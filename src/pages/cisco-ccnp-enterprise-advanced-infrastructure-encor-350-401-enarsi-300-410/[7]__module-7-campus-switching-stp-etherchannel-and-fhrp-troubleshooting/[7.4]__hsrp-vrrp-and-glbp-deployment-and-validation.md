## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Overview**

First-hop redundancy protocols (FHRPs) keep the default gateway available when a router or multilayer switch fails. In a typical LAN, hosts point to a single default gateway IP address, but the physical device behind that address can change without forcing host reconfiguration. The three protocols you need to know for CCNP-level deployment and validation are **HSRP** (Hot Standby Router Protocol), **VRRP** (Virtual Router Redundancy Protocol), and **GLBP** (Gateway Load Balancing Protocol).

These protocols solve the same availability problem in different ways:

- **HSRP** provides an active/standby gateway pair.
- **VRRP** provides a master/backup model standardized by the IETF.
- **GLBP** provides gateway redundancy plus load sharing across multiple routers.

At exam level, you should understand virtual IP behavior, virtual MAC learning, election logic, preemption, tracking, and the commands used to verify failover.

**Core FHRP concepts**

A host does not care which router owns the gateway as long as the gateway IP and MAC remain reachable. The protocol creates a **virtual IP address** that hosts use as their default gateway. The active device answers for that IP and forwards traffic. When failover occurs, another device takes over the same virtual IP, and in most cases the virtual MAC also remains stable from the host’s perspective.

Key terms:

- **Virtual IP**: the default gateway address configured on hosts.
- **Virtual MAC**: the Layer 2 address associated with the FHRP group or instance.
- **Active/Master**: the forwarding router in HSRP or VRRP.
- **Standby/Backup**: the router ready to take over.
- **Preemption**: the ability of a higher-priority router to reclaim the active role after recovery.
- **Tracking**: a mechanism that reduces priority when an interface or object fails.

Hosts learn the gateway MAC differently depending on the protocol:

- In **HSRP** and **VRRP**, hosts ARP for the virtual IP and learn the virtual MAC.
- In **GLBP**, hosts still use one virtual IP, but the active virtual gateway can hand out different virtual MACs to different hosts for load sharing.

> For IPv4 validation, use `show ip arp` on hosts or routers. For IPv6 validation, use `show ipv6 neighbors`; IPv6 does not use ARP.

**HSRP deployment**

HSRP is Cisco proprietary and is commonly used in Cisco-only environments. It creates a virtual router with one active router and one or more standby routers. The active router answers ARP requests for the virtual IP and forwards traffic. The standby router monitors hello messages and takes over if the active router fails.

HSRP versions:

- **HSRPv1**
  - Group range is `0-255`, but group `0` is reserved and not used in practice.
  - Virtual MAC format is `0000.0c07.acXX`, where `XX` is the group number in hexadecimal.
  - Supports IPv4 only.
- **HSRPv2**
  - Group range is `0-4095`.
  - Virtual MAC format is `0000.0c9f.fXXX`, where `XXX` is the group number in hexadecimal.
  - Supports IPv4 and is the version used for IPv6 HSRP configuration.

HSRP election behavior:

- Highest priority wins.
- Default priority is `100`.
- If priorities tie, the highest interface IP address wins.
- Preemption is not always enabled by default on every platform or release, so configure it explicitly when you want deterministic takeover after recovery.

Tracking is important in dual-uplink designs:

- **Interface tracking** lowers HSRP priority when a tracked interface goes down.
- **Object tracking** can track reachability of a route, IP SLA result, or other object.
- If the active router loses its upstream path, a lower-priority standby can take over even though the active router is still physically up.

Typical HSRP IPv4 configuration:

```cisco
interface GigabitEthernet0/0
 description LAN gateway
 ip address 10.10.10.2 255.255.255.0
 standby version 2
 standby 10 ip 10.10.10.1
 standby 10 priority 110
 standby 10 preempt
 standby 10 track GigabitEthernet0/1 20
```

A matching peer might use:

```cisco
interface GigabitEthernet0/0
 description LAN gateway
 ip address 10.10.10.3 255.255.255.0
 standby version 2
 standby 10 ip 10.10.10.1
 standby 10 priority 100
 standby 10 preempt
 standby 10 track GigabitEthernet0/1 20
```

What this does:

- Both routers share the virtual IP `10.10.10.1`.
- The router with priority `110` becomes active.
- If `GigabitEthernet0/1` fails on the active router, its priority drops by `20`.
- If the resulting priority falls below the peer, the standby router can take over.

HSRP for IPv6 uses the same design idea but different syntax:

```cisco
interface GigabitEthernet0/0
 ipv6 address 2001:db8:10:10::2/64
 standby version 2
 standby 10 ipv6 2001:db8:10:10::1
 standby 10 priority 110
 standby 10 preempt
```

Important IPv6 notes:

- Hosts use **Neighbor Discovery** instead of ARP.
- Validate with `show ipv6 neighbors`, not `show ip arp`.
- The virtual gateway still behaves like a shared first hop, but the host-side resolution process is IPv6-specific.

Useful HSRP verification commands:

```cisco
show standby brief
show standby
show standby interface GigabitEthernet0/0
show standby neighbors
show ip arp
show ipv6 neighbors
```

What to look for:

- `State is Active` or `Standby`
- `Active virtual MAC address`
- `Hello time` and `Hold time`
- `Priority`
- `Preemption enabled`
- `Track` status and decrements
- `Virtual IP address`

Example output cues:

- HSRPv1 virtual MAC for group `10` would be `0000.0c07.ac0a`.
- HSRPv2 virtual MAC for group `10` would be `0000.0c9f.f00a`.

**HSRP validation and failover workflow**

Use a repeatable test so you can prove both control-plane state and host reachability.

1. Verify the current role.
   ```cisco
   show standby brief
   ```
2. Confirm the host learned the virtual MAC.
   ```cisco
   show ip arp | include 10.10.10.1
   ```
3. Generate traffic through the gateway.
   ```cisco
   ping 10.10.10.1
   traceroute 8.8.8.8
   ```
4. Shut the active router’s tracked interface or the LAN interface.
   ```cisco
   interface GigabitEthernet0/1
    shutdown
   ```
5. Recheck the HSRP state and priority change.
   ```cisco
   show standby brief
   show standby
   ```
6. Confirm the host still reaches the same virtual IP after failover.
   ```cisco
   ping 10.10.10.1
   show ip arp | include 10.10.10.1
   ```

Expected behavior:

- The standby router becomes active if it has the higher effective priority.
- The virtual IP remains unchanged.
- The host ARP entry may refresh, but the default gateway IP stays the same.
- If preemption is enabled, the original higher-priority router can reclaim active status after recovery.

**VRRP deployment**

VRRP is an open standard defined by the IETF. It uses a **master/backup** model that is conceptually similar to HSRP. The master owns the virtual IP and responds to ARP for that address. Backup routers listen for advertisements and take over if the master fails.

VRRP election behavior:

- Highest priority wins.
- Default priority is `100`.
- The owner of the virtual IP has priority `255`.
- If priorities tie, the highest real interface IP address wins.
- Preemption behavior is implementation dependent, so configure and verify it rather than assuming a universal default.

VRRP virtual MAC format:

- IPv4 VRRP uses `0000.5e00.01XX`, where `XX` is the VRID in hexadecimal.
- The **VRID** is the Virtual Router Identifier, the group number for the VRRP instance.

Typical VRRP IPv4 configuration:

```cisco
interface GigabitEthernet0/0
 description LAN gateway
 ip address 10.20.20.2 255.255.255.0
 vrrp 20 ip 10.20.20.1
 vrrp 20 priority 110
 vrrp 20 preempt
```

Peer example:

```cisco
interface GigabitEthernet0/0
 description LAN gateway
 ip address 10.20.20.3 255.255.255.0
 vrrp 20 ip 10.20.20.1
 vrrp 20 priority 100
 vrrp 20 preempt
```

VRRP for IPv6 uses a similar model with IPv6 addressing and VRRPv3 support on platforms that implement it:

```cisco
interface GigabitEthernet0/0
 ipv6 address 2001:db8:20:20::2/64
 vrrp 20 ipv6 2001:db8:20:20::1
 vrrp 20 priority 110
 vrrp 20 preempt
```

Validation commands:

```cisco
show vrrp brief
show vrrp
show vrrp interface GigabitEthernet0/0
show ip arp
show ipv6 neighbors
```

What to inspect:

- `State` as `Master` or `Backup`
- `Virtual IP address`
- `Virtual MAC address`
- `Priority`
- `Preempt`
- `Advertisement interval`
- `Master down interval`
- `Track` status if configured

VRRP failover validation steps:

1. Confirm the master and backup roles.
   ```cisco
   show vrrp brief
   ```
2. Verify the host learned the virtual gateway MAC.
   ```cisco
   show ip arp | include 10.20.20.1
   ```
3. Test reachability through the gateway.
   ```cisco
   ping 10.20.20.1
   ```
4. Fail the master’s uplink or LAN interface.
   ```cisco
   interface GigabitEthernet0/1
    shutdown
   ```
5. Confirm the backup transitions to master.
   ```cisco
   show vrrp brief
   show vrrp
   ```
6. Restore the interface and confirm preemption behavior if enabled.
   ```cisco
   no shutdown
   ```

Practical VRRP notes:

- VRRP is often chosen when multi-vendor interoperability matters.
- The virtual MAC is standardized, which helps hosts keep a stable gateway identity.
- If you configure the owner with priority `255`, it is the natural master unless another design element changes the outcome.

**GLBP deployment**

GLBP combines redundancy with active load sharing. Unlike HSRP and VRRP, GLBP allows multiple routers to forward traffic for the same virtual IP at the same time. One router is the **Active Virtual Gateway** (AVG), and the others are **Active Virtual Forwarders** (AVFs).

How GLBP works:

- The AVG answers ARP requests for the virtual IP.
- The AVG assigns different virtual MAC addresses to hosts.
- Each AVF forwards traffic for the virtual MAC it owns.
- If an AVF fails, the AVG reassigns forwarding responsibility.

GLBP is useful when you want:

- Gateway redundancy
- Better utilization of multiple uplinks
- Simple host-side configuration with one default gateway IP

GLBP virtual MAC format:

- `0007.b400.XXYY`
- `XX` is the group number in hexadecimal.
- `YY` identifies the virtual forwarder.

Typical GLBP IPv4 configuration:

```cisco
interface GigabitEthernet0/0
 description LAN gateway
 ip address 10.30.30.2 255.255.255.0
 glbp 30 ip 10.30.30.1
 glbp 30 priority 110
 glbp 30 preempt
```

Peer example:

```cisco
interface GigabitEthernet0/0
 description LAN gateway
 ip address 10.30.30.3 255.255.255.0
 glbp 30 ip 10.30.30.1
 glbp 30 priority 100
 glbp 30 preempt
```

GLBP load-balancing modes:

- **Round-robin**
  - The AVG distributes hosts across AVFs in sequence.
  - This is commonly the default on many Cisco implementations, but verify on the platform you are studying.
- **Host-dependent**
  - The same host keeps receiving the same virtual MAC.
  - Useful when you want stable host-to-forwarder mapping.
- **Weighted**
  - The AVG assigns hosts based on router weight.
  - Useful when one router should carry more traffic than another.

Weighted GLBP is often paired with tracking so that forwarding responsibility shifts when a path fails. Exact command syntax can vary by platform and IOS/IOS XE release, so validate in your lab.

Example of a common weighted design pattern:

```cisco
interface GigabitEthernet0/0
 glbp 30 weighting 110
 glbp 30 weighting track GigabitEthernet0/1 20
 glbp 30 lower 90 upper 110
```

Use this as a conceptual pattern, not a universal syntax guarantee. Always confirm the exact command set on the platform you are configuring.

GLBP verification commands:

```cisco
show glbp brief
show glbp
show glbp interface GigabitEthernet0/0
show ip arp
```

What to inspect:

- `State` as `Active` or `Standby` for the AVG role
- `Forwarder` status for AVFs
- `Virtual MAC address`
- `Load balancing`
- `Weight`
- `Lower/upper thresholds`
- `Track` status
- `Hello` and `hold` timers

GLBP failover validation steps:

1. Confirm the AVG and AVF roles.
   ```cisco
   show glbp brief
   ```
2. Check that hosts learned different gateway MACs.
   ```cisco
   show ip arp | include 10.30.30.1
   ```
3. Generate traffic from multiple hosts and observe distribution.
   ```cisco
   ping 10.30.30.1
   ```
4. Fail one forwarding router or its tracked uplink.
   ```cisco
   interface GigabitEthernet0/1
    shutdown
   ```
5. Verify the AVG reassigns forwarding responsibility.
   ```cisco
   show glbp brief
   show glbp
   ```
6. Restore the interface and confirm the router rejoins the group.
   ```cisco
   no shutdown
   ```

**Comparison of HSRP, VRRP, and GLBP**

| Protocol | Standard | Gateway model | Load sharing | Virtual MAC behavior | Typical use |
|---|---|---|---|---|---|
| HSRP | Cisco proprietary | Active/standby | No | One virtual MAC per group | Cisco-only redundancy |
| VRRP | Open standard | Master/backup | No | One virtual MAC per VRID | Multi-vendor redundancy |
| GLBP | Cisco proprietary | AVG/AVF | Yes | Multiple virtual MACs per group | Redundancy plus load distribution |

Use this comparison to choose the right protocol:

- Choose **HSRP** when you want Cisco-native first-hop redundancy with familiar operational behavior.
- Choose **VRRP** when interoperability matters.
- Choose **GLBP** when you want multiple routers to actively forward traffic for the same subnet.

**Tracking, timers, and failure behavior**

Tracking is one of the most tested topics because it changes the election outcome without requiring a full device failure. You should understand the difference between a router being up and a router being a good gateway.

Common tracking patterns:

- Track the uplink toward the core or WAN.
- Track a static route or default route.
- Track an IP SLA probe to a next-hop or internet target.
- Reduce priority when the tracked object fails.

Why tracking matters:

- A router can still answer HSRP, VRRP, or GLBP hellos even if its upstream path is broken.
- Without tracking, hosts may keep using a gateway that cannot forward traffic effectively.
- Tracking makes failover reflect real forwarding health, not just interface state.

Timer concepts:

- **Hello time**: how often protocol messages are sent.
- **Hold time**: how long a peer waits before declaring failure.
- **Advertisement interval**: VRRP message interval.
- **Master down interval**: VRRP failover timer derived from advertisements and priority.

In most exam scenarios, you do not need to memorize every timer formula, but you should know that shorter timers speed failover at the cost of more control traffic.

**Validation checklist**

Use this checklist when troubleshooting FHRPs in a lab or exam scenario:

- Confirm the interface is up/up and in the correct VLAN or subnet.
- Confirm the virtual IP is configured identically on all peers.
- Confirm the group or VRID matches on all peers.
- Confirm priorities and preemption settings match the design.
- Confirm tracking is decrementing priority as expected.
- Confirm the host learned the correct virtual MAC.
- Confirm the protocol-specific show command reports the expected role.
- Confirm IPv4 validation uses ARP and IPv6 validation uses Neighbor Discovery.

Useful commands by protocol:

```cisco
! HSRP
show standby brief
show standby
show standby interface g0/0

! VRRP
show vrrp brief
show vrrp
show vrrp interface g0/0

! GLBP
show glbp brief
show glbp
show glbp interface g0/0
```

**Common platform and lab caveats**

- Command syntax can vary slightly across IOS, IOS XE, and older lab images.
- Some features, especially authentication and advanced weighting options, are platform dependent.
- HSRP version selection matters because the virtual MAC format and group range differ.
- VRRPv3 is the relevant version for IPv6 on supported platforms.
- GLBP weighting and tracking syntax should be verified in the specific software release you are using.
- Do not use `show ip arp` to validate IPv6 failover; use `show ipv6 neighbors`.

**Practical deployment patterns**

A common enterprise design uses two distribution switches with FHRP on the user VLANs:

- The primary distribution switch has higher priority.
- The secondary distribution switch has lower priority.
- Each VLAN uses a different active router to balance traffic when using HSRP or VRRP across multiple VLANs.
- Tracking ensures the active role follows the best upstream path.

Example design idea for HSRP across two VLANs:

- VLAN 10 active on Dist1
- VLAN 20 active on Dist2
- Both routers preempt after recovery
- Uplink tracking reduces priority if the core path fails

This pattern improves resilience and can also distribute gateway processing load without requiring GLBP.

**Exam focus summary**

Remember these high-value points:

- **HSRP** is Cisco proprietary and uses active/standby.
- **VRRP** is standardized and uses master/backup.
- **GLBP** provides redundancy plus load sharing.
- Virtual MAC formats are protocol-specific and are often tested.
- Preemption and tracking determine whether the preferred router regains or loses the active role.
- Validation requires protocol-specific show commands and the correct host table: ARP for IPv4, Neighbor Discovery for IPv6.
- GLBP load balancing can be round-robin, host-dependent, or weighted, depending on design and platform support.
