## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Spanning Tree context**

Rapid PVST+ and MST are both loop-prevention designs built on the same core goal: keep one active Layer 2 path while allowing fast recovery from failures. The difference is how they scale and how they represent VLANs.

- **Rapid PVST+** runs one Rapid Spanning Tree Protocol (RSTP) instance per VLAN.
- **MST** maps many VLANs into fewer spanning-tree instances to reduce control-plane load.
- Both rely on BPDUs (Bridge Protocol Data Units), but they do not process them the same way.
- Interoperability matters most at boundaries, where one side may run per-VLAN logic and the other side runs instance-based logic.

**Rapid PVST+ behavior**

Rapid PVST+ is Cisco’s per-VLAN implementation of RSTP. Each VLAN has its own topology, root election, and port roles. That gives you fine-grained control, but it also increases the number of spanning-tree instances and BPDUs.

Key operational points:

- Each VLAN can have a different root bridge.
- A port can be forwarding for one VLAN and blocking for another.
- Convergence is rapid when the link type and neighbor behavior support proposal/agreement handshakes.
- Edge ports can transition immediately when configured with PortFast.

RSTP uses the classic port roles plus rapid-transition logic:

- **Root port**: best path toward the root bridge for that VLAN.
- **Designated port**: best path away from the segment for that VLAN.
- **Alternate port**: backup path toward the root, usually blocking.
- **Backup port**: backup on a shared segment, less common in modern designs.
- **Edge port**: port connected to an end device; it skips the normal listening/learning delay when PortFast is enabled.

Rapid transition is not guaranteed on every link. It depends on conditions such as:

- Point-to-point link type
- Proposal/agreement exchange
- Neighbor support for RSTP behavior
- Whether the port is treated as an edge port

> PortFast is for end hosts, not for “trusted” uplinks. Do not describe it as a trust feature.

**MST architecture**

Multiple Spanning Tree (MST) groups VLANs into instances so the switch does not need a separate tree for every VLAN. This improves scalability in large Layer 2 domains.

MST has three important concepts:

- **CIST** (Common and Internal Spanning Tree): the common tree that connects MST regions and external STP domains.
- **IST** (Internal Spanning Tree): instance 0 inside the region; it carries the region’s internal view and the CIST information.
- **MSTI** (MST Instance): additional instances mapped to VLAN ranges.

A useful way to think about MST:

- Inside a region, VLANs are mapped to instances.
- Across regions, the CIST provides the common interconnection.
- The region boundary must interoperate with external STP domains using the common spanning-tree behavior and, when needed, PVST simulation for Cisco per-VLAN interoperability.

MST is based on RSTP behavior for the CIST/IST and uses rapid-transition concepts, but it is not simply “one RSTP per instance” in the same way Rapid PVST+ is. Each MST instance has its own topology and port roles, yet the protocol behavior is coordinated through the region and the CIST.

**MST region design**

All switches in the same MST region must match exactly on these items:

- **Region name**
- **Revision number**
- **VLAN-to-instance mapping**

If any of these differ, the switch treats the neighbor as being in a different region.

Practical configuration example:

```cisco
conf t
spanning-tree mode mst
spanning-tree mst configuration
 name CAMPUS-A
 revision 10
 instance 1 vlan 10,20
 instance 2 vlan 30,40
 instance 3 vlan 50-60
 exit
end
```

Verification commands:

```cisco
show spanning-tree mst configuration
show spanning-tree mst
show spanning-tree mst 1
show spanning-tree mst 2
```

What to look for:

- The region name and revision must match on all intended region members.
- The VLAN mapping must be identical.
- The digest should match across the region.

If the digest differs, the switch considers the neighbor a different region. Operationally, that means:

- The link becomes a **boundary** between regions.
- The switch uses CIST behavior toward that neighbor.
- Per-instance internal coordination stops at the boundary.

> A digest mismatch does not mean the link is down. It means the switches do not agree on region identity, so the port behaves as a boundary.

**MST port roles and topology behavior**

MST uses per-instance roles, so the same physical port can have different roles in different instances. That is normal and important for exam questions.

Common roles:

- **Root port**: best path to the CIST root or to the instance root, depending on the instance view.
- **Designated port**: forwarding port for that segment in that instance.
- **Alternate port**: backup path for that instance.
- **Backup port**: backup on the same segment, less common.
- **Master port**: a boundary-related role used in MST to reach the CIST root from inside the region.

The **master port** is often misunderstood. It is not a generic role like root or designated. It is the port on the boundary of an MST region that provides the best path from the region toward the CIST root. It matters when the region connects to the rest of the spanning-tree domain.

Practical interpretation:

- Inside the region, each MST instance elects its own root and port roles.
- The CIST connects the region to external STP domains.
- The master port helps the region reach the CIST root through the boundary.

Example of per-instance differences:

- Port `Gi1/0/1` may be root port in MST instance 1.
- The same port may be designated in MST instance 2.
- Another port may be alternate in one instance and forwarding in another.

That is expected because roles are evaluated per instance, not per physical interface globally.

**BPDU processing differences**

The BPDU model is one of the biggest exam topics.

Rapid PVST+:

- Sends a separate BPDU stream per VLAN.
- Each VLAN has its own RSTP state machine.
- Trunks carry per-VLAN spanning-tree information.

MST:

- Sends a single MST BPDU structure that carries the CIST information and MST instance information.
- Does not send a separate BPDU per VLAN the way Rapid PVST+ does.
- Uses the VLAN-to-instance mapping to decide which instance controls each VLAN.

This matters at boundaries:

- A Rapid PVST+ switch expects per-VLAN STP behavior on a trunk.
- An MST boundary must simulate per-VLAN behavior well enough for interoperability.
- Cisco devices use **PVST simulation** on the MST side when interoperating with Rapid PVST+.

That means the Rapid PVST+ side does not simply “see one tree.” It still runs per-VLAN STP, and the MST boundary must present compatible behavior for those VLANs.

**Mixed Rapid PVST+ and MST interoperability**

Mixed-mode links are common in migrations. The key is to understand that the boundary is not just a generic CIST-only exchange.

Operational behavior:

- The MST switch treats the link to Rapid PVST+ as a boundary.
- The MST side participates in PVST simulation for VLAN interoperability.
- The Rapid PVST+ side continues to run per-VLAN RSTP.
- The effective forwarding/blocking state must remain consistent enough to prevent loops.

Typical migration design:

- Keep the MST region stable.
- Match VLANs carefully across the trunk.
- Ensure the root placement is intentional.
- Verify boundary ports and per-VLAN states during cutover.

Example topology idea:

- Core switch runs MST.
- Access/distribution switch runs Rapid PVST+.
- Trunk between them carries VLANs 10, 20, 30, 40.

Operational expectation:

- The MST switch maps those VLANs into instances.
- The boundary port simulates per-VLAN behavior toward the Rapid PVST+ neighbor.
- One VLAN may forward while another blocks, depending on root placement and instance mapping.

**Rapid transition behavior**

RSTP-style rapid convergence depends on protocol conditions, not just the feature name.

Rapid transition is most likely when:

- The port is point-to-point.
- The neighbor participates in proposal/agreement.
- The port is not an edge port.
- The topology change does not require legacy timer-based fallback.

Rapid transition may be suppressed when:

- The link is treated as shared.
- The neighbor does not support RSTP-style handshakes.
- The port is a boundary with interoperability constraints.
- The topology requires compatibility behavior.

Classic STP timers still exist, but in rapid modes they are not the normal mechanism for everyday convergence. They matter more for compatibility and fallback than for the primary fast-transition path.

**Practical configuration examples**

Rapid PVST+:

```cisco
conf t
spanning-tree mode rapid-pvst
spanning-tree vlan 10 root primary
spanning-tree vlan 20 root secondary
interface gi1/0/24
 spanning-tree portfast
 spanning-tree bpduguard enable
end
```

MST:

```cisco
conf t
spanning-tree mode mst
spanning-tree mst configuration
 name CAMPUS-A
 revision 10
 instance 1 vlan 10,20
 instance 2 vlan 30,40
 exit
spanning-tree mst 1 root primary
spanning-tree mst 2 root secondary
interface gi1/0/24
 spanning-tree portfast
 spanning-tree bpduguard enable
end
```

Notes on these examples:

- `root primary` and `root secondary` influence root election for the selected VLANs or instances.
- `portfast` is appropriate only on edge ports connected to end devices.
- `bpduguard` protects edge ports from accidental switch connections.

**Verification and troubleshooting workflow**

Use a structured approach when checking Rapid PVST+ or MST behavior.

1. Confirm the spanning-tree mode.
   ```cisco
   show spanning-tree summary
   ```
   Look for `rapid-pvst` or `mst`.

2. Verify region consistency on MST switches.
   ```cisco
   show spanning-tree mst configuration
   ```
   Check the name, revision, and VLAN mapping.

3. Check the digest and region membership.
   ```cisco
   show spanning-tree mst
   ```
   Confirm whether the neighbor is in the same region or treated as a boundary.

4. Inspect root placement.
   ```cisco
   show spanning-tree mst 1
   show spanning-tree vlan 10
   ```
   Verify which switch is root for the instance or VLAN.

5. Check interface role and state.
   ```cisco
   show spanning-tree interface gi1/0/1 detail
   ```
   On some IOS platforms, this command is supported; on others, output format may vary. Use it to confirm role, state, and topology-change information.

6. Validate trunk and VLAN transport.
   ```cisco
   show interfaces trunk
   show vlan brief
   ```
   Make sure the VLANs are allowed and active.

7. Look for boundary behavior in mixed-mode links.
   ```cisco
   show spanning-tree mst detail
   show spanning-tree vlan 10 detail
   ```
   Check for evidence that the port is acting as a boundary and that per-VLAN behavior is being simulated correctly.

**Common exam scenarios**

- **Same MST region, different root per instance**
  - This is normal.
  - One switch can be root for instance 1 while another is root for instance 2.

- **MST region mismatch**
  - Different name, revision, or VLAN mapping causes a boundary.
  - The link still works, but it no longer behaves as one MST region.

- **Rapid PVST+ to MST trunk**
  - The MST side must interoperate with per-VLAN behavior.
  - Expect PVST simulation behavior at the boundary.

- **Port role differs by instance**
  - A port can be designated in one instance and alternate in another.
  - Always read the instance number in the output.

- **Edge port misconfiguration**
  - If you enable PortFast on a switch-to-switch link, you risk loops.
  - Use BPDU Guard on true access ports to protect against accidental bridging.

**Troubleshooting mixed-mode trunks**

When a Rapid PVST+ and MST link behaves unexpectedly, check these items in order:

1. Confirm both sides agree on the trunk.
   - Allowed VLANs
   - Native VLAN
   - Link status

2. Confirm the MST region is correct.
   - Name
   - Revision
   - VLAN mapping

3. Confirm the root bridge placement.
   - Decide which side should be root for each VLAN or instance.

4. Check for boundary behavior.
   - A mismatch creates a boundary.
   - Boundary ports may show different behavior than internal ports.

5. Check for blocked VLANs or unexpected forwarding.
   - Per-VLAN state on Rapid PVST+
   - Per-instance state on MST

Example troubleshooting commands:

```cisco
show spanning-tree vlan 10
show spanning-tree mst 1
show spanning-tree mst detail
show spanning-tree interface gi1/0/1 detail
show interfaces trunk
```

**Practical interpretation of outputs**

When reading spanning-tree output, focus on these fields:

- **Root ID**: identifies the root bridge for the VLAN or instance.
- **Bridge ID**: identifies the local switch.
- **Port role**: root, designated, alternate, backup, or master.
- **Port state**: forwarding, learning, discarding.
- **Boundary status**: indicates interaction with another region or STP domain.
- **Topology change**: helps explain recent reconvergence.

For MST, remember:

- Instance 0 is the IST.
- Other instances map to selected VLANs.
- Unmapped VLANs are assigned according to the MST configuration, but do not assume a universal default mapping display without checking the actual configuration.

**Quick comparison**

| Feature | Rapid PVST+ | MST |
|---|---|---|
| Control model | One instance per VLAN | Multiple VLANs per instance |
| BPDU behavior | Per-VLAN BPDUs | CIST BPDU plus instance information |
| Scaling | Higher overhead | Better scalability |
| Root control | Per VLAN | Per instance |
| Boundary behavior | Interoperates with MST through PVST simulation | Simulates per-VLAN behavior toward Rapid PVST+ |
| Role differences | Per VLAN | Per instance |

**What to remember for the exam**

- Rapid PVST+ is per-VLAN RSTP.
- MST groups VLANs into instances and uses a region concept.
- MST region identity depends on name, revision, and VLAN mapping.
- A digest mismatch creates a boundary.
- Mixed Rapid PVST+/MST links require PVST simulation behavior on the MST side.
- Port roles are evaluated per instance, so one physical port can have different roles in different instances.
- PortFast is for edge ports, not for trusted uplinks.
- Rapid convergence depends on link type and protocol conditions, not just the mode name.
