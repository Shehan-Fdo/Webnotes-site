## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Spanning Tree Protocol overview**

<abbr title="Spanning Tree Protocol">STP</abbr> prevents Layer 2 loops by building a loop-free tree and blocking redundant paths. In campus networks, you use STP to keep Ethernet redundancy without allowing broadcast storms, MAC table instability, or duplicate frames.

Cisco exam topics commonly focus on how STP elects the root bridge, how ports are assigned roles, and how guard features protect the topology from bad cabling, misconfigurations, or rogue switches. The key is to understand not just the commands, but the forwarding logic behind them.

- **PVST+** runs one STP instance per VLAN, so each VLAN can have a different root bridge.
- **Rapid PVST+** uses Rapid Spanning Tree behavior per VLAN and converges faster than classic 802.1D STP.
- **MST** (<abbr title="Multiple Spanning Tree">MST</abbr>) maps multiple VLANs to fewer spanning-tree instances, which reduces control-plane overhead in large environments.

**STP variants and where root placement matters**

Root placement is not just a design preference; it determines which links carry traffic for each VLAN or instance. In PVST+ and Rapid PVST+, you can intentionally place different roots for different VLANs to load-balance traffic across distribution switches. In MST, root placement is done per MST instance, and all VLANs mapped to that instance follow the same root and forwarding decisions.

| Mode | Root election scope | Typical use | Design implication |
|---|---|---|---|
| PVST+ | Per VLAN | Legacy Cisco campus | Different VLANs can use different roots |
| Rapid PVST+ | Per VLAN | Common Cisco campus | Faster convergence with per-VLAN control |
| MST | Per MST instance | Large multi-VLAN campus | VLANs in the same instance share one topology |

A practical design pattern is to make the distribution layer the root for access VLANs and, if needed, split primary roots across two distribution switches by VLAN. That keeps traffic local and avoids suboptimal paths through the core.

**Bridge ID and root bridge election**

STP elects the root bridge using the lowest <abbr title="Bridge Identifier">Bridge ID</abbr> (BID). The BID is made of:

- **Bridge priority**
- **Extended system ID** in VLAN-aware modes
- **MAC address** as the tie-breaker

In PVST+ and Rapid PVST+, the extended system ID incorporates the VLAN ID into the priority field. That means the effective priority is not just the configured value; it is the configured priority plus the VLAN identifier behavior used by Cisco STP. Because of this, you normally set priority in increments of 4096.

- Valid manual priority values are typically `0`, `4096`, `8192`, and so on up to `61440`.
- Lower priority wins.
- If priorities tie, the lower MAC address wins.

> In VLAN-aware STP modes, the root bridge can be different for each VLAN because each VLAN has its own election domain.

Cisco convenience commands calculate the priority for you:

```bash
spanning-tree vlan 10 root primary
spanning-tree vlan 10 root secondary
spanning-tree vlan 10,20,30 root primary
```

These commands do not create a special “secondary root” role. They simply set the bridge priority to a value that makes the switch likely to become root primary or a backup candidate if the current root fails. If you need deterministic control, set the priority manually in 4096 increments.

**Root bridge placement strategy**

You usually place the root bridge at the distribution layer, not at the access layer. The goal is to keep the shortest, most predictable Layer 2 paths between access switches and the default gateway location.

Common campus design goals:

- Keep the root close to the majority of access switches.
- Avoid placing the root on an access switch.
- Avoid placing the root on a core switch if the core is meant to stay transit-focused and Layer 3-centric.
- Use different roots per VLAN or instance to balance traffic when appropriate.

A simple two-distribution-switch design might look like this:

```text
           Core
         /      \
   Dist1          Dist2
   /  \            /  \
Acc1  Acc2      Acc3  Acc4
```

Example root placement:

- VLAN 10 root on `Dist1`
- VLAN 20 root on `Dist2`
- VLAN 30 root on `Dist1`

This lets traffic for different VLANs use different uplinks while still keeping each VLAN’s root in the distribution layer.

**How STP chooses ports**

STP selects roles based on the best path toward the root bridge. The main decision factors are:

1. Lowest root path cost to the root bridge
2. Lowest sender bridge ID
3. Lowest sender port ID
4. Local port ID as a final tie-breaker

Port roles differ by STP mode, but the logic is consistent:

- **Root port**: the best port on a non-root switch toward the root bridge
- **Designated port**: the forwarding port for a segment
- **Alternate port**: a backup path toward the root in Rapid STP modes
- **Backup port**: a backup on a shared segment, less common in modern campus designs

Path cost is critical. STP prefers the lowest cumulative cost, not necessarily the physically shortest cable. Higher-speed links have lower cost, so a 10G link is preferred over a 1G link if all else is equal.

Example cost logic:

- 10G uplink to Dist1: lower cost
- 1G uplink to Dist2: higher cost
- Result: the 10G path becomes the root port if it offers the best total cost

**Step-by-step root election example**

Consider this topology:

```text
        SW1
       /   \
     SW2   SW3
       \   /
        SW4
```

Assume all switches use the same STP mode and VLAN, and the priorities are:

- SW1: priority 4096
- SW2: priority 8192
- SW3: priority 8192
- SW4: priority 12288

Election outcome:

1. SW1 wins root bridge election because it has the lowest BID.
2. SW2 and SW3 each choose the best port toward SW1 as their root port.
3. SW4 compares paths through SW2 and SW3.
4. The path with the lower total cost becomes SW4’s root port.
5. The other uplink becomes blocked or alternate, depending on the STP mode.

If the costs are equal, STP uses bridge ID and then port ID tie-breakers. This is why predictable priority planning matters more than relying on MAC address outcomes.

**Classic STP, Rapid STP, and MST behavior**

Classic 802.1D STP converges slowly because it uses listening and learning states before forwarding. Rapid STP improves convergence by using proposal/agreement handshakes and faster role transitions. MST combines VLANs into instances so you can scale better in large environments.

Key differences:

- **802.1D STP**
  - Slower convergence
  - Uses blocking, listening, learning, forwarding
  - Topology changes are handled more conservatively

- **Rapid PVST+ / 802.1w behavior**
  - Faster convergence
  - Uses alternate and backup roles
  - Better for modern campus access/distribution designs

- **MST**
  - Maps VLANs to instances
  - One root per instance, not per VLAN
  - Useful when many VLANs would otherwise create too many STP instances

Root placement principles are similar across modes, but the scope changes:

- In PVST+ and Rapid PVST+, root placement is per VLAN.
- In MST, root placement is per MST instance.
- In all modes, you still want the root close to the access layer and aligned with traffic flow.

**Loop prevention features and where to use them**

Cisco provides several guard features that solve different problems. You should know where each one belongs.

| Feature | Protects against | Typical placement | Effect when triggered |
|---|---|---|---|
| BPDU Guard | Unexpected switch on an edge port | PortFast access ports | Err-disables the port |
| BPDU Filter | BPDU transmission/participation on edge ports | Very carefully, only when you truly want silence | Can suppress STP participation and is risky |
| Root Guard | A downstream switch trying to become root | Downstream-facing switch ports where root must never appear | Moves port to root-inconsistent |
| Loop Guard | Unidirectional link or missing BPDUs on non-designated ports | Redundant blocked ports and point-to-point links | Prevents accidental forwarding |
| UDLD | Unidirectional fiber/copper link detection | Fiber or links prone to one-way failures | Err-disables or flags the link |

**BPDU Guard**

BPDU Guard is the standard protection for PortFast edge ports. If a switch receives a BPDU on a port that should only connect to an end device, it assumes a switch was connected and shuts the port down.

Typical use:

```bash
interface range gigabitEthernet1/0/1-24
 spanning-tree portfast
 spanning-tree bpduguard enable
```

You can also enable it globally on many Cisco platforms for PortFast ports:

```bash
spanning-tree portfast default
spanning-tree bpduguard default
```

Operationally, this is the safest default for access ports because it protects against accidental switch connections without requiring per-port manual policing.

**BPDU Filter**

BPDU Filter is more dangerous than BPDU Guard because it suppresses STP participation. Use it only when you fully understand the consequences.

Important behavior distinctions:

- **Interface-level BPDU filter** on a PortFast port suppresses BPDUs on that interface and is intended for edge use.
- **Global BPDU filter** behaves differently and is tied to PortFast defaults on many Cisco platforms.
- On non-edge ports, BPDU filter can create loops if used incorrectly.

A safe exam takeaway is:

- Use BPDU Guard on edge ports.
- Avoid BPDU Filter unless you have a very specific design reason.
- Do not use BPDU Filter as a general loop-prevention tool.

> BPDU Filter can hide the very control traffic STP needs to detect loops, so it is usually a last-resort feature rather than a best practice.

**Root Guard**

Root Guard prevents a downstream switch from becoming root on a port where you expect the current root placement to remain stable. It is commonly applied on distribution switch ports that face access switches or other downstream switches.

Example:

```bash
interface range gigabitEthernet1/0/1-24
 spanning-tree guard root
```

Use Root Guard only on ports where a superior BPDU would be unexpected and undesirable. Do not apply it blindly to uplinks that should legitimately participate in root election. If a superior BPDU is received, the port transitions to root-inconsistent and stops forwarding until the condition clears.

Common use cases:

- Distribution-to-access downlinks
- Ports toward switches that must never become root
- Controlled campus hierarchies where root placement is fixed

**Loop Guard**

Loop Guard protects against a scenario where a non-designated port stops receiving BPDUs and might incorrectly transition to forwarding. This can happen with unidirectional failures or certain STP anomalies.

Use Loop Guard on redundant links where a blocked port should remain blocked unless BPDUs are consistently received.

Example:

```bash
spanning-tree loopguard default
```

Or per interface on supported platforms:

```bash
interface gigabitEthernet1/0/48
 spanning-tree guard loop
```

Loop Guard is especially useful on point-to-point uplinks in a redundant topology. It complements STP by preventing a blocked port from becoming forwarding just because BPDUs disappeared.

**UDLD and its relationship to STP**

<abbr title="Unidirectional Link Detection">UDLD</abbr> detects one-way link failures, especially on fiber links. STP assumes both directions of a link are working; UDLD verifies that assumption.

Use UDLD when:

- You have fiber links where one direction can fail independently.
- You want faster detection of one-way failures than STP alone provides.
- You need a complement to STP guard features, not a replacement.

Typical interface configuration on supported platforms:

```bash
interface tenGigabitEthernet1/1/1
 udld enable
```

Aggressive mode is platform-dependent and should be verified for the specific IOS/IOS XE train:

```bash
interface tenGigabitEthernet1/1/1
 udld aggressive
```

UDLD is complementary to STP because it detects physical one-way failures that STP may not immediately interpret correctly. In campus designs, UDLD is often preferred on fiber uplinks, while STP guard features protect against logical topology mistakes.

**PortFast interaction and edge-port caveats**

PortFast makes an access port transition to forwarding quickly by skipping the normal listening and learning delay. It is intended for end devices, not switches.

Best practices:

- Enable PortFast only on true edge ports.
- Pair PortFast with BPDU Guard.
- Avoid PortFast on trunk links unless the platform and design explicitly support an edge trunk use case.

Example:

```bash
interface gigabitEthernet1/0/10
 switchport mode access
 spanning-tree portfast
 spanning-tree bpduguard enable
```

If a switch is accidentally connected to a PortFast port, BPDU Guard is the safety net. Without it, the port may participate in STP in an unexpected way and create a loop risk.

**Topology change handling**

When STP detects a topology change, it updates forwarding behavior so the network can relearn MAC addresses and avoid stale entries. In classic 802.1D, the topology change notification causes switches to shorten MAC aging time and flush learned entries in a controlled way. In Rapid STP and MST, convergence and notification handling are faster and more efficient.

Exam-relevant points:

- A topology change does not mean “all MAC entries disappear instantly” in a generic sense.
- The TC flag influences how switches age and refresh MAC addresses.
- Rapid STP and MST handle changes more efficiently than classic STP.

**Verification and troubleshooting commands**

Use these commands to confirm root placement, port roles, and guard behavior:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree root
show spanning-tree interface gigabitEthernet1/0/1 detail
show spanning-tree inconsistentports
show spanning-tree summary
show udld neighbors
show udld interface tenGigabitEthernet1/1/1
show interfaces status err-disabled
show logging | include SPANTREE|UDLD|BPDU
```

What to look for:

- **Root bridge**: lowest BID for the VLAN or instance
- **Root port**: the best upstream path on non-root switches
- **Designated port**: forwarding port on a segment
- **Alternate port**: backup path in Rapid STP
- **Root-inconsistent**: Root Guard has blocked a port
- **Loop-inconsistent**: Loop Guard has protected a port
- **Err-disabled**: BPDU Guard or UDLD may have shut the port

Example output clues:

- `Root ID` shows the elected root bridge.
- `This bridge is the root` confirms the local switch is root.
- `Role Root` identifies the root port.
- `Role Altn` identifies an alternate port.
- `Inconsistent` states indicate guard features are actively protecting the topology.

**Practical configuration example**

A common distribution/access design might use the following pattern:

```bash
! Distribution switch 1
spanning-tree mode rapid-pvst
spanning-tree vlan 10,30 root primary
spanning-tree vlan 20 root secondary

interface range gigabitEthernet1/0/1-24
 description Downlink to access switch
 spanning-tree guard root

interface range gigabitEthernet1/0/49-52
 description Uplink to core or peer distribution
 no spanning-tree guard root
```

```bash
! Distribution switch 2
spanning-tree mode rapid-pvst
spanning-tree vlan 20 root primary
spanning-tree vlan 10,30 root secondary

interface range gigabitEthernet1/0/1-24
 description Downlink to access switch
 spanning-tree guard root
```

```bash
! Access switch
spanning-tree mode rapid-pvst
spanning-tree portfast default
spanning-tree bpduguard default
spanning-tree loopguard default

interface range gigabitEthernet1/0/1-24
 description User access ports
 switchport mode access
```

This design keeps the root in the distribution layer, protects edge ports from rogue switches, and uses guard features where they make operational sense.

**Exam tips**

- Know that root election is based on the lowest BID, not the highest bandwidth.
- Remember that PVST+ and Rapid PVST+ elect roots per VLAN, while MST elects roots per instance.
- Use Root Guard on downstream-facing switch ports where a superior BPDU must not win.
- Use BPDU Guard on PortFast edge ports.
- Use Loop Guard on redundant ports that should stay blocked if BPDUs stop arriving.
- Use UDLD to detect one-way link failures, especially on fiber.
- Understand that BPDU Filter is risky and should not be treated as a general-purpose protection feature.

**Quick comparison of guard features**

| Feature | Best for | Not for |
|---|---|---|
| BPDU Guard | End-device access ports | Switch-to-switch links |
| Root Guard | Fixed-root downstream links | Uplinks that may legitimately become root paths |
| Loop Guard | Redundant blocked links | Edge ports with PortFast |
| BPDU Filter | Rare special cases | General campus protection |
| UDLD | One-way link detection | Replacing STP |

**Common troubleshooting workflow**

1. Verify the STP mode and root bridge with `show spanning-tree`.
2. Check whether the expected distribution switch is root for the VLAN or instance.
3. Confirm port roles and costs on uplinks.
4. Look for inconsistent or err-disabled states.
5. Check logs for BPDU Guard, Root Guard, Loop Guard, or UDLD events.
6. Fix the physical or logical cause before clearing the port state.

A disciplined root placement plan plus the right guard features gives you predictable convergence and strong loop protection. In exam scenarios, always ask which device should be root, which ports are edge ports, and which links must remain protected from unexpected BPDUs.
