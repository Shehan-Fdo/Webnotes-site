## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**EtherChannel fundamentals**

EtherChannel bundles multiple physical links into one logical interface to increase bandwidth, improve resiliency, and simplify topology. Cisco platforms may call the logical interface a <abbr title="Port-channel interface">Port-Channel</abbr> or `PoX`. The bundle behaves as a single Layer 2 or Layer 3 interface, so spanning tree, routing, and many policies apply to the logical interface rather than each member link.

The key idea is that member links must be compatible. EtherChannel does not “average out” mismatched settings; it selects links that meet the bundle rules and places incompatible links in a non-bundled state. That behavior is central to troubleshooting because a link can be physically up but logically excluded.

- **Use cases**
  - Increase aggregate throughput for uplinks, server connections, and inter-switch trunks.
  - Provide link redundancy without changing the logical topology.
  - Reduce spanning tree blocking by treating parallel links as one logical path.
- **Common protocols**
  - **PAgP**: Port Aggregation Protocol, Cisco-proprietary negotiation.
  - **LACP**: Link Aggregation Control Protocol, IEEE 802.3ad / 802.1AX standard negotiation.
  - **Static**: `on` mode, no negotiation protocol.

**Negotiation modes and compatibility**

Negotiation determines whether both ends actively form a bundle or simply force membership. The exact mode names matter because Cisco uses different terms for PAgP and LACP.

- **PAgP modes**
  - `desirable`: actively negotiates.
  - `auto`: passively waits for a PAgP neighbor.
  - `on`: forces EtherChannel without PAgP.
- **LACP modes**
  - `active`: actively negotiates.
  - `passive`: waits for LACP neighbor activity.
  - `on`: forces EtherChannel without LACP.
- **Important distinction**
  - `silent` is not a PAgP mode.
  - On some platforms and contexts, “silent/non-silent” describes LACP behavior or platform-specific handling, but it is not a PAgP negotiation mode.

A bundle forms only when both sides agree on protocol and parameters. For exam purposes, remember the valid pairings and the fact that `auto` is passive.

| Local mode | Remote mode | Forms? | Notes |
|---|---:|---:|---|
| PAgP `desirable` | PAgP `desirable` | Yes | Both sides actively negotiate. |
| PAgP `desirable` | PAgP `auto` | Yes | Active side can recruit passive side. |
| PAgP `auto` | PAgP `auto` | No | Neither side initiates. |
| PAgP `on` | PAgP `on` | Yes | Static bundle, no PAgP. |
| LACP `active` | LACP `active` | Yes | Both sides can initiate. |
| LACP `active` | LACP `passive` | Yes | Active side starts negotiation. |
| LACP `passive` | LACP `passive` | No | Neither side initiates. |
| LACP `on` | LACP `on` | Yes | Static bundle, no LACP. |
| Any negotiation mode | `on` | Usually no | Mixed static and negotiated modes do not form a negotiated bundle. |

> A static `on` bundle is not the same as a negotiated bundle. If one side uses `on` and the other uses PAgP or LACP, the links typically do not negotiate correctly and may remain individual or suspended depending on platform behavior.

**Layer 2 and Layer 3 EtherChannel rules**

EtherChannel members must match in operational role. The most important rule is not “routed interfaces cannot join a bundle”; the real rule is that all members in a given EtherChannel must use the same Layer 2 or Layer 3 mode.

- **Layer 2 EtherChannel**
  - Member interfaces are switchports.
  - The port-channel can be an access port or trunk.
  - All members must match in VLAN, trunking, native VLAN, allowed VLANs, and related switchport settings.
- **Layer 3 EtherChannel**
  - Member interfaces are routed ports.
  - The port-channel itself is configured with an IP address.
  - Member interfaces should not carry Layer 2 switchport configuration.
- **Mismatch rule**
  - You cannot mix Layer 2 and Layer 3 member behavior in the same bundle.
  - A routed physical interface can participate in a Layer 3 port-channel, but not in a Layer 2 port-channel.

**What must match for a bundle to form**

Cisco checks many parameters before placing a port into the bundle. Exact enforcement can vary by platform and IOS/IOS XE release, but the common requirements are consistent.

- **Speed and duplex**
  - Must match across members.
  - Auto-negotiation settings should be consistent.
- **Switchport mode**
  - Access, trunk, or routed behavior must align.
- **VLAN settings**
  - Access VLAN must match for access bundles.
  - Native VLAN and allowed VLAN list must match for trunks.
- **Layer 3 settings**
  - IP addressing is configured on the port-channel, not on individual Layer 2 members.
- **MTU and encapsulation**
  - Must be consistent where enforced.
- **Channel protocol**
  - Both sides must use compatible negotiation or static mode.
- **Additional features**
  - Features such as voice VLAN, port security, storm control, and QoS can affect compatibility depending on platform.

**Practical configuration examples**

Use the port-channel as the logical interface and apply most policy there. Configure the physical members with only the settings required for membership.

```cisco
interface range GigabitEthernet1/0/1 - 2
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 channel-group 1 mode active
 no shutdown

interface Port-channel1
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
```

For a Layer 3 bundle:

```cisco
interface range TenGigabitEthernet1/1/1 - 2
 no switchport
 channel-group 10 mode active
 no shutdown

interface Port-channel10
 no switchport
 ip address 10.10.10.1 255.255.255.252
```

For a static bundle:

```cisco
interface range GigabitEthernet1/0/3 - 4
 switchport
 switchport mode access
 switchport access vlan 100
 channel-group 2 mode on

interface Port-channel2
 switchport
 switchport mode access
 switchport access vlan 100
```

**How negotiation fails**

Failure usually happens in one of three ways: the bundle never forms, one or more links are excluded, or the port-channel forms but traffic behaves unexpectedly. The physical link can still show `up/up` while the logical bundle is incomplete.

- **Protocol mismatch**
  - One side uses PAgP and the other uses LACP.
  - One side uses negotiation and the other uses static `on`.
- **Mode mismatch**
  - Both sides are passive (`auto/auto` for PAgP or `passive/passive` for LACP).
  - One side is configured to force a bundle while the other expects negotiation.
- **Parameter mismatch**
  - Trunk/access mismatch.
  - VLAN mismatch.
  - Speed/duplex mismatch.
  - Layer 2 versus Layer 3 mismatch.
- **Feature mismatch**
  - Some features are not compatible with bundling on a given platform.
  - The member may be excluded rather than fully joined.

In many cases, the interface is not err-disabled just because of EtherChannel mismatch. More commonly, the port is placed in a suspended, individual, or standby-like state depending on the platform and protocol. Err-disable is usually associated with separate protection mechanisms such as BPDU Guard, UDLD, port security, or other fault conditions.

**LACP behavior and selection logic**

LACP uses more than just “system information” and “port information.” It also relies on keys and state bits to decide whether links can aggregate. The actor and partner exchange information about identity, priority, key, and state so both sides can determine compatibility.

- **Actor and partner data**
  - System ID and system priority identify the device.
  - Port ID and port priority identify the member link.
  - Administrative key and operational key help determine which links belong together.
  - State bits indicate activity, synchronization, collecting, distributing, and timeout behavior.
- **Selection outcome**
  - Links with matching keys and compatible state can join the same aggregator.
  - Links with different keys or incompatible parameters remain out of the bundle.
- **Practical implication**
  - LACP can choose a subset of links if only some members are eligible.
  - A link may be physically up but not forwarding in the bundle if it is not synchronized.

**Useful verification commands**

Start with the port-channel summary, then inspect the logical interface and the members. The summary is usually the fastest way to see whether the bundle formed and which protocol is in use.

```cisco
show etherchannel summary
show interfaces port-channel 1
show interfaces trunk
show lacp neighbor
show pagp neighbor
show interfaces status
show running-config interface port-channel 1
show running-config interface gigabitEthernet1/0/1
```

Typical `show etherchannel summary` output uses flags to show the bundle state and protocol. The exact letters vary by platform, but the general meaning is consistent.

- **Common summary clues**
  - `SU`: Layer 2 port-channel in use.
  - `RU`: Layer 3 port-channel in use.
  - `P`: bundled in the port-channel.
  - `I`: stand-alone or not bundled.
  - `s`: suspended.
  - `D`: down.
- **Interpretation tip**
  - A member can be physically up but not marked `P`.
  - If the port is not bundled, inspect the mismatch reason and compare both ends.

Example:

```cisco
Switch# show etherchannel summary
Group  Port-channel  Protocol    Ports
------+-------------+-----------+-----------------------------------------------
1      Po1(SU)         LACP      Gi1/0/1(P) Gi1/0/2(P)
2      Po2(SU)         PAgP      Gi1/0/3(P) Gi1/0/4(P)
10     Po10(RU)        LACP      Te1/1/1(P) Te1/1/2(P)
```

This output shows three healthy bundles: one LACP Layer 2 bundle, one PAgP Layer 2 bundle, and one LACP Layer 3 bundle.

**Common failure modes and what they mean**

Troubleshooting is easier when you map symptoms to likely causes.

- **No bundle forms**
  - Check whether both sides use the same protocol.
  - Verify that one side is not `auto/passive` while the other side is also passive.
  - Confirm that the interfaces are not administratively shut.
- **Only some links bundle**
  - Compare speed, duplex, VLAN, trunking, and MTU.
  - Check whether one member has a different key or different physical capability.
- **Port-channel comes up but traffic fails**
  - Verify allowed VLANs and native VLAN on trunks.
  - Check Layer 3 addressing and routing adjacency on routed bundles.
  - Confirm that the upstream/downstream device is using the same hashing and bundle policy.
- **One link is excluded**
  - Look for configuration drift on the physical interface.
  - Check for feature incompatibility such as port security or STP-related settings.
- **Traffic imbalance**
  - EtherChannel load balancing is per-flow, not per-packet.
  - A single large flow usually uses one physical link.
  - Uneven distribution is normal if the traffic mix is small or asymmetric.

**Load balancing and hashing**

EtherChannel load balancing uses a hash to choose the member link for each flow. The hash is based on selected header fields such as source and destination MAC addresses, IP addresses, or Layer 4 ports, depending on platform and configuration.

- **Why it matters**
  - The bundle increases aggregate throughput, but one conversation still follows one physical link.
  - Poor hash diversity can create hot spots.
- **Typical methods**
  - Source MAC.
  - Destination MAC.
  - Source and destination MAC.
  - Source IP.
  - Destination IP.
  - Source and destination IP.
  - Layer 4 source and destination ports on supported platforms.
- **Verification**
  - Use the platform-supported load-balance display command.
  - `show etherchannel load-balance` is the common baseline command.
  - Exact hash-detail commands vary by platform, so verify locally before relying on them.

Example platform configuration may look like this, but accepted keywords vary by IOS XE release and hardware:

```cisco
port-channel load-balance src-dst-ip
```

Use the exact method supported by your platform. If the keyword is rejected, consult the device help output with `?` or the platform documentation.

**Troubleshooting workflow**

1. Verify physical link state with `show interfaces status` and `show interfaces`.
2. Check bundle membership with `show etherchannel summary`.
3. Compare both ends for protocol, mode, and interface settings.
4. Inspect the port-channel and member interface configuration.
5. Verify trunking, VLANs, or Layer 3 addressing as appropriate.
6. Review logs for suspension, protocol mismatch, or protection events.
7. Test with a minimal configuration if the bundle still fails.

**Practical mismatch examples**

A PAgP mismatch:

```cisco
! Side A
interface range Gi1/0/1 - 2
 channel-group 1 mode desirable

! Side B
interface range Gi1/0/1 - 2
 channel-group 1 mode auto
```

This forms because `desirable` actively negotiates with `auto`.

A non-forming PAgP pair:

```cisco
! Side A
interface range Gi1/0/1 - 2
 channel-group 1 mode auto

! Side B
interface range Gi1/0/1 - 2
 channel-group 1 mode auto
```

This does not form because neither side initiates negotiation.

A LACP mismatch:

```cisco
! Side A
interface range Gi1/0/1 - 2
 channel-group 1 mode active

! Side B
interface range Gi1/0/1 - 2
 channel-group 1 mode passive
```

This forms because `active` initiates LACP.

A Layer 2 versus Layer 3 mismatch:

```cisco
! Side A
interface range Gi1/0/1 - 2
 switchport
 channel-group 1 mode active

! Side B
interface range Gi1/0/1 - 2
 no switchport
 channel-group 1 mode active
```

This does not form correctly because the member roles do not match.

**Exam-focused takeaways**

- **PAgP** uses `desirable`, `auto`, and `on`.
- **LACP** uses `active`, `passive`, and `on`.
- `auto/auto` and `passive/passive` do not initiate a bundle.
- `desirable/auto` and `active/passive` can form.
- EtherChannel requires matching operational characteristics, not just physical link status.
- Most mismatches suspend or exclude links rather than immediately err-disabling them.
- Use `show etherchannel summary` first, then compare both ends and inspect the port-channel configuration.
