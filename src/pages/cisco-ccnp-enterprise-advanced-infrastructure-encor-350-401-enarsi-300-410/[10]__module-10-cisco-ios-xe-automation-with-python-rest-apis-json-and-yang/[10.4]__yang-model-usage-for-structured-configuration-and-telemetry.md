## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**YANG fundamentals**

YANG is a data modeling language used to describe how network configuration and operational data are structured. It does not move data by itself; instead, it defines the schema that management protocols use to read, validate, and modify device data.

A YANG model gives you:
- **Structure** for configuration and state data
- **Constraints** that validate values before they are accepted
- **Semantics** that make vendor and open models predictable
- **Interoperability** across NETCONF, RESTCONF, gNMI, and telemetry pipelines

YANG models commonly describe:
- **Configuration data** such as interface settings, routing policy, and system services
- **Operational state** such as counters, link status, and protocol adjacency information
- **Notifications** for event-driven updates
- **RPCs** (remote procedure calls) for device actions such as clear, reset, or operational requests

**Core YANG building blocks**

YANG organizes data into containers, lists, leaves, and leaf-lists. These objects map to hierarchical paths that management protocols expose.

- **container** groups related nodes
- **list** represents repeated entries keyed by one or more values
- **leaf** stores a single value
- **leaf-list** stores multiple values of the same type
- **choice/case** models mutually exclusive options
- **augment** extends an existing model without rewriting it
- **grouping/use** promotes reuse of common structures

A simple conceptual example looks like this:

```yang
module example-if {
  namespace "http://example.com/if";
  prefix exif;

  container interfaces {
    list interface {
      key "name";
      leaf name {
        type string;
      }
      leaf description {
        type string;
      }
      leaf enabled {
        type boolean;
        default "true";
      }
    }
  }
}
```

This model says that `interfaces/interface` is a list keyed by `name`, and each entry can carry a description and an enabled flag.

**YANG constraints and validation**

YANG is valuable because it validates data before the device accepts it. This reduces malformed configuration and makes automation safer.

Important constraint and validation concepts:
- **must** enforces an XPath-based rule that must evaluate to true
- **when** makes a node valid only under a specific condition
- **presence** indicates that an empty container has meaning and is not just a grouping node
- **mandatory** requires a leaf or leaf-list entry to be present
- **range** limits numeric values
- **length** limits string or binary size
- **pattern** constrains string format with regular expressions
- **default** supplies a value when the client omits the leaf

Practical implications:
- A `mandatory` leaf must be supplied during configuration.
- A `default` leaf may not appear in the payload, but the device still treats it as present logically.
- A `must` or `when` violation usually causes validation failure before commit or apply.
- `range`, `length`, and `pattern` reduce bad input at the protocol boundary.

Example of common constraint usage:

```yang
leaf mtu {
  type uint16 {
    range "576..9216";
  }
  default "1500";
}

leaf description {
  type string {
    length "0..240";
    pattern '[A-Za-z0-9 _.-]*';
  }
}
```

**Datastores and transaction behavior**

Structured configuration depends on the device datastore model. In NETCONF and related workflows, the datastore determines where data is read from and written to.

Common datastore concepts:
- **running** is the active configuration on the device
- **startup** is the configuration loaded at boot on platforms that use it
- **candidate** is a staging datastore used on devices that support edit-then-commit workflows
- **operational** is read-only state data, not a configuration target

On Cisco IOS XE, the most common operational behavior is direct interaction with the running configuration model. Candidate datastore support is platform and feature dependent, so do not assume every device supports the full NETCONF candidate workflow.

Transaction concepts to know:
- **edit-config** stages or applies configuration changes to a target datastore
- **validate** checks whether the candidate or proposed configuration is acceptable
- **commit** finalizes staged changes only when the platform supports a candidate-style workflow
- **confirmed-commit** allows a rollback timer if you do not confirm the change in time
- **rollback** restores a previous known-good state when supported by the platform or workflow

> Do not assume `commit` is universally available on Cisco IOS XE. Always verify the device’s NETCONF capabilities and supported datastores.

A practical mental model:
1. You retrieve the current configuration.
2. You edit the intended subtree.
3. The device validates syntax, schema, and constraints.
4. The device applies or stages the change.
5. You verify the resulting configuration and operational state.

**NETCONF for structured configuration**

NETCONF is an XML-based protocol for network configuration and state retrieval. It uses YANG models to define the data it manipulates.

Key NETCONF operations:
- **`<hello>`** exchanges capabilities and establishes session support
- **`<get>`** retrieves configuration and state data
- **`<get-config>`** retrieves configuration data only
- **`<edit-config>`** modifies configuration data
- **`<copy-config>`** copies one datastore to another
- **`<delete-config>`** removes a datastore where supported
- **`<lock>` / `<unlock>`** control datastore access
- **`<validate>`** checks candidate or proposed configuration
- **`<commit>`** applies staged changes on platforms that support it

Capability exchange matters because it tells you which models and features the device supports. A NETCONF client should inspect the server capabilities before attempting advanced workflows.

Example capability discovery from a NETCONF hello:

```xml
<hello xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <capabilities>
    <capability>urn:ietf:params:netconf:base:1.0</capability>
    <capability>urn:ietf:params:netconf:capability:writable-running:1.0</capability>
    <capability>urn:ietf:params:netconf:capability:validate:1.1</capability>
    <capability>urn:ietf:params:netconf:capability:xpath:1.0</capability>
  </capabilities>
</hello>
```

Example `get-config` request for interface configuration:

```xml
<rpc message-id="101" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get-config>
    <source>
      <running/>
    </source>
    <filter type="subtree">
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces"/>
    </filter>
  </get-config>
</rpc>
```

Example `edit-config` payload for a single interface entry:

```xml
<rpc message-id="102" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <edit-config>
    <target>
      <running/>
    </target>
    <config>
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
        <interface>
          <name>GigabitEthernet1</name>
          <description>Uplink to core</description>
          <enabled>true</enabled>
        </interface>
      </interfaces>
    </config>
  </edit-config>
</rpc>
```

In real deployments, the exact required leaves depend on the model and platform. For example, some interface types require additional leaves such as `type` or platform-specific augmentations.

**RESTCONF for REST-style access to YANG data**

RESTCONF exposes YANG-modeled data over HTTP using standard REST methods. It is useful when you want a simpler API style than NETCONF XML while still using the same underlying schema.

Common RESTCONF methods:
- **GET** reads data
- **POST** creates a new resource or invokes an RPC/action
- **PUT** replaces a resource
- **PATCH** partially updates a resource
- **DELETE** removes a resource

Resource path conventions:
- Paths map to YANG schema nodes
- List keys appear in the URI
- JSON payloads usually mirror the YANG tree
- Media types often use `application/yang-data+json` or `application/yang-data+xml`

Example GET for a specific interface entry:

```bash
curl -k -u admin:password \
  -H "Accept: application/yang-data+json" \
  https://router.example.com/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1
```

A typical response for a single list entry is an object, not an array:

```json
{
  "ietf-interfaces:interface": {
    "name": "GigabitEthernet1",
    "description": "Uplink to core",
    "enabled": true
  }
}
```

Example PATCH to update one leaf on that entry:

```bash
curl -k -u admin:password \
  -X PATCH \
  -H "Content-Type: application/yang-data+json" \
  -H "Accept: application/yang-data+json" \
  https://router.example.com/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1 \
  -d '{
    "ietf-interfaces:description": "Uplink to core - updated"
  }'
```

RESTCONF is especially useful when:
- You want HTTP-native integration
- You need lightweight automation from web tooling
- You are working with JSON-based workflows
- You want direct access to YANG data without XML parsing

**gNMI and model-driven telemetry**

gNMI, or gRPC Network Management Interface, is a protocol commonly used for model-driven telemetry and OpenConfig-based data access. It uses gRPC transport and Protobuf encoding.

gNMI is often used for:
- **Subscribe** operations for streaming telemetry
- **Get** operations for retrieving model data
- **Set** operations for configuration on platforms that support it, though configuration support is more limited and platform dependent than telemetry support

Telemetry modes to know:
- **ONCE** retrieves a snapshot
- **POLL** retrieves data repeatedly on request
- **STREAM** pushes updates continuously or periodically

Common subscription styles:
- **target-defined** lets the device determine update behavior
- **sample** sends periodic samples
- **on-change** sends updates when values change
- **suppress-redundant** reduces duplicate updates when supported

A conceptual gNMI subscribe request for interface counters might look like this:

```json
{
  "subscribe": {
    "subscription": [
      {
        "path": {
          "elem": [
            { "name": "interfaces" },
            { "name": "interface", "key": { "name": "GigabitEthernet1" } },
            { "name": "state" }
          ]
        },
        "mode": "SAMPLE",
        "sample_interval": 10000000000
      }
    ],
    "mode": "STREAM"
  }
}
```

The exact path syntax depends on the client library and schema. In OpenConfig, interface state and counters are usually modeled under the interface entry and its `state` subtree, with counters as leaves or nested containers depending on the model version.

Practical guidance:
- Use telemetry for high-frequency operational data.
- Use configuration APIs for intended state changes.
- Prefer state paths for counters, status, and live measurements.
- Prefer config paths for intended settings such as descriptions, shutdown state, or policy references.

**Model-driven telemetry versus traditional polling**

Model-driven telemetry reduces the inefficiency of repeated CLI scraping. Instead of polling text output, you subscribe to structured data and receive updates in a machine-readable format.

Why it matters:
- **Lower overhead** than frequent CLI polling
- **Better precision** because you subscribe to specific YANG paths
- **Faster detection** of changes and faults
- **Cleaner integration** with collectors, time-series databases, and analytics platforms

Typical telemetry pipeline:
1. The device exports YANG-modeled data.
2. A collector subscribes to the desired path.
3. The device streams updates over gRPC or another supported transport.
4. The collector stores, visualizes, or correlates the data.

Common encodings and transports:
- **gRPC** is the transport used by gNMI and many telemetry implementations
- **GPB** (Google Protocol Buffers) is compact and efficient
- **JSON_IETF** is human-readable and schema-aligned
- **XML** may appear in NETCONF-based workflows
- **YANG-Push** is a standards-based subscription mechanism often associated with NETCONF/RESTCONF ecosystems

**Cisco IOS XE telemetry considerations**

Cisco IOS XE support for telemetry is release- and platform-dependent. In exam study, focus on the concepts rather than memorizing one exact CLI syntax for every release.

What to remember:
- Telemetry often exports **operational state**, not configuration intent.
- Subscription syntax and show commands vary by release.
- Some platforms support native YANG-Push, others emphasize gNMI or platform-specific telemetry features.
- Collector configuration, destination setup, and encoding choices depend on the device version and feature set.

A practical workflow on IOS XE is usually:
- Enable the management protocol or telemetry feature required by the platform.
- Define the subscription or sensor path.
- Configure the destination collector and transport parameters.
- Verify session status, counters, and received updates on the collector side.

Because syntax varies, validate against the device release documentation or schema discovery tools before deploying.

**Schema discovery and model exploration**

Before automating against a device, you need to know which models and revisions it supports.

Useful discovery methods:
- **NETCONF hello** capabilities reveal supported protocol features and sometimes model hints
- **YANG library** exposes the set of implemented modules and revisions
- **RESTCONF root and schema resources** can help identify available data trees
- **Cisco YANG Suite** helps browse models, test payloads, and generate requests
- **pyang** validates and inspects YANG modules offline

Example workflow with `pyang`:
```bash
pyang -f tree ietf-interfaces.yang
pyang -f yin ietf-interfaces.yang
```

This helps you understand:
- Which leaves are mandatory
- Which containers are presence containers
- Which augmentations a vendor adds
- How list keys and paths are formed

**Operational state versus configuration**

YANG models often separate intended configuration from live state. This distinction is critical for telemetry and automation.

- **Configuration data** represents what you want the device to do
- **State data** represents what the device is currently doing
- **Operational data** may include counters, status, and runtime values
- **Config containers** are usually writable
- **State containers** are usually read-only

This separation matters because:
- Telemetry usually targets state data.
- RESTCONF and NETCONF can read both config and state, depending on the operation and model.
- Automation should not assume a state leaf can be written just because it appears in the same subtree as configuration.

**Practical workflow examples**

1. **Inspect the model**
   - Identify the module name, revision, and supported leaves.
   - Confirm whether the node is config or state.
   - Check constraints such as `mandatory`, `must`, and `when`.

2. **Read current data**
   - Use NETCONF `get` or RESTCONF `GET`.
   - Narrow the query to the subtree you need.
   - Verify list keys and current values.

3. **Apply configuration**
   - Use `edit-config`, RESTCONF `PUT`/`PATCH`, or a supported automation library.
   - Send only the subtree you intend to change.
   - Validate the response and any schema errors.

4. **Verify operational impact**
   - Read state data or telemetry output.
   - Confirm the device applied the intended change.
   - Check counters, adjacency state, or interface status as appropriate.

**Common exam distinctions**

- **NETCONF** is XML-based and transaction-oriented.
- **RESTCONF** is HTTP-based and resource-oriented.
- **gNMI** is gRPC-based and commonly used for telemetry.
- **YANG** defines the schema; it is not the transport.
- **State data** is read-only in normal operation.
- **Config data** is the intended device configuration.
- **Validation** happens against the YANG schema and device rules, not just syntax.

**Troubleshooting tips**

When a structured configuration or telemetry workflow fails, check:
- Whether the device supports the required protocol capability
- Whether the YANG model revision matches the client expectation
- Whether the path is correct for the schema and list keys
- Whether a `must`, `when`, `range`, or `mandatory` rule is being violated
- Whether the payload matches the resource type being updated
- Whether the target is config or state data
- Whether the transport, encoding, and authentication settings are correct

A useful troubleshooting mindset is to separate the problem into:
- **Schema issue**: wrong path, wrong leaf, wrong model revision
- **Protocol issue**: unsupported method, capability mismatch, transport failure
- **Data issue**: invalid value, missing mandatory leaf, constraint violation
- **Platform issue**: feature not enabled, release limitation, unsupported datastore

**Key takeaways**

- YANG defines the structure and rules for network data.
- NETCONF and RESTCONF use YANG for structured configuration and retrieval.
- gNMI is widely used for model-driven telemetry and OpenConfig access.
- Constraints such as `must`, `when`, `mandatory`, `range`, `length`, `pattern`, and `default` are central to validation.
- Datastore behavior matters, especially the difference between `running`, `startup`, `candidate`, and operational state.
- Telemetry usually focuses on state data, while configuration APIs change intended device behavior.
- Always verify model support, path syntax, and platform-specific capabilities before automating.
