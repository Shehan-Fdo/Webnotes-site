## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**JSON in network automation**

<abbr title="JavaScript Object Notation">JSON</abbr> is a lightweight data format used by APIs, automation tools, and controllers to exchange structured data. In enterprise networking, you commonly parse JSON responses from REST APIs, then construct JSON payloads to create, update, or query resources.

JSON is preferred because it is:
- Human-readable and machine-friendly.
- Easy to map to dictionaries, lists, and objects in most programming languages.
- Common in Cisco controller APIs, telemetry systems, and automation frameworks.

A typical workflow is:
1. Send a request to an API endpoint.
2. Receive a JSON response.
3. Parse the response into usable fields.
4. Build a new JSON payload for the next request.

**JSON structure fundamentals**

JSON uses key-value pairs, arrays, and nested objects.

```json
{
  "device": {
    "hostname": "edge-01",
    "mgmt_ip": "10.10.10.11",
    "roles": ["router", "wan-edge"],
    "enabled": true,
    "interfaces": [
      {
        "name": "GigabitEthernet0/0",
        "description": "Uplink",
        "status": "up"
      }
    ]
  }
}
```

Key rules to remember:
- Keys and string values use double quotes.
- Numbers, booleans, and `null` do not use quotes.
- Arrays preserve order and can contain objects or primitive values.
- Objects are unordered collections of key-value pairs.
- Trailing commas are invalid in strict JSON.

> JSON is not the same as Python dictionaries or YAML, even though they look similar. Always validate syntax before sending payloads to an API.

**Parsing JSON responses**

Parsing means converting a JSON string into a native data structure your tool can work with. In Python, that is usually a dictionary or list. In automation workflows, parsing lets you extract fields such as device IDs, interface states, or token values.

Example with Python:

```python
import json

response_text = '''
{
  "token": "abc123",
  "device": {
    "hostname": "edge-01",
    "status": "active"
  }
}
'''

data = json.loads(response_text)
print(data["token"])
print(data["device"]["hostname"])
```

Common parsing patterns:
- Use `json.loads()` for a JSON string.
- Use `json.load()` for a file object.
- Use `response.json()` in HTTP client libraries such as `requests`.
- Access nested values with chained keys or safe lookup methods.

Example with `requests`:

```python
import requests

resp = requests.get("https://api.example.com/devices/1", headers={"Accept": "application/json"})
payload = resp.json()

print(payload["id"])
print(payload["name"])
```

When parsing API output, check for:
- Missing keys.
- Unexpected data types.
- Empty arrays.
- Error objects returned instead of the expected resource.

**Constructing JSON payloads**

Payload construction is the process of building a JSON document to send in an API request body. This is common for `POST`, `PUT`, and `PATCH` operations.

A payload should match the API schema exactly. If the API expects nested objects, arrays, or specific field names, your payload must follow that structure.

Example payload for creating a device record:

```json
{
  "name": "edge-02",
  "site": "dc1",
  "management": {
    "ip": "10.10.10.12",
    "mask": "255.255.255.0",
    "gateway": "10.10.10.1"
  },
  "tags": ["branch", "wan"],
  "enabled": true
}
```

Practical construction tips:
- Build payloads from native data structures, then serialize them to JSON.
- Keep field names consistent with the API documentation.
- Use `null` when the API explicitly allows an empty value.
- Omit fields the API treats as optional unless they are required for defaults or validation.

Python example:

```python
import json

payload = {
    "name": "edge-02",
    "site": "dc1",
    "enabled": True,
    "interfaces": [
        {
            "name": "GigabitEthernet0/1",
            "description": "LAN",
            "enabled": True
        }
    ]
}

json_body = json.dumps(payload, indent=2)
print(json_body)
```

**Serialization and deserialization**

<dl>
<dt><strong>Deserialization</strong></dt>
<dd>Converting JSON text into a native object such as a dictionary or list.</dd>
<dt><strong>Serialization</strong></dt>
<dd>Converting a native object into JSON text for transport over HTTP or storage.</dd>
</dl>

These two operations are the core of JSON-based automation:
- Deserialize API responses before you inspect or transform them.
- Serialize payloads before you send them to an endpoint.

In Python:
- `json.loads()` deserializes a string.
- `json.dumps()` serializes an object.
- `json.dump()` writes JSON to a file.
- `json.load()` reads JSON from a file.

**API request examples**

Most REST APIs expect JSON in the request body and `Content-Type: application/json` in the headers.

```bash
curl -X POST "https://api.example.com/devices" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -d '{
    "name": "edge-03",
    "site": "dc2",
    "enabled": true
  }'
```

For `PATCH`, send only the fields you want to change if the API supports partial updates:

```bash
curl -X PATCH "https://api.example.com/devices/42" \
  -H "Content-Type: application/json" \
  -d '{
    "enabled": false
  }'
```

For `PUT`, many APIs expect the full resource representation, not just changed fields. Always confirm the API behavior.

**Validation and troubleshooting**

JSON errors often come from syntax mistakes or schema mismatches rather than transport issues.

Common problems:
- Single quotes instead of double quotes.
- Missing commas between fields.
- Extra trailing commas.
- Incorrect nesting level.
- Wrong data type, such as sending a string where the API expects a boolean.
- Encoding issues when reading files or command output.

Useful validation commands:
```bash
jq . payload.json
python -m json.tool payload.json
```

These tools help you confirm that the JSON is syntactically valid before you send it.

> If the API returns `400 Bad Request`, inspect both the JSON syntax and the field names. A valid JSON document can still fail schema validation.

**Working with `jq`**

`jq` is a command-line JSON processor that is useful for filtering, formatting, and extracting values from API responses.

Examples:

```bash
cat response.json | jq .
cat response.json | jq '.device.hostname'
cat response.json | jq '.interfaces[] | select(.status == "down")'
```

Use `jq` when you need to:
- Pretty-print large responses.
- Extract a single field for scripting.
- Filter arrays based on conditions.
- Transform JSON into a smaller structure.

**Python patterns for automation**

When building automation, create payloads from variables rather than hardcoding JSON strings. This reduces quoting errors and makes templates easier to maintain.

```python
import json

hostname = "edge-04"
site = "branch1"

payload = {
    "name": hostname,
    "site": site,
    "enabled": True
}

print(json.dumps(payload))
```

For safer parsing, use defensive checks:

```python
data = resp.json()

device = data.get("device", {})
hostname = device.get("hostname", "unknown")
status = device.get("status", "unknown")
```

This approach helps when APIs return partial data or optional fields.

**Real-world network automation use cases**

JSON parsing and payload construction appear in many enterprise workflows:
- Reading device inventory from a controller API.
- Creating VLANs, interfaces, or policy objects through REST calls.
- Extracting authentication tokens from login responses.
- Processing telemetry or event data from streaming systems.
- Updating configuration state in intent-based networking platforms.

A common pattern is:
- Query the controller for current state.
- Parse the response.
- Compare desired state to actual state.
- Build a JSON payload only for the differences.
- Send the update request.

**Best practices**

- Validate JSON before sending it.
- Use native language objects, not raw string concatenation.
- Match the API schema exactly.
- Keep payloads minimal when the API supports partial updates.
- Log requests and responses during troubleshooting, but avoid exposing secrets.
- Treat tokens, passwords, and API keys as sensitive data.
- Use consistent formatting and indentation for readability during development.

**Quick comparison**

| Task | Purpose | Common tools |
|---|---|---|
| Parse JSON | Convert text into usable data | `json.loads()`, `response.json()`, `jq` |
| Build payload | Create request body | Python dicts, templates, `json.dumps()` |
| Validate syntax | Catch formatting errors | `jq`, `python -m json.tool` |
| Send API request | Deliver JSON to endpoint | `curl`, Python `requests` |

**Exam-focused takeaways**

- JSON is the standard format for many network APIs.
- Parsing converts response text into data you can inspect and automate against.
- Payload construction creates the request body for API operations.
- `Content-Type: application/json` tells the server how to interpret the body.
- `jq` and Python JSON tools help you validate and manipulate data quickly.
- Syntax correctness and schema correctness are both required for successful API calls.
