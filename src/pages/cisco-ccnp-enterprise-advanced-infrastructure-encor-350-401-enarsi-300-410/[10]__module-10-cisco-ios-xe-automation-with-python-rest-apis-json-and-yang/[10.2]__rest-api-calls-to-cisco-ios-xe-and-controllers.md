## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**REST API fundamentals**

A <abbr title="Representational State Transfer">REST</abbr> API uses standard HTTP methods to create, read, update, and delete resources. In Cisco automation, you use REST APIs to interact with device configuration, operational state, and controller-managed services without logging into each device manually.

The key idea is resource orientation:

- A **resource** is an addressable object such as an interface, VLAN, wireless client, or network device.
- An **HTTP method** expresses intent:
  - `GET` reads data.
  - `POST` creates a resource or triggers an action.
  - `PUT` replaces a resource.
  - `PATCH` modifies part of a resource.
  - `DELETE` removes a resource.
- A **payload** is usually JSON and carries the data you want to send.
- A **status code** tells you whether the request succeeded, failed, or needs more action.

REST is common in Cisco controller platforms because controllers centralize policy and inventory. IOS XE also exposes RESTCONF, which maps REST operations to YANG data models on the device.

**RESTCONF on Cisco IOS XE**

RESTCONF is the RESTful protocol defined for accessing <abbr title="Yet Another Next Generation">YANG</abbr> data models. On IOS XE, it is the main REST interface for model-driven configuration and operational data.

Before you can use RESTCONF, the device must actually serve the API. Typical prerequisites include:

- Enable HTTPS server support.
- Enable RESTCONF.
- Create a local user or AAA-backed user with sufficient privilege.
- Ensure the device has a trustable certificate for secure access, especially in production.
- Confirm the relevant YANG models are supported on the platform and software release.

A common IOS XE setup looks like this:

```cisco
conf t
ip domain-name example.local
username apiuser privilege 15 secret Str0ngP@ssw0rd
crypto key generate rsa modulus 2048
ip http secure-server
restconf
aaa new-model
aaa authentication login default local
aaa authorization exec default local
end
write memory
```

In practice, you often also verify the HTTPS service and RESTCONF listener:

```cisco
show ip http server status
show running-config | include restconf|ip http secure-server|username|aaa
show platform software yang-management process
```

> RESTCONF access is not just “HTTPS is on.” If the API service, YANG management, or authorization is missing, the endpoint may be unreachable even though the web server responds.

**Authentication and authorization on IOS XE**

For IOS XE RESTCONF, you usually authenticate with HTTP Basic authentication over HTTPS. The username must be authorized to access the model-driven interface and the requested data.

Important points:

- **Privilege level** matters on many IOS XE deployments; `privilege 15` is a common lab and exam-safe choice.
- **AAA** can restrict access even when the username exists.
- **HTTPS** is the transport expected for secure RESTCONF use.
- **Certificates** matter because many clients reject self-signed certificates unless you explicitly trust them.

A typical request uses Basic auth:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces
```

Use `-k` only for lab testing when you intentionally ignore certificate validation. In production, install and trust the correct CA chain.

**YANG data model structure**

RESTCONF paths map to YANG modules and data nodes. You need to understand the path syntax to avoid malformed requests.

Key concepts:

- A **module** is identified by its namespace prefix, such as `ietf-interfaces`.
- A **container** groups related data.
- A **list** contains keyed entries.
- An **instance identifier** points to a specific list entry using key values.

Example path patterns:

- Collection: `/restconf/data/ietf-interfaces:interfaces`
- Specific interface: `/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1`
- Nested data: `/restconf/data/<module>:<container>/<list>=<key>/<leaf>`

For list keys, the exact key syntax depends on the YANG model. Some lists use a single key, others use multiple keys. Always check the model or device schema when building the URI.

**Reading configuration and operational data**

`GET` is the most common RESTCONF operation. It retrieves configuration or state data exposed by the model.

Example: list interfaces

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -H "Accept: application/yang-data+json" \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces
```

Example: read one interface

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -H "Accept: application/yang-data+json" \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1
```

Operational state is model-dependent. In many modern YANG models, state leaves are exposed under the same logical tree as configuration, often with separate `config false` leaves rather than a separate legacy `interfaces-state` container. Do not assume a single universal path across all releases.

Useful query options often include:

- `content=config` to return configuration only.
- `content=nonconfig` to return operational data only.
- `content=all` to return both where supported.
- `fields=` to limit returned leaves.
- `depth=` to control subtree expansion.

Example with filtering:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -H "Accept: application/yang-data+json" \
  "https://router1.example.local/restconf/data/ietf-interfaces:interfaces?content=config&fields=interface(name,enabled)"
```

**Creating and updating data**

RESTCONF supports both full replacement and partial modification, but the method you choose matters.

- Use **`PUT`** when you want to replace the target resource instance.
- Use **`PATCH`** when you want to change only specific fields.
- Use **`POST`** when you create a new list entry or invoke an action, depending on the model.

A common exam trap is assuming `PATCH` behaves like a generic “send the whole object” update. In RESTCONF, the request body should match the operation and target resource precisely.

Example: create or replace an interface configuration object with `PUT`:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -X PUT \
  -H "Content-Type: application/yang-data+json" \
  -d '{
    "ietf-interfaces:interface": {
      "name": "Loopback100",
      "description": "Mgmt loopback",
      "enabled": true
    }
  }' \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces/interface=Loopback100
```

Example: partial update with `PATCH`:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -X PATCH \
  -H "Content-Type: application/yang-data+json" \
  -d '{
    "ietf-interfaces:interface": {
      "description": "Updated by RESTCONF"
    }
  }' \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces/interface=Loopback100
```

Practical guidance:

- Use `PUT` when you know the full desired object and want deterministic replacement.
- Use `PATCH` when you want to minimize payload size and change only a few leaves.
- Verify the model’s required leaves before sending a `PUT`; missing mandatory fields can cause validation errors.
- Use the exact list key in the URI; the key value must match the instance you intend to modify.

**IPv4 addressing examples**

IP addressing models vary by platform and YANG revision. Some examples use `netmask`, while others use prefix length or platform-specific augmentations. For exam prep, focus on the concept rather than memorizing one universal payload.

A model-driven IPv4 configuration may look like this on one implementation:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -X PATCH \
  -H "Content-Type: application/yang-data+json" \
  -d '{
    "ietf-ip:ipv4": {
      "address": [
        {
          "ip": "192.0.2.10",
          "netmask": "255.255.255.0"
        }
      ]
    }
  }' \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1
```

On other releases or models, you may need a prefix-length or a different container structure. Always validate against the device schema before assuming a payload format.

**RESTCONF RPCs and actions**

Not every RESTCONF call is a CRUD operation. Some models expose **RPCs** (<abbr title="Remote Procedure Calls">RPCs</abbr>) or actions that trigger behavior rather than modify persistent configuration.

Typical use cases:

- Clear counters.
- Trigger diagnostics.
- Request operational actions.
- Execute model-defined workflows.

Example pattern for an RPC call:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -X POST \
  -H "Content-Type: application/yang-data+json" \
  -d '{
    "input": {
      "interface": "GigabitEthernet1"
    }
  }' \
  https://router1.example.local/restconf/operations/ietf-interfaces:clear-interface-counters
```

The exact RPC name and payload depend on the YANG model. In exam scenarios, the important point is that `POST` to `/restconf/operations/...` is used for actions, not for ordinary resource replacement.

**Common RESTCONF response codes and errors**

You should recognize the most common HTTP responses and what they imply.

- `200 OK` means the request succeeded and returned data.
- `201 Created` means a new resource was created.
- `204 No Content` means the request succeeded but returned no body.
- `400 Bad Request` usually means malformed JSON, invalid path syntax, or schema validation failure.
- `401 Unauthorized` means authentication failed.
- `403 Forbidden` means authentication succeeded but authorization failed.
- `404 Not Found` means the resource path does not exist or the model is unavailable.
- `409 Conflict` often indicates a duplicate key or conflicting edit.
- `415 Unsupported Media Type` usually means the `Content-Type` header is wrong.
- `500 Internal Server Error` suggests a server-side issue or unsupported operation.

RESTCONF error bodies are usually JSON and can include details about the failing path, invalid leaf, or schema constraint. Parse the body, not just the status code.

Example error handling with `jq`:

```bash
curl -sk -u apiuser:Str0ngP@ssw0rd \
  -H "Accept: application/yang-data+json" \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces/interface=BadName \
  | jq
```

**Discovering supported models and capabilities**

Modern RESTCONF implementations expose YANG library information so clients can discover supported modules and revisions. In current implementations, YANG Library 1.1 is the preferred discovery mechanism, not older `modules-state` examples.

Useful discovery steps:

1. Query the RESTCONF root or API entry point.
2. Inspect the YANG library data.
3. Confirm the module name, revision, and namespace before building payloads.
4. Check whether the model is config, state, or action-capable.

Example discovery request:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -H "Accept: application/yang-data+json" \
  https://router1.example.local/restconf/data/ietf-yang-library:yang-library
```

Depending on the platform, the exact schema path may differ, but the principle is the same: discover the supported model before you automate against it.

**Cisco IOS XE REST API troubleshooting**

When RESTCONF fails, troubleshoot from transport to model.

1. Verify HTTPS is enabled and reachable.
   - Test TCP/443 connectivity.
   - Confirm the device responds to HTTPS.
2. Verify RESTCONF is enabled.
   - Check the running configuration.
   - Confirm the RESTCONF process is active.
3. Verify authentication.
   - Confirm the username exists.
   - Confirm privilege and AAA authorization.
4. Verify certificate trust.
   - Check whether the client rejects the server certificate.
   - Use a trusted CA in production.
5. Verify the YANG model and URI.
   - Confirm the module name and revision.
   - Confirm the list key and path syntax.
6. Verify the payload.
   - Check `Content-Type`.
   - Check required leaves and JSON structure.

Useful CLI checks on IOS XE:

```cisco
show ip http server status
show running-config | section restconf
show running-config | include username|aaa|ip http secure-server
show platform software yang-management process
show logging | include RESTCONF|YANG|HTTP
```

If the device serves HTTPS but RESTCONF requests fail, the issue is often authorization, model availability, or an incorrect URI rather than basic connectivity.

**Controller API patterns**

Cisco controllers usually expose higher-level REST APIs than IOS XE. Instead of manipulating low-level device configuration directly, you interact with inventory, policy, templates, telemetry, or client data.

Common controller characteristics:

- Authentication is often token-based.
- Requests usually use JSON.
- APIs may be versioned.
- Endpoints are often grouped by function, such as inventory, assurance, wireless, or policy.
- Pagination and filtering are common because controller datasets can be large.

**Cisco Catalyst Center API workflow**

Catalyst Center commonly uses a token obtained from an authentication endpoint, then sends that token in `X-Auth-Token` for subsequent calls.

Typical flow:

1. Authenticate and retrieve a token.
2. Store the token securely.
3. Include `X-Auth-Token` in later requests.
4. Query inventory, devices, clients, or tasks.
5. Poll task endpoints when operations are asynchronous.

Example token request:

```bash
curl -k -X POST \
  https://catalyst.example.local/dna/system/api/v1/auth/token \
  -u apiuser:Str0ngP@ssw0rd
```

Example use of the token:

```bash
curl -k \
  -H "X-Auth-Token: <token>" \
  -H "Accept: application/json" \
  https://catalyst.example.local/dna/intent/api/v1/network-device
```

Common Catalyst Center endpoints include:

- `/dna/intent/api/v1/network-device`
- `/dna/intent/api/v1/client-health`
- `/dna/intent/api/v1/task/{taskId}`
- `/dna/intent/api/v1/site`
- `/dna/intent/api/v1/topology/physical-topology`

Practical note:

- Many Catalyst Center write operations are asynchronous.
- You often receive a task ID and must poll the task endpoint until completion.
- A successful submission does not always mean the change is already applied.

Example task polling:

```bash
curl -k \
  -H "X-Auth-Token: <token>" \
  https://catalyst.example.local/dna/intent/api/v1/task/12345
```

**Cisco SD-WAN API workflow**

Current Cisco SD-WAN/vManage automation is typically token-based and API-driven. Older login patterns exist in legacy environments, but for exam prep you should focus on the modern token workflow.

Common concepts:

- Authenticate to obtain a session or API token.
- Use the token in subsequent requests.
- Query device inventory, templates, alarms, and policies.
- Apply changes through template or policy endpoints rather than direct device edits.

Example pattern:

```bash
curl -k -X POST \
  https://sdwan.example.local/dataservice/client/token \
  -u apiuser:Str0ngP@ssw0rd
```

Then use the returned token according to the platform’s documented header or cookie mechanism.

Common SD-WAN API areas include:

- Device inventory.
- Real-time monitoring.
- Templates and feature templates.
- Centralized policy.
- Alarms and events.

Because SD-WAN APIs vary by release, always verify the exact endpoint and authentication method for the version you are targeting.

**Cisco wireless controller API workflow**

Wireless controllers and cloud-managed wireless platforms expose REST APIs for inventory, clients, APs, RF data, and configuration.

Typical use cases:

- Retrieve access point status.
- Query client associations.
- Read RF health or radio metrics.
- Automate site or WLAN-related operations.

Example controller-style request:

```bash
curl -k \
  -H "Authorization: Bearer <token>" \
  -H "Accept: application/json" \
  https://wireless.example.local/api/v1/clients
```

Wireless APIs often use:

- Bearer tokens.
- API keys.
- Session cookies.
- Versioned endpoints.

The exact authentication model depends on the platform, but the operational pattern is similar: authenticate, query resources, handle pagination, and process JSON responses.

**Pagination, filtering, and query parameters**

Controller APIs often return large datasets. To keep requests efficient, use query parameters when supported.

Common patterns:

- `limit` and `offset` for pagination.
- `page` and `pageSize` for page-based retrieval.
- `filter` or `query` for server-side filtering.
- `sort` for ordering results.
- `fields` for selecting only needed attributes.

Example:

```bash
curl -k \
  -H "X-Auth-Token: <token>" \
  "https://catalyst.example.local/dna/intent/api/v1/network-device?limit=50&offset=0"
```

Why this matters:

- You reduce bandwidth.
- You avoid client-side parsing of huge payloads.
- You can build reliable automation loops for inventory collection.

**RESTCONF versus NETCONF versus controller REST APIs**

These technologies overlap, but they are not interchangeable.

| Technology | Primary use | Data model | Transport | Best fit |
|---|---|---|---|---|
| RESTCONF | Model-driven device access | YANG | HTTPS | IOS XE configuration and state |
| NETCONF | Model-driven device access | YANG | SSH | Structured configuration and RPCs |
| Controller REST API | Platform services and orchestration | Platform-specific JSON | HTTPS | Inventory, policy, telemetry, workflows |

Practical distinctions:

- Use **RESTCONF** when you want REST semantics on YANG data over HTTPS.
- Use **NETCONF** when you need mature model-driven configuration over SSH or when a workflow is already built around NETCONF.
- Use **controller REST APIs** when the controller owns the workflow and the device is not the direct target.

**Exam-oriented troubleshooting mindset**

When a REST call fails, isolate the layer.

- Transport issue:
  - DNS, routing, firewall, TCP/443, certificate trust.
- Authentication issue:
  - Bad credentials, expired token, wrong auth header.
- Authorization issue:
  - Insufficient privilege, AAA restriction, role mismatch.
- Model issue:
  - Wrong YANG module, wrong revision, unsupported leaf.
- Syntax issue:
  - Bad JSON, wrong `Content-Type`, malformed URI.
- Workflow issue:
  - Asynchronous task not complete, wrong endpoint for the platform version.

A good automation workflow is:

1. Discover the API and supported model.
2. Authenticate.
3. Read a small known-good resource.
4. Modify one field.
5. Verify the result.
6. Expand to bulk automation.

**Practical examples of safe API usage**

A simple read-first approach reduces risk:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -H "Accept: application/yang-data+json" \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1
```

Then make a minimal change:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -X PATCH \
  -H "Content-Type: application/yang-data+json" \
  -d '{
    "ietf-interfaces:interface": {
      "description": "Uplink to core"
    }
  }' \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1
```

Then verify:

```bash
curl -k -u apiuser:Str0ngP@ssw0rd \
  -H "Accept: application/yang-data+json" \
  https://router1.example.local/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1
```

This read-modify-verify pattern is safer than pushing large payloads blindly.

**Key takeaways**

- REST APIs let you automate Cisco platforms with standard HTTP methods and JSON payloads.
- IOS XE uses RESTCONF for YANG-modeled device access over HTTPS.
- You must enable the API service, configure authentication, and verify authorization on the device.
- `GET`, `PUT`, `PATCH`, `POST`, and `DELETE` each have distinct meanings; do not treat them as interchangeable.
- Controller APIs such as Catalyst Center, SD-WAN, and wireless platforms are usually token-based and often asynchronous.
- Always confirm the exact endpoint, model revision, and payload structure for the platform and software version you are targeting.
