## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**NETCONF and RESTCONF overview**

NETCONF and RESTCONF are model-driven management protocols used to configure and monitor network devices through YANG data models. They reduce dependence on screen-scraping and CLI parsing by exposing structured data and operations.

- **NETCONF** uses an XML-based RPC model over an SSH session.
- **RESTCONF** uses HTTP methods over HTTPS and maps YANG data to REST-style resources.
- Both rely on **YANG** models to define configuration, state, and operations.
- On Cisco IOS XE, support depends on platform, release, and feature set, so you should always verify capability and model availability before building automation.

A useful mental model is:

1. Discover what the device supports.
2. Read current state or configuration.
3. Modify the target data.
4. Commit or confirm the change.
5. Verify the result and handle errors.

**Core YANG concepts**

YANG is the data modeling language that defines how configuration and operational data are structured. It is not a transport protocol; NETCONF and RESTCONF are the transports that expose YANG-defined data.

<dl>
<dt><strong>Configuration data</strong></dt>
<dd>Persistent settings intended to survive reboots, such as interface descriptions or routing policy.</dd>
<dt><strong>Operational data</strong></dt>
<dd>Live state reported by the device, such as interface counters or protocol neighbors.</dd>
<dt><strong>RPC</strong></dt>
<dd>Remote Procedure Call, a request that triggers an operation rather than reading or writing a datastore directly.</dd>
<dt><strong>Notification</strong></dt>
<dd>An event message emitted by the device, such as a link state change or alarm.</dd>
</dl>

YANG modules are often organized into namespaces and revisions. In practice, you need the exact module name and schema path to address a resource correctly.

**NETCONF workflow**

NETCONF is session-oriented. You open an SSH-based session, exchange capabilities, and then send XML RPCs inside `<rpc>` elements. The server replies with `<rpc-reply>` messages.

Typical NETCONF workflow:

1. Establish an SSH session to TCP port `830`.
2. Exchange capabilities in the `<hello>` message.
3. Read or edit configuration using NETCONF operations.
4. Commit, validate, or discard changes as needed.
5. Close the session cleanly.

A classic NETCONF 1.0 message terminator is `]]>]]>`. NETCONF 1.1 uses chunked framing instead, so do not assume the delimiter is always present.

```xml
<hello>
  <capabilities>
    <capability>urn:ietf:params:netconf:base:1.0</capability>
    <capability>urn:ietf:params:netconf:base:1.1</capability>
  </capabilities>
</hello>
]]>]]>
```

**Capability exchange and discovery**

Capabilities tell you what the server supports, such as base protocol version, writable-running support, candidate datastore support, confirmed commit, and notifications.

Common discovery steps:

- Read the server `<hello>` capabilities.
- Query YANG library data to learn supported modules and revisions.
- Confirm whether the device supports `candidate`, `running`, `startup`, or only a subset.

On many IOS XE platforms, YANG library information is exposed through model-driven interfaces and RESTCONF root resources. The exact discovery path can vary by release, but the goal is the same: identify supported modules before sending edits.

**NETCONF datastores and workflow**

NETCONF commonly works with these datastores:

- **running**: active configuration in use by the device.
- **candidate**: staging area for changes before commit, if supported.
- **startup**: configuration loaded at boot, if supported.

The datastore model matters because it determines whether you can stage changes, validate them, and commit atomically.

**Common NETCONF operations**

- `<get>`: retrieve operational and configuration data.
- `<get-config>`: retrieve configuration from a datastore.
- `<edit-config>`: modify configuration in a target datastore.
- `<copy-config>`: copy one datastore to another.
- `<validate>`: check whether a candidate or proposed configuration is valid.
- `<commit>`: activate staged changes.
- `<discard-changes>`: abandon uncommitted candidate changes.
- `<lock>` and `<unlock>`: protect a datastore from concurrent modification.
- `<close-session>`: end the NETCONF session.

**Locking and change control**

Locking prevents another client from editing the same datastore while you work. This is important in multi-automation environments where two tools might otherwise overwrite each other.

Typical workflow with candidate support:

1. `<lock>` the candidate datastore.
2. `<edit-config>` to stage changes.
3. `<validate>` the candidate if supported.
4. `<commit>` the changes.
5. `<unlock>` the datastore.

If validation fails or you want to abort, use `<discard-changes>` before unlocking.

> A common exam trap is assuming every device supports candidate and confirmed commit. Always verify capabilities first.

**NETCONF edit-config example**

The exact XML path depends on the YANG model. The example below shows the structure of a configuration edit, not a guaranteed IOS XE schema path.

```xml
<rpc message-id="101" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <edit-config>
    <target>
      <running/>
    </target>
    <config>
      <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
        <hostname>EDGE-R1</hostname>
      </native>
    </config>
  </edit-config>
</rpc>
```

**NETCONF validation and rollback behavior**

Validation checks whether the proposed configuration is syntactically and semantically acceptable before commit. This helps catch issues such as invalid interface names, unsupported values, or policy conflicts.

Confirmed commit is useful when you want a safety rollback window. If you do not confirm within the timeout, the device reverts the change automatically. This is valuable for remote maintenance where a bad change could lock you out.

**NETCONF troubleshooting clues**

- If the session opens but no RPC replies arrive, check SSH reachability, port `830`, and authentication.
- If the device rejects an edit, inspect the `<rpc-error>` element for `error-type`, `error-tag`, and `error-message`.
- If a path is wrong, the error often indicates an unknown element or invalid value.
- If a commit fails after validation, the issue may be a dependency or platform constraint rather than a syntax problem.

A practical troubleshooting habit is to compare the requested XML path against the YANG model rather than against the CLI hierarchy.

**RESTCONF workflow**

RESTCONF exposes YANG data through HTTP resources. It is generally treated as stateless request/response over HTTPS, with each request carrying the information needed to act on the target resource.

Typical RESTCONF workflow:

1. Discover the RESTCONF root and supported resources.
2. Inspect the YANG library or module list.
3. Use HTTP methods to read or modify resources.
4. Check response codes and payloads.
5. Verify the resulting configuration or state.

**RESTCONF discovery and root resources**

RESTCONF provides a root entry point that lets clients discover available top-level resources and supporting metadata. Common discovery patterns include:

- Reading the RESTCONF root resource to see top-level data nodes.
- Querying YANG library data to identify supported modules and revisions.
- Using resource paths that match the YANG schema exactly.

On Cisco IOS XE, the RESTCONF root often exposes model-driven data and operational resources under a `/restconf/data` hierarchy, with metadata and schema discovery available through the platform’s supported model set.

**RESTCONF media types and content negotiation**

RESTCONF uses HTTP content negotiation to determine how data is encoded.

Common media types:

- `application/yang-data+json`
- `application/yang-data+xml`

Use the same family for both request and response when possible. If you send JSON, the body must match the YANG structure and the server must accept JSON for that resource.

Example headers:

```bash
Accept: application/yang-data+json
Content-Type: application/yang-data+json
Authorization: Basic <credentials>
```

**RESTCONF methods**

- `GET`: read data.
- `POST`: create a new resource or invoke an operation, depending on the target.
- `PUT`: replace the target resource.
- `PATCH`: partially modify a resource.
- `DELETE`: remove a resource.

The method must match the resource type. For example, `POST` is appropriate for creating a new list entry under a list resource, while `PUT` is used to replace a specific resource at a known URI.

**RESTCONF filtering and query parameters**

RESTCONF supports query parameters that help reduce payload size and focus on relevant data.

Common parameters:

- `content=config|nonconfig|all`
- `fields`
- `depth`
- `filter`

Practical use cases:

- Use `content=config` when you only want configuration data.
- Use `content=nonconfig` when you only want operational state.
- Use `fields` to request only selected leaves.
- Use `depth` to limit nested output.
- Use `filter` to narrow results by subtree or selection criteria, depending on server support.

Example:

```bash
curl -k -u admin:password \
  -H 'Accept: application/yang-data+json' \
  'https://10.0.0.10/restconf/data/Cisco-IOS-XE-native:native/interface?content=config&fields=GigabitEthernet/name,description'
```

**RESTCONF resource modeling example**

For a list entry, target the list resource using the correct key syntax and send a body that matches the YANG list structure. The exact path depends on the module.

Example pattern for a keyed list entry:

```bash
curl -k -u admin:password \
  -X POST \
  -H 'Content-Type: application/yang-data+json' \
  -H 'Accept: application/yang-data+json' \
  https://10.0.0.10/restconf/data/example-module:interfaces \
  -d '{
    "interface": [
      {
        "name": "GigabitEthernet1",
        "description": "Uplink to core"
      }
    ]
  }'
```

For a leaf update, address the leaf or its parent resource according to the schema. A `PUT` replaces the target resource, so use it carefully when the resource is a single leaf or a tightly scoped container.

```bash
curl -k -u admin:password \
  -X PUT \
  -H 'Content-Type: application/yang-data+json' \
  -H 'Accept: application/yang-data+json' \
  https://10.0.0.10/restconf/data/example-module:system/hostname \
  -d '{
    "hostname": "EDGE-R1"
  }'
```

The exact URI and payload shape depend on the YANG model. In IOS XE, native models and OpenConfig models may expose different paths for the same conceptual setting.

**RESTCONF PATCH behavior**

`PATCH` is the preferred method when you want to modify only part of a resource without replacing the entire object. This is useful for incremental changes to a container or list entry.

Use `PATCH` when:

- You want to change one leaf in a larger structure.
- You want to avoid overwriting unrelated fields.
- The server supports partial updates for the target resource.

Be careful: some resources or implementations may have restrictions on patching list entries or nested containers. Always verify the server’s behavior.

**End-to-end interface workflow example**

A realistic automation workflow often looks like this:

1. Discover the device capabilities and supported model.
2. Read the current interface configuration.
3. Stage or apply a description and IP address.
4. Verify the interface state and counters.
5. Roll back or correct the change if validation fails.

NETCONF example for a conceptual interface edit:

```xml
<rpc message-id="201" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <edit-config>
    <target>
      <running/>
    </target>
    <config>
      <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
        <interface>
          <GigabitEthernet>
            <name>1</name>
            <description>Uplink to core</description>
            <ip>
              <address>
                <primary>
                  <address>192.0.2.2</address>
                  <mask>255.255.255.0</mask>
                </primary>
              </address>
            </ip>
          </GigabitEthernet>
        </interface>
      </native>
    </config>
  </edit-config>
</rpc>
```

RESTCONF example for a similar workflow, using a model path that matches the schema:

```bash
curl -k -u admin:password \
  -X PATCH \
  -H 'Content-Type: application/yang-data+json' \
  -H 'Accept: application/yang-data+json' \
  https://10.0.0.10/restconf/data/Cisco-IOS-XE-native:native/interface/GigabitEthernet=1 \
  -d '{
    "description": "Uplink to core"
  }'
```

The key lesson is not the exact syntax of one model, but the workflow: discover, target the correct schema node, modify minimally, and verify.

**NETCONF vs RESTCONF comparison**

| Aspect | NETCONF | RESTCONF |
|---|---|---|
| Transport | SSH | HTTPS |
| Encoding | XML | JSON or XML |
| Interaction style | Session-based RPC | HTTP request/response |
| Best for | Structured configuration workflows, transactional edits | Lightweight API access, web-friendly integration |
| Discovery | `<hello>` capabilities, YANG library | REST root, YANG library, resource discovery |
| Operations | `<get>`, `<edit-config>`, `<commit>`, `<lock>` | `GET`, `POST`, `PUT`, `PATCH`, `DELETE` |
| Transaction support | Stronger workflow control with candidate and commit features | Depends on server behavior and resource model |
| Typical use | Automation systems needing precise control | Applications and scripts that prefer HTTP APIs |

**When to prefer one over the other**

- Use **NETCONF** when you need datastore control, locking, validation, and commit workflows.
- Use **RESTCONF** when you want simpler HTTP integration or easier consumption by general-purpose tools.
- Use **NETCONF** for more explicit transaction handling on devices that support candidate and confirmed commit.
- Use **RESTCONF** when you want quick reads, lightweight writes, or easier integration with web services.

**IOS XE support and platform caveats**

Support is not identical across all IOS XE releases and platforms.

- Some devices support both NETCONF and RESTCONF.
- Some support one better than the other, depending on release and licensing.
- Native YANG models and OpenConfig models may differ in coverage and naming.
- Operational data availability can vary by model and platform.
- A feature visible in the CLI may not yet be exposed in the model-driven interface.

For exam and lab work, verify:

- The device version.
- Whether the required YANG module is present.
- Whether the transport and authentication method are enabled.
- Whether the target resource exists in the model you are using.

**Notifications and subscriptions**

Notifications let the device push event data to a client instead of waiting for a poll. In NETCONF, this is typically associated with notification capability and subscription support. In RESTCONF, event-driven workflows may be available through platform-specific support for subscriptions or event streams.

Use notifications when:

- You need near-real-time event handling.
- Polling would be inefficient.
- You want to react to link, routing, or telemetry events.

Be aware that notification support is implementation-dependent and may not be uniformly available across all IOS XE releases.

**Practical CLI and API examples**

Enable model-driven management on IOS XE using the platform’s supported configuration model and AAA design. Exact commands vary by release, but the general pattern is to enable secure management access and ensure the user has appropriate privilege.

```bash
conf t
username admin privilege 15 secret 0 Cisco123!
ip domain-name example.local
crypto key generate rsa modulus 2048
ip ssh version 2
end
```

A RESTCONF read example:

```bash
curl -k -u admin:Cisco123! \
  -H 'Accept: application/yang-data+json' \
  https://10.0.0.10/restconf/data/Cisco-IOS-XE-native:native/hostname
```

A NETCONF session test with `netconf-console` or a similar client:

```bash
netconf-console --host 10.0.0.10 --port 830 --user admin --password Cisco123!
```

If the device rejects the connection, check:

- SSH reachability to port `830`.
- User privilege and AAA policy.
- TLS/HTTPS readiness for RESTCONF.
- Whether the model-driven feature is enabled on the platform.

**Troubleshooting workflow**

1. Confirm transport reachability.
2. Confirm authentication and authorization.
3. Confirm the correct YANG module and path.
4. Confirm the request body matches the schema.
5. Inspect protocol-specific error messages.
6. Verify the resulting configuration or state.

Common failure patterns:

- `404 Not Found` in RESTCONF often means the URI does not match the schema or the resource is unavailable.
- `400 Bad Request` often means the JSON or XML body is malformed or the payload does not match the YANG model.
- `409 Conflict` can indicate a lock, edit conflict, or unsupported state transition.
- NETCONF `<rpc-error>` messages often reveal whether the issue is syntax, access, or data-model related.

**Exam focus points**

- Know the difference between **transport**, **encoding**, and **data model**.
- Know that NETCONF uses XML RPCs and session framing, while RESTCONF uses HTTP methods and resource paths.
- Know the role of **candidate**, **running**, and **startup** datastores.
- Know the purpose of `<lock>`, `<unlock>`, `<validate>`, `<copy-config>`, `<discard-changes>`, and `<commit>`.
- Know how to discover supported modules and capabilities before sending edits.
- Know that RESTCONF request and response media types matter.
- Know that URI paths must match the YANG schema exactly.
- Know that platform support and model coverage vary across IOS XE versions.

**Quick memory aid**

- **NETCONF**: session, XML, RPC, commit workflow.
- **RESTCONF**: HTTPS, HTTP verbs, resource paths, JSON/XML.
- **YANG**: schema that both protocols expose.
