## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Automation goals in enterprise networks**

Automation in Cisco enterprise environments usually serves three related goals:

- **Configuration push**: apply intended settings consistently across many devices.
- **Validation**: confirm the device state matches the intended state after change.
- **Compliance**: continuously compare actual state to a baseline or policy and report drift.

These are not the same activity. A push changes the device. Validation checks whether the change worked. Compliance checks whether the device still matches the approved standard over time. In practice, you often do all three in one workflow.

A useful mental model is:

1. Define desired state in a source of truth.
2. Push the change using the right interface or controller.
3. Validate operational and configuration state.
4. Record compliance and remediate drift.

**Choosing the right automation method**

Cisco platforms expose multiple automation paths, and the best choice depends on device support, task type, and how much structure you need.

| Method | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| CLI over SSH | Legacy devices, quick one-off tasks, troubleshooting | Universal, familiar, flexible | Harder to validate, parse, and scale safely |
| NETCONF | Structured configuration and state retrieval | Transactional, model-driven, good for validation | Requires YANG support and client tooling |
| RESTCONF | HTTP-based model-driven automation | Easy integration with web tools and APIs | Less common than NETCONF on some platforms |
| Ansible | Repeatable multi-device config and validation | Idempotent, readable, scalable | Depends on modules and inventory quality |
| Controller APIs | Intent-based workflows, assurance, compliance, inventory | Centralized policy and telemetry | Platform-specific, often more opinionated |

Use **CLI** when you need immediate access or the platform lacks richer APIs. Use **NETCONF/RESTCONF** when the device exposes YANG models and you want structured operations. Use **Ansible** when you need repeatable, idempotent orchestration across many devices. Use **controller APIs** when the controller is the source of truth or when the platform already manages policy, templates, and compliance.

> If the task is “make this one device work now,” CLI may be fastest. If the task is “keep 500 devices aligned to policy,” model-driven APIs or Ansible are usually better.

**Source of truth and desired state**

A **source of truth** is the authoritative place where intended configuration lives. It may be:

- A version-controlled repository with templates and variables
- A network inventory system
- A controller database
- A combination of all three

In mature workflows, you do not hand-edit devices as the primary method. Instead, you define desired state in templates, variables, or structured data, then generate and push configuration from that source.

Common source-of-truth patterns:

- **Git-backed desired state**
  - Store templates, variables, and policy files in version control.
  - Review changes through pull requests.
  - Use tags or releases to track approved baselines.
- **Inventory-driven automation**
  - Keep device roles, site data, and interface mappings in an inventory system.
  - Render device-specific configuration from shared templates.
- **Controller-managed intent**
  - Use Cisco Catalyst Center or another controller to define templates, policies, and compliance rules.
  - Let the controller push and monitor state.

**Configuration push methods**

Configuration push can be imperative or declarative.

- **Imperative tasks** describe exact commands to run.
  - Example: “configure this ACL line,” “add this VLAN,” or “set this interface description.”
  - Best for small, targeted changes or when the device state is not fully modeled.
- **Declarative tasks** describe the desired end state.
  - Example: “interface Gi1/0/1 must be access VLAN 20 and have portfast enabled.”
  - Best for repeatability, idempotency, and compliance.

Declarative automation is usually preferred for enterprise operations because it reduces drift and makes reruns safe. Imperative automation is still useful for troubleshooting, emergency changes, and unsupported features.

**Ansible for configuration push and validation**

Ansible is widely used because it is agentless, readable, and idempotent. In Cisco environments, you often combine inventory, variables, templates, and network modules.

Typical Ansible building blocks:

- **Inventory**: lists devices and groups
- **Playbook**: defines tasks and order
- **Modules**: perform device-specific actions
- **Templates**: render configuration from variables
- **Facts/outputs**: capture state for validation

A simple playbook structure often looks like this:

```yaml
---
- name: Push and validate enterprise config
  hosts: access_switches
  gather_facts: no
  connection: network_cli

  tasks:
    - name: Configure VLANs
      cisco.ios.ios_vlans:
        config:
          - vlan_id: 20
            name: USERS
          - vlan_id: 30
            name: VOICE
        state: merged

    - name: Configure trunk interface
      cisco.ios.ios_interfaces:
        config:
          - name: GigabitEthernet1/0/24
            description: Uplink to distribution
            enabled: true
        state: merged

    - name: Validate interface state
      cisco.ios.ios_command:
        commands:
          - show interfaces GigabitEthernet1/0/24 switchport
          - show vlan brief
      register: validation_output

    - name: Display validation
      debug:
        var: validation_output.stdout_lines
```

Common Cisco Ansible network modules include:

- `cisco.ios.ios_config` for CLI-style configuration lines
- `cisco.ios.ios_command` for show commands and validation
- `cisco.ios.ios_interfaces` for interface state
- `cisco.ios.ios_vlans` for VLAN configuration
- `cisco.ios.ios_acls` for ACLs
- `cisco.ios.ios_l3_interfaces` for Layer 3 interface settings
- `cisco.ios.ios_static_routes` for static routing
- `cisco.ios.ios_bgp_global` and related BGP modules where supported

For NX-OS, IOS XE, and other platforms, module names differ, but the pattern is the same: use a structured module when available, and fall back to `*_config` when you need CLI-like control.

**When to use declarative vs imperative tasks**

Use **declarative** tasks when:

- You want idempotency
- You need repeatable compliance checks
- You manage many similar devices
- You want reruns to be safe

Use **imperative** tasks when:

- The feature is not well modeled by a module
- You need a one-time operational action
- You are troubleshooting or making a temporary change
- The platform only exposes command-style control

A practical rule: if you can describe the target state cleanly, use declarative automation. If you must describe the sequence of actions, use imperative automation.

**Validation patterns after configuration push**

Validation should confirm both configuration and operational behavior. A change can be syntactically correct but still fail functionally.

Good validation checks include:

- **Interface state**
  - `show interfaces status`
  - `show interfaces <int> switchport`
  - `show ip interface brief`
- **Routing state**
  - `show ip route`
  - `show ip ospf neighbor`
  - `show bgp summary`
  - `show ip protocols`
- **Policy state**
  - `show access-lists`
  - `show policy-map interface`
  - `show route-map`
- **Layer 2 state**
  - `show vlan brief`
  - `show interfaces trunk`
  - `show spanning-tree`
- **Wireless or fabric state**
  - `show wireless summary`
  - `show ap summary`
  - `show fabric summary`
  - `show lisp session` or platform-specific fabric checks

Validation should answer three questions:

1. Did the device accept the configuration?
2. Is the intended configuration present?
3. Is the feature operating correctly?

For example, after pushing a trunk change, you should verify:

- The interface is up
- The trunk allowed VLAN list is correct
- The native VLAN matches policy
- The downstream device learns the expected VLANs

A sample validation workflow in Ansible might look like this:

```yaml
- name: Validate trunk and VLAN state
  hosts: access_switches
  gather_facts: no
  connection: network_cli

  tasks:
    - name: Collect show outputs
      cisco.ios.ios_command:
        commands:
          - show interfaces GigabitEthernet1/0/24 switchport
          - show interfaces trunk
          - show vlan brief
      register: trunk_check

    - name: Assert trunk is configured
      ansible.builtin.assert:
        that:
          - "'Operational Mode: trunk' in trunk_check.stdout[0]"
          - "'Vlans allowed on trunk' in trunk_check.stdout[1]"
          - "'20' in trunk_check.stdout[2]"
```

**Compliance baselines and drift detection**

Compliance compares actual device state to an approved baseline. The baseline may come from:

- A template repository
- A controller policy
- A golden configuration
- A compliance profile in a management platform

Compliance is broader than validation. Validation is usually immediate and change-specific. Compliance is ongoing and policy-driven.

Examples of compliance checks:

- Interface descriptions match naming standards
- Access ports use approved VLANs
- Trunks allow only sanctioned VLANs
- ACLs contain required permit/deny entries
- QoS policies are attached to the correct interfaces
- Routing policy matches approved prefix-lists and route-maps
- Wireless SSIDs, tags, or profiles match the design
- Fabric edge devices use the expected role and policy

A compliance workflow often includes:

1. Define the baseline.
2. Collect current state.
3. Compare actual vs intended.
4. Report drift.
5. Remediate or approve exceptions.

Version control is important here because it gives you traceability. If the baseline changes, you want to know who changed it, when, and why.

**Cisco Catalyst Center compliance and assurance workflows**

Cisco Catalyst Center supports centralized policy, templates, inventory, and assurance workflows. In practice, it is often used as the operational source of truth for enterprise deployments.

Typical workflow:

- Import devices into inventory.
- Assign them to sites, roles, and device families.
- Apply templates or network profiles.
- Sync devices so the controller has current state.
- Run compliance checks against templates or policy.
- Review assurance data for health, reachability, and anomalies.

Common operational patterns:

- **Templates**
  - Use them to standardize device configuration.
  - Parameterize values such as hostname, management IP, site code, and uplink details.
  - Push them to many devices with consistent syntax.
- **Device sync**
  - Refresh the controller’s view of the device after manual changes or outages.
  - Use sync before compliance checks so the controller evaluates current state.
- **Compliance reports**
  - Show whether a device matches the intended template or policy.
  - Highlight missing, extra, or mismatched configuration.
  - Help you prioritize remediation.

Catalyst Center is especially useful when you need both configuration and assurance in one place. It is less about one-off CLI replacement and more about lifecycle management.

> A common operational mistake is checking compliance before syncing inventory. If the controller has stale data, the report may be misleading.

**YANG models and model-driven automation**

YANG is a data modeling language used to describe configuration and state data. It is central to NETCONF and RESTCONF workflows.

Why YANG matters:

- It defines what can be configured.
- It defines how data is structured.
- It enables programmatic validation of configuration and state.
- It reduces ambiguity compared to free-form CLI text.

In model-driven automation, you do not send arbitrary commands. You manipulate structured data that maps to a YANG model.

Practical implications:

- A YANG model may define interfaces, routing, ACLs, or system settings.
- Your automation tool uses the model to build valid requests.
- Validation becomes easier because the response structure is predictable.
- Compliance checks can compare structured data instead of parsing CLI output.

YANG is especially valuable when you need:

- Consistent multi-vendor or multi-platform logic
- Strong validation of input data
- Reliable state retrieval for compliance
- Better integration with APIs and orchestration tools

**NETCONF operations and datastore concepts**

NETCONF is a network configuration protocol that uses structured XML messages over SSH. It is designed for configuration management and state retrieval.

Important datastore concepts:

- **running**: the active configuration currently in use
- **candidate**: a staging area where changes can be prepared before commit
- **startup**: the configuration loaded at boot on platforms that support it

Not every platform supports every datastore. Cisco platforms commonly expose `running`, and support for `candidate` or `startup` depends on the OS and feature set.

Key NETCONF operations:

- **`get-config`**
  - Retrieves configuration from a datastore such as `running`.
  - Use it when you want the current config state.
- **`get`**
  - Retrieves both configuration and operational state, depending on the filter and model.
  - Use it for broader state collection.
- **`edit-config`**
  - Sends configuration changes to a target datastore.
  - Use it to push intended changes.
- **`commit`**
  - Applies staged changes when a candidate datastore is used.
- **`discard-changes`**
  - Cancels uncommitted candidate changes.

A simplified NETCONF flow looks like this:

```xml
<rpc message-id="101" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get-config>
    <source>
      <running/>
    </source>
    <filter type="subtree">
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces"/>
    </filter>
  </get-config>
</rpc>
```

And a configuration push may use `edit-config`:

```xml
<rpc message-id="102" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <edit-config>
    <target>
      <running/>
    </target>
    <config>
      <!-- structured config payload -->
    </config>
  </edit-config>
</rpc>
```

Operationally, NETCONF is useful because it can support transaction-like behavior. If the platform supports candidate and commit, you can stage changes, validate them, and then commit. That reduces the risk of partial configuration application.

**RESTCONF operations and HTTP response handling**

RESTCONF exposes YANG-modeled data over HTTP. It is often easier to integrate with web tooling, scripts, and automation platforms that already speak REST.

Common RESTCONF methods:

- `GET` to retrieve config or state
- `POST` to create resources
- `PUT` to replace a resource
- `PATCH` to partially modify a resource
- `DELETE` to remove a resource

Typical response handling:

- `200 OK` or `204 No Content`: success
- `201 Created`: resource created
- `400 Bad Request`: malformed payload or invalid data
- `401 Unauthorized` / `403 Forbidden`: authentication or authorization issue
- `404 Not Found`: resource path or model not available
- `409 Conflict`: state conflict or duplicate resource
- `500` series: server-side failure

Example RESTCONF request:

```bash
curl -k -u admin:password \
  -H "Accept: application/yang-data+json" \
  https://router.example.com/restconf/data/ietf-interfaces:interfaces
```

Example RESTCONF patch:

```bash
curl -k -u admin:password \
  -H "Content-Type: application/yang-data+json" \
  -X PATCH \
  https://router.example.com/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1 \
  -d '{
    "ietf-interfaces:interface": {
      "name": "GigabitEthernet1",
      "description": "Uplink to core",
      "enabled": true
    }
  }'
```

For exam and operational purposes, remember that RESTCONF is not just “HTTP version of CLI.” It is a structured API tied to YANG models, and its error handling is based on HTTP semantics plus model validation.

**Error handling, retries, and rollback patterns**

Automation must assume failure. Good workflows handle errors explicitly instead of hoping the push succeeds.

Common failure patterns:

- Authentication failure
- Unsupported model or feature
- Invalid payload or syntax
- Device busy or transient transport failure
- Partial application of a change
- Post-change validation failure

Practical handling patterns:

- **Check status codes**
  - Treat `2xx` as success only if the response body or post-check confirms the intended state.
  - Treat `4xx` as a request or authorization problem.
  - Treat `5xx` as a server or platform problem.
- **Retry transient failures**
  - Retry on timeouts or temporary transport issues.
  - Do not blindly retry on validation errors or bad payloads.
- **Use rollback or compensation**
  - If the platform supports candidate rollback, discard uncommitted changes.
  - If not, apply a compensating change that restores the previous known-good state.
- **Validate after commit**
  - A successful API call does not guarantee the feature works.
  - Always verify operational state after the push.

A simple retry pattern in Ansible can look like this:

```yaml
- name: Push config with retries
  cisco.ios.ios_config:
    lines:
      - interface GigabitEthernet1/0/24
      - description Uplink to core
  register: push_result
  retries: 3
  delay: 5
  until: push_result is succeeded
```

For more complex workflows, you may need a rescue block:

```yaml
- block:
    - name: Apply change
      cisco.ios.ios_config:
        lines:
          - ip access-list extended WEB-FILTER
          - permit tcp any any eq 443
  rescue:
    - name: Restore previous config
      cisco.ios.ios_config:
        src: backup.cfg
```

**Practical validation examples by feature**

Different features need different validation methods.

- **Routing**
  - Verify route presence, next hop, and protocol adjacency.
  - Example checks: `show ip route`, `show bgp summary`, `show ip ospf neighbor`
- **ACLs**
  - Verify rule order, hit counts, and interface attachment.
  - Example checks: `show access-lists`, `show ip interface <int>`
- **QoS**
  - Verify policy attachment and counters.
  - Example checks: `show policy-map interface`, `show class-map`
- **VLANs and trunks**
  - Verify VLAN existence, trunk mode, allowed VLANs, and native VLAN.
  - Example checks: `show vlan brief`, `show interfaces trunk`
- **Wireless**
  - Verify SSID, AP join status, and policy/profile assignment.
  - Example checks: `show wireless summary`, controller assurance views
- **Fabric**
  - Verify fabric role, control-plane adjacency, and policy deployment.
  - Example checks: platform-specific fabric status commands and controller compliance

A good validation plan checks both configuration and behavior. For example, an ACL may exist in the running config but not be applied to the correct interface. A route may be present in the table but not installed for the intended reason. A QoS policy may be attached but not matching traffic as expected.

**End-to-end workflow example**

Consider a branch rollout where you need to standardize access switch configuration across multiple sites.

Desired outcome:

- VLAN 20 for users
- VLAN 30 for voice
- Access ports in VLAN 20
- Uplink trunk allows VLANs 20 and 30
- ACL applied to a guest-facing interface
- QoS policy attached to voice ports
- Compliance report confirms all devices match the template

A practical workflow might be:

1. Define the baseline in Git.
   - Store VLAN, interface, ACL, and QoS variables in version control.
2. Render configuration with Ansible.
   - Use templates for site-specific values.
3. Push the configuration.
   - Use `ios_vlans`, `ios_interfaces`, `ios_acls`, and `ios_config` where needed.
4. Validate the change.
   - Check VLANs, trunks, ACL attachment, and QoS counters.
5. Sync the controller.
   - If using Catalyst Center, refresh device inventory.
6. Run compliance.
   - Compare the device against the approved template or policy.
7. Remediate drift.
   - Fix mismatches or document approved exceptions.

Example mixed push and validation snippet:

```yaml
- name: Branch access switch standardization
  hosts: branch_switches
  gather_facts: no
  connection: network_cli

  tasks:
    - name: Ensure VLANs exist
      cisco.ios.ios_vlans:
        config:
          - vlan_id: 20
            name: USERS
          - vlan_id: 30
            name: VOICE
        state: merged

    - name: Configure access port
      cisco.ios.ios_interfaces:
        config:
          - name: GigabitEthernet1/0/10
            description: User workstation
            enabled: true
        state: merged

    - name: Apply interface config lines
      cisco.ios.ios_config:
        lines:
          - interface GigabitEthernet1/0/10
          - switchport mode access
          - switchport access vlan 20
          - spanning-tree portfast
          - spanning-tree bpduguard enable

    - name: Validate state
      cisco.ios.ios_command:
        commands:
          - show vlan brief
          - show interfaces GigabitEthernet1/0/10 switchport
          - show access-lists
      register: postcheck

    - name: Assert expected state
      ansible.builtin.assert:
        that:
          - "'20' in postcheck.stdout[0]"
          - "'Access Mode VLAN: 20' in postcheck.stdout[1]"
```

**Operational best practices**

- Keep desired state in version control.
- Prefer idempotent modules over raw command strings when possible.
- Validate both config and operational behavior.
- Use controller sync before compliance checks.
- Treat compliance as continuous, not one-time.
- Build rollback or compensation into every change workflow.
- Separate approved exceptions from true drift.
- Use structured data and YANG models when the platform supports them.
