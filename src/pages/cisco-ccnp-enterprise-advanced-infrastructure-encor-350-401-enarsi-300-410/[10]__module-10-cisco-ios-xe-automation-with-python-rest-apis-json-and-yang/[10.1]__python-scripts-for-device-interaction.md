## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Overview**

Python scripts for device interaction let you automate configuration, verification, and troubleshooting across Cisco devices. In CCNP Enterprise, you are expected to understand how scripts connect to devices, authenticate securely, send commands or API requests, parse structured and unstructured output, and recover cleanly from failures.

The key exam idea is not just “can you run a command,” but “can you build a reliable workflow.” That means you handle credentials safely, validate inputs, choose the right transport, verify results, and log enough detail to troubleshoot when something breaks.

**Where Python fits in Cisco automation**

Python is commonly used as the glue between your intent and the device interface. You can use it for:

- **CLI over SSH** for legacy or operationally simple workflows.
- **NETCONF** for structured configuration and state retrieval.
- **RESTCONF** for HTTP-based API access to YANG-modeled data.
- **Hybrid workflows** where you query one interface and verify with another.
- **Automation frameworks** such as pyATS and Unicon for test, parsing, and device session management.

A practical script usually follows this pattern:

1. Load inventory and credentials.
2. Validate inputs and target devices.
3. Open a session using the correct transport.
4. Collect state or push configuration.
5. Parse and verify the response.
6. Log success or failure.
7. Close the session and clean up.

**Secure authentication and secret handling**

Never hardcode passwords, API tokens, or private keys in scripts. Use environment variables, encrypted credential stores, or credential files with restricted permissions. For exam purposes, know the common secure patterns and why they matter.

- **Environment variables** are simple for lab use and CI/CD pipelines.
- **Credential files** can work if protected with file permissions and encryption.
- **SSH keys** are preferred over passwords for device access when supported.
- **Host key verification** helps prevent man-in-the-middle attacks by confirming the device identity.
- **Secrets managers** are best in production, though the exact product may vary by environment.

Example of reading credentials from environment variables:

```python
import os

username = os.getenv("NET_USER")
password = os.getenv("NET_PASS")

if not username or not password:
    raise ValueError("Missing NET_USER or NET_PASS environment variables")
```

Example of using an SSH private key with Paramiko:

```python
import paramiko

key = paramiko.RSAKey.from_private_key_file("/home/admin/.ssh/id_rsa")
client = paramiko.SSHClient()
client.load_system_host_keys()
client.set_missing_host_key_policy(paramiko.RejectPolicy())
client.connect(
    hostname="10.0.0.10",
    username="admin",
    pkey=key,
    look_for_keys=False,
    allow_agent=False
)
```

Important security points:

- **Avoid hardcoded passwords** in source code and notebooks.
- **Protect private keys** with file permissions and passphrases when possible.
- **Verify host keys** instead of blindly accepting unknown devices.
- **Prefer key-based SSH auth** for repeatable automation.
- **Use least privilege** accounts for read-only verification scripts.

> If you disable host key checking for convenience, you reduce security and can hide device impersonation problems.

**Transport choices: SSH vs NETCONF vs RESTCONF**

Choose the transport based on the task, device support, and data shape.

| Transport | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| SSH/CLI | Operational commands, older platforms, ad hoc automation | Familiar, widely supported | Output parsing can be brittle |
| NETCONF | Structured config/state, YANG-based workflows | Strong schema support, candidate/commit workflows | Requires device support and XML handling |
| RESTCONF | HTTP API access to YANG data | Easy integration with web tooling, JSON support | Feature support varies by platform/version |

Use SSH when you need quick operational access or the platform does not expose the needed model. Use NETCONF when you want structured configuration transactions and better idempotency. Use RESTCONF when you want HTTP-based integration and JSON payloads are convenient for your application.

**Core Python libraries and Cisco ecosystems**

Several libraries are common in Cisco automation workflows:

- **Netmiko** for SSH-based CLI automation across many network platforms.
- **Paramiko** for lower-level SSH control when you need custom behavior.
- **requests** for RESTCONF and other HTTP APIs.
- **ncclient** for NETCONF sessions and XML RPC operations.
- **pyATS** for test automation, device abstraction, and validation workflows.
- **Unicon** for connection management used by pyATS and for interactive device sessions.
- **Genie parsers** for structured parsing of CLI output into Python objects.

How they fit together:

- Use **Netmiko** when you want fast CLI automation with minimal setup.
- Use **Paramiko** when you need direct SSH control, custom channels, or lower-level handling.
- Use **ncclient** when you need NETCONF RPCs, candidate datastore operations, or YANG-driven config.
- Use **requests** when you interact with RESTCONF endpoints and want JSON handling.
- Use **pyATS/Unicon** when you need repeatable test workflows, connection abstraction, and verification at scale.

**SSH CLI automation with Netmiko**

Netmiko is a common choice for sending commands and collecting output. It handles many device-specific session details and is easier than raw Paramiko for most CLI tasks.

Example:

```python
from netmiko import ConnectHandler

device = {
    "device_type": "cisco_ios",
    "host": "10.0.0.10",
    "username": "admin",
    "password": "Cisco123!",
    "secret": "Enable123!"
}

with ConnectHandler(**device) as conn:
    conn.enable()
    output = conn.send_command("show ip interface brief")
    print(output)
```

Use this pattern when:

- You need operational verification.
- You are collecting text output from a CLI.
- You want a quick script for a lab or maintenance task.

Be aware of limitations:

- CLI output may change across software versions.
- Parsing text is less reliable than structured data.
- Interactive prompts can require extra handling.

**Parsing CLI output and structured data**

Automation scripts often need to convert raw output into usable data. The format you choose depends on the source.

- **Plain text** is common for CLI output and often needs regex or parser libraries.
- **JSON** is ideal for RESTCONF and many modern APIs because it maps naturally to Python dictionaries.
- **XML** is common in NETCONF because the protocol uses XML RPC payloads.

Use JSON when the API returns native JSON or when you want simple Python-native data structures. Use XML when working with NETCONF or XML-based device responses. Convert data carefully and validate expected keys or tags before using the result.

Example JSON handling with `requests`:

```python
import requests

url = "https://10.0.0.20/restconf/data/ietf-interfaces:interfaces"
headers = {
    "Accept": "application/yang-data+json",
    "Content-Type": "application/yang-data+json"
}

response = requests.get(url, headers=headers, auth=("admin", "Cisco123!"), verify=False)
response.raise_for_status()
data = response.json()

for interface in data["ietf-interfaces:interfaces"]["interface"]:
    print(interface["name"], interface.get("description", "no description"))
```

Example XML handling with `ncclient`:

```python
from ncclient import manager

with manager.connect(
    host="10.0.0.20",
    port=830,
    username="admin",
    password="Cisco123!",
    hostkey_verify=True,
    device_params={"name": "iosxe"},
    look_for_keys=False,
    allow_agent=False
) as m:
    reply = m.get_config(source="running", filter=("subtree", "<interfaces xmlns='urn:ietf:params:xml:ns:yang:ietf-interfaces'/>"))
    print(reply.xml)
```

**NETCONF session setup and authentication**

NETCONF uses an SSH transport and XML-based RPCs. In Cisco environments, you often use it for structured reads and transactional configuration.

Typical setup considerations:

- Enable NETCONF on the device.
- Ensure SSH access is working.
- Use the correct port, often `830`.
- Authenticate with username/password or SSH keys, depending on platform support.
- Verify the device supports the YANG model you plan to use.

Example end-to-end NETCONF connection:

```python
from ncclient import manager

with manager.connect(
    host="10.0.0.20",
    port=830,
    username="admin",
    password="Cisco123!",
    hostkey_verify=True,
    device_params={"name": "iosxe"},
    look_for_keys=False,
    allow_agent=False
) as m:
    print(m.server_capabilities)
    interfaces = m.get(filter=("subtree", """
        <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces"/>
    """))
    print(interfaces.xml)
```

Practical NETCONF workflow:

1. Connect and inspect capabilities.
2. Retrieve current state.
3. Build a candidate configuration payload.
4. Apply the change.
5. Verify the result.
6. Commit or roll back based on the outcome.

**RESTCONF session setup and headers**

RESTCONF uses HTTP methods such as `GET`, `POST`, `PUT`, `PATCH`, and `DELETE`. It commonly returns JSON, though XML is also possible.

Important headers:

- `Accept` tells the server what response format you want.
- `Content-Type` tells the server what format you are sending.
- `Authorization` carries credentials or tokens when required.

Example RESTCONF GET request:

```python
import requests

base_url = "https://10.0.0.20/restconf/data"
headers = {
    "Accept": "application/yang-data+json",
    "Content-Type": "application/yang-data+json"
}

response = requests.get(
    f"{base_url}/ietf-interfaces:interfaces",
    headers=headers,
    auth=("admin", "Cisco123!"),
    verify=False,
    timeout=10
)
response.raise_for_status()
print(response.json())
```

Example RESTCONF configuration update:

```python
payload = {
    "ietf-interfaces:interface": {
        "name": "Loopback100",
        "description": "Created by Python",
        "type": "iana-if-type:softwareLoopback",
        "enabled": True
    }
}

response = requests.put(
    f"{base_url}/ietf-interfaces:interfaces/interface=Loopback100",
    headers=headers,
    auth=("admin", "Cisco123!"),
    json=payload,
    verify=False,
    timeout=10
)
response.raise_for_status()
```

Use `PUT` for full replacement of a resource and `PATCH` for partial updates when supported. Verify the device’s RESTCONF implementation because support can vary.

**Exception handling, retries, and cleanup**

Robust scripts must handle failures gracefully. Python exception handling is essential in device automation because network sessions fail, devices reject payloads, and APIs return unexpected data.

Use `try`, `except`, and `finally` to protect the workflow:

- **try** contains the risky operation.
- **except** handles known failures.
- **finally** performs cleanup such as closing sessions.
- **raise** rethrows or creates meaningful errors when the script cannot continue.

Example with retry logic:

```python
import time
import requests

def get_with_retry(url, headers, auth, retries=3, delay=2):
    last_error = None
    for attempt in range(1, retries + 1):
        try:
            response = requests.get(url, headers=headers, auth=auth, verify=False, timeout=10)
            response.raise_for_status()
            return response.json()
        except (requests.Timeout, requests.ConnectionError) as exc:
            last_error = exc
            if attempt == retries:
                raise
            time.sleep(delay * attempt)
    raise last_error
```

Example with `try/except/finally` for SSH:

```python
from netmiko import ConnectHandler

conn = None
try:
    conn = ConnectHandler(**device)
    conn.enable()
    output = conn.send_command("show version")
    print(output)
except Exception as exc:
    raise RuntimeError(f"Device interaction failed: {exc}") from exc
finally:
    if conn:
        conn.disconnect()
```

Good retry behavior:

- Retry transient failures such as timeouts or temporary connection drops.
- Do not blindly retry authentication failures or invalid payloads.
- Use backoff to avoid hammering a device.
- Log each attempt so you can diagnose repeated failures.

**Logging and debugging**

Logging is critical for troubleshooting automation failures. Use the `logging` module instead of `print()` for anything beyond a quick lab test.

Recommended logging practices:

- Log the device name, action, and result.
- Use `INFO` for normal milestones.
- Use `DEBUG` for payloads and protocol details in lab environments.
- Use `WARNING` for recoverable issues.
- Use `ERROR` for failed operations.
- Avoid logging secrets, tokens, or private keys.

Example:

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s %(levelname)s %(name)s: %(message)s"
)
logger = logging.getLogger("device-automation")

logger.info("Connecting to %s", "10.0.0.20")
logger.debug("Payload: %s", payload)
```

Useful debugging techniques:

- Enable library debug logs when troubleshooting protocol issues.
- Capture raw device responses before parsing them.
- Validate that the device supports the expected YANG model or CLI command.
- Check SSH reachability, ACLs, and management VRF routing.
- Confirm that the device clock and certificates are valid when using HTTPS APIs.

**Idempotent configuration and rollback**

Idempotent automation means running the same script multiple times does not create duplicate or conflicting changes. This is a major exam concept because reliable automation should converge the device to the desired state.

A good idempotent script:

- Checks current state before changing anything.
- Applies only missing or incorrect settings.
- Verifies the final state after the change.
- Avoids re-adding existing objects.

Example logic for an interface description:

```python
desired = "Created by Python"
current = get_current_description()

if current != desired:
    apply_description(desired)
    verify_description(desired)
```

For configuration pushes, plan rollback behavior:

- Save the current configuration or use a candidate datastore when available.
- Apply the change.
- Verify the result.
- Roll back if verification fails.

Example rollback pattern:

```python
backup = conn.send_command("show running-config")
try:
    conn.send_config_set([
        "interface Loopback100",
        "description Created by Python"
    ])
    verify = conn.send_command("show run interface Loopback100")
    if "Created by Python" not in verify:
        raise RuntimeError("Verification failed")
except Exception:
    conn.send_config_set(["no interface Loopback100"])
    raise
```

On platforms that support candidate configuration and commit/rollback, prefer those features because they are safer than ad hoc CLI changes.

**Input validation and inventory schema checks**

Automation scripts should validate input before touching devices. This prevents malformed payloads, bad hostnames, and missing fields from causing partial changes.

Validate:

- IP addresses and hostnames.
- Required inventory keys such as `host`, `username`, and `device_type`.
- Interface names and VLAN IDs.
- JSON or YAML structure before use.
- File paths and file contents.

Example inventory validation:

```python
def validate_device(device):
    required = {"host", "username", "password"}
    missing = required - device.keys()
    if missing:
        raise ValueError(f"Missing keys: {sorted(missing)}")
    if not isinstance(device["host"], str) or not device["host"]:
        raise ValueError("Invalid host value")
```

For file-based inventories, validate the schema after loading YAML or JSON. This is especially useful when multiple engineers edit the same source of truth.

**pyATS and Unicon in device workflows**

pyATS is a Cisco automation and test framework used for structured validation, test orchestration, and device interaction. Unicon provides the connection layer that pyATS uses to manage device sessions.

Use them when you need:

- Repeatable test cases.
- Structured device connections.
- Parsing and verification at scale.
- Integration with testbeds and inventories.

Typical workflow:

1. Define a testbed file.
2. Connect to devices through Unicon.
3. Run commands or collect state.
4. Parse and validate outputs.
5. Report pass/fail results.

Example testbed-style connection concept:

```yaml
devices:
  r1:
    os: iosxe
    type: router
    connections:
      cli:
        protocol: ssh
        ip: 10.0.0.10
```

Example pyATS-style usage:

```python
from genie.testbed import load

testbed = load("testbed.yaml")
device = testbed.devices["r1"]
device.connect()
output = device.execute("show ip route")
print(output)
device.disconnect()
```

pyATS is especially useful when you want automation to behave like a test harness rather than a one-off script.

**Practical end-to-end example**

This example shows a simple workflow that connects, validates state, applies a change, verifies it, and logs the result.

```python
import logging
from netmiko import ConnectHandler

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

device = {
    "device_type": "cisco_ios",
    "host": "10.0.0.10",
    "username": "admin",
    "password": "Cisco123!",
    "secret": "Enable123!"
}

target_interface = "Loopback100"
target_description = "Created by Python"

with ConnectHandler(**device) as conn:
    conn.enable()

    current = conn.send_command(f"show run interface {target_interface}")
    if target_description in current:
        logger.info("No change needed on %s", target_interface)
    else:
        try:
            conn.send_config_set([
                f"interface {target_interface}",
                f"description {target_description}"
            ])
            verify = conn.send_command(f"show run interface {target_interface}")
            if target_description not in verify:
                raise RuntimeError("Verification failed after config push")
            logger.info("Configuration applied successfully")
        except Exception as exc:
            logger.error("Failed to configure %s: %s", target_interface, exc)
            raise
```

This pattern is exam-relevant because it combines state check, change, verification, and error handling.

**Troubleshooting checklist**

When a script fails, isolate the problem by checking the layer where it breaks.

- **Connectivity**
  - Can you ping or reach the management IP?
  - Is the correct VRF or interface used?
  - Are ACLs or firewalls blocking the port?

- **Authentication**
  - Are the username and password correct?
  - Is the SSH key valid and unencrypted if required?
  - Is the account authorized for the requested operation?

- **Protocol support**
  - Does the device support NETCONF or RESTCONF?
  - Is the correct port open, such as `830` for NETCONF?
  - Are the required YANG models available?

- **Parsing**
  - Did the device return the expected format?
  - Are you handling JSON, XML, or text correctly?
  - Did the output change because of a software upgrade?

- **Change verification**
  - Did the config apply but not persist?
  - Did the device reject part of the payload?
  - Did the script verify the wrong object?

**When to use each approach**

- Use **CLI/SSH** for quick operational tasks and broad compatibility.
- Use **NETCONF** for structured, transactional configuration and state retrieval.
- Use **RESTCONF** for HTTP-based integrations and JSON-friendly workflows.
- Use **pyATS/Unicon** when you need scalable validation and test automation.
- Use **Paramiko** when you need lower-level SSH control beyond Netmiko.
