## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Troubleshooting model and plane separation**

Operational troubleshooting is faster when you separate symptoms by plane:

- **Management plane**: access to the device itself, such as SSH, SNMP, NETCONF, or console.
- **Control plane**: protocol exchange and route/neighbor/state formation, such as OSPF, EIGRP, BGP, STP, HSRP, and ARP/ND processing.
- **Data plane**: actual packet forwarding through the device, including CEF, adjacency programming, ACLs, QoS, encapsulation, and tunneling.

A useful exam habit is to ask: “Is the device reachable, is the protocol formed, and is traffic forwarded?” That sequence narrows the fault domain quickly.

| Plane | What it does | Common symptoms | Typical checks |
|---|---|---|---|
| Management | Device access and monitoring | SSH fails, SNMP times out, API unreachable | Interface reachability, VRF, ACLs, CoPP, CPU, AAA |
| Control | Builds topology and forwarding state | Neighbors down, routes missing, STP blocked unexpectedly | Neighbor tables, timers, authentication, MTU, hello/dead settings |
| Data | Forwards packets | Ping fails, one-way traffic, blackholing | CEF, adjacency, ACL/QoS, MTU, encapsulation, counters |

> If control-plane state looks correct but traffic still fails, do not assume forwarding is healthy. Check CEF programming, adjacency resolution, policy, and encapsulation next.

**A practical troubleshooting workflow**

Use a repeatable sequence so you do not jump between unrelated commands.

1. Identify the symptom and scope.
   - Is the issue local, upstream, downstream, or only for one prefix, VLAN, VRF, or application?
2. Verify the management path.
   - Confirm you can reach the device and collect outputs.
3. Verify the control plane.
   - Check neighbors, adjacencies, route exchange, and protocol state.
4. Verify the data plane.
   - Confirm the route is installed and programmed for forwarding.
5. Test with a known-good source and destination.
   - Use `ping`, `traceroute`, and interface counters to isolate the failure point.
6. Check policy and platform features.
   - ACLs, QoS, PBR, NAT, VRF, tunnels, and hardware offload can change the result.

**Core verification commands**

These commands are common starting points on Cisco IOS XE and similar enterprise platforms:

```bash
show ip interface brief
show ipv6 interface brief
show interfaces status
show interfaces counters errors
show ip route
show ipv6 route
show arp
show ipv6 neighbors
show cdp neighbors detail
show lldp neighbors detail
show logging
show processes cpu sorted
show memory statistics
```

Use platform-specific commands carefully. Some outputs vary by release, hardware, and feature set.

**Control-plane troubleshooting fundamentals**

The control plane is where routing, switching state, and neighbor relationships are formed. If the control plane fails, the data plane often has nothing valid to forward with.

Common control-plane failure causes include:

- **Layer 1/2 mismatch**
  - Wrong VLAN, trunk allowed list, duplex, speed, or physical link issue.
- **Addressing or mask mismatch**
  - Incorrect IP, subnet mask, or IPv6 prefix length.
- **Authentication mismatch**
  - OSPF, EIGRP, BGP, HSRP, VRRP, or routing protocol authentication mismatch.
- **Timer mismatch**
  - Hello/dead timers, hold timers, or STP timers.
- **MTU mismatch**
  - Especially important for OSPF, OSPFv3, BGP, GRE, and tunnels.
- **ACL or policy filtering**
  - Control traffic blocked by interface ACLs, CoPP, or upstream policy.
- **VRF mismatch**
  - Neighbors or routes exist in different routing tables.
- **Passive interface or shutdown**
  - Protocol enabled but not active on the interface.
- **Duplicate addressing or ID conflicts**
  - Duplicate router ID, duplicate IP, or duplicate STP bridge ID.

**Cisco Express Forwarding and the data plane**

Cisco Express Forwarding (**CEF**) is the main forwarding mechanism on modern Cisco enterprise devices. It separates route computation from packet forwarding by programming a forwarding table and adjacency table.

CEF troubleshooting matters because a route can exist in the routing table but still fail in forwarding if the forwarding information base (**FIB**) or adjacency is incomplete.

Key CEF concepts:

- **RIB**: Routing Information Base, the control-plane routing table.
- **FIB**: Forwarding Information Base, the data-plane lookup table derived from the RIB.
- **Adjacency table**: Next-hop rewrite information, such as MAC address, encapsulation, or tunnel rewrite.
- **Recursive lookup**: A route may point to a next hop that must itself be resolved through another route.
- **Glean adjacency**: The router knows the connected network but must ARP/ND for the specific host.
- **Receive adjacency**: Traffic is destined to the router itself.
- **Drop adjacency**: Traffic is intentionally discarded, often due to a null route or policy.
- **Discard adjacency**: Similar to drop behavior in forwarding, often used for summary or null routes.

A route in `show ip route` but not in `show ip cef` usually indicates a programming, recursion, or feature interaction issue. On some platforms, CEF may be disabled or not fully populated for certain prefixes, so interpret the result in context.

Useful checks:

```bash
show ip cef
show ip cef exact-route 10.1.1.10 8.8.8.8
show adjacency
show adjacency detail
show ip cef summary
show platform hardware forwarding
```

**How CEF failures present**

- **Route exists, ping fails**
  - Check adjacency resolution, ARP/ND, ACLs, QoS, MTU, or tunnel encapsulation.
- **Route exists, but traffic blackholes**
  - Check recursive next hop resolution, PBR, VRF mismatch, or a drop adjacency.
- **Only some destinations fail**
  - Check glean behavior, host ARP/ND, or asymmetric return path.
- **CPU is high and forwarding is slow**
  - Check whether traffic is being punted to the control plane instead of hardware-forwarded.

> A route can be present in the RIB and still fail in the data plane if the next hop is unresolved, the adjacency is incomplete, or policy changes the forwarding path.

**Layer 2 and switching-plane checks**

Many enterprise outages begin as Layer 2 problems that look like routing failures.

Check these first when a VLAN, trunk, or access segment is involved:

```bash
show vlan brief
show interfaces trunk
show spanning-tree vlan 10
show spanning-tree interface gi1/0/1 detail
show mac address-table dynamic
show etherchannel summary
show lacp neighbor
```

Common Layer 2 issues:

- **STP blocking**
  - A port may be forwarding on one switch and blocking on another.
- **VLAN mismatch**
  - Access port in the wrong VLAN or trunk missing the VLAN.
- **Native VLAN mismatch**
  - Can cause untagged traffic issues and STP inconsistencies.
- **Port-channel mismatch**
  - LACP/PAgP settings, speed, duplex, or allowed VLAN mismatch.
- **MAC learning failure**
  - The switch does not learn the host MAC, often due to cabling, security, or VLAN issues.

**Spanning Tree Protocol troubleshooting**

Spanning Tree Protocol (**STP**) prevents loops but can also hide connectivity if the topology or root placement is wrong.

Focus on:

- Root bridge election
- Port roles and states
- Inconsistent or blocked ports
- Edge/PortFast behavior
- BPDU Guard and Root Guard events

Useful commands:

```bash
show spanning-tree
show spanning-tree root
show spanning-tree vlan 20
show spanning-tree inconsistentports
show spanning-tree interface gi1/0/10 detail
show logging | include SPANTREE|BPDU|ROOTGUARD
```

Common misconfigurations:

- **Wrong root bridge**
  - A distribution switch should usually be the root for user VLANs.
- **Unexpected blocking**
  - A redundant link is blocked because of cost or priority.
- **PortFast on non-edge ports**
  - Can create loops if used incorrectly.
- **BPDU Guard shutdown**
  - An access port goes err-disabled after receiving BPDUs.
- **Root Guard**
  - Prevents a downstream switch from becoming root, but can block a port if misapplied.

**First-hop redundancy troubleshooting**

First-hop redundancy protocols provide gateway availability, but they fail in different ways.

| Protocol | Key state checks | Common failure causes |
|---|---|---|
| HSRP | Active/Standby state, virtual IP, priority, preempt | Authentication mismatch, interface tracking, VLAN mismatch, hello/hold mismatch |
| VRRP | Master/Backup state, virtual IP, priority | Version mismatch, authentication, timer mismatch, interface issue |
| GLBP | Active virtual gateway, AVF load balancing | Group mismatch, weighting, authentication, tracking failure |

Useful commands:

```bash
show standby brief
show standby
show vrrp brief
show vrrp
show glbp brief
show glbp
show logging | include HSRP|VRRP|GLBP
```

Practical checks:

- Confirm both peers use the same group number and virtual IP.
- Verify the interface is up/up and in the correct VLAN or subnet.
- Check tracking objects if failover occurs unexpectedly.
- Confirm preemption behavior matches the design.
- Validate that hosts use the virtual IP as the default gateway.

Example HSRP configuration:

```cisco
interface Vlan10
 ip address 10.10.10.2 255.255.255.0
 standby version 2
 standby 10 ip 10.10.10.1
 standby 10 priority 110
 standby 10 preempt
 standby 10 track GigabitEthernet1/0/48 20
```

If the tracked interface fails, the priority drops and the active role may move to the peer.

**Routing protocol troubleshooting**

Routing protocols fail for predictable reasons: adjacency formation, authentication, network statements, timers, and topology filtering.

**OSPF**

Open Shortest Path First (**OSPF**) depends on matching area, timers, authentication, network type, and MTU behavior.

Useful commands:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip ospf interface gigabitEthernet1/0/1
show ip ospf database
show ip route ospf
show logging | include OSPF
```

Common OSPF issues:

- **Area mismatch**
  - Neighbors may not form if interfaces are in different areas.
- **Hello/dead mismatch**
  - Adjacency fails before full neighbor state.
- **Authentication mismatch**
  - Plain text or MD5 authentication must match.
- **MTU mismatch**
  - Neighbors may get stuck in `EXSTART` or `EXCHANGE`.
- **Network type mismatch**
  - Broadcast, point-to-point, and NBMA behavior differs.
- **Passive interface**
  - The interface advertises networks but does not form neighbors.
- **Duplicate router ID**
  - Can cause unstable or duplicate neighbor behavior.

OSPFv3 adds IPv6-specific considerations:

- Link-local addresses are used for neighbor formation.
- Interface IDs and IPv6 enablement matter.
- Neighbor discovery must work on the link.
- Authentication and area settings still matter, but the operational model is interface-centric.

Example OSPFv2 configuration:

```cisco
router ospf 10
 router-id 1.1.1.1
 passive-interface default
 no passive-interface GigabitEthernet1/0/1
 network 10.1.1.0 0.0.0.255 area 0
```

**EIGRP**

Enhanced Interior Gateway Routing Protocol (**EIGRP**) is sensitive to AS number, K-values, authentication, and interface reachability.

Useful commands:

```bash
show ip eigrp neighbors
show ip eigrp topology
show ip protocols
show ip route eigrp
show logging | include EIGRP
```

Common EIGRP issues:

- **AS mismatch**
  - Neighbors must use the same autonomous system number.
- **K-value mismatch**
  - Metric parameters must match.
- **Authentication mismatch**
  - Key chain or MD5 settings must align.
- **Passive interface**
  - Prevents neighbor formation.
- **Stub mismatch**
  - A stub router may not advertise or accept routes as expected.
- **ACL or policy filtering**
  - EIGRP uses IP protocol 88 for IPv4 and IPv6 EIGRP control traffic.

Example EIGRP configuration:

```cisco
router eigrp 100
 network 10.1.1.0 0.0.0.255
 passive-interface default
 no passive-interface GigabitEthernet1/0/1
 eigrp router-id 2.2.2.2
```

**BGP**

Border Gateway Protocol (**BGP**) is often the most policy-sensitive protocol in the enterprise edge.

Useful commands:

```bash
show ip bgp summary
show ip bgp neighbors
show ip bgp
show bgp ipv6 unicast summary
show bgp ipv6 unicast neighbors
show logging | include BGP
```

Platform syntax can vary, especially for IPv6 address families, so verify the exact command set on the device.

Common BGP issues:

- **Neighbor IP or remote-as mismatch**
  - Session never establishes.
- **Update-source mismatch**
  - Loopback peering fails if the source is wrong.
- **TTL or multihop issue**
  - Required for non-directly connected peers.
- **Authentication mismatch**
  - TCP session may not form.
- **Prefix filtering**
  - Route-map, prefix-list, or AS-path policy blocks routes.
- **Next-hop reachability**
  - Routes may be received but not installed.
- **Maximum-prefix**
  - Session may reset or stop accepting routes.

Example BGP configuration:

```cisco
router bgp 65001
 neighbor 192.0.2.2 remote-as 65002
 neighbor 192.0.2.2 description WAN-PEER
 address-family ipv4 unicast
  neighbor 192.0.2.2 activate
  network 10.10.10.0 mask 255.255.255.0
```

**IPv6 control-plane troubleshooting**

IPv6 introduces neighbor discovery behavior that differs from IPv4 ARP.

Important IPv6 control-plane elements:

- **Router Solicitation (RS)** and **Router Advertisement (RA)**
  - Hosts use these to learn prefixes and default gateways.
- **Neighbor Solicitation (NS)** and **Neighbor Advertisement (NA)**
  - Used for address resolution and reachability.
- **Link-local addresses**
  - Many IPv6 routing adjacencies use link-local next hops.
- **Duplicate Address Detection (DAD)**
  - Prevents duplicate IPv6 address assignment.

Useful commands:

```bash
show ipv6 interface brief
show ipv6 neighbors
show ipv6 route
show ipv6 ospf neighbor
show bgp ipv6 unicast summary
ping ipv6 fe80::1 source gi1/0/1
traceroute ipv6 2001:db8:100::10
```

Common IPv6 issues:

- **IPv6 not enabled on the interface**
  - The interface may have an address but not be forwarding as expected.
- **Missing link-local reachability**
  - Routing adjacencies fail if the link-local path is broken.
- **RA suppression or filtering**
  - Hosts do not learn a default gateway.
- **ND failure**
  - Similar to ARP failure in IPv4, but with multicast-based discovery.
- **OSPFv3 adjacency issues**
  - Often caused by interface enablement, MTU, or link-local problems.
- **BGP IPv6 next-hop issues**
  - The next hop must be reachable in the IPv6 table.

**Control Plane Policing and protection**

Control Plane Policing (**CoPP**) and Control Plane Protection (**CPPr**) protect the router CPU from excessive or malicious traffic. They can also block legitimate routing or management traffic if misconfigured.

Symptoms of CoPP issues:

- Routing neighbors flap while data forwarding seems normal.
- SSH, SNMP, or ICMP to the device fails intermittently.
- CPU stays protected, but control traffic is dropped.
- Only certain protocols fail, such as OSPF, BGP, or NTP.

Useful checks:

```bash
show policy-map control-plane
show policy-map interface control-plane
show access-lists
show logging | include DROP|POLICY|COPP
show processes cpu sorted
```

Practical interpretation:

- If the data plane forwards user traffic but the device cannot form neighbors, suspect CoPP or ACLs on the control path.
- If management traffic fails but routing is stable, check the management VRF, ACLs, and control-plane policy separately.

Example CoPP concept:

```cisco
class-map match-any CONTROL-PLANE-ROUTING
 match protocol ospf
 match protocol bgp
 match protocol eigrp
!
policy-map CONTROL-PLANE-POLICY
 class CONTROL-PLANE-ROUTING
  police 1000000 conform-action transmit exceed-action drop
 class class-default
  police 500000 conform-action transmit exceed-action drop
!
control-plane
 service-policy input CONTROL-PLANE-POLICY
```

**Management-plane troubleshooting**

Management-plane problems affect your ability to access the device, not necessarily the device’s ability to forward traffic.

Check:

- Interface and VRF reachability
- SSH configuration
- AAA authentication and authorization
- ACLs on VTY lines or interfaces
- SNMP community or user configuration
- NTP and DNS if name resolution is involved

Useful commands:

```bash
show run | section line vty
show run | include transport input|login local|access-class
show ip ssh
show aaa servers
show users
show access-lists
```

Example SSH hardening and access control:

```cisco
ip access-list standard MGMT-SSH
 permit 10.10.50.0 0.0.0.255
!
line vty 0 4
 login local
 transport input ssh
 access-class MGMT-SSH in
```

If SSH fails but ping works, check whether the VTY ACL, AAA, or CoPP is blocking TCP/22.

**Data-plane troubleshooting by symptom**

Use the symptom to guide the next check.

- **Ping to the next hop fails**
  - Check interface state, ARP/ND, VLAN, ACLs, and local addressing.
- **Ping to remote network fails but next hop works**
  - Check routing, recursive resolution, return path, and policy.
- **One-way traffic**
  - Check asymmetric routing, ACLs, NAT, firewall policy, or HSRP/VRRP gateway mismatch.
- **Intermittent loss**
  - Check interface errors, duplex mismatch, congestion, QoS drops, or flapping adjacencies.
- **Only large packets fail**
  - Check MTU, fragmentation, tunnel overhead, and PMTUD behavior.

Useful commands:

```bash
ping 10.1.1.2 source 10.1.1.1
traceroute 8.8.8.8 source 10.1.1.1
show interfaces gi1/0/1
show interfaces gi1/0/1 counters errors
show ip cef exact-route 10.1.1.10 8.8.8.8
```

**Encapsulation, tunneling, and MTU**

Encapsulation issues are common in advanced infrastructure troubleshooting because they affect both control and data planes.

Watch for:

- **MTU mismatch**
  - Packets may be dropped or fragmented unexpectedly.
- **GRE overhead**
  - Tunnel packets are larger than the original payload.
- **IPsec encryption**
  - Adds overhead and may require MSS adjustment.
- **MPLS label stack**
  - Can change effective MTU and forwarding behavior.
- **Tunnel source or destination mismatch**
  - Tunnel stays down or one-way.

Useful commands:

```bash
show interface tunnel 0
show ip interface tunnel 0
show crypto ipsec sa
show crypto session
show mpls interfaces
show mpls forwarding-table
```

Example GRE tunnel:

```cisco
interface Tunnel0
 ip address 172.16.0.1 255.255.255.252
 tunnel source GigabitEthernet0/0
 tunnel destination 203.0.113.2
 tunnel mode gre ip
 ip mtu 1400
 ip tcp adjust-mss 1360
```

If a tunnel works for small packets but fails for large ones, suspect MTU or fragmentation handling before assuming routing failure.

**Policy-based routing, VRF, and forwarding exceptions**

Some traffic fails because it is intentionally diverted from the normal routing table.

Check for:

- **Policy-Based Routing (PBR)**
  - Traffic is matched and sent to a different next hop.
- **VRF mismatch**
  - The source and destination are in different routing tables.
- **NAT translation issues**
  - Return traffic may not match the expected path.
- **Route leaking or missing import/export**
  - Prefixes are present in one VRF but not another.

Useful commands:

```bash
show ip policy
show route-map
show ip vrf
show vrf
show ip route vrf CUSTOMER
show ip cef vrf CUSTOMER
```

Example PBR:

```cisco
route-map PBR-TO-WAN permit 10
 match ip address 101
 set ip next-hop 192.0.2.254
!
interface GigabitEthernet1/0/10
 ip policy route-map PBR-TO-WAN
```

If traffic bypasses the expected route, verify whether PBR is active on the ingress interface.

**Packet capture and platform tools**

When CLI output is not enough, capture packets at the right point in the path.

Common tools and use cases:

- **Embedded Packet Capture (EPC)**
  - Capture traffic on the router or switch without external hardware.
- **SPAN/ERSPAN**
  - Mirror traffic to an analyzer for deeper inspection.
- **Packet tracer equivalents**
  - Some platforms offer feature-specific simulation or policy tracing.
- **Hardware counters**
  - Useful for drops, punts, and queue behavior.

Example EPC workflow:

```cisco
monitor capture buffer CAPBUF size 2048 max-size 1518 circular
monitor capture point ip cef CAPPOINT GigabitEthernet1/0/1 both
monitor capture point associate CAPPOINT CAPBUF
monitor capture start
monitor capture export flash:cap.pcap
```

Use captures to confirm whether packets arrive, are rewritten, or disappear after a specific hop.

**Reading logs and counters**

Logs and counters often reveal the root cause faster than protocol state alone.

Look for:

- Interface up/down events
- STP topology changes
- Neighbor adjacency resets
- Authentication failures
- MTU or encapsulation errors
- CPU spikes or process restarts
- Queue drops and input errors

Useful commands:

```bash
show logging
show logging | include UPDOWN|SPANTREE|OSPF|BGP|EIGRP|HSRP|VRRP
show interfaces counters errors
show interfaces queue
show platform software status control-processor
```

**Exam-oriented troubleshooting patterns**

Use these patterns to map symptoms to likely causes.

- **Neighbor down, interface up**
  - Check timers, authentication, MTU, network type, and ACLs.
- **Route present, traffic fails**
  - Check CEF, adjacency, PBR, VRF, ACLs, and encapsulation.
- **Device reachable by ping, not by SSH**
  - Check VTY ACLs, AAA, CoPP, and TCP reachability.
- **One switch forwards, another blocks**
  - Check STP root placement, port roles, and VLAN consistency.
- **Gateway failover occurs unexpectedly**
  - Check HSRP/VRRP/GLBP tracking and preemption.

**Quick decision tree**

1. Can you reach the device?
   - If no, check management plane, ACLs, VRF, and CoPP.
2. Is the protocol neighbor or adjacency up?
   - If no, check control-plane settings, timers, authentication, MTU, and interface state.
3. Is the route installed and programmed?
   - If no, check recursion, policy, VRF, and CEF.
4. Does the packet leave and return?
   - If no, check ACLs, NAT, PBR, tunnels, and upstream policy.
5. Does only large or specific traffic fail?
   - If yes, check MTU, fragmentation, QoS, or encapsulation overhead.

**Practical mini-scenarios**

A user cannot reach the default gateway after a switch change:

- Check `show spanning-tree vlan X` for blocked ports.
- Check `show vlan brief` and `show interfaces trunk`.
- Verify the SVI is up/up and the VLAN is active.
- Confirm HSRP/VRRP is active on the correct distribution switch.

A BGP neighbor is established but no routes are installed:

- Check `show ip bgp neighbors` for received prefixes and policy.
- Verify prefix-lists, route-maps, and next-hop reachability.
- Confirm the correct address family is activated.
- Check whether maximum-prefix or filtering is suppressing routes.

An OSPF adjacency is stuck in `EXSTART`:

- Check MTU on both sides.
- Verify area, authentication, and network type.
- Confirm the interface is not passive.
- Check for duplicate router IDs.

A tunnel passes pings but drops application traffic:

- Check MTU and MSS.
- Verify IPsec SA counters if encryption is used.
- Confirm the tunnel source and destination are correct.
- Inspect return path and fragmentation behavior.

**Key takeaways**

- Separate **management**, **control**, and **data** plane symptoms before changing configuration.
- Use CEF and adjacency checks to prove whether forwarding is actually programmed.
- Treat routing protocol issues as state problems first, then policy problems.
- Remember that STP, HSRP/VRRP/GLBP, CoPP, VRF, PBR, and tunnels can all create symptoms that look like simple reachability failures.
- Validate with both control-plane outputs and real packet tests.
