## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Template-based deployment overview**

Template-based deployment uses a reusable configuration model to generate device-specific configs from a common source of truth. You define the parts that stay consistent across devices, isolate the variables that change, and then render and push the result in a controlled workflow.

This approach reduces manual errors, improves consistency, and makes change management easier. It is especially useful when you deploy many similar devices, such as access switches, branch routers, or WAN edge nodes.

- **Template**: reusable configuration skeleton with placeholders for device-specific values.
- **Variables**: values such as hostname, interface IP addresses, VLAN IDs, or routing neighbors.
- **Source of truth**: the authoritative data set that drives the rendered configuration.
- **Render step**: the process of converting template plus variables into a device-ready config.
- **Validation step**: checks that the rendered output is syntactically and operationally safe before deployment.

**Why you use templates**

Templates help you standardize design intent and reduce drift. Instead of editing each device by hand, you generate configs from the same logic every time. That makes audits easier and supports repeatable change windows.

Common benefits include:

- Faster provisioning for new devices.
- Lower risk of inconsistent configuration.
- Easier bulk changes across many nodes.
- Better traceability when paired with version control.
- Cleaner rollback planning because the intended state is stored separately from the device state.

**Typical workflow stages**

A practical template-based workflow usually follows this sequence:

1. Define the desired configuration model.
2. Store variables in a structured format.
3. Render the template into a device-specific config.
4. Validate the rendered output.
5. Deploy the config to the target device.
6. Verify the resulting state.
7. Archive the rendered file and change record.

This workflow can be implemented with automation tools such as Ansible, Python, or a controller platform. The exact tool matters less than the discipline of separating data, logic, and deployment.

**Template design principles**

Good templates are readable, modular, and predictable. You want to avoid hardcoding values that should vary by device, and you want to keep the template logic simple enough to troubleshoot.

- Keep the template focused on one device role or platform family.
- Use variables for values that change per device.
- Prefer explicit defaults for optional settings.
- Avoid embedding operational assumptions that may not hold across all sites.
- Comment the template where logic is non-obvious.

A simple Jinja2-style template example:

```jinja2
hostname {{ hostname }}

interface Loopback0
 ip address {{ loopback_ip }} 255.255.255.255

router ospf {{ ospf_process_id }}
 router-id {{ loopback_ip }}
 network {{ loopback_ip }} 0.0.0.0 area 0
```

In this example, the template stays generic while the inventory or data file supplies the device-specific values.

**Data modeling and variable sources**

The variable source should be structured and easy to validate. YAML and JSON are common choices because they are human-readable and work well with automation tools.

Example YAML data:

```yaml
devices:
  dist-01:
    hostname: DIST-01
    loopback_ip: 10.255.1.1
    ospf_process_id: 10
  dist-02:
    hostname: DIST-02
    loopback_ip: 10.255.1.2
    ospf_process_id: 10
```

Practical guidance:

- Use consistent naming conventions for keys.
- Keep site-wide defaults separate from device-specific overrides.
- Validate data types where possible, especially for IP addresses and VLAN IDs.
- Avoid duplicating the same value in multiple places unless you have a clear override strategy.

**Rendering the template**

Rendering is the step where the automation engine combines the template with the variable data. In many workflows, you render locally on the control node before pushing anything to the network device. That gives you a chance to inspect the output and catch mistakes early.

Example with Jinja2 rendering in Python:

```python
from jinja2 import Template

template_text = """
hostname {{ hostname }}
interface Loopback0
 ip address {{ loopback_ip }} 255.255.255.255
"""

data = {
    "hostname": "DIST-01",
    "loopback_ip": "10.255.1.1"
}

rendered = Template(template_text).render(**data)
print(rendered)
```

Rendered output:

```text
hostname DIST-01
interface Loopback0
 ip address 10.255.1.1 255.255.255.255
```

Rendering before deployment is useful because it lets you:

- Review the exact config that will be applied.
- Compare intended changes against the current running configuration.
- Store the generated file for audit or rollback reference.
- Run syntax or policy checks before touching the device.

**Validation before deployment**

Validation is critical because a template can be logically correct but still produce an invalid or risky configuration. You should validate both the data and the rendered output.

Common validation checks include:

- Required variables are present.
- IP addresses and masks are valid.
- Interface names match the target platform.
- Routing process IDs and AS numbers are in range.
- The rendered config does not remove critical management access.

Useful examples:

```bash
python -m json.tool device.json
yamllint inventory.yml
```

For network configs, you can also compare the rendered file against a known-good baseline:

```bash
diff -u baseline/DIST-01.cfg rendered/DIST-01.cfg
```

> Do not assume a rendered config is safe just because the template compiled. Always validate the output against the intended design and the target platform.

**Ansible-based workflow example**

Ansible is commonly used for template-based deployment because it separates inventory, templates, and tasks cleanly. A typical pattern is to render the config with `template`, inspect or archive the result, and then push it with a network module that matches the platform and change type.

Example playbook:

```yaml
---
- name: Render and deploy device configuration
  hosts: edge_switches
  gather_facts: no
  connection: network_cli

  tasks:
    - name: Render device config locally
      ansible.builtin.template:
        src: templates/ios.j2
        dest: rendered/{{ inventory_hostname }}.cfg
      delegate_to: localhost

    - name: Review rendered config
      ansible.builtin.command: >
        sed -n '1,120p' rendered/{{ inventory_hostname }}.cfg
      delegate_to: localhost
      changed_when: false

    - name: Push configuration lines
      cisco.ios.ios_config:
        lines: "{{ lookup('file', 'rendered/' ~ inventory_hostname ~ '.cfg').splitlines() }}"
```

Notes on this workflow:

- `template` renders on the control node, not on the network device.
- `ios_config` is appropriate for line-based configuration changes on IOS/IOS XE.
- For larger changes, you may need module options such as `match`, `replace`, or a different deployment method depending on the platform and desired behavior.
- You should confirm module semantics for the exact collection version and device OS before relying on file-based push behavior.

**Deployment methods and when to use them**

Different platforms and change types require different deployment methods. A template-based workflow does not mean every device accepts the same push mechanism.

| Method | Best use case | Notes |
|---|---|---|
| CLI line push | Small, targeted changes | Good for incremental updates and idempotent tasks |
| Full config replace | Large, controlled rebuilds | Use only when the platform and workflow support it |
| Controller-driven push | Multi-device orchestration | Useful for policy-based or intent-based operations |
| Staged file + commit | Devices with candidate config models | Common in platforms that support commit workflows |

For Cisco IOS and IOS XE, `configure replace` can be useful in some recovery or controlled replacement scenarios, but it is not a universal deployment method across all Cisco platforms. Always verify platform support and operational impact before using it.

**Example: IOS/IOS XE file-based replacement concept**

A common IOS/IOS XE pattern is to save a known-good configuration snapshot and use it as a reference for recovery or controlled replacement.

```bash
copy running-config flash:prechange-DIST-01.cfg
configure replace flash:prechange-DIST-01.cfg force
```

Use this carefully:

- It can help restore a previous configuration state on supported platforms.
- It does not guarantee restoration of external dependencies, dynamic protocol state, or out-of-band changes.
- It should be tested in a lab before production use.
- It is not interchangeable with every Cisco OS or controller workflow.

**Rollback planning**

Rollback should be planned before deployment, not after a failure. In template-based workflows, rollback usually means restoring the previous intended configuration, not simply copying a file back to the device.

Good rollback practices:

- Archive the pre-change running configuration.
- Save the rendered intended config for the change window.
- Record the exact commands or module version used.
- Verify whether the rollback must include interface state, routing adjacencies, or controller policy.

Example snapshot workflow on IOS/IOS XE:

```bash
copy running-config flash:prechange-DIST-01.cfg
```

This snapshot is useful, but it is only one part of rollback planning. It does not automatically restore:

- Neighbor relationships.
- Session state.
- External controller intent.
- Non-configured operational state.

**Verification after deployment**

After pushing the config, verify both the configuration and the operational behavior. A config that looks correct may still fail if the device cannot form adjacencies, reach management services, or apply policy as expected.

Useful verification commands:

```bash
show running-config | section ^hostname|^interface|^router ospf
show ip interface brief
show ip ospf neighbor
show logging | last 20
```

The `section` filter is often safer than a complex `include` expression when you want grouped output. If you use regex filters, test them carefully because CLI parsing can behave differently than expected.

Verification should answer these questions:

- Did the intended lines land on the device?
- Did the device keep management access?
- Did routing and adjacency state remain healthy?
- Did the change introduce warnings or errors in logs?

**Controller-based deployment context**

Template-based deployment is also common in controller-managed environments. In those cases, the controller may own the source of truth and push policy or configuration to devices based on templates and variables.

Cisco DNA Center was renamed Cisco Catalyst Center; the platform name changed, but the workflow concept remains the same: you define intent centrally and deploy it consistently to managed devices.

Typical controller workflow characteristics:

- Centralized templates and variables.
- Device profiles or site-based assignments.
- Validation before deployment.
- Compliance checks after deployment.
- Automated remediation for drift.

This model is useful when you want policy consistency across many sites, but it may abstract away some device-level details. You should understand what the controller changes on your behalf and how to troubleshoot when the generated config differs from your expectations.

**Practical topology example**

A common enterprise topology for template-based deployment includes:

- One or more distribution switches.
- Access switches with repeated baseline settings.
- Branch routers with site-specific WAN parameters.
- A management network for automation access.
- A control node running Ansible or a similar tool.

Example deployment pattern:

- `DIST-01` and `DIST-02` share the same OSPF design.
- Each device gets a unique loopback IP and hostname.
- Access switches share VLAN and STP settings, but differ in uplink port assignments.
- Branch routers share the same security baseline, but differ in WAN peer addresses and site IDs.

This is where templates provide the most value: the design stays consistent while the variables adapt to each node.

**Troubleshooting common issues**

Template-based deployment failures usually fall into a few categories:

- Missing or misspelled variables.
- Wrong data type or invalid value.
- Template logic that generates duplicate or conflicting lines.
- Platform mismatch between template and target device.
- Push method that does not match the intended change scope.

Troubleshooting approach:

1. Render the template locally and inspect the output.
2. Compare rendered output with a known-good config.
3. Validate the target platform syntax.
4. Check module behavior and connection method.
5. Review device logs and command output after deployment.

Example of a quick local render-and-check workflow:

```bash
ansible-playbook deploy.yml --check
ansible-playbook deploy.yml --diff
```

Use `--check` and `--diff` when supported to preview changes before applying them. These options are especially helpful when you want to catch unexpected template output early.

**Best practices for exam and lab scenarios**

- Separate data from logic.
- Render locally before deployment.
- Validate the rendered config, not just the template.
- Match the deployment method to the platform and change size.
- Keep rollback artifacts from the pre-change state.
- Verify operational health after the push, not only config presence.
- Use controller workflows carefully and understand what the controller owns.

**Key takeaway**

Template-based deployment is about repeatability, safety, and scale. You reduce manual configuration by combining a reusable template with structured variables, then validating and deploying the rendered result through a method that fits the platform. The strongest workflows treat rendering, validation, deployment, and rollback as separate steps, which makes changes easier to audit and easier to recover.
