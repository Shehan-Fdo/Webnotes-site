## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Root-cause mindset**

Complex outages are rarely caused by a single obvious failure. You usually need to separate the symptom from the actual fault domain, then prove the failure with device state, packet evidence, and topology context. In CCNP Enterprise troubleshooting, the goal is not just to restore service; it is to identify *where* forwarding breaks and *why* it breaks.

A reliable workflow is:

- Confirm the scope.
- Identify the affected plane: control, data, or management.
- Compare expected behavior with actual device state.
- Test the path hop by hop.
- Validate the fix with the same evidence you used to find the fault.

**A practical troubleshooting model**

Use a layered approach so you do not chase symptoms in the wrong domain.

- **Layer 1/2** issues often show up as link loss, VLAN isolation, MAC flaps, STP instability, or EtherChannel mismatch.
- **Layer 3** issues often show up as missing adjacencies, missing routes, recursive next-hop failures, or policy mistakes.
- **Services** issues often show up as NAT, DHCP, HSRP, multicast, wireless, or WAN overlay problems.
- **Hardware or platform** issues often show up as drops, resource exhaustion, or feature-specific forwarding failures.

A useful mental model is: *if the packet never reaches the next hop, ask whether the control plane built the path and whether the data plane can actually forward it*.

**Triage first: define the outage**

Before you touch configuration, define the failure precisely. The same “network down” report can mean very different things.

- Is the issue isolated to one user, one VLAN, one site, one application, or the whole enterprise?
- Is the failure directional, such as client-to-server working but server-to-client failing?
- Is the problem constant, intermittent, or triggered by failover?
- Did the outage begin after a change, a power event, a link flap, or a software upgrade?

A fast triage checklist helps you avoid false assumptions:

1. Identify the first affected device or segment.
2. Confirm whether the issue is control-plane reachability, data-plane forwarding, or both.
3. Check recent changes, logs, and alarms.
4. Compare healthy and unhealthy paths.
5. Reproduce the failure with a controlled test.

**Evidence collection and verification**

You need evidence from multiple layers. A single `show` command rarely proves root cause by itself.

Common verification categories include:

- **Interface state**: `show interfaces status`, `show interfaces`, `show ip interface brief`
- **Neighbor state**: `show cdp neighbors`, `show lldp neighbors`, `show ip arp`, `show mac address-table`
- **Routing state**: `show ip route`, `show ip ospf neighbor`, `show ip eigrp neighbors`, `show ip bgp summary`
- **Redundancy state**: `show standby brief`, `show vrrp brief`, `show glbp brief`
- **Multicast state**: `show ip mroute`, `show ip pim neighbor`, `show ip pim interface`
- **Wireless state**: controller and AP join status, client association, RF health, and authentication logs
- **Packet evidence**: SPAN, embedded packet capture, or external capture

> A clean interface status does not guarantee end-to-end forwarding. You still need to verify the control plane, the forwarding table, and the path taken by the packet.

**Layer 2 outage root causes**

Layer 2 problems often create broad but confusing symptoms because they affect reachability before routing even begins.

Common root causes include:

- **STP instability**
  - Topology changes can cause temporary loss, MAC table churn, and intermittent flooding.
  - A loop can create broadcast storms, high CPU, and widespread packet loss.
- **EtherChannel misconfiguration**
  - Mismatched mode, speed, duplex, trunk settings, or allowed VLANs can prevent bundle formation or cause one-way forwarding.
- **VLAN pruning or missing VLANs**
  - A VLAN may exist on one switch but not be allowed across the trunk path.
- **Native VLAN mismatch**
  - Untagged traffic may be misclassified, causing unexpected drops or STP inconsistencies.
- **MAC flapping**
  - The same MAC appearing on multiple ports often indicates a loop, miswired topology, or a virtualization issue.
- **Port security or storm-control**
  - Security thresholds can block legitimate traffic after a topology change or device replacement.

Useful commands include:

```bash
show spanning-tree vlan 10
show spanning-tree inconsistentports
show etherchannel summary
show interfaces trunk
show vlan brief
show mac address-table dynamic
show logging | include SPAN|STP|EC|PSEC|MACFLAP
```

Interpretation tips:

- Frequent STP topology changes usually point to unstable links, loops, or misconfigured edge ports.
- `show etherchannel summary` should show all intended member links bundled consistently.
- If a VLAN is missing from `show interfaces trunk`, traffic for that VLAN will not cross the trunk even if the physical link is up.
- MAC flapping often means the network is forwarding the same source frame through more than one path.

**Layer 3 routing root causes**

Routing outages are often caused by adjacency failures, policy mistakes, or recursive next-hop problems. In exam scenarios, do not stop at “the route is missing”; determine *why* it is missing.

Common root causes include:

- **OSPF adjacency failure**
  - Area mismatch, hello/dead timer mismatch, authentication mismatch, MTU mismatch, passive interface, or network type mismatch.
- **EIGRP adjacency failure**
  - AS mismatch, K-value mismatch, authentication mismatch, stub design issues, or interface filtering.
- **BGP adjacency failure**
  - Neighbor reachability, ASN mismatch, update-source issues, TTL/ebgp-multihop requirements, authentication mismatch, or policy rejection.
- **Route redistribution loops**
  - Mutual redistribution without tags or filtering can re-inject routes and create instability.
- **Route filtering or policy mistakes**
  - Prefix lists, route maps, distribute lists, and policy-based routing can block expected routes.
- **Recursive routing failures**
  - A route may exist in the table, but its next hop resolves through another route that is missing or unstable.
- **Administrative distance or metric issues**
  - The “wrong” route may win, or a backup route may never install.

Useful commands include:

```bash
show ip route
show ip route <prefix>
show ip ospf neighbor
show ip ospf interface brief
show ip eigrp neighbors
show ip protocols
show ip bgp summary
show ip bgp
show ip cef <destination>
show ip cef exact-route <source-ip> <destination-ip>
```

Use these checks:

- If the adjacency is down, inspect timers, authentication, MTU, and interface state first.
- If the adjacency is up but the route is missing, inspect filtering, redistribution, and policy.
- If the route exists but traffic fails, inspect recursive resolution, next-hop reachability, and CEF forwarding.

A practical example of a routing fault:

```bash
router ospf 10
 network 10.10.10.0 0.0.0.255 area 0
 passive-interface default
 no passive-interface GigabitEthernet0/0
```

If `GigabitEthernet0/1` should form an OSPF neighbor but remains passive, the adjacency never forms. The fix is not in the neighbor device; it is on the local router where the interface was unintentionally suppressed.

**BGP-specific outage patterns**

BGP failures often look like “the WAN is up but the remote site is unreachable.”

Common indicators:

- `show ip bgp summary` shows the neighbor in `Idle`, `Active`, or `Established` with no useful prefixes.
- The session is established, but inbound or outbound policy filters all routes.
- The next hop is reachable in BGP but not in the IGP, so forwarding fails.
- A route reflector or confederation design issue prevents expected propagation.

Check:

```bash
show ip bgp summary
show ip bgp neighbors <neighbor-ip>
show ip bgp neighbors <neighbor-ip> advertised-routes
show ip bgp neighbors <neighbor-ip> routes
show ip route <next-hop-ip>
```

Remember:

- A BGP session in `Established` state does not guarantee usable forwarding.
- The next hop must be reachable in the forwarding table.
- Policy errors often affect only one direction, which creates asymmetric reachability.

**Redundancy and first-hop gateway failures**

Redundancy protocols can fail in ways that look like random client outages, especially during failover.

Common root causes include:

- **HSRP/VRRP/GLBP state mismatch**
  - The active router may not own the virtual IP, or preemption may not behave as expected.
- **Tracking failures**
  - Interface tracking or object tracking can force a router to relinquish active status.
- **ARP convergence delay**
  - After failover, clients may temporarily use stale ARP entries until the new active gateway is learned.
- **Upstream reachability loss**
  - The gateway may still be active, but its upstream path is broken.

Useful commands include:

```bash
show standby brief
show standby
show vrrp brief
show glbp brief
show ip arp
show track
show logging | include HSRP|VRRP|GLBP|TRACK
```

Practical interpretation:

- If the virtual gateway is active but upstream routing is broken, clients can reach the gateway and still fail beyond it.
- If failover occurs but clients do not recover, inspect ARP refresh, gratuitous ARP behavior, and upstream path health.
- The virtual MAC is deterministic; the issue is usually stale client state or a failed transition, not an “inconsistent virtual MAC.”

**Multicast outage root causes**

Multicast troubleshooting requires you to verify both control-plane membership and data-plane forwarding. Neighbor formation alone is not enough.

Common root causes include:

- **PIM neighbor or interface issues**
  - PIM may not be enabled on the correct interface, or the interface may be in the wrong mode.
- **RPF failure**
  - Reverse Path Forwarding checks fail when the unicast route to the source or RP points to the wrong interface.
- **Missing multicast routing state**
  - The multicast routing table may not contain the expected (*,G) or (S,G) entries.
- **RP or group mapping issues**
  - In sparse-mode designs, the RP must be reachable and correctly mapped for the group range.
- **IGMP snooping or querier problems**
  - Hosts may not join the group correctly, or switches may not learn where receivers exist.

Useful commands include:

```bash
show ip mroute
show ip pim neighbor
show ip pim interface
show ip igmp groups
show ip igmp snooping
show ip rpf <source-ip>
show ip route <source-ip>
show ip route <rp-ip>
```

Troubleshooting guidance:

- If PIM neighbors form but no data flows, check the multicast routing table and RPF state.
- If receivers never appear in the group table, inspect IGMP membership and snooping behavior.
- If the RP is unreachable or incorrectly mapped, multicast joins may fail even though unicast routing looks healthy.

> Do not assume a single command proves multicast health. You need the source route, the RP path when applicable, the RPF result, and the multicast forwarding state.

**WAN, edge, and Internet handoff root causes**

Enterprise outages often originate at the edge, where multiple technologies intersect.

Common root causes include:

- **MPLS handoff issues**
  - Provider circuit problems, wrong VRF attachment, missing routes, or label-switching path issues.
- **Internet edge failures**
  - Default route loss, upstream ACL changes, NAT failure, or ISP reachability problems.
- **IP SLA and object tracking**
  - A tracked object may withdraw a route or shut down a path when the probe fails.
- **NAT misconfiguration**
  - Missing inside/outside roles, ACL mismatch, overload pool exhaustion, or translation failures.
- **SD-WAN underlay vs overlay confusion**
  - The transport may be healthy while the overlay tunnel, control connection, or policy is broken.

Useful commands include:

```bash
show ip route 0.0.0.0
show ip sla statistics
show track
show ip nat translations
show ip nat statistics
show crypto ikev2 sa
show crypto ipsec sa
show sdwan control connections
show sdwan omp peers
```

Practical example:

```bash
ip sla 10
 icmp-echo 8.8.8.8 source-interface GigabitEthernet0/0
 frequency 5
ip sla schedule 10 life forever start-time now

track 10 ip sla 10 reachability

ip route 0.0.0.0 0.0.0.0 203.0.113.1 track 10
```

If the probe fails, the tracked default route is removed. That may be correct behavior, but it can also create an outage if the probe target is blocked by policy or if the source interface is wrong.

**Wireless outage root causes**

Wireless outages often require you to separate RF, controller, authentication, and client policy issues.

Common root causes include:

- **AP join failure**
  - CAPWAP discovery, controller reachability, certificate trust, software mismatch, or licensing issues.
- **RRM issues**
  - Radio Resource Management may choose poor channels or power levels if the RF environment changes rapidly.
- **Client authentication failure**
  - 802.1X, PSK, certificate, or AAA issues can prevent association or IP access.
- **VLAN or policy mismatch**
  - The client may authenticate but land in the wrong VLAN or policy profile.
- **RF interference or coverage gaps**
  - Clients may connect but experience poor throughput, retries, or roaming failures.

Useful checks include controller and AP status, client detail, and AAA logs. Example commands vary by platform, but the workflow is consistent:

```bash
show ap summary
show ap join stats summary
show wireless client summary
show wireless client mac <client-mac>
show logging | include CAPWAP|AAA|DOT1X|RRM
```

Troubleshooting guidance:

- If APs are down, verify controller reachability, time sync, certificates, and software compatibility.
- If APs are up but clients fail, inspect authentication, VLAN assignment, and policy.
- If clients connect but performance is poor, inspect RF health, channel utilization, and roaming behavior.

**Packet-level verification**

Packet evidence is often the fastest way to distinguish control-plane success from data-plane failure.

Use captures when you need to answer questions like:

- Is the packet leaving the source?
- Is the packet arriving at the next hop?
- Is the return traffic coming back?
- Is a device rewriting, dropping, or filtering the packet?

Capture options include:

- SPAN or RSPAN on a switch
- Embedded packet capture on supported platforms
- External capture with a tap or analyzer

What to look for:

- **No outbound packet**: the source host, application, or local stack may be the issue.
- **Outbound packet but no return**: upstream routing, ACL, NAT, or remote service may be the issue.
- **Packet arrives but is not forwarded**: local policy, FIB/CEF, ACL, or hardware forwarding may be the issue.
- **Asymmetric return path**: stateful devices may drop traffic if the return path bypasses them.

A simple capture workflow:

1. Capture near the source.
2. Capture near the suspected failure point.
3. Compare headers, TTL, MAC addresses, and sequence numbers.
4. Confirm whether the packet is dropped before or after the control-plane decision.

**Configuration-level troubleshooting example**

A common exam pattern is a misconfiguration that looks like a routing failure but is actually a policy issue.

Example: a route map blocks redistribution unintentionally.

```bash
route-map REDIST-OSPF deny 10
 match ip address prefix-list BLOCKED

route-map REDIST-OSPF permit 20

ip prefix-list BLOCKED seq 5 permit 10.10.50.0/24

router eigrp 100
 redistribute ospf 10 route-map REDIST-OSPF
```

If `10.10.50.0/24` must be redistributed but appears missing downstream, the route map is the root cause. The fix is to adjust the prefix list or route-map sequence so the intended prefix is permitted.

Another example is a trunk mismatch:

```bash
interface GigabitEthernet1/0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20
```

If VLAN 30 users lose connectivity after a change, the trunk may be up while VLAN 30 is silently excluded. The physical link is healthy, but the data plane for that VLAN is blocked.

**Decision matrix for fast root-cause isolation**

Use symptom patterns to narrow the likely fault domain.

| Symptom | Likely domain | Common root causes | First checks |
|---|---|---|---|
| One VLAN fails, others work | Layer 2 | VLAN pruning, trunk mismatch, STP, MAC flaps | `show interfaces trunk`, `show spanning-tree`, `show mac address-table` |
| Neighbor down, link up | Layer 3 | OSPF/EIGRP/BGP mismatch, auth, timers, passive interface | Protocol neighbor commands, interface config |
| Route present, traffic fails | Data plane | Recursive next hop, ACL, NAT, CEF issue | `show ip route`, `show ip cef`, packet capture |
| Failover causes brief outage | Redundancy | HSRP/VRRP tracking, ARP refresh, upstream path loss | `show standby brief`, `show track`, logs |
| Multicast joins but no stream | Multicast | RPF failure, RP mapping, missing mroute | `show ip mroute`, `show ip rpf`, `show ip pim neighbor` |
| APs join but clients fail | Wireless | AAA, VLAN, policy, RF | Client detail, AAA logs, AP status |
| Internet down, LAN fine | WAN/edge | Default route, NAT, ISP, SLA tracking | `show ip route`, `show ip nat statistics`, `show ip sla statistics` |

**Exam-oriented troubleshooting sequence**

When you face an outage scenario, follow a repeatable sequence.

1. Confirm the symptom and scope.
2. Identify the affected technology domain.
3. Check interface and neighbor state.
4. Verify the relevant control-plane table.
5. Verify the forwarding table or hardware path.
6. Use packet evidence if the failure remains ambiguous.
7. Correct the smallest plausible misconfiguration.
8. Re-test and document the result.

This sequence helps you avoid a common exam mistake: changing multiple things before you know which one caused the outage.

**What good root-cause analysis looks like**

A strong answer in an exam or real incident report should include:

- The exact failure domain.
- The evidence that proves the failure.
- The configuration or state that caused it.
- The fix applied.
- The validation that confirms service restoration.

For example:

- “OSPF adjacency failed because the interface was passive.”
- “VLAN 30 traffic failed because the trunk allowed list omitted VLAN 30.”
- “BGP routes were learned but not installed because the next hop was unreachable.”
- “Clients lost connectivity after HSRP failover because the tracked uplink forced the active role to move and the upstream path was down.”

**Key takeaways**

- Start with scope, then isolate the failure domain.
- Do not trust link-up status alone; verify control plane, forwarding, and packet flow.
- Use protocol-specific commands to prove adjacency, route installation, and forwarding behavior.
- Treat Layer 2, Layer 3, redundancy, multicast, WAN, and wireless as distinct fault domains.
- Validate the fix with the same evidence you used to find the problem.
