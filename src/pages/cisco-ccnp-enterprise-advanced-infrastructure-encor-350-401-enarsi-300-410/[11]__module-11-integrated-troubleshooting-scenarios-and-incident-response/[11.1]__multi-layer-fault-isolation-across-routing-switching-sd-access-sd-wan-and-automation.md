## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Fault isolation mindset**

Multi-layer troubleshooting starts with a simple rule: isolate the failure domain before you change anything. You verify the symptom, identify the control plane or data plane involved, and then move outward from the endpoint toward the core, or from the service edge back to the controller.

A good workflow is:

- Confirm the exact failure mode.
- Determine whether the issue is local, adjacent, or end-to-end.
- Check underlay reachability before overlay or policy.
- Validate control-plane state before assuming a forwarding problem.
- Compare the failing device or site against a known-good reference.

Use a layered approach because many enterprise outages are not caused by a single protocol. A host may fail because of VLAN mismatch, an SVI issue, a missing route, a broken tunnel, a policy denial, or an automation deployment error.

**Troubleshooting workflow and evidence collection**

Start with the simplest observable facts: interface status, IP addressing, ARP/neighbor resolution, routing table entries, and logs. Then move to protocol-specific verification. This reduces false positives and prevents you from chasing symptoms in the wrong layer.

Useful baseline commands include:

```bash
show clock
show logging
show ip interface brief
show ipv6 interface brief
show interfaces status
show interfaces counters errors
show arp
show ipv6 neighbors
show ip route
show ipv6 route
```

For structured troubleshooting, collect:

- **Interface state**: up/down, line protocol, errors, speed/duplex, MTU.
- **L2 adjacency**: VLAN membership, trunk allowed VLANs, STP state, MAC learning.
- **L3 reachability**: ARP, ND, routing table, next-hop resolution.
- **Control-plane health**: routing neighbors, controller sessions, fabric roles.
- **Policy and automation**: ACLs, QoS, segmentation, templates, API responses.

> If the data plane fails but the control plane looks healthy, verify forwarding tables, encapsulation, and policy enforcement before changing routing.

**Routing fault isolation**

Routing issues often appear as “no reachability,” but the root cause may be missing adjacency, incorrect redistribution, route filtering, next-hop recursion failure, or an administrative distance problem. Always validate the route source, the next hop, and the interface used to reach that next hop.

Common verification commands:

```bash
show ip route
show ip route <prefix>
show ip cef <prefix>
show ip protocols
show ip ospf neighbor
show ip ospf interface brief
show ip bgp summary
show ip bgp neighbors
show ip bgp <prefix>
show ipv6 route
show ipv6 neighbors
show ipv6 ospf neighbor
show eigrp neighbors
```

Key routing checks:

- **Connected reachability**: confirm the interface has the correct IP and is in the expected VRF if used.
- **Neighbor formation**: verify hello parameters, authentication, timers, area, AS number, and passive-interface settings.
- **Route installation**: confirm the route exists in the RIB and is not suppressed by filtering or a better route.
- **Next-hop resolution**: ensure the next hop is reachable in the same table or VRF.
- **Redistribution**: check route tags, metrics, and route maps to avoid loops or missing prefixes.

For OSPF, a healthy adjacency is often `FULL` on broadcast and point-to-point links, but not every topology behaves the same. On NBMA or special adjacency relationships, the expected state can differ. If the neighbor is stuck in `EXSTART`, `EXCHANGE`, or `LOADING`, inspect MTU, network type, authentication, and duplicate router IDs.

Example OSPF checks:

```bash
show ip ospf neighbor
show ip ospf interface g0/0
show ip ospf database
show ip ospf route
```

For BGP, focus on session state, received prefixes, policy, and next-hop reachability. A session in `Established` does not guarantee usable routes if inbound or outbound policy blocks them.

Example BGP checks:

```bash
show ip bgp summary
show ip bgp neighbors 192.0.2.2
show ip bgp neighbors 192.0.2.2 advertised-routes
show ip bgp neighbors 192.0.2.2 received-routes
show ip bgp 203.0.113.0
```

IPv6 troubleshooting deserves the same discipline as IPv4. Verify link-local neighbors, global addressing, router advertisements, and route installation. A common failure is a correct IPv6 address with missing neighbor discovery or a wrong default route.

Example IPv6 checks:

```bash
show ipv6 interface brief
show ipv6 neighbors
show ipv6 route
show ipv6 ospf neighbor
ping ipv6 2001:db8:1::1
traceroute ipv6 2001:db8:1::1
```

Static route issues are often overlooked. Validate the next hop, exit interface, and recursive resolution. If the next hop is unreachable, the static route may disappear from the routing table.

Example static route verification:

```bash
show ip route static
show running-config | include ^ip route
show ip cef 198.51.100.0
```

A practical static route example:

```bash
ip route 198.51.100.0 255.255.255.0 192.0.2.1
ip route vrf CORP 10.10.10.0 255.255.255.0 GigabitEthernet0/0 192.0.2.1
```

**First-hop redundancy troubleshooting**

First-hop redundancy protocols can create “intermittent gateway” symptoms that look like routing failures. Validate virtual IP ownership, active/standby state, tracking, and preemption behavior.

Check the protocol in use:

```bash
show standby brief
show vrrp brief
show glbp brief
```

What to verify:

- **HSRP**: active and standby roles, virtual IP, priority, preemption, interface tracking.
- **VRRP**: master election, virtual router ID, priority, and advertisement interval.
- **GLBP**: active virtual gateway, virtual forwarders, load-sharing behavior.

Common issues include:

- Duplicate virtual IP configuration.
- Tracking decrement causing unexpected failover.
- Preemption disabled when it should be enabled.
- ACLs or multicast filtering blocking hello packets.

**Switching fault isolation**

Layer 2 problems often manifest as VLAN reachability failures, asymmetric connectivity, or gateway loss. Start with port state, VLAN membership, trunking, MAC learning, and spanning tree.

Useful commands:

```bash
show interfaces status
show interfaces switchport
show vlan brief
show interfaces trunk
show mac address-table dynamic
show spanning-tree
show spanning-tree vlan 10
show spanning-tree root
```

Key Layer 2 checks:

- **Access ports**: confirm the correct access VLAN and that the port is not err-disabled.
- **Trunks**: verify encapsulation, native VLAN, allowed VLAN list, and pruning.
- **MAC learning**: confirm the switch learns the endpoint MAC on the expected port.
- **STP**: verify the root bridge, port roles, and blocking state.

STP root election matters because the wrong root can create suboptimal paths or loops. If a VLAN is unstable, check whether the intended distribution switch is actually the root and whether port priorities or bridge priorities are set correctly.

Example STP tuning:

```bash
spanning-tree vlan 10 root primary
spanning-tree vlan 20 priority 4096
```

Port-security can silently block traffic after a MAC move or a device replacement. Check for violation counters and err-disabled ports.

```bash
show port-security interface g1/0/10
show interfaces status err-disabled
show logging | include SECURITY|PSECURE|err-disable
```

Example port-security configuration:

```bash
interface GigabitEthernet1/0/10
 switchport mode access
 switchport access vlan 10
 switchport port-security
 switchport port-security maximum 2
 switchport port-security violation restrict
 switchport port-security mac-address sticky
```

DHCP snooping and Dynamic ARP Inspection (DAI) can also block traffic if trust boundaries are wrong. Verify trusted uplinks and binding table state.

```bash
show ip dhcp snooping
show ip dhcp snooping binding
show ip arp inspection
show ip arp inspection statistics
```

Common SVI and gateway issues:

- The SVI is down because no active ports exist in the VLAN.
- The SVI has the wrong IP or mask.
- The VLAN is missing on the trunk path.
- The default gateway on the host points to the wrong virtual IP or SVI.
- ACLs on the SVI block traffic unexpectedly.

Example SVI validation:

```bash
show ip interface brief
show interfaces vlan 10
show running-config interface vlan 10
show ip route connected
```

**SD-Access fault isolation**

SD-Access troubleshooting requires you to separate underlay, overlay, fabric role, endpoint onboarding, and policy enforcement. A fabric can have healthy transport but still fail endpoint access because of authentication, registration, or segmentation issues.

Start by checking fabric node roles and health:

```bash
show fabric summary
show fabric control-plane
show fabric border
show fabric edge
show lisp session
show lisp eid-table
show lisp map-cache
show lisp database
```

Useful indicators:

- **Fabric edge**: endpoint attachment, VLAN-to-VN mapping, and host onboarding.
- **Control-plane node**: LISP registration and mapping services.
- **Border node**: external routing exchange and transit to non-fabric networks.
- **Underlay**: IP reachability between fabric nodes, MTU, and routing adjacencies.

Endpoint onboarding failures often come from 802.1X, MAB (MAC Authentication Bypass), DHCP, or IP address assignment problems. Check whether the endpoint is authenticated, assigned to the correct virtual network, and mapped to the expected SGT (Security Group Tag).

Example checks:

```bash
show authentication sessions interface gigabitEthernet1/0/10 details
show access-session interface gigabitEthernet1/0/10 details
show cts role-based permissions
show cts role-based sgt-map all
show ip device tracking all
```

If an endpoint can authenticate but cannot reach other segments, verify policy and segmentation:

- The SGT is assigned correctly.
- The policy matrix allows the traffic.
- The endpoint is in the correct virtual network.
- The border node has the needed route-leak or external connectivity.

A practical fabric troubleshooting sequence is:

1. Verify the endpoint is physically connected and the access port is up.
2. Confirm authentication and IP assignment.
3. Check LISP registration and map-cache entries.
4. Validate the fabric edge and control-plane node health.
5. Confirm policy permits the traffic path.
6. Test reachability to the border and beyond.

If a host is reachable inside the fabric but not outside, inspect border-node routing, route leaking, and external handoff. If a host is unreachable only from certain segments, suspect policy or segmentation rather than transport.

**SD-WAN fault isolation**

SD-WAN troubleshooting spans transport, control connections, certificate trust, template deployment, and policy. A device may have WAN links up but still fail to join the fabric if certificates, system IP, or controller reachability are wrong.

Core verification commands include:

```bash
show control local-properties
show control connections
show omp peers
show omp routes
show sdwan bfd sessions
show sdwan interface
show sdwan policy from-vsmart
show certificate status
show crypto pki certificates
```

`show control connections` confirms control connections to SD-WAN controllers, but it does not by itself prove that every management-plane or data-plane function is healthy. Use it with OMP, BFD, and certificate checks.

What to validate:

- **Transport reachability**: WAN interface status, default route, NAT, and underlay connectivity.
- **Certificate onboarding**: valid device certificate, correct organization name, and trust chain.
- **Control connections**: vBond, vSmart, and vManage reachability.
- **OMP**: Overlay Management Protocol route exchange and policy distribution.
- **BFD**: Bidirectional Forwarding Detection sessions for data-plane liveliness.

A common onboarding workflow is:

1. Confirm the system IP, site ID, and organization name.
2. Verify the WAN transport can reach the controllers.
3. Check certificate installation and validity.
4. Confirm control connections form.
5. Verify OMP peers and route exchange.
6. Validate data-plane BFD sessions.

Example SD-WAN status interpretation:

- `control connection down`: often transport, certificate, DNS, or reachability related.
- `OMP down`: control connection may exist, but overlay route exchange is not established.
- `BFD down`: data-plane path or policy issue, even if control connections are up.
- `site-id mismatch`: routes or policies may not behave as expected across sites.

Template and policy deployment issues are common in production. If a device does not receive the intended configuration, verify:

- The device is attached to the correct template.
- Variables are populated correctly.
- The template is attached to the right device group.
- The policy is activated and pushed successfully.
- The device has no validation errors in vManage.

Example of a typical device-side check:

```bash
show running-config | section sdwan
show sdwan running-config
show sdwan policy
```

Certificate and trust problems often show up as failed control connections or repeated reboots of the control session. Check certificate expiration, clock synchronization, and trust anchor status.

**Automation and API fault isolation**

Automation failures usually fall into one of four categories: authentication, authorization, payload structure, or device-side validation. Treat API troubleshooting like any other network problem: verify connectivity, then credentials, then request format, then response.

Common interfaces include NETCONF, RESTCONF, and SSH-based automation. NETCONF uses XML over SSH, while RESTCONF uses HTTP methods and JSON or XML payloads.

Useful checks:

```bash
show netconf-yang sessions
show running-config | include netconf
show ip ssh
show logging | include NETCONF|RESTCONF|YANG
```

Example NETCONF enablement:

```bash
conf t
 netconf-yang
 ip ssh version 2
end
```

Example RESTCONF enablement:

```bash
conf t
 restconf
 ip http secure-server
 ip http authentication local
end
```

A valid RESTCONF request must match the YANG model structure. For `ietf-interfaces`, the interface list is typically under `ietf-interfaces:interface`.

Example RESTCONF payload:

```json
{
  "ietf-interfaces:interface": [
    {
      "name": "GigabitEthernet1",
      "description": "Uplink to core",
      "enabled": true
    }
  ]
}
```

Example `curl` request:

```bash
curl -k -u admin:password \
  -H "Content-Type: application/yang-data+json" \
  -H "Accept: application/yang-data+json" \
  -X PATCH \
  https://192.0.2.10/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1 \
  -d '{
    "ietf-interfaces:description": "Uplink to core",
    "ietf-interfaces:enabled": true
  }'
```

When an API call fails, inspect the HTTP status code:

- `200 OK` or `204 No Content`: request succeeded.
- `400 Bad Request`: malformed payload or invalid path.
- `401 Unauthorized`: bad credentials or token.
- `403 Forbidden`: authenticated but not permitted.
- `404 Not Found`: wrong URI or model path.
- `409 Conflict`: resource state conflict.
- `415 Unsupported Media Type`: wrong content type.
- `500 Internal Server Error`: device-side processing issue.

Practical troubleshooting steps:

1. Verify the device supports the model and feature.
2. Confirm the transport path and TLS/SSH trust.
3. Validate credentials and authorization.
4. Compare the request path to the YANG model.
5. Check device logs and the returned error body.
6. Confirm the configuration actually appears in the running state.

Example Python `requests` error handling:

```python
import requests

url = "https://192.0.2.10/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1"
headers = {
    "Content-Type": "application/yang-data+json",
    "Accept": "application/yang-data+json"
}

try:
    r = requests.patch(url, auth=("admin", "password"), headers=headers, json={
        "ietf-interfaces:description": "Uplink to core",
        "ietf-interfaces:enabled": True
    }, verify=False, timeout=10)

    print(r.status_code)
    print(r.text)
except requests.exceptions.Timeout:
    print("Request timed out")
except requests.exceptions.ConnectionError:
    print("Connection failed")
```

**Cross-layer symptom mapping**

Some symptoms point to a likely layer, but you should still verify adjacent layers before changing anything.

| Symptom | Likely area | What to check first |
|---|---|---|
| Host cannot reach default gateway | Switching or FHRP | VLAN, SVI, trunk, HSRP/VRRP/GLBP |
| Gateway reachable but remote subnet fails | Routing | Route table, next hop, redistribution |
| One VLAN fails, others work | Switching | Trunk allowed VLANs, STP, SVI state |
| Fabric host cannot reach external network | SD-Access | Border node, route leak, policy |
| SD-WAN site has control but no data traffic | SD-WAN | BFD, policy, transport, NAT |
| Automation push fails | Automation | Auth, payload, URI, HTTP status |

**Practical exam-style scenarios**

A host in VLAN 20 cannot reach the internet, but VLAN 10 works. Check whether VLAN 20 is allowed on the trunk, whether the SVI is up, and whether the default route exists on the gateway.

A BGP-learned prefix is present in `show ip bgp` but not in `show ip route`. Check administrative distance, next-hop reachability, route-map filtering, and whether the prefix is suppressed by policy.

An SD-Access endpoint authenticates successfully but cannot reach another subnet. Check the SGT assignment, policy matrix, LISP registration, and border-node route exchange.

An SD-WAN branch shows `control connection up` but no application traffic. Check OMP routes, BFD sessions, transport reachability, and whether the policy is steering traffic to the correct tunnel.

A RESTCONF automation job returns `400 Bad Request`. Compare the payload to the YANG model, verify the URI path, and confirm the content type header matches the API.

**High-value troubleshooting habits**

- Build a baseline from a known-good device or site.
- Change one variable at a time.
- Save command output before and after a change.
- Verify both the control plane and data plane.
- Use logs to confirm whether the device rejected, ignored, or partially applied the change.
- Prefer model-driven checks when the platform supports them.

**Summary**

Multi-layer fault isolation is about narrowing the problem domain quickly and proving each layer with the right command set. Routing, switching, fabric, SD-WAN, and automation each have their own failure signatures, but the same discipline applies: verify the physical and logical path, confirm control-plane health, inspect policy, and validate the final forwarding behavior.
