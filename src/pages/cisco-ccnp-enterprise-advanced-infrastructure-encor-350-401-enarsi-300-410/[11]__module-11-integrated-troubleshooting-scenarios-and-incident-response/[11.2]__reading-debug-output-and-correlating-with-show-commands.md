## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Why this topic matters**

Debug output shows the device’s live decision-making, while `show` commands reveal the current state. In troubleshooting, you use both to answer three questions:

- What is failing?
- Where is it failing?
- Is the failure control plane, data plane, or policy related?

A strong workflow is to start with low-risk verification commands, then enable targeted debugging only when you need packet-by-packet or event-by-event detail.

**How to read Cisco debug output**

Cisco debug lines are not just text; they are structured clues. Even when formatting varies across IOS, IOS XE, and platform-specific releases, you can usually extract the same core fields.

- **Process or feature name** identifies the subsystem generating the message, such as OSPF, BGP, EIGRP, DHCP, STP, NAT, or ACL.
- **Message code or event label** tells you what happened, such as adjacency change, packet received, state transition, or translation created.
- **Interface, neighbor, or address** narrows the scope.
- **Reason text** often explains the trigger, but it may be abbreviated or platform dependent.
- **Timestamps** help you correlate events across multiple devices.
- **Sequence and flags** may appear in packet-oriented debugs and can indicate packet type, direction, or state.

> Debug syntax and wording vary by platform. Treat the exact line as illustrative and focus on the event, state, and cause.

Common output patterns include:

- `*Mar 1 00:00:12.123: %OSPF-5-ADJCHG: Process 1, Nbr 10.1.1.2 on GigabitEthernet0/0 from LOADING to FULL, Loading Done`
- `*Mar 1 00:00:12.123: BGP: 10.1.1.2 active open`
- `*Mar 1 00:00:12.123: DHCPD: DHCPDISCOVER received from client`
- `*Mar 1 00:00:12.123: STP: VLAN0001 topology change detected`

Use the message to infer the state machine stage, then confirm with `show` commands.

**Safe debugging workflow**

Debugging can affect CPU and forwarding performance, especially on busy routers, switches, and WAN edges. Use a controlled workflow.

1. Verify the symptom with `show` commands first.
2. Narrow the scope to one protocol, one interface, one neighbor, or one host.
3. Enable only the needed debug.
4. Capture the event.
5. Disable debugging immediately.
6. Correlate the debug with state tables and counters.

Useful safety commands include:

```bash
terminal monitor
show logging
show processes cpu sorted 5sec
show platform hardware qfp active statistics drop
undebug all
no debug all
```

On many platforms, `undebug all` and `no debug all` are equivalent in practice, but use the form your platform supports. On remote sessions, `terminal monitor` is often required to see debug output.

**Interpreting debug output in context**

A debug line rarely proves the root cause by itself. It usually tells you where to look next.

- If a neighbor forms but routes do not appear, check route tables, filters, and redistribution.
- If packets are received but not forwarded, check ACLs, NAT, policy, CEF, or hardware drops.
- If a protocol never reaches adjacency, check timers, authentication, MTU, encapsulation, and addressing.
- If a service works intermittently, compare timestamps with interface flaps, CPU spikes, or control-plane resets.

A practical rule is to correlate each debug event with at least one `show` command from the same layer:

- Neighbor state with adjacency tables
- Route installation with routing tables
- Packet reception with interface counters
- Translation creation with NAT tables
- State changes with logs and timers

**Core correlation model**

Think in layers:

- **Control plane**: protocol neighbors, routing exchanges, STP elections, DHCP handshakes, BGP sessions
- **Data plane**: forwarding, drops, ACL hits, NAT translations, CEF adjacency resolution
- **Management plane**: logging, SSH, SNMP, NTP, AAA

If a debug shows the control plane is healthy but traffic still fails, move to the data plane. If both are healthy, check policy, path selection, or upstream/downstream devices.

**OSPF troubleshooting with debug and show**

OSPF is a common exam topic because its state machine produces useful debug output. The key is to match the adjacency state with the likely failure point.

Typical verification commands:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip ospf interface GigabitEthernet0/0
show ip route ospf
show ip protocols
show ip ospf database
```

Useful debug commands:

```bash
debug ip ospf adj
debug ip ospf hello
debug ip ospf packet
```

What to look for:

- **Down to Init/2-Way** often points to hello mismatch, area mismatch, authentication mismatch, passive interface, or one-way reachability.
- **Stuck in ExStart** commonly points to MTU mismatch or database description negotiation problems.
- **Stuck in Exchange or Loading** often indicates database sync issues, missing LSAs, or packet loss.
- **Full but no routes** suggests filtering, summarization, redistribution, or route preference issues.

> `ExStart` is the classic place to suspect MTU mismatch first. Duplicate router ID is more likely to prevent stable adjacency or cause neighbor confusion, not to be the primary ExStart explanation.

Example correlation:

```bash
R1# show ip ospf neighbor
Neighbor ID     Pri   State           Dead Time   Address         Interface
10.1.1.2          1   EXSTART/DR      00:00:33    10.0.12.2       GigabitEthernet0/0

R1# show ip ospf interface g0/0 | include MTU|State|Network Type
  Internet Address 10.0.12.1/24, Area 0
  MTU 1500, Cost 10, State DR, Network Type BROADCAST
```

If the neighbor is stuck in ExStart and the MTU values differ, align the interface MTU or use `ip ospf mtu-ignore` only when appropriate and understood.

**BGP troubleshooting with debug and show**

BGP is stateful and policy-heavy, so you must distinguish session establishment from route exchange.

Typical verification commands:

```bash
show ip bgp summary
show ip bgp neighbors
show ip bgp
show tcp brief all | include 179
show ip route bgp
```

Useful debug commands:

```bash
debug ip bgp
debug ip bgp events
debug ip bgp updates
```

What to look for:

- **Idle, Connect, Active** usually means TCP or reachability issues, wrong neighbor address, ACLs, source interface problems, or TTL/ebgp multihop issues.
- **OpenSent/OpenConfirm** means TCP is up and BGP OPEN messages are being exchanged.
- **Notification 2/2** indicates **Open Message Error / Bad Peer AS**, which usually means the remote AS number does not match the configured neighbor AS.
- **Established but no routes** suggests policy, route-map, prefix-list, next-hop reachability, or address-family activation issues.

Example correlation:

```bash
R1# show ip bgp summary
Neighbor        V    AS MsgRcvd MsgSent   TblVer  InQ OutQ Up/Down  State/PfxRcd
10.1.1.2        4 65002       0       0        0    0    0 00:00:12 Active
```

If the session never leaves Active, check:

- `show ip route 10.1.1.2`
- `show tcp brief all | include 179`
- `show run | section router bgp`
- `show access-lists`
- `show ip interface brief`

If the session reaches Established but no prefixes are received, verify:

- `show ip bgp neighbors 10.1.1.2 advertised-routes`
- `show ip bgp neighbors 10.1.1.2 routes` when supported
- `show route-map`
- `show ip prefix-list`
- `show ip bgp neighbors 10.1.1.2 | include route refresh|soft-reconfiguration`

> `received-routes` is not universally available unless the platform supports route refresh or inbound soft reconfiguration. Do not assume it works on every device.

**EIGRP troubleshooting with debug and show**

EIGRP adjacency and route exchange are often straightforward once you know whether the problem is neighbor formation or route propagation.

Typical verification commands:

```bash
show ip eigrp neighbors
show ip eigrp topology
show ip route eigrp
show ip protocols
show ip eigrp interfaces
```

Useful debug commands:

```bash
debug eigrp packets
debug eigrp fsm
debug eigrp neighbors
```

What to look for:

- **No neighbor adjacency** often points to K-value mismatch, AS mismatch, authentication mismatch, hello/hold timer mismatch, or multicast filtering.
- **Neighbor up but routes missing** usually points to filtering, stub behavior, summarization, passive interfaces, or topology table issues.
- **Stuck in active state** can indicate query propagation problems, stub design issues, or unreachable successors.

Important nuance:

- **K-value mismatch** prevents adjacency formation; it is not a typical explanation for routes disappearing after a stable neighbor relationship already exists.
- If neighbors are up, focus on topology, filters, and route advertisement behavior instead.

Named-mode verification can vary by platform and IOS XE release. Common forms include:

```bash
show eigrp address-family ipv4 unicast neighbors
show eigrp address-family ipv4 unicast topology
show eigrp address-family ipv4 unicast interfaces
```

On some platforms, you may need the process or ASN in the command, such as:

```bash
show eigrp address-family ipv4 unicast 100 neighbors
show eigrp address-family ipv4 unicast 100 topology
```

Use the syntax that matches the device and software train.

**HSRP and first-hop redundancy**

HSRP, VRRP, and GLBP are first-hop redundancy protocols that often fail due to timers, tracking, authentication, or interface reachability.

Typical verification commands:

```bash
show standby brief
show standby
show standby interface g0/0
show track
```

Useful debug commands:

```bash
debug standby events
debug standby packets
```

What to look for:

- **Init or Listen** on both routers may indicate missing hello exchange, multicast filtering, or interface issues.
- **Standby/Active mismatch** may indicate priority, preempt, or tracking behavior.
- **Frequent state changes** often point to interface flaps or tracked object instability.

Representative example:

```bash
R1# show standby brief
Interface   Grp  Pri P State   Active          Standby         Virtual IP
Gi0/0       1    110 P Standby  10.0.12.2       local           10.0.12.254
```

If the active router is not the expected one, check:

- `show standby`
- `show track`
- `show ip interface brief`
- `show logging`
- `show run interface g0/0`

**DHCP server, relay, and client troubleshooting**

DHCP issues often require you to separate server allocation problems from relay-path problems.

Typical verification commands:

```bash
show ip dhcp binding
show ip dhcp pool
show ip dhcp conflict
show ip dhcp server statistics
show run interface g0/0
show ip interface g0/0
```

Useful debug commands:

```bash
debug ip dhcp server events
debug ip dhcp server packet
debug ip dhcp relay events
debug ip dhcp relay packet
```

What to look for:

- **No DISCOVER seen on the server** suggests relay, VLAN, helper-address, ACL, or L3 reachability issues.
- **DISCOVER seen but no OFFER** suggests pool exhaustion, excluded addresses, wrong network statement, or server policy.
- **OFFER sent but no REQUEST/ACK completion** suggests client-side issues, broadcast filtering, or relay return-path problems.
- **Bindings exist but clients still fail** suggests conflicts, wrong default gateway, or option mismatch.

Example correlation:

```bash
R1# show ip dhcp pool
Pool VLAN10 :
 Utilization mark (high/low)    : 100 / 0
 Subnet size (first/next)       : 254 / 0
 Total addresses                : 254
 Leased addresses               : 254
 Excluded addresses             : 2
 Pending event                  : none
```

If the pool is exhausted, confirm with `show ip dhcp binding` and `show ip dhcp conflict`. If the server looks healthy but clients still fail, verify relay configuration:

```bash
interface Vlan10
 ip helper-address 10.1.1.10
```

Also check the client-facing interface and the path to the DHCP server with `show ip route`, `show arp`, and interface counters.

**STP, RSTP, and MST troubleshooting**

Spanning Tree Protocol issues often appear as loops, blocked ports, topology changes, or unexpected root elections.

Typical verification commands:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree mst
show spanning-tree inconsistentports
show interfaces status
show interfaces counters errors
```

Useful debug commands:

```bash
debug spanning-tree events
debug spanning-tree bpdu
```

What to look for:

- **Root bridge changes** indicate priority, path cost, or link changes.
- **Port role changes** help you identify which port is forwarding, alternate, or designated.
- **Inconsistent port states** may indicate BPDU Guard, Root Guard, Loop Guard, or EtherChannel mismatches.
- **Frequent topology changes** often correlate with unstable links, edge-port misconfiguration, or flapping uplinks.

Correlate STP debug with:

- `show spanning-tree detail`
- `show interfaces counters errors`
- `show logging`
- `show etherchannel summary`

If a port is unexpectedly blocking, verify VLAN membership, trunking, native VLAN consistency, and root placement.

**ACL and policy troubleshooting**

ACLs are a common source of “it pings sometimes” or “one direction works” problems. Debug output can show hits, but the exact format varies widely by platform and feature.

Typical verification commands:

```bash
show access-lists
show ip interface g0/0
show run interface g0/0
show policy-map interface
show class-map
```

Useful debug commands:

```bash
debug ip packet detail
debug ip icmp
debug platform packet-trace
```

Use caution:

- `debug ip packet detail` can be very noisy.
- On production devices, prefer targeted ACL counters and interface verification first.
- Exact debug lines are platform dependent and should be treated as illustrative, not canonical.

Example of a plausible ACL hit message:

```bash
*Mar 1 00:00:12.123: IP: s=10.1.1.10, d=203.0.113.10, len 84, access denied by access-group 101
```

Correlate with:

- `show access-lists 101`
- `show ip interface g0/0`
- `show run interface g0/0`
- `show ip cef exact-route 10.1.1.10 203.0.113.10`

If the ACL counters increment but traffic still fails, check direction, wildcard masks, implicit deny, and whether the ACL is applied inbound or outbound.

**NAT troubleshooting**

NAT problems often look like routing failures until you inspect translations and interface roles.

Typical verification commands:

```bash
show ip nat translations
show ip nat statistics
show run | include ip nat
show ip interface brief
show ip route
```

Useful debug commands:

```bash
debug ip nat
debug ip nat detailed
```

What to look for:

- **No translation created** may mean the inside/outside interfaces are wrong, the ACL does not match, or the route to the destination is missing.
- **Translation created but traffic fails** may mean return-path routing, overload pool exhaustion, or upstream filtering.
- **Unexpected translation entries** may indicate stale sessions or asymmetric routing.

Correlate NAT debug with:

- `show access-lists`
- `show ip route`
- `show arp`
- `show ip cef`
- `show ip nat translations verbose` when supported

**NTP and time synchronization**

NTP affects logs, certificates, and event correlation. If timestamps are wrong, debug correlation becomes unreliable.

Typical verification commands:

```bash
show ntp status
show ntp associations
show clock detail
show logging
```

Useful debug commands:

```bash
debug ntp packets
debug ntp events
```

What to look for:

- **Unsynchronized state** may indicate reachability, ACL, source-interface, or server issues.
- **Stratum and reachability** help you judge whether the device is receiving valid time.
- **Clock drift** can make logs appear out of order across devices.

If NTP fails, verify:

- UDP/123 reachability
- Source interface configuration
- ACLs and control-plane policing
- Correct server address and VRF, if used

**IP routing and CEF correlation**

When routing seems correct but traffic still fails, compare the routing table with CEF (Cisco Express Forwarding) and adjacency state.

Typical verification commands:

```bash
show ip route
show ip cef
show ip cef exact-route 10.1.1.10 203.0.113.10
show adjacency
show arp
show interfaces
```

Useful debug commands:

```bash
debug ip routing
debug ip cef
```

What to look for:

- **Route present but no adjacency** may indicate ARP failure, next-hop resolution issues, or interface problems.
- **CEF drop or glean behavior** can point to unresolved next-hop information.
- **Recursive routing** may reveal a bad next-hop chain.

This is especially useful when a protocol adjacency is healthy but traffic still blackholes.

**WAN, PPP, and serial troubleshooting**

WAN links often fail at the encapsulation, authentication, or keepalive layer before routing even starts.

Typical verification commands:

```bash
show interfaces serial 0/0/0
show controllers serial 0/0/0
show ppp all
show ppp multilink
show ip interface brief
show logging
```

Useful debug commands:

```bash
debug ppp negotiation
debug ppp authentication
debug serial interface
```

What to look for:

- **LCP (Link Control Protocol) failures** indicate link negotiation issues.
- **CHAP/PAP authentication failures** indicate credential mismatch.
- **Encapsulation mismatch** can prevent line protocol from coming up.
- **Clocking or physical layer issues** may appear in controller or interface counters.

**LACP and EtherChannel**

EtherChannel problems often show up as one link forwarding while others are suspended or individual links not bundling.

Typical verification commands:

```bash
show etherchannel summary
show interfaces port-channel 1
show lacp neighbor
show lacp internal
```

Useful debug commands:

```bash
debug lacp events
debug lacp packets
```

> `show lacp neighbor` and `show lacp internal` are platform and version dependent. Use them when available, but rely on `show etherchannel summary` as the most broadly useful first check.

What to look for:

- **Suspended members** may indicate speed/duplex mismatch, trunk mismatch, allowed VLAN mismatch, or LACP mode mismatch.
- **Individual links not bundled** often point to inconsistent configuration across members.
- **Port-channel up, member down** may indicate physical or negotiation issues.

**How to correlate debug with show commands**

A reliable troubleshooting sequence is to move from symptom to state to cause.

1. Identify the affected feature and scope.
2. Check the current state with the most relevant `show` command.
3. Enable a targeted debug for the same feature.
4. Reproduce the issue.
5. Match the debug event to the state table.
6. Confirm the root cause with a second `show` command.
7. Remove the debug.

Example workflow for OSPF:

1. `show ip ospf neighbor`
2. `show ip ospf interface g0/0`
3. `debug ip ospf adj`
4. Observe `EXSTART`
5. `show interface g0/0 | include MTU`
6. Fix MTU mismatch
7. `undebug all`

Example workflow for BGP:

1. `show ip bgp summary`
2. `show tcp brief all | include 179`
3. `debug ip bgp events`
4. Observe `OpenSent` or `Notification 2/2`
5. `show run | section router bgp`
6. Correct AS number or neighbor configuration
7. `undebug all`

**Practical end-to-end troubleshooting labs**

**Lab 1: OSPF adjacency stuck in ExStart**

Symptom: Two routers see each other, but the neighbor never reaches Full.

```bash
R1# show ip ospf neighbor
Neighbor ID     Pri   State           Dead Time   Address         Interface
10.1.1.2          1   EXSTART/DR      00:00:31    10.0.12.2       GigabitEthernet0/0
```

Next checks:

```bash
R1# show ip ospf interface g0/0
R1# show interface g0/0 | include MTU
```

Likely root cause: MTU mismatch or DBD negotiation issue.

Fix path:

- Align MTU on both sides.
- Recheck adjacency with `show ip ospf neighbor`.
- Confirm route installation with `show ip route ospf`.

**Lab 2: BGP session stuck in Active**

Symptom: No prefixes are exchanged.

```bash
R1# show ip bgp summary
Neighbor        V    AS MsgRcvd MsgSent   TblVer  InQ OutQ Up/Down  State/PfxRcd
10.1.1.2        4 65002       0       0        0    0    0 00:00:12 Active
```

Next checks:

```bash
R1# show ip route 10.1.1.2
R1# show tcp brief all | include 179
R1# show run | section router bgp
```

Likely root causes:

- No route to neighbor
- Wrong neighbor IP
- ACL blocking TCP/179
- Source interface mismatch
- eBGP multihop or TTL issue

Fix path:

- Restore reachability.
- Confirm TCP session establishment.
- Recheck `show ip bgp summary`.

**Lab 3: DHCP clients fail to obtain leases**

Symptom: Server has a pool, but clients remain APIPA or self-assigned.

```bash
R1# show ip dhcp pool
R1# show ip dhcp server statistics
R1# show ip dhcp binding
R1# show ip dhcp conflict
```

If the server sees no requests, check relay:

```bash
R1# show run interface vlan 10
R1# show ip interface vlan 10
```

Likely root causes:

- Missing `ip helper-address`
- Wrong VLAN or SVI state
- Pool exhaustion
- ACL blocking UDP/67 and UDP/68
- Wrong default gateway or option settings

**Debug-to-show quick reference**

| Feature | Debug clue | Best show commands | Common root cause |
|---|---|---|---|
| OSPF | `EXSTART`, `LOADING`, `FULL` transitions | `show ip ospf neighbor`, `show ip ospf interface`, `show ip route ospf` | MTU mismatch, auth mismatch, packet loss |
| BGP | `Idle`, `Active`, `OpenSent`, `Notification 2/2` | `show ip bgp summary`, `show tcp brief all`, `show ip bgp neighbors` | AS mismatch, reachability, ACL, policy |
| EIGRP | neighbor formation or active queries | `show ip eigrp neighbors`, `show ip eigrp topology` | AS mismatch, auth, timers, stub/query issues |
| HSRP | state changes, hello/hold behavior | `show standby brief`, `show standby`, `show track` | priority, preempt, tracking, interface flaps |
| DHCP | DISCOVER/OFFER/REQUEST/ACK flow | `show ip dhcp binding`, `show ip dhcp pool`, `show ip dhcp server statistics` | relay, pool exhaustion, ACL, conflicts |
| STP | root changes, port role changes | `show spanning-tree`, `show spanning-tree detail` | loops, BPDU issues, root misplacement |
| ACL | deny hits, packet drops | `show access-lists`, `show ip interface`, `show policy-map interface` | direction, implicit deny, wildcard mismatch |
| NAT | translation creation or failure | `show ip nat translations`, `show ip nat statistics` | inside/outside mismatch, ACL, return path |
| PPP/Serial | LCP, CHAP, line protocol events | `show interfaces serial`, `show ppp all`, `show controllers serial` | encapsulation, auth, clocking |
| CEF/Routing | route present but forwarding fails | `show ip route`, `show ip cef`, `show adjacency` | ARP, recursion, hardware drop |

**Exam tips**

- Start with the simplest `show` command that proves the current state.
- Match the debug to the protocol state machine, not just the text.
- Remember that a healthy neighbor does not guarantee healthy forwarding.
- Use `show logging` to correlate events across time, but do not assume every platform exposes the same logging detail.
- If a session never reaches the protocol exchange phase, check transport and reachability first.
- If a session is established but traffic fails, check policy, route installation, and data-plane forwarding.

**Common mistakes to avoid**

- Treating a debug line as proof without confirming with a `show` command.
- Using broad debugs on production devices without a scope.
- Assuming one platform’s debug format matches another.
- Confusing adjacency failure with route filtering.
- Assuming route presence guarantees packet forwarding.
- Overusing `received-routes` or other commands that may require platform support or special conditions.

**Mental model for troubleshooting**

Use this sequence:

- **Observe** the symptom.
- **Verify** the current state.
- **Debug** the smallest relevant scope.
- **Correlate** the event with counters, tables, and logs.
- **Fix** the root cause.
- **Confirm** recovery with a second `show` command.

When you practice this pattern, debug output becomes a fast way to explain behavior, and `show` commands become the evidence that proves the fix.
