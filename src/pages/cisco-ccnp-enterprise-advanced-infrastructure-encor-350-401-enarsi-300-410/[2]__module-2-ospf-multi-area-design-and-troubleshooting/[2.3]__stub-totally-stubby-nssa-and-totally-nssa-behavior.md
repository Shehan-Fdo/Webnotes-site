## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**OSPF area types: why they exist**

Open Shortest Path First (<abbr title="Open Shortest Path First">OSPF</abbr>) area types reduce routing overhead by limiting which <abbr title="Link-State Advertisement">LSA</abbr>s are flooded into an area. The main exam goal is to know which LSAs are blocked, which default routes appear, and how the area type affects external reachability.

These area types are not about changing the OSPF algorithm. They change the information an area accepts from the rest of the domain. In practice, you use them to simplify branch designs, reduce LSDB size, and keep edge routers from carrying full external routing tables.

- **Stub** areas block external LSAs from outside the area.
- **Totally stubby** areas block external LSAs and also suppress most inter-area summaries.
- **NSSA** areas allow redistribution inside the area without accepting normal external LSAs from outside.
- **Totally NSSA** areas behave like NSSA areas with additional summary suppression.

**Stub area behavior**

A stub area does not accept Type 5 external LSAs into the area. That means routes redistributed into OSPF from other protocols are not flooded as external routes inside the stub area. Instead, the Area Border Router (<abbr title="Area Border Router">ABR</abbr>) injects a default route so routers in the stub area can still reach destinations outside the area.

The key point is that stub areas still learn inter-area routes. They receive Type 3 summary LSAs for destinations in other OSPF areas, but they do not receive Type 5 external LSAs. Type 4 LSAs, which describe how to reach an Autonomous System Boundary Router (<abbr title="Autonomous System Boundary Router">ASBR</abbr>), are also not flooded into the stub area because external LSAs are blocked and the default route is used for outside reachability.

- **Allowed inside stub**:
  - Type 1 router LSAs
  - Type 2 network LSAs
  - Type 3 summary LSAs
  - Default route from the ABR
- **Not flooded into stub**:
  - Type 4 ASBR summary LSAs
  - Type 5 external LSAs
  - Type 7 LSAs

A stub area is useful when the area has no need to know the full external topology and can rely on a single exit point.

> A stub area must be configured consistently on all routers in the area, including the ABR. If one router expects stub behavior and another does not, adjacency formation fails.

**Totally stubby area behavior**

A totally stubby area is a Cisco extension that goes further than a normal stub area. It blocks Type 5 external LSAs and also suppresses Type 3 summary LSAs, except for the default route injected by the ABR. This makes the area even smaller from a routing-information perspective.

In a totally stubby area, routers usually see only:
- intra-area routes
- a default route to the ABR

This is why totally stubby areas are common in small branch designs where the branch should send all non-local traffic to one exit and does not need visibility into other OSPF areas.

Cisco IOS and IOS XE commonly use the `no-summary` option on the ABR to create this behavior.

```cisco
router ospf 10
 area 1 stub no-summary
```

On the non-ABR routers in the area, you still configure the area as stub:

```cisco
router ospf 10
 area 1 stub
```

The ABR typically injects a default route for the area, but that depends on correct configuration and platform behavior. Do not assume the default route appears if the ABR is not configured properly.

**NSSA behavior**

A Not-So-Stubby Area (<abbr title="Not-So-Stubby Area">NSSA</abbr>) is designed for areas that need stub-like filtering but also need redistribution from a router inside the area. This is the important distinction: an NSSA allows an internal ASBR to redistribute routes into OSPF without importing normal Type 5 external LSAs from outside the area.

Inside the NSSA, redistributed routes are carried as Type 7 LSAs. The router becomes an ASBR because it performs redistribution, not because the area is NSSA. The NSSA area type simply permits those redistributed routes to exist as Type 7 LSAs.

The ABR translates Type 7 LSAs to Type 5 LSAs when advertising them into the rest of the OSPF domain, unless translation is suppressed by policy or platform-specific behavior. This translation is what makes NSSA useful: the area can originate external information locally while still keeping outside external routes out.

- **Allowed inside NSSA**:
  - Type 1 router LSAs
  - Type 2 network LSAs
  - Type 3 summary LSAs
  - Type 7 LSAs for redistributed routes
- **Not flooded into NSSA**:
  - Type 5 external LSAs from other areas
  - Type 4 LSAs from other areas, because the area blocks normal external flooding
- **Default route**:
  - optional/configured, not automatic by definition

A common exam trap is to think NSSA means “stub plus default route.” That is incomplete. NSSA means “stub-like filtering, but allow redistribution inside the area.”

```cisco
router ospf 10
 area 2 nssa
 redistribute static subnets
```

In this example, the router becomes an ASBR because of redistribution. The redistributed static routes are represented as Type 7 LSAs inside area 2.

**Totally NSSA behavior**

A totally NSSA is the NSSA equivalent of a totally stubby area. It blocks Type 5 external LSAs and also suppresses Type 3 summary LSAs, while still allowing Type 7 LSAs inside the area. The ABR typically injects a default route so the area can reach destinations outside the area.

This design is useful when you want:
- local redistribution inside the area
- minimal routing information inside the area
- a single exit path to the rest of the OSPF domain

Cisco commonly uses the `no-summary` option on the ABR for this behavior.

```cisco
router ospf 10
 area 2 nssa no-summary
```

The ABR still participates in the rest of the OSPF domain normally. Type 4 LSAs may still exist elsewhere in the domain as needed to reach the ASBR that originated the Type 7 external information, but they are not flooded into the totally NSSA area.

**LSA comparison**

| Area type | Type 3 summaries | Type 4 ASBR summaries | Type 5 externals | Type 7 externals | Default route behavior |
|---|---:|---:|---:|---:|---|
| Stub | Yes | No | No | No | ABR injects default route |
| Totally stubby | No, except default | No | No | No | ABR injects default route |
| NSSA | Yes | No into the NSSA | No into the NSSA | Yes | Optional/configured |
| Totally NSSA | No, except default | No into the NSSA | No into the NSSA | Yes | ABR typically injects default route |

Use this table as a memory aid, but remember the nuance: the default route is a configured behavior, not a magical property of the area type itself.

**How default routes work in these areas**

Default route handling is one of the most tested topics.

- In a **stub** area, the ABR injects a default route instead of flooding external routes into the area.
- In a **totally stubby** area, the ABR injects a default route and suppresses other summary routes.
- In an **NSSA**, a default route is not inherent. It appears only if the ABR or another configured mechanism injects it.
- In a **totally NSSA**, the ABR typically injects a default route while suppressing other summaries.

On Cisco platforms, the exact command syntax can vary by software train and feature support. For exam purposes, know the intent:
- stub and totally stubby areas rely on the ABR for a default route
- NSSA areas may or may not have a default route depending on configuration
- totally NSSA areas usually combine NSSA behavior with summary suppression and a default route from the ABR

**Configuration patterns**

A basic stub area example:

```cisco
router ospf 10
 network 10.1.1.0 0.0.0.255 area 1
 network 10.1.2.0 0.0.0.255 area 1
 area 1 stub
```

On the ABR, the same area must also be marked stub:

```cisco
router ospf 10
 area 1 stub
```

A totally stubby area example:

```cisco
router ospf 10
 area 1 stub no-summary
```

An NSSA with redistribution inside the area:

```cisco
router ospf 10
 area 2 nssa
 redistribute static subnets
```

A totally NSSA example:

```cisco
router ospf 10
 area 2 nssa no-summary
 redistribute static subnets
```

If you want a default route in an NSSA, you typically configure it on the ABR or use the platform-supported NSSA default-origination method. The exact command can differ, so verify the syntax on the target platform.

**Verification commands**

Use the OSPF database and routing table to confirm what the area is receiving.

```cisco
show ip ospf
show ip ospf interface brief
show ip ospf database
show ip ospf database summary
show ip ospf database external
show ip ospf database nssa-external
show ip route ospf
```

What to look for:
- In a **stub** or **totally stubby** area, `show ip ospf database external` on a router inside the area should not show Type 5 LSAs in that router’s local LSDB.
- In an **NSSA**, `show ip ospf database nssa-external` should show Type 7 LSAs inside the NSSA.
- On the ABR, you may see Type 5 LSAs in the rest of the domain after translation from Type 7.
- A default route should appear in the routing table if the ABR is injecting one.

Example output clues:
- `O*IA 0.0.0.0/0` indicates an inter-area default route learned via OSPF.
- `O N2` or `O N1` indicates NSSA external routes in the routing table.
- `O E2` or `O E1` indicates external routes after translation outside the NSSA.

**Route types and what they mean**

- **O**: intra-area OSPF route
- **O IA**: inter-area OSPF route
- **O E1**: external Type 5 route with internal cost added
- **O E2**: external Type 5 route with external metric only
- **O N1**: NSSA external Type 7 route translated or retained as NSSA type 1 metric behavior
- **O N2**: NSSA external Type 7 route translated or retained as NSSA type 2 metric behavior

The exact route code shown depends on where you are in the domain and how the route was originated.

**Adjacency and area-type consistency**

OSPF neighbors must agree on the area type. If one router configures an area as stub and another configures the same area as NSSA, adjacency does not form. The same is true if one side expects totally stubby behavior and the other side does not match the required stub setting.

Important nuance:
- area type is configured per area, not as a global router personality
- a router can participate in multiple areas with different area types
- the same area must have the same type on every router in that area

This matters in hub-and-spoke designs where the hub may connect to several areas, each with different filtering requirements.

**Practical design guidance**

Use these area types when you want to reduce LSDB size and routing churn at the edge.

- Choose **stub** when the area only needs internal and inter-area reachability, and you want a default route for everything else.
- Choose **totally stubby** when the area should know only local routes plus a default route.
- Choose **NSSA** when the area needs to redistribute routes locally but should not accept external Type 5 LSAs from elsewhere.
- Choose **totally NSSA** when you want NSSA redistribution plus minimal summary visibility.

A simple branch topology often looks like this:

```text
Core Area 0 ---- ABR ---- Branch Area 1
                     |
                 default route
```

If the branch has a local firewall or static route redistribution, NSSA is often the better fit than stub because the branch can inject its own external information as Type 7 LSAs.

**Common exam traps**

- Thinking stub areas cannot learn inter-area routes. They can learn Type 3 summaries.
- Thinking NSSA automatically means a default route. It does not.
- Thinking the area type makes a router an ASBR. Redistribution makes the router an ASBR.
- Thinking Type 4 LSAs are “needed” inside stub or NSSA areas. They are not flooded into those areas because the area type suppresses the external reachability model inside the area.
- Thinking Type 7 LSAs are a separate route class. They are redistributed routes represented as NSSA LSAs.
- Thinking totally stubby and totally NSSA are standard OSPF area types. They are Cisco extensions.

**Quick memory model**

- **Stub**: no Type 5, no Type 4, yes Type 3, default route from ABR
- **Totally stubby**: no Type 5, no Type 4, no Type 3 except default, default route from ABR
- **NSSA**: no Type 5 from outside, yes Type 7 inside, Type 7 translated to Type 5 outside
- **Totally NSSA**: NSSA plus summary suppression, default route typically from ABR

**Troubleshooting checklist**

1. Verify the area type on every router in the area.
2. Confirm the ABR is configured for the same area type.
3. Check whether the router is actually redistributing routes if you expect Type 7 LSAs.
4. Inspect the local LSDB with `show ip ospf database`.
5. Confirm the default route exists in the routing table.
6. Check whether the ABR is translating Type 7 LSAs to Type 5 LSAs outside the NSSA.
7. Ensure the area type matches the design goal: stub for no redistribution, NSSA for local redistribution.

**Key takeaways**

- Stub and totally stubby areas reduce routing information by blocking external LSAs; totally stubby also suppresses summaries.
- NSSA and totally NSSA allow local redistribution using Type 7 LSAs.
- The ABR is central: it injects defaults, suppresses summaries, and translates Type 7 to Type 5 when needed.
- Area type consistency is mandatory for adjacency.
- Default route behavior is configuration-dependent in NSSA variants and ABR-dependent in stub variants.
