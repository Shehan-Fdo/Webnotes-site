## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Overview**

OSPF summarization reduces routing table size, limits LSA flooding, and improves stability by advertising an aggregate prefix instead of many more-specific routes. In Cisco OSPF, summarization is most commonly done at an **ABR** (Area Border Router) for inter-area routes and at an **ASBR** (Autonomous System Boundary Router) for external routes.

The key exam idea is that summarization happens at the boundary where routes leave a domain:
- **ABR summarization** compresses routes learned from one area before advertising them into another area.
- **ASBR summarization** compresses redistributed external routes before they are injected into OSPF as external LSAs.

Summarization is not the same as filtering. Summarization advertises a less-specific route; filtering suppresses routes entirely.

**Core OSPF roles**

<dl>
<dt><strong>ABR</strong></dt>
<dd>A router with interfaces in area 0 and at least one other OSPF area. It can summarize inter-area routes with area range configuration.</dd>
<dt><strong>ASBR</strong></dt>
<dd>A router that redistributes routes from another routing source into OSPF. It can summarize external redistributed routes with summary-address configuration.</dd>
<dt><strong>NSSA</strong></dt>
<dd>Not-So-Stubby Area. It carries external routes as Type 7 LSAs inside the area, then translates them to Type 5 LSAs at the NSSA ABR unless translation is suppressed or elected differently.</dd>
</dl>

**Why summarization matters**

- It reduces the number of prefixes in the routing table.
- It reduces LSA churn when a component subnet changes.
- It improves convergence by limiting the scope of topology updates.
- It can hide internal topology details from other areas or from the backbone.

**ABR summarization fundamentals**

An ABR summarizes routes between areas using the `area range` command. The ABR must actually have the component routes in its OSPF database and must be an ABR in the real sense: it needs connectivity to area 0 and at least one non-backbone area.

A summary is advertised only if at least one more-specific route exists in the area being summarized. If all component routes disappear, the summary can also disappear.

Typical Cisco IOS / IOS XE syntax:

```bash
router ospf 10
 area 10 range 10.10.0.0 255.255.0.0
```

This tells the ABR to summarize area 10 routes into `10.10.0.0/16` when advertising them into other areas.

**ABR summarization behavior**

- The summary is created at the area boundary, not inside the area.
- The ABR installs a discard route for the summary to prevent loops if the summary is selected.
- The summary mask must cover the component routes you intend to aggregate.
- If the summary is too broad, it can attract traffic for destinations that do not actually exist.

> A summary should be as specific as possible while still covering the intended subnets. Overly broad summaries can create black holes if the ABR does not have a valid path to every address in the aggregate.

**Example ABR topology**

- Area 0: `10.0.0.0/24`
- Area 10: `10.10.1.0/24`, `10.10.2.0/24`, `10.10.3.0/24`
- Area 20: receives a summary instead of three separate routes

On the ABR:

```bash
router ospf 10
 network 10.0.0.0 0.0.0.255 area 0
 network 10.10.1.0 0.0.0.255 area 10
 network 10.10.2.0 0.0.0.255 area 10
 network 10.10.3.0 0.0.0.255 area 10
 area 10 range 10.10.0.0 255.255.0.0
```

Expected result:
- Area 10 still sees its specific intra-area routes.
- Other areas see a single inter-area summary route for `10.10.0.0/16` if the component routes exist and the summary is valid.

**ABR summary route type**

- Inter-area routes are learned as **O IA** in the routing table.
- The summary LSA is a Type 3 summary LSA.
- In the OSPF database, you should expect to see summary information associated with the area boundary.

**ASBR summarization fundamentals**

An ASBR summarizes redistributed external routes using `summary-address`. This applies to routes injected into OSPF from another protocol such as connected, static, EIGRP, or BGP.

Typical Cisco IOS / IOS XE syntax:

```bash
router ospf 10
 redistribute static subnets
 summary-address 172.16.0.0 255.255.252.0
```

This is meaningful only if the redistributed routes fall within `172.16.0.0/22`.

**Important ASBR details**

- The command is configured on the ASBR, not on an ABR.
- It summarizes external routes before they are advertised as external LSAs.
- The summary can reduce the number of Type 5 LSAs generated.
- The ASBR still needs a valid route to the summarized destinations.

**External route types**

- **E1**: external metric plus internal OSPF cost to the ASBR
- **E2**: external metric only, unless tie-breaking requires internal cost comparison

When you summarize external routes, the resulting summary is still an external route advertisement, but the exact route type and metric behavior depend on the configuration and the route source.

**ABR vs ASBR summarization**

| Feature | ABR summarization | ASBR summarization |
|---|---|---|
| Command | `area <area-id> range <prefix> <mask>` | `summary-address <prefix> <mask>` |
| Location | ABR | ASBR |
| Route scope | Inter-area routes | External redistributed routes |
| LSA type affected | Type 3 | Type 5, and Type 7 in NSSA context before translation |
| Typical route code | `O IA` | `O E1` / `O E2` |
| Main goal | Reduce inter-area prefixes | Reduce external prefixes |

**NSSA summarization and translation**

A **NSSA** carries external routes as Type 7 LSAs inside the NSSA. The NSSA ABR translates Type 7 LSAs into Type 5 LSAs when advertising them into the rest of the OSPF domain.

Key points:
- The NSSA ASBR injects external routes as Type 7 LSAs.
- The NSSA ABR performs Type 7 to Type 5 translation.
- Translation behavior can be influenced by translator election and the `no-translate` option on some platforms or designs.
- Summarization in NSSA is design-sensitive because the summary may be applied before or after translation depending on where the command is configured and which router is acting as the translating ABR.

Practical exam takeaway:
- Do not assume a single command behaves identically in every NSSA design.
- Verify whether the router is the NSSA ASBR, the translating ABR, or both.
- Check whether the external route is being summarized as Type 7 inside the NSSA or as Type 5 outside it.

**Default route advertisement and summarization**

Default routes are common in OSPF designs and often appear in exam scenarios alongside summarization.

- An ABR can inject a default route into a stub or totally stubby area.
- An ASBR can originate a default route with redistribution or explicit default-information origination.
- Default route injection is not the same as summarization, but both reduce routing table size.

Common examples:

```bash
router ospf 10
 default-information originate
```

If the default route is not already present in the routing table, use:

```bash
router ospf 10
 default-information originate always
```

For stub areas, the ABR typically injects the default automatically according to area type.

**How OSPF chooses routes**

Within OSPF, route preference is based on OSPF route type and cost, not on administrative distance between two OSPF routes of the same protocol.

Use this mental model:
- Longest prefix match is always first in the routing table.
- If prefixes are equal, the router compares route sources.
- Between OSPF routes, the protocol prefers the best OSPF path based on OSPF rules and cost.
- Administrative distance matters when comparing OSPF against routes from other protocols, or when multiple sources compete for the same prefix.

**Verification commands**

Use these commands to confirm summarization behavior:

```bash
show ip ospf
show ip ospf database summary
show ip ospf database external
show ip route ospf
show ip route <prefix>
```

Useful checks:
- `show ip ospf database summary` to inspect Type 3 summary LSAs.
- `show ip ospf database external` to inspect Type 5 external LSAs.
- `show ip route ospf` to confirm whether the router installs the summary route.
- `show ip route <prefix>` to verify the exact prefix and route code.

On some platforms, additional commands may be available, but the commands above are the most reliable study and troubleshooting tools.

**What to look for in the OSPF database**

When summarization is working, you should expect:
- Fewer individual LSAs for the component subnets.
- A summary LSA with the aggregate prefix.
- The summary LSA advertising the aggregate into the receiving area or domain.
- A discard route or null route behavior on the summarizing router to prevent loops.

For example, if three `/24` routes are summarized into a `/16`, the database should show the `/16` summary instead of all three specifics in the receiving area.

**Troubleshooting checklist**

1. Confirm the router is the correct boundary device.
   - For inter-area summarization, it must be an ABR.
   - For external summarization, it must be an ASBR.

2. Confirm the component routes exist.
   - The summary cannot advertise routes that are not present.
   - Check the routing table and OSPF database on the boundary router.

3. Confirm the summary mask is correct.
   - A mask that is too narrow misses routes.
   - A mask that is too broad can create reachability problems.

4. Confirm the area design supports the summary.
   - The router must have the right area adjacencies.
   - NSSA and stub area behavior can change what is advertised.

5. Confirm translation behavior in NSSA.
   - Check whether the ABR is translating Type 7 LSAs.
   - Verify whether another ABR is elected as translator.

6. Confirm the receiving area sees the expected LSA type.
   - Type 3 for inter-area summaries.
   - Type 5 for external summaries outside NSSA.
   - Type 7 inside NSSA before translation.

**Practical lab: ABR summarization**

Topology:
- R1 in area 0
- R2 as ABR between area 0 and area 10
- R3 and R4 in area 10 advertising `192.168.10.0/24`, `192.168.11.0/24`, `192.168.12.0/24`

R2 configuration:

```bash
router ospf 1
 network 10.0.0.0 0.0.0.255 area 0
 network 10.10.0.0 0.0.0.255 area 10
 area 10 range 192.168.8.0 255.255.248.0
```

Expected result:
- R1 learns a single inter-area route for `192.168.8.0/21` if the component routes fit the aggregate.
- R2 still knows the specifics internally.
- The summary reduces the number of inter-area entries.

**Practical lab: ASBR summarization**

Topology:
- R5 redistributes static routes into OSPF
- Static routes:
  - `172.16.1.0/24`
  - `172.16.2.0/24`
  - `172.16.3.0/24`

R5 configuration:

```bash
ip route 172.16.1.0 255.255.255.0 Null0
ip route 172.16.2.0 255.255.255.0 Null0
ip route 172.16.3.0 255.255.255.0 Null0

router ospf 1
 redistribute static subnets
 summary-address 172.16.0.0 255.255.252.0
```

Expected result:
- Downstream routers see a summarized external route instead of three separate external prefixes.
- The summary should match the redistributed routes exactly enough to avoid leaking unrelated destinations.

**Summarization vs filtering**

These are related but different tools:
- **Summarization** advertises one aggregate route for many specifics.
- **Filtering** prevents selected routes from being advertised or installed.

Use summarization when you want scale and stability.
Use filtering when you need policy control or route suppression.

Do not use summarization as a substitute for route policy. A summary can still attract traffic for addresses that are not actually reachable if the aggregate is too broad.

**Common mistakes**

- Configuring `area range` on a non-ABR and expecting it to work.
- Using `summary-address` on a router that is not redistributing routes into OSPF.
- Summarizing prefixes that do not share a valid aggregate boundary.
- Forgetting that NSSA translation changes how external LSAs appear outside the area.
- Confusing route suppression with route aggregation.
- Verifying only the routing table and not the OSPF database.

**Exam focus summary**

- ABR summarization uses `area <id> range`.
- ASBR summarization uses `summary-address`.
- ABR summaries affect inter-area routes and Type 3 LSAs.
- ASBR summaries affect redistributed external routes and Type 5 LSAs.
- NSSA introduces Type 7 LSAs and translation at the ABR.
- Always verify both the routing table and the OSPF database.

**Quick reference**

