## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**OSPF Area 0 dependency**

Open Shortest Path First (<abbr title="Open Shortest Path First">OSPF</abbr>) uses Area 0, the backbone area, as the transit core for inter-area routing. Every non-backbone area must connect to Area 0 either directly or through a virtual link. If that dependency is broken, inter-area route exchange becomes inconsistent or fails, even if routers inside the isolated area still form neighbor adjacencies.

Area 0 matters because OSPF is a hierarchical link-state protocol. The hierarchy reduces flooding scope, limits SPF (Shortest Path First) recalculation, and gives the backbone a stable place to exchange summary information between areas. In exam scenarios, the key question is not just “is OSPF up?” but “is the area design valid?”

- **Area 0** is the backbone and the required transit domain for inter-area routing.
- **ABR** means Area Border Router; it connects multiple areas and injects summary routes.
- **ASBR** means Autonomous System Boundary Router; it injects external routes into OSPF.
- A broken backbone design often shows up as missing inter-area routes, not as a total OSPF failure.

**Why Area 0 dependency exists**

OSPF floods link-state information within an area, but it does not flood all LSAs everywhere. Area boundaries control what is summarized, what is hidden, and where route information can travel. Area 0 is the common meeting point for inter-area information, so every non-backbone area must have a path to it.

If an area does not connect to Area 0, routers in that area may still learn routes inside the area, but they cannot reliably exchange inter-area routes with the rest of the OSPF domain. Cisco designs therefore treat Area 0 as a structural requirement, not just a naming convention.

> If a non-backbone area has no path to Area 0, it is not a valid OSPF hierarchical design. The usual symptom is missing inter-area reachability, not necessarily missing neighbor adjacencies.

**Exam rule: how non-backbone areas connect**

A non-backbone area must connect to Area 0 in one of two ways:

1. **Directly**, with an ABR that has interfaces in Area 0 and the non-backbone area.
2. **Indirectly**, with a virtual link across a transit area when direct Area 0 connectivity is not possible.

If you violate this rule, the area becomes disconnected from the backbone from an OSPF design perspective. The routers may still exchange intra-area routes, but inter-area route propagation breaks.

**Common consequences of a broken Area 0 design**

- Inter-area routes do not appear in the routing table.
- Summary LSAs do not reach the isolated area.
- The ABR may still advertise routes into Area 0, but the remote area cannot receive them.
- External routes can also become unreachable if the path to the ASBR depends on missing inter-area information.
- Troubleshooting often shows full neighbor adjacency inside the area, which can mislead you into thinking OSPF is healthy.

**Route selection and dependency behavior**

OSPF route preference follows the usual hierarchy:

- **Intra-area routes** are preferred over inter-area routes.
- **Inter-area routes** are preferred over external routes.
- **External Type 1** and **External Type 2** routes are both less preferred than intra-area and inter-area routes, but Type 1 includes internal OSPF cost to the ASBR while Type 2 uses only the external metric.

Administrative distance also matters in the routing table:

- OSPF internal routes typically use administrative distance 110.
- If another protocol has a lower administrative distance, it may win even when OSPF has a valid path.
- In multi-ABR designs, the router chooses the best OSPF path based on cost and LSA information, not simply the first ABR it hears from.

**Area 0 design patterns**

A scalable enterprise design usually follows one of these patterns:

- **Preferred design**: every area connects directly to Area 0 through one or more ABRs.
- **Acceptable exception**: a virtual link is used temporarily or when redesign is not immediately possible.
- **Poor design**: long chains of areas or accidental backbone discontinuity.

Cisco generally recommends redesign over long-term virtual-link dependence. Virtual links solve a topology problem, but they add operational complexity and can make troubleshooting harder.

**End-to-end broken design example**

Consider this topology:

- Area 0 exists in the core.
- Area 10 is connected to Area 0 through ABR1.
- Area 20 is connected only to Area 10 through ABR2.
- Area 20 has no direct or virtual connection to Area 0.

In this design, Area 20 is not properly attached to the backbone. ABR2 can still form OSPF neighbors in Area 10 and Area 20, but Area 20 is not a valid backbone-connected area. Inter-area routes from Area 0 may not be installed correctly in Area 20.

A better design is either:

- Add an interface from ABR2 into Area 0, or
- Build a virtual link between ABR2 and the ABR that has a stable path through Area 10 to Area 0.

**Virtual links**

A virtual link creates a logical adjacency to Area 0 across a transit area. It is used when an ABR does not have direct physical connectivity to Area 0 but can reach another ABR through a common area.

The important exam distinction is this:

- The **transit area** must be a normal, non-stub area that both ABRs can fully reach through OSPF.
- The virtual link endpoints are the two ABRs.
- The virtual link itself behaves like a logical tunnel for backbone continuity.

Virtual links are not a substitute for good design. They are a repair mechanism.

**When to use a virtual link**

Use a virtual link when:

- You must restore backbone continuity without redesigning the topology immediately.
- A remote area is already deployed and cannot be renumbered or re-cabled quickly.
- You need a temporary migration path during an OSPF redesign.

Avoid a virtual link when:

- You can simply add a direct Area 0 connection.
- The transit area is unstable or likely to change.
- The transit area is a stub, totally stubby, NSSA, or totally NSSA area.

> The transit area for a virtual link must not be a stub-type area. The restriction applies to the transit area, not to the area containing the endpoints.

**Virtual link command context**

On Cisco IOS, the virtual link is configured under the OSPF process and the area statement for the transit area. You identify the remote ABR by its router ID.

```cisco
router ospf 10
 area 10 virtual-link 2.2.2.2
```

If authentication is required, the exact key syntax can vary by platform and IOS train, but the concept is the same: configure authentication for the virtual link and ensure both ends match.

```cisco
router ospf 10
 area 10 virtual-link 2.2.2.2 authentication message-digest
 area 10 virtual-link 2.2.2.2 message-digest-key 1 md5 <key-string>
```

Use the platform-specific syntax supported by your device. In labs and exams, focus on the relationship between the transit area, the remote router ID, and matching authentication rather than memorizing one exact vendor variant.

**How to identify the transit area and remote router ID**

1. Find the ABR that is not directly connected to Area 0.
2. Identify the area shared by both ABRs; that is the transit area.
3. Determine the remote ABR router ID with `show ip ospf` or `show ip ospf neighbor`.
4. Configure the virtual link under the transit area using that router ID.

Useful verification commands:

```cisco
show ip ospf
show ip ospf neighbor
show ip ospf virtual-links
show ip ospf interface brief
show ip route ospf
```

**What healthy virtual-link output looks like**

A healthy virtual link typically shows:

- State as `FULL`
- The transit area name or ID
- The remote router ID
- Authentication status if configured
- A stable adjacency with no repeated resets

Example output shape:

```cisco
show ip ospf virtual-links

Virtual Link OSPF_VL0 to router 2.2.2.2 is up
  Run as demand circuit
  Transit area 10, via interface GigabitEthernet0/1
  State POINT_TO_POINT, cost 10
  Authentication enabled
```

The exact formatting varies by IOS version, so use the fields, not the punctuation, as your study target.

**Troubleshooting virtual links**

Common failure causes include:

- The transit area is stub or NSSA.
- The remote router ID is wrong.
- The ABRs do not have full OSPF reachability through the transit area.
- Authentication mismatches exist.
- The transit path is unstable or filtered.
- The virtual link is configured under the wrong area number.

Symptoms often include:

- `INIT` or `EXSTART` state instead of `FULL`
- Virtual link flapping
- Missing inter-area routes beyond the broken area
- Inconsistent LSDB (Link-State Database) contents between ABRs

**LSA behavior and flooding scope**

OSPF uses Link-State Advertisements (<abbr title="Link-State Advertisement">LSAs</abbr>) to describe topology and reachability. Area design affects which LSAs are flooded and where they stop.

| LSA Type | Name | Main Role | Flooding Scope |
|---|---|---|---|
| Type 1 | Router LSA | Describes a router’s links inside an area | Area only |
| Type 2 | Network LSA | Describes a multiaccess network and its DR | Area only |
| Type 3 | Summary LSA | Advertises inter-area networks from an ABR | Between areas |
| Type 4 | ASBR Summary LSA | Reaches the ASBR from other areas | Between areas |
| Type 5 | External LSA | Advertises redistributed external routes | OSPF domain-wide, except stub restrictions |
| Type 7 | NSSA External LSA | Carries external routes inside an NSSA | NSSA only until translated |

Type 1 and Type 2 LSAs stay inside the area. Type 3 and Type 4 LSAs are generated by ABRs to move reachability information between areas. Type 5 LSAs are generated by ASBRs for redistributed routes and are flooded throughout the OSPF domain unless blocked by area type rules. Type 7 LSAs are used inside NSSA areas and are translated into Type 5 by the NSSA ABR when appropriate.

**Why Type 4 matters**

Type 4 LSAs help routers in other areas reach the ASBR itself. That matters because external routes are only useful if the router can also reach the ASBR when needed. However, a router in the same area as the ASBR can often use the external route without needing a Type 4 summary to reach the ASBR first.

So the practical rule is:

- **Same area as ASBR**: external route can be used directly.
- **Different area**: Type 4 helps the router find the ASBR that originated the external route.

**NSSA behavior**

A Not-So-Stubby Area (<abbr title="Not-So-Stubby Area">NSSA</abbr>) allows limited redistribution inside the area while still reducing external LSA flooding from the rest of the domain.

Key NSSA points:

- External routes inside the NSSA are carried as Type 7 LSAs.
- The NSSA ABR translates Type 7 to Type 5 when the route leaves the NSSA, unless configured otherwise.
- NSSA areas can still have a default route injected from the ABR.
- A virtual link cannot use an NSSA as the transit area.

**Stub area types compared**

| Area Type | External Type 5 LSAs | Type 7 LSAs | Default Route Behavior | Typical Use |
|---|---:|---:|---|---|
| Stub | Blocked | Not used | ABR injects a default route | Simple branch area |
| Totally stubby | Blocked | Not used | ABR injects only a default route and suppresses most summaries | Very small branch with minimal routing |
| NSSA | Blocked from outside; Type 7 allowed inside | Used | ABR can inject a default route | Branch that needs limited redistribution |
| Totally NSSA | Blocked from outside; Type 7 allowed inside | Used | ABR injects only a default route and suppresses most summaries | Branch with redistribution needs and minimal inter-area detail |

The exact Cisco command set differs slightly by platform, but the design intent is consistent: stub-style areas reduce routing table size and LSA flooding.

**Area-type configuration examples**

Stub area:

```cisco
router ospf 10
 area 20 stub
```

Totally stubby area on Cisco IOS:

```cisco
router ospf 10
 area 20 stub no-summary
```

NSSA:

```cisco
router ospf 10
 area 30 nssa
```

Totally NSSA:

```cisco
router ospf 10
 area 30 nssa no-summary
```

If you enable one of these area types, all routers in that area must agree on the area type, and the ABR must be configured accordingly.

**Redistribution and external routes**

When redistributing another protocol into OSPF, the ASBR injects external LSAs. In practice, you usually need to specify a metric or a default metric unless one is already configured.

```cisco
router ospf 10
 redistribute eigrp 100 subnets metric 20 metric-type 1
```

Practical notes:

- `subnets` is commonly used so OSPF advertises subnetted routes.
- `metric-type 1` makes the external cost include internal OSPF cost to the ASBR.
- `metric-type 2` uses only the external metric and is often preferred for simple external reachability, but Type 1 can be better when internal path cost matters.

**Practical topology scenario**

Imagine a campus with:

- Core in Area 0
- Distribution in Area 10
- A remote branch in Area 20
- The branch ABR only connects to Area 10

If Area 10 is a normal transit area and both ABRs can reach each other through it, you can build a virtual link to restore backbone continuity. If Area 10 is configured as stub or NSSA, the virtual link fails by design. The fix is either:

- Reconfigure the transit area as a normal area, or
- Redesign the topology so the branch ABR connects directly to Area 0.

This is why virtual links are usually a migration tool, not a long-term architecture.

**Verification and troubleshooting workflow**

1. Confirm the area map with `show ip ospf`.
2. Check neighbor states with `show ip ospf neighbor`.
3. Verify the LSDB with `show ip ospf database`.
4. Confirm inter-area routes with `show ip route ospf`.
5. If a virtual link is involved, check `show ip ospf virtual-links`.
6. Validate that the transit area is normal and that both ABRs can reach each other through it.

Useful commands:

```cisco
show ip ospf
show ip ospf database
show ip ospf database summary
show ip ospf database router
show ip ospf database external
show ip route ospf
show ip ospf virtual-links
```

**OSPFv2 and OSPFv3 considerations**

OSPFv2 is used for IPv4, while OSPFv3 is used for IPv6. The area design principles are similar:

- Area 0 remains the backbone.
- Non-backbone areas still need a path to Area 0.
- Virtual links are a design concept associated with backbone continuity.

The exact configuration syntax differs between OSPFv2 and OSPFv3, especially for interface-based configuration in OSPFv3. For exam prep, focus on the design rule first, then learn the platform-specific syntax.

**Design guidance for the exam**

Choose the simplest valid design first:

- Prefer direct Area 0 connectivity.
- Use summarization to reduce route scale where appropriate.
- Use stub or NSSA areas to limit LSA flooding at the edge.
- Use virtual links only when you cannot immediately fix the physical or logical topology.

Cisco’s scalable enterprise pattern is a stable backbone with clear area boundaries, predictable summarization, and minimal dependence on special-case repairs.

**High-value exam reminders**

- **Area 0** is mandatory for inter-area backbone continuity.
- A non-backbone area must connect to Area 0 directly or through a virtual link.
- The **transit area** for a virtual link must be a normal area.
- Stub and NSSA area types reduce LSA flooding and change default-route behavior.
- Type 7 LSAs exist inside NSSA and are translated by the ABR.
- Type 4 LSAs help reach the ASBR from other areas, but same-area routers can still use external routes without them.
- Virtual links are a workaround, not the preferred long-term design.
