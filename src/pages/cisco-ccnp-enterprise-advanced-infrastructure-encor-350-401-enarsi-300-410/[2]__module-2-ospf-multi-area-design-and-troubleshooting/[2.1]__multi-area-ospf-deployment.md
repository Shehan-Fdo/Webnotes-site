## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Multi-area OSPF design goals**

Open Shortest Path First (OSPF) scales better when you split a large domain into multiple areas. The main goals are to reduce link-state database size, limit flooding, and keep SPF (Shortest Path First) recalculation local to where changes occur.

A good design usually starts with a stable backbone and then adds non-backbone areas around it. You use areas to contain churn, not to create complexity for its own sake.

- **Area 0** is the backbone and must remain the transit core for inter-area routing.
- **ABR** means Area Border Router; it connects area 0 to one or more non-backbone areas.
- **ASBR** means Autonomous System Boundary Router; it injects external routes into OSPF.
- Keep area boundaries aligned with topology and failure domains, not arbitrary VLAN counts.
- Split areas when the link-state database, SPF load, or flooding scope becomes too large for a single domain.

**Backbone-first design rules**

Design the backbone first, then attach other areas to it. Every non-backbone area should have a clear path to area 0 through an ABR. If you cannot provide a physical backbone path, you may need a virtual link, but that is a repair mechanism, not a preferred design.

- Place high-speed, stable links in area 0 when possible.
- Keep area 0 contiguous and resilient.
- Avoid making access or unstable links part of the backbone.
- Use summarization at area boundaries to reduce route scale.
- Prefer simple hub-and-spoke or hierarchical topologies for large enterprises.

> A non-backbone area must still connect to area 0 logically. If the backbone is partitioned, inter-area routing breaks until connectivity is restored or a virtual link is used.

**OSPF area types and LSA behavior**

OSPF uses Link-State Advertisements (LSAs) to describe topology and reachability. Multi-area design matters because different area types filter or replace certain LSA types at the boundary.

| LSA Type | Name | Typical Scope | Notes |
|---|---|---:|---|
| Type 1 | Router LSA | Area | Generated by every OSPF router for each area |
| Type 2 | Network LSA | Area | Generated by the DR (Designated Router) on multiaccess networks |
| Type 3 | Summary LSA | Inter-area | Generated by ABRs to advertise routes between areas |
| Type 4 | ASBR Summary LSA | Inter-area | Generated by ABRs to reach an ASBR from other areas |
| Type 5 | External LSA | Domain-wide | Generated by ASBRs for redistributed routes in normal areas |
| Type 7 | NSSA External LSA | NSSA area | Generated inside NSSA, translated to Type 5 by the ABR |

The key exam idea is that area boundaries change what gets flooded and what gets hidden. Inside an area, routers see detailed Type 1 and Type 2 information. Across areas, ABRs summarize and translate reachability using Type 3 and Type 4 LSAs. External routes are usually Type 5, except in NSSA where they begin as Type 7.

**How LSAs move across area boundaries**

- **Type 1 and Type 2** stay inside the area.
- **Type 3** is created by ABRs to advertise inter-area prefixes.
- **Type 4** appears when an area needs a path to an ASBR located elsewhere.
- **Type 5** floods through normal areas but is blocked by stub-style areas.
- **Type 7** stays inside NSSA until the ABR translates it to Type 5.

This behavior explains why area type selection affects route visibility, default routes, and troubleshooting.

**ABR behavior and route selection**

An ABR maintains a separate SPF view for each attached area. It learns intra-area routes from one area and advertises them into another area as summary routes. The ABR does not simply “pass through” routes; it re-originates them with area-scoped summaries.

Route selection across areas depends on the best path to the destination prefix and the cost to reach the ABR advertising it. OSPF uses cumulative interface cost, so the total path cost includes:

- Cost from the local router to the ABR
- Cost from the ABR through the destination area
- Any additional intra-area cost to the destination

If multiple ABRs advertise the same inter-area prefix with equal cost, the router can install equal-cost paths if the platform supports ECMP (Equal-Cost Multi-Path).

- Prefer the lowest total OSPF cost, not the fewest hops.
- Multiple ABRs can provide resilience and load sharing.
- Summaries can change which ABR appears best by hiding more-specific routes.
- Unequal costs can create asymmetric traffic patterns between areas.

**Multi-area placement guidelines**

Use these guidelines when designing enterprise OSPF:

- Keep the backbone small, stable, and highly available.
- Place distribution or core routers as ABRs when they aggregate multiple access areas.
- Avoid stretching area 0 across low-speed or failure-prone links.
- Use one area per site, building, or major failure domain when it improves containment.
- Summarize at the edge of each area to reduce routing table growth.
- Do not over-segment small networks; the overhead may outweigh the benefit.

A common enterprise pattern is:

- Area 0 in the core
- One area per campus or region
- ABRs at the distribution layer
- ASBRs at controlled redistribution points

**Basic multi-area configuration**

On Cisco IOS, OSPFv2 commonly uses `network` statements to place interfaces into areas. The area number must match on all routers that share that segment.

```cisco
router ospf 10
 router-id 1.1.1.1
 network 10.0.0.0 0.0.0.255 area 0
 network 10.1.0.0 0.0.255.255 area 10
 network 10.2.0.0 0.0.255.255 area 20
```

This example places one interface range in the backbone and two ranges in separate areas. In practice, you should verify the interface masks and adjacency placement carefully so you do not accidentally split a transit link into the wrong area.

Useful verification commands:

```cisco
show ip ospf neighbor
show ip ospf interface brief
show ip ospf database
show ip route ospf
show ip ospf
```

**Inter-area summarization on ABRs**

Summarization reduces the number of prefixes advertised between areas. In OSPF, you configure inter-area summarization on the ABR using `area <area-id> range <prefix> <mask>`.

```cisco
router ospf 10
 area 10 range 10.10.0.0 255.255.0.0
```

This command summarizes routes from area 10 into a broader prefix when the ABR advertises them into other areas. The summary should align to contiguous prefixes that actually exist in the area. If the summary is too broad, it can create a black hole if the ABR advertises reachability for addresses it cannot truly reach.

For OSPFv2, the mask in `area range` is a subnet mask, not a wildcard mask.

```cisco
router ospf 10
 area 10 range 10.10.0.0 255.255.0.0
```

For OSPFv3, summarization is also configured on the ABR, but the syntax is interface- and prefix-oriented rather than the classic OSPFv2 network statement model.

```cisco
ipv6 router ospf 10
 router-id 1.1.1.1
 area 10 range 2001:db8:10::/48
```

OSPFv3 on Cisco platforms is activated on interfaces, not with the same `network` statement model used by OSPFv2.

```cisco
interface GigabitEthernet0/0
 ipv6 ospf 10 area 10
```

**External summarization on ASBRs**

If you redistribute routes into OSPF, you can summarize external prefixes on the ASBR. This reduces Type 5 LSA volume and helps keep the domain manageable.

```cisco
router ospf 10
 redistribute static subnets
 summary-address 172.16.0.0 255.255.0.0
```

External summarization is especially useful when redistributing many connected, static, or BGP routes. Use it carefully, because a summary can hide more-specific failures and make troubleshooting harder.

**Default route injection by area type**

Stub-style areas use a default route to reduce routing table size. The ABR injects the default instead of flooding external or inter-area detail that the area does not need.

| Area Type | Inter-area Type 3s | External Type 5s | Default Route | Notes |
|---|---:|---:|---:|---|
| Normal | Yes | Yes | Optional | Full OSPF visibility |
| Stub | Yes, but no Type 5s | No | Yes | ABR injects a default summary |
| Totally stubby | No, except default | No | Yes | Only default route plus intra-area routes |
| NSSA | Yes | No Type 5s inside area | Yes, optional | External routes enter as Type 7 |
| Totally NSSA | No, except default | No Type 5s inside area | Yes | Only default route plus intra-area routes |

The distinction between stub and totally stubby matters. In a totally stubby area, the ABR suppresses inter-area detail and injects only a default route, while the area still keeps its local intra-area routes.

**Stub area configuration**

On the ABR and all routers in the area, configure the area as stub.

```cisco
router ospf 10
 area 10 stub
```

If you want a totally stubby area on Cisco IOS, configure the ABR with `no-summary`.

```cisco
router ospf 10
 area 10 stub no-summary
```

The ABR still advertises a default route into the area, but it suppresses other inter-area and external detail.

**NSSA and totally NSSA**

Not-So-Stubby Area (NSSA) allows limited external redistribution inside the area. This is useful when a branch needs to inject a local static or connected route but still should not receive full external Type 5 flooding from the rest of the domain.

```cisco
router ospf 10
 area 20 nssa
```

Inside an NSSA, redistributed routes become Type 7 LSAs. The ABR translates Type 7 to Type 5 when sending them into normal areas, unless translation is suppressed or another ABR is elected translator.

For a totally NSSA area, the ABR suppresses inter-area detail and injects only a default route, while still allowing local Type 7 redistribution inside the NSSA.

```cisco
router ospf 10
 area 20 nssa no-summary
```

**Virtual links**

A virtual link is a logical tunnel used to connect an area to the backbone through a transit area. It is a repair tool for a broken or noncontiguous area 0 design, not a preferred architecture.

A virtual link is configured between two ABRs that share a common transit area. One ABR must already be connected to area 0, and the other ABR reaches area 0 through the transit area.

```cisco
router ospf 10
 area 10 virtual-link 2.2.2.2
```

In this example, the local router is an ABR in area 10, and `2.2.2.2` is the router ID of the remote ABR that has backbone connectivity through the same transit area.

Important virtual-link considerations:

- The transit area must be stable and fully adjacent on both sides.
- Authentication must match across the transit area if enabled.
- The virtual link inherits the transit area’s reliability, so failures there affect backbone reachability.
- You should use router IDs, not interface addresses, in the command.

Common failure causes:

- The transit area is not fully adjacent.
- One side is not actually an ABR.
- Router IDs are misconfigured or changed.
- Authentication mismatch exists on the transit area.
- The backbone is already broken in a way that prevents the virtual link from forming.

Verification commands:

```cisco
show ip ospf virtual-links
show ip ospf neighbor
show ip ospf database
show ip ospf interface
```

**OSPF cost and path selection across areas**

OSPF cost is interface-based and usually derived from bandwidth. The path cost to an inter-area destination is the sum of the local intra-area cost to the ABR plus the ABR’s advertised cost to the destination.

You can tune cost manually when needed:

```cisco
interface GigabitEthernet0/1
 ip ospf cost 10
```

This is useful when you want to influence which ABR or exit path is preferred. Be careful not to create unexpected asymmetry between areas.

**Equal-cost multi-path across ABRs**

If two ABRs advertise the same prefix with the same total cost, the router may install multiple next hops. This can improve resilience and load distribution.

- Verify that the destination prefix is truly equal cost from the local router’s perspective.
- Check whether the platform supports OSPF ECMP limits.
- Remember that summarization can hide more-specific differences and make paths appear equal when they are not.

**OSPFv3 multi-area considerations**

OSPFv3 carries IPv6 routes and uses interface-based activation. The multi-area design principles are the same, but the configuration model differs from OSPFv2.

Key differences:

- You enable OSPFv3 per interface.
- You still use areas and ABRs.
- Summarization is configured under the OSPFv3 process on the ABR.
- Verification uses IPv6-aware commands.

Example:

```cisco
ipv6 unicast-routing

ipv6 router ospf 10
 router-id 1.1.1.1
 area 0 range 2001:db8:100::/48

interface GigabitEthernet0/0
 ipv6 ospf 10 area 0

interface GigabitEthernet0/1
 ipv6 ospf 10 area 10
```

Useful verification:

```cisco
show ipv6 ospf neighbor
show ipv6 ospf interface brief
show ipv6 ospf database
show ipv6 route ospf
show ipv6 ospf border-routers
```

**Authentication in multi-area OSPF**

Authentication protects adjacencies from unauthorized peers and accidental mismatches. In classic OSPFv2 on IOS, the common options are plain text and MD5. More advanced authentication options may exist on some platforms or in newer feature sets, but they are not universal.

Examples:

```cisco
interface GigabitEthernet0/0
 ip ospf authentication message-digest
 ip ospf message-digest-key 1 md5 MySecretKey
```

For OSPFv3, authentication is handled differently depending on platform and implementation. Do not assume the OSPFv2 authentication commands map directly to OSPFv3.

> Authentication mismatches are a common cause of adjacency failure, especially on virtual links and transit areas.

**Troubleshooting multi-area OSPF**

Start by checking adjacency, then verify area membership, then inspect the LSDB (Link-State Database), and finally confirm the routing table.

1. Confirm neighbors are full.
2. Confirm each interface is in the intended area.
3. Inspect LSAs by type.
4. Check whether summaries or defaults are present.
5. Validate the best path and cost.

High-value commands:

```cisco
show ip ospf neighbor
show ip ospf interface brief
show ip ospf database router
show ip ospf database network
show ip ospf database summary
show ip ospf database external
show ip ospf database nssa-external
show ip route ospf
show ip ospf border-routers
```

What to look for:

- **Type 1/2** issues usually indicate local adjacency or DR problems.
- **Type 3** issues usually indicate ABR summarization or area boundary problems.
- **Type 4** issues usually indicate the path to an ASBR is missing.
- **Type 5** issues usually indicate redistribution or stub-area filtering problems.
- **Type 7** issues usually indicate NSSA translation or ABR election problems.

If a route is missing from a non-backbone area, check whether the area type blocks it by design. For example, a stub area will not receive Type 5 externals, and a totally stubby area will not receive normal Type 3 inter-area routes either.

**Common exam traps**

- Assuming any router in an area can configure inter-area summarization.
- Confusing stub, totally stubby, NSSA, and totally NSSA behavior.
- Forgetting that area 0 is the backbone and that virtual links are exceptions.
- Treating OSPFv3 like OSPFv2 with IPv6 addresses added.
- Ignoring the difference between Type 5 and Type 7 external LSAs.
- Assuming summarization always improves routing without side effects.

**Practical design example**

A campus has a core, two distribution blocks, and several access areas. A clean design might look like this:

- Core routers form area 0.
- Distribution block A is area 10.
- Distribution block B is area 20.
- Each block has an ABR at the distribution layer.
- Each block summarizes its local subnets toward the core.
- Branch redistribution, if needed, occurs at a controlled ASBR.

This design keeps the backbone stable, limits flooding, and makes failures easier to isolate.

**Quick verification checklist**

- `show ip ospf neighbor` shows all expected adjacencies in `FULL` state.
- `show ip ospf database summary` shows Type 3 LSAs from ABRs.
- `show ip ospf database external` shows Type 5 LSAs only where allowed.
- `show ip ospf database nssa-external` shows Type 7 LSAs inside NSSA.
- `show ip route ospf` shows the expected inter-area and external routes.
- `show ip ospf virtual-links` is empty unless you intentionally use a virtual link.

**Key takeaways**

Multi-area OSPF is about containment, scale, and control. You use area 0 as the backbone, ABRs to connect areas, summarization to reduce route volume, and special area types to limit what floods into smaller sites. The most important exam skill is knowing which LSA types appear where, and why a route is present, summarized, translated, or blocked.
