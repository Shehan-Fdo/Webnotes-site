## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**OSPF LSA fundamentals**

<abbr title="Open Shortest Path First">OSPF</abbr> uses <abbr title="Link-State Advertisement">LSA</abbr>s to describe topology and reachability. Each router builds a link-state database (<abbr title="Link-State Database">LSDB</abbr>) from received LSAs, then runs the Shortest Path First (<abbr title="SPF">SPF</abbr>) algorithm to compute routes.

The key exam idea is that LSAs do not all behave the same way. Their type determines:
- what information they carry,
- how far they flood,
- who originates them,
- and how they influence route selection.

A practical way to think about OSPF is:
- **LSAs describe topology**
- **SPF computes best paths**
- **the routing table installs selected results**

**Core flooding scope concepts**

Flooding scope is one of the most tested distinctions in OSPF.

- **Link-local** LSAs stay on a single link or segment.
- **Area-local** LSAs stay inside one OSPF area.
- **AS-wide** LSAs can cross the OSPF domain, subject to area type restrictions.
- **External** LSAs represent redistributed routes and are treated differently from internal topology LSAs.

> In exam scenarios, always ask: “Does this LSA stay on the link, stay in the area, or cross the domain?”

**OSPFv2 LSA types at a glance**

| Type | Name | Originator | Flooding scope | Main purpose |
|---|---|---|---|---|
| 1 | Router LSA | Every OSPF router per area | Area-local | Describes router links, link IDs, metrics, and attached networks |
| 2 | Network LSA | DR on multiaccess networks | Area-local | Describes the shared segment and attached routers |
| 3 | Summary LSA | ABR | Area-local into other areas | Advertises inter-area prefixes |
| 4 | ASBR Summary LSA | ABR | Area-local into other areas | Reaches the ASBR, not the external prefix |
| 5 | External LSA | ASBR | AS-wide, except stub-style areas that block externals | Advertises redistributed routes |
| 7 | NSSA External LSA | ASBR inside NSSA | NSSA-local until translated | Carries redistributed routes inside an NSSA |

**Type 1 Router LSA details**

A Type 1 LSA is generated by every router for each area it participates in. It describes the router’s links and the cost of those links. It is the foundation of SPF because it tells the router how the local topology is connected.

Important fields and behavior:
- **Link ID** identifies the neighbor router ID, transit network, or stub network depending on link type.
- **Link Data** carries interface-specific information such as IP address or subnet mask details, depending on the link type.
- **Metric** represents the cost of the link.
- **Flags and link types** help distinguish point-to-point, transit, and stub links.

Type 1 LSAs do not directly say “this router is adjacent to that router” as a separate adjacency field. Instead, adjacency is inferred from the link information and the presence of matching LSAs in the area.

Practical implication:
- If a router is missing from the LSDB, SPF cannot build a complete topology.
- If a link cost changes, the Type 1 LSA changes and SPF recalculates paths.

**Type 2 Network LSA details**

A Type 2 LSA is generated by the Designated Router (<abbr title="Designated Router">DR</abbr>) on a broadcast or nonbroadcast multiaccess segment. It represents the shared network and lists the routers attached to that segment.

Why it matters:
- It reduces LSDB complexity on multiaccess networks.
- It gives SPF a single node for the shared segment instead of modeling every neighbor relationship separately.

Typical contents:
- **Network mask**
- **Attached router IDs**
- **DR information for the segment**

On point-to-point links, Type 2 LSAs are not used because there is no DR election.

**Type 3 Summary LSAs and inter-area propagation**

Type 3 LSAs are originated by an Area Border Router (<abbr title="Area Border Router">ABR</abbr>) to advertise networks from one area into another. They are the primary mechanism for inter-area routing.

Key behavior:
- They are area-scoped from the perspective of each receiving area.
- They summarize reachability from one area into another.
- They are the basis for `O IA` routes in the routing table.

Important nuance:
- The next hop for an inter-area route is typically the ABR that originated the Type 3 LSA.
- The receiving router uses the ABR as the path toward the advertised prefix.

Example route code:
- `O IA` = OSPF inter-area route

**Type 4 ASBR Summary LSAs**

Type 4 LSAs are also originated by an ABR. They do not describe the external prefix itself. Instead, they tell routers how to reach the Autonomous System Boundary Router (<abbr title="Autonomous System Boundary Router">ASBR</abbr>) that injected external routes.

Why this matters:
- A router needs a path to the ASBR before it can use a Type 5 external route.
- Type 4 LSAs are therefore a “reach the ASBR” helper, not the external route advertisement itself.

Common route-table relationship:
- Type 4 supports the resolution of `O E1` and `O E2` routes.
- If the ASBR is unreachable in the LSDB, external routes may not install correctly.

**Type 5 External LSAs**

Type 5 LSAs are originated by an ASBR to advertise redistributed routes. They are flooded throughout the OSPF domain, but they are blocked by area types that do not accept external LSAs.

Area behavior:
- Allowed in normal areas
- Not allowed into stub areas
- Not allowed into totally stubby areas
- Not allowed into NSSA areas as Type 5s
- Not allowed into totally NSSA areas as Type 5s

Important distinction:
- NSSA areas do not accept Type 5 LSAs directly.
- An NSSA uses Type 7 LSAs internally, then an ABR translates them to Type 5s when they leave the NSSA.

External route types:
- **E1**: metric includes internal OSPF cost to the ASBR plus external metric
- **E2**: metric is primarily the external metric; internal cost to the ASBR is used for tie-breaking

Practical rule:
- Use **E1** when you want internal path cost to matter.
- Use **E2** when you want the external metric to dominate.

**Type 7 NSSA External LSAs**

Type 7 LSAs are used inside a Not-So-Stubby Area (<abbr title="Not-So-Stubby Area">NSSA</abbr>) to carry redistributed routes. They let an ASBR inside the NSSA inject external information without violating the NSSA restriction against Type 5 LSAs.

How they work:
- An ASBR inside the NSSA originates Type 7 LSAs.
- The NSSA ABR translates Type 7 LSAs into Type 5 LSAs when advertising them into the rest of the OSPF domain.
- The translated Type 5 is then flooded outside the NSSA.

Translation nuance:
- If multiple ABRs connect the NSSA to the backbone, one ABR becomes the translator for a given external LSA.
- Translation is influenced by the translator election process and router IDs.
- The translator role can change if the preferred ABR fails.

Route-table behavior:
- Inside the NSSA, the route may appear as `O N1` or `O N2`.
- Outside the NSSA, the same route is seen as `O E1` or `O E2` after translation.

**OSPFv3 LSA types**

OSPFv3 uses a different LSA model than OSPFv2. If you are studying broadly for CCNP, know that OSPFv3 introduces additional LSA types and separates topology from addressing more cleanly.

Common OSPFv3 LSA types include:
- **Type 1 Router LSA**
- **Type 2 Network LSA**
- **Type 3 Inter-Area Prefix LSA**
- **Type 4 Inter-Area Router LSA**
- **Type 5 AS-External LSA**
- **Type 7 NSSA LSA**
- **Type 8 Link LSA**
- **Type 9 Intra-Area Prefix LSA**
- **Type 10 Opaque LSA, area scope**
- **Type 11 Opaque LSA, AS scope**

Why this matters:
- OSPFv3 separates router topology from prefix information more explicitly.
- Type 8 and Type 9 are especially important for IPv6 prefix handling and link-local information.
- Type 10 and Type 11 are used for opaque extensions and scope-specific flooding.

**Flooding scope comparison**

| Scope | Typical LSA types | Where it stays | Exam takeaway |
|---|---|---|---|
| Link-local | OSPFv3 Type 8 | Single link | Used for link information and neighbor discovery support |
| Area-local | Type 1, 2, 3, 4, 7 in their respective area context; OSPFv3 Type 9, 10 | One area | Most topology and summary information stays here |
| AS-wide | Type 5, OSPFv3 Type 11 | Entire OSPF domain, if area type permits | Used for external reachability or domain-wide opaque info |

**LSA aging, sequence numbers, and refresh**

LSAs are not static. They age, refresh, and eventually flush from the LSDB if they are no longer valid.

Key concepts:
- **Sequence number** increases when an LSA is updated.
- **Checksum** helps verify integrity.
- **Age** tracks how long the LSA has existed.
- **MaxAge** is the point at which an LSA is considered expired and is flushed.

Operational behavior:
- A router refreshes self-originated LSAs periodically before they reach MaxAge.
- If a route disappears, the originating router can flush the LSA by advertising it with MaxAge.
- A newer LSA with a higher sequence number replaces an older one.

Typical OSPF refresh interval:
- Self-originated LSAs are refreshed periodically, commonly around 30 minutes in standard OSPF behavior.

Troubleshooting value:
- If an LSA persists unexpectedly, check whether the originator is still refreshing it.
- If two routers disagree on topology, compare sequence numbers and ages in the LSDB.

**How LSAs drive route selection**

OSPF route installation depends on the LSA type and the path cost computed by SPF.

Common route codes:
- `O` = intra-area route
- `O IA` = inter-area route
- `O E1` = external type 1
- `O E2` = external type 2
- `O N1` = NSSA external type 1
- `O N2` = NSSA external type 2

Selection logic:
- Intra-area routes are preferred over inter-area routes.
- Inter-area routes are preferred over external routes.
- For external routes, E1 and N1 include internal cost in the total metric.
- For E2 and N2, the external metric is primary unless there is a tie.

Practical example:
- If the same prefix is learned as `O IA` and `O E2`, the router prefers `O IA`.
- If two external paths exist, the one with the lower effective metric wins, with E1/N1 considering internal cost.

**Default route propagation**

Default route behavior depends on area type and configuration.

Common patterns:
- An ABR can inject a default route into a stub area automatically or by configuration.
- A totally stubby area typically receives only a default route from the ABR, not other inter-area summaries.
- An NSSA can also receive a default route from the ABR.

Cisco-style examples:
```bash
router ospf 10
 area 20 stub
```

```bash
router ospf 10
 area 20 stub no-summary
```

```bash
router ospf 10
 area 20 nssa
```

```bash
router ospf 10
 area 20 nssa default-information-originate
```

```bash
router ospf 10
 area 20 nssa no-summary
```

Notes on usage:
- `no-summary` is an ABR-side behavior for creating a totally stubby or totally NSSA area.
- `default-information-originate` under an NSSA area is used to inject a default route into that NSSA.
- Exact syntax and support can vary by platform and software release, so verify on the target IOS or IOS XE version.

**Area type behavior and LSA suppression**

| Area type | Type 3 summaries | Type 4 summaries | Type 5 externals | Type 7 NSSA externals | Default route behavior |
|---|---|---|---|---|---|
| Normal | Yes | Yes | Yes | No | Optional |
| Stub | Yes | Yes | No | No | ABR injects default |
| Totally stubby | ABR suppresses Type 3s except default | ABR suppresses Type 4s | No | No | Default only |
| NSSA | Yes | Yes | No | Yes | Optional default via ABR |
| Totally NSSA | ABR suppresses Type 3s except default | ABR suppresses Type 4s | No | Yes | Default only |

Important exam nuance:
- “Yes” for Type 3 in totally stubby or totally NSSA does not mean full summary support.
- It means the ABR suppresses all inter-area Type 3s except the default route.

**Virtual links and backbone considerations**

OSPF inter-area routing depends heavily on backbone connectivity. Area 0 is the transit backbone for inter-area LSAs in a standard design.

Why this matters:
- ABRs connect non-backbone areas to Area 0.
- Inter-area LSAs are expected to traverse the backbone.
- If the backbone is broken, route propagation can fail or become inconsistent.

Virtual links:
- A virtual link can logically extend Area 0 through a transit area.
- It is used to repair backbone continuity when a physical Area 0 path is missing.
- Virtual links are a design exception, not a preferred long-term architecture.

Troubleshooting impact:
- If an ABR is not properly connected to Area 0, Type 3 propagation may not behave as expected.
- A virtual link can restore inter-area visibility, but it adds complexity and should be verified carefully.

**Practical show commands**

Use these commands to inspect LSAs and route propagation:

```bash
show ip ospf
show ip ospf neighbor
show ip ospf database
show ip ospf database router
show ip ospf database network
show ip ospf database summary
show ip ospf database external
show ip route ospf
```

Useful targeted checks:
```bash
show ip ospf database router 1.1.1.1
show ip ospf database summary 10.10.10.0
show ip ospf database external 172.16.50.0
show ip ospf database nssa-external
```

What to look for:
- **Router LSA** presence for every router in the area
- **Network LSA** presence on multiaccess segments with a DR
- **Summary LSAs** on ABRs for inter-area prefixes
- **External LSAs** for redistributed routes
- **NSSA LSAs** inside NSSA areas before translation

**Packet and database troubleshooting workflow**

1. Check neighbor state with `show ip ospf neighbor`.
2. Verify the LSDB with `show ip ospf database`.
3. Confirm the expected LSA type exists in the correct area.
4. Compare the route table with `show ip route ospf`.
5. Validate area type and ABR/ASBR roles.
6. Check whether a default route or summary is being suppressed by area type.

Example troubleshooting scenario:
- A redistributed static route exists on an ASBR.
- Inside an NSSA, you expect `O N2`.
- Outside the NSSA, you expect `O E2` after translation.
- If the route is missing outside the NSSA, check the ABR translation role and whether the NSSA is configured correctly.

**End-to-end example topology**

Consider this simplified design:
- Area 0: core
- Area 10: normal area
- Area 20: NSSA with an internal ASBR redistributing static routes

Expected behavior:
- Routers in Area 10 learn inter-area prefixes as `O IA`.
- Routers in Area 20 learn redistributed routes as `O N2` or `O N1`.
- The ABR translates NSSA externals into Type 5 LSAs for the rest of the domain.
- The backbone carries the inter-area summaries between ABRs.

Example configuration snippets:

```bash
router ospf 100
 router-id 1.1.1.1
 network 10.0.0.0 0.0.0.255 area 0
 network 10.10.10.0 0.0.0.255 area 10
 network 10.20.20.0 0.0.0.255 area 20
```

```bash
router ospf 100
 area 20 nssa
 redistribute static subnets metric 20 metric-type 1
```

```bash
router ospf 100
 area 20 nssa default-information-originate
```

Practical interpretation:
- `redistribute static` creates external reachability at the ASBR.
- `area 20 nssa` defines the area behavior.
- The NSSA setting alone does not redistribute routes; redistribution must still be configured under `router ospf`.

**Common exam traps**

- Confusing Type 1 LSAs with neighbor tables.
- Assuming Type 5 LSAs enter all area types.
- Treating Type 7 LSAs as final outside the NSSA.
- Forgetting that Type 4 LSAs help reach the ASBR, not the external prefix.
- Assuming all summary routes are always present in stub-style areas.
- Mixing up `O IA`, `O E1`, `O E2`, `O N1`, and `O N2`.
- Forgetting that OSPFv3 adds Type 8, 9, 10, and 11 LSAs.

**Quick memory aid**

- **1** = router links
- **2** = multiaccess segment
- **3** = inter-area summary
- **4** = reach the ASBR
- **5** = external route
- **7** = external route inside NSSA
- **8/9/10/11** = OSPFv3-specific link, prefix, and opaque behavior

**Key takeaways**

- LSA type determines flooding scope and route behavior.
- Type 1 and Type 2 build the intra-area topology.
- Type 3 and Type 4 support inter-area reachability.
- Type 5 carries redistributed routes across the domain, except where blocked.
- Type 7 enables redistribution inside NSSA and is translated by an ABR.
- Aging, sequence numbers, and refresh behavior are essential for troubleshooting stale or missing routes.
- Backbone continuity and ABR placement strongly affect inter-area propagation.
