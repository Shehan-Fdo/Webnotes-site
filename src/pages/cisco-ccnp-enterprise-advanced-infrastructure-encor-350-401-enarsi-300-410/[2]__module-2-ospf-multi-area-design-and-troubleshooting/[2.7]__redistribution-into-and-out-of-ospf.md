## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**OSPF redistribution overview**

Redistribution injects routes learned from one routing domain into another. In OSPF, this usually means importing routes from connected, static, EIGRP, BGP, RIP, or another OSPF process into the OSPF domain, or exporting OSPF-learned routes into another protocol.

OSPF treats redistributed routes as **external routes**. They are not internal OSPF routes, so they carry different path-selection behavior and are advertised with external LSAs rather than normal intra-area or inter-area LSAs.

- **Intra-area** routes stay inside one OSPF area.
- **Inter-area** routes are learned from another OSPF area through an ABR.
- **External** routes are redistributed into OSPF and are usually represented as Type 5 LSAs.
- **NSSA** external routes use Type 7 LSAs inside the NSSA and are translated to Type 5 by an ABR when they leave the NSSA.

**Where redistribution fits in design**

Use redistribution when you must connect OSPF to another routing domain during migration, merger, or partial-domain integration. It is powerful, but it increases complexity and can create loops if you redistribute in both directions without controls.

Common use cases include:

- Migrating from RIP or EIGRP to OSPF.
- Connecting an OSPF core to a BGP edge.
- Advertising static routes for Internet or service reachability.
- Exchanging routes between two administrative domains with different protocols.

> Redistribution is a design tool, not a default best practice. Prefer summarization, route filtering, and clear domain boundaries whenever possible.

**OSPF route types and external behavior**

OSPF route selection is not just “lowest metric wins.” The protocol first prefers route type:

- **Intra-area** routes are preferred over inter-area routes.
- **Inter-area** routes are preferred over external routes.
- Among external routes, OSPF compares the external type and metric.
- **E1** routes include internal OSPF cost to the ASBR plus the external metric.
- **E2** routes use only the external metric for most comparisons; the internal cost to the ASBR is not added to the metric used for selection, though it still matters for reaching the ASBR.

A route can appear as both internal and external in a complex design. In that case, the internal OSPF path wins because OSPF prefers internal reachability over external reachability for the same prefix.

**External LSAs and forwarding address**

Redistributed routes are carried in external LSAs:

- **Type 5 LSA**: standard external LSA flooded throughout the OSPF domain, except into stub-style areas.
- **Type 7 LSA**: used inside an NSSA; translated to Type 5 by the NSSA ABR if translation is enabled and appropriate.

External LSAs can include a **forwarding address**. If the forwarding address is nonzero, routers may forward traffic toward that address instead of directly to the ASBR. This can affect path choice and is important in designs where the next hop is not the redistributing router itself.

**Area-type limitations**

OSPF area type determines whether external routes can enter the area.

- **Standard area**: accepts Type 5 external LSAs.
- **Stub area**: does not accept Type 5 external LSAs.
- **Totally stubby area**: does not accept Type 5 external LSAs and also suppresses most inter-area detail, leaving a default route.
- **NSSA**: does not accept Type 5 external LSAs, but it can originate Type 7 external LSAs inside the area.

Practical implications:

- You cannot inject normal external routes into a stub or totally stubby area.
- If a redistribution design must exist inside a non-backbone area, use **NSSA** instead of stub.
- An NSSA can originate a default route with `area <id> nssa default-information-originate` on the ABR in some designs, or by redistributing a default route into the NSSA from the ASBR depending on the topology and platform behavior.

**Redistribution into OSPF on Cisco IOS and IOS XE**

For IPv4 OSPF, redistribution is configured under the OSPF process:

```cisco
router ospf 10
 redistribute static subnets
 redistribute eigrp 100 subnets metric 50 metric-type 1
 redistribute bgp 65000 subnets route-map RM-BGP-TO-OSPF
```

Key points:

- `subnets` is important for IPv4 redistribution into OSPF on classic IOS and IOS XE because it allows classless prefixes to be redistributed.
- Without `subnets`, only classful networks are considered for redistribution in many IOS-style implementations.
- A route-map can filter routes and set attributes such as metric, metric type, and tag.
- If you do not explicitly set a metric, the platform may use a protocol-specific default. Do not memorize a single universal default for every source protocol and platform.

For **OSPFv3** and IPv6, redistribution is configured in the OSPFv3 process or address-family context, depending on platform and software release:

```cisco
router ospfv3 10
 address-family ipv6 unicast
  redistribute static metric 1000 metric-type 1
  redistribute connected route-map RM-CONN-TO-OSPFV3
 exit-address-family
```

**Important distinction**

- IPv4 OSPF uses `router ospf <process-id>`.
- OSPFv3 commonly uses `router ospfv3 <process-id>` with an address-family context.
- Do not assume the IPv4 redistribution syntax applies unchanged to IPv6.

**Common redistribution sources into OSPF**

You can redistribute several route sources into OSPF:

- **Connected** routes
- **Static** routes
- **EIGRP** routes
- **BGP** routes
- **RIP** routes
- Another **OSPF** process, if you are intentionally leaking between processes

Examples:

```cisco
router ospf 10
 redistribute connected subnets
 redistribute static subnets
 redistribute eigrp 100 subnets metric 20 metric-type 1
 redistribute rip subnets metric 200 metric-type 2
```

**Practical meaning of each source**

- **Connected**: useful for loopbacks, point-to-point links, or directly attached service networks.
- **Static**: useful for default routes, blackhole routes, or controlled reachability.
- **EIGRP/RIP/BGP**: useful during migration or when OSPF is the new core protocol.
- **Another OSPF process**: use carefully; it can create feedback loops if the same prefixes are reintroduced.

**Metric and metric type**

Redistributed OSPF routes need an external metric. You can set:

- **Metric**: the external cost value.
- **Metric type**: E1 or E2.

Example:

```cisco
router ospf 10
 redistribute static subnets metric 50 metric-type 1
 redistribute eigrp 100 subnets metric 20 metric-type 2
```

Guidance:

- Use **E1** when you want the internal OSPF path to the ASBR to influence selection more strongly.
- Use **E2** when you want the external metric to dominate and internal cost to the ASBR to matter less in route comparison.
- In multi-ASBR designs, E1 often gives more predictable path selection when you care about the closest exit.
- E2 is common for simple redistribution where the external metric is the main policy signal.

**Route tags and loop prevention**

Route tags help prevent feedback loops in mutual redistribution designs. A tag is a numeric marker attached to a redistributed route so that another redistribution point can recognize and filter it.

Example:

```cisco
route-map RM-STATIC-TO-OSPF permit 10
 set metric 100
 set metric-type type-1
 set tag 1001

router ospf 10
 redistribute static subnets route-map RM-STATIC-TO-OSPF
```

Then filter on the return path:

```cisco
route-map RM-OSPF-TO-BGP deny 10
 match tag 1001

route-map RM-OSPF-TO-BGP permit 20
```

Why this matters:

- Without tags, a route can leave OSPF, enter another protocol, and come back into OSPF as a different external route.
- That feedback can create routing loops, duplicate advertisements, or unstable best-path decisions.
- Tags are especially important in two-way redistribution between OSPF and BGP, OSPF and EIGRP, or OSPF and RIP.

**Mutual redistribution design patterns**

Two-way redistribution is risky because each protocol can re-advertise the other’s routes.

Common pitfalls:

- A route learned from OSPF is redistributed into BGP, then redistributed back into OSPF.
- Different metrics or administrative distances cause the same prefix to oscillate.
- Summaries hide the original source, making filtering harder.
- Multiple ASBRs inject the same prefix with different external types or metrics.

Safer patterns:

- Redistribute in only one direction when possible.
- Use route tags to mark redistributed routes.
- Filter with route-maps on both ingress and egress.
- Summarize at the redistribution boundary.
- Prefer a single redistribution point per domain.
- Use default routes instead of full redistribution when the design only needs reachability to one side.

**Default route injection**

A default route is often better than full redistribution when the remote domain only needs an exit path.

Example:

```cisco
ip route 0.0.0.0 0.0.0.0 203.0.113.1

router ospf 10
 default-information originate
```

Notes:

- `default-information originate` advertises a default route only if the router already has one in its routing table, unless `always` is added.
- `default-information originate always` advertises the default even if the router does not currently have a default route.
- This is commonly used at the edge toward an ISP or upstream routing domain.

For NSSA designs, default route handling is area-specific and often uses NSSA configuration on the ABR:

```cisco
router ospf 10
 area 1 nssa default-information-originate
```

Exact behavior can vary by platform and topology, so verify the resulting LSAs and routing table.

**Redistribution out of OSPF**

You can also export OSPF-learned routes into other protocols. This is common when OSPF is the enterprise core and another protocol is used at the edge or in a legacy domain.

Examples:

```cisco
router eigrp 100
 redistribute ospf 10 metric 10000 100 255 1 1500

router bgp 65000
 redistribute ospf 10 route-map RM-OSPF-TO-BGP

router rip
 redistribute ospf 10 metric 2
```

Important considerations:

- The destination protocol usually needs its own metric or policy.
- BGP redistribution often uses route-maps to control which OSPF routes are exported.
- EIGRP redistribution requires a composite metric unless a default metric is configured.
- RIP redistribution uses hop count style metrics and is usually limited to simple designs.

**Example: OSPF to BGP**

```cisco
route-map RM-OSPF-TO-BGP permit 10
 match ip address prefix-list OSPF-EXPORT

ip prefix-list OSPF-EXPORT seq 10 permit 10.10.0.0/16 le 24

router bgp 65000
 redistribute ospf 10 route-map RM-OSPF-TO-BGP
```

This pattern lets you export only selected OSPF prefixes into BGP instead of leaking the entire OSPF table.

**Example: OSPF to EIGRP**

```cisco
router eigrp 100
 default-metric 10000 100 255 1 1500
 redistribute ospf 10 route-map RM-OSPF-TO-EIGRP
```

Or set the metric directly on the redistribution statement if supported in your design:

```cisco
router eigrp 100
 redistribute ospf 10 metric 10000 100 255 1 1500
```

**Verification workflow**

Use a structured verification process after configuration.

1. Confirm the source route exists in the routing table.
2. Confirm the redistribution statement is present.
3. Confirm the route-map permits the intended prefixes.
4. Confirm the OSPF database contains the external LSA.
5. Confirm the destination router installs the route with the expected type and metric.

Useful commands:

```cisco
show ip route ospf
show ip route static
show ip ospf database external
show ip ospf database nssa-external
show ip protocols
show running-config | section router ospf
show running-config | section route-map
show route-map RM-STATIC-TO-OSPF
show ip ospf neighbor
```

For IPv6 / OSPFv3:

```cisco
show ipv6 route ospf
show ospfv3 database
show ospfv3 interface
show running-config | section ospfv3
```

**What to look for**

- `show ip route ospf` should display `O E1`, `O E2`, `O IA`, or `O` entries depending on the route type.
- `show ip ospf database external` should show Type 5 LSAs for standard external routes.
- `show ip ospf database nssa-external` should show Type 7 LSAs inside an NSSA.
- `show ip protocols` should confirm redistribution sources and metrics.
- `show route-map <name>` is platform-dependent; if unavailable, use `show running-config | section route-map`.

**Lab-style end-to-end example**

This example redistributes static routes into OSPF and verifies the result.

```cisco
ip route 172.16.50.0 255.255.255.0 Null0
ip route 172.16.60.0 255.255.255.0 10.1.1.2

route-map RM-STATIC-TO-OSPF permit 10
 match ip address prefix-list PL-STATIC
 set metric 100
 set metric-type type-1
 set tag 500

ip prefix-list PL-STATIC seq 10 permit 172.16.50.0/24
ip prefix-list PL-STATIC seq 20 permit 172.16.60.0/24

router ospf 10
 router-id 1.1.1.1
 redistribute static subnets route-map RM-STATIC-TO-OSPF
```

Verification:

```cisco
show ip route static
show ip ospf database external
show ip route ospf
show ip protocols
```

Expected outcome:

- The static routes appear in the local routing table.
- OSPF originates external LSAs for the permitted prefixes.
- Remote OSPF routers learn the routes as external OSPF entries.
- The tag is visible in the LSA details on platforms that display it.

**Troubleshooting checklist**

If a route does not appear in OSPF, check the following:

- The route exists in the local routing table.
- The route-map permits the prefix.
- The redistribution statement references the correct protocol and process.
- The area is not stub or totally stubby if you expect external LSAs to cross it.
- The prefix is not blocked by a distribute-list, prefix-list, or route-policy.
- The metric and metric type are valid.
- The ASBR has OSPF adjacency and can flood LSAs.
- For NSSA, confirm Type 7 generation and translation behavior.

If a route appears but is not preferred:

- Check whether an intra-area or inter-area route exists for the same prefix.
- Compare E1 versus E2 behavior.
- Check the external metric.
- Check the forwarding address.
- Check administrative distance if multiple protocols are competing.

**NSSA default and translation behavior**

NSSA is the OSPF area type designed for limited external redistribution inside a non-backbone area.

Key points:

- An NSSA can originate external routes as Type 7 LSAs.
- The ABR translates Type 7 LSAs to Type 5 LSAs when they leave the NSSA.
- Translation is typically performed by the elected translator ABR.
- A default route can be injected into an NSSA using NSSA-specific configuration on the ABR or by redistributing a default route from the NSSA ASBR, depending on the design.

Practical exam takeaway:

- If the question says “external routes must be redistributed in a non-backbone area,” the answer is usually **NSSA**, not stub.
- If the question says “no external routes allowed,” think **stub** or **totally stubby** depending on whether inter-area detail is also suppressed.

**Redistribution into OSPF versus route summarization**

Do not confuse redistribution with summarization.

- **Redistribution** imports routes from another protocol or process.
- **Summarization** reduces the number of prefixes advertised within the same protocol or at a boundary.

Use summarization to reduce LSDB size and limit churn. Use redistribution only when you must cross protocol boundaries.

**Exam-focused reminders**

- OSPF external routes are usually Type 5 LSAs; NSSA uses Type 7 internally.
- Stub and totally stubby areas do not carry external LSAs.
- `subnets` matters for IPv4 redistribution into OSPF on IOS-style platforms.
- Route-maps are essential for filtering and tagging redistributed routes.
- E1 and E2 are the key external route types for OSPF path selection.
- Mutual redistribution needs tags and filtering to prevent loops.
- OSPFv3 redistribution is configured in the OSPFv3 process or address-family context, not the classic IPv4-only OSPF mode.

**Quick reference examples**

Redistribute static into OSPF:

```cisco
router ospf 10
 redistribute static subnets metric 50 metric-type 2
```

Redistribute OSPF into BGP:

```cisco
router bgp 65000
 redistribute ospf 10 route-map RM-OSPF-TO-BGP
```

Inject a default route into OSPF:

```cisco
router ospf 10
 default-information originate
```

Redistribute into OSPFv3:

```cisco
router ospfv3 10
 address-family ipv6 unicast
  redistribute static metric 1000 metric-type 1
 exit-address-family
```

**Summary**

Redistribution into and out of OSPF is a core enterprise design skill. The main exam themes are route type behavior, external LSAs, area restrictions, metric control, and loop prevention. If you understand when to use Type 5 versus Type 7, how E1 and E2 differ, and how to verify redistribution with the routing table and LSDB, you can solve most redistribution scenarios confidently.
