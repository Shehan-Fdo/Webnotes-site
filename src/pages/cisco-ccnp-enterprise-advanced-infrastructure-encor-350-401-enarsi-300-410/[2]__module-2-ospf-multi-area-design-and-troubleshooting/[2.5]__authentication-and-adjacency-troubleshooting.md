## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Scope and troubleshooting mindset**

Authentication and adjacency problems often look similar at first: a neighbor does not form, a session flaps, or traffic never reaches the expected control-plane state. In practice, you troubleshoot by separating the failure domain into layers:

- **Link and transport**: physical link, VLAN, trunking, MTU, timers, multicast reachability, TCP reachability.
- **Protocol parameters**: area, AS, K-values, timers, network type, hello/dead intervals, authentication, router ID, capabilities.
- **Policy and control plane**: ACLs, CoPP, AAA, RADIUS/TACACS+, 802.1X, port security, STP, LACP, and platform-specific defaults.

A useful rule is to verify the underlay first, then the protocol-specific neighbor state, then authentication and policy. Many “authentication” issues are actually transport or interface-state issues that prevent the protocol from ever exchanging the packets needed to authenticate.

**Common symptoms and what they usually mean**

- Neighbor never appears:
  - Interface is down, passive, blocked, or in the wrong VLAN/VRF.
  - Multicast or TCP transport is filtered.
  - Authentication or key mismatch prevents packet acceptance.
- Neighbor appears briefly, then drops:
  - Key mismatch, timer mismatch, MTU mismatch, or control-plane policing.
  - For some protocols, the session may establish and then reset when database exchange or TCP negotiation fails.
- Neighbor is stuck in an intermediate state:
  - The protocol is exchanging hellos but not completing database or adjacency negotiation.
  - Common causes include MTU mismatch, authentication mismatch, K-value mismatch, or capability mismatch.

**Core verification workflow**

1. Confirm interface and Layer 2/Layer 3 reachability.
2. Check protocol-specific neighbor state.
3. Compare local and remote parameters.
4. Validate authentication configuration and key lifetimes.
5. Inspect logs, counters, and packet captures if available.
6. Test with a minimal, known-good configuration if the platform allows it.

**Useful baseline commands**

```bash
show ip interface brief
show interfaces status
show interfaces <interface>
show logging
show cdp neighbors detail
show lldp neighbors detail
show ip route
show arp
```

For Layer 2 adjacency problems, also check:

```bash
show etherchannel summary
show lacp neighbor
show spanning-tree interface <interface> detail
show spanning-tree vlan <vlan-id>
show interfaces trunk
```

**OSPFv2 adjacency troubleshooting**

OSPFv2 is one of the most common adjacency troubleshooting topics in CCNP Enterprise. The neighbor state tells you where the exchange is failing, which is often more useful than the final “down” condition.

**Neighbor states and what they imply**

- **Down**:
  - No hello packets received.
  - Wrong area, wrong network type, interface down, multicast blocked, or passive interface.
- **Init**:
  - Hellos are received, but your router ID is not seen in the neighbor’s hello.
  - Often indicates one-way communication or a mismatch in hello processing.
- **2-Way**:
  - Bidirectional hello exchange is working.
  - On broadcast networks, this may be normal for non-designated routers.
  - On point-to-point links, you usually expect progression beyond 2-Way.
- **ExStart**:
  - Routers are negotiating master/slave and starting database description exchange.
  - Common issues: MTU mismatch, duplicate router ID, or adjacency capability mismatch.
- **Exchange**:
  - Database description packets are being exchanged.
  - Stuck here often points to MTU mismatch, authentication mismatch, or packet loss.
- **Loading**:
  - Routers are requesting missing LSAs.
  - Problems here often involve database inconsistency, filtering, or packet loss.
- **Full**:
  - Adjacency is complete.

**OSPFv2 authentication behavior**

OSPFv2 authentication is platform and configuration dependent. On Cisco IOS and IOS XE, you may see interface authentication, area authentication, or key-chain-based authentication. Do not assume one syntax fits every release.

Common patterns include:

```bash
router ospf 10
 area 0 authentication message-digest
!
interface GigabitEthernet0/0
 ip ospf authentication message-digest
 ip ospf message-digest-key 1 md5 cisco123
```

Or key-chain-based authentication on supported platforms:

```bash
key chain OSPF-KEYS
 key 1
  key-string cisco123
!
interface GigabitEthernet0/0
 ip ospf authentication key-chain OSPF-KEYS
```

Troubleshooting points:

- Both sides must match on authentication type and key material.
- Area-level and interface-level settings must be consistent.
- Key-chain send/accept lifetimes must overlap during rotation.
- Authentication mismatch usually prevents stable adjacency formation; do not expect a healthy Full state if the keys do not match.

> If the neighbor reaches ExStart or Exchange and then drops, check MTU, authentication, and duplicate router ID before assuming a routing bug.

**OSPFv2 verification commands**

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip ospf interface GigabitEthernet0/0
show ip ospf database
show ip ospf statistics
show logging | include OSPF
```

What to look for:

- Neighbor state transitions and how often they flap.
- Hello and dead timer values.
- Area ID, network type, and authentication type.
- MTU reported on the interface.
- Duplicate router ID warnings.
- Log messages about adjacency changes or authentication failures.

**OSPFv2 common failure causes**

- **Hello/dead timer mismatch**:
  - Hellos are received, but adjacency does not progress.
- **Area mismatch**:
  - Neighbors never form.
- **Authentication mismatch**:
  - Hellos may be seen, but adjacency does not complete.
- **MTU mismatch**:
  - Often causes ExStart or Exchange stalls.
- **Passive interface**:
  - No hellos are sent.
- **Stub/NSSA mismatch**:
  - Adjacency may fail or remain unstable.
- **Duplicate router ID**:
  - Can cause strange neighbor behavior and database exchange issues.

**EIGRP adjacency troubleshooting**

EIGRP uses hello packets to discover neighbors and then forms adjacencies based on matching parameters. Classic EIGRP and named EIGRP use different configuration styles, so verify the syntax on the platform you are studying.

**What must match**

- Autonomous system number in classic EIGRP.
- Address-family and AS context in named EIGRP.
- K-values.
- Authentication settings.
- Hello and hold timers, where applicable.
- Primary interface reachability and multicast handling.

**Classic EIGRP example**

```bash
router eigrp 100
 network 10.1.1.0 0.0.0.255
 passive-interface default
 no passive-interface GigabitEthernet0/0
```

**Named EIGRP example**

```bash
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  network 10.1.1.0 0.0.0.255
  af-interface default
   passive-interface
  exit-af-interface
  af-interface GigabitEthernet0/0
   no passive-interface
  exit-af-interface
 exit-address-family
```

**EIGRP authentication**

On many IOS and IOS XE platforms, EIGRP authentication is configured per interface and must match on both sides. Named EIGRP commonly places interface behavior under the address-family interface context, so do not apply classic syntax blindly.

Representative classic-style example:

```bash
key chain EIGRP-KEYS
 key 1
  key-string cisco123
!
interface GigabitEthernet0/0
 ip authentication mode eigrp 100 md5
 ip authentication key-chain eigrp 100 EIGRP-KEYS
```

Named EIGRP may use address-family interface configuration depending on release:

```bash
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  af-interface GigabitEthernet0/0
   authentication mode md5
   authentication key-chain EIGRP-KEYS
  exit-af-interface
 exit-address-family
```

Troubleshooting points:

- Authentication must match on both peers.
- Key-chain lifetimes must overlap.
- K-values must match exactly; even one mismatch prevents adjacency.
- Passive interfaces suppress neighbor formation.
- Unequal hello/hold timers can cause instability on some designs, though EIGRP is more tolerant than OSPF in many cases.

**EIGRP verification commands**

```bash
show ip eigrp neighbors
show ip eigrp interfaces detail
show ip eigrp topology
show key chain
show running-config | section eigrp
show logging | include EIGRP
```

What to look for:

- Neighbor uptime and SRTT/RTO values.
- K-values in the topology and interface output.
- Authentication mode and key-chain association.
- Whether the interface is passive.
- Whether the neighbor is stuck in a reset loop.

**EIGRP failure indicators**

- **No neighbor entry**:
  - AS mismatch, passive interface, ACL, multicast blocked, or interface not participating.
- **Neighbor appears and disappears**:
  - Authentication mismatch, key lifetime mismatch, or transport instability.
- **Topology not installed**:
  - Feasible distance and reported distance issues, not adjacency failure, but often confused with it.
- **Stuck in active**:
  - Not an adjacency problem, but important in troubleshooting because it can look like a neighbor issue.

**K-value mismatch example**

```bash
show ip eigrp neighbors
show ip eigrp topology all-links
```

If K-values differ, the neighbor relationship does not form. Check both sides:

```bash
show ip protocols
```

**IS-IS adjacency troubleshooting**

IS-IS is especially important in enterprise and service-provider-style designs. Its troubleshooting is often more nuanced because adjacency depends on circuit type, level, and authentication behavior.

**Key IS-IS concepts**

- **Level 1**:
  - Intra-area adjacency and routing.
- **Level 2**:
  - Inter-area or backbone adjacency and routing.
- **Level 1-2**:
  - Participates in both levels.
- **Circuit type**:
  - Must align with the intended design and neighbor capability.
- **Network entity title (NET)**:
  - Must be valid and consistent with the design.
- **Authentication**:
  - Can be applied per interface and sometimes per level, depending on platform and release.

**IS-IS adjacency states**

- **Down**:
  - No IIH (IS-IS Hello) packets received.
- **Init**:
  - Hellos are seen, but adjacency is not yet bidirectional.
- **Up**:
  - Adjacency is established for the relevant level.

**Common IS-IS failure causes**

- Mismatched level capability.
- Mismatched area address or NET.
- Authentication mismatch.
- Interface not enabled for IS-IS.
- MTU mismatch.
- L2-only versus L1-only design mismatch.
- Passive or suppressed interface behavior.
- Circuit type mismatch on some platforms.

**IS-IS authentication note**

IS-IS authentication syntax varies by platform and release. Use the platform’s supported interface or level-specific commands rather than assuming a single universal form. For exam purposes, focus on the concept:

- Both neighbors must use compatible authentication.
- Authentication may be applied per interface, per level, or per circuit depending on implementation.
- A mismatch usually prevents adjacency from reaching Up.

**IS-IS verification commands**

```bash
show isis neighbors
show isis database
show isis interface
show isis adjacency
show clns interface
show clns neighbors
show logging | include ISIS
```

What to look for:

- Level 1, Level 2, or Level 1-2 adjacency state.
- Circuit type and interface MTU.
- Authentication-related log messages.
- Whether the neighbor is forming at one level but not the other.

**IS-IS troubleshooting example**

If a router forms Level 1 adjacency but not Level 2, check:

- Whether both routers are configured for Level 2 on that link.
- Whether the interface is intended to carry L2.
- Whether the NET and area design support the expected level.
- Whether authentication is applied consistently for the level in question.

**BGP adjacency and session troubleshooting**

BGP is a TCP-based protocol, so adjacency troubleshooting starts with TCP reachability and then moves to BGP parameters. Authentication problems often appear as TCP session failures rather than a simple neighbor state issue.

**BGP session prerequisites**

- TCP port 179 must be reachable.
- The neighbor IP must be reachable in the routing table.
- The source address must be stable and expected.
- TTL security, multihop, and authentication must be compatible.
- The remote AS and local AS must be correct.

**BGP authentication**

Cisco platforms commonly support TCP MD5 for BGP authentication, and some platforms or releases may support TCP Authentication Option (TCP-AO). Do not assume TCP-AO is universally available.

Representative TCP MD5 example:

```bash
router bgp 65001
 neighbor 192.0.2.2 remote-as 65002
 neighbor 192.0.2.2 password cisco123
```

Troubleshooting points:

- Passwords must match exactly.
- Authentication is applied to the TCP session, so a mismatch often prevents the session from establishing.
- If the session is already up and the password changes, the TCP connection may reset.
- TTL security and multihop settings must align with the topology.

**BGP verification commands**

```bash
show ip bgp summary
show ip bgp neighbors 192.0.2.2
show tcp brief
show tcp brief all
show logging | include BGP
```

Use these carefully:

- `show ip bgp summary` is the primary quick check for session state.
- `show ip bgp neighbors <neighbor>` gives detailed timers, capabilities, and last reset reason.
- `show tcp brief` helps confirm whether TCP 179 is established, but output varies by platform.
- `show tcp brief all` can be useful, but do not rely on it as the only BGP check.

**BGP debug commands**

Use debugs carefully in production:

```bash
debug ip bgp
debug bgp events
debug bgp updates
```

Avoid assuming a generic `debug bgp tcp` command exists on all IOS or IOS XE releases.

**BGP failure symptoms**

- **Idle or Active**:
  - TCP session not established, routing unreachable, ACL issue, or authentication mismatch.
- **Established then reset**:
  - Password mismatch, TTL security mismatch, multihop mismatch, or policy reset.
- **Neighbor stuck in OpenSent/OpenConfirm**:
  - Capability mismatch, AS mismatch, or transport instability.

**BGP and transport security interactions**

- **TTL security**:
  - Common with eBGP directly connected peers.
  - If enabled on one side only, the session may fail.
- **ebgp-multihop**:
  - Required for non-direct peers.
  - Must be consistent with the actual path and TTL expectations.
- **Authentication**:
  - TCP MD5/TCP-AO must match on both ends.
  - A route may exist, but the session still fails if the TCP parameters do not align.

**AAA, TACACS+, and RADIUS troubleshooting**

AAA (Authentication, Authorization, and Accounting) affects device access and can also influence management-plane reachability. When AAA fails, you may lose CLI access, privilege escalation, or authorization even if the network is otherwise healthy.

**AAA components**

- **Authentication**: who you are.
- **Authorization**: what you can do.
- **Accounting**: what you did.

**TACACS+ versus RADIUS**

| Feature | TACACS+ | RADIUS |
|---|---|---|
| Transport | TCP | UDP |
| Separation of auth/authorization | Yes | Limited |
| Common use | Device administration | Network access, 802.1X |
| Encryption | Entire payload encrypted except header | Password field protected; broader payload less protected |

**Modern TACACS+ example**

```bash
aaa new-model
tacacs server TAC1
 address ipv4 192.0.2.10
 key 0 cisco123
!
aaa group server tacacs+ TAC-GRP
 server name TAC1
!
ip tacacs source-interface Loopback0
aaa authentication login default group TAC-GRP local
aaa authorization exec default group TAC-GRP local
```

Troubleshooting points:

- Use the correct source-interface command for the platform.
- Ensure the source IP is reachable from the server.
- Confirm the server key matches exactly.
- Verify fallback to `local` or another method if the server is unavailable.
- Distinguish authentication failure from authorization failure.

**RADIUS example**

```bash
radius server RAD1
 address ipv4 192.0.2.20 auth-port 1812 acct-port 1813
 key 0 cisco123
!
aaa group server radius RAD-GRP
 server name RAD1
!
aaa authentication dot1x default group RAD-GRP local
aaa authorization network default group RAD-GRP
```

**AAA verification commands**

```bash
show aaa servers
show running-config | section aaa
show tacacs
show radius statistics
show logging | include AAA
test aaa group tacacs+ admin password legacy
```

What to look for:

- Server reachability and response time.
- Source-interface selection.
- Method-list order and fallback behavior.
- Whether authentication succeeds but authorization fails.
- Whether the device is using the expected server group.

**AAA failure patterns**

- **Server unreachable**:
  - Routing, ACL, VRF, or source-interface issue.
- **Authentication succeeds, authorization fails**:
  - Server policy or privilege mapping issue.
- **Fallback to local unexpectedly**:
  - Server timeout, deadtime, or group misconfiguration.
- **Login loop or lockout**:
  - Method-list misconfiguration or missing local user.

**802.1X and MAB troubleshooting**

802.1X and MAB (MAC Authentication Bypass) are common access-layer authentication topics. They affect whether a host gets network access, which makes them adjacency-like from an operational perspective because the port may remain unauthorized even though the physical link is up.

**802.1X flow**

- Supplicant connects to the switch port.
- The switch sends EAPOL (Extensible Authentication Protocol over LAN) messages.
- The switch relays authentication to a RADIUS server.
- The port becomes authorized if authentication succeeds.

**MAB flow**

- The switch uses the client MAC address as the identity.
- The RADIUS server accepts or rejects the MAC.
- MAB is often used as a fallback for devices without 802.1X supplicants.

**Common failure causes**

- Supplicant not running or not configured.
- EAPOL blocked or not exchanged correctly.
- RADIUS server timeout or unreachable server.
- Incorrect shared secret.
- Wrong identity, certificate, or supplicant policy.
- Guest VLAN or critical VLAN behavior not matching expectations.
- Port remains unauthorized because the server rejects the session.

**802.1X verification commands**

```bash
show authentication sessions
show authentication sessions interface GigabitEthernet1/0/10 details
show dot1x all
show dot1x interface GigabitEthernet1/0/10 details
show radius statistics
show logging | include DOT1X|AUTHMGR|RADIUS
```

What to look for:

- Session state: unauthorized, authorized, or server dead.
- Method used: dot1x or mab.
- Server response timeouts.
- Reauth timers and periodic reauthentication.
- VLAN assignment or downloadable ACL behavior.

**802.1X and MAB practical notes**

- If 802.1X fails, MAB may be the fallback depending on policy.
- If both fail, the port may enter a guest VLAN or remain blocked.
- Critical auth behavior depends on platform and policy design.
- A host can appear “connected” at Layer 1 while still being denied access at Layer 2.

**LACP, EtherChannel, and STP-related adjacency issues**

Not all adjacency problems are routing-protocol problems. Layer 2 control protocols can break the path that routing neighbors need.

**LACP and EtherChannel**

If one side of a port-channel is misconfigured, the link may be physically up but logically unusable.

Common causes:

- Channel-group mode mismatch.
- Speed/duplex or trunk mismatch.
- Allowed VLAN mismatch.
- LACP active/passive mismatch in some designs.
- One member suspended due to inconsistent parameters.

Verification:

```bash
show etherchannel summary
show lacp neighbor
show interfaces port-channel 1
show interfaces GigabitEthernet1/0/1 etherchannel
```

What to look for:

- Member ports in `P` or suspended states.
- Port-channel protocol status.
- Consistency of trunking and VLAN settings.
- Whether the logical port-channel is the one carrying the adjacency traffic.

**STP impact on adjacency**

Spanning Tree Protocol (STP) can block a port or delay forwarding, which prevents neighbor packets from passing.

Common symptoms:

- The interface is up, but no neighbor forms.
- One side sees hellos; the other side does not.
- The port is in blocking or listening/learning state.
- BPDU Guard, Root Guard, or Loop Guard changes forwarding behavior.

Verification:

```bash
show spanning-tree interface GigabitEthernet1/0/10 detail
show spanning-tree vlan 10
show errdisable recovery
show logging | include STP|BPDU|ERRDIS
```

Troubleshooting points:

- Confirm the port is forwarding for the VLAN carrying the adjacency.
- Check for err-disabled state.
- Verify trunk allowed VLANs and native VLAN consistency.
- Confirm that STP protection features are not intentionally blocking the port.

**Practical troubleshooting scenarios**

**Scenario 1: OSPF neighbor stuck in ExStart**

Likely causes:

- MTU mismatch.
- Duplicate router ID.
- Authentication mismatch.
- Packet loss or control-plane filtering.

Decision points:

1. Check `show ip ospf neighbor`.
2. Compare `show ip ospf interface` on both sides.
3. Verify MTU and authentication.
4. Check logs for adjacency resets.

**Scenario 2: EIGRP neighbor never appears**

Likely causes:

- AS mismatch.
- Passive interface.
- K-value mismatch.
- ACL or multicast issue.
- Authentication mismatch.

Decision points:

1. Check `show ip eigrp neighbors`.
2. Verify `show ip protocols`.
3. Confirm interface participation and passive status.
4. Compare key-chain and authentication settings.

**Scenario 3: BGP session stays Idle**

Likely causes:

- No route to neighbor.
- ACL blocking TCP 179.
- Wrong source address.
- Authentication mismatch.
- TTL security or multihop mismatch.

Decision points:

1. Check reachability with `ping` and routing table.
2. Check `show ip bgp summary`.
3. Inspect `show ip bgp neighbors <neighbor>`.
4. Verify TCP session and logs.

**Scenario 4: 802.1X port remains unauthorized**

Likely causes:

- RADIUS timeout.
- Wrong shared secret.
- Supplicant failure.
- Policy mismatch.
- Fallback to guest or critical VLAN.

Decision points:

1. Check `show authentication sessions`.
2. Verify RADIUS server reachability.
3. Inspect `show logging`.
4. Confirm the expected method order and VLAN policy.

**Scenario 5: EtherChannel forms, but routing neighbors do not**

Likely causes:

- VLAN not allowed on the port-channel.
- STP blocking the VLAN.
- One member inconsistent, causing traffic imbalance or drops.
- Neighbor is on a different logical interface than expected.

Decision points:

1. Check `show etherchannel summary`.
2. Check `show interfaces trunk`.
3. Check STP state for the VLAN.
4. Verify the routing protocol is enabled on the correct SVI or routed port.

**Troubleshooting checklist by protocol**

- **OSPFv2**:
  - Area, timers, MTU, authentication, network type, router ID.
- **EIGRP**:
  - AS, K-values, passive-interface, authentication, named versus classic syntax.
- **IS-IS**:
  - Level, NET, circuit type, MTU, authentication, interface enablement.
- **BGP**:
  - TCP reachability, AS numbers, source address, password, TTL security, multihop.
- **AAA**:
  - Server reachability, source-interface, keys, method lists, fallback.
- **802.1X/MAB**:
  - Supplicant, EAPOL, RADIUS, VLAN assignment, fallback policy.
- **LACP/STP**:
  - Port-channel consistency, forwarding state, errdisable, VLAN transport.

**High-value exam habits**

- Compare both sides of the link, not just one device.
- Read the neighbor state before changing configuration.
- Verify the transport path before blaming authentication.
- Use logs to confirm whether the failure is authentication, transport, or policy.
- Remember that syntax varies by platform and release, especially for OSPF, EIGRP named mode, IS-IS, and AAA.

**Quick command reference**

```bash
show ip ospf neighbor
show ip eigrp neighbors
show isis neighbors
show ip bgp summary
show authentication sessions
show etherchannel summary
show spanning-tree interface <interface> detail

```

**When to use packet captures or debugs**

Use packet captures or debugs when show commands do not explain the failure:

- Capture hellos, EAPOL, or TCP session setup.
- Confirm whether packets leave one side and arrive at the other.
- Use protocol debugs sparingly and only on controlled devices.

Examples:

```bash
debug ip ospf adj
debug ip eigrp packets
debug isis adj-packets
debug ip bgp
debug aaa authentication
debug dot1x all
```

> Disable debugs after use to avoid CPU impact and log flooding.

**Summary of what matters most**

Authentication and adjacency troubleshooting is about matching the protocol, the transport, and the policy. If the neighbor does not form, first prove that packets can flow, then prove that both sides agree on the protocol parameters, and finally prove that authentication and control-plane policy allow the session to complete. The most common exam mistakes come from assuming a single syntax or from skipping transport checks before diagnosing authentication.
