## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Route tagging fundamentals**

Route tags are 32-bit values carried in routing information to help you identify where a route came from and how it should be treated during redistribution. They are not a routing metric and do not influence best-path selection inside a protocol. Instead, you use them as policy markers to prevent a route from being sent back into the domain it came from.

In Cisco designs, tags are most useful when a route crosses protocol boundaries, such as OSPF to EIGRP, EIGRP to OSPF, static to OSPF, or OSPF NSSA translation. The tag survives redistribution if the receiving protocol supports carrying it, which lets you match on that tag later and block re-entry.

- **Purpose**: identify redistributed routes and stop feedback loops.
- **Scope**: commonly used on external routes, but can be applied wherever the platform and protocol support route attributes.
- **Behavior**: tags are matched in policy; they are not used by the routing decision process itself.

**Why loops happen during redistribution**

Redistribution creates a second copy of a route in another routing domain. If that route is redistributed back without a guardrail, the original domain may learn its own route as a new external route. That can create suboptimal paths, route churn, or persistent loops.

A classic failure pattern looks like this:

- OSPF learns a prefix from EIGRP through redistribution.
- OSPF redistributes that prefix back into EIGRP.
- EIGRP sees the prefix as external and may advertise it back toward the original boundary.
- Without filtering or tagging, the route can circulate between domains.

Route tags solve this by marking the route at the first redistribution point and denying it on the return path.

**How Cisco route tags work**

On Cisco IOS, IOS XE, and IOS XR, tagging is usually done in a policy object applied during redistribution. The exact syntax differs by platform, but the concept is the same: match a route and set a tag before it enters the new domain.

Common Cisco-style patterns include:

- **IOS / IOS XE**: set the tag in a `route-map` used under the redistribution command.
- **IOS XR**: set the tag in a `route-policy` attached to redistribution.
- **Protocol-specific redistribution**: some protocols let you attach a route-map directly under `redistribute`, while others use a policy statement with `set tag`.

Example on IOS / IOS XE:

```cisco
route-map TAG-TO-OSPF permit 10
 set tag 100

router ospf 1
 redistribute eigrp 10 subnets route-map TAG-TO-OSPF
```

Example on IOS XR:

```cisco
route-policy TAG-TO-OSPF
  set tag 100
  pass
end-policy

router ospf 1
 redistribute eigrp 10 route-policy TAG-TO-OSPF
```

The important exam point is that the tag is applied during redistribution, not by the routing protocol’s path selection logic.

**End-to-end redistribution flow**

A practical loop-avoidance design uses one tag per redistribution direction or per source domain. The tag is set when the route leaves its source domain and matched when it tries to return.

Example topology:

- R1 runs EIGRP on the left.
- R2 sits at the boundary and redistributes between EIGRP and OSPF.
- R3 runs OSPF on the right.

Flow:

1. R1 advertises `10.10.10.0/24` into EIGRP.
2. R2 redistributes that prefix into OSPF and sets tag `100`.
3. R3 learns the route in OSPF.
4. If R3 or another OSPF boundary redistributes back toward EIGRP, the route-map checks for tag `100`.
5. The route is denied on re-entry, so the original EIGRP domain does not learn its own route as a new external.

A simple policy sketch:

```cisco
route-map EIGRP-TO-OSPF permit 10
 set tag 100

route-map OSPF-TO-EIGRP deny 10
 match tag 100

route-map OSPF-TO-EIGRP permit 20
 set tag 200

router ospf 1
 redistribute eigrp 10 subnets route-map EIGRP-TO-OSPF

router eigrp 10
 redistribute ospf 1 route-map OSPF-TO-EIGRP
```

This pattern means:

- Routes already tagged `100` are blocked from going back into EIGRP.
- Other OSPF routes can still be redistributed, and you can optionally retag them as they cross.

> A route-map sequence with no `match` statements matches all routes in that sequence. Do not assume it matches nothing. If you need to block a specific tag, use an explicit `deny` sequence with `match tag`.

**Cisco route-map and tag syntax**

Route tags are typically set with `set tag <value>` inside a route-map or route-policy. Matching uses `match tag <value>`.

IOS / IOS XE example:

```cisco
route-map BLOCK-RETURN deny 10
 match tag 100

route-map BLOCK-RETURN permit 20
 set tag 200
```

Key behaviors:

- `match tag 100` selects routes that already carry tag `100`.
- `set tag 200` writes a new tag onto routes that pass the sequence.
- A `deny` sequence prevents matched routes from being redistributed.
- A `permit` sequence allows matched routes and can also modify attributes.

For redistribution, the route-map is attached to the protocol redistribution command, for example:

```cisco
router ospf 1
 redistribute eigrp 10 subnets route-map BLOCK-RETURN
```

or

```cisco
router eigrp 10
 redistribute ospf 1 route-map BLOCK-RETURN
```

**OSPF-specific considerations**

OSPF external routes can carry a tag in the external LSA (Link-State Advertisement). This is especially useful when routes are redistributed into OSPF and later redistributed out again.

Important OSPF points:

- **External Type 5 LSAs** can carry a tag.
- **NSSA Type 7 LSAs** can also carry a tag.
- During **NSSA translation**, the tag is generally preserved when a Type 7 LSA is translated to Type 5 by the ABR (Area Border Router), which is why tags are useful for loop prevention across OSPF areas and redistribution boundaries.
- Tags are not used by OSPF SPF (Shortest Path First) calculation; they are policy metadata.

Example redistribution into OSPF:

```cisco
route-map TO-OSPF permit 10
 set tag 110

router ospf 1
 redistribute eigrp 10 subnets route-map TO-OSPF
```

If you later redistribute OSPF back into another protocol, you can block the route based on tag:

```cisco
route-map FROM-OSPF deny 10
 match tag 110

route-map FROM-OSPF permit 20

router eigrp 10
 redistribute ospf 1 route-map FROM-OSPF
```

For exam prep, remember that OSPF tags are most visible in external route handling and redistribution policy, not in normal intra-area route selection.

**EIGRP-specific considerations**

EIGRP can carry route tags on redistributed routes and can match them in route-maps. This makes EIGRP a common place to enforce loop prevention at redistribution boundaries.

Typical use cases:

- Block routes that originated from OSPF from being redistributed back into OSPF.
- Mark static routes before injecting them into EIGRP.
- Distinguish between multiple redistribution sources entering the same EIGRP domain.

Example:

```cisco
route-map STATIC-TO-EIGRP permit 10
 set tag 300

router eigrp 10
 redistribute static route-map STATIC-TO-EIGRP
```

Then prevent return into OSPF:

```cisco
route-map EIGRP-TO-OSPF deny 10
 match tag 300

route-map EIGRP-TO-OSPF permit 20

router ospf 1
 redistribute eigrp 10 subnets route-map EIGRP-TO-OSPF
```

**BGP and route tags**

Route tags are not native BGP path attributes. BGP uses attributes such as:

- **AS_PATH**
- **LOCAL_PREF**
- **MED**
- **COMMUNITY**
- **WEIGHT**

Route tags can still matter at redistribution boundaries involving BGP, but only as carried route metadata during redistribution into or out of BGP. They are not part of BGP best-path selection.

Use this mental model:

- Inside BGP, prefer BGP attributes and communities for policy.
- At redistribution boundaries, route tags can help prevent routes from being re-injected into the wrong domain.

Example:

```cisco
route-map TO-BGP permit 10
 set tag 400

router bgp 65000
 redistribute ospf 1 route-map TO-BGP
```

If the route later leaves BGP and returns to OSPF or another protocol, you can match the tag in the redistribution policy. Do not describe tags as a BGP-native control mechanism.

**Static routes and tagging**

Static routes can also be tagged before redistribution. This is useful when a static route represents a summary, a default route, or a controlled leak into a dynamic protocol.

Example:

```cisco
ip route 192.0.2.0 255.255.255.0 Null0 tag 500
```

Then redistribute with a route-map that preserves or filters the tag:

```cisco
route-map STATIC-INTO-OSPF permit 10
 match tag 500
 set tag 500

router ospf 1
 redistribute static subnets route-map STATIC-INTO-OSPF
```

This is useful when you want to distinguish intentional static injections from other static routes.

**Loop-avoidance design patterns**

Route tags are one tool in a broader redistribution strategy. In exam scenarios, compare them with other controls rather than treating them as a standalone fix.

Common methods:

| Method | What it does | Best use |
|---|---|---|
| Route tags | Marks routes for later matching | Redistribution loop prevention |
| Route-maps | Filter or modify redistributed routes | Policy control at boundaries |
| Prefix-lists | Match prefixes precisely | Route selection and filtering |
| Distribute-lists | Filter routes by ACL or prefix-list depending on protocol | Basic inbound/outbound filtering |
| Route-policy | Policy engine on IOS XR and some platforms | Rich redistribution control |
| Administrative distance | Influences route preference | Backup path selection, not loop prevention |

Practical guidance:

- Use **tags** when the same prefix can re-enter from another domain and you need to recognize its origin.
- Use **prefix-lists** when you only need to permit or deny specific networks.
- Use **distribute-lists** when protocol support is limited and simple filtering is enough.
- Use **route-policy** on IOS XR for more expressive matching and setting.
- Use **administrative distance** only to prefer one route source over another; it does not stop redistribution loops.

**NSSA translation and tag preservation**

OSPF Not-So-Stubby Area (NSSA) translation is a common exam topic because it involves route transformation. A Type 7 external LSA inside an NSSA is translated to a Type 5 external LSA by the ABR when it leaves the NSSA.

Key point:

- The route tag is generally preserved through Type 7 to Type 5 translation.
- This allows downstream redistribution policies to recognize the route’s origin even after it changes LSA type.

Why this matters:

- A route injected into an NSSA can later appear as a Type 5 external outside the NSSA.
- If that route is redistributed into another protocol and then back toward OSPF, the preserved tag can still block the loop.

For exam accuracy, focus on the concept that translation changes the LSA type, not the policy metadata used to identify the route.

**Verification and troubleshooting**

Verification is usually done by checking the routing table and the protocol database, then confirming that the redistribution policy is applied.

Useful commands:

```cisco
show ip route
show ip route ospf
show ip route eigrp
show ip ospf database external
show ip ospf database nssa-external
show route-map
show ip protocols
```

What to look for:

- In `show ip route`, external routes often show the protocol source and may display a tag depending on platform and route type.
- In `show ip ospf database external`, check the external LSA details for the tag field.
- In `show ip ospf database nssa-external`, verify the tag on Type 7 LSAs.
- In `show route-map`, confirm sequence order, match conditions, and set actions.
- In `show ip protocols`, confirm that redistribution is enabled and the correct route-map is attached.

Example interpretation:

```cisco
R1# show ip route
O E2 10.10.10.0/24 [110/20] via 192.0.2.2, 00:00:12, GigabitEthernet0/0
```

This tells you:

- `O E2` means OSPF external type 2.
- The route is external, so it is a candidate for redistribution policy checks.
- The tag may not appear in this line on every platform, so inspect the OSPF database if you need the exact tag value.

Example OSPF database check:

```cisco
R1# show ip ospf database external

            OSPF Router with ID (1.1.1.1) (Process ID 1)

                Type-5 AS External Link States

  LS age: 123
  Options: (No TOS-capability, DC)
  LS Type: AS External Link
  Link State ID: 10.10.10.0 (External Network Number )
  Advertising Router: 2.2.2.2
  Network Mask: 255.255.255.0
  Metric Type: 2
  TOS: 0
  Metric: 20
  Forward Address: 0.0.0.0
  External Route Tag: 100
```

The `External Route Tag` field is the key value you use to confirm loop-prevention policy.

> Do not rely on a single platform-specific `show ip route tag ...` filter as a universal verification method. Use the routing table plus the protocol database for reliable validation.

**Common exam traps**

- **Confusing tags with metrics**: tags identify routes; metrics rank paths.
- **Assuming tags affect best path**: they do not.
- **Forgetting implicit route-map behavior**: a sequence with no `match` matches all routes.
- **Using BGP terminology incorrectly**: tags are not BGP path attributes.
- **Ignoring translation boundaries**: OSPF NSSA translation can preserve tags, which matters for loop prevention.
- **Using only one filter type**: tags are often paired with prefix-lists or route-maps for stronger control.

**Practical design example**

A clean redistribution design between OSPF and EIGRP can use one tag per direction:

```cisco
route-map OSPF-TO-EIGRP permit 10
 set tag 10

route-map EIGRP-TO-OSPF deny 10
 match tag 10

route-map EIGRP-TO-OSPF permit 20
 set tag 20

router eigrp 10
 redistribute ospf 1 route-map OSPF-TO-EIGRP

router ospf 1
 redistribute eigrp 10 subnets route-map EIGRP-TO-OSPF
```

How this works:

- OSPF routes entering EIGRP get tag `10`.
- If those routes try to return to OSPF, the deny sequence blocks them.
- Other EIGRP routes can still enter OSPF and receive tag `20`.

This pattern is simple, scalable, and easy to troubleshoot because each direction has a distinct tag.

**Quick study checklist**

- **Know** that route tags are redistribution metadata, not path-selection attributes.
- **Remember** that `set tag` is applied in route-maps or route-policies during redistribution.
- **Use** `match tag` to block routes from re-entering their source domain.
- **Verify** tags in protocol databases, especially OSPF external and NSSA external LSAs.
- **Distinguish** tags from BGP attributes, metrics, and administrative distance.
- **Understand** that NSSA translation preserves the route’s identity enough for tag-based loop prevention.
