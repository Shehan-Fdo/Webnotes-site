## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Redistribution fundamentals**

Route redistribution injects routes learned from one routing domain into another. You use it when two protocols must exchange reachability, such as an EIGRP core with an OSPF edge, or when BGP connects an enterprise to an internal IGP. Redistribution is powerful, but it breaks the clean separation between routing domains, so you must control metrics, route tags, and filtering carefully.

A redistributed route becomes a route learned from the receiving protocol, not a native route. That matters because the receiving protocol applies its own selection rules, administrative distance, and route advertisement behavior. In exam scenarios, always identify:
- The source protocol
- The destination protocol
- Whether redistribution is one-way or mutual
- How loops are prevented
- Which routes are intentionally excluded

**Key terms and concepts**

<dl>
<dt><strong>Source protocol</strong></dt>
<dd>The protocol that originally learned the route.</dd>
<dt><strong>Destination protocol</strong></dt>
<dd>The protocol that receives the redistributed route.</dd>
<dt><strong>Seed metric</strong></dt>
<dd>The initial metric assigned to a redistributed route so the destination protocol can install it.</dd>
<dt><strong>Route tag</strong></dt>
<dd>A value carried with a route to help prevent re-redistribution loops.</dd>
<dt><strong>Administrative distance</strong></dt>
<dd>The preference used when multiple routing sources advertise the same prefix.</dd>
</dl>

**Why redistribution is tricky**

Each protocol uses different route attributes:
- EIGRP uses composite metrics based on bandwidth and delay, with optional reliability and load.
- OSPF uses cost and external route types.
- BGP uses path attributes such as AS_PATH, local preference, MED, and next hop.

When you redistribute, some attributes are translated, some are approximated, and some are lost. That is why redistribution is often a policy problem, not just a configuration task.

**Protocol behavior at a glance**

| Source | Destination | What carries over well | What is lost or changed |
|---|---|---|---|
| EIGRP | OSPF | Prefix and seed metric | EIGRP composite metric details, variance behavior |
| OSPF | EIGRP | Prefix and seed metric | OSPF area/type context, LSA details |
| BGP | OSPF | Prefix reachability | BGP path attributes such as AS_PATH, local preference, communities |
| BGP | EIGRP | Prefix reachability | BGP policy attributes and path selection context |
| OSPF | BGP | Prefix reachability | OSPF cost and LSA type semantics |
| EIGRP | BGP | Prefix reachability | EIGRP metric semantics and topology context |

**Redistribution prerequisites**

Before a route can be redistributed, it must exist in the routing table or be otherwise eligible for origination by the source protocol. In practice:
- The source route must be active and reachable.
- The destination protocol must have a valid seed metric or default metric policy.
- The route must not be blocked by route-map, prefix-list, distribute-list, or area-type restrictions.
- The receiving protocol must be able to install the route based on administrative distance and next-hop resolution.

> A common exam trap is assuming a route is missing because redistribution is broken, when the real issue is that the source route is not present in the routing table or the destination protocol has no usable metric.

**EIGRP redistribution**

EIGRP requires a metric for redistributed routes. If you do not specify one, the route is not installed. You can set the metric directly under the redistribution command or define a `default-metric`.

Typical EIGRP seed metric components:
- Bandwidth
- Delay
- Reliability
- Load
- MTU

In most designs, bandwidth and delay are the important values. Reliability, load, and MTU are usually left at defaults unless you have a specific reason to tune them.

```cisco
router eigrp 100
 network 10.0.0.0 0.255.255.255
 redistribute ospf 1 metric 100000 100 255 1 1500
```

You can also define a default metric once and reuse it:
```cisco
router eigrp 100
 default-metric 100000 100 255 1 1500
 redistribute ospf 1
 redistribute bgp 65000
```

**EIGRP external routes**

Redistributed routes appear as external EIGRP routes:
- `D EX` for external EIGRP routes
- `D EX` routes are less preferred than internal EIGRP routes
- External routes may be filtered or tagged to prevent loops

EIGRP also supports route tags, which are useful when routes leave one domain and later re-enter through another redistribution point.

**OSPF redistribution**

OSPF requires an external metric and an external type. You normally choose between:
- **E1**: External cost plus internal OSPF cost to the ASBR
- **E2**: External cost only, which is the default in many Cisco examples

```cisco
router ospf 1
 redistribute eigrp 100 subnets metric 20 metric-type 1
 redistribute bgp 65000 subnets metric 50 metric-type 2
```

The `subnets` keyword is important because it allows redistribution of non-classful prefixes. Without it, you may miss many modern prefixes.

**OSPF external route types**

In the routing table, OSPF external routes appear as:
- `O E1`
- `O E2`
- `O N1`
- `O N2`

The LSA type in the database is different from the route code:
- Type 5 LSAs are used for external routes in normal areas
- Type 7 LSAs are used in NSSA areas and are translated by an ABR into Type 5 when appropriate

**How OSPF chooses external routes**

OSPF external route selection is nuanced:
- E1 is preferred over E2 when both exist for the same prefix.
- For E2 routes, the external metric is the primary value; internal cost to the ASBR is not added in the same way.
- For E1 routes, the total path cost includes the internal OSPF cost to reach the ASBR plus the external metric.
- In NSSA environments, `N1` and `N2` behave similarly to `E1` and `E2`, but they originate from Type 7 LSAs before translation.

If multiple ASBRs advertise the same external prefix, the best path depends on route type, metric, and tie-break behavior. Do not reduce this to a simple “lowest external cost wins” rule in all cases.

**BGP redistribution**

BGP redistribution is different from IGP redistribution because BGP is policy-driven and attribute-rich. When you redistribute into BGP, you are not assigning an IGP-style metric in the same way you would for OSPF or EIGRP. Instead, you typically control:
- Which prefixes are injected
- Which next hop is used
- Which BGP attributes are set by route-map
- Whether the route is originated by `network` or by redistribution

```cisco
router bgp 65000
 bgp log-neighbor-changes
 network 203.0.113.0 mask 255.255.255.0
 redistribute ospf 1 route-map OSPF-TO-BGP
 redistribute eigrp 100 route-map EIGRP-TO-BGP
```

A BGP `network` statement originates a prefix only when an exact matching route exists in the routing table. That route can be connected, static, or learned by another protocol, depending on platform behavior and the exact route present. The key exam point is that the prefix must be present in the local routing table with an exact match.

**BGP attributes and redistribution**

When routes are redistributed into BGP:
- The original IGP metric is not preserved as a native BGP path attribute in a meaningful way for normal selection.
- BGP path selection uses BGP attributes such as local preference, AS_PATH, origin, MED, and weight.
- A redistributed route may appear as locally originated in the BGP table, but that does not by itself prove whether it came from `network` or `redistribute`.

Example of a locally originated BGP route:
```cisco
show ip bgp 203.0.113.0
```

Possible output characteristics:
- Next hop may show `0.0.0.0`
- Weight may show Cisco’s local-origin default behavior
- The route may be originated by `network` or redistribution

Do not use the presence of `0.0.0.0` or a specific weight alone as proof of redistribution.

**Route-maps, filtering, and default deny behavior**

Route-maps are central to redistribution policy. They are evaluated top-down by sequence number:
- The first matching sequence is used
- Multiple `match` statements in the same sequence are ANDed unless the platform supports a different match behavior for the specific object type
- If no sequence matches, the route is denied by default

```cisco
route-map OSPF-TO-EIGRP permit 10
 match ip address prefix-list OSPF-REDIST
 set metric 100000 100 255 1 1500
 set tag 100

route-map OSPF-TO-EIGRP deny 20
 match tag 100
```

Important route-map behavior:
- `set` actions apply only when the sequence is permitted
- A deny sequence does not set values; it blocks the route
- If you need multiple policies, use multiple sequences with clear match logic
- A route-map used for redistribution should be tested carefully because an unmatched route is silently dropped

**Filtering tools**

Redistribution policy is stronger when you combine route-maps with protocol-level filters:
- `prefix-list` for prefix matching
- `distribute-list` for protocol-specific filtering
- `route-map` for attribute setting and conditional permit/deny
- `AS-path` access lists for BGP route selection
- `community` matching for BGP policy control

Examples:
```cisco
ip prefix-list OSPF-REDIST seq 10 permit 10.10.0.0/16 le 24
ip prefix-list OSPF-REDIST seq 20 permit 172.16.50.0/24
```

```cisco
router ospf 1
 distribute-list prefix OSPF-IN in
```

```cisco
ip as-path access-list 10 permit _65010_
route-map BGP-TO-OSPF permit 10
 match as-path 10
```

**Loop prevention**

Mutual redistribution can create routing loops when a route leaves one domain, enters another, and then re-enters the original domain as if it were new. The standard prevention methods are:
- Route tags
- Prefix filtering
- AS-path filtering for BGP
- Community-based policy for BGP
- Careful one-way redistribution design
- Summarization at redistribution boundaries

A common pattern is:
1. Tag routes when they are redistributed from OSPF into EIGRP.
2. Deny tagged routes when redistributing back from EIGRP into OSPF.
3. Permit only the intended prefixes in each direction.

Example:
```cisco
route-map OSPF-TO-EIGRP permit 10
 match ip address prefix-list OSPF-REDIST
 set tag 110

route-map EIGRP-TO-OSPF deny 10
 match tag 110

route-map EIGRP-TO-OSPF permit 20
 match ip address prefix-list EIGRP-REDIST
```

**One-way redistribution patterns**

One-way redistribution is safer than mutual redistribution. Common designs include:
- OSPF into EIGRP at a branch edge
- EIGRP into OSPF at a migration boundary
- IGP into BGP at the enterprise edge
- BGP into IGP only for selected external prefixes or a default route

Example: OSPF into EIGRP
```cisco
router eigrp 100
 default-metric 100000 100 255 1 1500
 redistribute ospf 1 route-map OSPF-TO-EIGRP
```

```cisco
route-map OSPF-TO-EIGRP permit 10
 match ip address prefix-list OSPF-REDIST
 set tag 110
```

Example: EIGRP into OSPF
```cisco
router ospf 1
 redistribute eigrp 100 subnets metric 20 metric-type 1 route-map EIGRP-TO-OSPF
```

Example: BGP into OSPF
```cisco
router ospf 1
 redistribute bgp 65000 subnets metric 50 metric-type 2 route-map BGP-TO-OSPF
```

**Default route redistribution**

Default routes are commonly tested because they are easy to misconfigure.

Options include:
- Redistribute a static default route
- Originate a default in OSPF with `default-information originate`
- Inject a default into EIGRP with redistribution or a static route
- Advertise a default from BGP using `network 0.0.0.0` only if the route exists, or by policy from an upstream source

Examples:
```cisco
ip route 0.0.0.0 0.0.0.0 192.0.2.1
router ospf 1
 default-information originate
```

```cisco
router eigrp 100
 redistribute static metric 100000 100 255 1 1500
```

```cisco
router bgp 65000
 network 0.0.0.0
```

Be careful:
- `default-information originate` advertises a default only if a default route exists, unless `always` is used
- Redistributing a static default can be broader than intended if your static route points to a generic next hop
- In stub or totally stub areas, default route behavior is often constrained by area type

**BGP to IGP redistribution specifics**

When you redistribute BGP into OSPF or EIGRP, you are usually importing selected external reachability from an edge or transit domain. The policy implications are significant:
- BGP may contain many more prefixes than you want inside the IGP
- BGP path attributes are not preserved in the IGP
- You often need prefix-lists to limit the imported set
- You may want to redistribute only customer, internal, or aggregate routes

Practical implications:
- OSPF receives the prefix as an external route with an assigned metric type and cost
- EIGRP receives the prefix as an external route with a seed metric
- The IGP does not understand AS_PATH, local preference, or communities
- If you later redistribute those IGP routes back into BGP, you can lose the original BGP policy context unless you preserve it with tags or careful filtering

**Troubleshooting workflow**

When redistribution fails, check the problem in layers:
1. Verify the source route exists in the routing table.
2. Verify the redistribution command is present under the correct process.
3. Verify the route-map or prefix-list matches the route.
4. Verify the destination protocol has a valid metric or default metric.
5. Verify the route is not blocked by area type, stub restrictions, or filtering.
6. Verify the route is not being overridden by a better route from another source.

Useful commands:
```cisco
show ip route
show ip protocols
show ip ospf database external
show ip ospf database nssa-external
show ip eigrp topology
show ip bgp
show ip bgp summary
show route-map
show ip prefix-list
```

For BGP-specific troubleshooting:
```cisco
show ip bgp
show ip bgp neighbors
show ip bgp neighbors advertised-routes
show ip bgp neighbors received-routes
```

For OSPF-specific troubleshooting:
```cisco
show ip ospf border-routers
show ip ospf database external
show ip ospf database nssa-external
show ip route ospf
```

For EIGRP-specific troubleshooting:
```cisco
show ip eigrp topology all-links
show ip route eigrp
show ip protocols
```

**Common failure causes**

- The route-map does not match the prefix.
- The prefix-list is too narrow or has the wrong mask length.
- EIGRP redistribution has no seed metric.
- OSPF redistribution omits `subnets`.
- The route is in a stub, totally stub, or NSSA area that limits external injection.
- The route is filtered by a distribute-list or route-map.
- The route exists in BGP but not in the local routing table for `network` origination.
- Administrative distance causes a different route source to win.
- Mutual redistribution reintroduces a tagged route and creates a loop, so the route is intentionally filtered.

**Exam-focused distinctions**

- **EIGRP** needs a metric for redistributed routes.
- **OSPF** needs a metric and a type; `E1` and `E2` matter.
- **BGP** redistribution is policy-driven; BGP attributes matter more than IGP-style metrics.
- **`network` vs `redistribute` in BGP** are not the same thing.
- **Route tags** help prevent loops, but they are not the only tool.
- **Type 5 and Type 7** are LSA types, not routing table codes.
- **`O E1`/`O E2`/`O N1`/`O N2`** are the routing table codes you should recognize.

**Practical design guidance**

Use redistribution only when you need it. Prefer:
- Summarization at boundaries
- One-way redistribution
- Explicit filtering
- Route tagging
- Minimal prefix exposure

If you must do mutual redistribution:
- Define a clear primary protocol
- Tag every redistributed route
- Filter re-entry at the opposite boundary
- Test default route behavior separately
- Verify external route preference in the destination protocol

**Quick configuration patterns**

EIGRP receiving OSPF:
```cisco
router eigrp 100
 default-metric 100000 100 255 1 1500
 redistribute ospf 1 route-map OSPF-TO-EIGRP
```

OSPF receiving EIGRP:
```cisco
router ospf 1
 redistribute eigrp 100 subnets metric 20 metric-type 1 route-map EIGRP-TO-OSPF
```

BGP receiving OSPF:
```cisco
router bgp 65000
 redistribute ospf 1 route-map OSPF-TO-BGP
```

OSPF receiving BGP:
```cisco
router ospf 1
 redistribute bgp 65000 subnets metric 50 metric-type 2 route-map BGP-TO-OSPF
```

**Troubleshooting example**

If a route is missing from OSPF after redistribution:
- Confirm the route exists in `show ip route`.
- Confirm the route-map permits it.
- Confirm `subnets` is present.
- Confirm the area is not stub or totally stub in a way that blocks the external route.
- Confirm the route is not being filtered by another policy.
- Confirm the route is not already present with a better administrative distance from another source.

If a route appears in BGP but not in the routing table:
- Check next-hop reachability.
- Check BGP best-path selection.
- Check whether the route is only in the BGP table and not installed due to administrative distance or policy.
- Check whether the route was originated by `network` or redistribution and whether the source route still exists.

**Summary**

Redistribution connects routing domains, but it also introduces policy, metric translation, and loop risk. EIGRP needs seed metrics, OSPF needs external type and cost, and BGP needs careful attribute and prefix control. The safest designs use one-way redistribution, route tags, and explicit filtering. For exam questions, always separate routing-table behavior from database behavior, and always verify whether the issue is origination, filtering, metric assignment, or route preference.
