## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**EIGRP feasible successor fundamentals**

Enhanced Interior Gateway Routing Protocol (<abbr title="Enhanced Interior Gateway Routing Protocol">EIGRP</abbr>) uses the topology table to keep loop-free backup paths ready before a failure occurs. In operational troubleshooting, the key question is not just “is there another path?” but “is there another path that satisfies the feasibility condition and can be promoted immediately?”

EIGRP maintains three important concepts for each destination:

- **Successor**: the current best path installed in the routing table.
- **Feasible successor**: a backup path that is guaranteed loop-free and can replace the successor immediately.
- **Reported distance** (<abbr title="Reported Distance">RD</abbr>): the metric a neighbor advertises to reach the destination.
- **Feasible distance** (<abbr title="Feasible Distance">FD</abbr>): the local router’s best known total metric to the destination.

The feasibility condition is the core rule:

- A neighbor path can become a feasible successor only if its RD is less than the current successor’s FD.
- If the condition is not met, the path may still exist in the topology table, but it is not eligible for instant promotion.

This distinction matters in troubleshooting because a route can be present in EIGRP, yet fail over slowly if no feasible successor exists.

**How to read the topology table**

The topology table is the best place to verify EIGRP path eligibility. It shows the successor, any feasible successors, and the metric relationship that determines whether a backup path is usable.

Typical output pattern:

```bash
show ip eigrp topology 10.10.10.0/24
```

Example interpretation:

```text
P 10.10.10.0/24, 1 successors, FD is 30720
        via 192.0.2.2 (30720/28160), GigabitEthernet0/0
        via 192.0.2.6 (40960/38400), GigabitEthernet0/1
```

In many Cisco outputs, the pair in parentheses is commonly read as:

- First value: local path metric for that neighbor path, often shown as the path’s FD in the topology display.
- Second value: the neighbor’s RD for that path.

For exam and troubleshooting purposes, focus on the relationship:

- The successor is the path with the lowest FD.
- A feasible successor must have RD < successor FD.
- If the backup path’s RD is equal to or greater than the successor FD, it is not feasible.

> Be careful with metric interpretation across examples and platforms. Always verify the successor, the displayed FD, and the neighbor RD relationship rather than relying on a memorized label order.

**Operational verification workflow**

When a route fails over unexpectedly or does not fail over at all, verify the path in a consistent order.

1. Check the routing table.
   ```bash
   show ip route eigrp
   ```
   - Confirm whether the destination is installed as an internal EIGRP route (`D`) or external EIGRP route (`D EX`).
   - Note the administrative distance is typically 90 for internal EIGRP and 170 for external EIGRP.

2. Check the topology table.
   ```bash
   show ip eigrp topology
   ```
   - Identify the successor.
   - Determine whether a feasible successor exists.
   - Compare the candidate backup path’s RD against the successor’s FD.

3. Check neighbor relationships.
   ```bash
   show ip eigrp neighbors
   ```
   - Confirm the neighbor advertising the backup path is up.
   - Verify the adjacency is stable and not flapping.

4. Check interface and metric inputs.
   ```bash
   show ip interface brief
   show interfaces gigabitEthernet0/0
   show ip protocols
   ```
   - EIGRP metrics depend on bandwidth, delay, and optionally reliability, load, and MTU in the composite metric calculation.
   - A changed interface metric can alter FD/RD relationships and remove feasibility.

5. Check for policy or design constraints.
   - Summarization can hide more-specific routes.
   - Stub routing can prevent a router from learning alternate paths.
   - Filtering can remove the only path that would have qualified as a feasible successor.

**Feasible successor selection in troubleshooting terms**

A practical way to think about selection is:

- EIGRP receives multiple paths for the same prefix.
- It computes the local metric for each path.
- It chooses the lowest-metric path as the successor.
- It compares every other path’s RD to the successor’s FD.
- Any path with RD lower than the successor FD becomes a feasible successor.

This is why a route can have multiple known paths but only one backup path eligible for immediate use.

Example logic:

- Successor FD = 30720
- Candidate path A RD = 28160 → feasible successor
- Candidate path B RD = 30720 → not feasible
- Candidate path C RD = 38400 → not feasible

Only path A can be promoted instantly if the successor fails.

**Why a path fails the feasibility condition**

A path fails the feasibility condition when the neighbor’s advertised distance is not strictly less than the current successor’s feasible distance. In practice, this often happens because:

- The alternate path is longer or less preferred.
- The topology is asymmetric.
- A summarization boundary changes the advertised metric.
- A stub router does not advertise transit paths.
- A route filter removes the better backup path.
- An external route is involved and the metric relationship is not favorable.

A failed feasibility check does not mean the route is unreachable. It means EIGRP cannot prove the path is loop-free using the feasibility condition.

**Internal versus external routes**

EIGRP internal routes and external routes both appear in the topology table, but external routes deserve extra caution during troubleshooting.

- **Internal routes** are learned from within the EIGRP domain and typically have lower administrative distance.
- **External routes** are redistributed into EIGRP and appear as `D EX` in the routing table.

Operationally:

- The feasibility condition still uses RD and FD logic.
- External routes can be trickier because the metric may reflect redistribution behavior rather than a simple internal path.
- When multiple external paths exist, verify the exact source of redistribution and the metric values before assuming a backup path is feasible.

Example verification:

```bash
show ip eigrp topology external
show ip route eigrp
```

Look for:

- The external route source.
- The metric advertised by the redistributing router.
- Whether the candidate backup path’s RD is lower than the current successor’s FD.

**Troubleshooting a route that is not a feasible successor**

Suppose a router learns `10.10.10.0/24` from two neighbors:

```text
P 10.10.10.0/24, 1 successors, FD is 30720
        via 192.0.2.2 (30720/28160), GigabitEthernet0/0
        via 192.0.2.6 (40960/30720), GigabitEthernet0/1
```

In this example:

- The first path is the successor.
- The second path is not a feasible successor because its RD equals the successor FD, not less than it.

Operational impact:

- If the successor fails, EIGRP cannot immediately promote the second path.
- The router must query neighbors or recompute the topology.
- Convergence is slower and may trigger a query chain if the network is not designed carefully.

How to fix it operationally:

- Improve the alternate path metric so its RD becomes lower than the successor FD.
- Adjust interface bandwidth or delay where appropriate and consistent with design.
- Rework summarization so the backup path advertises a better metric.
- Remove an unnecessary filter that blocks the better backup route.
- Consider redesigning the topology if the alternate path is inherently too costly.

**Variance and maximum-paths**

EIGRP unequal-cost load balancing uses `variance`, but variance does not override the feasibility condition.

Key rules:

- Only **feasible successors** are eligible for installation with variance.
- A path that is not a feasible successor cannot be installed just because its metric is within the variance multiplier.
- `maximum-paths` controls how many eligible paths can be installed, not whether a path is eligible.

Example:

```bash
router eigrp 100
 variance 2
 maximum-paths 4
```

Operational meaning:

- The router may install multiple feasible successors if their metrics are within the variance threshold.
- The path must still satisfy the feasibility condition first.
- If no feasible successor exists, variance does not create one.

**Named mode configuration example**

Named EIGRP is common in modern IOS and IOS-XE. In named mode, configuration is organized under the EIGRP instance and address-family.

Example:

```bash
router eigrp CORP
 !
 address-family ipv4 unicast autonomous-system 100
  network 10.10.10.0 0.0.0.255
  network 10.10.20.0 0.0.0.255
  af-interface default
   passive-interface
  exit-af-interface
  af-interface GigabitEthernet0/0
   no passive-interface
  exit-af-interface
 exit-address-family
```

Notes:

- Use the address-family context for IPv4 unicast.
- `network` statements in named mode are configured under the address-family.
- `passive-interface` behavior is often managed per address-family or interface context.
- Avoid assuming classic-process commands map 1:1 into named mode.

Verification:

```bash
show ip protocols
show ip eigrp topology
show ip route eigrp
```

**Classic mode comparison**

Classic EIGRP is still useful to recognize in labs and older deployments.

```bash
router eigrp 100
 network 10.10.10.0 0.0.0.255
 network 10.10.20.0 0.0.0.255
 no auto-summary
```

Use classic mode knowledge to interpret older configurations, but remember that named mode is the more modern operational model.

**Stub routing and feasible successors**

EIGRP stub routing reduces query scope and improves stability, but it can also limit backup path visibility.

Common stub behavior:

- A stub router advertises only selected route types.
- It does not act as a transit router for arbitrary EIGRP queries.
- It may advertise connected, summary, static, redistributed, or receive-only routes depending on configuration.

Example:

```bash
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  eigrp stub connected summary
 exit-address-family
```

Operational effect:

- A stub can prevent alternate paths from being learned through it.
- If the only potential backup path traverses a stub boundary, the route may never become a feasible successor.
- In troubleshooting, check whether the missing backup path is a design choice rather than a failure.

**Summarization and filtering effects**

Summarization can change the metric information that neighbors see. That can help or hurt feasibility.

- A summary route may hide more-specific alternate paths.
- The summary metric may be less favorable than the specific route metric.
- If the summary is the only advertisement a neighbor receives, the neighbor may not have enough information to form a feasible successor.

Filtering can be even more direct:

- Route maps, distribute lists, prefix lists, and ACL-based filters can suppress the alternate path.
- If the better backup route is filtered, the remaining path may fail the feasibility condition.

Useful checks:

```bash
show ip protocols
show running-config | section router eigrp
show route-map
show ip prefix-list
```

**Practical lab workflow**

A simple lab helps validate the full failover process.

1. Build a triangle topology with three routers.
   - R1 is the destination router.
   - R2 is the primary path.
   - R3 is the alternate path.

2. Advertise the same prefix through both neighbors.
   - Make the R2 path the successor.
   - Tune the R3 path so it is either feasible or non-feasible.

3. Verify the topology table on the receiving router.
   ```bash
   show ip eigrp topology 10.10.10.0/24
   ```

4. Shut down the primary link.
   ```bash
   interface gigabitEthernet0/0
    shutdown
   ```

5. Recheck the routing table and topology table.
   ```bash
   show ip route eigrp
   show ip eigrp topology 10.10.10.0/24
   ```

6. Confirm whether the backup path was promoted immediately.
   - If yes, it was a feasible successor.
   - If no, EIGRP had to recompute because no feasible successor existed.

**Example troubleshooting output interpretation**

If the route disappears briefly and then returns, suspect no feasible successor was available.

If the route remains installed with a new next hop immediately after failure, the backup path was eligible and promoted.

If the route is present in the topology table but not in the routing table, check:

- Administrative distance conflicts.
- Route filtering.
- Passive interfaces.
- Unequal metric relationships.
- Whether the path is external and being overridden by another route source.

**Common exam traps**

- Confusing a known alternate path with a feasible successor.
- Assuming variance can install any backup path.
- Forgetting that the feasibility condition is required before load balancing.
- Treating external routes exactly like internal routes without checking redistribution context.
- Assuming a summary route always improves backup behavior.
- Reading topology output without comparing RD to the successor FD.

**Quick reference**

- **Successor**: best path installed in the routing table.
- **Feasible successor**: backup path that satisfies the feasibility condition.
- **Feasibility condition**: candidate RD must be less than successor FD.
- **Variance**: allows unequal-cost load balancing only among feasible successors.
- **Maximum-paths**: limits how many eligible paths can be installed.
- **Topology table**: primary source for EIGRP path eligibility troubleshooting.

**Command set for fast verification**

```bash
show ip route eigrp
show ip eigrp topology
show ip eigrp neighbors
show ip protocols
show ip interface brief
show running-config | section router eigrp
```

Use these commands together to answer the operational question: is the backup path missing, present but ineligible, or eligible but blocked by policy or design?
