## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Overview**

Metric manipulation and route filtering are core policy tools in enterprise routing design. You use them to influence path selection, control which prefixes are accepted or advertised, and shape failover behavior without changing physical topology.

In exam scenarios, the key is to know two things:

- **Metric manipulation** changes how a routing protocol ranks paths.
- **Route filtering** changes which routes are installed, learned, or advertised.

These are related but not the same. A route can be filtered even if it has the best metric, and a route can be preferred by metric even if it is not the most direct path.

**Where these topics appear**

- **OSPF**: cost tuning, redistribution metrics, and limited filtering behavior.
- **EIGRP**: composite metric tuning, summaries, and distribute-lists.
- **IS-IS**: wide metrics, interface metric control, and policy via redistribution or route-maps.
- **BGP**: local preference, MED, weight, AS-path policy, communities, and prefix-based filtering.

**Metric manipulation fundamentals**

A routing metric is the value a protocol uses to compare candidate paths. Different protocols calculate metrics differently:

- **OSPF** uses interface cost.
- **EIGRP** uses a composite metric based on bandwidth, delay, reliability, load, and MTU, though only bandwidth and delay are used by default.
- **IS-IS** uses interface metric values assigned to links.
- **BGP** does not use a single IGP-style metric; it uses a best-path algorithm with attributes such as weight, local preference, AS-path, origin, MED, and next hop.

A useful exam mindset is to ask:

- Is the protocol choosing between internal paths?
- Is the route being redistributed from another protocol?
- Is the policy local to one router or propagated across the domain?

That determines whether you should tune a metric, filter a route, or redesign the policy.

**OSPF metric manipulation**

OSPF is a link-state protocol. Each router builds the same link-state database for an area and runs SPF (Shortest Path First) locally. You usually influence path choice by changing interface cost, not by filtering routes.

Common cost tuning methods:

- Set interface cost directly with `ip ospf cost`.
- Change the reference bandwidth with `auto-cost reference-bandwidth`.
- Adjust per-interface bandwidth if you want the default cost calculation to reflect a different link speed.

Example:

```bash
router ospf 10
 auto-cost reference-bandwidth 100000
!
interface GigabitEthernet0/0
 ip ospf cost 10
```

`auto-cost reference-bandwidth 100000` sets the reference bandwidth to `100000 Mbps`, which is `100 Gbps`. This matters on modern networks because default OSPF cost values can collapse multiple high-speed links into the same cost if you leave the default reference bandwidth too low.

**OSPF redistribution metrics**

When you redistribute another protocol into OSPF, you must assign an external metric unless the platform can derive one from the source protocol and configuration context. OSPF external routes use:

- **Metric value**: the external cost you assign.
- **Metric type**: `type-1` or `type-2`.

The distinction matters:

- **Type 1** adds the internal OSPF cost to the external metric, so the total path cost reflects both the external and internal portions.
- **Type 2** treats the external metric as dominant and adds internal cost only as a tie-breaker.

Example:

```bash
router ospf 10
 redistribute static subnets metric 50 metric-type 1
 redistribute eigrp 100 subnets metric 20 metric-type 2
```

On modern Cisco IOS and IOS XE, `subnets` is important when redistributing non-classful prefixes into OSPF. Without it, you may not redistribute the prefixes you expect.

**OSPF filtering limitations**

OSPF does not support generic inbound route filtering in the same way distance-vector protocols do. It does not filter LSAs in the LSDB with a simple distribute-list. Instead:

- You influence what enters OSPF through redistribution policy.
- You control what is advertised by area design, summarization, stub/NSSA behavior, and redistribution controls.
- You can sometimes filter what is installed into the local routing table, but that does not stop LSAs from being learned.

This is a common exam trap:

> OSPF route filtering is not a general mechanism for stopping LSA propagation. It is mainly a local routing-table control or a redistribution policy problem.

Useful verification commands:

```bash
show ip ospf interface brief
show ip ospf interface g0/0
show ip ospf database
show ip route ospf
show ip protocols
```

**EIGRP metric manipulation**

EIGRP uses a composite metric. In most enterprise designs, you manipulate it by changing interface bandwidth and delay, or by setting an explicit metric during redistribution.

Default metric components:

- **Bandwidth**
- **Delay**
- Reliability, load, and MTU are present in the formula but usually not used in common designs.

Example of interface-based influence:

```bash
interface GigabitEthernet0/1
 bandwidth 100000
 delay 10
```

EIGRP prefers the path with the lowest feasible distance. If you want to influence path selection, changing delay is often more predictable than changing bandwidth, because bandwidth may also affect QoS and other features.

**EIGRP redistribution metrics**

When redistributing into EIGRP, you must provide a metric unless a default metric is already configured.

Classic EIGRP example:

```bash
router eigrp 100
 default-metric 10000 100 255 1 1500
 redistribute ospf 10
 redistribute static
```

Or per-redistribution metric:

```bash
router eigrp 100
 redistribute ospf 10 metric 10000 100 255 1 1500
```

The metric values are:

- Bandwidth in Kbps
- Delay in tens of microseconds
- Reliability from 1 to 255
- Load from 1 to 255
- MTU in bytes

In ENARSI, be careful with syntax context:

- **Classic EIGRP** uses `router eigrp <AS>`.
- **Named EIGRP** uses `router eigrp <name>` with address-family configuration.

Named EIGRP example:

```bash
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  topology base
   redistribute ospf 10 metric 10000 100 255 1 1500
```

**EIGRP route filtering**

EIGRP supports route filtering more directly than OSPF. The most common tested mechanism is `distribute-list`, usually applied inbound on an interface or at the process level.

Typical inbound interface filter:

```bash
ip prefix-list EIGRP-IN seq 5 permit 10.10.0.0/16 le 24
!
router eigrp 100
 distribute-list prefix EIGRP-IN in GigabitEthernet0/0
```

Important behavior:

- EIGRP distribute-lists filter routes being installed into the topology table and routing table.
- In practice, interface-based inbound filtering is the most common exam scenario.
- Process-level filtering can affect routes from multiple neighbors, so always check the scope carefully.

Verification:

```bash
show ip eigrp neighbors
show ip eigrp topology
show ip eigrp interfaces
show ip route eigrp
```

**IS-IS metric manipulation**

IS-IS is also a link-state protocol. Like OSPF, you usually influence path selection by changing interface metrics rather than filtering routes.

Common metric controls:

- `isis metric <value>` on an interface
- wide metrics in modern deployments
- level-specific configuration when needed

Example:

```bash
interface GigabitEthernet0/2
 ip router isis
 isis metric 20 level-2
```

You can also tune the metric on a per-level basis in some IOS/IOS XE contexts. Lower metric values are preferred.

**IS-IS redistribution and policy**

When redistributing into IS-IS, you often use route-maps to control which routes are injected and what metric they receive.

Example:

```bash
route-map RM-TO-ISIS permit 10
 match ip address prefix-list PL-REDIST
 set metric 50
!
router isis CORE
 redistribute static route-map RM-TO-ISIS
```

IS-IS filtering is usually done through redistribution policy rather than generic route filtering of the link-state database.

**BGP metric manipulation and path selection**

BGP is policy-driven. It does not use a single metric like OSPF or EIGRP. Instead, it selects the best path using attributes in a defined order.

Common attributes:

- **Weight**: Cisco-local, highest value wins.
- **Local preference**: Influences outbound path choice within an AS, higher is better.
- **AS-path**: Shorter path is generally preferred.
- **Origin**: IGP is preferred over EGP over incomplete.
- **MED**: Multi-Exit Discriminator, lower is better, but comparison is limited by default.
- **Next hop**: Must be reachable.
- **Community**: Used to tag routes for policy decisions.

**Weight vs local preference**

- **Weight** is local to one router and affects only that router’s best-path selection.
- **Local preference** is propagated within the AS and is the standard tool for controlling outbound traffic across the enterprise.

Example:

```bash
route-map SET-LP permit 10
 set local-preference 200
!
router bgp 65000
 neighbor 192.0.2.1 route-map SET-LP in
```

**MED behavior**

MED is used to suggest which entry point another AS should prefer. It is commonly compared only when routes come from the same neighboring AS, unless you enable broader comparison with `bgp always-compare-med`.

Example:

```bash
route-map SET-MED permit 10
 set metric 50
!
router bgp 65000
 neighbor 203.0.113.1 route-map SET-MED out
```

**BGP filtering tools**

BGP offers several filtering mechanisms, and the right one depends on what you need to match.

- **Prefix-list**: Best for exact prefix and mask-length control.
- **AS-path access-list**: Best for filtering based on path origin or transit AS numbers.
- **Route-map**: Best for combining multiple match conditions and setting attributes.
- **Community list**: Best for policy tags across the AS or between peers.

Example prefix-list filter:

```bash
ip prefix-list BGP-IN seq 10 permit 10.0.0.0/8 le 24
!
router bgp 65000
 neighbor 192.0.2.2 prefix-list BGP-IN in
```

Example AS-path filter:

```bash
ip as-path access-list 10 permit _65010_
!
route-map BGP-FILTER deny 10
 match as-path 10
route-map BGP-FILTER permit 20
!
router bgp 65000
 neighbor 198.51.100.2 route-map BGP-FILTER in
```

Example community-based policy:

```bash
ip community-list standard NO-EXPORT permit 65000:100
!
route-map TAG-COMM permit 10
 match community NO-EXPORT
 set local-preference 50
```

**Route-map logic and implicit deny**

Route-maps are central to route filtering and metric manipulation. They are evaluated top-down, first match wins, and there is an implicit deny at the end.

Key rules:

- Sequence numbers are processed in ascending order.
- A `permit` sequence allows the route and can modify attributes.
- A `deny` sequence blocks the route.
- If no sequence matches, the route is denied by default.

Example:

```bash
route-map POLICY deny 10
 match ip address prefix-list BLOCKED
!
route-map POLICY permit 20
 set metric 100
```

If you forget a final permit sequence, you may accidentally filter everything not explicitly matched. This is one of the most common troubleshooting issues in ENARSI labs.

**Route filtering mechanisms compared**

| Mechanism | Best use | Typical protocols | Notes |
|---|---|---|---|
| Prefix-list | Exact prefix and mask control | BGP, redistribution | Fast and precise |
| ACL | Legacy matching or broad IP matching | Redistribution, some policy cases | Less precise than prefix-lists |
| Route-map | Multi-condition policy and attribute setting | BGP, redistribution, IS-IS | Supports match/set logic |
| AS-path ACL | Match AS numbers in BGP path | BGP | Useful for transit control |
| Community list | Tag-based policy | BGP | Scales well in large designs |
| Distribute-list | Filter installed routes | EIGRP, RIP, some limited cases | Scope matters |
| Summary | Hide specifics and reduce table size | OSPF, EIGRP, IS-IS, BGP | Can require route leaking |

**Route-map match conditions beyond prefixes**

For ENARSI-level policy design, you should know that route-maps can match more than prefixes.

Common BGP matches:

- `match ip address prefix-list`
- `match as-path`
- `match community`
- `match extcommunity`
- `match metric`
- `match tag`
- `match interface` in some redistribution contexts

This matters when you need to filter or classify routes based on origin or policy tags rather than just destination network.

Example:

```bash
route-map BGP-POLICY permit 10
 match community GOLD
 set local-preference 300
!
route-map BGP-POLICY permit 20
 match as-path 20
 set local-preference 100
```

**Summarization and route leaking**

Summarization reduces routing table size, but it can hide more-specific routes that you still need. In those cases, you may need route leaking or selective redistribution.

Common design patterns:

- Summarize at the edge to reduce churn.
- Leak specific prefixes back into a summarized domain when a downstream site still needs them.
- Use route-maps or prefix-lists to redistribute only selected specifics.

EIGRP summary example:

```bash
interface GigabitEthernet0/0
 ip summary-address eigrp 100 10.10.0.0 255.255.0.0
```

When EIGRP installs a summary, it also installs a `Null0` route for the summary prefix on the summarizing router. This prevents loops for destinations that fall within the summary but are not known more specifically.

Selective redistribution example:

```bash
route-map REDIST-ONLY-SPECIFICS permit 10
 match ip address prefix-list SPECIFICS
!
router ospf 10
 redistribute eigrp 100 subnets route-map REDIST-ONLY-SPECIFICS
```

**Practical troubleshooting workflow**

When routes are missing or the wrong path is chosen, work from policy outward:

1. Verify the route exists in the source protocol.
2. Check whether the route is being filtered before redistribution or advertisement.
3. Confirm the metric or attribute is being set as intended.
4. Verify the receiving protocol’s best-path logic.
5. Check for summarization, route leaking, or implicit deny behavior.

Useful commands:

```bash
show ip route
show ip protocols
show route-map
show ip prefix-list
show ip as-path access-list
show ip bgp
show ip bgp summary
show bgp ipv6 unicast
show ip ospf interface
show ip eigrp interfaces
```

For IPv6 BGP, remember that policy is commonly applied under the IPv6 address-family and neighbors often need activation there.

Example:

```bash
router bgp 65000
 address-family ipv6 unicast
  neighbor 2001:db8::2 activate
  neighbor 2001:db8::2 route-map V6-FILTER in
```

**Common exam traps**

- Confusing **metric** with **metric type** in OSPF redistribution.
- Assuming OSPF supports generic inbound route filtering like EIGRP.
- Forgetting the implicit deny at the end of a route-map.
- Using the wrong EIGRP syntax for classic versus named mode.
- Treating MED as universally compared in BGP without considering neighboring AS context.
- Assuming a distribute-list changes what a protocol advertises rather than what the router installs.
- Forgetting that BGP IPv6 policy is usually under the IPv6 address-family with neighbor activation.

**Quick protocol comparison**

| Protocol | Primary path control | Common filtering method | Notes |
|---|---|---|---|
| OSPF | Interface cost, reference bandwidth | Redistribution policy, limited local filtering | LSDB is not filtered by distribute-list |
| EIGRP | Composite metric, delay, bandwidth | Distribute-list, route-map redistribution | Inbound filtering is common |
| IS-IS | Interface metric | Route-map redistribution | Link-state policy is usually indirect |
| BGP | Attributes and best-path logic | Prefix-list, route-map, AS-path ACL, communities | Most flexible policy engine |

**Study takeaway**

If you need to influence which path is chosen, manipulate the metric or BGP attributes. If you need to control which routes are accepted or advertised, use filtering tools such as prefix-lists, route-maps, AS-path ACLs, communities, or protocol-specific controls. In ENCOR and ENARSI, the best answer is usually the one that matches the protocol’s design rather than forcing a generic filtering method onto every routing process.
