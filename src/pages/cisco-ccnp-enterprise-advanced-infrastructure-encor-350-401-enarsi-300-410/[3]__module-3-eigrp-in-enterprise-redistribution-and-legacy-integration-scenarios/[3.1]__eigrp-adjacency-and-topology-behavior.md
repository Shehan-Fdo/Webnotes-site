## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**EIGRP overview**

<abbr title="Enhanced Interior Gateway Routing Protocol">EIGRP</abbr> is an advanced distance-vector routing protocol that uses the <abbr title="Diffusing Update Algorithm">DUAL</abbr> algorithm to choose loop-free paths and converge quickly after a change. It forms neighbor adjacencies first, then exchanges topology information, and finally installs the best routes into the routing table.

EIGRP behavior is often tested in three layers:

- **Neighbor formation**: can two routers exchange hello packets and agree on key parameters?
- **Topology selection**: which paths are feasible, successors, or backups?
- **Convergence**: what happens when the best path fails?

**Neighbor adjacency fundamentals**

EIGRP neighbors exchange hello packets to discover and maintain adjacencies. On most broadcast and point-to-point Ethernet-style links, hellos are sent to multicast addresses and are received by all EIGRP-capable peers on the segment. On non-broadcast multi-access <abbr title="Non-Broadcast Multi-Access">NBMA</abbr> networks, multicast replication may not be inherent, so you may need static neighbor design or a topology that supports the desired reachability.

A neighbor relationship requires more than just IP reachability. Common requirements include:

- Matching EIGRP autonomous system settings for classic mode, or matching address-family autonomous-system values in named mode.
- Compatible K-values, because metric calculation must be identical.
- Matching authentication, if configured.
- Layer 3 connectivity and no filtering of EIGRP hello packets.
- Compatible interface behavior, including network type and timer expectations.

> Timer mismatches usually do not stop adjacency formation by themselves if hellos are still received within the hold time relationship. They more often affect stability and failure detection, not the initial ability to become neighbors.

**What EIGRP sends in hellos and updates**

EIGRP uses reliable transport for important packets and multicast for discovery and some updates. The protocol exchanges:

- **Hello** packets to discover and maintain neighbors.
- **Update** packets to advertise routes.
- **Query** packets when a route loses its feasible successor and DUAL must search for alternatives.
- **Reply** packets to answer queries.
- **ACK** packets to acknowledge reliable packets.

The reliable transport queue matters during troubleshooting. A persistently high `Q` count in `show ip eigrp neighbors` means packets are waiting for acknowledgment. That usually points to congestion, packet loss, or an overloaded peer, not a single generic “slow neighbor” condition.

**Classic and named mode adjacency requirements**

Classic EIGRP uses a process number, while named mode uses a routing process with one or more address families. The process name itself does not need to match across devices. What matters is that the neighbors are in the same EIGRP domain for the relevant address family and autonomous-system value.

Examples of common adjacency blockers:

- **ACLs** or firewall policies blocking protocol 88 or multicast EIGRP traffic.
- **Passive-interface** on the wrong interface.
- **Stub mismatches** that change query behavior and can prevent expected reachability patterns.
- **K-value mismatches**.
- **Authentication mismatches**.
- **Interface network type** that does not support the expected neighbor discovery method.
- **Split horizon** issues in hub-and-spoke designs, especially when routes must be reflected through a hub.

**Basic verification commands**

Use these commands to confirm neighbor state and packet exchange:

```bash
show ip eigrp neighbors
show ip eigrp interfaces detail
show ip protocols
show ip route eigrp
show ip eigrp topology
show ip eigrp traffic
```

For IPv6 and named mode, use the corresponding family-specific commands:

```bash
show ipv6 eigrp neighbors
show ipv6 eigrp topology
show ipv6 protocols
show ipv6 route eigrp
show eigrp address-family ipv4 unicast neighbors
show eigrp address-family ipv4 unicast topology
```

**Topology table behavior**

The EIGRP topology table stores all known routes for a prefix, not just the installed route. DUAL evaluates each path using the metric reported by the neighbor and the local feasible distance calculation.

Key terms:

- **Successor**: the best path installed in the routing table.
- **Feasible successor**: a backup path that is guaranteed loop-free and can replace the successor immediately.
- **Feasible distance**: the lowest known distance from the local router to the destination.
- **Reported distance**: the neighbor’s advertised distance to the destination.

The feasibility condition is critical: a backup path is a feasible successor only if its reported distance is less than the current feasible distance of the successor path. This check uses the neighbor’s reported distance, not the local total metric.

A simplified example:

- Router A learns prefix `10.10.10.0/24`.
- Path via Router B has the lowest metric and becomes the successor.
- Path via Router C is kept as a backup only if C’s reported distance is lower than A’s feasible distance for the B path.

If the condition is not met, the path may still exist in the topology table, but it is not immediately usable as a loop-free backup.

**Topology output interpretation**

A typical topology entry can show:

- The prefix and prefix length.
- The successor count.
- The feasible distance.
- The route state, such as passive or active.
- One or more next hops with their reported distance and interface.

Example:

```bash
Router# show ip eigrp topology 10.10.10.0/24
EIGRP-IPv4 Topology Entry for AS(100)/ID(1.1.1.1) for 10.10.10.0/24
  State is Passive, Query origin flag is 0, 1 Successor(s), FD is 28160
  Routing Descriptor Blocks:
  10.1.12.2 (GigabitEthernet0/0), from 10.1.12.2, Send flag is 0x0
      Composite metric is (28160/25600), Route is Internal
  10.1.13.3 (GigabitEthernet0/1), from 10.1.13.3, Send flag is 0x0
      Composite metric is (30720/28160), Route is Internal
```

How to read this:

- `State is Passive` means DUAL has converged for this prefix.
- `1 Successor(s)` means one best path is installed.
- `FD is 28160` is the current feasible distance.
- The first metric pair is often shown as `<feasible distance>/<reported distance>` for that path.
- The second path may be a feasible successor only if its reported distance is less than `28160`.

**DUAL states and convergence**

DUAL keeps routes in one of two broad states:

- **Passive**: the route is stable and fully converged.
- **Active**: the router is actively searching for a new path because no feasible successor is available.

When a successor fails:

1. The router checks for a feasible successor.
2. If one exists, it installs it immediately and stays passive.
3. If none exists, the route goes active.
4. The router sends queries to neighbors asking for an alternative path.
5. Neighbors reply with their own distances or indicate they also have no path.
6. Once all replies arrive, DUAL selects a new successor or marks the route unreachable.

A route that remains active too long can become **stuck in active** <abbr title="Stuck in Active">SIA</abbr>. This usually indicates that one or more neighbors did not reply in time, often because of congestion, packet loss, a neighbor failure, or an overly broad query domain.

**Query and reply behavior**

Queries are the mechanism that makes EIGRP scalable and loop-free, but they can also create troubleshooting complexity.

- If a router has a feasible successor, it does not need to query widely.
- If it has no feasible successor, it queries neighbors.
- Neighbors that also lack a path may need to query their own neighbors.
- This can create a query chain if the topology is large or poorly summarized.

To reduce query scope:

- Use route summarization at distribution points.
- Use EIGRP stub on spoke routers.
- Design clear hierarchy and avoid unnecessary redistribution loops.

**Stub routing and query containment**

Stub routing limits what a router advertises and how it participates in queries. It is especially useful in hub-and-spoke designs.

Common stub options include:

- `connected`
- `static`
- `summary`
- `redistributed`
- `receive-only`

Example:

```bash
router eigrp 100
 eigrp stub connected static summary
```

In named mode:

```bash
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  eigrp stub connected static
```

Stub routers reduce the chance of large query domains and help prevent SIA conditions in branch designs.

**Unequal-cost load balancing and variance**

EIGRP can install multiple paths when you intentionally allow unequal-cost load balancing with `variance`. The alternate path must still satisfy the feasibility condition, and its metric must be within the variance threshold relative to the successor.

Example:

```bash
router eigrp 100
 variance 2
 maximum-paths 4
```

Practical meaning:

- `maximum-paths` limits how many paths can be installed.
- `variance` allows paths with higher metrics to be used if they are still feasible and within the configured multiplier.
- Without `variance`, EIGRP typically uses only equal-cost paths for load balancing.

Use unequal-cost load balancing when you want to use a backup or secondary link without making it the primary path. This is common in WAN designs where one link is faster but another link is still acceptable for traffic.

**Neighbor formation and common failure causes**

A neighbor may fail to form even when IP connectivity exists. Check these causes first:

- **Protocol filtering**: ACLs or control-plane policies block EIGRP packets.
- **Authentication mismatch**: key-chain, key ID, or hash settings differ.
- **K-value mismatch**: metric weights differ, so neighbors refuse adjacency.
- **Passive interface**: the interface is not sending hellos.
- **Network type mismatch**: especially on NBMA or point-to-multipoint designs.
- **Stub mismatch**: not always an adjacency blocker, but it can change expected route exchange and query behavior.
- **Interface shutdown or wrong VRF**: the process is running, but not in the correct forwarding context.
- **Split horizon**: can prevent route advertisement back out the same interface in hub-and-spoke topologies.

Useful checks:

```bash
show ip eigrp neighbors detail
show ip eigrp interfaces
show ip eigrp topology all-links
show ip route
show access-lists
show run interface g0/0
```

**Network type effects**

Interface network type influences how EIGRP discovers neighbors and how it behaves on multi-access media.

- **Broadcast**: neighbors are discovered naturally with multicast hellos.
- **Point-to-point**: one neighbor relationship per link, simple and predictable.
- **Point-to-multipoint**: useful for hub-and-spoke over a shared medium.
- **NBMA**: may require static neighbor statements or careful design because multicast/broadcast replication is not inherent on the medium.

Example of static neighbors on a hub over NBMA:

```bash
router eigrp 100
 neighbor 172.16.1.2 GigabitEthernet0/0
 neighbor 172.16.1.3 GigabitEthernet0/0
```

Use static neighbors when the medium does not reliably deliver multicast discovery or when the design requires explicit control of adjacency formation.

**Authentication**

EIGRP authentication protects adjacency and route exchange from unauthorized peers. The exact syntax can vary by platform and software release, so verify the command set on the target device.

A common MD5 example on many IOS and IOS XE platforms:

```bash
key chain EIGRP-KEYS
 key 1
  key-string cisco123

interface GigabitEthernet0/0
 ip authentication mode eigrp 100 md5
 ip authentication key-chain eigrp 100 EIGRP-KEYS
```

Important points:

- Both neighbors must use the same authentication method and key material.
- Authentication failures usually prevent adjacency.
- Key rollover requires overlap in valid key lifetimes if you want seamless transition.

**IPv6 EIGRP behavior**

EIGRP for IPv6 uses the same DUAL concepts, but configuration and activation differ by platform and mode. In named mode on IOS and IOS XE, you commonly enable the address family and then activate interfaces or use `no shutdown` under the address-family.

Example:

```bash
ipv6 unicast-routing

router eigrp CORP
 address-family ipv6 unicast autonomous-system 100
  af-interface default
   passive-interface
  exit-af-interface
  af-interface GigabitEthernet0/0
   no passive-interface
  exit-af-interface
  no shutdown
```

Notes for exam prep:

- The need for `no shutdown` depends on the exact EIGRPv6 configuration model and platform.
- IPv6 neighbors still require link-local reachability and interface-level activation.
- Use `show ipv6 eigrp neighbors` and `show ipv6 eigrp topology` to verify operation.

**End-to-end lab scenario**

Consider a simple three-router topology:

- R1 connects to R2 and R3.
- R2 is the primary path to `10.10.10.0/24`.
- R3 is the backup path.
- EIGRP AS 100 runs on all routers.

Initial state:

- R1 learns the prefix from both neighbors.
- R2 becomes the successor because it has the lowest metric.
- R3 is a feasible successor if its reported distance is lower than R1’s feasible distance through R2.

Verification:

```bash
R1# show ip eigrp topology 10.10.10.0/24
R1# show ip route eigrp
R1# show ip eigrp neighbors
```

Failure event:

- The link to R2 fails.
- If R3 is a feasible successor, R1 immediately installs R3 and remains passive.
- If R3 is not feasible, R1 goes active and queries neighbors.

Recovery behavior:

- With a feasible successor, convergence is fast and local.
- Without one, convergence depends on query and reply completion across the topology.

This is the core exam pattern: identify whether a backup path is feasible before the failure, because that determines whether DUAL can switch instantly or must search.

**Troubleshooting workflow**

Use a structured approach:

1. Confirm the interface is up and in the correct VRF or routing context.
2. Verify hellos are being sent and received with `show ip eigrp neighbors`.
3. Check for authentication, K-value, and passive-interface mismatches.
4. Inspect `show ip eigrp interfaces detail` for network type and timer behavior.
5. Review `show ip eigrp topology all-links` to see all candidate paths.
6. Check `show ip eigrp traffic` for query, reply, and ACK patterns.
7. Look for SIA symptoms if routes remain active too long.
8. Validate ACLs, control-plane policies, and split horizon behavior.

**Practical configuration examples**

Classic mode:

```bash
router eigrp 100
 network 10.1.0.0 0.0.255.255
 no auto-summary
 eigrp router-id 1.1.1.1
 passive-interface default
 no passive-interface GigabitEthernet0/0
```

Named mode:

```bash
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  network 10.1.0.0 0.0.255.255
  eigrp router-id 1.1.1.1
  no shutdown
```

Interface-level tuning:

```bash
interface GigabitEthernet0/0
 ip hello-interval eigrp 100 5
 ip hold-time eigrp 100 15
```

Use timer tuning carefully. Matching timers is not usually required for adjacency, but extreme mismatches can affect failure detection and operational consistency.

**Exam focus summary**

- **Neighbor formation** depends on matching EIGRP domain parameters, not just IP reachability.
- **Feasible successors** are determined by the feasibility condition using reported distance versus feasible distance.
- **Passive** means converged; **active** means DUAL is searching.
- **Queries and replies** drive convergence when no feasible successor exists.
- **Stub routing** and summarization reduce query scope and improve stability.
- **Variance** enables unequal-cost load balancing when feasible alternate paths exist.
- **Troubleshooting** should always include neighbors, topology, traffic counters, interface details, and filtering checks.
