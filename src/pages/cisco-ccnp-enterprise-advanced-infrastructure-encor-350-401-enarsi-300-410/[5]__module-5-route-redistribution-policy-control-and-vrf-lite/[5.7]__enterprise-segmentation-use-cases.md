## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Segmentation goals and design drivers**

Enterprise segmentation limits who can talk to whom, reduces blast radius, and helps enforce policy at the right layer. In practice, you choose a segmentation method based on the control point you need: Layer 2 isolation, Layer 3 separation, identity-based policy, or centralized fabric policy.

Common drivers include:

- **Security**: isolate users, servers, guests, and IoT devices.
- **Compliance**: separate regulated workloads and log policy decisions.
- **Fault containment**: prevent broadcast, routing, or malware spread.
- **Operational clarity**: make policy easier to reason about and troubleshoot.
- **Scalability**: avoid per-subnet ACL sprawl when the environment grows.

A useful exam mindset is to map the requirement to the enforcement layer:

- **Layer 2**: private VLANs, VLAN design, port isolation.
- **Layer 3**: VRF-lite, route leaking, inter-VRF policy.
- **Identity/policy**: Cisco TrustSec, SGACLs, downloadable ACLs.
- **Traffic control**: zone-based firewall, ACLs, NAT boundaries.
- **Fabric**: SD-Access virtual networks and group-based policy.
- **WAN**: SD-WAN VPN segmentation and centralized policy.

**Private VLANs and Layer 2 isolation**

Private VLANs (PVLANs) let you subdivide a single VLAN into isolated and community segments. They are useful when you want hosts to share the same IP subnet but not directly communicate at Layer 2.

Typical use cases:

- **Shared services with isolation**: multiple servers in a rack or hosting environment.
- **Guest or kiosk networks**: clients can reach a gateway but not each other.
- **DMZ-style segments**: reduce east-west traffic between similar endpoints.

PVLAN roles:

- **Primary VLAN**: the parent VLAN.
- **Secondary isolated VLAN**: hosts cannot talk to other isolated hosts.
- **Secondary community VLAN**: hosts can talk within the same community.
- **Promiscuous port**: can communicate with all secondary VLANs, often the default gateway or firewall.

Example topology idea:

- `VLAN 100` primary
- `VLAN 101` isolated for user A
- `VLAN 102` isolated for user B
- `VLAN 110` community for app servers
- Firewall or router on a promiscuous port

Example configuration:

```cisco
vlan 100
 private-vlan primary
 private-vlan association 101,102,110

vlan 101
 private-vlan isolated

vlan 102
 private-vlan isolated

vlan 110
 private-vlan community

interface GigabitEthernet1/0/1
 switchport mode private-vlan host
 switchport private-vlan host-association 100 101

interface GigabitEthernet1/0/2
 switchport mode private-vlan host
 switchport private-vlan host-association 100 102

interface GigabitEthernet1/0/24
 switchport mode private-vlan promiscuous
 switchport private-vlan mapping 100 101,102,110
```

Operational notes:

- PVLANs solve **same-subnet isolation** without requiring separate subnets for every endpoint.
- They do not replace Layer 3 policy; traffic leaving the subnet still needs routing or firewall control.
- They are less common in modern campus designs than VRFs or fabric-based segmentation, but they remain relevant in data center and special-purpose access designs.

**VRF-lite and Layer 3 segmentation**

A VRF (Virtual Routing and Forwarding) instance creates a separate routing table on the same device. VRF-lite is the common enterprise deployment of VRFs without MPLS (Multiprotocol Label Switching) provider backbone integration.

Use VRF-lite when you need:

- **Hard L3 separation** between departments, tenants, or environments.
- **Overlapping IP address spaces** across segments.
- **Independent routing policies** for guest, corporate, and management traffic.
- **Cleaner firewall insertion** between routing domains.

Key points:

- Interfaces in different VRFs do not share routes by default.
- A VRF can have its own static routes, connected routes, and dynamic routing neighbors.
- Inter-VRF communication requires an explicit design, such as a firewall, route leaking, or a shared services router.

Example configuration:

```cisco
ip vrf CORP
 rd 65000:10

ip vrf GUEST
 rd 65000:20

interface GigabitEthernet1/0/10
 description Corp user VLAN
 ip vrf forwarding CORP
 ip address 10.10.10.1 255.255.255.0

interface GigabitEthernet1/0/20
 description Guest VLAN
 ip vrf forwarding GUEST
 ip address 10.20.20.1 255.255.255.0

ip route vrf CORP 0.0.0.0 0.0.0.0 10.10.10.254
ip route vrf GUEST 0.0.0.0 0.0.0.0 10.20.20.254
```

Important exam distinction:

- A static route inside a VRF only affects that VRF.
- A route pointing to a next hop in another VRF does **not** automatically create inter-VRF connectivity unless the next hop is reachable in the same VRF or you use a proper leaking or forwarding design.
- Common inter-VRF methods include:
  - firewall-based policy between VRFs
  - route leaking with import/export or static route redistribution between VRFs where supported
  - shared services interfaces or loopbacks with careful next-hop reachability

Example of a common troubleshooting symptom:

- `show ip route vrf CORP` shows the default route.
- `ping vrf CORP 10.20.20.10` fails.
- The issue is often missing route leaking, missing return path, or an ACL/firewall policy blocking the flow.

Useful verification commands:

```cisco
show vrf
show ip route vrf CORP
show ip interface brief vrf all
show running-config | section vrf
```

**TrustSec and identity-based segmentation**

Cisco TrustSec uses identity information, especially Security Group Tags (SGTs), to classify traffic and apply policy based on group membership rather than only IP addresses. This is useful when users move, addresses change, or policy must follow identity across the network.

Core concepts:

- **SGT**: Security Group Tag, an identity label assigned to a user, device, or endpoint.
- **SGACL**: Security Group Access Control List, policy that permits or denies traffic between groups.
- **Classification**: mapping an endpoint to an SGT.
- **Propagation**: carrying the SGT through the network where supported.
- **Enforcement**: applying policy at a device that understands TrustSec.

Typical use cases:

- **Role-based access**: finance, HR, contractors, printers, and IoT.
- **Dynamic policy**: users keep the same access even when they move ports or subnets.
- **Microsegmentation**: reduce reliance on large IP ACLs.

Example policy idea:

- `Finance` can reach `ERP`
- `Users` can reach `Printers`
- `Guests` can reach only Internet
- `IoT` can reach only management services

A conceptual SGACL example:

```text
permit Finance ERP
deny   Guests Internal
permit Users Printers
deny   IoT Users
```

Practical notes:

- TrustSec is often used with Cisco ISE for identity source and policy distribution.
- Command availability varies by platform and IOS XE release, so verification is not universal across all devices.
- On many platforms, you verify policy through device-specific TrustSec or policy-map outputs, ISE authorization results, and packet path behavior rather than assuming one universal `show cts` command set.

Troubleshooting clues:

- Endpoint gets the wrong SGT.
- SGT propagation is missing on a hop.
- SGACL is not downloaded or not enforced on the device.
- A fallback ACL or local policy overrides the intended behavior.

**Zone-Based Firewall and traffic control boundaries**

Zone-Based Firewall (ZBFW) is a stateful firewall feature in Cisco IOS XE that defines security zones and policies between them. It is useful when you want to control traffic at a routed boundary without deploying a dedicated firewall appliance.

Use ZBFW when you need:

- **Stateful inspection** between trust zones.
- **Simple branch or edge segmentation**.
- **Policy enforcement** between inside, outside, and DMZ-like zones.

Important limitation:

- ZBFW is not a full application-layer segmentation platform. It can inspect some protocols and enforce stateful policy, but it is not a general replacement for dedicated next-generation firewall or identity-based segmentation.

Basic model:

- **Zones** group interfaces.
- **Zone-pairs** define traffic direction between zones.
- **Class-maps** match traffic.
- **Policy-maps** define inspect, pass, or drop actions.

Example:

```cisco
zone security INSIDE
zone security OUTSIDE

class-map type inspect match-any CM-WEB
 match protocol http
 match protocol https

policy-map type inspect PM-INSIDE-OUTSIDE
 class type inspect CM-WEB
  inspect
 class class-default
  drop

zone-pair security ZP-IO source INSIDE destination OUTSIDE
 service-policy type inspect PM-INSIDE-OUTSIDE

interface GigabitEthernet1/0/1
 zone-member security INSIDE

interface GigabitEthernet1/0/2
 zone-member security OUTSIDE
```

Why this matters:

- `inspect` creates state so return traffic is allowed.
- `pass` allows traffic without state creation.
- `drop` blocks traffic.
- If you place an interface into a zone without a matching zone-pair policy, traffic is denied by default between zones.

Troubleshooting example:

- Users can browse some sites but not others.
- `show policy-map type inspect zone-pair` indicates drops in the class-default class.
- The issue may be missing protocol matches, asymmetric routing, or a policy that is too restrictive.

**SD-Access segmentation and policy flow**

SD-Access uses a fabric to separate traffic with **virtual networks** and apply policy with **group-based controls**. In this model, virtual networks provide the primary Layer 3 segmentation boundary, while SGTs and SGACLs provide policy within and across those networks.

Key terms:

- **Virtual Network (VN)**: logical L3 segment, often mapped to a VRF in the fabric.
- **SGT**: identity tag used for policy.
- **Fabric edge**: attaches endpoints and enforces policy.
- **Fabric border**: connects the fabric to external networks.
- **Control plane node**: distributes endpoint reachability information.

Policy flow concept:

1. An endpoint is assigned to a VN and an SGT.
2. The fabric edge encapsulates traffic and preserves policy context where supported.
3. The fabric enforces L3 separation by VN/VRF.
4. SGT-based policy is evaluated for permitted flows between groups.
5. The border node handles traffic to external networks and may enforce or translate policy at the fabric edge.

Exam-focused distinction:

- Use the VN/VRF to separate broad domains such as corporate, guest, and IoT.
- Use SGTs to control which groups inside or across those domains can communicate.
- Do not treat SGTs as the primary replacement for the VN boundary.

Example use case:

- `Guest VN` cannot reach internal resources.
- `Corporate VN` can reach shared services.
- `Printers` SGT can be allowed from `Users` SGT but denied from `Guests`.
- `IoT` devices can reach only a management server in a shared services VN.

Troubleshooting symptoms:

- Endpoint lands in the wrong VN.
- SGT assignment is missing from the policy source.
- Border policy blocks traffic leaving the fabric.
- Return traffic fails because the external network lacks the correct route or policy.

**SD-WAN segmentation terminology**

Cisco SD-WAN uses **VPNs** as the primary segmentation construct. These are logical routing and policy domains on the WAN edge, not the same thing as IPsec VPN tunnels in the generic sense.

Key ideas:

- **VPN 0**: transport side, typically underlay-facing.
- **VPN 512**: management plane on many deployments.
- **Service VPNs**: user or application segments, such as corporate, guest, or voice.
- **Centralized policy**: controls route advertisement, data policy, and application behavior across sites.

Use SD-WAN segmentation when you need:

- **Branch segmentation** across many sites.
- **Central policy control** for routing and traffic steering.
- **Separate business units** or guest traffic at the WAN edge.

Example conceptual design:

- `VPN 10` for corporate users
- `VPN 20` for guest Internet access
- `VPN 30` for voice
- Central policy allows only selected inter-VPN flows, such as voice to call control

Practical note:

- SD-WAN segmentation is often enforced by VPN separation plus centralized policy, not by a single ACL on one interface.
- Route leaking between VPNs is controlled by policy and route import/export behavior, not by ad hoc static routes alone.

**Choosing the right segmentation tool**

| Requirement | Best fit | Why |
|---|---|---|
| Same subnet, host-to-host isolation | PVLAN | Keeps one subnet while blocking peer-to-peer L2 traffic |
| Separate routing domains | VRF-lite | Strong L3 separation and overlapping IP support |
| Identity-based access | TrustSec | Policy follows the user or device |
| Stateful boundary control | ZBFW | Enforces traffic direction and session state |
| Fabric-wide campus segmentation | SD-Access | Combines VN separation with group policy |
| Branch WAN segmentation | SD-WAN VPNs | Centralized segmentation across sites |

Design guidance:

- Choose **PVLANs** for niche L2 isolation requirements.
- Choose **VRFs** when you need clear routing separation or overlapping addresses.
- Choose **TrustSec** when policy must follow identity rather than subnet.
- Choose **ZBFW** when you need a routed security boundary with stateful inspection.
- Choose **SD-Access** when you want campus-wide segmentation with centralized policy.
- Choose **SD-WAN** when segmentation must extend across branch WAN sites.

**Verification and troubleshooting workflow**

A practical troubleshooting flow helps you isolate whether the failure is in classification, routing, or policy enforcement.

1. Confirm the segment assignment.
   - Check VLAN, VRF, VN, or SGT membership.
   - Verify the endpoint is in the intended policy domain.

2. Confirm reachability inside the segment.
   - Test local gateway reachability.
   - Verify connected routes and ARP/ND resolution.

3. Confirm policy intent.
   - Review ACLs, SGACLs, zone-pairs, or centralized policy.
   - Check whether the policy permits the expected direction.

4. Confirm return path.
   - Ensure the destination can route back.
   - Verify route leaking, border policy, or firewall state.

5. Confirm enforcement point behavior.
   - Look for drops, class-default hits, or missing tags.
   - Check logs and counters where supported.

Example troubleshooting commands:

```cisco
show ip route vrf CORP
show ip arp vrf CORP
show zone security
show policy-map type inspect zone-pair
show access-lists
show cts role-based permissions
```

> `show cts role-based permissions` and similar TrustSec commands are platform-dependent. Use them only when the platform and IOS XE release support them.

Example failure patterns:

- **Wrong VRF**: interface is in the default VRF instead of the intended tenant VRF.
- **Missing return route**: forward path exists, reply path does not.
- **Policy too narrow**: class-map matches only `http` and `https`, but the application also needs DNS or TLS-related flows.
- **Asymmetric routing**: stateful firewall sees only one direction.
- **Missing tag propagation**: SGT is assigned at the edge but not carried to the enforcement point.

**Practical design scenarios**

- **Campus guest access**
  - Use a guest VRF or guest VN.
  - Send traffic to Internet only.
  - Avoid relying on split tunneling as a segmentation control; it is an access design choice, not a security boundary.

- **Finance and HR separation**
  - Use VRFs for hard separation.
  - Add TrustSec for role-based access to shared services.
  - Use ZBFW or firewall policy for north-south control.

- **IoT and printers**
  - Use SGTs to distinguish device classes.
  - Use SGACLs to allow only required services.
  - Consider PVLANs if the devices share a subnet and need L2 isolation.

- **Multi-tenant branch**
  - Use SD-WAN service VPNs for tenant separation.
  - Apply centralized policy for inter-VPN access.
  - Use route control to prevent unintended leakage.

- **Data center shared services**
  - Use VRFs for tenant separation.
  - Use firewall policy or route leaking for controlled access to shared services.
  - Use PVLANs for isolated server clusters when L2 adjacency must be restricted.

**Exam reminders**

- **VRF-lite** is about routing-table separation, not just interface configuration.
- **Route leaking** must be explicit and must preserve a valid return path.
- **TrustSec** is identity-based; SGTs and SGACLs are the key concepts.
- **ZBFW** is stateful and zone-based, but not a universal application segmentation solution.
- **SD-Access** uses VN/VRF for primary segmentation and SGT/SGACL for policy.
- **SD-WAN** uses VPNs and centralized policy for branch segmentation.
- **PVLANs** are useful for same-subnet isolation, especially in special-purpose L2 designs.
