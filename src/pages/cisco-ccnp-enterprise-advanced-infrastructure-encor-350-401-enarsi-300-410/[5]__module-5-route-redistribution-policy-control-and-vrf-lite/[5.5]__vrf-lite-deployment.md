## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**VRF-Lite fundamentals**

VRF-Lite uses <abbr title="Virtual Routing and Forwarding">VRF</abbr> instances to keep multiple routing tables on the same device. You use it when you need tenant separation, shared services, or management isolation without deploying MPLS <abbr title="Multiprotocol Label Switching">MPLS</abbr> VPNs.

A VRF changes the forwarding context for interfaces and services:
- Each VRF has its own routing table, <abbr title="Forwarding Information Base">FIB</abbr>, and often its own <abbr title="Cisco Express Forwarding">CEF</abbr> entries.
- Interfaces placed in different VRFs can reuse the same IP subnets without conflict.
- Traffic does not cross VRFs unless you explicitly leak routes or use a shared service design.

VRF-Lite is common on campus distribution switches, edge routers, firewalls, and management planes. It is also a frequent exam topic because it combines interface configuration, routing protocol behavior, and troubleshooting.

**VRF concepts you must separate**

A VRF is not the same as a route distinguisher or route target.

- **Route distinguisher (RD)** makes overlapping prefixes unique in MPLS VPN control planes.
- **Route target (RT)** controls import and export of VPN routes between VRFs in MPLS VPN designs.
- **VRF-Lite** usually does not require RD/RT for local forwarding separation, but some platforms still use them for route leaking or VPN-style policy.

In practical terms:
- Use VRF-Lite when you want local segmentation on one device or a small set of devices.
- Use RD/RT concepts when the design involves VPNv4/VPNv6 exchange, provider-style route distribution, or route leaking through BGP policy.
- Do not assume that creating a VRF automatically shares routes with another VRF; it only isolates them.

**Platform syntax and creation patterns**

Cisco IOS and IOS XE support VRF creation with slightly different command styles depending on release and feature set. The most common modern syntax is:

```cisco
vrf definition CUST-A
 rd 65000:10
 address-family ipv4
 exit-address-family
 address-family ipv6
 exit-address-family
```

Older or alternate syntax may appear on some platforms:

```cisco
ip vrf CUST-A
 rd 65000:10
```

Use the platform-specific syntax that your device supports. On IOS XE, `vrf definition` is common and more flexible because it can include address-family configuration. On older IOS, `ip vrf` is often seen.

After creating the VRF, you bind interfaces to it:

```cisco
interface GigabitEthernet0/0/0
 description Customer A
 vrf forwarding CUST-A
 ip address 10.10.10.1 255.255.255.0
```

> `vrf forwarding` clears the interface IP address on many platforms. Reapply the address after assigning the VRF.

**Interface membership and forwarding behavior**

When you place an interface into a VRF:
- The interface stops using the global routing table.
- Connected routes appear only in that VRF.
- ARP, adjacency resolution, and recursive next-hop lookup happen inside that VRF.
- Management and service traffic sourced from that interface may also need VRF-aware configuration.

Typical use cases:
- Customer-facing interfaces in separate VRFs.
- A management VRF for out-of-band access.
- Shared-services VRFs for DNS, NTP, AAA, or logging.

**Verification of VRF creation and membership**

Use multiple commands because output varies by platform and release.

```cisco
show vrf
show vrf detail
show ip interface brief vrf all
show ip route vrf CUST-A
show ip cef vrf CUST-A
```

What to look for:
- VRF name exists.
- Interfaces are listed under the correct VRF.
- Connected routes appear in the expected VRF table.
- CEF shows the correct next-hop and adjacency resolution.

If a route is missing, check:
- Interface state
- Correct VRF assignment
- IP address and mask
- Duplicate subnet assumptions
- Routing protocol activation in the correct VRF

**Static routing inside a VRF**

Static routes are the simplest way to populate a VRF routing table. The next hop must be reachable in that same VRF, or you must specify an outgoing interface that is valid in that VRF.

Example:

```cisco
vrf definition CUST-A
 address-family ipv4
 exit-address-family

interface GigabitEthernet0/0/0
 vrf forwarding CUST-A
 ip address 10.10.10.1 255.255.255.0

interface GigabitEthernet0/0/1
 vrf forwarding CUST-A
 ip address 10.10.20.1 255.255.255.0

ip route vrf CUST-A 10.10.30.0 255.255.255.0 10.10.20.2
```

This works only if `10.10.20.2` is reachable inside `CUST-A`. If the next hop is not in the VRF, the route may install as unresolved or fail to forward.

For IPv6, the same principle applies:

```cisco
ipv6 route vrf CUST-A 2001:db8:30::/64 2001:db8:20::2
```

On some topologies, you may need an outgoing interface or a link-local next hop. Always verify the platform’s syntax and the neighbor’s reachability in the same VRF.

**Dynamic routing inside a VRF**

Dynamic routing protocols run independently per VRF. The exact syntax varies by protocol and platform, but the key idea is always the same: the process must be bound to the VRF.

**OSPF in a VRF**

A common IOS XE pattern is:

```cisco
router ospf 10 vrf CUST-A
 router-id 10.10.10.1
 passive-interface default
 no passive-interface GigabitEthernet0/0/0
```

Then enable OSPF on the interface or network as required by the platform:

```cisco
interface GigabitEthernet0/0/0
 vrf forwarding CUST-A
 ip address 10.10.10.1 255.255.255.0
 ip ospf 10 area 0
```

Notes:
- Some platforms use `router ospf <pid> vrf <name>`.
- Some use interface-based OSPF only.
- Do not assume global OSPF neighbors will form across a VRF boundary.

**EIGRP in a VRF**

Named mode is common in modern deployments:

```cisco
router eigrp CUST-A
 address-family ipv4 unicast autonomous-system 100
  af-interface default
   passive-interface
  exit-af-interface
  af-interface GigabitEthernet0/0/0
   no passive-interface
  exit-af-interface
 exit-address-family
```

Bind the interface to the VRF first:

```cisco
interface GigabitEthernet0/0/0
 vrf forwarding CUST-A
 ip address 10.10.10.1 255.255.255.0
```

Depending on platform and release, the VRF may need to be specified in the address-family or interface context. Verify the exact syntax before lab or exam use.

**BGP in a VRF**

BGP is often used for customer edge routing, route leaking, or service separation. For VRF-aware BGP, the neighbor relationship and activation must occur in the VRF address-family context.

Example pattern:

```cisco
router bgp 65000
 bgp log-neighbor-changes
 neighbor 10.10.10.2 remote-as 65010

 address-family ipv4 vrf CUST-A
  neighbor 10.10.10.2 activate
  neighbor 10.10.10.2 remote-as 65010
  redistribute connected
 exit-address-family
```

Important points:
- A neighbor in the global BGP context is not automatically valid for a VRF address-family.
- The neighbor must be reachable in the VRF.
- Some platforms require the neighbor to be defined under the VRF AF or with a VRF-specific update-source or interface context.
- `redistribute connected` only injects connected routes into that VRF’s BGP table; it does not leak routes into another VRF by itself.

**Route leaking between VRFs**

Route leaking is the controlled sharing of selected routes between VRFs. This is one of the most important VRF-Lite design topics.

There are three common approaches:

| Method | Best for | Strengths | Limitations |
|---|---|---|---|
| Static route leaking | Small, simple designs | Easy to understand, deterministic | Poor scalability, manual maintenance |
| Inter-VRF redistribution | Controlled lab or edge cases | Flexible policy control | Risk of loops and complexity |
| BGP import/export or VPN-style RT policy | Larger designs | Scalable and policy-driven | More complex, platform-dependent |

**Static route leaking**

A static route in one VRF can point to a next hop in another VRF only if the platform supports that behavior and the next hop is resolvable in the correct forwarding context. A more common and safer pattern is to use a shared-services interface or a dedicated transit VRF.

Example of a valid static route pattern:

```cisco
interface GigabitEthernet0/0/2
 description Shared-services transit
 vrf forwarding SHARED
 ip address 192.0.2.1 255.255.255.252

interface GigabitEthernet0/0/3
 description Customer A transit
 vrf forwarding CUST-A
 ip address 192.0.2.5 255.255.255.252

ip route vrf CUST-A 10.10.20.0 255.255.255.0 192.0.2.6
```

Here, `192.0.2.6` must be reachable in `CUST-A`. If you want `CUST-A` to reach a shared service, the shared service must be reachable through a valid path in that VRF or through an explicit leaking design.

**BGP-based route leaking**

In MPLS VPN-style designs, route leaking usually relies on route targets:
- Export routes from one VRF with an RT.
- Import those routes into another VRF with a matching RT.

Conceptually:

```cisco
vrf definition CUST-A
 rd 65000:10
 route-target export 65000:10
 route-target import 65000:100

vrf definition SHARED
 rd 65000:100
 route-target export 65000:100
 route-target import 65000:10
```

This is the cleanest policy model when the platform and design support it. It is not the same as simply redistributing connected routes into multiple VRF BGP address-families.

**Inter-VRF redistribution**

Some designs use a transit VRF or a pair of routing processes with redistribution and filtering. This can work, but you must control:
- Prefix filtering
- Administrative distance
- Route tags
- Redistribution loops

Use this only when route-target-based leaking is unavailable or not appropriate.

**Default route handling in VRFs**

Default routes are common in customer and management VRFs. You may need to:
- Inject a default route from a static next hop.
- Learn a default route from a routing protocol.
- Leak a default route from a shared-services or internet VRF.

Examples:

```cisco
ip route vrf MGMT 0.0.0.0 0.0.0.0 192.0.2.254
```

```cisco
router ospf 10 vrf CUST-A
 default-information originate
```

```cisco
router bgp 65000
 address-family ipv4 vrf SHARED
  network 0.0.0.0
```

Use default routes carefully:
- A management VRF often needs only a narrow set of destinations.
- A customer VRF may need a default route to a firewall or internet edge.
- A leaked default route can create unintended reachability if you do not filter it.

**Management and service VRFs**

VRFs are often used to isolate management traffic from production traffic. Common services include DNS, NTP, SNMP, syslog, TACACS+, and SSH.

Platform support varies, so treat service commands as feature-dependent. Common patterns include:
- Source the service from an interface in the management VRF.
- Specify the VRF in the service command if the platform supports it.
- Use a loopback in the management VRF as a stable source address.

Examples that may be platform-dependent:

```cisco
ip name-server vrf MGMT 192.0.2.53
snmp-server host 192.0.2.60 version 2c PUBLIC vrf MGMT
logging host 192.0.2.70 vrf MGMT
```

If your platform does not support VRF keywords for a service, configure a source interface or use a management-plane feature that binds the service to the VRF.

**DHCP in VRFs**

DHCP relay and DHCP server behavior can be VRF-aware. This matters when clients live in a customer VRF and the DHCP server lives in a shared-services VRF.

Typical relay pattern:

```cisco
interface GigabitEthernet0/0/0
 vrf forwarding CUST-A
 ip address 10.10.10.1 255.255.255.0
 ip helper-address 192.0.2.100
```

If the helper address is in another VRF, the platform may require:
- A VRF-aware helper syntax
- A reachable shared-services path
- Policy-based route leaking
- A relay agent configured in the correct VRF

Troubleshooting DHCP in VRFs usually starts with:
- Confirming the client interface VRF
- Verifying the helper address reachability
- Checking UDP forwarding and ACLs
- Confirming the server return path into the same VRF or through a valid leak

**Troubleshooting workflow**

Use a structured approach when VRF traffic fails.

1. Confirm the VRF exists.
2. Confirm the interface is in the correct VRF.
3. Confirm the route exists in the correct VRF table.
4. Confirm the next hop is reachable in that VRF.
5. Confirm the routing protocol is bound to the VRF.
6. Confirm services are VRF-aware if the traffic is management or control-plane related.

Useful commands:

```cisco
show vrf detail
show ip interface brief vrf all
show ip route vrf CUST-A
show ip route vrf CUST-A 10.10.30.0
show ip cef vrf CUST-A 10.10.30.0 detail
show arp vrf CUST-A
show ip ospf vrf CUST-A neighbor
show ip eigrp vrf CUST-A neighbors
show bgp ipv4 unicast vrf CUST-A summary
```

For deeper forwarding checks:
- `show ip cef vrf <name> <prefix> detail` helps confirm recursive resolution.
- `show adjacency vrf <name>` can help on some platforms.
- `ping vrf <name> <destination>` and `traceroute vrf <name> <destination>` validate data-plane reachability.

**Common failure cases**

- You assign the interface to the VRF after configuring the IP address, and the address disappears.
- You configure a route in the wrong VRF and the prefix never appears where you expect.
- You form a routing neighbor in the global table instead of the VRF.
- You leak a route with redistribution but forget to filter or tag it, creating loops.
- You assume a service command supports VRF on every platform and release.
- You use overlapping subnets correctly in separate VRFs, but forget that shared services still need explicit reachability.

**End-to-end lab example**

This example shows two customer VRFs and one shared-services VRF. The goal is to let both customers reach a DNS server in the shared VRF while keeping customer routes isolated from each other.

Topology:
- `CUST-A` on `GigabitEthernet0/0/0`
- `CUST-B` on `GigabitEthernet0/0/1`
- `SHARED` on `GigabitEthernet0/0/2`
- DNS server reachable in `SHARED`

Configuration:

```cisco
vrf definition CUST-A
 rd 65000:10
 address-family ipv4
 exit-address-family

vrf definition CUST-B
 rd 65000:20
 address-family ipv4
 exit-address-family

vrf definition SHARED
 rd 65000:100
 address-family ipv4
 exit-address-family

interface GigabitEthernet0/0/0
 description CUST-A LAN
 vrf forwarding CUST-A
 ip address 10.10.10.1 255.255.255.0

interface GigabitEthernet0/0/1
 description CUST-B LAN
 vrf forwarding CUST-B
 ip address 10.20.20.1 255.255.255.0

interface GigabitEthernet0/0/2
 description Shared services
 vrf forwarding SHARED
 ip address 192.0.2.1 255.255.255.252

ip route vrf CUST-A 192.0.2.100 255.255.255.255 192.0.2.2
ip route vrf CUST-B 192.0.2.100 255.255.255.255 192.0.2.2
```

This static example is only valid if `192.0.2.2` is the next hop reachable in the same VRF context for each customer. In a real design, you often use a shared-services transit or a route-leaking mechanism instead of pointing customer VRFs directly at a shared address without a valid path.

A more scalable design uses route targets:

```cisco
vrf definition CUST-A
 rd 65000:10
 route-target export 65000:10
 route-target import 65000:100
 address-family ipv4
 exit-address-family

vrf definition SHARED
 rd 65000:100
 route-target export 65000:100
 route-target import 65000:10
 route-target import 65000:20
 address-family ipv4
 exit-address-family
```

This lets you selectively share only the prefixes you intend to leak.

**Exam focus and design tradeoffs**

VRF-Lite questions often test whether you understand separation, not just syntax.

- Use VRFs to isolate overlapping address spaces.
- Bind interfaces and routing protocols to the correct VRF.
- Verify forwarding in the correct table, not just global routing.
- Choose route leaking carefully based on scale and policy.
- Remember that service-plane support is platform-specific.
- Distinguish local VRF separation from MPLS VPN control-plane concepts.

A good mental model is:
- VRF = separate routing context
- RD = unique prefix identity
- RT = import/export policy
- Route leaking = explicit sharing, never accidental

**Key takeaways**

- VRF-Lite gives you isolated routing tables on the same device.
- Interface membership determines which VRF owns connected routes and forwarding.
- Dynamic routing must be VRF-aware to form adjacencies and exchange routes.
- Route leaking requires explicit policy; redistribution alone is not enough.
- Verification must include both route-table and forwarding-plane checks.
- Platform syntax varies, so always confirm the exact IOS or IOS XE command set before deployment.
