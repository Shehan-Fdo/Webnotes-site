## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Mutual redistribution overview**

Mutual redistribution occurs when two routing domains exchange routes in both directions, such as OSPF into EIGRP and EIGRP back into OSPF. It is powerful for migrations and inter-domain connectivity, but it creates a risk of route feedback, duplicate advertisements, and unstable path selection.

The core hazard is simple: a route learned from one protocol can be redistributed into another protocol, then learned back again as if it were new information. Without controls, each domain may prefer the redistributed copy over the original, causing loops or route churn.

- **Route feedback** happens when a route leaves one protocol, enters another, and returns to the original domain.
- **Suboptimal routing** appears when the redistributed path wins over a better native path because of metric differences.
- **Route duplication** can occur when the same prefix exists in multiple domains with different attributes.
- **Default routes** can amplify the problem because they are easy to inject and easy to re-advertise.

**Why redistribution becomes dangerous**

Different routing protocols do not understand each other’s native metrics, administrative distance, or loop-prevention mechanisms. When you redistribute, you translate routes from one protocol’s worldview into another’s. That translation is lossy unless you add policy.

For example:

- OSPF uses cost and external route types.
- EIGRP uses composite metrics and external/internal route distinction.
- BGP uses path attributes and policy.
- RIP uses hop count and has a very small metric range.

Because each protocol evaluates routes differently, a route can be preferred in one domain even if it is already a redistributed copy from another domain. This is the classic feedback loop problem.

**Common loop patterns**

Mutual redistribution hazards show up most often in these pairings:

- **OSPF ↔ EIGRP**
  - A route learned from OSPF is redistributed into EIGRP as an external route.
  - EIGRP then redistributes it back into OSPF.
  - OSPF may prefer the reintroduced external route if policy is not controlled.
- **OSPF ↔ BGP**
  - BGP often carries many prefixes from multiple sources.
  - Redistributing BGP into OSPF can flood the IGP with routes that should remain external.
  - Redistributing OSPF back into BGP can create path ambiguity unless tags and policy are used.
- **EIGRP ↔ BGP**
  - BGP may inject a large set of prefixes into EIGRP.
  - EIGRP external routes can then be re-exported into BGP, creating feedback and poor path selection.
- **RIP with any dynamic protocol**
  - RIP’s low hop-count ceiling and simplistic metric make it especially vulnerable to poor redistribution design.
  - Mutual redistribution with RIP is usually a migration-only design and should be tightly filtered.

**How protocols treat redistributed routes**

Redistributed routes are usually marked as external in the receiving protocol.

- **OSPF external routes**
  - Type 5 LSAs are external routes flooded throughout the OSPF domain.
  - Type 7 LSAs are used in NSSA areas and translated to Type 5 by the ABR.
  - **E1** routes include the external metric plus the internal cost to reach the ASBR.
  - **E2** routes use only the external metric for route selection; the internal cost to the ASBR is used only as a tie-breaker when external metrics are equal.
- **EIGRP external routes**
  - External routes are tagged as EIGRP external and use a different administrative distance than internal routes.
  - The metric must be explicitly set or derived from a default metric during redistribution.
- **BGP redistributed routes**
  - Routes can carry communities, local preference, MED, and AS-path information, but redistribution into IGPs strips most BGP path semantics.
- **RIP redistributed routes**
  - Routes are converted to hop count, which is often a poor representation of real path quality.

> <mark>Important:</mark> OSPF E1 does not mean “internal cost is included in the external metric itself.” It means the total path cost is the external metric plus the internal cost to reach the ASBR.

**Why tags matter**

Route tags are one of the most important tools for preventing feedback loops. A tag is a numeric marker you attach to a route during redistribution so that downstream policy can recognize the route’s origin.

Tags are not automatically preserved in every scenario. You must explicitly set them in redistribution policy and then match them on the return path.

Typical uses:

- Mark routes redistributed from OSPF into EIGRP with one tag value.
- Block those tagged routes when redistributing EIGRP back into OSPF.
- Use different tags for different source domains or edge locations.
- Combine tags with prefix lists so you filter both by origin and by prefix.

**Practical redistribution example: OSPF to EIGRP and back**

In classic EIGRP mode, redistribution syntax is straightforward. In named mode, the configuration structure differs, so do not assume the same command placement or syntax.

Classic EIGRP example:

```cisco
router eigrp 100
 default-metric 10000 100 255 1 1500
 redistribute ospf 1 route-map OSPF-TO-EIGRP
```

OSPF example:

```cisco
router ospf 1
 redistribute eigrp 100 subnets route-map EIGRP-TO-OSPF
```

Route-map with tags:

```cisco
route-map OSPF-TO-EIGRP permit 10
 match ip address prefix-list OSPF-REDIST
 set tag 1001

route-map EIGRP-TO-OSPF permit 10
 match ip address prefix-list EIGRP-REDIST
 match tag 1001
 deny 20
```

Prefix lists:

```cisco
ip prefix-list OSPF-REDIST seq 10 permit 10.10.0.0/16 le 24
ip prefix-list EIGRP-REDIST seq 10 permit 10.20.0.0/16 le 24
```

This design does two things:

- It limits which prefixes are redistributed.
- It prevents routes that originated in OSPF and were tagged `1001` from being sent back into OSPF again.

**EIGRP redistribution specifics**

EIGRP redistribution requires special attention because the protocol needs a metric for external routes.

You can provide the metric in one of two ways:

- Set a **default-metric** under the EIGRP process.
- Specify the metric directly in the redistribution command or route-map, depending on platform and mode.

Classic EIGRP example:

```cisco
router eigrp 100
 default-metric 10000 100 255 1 1500
 redistribute ospf 1 route-map OSPF-TO-EIGRP
```

The metric values represent:

- Bandwidth in Kbps
- Delay in tens of microseconds
- Reliability
- Load
- MTU

If you omit the metric and do not configure a default metric, redistribution may fail for EIGRP external routes.

Named EIGRP uses a different configuration hierarchy, so verify the exact syntax on the platform and software release you are studying.

**OSPF redistribution specifics**

OSPF redistribution is usually more forgiving than EIGRP because you can set a default metric for external routes.

Example:

```cisco
router ospf 1
 redistribute eigrp 100 subnets metric 20 metric-type 1 route-map EIGRP-TO-OSPF
```

Key points:

- `subnets` is commonly required when redistributing classless routes.
- `metric` sets the external cost.
- `metric-type 1` or `metric-type 2` changes how the route is evaluated.
- `route-map` lets you filter and tag routes.

Use **E1** when the internal cost to the ASBR should influence selection across the OSPF domain. Use **E2** when you want the external metric to dominate and internal cost to matter only as a tie-breaker.

**BGP redistribution hazards**

BGP is often used as a transit or edge protocol, so redistribution into and out of BGP can create large-scale feedback if you are not careful.

Common risks:

- Redistributing many IGP prefixes into BGP without filtering.
- Reinjecting BGP-learned routes back into an IGP and then back into BGP.
- Losing BGP path attributes when routes enter an IGP.
- Accidentally advertising internal infrastructure routes to external peers.

Example:

```cisco
router bgp 65001
 redistribute ospf 1 route-map OSPF-TO-BGP
```

If you later redistribute BGP into OSPF, you should filter aggressively:

```cisco
router ospf 1
 redistribute bgp 65001 subnets route-map BGP-TO-OSPF
```

Use prefix lists, route maps, and tags to prevent the same prefix from bouncing between the protocols.

**RIP redistribution hazards**

RIP is especially fragile in redistribution designs because it uses hop count and has a maximum usable metric of 15.

Hazards include:

- Large topologies becoming unreachable because redistributed hop counts exceed the limit.
- Routes bouncing between RIP and another protocol with poor metric translation.
- Default routes being overused as a workaround, hiding real reachability issues.

RIP redistribution is usually acceptable only in small migration scenarios or isolated edge segments.

**Default route hazards**

Default routes are a common source of accidental feedback.

Typical patterns:

- `default-information originate` in OSPF injects a default only if the router already has one, unless `always` is used.
- `redistribute static` can inject a static default into a dynamic protocol.
- A default learned from one protocol can be redistributed into another and then re-learned back.

Example:

```cisco
router ospf 1
 default-information originate
```

Example with a static default:

```cisco
ip route 0.0.0.0 0.0.0.0 192.0.2.1
router eigrp 100
 redistribute static route-map STATIC-DEFAULT
```

Be careful not to create a situation where multiple domains keep re-advertising the same default route. That can hide failures and make troubleshooting harder.

**Filtering methods beyond route maps**

Route maps are the most flexible tool, but they are not the only one.

- **Prefix lists**
  - Best for matching exact prefixes or ranges.
  - Useful for controlling what enters redistribution.
- **Distribute-lists**
  - Filter routes entering or leaving a routing process in some protocols.
  - Less expressive than route maps.
- **Route-policy concepts**
  - Common in platforms and protocols that support policy-based routing decisions more directly.
  - Often used in larger designs or on platforms with richer policy syntax.
- **Administrative distance tuning**
  - Can influence which protocol wins locally.
  - Does not solve feedback by itself.
- **Summarization**
  - Reduces the number of specific routes redistributed.
  - Can hide detail and reduce churn, but may also obscure failures.

A good design often combines multiple controls:

- Filter at the source.
- Tag on redistribution.
- Block tagged routes on re-entry.
- Summarize where appropriate.
- Avoid redistributing everything by default.

**Recognizing redistribution loops in the field**

You can often spot a redistribution problem from the routing table and protocol databases before users report a full outage.

Symptoms include:

- The same prefix appears as external in multiple protocols.
- Route ownership changes repeatedly after convergence.
- Metrics fluctuate without a real topology change.
- A route is present in one protocol’s database but missing from another’s expected source.
- Traffic follows a longer or unexpected path after redistribution.

Useful verification commands:

```cisco
show ip route
show ip route <prefix>
show ip protocols
show ip ospf database external
show ip eigrp topology
show ip bgp
show route-map
show ip route tag <tag-value>
```

Depending on platform and IOS XE release, `show route-map` output may vary, so confirm the exact syntax supported on your device.

**Troubleshooting workflow**

1. Identify the prefix and determine which protocol should be the original source.
2. Check whether the route is being redistributed into another domain.
3. Verify tags, prefix lists, and route-map matches.
4. Confirm the receiving protocol’s metric requirements are satisfied.
5. Look for the same prefix being reintroduced from the opposite direction.
6. Validate whether a default route is masking the real path.
7. Check protocol databases for external route type and origin information.

Useful debugs depend on the platform and issue, but these are more relevant than protocol-specific finite-state-machine debugs for redistribution analysis:

```cisco
debug ip routing
debug ip eigrp
debug ip ospf events
```

Use debugs carefully in production because they can be CPU-intensive.

**Example loop scenario**

Imagine this topology:

- Site A runs OSPF.
- Site B runs EIGRP.
- A border router redistributes OSPF into EIGRP and EIGRP into OSPF.
- A prefix `10.10.10.0/24` originates in OSPF at Site A.

Without protection:

- The prefix enters EIGRP as an external route.
- EIGRP advertises it back into OSPF.
- OSPF may see the reintroduced route as a valid external path.
- If the original source changes or fails, the redistributed copy can become the preferred path, even though it is just a reflection of the same route.

With protection:

- Tag the route when it leaves OSPF.
- Block that tag when it returns from EIGRP to OSPF.
- Filter the prefix so only intended networks are redistributed.
- Use appropriate metrics so the receiving protocol does not prefer an unintended path.

**Design best practices**

- **Redistribute only what you must.**
  - Smaller policy sets are easier to verify and less likely to loop.
- **Use tags consistently.**
  - Treat tags as origin markers, not as decoration.
- **Filter at every boundary.**
  - Do not rely on a single route map to solve all directions.
- **Summarize where possible.**
  - Fewer specifics mean fewer opportunities for feedback.
- **Avoid mutual redistribution unless required.**
  - A single redistribution direction is usually safer.
- **Document the source of truth.**
  - Decide which protocol owns each prefix.
- **Test failure cases.**
  - Verify what happens when one side loses reachability or a default route appears.

**Quick comparison of hazards**

| Pairing | Main risk | Typical control |
|---|---|---|
| OSPF ↔ EIGRP | External route feedback and metric mismatch | Tags, route maps, explicit metrics |
| OSPF ↔ BGP | Large-scale route leakage and re-advertisement | Prefix filtering, tags, selective redistribution |
| EIGRP ↔ BGP | External route churn and path ambiguity | Policy, summarization, strict redistribution boundaries |
| RIP ↔ Any IGP | Hop-count exhaustion and poor metric translation | Very tight filtering, migration-only use |

**Exam focus points**

- Know that mutual redistribution can create loops even when each protocol is stable on its own.
- Distinguish **E1** and **E2** behavior in OSPF external route selection.
- Remember that EIGRP redistribution needs a usable metric.
- Understand that tags are a policy tool, not automatic loop prevention.
- Recognize that default routes can be redistributed and reintroduced just like any other prefix.
- Be able to explain why filtering must happen in both directions.

**Key takeaway**

Mutual redistribution is not inherently wrong, but it is inherently risky. The safest designs define a clear route source, translate only the required prefixes, tag redistributed routes, and block them from coming back in the opposite direction. If you can explain where a route originated, how it was translated, and why it cannot re-enter the source domain, you have solved the core redistribution hazard.
