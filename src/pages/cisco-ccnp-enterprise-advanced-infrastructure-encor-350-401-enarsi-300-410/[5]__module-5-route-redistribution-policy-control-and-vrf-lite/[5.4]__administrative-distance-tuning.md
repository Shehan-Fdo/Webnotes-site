## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Administrative distance fundamentals**

Administrative distance (AD) is the local router’s trust ranking for routes learned from different sources. Lower values are preferred. It is not advertised to neighbors and does not change the routing protocol itself; it only affects how the local device chooses the installed route when multiple candidates exist.

Common default AD values on Cisco IOS/IOS XE include:

- **Connected**: 0
- **Static**: 1
- **eBGP**: 20
- **EIGRP internal**: 90
- **OSPF**: 110
- **IS-IS**: 115
- **RIP**: 120
- **EIGRP external**: 170
- **iBGP**: 200
- **Unknown/untrusted**: 255

A route with a lower AD wins over a route with a higher AD for the same prefix. If AD is equal, Cisco then uses the protocol’s own best-path logic or route selection rules. For example, OSPF compares cost, EIGRP compares composite metric, and BGP uses BGP best-path attributes rather than a generic IGP-style metric.

**Why AD tuning matters**

You tune AD when you want to influence which route the local router installs without changing the routing design everywhere else. Typical use cases include:

- Preferring a static backup route over a dynamic route only when the dynamic route fails.
- Making one routing protocol win over another for the same destination.
- Controlling redistribution outcomes in a lab or migration scenario.
- Protecting against accidental route installation from a less trusted source.

> AD tuning is local to the router where you configure it. It does not propagate to neighbors and does not change how other routers rank the same prefix.

**How Cisco compares routes**

Cisco route selection for a prefix generally follows this order:

1. **Longest prefix match** first.
2. **Administrative distance** among routes for the same prefix length.
3. **Protocol-specific selection** if multiple candidates from the same source remain.
4. **Equal-cost multipath (ECMP)** if the protocol and platform allow multiple equal best paths.

This means AD does not compete across different prefix lengths. A `/24` route always beats a covering `/16` for a matching destination, even if the `/16` has a lower AD.

**Core configuration patterns**

Cisco IOS/IOS XE provides several `distance` forms. The exact syntax supported can vary by platform and release, so verify on the target image.

**Global AD override for a routing protocol**

Use this when you want to change the default trust level for all routes from a protocol on the local router.

```cisco
router ospf 10
 distance 90
!
router eigrp 100
 distance 95
!
router bgp 65000
 distance bgp 20 200 200
```

- OSPF example lowers OSPF AD from 110 to 90.
- EIGRP example raises internal EIGRP AD from 90 to 95.
- BGP example sets AD for eBGP, iBGP, and local routes in one command.

**Per-source AD tuning**

Use this when you want to trust only specific neighbors or sources more or less than the protocol default.

```cisco
router ospf 10
 distance 110 10.1.1.2 0.0.0.0
```

This form is platform-dependent and is used to set a different AD for routes learned from a specific OSPF neighbor or source. In practice, IOS/IOS XE commonly supports source-based AD tuning with wildcard masks, but you should confirm the exact syntax on the device because support and parsing can differ by release.

For EIGRP, the common lab form is:

```cisco
router eigrp 100
 distance 90 10.1.1.0 0.0.0.255
```

This changes the AD for routes learned from sources matching the address and wildcard mask. Use this when one neighbor should be preferred over another for the same prefix.

**BGP-specific distance tuning**

BGP has a dedicated `distance bgp` command because its route preference is often compared against IGPs and static routes.

```cisco
router bgp 65000
 distance bgp 20 200 200
```

- First value: eBGP AD
- Second value: iBGP AD
- Third value: local BGP route AD

This is useful when you want BGP to win over or lose to an IGP on the local router. Be careful: lowering BGP AD too much can cause BGP-learned routes to override routes you intended to keep from the IGP.

**Static route AD tuning**

Static routes use the optional AD at the end of the route statement.

```cisco
ip route 203.0.113.0 255.255.255.0 192.0.2.1 200
ip route 198.51.100.0 255.255.255.0 192.0.2.2 1
```

- The first static route is a floating static backup because AD 200 is worse than most dynamic protocols.
- The second static route is a primary static route because AD 1 is very trusted.

A floating static route is one of the most common AD tuning patterns in enterprise labs.

**Exam-style route preference examples**

**Static vs OSPF**

If both routes exist for `10.10.10.0/24`, the lower AD wins.

```cisco
ip route 10.10.10.0 255.255.255.0 192.0.2.1 1
router ospf 10
 network 10.10.10.0 0.0.0.255 area 0
```

Expected outcome:

- Static route installs first because AD 1 is lower than OSPF 110.
- If you change the static route to AD 200, OSPF wins and the static becomes a backup.

**EIGRP vs OSPF**

```cisco
router eigrp 100
 network 10.20.20.0 0.0.0.255
!
router ospf 10
 network 10.20.20.0 0.0.0.255 area 0
```

Expected outcome:

- EIGRP internal route wins by default because AD 90 is lower than OSPF 110.
- If you raise EIGRP AD above 110, OSPF becomes preferred on that router.

**BGP vs IGP**

```cisco
router bgp 65000
 neighbor 192.0.2.9 remote-as 65100
 network 172.16.50.0 mask 255.255.255.0
!
router ospf 10
 network 172.16.50.0 0.0.0.255 area 0
```

Expected outcome:

- OSPF wins over iBGP by default because 110 is lower than 200.
- eBGP can win over OSPF by default because 20 is lower than 110.

**Equal AD behavior**

If two routes to the same prefix have the same AD, Cisco does not choose based on AD anymore. It then uses the protocol’s own selection process.

- For OSPF, lower cost wins.
- For EIGRP, lower composite metric wins.
- For BGP, best-path attributes decide.
- For static routes, multiple equal static routes can be installed if they are valid and the platform supports load sharing.

If the routes are truly equal and the platform supports it, Cisco may install multiple next hops for ECMP.

**Verification commands**

Use `show ip route` to confirm which route is installed and why.

```cisco
show ip route 10.10.10.0
show ip route
show ip route static
show ip route ospf
show ip route eigrp
```

Typical output clues:

- The route code shows the source protocol, such as `S`, `O`, `D`, or `B`.
- The bracketed value shows `[AD/metric]`.
- The next hop and outgoing interface show the installed path.

Example:

```cisco
R1# show ip route 10.10.10.0
Routing entry for 10.10.10.0/24
  Known via "ospf 10", distance 110, metric 20
  Routing Descriptor Blocks:
  * 192.0.2.1, from 192.0.2.1, 00:00:12 ago
      Route metric is 20, traffic share count is 1
```

To verify protocol-level AD settings, use `show ip protocols`.

```cisco
R1# show ip protocols
Routing Protocol is "ospf 10"
  Distance: 90
```

This is important because the routing table alone may not show the configured override clearly.

**Practical lab example: floating static backup**

A common design is to prefer OSPF during normal operation and use a static route only if OSPF disappears.

```cisco
ip route 10.1.50.0 255.255.255.0 192.0.2.254 200
router ospf 10
 network 10.1.50.0 0.0.0.255 area 0
```

Behavior:

- OSPF route installs while it exists.
- If OSPF fails, the static route with AD 200 becomes active.
- This is useful for WAN backup, migration windows, and emergency reachability.

**Practical lab example: preferring one redistribution source**

In redistribution scenarios, the same prefix can appear from multiple sources. AD tuning helps the local router choose the desired path.

```cisco
router ospf 10
 redistribute eigrp 100 subnets
 distance 95
!
router eigrp 100
 redistribute ospf 10 metric 10000 100 255 1 1500
```

Use this carefully:

- Redistribution can create duplicate routes from different protocols.
- AD tuning decides which redistributed copy the local router installs.
- It does not prevent redistribution loops by itself; you still need route tagging, filtering, and design controls.

**Important distinction: local preference vs domain-wide preference**

AD affects only the router where it is configured. It is not a routing-domain-wide policy.

- It does not change what neighbors advertise.
- It does not alter the protocol’s metric in updates.
- It does not replace route filtering or redistribution policy.
- It does not solve all path-selection problems in a multi-router design.

If you need to influence the whole network, use protocol design tools such as:

- OSPF cost
- EIGRP metric manipulation
- BGP local preference, weight, AS-path prepending, or MED
- Route maps and prefix lists for redistribution control

**Common commands and syntax patterns**

```cisco
! Static route with AD
ip route <prefix> <mask> <next-hop> <distance>

! OSPF global distance
router ospf <process-id>
 distance <value>

! OSPF source-based distance
router ospf <process-id>
 distance <value> <ip-address> <wildcard-mask>

! EIGRP global distance
router eigrp <asn>
 distance <value>

! EIGRP source-based distance
router eigrp <asn>
 distance <value> <ip-address> <wildcard-mask>

! BGP distance
router bgp <asn>
 distance bgp <ebgp> <ibgp> <local>
```

**Troubleshooting checklist**

1. Confirm the prefix exists in more than one source with `show ip route`.
2. Check the bracketed `[AD/metric]` values for each candidate.
3. Verify the configured AD with `show ip protocols`.
4. Confirm the route is not being hidden by a longer prefix match.
5. Check whether the route is from the same protocol, where protocol-specific best-path rules may apply after AD ties.
6. Validate platform support if a `distance` form is rejected or behaves differently.

**Platform and version cautions**

Cisco IOS and IOS XE support most common AD tuning features, but exact syntax and behavior can vary by release and feature set.

- Some per-neighbor or per-source forms are more limited than global protocol distance changes.
- BGP distance syntax is widely supported, but route selection behavior still follows BGP best-path rules before installation.
- Lab images and production images may differ in parser strictness.
- Always verify on the target platform with `?`, `show ip protocols`, and `show running-config`.

**Key exam takeaways**

- **Lower AD wins** for the same prefix.
- AD is **local only** and is not advertised.
- AD is compared **before** protocol-specific metrics or best-path logic.
- Static routes can be made **floating** by assigning a higher AD.
- `show ip route` confirms the installed route; `show ip protocols` confirms protocol AD settings.
- BGP uses **best-path attributes**, not a generic IGP metric, so treat BGP distance tuning separately from IGP metric logic.
