## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Redistribution fundamentals**

Route redistribution injects routes learned in one routing domain into another routing domain. You use it when you must connect separate protocols, migrate between protocols, or leak selected routes between administrative boundaries.

The key exam idea is that redistribution is not a simple copy operation. Each protocol has its own route attributes, metric model, and loop-prevention behavior. When a route crosses protocol boundaries, some information is translated, some is discarded, and some must be supplied by policy.

- **Source protocol** is the routing process that originally learned the route.
- **Destination protocol** is the routing process that receives the redistributed route.
- **Seed metric** is the starting metric assigned to the redistributed route in the destination protocol.
- **Administrative distance** still decides which route wins if multiple protocols advertise the same prefix.

A route can be redistributed from:
- Connected interfaces
- Static routes
- OSPF
- EIGRP
- BGP

In practice, redistribution is usually controlled with:
- `route-map`
- `prefix-list`
- `distribute-list`
- `tag`
- `default-metric`
- protocol-specific metric statements

> Redistribution is a policy decision. If you do not control it carefully, you can create loops, suboptimal paths, or route loss.

**Core design rules**

Before configuring redistribution, decide:
- Which prefixes should move
- Which direction the routes should flow
- What metric the destination protocol should use
- How you will prevent feedback loops
- How you will verify the result

A good design usually includes:
- Explicit filtering
- Explicit metrics
- Route tagging
- Clear default-route handling
- Verification commands for both the source and destination protocols

**OSPF redistribution**

OSPF (Open Shortest Path First) requires an external metric for redistributed routes. On Cisco IOS/IOS XE, you must provide a metric directly or via a route-map. OSPF does not rely on a generic `default-metric` mechanism the way EIGRP does.

Typical external route types:
- **E1**: external metric plus internal OSPF cost to the ASBR
- **E2**: external metric only, unless tie-breaking requires internal cost comparison

E2 is the common default for redistributed routes. E1 is often preferred when you want the route to reflect distance to the ASBR as well as the external cost.

Example redistribution from static routes into OSPF:

```cisco
router ospf 10
 router-id 1.1.1.1
 redistribute static subnets metric 20 metric-type 1
```

If you use a route-map, the route-map can set the metric and metric type:

```cisco
route-map STATIC-TO-OSPF permit 10
 match ip address prefix-list STATIC-ROUTES
 set metric 50
 set metric-type type-2
 set tag 100

ip prefix-list STATIC-ROUTES seq 5 permit 10.10.10.0/24
ip prefix-list STATIC-ROUTES seq 10 permit 10.20.20.0/24

router ospf 10
 redistribute static subnets route-map STATIC-TO-OSPF
```

Important OSPF points:
- `subnets` is commonly used with redistribution so classless routes are included.
- External routes appear as `O E1` or `O E2` in the routing table.
- OSPF external LSAs are Type 5 in normal areas.
- In NSSA areas, redistributed routes are carried as Type 7 LSAs and translated to Type 5 by the NSSA ABR if translation is enabled.

**OSPF NSSA redistribution**

An NSSA (Not-So-Stubby Area) allows external routes inside a stub-like area. The ASBR inside the NSSA injects redistributed routes as Type 7 LSAs. The NSSA ABR can translate Type 7 to Type 5 when the route leaves the NSSA.

Key behaviors:
- Type 7 LSAs stay inside the NSSA until translated.
- The ABR performs translation to Type 5 for the rest of the OSPF domain.
- If multiple ABRs exist, translation and forwarding behavior can affect which external route is preferred.

Useful verification:
- `show ip ospf database nssa-external`
- `show ip ospf database external`
- `show ip route ospf`

Example NSSA configuration:

```cisco
router ospf 10
 area 1 nssa
 redistribute static subnets metric 20 metric-type 1
```

If the area is configured as NSSA, redistributed routes inside that area are not handled the same way as in a normal area. For exam questions, always identify whether the ASBR is inside the NSSA and whether the route must be translated at the ABR.

**EIGRP redistribution**

EIGRP (Enhanced Interior Gateway Routing Protocol) needs a seed metric for redistributed routes. You can provide the metric directly in the redistribution command or configure a `default-metric` under the EIGRP process.

EIGRP metric components:
- Bandwidth
- Delay
- Reliability
- Load
- MTU (Maximum Transmission Unit, carried but not used in the composite metric)

The classic exam-safe approach is to set a `default-metric` before redistribution if you are redistributing multiple route sources.

Example using `default-metric`:

```cisco
router eigrp 100
 network 10.0.0.0 0.255.255.255
 default-metric 100000 100 255 1 1500
 redistribute static
 redistribute ospf 10
```

Example using an explicit metric on redistribution:

```cisco
router eigrp 100
 redistribute static metric 100000 100 255 1 1500
```

If you omit both the explicit metric and `default-metric`, redistributed routes do not have a usable EIGRP metric and will not be installed as expected.

Route-map example for EIGRP redistribution:

```cisco
route-map OSPF-TO-EIGRP permit 10
 match tag 100
 set metric 100000 100 255 1 1500
 set tag 200

router eigrp 100
 redistribute ospf 10 route-map OSPF-TO-EIGRP
```

This pattern is useful when you want to:
- Match only routes previously tagged by another redistribution point
- Assign a consistent EIGRP seed metric
- Rewrite the tag to mark the route as having crossed a boundary

**EIGRP metric manipulation**

When redistributing into EIGRP, you often need to preserve a realistic metric so the route is preferred correctly inside the EIGRP domain. The metric should reflect the path you want EIGRP to believe is best.

Practical guidance:
- Use a low delay and reasonable bandwidth for preferred routes.
- Keep the metric consistent across all redistributed prefixes if they share the same policy.
- Use route-maps to assign different metrics to different prefix groups.

Example with multiple metric classes:

```cisco
route-map STATIC-TO-EIGRP permit 10
 match ip address prefix-list CRITICAL
 set metric 10000 10 255 1 1500
 set tag 10

route-map STATIC-TO-EIGRP permit 20
 match ip address prefix-list NONCRITICAL
 set metric 100000 100 255 1 1500
 set tag 20

router eigrp 100
 redistribute static route-map STATIC-TO-EIGRP
```

**BGP redistribution**

BGP (Border Gateway Protocol) behaves differently from IGPs. A prefix is not advertised just because it exists somewhere in the topology. It must be present in the BGP table through `network`, redistribution, or learned BGP paths, and policy must permit advertisement.

Common ways to originate a prefix into BGP:
- `network` statement
- `redistribute connected`
- `redistribute static`
- `redistribute ospf`
- `redistribute eigrp`

Example:

```cisco
router bgp 65000
 bgp log-neighbor-changes
 network 203.0.113.0 mask 255.255.255.0
 redistribute static
```

Important BGP caveats:
- `network` requires an exact match in the routing table unless the prefix is present through a more specific route that satisfies the BGP process on the platform.
- Redistribution into BGP can leak many routes if you do not filter carefully.
- BGP attributes such as local preference, AS path, and MED are not preserved when routes are injected into an IGP.
- When BGP routes are redistributed into an IGP, the IGP sees the ASBR as the originator of the external route, but forwarding still depends on the ASBR’s reachability to the BGP next hop.

**BGP to IGP redistribution**

When you redistribute BGP into OSPF or EIGRP, you are converting policy-rich BGP routes into simpler IGP external routes. This loses BGP-specific attributes.

What is lost:
- AS path
- Local preference
- MED (Multi-Exit Discriminator)
- Communities
- Next-hop attribute as a BGP attribute

What remains:
- The prefix
- The redistributed metric you assign
- Optional tag values if you set them

Example BGP into OSPF:

```cisco
route-map BGP-TO-OSPF permit 10
 match ip address prefix-list BGP-EXPORT
 set metric 100
 set metric-type type-2
 set tag 65000

ip prefix-list BGP-EXPORT seq 5 permit 198.51.100.0/24

router ospf 10
 redistribute bgp 65000 subnets route-map BGP-TO-OSPF
```

Example BGP into EIGRP:

```cisco
route-map BGP-TO-EIGRP permit 10
 match ip address prefix-list BGP-EXPORT
 set metric 100000 100 255 1 1500
 set tag 65000

router eigrp 100
 redistribute bgp 65000 route-map BGP-TO-EIGRP
```

If the BGP route is learned from an external neighbor and then redistributed into an IGP, make sure the ASBR has a valid route to the BGP next hop. Otherwise, the IGP may install the external prefix, but forwarding can fail because the ASBR cannot reach the actual next hop.

**Static route redistribution**

Static routes are often redistributed when you want to inject:
- A default route
- A small set of controlled prefixes
- Summaries or stub networks

Example:

```cisco
ip route 10.10.10.0 255.255.255.0 Null0
ip route 0.0.0.0 0.0.0.0 192.0.2.1
```

Redistribute static into OSPF:

```cisco
router ospf 10
 redistribute static subnets metric 20 metric-type 1
```

Redistribute static into EIGRP:

```cisco
router eigrp 100
 default-metric 100000 100 255 1 1500
 redistribute static
```

Redistribute static into BGP:

```cisco
router bgp 65000
 redistribute static
```

Default route handling differs by protocol:
- In OSPF, you often use `default-information originate` for a controlled default route, or redistribute a static default if you want it treated as an external route.
- In EIGRP, you can redistribute a static default or use `ip default-network` in older designs, though redistribution is more common in modern labs.
- In BGP, you can advertise a default with `network 0.0.0.0` only if the route exists in the routing table, or use `default-originate` toward a neighbor.

**Administrative distance and route selection**

Administrative distance (AD) decides which routing source is trusted when multiple protocols advertise the same prefix. Redistribution can create competing paths from different protocols, so AD matters during troubleshooting.

Common Cisco default AD values:
- Connected: 0
- Static: 1
- eBGP: 20
- EIGRP internal: 90
- OSPF: 110
- IS-IS: 115
- RIP: 120
- EIGRP external: 170
- iBGP: 200

Implications:
- A static route usually wins over redistributed dynamic routes unless you change its AD.
- An EIGRP external route loses to OSPF internal routes if both exist for the same prefix.
- A BGP-learned route may not install if a lower-AD route already exists in the RIB (Routing Information Base).

Example troubleshooting scenario:
- OSPF redistributes a static route.
- EIGRP also redistributes the same static route.
- The router may prefer the lower-AD route source, but the destination protocol still maintains its own best path selection internally.

**Route tags and loop prevention**

Route tags help you identify routes that have already crossed a redistribution boundary. They are essential in multi-protocol designs where a route could be reintroduced into its source protocol.

Typical loop-prevention pattern:
- Tag routes when they leave the source domain.
- Match the tag on re-entry.
- Deny or rewrite the route-map accordingly.

Example design:
- Static routes are redistributed into OSPF with tag `100`.
- OSPF routes are redistributed into EIGRP only if they do not carry tag `100`.
- EIGRP routes are redistributed back into OSPF only if they carry a different tag or no tag.

Example:

```cisco
route-map STATIC-TO-OSPF permit 10
 match ip address prefix-list STATIC-ROUTES
 set tag 100
 set metric 20
 set metric-type type-2

route-map OSPF-TO-EIGRP deny 10
 match tag 100

route-map OSPF-TO-EIGRP permit 20
 set metric 100000 100 255 1 1500
 set tag 200
```

This is more reliable than simply matching a tag and rewriting it without a clear deny/permit structure. The deny clause blocks routes that already came from the static-to-OSPF path.

**Filtering techniques**

Redistribution should almost never be unrestricted in production or exam labs. You can filter at several points.

Common tools:
- `route-map`
- `prefix-list`
- `distribute-list`
- `offset-list` in some EIGRP scenarios
- `summary-address` or route summarization
- `default-information originate` instead of full redistribution for default routes

Examples:
- Use a `prefix-list` to select only approved prefixes.
- Use a `route-map` to set metrics and tags while filtering.
- Use `distribute-list` to filter routes entering or leaving a routing process where supported.

Example `distribute-list` with EIGRP:

```cisco
ip prefix-list EIGRP-IN seq 5 permit 10.0.0.0/8 le 24

router eigrp 100
 distribute-list prefix EIGRP-IN in
```

Example `route-map` with OSPF redistribution:

```cisco
route-map STATIC-TO-OSPF deny 5
 match ip address prefix-list BLOCKED

route-map STATIC-TO-OSPF permit 10
 match ip address prefix-list ALLOWED
 set metric 50
 set metric-type type-1
```

**Default route origination versus redistribution**

A default route can be injected in multiple ways, and the method matters.

Use `default-information originate` when:
- You want OSPF to advertise a default only if the router already has one in its routing table
- You want controlled default propagation

Use `default-information originate always` when:
- You want OSPF to advertise a default regardless of whether the router currently has one

Use `redistribute static` when:
- You want to inject a static default as an external route
- You also want to redistribute other static routes

Example OSPF default origination:

```cisco
router ospf 10
 default-information originate
```

Example BGP default origination toward a neighbor:

```cisco
router bgp 65000
 neighbor 192.0.2.2 default-originate
```

Example EIGRP default redistribution:

```cisco
ip route 0.0.0.0 0.0.0.0 192.0.2.1

router eigrp 100
 default-metric 100000 100 255 1 1500
 redistribute static
```

**End-to-end lab scenario**

Consider this topology:

- R1 runs OSPF area 0 and has static routes to a server network.
- R2 runs EIGRP and connects to R1.
- R3 runs BGP and connects to R2.
- You want selected static routes from R1 to reach R3, and selected BGP routes from R3 to reach R1.

A practical redistribution plan:
- R1 redistributes static into OSPF with tags.
- R2 redistributes OSPF into EIGRP with a route-map that blocks tagged routes from looping back.
- R3 redistributes BGP into EIGRP or OSPF only for approved prefixes.
- Default routes are handled separately with explicit origination.

Example configuration fragments:

```cisco
! R1
ip prefix-list STATIC-ROUTES seq 5 permit 10.10.10.0/24
route-map STATIC-TO-OSPF permit 10
 match ip address prefix-list STATIC-ROUTES
 set metric 20
 set metric-type type-1
 set tag 100

router ospf 10
 redistribute static subnets route-map STATIC-TO-OSPF
```

```cisco
! R2
route-map OSPF-TO-EIGRP deny 10
 match tag 100

route-map OSPF-TO-EIGRP permit 20
 set metric 100000 100 255 1 1500
 set tag 200

router eigrp 100
 redistribute ospf 10 route-map OSPF-TO-EIGRP
```

```cisco
! R3
ip prefix-list BGP-EXPORT seq 5 permit 198.51.100.0/24
route-map BGP-TO-EIGRP permit 10
 match ip address prefix-list BGP-EXPORT
 set metric 100000 100 255 1 1500
 set tag 300

router eigrp 100
 redistribute bgp 65000 route-map BGP-TO-EIGRP
```

Expected outcomes:
- R1 advertises only approved static routes into OSPF.
- R2 accepts OSPF routes except those tagged as already redistributed from static.
- R3 injects only approved BGP prefixes into EIGRP.
- Tags help you identify the route’s path during troubleshooting.

**Verification commands**

Use protocol-specific commands to confirm both control-plane and RIB behavior.

OSPF:
- `show ip ospf`
- `show ip ospf database external`
- `show ip ospf database nssa-external`
- `show ip route ospf`

EIGRP:
- `show ip eigrp topology`
- `show ip eigrp topology all-links`
- `show ip route eigrp`

BGP:
- `show ip bgp`
- `show ip bgp rib-failure`
- `show ip bgp neighbors`
- `show ip route bgp`

Static and general:
- `show ip route`
- `show running-config | section router`
- `show route-map`
- `show ip prefix-list`

Useful troubleshooting workflow:
1. Confirm the source route exists in the source protocol or RIB.
2. Confirm the redistribution policy matches the route.
3. Confirm the destination protocol has the required metric.
4. Confirm the route appears in the destination protocol database.
5. Confirm the route installs in the routing table.
6. Check for AD conflicts, recursion issues, or filtering.

**Common failure cases**

Missing metric:
- EIGRP redistribution fails or produces unusable routes if you do not set a seed metric.

Route-map mismatch:
- The route exists, but the `match` clause does not select it, so nothing is redistributed.

Tag loop:
- A route is redistributed out of one protocol and then re-entered through another path because the tag filter is missing or incorrect.

Recursion problem:
- The ASBR cannot reach the BGP next hop, so the redistributed route is present but forwarding fails.

RIB failure:
- BGP learns a route, but a lower-AD route already exists, so the BGP route does not install in the main routing table.

NSSA translation issue:
- A Type 7 route does not appear outside the NSSA because translation or area design is incorrect.

**Exam-focused reminders**

- OSPF redistribution needs an external metric.
- EIGRP redistribution needs a seed metric or `default-metric`.
- BGP redistribution is policy-driven and can lose BGP attributes when injected into IGPs.
- Tags are one of the best tools for loop prevention.
- Default route origination is not the same as full redistribution.
- Administrative distance can explain why a route is learned but not installed.
- Verification must include both the routing protocol database and the routing table.

**Quick comparison**

| Topic | OSPF | EIGRP | BGP | Static |
|---|---|---|---|---|
| Metric requirement | Required | Required | Not used the same way | N/A |
| Common redistribution control | `route-map`, `metric`, `metric-type` | `route-map`, `default-metric`, explicit metric | `network`, `redistribute`, policy | `redistribute static` |
| External route handling | Type 5 or Type 7 in NSSA | External EIGRP route | BGP path attributes | Source of injected routes |
| Loop prevention | Tags, filtering | Tags, filtering | Policy, filtering | Tags, filtering |

**Practical command patterns**

Redistribute static into OSPF with explicit external cost:

```cisco
router ospf 10
 redistribute static subnets metric 20 metric-type 2
```

Redistribute OSPF into EIGRP with a route-map and seed metric:

```cisco
route-map OSPF-TO-EIGRP permit 10
 set metric 100000 100 255 1 1500
 set tag 200

router eigrp 100
 redistribute ospf 10 route-map OSPF-TO-EIGRP
```

Redistribute selected BGP routes into OSPF:

```cisco
ip prefix-list BGP-EXPORT seq 5 permit 203.0.113.0/24

route-map BGP-TO-OSPF permit 10
 match ip address prefix-list BGP-EXPORT
 set metric 100
 set metric-type type-1
 set tag 65000

router ospf 10
 redistribute bgp 65000 subnets route-map BGP-TO-OSPF
```

**Troubleshooting checklist**

- Verify the route exists in the source table.
- Verify the route-map permits the prefix.
- Verify the destination protocol has the correct metric.
- Verify tags are not blocking the route unintentionally.
- Verify the route is not filtered by a distribute-list or prefix-list.
- Verify the route is not failing due to recursion or next-hop reachability.
- Verify the route is not losing to a lower-AD route in the RIB.
