## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Route-map fundamentals**

A **route-map** is a policy engine used by Cisco IOS and IOS XE to match routes or packets and then apply actions. It is most commonly used with BGP, redistribution, and policy-based routing (PBR). The design goal is simple: match specific traffic or routes, then either permit, deny, or modify attributes.

A route-map is built from ordered sequences. Each sequence has:
- a sequence number
- an action: `permit` or `deny`
- one or more `match` statements
- optional `set` statements

If a sequence has no `match` statements, it matches everything. That is powerful, but dangerous if you intend a narrow policy.

**Core syntax**

```cisco
route-map NAME permit|deny SEQ
 match ...
 set ...
```

Example skeleton:

```cisco
route-map RM-BGP-IN permit 10
 match ip address prefix-list PL-IN
 set local-preference 200
```

Important design points:
- **Sequence order** matters; Cisco evaluates sequences from lowest to highest number.
- A sequence only applies when its match criteria are satisfied.
- The effect of a `permit` or `deny` depends on the feature using the route-map.
- Unmatched items are handled by the feature, not by a universal route-map rule.

> Route-map behavior is feature-specific. Do not assume BGP, redistribution, and PBR treat `permit` and `deny` the same way.

**Route-map structure and editing**

Route-maps are often modified in production, so you need to know how to manage sequence numbers safely.

Common operations:
- Add a new sequence between existing ones by choosing an unused number, such as `15`.
- Replace a sequence by re-entering the same sequence number.
- Remove a sequence with `no route-map NAME permit|deny SEQ`.
- Rename or rebuild policies by creating a new route-map and swapping references during maintenance.

Example:

```cisco
route-map RM-EDGE permit 10
 match ip address prefix-list PL-EDGE
 set community 65000:100 additive

route-map RM-EDGE permit 20
 set local-preference 150
```

Useful operational commands:
```cisco
show route-map
show running-config | section route-map
show ip protocols
```

**Match logic**

A route-map sequence can include multiple `match` statements. Cisco treats multiple match statements within the same sequence as an **AND** relationship for many features: all required conditions must be satisfied for the sequence to match. Some match types are feature-dependent, so always verify the platform and use case.

Common match objects:
- `match ip address prefix-list`
- `match ip address`
- `match as-path`
- `match community`
- `match metric`
- `match tag`
- `match interface`
- `match length`
- `match ip next-hop`

Examples:

```cisco
route-map RM-BGP-IN permit 10
 match ip address prefix-list PL-CUSTOMERS
 match community COMM-CUST
 set local-preference 200
```

```cisco
route-map RM-REDIST deny 10
 match tag 100
```

This is a common loop-prevention pattern: routes tagged on one redistribution path are blocked from being re-injected elsewhere.

**Set actions and feature support**

Not every `set` action works in every feature. This is one of the most important exam topics.

Typical `set` actions by use case:

- **BGP policy**
  - `set local-preference`
  - `set as-path prepend`
  - `set community`
  - `set metric` or `set metric-type` in some redistribution contexts
  - `set weight` on some platforms/features
  - `set origin`
  - `set next-hop`
- **Redistribution**
  - `set metric`
  - `set metric-type`
  - `set tag`
  - sometimes `set community` when redistributing into BGP
- **PBR**
  - `set ip next-hop`
  - `set ip default next-hop`
  - `set interface`
  - `set default interface` on some platforms

Unsupported or ignored actions are a common source of confusion. For example, `set local-preference` is meaningful for BGP policy, but not for PBR. Likewise, `set ip next-hop` is a PBR action, not a redistribution action.

**BGP route-map design**

BGP uses route-maps for both filtering and attribute manipulation.

Typical inbound uses:
- increase `local-preference` for preferred paths
- tag or classify routes with communities
- filter routes by prefix, AS path, or community

Typical outbound uses:
- prepend the AS path to make a path less preferred
- set communities for downstream policy
- filter which routes are advertised

Example inbound policy:

```cisco
ip prefix-list PL-CUST1 seq 10 permit 10.10.0.0/16 le 24
route-map RM-BGP-IN permit 10
 match ip address prefix-list PL-CUST1
 set local-preference 200
 set community 65000:10 additive
route-map RM-BGP-IN deny 100
```

Example outbound policy:

```cisco
ip prefix-list PL-OUT seq 10 permit 172.16.0.0/16 le 24
route-map RM-BGP-OUT permit 10
 match ip address prefix-list PL-OUT
 set as-path prepend 65000 65000 65000
 set community 65000:200 additive
route-map RM-BGP-OUT deny 100
```

Design notes:
- Use prefix-lists or communities to keep policies readable.
- Use `additive` when you want to preserve existing communities.
- Be careful with a sequence that has no `match` statements; it matches everything and can affect all advertised routes if applied outbound.
- Use explicit deny sequences when you want to block specific routes, but remember the feature determines how a deny is interpreted.

BGP verification commands:
```cisco
show ip bgp
show ip bgp neighbors 192.0.2.2 advertised-routes
show ip bgp neighbors 192.0.2.2 received-routes
show ip bgp neighbors 192.0.2.2 routes
show ip bgp community
```

> `show ip bgp neighbors ... routes` is not available on every IOS/IOS XE release. Use `advertised-routes` or `received-routes` when needed.

**BGP attribute design patterns**

Common policy goals:
- **Inbound preference**: set higher `local-preference` for preferred exits.
- **Outbound traffic engineering**: use AS-path prepending to make a path less attractive to external peers.
- **Route classification**: use communities to mark routes for downstream policy.
- **Selective advertisement**: use prefix-lists and route-maps to control what leaves the AS.

Example of a simple dual-ISP outbound design:

```cisco
ip prefix-list PL-TO-ISP1 seq 10 permit 203.0.113.0/24
ip prefix-list PL-TO-ISP2 seq 10 permit 203.0.113.0/24

route-map RM-ISP1-OUT permit 10
 match ip address prefix-list PL-TO-ISP1
 set as-path prepend 65000 65000

route-map RM-ISP2-OUT permit 10
 match ip address prefix-list PL-TO-ISP2
```

This keeps the policies aligned with the intended prefixes. Do not reuse the wrong prefix-list on both peers unless you intentionally want identical filtering.

**Redistribution route-map design**

Redistribution is where route-maps are especially important for control and loop prevention. A route-map can:
- filter which routes are redistributed
- set a metric
- tag routes so they are not re-redistributed later
- classify routes by source or prefix

Example: OSPF into EIGRP with tagging

```cisco
route-map RM-OSPF-TO-EIGRP permit 10
 match tag 10
 set metric 10000 100 255 1 1500
 set tag 110

route-map RM-OSPF-TO-EIGRP deny 20
 match tag 20
```

Example redistribution command structure:

```cisco
router eigrp 100
 redistribute ospf 10 metric 10000 100 255 1 1500 route-map RM-OSPF-TO-EIGRP
```

Better practice is to keep the redistribution command syntax correct for the target protocol and use the route-map for policy decisions. For EIGRP, the seed metric must be valid for EIGRP. The route-map does not replace the need for a proper metric.

Key redistribution design rules:
- Use **tags** to prevent feedback loops between routing domains.
- Set a valid **metric** for the target protocol.
- Use explicit deny sequences when you want to block a class of routes.
- Remember that a deny sequence with no match statements is a catch-all deny and matches everything not already matched above it.

Example of a catch-all deny:

```cisco
route-map RM-REDIST deny 100
```

This sequence matches all remaining routes and blocks them. It is useful when you want only explicitly permitted routes to pass.

**Protocol-specific redistribution notes**

- **OSPF**
  - External routes can be redistributed as type 1 or type 2.
  - `set metric-type` can influence external type selection in some designs.
  - Tags are commonly used to avoid re-injection.
- **EIGRP**
  - Requires a valid composite metric.
  - Seed metric handling is critical.
  - Route-maps are often used to tag and filter, but they do not eliminate the need for correct EIGRP metric configuration.
- **BGP redistribution**
  - Redistributed routes often need communities, origin, or MED-related policy.
  - Be careful not to leak internal routes unintentionally.
  - Filtering by tag or prefix-list is common before redistribution into BGP.

**Policy-based routing design**

PBR uses route-maps to override normal destination-based forwarding for selected traffic. It is typically applied inbound on an interface.

Basic PBR example:

```cisco
ip access-list extended ACL-PBR
 permit ip 10.10.10.0 0.0.0.255 any

route-map RM-PBR permit 10
 match ip address ACL-PBR
 set ip next-hop 192.0.2.1
```

Apply it to an interface:

```cisco
interface GigabitEthernet0/0
 ip policy route-map RM-PBR
```

PBR design notes:
- PBR affects packets entering the interface where the policy is applied.
- `set ip next-hop` sends matched traffic to a specified next hop.
- `set ip default next-hop` is a fallback-style action used when the destination is not in the routing table, subject to platform behavior.
- If the policy next hop is unreachable, the router may fall back to normal routing depending on the platform and configuration.
- Recursive next-hop resolution matters; the next hop must be reachable in the routing table for the policy to work as intended.

Example with default next hop:

```cisco
route-map RM-PBR permit 10
 match ip address ACL-PBR
 set ip default next-hop 192.0.2.253
```

Use this when you want policy routing only if the normal routing table does not already provide a usable path, but verify platform behavior carefully.

Useful PBR verification commands:
```cisco
show ip policy
show route-map
show access-lists ACL-PBR
show ip cef exact-route 10.10.10.10 8.8.8.8
show ip cef 192.0.2.1
```

**PBR failover and tracking**

For resilient designs, pair PBR with object tracking so the policy next hop is removed when the path fails.

Example:

```cisco
track 1 ip sla 1 reachability

ip sla 1
 icmp-echo 192.0.2.1 source-interface GigabitEthernet0/0
 frequency 5
ip sla schedule 1 life forever start-time now

route-map RM-PBR permit 10
 match ip address ACL-PBR
 set ip next-hop verify-availability 192.0.2.1 1 track 1
```

This design helps avoid blackholing traffic when the preferred next hop fails.

Design considerations:
- Use tracking for critical traffic steering.
- Verify that the next hop is reachable through the intended path.
- Test failover behavior under real link loss, not just administrative shutdown.

**Route-map behavior matrix**

| Feature | Common use | Typical `set` actions | Deny behavior |
|---|---|---|---|
| BGP | Filter and modify route attributes | `local-preference`, `community`, `as-path prepend`, `next-hop` | Usually blocks the route from that policy action |
| Redistribution | Control which routes move between protocols | `metric`, `metric-type`, `tag` | Usually prevents redistribution of matched routes |
| PBR | Steer packets | `ip next-hop`, `ip default next-hop`, `interface` | Usually means the packet is not policy-routed by that sequence |

This table is a study aid, not a universal rulebook. Always confirm the exact feature behavior on the platform and IOS family you are using.

**Troubleshooting route-maps**

When a route-map does not behave as expected, check the policy from the outside in:
1. Confirm the route-map is applied to the correct feature and direction.
2. Verify the match object actually matches the intended routes or packets.
3. Check sequence order and whether a broader sequence is catching traffic first.
4. Confirm the `set` action is valid for that feature.
5. Inspect counters and the resulting routing table or forwarding behavior.

Common misconfigurations:
- Prefix-list does not match because of wrong mask or `le`/`ge` values.
- Community match fails because `send-community` is missing on BGP neighbors.
- Redistribution policy blocks everything because a catch-all deny sequence is placed too early.
- PBR next hop is unreachable, so traffic follows normal routing or is dropped depending on platform behavior.
- A route-map sequence with no match statements unintentionally matches all routes.

Example troubleshooting workflow:

```cisco
show route-map RM-BGP-IN
show ip prefix-list PL-CUST1
show ip bgp neighbors 192.0.2.2 received-routes
show ip bgp neighbors 192.0.2.2 advertised-routes
show ip policy
show ip cef exact-route 10.10.10.10 8.8.8.8
```

**Production change and safety practices**

Route-maps are often tied to live routing policy, so changes should be controlled.

Recommended practices:
- Build a new route-map and test it before replacing the old one.
- Use sequence numbers to insert policy without rewriting the entire map.
- Document the intent of each sequence.
- Use explicit prefix-lists and tags to avoid accidental broad matches.
- Validate the effect on a maintenance window when the policy impacts transit or redistribution.

Safe modification example:

```cisco
route-map RM-EDGE permit 10
 match ip address prefix-list PL-EDGE
 set community 65000:100 additive

route-map RM-EDGE permit 20
 match ip address prefix-list PL-EDGE-BACKUP
 set community 65000:200 additive
```

If you need to remove a sequence:

```cisco
no route-map RM-EDGE permit 20
```

**Exam-focused takeaways**

- A route-map is an ordered policy tool with `permit` and `deny` sequences.
- Sequence numbers control evaluation order and make incremental edits easier.
- Match logic and set actions are feature-dependent.
- BGP route-maps often manipulate attributes such as `local-preference`, communities, AS-path, and next hop.
- Redistribution route-maps commonly filter routes, set metrics, and tag routes to prevent loops.
- PBR route-maps steer packets with next-hop or interface actions and may use tracking for failover.
- A sequence with no match statements matches everything, so use it carefully.
- Always verify the exact behavior on the feature and platform you are configuring.
