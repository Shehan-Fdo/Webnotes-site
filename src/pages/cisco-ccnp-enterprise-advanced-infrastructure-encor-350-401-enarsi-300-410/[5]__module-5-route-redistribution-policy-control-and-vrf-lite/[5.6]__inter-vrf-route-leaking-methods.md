## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Inter-VRF route leaking overview**

Inter-VRF route leaking lets you share selected routes between separate <abbr title="Virtual Routing and Forwarding">VRF</abbr>s without merging their full routing tables. You use it when two tenants, departments, or services must communicate while keeping most prefixes isolated.

On Cisco IOS and IOS XE, the practical methods you should know are:

- **Static route leaking** for simple, controlled point-to-point sharing.
- **BGP-based leaking** when you want policy control and scalable route exchange.
- **MPLS L3VPN route-target import/export** when VRFs participate in a provider-style VPN design.
- **Redistribution-based designs** as a broader control-plane pattern, usually with careful filtering.

A key exam distinction is that VRF-lite enterprise leaking and MPLS VPN route exchange are related ideas, but not the same operational model. In VRF-lite, you often leak routes on a single router or pair of routers. In MPLS Layer 3 VPN, route-targets and MP-BGP carry VPN routes between provider edge devices.

**Core concepts**

A VRF has its own:

- Routing table
- Forwarding table
- Interfaces
- Neighbor relationships, depending on the protocol

For leaking to work, you must solve two problems:

- **Route visibility**: the destination VRF must learn the prefix.
- **Next-hop reachability**: the destination VRF must be able to forward traffic toward the advertised next hop.

That second point is where many designs fail. A route can appear in the table but still not forward if the next hop is unreachable inside that VRF.

**Common design patterns**

| Method | Best for | Strengths | Limitations |
|---|---|---|---|
| Static leaking | Small, predictable route sets | Simple, deterministic, easy to troubleshoot | Poor scalability, manual maintenance |
| BGP leaking | Policy-based exchange | Scales better, supports filtering and attributes | More configuration, platform behavior varies |
| MPLS L3VPN RT import/export | Provider VPNs and PE-based VRFs | Standard VPN control plane, scalable | Requires MP-BGP and VPN infrastructure |
| Redistribution | Controlled redistribution between routing domains | Flexible in lab and service designs | Easy to create loops or leaks without filtering |

**Static inter-VRF route leaking**

Static leaking is the most direct method. You install a route in one VRF that points to a next hop reachable through another VRF or through the global table, depending on the platform and design.

A common enterprise pattern is a shared-services router or firewall that sits between VRFs. Another pattern is leaking a route from a VRF to the global table and then from the global table into another VRF. This is useful when the global table acts as a transit or services domain.

> Be careful: static VRF route syntax and next-hop resolution behavior are platform and release dependent. Always verify the exact IOS/IOS XE syntax on the target platform.

A typical VRF-lite interface and static route example looks like this:

```cisco
vrf definition BLUE
 address-family ipv4
 exit-address-family

vrf definition RED
 address-family ipv4
 exit-address-family

interface GigabitEthernet0/0
 description BLUE-LAN
 vrf forwarding BLUE
 ip address 10.10.10.1 255.255.255.0

interface GigabitEthernet0/1
 description RED-LAN
 vrf forwarding RED
 ip address 10.20.20.1 255.255.255.0
```

If you want BLUE to reach a RED subnet through a shared next hop, you may use a VRF-aware static route form supported by the platform, or a route through a transit interface in the same VRF. The exact syntax can differ, so treat the following as a design pattern rather than a universal command:

```cisco
ip route vrf BLUE 10.20.20.0 255.255.255.0 10.10.10.254
```

If the next hop is not directly connected in that VRF, you may need recursive resolution or a fully specified route, depending on platform support. In lab work, always confirm with:

```cisco
show ip route vrf BLUE
show ip cef vrf BLUE
show vrf
show ip interface brief vrf all
```

For IPv6, the same caution applies. A route may require a reachable next hop inside the VRF:

```cisco
ipv6 route vrf BLUE 2001:db8:20::/64 2001:db8:10::2
```

Use static leaking when:

- Only a few prefixes need to be shared.
- You want the simplest possible control plane.
- You can tolerate manual updates.

Avoid it when:

- Many prefixes change frequently.
- You need policy-based filtering.
- You need dynamic failover or attribute-based path selection.

**BGP-based inter-VRF leaking**

BGP is the most flexible enterprise-style leaking method when you need policy control. The important idea is that BGP does not magically leak routes between VRFs just because you enable it. You must explicitly import routes into the target VRF, or build a design where one VRF advertises routes that another VRF learns through a controlled BGP relationship.

A common IOS/IOS XE approach is to use a route reflector-like or shared-services topology with VRF-aware BGP sessions and route policies. Another approach is to use the global table as a transit domain and selectively redistribute or import routes. The exact mechanism depends on the platform and design.

A simple conceptual topology:

- VRF `BLUE` contains user networks.
- VRF `SHARED` contains services such as DNS, AD, or management.
- You want `BLUE` to reach only selected `SHARED` prefixes.

One practical pattern is to run BGP in each VRF and control what is advertised and accepted with route-maps and prefix-lists:

```cisco
router bgp 65000
 bgp log-neighbor-changes

 address-family ipv4 vrf BLUE
  redistribute connected route-map BLUE-CONN
  neighbor 10.255.255.2 remote-as 65000
  neighbor 10.255.255.2 activate
 exit-address-family

 address-family ipv4 vrf SHARED
  redistribute connected route-map SHARED-CONN
  neighbor 10.255.255.1 remote-as 65000
  neighbor 10.255.255.1 activate
 exit-address-family
```

This alone only places routes into BGP within each VRF. To actually leak routes, you need a policy that imports the desired prefixes into the other VRF or a design where the receiving VRF learns them through a controlled BGP adjacency and redistribution path. In practice, that usually means one of these:

- A dedicated services router or firewall that participates in both VRFs.
- A route-leaking policy using BGP import/export features supported by the platform.
- MPLS VPN-style route-target import/export if the design is provider-based.

A more explicit route-policy example for a shared-services design is:

```cisco
ip prefix-list BLUE-TO-SHARED seq 10 permit 10.10.10.0/24
ip prefix-list SHARED-TO-BLUE seq 10 permit 10.30.30.0/24

route-map BLUE-OUT permit 10
 match ip address prefix-list BLUE-TO-SHARED

route-map SHARED-OUT permit 10
 match ip address prefix-list SHARED-TO-BLUE
```

Then apply the policy to the relevant BGP process or redistribution point. The exact syntax varies by IOS XE release and topology, so the exam-safe takeaway is:

- BGP gives you filtering and policy.
- BGP alone does not create inter-VRF reachability unless routes are intentionally imported or redistributed.
- You must verify both the BGP table and the VRF routing table.

Useful verification commands include:

```cisco
show ip bgp vpnv4 all
show ip bgp vpnv4 vrf BLUE
show bgp ipv4 unicast vrf BLUE
show ip route vrf BLUE
show ip route vrf SHARED
```

On newer IOS XE releases, `show bgp ipv4 unicast vrf <name>` is common. On MPLS VPN platforms, `show ip bgp vpnv4 vrf <name>` is often the more relevant command. Use the command family that matches the feature set on the device.

**MPLS L3VPN route-target import/export**

Route-targets are the standard control-plane mechanism for MPLS Layer 3 VPNs. They are not, by themselves, a complete leaking solution. They work only when the PE router also has:

- A working MP-BGP <abbr title="Multiprotocol Border Gateway Protocol">MP-BGP</abbr> VPNv4/VPNv6 exchange
- VRFs attached to customer-facing interfaces
- Customer routes present in those VRFs

A VRF definition with route-targets looks like this:

```cisco
vrf definition CUST-A
 rd 65000:10
 address-family ipv4
  route-target export 65000:100
  route-target import 65000:200
 exit-address-family

vrf definition CUST-B
 rd 65000:20
 address-family ipv4
  route-target export 65000:200
  route-target import 65000:100
 exit-address-family
```

This configuration means:

- `CUST-A` exports routes with RT `65000:100`.
- `CUST-B` imports routes with RT `65000:100`.
- `CUST-B` exports routes with RT `65000:200`.
- `CUST-A` imports routes with RT `65000:200`.

That is the route-leaking policy. But it only works when the PE advertises VPN routes through MP-BGP and the routes are actually learned from attached interfaces or other valid sources.

A minimal PE-side context often includes:

```cisco
router bgp 65000
 address-family vpnv4
  neighbor 192.0.2.1 activate
  neighbor 192.0.2.1 send-community extended
 exit-address-family

 address-family ipv4 vrf CUST-A
  redistribute connected
 exit-address-family

 address-family ipv4 vrf CUST-B
  redistribute connected
 exit-address-family
```

Important points:

- `send-community extended` is required so route-targets are carried.
- The VRF must have routes to advertise.
- The PE must exchange VPN routes with other PEs or with itself in a lab topology.

Verification commands commonly used in MPLS VPN designs:

```cisco
show ip bgp vpnv4 all
show ip bgp vpnv4 vrf CUST-A
show ip route vrf CUST-A
show ip route vrf CUST-B
show mpls ldp neighbor
show bgp vpnv4 unicast all
```

If the route appears in BGP but not in the VRF table, check:

- Route-target import/export mismatch
- Missing `send-community extended`
- No valid route in the source VRF
- Missing MP-BGP adjacency
- Interface not bound to the expected VRF

**Redistribution-based leaking**

Redistribution is often used as part of a leaking design, but it is not a complete solution by itself. You can redistribute connected, static, or BGP-learned routes into another routing process or VRF-aware BGP instance, then filter carefully.

This is powerful, but it is also the easiest way to create loops. Use route-maps, prefix-lists, and tag values to prevent re-injection.

Example of a controlled redistribution pattern:

```cisco
route-map TAG-BLUE permit 10
 set tag 110

route-map REDIST-BLUE permit 10
 match tag 110

router bgp 65000
 address-family ipv4 vrf BLUE
  redistribute connected route-map TAG-BLUE
 exit-address-family

 address-family ipv4 vrf SHARED
  redistribute bgp 65000 route-map REDIST-BLUE
 exit-address-family
```

This kind of design is highly topology dependent. In many real deployments, you would prefer route-targets or a dedicated services router instead of broad redistribution.

Use redistribution when:

- You need to move a small, controlled set of routes between domains.
- You can tag and filter routes reliably.
- You understand the return path and loop-prevention requirements.

**End-to-end lab patterns**

A good exam-prep lab should show source VRF, destination VRF, route exchange, and verification.

1. **Static leak lab**
   - Create `BLUE` and `RED` VRFs.
   - Bind one interface to each VRF.
   - Add a static route in `BLUE` pointing to a reachable next hop that leads to `RED`.
   - Verify with `show ip route vrf BLUE` and a ping sourced from the BLUE interface.

2. **BGP leak lab**
   - Create two VRFs and assign interfaces.
   - Run VRF-aware BGP in each VRF.
   - Advertise only the required prefixes with prefix-lists and route-maps.
   - Confirm the route appears in the target VRF table, not just in BGP.

3. **MPLS VPN lab**
   - Create two customer VRFs with complementary route-targets.
   - Attach customer interfaces to each VRF.
   - Enable MP-BGP VPNv4 and extended communities.
   - Verify route import with `show ip bgp vpnv4 vrf <name>` and test reachability.

A simple verification sequence is:

```cisco
show vrf
show ip interface brief vrf all
show ip route vrf BLUE
show ip route vrf RED
show ip bgp vpnv4 vrf BLUE
ping vrf BLUE 10.20.20.10 source 10.10.10.1
traceroute vrf BLUE 10.20.20.10
```

**Troubleshooting checklist**

When a leak fails, check the problem in layers:

- **Interface binding**
  - Confirm the interface is in the correct VRF.
  - Confirm the IP address is on the VRF interface, not the global table.

- **Route presence**
  - Confirm the source VRF actually has the route.
  - Confirm the destination VRF imports or learns it.

- **Next-hop reachability**
  - Confirm the next hop is reachable inside the destination VRF.
  - Check CEF if the route exists but forwarding fails.

- **Policy**
  - Check prefix-lists, route-maps, and tags.
  - Confirm route-targets match exactly in MPLS VPN designs.

- **Protocol state**
  - Check BGP neighbors, address-family activation, and community handling.
  - Confirm `extended communities` are sent for VPN route-targets.

Useful commands:

```cisco
show vrf detail
show ip route vrf <name>
show ip cef vrf <name> <prefix>
show ip bgp vpnv4 all
show ip bgp vpnv4 vrf <name>
show bgp ipv4 unicast vrf <name>
show run | section vrf
show run | section router bgp
```

**Exam focus and practical guidance**

For ENCOR and ENARSI, focus on the following distinctions:

- **VRF-lite static leaking** is the simplest enterprise answer.
- **BGP-based leaking** is the policy-driven enterprise answer.
- **MPLS L3VPN route-targets** are the provider VPN answer.
- **Redistribution** is a supporting technique, not a magic leak.

Remember these rules:

- A route must be learned in the source VRF before it can be leaked.
- A route-target only matters in an MPLS VPN control plane.
- A BGP route in one VRF is not automatically visible in another VRF.
- Verification must include both the routing table and the control plane.

**Quick reference**

<dl>
<dt>VRF-lite</dt>
<dd>Enterprise VRF separation on a single router or campus edge, usually without MPLS.</dd>
<dt>Route leaking</dt>
<dd>Selective sharing of routes between VRFs while preserving isolation for other prefixes.</dd>
<dt>Route-target</dt>
<dd>An extended community used in MPLS VPNs to control which VRFs import or export routes.</dd>
<dt>MP-BGP</dt>
<dd>Multiprotocol BGP, used to carry VPNv4 and VPNv6 routes in MPLS L3VPN designs.</dd>
</dl>

**Practical takeaway**

If you need a small, deterministic leak, use static routes. If you need policy and scale, use BGP with explicit import/export behavior. If you are in an MPLS VPN environment, use route-targets and MP-BGP. In every case, verify the VRF table, the control plane, and the next-hop path before you assume the leak works.
