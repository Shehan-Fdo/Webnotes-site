## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Next-hop fundamentals**

A **next hop** is the immediate forwarding target a router uses to send traffic toward a destination. Cisco IOS and IOS XE decide whether a route is usable by checking both the route itself and whether the next hop can be resolved to an adjacency in the forwarding plane.

At exam level, you should separate three ideas:

- **Routing decision**: which route wins in the routing table.
- **Recursive resolution**: how Cisco finds the outgoing interface and/or connected next hop.
- **Forwarding adjacency**: how CEF (Cisco Express Forwarding) actually sends packets.

A route can look valid in the routing table but still not forward traffic if the next hop cannot be resolved into a usable adjacency.

**Core forwarding model**

Cisco typically resolves a route in this order:

1. The router selects the best route in the RIB (Routing Information Base).
2. If the route points to a next-hop IP, Cisco recursively resolves that IP through other routes.
3. The router resolves the final path to an outgoing interface and adjacency.
4. CEF installs the forwarding entry when resolution succeeds.

A simple flow looks like this:

```text
Destination prefix
  -> best route in RIB
    -> next-hop IP
      -> recursive lookup
        -> connected or directly resolvable path
          -> CEF adjacency
            -> packet forwarding
```

**Recursive resolution**

Recursive resolution is common with static routes and many dynamic routes. Cisco follows the next-hop chain until it reaches a directly connected network or another resolvable path.

Example:

```text
10.10.10.0/24 via 192.0.2.2
192.0.2.2 via 203.0.113.1
203.0.113.1 is directly connected on GigabitEthernet0/0
```

In this case, the router forwards traffic for `10.10.10.0/24` out `GigabitEthernet0/0` after resolving the chain.

Important exam points:

- Recursive resolution is implementation-dependent and not unlimited.
- Cisco may resolve multiple levels, but you should not assume arbitrary recursion depth.
- If the final next hop is not reachable, the route may remain in the RIB but not become usable in the FIB (Forwarding Information Base).

**Static route next-hop behavior**

Static routes are the clearest place to study next-hop handling because you control the exact syntax.

Common forms:

```bash
ip route 203.0.113.0 255.255.255.0 192.0.2.2
ip route 203.0.113.0 255.255.255.0 GigabitEthernet0/0 192.0.2.2
ip route 0.0.0.0 0.0.0.0 192.0.2.1
```

Key behaviors:

- A **next-hop-only** static route relies on recursive resolution.
- An **exit-interface-only** static route on a point-to-point link is often valid and efficient.
- An **exit-interface plus next-hop** static route is often preferred on multiaccess networks because it avoids unnecessary ARP for every destination in the prefix.

On a multiaccess Ethernet, an interface-only static route can cause Cisco to treat the prefix as directly reachable on that interface and attempt ARP resolution for destinations in the prefix. That is usually not recommended unless the prefix is truly on-link.

> Use interface-only static routes carefully on Ethernet. If the destination network is not actually on that segment, you can create excessive ARP and blackholing.

A practical pattern:

```bash
! Point-to-point link
ip route 198.51.100.0 255.255.255.0 Serial0/0/0

! Multiaccess link with explicit next hop
ip route 198.51.100.0 255.255.255.0 GigabitEthernet0/0 192.0.2.2
```

**Default route next-hop handling**

Default routes are a common exam scenario because they behave like any other route, but they match all unknown destinations.

IPv4:

```bash
ip route 0.0.0.0 0.0.0.0 192.0.2.1
ip route 0.0.0.0 0.0.0.0 GigabitEthernet0/0 192.0.2.1
```

IPv6:

```bash
ipv6 route ::/0 2001:db8:1::1
ipv6 route ::/0 GigabitEthernet0/0 2001:db8:1::1
```

Exam-relevant behavior:

- Cisco resolves the default route like any other route.
- If the next hop is recursive, the router must still find a connected path to that next hop.
- A default route with an unresolved next hop is not useful for forwarding, even if it appears in configuration.

**IPv6 next-hop specifics**

IPv6 next-hop handling is similar in concept to IPv4, but link-local addressing adds an important twist.

A link-local next hop must be paired with an outgoing interface because the address is only valid on that link:

```bash
ipv6 route 2001:db8:100::/64 GigabitEthernet0/0 fe80::1
```

Why this matters:

- `fe80::/10` addresses are link-local and not globally unique.
- The router needs the interface to know which link-local scope to use.
- Without the interface, the next hop is not usable because the address is only meaningful on a specific link.

Useful IPv6 troubleshooting commands:

```bash
show ipv6 route
show ipv6 cef
show ipv6 neighbors
```

Common IPv6 next-hop issues:

- The next hop is configured but Neighbor Discovery (ND) has not resolved the neighbor yet.
- The outgoing interface is wrong for a link-local next hop.
- The route exists, but the neighbor entry is incomplete or stale.

**CEF adjacency types**

CEF uses adjacency types to represent how packets are forwarded. These are important for troubleshooting because they reveal whether the router can actually send traffic.

Common adjacency types:

- **Connected**: the destination is directly reachable on the interface.
- **Receive**: traffic destined to the router itself.
- **Drop**: traffic is intentionally discarded, often for null routes or unreachable paths.
- **Punt**: traffic is sent to the CPU for process switching or special handling.
- **Glean**: the router knows the outgoing interface for a directly connected network but still needs ARP or ND for the destination host.

A glean adjacency is specific to a host on a directly connected network when the router must resolve the destination MAC address before forwarding.

Example symptom:

```bash
show ip cef 192.0.2.10 detail
```

You may see a glean-related entry when the router has the connected prefix but has not yet resolved the host address.

Verification examples:

```bash
show ip cef 203.0.113.0/24
show adjacency GigabitEthernet0/0 detail
show adjacency
```

> `show adjacency <interface> detail` is often the most useful form when you want to inspect a specific link and its adjacency state.

**RIB versus FIB**

The RIB is the control-plane routing table. The FIB is the data-plane forwarding table.

Practical distinction:

- The RIB decides which route is best.
- The FIB decides how packets are forwarded.
- A route may be selected by the RIB but still not produce a usable forwarding entry if the next hop cannot be resolved.

Typical troubleshooting pattern:

```bash
show ip route 203.0.113.0
show ip cef 203.0.113.0
show ip cef exact-route 10.1.1.1 203.0.113.10
```

What to look for:

- The route is present in `show ip route`.
- The CEF entry is missing, incomplete, or points to an unresolved adjacency.
- The next hop is not reachable through any connected path.

**OSPF next-hop behavior**

OSPF (Open Shortest Path First) next-hop selection depends on route type and topology. Do not oversimplify it to “the advertising router is always the next hop.”

General exam guidance:

- For many **intra-area** routes on broadcast multiaccess networks, the next hop is often the advertising router.
- For **inter-area** routes, the next hop is typically the ABR (Area Border Router) that provides the best path into the area.
- For **external** routes, the next hop may be the ASBR (Autonomous System Boundary Router) or an ABR depending on the topology and SPF result.

Useful verification:

```bash
show ip route ospf
show ip ospf neighbor
show ip ospf database
show ip cef <prefix>
```

Practical example:

```bash
router ospf 10
 network 10.1.1.0 0.0.0.255 area 0
 network 10.2.2.0 0.0.0.255 area 0
```

If a router learns `10.3.3.0/24` through OSPF, the next hop in the routing table depends on the SPF path to the advertising router, not just the fact that the route was advertised.

Troubleshooting clues:

- If the OSPF route is present but traffic fails, check whether the next hop is reachable.
- If the next hop is reachable but the route still does not forward, inspect the CEF entry and adjacency state.
- On multiaccess networks, verify DR/BDR behavior only when it affects the topology; the next hop is still determined by the SPF path.

**EIGRP next-hop behavior**

EIGRP (Enhanced Interior Gateway Routing Protocol) uses the topology table and successor selection, so next-hop handling is tied to the best path calculation rather than a simple “neighbor advertised it, so use that neighbor” rule.

Important points:

- The successor is the best path installed in the routing table.
- The next hop is usually the neighbor that provides the successor path.
- Feasible successors can exist as backups, but they do not automatically become the next hop unless selected.

On multiaccess networks, the advertising neighbor is often the next hop, but split horizon, summarization, and redistribution can change what you see.

Example configuration:

```bash
router eigrp 100
 network 10.1.1.0 0.0.0.255
 network 10.2.2.0 0.0.0.255
 no auto-summary
```

Verification:

```bash
show ip eigrp topology
show ip route eigrp
show ip cef <prefix>
```

IPv6 EIGRP adds an important detail:

- Link-local addresses are commonly used as next hops.
- You must verify the neighbor relationship and the outgoing interface.
- The interface matters because the link-local next hop is only valid on that link.

Example:

```bash
ipv6 router eigrp 100
 no shutdown
!
interface GigabitEthernet0/0
 ipv6 eigrp 100
```

Troubleshooting checklist:

- Confirm the EIGRP neighbor is up.
- Confirm the successor path is installed.
- Check whether summarization or split horizon is changing the advertised next hop.
- Verify that the outgoing interface can resolve the neighbor at Layer 2.

**RIP next-hop behavior**

RIP (Routing Information Protocol) is simpler than OSPF or EIGRP in next-hop handling. In general, the router uses the neighbor that advertised the route as the forwarding direction, and there is no separate next-hop attribute comparable to BGP.

Exam focus:

- RIP is distance-vector based.
- The route learned from a neighbor is forwarded toward that neighbor unless another route is preferred.
- RIP does not use a rich next-hop policy model like BGP.

Verification:

```bash
show ip route rip
show ip protocols
debug ip rip
```

Practical note:

- If a RIP route is present but traffic fails, check the neighbor relationship, split horizon, and whether the route is being advertised back out the same interface.
- On multiaccess networks, the forwarding path still depends on the resolved adjacency to the neighbor.

**BGP next-hop behavior**

BGP (Border Gateway Protocol) has the most exam-relevant next-hop rules, especially for iBGP (internal BGP), eBGP (external BGP), route reflectors, and confederations.

Core rules:

- In **eBGP**, the next hop is normally changed to the local eBGP neighbor by default.
- In **iBGP**, the next hop is generally preserved unless you change it with policy.
- `next-hop-self` forces the advertising router to become the next hop.
- `next-hop-unchanged` preserves the original next hop in scenarios where the default behavior would change it.

Basic examples:

```bash
router bgp 65001
 neighbor 192.0.2.2 remote-as 65002
 neighbor 192.0.2.2 next-hop-self
```

```bash
router bgp 65001
 neighbor 10.0.0.2 remote-as 65001
 neighbor 10.0.0.2 next-hop-self
```

Why this matters:

- In iBGP, a route can be learned but not usable if the next hop is not reachable inside the AS.
- Route reflectors often need `next-hop-self` or careful design so clients can reach the next hop.
- Confederations can alter propagation behavior, so you should verify the effective next hop rather than assume it.

Route reflector considerations:

- A route reflector may reflect a route without changing the next hop unless configured to do so.
- Clients must be able to reach the reflected next hop.
- If the next hop is external and not reachable in the iBGP domain, the route may be present but not installed for forwarding.

Confederation considerations:

- Confederation sub-AS behavior can resemble iBGP in some respects and eBGP in others.
- Always verify the actual next hop in the BGP table and the RIB.

Useful commands:

```bash
show ip bgp
show ip bgp <prefix>
show ip route <prefix>
show ip cef <prefix>
show bgp ipv4 unicast summary
```

A common troubleshooting pattern:

- The prefix appears in `show ip bgp`.
- The prefix does not appear in `show ip route`.
- The next hop is unreachable, the route loses best-path selection, or policy prevents installation.
- Fixing next-hop reachability often restores installation, but also check administrative distance, attributes, and route policy.

> A BGP prefix can be present in the BGP table and still not install in the RIB for reasons other than next-hop reachability, including best-path selection, policy, or administrative distance.

**BGP next-hop propagation scenarios**

Use this mental model:

| Scenario | Default next-hop behavior | Common fix |
|---|---|---|
| eBGP to eBGP | Next hop becomes the local eBGP neighbor | Usually none |
| eBGP to iBGP | Next hop is often preserved into iBGP | `next-hop-self` on the edge router |
| iBGP to iBGP | Next hop is preserved | Ensure reachability or use `next-hop-self` |
| Route reflector | Next hop may be preserved when reflected | `next-hop-self` on RR or edge |
| Confederation | Depends on sub-AS and policy | Verify actual advertised next hop |

Example route reflector design:

```bash
router bgp 65001
 neighbor 10.0.0.2 remote-as 65001
 neighbor 10.0.0.2 route-reflector-client
 neighbor 10.0.0.2 next-hop-self
```

**Default route and summary route next-hop behavior**

Default routes and summary routes are often used to simplify forwarding, but they still rely on next-hop resolution.

Examples:

```bash
ip route 0.0.0.0 0.0.0.0 192.0.2.1
ipv6 route ::/0 2001:db8:1::1
```

For summaries:

- The summary route may point to a specific next hop.
- If the summary next hop fails, traffic can blackhole unless you provide a backup path.
- In dynamic routing, summary advertisements can change the apparent next hop seen by downstream routers.

Practical design tip:

- Verify that the default route’s next hop is reachable from the router that installs it.
- Confirm that the upstream device actually forwards toward the intended exit path.

**Troubleshooting workflow**

Use a consistent workflow when a route exists but traffic fails.

1. Check the route in the RIB.
   ```bash
   show ip route <prefix>
   ```
2. Check the forwarding entry.
   ```bash
   show ip cef <prefix>
   ```
3. Check the adjacency.
   ```bash
   show adjacency <interface> detail
   ```
4. Verify the next hop is reachable.
   ```bash
   ping <next-hop-ip>
   traceroute <destination>
   ```
5. Check the protocol-specific table.
   ```bash
   show ip bgp <prefix>
   show ip ospf database
   show ip eigrp topology
   ```
6. Confirm Layer 2 resolution.
   ```bash
   show ip arp
   show ipv6 neighbors
   ```

Common symptoms and likely causes:

- **Route in RIB, no forwarding**: unresolved next hop, policy issue, or route not selected for installation.
- **CEF entry exists but traffic drops**: adjacency problem, ARP/ND failure, or wrong exit interface.
- **BGP route present but not installed**: next-hop unreachable, better route exists, or policy/administrative distance blocks installation.
- **IPv6 route present but no traffic**: wrong interface for link-local next hop, ND failure, or missing neighbor discovery.

**Platform behavior and limitations**

Cisco IOS and IOS XE are similar in concept, but exact output and edge behavior can vary by platform, feature set, and hardware forwarding architecture.

Keep these cautions in mind:

- Some commands show slightly different output depending on platform.
- Hardware offload can affect how quickly a route appears in the FIB.
- Recursive resolution behavior is not identical across all platforms and software trains.
- A route that looks valid in the control plane may still fail in the data plane if the adjacency is incomplete.

For exam purposes, focus on the principle:

- The control plane chooses the route.
- The forwarding plane needs a resolvable next hop and adjacency.
- If either part fails, traffic fails.

**Practical lab scenarios**

Scenario 1: Static route with unresolved next hop

```bash
ip route 203.0.113.0 255.255.255.0 192.0.2.2
```

Symptoms:

- `show ip route 203.0.113.0` shows the route.
- `show ip cef 203.0.113.0` shows incomplete resolution or no usable adjacency.
- Ping fails.

Fix:

- Make sure `192.0.2.2` is reachable through a connected interface.
- Verify the upstream interface and ARP entry.
- If needed, use an exit interface plus next hop on a multiaccess network.

Scenario 2: iBGP route not forwarding

```bash
show ip bgp 10.10.10.0
show ip route 10.10.10.0
```

Symptoms:

- The prefix is in BGP.
- The prefix is not in the RIB or not forwarding.
- The next hop is an external address not reachable inside the AS.

Fix:

- Configure `next-hop-self` on the iBGP edge or route reflector.
- Ensure the next hop is reachable through an IGP or static route.

Scenario 3: IPv6 static route with link-local next hop

```bash
ipv6 route 2001:db8:100::/64 GigabitEthernet0/0 fe80::1
```

Symptoms:

- The route fails if the interface is omitted.
- Neighbor discovery does not resolve the next hop.

Fix:

- Include the outgoing interface.
- Verify the neighbor is present with `show ipv6 neighbors`.

**Quick comparison**

| Topic | Key next-hop rule | Common exam trap |
|---|---|---|
| Static routes | Recursive resolution is common | Assuming interface-only static routes are always safe |
| OSPF | Next hop depends on route type and SPF path | Assuming the advertising router is always the next hop |
| EIGRP | Next hop follows successor selection and topology logic | Treating it like a simple neighbor-forwarding protocol |
| RIP | Simpler neighbor-based forwarding | Inventing a special next-hop mechanism |
| BGP | iBGP preserves next hop by default; eBGP usually changes it | Confusing `next-hop-self` with multihop behavior |
| IPv6 | Link-local next hops require an interface | Omitting the interface for `fe80::` routes |

**Exam takeaways**

- Always verify both the route and the next hop.
- Distinguish between control-plane presence and data-plane usability.
- Remember that BGP next-hop behavior is policy-sensitive and topology-sensitive.
- Treat IPv6 link-local next hops as interface-scoped.
- Use CEF and adjacency output to confirm whether forwarding can actually happen.
