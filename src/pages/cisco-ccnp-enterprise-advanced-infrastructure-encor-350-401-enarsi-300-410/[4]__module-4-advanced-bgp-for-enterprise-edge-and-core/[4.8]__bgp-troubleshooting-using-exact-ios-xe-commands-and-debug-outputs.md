## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**BGP troubleshooting workflow**

Border Gateway Protocol (<abbr title="Border Gateway Protocol">BGP</abbr>) issues usually fall into one of four buckets: session establishment, capability negotiation, policy filtering, or route installation/selection. On IOS XE, you troubleshoot fastest when you verify the transport first, then the neighbor state, then the BGP table, and finally the Routing Information Base (<abbr title="Routing Information Base">RIB</abbr>) and forwarding path.

A practical order is:

1. Confirm IP reachability and TCP/179 connectivity.
2. Check neighbor state and negotiated capabilities.
3. Verify that routes are learned, accepted, and best.
4. Confirm the route wins the RIB and resolves next hop correctly.
5. Use debug only after you narrow the failure domain.

> Use `debug ip bgp` carefully on production routers. Prefer conditional debugging and short capture windows.

**Core verification commands**

These commands are the backbone of IOS XE BGP troubleshooting. Use them before you enable debug.

```bash
show ip bgp summary
show ip bgp
show ip bgp neighbors
show ip bgp neighbors 192.0.2.2 advertised-routes
show ip bgp neighbors 192.0.2.2 routes
show bgp all neighbors
show bgp ipv4 unicast summary
show bgp ipv4 unicast neighbors 192.0.2.2
show bgp ipv4 unicast neighbors 192.0.2.2 advertised-routes
show bgp ipv4 unicast neighbors 192.0.2.2 routes
show ip route bgp
show ip route 203.0.113.0
show ip cef 203.0.113.0
show tcp brief | include :179
```

Key interpretation points:

- **`show ip bgp summary`** tells you whether the session is established and how many prefixes are received from each neighbor.
- **`State/PfxRcd`** shows a prefix count only when the session is Established.
- In non-established states, `State/PfxRcd` shows the state name or a reset reason, not a prefix count.
- **`show bgp all neighbors`** is useful when you need to inspect all address families and VRFs in one view.
- **`show ip bgp neighbors <peer> routes`** shows routes received from the neighbor when the router stores inbound routes for inspection.
- **`show ip bgp neighbors <peer> advertised-routes`** shows what you send to the neighbor.
- **`show ip route bgp`** confirms whether BGP-learned routes actually enter the RIB.
- **`show ip cef`** helps confirm forwarding and recursive next-hop resolution.

**Neighbor state troubleshooting**

BGP neighbor states tell you where the session is failing. On IOS XE, the common states are Idle, Connect, Active, OpenSent, OpenConfirm, and Established.

<dl>
<dt><strong>Idle</strong></dt>
<dd>The router is not actively trying to establish the session or has backed off after a failure. Common causes include missing neighbor configuration, shutdown, authentication mismatch, policy issues that prevent retry, or a manually cleared session.</dd>
<dt><strong>Connect</strong></dt>
<dd>The router has initiated TCP connection attempts and is waiting for the transport to succeed. This often points to reachability, ACL, firewall, source address, or TCP/179 problems.</dd>
<dt><strong>Active</strong></dt>
<dd>The router is retrying TCP connection attempts after a failure. This usually indicates the peer is unreachable, TCP is blocked, the remote BGP process is down, or the source/destination address is wrong.</dd>
<dt><strong>OpenSent</strong></dt>
<dd>The TCP session is up and the router has sent a BGP OPEN message. Failures here often involve capability mismatch, AS number mismatch, authentication, or hold-time negotiation problems.</dd>
<dt><strong>OpenConfirm</strong></dt>
<dd>The router has exchanged OPEN messages and is waiting for KEEPALIVE. Problems here often involve timer issues, policy resets, or a peer that accepts TCP but does not complete BGP negotiation.</dd>
<dt><strong>Established</strong></dt>
<dd>The session is up and routes can be exchanged. If routes are missing, focus on policy, AFI/SAFI activation, next-hop reachability, and best-path selection.</dd>
</dl>

Typical IOS XE summary output:

```bash
R1# show ip bgp summary
BGP router identifier 10.0.0.1, local AS number 65001
BGP table version is 12, main routing table version 12

Neighbor        V    AS MsgRcvd MsgSent   TblVer  InQ OutQ Up/Down  State/PfxRcd
192.0.2.2       4 65002      18      19       12    0    0 00:12:31        24
198.51.100.2    4 65003       0       0        0    0    0 00:00:14        Active
203.0.113.2     4 65004       0       0        0    0    0 00:00:09        Idle
```

How to read it:

- `192.0.2.2` is Established because `State/PfxRcd` is numeric.
- `198.51.100.2` is stuck in Active, so the transport or remote process is not completing.
- `203.0.113.2` is Idle, so the session is not progressing to TCP/BGP negotiation.

**Transport and TCP checks**

Before blaming BGP policy, verify the underlying transport. BGP uses TCP port 179, so a session can fail even when ping works.

Useful checks:

```bash
ping 192.0.2.2 source 10.0.0.1
traceroute 192.0.2.2 source 10.0.0.1
show ip interface brief
show ip route 192.0.2.2
show access-lists
show tcp brief | include :179
show ip sockets | include 179
```

Common transport problems:

- The update-source address is wrong, so the peer sees a different source IP than expected.
- An ACL or firewall blocks TCP/179.
- The peer address is reachable by ping but not by the exact source address used by BGP.
- A loopback is used without a route to the loopback.
- A VRF is missing the correct route or interface binding.

Example of a source mismatch:

```bash
router bgp 65001
 neighbor 192.0.2.2 remote-as 65002
 neighbor 192.0.2.2 update-source Loopback0
```

If the peer expects the session from `10.0.0.1` but the router sources from another interface, the TCP handshake may never complete.

**Debug outputs for session establishment**

Use debug only after you have a focused hypothesis. The most useful debug families are BGP events, BGP updates, and TCP.

```bash
debug ip bgp events
debug ip bgp updates
debug ip tcp transactions
debug ip tcp packet
```

Example debug patterns for a successful session:

```bash
*Mar  1 10:15:21.123: BGP: 192.0.2.2 active open
*Mar  1 10:15:21.124: BGP: 192.0.2.2 sending OPEN, version 4, my AS 65001, holdtime 180, id 10.0.0.1
*Mar  1 10:15:21.130: BGP: 192.0.2.2 rcv OPEN, version 4, remote AS 65002, holdtime 180, id 10.0.0.2
*Mar  1 10:15:21.131: BGP: 192.0.2.2 went from OpenSent to OpenConfirm
*Mar  1 10:15:21.140: BGP: 192.0.2.2 went from OpenConfirm to Established
*Mar  1 10:15:21.141: BGP: 192.0.2.2 received KEEPALIVE
```

Example debug patterns for a TCP failure:

```bash
*Mar  1 10:16:02.001: BGP: 192.0.2.2 active open failed
*Mar  1 10:16:02.002: BGP: 192.0.2.2 connection refused
*Mar  1 10:16:02.003: BGP: 192.0.2.2 went from Active to Idle
```

Example debug patterns for authentication failure:

```bash
*Mar  1 10:17:10.221: BGP: 192.0.2.2 open failed: authentication failure
*Mar  1 10:17:10.222: BGP: 192.0.2.2 went from OpenSent to Idle
```

Example debug patterns for capability mismatch:

```bash
*Mar  1 10:18:44.501: BGP: 192.0.2.2 rcv OPEN, version 4, remote AS 65002
*Mar  1 10:18:44.502: BGP: 192.0.2.2 unsupported capability 64
*Mar  1 10:18:44.503: BGP: 192.0.2.2 notification sent (unsupported capability)
*Mar  1 10:18:44.504: BGP: 192.0.2.2 went from OpenSent to Idle
```

**Capability negotiation problems**

BGP peers must agree on capabilities during OPEN. If they do not, the session may reset even though TCP is up.

Common capabilities to verify:

- **Route refresh** support, which allows a peer to resend routes after policy changes.
- **4-byte AS** support, which is required when either side uses an AS number above 65535.
- **Graceful restart** support, which helps preserve forwarding during restarts.
- **AFI/SAFI** support, which determines whether IPv4 unicast, IPv6 unicast, VPNv4, or other families are negotiated.

Useful commands:

```bash
show ip bgp neighbors 192.0.2.2
show bgp ipv4 unicast neighbors 192.0.2.2
show bgp all neighbors 192.0.2.2
```

What to look for:

- Negotiated capabilities listed in the neighbor detail.
- Whether the peer supports the address family you configured.
- Whether the session resets immediately after OPEN exchange.

Typical mismatch symptoms:

- The session reaches OpenSent and then resets.
- The neighbor detail shows the peer does not advertise the expected AFI/SAFI.
- A 4-byte AS mismatch appears when one side uses a 32-bit AS and the other side is misconfigured or too old.
- Graceful restart differences usually do not prevent adjacency, but they can affect convergence behavior and stale route handling.

**Policy and route filtering**

Once the session is Established, missing routes are often caused by policy. On IOS XE, policy can be applied inbound, outbound, or through route-maps, prefix lists, AS-path lists, communities, and distribute lists.

Useful verification commands:

```bash
show ip bgp neighbors 192.0.2.2 received-routes
show ip bgp neighbors 192.0.2.2 advertised-routes
show ip bgp neighbors 192.0.2.2 routes
show route-map
show ip prefix-list
show ip as-path access-list
show ip community-list
show bgp ipv4 unicast neighbors 192.0.2.2 advertised-routes
```

Important notes:

- `received-routes` is only available when the router stores inbound routes for inspection, such as with inbound soft reconfiguration.
- Route refresh alone does not guarantee `received-routes` output.
- `routes` shows routes accepted from the neighbor in the current BGP table view.
- `advertised-routes` helps confirm outbound policy and what the neighbor should receive.

Example route-map issue:

```bash
route-map RM-IN deny 10
 match ip address prefix-list PL-BLOCK
route-map RM-IN permit 20
```

If `PL-BLOCK` matches the route, the route is denied before it enters BGP. Verify with:

```bash
show ip bgp neighbors 192.0.2.2 routes
show ip bgp 203.0.113.0
show route-map RM-IN
show ip prefix-list PL-BLOCK
```

**Best-path selection and why a route is not installed**

A route can appear in BGP but still not be installed in the RIB. That usually means another route won the administrative distance or the BGP path lost best-path selection.

Check:

```bash
show ip bgp 203.0.113.0
show ip route 203.0.113.0
show ip route bgp
show ip cef 203.0.113.0
```

Example BGP table output:

```bash
R1# show ip bgp 203.0.113.0
BGP routing table entry for 203.0.113.0/24, version 12
Paths: (2 available, best #2, table default)
  Not advertised to any peer
  65002
    192.0.2.2 from 192.0.2.2 (10.0.0.2)
      Origin IGP, metric 0, localpref 100, valid, external
  65003
    198.51.100.2 from 198.51.100.2 (10.0.0.3)
      Origin IGP, metric 0, localpref 200, valid, internal, best
```

Interpretation:

- The second path is best because it has higher local preference.
- If `show ip route 203.0.113.0` does not show BGP, another protocol may have a lower administrative distance and is winning the RIB.
- If the route is best in BGP but missing from the RIB, check for redistribution, static routes, or policy that suppresses installation.

Administrative distance reminders:

- eBGP default administrative distance is 20.
- iBGP default administrative distance is 200.
- Internal routes from OSPF, EIGRP, or static routes may win over BGP depending on the values configured.

**Next-hop reachability and recursive resolution**

A very common BGP problem is that the route is learned but not usable because the next hop is unreachable. This is especially important in iBGP and route-reflector designs.

Check:

```bash
show ip bgp 203.0.113.0
show ip route <next-hop>
show ip cef 203.0.113.0 detail
show ip cef exact-route <src> <dst>
```

What to verify:

- The BGP next hop exists in the routing table.
- Recursive resolution succeeds all the way to an outgoing interface.
- The next hop is not pointing to an unreachable loopback or remote address.
- The next hop is preserved or rewritten as intended.

Common design cases:

- **eBGP** usually rewrites next hop to the neighbor address.
- **iBGP** preserves next hop by default.
- A route reflector reflects routes without changing next hop unless configured to do so.
- `next-hop-self` is useful when the receiving router cannot reach the original next hop.

Example configuration:

```bash
router bgp 65001
 neighbor 10.0.0.2 remote-as 65001
 neighbor 10.0.0.2 next-hop-self
```

Use `next-hop-self` when:

- An iBGP peer cannot reach the original eBGP next hop.
- A route reflector client needs the reflector to become the reachable next hop.
- You want to simplify recursive resolution in a hub-and-spoke design.

Do not assume it is always required:

- If the original next hop is already reachable, `next-hop-self` may be unnecessary.
- In some topologies, preserving the original next hop is desirable for optimal forwarding.

Example of recursive failure symptoms:

```bash
R1# show ip bgp 203.0.113.0
*> 203.0.113.0/24 192.0.2.2 0 65002 i
R1# show ip route 192.0.2.2
% Network not in table
```

The BGP route exists, but the next hop is not reachable, so forwarding fails.

**Route reflector troubleshooting**

Route reflectors can hide problems because the session is up but reflected routes do not appear where expected. Focus on cluster design, client relationships, next-hop reachability, and policy.

Check:

```bash
show ip bgp neighbors
show ip bgp neighbors 10.0.0.2
show ip bgp 203.0.113.0
show ip bgp regexp _65002_
```

Useful checks:

- Confirm the neighbor is configured as a route-reflector client.
- Confirm the reflected route is not being filtered by policy.
- Confirm the receiving client can reach the next hop.
- Confirm cluster IDs are not causing unexpected reflection loops or suppression.

Important nuance:

- Duplicate cluster IDs can cause reflection issues, but they are not the only cause.
- `next-hop-self` may help in some designs, but it is not universally required for route reflection.
- If a route is reflected but not installed, the problem may still be next-hop reachability or administrative distance.

**Legacy synchronization behavior**

BGP synchronization is a legacy concept that is mostly disabled in modern enterprise designs, but it still appears in troubleshooting discussions and exam questions.

Conceptually:

- Synchronization required an iBGP-learned route to be present in an IGP before it could be advertised to eBGP peers.
- This was intended to prevent blackholing in transit networks that did not use full-mesh iBGP.
- In modern designs with route reflectors and full iBGP reachability, synchronization is usually not used.

Check the configuration if routes are unexpectedly suppressed:

```bash
show running-config | section router bgp
```

If you see synchronization-related behavior in a legacy environment, verify whether the route exists in the IGP and whether the design still depends on that old rule.

**BGP dampening and route flap suppression**

If a route appears and disappears repeatedly, dampening or flap suppression may be involved. This is less common in modern enterprise deployments, but it remains relevant for troubleshooting unstable prefixes.

Symptoms:

- A prefix is learned, then suppressed.
- The route is present in BGP but not selected for forwarding.
- The route flaps after interface or neighbor instability.

Useful checks:

```bash
show ip bgp dampening
show ip bgp 203.0.113.0
show ip bgp flap-statistics
show ip bgp neighbors 192.0.2.2
```

What to look for:

- Suppressed routes.
- Penalty values.
- Reuse and suppress thresholds.
- Repeated neighbor resets causing route churn.

**VRF-aware BGP troubleshooting**

When BGP runs in a <abbr title="Virtual Routing and Forwarding">VRF</abbr>, you must verify the correct routing table, interface binding, and address family context.

Useful commands:

```bash
show bgp vpnv4 unicast all summary
show bgp ipv4 unicast vrf CUSTOMER1 summary
show bgp ipv4 unicast vrf CUSTOMER1 neighbors
show bgp ipv4 unicast vrf CUSTOMER1 neighbors 10.1.1.2 advertised-routes
show bgp ipv4 unicast vrf CUSTOMER1 neighbors 10.1.1.2 routes
show ip route vrf CUSTOMER1
show ip bgp vpnv4 all
show ip bgp vpnv4 all neighbors
```

Common VRF issues:

- The neighbor is configured in the wrong VRF.
- The source interface is in a different VRF.
- The route to the neighbor exists in the global table, not the VRF table.
- The address family is activated in the wrong context.
- The route-target import/export policy is missing, so routes never appear in the target VRF.

Example VRF configuration:

```bash
router bgp 65001
 address-family ipv4 vrf CUSTOMER1
  neighbor 10.1.1.2 remote-as 65002
  neighbor 10.1.1.2 activate
  neighbor 10.1.1.2 route-map RM-CUST1-IN in
```

**Worked example 1: Session stuck in Active**

Topology:

- R1 `10.0.0.1` in AS 65001
- R2 `10.0.0.2` in AS 65002
- eBGP over loopbacks

Configuration on R1:

```bash
router bgp 65001
 neighbor 10.0.0.2 remote-as 65002
 neighbor 10.0.0.2 update-source Loopback0
```

Problem: The session stays in Active.

Step-by-step diagnosis:

1. Check reachability to the peer loopback.
   ```bash
   R1# ping 10.0.0.2 source 10.0.0.1
   ```
2. Check the route to the peer.
   ```bash
   R1# show ip route 10.0.0.2
   ```
3. Check TCP/179.
   ```bash
   R1# show tcp brief | include :179
   ```
4. Check the neighbor state.
   ```bash
   R1# show ip bgp summary
   ```
5. Enable targeted debug.
   ```bash
   debug ip bgp events
   debug ip tcp transactions
   ```

Sample debug:

```bash
*Mar  1 11:02:10.101: BGP: 10.0.0.2 active open
*Mar  1 11:02:10.102: BGP: 10.0.0.2 active open failed
*Mar  1 11:02:10.103: BGP: 10.0.0.2 went from Active to Idle
```

Diagnosis:

- The router cannot complete the TCP handshake.
- The most likely causes are missing route to the loopback, ACL blocking TCP/179, or wrong source address.

Fix:

- Add the correct route to the peer loopback.
- Verify the peer allows TCP/179 from the source loopback.
- Confirm both sides use the same update-source and neighbor address.

**Worked example 2: Session Established, but no routes installed**

Topology:

- iBGP between R1 and R2
- R2 advertises `203.0.113.0/24`
- R1 receives the route in BGP but not in the RIB

Commands:

```bash
R1# show ip bgp summary
R1# show ip bgp 203.0.113.0
R1# show ip route 203.0.113.0
R1# show ip route 10.0.0.2
R1# show ip cef 203.0.113.0 detail
```

Sample output:

```bash
R1# show ip bgp 203.0.113.0
BGP routing table entry for 203.0.113.0/24, version 18
Paths: (1 available, best #1, table default)
  65002
    10.0.0.2 from 10.0.0.2 (10.0.0.2)
      Origin IGP, localpref 100, valid, internal
```

But:

```bash
R1# show ip route 203.0.113.0
% Network not in table
R1# show ip route 10.0.0.2
% Network not in table
```

Diagnosis:

- The BGP path is valid, but the next hop is unreachable.
- Because iBGP preserves next hop by default, R1 cannot resolve the route.

Fix options:

- Advertise the next hop in the IGP.
- Configure `next-hop-self` on the advertising router if the design allows it.

**Worked example 3: Policy denies outbound advertisement**

Commands:

```bash
R1# show ip bgp neighbors 192.0.2.2 advertised-routes
R1# show route-map
R1# show ip prefix-list
R1# show ip bgp 198.51.100.0
```

Sample route-map:

```bash
route-map OUT deny 10
 match ip address prefix-list BLOCKED
route-map OUT permit 20
```

If `198.51.100.0/24` is missing from `advertised-routes`, the route-map likely denies it.

Fix:

- Adjust the prefix list or route-map sequence.
- Re-evaluate with route refresh or clear the neighbor carefully.

**Useful clear and refresh commands**

Use clears sparingly and only after you understand the impact.

```bash
clear ip bgp 192.0.2.2 soft in
clear ip bgp 192.0.2.2 soft out
clear ip bgp 192.0.2.2
clear bgp ipv4 unicast 192.0.2.2 soft in
clear bgp ipv4 unicast 192.0.2.2 soft out
clear bgp ipv4 unicast vrf CUSTOMER1 10.1.1.2 soft in
```

Guidance:

- Use soft clears when you want to reapply policy without dropping the session.
- Use hard clears only when the session is broken or capability negotiation changed.
- In VRF contexts, include the VRF name and address family.

**Quick decision tree**

1. If the neighbor is not Established, check transport, source address, ACLs, and authentication.
2. If the neighbor is Established but no routes appear, check AF activation, policy, and capability negotiation.
3. If the route appears in BGP but not in the RIB, check next-hop reachability and administrative distance.
4. If the route is in the RIB but traffic fails, check CEF, recursive resolution, and forwarding adjacency.
5. If routes flap, check dampening, interface instability, and neighbor resets.

**Troubleshooting mindset**

BGP troubleshooting is not just about whether the session is up. You must determine whether the route is:

- learned,
- accepted,
- selected as best,
- installed in the RIB,
- and reachable in the forwarding plane.

That sequence prevents false conclusions. A neighbor can be Established while policy blocks routes, a route can be best in BGP but lose to another protocol in the RIB, and a route can be installed but still fail forwarding because the next hop is unresolved.
