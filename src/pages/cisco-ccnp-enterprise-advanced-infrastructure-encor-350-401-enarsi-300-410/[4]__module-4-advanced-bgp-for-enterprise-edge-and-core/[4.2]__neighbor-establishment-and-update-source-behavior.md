## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Neighbor establishment overview**

Neighbor formation is the process by which routing devices discover each other, verify compatibility, and exchange enough control information to build an adjacency or session. The exact mechanics vary by protocol, but the same troubleshooting pattern appears repeatedly: transport reachability must work, protocol parameters must match, and the chosen source address must be reachable in both directions.

For exam purposes, separate three ideas:

- **Neighbor discovery** is the initial hello exchange.
- **Adjacency or session establishment** is the point where the protocol considers the peer usable.
- **Source selection** determines which local address the router uses to originate packets, especially important for loopback-based peering.

A useful mental model is:

```text
Configured neighbor address  --->  destination for control packets
Local source address         --->  source seen by the peer
Return path                  --->  must reach the chosen source
```

If the destination is correct but the source is wrong or unreachable, the protocol may fail even though basic IP connectivity exists.

**Protocol comparison at a glance**

| Protocol | Neighboring mechanism | Source behavior | Common exam focus |
|---|---|---|---|
| OSPF | Hellos on an interface, then state progression | Usually uses the interface IP; loopbacks are typically advertised as host routes and are not used for adjacency formation | Network type, passive interfaces, area/authentication mismatch |
| EIGRP | Multicast Hellos and reliable updates | Uses the interface IP unless you build the adjacency over a different interface or tunnel | AS mismatch, K-values, stubs, split horizon, timers |
| IS-IS | CLNS-based hellos on the circuit | Runs directly over Layer 2; source is tied to the interface/system identity rather than IP transport | Level mismatch, circuit type, adjacency state, encapsulation |
| BGP | TCP session to port 179 | Can source from a loopback with `update-source` | `update-source`, TTL/multihop, reachability to the source, neighbor state |

**OSPF neighbor establishment**

OSPF forms adjacencies by sending Hello packets on an interface. The router ID, area ID, authentication, timers, network type, and neighbor eligibility all influence whether the adjacency reaches Full.

The neighbor state progression is commonly:

- `Down`
- `Init`
- `2-Way`
- `ExStart`
- `Exchange`
- `Loading`
- `Full`

The exact path depends on the link type. On broadcast and NBMA networks, non-DR/BDR neighbors often remain in `2-Way` and that is normal. On point-to-point links, you expect a Full adjacency if the parameters match.

Key OSPF points:

- **Router ID** must be unique.
- **Area ID** must match on the link.
- **Authentication** must match if enabled.
- **Timers** should match on most shared links.
- **Network type** affects DR/BDR election and adjacency behavior.
- **Passive interfaces** send no Hellos, so they do not form neighbors.

Loopback interfaces deserve special attention:

- You can enable OSPF on a loopback.
- On Cisco IOS, a loopback is typically advertised as a host route.
- It is not used for normal neighbor adjacency formation because it behaves like a passive-style interface for adjacency purposes.

Practical OSPF examples:

```cisco
router ospf 10
 router-id 1.1.1.1
 network 10.0.12.0 0.0.0.3 area 0
 passive-interface default
 no passive-interface GigabitEthernet0/0
```

```cisco
interface GigabitEthernet0/0
 ip address 10.0.12.1 255.255.255.252
 ip ospf 10 area 0
 ip ospf network point-to-point
```

Use `ip ospf network point-to-point` when you want to avoid DR/BDR behavior on a link that should behave like a simple point-to-point adjacency. This is common on Ethernet links in lab and exam scenarios when you want deterministic neighbor formation.

Useful verification:

```cisco
show ip ospf neighbor
show ip ospf interface g0/0
show ip ospf interface brief
show ip protocols
```

Troubleshooting cues:

- `Init` usually means you receive Hellos but your router ID is not yet seen back in the neighbor’s Hello.
- `2-Way` is normal on broadcast/NBMA links for non-DR/BDR neighbors.
- Stuck in `ExStart` or `Exchange` often points to MTU mismatch or database exchange issues.
- No neighbor at all often points to passive interface, area mismatch, authentication mismatch, or a missing Hello due to network type or ACL issues.

> On point-to-point links, `2-Way` is not the expected end state; you normally want Full adjacency.

**EIGRP neighbor establishment**

EIGRP uses Hello packets to discover neighbors and then builds a reliable adjacency for route exchange. In modern Cisco deployments, you may see both classic mode and named mode, but the neighbor logic is the same: the peers must agree on the core parameters and must be able to exchange Hellos before the hold timer expires.

Important EIGRP concepts:

- **AS number** must match.
- **K-values** must match.
- **Authentication** must match if configured.
- **Hello/hold timers** do not need to be identical, but Hellos must arrive before the hold timer expires.
- **Stub** settings affect query behavior and can influence troubleshooting in partial-mesh topologies.
- **Split horizon** can prevent route advertisement back out the same interface, which matters in hub-and-spoke designs.

EIGRP does not use OSPF-style neighbor states such as `Init` or `2-Way`. Instead, you typically verify whether the neighbor appears in the table and whether the adjacency is stable.

Classic mode example:

```cisco
router eigrp 100
 network 10.0.0.0 0.0.0.255
 no auto-summary
```

Named mode example:

```cisco
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  network 10.0.0.0 0.0.0.255
  af-interface default
   passive-interface
  exit-af-interface
  af-interface GigabitEthernet0/0
   no passive-interface
  exit-af-interface
 exit-address-family
```

Neighbor verification:

```cisco
show ip eigrp neighbors
show ip eigrp interfaces
show ip eigrp topology
show ip protocols
```

Practical troubleshooting notes:

- If you see no neighbor, check AS number, K-values, authentication, and whether the interface is passive.
- If routes are missing in a hub-and-spoke design, check stub configuration and split horizon.
- If Hellos are seen but adjacency is unstable, inspect timers, packet loss, and interface errors.

A common exam trap is assuming EIGRP behaves like OSPF state-wise. It does not use OSPF neighbor states, so do not map `Init` or `2-Way` to EIGRP.

**IS-IS neighbor establishment**

IS-IS runs directly over Layer 2 using CLNS packets, not over IP transport in the same way as OSPF or BGP. It forms adjacencies on circuits and uses Hello PDUs to discover and maintain neighbors.

Core IS-IS terms:

- **Circuit**: the interface participating in IS-IS.
- **Level 1**: intra-area routing.
- **Level 2**: inter-area/backbone-style routing.
- **Level 1-2**: a router capable of both levels.
- **Adjacency**: the neighbor relationship formed on the circuit.

IS-IS adjacency depends on:

- Matching **circuit type** expectations
- Compatible **encapsulation**
- Correct **level** configuration
- Matching **authentication** if used
- Proper **MTU** and link health

Unlike OSPF, IS-IS does not use an OSPF-style network type concept. For exam purposes, focus on circuit type and interface participation instead.

Typical configuration:

```cisco
router isis CORE
 net 49.0001.0000.0000.0001.00
 is-type level-2-only
```

```cisco
interface GigabitEthernet0/0
 ip router isis CORE
 isis network point-to-point
 isis circuit-type level-2-only
```

Verification commands:

```cisco
show isis neighbors
show isis interface
show isis database
show clns neighbors
```

Adjacency state ideas to remember:

- `Init`-like symptoms usually mean the neighbor is heard but the relationship is not fully bidirectional or not yet accepted at the expected level.
- Level mismatch can prevent the adjacency from reaching the desired state.
- A Level-1 router and a Level-2-only router may not form the adjacency you expect unless the circuit and level settings align.

Practical exam guidance:

- Check whether the interface is enabled for IS-IS.
- Check whether the router is Level-1, Level-2, or Level-1-2.
- Check whether the neighbor is on the same circuit type and whether the link is point-to-point or broadcast.
- Check the IS-IS adjacency and database output before assuming a transport problem.

**BGP neighbor establishment and update-source**

BGP uses TCP port 179 for its session. The neighbor statement identifies the remote peer address, while `update-source` controls the local source address used for the TCP session.

This distinction is critical:

- `neighbor X.X.X.X remote-as Y` tells BGP who the peer is.
- `neighbor X.X.X.X update-source Z` tells BGP which local interface or address to use as the source.
- The remote peer must be able to route back to that source address.
- The TCP session is established to the configured neighbor address, not to whatever source address the router happens to choose by default.

Directly connected eBGP often works without `update-source` because the outgoing interface address is suitable and the TTL of 1 is enough for a single hop. Loopback-based peering changes the rules.

For **iBGP**, loopback peering is common because it provides stability and survives interface failures. Since iBGP peers are usually within the same AS, you typically:

- Configure the neighbor using the loopback address.
- Use `update-source loopback0` on both sides.
- Ensure IGP or static routing can reach the loopback addresses.

For **eBGP**, peering to a loopback requires `update-source` and also `ebgp-multihop` because the default TTL of 1 is not enough when the peer is not directly connected. The multihop value must be greater than the actual hop count.

Example: iBGP over loopbacks

```cisco
router bgp 65001
 neighbor 10.255.255.2 remote-as 65001
 neighbor 10.255.255.2 update-source Loopback0
```

```cisco
interface Loopback0
 ip address 10.255.255.1 255.255.255.255
```

Example: eBGP over loopbacks

```cisco
router bgp 65001
 neighbor 203.0.113.2 remote-as 65002
 neighbor 203.0.113.2 update-source Loopback0
 neighbor 203.0.113.2 ebgp-multihop 2
```

```cisco
ip route 203.0.113.2 255.255.255.255 192.0.2.2
```

Why this matters:

- Without `update-source`, the session may source from the wrong interface.
- Without a return route to the source, the peer may receive the SYN but cannot complete the TCP handshake.
- Without `ebgp-multihop` for loopback-based eBGP, the TTL may expire before the packet reaches the peer.

A simple packet-flow view:

```text
R1 Loopback0 10.255.255.1
   |
   | TCP SYN to 10.255.255.2:179
   v
R2 Loopback0 10.255.255.2

Source address on SYN:
- default: often the outgoing interface address
- with update-source: the specified interface/address
```

BGP neighbor states to know:

- `Idle`
- `Connect`
- `Active`
- `OpenSent`
- `OpenConfirm`
- `Established`

State meanings:

- **Idle**: BGP is not actively trying, or it is waiting for a retry.
- **Connect**: TCP connection attempt is in progress.
- **Active**: BGP is retrying after a failed attempt.
- **OpenSent**: TCP is up and BGP Open has been sent.
- **OpenConfirm**: Keepalive exchange is underway.
- **Established**: The session is up and route exchange can begin.

Common BGP troubleshooting causes:

- Wrong neighbor IP
- Wrong remote AS
- Missing route to the neighbor address
- Missing route back to the chosen source address
- Incorrect `update-source`
- Missing `ebgp-multihop` for loopback-based eBGP
- ACL or firewall blocking TCP/179
- TTL expiration on multihop eBGP

Useful verification:

```cisco
show ip bgp summary
show ip bgp neighbors
show tcp brief
show ip route <neighbor-ip>
```

> `show tcp brief` syntax can vary by platform and software release, so use the local IOS/IOS XE help if the exact form differs.

**Troubleshooting patterns by protocol**

| Symptom | Likely cause | Protocols |
|---|---|---|
| No neighbor at all | Interface not participating, passive interface, wrong AS/area/level, authentication mismatch, ACL blocking control traffic | OSPF, EIGRP, IS-IS, BGP |
| Neighbor stuck in `Init` | One-way Hello reception or router ID not seen back | OSPF |
| Neighbor stuck in `2-Way` | Normal on broadcast/NBMA for non-DR/BDR neighbors; on point-to-point, adjacency should continue to Full | OSPF |
| Neighbor stuck in `ExStart` or `Exchange` | MTU mismatch or database exchange problem | OSPF |
| Neighbor missing from table | AS mismatch, K-value mismatch, passive interface, stub/topology issue, authentication mismatch | EIGRP |
| Adjacency not forming at expected level | Level mismatch, circuit type mismatch, encapsulation issue | IS-IS |
| BGP stuck in `Idle` or `Active` | Wrong neighbor, no route to peer, TCP/179 blocked, source address problem | BGP |
| BGP stuck in `OpenSent` or `OpenConfirm` | ASN mismatch, capability mismatch, authentication issue, policy issue | BGP |

**Update-source behavior in practice**

`update-source` changes the source IP address used by the protocol. It does not change the destination neighbor address. That distinction is the heart of many troubleshooting cases.

Think of it this way:

- The **neighbor address** is where you send the packet.
- The **update-source** is where the packet appears to come from.
- The **return path** must know how to reach that source.

Practical scenarios:

- If you peer to a directly connected interface address, you often do not need `update-source`.
- If you peer to a loopback address, you usually need `update-source` so the session uses the loopback as the source.
- If you peer to a loopback in eBGP, you also need `ebgp-multihop`.
- If the remote side has no route back to the chosen source, the session may fail even though the outbound packet leaves successfully.

Example of a failed BGP loopback peering setup:

```cisco
router bgp 65001
 neighbor 10.255.255.2 remote-as 65001
 ! missing update-source Loopback0
```

Symptoms:

- `show ip bgp summary` shows `Idle` or `Active`
- TCP connection attempts may target the neighbor address, but the source is not the loopback you intended
- The peer may never see a valid session from the expected source path

Corrected version:

```cisco
router bgp 65001
 neighbor 10.255.255.2 remote-as 65001
 neighbor 10.255.255.2 update-source Loopback0
```

If this is eBGP over loopbacks, add:

```cisco
 neighbor 10.255.255.2 ebgp-multihop 2
```

**Exam-focused distinctions**

- **OSPF**: adjacency depends heavily on interface behavior, network type, and DR/BDR logic.
- **EIGRP**: adjacency depends on matching core parameters and stable Hello exchange; named mode and stub behavior matter in modern designs.
- **IS-IS**: adjacency is circuit-based and Layer 2 oriented; think levels, circuit type, and encapsulation.
- **BGP**: session establishment depends on TCP reachability, correct neighbor address, correct source address, and TTL/multihop when peering beyond a single hop.

A quick decision tree:

1. If the protocol uses **Hello packets**, verify interface participation and parameter matching.
2. If the protocol uses **TCP**, verify neighbor IP, source address, routing, and port reachability.
3. If the protocol uses **loopbacks**, verify both the source configuration and the return path.
4. If the protocol uses **levels or areas**, verify the topology role and adjacency expectations.

**Practical verification workflow**

1. Confirm the interface or peer address is reachable.
2. Check whether the protocol is enabled on the correct interface or address family.
3. Verify matching parameters:
   - OSPF: area, timers, authentication, network type
   - EIGRP: AS, K-values, authentication, stub behavior
   - IS-IS: level, circuit type, encapsulation, authentication
   - BGP: remote AS, neighbor IP, source address, TTL/multihop
4. Inspect neighbor state or session state.
5. Compare the local source address with the return route on the peer.
6. Fix the smallest mismatch first, then recheck adjacency.

**Common CLI snippets**

```cisco
show ip ospf neighbor
show ip eigrp neighbors
show isis neighbors
show ip bgp summary
show ip route
```

```cisco
ping <neighbor-ip> source <loopback-interface>
traceroute <neighbor-ip> source <loopback-interface>
```

These source-based tests are especially useful before you troubleshoot BGP loopback peering. If the source address cannot reach the peer or the peer cannot return traffic to that source, the session will not establish reliably.

**Key takeaways**

- Neighbor establishment is always about more than reachability; protocol-specific parameters must align.
- `update-source` controls the local source address, not the neighbor destination.
- OSPF uses interface-based adjacency logic and is sensitive to network type and DR/BDR behavior.
- EIGRP requires matching core parameters and stable Hello exchange, but its neighbor troubleshooting does not use OSPF state names.
- IS-IS is Layer 2 and circuit-based, so focus on levels and circuit type rather than OSPF-style network type.
- BGP over loopbacks requires both `update-source` and a valid return path; eBGP over loopbacks also requires multihop.
