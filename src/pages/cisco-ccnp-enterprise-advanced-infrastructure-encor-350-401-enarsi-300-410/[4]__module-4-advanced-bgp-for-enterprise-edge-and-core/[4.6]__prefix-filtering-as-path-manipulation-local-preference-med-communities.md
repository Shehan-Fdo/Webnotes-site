## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**BGP policy overview**

Border Gateway Protocol (BGP) policy controls which routes you accept, advertise, and prefer. In enterprise designs, you usually combine multiple tools:

- **Prefix filtering** to control exact networks and route ranges.
- **AS-path manipulation** to influence outbound path choice from remote networks.
- **Local preference** to choose the preferred exit inside your autonomous system (AS).
- **MED** (Multi-Exit Discriminator) to suggest an entry point into your AS from a neighbor AS.
- **Communities** to tag routes so downstream policy can match them consistently.

A useful mental model is:

```text
Inbound from neighbor -> filter and tag -> best-path selection inside AS -> outbound advertisement to other peers
```

Policy direction matters:

- **Inbound policy** changes what you accept and how you rank it internally.
- **Outbound policy** changes what you advertise and how others may see your paths.
- **Internal policy** often uses communities as a reusable control plane signal.

**Prefix filtering fundamentals**

Prefix lists are the preferred route-filtering tool in Cisco BGP policy because they are more precise and efficient than standard ACLs for prefix matching. They match both the network address and the prefix length, which makes them ideal for controlling exact routes and route ranges.

A prefix-list entry has three important parts:

- The **prefix** itself, such as `10.10.0.0/16`.
- Optional length operators such as `ge` and `le`.
- A sequence number that controls evaluation order.

Example:

```cisco
ip prefix-list CUSTOMER-IN seq 10 permit 10.10.0.0/16
ip prefix-list CUSTOMER-IN seq 20 permit 10.20.0.0/16 ge 24 le 28
ip prefix-list CUSTOMER-IN seq 30 deny 0.0.0.0/0 le 32
```

Key behavior:

- `permit 10.10.0.0/16` is an **exact match** for only `10.10.0.0/16`.
- `permit 10.20.0.0/16 ge 24 le 28` matches more-specific routes under `10.20.0.0/16` with prefix lengths from `/24` through `/28`.
- The final deny is a common explicit catch-all, but remember that prefix-lists have an implicit deny at the end anyway.

Important matching rules:

- Without `ge` or `le`, the prefix length must match exactly.
- `ge` means “greater than or equal to” the specified prefix length.
- `le` means “less than or equal to” the specified prefix length.
- If you use both, the route must fall within the range.
- Sequence order matters because the first matching entry wins.

> A common exam trap is assuming a range statement includes the base prefix automatically. It does not unless the length range allows it.

A more precise example for blocking only the default route while allowing everything else:

```cisco
ip prefix-list BLOCK-DEFAULT seq 5 deny 0.0.0.0/0
ip prefix-list BLOCK-DEFAULT seq 10 permit 0.0.0.0/0 ge 1 le 32
```

This denies only `0.0.0.0/0` and permits all other IPv4 prefixes.

**Prefix-list design patterns**

Use prefix-lists for these common tasks:

- Permit only customer-owned networks.
- Block default routes from a peer.
- Allow only specific summary routes.
- Allow only more-specifics within a known aggregate.
- Protect route redistribution boundaries.

Examples:

```cisco
ip prefix-list ONLY-DEFAULT seq 10 permit 0.0.0.0/0
ip prefix-list NO-DEFAULT seq 10 deny 0.0.0.0/0
ip prefix-list NO-DEFAULT seq 20 permit 0.0.0.0/0 ge 1 le 32
```

For a route range:

```cisco
ip prefix-list CUSTOMER-MORE-SPECIFICS seq 10 permit 172.16.0.0/16 ge 24 le 28
```

This matches routes such as `172.16.1.0/24` or `172.16.32.0/27`, but not the `/16` itself because `ge 24` excludes it.

**IPv6 prefix-lists**

ENCOR and ENARSI breadth includes IPv6 policy as well. IPv6 prefix-lists work similarly, but they use IPv6 prefixes and are applied under IPv6 BGP address-family configuration.

Example:

```cisco
ipv6 prefix-list V6-CUSTOMER-IN seq 10 permit 2001:db8:100::/48 ge 64 le 64
ipv6 prefix-list V6-CUSTOMER-IN seq 20 deny ::/0 le 128
```

Notes:

- `::/0` is the IPv6 default route.
- Prefix-length ranges are especially useful for controlling customer deaggregation.
- As with IPv4, sequence order and first-match behavior still apply.

**Applying prefix-lists in BGP**

You usually attach prefix-lists to neighbors with route-maps or directly with `prefix-list` filters depending on the policy goal.

Example inbound filter:

```cisco
router bgp 65000
 neighbor 203.0.113.2 remote-as 65100
 address-family ipv4 unicast
  neighbor 203.0.113.2 prefix-list CUSTOMER-IN in
```

Example outbound filter:

```cisco
router bgp 65000
 address-family ipv4 unicast
  neighbor 203.0.113.2 prefix-list ONLY-DEFAULT out
```

Use prefix-lists when you want to control route presence before any path manipulation occurs.

**AS-path manipulation**

AS-path manipulation changes how remote ASes perceive the attractiveness of a route. The most common technique is **AS-path prepending**, where you add your own AS number multiple times to make a path look longer and therefore less preferred.

Example:

```cisco
route-map PREPEND-OUT permit 10
 set as-path prepend 65000 65000 65000
```

Applied outbound:

```cisco
router bgp 65000
 neighbor 198.51.100.1 remote-as 65200
 address-family ipv4 unicast
  neighbor 198.51.100.1 route-map PREPEND-OUT out
```

Why it works:

- Many BGP decisions prefer the shortest AS-path.
- Prepending makes one exit appear less attractive to external peers.
- It is commonly used for traffic engineering across dual-homed links.

Practical use cases:

- Prefer ISP-A for inbound traffic by prepending on ISP-B.
- Make a backup link less attractive without shutting it down.
- Influence remote ingress during maintenance windows.

**AS-path filtering with regex**

AS-path access lists let you match routes based on the AS sequence. This is useful for filtering, tagging, or selecting routes in route-maps.

Example:

```cisco
ip as-path access-list 10 permit ^65010_
ip as-path access-list 20 permit _65100$
ip as-path access-list 30 deny _64512_
```

Regex ideas:

- `^65010_` matches paths that begin with AS `65010`.
- `_65100$` matches paths that end with AS `65100`.
- `_64512_` matches any path containing AS `64512`.

Common exam-oriented patterns:

- `^$` matches locally originated routes with an empty AS-path.
- `_65000_` matches any path containing AS `65000`.
- `^65010_65020_` matches a specific AS sequence at the start.

Use AS-path ACLs in route-maps:

```cisco
route-map TAG-TRANSIT permit 10
 match as-path 10
 set local-preference 200
```

Route-map logic matters:

- A route-map sequence is processed top-down.
- If a sequence has multiple `match` statements, Cisco uses **AND** logic across different match types in the same sequence.
- Multiple entries within the same match type often behave as **OR**.
- A route-map `deny` sequence drops matched routes when applied in a filtering context.
- A route-map `permit` sequence allows matched routes and can modify attributes.

**Local preference**

Local preference is a BGP attribute used to choose the preferred exit path inside your AS. Higher local preference wins.

Key points:

- It is propagated throughout the AS via iBGP.
- It is not advertised to external BGP peers.
- It is commonly set inbound on edge routers to influence internal forwarding.

Typical values:

- Default is usually `100`.
- Higher values are preferred.
- Enterprises often use `200` or `300` for preferred exits.

Example:

```cisco
route-map SET-LP-PRIMARY permit 10
 match ip address prefix-list PRIMARY-ROUTES
 set local-preference 200

route-map SET-LP-BACKUP permit 10
 set local-preference 50
```

Applied inbound:

```cisco
router bgp 65000
 neighbor 203.0.113.2 remote-as 65100
 address-family ipv4 unicast
  neighbor 203.0.113.2 route-map SET-LP-PRIMARY in
```

Why inbound is common:

- You want the entire AS to agree on the same preferred exit.
- Setting local preference at the edge lets iBGP propagate the decision.
- It is more scalable than changing static routes or interface costs everywhere.

**MED**

MED is the **Multi-Exit Discriminator**. It suggests to an external neighbor which entry point into your AS it should prefer. Lower MED is preferred.

Important caveats:

- MED is typically compared only among routes received from the same neighboring AS.
- Cisco best-path behavior can be changed with `bgp always-compare-med`, which makes MED comparison broader.
- `bgp deterministic-med` helps make MED comparisons more consistent across multiple paths.

Example:

```cisco
route-map SET-MED permit 10
 set metric 50
```

In BGP route-maps, `set metric` sets the MED value.

Applied outbound to a neighbor:

```cisco
router bgp 65000
 neighbor 198.51.100.1 remote-as 65200
 address-family ipv4 unicast
  neighbor 198.51.100.1 route-map SET-MED out
```

Use MED when:

- You have multiple links to the same external AS.
- You want to influence how that AS enters your network.
- You need a softer hint than AS-path prepending.

MED comparison behavior:

- Lower is better.
- It is often considered after local preference and AS-path length.
- It is not as universally influential as local preference because many networks ignore it or compare it only in limited cases.

Useful knobs:

```cisco
router bgp 65000
 bgp deterministic-med
 bgp always-compare-med
```

Interpretation:

- `deterministic-med` improves consistency when multiple paths from the same AS exist.
- `always-compare-med` forces comparison across different neighboring ASes, which can change best-path selection.

**Communities**

Communities are tags attached to BGP routes so you can apply policy consistently. They are especially useful in large networks because they separate route classification from route action.

Standard community format:

- 32-bit value written as `ASN:VALUE`, such as `65000:100`.
- Communities are transitive by default unless explicitly handled otherwise.

Common uses:

- Mark customer routes for higher local preference.
- Mark backup routes for lower preference.
- Tag routes that should not be advertised to certain peers.
- Apply consistent policy across many edge routers.

Example:

```cisco
route-map TAG-CUSTOMER permit 10
 set community 65000:100
```

To preserve existing communities, use additive behavior:

```cisco
route-map TAG-CUSTOMER permit 10
 set community 65000:100 additive
```

Without `additive`, the new community replaces existing communities in many Cisco policy contexts.

**Community matching in policy**

You can match communities in route-maps to drive local preference, filtering, or redistribution decisions.

Example inbound policy:

```cisco
ip community-list standard CUSTOMER-PREF permit 65000:100

route-map SET-HIGH-LP permit 10
 match community CUSTOMER-PREF
 set local-preference 200

route-map SET-HIGH-LP permit 20
 set local-preference 100
```

This pattern lets you:

- Tag routes at one edge.
- Match the tag at another policy point.
- Apply a consistent internal preference.

Example filtering:

```cisco
ip community-list standard BLOCK-TRANSIT permit 65000:666

route-map FILTER-BAD permit 10
 match community BLOCK-TRANSIT
 deny
```

Community lists can be standard or expanded:

- **Standard community lists** match exact community values.
- **Expanded community lists** use regex-like matching for more flexible patterns.

**Well-known communities**

Cisco exam prep should distinguish ordinary communities from well-known ones.

Common well-known communities include:

- `no-export` — do not advertise beyond the local AS.
- `no-advertise` — do not advertise to any BGP peer.
- `internet` — advertise to the Internet; this is the default concept in many contexts.
- `local-AS` — used with BGP confederations, not as a general-purpose community for ordinary enterprise policy.

Use `no-export` when you want a route to stay inside your AS and not leave to external peers.

Example:

```cisco
route-map TAG-PRIVATE permit 10
 set community no-export additive
```

**Extended and large communities**

Standard communities are 32 bits, which can be limiting in large designs. Cisco exam preparation should include the broader community types.

- **Extended communities** are 64-bit and are often used for VPN-related signaling, route targets, and more structured policy.
- **Large communities** are 96-bit and are useful when you need more namespace flexibility than standard communities provide.

When to prefer each:

- Use **standard communities** for simple enterprise tagging.
- Use **extended communities** when the feature or VPN design requires them.
- Use **large communities** when you need scalable tagging across many customers, regions, or policy domains.

Example large community syntax:

```cisco
route-map TAG-LARGE permit 10
 set large-community 65000:100:1
```

Example extended community syntax varies by feature and platform, but route-target style policy is common in MPLS VPN designs.

**Route-map behavior and policy logic**

Route-maps are the glue that connects prefix-lists, AS-path ACLs, and communities to BGP actions.

Core rules:

- Sequences are evaluated in ascending order.
- The first matching sequence is used.
- A `permit` sequence allows the route and can modify attributes.
- A `deny` sequence blocks the route when the route-map is used for filtering.
- If no sequence matches, the implicit result is deny in filtering contexts.

Multiple match types:

- Different match types in the same sequence are typically ANDed.
- Multiple entries under the same match type are often ORed.
- This makes route-maps powerful but easy to misread.

Example combining conditions:

```cisco
route-map INBOUND-POLICY permit 10
 match ip address prefix-list CUSTOMER-IN
 match as-path 10
 set local-preference 200
 set community 65000:100 additive
```

This sequence only matches routes that satisfy both the prefix-list and the AS-path ACL.

**End-to-end enterprise example**

Consider a dual-homed enterprise with two ISPs and two edge routers. You want:

- ISP-A to be the primary inbound and outbound path.
- ISP-B to be backup.
- Customer routes to be accepted only if they match approved prefixes.
- Internal routers to prefer routes learned from the primary edge.
- Remote networks to prefer one of your exits for inbound traffic.

A practical design might look like this:

```text
          ISP-A
            |
         Edge-1 -------- iBGP -------- Core
            |
         Edge-2
            |
          ISP-B
```

Policy plan:

- Inbound from each ISP:
  - Filter with prefix-lists.
  - Tag primary routes with higher local preference.
  - Tag backup routes with lower local preference.
- Outbound to ISP-B:
  - Prepend your AS to make it less attractive.
- To the same external AS on two links:
  - Use MED to suggest the preferred entry point.
- Across the enterprise:
  - Use communities to carry policy intent between edge routers and route reflectors.

Example configuration sketch:

```cisco
ip prefix-list ISP-A-IN seq 10 permit 203.0.113.0/24
ip prefix-list ISP-B-IN seq 10 permit 198.51.100.0/24

ip community-list standard PRIMARY permit 65000:100
ip community-list standard BACKUP permit 65000:50

route-map ISP-A-IN permit 10
 match ip address prefix-list ISP-A-IN
 set local-preference 200
 set community 65000:100 additive

route-map ISP-B-IN permit 10
 match ip address prefix-list ISP-B-IN
 set local-preference 50
 set community 65000:50 additive

route-map ISP-B-OUT permit 10
 set as-path prepend 65000 65000 65000
```

This design gives you:

- Strong inbound control through prefix filtering.
- Internal path preference through local preference.
- External traffic engineering through AS-path prepending.
- Reusable route tagging through communities.

**Verification and troubleshooting**

Use verification commands that match the policy you configured.

Prefix-list checks:

```cisco
show ip prefix-list
show ip prefix-list CUSTOMER-IN
```

Useful output includes:

- Sequence hit counts.
- Whether entries are active.
- Which prefixes matched.

BGP route inspection:

```cisco
show ip bgp
show ip bgp 10.10.10.0
show ip bgp neighbors 203.0.113.2 advertised-routes
show ip bgp neighbors 203.0.113.2 received-routes
```

Notes:

- `received-routes` may require soft reconfiguration inbound or route-refresh support.
- `advertised-routes` helps confirm outbound policy.
- `show ip bgp <prefix>` is often the fastest way to inspect attributes such as local preference, MED, AS-path, and communities.

Community inspection:

```cisco
show ip bgp 10.10.10.0
show ip bgp regexp _65000_
```

AS-path and regex troubleshooting:

```cisco
show ip bgp regexp ^65010_
show ip bgp regexp _65100$
```

Route-map and policy hit counts:

```cisco
show route-map
show ip bgp neighbors 203.0.113.2 routes
```

Be careful:

- `show ip bgp neighbors <neighbor> routes` is not the standard command for received routes on Cisco IOS/IOS XE.
- Use `received-routes` for inbound-learned routes when supported.
- Route-map hit counts help confirm whether your match logic is actually triggering.

**Troubleshooting workflow**

1. Verify the route exists in BGP.
2. Check whether the prefix-list permits or denies it.
3. Confirm the route-map sequence matched.
4. Inspect the resulting attributes in the BGP table.
5. Validate best-path selection on the router that makes the forwarding decision.
6. Confirm outbound advertisement if the issue is on the egress side.

A practical checklist:

- If a route is missing, check prefix filtering first.
- If a route is present but not preferred, check local preference and AS-path length.
- If remote ingress is wrong, check AS-path prepending and MED.
- If policy is inconsistent across routers, check community tagging and matching.

**Best-path attribute order to remember**

BGP best-path selection is vendor-specific in details, but for Cisco exam purposes, these attributes are the ones you most often manipulate:

- Highest local preference.
- Shortest AS-path.
- Lowest origin type.
- Lowest MED, usually among routes from the same neighboring AS.
- eBGP over iBGP.
- Lowest IGP cost to next hop.

This is why:

- Local preference is the main internal exit-control tool.
- AS-path prepending is the main external ingress-control tool.
- MED is a softer hint for multi-exit links to the same AS.
- Communities are the scalable way to carry policy intent.

**Quick comparison**

| Attribute / Tool | Direction | Main purpose | Typical preference rule |
|---|---|---|---|
| Prefix-list | Inbound or outbound | Filter routes | Permit or deny by prefix and length |
| AS-path prepend | Outbound | Influence remote ingress | Longer path is less preferred |
| Local preference | Inbound/internal | Choose exit inside AS | Higher is better |
| MED | Outbound/inbound hint | Suggest entry into AS | Lower is better |
| Communities | Both | Tag and reuse policy | Match tags in route-maps |

**Exam reminders**

- Prefix-lists match prefix and length; exact match is not the same as range match.
- Sequence order matters in prefix-lists and route-maps.
- `set local-preference` is an internal BGP decision tool.
- `set metric` in a BGP route-map sets MED.
- MED is usually compared only within the same neighboring AS unless you change behavior.
- `no-export` and `no-advertise` are well-known communities; `local-AS` is tied to confederations.
- Standard communities are simple; extended and large communities scale better for complex designs.
- Use `show ip bgp <prefix>` and route-map hit counts to validate policy.
