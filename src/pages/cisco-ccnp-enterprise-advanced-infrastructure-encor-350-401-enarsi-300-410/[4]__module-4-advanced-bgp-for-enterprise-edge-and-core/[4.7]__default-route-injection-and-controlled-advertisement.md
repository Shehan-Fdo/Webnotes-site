## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Overview**

A default route is the route of last resort. You inject it when you want downstream routers to send unknown destinations toward a gateway, edge, or exit point instead of carrying a full routing table. Controlled advertisement means you do not blindly flood that default everywhere; you use protocol features and policy to decide where it appears and who receives it.

In enterprise designs, default routes are common at:
- Internet edges
- WAN hubs
- Data center borders
- Stub and branch sites
- Redistribution boundaries between routing domains

The exam focus is not just “how to advertise `0.0.0.0/0`,” but also:
- when a protocol requires the route to exist locally
- how to advertise only to selected neighbors
- how to prevent accidental transit
- how to verify the route is actually installed downstream

**Core concepts**

A default route is usually represented as:
- IPv4: `0.0.0.0/0`
- IPv6: `::/0`

A router can learn or create a default route in several ways:
- Static route
- Dynamic routing protocol origination
- Redistribution from another protocol
- Summarization or special area behavior in some protocols
- Policy-based advertisement to specific peers

Downstream routers choose among multiple defaults using normal route selection:
- **Administrative distance (AD)** decides between different sources
- **Metric** decides between routes from the same source
- **Next hop reachability** must be valid
- Policy can suppress a route before it is installed

> A default route that is advertised is not always a default route that is installed. Always verify both the control plane and the routing table.

**Static default routes and local origination**

A static default route is the simplest starting point. You often create it at the edge, then advertise it into the routing domain.

```bash
ip route 0.0.0.0 0.0.0.0 203.0.113.1
ipv6 route ::/0 2001:db8:100::1
```

Common uses:
- Internet edge pointing to ISP next hop
- Stub router pointing to hub
- Backup path with higher administrative distance

You can also point a static default to an exit interface on point-to-point links:
```bash
ip route 0.0.0.0 0.0.0.0 GigabitEthernet0/0/0
```

Be careful on multiaccess networks:
- An interface-only default can cause ARP or next-hop ambiguity
- A next-hop IP is usually safer and clearer

**OSPF default route injection**

OSPF (Open Shortest Path First) commonly injects a default route from an ASBR (Autonomous System Boundary Router) or an ABR (Area Border Router), depending on area type and design.

**1. `default-information originate`**

This command originates a default route into OSPF when the router has a matching default in its routing table, unless you add `always`.

```bash
router ospf 10
 default-information originate
```

If you want to advertise the default even when the router does not currently have one in the RIB (Routing Information Base), use:

```bash
router ospf 10
 default-information originate always
```

You can also set metric attributes:

```bash
router ospf 10
 default-information originate metric 10 metric-type 1
```

Key points:
- The default is advertised as an external LSA, usually Type 5
- In an NSSA (Not-So-Stubby Area), it is initially Type 7 inside the NSSA and may be translated by the ABR
- `metric-type 1` means the external metric is added to the internal cost to reach the ASBR
- `metric-type 2` means the external metric is preferred first; internal cost is used only as a tie-breaker

**2. Route-map control**

You can conditionally originate the default with a route-map.

```bash
route-map OSPF-DEFAULT permit 10
 match ip address prefix-list DEFAULT-OK

ip prefix-list DEFAULT-OK seq 5 permit 0.0.0.0/0

router ospf 10
 default-information originate route-map OSPF-DEFAULT
```

Use this when you want the router to advertise a default only if some condition is true, such as:
- a tracked static route is present
- a specific prefix exists in the RIB
- a policy condition is met

Important:
- The route-map controls whether the default is originated
- It does not change the contents of the default LSA itself
- The match must evaluate against the route or condition you intend to test

A common design is to tie the default to tracking:
```bash
track 1 ip route 0.0.0.0 0.0.0.0 reachability

route-map OSPF-DEFAULT permit 10
 match track 1
```

**3. OSPF area behavior**

OSPF area type affects default route behavior:

- **Normal area**
  - No automatic default injection
  - You must configure `default-information originate` or redistribute a static default
- **Stub area**
  - The ABR injects a default route into the stub area automatically
  - External LSAs are blocked
- **Totally stubby area**
  - The ABR injects a default route
  - Inter-area summary routes are also suppressed, leaving mostly the default plus intra-area routes
- **NSSA**
  - External routes are allowed in a limited form
  - The ABR can inject a default into the NSSA
  - External defaults may appear as Type 7 inside the NSSA
- **Totally NSSA**
  - Similar to NSSA, but summary routes are suppressed
  - The ABR still provides a default route

Typical ABR configuration for a stub area:
```bash
router ospf 10
 area 0 range 10.10.0.0 255.255.0.0
 area 10 stub
```

Typical totally stubby area behavior on Cisco IOS:
```bash
router ospf 10
 area 10 stub no-summary
```

Typical NSSA default injection:
```bash
router ospf 10
 area 10 nssa
 area 10 nssa default-information-originate
```

For totally NSSA:
```bash
router ospf 10
 area 10 nssa no-summary
 area 10 nssa default-information-originate
```

**4. Verification**

Useful commands:
```bash
show ip route ospf
show ip ospf database external
show ip ospf database nssa-external
show ip ospf neighbor
show ip ospf interface brief
```

What to look for:
- `O*E2` or `O*E1` in the routing table for an external default
- `O*IA` or a default learned from an ABR in stub-style designs
- Type 5 external LSA for normal OSPF default origination
- Type 7 LSA in NSSA before translation

Example output clue:
- `O*E2 0.0.0.0/0` means an external default route is installed
- `O*IA 0.0.0.0/0` may appear in some area designs where the ABR supplies the default

**OSPF troubleshooting**

If the default is missing:
- Confirm the router actually has a default route in the RIB when not using `always`
- Confirm the route-map matches the intended route or condition
- Confirm the area type allows the expected behavior
- Confirm the neighbor adjacency is full
- Confirm the LSA is present in the database
- Confirm downstream routers are not filtering the route

If the default is present in OSPF but not installed downstream:
- Check administrative distance conflicts
- Check for a better default from another protocol
- Check route filtering on the receiving router
- Check next-hop reachability

**EIGRP default route injection**

EIGRP (Enhanced Interior Gateway Routing Protocol) supports several default-route mechanisms. Exam questions often test whether you know the difference between a true default route, a candidate default, and a summary-generated default.

**1. Redistribute a static default**

A common method is to create a static default and redistribute it into EIGRP.

```bash
ip route 0.0.0.0 0.0.0.0 198.51.100.1

router eigrp 100
 redistribute static metric 10000 100 255 1 1500
```

Important:
- EIGRP redistribution requires a metric
- If you do not set a metric or a `default-metric`, redistribution fails
- The metric values are bandwidth, delay, reliability, load, and MTU in that order

You can also set a default metric once:
```bash
router eigrp 100
 default-metric 10000 100 255 1 1500
 redistribute static
```

**2. `ip default-network`**

`ip default-network` is a legacy mechanism that can still appear in troubleshooting and older designs.

```bash
ip default-network 172.16.0.0
```

Behavior:
- The network must be classful and known in the routing table
- The route becomes a candidate default
- Downstream routers may treat it as a default route

This feature is largely legacy, but it still exists in IOS and can appear in exam scenarios.

**3. Summary route to `0.0.0.0/0`**

EIGRP can advertise a summary default on an interface.

```bash
interface GigabitEthernet0/0/0
 ip summary-address eigrp 100 0.0.0.0 0.0.0.0
```

Use this carefully:
- It creates a summary route toward neighbors on that interface
- It is useful when you want to suppress more-specific routes and advertise only a default
- It is interface-specific, not a global process command

**4. Controlled advertisement with distribute-lists and route-maps**

EIGRP can filter routes with distribute-lists, but the tool and direction matter.

Example using a prefix-list to block only the default route on inbound updates:
```bash
ip prefix-list BLOCK-DEFAULT seq 5 deny 0.0.0.0/0
ip prefix-list BLOCK-DEFAULT seq 10 permit 0.0.0.0/0 le 32

router eigrp 100
 distribute-list prefix BLOCK-DEFAULT in GigabitEthernet0/0/0
```

This example is subtle:
- The deny line blocks the exact default route
- The permit line allows all other prefixes
- The interface direction matters because EIGRP distribute-lists can be applied inbound or outbound depending on design

For more complex policy, use route-maps with redistribution:
```bash
route-map EIGRP-REDIST permit 10
 match ip address prefix-list DEFAULT-ONLY

ip prefix-list DEFAULT-ONLY seq 5 permit 0.0.0.0/0

router eigrp 100
 redistribute static route-map EIGRP-REDIST metric 10000 100 255 1 1500
```

**5. Verification**

```bash
show ip route eigrp
show ip eigrp topology
show ip protocols
show ip eigrp neighbors
```

Look for:
- `D EX 0.0.0.0/0` for external EIGRP default
- `D*` for a candidate default in some scenarios
- topology entries showing the route source and metric

**EIGRP troubleshooting**

If the default is not advertised:
- Confirm a metric is configured for redistribution
- Confirm the static default exists if you are redistributing static
- Confirm the summary is applied on the correct interface
- Confirm the distribute-list is not blocking the route
- Confirm the route is not filtered by an offset or route-map

If downstream routers do not use the default:
- Check whether another default has a lower AD
- Check whether the next hop is reachable
- Check whether the route is external and being filtered by policy

**BGP default route injection**

BGP (Border Gateway Protocol) gives you precise control over default advertisement. The main methods are `network`, `default-originate`, and redistribution.

**1. `network 0.0.0.0`**

A BGP `network` statement originates a prefix only if an exact matching route exists in the local routing table.

```bash
router bgp 65000
 network 0.0.0.0
```

For IPv4, the exact `0.0.0.0/0` route must be present in the RIB. The same principle applies to IPv6:

```bash
router bgp 65000
 address-family ipv6 unicast
  network ::/0
```

Use this when:
- you want BGP to advertise a default that already exists locally
- you want deterministic origination based on the RIB
- you want the route to be part of the BGP table and eligible for propagation

**2. `default-originate`**

`default-originate` advertises a default to a specific neighbor. It does not require the local BGP table to contain `0.0.0.0/0`.

```bash
router bgp 65000
 neighbor 192.0.2.2 remote-as 65001
 neighbor 192.0.2.2 default-originate
```

This is neighbor-specific and commonly paired with a route-map:
```bash
route-map BGP-DEFAULT permit 10
 match ip address prefix-list DEFAULT-OK

ip prefix-list DEFAULT-OK seq 5 permit 0.0.0.0/0

router bgp 65000
 neighbor 192.0.2.2 default-originate route-map BGP-DEFAULT
```

Use this when:
- you want to send a default only to one peer or one peer group
- you want conditional advertisement
- you do not want to install a default in the local BGP table

**3. Redistribution into BGP**

You can redistribute a static default into BGP.

```bash
ip route 0.0.0.0 0.0.0.0 203.0.113.1

router bgp 65000
 redistribute static
```

In practice, redistribution is often controlled with a route-map:
```bash
route-map REDIST-DEFAULT permit 10
 match ip address prefix-list DEFAULT-ONLY

ip prefix-list DEFAULT-ONLY seq 5 permit 0.0.0.0/0

router bgp 65000
 redistribute static route-map REDIST-DEFAULT
```

**4. `aggregate-address` and default routes**

`aggregate-address` is not the normal way to create a default route. It summarizes more-specific routes into a less-specific prefix when the component routes exist.

Use it for aggregation, not for default origination unless your design specifically creates a summary that includes the default prefix, which is uncommon and usually not the intent.

**5. Controlled advertisement with route-maps**

A route-map on a BGP neighbor can filter outbound updates.

```bash
ip prefix-list ONLY-DEFAULT seq 5 permit 0.0.0.0/0

route-map BGP-OUT permit 10
 match ip address prefix-list ONLY-DEFAULT

router bgp 65000
 neighbor 192.0.2.2 remote-as 65001
 neighbor 192.0.2.2 route-map BGP-OUT out
```

This means:
- only routes matching the permit sequence are advertised
- everything else is implicitly denied unless another permit sequence matches
- the route-map is evaluated top-down

If you want to advertise only the default and nothing else, this is a common pattern.

**6. Verification**

```bash
show ip bgp 0.0.0.0
show ip bgp summary
show ip bgp neighbors 192.0.2.2 advertised-routes
show ip route 0.0.0.0
```

For IPv6:
```bash
show bgp ipv6 unicast ::/0
show bgp ipv6 unicast summary
show bgp ipv6 unicast neighbors 2001:db8:1::2 advertised-routes
show ipv6 route ::/0
```

Platform syntax can vary, so use the command set supported by the IOS or IOS XE release you are studying.

**BGP troubleshooting**

If `network 0.0.0.0` does not work:
- Confirm an exact `0.0.0.0/0` route exists in the local RIB
- Confirm the BGP address family is active
- Confirm the route is not suppressed by policy or next-hop issues

If `default-originate` does not work:
- Confirm the neighbor is up
- Confirm the route-map, if used, permits advertisement
- Confirm outbound policy is not blocking the update
- Confirm the neighbor is in the correct address family

If the neighbor receives too many routes:
- Check outbound route-maps
- Check prefix-list logic
- Check whether the route-map has an implicit deny at the end

**IPv6 default route injection**

IPv6 uses the same conceptual default route, `::/0`, but the configuration is protocol-specific.

**1. Static IPv6 default**

```bash
ipv6 route ::/0 2001:db8:100::1
```

**2. OSPFv3**

OSPFv3 can inject a default route similarly to OSPFv2, but the configuration is under the IPv6 routing process.

```bash
ipv6 router ospf 10
 default-information originate
```

Interface activation:
```bash
interface GigabitEthernet0/0/0
 ipv6 ospf 10 area 0
```

Verification:
```bash
show ipv6 route ospf
show ipv6 ospf database
show ipv6 ospf neighbor
```

**3. EIGRP for IPv6**

EIGRP for IPv6 uses the IPv6 address-family style configuration.

```bash
ipv6 router eigrp 100
 eigrp router-id 1.1.1.1
 redistribute static metric 10000 100 255 1 1500
```

Interface activation:
```bash
interface GigabitEthernet0/0/0
 ipv6 eigrp 100
```

Verification:
```bash
show ipv6 route eigrp
show ipv6 eigrp topology
show ipv6 eigrp neighbors
```

**4. MP-BGP for IPv6**

For IPv6 BGP, the exact prefix must exist in the IPv6 routing table if you use `network ::/0`.

```bash
router bgp 65000
 address-family ipv6 unicast
  network ::/0
```

For neighbor-specific default advertisement:
```bash
router bgp 65000
 address-family ipv6 unicast
  neighbor 2001:db8:1::2 activate
  neighbor 2001:db8:1::2 default-originate
```

Verification:
```bash
show bgp ipv6 unicast ::/0
show bgp ipv6 unicast neighbors 2001:db8:1::2 advertised-routes
show ipv6 route ::/0
```

**Controlled advertisement techniques**

Controlled advertisement means you decide which routes are allowed to leave a router or enter a routing process.

**Common tools**
- **Route-maps** for policy logic
- **Prefix-lists** for exact prefix matching
- **Distribute-lists** for protocol-specific filtering
- **Route filtering** at redistribution boundaries
- **Conditional advertisement** based on route presence or tracking
- **Summarization** to reduce route detail

**Route-map behavior**
- Evaluated top-down
- First match wins
- `permit` allows the route if the match succeeds
- `deny` blocks the route if the match succeeds
- If no sequence matches, the implicit result is deny

Example:
```bash
route-map OUT-DEFAULT permit 10
 match ip address prefix-list DEFAULT-ONLY

route-map OUT-DEFAULT deny 20
 match ip address prefix-list BLOCK-ALL
```

**Prefix-list pitfalls**
- `permit 0.0.0.0/0` matches only the exact default route
- `permit 0.0.0.0/0 le 32` matches everything in IPv4, including the default
- For IPv6, use IPv6 prefix-list syntax and match the exact `::/0` when needed

Example exact match:
```bash
ip prefix-list DEFAULT-ONLY seq 5 permit 0.0.0.0/0
ipv6 prefix-list DEFAULT6-ONLY seq 5 permit ::/0
```

**Distribute-list vs route-map**
- Distribute-lists filter routes in or out of a routing process
- Route-maps are more flexible and often used with redistribution or neighbor policy
- BGP commonly uses route-maps for outbound and inbound policy
- EIGRP and OSPF use protocol-specific policy features, but redistribution often relies on route-maps

**Multiple default routes and route selection**

A router may learn more than one default route. The installed route depends on:
- Administrative distance
- Metric
- Protocol preference
- Policy and next-hop reachability

Typical AD values vary by protocol and platform defaults, but common Cisco defaults are:
- Connected: 0
- Static: 1
- eBGP: 20
- EIGRP internal: 90
- OSPF: 110
- IS-IS: 115
- RIP: 120
- EIGRP external: 170
- iBGP: 200

Practical implications:
- A static default usually beats a dynamic default unless you raise its AD
- An eBGP-learned default often beats OSPF and EIGRP defaults
- If you want a backup default, use a higher AD on the backup static route

Example primary and backup defaults:
```bash
ip route 0.0.0.0 0.0.0.0 203.0.113.1
ip route 0.0.0.0 0.0.0.0 198.51.100.1 200
```

This installs the first route as primary and the second as floating static backup.

**End-to-end lab patterns**

**Lab 1: OSPF edge default**
1. Configure a static default on the edge router.
2. Originate it into OSPF with `default-information originate`.
3. Verify downstream routers install `O*E2 0.0.0.0/0`.
4. Remove the static default and confirm the OSPF default disappears unless `always` is configured.

Example:
```bash
ip route 0.0.0.0 0.0.0.0 203.0.113.1

router ospf 10
 default-information originate
```

**Lab 2: EIGRP hub default**
1. Configure a static default on the hub.
2. Redistribute static into EIGRP with a metric or `default-metric`.
3. Verify spokes learn `D EX 0.0.0.0/0`.
4. Add a distribute-list to block the default on one spoke and confirm only that spoke loses it.

Example:
```bash
ip route 0.0.0.0 0.0.0.0 198.51.100.1

router eigrp 100
 default-metric 10000 100 255 1 1500
 redistribute static
```

**Lab 3: BGP edge default to one neighbor**
1. Install a static default in the edge RIB.
2. Use `network 0.0.0.0` if you want the default in BGP globally.
3. Use `default-originate` if you want to send it only to one neighbor.
4. Apply a route-map to control advertisement.
5. Verify with advertised-routes and the neighbor’s routing table.

Example:
```bash
ip route 0.0.0.0 0.0.0.0 203.0.113.1

router bgp 65000
 network 0.0.0.0
 neighbor 192.0.2.2 remote-as 65001
 neighbor 192.0.2.2 default-originate
```

**Troubleshooting checklist**

When a default route is missing, ask:
- Is the default present in the local RIB?
- Does the protocol require an exact match?
- Is the area or neighbor type correct?
- Is a route-map denying origination?
- Is redistribution missing a metric?
- Is the route filtered on the receiving side?
- Is a better default already installed?
- Is the next hop reachable?

When a default route is present but not used:
- Check AD and metric
- Check route preference between protocols
- Check for policy-based filtering
- Check for recursive next-hop failure
- Check whether the route is only advertised to one neighbor

**Quick comparison**

| Protocol | Common default method | Key prerequisite | Typical verification |
|---|---|---|---|
| OSPF | `default-information originate` | Default in RIB unless `always` | `show ip ospf database external` |
| OSPFv3 | `default-information originate` | IPv6 default in RIB unless `always` | `show ipv6 ospf database` |
| EIGRP | Redistribute static or summary default | Metric required for redistribution | `show ip eigrp topology` |
| EIGRP for IPv6 | Redistribute static or summary default | Metric required for redistribution | `show ipv6 eigrp topology` |
| BGP | `network 0.0.0.0` or `default-originate` | Exact RIB match for `network` | `show ip bgp 0.0.0.0` |
| MP-BGP IPv6 | `network ::/0` or `default-originate` | Exact IPv6 RIB match for `network` | `show bgp ipv6 unicast ::/0` |

**Exam reminders**

- `default-information originate` is not the same as area summarization.
- `always` overrides the need for a local default route in OSPF.
- EIGRP redistribution needs a metric.
- `network 0.0.0.0` in BGP requires an exact default route in the RIB.
- `default-originate` is neighbor-specific in BGP.
- Route-maps use top-down logic with an implicit deny at the end.
- Stub, totally stubby, NSSA, and totally NSSA areas have built-in default behavior that differs from normal areas.
- IPv6 uses the same concepts, but configuration and verification are address-family specific.
