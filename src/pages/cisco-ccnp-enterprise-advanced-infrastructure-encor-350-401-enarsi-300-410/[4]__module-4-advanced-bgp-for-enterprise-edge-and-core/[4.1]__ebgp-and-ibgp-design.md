## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**BGP design goals**

Border Gateway Protocol (BGP) is the interdomain routing protocol used to exchange reachability between autonomous systems (ASes). In enterprise designs, you use **eBGP** (external BGP) at the edge and **iBGP** (internal BGP) inside the AS to control policy, scale, and path selection.

Good BGP design balances:
- **Reachability** across the underlay and overlay
- **Policy control** for inbound and outbound traffic
- **Scalability** for many routers and prefixes
- **Stability** under failure and convergence events
- **Security** against spoofing and session abuse

BGP is not a fast-flooding protocol like an IGP. It depends on TCP port 179, explicit neighbor relationships, and a working forwarding path for next-hop resolution. That means design choices around loopbacks, multihop, route reflectors, and next-hop handling matter as much as the route policy itself.

**eBGP vs iBGP fundamentals**

eBGP connects routers in different ASes. iBGP connects routers within the same AS. The protocol behavior differs in ways that affect design:

- **eBGP** typically uses directly connected neighbors and a TTL of 1 by default.
- **iBGP** can peer over loopbacks and often uses the IGP to provide reachability.
- **AS_PATH** is extended by eBGP when routes cross AS boundaries.
- **iBGP split-horizon behavior** prevents a route learned from one iBGP peer from being advertised to another iBGP peer unless you use a route reflector or confederation.
- **Next-hop handling** differs: eBGP commonly sets next hop to the advertising router for routes sent to an eBGP peer, while iBGP usually preserves the next hop unless you change it with policy.

A practical design rule is simple: use eBGP at the edge for policy and iBGP inside for distribution, then add route reflectors or confederations when full-mesh iBGP no longer scales.

**Neighbor establishment requirements**

A BGP session forms only when several conditions are met. In exam and lab scenarios, verify these first:

- The neighbor IP address is reachable at the IP layer.
- Both peers use the correct **remote AS**.
- TCP/179 is not blocked by ACLs, firewalls, or control-plane policy.
- The source address matches what the peer expects, especially when peering from loopbacks.
- Authentication, if configured, matches on both sides.
- TTL and multihop settings are compatible with the topology.

For loopback peering, you usually configure an update source and ensure the IGP or static routing can reach the loopback addresses.

```cisco
router bgp 65001
 neighbor 10.0.0.2 remote-as 65002
 neighbor 10.0.0.2 update-source Loopback0
```

If the peer is not directly connected, you also need multihop on the eBGP side:

```cisco
router bgp 65001
 neighbor 10.0.0.2 ebgp-multihop 2
 neighbor 10.0.0.2 update-source Loopback0
```

Use multihop carefully. It is common in routed edge designs, but it increases the importance of underlay reachability and security controls.

**eBGP design patterns**

eBGP is the preferred choice for edge routing because it gives you clear policy boundaries and strong path selection signals. Common patterns include:
- Internet edge peering with a provider
- WAN edge peering between enterprise ASes
- Data center interconnect between administrative domains
- Backup or selective transit links

A basic directly connected eBGP edge looks like this:

```cisco
interface GigabitEthernet0/0
 ip address 203.0.113.2 255.255.255.252

router bgp 65001
 bgp log-neighbor-changes
 neighbor 203.0.113.1 remote-as 65010
 !
 address-family ipv4 unicast
  network 198.51.100.0 mask 255.255.255.0
  neighbor 203.0.113.1 activate
 exit-address-family
```

Important design points:
- **Direct peering** is simplest and most secure when the link is point-to-point.
- **Loopback peering** improves resilience when the physical interface fails, but it requires multihop and a stable underlay.
- **Policy separation** is easier when you keep edge peers in their own peer-groups or templates.
- **Default route origination** in BGP is typically done with `network 0.0.0.0 mask 0.0.0.0` or `neighbor ... default-originate`, not with IGP commands.

Example default origination toward a downstream peer:

```cisco
router bgp 65001
 address-family ipv4 unicast
  network 0.0.0.0
  neighbor 203.0.113.1 default-originate
```

Use `default-originate` when you want to advertise a default only to a specific neighbor, often after verifying that a valid upstream default exists in the local routing table.

**iBGP design and scaling**

iBGP distributes external reachability inside the AS. The classic requirement is a full mesh of iBGP sessions so every router learns every BGP route. That works only for small topologies.

Scaling options:
- **Full mesh iBGP** for very small networks
- **Route reflectors** for most enterprise and service-provider-like designs
- **Confederations** when you want to split a large AS into smaller sub-ASes while preserving a single external AS identity

iBGP does not change the AS_PATH when advertising routes inside the AS. Because of that, iBGP relies on the IGP for next-hop reachability and on additional mechanisms for route distribution.

A common iBGP design uses loopbacks:
- Peers source BGP from loopback interfaces.
- The IGP carries loopback reachability.
- BGP next hops are resolved recursively through the IGP.

This design improves stability because a physical interface failure does not necessarily tear down the BGP session if the loopback remains reachable through another path.

**Loopback peering and recursive next-hop resolution**

Loopback peering is common for both iBGP and multihop eBGP. It gives you a stable endpoint, but it adds an underlay dependency.

You must ensure:
- The loopback addresses are advertised in the IGP or static routes.
- The next hop for BGP-learned routes is reachable in the routing table.
- The path to the loopback survives link or path failures.

Example:

```cisco
interface Loopback0
 ip address 10.255.0.1 255.255.255.255

router ospf 10
 network 10.255.0.1 0.0.0.0 area 0

router bgp 65001
 neighbor 10.255.0.2 remote-as 65001
 neighbor 10.255.0.2 update-source Loopback0
```

If the IGP cannot reach the BGP next hop, the route may appear in the BGP table but not install in the routing table. This is a common troubleshooting point.

**Route reflectors**

A route reflector (RR) solves the iBGP full-mesh problem by allowing one router to reflect routes between clients. This reduces session count and improves scale.

Core behavior:
- A route learned from one **client** can be reflected to other clients.
- A route learned from a client can also be advertised to non-client iBGP peers, subject to policy.
- A route learned from a non-client is reflected to clients, but not automatically to other non-clients.

Design guidance:
- Place RRs close to the center of the topology, not at the edge.
- Use at least two RRs for redundancy.
- Make sure clients have multiple RR paths when possible.
- Keep the IGP underlay stable so clients can reach the RR loopbacks.
- Plan for failure domains; do not put all clients behind a single physical or logical point of failure.

Example RR configuration:

```cisco
router bgp 65001
 bgp cluster-id 1.1.1.1
 neighbor 10.255.0.2 remote-as 65001
 neighbor 10.255.0.2 update-source Loopback0
 !
 address-family ipv4 unicast
  neighbor 10.255.0.2 route-reflector-client
```

Useful RR considerations:
- **Cluster ID** helps prevent reflection loops when multiple RRs exist.
- **Client-to-client reflection** allows clients to learn routes from each other through the RR, which reduces the need for direct peering.
- **Best-path selection** still happens on each router; the RR does not force all clients to use the same exit unless policy and attributes align.
- **Scaling tradeoff:** fewer sessions, but more dependence on RR placement and consistent policy.

In multi-RR designs, use consistent policy and cluster planning. If you deploy multiple RRs, ensure clients can reach both and that route reflection does not create unexpected path hiding.

**Confederations**

A BGP confederation splits one large AS into smaller sub-ASes for internal scaling while presenting a single external AS number to outside peers. This is useful when you want some of the benefits of separate domains without fully exposing multiple ASes to the Internet or to external partners.

Typical configuration elements:
- `bgp confederation identifier` sets the public-facing AS number.
- `bgp confederation peers` lists the internal sub-AS numbers.

Example:

```cisco
router bgp 65000
 bgp confederation identifier 65000
 bgp confederation peers 65001 65002
```

Important behavior:
- Inside the confederation, routers exchange routes between sub-ASes using eBGP-like sessions.
- The AS_PATH still carries sub-AS information internally, often represented in parentheses.
- To external peers, the confederation appears as one AS, but path attributes are not magically erased; the internal structure is just abstracted from normal external policy decisions.

When to prefer confederations:
- You need stronger administrative separation than route reflection alone provides.
- You want to divide a very large AS into policy domains.
- You need eBGP-like behavior internally for path selection or policy control.

When to prefer route reflectors:
- You mainly need scale reduction.
- You want simpler operational behavior.
- You do not need sub-AS policy boundaries.

In many enterprise designs, route reflectors are simpler and more common than confederations.

**Address-family design, including IPv6**

Modern BGP design is not limited to IPv4 unicast. Cisco platforms commonly use Multiprotocol BGP (MP-BGP) to activate address families such as IPv4 unicast and IPv6 unicast under the same neighbor.

For IPv6, you usually:
- Configure IPv6 addressing on interfaces or loopbacks.
- Ensure IPv6 reachability in the underlay.
- Activate the IPv6 unicast address family for the neighbor.

Example:

```cisco
ipv6 unicast-routing

interface Loopback0
 ipv6 address 2001:db8:10::1/128

router bgp 65001
 neighbor 2001:db8:10::2 remote-as 65001
 neighbor 2001:db8:10::2 update-source Loopback0
 address-family ipv6 unicast
  neighbor 2001:db8:10::2 activate
  network 2001:db8:100::/64
```

Design notes:
- IPv4 and IPv6 can use the same BGP process and neighbor relationship, but each address family must be activated as needed.
- Next-hop reachability must exist for the specific address family.
- Policy can differ by address family, so do not assume IPv4 settings automatically apply to IPv6.

**Path selection and best-path behavior**

BGP best-path selection is attribute-driven, but the exact order can vary by platform, release, and feature set. Do not memorize a simplified list as if it were universal. Instead, understand the major attributes and their intent.

Common attributes that influence selection:
- **Weight** on Cisco devices, which is local to the router
- **Local preference**, which influences outbound path choice within the AS
- **AS_PATH** length, which often prefers shorter external paths
- **Origin** type
- **MED** (Multi-Exit Discriminator), which can influence inbound path choice from a neighboring AS
- **eBGP over iBGP** preference in many cases
- **IGP metric to next hop**
- **Router ID** and tie-breakers

Practical interpretation:
- Use **weight** for local, router-specific preference.
- Use **local preference** for AS-wide outbound policy.
- Use **AS_PATH prepending** or provider communities for inbound traffic engineering.
- Use **MED** when you want to suggest an entry point to a neighboring AS, but remember that many providers only compare MED under specific conditions.

Example local preference policy:

```cisco
route-map SET-HIGH-LP permit 10
 set local-preference 200

router bgp 65001
 address-family ipv4 unicast
  neighbor 10.255.0.2 route-map SET-HIGH-LP in
```

**Traffic engineering**

BGP is often used to influence how traffic enters and leaves your network.

Outbound traffic engineering:
- Use local preference to choose the preferred exit from your AS.
- Use policy on iBGP routes so internal routers agree on the exit.
- Prefer local preference over AS_PATH manipulation for internal consistency.

Inbound traffic engineering:
- Use AS_PATH prepending to make a path less attractive.
- Use MED when the upstream honors it and the design supports it.
- Use provider communities to request actions such as local preference changes, blackholing, or regional preference.

Example AS_PATH prepending:

```cisco
route-map PREPEND-OUT permit 10
 set as-path prepend 65001 65001 65001

router bgp 65001
 address-family ipv4 unicast
  neighbor 203.0.113.1 route-map PREPEND-OUT out
```

Community-based upstream signaling is often more effective than raw attribute manipulation when the provider documents supported communities.

**Security design**

BGP security is about protecting the session and limiting abuse.

Common controls:
- **Prefix filtering** to accept only expected routes
- **Max-prefix** limits to protect against route leaks
- **Authentication** to reduce unauthorized session establishment
- **TTL security** to protect directly connected or carefully designed multihop sessions
- **Control-plane policing** to protect the router CPU
- **BFD** (Bidirectional Forwarding Detection) when you need faster failure detection and the platform supports it

Example max-prefix and authentication:

```cisco
router bgp 65001
 neighbor 203.0.113.1 password 7 030752180500
 address-family ipv4 unicast
  neighbor 203.0.113.1 maximum-prefix 100000 90 restart 5
```

Notes on security:
- Type 7 passwords are legacy obfuscation, not strong encryption. Prefer stronger platform-supported authentication options where available.
- TTL security is most straightforward on directly connected eBGP peers, but it can also be used in some multihop designs with careful hop-count planning.
- BFD is useful when you need faster failure detection than BGP timers alone, but it adds operational complexity and should be validated with the underlay and platform support.

**Timers and convergence**

BGP keepalive and hold timers affect how quickly a neighbor failure is detected. Default values are often conservative, which is good for stability but slower for convergence.

Typical design considerations:
- Lower timers can improve failure detection but increase control-plane load.
- Keep timers consistent with the transport quality and platform capability.
- Use BFD when you need fast detection without making BGP timers overly aggressive.

Example:

```cisco
router bgp 65001
 neighbor 203.0.113.1 timers 10 30
```

Use aggressive timers only when the path is stable and the platform can handle the churn. In unstable WANs, overly low timers can create unnecessary flaps.

**Troubleshooting workflow**

When BGP fails, work from transport to policy to installation.

1. Verify IP reachability to the neighbor or loopback.
2. Check that the remote AS and source address are correct.
3. Confirm TCP/179 connectivity and any ACL or firewall policy.
4. Verify the BGP session state with `show ip bgp summary`.
5. Check whether routes are received, selected, and installed.
6. Inspect next-hop reachability in the routing table.
7. Review route maps, prefix lists, communities, and maximum-prefix settings.
8. Confirm that the correct address family is activated.

Useful commands:

```cisco
show ip bgp summary
show ip bgp neighbors 203.0.113.1
show ip bgp
show ip route 203.0.113.0
show bgp ipv6 unicast summary
show bgp ipv6 unicast neighbors 2001:db8:10::2
```

Be careful with platform-dependent commands:
- Route-refresh support may allow you to request updates without clearing the session.
- Some platforms support `show ip bgp neighbors <neighbor> routes` and `advertised-routes`.
- `received-routes` may require soft reconfiguration inbound on classic IOS or may not be available in the same form on all platforms.

Selective clearing is often safer than a full reset:

```cisco
clear ip bgp 203.0.113.1 soft in
clear ip bgp 203.0.113.1 soft out
```

Use a hard clear only when necessary, because it disrupts traffic and can trigger reconvergence.

**Common exam and lab pitfalls**

- **Missing underlay reachability** for loopback peering
- **Forgetting address-family activation** for IPv4 or IPv6
- **Assuming iBGP will re-advertise routes** without route reflectors or confederations
- **Misunderstanding next-hop behavior** and recursive resolution
- **Using the wrong command for default origination**
- **Overlooking policy order** in route maps and prefix lists
- **Applying security controls without testing platform support**
- **Treating simplified best-path order as universal**

**Quick design checklist**

- Use eBGP at the edge and iBGP inside the AS.
- Prefer loopback peering for resilience, but provide underlay reachability.
- Add route reflectors when full-mesh iBGP does not scale.
- Consider confederations when you need sub-AS structure and policy separation.
- Activate the correct address families, including IPv6.
- Design inbound and outbound traffic engineering separately.
- Protect sessions with filtering, authentication, TTL security, and max-prefix limits.
- Validate next-hop reachability and route installation, not just neighbor state.

**Practical verification examples**

```cisco
show ip bgp summary
show ip bgp neighbors 203.0.113.1 advertised-routes
show ip route bgp
show ip bgp 198.51.100.0
show bgp ipv6 unicast summary
show bgp ipv6 unicast neighbors 2001:db8:10::2 advertised-routes
```

If a route is present in the BGP table but not in the routing table, check:
- Next-hop reachability
- Better path from another protocol
- Policy that suppresses installation
- Address-family mismatch
- Route filtering or maximum-prefix behavior

**Summary**

eBGP and iBGP design is about more than session formation. You need to plan underlay reachability, next-hop behavior, scaling mechanisms, policy control, and security together. eBGP gives you clean boundaries at the edge, while iBGP distributes routes inside the AS. Route reflectors and confederations solve scale problems in different ways, and IPv6 requires explicit address-family design. If you understand how BGP sessions form, how routes are selected, and how next hops are resolved, you can troubleshoot most enterprise BGP issues quickly and design networks that converge predictably.
