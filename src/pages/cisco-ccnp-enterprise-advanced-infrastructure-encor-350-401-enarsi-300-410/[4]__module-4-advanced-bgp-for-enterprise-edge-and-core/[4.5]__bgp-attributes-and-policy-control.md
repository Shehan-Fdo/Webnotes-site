## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**BGP policy model**

Border Gateway Protocol (BGP) is a path-vector protocol. You do not usually influence forwarding by changing the destination prefix itself; you influence which path is preferred by manipulating attributes attached to routes. Those attributes can be set locally, learned from neighbors, or carried across the AS path depending on the attribute type.

BGP policy is typically applied in three places:

- **Inbound** from a neighbor, to influence how you select routes.
- **Outbound** toward a neighbor, to influence what you advertise.
- **Redistribution or injection** into BGP, to control which routes enter BGP in the first place.

A useful mental model is:

1. Receive a route.
2. Apply inbound policy.
3. Compare attributes against other paths.
4. Install the best path.
5. Apply outbound policy before advertisement.

> In BGP, a route-map `deny` sequence does not always mean “drop the route from the routing table.” It means “stop processing this sequence.” Whether the route is rejected depends on where the route-map is applied and whether a later sequence matches.

**Core BGP attributes**

BGP attributes are the metadata that drive path selection and policy. Some are well-known and mandatory, some are well-known and discretionary, and some are optional transitive or optional non-transitive.

<dl>
<dt><strong>Weight</strong></dt>
<dd>Cisco-local attribute only. Highest weight wins. It is not advertised to neighbors.</dd>
<dt><strong>Local Preference</strong></dt>
<dd>AS-wide attribute used inside one autonomous system (AS). Higher local preference wins.</dd>
<dt><strong>AS_PATH</strong></dt>
<dd>List of AS numbers a route has traversed. Shorter path is usually preferred. Also used for loop prevention.</dd>
<dt><strong>Origin</strong></dt>
<dd>Indicates how the route entered BGP: IGP, EGP, or incomplete. Lower is better.</dd>
<dt><strong>MED</strong></dt>
<dd>Multi-Exit Discriminator. Lower MED is preferred when compared. Used to suggest entry point into an AS.</dd>
<dt><strong>Next hop</strong></dt>
<dd>IP address used to reach the BGP prefix. It must be reachable in the IGP or via a recursive route.</dd>
<dt><strong>Community</strong></dt>
<dd>Tag used for policy signaling. Communities are matched by policy and can be standard, extended, or large.</dd>
</dl>

**Best-path selection order**

Cisco BGP best-path selection is often tested as a sequence. Exact behavior can vary slightly by platform and release, but the major order is consistent enough for exam prep.

A practical exam-focused order is:

1. Highest **weight**
2. Highest **local preference**
3. Locally originated route
4. Shortest **AS_PATH**
5. Lowest **origin** type
6. Lowest **MED** when comparable
7. eBGP over iBGP
8. Lowest IGP cost to next hop
9. Oldest path
10. Lowest BGP router ID
11. Lowest neighbor IP address

The important exam point is not just memorizing the list, but understanding where each attribute is meaningful:

- **Weight** affects only the local router.
- **Local preference** affects all routers in the AS if it is propagated.
- **AS_PATH** is visible across AS boundaries and is a major external selection signal.
- **MED** is a hint to a neighboring AS and is often compared only under specific conditions.

**Attribute comparison summary**

| Attribute | Scope | Default behavior | Typical use case |
|---|---|---:|---|
| Weight | Local router only | 0 | Force one exit path on a single router |
| Local preference | Entire AS | 100 | Prefer one egress point across the AS |
| AS_PATH | Inter-AS | Learned path length | Prefer shorter external path; prepend to de-prefer |
| Origin | Inter-AS | Depends on injection method | Minor tie-breaker in best-path selection |
| MED | Neighboring AS relationship | 0 or unset depending on platform/route | Suggest preferred ingress point |
| Next hop | Forwarding reachability | Must be resolvable | Ensure recursive forwarding works |
| Community | Policy tag | None | Mark routes for later filtering or action |

**Weight and local preference**

Weight is Cisco-specific and is one of the fastest ways to influence a single router’s decision. Because it is not advertised, it is ideal when you want a local preference without changing the rest of the AS.

Local preference is the standard tool for choosing an exit path across an AS. You usually set it inbound on edge routers so the value propagates through iBGP.

Example: prefer routes learned from ISP-A across the AS.

```cisco
route-map SET-LP-ISP-A permit 10
 set local-preference 200

router bgp 65000
 neighbor 203.0.113.1 route-map SET-LP-ISP-A in
```

Example: set weight on one router only.

```cisco
route-map SET-WEIGHT permit 10
 set weight 500

router bgp 65000
 neighbor 203.0.113.1 route-map SET-WEIGHT in
```

Key exam distinction:

- Use **weight** when only one router should prefer the path.
- Use **local preference** when the whole AS should prefer the path.

**AS_PATH and origin**

AS_PATH is both a policy attribute and a loop-prevention mechanism. A route with a shorter AS_PATH is usually preferred over a longer one. You can intentionally lengthen the path with AS-path prepending to make a route less attractive.

Origin indicates how BGP learned the route:

- **IGP**: originated by `network` statement or equivalent controlled injection.
- **EGP**: historical value, rarely seen in modern networks.
- **incomplete**: learned by redistribution or other non-originated means.

Lower origin is preferred, so the order is:

- `i` = IGP
- `e` = EGP
- `?` = incomplete

Example: prepend your AS to make an exit less attractive.

```cisco
route-map PREPEND-OUT permit 10
 set as-path prepend 65000 65000 65000

router bgp 65000
 neighbor 198.51.100.2 route-map PREPEND-OUT out
```

Use AS-path prepending when you want to influence inbound traffic from another AS. It is a hint, not a guarantee, because the remote AS can still override it with local policy.

**MED behavior and comparison nuances**

MED is a lower-is-better attribute used to suggest which entry point another AS should use when multiple links exist. It is commonly compared only among routes received from the same neighboring AS unless you enable broader comparison.

Important nuances:

- By default, many Cisco implementations compare MED only when the routes come from the same neighboring AS.
- `bgp always-compare-med` forces MED comparison across different neighboring ASes.
- `bgp deterministic-med` makes MED comparison more consistent by grouping paths before selection, which helps avoid path oscillation or surprising results.

Example:

```cisco
router bgp 65000
 bgp deterministic-med
 bgp always-compare-med
```

Example: set MED outbound toward a neighbor.

```cisco
route-map SET-MED permit 10
 set metric 50

router bgp 65000
 neighbor 198.51.100.2 route-map SET-MED out
```

Use MED when you want to influence how a neighboring AS enters your network, especially when you have multiple links to the same provider. Do not rely on MED as a universal inter-AS policy tool, because many networks ignore it or override it.

**Next-hop control and reachability**

Next-hop reachability is a common source of BGP problems. A route can be received successfully but still not be usable if the next hop is unreachable in the local routing table.

Typical rules:

- In **eBGP**, the next hop is usually changed to the neighbor address.
- In **iBGP**, the next hop is usually preserved.
- If iBGP peers are not directly connected through IGP reachability, you often need `next-hop-self`.

Example:

```cisco
router bgp 65000
 neighbor 10.0.0.2 remote-as 65000
 neighbor 10.0.0.2 next-hop-self
```

Use `next-hop-self` when:

- You advertise external routes into iBGP and want internal routers to reach the advertising router instead of the original external next hop.
- You have a hub-and-spoke design and want the hub to become the reachable next hop for spokes.

Be careful with route reflection and multi-hop designs. If the next hop is not reachable, the route may appear in BGP but not be installed in the forwarding table.

**Communities and community policy**

Communities are tags used to mark routes for later policy decisions. They do not directly change forwarding by themselves; they become useful when matched by route-maps, prefix-lists, or community-lists.

Common well-known communities:

- `no-export`: do not advertise beyond the local AS confederation boundary or to eBGP peers, depending on context.
- `no-advertise`: do not advertise to any peer.
- `internet`: advertise normally.

Community types:

- **Standard community**: 32-bit value written as `ASN:VALUE`, such as `65000:100`.
- **Extended community**: larger format used for VPN and advanced policy signaling.
- **Large community**: 96-bit format written as `ASN:VALUE:VALUE`, useful when standard communities are too small.

Example: tag routes with a standard community.

```cisco
route-map TAG-ROUTES permit 10
 set community 65000:100 additive

router bgp 65000
 neighbor 192.0.2.2 route-map TAG-ROUTES out
```

The `additive` keyword preserves existing communities. Without it, the new community list typically replaces the old one.

Example: remove a community before advertising.

```cisco
route-map CLEANUP permit 10
 set comm-list 10 delete

ip community-list standard 10 permit 65000:100

router bgp 65000
 neighbor 192.0.2.2 route-map CLEANUP out
```

Platform syntax varies for sending communities. On Cisco IOS XE, you often need to explicitly enable community propagation to a neighbor, commonly with one of these forms:

```cisco
router bgp 65000
 neighbor 192.0.2.2 send-community
 neighbor 192.0.2.2 send-community extended
 neighbor 192.0.2.2 send-community both
```

Use the syntax supported by your platform and release. For exam purposes, remember the concept: communities are not always sent unless you enable them.

**Route-map behavior in BGP policy**

Route-maps are the main policy tool for BGP. They combine match conditions and set actions.

General flow:

- A route enters the route-map at sequence 10, 20, 30, and so on.
- If the route matches the sequence, the `set` actions run.
- If the sequence is `permit`, the route is allowed to continue.
- If the sequence is `deny`, the route is rejected at that policy point.
- If no sequence matches, the route is implicitly denied.

This is critical in BGP because route-map behavior differs depending on direction:

- **Inbound route-map**: unmatched routes are denied from being accepted by that policy.
- **Outbound route-map**: unmatched routes are not advertised.
- A `permit` sequence with no `set` action still permits the route.
- A `deny` sequence with a match condition blocks only the matched routes.

Example: block the default route but permit all other prefixes.

```cisco
ip prefix-list BLOCK-DEFAULT seq 5 deny 0.0.0.0/0
ip prefix-list BLOCK-DEFAULT seq 10 permit 0.0.0.0/0 le 32

route-map OUT-FILTER deny 10
 match ip address prefix-list BLOCK-DEFAULT

route-map OUT-FILTER permit 20

router bgp 65000
 neighbor 192.0.2.2 route-map OUT-FILTER out
```

The second sequence is important: it permits all routes that do not match the default-route deny sequence. Without it, the implicit deny would suppress all outbound advertisements.

**Network statements, redistribution, and route injection**

BGP does not automatically advertise every route in the routing table. You must inject routes into BGP using `network` statements, redistribution, or policy-based mechanisms.

Common methods:

- `network` statement: advertises a route only if it exists in the local routing table with an exact match.
- `redistribute`: injects routes from another protocol into BGP.
- Route-map with redistribution: filters or tags redistributed routes.

Example: originate a prefix only if it exists in the RIB.

```cisco
router bgp 65000
 network 203.0.113.0 mask 255.255.255.0
```

Example: redistribute static routes into BGP with policy.

```cisco
ip prefix-list REDIST-STATIC seq 10 permit 203.0.113.0/24

route-map REDIST-TO-BGP permit 10
 match ip address prefix-list REDIST-STATIC
 set community 65000:200 additive

router bgp 65000
 redistribute static route-map REDIST-TO-BGP
```

Exam point: `network` is precise and controlled. Redistribution is broader and can accidentally leak routes if you do not filter carefully.

**Practical policy examples**

Prefer one upstream for outbound traffic:

```cisco
route-map PREFER-UPSTREAM-A permit 10
 set local-preference 200

route-map PREFER-UPSTREAM-B permit 10
 set local-preference 100

router bgp 65000
 neighbor 203.0.113.1 route-map PREFER-UPSTREAM-A in
 neighbor 203.0.113.2 route-map PREFER-UPSTREAM-B in
```

De-prefer one exit by prepending on outbound advertisements:

```cisco
route-map DEPREFER-OUT permit 10
 set as-path prepend 65000 65000

router bgp 65000
 neighbor 198.51.100.2 route-map DEPREFER-OUT out
```

Set a lower MED for a preferred ingress link:

```cisco
route-map SET-LOW-MED permit 10
 set metric 10

router bgp 65000
 neighbor 198.51.100.2 route-map SET-LOW-MED out
```

Ensure internal routers can reach external next hops:

```cisco
router bgp 65000
 neighbor 10.0.0.2 remote-as 65000
 neighbor 10.0.0.2 next-hop-self
```

**Conditional advertisement**

Conditional advertisement lets you advertise a route only when another route exists or does not exist. This is useful for backup links, failover, and selective announcements.

Key terms:

- **advertise-map**: routes to advertise conditionally.
- **exist-map**: routes that must exist for advertisement to occur.
- **non-exist-map**: routes that must not exist for advertisement to occur.

Typical use case:

- Advertise a backup prefix only if the primary path is unavailable.
- Suppress a more specific route unless a tracking prefix disappears.

Example concept:

```cisco
router bgp 65000
 neighbor 192.0.2.2 advertise-map ADV-MAP exist-map PRIMARY-MAP
```

Interpretation:

- If the routes in `PRIMARY-MAP` exist, advertise the routes in `ADV-MAP`.
- If they do not exist, suppress them.

This feature is more advanced than basic route-maps and is often used in edge failover designs.

**Common exam traps**

- **Route-map deny does not always mean total rejection**; it only rejects matched routes at that sequence.
- **Local preference** affects only your AS, not the upstream neighbor.
- **Weight** is local to one router and is not shared.
- **MED** is not always compared across all paths unless you enable it.
- **Next-hop-self** is often required in iBGP to avoid unreachable next hops.
- **Communities** are policy tags, not forwarding instructions by themselves.
- **AS-path prepending** influences inbound traffic but does not guarantee it.
- **`network` statements** require an exact route in the local table.
- **Redistribution** can leak routes if you do not filter with a route-map.

**Troubleshooting commands**

Use these commands to inspect BGP policy and path selection:

```cisco
show ip bgp
show ip bgp summary
show ip bgp <prefix>
show ip bgp neighbors
show ip bgp neighbors <neighbor-ip> advertised-routes
show ip bgp neighbors <neighbor-ip> routes
show ip route <prefix>
show ip protocols
```

Useful checks:

- Confirm the route is present in BGP but not in the routing table.
- Verify the next hop is reachable.
- Compare attributes across multiple paths.
- Check whether a route-map or prefix-list is filtering the route.
- Confirm communities are actually being sent and matched.

For community troubleshooting, inspect route detail or community-lists rather than relying on a single universal command, because syntax varies by platform and release.

**Quick exam workflow**

When you see a BGP policy question, work through it in this order:

1. Identify whether the policy is **inbound** or **outbound**.
2. Determine whether the goal is local preference, weight, MED, AS-path length, or filtering.
3. Check whether the attribute is local-only or propagated.
4. Confirm whether the route is injected by `network`, redistribution, or learned from a neighbor.
5. Verify next-hop reachability.
6. Check for implicit deny in route-maps and prefix-lists.
7. Confirm whether communities are being sent and matched correctly.

**Scenario: dual-homed enterprise edge**

A common design has two ISPs and one internal AS.

- ISP-A should be the preferred outbound path.
- ISP-B should be backup.
- Inbound traffic should prefer ISP-A for the primary prefix.
- Internal routers should always reach external next hops.

A typical policy set is:

- Set higher **local preference** on routes learned from ISP-A.
- Set lower **local preference** on routes learned from ISP-B.
- Use **AS-path prepending** on ISP-B advertisements for inbound de-preference.
- Use **next-hop-self** on iBGP peers.
- Use **communities** to tag routes for special handling or filtering.

Expected behavior:

- Before failure, internal routers choose ISP-A because local preference is higher.
- If ISP-A fails, routes from ISP-B remain available and become best.
- Remote networks may prefer ISP-A because ISP-B advertisements are prepended.
- If next-hop-self is missing in iBGP, internal routers may see the route but fail to forward traffic correctly.

**Minimal configuration pattern**

```cisco
router bgp 65000
 bgp log-neighbor-changes
 neighbor 10.0.0.2 remote-as 65000
 neighbor 10.0.0.2 next-hop-self
 neighbor 203.0.113.1 remote-as 64501
 neighbor 198.51.100.1 remote-as 64502

route-map ISP-A-IN permit 10
 set local-preference 200

route-map ISP-B-IN permit 10
 set local-preference 100

route-map ISP-B-OUT permit 10
 set as-path prepend 65000 65000

router bgp 65000
 neighbor 203.0.113.1 route-map ISP-A-IN in
 neighbor 198.51.100.1 route-map ISP-B-IN in
 neighbor 198.51.100.1 route-map ISP-B-OUT out
```

**What to remember for the exam**

- **Weight** is Cisco-local and wins first.
- **Local preference** is the main AS-wide exit-selection tool.
- **AS_PATH** length and **origin** are major path-selection attributes.
- **MED** is a lower-is-better hint and has comparison caveats.
- **Next-hop-self** solves many iBGP reachability issues.
- **Communities** are the standard way to tag routes for later policy.
- **Route-maps** control both filtering and attribute manipulation.
- **Conditional advertisement** is useful for advanced failover logic.
