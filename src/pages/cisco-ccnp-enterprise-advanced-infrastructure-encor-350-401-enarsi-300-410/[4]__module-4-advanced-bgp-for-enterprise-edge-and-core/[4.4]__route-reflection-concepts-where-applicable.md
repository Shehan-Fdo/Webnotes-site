## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Route reflection overview**

Border Gateway Protocol (BGP) iBGP (internal BGP) has a built-in rule: routes learned from one iBGP peer are not advertised to another iBGP peer. This prevents loops, but it also creates a full-mesh requirement if you want every router to learn every internal route. Route reflection removes that scaling problem by allowing a designated router, the **route reflector (RR)**, to re-advertise iBGP-learned routes to other iBGP peers.

Route reflection is most useful in larger enterprise or service-provider-style topologies where a full iBGP mesh becomes operationally expensive. It is not a replacement for good design; it is a scaling tool that trades simplicity of full mesh for reflector-based hierarchy.

- **iBGP full mesh** requires every iBGP speaker to peer with every other iBGP speaker.
- **Route reflector** allows clients to avoid full mesh while still learning internal BGP routes.
- **Cluster design** matters because reflection introduces new loop-prevention rules.

**Why route reflection exists**

Without route reflection, iBGP has two major scaling issues:

- The number of sessions grows quickly as routers are added.
- Operational overhead increases because every new router requires many new peering relationships.

For example, a full mesh of 10 routers requires 45 iBGP sessions. At 20 routers, it requires 190 sessions. Route reflection reduces this to a hub-and-spoke-like control plane, where clients peer only with one or more reflectors.

**Core route reflection roles**

A route reflector has two types of iBGP neighbors:

- **Clients**: routers that rely on the RR to propagate routes.
- **Non-clients**: iBGP peers that are not RR clients, often other reflectors or core routers.

The RR applies special advertisement rules:

- Routes learned from a client can be advertised to:
  - other clients
  - non-clients
  - eBGP peers
- Routes learned from a non-client can be advertised to:
  - clients
  - eBGP peers
- Routes learned from one non-client are not reflected to another non-client.

This behavior preserves loop prevention while allowing route propagation.

**Loop prevention mechanisms**

Route reflection introduces two key attributes used to prevent loops:

- **ORIGINATOR_ID**: identifies the original iBGP router that injected the route into the RR domain.
- **CLUSTER_LIST**: records the route reflector cluster path.

If a router sees its own router ID in the ORIGINATOR_ID, it rejects the route. If a route reflector sees its own cluster ID in the CLUSTER_LIST, it rejects the route.

These attributes are added automatically by the RR and are critical to safe operation.

> A route reflector does not remove the need for careful design. Poorly planned reflection can still create suboptimal paths, hidden reachability issues, or unexpected best-path choices.

**Typical topology patterns**

A common enterprise design uses:

- Two route reflectors for redundancy
- Multiple clients in access, distribution, or core layers
- iBGP between reflectors as non-client peering
- eBGP at the edge for external connectivity

This design avoids a single point of failure and reduces the risk of route loss if one reflector fails.

**Example topology behavior**

Consider three routers:

- `R1` and `R2` are clients
- `RR1` is the route reflector

If `R1` advertises a prefix to `RR1`, the RR can reflect it to `R2` even though `R1` and `R2` are not directly peered. If `RR1` also peers with `RR2` as a non-client, route propagation between reflectors depends on the reflection rules and cluster design.

**Cisco IOS XE configuration example**

A basic route reflector configuration uses the `route-reflector-client` command under the BGP neighbor.

```bash
router bgp 65000
 bgp router-id 1.1.1.1
 neighbor 10.0.0.2 remote-as 65000
 neighbor 10.0.0.2 route-reflector-client
 neighbor 10.0.0.3 remote-as 65000
 neighbor 10.0.0.3 route-reflector-client
 neighbor 10.0.0.4 remote-as 65000
```

In this example:

- `10.0.0.2` and `10.0.0.3` are clients
- `10.0.0.4` is a non-client iBGP peer

The RR reflects client-learned routes to both clients and non-clients, but it does not reflect non-client-learned routes to other non-clients.

**Cluster ID and router ID considerations**

Each route reflector belongs to a **cluster**. If you do not configure a cluster ID, Cisco IOS XE typically uses the router ID as the cluster ID. In multi-reflector designs, you should understand how cluster IDs affect loop prevention and route acceptance.

Use a unique cluster ID when:

- You want to explicitly define reflector domains
- You have multiple reflectors in the same logical cluster
- You need predictable behavior across redundant reflectors

Example:

```bash
router bgp 65000
 bgp cluster-id 100
 neighbor 10.0.0.2 remote-as 65000
 neighbor 10.0.0.2 route-reflector-client
```

**Best-path selection and path visibility**

Route reflection can influence which path a router chooses because clients may not see all possible iBGP paths. This can lead to:

- **Suboptimal routing** if the RR hides a better path from some clients
- **Path asymmetry** if different clients receive different reflected paths
- **Convergence differences** because the RR becomes a control-plane dependency

To reduce these issues:

- Deploy multiple reflectors
- Place reflectors centrally in the topology
- Ensure consistent policy across reflectors
- Consider add-path or other enhancements where supported and needed

**Operational commands**

Use these commands to verify route reflection behavior on Cisco devices:

```bash
show ip bgp summary
show ip bgp
show ip bgp neighbors
show ip bgp neighbors 10.0.0.2 advertised-routes
show ip bgp neighbors 10.0.0.2 routes
show ip bgp 192.0.2.0
```

Useful things to check:

- Neighbor state is `Established`
- Routes are present in the BGP table
- Reflected routes carry `ORIGINATOR_ID` and `CLUSTER_LIST`
- The expected prefixes are advertised to clients

For IPv6, use the equivalent commands:

```bash
show bgp ipv6 unicast summary
show bgp ipv6 unicast
show bgp ipv6 unicast neighbors
```

**Troubleshooting route reflection**

When routes do not appear where expected, check the following:

1. Verify the iBGP session is up and in the correct address family.
2. Confirm the neighbor is configured as a `route-reflector-client` on the RR.
3. Check whether route policy is filtering the prefix.
4. Inspect `ORIGINATOR_ID` and `CLUSTER_LIST` for loop-prevention drops.
5. Confirm next-hop reachability inside the IGP or via static routes.
6. Validate that the route is actually best on the RR before expecting reflection.

Common failure causes include:

- Missing `route-reflector-client`
- Incorrect `remote-as`
- Next-hop unreachable from clients
- Route-map or prefix-list filtering
- Conflicting cluster IDs in a multi-RR design

**Design guidance**

Route reflection works best when you treat reflectors as infrastructure devices, not as ordinary transit routers.

- Place reflectors where they have stable reachability to clients.
- Use at least two reflectors for redundancy in production designs.
- Keep cluster design simple unless the topology requires hierarchy.
- Avoid unnecessary policy differences between reflectors.
- Document which routers are clients and which are non-clients.

A hierarchical design may be appropriate in very large networks, but each additional layer increases complexity and can make troubleshooting harder.

**Route reflection vs full mesh**

| Aspect | Full iBGP mesh | Route reflection |
|---|---|---|
| Session count | Grows rapidly | Scales better |
| Configuration effort | High | Lower |
| Path visibility | Complete | Potentially partial |
| Loop prevention | Simpler | Uses ORIGINATOR_ID and CLUSTER_LIST |
| Troubleshooting | Easier | More complex |

Full mesh is still attractive in small topologies because it is straightforward and gives every router full path visibility. Route reflection is the practical choice when scale matters.

**Practical verification example**

A client receiving a reflected route may show attributes similar to the following:

```bash
R2# show ip bgp 203.0.113.0
BGP routing table entry for 203.0.113.0/24, version 12
Paths: (1 available, best #1, table default)
  Not advertised to any peer
  65001
    10.0.0.1 from 10.0.0.1 (1.1.1.1)
      Origin IGP, metric 0, localpref 100, valid, internal, best
      Originator: 2.2.2.2, Cluster list: 1.1.1.1
```

Interpretation:

- `10.0.0.1` is the RR
- `Originator` identifies the original router
- `Cluster list` shows the reflector path
- The route is valid and best on the receiving client

**Key takeaways**

- Route reflection solves the iBGP full-mesh scaling problem.
- Clients rely on reflectors to propagate internal routes.
- `ORIGINATOR_ID` and `CLUSTER_LIST` prevent reflection loops.
- Multiple reflectors improve resilience and reduce control-plane dependency.
- Route reflection simplifies scaling but can hide paths and complicate troubleshooting.
