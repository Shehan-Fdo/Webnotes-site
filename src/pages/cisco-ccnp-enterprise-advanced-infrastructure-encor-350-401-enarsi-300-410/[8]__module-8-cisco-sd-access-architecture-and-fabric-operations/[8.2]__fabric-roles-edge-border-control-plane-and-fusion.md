## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Fabric role overview**

Cisco SD-Access uses a small set of fabric roles to separate responsibilities inside the overlay. The main roles are **edge**, **border**, and **control plane**. A related design concept, **fusion**, is used to connect fabric VRFs to external or shared-services routing domains, but it is not a standard fabric node role.

At a high level:

- **Edge nodes** attach endpoints to the fabric and encapsulate traffic into the overlay.
- **Border nodes** connect the fabric to nonfabric networks or other fabric domains.
- **Control-plane nodes** maintain endpoint-to-location mappings for the fabric using LISP.
- **Fusion** is a handoff design that links fabric routing to external routing, often through a border or dedicated external device.

**Role prerequisites and platform support**

Not every Catalyst platform can perform every fabric role. In exam scenarios, always check platform capability, software support, and design intent before assigning a role.

- **Edge node support** is common on access-layer Catalyst switches that support SD-Access fabric features.
- **Border node support** is typically on higher-capacity Catalyst switches or routers that can handle route exchange and external connectivity.
- **Control-plane node support** requires a platform and release that support the LISP control-plane function at scale.
- **Wireless integration** usually involves fabric-capable wireless controllers and APs that map wireless clients into the fabric as endpoints.

> Do not assume a device can be any fabric role just because it runs IOS XE. Role support depends on model, license, and release.

**SD-Access control-plane mechanics**

The control-plane node is the LISP <abbr title="Locator/ID Separation Protocol">LISP</abbr> mapping function for the fabric. It is not a generic registration authority; its job is to maintain **EID-to-RLOC** mappings.

- **EID** means Endpoint Identifier, the identity of the endpoint.
- **RLOC** means Routing Locator, the fabric location used to reach that endpoint.
- **Map-Server** receives endpoint registrations from edge nodes.
- **Map-Resolver** answers mapping requests when a node needs to learn where an endpoint lives.
- **Map-cache** on fabric nodes stores recently learned mappings to reduce repeated lookups.

A typical flow looks like this:

1. An endpoint connects to an edge node.
2. The edge node learns the endpoint identity and registers the mapping to the control plane.
3. Another fabric node needs to reach that endpoint.
4. The node queries the map-resolver function.
5. The control plane returns the current RLOC for the destination EID.
6. The source node caches the result in its map-cache.

This separation matters because the control plane does not forward user traffic. It only helps fabric nodes locate endpoints efficiently.

**Edge node behavior**

Edge nodes are the fabric attachment points for wired and wireless endpoints. They classify traffic, apply policy, and encapsulate packets into the overlay toward the destination fabric node.

Common edge responsibilities include:

- **Endpoint onboarding** for wired hosts, printers, phones, and wireless clients.
- **VXLAN** encapsulation and decapsulation for fabric traffic.
- **LISP registration** of local endpoint mappings to the control plane.
- **Policy enforcement** using group-based policy and segmentation constructs.
- **Anycast gateway** support so endpoints can keep the same default gateway across fabric nodes.

Edge nodes are often the first place where endpoint identity is learned. They may learn identity from MAC address, IP address, VLAN, or policy context depending on the design.

**Wired and wireless edge considerations**

Wireless endpoints are usually integrated through a fabric-capable wireless design rather than being treated as ordinary wired hosts.

- Wireless clients can be mapped into the fabric through a wireless controller that participates in the fabric design.
- APs may connect to an edge node and tunnel client traffic through the wireless control plane before it enters the fabric overlay.
- The fabric edge still matters because it provides the fabric attachment and policy enforcement point for the client traffic.

In exam questions, distinguish between:

- **AP transport**: how the access point reaches the network.
- **Client fabric attachment**: how the wireless client is represented inside the fabric.
- **Policy enforcement**: where segmentation and access control are applied.

**Border node behavior**

Border nodes connect the fabric to external routing domains, shared services, or other fabrics. They are usually discussed as **L3 borders** because route exchange is the common exam-relevant behavior.

Border nodes can be:

- **External border**: connects the fabric to an outside enterprise WAN, data center, Internet edge, or another routing domain.
- **Internal border**: connects one fabric site or fabric domain to another internal routing boundary or shared-services domain.

Border behavior depends on the design:

- Some borders redistribute or leak routes between the fabric and external routing domains.
- Some borders participate in route exchange with specific protocols such as OSPF, EIGRP, or BGP, depending on the design.
- Some borders are used to connect to shared services without making those services part of the fabric overlay.

> Avoid assuming a border always redistributes all routes. The exact behavior depends on whether the border is external or internal, and on the chosen routing design.

**Border subtypes and exam distinctions**

| Border type | Typical purpose | Exam focus |
|---|---|---|
| External border | Connects fabric to nonfabric routing domains | Route exchange, default route handling, external reachability |
| Internal border | Connects fabric to internal enterprise domains or another fabric boundary | Segmentation and controlled route exchange |
| L2 border | Limited, design-specific Layer 2 extension behavior | Less common; treat as special-case design, not the default assumption |

For most exam scenarios, think of border nodes as the fabric’s Layer 3 handoff point. If a question suggests a Layer 2 border, read carefully and verify the design intent.

**Fusion design pattern**

Fusion is not a fabric node role. It is a design pattern used to connect fabric VRFs to external routing or shared-services domains in a controlled way.

Use fusion when you need to:

- Connect fabric segments to a shared-services network.
- Keep the fabric overlay separate from external routing complexity.
- Provide a clean handoff between fabric VRFs and nonfabric routing domains.

Fusion is often implemented:

- On a border node in smaller designs.
- On a dedicated external router or firewall in larger designs.
- With route leaking or redistribution policies that are intentionally limited.

A simple fusion topology might look like this:

```text
Fabric Edge Nodes
        |
   Fabric Overlay
        |
   Fabric Border
        |
   Fusion Device / Shared Services Router
        |
   External VRF / WAN / Data Center / Services
```

In practice, fusion is about routing separation and controlled handoff, not about adding another fabric role.

**Practical design example**

A common enterprise design uses:

- Edge nodes in access closets for wired and wireless endpoint attachment.
- A pair of border nodes at the site edge for external connectivity.
- A control-plane node in the campus core or distribution layer.
- A fusion router or firewall for shared services and WAN handoff.

This design lets you keep endpoint mobility and segmentation inside the fabric while still reaching services outside the overlay.

**Configuration concepts and examples**

The exact commands vary by platform and release, but the configuration workflow usually follows the same pattern.

1. Enable fabric features on the device.
2. Assign the node role.
3. Define underlay reachability.
4. Configure overlay parameters.
5. Verify role-specific behavior.

Example underlay and loopback reachability on a fabric-capable switch:

```cisco
interface Loopback0
 ip address 10.255.0.11 255.255.255.255

interface GigabitEthernet1/0/1
 description Underlay uplink
 ip address 172.16.0.2 255.255.255.252
 ip ospf 10 area 0

router ospf 10
 router-id 10.255.0.11
 passive-interface default
 no passive-interface GigabitEthernet1/0/1
 network 10.255.0.11 0.0.0.0 area 0
 network 172.16.0.0 0.0.0.3 area 0
```

Example of a fabric role assignment workflow on a Catalyst device:

```cisco
fabric enabled
fabric role edge
```

Example of a border-oriented routing handoff concept:

```cisco
router bgp 65000
 neighbor 192.0.2.1 remote-as 65100
 address-family ipv4 unicast
  redistribute connected
  redistribute ospf 10
```

Use these examples as conceptual references. In real deployments, the exact syntax depends on the platform, the automation workflow, and whether the fabric is managed through Cisco DNA Center or CLI.

**Verification commands**

Use platform-specific verification commands rather than generic guesses. Common checks include:

```cisco
show ip interface brief
show ip route
show lisp session
show lisp database
show lisp map-cache
show lisp control-plane
show fabric summary
show fabric role
show cdp neighbors
show ospf neighbor
show bgp summary
```

What to look for:

- **Edge nodes** should show fabric role status, endpoint learning, and LISP registration activity.
- **Border nodes** should show external routing adjacencies and route exchange behavior.
- **Control-plane nodes** should show LISP mapping state and registration counts.
- **Fusion devices** should show the expected routing adjacency or route leak path to shared services.

> Do not rely on `show vxlan` as a universal SD-Access verification command. Availability and output vary by platform and release.

**Troubleshooting by role**

When a fabric role fails, isolate the problem by function.

- **Edge issues**
  - Endpoint does not get the expected gateway or policy.
  - LISP registration fails.
  - Underlay reachability to the control plane is missing.
- **Border issues**
  - External routes are not learned or advertised as expected.
  - Route leaking is incomplete.
  - The border is connected to the wrong VRF or routing domain.
- **Control-plane issues**
  - Map registrations are missing.
  - Map-cache entries are stale or absent.
  - Nodes cannot resolve EID-to-RLOC mappings.
- **Fusion issues**
  - Shared services are reachable from one VRF but not another.
  - Route leaking is too broad or too narrow.
  - The handoff device is not aligned with the intended VRF design.

A practical troubleshooting sequence is:

1. Confirm underlay reachability between fabric nodes.
2. Verify role assignment and fabric enablement.
3. Check LISP registration and map-cache state.
4. Confirm border routing adjacencies.
5. Validate route leaking or redistribution policy.
6. Test endpoint reachability from inside and outside the fabric.

**Common exam traps**

- **Control plane confusion**: The control-plane node maps EIDs to RLOCs; it does not forward user traffic.
- **Border overgeneralization**: Border nodes do not automatically exchange every route with every external protocol.
- **Fusion mislabeling**: Fusion is a design pattern, not a standard fabric node role.
- **Layer 2 border assumptions**: Treat L2 border behavior as special-case and design-specific.
- **Platform assumptions**: Not every Catalyst model supports every fabric role.

**Quick comparison**

| Concept | Primary job | Traffic forwarding? | Exam takeaway |
|---|---|---:|---|
| Edge | Attach endpoints and encapsulate traffic | Yes | First fabric touchpoint for hosts and clients |
| Border | Connect fabric to external domains | Yes | Usually the L3 handoff point |
| Control plane | Maintain endpoint mappings | No | LISP map-server/map-resolver function |
| Fusion | Controlled handoff to shared services or external routing | Depends on implementation | Design pattern, not a fabric role |

**Key takeaways**

- The **edge** is where endpoints join the fabric.
- The **border** is where the fabric meets external routing or shared services.
- The **control plane** stores and resolves LISP endpoint mappings.
- **Fusion** is the routing handoff design, not a fabric node role.
- Platform support, routing design, and role assignment all matter in real deployments.
