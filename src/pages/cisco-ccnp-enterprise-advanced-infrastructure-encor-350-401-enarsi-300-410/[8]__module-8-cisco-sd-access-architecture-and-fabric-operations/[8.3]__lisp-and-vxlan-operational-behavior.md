## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**LISP operational model**

<abbr title="Locator/ID Separation Protocol">LISP</abbr> separates endpoint identity from location. You identify a host or subnet with an <strong>EID</strong> (Endpoint Identifier) and reach it through one or more <strong>RLOCs</strong> (Routing Locators). This separation lets the network change the underlay path without changing the endpoint address seen by applications.

The key operational idea is that LISP uses a control plane to learn where EIDs live, then uses a data plane to encapsulate traffic toward the selected RLOC. That split is central to troubleshooting: if mapping is wrong, traffic fails before encapsulation; if encapsulation is correct but transport fails, the underlay is the problem.

- **EID**: the address prefix or host address that identifies the endpoint.
- **RLOC**: the reachable transport address of the LISP tunnel router.
- **xTR**: the device that performs encapsulation and decapsulation.
- **ITR/ETR**: ingress tunnel router and egress tunnel router roles on the xTR.
- **Map-cache**: the local cache of EID-to-RLOC mappings used for forwarding.
- **Map-server**: stores registrations from ETRs and can assist with reachability of EID prefixes.
- **Map-resolver**: receives map-requests and helps the ITR find the correct mapping.

A practical mental model is:

1. A source xTR receives a packet for a remote EID.
2. It checks the map-cache for an EID-to-RLOC entry.
3. If the mapping exists, it encapsulates immediately.
4. If the mapping is missing, it sends a map-request and waits for a map-reply or uses platform-specific queuing behavior.
5. Once the mapping arrives, it updates the map-cache and forwards subsequent packets.

> Do not assume the first packet is always dropped. Many implementations punt, queue, or briefly buffer traffic while map-resolution completes, and behavior varies by platform and configuration.

**LISP control plane vs data plane**

LISP uses different UDP ports for different functions:

- **UDP 4342**: control-plane signaling such as map-request and map-reply.
- **UDP 4341**: data-plane encapsulation for user traffic.

That distinction matters on exams and in packet captures. If you see UDP 4342, you are usually looking at mapping exchange. If you see UDP 4341, you are usually looking at encapsulated data traffic.

A typical encapsulated packet adds outer headers around the original packet:

```text
Outer Ethernet
Outer IP
UDP 4341
LISP header
Inner IP packet
```

This overhead affects MTU planning. If the underlay MTU is too small, you can see fragmentation, drops, or path-MTU issues. In practice, you should size the underlay to carry the extra encapsulation overhead with room for the inner payload.

**LISP mapping workflow**

LISP forwarding depends on the relationship between the map-cache, map-server, and map-resolver.

- The **map-cache** is the forwarding table on the ITR.
- The **map-resolver** answers the ITR’s map-request by helping it find the correct mapping.
- The **map-server** receives registrations from ETRs and maintains authoritative registration state for EID prefixes.

Operationally, the map-server is not the normal device that directly answers every mapping query. Instead, it supports the registration and resolution ecosystem, while the map-resolver handles the request path for lookups.

A common sequence is:

1. The ETR registers its EID prefix with the map-server.
2. The ITR needs to send traffic to that EID prefix.
3. The ITR sends a map-request to the map-resolver.
4. The map-resolver returns or forwards the mapping information.
5. The ITR installs the result in the map-cache and forwards traffic.

**Locator selection, priority, and weight**

LISP can advertise multiple RLOCs for the same EID prefix. The locator set controls how traffic is distributed and which locators are preferred.

- **Priority** determines preference.
  - Lower priority values are preferred over higher values.
  - If a lower-priority locator is available, it is chosen before less preferred locators.
- **Weight** influences load sharing among locators with the same priority.
  - Higher weight generally receives more traffic.
  - Weight is used for distribution, not preference.
- **Reachability state** affects whether a locator is usable.
  - If a locator becomes unreachable, it should be removed from active selection.

This is important for failover and load balancing. If you have two RLOCs with equal priority but different weights, the ITR can distribute traffic proportionally. If one RLOC fails, the mapping system should converge so the ITR stops using it.

**LISP site registration and reachability changes**

LISP is not static. EID-to-RLOC mappings can change when a site moves, a link fails, or a locator becomes unreachable. The control plane updates the mapping state so new traffic uses the current RLOC set.

- **Registration** tells the map-system that a site owns a prefix.
- **Reachability changes** can trigger updated locator status.
- **Map-cache refresh** ensures the ITR does not keep using stale mappings.

For exam purposes, remember that LISP is designed to react to topology changes without renumbering endpoints. The control plane is what keeps the data plane aligned with current reachability.

**LISP packet behavior and troubleshooting**

When troubleshooting LISP, separate the problem into three layers:

- **Registration**: did the ETR register the EID prefix?
- **Resolution**: did the ITR learn the mapping?
- **Transport**: can the underlay carry the encapsulated packet?

Useful checks include:

```bash
show lisp session
show lisp map-cache
show lisp database
show lisp site
show lisp control-plane
show lisp forwarding
```

Interpretation guidance:

- If the **database** is present but the **map-cache** is empty, the ITR may not have resolved the remote EID yet.
- If the **map-cache** exists but traffic fails, check underlay reachability, ACLs, MTU, and UDP 4341 forwarding.
- If registrations are missing, verify the ETR-to-map-server relationship and the EID prefix advertisement.

A packet capture can help distinguish control-plane and data-plane issues:

- UDP 4342 traffic suggests map exchange.
- UDP 4341 traffic suggests encapsulated user traffic.
- No UDP traffic at all may indicate routing, ACL, or adjacency problems before LISP is even used.

**VXLAN operational model**

<abbr title="Virtual Extensible LAN">VXLAN</abbr> extends Layer 2 segments over a Layer 3 underlay by encapsulating an inner Ethernet frame inside UDP/IP. The tunnel endpoint is the <strong>VTEP</strong> (VXLAN Tunnel Endpoint). The VTEP adds and removes the VXLAN encapsulation.

VXLAN is not a separate “packet type” for routed traffic. It always carries an inner Ethernet frame. Routing between segments is handled by the fabric design, often through an <strong>L3VNI</strong> in an EVPN-based architecture.

A VXLAN packet typically looks like this:

```text
Outer Ethernet
Outer IP
UDP 4789
VXLAN header
Inner Ethernet frame
```

The standard UDP destination port for VXLAN is **4789**.

**VTEP roles and forwarding behavior**

A VTEP can act as both ingress and egress tunnel endpoint:

- **Ingress VTEP**: encapsulates the original Ethernet frame into VXLAN.
- **Egress VTEP**: decapsulates the VXLAN packet and forwards the inner Ethernet frame on the destination segment.

In a fabric, the VTEP also participates in learning and replication decisions. The exact behavior depends on whether the design uses flood-and-learn or EVPN control-plane learning.

- In **flood-and-learn**, the fabric learns MAC addresses from data-plane traffic and floods unknown traffic.
- In **EVPN-based VXLAN**, the control plane distributes MAC/IP reachability, reducing reliance on flooding.

**BUM traffic in VXLAN**

BUM means **broadcast, unknown unicast, and multicast**.

- **Broadcast**: ARP requests and other broadcast frames.
- **Unknown unicast**: destination MAC not yet known in the forwarding table.
- **Multicast**: frames sent to a multicast MAC address.

In VXLAN, BUM handling depends on the design:

- With multicast replication, the underlay multicast group can replicate BUM traffic to interested VTEPs.
- With ingress replication, the source VTEP sends copies to remote VTEPs.
- With EVPN, the control plane can reduce unknown unicast flooding by advertising MAC reachability and suppressing unnecessary flooding.

> Be careful not to treat multicast as “flooded” in the same way as broadcast and unknown unicast. Multicast is a distinct traffic class and is handled according to the replication design.

**Flood-and-learn vs EVPN control plane**

VXLAN can operate with either data-plane learning or EVPN control-plane learning.

| Aspect | Flood-and-learn | EVPN-based VXLAN |
|---|---|---|
| MAC learning | Learned from incoming frames | Advertised by BGP EVPN |
| Unknown unicast | Often flooded until learned | Reduced by control-plane distribution |
| ARP/ND behavior | More dependence on flooding | Can use suppression and proxying |
| Scalability | Lower | Higher |
| Troubleshooting | More dependent on data-plane visibility | More dependent on BGP EVPN state |

In EVPN, the control plane distributes endpoint information so the fabric can forward more intelligently. That is why EVPN is the common enterprise design for scalable VXLAN fabrics.

**EVPN operational behavior**

<abbr title="Ethernet VPN">EVPN</abbr> uses BGP as the control plane to advertise MAC and IP reachability. It maps control-plane state to forwarding behavior in the VTEPs.

Important EVPN concepts:

- **Route distinguisher (RD)** makes overlapping prefixes unique in BGP.
- **Route target (RT)** controls import and export of EVPN routes between VTEPs and VNIs.
- **MAC mobility sequence number** helps the fabric determine which VTEP currently owns a moving MAC address.
- **Type 2 routes** commonly carry MAC/IP reachability.
- **Type 3 routes** commonly carry inclusive multicast or replication information, depending on design.
- **Type 5 routes** commonly carry IP prefix reachability for routed traffic in L3VNI designs.

The operational value of EVPN is that it gives the fabric a control-plane view of endpoint location. When a host moves, the new VTEP advertises the updated MAC/IP information, and the sequence number helps other VTEPs prefer the newest location.

**MAC mobility and forwarding updates**

When a MAC address moves from one VTEP to another, EVPN uses sequence numbers to prevent stale forwarding entries from winning over newer ones.

- The new VTEP advertises the MAC with a higher sequence number.
- Remote VTEPs update their forwarding state.
- Traffic follows the new location without relying on slow data-plane relearning.

This is a major operational advantage over pure flood-and-learn designs. It reduces blackholing during host mobility and improves convergence.

**L3VNI and inter-subnet routing**

A **L3VNI** is used for routed traffic between subnets in a VXLAN EVPN fabric. It is not a different encapsulation format; it is a design construct that lets the fabric carry Layer 3 reachability across the VXLAN overlay.

Operationally:

- A **L2VNI** carries bridged traffic for a VLAN/bridge domain.
- A **L3VNI** carries routed traffic between subnets and VRFs.
- The VTEP performs routing and then encapsulates the routed packet across the overlay.

This matters because inter-subnet traffic in a VXLAN fabric is usually not forwarded as a simple bridged frame across the whole fabric. Instead, the ingress VTEP routes the packet, then sends it over the L3VNI to the remote VTEP that owns the destination subnet or next hop.

A simplified flow is:

1. Host A sends traffic to a remote subnet.
2. The ingress VTEP routes the packet in the VRF.
3. The VTEP selects the remote VTEP based on EVPN reachability.
4. The packet is encapsulated in VXLAN over the L3VNI.
5. The egress VTEP decapsulates and forwards toward the destination.

**VXLAN and EVPN configuration examples**

The exact syntax varies by platform and software release, so treat these as conceptual examples rather than copy-paste templates.

```cisco
! Conceptual VXLAN EVPN building blocks
feature nv overlay
feature vn-segment-vlan-based
feature bgp

interface nve1
  no shutdown
  source-interface loopback0
  host-reachability protocol bgp
  member vni 10010
    ingress-replication protocol bgp
  member vni 50000 associate-vrf

vlan 10
  vn-segment 10010

vrf context TENANT-A
  vni 50000
```

A corresponding BGP EVPN control-plane example:

```cisco
router bgp 65001
  address-family l2vpn evpn
    neighbor 10.0.0.2 activate
    neighbor 10.0.0.2 send-community extended
```

Use these examples to understand the roles of the components:

- `nve1` is the VXLAN tunnel interface.
- `source-interface loopback0` provides the stable VTEP source address.
- `host-reachability protocol bgp` indicates EVPN control-plane learning.
- `member vni` binds a VNI to the overlay.
- `associate-vrf` ties the L3VNI to the VRF.

**LISP configuration example**

LISP syntax varies significantly by platform and release, so avoid memorizing one exact command string as universal truth. Focus on the roles and the workflow.

```cisco
! Conceptual LISP building blocks
router lisp
  site SITE1
    eid-prefix 10.10.10.0/24
  itr map-resolver 192.0.2.10
  etr map-server 192.0.2.20
```

Use this as a conceptual reminder only:

- The site advertises an EID prefix.
- The ITR points to a map-resolver.
- The ETR registers with a map-server.
- The xTR handles encapsulation and decapsulation.

**Verification and troubleshooting commands**

For LISP:

```bash
show lisp map-cache
show lisp database
show lisp site
show lisp control-plane
show lisp forwarding
show lisp session
```

For VXLAN and EVPN:

```bash
show nve interface
show nve peers
show nve vni
show bgp l2vpn evpn
show mac address-table dynamic
show ip route vrf <VRF-NAME>
show fabric forwarding ip route
```

What to look for:

- **NVE peers**: confirm remote VTEPs are up and reachable.
- **VNI state**: confirm the VNI is up and mapped correctly.
- **BGP EVPN routes**: confirm MAC/IP and prefix reachability are being advertised.
- **MAC table**: confirm local and remote MAC learning is consistent with the design.
- **VRF routes**: confirm L3VNI routing is installed for inter-subnet forwarding.

**Common failure symptoms**

- **Missing LISP map-cache entries**: the ITR has not resolved the remote EID or the map-request path is broken.
- **Stale LISP mappings**: the control plane has not converged after a reachability change.
- **VXLAN peer down**: underlay reachability, source-interface, or NVE configuration is wrong.
- **Unknown unicast flooding**: EVPN learning is incomplete or the MAC/IP route is missing.
- **Inter-subnet traffic fails**: the L3VNI, VRF, or EVPN Type 5 route is missing or inconsistent.
- **MTU-related drops**: the underlay cannot carry the encapsulated packet size.

A useful troubleshooting sequence is:

1. Verify underlay IP reachability between loopbacks.
2. Verify the tunnel source interface and NVE/LISP source addresses.
3. Verify control-plane adjacency or registration.
4. Verify mapping or EVPN route installation.
5. Verify MTU and ACLs for UDP 4341, UDP 4342, and UDP 4789 as appropriate.

**LISP vs VXLAN operational comparison**

| Topic | LISP | VXLAN |
|---|---|---|
| Main purpose | Separates identity from location | Extends Layer 2 and Layer 3 overlays over IP underlay |
| Control plane | Map-server/map-resolver ecosystem | Often BGP EVPN |
| Data-plane port | UDP 4341 | UDP 4789 |
| Control-plane port | UDP 4342 | BGP over TCP 179 for EVPN |
| Endpoint identity | EID | MAC/IP in EVPN, VNI-based segmentation |
| Mobility handling | Mapping updates and locator changes | EVPN MAC mobility and route updates |
| Common exam focus | Map-cache, RLOCs, registration, resolution | VTEPs, VNIs, EVPN routes, L3VNI, BUM handling |

**Exam-focused takeaways**

- LISP uses a control plane to map EIDs to RLOCs and a data plane to encapsulate traffic toward the selected locator.
- LISP control traffic uses UDP 4342, while LISP data traffic uses UDP 4341.
- VXLAN encapsulates inner Ethernet frames over UDP 4789 and relies on VTEPs to add and remove the overlay header.
- BUM means broadcast, unknown unicast, and multicast, and the replication method depends on the fabric design.
- EVPN improves VXLAN scalability by distributing MAC/IP reachability and mobility information in the control plane.
- L3VNI is the routed overlay construct that carries inter-subnet traffic in a VXLAN EVPN fabric.
- For troubleshooting, always separate control-plane failure, mapping failure, and underlay transport failure before changing configuration.
