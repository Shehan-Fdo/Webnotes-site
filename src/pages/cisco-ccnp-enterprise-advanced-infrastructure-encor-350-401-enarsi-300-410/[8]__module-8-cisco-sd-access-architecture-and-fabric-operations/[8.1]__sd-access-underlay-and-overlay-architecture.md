## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**SD-Access architecture overview**

Cisco Software-Defined Access (SD-Access) separates the fabric into an **underlay** and an **overlay**. The underlay is the IP transport that provides reachability between fabric nodes. The overlay is the virtualized fabric built on top of that transport, using **LISP** (Locator/ID Separation Protocol) for control-plane mapping and **VXLAN** (Virtual Extensible LAN) for data-plane encapsulation.

The key exam idea is that the underlay and overlay solve different problems:

- The **underlay** provides stable IP connectivity, typically with routing only.
- The **overlay** provides segmentation, endpoint mobility, and policy enforcement.
- The **control plane** maps endpoint identity to location.
- The **data plane** carries encapsulated traffic across the fabric.

A useful mental model is:

- Underlay = “How do fabric nodes reach each other?”
- Control plane = “Where is an endpoint located?”
- Overlay data plane = “How do packets cross the fabric while preserving segmentation?”

**Underlay design fundamentals**

The underlay is usually built with a routed topology, often using OSPF, IS-IS, or eBGP depending on the design. Its job is to provide IP reachability between all fabric nodes and to support the transport requirements of VXLAN and LISP.

Important underlay characteristics include:

- **Loop-free routing** between fabric nodes.
- **Consistent MTU** to support encapsulation overhead.
- **Reachability** to loopback addresses used by control-plane and VXLAN endpoints.
- **Fast convergence** so fabric traffic remains stable during failures.

A common design choice is to use routed point-to-point links between fabric nodes. This reduces Layer 2 dependency and simplifies failure domains.

> <mark>Do not assume the underlay carries endpoint subnets in every design.</mark> In SD-Access, endpoint reachability is primarily handled by the fabric control plane and overlay. Some prefixes may still appear in border or inter-site designs, but the underlay itself is not the place where endpoint mobility is solved.

**Underlay addressing and MTU**

Fabric nodes typically use loopback interfaces for control-plane and VXLAN source addresses. Physical links use routed interfaces or routed port-channels. The underlay must support the encapsulation overhead introduced by VXLAN and, in some cases, additional headers from tunneling or transport features.

Practical guidance:

- Use a larger MTU on underlay links, commonly at least `9100` on many campus designs, if supported end to end.
- Verify that all hops support the same or greater MTU.
- Ensure the routing protocol advertises loopback reachability.

Example underlay interface pattern:

```cisco
interface TenGigabitEthernet1/0/1
 description To-Distribution-1
 no switchport
 ip address 10.0.0.1 255.255.255.252
 ip mtu 9100
 no shutdown

interface Loopback0
 ip address 10.255.255.1 255.255.255.255
```

Typical verification commands:

```cisco
show ip interface brief
show ip route
show ip ospf neighbor
show isis neighbors
show bgp ipv4 unicast summary
ping 10.255.255.2 source 10.255.255.1
traceroute 10.255.255.2 source 10.255.255.1
```

**Fabric node roles**

SD-Access uses specific node roles. These roles matter because each node participates differently in the underlay, control plane, and overlay.

- **Edge node**
  - Connects endpoints to the fabric.
  - Provides first-hop gateway functions for fabric VNIs when configured.
  - Encapsulates and decapsulates VXLAN traffic for attached endpoints.
  - Registers endpoint information with the control plane.

- **Control-plane node**
  - Stores endpoint-to-location mappings.
  - Acts as the LISP map-server and map-resolver in the fabric.
  - Helps fabric nodes locate remote endpoints.

- **Border node**
  - Connects the fabric to external routing domains or to other fabric domains, depending on the design.
  - Exchanges routes between the fabric and non-fabric networks.
  - Can perform route leaking and VRF handoff behavior.

- **Intermediate node**
  - Carries fabric underlay traffic between edge, border, and control-plane nodes.
  - Does not attach endpoints as a fabric access role.
  - Provides transit connectivity in the routed underlay.

These roles are not just labels; they determine which protocols and fabric functions a device must support.

**Edge node behavior**

The edge node is where most endpoint-facing fabric behavior occurs. It is often the access switch that connects wired hosts, and in wireless designs it may also participate in fabric termination for wireless client traffic depending on the deployment model.

Edge nodes typically:

- Learn endpoint MAC and IP information from attached devices.
- Register endpoint identity and location with the control plane.
- Encapsulate traffic into VXLAN when sending to remote fabric nodes.
- Decapsulate VXLAN traffic when receiving traffic for local endpoints.
- Provide anycast gateway behavior for fabric VNIs when configured.

A first-hop gateway on a fabric edge is often an anycast gateway shared across fabric nodes in the same virtual network. This allows endpoint mobility without changing the default gateway address.

Example conceptual SVI pattern:

```cisco
interface Vlan10
 description Users-VN
 ip address 192.168.10.1 255.255.255.0
 fabric forwarding mode anycast-gateway
```

This snippet is only a conceptual example. In real SD-Access deployments, fabric-enabled SVIs are created and managed through the fabric workflow, not as isolated standalone configuration.

**Control-plane architecture and LISP**

In SD-Access, **LISP** is the control-plane protocol used to map endpoint identifiers to routing locators. The endpoint identity is often called an **EID** (Endpoint ID), and the fabric node address used for reachability is the **RLOC** (Routing Locator).

The control plane is not a generic directory for all traffic. It specifically helps fabric nodes answer questions such as:

- Where is this endpoint currently attached?
- Which fabric node should receive encapsulated traffic for this destination?
- What is the current location of a moved endpoint?

The basic flow is:

- An edge node learns an endpoint.
- The edge node registers the endpoint mapping with the control-plane node.
- Another fabric node queries the control plane when it needs to reach that endpoint.
- The control plane returns the current location information.
- The sending node encapsulates traffic to the correct remote fabric node.

This is why SD-Access supports endpoint mobility well: the identity stays the same while the location can change.

Important LISP terms:

- **Map-register**: endpoint location registration from edge to control plane.
- **Map-request**: query for endpoint location.
- **Map-reply**: response containing the current mapping.
- **xTR**: tunnel router role; in fabric terms, the edge often behaves as the registration and encapsulation point.

**VXLAN data plane**

VXLAN is the overlay data-plane technology. It encapsulates original endpoint frames inside UDP/IP so they can traverse the routed underlay.

Why VXLAN is used:

- It scales Layer 2 and Layer 3 segmentation better than traditional VLAN-only designs.
- It allows the fabric to carry tenant or virtual-network separation across the campus.
- It supports endpoint mobility across fabric nodes.

VXLAN uses:

- A UDP transport, commonly destination port `4789`.
- A **VNI** (VXLAN Network Identifier) to identify the overlay segment.
- Underlay IP addresses as tunnel endpoints.

In SD-Access, the VNI is tied to fabric segmentation constructs such as virtual networks and Layer 2 segments. The underlay only sees IP packets between fabric nodes; the overlay preserves the fabric context.

Example packet flow concept:

- Host sends a frame to its default gateway on the local edge node.
- The edge node determines whether the destination is local or remote.
- If remote, the edge node encapsulates the packet in VXLAN and sends it across the underlay.
- The remote edge decapsulates and forwards the packet to the destination endpoint.

**Fabric segmentation and policy**

SD-Access uses segmentation to isolate traffic and apply policy. The main constructs are:

- **Virtual Network (VN)**: logical segmentation boundary.
- **Layer 2 VNI**: used for bridged segments.
- **Layer 3 VNI**: used for routed segments within a VN.
- **Scalable Group Tag (SGT)**: identity used for policy enforcement in TrustSec-based designs.

The practical effect is that two endpoints can be in different virtual networks even if they share the same physical fabric. Policy can then control whether traffic is allowed between them.

Common use cases:

- Separate corporate, guest, and IoT traffic.
- Keep branch or campus departments isolated.
- Apply identity-based policy instead of relying only on IP subnets.

**Fabric site and domain hierarchy**

SD-Access is often deployed with a hierarchy that includes fabric sites and, in larger designs, multiple sites connected through a broader fabric architecture.

At a conceptual level:

- A **fabric site** is a logical campus or campus segment built with SD-Access.
- A **fabric domain** may refer to the fabric control and policy scope within that site or architecture.
- Multi-site designs connect fabric sites while preserving segmentation and policy intent.

In multi-site architectures, the design goal is to extend segmentation between sites without flattening the entire campus into one large Layer 2 domain. Route exchange and policy handoff are handled at borders and inter-site connections rather than by stretching access VLANs everywhere.

Exam takeaway:

- Know that SD-Access is not just a single fabric switch pair.
- Understand that site boundaries matter for control-plane scale, route distribution, and policy containment.
- Expect border nodes and route exchange to be central in multi-site designs.

**Border node behavior**

Border nodes connect the fabric to external networks or to other fabric domains, depending on the design.

Two common border roles are:

- **External border**
  - Connects the fabric to non-fabric networks such as the enterprise core, WAN, data center, or Internet edge.
  - Exchanges routes between fabric VRFs and external routing domains.
  - Often performs route leaking or redistribution under controlled policy.

- **Internal border**
  - Connects fabric sites or fabric domains within the same SD-Access architecture.
  - Supports inter-site or inter-domain route exchange while preserving segmentation.
  - Helps extend reachability without making every site a single flat routing domain.

Border behavior is more specific than “connects one fabric to another.” The exact route exchange depends on whether the border is external or internal, which VRFs are involved, and how policy is defined.

Practical border concepts:

- Border nodes often participate in multiple VRFs.
- They may import and export routes selectively.
- They are a common place for handoff to traditional routing domains.

**Wireless integration**

Wireless in SD-Access is integrated through specific deployment models rather than by treating APs as fabric nodes in the same way as edge or border switches.

Key ideas:

- Access points connect through the wired underlay, but they are not fabric control-plane nodes.
- Wireless client traffic is integrated into the fabric according to the chosen wireless deployment model.
- The wireless LAN controller (WLC) may be centralized, converged, or otherwise integrated depending on the architecture.

What matters for exam prep is the traffic termination point:

- Wireless control and data paths do not always terminate in the same place.
- The WLC role depends on the deployment model.
- Client traffic is then mapped into the fabric overlay so policy and segmentation remain consistent with wired endpoints.

Be careful not to oversimplify wireless as “APs just connect to the fabric.” The AP uplink is part of the underlay, but the wireless service model determines how clients are anchored and how traffic enters the overlay.

**Multi-site and route exchange concepts**

In larger deployments, SD-Access must support multiple sites while preserving segmentation and scale. This introduces route exchange between fabric domains and border nodes.

Important concepts:

- Endpoint mobility is local to the fabric control plane, but inter-site reachability depends on border design.
- Routes may be summarized or selectively leaked between sites.
- VRF-to-VRF handoff can occur at borders to preserve tenant separation.
- The design should avoid unnecessary flooding or broad route redistribution.

A good mental model is:

- Inside a site, the fabric control plane handles endpoint location.
- Between sites, borders handle route exchange and policy boundaries.
- The overlay remains segmented even when traffic crosses sites.

**Underlay multicast and replication considerations**

VXLAN traffic may require replication for broadcast, unknown unicast, and multicast traffic, often abbreviated as **BUM**.

Depending on the design, replication can be handled by:

- Multicast in the underlay.
- Head-end replication from the fabric node.
- A design-specific combination of both.

Why this matters:

- Broadcast and unknown destination traffic still need a delivery mechanism.
- The underlay must support the chosen replication model.
- Multicast design affects scalability and operational complexity.

Practical exam point:

- Know that replication is a design choice, not a universal single method.
- Validate that the underlay supports the selected approach before enabling fabric services.

**Verification and troubleshooting**

Verification in SD-Access should confirm underlay health, control-plane registration, and overlay forwarding.

Useful checks include:

```cisco
show ip route
show ip ospf neighbor
show isis neighbors
show lisp session
show lisp eid-table
show lisp map-cache
show lisp database
show fabric summary
show fabric control-plane
show fabric edge
show fabric border
show fabric intermediate
show fabric multicast
show vxlan
```

Depending on platform and software release, some commands may vary. Use the fabric-specific commands available on the device and in the controller workflow.

What to verify:

- Underlay neighbors are up and routes are present.
- Loopback reachability exists between fabric nodes.
- LISP sessions are established with the control-plane node.
- Endpoint mappings appear in the database and map-cache.
- Fabric role status matches the intended node function.
- VXLAN tunnels or fabric encapsulation are operational where supported.

Example troubleshooting sequence:

1. Confirm underlay adjacency and routing.
2. Confirm loopback reachability between fabric nodes.
3. Confirm LISP control-plane registration.
4. Confirm endpoint appears in the database.
5. Confirm the edge has a valid map-cache entry for remote destinations.
6. Confirm traffic is being encapsulated and decapsulated as expected.

Example output patterns to look for:

```cisco
show lisp session
show lisp database
show lisp map-cache
```

You want to see:

- Established control-plane sessions.
- Local endpoint registrations.
- Remote endpoint mappings with correct RLOCs.
- No stale or unresolved entries for active endpoints.

**Practical configuration workflow**

In real deployments, SD-Access is usually provisioned through Cisco Catalyst Center rather than by manually building every fabric feature on each switch. The controller defines the fabric, assigns roles, and pushes the required configuration.

A typical workflow is:

1. Build the underlay with routed links and routing protocol adjacency.
2. Define fabric sites and assign devices as edge, border, control-plane, or intermediate nodes.
3. Configure virtual networks and segmentation policy.
4. Provision access ports and fabric-enabled interfaces.
5. Validate control-plane registration and endpoint learning.
6. Test wired and wireless traffic flows.

A simplified underlay example for a routed fabric node:

```cisco
router ospf 10
 router-id 10.255.255.1
 passive-interface default
 no passive-interface TenGigabitEthernet1/0/1
 network 10.0.0.0 0.0.0.3 area 0
 network 10.255.255.1 0.0.0.0 area 0
```

A simplified verification example:

```cisco
show ip ospf neighbor
show ip route 10.255.255.2
show lisp session
show lisp map-cache
show fabric summary
```

These examples are intentionally generic. Exact commands and syntax vary by platform, software release, and whether the device is managed by Catalyst Center.

**Common exam pitfalls**

- Confusing the **underlay** with the overlay.
- Treating LISP as a generic routing protocol instead of a mapping system for EID-to-RLOC resolution.
- Assuming every fabric node performs the same role.
- Overstating what the underlay carries in every design.
- Using generic VXLAN or NVE commands without checking platform applicability.
- Ignoring the difference between external and internal border behavior.
- Oversimplifying wireless integration and client anchoring.

**Quick comparison**

| Component | Primary job | Key technology |
|---|---|---|
| Underlay | Transport between fabric nodes | IP routing |
| Control plane | Endpoint location mapping | LISP |
| Data plane | Encapsulated traffic forwarding | VXLAN |
| Edge node | Endpoint attachment and encapsulation | Fabric edge functions |
| Border node | External or inter-site route exchange | VRF handoff, redistribution, policy |
| Intermediate node | Transit in the underlay | Routed forwarding |

**High-yield summary**

SD-Access uses a routed underlay for transport, a LISP-based control plane for endpoint location, and a VXLAN overlay for segmented forwarding. Edge nodes attach endpoints and register them with the control plane. Border nodes connect the fabric to external or inter-site domains. Intermediate nodes provide transit. Wireless integrates through specific deployment models, and verification should always confirm underlay reachability, control-plane registration, and overlay forwarding.
