## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Fabric troubleshooting mindset**

Cisco SD-Access troubleshooting works best when you separate the problem into layers: underlay, control plane, data plane, policy, and services. A symptom at the edge often originates elsewhere. For example, an endpoint may fail to reach a server because the fabric edge cannot register the endpoint, the control plane node cannot resolve the endpoint location, the border cannot leak routes, or the underlay cannot carry the encapsulated traffic.

A practical workflow is to start with the simplest question: is the failure local to one endpoint, one fabric node, one site, or the entire fabric? Then verify the role of the affected node. Edge nodes, control-plane nodes, border nodes, and intermediate underlay nodes fail in different ways and expose different clues.

**Fabric node roles and what to check**

- **Edge node** issues usually affect endpoint attachment, authentication, SGT assignment, and local VXLAN encapsulation or decapsulation.
- **Control-plane node** issues usually affect endpoint registration, LISP mapping, and reachability of remote EIDs.
- **Border node** issues usually affect route exchange with external networks, route leaking, and traffic entering or leaving the fabric.
- **Intermediate underlay node** issues usually affect transport reachability, MTU, routing adjacency, and packet loss across the fabric transport.

A useful mental model is:

- **EID**: Endpoint Identifier, the fabric-facing identity of an endpoint.
- **RLOC**: Routing Locator, the underlay-reachable location of a fabric node.
- **LISP**: Locator/ID Separation Protocol, used to map EIDs to RLOCs in many SD-Access designs.
- **VXLAN**: Virtual Extensible LAN, the encapsulation used for fabric data-plane transport.

**Troubleshooting workflow**

1. Confirm the symptom and scope.
   - Identify source, destination, VLAN, VRF, SGT, and whether the issue is wired or wireless.
   - Determine whether the failure is one-way or bidirectional.
2. Verify the underlay first.
   - Check routing adjacencies, reachability, MTU, and packet loss.
   - Confirm that fabric node loopbacks and transport addresses are reachable.
3. Verify control-plane registration.
   - Check whether the endpoint is registered in the fabric control plane.
   - Confirm that the node can resolve remote endpoint locations.
4. Verify data-plane forwarding.
   - Confirm VXLAN encapsulation and decapsulation.
   - Check that the correct VTEP-like fabric node is used as the next hop.
5. Verify policy and segmentation.
   - Confirm SGT assignment, SGACL enforcement, and VRF mapping.
6. Verify border and external connectivity.
   - Check route exchange, route leaking, and handoff to nonfabric networks.
7. Use assurance and telemetry.
   - Correlate alarms, path traces, and endpoint history in Cisco DNA Center or Catalyst Center.

**Underlay troubleshooting**

The underlay is the transport foundation. If the underlay is unstable, the fabric may still form partially, but endpoint registration and VXLAN forwarding become unreliable. Underlay problems often appear as intermittent reachability, slow convergence, or fabric nodes that can see some peers but not others.

Common underlay failure modes include:

- Missing or incorrect routing adjacency between fabric nodes.
- MTU mismatch causing fragmentation or dropped encapsulated packets.
- ACLs or firewalls blocking required transport traffic.
- Loopback or transport interface reachability failures.
- Asymmetric routing that breaks return traffic or control-plane sessions.
- Loss or flapping of the routing protocol used in the underlay.

Useful checks include:

```bash
show ip interface brief
show ip route
show ip ospf neighbor
show ip eigrp neighbors
show ip bgp summary
show ip cef <peer-loopback>
ping <peer-loopback> source <loopback-interface>
traceroute <peer-loopback> source <loopback-interface>
show interfaces <interface> | include MTU|error|drop
show platform hardware qfp active statistics drop
```

If the fabric uses OSPF, EIGRP, or BGP in the underlay, verify adjacency health and route installation. A node may have a neighbor relationship but still fail to forward if the route to the remote loopback is missing or if the MTU is too small for encapsulated traffic.

> **Critical warning:** Do not assume a successful ping proves the underlay is healthy. Small ICMP packets can pass even when MTU, fragmentation, or QoS issues break VXLAN traffic.

A practical MTU check is to test the largest expected packet size along the path. If the fabric adds encapsulation overhead, the underlay must support it end to end. Symptoms of MTU problems often include:

- Endpoint traffic works for small packets but fails for large transfers.
- Control-plane sessions stay up, but data-plane traffic drops.
- Path traces show reachability, but applications time out.

**Control-plane troubleshooting**

The control plane maps endpoint identities to fabric locations. In SD-Access, this is commonly handled through LISP-based registration and lookup behavior. The exact commands and output vary by platform and IOS XE release, so verify on the specific node role and software version.

The main questions are:

- Did the endpoint register successfully?
- Does the control plane know the correct location?
- Can the fabric node resolve the remote endpoint?
- Is the mapping stale, missing, or pointing to the wrong RLOC?

Typical verification commands include:

```bash
show lisp session
show lisp control-plane
show lisp map-cache
show lisp database
show lisp eid-table
show lisp site
show lisp instance-id
```

On some platforms, endpoint visibility is better checked with role-specific fabric commands rather than a single universal LISP command. Use the command set that matches the node role and software release.

Healthy control-plane behavior usually shows:

- A registered endpoint entry with the correct EID.
- A reachable RLOC for the fabric node.
- A populated map-cache for remote destinations.
- Stable session state between fabric nodes and the control-plane node.

Example of a healthy-looking mapping conceptually:

```text
EID: 10.10.10.25/32
RLOC: 172.16.0.11
State: registered
Map-cache: resolved
```

Example of a problem pattern:

```text
EID: 10.10.10.25/32
RLOC: unresolved
State: pending
Map-cache: negative-cache entry
```

A negative cache entry means the node recently failed to resolve the destination and is temporarily remembering that failure. This can happen during convergence, after endpoint movement, or when the control plane has not yet learned the new location.

Common control-plane causes include:

- Endpoint authentication failure at the edge.
- Incorrect VLAN-to-VNI or SGT-to-policy mapping.
- Control-plane node reachability loss.
- Stale endpoint registration after mobility or link flap.
- Duplicate endpoint identity or conflicting registration.
- Underlay instability preventing registration refresh.

**Data-plane troubleshooting**

The data plane carries encapsulated traffic between fabric nodes. In SD-Access, the forwarding path depends on correct encapsulation, decapsulation, and lookup of the remote endpoint location. A control-plane entry may exist, but traffic can still fail if the data plane cannot forward the packet.

Focus on these questions:

- Is the packet being encapsulated at the source edge?
- Is the packet reaching the correct remote fabric node?
- Is the packet being decapsulated and delivered to the endpoint?
- Is the return path symmetric enough for the application?

Useful verification commands vary by platform, but common checks include:

```bash
show ip cef <destination-ip> detail
show ip route vrf <vrf-name> <destination-ip>
show interfaces nve1
show nve peers
show nve vni
show platform hardware forwarding
show platform hardware qfp active feature vxlan
show platform hardware qfp active statistics drop
```

Not every platform exposes the same NVE-style commands in the same way, but the concepts remain the same: confirm tunnel/overlay peer health, VNI membership, and forwarding-table installation.

A healthy data plane typically shows:

- The overlay peer is up.
- The VNI is active and mapped to the correct VRF or bridge domain.
- The forwarding table contains the expected remote location.
- Encapsulated packets are counted on the source node and decapsulated on the destination node.

Common data-plane failure modes include:

- VXLAN encapsulation missing because the endpoint is not in the fabric table.
- Wrong VNI or VRF association.
- Forwarding entry installed to the wrong remote node.
- Underlay drops due to MTU or ACL issues.
- Hardware forwarding table exhaustion or programming failure.
- Asymmetric return path causing stateful application failure.

**Control plane vs data plane comparison**

| Area | What it does | Typical symptom | What to verify |
|---|---|---|---|
| Control plane | Learns and distributes endpoint location | Endpoint appears unreachable across the fabric | LISP registration, map-cache, endpoint database |
| Data plane | Forwards encapsulated traffic | Control-plane looks healthy but traffic drops | VXLAN peers, VNI state, hardware forwarding, drops |
| Underlay | Carries transport between fabric nodes | Multiple fabric functions fail or flap | Routing adjacency, MTU, loopback reachability |
| Policy | Enforces segmentation and access rules | Reachability exists but traffic is denied | SGT, SGACL, VRF, policy mapping |

**Policy and segmentation troubleshooting**

Policy issues are often mistaken for routing failures. In SD-Access, segmentation commonly uses Security Group Tags (SGTs) and Security Group Access Control Lists (SGACLs). If routing works but traffic is blocked, policy is a strong suspect.

Check these items:

- The endpoint received the expected SGT.
- The policy matrix allows the source and destination groups.
- The fabric node has the correct policy mapping.
- The traffic is not being denied by an SGACL or external ACL.
- The VRF is correct and the destination is in the expected segment.

Useful commands may include:

```bash
show cts role-based sgt-map all
show cts role-based permissions
show access-lists
show ip access-lists
show policy-map interface
show fabric forwarding policy
```

Command availability varies by platform and release. In SD-Access, focus on the actual policy model in use rather than legacy TrustSec terminology alone. If the issue is policy-related, you often see:

- Ping works between endpoints in the same segment but fails across segments.
- One application port works while another is denied.
- The endpoint is reachable from the fabric but not from a specific group.
- Logs or counters show drops without route failures.

A useful test is to compare a known-allowed source and a known-denied source. If both fail, the issue is probably not policy alone. If only one fails, policy is more likely.

**Border node troubleshooting**

Border nodes connect the fabric to external networks. They handle route exchange, route leaking, and traffic handoff between fabric VRFs and nonfabric routing domains. Border behavior depends on the design, but the key idea is that the border must learn external routes and advertise fabric routes correctly.

Troubleshoot border issues by checking:

- Whether the border has the correct external adjacency.
- Whether routes are learned from the external domain.
- Whether fabric prefixes are advertised outward as intended.
- Whether the correct VRF is used for the handoff.
- Whether route leaking or redistribution is configured correctly.

Useful commands include:

```bash
show ip route vrf <vrf-name>
show bgp ipv4 unicast summary
show bgp l2vpn evpn summary
show ip protocols
show lisp site
show lisp map-server
show lisp map-resolver
```

The exact border design may use BGP, static routes, or other route-exchange mechanisms depending on the deployment. The important exam point is that the border is not just a generic router; it is a fabric role with specific route-leaking and policy implications.

Common border symptoms include:

- Fabric endpoints can reach each other, but not external networks.
- External networks can reach some fabric prefixes but not others.
- Return traffic enters the wrong VRF.
- Default route exists, but specific routes are missing.
- Border adjacency is up, but redistribution is incomplete.

**Wireless fabric troubleshooting**

Wireless endpoints add another layer: the access point, wireless controller, mobility behavior, and fabric integration. A wireless issue may look like a fabric problem, but the root cause can be AP registration, client roaming, or controller policy.

Check:

- AP join status and controller reachability.
- Client association and authentication.
- Client mobility state during roaming.
- Fabric role of the wireless termination point.
- Whether the client is anchored or local to the controller.

Useful commands include:

```bash
show wireless client summary
show wireless client mac <client-mac>
show ap summary
show ap join stats detailed <ap-name>
show mobility summary
show wireless mobility summary
```

Wireless-specific failure patterns include:

- Client connects but never gets an IP address.
- Client roams and loses connectivity for longer than expected.
- Client is anchored to the wrong controller or mobility domain.
- Fabric policy is correct for wired clients but not for wireless clients.
- AP joins fail because of underlay or controller reachability issues.

Roaming problems often involve timing. A brief interruption can be normal, but prolonged loss suggests mobility database issues, controller reachability problems, or inconsistent policy state during handoff.

**Cisco DNA Center / Catalyst Center assurance workflow**

Assurance tools help you correlate symptoms with fabric state. They are especially useful when the issue is intermittent or affects multiple nodes. Instead of checking one device at a time, you can trace the endpoint, path, and health indicators across the fabric.

Use assurance to look for:

- Endpoint history and movement.
- Path trace results.
- Device health and interface drops.
- Control-plane registration failures.
- Policy violations or segmentation mismatches.
- Wireless client roam events.

A practical workflow is:

1. Search for the endpoint or client.
2. Review the timeline for moves, failures, and policy changes.
3. Run a path trace from source to destination.
4. Compare the intended path with the actual path.
5. Correlate device health, interface errors, and control-plane alarms.

Assurance is not a replacement for CLI verification, but it often narrows the problem quickly and tells you which node to inspect first.

**Practical troubleshooting scenarios**

Scenario 1: Wired endpoint can ping the gateway but not a remote server.

- The endpoint is attached and has basic local connectivity.
- The issue may be remote endpoint resolution, policy, or border routing.
- Check the control-plane mapping for the server and the border route exchange if the server is external.
- Verify SGACLs if the server is in a different segment.

Scenario 2: Multiple endpoints on one edge switch fail after a link flap.

- The edge node may have lost control-plane reachability or failed to refresh registrations.
- Check underlay stability, LISP session state, and endpoint database entries.
- Verify that the edge node still has the correct fabric role and VNI mappings.

Scenario 3: Fabric-to-external traffic fails, but internal fabric traffic works.

- The border node is the primary suspect.
- Check route leaking, redistribution, VRF handoff, and external adjacency.
- Confirm that return routes exist and that the external domain knows the fabric prefixes.

Scenario 4: Wireless client roams and loses access to one application only.

- The issue may be policy or mobility state rather than pure connectivity.
- Check client SGT assignment before and after roam.
- Verify controller and mobility state consistency.
- Compare the application’s ACL or SGACL treatment with a known-working client.

**Sample verification snippets**

```bash
show lisp map-cache | include 10.10.10.25
show lisp database | include 10.10.10.25
show ip cef 10.10.10.25 detail
show platform hardware forwarding | include 10.10.10.25
show interfaces nve1
show nve peers
```

Example of a healthy endpoint flow:

```text
Endpoint attached -> registered in control plane -> map-cache resolved -> VXLAN encapsulated -> remote edge decapsulates -> policy permits traffic
```

Example of a broken flow:

```text
Endpoint attached -> registration missing -> map-cache unresolved -> no valid forwarding entry -> traffic dropped
```

**How to interpret symptoms quickly**

- If only one endpoint fails, suspect attachment, registration, or policy.
- If many endpoints on one node fail, suspect the node, its underlay, or its role configuration.
- If internal fabric traffic works but external traffic fails, suspect the border.
- If control-plane looks healthy but traffic drops, suspect data-plane programming, MTU, or policy.
- If wireless clients fail but wired clients work, suspect AP, controller, or mobility behavior.

**Exam-focused reminders**

- Verify the node role before choosing commands.
- Do not overgeneralize command output across platforms or releases.
- Separate routing reachability from fabric endpoint reachability.
- Treat policy denials differently from forwarding failures.
- Use assurance to correlate symptoms, but confirm with CLI and packet-path reasoning.
- Remember that fabric troubleshooting is often about finding the first broken layer, not the last visible symptom.

**Quick reference checklist**

- Underlay reachability and MTU
- Control-plane registration and map-cache
- Data-plane encapsulation and forwarding
- Policy and segmentation state
- Border route exchange and VRF handoff
- Wireless join, roam, and mobility state
- Assurance correlation and endpoint history
