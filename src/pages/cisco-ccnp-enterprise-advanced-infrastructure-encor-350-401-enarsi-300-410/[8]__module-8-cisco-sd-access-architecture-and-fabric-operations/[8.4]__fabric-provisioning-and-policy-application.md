## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**SD-Access fabric overview**

Cisco Software-Defined Access (SD-Access) builds an overlay network on top of a routed underlay. The underlay provides IP reachability, while the fabric adds endpoint mobility, segmentation, and policy enforcement. In Cisco exam terminology, the fabric is typically orchestrated by Cisco DNA Center, now Cisco Catalyst Center, which designs the fabric, assigns node roles, and pushes the required configuration.

At a high level, you provision the fabric in layers:

- **Underlay**: routed transport with loopbacks, point-to-point links, and dynamic routing.
- **Overlay**: VXLAN encapsulation for data-plane transport and LISP for endpoint location mapping.
- **Policy**: Cisco TrustSec with Security Group Tags (SGTs) and Security Group Access Control Lists (SGACLs).

The key idea is that you do not manually build every tunnel and policy binding. You define intent in the controller, then the controller programs the fabric nodes.

**Fabric design objects in Catalyst Center**

Before you provision devices, you design the fabric site and supporting objects in the controller. These objects determine how the controller generates configuration and how endpoints are mapped into the fabric.

Typical design inputs include:

- **Fabric site hierarchy**: campus, building, and floor structure.
- **IP pools**: used for underlay links, loopbacks, and client subnets.
- **Transit addressing**: routed links between fabric nodes and external networks.
- **Virtual networks (VNs)**: segmentation containers that map to VRFs.
- **Device roles**: edge, control plane, border, and wireless roles.

A practical workflow usually looks like this:

1. Create the site hierarchy in Catalyst Center.
2. Define IP pools for infrastructure and client segments.
3. Assign devices to the fabric site.
4. Select the fabric role for each device.
5. Provision the fabric so the controller pushes underlay, overlay, and policy settings.
6. Validate registration, tunnels, and endpoint learning.

> <mark>Important:</mark> Fabric provisioning is not just device discovery. The controller must know the site design, IP pools, and intended roles before it can generate a consistent fabric.

**Fabric node roles and behavior**

Fabric roles determine what each node does in the overlay and how it participates in control-plane and policy functions.

- **Fabric edge node**
  - Connects wired endpoints and access switches into the fabric.
  - Performs VXLAN encapsulation and decapsulation for client traffic.
  - Often provides the anycast gateway for client VLANs.
  - Uses DHCP relay or local DHCP services depending on design.

- **Control plane node**
  - Maintains LISP endpoint-to-location mappings.
  - Answers map requests from fabric nodes.
  - Can be redundant by deploying multiple control plane nodes.
  - Should be sized for endpoint scale and resiliency.

- **Border node**
  - Connects the fabric to external routing domains.
  - Exchanges routes between the fabric and non-fabric networks.
  - Common types are:
    - **Internal border**
    - **External border**
    - **Dual border**
  - Dual border combines internal and external connectivity in the same node when the design and platform support it.

- **Wireless controller role**
  - Integrates wireless clients into the fabric.
  - Maps wireless client identity and policy into the fabric model.
  - Supports fabric-enabled SSIDs and roaming across fabric access points.

Role placement matters. For example, a control plane node is not the same as a border node, and a border node is not simply “any device that connects outside.” The role determines the control-plane behavior, route exchange, and policy scope.

**Fabric provisioning workflow**

Catalyst Center uses a guided workflow to provision the fabric. The exact UI labels can vary by release, but the intent is consistent.

A representative workflow includes:

1. **Create the fabric site**
   - Define the site in the hierarchy.
   - Associate the site with the fabric intent.

2. **Define pools and addressing**
   - Allocate underlay link pools.
   - Allocate loopback pools for node identifiers.
   - Allocate client subnet pools for VN segments.

3. **Assign device roles**
   - Mark devices as edge, control plane, border, or wireless.
   - Confirm that the device software and platform support the selected role.

4. **Provision the fabric**
   - The controller pushes underlay routing, LISP, VXLAN, and TrustSec-related configuration.
   - The controller also programs anycast gateway and VRF/VN mappings where required.

5. **Validate the fabric**
   - Check node registration.
   - Check endpoint learning.
   - Check route exchange at the border.
   - Check policy deployment.

A common exam point is that provisioning is controller-driven, but the fabric still depends on correct underlay reachability. If the underlay is broken, overlay registration and endpoint mobility fail.

**Underlay design and node onboarding**

The underlay must provide stable IP reachability between all fabric nodes. It is usually built with routed links and a dynamic routing protocol such as OSPF or IS-IS. The exact design depends on the platform and topology, so avoid assuming one universal interface setting for every fabric link.

Common underlay requirements include:

- Unique loopback addresses for node identity.
- Routed point-to-point links between fabric nodes.
- Consistent MTU to support VXLAN overhead.
- Reachability to the control plane node and border nodes.
- Dynamic routing adjacency across the fabric transport.

A typical underlay configuration pattern on a routed link might look like this:

```cisco
interface GigabitEthernet1/0/1
 description Fabric underlay link
 no switchport
 ip address 10.0.12.1 255.255.255.252
 ip ospf 10 area 0
 ip mtu 9100
```

Loopbacks are commonly used for router IDs and overlay reachability:

```cisco
interface Loopback0
 ip address 10.255.0.11 255.255.255.255
```

If you use OSPF, you may advertise loopbacks and routed links in the underlay process:

```cisco
router ospf 10
 router-id 10.255.0.11
 passive-interface default
 no passive-interface GigabitEthernet1/0/1
 network 10.0.12.0 0.0.0.3 area 0
 network 10.255.0.11 0.0.0.0 area 0
```

> <mark>Important:</mark> Underlay design is topology- and platform-dependent. Do not treat any single interface command as mandatory for all fabric links.

**Edge node behavior and client attachment**

Fabric edge nodes connect access ports to the overlay. They map client VLANs into the fabric virtual network and provide the first-hop gateway for endpoints.

Key edge behaviors include:

- **Anycast gateway**
  - Multiple fabric edge nodes can present the same gateway IP and MAC.
  - This supports seamless mobility when clients move between access switches.

- **VLAN-to-VN mapping**
  - A client VLAN is associated with a virtual network.
  - The VN maps to a VRF, which provides routing separation.

- **DHCP relay**
  - The edge node often relays DHCP requests to the correct server.
  - The relay target may be inside the same VN or external, depending on design.

- **Endpoint registration**
  - The edge learns the endpoint and registers its location through the control plane.
  - The fabric can then move the endpoint without changing the client’s IP in many cases.

A representative access-side configuration pattern might look like this:

```cisco
interface Vlan10
 description Users
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 192.168.100.10
 fabric forwarding mode anycast-gateway
```

On many Catalyst platforms, the controller generates much of this configuration. The important exam concept is the relationship between the client subnet, the VN/VRF, and the fabric edge role.

**Control plane node operation**

The control plane node is the LISP mapping system for the fabric. LISP, or Locator/ID Separation Protocol, separates endpoint identity from location. Fabric nodes register endpoint reachability with the control plane, and other nodes query it when they need to forward traffic.

Control plane responsibilities include:

- Maintaining endpoint-to-edge mappings.
- Answering map requests from fabric nodes.
- Supporting mobility when endpoints move between edges.
- Providing redundancy when multiple control plane nodes are deployed.

A control plane node is not a transit router in the traditional sense. Its job is to maintain the mapping database that lets the fabric locate endpoints efficiently.

Useful verification commands include:

```cisco
show lisp session
show lisp database
show lisp map-cache
show lisp site
```

Depending on platform and software release, the exact output varies. The exam focus is on what the commands prove:

- The node is registered to the control plane.
- Endpoint mappings exist.
- The map-cache contains active destinations.
- The site and instance information match the intended fabric design.

**Border node types and route exchange**

Border nodes connect the fabric to external routing domains such as the campus core, data center, WAN, or Internet edge. Cisco commonly describes border nodes as internal border, external border, or dual border.

- **Internal border**
  - Connects fabric segments to internal enterprise routing domains.
  - Often used when the fabric must exchange routes with another internal network.

- **External border**
  - Connects the fabric to external domains such as WAN or Internet edge.
  - Often used for northbound connectivity outside the campus fabric.

- **Dual border**
  - Performs both internal and external border functions when the design supports it.
  - Useful when a single node must exchange routes with multiple domains.

Border nodes commonly use BGP, but the exact routing protocol depends on the design. The controller orchestrates the route exchange and VRF associations. A border node must advertise the correct prefixes for each VN and import the correct external routes.

A representative BGP pattern for a border node might look like this:

```cisco
router bgp 65001
 bgp log-neighbor-changes
 neighbor 172.16.0.2 remote-as 65100
 !
 address-family ipv4 vrf CORP
  neighbor 172.16.0.2 activate
  redistribute connected
 exit-address-family
```

This is only a simplified example. In real SD-Access deployments, the controller typically generates additional VRF, route-target, and policy-related configuration.

Verification on a border node often includes:

```cisco
show ip route vrf CORP
show bgp ipv4 unicast vrf CORP
show lisp service
show lisp eid-table
```

**TrustSec policy model**

Cisco SD-Access uses Cisco TrustSec to apply scalable segmentation and policy. The core policy object is the SGT, which identifies the security group of an endpoint or traffic flow. SGACLs define what traffic is permitted or denied between groups.

Policy concepts to know:

- **SGT**
  - Security Group Tag.
  - Represents the role or trust level of an endpoint.

- **SGACL**
  - Security Group Access Control List.
  - Defines permit/deny behavior between SGT pairs.

- **Policy matrix**
  - The controller defines which SGTs can communicate.
  - The fabric enforces the resulting policy where traffic is switched or routed.

- **Propagation**
  - SGT information can be carried inline in supported paths.
  - If inline tagging is not available, SXP may be used in some designs to propagate tags between devices.

The policy model is not just “a separate policy plane.” In SD-Access, policy is tied to the fabric control and data planes, with Catalyst Center orchestrating the policy intent and TrustSec mechanisms carrying and enforcing it.

A simple policy intent example is:

- Users SGT can access App SGT on TCP 443.
- Users SGT cannot access Admin SGT.
- Printers SGT can reach print services but not user subnets.

**SGT propagation and enforcement**

SGT propagation can occur in different ways depending on the path and device capabilities.

- **Inline tagging**
  - The tag is carried in the packet path where supported.
  - Common in fabric-capable paths.

- **SXP**
  - Security Group eXchange Protocol.
  - Used to exchange SGT-to-IP bindings between devices that do not carry inline tags.
  - Helpful at boundaries or with legacy devices.

- **Enforcement points**
  - Fabric edge nodes can enforce policy for wired clients.
  - Wireless fabric paths can enforce policy for wireless clients.
  - Border nodes may enforce policy when traffic crosses fabric boundaries.
  - Non-fabric devices may rely on SXP-fed mappings and local ACL behavior.

A practical way to think about it is:

- The controller defines the policy.
- The fabric learns who the endpoint is.
- The data path carries the SGT or binding.
- The enforcement point applies the SGACL.

Example TrustSec object configuration on a device may look like this:

```cisco
cts role-based sgt-map 10 USERS
cts role-based sgt-map 20 APP
cts role-based permissions from 10 to 20 permit tcp eq 443
cts role-based permissions from 10 to 20 deny ip
```

The exact syntax varies by platform and release, but the exam focus is on the relationship between SGTs, mappings, and permissions.

**Wireless fabric integration**

Wireless in SD-Access extends the fabric to wireless clients through fabric-enabled access points and controller integration. The wireless controller participates in the fabric design so that wireless clients receive the same segmentation and mobility benefits as wired clients.

Important wireless concepts include:

- **Fabric SSID**
  - The SSID is associated with a VN and policy profile.
  - The controller maps the wireless client into the fabric segmentation model.

- **Identity-based access**
  - 802.1X is common for identity-driven policy.
  - Web authentication may also be used in some designs.
  - PSK-based designs do not typically provide the same identity-centric policy model as 802.1X.

- **Roaming**
  - Clients can roam between access points while retaining policy and, in many cases, IP continuity.
  - The fabric updates the client location without requiring a full readdressing event.

- **Policy mapping**
  - The wireless controller and fabric edge cooperate to apply the correct VN and SGT.

A representative wireless design flow is:

1. Define the wireless policy profile in Catalyst Center.
2. Map the SSID to the correct VN.
3. Associate the authentication method with the intended identity source.
4. Assign the SGT or policy profile for the user group.
5. Provision the wireless fabric and validate client onboarding.

**Verification commands and what they prove**

Verification is a major exam topic. You should know how to confirm underlay, overlay, and policy state.

Useful commands include:

```cisco
show cdp neighbors
show ip ospf neighbor
show ip route
show lisp session
show lisp map-cache
show lisp database
show vxlan tunnel
show fabric edge
show fabric border
show fabric control-plane
show cts role-based permissions
show cts role-based sgt-map
```

What to look for:

- **Underlay**
  - Routing neighbors are up.
  - Loopbacks are reachable.
  - MTU is sufficient for overlay traffic.

- **Overlay**
  - LISP sessions are established.
  - Endpoint mappings exist.
  - VXLAN tunnels are up.

- **Policy**
  - SGT mappings are present.
  - SGACL permissions match the intended policy.
  - Traffic is being tagged and enforced as expected.

Catalyst Center also provides operational views that show fabric node health, endpoint location, and policy deployment status. Those views are often the fastest way to confirm whether the controller successfully pushed intent to the fabric.

**Common provisioning failures and troubleshooting**

Fabric provisioning issues usually fall into a few categories.

- **Underlay discovery failure**
  - The device is not reachable from the controller.
  - Routing adjacency is missing.
  - Loopback or transit addressing is incorrect.
  - MTU mismatch prevents overlay formation.

- **Control plane registration failure**
  - The node cannot reach the control plane.
  - LISP parameters do not match.
  - The node role is not correctly assigned.

- **Border route leak failure**
  - The border does not import or export the expected prefixes.
  - VRF associations are wrong.
  - External routing adjacency is down.
  - Route targets or redistribution are incomplete.

- **Policy push failure**
  - SGTs or SGACLs are not deployed to the correct devices.
  - The device does not support the required TrustSec feature.
  - The policy matrix is incomplete or inconsistent.

- **Wireless onboarding failure**
  - The SSID is not mapped to the correct VN.
  - Authentication does not match the intended policy model.
  - The controller or AP is not fabric-enabled.

A practical troubleshooting sequence is:

1. Confirm underlay reachability with routing and ping tests.
2. Confirm node role assignment in Catalyst Center.
3. Check LISP registration and map-cache state.
4. Verify VXLAN tunnel formation.
5. Confirm border route exchange.
6. Validate SGT and SGACL deployment.
7. Test endpoint mobility and policy enforcement.

Example checks:

```cisco
ping 10.255.0.1 source 10.255.0.11
show ip ospf neighbor
show lisp session
show lisp map-cache
show bgp ipv4 unicast vrf CORP
show cts role-based permissions
```

**Exam-focused distinctions**

A few distinctions are especially important for CCNP-level questions:

- **Fabric provisioning** is controller-driven intent deployment, not manual per-device overlay construction.
- **Border node types** are internal, external, and dual border.
- **Control plane nodes** maintain LISP mappings and can be redundant.
- **Edge nodes** connect endpoints, provide anycast gateway behavior, and register clients into the fabric.
- **Policy** is based on TrustSec SGTs and SGACLs, with propagation methods such as inline tagging and SXP depending on the path.
- **Wireless fabric** is typically identity-driven and integrated with the same segmentation model as wired clients.

**Quick reference: fabric objects and their purpose**

| Object | Purpose | Typical exam focus |
|---|---|---|
| Site hierarchy | Organizes the fabric by location | Design and provisioning workflow |
| IP pool | Supplies underlay and client addressing | Correct allocation and reuse |
| VN / VRF | Segments traffic | Mapping clients to policy domains |
| Edge node | Connects endpoints to the fabric | Anycast gateway and endpoint registration |
| Control plane node | Stores endpoint mappings | LISP registration and redundancy |
| Border node | Exits the fabric | Route exchange and border type |
| SGT | Identifies security groups | Policy classification |
| SGACL | Enforces group policy | Permit/deny behavior |

**Practical mental model**

When you study SD-Access, keep this sequence in mind:

- The controller designs the fabric.
- The underlay provides reachability.
- The edge learns the endpoint.
- The control plane records the location.
- The border exchanges routes with external domains.
- TrustSec applies policy based on SGTs.

If one layer is wrong, the next layer usually fails in a predictable way. That is why troubleshooting should always start with underlay and role assignment before moving to policy.
