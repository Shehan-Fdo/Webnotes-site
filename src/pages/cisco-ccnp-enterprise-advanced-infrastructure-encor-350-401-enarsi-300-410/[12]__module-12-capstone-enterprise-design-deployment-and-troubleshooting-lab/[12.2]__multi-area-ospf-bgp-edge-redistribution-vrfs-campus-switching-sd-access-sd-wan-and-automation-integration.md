## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Multi-area OSPF fundamentals**

OSPF (Open Shortest Path First) is a link-state IGP (Interior Gateway Protocol) that scales by dividing the topology into areas. Area 0 is the backbone, and all other areas must connect to it directly or through a virtual link in legacy designs. Multi-area design reduces SPF (Shortest Path First) calculation scope and limits LSA flooding.

- **Router ID** is a 32-bit identifier chosen from the highest loopback IP, then highest active interface IP if no loopback exists.
- **Neighbor adjacency** requires matching area ID, hello/dead timers, authentication, network type compatibility, and MTU behavior.
- **LSA** types matter for troubleshooting:
  - Type 1: Router LSA, generated within an area.
  - Type 2: Network LSA, generated by the DR (Designated Router).
  - Type 3: Summary LSA, generated by ABR (Area Border Router).
  - Type 4: ASBR Summary LSA, tells other areas how to reach the ASBR.
  - Type 5: External LSA, used for redistributed routes in normal areas.
  - Type 7: NSSA (Not-So-Stubby Area) external LSA, translated to Type 5 by the ABR.
- **ABR behavior** is central to multi-area OSPF because it summarizes and filters inter-area routes.
- **Passive interfaces** stop OSPF hellos on that interface, but the connected network can still be advertised if OSPF is enabled on the interface or via network statements.

A practical multi-area design often uses loopbacks for stable router IDs and management reachability. Loopbacks are commonly advertised as /32 host routes, which makes them ideal for stable endpoints and test reachability.

```cisco
router ospf 10
 router-id 10.10.10.10
 passive-interface default
 no passive-interface GigabitEthernet0/0
 no passive-interface GigabitEthernet0/1
 network 10.0.0.0 0.0.0.255 area 0
 network 10.1.1.1 0.0.0.0 area 10
 network 10.2.2.2 0.0.0.0 area 10
 area 10 range 10.1.0.0 255.255.0.0
```

- **`passive-interface default`** is useful on access-facing links and loopbacks because it suppresses neighbor formation while still allowing route advertisement.
- **`area range`** summarizes inter-area routes at the ABR and reduces LSDB size.
- **`network` statements** match interface IPs; on modern IOS XE, interface-level OSPF configuration is also common.

**OSPF network types and DR/BDR behavior**

OSPF network type affects neighbor discovery and DR/BDR election. This is a frequent exam topic because mismatched network types can prevent adjacency or change convergence behavior.

| Network type | DR/BDR election | Typical use | Notes |
|---|---:|---|---|
| Broadcast | Yes | Ethernet multiaccess | Uses multicast hellos and elects DR/BDR |
| Non-broadcast | Yes | Frame Relay, some NBMA designs | Manual neighbor statements often needed |
| Point-to-point | No | Serial links, tunnels | Simplest adjacency model |
| Point-to-multipoint | No | Hub-and-spoke overlays | Treats each neighbor as point-to-point |
| Point-to-multipoint non-broadcast | No | Legacy NBMA overlays | Manual neighbors may be required |

- **DR/BDR** reduce adjacency count on broadcast and NBMA networks.
- **Point-to-point** avoids DR/BDR overhead and is common on routed links and many tunnels.
- **Loopback interfaces** are advertised as host routes and do not form adjacencies, even if OSPF is enabled on them.

**OSPFv3 and IPv6 considerations**

OSPFv3 carries IPv6 routing information and uses link-local addresses for neighbor relationships. It is still important to understand the control-plane differences even when the exam focuses on IPv4.

- **OSPFv3** uses interface-based activation more often than network statements.
- **Link-local addresses** are used for next-hop and adjacency formation.
- **Instance ID** can separate multiple OSPFv3 processes on the same link.
- **Authentication** is handled differently from classic OSPFv2 and depends on platform support and configuration model.

```cisco
ipv6 unicast-routing

interface GigabitEthernet0/0
 ipv6 address 2001:db8:10:1::1/64
 ospfv3 10 ipv6 area 0

router ospfv3 10
 router-id 10.10.10.10
```

**OSPF troubleshooting and verification**

Use verification commands to confirm adjacency, LSDB content, and route installation.

```cisco
show ip ospf neighbor
show ip ospf interface brief
show ip ospf database
show ip route ospf
show ip protocols
```

- **`show ip ospf neighbor`** confirms adjacency state and DR/BDR roles.
- **`show ip ospf database`** helps identify missing LSAs or area mismatches.
- **MTU mismatch** commonly stalls adjacency in `EXSTART` or `EXCHANGE` on many platforms unless MTU checking is disabled on both sides.
- **Hello/dead timer mismatch** prevents adjacency unless both sides match.
- **Authentication mismatch** often appears as repeated neighbor resets.

> If you see neighbors stuck in `EXSTART` or `EXCHANGE`, check MTU first, then area type, timers, and authentication.

**BGP edge design and best-path selection**

BGP (Border Gateway Protocol) is the standard interdomain routing protocol and is also used inside enterprises for edge routing, VRFs, and route exchange between sites. It is policy-driven rather than purely metric-driven.

BGP neighbor establishment progresses through states such as Idle, Connect, Active, OpenSent, OpenConfirm, and Established. A stable session in `Established` is required before routes can be exchanged.

- **Weight** is Cisco-local and is evaluated early in best-path selection.
- **Local preference** influences outbound path choice within an AS (Autonomous System).
- **Locally originated routes** are preferred over learned routes.
- **AS_PATH** length is a major inter-AS tie-breaker.
- **Origin code** prefers IGP over EGP over incomplete.
- **MED** (Multi-Exit Discriminator) is compared when applicable and is often used to influence inbound traffic from a neighboring AS.
- **eBGP over iBGP** is preferred before IGP metric to next hop in the common Cisco decision process.
- **IGP metric to next hop** helps choose the closest exit when other attributes tie.
- **Oldest path, router ID, and neighbor IP** can appear later in the tie-break sequence depending on platform and conditions.

A practical way to think about BGP is that you first shape policy with attributes, then let the protocol choose the best path among acceptable candidates.

```cisco
router bgp 65001
 bgp log-neighbor-changes
 neighbor 203.0.113.2 remote-as 65002
 neighbor 203.0.113.2 description ISP-A
 neighbor 198.51.100.2 remote-as 65003
 neighbor 198.51.100.2 description ISP-B

 address-family ipv4
  network 10.10.0.0 mask 255.255.0.0
  neighbor 203.0.113.2 activate
  neighbor 198.51.100.2 activate
 exit-address-family
```

**BGP policy tools**

Use route maps, prefix lists, and AS-path access lists to control what BGP accepts and advertises.

- **Prefix lists** are efficient for matching prefixes and prefix lengths.
- **Route maps** are flexible and can match multiple conditions, then set attributes.
- **AS-path access lists** filter based on AS_PATH content.
- **Communities** tag routes for downstream policy decisions.

```cisco
ip prefix-list PL-DEFAULT seq 5 permit 0.0.0.0/0
ip prefix-list PL-CORP seq 10 permit 10.10.0.0/16 le 24

route-map RM-IN permit 10
 match ip address prefix-list PL-CORP
 set local-preference 200

route-map RM-OUT permit 10
 match ip address prefix-list PL-DEFAULT
 set community 65001:100 additive
```

- **`set local-preference`** is used inbound on iBGP-learned routes to influence exit choice.
- **`set community`** is useful for tagging routes for later policy.
- **`additive`** preserves existing communities.

**BGP verification and route visibility**

```cisco
show ip bgp
show ip bgp summary
show ip bgp neighbors 203.0.113.2 advertised-routes
show ip bgp neighbors 203.0.113.2 received-routes
show ip bgp neighbors 203.0.113.2 routes
```

- **`advertised-routes`** shows what you send to the neighbor.
- **`received-routes`** typically requires inbound soft reconfiguration or route refresh support and the right platform/context.
- **`routes`** shows routes accepted from that neighbor into the BGP table.
- **`show ip bgp`** reveals path attributes, origin code, next hop, and best-path selection.

**Default origination and aggregation**

Default route advertisement in BGP is nuanced. `neighbor x.x.x.x default-originate` advertises `0.0.0.0/0` only when the router has a default route, unless a route map is used to control or force origination.

```cisco
router bgp 65001
 address-family ipv4
  neighbor 203.0.113.2 default-originate
```

- **`default-originate`** is often used toward branches or downstream peers.
- **`aggregate-address`** summarizes multiple more-specific routes into a less-specific route.
- **`atomic-aggregate`** and **`aggregator`** attributes may appear with aggregation.

```cisco
router bgp 65001
 address-family ipv4
  aggregate-address 10.10.0.0 255.255.0.0 summary-only
```

**Redistribution between protocols**

Redistribution injects routes from one routing domain into another. It is powerful but risky because it can create loops, suboptimal paths, and route feedback if you do not tag and filter carefully.

- **Seed metric** is required by some protocols when redistributing into them.
- **Metric type** matters in OSPF redistribution:
  - Type 1: external cost plus internal cost to ASBR.
  - Type 2: external cost only, default in many IOS/IOS XE cases.
- **Route tags** help prevent loops when routes are redistributed back into the source domain.
- **Administrative distance** affects which route wins when multiple protocols advertise the same prefix.

**Redistribution into OSPF**

When redistributing into OSPF, you usually set the external metric and metric type under the `redistribute` command, not in the route map.

```cisco
router ospf 10
 redistribute bgp 65001 subnets metric 20 metric-type 1 route-map BGP-TO-OSPF
 redistribute connected subnets metric 50 metric-type 2 route-map CONN-TO-OSPF
```

- **`subnets`** is important when redistributing classless routes.
- **`metric-type 1`** is often preferred when you want internal cost to influence path selection.
- **`metric-type 2`** is common when external cost should dominate and internal cost should not accumulate.

**Redistribution into BGP**

BGP redistribution is usually done under the relevant address-family. In VRF contexts, the configuration belongs under that VRF address-family; global BGP uses the global address-family.

```cisco
router bgp 65001
 address-family ipv4
  redistribute ospf 10 route-map OSPF-TO-BGP
```

For a VRF:

```cisco
router bgp 65001
 address-family ipv4 vrf CORP
  redistribute ospf 10 route-map OSPF-TO-BGP
```

- **Global BGP** and **VRF BGP** are separate routing tables.
- **Route targets** are used for MPLS VPN route exchange, not for plain VRF-lite redistribution.
- **VRF address-family context** is required when redistributing routes for that VRF.

**Loop prevention with route tags**

A meaningful loop-prevention pattern is to tag routes when they enter a domain and deny those tags when they come back.

```cisco
route-map OSPF-TO-BGP permit 10
 match tag 200
 set community 65001:200 additive

route-map OSPF-TO-BGP permit 20
 set tag 200

route-map BGP-TO-OSPF deny 10
 match tag 200

route-map BGP-TO-OSPF permit 20
 set tag 100
```

- **Deny tagged routes** on re-entry to the original domain.
- **Tag consistently** so you can trace route origin during troubleshooting.
- **Filter at redistribution boundaries** rather than relying on downstream behavior.

> A route-map that only matches a tag and then permits it does not prevent loops by itself. You need explicit deny logic or a different filtering boundary.

**VRFs and VRF-lite**

VRF (Virtual Routing and Forwarding) creates separate routing tables on the same device. VRF-lite is VRF without MPLS label switching, commonly used in enterprise edge and campus segmentation.

Current IOS XE commonly uses `vrf definition` rather than legacy `ip vrf`.

```cisco
vrf definition CORP
 rd 65001:10
 route-target export 65001:10
 route-target import 65001:10
 address-family ipv4
 exit-address-family

interface GigabitEthernet0/1
 vrf forwarding CORP
 ip address 10.10.10.1 255.255.255.0
```

- **RD** (Route Distinguisher) makes overlapping prefixes unique in VPN contexts.
- **RT** (Route Target) controls import/export of VPN routes in MPLS L3VPN designs.
- **`vrf forwarding`** moves the interface into the VRF and clears the IP address on many platforms, so configure the address after applying the VRF.

**VRF routing behavior**

- **Connected routes** stay inside the VRF unless leaked.
- **Static routes** can point to next hops within the same VRF or across VRFs with special syntax.
- **Inter-VRF route leaking** can be done with static routes, route maps, or MP-BGP depending on design.

```cisco
ip route vrf CORP 0.0.0.0 0.0.0.0 10.10.10.254
ip route vrf CORP 192.0.2.0 255.255.255.0 GigabitEthernet0/2 10.20.20.2
```

**Route leaking examples**

A simple static leak between VRFs can be used for limited services, but it does not scale as well as policy-based designs.

```cisco
ip route vrf CORP 172.16.100.0 255.255.255.0 10.10.10.2
ip route vrf GUEST 10.10.10.0 255.255.255.0 10.20.20.2
```

For scalable VRF route exchange, MP-BGP is common in MPLS VPNs and some advanced enterprise designs.

- **MP-BGP** carries VPNv4/VPNv6 routes with RD and RT information.
- **VRF-lite** usually uses separate IPv4/IPv6 address-families without MPLS labels.
- **Route leaking** should be tightly controlled to avoid unintended lateral movement.

**Campus switching essentials**

Campus switching focuses on VLANs, trunks, spanning tree, first-hop redundancy, and edge protections. These features are foundational for ENCOR and often appear in troubleshooting scenarios.

**VLANs and trunks**

- **VLAN**s segment Layer 2 broadcast domains.
- **802.1Q trunking** carries multiple VLANs over one link.
- **Native VLAN** is untagged on 802.1Q trunks and should be consistent on both ends.
- **Allowed VLAN lists** reduce unnecessary flooding and improve security.

```cisco
vlan 10
 name USERS
vlan 20
 name VOICE

interface GigabitEthernet1/0/1
 switchport mode access
 switchport access vlan 10
 spanning-tree portfast
 spanning-tree bpduguard enable

interface GigabitEthernet1/0/48
 switchport mode trunk
 switchport trunk allowed vlan 10,20
 switchport trunk native vlan 999
```

**Spanning Tree Protocol**

STP (Spanning Tree Protocol) prevents Layer 2 loops. Rapid PVST+ is Cisco-proprietary and provides per-VLAN rapid convergence. MST (Multiple Spanning Tree) is IEEE standard and maps VLANs to instances for scale.

- **Root bridge placement** should be intentional and predictable.
- **PortFast** speeds up edge port forwarding.
- **BPDU Guard** shuts down an edge port if it receives BPDUs.
- **Root Guard** prevents an access switch from becoming root.
- **Loop Guard** protects against unidirectional link failures that could create loops.

```cisco
spanning-tree mode rapid-pvst

interface range GigabitEthernet1/0/1 - 24
 spanning-tree portfast
 spanning-tree bpduguard enable

interface GigabitEthernet1/0/48
 spanning-tree guard root
```

- **Rapid PVST+** is common in Cisco campus designs.
- **MST** is useful when many VLANs exist and you want fewer spanning-tree instances.
- **STP protections** are critical on user-facing ports and distribution uplinks.

**First-hop redundancy**

First-hop redundancy protocols provide a virtual default gateway for hosts.

| Protocol | Vendor | Load sharing | Notes |
|---|---|---:|---|
| HSRP | Cisco | Limited | Common in Cisco-only environments |
| VRRP | Standard | Limited | Interoperable across vendors |
| GLBP | Cisco | Yes | Provides gateway load balancing |

```cisco
interface Vlan10
 ip address 10.10.10.2 255.255.255.0
 standby 10 ip 10.10.10.1
 standby 10 priority 110
 standby 10 preempt
```

- **HSRP** uses a virtual IP and virtual MAC for gateway redundancy.
- **Preempt** allows the higher-priority router to reclaim active role after recovery.
- **Tracking** can reduce priority when an uplink fails.

**Campus security and edge controls**

- **Storm-control** limits broadcast, multicast, or unknown unicast storms.
- **DHCP snooping** helps block rogue DHCP servers.
- **Dynamic ARP Inspection** uses DHCP snooping bindings to validate ARP.
- **Port security** limits MAC addresses on access ports.

```cisco
interface GigabitEthernet1/0/10
 switchport mode access
 switchport access vlan 10
 storm-control broadcast level 1.00
 storm-control multicast level 1.00
```

**SD-Access architecture**

SD-Access (Software-Defined Access) uses a fabric to automate segmentation and policy in the campus. The fabric relies on control-plane and data-plane separation.

- **Fabric edge nodes** perform VXLAN (Virtual Extensible LAN) encapsulation and decapsulation for user traffic and register endpoints with the control plane using LISP (Locator/ID Separation Protocol).
- **Fabric control-plane nodes** maintain endpoint-to-location mappings.
- **Fabric border nodes** connect the fabric to external networks.
- **Fabric intermediate nodes** forward fabric traffic without endpoint registration duties.

The key idea is that endpoint identity is decoupled from location. This enables mobility and consistent policy enforcement.

- **VXLAN** provides the data-plane overlay.
- **LISP** provides endpoint mapping and control-plane registration.
- **Scalable Group Tags (SGTs)** and policy can be used for segmentation and access control.
- **Virtual Networks (VNs)** separate traffic logically, similar to VRFs in intent-based campus designs.

```text
Endpoint -> Fabric Edge -> VXLAN overlay -> Fabric Edge -> Destination
                 |             |
                 +-- LISP registration to control-plane
```

**SD-Access operational concepts**

- **Fabric edge** learns endpoints and registers them to the control plane.
- **Fabric border** handles external connectivity and can be internal or external border depending on design.
- **Anycast gateway** is commonly used so hosts keep the same default gateway across fabric nodes.
- **Policy** is often expressed through group-based access control rather than only VLANs and ACLs.

**SD-Access verification**

```cisco
show fabric summary
show lisp session
show lisp eid-table
show lisp map-cache
show cts role-based permissions
```

- **`show lisp eid-table`** helps confirm endpoint registration.
- **`show lisp map-cache`** shows learned mappings and forwarding state.
- **`show fabric summary`** gives a high-level fabric health view.

**SD-WAN architecture**

SD-WAN (Software-Defined WAN) separates management, control, and data functions across controllers and edge devices.

- **vManage** is the management and orchestration plane.
- **vSmart** is the control plane and distributes centralized policy.
- **vBond** is the orchestrator that helps with onboarding and initial connectivity.
- **cEdge** devices are IOS XE-based SD-WAN edges.
- **vEdge** devices are the original SD-WAN edge platform in many deployments.

Centralized policy is typically configured in vManage and enforced through vSmart. That distinction matters for exam questions.

- **Control connections** establish overlay reachability and policy distribution.
- **Data-plane tunnels** carry user traffic over transport links.
- **OMP** (Overlay Management Protocol) distributes routes, TLOCs (Transport Locator), and policies.

**OMP route types**

- **OMP routes** can represent service, transport, and VPN reachability depending on context.
- **TLOCs** identify how to reach an edge over a specific transport.
- **Centralized policy** can influence route preference, service insertion, and traffic steering.

```text
Branch cEdge <--- IPsec tunnel ---> vEdge/cEdge
      |                               |
      +---- OMP route exchange -------+
                 via vSmart
```

**SD-WAN verification**

```cisco
show sdwan control connections
show sdwan omp routes
show sdwan bfd sessions
show sdwan policy from-vsmart
show sdwan interface
```

- **BFD** (Bidirectional Forwarding Detection) is often used for fast path liveliness.
- **`show sdwan policy from-vsmart`** helps confirm received policy.
- **Transport tunnels** should be healthy before you expect stable overlay routing.

**Automation integration**

Automation is increasingly part of routing and switching validation. In ENCOR-level work, you should understand how APIs, YANG models, NETCONF, RESTCONF, and telemetry fit into operations.

- **NETCONF** is a structured configuration and state protocol that uses XML over SSH.
- **RESTCONF** exposes YANG-modeled data over HTTP/HTTPS and is often easier to integrate with web tooling.
- **YANG** defines the data model for configuration and operational state.
- **Streaming telemetry** pushes state updates to collectors with lower polling overhead than SNMP in many designs.

**NETCONF example**

NETCONF requires device support for the NETCONF server and the relevant YANG models.

```python
from ncclient import manager

with manager.connect(
    host="10.0.0.10",
    port=830,
    username="admin",
    password="cisco123",
    hostkey_verify=False,
    device_params={"name": "iosxe"},
) as m:
    print(m.get_config(source="running").xml)
```

- **`get_config`** retrieves configuration from a NETCONF-capable device.
- **Model support** determines what data you can retrieve or edit.
- **Transport and credentials** must be enabled on the device.

**RESTCONF example**

```bash
curl -k -u admin:cisco123 \
  https://10.0.0.10/restconf/data/ietf-interfaces:interfaces
```

- **RESTCONF** is useful for inventory, interface state, and configuration workflows.
- **JSON** is commonly used in RESTCONF responses, though XML may also be supported.

**Netmiko example for multiple show commands**

Use one command per `send_command()` call, or loop through commands.

```python
from netmiko import ConnectHandler

device = {
    "device_type": "cisco_ios",
    "host": "10.0.0.10",
    "username": "admin",
    "password": "cisco123",
}

commands = ["show ip ospf neighbor", "show ip bgp summary", "show ip route"]

with ConnectHandler(**device) as conn:
    for cmd in commands:
        output = conn.send_command(cmd)
        print(f"\n### {cmd}\n{output}")
```

- **`send_command`** is for a single command at a time.
- **`send_config_set`** is for configuration commands, not show commands.
- **Looping** is the simplest reliable way to collect multiple outputs.

**Practical troubleshooting workflow**

1. Verify the underlay first with interface status, routing adjacency, and reachability.
2. Confirm the control plane next with OSPF neighbors, BGP sessions, or SD-WAN control connections.
3. Check the data plane with route tables, next hops, and encapsulation state.
4. Validate policy last, including route maps, prefix lists, VRF import/export, and SD-Access or SD-WAN policy.

**Common exam-style scenarios**

- **OSPF adjacency fails** because of MTU mismatch, area mismatch, or authentication mismatch.
- **BGP routes are missing** because the prefix is not in the BGP table, the neighbor is not activated in the address-family, or outbound filtering blocks advertisement.
- **Redistribution loops** happen when routes are reintroduced without tags or filtering.
- **VRF traffic leaks unexpectedly** because of incorrect static routes or overly broad route import policy.
- **Campus loops** appear when edge protections like BPDU Guard or Root Guard are absent.
- **SD-Access endpoints do not appear** when fabric edge registration or control-plane mapping fails.
- **SD-WAN tunnels are up but traffic fails** when policy, route advertisement, or service-side reachability is missing.

**Key takeaways**

- **OSPF** scales with areas, summarization, and careful adjacency design.
- **BGP** is policy-driven and requires deliberate attribute control.
- **Redistribution** needs metrics, tags, and filtering to avoid loops.
- **VRFs** isolate routing tables and require explicit route leaking.
- **Campus switching** depends on VLANs, trunks, STP, and first-hop redundancy.
- **SD-Access** uses VXLAN and LISP to abstract endpoint location.
- **SD-WAN** separates management, control, and data planes with centralized policy.
- **Automation** ties everything together through APIs, YANG, and telemetry.
