## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Enterprise build overview**

An end-to-end enterprise network build ties together campus switching, routing, WAN edge, wireless, security, services, and operations into one design that can be deployed, verified, and troubleshot consistently. For CCNP Enterprise, you need to understand not just individual features, but where each feature belongs in the architecture and why one design choice is preferred over another.

A practical build usually starts with requirements:
- User access and segmentation
- Application reachability and performance
- High availability and failure domains
- Security policy and identity
- Manageability and observability
- Scalability for growth and change

The best exam answers usually reflect architecture first, then configuration details. If a feature can be placed at multiple layers, choose the placement that minimizes blast radius and simplifies operations.

**Campus architecture and role separation**

A classic enterprise campus uses a hierarchical model with **access**, **distribution**, and **core** layers. The access layer connects endpoints, the distribution layer aggregates access switches and enforces policy, and the core provides fast transport between major blocks.

Common design choices:
- **Three-tier campus** for large, geographically or functionally separated environments
- **Collapsed core** when distribution and core are combined for smaller or medium sites
- **Routed access** when you want Layer 3 at the access edge to reduce spanning-tree dependence and improve convergence
- **Multilayer campus** when you need both Layer 2 and Layer 3 boundaries for legacy or mobility requirements

Use routed access when you want:
- Smaller Layer 2 domains
- Faster failure recovery
- Better ECMP (equal-cost multipath) and summarization opportunities
- Cleaner policy boundaries

Use a collapsed core when:
- The site is smaller
- You want fewer devices and simpler operations
- Redundancy can still be achieved with a pair of distribution/core switches

A common campus design pattern is:
- Access switches run edge features such as port security, 802.1X, DHCP snooping, and QoS trust boundaries
- Distribution switches terminate VLANs or routed links, run first-hop redundancy, and apply inter-VLAN policy
- Core switches forward traffic with minimal policy and no unnecessary features

**End-to-end topology example**

A realistic enterprise build often includes:
- Dual campus core/distribution switches
- Multiple access blocks for users, voice, and IoT
- Wireless LAN controller or Catalyst 9800 pair
- WAN edge routers or SD-WAN edge devices
- Internet edge firewall pair
- Services such as DHCP, DNS, NTP, AAA, and logging
- Management network with out-of-band access

Example logical layout:
- Users in VLAN 10
- Voice in VLAN 20
- Guest in VLAN 30
- IoT in VLAN 40
- Network management in VLAN 99
- Server segment in VLAN 100
- Transit links between layers using /31 or /30 point-to-point subnets

A simple routed campus core example:
```text
Access SW1 ---\
               \        +-------------------+
Access SW2 -----+------->| Distribution/Core |
                        /+-------------------+
Access SW3 ---/        /
                      /   +-------------------+
                     /--->| WAN Edge / Firewall|
                    /     +-------------------+
                   /
              Wireless / Services / Internet
```

**Layer 2 and Layer 3 placement**

Layer 2 is still useful at the edge for endpoint attachment, VLAN membership, and mobility. Layer 3 is preferred for transport between major blocks because it improves convergence and reduces spanning-tree complexity.

Use Layer 2 where you need:
- Endpoint VLAN assignment
- Legacy devices that expect the same subnet
- Some wireless guest or mobility designs
- Simple access edge attachment

Use Layer 3 where you need:
- Fast convergence
- Route summarization
- Better fault isolation
- Reduced spanning-tree scope

Important design principle:
- Keep Layer 2 domains as small as practical
- Keep Layer 3 boundaries close to the edge when operationally feasible

**Routing design and summarization**

Enterprise routing commonly uses OSPF, EIGRP, or BGP depending on the domain. In campus and internal WAN designs, OSPF is common for inter-area routing and summarization, while BGP is common at the edge and in SD-WAN or multi-homed designs.

OSPF design notes:
- Place the backbone in area 0
- Use ABRs (Area Border Routers) to summarize inter-area routes
- Summarization reduces routing table size and limits topology churn

Example OSPF inter-area summarization on an ABR:
```cisco
router ospf 10
 router-id 1.1.1.1
 area 10 range 10.10.0.0 255.255.0.0
```

This command belongs on the ABR that connects area 10 to another area. The summary mask is a subnet mask, and the summarized prefixes must actually exist in the area being summarized.

EIGRP design notes:
- Classic EIGRP syntax is still seen in older networks
- Modern IOS XE designs often use named mode, which is more flexible and aligns better with current operations

Classic EIGRP example:
```cisco
router eigrp 100
 network 10.0.0.0 0.255.255.255
 no auto-summary
```

Named mode example:
```cisco
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  network 10.0.0.0 0.255.255.255
  af-interface default
   passive-interface
  exit-af-interface
 exit-address-family
```

Use named mode when:
- You want clearer multi-AF configuration
- You need more structured policy and interface control
- You are working in newer IOS XE environments

BGP design notes:
- Use BGP at the WAN edge, internet edge, or between administrative domains
- Use route filtering and summarization carefully
- Prefer stable policy boundaries over uncontrolled redistribution

**WAN edge and SD-WAN**

The WAN edge connects the enterprise to branches, data centers, cloud, and internet services. Traditional WAN designs often use static routes, OSPF, EIGRP, or BGP over MPLS, leased lines, or internet VPNs. Modern designs increasingly use **SD-WAN** (Software-Defined WAN) for centralized policy, application-aware path selection, and simplified branch onboarding.

Traditional WAN edge is a good fit when:
- The environment is small or stable
- You already operate a mature routing design
- You need direct control over routing policy and failover

SD-WAN is a good fit when:
- You need centralized orchestration
- You want application-aware steering across multiple transports
- You need segmentation across branches and shared services
- You want faster branch deployment and policy consistency

Common SD-WAN concepts:
- **Control plane** for route and policy distribution
- **Data plane** for encrypted traffic forwarding
- **OMP** (Overlay Management Protocol) in Cisco SD-WAN for route and policy exchange
- Transport independence across MPLS, broadband, and LTE/5G

A practical WAN design often includes:
- Primary MPLS or private transport
- Secondary internet transport
- Optional LTE/5G backup
- Dynamic routing or SD-WAN overlay
- Edge firewall or integrated security services

**Security placement and policy boundaries**

Security should be placed where it best protects trust boundaries without creating unnecessary bottlenecks. In an enterprise build, policy is usually enforced at the access edge, distribution boundary, WAN edge, internet edge, and management plane.

Typical security controls by zone:
- **Access edge**: 802.1X, MAB (MAC Authentication Bypass), DHCP snooping, Dynamic ARP Inspection, port security
- **Distribution/core boundary**: ACLs for inter-VLAN control, segmentation, and service isolation
- **WAN edge**: VPN, firewall policy, NAT, route filtering
- **Internet edge**: stateful inspection, NAT, DMZ controls, URL or threat services
- **Management plane**: AAA, SSH, SNMPv3, logging, and restricted access

ACL placement guidance:
- Use ACLs close to the source when you want to stop unwanted traffic early
- Use ACLs at the boundary when policy is based on zone or service
- Use firewall policy when you need stateful inspection, application awareness, or NAT integration

Example ACL for guest isolation:
```cisco
ip access-list extended GUEST-IN
 deny ip 10.30.0.0 0.0.255.255 10.0.0.0 0.255.255.255
 permit ip any any
```

Example interface application:
```cisco
interface Vlan30
 ip access-group GUEST-IN in
```

For firewall placement, think in terms of zones:
- Guest to internet only
- User to internal apps only
- Management to infrastructure only
- DMZ to internal services only when explicitly required

> Do not rely on ACLs alone for stateful internet or guest policy when a firewall is required by the design.

**DHCP, DNS, and core services**

Enterprise builds depend on services that are often overlooked in routing-focused study. DHCP and DNS are especially important because they affect endpoint onboarding and application resolution.

DHCP design considerations:
- Use a centralized DHCP server for consistency when the network is manageable
- Use DHCP relay (`ip helper-address`) on Layer 3 interfaces when clients are on remote subnets
- Exclude infrastructure addresses from pools
- Consider helper forwarding for multiple UDP services if needed

Example DHCP server configuration:
```cisco
ip dhcp excluded-address 10.10.10.1 10.10.10.20

ip dhcp pool USERS
 network 10.10.10.0 255.255.255.0
 default-router 10.10.10.1
 dns-server 10.100.100.10 10.100.100.11
 domain-name corp.example
 lease 7
```

Example DHCP relay:
```cisco
interface Vlan10
 ip address 10.10.10.1 255.255.255.0
 ip helper-address 10.100.100.50
```

DNS design considerations:
- Use internal DNS for internal hostnames and split-horizon records if needed
- Ensure clients can resolve both internal and external names appropriately
- Place DNS servers in a protected services segment
- Verify that DHCP hands out the correct DNS server addresses

Other common services:
- **NTP** for time synchronization
- **AAA** for authentication, authorization, and accounting
- **Syslog** for event collection
- **PKI** (Public Key Infrastructure) for certificates, especially wireless and VPN
- **TACACS+** or RADIUS for device and user authentication

Example NTP and logging:
```cisco
ntp server 10.100.100.20 prefer
logging host 10.100.100.30
service timestamps log datetime msec
```

**Wireless deployment and controller integration**

Modern enterprise wireless commonly uses lightweight access points that join a controller using CAPWAP (Control and Provisioning of Wireless Access Points). The controller centralizes policy, RF management, and client services.

Wireless deployment concepts:
- APs discover and join the controller using DHCP, DNS, or broadcast discovery
- CAPWAP tunnels carry control traffic and often client data depending on design
- AP modes may include local, FlexConnect, monitor, or bridge depending on site needs
- Policy is applied through WLANs, policy profiles, and AAA integration

Catalyst 9800-style design elements:
- **WLAN profile** defines SSID behavior
- **Policy profile** defines VLAN, ACL, QoS, and central switching behavior
- **AP profile** groups AP-specific settings
- **Policy tag** maps WLANs to policy profiles
- **Site tag** and **RF tag** control AP site and radio behavior

A simplified wireless build flow:
1. Define AAA and RADIUS for 802.1X authentication.
2. Create WLAN and policy profiles.
3. Map the WLAN to the correct policy profile and VLAN.
4. Ensure APs can join the controller over CAPWAP.
5. Verify client authentication, IP assignment, and roaming.

Example verification commands on a Catalyst 9800:
```cisco
show wireless summary
show ap summary
show wlan summary
show wireless client summary
show wireless client mac <client-mac>
```

Example conceptual WLAN configuration on modern platforms:
```cisco
wlan CORP-WIFI 10 CORP-WIFI
 security wpa psk set-key ascii 0 ExampleKey123
 no shutdown
```

For enterprise 802.1X, the WLAN usually uses RADIUS rather than PSK:
```cisco
aaa new-model
radius server ISE
 address ipv4 10.100.100.40 auth-port 1812 acct-port 1813
 key Cisco123
!
aaa authentication dot1x default group radius
aaa authorization network default group radius
```

Wireless operational considerations:
- Use 5 GHz and 6 GHz planning where supported
- Design RF coverage for capacity, not only signal strength
- Separate guest and corporate traffic with policy and segmentation
- Verify roaming behavior and authentication latency

**Access edge security and identity**

The access layer is where endpoint identity and basic trust enforcement begin. 802.1X is the primary method for port-based access control, while MAB is a fallback for devices that cannot do 802.1X.

Common access security features:
- **802.1X** for user/device authentication
- **MAB** for printers, phones, and IoT devices that lack supplicant support
- **Port security** for limiting MAC addresses
- **DHCP snooping** to block rogue DHCP servers
- **Dynamic ARP Inspection** to protect against ARP spoofing
- **BPDU Guard** to protect edge ports from rogue switches

Example access port configuration:
```cisco
interface GigabitEthernet1/0/10
 description User Access Port
 switchport mode access
 switchport access vlan 10
 spanning-tree portfast
 spanning-tree bpduguard enable
 authentication port-control auto
 mab
 dot1x pae authenticator
```

This is only part of a full 802.1X design. In practice, you also need AAA, RADIUS, and often critical VLAN or guest VLAN behavior.

Example supporting AAA configuration:
```cisco
aaa new-model
radius server ISE
 address ipv4 10.100.100.40 auth-port 1812 acct-port 1813
 key Cisco123
!
aaa authentication dot1x default group radius
aaa authorization network default group radius
dot1x system-auth-control
```

Operationally, you should verify:
- The port is in the correct VLAN or dynamic authorization state
- The client receives the expected role or VLAN
- RADIUS responses are successful
- Fallback behavior is correct when authentication fails

**Multicast design**

Multicast is used when one sender must efficiently reach many receivers, such as IPTV, market data, or some collaboration applications. Enterprise multicast design usually centers on PIM (Protocol Independent Multicast) and an RP (Rendezvous Point) for sparse-mode operation.

Common multicast components:
- **IGMP** (Internet Group Management Protocol) on receiver access segments
- **PIM sparse mode** on routed interfaces
- **RP** for group registration and tree building
- Optional **SSM** (Source-Specific Multicast) for simpler source-based delivery

Example multicast configuration:
```cisco
ip multicast-routing

interface Vlan50
 ip pim sparse-mode
 ip igmp version 2

interface GigabitEthernet0/0
 ip pim sparse-mode

ip pim rp-address 10.100.100.60
```

Operational checks:
```cisco
show ip igmp groups
show ip mroute
show ip pim neighbor
show ip pim rp mapping
```

Design guidance:
- Enable multicast only where needed
- Keep receiver segments well-defined
- Use SSM when applications support it because it reduces RP dependence
- Verify that Layer 2 switches support multicast snooping if required

**QoS design and trust boundaries**

QoS (Quality of Service) protects important traffic when links are congested. In enterprise builds, QoS is usually applied at the edge and WAN, not everywhere.

Typical QoS goals:
- Prioritize voice and interactive video
- Protect control traffic
- Prevent bulk traffic from starving critical applications
- Mark traffic consistently at trust boundaries

Common QoS actions:
- Classify traffic using ACLs, NBAR, or DSCP values
- Mark traffic at the edge
- Queue and schedule on congested links
- Police or shape traffic where appropriate

Example trust boundary on an access port:
```cisco
interface GigabitEthernet1/0/20
 mls qos trust dscp
```

Example MQC (Modular QoS CLI) policy:
```cisco
class-map match-any VOICE
 match dscp ef

policy-map WAN-OUT
 class VOICE
  priority percent 20
 class class-default
  fair-queue

interface GigabitEthernet0/0
 service-policy output WAN-OUT
```

Design guidance:
- Trust only known devices such as IP phones or managed uplinks
- Mark once, then preserve markings through the network
- Shape at WAN edges when the provider circuit is slower than the LAN
- Use policing carefully because it can drop bursts abruptly

**Monitoring, telemetry, and troubleshooting**

An enterprise build is not complete until you can observe it. Monitoring tools help you validate design intent and troubleshoot failures quickly.

Core monitoring tools:
- **SNMP** for polling and traps
- **Syslog** for event logging
- **NetFlow/Flexible NetFlow** for traffic visibility
- **SPAN** for local packet capture
- **RSPAN** for remote switched port analysis
- **ERSPAN** for encapsulated remote capture across Layer 3
- **IP SLA** for active path testing
- **NTP** for consistent timestamps

Example SNMPv3 configuration:
```cisco
snmp-server group NMS v3 priv
snmp-server user nmsuser NMS v3 auth sha AuthPass123 priv aes 128 PrivPass123
snmp-server host 10.100.100.70 version 3 priv nmsuser
```

Example SPAN session:
```cisco
monitor session 1 source interface GigabitEthernet1/0/10 both
monitor session 1 destination interface GigabitEthernet1/0/48
```

Example Flexible NetFlow:
```cisco
flow record CORP-REC
 match ipv4 source address
 match ipv4 destination address
 match transport source-port
 match transport destination-port
 collect counter bytes
 collect counter packets

flow exporter CORP-EXP
 destination 10.100.100.80
 source GigabitEthernet0/0
 transport udp 2055

flow monitor CORP-MON
 record CORP-REC
 exporter CORP-EXP

interface GigabitEthernet0/0
 ip flow monitor CORP-MON input
```

Verification commands:
```cisco
show flow monitor CORP-MON cache
show ip traffic
show logging
show snmp user
show monitor session 1
```

Troubleshooting approach:
1. Confirm physical and link status.
2. Verify addressing, VLANs, and routing.
3. Check policy enforcement points.
4. Validate services such as DHCP, DNS, and AAA.
5. Inspect logs, counters, and captures.
6. Test end-to-end from client to application.

**Integrated build example**

A practical enterprise build ties all components together. The campus provides user access and segmentation, the wireless system extends identity-based access, the WAN edge connects branches and cloud, and the security layer enforces trust boundaries. Services such as DHCP, DNS, NTP, and AAA support the entire environment, while monitoring confirms that the design works under real traffic.

Example integrated design choices:
- Access switches use 802.1X, DHCP snooping, and BPDU Guard
- Distribution switches provide SVIs, inter-VLAN routing, and ACLs
- Core switches forward transit traffic with minimal policy
- Wireless uses 802.1X with RADIUS and policy profiles
- WAN edge uses BGP or SD-WAN depending on scale and operational model
- Internet edge uses firewall policy and NAT
- Management uses a separate VLAN or VRF with restricted access

Example end-to-end configuration fragments:
```cisco
! SVI and DHCP relay
interface Vlan10
 description Users
 ip address 10.10.10.1 255.255.255.0
 ip helper-address 10.100.100.50
 ip access-group USERS-IN in

! OSPF on routed uplink
interface GigabitEthernet0/1
 no switchport
 ip address 10.255.0.1 255.255.255.252
 ip ospf 10 area 0

router ospf 10
 router-id 1.1.1.1
 passive-interface default
 no passive-interface GigabitEthernet0/1
 network 10.10.10.0 0.0.0.255 area 10
 network 10.255.0.0 0.0.0.3 area 0
```

This kind of build is useful for exam prep because it forces you to think about:
- Where routing boundaries belong
- How clients get addresses and names
- Where identity is enforced
- How traffic is secured and observed
- How the network recovers from failure

**Verification checklist**

Use a structured checklist after deployment:
- Confirm all uplinks are up and routed correctly
- Verify routing adjacencies and route propagation
- Test DHCP, DNS, and default gateway reachability
- Validate 802.1X, MAB, and guest fallback behavior
- Confirm wireless AP join status and client authentication
- Test WAN failover and application path selection
- Verify ACL and firewall policy matches the intended zones
- Check SNMP, syslog, NetFlow, and NTP
- Capture traffic if symptoms are unclear

Useful commands:
```cisco
show ip interface brief
show ip route
show ip ospf neighbor
show etherchannel summary
show spanning-tree summary
show authentication sessions
show wireless client summary
show bgp summary
show access-lists
show policy-map interface
```

**Common design mistakes**

- Making Layer 2 domains too large
- Placing policy in the wrong layer
- Using ACLs where stateful firewall policy is required
- Forgetting DHCP relay on remote VLANs
- Overlooking DNS and NTP during build-out
- Trusting QoS markings from untrusted endpoints
- Deploying wireless without validating AAA and AP join behavior
- Applying summarization without confirming the actual route set
- Treating SD-WAN as only a transport replacement instead of a policy framework

**Exam focus summary**

For CCNP Enterprise, the key is to connect features to architecture:
- Campus design determines where routing, redundancy, and policy live
- WAN design determines how branches and cloud connect and fail over
- Security design determines where trust boundaries are enforced
- Services design determines whether clients can join, resolve, and authenticate
- Monitoring design determines whether you can prove the network works

If you can explain the role of each layer and then map the correct Cisco feature to that role, you are thinking like a network engineer rather than memorizing isolated commands.
