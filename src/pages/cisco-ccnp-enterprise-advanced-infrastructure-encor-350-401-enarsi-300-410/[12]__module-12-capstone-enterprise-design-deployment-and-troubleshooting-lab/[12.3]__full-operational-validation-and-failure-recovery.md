## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Operational validation mindset**

Operational validation is the discipline of proving that a network change, recovery action, or failover event actually works in production conditions. You do not stop at “the config is in place”; you verify control plane, data plane, and service behavior after the change.

A good validation plan answers three questions:

- Did the intended state take effect?
- Did anything unintended break?
- Can you recover quickly if the change fails?

In enterprise environments, validation must cover routing, switching, security, management access, and application reachability. Recovery planning must also account for platform differences, storage locations, and feature support across IOS, IOS XE, and hardware families.

**Validation layers**

Think in layers so you do not miss hidden failures:

- **Physical and link layer**: interface status, optics, errors, speed/duplex, EtherChannel member health.
- **Layer 2**: VLAN membership, trunking, STP (Spanning Tree Protocol) state, MAC learning, port-channel forwarding.
- **Layer 3**: routing table, adjacency formation, next-hop resolution, VRF (Virtual Routing and Forwarding) separation.
- **Services**: DHCP, DNS, NAT, AAA (Authentication, Authorization, and Accounting), multicast, management-plane access.
- **Application path**: end-to-end reachability from the user or server perspective.

A change can look correct in the configuration and still fail at one of these layers. For example, a route may exist, but recursive next-hop resolution may fail; a trunk may be up, but the VLAN may be pruned; a NAT rule may be configured, but the translation may never be created for the tested flow.

**Post-change validation checklist**

Use a repeatable checklist after every change window.

1. Confirm the device is healthy.
   - Check CPU, memory, logs, and interface state.
   - Verify there are no unexpected reloads or crash messages.
2. Confirm the intended configuration is active.
   - Review the running configuration and relevant feature state.
   - Compare against the change plan or golden config.
3. Confirm control-plane convergence.
   - Verify routing neighbors, STP roles, HSRP/VRRP state, and adjacency tables.
4. Confirm data-plane forwarding.
   - Test traffic through the path you changed.
   - Validate translations, encapsulation, and policy behavior if applicable.
5. Confirm service behavior.
   - Test DNS resolution, DHCP lease assignment, AAA login, and management access.
6. Confirm rollback readiness.
   - Ensure you can restore the previous state if the change is unstable.

**Core verification commands**

Use commands that match the feature and platform. Some syntax varies by IOS/IOS XE release and hardware family, so always confirm the exact command set on the target device.

```bash
show version
show inventory
show logging
show processes cpu sorted
show memory statistics
show interfaces status
show interfaces counters errors
show ip interface brief
show ip route
show ip arp
show mac address-table
show spanning-tree
show etherchannel summary
show cdp neighbors detail
show lldp neighbors detail
```

For routing and forwarding validation, use targeted checks:

```bash
show ip route <prefix>
show ip cef <prefix>
show ip protocols
show ip ospf neighbor
show ip eigrp neighbors
show ip bgp summary
show ip bgp <prefix>
```

For VRF-aware validation, use the VRF context explicitly where supported:

```bash
show ip route vrf <VRF-NAME>
show ip arp vrf <VRF-NAME>
show ip cef vrf <VRF-NAME>
ping vrf <VRF-NAME> <destination>
traceroute vrf <VRF-NAME> <destination>
```

> VRF command syntax can differ across platforms and software trains. If a command is not accepted, verify the exact IOS XE syntax for that release.

**Layer 2 failure recovery validation**

Layer 2 issues often appear as “partial reachability” or “some hosts work, some do not.” Validate the switching path carefully.

- **STP convergence**: confirm the intended root bridge, port roles, and blocked ports.
- **EtherChannel/LACP**: verify the bundle is formed and all members are consistent.
- **Trunking**: confirm allowed VLANs, native VLAN, and encapsulation where relevant.
- **Access ports**: confirm the correct VLAN assignment and port state.
- **MAC learning**: verify the switch learns source MAC addresses on the expected port or port-channel.

Useful commands:

```bash
show spanning-tree vlan <vlan-id>
show spanning-tree root
show etherchannel summary
show lacp neighbor
show interfaces trunk
show interfaces <interface> switchport
show mac address-table dynamic
```

A practical EtherChannel validation sequence:

1. Check that the port-channel is up and bundled.
2. Confirm all member links are compatible.
3. Verify the trunk or access mode matches the design.
4. Test traffic across the bundle.
5. Shut one member link and confirm forwarding continues.

Example configuration:

```cisco
interface range GigabitEthernet1/0/1-2
 switchport
 switchport mode trunk
 channel-group 1 mode active

interface Port-channel1
 switchport
 switchport mode trunk
```

If one member fails, the bundle should remain forwarding if the remaining links are healthy and the channel parameters still match.

**Routing and failover validation**

Routing validation is not just “does the route exist?” You must confirm the correct path is selected and that failover occurs when the preferred path disappears.

Check:

- Route presence in the routing table.
- Next-hop reachability.
- Administrative distance and metric selection.
- Redistribution behavior, if used.
- Convergence after a link or neighbor failure.

Commands:

```bash
show ip route
show ip route <prefix>
show ip cef <prefix>
show ip ospf database
show ip ospf interface brief
show ip eigrp topology
show ip bgp neighbors
show ip bgp neighbors <neighbor-ip> advertised-routes
```

For BGP, remember that advertised-routes and received-routes are often context-sensitive and may require the correct address-family view. If the command is unavailable, check the relevant AFI/SAFI (Address Family Identifier/Subsequent Address Family Identifier) context.

A simple failover validation workflow:

1. Verify the primary path is active.
2. Confirm the backup path is present but not preferred.
3. Remove or fail the primary link.
4. Confirm the routing table converges to the backup path.
5. Test traffic end to end.
6. Restore the primary path and confirm the intended preference returns.

**HSRP, VRRP, and gateway redundancy**

First Hop Redundancy Protocols (FHRPs) protect the default gateway. Validate both the active role and the failover behavior.

For HSRP (Hot Standby Router Protocol):

- Confirm the active and standby routers.
- Verify the virtual IP address is reachable.
- Check tracking objects if configured.
- Test preemption behavior if the design requires it.

Useful commands:

```bash
show standby brief
show standby
show track
```

Tracking reduces priority; it does not automatically force a role change unless the priority drops below the peer and the preemption/role conditions allow it. Validate the actual election outcome, not just the tracking state.

Example HSRP configuration with tracking:

```cisco
interface GigabitEthernet0/0
 ip address 10.10.10.2 255.255.255.0
 standby 10 ip 10.10.10.1
 standby 10 priority 110
 standby 10 preempt
 standby 10 track GigabitEthernet0/1 20
```

Validation steps:

1. Confirm the active router owns the virtual IP.
2. Shut or fail the tracked interface.
3. Confirm priority decreases as expected.
4. Verify the standby router becomes active only if the election conditions are met.
5. Restore the interface and confirm preemption behavior.

**NAT validation**

Network Address Translation (NAT) validation must prove that the tested traffic is actually translated, not just that a translation table exists.

Check:

- Inside and outside interface roles.
- ACL or route-map match conditions.
- Translation creation for the specific flow.
- Return traffic path.
- VRF-aware behavior if NAT is used with VRFs.

Useful commands:

```bash
show ip nat statistics
show ip nat translations
show ip nat translations verbose
show access-lists
show ip route
```

A practical test sequence:

1. Generate traffic from an inside host to an outside destination.
2. Confirm a translation entry appears for that flow.
3. Verify the packet counters increase on the NAT rule or ACL.
4. Confirm the return traffic reaches the inside host.
5. Clear translations only if you need to retest from a clean state.

Example NAT configuration:

```cisco
interface GigabitEthernet0/0
 ip address 192.168.10.1 255.255.255.0
 ip nat inside

interface GigabitEthernet0/1
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

access-list 10 permit 192.168.10.0 0.0.0.255

ip nat inside source list 10 interface GigabitEthernet0/1 overload
```

**VPN and IPsec validation**

For modern IOS XE deployments, validate IKEv2 (Internet Key Exchange version 2) and IPsec state directly. IKEv1/ISAKMP commands still exist on some platforms, but they are legacy in many designs.

Useful commands:

```bash
show crypto ikev2 sa
show crypto ipsec sa
show crypto session
show crypto session detail
```

What to confirm:

- IKE SA (Security Association) is established.
- IPsec SA is active and encrypting traffic.
- Interesting traffic matches the crypto policy.
- Counters increase in both directions.
- Tunnel endpoints are reachable and routing is correct.

If the tunnel is up but traffic fails, check:

- ACL selectors or traffic match statements.
- NAT exemption.
- Routing symmetry.
- MTU and fragmentation issues.
- Firewall or policy restrictions.

**DMVPN validation**

Dynamic Multipoint VPN (DMVPN) validation usually involves multiple commands, not a single universal one. Validate the tunnel, NHRP (Next Hop Resolution Protocol), and crypto state together.

Useful commands:

```bash
show ip nhrp
show dmvpn
show crypto ikev2 sa
show crypto ipsec sa
show interface tunnel <id>
show ip route
```

Because platform support varies, do not rely on one command alone. Confirm:

- The tunnel interface is up/up.
- NHRP registrations are present.
- Spoke-to-hub and spoke-to-spoke reachability works as designed.
- Routing over the tunnel is stable.

**AAA, TACACS+, and RADIUS validation**

AAA validation should prove that authentication and authorization work for the actual login path you use.

Common checks:

```bash
show aaa servers
show tacacs
show radius statistics
test aaa group tacacs+ <username> <password> legacy
test aaa group radius <username> <password> legacy
```

Platform support varies, so use the command set available on the device. If a server is unreachable, validate:

- Source interface or source IP for AAA traffic.
- Routing to the AAA server.
- Shared secret consistency.
- UDP reachability for RADIUS or TCP reachability for TACACS+.
- Fallback behavior to local authentication.

Example AAA configuration:

```cisco
aaa new-model
tacacs server TAC1
 address ipv4 198.51.100.10
 key 0 MySecretKey

aaa group server tacacs+ TAC-GRP
 server name TAC1

aaa authentication login default group TAC-GRP local
aaa authorization exec default group TAC-GRP local
```

**DHCP and DNS service validation**

Service-level checks matter because the network can be “up” while users still cannot work.

For DHCP (Dynamic Host Configuration Protocol), validate:

- The client receives an address from the correct scope.
- The default gateway, DNS server, and lease time are correct.
- Relay agents forward requests to the right server.

Useful commands:

```bash
show ip dhcp binding
show ip dhcp pool
show ip dhcp server statistics
show ip helper-address
```

For DNS (Domain Name System), validate:

- Name resolution works from the client and from the device.
- The correct DNS server is reachable.
- Search domains and split-horizon behavior are correct if used.

Useful commands:

```bash
show hosts
ping <hostname>
nslookup <hostname>
```

**Multicast validation**

Multicast recovery is often overlooked, but it is important for enterprise video, discovery, and some routing designs. Validate both control plane and data plane.

Check:

- PIM (Protocol Independent Multicast) neighbor formation.
- Rendezvous Point (RP) reachability if used.
- IGMP (Internet Group Management Protocol) joins on receiver segments.
- Multicast routing table entries and forwarding state.

Useful commands:

```bash
show ip pim neighbor
show ip pim rp mapping
show ip igmp groups
show ip mroute
show ip mroute count
```

If multicast fails, check:

- Unicast routing to the RP or source.
- PIM mode consistency.
- ACLs and TTL thresholds.
- IGMP snooping on Layer 2 switches.

**Archive, checkpoint, and rollback workflows**

Rollback is your safety net. The exact syntax varies by platform, but the concepts are consistent: save a known-good state, make a change, and restore if needed.

Common IOS/IOS XE tools include:

- `archive` for automatic config snapshots.
- `configure replace` for restoring a saved configuration.
- `copy running-config startup-config` for persistence.
- `show archive` for history and status.

Example archive configuration:

```cisco
archive
 path bootflash:archive
 write-memory
 time-period 1440
```

This example uses `bootflash:` because that is common on many IOS XE platforms. Some devices use different storage names, so verify the filesystem before applying the command.

Example restore workflow:

```cisco
copy running-config bootflash:golden-config
configure replace bootflash:golden-config force
```

Important points:

- `configure replace` requires a valid configuration file path.
- `force` suppresses interactive prompts.
- Some platforms support `list` to preview differences before replacement.
- The storage path may be `bootflash:`, `flash:`, or another filesystem depending on the device.

A safer rollback procedure:

1. Save the current running config to a known location.
2. Confirm the backup file is readable.
3. Apply the change.
4. If validation fails, restore the saved config with `configure replace`.
5. Re-verify the critical services after rollback.

> Do not assume the same storage path exists on every platform. Always confirm the filesystem with `dir` or `show file systems` before using archive or restore commands.

**Legacy and modern recovery modes**

Platform recovery is broader than configuration rollback. You may need to recover from a bad image, a lost password, or a boot failure.

Common recovery scenarios:

- **ROMMON** (ROM Monitor) or boot loader recovery after a failed boot.
- **Password recovery** when local access is lost.
- **Image reinstallation** after corruption or an upgrade failure.
- **Rollback to a previous image** when the new release is unstable.

Typical recovery concepts:

- Interrupt boot and enter ROMMON/boot loader.
- Set the boot variable or boot manually from a known-good image.
- Copy a valid image to local storage.
- Verify checksums when possible.
- Save boot variables and reload.

Example boot variable checks:

```bash
show boot
show version
dir bootflash:
```

Example boot configuration on IOS XE:

```cisco
conf t
 boot system bootflash:cat9k_iosxe.17.09.04.SPA.bin
end
write memory
reload
```

If the device fails to boot the intended image, verify:

- The image file exists and is not corrupted.
- The boot variable points to the correct filename.
- The device has enough storage and memory for the image.
- The package or install mode matches the platform requirements.

Password recovery is platform-specific, but the general workflow is:

1. Access ROMMON or boot loader.
2. Change the boot behavior to bypass the startup configuration.
3. Boot the device.
4. Copy or modify the configuration as needed.
5. Restore normal boot behavior and reload.

Always follow the vendor procedure for the exact platform, because the steps differ between Catalyst, ISR, ASR, and Nexus families.

**Image upgrade and rollback planning**

Image changes are a common source of outages, so treat them like any other change.

Before upgrading:

- Verify compatibility with hardware and feature requirements.
- Check available storage and memory.
- Back up the running configuration and boot variables.
- Confirm the rollback image is present and bootable.
- Schedule a maintenance window and define success criteria.

During upgrade:

- Copy the image to local storage.
- Verify the checksum if available.
- Set the boot variable or install the package according to platform requirements.
- Reload only after confirming the image is staged correctly.

After upgrade:

- Confirm the device boots the expected image.
- Validate interfaces, routing, and management access.
- Check logs for crash, license, or feature warnings.
- Keep the previous image until the new release is stable.

**Practical IOS XE upgrade example**

```cisco
copy scp://user@198.51.100.20/cat9k_iosxe.17.09.04.SPA.bin bootflash:
verify /md5 bootflash:cat9k_iosxe.17.09.04.SPA.bin
conf t
 boot system bootflash:cat9k_iosxe.17.09.04.SPA.bin
end
write memory
reload
```

The exact upgrade workflow may differ if the platform uses install mode, package mode, or a different boot process. Use the platform-specific procedure rather than assuming a universal command sequence.

**Management-plane and service access validation**

A network change is not complete until you verify that operators can still manage the device and users can still reach critical services.

Validate:

- SSH access to the device.
- SNMP (Simple Network Management Protocol) polling and traps if used.
- Syslog delivery.
- NTP (Network Time Protocol) synchronization.
- AAA login and privilege escalation.
- Reachability to monitoring and backup servers.

Useful commands:

```bash
show ip ssh
show ntp status
show ntp associations
show logging
show snmp
```

Service-level validation examples:

- Can you SSH to the device from the management network?
- Can the NMS (Network Management System) poll interface counters?
- Are logs still reaching the syslog server?
- Is the clock synchronized so logs and certificates remain valid?

**Troubleshooting workflow after a failed change**

Use a disciplined sequence instead of random command execution.

1. Identify the symptom.
   - Is the failure total, partial, or intermittent?
2. Scope the impact.
   - One host, one VLAN, one VRF, one site, or the whole network?
3. Check the last change.
   - Compare the intended change with the actual running state.
4. Validate the control plane.
   - Neighbors, routes, adjacencies, and redundancy roles.
5. Validate the data plane.
   - Ping, traceroute, counters, translations, and encapsulation.
6. Roll back if the issue is not quickly isolated.
   - Restore the known-good config or image.
7. Re-test the original service.
   - Do not stop at “the device is reachable.”

**Practical end-to-end validation example**

Suppose you change a branch router that provides WAN access, NAT, HSRP, and DHCP relay.

```cisco
interface GigabitEthernet0/0
 description LAN
 ip address 10.10.10.2 255.255.255.0
 standby 10 ip 10.10.10.1
 standby 10 priority 110
 standby 10 preempt
 ip helper-address 192.0.2.50
 ip nat inside

interface GigabitEthernet0/1
 description WAN
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

access-list 10 permit 10.10.10.0 0.0.0.255
ip nat inside source list 10 interface GigabitEthernet0/1 overload
```

Validation sequence:

- `show standby brief` to confirm gateway redundancy.
- `show ip interface brief` to confirm interface state.
- `show ip nat statistics` and `show ip nat translations` to confirm translation activity.
- `show ip dhcp relay information` or relevant DHCP relay checks to confirm forwarding.
- `ping` and `traceroute` from a client to confirm end-to-end reachability.
- Shut the WAN interface or fail the upstream path and confirm the backup design behaves as expected.

**What exam scenarios often test**

Expect questions that ask you to choose the best verification or recovery action, not just the feature command.

Common themes:

- Which command best confirms the feature is active?
- Which check proves traffic is actually using the intended path?
- Which rollback method restores a known-good configuration fastest?
- Which recovery step is appropriate after a failed boot?
- Which validation step catches a Layer 2 or redundancy issue before users report it?

The safest answer usually combines configuration verification with traffic validation and a rollback plan.

**Quick reference**

- Use `show` commands to confirm state before and after a change.
- Validate both control plane and data plane.
- Test the actual user flow, not only the device state.
- Keep a known-good config and image ready for rollback.
- Treat platform syntax as variable, especially for archive, restore, AAA, VRF, and VPN commands.
- Include service checks such as DHCP, DNS, AAA, NTP, and management access in every post-change review.
