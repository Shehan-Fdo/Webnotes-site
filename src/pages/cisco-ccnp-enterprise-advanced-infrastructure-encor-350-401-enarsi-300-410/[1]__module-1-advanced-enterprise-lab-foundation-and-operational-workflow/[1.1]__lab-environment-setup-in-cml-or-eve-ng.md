## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Lab platform overview**

Cisco Modeling Labs (CML) and EVE-NG are the two most common virtual lab platforms for CCNP Enterprise practice. Both let you build repeatable topologies, connect virtual routers and switches, and test routing, switching, and troubleshooting workflows without physical hardware.

Use **CML** when you want a supported Cisco platform with a polished UI, built-in images, and simpler licensing. Use **EVE-NG** when you want broader image flexibility, more community content, and a lab environment that can scale across many vendor images. In practice, many candidates choose based on image availability, host resources, and how much manual setup they are willing to manage.

**Platform selection and real-world decision**

A practical decision rule is:

- Choose **CML** if you want faster onboarding, fewer image-management tasks, and a cleaner path for Cisco-only study.
- Choose **EVE-NG** if you need mixed-vendor labs, custom images, or more control over templates and topology design.
- Choose **CML** if you prefer a supported subscription model and predictable appliance behavior.
- Choose **EVE-NG** if you are comfortable handling image conversion, permissions, and template mapping.

A common CCNP workflow is to use CML for quick routing and automation labs, then move to EVE-NG when you need more complex topologies or third-party appliances.

**Host and hypervisor prerequisites**

Both platforms depend on hardware virtualization. You should verify that the host CPU supports Intel VT-x or AMD-V and that virtualization is enabled in BIOS or UEFI.

Typical host considerations include:

- **CPU**: modern multi-core processor with hardware virtualization enabled
- **Memory**: 16 GB minimum for small labs; 32 GB or more is more realistic for CCNP practice
- **Storage**: SSD strongly preferred because image boot times and snapshot operations are much faster
- **Network**: at least one bridged or trunk-capable adapter for external connectivity
- **Hypervisor**: VMware ESXi, VMware Workstation/Player, VirtualBox, KVM, or bare-metal appliance deployment depending on platform

For host networking, separate management access from lab traffic whenever possible:

- **Management network**: used for browser access, SSH, and platform administration
- **Lab/data network**: used by virtual nodes for inter-device connectivity and external breakout

A simple mental model is:

```text
[Your laptop/PC] ---> [Management NIC] ---> [CML/EVE-NG management IP]
                         |
                         +--> [Bridged/trunk NIC] ---> [Lab nodes / external network]
```

> If you place lab traffic on the same network as your production LAN, be careful with bridging, DHCP, and accidental routing leaks.

**CML deployment and first login**

CML is usually deployed as a virtual appliance or on supported infrastructure. The exact packaging can vary by version, but the setup flow is consistent: deploy the appliance, assign resources, boot, and complete first-login configuration.

A typical CML deployment sequence is:

1. Download the CML appliance or installer from Cisco using your licensed account.
2. Deploy it on a supported hypervisor such as ESXi or a compatible virtualization host.
3. Allocate enough resources for your intended lab size.
4. Boot the appliance and wait for the services to initialize.
5. Open the web UI from the management IP.
6. Complete the initial admin setup and confirm licensing status.

Recommended starting resources for a small CCNP lab:

- **2 vCPU** for the appliance itself, more if you plan to run many nodes
- **8 GB RAM minimum** for the platform, with additional memory reserved for nodes
- **100 GB or more** of storage if you plan to keep multiple labs and images
- **Bridged management access** so you can reach the web UI from your workstation

CML licensing matters because the platform may restrict features or node counts depending on the license tier. Always confirm:

- the license is activated and visible in the UI
- the entitlement matches the version you installed
- the node count and image set you need are permitted

A common first-login checklist is:

- change the default password if prompted
- verify time, DNS, and NTP settings
- confirm the management IP is reachable
- check that the image catalog is available
- create a test lab and power on a small topology

**EVE-NG editions and deployment**

EVE-NG is available in different editions, and the edition affects setup and features. The most common distinction is between **Community** and **Professional** editions.

| Edition | Typical use | Key differences |
|---|---|---|
| Community | Personal study and basic labs | More manual setup, fewer convenience features, limited support |
| Professional | Heavier lab use and advanced workflows | Better multi-user and management features, more polished experience |

The exact feature set can vary by release, so verify the edition documentation before planning a large lab environment.

A typical EVE-NG deployment flow is:

1. Deploy the EVE-NG OVA or install it on a supported hypervisor or bare-metal host.
2. Assign CPU, RAM, and storage based on expected lab size.
3. Boot the system and access the web interface.
4. Complete the initial password and network configuration.
5. Upload images into the correct directories.
6. Fix permissions and create templates for each image type.

Recommended starting resources for CCNP practice:

- **4 vCPU** or more for the host VM
- **8–16 GB RAM minimum** for small labs
- **100–200 GB SSD storage** for images and saved labs
- **Promiscuous/bridged networking** if you need external connectivity or nested lab access

EVE-NG often runs well on:

- **VMware ESXi** for stable nested virtualization
- **VMware Workstation** for a desktop lab
- **KVM** for Linux-based hosts
- **VirtualBox** for small personal labs, though performance and networking can be less predictable

**Image workflow in EVE-NG**

EVE-NG is powerful because it supports many image types, but that flexibility means you must manage image names, formats, and templates carefully. A mismatch between the image filename, template type, and expected platform is one of the most common causes of boot failure.

Typical image workflow:

1. Upload the image to the correct folder.
2. Convert the image if required.
3. Set ownership and permissions.
4. Create or edit the node template.
5. Add the node to a lab and verify boot output.

Common Cisco image examples include:

- **IOSv** for basic routing labs
- **IOSvL2** for Layer 2 switching behavior
- **CSR1000v** for enterprise routing and VPN features
- **NX-OSv** for data center-style switching practice when available
- **ASAv** for firewall labs when supported and licensed

Example directory layout on EVE-NG:

```bash
/opt/unetlab/addons/qemu/
├── iosv-15.9/
├── iosvl2-15.2/
├── csr1000v-17.3/
└── nxosv-9.3/
```

Example permission fix after upload:

```bash
/opt/unetlab/wrappers/unl_wrapper -a fixpermissions
```

If an image does not boot, check these common causes:

- the image folder name does not match the template expectation
- the disk image format is wrong, such as raw versus qcow2
- the image is missing the expected filename, often `virtioa.qcow2` or a vendor-specific disk name
- the node template points to the wrong platform type
- the VM lacks nested virtualization support on the host

A practical troubleshooting pattern is to compare the template settings against a known-good image of the same family. For example, an IOSv template may boot with one disk layout, while a CSR1000v image expects a different disk name and more memory.

**CML image and node workflow**

CML is simpler because Cisco provides a curated image workflow, but you still need to understand how images and node definitions affect lab behavior. In many environments, you import or enable the images available to your license and then place nodes directly into the topology.

Typical CML workflow:

1. Log in to the web UI.
2. Open the image or node catalog.
3. Confirm the required Cisco images are available.
4. Create a lab and drag nodes into the canvas.
5. Connect interfaces and start the lab.
6. Open console sessions and validate reachability.

A small CCNP starter lab in CML might include:

- 2 x IOSv routers
- 1 x IOSvL2 switch
- 1 x management or external connector node

Example lab goal:

- R1 and R2 form an OSPF adjacency
- the switch provides a VLAN access segment
- one node reaches the host network through an external connector

**Hypervisor and network adapter considerations**

The host hypervisor affects both performance and connectivity. Nested virtualization must be enabled if the lab appliance itself runs as a VM and then hosts additional virtual routers.

Key considerations by platform:

- **ESXi**: strong choice for nested labs; use a port group that allows the required MAC and promiscuous settings when bridging lab traffic
- **VMware Workstation**: convenient for desktop use; ensure the VM network mode matches your lab design
- **VirtualBox**: workable for small labs; bridged adapters can be less predictable than on ESXi or Workstation
- **KVM**: efficient on Linux hosts; verify CPU passthrough and bridge configuration
- **Bare metal**: best performance when supported, but less flexible for quick snapshots of the whole appliance

For external connectivity, you usually need one of these patterns:

- **Bridged adapter** to place the lab on the same L2 segment as the host network
- **NAT** for isolated internet access without exposing lab nodes directly
- **Trunked bridge** if you want multiple VLANs to pass between the host and the lab

Firewall and port planning also matters. At minimum, allow:

- web UI access over **HTTPS**
- SSH for appliance administration if enabled
- any platform-specific console or management services
- outbound access for license checks, image downloads, or package updates

**Initial lab bring-up and validation**

A good first lab should prove that management access, node boot, interface wiring, and external connectivity all work before you build a larger topology.

Use this end-to-end sequence:

1. Deploy the appliance and confirm you can reach the web UI.
2. Verify licensing or edition status.
3. Create a small topology with two routers and one switch.
4. Connect router interfaces to the switch or directly to each other.
5. Assign management or lab IP addresses.
6. Start the nodes and open the console.
7. Confirm interface status and routing adjacency.
8. Test reachability with `ping` and `show` commands.

Example validation commands on Cisco IOS XE-style nodes:

```bash
show ip interface brief
show cdp neighbors
show lldp neighbors
show ip ospf neighbor
ping 10.0.0.2
traceroute 10.0.0.2
```

Example basic interface configuration:

```cisco
conf t
interface gigabitEthernet0/0
 ip address 10.0.0.1 255.255.255.0
 no shutdown
exit
router ospf 1
 network 10.0.0.0 0.0.0.255 area 0
end
write memory
```

If you are validating Layer 2 behavior, add a VLAN and access port test:

```cisco
conf t
vlan 10
 name USERS
interface gigabitEthernet0/1
 switchport mode access
 switchport access vlan 10
 no shutdown
end
show vlan brief
show interfaces status
```

**Backup, export, snapshot, and restore**

Preserving labs is essential when you build reusable topologies for exam practice. You do not want to rebuild the same underlay every time you study a new feature.

Useful preservation methods include:

- **Snapshots** for quick rollback before risky changes
- **Clones** for creating a copy of a known-good lab
- **Export/import** for moving labs between hosts or archiving them
- **Saved topologies** for reusing a standard CCNP practice design

Best practices:

- save a clean baseline after the lab first works
- keep one snapshot before major protocol changes
- export the lab after you finish a stable configuration set
- document image versions so the lab can be recreated later

A practical workflow is:

1. Build the base topology.
2. Verify connectivity.
3. Save a snapshot named `baseline-working`.
4. Clone the lab for a new scenario, such as redistribution or route filtering.
5. Restore the baseline if a change breaks the topology.

> Snapshots are not a substitute for image backups. If you lose the underlying image files, the lab may not restore correctly even if the topology file still exists.

**Troubleshooting common setup problems**

Most setup issues fall into a few categories: resource shortages, image mismatches, network misconfiguration, or licensing problems.

Common symptoms and likely causes:

- **Nodes boot slowly or fail to start**
  - insufficient RAM or CPU
  - nested virtualization disabled
  - wrong image type for the template
- **Web UI unreachable**
  - wrong management IP
  - host firewall blocking access
  - adapter attached to the wrong network
- **No external connectivity**
  - bridge not configured correctly
  - NAT or trunk settings missing
  - host firewall or upstream switch blocking traffic
- **Console opens but interfaces stay down**
  - wrong cabling in the topology
  - interface not enabled with `no shutdown`
  - image does not support the selected interface type

Useful host-side checks:

```bash
ip addr show
ip route
ping <management-ip>
ss -tulpn
```

Useful guest-side checks:

```cisco
show version
show platform
show interfaces
show ip route
show logging
```

**Example CCNP lab build**

A realistic starter lab for ENCOR or ENARSI practice might look like this:

- 1 x management workstation
- 1 x CML or EVE-NG appliance VM
- 2 x IOSv routers
- 1 x IOSvL2 switch
- 1 x external connector or bridged interface

Suggested resource plan:

| Component | vCPU | RAM | Notes |
|---|---:|---:|---|
| Lab appliance | 2-4 | 8-16 GB | More if many nodes run at once |
| IOSv router | 1 | 512 MB-1 GB | Depends on image version |
| IOSvL2 switch | 1 | 512 MB-1 GB | Keep small labs lightweight |
| CSR1000v | 2 | 3-4 GB | Heavier but useful for advanced features |

A simple bring-up sequence:

1. Deploy the appliance on ESXi, Workstation, KVM, or another supported host.
2. Confirm the management interface is reachable from your browser.
3. Import or enable IOSv and IOSvL2 images.
4. Create a lab with R1, R2, and SW1.
5. Connect R1 and R2 through SW1 or directly for a point-to-point test.
6. Assign IP addresses and enable interfaces.
7. Verify OSPF or static routing.
8. Add an external connector if you need host-to-lab access.
9. Save the working lab as a baseline snapshot.

Example end-to-end verification:

```cisco
R1# show ip interface brief
R1# show ip ospf neighbor
R1# ping 10.10.10.2

R2# show ip route ospf
R2# show cdp neighbors
```

**When to use each platform**

Use **CML** when you want a streamlined Cisco-centric lab with less image handling and a more guided setup. It is a strong choice for candidates who want to spend more time on protocol behavior and less time on platform maintenance.

Use **EVE-NG** when you need flexibility, mixed images, or a more customizable environment. It is especially useful when you want to practice with multiple Cisco platforms or build large reusable topologies for repeated exam drills.

A good rule of thumb is:

- **CML** for speed, simplicity, and Cisco-supported workflows
- **EVE-NG** for flexibility, scale, and image diversity

**Key takeaways**

A successful lab environment depends on more than just installing a VM. You need the right host resources, a clear separation between management and lab traffic, correct image handling, and a repeatable validation process.

Focus on these priorities:

- verify virtualization support before deployment
- size the host for the number and type of nodes you plan to run
- match images to templates carefully, especially in EVE-NG
- confirm licensing or edition limits early
- save snapshots and exports once the lab is stable
- validate with real Cisco commands before moving to advanced scenarios
