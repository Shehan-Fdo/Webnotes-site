## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Verification discipline: what Cisco expects you to do**

Verification is the habit of proving that a device, protocol, or service is in the state you intended. In Cisco enterprise environments, you do not troubleshoot by guessing; you compare observed state against intended state, then narrow the fault domain until you can act with confidence.

A strong workflow usually follows the Cisco-style sequence:

1. **Define the problem**
2. **Gather information**
3. **Consider possibilities**
4. **Act**
5. **Evaluate**

This is not just theory. It helps you avoid changing the wrong layer, especially when a symptom could come from routing, switching, security policy, or automation drift.

- **Define the problem** by stating the exact symptom, scope, and impact.
- **Gather information** from logs, counters, topology, and control-plane state.
- **Consider possibilities** by ranking likely causes from most to least probable.
- **Act** with the smallest safe change or test.
- **Evaluate** by verifying that the symptom is gone and no new issue appears.

> If you cannot describe the failure in one sentence, you are probably not ready to fix it.

**Core troubleshooting mindset**

Good troubleshooting is evidence-driven and reversible. You should prefer commands that show state over commands that only show configuration, because running configuration alone does not prove protocol health.

Useful habits:

- Start at the **highest layer that still shows the symptom**, then move downward only as needed.
- Compare **intended state** to **actual state**.
- Change one variable at a time so you can attribute the result.
- Keep a clear boundary between **local device issues**, **adjacent device issues**, and **network-wide issues**.
- Verify both **data plane** and **control plane** when the symptom is intermittent or asymmetric.

**A practical isolation model**

A clean way to isolate enterprise problems is to divide them into domains:

- **Physical and link**
  - Interface status, errors, speed/duplex, optics, cabling, PoE, MTU
- **Layer 2**
  - VLAN membership, trunking, STP, EtherChannel, MAC learning
- **Layer 3**
  - IP addressing, ARP/ND, routing table, VRF, recursion, next hop reachability
- **Policy and services**
  - ACLs, NAT, QoS, firewall, IPsec, PBR, DHCP
- **Control plane**
  - OSPF, EIGRP, BGP, HSRP/VRRP/GLBP, redistribution, summarization
- **Automation and management**
  - API state, telemetry, templates, config drift, orchestration output

This separation matters because a symptom like “host cannot reach the internet” may be caused by a missing default route, a NAT rule mismatch, an ACL, or a tunnel failure. Verification tells you which domain is actually broken.

**Define the problem clearly**

Before you touch the device, write down:

- What is broken?
- Who is affected?
- Since when?
- Is it constant or intermittent?
- Is it one direction or both?
- Is it all traffic or only specific prefixes, ports, or applications?

A precise problem statement might be:

- “Branch users in `VRF-CUST1` can reach local resources but cannot reach the data center after 14:20 UTC.”
- “OSPF adjacency between `R1` and `R2` is stuck in `EXSTART` after a maintenance change.”
- “BGP routes are present in the table but traffic is blackholed because the next hop is unreachable.”

That level of specificity helps you choose the right verification commands immediately.

**Gather information before changing anything**

Start with the simplest evidence:

- Interface state and counters
- Recent logs
- Neighbor relationships
- Routing table entries
- Policy matches
- Configuration diffs
- Automation job output

Common commands:

```bash
show clock
show logging
show interfaces status
show interfaces counters errors
show ip interface brief
show ipv6 interface brief
show running-config
show startup-config
show archive config differences
```

For a first-pass triage, you often want to answer these questions:

- Is the interface up/up?
- Is the address correct?
- Is the route present?
- Is the neighbor adjacent?
- Is the policy matching traffic?
- Is the tunnel or overlay established?
- Is the device actually running the intended config?

**Cisco-recognized workflow framing**

Cisco exam scenarios often reward a disciplined sequence that looks like this:

- **Identify symptoms**
- **Collect facts**
- **Form hypotheses**
- **Test the most likely cause first**
- **Confirm the fix**
- **Document the root cause**

This is especially useful when multiple technologies interact. For example, a routing failure may actually be caused by an ACL, a VRF leak, or a tunnel state issue. The workflow keeps you from overcorrecting.

**Layer 1 and Layer 2 verification**

Physical and switching issues are often the fastest to prove or eliminate. You should verify link state, error counters, VLAN membership, trunking, and spanning tree behavior before assuming a routing problem.

Key commands:

```bash
show interfaces GigabitEthernet1/0/1
show interfaces counters errors
show interfaces trunk
show vlan brief
show mac address-table dynamic
show spanning-tree vlan 10
show etherchannel summary
```

What to look for:

- **Interface state**
  - `up/up` usually means the physical link and line protocol are healthy.
  - `up/down` often points to encapsulation, keepalive, or Layer 2 mismatch.
  - `administratively down` means the interface is shut.
- **Errors**
  - CRC, input errors, runts, giants, and drops can indicate cabling, duplex, MTU, or congestion issues.
- **Trunking**
  - Verify allowed VLANs, native VLAN, and trunk status.
- **STP**
  - Confirm the port role and state match the design.
  - A port in `blocking`, `listening`, or `learning` may be expected or may indicate a topology issue.
- **EtherChannel**
  - Member links should be bundled consistently.
  - Mismatched speed, duplex, or channel settings can prevent bundling.

Example interpretation:

```bash
show spanning-tree vlan 10
```

Possible useful clues:

- `Root` and `Designated` roles show normal forwarding paths.
- `Alternate` or `Blocking` may be correct for redundancy.
- A port stuck in `Discarding` on Rapid PVST+ can indicate convergence or a topology mismatch.
- If the port is unexpectedly blocking, check for root bridge placement, VLAN pruning, or a loop.

**Layer 3 verification**

At Layer 3, you verify addressing, neighbor discovery, and route selection. A route can exist in the table but still fail if the next hop is unreachable or if the route is installed in the wrong VRF.

Core commands:

```bash
show ip interface brief
show ip route
show ip route 0.0.0.0
show ip arp
show cdp neighbors detail
show lldp neighbors detail
show ip cef exact-route 10.1.1.10 8.8.8.8
```

For IPv6, use:

```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
show ipv6 cef
show ipv6 protocols
```

Important IPv6 checks:

- `show ipv6 route` confirms whether the prefix is installed and which source learned it.
- `show ipv6 neighbors` verifies Neighbor Discovery Protocol (NDP) resolution.
- `show ipv6 interface brief` confirms global and link-local addressing.
- `show ipv6 protocols` helps confirm which routing processes are active and advertising.

When troubleshooting IPv6, remember that link-local addresses often appear in adjacency and next-hop information. If a route points to a link-local next hop, you must verify the outgoing interface as well.

**Route selection and interpretation**

You should be able to read the routing table quickly:

- **Code** tells you the source of the route.
- **Administrative distance** influences which source wins.
- **Metric** influences selection within the same protocol.
- **Next hop** tells you where traffic is forwarded.
- **Outgoing interface** tells you how the device will send the packet.

Example:

```bash
show ip route 10.10.20.0
```

You may see multiple candidates:

- Connected route
- Static route
- OSPF route
- EIGRP route
- BGP route

The installed route is the one with the best administrative distance and protocol-specific preference. If the route is missing, ask whether the protocol learned it, whether filtering blocked it, or whether recursion failed.

**VRF-specific verification**

VRFs (Virtual Routing and Forwarding) are a common source of confusion because a route can exist in one table and be absent in another. Always verify the correct VRF before assuming the route is missing.

Useful commands:

```bash
show vrf
show ip route vrf CUST1
show ip arp vrf CUST1
show ip cef vrf CUST1
show ip interface brief vrf CUST1
show ipv6 route vrf CUST1
show ipv6 neighbors vrf CUST1
```

What to check:

- The interface is assigned to the expected VRF.
- The route exists in the correct VRF table.
- The next hop is reachable inside that VRF.
- Any route leaking or import/export policy is working as intended.

A common failure pattern is:

- Interface is in `VRF-A`
- Static route is configured in the global table
- Traffic never leaves the VRF because the route is invisible there

That is why VRF-aware verification is essential in ENARSI scenarios.

**Dynamic routing protocol verification**

Dynamic routing problems are often about adjacency, timers, authentication, filtering, or topology mismatch. You should verify both neighbor formation and route exchange.

**OSPF verification**

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip ospf interface GigabitEthernet0/0
show ip route ospf
show ip protocols
```

Interpretation tips:

- Neighbor state should usually reach `FULL` on broadcast and point-to-point links.
- `INIT`, `2-WAY`, `EXSTART`, or `EXCHANGE` indicate adjacency problems.
- `EXSTART` often points to MTU mismatch, DBD negotiation issues, or duplicate router IDs.
- `show ip ospf interface` helps confirm area, timers, network type, and authentication.

Example configuration and verification:

```bash
router ospf 10
 router-id 1.1.1.1
 network 10.1.1.0 0.0.0.255 area 0
```

```bash
show ip ospf neighbor
show ip route ospf
```

If the neighbor is missing, verify:

- Area ID
- Hello and dead timers
- Authentication
- Network type
- Passive interface settings
- MTU consistency
- Interface IP reachability

**EIGRP verification**

```bash
show ip eigrp neighbors
show ip eigrp topology
show ip route eigrp
show ip protocols
```

Interpretation tips:

- A missing neighbor usually means hello mismatch, K-value mismatch, authentication mismatch, or interface filtering.
- The topology table shows feasible successors and reported distances.
- If a route is not installed, check feasibility conditions and metric consistency.

Example:

```bash
router eigrp 100
 network 10.1.0.0 0.0.255.255
 no auto-summary
```

Useful checks:

- Autonomous system number matches
- K-values match
- Stub settings are compatible
- Summarization is not hiding needed routes
- Passive interfaces are intentional

**BGP verification**

```bash
show ip bgp summary
show ip bgp
show ip bgp neighbors
show ip route bgp
show bgp ipv4 unicast summary
```

Interpretation tips:

- `Established` means the TCP session and BGP FSM are up.
- `Idle`, `Active`, or repeated resets indicate transport, authentication, ASN, or policy issues.
- A route can be received in BGP but not installed in the routing table if a better route exists or if next-hop reachability fails.

Example configuration:

```bash
router bgp 65001
 neighbor 192.0.2.2 remote-as 65002
 address-family ipv4 unicast
  network 10.10.10.0 mask 255.255.255.0
```

Useful checks:

- Neighbor state and uptime
- Prefix counts sent and received
- Route-maps and prefix-lists
- Next-hop reachability
- `maximum-prefix` limits
- Route reflector or confederation behavior if applicable

A common BGP troubleshooting pattern is:

- Session is up
- Routes are received
- Routes are not installed or not forwarded
- Cause is often next-hop recursion, policy filtering, or missing redistribution

**NAT verification**

NAT (Network Address Translation) issues often appear as “inside can reach some destinations but not others.” You should verify translation creation, ACL matching, and interface roles.

Commands:

```bash
show ip nat translations
show ip nat statistics
show access-lists
show running-config | section ip nat
```

Example configuration:

```bash
interface GigabitEthernet0/0
 ip nat inside

interface GigabitEthernet0/1
 ip nat outside

access-list 1 permit 10.10.10.0 0.0.0.255

ip nat inside source list 1 interface GigabitEthernet0/1 overload
```

What to verify:

- Inside and outside interfaces are correct
- ACL matches the source addresses you expect
- Overload or static mappings are present
- Translations appear when traffic is generated
- Return traffic is not blocked by policy or routing

If translations do not appear, check:

- ACL mismatch
- Wrong interface role
- Missing route to the outside
- Asymmetric routing
- NAT exemption rules

**IPsec and tunnel verification**

VPN and tunnel problems often require you to verify both the tunnel interface and the underlying security associations. A tunnel can be up while traffic still fails if routing, crypto policy, or selectors are wrong.

Commands:

```bash
show crypto isakmp sa
show crypto ikev2 sa
show crypto ipsec sa
show crypto session
show interface tunnel0
show ip route
```

Interpretation tips:

- IKE (Internet Key Exchange) or IKEv2 should complete before IPsec data flows.
- `QM_IDLE` or equivalent healthy states indicate successful negotiation in many classic deployments.
- IPsec packet counters should increase when traffic passes.
- A tunnel interface may be `up/up` but still carry no traffic if the route to the remote subnet is missing.

Example tunnel check:

```bash
show interface tunnel0
show crypto ipsec sa
```

Look for:

- Encapsulation and decapsulation counters increasing
- Correct peer address
- Correct transform set or proposal
- Matching traffic selectors
- No repeated rekey failures

**Automation and programmability verification**

ENCOR expects you to understand that verification is not only CLI-based. In modern networks, you also verify state through APIs, model-driven telemetry, and automation outputs.

The key idea is simple: compare the **intended state** from your source of truth with the **observed state** on the device.

Common verification methods:

- **REST API** queries to confirm configuration or operational state
- **NETCONF/RESTCONF** retrieval of structured data
- **gNMI** or telemetry subscriptions for streaming state
- **Ansible** or other automation output validation
- **Diff checks** between intended templates and running configuration

Example RESTCONF-style verification concept:

```bash
curl -k -u admin:password \
  -H "Accept: application/yang-data+json" \
  https://router.example.com/restconf/data/ietf-interfaces:interfaces
```

Example automation validation pattern:

```yaml
- name: Collect interface state
  cisco.ios.ios_facts:
    gather_subset:
      - interfaces

- name: Assert expected interface is up
  ansible.builtin.assert:
    that:
      - ansible_net_interfaces['GigabitEthernet1']['operstatus'] == 'up'
```

What to verify in automation workflows:

- The device accepted the intended configuration
- The resulting state matches the template or policy
- The API response reflects the same state as the CLI
- Telemetry shows the expected counters, adjacency, or route changes
- The automation job did not partially fail

A useful discipline is to validate both:

- **Configuration state**: what was pushed
- **Operational state**: what the device is actually doing

That distinction matters because a successful push does not guarantee a successful outcome.

**Interpreting common outputs**

You should practice reading outputs as evidence, not as decoration.

Examples:

- `show ip interface brief`
  - Fast check for address and line status
- `show ip route`
  - Confirms route source, next hop, and installation
- `show ip ospf neighbor`
  - Confirms adjacency state and neighbor ID
- `show spanning-tree`
  - Confirms forwarding/blocking roles and root placement
- `show crypto ipsec sa`
  - Confirms encrypted traffic counters
- `show ip nat translations`
  - Confirms active translations and direction

A few high-value interpretations:

- **Route present but ping fails**
  - Check ACLs, return path, NAT, and CEF recursion
- **Neighbor stuck in `EXSTART`**
  - Check MTU, duplicate router ID, or adjacency mismatch
- **STP port blocking unexpectedly**
  - Check root bridge, port priority, VLAN pruning, or loop protection
- **Tunnel up but no traffic**
  - Check routing, crypto selectors, and counters
- **BGP established but no prefixes installed**
  - Check policy, next-hop reachability, and route selection

**Structured escalation and isolation boundaries**

When a problem spans multiple devices or domains, isolate the fault boundary before escalating. This prevents “blame ping-pong” between teams.

A practical escalation model:

1. **Local device**
   - Interface, config, counters, logs
2. **Adjacent device**
   - Neighbor state, trunking, routing adjacency, tunnel peer
3. **Path**
   - Intermediate hops, policy devices, VRFs, NAT, security zones
4. **Service**
   - DNS, DHCP, AAA, application reachability
5. **Automation/control**
   - Templates, API state, orchestration, telemetry

Use this boundary model to answer:

- Is the fault on the source device?
- Is it on the neighbor?
- Is it in transit?
- Is it in a shared service?
- Is it only in the automation layer?

This is especially important in enterprise environments where a single symptom may cross switching, routing, security, and orchestration teams.

**A repeatable troubleshooting procedure**

1. **State the symptom precisely**
   - Include source, destination, protocol, and time.
2. **Check the most local evidence**
   - Interface status, counters, logs, and config.
3. **Verify protocol state**
   - Neighbor, route, tunnel, or translation status.
4. **Compare intended and actual state**
   - Use CLI, API, or telemetry.
5. **Test one hypothesis**
   - Change the smallest safe variable.
6. **Re-verify**
   - Confirm the fix and watch for side effects.
7. **Document the root cause**
   - Record what failed and why the fix worked.

**Practical verification examples**

**Example 1: OSPF adjacency failure**

```bash
show ip ospf neighbor
show ip ospf interface g0/0
show ip interface brief
show logging
```

If the neighbor is stuck in `EXSTART`, verify:

- MTU matches on both sides
- Area and timers match
- Authentication matches
- Router IDs are unique
- The link is not filtered by ACLs

**Example 2: IPv6 host cannot reach remote subnet**

```bash
show ipv6 interface brief
show ipv6 neighbors
show ipv6 route
show ipv6 protocols
ping ipv6 2001:db8:2::1
```

Check:

- Global and link-local addresses
- Neighbor discovery entries
- Route presence in the correct VRF
- Default route or specific prefix
- Any RA, DHCPv6, or routing process issues

**Example 3: NAT translation not appearing**

```bash
show ip nat statistics
show ip nat translations
show access-lists
show ip route 0.0.0.0
```

Check:

- ACL matches the inside source
- Inside/outside roles are correct
- Outside route exists
- Return traffic is not asymmetric

**Example 4: BGP routes learned but not used**

```bash
show ip bgp summary
show ip bgp 203.0.113.0
show ip route 203.0.113.0
show ip cef 203.0.113.0
```

Check:

- Next hop is reachable
- Policy does not filter the route
- Another route has better administrative distance
- The prefix is not suppressed or dampened

**Example 5: IPsec tunnel established but traffic fails**

```bash
show crypto ikev2 sa
show crypto ipsec sa
show interface tunnel0
show ip route
```

Check:

- Security associations are active
- Encapsulation and decapsulation counters increase
- Tunnel source and destination are correct
- Routes point into the tunnel as intended

**Verification checklist for exam scenarios**

- Confirm the symptom with a test, not an assumption.
- Verify the correct VRF before checking routes.
- Check IPv6 with IPv6-specific commands, not only IPv4 equivalents.
- Read protocol state, not just configuration.
- Validate adjacency, route installation, and forwarding separately.
- Use counters to prove traffic flow.
- Compare automation output to intended state.
- Escalate only after you isolate the fault boundary.

**Key takeaway**

Verification discipline is the difference between random changes and controlled troubleshooting. In ENCOR and ENARSI scenarios, you should prove the failure, isolate the layer, confirm the protocol state, and validate the fix with evidence. The best answer is usually the one that explains not only what is broken, but where it is broken and how you know.
