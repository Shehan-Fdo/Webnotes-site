## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Configuration backup, rollback, and documentation workflow**

A reliable change workflow reduces outage risk and makes recovery predictable. In Cisco enterprise environments, you usually combine configuration backups, versioned archives, validation checks, and clear documentation so you can restore service quickly if a change fails.

The core idea is simple:

- Save a known-good configuration before change.
- Make the change in a controlled window.
- Verify behavior immediately after the change.
- Roll back quickly if the result is wrong.
- Record what changed so the next engineer can repeat or reverse it safely.

**Why this matters in enterprise operations**

Backups are not only for disaster recovery. They also support:

- **Change control** by preserving a pre-change baseline.
- **Troubleshooting** by comparing current and previous states.
- **Compliance** by proving what was changed and when.
- **Recovery** after bad ACLs, routing changes, AAA mistakes, or interface edits.

A good workflow is both technical and procedural. The technical part is the actual copy, archive, and restore commands. The procedural part is naming, storage, access control, and documentation discipline.

**Typical change lifecycle**

```text
Plan -> Backup -> Implement -> Validate -> Roll back if needed -> Document -> Archive
```

**Backup strategy fundamentals**

A backup strategy should answer four questions:

- What do you back up?
- Where do you store it?
- How do you restore it?
- How do you prove it is usable?

On Cisco routers and switches, the most important configuration artifacts are:

- `running-config` for the active configuration in RAM.
- `startup-config` for the saved boot configuration in NVRAM.
- Archive copies stored on local flash, remote servers, or both.

**Running configuration vs startup configuration**

- **`running-config`** changes immediately when you enter commands.
- **`startup-config`** is what the device loads at boot.
- If you do not save changes, a reload can discard them.
- If you restore the wrong file, you can overwrite a working state with stale settings.

A common operational habit is to save after every approved change window:

```bash
copy running-config startup-config
```

On many platforms, you can also use:

```bash
write memory
```

or the abbreviated form:

```bash
wr mem
```

Use the explicit `copy` form when you want clarity in scripts and runbooks.

**Backup destinations and transport choices**

Choose a destination based on security, scale, and recovery speed.

| Destination | Strengths | Tradeoffs |
|---|---|---|
| Local flash | Fast, simple, useful for immediate rollback | Limited space, device loss risk |
| USB storage | Portable, useful in recovery scenarios | Physical handling risk, easy to misplace |
| TFTP server | Simple and common in labs | No encryption, weak security |
| FTP server | Widely supported | Credentials and data are not encrypted |
| SCP server | Encrypted transport over SSH | Requires SSH and server support |
| SFTP server | Encrypted transport over SSH | Requires server support and proper permissions |

For production, prefer encrypted transport such as **SCP** (Secure Copy Protocol) or **SFTP** (SSH File Transfer Protocol) when the platform supports it.

> Avoid treating TFTP as a secure backup method. It is useful in labs and isolated recovery networks, but it does not protect credentials or configuration contents in transit.

**Practical backup commands**

A local backup to flash can be useful before a risky change:

```bash
copy running-config flash:prechange-rtr1.cfg
```

A remote backup to a TFTP server typically requires the destination host and filename:

```bash
copy running-config tftp:
```

The device then prompts you for the remote host and filename on many IOS/IOS XE releases. In some environments, you may specify the full destination interactively or with a URL-style path, depending on platform support.

For encrypted transfer, use SCP or SFTP in the platform-supported form. The exact syntax can vary by IOS/IOS XE release and device family, so verify the command on the target platform:

```bash
copy running-config scp:
copy running-config sftp:
```

These forms are often interactive and may prompt for the remote host, username, and destination path. Do not assume a single universal one-line syntax across all platforms.

**Restoring configurations**

Restoration can mean different things:

- Merge a saved config into the current running configuration.
- Replace the current configuration with a known-good file.
- Restore startup configuration after a reload.
- Recover from a failed change using an archive or backup file.

The safest method depends on the situation.

**Merge versus replace**

- **Merge** adds or changes lines from the source file into the current running configuration.
- **Replace** attempts to make the running configuration match the source more closely.
- Merge is useful for small, controlled additions.
- Replace is better when you need to revert many changes at once.

A simple merge from a file into the running configuration may look like this:

```bash
copy flash:prechange-rtr1.cfg running-config
```

This can be useful, but be careful: the result depends on the file contents and platform behavior. It is not always a clean “undo.” Conflicting lines may remain unless you explicitly remove them or use a replace-style workflow.

**Restoring startup configuration**

If the device boots with a bad or empty startup configuration, you can restore from a backup file after access is regained:

```bash
copy tftp: startup-config
```

On many devices, this is interactive and requires the source filename and server details. After restoring `startup-config`, you usually reload the device to boot from the recovered file:

```bash
reload
```

Use this method when the saved startup configuration is the desired recovery target and you want the device to come up with that baseline on the next boot.

**Archive-based rollback on IOS and IOS XE**

Cisco archive features help you keep versioned configuration copies and roll back changes. This is especially useful when you want a local history of changes on the device itself.

A common archive configuration includes:

```bash
configure terminal
 archive
  path flash:archive/
  maximum 10
  write-memory
end
```

What this does:

- Stores archived copies in `flash:archive/`.
- Keeps up to 10 versions.
- Saves an archive copy when you write memory.

You can also create an archive copy manually before a change:

```bash
archive config
```

or, on some platforms, use the archive command sequence supported by that release.

**Rollback with archive files**

Rollback syntax is platform and release dependent. On many IOS/IOS XE devices, you use archive-related commands or `configure replace` rather than a universal checkpoint command.

A common replace workflow looks like this:

```bash
configure replace flash:prechange-rtr1.cfg force
```

This tells the device to replace the current running configuration with the contents of the specified file. The `force` keyword suppresses some prompts and can be useful in controlled maintenance windows.

Important cautions:

- Support varies by platform and software release.
- Some devices support `configure replace` differently or with limitations.
- Always test the exact command on the target platform before relying on it in production.

**Checkpoint and rollback concepts**

Some Cisco platforms and software trains support checkpoint-style workflows, but the exact syntax is not universal across IOS XE devices. For exam prep, focus on the concept rather than memorizing a single command that may not apply everywhere.

Use checkpoints when you need:

- A named pre-change state.
- A quick way to compare current and saved configurations.
- A controlled rollback path during maintenance.

If your platform supports archive or replace workflows, those are often the more broadly applicable IOS/IOS XE methods to know.

**Validation after change**

A backup is only useful if you can confirm it is valid and current. After every change, verify both the configuration and the operational state.

Common checks include:

```bash
show running-config
show startup-config
show archive
show version
show ip interface brief
show interfaces status
show ip route
```

Use the right verification command for the feature you changed:

- For interface work, check line protocol, errors, and VLAN membership.
- For routing changes, confirm neighbors, routes, and reachability.
- For ACL changes, test the intended permit/deny behavior.
- For AAA changes, verify you still have administrative access before ending the session.

**Side-by-side behavior summary**

| Topic | IOS / IOS XE practical guidance |
|---|---|
| Save active config | `copy running-config startup-config` or `write memory` |
| Local backup | `copy running-config flash:filename` |
| Remote backup | `copy running-config tftp:` or encrypted SCP/SFTP forms |
| Restore to running | `copy file running-config` may merge, depending on file and platform |
| Replace running config | `configure replace <file> force` on supported platforms |
| Verify boot config | `show startup-config`, `show version`, and platform-specific boot checks |
| Archive history | `archive` with `path` and `maximum` settings |

**Documentation workflow**

Documentation is part of the recovery plan. If you cannot explain what changed, you cannot reliably reverse it later.

Your change record should include:

- Device name and management IP.
- Date and time of change.
- Engineer name or ticket reference.
- Baseline configuration filename.
- Exact commands entered.
- Validation results.
- Rollback steps, if used.
- Final approved state.

A concise change log entry might look like this:

```text
Device: DIST-SW1
Change window: 2026-04-26 22:00 UTC
Ticket: CHG-18422
Backup: flash:prechange-dist-sw1.cfg
Change: Added VLAN 240 and trunk allowed list update
Validation: Ping to gateway successful, trunk up, STP stable
Rollback: Not required
```

**Recommended file naming**

Use names that sort well and identify the device and time.

Examples:

```text
rtr1-prechange-20260426-2200.cfg
dist-sw1-postchange-20260426-2230.cfg
core1-archive-20260426.cfg
```

Good names help you:

- Find the right file quickly.
- Avoid overwriting older backups.
- Match a backup to a change ticket.

**Storage and retention considerations**

Configuration archives consume storage over time. Plan for cleanup and retention.

Consider:

- How many versions you need locally.
- Whether flash space is limited.
- Whether old backups should be copied off-device.
- Whether backups should be retained for audit requirements.

Useful operational checks include:

```bash
dir flash:
show file systems
```

If archive files accumulate, remove only the files you no longer need and confirm you are not deleting the last known-good copy.

**Security for backup files**

Backup files often contain sensitive information such as:

- Password hashes.
- SNMP community strings or credentials.
- Shared secrets.
- ACLs and management addresses.
- VPN and routing details.

Protect them with the same care as the live device.

Best practices:

- Restrict access to backup repositories.
- Use encrypted transport such as SCP or SFTP.
- Store backups on secured servers with role-based access control.
- Avoid leaving sensitive files on shared jump hosts.
- Protect backup directories from unauthorized modification.
- Consider encryption at rest on backup servers or storage systems.

> A backup that is easy to read by everyone is also easy to misuse by everyone.

**Recovery scenarios**

Different failures call for different restore methods.

- If you made a bad live change and still have access, restore the saved file to `running-config` or use `configure replace` if supported.
- If the device reloads with a bad startup configuration, restore the saved file to `startup-config` and reload.
- If the device is partially reachable, use console access and a local flash backup.
- If local storage is damaged, recover from an external backup server.

A practical recovery sequence often looks like this:

1. Stop making changes.
2. Confirm the failure scope.
3. Retrieve the last known-good backup.
4. Restore using the least disruptive supported method.
5. Verify services and access.
6. Document the incident and the final state.

**Example workflow for a controlled change**

```text
1. Back up the current configuration.
2. Apply the change.
3. Verify interfaces, routing, and access.
4. If validation fails, restore the backup.
5. Save the recovered state.
6. Update the change record.
```

**Example: interface and VLAN change with rollback plan**

```bash
copy running-config flash:dist-sw1-prechange.cfg
configure terminal
 interface GigabitEthernet1/0/10
  switchport mode access
  switchport access vlan 240
end
show interfaces status
show vlan brief
copy running-config startup-config
```

If validation fails, you can restore the saved file:

```bash
copy flash:dist-sw1-prechange.cfg running-config
```

Then verify the result again:

```bash
show running-config interface GigabitEthernet1/0/10
show vlan brief
```

**Example: archive-based operational pattern**

```bash
configure terminal
 archive
  path flash:archive/
  maximum 5
  write-memory
end
copy running-config startup-config
show archive
```

This pattern gives you a local history of saved configurations. It is useful when you need to compare versions or recover from a recent mistake without relying only on an external server.

**Common mistakes to avoid**

- Saving a backup after the change instead of before it.
- Keeping only one copy of the configuration.
- Using insecure transport for sensitive backups.
- Assuming a restore command behaves the same on every platform.
- Forgetting to verify the restored state.
- Overwriting a good backup with an untested one.
- Neglecting flash space and archive retention.
- Failing to document the exact commands used.

**Exam-focused takeaways**

- **Backup before change** is the safest default.
- **`running-config`** is the live configuration; **`startup-config`** is the boot configuration.
- **Archive and replace** workflows are common on IOS and IOS XE, but exact syntax and support vary.
- **SCP/SFTP** are preferred over unencrypted file transfer when available.
- **Verification** is part of the workflow, not an optional extra.
- **Documentation** makes rollback repeatable and auditable.

**Quick reference checklist**

- Confirm the maintenance window and ticket.
- Save a pre-change backup.
- Use a clear filename and storage location.
- Apply the change.
- Validate the service and control plane.
- Roll back if needed using the supported method.
- Save the final approved state.
- Record what changed and where the backup is stored.
