## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Overview**

Cisco IOS XE image selection is not just about finding a file that boots. You choose an image based on the platform family, intended device role, feature set, boot model, licensing, and upgrade strategy. In enterprise networks, the wrong image can create avoidable downtime, missing features, or unsupported behavior.

Device role planning means matching the router or switch to the job it must perform. A branch edge router, campus access switch, distribution switch, WAN aggregation node, or wireless controller host all have different requirements for throughput, redundancy, crypto, telemetry, and management.

**IOS XE image naming and what it tells you**

IOS XE filenames usually encode useful information about the platform, release, and packaging. You should read the filename carefully, but never assume the name alone guarantees every feature.

Common elements include:

- **Platform family**: often embedded in the prefix, such as `isr4k`, `cat9k`, or `c8000`.
- **Release train**: for example `17.09.04` indicates the software release.
- **Image type**: `universalk9` is a common Cisco naming convention for a universal IOS XE image with K9 crypto support.
- **Packaging**: `.bin` is commonly used for the installable image file on many IOS XE platforms.
- **Special tags**: `SPA` is a Cisco packaging marker often seen in IOS XE filenames.

A filename such as `cat9k_iosxe.17.09.04.SPA.bin` suggests a Catalyst 9000 IOS XE image for release 17.9.4, but you still need to verify platform support and feature availability in the release notes.

> Do not treat `universalk9` as a promise that every feature is included. Feature availability still depends on the platform, software release, licensing, and whether the platform runs in install mode or another supported boot model.

**Platform and role planning**

Before you choose an image, define the device role. The role determines the minimum feature set and operational constraints.

- **Branch edge router**
  - Needs WAN interfaces, routing protocols, NAT, VPN, and often QoS.
  - May require crypto acceleration or hardware offload.
  - Often prioritizes stability and remote recoverability.

- **Campus access switch**
  - Needs VLANs, STP, EtherChannel, 802.1X, PoE, and access control features.
  - Often requires stack or chassis support, depending on platform.
  - May need telemetry, segmentation, and automation hooks.

- **Distribution or core switch**
  - Needs high forwarding capacity, redundancy, routing scale, and fast convergence.
  - Often requires advanced routing, multicast, and resilient upgrade behavior.

- **WAN aggregation or edge**
  - Needs route scale, policy control, tunnel support, and service insertion features.
  - May require high availability and careful image compatibility with line cards or modules.

- **Management or services node**
  - May host controller functions, telemetry collectors, or specialized services.
  - Needs compatibility with the service role, not just the base platform.

A good planning workflow is to start with the role, then narrow to platform, then release, then boot mode, then licensing.

**Install mode, bundle mode, and boot behavior**

IOS XE boot behavior varies by platform. On many modern Catalyst platforms, **install mode** is the preferred operational model. In install mode, the system extracts and activates packages from the image during installation, and the device boots from the installed package set rather than directly from a single monolithic file.

Bundle mode exists on some platforms and releases, but you should not assume it is the normal or preferred method everywhere. Some platforms support it for specific recovery or compatibility scenarios, while others do not recommend it for day-to-day operation.

Key points:

- **Install mode**
  - Common on Catalyst 9000 platforms.
  - Uses the `install` workflow to add, activate, and commit software.
  - Better suited to managed upgrades and rollback control.

- **Bundle mode**
  - Boots from a single image file on platforms that support it.
  - May be used in limited scenarios, but support and recommendation vary by platform.
  - Can simplify some recovery situations, but it is not a universal fallback.

- **Boot variables**
  - Boot variables and boot commands differ by platform and mode.
  - Do not assume a single `boot system flash:...` line applies everywhere.

A typical install-mode upgrade on a Catalyst 9000 platform looks like this:

```bash
show version
show install summary
dir flash:
install add file flash:cat9k_iosxe.17.09.04.SPA.bin activate commit
show install summary
show version
```

This workflow is preferred because it validates the package, stages it, activates it, and commits it in a controlled sequence.

**How to verify the right image before upgrade**

You should verify compatibility before you copy or activate an image. The exact commands vary by platform and release, so use platform documentation and release notes as the final authority.

Useful checks include:

- `show version`
  - Confirms platform model, current release, uptime, and boot mode.
- `show inventory`
  - Identifies hardware modules and serial numbers.
- `show install summary`
  - Shows install state on platforms that support install mode.
- `show boot`
  - Displays boot variables on platforms that expose them.
- `dir flash:`
  - Confirms available storage and image presence.
- `show platform`
  - Helps identify platform-specific state and hardware details.

Example:

```bash
show version
show inventory
show install summary
show boot
dir flash:
```

> Commands such as `verify /sha256` or `show rom-monitor` are not universal across all IOS XE platforms and releases. Use them only when you know the platform supports them.

**Image selection criteria**

Choose the image that matches the operational need, not just the newest release.

- **Platform support**
  - Confirm the image is built for the exact hardware family.
  - Check whether the platform uses a unified image, install packages, or a platform-specific package set.

- **Feature requirements**
  - Confirm support for routing, switching, security, telemetry, or wireless functions you need.
  - Check whether the feature is native to the platform or requires a license or add-on package.

- **Release maturity**
  - Prefer a release with known stability for your platform and feature set.
  - Avoid jumping to a release only because it is newer.

- **Memory and storage**
  - Ensure flash, DRAM, and boot storage meet the image requirements.
  - Leave room for rollback images, logs, and package extraction.

- **Upgrade path**
  - Confirm whether you can upgrade directly or need an intermediate release.
  - Review any ROMMON, bootloader, or prerequisite package requirements.

- **Operational constraints**
  - Consider maintenance windows, remote access, and rollback options.
  - Plan for console access if the upgrade fails.

**Licensing and feature availability**

Licensing on IOS XE is platform-specific and release-specific. Do not oversimplify it as a generic unlock mechanism.

On Catalyst platforms, licensing may involve:

- **Network licenses**
  - Often tied to base switching or routing capabilities.
  - May affect access to certain forwarding or protocol features.

- **DNA subscriptions**
  - Such as DNA Essentials or DNA Advantage on supported Catalyst platforms.
  - These are management and automation subscriptions, not a universal feature switch.
  - They may influence access to certain assurance, automation, or policy capabilities depending on platform and release.

- **Evaluation or smart licensing state**
  - Some platforms operate with smart licensing registration and entitlement reporting.
  - License state can affect warnings, grace periods, or feature enforcement.

Practical planning rule:

- Verify the platform’s licensing model before deployment.
- Confirm whether the feature you need is controlled by hardware capability, network license, DNA subscription, or software release.
- Do not assume a subscription alone enables every advanced function.

**Planning for rollback and recovery**

Rollback planning is part of image selection. You should always know how to return the device to service if the new image fails.

Good rollback planning includes:

- Keeping a known-good image available.
- Confirming console or out-of-band access.
- Saving the current configuration before upgrade.
- Checking storage space for both current and fallback software.
- Understanding whether rollback uses install rollback, boot variable changes, or a platform-specific recovery method.

Example planning checklist:

1. Save the running configuration.
2. Record the current release and boot state.
3. Confirm the fallback image is present and valid.
4. Verify console access.
5. Perform the upgrade during a maintenance window.
6. Validate forwarding, management, and routing after reboot.

Example commands:

```bash
copy running-config startup-config
show version
show install summary
dir flash:
```

> Copying an old `.bin` file to USB does not by itself create a valid rollback plan. Rollback depends on the platform boot model and whether the device uses install mode, bundle mode, or another recovery workflow.

**Practical examples by device role**

A branch router and a campus switch may both run IOS XE, but they need different planning.

- **Branch router example**
  - Prioritize WAN interfaces, VPN, NAT, and routing scale.
  - Confirm crypto support and throughput on the exact hardware.
  - Choose a release that supports the required tunnel and routing features.

- **Campus access switch example**
  - Prioritize access control, PoE, stacking, and endpoint onboarding.
  - Confirm support for 802.1X, DHCP snooping, Dynamic ARP Inspection, and segmentation features.
  - Ensure the image supports the intended operational mode and management tooling.

- **Distribution switch example**
  - Prioritize routing scale, redundancy, and convergence.
  - Confirm support for routing protocols, first-hop redundancy, and high availability features.
  - Validate upgrade behavior carefully because downtime affects many downstream devices.

**Common mistakes to avoid**

- Choosing an image by filename alone without checking release notes.
- Assuming `universalk9` means every feature is included.
- Treating install mode and bundle mode as interchangeable on all platforms.
- Using a boot command from one platform as a universal example.
- Ignoring licensing state when planning feature availability.
- Forgetting to verify storage, memory, and rollback options before upgrade.
- Assuming a command exists on every IOS XE platform.

**Operational workflow for image planning**

Use a repeatable process when selecting IOS XE software.

1. Identify the exact platform model.
2. Define the device role and required features.
3. Check the supported release train for that platform.
4. Confirm boot mode and software packaging model.
5. Review licensing requirements.
6. Verify storage, memory, and upgrade prerequisites.
7. Stage the image and test in a maintenance window.
8. Validate post-upgrade behavior with `show version`, `show install summary`, and role-specific checks.

**Example validation commands after upgrade**

```bash
show version
show ip interface brief
show ip route
show spanning-tree summary
show install summary
```

Use role-specific checks after the reboot. For a router, validate routing neighbors and tunnel status. For a switch, validate VLANs, trunks, STP, and endpoint connectivity. For a wireless or services role, validate the service-specific control plane and management reachability.

**Key takeaways**

- Image selection starts with the device role, not the filename.
- IOS XE packaging and boot behavior vary by platform.
- Install mode is common and preferred on many Catalyst platforms, but you must confirm the supported workflow.
- Licensing affects feature availability in platform-specific ways.
- Rollback planning is part of the upgrade design, not an afterthought.
- Always validate the exact platform, release, and supported commands before making changes.
