## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**OSPFv3 and BGP IPv6 overview**

OSPFv3 and BGP are the two most common control-plane choices for IPv6 routing in Cisco enterprise networks. OSPFv3 is an IGP (Interior Gateway Protocol) used for fast convergence inside an autonomous system, while BGP is typically used for policy control, edge routing, and route exchange between domains.

For exam prep, focus on three ideas:

- **OSPFv3** runs per link and uses IPv6 link-local addresses for neighbor relationships.
- **BGP IPv6** uses the MP-BGP (Multiprotocol BGP) address-family model on modern Cisco IOS XE.
- Route installation depends on best-path selection, administrative distance, and whether the route is actually eligible for the RIB (Routing Information Base).

**OSPFv3 fundamentals**

OSPFv3 is the IPv6 version of OSPF. It keeps the same core link-state design as OSPFv2, but the protocol is adapted for IPv6 addressing and modern multi-address-family operation.

Key differences from OSPFv2:

- **Neighboring** uses IPv6 link-local addresses, not global unicast addresses.
- **Interface activation** is often done under the interface or under an address-family, depending on IOS XE syntax.
- **Authentication** is not the same as OSPFv2 simple or MD5 authentication fields; modern deployments may use IPv6 authentication trailer or IPsec-related protection depending on platform and release.
- **LSAs** are still the core database objects, but OSPFv3 separates topology information from IPv6 prefix information more cleanly than OSPFv2.

OSPFv3 packet types are the same classic OSPF packet families:

- Hello
- Database Description (DBD)
- Link State Request (LSR)
- Link State Update (LSU)
- Link State Acknowledgment (LSAck)

OSPFv3 packets include an OSPF header with fields such as the **Instance ID**, which helps distinguish multiple OSPFv3 instances on the same link. The protocol is also more explicitly per-link than per-interface in how it treats adjacency formation and flooding scope.

**Neighbor formation and packet behavior**

OSPFv3 neighbors form using link-local addresses and multicast hellos on the local link. The common multicast addresses are:

- `ff02::5` for all OSPF routers
- `ff02::6` for designated routers and backup designated routers

That said, not all OSPFv3 traffic is multicast. After adjacency formation, some exchanges can be unicast to a specific neighbor, especially during database synchronization and retransmission. The important troubleshooting point is that OSPFv3 depends on link-local reachability and correct interface-level behavior, not just global IPv6 routing.

A typical adjacency sequence is:

1. Hellos are exchanged on the link.
2. Neighbor parameters are matched.
3. The DR/BDR election occurs on broadcast and NBMA-like multiaccess networks.
4. DBD packets synchronize the database.
5. LSR and LSU packets request and flood missing LSAs.
6. LSAck confirms receipt.

Common neighbor requirements:

- Matching area ID
- Matching hello and dead timers
- Matching authentication/security settings, if used
- Matching network type expectations
- No duplicate router IDs
- Reachable link-local adjacency on the same segment

> OSPFv3 adjacency problems often look like “hello received, but no full adjacency.” That usually points to a mismatch in timers, area, MTU, network type, or security settings rather than a pure IPv6 routing issue.

**DR/BDR election and multiaccess behavior**

On broadcast and some multiaccess segments, OSPFv3 elects a Designated Router (DR) and Backup Designated Router (BDR) just like OSPFv2. The election is based on:

- Interface priority
- Router ID as the tiebreaker

Practical implications:

- DR/BDR reduce flooding overhead on shared segments.
- A router with priority `0` does not participate in the election.
- If the DR changes, adjacency state can briefly churn while LSAs reflood.

On point-to-point links, there is no DR/BDR election. This is a common exam distinction.

**OSPFv3 areas and LSA behavior**

OSPFv3 still uses areas to scale the topology database. Area design affects which LSAs are allowed and how external routes are handled.

Common area types:

- **Backbone area 0**: required transit area for inter-area routing in standard designs.
- **Normal area**: accepts intra-area, inter-area, and external LSAs.
- **Stub area**: blocks external LSAs and uses a default route from the ABR (Area Border Router).
- **Totally stubby area**: Cisco-specific behavior that suppresses more inter-area detail and relies heavily on a default route.
- **NSSA** (Not-So-Stubby Area): allows limited external route injection via NSSA externals, then translates them at the ABR.

Behavior to remember:

- Stub and totally stubby areas reduce LSDB size.
- NSSA is useful when a branch must inject local external routes but should not receive full external tables.
- External LSAs are filtered differently depending on area type and ABR role.

In OSPFv3, the exact LSA types and how prefixes are carried can differ from OSPFv2 in implementation details, but the exam-relevant point is that OSPFv3 still performs link-state flooding, summarization at ABRs/ASBRs, and area-based filtering.

**OSPFv3 configuration patterns on Cisco IOS XE**

Modern IOS XE commonly supports OSPFv3 in an address-family style. Exact syntax varies by release, so verify the platform before lab work.

Example interface-based activation:

```cisco
ipv6 unicast-routing

ipv6 router ospf 10
 router-id 10.10.10.10

interface GigabitEthernet0/0
 ipv6 address 2001:db8:1::1/64
 ipv6 ospf 10 area 0
```

Example address-family style:

```cisco
router ospfv3 10
 router-id 10.10.10.10

address-family ipv6 unicast
 exit-address-family

interface GigabitEthernet0/0
 ospfv3 10 ipv6 area 0
```

Use the syntax that matches your IOS XE release and lab platform. The exam may test the concept more than the exact command spelling, but you should still recognize both styles.

Useful verification commands:

```cisco
show ipv6 ospf neighbor
show ipv6 ospf interface
show ipv6 ospf database
show ipv6 route ospf
show ipv6 protocols
```

Platform-dependent commands may exist for brief summaries, but `show ipv6 ospf interface` and `show ipv6 ospf neighbor` are the safest baseline commands to know.

**OSPFv3 security and authentication**

OSPFv3 does not use the OSPFv2 authentication field model. On modern Cisco platforms, OSPFv3 security may be implemented with IPv6 authentication trailer support or IPsec-related mechanisms, depending on software release and feature set.

Exam-relevant takeaways:

- Do not assume OSPFv3 is “unauthenticated by design.”
- Do not troubleshoot OSPFv3 adjacency as if OSPFv2 MD5 were the default mechanism.
- If neighbors fail to form and all basic parameters match, security configuration is a valid suspect.

When troubleshooting, check:

- Whether the platform supports the configured security method
- Whether both sides use the same protection model
- Whether the interface is actually applying the intended OSPFv3 instance

**BGP IPv6 behavior with MP-BGP**

IPv6 BGP on Cisco IOS XE is typically implemented with MP-BGP. That means you configure a global BGP process and then activate the IPv6 address-family.

Core concepts:

- The BGP process is created globally with an AS number.
- Neighbors are defined globally.
- IPv6 route exchange happens under `address-family ipv6 unicast`.
- Routes are advertised only if they are eligible for BGP origination and the address-family is activated.

Example baseline configuration:

```cisco
router bgp 65000
 bgp router-id 10.10.10.10
 neighbor 2001:db8:12::2 remote-as 65001
 neighbor 2001:db8:12::2 update-source GigabitEthernet0/0

 address-family ipv6 unicast
  neighbor 2001:db8:12::2 activate
  network 2001:db8:100::/64
 exit-address-family
```

Important behavior:

- The `network` statement originates a prefix only if an exact matching route exists in the local RIB.
- Redistribution is another way to originate routes, but it is not the only way.
- BGP does not automatically advertise connected IPv6 routes unless they are matched by a `network` statement or redistributed into BGP.

**BGP IPv6 neighbor formation requirements**

IPv6 BGP sessions often fail for predictable reasons. The most common are:

- Wrong remote AS
- Missing `neighbor ... activate` under the IPv6 address-family
- Missing `update-source` when peering to a loopback
- No route to the neighbor’s source address
- TTL/hop-count issues for multihop peering
- ACLs or control-plane policing blocking TCP port 179

For directly connected eBGP (external BGP) peers, the default hop limit is usually sufficient. For loopback-to-loopback peering or non-directly connected peers, you typically need multihop behavior.

Example loopback peering pattern:

```cisco
router bgp 65000
 neighbor 2001:db8:ff::2 remote-as 65001
 neighbor 2001:db8:ff::2 update-source Loopback0
 neighbor 2001:db8:ff::2 ebgp-multihop 2

 address-family ipv6 unicast
  neighbor 2001:db8:ff::2 activate
```

Practical notes:

- For iBGP (internal BGP), loopback peering is common and stable.
- For eBGP over loopbacks, you need both `update-source` and multihop.
- The neighbor must be reachable in the routing table before the session can establish.

**BGP IPv6 next-hop and route installation**

BGP stores paths in the BGP table first. A route is installed into the IPv6 RIB only if it wins best-path selection and the next-hop is resolvable.

Key points:

- A route can appear in `show bgp ipv6 unicast` but not in `show ipv6 route` if it is not the best path.
- A route can also fail RIB installation if its next-hop is unreachable.
- Administrative distance still matters when multiple protocols offer the same prefix.

Typical administrative distance values:

- Connected: 0
- Static: 1
- eBGP: 20
- EIGRP internal: 90
- OSPF: 110
- iBGP: 200

This means:

- eBGP usually beats OSPFv3 for the same prefix.
- Static routes usually beat both, unless you intentionally change distance.
- iBGP often loses to IGP-learned routes unless policy changes the outcome.

A common troubleshooting pattern is:

- The prefix is present in BGP.
- The next-hop is not reachable in the IPv6 routing table.
- The route is therefore not installed in the RIB.

Verification commands:

```cisco
show bgp ipv6 unicast
show bgp ipv6 unicast summary
show bgp ipv6 unicast neighbors
show bgp ipv6 unicast neighbors 2001:db8:12::2 advertised-routes
show ipv6 route
show ipv6 route 2001:db8:100::/64
```

Some platforms support `received-routes` only if route refresh or soft-reconfiguration is available. Do not rely on a single neighbor route-display command across all IOS XE releases.

**Redistribution between OSPFv3 and BGP**

Redistribution is powerful but risky. It moves routes between protocol domains, which can create loops if you do not tag and filter carefully.

Design goals:

- Inject only the routes you intend to share.
- Preserve route origin information with tags.
- Prevent routes from being reintroduced back into the source domain.
- Keep metrics and policy consistent.

Common redistribution concerns:

- **Feedback loops**: a route learned from OSPFv3 is redistributed into BGP, then redistributed back into OSPFv3.
- **Metric mismatch**: external route cost may not reflect the intended preference.
- **Route explosion**: too many prefixes are leaked between domains.
- **Asymmetric policy**: one direction is filtered differently from the return path.

A safer design uses route tags and route-maps.

Example concept:

```cisco
route-map OSPF-TO-BGP permit 10
 match ipv6 address prefix-list OSPF-EXPORT
 set tag 100

ipv6 prefix-list OSPF-EXPORT seq 10 permit 2001:db8:10::/48 le 64
```

Then apply redistribution with policy where supported by your IOS XE release.

Example policy logic:

- Tag routes when they leave OSPFv3.
- Deny tagged routes when they come back from BGP.
- Permit only the intended summary or edge prefixes.

> Redistribution syntax is platform and release dependent on IOS XE. In exam labs, focus on the policy model, route tagging, and the direction of redistribution rather than memorizing one fragile command form.

**Practical route-flow example**

Consider this topology:

- R1 runs OSPFv3 inside the campus.
- R2 is the edge router.
- R2 peers with an ISP over eBGP IPv6.
- R1 advertises internal IPv6 prefixes to R2.
- R2 advertises a default route or selected external prefixes back into OSPFv3.

Route flow:

1. R1 learns `2001:db8:10::/64` via OSPFv3.
2. R2 receives it and may redistribute or originate it into BGP policy.
3. The ISP learns the prefix via BGP.
4. R2 learns a default route or external prefix from the ISP.
5. R2 injects the default route into OSPFv3 for internal reachability.

A simple conceptual diagram:

```text
[LAN] -- OSPFv3 -- [R1] -- OSPFv3 -- [R2] -- eBGP IPv6 -- [ISP]
                     ^                         |
                     |---- default route ------|
```

This design works well when:

- Internal prefixes stay internal unless explicitly exported.
- External routes are summarized or filtered before entering the IGP.
- The default route is preferred for outbound Internet access.

**Troubleshooting workflow**

Use a layered approach instead of jumping straight to redistribution.

1. Verify IPv6 is enabled and interfaces have addresses.
2. Verify OSPFv3 neighbors are full.
3. Verify BGP neighbors are established.
4. Verify the route exists in the source protocol table.
5. Verify the next-hop is reachable.
6. Verify redistribution or `network` statements are correct.
7. Verify the route is installed in the RIB.

Useful commands:

```cisco
show ipv6 interface brief
show ipv6 ospf neighbor
show ipv6 ospf interface
show bgp ipv6 unicast summary
show bgp ipv6 unicast
show ipv6 route
show ipv6 route ospf
show ipv6 route bgp
```

Common symptoms and likely causes:

- **OSPFv3 neighbor stuck in INIT**: one-way hello reception, multicast filtering, or interface mismatch.
- **OSPFv3 neighbor stuck in EXSTART/EXCHANGE**: MTU mismatch or database sync issue.
- **BGP Idle/Active**: reachability, AS mismatch, TCP 179 filtering, or missing update-source.
- **BGP route present but not installed**: next-hop unreachable or better route exists.
- **Route missing from both protocols**: redistribution or `network` statement mismatch.

**Verification examples**

OSPFv3 neighbor state example:

```cisco
show ipv6 ospf neighbor

Neighbor ID     Pri   State           Dead Time   Interface ID    Interface
10.10.10.2        1   FULL/DR         00:00:31    7               GigabitEthernet0/0
```

Interpretation:

- `FULL` means adjacency is complete.
- `DR` indicates the neighbor is the Designated Router on that segment.
- The interface ID helps correlate the adjacency to the local link.

BGP IPv6 best-path example:

```cisco
show bgp ipv6 unicast 2001:db8:100::/64

BGP routing table entry for 2001:db8:100::/64, version 12
Paths: (1 available, best #1, table default)
  Advertised to update-groups:
     1
  65001
    2001:db8:12::2 from 2001:db8:12::2 (10.10.10.2)
      Origin IGP, metric 0, localpref 100, valid, external, best
```

Interpretation:

- `best` means this path won selection.
- `external` indicates eBGP-learned.
- If the route is valid and best but still missing from `show ipv6 route`, check next-hop recursion and RIB installation conditions.

**OSPFv3 versus BGP IPv6 comparison**

| Feature | OSPFv3 | BGP IPv6 |
|---|---|---|
| Primary role | Intra-domain routing | Policy and inter-domain routing |
| Neighbor basis | Link-local adjacency on a link | TCP session between peers |
| Convergence | Fast | Slower, policy-driven |
| Path selection | SPF (Shortest Path First) | Best-path algorithm |
| Scaling model | Areas and LSDB flooding | Route policy and attributes |
| Typical use | Campus, enterprise core | Edge, WAN, Internet, VPNs |
| Route origin | LSAs | Network statements, redistribution, or learned paths |

The most important exam distinction is that OSPFv3 is topology-driven, while BGP is policy-driven. OSPFv3 chooses the shortest path inside the domain; BGP chooses the best policy path across domains.

**Exam-focused troubleshooting checklist**

- Confirm `ipv6 unicast-routing` is enabled.
- Confirm OSPFv3 is enabled on the correct interface and area.
- Confirm OSPFv3 neighbors are `FULL`.
- Confirm BGP neighbors are `Established`.
- Confirm the BGP IPv6 address-family is activated.
- Confirm the route exists in the local RIB before using a `network` statement.
- Confirm next-hop reachability for BGP-learned prefixes.
- Confirm administrative distance is not hiding the expected route.
- Confirm redistribution policy does not filter the prefix.

**Key takeaways**

OSPFv3 and BGP IPv6 solve different problems, but they often coexist in the same enterprise design. OSPFv3 provides fast internal convergence and area-based scaling. BGP provides policy control, route origination, and edge connectivity. Most troubleshooting comes down to neighbor formation, address-family activation, next-hop reachability, and route-policy correctness.

If you remember only a few exam points, remember these:

- OSPFv3 neighbors use link-local addresses and multicast hellos, but some exchanges can be unicast.
- OSPFv3 DR/BDR election still matters on multiaccess links.
- BGP IPv6 on IOS XE uses the global BGP process plus IPv6 address-family activation.
- A BGP route must be best and have a reachable next-hop before it installs in the IPv6 RIB.
- Redistribution should be controlled with tags and filters to prevent loops.
