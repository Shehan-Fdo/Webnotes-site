## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Dual-stack troubleshooting mindset**

Dual-stack issues usually fall into one of four buckets: addressing, neighbor discovery, routing adjacency, or data-plane filtering. You troubleshoot faster when you separate IPv4 and IPv6 paths instead of assuming a single fault affects both.

A practical workflow is:

- Verify local interface state and addressing.
- Verify neighbor resolution:
  - IPv4 uses **ARP** (Address Resolution Protocol).
  - IPv6 uses **NDP** (Neighbor Discovery Protocol) and multicast.
- Verify routing adjacencies and learned routes.
- Verify first-hop redundancy and policy filters.
- Test end-to-end with `ping`, `traceroute`, and route lookups.

> If IPv4 works but IPv6 fails, suspect ICMPv6, multicast, link-local next-hop handling, or missing IPv6 address-family activation before you suspect the physical link.

**Layer 1 and Layer 2 checks**

Start with the basics because dual-stack failures often share the same physical or VLAN root cause.

- Confirm the interface is up/up with `show ip interface brief` and `show ipv6 interface brief`.
- Check speed, duplex, MTU, and encapsulation consistency.
- Verify the VLAN exists and the access/trunk configuration matches the topology.
- Confirm the SVI or routed interface has the expected IPv4 and IPv6 addresses.

Useful commands:

```bash
show interfaces status
show interfaces gigabitEthernet0/0
show ip interface brief
show ipv6 interface brief
show running-config interface gigabitEthernet0/0
```

For switched environments, also verify that the host VLAN is active and that the port is not err-disabled or blocked by a policy feature.

**IPv4 neighbor resolution and ARP troubleshooting**

IPv4 reachability depends on ARP for next-hop MAC resolution on the local segment. If ARP fails, the host may have a valid IP address and route but still cannot send frames to the gateway or peer.

Common symptoms:

- `ping` to the default gateway fails.
- The ARP table is incomplete or missing the next-hop entry.
- The interface shows packets out but no replies.
- The route exists, but the next-hop MAC never resolves.

Verification commands:

```bash
show ip arp
show arp
show ip route
ping <next-hop-ipv4>
traceroute <destination-ipv4>
```

Things to check:

- The default gateway IP is correct.
- The subnet mask matches the peer.
- The VLAN or subnet is correct on both ends.
- The gateway interface is up and has the expected MAC address.
- No ACL, port security, or DHCP snooping feature is blocking ARP or traffic.

A common IPv4 failure pattern is a host with the wrong mask. For example, a `/25` host trying to reach a gateway in the other half of the subnet may ARP for the wrong device or send traffic to the wrong next hop.

**IPv6 neighbor discovery and multicast dependency**

IPv6 does not use ARP. It uses **NDP**, which depends on ICMPv6 and multicast. That means IPv6 reachability can fail even when unicast routing looks correct if multicast or ICMPv6 is filtered.

Key IPv6 neighbor functions:

- **Neighbor Solicitation (NS)** discovers a neighbor’s link-layer address.
- **Neighbor Advertisement (NA)** replies with the MAC address.
- **Router Solicitation (RS)** asks for router information.
- **Router Advertisement (RA)** advertises prefixes, default gateway, and flags.

Verification commands:

```bash
show ipv6 neighbors
show ipv6 interface
show ipv6 route
ping ipv6 <destination>
traceroute ipv6 <destination>
```

Important troubleshooting points:

- IPv6 link-local addresses are always present on enabled interfaces.
- Many routing protocols and static routes use link-local next hops.
- If multicast is broken, NDP and routing protocol hellos may fail.
- If ICMPv6 is blocked, IPv6 breaks in subtle ways because ICMPv6 is required for control-plane functions, not just diagnostics.

> Do not treat ICMPv6 as optional. Blocking it can break neighbor discovery, PMTUD, and router discovery.

**IPv6 link-local next-hop troubleshooting**

IPv6 routes often point to a link-local next hop such as `fe80::1`. That is normal, but it changes how you verify reachability.

When a route uses a link-local next hop:

- The outgoing interface matters because the same link-local address can exist on multiple links.
- Recursive resolution must identify both the next-hop address and the egress interface.
- You should verify the neighbor table on the correct interface, not just the global routing table.

Example static route:

```bash
ipv6 route 2001:db8:10:1::/64 GigabitEthernet0/0 fe80::1
```

Verification:

```bash
show ipv6 route 2001:db8:10:1::/64
show ipv6 neighbors gigabitEthernet0/0
show ipv6 interface gigabitEthernet0/0
```

If the route is installed but traffic fails:

- Confirm the neighbor entry exists on the correct interface.
- Confirm the next-hop link-local address matches the neighbor’s interface.
- Confirm the upstream device is sending RAs or routing protocol hellos if dynamic routing is used.
- Confirm the interface is not filtered by an ACL that blocks ICMPv6 or routing protocol multicast.

**MTU, fragmentation, and Path MTU Discovery**

MTU mismatches can affect both IPv4 and IPv6, but the failure mode differs.

IPv4 behavior:

- IPv4 routers can fragment packets unless the DF (Don’t Fragment) bit is set.
- Hosts may still communicate even with some MTU mismatch, though performance suffers.
- PMTUD relies on ICMP Type 3 Code 4 messages.

IPv6 behavior:

- IPv6 routers do not fragment transit packets.
- The source host must adjust packet size based on **Packet Too Big** ICMPv6 messages.
- If ICMPv6 Packet Too Big is blocked, large IPv6 flows fail or stall.

Verification commands:

```bash
show interfaces gigabitEthernet0/0 | include MTU
ping ipv6 <destination> size 1400
ping ipv6 <destination> size 1400 df-bit
traceroute ipv6 <destination>
```

Troubleshooting guidance:

- Compare MTU values end to end, including tunnels and overlays.
- Check for ACLs or firewalls blocking ICMPv6 Packet Too Big.
- Remember that tunnels reduce effective MTU.
- If only large transfers fail, suspect PMTUD before routing.

**OSPFv2 and OSPFv3 adjacency troubleshooting**

OSPF troubleshooting depends on the address family in use. OSPFv2 is for IPv4. OSPFv3 is commonly used for IPv6, and on some platforms it can also support IPv4 through address-family configuration.

For OSPFv2, verify with:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip route ospf
```

For OSPFv3, verify with:

```bash
show ipv6 ospf neighbor
show ospfv3 neighbor
show ospfv3 interface brief
show ipv6 route ospf
```

Common adjacency problems:

- Area mismatch.
- Hello and dead timer mismatch.
- Authentication mismatch.
- MTU mismatch.
- Passive interface enabled.
- Network type mismatch.
- Missing interface activation.
- Wrong router ID.
- ACL blocking protocol multicast or control traffic.

Example OSPFv3 interface-based configuration for IPv6:

```bash
ipv6 unicast-routing

ipv6 router ospf 10
 router-id 10.10.10.10

interface GigabitEthernet0/0
 ipv6 address 2001:db8:1:1::1/64
 ipv6 ospf 10 area 0
```

Example OSPFv3 address-family style configuration on platforms that support it:

```bash
router ospfv3 10
 router-id 10.10.10.10
 address-family ipv4 unicast
  area 0
 exit-address-family
 address-family ipv6 unicast
  area 0
 exit-address-family
```

> OSPFv3 IPv4 support depends on platform and software support. Do not assume every OSPFv3 deployment carries IPv4 routes.

Practical checks:

- Confirm the interface is enabled for the correct OSPF process or address family.
- Confirm the neighbor is in the expected state, not stuck in `INIT`, `EXSTART`, or `EXCHANGE`.
- Use `show ipv6 ospf interface` or `show ospfv3 interface` to verify timers and MTU.
- If the adjacency forms but routes do not appear, check area type, summarization, and route filtering.

**EIGRP dual-stack and named-mode troubleshooting**

EIGRP can run for IPv4 and IPv6, including named mode on many Cisco platforms. The most common mistake is configuring the process but forgetting to activate the address family on the interface.

Classic IPv4 example:

```bash
router eigrp 100
 network 10.0.0.0 0.0.0.255
 no auto-summary
```

Classic IPv6 example:

```bash
ipv6 unicast-routing

ipv6 router eigrp 100
 eigrp router-id 10.10.10.10

interface GigabitEthernet0/0
 ipv6 address 2001:db8:1:1::1/64
 ipv6 eigrp 100
```

Named-mode example:

```bash
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  network 10.0.0.0 0.0.0.255
 exit-address-family
 address-family ipv6 unicast autonomous-system 100
  af-interface default
   no shutdown
  exit-af-interface
  af-interface GigabitEthernet0/0
   no shutdown
  exit-af-interface
 exit-address-family
```

Verification commands:

```bash
show ip eigrp neighbors
show ipv6 eigrp neighbors
show eigrp address-family ipv6 neighbors
show ip route eigrp
show ipv6 route eigrp
```

Common issues:

- The interface is not activated under the IPv6 address-family.
- K-values do not match.
- Authentication differs.
- Split horizon or summarization blocks expected routes.
- The router ID is missing or duplicated.
- Passive-interface is enabled on the wrong interface.

**BGP dual-stack troubleshooting**

BGP often fails in dual-stack environments because the session comes up for one address family but not the other, or because the next-hop is not reachable.

Key concepts:

- BGP neighbors are activated per address family on many platforms.
- IPv6 BGP commonly uses a global or link-local neighbor address.
- Route policy mistakes often affect only one address family.
- Next-hop reachability is critical before routes install.

Example configuration:

```bash
router bgp 65000
 neighbor 192.0.2.2 remote-as 65001
 neighbor 2001:db8:ffff::2 remote-as 65001

 address-family ipv4 unicast
  neighbor 192.0.2.2 activate
  neighbor 192.0.2.2 next-hop-self
 exit-address-family

 address-family ipv6 unicast
  neighbor 2001:db8:ffff::2 activate
  neighbor 2001:db8:ffff::2 next-hop-self
 exit-address-family
```

Verification commands:

```bash
show ip bgp summary
show bgp ipv6 unicast summary
show ip bgp
show bgp ipv6 unicast
show bgp ipv6 unicast neighbors
show bgp ipv6 unicast neighbors 2001:db8:ffff::2 advertised-routes
```

Common BGP dual-stack problems:

- The neighbor is configured but not activated under the address family.
- The update-source is wrong for loopback peering.
- The IPv6 next hop is unreachable.
- Route maps or prefix lists filter only one family.
- A route reflector reflects one family but not the other due to missing activation or policy.

Example of a loopback-based IPv6 peering pattern:

```bash
router bgp 65000
 neighbor 2001:db8:100::2 remote-as 65001
 neighbor 2001:db8:100::2 update-source Loopback0

 address-family ipv6 unicast
  neighbor 2001:db8:100::2 activate
  neighbor 2001:db8:100::2 next-hop-self
 exit-address-family
```

If the session is established but routes are missing:

- Check `show bgp ipv6 unicast neighbors <peer> received-routes`.
- Verify the prefix-list and route-map sequence order.
- Confirm the next-hop is reachable in the IPv6 routing table.
- Check whether the peer is advertising only link-local next hops and whether the local device can resolve them.

**First-hop redundancy: HSRP, VRRP, and GLBP**

First-hop redundancy problems often look like routing failures because hosts can reach the subnet but not the default gateway.

HSRP, VRRP, and GLBP provide a virtual default gateway for hosts.

HSRP example for IPv4:

```bash
interface Vlan10
 ip address 10.10.10.2 255.255.255.0
 standby 10 ip 10.10.10.1
 standby 10 priority 110
 standby 10 preempt
```

HSRP example for IPv6:

```bash
interface Vlan10
 ipv6 address 2001:db8:10:10::2/64
 standby version 2
 standby 10 ipv6 2001:db8:10:10::1
 standby 10 priority 110
 standby 10 preempt
```

VRRP example:

```bash
interface Vlan20
 ip address 10.20.20.2 255.255.255.0
 vrrp 20 ip 10.20.20.1
 vrrp 20 priority 110
 vrrp 20 preempt
```

GLBP example:

```bash
interface Vlan30
 ip address 10.30.30.2 255.255.255.0
 glbp 30 ip 10.30.30.1
 glbp 30 priority 110
 glbp 30 preempt
```

Verification commands:

```bash
show standby brief
show standby
show vrrp brief
show glbp brief
show ipv6 neighbors
show ipv6 route
```

Troubleshooting points:

- The virtual IP is not in the same subnet as the hosts.
- Preemption is disabled when it should be enabled.
- Tracking lowers priority unexpectedly.
- The virtual MAC is not learned by the switch or host.
- For IPv6, router advertisements may still advertise the wrong default gateway behavior if the interface is misconfigured.

For IPv6 first-hop redundancy, remember:

- Hosts may learn the default gateway through RA.
- The virtual router must advertise the correct prefix and flags.
- Duplicate default gateway advertisements can confuse hosts.
- If the virtual IPv6 gateway is reachable but hosts still fail, inspect RA behavior and neighbor cache entries.

**ACLs, ICMP, and control-plane filtering**

ACLs often cause dual-stack failures because they block the control traffic needed for routing and neighbor discovery.

IPv4 ACLs may block:

- ICMP echo and reply.
- OSPF protocol 89.
- EIGRP protocol 88.
- BGP TCP 179.
- ARP is not filtered by standard IP ACLs, but Layer 2 features can still affect it.

IPv6 ACLs may block:

- ICMPv6 neighbor discovery.
- ICMPv6 router solicitation and advertisement.
- ICMPv6 Packet Too Big.
- OSPFv3.
- EIGRP for IPv6.
- BGP TCP 179.

Example IPv6 ACL that preserves essential control traffic:

```bash
ipv6 access-list V6-FILTER
 permit icmp any any nd-na
 permit icmp any any nd-ns
 permit icmp any any router-advertisement
 permit icmp any any router-solicitation
 permit icmp any any packet-too-big
 permit icmp any any echo-request
 permit icmp any any echo-reply
 permit ospf any any
 permit tcp any any eq 179
 deny ipv6 any any log
```

Apply it carefully:

```bash
interface GigabitEthernet0/0
 ipv6 traffic-filter V6-FILTER in
```

Troubleshooting approach:

- Start with a temporary permit-all test if policy allows.
- Check ACL hit counters.
- Verify whether the filter is applied inbound or outbound.
- Remember that IPv6 control traffic is more sensitive to filtering than IPv4.

**Multicast and MLD considerations**

IPv6 neighbor discovery and router discovery depend on multicast. In some environments, multicast suppression, snooping issues, or Layer 2 filtering can break IPv6 even when unicast looks fine.

Relevant concepts:

- **MLD** (Multicast Listener Discovery) is the IPv6 equivalent of IGMP for multicast group membership.
- NDP uses solicited-node multicast addresses.
- Routing protocols may use multicast hello packets.
- Some switches or wireless systems suppress unknown multicast or require snooping support.

Symptoms:

- IPv6 neighbors never appear.
- RAs are not received.
- OSPFv3 or EIGRPv6 neighbors do not form.
- Only one direction of IPv6 communication works.

Checks:

```bash
show ipv6 neighbors
show ipv6 interface
show ipv6 mld interface
show ipv6 mld groups
show mac address-table multicast
```

If multicast is the issue:

- Verify snooping and querier behavior.
- Confirm the VLAN permits IPv6 multicast.
- Check for storm-control or suppression features.
- Validate wireless or overlay policies that may drop link-local multicast.

**Route verification and recursive lookup**

A route can exist in the table but still fail because the next hop is unresolved or recursively broken.

Verify both families:

```bash
show ip route
show ipv6 route
show ip cef
show ipv6 cef
```

What to look for:

- Connected routes for the next hop.
- Recursive resolution to a valid outgoing interface.
- A fully installed CEF entry, not just a RIB entry.
- A route that points to a link-local next hop with the correct interface.

If a route is present but traffic fails:

- Check whether the next hop is reachable.
- Check whether the outgoing interface is administratively down.
- Check whether policy-based routing or VRF separation is interfering.
- Check whether the route is installed in the correct VRF.

**Practical end-to-end troubleshooting workflow**

Use a repeatable sequence so you do not chase symptoms.

1. Verify the interface and addressing.
2. Verify local neighbor resolution.
3. Verify the routing table and next hop.
4. Verify the dynamic routing adjacency.
5. Verify first-hop redundancy.
6. Verify ACLs, multicast, and ICMP control traffic.
7. Test with targeted pings and traceroutes.

Example workflow for an IPv6 failure:

```bash
show ipv6 interface brief
show ipv6 neighbors
show ipv6 route
show ipv6 ospf neighbor
show ipv6 access-list
ping ipv6 2001:db8:200::1
traceroute ipv6 2001:db8:200::1
```

Example workflow for an IPv4 failure:

```bash
show ip interface brief
show ip arp
show ip route
show ip ospf neighbor
show access-lists
ping 10.10.10.1
traceroute 10.10.10.1
```

**Worked scenario: IPv6 host cannot reach the gateway**

Symptom: A host has a valid IPv6 address but cannot ping the default gateway.

Likely causes:

- RA is missing or blocked.
- NDP multicast is filtered.
- The gateway interface lacks IPv6 addressing.
- The host has a stale neighbor cache entry.
- An IPv6 ACL blocks ICMPv6.

Verification:

```bash
show ipv6 interface vlan 10
show ipv6 neighbors
show ipv6 access-list
show ipv6 route
```

Fixes:

- Ensure `ipv6 unicast-routing` is enabled where needed.
- Confirm the gateway interface has a global IPv6 address.
- Permit ICMPv6 ND and Packet Too Big messages.
- Clear stale neighbor entries if necessary.

**Worked scenario: OSPFv3 neighbors stuck in EXSTART**

Symptom: OSPFv3 adjacency forms partially but never reaches FULL.

Likely causes:

- MTU mismatch.
- Duplicate router ID.
- Authentication mismatch.
- Interface activation mismatch.

Verification:

```bash
show ipv6 ospf neighbor
show ipv6 ospf interface
show interfaces gigabitEthernet0/0 | include MTU
show running-config | section ospf
```

Fixes:

- Match MTU on both sides.
- Confirm the same area and timers.
- Ensure the interface is enabled under the correct OSPFv3 process or address family.
- Verify authentication settings match exactly.

**Worked scenario: BGP IPv6 routes missing after session establishment**

Symptom: The BGP session is established, but no IPv6 prefixes appear in the table.

Likely causes:

- The IPv6 address family is not activated.
- The peer is not advertising IPv6 routes.
- A route-map or prefix-list filters the routes.
- The next hop is unreachable.

Verification:

```bash
show bgp ipv6 unicast summary
show bgp ipv6 unicast neighbors 2001:db8:ffff::2
show bgp ipv6 unicast neighbors 2001:db8:ffff::2 advertised-routes
show ipv6 route
```

Fixes:

- Activate the neighbor under `address-family ipv6 unicast`.
- Add or correct `next-hop-self` where appropriate.
- Review route policy and prefix filters.
- Confirm the peer’s update-source and reachability.

**Key takeaways**

- IPv4 troubleshooting often starts with ARP and subnet correctness.
- IPv6 troubleshooting often starts with NDP, multicast, and ICMPv6.
- Link-local next hops require interface-aware verification.
- MTU and PMTUD issues are more disruptive in IPv6 because routers do not fragment transit packets.
- Dynamic routing failures often come from missing address-family activation, policy filters, or control-plane ACLs.
- First-hop redundancy must be verified separately for IPv4 and IPv6, including virtual addresses and RA behavior.
