## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Redistribution fundamentals**

IPv6 redistribution injects routes learned by one routing source into another routing protocol so that separate routing domains can reach each other. In Cisco IOS XE, the exact command placement depends on the protocol and whether you use classic or named configuration. The exam focus is not just syntax; it is understanding which protocol needs a seed metric, how filtering is applied, and how to prevent loops when routes move between multiple domains.

A redistributed route is usually treated as an **external** route by the receiving protocol. That matters because external routes often have different administrative distance, metric behavior, and selection rules than internal routes.

- **Seed metric**: the starting metric assigned to redistributed routes.
- **Route-map**: the policy tool used to match, permit, deny, tag, and set attributes.
- **Tag**: a numeric marker used to identify redistributed routes and prevent re-redistribution loops.

**Where redistribution is configured**

Different protocols place redistribution in different configuration modes:

- **OSPFv3**: typically under `ipv6 router ospf <process-id>` or the relevant address-family context on platforms using named mode.
- **EIGRP for IPv6**: under `ipv6 router eigrp <asn>` or the named EIGRP IPv6 address-family context.
- **BGP for IPv6**: under `router bgp <asn>` and then `address-family ipv6 unicast`.

The practical exam skill is to recognize the correct context before you type the redistribution command. A command that is valid in one protocol family may be invalid in another.

**OSPFv3 redistribution behavior**

OSPFv3 accepts redistributed routes as external LSAs (Link-State Advertisements). External routes can be type 1 or type 2:

- **E1**: external metric plus internal cost to the ASBR (Autonomous System Boundary Router).
- **E2**: external metric only, which is the default in many configurations.

E1 is often preferred when you want the path cost to reflect both the external route and the cost to reach the redistributing router. E2 is often used when the external metric should dominate and internal topology should not influence the choice as much.

> OSPF external route type affects path selection, but it does not change the administrative distance. OSPF internal and external routes both use AD 110 on Cisco IOS XE.

A typical OSPFv3 redistribution example looks like this:

```cisco
ipv6 router ospf 10
 router-id 10.10.10.10
 redistribute eigrp 100 metric 20 metric-type 1 route-map RM-EIGRP-TO-OSPF
 redistribute static metric 50 metric-type 2
```

Key points:

- `metric` sets the external cost.
- `metric-type 1` or `metric-type 2` selects E1 or E2.
- `route-map` can filter routes and set tags or metrics, but it does not directly set OSPF external type; that is set on the `redistribute` command.

A route-map example:

```cisco
route-map RM-EIGRP-TO-OSPF permit 10
 match ipv6 address prefix-list PL-REDIST
 set tag 10010
 set metric 20
```

Important behavior:

- If the route-map has no matching permit sequence, the route is denied by the route-map’s implicit deny.
- If you want to allow only selected prefixes, you must explicitly permit them.
- A route-map can set a tag for loop prevention, which is critical in mutual redistribution designs.

**EIGRP for IPv6 redistribution behavior**

EIGRP requires a seed metric for redistributed routes. On IOS XE, if you do not provide a metric or a `default-metric`, redistributed routes are not installed into EIGRP as expected because EIGRP needs bandwidth, delay, reliability, load, and MTU values to compute its composite metric.

A common configuration pattern is:

```cisco
ipv6 router eigrp 100
 eigrp router-id 100.100.100.100
 redistribute ospf 10 metric 100000 100 255 1 1500 route-map RM-OSPF-TO-EIGRP
 default-metric 100000 100 255 1 1500
```

Notes:

- The metric values are in the EIGRP order: bandwidth, delay, reliability, load, MTU.
- `default-metric` is useful when multiple redistribution statements should share the same seed metric.
- A route-map can still override or filter routes, but the receiving EIGRP process still needs a usable metric.

EIGRP verification commonly uses:

```cisco
show ipv6 eigrp topology
show ipv6 route eigrp 100
show ipv6 protocols
```

Depending on platform and IOS XE release, `show ipv6 route eigrp` may be supported with or without the ASN. For exam and lab work, `show ipv6 eigrp topology` is often the most reliable way to confirm redistributed entries and their metrics.

**BGP IPv6 redistribution behavior**

BGP is often used as a redistribution hub because it can carry policy-rich route attributes and scale better than many IGPs. For IPv6, redistribution is typically configured under the IPv6 address family:

```cisco
router bgp 65000
 bgp router-id 1.1.1.1
 address-family ipv6 unicast
  redistribute ospf 10 route-map RM-OSPF-TO-BGP
  redistribute static route-map RM-STATIC-TO-BGP
```

Important BGP points:

- The IPv6 address family must be activated.
- The routes must exist in the local routing table or be eligible for redistribution.
- Redistribution does not guarantee advertisement unless the routes are accepted into BGP and pass policy.
- BGP policy often uses prefix-lists, route-maps, communities, and tags together.

A practical BGP policy example:

```cisco
ipv6 prefix-list PL-BGP-REDIST seq 10 permit 2001:db8:10::/48
ipv6 prefix-list PL-BGP-REDIST seq 20 permit 2001:db8:20::/48

route-map RM-STATIC-TO-BGP permit 10
 match ipv6 address prefix-list PL-BGP-REDIST
 set community 65000:100 additive
 set tag 65010

route-map RM-STATIC-TO-BGP deny 100
```

This design does three things:

- Allows only selected prefixes.
- Tags redistributed routes for later filtering or loop prevention.
- Adds a community so downstream BGP policy can recognize the route.

**Filtering methods and where they apply**

Filtering is one of the most tested redistribution topics. The key idea is that a prefix-list alone does nothing until it is referenced by a route-map, distribute-list, or another policy mechanism supported by the protocol.

Common filtering tools:

- **Prefix-list**: matches prefixes and prefix lengths.
- **Route-map**: applies match and set logic.
- **Distribute-list**: filters routes in some routing protocols and contexts.
- **Prefix-based policy in BGP**: often uses prefix-lists directly in route-maps or neighbor policies.

A prefix-list example:

```cisco
ipv6 prefix-list PL-REDIST seq 10 permit 2001:db8:100::/48
ipv6 prefix-list PL-REDIST seq 20 permit 2001:db8:200::/48
ipv6 prefix-list PL-REDIST seq 30 deny ::/0 le 128
```

This list only becomes useful when referenced:

```cisco
route-map RM-FILTER permit 10
 match ipv6 address prefix-list PL-REDIST
```

If you forget to attach the prefix-list to a route-map or protocol filter, it has no effect.

**Route-map logic and implicit deny**

Route-maps are evaluated top-down by sequence number. The first matching sequence decides the action.

- `permit` sequence: the route is allowed and any `set` actions are applied.
- `deny` sequence: the route is blocked.
- No match in any sequence: implicit deny.

Example:

```cisco
route-map RM-OSPF-TO-EIGRP permit 10
 match ipv6 address prefix-list PL-OSPF-ALLOW
 set tag 20010

route-map RM-OSPF-TO-EIGRP deny 20
 match ipv6 address prefix-list PL-OSPF-BLOCK

route-map RM-OSPF-TO-EIGRP permit 100
```

Interpretation:

- Sequence 10 permits selected routes and tags them.
- Sequence 20 explicitly blocks a known bad set.
- Sequence 100 is a catch-all permit for anything not matched earlier.

If you omit the final permit and no earlier sequence matches, the route is denied by the implicit deny. That is a common troubleshooting issue.

**Route tagging and loop prevention**

Tagging is essential in mutual redistribution. Without tags, a route learned from OSPFv3 can be redistributed into EIGRP, then redistributed back into OSPFv3, creating feedback loops or route churn.

A typical loop-prevention design:

- Tag routes when they enter a new domain.
- Match the tag on the return path.
- Deny re-redistribution of tagged routes.

Example:

```cisco
route-map RM-EIGRP-TO-OSPF permit 10
 match ipv6 address prefix-list PL-EIGRP-EXPORT
 set tag 100

route-map RM-OSPF-TO-EIGRP deny 10
 match tag 100

route-map RM-OSPF-TO-EIGRP permit 20
 match ipv6 address prefix-list PL-OSPF-EXPORT
 set tag 200
```

How this works:

- EIGRP routes entering OSPF get tag `100`.
- When OSPF tries to send routes back into EIGRP, any route with tag `100` is blocked.
- The same pattern can be extended across BGP, OSPFv3, and EIGRP.

This is one of the most important CCNP-level redistribution concepts because it prevents loops in multi-protocol designs.

**Mutual redistribution design patterns**

Mutual redistribution means two protocols exchange routes in both directions. This is common in migration and integration scenarios, but it is also where loops and suboptimal path selection appear.

Typical examples:

- **OSPFv3 <-> EIGRP for IPv6**
- **OSPFv3 <-> BGP IPv6**
- **EIGRP for IPv6 <-> BGP IPv6**

Best practices:

- Tag routes on ingress.
- Filter tagged routes on re-entry.
- Use route-maps for both filtering and attribute setting.
- Keep redistribution points limited to a small number of routers.
- Prefer a single redistribution hub when possible.

A simple topology idea:

- Site A runs OSPFv3.
- Site B runs EIGRP for IPv6.
- A border router redistributes between them.
- The border router tags routes from each side and blocks them from returning to the source protocol.

**Administrative distance and route selection**

Administrative distance (AD) is the trust value used when multiple routing sources advertise the same prefix. Lower AD wins.

Common IPv6-related Cisco AD values:

| Route source | Typical AD |
|---|---:|
| Connected | 0 |
| Static | 1 |
| eBGP | 20 |
| EIGRP internal | 90 |
| IGRP | 100 |
| OSPF internal/external | 110 |
| IS-IS | 115 |
| RIPng | 120 |
| EIGRP external | 170 |
| iBGP | 200 |

Important exam points:

- OSPF internal and OSPF external both use AD 110.
- EIGRP external routes use AD 170, which is worse than EIGRP internal routes.
- A redistributed route does not automatically win just because it was redistributed; AD and metric still matter.
- In a clean design, you usually avoid having a directly connected route and a redistributed route for the exact same prefix on the same router.

**Filtering with distribute-lists in IPv6**

Distribute-lists can filter routes in some IPv6 routing contexts, but support varies by protocol and IOS XE release. In practice, route-maps and prefix-lists are usually more flexible and more common for redistribution policy.

A conceptual example in a protocol that supports it:

```cisco
ipv6 prefix-list PL-IN seq 10 permit 2001:db8:1::/48
ipv6 prefix-list PL-IN seq 20 deny ::/0 le 128

ipv6 router eigrp 100
 distribute-list prefix PL-IN in
```

Use this carefully:

- It filters routes learned by the protocol, not necessarily redistributed routes unless the protocol supports that placement.
- It is less expressive than route-maps for tagging and attribute manipulation.
- For redistribution design, route-maps are usually the primary tool.

**Verification workflow**

Use a layered verification approach: confirm the source route exists, confirm the redistribution policy matches it, and confirm the destination protocol installed it.

1. Verify the source route is present.
2. Verify the redistribution configuration and route-map logic.
3. Verify the destination protocol table and routing table.
4. Check tags, metrics, and route type.
5. Confirm the route is not blocked by an implicit deny.

Useful commands:

```cisco
show ipv6 route
show ipv6 route ospf
show ipv6 route eigrp 100
show ipv6 route bgp
show ipv6 protocols
show ipv6 ospf database external
show ipv6 eigrp topology
show bgp ipv6 unicast
show route-map
show ipv6 prefix-list
```

Sample interpretation:

- If the source route exists in `show ipv6 route` but not in the destination protocol, the redistribution policy is likely blocking it.
- If the route appears in the destination protocol but not in the global routing table, another route with a better AD may be winning.
- If the route-map counters stay at zero, the match statement is probably wrong or the route is not being considered for redistribution.

**Practical troubleshooting examples**

If EIGRP does not install redistributed IPv6 routes:

- Check that a seed metric or `default-metric` is configured.
- Confirm the route-map permits the prefix.
- Verify the source route exists and is active.
- Check whether the route is being filtered by an implicit deny.

If OSPFv3 does not advertise redistributed routes:

- Confirm the route is present in the routing table.
- Verify the `redistribute` command is under the correct OSPFv3 process or address-family context.
- Check whether the route-map denies the prefix.
- Confirm the external LSA appears in the OSPFv3 database.

If BGP does not show redistributed IPv6 routes:

- Confirm `address-family ipv6 unicast` is active.
- Verify the route exists in the local routing table.
- Check the route-map and prefix-list match.
- Confirm the route is accepted into the BGP table with `show bgp ipv6 unicast`.

**Sample end-to-end design**

A common enterprise design uses OSPFv3 in the campus, EIGRP for IPv6 in a branch, and BGP at the edge.

```cisco
ipv6 prefix-list PL-CAMPUS seq 10 permit 2001:db8:10::/48
ipv6 prefix-list PL-BRANCH seq 10 permit 2001:db8:20::/48

route-map RM-OSPF-TO-EIGRP deny 10
 match tag 100

route-map RM-OSPF-TO-EIGRP permit 20
 match ipv6 address prefix-list PL-CAMPUS
 set tag 100

route-map RM-EIGRP-TO-OSPF deny 10
 match tag 200

route-map RM-EIGRP-TO-OSPF permit 20
 match ipv6 address prefix-list PL-BRANCH
 set tag 200
```

This design:

- Limits redistribution to known prefixes.
- Tags routes on entry.
- Blocks routes from returning to their source domain.
- Reduces the chance of loops and route feedback.

**Exam focus summary**

- **OSPFv3** uses external route types E1 and E2; external type is set on the redistribution command.
- **EIGRP for IPv6** requires a seed metric or `default-metric`.
- **BGP IPv6** redistribution happens under `address-family ipv6 unicast`.
- **Route-maps** control filtering, tagging, and attribute setting.
- **Prefix-lists** must be referenced to have any effect.
- **Tags** are the primary loop-prevention tool in mutual redistribution.
- **Implicit deny** in route-maps is a frequent cause of missing routes.
- **Verification** should check source routes, policy matches, and destination protocol tables.

**Quick comparison**

| Protocol | Redistribution context | Metric requirement | Common external behavior |
|---|---|---|---|
| OSPFv3 | OSPFv3 process or address-family | External metric required on redistribute | E1/E2 external LSAs |
| EIGRP for IPv6 | EIGRP IPv6 process or address-family | Seed metric required | External routes use AD 170 |
| BGP IPv6 | `address-family ipv6 unicast` | No EIGRP-style seed metric, but policy still matters | Policy-rich redistribution and advertisement |

**Common mistakes**

- Forgetting the EIGRP seed metric.
- Applying a prefix-list but never referencing it in a route-map.
- Assuming route-maps can set OSPF external type.
- Omitting tags in mutual redistribution.
- Misplacing the redistribution command in the wrong protocol context.
- Trusting a route-map permit sequence without checking for an earlier match or implicit deny.

**Practical lab checklist**

- Confirm the source prefix exists.
- Confirm the destination protocol is enabled for IPv6.
- Add a route-map with explicit permit and deny logic.
- Set a tag on redistributed routes.
- Block tagged routes from being re-redistributed.
- Verify with protocol-specific show commands.
- Compare route selection using AD and protocol metrics.
