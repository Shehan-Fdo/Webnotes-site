## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**What link-local next hops are**

An IPv6 **link-local address** is an address that is valid only on the local Layer 2 segment. It is not routed beyond the local link, and it is commonly used for neighbor discovery, routing protocol adjacencies, and next-hop resolution.

Cisco devices often prefer link-local next hops in IPv6 because they are stable within the local segment and do not depend on a globally routed address. This is especially useful in routed point-to-point links and in dynamic routing protocols where the neighbor’s global address may change.

**Why Cisco uses link-local next hops**

Link-local next hops reduce dependence on global addressing and make routing more resilient on directly connected links. The router can identify the neighbor by its local interface context rather than by a globally unique address that might be renumbered.

This matters in both static and dynamic routing:

- **Static routes** can point to a link-local next hop, but Cisco IOS and IOS XE require the outgoing interface to be specified.
- **Dynamic routing protocols** often advertise or install routes with link-local next hops automatically, and the interface context is carried in the protocol state or RIB entry.
- **Verification output** may show the link-local address plus the outgoing interface, which is the key clue that the route is resolved correctly.

> A link-local next hop is only meaningful on the local segment. If the router cannot associate it with the correct outgoing interface, the route is not usable.

**Static IPv6 routes with link-local next hops**

For Cisco IOS/IOS XE, a static IPv6 route that uses a link-local next hop must include the outgoing interface. This is the exam-critical rule.

Typical syntax:

```bash
ipv6 route <prefix>/<length> <exit-interface> <link-local-next-hop>
```

Example:

```bash
ipv6 route 2001:db8:10:1::/64 GigabitEthernet0/0 fe80::2
```

Why the interface is mandatory:

- A link-local address is not globally unique.
- The router needs the interface to know which neighbor table entry to use.
- Without the interface, the route is ambiguous and is generally rejected or not installed as intended.

If you omit the interface for a link-local next hop, Cisco IOS/IOS XE treats the route as invalid for that form of next-hop specification.

A common exam trap is confusing this with a global unicast next hop. A global unicast next hop may be used in a static route when it is directly reachable and resolvable, but that is a different case from a link-local next hop.

**Static route examples and behavior**

A directly connected global unicast next hop can sometimes be configured without an explicit interface if the router can resolve it on the local link:

```bash
ipv6 route 2001:db8:10:1::/64 2001:db8:12::2
```

Use this carefully in lab and exam contexts:

- It works only when the next hop is directly connected and resolvable.
- It is less explicit than specifying the interface.
- For exam answers, the safest and clearest design is to specify the outgoing interface when there is any ambiguity.

A more explicit and robust form is:

```bash
ipv6 route 2001:db8:10:1::/64 GigabitEthernet0/0 2001:db8:12::2
```

**Verification of static link-local routes**

Use `show ipv6 route` to confirm that the route is installed in the IPv6 routing table.

Example output pattern:

```bash
R1# show ipv6 route static
IPv6 Routing Table - 2001:DB8:10:1::/64
  Known via "static", distance 1, metric 0
  Routing Descriptor Blocks:
  * via FE80::2, GigabitEthernet0/0
```

What to look for:

- The route appears under the expected prefix.
- The next hop is shown as a link-local address.
- The outgoing interface is displayed alongside it.
- The route is marked as active with `*` when installed.

`show ipv6 route` can also reveal whether the route is missing entirely, which usually means the next hop is unresolved, the interface is down, or the syntax is wrong.

**CEF and adjacency resolution**

Cisco Express Forwarding (<abbr title="Cisco Express Forwarding">CEF</abbr>) is important because the route may exist in the Routing Information Base (<abbr title="Routing Information Base">RIB</abbr>) but still fail to forward if the adjacency is unresolved.

Useful commands:

```bash
show ipv6 cef
show ipv6 cef 2001:db8:10:1::/64
show ipv6 neighbors
```

What you want to confirm:

- The next hop resolves to a neighbor entry on the correct interface.
- The adjacency is not incomplete or stale.
- The router has a valid Layer 2 mapping for the link-local neighbor.

If the neighbor entry is missing, the route may appear in the RIB but traffic still fails because CEF cannot build a usable adjacency.

**Dynamic routing protocols and link-local next hops**

Cisco IPv6 routing protocols commonly use link-local addresses for next-hop behavior. This is normal and expected.

**OSPFv3**

OSPFv3 neighbors are identified using link-local addresses on the local link. Routes learned through OSPFv3 often show a link-local next hop in the routing table.

Typical verification:

```bash
show ipv6 ospf neighbor
show ipv6 route ospf
```

What to recognize:

- Neighbor adjacency forms over link-local addresses.
- The route entry may show `via FE80::...` with the outgoing interface.
- The interface context is essential because the same link-local address format can exist on multiple links.

**EIGRP for IPv6**

EIGRP for IPv6 also uses link-local addresses for neighbor relationships and route next hops.

Typical verification:

```bash
show ipv6 eigrp neighbors
show ipv6 route eigrp
```

What to recognize:

- The neighbor table lists link-local addresses.
- The route table may show a link-local next hop and the interface.
- Successor and feasible successor logic still applies, but the next-hop address itself is usually link-local.

**IS-IS for IPv6**

IS-IS commonly uses link-local addressing on the interface for adjacency formation and route resolution.

Typical verification:

```bash
show isis neighbors
show ipv6 route isis
```

What to recognize:

- Adjacencies are tied to the local interface.
- The next hop in the IPv6 route table may be link-local.
- The protocol relies on the interface context to distinguish neighbors.

**RIPng**

RIPng uses link-local addresses for next-hop information on the local link.

Typical verification:

```bash
show ipv6 rip
show ipv6 route rip
```

What to recognize:

- Learned routes often show a link-local next hop.
- The outgoing interface is part of the route resolution.
- This is normal and not a sign of misconfiguration by itself.

**How to read protocol-learned link-local next hops**

When a route is learned dynamically, you usually do not manually configure the next hop. The protocol carries the neighbor identity and interface context for you.

Key exam idea:

- The next hop may be link-local.
- The interface is still required for resolution.
- The protocol, not the administrator, supplies the relationship between the address and the link.

This is why link-local next hops are common in routing tables even when you never type them into a static route.

**Lab topology example**

A simple point-to-point topology helps illustrate the behavior:

```text
R1 G0/0 ---- G0/0 R2
R1: 2001:db8:12::1/64
R2: 2001:db8:12::2/64
R2 link-local: fe80::2
```

On R1, a static route to a remote prefix behind R2:

```bash
ipv6 route 2001:db8:20:1::/64 GigabitEthernet0/0 fe80::2
```

Verification:

```bash
R1# show ipv6 route 2001:db8:20:1::/64
IPv6 Routing Table - 2001:DB8:20:1::/64
  Known via "static", distance 1, metric 0
  Routing Descriptor Blocks:
  * via FE80::2, GigabitEthernet0/0
```

If the route is missing, check:

```bash
show ipv6 interface brief
show ipv6 neighbors
show running-config | include ipv6 route
```

**Troubleshooting a failing link-local static route**

A common failure is configuring a link-local next hop without the outgoing interface.

Bad configuration:

```bash
ipv6 route 2001:db8:20:1::/64 fe80::2
```

Likely result:

- The route is not installed as intended.
- The router cannot uniquely identify the neighbor.
- Traffic to the destination prefix fails.

How to isolate the issue:

1. Check the route configuration.
   ```bash
   show running-config | include ipv6 route
   ```
2. Check whether the interface is up/up.
   ```bash
   show ipv6 interface brief
   ```
3. Check whether the neighbor is reachable on that link.
   ```bash
   show ipv6 neighbors
   ```
4. Check whether the route is in the IPv6 table.
   ```bash
   show ipv6 route static
   ```
5. Check CEF adjacency resolution.
   ```bash
   show ipv6 cef 2001:db8:20:1::/64
   ```

Corrected configuration:

```bash
conf t
ipv6 route 2001:db8:20:1::/64 GigabitEthernet0/0 fe80::2
end
```

If the route still fails after correction, the likely causes are:

- The interface is down.
- The neighbor is not present on the local link.
- The Layer 2 adjacency is broken.
- The wrong interface was specified.
- The next-hop address belongs to a different neighbor than expected.

**Common verification patterns**

Use these commands to build a complete picture:

```bash
show ipv6 route
show ipv6 route static
show ipv6 neighbors
show ipv6 cef
show ipv6 interface brief
show running-config | section ipv6 route
```

Interpretation tips:

- `show ipv6 route` tells you whether the route is installed.
- `show ipv6 neighbors` tells you whether the next hop is discoverable on the link.
- `show ipv6 cef` tells you whether forwarding can actually occur.
- `show ipv6 interface brief` tells you whether the outgoing interface is operational.

**Design guidance for exams and labs**

Use link-local next hops when you want stable local-link resolution and protocol consistency. This is especially common on point-to-point links and in routing protocols that already operate with interface-local neighbor relationships.

Prefer explicit interface specification for static routes when:

- The next hop is link-local.
- The topology has multiple possible exits.
- You want unambiguous exam-safe configuration.
- You are troubleshooting a route that is not installing correctly.

Use a global unicast next hop only when it is clearly directly connected and the lab or platform behavior supports that resolution. Do not assume that a global next hop alone is always sufficient in every Cisco context.

**Quick comparison**

| Case | Static route syntax | Notes |
|---|---|---|
| Link-local next hop | `ipv6 route 2001:db8:20:1::/64 GigabitEthernet0/0 fe80::2` | Interface is required |
| Directly connected global next hop | `ipv6 route 2001:db8:20:1::/64 2001:db8:12::2` | Works only when resolvable on-link |
| Safer explicit global next hop | `ipv6 route 2001:db8:20:1::/64 GigabitEthernet0/0 2001:db8:12::2` | Clear and exam-friendly |

**Key exam takeaways**

- **Link-local addresses** are local to the link and are not routed.
- A static IPv6 route with a **link-local next hop** must include the outgoing interface on Cisco IOS/IOS XE.
- Dynamic routing protocols commonly use link-local next hops automatically.
- Verification should include the routing table, neighbor table, and CEF adjacency state.
- If the route is present but traffic fails, suspect adjacency or interface resolution before assuming the prefix is wrong.

**Practical mental model**

Think of a link-local next hop as “the neighbor on this interface,” not as a globally reachable destination. The address alone is not enough; the interface gives the router the context needed to forward packets correctly.
