## **Course:** Cisco CCNP Enterprise Advanced Infrastructure (ENCOR 350-401 & ENARSI 300-410)

**Dual-stack design overview**

Dual-stack means you run **IPv4 and IPv6 at the same time** on the same network, devices, and often the same links. It is the most common enterprise transition model because it preserves IPv4 service while enabling IPv6 adoption without forcing an immediate cutover.

The design goal is not just “make IPv6 work.” You need to decide how both protocols will coexist, how routes will be exchanged, how traffic will be forwarded, and how you will migrate in phases without breaking applications.

- **IPv4 continuity** keeps legacy applications and dependencies working.
- **IPv6 enablement** prepares the network for growth, modern services, and future-only IPv6 environments.
- **Operational parity** matters: monitoring, security, and troubleshooting must work for both stacks.
- **Design consistency** reduces risk when you deploy routing protocols, summarization, and policy.

**Core dual-stack principles**

Dual-stack is not tunneling and not translation. Each host or router has separate IPv4 and IPv6 control-plane and data-plane behavior, and each protocol has its own addressing, routing, and neighbor discovery mechanisms.

Key implications:

- **Separate routing tables** exist for IPv4 and IPv6.
- **Separate adjacencies** form for routing protocols unless the protocol explicitly shares a process with multiple address families.
- **Separate ACLs, QoS policies, and management reachability** may be required.
- **Separate troubleshooting paths** are often needed because a device can be healthy in one stack and broken in the other.

> <mark>Important:</mark> Dual-stack does not automatically mean equal path selection. IPv4 and IPv6 may follow different topologies, policies, or next hops unless you intentionally align them.

**Addressing and forwarding behavior**

IPv6 forwarding depends heavily on **Neighbor Discovery (ND)**, which replaces ARP for many local-link functions. ND handles neighbor resolution, router discovery, and reachability detection. In dual-stack environments, this means IPv6 forwarding can fail even when routing looks correct if ND is broken.

Important IPv6 forwarding behaviors:

- **Link-local addresses** are always present on enabled IPv6 interfaces and are used for next-hop resolution and routing protocol adjacencies.
- **Neighbor Discovery** resolves the Layer 2 destination for IPv6 next hops.
- **Source address selection** affects how hosts and routers choose the source for locally generated traffic.
- **Destination selection** can influence whether traffic prefers IPv4 or IPv6 when both are available.

Practical troubleshooting considerations:

- If a route exists but traffic fails, verify the **next-hop reachability** and ND state.
- If a router can ping a destination but applications fail, check **source address selection** and policy routing.
- If one stack works and the other does not, compare interface status, MTU, ACLs, and routing advertisements for both protocols.

**Dual-stack deployment patterns**

Enterprises usually deploy dual-stack in phases. The right sequence depends on risk, application dependency, and operational maturity.

Common rollout patterns:

- **Pilot-first**
  - Enable dual-stack on a small user segment or lab-like production segment.
  - Validate routing, DNS, security, and monitoring before broad rollout.
- **Core-first**
  - Enable IPv6 in the backbone and distribution layers first.
  - Extend to access and edge later.
  - Useful when you want routing infrastructure ready before endpoints.
- **Edge-first**
  - Enable dual-stack at internet-facing or partner-facing edges first.
  - Useful when external services or SaaS integration drives IPv6 adoption.
- **Per-VRF or per-segment**
  - Roll out one VRF, campus block, or business unit at a time.
  - Helps isolate risk and simplify rollback.

A practical migration plan often follows this order:

1. Build IPv6 addressing and routing design.
2. Enable dual-stack on core and distribution infrastructure.
3. Validate management access, logging, and monitoring.
4. Extend to access segments and critical services.
5. Migrate applications and user groups in controlled waves.
6. Optimize summarization, policy, and security after stabilization.

**IPv6 addressing design**

A clean IPv6 plan improves summarization, troubleshooting, and growth. Most enterprise designs allocate a large prefix from the ISP or internal registry and subdivide it by site, function, or VRF.

Common design practices:

- Use a **consistent hierarchical allocation** so summaries are predictable.
- Assign prefixes by **site, building, floor, or VRF**.
- Reserve space for growth and future segments.
- Keep infrastructure and user prefixes distinct when possible.

Example allocation model:

- `2001:db8:1000::/48` for the enterprise
- `2001:db8:1000:10::/64` for campus core
- `2001:db8:1000:20::/64` for users
- `2001:db8:1000:30::/64` for servers
- `2001:db8:1000:40::/64` for voice or IoT

**Cisco IOS XE dual-stack interface configuration**

On Cisco IOS XE, you typically enable IPv4 and IPv6 on the same interface. IPv6 requires global IPv6 routing to be enabled first.

```cisco
conf t
ipv6 unicast-routing

interface GigabitEthernet0/0
 description Uplink to Distribution
 ip address 10.10.10.1 255.255.255.252
 ipv6 address 2001:db8:1000:10::1/64
 ipv6 enable
 no shutdown
end
```

Notes:

- `ipv6 unicast-routing` enables IPv6 forwarding on the router.
- `ipv6 address` assigns a global unicast address.
- `ipv6 enable` ensures a link-local address exists if needed.
- The interface can participate in both stacks independently.

Verification commands:

```cisco
show ip interface brief
show ipv6 interface brief
show ipv6 interface GigabitEthernet0/0
show ipv6 neighbors
```

**Routing protocol design in dual-stack networks**

Each routing protocol handles dual-stack differently. The exam often tests whether you understand the protocol model, not just the syntax.

**OSPFv2 and OSPFv3**

OSPFv2 is IPv4-only. OSPFv3 was originally IPv6-only, and on some modern Cisco platforms and releases it can support IPv4 address families, but that support is platform and version dependent. Do not assume IPv4 OSPFv3 behavior is universal.

Design implications:

- Use **OSPFv2** for IPv4 in traditional deployments.
- Use **OSPFv3** for IPv6 where you want a separate IPv6 control plane.
- In some environments, OSPFv3 address-family support can reduce protocol sprawl, but verify platform support before designing around it.

Classic OSPFv3 IPv6 example:

```cisco
conf t
ipv6 unicast-routing

ipv6 router ospf 10
 router-id 1.1.1.1

interface GigabitEthernet0/0
 ipv6 ospf 10 area 0
end
```

Verification:

```cisco
show ipv6 ospf neighbor
show ipv6 route ospf
show ipv6 ospf interface brief
```

If your platform supports OSPFv3 address families, the configuration may look different and can include IPv4 or IPv6 AF-specific activation. Always validate the exact IOS XE release behavior.

**EIGRP for IPv4 and IPv6**

EIGRP can run dual-stack, but the configuration model differs between classic and named mode. In modern IOS XE, named mode is common and uses address families.

Important design points:

- IPv4 and IPv6 adjacencies are separate.
- IPv6 EIGRP uses the **address-family** model.
- In named mode, IPv6 does not use the same `network` statement style as IPv4.
- You must activate the AF on the interface and ensure the process is not administratively shut down.

Example named-mode EIGRP for dual-stack:

```cisco
conf t
router eigrp CORP
 address-family ipv4 unicast autonomous-system 100
  network 10.10.10.0 0.0.0.255
  af-interface default
   passive-interface
  exit-af-interface
  af-interface GigabitEthernet0/0
   no passive-interface
  exit-af-interface
 exit-address-family

 address-family ipv6 unicast autonomous-system 100
  af-interface default
   passive-interface
  exit-af-interface
  af-interface GigabitEthernet0/0
   no passive-interface
  exit-af-interface
  topology base
   exit-af-topology
 exit-address-family
end
```

A more interface-centric IPv6 activation example may be used depending on platform and design:

```cisco
conf t
ipv6 unicast-routing

interface GigabitEthernet0/0
 ipv6 eigrp 100
end
```

Verification commands on modern IOS XE often use AF-specific forms:

```cisco
show eigrp address-family ipv4 neighbors
show eigrp address-family ipv6 neighbors
show eigrp address-family ipv4 topology
show eigrp address-family ipv6 topology
```

> <mark>Important:</mark> In named mode, use the AF-specific verification commands when possible; generic `show eigrp neighbors` output may be incomplete or misleading depending on platform and mode.

**BGP in dual-stack designs**

BGP is common at the edge, in WAN designs, and in large enterprise cores. Dual-stack BGP usually means one BGP process with separate address families for IPv4 and IPv6.

Typical design choices:

- Use **iBGP** inside the enterprise core or between route reflectors and clients.
- Use **eBGP** at the internet edge, partner edge, or between administrative domains.
- Use separate AFs for IPv4 and IPv6 to keep policy clear.

Example dual-stack BGP:

```cisco
conf t
router bgp 65000
 bgp log-neighbor-changes
 neighbor 192.0.2.2 remote-as 65001
 neighbor 2001:db8:ffff::2 remote-as 65001

 address-family ipv4 unicast
  neighbor 192.0.2.2 activate
  network 10.10.10.0 mask 255.255.255.0
 exit-address-family

 address-family ipv6 unicast
  neighbor 2001:db8:ffff::2 activate
  network 2001:db8:10::/48
 exit-address-family
end
```

Important BGP behavior:

- The `network` statement advertises a prefix only if that exact prefix exists in the routing table.
- For IPv6, the route must be present in the IPv6 RIB with the same prefix length.
- If the route is missing, BGP will not originate it.

Verification:

```cisco
show bgp ipv6 unicast summary
show bgp ipv6 unicast
show ipv6 route 2001:db8:10::/48
show bgp ipv6 unicast neighbors
```

**Route selection and policy in dual-stack environments**

IPv4 and IPv6 route selection are independent. A router may prefer one path for IPv4 and a different path for IPv6 because of different metrics, policies, or protocol attributes.

Design considerations:

- Align **IGP metrics** if you want similar forwarding paths for both stacks.
- Apply **route maps** and prefix lists separately for IPv4 and IPv6.
- Remember that IPv6 uses its own ACL syntax and prefix-list objects.
- Verify redistribution carefully because a mistake in one AF does not automatically affect the other.

Example IPv6 prefix list and route map:

```cisco
ipv6 prefix-list PL-V6-CORE seq 10 permit 2001:db8:1000::/48 le 64

route-map RM-V6-REDIST permit 10
 match ipv6 address prefix-list PL-V6-CORE
 set metric 100
```

**Summarization design**

Summarization is valuable in dual-stack networks, but it must be planned carefully. IPv6 summarization is often cleaner than IPv4 because of the large address space and hierarchical allocation, but it still needs discipline.

When summarization helps:

- At **distribution-to-core boundaries**
- Between **sites** or **VRFs**
- At **route reflectors** or aggregation points
- In **WAN hub-and-spoke** designs

When to avoid summarization:

- When it hides important failure domains
- When it causes black holes due to incomplete reachability
- When the summary covers discontiguous or poorly planned allocations

Operational differences from IPv4:

- IPv6 summaries are usually easier to allocate cleanly because you can reserve aligned blocks.
- A summary route can still attract traffic for more-specific destinations that are unreachable if the summary is advertised too broadly.
- In dual-stack environments, summarize both stacks consistently when possible to avoid asymmetric reachability.

Example static summary concept:

```cisco
ipv6 route 2001:db8:1000::/48 Null0
```

This can be used as an aggregate anchor in some designs, but only if the more-specific routes are actually reachable through the intended path and the summary does not create a black hole.

**Dual-stack migration and phasing strategy**

A good migration plan reduces risk and makes troubleshooting easier. The most common enterprise approach is to keep IPv4 stable while introducing IPv6 in controlled phases.

Recommended phases:

1. **Assess readiness**
   - Inventory applications, routing devices, firewalls, and monitoring tools.
   - Confirm IPv6 support in critical infrastructure.
2. **Design addressing and routing**
   - Allocate prefixes by site and function.
   - Decide where OSPF, EIGRP, or BGP will run.
3. **Enable infrastructure**
   - Turn on IPv6 routing on core and distribution devices.
   - Verify management reachability and logging.
4. **Pilot segments**
   - Enable dual-stack on a limited user or server segment.
   - Validate DNS, DHCPv6 if used, and security policy.
5. **Expand gradually**
   - Roll out to additional segments and edges.
   - Monitor route stability, neighbor tables, and application behavior.
6. **Optimize**
   - Refine summarization, policy, and troubleshooting procedures.

Common phasing models:

- **Core-first** is best when the backbone must be ready before endpoints.
- **Edge-first** is useful when external connectivity drives the project.
- **Per-VRF rollout** is useful in large enterprises with segmented routing domains.

**End-to-end example topology**

Consider a small enterprise with a core router, a distribution router, and an edge router. IPv4 uses OSPFv2 internally and BGP externally. IPv6 uses OSPFv3 internally and BGP externally.

Topology:

- Core: `R1`
- Distribution: `R2`
- Edge: `R3`
- Internal IPv4: `10.10.0.0/16`
- Internal IPv6: `2001:db8:1000::/48`
- WAN/ISP IPv4: `192.0.2.0/30`
- WAN/ISP IPv6: `2001:db8:ffff::/64`

Example configuration sketch:

```cisco
! R1 Core
conf t
ipv6 unicast-routing

interface GigabitEthernet0/0
 ip address 10.10.12.1 255.255.255.252
 ipv6 address 2001:db8:1000:12::1/64
 ipv6 ospf 10 area 0
 ip ospf 10 area 0

router ospf 10
 router-id 1.1.1.1
 network 10.10.12.0 0.0.0.3 area 0

ipv6 router ospf 10
 router-id 1.1.1.1
end
```

```cisco
! R2 Distribution
conf t
ipv6 unicast-routing

interface GigabitEthernet0/0
 ip address 10.10.12.2 255.255.255.252
 ipv6 address 2001:db8:1000:12::2/64
 ipv6 ospf 10 area 0
 ip ospf 10 area 0

interface GigabitEthernet0/1
 ip address 10.10.23.1 255.255.255.252
 ipv6 address 2001:db8:1000:23::1/64
 ipv6 ospf 10 area 0
 ip ospf 10 area 0

router ospf 10
 router-id 2.2.2.2
 network 10.10.12.0 0.0.0.3 area 0
 network 10.10.23.0 0.0.0.3 area 0

ipv6 router ospf 10
 router-id 2.2.2.2
end
```

```cisco
! R3 Edge
conf t
ipv6 unicast-routing

interface GigabitEthernet0/0
 ip address 10.10.23.2 255.255.255.252
 ipv6 address 2001:db8:1000:23::2/64
 ipv6 ospf 10 area 0
 ip ospf 10 area 0

interface GigabitEthernet0/1
 ip address 192.0.2.2 255.255.255.252
 ipv6 address 2001:db8:ffff::2/64

router ospf 10
 router-id 3.3.3.3
 network 10.10.23.0 0.0.0.3 area 0

ipv6 router ospf 10
 router-id 3.3.3.3

router bgp 65000
 neighbor 192.0.2.1 remote-as 65001
 neighbor 2001:db8:ffff::1 remote-as 65001

 address-family ipv4 unicast
  neighbor 192.0.2.1 activate
  network 10.10.0.0 mask 255.255.0.0
 exit-address-family

 address-family ipv6 unicast
  neighbor 2001:db8:ffff::1 activate
  network 2001:db8:1000::/48
 exit-address-family
end
```

Verification on the core and distribution:

```cisco
show ip ospf neighbor
show ipv6 ospf neighbor
show ip route ospf
show ipv6 route ospf
show bgp ipv4 unicast summary
show bgp ipv6 unicast summary
```

**Troubleshooting dual-stack routing**

Troubleshooting should be protocol-specific and stack-specific. Do not assume that a working IPv4 path proves IPv6 is healthy.

A practical workflow:

1. Verify interface status and addressing.
2. Verify neighbor discovery or adjacency formation.
3. Verify routing table entries in the correct AF.
4. Verify next-hop reachability.
5. Verify policy, ACLs, and MTU.
6. Test end-to-end with source-aware tools.

Useful commands:

```cisco
show ip interface brief
show ipv6 interface brief
show ipv6 neighbors
show ip route
show ipv6 route
show ip cef
show ipv6 cef
```

Protocol-specific checks:

```cisco
show ip ospf neighbor
show ipv6 ospf neighbor
show eigrp address-family ipv4 neighbors
show eigrp address-family ipv6 neighbors
show bgp ipv4 unicast summary
show bgp ipv6 unicast summary
```

Testing reachability:

```cisco
ping 10.10.10.10 source 10.10.10.1
ping ipv6 2001:db8:1000:30::10 source 2001:db8:1000:10::1
traceroute 10.10.10.10
traceroute ipv6 2001:db8:1000:30::10
```

> The exact IPv6 traceroute syntax can vary by IOS XE platform and release. If one form fails, check the device help output with `?` and use the platform-supported variant.

**IPv6-specific troubleshooting details**

IPv6 introduces a few behaviors that are easy to overlook in exam scenarios and real networks.

Watch for:

- **Link-local next hops** in routing tables and protocol adjacencies.
- **ND cache** issues that look like routing failures.
- **Source address selection** causing unexpected return paths.
- **RA/RS behavior** on access segments if hosts rely on router advertisements.
- **MTU and fragmentation** differences because IPv6 routers do not fragment packets in transit.

Useful commands:

```cisco
show ipv6 neighbors
show ipv6 route
show ipv6 cef
show ipv6 traffic
show ipv6 interface GigabitEthernet0/0
```

**Security and policy considerations**

Dual-stack doubles the number of things you must secure. A common failure is to harden IPv4 while leaving IPv6 open.

Design checklist:

- Apply **ACLs** to both IPv4 and IPv6 interfaces.
- Mirror **control-plane protection** for both stacks.
- Ensure **management access** is available and restricted for both protocols.
- Validate **RA guard**, **DHCPv6 guard**, and **IPv6 ACLs** where appropriate.
- Review firewall and NAT behavior carefully because IPv6 often reduces or removes NAT dependence.

Example IPv6 ACL:

```cisco
ipv6 access-list V6-MGMT
 permit tcp 2001:db8:1000:10::/64 host 2001:db8:1000:1::10 eq 22
 deny ipv6 any any log
```

**Exam-focused takeaways**

- Dual-stack means **both IPv4 and IPv6 are active**, not that one protocol is translated into the other.
- IPv4 and IPv6 have **separate routing, neighbor, and policy behavior**.
- OSPFv2 is IPv4-only; OSPFv3 is IPv6-first, with IPv4 AF support only on some platforms and releases.
- EIGRP named mode uses **address families**; IPv6 verification is often AF-specific.
- BGP `network` statements advertise only prefixes that exist in the routing table.
- IPv6 forwarding depends on **Neighbor Discovery**, link-local next hops, and correct source selection.
- Summarization is powerful, but only when the address plan is hierarchical and the summary will not create black holes.
- Migration is usually safest when you phase deployment by **pilot, core, edge, or VRF**.
