## **Course:** Google Cloud Associate Cloud Engineer (ACE)

**Why cleanup matters**

Test environments in Google Cloud can keep generating charges after you finish a lab or proof of concept. Cleanup is not just about deleting a VM; you also need to remove attached storage, reserved IPs, snapshots, load balancers, and any managed services you created. The goal is to leave no billable resources behind and to confirm that the project is safe to keep or delete.

A good cleanup workflow is:

- Identify all resources created for the test.
- Remove dependencies in the correct order.
- Verify that nothing billable remains.
- Delete the project if the environment is disposable.

**Common resource types to remove**

Different services create different kinds of billable objects. Some resources stop when you delete the main service, but others persist independently.

- **Compute Engine** instances can leave behind persistent disks, static external IP addresses, and snapshots.
- **Cloud Storage** buckets continue to incur storage and request costs until emptied and deleted.
- **Cloud SQL** instances may retain backups, read replicas, and automated storage allocations until removed.
- **GKE** clusters can leave node pools, persistent volumes, load balancers, and IP addresses.
- **Networking** resources such as forwarding rules, firewall rules, routes, and reserved addresses may remain after workloads are gone.

> Deleting a workload does not always delete every related resource. Always check for attached disks, reserved IPs, and service-specific artifacts before assuming cleanup is complete.

**Use the console, CLI, or project deletion**

You can clean up resources through the Google Cloud console, `gcloud`, or by deleting the entire project. The best choice depends on whether you want to preserve anything in the project.

- Use the **console** when you want a visual inventory and quick manual deletion.
- Use `gcloud` when you want repeatable cleanup and scriptable verification.
- Delete the **project** when the environment is temporary and nothing in it needs to be preserved.

A project deletion is the most complete cleanup method, but it is also the most destructive. It removes all resources in the project and can affect linked billing and IAM (Identity and Access Management) relationships.

**Inspect resources before deleting**

Before you remove anything, list what exists so you can avoid orphaned resources. This is especially important when multiple services were used together.

```bash
gcloud compute instances list
gcloud compute disks list
gcloud compute addresses list
gcloud compute snapshots list
gcloud compute firewall-rules list
gcloud storage buckets list
gcloud sql instances list
gcloud container clusters list
```

These commands help you find resources that are easy to miss:

- **Static IP addresses** that remain reserved after VM deletion.
- **Persistent disks** that outlive the instance that used them.
- **Snapshots** that continue to store data and cost money.
- **Buckets** that still contain objects.
- **Managed service instances** that have their own lifecycle.

**Delete resources in dependency order**

Some resources cannot be deleted until dependent objects are removed first. Cleanup is easier when you delete from the outside in.

1. Stop or delete workloads that use the resource.
2. Remove attached or dependent resources.
3. Delete the primary resource.
4. Verify that no related objects remain.

Examples of dependency-aware cleanup:

- Delete a **GKE** workload before deleting the cluster if the cluster is still running services or persistent volumes.
- Remove a **Cloud SQL** read replica before deleting the primary instance.
- Empty a **Cloud Storage** bucket before deleting the bucket itself.
- Release a **static external IP** after deleting the VM or load balancer that used it.

**Compute Engine cleanup examples**

A VM often has several related resources. If you only delete the instance, disks and IPs may remain.

```bash
gcloud compute instances delete my-vm --zone=us-central1-a
gcloud compute disks list
gcloud compute addresses list
```

If the boot disk was set to auto-delete, it is removed with the VM. If not, delete it separately:

```bash
gcloud compute disks delete my-vm-disk --zone=us-central1-a
```

If you reserved a static external IP, release it after the VM is gone:

```bash
gcloud compute addresses delete my-static-ip --region=us-central1
```

Useful checks:

- Confirm whether the disk is **boot** or **persistent**.
- Check whether the IP is **ephemeral** or **static**.
- Verify whether the instance used a custom service account or attached service resources that also need cleanup.

**Cloud Storage cleanup examples**

Buckets must be empty before deletion. If versioning is enabled, older object versions may still exist even after you delete the visible object.

```bash
gsutil ls
gsutil rm gs://my-test-bucket/**
gsutil rb gs://my-test-bucket
```

If object versioning is enabled, you may need to remove all versions:

```bash
gsutil rm -a gs://my-test-bucket/**
```

Practical cleanup checks:

- Confirm that no objects remain in the bucket.
- Check for lifecycle rules that may keep versions or temporary objects.
- Remove any bucket-level IAM bindings only if the bucket itself will remain; otherwise deletion is enough.

**Cloud SQL cleanup examples**

Cloud SQL resources can include backups and replicas. Deleting the instance usually removes the primary database service, but you should still confirm that no dependent resources remain.

```bash
gcloud sql instances list
gcloud sql instances delete my-sql-instance
```

Before deletion, consider:

- Exporting data if you need to preserve it.
- Removing replicas first if the service requires it.
- Checking whether automated backups or retained backups matter for your retention policy.

**GKE cleanup examples**

A Kubernetes cluster can leave behind multiple billable components. Deleting the cluster usually removes the control plane and node pools, but persistent disks and load balancer resources may remain if they were created separately.

```bash
gcloud container clusters list
gcloud container clusters delete my-cluster --zone=us-central1-a
kubectl get all -A
kubectl delete svc my-service
kubectl delete ingress my-ingress
```

Cleanup considerations:

- Delete Kubernetes Services of type `LoadBalancer` so their external IPs can be released.
- Remove persistent volume claims if the underlying storage should be deleted.
- Check for namespaces with remaining workloads before deleting the cluster.

**Networking resources to verify**

Networking objects are easy to overlook because they are not always tied to a single VM.

- **Firewall rules** may remain after the test workload is gone.
- **Forwarding rules** can keep load balancers active.
- **Routes** may still exist if they were created manually.
- **Reserved addresses** continue to be billable in some cases.
- **NAT** or proxy-related resources may persist depending on the architecture.

Useful commands:

```bash
gcloud compute firewall-rules list
gcloud compute forwarding-rules list
gcloud compute routes list
gcloud compute addresses list
```

**Project deletion as final cleanup**

If the project exists only for testing, deleting the project is the cleanest way to ensure all resources are removed. This is often the safest option when you do not need to preserve anything.

Typical workflow:

1. Confirm the project contains only disposable resources.
2. Export any data you need to keep.
3. Remove or note any linked billing or IAM dependencies.
4. Delete the project from the console or CLI.
5. Verify the project no longer appears in your project list.

CLI example:

```bash
gcloud projects delete PROJECT_ID
```

Important implications:

- All resources in the project are deleted.
- APIs, service accounts, and IAM bindings associated with the project are removed.
- Some deletions are asynchronous, so charges may continue briefly until cleanup completes.

> Project deletion is irreversible after the grace period. Use it only when you are certain the project is disposable.

**Verification after cleanup**

Cleanup is not complete until you verify that the environment is empty or intentionally preserved. Verification reduces the chance of surprise billing and helps you catch orphaned resources.

Recommended checks:

- List resources in each service you used.
- Review the Billing page for recent usage if you have access.
- Confirm that static IPs, disks, snapshots, and buckets are gone.
- Check for lingering Kubernetes services, load balancers, and persistent volumes.
- Ensure the project is deleted if that was the intended outcome.

A simple verification pass might look like this:

```bash
gcloud compute instances list
gcloud compute disks list
gcloud compute addresses list
gcloud compute snapshots list
gcloud storage buckets list
gcloud sql instances list
gcloud container clusters list
```

If any command still returns resources, inspect whether they are intentional or leftover test artifacts.

**Practical cleanup checklist**

Use this checklist when finishing a lab or test deployment:

- Delete application workloads first.
- Remove load balancers and forwarding rules.
- Delete VMs, clusters, and managed services.
- Empty and delete storage buckets.
- Delete snapshots, disks, and reserved IPs.
- Verify firewall rules and routes.
- Delete the project if nothing needs to remain.

**Exam-focused takeaways**

For ACE-style tasks, focus on the relationship between a resource and its dependencies. The exam often expects you to know that cleanup is broader than deleting the main service.

- **Compute Engine** cleanup often includes disks and IP addresses.
- **Cloud Storage** buckets must be emptied before deletion.
- **GKE** cleanup may require deleting Services and persistent storage artifacts.
- **Project deletion** is the fastest full cleanup method when the environment is disposable.
- **Verification** is part of the task, not an optional final step.
