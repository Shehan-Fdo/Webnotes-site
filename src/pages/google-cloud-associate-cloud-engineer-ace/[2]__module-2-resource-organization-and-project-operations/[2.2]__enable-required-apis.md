## **Course:** Google Cloud Associate Cloud Engineer (ACE)

**Why API enablement matters**

Google Cloud services are exposed through APIs (Application Programming Interfaces). In practice, you often cannot use a product until its API is enabled in the target project. This is a common first step when you create infrastructure, deploy applications, or automate tasks with `gcloud`, Terraform, or client libraries.

API enablement is project-scoped. That means one project can have an API enabled while another project does not, even if both belong to the same organization or billing account. For exam scenarios, always check the project context before troubleshooting a “service not enabled” error.

- **Project scope** controls whether an API is available in that project.
- **Service Usage API** is the control plane used to list and manage enabled services.
- **Permissions** matter: you need the right IAM roles to enable APIs.
- **Billing** may be required for some services, but billing and API enablement are separate concerns.

**Common reasons you enable APIs**

You enable APIs when a service, tool, or deployment depends on a Google Cloud backend that is not active yet. Typical examples include:

- Creating Compute Engine resources often requires the Compute Engine API.
- Using Cloud Storage from code may require the Cloud Storage JSON API or related service access.
- Deploying serverless workloads may require Cloud Run, Cloud Functions, Artifact Registry, or Cloud Build APIs.
- Managing infrastructure with automation often requires multiple APIs at once.

> If an API is disabled, the service may return `403`, `404`, or a “service usage disabled” style error depending on the product and call path.

**How Google Cloud API enablement works**

When you enable an API, Google Cloud registers that service for the project. After that, the service can accept requests from authenticated users, service accounts, or workloads that have the correct IAM permissions.

A typical flow looks like this:

1. You select the correct project.
2. You enable the required API.
3. You wait for propagation if needed.
4. You verify the API is active.
5. You retry the original operation.

This is especially important in automation, where a deployment can fail early if a prerequisite API is missing.

**Enable APIs from the Google Cloud console**

The console is the most direct way to enable a service when you are exploring a project or troubleshooting manually.

1. Open the Google Cloud console.
2. Select the correct project from the project picker.
3. Go to **APIs & Services** > **Library**.
4. Search for the API you need.
5. Open the API page and click **Enable**.
6. Wait for the status to change to enabled.
7. Retry the operation that failed.

Practical examples:

- Enable the Compute Engine API before creating VM instances.
- Enable the Cloud Run API before deploying a containerized service.
- Enable the Artifact Registry API before pushing container images.

**Enable APIs with `gcloud`**

The `gcloud` CLI is the preferred method for repeatable administration and scripting. It is especially useful in labs, CI/CD pipelines, and exam tasks where you need to act quickly.

```bash
gcloud config set project PROJECT_ID
gcloud services enable compute.googleapis.com
gcloud services enable run.googleapis.com
gcloud services enable artifactregistry.googleapis.com
```

To verify enabled services:

```bash
gcloud services list --enabled
```

To check whether a specific API is enabled:

```bash
gcloud services list --enabled --filter="config.name:compute.googleapis.com"
```

To disable an API when cleanup is required:

```bash
gcloud services disable run.googleapis.com
```

Notes:

- Use the service name, not the human-readable product name.
- The command affects only the currently selected project unless you specify another project with `--project`.
- Some services may take a short time to become usable after enablement.

**Enable multiple APIs efficiently**

Many workloads need more than one API. For example, a container deployment might require:

- `run.googleapis.com` for Cloud Run
- `artifactregistry.googleapis.com` for image storage
- `cloudbuild.googleapis.com` for builds
- `iam.googleapis.com` for identity and policy operations

You can enable several services in one command:

```bash
gcloud services enable run.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com iam.googleapis.com
```

This is useful when preparing a new project or building a reusable bootstrap script.

**Common APIs and when you need them**

| Service | API name | Typical use |
|---|---|---|
| Compute Engine | `compute.googleapis.com` | Virtual machines, disks, networks |
| Cloud Run | `run.googleapis.com` | Containerized serverless services |
| Cloud Functions | `cloudfunctions.googleapis.com` | Event-driven functions |
| Artifact Registry | `artifactregistry.googleapis.com` | Container and package repositories |
| Cloud Build | `cloudbuild.googleapis.com` | Builds and CI/CD pipelines |
| Cloud Storage | `storage.googleapis.com` | Object storage access |
| IAM | `iam.googleapis.com` | Identity and policy management |
| Service Usage | `serviceusage.googleapis.com` | API enablement and service listing |
| Secret Manager | `secretmanager.googleapis.com` | Secret storage and retrieval |
| Pub/Sub | `pubsub.googleapis.com` | Messaging and event delivery |

Use the exact API name shown by Google Cloud documentation or the console. Product branding and API service names do not always match one-to-one.

**Permissions required to enable APIs**

You need IAM permissions to enable services. In many environments, the following roles are relevant:

- **Service Usage Admin** can manage service enablement.
- **Project Editor** may have broad permissions, but least privilege is preferred.
- **Project Owner** can usually enable APIs, but this role is broader than necessary.

The specific permission commonly associated with enabling services is `serviceusage.services.enable`.

Example role grant:

```bash
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="user:you@example.com" \
  --role="roles/serviceusage.serviceUsageAdmin"
```

If you cannot enable an API, check:

- Whether you are in the correct project.
- Whether your account has the required IAM role.
- Whether an organization policy restricts service usage.
- Whether the API is already enabled but propagation is still in progress.

**Troubleshooting API-related failures**

API enablement issues often look like service failures, but the root cause is usually missing prerequisites.

Common symptoms:

- `API has not been used in project before or it is disabled`
- `403 Permission denied`
- `404 Not Found` when the backend service is not active
- Deployment tools failing during initialization or validation

Troubleshooting approach:

1. Confirm the active project with `gcloud config get-value project`.
2. Check whether the API is enabled with `gcloud services list --enabled`.
3. Enable the missing API.
4. Verify your IAM permissions.
5. Retry after a short wait if the service was just enabled.

Example verification commands:

```bash
gcloud config get-value project
gcloud services list --enabled
gcloud services list --available --filter="name:run.googleapis.com"
```

**Automation and infrastructure as code**

In real environments, API enablement is often part of bootstrap automation. This prevents deployments from failing because a dependency was forgotten.

Terraform example:

```hcl
resource "google_project_service" "run" {
  project = var.project_id
  service = "run.googleapis.com"

  disable_on_destroy = false
}
```

Terraform notes:

- Use `google_project_service` to manage a single API.
- Use `google_project_service` resources or a module pattern to enable a set of APIs.
- Set `disable_on_destroy = false` carefully if you do not want Terraform to turn off shared services during teardown.

A shell bootstrap example:

```bash
#!/usr/bin/env bash
set -euo pipefail

PROJECT_ID="$1"

gcloud config set project "$PROJECT_ID"
gcloud services enable \
  serviceusage.googleapis.com \
  iam.googleapis.com \
  compute.googleapis.com \
  artifactregistry.googleapis.com \
  run.googleapis.com
```

This pattern is useful for repeatable environment setup in development, test, and lab projects.

**Best practices**

- Enable only the APIs you need.
- Use the correct project before making changes.
- Prefer automation for repeatable environments.
- Verify enablement before deploying dependent resources.
- Keep least privilege in mind when assigning IAM roles.
- Treat API enablement as a prerequisite, not a troubleshooting afterthought.

**Exam-focused reminders**

For ACE scenarios, remember these practical points:

- API enablement is usually a project-level prerequisite.
- The console path is **APIs & Services** > **Library**.
- `gcloud services enable` is the standard CLI command.
- Missing APIs often cause deployment failures that look like permission or connectivity issues.
- The Service Usage API is central to managing enabled services.

**Quick reference**

```bash
# Set the active project
gcloud config set project PROJECT_ID

# Enable a single API
gcloud services enable compute.googleapis.com

# Enable several APIs
gcloud services enable run.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com

# List enabled APIs
gcloud services list --enabled

# Check a specific API
gcloud services list --enabled --filter="config.name:run.googleapis.com"
```
