## **Course:** Google Cloud Associate Cloud Engineer (ACE)

**Network tags: what they are and why they matter**

Network tags are simple labels you attach to a Compute Engine VM instance so that Google Cloud firewall rules can target that VM. They are not the same as labels: labels help with organization and billing, while tags are used for network policy selection.

In practice, you use tags to say, “this VM should receive traffic allowed by this firewall rule.” That makes tags useful when you want to apply the same network access pattern to a group of instances without listing each instance name.

- **Targeting**: firewall rules can match instances by tag.
- **Scope**: tags apply at the VM instance level.
- **Use case**: allow SSH only to bastion hosts, or allow HTTP only to web servers.

**How network tags work with firewall rules**

A firewall rule can use a target tag so that only instances with that tag are affected. For example, a rule with target tag `web` can allow TCP port 80 only to VMs tagged `web`.

This is especially useful in layered architectures:

- `web` tag for frontend instances
- `app` tag for application instances
- `db` tag for database instances

You can then write firewall rules that allow traffic between tiers while keeping the rest blocked.

```bash
gcloud compute firewall-rules create allow-web-http \
  --network=default \
  --allow=tcp:80 \
  --target-tags=web \
  --source-ranges=0.0.0.0/0
```

In this example, only VMs with the `web` tag receive the rule. If a VM does not have the tag, the rule does not apply.

**Assigning a network tag to a VM**

You can assign tags when you create the VM or update an existing VM later. Tags are part of the instance metadata and are managed on the VM resource.

1. Create a VM with tags:
   ```bash
   gcloud compute instances create web-vm \
     --zone=us-central1-a \
     --tags=web,http-server
   ```

2. Add or replace tags on an existing VM:
   ```bash
   gcloud compute instances add-tags web-vm \
     --zone=us-central1-a \
     --tags=web,http-server
   ```

3. Remove tags from an existing VM:
   ```bash
   gcloud compute instances remove-tags web-vm \
     --zone=us-central1-a \
     --tags=http-server
   ```

When you update tags, you are changing the instance’s network identity for firewall matching. The VM does not need a new IP address for the tag change to take effect.

**Console workflow**

In the Google Cloud Console, you typically assign tags from the VM details page or during instance creation.

1. Go to **Compute Engine** > **VM instances**.
2. Select the VM or click **Create instance**.
3. Open the **Networking** or **Management, security, disks, networking** section, depending on the UI layout.
4. Find **Network tags**.
5. Enter one or more tags, separated by commas.
6. Save or create the instance.

If the VM already exists, you can edit the instance and update the tags. The console writes the same underlying instance metadata that `gcloud` changes.

**Firewall rule targeting patterns**

Tags are most effective when paired with clear firewall design. A common pattern is to use one rule per service tier.

| Tier | Tag | Example allowed traffic |
|---|---|---|
| Web | `web` | TCP 80, TCP 443 from the internet |
| App | `app` | TCP 8080 from `web` tier |
| DB | `db` | TCP 5432 or 3306 from `app` tier |

Example firewall rule for internal app access:

```bash
gcloud compute firewall-rules create allow-app-from-web \
  --network=default \
  --allow=tcp:8080 \
  --source-tags=web \
  --target-tags=app
```

This rule allows traffic from instances tagged `web` to instances tagged `app`. Use this pattern when you want instance-to-instance filtering based on workload role.

**Important differences: tags, labels, and service accounts**

These concepts are often confused on the exam.

- **Network tags**: used by firewall rules to select VM instances.
- **Labels**: used for organization, filtering, and billing reports.
- **Service accounts**: used for identity and API authorization.

A VM can have all three at once, but each serves a different purpose. If you need firewall targeting, use tags. If you need cost allocation or inventory grouping, use labels.

> Tags do not grant access by themselves. They only make a VM match a firewall rule that already allows or denies traffic.

**Practical examples**

A common bastion host setup uses a tag like `bastion`:

```bash
gcloud compute instances create bastion-vm \
  --zone=us-central1-a \
  --tags=bastion
```

Then you allow SSH only to that tag:

```bash
gcloud compute firewall-rules create allow-ssh-bastion \
  --network=default \
  --allow=tcp:22 \
  --source-ranges=203.0.113.0/24 \
  --target-tags=bastion
```

For a web server, you might use:

```bash
gcloud compute instances create web-vm \
  --zone=us-central1-a \
  --tags=web
```

And allow public HTTP/HTTPS:

```bash
gcloud compute firewall-rules create allow-web \
  --network=default \
  --allow=tcp:80,tcp:443 \
  --source-ranges=0.0.0.0/0 \
  --target-tags=web
```

**Verification and troubleshooting**

After assigning tags, verify both the instance and the firewall rule.

```bash
gcloud compute instances describe web-vm \
  --zone=us-central1-a \
  --format="get(tags.items)"
```

You can also inspect the full instance configuration:

```bash
gcloud compute instances describe web-vm --zone=us-central1-a
```

If traffic is not working, check these common issues:

- The VM does not have the expected tag.
- The firewall rule uses the wrong `--target-tags` value.
- The source range is too restrictive.
- Another deny rule or higher-priority rule blocks the traffic.
- You updated the wrong zone or instance name.

For deeper validation, test connectivity from another VM or use packet-level troubleshooting tools available in Compute Engine environments.

**Exam-focused takeaways**

- Assign tags to VMs when you want firewall rules to target workloads by role.
- Use `gcloud compute instances create --tags=...` for new VMs.
- Use `gcloud compute instances add-tags` and `remove-tags` for existing VMs.
- Remember that tags are for network policy, not organization or IAM.
- Pair tags with clear firewall naming and tier-based design.

**Quick reference**

```bash
# Create VM with tags
gcloud compute instances create vm-1 --zone=us-central1-a --tags=web,https-server

# Add tags to existing VM
gcloud compute instances add-tags vm-1 --zone=us-central1-a --tags=web

# Remove a tag
gcloud compute instances remove-tags vm-1 --zone=us-central1-a --tags=https-server

# Show tags on a VM
gcloud compute instances describe vm-1 --zone=us-central1-a --format="get(tags.items)"
```
