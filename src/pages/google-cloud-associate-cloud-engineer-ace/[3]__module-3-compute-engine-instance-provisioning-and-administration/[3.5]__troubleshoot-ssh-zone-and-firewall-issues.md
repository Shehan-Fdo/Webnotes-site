## **Course:** Google Cloud Associate Cloud Engineer (ACE)

**SSH troubleshooting in Google Cloud**

SSH (Secure Shell) access problems usually come from one of four areas: the VM is in the wrong zone, the network path is blocked, the firewall does not allow TCP 22, or the instance itself is not reachable through the expected access method. In Google Cloud, you often troubleshoot by verifying the instance location, checking VPC (Virtual Private Cloud) firewall rules, and confirming that the VM has a valid external or internal path for the type of SSH you are using.

A practical workflow is to start with the instance metadata and network settings, then move outward to the VPC and routing layer. This avoids guessing and helps you isolate whether the issue is with the VM, the subnet, or the perimeter controls.

**Common SSH failure categories**

- **Wrong zone**: You are targeting an instance name that exists in a different zone, or you are using the wrong zone in `gcloud compute ssh`.
- **Firewall denied**: No ingress rule allows TCP port `22` from your source IP or from the IAP (Identity-Aware Proxy) range.
- **No external IP**: The VM has no public address, so direct internet SSH cannot work.
- **OS-level access issue**: The network is open, but the guest OS blocks SSH or the SSH daemon is not running.
- **Route or NAT gap**: Internal-only access depends on correct routing, Cloud NAT, or a bastion host.

**Verify the instance zone first**

Google Cloud resources are regional or zonal, and Compute Engine VMs are zonal. If you use the wrong zone, `gcloud` may fail to find the instance or may target a different VM with the same name in another zone.

Useful checks:

```bash
gcloud compute instances list --filter="name=('my-vm')"
gcloud compute instances describe my-vm --zone=us-central1-a
gcloud config get-value compute/zone
```

If you are unsure of the zone, list instances across zones and confirm the exact location before attempting SSH.

**Typical zone-related symptoms**

- `gcloud compute ssh` reports that the resource was not found.
- The VM name exists, but the command points to the wrong zone.
- You can see the instance in the console, but your CLI context uses a different default zone.
- A startup script or automation created the VM in a different zone than expected.

**Correct the zone in your CLI context**

If you work repeatedly in one zone, set the default:

```bash
gcloud config set compute/zone us-central1-a
```

You can also override the zone per command:

```bash
gcloud compute ssh my-vm --zone=us-central1-a
```

This is especially useful when you manage multiple environments or when a lab environment creates resources in a nondefault zone.

**Check whether the VM is reachable by the SSH method you intend**

There are three common SSH paths in Google Cloud:

- **Direct external SSH**: The VM has an external IP and your source IP is allowed by firewall rules.
- **IAP SSH**: You connect through Identity-Aware Proxy without exposing port 22 to the internet.
- **Internal SSH**: You connect from another VM or a bastion host inside the VPC.

Each path has different prerequisites. If one path fails, another may still work.

**Firewall rules and TCP 22**

A VPC firewall rule must allow ingress on TCP port `22` for SSH to work. Google Cloud firewall rules are stateful, so return traffic is allowed automatically once the ingress rule permits the connection.

A basic allow rule looks like this:

```bash
gcloud compute firewall-rules create allow-ssh \
  --network=default \
  --allow=tcp:22 \
  --source-ranges=203.0.113.0/24 \
  --target-tags=ssh
```

This example allows SSH only from a specific source range and only to instances with the `ssh` network tag. That is safer than opening SSH to all sources.

**What to verify in firewall rules**

- **Direction**: Ingress rules control incoming SSH traffic.
- **Protocol and port**: You need `tcp:22`.
- **Source range**: Your public IP, office CIDR, VPN range, or IAP range must be allowed.
- **Targeting**: The rule must apply to the VM, either by network tag or service account.
- **Priority**: A lower-numbered deny rule can override an allow rule.

**Common firewall troubleshooting commands**

```bash
gcloud compute firewall-rules list --filter="allowed.tcp:22"
gcloud compute firewall-rules describe allow-ssh
gcloud compute instances describe my-vm --zone=us-central1-a --format="get(tags.items)"
```

If the instance uses network tags, confirm that the tag on the VM matches the firewall rule target tag. If the rule targets a service account, confirm the VM is using that service account.

**Understand source IP matching**

When you SSH from your laptop, your public IP may differ from your private LAN IP. The firewall must allow the public source IP as seen by Google Cloud. If your IP changes frequently, a narrow source range may break access.

For example:

- Home network: your ISP-assigned public IP may change.
- Corporate network: traffic may exit through a shared NAT gateway.
- VPN: your source IP may become the VPN egress address.

If you are unsure of your current public IP, check it from your workstation and compare it with the firewall source range.

**Use IAP when the VM has no external IP**

IAP TCP forwarding is a common solution for SSH to private instances. It lets you connect without assigning an external IP to the VM.

A typical IAP SSH command is:

```bash
gcloud compute ssh my-vm \
  --zone=us-central1-a \
  --tunnel-through-iap
```

For IAP to work, you generally need:

- The VM to be reachable on port `22` from the IAP source range.
- The correct IAM (Identity and Access Management) permissions for the user.
- The IAP TCP forwarding firewall rule in the VPC.

A common firewall rule for IAP SSH is:

```bash
gcloud compute firewall-rules create allow-iap-ssh \
  --network=default \
  --allow=tcp:22 \
  --source-ranges=35.235.240.0/20 \
  --target-tags=ssh
```

The source range `35.235.240.0/20` is the well-known IAP TCP forwarding range used for SSH access through IAP.

**When to prefer IAP**

- The VM has no external IP.
- You want to avoid exposing SSH to the public internet.
- You need centralized access control through IAM.
- You are connecting to production systems with tighter security requirements.

**OS-level SSH checks**

If the network path is open but SSH still fails, the guest OS may be the problem. Common causes include:

- The SSH daemon is not running.
- The OS firewall blocks port `22`.
- The user account does not have a valid SSH key.
- The instance metadata or OS Login configuration prevents login.

Useful guest-side checks on Linux:

```bash
sudo systemctl status ssh
sudo systemctl status sshd
sudo ss -tlnp | grep :22
sudo ufw status
sudo firewall-cmd --list-all
```

Different Linux distributions use different service names. Debian and Ubuntu often use `ssh`, while RHEL-based systems often use `sshd`.

**Metadata and OS Login considerations**

Google Cloud can manage SSH keys through instance metadata, project metadata, or OS Login. If OS Login is enabled, manual metadata keys may not behave as expected.

Things to verify:

- **OS Login enabled**: Check whether the project or instance metadata has `enable-oslogin=TRUE`.
- **IAM permissions**: The user needs the appropriate OS Login role to log in.
- **SSH keys**: If OS Login is disabled, confirm that the user’s public key is present in metadata or that the console-generated key was accepted.

Example metadata check:

```bash
gcloud compute instances describe my-vm \
  --zone=us-central1-a \
  --format="get(metadata.items)"
```

If you use OS Login, the login flow depends on IAM rather than static SSH keys. That is often more secure and easier to manage at scale.

**Routes, NAT, and internal access**

If you are SSHing from one VM to another inside the same VPC, the issue may not be firewall-only. Internal SSH requires:

- A valid internal IP address.
- A route between the source and destination subnets.
- A firewall rule allowing TCP `22` from the source subnet or source tag.
- If the destination needs outbound internet for updates or package installs, Cloud NAT or another egress path.

Example internal SSH from a bastion host:

```bash
ssh myuser@10.128.0.5
```

If the destination VM has no external IP, a bastion host is a common pattern. You SSH to the bastion first, then to the private VM.

**Troubleshooting sequence you can follow**

1. Confirm the instance exists in the expected zone.
2. Confirm the VM is running and has the expected network interface.
3. Check whether the VM has an external IP or requires IAP/internal access.
4. Inspect firewall rules for TCP `22`.
5. Verify the rule targets the VM correctly through tags or service account.
6. Confirm your source IP matches the allowed source range.
7. Check guest OS SSH service, OS firewall, and login configuration.
8. If using IAP, verify IAM permissions and the IAP firewall rule.

This sequence works well because it moves from the most common and easiest-to-verify causes to the more specific guest-level causes.

**Practical command set for diagnosis**

```bash
# List the instance and confirm zone
gcloud compute instances list --filter="name=('my-vm')"

# Describe the instance network details
gcloud compute instances describe my-vm --zone=us-central1-a \
  --format="yaml(name,zone,status,networkInterfaces,tags,serviceAccounts,metadata)"

# Check firewall rules that mention SSH
gcloud compute firewall-rules list --filter="allowed.tcp:22"

# Try SSH with an explicit zone
gcloud compute ssh my-vm --zone=us-central1-a

# Try IAP if there is no external IP
gcloud compute ssh my-vm --zone=us-central1-a --tunnel-through-iap
```

**How to read the results**

- If the instance is missing from the zone, you are targeting the wrong zone or project.
- If SSH times out, the firewall or route is usually the first suspect.
- If SSH is refused immediately, the guest OS may not be listening on port `22`.
- If IAP fails but direct SSH works, the IAP IAM or firewall path is likely misconfigured.
- If direct SSH fails but IAP works, the public firewall or external IP path is the issue.

**Example firewall and tag alignment**

A common pattern is to tag the VM and target the firewall rule to that tag:

```bash
gcloud compute instances add-tags my-vm \
  --zone=us-central1-a \
  --tags=ssh
```

Then create the allow rule:

```bash
gcloud compute firewall-rules create allow-ssh-from-office \
  --network=default \
  --allow=tcp:22 \
  --source-ranges=198.51.100.0/24 \
  --target-tags=ssh
```

This keeps SSH exposure limited to the intended instances instead of every VM in the network.

**Security best practices**

- **Prefer IAP** for private instances instead of opening SSH to the internet.
- **Restrict source ranges** to known IPs or trusted networks.
- **Use tags or service accounts** to scope firewall rules narrowly.
- **Avoid `0.0.0.0/0`** for SSH unless it is a temporary lab setup.
- **Use OS Login** where possible to centralize access control.
- **Remove stale rules** after troubleshooting to reduce attack surface.

> If you open TCP `22` broadly during troubleshooting, close it again after validation. Temporary access often becomes permanent by accident.

**Quick comparison of likely root causes**

| Symptom | Likely cause | What to check first |
|---|---|---|
| Instance not found | Wrong zone or project | `gcloud compute instances list` |
| SSH times out | Firewall or route issue | Ingress rule for `tcp:22` |
| Connection refused | SSH service not running | `systemctl status sshd` |
| Works with IAP only | No external IP or public firewall blocked | External IP and ingress rule |
| Works from one network but not another | Source range mismatch | Public egress IP and firewall source range |

**Key exam takeaways**

- Compute Engine VMs are **zonal**, so zone mismatches are a common SSH troubleshooting mistake.
- SSH requires both a reachable network path and an allow rule for **TCP 22**.
- Firewall rules must match the VM by **tag** or **service account** and must allow the correct **source range**.
- **IAP** is the preferred method for SSH to private VMs.
- If the network is correct, check the guest OS, SSH daemon, and login configuration.
