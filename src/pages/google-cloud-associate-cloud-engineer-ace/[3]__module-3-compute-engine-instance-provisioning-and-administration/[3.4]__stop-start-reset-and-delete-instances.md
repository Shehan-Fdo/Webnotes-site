## **Course:** Google Cloud Associate Cloud Engineer (ACE)

**Overview**

Google Compute Engine instances have distinct lifecycle actions that affect power state, billing, attached storage, and data persistence. For the ACE exam, you need to know when to use `stop`, `start`, `reset`, and `delete`, and what each action does to the virtual machine (VM), persistent disks, and ephemeral resources.

A good mental model is:

- **Stop** powers off the VM but keeps the instance configuration and persistent disks.
- **Start** powers on a stopped VM and reattaches its disks and network settings.
- **Reset** reboots the VM without changing its configuration or attached disks.
- **Delete** removes the VM resource and can optionally delete attached disks.

**Instance lifecycle basics**

Compute Engine instances are virtual machines that run on Google infrastructure. Their state changes affect both availability and cost:

- A running VM consumes compute charges.
- A stopped VM does not consume vCPU and memory charges, but attached persistent disks and reserved IPs may still incur charges.
- A deleted VM no longer exists as a compute resource.
- Ephemeral data on local SSD or temporary files on the boot disk is not guaranteed to survive lifecycle changes.

Use lifecycle actions when you need maintenance, troubleshooting, cost control, or cleanup. Choose the least destructive action that meets the goal.

**Stop an instance**

Stopping an instance shuts down the guest OS and releases the VM’s compute resources. The instance remains defined in Compute Engine, so you can start it later with the same configuration.

Typical reasons to stop a VM:

- Reduce cost when the VM is not needed.
- Perform maintenance on the guest OS.
- Preserve the instance definition while pausing work.
- Prepare for changes that require the VM to be offline.

When you stop a VM:

- The instance enters a stopped state.
- Persistent disks remain attached to the instance definition.
- Static external IP addresses can remain reserved, depending on how they were assigned.
- Local SSD data is lost because local SSD is ephemeral.
- The VM can be started again later.

CLI example:

```bash
gcloud compute instances stop INSTANCE_NAME \
  --zone=ZONE
```

If you want to stop multiple instances, you can script the command or use instance groups where appropriate, but the exam focus is usually on the effect of the action rather than bulk automation.

> Stopping a VM is not the same as deleting it. Stop preserves the instance resource; delete removes it.

**Start an instance**

Starting an instance powers on a previously stopped VM. Compute Engine recreates the running VM from the saved instance configuration and reattaches persistent disks.

Use start when:

- You need the same VM back online after a planned shutdown.
- You want to resume a development or test environment.
- You are restoring service after a temporary pause.

When you start a VM:

- The instance returns to the running state.
- Persistent disks are reattached.
- The guest OS boots again.
- Any startup scripts configured on the instance may run during boot.
- The external IP may change if it was ephemeral and not reserved.

CLI example:

```bash
gcloud compute instances start INSTANCE_NAME \
  --zone=ZONE
```

If the VM uses an ephemeral external IP and you stop it, the address may be released. If you need a stable address across stop/start cycles, use a reserved static external IP.

**Reset an instance**

Reset restarts the VM immediately without going through a graceful shutdown. It is similar to pressing a physical reset button on a server.

Use reset when:

- The VM is unresponsive but still exists.
- You need a fast reboot after a crash or hang.
- You want to restart the guest OS without deleting the instance.

Important characteristics of reset:

- The instance remains the same resource.
- Attached persistent disks remain attached.
- The VM does not go through a clean shutdown sequence.
- Unsaved in-memory state is lost.
- Applications may not have time to flush buffers or close files cleanly.

CLI example:

```bash
gcloud compute instances reset INSTANCE_NAME \
  --zone=ZONE
```

Because reset is abrupt, prefer stop/start when you can safely shut down the guest OS. Use reset only when the VM is stuck or you need immediate recovery.

**Delete an instance**

Deleting an instance removes the VM resource from Compute Engine. This is the most destructive lifecycle action in this topic.

Use delete when:

- The VM is no longer needed.
- You are cleaning up test or temporary environments.
- You want to remove a broken instance and recreate it from a template or image.
- You are decommissioning a workload.

When you delete a VM:

- The instance resource is removed.
- The VM no longer appears in the project as an instance.
- Persistent disks may be deleted or preserved depending on deletion settings.
- Ephemeral data is lost.
- Any attached local SSD is lost.

CLI example:

```bash
gcloud compute instances delete INSTANCE_NAME \
  --zone=ZONE
```

If you want to keep attached persistent disks, ensure the boot disk and any additional disks are configured not to auto-delete, or detach them before deleting the instance. This is a common exam concept because disk deletion behavior is separate from instance deletion.

**Disk deletion behavior**

Instance deletion does not always mean disk deletion. Persistent disks are durable storage, but they can still be removed if configured to auto-delete with the VM.

Key points:

- **Boot disk** can be set to auto-delete when the instance is deleted.
- **Additional persistent disks** can also be set to auto-delete or preserved.
- **Local SSD** is always ephemeral and is lost when the VM stops or is deleted.
- Detached persistent disks can be reused by other VMs in the same zone, subject to compatibility.

Example of checking disk settings:

```bash
gcloud compute instances describe INSTANCE_NAME \
  --zone=ZONE \
  --format="yaml(disks)"
```

Look for fields related to `autoDelete` and disk source information. If you are preserving data, verify the disk lifecycle before deleting the VM.

**Common command patterns**

You often manage instances with `gcloud compute instances` commands:

```bash
gcloud compute instances stop INSTANCE_NAME --zone=ZONE
gcloud compute instances start INSTANCE_NAME --zone=ZONE
gcloud compute instances reset INSTANCE_NAME --zone=ZONE
gcloud compute instances delete INSTANCE_NAME --zone=ZONE
```

Useful inspection commands:

```bash
gcloud compute instances list
gcloud compute instances describe INSTANCE_NAME --zone=ZONE
gcloud compute disks list
gcloud compute addresses list
```

These commands help you confirm the current state, attached disks, and whether an external IP is static or ephemeral.

**Practical decision guide**

Use the least destructive action that solves the problem:

1. If the VM is healthy but temporarily not needed, **stop** it.
2. If the VM is stopped and you need it back, **start** it.
3. If the VM is frozen or unresponsive, **reset** it.
4. If the VM is obsolete, **delete** it.

This sequence matters because it reduces the risk of data loss and unnecessary recreation work.

**Real-world examples**

A development VM used only during business hours can be stopped at night and started in the morning to save cost.

```bash
gcloud compute instances stop dev-vm --zone=us-central1-a
gcloud compute instances start dev-vm --zone=us-central1-a
```

A production VM that stops responding may need a reset if SSH is unavailable and the guest OS is hung.

```bash
gcloud compute instances reset prod-vm --zone=us-central1-a
```

A temporary test VM can be deleted after validation, but you should first confirm whether any disks need to be preserved.

```bash
gcloud compute instances delete test-vm --zone=us-central1-a
```

**Operational cautions**

> Do not assume that stopping a VM preserves all data. Data on local SSD and other ephemeral storage is not durable.

> Do not delete an instance until you confirm whether its boot disk or attached data disks are set to auto-delete.

> Reset is disruptive because it does not provide a graceful shutdown path for applications.

**Exam focus**

For ACE, expect questions that ask you to choose the correct lifecycle action based on a scenario:

- Need to pause compute charges but keep the VM definition: **stop**
- Need to bring back a stopped VM: **start**
- Need to recover a hung VM quickly: **reset**
- Need to remove the VM entirely: **delete**
- Need to preserve data before deletion: verify disk auto-delete settings or detach disks first

**Quick comparison**

| Action | VM resource kept? | Guest OS shutdown? | Persistent disks kept? | Local SSD kept? | Typical use |
|---|---:|---:|---:|---:|---|
| Stop | Yes | Yes, graceful | Yes | No | Pause and save cost |
| Start | Yes | N/A | Yes | N/A | Resume a stopped VM |
| Reset | Yes | No | Yes | No change to attachment, but in-memory state is lost | Recover a hung VM |
| Delete | No | Depends on deletion path | Maybe | No | Remove the VM permanently |

**Summary**

The key distinction is whether you are changing power state or removing the resource. Stop and start preserve the instance; reset reboots it abruptly; delete removes it. Always check disk behavior before deleting, especially when persistent data matters.
