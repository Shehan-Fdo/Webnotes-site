## **Course:** Google Cloud Associate Cloud Engineer (ACE)

**Overview**

Verifying a simple service on a virtual machine (VM) means confirming that the process is running, listening on the expected port, reachable from the right network path, and producing the expected response. In Google Cloud, this usually involves connecting to the VM with SSH, checking the service manager, validating firewall rules, and testing the endpoint from inside and outside the instance.

For the ACE exam, focus on practical verification steps rather than deep application debugging. You should know how to confirm that a service starts correctly, how to inspect logs when it does not, and how to distinguish between a local-only problem and a network exposure problem.

**What “simple service” usually means**

A simple service is often a lightweight web server or test daemon such as:

- `nginx`
- `apache2`
- `python3 -m http.server`
- a custom app listening on a TCP port like `8080`

The goal is to verify basic availability, not to perform full application health analysis.

**Typical verification flow**

1. Connect to the VM with SSH.
2. Confirm the process or service is running.
3. Confirm the service is listening on the expected port.
4. Test the service locally on the VM.
5. Test access from another host if external access is expected.
6. Check logs if the service fails or returns errors.

**Connect to the VM**

Use SSH from Cloud Shell, your terminal, or the Google Cloud console.

```bash
gcloud compute ssh VM_NAME --zone=ZONE
```

If you are using the console, the SSH button opens a browser-based session. Once connected, you can verify the service from the VM itself.

**Check whether the service is running**

If the service is managed by `systemd`, use `systemctl`:

```bash
sudo systemctl status nginx
sudo systemctl status apache2
sudo systemctl status my-app.service
```

Useful status indicators:

- `active (running)` means the service is up.
- `inactive` means it is stopped.
- `failed` means startup or runtime errors occurred.
- `enabled` means it starts at boot.
- `disabled` means it does not start automatically.

If the service is not managed by `systemd`, inspect the process directly:

```bash
ps aux | grep http.server
pgrep -a python3
```

**Start, stop, and restart for verification**

You often restart a service after changing configuration or deploying a new version.

```bash
sudo systemctl restart nginx
sudo systemctl restart apache2
sudo systemctl restart my-app.service
```

If you changed configuration, validate it before restarting when possible.

```bash
sudo nginx -t
sudo apachectl configtest
```

A configuration test helps you catch syntax errors before the service fails to start.

**Confirm the service is listening on the expected port**

A running process is not enough; it must also bind to the correct port and interface.

```bash
sudo ss -tulpn
sudo ss -tulpn | grep :80
sudo ss -tulpn | grep :8080
```

You can also use `netstat` if installed, though `ss` is more common on modern Linux systems.

What to look for:

- The correct port number, such as `80`, `443`, or `8080`
- The correct protocol, such as `tcp`
- The correct bind address:
  - `0.0.0.0` means all IPv4 interfaces
  - `127.0.0.1` means localhost only
  - a specific VM IP means only that interface

> If a service binds only to `127.0.0.1`, it works locally but is not reachable from other machines.

**Test locally on the VM**

Local testing confirms the application responds before you troubleshoot networking.

```bash
curl http://localhost
curl http://127.0.0.1:8080
curl -I http://localhost
```

Common checks:

- `curl` returns HTML, JSON, or another expected payload
- `curl -I` returns headers and status code
- `curl -v` shows connection details when debugging

Example with a simple Python HTTP server:

```bash
python3 -m http.server 8080
```

Then verify it locally:

```bash
curl http://localhost:8080
```

**Test from another host**

If the service should be reachable externally, test from a different machine or from Cloud Shell.

```bash
curl http://EXTERNAL_IP
curl http://EXTERNAL_IP:8080
```

If the service is only intended for internal access, test from another VM in the same VPC network or from a bastion host.

**Understand the difference between local and external reachability**

A service can be healthy on the VM but still unreachable from outside because of network controls.

Common causes:

- The service listens only on localhost.
- The Google Cloud firewall blocks the port.
- The VM OS firewall blocks the port.
- The wrong external IP or port is used.
- The service is listening on a different port than expected.

A useful mental model:

- **Process layer**: Is the app running?
- **Socket layer**: Is it listening on the right port?
- **Host layer**: Is the OS allowing traffic?
- **Network layer**: Is Google Cloud allowing traffic?
- **Client layer**: Are you connecting to the right address and protocol?

**Check logs when the service fails**

Logs often explain why a service does not start or why it exits immediately.

For `systemd` services:

```bash
sudo journalctl -u nginx
sudo journalctl -u apache2
sudo journalctl -u my-app.service
sudo journalctl -u my-app.service -f
```

For application-specific logs, check common locations such as:

```bash
/var/log/nginx/
/var/log/apache2/
/var/log/syslog
```

Useful patterns to search for:

- permission denied
- address already in use
- configuration syntax error
- port conflict
- missing file or directory
- failed dependency

**Verify firewall rules in Google Cloud**

If the service works locally but not externally, inspect VPC firewall rules. The VM may be healthy, but ingress traffic may still be blocked.

Typical checks:

```bash
gcloud compute firewall-rules list
gcloud compute firewall-rules describe RULE_NAME
```

Look for:

- an ingress rule
- the correct protocol and port, such as `tcp:80`
- the correct target tags or service account
- the correct source ranges, such as `0.0.0.0/0` for public access

Example firewall rule for HTTP:

```bash
gcloud compute firewall-rules create allow-http \
  --allow tcp:80 \
  --direction INGRESS \
  --target-tags http-server \
  --source-ranges 0.0.0.0/0
```

If the VM uses network tags, confirm the tag is attached:

```bash
gcloud compute instances describe VM_NAME --zone=ZONE --format="get(tags.items)"
```

**Check the VM’s internal and external IPs**

Make sure you are testing the correct address.

```bash
gcloud compute instances describe VM_NAME --zone=ZONE \
  --format="get(networkInterfaces[0].networkIP,networkInterfaces[0].accessConfigs[0].natIP)"
```

Interpretation:

- `networkIP` is the internal IP
- `natIP` is the external IP, if assigned

If the VM has no external IP, you cannot reach it directly from the public internet unless you use a proxy, load balancer, bastion host, or VPN.

**Common troubleshooting scenarios**

- Service is `active` but browser shows connection refused:
  - The app may be listening on the wrong port.
  - The firewall may block the port.
  - The app may bind only to localhost.

- Service fails immediately after restart:
  - Check `journalctl` for syntax errors or missing files.
  - Run the service’s config test if available.

- Local `curl` works but remote access fails:
  - Check Google Cloud firewall rules.
  - Check the VM’s OS firewall.
  - Confirm the service listens on `0.0.0.0`, not `127.0.0.1`.

- Port is in use:
  - Another process may already be bound to the port.
  - Find the process with `ss -tulpn` or `lsof -i :PORT`.

```bash
sudo lsof -i :80
sudo lsof -i :8080
```

**Example: verify a Python HTTP server**

This is a common exam-style test because it is simple and easy to validate.

Start the server:

```bash
python3 -m http.server 8080 --bind 0.0.0.0
```

Verify the port is open:

```bash
sudo ss -tulpn | grep :8080
```

Test locally:

```bash
curl http://localhost:8080
```

If you want external access, ensure the firewall allows `tcp:8080`:

```bash
gcloud compute firewall-rules create allow-8080 \
  --allow tcp:8080 \
  --direction INGRESS \
  --target-tags test-server \
  --source-ranges 0.0.0.0/0
```

Then confirm the VM has the `test-server` tag and test from another host:

```bash
curl http://EXTERNAL_IP:8080
```

**Example: verify an NGINX service**

Check status:

```bash
sudo systemctl status nginx
```

Validate configuration:

```bash
sudo nginx -t
```

Restart after changes:

```bash
sudo systemctl restart nginx
```

Confirm listening socket:

```bash
sudo ss -tulpn | grep :80
```

Test locally:

```bash
curl -I http://localhost
```

If external access fails, verify the firewall rule and the VM tag.

**Practical checklist**

- Confirm the service is running with `systemctl status`.
- Confirm the port is listening with `ss -tulpn`.
- Confirm local response with `curl localhost`.
- Confirm external reachability if required.
- Check `journalctl` for startup or runtime errors.
- Check Google Cloud firewall rules and VM network tags.
- Confirm the service binds to the correct interface.

**Exam-focused takeaways**

For ACE, you should be able to:

- SSH into a VM and inspect a service.
- Use `systemctl`, `ss`, and `curl` to verify availability.
- Distinguish between application failure and network exposure failure.
- Check logs when the service does not start.
- Recognize that Google Cloud firewall rules control ingress to the VM.

**Quick reference commands**

```bash
gcloud compute ssh VM_NAME --zone=ZONE
sudo systemctl status SERVICE
sudo systemctl restart SERVICE
sudo journalctl -u SERVICE -f
sudo ss -tulpn | grep :PORT
curl http://localhost:PORT
gcloud compute firewall-rules list
gcloud compute instances describe VM_NAME --zone=ZONE
```
