## **Course:** Google Cloud Associate Cloud Engineer (ACE)

**Why context matters**

Before you create resources, deploy apps, or troubleshoot access, you need to know which <abbr title="Google Cloud SDK command-line tool">gcloud</abbr> identity and project are active. Many mistakes in Google Cloud come from running commands against the wrong account, the wrong project, or the wrong configuration. Inspecting context is a fast habit that prevents accidental billing, permission errors, and changes in the wrong environment.

In practice, your active context comes from the current `gcloud` configuration, which includes the selected account, project, region, zone, and other defaults. These values affect how commands resolve resources when you do not specify them explicitly.

**Core concepts**

- **Active account** is the identity `gcloud` uses for authentication and authorization.
- **Active project** is the default Google Cloud project used by many commands.
- **Configuration** is the set of stored defaults that `gcloud` reads from the active profile.
- **Application Default Credentials (ADC)** are separate from the active `gcloud` account and are often used by client libraries and some local development tools.

> The active `gcloud` account is not always the same as the credentials used by an application. Always verify both when troubleshooting access.

**Inspect the current account**

Use `gcloud auth list` to see which accounts are available and which one is active.

```bash
gcloud auth list
```

Typical output shows a list of authenticated accounts, with an asterisk marking the active one:

```text
Credentialed Accounts
ACTIVE  ACCOUNT
*       user@example.com
        service-account@my-project.iam.gserviceaccount.com
```

What to look for:

- The `*` indicates the account currently used by `gcloud`.
- Multiple accounts may be stored locally, but only one is active at a time.
- If the wrong account is active, commands may fail with permission errors or affect the wrong resources.

If you need to switch accounts, use:

```bash
gcloud config set account user@example.com
```

If the account is not yet authenticated, sign in first:

```bash
gcloud auth login
```

For service accounts, you typically activate a key file when needed:

```bash
gcloud auth activate-service-account --key-file=/path/to/key.json
```

**Inspect the active project**

Use `gcloud config get-value project` to display the current default project.

```bash
gcloud config get-value project
```

If a project is set, the command returns the project ID. If no project is configured, the output may be empty or indicate that the property is unset.

You can also inspect the full configuration:

```bash
gcloud config list
```

Example output:

```text
[core]
account = user@example.com
project = my-ace-project
disable_usage_reporting = False
```

Useful checks:

- `core/account` shows the active account.
- `core/project` shows the default project ID.
- Other properties may include region and zone defaults.

To set the active project:

```bash
gcloud config set project my-ace-project
```

**Inspect region and zone defaults**

Many Compute Engine and Kubernetes-related commands rely on default location settings. If you omit them, `gcloud` may use the configured region or zone, or prompt you to specify one.

Check the current values with:

```bash
gcloud config get-value compute/region
gcloud config get-value compute/zone
```

Set them when you want commands to target a specific location by default:

```bash
gcloud config set compute/region us-central1
gcloud config set compute/zone us-central1-a
```

These defaults are especially useful when working with zonal resources such as virtual machine instances or persistent disks.

**Understand configuration scopes**

`gcloud` can store settings in different configurations. This helps you separate work across environments such as personal projects, labs, and production.

Common patterns:

- **Default configuration** is the active profile most users work with.
- **Named configurations** let you switch between sets of settings.
- **Environment variables** can override some behavior in scripts and shells.

List configurations with:

```bash
gcloud config configurations list
```

Create a new configuration for a separate environment:

```bash
gcloud config configurations create lab
gcloud config configurations activate lab
gcloud config set account user@example.com
gcloud config set project lab-project-id
```

This is useful when you want to avoid mixing credentials or project settings between tasks.

**Check authentication and application credentials**

`gcloud auth list` tells you which account is active for the CLI, but applications may use ADC instead. To inspect ADC, use:

```bash
gcloud auth application-default print-access-token
```

If ADC is configured, this command returns a token. If not, it fails and indicates that application default credentials are missing.

To initialize ADC for local development:

```bash
gcloud auth application-default login
```

Use this when testing client libraries or tools that rely on ADC rather than the active `gcloud` account.

**Practical troubleshooting workflow**

When a command behaves unexpectedly, verify context in a consistent order:

1. Check the active account with `gcloud auth list`.
2. Check the active project with `gcloud config get-value project`.
3. Check location defaults with `gcloud config get-value compute/region` and `gcloud config get-value compute/zone`.
4. Confirm whether the command uses `gcloud` credentials or ADC.
5. Re-run the command with explicit flags if needed.

Example of making a command unambiguous:

```bash
gcloud compute instances list \
  --project=my-ace-project \
  --zone=us-central1-a
```

Explicit flags reduce dependence on local defaults and make scripts more reliable.

**Common pitfalls**

- **Wrong account active**: You may have access to the project, but under a different identity.
- **Wrong project set**: Resources may be created in an unintended billing or lab project.
- **Missing project**: Some commands require a project and fail if none is configured.
- **ADC mismatch**: A local application may authenticate differently from the CLI.
- **Location defaults missing**: Zonal or regional commands may fail or target the wrong location.

> Be careful when using shared terminals or lab environments. Always inspect context before running destructive commands such as deletes, resets, or quota-heavy deployments.

**Useful command summary**

| Purpose | Command |
|---|---|
| Show authenticated accounts | `gcloud auth list` |
| Show active project | `gcloud config get-value project` |
| Show full config | `gcloud config list` |
| Set active account | `gcloud config set account ACCOUNT` |
| Set active project | `gcloud config set project PROJECT_ID` |
| Set region | `gcloud config set compute/region REGION` |
| Set zone | `gcloud config set compute/zone ZONE` |
| List configurations | `gcloud config configurations list` |
| Activate a configuration | `gcloud config configurations activate NAME` |
| Login for ADC | `gcloud auth application-default login` |

**Exam focus**

For ACE-style tasks, expect to verify and adjust context before performing resource operations. You should know how to:

- Identify the active account.
- Identify the active project.
- Set project, region, and zone defaults.
- Switch between configurations when working across environments.
- Distinguish CLI authentication from application default credentials.

A strong workflow is to inspect first, then act. That habit helps you avoid mistakes and makes your commands more predictable in both the exam and real projects.
