## **Course:** Google Cloud Associate Cloud Engineer (ACE)

**Overview**

Google Cloud resources are created in a specific **project**, and many compute resources also require a **region** and **zone**. Setting defaults in the Google Cloud CLI (`gcloud`) reduces repetitive typing, helps you avoid accidental deployments in the wrong location, and makes scripts easier to read.

A **project** is the top-level container for billing, APIs, IAM (Identity and Access Management), and resources. A **region** is a geographic area such as `us-central1`, and a **zone** is a sub-area within a region such as `us-central1-a`. Some services are regional, some are zonal, and some are global, so you should understand which default matters for the service you are using.

**Why defaults matter**

When you work in the CLI, `gcloud` uses configuration values to fill in missing parameters. If you set a default project, region, and zone, commands become shorter and less error-prone.

- **Project** default controls where `gcloud` sends API requests for most resources.
- **Region** default helps commands that create regional resources, such as some networking and managed service components.
- **Zone** default helps commands that create zonal resources, especially Compute Engine virtual machines.

> A default region or zone does not override a service’s actual placement rules. You still need to choose a valid location for the resource type.

**Core concepts**

The `gcloud` CLI stores settings in a named **configuration**. The active configuration is often `default`, but you can create additional configurations for different environments or accounts.

- **Configuration**: a set of CLI properties such as account, project, region, and zone.
- **Property**: a specific setting inside a configuration, such as `compute/region`.
- **Active configuration**: the configuration currently used by `gcloud` commands.

Common properties for this topic include:

- `core/project`
- `compute/region`
- `compute/zone`

You can inspect current values with:

```bash
gcloud config list
gcloud config list --all
```

To see only one property:

```bash
gcloud config get-value project
gcloud config get-value compute/region
gcloud config get-value compute/zone
```

**Set the default project**

Set the active project when you want commands to target a specific Google Cloud project without repeating `--project` each time.

```bash
gcloud config set project PROJECT_ID
```

Example:

```bash
gcloud config set project my-sample-project
```

This is especially useful when:

- You manage multiple projects.
- You switch between lab, development, and production environments.
- You run commands that create or read resources in the current project.

You can also override the default for a single command:

```bash
gcloud compute instances list --project=my-sample-project
```

That override is useful when you need one-off access to a different project without changing your active configuration.

**Set the default region**

Set the default region when you frequently create regional resources or when a command asks for a region and you want to avoid typing it repeatedly.

```bash
gcloud config set compute/region REGION
```

Example:

```bash
gcloud config set compute/region us-central1
```

Use a region that matches your workload, latency, compliance, or service availability needs. Regional placement can affect:

- Network latency to users and other services
- Data residency and compliance requirements
- Availability design for multi-zone or regional architectures

**Set the default zone**

Set the default zone when you often create zonal resources, especially Compute Engine instances.

```bash
gcloud config set compute/zone ZONE
```

Example:

```bash
gcloud config set compute/zone us-central1-a
```

A zone is narrower than a region, so it is useful when you need a specific failure domain or when a lab or script expects a zonal default.

**Typical workflow**

A common setup sequence is to choose the project first, then region, then zone.

1. Select the project.
2. Choose the region.
3. Choose the zone.
4. Verify the active values.

Example:

```bash
gcloud config set project my-sample-project
gcloud config set compute/region us-central1
gcloud config set compute/zone us-central1-a
gcloud config list
```

This workflow helps ensure that subsequent commands use the intended environment.

**Practical examples**

Create a Compute Engine VM without specifying project, region, or zone every time:

```bash
gcloud compute instances create ace-vm-1 \
  --machine-type=e2-medium
```

If the zone is set, `gcloud` can use it automatically for zonal commands. If the zone is not set, you may need to provide `--zone`.

List instances in the current project:

```bash
gcloud compute instances list
```

Create a regional managed instance group or other regional resource with the configured region when supported by the command:

```bash
gcloud config set compute/region europe-west1
```

Then use commands that accept the region implicitly or explicitly depending on the service.

**How defaults interact with commands**

`gcloud` resolves settings in this general order:

- Explicit command-line flags such as `--project`, `--region`, or `--zone`
- Active configuration values
- Service-specific defaults or prompts, when available

This means a command-line flag always wins over the saved default. That behavior is useful when you need a temporary exception.

Example:

```bash
gcloud compute instances create test-vm \
  --zone=us-east1-b \
  --project=other-project
```

Even if your configuration points elsewhere, the flags above take precedence for that command.

**Multiple configurations**

If you work across several environments, separate configurations can prevent mistakes.

```bash
gcloud config configurations create dev
gcloud config configurations activate dev
gcloud config set project dev-project
gcloud config set compute/region us-central1
gcloud config set compute/zone us-central1-a
```

Switch back later:

```bash
gcloud config configurations activate default
```

This approach is useful when:

- You manage multiple accounts or projects
- You want isolated settings for labs and production
- You need reproducible CLI behavior across sessions

**Verify and troubleshoot**

Always verify your settings before creating resources.

```bash
gcloud config list
gcloud config get-value project
gcloud config get-value compute/region
gcloud config get-value compute/zone
```

If a command fails because of location issues, check for these common causes:

- The region or zone is invalid for the service.
- The zone does not exist in the selected region.
- The project is not the one you intended.
- The resource type is global, so region or zone flags are unnecessary or ignored.

> If a command appears to ignore your defaults, check whether the command requires a different flag name or whether an explicit flag is overriding the configuration.

**Reset or remove defaults**

You can unset a property if you no longer want a default value.

```bash
gcloud config unset project
gcloud config unset compute/region
gcloud config unset compute/zone
```

This is useful when:

- You want to force yourself to specify values explicitly
- You are cleaning up a temporary lab configuration
- You are switching to a different environment model

**Exam-focused takeaways**

For ACE-style tasks, you should know how to:

- Set the active project with `gcloud config set project PROJECT_ID`
- Set the default region with `gcloud config set compute/region REGION`
- Set the default zone with `gcloud config set compute/zone ZONE`
- Confirm settings with `gcloud config list`
- Override defaults per command with flags like `--project` and `--zone`

**Quick reference**

| Setting | Property | Example command | Typical use |
|---|---|---|---|
| Project | `core/project` | `gcloud config set project my-sample-project` | Target the correct Google Cloud project |
| Region | `compute/region` | `gcloud config set compute/region us-central1` | Regional resources and location defaults |
| Zone | `compute/zone` | `gcloud config set compute/zone us-central1-a` | Zonal resources such as VM instances |

**Common command set**

```bash
gcloud config set project my-sample-project
gcloud config set compute/region us-central1
gcloud config set compute/zone us-central1-a
gcloud config list
gcloud compute instances create ace-vm-1 --machine-type=e2-medium
```

**Best practices**

- Use a consistent naming convention for projects and environments.
- Verify the active configuration before destructive actions.
- Prefer explicit flags in automation when reproducibility matters.
- Use separate configurations when you regularly switch contexts.
- Choose region and zone based on service requirements, not convenience alone.
