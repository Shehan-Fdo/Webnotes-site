## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure routing fundamentals**

Azure routing determines how packets move between subnets, VNets, the internet, and on-premises networks. You usually troubleshoot routing by checking the **effective routes** on a NIC or subnet and then comparing them with the intended design.

Azure uses several route sources:

- **System routes** are created automatically by Azure.
- **User-defined routes (UDRs)** come from route tables you attach to subnets.
- **BGP routes** are learned from VPN Gateway or ExpressRoute.
- **Peering routes** are injected when VNets are peered.

Azure route selection follows **longest prefix match** first. If multiple routes match the same prefix length, Azure uses source preference in this order:

1. **User-defined route**
2. **BGP route**
3. **System route**

That order matters for AZ-104 questions because a UDR can override a BGP-learned path when the prefix length is the same.

**Common next hop types**

A route entry includes a destination prefix and a **next hop type**. The next hop tells Azure where to send traffic after the route is selected.

- **Virtual network**: traffic stays inside the VNet.
- **VNet peering**: traffic goes to a peered VNet.
- **Internet**: traffic exits to the public internet.
- **Virtual appliance**: traffic goes to an NVA (network virtual appliance), such as a firewall or router.
- **None**: traffic is dropped.

Use these next hop types to force inspection, segment traffic, or override default behavior.

**Default, peering, and propagated routes**

Azure always provides a default route for local VNet traffic and internet access behavior, but the exact effective route set depends on peering, gateways, and UDRs.

- **Default route** `0.0.0.0/0` is often used in UDRs to force all outbound traffic to a firewall or NVA.
- **Peering routes** are automatically added for address spaces in peered VNets.
- **Gateway-learned routes** arrive through BGP when you connect to on-premises networks.
- **Propagated routes** can be disabled on a route table if you do not want BGP routes to influence a subnet.

> If you disable BGP propagation on a route table, Azure stops using learned BGP routes for subnets associated with that table. This is a common exam detail when a custom default route should win over on-premises routes.

**Route tables and propagation**

A route table is attached to a subnet, not to a NIC. When you associate a route table, Azure applies those UDRs to all resources in that subnet.

Typical use cases:

- Force outbound internet traffic through Azure Firewall.
- Send traffic to a hub VNet through an NVA.
- Override a learned route for a specific prefix.
- Block a path by using `None`.

Example UDR configuration:

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

Associate the route table to a subnet:

```bash
az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name vnet-spoke \
  --name app-subnet \
  --route-table rt-spoke
```

**Effective route troubleshooting**

When traffic does not flow as expected, inspect the effective routes on the NIC or subnet.

Portal workflow:

1. Open the VM or NIC.
2. Select **Effective routes**.
3. Compare the selected route with the intended path.
4. Check whether a UDR, BGP route, or peering route is overriding the default.

CLI example:

```bash
az network nic show-effective-route-table \
  --resource-group rg-network \
  --name vm1-nic
```

Useful checks:

- Confirm the destination prefix matches the traffic.
- Confirm the next hop type is correct.
- Confirm the route table is associated with the right subnet.
- Confirm BGP propagation is enabled or disabled as intended.
- Confirm peering is configured with the correct address spaces.

**Traffic flow mental model**

```text
Source VM
  |
  |-- same subnet / same VNet --> system route or UDR
  |
  |-- peered VNet -------------> peering route or UDR
  |
  |-- on-prem via gateway ------> BGP route or UDR
  |
  |-- internet -----------------> Internet route or UDR to firewall
```

**Service endpoints**

A **service endpoint** extends a VNet subnet identity to an Azure PaaS service over the Azure backbone. The service still uses its public endpoint, but Azure recognizes the source subnet and can allow access based on that subnet.

Use service endpoints when you want:

- Simpler private access to supported PaaS services.
- Traffic to stay on the Microsoft backbone.
- Access control based on subnet identity rather than public IPs.

Common supported services include Azure Storage, Azure SQL Database, Azure Key Vault, and others, depending on the service.

Important characteristics:

- The service keeps a public IP endpoint.
- The subnet becomes an allowed source for the service.
- You usually pair this with service firewall rules on the target service.
- The traffic does not get a private IP in your VNet.

Enable a service endpoint on a subnet:

```bash
az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name vnet-app \
  --name app-subnet \
  --service-endpoints Microsoft.Storage Microsoft.Sql
```

Then allow the subnet on the service, for example Storage:

```bash
az storage account network-rule add \
  --resource-group rg-data \
  --account-name mystorageacct \
  --subnet /subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/virtualNetworks/vnet-app/subnets/app-subnet
```

**Service endpoint policies**

A **service endpoint policy** restricts which Azure service resources a subnet can reach through a service endpoint. This adds a layer of control beyond allowing the subnet itself.

Use policies when you want to:

- Allow access only to specific storage accounts or service resources.
- Reduce lateral access to other resources of the same service type.
- Enforce tighter outbound control from a subnet.

Key distinction:

- **Service endpoints** control the path and subnet identity.
- **Service endpoint policies** control which service resources are reachable.

**Private endpoints**

A **private endpoint** maps a PaaS service to a private IP address in your VNet. This is the most restrictive and commonly tested private access model in AZ-104.

Use private endpoints when you want:

- Private IP access to a PaaS service.
- To disable public access to the service.
- Stronger network isolation and simpler private routing from workloads.

Private endpoints are used with services such as:

- Azure Storage
- Azure SQL Database
- Azure Key Vault
- Azure App Service
- Azure Cosmos DB

Each service uses one or more **group IDs** that identify the subresource you are connecting to. For example, Storage uses group IDs such as `blob`, `file`, `queue`, and `table`, depending on the service capability.

**Private endpoint network policies**

Subnets that host private endpoints often require **private endpoint network policies** to be disabled. This allows Azure to place and manage the private endpoint NIC correctly.

Typical subnet setting:

- Disable private endpoint network policies when creating or using private endpoints.
- Re-enable only if your design or feature requires it and you understand the impact.

Example:

```bash
az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name vnet-app \
  --name pe-subnet \
  --disable-private-endpoint-network-policies true
```

**Private endpoint creation**

A private endpoint is created in a subnet and linked to a specific resource and subresource.

Example for a Storage account blob endpoint:

```bash
az network private-endpoint create \
  --resource-group rg-network \
  --name pe-storage-blob \
  --vnet-name vnet-app \
  --subnet pe-subnet \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/mystorageacct \
  --group-ids blob \
  --connection-name pe-storage-blob-conn
```

Depending on CLI version and resource type, you may need to pass the subnet as a resource ID. The important exam concept is that the private endpoint binds to a subnet and a specific service subresource.

**Private DNS zones**

Private endpoints usually need **Private DNS zones** so clients resolve the service name to the private IP instead of the public endpoint.

Common pattern:

- Create the private DNS zone for the service.
- Link the zone to one or more VNets.
- Create an A record automatically through a zone group or manually if needed.
- Ensure clients query DNS that can resolve the private zone.

Example for Storage blob:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.blob.core.windows.net \
  --name link-vnet-app \
  --virtual-network vnet-app \
  --registration-enabled false
```

A private DNS zone can be linked to multiple VNets. That means workloads in different VNets can resolve the same private endpoint name if the zone is linked appropriately and network access exists.

> If you use custom DNS servers, they must be able to resolve the private zone names. In practice, that usually means conditional forwarding to Azure-provided DNS or to a DNS solution that can resolve the private zone. Without that, clients may still resolve the public endpoint.

**Private DNS zone group**

A private DNS zone group associates the private endpoint with the correct DNS zone and creates the needed records.

Example:

```bash
az network private-endpoint dns-zone-group create \
  --resource-group rg-network \
  --endpoint-name pe-storage-blob \
  --name default \
  --private-dns-zone privatelink.blob.core.windows.net
```

If the CLI version expects a different parameter shape, the exam-relevant idea is still the same: the private endpoint must be linked to the correct private DNS zone for the service.

**Private endpoint approval and validation**

Some private endpoint connections require approval from the service owner. In the portal, you can check:

1. The private endpoint connection status.
2. The private IP assigned in the subnet.
3. The DNS zone group and A record.
4. The target service public network access setting.

Validation steps:

- Confirm the private endpoint connection is **Approved**.
- Confirm the NIC in the subnet has the expected private IP.
- Confirm DNS resolves the service FQDN to the private IP.
- Confirm the service is not blocking access through public network restrictions if you expect private-only access.

CLI checks:

```bash
az network private-endpoint show \
  --resource-group rg-network \
  --name pe-storage-blob \
  --query "{name:name, provisioningState:provisioningState, customDnsConfigs:customDnsConfigs}"
```

```bash
az network nic list \
  --resource-group rg-network \
  --query "[?contains(name, 'pe-storage-blob')].{name:name, ipConfigurations:ipConfigurations}"
```

For DNS validation, test name resolution from a VM in the VNet:

```bash
nslookup mystorageacct.privatelink.blob.core.windows.net
```

Or test the service name that your application uses, then confirm it resolves to the private IP.

**Service endpoints vs private endpoints**

| Aspect | Service endpoints | Private endpoints |
|---|---|---|
| IP model | Public service endpoint with subnet identity | Private IP in your VNet |
| Traffic path | Azure backbone to public service endpoint | Private link to service over private IP |
| Public access | Service may still have public endpoint | Public access can be disabled |
| DNS change | Usually no DNS change | Usually requires private DNS zone |
| Control point | Service firewall rules and subnet allow-list | Private endpoint approval, subnet path, and service public access setting |
| Best for | Simpler secure access to supported services | Strong isolation and private-only access |

**When to choose each**

- Choose **service endpoints** when you want quick subnet-based access to a supported PaaS service and do not need a private IP.
- Choose **private endpoints** when you need private IP connectivity, tighter isolation, or public network access disabled.
- Choose **route tables and NVAs** when you need traffic inspection, forced tunneling, or custom routing.

**Troubleshooting scenarios**

A failed service endpoint usually means one of these:

- The subnet does not have the service endpoint enabled.
- The target service firewall does not allow the subnet.
- The service is configured to deny public network access in a way that conflicts with your design.
- The workload is using a different subnet than expected.

A failed private endpoint usually means one of these:

- The private endpoint connection is pending or rejected.
- The subnet does not allow private endpoint placement.
- The private DNS zone is missing or not linked.
- The client DNS server cannot resolve the private zone.
- The service subresource or group ID is wrong.
- The service public access setting does not match the intended design.

End-to-end example for a private endpoint issue:

1. Check the private endpoint status in the portal.
2. Confirm the private IP exists on the endpoint NIC.
3. Verify the private DNS zone link to the VNet.
4. Run `nslookup` from the client VM.
5. Check the target service networking settings.
6. If custom DNS is used, verify forwarding and zone resolution.

**Portal workflow for exam prep**

Use the portal to validate the full path:

- For routing:
  - Open the NIC.
  - Review **Effective routes**.
  - Review **Effective security rules** if NSGs are also involved.
- For service endpoints:
  - Open the subnet.
  - Confirm the service endpoint is enabled.
  - Open the target service networking blade.
  - Confirm the subnet is allowed.
- For private endpoints:
  - Open the private endpoint.
  - Confirm connection state and DNS zone group.
  - Open the private DNS zone.
  - Confirm the A record exists.
  - Test resolution from a VM.

**Practical design patterns**

- Use a hub-and-spoke topology with a firewall in the hub and UDRs in spokes.
- Use service endpoints for moderate isolation when the service still supports public endpoint access.
- Use private endpoints for data services that should not be reachable over the public internet.
- Use private DNS zones linked to all VNets that need name resolution.
- Use BGP propagation carefully when on-premises routes should not override your intended path.

**Quick exam reminders**

- Longest prefix match wins before source preference.
- Among equal prefixes, UDR beats BGP, and BGP beats system routes.
- Route tables attach to subnets.
- Service endpoints do not give the service a private IP.
- Private endpoints do give the service a private IP in your VNet.
- Private endpoints usually need private DNS.
- Custom DNS must be able to resolve private zone names.
- Private endpoint subnets often require network policies disabled.
- Service endpoint policies restrict which service resources a subnet can reach.
