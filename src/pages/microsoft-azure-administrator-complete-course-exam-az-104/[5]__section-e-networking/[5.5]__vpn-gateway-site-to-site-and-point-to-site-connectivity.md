## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**VPN Gateway overview**

Azure VPN Gateway provides encrypted connectivity between Azure virtual networks and other networks over the public internet using IPsec/IKE tunnels. It is the standard choice when you need secure, routed connectivity without dedicated private circuits.

Use VPN Gateway when you need:

- **Site-to-site (S2S)** connectivity between an on-premises network and Azure.
- **Point-to-site (P2S)** connectivity for individual clients such as laptops or admin workstations.
- **VNet-to-VNet** connectivity between Azure virtual networks.
- **Multi-site** connectivity to multiple on-premises locations through one gateway.

A VPN gateway is deployed into a dedicated subnet named `GatewaySubnet`. Azure manages the gateway infrastructure, but you control the gateway type, SKU, routing, authentication, and tunnel configuration.

> A VPN gateway is not the same as a virtual network appliance. It is a managed Azure service with specific SKU and feature limits.

**Core architecture and prerequisites**

A VPN Gateway deployment depends on a few required building blocks:

- A **virtual network** with a dedicated `GatewaySubnet`.
- A **public IP address** for the gateway.
- A **gateway type** of `VPN`.
- A **VPN type** of `Route-based` for most modern scenarios.
- A compatible **gateway SKU** that matches the features you need.
- A local device or client configuration that matches the tunnel and authentication settings.

The `GatewaySubnet` name is mandatory because Azure uses it to identify the subnet reserved for gateway resources. You should not place other workloads there. The subnet size affects future scaling and SKU changes; Azure commonly recommends `/27` or larger, but smaller sizes can work in some cases depending on the design and region capacity.

Typical planning considerations:

- Leave enough address space for gateway scaling and SKU upgrades.
- Avoid overlapping address ranges between Azure and on-premises networks.
- Plan DNS, routing, and firewall rules before deployment.
- Decide whether you need BGP, active-active, zone redundancy, or P2S support.

**Gateway SKUs and feature selection**

Gateway SKUs determine throughput, supported features, and scale. For exam purposes, focus on the relationship between SKU family and capabilities rather than memorizing every throughput number.

Common SKU families include:

- **Basic**: legacy, limited features, not suitable for many production designs.
- **VpnGw1/VpnGw2/VpnGw3**: standard route-based VPN gateway SKUs with increasing scale.
- **VpnGw1AZ/VpnGw2AZ/VpnGw3AZ**: zone-redundant variants in supported regions.
- Higher SKUs such as **VpnGw4/VpnGw5** may be available for larger throughput needs.

Feature guidance:

- **Active-active** is supported on specific route-based VpnGw SKUs, not on Basic.
- **BGP** is supported on route-based VpnGw SKUs and is commonly used for dynamic route exchange.
- **Zone redundancy** is available on AZ SKUs in supported regions.
- **P2S protocol support** depends on both the gateway SKU and the authentication method.
- **OpenVPN** is the most flexible P2S protocol for modern auth methods such as Azure AD.

A practical exam rule: if you need modern features like active-active, BGP, or Azure AD-based P2S, choose a route-based VpnGw SKU rather than Basic.

**VPN Gateway creation in the Azure portal**

The portal workflow is a common AZ-104 topic. You should know the key fields and why they matter.

1. Create or open the target **virtual network**.
2. Add a subnet named `GatewaySubnet`.
3. Create a **public IP address** for the gateway.
4. Create a **VPN gateway** resource.
5. Select the correct **region** and ensure it matches the virtual network region.
6. Choose **VPN** as the gateway type.
7. Choose **Route-based** as the VPN type for most scenarios.
8. Select the required **SKU**.
9. Associate the gateway with the virtual network and public IP.
10. Wait for deployment, which can take a significant amount of time.

Portal fields you should recognize:

- **Gateway type**: `VPN` for encrypted tunnels.
- **VPN type**: `Route-based` or `Policy-based` where supported.
- **SKU**: determines scale and features.
- **Generation**: affects platform capabilities in some deployments.
- **Public IP**: the internet-facing endpoint for the gateway.
- **Active-active configuration**: creates two gateway instances and two public IPs.
- **BGP settings**: ASN and peer IPs for dynamic routing.

**Azure CLI example: create a route-based VPN gateway**

The CLI workflow requires a virtual network with `GatewaySubnet` already present.

```bash
# Create the gateway subnet if it does not already exist
az network vnet subnet create \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --name GatewaySubnet \
  --address-prefixes 10.10.255.0/27

# Create a public IP for the gateway
az network public-ip create \
  --resource-group rg-network \
  --name pip-vpngw \
  --sku Standard \
  --allocation-method Static

# Create the VPN gateway
az network vnet-gateway create \
  --resource-group rg-network \
  --name vpngw-hub \
  --public-ip-addresses pip-vpngw \
  --vnet vnet-hub \
  --gateway-type Vpn \
  --vpn-type RouteBased \
  --sku VpnGw1 \
  --no-wait
```

> The gateway subnet must already exist before gateway creation. In many environments, the gateway deployment fails if the subnet is missing, too small for the chosen design, or overlaps with other planned address space.

**Site-to-site VPN**

S2S VPN connects an on-premises VPN device to an Azure VPN gateway. It is the standard choice for persistent network-to-network connectivity.

Use S2S when:

- You need always-on connectivity between sites.
- You want routed access between on-premises and Azure workloads.
- You need a lower-cost alternative to dedicated private circuits.
- You can support IPsec/IKE configuration on the on-premises device.

S2S design elements:

- **Local network gateway**: represents the on-premises VPN device and address prefixes in Azure.
- **Connection**: binds the Azure VPN gateway to the local network gateway.
- **Shared key**: pre-shared key used by both sides of the tunnel.
- **IPsec/IKE policy**: optional custom cryptographic settings when defaults do not match the on-premises device.
- **BGP**: optional dynamic routing for route exchange and failover.

A local network gateway typically includes:

- The on-premises VPN device public IP.
- On-premises address prefixes.
- Optional BGP peer IP and ASN if BGP is enabled.

**Azure CLI example: create a local network gateway and S2S connection**

```bash
# Create the local network gateway representing the on-premises site
az network local-gateway create \
  --resource-group rg-network \
  --name lng-branch1 \
  --gateway-ip-address 203.0.113.10 \
  --local-address-prefixes 192.168.10.0/24 192.168.20.0/24

# Create the VPN connection
az network vpn-connection create \
  --resource-group rg-network \
  --name conn-branch1 \
  --vnet-gateway1 vpngw-hub \
  --local-gateway2 lng-branch1 \
  --shared-key "StrongSharedKeyHere"
```

For a standard single-site S2S connection, the connection references one local network gateway. In some CLI versions, the parameter naming can vary, but the concept remains the same: the Azure VPN gateway connects to the on-premises local network gateway using a shared key and matching tunnel settings.

**Custom IPsec/IKE policy**

Use a custom IPsec/IKE policy when the default Azure negotiation parameters do not match the on-premises VPN device or when your organization requires specific cryptographic standards.

Common reasons to configure a custom policy:

- The on-premises firewall or VPN appliance supports only certain encryption suites.
- Security policy requires stronger or specific algorithms.
- You need to troubleshoot negotiation failures caused by mismatched proposals.

A policy can specify:

- **IKE version**: IKEv1 or IKEv2.
- **Encryption**: for example `AES256`.
- **Integrity**: for example `SHA256`.
- **DH group**: Diffie-Hellman group for key exchange.
- **PFS**: Perfect Forward Secrecy group.
- **SA lifetime**: Security Association lifetime.

Example policy snippet:

```bash
az network vpn-connection ipsec-policy add \
  --resource-group rg-network \
  --connection-name conn-branch1 \
  --ike-encryption AES256 \
  --ike-integrity SHA256 \
  --ipsec-encryption AES256 \
  --ipsec-integrity SHA256 \
  --dh-group DHGroup14 \
  --pfs-group PFS14 \
  --sa-lifetime 3600 \
  --sa-data-size 102400000
```

Use custom policies carefully. A mismatch between Azure and the on-premises device is a common cause of tunnel establishment failures.

**BGP and route propagation**

Border Gateway Protocol (BGP) enables dynamic route exchange between Azure and connected networks. It reduces manual route maintenance and improves failover behavior.

Use BGP when:

- You have multiple on-premises sites.
- You want automatic route advertisement and withdrawal.
- You need more scalable routing than static prefixes.
- You are building hub-and-spoke or multi-site topologies.

Key BGP concepts:

- **ASN**: Autonomous System Number used to identify the routing domain.
- **BGP peer IP**: the IP address used for BGP peering.
- **Route propagation**: Azure learns and advertises routes through the gateway.
- **Effective routes**: the routes seen by a subnet or NIC after propagation and UDR evaluation.

Route behavior to remember:

- Azure propagates gateway-learned routes to subnets unless route propagation is disabled.
- A **user-defined route (UDR)** can override or suppress propagated routes.
- If you attach a route table with `disableBgpRoutePropagation` set to `true`, BGP-learned routes do not reach that subnet.
- In hub-and-spoke designs, route tables often control whether spoke traffic uses the gateway or a firewall/NVA.

Example route table setting:

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke1

az network route-table update \
  --resource-group rg-network \
  --name rt-spoke1 \
  --disable-bgp-route-propagation true
```

**Multi-site and active-active scenarios**

Multi-site VPN lets one Azure VPN gateway connect to multiple on-premises sites. This is common in branch office designs.

Typical multi-site pattern:

- One Azure hub VNet with a VPN gateway.
- Multiple local network gateways, one per branch.
- One connection per branch.
- Optional BGP for route exchange.

Active-active gateways improve availability by deploying two gateway instances with separate public IPs. This is useful when you want higher resiliency and supported failover behavior.

Important points:

- Active-active requires a supported route-based VpnGw SKU.
- You configure two public IP addresses.
- The on-premises device must support connecting to both gateway IPs.
- Active-active is not the same as zone redundancy, though both improve resilience in different ways.

**Point-to-site VPN**

P2S VPN connects individual client devices to Azure over an encrypted tunnel. It is ideal for administrators, developers, and remote users who need secure access without a site VPN device.

Use P2S when:

- You need per-user remote access.
- You do not have a VPN appliance on the remote side.
- You want a quick secure connection for admin access.
- You need to support roaming clients.

P2S authentication options:

- **Certificate authentication**: client certificate installed on the device.
- **RADIUS authentication**: integrates with an external identity source.
- **Azure AD authentication**: supports modern identity-based access for OpenVPN clients.

Protocol support depends on the auth method and gateway SKU:

| Authentication | Common protocol support | Notes |
|---|---|---|
| Certificate | OpenVPN, IKEv2 | Depends on client OS and gateway SKU |
| RADIUS | OpenVPN, IKEv2 | Often used with existing identity infrastructure |
| Azure AD | OpenVPN | Supported with compatible clients and gateway configuration |

Practical guidance:

- Use **OpenVPN** when you need Azure AD authentication.
- Use **IKEv2** when you want native client support on supported platforms and certificate or RADIUS auth.
- Verify client OS support before choosing the protocol.

**Azure AD P2S example**

Azure AD-based P2S is a common modern setup for user access.

High-level steps:

1. Register the VPN application in Microsoft Entra ID.
2. Configure the VPN gateway for Azure AD authentication.
3. Set the tenant, issuer, and audience values.
4. Download the VPN client profile.
5. Install the client and sign in with your Entra ID account.

Example configuration values often include:

- **Tenant**: `https://login.microsoftonline.com/<tenant-id>/`
- **Issuer**: the Entra ID issuer URL
- **Audience**: the application ID URI or client ID used by the VPN app

Example CLI pattern:

```bash
az network vnet-gateway update \
  --resource-group rg-network \
  --name vpngw-hub \
  --set vpnClientConfiguration.vpnAuthenticationTypes='["AAD"]' \
       vpnClientConfiguration.vpnClientProtocols='["OpenVPN"]'
```

**RADIUS-based P2S**

RADIUS is useful when you already have centralized authentication, MFA, or policy enforcement outside Entra ID.

Use RADIUS when:

- You need integration with an existing NPS or RADIUS server.
- You want to reuse enterprise authentication policies.
- You need certificate-based or username/password-based validation through your identity stack.

Considerations:

- The RADIUS server must be reachable from the VPN gateway.
- Network security groups and firewall rules must allow the required traffic.
- Authentication failures may come from the RADIUS server, not the gateway itself.

**Certificate-based P2S**

Certificate authentication uses a trusted root certificate uploaded to the gateway and client certificates installed on user devices.

Why use it:

- It is simple for small environments.
- It avoids dependency on external identity services.
- It works well for controlled admin access.

Operational notes:

- Upload the root certificate public key to Azure.
- Issue client certificates from that root.
- Install the client certificate on the device that connects.
- Revoke or replace certificates when access changes.

**P2S client configuration and deployment**

After you configure P2S, Azure generates a client package. The package contains the settings needed by the VPN client.

Typical workflow:

1. Configure authentication and protocol settings in the gateway.
2. Download the VPN client package from the portal.
3. Install the appropriate client software.
4. Import the profile or package.
5. Connect and verify access to Azure resources.

Common client considerations:

- Windows, macOS, and Linux support varies by protocol and auth method.
- Azure AD authentication requires OpenVPN-compatible clients.
- Split tunneling may be enabled depending on the profile and design.
- DNS settings matter if you need name resolution for private resources.

**NAT traversal and VPN Gateway NAT**

NAT traversal is relevant when the VPN device or gateway sits behind a NAT device or when address translation is required for overlapping networks.

Use NAT rules when:

- The on-premises VPN device is behind NAT.
- You need to translate overlapping address spaces.
- You want to present different prefixes across the tunnel.

Azure VPN Gateway supports NAT in specific scenarios using NAT rules on the gateway connection. This is especially useful in mergers, acquisitions, or lab environments where overlapping RFC 1918 address ranges exist.

Practical implications:

- NAT can simplify connectivity when address spaces overlap.
- NAT adds complexity to troubleshooting because translated prefixes appear in routes and logs.
- You must verify both tunnel negotiation and translated route behavior.

**Troubleshooting and validation**

Do not rely on `ping` alone to validate VPN connectivity. ICMP is often blocked by guest OS firewalls, Azure NSGs, or on-premises security devices.

Better validation methods:

- Test TCP connectivity with `Test-NetConnection` on Windows.
- Use `curl`, `nc`, or application-specific checks.
- Verify route tables and effective routes.
- Check gateway and connection status in the portal.
- Review VPN diagnostics and logs.

Useful commands:

```powershell
Test-NetConnection 10.10.1.4 -Port 443
tracert 10.10.1.4
```

```bash
az network vpn-connection show \
  --resource-group rg-network \
  --name conn-branch1 \
  --query "{status:connectionStatus, ingress:ingressBytesTransferred, egress:egressBytesTransferred}"
```

Troubleshooting checklist:

- Confirm the gateway is `Connected` or `Provisioning succeeded`.
- Verify the shared key matches on both sides.
- Check that address spaces do not overlap unexpectedly.
- Confirm the on-premises device public IP is correct.
- Validate that the correct IPsec/IKE policy is configured on both ends.
- Ensure route propagation is not blocked by a UDR or disabled BGP propagation.
- Check NSGs, firewalls, and guest OS rules for the target ports.
- Review logs for tunnel negotiation failures, authentication failures, or route issues.

**Diagnostics and monitoring**

Azure provides several ways to observe VPN health and traffic.

Useful monitoring sources:

- **Connection status** in the portal.
- **Metrics** such as tunnel ingress and egress bytes.
- **Diagnostic logs** for gateway and connection events.
- **Azure Monitor** for alerts and dashboards.
- **Network Watcher** for network troubleshooting workflows.

Common log categories may include:

- Gateway diagnostic logs.
- Tunnel diagnostic logs.
- Route diagnostic information.
- Connection events.

Use diagnostics when you need to distinguish between:

- A tunnel that is up but no traffic flows.
- A routing issue versus an authentication issue.
- A client problem versus a gateway problem.

**S2S, P2S, and ExpressRoute comparison**

Choose the connectivity model based on performance, cost, and operational needs.

| Option | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| S2S VPN | Site connectivity | Lower cost, encrypted, persistent | Internet-dependent, variable latency |
| P2S VPN | Individual users | Fast to deploy, flexible remote access | Not ideal for site-wide connectivity |
| ExpressRoute | Enterprise private connectivity | Predictable latency, private circuit, high bandwidth | Higher cost, provider dependency |

Decision guidance:

- Choose **S2S** for branch office connectivity and general hybrid networking.
- Choose **P2S** for admin access, remote work, and temporary secure access.
- Choose **ExpressRoute** when you need private connectivity, predictable performance, or enterprise-scale bandwidth.

**Common exam scenarios**

- A branch office needs secure connectivity to Azure with a VPN appliance: use **S2S**.
- A remote admin needs access from a laptop without a site appliance: use **P2S**.
- Multiple branches must advertise routes dynamically: use **BGP** with S2S.
- The organization requires high availability for the gateway: use an **active-active** supported VpnGw SKU.
- The security team requires specific encryption settings: configure a **custom IPsec/IKE policy**.
- The company uses Entra ID for user authentication: use **Azure AD-based P2S** with **OpenVPN**.

**Practical design tips**

- Place the VPN gateway in a hub VNet and connect spokes through routing controls.
- Keep the `GatewaySubnet` dedicated and appropriately sized for future growth.
- Use route tables intentionally; do not assume Azure will always choose the path you expect.
- Prefer route-based gateways for modern deployments and exam scenarios.
- Validate both tunnel status and application reachability before declaring success.
- Document shared keys, BGP settings, and IPsec policies in a secure configuration repository.

**Quick reference**

- `GatewaySubnet` is required and must be dedicated to the gateway.
- Route-based VPN is the default choice for most Azure designs.
- S2S connects networks; P2S connects users.
- BGP simplifies route exchange and failover.
- Custom IPsec/IKE policies solve interoperability issues.
- Do not use `ping` as your only validation method.
- Azure AD P2S uses OpenVPN.
- Active-active and zone redundancy depend on specific VpnGw SKUs and region support.
