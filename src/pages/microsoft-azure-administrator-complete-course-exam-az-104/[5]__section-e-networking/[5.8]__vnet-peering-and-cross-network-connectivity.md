## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**VNet peering fundamentals**

Azure Virtual Network (VNet) peering connects two VNets through the Microsoft backbone network so resources can communicate privately with low latency. You use it when you want workloads in separate VNets to behave like they are on the same network, without using VPN gateways or public IPs.

Peering is common in AZ-104 scenarios for:

- **Hub-and-spoke** architectures
- Shared services such as DNS, Azure Bastion, firewalls, and jump hosts
- Application tier separation across VNets
- Cross-subscription or cross-region connectivity

A peered network can route traffic directly to the other VNet’s private IP addresses, but peering does not create a single merged network. Each VNet still keeps its own address space, security rules, and management boundaries.

**Core peering behavior and limitations**

VNet peering is fast and simple, but it has important constraints that often appear in exam questions.

- **No transitive routing by default**: if VNet A peers with VNet B, and VNet B peers with VNet C, VNet A cannot reach VNet C through B unless you add routing devices such as Azure Firewall or a virtual appliance.
- **No overlapping address spaces**: the address ranges of peered VNets must not overlap.
- **Private IP only**: peering uses private addresses; it does not require public IPs for VNet-to-VNet traffic.
- **No gateway sharing unless configured**: gateway transit is optional and must be enabled explicitly.
- **Regional and global peering**: you can peer VNets in the same region or different Azure regions, but cross-region peering has different cost and latency considerations.

> If address spaces overlap, Azure blocks peering creation or the peering remains unusable for routing. Plan IP ranges before deployment.

**Peering types and when to use them**

| Type | Scope | Typical use |
|---|---|---|
| Regional peering | Same Azure region | Low-latency connectivity between VNets in one region |
| Global peering | Different Azure regions | Multi-region application connectivity and DR designs |

Regional peering is usually the simplest option for hub-and-spoke designs. Global peering is useful when you need private connectivity across regions, but you should account for bandwidth charges and the fact that traffic still traverses Microsoft’s backbone between regions.

**Permissions and RBAC requirements**

To create and manage peering, you need appropriate Azure Role-Based Access Control (RBAC) permissions on both VNets.

- **Network Contributor** is the common built-in role used to create and manage peerings.
- You need permission on each VNet involved, not just one side.
- In cross-subscription scenarios, the identity creating the peering must have access in both subscriptions or use delegated access.

Common exam trap:

- You can see one VNet but still fail to create peering if you lack permissions on the remote VNet.
- A peering can be created in one direction first, but it is not fully functional until the reciprocal peering exists and is connected.

**How peering is configured**

A peering relationship is effectively two linked configurations, one on each VNet. Each side can have different options enabled.

Typical options include:

- **Allow virtual network access**: permits traffic between the VNets.
- **Allow forwarded traffic**: allows traffic that originated elsewhere and is being forwarded by a network virtual appliance or firewall.
- **Allow gateway transit**: lets one VNet share its VPN or ExpressRoute gateway with the peered VNet.
- **Use remote gateways**: lets the peered VNet use the remote VNet’s gateway.

A common misconfiguration is enabling gateway transit on the wrong side. The VNet that owns the gateway enables **Allow gateway transit**. The peered VNet that wants to use that gateway enables **Use remote gateways**.

**Gateway transit prerequisites and constraints**

Gateway transit is used when you want a spoke VNet to reach on-premises networks through a hub VNet’s VPN gateway or ExpressRoute gateway.

Key rules:

- The hub VNet must already have a deployed gateway.
- The hub peering must have **Allow gateway transit** enabled.
- The spoke peering must have **Use remote gateways** enabled.
- A VNet can use only one remote gateway at a time.
- If a VNet already has its own gateway, it cannot use a remote gateway.
- Gateway transit is not the same as transitive routing between spokes; it only shares gateway access.

Practical misconfiguration examples:

- Enabling **Use remote gateways** on the hub instead of the spoke.
- Forgetting to enable **Allow gateway transit** on the hub side.
- Expecting spoke-to-spoke traffic to flow through the hub gateway automatically.
- Trying to use remote gateways when the spoke already has a VPN gateway.

**Hub-and-spoke design**

Hub-and-spoke is the most common enterprise pattern for Azure network segmentation. The hub hosts shared services, and spokes host workloads.

Typical hub services:

- **Azure Bastion** for secure browser-based RDP/SSH access
- Azure Firewall or a third-party NVA for inspection and egress control
- DNS forwarders or Azure DNS Private Resolver
- VPN or ExpressRoute gateway for on-premises connectivity
- Jump boxes or management VMs

Typical spoke services:

- Application VMs
- App tiers separated by environment or business unit
- Databases or internal services isolated from other workloads

A simple traffic model:

- Spoke VM to spoke VM in another spoke: traffic goes directly over peering only if the route exists and no firewall/NVA is required.
- Spoke VM to on-premises: traffic goes to the hub gateway if gateway transit is configured.
- Admin access to spoke VM: you often connect through Azure Bastion in the hub rather than exposing public IPs.

**End-to-end hub-and-spoke example**

Consider this layout:

- Hub VNet: `10.0.0.0/16`
  - `AzureBastionSubnet`: `10.0.1.0/26`
  - `GatewaySubnet`: `10.0.2.0/27`
  - `AzureFirewallSubnet`: `10.0.3.0/26`
- Spoke1 VNet: `10.1.0.0/16`
  - App subnet: `10.1.1.0/24`
- Spoke2 VNet: `10.2.0.0/16`
  - App subnet: `10.2.1.0/24`

Expected design:

- Spoke1 and Spoke2 peer to Hub.
- Hub peering enables **Allow gateway transit** if the hub has a VPN/ExpressRoute gateway.
- Each spoke peering enables **Use remote gateways** if it needs on-prem access.
- Spoke subnets use NSGs to allow only required east-west traffic.
- Azure Firewall in the hub can inspect traffic if you force routes through it.

Example route table for a spoke subnet when using a firewall in the hub:

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke1

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke1 \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.3.4
```

Example NSG rule allowing app-to-app traffic from another spoke:

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-spoke1 \
  --name Allow-Spoke2-App \
  --priority 200 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.2.1.0/24 \
  --source-port-ranges '*' \
  --destination-address-prefixes 10.1.1.0/24 \
  --destination-port-ranges 443
```

Expected traffic flow:

- Spoke2 app reaches Spoke1 app on TCP 443 if peering exists and NSGs allow it.
- Spoke1 admin reaches a VM through Azure Bastion in the hub using RDP or SSH.
- Spoke1 reaches on-premises through the hub gateway if remote gateway use is enabled.
- Internet-bound traffic can be forced through Azure Firewall if the route table sends `0.0.0.0/0` to the firewall.

**Service endpoints and private endpoints with peering**

Service endpoints and private endpoints are often confused with peering, but they solve different problems.

- **Service endpoints** extend a VNet identity to a supported Azure service over the Azure backbone.
- **Private endpoints** assign a private IP from a VNet to a specific Azure PaaS resource.

How they interact with peering:

- A private endpoint in one VNet is reachable from peered VNets if routing and DNS are configured correctly.
- A service endpoint is enabled on a subnet, not on the peering itself.
- Peering does not automatically make private endpoint DNS work across VNets.
- For cross-VNet private endpoint access, you usually need private DNS zone links or custom DNS forwarding.

Practical considerations:

- If a storage account uses a private endpoint in the hub, spokes need DNS resolution to the private IP.
- If a spoke uses service endpoints to access a storage account, the storage firewall must allow the spoke subnet.
- Private endpoints are usually preferred for tighter isolation and explicit private IP access.

Example private DNS zone link for cross-VNet name resolution:

```bash
az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.blob.core.windows.net \
  --name link-spoke1 \
  --virtual-network /subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/virtualNetworks/vnet-spoke1 \
  --registration-enabled false
```

Without DNS integration, a VM in a peered VNet may still resolve the public endpoint instead of the private endpoint, causing connectivity failures or unexpected internet routing.

**DNS across peered VNets**

DNS is a frequent source of “peering works but the app does not” issues.

Common patterns:

- Use Azure-provided DNS for simple environments.
- Use custom DNS servers in the hub for centralized name resolution.
- Forward queries from spokes to hub DNS servers.
- Link private DNS zones to all VNets that need to resolve private endpoint names.

Troubleshooting checklist:

- Confirm the VM uses the expected DNS server.
- Verify the private DNS zone is linked to the correct VNets.
- Check whether the name resolves to a private IP or public IP.
- If using custom DNS, ensure conditional forwarders exist for Azure private zones.

Useful commands:

```bash
nslookup myapp.privatelink.database.windows.net
Resolve-DnsName myapp.privatelink.database.windows.net
```

**Security controls: NSGs, UDRs, and firewalls**

Peering does not bypass security controls. Network Security Groups (NSGs), User Defined Routes (UDRs), and firewalls still apply.

- **NSGs** filter traffic at subnet or NIC level.
- **UDRs** control next-hop selection.
- **Azure Firewall** or NVAs can inspect and log traffic.
- **Forced tunneling** can send internet-bound traffic through a central inspection point.

Important exam point:

- If peering is connected but traffic fails, check NSGs and route tables before assuming peering is broken.
- If you want spoke-to-spoke traffic to pass through a hub firewall, you need routing that sends traffic to the firewall and peering that allows forwarded traffic.

**Troubleshooting peering issues**

Common failure states and causes:

- **Initiated**: one side created the peering, but the remote side has not accepted or completed it.
- **Disconnected**: the relationship was removed or is no longer synchronized.
- **Connected but no traffic**: NSG, route, DNS, or gateway transit issue.

Troubleshooting steps:

1. Verify both peerings exist and are in `Connected` state.
2. Confirm address spaces do not overlap.
3. Check that both VNets are in supported regions for the chosen peering type.
4. Review NSG rules on source and destination subnets.
5. Inspect UDRs for forced tunneling or blackhole routes.
6. Validate DNS resolution for private endpoints or internal names.
7. Confirm gateway transit settings on the correct peering side.
8. Test with Network Watcher tools.

Useful Network Watcher commands:

```bash
az network watcher test-connectivity \
  --source-resource /subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Compute/virtualMachines/vm-spoke1 \
  --dest-address 10.2.1.4 \
  --dest-port 443
```

```bash
az network watcher show-next-hop \
  --resource-group rg-network \
  --vm vm-spoke1 \
  --source-ip 10.1.1.4 \
  --dest-ip 10.2.1.4
```

You can also use:

- **Connection troubleshoot** to test end-to-end reachability
- **Effective routes** to see whether traffic uses peering, a firewall, or a gateway
- **IP flow verify** to determine whether an NSG blocks the packet

**Practical exam scenarios**

- If two VNets in different subscriptions need private communication, create peering in both directions and ensure RBAC allows it.
- If a spoke must reach on-premises, configure gateway transit on the hub and remote gateway use on the spoke.
- If a workload in one spoke must reach another spoke through a firewall, use UDRs and allow forwarded traffic.
- If a private endpoint is unreachable from a peered VNet, check DNS first, then peering, then NSGs and routes.
- If Azure Bastion is deployed in the hub, use it for secure admin access to spoke VMs without public IPs.

**Quick configuration checklist**

- Confirm non-overlapping address spaces.
- Create peering on both VNets.
- Enable **Allow virtual network access**.
- Enable **Allow forwarded traffic** when using NVAs or Azure Firewall.
- Enable **Allow gateway transit** only on the gateway-owning VNet.
- Enable **Use remote gateways** only on the spoke that should consume the hub gateway.
- Link private DNS zones where private endpoints are used.
- Validate NSGs and UDRs.
- Test with Network Watcher.

**Key takeaways**

VNet peering is the standard Azure method for private VNet-to-VNet connectivity, but it is not a full routing solution by itself. You must still design for address space planning, RBAC, DNS, security rules, and gateway behavior. In AZ-104, the most important distinctions are between direct peering, gateway transit, and routed hub-and-spoke traffic.
