## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Network Security Groups overview**

An <abbr title="Network Security Group">NSG</abbr> is Azure’s primary Layer 4 traffic filter for virtual networks. You use it to allow or deny traffic based on source, destination, protocol, port, and direction. NSGs help you enforce segmentation, reduce attack surface, and apply consistent access rules to subnets and individual network interfaces.

NSGs are stateful. If you allow an inbound flow, the return traffic is automatically allowed without needing a separate outbound rule for the response. This matters when you design rules for web apps, management access, and internal service communication.

- **Scope** matters: you can associate an NSG with a subnet or a network interface card (<abbr title="Network Interface Card">NIC</abbr>).
- **Stateful** filtering means established sessions are tracked automatically.
- **Layer 4** filtering means rules match on IP, protocol, and port, not on application content.
- **Priority** determines which rule is evaluated first; lower numbers have higher priority.

**Where NSGs fit in traffic filtering**

NSGs are commonly used with other Azure networking controls such as Azure Firewall, route tables, and application gateways. Use an NSG when you need simple, fast, distributed filtering close to the workload. Use a more advanced control when you need centralized inspection, logging, or application-aware policy.

A practical pattern is to place broad segmentation at the subnet level and tighter exceptions at the NIC level. For example, a subnet NSG can block all inbound traffic except web ports, while a NIC NSG can further restrict administrative access to a single VM.

**Rule evaluation model**

Azure evaluates NSG rules by direction and priority. For a given flow, Azure checks the applicable rules in order until it finds a match. If a rule allows the traffic, the flow is permitted; if a rule denies it, the flow is blocked.

Important evaluation points:

- **Inbound and outbound** rules are evaluated separately.
- **Lower priority numbers** are processed before higher numbers.
- **Both NSGs must allow the flow** when a subnet NSG and a NIC NSG are both associated with the resource.
- A deny in either scope blocks the traffic.
- Default rules still apply unless you override them with a higher-priority custom rule.

> Azure does not merge subnet and NIC NSG rules by choosing the “most restrictive” match. It evaluates the effective rules from both scopes, and the flow is allowed only when neither scope blocks it.

**Default security rules**

Every NSG includes default rules that provide baseline connectivity and protection. You cannot delete these rules, but you can override them with custom rules that have higher priority.

| Priority | Name | Direction | Action | Purpose |
|---:|---|---|---|---|
| 65000 | AllowVNetInBound | Inbound | Allow | Allows traffic within the virtual network |
| 65001 | AllowAzureLoadBalancerInBound | Inbound | Allow | Allows traffic from the `AzureLoadBalancer` service tag, commonly used for load balancer health probes and related infrastructure traffic |
| 65500 | DenyAllInBound | Inbound | Deny | Blocks all other inbound traffic |
| 65000 | AllowVNetOutBound | Outbound | Allow | Allows traffic within the virtual network |
| 65001 | AllowInternetOutBound | Outbound | Allow | Allows outbound traffic to the internet |
| 65500 | DenyAllOutBound | Outbound | Deny | Blocks all other outbound traffic |

These defaults are useful for understanding why traffic may still flow even when you have not created custom rules. For example, internal VM-to-VM traffic inside the same virtual network is allowed by default unless you override it.

**Rule components**

An NSG rule contains the fields you use to define the match criteria and action.

- **Name**: Human-readable identifier.
- **Priority**: Integer from 100 to 4096 for custom rules; lower is evaluated first.
- **Source**: IP address, CIDR range, service tag, application security group, or `Any`.
- **Source port range**: Usually `*` because clients use ephemeral ports.
- **Destination**: IP address, CIDR range, service tag, application security group, or `Any`.
- **Destination port range**: Single port or range such as `80`, `443`, or `10000-20000`.
- **Protocol**: `TCP`, `UDP`, or `Any`.
- **Action**: `Allow` or `Deny`.

A common mistake is to restrict the source port on inbound web traffic. In most cases, the client source port is ephemeral and unpredictable, so you should filter on destination port instead.

**Service tags and application security groups**

Service tags simplify rule management by replacing large IP lists with Azure-managed labels. They are especially useful for platform services and internet-facing patterns.

Examples of service tags include:

- `VirtualNetwork`
- `AzureLoadBalancer`
- `Internet`
- `Storage`
- `Sql`
- `AzureCloud`

<mark>Service tags</mark> reduce maintenance because Azure updates the underlying IP ranges automatically. Use them when the traffic source or destination maps cleanly to a Microsoft-managed service boundary.

<abbr title="Application Security Group">ASG</abbr>s let you group VMs by workload role and reference the group in NSG rules. This is useful when you want rules like “allow web tier to app tier on port 443” without hardcoding IP addresses.

- Use **service tags** for platform-defined endpoints and broad categories.
- Use **ASGs** for workload segmentation inside your environment.
- Use **CIDR ranges** when you need explicit IP control or on-premises address matching.

**Subnet NSG versus NIC NSG**

You can associate an NSG at the subnet level, the NIC level, or both. The choice affects how broadly the rule applies.

| Scope | Best for | Typical use |
|---|---|---|
| Subnet NSG | Shared policy for many resources | Block inbound internet access to an entire tier |
| NIC NSG | Per-VM exceptions | Allow RDP or SSH only to a specific admin VM |
| Both | Layered defense | Broad subnet policy plus targeted VM-specific exceptions |

A subnet NSG is easier to manage at scale because one rule set covers many resources. A NIC NSG is useful when one VM needs a special exception that should not apply to the whole subnet.

When both are present:

- Azure evaluates the effective rules from both NSGs.
- Traffic is allowed only if both scopes permit it.
- A deny at either scope blocks the flow.

**Common traffic filtering scenarios**

A typical three-tier application uses NSGs to separate web, app, and data tiers.

- **Web tier**:
  - Allow inbound `TCP 80` and `TCP 443` from `Internet` or a load balancer.
  - Deny all other inbound traffic.
- **App tier**:
  - Allow inbound from the web tier on the application port.
  - Deny direct internet access.
- **Data tier**:
  - Allow inbound only from the app tier on database ports such as `1433` or `3306`.
  - Deny all other inbound traffic.

For administrative access:

- Prefer <abbr title="Just-In-Time">JIT</abbr> access through Microsoft Defender for Cloud when available.
- If you must open management ports, restrict them to trusted public IPs and short time windows.
- Avoid leaving `3389` (RDP) or `22` (SSH) open to `Any`.

**Example NSG rule set**

The following example shows a simple inbound policy for a web subnet.

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-web \
  --name Allow-HTTP-HTTPS \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes Internet \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 80 443
```

A deny rule can then block everything else:

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-web \
  --name Deny-All-Inbound \
  --priority 200 \
  --direction Inbound \
  --access Deny \
  --protocol '*' \
  --source-address-prefixes '*' \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges '*'
```

This pattern works because the allow rule has a higher priority than the deny rule. If you reversed the priorities, the deny would match first and block the web traffic.

**Azure CLI and PowerShell examples**

You often need to inspect NSGs, rules, and associations during troubleshooting.

```bash
az network nsg list -g rg-network -o table
az network nsg rule list -g rg-network --nsg-name nsg-web -o table
az network nic show -g rg-app --name vm1-nic --query "networkSecurityGroup"
az network vnet subnet show -g rg-network --vnet-name vnet-prod --name subnet-web --query "networkSecurityGroup"
```

For PowerShell:

```powershell
Get-AzNetworkSecurityGroup -ResourceGroupName rg-network
Get-AzNetworkSecurityRuleConfig -NetworkSecurityGroup (Get-AzNetworkSecurityGroup -ResourceGroupName rg-network -Name nsg-web)
Get-AzNetworkInterface -ResourceGroupName rg-app -Name vm1-nic | Select-Object -ExpandProperty NetworkSecurityGroup
```

Effective rule inspection can vary by Azure CLI version and tooling. Use the portal or the currently installed CLI/PowerShell cmdlets to verify effective security rules in your environment rather than relying on a single command syntax.

**Troubleshooting traffic filtering**

When traffic is blocked unexpectedly, work through the path systematically.

1. Confirm the source and destination IPs, ports, and protocol.
2. Check whether the NSG is attached to the subnet, the NIC, or both.
3. Review rule priority and make sure an earlier deny is not matching first.
4. Verify whether the traffic is intra-subnet, inter-subnet, or internet-bound.
5. Check default rules and service tag behavior.
6. Validate that the VM OS firewall is not blocking the traffic after Azure allows it.

NSGs only control traffic at the Azure network layer. A packet can still be blocked by the guest operating system firewall, an application firewall, or a missing listener on the destination port.

**Practical design guidance**

Use NSGs to express intent clearly.

- Allow only the ports you need.
- Deny everything else explicitly when the workload requires tight control.
- Prefer service tags and ASGs over raw IP lists when possible.
- Keep rule names descriptive, such as `Allow-Web-From-Internet` or `Deny-Admin-From-Internet`.
- Document why each exception exists so future changes do not break production traffic.

A good rule set is easy to read, easy to audit, and narrow enough to reduce risk without creating unnecessary operational overhead.

**Key exam points**

- NSGs filter traffic at Layer 4 using source, destination, protocol, and port.
- NSGs are stateful.
- Lower priority numbers are evaluated first.
- Subnet and NIC NSGs can both apply to the same flow.
- Both NSGs must allow the traffic for the flow to succeed.
- Default rules include virtual network allow, Azure Load Balancer allow, and deny-all rules.
- Service tags and ASGs simplify rule management.
- NSGs do not inspect application payloads; use other services for deeper inspection.

**Quick reference**

- **Inbound rule**: Controls traffic entering Azure resources.
- **Outbound rule**: Controls traffic leaving Azure resources.
- **Priority range**: Custom rules typically use `100-4096`.
- **Stateful behavior**: Return traffic is automatically allowed.
- **Best practice**: Start restrictive, then add only the minimum required exceptions.
