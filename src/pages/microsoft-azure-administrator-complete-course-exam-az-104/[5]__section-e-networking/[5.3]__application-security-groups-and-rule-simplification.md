## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Application Security Groups (ASGs) overview**

Application Security Groups let you group Azure virtual machine network interfaces by application role instead of by IP address. You then reference the group in a Network Security Group (NSG) rule, which makes rules easier to read, maintain, and scale.

ASGs are most useful when your workload has stable roles such as web, app, and database tiers. Instead of updating multiple IP-based rules whenever a VM is replaced or its private IP changes, you update membership in the ASG and keep the NSG rule logic intact.

- **ASG** means a logical grouping of NICs for network policy.
- **NSG** means Network Security Group, the Azure stateful packet filter used to allow or deny traffic.
- ASGs simplify rule management, but they do not replace NSGs.

**Core concept and scope**

ASGs are regional resources and are used with NSG rules in the same Azure region. They are not a general-purpose network segmentation feature across regions or across peered virtual networks. In exam scenarios, assume ASG-based NSG rules work only when the ASG and the NSG are in the same region.

ASGs are also limited to NSG-based filtering. They are not used with Azure Firewall, user-defined routes (UDRs), route tables, or other non-NSG traffic controls. If you need centralized inspection, routing, or application-layer filtering, use the appropriate Azure networking service instead of ASGs.

<dt>Why this matters</dt>
<dd>ASGs reduce rule sprawl, but they do not change the underlying network path or bypass other controls.</dd>
<dt>What they reference</dt>
<dd>Network interface cards (NICs), not subnets or entire virtual networks.</dd>

**How ASGs simplify NSG rules**

Without ASGs, you often write rules using source and destination IP addresses or address prefixes. That works, but it becomes brittle when VMs are redeployed, autoscaled, or replaced.

With ASGs, you write rules around application roles:

- `asg-web`
- `asg-app`
- `asg-db`

This lets you express intent directly:

- Allow HTTP/HTTPS from the internet to the web tier.
- Allow app-tier traffic from the web tier.
- Allow database traffic only from the app tier.

A practical benefit is that the NSG rule stays stable while the NIC membership changes.

**Inbound and outbound rule direction**

Direction matters when you use ASGs in NSG rules.

- For **inbound** rules, the source is typically the client or upstream tier, and the destination is the protected tier.
- For **outbound** rules, the source is the originating tier, and the destination is the target tier or service.

Examples:

- Inbound to web tier: source can be `asg-app` or `Internet`, destination can be `asg-web`.
- Inbound to app tier: source can be `asg-web`, destination can be `asg-app`.
- Inbound to database tier: source can be `asg-app`, destination can be `asg-db`.

For outbound filtering, reverse the logic:

- Outbound from web tier to app tier: source `asg-web`, destination `asg-app`.
- Outbound from app tier to database tier: source `asg-app`, destination `asg-db`.

> ASGs are evaluated by NSG rules, so the rule direction and priority still determine the final decision.

**Typical three-tier example**

A common design uses one ASG per tier and one NSG per subnet or workload boundary.

```text
Internet
   |
   v
[Web subnet]  -- NSG: allow 80/443 from Internet to asg-web
   |
   v
[App subnet]  -- NSG: allow 8080 from asg-web to asg-app
   |
   v
[DB subnet]   -- NSG: allow 1433 from asg-app to asg-db
```

This model is easier to maintain than writing rules for each VM IP address.

**Rule simplification before and after**

Before ASGs, you might need rules like:

- Allow `10.0.1.4` to `10.0.2.4` on TCP 8080
- Allow `10.0.1.5` to `10.0.2.4` on TCP 8080
- Allow `10.0.1.6` to `10.0.2.4` on TCP 8080

After ASGs, one rule can cover the whole tier:

- Allow `asg-web` to `asg-app` on TCP 8080

This is especially valuable when VMs scale out or are rebuilt frequently.

**Portal workflow**

Use the Azure portal when you want to create or verify ASGs quickly.

1. Create an Application Security Group.
2. Add one or more NICs to the ASG.
3. Open the NSG attached to the subnet or NIC.
4. Create a security rule.
5. Set the source or destination to the ASG.
6. Save and test connectivity.

The portal is useful for learning, but in production you often automate ASG membership and NSG deployment with Bicep, ARM templates, or Azure CLI.

**Azure CLI examples**

Create an ASG:

```bash
az network asg create \
  --resource-group rg-network \
  --name asg-web \
  --location eastus
```

Add a NIC to the ASG:

```bash
az network nic ip-config address-pool add \
  --resource-group rg-network \
  --nic-name nic-web-01 \
  --ip-config-name ipconfig1 \
  --address-pool asg-web
```

> CLI commands can vary by Azure CLI version and resource type support. Verify the exact command syntax in your environment if a command fails.

Create an NSG rule that allows HTTP from a web ASG to an app ASG:

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-app \
  --name allow-web-to-app-8080 \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-asgs asg-web \
  --destination-asgs asg-app \
  --destination-port-ranges 8080
```

Create a rule that allows HTTPS from the internet to the web tier:

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-web \
  --name allow-https-in \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes Internet \
  --destination-asgs asg-web \
  --destination-port-ranges 443
```

**Bicep pattern for ASG-based NSG rules**

A reliable pattern is to deploy the ASG and NSG separately, then create NSG security rules as child resources. This avoids schema ambiguity and matches common ARM/Bicep usage.

```bicep
param location string = resourceGroup().location

resource asgWeb 'Microsoft.Network/applicationSecurityGroups@2023-09-01' = {
  name: 'asg-web'
  location: location
}

resource asgApp 'Microsoft.Network/applicationSecurityGroups@2023-09-01' = {
  name: 'asg-app'
  location: location
}

resource nsgApp 'Microsoft.Network/networkSecurityGroups@2023-09-01' = {
  name: 'nsg-app'
  location: location
}

resource allowWebToApp 'Microsoft.Network/networkSecurityGroups/securityRules@2023-09-01' = {
  parent: nsgApp
  name: 'allow-web-to-app-8080'
  properties: {
    priority: 100
    direction: 'Inbound'
    access: 'Allow'
    protocol: 'Tcp'
    sourcePortRange: '*'
    destinationPortRange: '8080'
    sourceApplicationSecurityGroups: [
      {
        id: asgWeb.id
      }
    ]
    destinationApplicationSecurityGroups: [
      {
        id: asgApp.id
      }
    ]
  }
}
```

For single ports, use `sourcePortRange` and `destinationPortRange`. For multiple ports, use the plural array properties such as `sourcePortRanges` and `destinationPortRanges` when the schema supports them.

**Practical deployment scenario**

Imagine a small application with three VMs:

- `vm-web-01` and `vm-web-02` in the web tier
- `vm-app-01` in the app tier
- `vm-db-01` in the database tier

You create:

- `asg-web` and add both web NICs
- `asg-app` and add the app NIC
- `asg-db` and add the database NIC

Then you define rules:

- Web NSG: allow inbound TCP 80 and 443 from `Internet` to `asg-web`
- App NSG: allow inbound TCP 8080 from `asg-web` to `asg-app`
- DB NSG: allow inbound TCP 1433 from `asg-app` to `asg-db`

This design gives you a clear policy map and makes future changes easier. If `vm-web-02` is replaced, you only update ASG membership, not every rule that references its IP.

**Troubleshooting ASG-based rules**

When traffic fails, check the problem in layers.

1. Confirm the NIC is a member of the expected ASG.
2. Confirm the NSG is associated with the subnet or NIC you expect.
3. Confirm the rule priority allows the traffic before any deny rule.
4. Confirm the rule direction matches the traffic flow.
5. Confirm the ASG and NSG are in the same region.
6. Confirm the source and destination ports are correct.
7. Confirm no other NSG on the subnet or NIC blocks the flow.

A common misconfiguration is placing the app VM NIC in the wrong ASG. For example, if `vm-app-01` is accidentally added to `asg-web`, the rule `allow-web-to-app-8080` may not match the intended traffic path. The fix is to move the NIC to the correct ASG and recheck the rule direction.

Another common issue is priority. If a deny rule with a lower number appears before the allow rule, the allow rule never takes effect. In Azure NSGs, lower priority numbers are evaluated first.

**Connectivity testing**

Use packet-flow and effective-security-rule checks to validate NSG behavior. These tools help you confirm whether the NSG allows or denies traffic, but they do not validate ASG membership by themselves.

Useful commands:

```bash
az network nic show-effective-nsg \
  --resource-group rg-network \
  --nic-name nic-app-01
```

```bash
az network nic list-effective-nsg \
  --resource-group rg-network \
  --nic-name nic-app-01
```

For packet-level testing, use tools such as `Test-NetConnection` from a Windows VM or `nc`/`curl` from Linux, depending on the port and protocol.

```powershell
Test-NetConnection -ComputerName 10.0.2.4 -Port 8080
```

```bash
curl -I http://10.0.2.4:8080
```

**What to remember for the exam**

- ASGs group NICs by application role.
- ASGs simplify NSG rules by replacing IP-based references with logical groups.
- ASGs are regional and must be used in the same region as the NSG rule.
- ASGs work with NSGs, not with Azure Firewall or UDRs.
- Inbound rules usually use a source ASG and destination ASG to express tier-to-tier access.
- Outbound rules reverse the flow and should be designed accordingly.
- Troubleshooting starts with ASG membership, NSG association, rule priority, and region scope.

**Quick comparison**

| Item | IP-based NSG rule | ASG-based NSG rule |
|---|---|---|
| Maintenance | High when IPs change | Lower because membership changes instead |
| Readability | Harder to understand | Easier to map to app tiers |
| Scale | Becomes verbose | Scales better for repeated patterns |
| Scope | Works with explicit addresses | Works with NIC groups in the same region |

**Exam tip**

If a question asks how to reduce NSG rule complexity for a multi-tier app, choose ASGs when the goal is to group NICs by role and keep rules stable as VMs change. If the question involves cross-region traffic, Azure Firewall, or routing, ASGs are not the right answer.
