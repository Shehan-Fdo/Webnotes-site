## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure Load Balancer overview**

Azure Load Balancer is a Layer 4 <abbr title="Transmission Control Protocol">TCP</abbr>/<abbr title="User Datagram Protocol">UDP</abbr> load-balancing service that distributes traffic to healthy backend instances. It is designed for high availability, low latency, and scale for both inbound and outbound scenarios.

Use it when you need:
- **Many-to-one** distribution for a service such as web, API, or custom TCP/UDP workloads.
- **One-to-one** access to a specific backend instance through inbound NAT rules.
- **Internal** load balancing for private-only services inside a virtual network.
- **Outbound** connectivity patterns, when you explicitly configure them.

Azure Load Balancer is not an application-layer device. It does not inspect HTTP headers, cookies, or URLs. If you need Layer 7 features such as path-based routing or TLS termination, use Application Gateway or Front Door instead.

**Core architecture**

A load balancer configuration usually includes:
- A **frontend IP configuration** that represents the public or private entry point.
- One or more **backend pools** containing NICs, VM scale set instances, or IP-based members.
- **Health probes** that determine whether a backend is available.
- **Load balancing rules** that map frontend ports to backend ports.
- Optional **inbound NAT rules** for direct access to a single backend.
- Optional **outbound rules** for controlled outbound SNAT behavior.

The frontend can be:
- **Public**, using a public IP address.
- **Internal**, using a private IP address from a virtual network subnet.

**SKU differences: Basic vs Standard**

For AZ-104, the most important distinction is the SKU.

| Feature | Basic Load Balancer | Standard Load Balancer |
|---|---|---|
| Security posture | Not secure by default in the same way as Standard | Secure by default; traffic is denied unless allowed by rules and NSGs |
| Availability zones | No zone-redundant design | Supports zone-redundant and zonal deployment patterns |
| Backend scale | Limited compared with Standard | Higher scale and more flexible |
| Health probes | Supported | Supported |
| Outbound behavior | Can support outbound in limited scenarios | No implicit outbound access; configure outbound rules, NAT Gateway, or another method |
| Recommended for new deployments | No | Yes |

> For exam questions, remember that Standard Load Balancer does **not** provide implicit outbound internet access. If you need outbound connectivity, you must configure it explicitly or use another outbound method such as NAT Gateway or a public IP on the instance.

**Availability and resiliency**

Standard Load Balancer supports availability zones in two common ways:
- **Zone-redundant frontend**: the frontend IP is resilient across zones and can serve backends in multiple zones.
- **Zonal frontend**: the frontend is pinned to a specific zone.

This matters when designing for failure domains:
- Use **zone-redundant** designs when you want the frontend to survive a zone failure.
- Use **zonal** designs when you intentionally place the service in one zone and accept that zone as the boundary.

Backend instances can also be spread across zones. The load balancer only sends traffic to healthy instances, so zone-aware placement improves resilience when combined with multiple backend instances and correct probe design.

**Backend pool membership**

Backend pools can include:
- Standalone VM NICs.
- VM scale set instances.
- IP address-based members in supported scenarios.

Practical differences:
- For **standalone VMs**, you typically associate the NIC with the backend pool.
- For **VM scale sets**, backend pool membership is usually integrated with the scale set model, so new instances join automatically when the scale set is configured to use the pool.

This distinction matters in troubleshooting:
- A VM may exist but not be attached to the backend pool.
- A scale set instance may be healthy in the OS but not yet registered in the pool.
- A NIC can be in the pool while the service on the VM is still blocked by the guest firewall.

**Load balancing rules**

A load balancing rule maps a frontend port to a backend port and defines how traffic is distributed.

Common rule settings:
- **Protocol**: TCP or UDP.
- **Frontend port** and **backend port**: can be the same or different.
- **Session persistence**: controls whether repeated flows from the same client stay on the same backend.
- **Idle timeout**: controls how long an idle flow stays open.
- **TCP reset**: can send a reset when the idle timeout expires, which helps clients fail fast.
- **Floating IP**: changes how the frontend and backend ports are mapped for specific scenarios.

Session persistence options:
- **None**: each new flow can go to any healthy backend. Use this for stateless services.
- **Client IP**: flows from the same client IP are pinned to the same backend. Use this when you want some affinity without tying to protocol.
- **Client IP and protocol**: flows from the same client IP and protocol are pinned together. Use this when TCP and UDP traffic from the same client should be treated separately.

Practical guidance:
- Use **None** for stateless web tiers and APIs.
- Use **Client IP** when a backend caches session state and you want better stickiness.
- Use **Client IP and protocol** when the same client may use both TCP and UDP and you want separate affinity behavior.

Idle timeout and TCP reset:
- A longer **idle timeout** helps long-lived connections, but it can also keep stale sessions open longer.
- Enabling **TCP reset** can improve client behavior when a connection is dropped due to timeout, because the client gets an explicit failure instead of waiting silently.
- For long-lived protocols, verify that the timeout matches the application’s connection pattern.

**Health probes**

Health probes tell Azure Load Balancer whether a backend is ready to receive traffic. A backend is considered healthy only when it responds successfully to the probe.

Probe characteristics:
- **Protocol**: TCP, HTTP, or HTTPS depending on the design and SKU support.
- **Port**: can be the service port or a dedicated health endpoint port.
- **Interval**: how often Azure checks the backend.
- **Threshold**: how many consecutive failures are required before the backend is marked unhealthy.

Why probe design matters:
- The probe port does **not** have to be the same as the service port.
- A dedicated probe port is useful when the application port is busy or when you want a lightweight health endpoint.
- A backend can accept TCP connections but still be unhealthy from the application’s point of view, so probe logic should reflect real service readiness.

Example design:
- Web traffic on port `80`
- Health probe on port `8080`
- Application returns a simple success response on `/health`

Example backend health endpoint:
```bash
curl -i http://10.0.1.4:8080/health
```

Important exam point:
- If the probe fails, the load balancer stops sending new flows to that backend.
- Existing flows may continue depending on the protocol and connection state, but you should not rely on unhealthy backends for new traffic.

**Inbound load balancing vs inbound NAT**

These are often confused, so keep the distinction clear.

- **Load balancing rule**
  - Many-to-one distribution.
  - Multiple backend instances share the same frontend port.
  - Use this for application traffic.

- **Inbound NAT rule**
  - One-to-one access to a specific backend instance.
  - A unique frontend port maps to one backend VM port.
  - Use this for administrative access such as RDP or SSH.

Example:
- Load balancing rule: `public-ip:80` distributes to `vm1:80`, `vm2:80`, `vm3:80`
- NAT rule: `public-ip:50001` maps to `vm1:3389`

Use NAT rules when you need direct access to a specific VM behind the load balancer. Use load balancing rules when you want traffic spread across multiple healthy instances.

**Outbound connectivity and SNAT**

Azure Load Balancer can support outbound connectivity through Source Network Address Translation (<abbr title="Source Network Address Translation">SNAT</abbr>), but outbound behavior must be designed explicitly.

Key points:
- Standard Load Balancer does **not** provide implicit outbound internet access.
- You can enable outbound connectivity with:
  - **Outbound rules**
  - **NAT Gateway**
  - **Instance-level public IPs**
  - Other supported outbound paths

SNAT port exhaustion:
- Each outbound connection consumes SNAT ports.
- High connection churn, many clients, or chatty applications can exhaust available ports.
- Symptoms include intermittent outbound failures, connection resets, or timeouts under load.

When to prefer NAT Gateway:
- You need large-scale, predictable outbound internet access.
- You want simpler outbound design with more predictable port allocation.
- You do not need the load balancer to be the outbound control point.

When outbound rules make sense:
- You already use Standard Load Balancer for inbound traffic and want to define outbound behavior on the same resource.
- You need explicit control over outbound translation for a backend pool.

Practical guidance:
- Use outbound rules when you need to define how backend instances share frontend public IPs for egress.
- Use NAT Gateway when outbound internet access is the primary requirement and you want a dedicated outbound service.
- Do not assume a load balancer alone will provide egress.

**Floating IP**

Floating IP changes how the frontend and backend ports are mapped for specific scenarios. It is used for workloads that require the frontend port to be preserved or for special clustering patterns.

Use it carefully:
- It is not a general-purpose feature.
- It is not equivalent to full direct server return in every scenario.
- It is typically used when the backend service expects traffic in a specific port-mapping pattern.

Exam takeaway:
- Treat Floating IP as a specialized configuration option, not a default setting.

**Practical configuration examples**

**Azure CLI: create a public Standard Load Balancer**

```bash
# Resource group and network
az group create -n rg-lb-demo -l eastus
az network vnet create \
  -g rg-lb-demo \
  -n vnet-demo \
  --address-prefix 10.0.0.0/16 \
  --subnet-name subnet-app \
  --subnet-prefix 10.0.1.0/24

# Public IP for the frontend
az network public-ip create \
  -g rg-lb-demo \
  -n pip-lb-demo \
  --sku Standard \
  --allocation-method Static

# Load balancer
az network lb create \
  -g rg-lb-demo \
  -n lb-demo \
  --sku Standard \
  --public-ip-address pip-lb-demo \
  --frontend-ip-name fe-lb \
  --backend-pool-name be-lb

# Health probe
az network lb probe create \
  -g rg-lb-demo \
  --lb-name lb-demo \
  -n probe-http \
  --protocol tcp \
  --port 80

# Load balancing rule
az network lb rule create \
  -g rg-lb-demo \
  --lb-name lb-demo \
  -n rule-http \
  --protocol tcp \
  --frontend-port 80 \
  --backend-port 80 \
  --frontend-ip-name fe-lb \
  --backend-pool-name be-lb \
  --probe-name probe-http \
  --idle-timeout 15 \
  --enable-tcp-reset true
```

**Azure CLI: add a VM NIC to the backend pool**

```bash
az network nic ip-config address-pool add \
  -g rg-lb-demo \
  --nic-name vm1-nic \
  --ip-config-name ipconfig1 \
  --lb-name lb-demo \
  --address-pool be-lb
```

**Azure CLI: create an inbound NAT rule for RDP**

```bash
az network lb inbound-nat-rule create \
  -g rg-lb-demo \
  --lb-name lb-demo \
  -n nat-rdp-vm1 \
  --protocol tcp \
  --frontend-port 50001 \
  --backend-port 3389 \
  --frontend-ip-name fe-lb
```

**Azure CLI: create an outbound rule**

```bash
az network lb outbound-rule create \
  -g rg-lb-demo \
  --lb-name lb-demo \
  -n outbound-demo \
  --protocol All \
  --frontend-ip-configs fe-lb \
  --backend-pool-name be-lb \
  --idle-timeout 15 \
  --allocated-outbound-ports 1024
```

> CLI parameter names can vary by Azure CLI version. Verify with `az network lb outbound-rule create -h` before using in production scripts.

**PowerShell: inspect a load balancer**

```powershell
Get-AzLoadBalancer -ResourceGroupName rg-lb-demo -Name lb-demo
Get-AzLoadBalancerProbeConfig -LoadBalancer (Get-AzLoadBalancer -ResourceGroupName rg-lb-demo -Name lb-demo)
Get-AzLoadBalancerRuleConfig -LoadBalancer (Get-AzLoadBalancer -ResourceGroupName rg-lb-demo -Name lb-demo)
```

**Internal Load Balancer example**

An internal load balancer is used when the service must stay private inside the virtual network.

Typical pattern:
- Frontend IP: private address such as `10.0.1.10`
- Backend pool: app servers in the same virtual network or peered networks
- Rule: TCP `1433` for a database listener or TCP `443` for an internal API
- Probe: dedicated health port such as `59999`

Example topology:
- `10.0.1.10:443` on the internal load balancer
- Backend VMs: `10.0.1.4`, `10.0.1.5`
- Probe: `10.0.1.4:59999`, `10.0.1.5:59999`

This pattern is common for:
- Internal APIs
- Line-of-business applications
- Clustered services
- Private service tiers that should not be exposed to the internet

**HA Ports scenario**

HA Ports is a special load balancing pattern that forwards all ports for a given protocol to the backend pool. It is useful for appliances, NVAs (<abbr title="Network Virtual Appliance">NVA</abbr>), and certain clustered workloads.

Use it when:
- You need to pass a wide range of ports without creating many individual rules.
- You are building a network appliance or a service that listens on many ports.
- You want a simplified rule set for internal traffic distribution.

Be careful:
- HA Ports is not a substitute for application-aware routing.
- You still need correct backend health, NSG rules, and OS firewall settings.

**Diagnostics and monitoring**

Azure provides several ways to verify load balancer health and behavior.

In the Azure portal:
- Check the **Backend health** view to see whether each backend is healthy or unhealthy.
- Review the **Load balancing rules** and **Health probes** blades for configuration errors.
- Confirm that the backend pool contains the expected NICs or scale set instances.

Useful monitoring and troubleshooting signals:
- **Metrics** for data path and health-related behavior.
- **Diagnostic logs** where enabled.
- **Network Watcher** tools such as connection troubleshoot and IP flow verify.
- **Effective security rules** on the NIC or subnet.

Practical checks:
- Confirm the probe succeeds from the backend VM.
- Confirm the service listens on the expected port.
- Confirm the NSG allows the probe source and the client traffic.
- Confirm the guest OS firewall allows the service port.

**Troubleshooting checklist**

When traffic does not reach the backend, check these in order:
1. Verify the frontend IP, rule, and backend pool association.
2. Verify the backend VM or scale set instance is actually in the pool.
3. Verify the health probe succeeds.
4. Verify the NSG allows the required traffic.
5. Verify the guest OS firewall allows the port.
6. Verify the application is listening on the expected port.
7. Verify there is no asymmetric routing caused by custom routes or a second network path.
8. Verify outbound dependencies if the service needs to call other systems.

Common failure causes:
- **Asymmetric routing**: return traffic leaves through a different path than it entered, breaking stateful flows.
- **Backend OS firewall**: Windows Firewall or `iptables`/`nftables` blocks the probe or service port.
- **Probe-specific NSG rules**: the service port is open, but the probe port is blocked.
- **Wrong backend port**: the rule points to the wrong service port.
- **Missing backend registration**: the VM exists but is not in the backend pool.
- **SNAT exhaustion**: outbound connections fail under load.
- **Idle timeout mismatch**: long-lived sessions drop unexpectedly.

**Exam-focused design patterns**

- Use **Standard Load Balancer** for new production designs.
- Use **session persistence** only when the application needs affinity.
- Use **health probes** that reflect real service readiness, not just port openness.
- Use **inbound NAT rules** for admin access to specific VMs.
- Use **internal load balancers** for private service tiers.
- Use **NAT Gateway** when outbound internet access is the primary requirement.
- Use **zone-redundant** designs when you need resilience across availability zones.
- Use **HA Ports** for appliances or broad port forwarding scenarios.

**Quick comparison of common scenarios**

| Scenario | Best fit | Why |
|---|---|---|
| Public web tier with multiple VMs | Standard Load Balancer | Distributes inbound traffic across healthy instances |
| Private API inside a VNet | Internal Load Balancer | Keeps traffic private |
| RDP to a specific VM behind a public frontend | Inbound NAT rule | One-to-one access |
| Large-scale outbound internet access | NAT Gateway | Predictable outbound SNAT at scale |
| Cluster or appliance with many ports | HA Ports | Simplifies broad port forwarding |

**Key exam reminders**

- Standard Load Balancer is the default choice for modern Azure designs.
- It is a Layer 4 service, so it does not do HTTP-aware routing.
- Health probes determine backend eligibility.
- Inbound NAT is for one backend at a time; load balancing rules are for many backends.
- Outbound access must be designed explicitly.
- Zone support and backend health are important for resiliency questions.
- Session persistence, idle timeout, and TCP reset can affect real application behavior.
