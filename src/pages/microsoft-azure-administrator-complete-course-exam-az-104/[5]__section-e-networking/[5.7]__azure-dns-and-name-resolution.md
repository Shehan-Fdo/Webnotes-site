## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure DNS fundamentals**

Azure DNS is a hosting service for DNS zones in Azure. You use it to publish public DNS records for internet-facing names and to manage private DNS zones for internal name resolution in Azure and hybrid environments. DNS (Domain Name System) translates names to IP addresses and other records, so understanding where queries are answered is essential for AZ-104.

Azure DNS is authoritative for zones you host there, but it is not a recursive resolver for arbitrary internet lookups. In practice, you combine Azure DNS with Azure-provided DNS, custom DNS servers, conditional forwarders, and private DNS zones to build the resolution path you need.

- **Public DNS zones** are used for internet-facing names such as `www.contoso.com`.
- **Private DNS zones** are used for internal names such as `app.internal.contoso.com` or Azure service private endpoints.
- **Record sets** group records of the same name and type within a zone.
- **TTL** (time to live) controls how long resolvers cache answers.

**Azure-provided DNS in virtual networks**

By default, Azure virtual machines (VMs) use Azure-provided DNS. Azure assigns a built-in resolver address in the VNet, commonly `168.63.129.16`, which is reachable from Azure resources and provides name resolution for Azure internal names and linked private DNS zones.

This default behavior is convenient because you do not need to deploy DNS servers for basic VM name resolution. However, Azure-provided DNS has limits:

- It is not a replacement for enterprise DNS features such as custom forwarding policies, split-brain zones, or advanced logging.
- It resolves names based on Azure DNS and linked private DNS zones, not your on-premises Active Directory (AD) zones unless you integrate forwarding.
- It does not make your VNet a recursive resolver for arbitrary custom namespaces.

A VM in a VNet typically resolves:
- Other Azure VM hostnames in the same VNet or peered VNets when name resolution is configured appropriately.
- Private DNS zone records linked to the VNet.
- Public internet names through Azure’s recursive infrastructure, unless custom DNS changes that path.

**Custom DNS servers in Azure VNets**

You use custom DNS when you need enterprise control, hybrid name resolution, or split-brain DNS. Common examples include:
- On-premises AD DNS servers reachable over VPN or ExpressRoute.
- Azure-hosted DNS servers that forward queries to on-premises or upstream resolvers.
- Third-party DNS appliances.

To configure a VNet to use custom DNS servers, set the DNS server IP addresses on the VNet. Existing VMs usually need a DHCP renewal or restart to pick up the new settings.

```bash
az network vnet update \
  --resource-group rg-network \
  --name vnet-prod \
  --dns-servers 10.10.0.4 10.10.0.5
```

After you set custom DNS:
- VMs in the VNet receive those DNS server addresses via DHCP.
- The custom DNS servers must be reachable from the VNet.
- You should configure forwarding rules so external names still resolve.

A practical forwarding pattern is:
- On-premises DNS forwards Azure private zones to Azure DNS Private Resolver or to DNS servers that can resolve them.
- Azure-hosted DNS forwards on-premises zones to on-premises DNS.
- Both sides forward unknown queries to public resolvers or recursive infrastructure.

**Split-brain and split-horizon DNS**

Split-brain DNS means the same DNS name resolves differently depending on where the query originates. This is common in hybrid environments where internal users should reach private IPs while external users reach public IPs.

Typical use cases:
- `app.contoso.com` resolves to a public IP from the internet.
- `app.contoso.com` resolves to a private IP from the corporate network or Azure VNet.

You implement this with:
- Separate public and private zones using the same name.
- Conditional forwarding based on source or namespace.
- Private DNS zone links to selected VNets.
- On-premises DNS policies or zone scoping, when available.

**Public DNS zones and delegation**

For public DNS, Azure DNS can host the authoritative zone, but delegation still depends on the parent zone or registrar. If you register `contoso.com` with a registrar and host `contoso.com` in Azure DNS, you must update the registrar or parent zone with the Azure DNS name server (NS) records.

End-to-end delegation flow:
1. You create a public DNS zone in Azure.
2. Azure assigns authoritative name servers for that zone.
3. You copy the NS records from Azure DNS.
4. You update the registrar or parent zone to delegate the domain to those name servers.
5. Recursive resolvers query the parent zone, receive the delegation, and then query Azure DNS for the final answer.

If delegation is missing, the zone exists in Azure but the public internet cannot find it.

> A hosted zone is not publicly reachable until the parent zone or registrar points to the Azure DNS name servers.

**Azure DNS record sets**

Azure DNS manages records in record sets. A record set is the collection of records with the same name and type in a zone. This matters because Azure DNS does not treat each record as a completely separate object in the same way some other DNS systems do.

Key behaviors:
- You create or update a record set at a name and type.
- Multiple records of the same type can exist in one set, such as multiple `A` records for load balancing.
- TTL is configured on the record set.
- Deleting a record set removes all records in that set.

Common record types:
- `A` maps a name to an IPv4 address.
- `AAAA` maps a name to an IPv6 address.
- `CNAME` aliases one name to another name.
- `MX` defines mail exchange servers.
- `TXT` stores text, often for verification or SPF.
- `NS` delegates a zone or subzone.
- `SOA` (Start of Authority) defines zone metadata and is created automatically.
- `SRV` publishes service location data.

Practical constraints:
- The zone apex cannot use a `CNAME` record because the apex must also host `SOA` and `NS` records.
- Some record types are not supported in all Azure DNS scenarios or may have service-specific constraints.
- Alias records are often preferred for Azure resource integration when supported.

**Alias records**

Alias records let you point a DNS name to an Azure resource instead of a fixed IP address or hostname. They are useful when the target IP can change, such as with public IP resources or some Azure services.

Why alias records matter:
- They reduce manual updates when Azure resource IPs change.
- They help avoid stale DNS entries.
- They are commonly used for Azure-integrated public endpoints.

Examples of alias targets can include:
- Public IP addresses.
- Traffic Manager profiles.
- Some Azure resource endpoints, depending on service support.

**Private DNS zones**

Private DNS zones provide name resolution inside Azure VNets without exposing records to the public internet. You link a private DNS zone to one or more VNets so resources in those VNets can resolve names in the zone.

Typical uses:
- Private endpoints for Azure PaaS services.
- Internal application names.
- Hybrid name resolution for Azure-hosted workloads.

A private DNS zone must be linked to the VNet that needs to resolve it. Without the link, the zone exists but the VNet cannot use it.

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.database.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.database.windows.net \
  --name link-prod-vnet \
  --virtual-network vnet-prod \
  --registration-enabled false
```

Important behaviors:
- `registration-enabled` is typically `false` for service private zones.
- Auto-registration is useful for some internal zones, but not for most Azure service private endpoint zones.
- Private DNS zones are separate from public DNS zones, even if the names look similar.

**Private endpoints and DNS**

Private endpoints expose Azure PaaS services over a private IP in your VNet. DNS is critical because clients must resolve the service name to the private endpoint IP rather than the public service IP.

The usual pattern is:
- The service’s public DNS name resolves to a `CNAME`.
- That `CNAME` points to a `privatelink` zone name.
- The `privatelink` zone contains an `A` record for the private endpoint IP.
- Clients in the linked VNet resolve the chain and connect privately.

Example for Azure SQL:
- `myserver.database.windows.net` may CNAME to `myserver.privatelink.database.windows.net`.
- The private zone `privatelink.database.windows.net` contains an `A` record for the private endpoint IP.

Common failure modes:
- The private DNS zone name is wrong for the service.
- The private endpoint is created but not associated with a DNS zone group.
- The CNAME chain is broken because the zone or record is missing.
- The VNet is not linked to the private DNS zone.
- The client is using a DNS server that cannot resolve the private zone.

A zone group associates a private endpoint with one or more private DNS zones. The current Azure CLI pattern is to create the zone group with zone configuration entries rather than assuming a single legacy parameter shape.

```bash
az network private-endpoint dns-zone-group create \
  --resource-group rg-network \
  --endpoint-name pe-sql-prod \
  --name default \
  --private-dns-zone-configs name=sqlzone private-dns-zone=/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-network/providers/Microsoft.Network/privateDnsZones/privatelink.database.windows.net
```

If your CLI version differs, check `az network private-endpoint dns-zone-group create -h` because parameter names can vary across releases.

**DNS propagation, caching, and TTL**

DNS changes are not instant everywhere because resolvers cache answers. TTL controls how long a resolver can reuse a response before asking again.

Exam-relevant points:
- Lower TTL values reduce cache lifetime and speed up changes, but increase query volume.
- Higher TTL values improve performance and reduce load, but delay propagation.
- Public DNS changes can take time to appear globally because recursive resolvers honor cached data.
- Private DNS changes can also appear delayed if clients or local resolvers cache old answers.

Practical implications:
- If you change a public `A` record, some users may still see the old IP until TTL expires.
- If you update a private endpoint record, clients may continue using the old address until their DNS cache is cleared.
- Windows clients may need `ipconfig /flushdns`.
- Linux clients may need to restart the caching service or clear the resolver cache, depending on the distribution.

**Hybrid DNS forwarding patterns**

Hybrid DNS usually requires forwarding between Azure and on-premises DNS. The goal is to ensure each side can resolve the other side’s private namespaces without exposing internal zones publicly.

Common patterns:
- On-premises DNS forwards Azure private zones to Azure DNS Private Resolver inbound endpoints.
- Azure DNS servers or appliances forward on-premises zones to on-premises DNS.
- Conditional forwarders route only specific namespaces, such as `corp.contoso.com` or `privatelink.database.windows.net`.

Windows DNS conditional forwarder example:

```powershell
Add-DnsServerConditionalForwarderZone `
  -Name "corp.contoso.com" `
  -MasterServers 10.1.0.4,10.1.0.5
```

If the DNS server is AD-integrated, you may choose a replication scope appropriate to your environment. That setting is not required for every DNS server and should match your directory design.

Linux `resolv.conf` example for a simple lab is:

```conf
nameserver 10.10.0.4
search corp.contoso.com
```

In enterprise environments, a single DNS server entry is often not enough unless that server forwards external queries correctly and is highly available. Prefer multiple resolvers or a resilient forwarding design.

A more realistic forwarding setup on a Linux DNS server using `named` might include a forward zone:

```conf
zone "corp.contoso.com" {
    type forward;
    forward only;
    forwarders { 10.1.0.4; 10.1.0.5; };
};
```

**Azure DNS Private Resolver**

Azure DNS Private Resolver helps bridge DNS between on-premises and Azure without deploying your own DNS VMs. It provides:
- An **inbound endpoint** for on-premises clients to send DNS queries into Azure.
- An **outbound endpoint** for Azure workloads to forward queries to on-premises DNS.
- **DNS forwarding rulesets** to define which namespaces are forwarded and where.

How it works in practice:
- On-premises clients query the inbound endpoint IP.
- Azure DNS resolution infrastructure answers using linked private DNS zones and Azure name resolution context.
- Azure workloads send selected queries to the outbound endpoint, which forwards them according to rules.

This is especially useful for split-brain DNS and for resolving private endpoints from on-premises networks.

**Troubleshooting name resolution**

When DNS fails, isolate the failure domain first: client, resolver, zone, record, or network path.

Useful tools:
- `nslookup` for quick query testing.
- `Resolve-DnsName` on Windows for detailed diagnostics.
- `dig` on Linux for explicit query tracing.
- `ipconfig /displaydns` and `ipconfig /flushdns` on Windows.
- `resolvectl query` on systemd-based Linux distributions.

Examples:

```powershell
nslookup myserver.database.windows.net
Resolve-DnsName myserver.database.windows.net
Resolve-DnsName myserver.database.windows.net -Server 10.10.0.4
```

```bash
dig @10.10.0.4 myserver.database.windows.net
dig +trace contoso.com
resolvectl query myserver.database.windows.net
```

Troubleshooting checklist:
- Verify the VNet is linked to the correct private DNS zone.
- Verify the private endpoint has the correct zone group association.
- Verify the record name matches the service’s expected `privatelink` zone.
- Verify the client is using the intended DNS server.
- Verify the DNS server can reach its forwarders or upstream resolvers.
- Verify NS delegation exists for public zones.
- Verify TTL and caching are not hiding a recent change.

**Common exam scenarios**

You may be asked to choose the correct DNS design for a workload. Focus on the resolution path, not just the record type.

Typical scenarios:
- A public website needs global name resolution: use Azure DNS public zone and delegate from the registrar.
- An Azure SQL private endpoint must be reachable from a VNet: use a private DNS zone linked to the VNet and a zone group on the private endpoint.
- On-premises users must resolve Azure private endpoints: use Azure DNS Private Resolver or conditional forwarding to a DNS server that can resolve the private zone.
- Internal and external users need different answers for the same name: use split-brain DNS.

**Practical design guidance**

Use Azure-provided DNS when:
- You need simple VM name resolution in Azure.
- You do not need custom forwarding or enterprise DNS policies.

Use custom DNS when:
- You need hybrid resolution with on-premises zones.
- You need conditional forwarding or split-brain behavior.
- You need centralized DNS logging or security controls.

Use private DNS zones when:
- You want private endpoint names to resolve inside VNets.
- You want to avoid exposing internal service endpoints publicly.

Use public Azure DNS zones when:
- You host internet-facing domains.
- You need authoritative DNS managed in Azure.

**Quick reference**

| Topic | Key point |
|---|---|
| Azure-provided DNS | Default VNet DNS uses Azure infrastructure and supports basic Azure name resolution |
| Custom DNS | Required for hybrid forwarding, split-brain, and enterprise control |
| Public DNS delegation | Registrar or parent zone must point NS records to Azure DNS |
| Private DNS zones | Must be linked to VNets to resolve privately |
| Private endpoints | Need correct `privatelink` zone and zone group association |
| TTL | Controls cache duration and propagation speed |
| Alias records | Help map DNS names to Azure resources without fixed IPs |

**What to remember for AZ-104**

- DNS is about the full resolution path, not just where the record lives.
- Azure DNS public zones require delegation from the registrar or parent zone.
- Azure-provided DNS is the default in VNets, but custom DNS is needed for hybrid scenarios.
- Private endpoints depend on correct private DNS zones, zone links, and record chains.
- Caching and TTL explain many “it works here but not there” issues.
- Use the right query tool from the right place: on-premises, Azure VM, or linked VNet.
