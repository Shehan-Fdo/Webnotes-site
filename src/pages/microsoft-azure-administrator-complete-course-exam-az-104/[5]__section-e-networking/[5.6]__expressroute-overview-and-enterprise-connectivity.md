## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**ExpressRoute fundamentals**

Azure ExpressRoute provides private connectivity between your on-premises network and Microsoft cloud services without traversing the public internet. You use it when you need predictable latency, higher reliability, and enterprise-grade network control for hybrid connectivity.

ExpressRoute is commonly used for:
- Private access to Azure virtual networks
- Access to Microsoft services over private connectivity
- Hybrid architectures that need stable throughput and SLA-backed connectivity
- Large-scale enterprise WAN integration

Unlike a site-to-site VPN, ExpressRoute is not an encrypted tunnel by default. It is a private circuit delivered through a connectivity provider or through ExpressRoute Direct. If you need encryption, you typically add IPsec over the circuit or use another encryption layer.

**Connectivity models and circuit delivery**

ExpressRoute circuits are provisioned through one of the supported connectivity paths:
- **Colocation**: Your network equipment is in a provider or exchange facility with direct peering to Microsoft.
- **Point-to-point Ethernet**: A dedicated Ethernet connection links your WAN edge to Microsoft.
- **Any-to-any IP VPN**: A provider-managed WAN can carry ExpressRoute traffic.
- **ExpressRoute Direct**: You connect directly to Microsoft at a peering location with high bandwidth and more control.

A circuit is the logical resource you create in Azure, but the physical path is established by the provider or by direct peering. The provider typically handles last-mile connectivity, while Azure manages the cloud-side termination.

**Circuit SKUs, billing, and capacity planning**

ExpressRoute circuit SKUs matter for both capability and cost. In practice, you choose a bandwidth tier and a billing model that matches your traffic profile.

Common considerations:
- **Standard**: Baseline ExpressRoute capabilities for a region and its connected VNets.
- **Premium**: Adds broader reach, more route scale, and more VNets/circuits support.
- **Local**: Designed for low-latency access to a specific Azure region with data transfer included in the circuit model.
- **Metered data**: Charges apply for outbound data transfer beyond the circuit fee.
- **Unlimited data**: Higher fixed cost, useful for sustained high-volume traffic.

> Billing and feature availability can vary by region and SKU generation. Always verify the current Azure documentation before deployment.

A practical rule:
- Use **Standard** for a single-region or limited hybrid footprint.
- Use **Premium** when you need larger scale, more routes, or broader geographic reach.
- Use **Local** when your workloads and users are concentrated around one Azure region and you want a region-focused design.

**Private peering vs Microsoft peering**

ExpressRoute supports different traffic types, and exam questions often test the distinction.

- **Private peering**
  - Used for access to Azure virtual networks.
  - Carries RFC 1918 private IP traffic.
  - Commonly used for IaaS, internal apps, and hub-and-spoke connectivity.

- **Microsoft peering**
  - Used for supported Microsoft SaaS and PaaS services that expose public endpoints through ExpressRoute.
  - Requires route filtering to advertise only the Microsoft service prefixes you want.
  - Often used for services such as Microsoft 365-related scenarios where supported and permitted by the service.

Private peering is the core Azure networking use case. Microsoft peering is more specialized and requires careful route control to avoid advertising or receiving unnecessary prefixes.

**Peering configuration details**

ExpressRoute peering is not just “turn it on.” You must coordinate several values with your provider and Azure.

Important configuration elements:
- **VLAN ID**: Used to identify the peering VLAN on the circuit.
- **Primary and secondary peer IPs**: Typically /30 or /31-style point-to-point addressing, depending on the peering type and provider design.
- **ASN**: The autonomous system number used for BGP (Border Gateway Protocol) on your side and by Azure/provider.
- **BGP peering**: Establishes route exchange between your edge and Microsoft.

A typical workflow is:
1. Create the ExpressRoute circuit in Azure.
2. Receive the service key and give it to the connectivity provider.
3. Provider provisions the circuit and establishes the physical/virtual handoff.
4. Configure peering parameters such as VLAN, peer IPs, and ASN.
5. Validate BGP session state and route exchange.

Example Azure CLI for circuit creation:
```bash
az network express-route create \
  --resource-group rg-network \
  --name er-prod-eastus \
  --location eastus \
  --bandwidth 1000 \
  --peering-location "Silicon Valley" \
  --sku-family MeteredData \
  --sku-tier Standard
```

Example of checking circuit state:
```bash
az network express-route show \
  --resource-group rg-network \
  --name er-prod-eastus \
  --query "{name:name, provisioningState:provisioningState, serviceProviderProvisioningState:serviceProviderProvisioningState, sku:sku}"
```

**BGP routing behavior**

ExpressRoute uses BGP to exchange routes between your on-premises network and Azure. BGP is the control plane that tells each side which prefixes are reachable.

For private peering, you should expect:
- Your on-premises prefixes advertised to Azure
- Azure VNet address spaces and connected prefixes advertised back to on-premises
- Route propagation through the ExpressRoute gateway into the VNet

A minimal BGP example on a network device might look like this:
```bash
router bgp 65010
 neighbor 10.10.10.2 remote-as 12076
 neighbor 10.10.10.2 description Azure-ExpressRoute-Primary
 network 10.1.0.0 mask 255.255.0.0
```

This is only illustrative. In a real deployment, you must use the exact primary and secondary peer IPs, VLAN, and ASN values assigned for the circuit. You also need to advertise the correct on-premises prefixes and ensure your firewall and routing policy allow BGP.

**ExpressRoute gateway and Azure virtual networks**

An ExpressRoute virtual network gateway connects an Azure VNet to the ExpressRoute circuit. It is the Azure-side gateway resource that enables route exchange and connectivity into the VNet.

Key points:
- The gateway is deployed into a dedicated subnet named `GatewaySubnet`.
- The gateway type must support ExpressRoute.
- The gateway SKU affects throughput and scale.
- The gateway participates in route exchange, but supported data-plane traffic can bypass the gateway data path when **FastPath** is enabled.

Example gateway creation:
```bash
az network vnet-gateway create \
  --resource-group rg-network \
  --name vnet-gw-er \
  --location eastus \
  --vnet vnet-hub \
  --public-ip-address pip-vnet-gw \
  --gateway-type ExpressRoute \
  --sku ErGw2AZ
```

> The gateway is part of the connectivity path, but it is not always the forwarding bottleneck. With FastPath, supported traffic can flow directly to the VNet, improving performance.

**FastPath**

FastPath improves data-plane performance by sending supported traffic directly from the ExpressRoute circuit into the VNet, bypassing the gateway data path for that traffic.

Use FastPath when:
- You need lower latency for supported workloads
- You want to reduce gateway throughput pressure
- Your design uses supported gateway and circuit combinations

FastPath does not remove the need for the gateway. It changes how supported traffic is forwarded after route resolution.

**Virtual network peering and hub-and-spoke design**

ExpressRoute is often deployed in a hub-and-spoke topology.

Typical pattern:
- The **hub VNet** contains the ExpressRoute gateway.
- **Spoke VNets** peer to the hub.
- On-premises routes enter the hub through ExpressRoute.
- Spokes reach on-premises through the hub if gateway transit is configured.

Important peering concepts:
- **Gateway transit**: The hub VNet shares its gateway with peered spokes.
- **Use remote gateways**: A spoke can use the hub’s gateway for on-premises connectivity.
- **Allow gateway transit**: The hub permits gateway use by peers.

Routing limitations to remember:
- VNet peering is not transitive by default.
- A spoke does not automatically learn routes from another spoke through the hub unless the design explicitly supports that path.
- User-defined routes (UDRs) and network virtual appliances (NVAs) can alter traffic flow, but they do not magically create full transitive routing.

A common hub-and-spoke layout:
```text
On-premises
   |
ExpressRoute Circuit
   |
ExpressRoute Gateway in Hub VNet
   |
Hub VNet ---- VNet Peering ---- Spoke VNet A
   |
   +------ VNet Peering ---- Spoke VNet B
```

**Premium add-on capabilities**

ExpressRoute Premium extends the base circuit with broader enterprise reach and scale.

Common Premium benefits:
- More VNets can connect to a circuit
- More routes can be advertised and learned
- Global reach across regions for supported scenarios
- Better fit for large, multi-region enterprise networks

Use Premium when:
- You connect many VNets to one circuit
- You need cross-region connectivity beyond Standard limits
- You have a large route table footprint
- You are building a global hub architecture

Premium is not automatically required for every enterprise deployment. Many single-region or moderate-scale designs work well with Standard.

**Redundancy and high availability**

ExpressRoute is designed for enterprise resilience, but you still need to architect for failure.

Common resilience patterns:
- **Dual circuits** in separate peering locations or with separate providers
- **Dual connections** from your edge to the same circuit
- **Provider diversity** to reduce shared-failure risk
- **Redundant routers** on the customer side
- **Zone-redundant gateway SKUs** where supported

Design guidance:
- Do not rely on a single physical path if the workload is business critical.
- Use separate devices, links, and provider paths where possible.
- Validate failover behavior before production cutover.

**Route propagation and validation**

You should verify routing in both directions: on-premises to Azure and Azure to on-premises.

Useful validation steps:
1. Confirm the circuit and peering are provisioned.
2. Check BGP session status on your edge device.
3. Verify advertised prefixes from on-premises.
4. Confirm Azure learned routes in the VNet.
5. Test reachability with `ping`, `traceroute`, or application-level probes.
6. Review effective routes on Azure NICs and route tables.

Azure CLI examples:
```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name nic-app01
```

```bash
az network express-route list-route-tables \
  --resource-group rg-network \
  --name er-prod-eastus \
  --peering-name AzurePrivatePeering
```

```bash
az network express-route list-arp-tables \
  --resource-group rg-network \
  --name er-prod-eastus \
  --peering-name AzurePrivatePeering
```

On a Linux VM, you can validate path behavior with:
```bash
ip route
traceroute 10.1.0.4
ping -c 4 10.1.0.4
```

**Troubleshooting signals**

When ExpressRoute fails, the issue is often in one of four places: circuit provisioning, BGP, routing, or provider handoff.

Look for:
- Circuit stuck in provisioning
- BGP session down or flapping
- Missing prefixes on one side
- Wrong VLAN or peer IP configuration
- ASN mismatch
- Firewall blocking TCP 179 for BGP
- UDRs overriding expected paths
- Gateway subnet misconfiguration

If traffic reaches Azure but not the target workload, check:
- NSGs (Network Security Groups)
- Azure Firewall or NVAs
- Guest OS firewall
- Route asymmetry between return and forward paths

**Microsoft peering and route filters**

Microsoft peering requires more control than private peering because you are reaching Microsoft service prefixes rather than your own VNets.

Key points:
- You typically use **route filters** to select the Microsoft services you want.
- Route filters help limit advertised service prefixes.
- Not every Microsoft service is available over Microsoft peering.
- Service support and routing behavior can change, so verify current service eligibility before design.

This is especially important in exam scenarios where the question asks how to restrict access to specific Microsoft services over ExpressRoute.

**ExpressRoute vs VPN Gateway**

ExpressRoute and VPN Gateway solve similar hybrid connectivity problems, but they are not the same.

| Feature | ExpressRoute | VPN Gateway |
|---|---|---|
| Transport | Private circuit | Public internet |
| Encryption | Not inherent | IPsec/IKE built in |
| Latency consistency | Higher and more predictable | Depends on internet path |
| Bandwidth | Higher, enterprise-grade | Lower, SKU-dependent |
| Typical use | Mission-critical hybrid networking | Cost-effective site-to-site connectivity |

Use ExpressRoute when you need predictable performance and private connectivity. Use VPN Gateway when you need simpler deployment, lower cost, or encrypted internet-based connectivity.

**Practical design examples**

A common enterprise pattern is:
- Primary region hosts the hub VNet and ExpressRoute gateway
- Spoke VNets host application tiers
- On-premises datacenter connects through ExpressRoute
- Azure Firewall or an NVA inspects north-south traffic
- BGP advertises on-premises and Azure prefixes
- FastPath is enabled for supported workloads

Another pattern is:
- Two ExpressRoute circuits from different providers
- Separate edge routers in the datacenter
- Active-active routing with BGP
- Shared hub VNet with gateway transit
- Regional failover tested with planned circuit withdrawal

**Exam-focused takeaways**

- ExpressRoute provides private connectivity and is not the same as a VPN.
- Private peering is for Azure VNets; Microsoft peering is for supported Microsoft services.
- BGP, VLANs, peer IPs, and ASN values are essential to successful peering.
- ExpressRoute gateways live in `GatewaySubnet` and support hub-and-spoke designs.
- FastPath improves supported data-plane traffic flow.
- Premium adds scale and broader reach.
- ExpressRoute does not inherently encrypt traffic.
- Redundancy requires deliberate design; do not assume the circuit alone guarantees high availability.
