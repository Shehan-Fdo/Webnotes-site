## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Enterprise networking lab mindset**

In AZ-104-style scenarios, you usually need to connect identity, routing, DNS, security, and service access into one working design. The exam often tests whether you can choose the right Azure networking service for a requirement, then validate the path end to end.

A practical lab approach is to start with the user story:
- Who needs access?
- From where?
- To what resource?
- Over which path?
- With what security controls?

That framing helps you avoid common mistakes such as assuming every problem is a firewall issue or that every private service should use the same DNS pattern.

**Core Azure networking building blocks**

Azure networking scenarios usually combine these components:
- **Virtual network (VNet)** for private IP space and segmentation.
- **Subnets** to isolate workloads and apply subnet-level controls.
- **Network security groups (NSGs)** for stateful filtering.
- **Route tables** for user-defined routing.
- **Private Endpoint** for private access to PaaS services.
- **VPN Gateway** or **ExpressRoute** for hybrid connectivity.
- **Azure DNS** and **Azure DNS Private Resolver** for name resolution.
- **Azure Bastion** for secure VM administration without public IPs.
- **NAT Gateway** for predictable outbound internet access.
- **Azure Firewall** or another network virtual appliance for centralized inspection.
- **DDoS Protection Standard** for enhanced protection of public-facing VNets.

**Virtual network design and segmentation**

A VNet is the boundary for private IP communication in Azure. In enterprise labs, you often create separate subnets for application tiers, management, private endpoints, and gateways. This separation makes routing and security easier to reason about.

Typical design goals:
- Keep management access separate from application traffic.
- Reserve dedicated subnets for platform services such as Bastion, gateways, and private endpoints.
- Use address spaces that do not overlap with on-premises ranges if you plan hybrid connectivity.
- Plan for growth; renumbering later is painful.

Example VNet and subnet creation:
```bash
az network vnet create \
  --resource-group rg-net \
  --name vnet-prod \
  --address-prefixes 10.10.0.0/16 \
  --subnet-name snet-app \
  --subnet-prefixes 10.10.1.0/24

az network vnet subnet create \
  --resource-group rg-net \
  --vnet-name vnet-prod \
  --name snet-db \
  --address-prefixes 10.10.2.0/24
```

Useful subnet planning notes:
- **GatewaySubnet** is required for VPN or ExpressRoute gateways and must use that exact name.
- **AzureBastionSubnet** is required for Bastion and should be sized generously enough for the service.
- Private endpoints are commonly placed in a dedicated subnet to simplify policy and DNS troubleshooting.

**Network Security Groups and traffic evaluation**

NSGs are stateful packet filters applied to subnets or NICs. They are a frequent source of lab issues because a rule can be technically correct but still lose to another rule with higher priority.

Key NSG concepts:
- Rules are evaluated by **priority**, where a lower number wins.
- Default rules exist and can allow traffic you did not explicitly configure.
- NSGs are **stateful**, so return traffic for an allowed flow is automatically permitted.
- You can use **service tags** to simplify rules for Azure services.
- You can use **application security groups (ASGs)** to group VMs logically and write rules against the group instead of individual IPs.

Common service tag examples:
- `VirtualNetwork`
- `AzureLoadBalancer`
- `Internet`
- `Storage`
- `Sql`

Example NSG rule creation:
```bash
az network nsg rule create \
  --resource-group rg-net \
  --nsg-name nsg-app \
  --name Allow-HTTPS-From-FrontEnd \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.10.1.0/24 \
  --source-port-ranges '*' \
  --destination-address-prefixes 10.10.2.4 \
  --destination-port-ranges 443
```

NSG troubleshooting nuances:
- Check both the subnet NSG and NIC NSG if both exist.
- Confirm the effective rule set rather than only the rule you created.
- Remember that an allow rule on one side does not override a deny on the other side if the deny is evaluated first.
- Verify whether the source is coming from a service tag, peered VNet, or on-premises range.
- Use IP flow verification when you need to test a specific 5-tuple path.

**NSG rule order and common exam traps**

NSG rules are not processed top to bottom in the file; they are processed by priority number. This matters when you create a broad deny rule and then wonder why a later allow rule never works.

Typical pattern:
- Priority `100` for specific allows.
- Priority `200` for broader allows.
- Priority `4096` or similar for custom deny catch-alls, if needed.

Common traps:
- Allowing `10.0.0.0/8` when the actual source is from a peered VNet with a different range.
- Forgetting that platform traffic may require specific service tags.
- Blocking `AzureLoadBalancer` and breaking health probes.
- Applying a deny rule to a subnet that also hosts a private endpoint or gateway service.

**Azure Bastion for secure VM administration**

Azure Bastion provides browser-based RDP (Remote Desktop Protocol) and SSH (Secure Shell) access to VMs without exposing public IPs. In enterprise labs, it is the preferred answer when you need secure admin access from the Azure portal.

Why you use Bastion:
- No public IP is required on the VM.
- You reduce attack surface.
- You can manage VMs from the portal over TLS (Transport Layer Security).

Deployment requirements:
- A dedicated `AzureBastionSubnet`.
- A public IP for the Bastion host.
- Network access that allows the Bastion service to reach target VMs on RDP or SSH.

Typical workflow:
1. Deploy Bastion into the VNet.
2. Remove public IPs from VMs where possible.
3. Connect from the Azure portal using Bastion.
4. Use it for secure troubleshooting and patching.

**Private Endpoint and private DNS behavior**

A Private Endpoint gives a private IP address from your VNet to a PaaS service such as Azure SQL Database, Storage, or Key Vault. This changes the access path from public internet to private network traffic.

Important behavior:
- The service still has a public DNS name.
- Public name resolution typically follows a CNAME chain to a `privatelink` zone.
- The private DNS zone returns the private endpoint IP address.
- You usually need DNS integration for clients to resolve the private address correctly.

Example flow for Azure SQL:
- Client queries `sql1.database.windows.net`.
- Public DNS returns a CNAME to `sql1.privatelink.database.windows.net`.
- Private DNS zone `privatelink.database.windows.net` returns the A record for the private endpoint IP.

Private endpoint lab considerations:
- Disable or review **private endpoint network policies** on the subnet if required by the service design.
- Understand that multiple private endpoints for the same service can create multiple A records or separate records depending on the zone and naming.
- Split-horizon DNS is common: internal clients resolve to private IPs, while external clients may still resolve public endpoints.
- If DNS is wrong, the service may be reachable by IP but not by name.

Example private endpoint creation:
```bash
az network private-endpoint create \
  --resource-group rg-net \
  --name pe-sql1 \
  --vnet-name vnet-prod \
  --subnet snet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Sql/servers/sql1 \
  --group-id sqlServer \
  --connection-name pe-sql1-conn
```

Private DNS zone link example:
```bash
az network private-dns zone create \
  --resource-group rg-net \
  --name privatelink.database.windows.net

az network private-dns link vnet create \
  --resource-group rg-net \
  --zone-name privatelink.database.windows.net \
  --name link-vnet-prod \
  --virtual-network vnet-prod \
  --registration-enabled false
```

Private endpoint troubleshooting checklist:
- Confirm the private endpoint is approved.
- Confirm the NIC received a private IP in the expected subnet.
- Confirm the private DNS zone exists and is linked to the correct VNet.
- Confirm the A record exists for the service name.
- Confirm the client is using the intended DNS server.
- Confirm the service firewall or public network access setting matches the design.

**Azure DNS Private Resolver**

Azure DNS Private Resolver helps you bridge DNS between Azure and on-premises without deploying your own DNS VM. It is especially useful in hybrid labs where Azure workloads must resolve on-premises names and on-premises clients must resolve Azure private names.

Core concept:
- **Inbound endpoint**: on-premises DNS clients or servers send queries to Azure DNS Private Resolver to resolve Azure private DNS zones.
- **Outbound endpoint**: Azure forwards selected DNS queries to on-premises DNS servers through a forwarding ruleset.

This distinction matters:
- Inbound is for queries coming into Azure DNS Private Resolver.
- Outbound is for forwarding from Azure to on-premises DNS.

Typical setup steps:
1. Create the DNS Private Resolver in a VNet.
2. Create an inbound endpoint in a dedicated subnet.
3. Create an outbound endpoint in a dedicated subnet.
4. Create a forwarding ruleset.
5. Add forwarding rules for specific zones.
6. Link the ruleset to the VNet that needs forwarding.
7. Configure on-premises DNS to forward the relevant Azure private zones to the inbound endpoint IPs.

Example forwarding rule for an on-premises zone:
```bash
az network dns-resolver forwarding-rule create \
  --resource-group rg-net \
  --dns-forwarding-ruleset-name frs-hybrid \
  --name onprem-contoso \
  --domain-name contoso.local \
  --target-dns-servers 10.1.0.4 10.1.0.5 \
  --target-dns-servers-port 53 \
  --enabled true
```

Practical DNS lab pattern:
- On-premises DNS forwards `privatelink.database.windows.net` to the inbound endpoint.
- Azure workloads use the outbound endpoint and forwarding ruleset to resolve on-premises zones such as `contoso.local`.
- You do not usually forward a wildcard `privatelink.*` pattern; you forward the specific private DNS zone names you need.

**Azure DNS and private zone integration**

Azure DNS Private Zones are often used with Private Endpoints. The zone must be linked to the VNets that need name resolution.

Common zones:
- `privatelink.database.windows.net`
- `privatelink.blob.core.windows.net`
- `privatelink.vaultcore.azure.net`

Key behaviors:
- A record creation is usually automatic when the private endpoint is integrated correctly.
- Multiple endpoints can create multiple records or require careful naming.
- If you manually create records, ensure they match the service’s expected private link name.

Troubleshooting DNS:
- Use `nslookup` from the client VM.
- Compare resolution from inside the VNet and from on-premises.
- Check which DNS server the client is using.
- Verify the private DNS zone link and record set.

Example:
```bash
nslookup sql1.database.windows.net
nslookup sql1.privatelink.database.windows.net
```

**VPN Gateway for site-to-site and point-to-site connectivity**

VPN Gateway provides encrypted connectivity over the public internet. It is common in labs that require hybrid access without ExpressRoute.

Connection types:
- **Site-to-site (S2S)** for network-to-network connectivity.
- **Point-to-site (P2S)** for individual client devices.
- **VNet-to-VNet** for Azure-to-Azure connectivity.

P2S configuration details to know:
- Authentication can use **certificate**, **Microsoft Entra ID** (formerly Azure Active Directory), or **RADIUS** depending on gateway type and design.
- You must define a **client address pool** that does not overlap with VNet or on-premises ranges.
- DNS settings matter if clients need to resolve private names after connecting.
- Route propagation determines whether clients learn the right prefixes.

Example P2S considerations:
- Use a dedicated pool such as `172.16.201.0/24`.
- Push internal DNS servers or Azure DNS Private Resolver addresses if name resolution is required.
- Verify the client receives routes to the target VNet prefixes.

Common VPN troubleshooting steps:
- Confirm the gateway is provisioned and the public IP is correct.
- Confirm the tunnel status is connected.
- Check shared key or certificate trust for S2S.
- Verify client profile installation for P2S.
- Confirm address pool overlap is not causing route conflicts.
- Check effective routes on the VM or NIC.

**ExpressRoute fundamentals**

ExpressRoute provides private connectivity between on-premises networks and Azure through a connectivity provider. It is used when you need more predictable latency, higher reliability, or private connectivity that does not traverse the public internet.

Important concepts:
- **Private peering** is used for access to Azure private resources.
- **Microsoft peering** is used for Microsoft public services over ExpressRoute, where supported and configured.
- **Route propagation** determines which prefixes are advertised and learned.
- **Gateway coexistence** can allow VPN and ExpressRoute to exist together in some designs.
- **Failover** planning matters if you have both primary and backup connectivity paths.

Exam-relevant notes:
- Private peering is the common answer for VNet access.
- Microsoft peering is not the same as private peering.
- ExpressRoute gateway SKU and circuit design affect throughput and redundancy.
- Route filters and BGP (Border Gateway Protocol) influence what is advertised.

Troubleshooting ideas:
- Check whether the circuit is provisioned and authorized.
- Verify BGP peering state.
- Confirm the gateway subnet and gateway SKU support the design.
- Check route tables and effective routes for unexpected next hops.

**Azure Load Balancer behavior**

Azure Load Balancer distributes traffic at Layer 4. It is appropriate when you need TCP or UDP load distribution without application-layer inspection.

SKU differences:
- **Basic** is limited and generally not preferred for enterprise designs.
- **Standard** is the recommended SKU for production and supports more enterprise features.

Important behaviors:
- Health probes determine backend availability.
- If the probe fails, the backend is removed from rotation.
- Backend health is often the first place to look when traffic fails.
- Load Balancer does not inspect HTTP content; it works at transport level.

Probe configuration:
- Use a TCP probe when you only need port reachability.
- Use an HTTP probe when the service exposes an HTTP endpoint and you want a path check.
- Make sure the probe path returns a healthy status code.

Example probe:
```bash
az network lb probe create \
  --resource-group rg-net \
  --lb-name lb-web \
  --name probe-http \
  --protocol Http \
  --port 80 \
  --path /health
```

Common misconfigurations:
- Backend VM NSG blocks the probe source.
- The application listens on a different port than the probe.
- The health endpoint returns a redirect or error code.
- The backend pool contains the wrong NIC or IP configuration.

**Application Gateway behavior**

Application Gateway is a Layer 7 load balancer for HTTP and HTTPS traffic. It is commonly used for web applications that need path-based routing, TLS termination, or web application firewall integration.

Key features:
- **TLS termination** at the gateway.
- **Path-based routing** to different backend pools.
- **Cookie-based affinity** when session stickiness is needed.
- **Web Application Firewall (WAF)** for application-layer protection.
- **Health probes** that validate backend application health.

SKU notes:
- Use the modern v2 family for most enterprise scenarios.
- WAF-enabled deployments are common for internet-facing apps.

Probe behavior:
- A backend can be reachable at the network layer but still unhealthy at the application layer.
- Probe path and host header must match the application’s expected response.
- A 200-series response is usually expected unless configured otherwise.

Example backend health probe:
```bash
az network application-gateway probe create \
  --resource-group rg-net \
  --gateway-name appgw-prod \
  --name probe-app \
  --protocol Http \
  --host 10.10.2.4 \
  --path /health \
  --interval 30 \
  --timeout 30 \
  --threshold 3
```

Common misconfigurations:
- Wrong backend port.
- Missing or incorrect host header.
- Probe path returns 301/302 redirect instead of a healthy response.
- NSG or route table blocks gateway-to-backend traffic.
- Backend certificate trust issues when using HTTPS to the backend.

> The gateway subnet must be dedicated, and the gateway design must match the SKU and feature set you plan to use.

**NAT Gateway for outbound internet access**

NAT Gateway provides predictable outbound internet connectivity for resources in a subnet. It is useful when you want stable outbound public IPs without using a load balancer for egress.

Why you use it:
- Simplifies outbound IP management.
- Supports scalable outbound access.
- Avoids per-VM public IPs for internet egress.

Typical use cases:
- App VMs that need outbound package updates.
- Private workloads that need controlled internet access.
- Scenarios where you want a known source IP for external allowlists.

Example association:
```bash
az network nat gateway create \
  --resource-group rg-net \
  --name nat-prod \
  --public-ip-addresses pip-nat \
  --idle-timeout 10

az network vnet subnet update \
  --resource-group rg-net \
  --vnet-name vnet-prod \
  --name snet-app \
  --nat-gateway nat-prod
```

**Azure Firewall and routing choices**

Azure Firewall is a managed firewall service often used for centralized inspection and egress control. It is not the only possible next hop in Azure, so always check the scenario.

Possible next hops:
- **Internet** for direct outbound access.
- **Virtual appliance** such as Azure Firewall or an NVA.
- **Virtual network gateway** for hybrid routing.
- **None** when system routes are sufficient.

Route table example:
```bash
az network route-table route create \
  --resource-group rg-net \
  --route-table-name rt-spoke \
  --name DefaultToFirewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.10.100.4
```

Design caution:
- Do not assume every spoke must route through Azure Firewall.
- Some workloads need direct internet, some need inspection, and some need hybrid next hops.
- The correct answer depends on the security and connectivity requirement.

**DDoS Protection Standard**

DDoS Protection Standard adds enhanced protection for public IP resources in protected VNets. It is relevant when you need stronger resilience for internet-facing services.

What to remember:
- It is enabled at the VNet level.
- It provides telemetry and mitigation for supported public IP resources in that VNet.
- It is often paired with Application Gateway, Load Balancer, or public-facing services.

When it matters:
- Internet-facing enterprise applications.
- Critical workloads with availability requirements.
- Scenarios where attack visibility and mitigation are important.

**Network Watcher and troubleshooting workflow**

Network Watcher is the main Azure troubleshooting toolkit for network path analysis. In labs, it helps you verify whether a packet can flow, where it stops, and what route it takes.

Useful capabilities:
- **IP flow verify** to test whether NSGs allow or deny a flow.
- **Next hop** to see the route a packet would take.
- **Connection troubleshoot** to test reachability between endpoints.
- **Packet capture** for deeper analysis.
- **NSG flow logs** for historical traffic visibility.

Example next hop check:
```bash
az network watcher show-next-hop \
  --resource-group rg-net \
  --vm vm-app \
  --source-ip 10.10.1.4 \
  --destination-ip 10.10.2.4
```

Example IP flow verification:
```bash
az network watcher test-ip-flow \
  --resource-group rg-net \
  --vm vm-app \
  --direction Inbound \
  --protocol TCP \
  --local 10.10.2.4:443 \
  --remote 10.10.1.4:51514
```

Practical troubleshooting workflow:
1. Confirm DNS resolution first.
2. Confirm the route with next hop or effective routes.
3. Confirm NSG rules with IP flow verification.
4. Confirm the service is listening on the expected port.
5. Confirm the backend health probe is passing.
6. Confirm the correct public or private endpoint is being used.

**End-to-end lab scenarios**

Scenario 1: VM cannot reach a private SQL database
- Check `nslookup sql1.database.windows.net`.
- Confirm the private DNS zone `privatelink.database.windows.net` is linked to the VNet.
- Confirm the private endpoint is approved and has a private IP.
- Check NSG rules on the VM subnet and private endpoint subnet.
- Verify the SQL server public network access setting matches the design.

Scenario 2: On-premises users cannot resolve Azure private names
- Confirm on-premises DNS forwards the specific private zone, such as `privatelink.database.windows.net`, to the Azure DNS Private Resolver inbound endpoint.
- Confirm the inbound endpoint IPs are reachable from on-premises.
- Confirm the private DNS zone exists and contains the expected A record.
- Test resolution from both on-premises and an Azure VM.

Scenario 3: Web app behind Application Gateway returns 502
- Check backend health in the gateway.
- Verify the probe path and host header.
- Confirm the backend NSG allows traffic from the gateway subnet.
- Confirm the backend service listens on the configured port.
- Check whether HTTPS backend trust or certificate issues are causing probe failure.

Scenario 4: Spoke VM has no internet access
- Check whether a UDR sends `0.0.0.0/0` to Azure Firewall or another appliance.
- Confirm the firewall or NVA allows outbound traffic.
- If using NAT Gateway, confirm it is associated with the subnet.
- Check NSG outbound rules and effective routes.

Scenario 5: P2S VPN client connects but cannot resolve names
- Confirm the VPN client received the correct DNS servers.
- Confirm the client address pool does not overlap with local networks.
- Confirm routes to the VNet are present.
- If needed, point the client to Azure DNS Private Resolver or internal DNS servers.

**Practical command patterns**

List effective routes on a NIC:
```bash
az network nic show-effective-route-table \
  --resource-group rg-net \
  --name nic-app
```

List effective NSG rules:
```bash
az network nic list-effective-nsg \
  --resource-group rg-net \
  --name nic-app
```

Check private endpoint connections on the target service:
```bash
az network private-endpoint-connection list \
  --id /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Sql/servers/sql1
```

View backend health for Application Gateway:
```bash
az network application-gateway show-backend-health \
  --resource-group rg-net \
  --name appgw-prod
```

**Exam focus summary**

For AZ-104, focus on choosing the right service and validating the path:
- Use **Bastion** for secure VM admin access.
- Use **Private Endpoint** plus private DNS for private PaaS access.
- Use **DNS Private Resolver** for hybrid DNS forwarding.
- Use **VPN Gateway** or **ExpressRoute** for hybrid connectivity.
- Use **NSGs** and **route tables** to control traffic flow.
- Use **Load Balancer** for Layer 4 distribution and **Application Gateway** for Layer 7 web routing.
- Use **NAT Gateway** for predictable outbound internet access.
- Use **Network Watcher** to prove where traffic fails.

The strongest lab answers are usually the ones that explain both the service choice and the validation step. If you can describe the DNS path, the route path, and the security path, you are usually close to the correct solution.
