## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Network troubleshooting workflow**

Azure network troubleshooting is most effective when you move from the symptom to the layer that is most likely failing. Start by identifying whether the issue is DNS, routing, security filtering, platform health, or application behavior. Then validate the path hop by hop: client, source subnet, virtual network, network security group (NSG), user-defined route (UDR), firewall or network virtual appliance (NVA), load balancer, private endpoint, and destination.

A practical workflow is:

1. Confirm the exact failure mode.
   - Is it name resolution, TCP connection failure, intermittent loss, or slow performance?
   - Is the problem inbound, outbound, or east-west between workloads?
2. Identify the source and destination.
   - Use IP address, FQDN, port, protocol, and direction.
   - Note whether the destination is a VM, PaaS service, private endpoint, or on-premises network.
3. Check DNS first when the target is referenced by name.
   - A wrong IP from DNS often looks like a network failure.
4. Validate routing and next hop.
   - Confirm the effective route table and whether traffic is forced through a firewall or NVA.
5. Check security controls.
   - Review NSG rules, Azure Firewall rules, and appliance policies.
6. Verify platform-specific components.
   - Load balancer health probes, backend pool membership, private endpoint approval, or VPN/BGP status.
7. Capture evidence.
   - Use Network Watcher tools, logs, and packet capture to confirm the actual path and failure point.

> Do not assume the first visible symptom is the root cause. A timeout can be caused by DNS, a denied NSG rule, a missing route, a failed health probe, or an upstream appliance drop.

**Core Azure Network Watcher tools**

Azure Network Watcher provides the main diagnostics surface for Azure networking. It helps you test connectivity, inspect effective configuration, capture packets, and view topology. In AZ-104 scenarios, you should know when to use each tool and what it can prove.

- **Connection troubleshoot** checks whether Azure can reach a destination and reports the likely failing hop or rule.
- **IP flow verify** tests whether a specific 5-tuple is allowed or denied by NSG rules.
- **Next hop** shows the route Azure will use from a VM to a destination IP.
- **Effective security rules** shows the combined NSG rules applied to a NIC or subnet.
- **Effective routes** shows system, BGP, and user-defined routes that affect forwarding.
- **Packet capture** records traffic for deeper analysis.
- **Topology** visualizes resources and connections in a region, but it does not replace packet-level or policy-level diagnostics.

**DNS troubleshooting workflow**

DNS is often the first thing to validate because a wrong name resolution result can mimic a network outage. Azure supports Azure-provided DNS, custom DNS servers, and private DNS zones, and each introduces different failure modes.

When troubleshooting DNS, check these points:

- **Azure-provided DNS**
  - Virtual machines use Azure’s recursive resolver by default unless you configure custom DNS.
  - Validate that the VM can resolve public names and internal Azure names as expected.
- **Custom DNS servers**
  - Confirm the DNS server IPs are configured on the virtual network and reachable from the VM.
  - Verify that the DNS server itself can resolve upstream names and is not blocked by NSGs or routes.
- **Private DNS zones**
  - Confirm the zone is linked to the correct virtual network.
  - Verify the record exists and points to the correct private IP.
  - Check for overlapping records or stale cached responses.
- **Split-brain or conditional forwarding**
  - Ensure on-premises DNS forwards the correct zones to Azure when hybrid name resolution is required.

Useful checks from a VM:

```bash
nslookup myapp.internal.contoso.com
nslookup myvm.eastus.cloudapp.azure.com
Resolve-DnsName myapp.internal.contoso.com
```

If resolution fails, isolate the layer:

- If public names fail, check the VM’s DNS settings, outbound connectivity to the resolver, and any custom DNS server health.
- If private names fail, check private DNS zone links, record sets, and whether the VM is using the intended resolver.
- If the name resolves to the wrong IP, check stale records, split-horizon DNS, or an incorrect private endpoint mapping.

**Connection troubleshoot and interpreting results**

Connection troubleshoot in Network Watcher is used to test connectivity between a source and a destination and to identify where the path fails. It is especially useful when you need to validate a VM-to-VM, VM-to-IP, or VM-to-FQDN path without manually tracing every hop.

Typical source and destination types include:

- **Source**
  - Azure VM
  - Azure virtual machine scale set instance
  - On-premises endpoint in some hybrid scenarios, depending on the tool path and configuration
- **Destination**
  - IP address
  - FQDN
  - Azure resource endpoint, depending on the scenario

A typical CLI pattern is:

```bash
az network watcher test-connectivity \
  --source-resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/<vmName> \
  --dest-address 10.10.2.4 \
  --dest-port 443
```

Use the output to look for:

- **Reachable** or **unreachable** status.
- The hop where the path stops.
- Whether the failure is due to DNS, routing, NSG, or an application-level refusal.
- Latency and hop details when available.

Common interpretations:

- **DNS resolution failure**
  - The destination name cannot be resolved, so the test never reaches the network path.
- **TCP timeout**
  - Often indicates a blocked port, missing route, firewall drop, or a service not listening.
- **Connection refused**
  - The destination is reachable, but the application is not listening on the port.
- **Asymmetric path symptoms**
  - Forward traffic reaches the destination, but return traffic is routed elsewhere or blocked.

**IP flow verify and NSG troubleshooting**

IP flow verify is the fastest way to determine whether an NSG allows or denies a specific packet. It evaluates the effective NSG rules for a NIC or subnet using source IP, destination IP, destination port, protocol, and direction.

Use it when you need to answer: “Would this packet be allowed?”

Example CLI pattern:

```bash
az network watcher test-ip-flow \
  --direction Inbound \
  --protocol TCP \
  --local 10.1.0.4:3389 \
  --remote 203.0.113.10:51544 \
  --nic /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Network/networkInterfaces/<nicName>
```

What to check:

- **Direction**
  - Inbound and outbound rules are evaluated separately.
- **Protocol**
  - TCP, UDP, or Any.
- **Source and destination**
  - Match the actual packet, not just the expected subnet.
- **Priority**
  - Lower numbers win.
- **Service tags and application security groups**
  - These can simplify rules but can also hide the effective match if you do not inspect carefully.

NSG troubleshooting tips:

- Check both the subnet NSG and NIC NSG, because either can deny traffic.
- Remember that an allow rule must exist for the exact port and direction.
- Verify that a higher-priority deny rule is not shadowing the intended allow rule.
- Use flow logs when you need historical evidence of allowed and denied flows.

**NSG flow logs and Traffic Analytics**

NSG flow logs record IP traffic metadata that passes through NSGs. They are useful for understanding who talked to whom, on what port, and whether the flow was allowed or denied. Flow logs are not packet captures; they do not show payloads.

Important considerations:

- **Prerequisites**
  - Network Watcher must be enabled in the region.
  - A storage account is required for log storage.
  - Traffic Analytics is optional but useful for aggregation and visualization.
- **Versions**
  - Flow log implementations have evolved over time, so the exact schema and storage format depend on the version and configuration.
  - In exam scenarios, focus on the concept: flow logs provide metadata for NSG traffic analysis.
- **Use cases**
  - Confirm whether traffic reached the NSG.
  - Identify denied flows to a port or subnet.
  - Correlate repeated connection failures with rule changes.

Practical guidance:

- Use flow logs when you need historical evidence after the issue has already occurred.
- Use IP flow verify when you need an immediate rule evaluation for a specific packet.
- Use packet capture when you need packet-level detail, retransmissions, or handshake analysis.

**Packet capture workflow**

Packet capture helps you inspect actual packets on a VM NIC or at the Network Watcher level. It is useful for diagnosing TCP handshakes, retransmissions, resets, MTU-related symptoms, and whether traffic is leaving or arriving at all.

A more realistic Azure CLI workflow usually includes a storage account and a target VM or NIC context:

```bash
az network watcher packet-capture create \
  --resource-group <rg> \
  --vm <vmName> \
  --name cap1 \
  --storage-account <storageAccountName> \
  --filters '[{"protocol":"TCP","localPort":"443","remotePort":"*","localIPAddress":"*","remoteIPAddress":"*"}]'
```

Use filters to reduce noise:

- Limit by protocol.
- Limit by local or remote port.
- Limit by IP address when you know the suspected path.

How to analyze results:

- **SYN without SYN-ACK**
  - The packet leaves the source, but the destination does not respond. This often points to a blocked path, wrong route, or service not listening.
- **SYN, SYN-ACK, then RST**
  - The destination is reachable, but the application or intermediary resets the session.
- **Repeated retransmissions**
  - Suggest loss, filtering, or asymmetric routing.
- **ICMP unreachable**
  - Can indicate routing or policy issues, depending on the source of the message.

Packet capture is most useful when combined with a hypothesis. For example, if `test-connectivity` says the path is unreachable, capture traffic on the source VM and confirm whether packets leave the NIC and whether any return traffic arrives.

**Routing, UDRs, and forced tunneling**

Routing issues are common when traffic is intentionally steered through a firewall, NVA, or on-premises network. Azure uses system routes, BGP routes, and UDRs, and the effective route table determines the actual next hop.

Key concepts:

- **System routes**
  - Automatically created by Azure for local VNet, peering, Internet, and other platform paths.
- **User-defined routes**
  - Custom routes you add to override or steer traffic.
- **BGP routes**
  - Learned from VPN Gateway or ExpressRoute when dynamic routing is in use.

Troubleshooting steps:

1. Check the effective routes on the NIC.
2. Confirm the destination prefix matches the intended route.
3. Verify the next hop type is correct.
4. Validate that the firewall or NVA has a return path.
5. Check for asymmetric routing.

Example route table snippet:

```json
{
  "name": "route-to-firewall",
  "properties": {
    "addressPrefix": "0.0.0.0/0",
    "nextHopType": "VirtualAppliance",
    "nextHopIpAddress": "10.1.0.4"
  }
}
```

Common forced-tunneling failure patterns:

- The UDR sends traffic to the firewall, but the firewall lacks a route back to the source subnet.
- The firewall allows outbound traffic, but return traffic bypasses the firewall and is dropped by stateful inspection.
- A more specific route overrides the intended default route.
- BGP-learned routes conflict with UDRs or create unexpected path selection.

**Azure Firewall and NVA troubleshooting**

When traffic is forced through Azure Firewall or another NVA, you must validate both Azure-side routing and appliance-side policy. A successful Azure route does not guarantee the firewall will allow the flow.

Check these items:

- **Firewall rules**
  - Network rules for IP/port/protocol.
  - Application rules for FQDN-based outbound access.
  - DNAT rules for inbound publishing.
- **SNAT behavior**
  - Source network address translation may be required for outbound flows.
  - If SNAT ports are exhausted or misconfigured, connections can fail intermittently.
- **Asymmetric routing**
  - Ensure return traffic comes back through the same stateful device when required.
- **Logging**
  - Review firewall logs for denies, rule matches, and translated addresses.

Practical symptoms:

- Outbound web traffic fails only when forced through the firewall.
- Internal-to-internal traffic works until a UDR sends it through the appliance.
- Inbound publishing works for some ports but not others because the DNAT rule or backend path is incomplete.

**Load balancer-specific diagnostics**

Load balancers affect connectivity in ways that are easy to miss because the frontend may be reachable while the backend is unhealthy. Troubleshooting requires checking the frontend, probe, backend pool, and any inbound NAT rules.

Validate these components:

- **Health probes**
  - The backend must respond on the probe port and path.
  - A failed probe removes the instance from service.
- **Backend pool membership**
  - Confirm the VM NIC is associated with the correct backend pool.
- **Inbound NAT rules**
  - These map a frontend port to a specific backend VM port.
  - Missing or incorrect NAT rules can make RDP or SSH appear broken.
- **Load balancing rules**
  - Confirm frontend port, backend port, protocol, and probe alignment.

Example failure patterns:

- You can reach the load balancer frontend, but the backend VM is not in the pool.
- The probe uses HTTP on port 80, but the application listens on 8080.
- RDP fails because the inbound NAT rule is missing or points to the wrong backend instance.
- Health probes fail because an NSG blocks the probe source or port.

**Private Endpoint and Private Link diagnostics**

Private Endpoint connectivity issues are common in Azure because they combine DNS, routing, and access control. A private endpoint gives a service a private IP in your VNet, but the client must resolve the service name to that private IP and be allowed by the service’s private access settings.

Troubleshooting checklist:

- **DNS resolution**
  - Confirm the service FQDN resolves to the private endpoint IP.
  - Verify the private DNS zone is linked to the VNet.
- **Network path**
  - Ensure the client can reach the private IP within the VNet or peered network.
  - Check NSGs and UDRs that might block the private IP.
- **Private endpoint state**
  - Confirm the endpoint is approved and provisioned.
- **Service-side access**
  - Verify the PaaS resource allows private access and that public access settings do not conflict with your design.

Example scenario:

- A VM tries to reach a storage account.
- DNS resolves the storage FQDN to the public IP instead of the private endpoint IP.
- The connection fails because public access is disabled.
- Fix the private DNS zone link or record so the name resolves to the private IP.

**Hybrid connectivity and BGP validation**

Hybrid connectivity issues often involve VPN Gateway, ExpressRoute, BGP (Border Gateway Protocol), and route propagation. The key exam skill is to determine whether the route exists, whether it is advertised, and whether both sides agree on the path.

Validate:

- **Gateway status**
  - The VPN or ExpressRoute gateway is provisioned and connected.
- **BGP peering**
  - Confirm BGP sessions are established.
  - Check advertised and learned prefixes.
- **Route propagation**
  - Ensure the subnet receives the expected routes.
- **On-premises routing**
  - Verify the on-premises device has the correct return route to Azure prefixes.

Common misconfigurations:

- The Azure side learns the on-premises prefix, but the on-premises router does not advertise the Azure prefix back.
- A UDR disables route propagation where it is needed.
- A more specific route sends traffic to the wrong next hop.
- The on-premises firewall blocks the return path even though the tunnel is up.

Useful validation commands and checks:

```bash
az network nic show-effective-route-table \
  --resource-group <rg> \
  --name <nicName>

az network nic show-effective-nsg \
  --resource-group <rg> \
  --name <nicName>
```

For VPN and ExpressRoute, also inspect gateway diagnostics and connection status in the portal when route tables alone do not explain the issue.

**Network Watcher topology: when to use it and when not to**

Topology is useful for visualizing relationships between resources in a region, such as VMs, NICs, subnets, NSGs, and load balancers. It helps you understand the architecture quickly and spot missing associations.

Use topology when you need to:

- Confirm which NIC belongs to which VM.
- See whether an NSG or route table is attached.
- Understand the high-level layout of a subnet or resource group.

Do not rely on topology when you need:

- Packet-level evidence.
- Exact rule evaluation.
- DNS resolution results.
- Detailed route precedence or BGP behavior.

Topology is a map, not a diagnosis. It helps you orient yourself before using more precise tools.

**End-to-end troubleshooting examples**

**1. DNS failure to a private service**

A VM cannot reach `myapp.privatelink.database.windows.net`.

1. Run `nslookup` from the VM.
2. Confirm the name resolves to the private endpoint IP.
3. Check the private DNS zone link to the VNet.
4. Verify the private endpoint is approved.
5. Test connectivity to the resolved IP and port.

If the name resolves to a public IP, the issue is usually DNS configuration, not routing.

**2. UDR and forced tunneling failure**

A VM cannot browse the internet after a new route table is associated.

1. Check effective routes on the NIC.
2. Confirm `0.0.0.0/0` points to the intended firewall or NVA.
3. Verify the appliance has outbound rules and SNAT configured.
4. Confirm return traffic is routed back through the same device.
5. Use packet capture to see whether SYN packets leave and whether replies return.

If the firewall logs show denies, the route is working and the policy is the problem. If no packets reach the firewall, the route or next hop is wrong.

**3. Load balancer backend failure**

Users can reach the frontend IP, but the application is unavailable.

1. Check backend pool membership.
2. Validate the health probe port and path.
3. Confirm the backend VM listens on the expected port.
4. Review NSG rules on the backend subnet or NIC.
5. Test the backend directly from another VM in the VNet.

If the probe fails but direct VM access works, the probe configuration is likely incorrect.

**4. Private Endpoint access failure**

A storage account is configured for private access, but the VM gets a timeout.

1. Resolve the storage FQDN from the VM.
2. Confirm the private DNS zone is linked.
3. Check that the private endpoint IP is reachable.
4. Verify no UDR or NSG blocks the private IP.
5. Confirm the storage account allows the private endpoint connection.

If DNS points to the public endpoint, the private endpoint is not being used even if the network path is healthy.

**5. Hybrid route advertisement issue**

A branch site can reach Azure, but Azure cannot reach the branch subnet.

1. Check VPN or ExpressRoute connection status.
2. Review BGP learned routes on the Azure gateway.
3. Confirm the branch router advertises the subnet.
4. Verify the Azure subnet receives the route.
5. Check on-premises firewall and return routing.

If the route is missing on one side, the tunnel may still be up while the network is effectively unreachable.

```bash
# DNS lookup
nslookup contoso.internal

# Effective routes on a NIC
az network nic show-effective-route-table \
  --resource-group <rg> \
  --name <nicName>

# Effective NSG on a NIC
az network nic show-effective-nsg \
  --resource-group <rg> \
  --name <nicName>

# Connection troubleshoot
az network watcher test-connectivity \
  --source-resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/<vmName> \
  --dest-address 10.10.2.4 \
  --dest-port 443

# IP flow verify
az network watcher test-ip-flow \
  --direction Outbound \
  --protocol TCP \
  --local 10.1.0.4:51544 \
  --remote 10.10.2.4:443 \
  --nic /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Network/networkInterfaces/<nicName>
```

**Exam-oriented decision guide**

Choose the tool based on the question you need to answer:

- **“Can this name resolve?”**
  - Use `nslookup`, `Resolve-DnsName`, and private DNS checks.
- **“Is this packet allowed?”**
  - Use IP flow verify and effective NSG rules.
- **“Which route will Azure use?”**
  - Use effective routes and next hop.
- **“Is the connection actually reaching the destination?”**
  - Use connection troubleshoot and packet capture.
- **“Is the backend healthy behind a load balancer?”**
  - Check probes, backend pool membership, and NAT rules.
- **“Is the private endpoint path correct?”**
  - Check DNS, private endpoint state, and service access settings.
- **“Is the hybrid route advertised and learned?”**
  - Check BGP, gateway status, and route propagation.

**Key takeaways**

- Start with DNS, then routing, then security, then platform-specific components.
- Use Network Watcher to prove the path, not just guess it.
- Distinguish between a policy deny, a routing issue, and an application refusal.
- Combine effective routes, effective NSG, connection troubleshoot, and packet capture for a complete diagnosis.
- Remember that Azure networking problems often involve multiple layers at once.
