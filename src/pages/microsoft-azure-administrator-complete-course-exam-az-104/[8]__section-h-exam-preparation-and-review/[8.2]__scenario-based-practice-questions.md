## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**How to approach AZ-104 scenarios**

AZ-104 scenario questions test whether you can choose the right Azure service, scope, and configuration under constraints such as cost, downtime, security, and manageability. The best answer is often the one that satisfies the requirement with the least operational complexity.

Focus on these decision signals:

- **Scope**: subscription, resource group, resource, management group, or tenant.
- **Access model**: Azure role-based access control (Azure RBAC), Microsoft Entra ID, or data-plane permissions.
- **Network path**: public endpoint, service endpoint, private endpoint, VPN, ExpressRoute, or internal load balancer.
- **Operational impact**: restart required, downtime, replication delay, or policy enforcement timing.
- **Cost and supportability**: choose the simplest service that meets the requirement.

> If a question says “least administrative effort,” prefer built-in Azure features over custom automation unless automation is explicitly required.

**Identity and access scenarios**

Microsoft Entra ID is the identity plane for users, groups, guests, and role assignments. In AZ-104 scenarios, you often need to decide whether to assign permissions to a user, a group, or a managed identity.

Key patterns:

- **User lifecycle**
  - Create users for internal staff.
  - Use guest users for external collaboration.
  - Disable accounts instead of deleting them when you need to preserve audit history.
- **Group lifecycle**
  - Use security groups to simplify repeated access assignments.
  - Prefer group-based role assignment for teams that change over time.
- **Administrative units**
  - Use administrative units when delegated admins should manage only a subset of users or groups.
- **Role assignment scope**
  - Assign at the narrowest scope that satisfies the requirement.
  - Use subscription scope for broad administration.
  - Use resource group scope for team-owned workloads.
  - Use resource scope for a single resource.

Practical CLI examples:

```bash
# Create a user
az ad user create \
  --display-name "Ava Patel" \
  --user-principal-name ava.patel@contoso.com \
  --password "Use-A-Strong-Password-Here"

# Create a security group
az ad group create \
  --display-name "AppOps-Readers" \
  --mail-nickname AppOpsReaders

# Add a member to a group
az ad group member add \
  --group AppOps-Readers \
  --member-id <user-object-id>

# Assign a role at resource group scope
az role assignment create \
  --assignee-object-id <user-or-group-object-id> \
  --role "Reader" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-app
```

Important exam nuance:

- **Assignee resolution** can fail if you use a UPN that is not resolvable in the current tenant context.
- **Object ID** is the safest choice in CLI examples.
- **Guest users** may require careful tenant context and invitation status before role assignment.
- **Data-plane access** is separate from management-plane access for services like Storage and Key Vault.

**Storage scenarios**

Storage questions often test account type, redundancy, access tier, and authorization model. Choose the storage configuration based on durability, performance, and access pattern.

Common decisions:

- **Redundancy**
  - **LRS**: lowest cost, copies within one datacenter.
  - **ZRS**: higher resilience within a region.
  - **GRS**: asynchronous replication to a paired region.
  - **GZRS**: combines zone resilience with geo-replication.
- **Access tiers**
  - **Hot** for frequent access.
  - **Cool** for infrequent access with lower storage cost.
  - **Archive** for long-term retention with rehydration delay.
- **Blob access**
  - Use private containers for sensitive data.
  - Use anonymous access only when public read is explicitly required.
- **Authorization**
  - Use Microsoft Entra ID and Azure RBAC for blob data access when supported.
  - Use shared access signatures (SAS) for time-limited delegated access.
  - Use storage account keys only when necessary, because they grant broad access.

Example storage account creation:

```bash
az storage account create \
  --name mystorageacct01 \
  --resource-group rg-data \
  --location eastus \
  --sku Standard_GZRS \
  --kind StorageV2 \
  --access-tier Hot \
  --allow-blob-public-access false
```

Blob access example:

```bash
# Upload a blob using Entra ID auth
az storage blob upload \
  --account-name mystorageacct01 \
  --container-name appdata \
  --name report.csv \
  --file ./report.csv \
  --auth-mode login
```

Scenario guidance:

- Use **Azure Files** when you need SMB file shares for lift-and-shift apps or shared drive scenarios.
- Use **blob storage** for object data, logs, backups, and static content.
- Use **file shares** when multiple VMs or users need shared file access.
- Use **private endpoints** when the requirement is private network access to the storage service, but remember that some scenarios can also use firewall rules or service endpoints depending on the service and design.

> For blob data access with Entra ID, the storage account must support Azure AD authorization for data-plane access, and the application must use the correct SDK or endpoint.

**Compute and VM scenarios**

Compute questions often ask you to choose between a VM, VM scale set, container service, or platform service. For AZ-104, the most common compute scenarios involve virtual machines, images, extensions, and scale sets.

Use a VM when:

- You need full OS control.
- You must install legacy software.
- You need custom networking or agent-level configuration.

Use a VM scale set when:

- You need horizontal scaling.
- You want identical instances.
- You need autoscale based on metrics or schedules.

Use an image when:

- You want repeatable deployments.
- You need a standardized OS and software baseline.

Use cloud-init or extensions when:

- You need first-boot configuration.
- You want to install software or run scripts after provisioning.

Example: Custom Script Extension on Linux VM

```bash
az vm extension set \
  --resource-group rg-app \
  --vm-name vm01 \
  --name CustomScript \
  --publisher Microsoft.Azure.Extensions \
  --settings '{
    "fileUris": ["https://contoso.blob.core.windows.net/scripts/setup.sh"],
    "commandToExecute": "bash setup.sh"
  }'
```

Example: VM scale set with autoscale concept

```bash
az vmss create \
  --resource-group rg-web \
  --name webss01 \
  --image Ubuntu2204 \
  --instance-count 2 \
  --upgrade-policy-mode automatic
```

Scenario clues:

- Use **cloud-init** for Linux bootstrap tasks that should run during provisioning.
- Use **VM extensions** for post-deployment configuration, monitoring agents, or custom scripts.
- Use **managed disks** for persistent VM storage.
- Use **availability zones** when the requirement is zone-level resilience within a region.
- Use **availability sets** when the requirement is fault domain and update domain separation for classic VM high availability.

**Networking scenarios**

Networking questions often hinge on traffic flow, name resolution, segmentation, and security controls. Think in layers: subnet design, filtering, routing, and service exposure.

Core concepts:

- **NSG (Network Security Group)**: stateful packet filtering at subnet or NIC level.
- **ASG (Application Security Group)**: logical grouping of VMs for simpler NSG rules.
- **VNet peering**: private connectivity between VNets.
- **Azure Firewall**: centralized stateful network security and logging.
- **NAT Gateway**: outbound internet access with stable public IPs.
- **Load Balancer**: Layer 4 distribution with health probes.
- **Application Gateway**: Layer 7 HTTP/HTTPS routing with optional WAF (Web Application Firewall).
- **Private Endpoint**: private IP access to supported PaaS services.
- **VPN Gateway**: encrypted site-to-site or point-to-site connectivity.
- **ExpressRoute**: private dedicated connectivity to Azure.

NSG and ASG example:

```bash
az network nsg create \
  --resource-group rg-net \
  --name nsg-web

az network asg create \
  --resource-group rg-net \
  --name asg-web

az network nsg rule create \
  --resource-group rg-net \
  --nsg-name nsg-web \
  --name Allow-HTTPS \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes Internet \
  --destination-port-ranges 443
```

Peering example:

```bash
az network vnet peering create \
  --resource-group rg-net \
  --name vnet1-to-vnet2 \
  --vnet-name vnet1 \
  --remote-vnet vnet2 \
  --allow-vnet-access
```

DNS and routing scenario notes:

- Use **Azure-provided DNS** for simple environments.
- Use **custom DNS** when you need on-premises name resolution or split-horizon behavior.
- Use **private DNS zones** with private endpoints so clients resolve the private IP.
- Use **user-defined routes** when you need to force traffic through a firewall or NVA (network virtual appliance).

> A private endpoint alone does not solve name resolution. You usually also need the correct private DNS zone and VNet link.

Load balancer health probe example:

```bash
az network lb probe create \
  --resource-group rg-net \
  --lb-name lb-web \
  --name http-probe \
  --protocol Http \
  --port 80 \
  --path /health
```

**Governance and policy scenarios**

Governance questions test whether you can enforce standards without breaking deployments. Use the least disruptive control that still meets the requirement.

Main tools:

- **Azure Policy** for compliance and enforcement.
- **Management locks** for protection against accidental deletion or modification.
- **Tags** for cost allocation and organization.
- **RBAC** for access control.
- **Blueprint-style governance** concepts may appear in older materials, but Azure Policy and management groups are the primary modern controls.

Policy behavior:

- **Audit** reports noncompliance without blocking.
- **Deny** blocks noncompliant requests.
- **DeployIfNotExists** can remediate missing resources or settings.
- **Modify** can change request properties before deployment when supported.

Example policy assignment pattern:

```bash
az policy assignment create \
  --name require-tags \
  --scope /subscriptions/<sub-id> \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policy-definition-name> \
  --params '{
    "tagName": { "value": "CostCenter" }
  }'
```

Use this pattern carefully:

- Make sure the policy definition ID is valid.
- Use an initiative when multiple related policies must be assigned together.
- Use remediation tasks for existing resources when the policy supports it.

Management lock nuance:

- A **CanNotDelete** lock prevents deletion of the locked scope and child resources where applicable.
- A **ReadOnly** lock blocks most write operations.
- Scope matters:
  - At **resource group** scope, the lock affects the group and contained resources.
  - At **resource** scope, the lock affects only that resource.
- Locks do not replace RBAC; they add protection on top of permissions.

Example:

```bash
az lock create \
  --name rg-protect \
  --lock-type CanNotDelete \
  --resource-group rg-prod
```

**Monitoring and alerting scenarios**

Monitoring questions often ask whether to use metrics, logs, diagnostic settings, or alerts. Choose based on data type and response time.

Use **metrics** when:

- You need near-real-time numeric signals.
- You want threshold-based alerting on CPU, latency, or availability.

Use **logs** when:

- You need queryable event history.
- You need correlation across resources.
- You need detailed troubleshooting.

Use **diagnostic settings** when:

- You want to send platform logs and metrics to Log Analytics, Storage, or Event Hub.

Diagnostic settings example:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm01 \
  --workspace /subscriptions/<sub-id>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law01 \
  --logs '[{"category":"Administrative","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Alerting guidance:

- Use **metric alerts** for fast response to numeric thresholds.
- Use **log alerts** for complex conditions, correlation, or when the signal exists only in logs.
- Set the alert **scope** correctly; a wrong scope is a common exam trap.
- Use **action groups** to notify email, SMS, push, webhook, or automation targets.

Example metric alert pattern:

```bash
az monitor metrics alert create \
  --name cpu-high \
  --resource-group rg-monitor \
  --scopes /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm01 \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --action-group /subscriptions/<sub-id>/resourceGroups/rg-monitor/providers/microsoft.insights/actionGroups/ag01
```

**Backup and disaster recovery scenarios**

Backup and disaster recovery questions test whether you can protect data and restore service with the right recovery objective.

Key terms:

- **RPO (Recovery Point Objective)**: acceptable data loss window.
- **RTO (Recovery Time Objective)**: acceptable downtime window.
- **Backup vault / Recovery Services vault**: stores backup configuration and recovery points.
- **Site Recovery**: orchestrates replication and failover for workloads.

Backup decisions:

- Use backup for accidental deletion, corruption, and point-in-time restore.
- Use Site Recovery for regional outage recovery and workload failover.
- Use snapshots for quick disk-level recovery, but do not treat them as a full backup strategy unless the scenario explicitly allows it.

Example backup policy concept:

- Daily backup for standard workloads.
- Short retention for operational recovery.
- Longer retention for compliance or archival needs.

Site Recovery prerequisites to remember:

- Source and target regions must support the workload.
- Network mapping and target resources must exist.
- Replication must be enabled before failover testing.
- You should test failover before a real disaster.

Scenario clue:

- If the requirement is “restore a deleted file from yesterday,” use backup recovery.
- If the requirement is “keep the app running after a regional outage,” use replication and failover.

**End-to-end scenario patterns**

These patterns help you choose the right answer quickly.

1. **Private access to a PaaS service**
   - Use a private endpoint if the service supports it.
   - Add the private DNS zone.
   - Disable public network access when required.
   - Verify client DNS resolution.

2. **Team-based administration**
   - Create a security group.
   - Assign the role to the group at the narrowest scope.
   - Add and remove users from the group instead of changing role assignments repeatedly.

3. **Secure blob access from an app**
   - Use a managed identity for the app.
   - Grant a data-plane role such as `Storage Blob Data Contributor`.
   - Ensure the storage account supports Entra ID authorization for blob access.
   - Use the correct SDK and endpoint.

4. **Standardized VM deployment**
   - Build a custom image or use a marketplace image.
   - Apply cloud-init or extensions for bootstrap tasks.
   - Use an ARM template, Bicep, or automation for repeatability.

5. **Protect production resources**
   - Apply a `CanNotDelete` lock at the appropriate scope.
   - Use RBAC for access control.
   - Use Azure Policy for configuration compliance.
   - Use tags for ownership and cost tracking.

**Practice question patterns and decision cues**

Use these cues to eliminate distractors:

- If the question says **“least privilege”**, choose the narrowest scope and the smallest role that works.
- If the question says **“no public internet exposure”**, consider private endpoint, VPN, ExpressRoute, or internal load balancer depending on the service.
- If the question says **“automate repeated deployments”**, prefer templates, images, or scale sets.
- If the question says **“monitor without blocking”**, use audit policy or alerts rather than deny.
- If the question says **“protect from deletion”**, use a lock, not just RBAC.
- If the question says **“shared access for multiple VMs”**, consider Azure Files or a shared data service.
- If the question says **“fast failover”**, think replication and pre-provisioned target resources.

**Common exam traps**

- Confusing **management-plane** permissions with **data-plane** permissions.
- Choosing a **private endpoint** when the service or requirement actually fits a firewall rule, service endpoint, or routing control.
- Using a **role assignment** at too broad a scope.
- Assuming a **lock** prevents all changes.
- Treating **audit** and **deny** as if they can be combined in one policy rule without an initiative or separate assignments.
- Forgetting that **diagnostic settings** must be configured per resource or resource type.
- Selecting **metric alerts** when the signal exists only in logs.
- Ignoring **DNS** when private connectivity is involved.

**Quick reference checklist**

- Identify the service and plane: management or data.
- Check whether the requirement is about access, network, compliance, monitoring, or recovery.
- Choose the narrowest scope that satisfies the scenario.
- Verify whether the service supports the feature you want.
- Prefer built-in Azure controls before custom automation.
- Confirm dependencies such as DNS, tenant context, replication, or authorization mode.

**Mini scenario drills**

- A developer needs read-only access to one storage account for blob data.
  - Use a data-plane role at the storage account scope.
  - Prefer a group or managed identity if the access is recurring.

- A web app must reach a database privately.
  - Use the service’s supported private connectivity option.
  - Add DNS resolution and network rules.
  - Do not assume every backend supports the same private access model.

- A production VM must not be deleted accidentally.
  - Apply a `CanNotDelete` lock at the VM or resource group scope depending on what must be protected.
  - Remember that scope determines what the lock blocks.

- A team needs to manage only users in one department.
  - Use an administrative unit and delegated admin roles.
  - Keep the scope limited to that department.

- You need to know when a VM CPU spike occurred and who changed the VM.
  - Use metrics for CPU.
  - Use activity logs or diagnostic logs for change history.
  - Route logs to Log Analytics for querying.

**Final exam strategy**

Read the requirement first, then map it to the Azure control plane:

- **Identity** for who can do it.
- **Network** for how traffic reaches it.
- **Governance** for what is allowed.
- **Monitoring** for how you detect it.
- **Backup/DR** for how you recover it.
- **Compute/Storage** for where the workload lives.

If two answers both seem possible, choose the one that best matches the stated constraint, not the one with the most features.
