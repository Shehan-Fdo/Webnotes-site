## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Purpose of this checklist**

This topic helps you assess whether you are ready for the AZ-104 exam and where to focus your final review. The goal is not just to recognize concepts, but to prove that you can perform common Azure administration tasks in the portal, Azure CLI, PowerShell, and ARM/Bicep-based workflows.

A strong readiness assessment combines three signals:

- **Knowledge**: you understand the service, feature, and limitation.
- **Execution**: you can configure it correctly in a lab.
- **Judgment**: you can choose the right Azure service or setting for the scenario.

**AZ-104 readiness mindset**

AZ-104 emphasizes day-to-day Azure administration. You should expect scenario-based questions that test whether you can:

- Create and manage identities, subscriptions, and governance controls.
- Deploy and monitor compute, storage, networking, and virtual machines.
- Secure resources with role-based access control and policy.
- Troubleshoot access, connectivity, and resource configuration issues.
- Use the Azure portal, CLI, and PowerShell to complete tasks efficiently.

A good self-check is whether you can explain not only *how* to configure something, but also *why* you would choose one option over another.

**Core exam domains to validate**

Use the exam skills outline as a checklist, then verify each area in a lab.

- **Identity and governance**
  - Manage Microsoft Entra ID users, groups, and role assignments.
  - Apply Azure RBAC at the correct scope.
  - Use Azure Policy to enforce standards.
  - Work with subscriptions, resource groups, and locks.
- **Storage**
  - Configure storage accounts, redundancy, access tiers, and networking.
  - Secure blob access with SAS, access keys, or Microsoft Entra-based authorization where appropriate.
  - Manage file shares, containers, and lifecycle rules.
- **Compute**
  - Deploy and manage virtual machines, VM scale sets, and availability options.
  - Configure extensions, disks, and backup.
  - Understand when to use App Service, containers, or VMs at a high level.
- **Networking**
  - Configure virtual networks, subnets, NSGs, route tables, and peering.
  - Troubleshoot DNS, private access, and connectivity.
  - Understand load balancing and application delivery options.
- **Monitoring and backup**
  - Use Azure Monitor, Log Analytics, alerts, and metrics.
  - Interpret diagnostic logs and activity logs.
  - Configure backup and recovery for supported workloads.

**What you should be able to do in the portal**

If you can only describe a feature but cannot perform it, you are not fully ready. Practice these tasks until they feel routine:

- Create a resource group and apply a lock.
- Assign a built-in role to a user or group at the correct scope.
- Create a storage account and restrict network access.
- Deploy a VM from an image and connect securely.
- Create a virtual network with subnets and attach an NSG.
- Configure a basic alert rule and verify it triggers.
- Review activity logs and resource diagnostics.

**CLI and PowerShell readiness checks**

You do not need to memorize every command, but you should be comfortable reading and adapting common commands.

```bash
# Azure CLI examples
az login
az account show
az group create --name rg-az104-lab --location eastus
az network vnet create \
  --resource-group rg-az104-lab \
  --name vnet-lab \
  --address-prefix 10.10.0.0/16 \
  --subnet-name subnet-app \
  --subnet-prefix 10.10.1.0/24

az role assignment create \
  --assignee user@contoso.com \
  --role "Reader" \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-az104-lab
```

```powershell
# Azure PowerShell examples
Connect-AzAccount
Get-AzContext
New-AzResourceGroup -Name rg-az104-lab -Location eastus
New-AzVirtualNetwork -Name vnet-lab -ResourceGroupName rg-az104-lab -Location eastus -AddressPrefix 10.10.0.0/16
New-AzRoleAssignment -SignInName user@contoso.com -RoleDefinitionName Reader -ResourceGroupName rg-az104-lab
```

You should know when to use each tool:

- **Azure CLI** is often faster for repeatable, cross-platform administration.
- **Azure PowerShell** is useful when you prefer object-based output and Microsoft-centric automation.
- **Portal** is best for visual validation, quick troubleshooting, and learning service relationships.

**Infrastructure as Code awareness**

AZ-104 is not an IaC-heavy exam, but you should understand the basics of deploying and reviewing templates.

- **ARM** templates define Azure resources declaratively in JSON.
- **Bicep** is a simpler language that compiles to ARM templates.
- You should recognize common deployment patterns:
  - Parameterized resource groups and locations.
  - Reusable modules.
  - Outputs for resource IDs, names, and endpoints.

Example Bicep snippet:

```bicep
param location string = resourceGroup().location
param vnetName string = 'vnet-lab'

resource vnet 'Microsoft.Network/virtualNetworks@2023-11-01' = {
  name: vnetName
  location: location
  properties: {
    addressSpace: {
      addressPrefixes: [
        '10.10.0.0/16'
      ]
    }
  }
}
```

**Identity and access readiness**

This area is frequently tested because it affects nearly every Azure service.

- **Microsoft Entra ID** manages identities, groups, and authentication.
- **Azure RBAC** controls what a principal can do at a scope.
- **Scopes** include management group, subscription, resource group, and resource.
- **Least privilege** means you assign only the permissions required for the task.

You should be able to distinguish these common concepts:

- **Authentication** answers “Who are you?”
- **Authorization** answers “What can you do?”
- **Role assignment** grants permissions at a specific scope.
- **Custom roles** are useful when built-in roles are too broad.

Practical validation tasks:

- Assign `Reader`, `Contributor`, or `Virtual Machine Contributor` at different scopes.
- Verify inherited permissions from parent scopes.
- Use `az role assignment list` or the portal to confirm effective access.

```bash
az role assignment list \
  --assignee user@contoso.com \
  --all \
  --output table
```

**Governance and compliance readiness**

You should understand how Azure governance tools work together.

- **Management groups** organize subscriptions.
- **Azure Policy** enforces or audits configuration.
- **Initiatives** group multiple policies into one assignment.
- **Resource locks** prevent accidental deletion or modification.
- **Tags** help with cost allocation, ownership, and automation.

A practical governance checklist:

- Apply a tag standard such as `Environment`, `Owner`, and `CostCenter`.
- Create a policy assignment that audits allowed locations.
- Use a lock on critical resource groups.
- Confirm that policy compliance appears in the portal.

> Be careful not to confuse Azure Policy with Azure RBAC. Policy evaluates resource configuration; RBAC controls user permissions.

**Networking readiness**

Networking questions often require you to reason through traffic flow.

You should be able to explain:

- How a **virtual network** isolates resources.
- How **subnets** segment workloads.
- How an **NSG** filters traffic using rules.
- How **peering** connects VNets privately.
- How **DNS** affects name resolution and private endpoints.

Common lab tasks:

- Create a VNet with separate app and data subnets.
- Attach an NSG and allow only required ports.
- Test connectivity with `ping`, `Test-NetConnection`, or `az network watcher`.
- Verify route behavior when using user-defined routes or peering.

```powershell
Test-NetConnection 10.10.1.4 -Port 3389
```

```bash
az network watcher test-connectivity \
  --source-resource <vm-resource-id> \
  --dest-address 10.10.1.4 \
  --dest-port 3389
```

You should also know the purpose of:

- **Private endpoints** for private access to PaaS services.
- **Service endpoints** for optimized service access over the Azure backbone.
- **Azure Load Balancer** for Layer 4 distribution.
- **Application Gateway** for Layer 7 routing and web application delivery.

**Storage readiness**

Storage questions often test both configuration and access control.

You should understand:

- **Storage account types** and redundancy options.
- **Blob**, **file**, **queue**, and **table** services at a high level.
- **Access tiers** such as hot, cool, and archive for blob data.
- **Shared access signatures (SAS)** for delegated access.
- **Lifecycle management** for cost optimization.

Practical checks:

- Create a storage account with the correct redundancy for the scenario.
- Restrict public access when required.
- Upload and download a blob using the portal or CLI.
- Configure a file share and map it from a client if needed.
- Review how network rules affect access.

```bash
az storage account create \
  --name staz104lab01 \
  --resource-group rg-az104-lab \
  --location eastus \
  --sku Standard_LRS \
  --kind StorageV2
```

```bash
az storage container create \
  --account-name staz104lab01 \
  --name data \
  --auth-mode login
```

**Compute readiness**

You should be able to deploy and manage virtual machines confidently.

Focus on:

- VM sizing and workload fit.
- OS disk versus data disk usage.
- Availability sets and availability zones.
- VM extensions for configuration and monitoring.
- Backup and restore basics.

Validation tasks:

- Deploy a Linux or Windows VM.
- Connect using SSH or RDP securely.
- Resize the VM and observe the effect.
- Attach a data disk and format it.
- Install a monitoring or custom script extension.

```bash
az vm create \
  --resource-group rg-az104-lab \
  --name vm01 \
  --image Ubuntu2204 \
  --admin-username azureuser \
  --generate-ssh-keys
```

You should also know when a VM is the right choice versus a platform service:

- Use a **VM** when you need full OS control or legacy compatibility.
- Use **App Service** when you want managed web hosting.
- Use **containers** when portability and packaging matter.
- Use **VM scale sets** when you need repeatable horizontal scaling.

**Monitoring and troubleshooting readiness**

Monitoring is not just about collecting data; it is about proving what happened and responding quickly.

You should be comfortable with:

- **Azure Monitor** for platform-wide observability.
- **Metrics** for near-real-time numeric data.
- **Logs** for detailed event and diagnostic records.
- **Log Analytics** for querying and correlating data.
- **Alerts** for proactive notification and automation.

Useful Kusto Query Language (KQL) example:

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where ActivityStatusValue == "Failed"
| project TimeGenerated, Caller, OperationNameValue, ResourceGroup, ActivityStatusValue
| order by TimeGenerated desc
```

Practical troubleshooting workflow:

1. Check the **activity log** for control-plane changes.
2. Review **metrics** for service health or saturation.
3. Inspect **diagnostic settings** and send logs to Log Analytics.
4. Validate network paths, DNS, and NSG rules.
5. Confirm identity and RBAC assignments if access fails.

**Backup and recovery readiness**

You should understand the difference between protecting data and protecting workloads.

- **Azure Backup** protects supported workloads and VMs.
- **Recovery Services vaults** store backup data and policies.
- **Backup policies** define schedule and retention.
- **Restore points** help you recover from deletion, corruption, or accidental changes.

Check that you can:

- Create a vault and enable backup for a VM.
- Run an on-demand backup.
- Restore files or a full VM when required.
- Explain retention and recovery point objectives at a basic level.

**Hands-on lab checklist**

Use a lab subscription and verify each item without looking up the steps.

- Create and delete a resource group.
- Assign and remove a role assignment.
- Apply a policy assignment and check compliance.
- Create a VNet, subnet, and NSG.
- Deploy a VM and connect to it.
- Create a storage account and container.
- Configure a diagnostic setting and query logs.
- Create an alert rule and test notification delivery.
- Enable backup for a VM and review recovery options.

If you cannot complete a task from memory, revisit the service and repeat the lab until the workflow feels predictable.

**Readiness self-assessment rubric**

Use this simple scoring model to identify weak areas.

| Area | Can explain | Can configure in lab | Can troubleshoot |
|---|---:|---:|---:|
| Identity and RBAC | Yes/No | Yes/No | Yes/No |
| Governance and policy | Yes/No | Yes/No | Yes/No |
| Networking | Yes/No | Yes/No | Yes/No |
| Storage | Yes/No | Yes/No | Yes/No |
| Compute | Yes/No | Yes/No | Yes/No |
| Monitoring | Yes/No | Yes/No | Yes/No |
| Backup and recovery | Yes/No | Yes/No | Yes/No |

Interpretation:

- **Mostly yes** across all rows means you are likely ready for final review.
- **Several no answers** in one domain means you need targeted lab practice.
- **Troubleshoot gaps** are especially important because scenario questions often hide the root cause.

**Common readiness gaps**

These are frequent weak spots for AZ-104 candidates:

- Confusing **RBAC** with **Azure Policy**.
- Forgetting scope inheritance for role assignments.
- Misconfiguring NSGs and assuming traffic is blocked elsewhere.
- Not understanding storage network rules versus access keys.
- Choosing the wrong VM size or availability option for the scenario.
- Skipping diagnostic settings and then lacking evidence during troubleshooting.
- Knowing the portal path but not the CLI or PowerShell equivalent.

**Exam-day strategy**

Your final review should focus on accuracy and speed, not cramming new material.

- Read the scenario carefully and identify the resource scope.
- Look for keywords such as `least privilege`, `private access`, `high availability`, or `cost optimization`.
- Eliminate answers that solve the wrong problem.
- Prefer the simplest service that meets the requirement.
- If a question asks for a management task, think in terms of Azure administration, not application development.

A useful final check is whether you can answer these questions confidently:

- Can you assign access at the correct scope?
- Can you secure a storage account without breaking required access?
- Can you build and validate a network path?
- Can you monitor and troubleshoot a resource using logs and metrics?
- Can you explain the difference between a control-plane change and a data-plane operation?

**Final readiness checklist**

Before scheduling the exam, confirm that you can do the following without assistance:

- Navigate the Azure portal efficiently.
- Use Azure CLI and PowerShell for common admin tasks.
- Interpret RBAC, policy, and diagnostic settings.
- Deploy and manage core Azure resources.
- Troubleshoot access, networking, and monitoring issues.
- Explain service selection decisions in scenario-based terms.

**Recommended final review routine**

1. Revisit weak domains from your rubric.
2. Rebuild one small lab for identity, networking, storage, and compute.
3. Review monitoring and backup workflows.
4. Practice a few CLI and PowerShell commands from memory.
5. Take a timed practice exam and review every missed question.
6. Repeat the lab steps that caused hesitation.

**What readiness looks like**

You are ready when you can move from a scenario description to a correct Azure configuration without relying on hints. The exam rewards practical understanding, so your final preparation should prove that you can administer Azure resources safely, consistently, and with good judgment.
