## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Purpose of this review**

This topic consolidates the AZ-104 domains into a single operational view. The exam expects you to choose the right Azure service, scope, and configuration for a scenario, not just define terms. Focus on how governance, identity, networking, storage, compute, and monitoring interact in real deployments.

**AZ-104 domain map**

- **Governance and identity**: management groups, subscriptions, resource groups, RBAC, policy, locks, and tags.
- **Compute**: virtual machines, availability, disks, images, and backup/restore.
- **Networking**: VNets, subnets, DNS, routing, private access, and load balancing.
- **Storage**: storage accounts, blobs, files, access tiers, security, and lifecycle.
- **Monitoring and recovery**: Azure Monitor, Log Analytics, alerts, Service Health, Advisor, and Backup.

**Governance hierarchy and scope inheritance**

Azure governance is hierarchical. Management groups sit above subscriptions and let you apply policy and RBAC at scale. Subscriptions contain resource groups, and resource groups contain resources. Permissions and policy assignments can inherit downward, so a decision at a higher scope affects everything below it unless you explicitly exclude or override it.

<dl>
<dt>Management group</dt>
<dd>A container for subscriptions used to organize governance and apply inherited policy or RBAC.</dd>
<dt>Subscription</dt>
<dd>The billing and access boundary for Azure resources.</dd>
<dt>Resource group</dt>
<dd>A logical container for resources that usually share lifecycle and access patterns.</dd>
</dl>

A common exam pattern is deciding where to apply a control:

- Use a **management group** when you need consistent governance across many subscriptions.
- Use a **subscription** when the boundary is billing, quota, or environment-wide access.
- Use a **resource group** when the control applies to a workload or app stack.
- Use a **resource** scope when only one object needs the setting.

**Role-based access control**

Role-based access control (RBAC) answers the question: who can do what, and where? Azure uses role assignments at a scope. Built-in roles cover common tasks, while custom roles let you narrow permissions.

Key RBAC concepts:

- **Role definition**: the permission set, such as Reader or Contributor.
- **Role assignment**: the binding of a principal to a role at a scope.
- **Scope**: management group, subscription, resource group, or resource.
- **Inheritance**: permissions flow downward from parent scopes.

Practical examples:

```bash
# Assign a built-in role at a resource group scope
az role assignment create \
  --assignee user@contoso.com \
  --role Reader \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg-name>
```

```bash
# View role assignments at a subscription
az role assignment list \
  --scope /subscriptions/<sub-id> \
  --output table
```

Use RBAC when you need access control. Use policy when you need to enforce configuration. Use locks when you need to protect against accidental deletion or modification.

**Azure resource locks**

Resource locks are a simple but important governance control.

- **CanNotDelete** prevents deletion but allows reads and updates.
- **ReadOnly** prevents changes and deletion, but still allows read operations.

Locks are useful for shared resources, critical networking components, and production systems where accidental removal is a risk. They are not a substitute for RBAC because a user with sufficient permissions can still manage locks depending on scope and rights.

Example:

```bash
# Create a delete lock on a resource group
az lock create \
  --name rg-protect-lock \
  --lock-type CanNotDelete \
  --resource-group <rg-name>
```

> A ReadOnly lock can break normal operations such as scaling, patching, or configuration changes. Use it only when you truly want to freeze the resource.

**Azure Policy and compliance**

Azure Policy evaluates resources against rules. It helps you enforce standards such as allowed locations, required tags, secure transfer, or approved SKUs. Policy can deny noncompliant deployments, audit existing resources, or deploy missing settings.

Common policy effects:

- **Deny** blocks noncompliant requests.
- **Audit** records noncompliance without blocking.
- **Append** adds fields to requests when possible.
- **DeployIfNotExists** deploys a related resource or configuration.
- **Modify** changes request properties before deployment.

Policy assignment example:

```bash
az policy assignment create \
  --name require-tags \
  --display-name "Require cost center tag" \
  --policy-definition "/providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>" \
  --scope /subscriptions/<sub-id>
```

For exam purposes, remember:

- Assign policy at the highest practical scope to maximize reuse.
- Use **initiatives** when you want to group multiple policies.
- Use **remediation** for existing resources when the effect supports it.
- Use **exemptions** sparingly and document the reason.

**Tags and resource organization**

Tags are metadata, not security boundaries. They help with cost allocation, ownership, automation, and reporting.

Useful tag patterns:

- `CostCenter=FIN-001`
- `Owner=platform-team`
- `Environment=prod`
- `Application=orders`

Tags are often enforced with policy. If a scenario asks for chargeback or inventory, tags are usually the right answer. If it asks for access control, tags are not enough.

**Compute: virtual machines and administration**

Azure Virtual Machines (VMs) are IaaS (Infrastructure as a Service) compute resources. AZ-104 expects you to know how to deploy, size, resize, secure, and recover them.

Important VM administration topics:

- **Sizing**: choose CPU, memory, and storage performance based on workload.
- **Resizing**: change the VM size when workload demand changes.
- **Managed disks**: Azure-managed storage for VM disks; preferred over unmanaged disks.
- **Availability sets**: protect against datacenter failures by spreading VMs across fault and update domains.
- **Availability zones**: place VMs in physically separate zones for higher resiliency where supported.
- **Disk encryption**: use platform-managed encryption by default; consider customer-managed keys when required by policy or compliance.
- **Extensions**: install agents or automate configuration tasks.

Example resize workflow:

```bash
az vm list-sizes \
  --location eastus \
  --output table

az vm resize \
  --resource-group <rg-name> \
  --name <vm-name> \
  --size Standard_D4s_v5
```

Managed disks are the default choice because they simplify management, support snapshots and backup, and integrate with availability features. Unmanaged disks are legacy and generally not the exam-preferred answer.

**Availability sets and domains**

Availability sets distribute VMs across:

- **Fault domains**: separate physical hardware and shared power/network components.
- **Update domains**: separate maintenance groups so Azure does not reboot all VMs at once.

Use availability sets when you need high availability within a region and do not need zone-level separation. Use zones when the scenario requires stronger isolation across datacenters.

**VM images, extensions, and automation**

You can deploy VMs from marketplace images, custom images, or shared image galleries. Custom images are useful when you need a standardized baseline. Extensions help with post-deployment tasks such as monitoring, configuration management, or security agents.

Example extension install:

```bash
az vm extension set \
  --resource-group <rg-name> \
  --vm-name <vm-name> \
  --name CustomScript \
  --publisher Microsoft.Azure.Extensions \
  --settings '{"fileUris":["https://<storage>.blob.core.windows.net/scripts/setup.sh"],"commandToExecute":"bash setup.sh"}'
```

**VM backup and restore**

Azure Backup protects VMs with Recovery Services vaults. The vault stores backup data and policies. Backup behavior depends on workload and configuration, so do not assume every restore option is available in every scenario.

Key terms:

- **Recovery Services vault**: classic vault for Azure VM backup and many workload backups.
- **Backup policy**: defines schedule and retention.
- **Instant restore**: keeps recent recovery points available for faster restore operations.
- **Soft delete**: protects backup data from accidental deletion for a retention period.

Typical restore choices include:

- Restore the entire VM.
- Restore disks and create a new VM.
- Restore files in supported scenarios.
- Restore application-consistent data when the workload supports it.

> File-level restore is not universal across all workloads and operating systems. Check the backup scenario before choosing it as the answer.

**Storage accounts and security settings**

Storage accounts are the foundation for blobs, files, queues, and tables. AZ-104 often tests which account settings improve security and compatibility.

Important account-level settings:

- **Access tier**: Hot, Cool, or Archive for blob data.
- **Secure transfer required**: forces HTTPS for requests.
- **Minimum TLS version**: limits older protocol versions.
- **Public network access**: can be restricted or disabled.
- **Replication**: LRS, ZRS, GRS, GZRS, and read-access variants depending on the scenario.

Blob access levels matter:

- **Private**: no anonymous access.
- **Blob**: anonymous read access to blobs only.
- **Container**: anonymous read access to the container and blobs.

Example storage account creation with secure defaults:

```bash
az storage account create \
  --name mystorageacct123 \
  --resource-group <rg-name> \
  --location eastus \
  --sku Standard_GRS \
  --kind StorageV2 \
  --https-only true \
  --min-tls-version TLS1_2 \
  --allow-blob-public-access false
```

Use the access tier to optimize cost:

- **Hot** for frequent access.
- **Cool** for infrequent access with lower storage cost.
- **Archive** for long-term retention with rehydration delay.

**Blob lifecycle management**

Lifecycle management automates tier changes and deletion based on age or last access. This is useful for cost control and retention policies.

Example lifecycle policy:

```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-old-blobs-to-cool",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool": {
              "daysAfterModificationGreaterThan": 30
            },
            "delete": {
              "daysAfterModificationGreaterThan": 365
            }
          }
        }
      }
    }
  ]
}
```

This is the kind of configuration you use when the scenario asks for automated retention or cost optimization without manual intervention.

**Azure Files and identity-based access**

Azure Files supports SMB and, in some scenarios, NFS. The authentication model depends on the protocol and environment.

SMB identity options include:

- **Microsoft Entra Kerberos** for modern identity-based access in supported environments.
- **Active Directory Domain Services (AD DS)** integration for traditional domain-joined access.
- **Microsoft Entra Domain Services** for managed domain scenarios.

NFS access is different:

- NFS uses Unix-style permissions and network-based access patterns.
- NFS does not use the same SMB identity-based mount flow.

Practical distinction:

- Use **SMB** when you need Windows-style file sharing or identity-based access with AD integration.
- Use **NFS** when you need Linux/Unix semantics and supported NFS workloads.

Example SMB mount on Linux using a storage account key:

```bash
sudo mkdir -p /mnt/azfiles
sudo mount -t cifs //<storage-account>.file.core.windows.net/<share-name> /mnt/azfiles \
  -o vers=3.0,username=<storage-account>,password=<storage-key>,dir_mode=0777,file_mode=0777,serverino
```

> This is an SMB/CIFS mount pattern. Do not confuse it with NFS, which uses different mount options and authentication behavior.

**Networking fundamentals**

Azure networking questions often test how traffic flows between subnets, private services, and the internet. Start with the virtual network (VNet), then layer on routing, DNS, and security.

Core concepts:

- **VNet**: isolated private network in Azure.
- **Subnet**: address range within a VNet.
- **NSG**: network security group for stateful filtering.
- **UDR**: user-defined route for custom next-hop behavior.
- **DNS**: name resolution for private and public endpoints.

**Subnet delegation, service endpoints, and private endpoints**

These are commonly confused, so separate them clearly.

| Feature | What it does | Typical use |
|---|---|---|
| Subnet delegation | Grants a subnet to a specific Azure service | App Service Environment, Azure Container Instances, delegated PaaS integrations |
| Service endpoint | Extends VNet identity to a service over the Azure backbone | Secure access to storage or SQL from a subnet |
| Private endpoint | Gives a private IP in your VNet for a PaaS service | Private access to storage, SQL, Key Vault, and similar services |

Use a **service endpoint** when you want to keep traffic on the Azure backbone and restrict access by VNet/subnet. Use a **private endpoint** when you want the service reachable through a private IP in your VNet and want to eliminate public exposure.

Private endpoint DNS is a frequent exam topic. When you create a private endpoint, you usually need private DNS zone integration so the service name resolves to the private IP.

Example private DNS zone link:

```bash
az network private-dns zone create \
  --resource-group <rg-name> \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group <rg-name> \
  --zone-name privatelink.blob.core.windows.net \
  --name link-vnet1 \
  --virtual-network <vnet-name> \
  --registration-enabled false
```

**Network security groups and routing**

NSGs filter traffic at the subnet or NIC level. They are stateful, so return traffic is automatically allowed when a flow is permitted.

Use NSGs to:

- Allow required ports.
- Deny unwanted inbound traffic.
- Segment tiers such as web, app, and data.

UDRs let you steer traffic to a firewall, virtual appliance, or gateway. This is important when the scenario requires inspection or forced tunneling.

Example route table association:

```bash
az network route-table create \
  --resource-group <rg-name> \
  --name rt-spoke

az network route-table route create \
  --resource-group <rg-name> \
  --route-table-name rt-spoke \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.10.0.4
```

**Azure Firewall and forced tunneling**

Azure Firewall is a managed, stateful network security service. Use it when you need centralized control, logging, and rule management across spokes or workloads.

Typical scenario fit:

- Central egress control.
- FQDN-based filtering.
- Network and application rules.
- Hub-and-spoke inspection.

A common design is:

- Hub VNet contains Azure Firewall and shared services.
- Spoke VNets host workloads.
- UDRs send outbound traffic from spokes to the firewall.

**Load balancing and application delivery**

AZ-104 expects you to choose the right service based on traffic type and layer.

| Service | Layer | Best for |
|---|---|---|
| Azure Load Balancer | Layer 4 | TCP/UDP distribution, low latency, VM backends |
| Application Gateway | Layer 7 | HTTP/HTTPS routing, TLS termination, path-based routing, WAF |
| Front Door | Global Layer 7 | Global HTTP/HTTPS acceleration, failover, edge routing |

Important components:

- **Backend pool**: the set of targets receiving traffic.
- **Health probe**: checks whether a backend is healthy.
- **Rule**: maps frontend traffic to backend behavior.

Choose **Load Balancer** for non-HTTP workloads or when you need simple L4 distribution. Choose **Application Gateway** when you need web application routing, TLS offload, or Web Application Firewall (WAF). Choose **Front Door** when you need global entry, edge optimization, or multi-region web failover.

**Monitoring, logging, and alerting**

Azure Monitor is the umbrella service for metrics, logs, and alerts. Log Analytics workspace is the central store for queryable logs.

Core monitoring concepts:

- **Metrics**: near-real-time numeric data.
- **Logs**: detailed records stored in Log Analytics.
- **Alerts**: notifications or actions triggered by conditions.
- **Action groups**: define who gets notified and how.

Log Analytics is especially important because many Azure services can send diagnostic data there. Workspace-based logs are the modern model for centralized analysis.

Example KQL (Kusto Query Language) query:

```kusto
AzureActivity
| where ActivityStatusValue == "Failed"
| summarize Failures = count() by Caller, OperationNameValue
| order by Failures desc
```

Use KQL to:

- Filter events by time, resource, or status.
- Summarize trends.
- Correlate activity across services.
- Investigate failures and access patterns.

**Alert types and practical use**

Common alert categories:

- **Metric alerts**: trigger on numeric thresholds such as CPU or disk queue length.
- **Log alerts**: trigger from KQL query results.
- **Activity log alerts**: trigger on subscription-level control plane events.

Application Insights also supports smart detection and anomaly-style capabilities, but do not treat that as a generic Azure Monitor alert type.

Example metric alert creation:

```bash
az monitor metrics alert create \
  --name high-cpu-alert \
  --resource-group <rg-name> \
  --scopes /subscriptions/<sub-id>/resourceGroups/<rg-name>/providers/Microsoft.Compute/virtualMachines/<vm-name> \
  --condition "avg Percentage CPU > 80" \
  --description "Alert when CPU exceeds 80 percent"
```

**Workbooks and dashboards**

Azure Monitor workbooks provide interactive reports that combine logs, metrics, and parameters. Use them when you need a reusable operational view for teams or troubleshooting.

Good workbook use cases:

- VM health overview.
- Security event investigation.
- Storage capacity and transaction trends.
- Application performance summaries.

**Service Health and Advisor**

These services are often overlooked but are important for operations.

- **Service Health** tells you about Azure platform incidents, planned maintenance, and advisories that affect your resources.
- **Azure Advisor** gives recommendations across cost, security, reliability, operational excellence, and performance.

Use Service Health when the issue may be on Microsoft’s side. Use Advisor when you want improvement recommendations for your own environment.

**Backup vaults and workload protection**

Azure Backup uses different vault types depending on the workload.

- **Recovery Services vault**: traditional vault for Azure VMs and many classic backup scenarios.
- **Backup vault**: used in newer backup scenarios and some modern workloads.

For AZ-104, the key is to know that vault choice depends on the workload and backup feature set. Do not assume one vault type covers every backup scenario.

**Scenario-based decision patterns**

When you see a question, identify the control plane first:

- If the issue is about who can do something, think **RBAC**.
- If the issue is about enforcing a standard, think **Policy**.
- If the issue is about preventing deletion, think **Locks**.
- If the issue is about network reachability, think **NSG**, **UDR**, **service endpoint**, or **private endpoint**.
- If the issue is about cost or retention, think **storage tiers** or **lifecycle management**.
- If the issue is about availability, think **availability sets**, **zones**, or **load balancing**.
- If the issue is about troubleshooting, think **Azure Monitor**, **Log Analytics**, **Service Health**, or **Advisor**.

**Practical governance example**

A common enterprise pattern is:

- Apply a policy at the management group to require tags and restrict regions.
- Assign RBAC at the subscription or resource group for team access.
- Add a CanNotDelete lock to critical shared resources.
- Use private endpoints for PaaS services that should not be public.
- Send diagnostics to Log Analytics and create alerts for failures.

Example of a simple governance stack:

```text
Management group
  └─ Subscription
      ├─ Resource group: shared-network
      │   ├─ VNet
      │   ├─ Azure Firewall
      │   └─ Private DNS zones
      └─ Resource group: app-prod
          ├─ VM scale set
          ├─ Storage account
          └─ Application Gateway
```

**Exam consolidation checklist**

- Know where to apply **RBAC**, **policy**, and **locks**.
- Distinguish **management groups** from subscriptions and resource groups.
- Prefer **managed disks** and understand VM sizing and availability options.
- Separate **service endpoints** from **private endpoints** and know when DNS changes are required.
- Understand **Load Balancer**, **Application Gateway**, and **Front Door** by layer and use case.
- Know storage security settings such as **secure transfer**, **minimum TLS**, and **public access**.
- Use **Log Analytics** and **KQL** for investigation, not just metrics.
- Remember **Service Health** and **Advisor** for operational awareness.
- Treat backup and restore as scenario-dependent, especially for file-level restore and vault type.

**Final review mindset**

AZ-104 rewards practical judgment. When two answers look plausible, choose the one that matches the scope, the service layer, and the operational goal. Think in terms of control, reach, resilience, and recoverability. If you can explain why a setting belongs at a specific scope and what problem it solves, you are usually close to the correct exam answer.
