## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Troubleshooting mindset for AZ-104**

AZ-104 questions often test whether you can identify the *most likely* cause of a problem and the *best* remediation, not just a technically valid one. In Azure, multiple answers may work, but the exam usually rewards the option that is least disruptive, most cost-effective, and aligned with the service’s native capabilities.

A strong troubleshooting approach starts with scope:

- Identify whether the issue is at the **identity**, **network**, **compute**, **storage**, or **governance** layer.
- Check whether the problem is caused by **permissions**, **configuration**, or **service health**.
- Prefer the smallest change that resolves the issue.
- Validate assumptions with Azure-native tools before making broad changes.

A useful mental model is:

1. Confirm the symptom.
2. Isolate the layer.
3. Check the most common misconfiguration.
4. Choose the least disruptive fix.
5. Verify the result.

**How to read AZ-104 troubleshooting questions**

Exam questions often include extra details that are not relevant. Your job is to separate signal from noise. Look for the exact failure mode, the resource scope, and the constraint that matters most.

Common clues include:

- “Users cannot sign in” usually points to **Microsoft Entra ID** permissions, conditional access, MFA, or account state.
- “VM is unreachable” often points to **NSG**, route table, public IP, DNS, or guest OS firewall.
- “Storage access denied” may involve **RBAC**, shared key access, SAS, firewall rules, or private endpoint configuration.
- “Deployment failed” often indicates **policy**, **quota**, **location**, or **template** issues.
- “Need to minimize administrative effort” usually favors **role assignments**, **policy**, or **automation** over manual per-resource fixes.

> If two answers both work, the exam usually prefers the one that is more Azure-native, more scalable, and less disruptive.

**Identity and access troubleshooting**

Identity issues are common because Azure uses multiple authorization layers. You may need to distinguish between authentication, authorization, and directory configuration.

- **Authentication** answers the question: “Who are you?”
- **Authorization** answers the question: “What can you do?”
- **Microsoft Entra ID** is the identity provider for Azure sign-in and access control.

Typical troubleshooting areas:

- Verify the user or group has the correct **role assignment** at the right scope.
- Check whether **deny assignments** or **Azure Policy** are blocking the action.
- Confirm the user is in the correct tenant and subscription context.
- Review **Privileged Identity Management (PIM)** if the role is eligible rather than active.
- Check whether **Conditional Access** or MFA is preventing sign-in.

Example Azure CLI checks:

```bash
az account show
az role assignment list --assignee user@contoso.com --all -o table
az role assignment list --scope /subscriptions/<sub-id>/resourceGroups/<rg-name> -o table
```

Practical interpretation:

- If a user can see a resource but cannot modify it, the issue is often **RBAC**.
- If a user cannot sign in at all, the issue is often **authentication** or **Conditional Access**.
- If a deployment fails even with correct permissions, check **policy** and **resource provider registration**.

**Network troubleshooting**

Networking questions often hide the real issue behind a generic “cannot connect” statement. Start by identifying the traffic path.

Common Azure network components to check:

- **Network Security Group (NSG)** rules
- **User-defined routes (UDRs)** and route tables
- **Azure Firewall**
- **Load balancer** health probes
- **DNS** resolution
- **Private endpoints**
- **Virtual network peering**
- **Guest OS firewall**

A good sequence is:

1. Confirm the source and destination.
2. Check name resolution.
3. Check effective routes.
4. Check effective security rules.
5. Check service health and backend status.

Useful CLI commands:

```bash
az network nic show-effective-nsg --resource-group <rg-name> --name <nic-name>
az network nic show-effective-route-table --resource-group <rg-name> --name <nic-name>
az network watcher test-connectivity \
  --source-resource <vm-resource-id> \
  --dest-address 10.1.0.4 \
  --dest-port 443
```

Common exam traps:

- A **public IP** exists, but the NSG blocks inbound traffic.
- A **route table** sends traffic to a virtual appliance that is not reachable.
- DNS resolves to a **private endpoint** address, but the client is outside the private network.
- A load balancer is configured, but the backend pool is unhealthy.
- VNet peering exists, but **gateway transit** or **allow forwarded traffic** is missing.

**Storage troubleshooting**

Storage questions often involve access control, network restrictions, or the difference between data-plane and management-plane permissions.

Key distinctions:

- **Management plane**: creating or configuring the storage account.
- **Data plane**: reading or writing blobs, files, queues, or tables.

Common causes of access failure:

- Missing **RBAC** role such as Storage Blob Data Reader or Contributor.
- Storage firewall blocks the client IP or subnet.
- Private endpoint is configured, but DNS is not resolving correctly.
- Shared key access is disabled, but the client still uses account keys.
- SAS token is expired or lacks required permissions.

Example checks:

```bash
az storage account show --name <storageacct> --resource-group <rg-name>
az storage account network-rule list --account-name <storageacct> --resource-group <rg-name>
az role assignment list --scope /subscriptions/<sub-id>/resourceGroups/<rg-name>/providers/Microsoft.Storage/storageAccounts/<storageacct> -o table
```

When choosing an answer, remember:

- If the question is about accessing blob data, prefer a **data-plane role** over a management role.
- If the question is about restricting access to a storage account, prefer **firewall rules** or **private endpoints** depending on the scenario.
- If the question mentions “from on-premises only,” a **private endpoint** or VPN-connected network is often the best fit.

**Compute and VM troubleshooting**

VM issues usually fall into one of four categories: provisioning, connectivity, performance, or guest OS configuration.

Provisioning problems often involve:

- Quota limits
- Unsupported VM size in the region
- Unavailable image
- Locked resource group
- Policy restriction

Connectivity problems often involve:

- NSG rules
- Public IP assignment
- Bastion configuration
- OS firewall
- SSH/RDP service status

Performance problems often involve:

- Disk type mismatch
- CPU or memory pressure
- Burstable VM credit exhaustion
- Incorrect sizing

Guest OS checks can include:

```bash
# Linux
systemctl status sshd
sudo ufw status
ip addr show

# Windows
Get-Service -Name TermService
Get-NetFirewallRule | Where-Object {$_.Enabled -eq 'True'}
```

Exam pattern:

- If the VM is running but unreachable, check **network path** before resizing.
- If the VM is slow, check **disk performance** and **CPU utilization** before changing the OS.
- If deployment fails, check **quota** and **region availability** before changing the template.

**Azure Resource Manager and deployment troubleshooting**

Azure Resource Manager (ARM) deployment issues often come from template syntax, parameter values, permissions, or provider registration.

Common causes:

- Invalid parameter type or missing required parameter
- Resource name not meeting naming rules
- Dependency order issue in the template
- Resource provider not registered
- Policy denies the deployment
- Quota exceeded

Useful commands:

```bash
az deployment group validate \
  --resource-group <rg-name> \
  --template-file main.json \
  --parameters @params.json

az provider list --query "[?registrationState!='Registered'].namespace" -o tsv
az deployment group create \
  --resource-group <rg-name> \
  --template-file main.json \
  --parameters @params.json
```

Best-answer trap:

- If the question asks how to catch issues before deployment, choose **validate** or **what-if** rather than deploying first.
- If the question asks how to reduce repeated manual configuration, choose **ARM templates**, **Bicep**, or **automation**.
- If the issue is at subscription scope, check **provider registration** and **policy** before editing the template.

**Monitoring and diagnostics**

Troubleshooting in Azure is easier when you use the platform’s observability tools. The exam often expects you to know which tool fits which problem.

Common tools:

- **Azure Monitor** for metrics and alerts
- **Log Analytics** for querying logs
- **Activity Log** for control-plane events
- **Network Watcher** for network diagnostics
- **Resource Health** for resource-specific availability
- **Service Health** for Azure platform incidents

Example Log Analytics query:

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where OperationNameValue contains "Microsoft.Compute/virtualMachines/write"
| project TimeGenerated, ResourceGroup, ActivityStatusValue, Caller, CorrelationId
| order by TimeGenerated desc
```

Use the right tool for the question:

- **Activity Log**: who changed what in Azure.
- **Metrics**: performance and near-real-time resource behavior.
- **Logs**: detailed event analysis and correlation.
- **Service Health**: whether Azure has a regional or service-wide issue.
- **Resource Health**: whether a specific resource is impacted.

**Common best-answer traps**

AZ-104 often includes answers that are technically possible but not the best choice. Watch for these patterns:

- **Overly broad fixes**
  - Recreating a resource when a single rule change would work.
  - Moving a workload when a configuration issue is more likely.
- **Wrong scope**
  - Assigning permissions at subscription scope when resource-group scope is enough.
  - Changing a global setting when the issue is isolated to one subnet or VM.
- **Wrong plane**
  - Using management-plane roles for data access.
  - Using storage account keys when RBAC is the intended solution.
- **Ignoring native features**
  - Manually patching access instead of using policy, PIM, or automation.
  - Using custom scripts when built-in diagnostics answer the question.
- **Choosing a disruptive fix**
  - Redeploying or resizing before checking logs and effective rules.
  - Replacing networking components before validating DNS and routing.

A quick decision filter:

| Question clue | Likely best answer style |
|---|---|
| “Least administrative effort” | Role assignment, policy, automation |
| “Minimize downtime” | Non-disruptive config change |
| “Prevent future issues” | Policy, monitoring, automation |
| “Temporary access” | Just-in-time access, PIM, SAS |
| “Access to data only” | Data-plane role or SAS |
| “Network-only access” | NSG, route, private endpoint |

**How to eliminate distractors**

When two answers seem correct, compare them against the scenario constraints.

Ask yourself:

- Does the answer solve the exact problem, or only a related one?
- Does it require more changes than necessary?
- Does it match the scope in the question?
- Does it use the Azure feature that is designed for this purpose?
- Does it preserve security and least privilege?

Examples:

- If the question asks for access to a single storage container, do not choose a subscription-wide role if a narrower scope works.
- If the question asks to block internet access to a VM, do not choose a DNS change if an NSG or route table is the real control.
- If the question asks to audit changes, choose **Activity Log** or **Azure Monitor** rather than manual screenshots.

**Practical troubleshooting workflow**

Use a repeatable workflow to avoid jumping to conclusions.

1. Read the question twice and identify the exact failure.
2. Classify the issue as identity, network, storage, compute, or deployment.
3. Identify the scope: subscription, resource group, resource, subnet, or user.
4. Check the most likely Azure-native diagnostic tool.
5. Choose the least disruptive fix that satisfies the requirement.
6. Verify whether the answer also meets security and operational constraints.

**CLI and portal habits that help in the exam**

You do not need to memorize every command, but you should recognize what each tool does.

```bash
# Check current subscription and context
az account show

# Inspect role assignments
az role assignment list --all -o table

# Review storage network rules
az storage account network-rule list --account-name <storageacct> --resource-group <rg-name>

# Check effective NSG on a NIC
az network nic show-effective-nsg --resource-group <rg-name> --name <nic-name>

# Validate an ARM deployment
az deployment group validate --resource-group <rg-name> --template-file main.bicep
```

Portal habits:

- Use **Activity Log** for recent control-plane changes.
- Use **Resource Health** when a single resource is unavailable.
- Use **Service Health** when the issue may be regional.
- Use **Diagnose and solve problems** when the portal offers a targeted workflow.

**High-yield exam reminders**

- **RBAC** controls access to Azure resources; data access may require separate data-plane roles.
- **NSG** rules are stateful and can block traffic even when a public IP exists.
- **DNS** issues can mimic network failures.
- **Policy** can deny deployments even when permissions are correct.
- **PIM** roles may not be active until you activate them.
- **Service Health** is for platform incidents, not your own configuration mistakes.
- **Least privilege** and **least disruption** are frequent deciding factors.

**Final exam strategy**

If you are unsure, eliminate answers that are too broad, too disruptive, or not aligned with the scenario. Prefer the option that:

- fixes the root cause,
- uses the native Azure feature,
- minimizes operational impact,
- and matches the scope described in the question.

That approach is usually the safest path to the best answer on AZ-104.
