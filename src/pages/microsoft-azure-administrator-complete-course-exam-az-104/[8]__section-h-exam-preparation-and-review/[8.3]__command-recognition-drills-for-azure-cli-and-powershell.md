## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Purpose and exam strategy**

AZ-104 often tests whether you can recognize the right tool, not memorize every parameter. Azure CLI (`az`) is common for cross-platform automation, while Azure PowerShell (`Az` module) is common in Windows-centric administration and object-based workflows. You should know the service area, the command family, and the typical verbs used for create, update, list, show, and delete operations.

A useful mental model is:

- **Azure CLI** uses `az <group> <command>` and returns JSON by default.
- **Azure PowerShell** uses `Get-`, `New-`, `Set-`, `Remove-`, and `Update-` cmdlets and returns objects.
- **Management-plane** commands change Azure resources.
- **Data-plane** commands operate on service data, such as blobs, files, or secrets.

> If a command looks plausible but does not match the service’s normal command family, treat it cautiously. AZ-104 questions often reward recognition of the standard pattern.

**Azure CLI and PowerShell basics**

Before drilling service commands, know the common shell patterns and how to inspect output.

```bash
az login
az account show
az account set --subscription "My Subscription"
az group list -o table
```

```powershell
Connect-AzAccount
Get-AzContext
Set-AzContext -Subscription "My Subscription"
Get-AzResourceGroup
```

- `az login` authenticates the CLI session.
- `Connect-AzAccount` authenticates the PowerShell session.
- `az account show` displays the active subscription object.
- `Get-AzContext` returns the current Azure context, including subscription and tenant details.
- `Set-AzContext` changes the active subscription or context in PowerShell.

**Resource groups and general resource management**

Resource groups are the first place to look when you need to create, move, tag, or delete resources.

```bash
az group create --name rg-prod --location eastus
az group show --name rg-prod
az group update --name rg-prod --set tags.env=prod
az group delete --name rg-prod --yes --no-wait
```

```powershell
New-AzResourceGroup -Name rg-prod -Location eastus
Get-AzResourceGroup -Name rg-prod
Set-AzResourceGroup -Name rg-prod -Tag @{ env = "prod" }
Remove-AzResourceGroup -Name rg-prod -Force
```

- **Create** commands usually use `create` in CLI and `New-` in PowerShell.
- **Update** commands often use `update` in CLI and `Set-` in PowerShell.
- **Delete** commands often require confirmation flags in CLI and `-Force` in PowerShell.
- Tags are commonly updated with `--set` in CLI or a hashtable in PowerShell.

**Compute: virtual machines and extensions**

VM commands are common in AZ-104. You should recognize start, stop, deallocate, resize, and extension patterns.

```bash
az vm create \
  --resource-group rg-compute \
  --name vm01 \
  --image Ubuntu2204 \
  --admin-username azureuser \
  --generate-ssh-keys

az vm start --resource-group rg-compute --name vm01
az vm stop --resource-group rg-compute --name vm01
az vm deallocate --resource-group rg-compute --name vm01
az vm restart --resource-group rg-compute --name vm01
az vm resize --resource-group rg-compute --name vm01 --size Standard_DS2_v2
```

```powershell
$vm = New-AzVMConfig -VMName vm01 -VMSize Standard_DS2_v2
Start-AzVM -ResourceGroupName rg-compute -Name vm01
Stop-AzVM -ResourceGroupName rg-compute -Name vm01
Stop-AzVM -ResourceGroupName rg-compute -Name vm01 -StayProvisioned:$false
Restart-AzVM -ResourceGroupName rg-compute -Name vm01
Update-AzVM -ResourceGroupName rg-compute -VM $vm
```

- `az vm stop` stops the VM but does **not** deallocate it; billing for compute continues until deallocation.
- `az vm deallocate` releases compute resources and is the command to remember for stopping compute charges.
- In PowerShell, `Stop-AzVM` can stop a VM, and `-StayProvisioned:$false` is the key idea when you want deallocation behavior.
- `Update-AzVM` is the standard pattern after modifying a VM object in PowerShell.

VM extensions are often used for configuration, monitoring, or bootstrap tasks.

```bash
az vm extension set \
  --resource-group rg-compute \
  --vm-name vm01 \
  --name CustomScript \
  --publisher Microsoft.Azure.Extensions \
  --settings '{"fileUris":["https://example.com/install.sh"],"commandToExecute":"bash install.sh"}'
```

```powershell
Set-AzVMExtension `
  -ResourceGroupName rg-compute `
  -VMName vm01 `
  -Name CustomScript `
  -Publisher Microsoft.Azure.Extensions `
  -ExtensionType CustomScript `
  -TypeHandlerVersion 2.1 `
  -SettingString '{"fileUris":["https://example.com/install.sh"],"commandToExecute":"bash install.sh"}'
```

- **Custom Script Extension** is a common recognition target for post-deployment configuration.
- Extension commands usually include publisher, type, and settings.
- Use extensions when you need to run scripts inside the guest OS after deployment.

**Managed identities and role assignment**

Managed identities let Azure resources authenticate without storing credentials. AZ-104 often tests the difference between enabling the identity and granting it access.

```bash
az vm identity assign --resource-group rg-compute --name vm01
az webapp identity assign --resource-group rg-app --name webapp01
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/<subId>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/stg01
```

```powershell
$vm = Get-AzVM -ResourceGroupName rg-compute -Name vm01
Update-AzVM -ResourceGroupName rg-compute -VM $vm -IdentityType SystemAssigned

New-AzRoleAssignment `
  -ObjectId <principal-object-id> `
  -RoleDefinitionName "Storage Blob Data Reader" `
  -Scope "/subscriptions/<subId>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/stg01"
```

- **System-assigned managed identity** is enabled on the resource itself.
- **User-assigned managed identity** is a separate resource that can be attached to multiple services.
- Enabling identity does not grant permissions by itself.
- Use Azure RBAC role assignments to authorize the identity on the target scope.

**Virtual networks, subnets, NICs, and peering**

Networking commands are a major AZ-104 area. You should recognize VNet creation, subnet updates, NIC attachment, and peering.

```bash
az network vnet create \
  --resource-group rg-net \
  --name vnet01 \
  --address-prefix 10.10.0.0/16 \
  --subnet-name subnet-app \
  --subnet-prefix 10.10.1.0/24

az network vnet subnet create \
  --resource-group rg-net \
  --vnet-name vnet01 \
  --name subnet-db \
  --address-prefixes 10.10.2.0/24

az network vnet subnet update \
  --resource-group rg-net \
  --vnet-name vnet01 \
  --name subnet-db \
  --delegations Microsoft.Web/serverFarms
```

```powershell
$vnet = New-AzVirtualNetwork -ResourceGroupName rg-net -Location eastus -Name vnet01 -AddressPrefix 10.10.0.0/16
Add-AzVirtualNetworkSubnetConfig -Name subnet-app -AddressPrefix 10.10.1.0/24 -VirtualNetwork $vnet
$vnet | Set-AzVirtualNetwork
```

- `az network vnet create` is the standard VNet creation command.
- `az network vnet subnet create` adds a subnet to an existing VNet.
- In PowerShell, subnet changes are often made on the in-memory VNet object and then persisted.
- Delegation is important when a subnet must host a platform service such as App Service or Azure Container Instances.

NIC management is often tested with VM attachment and IP configuration.

```bash
az network nic create \
  --resource-group rg-net \
  --name nic01 \
  --vnet-name vnet01 \
  --subnet subnet-app

az network nic ip-config update \
  --resource-group rg-net \
  --nic-name nic01 \
  --name ipconfig1 \
  --private-ip-address 10.10.1.10
```

```powershell
$nic = New-AzNetworkInterface -ResourceGroupName rg-net -Location eastus -Name nic01 -SubnetId $subnet.Id
```

- NICs connect VMs to subnets.
- Private IP assignment can be dynamic or static.
- Exam questions may ask you to identify whether a NIC or a subnet change is the correct fix.

Peering connects VNets without using a gateway for direct private connectivity.

```bash
az network vnet peering create \
  --resource-group rg-net \
  --vnet-name vnet01 \
  --name peer-to-vnet02 \
  --remote-vnet /subscriptions/<subId>/resourceGroups/rg-net/providers/Microsoft.Network/virtualNetworks/vnet02 \
  --allow-vnet-access
```

```powershell
Add-AzVirtualNetworkPeering `
  -Name peer-to-vnet02 `
  -VirtualNetwork $vnet1 `
  -RemoteVirtualNetworkId $vnet2.Id `
  -AllowVirtualNetworkAccess
```

- **Peering** is non-transitive by default.
- Use peering for low-latency private connectivity between VNets.
- Gateway transit and forwarded traffic are separate options you may need to recognize.

**Private endpoints and private DNS**

Private endpoints are common in secure designs. The `--group-id` must match the supported subresource for the target service.

```bash
az network private-endpoint create \
  --resource-group rg-net \
  --name pe-stg01 \
  --vnet-name vnet01 \
  --subnet subnet-private \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/stg01 \
  --group-id blob \
  --connection-name stg01-blob-conn
```

```bash
az network private-dns zone create --resource-group rg-net --name privatelink.blob.core.windows.net
az network private-dns link vnet create \
  --resource-group rg-net \
  --zone-name privatelink.blob.core.windows.net \
  --name link-vnet01 \
  --virtual-network vnet01 \
  --registration-enabled false
```

```powershell
$zone = New-AzPrivateDnsZone -ResourceGroupName rg-net -Name "privatelink.blob.core.windows.net"
New-AzPrivateDnsVirtualNetworkLink -ResourceGroupName rg-net -ZoneName $zone.Name -Name link-vnet01 -VirtualNetworkId $vnet.Id
```

- For Azure Storage private endpoints, common `--group-id` values include `blob`, `file`, `queue`, and `table`.
- Private DNS zones resolve private endpoint names to private IP addresses.
- Linking the VNet to the private DNS zone is required for name resolution in that network.

**Load balancing and application gateways**

You may need to recognize the service family even if you do not memorize every parameter.

```bash
az network lb create --resource-group rg-net --name lb01 --sku Standard --frontend-ip-name fe01 --backend-pool-name be01
az network application-gateway create --resource-group rg-net --name agw01 --location eastus --sku Standard_v2 --vnet-name vnet01 --subnet appgw-subnet
```

- **Load Balancer** is layer 4.
- **Application Gateway** is layer 7 and often includes web traffic routing features.
- Standard SKU is the common exam-relevant choice for modern deployments.

**Storage accounts, blobs, files, and queues**

Storage questions often mix management-plane and data-plane operations. Recognize the difference.

```bash
az storage account create \
  --resource-group rg-storage \
  --name stg12345 \
  --location eastus \
  --sku Standard_LRS \
  --kind StorageV2

az storage container create \
  --account-name stg12345 \
  --name data \
  --auth-mode login

az storage blob upload \
  --account-name stg12345 \
  --container-name data \
  --name report.csv \
  --file ./report.csv \
  --auth-mode login
```

```powershell
New-AzStorageAccount -ResourceGroupName rg-storage -Name stg12345 -Location eastus -SkuName Standard_LRS -Kind StorageV2
$ctx = (Get-AzStorageAccount -ResourceGroupName rg-storage -Name stg12345).Context
New-AzStorageContainer -Name data -Context $ctx
Set-AzStorageBlobContent -File .\report.csv -Container data -Blob report.csv -Context $ctx
```

- **Management-plane** commands create the storage account itself.
- **Data-plane** commands create containers and upload blobs.
- In PowerShell, many storage cmdlets require a storage context.

Blob access tier and lifecycle management are common operational topics.

```bash
az storage blob set-tier \
  --account-name stg12345 \
  --container-name data \
  --name report.csv \
  --tier Cool \
  --auth-mode login
```

```powershell
Set-AzStorageBlob -Container data -Blob report.csv -StandardBlobTier Cool -Context $ctx
```

- Access tiers include **Hot**, **Cool**, and **Archive**.
- Lifecycle management policies automate tier changes and deletion based on age or conditions.
- Archive is for infrequently accessed data and has rehydration delay considerations.

File shares are another frequent exam area.

```bash
az storage share-rm create --resource-group rg-storage --storage-account stg12345 --name share01
az storage file upload --account-name stg12345 --share-name share01 --source ./notes.txt --path notes.txt --auth-mode login
```

```powershell
New-AzStorageShare -Name share01 -Context $ctx
Set-AzStorageFileContent -ShareName share01 -Source .\notes.txt -Path notes.txt -Context $ctx
```

- Azure Files supports SMB and NFS scenarios depending on account and share configuration.
- Share-level permissions and identity-based access are separate from blob access patterns.
- File share operations often use a storage context plus share-specific cmdlets.

Queues are also part of the storage service family.

```bash
az storage queue create --account-name stg12345 --name queue01 --auth-mode login
az storage message put --account-name stg12345 --queue-name queue01 --content "hello" --auth-mode login
```

```powershell
New-AzStorageQueue -Name queue01 -Context $ctx
Add-AzStorageQueueMessage -Queue queue01 -Message "hello" -Context $ctx
```

- Queue commands are useful for decoupled messaging.
- Recognize the difference between queue data operations and storage account management.

**Key Vault and secrets**

Key Vault commands often appear in identity and application configuration scenarios.

```bash
az keyvault create --resource-group rg-sec --name kv-prod --location eastus
az keyvault secret set --vault-name kv-prod --name DbPassword --value "P@ssw0rd!"
az keyvault secret show --vault-name kv-prod --name DbPassword
```

```powershell
New-AzKeyVault -ResourceGroupName rg-sec -VaultName kv-prod -Location eastus
Set-AzKeyVaultSecret -VaultName kv-prod -Name DbPassword -SecretValue (ConvertTo-SecureString "P@ssw0rd!" -AsPlainText -Force)
$secret = Get-AzKeyVaultSecret -VaultName kv-prod -Name DbPassword
```

- `Get-AzKeyVaultSecret` returns a secret object; you may need to access the value explicitly depending on the workflow and object type.
- Key Vault access can be controlled with access policies or Azure RBAC, depending on the vault configuration.
- Secrets, keys, and certificates are separate object types.

**App Service and web apps**

App Service commands are common for web app deployment and configuration. Recognize the plan, app, settings, and slot patterns.

```bash
az appservice plan create --resource-group rg-app --name asp-web --sku B1 --is-linux
az webapp create --resource-group rg-app --plan asp-web --name webapp01 --runtime "DOTNETCORE:8.0"
az webapp config appsettings set --resource-group rg-app --name webapp01 --settings ENV=prod
az webapp deployment slot create --resource-group rg-app --name webapp01 --slot staging
```

```powershell
New-AzAppServicePlan -ResourceGroupName rg-app -Name asp-web -Location eastus -Tier Basic -NumberofWorkers 1 -WorkerSize Small
New-AzWebApp -ResourceGroupName rg-app -Name webapp01 -Location eastus -AppServicePlan asp-web
Set-AzWebApp -ResourceGroupName rg-app -Name webapp01 -AppSettings @{ ENV = "prod" }
New-AzWebAppSlot -ResourceGroupName rg-app -Name webapp01 -Slot staging
```

- The App Service plan is the compute container for the web app.
- Deployment slots support staged releases and slot swaps.
- App settings are commonly used for environment-specific configuration.
- Be careful with PowerShell examples: the plan must exist, and the app should be created in a way that matches the module version and runtime requirements.

Scaling is another important recognition area.

```bash
az appservice plan update --resource-group rg-app --name asp-web --sku S1
az webapp scale --resource-group rg-app --name webapp01 --number-of-workers 2
```

```powershell
Set-AzAppServicePlan -ResourceGroupName rg-app -Name asp-web -Tier Standard
```

- Scale up changes the SKU.
- Scale out changes the number of instances.
- App Service autoscale is often configured through Azure Monitor rather than the web app command itself.

**Monitoring, diagnostics, and alerts**

Monitoring questions often distinguish metrics, logs, and diagnostic settings. Metric alerts and log alerts use different command families.

```bash
az monitor diagnostic-settings create \
  --name diag01 \
  --resource /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Web/sites/webapp01 \
  --workspace /subscriptions/<subId>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law01 \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

```bash
az monitor metrics alert create \
  --name HighCPU \
  --resource-group rg-monitor \
  --scopes /subscriptions/<subId>/resourceGroups/rg-compute/providers/Microsoft.Compute/virtualMachines/vm01 \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m
```

```powershell
Set-AzDiagnosticSetting -Name diag01 -ResourceId $resourceId -WorkspaceId $workspaceId -Enabled $true
Add-AzMetricAlertRuleV2 -Name HighCPU -ResourceGroupName rg-monitor -TargetResourceId $vmId -Condition (New-AzMetricAlertRuleV2Criteria -MetricName "Percentage CPU" -TimeAggregation Average -Operator GreaterThan -Threshold 80)
```

- **Diagnostic settings** send logs and metrics to a destination such as Log Analytics, Event Hub, or Storage.
- **Metric alerts** trigger on numeric thresholds.
- **Log alerts** use query results from Log Analytics and are often based on KQL.
- `az monitor metrics alert create` is the correct CLI family for metric alerts.

A simple KQL example helps with recognition.

```kusto
AzureActivity
| where ActivityStatusValue == "Failed"
| summarize Count = count() by OperationNameValue
| order by Count desc
```

- KQL stands for **Kusto Query Language**.
- Use KQL in Log Analytics queries, log alerts, and workbook-style investigations.
- Activity logs are not the same as resource health or service health.

Resource health and service health are separate from activity logs.

- **Resource Health** tells you whether a specific resource is available and why it may be unavailable.
- **Service Health** reports Azure platform incidents, advisories, and planned maintenance.
- Activity logs show subscription-level control-plane events, not the same thing as health status.

**Backup and recovery**

Backup and restore are important for VM and workload protection. Recognize the recovery services vault pattern.

```bash
az backup vault create --resource-group rg-backup --name rsv01 --location eastus
az backup protection enable-for-vm --vault-name rsv01 --resource-group rg-backup --vm vm01 --policy-name DefaultPolicy
az backup recoverypoint list --vault-name rsv01 --resource-group rg-backup --container-name vm01 --item-name vm01
```

```powershell
$vault = Get-AzRecoveryServicesVault -Name rsv01 -ResourceGroupName rg-backup
Set-AzRecoveryServicesVaultContext -Vault $vault
Enable-AzRecoveryServicesBackupProtection -Item $vmItem -Policy $policy
Get-AzRecoveryServicesBackupRecoveryPoint -Item $vmItem
```

- Backup protection is enabled through a vault and policy.
- Recovery points are the restore targets.
- Restore commands usually require the item, recovery point, and restore configuration.

A typical restore flow is:

1. Identify the protected item.
2. List recovery points.
3. Choose the correct restore point.
4. Restore to the original location or a new location, depending on the scenario.

**Azure Policy and governance**

Policy is a common AZ-104 governance topic and is often tested alongside RBAC and tagging.

```bash
az policy assignment create \
  --name enforce-tags \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/<policyDefinitionId>" \
  --scope /subscriptions/<subId>/resourceGroups/rg-prod
```

```powershell
New-AzPolicyAssignment -Name enforce-tags -PolicyDefinition $policyDefinition -Scope "/subscriptions/<subId>/resourceGroups/rg-prod"
```

- **Azure Policy** enforces rules and effects such as deny, audit, or append.
- Policy is not the same as Azure RBAC.
- Use policy for compliance and standardization, not for granting access.

**Azure Automation and runbooks**

Automation is useful for scheduled or event-driven operational tasks.

```bash
az automation account create --resource-group rg-auto --name auto01 --location eastus
```

```powershell
New-AzAutomationAccount -ResourceGroupName rg-auto -Name auto01 -Location eastus
```

- Automation accounts host runbooks, modules, and schedules.
- Runbooks can be PowerShell or Python depending on the environment.
- Recognize Automation as a control-plane orchestration service, not a replacement for Azure Functions.

**Identity and access management**

AZ-104 expects you to distinguish Microsoft Entra ID roles from Azure RBAC roles.

- **Microsoft Entra ID roles** manage directory-level permissions such as user administrator or global administrator.
- **Azure RBAC** manages access to Azure resources such as Reader, Contributor, or Storage Blob Data Contributor.
- Service principals represent applications in Entra ID and are often used for automation.

```bash
az ad sp create-for-rbac --name sp-automation --role Contributor --scopes /subscriptions/<subId>
az role assignment list --assignee <appId-or-objectId>
```

```powershell
New-AzADServicePrincipal -DisplayName sp-automation
Get-AzRoleAssignment -ObjectId <principal-object-id>
```

- A service principal is the security identity used by an application or automation process.
- Managed identity is preferred when the workload runs in Azure and can use a built-in identity.
- Use Azure RBAC role assignments to grant access at the right scope.

**CLI and PowerShell recognition map**

| Task | Azure CLI | Azure PowerShell |
|---|---|---|
| Sign in | `az login` | `Connect-AzAccount` |
| Show current subscription | `az account show` | `Get-AzContext` |
| Create resource group | `az group create` | `New-AzResourceGroup` |
| Create VM | `az vm create` | `New-AzVM` |
| Stop VM | `az vm stop` | `Stop-AzVM` |
| Deallocate VM | `az vm deallocate` | `Stop-AzVM -StayProvisioned:$false` |
| Create VNet | `az network vnet create` | `New-AzVirtualNetwork` |
| Create storage account | `az storage account create` | `New-AzStorageAccount` |
| Create Key Vault | `az keyvault create` | `New-AzKeyVault` |
| Create web app | `az webapp create` | `New-AzWebApp` |
| Create metric alert | `az monitor metrics alert create` | `Add-AzMetricAlertRuleV2` |

- The table is a recognition aid, not a guarantee that every parameter is identical.
- PowerShell often requires object creation and pipeline persistence.
- CLI often uses explicit flags and is easier to read as a single command line.

**Practical drill patterns**

When you see a scenario, identify the service first, then the operation, then the tool family.

- If the question mentions a VM that should stop billing, look for **deallocate**.
- If the question mentions private connectivity to Storage, look for **private endpoint** plus the correct `group-id`.
- If the question mentions logs sent to Log Analytics, look for **diagnostic settings**.
- If the question mentions a web app deployment environment, look for **deployment slots**.
- If the question mentions access without passwords, look for **managed identity**.
- If the question mentions compliance rules, look for **Azure Policy**.
- If the question mentions scheduled automation, look for **Automation account** or runbook.

**Common pitfalls to avoid**

> Do not confuse stopping a VM with deallocating it. The first stops the guest; the second releases compute resources.

> Do not confuse Azure RBAC with Microsoft Entra ID roles. They solve different authorization problems.

> Do not assume a data-plane command can replace a management-plane command. Storage account creation and blob upload are different layers.

> Do not treat activity logs as a substitute for resource health or service health.

**Quick command drills**

```bash
# VM lifecycle
az vm start -g rg-compute -n vm01
az vm stop -g rg-compute -n vm01

# Networking

# Storage
az storage blob list --account-name stg12345 --container-name data --auth-mode login

# Monitoring
az monitor metrics alert create --name HighCPU --resource-group rg-monitor --scopes <resourceId> --condition "avg Percentage CPU > 80" --window-size 5m --evaluation-frequency 1m

# Identity
az role assignment create --assignee-object-id <principal-object-id> --role Reader --scope /subscriptions/<subId>
```

```powershell
# VM lifecycle
Start-AzVM -ResourceGroupName rg-compute -Name vm01
Stop-AzVM -ResourceGroupName rg-compute -Name vm01

# Networking
Add-AzVirtualNetworkPeering -Name peer01 -VirtualNetwork $vnet1 -RemoteVirtualNetworkId $vnet2.Id -AllowVirtualNetworkAccess

# Storage
Get-AzStorageBlob -Container data -Context $ctx

# Monitoring
Add-AzMetricAlertRuleV2 -Name HighCPU -ResourceGroupName rg-monitor -TargetResourceId $vmId

# Identity
New-AzRoleAssignment -ObjectId <principal-object-id> -RoleDefinitionName Reader -Scope "/subscriptions/<subId>"
```

**Final exam reminders**

- Learn the service family first, then the command verb.
- Prefer the standard Az cmdlets and current Azure CLI command groups.
- Watch for management-plane versus data-plane wording.
- Use the command shape to eliminate distractors quickly.
- When in doubt, choose the command that matches the resource type and the operation being performed.
