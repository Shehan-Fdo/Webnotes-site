## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Blueprint overview**

The AZ-104 exam tests whether you can administer Azure in day-to-day operations, not just define services. Expect scenario-based questions that combine identity, governance, networking, storage, compute, monitoring, and backup into one workflow.

A strong final review focuses on:
- What each service does.
- How services connect in real deployments.
- Which tool you use for a given task: portal, Azure CLI, PowerShell, ARM, or Bicep.
- Common failure points such as permissions, DNS, network rules, and resource scope.

**Exam domains to prioritize**

The exam blueprint typically centers on these operational areas:
- **Identity and governance**: Microsoft Entra ID, Azure RBAC, subscriptions, management groups, locks, tags, and policy.
- **Storage**: storage accounts, access tiers, SAS, access keys, Azure Files, and network controls.
- **Compute**: virtual machines, VM scale sets, extensions, managed identities, and availability options.
- **Networking**: VNets, subnets, NSGs, peering, VPN, load balancing, private endpoints, and routing.
- **Monitoring and backup**: Azure Monitor, diagnostic settings, alerts, Recovery Services vaults, and restore workflows.

**Governance hierarchy and scope**

Azure governance is built around scope. You assign access and apply policy at the right level so controls inherit downward.

- **Management groups** organize subscriptions and let you apply governance across multiple subscriptions.
- **Subscriptions** are the billing and quota boundary for most resources.
- **Resource groups** are lifecycle containers for related resources.
- **Resources** inherit many controls from higher scopes, but you can override some settings at lower scopes.

A practical hierarchy looks like this:

```text
Management Group
└── Subscription
    ├── Resource Group: rg-prod-app
    │   ├── Virtual Network
    │   ├── VM
    │   └── Storage Account
    └── Resource Group: rg-prod-data
        └── SQL / Storage / Key Vault
```

Use management groups when you need consistent governance across multiple subscriptions, such as separating production and nonproduction while keeping shared policy at the top.

**Azure RBAC and directory roles**

Azure uses two different permission systems that are often confused:

- **Azure RBAC** controls access to Azure resources.
- **Microsoft Entra directory roles** control identity and directory administration in Entra ID.

Important exam distinctions:
- Azure RBAC assignments are scoped to management group, subscription, resource group, or resource.
- Role assignment inheritance flows from parent scope to child scope.
- A user can have access through multiple assignments; effective permissions are the union of allowed actions minus explicit denies where applicable.
- Custom roles are useful when built-in roles are too broad.
- Directory roles such as Global Administrator do not automatically grant Azure resource access unless paired with Azure RBAC or specific tenant-level permissions.

Common built-in Azure RBAC roles:
- **Owner**: full access plus permission to assign roles.
- **Contributor**: full resource management, but no role assignment.
- **Reader**: view-only access.
- **User Access Administrator**: manage role assignments.
- Service-specific roles such as **Virtual Machine Contributor** or **Storage Blob Data Contributor** reduce privilege.

A custom role definition uses allowed actions and notActions:

```json
{
  "Name": "VM Restart Operator",
  "IsCustom": true,
  "Description": "Can restart VMs but not delete them",
  "Actions": [
    "Microsoft.Compute/virtualMachines/read",
    "Microsoft.Compute/virtualMachines/restart/action"
  ],
  "NotActions": [
    "Microsoft.Compute/virtualMachines/delete"
  ],
  "AssignableScopes": [
    "/subscriptions/<subscription-id>"
  ]
}
```

**Policy, tags, and locks**

Azure Policy enforces standards; tags help organize and report; locks protect resources from accidental deletion or modification.

- **Policy** can deny, audit, append, or deploy configurations.
- **Initiatives** group multiple policies into one assignment.
- **Tags** support cost allocation, ownership, and environment tracking.
- **Locks** come in two main forms:
  - **CanNotDelete**
  - **ReadOnly**

Use policy when you want to enforce compliance, such as allowed regions or required tags. Use locks when you want to protect a specific resource from accidental change.

Example policy assignment with a built-in definition ID:

```bash
az policy assignment create \
  --name require-costcenter-tag \
  --display-name "Require CostCenter tag" \
  --scope /subscriptions/<subscription-id> \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>
```

Trigger a compliance scan in the current subscription context:

```bash
az account set --subscription <subscription-id>
az policy state trigger-scan
```

**Storage account fundamentals**

Storage accounts are a frequent AZ-104 topic because they combine identity, networking, and data protection.

Core concepts:
- **General-purpose v2** is the default exam-relevant storage account type.
- **Replication** options include LRS, ZRS, GRS, and GZRS.
- **Access tiers** for blobs include hot, cool, and archive.
- **Endpoints** can be public, private, or restricted by firewall rules.

Authentication options:
- **Access keys** provide full account-level access and are powerful but broad.
- **Connection strings** bundle endpoint and credential information for applications.
- **Shared Access Signature (SAS)** grants limited, time-bound access.
- **User delegation SAS** uses Microsoft Entra credentials and is preferred for blob access when possible.
- **Service SAS** is scoped to a service such as Blob or File.
- **Account SAS** can span multiple services and resource types.

Practical CLI examples:

```bash
az storage account create \
  --name mystorageacct123 \
  --resource-group rg-data \
  --location eastus \
  --sku Standard_GRS \
  --kind StorageV2
```

List keys and retrieve a connection string:

```bash
az storage account keys list \
  --resource-group rg-data \
  --account-name mystorageacct123

az storage account show-connection-string \
  --resource-group rg-data \
  --name mystorageacct123
```

Generate a user delegation SAS for blob access:

```bash
az storage blob generate-sas \
  --account-name mystorageacct123 \
  --container-name reports \
  --name q1.csv \
  --permissions r \
  --expiry 2026-05-01T00:00Z \
  --auth-mode login
```

**Storage networking and firewall rules**

Storage security is not just about private endpoints. You also need to understand firewall behavior and service access.

- **Storage firewall** can allow selected public IPs, selected VNets, or deny all public access.
- **Service endpoints** extend a subnet’s identity to a service over the Azure backbone.
- **Private endpoints** map a storage service to a private IP in your VNet.
- **Private DNS** is required so clients resolve the storage FQDN to the private IP.

Use service endpoints when you want simpler VNet-based access without private IPs. Use private endpoints when you want the service fully reachable through private addressing and tighter isolation.

> A private endpoint does not automatically fix DNS. If name resolution still points to the public endpoint, clients may fail even though the private endpoint exists.

Example private endpoint for Blob storage:

```bash
az network private-endpoint create \
  --name pe-storage-blob \
  --resource-group rg-network \
  --vnet-name vnet-prod \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subscription-id>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/mystorageacct123 \
  --group-ids blob \
  --connection-name pe-storage-blob-conn
```

Blob is only one subresource. Other storage services use different group IDs such as `file`, `queue`, `table`, or `web` depending on the account features you expose.

**Azure Files and file sharing**

Azure Files provides managed SMB and NFS file shares. It is commonly used for lift-and-shift file workloads, shared application content, and hybrid file access.

Key points:
- **SMB** is common for Windows clients and many cross-platform scenarios.
- **NFS** is used for Linux-oriented workloads where supported.
- **Snapshots** provide point-in-time share recovery.
- **Azure File Sync** extends on-premises Windows Server caching and synchronization to Azure Files.

When to use Azure Files:
- You need a shared file system without managing file servers.
- You want hybrid access from on-premises and Azure.
- You need centralized file storage for multiple VMs or apps.

Example share creation:

```bash
az storage share create \
  --account-name mystorageacct123 \
  --name appshare \
  --quota 100
```

Example mount on Linux using SMB:

```bash
sudo mkdir -p /mnt/appshare
sudo mount -t cifs //<account>.file.core.windows.net/appshare /mnt/appshare \
  -o vers=3.0,username=<account>,password=<key>,dir_mode=0777,file_mode=0777,serverino
```

Azure File Sync is useful when:
- You need local caching at branch offices.
- You want cloud tiering.
- You need centralized backup and disaster recovery for file data.

**Compute administration**

Virtual machines are tested from both provisioning and operations angles. You should know how to size, secure, extend, and troubleshoot them.

Important VM topics:
- **Sizing** affects CPU, memory, disk throughput, and network limits.
- **Resizing** may require the VM to be stopped/deallocated if the target size is not available on the current host.
- **Managed disks** are the default storage choice for VM OS and data disks.
- **Availability sets** protect against hardware failure within a datacenter.
- **Availability zones** protect against datacenter-level failures in a region.

VM extensions are a high-yield topic because they automate post-deployment configuration:
- **Custom Script Extension** runs scripts after deployment.
- **VMAccess** can help reset credentials or SSH configuration.
- Monitoring and security extensions can install agents and configure telemetry.

Example Custom Script Extension:

```bash
az vm extension set \
  --resource-group rg-compute \
  --vm-name vm-app01 \
  --name CustomScript \
  --publisher Microsoft.Azure.Extensions \
  --settings '{
    "fileUris": ["https://example.com/install.sh"],
    "commandToExecute": "bash install.sh"
  }'
```

Managed identity is important when a VM needs to access Azure services without storing secrets.

```bash
az vm identity assign \
  --resource-group rg-compute \
  --name vm-app01
```

Then grant the VM a role such as `Storage Blob Data Reader` on the target storage account or container scope.

**Virtual machine scale sets**

VM scale sets let you manage a group of identical VMs with autoscale and centralized configuration.

Use them when:
- You need horizontal scaling.
- You want consistent deployment across many instances.
- You need rolling upgrades or instance replacement.

Know the difference between:
- **Uniform orchestration** for identical instances.
- **Flexible orchestration** for mixed instance types and more advanced scenarios.

**Networking fundamentals**

Networking questions often combine subnet design, security, routing, and service connectivity.

Core objects:
- **Virtual network (VNet)**: private network boundary.
- **Subnet**: segmented range inside a VNet.
- **Network security group (NSG)**: stateful allow/deny rules.
- **Route table**: user-defined routes for traffic steering.
- **Public IP**: internet-facing address for Azure resources.

NSG rules are evaluated by priority, where lower numbers win. A deny at a higher priority overrides later allows.

Example NSG rule:

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-app \
  --name Allow-HTTPS \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes Internet \
  --source-port-ranges '*' \
  --destination-port-ranges 443
```

Use NSGs to control traffic at subnet or NIC level. Use application security groups when you want to group VMs logically instead of managing IP lists.

**Peering, DNS, and routing**

VNet peering connects VNets privately over the Azure backbone. It is not transitive by default.

Important peering details:
- Address spaces must not overlap.
- Peering is regional or global depending on the VNets.
- Traffic is private, but you still need route and DNS planning.
- Gateway transit and use remote gateways are advanced options that affect shared connectivity.

DNS is often the hidden cause of connectivity failures:
- Private endpoints require private DNS zones.
- Custom DNS servers must forward Azure zones correctly.
- Name resolution issues can look like firewall problems.

**Service endpoints vs private endpoints**

These are commonly compared in AZ-104.

| Feature | Service Endpoint | Private Endpoint |
|---|---|---|
| Connectivity | Uses service public endpoint over Azure backbone | Uses private IP in your VNet |
| DNS change | Usually no special private DNS requirement | Requires private DNS for correct resolution |
| Exposure | Service still has public endpoint | Public access can be disabled |
| Scope | Subnet-based access | Resource-specific private link |
| Typical use | Simpler VNet restriction | Stronger isolation and private access |

Use service endpoints when you want to restrict access from a subnet without introducing private IPs. Use private endpoints when you need the service to appear inside your network.

**Azure Firewall and load balancing**

Azure Firewall is a managed, stateful network security service for centralized traffic control.

Use Azure Firewall when you need:
- Central egress control.
- FQDN-based filtering.
- Threat intelligence filtering.
- Hub-and-spoke inspection.

Basic deployment pattern:
- Hub VNet contains Azure Firewall.
- Spoke VNets peer to the hub.
- User-defined routes send outbound traffic to the firewall.

Load balancing concepts matter because Azure has multiple options:
- **Azure Load Balancer** operates at Layer 4 and distributes TCP/UDP traffic.
- **Application Gateway** operates at Layer 7 and supports HTTP(S), TLS termination, and web application firewall features.
- **Traffic Manager** is DNS-based global traffic distribution.

Use Load Balancer for non-HTTP workloads or simple TCP/UDP distribution. Use Application Gateway when you need HTTP-aware routing, path-based rules, or web protection.

**Monitoring and diagnostics**

Azure Monitor is more than alerts. It includes metrics, logs, activity logs, and diagnostic settings.

Key distinctions:
- **Metrics** are near real-time numeric measurements.
- **Logs** are detailed records stored in Log Analytics or another destination.
- **Activity logs** track subscription-level control-plane events such as create, update, and delete operations.
- **Diagnostic settings** send resource logs and metrics to destinations such as Log Analytics, Storage, or Event Hub.

Diagnostic settings are critical because many services do not retain detailed logs long-term unless you export them.

Example diagnostic setting to Log Analytics:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<subscription-id>/resourceGroups/rg-compute/providers/Microsoft.Compute/virtualMachines/vm-app01 \
  --workspace /subscriptions/<subscription-id>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --logs '[{"category":"Administrative","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Alerts are typically based on:
- **Metric alerts** for numeric thresholds.
- **Log alerts** for KQL-based queries.
- **Activity log alerts** for subscription events.

A practical metric alert usually needs a target resource and an action group:

```bash
az monitor action-group create \
  --resource-group rg-monitor \
  --name ag-ops \
  --short-name ops \
  --action email ops-team ops@example.com

az monitor metrics alert create \
  --name vm-cpu-high \
  --resource-group rg-monitor \
  --scopes /subscriptions/<subscription-id>/resourceGroups/rg-compute/providers/Microsoft.Compute/virtualMachines/vm-app01 \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --action ag-ops
```

Activity log alerts also need a meaningful scope and action group:

```bash
az monitor activity-log alert create \
  --name rg-delete-alert \
  --resource-group rg-monitor \
  --scopes /subscriptions/<subscription-id>/resourceGroups/rg-compute \
  --condition category=Administrative and operationName=Microsoft.Resources/subscriptions/resourceGroups/delete \
  --action-group ag-ops
```

**Backup and recovery**

Azure Backup protects VMs, files, and other supported workloads through a Recovery Services vault.

Core concepts:
- **Recovery Services vault** stores backup data and policy configuration.
- **Backup policy** defines schedule and retention.
- **Recovery point** is the restore point created by a backup job.
- **Soft delete** helps protect backup data from accidental deletion.

Backup is not the same as replication. Backup supports point-in-time recovery; replication supports failover.

For VMs, backup can help you recover from:
- Accidental deletion, if the backup and retention state still exist.
- Corruption or ransomware, if you have a clean recovery point.
- Operational mistakes such as bad patching or configuration changes.

> Recovery is not guaranteed in every case. It depends on whether the VM was protected, whether the vault and backup data still exist, and whether the retention window still covers the needed restore point.

Example vault creation:

```bash
az backup vault create \
  --resource-group rg-backup \
  --name rsv-prod \
  --location eastus
```

Example backup policy and VM protection workflow:

```bash
az backup policy create \
  --resource-group rg-backup \
  --vault-name rsv-prod \
  --name daily-policy \
  --backup-management-type AzureIaasVM \
  --policy '{"schedulePolicy":{"scheduleRunFrequency":"Daily","scheduleRunTimes":["2026-04-26T02:00:00Z"]},"retentionPolicy":{"dailySchedule":{"retentionTimes":["2026-04-26T02:00:00Z"],"retentionDuration":{"count":30,"durationType":"Days"}}}}'
```

**Deployment and infrastructure as code**

AZ-104 expects you to understand how Azure resources are deployed and updated, even if you are not writing complex templates every day.

Key deployment concepts:
- **ARM templates** are JSON-based declarative deployments.
- **Bicep** is a cleaner language that compiles to ARM.
- **Deployment scope** can be management group, subscription, resource group, or tenant depending on the template and command.
- **What-if** helps you preview changes before deployment.

Useful commands:

```bash
az deployment group what-if \
  --resource-group rg-app \
  --template-file main.bicep \
  --parameters @prod.parameters.json
```

Deploy at resource group scope:

```bash
az deployment group create \
  --resource-group rg-app \
  --template-file main.bicep \
  --parameters @prod.parameters.json
```

Deploy at subscription scope:

```bash
az deployment sub create \
  --location eastus \
  --template-file subscope.bicep
```

Troubleshooting tips:
- Check parameter names and types first.
- Validate resource provider registration.
- Confirm the deployment scope matches the resource type.
- Use `what-if` to catch destructive changes before execution.

**Practical exam scenarios**

Expect questions that combine multiple services:
- A VM cannot reach a storage account because the storage firewall blocks the subnet.
- A private endpoint exists, but DNS still resolves to the public endpoint.
- A user can read a resource group but cannot assign roles because they lack `Owner` or `User Access Administrator`.
- A backup restore is available only if the retention policy still contains the needed recovery point.
- A policy assignment denies resource creation in an unsupported region.
- A VM extension fails because the script URL is unreachable or the command syntax is wrong.

**High-yield CLI workflow examples**

Set the subscription context before running most administrative commands:

```bash
az account set --subscription <subscription-id>
az account show
```

Create a resource group and a VM:

```bash
az group create --name rg-lab --location eastus

az vm create \
  --resource-group rg-lab \
  --name vm01 \
  --image Ubuntu2204 \
  --admin-username azureuser \
  --generate-ssh-keys
```

Open an NSG port for testing:

```bash
az vm open-port \
  --resource-group rg-lab \
  --name vm01 \
  --port 80
```

Assign a role to a user at resource group scope:

```bash
az role assignment create \
  --assignee user@example.com \
  --role Reader \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-lab
```

**Final study strategy**

Focus on scenario thinking rather than memorization alone.

- Map each service to its scope, security model, and operational purpose.
- Practice reading a question for the hidden dependency: DNS, RBAC, policy, or network access.
- Learn the difference between control plane and data plane actions.
- Review commands enough to recognize syntax, but prioritize understanding the outcome.
- Rehearse common troubleshooting paths:
  - permissions first,
  - then network,
  - then DNS,
  - then service configuration,
  - then logs and metrics.

A good final mock exam blueprint is not a list of facts; it is a checklist of how Azure behaves when multiple services interact.
