## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure RBAC fundamentals**

<abbr title="Role-Based Access Control">RBAC</abbr> is Azure’s authorization system for controlling who can do what on which resources. It answers three questions:

- **Who** is the security principal: user, group, service principal, or managed identity.
- **What** is the allowed action: for example, read, write, delete, or a custom action.
- **Where** is the permission valid: the scope, such as management group, subscription, resource group, or resource.

Azure RBAC is separate from authentication. Microsoft Entra ID authenticates the identity; Azure RBAC authorizes access to Azure resources. This distinction matters because a user can sign in successfully and still be denied access if no role assignment grants the required permissions.

A role assignment is the link between a principal, a role definition, and a scope. In practice, you use built-in roles for common tasks and custom roles when built-in roles are too broad or too narrow.

- **Security principal**: the identity receiving access.
- **Role definition**: the set of allowed actions and not-actions.
- **Scope**: the boundary where the role applies.
- **Role assignment**: the actual permission grant.

**Scope hierarchy and inheritance**

Azure scopes form a hierarchy. Permissions assigned at a higher level flow down to lower levels unless explicitly restricted by the role definition or overridden by a more specific assignment.

Typical hierarchy:

1. **Management group**
2. **Subscription**
3. **Resource group**
4. **Resource**

A role assignment at the subscription scope usually applies to all resource groups and resources inside that subscription. A role assignment at a resource group scope applies to all resources in that group, but not to other groups in the same subscription. A role assignment at a resource scope applies only to that single resource.

This inheritance model is useful when you want to manage access once and let it cascade. It also reduces administrative overhead, but it can create accidental overexposure if you assign broad roles too high in the hierarchy.

> A role assignment at a parent scope is inherited by child scopes. Always choose the narrowest scope that still meets the operational need.

**Common scope examples**

- **Management group**: use for enterprise-wide governance across multiple subscriptions.
- **Subscription**: use for platform teams or shared services that need broad access within one subscription.
- **Resource group**: use for application teams that manage a specific workload.
- **Resource**: use for tightly controlled access to one VM, storage account, or database.

A practical example is granting a DevOps team the ability to manage only the `rg-app-prod` resource group instead of the entire subscription. That keeps the team productive while limiting blast radius.

**Built-in roles and least privilege**

Azure includes many built-in roles. The most common ones in AZ-104 scenarios are:

- **Owner**: full access to all resources, including the ability to delegate access.
- **Contributor**: can create and manage resources, but cannot grant access.
- **Reader**: can view resources, but cannot modify them.
- **User Access Administrator**: can manage access assignments, but not resource content.

The principle of **least privilege** means you grant only the permissions required for the task. In most environments, you avoid using Owner unless there is a strong administrative reason. Contributor is often enough for deployment and operations, while User Access Administrator is reserved for access management workflows.

**Actions, notActions, and data actions**

Role definitions contain permission sets that determine what the role can do.

- **Actions** control management-plane operations, such as creating a VM or changing a network rule.
- **NotActions** exclude specific management actions from an otherwise allowed set.
- **DataActions** control access to data inside a service, such as reading blobs in Azure Storage.
- **NotDataActions** exclude specific data-plane operations.

This distinction is important because some roles let you manage a resource without letting you read the data inside it. For example, a user may be able to manage a storage account but still not be able to read blob contents unless the role includes the relevant data actions.

**Role assignment behavior**

Role assignments are additive. If a user receives multiple role assignments, Azure combines the permissions from all of them. There is no “deny” assignment in standard Azure RBAC to cancel out another allow. If access seems broader than expected, check all inherited assignments and group memberships.

Common sources of access include:

- Direct assignment to the user.
- Assignment to an Azure AD group the user belongs to.
- Assignment to a service principal or managed identity.
- Inherited assignment from a parent scope.

Because group-based access is easier to manage at scale, you usually assign roles to groups rather than individual users.

**Practical CLI examples**

Use the Azure CLI to inspect and manage role assignments.

```bash
# Sign in
az login

# List role assignments for a user
az role assignment list --assignee user@contoso.com --all -o table

# Assign Reader at subscription scope
az role assignment create \
  --assignee user@contoso.com \
  --role Reader \
  --scope /subscriptions/<subscription-id>

# Assign Contributor at resource group scope
az role assignment create \
  --assignee <object-id> \
  --role Contributor \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-app-prod
```

You can also inspect role definitions to understand what a role actually permits.

```bash
# Show built-in role details
az role definition list --name "Contributor" -o jsonc

# Query a custom role by name
az role definition list --custom-role-only true --name "My Storage Operator" -o jsonc
```

**Azure PowerShell examples**

PowerShell is common in administration tasks and exam scenarios.

```powershell
Connect-AzAccount

# List assignments for a principal
Get-AzRoleAssignment -SignInName user@contoso.com

# Assign Reader at subscription scope
New-AzRoleAssignment `
  -SignInName user@contoso.com `
  -RoleDefinitionName Reader `
  -Scope "/subscriptions/<subscription-id>"

# Assign Contributor at resource group scope
New-AzRoleAssignment `
  -ObjectId "<object-id>" `
  -RoleDefinitionName Contributor `
  -Scope "/subscriptions/<subscription-id>/resourceGroups/rg-app-prod"
```

**Scope strings and resource IDs**

Azure RBAC uses resource IDs to identify scope. The format becomes more specific as you move down the hierarchy.

- Subscription scope: `/subscriptions/<subscription-id>`
- Resource group scope: `/subscriptions/<subscription-id>/resourceGroups/<rg-name>`
- Resource scope: `/subscriptions/<subscription-id>/resourceGroups/<rg-name>/providers/<provider>/<type>/<name>`

Example for a storage account:

```text
/subscriptions/11111111-2222-3333-4444-555555555555/resourceGroups/rg-app-prod/providers/Microsoft.Storage/storageAccounts/stapp01
```

Knowing the exact scope string helps you troubleshoot why an assignment does or does not apply.

**Inheritance and effective access**

Effective access is the final permission set a principal has after Azure evaluates all assignments and scope inheritance. When troubleshooting, do not assume the visible assignment list is the whole story. Check:

- Parent scopes for inherited permissions.
- Group memberships.
- Multiple role assignments at different scopes.
- Data-plane versus management-plane permissions.
- Resource locks or service-specific access controls that may affect operations separately from RBAC.

A user may have Reader on a subscription and Contributor on one resource group. In that case, they can modify resources only in the resource group where Contributor applies.

**When to use each scope**

| Scope | Best use case | Risk level |
|---|---|---|
| Management group | Central governance across many subscriptions | High if overused |
| Subscription | Shared platform administration | Medium |
| Resource group | Application or workload ownership | Lower |
| Resource | Highly specific access | Lowest blast radius |

Choose the narrowest scope that still supports the job. If a team only manages one application, resource group scope is usually better than subscription scope.

**Custom roles**

Use custom roles when built-in roles do not match your operational needs. A custom role lets you define a precise set of allowed actions and data actions.

Example custom role JSON:

```json
{
  "Name": "Storage Blob Reader Limited",
  "IsCustom": true,
  "Description": "Can read blob data in selected storage accounts.",
  "Actions": [
    "Microsoft.Storage/storageAccounts/read"
  ],
  "DataActions": [
    "Microsoft.Storage/storageAccounts/blobServices/containers/blobs/read"
  ],
  "NotActions": [],
  "NotDataActions": [],
  "AssignableScopes": [
    "/subscriptions/<subscription-id>"
  ]
}
```

Use custom roles carefully:

- Keep them narrowly scoped.
- Document the business need.
- Review them periodically.
- Prefer built-in roles when they already fit the requirement.

**Troubleshooting access issues**

If a user cannot perform an action, work through the problem systematically:

1. Confirm the identity is correct and the user is signed into the expected tenant.
2. Check whether the required permission is management-plane or data-plane.
3. Review role assignments at the resource, resource group, subscription, and management group scopes.
4. Check group memberships and inherited assignments.
5. Verify the role includes the needed action or data action.
6. Confirm there is no separate service-level permission model involved.

Common symptoms:

- A user can see a resource but cannot change it: likely Reader or insufficient write permissions.
- A user can manage a resource but cannot access its contents: likely missing data actions.
- A user has access in one resource group but not another: likely scope mismatch.
- A user can assign roles but cannot edit resources: likely User Access Administrator without Contributor.

**Exam-focused distinctions**

- **Authorization** is what RBAC controls.
- **Authentication** is handled by Microsoft Entra ID.
- **Scope inheritance** flows from parent to child.
- **Least privilege** reduces risk and is the preferred design.
- **Owner** can manage access; **Contributor** cannot.
- **Reader** is view-only.
- **User Access Administrator** manages access, not resource configuration.

**Practical design guidance**

In real environments, you usually combine RBAC with groups, naming standards, and scoped delegation.

- Assign roles to groups, not individuals, for easier lifecycle management.
- Use resource group scope for application teams.
- Use subscription scope for platform operations.
- Reserve management group scope for governance and policy-driven administration.
- Review assignments regularly to remove stale access.

A simple access model might look like this:

- Platform team: Contributor at subscription scope.
- Security team: User Access Administrator at subscription scope.
- App team: Contributor at `rg-app-prod`.
- Auditors: Reader at management group scope.

This structure keeps administration manageable while preserving separation of duties.

**Quick reference**

- **RBAC**: authorization model for Azure resources.
- **Principal**: user, group, service principal, or managed identity.
- **Role definition**: permissions granted by a role.
- **Role assignment**: principal + role + scope.
- **Scope**: management group, subscription, resource group, or resource.
- **Inheritance**: parent scope permissions apply to child scopes.
- **Least privilege**: grant only what is needed.
