## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Scenario focus and lab mindset**

This lab-style topic ties together Azure identity, access control, and governance features the way they appear in enterprise environments. You are not just memorizing definitions; you are deciding which control belongs where, how access is granted, how it is reviewed, and how to avoid breaking production.

The core exam skill is to distinguish between:
- **Authentication** in Microsoft Entra ID
- **Authorization** through Azure RBAC
- **Privileged access** through Privileged Identity Management (PIM)
- **Governance controls** such as Azure Policy, locks, and access reviews

A useful mental model is:
- Entra ID answers **who you are**
- RBAC answers **what you can do**
- Policy answers **what is allowed to exist**
- Locks answer **what cannot be changed or deleted**
- PIM answers **when privileged access is active**

**Azure RBAC fundamentals in enterprise scenarios**

Azure Role-Based Access Control (RBAC) is the primary authorization system for Azure resource management. It is evaluated at the scope of the request and combines assignments from higher and lower scopes. In practice, a user may receive access from multiple role assignments, and the effective permissions are the union of allowed actions unless a specific action is not included in any assignment.

Important scope behavior:
- **Management group** scope applies to subscriptions beneath it
- **Subscription** scope applies to resource groups and resources beneath it
- **Resource group** scope applies to resources in that group
- **Resource** scope applies only to that resource

RBAC inheritance is additive:
- A role assigned at a higher scope flows down to child scopes
- A role assigned at a lower scope adds to the effective permissions
- A lower-scope assignment does not remove permissions granted above it

> Locks and Azure Policy do not stop RBAC inheritance. They can block or alter operations, but they do not change how RBAC assignments flow through scope.

A practical example:
- A user has **Reader** at the subscription
- The same user has **Contributor** on one resource group
- The user can read everything in the subscription and manage resources only in that one resource group

**Built-in roles and when to use them**

Built-in roles are the fastest way to grant common access patterns. In exam scenarios, choose the least-privileged role that matches the task.

Common management-plane roles:
- **Owner**: full access plus role assignment management
- **Contributor**: manage resources but cannot grant access
- **Reader**: view-only access
- **User Access Administrator**: manage role assignments without managing resources

Data-plane roles are separate from management-plane roles:
- **Storage Blob Data Reader**: read blob data
- **Storage Blob Data Contributor**: read/write blob data
- **Key Vault Secrets User**: read secrets from Key Vault data plane
- **Key Vault Administrator**: manage Key Vault data-plane permissions and objects, depending on configuration

This distinction matters because:
- A user can be **Contributor** on a storage account and still not read blob contents
- A user may need a data-plane role to access secrets, files, or blobs even if they can manage the resource in Azure

A quick CLI check for a built-in role:
```bash
az role definition list --name "Contributor" --query "[0].permissions" -o json
```

If multiple matches are possible, inspect the full object:
```bash
az role definition list --name "Reader" -o jsonc
```

**Role assignment patterns and effective access**

Enterprise environments often assign roles to groups rather than individuals. This reduces administrative overhead and makes access changes easier to audit.

Typical pattern:
- Create an Entra ID security group
- Assign the group a role at the required scope
- Add or remove users from the group as job responsibilities change

Why this matters:
- Group-based assignment scales better than per-user assignment
- Access changes take effect when group membership updates are processed
- It supports joiner-mover-leaver workflows

Example:
- `Ops-VM-Admins` group gets **Virtual Machine Contributor** on a resource group
- New operators are added to the group
- They inherit the same access without creating new role assignments

Useful CLI examples:
```bash
az role assignment create \
  --assignee-object-id <group-object-id> \
  --role "Virtual Machine Contributor" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg-name>
```

List assignments at a scope:
```bash
az role assignment list \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg-name> \
  -o table
```

**Management group governance**

Management groups are used to organize subscriptions and apply governance consistently across a large tenant. They are especially useful when you need standard policy, RBAC, and compliance boundaries across many subscriptions.

Use management groups when you need:
- Shared policy across multiple subscriptions
- Centralized access control for platform teams
- A hierarchy that mirrors business units or environments

Practical considerations:
- Not every role assignment scenario is ideal at management group scope
- Some operations are easier to manage at subscription or resource group scope
- Moving subscriptions between management groups can change inherited policy and access, so plan carefully

A common enterprise layout:
- `Tenant Root Group`
  - `Platform`
    - `Shared Services`
    - `Connectivity`
  - `Landing Zones`
    - `Prod`
    - `NonProd`

This structure helps you:
- Apply baseline policy at the top
- Delegate subscription administration lower in the tree
- Separate production from nonproduction controls

**Azure Policy: control what can be deployed**

Azure Policy evaluates resources against rules. It is used for compliance, standardization, and guardrails. Unlike RBAC, it does not grant access; it constrains resource configuration.

Common policy effects:
- **Deny**: block noncompliant deployments
- **Audit**: allow but flag noncompliance
- **Append**: add fields to requests
- **Modify**: change request properties before deployment
- **DeployIfNotExists**: deploy a related resource if it is missing

A simple tag policy example:
```json
{
  "mode": "Indexed",
  "policyRule": {
    "if": {
      "field": "tags['Environment']",
      "exists": "false"
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

A more realistic modify policy often needs a managed identity and remediation:
- The policy assignment uses a system-assigned or user-assigned managed identity
- The identity must have permissions to make the required changes
- Existing noncompliant resources may need a remediation task

Example remediation scenario:
- You assign a policy that adds a diagnostic setting using `deployIfNotExists`
- New resources are evaluated at creation time
- Existing resources are not automatically fixed unless you run remediation

Operational steps:
1. Assign the policy at the correct scope.
2. Enable or attach the managed identity if the effect requires deployment or modification.
3. Grant the identity the permissions it needs.
4. Create a remediation task for existing resources when required.

CLI examples:
```bash
az policy assignment create \
  --name "require-tags" \
  --display-name "Require Environment Tag" \
  --scope /subscriptions/<sub-id> \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>
```

List policy assignments:
```bash
az policy assignment list --scope /subscriptions/<sub-id> -o table
```

Check compliance state:
```bash
az policy state list --scope /subscriptions/<sub-id> -o table
```

**Resource locks and operational protection**

Locks protect resources from accidental change. They are useful for critical infrastructure, but they do not replace RBAC or policy.

Lock types:
- **CanNotDelete**: users can read and modify, but cannot delete the locked scope
- **ReadOnly**: users can read, but cannot modify or delete the locked scope

Important behavior:
- Locks are inherited by child resources
- A lock at a resource group can block deletion of resources inside it
- A lock at a child resource can block deletion of that child resource
- Some update operations may also fail under a ReadOnly lock

Use locks for:
- Production resource groups
- Shared networking components
- Critical recovery resources

Example:
```bash
az lock create \
  --name "rg-protect" \
  --lock-type CanNotDelete \
  --resource-group <rg-name>
```

List locks:
```bash
az lock list --resource-group <rg-name> -o table
```

> A lock does not stop RBAC inheritance. It only restricts operations at the locked scope and below.

**Privileged Identity Management (PIM)**

PIM provides just-in-time privileged access. It is a governance control for reducing standing privilege and improving auditability.

Use PIM when you want:
- Eligible rather than permanent privileged access
- Approval before activation
- Time-bound elevation
- MFA (Multi-Factor Authentication) and justification at activation

Common PIM objects:
- Azure resource roles
- Entra ID directory roles
- Privileged access groups

Typical workflow:
1. Make a user eligible for a role.
2. Require activation settings such as MFA, justification, or approval.
3. The user activates the role when needed.
4. Access expires after the configured duration.

Example use case:
- A help desk operator is eligible for **User Access Administrator**
- They activate it only when they need to troubleshoot role assignments
- The activation is logged and time-limited

Portal configuration path:
1. Open **Microsoft Entra ID**.
2. Go to **Privileged Identity Management**.
3. Select the target role or resource.
4. Add an eligible assignment.
5. Configure activation requirements.
6. Review audit logs and activation history.

CLI and automation are possible through Microsoft Graph and Azure PowerShell in many environments, but portal-based configuration is common in labs and exams.

**Conditional Access and how it differs from RBAC**

Conditional Access is an Entra ID sign-in control. It decides whether a sign-in is allowed and under what conditions. It is not an Azure RBAC permission system.

Use Conditional Access to enforce:
- MFA
- Compliant device requirements
- Location-based restrictions
- Session controls

Key difference:
- **RBAC** controls access to Azure resources after authentication
- **Conditional Access** controls the sign-in experience before or during token issuance

Example enterprise pattern:
- Break-glass accounts are excluded from Conditional Access
- Normal admin accounts require MFA and compliant devices
- PIM activation may still require Conditional Access compliance before access is granted

This matters because a user can be assigned the correct Azure role and still fail to access resources if Conditional Access blocks the sign-in.

**Access reviews and entitlement management**

Entra ID governance features help you review and manage access over time. These are especially important for group-based access and privileged roles.

Access reviews are used to:
- Periodically verify whether users still need access
- Review group membership
- Review application access
- Review privileged role assignments in supported scenarios

Operationally, you configure:
- The review target, such as a group or role
- Reviewers, such as managers or resource owners
- The schedule and recurrence
- Automatic removal of access if not approved

Entitlement management is used to:
- Package access into access packages
- Allow users to request access through a governed workflow
- Apply approval, expiration, and review policies

Typical enterprise use:
- Contractors request access to a project group through an access package
- Approval is required from the project owner
- Access expires automatically after the engagement ends

Portal path for access reviews:
1. Open **Microsoft Entra ID**.
2. Go to **Identity Governance**.
3. Select **Access reviews**.
4. Create a review for a group, application, or privileged role.
5. Configure reviewers, recurrence, and auto-apply results.

**Break-glass accounts and emergency access**

Break-glass accounts are emergency access accounts used when normal identity controls fail. They should be highly protected and rarely used.

Best practices:
- Keep at least two emergency accounts
- Exclude them from Conditional Access policies that could lock out administrators
- Use strong passwords and secure storage
- Monitor sign-ins and alert on use
- Avoid assigning them everyday roles unless necessary

Why they matter:
- They provide a recovery path if PIM, Conditional Access, or federation issues block normal admin access
- They reduce the risk of complete tenant lockout

**Common enterprise lab scenarios**

Scenario 1: A team needs to manage VMs in one subscription only
- Assign **Virtual Machine Contributor** at the resource group
- Use a security group for the assignment
- Avoid subscription-wide Contributor unless required

Scenario 2: A storage team needs to read blobs but not manage the account
- Assign a data-plane role such as **Storage Blob Data Reader**
- Do not rely on management-plane Contributor alone

Scenario 3: Production resources must not be deleted
- Apply a **CanNotDelete** lock to the resource group
- Use RBAC to limit who can modify the resources
- Use Policy to enforce tags or configuration standards

Scenario 4: A security admin needs temporary access to role assignments
- Make the admin eligible for **User Access Administrator** in PIM
- Require MFA and justification
- Set a short activation window

Scenario 5: Contractors need time-limited access to a project group
- Use an access package or governed group membership
- Add approval and expiration
- Run access reviews to confirm continued need

**Troubleshooting access issues**

When access fails, check the layers in order:
1. Confirm the user authenticated successfully.
2. Verify the correct Azure role assignment and scope.
3. Check whether the role is management-plane or data-plane.
4. Review Conditional Access for sign-in blocks.
5. Check PIM activation status if the role is eligible.
6. Inspect locks and policy for blocked operations.
7. Confirm group membership and propagation timing.

Common symptoms:
- User can see a resource but cannot modify it: likely insufficient RBAC or a lock
- User can manage a storage account but not read blobs: missing data-plane role
- Deployment fails with policy error: policy deny or modify requirement not met
- Delete fails unexpectedly: lock at the resource group or child resource scope
- Admin role appears assigned but access is denied: PIM not activated or Conditional Access blocked sign-in

**CLI and portal quick reference**

Useful Azure CLI commands:
```bash
az role assignment list --scope /subscriptions/<sub-id> -o table
az role assignment create --assignee <user-upn-or-object-id> --role Reader --scope /subscriptions/<sub-id>
az lock list --resource-group <rg-name> -o table
az policy assignment list --scope /subscriptions/<sub-id> -o table
az policy state list --scope /subscriptions/<sub-id> -o table
```

Useful portal locations:
- **Microsoft Entra ID** for users, groups, Conditional Access, PIM, and governance
- **Subscriptions** for RBAC at subscription scope
- **Resource groups** for scoped access and locks
- **Policy** for assignments, compliance, and remediation

**Exam-ready distinctions**

- **RBAC** grants permissions
- **Policy** enforces standards
- **Locks** protect against accidental operations
- **PIM** makes privilege temporary
- **Conditional Access** controls sign-in conditions
- **Access reviews** validate continued need
- **Entitlement management** packages and governs access requests

If a question asks what to use for:
- Temporary admin access: **PIM**
- Preventing deletion: **CanNotDelete lock**
- Enforcing tags or allowed SKUs: **Azure Policy**
- Granting access to Azure resources: **RBAC**
- Requiring MFA for admin sign-in: **Conditional Access**
- Periodic access validation: **Access reviews**

**Practical lab checklist**

1. Create or identify a security group for the team.
2. Assign the least-privileged RBAC role at the correct scope.
3. Verify whether the task needs a management-plane or data-plane role.
4. Add a lock if the resource must resist accidental deletion.
5. Assign policy for standards such as tags, locations, or SKUs.
6. Configure PIM for privileged roles that should not be permanent.
7. Set up access reviews for group or role membership where governance is required.
8. Test with a non-admin account and confirm effective access.

**Key takeaways**

Enterprise identity and governance is about layering controls correctly. You use Entra ID for identity, RBAC for authorization, PIM for just-in-time privilege, Policy for compliance, locks for protection, and Conditional Access for sign-in control. The exam often tests whether you can choose the right control for the right problem and understand how those controls interact without confusing one for another.
