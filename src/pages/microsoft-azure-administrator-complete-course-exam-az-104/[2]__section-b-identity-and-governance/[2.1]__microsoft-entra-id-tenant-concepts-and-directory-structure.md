## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Microsoft Entra ID tenant fundamentals**

A Microsoft Entra ID tenant is the dedicated, isolated identity boundary for an organization in Microsoft cloud services. It stores identity objects such as users, groups, devices, applications, and administrative settings. In exam terms, think of the tenant as the security and identity container that Azure and Microsoft 365 use to authenticate users and authorize access.

A tenant is typically created when an organization signs up for a Microsoft cloud service that requires one. In most cases, the first such signup creates a new tenant unless the user is joining an existing tenant. The tenant is not the same thing as a subscription:

- **Tenant**: identity boundary and directory.
- **Subscription**: billing and resource container for Azure services.
- **Directory**: the set of identity objects and policies inside the tenant.

A single tenant can have multiple Azure subscriptions, and a single organization can also operate multiple tenants for separation of environments, mergers, subsidiaries, or regulatory boundaries.

**Tenant creation, switching, and multi-tenant management**

When you create or join a tenant, you are establishing where identity objects live and where directory policies apply. In the Azure portal and Microsoft Entra admin center, you can switch between tenants if your account has access to more than one.

Common tenant management actions include:

1. Create or join a tenant from the Microsoft Entra or Azure portal experience.
2. Switch directory context to manage the correct tenant.
3. Verify that subscriptions are associated with the intended tenant.
4. Move or reassign administrative access where needed.

Practical CLI checks can help confirm the current tenant context:

```bash
az account show --query tenantId -o tsv
az account list --output table
az account set --subscription "<subscription-id>"
```

For identity object administration, Microsoft Graph PowerShell and Microsoft Graph API are the preferred modern tools. Azure CLI is useful for subscription and context checks, but it is not the primary interface for detailed Entra object management.

**Tenant, directory, and object hierarchy**

The directory contains the objects that represent identities and resources. A useful mental model is:

```text
Tenant
└── Directory
    ├── Users
    ├── Groups
    ├── Devices
    ├── Applications
    ├── Service principals
    ├── Administrative units
    └── Policies and governance settings
```

The tenant is the boundary, the directory is the identity store, and objects are the individual entries inside that store. This distinction matters because many settings are tenant-wide, while permissions and assignments are object-specific.

**Core directory objects**

Microsoft Entra ID stores several object types that you must recognize for AZ-104.

- **User**: a human identity used for sign-in and access.
- **Group**: a collection of users and, in some cases, devices for access management.
- **Device**: a registered endpoint that can be managed and used in conditional access decisions.
- **Application**: the app definition or registration that describes how an application integrates with Entra ID.
- **Service principal**: the tenant-local instance of an application used for access control and permissions.
- **Administrative unit**: a scope boundary for delegated administration over a subset of users or groups.

A practical way to think about it:

- The **app registration** is the application’s identity definition.
- The **service principal** is the usable identity for that application in a specific tenant.
- The **user**, **group**, and **device** objects are the identities and targets you assign access to.

**App registrations and enterprise applications**

App registration and enterprise application are related but not identical.

- **App registration**: the application object in the home tenant; it defines the app’s identity, redirect URIs, credentials, and permissions.
- **Enterprise application**: the service principal representation of that app in a tenant; it is what you assign users, groups, and permissions to.

For a single-tenant app, the service principal exists in the home tenant alongside the registration. For a multi-tenant app, a service principal is created in each tenant when that tenant consents to or uses the application.

This distinction is important because:

- Authentication uses the app registration metadata.
- Authorization and assignment often happen on the enterprise application/service principal.
- Troubleshooting sign-in issues often requires checking both objects.

Example Microsoft Graph PowerShell pattern:

```powershell
Connect-MgGraph -Scopes "Application.Read.All","Directory.Read.All"

Get-MgApplication -Filter "displayName eq 'Contoso App'"
Get-MgServicePrincipal -Filter "displayName eq 'Contoso App'"
```

**Users, groups, and membership models**

Groups simplify access management by letting you assign permissions once and reuse them across many resources. In Entra ID, group type determines what can be a member and what the group is best used for.

| Group type | Typical use | Common member types |
|---|---|---|
| Security group | Access control for apps, Azure roles, and resources | Users, devices, and nested groups in supported scenarios |
| Microsoft 365 group | Collaboration for Teams, SharePoint, Outlook | Users only |
| Role-assignable group | Assign Entra directory roles to a group | Users only, with stricter controls |

Important exam distinctions:

- **Microsoft 365 groups** are for collaboration, not device membership.
- **Security groups** are the general-purpose access control group.
- **Role-assignable groups** are security groups with special settings that allow directory role assignment.
- Group nesting and supported member types depend on the group type and the target service.

A role-assignable group is useful when you want to manage privileged access through group membership instead of assigning roles directly to individual admins. This improves auditability and reduces manual role sprawl.

**Role assignment mechanics**

Entra directory roles control who can administer identity objects and tenant settings. Common examples include Global Administrator, User Administrator, Groups Administrator, and Privileged Role Administrator.

You can assign roles:

- Directly to a user.
- To a role-assignable group.
- In some cases, scoped to an administrative unit.

Role assignment through groups is especially useful for operational teams because you can add or remove users from the group without changing the role assignment itself.

Example conceptual flow:

1. Create a role-assignable security group.
2. Assign the directory role to the group.
3. Add approved administrators to the group.
4. Review membership regularly.

**Administrative units and scoped administration**

Administrative units let you delegate management over a subset of users or groups instead of the entire tenant. This is useful for regional IT teams, business units, or subsidiaries.

Use administrative units when you need:

- Delegated user management for a department or region.
- Limited scope for help desk or HR-driven administration.
- Cleaner separation of duties in a large tenant.

They do not replace tenant-wide roles; they narrow the scope of some role assignments.

**Domains, UPNs, and sign-in identity**

A tenant starts with an initial `onmicrosoft.com` domain. That domain is the default domain. You can add and verify custom domains such as `contoso.com` and use them for user principal names (UPNs), email, and sign-in.

Key terms:

- **Default domain**: the initial `onmicrosoft.com` domain created with the tenant.
- **Custom domain**: a domain you own and verify in Entra ID.
- **UPN**: user principal name, the sign-in name for a user such as `alice@contoso.com`.
- **Primary domain**: the domain used as the default sign-in suffix for new users and some creation workflows; it is not always the same as the default domain.

Typical custom domain workflow:

1. Add the domain in Microsoft Entra ID.
2. Verify ownership by adding the required DNS record.
3. Update user UPNs to the new suffix if needed.
4. Set the preferred domain for sign-in defaults where appropriate.

Example PowerShell pattern with Microsoft Graph:

```powershell
Connect-MgGraph -Scopes "Domain.ReadWrite.All","User.ReadWrite.All"

Get-MgDomain
# Add and verify domain steps are typically completed in the portal with DNS validation.
```

A practical UPN change example:

```powershell
Update-MgUser -UserId "alice@contoso.onmicrosoft.com" -UserPrincipalName "alice@contoso.com"
```

> Changing a UPN suffix does not automatically change every related email address or application configuration. Check downstream dependencies such as Exchange, app claims, and scripts.

**Tenant-level licensing and feature availability**

Entra ID licensing affects which identity and governance features are available. The main editions are:

- **Free**: basic identity and directory capabilities.
- **Premium P1**: adds features such as Conditional Access, dynamic groups, self-service group management capabilities, and more advanced identity controls.
- **Premium P2**: adds identity protection and privileged identity management capabilities on top of P1.

For AZ-104, focus on how licensing changes the available control model:

- Conditional Access requires Entra ID P1 or equivalent licensing.
- Security defaults are available in tenants without Conditional Access licensing, but they are not a substitute for full policy design.
- P2 is relevant when you need stronger privileged access governance and risk-based controls.

**Security defaults and Conditional Access**

Security defaults provide a baseline security posture with built-in protections. Conditional Access provides policy-based control over sign-in conditions, device state, location, risk, and app access.

Important platform rule:

- If you use Conditional Access, security defaults must be disabled in that tenant.

Use Conditional Access when you need:

- MFA enforcement based on user, app, or location.
- Device compliance requirements.
- Risk-based access decisions.
- More granular policy control.

Use security defaults when you need:

- A simple baseline for a small tenant.
- Basic protections without custom policy design.

**Devices and join states**

Devices are directory objects that represent endpoints. The join state determines how the device is associated with Entra ID and how it participates in management and access decisions.

Common states:

- **Registered device**: the device is registered with Entra ID, often for personal or bring-your-own-device scenarios.
- **Joined device**: the device is joined to Entra ID and typically used in cloud-first managed environments.
- **Hybrid joined device**: the device is joined to both on-premises Active Directory and Entra ID.

Why this matters:

- Device state can influence Conditional Access.
- Hybrid join supports organizations that still rely on on-premises AD as a source of authority for device identity.
- Registered devices are common in BYOD scenarios where you want limited identity association without full join.

Example checks from Windows:

```powershell
dsregcmd /status
```

Look for fields such as:

- `AzureAdJoined`
- `DomainJoined`
- `WorkplaceJoined`

These help you determine whether the device is registered, joined, or hybrid joined.

**Directory synchronization and hybrid identity**

Hybrid identity connects on-premises Active Directory with Microsoft Entra ID. This is common when organizations are migrating gradually or need to keep on-premises identity as part of their source of authority.

Core synchronization and authentication patterns:

- **Password hash synchronization (PHS)**: hashes of on-premises passwords are synchronized to Entra ID; users authenticate in the cloud.
- **Pass-through authentication (PTA)**: authentication requests are validated against on-premises agents in real time.
- **Federation**: authentication is redirected to an external identity provider, often for advanced on-premises control or legacy requirements.

High-level comparison:

| Method | Where authentication is validated | Typical use |
|---|---|---|
| Password hash synchronization | Microsoft Entra ID | Simpler hybrid identity, strong cloud integration |
| Pass-through authentication | On-premises agent | Real-time on-prem validation without password hash sync |
| Federation | External identity provider | Specialized or legacy scenarios |

Source of authority matters:

- For synced users, some attributes are mastered on-premises and synchronized to Entra ID.
- Cloud-only users are managed directly in Entra ID.
- Device identity may also be hybrid-managed depending on the join model.

**Guest access and B2B collaboration**

Microsoft Entra B2B (Business-to-Business) collaboration lets external users access your tenant as guests. This is common for vendors, partners, and contractors.

Guest access can involve different identity sources depending on the guest’s account type and your cross-tenant settings:

- Home organization account.
- Microsoft account.
- Other supported external identity providers.

Use guest users when you need:

- External collaboration without creating full internal accounts.
- Controlled access to Teams, SharePoint, apps, or Azure resources.
- Auditability and lifecycle management for external identities.

Example guest invitation flow:

1. Invite the external user to the tenant.
2. The user accepts the invitation and signs in with their available identity.
3. Assign the guest to groups, apps, or resources as needed.
4. Review and remove access when the engagement ends.

PowerShell example:

```powershell
Connect-MgGraph -Scopes "User.Invite.All"

New-MgInvitation -InvitedUserEmailAddress "partner@external.com" `
  -InviteRedirectUrl "https://myapps.microsoft.com" `
  -SendInvitationMessage
```

**Governance controls and tenant restrictions**

Large tenants need controls that reduce identity sprawl and privilege creep. Common governance features include:

- **Naming policies**: enforce consistent names for groups and resources.
- **Group expiration**: automatically expire inactive groups.
- **Access reviews**: periodically review user, guest, and group access.
- **Entitlement management**: package access into catalogs and access packages.
- **Privileged Identity Management (PIM)**: provide just-in-time privileged role activation, available with Entra ID P2.

These controls help you answer exam questions about how to keep a tenant manageable over time, not just how to create objects.

**Practical admin examples**

Create a custom domain and verify it:

1. Add the domain in the Entra admin center.
2. Publish the required DNS verification record.
3. Confirm verification status.
4. Update user UPNs if you want sign-in names to use the new suffix.

Create a guest user:

1. Open Microsoft Entra ID.
2. Select the user invitation workflow.
3. Enter the external email address.
4. Send the invitation and assign access after acceptance.

Assign a role to a group:

1. Create a security group.
2. Enable it as role-assignable if the role supports group assignment.
3. Assign the directory role to the group.
4. Add approved users to the group.

Switch tenants in the portal:

1. Open the account or directory switcher.
2. Select the target tenant.
3. Confirm the portal context before making changes.

**Troubleshooting and exam traps**

- A tenant is not the same as a subscription.
- The initial `onmicrosoft.com` domain is the default domain, but the sign-in domain users see may be a custom domain.
- App registration and enterprise application are related but distinct objects.
- Security defaults are not the same as Conditional Access, and they cannot both be active in the same tenant.
- Microsoft 365 groups are for collaboration, not device membership.
- Hybrid identity does not mean every object is mastered on-premises; only specific synced attributes are.
- A service principal is the tenant-local identity for an application, not just a duplicate of the app registration.

**Quick reference summary**

- **Tenant**: identity boundary for the organization.
- **Directory**: the object store inside the tenant.
- **Users, groups, devices, apps**: core directory objects.
- **App registration**: application definition.
- **Enterprise application**: service principal instance in a tenant.
- **Default domain**: initial `onmicrosoft.com` domain.
- **Custom domain**: verified domain you own.
- **PHS/PTA/Federation**: hybrid authentication patterns.
- **P1/P2**: licensing tiers that unlock advanced identity features.
