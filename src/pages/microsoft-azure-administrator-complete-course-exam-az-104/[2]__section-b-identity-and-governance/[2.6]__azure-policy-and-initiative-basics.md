## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure governance overview**

Azure Policy is the control plane tool you use to enforce organizational standards and assess resource compliance. It helps you answer questions such as whether resources are deployed in approved regions, whether required tags exist, or whether specific settings are allowed.

Policy works by evaluating resource properties against rules. It does not replace role-based access control (RBAC); instead, it complements RBAC by governing *what* can be created or how it must be configured.

- **Policy definition**: the rule and effect you want Azure to evaluate.
- **Assignment**: the scope where the policy applies.
- **Compliance result**: the evaluation outcome for resources in scope.
- **Initiative**: a grouped set of related policy definitions.

**Core concepts and scope**

Azure Policy is hierarchical. Definitions are created once and then assigned at a management group, subscription, resource group, or resource scope. Child scopes inherit assignments unless you explicitly exclude them with `notScopes`.

A useful mental model is:

```text
Policy definition -> Assignment at scope -> Evaluation of resources -> Compliance reporting
Initiative definition -> Assignment at scope -> Multiple child policies evaluated together
```

Scope matters because it determines which resources are evaluated and where compliance data is aggregated. A policy assigned at the management group level can influence many subscriptions, while a resource group assignment targets only that group.

**Policy definition structure**

A policy definition is a JSON document that describes the rule Azure evaluates. The most important elements are:

- **`mode`**: controls which resource types the policy can evaluate.
  - `All` is common for most modern policies.
  - `Indexed` is used when the policy targets tag and location-related scenarios or resource types that support indexing.
- **`policyRule`**: contains the `if` condition and the `then` effect.
- **`parameters`**: makes the policy reusable by letting you pass values at assignment time.
- **`metadata`**: stores versioning, category, and descriptive information.

A simplified example:

```json
{
  "mode": "All",
  "parameters": {
    "allowedLocations": {
      "type": "Array",
      "metadata": {
        "displayName": "Allowed locations"
      }
    }
  },
  "policyRule": {
    "if": {
      "not": {
        "field": "location",
        "in": "[parameters('allowedLocations')]"
      }
    },
    "then": {
      "effect": "deny"
    }
  },
  "metadata": {
    "category": "General",
    "version": "1.0.0"
  }
}
```

The `field` value often uses an Azure Policy alias, which maps a readable policy path to a resource property. Aliases are important because Azure Policy does not usually reference raw ARM property paths directly.

**Policy aliases**

Policy aliases are the bridge between policy rules and resource properties. They let you target specific settings on a resource type without memorizing the underlying schema.

Common uses include:

- Checking whether a tag exists.
- Validating a VM size or SKU.
- Enforcing allowed locations.
- Inspecting configuration properties on supported resource providers.

Example CLI query:

```bash
az policy alias list --namespace Microsoft.Compute --resource-type virtualMachines -o table
```

Aliases are especially useful when you need to build custom policies or troubleshoot why a rule does not match the property you expect.

**Policy assignment details**

An assignment applies a policy definition or initiative to a scope. In practice, you configure:

- **Assignment name**: the identifier for the assignment.
- **Scope**: management group, subscription, resource group, or resource.
- **Parameters**: values supplied to the definition.
- **Enforcement mode**: whether the policy is actively enforced or only evaluated.
- **Identity**: required for effects such as `deployIfNotExists` and `modify`.

Assignment parameters are set in the portal or CLI when you bind the definition to a scope. The same definition can be assigned multiple times with different parameter values.

Example CLI assignment for an allowed locations policy:

```bash
az policy assignment create \
  --name "allowed-locations" \
  --display-name "Allowed locations" \
  --scope "/subscriptions/<subscription-id>" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>" \
  --params '{
    "allowedLocations": {
      "value": ["eastus", "eastus2"]
    }
  }'
```

For policies that deploy or modify resources, you typically also need a managed identity and a location for the assignment:

```bash
az policy assignment create \
  --name "deploy-diagnostics" \
  --display-name "Deploy diagnostics settings" \
  --scope "/subscriptions/<subscription-id>" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>" \
  --identity-type SystemAssigned \
  --location eastus \
  --params '{
    "logAnalyticsWorkspaceId": {
      "value": "/subscriptions/<subscription-id>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspace>"
    }
  }'
```

**Policy effects**

The effect determines what Azure does when the `if` condition matches. The most common effects for AZ-104 are:

| Effect | What it does | Typical use |
|---|---|---|
| `deny` | Blocks noncompliant requests | Prevent unsupported regions or SKUs |
| `audit` | Allows the request but records noncompliance | Visibility without blocking |
| `append` | Adds or sets supported request body fields before the resource provider processes the request | Add required tags or default values in supported scenarios |
| `modify` | Changes supported request properties during create or update | Enforce tags or normalize settings |
| `deployIfNotExists` | Deploys a related resource if it is missing | Add diagnostics, monitoring, or security settings |
| `disabled` | Turns the policy off | Temporary testing or staged rollout |

A few practical notes:

- **`deny`** is best when you must prevent drift immediately.
- **`audit`** is best when you want visibility before enforcement.
- **`append`** and **`modify`** are not interchangeable.
  - `append` works only for supported request properties and request-time changes.
  - `modify` is broader for supported resource types and can be used to correct or set properties during create or update.
- **`deployIfNotExists`** is the main effect for automatic remediation of missing related resources.

> `Exempt` is not a standard policy compliance state. Exemptions are tracked separately from compliance results.

**Compliance evaluation and timing**

Azure Policy evaluates resources automatically after assignment and during normal platform activity. Compliance is not always instant because Azure must process the resource state and aggregate results.

You should know these timing behaviors:

- New or updated resources are evaluated during create or update operations.
- Existing resources are evaluated during periodic compliance scans.
- Some changes may not appear immediately in compliance reports.
- You can trigger a scan when you need faster reporting.

CLI example:

```bash
az policy state trigger-scan --subscription <subscription-id>
```

Depending on your scope and CLI version, you can also trigger scans at resource group or management group scope. Use the scope that matches the resources you want to refresh.

Compliance reporting is aggregated by scope:

- Management group compliance rolls up child subscriptions.
- Subscription compliance rolls up resource groups and resources.
- Resource group compliance shows only that group.
- Resource-level views show the specific resource state.

Policy Insights is the reporting layer behind compliance data. It is useful for dashboards and queries, but it is not a real-time transaction log. Treat it as governance reporting, not as a substitute for operational monitoring.

**Compliance states and exemptions**

Typical compliance states include:

- **Compliant**: the resource matches the policy.
- **Non-compliant**: the resource violates the policy.
- **Not started**: evaluation has not completed yet.
- **Conflict**: another policy or assignment prevents a clear result.

Exemptions are separate governance objects. You use them when a resource or scope is intentionally allowed to deviate for a valid reason, such as a temporary exception or a business-approved waiver.

Important distinction:

- **`notScopes`** excludes child scopes from an assignment.
- **Exemptions** document approved exceptions and remain visible in governance reporting.

Use `notScopes` when you want to prevent evaluation at specific child scopes. Use exemptions when you want the resource to remain in scope but be formally excused.

**Initiatives**

An initiative, also called a policy set definition, groups multiple policy definitions under one assignment. Use initiatives when you want to manage a governance standard as a package instead of assigning policies one by one.

Initiatives are useful for:

- Tag governance bundles.
- Security baseline packages.
- Region and SKU control plus diagnostics requirements.
- Environment-specific standards such as production hardening.

An initiative definition contains:

- **`policyDefinitions`**: the child policies included in the set.
- **Parameters**: values that can be passed to one or more child policies.
- **Metadata**: category, version, and description.

Example initiative definition:

```json
{
  "properties": {
    "displayName": "Tagging baseline",
    "description": "Requires core tags across resources.",
    "metadata": {
      "category": "Tags"
    },
    "parameters": {
      "requiredTagName": {
        "type": "String",
        "metadata": {
          "displayName": "Required tag name"
        }
      }
    },
    "policyDefinitions": [
      {
        "policyDefinitionId": "/providers/Microsoft.Authorization/policyDefinitions/<tag-policy-id>",
        "parameters": {
          "tagName": {
            "value": "[parameters('requiredTagName')]"
          }
        }
      },
      {
        "policyDefinitionId": "/providers/Microsoft.Authorization/policyDefinitions/<another-tag-policy-id>"
      }
    ]
  }
}
```

The difference between the objects is important:

- **Initiative definition**: the reusable bundle.
- **Initiative assignment**: the bundle applied to a scope.
- **Child policy definitions**: the individual rules inside the bundle.

**When to use a policy versus an initiative**

Use this decision pattern:

- Use a **single policy** when you need one clear rule, such as allowed locations.
- Use an **initiative** when you need a governance standard with multiple related controls.
- Use **exemptions** when a valid exception is approved.
- Use **remediation** when you need Azure to correct supported existing resources.

A simple comparison:

| Item | Best for | Scope of control |
|---|---|---|
| Policy definition | One rule | Single control |
| Policy assignment | Applying one rule | One scope |
| Initiative definition | Multiple related rules | Governance bundle |
| Initiative assignment | Applying the bundle | One scope |
| Exemption | Approved exception | Specific resource or scope |
| Remediation task | Correcting supported noncompliance | Existing resources |

**Remediation**

Remediation is the process of bringing existing resources into compliance when the policy effect supports it. It is most relevant for `deployIfNotExists` and some `modify` scenarios.

Key points:

- **`deployIfNotExists`** commonly uses remediation tasks to deploy missing resources or settings.
- **`modify`** can correct supported properties during create or update, and some existing resources may also be remediated depending on the policy and assignment behavior.
- **`audit`** does not remediate.
- **`deny`** prevents future noncompliance but does not fix existing resources.

Example remediation task creation:

```bash
az policy remediation create \
  --name "remediate-diagnostics" \
  --policy-assignment "/subscriptions/<subscription-id>/providers/Microsoft.Authorization/policyAssignments/<assignment-name>" \
  --scope "/subscriptions/<subscription-id>"
```

Remediation tasks are not magic repair jobs for every policy. They only work when the policy definition and effect support corrective action.

**Portal workflow**

A practical portal workflow helps you remember the order of operations.

1. Create or select a policy definition.
2. Review the rule, parameters, and effect.
3. Assign the policy at the correct scope.
4. Set assignment parameters and enforcement mode.
5. Wait for evaluation or trigger a scan.
6. Review compliance results in Policy Insights.
7. Create an exemption if the deviation is approved.
8. Create a remediation task if the policy supports correction.

For an initiative, the workflow is similar:

1. Create the initiative definition.
2. Add child policy definitions.
3. Map initiative parameters to child policy parameters.
4. Assign the initiative at the target scope.
5. Review compliance at the initiative and child-policy levels.
6. Remediate supported noncompliance where needed.

**Real-world scenarios**

A few end-to-end examples make the governance model easier to apply.

- **Region control**
  - You create a policy that denies resource creation outside approved regions.
  - You assign it at the subscription or management group level.
  - You use an exemption only for a documented business exception.
  - You do not use remediation because the effect is preventive, not corrective.

- **Tag enforcement**
  - You create a policy or initiative that requires standard tags such as `CostCenter` and `Owner`.
  - You use `modify` or `append` where supported to help standardize new resources.
  - You use compliance reports to find missing tags on existing resources.
  - You use remediation only if the policy supports it for existing resources.

- **Security baseline**
  - You create an initiative that includes diagnostics, logging, and configuration policies.
  - You assign the initiative to production subscriptions.
  - You use remediation tasks for missing diagnostic settings where supported.
  - You use exemptions sparingly and document the business reason.

**Practical CLI examples**

List policy definitions:

```bash
az policy definition list -o table
```

Show a specific policy definition:

```bash
az policy definition show --name <policy-definition-name>
```

Create a policy assignment:

```bash
az policy assignment create \
  --name "deny-unsupported-regions" \
  --scope "/subscriptions/<subscription-id>" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>"
```

List assignments at a scope:

```bash
az policy assignment list --scope "/subscriptions/<subscription-id>" -o table
```

List compliance states:

```bash
az policy state list --subscription <subscription-id> -o table
```

Trigger a compliance scan:

```bash
az policy state trigger-scan --resource-group <rg-name>
```

Create an exemption:

```bash
az policy exemption create \
  --name "temporary-exception" \
  --scope "/subscriptions/<subscription-id>/resourceGroups/<rg-name>" \
  --policy-assignment "/subscriptions/<subscription-id>/providers/Microsoft.Authorization/policyAssignments/<assignment-name>" \
  --exemption-category Waiver \
  --display-name "Approved temporary exception"
```

**Exam-focused distinctions**

Remember these high-value differences:

- **Policy definition** is the rule.
- **Assignment** is where the rule applies.
- **Initiative** is a bundle of rules.
- **Exemption** is an approved exception.
- **Remediation** is corrective action for supported effects.
- **Compliance scan** is the evaluation process, not the policy itself.

Also remember that Azure Policy is about governance and enforcement, while RBAC is about authorization. They solve different problems and are often used together.

**Common pitfalls**

- Confusing `notScopes` with exemptions.
- Assuming every policy effect can remediate existing resources.
- Treating compliance as instant after every change.
- Forgetting that initiative parameters can map to child policy parameters.
- Assuming a policy assignment automatically fixes existing noncompliant resources.
- Using a policy alias incorrectly and targeting the wrong property.

**Quick memory aid**

- **Definition**: what to check.
- **Assignment**: where to check it.
- **Initiative**: group related checks.
- **Exemption**: approved exception.
- **Remediation**: fix supported drift.
