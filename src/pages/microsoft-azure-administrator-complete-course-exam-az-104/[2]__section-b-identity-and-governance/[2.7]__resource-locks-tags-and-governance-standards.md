## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Governance overview**

Azure governance is the set of controls you use to keep subscriptions, resource groups, and resources organized, secure, and compliant. In practice, you combine **resource locks** to prevent accidental changes, **tags** to classify and report on assets, and broader governance standards such as Azure Policy and management groups to enforce consistency at scale.

A useful mental model is:

- **Locks** protect against destructive operations.
- **Tags** describe and classify resources.
- **Policy** enforces rules and audits compliance.
- **Management groups** apply governance across many subscriptions.

**Resource locks**

Resource locks help you prevent accidental deletion or modification of critical Azure resources. They are especially useful for production databases, networking components, shared services, and resource groups that should not be changed casually.

Azure supports two lock types:

- **CanNotDelete** prevents deletion but allows read and update operations.
- **ReadOnly** prevents deletion and modification; only read operations are allowed.

Locks are applied at a scope, and the scope can be:

- A subscription
- A resource group
- A specific resource

When you apply a lock at a higher scope, it affects child resources in that scope. For example, a lock on a resource group blocks delete or write operations on resources inside that group when the operation is covered by the lock.

> A lock does not replace RBAC (Role-Based Access Control). A user may still have permission to perform an action, but the lock can block the operation at the resource provider layer.

**Lock behavior and practical impact**

Use locks when you want a safety net against human error, automation mistakes, or accidental portal actions. They are common on:

- Production resource groups
- Virtual networks and subnets
- Key infrastructure resources
- Shared platform services

A **CanNotDelete** lock is often the best default for production because it protects against accidental removal while still allowing configuration changes. A **ReadOnly** lock is stronger, but it can break normal operations because many management actions require write access.

Typical effects:

- Deleting a locked resource fails.
- Updating a locked resource fails if the lock is ReadOnly.
- Deleting a resource group fails if the group or a child scope is locked.
- Some portal actions may appear available but fail when submitted.

**CLI examples for locks**

Use the Azure CLI `az lock` commands to create and manage locks. Scope-based targeting is the safest approach because it is explicit and works well across resource types.

```bash
# Lock a resource group
az lock create \
  --name rg-prod-lock \
  --lock-type CanNotDelete \
  --resource-group rg-prod

# Lock a specific resource using an explicit scope
az lock create \
  --name vm01-lock \
  --lock-type CanNotDelete \
  --scope /subscriptions/<subId>/resourceGroups/rg-prod/providers/Microsoft.Compute/virtualMachines/vm01

# List locks at a scope
az lock list --resource-group rg-prod

# Remove a lock
az lock delete --name rg-prod-lock --resource-group rg-prod
```

For resource-level locks, using `--scope` is often clearer than relying on resource-name/type combinations. It reduces ambiguity when the resource provider has multiple nested resource types.

**PowerShell examples for locks**

In Azure PowerShell, prefer scope-based targeting when possible. The exact parameter set can vary by module version, so `-ResourceId` is the most reliable way to target a specific resource.

```powershell
# Create a lock on a resource group
New-AzResourceLock -LockName "rg-prod-lock" `
  -LockLevel CanNotDelete `
  -ResourceGroupName "rg-prod"

# Create a lock on a specific resource by resource ID
New-AzResourceLock -LockName "vm01-lock" `
  -LockLevel CanNotDelete `
  -ResourceId "/subscriptions/<subId>/resourceGroups/rg-prod/providers/Microsoft.Compute/virtualMachines/vm01"

# View locks
Get-AzResourceLock -ResourceGroupName "rg-prod"

# Remove a lock
Remove-AzResourceLock -LockName "rg-prod-lock" -ResourceGroupName "rg-prod"
```

**Lock scope and inheritance**

Locks are enforced by scope. If you lock a resource group, operations on resources inside that group are blocked when the operation falls under the locked scope. This is why a lock on a parent scope is a strong control for protecting a whole environment.

Use this rule of thumb:

- Lock the **resource group** when you want broad protection for an environment.
- Lock the **resource** when only one critical item needs protection.
- Lock the **subscription** only when you need very strong guardrails and understand the operational impact.

**Common lock pitfalls**

- A **ReadOnly** lock can interfere with portal edits, automation, and some service operations.
- Locks can block legitimate maintenance if you forget to remove them first.
- Deleting a resource group may fail if any resource inside it is locked.
- Locks are not a substitute for backups or change control.

**Tags**

Tags are name/value pairs that you attach to Azure resources and resource groups to organize, classify, and report on them. They are essential for cost management, ownership tracking, environment separation, and automation.

Common tag examples:

- `Environment=Prod`
- `Owner=platform-team`
- `CostCenter=FIN-102`
- `Application=payments`
- `Criticality=high`

Tags are useful because they let you:

- Filter resources in the portal
- Group costs in Cost Management
- Drive automation and reporting
- Support chargeback and showback
- Identify ownership and support contacts

A tag is descriptive metadata, not a security boundary. Do not use tags to enforce access control.

**Tag constraints and behavior**

Azure resources and resource groups commonly support up to 50 tags, but exact behavior can vary by resource provider and resource type. Treat the limit as a practical guideline and verify provider-specific constraints when needed.

Important tag characteristics:

- Tag names are case-insensitive in many management experiences, but you should standardize casing for consistency.
- Tag values are strings.
- Tags are not inherited automatically from a resource group to child resources.
- Some services may expose their own tagging behavior or limitations.
- Tags can be updated without redeploying the resource in many cases.

Use a naming standard so reports stay clean. For example, choose one format for environment values and stick to it:

- `Prod`, `Test`, `Dev`
- or `production`, `test`, `development`

Do not mix formats unless your organization explicitly allows it.

**CLI examples for tags**

```bash
# Add or update tags on a resource group
az group update \
  --name rg-prod \
  --set tags.Environment=Prod tags.Owner=platform-team tags.CostCenter=FIN-102

# Add a tag to a VM
az resource tag \
  --ids /subscriptions/<subId>/resourceGroups/rg-prod/providers/Microsoft.Compute/virtualMachines/vm01 \
  --tags Environment=Prod Owner=platform-team Application=payments

# Show tags on a resource group
az group show --name rg-prod --query tags

# Remove a tag by replacing the tag set
az group update \
  --name rg-prod \
  --remove tags.Owner
```

**PowerShell examples for tags**

```powershell
# Set tags on a resource group
Set-AzResourceGroup -Name "rg-prod" -Tag @{
  Environment = "Prod"
  Owner       = "platform-team"
  CostCenter  = "FIN-102"
}

# Update tags on a resource
$resource = Get-AzResource -ResourceGroupName "rg-prod" -ResourceName "vm01" -ResourceType "Microsoft.Compute/virtualMachines"
$resource.Tags["Application"] = "payments"
Set-AzResource -ResourceId $resource.ResourceId -Tag $resource.Tags -Force
```

**Tagging strategy**

A good tagging strategy is simple, mandatory for key fields, and consistent across subscriptions. Start with a small set of tags that answer the questions operations and finance ask most often.

Recommended baseline tags:

- **Environment**: `Prod`, `Test`, `Dev`
- **Owner**: team or person responsible
- **CostCenter**: finance or billing code
- **Application**: service or workload name
- **Criticality**: business importance
- **DataClassification**: public, internal, confidential, restricted

Keep the tag set small enough that people actually use it. Too many tags reduce compliance and make reporting noisy.

**Governance standards**

Governance standards are the organizational rules that define how Azure should be used. In Azure, these standards are usually implemented with a combination of:

- **Management groups** for hierarchy and scope
- **Azure Policy** for enforcement and auditing
- **RBAC** for access control
- **Tags** for classification
- **Locks** for protection
- **Naming conventions** for consistency

A practical governance model often looks like this:

- Management group for the enterprise
- Child management groups for business units or environments
- Policies assigned at management group or subscription scope
- Standard tags required on all production resources
- Locks applied to critical resource groups

**Azure Policy and tags**

Azure Policy can require tags, append tags, or deny resources that do not meet tagging standards. This is one of the most important governance tools for AZ-104 because it turns a recommendation into an enforceable rule.

Common policy effects for tags:

- **Deny** blocks creation or update if required tags are missing.
- **Append** adds a tag if it is missing.
- **Modify** can add or change tags during deployment or update.
- **Audit** reports noncompliance without blocking.

Example policy definition that audits missing `CostCenter` tags:

```json
{
  "mode": "All",
  "policyRule": {
    "if": {
      "field": "tags['CostCenter']",
      "exists": "false"
    },
    "then": {
      "effect": "audit"
    }
  }
}
```

This is a teaching example. In real environments, alias support and tag evaluation can vary by resource type, so validate the policy against the resources you manage.

A more operational policy might deny resources without an `Environment` tag:

```json
{
  "mode": "All",
  "policyRule": {
    "if": {
      "field": "tags['Environment']",
      "exists": "false"
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

**Policy assignment and remediation**

Policy assignment is how you apply a policy definition to a scope such as a management group, subscription, or resource group. If you use `audit`, you can identify noncompliant resources without disrupting workloads. If you use `deny`, you prevent drift from the standard.

A typical workflow is:

1. Define the policy.
2. Assign it at the correct scope.
3. Review compliance results.
4. Remediate existing resources if needed.
5. Enforce the policy for new deployments.

Example Azure CLI assignment pattern:

```bash
az policy assignment create \
  --name require-environment-tag \
  --policy /subscriptions/<subId>/providers/Microsoft.Authorization/policyDefinitions/<policyId> \
  --scope /subscriptions/<subId>
```

For remediation, you usually need a policy that supports remediation and a managed identity with sufficient permissions. Not every policy effect can automatically fix existing resources.

**Real-world governance scenario**

A finance application team deploys a production workload in `rg-fin-prod`.

Recommended controls:

- Apply a **CanNotDelete** lock to the resource group.
- Require tags such as `Environment`, `Owner`, `CostCenter`, and `Application`.
- Assign a policy at the subscription scope to deny resources missing `Environment`.
- Use RBAC so only the platform team can modify the resource group.
- Use Cost Management reports filtered by `CostCenter=FIN-102`.

This combination gives you protection, accountability, and reporting:

- The lock reduces accidental deletion.
- The tags support chargeback and ownership.
- The policy enforces standards for new resources.
- RBAC limits who can make changes.

**Second real-world scenario**

A shared networking subscription contains virtual networks, route tables, and private DNS zones.

Recommended controls:

- Lock the networking resource group with `CanNotDelete`.
- Tag each resource with `Owner=network-team` and `Criticality=high`.
- Use policy to require tags and audit unsupported configurations.
- Avoid `ReadOnly` unless you fully understand the impact on network operations.

This is a good example of where a stronger lock can be harmful. Network teams often need to update routes, DNS records, or peering settings, so `CanNotDelete` is usually safer than `ReadOnly`.

**Locks vs tags vs policy**

| Feature | Primary purpose | Best use case | Main limitation |
|---|---|---|---|
| Locks | Prevent accidental change or deletion | Protect production resources | Can block legitimate operations |
| Tags | Classify and report resources | Cost tracking and ownership | Do not enforce security |
| Policy | Enforce standards and audit compliance | Require tags or approved settings | Needs correct scope and design |

**Exam focus points**

- Know the difference between **CanNotDelete** and **ReadOnly**.
- Understand that locks apply by scope and affect child resources in that scope.
- Remember that tags are for organization and reporting, not access control.
- Know that Azure Policy can require, audit, append, or deny tags.
- Be able to explain why governance usually combines policy, tags, locks, and RBAC.

**Troubleshooting tips**

If a delete or update fails unexpectedly:

- Check whether a lock exists at the resource, resource group, or subscription scope.
- Verify whether the lock is `CanNotDelete` or `ReadOnly`.
- Confirm whether a policy is denying the operation.
- Review RBAC permissions, but do not assume permissions are the only issue.
- Inspect tags and naming standards if automation is failing.

If tagging is inconsistent:

- Standardize tag names and allowed values.
- Use policy to enforce required tags.
- Automate tag assignment in deployment templates or pipelines.
- Audit existing resources and remediate drift.

**Practical deployment example**

A Bicep template can apply tags during deployment so resources start compliant.

```bicep
param location string = resourceGroup().location
param environment string = 'Prod'
param owner string = 'platform-team'
param costCenter string = 'FIN-102'

resource stg 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: 'stgprod001'
  location: location
  tags: {
    Environment: environment
    Owner: owner
    CostCenter: costCenter
  }
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
}
```

This approach reduces manual tagging errors and makes policy compliance easier from the start.

**Key takeaways**

- Use **locks** to protect critical resources from accidental deletion or modification.
- Use **tags** to classify resources for cost, ownership, and reporting.
- Use **Azure Policy** to enforce tagging and governance standards.
- Apply governance at the right scope so controls are effective but not overly restrictive.
- Prefer simple, consistent standards that teams can follow every day.
