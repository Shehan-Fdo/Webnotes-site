## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure RBAC overview**

<abbr title="Role-Based Access Control">RBAC</abbr> is the authorization system you use to control who can do what in Azure. You assign a role to a security principal at a scope, and Azure evaluates that assignment to determine effective permissions.

A role assignment is the combination of:

- **Security principal**: user, group, service principal, or managed identity
- **Role definition**: built-in or custom role
- **Scope**: management group, subscription, resource group, or resource

Azure RBAC is additive. If a user receives multiple role assignments, Azure combines the allowed actions. Deny assignments exist in some scenarios, but for AZ-104 you usually focus on allow-based role assignments and scope inheritance.

**Built-in roles: what they are and why they matter**

Built-in roles are predefined role definitions maintained by Microsoft. You use them when an existing role matches the job requirement closely enough. They are the fastest and safest choice because Microsoft manages the permissions set and updates it as Azure services evolve.

Common exam-relevant built-in roles include:

- **Owner**: full access to all resources at the assigned scope, including the ability to grant access to others through `Microsoft.Authorization/roleAssignments/*`
- **Contributor**: can create and manage resources, but cannot grant access to others
- **Reader**: can view resources, but cannot make changes
- **User Access Administrator**: can manage role assignments, but does not automatically manage resources themselves

Several service-specific roles are especially important in AZ-104 scenarios:

- **Virtual Machine Contributor**: can manage virtual machines, but not the networking or storage resources attached to them unless separately granted
- **Network Contributor**: can manage network resources such as virtual networks, subnets, network interfaces, public IPs, load balancers, and network security groups
- **Storage Account Contributor**: can manage storage accounts and their configuration, but not necessarily the data inside blobs, files, queues, or tables
- **Key Vault Administrator**: manages Key Vault resources and, in RBAC-enabled vaults, can administer access to the vault
- **Key Vault Secrets Officer**: can manage secrets data-plane operations in a Key Vault that uses Azure RBAC
- **Key Vault Secrets User**: can read secrets data-plane operations in a Key Vault that uses Azure RBAC

The difference between a general role like **Contributor** and a narrower role like **Virtual Machine Contributor** is scope of capability. Contributor can manage most resource types at the assigned scope, but it does not include access management. A service-specific role limits actions to that service, which is useful for least privilege.

**How built-in roles differ in practice**

Use built-in roles when the requirement is broad and standard:

- Grant **Reader** for audit, reporting, or troubleshooting without change rights
- Grant **Contributor** for teams that deploy resources but should not delegate access
- Grant **User Access Administrator** only when someone must manage access control
- Grant service-specific roles when the user should manage one workload without broader subscription rights

A practical distinction:

- **Contributor** can create a virtual machine, but it may fail if the user also needs to create or attach network interfaces, disks, or public IPs and lacks those permissions at the relevant scope
- **Virtual Machine Contributor** focuses on VM lifecycle operations, but it does not replace network or storage roles
- **Network Contributor** is needed when the task includes creating or modifying virtual networks, subnets, or NICs
- **Storage Account Contributor** is needed for storage account configuration, while data access often requires separate data roles or shared access mechanisms

**Scope and inheritance**

Azure RBAC scopes form a hierarchy:

- Management group
- Subscription
- Resource group
- Resource

Permissions assigned at a higher scope inherit to lower scopes. For example, a role assignment at the subscription scope applies to all resource groups and resources in that subscription.

This inheritance is critical for exam questions:

- Assigning **Reader** at the subscription scope gives read access everywhere in that subscription
- Assigning **Contributor** at a resource group scope gives management rights only within that resource group
- Assigning a role at a resource scope limits access to that single resource

Effective access is the result of all applicable assignments. If a user has **Reader** at the subscription scope and **Contributor** at one resource group, the user can read everything in the subscription and modify resources in that one resource group.

> A lower-scope assignment does not remove permissions inherited from a higher scope; it only adds permissions. To reduce access, you must change or remove the broader assignment.

**Common built-in roles for AZ-104**

| Role | What it can do | What it cannot do |
|---|---|---|
| Owner | Manage resources and access | Nothing within the assigned scope, except service-specific limits |
| Contributor | Manage resources | Grant access to others |
| Reader | View resources | Modify resources |
| User Access Administrator | Manage role assignments | Create or manage most resources by itself |
| Virtual Machine Contributor | Manage VMs | Manage access or unrelated resources by default |
| Network Contributor | Manage network resources | Manage compute or storage resources by default |
| Storage Account Contributor | Manage storage accounts | Access data in blobs/files/queues/tables by default |
| Key Vault Administrator | Administer Key Vault configuration and access in RBAC mode | Automatically read secret values unless data-plane permissions allow it |
| Key Vault Secrets Officer | Manage secrets in RBAC-enabled vaults | Manage unrelated Azure resources |
| Key Vault Secrets User | Read secrets in RBAC-enabled vaults | Create or delete secrets |

**Key Vault authorization model**

Some Azure services use their own authorization model in addition to or instead of Azure RBAC. Key Vault is a common exam example.

Key Vault can use:

- **Azure RBAC** for management and data-plane authorization in RBAC-enabled vaults
- **Access policies** in the older model, where secret, key, and certificate permissions are granted directly on the vault

This matters because the correct role depends on the vault configuration:

- If the vault uses **Azure RBAC**, you assign roles such as **Key Vault Secrets User** or **Key Vault Administrator**
- If the vault uses **access policies**, RBAC roles alone may not grant secret access
- If a question says the user can manage the vault but cannot read secrets, check whether the data-plane permission model is separate from management permissions

Other Azure services may also have service-specific authorization patterns, so always read the scenario carefully. In exam questions, the service’s own model can override your assumption that a general Azure role is enough.

**Custom roles: when built-in roles are not enough**

Custom roles let you define a permission set tailored to a specific job. Use them when built-in roles are too broad or too narrow.

Typical reasons to create a custom role:

- A team needs to manage only a subset of actions on a resource type
- You want least privilege for a specialized operational task
- You need to exclude a sensitive action from a broader built-in role
- You need a role that spans multiple services in a precise way

A custom role definition contains:

- **Name**
- **Description**
- **Permissions**
  - `actions`
  - `notActions`
  - `dataActions`
  - `notDataActions`
- **Assignable scopes**

`actions` and `notActions` control management-plane permissions. `dataActions` and `notDataActions` control data-plane permissions for services that support them.

**Custom role limitations and constraints**

Custom roles are powerful, but they have important limits:

- You can create a limited number of custom roles per subscription and per tenant; the exact limit depends on the Azure environment and is commonly tested as a platform constraint rather than a memorized number
- Role names must be unique within the directory/tenant context where they are created
- Role definition names and IDs must be unique
- A custom role cannot include every possible permission in every scenario
- Some permissions are not allowed in custom roles, especially certain privileged or unsupported operations
- You cannot use custom roles to bypass service-specific authorization models
- `notActions` and `notDataActions` only subtract from the permissions granted by `actions` and `dataActions`; they do not create new permissions
- Assignable scopes must be within the scope where the role is defined or below it

For exam purposes, remember the practical rule: if Azure does not expose a permission in the role definition model for custom roles, you cannot invent it.

**AssignableScopes rule**

`assignableScopes` defines where the custom role can be assigned. It is a list of scopes such as a subscription or management group.

Example of a valid relationship:

- You define a custom role at the subscription scope
- You set `assignableScopes` to that subscription
- You can assign the role to a resource group or resource inside that subscription

Example of an invalid relationship:

- You define a custom role with `assignableScopes` set to Subscription A
- You try to assign it in Subscription B
- Azure rejects the assignment because Subscription B is outside the allowed assignable scope

A useful mental model:

- **Definition scope** controls where the role exists
- **Assignable scope** controls where the role can be used
- **Assignment scope** controls where the permissions apply

**Creating a custom role with Azure CLI**

You usually create a custom role by exporting a JSON template, editing it, and then creating the role.

Example JSON:

```json
{
  "Name": "VM Restart Operator",
  "IsCustom": true,
  "Description": "Can restart virtual machines and read their properties.",
  "Actions": [
    "Microsoft.Compute/virtualMachines/read",
    "Microsoft.Compute/virtualMachines/restart/action"
  ],
  "NotActions": [],
  "DataActions": [],
  "NotDataActions": [],
  "AssignableScopes": [
    "/subscriptions/00000000-0000-0000-0000-000000000000"
  ]
}
```

Create the role:

```bash
az role definition create --role-definition vm-restart-operator.json
```

Verify it:

```bash
az role definition list --name "VM Restart Operator" -o table
```

**Creating a custom role in the Azure portal**

You can also create custom roles in the portal, which is a common practical scenario.

1. Open the Azure portal and go to **Subscriptions** or **Management groups**.
2. Select the scope where you want the role definition to be available.
3. Open **Access control (IAM)**.
4. Select **Roles** and then **+ Add** or **+ Create custom role**.
5. On the **Basics** tab, enter the role name and description.
6. On the **Permissions** tab, add allowed actions and remove any unnecessary permissions.
7. On the **Assignable scopes** tab, choose the scopes where the role can be assigned.
8. Review and create the role.

Portal creation is useful when you want to visually inspect permissions, but CLI is often better for repeatability and source control.

**Assigning built-in and custom roles**

You assign roles the same way whether the role is built-in or custom.

Azure CLI example:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Virtual Machine Contributor" \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app
```

Assign a custom role:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "VM Restart Operator" \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm01
```

PowerShell example:

```powershell
New-AzRoleAssignment `
  -ObjectId (Get-AzADUser -UserPrincipalName "user@contoso.com").Id `
  -RoleDefinitionName "Network Contributor" `
  -Scope "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-network"
```

**Effective permissions and overlapping assignments**

Effective access is the union of all allowed permissions from all applicable assignments, minus any explicit exclusions in the role definition and any service-specific constraints.

Important exam scenarios:

- A user has **Reader** at subscription scope and **Contributor** at resource group scope
  - The user can read all resources in the subscription
  - The user can modify resources only in that resource group
- A user has **Virtual Machine Contributor** at a resource group and **Network Contributor** at the subscription
  - The user can manage VMs in the resource group
  - The user can manage network resources across the subscription
  - The user may still fail a VM deployment if the deployment requires permissions not covered by the combination at the exact resource scope
- A user has a custom role at a resource scope and a broader Reader role at the subscription
  - The user can perform the custom actions on that resource
  - The user can still view other resources in the subscription

When troubleshooting, distinguish between:

- **Role assignment inheritance**: permissions flow down from parent scopes
- **Effective permissions**: the final set after combining all assignments and service rules

**Practical end-to-end scenario**

Requirement: A support engineer must restart virtual machines in one resource group, view their details, and must not be able to change network settings or grant access.

Recommended approach:

1. Choose a role that matches the task.
   - `Virtual Machine Contributor` is a strong candidate if the engineer needs VM lifecycle management.
2. Choose the narrowest scope that satisfies the requirement.
   - Use the resource group containing the VMs, not the subscription.
3. Verify whether additional roles are needed.
   - If the engineer must only restart VMs and not manage all VM settings, a custom role may be better.
4. Assign the role.
5. Test the access.
   - Confirm the engineer can restart a VM.
   - Confirm the engineer cannot modify network resources or role assignments.

Example custom role for a narrower requirement:

```json
{
  "Name": "VM Restart and Read Only",
  "IsCustom": true,
  "Description": "Can read and restart virtual machines only.",
  "Actions": [
    "Microsoft.Compute/virtualMachines/read",
    "Microsoft.Compute/virtualMachines/restart/action"
  ],
  "NotActions": [],
  "DataActions": [],
  "NotDataActions": [],
  "AssignableScopes": [
    "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app"
  ]
}
```

This is more restrictive than **Virtual Machine Contributor** because it excludes broader VM management actions.

**Troubleshooting access**

When a user cannot perform an action, check in this order:

1. Confirm the correct principal was assigned.
2. Confirm the role is assigned at the correct scope.
3. Check whether the role is inherited from a parent scope.
4. Check whether another assignment grants broader access than expected.
5. Check whether the service uses a separate authorization model.
6. Verify whether the action is management-plane or data-plane.
7. Confirm the role definition actually includes the needed action.

Useful CLI commands:

```bash
az role assignment list --assignee user@contoso.com --all -o table
az role definition list --name "Virtual Machine Contributor" -o jsonc
az role assignment list --scope /subscriptions/00000000-0000-0000-0000-000000000000 -o table
```

For Key Vault, also verify the vault’s permission model and whether the user needs a data-plane role or access policy.

**Exam patterns to recognize**

- If the question asks for the least privilege role to manage access, think **User Access Administrator**
- If the question asks for full resource management but no access delegation, think **Contributor**
- If the question asks for read-only visibility, think **Reader**
- If the question asks for a workload-specific admin role, think of a service-specific built-in role before creating a custom role
- If the question asks for a very narrow permission set, think **custom role**
- If the question involves secrets, keys, or certificates, check whether the service uses **Azure RBAC** or **access policies**

**Quick comparison of built-in vs custom roles**

| Aspect | Built-in role | Custom role |
|---|---|---|
| Managed by | Microsoft | You |
| Speed of use | Immediate | Requires definition and testing |
| Least privilege fit | Sometimes approximate | Usually precise |
| Maintenance | Automatic updates | Manual updates |
| Exam use case | Standard administrative tasks | Specialized or restricted tasks |

**Key takeaways**

- Use built-in roles first when they match the requirement.
- Use custom roles when built-in roles are too broad or too narrow.
- Scope determines where permissions apply, and higher scopes inherit downward.
- Effective access is the combination of all applicable assignments.
- Some services, especially Key Vault, may use their own authorization model or data-plane roles.
- For AZ-104, always match the role, scope, and service authorization model to the scenario.
