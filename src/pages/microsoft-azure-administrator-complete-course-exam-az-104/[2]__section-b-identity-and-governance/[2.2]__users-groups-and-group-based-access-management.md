## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Identity administration in Azure**

Microsoft Entra ID is the identity plane for Azure administration. For AZ-104, you need to understand how users, groups, and directory roles work together to control access to Azure resources and Microsoft 365-style collaboration features. The exam often tests whether you choose direct assignment, group-based assignment, or a directory role for a given scenario.

A useful mental model is:

- **Users** represent people or service identities in the directory.
- **Groups** collect users and sometimes devices for easier management.
- **Roles** grant administrative permissions in Entra ID or Azure.
- **Azure RBAC** controls access to Azure resources such as subscriptions, resource groups, and resources.

The key exam skill is choosing the right layer of control. If you need someone to manage users, you use an Entra role. If you need someone to read a storage account, you use Azure RBAC. If you need to simplify repeated access grants, you use a group.

**Microsoft Entra users**

A user object is the identity you authenticate and authorize. In Entra ID, users can be created in several ways:

- **Cloud-only users** are created directly in Entra ID.
- **Synced users** come from an on-premises directory through Microsoft Entra Connect.
- **Guest users** are external identities invited into your tenant for collaboration.

User properties commonly relevant to AZ-104 include:

- Display name
- User principal name (`user@domain.com`)
- Usage location
- Job title, department, and company name
- Account enabled/disabled state
- Assigned roles and group membership

When you manage users, remember that not every admin can do everything. Typical constraints include:

- A **User Administrator** can create and manage most users and groups, but not all privileged role assignments.
- A **Global Administrator** has broad control across the tenant.
- A **Privileged Role Administrator** manages role assignments in Entra ID.
- A **Groups Administrator** manages groups, but not necessarily all user lifecycle tasks.
- A **User Administrator** can usually create users and reset passwords for non-administrative users, but cannot manage some privileged accounts unless additional permissions are granted.

> In exam scenarios, if the task says “create users and reset passwords for standard employees,” a User Administrator is often sufficient. If it says “assign Entra roles,” you usually need a role with privileged role management capability.

**User creation and management workflow**

In the Entra admin center, you typically manage users from the Users blade. Common tasks include creating a user, resetting a password, blocking sign-in, assigning groups, and reviewing audit activity.

A typical portal workflow is:

1. Open the Entra admin center.
2. Go to **Users**.
3. Select **New user** or open an existing user.
4. Set identity details such as name, username, and password.
5. Assign groups or roles if required.
6. Save and verify the account status.

Example Azure CLI commands for user management:

```bash
# Create a cloud-only user
az ad user create \
  --display-name "Ava Patel" \
  --user-principal-name ava.patel@contoso.com \
  --password "P@ssw0rd-ChangeMe123"

# Show a user
az ad user show --id ava.patel@contoso.com

# Block sign-in for a user
az ad user update \
  --id ava.patel@contoso.com \
  --account-enabled false
```

For Microsoft Graph PowerShell, the modern approach is often preferred over older AzureAD module examples:

```powershell
Connect-MgGraph -Scopes "User.ReadWrite.All"

New-MgUser `
  -AccountEnabled `
  -DisplayName "Ava Patel" `
  -MailNickname "ava.patel" `
  -UserPrincipalName "ava.patel@contoso.com" `
  -PasswordProfile @{ Password = "P@ssw0rd-ChangeMe123"; ForceChangePasswordNextSignIn = $true }
```

Common user-management outcomes to recognize:

- A user can be created without being assigned Azure resource access.
- A user can be a member of many groups.
- A user can be a guest with limited collaboration permissions.
- A user can be blocked from sign-in while still existing in the directory.

**Guest users and external collaboration**

Guest users are external identities invited into your tenant. They are not the same as internal member users. This distinction matters in administration and collaboration scenarios.

Guest access is typically used when:

- You need to share files, apps, or resources with an external partner.
- You want limited access without creating a full internal account.
- You need to collaborate across tenants.

Guest users usually have different default permissions and lifecycle behavior than members:

- They are often restricted by external collaboration settings.
- They may not be able to perform the same administrative actions as members.
- They are commonly managed through invitation and redemption workflows.
- Their sign-in and access can be governed by conditional access and entitlement policies, depending on tenant configuration.

A common workflow is:

1. Invite the external user.
2. The user redeems the invitation.
3. The guest account appears in your directory.
4. You assign the guest to groups or resources as needed.
5. You review access periodically and remove it when collaboration ends.

Example invitation with Microsoft Graph PowerShell:

```powershell
Connect-MgGraph -Scopes "User.Invite.All"

New-MgInvitation `
  -InvitedUserEmailAddress "partner.user@external.com" `
  -InviteRedirectUrl "https://myapps.microsoft.com" `
  -SendInvitationMessage
```

Practical exam point:

- If the scenario says “external consultant needs access to one app and one SharePoint site,” a guest user is usually the right identity type.
- If the scenario says “employee needs access to internal Azure resources,” use a member user.

**Groups in Microsoft Entra ID**

Groups simplify access management by letting you assign permissions once to a collection of users. This reduces repetitive administration and lowers the chance of inconsistent access.

Common group types include:

- **Security groups** for access control.
- **Microsoft 365 groups** for collaboration across Outlook, Teams, SharePoint, and related services.
- **Mail-enabled security groups** for both email distribution and security-based access scenarios.

Group properties and concepts often tested in AZ-104:

- **Membership type**
  - Assigned: you add members manually.
  - Dynamic user: membership is rule-based.
  - Dynamic device: membership is rule-based for devices.
- **Ownership**
  - Owners can manage membership and some settings.
- **Expiration**
  - Group expiration can help with lifecycle governance.
- **Naming policy**
  - Helps standardize group names.
- **Role-assignable groups**
  - Special security groups that can be assigned Microsoft Entra roles.

A quick comparison:

| Group type | Primary use | Can be used for access control | Can be mail-enabled |
|---|---|---:|---:|
| Security group | Authorization | Yes | Sometimes, if mail-enabled security group |
| Microsoft 365 group | Collaboration | Limited for Azure RBAC scenarios | Yes |
| Mail-enabled security group | Email + access | Yes | Yes |
| Role-assignable group | Entra role assignment | Yes, for directory roles | No typical collaboration use |

**Group creation and management**

You can create and manage groups in the Entra admin center. Common tasks include creating the group, adding members, adding owners, reviewing membership, and deleting or expiring the group.

Portal actions to know:

- Create a group from **Groups** > **New group**.
- Choose the group type and membership type.
- Add members directly or configure dynamic rules.
- Add owners so group administration is delegated.
- Review **Members** and **Owners** tabs to audit access.
- Remove users when access is no longer needed.

Example Azure CLI group creation:

```bash
# Create a security group
az ad group create \
  --display-name "RG-Readers" \
  --mail-nickname "rg-readers"

# Add a member to the group
az ad group member add \
  --group "RG-Readers" \
  --member-id "11111111-1111-1111-1111-111111111111"

# Show group members
az ad group member list --group "RG-Readers"
```

Example Microsoft Graph PowerShell group creation:

```powershell
Connect-MgGraph -Scopes "Group.ReadWrite.All"

New-MgGroup `
  -DisplayName "RG-Readers" `
  -MailEnabled:$false `
  -MailNickname "rg-readers" `
  -SecurityEnabled:$true
```

Important management behaviors:

- Owners can often add or remove members without being full administrators.
- Membership changes are reflected in access checks after propagation.
- Dynamic groups reduce manual work but require correct rule design.
- Group expiration can remove stale groups from the tenant lifecycle.

**Role-assignable groups**

Role-assignable groups are a critical AZ-104 concept. They are security groups that can be assigned Microsoft Entra directory roles. This is different from Azure RBAC role assignment at the subscription or resource level.

Use role-assignable groups when you want:

- A group of admins to share a directory role.
- Easier role lifecycle management.
- Consistent privileged access assignment.

Constraints to remember:

- Not every group can be assigned a directory role.
- Role-assignable groups are a special group type and must be created with that capability enabled.
- They are intended for Entra role assignments, not general collaboration.
- Membership and ownership of these groups should be tightly controlled.

Example use case:

- You create a role-assignable group called `Entra-Helpdesk-Admins`.
- You assign the **Helpdesk Administrator** role to the group.
- You add helpdesk staff to the group instead of assigning the role individually.

This makes onboarding and offboarding easier and reduces the risk of orphaned privileged assignments.

**Azure RBAC and group-based access**

Azure Role-Based Access Control (RBAC) authorizes access to Azure resources. You can assign roles to users, groups, service principals, and managed identities at scopes such as management group, subscription, resource group, or resource.

Group-based RBAC is preferred when:

- Multiple users need the same access.
- You want to manage access centrally.
- You need to reduce direct role assignments.

Typical pattern:

- Assign the **Reader** role to a group at the resource group scope.
- Add users to the group.
- Users inherit the access through group membership.

Example Azure CLI role assignment:

```bash
# Assign Reader to a group at resource group scope
az role assignment create \
  --assignee-object-id "22222222-2222-2222-2222-222222222222" \
  --assignee-principal-type Group \
  --role Reader \
  --scope "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app"
```

**Nested group limitations and behavior**

Nested groups are a common source of exam traps. In Azure RBAC, group nesting support is limited and should not be assumed to work everywhere. The safest exam answer is:

- Do not rely on nested groups for Azure RBAC unless the scenario explicitly confirms support in that context.
- Prefer direct membership for predictable authorization.
- If access does not appear to apply, check whether the user is only indirectly included through nested groups.

For Entra directory role assignments, nested group behavior is also limited and should not be treated as a universal solution. Role-assignable groups are the supported pattern for assigning directory roles to a group, but you should not assume nested group inheritance will solve privileged access delegation.

Practical takeaway:

- If a user needs access, add the user directly to the access group.
- If a group needs a role, use a role-assignable group where supported.
- If a scenario depends on nested groups, verify the exact service behavior before choosing it.

> For AZ-104, nested groups are usually a poor default answer when a direct group assignment is available.

**Administrative units and scoped administration**

Administrative units let you delegate management of a subset of users and groups within Entra ID. They are useful when you want regional, departmental, or business-unit admins to manage only their own population.

Use administrative units when:

- HR admins should manage only employees in one region.
- Helpdesk staff should reset passwords only for a department.
- Group administrators should manage only a subset of groups.

This is a directory-scoping concept, not an Azure RBAC scope. It helps you delegate identity administration without granting tenant-wide privileges.

Exam-relevant idea:

- If the question asks for delegated user management limited to a department, administrative units are often the right answer.
- If the question asks for access to Azure resources, use Azure RBAC instead.

**Choosing between direct assignment, group assignment, and Entra role assignment**

AZ-104 often tests the best access model for a scenario. Use this decision logic:

- **Direct user assignment**
  - Use when only one or two users need access and the access is temporary or highly specific.
  - Avoid when many users need the same permissions.
- **Security group assignment**
  - Use when multiple users need the same Azure resource access.
  - Best for scalable Azure RBAC management.
- **Entra role assignment**
  - Use when the user needs to manage identities, groups, or directory settings.
  - Often assigned to a role-assignable group for scale.

Examples:

- A developer needs read access to one storage account: assign an Azure RBAC role directly or through a security group.
- A helpdesk team needs to reset passwords: assign an Entra role such as Helpdesk Administrator, preferably through a role-assignable group.
- A project team needs access to a resource group: assign the group a resource role like Contributor or Reader.

**Common portal tasks and where to do them**

In the Entra admin center, you should know where to perform common tasks:

- **Users**
  - Create users
  - Reset passwords
  - Block sign-in
  - Assign groups and roles
- **Groups**
  - Create security groups
  - Add or remove members
  - Add or remove owners
  - Configure dynamic membership
  - Review group activity
- **Roles and administrators**
  - Assign directory roles
  - Review privileged assignments
- **Administrative units**
  - Scope delegated administration

A practical membership review process:

1. Open the group.
2. Check **Members** for current access.
3. Check **Owners** for delegated administration.
4. Review **Audit logs** for recent changes.
5. Remove stale members or expired access.

**Real-world end-to-end scenarios**

Scenario 1: Departmental Azure access

A finance team needs read-only access to a resource group containing reporting VMs and storage accounts.

Recommended approach:

- Create a security group named `Finance-RG-Readers`.
- Add finance users to the group.
- Assign the group the **Reader** role at the resource group scope.

Why this works:

- You manage access once at the group level.
- Users inherit permissions automatically.
- Offboarding is simple: remove the user from the group.

Scenario 2: Helpdesk identity administration

A support team needs to reset passwords and manage basic user attributes, but not assign subscription permissions.

Recommended approach:

- Create a role-assignable group.
- Assign the appropriate Entra directory role to the group.
- Add helpdesk staff as members.
- Optionally scope their work with administrative units.

Why this works:

- Directory administration stays separate from Azure resource access.
- Role assignment is easier to maintain than individual user assignments.

Scenario 3: External partner collaboration

A vendor needs access to a shared app and a document library for a short project.

Recommended approach:

- Invite the vendor as a guest user.
- Add the guest to a collaboration group or assign access directly as needed.
- Review access at project end and remove the guest.

Why this works:

- Guest access limits exposure.
- You avoid creating a full internal account for an external person.

**Troubleshooting access issues**

When access does not work as expected, check these common causes:

- The user is in the wrong group.
- The group has not been assigned the correct role.
- The role was assigned at the wrong scope.
- The user is a guest and lacks the expected permissions.
- The group is nested and the service does not honor that nesting for the scenario.
- The user has not signed out and back in after a role or group change.
- The access change has not fully propagated yet.

Useful checks:

```bash
# Show role assignments at a scope
az role assignment list \
  --scope "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app" \
  --output table

# Show group membership
az ad group member list --group "Finance-RG-Readers" --output table
```

If a user says they still cannot access a resource after being added to a group:

- Confirm the group has the correct Azure RBAC role.
- Confirm the role is assigned at the correct scope.
- Confirm the user is a direct member if nested membership is unsupported.
- Confirm the user is using the correct account and tenant.

**Exam tips and common traps**

- **Do not confuse** Entra directory roles with Azure RBAC roles.
- **Do not assume** Microsoft 365 groups are the best choice for Azure resource authorization.
- **Do not rely** on nested groups unless the scenario explicitly supports them.
- **Do use** security groups for scalable Azure RBAC access.
- **Do use** role-assignable groups for Entra role delegation.
- **Do use** guest users for external collaboration.
- **Do use** administrative units when delegation must be scoped to a subset of the directory.

```bash

# Add a user to the group
az ad group member add \
  --group "App-Operators" \
  --member-id "33333333-3333-3333-3333-333333333333"

# Assign Contributor to the group at subscription scope
az role assignment create \
  --assignee-object-id "44444444-4444-4444-4444-444444444444" \
  --assignee-principal-type Group \
  --role Contributor \
  --scope "/subscriptions/00000000-0000-0000-0000-000000000000"
```

These patterns are useful because they mirror how you should think during the exam: create the identity container, place users into it, then assign the permission at the correct scope.
