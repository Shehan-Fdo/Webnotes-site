## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure portal overview**

The Azure portal is the browser-based management plane for Azure resources. You use it to create, configure, monitor, and govern services through a graphical interface that maps to Azure Resource Manager (ARM) operations. For AZ-104, you should understand not just where things are, but how the portal behaves when you move between resources, scopes, and tenants.

The portal is organized around **blades**. A blade is a pane that opens from the left or from another blade, letting you drill into a resource without losing the parent context. This multi-blade model is central to portal navigation and is one of the most common sources of confusion for new administrators.

- The portal is optimized for **control-plane** tasks such as creating resources, changing configuration, assigning RBAC (role-based access control), and applying locks.
- Data-plane actions happen inside the service itself, such as reading blob contents or querying a database, and may not always be represented the same way in portal logs.
- Many portal actions generate ARM requests and can appear in the **Activity log** at the appropriate scope.

**Portal layout and navigation model**

The left navigation bar gives you entry points to common services and management surfaces. You can search globally, open favorites, and jump to recently used resources. The top bar contains the directory and subscription context switcher, notifications, settings, help, and account controls.

A practical mental model is:

```text
Home / Dashboard
  ├─ Global search
  ├─ Favorites
  ├─ All resources
  ├─ Resource groups
  ├─ Monitor
  ├─ Cost Management
  ├─ Advisor
  └─ Individual resource blades
```

- **Global search** helps you find resources, services, documentation, and marketplace items quickly.
- **Favorites** are pinned navigation items, not the same as dashboard tiles.
- **All resources** shows resources across the currently selected subscription and directory context.
- **Resource groups** is a core blade for managing grouped resources as a unit.
- **Monitor**, **Cost Management**, and **Advisor** are common admin entry points for health, spending, and recommendations.

**Directory, subscription, and tenant context**

The directory and subscription context switcher is critical because the portal only shows resources that exist in the selected scope and that your account can access. If a resource seems to be missing, the first thing to check is whether you are in the correct directory, tenant, and subscription.

- A **directory** in the portal usually refers to Microsoft Entra ID tenant context.
- A **subscription** is the billing and access boundary for Azure resources.
- A single account can have access to multiple directories and subscriptions.
- Resource visibility depends on both your selected context and your permissions.

Typical workflow when a resource is missing:

1. Open the directory and subscription switcher in the top bar.
2. Verify the correct tenant or directory is selected.
3. Check the subscription list and ensure the target subscription is enabled.
4. Confirm you have the required RBAC role at the right scope.
5. Search again using the global search box or browse through **All resources**.

> If you are in the wrong directory or subscription, the portal may appear empty even though the resource exists.

**Blades, drill-down, and multi-blade navigation**

The portal uses a stacked blade experience. When you open a resource, its overview blade appears. From there, you can open related blades such as settings, access control, locks, metrics, or activity log. Each new blade opens alongside the previous one, so you can compare context without losing your place.

Common blade behavior:

- Clicking a resource often opens its **Overview** blade.
- Selecting a menu item inside a blade opens a new blade to the right.
- You can close a blade to return to the previous context.
- Some blades open in a new browser tab or a separate page, but most remain in the blade stack.

A simple visual model:

```text
Dashboard -> Resource Group -> Virtual Machine -> Metrics
                |                  |               |
             parent blade      child blade     child blade
```

- Use the blade stack to trace dependencies, such as a VM inside a resource group.
- Close deeper blades when you want to reduce clutter and return to a higher-level scope.
- Keep in mind that some actions are scope-sensitive; the blade title usually tells you whether you are at subscription, resource group, or resource scope.

**Home, Dashboard, and pinning**

The **Dashboard** is a customizable landing page made of tiles. You can pin entire blades, specific charts, metrics, or resource views to a dashboard for quick access. This is useful when you repeatedly monitor a small set of resources or KPIs.

Pinning is not the same as favoriting:

- **Pin to dashboard** creates a tile on a dashboard.
- **Pin to favorites** adds a navigation shortcut in the portal menu.
- You can unpin dashboard tiles later if they are no longer useful.

Practical examples of what you can pin:

- A resource overview blade for a virtual machine or storage account.
- A **Metrics** chart showing CPU, requests, or latency.
- A **Resource group** overview for a project environment.
- A **Monitor** workbook or chart used during operations.

Step-by-step example: pin a resource to a dashboard

1. Open the resource blade, such as a virtual machine.
2. Select **Pin to dashboard** from the top command bar or blade menu.
3. Choose an existing dashboard or create a new one.
4. Confirm the tile appears on the dashboard.
5. Open the dashboard and rearrange or resize the tile if needed.

Step-by-step example: unpin a dashboard tile

1. Open the dashboard.
2. Hover over the tile.
3. Select the tile menu.
4. Choose **Unpin from dashboard** or remove the tile.
5. Save the dashboard if prompted.

- Pin charts when you need operational visibility.
- Pin blades when you need fast access to a resource or scope.
- Use dashboards for team views and recurring admin tasks.

**Portal settings, language, region, and theme**

The portal includes user-level settings that affect how it looks and behaves. These settings are useful for accessibility, localization, and personal workflow preferences.

Common customization options include:

- **Language and region** for portal display and formatting.
- **Theme** such as light, dark, or high-contrast variants.
- **Directory and subscription filter behavior** to control how the portal remembers or applies scope selections.
- **Notifications** and other user preferences.
- **Accessibility** options such as keyboard navigation and contrast support.

Why this matters:

- Language and region settings affect labels, date formats, and some localized experiences.
- Theme selection can improve readability in long admin sessions.
- Subscription filtering helps reduce clutter when you manage many subscriptions.
- Remembering the right context reduces mistakes when creating or modifying resources.

Example of a common admin habit:

```text
Before making changes:
1. Confirm the correct directory.
2. Confirm the correct subscription.
3. Confirm the correct resource group.
4. Confirm the correct resource.
```

**Core navigation surfaces**

Several portal entry points are especially important for AZ-104 and day-to-day administration.

- **Resource groups**: Manage resources that share a lifecycle, such as a web app, database, and storage account for one application.
- **All resources**: Quickly locate any resource you can access in the selected subscription.
- **Monitor**: Central place for metrics, alerts, logs, and service health-related workflows.
- **Cost Management**: Review spending, budgets, and cost analysis.
- **Advisor**: Review recommendations for reliability, security, performance, and cost optimization.
- **Activity log**: Review subscription-scoped control-plane events and changes.

Use the right surface for the job:

- Use **Resource groups** when you want to manage a workload as a unit.
- Use **All resources** when you know the resource name but not its group.
- Use **Monitor** when you need operational insight or alerting.
- Use **Cost Management** when you need financial visibility.
- Use **Advisor** when you want Azure recommendations for improvement.

**Resource group blade and why it matters**

The **Resource Group** blade is a core navigation concept distinct from **All resources** and from individual resource blades. A resource group is a logical container for resources that share a lifecycle, deployment, or ownership boundary.

In practice, the resource group blade helps you:

- View all resources in one application or environment.
- Apply tags, locks, and access control at a shared scope.
- Delete or move resources with awareness of dependencies.
- Review deployments and activity at the group level.

Common use case:

- You deploy a web application into a resource group containing an App Service, SQL database, Key Vault, and virtual network components.
- You use the resource group blade to inspect the whole stack instead of opening each resource separately.

**Activity log scope and event types**

The **Activity log** records control-plane events in Azure. It is available at multiple scopes, including subscription, resource group, and individual resource scope. This makes it useful for auditing who changed what and when.

The Activity log typically records events such as:

- Create, update, and delete operations on resources.
- Role assignment changes.
- Lock creation or removal.
- Resource move operations.
- Policy-related events and some administrative actions.

Important distinction:

- **Control-plane** actions change Azure resource configuration and are usually visible in the Activity log.
- **Data-plane** actions operate on the data inside a service and may be logged elsewhere, depending on the service.

Practical examples:

- If someone changes a VM size, you can often find the event in the Activity log.
- If someone reads a blob from storage, that is a data-plane action and is not the same as a portal control-plane change.
- If a resource is moved between resource groups, the Activity log can help you trace the operation.

**Common admin tasks in the portal**

The portal is often used for quick administrative changes that are easier to validate visually than through CLI alone.

Examples include:

- Changing **RBAC** assignments on a resource group or resource.
- Adding or removing **locks** to prevent accidental deletion or modification.
- Moving resources between resource groups when dependencies allow it.
- Checking **metrics** before and after a configuration change.
- Reviewing **Activity log** entries after a deployment or incident.

Step-by-step example: switch subscriptions before making a change

1. Open the directory and subscription switcher.
2. Select the correct directory if needed.
3. Enable the target subscription.
4. Search for the resource or open the resource group.
5. Verify the scope shown in the blade title before editing.

Step-by-step example: locate a resource through the portal menu

1. Open **All resources**.
2. Filter by subscription, resource type, or name.
3. Open the target resource.
4. Confirm the resource group and subscription in the overview.
5. Use the left menu to navigate to settings, monitoring, or access control.

Step-by-step example: review a resource change in Activity log

1. Open the subscription, resource group, or resource blade.
2. Select **Activity log** from the left menu.
3. Filter by time range, operation, or status.
4. Open an event to inspect the caller, timestamp, and operation details.
5. Correlate the event with the change you made in the portal.

**Practical CLI equivalents**

Even though this topic focuses on the portal, AZ-104 expects you to understand that many portal actions map to Azure CLI or PowerShell operations. This helps you validate what the portal is doing and automate repeatable tasks.

```bash
# List subscriptions you can access
az account list -o table

# Set the active subscription
az account set --subscription "Contoso-Prod"

# List resource groups in the active subscription
az group list -o table

# Show a specific resource group
az group show --name rg-app-prod

# View activity log events for a resource group
az monitor activity-log list \
  --resource-group rg-app-prod \
  --offset 24h \
  -o table
```

PowerShell example:

```powershell
Get-AzSubscription
Set-AzContext -Subscription "Contoso-Prod"
Get-AzResourceGroup -Name "rg-app-prod"
Get-AzActivityLog -ResourceGroupName "rg-app-prod" -StartTime (Get-Date).AddHours(-24)
```

- Use CLI when you need repeatability or bulk operations.
- Use the portal when you need visual confirmation, discovery, or quick troubleshooting.
- Compare portal actions with CLI output to understand scope and effect.

**Portal navigation scenarios**

Scenario 1: You cannot find a VM

- Check the directory and subscription context first.
- Search by name in the global search box.
- Open **All resources** and filter by type `Virtual machine`.
- If still missing, verify RBAC permissions and whether the VM exists in another subscription.

Scenario 2: You need to review a failed deployment

- Open the resource group blade.
- Check **Deployments** for ARM deployment history.
- Review the **Activity log** for failed or denied operations.
- Open the affected resource blade if the issue is isolated to one service.

Scenario 3: You need to monitor a production workload

- Open the resource or resource group.
- Pin a key **Metrics** chart to the dashboard.
- Use **Monitor** for alerts and log queries.
- Add the dashboard tile to a shared operational dashboard.

Scenario 4: You need to change access safely

- Open the resource group or resource blade.
- Go to **Access control (IAM)**.
- Review existing role assignments before adding a new one.
- Confirm the scope is correct so you do not grant access too broadly.

**Exam-focused takeaways**

- The Azure portal is a **scope-aware** UI built around blades and context.
- The directory/subscription switcher explains why resources may not appear.
- Resource groups are a primary management boundary, not just a folder.
- The Activity log is for **control-plane** auditing at subscription, resource group, and resource scope.
- Dashboards are for pinned tiles and charts; favorites are for navigation shortcuts.
- Monitor, Cost Management, and Advisor are important admin surfaces you should recognize quickly.

**Quick reference**

| Concept | What it does | Why it matters |
|---|---|---|
| Blade | Opens a contextual pane for a resource or setting | Preserves navigation context |
| Dashboard tile | Pinned view or chart | Supports monitoring and quick access |
| Favorites | Portal navigation shortcut | Speeds up repeated navigation |
| Resource group | Logical container for related resources | Simplifies lifecycle management |
| Activity log | Records control-plane events | Supports auditing and troubleshooting |
| Directory/subscription switcher | Changes visible scope | Prevents “missing resource” confusion |

**Common mistakes to avoid**

- Assuming a resource is missing when you are simply in the wrong subscription.
- Confusing dashboard pinning with favorites.
- Looking for data-plane actions in the Activity log when the service logs them elsewhere.
- Editing a resource without checking the blade scope first.
- Treating a resource group like a folder instead of a management boundary.
