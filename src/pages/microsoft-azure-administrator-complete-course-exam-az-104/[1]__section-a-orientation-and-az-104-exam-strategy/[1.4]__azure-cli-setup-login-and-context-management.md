## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure CLI overview**

The Azure CLI is Microsoft’s cross-platform command-line tool for managing Azure resources. It is useful when you want repeatable administration, scripting, automation, and quick access to Azure without using the portal.

Use the CLI when you need to:

- Automate common tasks in shell scripts or pipelines.
- Work consistently across Windows, macOS, and Linux.
- Query resource state quickly with `--query` and `-o`.
- Switch between subscriptions, tenants, and clouds without changing tools.

The CLI is built around the `az` command. Most workflows follow a simple pattern: install, sign in, select the correct subscription, and verify the active context before making changes.

**Installation and version checks**

Install Azure CLI using the method that matches your platform and management standards. In practice, package managers are common for developer workstations, while enterprise environments may prefer controlled installation methods.

Typical version verification:

```bash
az version
az --version
```

Useful checks after installation:

- Confirm the CLI is available in your shell path.
- Verify the installed version before troubleshooting command behavior.
- Update regularly to get current command behavior and security fixes.

Common installation examples:

```bash
# Windows (PowerShell)
winget install -e --id Microsoft.AzureCLI

# macOS
brew update
brew install azure-cli

# Ubuntu/Debian (example)
curl -sL https://aka.ms/InstallAzureCLIDeb | sudo bash
```

If you manage multiple environments, keep in mind that command availability and defaults can vary slightly by version. For exam and production work, always validate the exact syntax with `az <group> <command> -h`.

**Authentication model**

Azure CLI authentication is usually interactive and browser-based. When you run `az login`, the CLI opens a browser sign-in flow unless you explicitly choose another method.

Common sign-in patterns:

```bash
az login
```

If you are on a machine without a browser or you need a code-based sign-in, request device code flow explicitly:

```bash
az login --use-device-code
```

This is useful for:

- Headless servers.
- Remote shells without GUI access.
- Restricted environments where browser sign-in is not practical.

For service automation, use a service principal instead of a user account:

```bash
az login --service-principal \
  --username <appId> \
  --password <client-secret-or-cert> \
  --tenant <tenant-id>
```

For managed identity on Azure resources, use:

```bash
az login --identity
```

Authentication choices matter because they affect permissions, token lifetime, and how the CLI resolves your default tenant and subscription.

**Tenant and subscription selection**

A tenant is the Microsoft Entra ID boundary for identities, while a subscription is the billing and resource container where Azure resources live. You often authenticate into one tenant and then choose one subscription from that tenant.

Important distinction:

- Use `az login` to choose or specify the tenant.
- Use `az account set` to choose the active subscription.

Example:

```bash
az login --tenant <tenant-id>
az account set --subscription <subscription-id-or-name>
```

> `az account set` does not select a tenant. If you need a different tenant, sign in again with `az login --tenant <tenant-id>`.

After login, inspect the current account context:

```bash
az account show
```

A practical query for troubleshooting is to inspect the subscription and tenant fields directly:

```bash
az account show --query "{name:name, id:id, tenantId:tenantId}" -o json
```

Depending on the account type and authentication flow, the shape of the returned user information can vary, so avoid assuming every field is always populated the same way.

**Subscription context management**

The active subscription determines where most CLI operations run. If you forget to set it, you may create or modify resources in the wrong subscription.

List available subscriptions:

```bash
az account list -o table
```

Set the active subscription:

```bash
az account set --subscription "<subscription-id-or-name>"
```

Verify the active subscription:

```bash
az account show --query "{name:name, id:id, tenantId:tenantId}" -o table
```

Practical workflow:

1. Sign in to the correct tenant.
2. List subscriptions you can access.
3. Set the intended subscription.
4. Confirm the active context before running destructive commands.

This is especially important when you work across:

- Multiple subscriptions in the same tenant.
- Separate subscriptions for dev, test, and production.
- Cross-tenant administration scenarios.

**Cloud environment selection**

Azure CLI can target different Azure clouds, such as the public cloud and sovereign clouds. The cloud setting affects endpoints, authentication, and resource management behavior.

List available clouds:

```bash
az cloud list -o table
```

Show the current cloud:

```bash
az cloud show
```

Switch cloud when needed:

```bash
az cloud set --name AzureCloud
```

Other cloud names may exist depending on your environment, such as government or regional sovereign clouds. After changing clouds, sign in again because the authentication context is tied to the selected cloud environment.

**Configuration and defaults**

Azure CLI supports local configuration for behavior such as default output format, default group, and default location. This reduces repetitive typing and helps standardize your workflow.

Set a default resource group:

```bash
az config set defaults.group=my-rg
```

Set a default location:

```bash
az config set defaults.location=eastus
```

Set a default output format:

```bash
az config set core.output=json
```

Check a specific setting:

```bash
az config get defaults.group
az config get core.output
```

Use defaults carefully:

- They speed up repetitive administration.
- They can hide mistakes if you forget which defaults are active.
- They are local to the CLI profile on that machine and user context.

For exam scenarios, remember that defaults do not replace explicit parameters when precision matters. If a command must target a specific resource group or subscription, specify it directly.

**Practical login and context workflow**

A reliable admin workflow is to authenticate, confirm the subscription, and validate the environment before making changes.

Example session:

```bash
az login --tenant <tenant-id>
az account list -o table
az account set --subscription "<subscription-name-or-id>"
az account show --query "{name:name, id:id, tenantId:tenantId}" -o table
az config set defaults.group=my-rg
az config set defaults.location=eastus
```

If you need to work in a different cloud:

```bash
az cloud set --name AzureCloud
az login --tenant <tenant-id>
az account set --subscription "<subscription-id>"
```

This sequence prevents accidental changes in the wrong tenant, subscription, or cloud.

**Querying and output formatting**

The Azure CLI is often most useful when combined with JMESPath queries and structured output. This helps you extract only the fields you need.

Common output options:

- `-o table` for quick human-readable summaries.
- `-o json` for scripts and automation.
- `-o tsv` for shell-friendly values.
- `-o yaml` for readable structured output.

Examples:

```bash
az account list -o table
az group list --query "[].{Name:name, Location:location}" -o table
az vm list --query "[].{Name:name, PowerState:powerState}" -o table
```

Use queries to reduce noise and make scripts more reliable. For example, extracting a subscription ID:

```bash
az account show --query id -o tsv
```

**Troubleshooting login and context issues**

Most Azure CLI problems in administration workflows come from the wrong tenant, wrong subscription, stale context, or missing permissions.

Common checks:

- Confirm you are signed in with `az account show`.
- Confirm the active subscription with `az account list -o table`.
- Confirm the cloud with `az cloud show`.
- Confirm the current defaults with `az config get <setting>`.

Typical symptoms and causes:

- You can see resources in the portal but not in CLI output.
  - You may be in the wrong subscription or tenant.
- A command fails with authorization errors.
  - Your account may lack the required role assignment.
- A resource appears to be missing.
  - You may be querying the wrong region, resource group, or subscription.
- A script works on one machine but not another.
  - Different CLI versions or different local defaults may be in use.

Useful reset actions:

```bash
az logout
az login
az account set --subscription "<subscription-id-or-name>"
```

If you suspect local configuration is causing confusion, inspect the active settings rather than assuming defaults:

```bash
az config get defaults.group
az config get defaults.location
az config get core.output
```

**Service principal and automation scenarios**

For automation, CI/CD, and non-interactive administration, use a service principal or managed identity rather than a personal user account. This improves repeatability and reduces dependency on human sign-in.

Service principal login example:

```bash
az login --service-principal \
  --username <appId> \
  --password <client-secret> \
  --tenant <tenant-id>
az account set --subscription "<subscription-id>"
```

Managed identity example from an Azure VM or other supported resource:

```bash
az login --identity
az account show
```

Use these patterns when:

- Running scripts in pipelines.
- Managing Azure resources from an Azure-hosted workload.
- Avoiding interactive authentication in production automation.

**Security and operational best practices**

Treat CLI context as sensitive operational state. A valid login does not guarantee you are in the correct scope.

Best practices:

- Always verify the active subscription before running create, update, or delete commands.
- Prefer explicit parameters in scripts instead of relying only on defaults.
- Use least privilege roles for automation identities.
- Re-authenticate after switching clouds or tenants.
- Avoid sharing long-lived secrets when managed identity is available.

> Do not assume the last successful login is still the correct context for the task you are about to run.

**Quick reference commands**

```bash
# Install/version
az version

# Sign in interactively
az login

# Sign in with device code
az login --use-device-code

# Sign in to a specific tenant
az login --tenant <tenant-id>

# Set active subscription
az account set --subscription "<subscription-id-or-name>"

# Show current account context
az account show

# List subscriptions
az account list -o table

# Show current cloud
az cloud show

# Switch cloud
az cloud set --name AzureCloud

# Set defaults
az config set defaults.group=my-rg
az config set defaults.location=eastus
az config set core.output=json
```

**Exam focus**

For AZ-104, focus on the relationship between identity, tenant, subscription, and cloud context. The CLI is not just a command runner; it is a stateful tool whose current context determines where your commands act.

Key takeaways:

- `az login` handles authentication and tenant selection.
- `az account set` selects the active subscription.
- `az cloud set` changes the target Azure cloud.
- `az config set` stores local CLI defaults.
- Always verify context before making changes.
