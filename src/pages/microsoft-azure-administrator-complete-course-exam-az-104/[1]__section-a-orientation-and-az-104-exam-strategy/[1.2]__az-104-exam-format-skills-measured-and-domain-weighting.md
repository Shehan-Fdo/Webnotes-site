## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**What AZ-104 measures**

AZ-104 validates that you can administer Azure services in day-to-day operations. The exam focuses on practical administration rather than architecture design. You are expected to manage identities, storage, compute, networking, monitoring, and governance using the Azure portal, Azure CLI, PowerShell, and ARM/Bicep concepts at an operational level.

The exam is best approached as a hands-on administration test. You should know how to create, configure, troubleshoot, and secure resources, not just define them. Many questions present a scenario and ask you to choose the most appropriate administrative action.

**Exam delivery and format**

AZ-104 is delivered through Microsoft’s standard certification delivery options, which can include online proctoring or a test center, depending on availability in your region. The exact delivery method is not fixed to one model, so always check the current scheduling options when you book.

The item mix can vary over time. Expect a combination of:
- Multiple-choice questions
- Multiple-select questions
- Drag-and-drop or matching-style items
- Scenario-based questions
- Case-study style sets with several related questions
- Items that may include exhibits, logs, or configuration snippets

The exam often feels operational and situational. Instead of asking “What is Azure Monitor?”, it may ask which alert rule, access control change, or network setting best solves a problem in a given environment.

> Microsoft can change exam length, item count, and item types without notice. Do not memorize outdated numbers from older study guides.

**How the exam feels in practice**

AZ-104 usually rewards careful reading and elimination. Scenario questions often include:
- A current state description
- A target state or business requirement
- Constraints such as cost, security, or minimal disruption
- One or more “best” answers rather than a single obvious fact

Case studies are especially important to handle well. You may need to read a longer business scenario, identify the relevant services, and answer several questions from the same context. The key skill is mapping requirements to Azure administration actions quickly.

Practical exam habits:
- Read the requirement first, then inspect the scenario details.
- Watch for words like `least privilege`, `minimize cost`, `high availability`, `private access`, and `no downtime`.
- Eliminate answers that solve the problem but violate a constraint.
- Treat “best” as “best fit for the stated requirement,” not “technically possible.”

**Skills measured by AZ-104**

The official skills outline changes over time, but the exam consistently covers the following administration areas.

**Manage Azure identities and governance**

This domain focuses on controlling access and organizing resources.

You should understand:
- Microsoft Entra ID user, group, and role concepts
- Role-based access control (RBAC) assignments at subscription, resource group, and resource scope
- Azure subscriptions, management groups, and resource groups
- Azure Policy for compliance and guardrails
- Resource locks and tagging for governance

Practical examples:
- Use RBAC when you need to grant a user permission to manage a VM without giving subscription-wide access.
- Use Azure Policy when you need to enforce standards such as allowed regions or required tags.
- Use management groups when you need to apply governance across multiple subscriptions.

Example CLI:
```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Virtual Machine Contributor" \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-prod-vm
```

Example policy assignment:
```bash
az policy assignment create \
  --name require-tags \
  --policy "Require a tag and its value" \
  --scope /subscriptions/<subscription-id>
```

**Implement and manage storage**

This domain covers Azure Storage accounts and common data access patterns.

You should know:
- Storage account types and redundancy options
- Blob, file, queue, and table storage use cases
- Access tiers such as hot, cool, and archive for blobs
- Shared access signatures (SAS), access keys, and Microsoft Entra ID-based authorization
- Storage networking controls such as firewalls and private access options
- Lifecycle management for blob data

Practical examples:
- Use blob storage for unstructured objects such as backups, images, and logs.
- Use Azure Files when you need SMB or NFS-style file shares.
- Use lifecycle rules to move older blobs to cooler tiers automatically.

Example CLI:
```bash
az storage account create \
  --name mystorageacct01 \
  --resource-group rg-data \
  --location eastus \
  --sku Standard_LRS \
  --kind StorageV2
```

Example blob lifecycle policy:
```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-to-cool",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool": { "daysAfterModificationGreaterThan": 30 },
            "tierToArchive": { "daysAfterModificationGreaterThan": 90 }
          }
        }
      }
    }
  ]
}
```

**Implement and manage compute resources**

This domain focuses on Azure virtual machines and related compute services.

You should understand:
- VM creation, sizing, availability, and scaling concepts
- Availability sets and availability zones
- VM extensions and custom script automation
- Azure Virtual Machine Scale Sets
- App Service and container-related administration at a basic operational level
- Backup and restore considerations for compute workloads

Practical examples:
- Use availability zones when you need resilience across physically separate datacenters in a region.
- Use an availability set when you need to protect against host or rack-level failures in a single datacenter cluster.
- Use scale sets when you need to automatically increase or decrease VM instances based on demand.

Example CLI:
```bash
az vm create \
  --resource-group rg-app \
  --name vm-web01 \
  --image Ubuntu2204 \
  --admin-username azureadmin \
  --generate-ssh-keys
```

Example extension install:
```bash
az vm extension set \
  --resource-group rg-app \
  --vm-name vm-web01 \
  --name CustomScript \
  --publisher Microsoft.Azure.Extensions \
  --settings '{"fileUris":["https://example.com/install.sh"],"commandToExecute":"bash install.sh"}'
```

**Configure and manage virtual networking**

This domain is one of the most important AZ-104 areas because many scenarios depend on network access and segmentation.

You should know:
- Virtual networks, subnets, and IP addressing
- Network security groups (NSGs) and application security groups
- Route tables and user-defined routes
- DNS basics in Azure
- VPN Gateway and ExpressRoute concepts at a high level
- Private endpoints and service endpoints
- Load balancer and application gateway basics
- Network peering and traffic flow

Practical distinctions:
- Use an **NSG** to filter traffic at subnet or NIC level.
- Use a **route table** when you need to override system routes.
- Use **VNet peering** when you need private connectivity between VNets.
- Use a **private endpoint** when you want supported PaaS access through a private IP in your VNet, usually with DNS and public access settings configured to make the service effectively private-only.
- Use a **service endpoint** when you want to extend VNet or subnet identity to a supported service while traffic still reaches the service’s public endpoint over the Microsoft backbone.

Example NSG rule:
```bash
az network nsg rule create \
  --resource-group rg-net \
  --nsg-name nsg-web \
  --name Allow-HTTP \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes Internet \
  --source-port-ranges '*' \
  --destination-port-ranges 80
```

Example VNet peering:
```bash
az network vnet peering create \
  --resource-group rg-net \
  --vnet-name vnet-hub \
  --name hub-to-spoke1 \
  --remote-vnet /subscriptions/<subscription-id>/resourceGroups/rg-net/providers/Microsoft.Network/virtualNetworks/vnet-spoke1 \
  --allow-vnet-access
```

**Monitor and maintain Azure resources**

This domain covers observability, alerting, and operational troubleshooting.

You should understand:
- Azure Monitor and Log Analytics
- Metrics versus logs
- Alerts, action groups, and alert rules
- Diagnostic settings
- Activity logs and resource logs
- VM insights and service health concepts

Practical examples:
- Use metrics for near-real-time numeric signals such as CPU percentage.
- Use logs when you need queryable historical detail.
- Use diagnostic settings to send platform logs to a Log Analytics workspace, storage account, or event hub.

Example diagnostic setting:
```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<subscription-id>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm-web01 \
  --workspace /subscriptions/<subscription-id>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --logs '[{"category":"Administrative","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

**Typical domain weighting**

Microsoft can adjust the exact weighting, so treat published percentages as guidance rather than a fixed contract. The current AZ-104 skills outline generally emphasizes:
- **Manage Azure identities and governance**
- **Implement and manage storage**
- **Deploy and manage Azure compute resources**
- **Configure and manage virtual networking**
- **Monitor and maintain Azure resources**

A practical way to study is to prioritize the largest operational areas first:
- Networking and identity usually deserve the most hands-on practice.
- Storage and compute follow closely because many scenarios combine them with access control.
- Monitoring and governance often appear as supporting requirements in larger scenarios.

A useful study mindset is to think in workflows rather than isolated services:
- A user needs access to a VM in a locked-down subnet.
- The VM stores logs in blob storage.
- Alerts notify operations when CPU or disk thresholds are crossed.
- Policy enforces tags and region restrictions.
- RBAC limits who can modify the resource group.

That kind of end-to-end scenario reflects how AZ-104 questions are often framed.

**How to prepare efficiently**

Focus on doing, not just reading. AZ-104 rewards familiarity with the Azure portal and command line.

A strong preparation loop looks like this:
1. Read the skill objective.
2. Build the resource in a lab subscription.
3. Break it on purpose by changing permissions, network rules, or settings.
4. Fix it using portal, CLI, or PowerShell.
5. Verify the result with logs, metrics, or access tests.

Useful CLI habits:
- Use `az find` to discover commands.
- Use `--help` to inspect parameters.
- Use `--query` and `-o table` to inspect output quickly.

Example:
```bash
az vm list --resource-group rg-app -o table
az network vnet list --query "[].{Name:name,Location:location}" -o table
```

**Common exam traps**

AZ-104 questions often test whether you understand the difference between similar Azure features.

Watch for these patterns:
- **RBAC vs Azure Policy**: RBAC grants access; Policy enforces rules.
- **NSG vs route table**: NSG filters traffic; route table changes path selection.
- **Private endpoint vs service endpoint**: private endpoint gives private IP access to supported PaaS; service endpoint keeps public endpoint access but restricts it to selected VNets/subnets.
- **Availability set vs availability zone**: availability set protects against host/rack failures; zones protect against datacenter-level failures within a region.
- **Storage key vs SAS vs Entra ID**: keys are broad and powerful, SAS is scoped and time-bound, Entra ID is preferred where supported for identity-based access.

> If a question asks for the most secure or most controlled option, prefer identity-based access, least privilege, and private connectivity where the scenario supports it.

**A practical end-to-end scenario**

Imagine a company hosting a web app and file uploads in Azure:
- The app runs on a VM or App Service.
- Users upload documents to Blob Storage.
- Only the app subnet should reach the storage account.
- Operations wants alerts for CPU spikes and failed requests.
- Compliance requires tags and restricted regions.

A good AZ-104 answer set would likely involve:
- RBAC for administrators and app identities
- Storage firewall rules or private endpoint access
- NSGs and possibly application gateway or load balancer for traffic control
- Azure Monitor alerts and diagnostic settings
- Azure Policy for tagging and location enforcement

This kind of scenario ties together multiple exam domains and reflects the real administrative work the exam is designed to measure.

**Study priorities by confidence level**

If you are short on time, prioritize in this order:
- Identity and governance
- Networking
- Storage
- Compute
- Monitoring

If you already work in Azure, spend extra time on:
- CLI and PowerShell equivalents
- Private access patterns
- RBAC scope inheritance
- Diagnostic settings and alerting
- Troubleshooting network reachability

**Final exam strategy**

AZ-104 is less about memorizing definitions and more about choosing the right administrative action under constraints. Build muscle memory for common tasks, understand how services interact, and practice reading scenarios carefully.

When you study, ask:
- What is the requirement?
- What is the constraint?
- Which Azure feature solves it with the least complexity?
- What side effect might make another option wrong?

That approach matches the exam’s practical, administrator-focused design.
