## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Why naming, tagging, and organization matter**

Azure environments become difficult to operate when resources are created ad hoc. Consistent naming, tags, and scope design help you find resources quickly, apply governance, control access, and understand cost ownership. For AZ-104, you should know how these controls differ and when to use each one.

- **Naming** identifies a resource and often must satisfy service-specific rules.
- **Tags** add metadata for filtering, reporting, automation, and chargeback.
- **Resource organization** determines how you apply RBAC, Azure Policy, and lifecycle management.

**Azure resource hierarchy and scope**

Azure governance is applied at different scopes. Understanding scope is essential because many exam questions ask where to assign permissions or policy.

- **Management group**: Organizes subscriptions and is the best place for standards that must apply across multiple subscriptions.
- **Subscription**: Common boundary for billing, quota, and many governance assignments.
- **Resource group**: Logical container for resources that share a lifecycle.
- **Resource**: Individual service instance, such as a virtual machine or storage account.

A higher scope can usually govern lower scopes through inheritance. For example, a role assignment or policy at a management group can affect subscriptions beneath it. However, the exact behavior depends on the feature and the scope hierarchy; do not assume every setting flows everywhere.

**Naming conventions**

Naming conventions make resources predictable and searchable. In Azure, many services have strict naming rules, so a good convention must be both human-readable and technically valid.

A practical naming pattern often includes:

- **Environment**: `dev`, `test`, `uat`, `prod`
- **Application or workload**: `web`, `api`, `erp`
- **Region**: `weu`, `eus`, `uks`
- **Instance or sequence**: `01`, `02`

Example pattern:

```text
<org>-<app>-<env>-<region>-<instance>
```

Example names:

```text
contoso-web-prod-weu-01
contoso-api-dev-eus-01
```

Use names that are consistent, short enough to fit service limits, and meaningful to operators. Avoid embedding secrets, ticket numbers, or overly long descriptions.

> Storage account names are globally unique and must use only lowercase letters and numbers. Do not treat a sample name as reusable across tenants or subscriptions.

**Common service-specific naming constraints**

Some services are frequently tested because their naming rules are strict.

| Service | Common constraint | Notes |
|---|---|---|
| Storage account | 3-24 characters, lowercase letters and numbers only, globally unique | No hyphens or uppercase letters |
| Key Vault | 3-24 characters, alphanumeric and hyphens, globally unique DNS name | Must be valid as a DNS label |
| Public IP DNS label | Regionally unique within the Azure region and DNS namespace | Used for the public FQDN |
| Resource group | 1-90 characters, supports many characters | Best kept simple for readability |
| Virtual network | Follows Azure resource naming rules, not DNS-specific | Use consistent workload naming |
| App Service app name | Must be unique in the App Service DNS namespace | Often becomes part of the default hostname |

When in doubt, check the service documentation before standardizing a pattern. Exam questions often test whether you know that some names are globally unique while others are only unique within a scope or namespace.

**Tagging standards**

Tags are name-value pairs attached to supported Azure resources, resource groups, and subscriptions. They help you classify resources without changing the resource name.

Common tag keys include:

- **Owner**: Team or person responsible for the resource
- **CostCenter**: Billing or chargeback code
- **Environment**: `dev`, `test`, `prod`
- **Application**: Workload or business service name
- **Criticality**: `low`, `medium`, `high`
- **DataClassification**: `public`, `internal`, `confidential`

Example:

```text
Owner=PlatformTeam
CostCenter=FIN-204
Environment=prod
Application=webshop
Criticality=high
```

Tags are useful for:

- Cost analysis and chargeback
- Inventory filtering
- Automation and reporting
- Governance and lifecycle management

Important limitations:

- Tags are not automatically inherited by child resources in all cases.
- Inheritance usually requires Azure Policy, automation, or deployment templates.
- Not every Azure resource type supports tags in the same way.
- Some services have special tag behavior or restrictions, so verify support before relying on tags for governance.

**Tagging versus naming**

Use names for identity and tags for metadata.

- **Name** answers: “What is this resource?”
- **Tag** answers: “Who owns it, what is it for, and how should it be managed?”

A resource name should remain stable and concise. Tags can change as ownership, cost center, or environment changes.

**Azure Policy for naming and tagging**

Azure Policy enforces standards at scale. It is the preferred tool when you need to audit, deny, or remediate noncompliant resources.

Common policy effects include:

- **Deny**: Blocks noncompliant requests.
- **Audit**: Allows the request but records noncompliance.
- **Append**: Adds fields to the request before evaluation in limited scenarios; it is a legacy effect and not a general-purpose remediation tool.
- **Modify**: Changes supported request properties during deployment and is the preferred option for many tag-remediation scenarios when supported.

A simple policy idea for required tags:

```json
{
  "mode": "Indexed",
  "policyRule": {
    "if": {
      "field": "tags['Environment']",
      "exists": "false"
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

A policy that restricts allowed values:

```json
{
  "mode": "Indexed",
  "policyRule": {
    "if": {
      "not": {
        "field": "tags['Environment']",
        "in": ["dev", "test", "prod"]
      }
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

A policy that audits a naming pattern:

```json
{
  "mode": "All",
  "policyRule": {
    "if": {
      "not": {
        "field": "name",
        "like": "contoso-*"
      }
    },
    "then": {
      "effect": "audit"
    }
  }
}
```

For tag enforcement, remember:

- **Deny** is best when you want to prevent noncompliant deployments.
- **Audit** is best when you want visibility before enforcement.
- **Modify** can help add or correct tags, but it requires a managed identity and support from the target resource type.
- **Append** does not replace a full tagging strategy and should not be treated as universal tag inheritance.

**Resource groups and lifecycle design**

Resource groups are logical containers for resources that share a lifecycle. Put resources together when you expect to deploy, update, and delete them together.

Use a resource group when:

- Resources belong to the same application or environment.
- You want a common lifecycle boundary.
- You need a simple unit for access control and deployment.

Do not use a resource group as a cross-subscription container. A resource group belongs to one subscription only.

Practical guidance:

- Keep production and nonproduction resources separate.
- Group resources by workload, not by department alone.
- Avoid mixing unrelated applications in one resource group unless there is a strong operational reason.

Example topology:

```text
Management group
└── Subscription: Contoso-Prod
    ├── Resource group: rg-web-prod-weu
    └── Resource group: rg-data-prod-weu

Subscription: Contoso-Dev
└── Resource group: rg-web-dev-eus
```

**Management groups versus subscriptions versus resource groups**

Choose the scope based on what you need to control.

| Scope | Best for | Typical use |
|---|---|---|
| Management group | Cross-subscription governance | Policy, RBAC, standardization |
| Subscription | Billing, quota, isolation | Environment separation, chargeback |
| Resource group | Shared lifecycle | Deployment and deletion boundary |

Exam decision rule:

- Use a **management group** when the same policy or access model must apply across multiple subscriptions.
- Use a **subscription** when you need a strong boundary for billing, quota, or isolation.
- Use a **resource group** when resources should be managed and deployed together.

**RBAC and scope inheritance**

Role-Based Access Control (RBAC) determines who can do what at a given scope.

Key points:

- A role assignment at a higher scope usually applies to child scopes through inheritance.
- A role assigned at a resource group scope affects resources in that group.
- A role assigned at a subscription scope affects resource groups and resources in that subscription.
- A role assigned at a management group scope can affect subscriptions beneath it.

Use the least privileged scope that still meets the operational need.

Examples:

- Grant a support team read access at the subscription level.
- Grant an application team contributor access only to its resource group.
- Grant a security team policy or reader access at the management group level.

Do not confuse RBAC inheritance with Azure Policy. Policy evaluates compliance; RBAC controls authorization.

**Resource locks**

Resource locks protect important resources from accidental deletion or modification. They are a governance control, not an access control replacement.

Lock types:

- **CanNotDelete**: Users can read and modify, but cannot delete the resource.
- **ReadOnly**: Users can read but cannot modify or delete.

Use locks when:

- A resource is critical and should not be removed accidentally.
- You want an extra safeguard for shared infrastructure.

Be careful:

- Locks can interfere with deployments, updates, and automation.
- Locks apply at the scope where they are set and affect child resources where applicable.
- Use them selectively, especially in environments with frequent change.

**Azure Blueprints and equivalent governance packaging**

Azure Blueprints was designed to package governance artifacts such as policy assignments, role assignments, and ARM templates for repeatable environment deployment. In many modern environments, similar outcomes are achieved with management groups, Azure Policy, RBAC, ARM templates, Bicep, and automation pipelines.

For exam purposes, understand the concept:

- Governance can be standardized and deployed repeatedly.
- Policy, RBAC, and templates can be combined to create a compliant landing zone.
- At scale, organizations often use management groups plus infrastructure-as-code rather than manual setup.

**Deployment automation with ARM and Bicep**

Naming and tagging standards should be enforced during deployment, not only after resources exist. Infrastructure as code helps you keep standards consistent.

Example Bicep snippet:

```bicep
param location string = resourceGroup().location
param appName string
param env string = 'prod'

resource stg 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: toLower('${appName}${env}weu01')
  location: location
  tags: {
    Environment: env
    Application: appName
    Owner: 'PlatformTeam'
  }
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
}
```

Practical notes:

- Use parameters for environment, region, and application name.
- Apply tags in the template so every deployment starts compliant.
- Validate naming rules before deployment to avoid failed provisioning.
- Use `what-if` to preview changes when possible.

Example Azure CLI deployment:

```bash
az deployment group create \
  --resource-group rg-web-prod-weu \
  --template-file main.bicep \
  --parameters appName=webshop env=prod
```

**Practical governance workflow**

1. Define naming and tagging standards.
2. Assign policy at the management group or subscription scope.
3. Use RBAC to delegate only the required permissions.
4. Deploy resources through ARM templates, Bicep, or automation.
5. Audit compliance and remediate drift.
6. Add locks for critical resources where deletion risk is high.

**Exam-style scenarios**

- If you need a standard to apply across many subscriptions, use a **management group** with policy and RBAC.
- If you need to isolate billing and quotas for a workload, use a **subscription** boundary.
- If resources must be deployed and deleted together, place them in the same **resource group**.
- If you need to prevent accidental deletion of a key resource, use a **resource lock**.
- If you need metadata for reporting and chargeback, use **tags**.
- If you need to block noncompliant deployments, use **Azure Policy** with `deny`.

**Common mistakes to avoid**

- Assuming tags are inherited automatically by all child resources.
- Using resource groups as if they can span subscriptions.
- Treating Azure Policy as an RBAC feature.
- Using `append` as a universal tag-enforcement mechanism.
- Ignoring service-specific naming rules such as global uniqueness or DNS constraints.
- Overloading names with information that belongs in tags.

**Quick reference**

- **Naming**: Human-readable identity with service-specific constraints.
- **Tags**: Metadata for ownership, cost, and governance.
- **Resource groups**: Lifecycle boundary inside one subscription.
- **Subscriptions**: Billing, quota, and isolation boundary.
- **Management groups**: Cross-subscription governance boundary.
- **RBAC**: Authorization.
- **Azure Policy**: Compliance enforcement.
- **Locks**: Protection against accidental change or deletion.
