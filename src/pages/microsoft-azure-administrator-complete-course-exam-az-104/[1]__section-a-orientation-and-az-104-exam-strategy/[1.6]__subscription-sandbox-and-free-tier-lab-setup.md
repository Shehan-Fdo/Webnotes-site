## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Why this setup matters**

AZ-104 labs often fail for reasons that are not about Azure administration skills at all: the wrong subscription is active, the sandbox expired, a provider is not registered, or a free-tier limit blocks deployment. You need a reliable lab environment so you can focus on identity, networking, storage, governance, and compute tasks instead of troubleshooting access.

A good lab setup gives you:
- A subscription you can safely modify.
- A predictable way to switch between subscriptions and directories.
- A low-cost or free environment for practice.
- A cleanup routine that prevents surprise charges.

**Azure subscription types for AZ-104 practice**

An Azure <abbr title="subscription is a billing and access boundary in Azure">subscription</abbr> is the main container for billing, quotas, and resource deployment. For AZ-104, the subscription type affects what you can create, how much you can spend, and whether you can use certain services.

Common subscription types include:

- **Free trial**
  - Best for initial hands-on practice and short labs.
  - Usually includes a limited credit amount for a limited time.
  - Often requires a credit card or phone verification, depending on region and current signup rules.
  - Good for learning portal navigation, resource groups, virtual networks, storage accounts, and basic VM deployment.

- **Pay-as-you-go**
  - Best when you need longer-term practice or repeated lab work.
  - Charges are based on usage.
  - Useful for AZ-104 scenarios that need more flexibility, such as multiple VMs, public IPs, load balancers, or longer-running labs.
  - Requires active cost management because resources can keep billing even when idle.

- **Azure for Students**
  - Best for eligible students who want a no-credit-card entry path.
  - Typically includes a credit grant and access to many core services.
  - Good for practice labs, but some services, quotas, or marketplace items may be restricted.
  - Useful when you want to avoid payment setup during study.

- **Sandbox or learning environment**
  - Best for guided training modules and temporary practice.
  - Often preconfigured with a time limit and a fixed set of permissions.
  - Useful for quick exercises, but not ideal for long-running or custom lab topologies.

> <mark>Always confirm which subscription is active before you create resources.</mark> Many AZ-104 lab mistakes come from deploying into the wrong subscription or directory.

**How to create or obtain an Azure free account**

If you do not already have a lab subscription, the usual path is to create an Azure free account. The exact signup flow can vary by region and Microsoft policy, but the general process is consistent.

1. Go to the Azure free account signup page.
2. Sign in with a Microsoft account or create one if needed.
3. Complete identity verification steps, which may include phone verification and payment method validation.
4. Accept the terms and create the subscription.
5. Open the Azure portal and confirm the subscription appears in your account.

Typical free account benefits include:
- A limited credit amount for a trial period.
- Access to a set of free services for a limited time.
- Some services that remain free within monthly usage limits.
- Access to common AZ-104 practice resources such as resource groups, storage accounts, virtual networks, and basic compute testing, subject to quota and region availability.

Common free-tier limitations:
- Credits expire after the trial period.
- Some VM sizes, premium storage options, and marketplace images may not be available.
- Not all regions or services are eligible.
- You may need to upgrade to pay-as-you-go for advanced scenarios.

**Azure free services and limits to know**

The free tier is useful, but it is not unlimited. For AZ-104, you should know the difference between “free for a period” and “free within a quota.”

Typical free-tier patterns:
- Some services are free for a fixed period after signup.
- Some services are free only up to a monthly quota.
- Some services are always free only for small usage levels.

Examples of common free or low-cost practice targets:
- Resource groups
- Azure <abbr title="Role-Based Access Control">RBAC</abbr> practice with users and groups
- Storage accounts with small amounts of data
- Virtual networks and subnets
- Network security groups
- Key Vault basics, if available in your subscription and region
- Azure Monitor basics, with attention to log ingestion costs

Be careful with:
- Virtual machines, which can generate compute, disk, and public IP charges.
- Managed disks, snapshots, and images, which can remain billable after a VM is deleted.
- Log Analytics workspaces, which can incur ingestion and retention costs.
- Public IP addresses, which may bill even when attached to stopped resources depending on SKU and configuration.

**Sandbox use for AZ-104 labs**

A sandbox is a temporary Azure environment provided by a training platform or Microsoft Learn-style exercise. It is useful when you want a safe, disposable environment without setting up your own subscription.

Use a sandbox when:
- You need a quick, guided lab.
- You want to avoid billing risk.
- The exercise is designed for a temporary environment.
- You only need a small set of services and permissions.

Use your own subscription when:
- You need to repeat labs many times.
- You need custom networking or multi-resource topologies.
- You want to test cleanup, cost controls, or governance.
- The sandbox does not support the service or region you need.

Common sandbox limitations that affect AZ-104 labs:
- **Unsupported services**
  - Some resource types are blocked entirely.
  - Marketplace images may be unavailable.
  - Certain identity, networking, or monitoring features may be restricted.

- **Region restrictions**
  - You may only deploy in one or a few allowed regions.
  - Some services are not available in the sandbox region.
  - A lab may fail if you choose a region outside the allowed list.

- **Resource type restrictions**
  - You may not be able to create public IPs, specific VM sizes, managed identities, or advanced networking components.
  - Some labs require features such as peering, private endpoints, or custom DNS that the sandbox does not permit.

- **Quota and time limits**
  - The sandbox can expire during a long lab.
  - Resource quotas may be lower than in a personal subscription.
  - Repeated deployments can hit temporary limits.

Workarounds for sandbox constraints:
- Use the exact region specified by the lab.
- Choose smaller VM sizes if the lab allows it.
- Replace unsupported services with the closest supported equivalent when the lab objective permits it.
- Recreate the lab in your own subscription if the sandbox blocks a required feature.
- Save commands and configuration locally so you can rerun them quickly after expiration.

> If a sandbox expires mid-lab, do not assume your commands are wrong. Check the timer first, then redeploy in a fresh sandbox or your own subscription.

**Azure Cloud Shell versus local CLI or PowerShell**

Azure Cloud Shell is a browser-based shell that gives you access to Azure CLI or Azure PowerShell without installing tools locally. It is especially useful for AZ-104 because it reduces setup time and works well in locked-down environments.

Use **Cloud Shell** when:
- You want a quick, browser-based lab environment.
- You are on a shared or managed device.
- You need Azure CLI or PowerShell without local installation.
- You want built-in authentication to the current Azure session.

Use **local Azure CLI or PowerShell** when:
- You need offline scripting or repeated automation.
- You want to use local files, editors, or automation tools.
- You need to test scripts against a local environment before running them in Azure.
- You want more control over versions and extensions.

Cloud Shell practical notes:
- It persists files in a mounted storage account.
- It starts in the context of the signed-in account and selected subscription.
- It can be used for quick verification commands, resource creation, and cleanup.

Common commands in Cloud Shell or local CLI:

```bash
az login
az account list -o table
az account set --subscription "My Lab Subscription"
az group list -o table
```

PowerShell equivalents:

```powershell
Connect-AzAccount
Get-AzSubscription
Set-AzContext -Subscription "My Lab Subscription"
Get-AzResourceGroup
```

**Portal-based lab setup and context verification**

In the Azure portal, the most important habit is verifying the active directory and subscription before you create anything.

Where to check context in the portal:
- Use the top-right account and directory area to confirm the signed-in identity.
- Open the subscription filter or directory + subscription settings to see which subscription is selected.
- Confirm the current directory if you work across multiple tenants.
- Check the portal header when switching between lab accounts.

Practical portal workflow:
1. Sign in to the Azure portal.
2. Open the directory and subscription selector.
3. Confirm the correct tenant or directory.
4. Select the intended subscription.
5. Refresh the portal if the resource list does not match expectations.
6. Create or modify resources only after verification.

Why this matters:
- A resource created in the wrong subscription may be hard to find later.
- RBAC assignments can fail if you are in the wrong directory.
- Some labs require a specific tenant context for identity tasks.

**Multiple subscriptions and directory switching**

AZ-104 scenarios often involve more than one subscription. You may have a free trial, a student subscription, and a work or pay-as-you-go subscription at the same time.

Best practices:
- Name subscriptions clearly so you can identify them quickly.
- Use tags or resource group naming conventions to separate lab work from personal resources.
- Verify the active context before every deployment.
- Recheck context after signing out, using a different browser profile, or opening a fresh Cloud Shell session.

CLI examples for switching context:

```bash
az account list --output table
az account set --subscription "AZ104-Lab-01"
az account show --output table
```

PowerShell examples:

```powershell
Get-AzSubscription | Format-Table Name, Id, State
Set-AzContext -Subscription "AZ104-Lab-01"
Get-AzContext
```

**Provider registration and common deployment failures**

Many Azure services require the resource provider to be registered in the subscription. In AZ-104 labs, a deployment can fail even when your permissions are correct if the provider is not registered.

Common symptoms:
- “The subscription is not registered to use namespace ...”
- A resource creation blade fails after validation.
- CLI or PowerShell returns a provider-related error.

Useful checks:
```bash
az provider list --query "[?registrationState!='Registered'].{Namespace:namespace,State:registrationState}" -o table
az provider register --namespace Microsoft.Network
az provider register --namespace Microsoft.Compute
```

PowerShell:
```powershell
Get-AzResourceProvider | Where-Object {$_.RegistrationState -ne "Registered"} | Select-Object ProviderNamespace, RegistrationState
Register-AzResourceProvider -ProviderNamespace Microsoft.Network
```

AZ-104 exam-oriented examples:
- You try to create a virtual network and the provider is not registered.
- A sandbox allows the portal blade but blocks the backend deployment.
- A resource group exists, but the subscription cannot deploy a specific service until registration completes.

**Sandbox expiration and lab recovery**

A sandbox can expire while you are still working. This is common in timed practice environments and can interrupt long deployments.

What to do:
- Save commands and templates outside the sandbox.
- Keep deployment scripts in a local editor or repository.
- Recreate the environment quickly in a fresh sandbox or your own subscription.
- Avoid starting long-running deployments near the end of the timer.

Recovery strategy:
1. Export or copy the commands you used.
2. Note the resource group names, regions, and parameters.
3. Reopen the lab in a new sandbox if available.
4. Re-run the deployment with the same naming pattern.
5. Verify the result before continuing.

**Cost management and cleanup**

Cleanup is not optional in Azure practice. Even small lab mistakes can create ongoing charges.

Before you finish a lab:
- Delete the resource group if the lab is disposable.
- Confirm that all dependent resources are removed.
- Check for orphaned managed disks.
- Check for public IP addresses that remain after VM deletion.
- Check for snapshots and images that were created during testing.
- Review storage accounts, log analytics workspaces, and backup items if they were used.

Useful cleanup commands:

```bash
az group delete --name rg-az104-lab --yes --no-wait
az disk list -o table
az network public-ip list -o table
az snapshot list -o table
```

PowerShell:

```powershell
Remove-AzResourceGroup -Name "rg-az104-lab" -Force
Get-AzDisk
Get-AzPublicIpAddress
Get-AzSnapshot
```

Cost controls to use early:
- Create a budget in Cost Management.
- Add alert thresholds for spending.
- Review cost analysis regularly.
- Use tags such as `env=lab` or `owner=study` to filter resources.
- Prefer smaller VM sizes and shorter runtime windows.

Example budget idea:
- Set a monthly budget for your lab subscription.
- Trigger alerts at 50%, 80%, and 100%.
- Send alerts to your email so you notice unexpected usage quickly.

**Resource locks and why they matter**

Resource locks protect resources from accidental deletion or modification. They are useful in production, but they can interfere with lab cleanup and testing.

Common lock types:
- **CanNotDelete**
  - Prevents deletion but allows changes.
- **ReadOnly**
  - Prevents changes and deletion.

Why locks matter in AZ-104 labs:
- A locked resource group may block cleanup.
- A lock can make a delete command fail even when permissions are correct.
- A lab may intentionally use locks to test governance behavior.

If cleanup fails, check for locks:
```bash
az lock list --resource-group rg-az104-lab -o table
```

Remove a lock if appropriate:
```bash
az lock delete --name lock1 --resource-group rg-az104-lab
```

**Practical lab setup checklist**

1. Choose the right subscription type for the lab duration and cost tolerance.
2. Confirm the active directory and subscription in the portal.
3. Use Cloud Shell for quick tasks and local tools for repeatable automation.
4. Verify provider registration before deploying services.
5. Respect sandbox region and service limits.
6. Track costs with budgets and alerts.
7. Clean up disks, IPs, snapshots, and resource groups after each lab.
8. Check for locks if deletion fails.
9. Save commands so you can recover quickly if the sandbox expires.

**AZ-104 scenario examples tied to setup**

- You need to test RBAC across two subscriptions.
  - Use a subscription selector and confirm the correct context before assigning roles.

- A virtual network deployment fails in a sandbox.
  - Check whether the region is allowed and whether `Microsoft.Network` is registered.

- A VM lab creates charges after you stop the machine.
  - Review disks, public IPs, and snapshots, then delete the resource group if possible.

- A lab expires before you finish storage configuration.
  - Reuse saved CLI commands in a new sandbox or your own subscription.

- A resource group will not delete.
  - Check for locks, then remove the lock if the lab permits it.

**Key takeaways**

- Use a free trial, student subscription, or pay-as-you-go subscription based on how long and how often you practice.
- Treat the sandbox as temporary and limited.
- Verify subscription and directory context before every deployment.
- Use Cloud Shell for fast browser-based work and local tools for repeatable automation.
- Watch for provider registration issues, region restrictions, and sandbox expiration.
- Clean up aggressively and use budgets, alerts, and lock checks to avoid surprises.
