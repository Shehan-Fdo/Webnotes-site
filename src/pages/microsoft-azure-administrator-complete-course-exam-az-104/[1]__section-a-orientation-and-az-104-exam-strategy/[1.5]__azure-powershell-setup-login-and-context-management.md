## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure PowerShell overview**

Azure PowerShell is a set of PowerShell modules that lets you manage Azure resources from the command line, scripts, and automation pipelines. For AZ-104, you need to know how to install the `Az` module, authenticate correctly, and select the right subscription and tenant before running administrative commands.

The `Az` module is the current Azure PowerShell family. It replaces the older `AzureRM` modules, which are deprecated. In practice, you use `Connect-AzAccount` to sign in, `Get-AzContext` to inspect the current session, and `Set-AzContext` to switch the active subscription or tenant.

**Why context matters**

Azure PowerShell is stateful. Many commands run against the current context unless you explicitly specify a subscription or resource scope. If your context points to the wrong subscription, you can create, modify, or delete resources in the wrong place.

A typical context includes:
- **Account**: the signed-in identity
- **Tenant**: the Microsoft Entra ID tenant
- **Subscription**: the active Azure subscription
- **Environment**: the Azure cloud, such as AzureCloud or AzureUSGovernment

For day-to-day administration, the most important operational choices are the account, tenant, and subscription.

**Install and update the Az module**

Install the module from the PowerShell Gallery when you need a fresh environment or a lab machine. Use an elevated session if your execution policy or system scope requires it.

```powershell
# Install for the current user
Install-Module -Name Az -Scope CurrentUser -Repository PSGallery -Force

# Update the installed Az module
Update-Module -Name Az

# Check the installed version
Get-InstalledModule -Name Az
```

If you are troubleshooting version issues, verify the module path and loaded version in the current session:

```powershell
Get-Module Az -ListAvailable
Get-Module Az.Accounts
$PSVersionTable.PSVersion
```

A few practical points:
- **PowerShell 7** works well for Azure PowerShell and is common in modern automation.
- **Windows PowerShell 5.1** is still supported in many environments, but PowerShell 7 is preferred for new work.
- The `Az.Accounts` module provides sign-in and context cmdlets, while other `Az.*` modules provide service-specific management commands.

> If you have both `AzureRM` and `Az` installed, avoid mixing them in the same session. Module conflicts can cause confusing command resolution and authentication behavior.

**Authentication methods**

Azure PowerShell supports several sign-in methods. Choose the method that fits the environment and identity type.

**Interactive user sign-in**

Use interactive sign-in for local administration and labs.

```powershell
Connect-AzAccount
```

If you need to target a specific tenant, use the tenant ID as the standard and most reliable value:

```powershell
Connect-AzAccount -Tenant '00000000-0000-0000-0000-000000000000'
```

You can also use a tenant domain in some supported scenarios, but the tenant ID is the safest choice for exam and production guidance.

**Device code authentication**

Use device code flow when browser-based sign-in is not practical, such as in a non-GUI session, remote shell, or constrained environment.

```powershell
Connect-AzAccount -UseDeviceAuthentication
```

This method is common in PowerShell 7 sessions, jump boxes, and remote terminals where a browser cannot open directly.

**Service principal authentication**

Use a service principal for automation, CI/CD, and non-interactive scripts. The credential must be built from the application registration details:
- The username is the **application (client) ID**
- The password is the **client secret**

```powershell
$tenantId = '00000000-0000-0000-0000-000000000000'
$appId    = '11111111-1111-1111-1111-111111111111'
$secret   = ConvertTo-SecureString 'client-secret-value' -AsPlainText -Force
$cred     = [pscredential]::new($appId, $secret)

Connect-AzAccount -ServicePrincipal -Tenant $tenantId -Credential $cred
```

Use this pattern when you need repeatable automation without user interaction. Assign only the permissions the app needs.

**Managed identity authentication**

Use managed identity when code runs on an Azure resource such as a virtual machine, automation account, function app, or app service. This removes the need to store secrets.

```powershell
Connect-AzAccount -Identity
```

For a user-assigned managed identity, the identity must already be assigned to the Azure resource. Depending on the scenario and module behavior, you may need to specify the identity by client ID or object ID.

```powershell
# Example pattern for a user-assigned managed identity
Connect-AzAccount -Identity -AccountId '11111111-1111-1111-1111-111111111111'
```

Use the system-assigned identity without `-AccountId`. Use the user-assigned identity only when the resource has that identity attached.

**Inspect the current context**

After sign-in, verify where your session is pointed before you run administrative commands.

```powershell
Get-AzContext
```

This shows the active account, tenant, subscription, and environment. For a broader view of available subscriptions in the signed-in tenant, use:

```powershell
Get-AzSubscription
```

To see the current context in a compact form:

```powershell
Get-AzContext | Select-Object Account, Tenant, Subscription, Environment
```

To inspect all available contexts stored in the current profile, use:

```powershell
Get-AzContext -ListAvailable
```

That cmdlet lists saved contexts, not subscriptions. Do not confuse the two.

**Switch subscriptions and tenants**

Use `Set-AzContext` when you need to change the active subscription or tenant for the current session.

```powershell
# Switch to a specific subscription by name
Set-AzContext -Subscription 'Production-Sub'

# Switch by subscription ID
Set-AzContext -SubscriptionId '22222222-2222-2222-2222-222222222222'
```

If you work across multiple tenants, you may need to reconnect or explicitly target the tenant first, then select the subscription in that tenant.

A practical workflow is:
1. Sign in with the correct identity.
2. Confirm the tenant.
3. Select the intended subscription.
4. Recheck the context before making changes.

**Context persistence and autosave**

Azure PowerShell can persist context information locally so you do not need to sign in repeatedly in the same environment. This is useful for interactive work, but it is not a universal portable backup of your session state.

```powershell
# Check whether context autosave is enabled
Get-AzContextAutosaveSetting

# Disable autosave for the current process or user scope as needed
Disable-AzContextAutosave -Scope Process
```

You can export and import context data, but treat it as environment-dependent. The saved file may rely on the associated account state, token cache, and module version.

```powershell
# Export the current context
Save-AzContext -Path .\az-context.json

# Import a saved context
Import-AzContext -Path .\az-context.json
```

Use this for convenience in controlled lab environments, not as a guaranteed portable restore mechanism across machines or users.

**Practical admin patterns**

When you automate Azure administration, make context explicit instead of relying on whatever is currently active in the shell.

```powershell
Connect-AzAccount -Tenant '00000000-0000-0000-0000-000000000000'
Set-AzContext -SubscriptionId '22222222-2222-2222-2222-222222222222'

Get-AzResourceGroup
```

You can also pass a context object to some cmdlets to avoid ambiguity:

```powershell
$ctx = Get-AzContext
Get-AzVM -ResourceGroupName 'rg-prod' -DefaultProfile $ctx
```

This is useful in scripts that must operate against a specific subscription even if the user’s default context changes.

**Common troubleshooting checks**

If sign-in or context selection fails, verify the basics first:
- **Module version**: confirm `Az.Accounts` and `Az` are installed and current enough for your environment
- **PowerShell version**: confirm whether you are using Windows PowerShell or PowerShell 7
- **Tenant ID**: use the correct tenant GUID when targeting a specific directory
- **Subscription access**: ensure the signed-in identity has access to the subscription
- **Identity type**: confirm whether you are using a user account, service principal, or managed identity

Useful commands:

```powershell
# Confirm the signed-in account and active subscription
Get-AzContext

# List subscriptions available to the signed-in identity
Get-AzSubscription

# Reconnect if the token is stale
Disconnect-AzAccount
Connect-AzAccount
```

If you are using a service principal, also confirm that:
- The app registration exists
- The secret or certificate is valid
- The service principal has the required role assignments
- The tenant ID matches the directory where the app is registered

**Exam-focused distinctions**

AZ-104 often tests whether you understand the difference between authentication and context:
- **Authentication** answers: who are you?
- **Context** answers: where are you operating?

That distinction matters because a successful sign-in does not guarantee you are in the correct subscription. Always verify the active context before making changes.

It also helps to remember:
- `Get-AzSubscription` lists subscriptions
- `Get-AzContext` shows the active session context
- `Set-AzContext` changes the active subscription or context
- `Connect-AzAccount` starts authentication

**Quick reference**

| Task | Cmdlet | Notes |
|---|---|---|
| Install Azure PowerShell | `Install-Module Az` | Prefer `-Scope CurrentUser` for local installs |
| Sign in interactively | `Connect-AzAccount` | Use tenant ID when you need to target a specific tenant |
| Device code sign-in | `Connect-AzAccount -UseDeviceAuthentication` | Useful in non-GUI or remote sessions |
| Service principal sign-in | `Connect-AzAccount -ServicePrincipal` | Use app ID as username and client secret or certificate |
| Managed identity sign-in | `Connect-AzAccount -Identity` | Works on Azure resources with managed identity enabled |
| View active session | `Get-AzContext` | Shows account, tenant, subscription, environment |
| List subscriptions | `Get-AzSubscription` | Use this for subscription discovery |
| Switch subscription | `Set-AzContext` | Use subscription name or ID |
| Save context | `Save-AzContext` | Environment-dependent convenience, not a universal backup |
| Import context | `Import-AzContext` | Works best in the same profile and module environment |

**Minimal lab workflow**

A simple lab sequence often looks like this:

```powershell
Install-Module Az -Scope CurrentUser -Force
Connect-AzAccount
Get-AzSubscription
Set-AzContext -SubscriptionId '22222222-2222-2222-2222-222222222222'
Get-AzContext
```

This pattern keeps your session predictable and reduces the risk of working in the wrong subscription.

**Key takeaways**

- Use the `Az` module family for current Azure PowerShell administration.
- Prefer tenant ID when you need to target a specific tenant.
- Use `Get-AzSubscription` for subscription discovery and `Get-AzContext` for the active session.
- Treat context as session state, not as a portable backup.
- Use service principals or managed identities for automation, and keep credentials out of scripts whenever possible.
