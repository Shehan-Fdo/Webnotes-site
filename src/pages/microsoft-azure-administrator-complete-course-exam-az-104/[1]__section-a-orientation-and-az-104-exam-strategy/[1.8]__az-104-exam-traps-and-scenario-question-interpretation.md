## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**How to read AZ-104 scenario questions**

AZ-104 questions often test whether you can map a business requirement to the *right Azure control plane feature*, not just whether you know a service name. The trap is usually in the wording: scope, identity type, network path, or authorization model.

A good approach is to identify four things before choosing an answer:

- **Scope**: subscription, management group, resource group, resource, or object level.
- **Plane**: management plane versus data plane.
- **Path**: public endpoint, private endpoint, service endpoint, VPN, or ExpressRoute.
- **Control**: RBAC, policy, locks, access policies, NSGs, or DNS.

If you train yourself to classify the requirement first, you avoid answers that are technically related but wrong for the exact scenario.

**Scope traps: subscription, management group, resource group, and resource**

Scope is one of the most common exam traps. Azure RBAC and Azure Policy both inherit downward, but they solve different problems.

- **Management group**:
  - Use it when you need consistent governance across multiple subscriptions.
  - Common for policy assignment, RBAC delegation, and standardization.
- **Subscription**:
  - Use it when the requirement applies to all resources in one billing boundary.
  - Often the right answer for broad access or policy within a single subscription.
- **Resource group**:
  - Use it when the requirement applies to a workload boundary.
  - Useful for lifecycle management, delegation, and cleanup.
- **Resource**:
  - Use it when only one resource needs a setting, lock, or role assignment.

A common trap is assigning a role at the resource group level and expecting it to control data access everywhere. RBAC at the resource group scope grants management permissions over resources in that group, but it does not automatically grant data-plane access unless the service uses Azure RBAC for data access and the role includes those permissions.

For example:

- A **Contributor** role on a resource group lets you manage resources.
- It does **not** automatically let you read storage blobs or Key Vault secrets.
- For Key Vault, the vault uses **either** Azure RBAC **or** access policies, depending on its authorization model. You do not evaluate both models at the same time for the same permission path.

**Identity traps: app registration, enterprise application, service principal, and managed identity**

AZ-104 questions often blur identity objects. You need to separate the application definition from the runtime identity.

- **App registration**:
  - The application object in Entra ID.
  - Defines how an app authenticates.
  - Used when you create an app for OAuth, certificates, secrets, or API permissions.
- **Enterprise application**:
  - The service principal representation in a tenant.
  - Used for assignment, sign-in, and access control.
- **Service principal**:
  - The security identity used by apps and automation in a tenant.
  - Often the object you grant RBAC roles to.
- **Managed identity**:
  - A service principal managed by Azure.
  - Best when an Azure resource needs to authenticate without storing secrets.

Scenario clue examples:

- If the question says “an Azure VM must access Key Vault without storing credentials,” think **managed identity**.
- If it says “a third-party app needs access to Microsoft Graph,” think **app registration** and **service principal**.
- If it says “grant a workload access to Azure resources with no secret rotation,” managed identity is usually the best answer.
- If it says “allow a user to sign in to an app,” think **enterprise application** and assignment.

A useful mental model is:

- App registration = *definition*
- Service principal = *instance in a tenant*
- Managed identity = *Azure-managed service principal*

**Authorization traps: RBAC, access policies, and locks**

Azure authorization questions often hide the difference between management access and data access.

- **Azure RBAC**:
  - Controls who can manage Azure resources.
  - Can also control data access for some services, such as Storage and Key Vault, when the service supports RBAC for data plane.
- **Access policies**:
  - Used by some services, especially Key Vault in the access policy model.
  - Control data-plane permissions directly.
- **Resource locks**:
  - Prevent accidental deletion or modification.
  - Do not grant access.
  - Are a common exam trap when the requirement is “protect from deletion” rather than “restrict who can change settings.”

Lock types:

- **CanNotDelete**:
  - Users can read and modify, but not delete.
- **ReadOnly**:
  - Users can read, but not modify or delete.

Use locks when the requirement is protection against accidental change, not delegation. If a question says “ensure a resource cannot be deleted even by users with Contributor,” a lock is often the correct answer.

> A lock can block operations even when RBAC allows them. If the scenario is about preventing deletion or modification, check for a lock before changing roles.

**Azure Policy traps: compliance versus access**

Azure Policy evaluates whether resources comply with rules. It does not replace RBAC.

- **Policy**:
  - Enforces standards.
  - Can deny, audit, append, or deploy configurations.
- **RBAC**:
  - Grants permissions.
  - Does not enforce configuration compliance.

Common exam trap:

- “Only allow storage accounts in a specific region” is a **policy** requirement.
- “Allow a user to create storage accounts” is an **RBAC** requirement.

Policy scope matters:

- Assign at **management group** for organization-wide standards.
- Assign at **subscription** for one environment.
- Assign at **resource group** for a workload-specific boundary.

Example policy assignment with Azure CLI:

```bash
az policy assignment create \
  --name deny-nonapproved-locations \
  --scope /subscriptions/<subscription-id> \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id> \
  --params '{"listOfAllowedLocations":{"value":["eastus","westus2"]}}'
```

If the question asks for “prevent noncompliant resources from being created,” choose a **deny** policy. If it asks for “report on noncompliance,” choose **audit**.

**Storage traps: authentication, SAS, account keys, and access tiers**

Storage questions often test whether you know which authentication method fits the scenario.

- **Account key**:
  - Full access to the storage account.
  - Simple, but broad and risky.
  - Best avoided for user-facing or long-lived access when alternatives exist.
- **Shared Access Signature (SAS)**:
  - Delegated access with limited permissions and expiry.
  - Good for temporary access to blobs, files, queues, or tables depending on the SAS type.
- **User delegation SAS**:
  - A SAS signed with Microsoft Entra ID credentials for Blob storage.
  - Preferred when you want temporary blob access without exposing account keys.
- **Microsoft Entra ID authorization**:
  - Best for applications and users when supported.
  - Stronger governance and easier revocation.

Scenario clues:

- “Temporary upload link for a vendor” often points to **SAS**.
- “App should access blobs without storing the account key” often points to **managed identity** or **user delegation SAS**.
- “Break-glass administrative access to storage” may point to **account key** if the scenario explicitly needs full control.

Example SAS generation for a blob container:

```bash
az storage container generate-sas \
  --account-name mystorageacct \
  --name uploads \
  --permissions acdlrw \
  --expiry 2026-05-01T00:00Z \
  --auth-mode login \
  --as-user
```

Access tiers are another trap:

- **Hot**:
  - Frequent access.
- **Cool**:
  - Less frequent access, lower storage cost, higher access cost.
- **Archive**:
  - Lowest storage cost, highest retrieval delay and rehydration time.

Lifecycle management can move blobs between tiers, but not every blob type or condition supports every transition in every context. Archive is not instant access, so do not choose it when the scenario requires immediate reads.

Example lifecycle rule:

```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-to-archive",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToArchive": {
              "daysAfterModificationGreaterThan": 180
            }
          }
        }
      }
    }
  ]
}
```

**Networking traps: NSGs, ASGs, UDRs, service endpoints, and private endpoints**

Networking questions often look similar but differ by traffic path and control point.

- **Network Security Group (NSG)**:
  - Stateful packet filtering at subnet or NIC level.
  - Use it to allow or deny traffic by IP, port, and protocol.
- **Application Security Group (ASG)**:
  - Logical grouping of VMs for NSG rules.
  - Use it to simplify rules when many VMs share a role.
- **User Defined Route (UDR)**:
  - Custom route table entry.
  - Use it to force traffic through a firewall, NVA, or hub.
- **Service endpoint**:
  - Extends a VNet identity to a service over the Azure backbone.
  - The service still uses its public endpoint, but access is restricted to selected subnets/VNets.
- **Private endpoint**:
  - Gives the service a private IP in your VNet.
  - Traffic stays private and resolves to a private address through DNS.

A frequent exam trap is confusing service endpoints with private endpoints.

- With a **service endpoint**, the service is still reached via its public endpoint, but only approved subnets can access it.
- With a **private endpoint**, the service is reached via a private IP in your VNet.

Use a private endpoint when the requirement says:

- no public exposure,
- private IP access,
- access from on-premises through private routing,
- or strict data exfiltration control.

Example private endpoint creation:

```bash
az network private-endpoint create \
  --name pe-storage \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id /subscriptions/<sub-id>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct \
  --group-id blob \
  --connection-name pe-storage-conn
```

Private endpoint DNS is a major trap. If name resolution is wrong, the endpoint exists but clients still hit the public address.

- Use a **private DNS zone** for the service, such as `privatelink.blob.core.windows.net`.
- Link the zone to the VNet.
- Ensure custom DNS servers can resolve or forward the zone correctly.
- For hybrid connectivity, make sure on-premises DNS can resolve the private name through conditional forwarders or Azure DNS Private Resolver.

If the question says “the private endpoint is deployed but the app still cannot connect,” check DNS before changing the network rule.

**Hybrid connectivity traps: VPN Gateway versus ExpressRoute**

Hybrid questions usually ask whether you need internet-based encrypted connectivity or private dedicated connectivity.

- **VPN Gateway**:
  - Uses encrypted tunnels over the public internet.
  - Good for site-to-site, point-to-site, and smaller or faster-to-deploy hybrid needs.
- **ExpressRoute**:
  - Private connectivity through a connectivity provider.
  - Better for predictable latency, higher throughput, and enterprise-grade private links.

Scenario clues:

- “Quickly connect a branch office securely” often points to **VPN Gateway**.
- “Dedicated private connection with predictable performance” often points to **ExpressRoute**.
- “Need failover between internet VPN and private circuit” may require both, with routing designed for resilience.

Do not confuse “private” in the business sense with “private endpoint” in the Azure networking sense. ExpressRoute is private connectivity to Azure, while private endpoints are private IPs to specific PaaS services.

**DNS traps: private DNS zones, custom DNS, and resolution order**

DNS is a common hidden dependency in Azure scenarios. Many “network” problems are actually name resolution problems.

- **Private DNS zone**:
  - Resolves private endpoint names to private IPs.
  - Must be linked to the VNets that need resolution.
- **Custom DNS**:
  - Often used in hybrid environments.
  - Must forward Azure private zones correctly.
- **Azure-provided DNS**:
  - Works for many simple scenarios.
  - May not be enough for hybrid or custom name resolution needs.

Common trap patterns:

- A private endpoint works from one VNet but not another because the private DNS zone is not linked.
- On-premises clients cannot resolve the private endpoint because the custom DNS server lacks a forwarder.
- The resource is reachable by IP but not by name because the private zone record is missing.

Example private DNS zone setup:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.blob.core.windows.net \
  --name link-hub-vnet \
  --virtual-network vnet-hub \
  --registration-enabled false
```

**Load balancing and availability traps: availability sets, availability zones, scale sets, and autoscale**

Availability questions often test whether you are solving for hardware failure, datacenter failure, or variable demand.

- **Availability set**:
  - Protects against planned and unplanned host failures within a datacenter.
  - Use for legacy or single-region VM resilience.
- **Availability zones**:
  - Protect against datacenter-level failure within a region.
  - Use when the region supports zones and the workload needs higher resilience.
- **Virtual Machine Scale Sets (VMSS)**:
  - Deploy and manage identical VMs at scale.
  - Use for horizontal scaling and consistent configuration.
- **Autoscale**:
  - Adjusts instance count based on metrics or schedules.
  - Use when demand changes over time.

Scenario interpretation:

- If the question says “survive a datacenter outage,” think **availability zones**.
- If it says “keep VMs available during host maintenance,” think **availability set**.
- If it says “handle variable web traffic automatically,” think **VMSS with autoscale**.
- If it says “manually add more VMs for a seasonal event,” VMSS still fits, but autoscale is the stronger answer when the requirement is automatic scaling.

For load balancing:

- **Azure Load Balancer**:
  - Layer 4.
  - Good for TCP/UDP and non-HTTP traffic.
- **Application Gateway**:
  - Layer 7.
  - Good for HTTP/HTTPS, path-based routing, TLS termination, and web application scenarios.
- **Front Door**:
  - Global HTTP/HTTPS entry point with edge-based routing and acceleration.
  - Good for global web apps and failover.
- **Traffic Manager**:
  - DNS-based global routing.
  - Good for directing clients to healthy endpoints, but it does not proxy traffic or provide true edge-layer HTTP features.

A common trap is saying Traffic Manager gives “nearest endpoint” in the same way Front Door does. Traffic Manager makes DNS decisions; it does not inspect or proxy HTTP traffic.

**Monitoring traps: diagnostic settings, metrics, logs, and destination limits**

Azure Monitor questions often ask where to send logs and metrics, but the exact support depends on the resource type.

- **Diagnostic settings**:
  - Send platform logs and supported metrics to destinations such as Log Analytics, Storage, or Event Hub.
  - Not every resource supports every destination or every category.
- **Metrics**:
  - Numeric time-series data.
  - Useful for alerting and performance trends.
- **Logs**:
  - Event records and diagnostic details.
  - Useful for troubleshooting and auditing.

Scenario clues:

- “Centralize logs for query and alerting” often points to **Log Analytics**.
- “Archive logs for long-term retention” often points to **Storage account**.
- “Stream events to a SIEM or custom consumer” often points to **Event Hub**.

Example diagnostic settings CLI:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.Web/sites/myapp \
  --workspace /subscriptions/<sub-id>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

If the question implies every resource can send every metric category to every destination, be cautious. Resource support varies.

**Backup and recovery traps: vault types, soft delete, and immutability**

Backup questions often hinge on which vault type and protection feature the service uses.

- **Recovery Services vault**:
  - Common for Azure VM backup and many classic backup scenarios.
  - Supports backup policies, restore points, and soft delete behavior for protected items.
- **Backup vault**:
  - Used for newer backup scenarios and certain workloads.
  - Supports modern backup capabilities depending on workload type.
- **Soft delete**:
  - Helps protect backup data from accidental deletion for supported services and configurations.
  - Not universal across all Azure services in the same way.
- **Immutability**:
  - Prevents backup data from being altered or deleted for a retention period or permanently, depending on configuration and service support.

Scenario clues:

- “Protect backups from accidental deletion” may point to **soft delete** or **immutability**, depending on the service and requirement.
- “Prevent anyone from changing backup data” often points to **immutability**.
- “Restore a deleted VM from backup” often points to a **Recovery Services vault**.

Be precise: soft delete is not a generic guarantee for all blobs, backups, or Key Vault items. It is a service-specific protection feature.

**Entra ID and Conditional Access traps**

Conditional Access questions usually test whether you understand policy conditions and enforcement.

- **Conditional Access**:
  - Evaluates user, app, device, location, and risk conditions.
  - Can require MFA, compliant devices, or block access.
- **MFA**:
  - Multi-Factor Authentication.
  - Often required by policy rather than configured per app.
- **Compliant device**:
  - A device that meets Intune or device compliance requirements.

Example scenario:

- “Require MFA only when users access the Azure portal from outside the corporate network” is a **Conditional Access** policy.
- “Block legacy authentication” is also a common Conditional Access use case.

Example policy conceptually:

```text
Users: All users
Cloud apps: Microsoft Azure Management
Conditions: Locations = outside trusted locations
Grant: Require multi-factor authentication
```

If the question asks for a tenant-wide sign-in control, think Conditional Access rather than RBAC.

**Common CLI patterns you should recognize**

You do not need to memorize every command, but you should recognize the shape of common Azure CLI operations.

Role assignment:

```bash
az role assignment create \
  --assignee <user-or-sp-object-id-or-upn> \
  --role "Reader" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-app
```

Resource lock:

```bash
az lock create \
  --name prevent-delete \
  --lock-type CanNotDelete \
  --resource-group rg-app
```

Private endpoint:

```bash
az network private-endpoint create \
  --name pe-kv \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id /subscriptions/<sub-id>/resourceGroups/rg-sec/providers/Microsoft.KeyVault/vaults/kv-prod \
  --group-id vault \
  --connection-name pe-kv-conn
```

Policy assignment:

```bash
az policy assignment create \
  --name enforce-tags \
  --scope /subscriptions/<sub-id> \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policy-id>
```

These examples help you map exam wording to the right control plane action.

**Scenario interpretation checklist**

When you read an AZ-104 scenario, ask yourself:

1. What is the exact goal: access, protection, compliance, routing, availability, or recovery?
2. Is the requirement about management access or data access?
3. What scope does the requirement apply to?
4. Does the service need public access, private access, or both?
5. Is the issue identity, DNS, routing, or authorization?
6. Is the requirement preventive, detective, or corrective?

A few high-value exam patterns:

- “Prevent deletion” → **resource lock**
- “Enforce allowed regions” → **Azure Policy**
- “Grant app access without secrets” → **managed identity**
- “Temporary external access to blob data” → **SAS**
- “Private access to PaaS” → **private endpoint**
- “Restrict access to a subnet but keep public endpoint” → **service endpoint**
- “Global HTTP failover with proxying” → **Front Door**
- “DNS-based endpoint selection” → **Traffic Manager**
- “Private connectivity to on-premises” → **VPN Gateway** or **ExpressRoute**
- “Scale out automatically” → **VMSS with autoscale**

**Mini-scenarios and the likely answer**

- A finance team wants all subscriptions under one governance model with the same allowed regions and tagging rules.
  - Use **management group** scope with **Azure Policy**.
- A web app needs to read secrets from Key Vault without storing credentials.
  - Use **managed identity** and the vault’s chosen authorization model.
- A storage container must be shared with a vendor for 24 hours only.
  - Use a **SAS** with expiry and least privilege.
- A VM must be protected from accidental deletion by admins.
  - Use a **CanNotDelete** lock.
- A private endpoint is deployed, but the app still resolves the public address.
  - Fix **private DNS** and name resolution.
- A workload needs private connectivity from on-premises with predictable performance.
  - Use **ExpressRoute** if the requirement is dedicated private connectivity; use **VPN Gateway** if encrypted internet tunneling is sufficient.

**Practical exam habits**

- Read for the *constraint*, not just the service name.
- Eliminate answers that solve the wrong plane.
- Watch for words like “only,” “must,” “without,” “prevent,” and “temporary.”
- Treat DNS as a first-class dependency in private networking questions.
- Assume the simplest service that satisfies the requirement, but not a weaker one.

If two answers both sound plausible, the better one usually matches the exact control point: policy for compliance, RBAC for access, lock for protection, DNS for name resolution, and private endpoint for private data access.
