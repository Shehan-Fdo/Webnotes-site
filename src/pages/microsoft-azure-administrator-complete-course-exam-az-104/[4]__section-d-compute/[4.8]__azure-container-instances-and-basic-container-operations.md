## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure Container Instances overview**

Azure Container Instances (ACI) lets you run containers without managing virtual machines or a full container orchestrator. You use it when you want fast, isolated container execution with minimal infrastructure overhead.

ACI is a strong fit for:
- Short-lived jobs and batch tasks
- Burst workloads that need quick start-up
- Simple containerized services
- Build/test automation and scheduled tasks
- Sidecar-style helper containers in the same container group

ACI is not a full application platform like Azure App Service or Kubernetes. It does not provide built-in autoscaling, rolling updates, service discovery, or advanced orchestration features. For AZ-104, focus on when ACI is the right lightweight choice and how to deploy, secure, monitor, and troubleshoot it.

**Core concepts**

A container instance in Azure is deployed inside a **container group**. The container group is the atomic deployment unit:
- All containers in the group share the same lifecycle.
- They share the same network namespace and IP address.
- They can share mounted volumes.
- They are scheduled together on the same host.

This model is useful for sidecars, log shippers, and helper containers that must run beside the main app.

<dl>
<dt><strong>Container group</strong></dt>
<dd>The deployment boundary in ACI. You create, update, and delete the group as a unit.</dd>
<dt><strong>Restart policy</strong></dt>
<dd>Controls whether containers restart automatically after exit. Common values are <code>Always</code>, <code>OnFailure</code>, and <code>Never</code>.</dd>
<dt><strong>Image</strong></dt>
<dd>The container image pulled from a registry such as Docker Hub or Azure Container Registry (ACR).</dd>
<dt><strong>CPU and memory</strong></dt>
<dd>Resources requested for the container group. You size them when you create the group.</dd>
</dl>

**When to use ACI**

Use ACI when you need:
- Fast provisioning without VM management
- Per-run isolation for jobs or scripts
- A simple way to run a container in a private network
- A temporary endpoint for testing or demos
- A lightweight companion container beside a primary container

Use something else when you need:
- Horizontal autoscaling
- Complex multi-service orchestration
- Persistent application hosting with rich platform features
- Advanced traffic management and service discovery

> ACI is best thought of as a container runtime service, not a general-purpose web hosting platform.

**Container group design**

A container group can contain one or more containers. Multi-container groups are useful when containers must share:
- Network access
- Storage mounts
- Lifecycle
- Environment context

Common patterns:
- **Main app + sidecar**: app container plus log collector or proxy
- **Worker + helper**: job container plus a small utility container
- **Init-like behavior**: one container prepares data while another consumes it, though ACI does not provide true init containers like Kubernetes

Example sidecar layout:
- `app`: serves the application
- `log-agent`: tails files and forwards logs
- Shared Azure File share: both containers read/write the same mounted path

**Deployment options**

You can create ACI from:
- Azure portal
- Azure CLI
- ARM templates or Bicep
- Azure PowerShell
- Automation pipelines

The portal is useful for learning the required fields and for quick deployments. CLI and templates are better for repeatable administration.

**Azure portal deployment workflow**

In the Azure portal, you typically create a container instance by filling in:
- Subscription and resource group
- Container name
- Region
- Image source and image name
- OS type: Linux or Windows
- Size: CPU cores and memory
- Restart policy
- Networking settings
- Environment variables
- Command override
- Volumes and mounts
- Monitoring settings

Key portal areas to know:
- **Basics**: name, region, image, size, restart policy
- **Networking**: public IP, DNS label, ports, virtual network placement
- **Advanced**: environment variables, command override, identity, GPU options if available in your region and SKU context
- **Monitoring**: diagnostics, logs, and Log Analytics workspace selection

**CLI creation example**

A basic Linux container group with a public IP:

```bash
az container create \
  --resource-group rg-aci \
  --name aci-web \
  --image mcr.microsoft.com/azuredocs/aci-helloworld \
  --cpu 1 \
  --memory 1.5 \
  --ports 80 \
  --dns-name-label aciwebdemo12345 \
  --restart-policy Always
```

What this does:
- Pulls a public image
- Allocates 1 vCPU and 1.5 GiB memory
- Exposes port 80
- Assigns a public DNS name
- Restarts the container if it exits

Common checks:
```bash
az container show \
  --resource-group rg-aci \
  --name aci-web \
  --query "{state:instanceView.state, ip:ipAddress.ip, fqdn:ipAddress.fqdn}" \
  --output table
```

**Resource sizing and limits**

ACI requires you to specify CPU and memory at deployment time. These are not elastic in the same way as a VM scale set or Kubernetes pod with autoscaling.

Important sizing points:
- You request CPU and memory for the container group.
- The platform schedules the group based on available capacity.
- If the container exceeds its memory allocation, it is typically terminated by the platform.
- CPU contention can slow the workload, but memory exhaustion is usually the more visible failure mode.

Practical guidance:
- Start with the smallest size that fits the workload.
- Increase memory first when troubleshooting crashes or OOM-like behavior.
- Use logs and metrics to confirm whether the workload is CPU-bound or memory-bound.

Example sizing for a small API or worker:
- 0.5 CPU, 1.0 GiB memory for light tasks
- 1 CPU, 1.5–2.0 GiB memory for moderate workloads
- Larger sizes for build jobs, data processing, or memory-heavy apps

**Environment variables and command overrides**

Environment variables let you inject configuration without rebuilding the image. This is one of the most common ACI administration tasks.

Example:
```bash
az container create \
  --resource-group rg-aci \
  --name aci-app \
  --image myregistry.azurecr.io/app:1.0 \
  --cpu 1 \
  --memory 1.5 \
  --environment-variables \
    ASPNETCORE_ENVIRONMENT=Production \
    LOG_LEVEL=Information \
  --secure-environment-variables \
    DB_PASSWORD='P@ssw0rd!' \
  --ports 8080
```

Use:
- `--environment-variables` for non-sensitive values
- `--secure-environment-variables` for secrets

Command override is useful when the image entrypoint is not suitable for the task. You can override the default startup command to run a script, a shell, or a one-off job.

Example:
```bash
az container create \
  --resource-group rg-aci \
  --name aci-job \
  --image alpine:3.20 \
  --command-line "/bin/sh -c 'echo hello from ACI && sleep 30'" \
  --restart-policy Never
```

Use command overrides when:
- The image is generic and needs a custom startup command
- You want to run a one-time task
- You need to debug an image interactively

**Restart policy and workload behavior**

Restart policy is important for exam scenarios because it changes how ACI behaves after the container exits.

Common policies:
- `Always`: restart after exit, suitable for long-running services
- `OnFailure`: restart only if the process exits with a non-zero code, useful for jobs that may retry
- `Never`: do not restart, best for one-shot tasks and troubleshooting

Use cases:
- **Long-running service**: `Always`
- **Batch job**: `Never` or `OnFailure`
- **Transient worker with retry**: `OnFailure`

Example batch job:
```bash
az container create \
  --resource-group rg-aci \
  --name aci-batch \
  --image mcr.microsoft.com/azuredocs/aci-helloworld \
  --restart-policy Never
```

Troubleshooting tip:
- If a job exits successfully but keeps restarting, the restart policy is too aggressive.
- If a service exits unexpectedly and does not come back, check the restart policy and the container logs.

**Networking basics**

ACI supports public and private networking scenarios.

Public networking:
- The container group gets a public IP address.
- You can optionally assign a DNS label.
- You expose one or more ports from the container group.

Private networking:
- The container group is deployed into a delegated subnet in a virtual network.
- The group receives a private IP address.
- You use internal routing and private DNS patterns to reach it.

Important port rule:
- ACI does not translate ports automatically.
- The application must listen on the same port that you expose, or you must configure the app to listen on the exposed port.

Example:
- If the app listens on `8080`, expose `8080`
- If you expose `80`, the app must listen on `80` unless you change the app configuration

Public deployment example:
```bash
az container create \
  --resource-group rg-aci \
  --name aci-public \
  --image nginx \
  --ports 80 \
  --dns-name-label aci-nginx-demo-12345 \
  --ip-address Public
```

**Private deployment prerequisites**

For private ACI, the subnet must be prepared correctly before deployment.

Required considerations:
- The subnet must be delegated to `Microsoft.ContainerInstance/containerGroups`
- The subnet should have enough free IP addresses for the container group
- The subnet must not conflict with other delegated services or address planning
- Network security groups and route tables must allow the required traffic
- Private DNS may be needed for name resolution if the workload must be reached by name

Typical subnet delegation command:
```bash
az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name vnet-app \
  --name aci-subnet \
  --delegations Microsoft.ContainerInstance/containerGroups
```

Private deployment example:
```bash
az container create \
  --resource-group rg-aci \
  --name aci-private \
  --image myregistry.azurecr.io/app:1.0 \
  --vnet vnet-app \
  --subnet aci-subnet \
  --ip-address Private \
  --ports 8080
```

Common private-network failure points:
- Subnet not delegated
- Subnet too small
- NSG blocks required traffic
- DNS not configured for internal resolution
- Image pull fails because the registry is not reachable or credentials are missing

**Storage and volume mounts**

ACI supports mounting storage, commonly Azure Files, so containers can persist data beyond the container lifecycle.

Why this matters:
- Container filesystems are ephemeral
- Logs, uploads, and job outputs disappear when the container group is deleted
- Shared storage lets multiple containers in the same group access the same files

Azure Files mount example:
```bash
az container create \
  --resource-group rg-aci \
  --name aci-files \
  --image mcr.microsoft.com/azuredocs/aci-helloworld \
  --cpu 1 \
  --memory 1.5 \
  --azure-file-volume-account-name mystorageacct \
  --azure-file-volume-account-key '<storage-key>' \
  --azure-file-volume-share-name acishare \
  --azure-file-volume-mount-path /mnt/data
```

Use mounted storage for:
- Uploads
- Shared configuration
- Job output
- Temporary persistence across restarts

Do not rely on the container filesystem for durable data.

**Multi-container groups and sidecars**

Multi-container groups are one of the most useful ACI features for exam scenarios.

Example pattern:
- `web`: application container
- `logger`: sidecar that ships logs
- Shared Azure File mount for log files

Example YAML:
```yaml
apiVersion: 2019-12-01
location: eastus
name: aci-sidecar-demo
properties:
  containers:
    - name: app
      properties:
        image: nginx
        resources:
          requests:
            cpu: 1
            memoryInGB: 1.5
        ports:
          - port: 80
        volumeMounts:
          - name: sharedfiles
            mountPath: /usr/share/nginx/html
    - name: helper
      properties:
        image: alpine:3.20
        command:
          - /bin/sh
          - -c
          - while true; do echo "sidecar alive"; sleep 30; done
        resources:
          requests:
            cpu: 0.5
            memoryInGB: 1.0
        volumeMounts:
          - name: sharedfiles
            mountPath: /data
  osType: Linux
  restartPolicy: Always
  volumes:
    - name: sharedfiles
      azureFile:
        shareName: acishare
        storageAccountName: mystorageacct
        storageAccountKey: "<storage-key>"
```

Sidecar guidance:
- Keep the sidecar lightweight
- Make sure both containers tolerate the same restart policy
- Remember that the group is the deployment unit, so one container’s failure can affect the whole group depending on the restart behavior

**Identity and registry access**

ACI can use managed identity to access Azure resources such as Azure Key Vault or Azure Container Registry, depending on the scenario and permissions.

Common use cases:
- Pulling images from a private registry
- Reading secrets from Key Vault
- Accessing storage or other Azure services without embedding credentials

For private registry access, you can also use registry credentials:
```bash
az container create \
  --resource-group rg-aci \
  --name aci-private-registry \
  --image myregistry.azurecr.io/app:1.0 \
  --registry-login-server myregistry.azurecr.io \
  --registry-username myregistry \
  --registry-password '<password>'
```

Managed identity workflow:
1. Enable a system-assigned or user-assigned identity on the container group.
2. Assign the required Azure role to that identity.
3. Reference the identity from your app code or deployment logic.

Example role assignment concept:
```bash
az role assignment create \
  --assignee-object-id <identity-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "AcrPull" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.ContainerRegistry/registries/<acr-name>
```

**Monitoring and diagnostics**

ACI integrates with Azure Monitor and Log Analytics for operational visibility.

What to monitor:
- Container start and stop events
- Exit codes
- Restart counts
- CPU and memory usage
- Application logs
- Image pull failures
- Network reachability issues

Best practice:
- Send logs to a Log Analytics workspace for centralized troubleshooting.
- Use Azure Monitor metrics and logs to correlate failures with deployment changes.

Portal monitoring setup:
- In the container group creation flow, enable diagnostics or connect a Log Analytics workspace.
- After deployment, use the Logs blade and metrics to inspect runtime behavior.

CLI example with Log Analytics:
```bash
az container create \
  --resource-group rg-aci \
  --name aci-monitored \
  --image mcr.microsoft.com/azuredocs/aci-helloworld \
  --log-analytics-workspace <workspace-id> \
  --log-analytics-workspace-key '<workspace-key>'
```

Useful troubleshooting commands:
```bash
az container logs \
  --resource-group rg-aci \
  --name aci-web

az container attach \
  --resource-group rg-aci \
  --name aci-web
```

**Troubleshooting scenarios**

If the container does not start:
- Check the image name and tag
- Verify registry credentials or managed identity permissions
- Confirm the region supports the image and requested features
- Review logs for startup exceptions

If the app is unreachable:
- Confirm the container listens on the exposed port
- Verify the port is open in the container group configuration
- Check DNS label or private DNS resolution
- Confirm NSG and routing rules for private deployments

If the container keeps restarting:
- Check the restart policy
- Inspect the process exit code
- Determine whether the app is crashing or intentionally exiting
- For jobs, use `Never` so completion is visible

If the container runs but behaves oddly:
- Verify environment variables
- Check command override syntax
- Confirm mounted paths exist and permissions are correct
- Validate CPU and memory sizing

**Update and lifecycle management**

ACI is often managed by redeployment rather than in-place editing.

Important lifecycle behavior:
- Some settings can be changed, but many meaningful changes require deleting and recreating the container group.
- Image updates usually mean redeploying with a new tag or digest.
- Network changes often require recreation.
- Storage and identity changes may also require redeployment depending on the property.

Practical rule:
- Treat the container group as an immutable deployment unit for most exam scenarios.

Recommended workflow:
1. Export or store the deployment definition.
2. Change the image tag, environment variables, or resource settings.
3. Recreate the container group.
4. Validate logs and connectivity.

**Real-world walkthrough: private web app with logs**

Scenario:
You need to run a small internal web app in a private subnet and capture logs for troubleshooting.

Approach:
1. Create or choose a delegated subnet for ACI.
2. Deploy the container group into that subnet with a private IP.
3. Expose the correct application port.
4. Mount Azure Files for shared logs or app data.
5. Send logs to Log Analytics.
6. Test connectivity from a VM in the same virtual network or peered network.

Example deployment shape:
- `app` container listens on `8080`
- Private IP only
- Azure Files mounted at `/var/log/app`
- Log Analytics enabled
- Restart policy `Always`

This pattern is common in exam questions because it combines networking, storage, monitoring, and lifecycle behavior.

**Real-world walkthrough: one-shot batch job**

Scenario:
You need to run a script that processes files and exits.

Approach:
1. Use a small Linux image with your script baked in.
2. Set `--restart-policy Never`.
3. Pass input values through environment variables.
4. Mount Azure Files or use blob storage for input and output.
5. Check logs after completion.

Example:
```bash
az container create \
  --resource-group rg-aci \
  --name aci-transform \
  --image myregistry.azurecr.io/transform:2.1 \
  --restart-policy Never \
  --environment-variables INPUT_PATH=/mnt/input OUTPUT_PATH=/mnt/output \
  --azure-file-volume-account-name mystorageacct \
  --azure-file-volume-account-key '<storage-key>' \
  --azure-file-volume-share-name batchshare \
  --azure-file-volume-mount-path /mnt
```

This is a better fit for ACI than a long-running web app because the workload is finite and does not need orchestration features.

**Exam focus reminders**

- Know that ACI runs containers without managing VMs.
- Know that the container group is the deployment unit.
- Know the difference between public and private networking.
- Know that restart policy affects long-running services and one-shot jobs differently.
- Know that environment variables, command overrides, and mounted storage are common configuration tools.
- Know that monitoring usually involves Azure Monitor and Log Analytics.
- Know that many changes require redeployment rather than in-place modification.
- Know that ACI is lightweight and not a replacement for Kubernetes or a full app hosting platform.

**Quick comparison**

| Capability | ACI | VM | Kubernetes |
|---|---:|---:|---:|
| Infrastructure management | Low | High | Medium to high |
| Startup speed | Fast | Slower | Fast to moderate |
| Autoscaling | No built-in | External | Yes |
| Orchestration depth | Low | None | High |
| Best for | Simple container workloads | Full OS control | Complex container platforms |

**Key commands to remember**

```bash
az container create
az container show
az container logs
az container attach
az container delete
az container exec
```

Use `az container exec` when you need an interactive shell in a running container for debugging, if supported by the image and state of the container.

**Common pitfalls**

- Exposing the wrong port and assuming ACI maps it automatically
- Using the container filesystem for data that must persist
- Choosing `Always` for a job that should exit once
- Forgetting subnet delegation for private deployments
- Missing registry permissions for private images
- Expecting autoscaling or rolling updates from ACI
- Treating a container group like a mutable VM instead of a redeployable unit
