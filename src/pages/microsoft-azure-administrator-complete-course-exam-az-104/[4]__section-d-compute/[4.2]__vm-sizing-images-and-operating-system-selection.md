## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**VM sizing fundamentals**

Azure VM size selection is a capacity decision, not just a pricing decision. You choose a size based on CPU, memory, storage throughput, disk count, network performance, and feature support. For AZ-104, expect scenario questions that ask you to match a workload to a VM family and to explain why one size is better than another.

A VM size name often encodes useful information. For example, `Standard_D2s_v5` usually means:

- **Standard**: the SKU family prefix used by Azure.
- **D**: the series family.
- **2**: number of vCPUs.
- **s**: premium storage support.
- **v5**: generation/version of the series.

Common families you should recognize include:

- **B-series**: burstable VMs for low, intermittent workloads.
- **D-series**: general purpose for balanced CPU and memory.
- **E-series**: memory optimized for in-memory databases and analytics.
- **F-series**: compute optimized for CPU-heavy workloads.
- **L-series**: storage optimized for high disk throughput and low latency.
- **N-series**: GPU-enabled for graphics, AI, and specialized compute.

A practical way to think about sizing is to map the workload bottleneck:

- **CPU-bound** workloads benefit from compute-optimized sizes such as F-series.
- **Memory-bound** workloads often fit E-series.
- **Bursting workloads** can use B-series when sustained high CPU is not required.
- **Disk-heavy workloads** may need L-series or a size with higher storage throughput.
- **GPU workloads** require N-series and compatible images/drivers.

**How to interpret size names**

Azure size names are useful when you need to compare options quickly. The suffixes matter because they affect capabilities, not just performance.

- **`s`** usually indicates support for premium storage.
- **`v2`, `v3`, `v4`, `v5`** indicate newer generations with different hardware and feature sets.
- **`a`** or other letters may appear in some families and can indicate hardware or feature variants.
- **`ds`** often means a D-series size with premium storage support.

When you compare sizes, do not focus only on vCPU count. Two sizes with the same number of cores can differ in memory, disk throughput, NIC performance, and supported features.

**Choosing a size in practice**

Use measurable workload needs rather than guesswork. A few exam-style examples:

- A small internal web server with light traffic can start on a B-series or a small D-series VM.
- A SQL Server VM with a memory-heavy workload usually needs an E-series size and premium disks.
- A build agent that compiles code all day is often better on an F-series size.
- A file-processing workload that reads and writes large datasets may need an L-series size with high disk throughput.
- A remote visualization or AI inference workload may require an N-series VM.

If you are unsure, start with a conservative general-purpose size, monitor CPU, memory, disk queue length, and network usage, then resize based on actual telemetry.

**Azure VM images**

An Azure VM image is the template used to create the operating system and preinstalled software state for a VM. Image choice affects patching, maintenance, compatibility, and deployment speed.

There are three common image sources:

- **Marketplace images**: published by Microsoft or third parties in Azure Marketplace.
- **Custom images**: created by your organization from a configured VM or captured source.
- **Azure Compute Gallery images**: centrally managed image definitions and versions for repeatable deployment.

Azure Compute Gallery is the current name you will often see for what was previously called Shared Image Gallery.

**Marketplace images**

Marketplace images are the fastest way to deploy common operating systems and applications. They are useful when you want a supported base image without maintaining your own image pipeline.

Typical uses:

- Standard Windows Server or Linux deployments.
- Prebuilt application stacks.
- Quick lab or proof-of-concept environments.

Marketplace images can include purchase terms or plan information for some third-party images. In those cases, the deployment must include the required plan details in the portal, ARM template, Bicep, or CLI workflow. If the image requires acceptance of terms, you must accept them before deployment.

A practical CLI pattern for checking image metadata is:

```bash
az vm image list --publisher Canonical --offer UbuntuServer --sku 22_04-lts --all -o table
```

If a Marketplace image requires terms acceptance, you can inspect and accept them with the image-specific metadata returned by Azure. The exact publisher, offer, SKU, and plan values must match the image definition.

> Be careful with Marketplace images that include a purchase plan. The plan fields must match the image metadata exactly, or deployment can fail.

**Custom images**

Custom images are best when you need a standardized corporate baseline. They reduce configuration drift and let you preinstall agents, security tools, and application dependencies.

Use custom images when:

- You need the same hardened build across many VMs.
- You want faster provisioning with preconfigured software.
- You need to control patching and baseline settings centrally.

Trade-offs:

- You own image maintenance and patch refresh cycles.
- Old images can drift from current security baselines.
- You must rebuild or update images regularly to avoid stale software.

**Azure Compute Gallery images**

Azure Compute Gallery is the preferred way to distribute custom images at scale. It supports versioning, replication, and regional deployment control.

Key concepts:

- **Image definition**: the template metadata for an image family.
- **Image version**: a specific published build, such as `1.0.0`.
- **Replication**: copies of the image version placed into selected regions.
- **Regional deployment**: VMs can be created from the nearest replicated version for better availability and lower latency.

This model is useful when you need consistent images across subscriptions or regions. It also helps with controlled rollout: publish a new version, replicate it to test regions first, then expand to production regions.

Example gallery workflow:

```bash
az sig image-definition create \
  --resource-group rg-images \
  --gallery-name corpGallery \
  --gallery-image-definition win2022-base \
  --publisher Contoso \
  --offer win2022 \
  --sku base \
  --os-type Windows \
  --os-state generalized
```

```bash
az sig image-version create \
  --resource-group rg-images \
  --gallery-name corpGallery \
  --gallery-image-definition win2022-base \
  --gallery-image-version 1.0.0 \
  --managed-image /subscriptions/<subId>/resourceGroups/rg-images/providers/Microsoft.Compute/images/win2022-base-img \
  --target-regions eastus=1 westus2=1
```

**Generalized vs specialized images**

Image state matters for how the VM is created and what happens on first boot.

- **Generalized image**:
  - The source VM is prepared for reuse.
  - Machine-specific data is removed.
  - The new VM gets a fresh identity and initialization.
  - This is the common choice for reusable templates and galleries.

- **Specialized image**:
  - The source VM keeps its machine-specific state.
  - The new VM is effectively a clone of the original machine.
  - This is useful when you need to preserve a configured system exactly as-is.

For AZ-104, generalized images are usually the default answer when you want repeatable deployments. Specialized images are more niche and are often used when cloning a specific machine state is required.

**Operating system selection**

Your OS choice affects administration, tooling, authentication, and automation. Azure supports both Windows and Linux VMs, and the right choice depends on the workload and the team that manages it.

**Windows VM considerations**

Windows VMs are common for:

- Active Directory-related workloads.
- Microsoft application stacks.
- GUI-based administration.
- Applications that require Windows-only components.

Typical management tools:

- **RDP** (Remote Desktop Protocol) for interactive access.
- PowerShell for automation.
- Azure VM extensions for configuration and monitoring.

Example Windows VM creation:

```bash
az vm create \
  --resource-group rg-demo \
  --name vm-win01 \
  --image Win2022Datacenter \
  --admin-username azureadmin \
  --admin-password 'Use-A-Strong-Password-Here!' \
  --size Standard_D2s_v5
```

**Linux VM considerations**

Linux VMs are common for:

- Web servers and application servers.
- Containers and DevOps tooling.
- Open-source databases and services.
- Automation-heavy environments.

Typical management tools:

- **SSH** (Secure Shell) for remote access.
- Bash, Python, and cloud-init for automation.
- Azure VM extensions for monitoring and configuration.

Example Linux VM creation:

```bash
az vm create \
  --resource-group rg-demo \
  --name vm-linux01 \
  --image Ubuntu2204 \
  --admin-username azureadmin \
  --generate-ssh-keys \
  --size Standard_D2s_v5
```

`Ubuntu2204` is a CLI convenience alias in many environments. If you need precise control, use the full Marketplace URN or gallery image reference.

**OS lifecycle and support**

Operating system support lifecycle matters in Azure because an unsupported OS increases security and compliance risk.

- Prefer currently supported Windows Server and Linux releases.
- Check end-of-support dates before standardizing on an image.
- Plan image refreshes before the OS reaches end of support.
- Avoid building new workloads on obsolete versions unless you have a migration plan.

In exam scenarios, if two images are functionally similar, choose the supported version with the longer remaining lifecycle.

**Generation 1 vs Generation 2**

Azure VM generation affects boot architecture and feature support.

- **Gen 1**:
  - Legacy BIOS-based boot model.
  - Broad compatibility with older images and some older VM sizes.
  - Useful when a workload or image does not support Gen 2.

- **Gen 2**:
  - UEFI-based boot model.
  - Supports newer platform capabilities such as **Secure Boot**, **vTPM** (virtual Trusted Platform Module), and **Trusted Launch** on supported configurations.
  - Often preferred for modern deployments when the image and size support it.

Do not assume Gen 2 is always the right answer. Some images, extensions, or VM size combinations may still require Gen 1 or may not support every Gen 2 feature in every region.

**Trusted Launch and security features**

Trusted Launch improves VM boot security by combining platform protections.

Common features include:

- **Secure Boot** to help protect against boot-level malware.
- **vTPM** to support measured boot and key protection.
- **Boot integrity monitoring** for additional attestation.

These features are typically associated with Gen 2 VMs and supported images. If a scenario asks for stronger boot security, Gen 2 with Trusted Launch is often the right direction, assuming the workload and image support it.

**Disk and storage implications**

VM size selection affects disk capabilities. This is important because the image and the VM size must work together.

Consider:

- Maximum number of data disks supported by the VM size.
- Maximum disk throughput and IOPS.
- Whether the size supports premium storage.
- Whether the OS disk and data disks need different performance tiers.
- Whether the workload needs managed disks for simpler operations.

Managed disks are the standard choice in modern Azure designs because Azure handles storage account placement and scaling concerns for you. Unmanaged disks are legacy and not recommended for new deployments.

A practical example:

- A database server may need a size that supports many data disks and high throughput.
- A small app server may only need one OS disk and one data disk.
- A file server may need a size with enough disk bandwidth to avoid storage bottlenecks.

If the VM size cannot support the required disk count or throughput, the workload may perform poorly even if CPU and memory look sufficient.

**Windows vs Linux administration differences**

The OS you choose changes how you operate the VM day to day.

- **Windows**:
  - Commonly managed with RDP, PowerShell, and Group Policy in domain-joined environments.
  - Often uses Windows-specific agents and extensions.
  - Fits Microsoft-centric enterprise administration patterns.

- **Linux**:
  - Commonly managed with SSH, shell scripts, and configuration management tools.
  - Often uses cloud-init for first-boot customization.
  - Fits automation-heavy and open-source stacks.

For AZ-104, the key is not memorizing every tool. It is understanding which OS aligns with the workload and the administration model.

**Image deployment behavior and patching responsibility**

The image type changes who owns the maintenance burden.

- **Marketplace image**:
  - Faster to deploy.
  - Vendor or Microsoft provides the base image.
  - You still patch and secure the running VM.

- **Custom image**:
  - More control and consistency.
  - You own the image pipeline and refresh cycle.
  - Better for standardized enterprise builds.

- **Compute Gallery image**:
  - Best for controlled distribution of custom images.
  - Supports versioning and regional replication.
  - Good for repeatable enterprise deployments.

A common exam trap is assuming the image itself removes patching responsibility. It does not. Even if you deploy from a curated image, you still need OS updates, application updates, and security hardening after provisioning.

**Practical selection scenarios**

1. A development team needs a cheap test VM that is idle most of the day but spikes during builds.
   - Choose a **B-series** size.
   - Use a supported Linux or Windows image based on the team’s tooling.
   - Prefer a generalized image if you need repeatable rebuilds.

2. A finance application runs memory-heavy reporting jobs.
   - Choose an **E-series** size.
   - Use premium disks if the workload is storage sensitive.
   - Consider a Gen 2 image if the application stack supports it.

3. A media processing server performs CPU-intensive encoding.
   - Choose an **F-series** size.
   - Use an image with the required codecs or processing tools.
   - Verify the OS and drivers support the workload.

4. A database server needs high disk throughput and many attached disks.
   - Choose an **L-series** or another size with strong storage performance.
   - Use managed disks and verify disk controller support.
   - Select an image that supports the database platform and OS lifecycle requirements.

5. A secure workload requires boot attestation and modern platform protections.
   - Choose a **Gen 2** image and a compatible VM size.
   - Enable Trusted Launch features where supported.
   - Confirm the image supports Secure Boot and vTPM.

**Troubleshooting and validation**

When a VM deployment fails, check the size, image, and region together. Many issues are caused by mismatched assumptions rather than a single broken setting.

Useful validation commands:

```bash
az vm list-sizes --location eastus -o table
```

```bash
az vm image list-offers --location eastus --publisher Canonical -o table
```

```bash
az vm image list-skus --location eastus --publisher Canonical --offer UbuntuServer -o table
```

Use `az vm list-sizes` when you want to see available VM sizes directly. Use image listing commands when you need to confirm the exact image family, offer, and SKU.

Common failure causes:

- The selected VM size is not available in the region.
- The image is not available in the region.
- The image generation does not match the requested deployment type.
- The VM size does not support the required disk or security feature.
- Marketplace terms or plan details are missing for a paid image.
- The OS version is unsupported by the chosen feature set.

**Exam-focused takeaways**

- Match the VM family to the workload bottleneck, not just the core count.
- Read size names carefully; suffixes and versions matter.
- Use Marketplace images for speed, custom images for standardization, and Azure Compute Gallery for scalable image distribution.
- Prefer generalized images for reusable deployments and specialized images only when you need a cloned machine state.
- Check OS lifecycle before standardizing on an image.
- Use Gen 2 when you need modern security features and the image supports them.
- Remember that disk limits, throughput, and premium storage support are part of VM sizing.
- Choose Windows or Linux based on administration model, application requirements, and team skills.
