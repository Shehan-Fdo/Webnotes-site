## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Availability sets: the core idea**

An **availability set** is a logical grouping for Azure virtual machines (VMs) that helps keep an application available during planned maintenance and unplanned hardware failures. Azure places VMs in the set across separate **fault domains** and **update domains** so a single rack failure or a single maintenance event does not take down every VM at once.

Use availability sets when you need high availability for VMs in a single region but do not need zone-level isolation. They are a classic AZ-104 design choice for legacy or regional VM workloads, especially when you want multiple VMs behind a load balancer or Application Gateway.

- **Fault domain** separation protects against shared hardware failure.
- **Update domain** separation protects against simultaneous platform maintenance.
- Availability sets are a **regional** resiliency feature, not a cross-region disaster recovery solution.

**What an availability set actually guarantees**

Azure distributes VMs in an availability set across different physical infrastructure within the same region. You do not choose the exact rack or datacenter. Instead, Azure manages placement so that VMs are spread across failure boundaries.

Important exam points:

- All VMs in an availability set must be in the **same region**.
- All VMs in an availability set must be in the **same resource group**.
- A VM can belong to only one availability set.
- Availability sets are created before or during VM deployment; you cannot move an existing VM into an availability set after creation in the same way you can attach some other resources.

> Availability sets do not provide cross-region disaster recovery. For DR, pair them with **Azure Backup** and/or **Azure Site Recovery (ASR)**.

**Fault domains and update domains**

A **fault domain (FD)** is a group of VMs that share a common power source and network switch path. If that underlying hardware fails, all VMs in that fault domain can be affected.

An **update domain (UD)** is a logical grouping used by Azure during planned maintenance. Azure updates one update domain at a time, which reduces the chance that all VMs in the set restart together.

Key design implications:

- Fault domains reduce the impact of **hardware failure**.
- Update domains reduce the impact of **platform maintenance**.
- You design for both because availability is about surviving different failure types.

Typical exam rule of thumb:

- Availability sets commonly use a region-dependent fault domain count, often **2 or 3**.
- Update domain count is configurable at creation time and is commonly set to a value such as **5** or more depending on the design.
- These values are generally **set when the availability set is created** and are not something you casually change later.

**Aligned vs Classic availability sets**

Azure has two availability set architectures:

| Type | Status | Notes |
|---|---|---|
| **Aligned** | Modern/default choice | Uses managed disks and supports current Azure VM capabilities. This is the expected choice for AZ-104 scenarios. |
| **Classic** | Legacy | Older model associated with unmanaged disks and older deployment patterns. Rarely the right answer on modern exams. |

For exam purposes, assume you should choose **Aligned** unless a question explicitly describes a legacy environment.

**How Azure places VMs in an availability set**

When you create multiple VMs in the same availability set, Azure spreads them across fault domains and update domains automatically. This is especially useful for:

- Two-tier applications with separate web and app VMs
- N-tier applications with multiple front-end or middle-tier instances
- Stateful workloads that still need some level of infrastructure resilience
- VM-based services behind a load balancer

A common topology looks like this:

```text
Internet
   |
Azure Load Balancer / Application Gateway
   |
Availability Set
   |-- VM1 (FD 0, UD 0)
   |-- VM2 (FD 1, UD 1)
   |-- VM3 (FD 2, UD 2)
```

The load balancer or Application Gateway sends traffic only to healthy instances. If one VM or one fault domain fails, the remaining instances continue serving requests.

**Availability sets with load balancers and Application Gateway**

Availability sets are often paired with a **load balancer** or **Application Gateway** because the availability set itself does not distribute traffic. It only improves placement and maintenance resilience.

Use this pattern when:

- You have multiple identical web servers.
- You want active-active traffic distribution.
- You need health probes and automatic failover between instances.

Practical example:

- Two IIS web servers in an availability set
- Azure Load Balancer in front
- Health probe checks TCP 80 or HTTP 80
- If one VM fails, traffic continues to the other VM

Example Azure CLI pattern for a Windows VM in an availability set:

```bash
az group create \
  --name rg-web-prod \
  --location eastus

az network vnet create \
  --resource-group rg-web-prod \
  --name vnet-web \
  --address-prefix 10.10.0.0/16 \
  --subnet-name subnet-web \
  --subnet-prefix 10.10.1.0/24

az vm availability-set create \
  --resource-group rg-web-prod \
  --name avset-web \
  --platform-fault-domain-count 2 \
  --platform-update-domain-count 5

az vm create \
  --resource-group rg-web-prod \
  --name vm-web-01 \
  --image Win2022Datacenter \
  --admin-username azureadmin \
  --admin-password 'Use-A-Strong-Password!123' \
  --vnet-name vnet-web \
  --subnet subnet-web \
  --availability-set avset-web
```

For Windows VMs, use an admin password or another Windows authentication approach. Do not rely on SSH key generation as the primary creation method.

**PowerShell example pattern**

In PowerShell, create the availability set first, then build a full VM configuration object. `New-AzVM` is not usually a one-line shortcut for a complete VM deployment.

```powershell
$rg = "rg-web-prod"
$loc = "eastus"
$avset = New-AzAvailabilitySet -ResourceGroupName $rg -Name "avset-web" -Location $loc `
  -PlatformFaultDomainCount 2 -PlatformUpdateDomainCount 5 -Sku Aligned

$cred = Get-Credential

$vmConfig = New-AzVMConfig -VMName "vm-web-01" -VMSize "Standard_D2s_v5" -AvailabilitySetId $avset.Id
$vmConfig = Set-AzVMOperatingSystem -VM $vmConfig -Windows -ComputerName "vm-web-01" -Credential $cred
$vmConfig = Add-AzVMNetworkInterface -VM $vmConfig -Id $nic.Id

New-AzVM -ResourceGroupName $rg -Location $loc -VM $vmConfig
```

This pattern matters because Azure PowerShell VM creation typically requires a prepared configuration object with OS and networking settings.

**Design constraints you should remember**

Availability sets have several constraints that often appear in exam questions:

- VMs must be in the **same region**.
- VMs must be in the **same resource group**.
- The availability set is a **single-region** construct.
- You cannot use an availability set to span regions for disaster recovery.
- Fault domain and update domain counts are generally chosen at creation time.
- You should plan the number of VMs so Azure can spread them meaningfully across domains.

If you need multi-region resilience, use:

- **Azure Site Recovery** for VM replication and failover
- **Azure Backup** for restore and retention
- A multi-region application design for active-active or active-passive patterns

**Fault domain count and update domain count**

The platform fault domain count is region-dependent and typically 2 or 3. Azure abstracts the physical layout, so you do not manually assign a VM to a specific fault domain.

Update domain count is a logical setting that controls how Azure sequences maintenance. More update domains can reduce the number of VMs affected by a single maintenance wave, but the exact value should match the application design and operational tolerance.

Practical guidance:

- Use at least two VMs for meaningful availability set protection.
- Spread critical tiers across multiple VMs.
- Keep application state external when possible, such as in Azure SQL Database, Azure Cache for Redis, or shared storage designed for the workload.
- Test failover behavior at the application layer, not just the infrastructure layer.

**Can you change fault domains or update domains later?**

In practice, you should treat fault domain count and update domain count as **creation-time design choices**. Azure portal, CLI, and PowerShell workflows are centered on defining these values when the availability set is created.

Operationally:

- Plan the counts before deployment.
- Recreate the availability set if you need a different domain design.
- Expect that moving existing VMs into a new availability set may require redeployment or rebuild, depending on the workload and migration path.

**Availability sets vs availability zones vs VM scale sets**

This comparison is a common AZ-104 decision point.

| Feature | Availability Sets | Availability Zones | VM Scale Sets |
|---|---|---|---|
| Scope | Single region | Multiple datacenter zones in a region | Single region, scalable VM fleet |
| Main goal | Protect against host/rack failure and maintenance | Protect against datacenter-level failure | Scale out/in and automate VM fleet management |
| Traffic distribution | Needs load balancer/Application Gateway | Needs load balancer/Application Gateway | Built-in scaling patterns, often with load balancing |
| Best for | Legacy or regional VM HA | Higher resilience for zone-aware apps | Elastic, repeatable VM deployments |
| Exam clue | “Same region, separate fault/update domains” | “Survive a zone outage” | “Autoscale many identical VMs” |

How to choose:

- Choose **availability sets** when the question describes VMs in one region that need basic high availability and no zone requirement.
- Choose **availability zones** when the question explicitly requires datacenter-level isolation within a region.
- Choose **VM scale sets** when the question emphasizes autoscaling, many identical instances, or fleet management.

> If a scenario asks for the highest resilience within a region and the workload supports it, availability zones are usually stronger than availability sets.

**Availability sets vs VM scale sets**

Availability sets and VM scale sets are related but not interchangeable.

- Availability sets focus on **placement resilience** for a small number of VMs.
- VM scale sets focus on **scaling and orchestration** for many similar VMs.
- VM scale sets can also use availability zones or spread across fault domains depending on configuration.

Exam-oriented distinction:

- Use availability sets for a fixed number of VMs with HA requirements.
- Use VM scale sets when you need autoscale, rolling upgrades, or large-scale stateless compute.

**Common deployment patterns**

Two-tier pattern:

- Web tier: 2 VMs in an availability set
- App tier: 2 VMs in a separate availability set
- Database tier: managed database service or a separate HA design

N-tier pattern:

- Front-end VMs behind Application Gateway
- Middle-tier VMs in another availability set
- Back-end services externalized where possible

This design helps you isolate failure domains by tier and avoid a single point of failure.

**Portal workflow**

A typical portal workflow is:

1. Create the availability set first.
2. Choose the region and resource group.
3. Select the **Aligned** SKU.
4. Set fault domain and update domain counts.
5. Create VMs and attach them to the availability set.
6. Place a load balancer or Application Gateway in front if the workload needs traffic distribution.

**CLI workflow**

A concise CLI workflow for a Linux or Windows VM follows the same structure:

```bash
az vm availability-set create \
  --resource-group rg-app \
  --name avset-app \
  --platform-fault-domain-count 2 \
  --platform-update-domain-count 5

az vm create \
  --resource-group rg-app \
  --name vm-app-01 \
  --image Ubuntu2204 \
  --admin-username azureadmin \
  --generate-ssh-keys \
  --availability-set avset-app
```

For Windows, replace SSH key generation with Windows authentication options such as `--admin-password`.

**Troubleshooting and validation**

To verify placement and configuration, inspect the availability set and VM properties.

Useful CLI commands:

```bash
az vm availability-set show \
  --resource-group rg-app \
  --name avset-app \
  --output table

az vm show \
  --resource-group rg-app \
  --name vm-app-01 \
  --query "availabilitySet.id" \
  --output tsv
```

Things to check:

- The VM is attached to the expected availability set.
- The availability set is in the correct region.
- The application has at least two instances if you expect HA.
- The load balancer health probe is passing.
- The app does not store critical state only on one VM.

**Exam traps and how to avoid them**

- Do not confuse **availability sets** with **availability zones**.
- Do not assume availability sets provide **cross-region DR**.
- Do not assume you can pick a specific datacenter.
- Do not use availability sets as a substitute for **backup**.
- Do not choose VM scale sets when the question only asks for a small fixed HA pair.
- Do not choose classic availability sets unless the scenario is clearly legacy.

**Quick decision guide**

- Need two or more VMs in one region with basic HA? Use an **availability set**.
- Need protection from a zone outage? Use **availability zones**.
- Need autoscaling identical VMs? Use **VM scale sets**.
- Need disaster recovery to another region? Use **Azure Site Recovery** and/or **Azure Backup**.

**Key takeaways**

Availability sets are a foundational Azure HA feature for VM-based workloads. They reduce risk by spreading VMs across fault domains and update domains within a single region. For AZ-104, focus on the design constraints, the modern **Aligned** model, the relationship to load balancers, and the distinction between availability sets, zones, and scale sets.
