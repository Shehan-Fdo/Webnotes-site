## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**High availability design for Azure compute**

High availability (HA) reduces downtime by removing single points of failure and spreading risk across fault domains, update domains, and availability zones. For Azure virtual machines (VMs) and VM scale sets (VMSS), you choose the right resilience pattern based on the service tier, regional support, and the recovery objective you need.

- **Availability set**: protects against localized hardware and platform maintenance failures within a datacenter.
- **Availability zone**: isolates workloads across physically separate datacenters in a region.
- **Zone-redundant design**: spreads instances across zones so a zone outage does not take down the service.
- **Zonal design**: pins a resource to one zone for lower latency or simpler architecture, but it has less resilience than zone-redundant deployment.

An availability set uses:
- **Fault domains** to separate VMs across different physical racks and shared power/network components.
- **Update domains** to control how Azure rolls platform maintenance so not all VMs restart at once.

A practical rule is:
- Use an **availability set** when you need intra-datacenter resilience and the region or workload does not require zone-level isolation.
- Use **availability zones** when the service supports them and you need stronger regional resilience.
- Use **VMSS** when you need horizontal scale and automated instance management.

> Availability zones are not supported by every Azure service or every region. Always verify service and region support before designing around zones.

**Availability sets and fault/update domain behavior**

Availability sets help you distribute VMs so a single maintenance event or hardware failure does not affect every instance. Azure places VMs in different fault domains and update domains automatically when possible, but you still need to design the application tier to tolerate instance loss.

- Put at least two VMs in the set for meaningful redundancy.
- Keep stateless tiers behind a load balancer or application gateway.
- Store state externally in managed services such as Azure SQL Database, Azure Files, or Cosmos DB when possible.
- Use the same availability set for VMs that form one tier of an application.

Example creation with Azure CLI:

```bash
az vm availability-set create \
  --resource-group rg-app \
  --name as-web \
  --platform-fault-domain-count 2 \
  --platform-update-domain-count 5
```

Example VM creation into an availability set:

```bash
az vm create \
  --resource-group rg-app \
  --name vm-web-01 \
  --image Ubuntu2204 \
  --admin-username azureuser \
  --generate-ssh-keys \
  --availability-set as-web
```

**Availability zones and deployment patterns**

Availability zones provide physically separate locations within a region. They are the preferred HA option when supported because they protect against datacenter-level failures.

Common patterns:
- **Zone-redundant load-balanced tier**: place multiple VMs or VMSS instances across zones and front them with a zone-aware load balancer or application gateway.
- **Active-active across zones**: run instances in multiple zones and keep traffic flowing to healthy zones.
- **Active-passive across zones**: keep a warm standby in another zone for simpler failover.

Important design points:
- A zonal VM lives in one zone only.
- A zone-redundant service can distribute traffic across zones.
- Some resources are zone-redundant by design, while others are zonal or do not support zones at all.
- Data replication and application session handling matter as much as compute placement.

Example VMSS zonal deployment:

```bash
az vmss create \
  --resource-group rg-app \
  --name vmss-web \
  --image Ubuntu2204 \
  --upgrade-policy-mode automatic \
  --orchestration-mode Uniform \
  --zones 1 2 3 \
  --admin-username azureuser \
  --generate-ssh-keys
```

**Load balancing for resilient compute**

A load balancer distributes traffic across healthy backend instances. It does not make the application itself stateless, so your app must still tolerate instance loss.

Use cases:
- **Azure Load Balancer** for TCP/UDP traffic at layer 4.
- **Application Gateway** for HTTP/HTTPS at layer 7 with TLS termination, path-based routing, and web application firewall features.
- **Health probes** to determine whether a backend instance should receive traffic.

A common exam trap is confusing the load-balancing rule protocol with the probe protocol:
- The backend traffic rule may use **TCP** for HTTP applications.
- The health probe can be **HTTP** or **HTTPS** to check application responsiveness.
- The application still serves HTTP/HTTPS; the load balancer is not interpreting the app payload at layer 4.

Example Azure Load Balancer rule and probe:

```bash
az network lb probe create \
  --resource-group rg-app \
  --lb-name lb-web \
  --name probe-http \
  --protocol http \
  --port 80 \
  --path /health

az network lb rule create \
  --resource-group rg-app \
  --lb-name lb-web \
  --name rule-web \
  --protocol tcp \
  --frontend-port 80 \
  --backend-port 80 \
  --frontend-ip-name fe-web \
  --backend-pool-name be-web \
  --probe-name probe-http
```

**VM Scale Sets and autoscaling**

VM Scale Sets provide a way to deploy and manage a group of identical VMs. They are the primary Azure compute service for horizontal scaling and are commonly paired with load balancers or application gateways.

VMSS benefits:
- Consistent VM configuration across instances.
- Easier scale-out and scale-in.
- Integration with Azure Monitor autoscale.
- Support for rolling upgrades and instance-level operations.

Autoscale in Azure is typically configured through **Azure Monitor autoscale**. Support varies by resource type, so do not assume every PaaS service uses the same autoscale model.

Scale triggers commonly include:
- **CPU percentage**
- **Memory usage** when guest metrics are available
- **Queue length** for message-driven workloads
- **Schedule-based scaling** for predictable business hours or batch windows
- **Predictive autoscale** for workloads with recurring patterns, where supported

Example autoscale profile for a VMSS:

```bash
az monitor autoscale create \
  --resource-group rg-app \
  --resource vmss-web \
  --resource-type Microsoft.Compute/virtualMachineScaleSets \
  --name autoscale-web \
  --min-count 2 \
  --max-count 10 \
  --count 2
```

Example scale rule based on CPU:

```bash
az monitor autoscale rule create \
  --resource-group rg-app \
  --autoscale-name autoscale-web \
  --condition "Percentage CPU > 70 avg 5m" \
  --scale out 1
```

Example schedule-based profile concept:

```json
{
  "name": "business-hours",
  "capacity": {
    "minimum": "2",
    "maximum": "8",
    "default": "4"
  },
  "rules": []
}
```

Operational guidance:
- Use **scale out** for demand spikes.
- Use **scale in** carefully to avoid removing instances that still hold active sessions.
- Keep session state external, such as in Redis or a database, so instances remain disposable.
- Set cooldown periods to prevent oscillation.

Predictive autoscale considerations:
- It works best for workloads with stable recurring patterns.
- It is not a substitute for capacity planning.
- Validate it with real traffic patterns before relying on it for critical workloads.

**VMSS instance protection and upgrade policies**

VMSS includes controls that affect availability during maintenance and scaling.

Key concepts:
- **Instance protection** lets you protect specific instances from scale-in or upgrades.
- **Upgrade policy** controls how model changes are applied.
- **Manual upgrades** require you to apply changes explicitly.
- **Automatic upgrades** apply changes automatically.
- **Rolling upgrades** update instances in batches to reduce impact.

Why this matters:
- Protecting a critical instance prevents autoscale from removing it.
- Rolling upgrades reduce blast radius during model changes.
- Automatic upgrades simplify operations but require confidence in the image and extension lifecycle.

Example of checking instance state:

```bash
az vmss list-instances \
  --resource-group rg-app \
  --name vmss-web \
  --output table
```

**Operational management of VM fleets**

Operational management covers the day-to-day tasks of keeping VMs healthy, compliant, and supportable. For Azure admins, this includes patching, configuration drift control, access recovery, monitoring, and extension management.

Common fleet tools:
- **Azure Automation** for runbooks, scheduled tasks, and patch orchestration.
- **Desired State Configuration (DSC)** for declarative configuration on Windows systems.
- **Azure Machine Configuration** and **guest configuration** for compliance and policy-driven settings.
- **Azure Policy** to enforce standards such as allowed images, tags, or extensions.

Use these tools when:
- You need repeatable configuration across many VMs.
- You want to detect drift from a baseline.
- You need scheduled operational tasks such as service restarts or log collection.

Example PowerShell DSC concept:

```powershell
Configuration WebServerConfig {
  Node "localhost" {
    WindowsFeature IIS {
      Name   = "Web-Server"
      Ensure = "Present"
    }
  }
}
```

Example Azure Automation runbook idea:
- Restart a service on a schedule.
- Collect logs from a VM extension failure.
- Rotate local admin credentials.

**Patch management and maintenance**

Patch management keeps guest operating systems and applications current. In Azure, patching is not only about installing updates; it is also about controlling when reboots happen and how maintenance is orchestrated.

Important concepts:
- **Maintenance configurations** let you control when Azure platform maintenance is applied to supported resources.
- **Update orchestration** can be manual, scheduled, or automated depending on the service.
- **Reboot control** matters because some updates require a restart to complete.
- **Maintenance windows** reduce business impact by aligning updates with low-traffic periods.

Practical guidance:
- Separate OS patching from application patching.
- Test patches in a nonproduction environment first.
- Use maintenance windows for production systems.
- Monitor failed patch deployments and extension health after updates.

Example Azure CLI to install updates on a VM is typically handled through guest OS tooling or automation rather than a single universal command, so the exact method depends on the OS and management approach. For Windows, you might use Update Management or native Windows Update controls; for Linux, use the distro package manager.

Linux example:

```bash
sudo apt update
sudo apt upgrade -y
sudo reboot
```

Windows example:

```powershell
Install-WindowsUpdate -AcceptAll -AutoReboot
```

> Patch orchestration differs by resource type and OS. Do not assume one command or one Azure feature covers every VM scenario.

**Backup, restore, and disaster recovery**

Backup and disaster recovery solve different problems:
- **Backup** protects against accidental deletion, corruption, and ransomware recovery scenarios.
- **Disaster recovery (DR)** helps you fail over to another region or site when the primary site is unavailable.

Azure Backup for VMs protects the VM by creating recovery points from which you can restore the VM or its disks. It is not the same as a platform-native redeploy, and it does not generally recreate every aspect of the original VM configuration in the way a full redeployment workflow would.

Use Azure Backup when:
- You need point-in-time recovery.
- You need retention policies for compliance.
- You want operational recovery from accidental changes or data loss.

Use **Azure Site Recovery (ASR)** when:
- You need cross-region failover for a workload.
- You want to orchestrate replication, failover, and failback.
- You need a DR plan for compute workloads, not just backup.

Key distinction:
- Backup restores data and VM disks from recovery points.
- Site Recovery replicates workloads to a secondary location and supports failover workflows.

Example Azure Backup concepts:
- Recovery vault stores backup data.
- Backup policy defines schedule and retention.
- Restore can target a new VM, existing disks, or files depending on the scenario.

Example ASR concepts:
- Source region replicates to a target region.
- Recovery plan coordinates multi-tier failover.
- Test failover validates DR without impacting production.

**VM restore points and recovery workflows**

VM restore points provide a point-in-time consistency mechanism for a VM and its attached disks. They are useful for operational recovery and deployment rollback scenarios, but they are not a simple snapshot clone and have specific limitations.

Use VM restore points when:
- You need a consistent rollback point before a change.
- You want to recover a VM configuration and disk state together within the feature’s limits.
- You are managing application deployment checkpoints.

Be careful:
- Restore points are not a substitute for full backup strategy.
- They are not the same as Azure Backup recovery points.
- They do not replace cross-region disaster recovery.

Common recovery actions for Azure VMs:
- **Redeploy** moves the VM to a new Azure host when the underlying host is unhealthy.
- **Reset password/SSH key** restores access when credentials are lost.
- **Screenshot and serial console** help diagnose boot or OS issues.
- **Reapply** can refresh the VM’s Azure configuration state in some scenarios.

Example CLI recovery actions:

```bash
az vm redeploy \
  --resource-group rg-app \
  --name vm-web-01
```

```bash
az vm user update \
  --resource-group rg-app \
  --name vm-web-01 \
  --username azureuser \
  --password 'StrongP@ssw0rd!'
```

```bash
az vm boot-diagnostics get-boot-log \
  --resource-group rg-app \
  --name vm-web-01
```

**Operational troubleshooting and monitoring**

Troubleshooting should start with the broadest signal and then narrow to the specific resource. For regional or platform-wide issues, **Azure Service Health** is often the best first check. For a specific resource, **Azure Resource Health** tells you whether Azure sees the resource as available, degraded, or unavailable.

Useful troubleshooting tools:
- **Activity Log** for control-plane operations such as create, update, delete, and policy actions.
- **Azure Monitor metrics** for CPU, disk, network, and platform signals.
- **Azure Monitor logs** for queryable telemetry and correlation.
- **Boot diagnostics** for VM startup issues.
- **Serial console** for low-level access when the OS is reachable enough to accept it.
- **Extension logs** for failures in agents such as Custom Script, DSC, or monitoring extensions.

Common VM extension failure causes:
- Bad script path or syntax.
- Missing permissions or blocked outbound connectivity.
- Unsupported OS version.
- Timeout during installation.
- Conflicting extension versions.

Remediation approach:
1. Check the extension status in the portal or CLI.
2. Review the extension logs on the VM.
3. Verify network access to required endpoints.
4. Remove and reinstall the extension if it is stuck.
5. Reboot the VM if the extension requires it.

Example CLI to inspect VM status:

```bash
az vm get-instance-view \
  --resource-group rg-app \
  --name vm-web-01 \
  --query instanceView.statuses
```

Example to view Activity Log entries:

```bash
az monitor activity-log list \
  --resource-group rg-app \
  --status Failed \
  --max-events 10
```

Example to query metrics:

```bash
az monitor metrics list \
  --resource /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm-web-01 \
  --metric Percentage CPU
```

**Common recovery and maintenance actions**

These actions are frequently tested because they map to real operational incidents.

- **Stop vs deallocate**:
  - Stopping a VM from inside the guest OS can leave the resource allocated depending on the workflow.
  - Deallocating releases compute capacity and usually stops compute billing for the VM.
  - Use deallocate when you want to stop compute charges and you do not need the VM running.

- **Restart**:
  - Use for transient OS or application issues.
  - It keeps the VM resource and usually preserves the disk state.

- **Redeploy**:
  - Use when the VM is stuck on a bad host or has platform-level issues.
  - Azure moves the VM to a new host.

- **Resize**:
  - Use when the VM needs more CPU, memory, or different capabilities.
  - Check whether the new size is available in the region and compatible with the current disk and networking setup.

- **Reset access**:
  - Use when you lose SSH or RDP access.
  - Prefer key-based SSH for Linux and strong credential management for Windows.

**Example operational checklist for a production VM tier**

1. Confirm service impact in **Azure Service Health**.
2. Check the specific VM in **Azure Resource Health**.
3. Review the **Activity Log** for recent changes.
4. Inspect **metrics** for CPU, memory, disk, and network saturation.
5. Validate load balancer health probes and backend membership.
6. Check VM extension status and logs.
7. Use serial console or boot diagnostics if the OS is not reachable.
8. Redeploy, resize, or restore only after identifying the failure domain.

**Practical design patterns**

A few patterns appear often in Azure compute deployments:

- **Single VM with backup**:
  - Lowest complexity.
  - Suitable for dev/test or low-criticality workloads.
  - Limited HA.

- **Two VMs in an availability set**:
  - Good for legacy apps that are not zone-aware.
  - Requires application-level redundancy.

- **Multi-zone VMSS behind a load balancer**:
  - Stronger resilience and easier scale-out.
  - Best for stateless web or API tiers.

- **Primary region plus ASR to secondary region**:
  - Best when regional outage recovery is required.
  - More complex and more expensive than backup alone.

- **VMSS with autoscale and rolling upgrades**:
  - Good for elastic workloads.
  - Requires careful health probe and deployment validation.

**Quick comparison of resilience tools**

| Feature | Primary purpose | Best for | Key limitation |
|---|---|---|---|
| Availability set | Protect against host and maintenance failures | Legacy or non-zone workloads | No datacenter-level isolation |
| Availability zone | Protect against datacenter failure | Critical regional workloads | Not supported everywhere |
| VMSS | Scale and manage identical VMs | Stateless tiers | App must tolerate instance churn |
| Azure Backup | Point-in-time recovery | Accidental deletion, corruption | Not DR by itself |
| Azure Site Recovery | Disaster recovery | Regional failover | Requires replication planning |

**Exam focus reminders**

- Know the difference between **backup**, **restore**, and **disaster recovery**.
- Distinguish **availability sets** from **availability zones**.
- Understand that **VMSS** is for horizontal scale and instance lifecycle management.
- Remember that **Azure Monitor autoscale** is the scaling mechanism, and support depends on the resource.
- Be able to explain **health probes**, **load balancing**, and **application health** separately.
- Know the operational tools: **Activity Log**, **Azure Monitor**, **Resource Health**, **Service Health**, **boot diagnostics**, and **serial console**.
- Recognize when to use **redeploy**, **reset access**, **resize**, or **restore**.
