## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Enterprise compute design goals**

A realistic Azure compute lab usually combines availability, security, identity, storage, and observability. For AZ-104, you need to choose the right compute service and then prove you can operate it after deployment.

- **Availability**: place workloads across availability zones or availability sets when the application needs resilience.
- **Scalability**: use VM Scale Sets (VMSS) or autoscale when demand changes.
- **Security**: prefer private access, managed identities, and least-privilege role assignments.
- **Operations**: monitor health, patching, backup, and lifecycle actions such as resize or redeploy.

**VM selection and sizing**

Choose a VM SKU based on workload characteristics, not just cost. Azure VM families differ in CPU, memory, storage throughput, and accelerator support.

- **General purpose**: balanced CPU and memory for domain controllers, small app servers, and utility hosts.
- **Compute optimized**: higher CPU-to-memory ratio for batch processing and web front ends.
- **Memory optimized**: larger memory footprint for in-memory caches, databases, and analytics.
- **Storage optimized**: high disk throughput for log-heavy or data-intensive workloads.
- **GPU-enabled**: specialized workloads such as graphics rendering or ML inference.
- **Burstable**: low-cost dev/test or intermittent workloads, but not ideal for sustained production load.

When you size a VM, check:
- vCPU count and memory
- Premium SSD or Ultra Disk support
- NIC limits and accelerated networking support
- Temporary disk size
- Region availability for the SKU
- Quotas in the subscription

A practical sizing workflow is:
1. Identify the workload profile and peak usage.
2. Select a family that matches CPU, memory, and storage needs.
3. Verify the SKU exists in the target region.
4. Confirm quota and feature support.
5. Start with a smaller size if you can resize later without downtime.

**VM creation choices that matter**

During deployment, several options affect resilience and operations.

- **Availability set**: use for classic intra-datacenter fault and update domain separation when zones are not required.
- **Availability zone**: use for higher resilience across physically separate datacenters in a region.
- **Public IP**: attach only when direct internet access is required; otherwise prefer Bastion, VPN, or private access.
- **OS disk type**: choose Standard HDD for low-cost labs, Standard SSD for general use, or Premium SSD for production performance.
- **Data disks**: separate application data from the OS disk for easier management and performance tuning.

> For zone-redundant designs, the load balancer or gateway must also support the zone design and SKU you choose. A zone-aware VM placement alone does not make the full solution zone resilient.

Example CLI for a VM with a private IP and no public exposure:
```bash
az vm create \
  --resource-group rg-compute-lab \
  --name vm-app01 \
  --image Ubuntu2204 \
  --size Standard_D2s_v5 \
  --admin-username azureadmin \
  --generate-ssh-keys \
  --vnet-name vnet-hub \
  --subnet app-subnet \
  --public-ip-address "" \
  --os-disk-size-gb 64 \
  --storage-sku Premium_LRS
```

Example with a public IP for a lab host:
```bash
az vm create \
  --resource-group rg-compute-lab \
  --name vm-admin01 \
  --image Win2022Datacenter \
  --size Standard_D2s_v5 \
  --admin-username azureadmin \
  --admin-password 'Use-A-Strong-Password-Here!' \
  --public-ip-sku Standard
```

**VM lifecycle operations**

You should know the difference between common lifecycle actions because they affect billing, availability, and troubleshooting.

- **Stop**: shuts down the guest OS, but the VM may still be allocated.
- **Deallocate**: releases compute resources and stops compute billing for the VM.
- **Restart**: reboots the guest and keeps the VM resource.
- **Redeploy**: moves the VM to a new Azure host; useful for host-level issues.
- **Resize**: changes the VM size if the target SKU is available and the VM can be moved.
- **Reimage**: rebuilds the OS disk from the original image and removes OS disk changes.
- **Delete**: removes the VM resource; disks and NICs may remain if you choose to keep them.

Common commands:
```bash
az vm stop -g rg-compute-lab -n vm-app01
az vm deallocate -g rg-compute-lab -n vm-app01
az vm restart -g rg-compute-lab -n vm-app01
az vm redeploy -g rg-compute-lab -n vm-app01
az vm resize -g rg-compute-lab -n vm-app01 --size Standard_D4s_v5
```

To inspect instance state and power status, use instance view:
```bash
az vm get-instance-view \
  -g rg-compute-lab \
  -n vm-app01 \
  --query "{name:name, powerState:instanceView.statuses[?starts_with(code, 'PowerState/')].displayStatus | [0], provisioning:provisioningState}" \
  -o json
```

**VM Scale Sets and autoscale**

VMSS lets you manage a group of identical VMs as one resource. It is the preferred pattern when you need horizontal scale and consistent configuration.

Key VMSS concepts:
- **Orchestration mode**: modern deployments use Flexible or Uniform depending on the scenario and feature needs.
- **Upgrade policy**: Manual, Automatic, or Rolling behavior depends on the VMSS configuration and platform support.
- **Instance management**: you can list, restart, reimage, or upgrade individual instances.
- **Autoscale**: scales the instance count based on metrics or schedules.

Use VMSS when:
- you need stateless scale-out web or API tiers
- you want consistent deployment of many identical instances
- you need autoscale based on CPU, queue length, or schedule
- you want easier rolling updates across instances

Example VMSS creation:
```bash
az vmss create \
  -g rg-compute-lab \
  -n vmss-web \
  --image Ubuntu2204 \
  --instance-count 2 \
  --admin-username azureadmin \
  --generate-ssh-keys \
  --upgrade-policy-mode Manual \
  --vnet-name vnet-hub \
  --subnet app-subnet
```

Example autoscale profile using a target resource ID:
```bash
VMSS_ID=$(az vmss show -g rg-compute-lab -n vmss-web --query id -o tsv)

az monitor autoscale create \
  --resource-group rg-compute-lab \
  --resource $VMSS_ID \
  --resource-type Microsoft.Compute/virtualMachineScaleSets \
  --name vmss-web-autoscale \
  --min-count 2 \
  --max-count 6 \
  --count 2
```

Add a CPU-based scale rule:
```bash
az monitor autoscale rule create \
  --resource-group rg-compute-lab \
  --autoscale-name vmss-web-autoscale \
  --condition "Percentage CPU > 70 avg 5m" \
  --scale out 1
```

Scale set instance operations:
```bash
az vmss list-instances -g rg-compute-lab -n vmss-web -o table
az vmss restart-instances -g rg-compute-lab -n vmss-web --instance-ids 0 1
az vmss reimage -g rg-compute-lab -n vmss-web --instance-ids 0
```

**Disks and storage design**

Azure VM storage design affects performance, cost, and recoverability. Separate OS and data responsibilities whenever possible.

- **OS disk**: hosts the operating system.
- **Data disk**: stores application data, logs, or databases.
- **Temporary disk**: local ephemeral storage that can be lost on host move or redeploy.
- **Managed disk**: Azure-managed storage abstraction for easier lifecycle control.

Disk types:
- **Standard HDD**: lowest cost, suitable for noncritical workloads.
- **Standard SSD**: balanced cost and performance.
- **Premium SSD**: higher IOPS and lower latency for production workloads.
- **Ultra Disk**: very high performance with tunable IOPS and throughput, for specialized workloads.

Important disk features:
- **Snapshots**: point-in-time copies of managed disks for backup-like workflows or cloning.
- **Shared disks**: allow multiple VMs to attach the same disk for clustered applications that support shared storage.
- **Ephemeral OS disks**: place the OS on local temporary storage for fast provisioning and stateless workloads; data on the OS disk is not durable across host events.

Example snapshot creation:
```bash
az snapshot create \
  -g rg-compute-lab \
  -n snap-app01-os \
  --source /subscriptions/<subId>/resourceGroups/rg-compute-lab/providers/Microsoft.Compute/disks/vm-app01_OsDisk_1
```

Example data disk attachment:
```bash
az vm disk attach \
  -g rg-compute-lab \
  --vm-name vm-app01 \
  --name data01 \
  --new \
  --size-gb 128 \
  --sku Premium_LRS
```

**Identity and access for VMs**

Managed identities let a VM authenticate to Azure services without storing secrets in code or configuration.

- **System-assigned identity**: tied to one VM and deleted with it.
- **User-assigned identity**: standalone identity that you can attach to one or more resources.

Choose a system-assigned identity when:
- the VM is the only consumer
- you want simpler lifecycle management
- you do not need identity reuse

Choose a user-assigned identity when:
- multiple VMs need the same identity
- you want identity continuity across VM replacement
- you need to pre-provision access before the VM exists

Enable a system-assigned identity:
```bash
az vm identity assign -g rg-compute-lab -n vm-app01
```

Create and attach a user-assigned identity:
```bash
az identity create -g rg-compute-lab -n id-app01
IDENTITY_ID=$(az identity show -g rg-compute-lab -n id-app01 --query id -o tsv)

az vm identity assign \
  -g rg-compute-lab \
  -n vm-app01 \
  --identities $IDENTITY_ID
```

Grant access to Key Vault depends on the permission model:
- If the vault uses **Azure RBAC**, assign roles such as `Key Vault Secrets User`.
- If the vault uses **access policies**, configure an access policy instead of relying on RBAC roles.

Example RBAC assignment:
```bash
az role assignment create \
  --assignee-object-id <principalObjectId> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope $(az keyvault show -g rg-compute-lab -n kv-lab --query id -o tsv)
```

Example access policy:
```bash
az keyvault set-policy \
  -g rg-compute-lab \
  -n kv-lab \
  --object-id <principalObjectId> \
  --secret-permissions get list
```

Token retrieval from inside the VM uses the Instance Metadata Service (IMDS). The request must come from the VM and include the metadata header:
```bash
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2021-02-01&resource=https%3A%2F%2Fvault.azure.net"
```

**Key Vault and secret validation**

A common enterprise pattern is to store application secrets in Key Vault and let the VM read them using managed identity.

Validation steps:
1. Confirm the VM has a managed identity.
2. Confirm the identity has the correct Key Vault permission model access.
3. Confirm network access to the vault, including firewall or private endpoint rules.
4. Test secret retrieval from the VM.

Example secret read with Azure CLI after login through managed identity:
```bash
az login --identity
az keyvault secret show --vault-name kv-lab --name app-secret --query value -o tsv
```

If the vault uses private access, ensure the VM can resolve and reach the private endpoint. DNS is often the failure point in lab scenarios.

**Monitoring and diagnostics**

Azure Monitor is the core platform for metrics, logs, alerts, and action routing. For VM operations, combine platform metrics with guest-level telemetry.

Key components:
- **Azure Monitor Agent (AMA)**: collects guest OS logs and performance data.
- **Log Analytics workspace**: stores and queries collected logs.
- **Data Collection Rule (DCR)**: defines what data AMA collects and where it goes.
- **Alert rule**: triggers on metric or log conditions.
- **Action group**: sends notifications or runs automation.

Typical monitoring flow:
1. Install or enable AMA on the VM.
2. Associate a DCR with the VM or VMSS.
3. Send data to a Log Analytics workspace.
4. Create metric and log alerts.
5. Attach an action group for email, SMS, webhook, or automation.

Example CLI to create a workspace:
```bash
az monitor log-analytics workspace create \
  -g rg-compute-lab \
  -n law-compute-lab
```

Example VM extension install for AMA on Windows:
```bash
az vm extension set \
  -g rg-compute-lab \
  --vm-name vm-app01 \
  --publisher Microsoft.Azure.Monitor \
  --name AzureMonitorWindowsAgent
```

Example metric alert for high CPU:
```bash
az monitor metrics alert create \
  -g rg-compute-lab \
  -n alert-vm-cpu \
  --scopes $(az vm show -g rg-compute-lab -n vm-app01 --query id -o tsv) \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m
```

Example action group:
```bash
az monitor action-group create \
  -g rg-compute-lab \
  -n ag-ops \
  --short-name ops \
  --action email ops-team ops@example.com
```

**Backup and restore**

For Azure VMs, Azure Backup provides recovery points and restore workflows. Use the correct terminology: VM backup is not the same as continuous database-style point-in-time recovery.

Core objects:
- **Recovery Services vault**: stores backup configuration and recovery points.
- **Backup policy**: defines schedule and retention.
- **Recovery point**: a restore point created by the backup service.
- **Restore**: creates a new VM, restores disks, or restores files depending on the workflow.

Typical setup:
1. Create a Recovery Services vault.
2. Configure a backup policy.
3. Enable backup for the VM.
4. Verify recovery points are created.
5. Test restore to a new VM or disk.

Example vault creation:
```bash
az backup vault create \
  -g rg-compute-lab \
  -n rsv-compute-lab \
  -l eastus
```

Example backup policy creation:
```bash
az backup policy create \
  -g rg-compute-lab \
  -v rsv-compute-lab \
  -n daily-policy \
  --policy "@policy.json"
```

A simple policy file can define daily backup and retention settings. Exact JSON structure varies by workload and API version, so validate against the current Azure CLI help before use.

Restore options you should know:
- **Create new VM**: fastest for full VM recovery in many labs.
- **Restore disks**: useful when you want to attach recovered disks to an existing VM.
- **File recovery**: mount recovery data to extract individual files.

**Extensions and configuration management**

VM extensions automate post-deployment configuration. They are common in AZ-104 scenarios for bootstrapping software, monitoring, and security.

Examples:
- **Custom Script Extension**: downloads and runs scripts after deployment.
- **VMAccess**: resets credentials or SSH configuration in some scenarios.
- **Monitoring extensions**: install agents for telemetry collection.

Be careful with extension names and handlers because they differ by OS and generation. For Linux, the legacy handler name may differ from newer extension packaging.

Example Linux Custom Script Extension:
```bash
az vm extension set \
  -g rg-compute-lab \
  --vm-name vm-app01 \
  --publisher Microsoft.Azure.Extensions \
  --name CustomScript \
  --settings '{"fileUris":["https://storageaccount.blob.core.windows.net/scripts/setup.sh"],"commandToExecute":"bash setup.sh"}'
```

Example Windows Custom Script Extension:
```powershell
Set-AzVMCustomScriptExtension `
  -ResourceGroupName rg-compute-lab `
  -VMName vm-app01 `
  -Name "CustomScript" `
  -FileUri "https://storageaccount.blob.core.windows.net/scripts/setup.ps1" `
  -Run "setup.ps1"
```

**Connectivity and secure access**

For enterprise labs, avoid exposing management ports directly to the internet unless the scenario requires it.

Preferred access patterns:
- **Azure Bastion**: browser-based RDP/SSH without public IPs on VMs.
- **VPN or ExpressRoute**: private access from on-premises or a connected network.
- **Jump host**: only when Bastion is unavailable in the scenario.

If you use Bastion:
- deploy it in the virtual network
- use the required subnet name and size
- ensure the VM has a private IP in a reachable subnet
- keep NSG rules aligned with the access path

**Load balancing and application delivery**

Use the right front-end service for the traffic pattern.

- **Azure Load Balancer**: Layer 4 distribution for TCP/UDP workloads.
- **Application Gateway**: Layer 7 HTTP/HTTPS routing, TLS termination, and web application firewall options.
- **Front Door**: global HTTP(S) entry point for multi-region apps.

For zone-aware deployments, ensure the chosen load balancer or gateway SKU and configuration support the zone design. A mismatch can break the resilience story even if the VMs are spread across zones.

**Troubleshooting workflow**

When a VM or VMSS problem appears, isolate the layer first.

1. Check Azure resource state and provisioning status.
2. Check instance view for power state and extension health.
3. Verify network reachability and NSG rules.
4. Confirm identity and authorization.
5. Review guest logs and Azure Monitor data.
6. Redeploy or reimage only after you identify whether the issue is host-level or guest-level.

Useful commands:
```bash
az vm show -g rg-compute-lab -n vm-app01 -o jsonc
az vm get-instance-view -g rg-compute-lab -n vm-app01 -o jsonc
az vm extension list -g rg-compute-lab --vm-name vm-app01 -o table
az network nic show-effective-nsg -g rg-compute-lab -n nic-app01
```

For Windows guest checks, prefer Azure portal diagnostics, extension status, and VM Agent health over relying only on legacy service names. For Linux, confirm the agent and extension logs if the scenario includes them.

**Practical lab topology**

A common AZ-104 enterprise lab pattern looks like this:

- **Hub VNet**
  - Bastion subnet
  - shared services subnet
- **Spoke VNet**
  - app subnet with VM or VMSS
  - data subnet with private endpoints
- **Security**
  - NSGs on subnets
  - managed identity for app access to Key Vault
- **Operations**
  - Log Analytics workspace
  - Azure Monitor alerts
  - Recovery Services vault
- **Scale**
  - Load Balancer or Application Gateway in front of app tier
  - VMSS for stateless web nodes

This design lets you test:
- secure admin access
- private secret retrieval
- autoscale behavior
- backup and restore
- monitoring and alerting
- lifecycle operations such as resize, redeploy, and reimage

**Exam focus reminders**

- Know when to use a VM, VMSS, or PaaS service.
- Know the difference between stop and deallocate.
- Know how managed identities differ from service principals.
- Know the Key Vault permission model before assigning roles.
- Know the difference between disks, snapshots, and backup recovery points.
- Know how to validate VM health with instance view and Azure Monitor.
- Know that zone resilience requires the full path, not just the VM placement.
