## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure VM provisioning overview**

Azure Virtual Machines (VMs) are IaaS (Infrastructure as a Service) compute resources that you control at the operating system level. You choose the image, size, storage, networking, and access model, then manage the guest OS like a traditional server.

For AZ-104, focus on how you provision VMs, secure access, attach storage, monitor health, and perform lifecycle operations. The exam often tests whether you know which settings belong to the VM, the NIC, the disk, or the network.

- **Compute**: VM size determines CPU, memory, and supported features.
- **Storage**: OS disks and data disks have different performance and lifecycle implications.
- **Networking**: The NIC controls IP configuration, acceleration, and routing behavior.
- **Access**: Use SSH keys for Linux and passwords or domain-based access for Windows.
- **Operations**: Start, stop, deallocate, resize, and redeploy affect state and billing differently.

**Provisioning methods**

You can create VMs through the Azure portal, Azure CLI, PowerShell, ARM templates, or Bicep. For repeatable deployments, use templates or Bicep. For quick administration and exam scenarios, CLI is especially useful because it exposes the underlying resource settings clearly.

Common provisioning inputs include:

- **Subscription and resource group**
- **Region**
- **Image**
- **Size**
- **Authentication method**
- **Virtual network and subnet**
- **Public IP or Bastion access**
- **OS disk and data disk settings**
- **Availability options**

A practical CLI example for Linux uses SSH key authentication and a standard public IP:

```bash
az vm create \
  --resource-group rg-prod \
  --name vm-linux-01 \
  --image Ubuntu2204 \
  --size Standard_B2s \
  --admin-username azureadmin \
  --ssh-key-values ~/.ssh/id_rsa.pub \
  --vnet-name vnet-prod \
  --subnet subnet-app \
  --public-ip-sku Standard
```

A Windows VM example uses an admin password and a Standard public IP:

```bash
az vm create \
  --resource-group rg-prod \
  --name vm-win-01 \
  --image Win2022Datacenter \
  --size Standard_D2s_v5 \
  --admin-username azureadmin \
  --admin-password 'Use-A-Strong-Password!123' \
  --vnet-name vnet-prod \
  --subnet subnet-app \
  --public-ip-sku Standard
```

> Standard public IP is the current recommended choice for most scenarios. Basic public IP is legacy and being retired in many environments.

**Images and image sources**

The image defines the base operating system and sometimes preinstalled software. Azure supports marketplace images, custom images, and images from Azure Compute Gallery.

Use the right image source based on repeatability and governance:

- **Marketplace image**: Fastest way to deploy a common OS or application image.
- **Custom image**: Useful when you need a standardized internal build.
- **Azure Compute Gallery**: Best for versioning, replication, and controlled sharing across subscriptions and regions.

Azure Compute Gallery supports both generalized and specialized image scenarios depending on the use case. Generalized images are the standard choice for reusable templates and scale-out deployments because they remove machine-specific state. Specialized images preserve the exact machine state and are useful when you need an exact clone.

Typical workflow:

1. Prepare the source VM.
2. Capture or publish the image to Azure Compute Gallery.
3. Replicate the image to target regions if needed.
4. Deploy new VMs from the image version.

**VM size selection and resizing**

VM size affects CPU, memory, network throughput, disk throughput, and feature support. Choose a size that matches the workload rather than overprovisioning by default.

Common size families include:

- **B-series**: Burstable workloads with variable CPU usage.
- **D-series**: General-purpose workloads.
- **E-series**: Memory-optimized workloads.
- **F-series**: Compute-optimized workloads.
- **L-series**: Storage-optimized workloads.

When resizing, remember these exam-relevant constraints:

- You often must **deallocate** the VM before changing to a different size family or when the target size is not available on the current host.
- The target size must be available in the VM’s region and sometimes in the specific availability zone.
- Some sizes do not support the same features, such as accelerated networking or premium storage.
- Resizing can fail if the new size is not supported by the current hardware cluster.

CLI example:

```bash
az vm deallocate --resource-group rg-prod --name vm-linux-01
az vm resize --resource-group rg-prod --name vm-linux-01 --size Standard_D4s_v5
az vm start --resource-group rg-prod --name vm-linux-01
```

**Lifecycle operations and billing impact**

VM state matters for both administration and cost.

- **Start**: Powers on the VM and resumes compute billing.
- **Stop**: Stops the guest OS, but if the VM remains allocated, compute resources may still be reserved.
- **Deallocate**: Releases the underlying compute host and stops compute billing.
- **Restart**: Reboots the guest OS and may briefly interrupt service.
- **Redeploy**: Moves the VM to a new Azure host; useful for some host-level issues.
- **Delete**: Removes the VM resource; disks and NICs may remain if you choose to preserve them.

Use deallocation when you want to stop compute charges for a VM that is not in use. Use restart for routine maintenance. Use redeploy when you suspect host-level problems and need Azure to place the VM on new infrastructure.

Examples:

```bash
az vm stop --resource-group rg-prod --name vm-linux-01
az vm deallocate --resource-group rg-prod --name vm-linux-01
az vm start --resource-group rg-prod --name vm-linux-01
az vm restart --resource-group rg-prod --name vm-linux-01
az vm redeploy --resource-group rg-prod --name vm-linux-01
az vm delete --resource-group rg-prod --name vm-linux-01 --yes
```

**Networking configuration**

The VM’s NIC is a separate resource and is central to connectivity, routing, and performance. In AZ-104, be clear about what belongs to the NIC versus the VM.

Important NIC settings include:

- **Private IP**: Static or dynamic assignment within the subnet.
- **Public IP**: Optional internet-facing address.
- **Accelerated networking**: Reduces latency and improves throughput on supported VM sizes and images.
- **IP forwarding**: Required when the VM acts as a router, firewall, or network virtual appliance.
- **Multiple NICs**: Supported on many VM sizes for advanced routing or segmentation.
- **DNS settings**: Can be inherited from the VNet or customized.

Accelerated networking is especially relevant for high-throughput or latency-sensitive workloads. IP forwarding is required if the VM forwards traffic between interfaces or subnets.

Example NIC-related CLI patterns:

```bash
az network nic create \
  --resource-group rg-prod \
  --name nic-app-01 \
  --vnet-name vnet-prod \
  --subnet subnet-app \
  --accelerated-networking true
```

```bash
az network nic update \
  --resource-group rg-prod \
  --name nic-app-01 \
  --ip-forwarding true
```

If you need multiple NICs, ensure the VM size supports them and that your routing design is intentional. Multiple NICs are common for network appliances, not for ordinary application servers.

**Access methods and secure administration**

For Linux VMs, SSH (Secure Shell) key authentication is the preferred method. For Windows VMs, use a strong local administrator password or integrate with domain-based access patterns.

Recommended access options:

- **SSH keys** for Linux
- **RDP (Remote Desktop Protocol)** for Windows
- **Azure Bastion** for browser-based private access
- **Just-in-time access** through Microsoft Defender for Cloud when available
- **Domain join** for enterprise Windows administration
- **Azure AD login** for supported Windows and Linux scenarios

Azure Bastion lets you connect to a VM over the Azure portal without exposing RDP or SSH to the public internet. The VM must be reachable over its private IP in the connected virtual network or peered network, and the Bastion SKU and peering configuration must support the scenario.

Example Bastion deployment pattern:

- Place Bastion in a dedicated subnet named `AzureBastionSubnet`.
- Keep the VM on a private subnet without a public IP.
- Connect through the portal using Bastion instead of opening inbound management ports.

Windows domain join example with CLI extension:

```bash
az vm extension set \
  --resource-group rg-prod \
  --vm-name vm-win-01 \
  --name JsonADDomainExtension \
  --publisher Microsoft.Compute \
  --settings '{
    "Name": "contoso.com",
    "OUPath": "OU=Servers,DC=contoso,DC=com",
    "User": "contoso\\domainjoin",
    "Restart": "true",
    "Options": "3"
  }' \
  --protected-settings '{
    "Password": "Use-A-Strong-Password!123"
  }'
```

For Azure AD login on supported Windows VMs, you typically enable the appropriate extension and role assignments, then sign in with an Azure AD identity. This is useful when you want centralized identity and reduced local admin sprawl.

**Disks, storage, and performance**

Azure VMs use an OS disk and can use one or more data disks. Disk choice affects performance, cost, and durability.

Disk types commonly tested in AZ-104:

- **Standard HDD**: Lowest cost, lowest performance.
- **Standard SSD**: Balanced cost and performance.
- **Premium SSD**: Higher performance for production workloads.
- **Ultra Disk**: Very high performance and tunable IOPS/throughput for specialized workloads.

Key disk concepts:

- **OS disk**: Holds the operating system.
- **Data disk**: Stores application or user data.
- **Temporary disk**: Local ephemeral storage; do not store critical data here.
- **Caching**: Improves read/write performance depending on disk role and workload.

> Do not rely on the temporary disk for durable data. It can be lost during host moves, deallocation, or reimage operations.

Example of attaching a new managed data disk with an explicit name:

```bash
az vm disk attach \
  --resource-group rg-prod \
  --vm-name vm-linux-01 \
  --name vm-linux-01-datadisk01 \
  --new \
  --size-gb 128 \
  --sku Premium_LRS
```

After attachment, you must initialize and format the disk inside the guest OS.

Linux example:

```bash
lsblk
sudo fdisk /dev/sdc
sudo mkfs.ext4 /dev/sdc1
sudo mkdir -p /data
sudo mount /dev/sdc1 /data
echo '/dev/sdc1 /data ext4 defaults,nofail 0 2' | sudo tee -a /etc/fstab
```

Windows example:

```powershell
Get-Disk
Initialize-Disk -Number 2 -PartitionStyle GPT
New-Partition -DiskNumber 2 -UseMaximumSize -AssignDriveLetter
Format-Volume -DriveLetter F -FileSystem NTFS -NewFileSystemLabel "Data01" -Confirm:$false
```

For Linux, verify the device name with `lsblk` before formatting. Device names can change, especially after reboots or when multiple disks are attached.

**Disk encryption and key management**

Azure provides multiple encryption layers for VM disks. Know the difference between platform-managed encryption and customer-managed encryption.

- **Server-side encryption (SSE)**: Enabled by default for managed disks and protects data at rest.
- **Customer-managed keys (CMK)**: You control the encryption key in Azure Key Vault or Managed HSM.
- **Azure Disk Encryption (ADE)**: Guest-level encryption using BitLocker for Windows or DM-Crypt for Linux; used in some legacy or specialized compliance scenarios.

Use CMK when you need stronger control over key lifecycle and compliance requirements. Use ADE only when the scenario specifically requires guest-level encryption, because it adds operational complexity.

Practical considerations:

- Ensure the identity used by the VM or deployment has access to the key vault.
- Plan key rotation and recovery procedures.
- Confirm that backup and restore workflows are compatible with the chosen encryption model.

**Availability and resiliency design**

Azure offers several availability patterns, and the right choice depends on the failure domain you want to protect against.

- **Availability set**: Spreads VMs across fault domains and update domains within a single datacenter cluster.
- **Availability zone**: Places VMs in physically separate datacenters within a region.
- **Zone-redundant design**: Uses multiple zones for higher resiliency.
- **Load balancing**: Distributes traffic across instances in one or more zones.

Availability sets protect against host maintenance and some hardware failures, but they do not protect against a full datacenter outage. Availability zones provide stronger protection because they span separate datacenters within the region.

Design guidance:

- Use **availability sets** for legacy or intra-datacenter resiliency needs.
- Use **availability zones** for higher regional resiliency when the region supports them.
- Use **zone-redundant load balancing** when you deploy multiple VM instances across zones.
- Use **scale sets** when you need automated instance management and horizontal scaling.

**Monitoring, diagnostics, and alerting**

VM monitoring should include both Azure-side resource metrics and guest-level telemetry. Azure Monitor is the central service for metrics, logs, and alerts.

Common monitoring components:

- **Boot diagnostics**: Captures console output and screenshots for startup troubleshooting.
- **Azure Monitor metrics**: Tracks CPU, network, disk, and platform metrics.
- **Log Analytics**: Stores and queries guest and platform logs.
- **VM insights**: Provides performance and dependency visibility when configured.
- **Alerts**: Notify you when thresholds or conditions are met.

Boot diagnostics is useful when a VM fails to boot or the OS is inaccessible. For managed boot diagnostics, enable the feature explicitly in the VM configuration or deployment template.

Example CLI pattern:

```bash
az vm create \
  --resource-group rg-prod \
  --name vm-linux-02 \
  --image Ubuntu2204 \
  --size Standard_B2s \
  --admin-username azureadmin \
  --ssh-key-values ~/.ssh/id_rsa.pub \
  --boot-diagnostics true
```

For deeper monitoring, install the Azure Monitor Agent and connect the VM to a Data Collection Rule (DCR). This is the modern path for guest telemetry and log collection.

Example extension installation:

```bash
az vm extension set \
  --resource-group rg-prod \
  --vm-name vm-linux-01 \
  --name AzureMonitorLinuxAgent \
  --publisher Microsoft.Azure.Monitor
```

Use alerts for:

- CPU saturation
- Disk queue or latency issues
- Memory pressure from guest metrics
- VM availability or heartbeat loss
- Log-based conditions from application or system logs

**Backup and recovery**

Azure Backup protects VMs from accidental deletion, corruption, and some disaster scenarios. It is not a replacement for high availability, but it is essential for recovery.

Backup planning considerations:

- **Recovery Services vault** stores backup data and policies.
- **Backup policy** defines schedule and retention.
- **Application-consistent backups** are preferable for transactional workloads.
- **File-level restore** can help recover individual files without restoring the whole VM.
- **Restore options** may include creating a new VM or restoring disks.

Use backup for recovery from user error, malware, or application corruption. Use availability features for uptime. These are complementary, not interchangeable.

**Patching and update management**

VM patching is part of ongoing administration. Azure Update Manager helps you assess and apply OS updates across Azure VMs and hybrid machines.

Key patching concepts:

- **Assessment**: Determine missing updates.
- **Maintenance window**: Control when updates are applied.
- **Reboot behavior**: Plan for reboots when patches require them.
- **Scope**: Apply updates to one VM, a group, or a fleet.

For Windows and Linux, you can also configure native OS update settings inside the guest, but Azure Update Manager gives you centralized control and reporting.

Operational guidance:

- Schedule patching during low-traffic windows.
- Coordinate with application owners for reboot-sensitive workloads.
- Combine patching with monitoring and backup validation.

**VM extensions and post-deployment configuration**

VM extensions let you automate guest configuration after provisioning. They are important for configuration management, monitoring, and application deployment.

Common extensions include:

- **Custom Script Extension**: Downloads and runs scripts during or after deployment.
- **DSC (Desired State Configuration)**: Enforces Windows configuration state.
- **Azure Monitor Agent**: Sends telemetry to Azure Monitor.
- **Dependency agent**: Supports VM insights and dependency mapping.
- **Domain join extensions**: Join Windows VMs to Active Directory.

Custom Script Extension example:

```bash
az vm extension set \
  --resource-group rg-prod \
  --vm-name vm-linux-01 \
  --name customScript \
  --publisher Microsoft.Azure.Extensions \
  --settings '{
    "fileUris": ["https://storageaccount.blob.core.windows.net/scripts/install-nginx.sh"],
    "commandToExecute": "bash install-nginx.sh"
  }'
```

Use extensions when you need repeatable post-provisioning actions. Avoid putting too much configuration logic directly into manual provisioning steps.

**Practical provisioning patterns**

A secure Linux VM pattern often looks like this:

1. Create the VNet and subnet.
2. Create the VM without a public IP.
3. Enable Bastion for browser-based access.
4. Attach data disks if needed.
5. Install monitoring and backup agents.
6. Apply NSGs (Network Security Groups) and least-privilege rules.

A secure Windows VM pattern often looks like this:

1. Create the VM in a private subnet.
2. Use Bastion or a jump host for administration.
3. Join the VM to the domain if required.
4. Enable monitoring and backup.
5. Configure patching and alerting.
6. Restrict RDP exposure and use role-based access control.

**Common exam pitfalls**

- **Authentication flags**: For `az vm create`, Linux SSH access is typically provided by `--ssh-key-values`; do not assume an `--authentication-type ssh` flag is required or valid.
- **Windows passwords**: Use `--admin-password` for Windows provisioning rather than a generic authentication flag.
- **Disk formatting**: Always verify the disk device name after attachment before formatting.
- **Public IPs**: Prefer Standard public IPs; Basic is legacy in many scenarios.
- **Availability sets**: They do not protect against region-wide or datacenter-wide outages.
- **Boot diagnostics**: Enable it explicitly when you need startup troubleshooting.
- **Resizing**: Deallocate first when required, and confirm the target size exists in the region or zone.
- **Temporary disk**: Never store durable data there.

**Quick reference commands**

```bash
# Show VM details
az vm show --resource-group rg-prod --name vm-linux-01 -o table

# List available sizes in a region
az vm list-sizes --location eastus -o table

# Stop and deallocate
az vm deallocate --resource-group rg-prod --name vm-linux-01

# Start the VM
az vm start --resource-group rg-prod --name vm-linux-01

# Restart the VM
az vm restart --resource-group rg-prod --name vm-linux-01

# Resize the VM
az vm resize --resource-group rg-prod --name vm-linux-01 --size Standard_D4s_v5

# Attach a data disk
az vm disk attach --resource-group rg-prod --vm-name vm-linux-01 --name vm-linux-01-datadisk01 --new --size-gb 128 --sku Premium_LRS
```

**What to remember for AZ-104**

- Choose the right VM size, image, and region before deployment.
- Use SSH keys for Linux and strong password or directory-based access for Windows.
- Understand the difference between start, stop, and deallocate.
- Know how NIC settings affect routing, acceleration, and forwarding.
- Use managed disks, and format data disks inside the guest OS after attachment.
- Apply monitoring, backup, and patching as part of the VM lifecycle.
- Prefer availability zones for stronger resiliency when the workload requires it.
