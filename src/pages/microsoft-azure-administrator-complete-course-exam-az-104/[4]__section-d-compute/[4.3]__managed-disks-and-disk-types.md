## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Managed disks overview**

Azure managed disks are the default storage choice for Azure virtual machines (VMs). Azure handles the storage account lifecycle, placement, and availability concerns for you, so you manage the disk as a first-class resource instead of managing page blobs in a storage account.

A managed disk is useful when you want:
- Simplified VM storage management.
- Better isolation than unmanaged disks.
- Easier scaling, snapshots, and backup integration.
- Clear performance tiers tied to the disk SKU.

For AZ-104, focus on how disk type, performance tier, caching, encryption, and lifecycle behavior affect VM design and operations.

**Core managed disk concepts**

A managed disk is an Azure resource that represents block storage attached to a VM or used independently for snapshots, images, or shared scenarios.

Key properties you should recognize:
- **Disk SKU** determines performance and redundancy characteristics.
- **Size** affects capacity and often performance limits.
- **Caching** affects read/write behavior for attached VM disks.
- **Encryption** protects data at rest.
- **Lifecycle** determines whether the disk is deleted or retained when the VM is removed.

Managed disks are region-scoped resources. You typically place the disk in the same region as the VM for best performance and to avoid unsupported attachment scenarios.

**Disk creation options**

You can create managed disks in several ways, and exam questions often test which source is appropriate.

Common creation sources:
- **Empty disk** for new data volumes.
- **Snapshot** to preserve a point-in-time copy of an existing disk.
- **Image** to create a VM from a generalized OS image.
- **Restore** from backup or recovery workflows, depending on the service used.

Typical use cases:
- Use an empty disk when you need a new data volume for an application.
- Use a snapshot when you need fast recovery or cloning.
- Use an image when you need to deploy a new VM with a known OS baseline.
- Use a restored disk when recovering from a failure or rollback event.

> A VM disk created from a snapshot or image is not the same as a live clone of the original workload state unless the source was captured consistently.

**Managed disk SKUs and when to use them**

Azure expresses managed disk performance and redundancy through the disk SKU. Not every SKU supports every redundancy option, so read the SKU name carefully.

| SKU | Typical use | Notes |
|---|---|---|
| `Standard_LRS` | Dev/test, low-cost workloads | Locally redundant within a region; good for noncritical data |
| `StandardSSD_LRS` | General-purpose production with moderate performance | Better latency than Standard HDD |
| `Premium_LRS` | Production workloads needing low latency | Backed by SSD; common for OS and data disks |
| `StandardSSD_ZRS` | Higher availability for supported scenarios | Zone-redundant; availability depends on region and support |
| `Premium_ZRS` | Premium performance with zone redundancy | Supported only in specific regions and scenarios |
| `Premium SSD v2` | Fine-grained performance tuning | Separate provisioning model; not identical to classic Premium SSD |
| `Ultra Disk` | Very high IOPS and throughput | Typically for data disks only; strict configuration constraints |

Important distinctions:
- **Standard HDD** is the lowest-cost option and is suitable for infrequent access or nonproduction workloads.
- **Standard SSD** is a balanced option for many general workloads.
- **Premium SSD** is the common choice for production VMs that need predictable latency.
- **Premium SSD v2** gives more flexible performance tuning than classic Premium SSD, which can help when you want to adjust performance without changing the entire disk size in the same way as older models.
- **Ultra Disk** is designed for extreme performance needs such as large databases or latency-sensitive workloads, but it is not the default choice for general VM storage.

**Premium SSD, Premium SSD v2, and Ultra Disk**

These three are often compared in exam questions.

- **Premium SSD**
  - Best for most production VM disks.
  - Performance is tied to disk size and SKU.
  - Common for OS disks and application data disks.

- **Premium SSD v2**
  - Better for workloads that need more flexible performance tuning.
  - Useful when you want to adjust IOPS and throughput more granularly.
  - Not every region or VM size supports it.

- **Ultra Disk**
  - Best for the most demanding data workloads.
  - Usually used as a data disk, not an OS disk.
  - Requires specific VM size and region support.
  - Often used for databases, transaction-heavy systems, and high-throughput analytics.

A practical rule:
- Choose **Premium SSD** for most production VMs.
- Choose **Premium SSD v2** when you need more tunable performance and the region supports it.
- Choose **Ultra Disk** only when you have a clearly justified high-performance requirement.

**Redundancy and availability**

Managed disk redundancy is expressed through the SKU, not as a generic property applied the same way to every disk type.

What to remember:
- `LRS` means **locally redundant storage**.
- `ZRS` means **zone-redundant storage**.
- Some managed disk SKUs support `ZRS`, but not all do.
- Redundancy choice affects resilience, not just cost.

Use cases:
- Choose `LRS` when you want lower cost and can tolerate regional-zone failure impact on the disk.
- Choose `ZRS` when you want the disk replicated across availability zones for higher resilience in supported regions.

For AZ-104, be careful not to assume that all managed disks can be made zone redundant. The exact SKU and region support matter.

**OS disks, data disks, and ephemeral OS disks**

Azure VM storage is usually discussed in terms of disk role.

- **OS disk**
  - Contains the operating system.
  - Usually smaller than data disks.
  - Commonly cached as `ReadWrite` or `ReadOnly` depending on platform guidance and workload.

- **Data disk**
  - Stores application data, databases, logs, or file shares.
  - Can be attached and detached independently.
  - Often uses `None` or `ReadOnly` caching depending on workload.

- **Ephemeral OS disk**
  - Temporary OS storage that is not persisted on the managed disk service.
  - Useful for stateless workloads, scale sets, or scenarios where fast reimaging matters more than persistence.
  - Not suitable when you need OS disk persistence across host moves or deallocation events.

Ephemeral OS disks are a common exam distractor because they look like a normal OS disk choice, but they behave differently. If the workload must survive host-level changes with the OS disk intact, use a managed OS disk instead.

**Caching recommendations**

Disk caching affects how Azure VM reads and writes data. The right setting depends on the disk role and workload.

General guidance:
- **OS disk**
  - Often use `ReadWrite` caching for Windows and Linux OS disks.
  - This helps boot and OS responsiveness.
- **Data disk**
  - Often use `None` for write-heavy workloads.
  - Use `ReadOnly` for read-heavy workloads where caching improves performance.
- **Log disk**
  - Usually use `None`.
  - Logs are write-intensive and should avoid caching that can interfere with write ordering.

Practical workload mapping:
- **Windows OS disk**: `ReadWrite` is commonly recommended.
- **Linux OS disk**: `ReadWrite` is commonly recommended.
- **Database data files**: often `ReadOnly` or `None` depending on access pattern and product guidance.
- **Transaction logs**: usually `None`.
- **General application data**: choose based on whether reads dominate writes.

> Do not assume caching is always beneficial. For write-heavy workloads, the wrong cache mode can reduce consistency or performance.

**Encryption and security options**

Azure disk security is an important AZ-104 topic. There are multiple layers, and they are not interchangeable.

Main options:
- **Server-side encryption (SSE)**
  - Azure encrypts data at rest by default.
  - Uses platform-managed keys unless you configure customer-managed keys.
- **Customer-managed keys (CMK)**
  - You control the key in Azure Key Vault or Managed HSM.
  - Useful for compliance or key ownership requirements.
- **Azure Disk Encryption (ADE)**
  - Uses BitLocker for Windows or DM-Crypt for Linux.
  - Encrypts the VM disks from inside the guest OS.
  - Often used for legacy or specific compliance scenarios, but it is more operationally complex than SSE.

Important exam distinction:
- **SSE with platform-managed keys** is the default baseline.
- **SSE with customer-managed keys** is the modern control-plane encryption choice for many compliance scenarios.
- **ADE** is guest-level encryption and is not the same thing as SSE.

When to choose each:
- Use **SSE** for default protection at rest.
- Use **CMK** when policy requires customer control over encryption keys.
- Use **ADE** only when a specific requirement calls for guest-based encryption or legacy compatibility.

**Shared disks**

Azure shared disks let multiple VMs attach to the same managed disk in supported configurations.

Use cases:
- Clustered applications.
- High-availability failover clusters.
- Shared application state that requires coordinated access.

Constraints:
- Not every disk SKU supports sharing.
- The workload must be designed for concurrent access.
- You need application-level coordination or a cluster file system.

Shared disks are not a general-purpose way to share a data volume between unrelated VMs. They are for specialized clustered workloads.

**Disk performance and sizing**

Disk performance is influenced by more than just the SKU.

Factors that matter:
- **Disk size**
- **Disk SKU**
- **VM size**
- **Caching mode**
- **Region and feature support**
- **Number of attached disks and striping design**

Practical implications:
- A fast disk attached to a small VM may still be bottlenecked by the VM’s IOPS and throughput limits.
- Increasing disk size can increase performance in some SKUs.
- Ultra Disk and Premium SSD v2 allow more granular performance tuning than classic fixed-tier models.

For exam scenarios, always check whether the VM size can actually consume the disk performance you are trying to provision.

**Lifecycle behavior when deleting a VM**

Deleting a VM does not automatically mean every attached disk is deleted. The outcome depends on the VM’s delete options and how the disks were configured.

What to remember:
- OS disk and data disk deletion behavior can be configured separately.
- Some deployments retain disks after VM deletion so you can recover data or redeploy later.
- Other deployments delete disks to avoid orphaned resources and cost.

This matters operationally:
- Retain disks when you need recovery or forensic access.
- Delete disks when the VM is disposable and you want to avoid storage charges.

A safe exam mindset:
- Check the delete behavior for the VM, OS disk, and data disks.
- Do not assume a fixed default across all creation methods or portals.

**Migration and upgrade scenarios**

You may need to move a disk to a different performance tier or change its configuration over time.

Common scenarios:
- Upgrade from `StandardSSD_LRS` to `Premium_LRS` for better latency.
- Move from classic Premium SSD to Premium SSD v2 when you need more flexible tuning.
- Resize a disk to increase capacity and, in some SKUs, performance.
- Change caching when the workload changes from write-heavy to read-heavy.

Typical workflow:
1. Verify the target SKU is supported in the region and by the VM.
2. Check whether the disk can be changed online or requires a stop/deallocate.
3. Update the disk configuration.
4. Validate performance and application behavior after the change.

> Some disk changes are not instantaneous or may require VM downtime. Always verify the supported path before modifying production storage.

**CLI examples**

Use Azure CLI to inspect and manage disks.

List disks:
```bash
az disk list -o table
```

Show a specific disk:
```bash
az disk show --resource-group rg-app --name appDataDisk01
```

Create an empty managed disk:
```bash
az disk create \
  --resource-group rg-app \
  --name appDataDisk01 \
  --size-gb 128 \
  --sku Premium_LRS
```

Create a disk from a snapshot:
```bash
az disk create \
  --resource-group rg-app \
  --name appDataDisk01-copy \
  --source appDataSnapshot01
```

Attach a disk to a VM:
```bash
az vm disk attach \
  --resource-group rg-app \
  --vm-name vm01 \
  --name appDataDisk01
```

Resize a disk:
```bash
az disk update \
  --resource-group rg-app \
  --name appDataDisk01 \
  --size-gb 256
```

Update caching on an attached disk:
```bash
az vm update \
  --resource-group rg-app \
  --name vm01 \
  --set storageProfile.dataDisks[0].caching=None
```

**Bicep example**

A complete VM deployment needs an image, OS disk settings, and any attached data disks. This example shows a valid pattern for a Linux VM with one managed data disk.

```bicep
param location string = resourceGroup().location
param vmName string = 'vm01'
param adminUsername string = 'azureuser'
@secure()
param adminPassword string

resource dataDisk 'Microsoft.Compute/disks@2023-04-02' = {
  name: '${vmName}-data01'
  location: location
  sku: {
    name: 'Premium_LRS'
  }
  properties: {
    diskSizeGB: 128
    creationData: {
      createOption: 'Empty'
    }
  }
}

resource nic 'Microsoft.Network/networkInterfaces@2023-11-01' = {
  name: '${vmName}-nic'
  location: location
  properties: {
    ipConfigurations: [
      {
        name: 'ipconfig1'
        properties: {
          subnet: {
            id: '/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Network/virtualNetworks/<vnet>/subnets/<subnet>'
          }
          privateIPAllocationMethod: 'Dynamic'
        }
      }
    ]
  }
}

resource vm 'Microsoft.Compute/virtualMachines@2023-09-01' = {
  name: vmName
  location: location
  properties: {
    hardwareProfile: {
      vmSize: 'Standard_DS2_v2'
    }
    osProfile: {
      computerName: vmName
      adminUsername: adminUsername
      adminPassword: adminPassword
      linuxConfiguration: {
        disablePasswordAuthentication: false
      }
    }
    storageProfile: {
      imageReference: {
        publisher: 'Canonical'
        offer: '0001-com-ubuntu-server-jammy'
        sku: '22_04-lts'
        version: 'latest'
      }
      osDisk: {
        createOption: 'FromImage'
        managedDisk: {
          storageAccountType: 'Premium_LRS'
        }
      }
      dataDisks: [
        {
          lun: 0
          createOption: 'Attach'
          managedDisk: {
            id: dataDisk.id
          }
          caching: 'None'
        }
      ]
    }
    networkProfile: {
      networkInterfaces: [
        {
          id: nic.id
        }
      ]
    }
  }
}
```

Why this matters:
- The VM includes a valid image reference.
- The OS disk is created from the image.
- The data disk is created separately and attached.
- The caching mode is set explicitly for the data disk.

**Linux disk initialization example**

After attaching a new data disk to Linux, you usually partition, format, and mount it. Do not format a partition before the kernel recognizes it.

Example workflow:
```bash
lsblk
sudo fdisk /dev/sdc
sudo partprobe /dev/sdc
lsblk
sudo mkfs.ext4 /dev/sdc1
sudo mkdir -p /data
sudo mount /dev/sdc1 /data
df -h /data
```

What each step does:
- `lsblk` confirms the new disk name.
- `fdisk` creates the partition.
- `partprobe` asks the kernel to re-read the partition table.
- `mkfs.ext4` formats the new partition.
- `mount` makes the filesystem available.

To persist the mount:
```bash
sudo blkid /dev/sdc1
sudo nano /etc/fstab
```

Example `/etc/fstab` entry:
```fstab
UUID=<uuid-from-blkid>  /data  ext4  defaults,nofail  0  2
```

**Windows disk initialization and expansion**

In Windows, you typically initialize the disk, create a partition, format it, and then extend it when the Azure disk grows.

Initialize and format a new disk:
```powershell
Get-Disk
Initialize-Disk -Number 2 -PartitionStyle GPT
New-Partition -DiskNumber 2 -UseMaximumSize -AssignDriveLetter
Format-Volume -DriveLetter F -FileSystem NTFS -NewFileSystemLabel "Data01" -Confirm:$false
```

Extend a volume after increasing the Azure disk size:
```powershell
Get-PartitionSupportedSize -DriveLetter F
Resize-Partition -DriveLetter F -Size (Get-PartitionSupportedSize -DriveLetter F).SizeMax
```

Why this approach is preferred:
- You query the maximum supported size first.
- You extend to the full available size instead of hard-coding a value.
- You avoid mismatches between the Azure disk size and the guest partition size.

**Portal workflow for a typical VM disk setup**

A common exam scenario is creating a VM with one OS disk and one data disk for an application server.

Recommended choices:
1. Select a production-capable VM size that supports the required disk performance.
2. Choose a managed OS disk with `Premium_LRS` if the VM needs low latency.
3. Add a data disk with `Premium_LRS` or `StandardSSD_LRS` depending on workload.
4. Set OS disk caching to the platform-recommended value.
5. Set data disk caching to `None` for write-heavy workloads or `ReadOnly` for read-heavy workloads.
6. Decide whether the disk should be deleted with the VM or retained for recovery.
7. Enable encryption policy or CMK if required by compliance.

Example workload choices:
- **Web server**: Premium SSD OS disk, Standard SSD or Premium SSD data disk, moderate caching.
- **Database server**: Premium SSD or Ultra Disk data disks, `None` caching for logs, careful sizing.
- **Dev/test VM**: Standard HDD or Standard SSD to reduce cost.
- **Stateless scale-out VM**: Ephemeral OS disk may be appropriate if persistence is not required.

**Troubleshooting and exam traps**

Common mistakes to avoid:
- Assuming every managed disk supports zone redundancy.
- Treating `Ultra Disk` as a general-purpose OS disk.
- Using the wrong cache mode for logs or write-heavy data.
- Forgetting that disk performance depends on both the disk and the VM size.
- Formatting a Linux partition before the kernel has re-read the partition table.
- Hard-coding a Windows partition size instead of extending to the maximum supported size.
- Assuming a VM deletion always deletes attached disks.

Useful checks:
- Verify the disk SKU.
- Verify the region supports the feature.
- Verify the VM size supports the required IOPS and throughput.
- Verify the disk role before choosing caching.
- Verify whether the disk should be retained or deleted with the VM.

**Quick exam summary**

- Managed disks are the standard Azure VM storage model.
- Disk behavior is defined by the SKU, size, caching, and encryption settings.
- `Standard HDD`, `Standard SSD`, `Premium SSD`, `Premium SSD v2`, and `Ultra Disk` serve different performance needs.
- `LRS` and `ZRS` are not generic options for every managed disk SKU.
- OS disks, data disks, and ephemeral OS disks behave differently.
- Caching should match workload type, especially for logs and databases.
- Encryption can be handled by SSE, CMK, or ADE depending on the requirement.
- Disk deletion behavior depends on VM and disk delete options.
- Always validate region and VM-size support before choosing advanced disk features.
