## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure App Service overview**

Azure App Service is a managed platform for hosting web applications, REST APIs, and mobile back ends without managing the underlying operating system. You deploy code or containers, and Azure handles much of the infrastructure, patching, load balancing, and scaling behavior.

App Service is built around three related concepts:

- **App Service plan**: the compute and pricing boundary.
- **Web App**: the application endpoint that runs in a plan.
- **Deployment slot**: a separate deployment target within the same app and plan.

A useful mental model is:

| Concept | What it represents | Why it matters |
|---|---|---|
| App Service plan | The VM-like compute allocation and SKU | Determines region, OS, scale, and features |
| Web App | The hosted application | Holds code, settings, identity, and runtime config |
| Deployment slot | A live staging endpoint | Enables safe release testing and swap-based deployments |

**App Service plan fundamentals**

An App Service plan defines the compute resources that host one or more apps. Multiple web apps can share the same plan, which is often cost-effective when apps have similar performance needs and lifecycle patterns.

Key plan characteristics:

- **Region**: the plan is created in a specific Azure region.
- **OS type**: Windows or Linux.
- **SKU/tier**: determines scale, features, and pricing.
- **Scale unit**: instances can be scaled out horizontally within the plan.

Because apps in the same plan share the same underlying compute pool, noisy-neighbor effects can occur if one app consumes excessive CPU, memory, or connections. For that reason, you often separate production workloads from dev/test workloads or from apps with different scaling profiles.

**Plan, app, and slot relationship**

The plan is the billing and compute container, while the app is the runtime identity and configuration boundary. Slots belong to a specific web app and inherit the app’s plan.

- A single plan can host multiple web apps.
- A web app can have multiple deployment slots.
- All slots for a given app run in the same App Service plan.
- Slots are not independent apps; they are deployment targets tied to the parent app.

This matters for AZ-104 because many features are controlled by the plan tier, not by the app itself.

**SKU and feature boundaries**

App Service features vary by tier. For exam prep, the most important boundaries are:

- **Free and Shared**: basic hosting only; no deployment slots.
- **Basic**: supports custom domains and SSL options, but not deployment slots or autoscale.
- **Standard**: introduces deployment slots and autoscale.
- **Premium tiers**: add more scale, performance, and advanced capabilities.
- **Isolated / ASE**: dedicated environment for network isolation and enterprise scenarios.

A practical summary:

| Tier | Deployment slots | Autoscale | Backups | Typical use |
|---|---:|---:|---:|---|
| Free | No | No | No | Learning, testing |
| Shared | No | No | No | Very small dev/test |
| Basic | No | No | No | Small production apps without slot needs |
| Standard | Yes | Yes | Yes | Production web apps |
| Premium | Yes | Yes | Yes | Higher scale and performance |
| Isolated | Yes | Yes | Yes | Private, dedicated hosting |

> Deployment slots start at **Standard** tier and above. If you need slots, do not choose Basic.

**Creating an App Service plan with Azure CLI**

You create the plan first, then deploy apps into it. For Windows plans, omit `--is-linux`. Use `--is-linux` only when you want a Linux plan.

```bash
# Windows App Service plan
az appservice plan create \
  --name asp-prod-eastus \
  --resource-group rg-web \
  --sku S1 \
  --location eastus

# Linux App Service plan
az appservice plan create \
  --name asp-linux-eastus \
  --resource-group rg-web \
  --sku S1 \
  --is-linux \
  --location eastus
```

Common plan inspection commands:

```bash
az appservice plan show \
  --name asp-prod-eastus \
  --resource-group rg-web

az appservice plan list \
  --resource-group rg-web \
  --output table
```

**Web Apps and runtime configuration**

A web app is the application resource that runs your code. It stores app settings, connection strings, identity configuration, deployment source, and runtime-specific options.

Important app settings and behaviors:

- **Application settings**: environment variables for the app runtime.
- **Connection strings**: managed separately from general settings.
- **Always On**: keeps the app loaded to reduce cold starts.
- **Health check**: probes a path and helps Azure detect unhealthy instances.
- **Startup command**: used for Linux apps and custom containers.
- **Platform stack**: .NET, Node.js, Python, Java, PHP, or custom container.

For production workloads, you usually enable **Always On** when the tier supports it, especially for apps that must respond quickly after idle periods. Health checks are useful when you want Azure to remove unhealthy instances from rotation and replace them.

Example: create a Windows web app in an existing plan.

```bash
az webapp create \
  --name app-prod-001 \
  --resource-group rg-web \
  --plan asp-prod-eastus \
  --runtime "DOTNET|8.0"
```

Example: create a Linux web app.

```bash
az webapp create \
  --name app-linux-001 \
  --resource-group rg-web \
  --plan asp-linux-eastus \
  --runtime "NODE:20-lts"
```

Useful configuration commands:

```bash
# Set app settings
az webapp config appsettings set \
  --name app-prod-001 \
  --resource-group rg-web \
  --settings ماحول=prod FEATURE_FLAG=true

# Enable Always On
az webapp config set \
  --name app-prod-001 \
  --resource-group rg-web \
  --always-on true

# Configure a health check path
az webapp config set \
  --name app-prod-001 \
  --resource-group rg-web \
  --health-check-path /health

# Set a startup command for Linux/custom container scenarios
az webapp config set \
  --name app-linux-001 \
  --resource-group rg-web \
  --startup-file "npm start"
```

> `Always On` is not available in every tier. Check the plan SKU before relying on it for production behavior.

**Deployment options**

App Service supports several deployment methods:

- **Local Git**: push code directly to a Git endpoint.
- **GitHub Actions / Azure DevOps**: CI/CD pipelines for automated releases.
- **Zip deploy**: upload a ZIP package to the app.
- **Container image**: deploy from Azure Container Registry or another registry.
- **FTP/FTPS**: legacy or simple file transfer scenarios.

For AZ-104, the most practical methods are ZIP deployment and pipeline-based deployment.

ZIP deployment example:

```bash
az webapp deployment source config-zip \
  --resource-group rg-web \
  --name app-prod-001 \
  --src ./app.zip
```

You can also deploy from a local path with `az webapp deploy` in supported scenarios:

```bash
az webapp deploy \
  --resource-group rg-web \
  --name app-prod-001 \
  --src-path ./app.zip
```

After deployment, verify the app:

```bash
az webapp browse \
  --resource-group rg-web \
  --name app-prod-001
```

**Deployment slots**

Deployment slots let you stage a new version of an app in a live environment before swapping it into production. Each slot has its own hostname and can run a different build, configuration, or test data path.

Typical slot workflow:

- Deploy new code to a staging slot.
- Validate the app under real App Service conditions.
- Swap staging into production.
- Keep the previous production version in the staging slot for rollback.

Slot support and counts are tier-dependent:

- **Standard**: supports slots, typically up to 5 total slots including production.
- **Premium**: supports more slots, commonly up to 20 total slots including production.
- **Isolated**: supports the highest slot counts, commonly up to 20 total slots depending on the plan family.
- **Basic and below**: no deployment slots.

The exact maximum can vary by SKU family, but the exam-relevant rule is simple: slots begin at Standard.

**Slot swap behavior**

A swap exchanges the content and selected configuration between two slots. Azure performs warm-up and routing steps to reduce downtime.

What usually swaps:

- Application content
- Most app settings
- Connection strings, depending on slot setting behavior
- Hostnames and routing target

What can stay sticky to a slot:

- **Slot-specific settings** marked as deployment slot settings
- Some connection strings or app settings you explicitly pin to a slot
- Custom warm-up or test settings

This is important because not every setting should move with the code. For example, a staging slot may point to a test database, while production points to a live database.

Example: create and use a staging slot.

```bash
az webapp deployment slot create \
  --resource-group rg-web \
  --name app-prod-001 \
  --slot staging
```

Deploy to the staging slot:

```bash
az webapp deployment source config-zip \
  --resource-group rg-web \
  --name app-prod-001 \
  --slot staging \
  --src ./app.zip
```

Swap staging into production:

```bash
az webapp deployment slot swap \
  --resource-group rg-web \
  --name app-prod-001 \
  --slot staging \
  --target-slot production
```

**Slot settings and sticky configuration**

Some settings should remain tied to a slot rather than move during a swap. These are often called **slot settings** or sticky settings.

Use slot settings for:

- Connection strings to different databases
- Environment-specific API endpoints
- Feature flags for staged testing
- Diagnostic or logging differences between environments

Example: mark an app setting as sticky to the slot.

```bash
az webapp config appsettings set \
  --resource-group rg-web \
  --name app-prod-001 \
  --slot-settings DB_ENV=staging FEATURE_FLAG=true
```

If you do not mark a setting as sticky, a swap may move it with the content, which can cause production to point at the wrong backend.

**Autoscale**

Autoscale adjusts the number of instances in a plan based on metrics or schedules. It is a plan-level feature, not an app-level feature.

Use autoscale when:

- Traffic varies by time of day or season.
- You need to respond to CPU, memory, or queue length changes.
- You want to reduce manual scaling operations.

Autoscale is supported in Standard and above. Basic does not support autoscale.

Example: create an autoscale setting with a CPU-based rule.

```bash
az monitor autoscale create \
  --resource-group rg-web \
  --resource asp-prod-eastus \
  --resource-type Microsoft.Web/serverfarms \
  --name autoscale-web

az monitor autoscale rule create \
  --resource-group rg-web \
  --autoscale-name autoscale-web \
  --condition "Percentage CPU > 70 avg 5m" \
  --scale out 1
```

A more complete autoscale profile often includes both scale-out and scale-in rules, plus minimum and maximum instance counts.

**Backups**

App Service backups protect app content and some configuration data. Backups are supported on Standard and above, not on Basic or Free/Shared tiers.

Use backups when:

- You need a restore point before a risky deployment.
- You want recovery for app content and configuration.
- You manage production apps with change control requirements.

Example: configure a backup schedule.

```bash
az webapp config backup create \
  --resource-group rg-web \
  --webapp-name app-prod-001 \
  --container-url "https://mystorageaccount.blob.core.windows.net/backups?sas-token" \
  --frequency 24 \
  --retain-one true
```

Backup storage usually requires a storage account container with appropriate access. In practice, you should test restore procedures, not just backup creation.

**Custom domains and TLS/SSL**

App Service supports custom domains and TLS/SSL bindings. You typically map a DNS record to the app hostname and then bind a certificate.

Common patterns:

- **CNAME** for subdomains like `www.contoso.com`
- **A record** for apex/root domains when needed
- **Managed certificate** for simpler TLS management in supported scenarios

Example: add a hostname binding.

```bash
az webapp config hostname add \
  --webapp-name app-prod-001 \
  --resource-group rg-web \
  --hostname www.contoso.com
```

For exam purposes, remember that domain binding and certificate management are app-level concerns, while the plan determines the hosting capabilities.

**Linux, Windows, and custom containers**

App Service supports both Windows and Linux hosting. The choice affects runtime behavior, deployment style, and configuration.

Windows apps are often used for:

- .NET Framework workloads
- IIS-based applications
- Windows-specific dependencies

Linux apps are often used for:

- Node.js, Python, Java, PHP
- Container-based deployments
- Open-source runtime stacks

Custom containers are useful when you need full control over the application environment, dependencies, or startup process. In that case, you often configure:

- Container image source
- Registry credentials
- Startup command or entrypoint behavior
- Health check path

Example: configure a container image.

```bash
az webapp create \
  --resource-group rg-web \
  --plan asp-linux-eastus \
  --name app-container-001 \
  --deployment-container-image-name myregistry.azurecr.io/myapp:1.0
```

**Operational troubleshooting**

When an App Service app misbehaves, check the app, the plan, and the deployment pipeline together. Many issues are caused by configuration drift rather than code defects.

Common checks:

- Verify the app is in the expected plan and region.
- Confirm the tier supports the feature you are using.
- Review app settings and connection strings.
- Check deployment logs and swap history.
- Validate health check responses.
- Confirm the app is not hitting instance or quota limits.

Useful CLI commands:

```bash
az webapp log tail \
  --resource-group rg-web \
  --name app-prod-001

az webapp show \
  --resource-group rg-web \
  --name app-prod-001 \
  --query "{state:state, hostNames:hostNames, defaultHostName:defaultHostName}"
```

**AZ-104 exam focus**

For the exam, focus on the relationship between tier and feature. Many questions are really asking which SKU is required for a capability.

High-value facts:

- **Deployment slots** start at Standard.
- **Autoscale** starts at Standard.
- **Backups** start at Standard.
- **Basic** does not include slots or autoscale.
- **The plan** controls compute and feature availability.
- **The app** holds runtime configuration and deployment settings.
- **Slots** are for safe release management and rollback.

A simple release topology looks like this:

```text
Internet
   |
DNS / Custom Domain
   |
Production slot  <---- swap ---->  Staging slot
   |
App Service Plan
```

**Practical deployment pattern**

A common production pattern is:

1. Create a Standard or Premium App Service plan.
2. Deploy the production web app into the plan.
3. Create a staging slot.
4. Deploy new builds to staging.
5. Run smoke tests and validate health checks.
6. Swap staging into production.
7. Keep the old version available for quick rollback.

This pattern reduces downtime and gives you a safer release process than direct production deployment.

**Quick comparison**

| Feature | Controlled by | Notes |
|---|---|---|
| Compute size and cost | App Service plan | SKU determines capacity and features |
| Runtime settings | Web App | App settings, stack, identity, health check |
| Safe pre-production release | Deployment slot | Requires Standard or above |
| Horizontal scaling | App Service plan | Autoscale is plan-level |
| Restore capability | Web App backup | Supported on Standard and above |

**Key takeaways**

- The **plan** is the hosting and pricing boundary.
- The **web app** is the application resource and configuration boundary.
- The **slot** is a deployment target for safe releases.
- **Standard** is the minimum tier for slots, autoscale, and backups.
- Use **swap** to promote tested code with minimal downtime.
- Keep environment-specific settings sticky to avoid accidental production changes.
