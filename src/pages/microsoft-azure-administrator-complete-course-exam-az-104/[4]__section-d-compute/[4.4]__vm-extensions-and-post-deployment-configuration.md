## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**VM extensions overview**

Azure VM extensions are small post-deployment components that run inside the guest operating system to automate configuration tasks after the VM is provisioned. You use them when ARM/Bicep deployment alone is not enough and you need guest-level actions such as installing software, joining a domain, configuring monitoring, or applying security settings.

Extensions are managed by the Azure VM agent on the guest. On Windows, this is the Windows Azure Guest Agent; on Linux, it is typically the Azure Linux Agent (`waagent`) or a compatible agent path. The extension handler is OS-specific, so the publisher, type, and version must match the VM operating system and the extension family exactly.

- **Guest-level automation** runs inside the VM, not from the Azure control plane alone.
- **Handler compatibility** matters: Windows and Linux often use different publishers and types.
- **Post-deployment** tasks are ideal for extensions because they are repeatable and declarative.

**When to use extensions**

Use extensions when you need a one-time or repeatable configuration step after VM creation. Common examples include:

- Installing a monitoring agent
- Joining a Windows VM to Active Directory
- Running a bootstrap script on first boot
- Enabling backup-related guest configuration
- Deploying security or compliance tooling

Use other services when the requirement is broader than a single VM or needs ongoing policy enforcement. For example, Azure Policy with Guest Configuration is better for compliance drift detection and continuous assessment, while VM extensions are better for immediate guest actions.

**Core extension concepts**

Extensions are usually deployed as child resources of a VM. In ARM templates and Bicep, they are modeled under the VM resource scope. In the Azure portal, they appear under the VM’s Extensions + applications blade.

Important properties commonly include:

- `publisher`: The extension vendor or Microsoft publisher namespace
- `type`: The extension handler type
- `typeHandlerVersion`: The handler version to use
- `autoUpgradeMinorVersion`: Whether Azure can automatically use newer minor versions
- `settings`: Non-secret configuration values
- `protectedSettings`: Secret values such as passwords, keys, or tokens

A typical extension lifecycle is:

1. Azure places the extension resource on the VM.
2. The VM agent downloads the handler package.
3. The handler executes the requested configuration.
4. Status is reported back to Azure as success or failure.

**Versioning and upgrade behavior**

Versioning is a frequent exam topic because extension behavior can change across versions. You should understand the difference between pinning and auto-upgrading.

- **Pinned version** means you specify an exact handler version and keep behavior stable.
- **Auto-upgrade minor version** allows Azure to move to newer minor releases within the same major line.
- **Handler changes** can affect production VMs, so test upgrades before broad rollout.
- **OS-specific versions** may differ between Windows and Linux even for similar tasks.

For production workloads, pin versions when you need predictable behavior, and use auto-upgrade only when you are comfortable with vendor-managed minor updates.

> Extension versioning is not the same as VM image versioning. A VM can stay on the same image while its extension handler changes behavior if auto-upgrade is enabled.

**Common extension types and use cases**

| Use case | Typical extension | Notes |
|---|---|---|
| Run a script after deployment | Custom Script Extension | Common for bootstrap and configuration |
| Join a domain | JsonADDomainExtension or domain join extension | Windows-focused scenario |
| Install monitoring | Azure Monitor Agent or legacy monitoring extension | Exact resource shape depends on OS and deployment path |
| Enable backup-related guest actions | Backup-related extension or policy-driven guest configuration | Often paired with Azure Backup and policy |
| Security baseline or compliance | Guest Configuration | Better for ongoing compliance than one-time changes |

**Custom Script Extension**

The Custom Script Extension is one of the most common post-deployment tools. It downloads and runs scripts from a storage account, Git repository, or other reachable location. It is useful for bootstrap tasks such as installing packages, configuring services, or applying local settings.

For Windows, the handler family is different from Linux. A Windows example typically uses the Microsoft Compute publisher and a Windows-specific type. A Linux example typically uses the Microsoft Azure Extensions publisher and a Linux-specific type. Do not mix them.

Windows example:

```json
{
  "type": "Microsoft.Compute/virtualMachines/extensions",
  "apiVersion": "2023-09-01",
  "name": "myVm/customScript",
  "location": "eastus",
  "properties": {
    "publisher": "Microsoft.Compute",
    "type": "CustomScriptExtension",
    "typeHandlerVersion": "1.10",
    "autoUpgradeMinorVersion": true,
    "settings": {
      "fileUris": [
        "https://mystorage.blob.core.windows.net/scripts/install.ps1"
      ],
      "commandToExecute": "powershell -ExecutionPolicy Unrestricted -File install.ps1"
    }
  }
}
```

Linux example:

```json
{
  "type": "Microsoft.Compute/virtualMachines/extensions",
  "apiVersion": "2023-09-01",
  "name": "myVm/customScript",
  "location": "eastus",
  "properties": {
    "publisher": "Microsoft.Azure.Extensions",
    "type": "CustomScript",
    "typeHandlerVersion": "2.1",
    "autoUpgradeMinorVersion": true,
    "settings": {
      "fileUris": [
        "https://mystorage.blob.core.windows.net/scripts/install.sh"
      ],
      "commandToExecute": "bash install.sh"
    }
  }
}
```

In Bicep, define the extension as a child resource of the VM so the relationship is explicit:

```bicep
resource vm 'Microsoft.Compute/virtualMachines@2023-09-01' = {
  name: 'myVm'
  location: resourceGroup().location
  properties: {
    hardwareProfile: {
      vmSize: 'Standard_B2s'
    }
    osProfile: {
      computerName: 'myVm'
      adminUsername: 'azureuser'
      adminPassword: 'P@ssw0rd1234!'
    }
  }
}

resource customScript 'Microsoft.Compute/virtualMachines/extensions@2023-09-01' = {
  parent: vm
  name: 'customScript'
  location: vm.location
  properties: {
    publisher: 'Microsoft.Azure.Extensions'
    type: 'CustomScript'
    typeHandlerVersion: '2.1'
    autoUpgradeMinorVersion: true
    settings: {
      fileUris: [
        'https://mystorage.blob.core.windows.net/scripts/install.sh'
      ]
      commandToExecute: 'bash install.sh'
    }
  }
}
```

**Practical script considerations**

- Keep scripts idempotent so reruns do not break the VM.
- Store scripts in a trusted location with controlled access.
- Use `protectedSettings` for secrets instead of embedding them in plain text.
- Log output to a file so you can troubleshoot failures later.

Example Linux bootstrap script:

```bash
#!/bin/bash
set -euo pipefail

apt-get update
apt-get install -y nginx
systemctl enable nginx
systemctl start nginx
echo "Nginx installed" > /var/log/bootstrap.log
```

**Domain join and identity-related configuration**

A common Windows post-deployment task is joining a VM to an Active Directory domain. This is often done with a domain join extension or through automation that runs inside the guest. The extension typically needs domain name, OU path, username, and password or a secure credential reference.

Use this pattern when:

- You need the VM to participate in on-premises authentication
- You want centralized Group Policy management
- You are building line-of-business servers that must be domain joined immediately after deployment

Be careful with secrets. Domain credentials should be stored securely, often in Key Vault and referenced through deployment automation rather than hardcoded in templates.

**Monitoring and backup-related configuration**

Post-deployment configuration often includes installing monitoring or backup components.

- **Azure Monitor Agent (AMA)** is the modern monitoring agent used for collecting logs and metrics through Data Collection Rules.
- **Legacy monitoring extensions** may still appear in older environments, but you should recognize that deployment paths vary by OS and platform.
- **Azure Backup** may require guest-side coordination for application-consistent backups or workload-specific protection.

A practical workflow is:

1. Deploy the VM.
2. Install the monitoring agent.
3. Associate the VM with a Data Collection Rule.
4. Enable backup policy and verify the vault association.

Example CLI to install an extension:

```bash
az vm extension set \
  --resource-group rg-prod \
  --vm-name myVm \
  --name CustomScript \
  --publisher Microsoft.Azure.Extensions \
  --version 2.1 \
  --settings '{"fileUris":["https://mystorage.blob.core.windows.net/scripts/install.sh"],"commandToExecute":"bash install.sh"}'
```

Example PowerShell to add an extension:

```powershell
Set-AzVMExtension `
  -ResourceGroupName "rg-prod" `
  -VMName "myVm" `
  -Name "CustomScript" `
  -Publisher "Microsoft.Azure.Extensions" `
  -ExtensionType "CustomScript" `
  -TypeHandlerVersion "2.1" `
  -Location "eastus" `
  -SettingString '{ "fileUris": ["https://mystorage.blob.core.windows.net/scripts/install.sh"], "commandToExecute": "bash install.sh" }'
```

**Azure Portal management**

In the Azure portal, you manage extensions from the VM blade.

- Open the VM and select **Extensions + applications**.
- Review installed extensions and their provisioning state.
- Add a new extension from the gallery.
- Remove an extension when it is no longer needed.
- Check **Instance view** for runtime status and error details.

Portal workflows are useful for quick validation, but for repeatable administration you should prefer CLI, PowerShell, ARM, or Bicep.

**Azure CLI and PowerShell administration**

You should know how to inspect, update, and remove extensions from the command line.

List extensions on a VM:

```bash
az vm extension list \
  --resource-group rg-prod \
  --vm-name myVm \
  -o table
```

Show instance view and status:

```bash
az vm extension show \
  --resource-group rg-prod \
  --vm-name myVm \
  --name CustomScript \
  --expand instanceView
```

Remove an extension:

```bash
az vm extension delete \
  --resource-group rg-prod \
  --vm-name myVm \
  --name CustomScript
```

PowerShell equivalent:

```powershell
Get-AzVMExtension -ResourceGroupName "rg-prod" -VMName "myVm"

Remove-AzVMExtension -ResourceGroupName "rg-prod" -VMName "myVm" -Name "CustomScript"
```

If an extension is stuck or misconfigured, a common recovery pattern is to remove it and redeploy it cleanly.

**Troubleshooting extensions**

Extension failures usually show up as provisioning errors, timeout errors, or handler-specific exit codes. Start with the extension status in the portal or CLI, then inspect guest logs.

Useful troubleshooting steps:

1. Check the extension provisioning state and substatus.
2. Review the handler output and guest logs.
3. Confirm network access to script locations or package repositories.
4. Verify that the extension publisher, type, and version match the OS.
5. Remove and redeploy the extension if the state is corrupted.

Common log locations include:

- Windows:
  - `C:\WindowsAzure\Logs\WaAppAgent.log`
  - `C:\Packages\Plugins\<publisher>.<extension>\`
  - `C:\WindowsAzure\Logs\Plugins\`
- Linux:
  - `/var/log/waagent.log`
  - `/var/lib/waagent/`
  - `/var/log/azure/<extension>/`

On newer Linux images, extension-specific logs are often under `/var/log/azure/<extension>/`, while the agent state may still be tracked by `waagent`. Use both locations when investigating failures.

Common failure patterns:

- **Provisioning failed**: The handler returned a nonzero exit code or could not complete setup.
- **Timeout**: The script or install step took too long.
- **Download failure**: The VM could not reach the script URI or package source.
- **Permission denied**: The script lacks execute permission or the account lacks access.
- **Version mismatch**: The extension family does not match the OS or VM type.

> If the extension depends on outbound internet access, NSG rules, firewall rules, or proxy settings can cause silent download failures.

**Extension limits and operational constraints**

Extensions are powerful, but they are not unlimited. You should understand operational constraints because AZ-104 scenarios often test what happens when multiple extensions are involved.

- There is a practical limit to how many extensions a VM can host, and the exact limit depends on platform and extension type.
- Extensions may not execute in parallel when they depend on the same resources.
- Some extensions conflict with others if they modify the same services, registry keys, or files.
- Extension order matters when one task depends on another, such as installing prerequisites before application deployment.
- Long-running scripts can cause timeouts and failed provisioning states.

Design your deployment so each extension has a single responsibility. If you need complex orchestration, use Azure Automation, DevOps pipelines, or configuration management tools instead of stacking many extensions on one VM.

**VM extensions vs Azure Policy and Guest Configuration**

This distinction is important for exam questions.

- **VM extensions** perform actions inside the guest, usually at deployment time or on demand.
- **Azure Policy** evaluates resources against rules and can deploy extensions or enforce settings.
- **Guest Configuration** checks and reports on in-guest settings for compliance and can remediate some scenarios.

Use VM extensions when you need to make a change. Use Guest Configuration when you need to verify or continuously assess a change. Use Azure Policy when you need governance at scale.

**Practical scenario patterns**

- **New web server build**
  - Deploy the VM.
  - Use Custom Script Extension to install IIS or Nginx.
  - Install monitoring agent.
  - Register with backup policy.
- **Enterprise Windows server**
  - Deploy the VM.
  - Join the domain.
  - Install endpoint protection and monitoring.
  - Apply baseline configuration.
- **Compliance-focused workload**
  - Deploy the VM.
  - Use Guest Configuration to assess drift.
  - Use Policy to ensure required extensions are present.

**Exam tips**

- Match the extension publisher, type, and version to the OS and extension family.
- Remember that Windows and Linux custom script extensions are not interchangeable.
- Know how to view extension status in the portal and with CLI or PowerShell.
- Distinguish one-time configuration from ongoing compliance.
- Expect troubleshooting questions about logs, timeouts, download access, and handler failures.
- Recognize that version pinning can prevent unexpected behavior in production.

**Quick reference**

- **Portal path:** VM > **Extensions + applications**
- **CLI list:** `az vm extension list`
- **CLI status:** `az vm extension show --expand instanceView`
- **CLI remove:** `az vm extension delete`
- **PowerShell list:** `Get-AzVMExtension`
- **PowerShell remove:** `Remove-AzVMExtension`

**Key takeaways**

VM extensions are the standard Azure mechanism for guest-level post-deployment configuration. They are useful for scripts, domain join, monitoring, and other bootstrap tasks, but they must be matched carefully to the VM OS and extension family. For production, pay attention to versioning, logging, and operational limits, and use Azure Policy or Guest Configuration when the goal is compliance rather than one-time configuration.
