## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Virtual Machine Scale Sets overview**

Azure Virtual Machine Scale Sets (VMSS) let you deploy and manage a group of identical virtual machines as a single resource. You use them when you need to scale out compute capacity, keep configuration consistent, and simplify operational management across many instances.

VMSS is commonly used for horizontally scalable applications, but it is not limited to purely stateless workloads. With the right architecture, especially in **Flexible** orchestration mode, VMSS can also support scenarios that need more VM-like control and placement options. For AZ-104, focus on when VMSS is the right fit for repeatable, managed instance fleets rather than treating it as a separate product from Azure VMs.

- **Scale out** by adding instances when demand increases.
- **Scale in** by removing instances when demand decreases.
- **Consistency** comes from a shared model for image, extensions, networking, and policy.
- **Automation** is typically paired with Azure Monitor autoscale or custom orchestration.

**Core VMSS concepts**

A VMSS deployment starts with a model. The model defines the image, size, networking, extensions, and scaling behavior. Azure then applies that model to create instances.

Important concepts to know for the exam:

- **Instance**: one VM created from the scale set model.
- **Capacity**: the target number of instances in the scale set.
- **Model**: the desired configuration for all instances.
- **Instance view**: runtime status and health details for a specific instance.
- **Upgrade policy**: how model changes are applied to existing instances.
- **Autoscale**: rules that change capacity automatically based on metrics or schedules.

A practical mental model is: you define the desired state once, and Azure keeps the fleet aligned to that state as capacity changes.

**Orchestration modes: Uniform and Flexible**

Azure VMSS supports two orchestration modes, and this is a common AZ-104 distinction.

**Uniform orchestration**

Uniform mode is designed for large sets of identical instances managed as a group. It is the classic scale-set experience and is a strong fit when you want tight standardization and predictable fleet operations.

- Instances share a common model.
- Management is centered on the scale set resource.
- It is well suited to load-balanced application tiers and repeatable deployments.
- It supports scale-set style operations such as model updates and instance upgrades.

**Flexible orchestration**

Flexible mode gives you broader VM-like management and placement options while still benefiting from scale-set management. It is not the same as standalone VMs, but it offers more flexibility than Uniform mode.

- You can manage instances more individually.
- It supports a wider range of placement and operational patterns.
- It is useful when you need scale-set benefits but also need more control over instance behavior.
- It can support mixed or more state-aware designs better than the classic “all instances are identical” model.

> Do not assume Flexible means “just like standalone VMs.” It still uses VMSS management constructs and is not a full replacement for all VM behaviors.

**When to choose each mode**

- Use **Uniform** when you want a highly standardized fleet with simple scaling and consistent configuration.
- Use **Flexible** when you need more VM-like control, broader placement options, or a design that does not fit the strict uniform model.
- Check the required features of your workload before choosing, because some management and networking behaviors differ by mode.

**Deployment and creation options**

VMSS creation is commonly tested at a practical level. You should know the main choices you make during deployment.

- **Image source**
  - Azure Marketplace image
  - Custom image from Azure Compute Gallery
  - Managed image in some scenarios
- **Instance count**
  - Initial capacity is set at deployment time.
  - You can later change capacity manually or through autoscale.
- **VM size**
  - Choose based on CPU, memory, storage, and network needs.
- **Overprovisioning**
  - Azure may create extra instances during deployment to improve success rates, then remove the extras after provisioning.
  - This is useful for large deployments, but you should understand that the final capacity is the requested capacity.
- **Networking**
  - Instances can be placed behind a load balancer or integrated with other traffic distribution patterns.
  - You can also configure public IP behavior depending on the design.
- **Public IPs**
  - You can expose instances directly in some designs, but production patterns usually place them behind a load balancer or gateway.
- **Extensions**
  - VM extensions can install agents, configure software, or integrate with monitoring and security tooling.

A reliable Azure CLI example uses explicit image publisher, offer, and SKU values rather than ambiguous aliases:

```bash
az vmss create \
  --resource-group rg-app \
  --name vmss-web \
  --image Canonical:0001-com-ubuntu-server-jammy:22_04-lts-gen2:latest \
  --vm-sku Standard_DS2_v2 \
  --instance-count 2 \
  --admin-username azureuser \
  --generate-ssh-keys \
  --upgrade-policy-mode automatic
```

**Networking and traffic distribution**

VMSS commonly sits behind Azure Load Balancer or Application Gateway, depending on the traffic pattern.

**Azure Load Balancer integration**

Load Balancer is the classic front end for VMSS.

- Use a **backend pool** to group VMSS instances.
- Use a **health probe** to check whether instances are responding.
- Use **load-balancing rules** to map frontend ports to backend ports.
- Use **inbound NAT rules** when you need direct RDP or SSH access to specific instances.

Example topology:

```text
Internet
   |
Public IP
   |
Azure Load Balancer
   |
Backend pool -> VMSS instances
   |
App tier
```

Key exam points:

- A backend pool can target the scale set rather than individual VMs.
- Health probes help the load balancer avoid sending traffic to unhealthy instances.
- NAT rules are useful for administrative access to specific instances without exposing all instances publicly.

**Application Gateway integration**

Application Gateway is useful when you need Layer 7 routing.

- It can route based on host name or URL path.
- It can terminate TLS.
- It can provide web application firewall capabilities when configured with WAF.
- It is often used for web apps that need smarter HTTP routing than a Layer 4 load balancer.

**Public access patterns**

You should understand the difference between direct exposure and controlled exposure.

- Direct public IP per instance is possible in some designs, but it increases management and security complexity.
- A front-end service such as Load Balancer or Application Gateway is usually preferred.
- For admin access, use jump hosts, Bastion, or inbound NAT rules rather than broad public exposure.

**Health, repair, and instance replacement**

Health in VMSS is not just about whether a VM exists. Azure can use health signals to decide whether an instance should remain in service or be repaired.

Important health concepts:

- **Platform health**: Azure infrastructure state and instance status.
- **Load balancer health probe**: whether the instance responds to the configured probe.
- **Application health**: whether the app itself is healthy, often exposed through a custom endpoint.
- **Automatic instance repair**: Azure can replace or reimage unhealthy instances based on configured health signals and repair policy.

Health probes do not directly control autoscale in the normal case, but they do matter for traffic routing and for repair workflows. In practice, a healthy probe helps keep traffic flowing, while unhealthy instances may be removed from service or repaired depending on configuration.

Example health probe endpoint:

```bash
curl -i http://10.0.0.4:8080/health
```

A typical app should return a simple success response such as `200 OK` when it is ready to receive traffic.

**Upgrade policy and model changes**

When you change the VMSS model, Azure must decide how to apply that change to existing instances. This is the role of the upgrade policy.

**Manual upgrades**

- Azure does not automatically apply model changes to existing instances.
- You choose when to update instances.
- This gives you the most control, but it requires more operational effort.

**Automatic upgrades**

- Azure applies changes to instances automatically.
- This reduces manual work.
- It can be useful when you want the fleet to converge quickly to the latest model.

**Rolling upgrades**

- Azure updates instances in batches rather than all at once.
- This reduces blast radius.
- It is useful when you want to preserve availability during changes.

Rolling behavior is especially important in production because it helps avoid taking the entire fleet down at once. In practice, Azure may use upgrade domains or health-based checks depending on orchestration mode and configuration. The exact behavior is not identical across all VMSS modes, so do not assume a single domain model applies everywhere.

Practical considerations:

- Test model changes in a nonproduction environment first.
- Use health checks to reduce the chance of rolling out a bad configuration.
- Keep application startup and readiness checks fast and reliable.

**Autoscale fundamentals**

Azure Monitor autoscale changes VMSS capacity based on rules and schedules. It is one of the most important AZ-104 topics in this area.

Autoscale is driven by:

- **Metric rules**: scale based on CPU, memory, queue length, or custom metrics.
- **Schedule rules**: scale at specific times or days.
- **Profiles**: sets of rules and capacity settings for different time windows.
- **Default profile**: the fallback profile when no other profile matches.

Autoscale is not limited to CPU. You can use Azure Monitor metrics, custom metrics, and application telemetry if you publish it correctly.

Common metric sources:

- Azure platform metrics such as CPU percentage.
- Guest-level metrics when the monitoring agent or diagnostic pipeline is configured.
- Application metrics exposed through Azure Monitor custom metrics.
- Queue length or backlog metrics from services such as Storage queues or Service Bus, when your design uses them as scaling signals.

**Autoscale rule logic**

Autoscale rules compare a metric to a threshold over a time window.

Typical rule components:

- **Metric namespace** and metric name
- **Condition** such as greater than or less than
- **Threshold**
- **Aggregation** such as average, minimum, maximum, or total
- **Time grain** and **time window**
- **Action** to scale out or scale in
- **Cooldown** to prevent rapid repeated changes

You can define multiple rules in a profile. Azure evaluates them and applies the resulting action according to the autoscale engine’s logic. If both scale-out and scale-in conditions are met, the service uses the rule evaluation and profile behavior to determine the final action, so you should design thresholds with enough separation to avoid oscillation.

Example conceptual autoscale profile:

```json
{
  "name": "weekday-profile",
  "capacity": {
    "minimum": "2",
    "default": "2",
    "maximum": "10"
  },
  "rules": [
    {
      "metric": "Percentage CPU",
      "operator": "GreaterThan",
      "threshold": 70,
      "direction": "Increase",
      "changeCount": 2,
      "cooldown": "PT10M"
    },
    {
      "metric": "Percentage CPU",
      "operator": "LessThan",
      "threshold": 30,
      "direction": "Decrease",
      "changeCount": 1,
      "cooldown": "PT10M"
    }
  ]
}
```

This is conceptual, not a literal ARM template. In real Azure Monitor autoscale, the schema uses specific ARM fields and nested profile/rule structures.

**Autoscale schedules and profiles**

Schedules let you pre-scale for predictable demand.

- Use a weekday profile for business hours.
- Use a weekend profile for lower demand.
- Use a default profile for all other times.

Practical behavior:

- Profiles can have different minimum, maximum, and default capacities.
- Time-based profiles are useful when traffic patterns are predictable.
- Metric-based rules are better when demand is variable or event-driven.

Example schedule idea:

- Weekdays 08:00–18:00: minimum 4, default 6, maximum 12
- Nights and weekends: minimum 2, default 2, maximum 6

**Autoscale with CLI**

You can create or update autoscale settings with Azure CLI. A common pattern is to attach autoscale to a VMSS resource ID.

```bash
az monitor autoscale create \
  --resource-group rg-app \
  --resource vmss-web \
  --resource-type Microsoft.Compute/virtualMachineScaleSets \
  --name vmss-web-autoscale \
  --min-count 2 \
  --max-count 10 \
  --count 2
```

Add a rule based on CPU:

```bash
az monitor autoscale rule create \
  --resource-group rg-app \
  --autoscale-name vmss-web-autoscale \
  --condition "Percentage CPU > 70 avg 5m" \
  --scale out 2
```

Add a scale-in rule:

```bash
az monitor autoscale rule create \
  --resource-group rg-app \
  --autoscale-name vmss-web-autoscale \
  --condition "Percentage CPU < 30 avg 5m" \
  --scale in 1
```

**Autoscale troubleshooting**

When autoscale does not behave as expected, check the following:

- The VMSS is connected to the correct autoscale setting.
- The metric exists and is being emitted.
- The threshold is realistic for the workload.
- The cooldown period is not too long.
- The minimum and maximum capacity allow the desired action.
- The time window is long enough to avoid reacting to noise.
- The activity log shows whether a scale action was attempted or blocked.

Useful commands:

```bash
az monitor autoscale show \
  --resource-group rg-app \
  --name vmss-web-autoscale
```

```bash
az monitor activity-log list \
  --resource-group rg-app \
  --status Succeeded \
  --max-events 20
```

**Instance management operations**

VMSS supports operational actions on individual instances and on the model as a whole.

Common actions:

- **List instances** to inspect health and status.
- **Restart** an instance when troubleshooting.
- **Reimage** an instance to rebuild it from the model.
- **Delete** an instance to let the scale set replace it if capacity requires it.
- **Upgrade** instances after a model change.
- **Protect** instances from scale-in or scale-out in some scenarios.

Instance protection is important when you have a special node that should not be removed automatically. This is especially relevant in mixed or state-aware designs.

Example CLI commands:

```bash
az vmss list-instances \
  --resource-group rg-app \
  --name vmss-web \
  --output table
```

```bash
az vmss restart \
  --resource-group rg-app \
  --name vmss-web \
  --instance-ids 3
```

```bash
az vmss reimage \
  --resource-group rg-app \
  --name vmss-web \
  --instance-ids 3
```

**Monitoring and diagnostics**

Monitoring is essential for both scaling and troubleshooting.

What to monitor:

- **CPU percentage**
- **Network in/out**
- **Disk read/write**
- **Instance health**
- **Autoscale history**
- **Activity log**
- **Application-specific metrics**

Azure Monitor is the main source for platform metrics and autoscale history. The Activity Log helps you confirm whether Azure performed a scale action, changed a model, or encountered a control-plane issue.

Practical troubleshooting workflow:

1. Check whether the VMSS instance is healthy.
2. Review the autoscale setting and rule thresholds.
3. Inspect Azure Monitor metrics over the same time window.
4. Review the Activity Log for scale actions or failures.
5. Confirm the application is emitting the expected metric or health signal.
6. Validate networking, probe configuration, and backend pool membership.

Use commands that are explicit and current enough for exam study.

Azure CLI example for creating a scale set with a marketplace image:

```bash
az vmss create \
```


PowerShell cmdlets and parameters can vary by module version and orchestration mode, so use them as inspection and management examples rather than assuming a single creation pattern works everywhere.

**Common exam scenarios**

- You need to host a web app that scales out under load and stays consistent across instances.
- You need scheduled scaling for predictable business hours.
- You need health-based replacement for failed instances.
- You need a front end that distributes traffic across multiple VM instances.
- You need to choose between Uniform and Flexible orchestration based on management and placement requirements.
- You need to troubleshoot why autoscale did not trigger even though CPU was high.

**Key distinctions to remember**

| Topic | What to remember |
|---|---|
| VMSS purpose | Manage a fleet of similar VMs as one resource |
| Uniform mode | Best for standardized fleets and classic scale-set behavior |
| Flexible mode | More VM-like management and placement options |
| Autoscale | Uses Azure Monitor metrics and schedules to change capacity |
| Health probes | Control traffic routing and contribute to repair workflows |
| Upgrade policy | Determines how model changes reach existing instances |
| Load Balancer | Common Layer 4 front end for VMSS |
| Application Gateway | Common Layer 7 front end for web workloads |

**Practical design guidance**

- Use a load balancer or gateway instead of exposing every instance directly.
- Keep the application stateless where possible to simplify scaling.
- If state is required, externalize it to managed services such as databases, caches, or storage.
- Use health endpoints that reflect real readiness, not just process liveness.
- Set autoscale thresholds with enough gap to avoid rapid scale oscillation.
- Validate image, extension, and networking settings before enabling production autoscale.
