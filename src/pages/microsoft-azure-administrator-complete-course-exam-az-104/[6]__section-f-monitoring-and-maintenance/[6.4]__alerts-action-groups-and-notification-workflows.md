## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure Monitor alerting overview**

Azure Monitor alerting turns telemetry into actionable notifications. You define an alert rule against a **signal** such as metrics, logs, activity log events, or service health events, then attach an **action group** to decide what happens when the rule fires.

The core exam idea is that alerts are not just “notifications.” They are a pipeline:

- **Signal**: the data source being evaluated.
- **Condition**: the logic that decides whether the signal is abnormal.
- **Action group**: the response workflow.
- **Alert instance**: the fired occurrence that appears in alert history and can trigger actions.

Alerting is used to detect availability issues, performance degradation, configuration changes, and platform incidents. In practice, you often combine multiple alert types so that one issue can generate both an operational notification and an incident ticket.

**Alert rule components and terminology**

An Azure Monitor alert rule usually includes these parts:

- **Scope**: the resource, resource group, subscription, or workspace being monitored.
- **Signal type**: metric, log, activity log, or service health.
- **Condition**: the rule logic, such as threshold comparison or query result.
- **Threshold**: the value that triggers the alert.
- **Aggregation**: how data is summarized, such as `Average`, `Count`, `Maximum`, or `Minimum`.
- **Evaluation frequency**: how often Azure checks the rule.
- **Window size**: the time range used for each evaluation.
- **Dimensions**: metric filters that split data by properties such as instance, API name, or status code.

A practical way to think about it is:

- **Evaluation frequency** answers “how often do you check?”
- **Window size** answers “how much recent data do you inspect?”
- **Aggregation** answers “how do you summarize the data?”
- **Dimensions** answer “which slice of the metric do you care about?”

> Alert rules can be noisy if the window is too short or the threshold is too sensitive. Tune the evaluation frequency and window size together.

**Alert rule types you should know**

Azure Monitor supports several common alert patterns:

- **Metric alerts**
  - Best for near-real-time performance and availability signals.
  - Can use **static threshold** rules or **dynamic threshold** rules.
  - Can be single-resource or multi-resource depending on the metric and scope.
- **Log alerts**
  - Use a Kusto Query Language (<abbr title="Kusto Query Language">KQL</abbr>) query against Log Analytics or Azure Monitor Logs.
  - Useful for complex conditions, correlation, and multi-resource analysis.
  - Can be **multi-resource log alerts** when the query spans multiple resources in a workspace.
- **Activity log alerts**
  - Trigger on subscription-level control plane events such as create, delete, write, or failed operations.
  - Useful for governance and change tracking.
- **Service Health alerts**
  - Notify you about Azure platform incidents, planned maintenance, security advisories, and health advisories that affect your services.
- **Resource Health alerts**
  - Notify you when a specific Azure resource changes health state, such as unavailable, degraded, or unknown.

**Metric alerts in practice**

Metric alerts are common for VM, App Service, storage, and networking scenarios. They are usually the first choice when you want fast detection of performance or availability issues.

Typical metric alert design:

- **Signal**: CPU percentage, disk queue length, request count, latency, or availability.
- **Condition**: greater than, less than, or equal to a threshold.
- **Aggregation**: often `Average` or `Maximum`.
- **Dimensions**: optional filters for instance-level or endpoint-level targeting.

Examples of when to use metric alerts:

- VM CPU stays high for several minutes.
- Storage account availability drops below a target.
- Application Gateway backend response time increases.
- A load balancer backend health metric indicates failure.

Dynamic threshold alerts are useful when the “normal” value changes over time. They reduce manual threshold tuning, but they are not ideal for every workload. Static thresholds are simpler and more predictable when you already know the expected baseline.

**Log alerts and KQL**

Log alerts are powerful when you need richer logic than a metric can provide. You write a KQL query, then alert when the query returns results or crosses a numeric threshold.

Example KQL for failed requests in Application Insights:

```kusto
requests
| where success == false
| summarize FailedRequests = count() by bin(timestamp, 5m)
| where FailedRequests > 10
```

Use log alerts when you need to:

- Correlate multiple signals.
- Detect patterns across several resources.
- Alert on application-level conditions that are not exposed as metrics.
- Build multi-resource operational views from a workspace.

A common exam distinction:

- **Metric alerts** are faster and simpler.
- **Log alerts** are more flexible and query-driven.

**Scope and target selection**

Scope determines what the alert evaluates. This is a frequent source of confusion in Azure Monitor.

Common scope patterns:

- **Resource-level scope**
  - Targets one VM, one storage account, one App Service, or another single resource.
  - Best for resource-specific health and performance.
- **Resource group scope**
  - Useful when you want to monitor a set of related resources together.
  - Often used for governance or shared operational patterns.
- **Subscription scope**
  - Common for activity log alerts and governance-related monitoring.
  - Useful for broad operational visibility.
- **Workspace scope**
  - Used for log alerts and workspace-based monitoring.
  - Best when the data lives in Log Analytics.
- **Multi-resource scope**
  - Useful for standardized alerting across many similar resources.
  - Often paired with dimensions or KQL filters.

Practical guidance:

- Use the narrowest scope that still captures the problem you want to detect.
- Use dimensions or query filters to avoid alerting on irrelevant instances.
- For shared services, consider subscription or workspace scope to centralize operations.

**Action groups**

An **action group** is the response engine attached to an alert rule. It defines who gets notified and what automation runs when the alert fires.

Common action group actions:

- **Email/SMS/Push/Voice**
  - Sends human notifications.
- **Webhook**
  - Calls an external HTTP endpoint.
- **Azure Function**
  - Runs serverless code for custom automation.
- **Logic App**
  - Starts a workflow for routing, enrichment, or ticket creation.
- **Automation Runbook**
  - Starts a PowerShell or Python runbook in Azure Automation.
- **ITSM connector**
  - Creates or updates incidents in IT service management systems.
- **Event Hub**
  - Streams alert data to downstream systems for analytics or SIEM integration.

Action group design details:

- **Name** should be descriptive and consistent with your operations model.
- **Short name** is a compact identifier used in notifications and has length constraints.
- **Receivers** should be validated where possible so you do not discover bad contact data during an incident.
- **Common alert schema** should usually be enabled for webhook-based integrations unless a downstream system requires a custom payload.

> The common alert schema standardizes alert payloads across alert types, but downstream systems must still be tested because field availability can vary by alert source.

**Action group configuration considerations**

When you configure an action group, pay attention to these details:

- **Receiver validation**
  - Email receivers often require confirmation.
  - Webhook endpoints should be reachable and return a successful HTTP status.
- **Common alert schema**
  - Simplifies parsing across alert types.
  - Recommended for Logic Apps, Functions, and many webhook integrations.
- **Naming constraints**
  - Action group short names are limited in length.
  - Keep names short but meaningful for operational clarity.
- **Regional behavior**
  - Some action group components are global, but downstream automation may be region-specific.

A practical naming pattern:

- `ag-prod-ops`
- `ag-sec-incidents`
- `ag-platform-critical`

**Notification workflows and incident routing**

Alerting is most useful when it fits into a workflow, not just an inbox. A typical workflow looks like this:

1. Azure Monitor detects the condition.
2. The alert rule fires.
3. The action group sends notifications and triggers automation.
4. A workflow system enriches the event.
5. An incident is created, updated, or escalated.
6. The operations team acknowledges and resolves the issue.

Common workflow patterns:

- **Email-only**
  - Simple and fast for small environments.
  - Not ideal for large operations teams.
- **Email plus ticketing**
  - Good for auditability and assignment.
- **Webhook to Logic App**
  - Good for enrichment, routing, and conditional branching.
- **Runbook automation**
  - Good for remediation tasks such as restarting a service or scaling a resource.
- **ITSM integration**
  - Good for enterprise incident management and SLA tracking.

Example Logic App pattern:

- Trigger from an action group webhook.
- Parse the alert payload.
- Check severity and resource type.
- Create a ticket in ServiceNow, Jira, or another ITSM tool.
- Notify Teams or email the on-call team.

**Example webhook payload handling**

Webhook payloads vary by alert type and whether the common alert schema is enabled. Do not assume a fixed JSON shape for every alert.

A safer Logic App approach is to parse the incoming body dynamically and inspect the top-level fields first:

```json
{
  "schemaId": "azureMonitorCommonAlertSchema",
  "data": {
    "essentials": {
      "alertRule": "vm-high-cpu",
      "severity": "Sev2",
      "signalType": "Metric",
      "monitorCondition": "Fired"
    }
  }
}
```

Use this as a conceptual shape only. In real deployments, validate the exact payload from your alert source and integration type.

**Service Health vs Resource Health**

These two are often confused, but they solve different problems.

| Feature | Service Health | Resource Health |
|---|---|---|
| Scope | Azure service or region impact | Specific Azure resource |
| Purpose | Platform incidents, maintenance, advisories | Health state of one resource |
| Typical question | “Is Azure affecting my service?” | “Is my VM or database healthy?” |
| Best for | Broad platform awareness | Resource-specific troubleshooting |

Use **Service Health** when you need to know about Azure platform events that may affect many customers. Use **Resource Health** when you need to know whether a specific resource is currently available, degraded, or unavailable.

**Activity log alerts**

Activity log alerts monitor subscription-level control plane events. They are useful for governance, security, and change tracking.

Common use cases:

- Resource creation or deletion.
- Role assignment changes.
- Policy changes.
- Failed administrative operations.
- Service health events at the subscription level.

Important points:

- Activity log data is subscription-scoped.
- These alerts are not the same as metric alerts.
- They are often used to detect unauthorized or unexpected changes.

Example Azure CLI pattern for an activity log alert rule:

```bash
az monitor activity-log alert create \
  --name alert-rg-delete \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<subscription-id> \
  --condition category=Administrative \
  --action-group <action-group-resource-id>
```

**Metric alert example with Azure CLI**

A metric alert typically targets a resource and uses a metric condition. The exact syntax can vary by CLI version and resource type, so always verify against the current Azure CLI help.

Example pattern:

```bash
az monitor metrics alert create \
  --name vm-high-cpu \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<subscription-id>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm01 \
  --condition "avg Percentage CPU > 80" \
  --description "Alert when VM CPU exceeds 80 percent" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 2 \
  --action <action-group-resource-id>
```

Use this as a pattern, not a guaranteed copy-paste command. Metric names, dimensions, and supported operators depend on the resource provider and metric namespace.

**Log alert example with KQL and Azure CLI**

A log alert is usually created from a query and a threshold. The query should return a numeric result or a set of rows that the alert engine can evaluate.

Example pattern:

```bash
az monitor scheduled-query create \
  --name failed-requests-alert \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<subscription-id>/resourceGroups/rg-app/providers/Microsoft.Insights/components/appi01 \
  --condition "count 'requests | where success == false' > 10" \
  --description "Alert when failed requests exceed threshold" \
  --severity 2
```

Because CLI syntax changes over time, confirm the current command shape before using it in production. The exam focus is the concept: query, threshold, scope, and action group.

**Portal workflow for creating alerts**

A typical portal workflow is:

1. Open **Azure Monitor**.
2. Select **Alerts**.
3. Choose **Create** and pick the alert rule type.
4. Select the **scope**.
5. Define the **condition** and threshold.
6. Set **evaluation frequency** and **window size**.
7. Attach an **action group**.
8. Review and create the rule.

This workflow matters because the portal exposes the same concepts you must understand for CLI, PowerShell, and IaC deployments.

**PowerShell and automation considerations**

Azure PowerShell and ARM/Bicep are common for repeatable deployments. Prefer current Az modules and resource templates over older cmdlets when possible.

Example PowerShell pattern for creating an action group:

```powershell
$actionGroup = New-AzActionGroup `
  -ResourceGroupName "rg-monitoring" `
  -Name "ag-prod-ops" `
  -ShortName "prodops" `
  -Receiver @{
      Name = "OpsEmail"
      EmailAddress = "ops@example.com"
    }
```

Example Bicep for an action group:

```bicep
resource actionGroup 'Microsoft.Insights/actionGroups@2023-01-01' = {
  name: 'ag-prod-ops'
  location: 'global'
  properties: {
    groupShortName: 'prodops'
    enabled: true
    emailReceivers: [
      {
        name: 'OpsEmail'
        emailAddress: 'ops@example.com'
        useCommonAlertSchema: true
      }
    ]
  }
}
```

For end-to-end alerting, pair the action group with an alert rule resource in the same deployment or in a separate template.

**Alert lifecycle and management**

Alert rules are not static forever. You should manage them as living operational assets.

Key lifecycle actions:

- **Enable or disable** a rule during maintenance or testing.
- **Update** thresholds, scopes, or receivers as workloads change.
- **Delete** obsolete rules to reduce noise and cost.
- **Review alert history** to understand firing patterns.
- **Inspect alert instances** to see when and why a rule fired.

Operationally, you should review:

- Whether the rule is firing too often.
- Whether the threshold is too low or too high.
- Whether the scope includes resources that should not be monitored.
- Whether the action group still reaches the right people.

**Troubleshooting alert delivery**

When an alert fires but no one is notified, check the workflow from end to end.

Common failure points:

- **Wrong scope**
  - The rule monitors the wrong resource or subscription.
- **Disabled rule**
  - The alert exists but is not enabled.
- **Bad receiver data**
  - Email, webhook, or phone details are incorrect.
- **Webhook authentication failure**
  - The endpoint rejects the request or requires a token you did not provide.
- **Logic App or Function authorization**
  - Managed identity or access policy is missing.
- **Runbook permissions**
  - The automation account or runbook identity lacks rights to the target resource.
- **Endpoint timeout or non-2xx response**
  - The downstream service does not acknowledge the request correctly.
- **Common alert schema mismatch**
  - The integration expects a different payload shape.

Troubleshooting steps:

1. Confirm the alert rule fired in alert history.
2. Verify the action group is attached to the rule.
3. Check receiver validation and endpoint health.
4. Review downstream logs in Logic Apps, Functions, or Automation.
5. Confirm permissions for the identity used by the workflow.
6. Test the endpoint independently with a sample payload.

**Practical examples of integrations**

Common real-world integrations include:

- **Azure Monitor to Microsoft Teams**
  - Usually through Logic Apps or a webhook relay.
- **Azure Monitor to ServiceNow**
  - Often through ITSM connector or Logic App.
- **Azure Monitor to email distribution list**
  - Simple and effective for small teams.
- **Azure Monitor to runbook**
  - Useful for auto-remediation such as restarting a service.
- **Azure Monitor to Event Hub**
  - Useful for SIEM, analytics, or centralized event processing.

**VM Insights, Application Insights, and other monitoring sources**

You may encounter alerting on top of Azure Monitor integrations such as:

- **VM Insights**
  - Helps monitor guest OS performance and dependency data.
  - Often paired with metric and log alerts for VM health.
- **Application Insights**
  - Common for application performance, failures, and request latency.
  - Often uses log alerts and availability tests.
- **Container insights**
  - Useful for Kubernetes and container workloads.
  - Often alerts on node pressure, pod failures, or cluster health.

These integrations do not replace alert rules. They provide richer telemetry that alert rules can evaluate.

**Exam-focused distinctions**

- **Metric alert**
  - Best for fast, numeric resource telemetry.
- **Log alert**
  - Best for query-based, cross-resource, or application-level conditions.
- **Activity log alert**
  - Best for subscription events and governance.
- **Service Health alert**
  - Best for Azure platform incidents and advisories.
- **Resource Health alert**
  - Best for the health of a specific resource.
- **Action group**
  - Best for routing notifications and automation.
- **Common alert schema**
  - Best for standardized webhook processing.

**Quick reference checklist**

- Define the **signal** before choosing the alert type.
- Set the **scope** as narrowly as practical.
- Tune **threshold**, **aggregation**, **evaluation frequency**, and **window size** together.
- Use **dimensions** to reduce noise.
- Attach an **action group** that matches the operational workflow.
- Enable **common alert schema** for most webhook-based automations.
- Validate receivers and test the downstream workflow.
- Review alert history and retire noisy rules.
