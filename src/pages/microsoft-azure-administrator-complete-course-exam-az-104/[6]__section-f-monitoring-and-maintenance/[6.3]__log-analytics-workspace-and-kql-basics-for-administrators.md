## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Log Analytics workspace overview**

A **Log Analytics workspace** is the central Azure Monitor data store for logs, queries, alerts, and analysis. You use it to collect telemetry from Azure resources, virtual machines, on-premises servers, and other supported sources, then query that data with **Kusto Query Language (KQL)**.

For AZ-104, focus on the workspace as both:
- A **storage and query boundary** for log data
- A **management boundary** for access, retention, and alerting

A workspace is often the place where you verify whether data is arriving, whether a data source is configured correctly, and whether a query can support an alert or investigation.

**Accessing workspaces in the Azure portal**

You typically access a workspace from the Azure portal by searching for **Log Analytics workspaces** or by opening **Azure Monitor** and selecting **Logs**. From there, you can choose the target workspace and run queries against its tables.

Common portal paths:
- **Azure portal** > **Log Analytics workspaces** > select workspace
- **Azure portal** > **Monitor** > **Logs**
- **Azure portal** > resource blade > **Logs** for resource-context querying

When you open **Logs** from a resource, Azure can scope the query to that resource automatically. When you open the workspace directly, you query in **workspace context** and can target any table in that workspace.

**Workspace context vs resource context**

This distinction matters in exams and in day-to-day administration.

- **Workspace-context query**
  - Runs against the workspace as the primary scope
  - Best when you want to search across multiple resources that send data to the same workspace
  - Common for centralized operations, hunting, and cross-resource correlation

- **Resource-context query**
  - Starts from a specific Azure resource
  - Best when you want to inspect logs for one VM, app, or service
  - Useful when troubleshooting from the resource blade

A resource-context query may still read from the workspace behind the scenes, but the portal scopes the query to the selected resource and time range.

**Permissions and access control**

Workspace access is not the same as resource access. This is a common exam trap.

- **Workspace permissions**
  - Control who can query, manage, and configure the workspace
  - Often assigned through Azure role-based access control (**RBAC**) at the workspace scope or higher
  - Examples include roles that allow reading logs, contributing to workspace settings, or managing solutions

- **Resource permissions**
  - Control access to the Azure resource that emits logs
  - A user may have permission to view a VM or storage account but still lack permission to query the workspace that stores its logs
  - Conversely, a user may query the workspace without having full management rights on the source resource

In practice, you need both the right **source-side permissions** and the right **workspace-side permissions** for full troubleshooting workflows.

> If a user can open a resource blade but cannot see logs, check whether they have permission on the workspace and whether the resource is actually sending data there.

**Workspace-linked solutions and data sources**

A workspace can be connected to multiple **data sources** and **solutions**. These integrations determine what data lands in the workspace and how it is organized.

Common data sources include:
- **Azure resources** sending platform logs or diagnostic logs
- **Virtual machines** using the Azure Monitor agent
- **Windows Event Logs** and **Syslog**
- **Custom logs** and application telemetry
- **Activity logs** forwarded into the workspace

Common workspace-linked solutions include:
- Security and operations solutions that add dashboards, tables, or saved queries
- Monitoring packs or integrations that enrich the workspace with specialized data

For AZ-104, the key idea is that the workspace is not just a passive bucket. It is the destination for multiple ingestion paths, and each path affects table availability, schema, and retention.

**Retention, archive, and search**

Retention controls how long data stays queryable in the workspace. Azure Monitor supports table-level retention settings, and this is important when you need to balance cost and investigation depth.

Typical retention concepts:
- **Interactive retention**
  - Data remains readily queryable in the workspace
  - Used for active investigations and dashboards

- **Archive retention**
  - Older data can be moved to a lower-cost archive state, if configured for the table
  - Archived data is not used the same way as hot data and may require special search or restore workflows depending on the feature set available in your environment

- **Search capabilities**
  - You can search recent data directly with KQL
  - For older or archived data, availability and performance depend on the table configuration and Azure Monitor features enabled

Table-level retention is useful when different tables have different business value. For example, security logs may need longer retention than verbose application traces.

**Table-level retention configuration**

Retention is often configured per table rather than only at the workspace level. This lets you tune cost and compliance requirements more precisely.

Practical implications:
- High-volume tables may need shorter interactive retention
- Critical audit tables may need longer retention
- Archive settings can reduce cost while preserving historical data
- Retention changes do not instantly rewrite historical behavior; they affect future handling and lifecycle policies

When studying, remember that retention is a **table property** as well as a workspace concern.

**KQL fundamentals**

**Kusto Query Language (KQL)** is the query language used to analyze log data in Log Analytics. KQL is optimized for read-heavy analytics, filtering, shaping, and aggregating large datasets.

A basic query usually follows this pattern:
1. Choose a table
2. Filter rows
3. Project or extend columns
4. Aggregate or sort results

Example:
```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where ActivityStatusValue == "Failed"
| project TimeGenerated, Caller, OperationNameValue, ResourceGroup, ActivityStatusValue
| order by TimeGenerated desc
```

**Syntax essentials**

KQL is case-insensitive for keywords, but string comparisons and some operators can behave differently depending on the operator you choose. Be precise with column names and values, and do not assume every comparison is case-insensitive.

Important syntax ideas:
- Use `|` to pipe data from one operator to the next
- Use `//` for comments in query examples
- Use `==` for equality checks
- Use `=~` for case-insensitive string equality in many scenarios
- Use `has` for token-based matching
- Use `contains` for substring matching
- Use `in (...)` for membership checks

KQL also works with multiple data types:
- **string**
- **datetime**
- **int**
- **long**
- **real**
- **bool**
- **dynamic**

The **dynamic** type is especially important because many Azure logs store JSON-like payloads in dynamic fields. You often need to extract values before filtering or aggregating.

Example of parsing a dynamic field:
```kusto
AzureDiagnostics
| where TimeGenerated > ago(1h)
| extend props = parse_json(Properties)
| extend status = tostring(props.status)
| project TimeGenerated, Resource, status, props
```

**Nulls and type conversion**

Log data is often messy. Fields may be missing, empty, or stored as the wrong type. Use conversion functions to avoid query failures and to make comparisons reliable.

Common conversion functions:
- `tostring()` for string conversion
- `toint()` for integer conversion
- `toreal()` for decimal values
- `todatetime()` for date/time conversion
- `tobool()` for boolean conversion

Example:
```kusto
Heartbeat
| extend ComputerName = tostring(Computer)
| extend CpuPercent = toint(CounterValue)
| where isnotnull(ComputerName)
| where CpuPercent > 80
```

Useful null-handling functions and checks:
- `isnull()`
- `isnotnull()`
- `coalesce()` to choose the first non-null value

Example:
```kusto
AzureActivity
| extend Actor = coalesce(Caller, tostring(Claims))
| where isnotnull(Actor)
```

**Core operators you should know**

These operators appear frequently in exam-style questions and real troubleshooting.

- **`where`**
  - Filters rows
  - Use it early to reduce data volume

- **`project`**
  - Selects and renames columns
  - Use it to simplify output

- **`extend`**
  - Adds calculated columns without removing existing ones
  - Useful for parsing, conversions, and derived values

- **`summarize`**
  - Aggregates data into grouped results
  - Essential for counts, trends, and top-N analysis

- **`join`**
  - Correlates rows from two tables or two filtered datasets
  - Useful for linking activity with resource inventory or alerts with events

- **`sort` / `order by`**
  - Sorts results
  - `order by` is commonly used in KQL examples

- **`take`**
  - Returns a limited number of rows without guaranteeing sort order
  - Useful for quick sampling

- **`limit`**
  - Similar to `take` in many practical uses
  - Often used to cap result size during exploration

A practical distinction:
- Use `take` or `limit` when you want a quick sample
- Use `order by ... desc` plus `take` when you want the newest records
- Use `summarize` when you need grouped totals

**Aggregation patterns**

`count()` is only the starting point. AZ-104-style analysis often requires more than a raw row count.

Common aggregation patterns:
- `count()` for total rows
- `countif()` for conditional counts
- `sum()` for totals
- `avg()` for averages
- `min()` and `max()` for ranges
- `dcount()` for approximate distinct counts
- `make_list()` or `make_set()` for collecting values
- `arg_max()` for the latest row per group

Example:
```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| summarize
    Total=count(),
    Failed=countif(ActivityStatusValue == "Failed"),
    Succeeded=countif(ActivityStatusValue == "Succeeded")
  by ResourceGroup
| order by Failed desc
```

Example of latest event per computer:
```kusto
Heartbeat
| summarize arg_max(TimeGenerated, *) by Computer
| project Computer, TimeGenerated, OSType, Version
```

**Joining and correlating data**

A **join** is one of the most useful KQL techniques for administrators because it lets you correlate related records across tables.

Common join use cases:
- Match failed sign-ins with audit events
- Correlate VM heartbeat data with performance counters
- Link resource changes to activity log entries

Example:
```kusto
let FailedOps =
    AzureActivity
    | where TimeGenerated > ago(24h)
    | where ActivityStatusValue == "Failed"
    | project CorrelationId, TimeGenerated, Caller, OperationNameValue;
let RelatedEvents =
    AzureActivity
    | where TimeGenerated > ago(24h)
    | project CorrelationId, ResourceGroup, ResourceProviderValue;
FailedOps
| join kind=leftouter RelatedEvents on CorrelationId
| project TimeGenerated, Caller, OperationNameValue, ResourceGroup, ResourceProviderValue
| order by TimeGenerated desc
```

Join guidance:
- Use `inner` when you only want matching rows
- Use `leftouter` when you want all rows from the left side plus matches from the right
- Keep the join key small and well-defined to avoid noisy results

**Query scope and time range**

KQL results depend heavily on the selected scope and time range.

- **Scope**
  - Workspace scope searches the workspace tables
  - Resource scope narrows the query to a specific resource
  - Cross-resource queries may require explicit workspace selection or broader scope configuration

- **Time range**
  - Portal time pickers can limit results even if the query itself is broader
  - `ago()` is the most common way to define relative time windows in KQL

Example:
```kusto
SecurityEvent
| where TimeGenerated between (ago(7d) .. now())
| summarize Events=count() by EventID
```

**Practical query examples**

Use these patterns to verify data arrival and troubleshoot ingestion.

Check whether a table has recent data:
```kusto
Heartbeat
| where TimeGenerated > ago(15m)
| summarize LastSeen=max(TimeGenerated), Computers=dcount(Computer)
```

Find recent failures in Azure activity:
```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where ActivityStatusValue != "Succeeded"
| project TimeGenerated, Caller, OperationNameValue, ActivityStatusValue, ResourceGroup
| order by TimeGenerated desc
```

Inspect raw records when schema is unclear:
```kusto
AzureDiagnostics
| where TimeGenerated > ago(1h)
| take 20
```

Search for a specific resource name:
```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where Resource has "myvm01"
| project TimeGenerated, OperationNameValue, Resource, Caller
```

**Alerting with log queries**

Log alerts use KQL to evaluate log data on a schedule. They are different from **metric alerts**, which evaluate numeric platform metrics.

Key log alert concepts:
- **Log alert rule**
  - Runs a query on a schedule
  - Triggers when the query returns matching results or crosses a threshold

- **Evaluation frequency**
  - How often Azure runs the query
  - Commonly every few minutes, depending on the rule configuration

- **Lookback window**
  - The time range the query evaluates each time it runs
  - Must be long enough to catch the event but short enough to avoid duplicate noise

- **Threshold**
  - The condition that determines whether the alert fires
  - Can be based on row count or aggregated values

Log alerts are useful when the signal exists only in logs, such as failed administrative operations or application-specific errors. Metric alerts are better when Azure already exposes a native metric like CPU percentage or disk queue length.

**Troubleshooting workspace ingestion**

When data does not appear, check the pipeline from source to workspace.

Common issues:
- **Ingestion latency**
  - Data may take time to arrive
  - Recent events may not appear immediately

- **Data collection failures**
  - Agent misconfiguration
  - Diagnostic settings not enabled
  - Missing permissions on the source or destination

- **Schema mismatch**
  - Data may land in a different table than expected
  - Custom logs may use different column names or dynamic fields

- **Workspace access issues**
  - You may be querying the wrong workspace
  - You may have resource access but not workspace query rights

Verification steps:
1. Confirm the source is configured to send data to the workspace.
2. Query the expected table with a broad time window.
3. Use `take` to inspect raw rows.
4. Check whether the portal scope is workspace-context or resource-context.
5. Review agent health or diagnostic settings if no rows appear.

Example health check query:
```kusto
Heartbeat
| where TimeGenerated > ago(1h)
| summarize LastHeartbeat=max(TimeGenerated), Machines=dcount(Computer)
```

**Workspace administration tips**

A workspace is easier to manage when you standardize naming, access, and retention.

Good practices:
- Keep workspaces aligned to operational boundaries, such as environment or business unit
- Use RBAC to separate readers, contributors, and administrators
- Review table retention regularly to control cost
- Use saved queries for repeatable investigations
- Validate alert queries against real data before enabling production alerts

**Exam-oriented takeaways**

For AZ-104, remember these high-value points:
- You access logs through **Azure Monitor**, **Log Analytics workspaces**, or a **resource blade**
- Workspace permissions and resource permissions are related but not the same
- KQL commonly uses `where`, `project`, `extend`, `summarize`, `join`, and `order by`
- `take` and `limit` are for sampling, not deterministic sorting
- Use conversion functions like `tostring()` and `toint()` when fields are dynamic or inconsistent
- Log alerts evaluate queries on a schedule and differ from metric alerts
- Troubleshooting often starts by confirming the correct workspace, scope, and time range

**Quick reference**

| Concept | What it does | Why it matters |
|---|---|---|
| Workspace | Stores and queries log data | Central place for monitoring and investigation |
| Resource context | Queries from a specific Azure resource | Faster troubleshooting for one service |
| Workspace context | Queries across the workspace | Better for centralized analysis |
| `extend` | Adds calculated columns | Helps parse and normalize data |
| `summarize` | Aggregates rows | Essential for trends and counts |
| `join` | Correlates datasets | Useful for root-cause analysis |
| Log alert | Query-based alert | Detects conditions only visible in logs |
| Metric alert | Metric-based alert | Better for native Azure metrics |
