## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure monitoring fundamentals**

Azure monitoring is built around three core data types: **metrics**, **logs**, and **alerts**. Metrics are near real-time numeric values, logs are detailed records stored in Log Analytics or other destinations, and alerts turn those signals into action. For AZ-104, you need to know how to choose the right signal for the right problem.

- **Metrics** are best for fast detection of performance or availability changes.
- **Logs** are best for investigation, correlation, and historical analysis.
- **Alerts** notify you when a condition is met and can trigger action groups.

Azure Monitor is the umbrella service. It includes:
- **Metrics Explorer** for charting platform metrics.
- **Log Analytics** for Kusto Query Language (**KQL**) queries.
- **Alerts** for proactive notification and automation.
- **Workbooks** for dashboards and operational views.
- **Application Insights** for application telemetry, especially for App Service and custom apps.

A practical rule is:
- Use metrics to ask, “Is something wrong now?”
- Use logs to ask, “Why did it happen?”
- Use alerts to ask, “Who should be notified, and what should run next?”

**Alerting model and Azure Monitor Alerts v2**

Azure Monitor alerts v2 is the current alerting model used for most modern alert rules. It supports:
- **Scope**: the resource or resource group the rule evaluates.
- **Condition**: the metric or log query that defines the trigger.
- **Severity**: a numeric priority from 0 to 4, where 0 is the most critical.
- **Dimensions**: filters that split a metric by instance, API name, status code, or other attributes.
- **Evaluation frequency** and **window size**: how often Azure checks and over what time range.

This matters because many exam scenarios are really about choosing the correct scope and signal:
- A VM CPU alert usually scopes to the VM or VM scale set instance.
- A storage latency alert usually scopes to the storage account and the correct service namespace.
- A web app availability alert may scope to the App Service resource or an Application Insights test.

Common alert types:
- **Metric alerts** for platform metrics such as CPU, disk, or request count.
- **Log alerts** for KQL-based conditions from Log Analytics.
- **Activity Log alerts** for subscription-level events such as resource creation, deletion, or service health events.
- **Smart detection** and application alerts for application telemetry, where available.

> Alert queries must return data in the way the alert engine expects. A KQL query that is useful for analysis is not always valid as an alert rule without proper aggregation and time filtering.

**Activity Log, Service Health, and Resource Health**

These are often confused, but they serve different purposes:
- **Activity Log** records control-plane events in Azure, such as create, update, delete, and policy actions.
- **Service Health** reports Azure platform incidents, planned maintenance, and health advisories that affect your services.
- **Resource Health** shows the health state of a specific resource, such as Available, Unavailable, or Degraded.

Use them together:
- Use **Activity Log alerts** when you need to know about administrative changes.
- Use **Service Health alerts** when Azure has a regional or service-wide issue.
- Use **Resource Health** when you need to confirm whether a specific resource is impacted.

A common operational pattern is:
1. Check **Resource Health** for the affected resource.
2. Check **Activity Log** for recent changes.
3. Check **Service Health** for platform incidents.
4. Drill into metrics and logs for root cause.

**Virtual machine monitoring**

VM monitoring is a mix of Azure platform metrics, guest OS counters, and log-based troubleshooting. For AZ-104, you should understand the difference between what Azure sees from the host and what the guest operating system sees inside the VM.

Platform-side VM metrics commonly include:
- **Percentage CPU**
- **Network In Total**
- **Network Out Total**
- **Disk Read Bytes**
- **Disk Write Bytes**
- **Disk Read Operations/Sec**
- **Disk Write Operations/Sec**

Guest OS monitoring often includes:
- **Memory Available Bytes**
- **Pages/sec**
- **Processor queue length**
- **Logical disk free space**
- **Disk queue length**
- **Disk latency**
- **TCP connections** or service-specific counters

These guest counters are important because Azure platform metrics do not always show memory pressure or inside-the-OS bottlenecks. A VM can have low CPU but still be slow because of memory exhaustion, disk contention, or application-level issues.

**VM Insights and Log Analytics**

**VM Insights** is a monitoring feature that uses Azure Monitor and Log Analytics to collect performance and dependency data from VMs. It helps you:
- View CPU, memory, disk, and network trends.
- Inspect top processes.
- Correlate performance with guest-level telemetry.
- Troubleshoot dependencies and application response patterns.

Typical data sources include:
- **Perf** table for performance counters.
- **Heartbeat** table to confirm agent connectivity.
- **InsightsMetrics** for VM Insights metrics.
- **VMConnection** and related tables for dependency visibility, depending on configuration.

A simple KQL example for VM heartbeat validation:
```kusto
Heartbeat
| summarize LastSeen=max(TimeGenerated) by Computer
| where LastSeen < ago(5m)
```

For alerting, you usually refine the query so it returns rows only when a problem exists within the evaluation window. For example, you might alert when a machine has not sent a heartbeat in the last 5 to 10 minutes, but the exact query structure should match the alert rule’s evaluation settings.

A useful operational workflow:
1. Check the VM’s **Overview** and **Metrics** blade for platform symptoms.
2. Open **Insights** to see guest-level performance.
3. Query **Heartbeat** to confirm the agent is alive.
4. Use **Run Command** or remote access to inspect the guest OS if needed.

**VM power state and instance view**

Azure VM power state is best confirmed through **instance view**, because it reflects runtime status such as running, stopped, deallocated, or starting. The standard `az vm show` command returns resource properties and provisioning state, but runtime power state is more reliably obtained from instance view.

Examples:
```bash
az vm get-instance-view -g rg-demo -n vm01 --query instanceView.statuses
```

```bash
az vm show -g rg-demo -n vm01 --show-details --query "{provisioningState:provisioningState,powerState:powerState}"
```

Use instance view when you need the current operational state:
- **Running**
- **Stopped**
- **Deallocated**
- **Starting**
- **Stopping**

This distinction matters in troubleshooting because a VM can be provisioned successfully but still not be running.

**Storage monitoring**

Storage monitoring is service-specific. The exact metrics and diagnostic categories depend on the storage service and the diagnostic mode you enable. Do not assume every storage account exposes the same logs or metrics in the same way.

Common monitoring areas include:
- **Availability**
- **Transaction rate**
- **Ingress and egress**
- **Latency**
- **Throttling**
- **Authentication failures**
- **Capacity usage**

For Azure Storage, you often monitor at the account level and then narrow to the service:
- **Blob**
- **File**
- **Queue**
- **Table**

Platform metrics vary by service and namespace, so always verify the metric list for the specific storage resource you are monitoring. Some metrics are only available for certain services or account types.

Typical operational uses:
- Detect spikes in request volume.
- Identify throttling or server-side latency.
- Track capacity growth.
- Investigate authorization failures or network access issues.

A practical CLI example to inspect available metrics:
```bash
az monitor metrics list-definitions \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<accountName>
```

And to query a metric that exists for the resource:
```bash
az monitor metrics list \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<accountName> \
  --metric "<metric-name>"
```

Because metric names are resource-specific, always confirm the exact name before building an alert.

**Storage diagnostic settings**

Diagnostic settings send logs and metrics to destinations such as:
- **Log Analytics workspace**
- **Storage account**
- **Event Hub**
- **Partner solution**

For storage accounts, diagnostic categories depend on the service and the current Azure Monitor schema. In practice, you should:
- Open the storage account in the portal.
- Go to **Diagnostic settings**.
- Review the available categories for that resource.
- Send logs to Log Analytics for query and alerting.

A common setup pattern:
1. Open the storage account.
2. Select **Diagnostic settings**.
3. Add a setting and choose a destination.
4. Enable the relevant logs and metrics.
5. Save and validate ingestion in Log Analytics.

Example portal outcome:
- Storage service logs flow into a workspace.
- Metrics remain available in Metrics Explorer.
- Alerts can be built from either metrics or KQL queries.

**Storage troubleshooting examples**

When storage is slow or failing, look for:
- Increased latency.
- Throttled requests.
- Authentication failures.
- Network restrictions such as firewall rules or private endpoint issues.
- Client-side retry storms.

A useful KQL pattern in Log Analytics is to summarize request outcomes by status code or operation, but the exact table and schema depend on the diagnostic configuration. For exam purposes, focus on the workflow:
- Identify the affected storage service.
- Check metrics for latency and request volume.
- Check logs for failures and authorization issues.
- Validate network access and identity permissions.

**Networking monitoring and troubleshooting**

Networking monitoring is a major AZ-104 operational area. Azure provides **Network Watcher** and related tools to help you understand connectivity, routing, packet flow, and security rule behavior.

Key Network Watcher capabilities:
- **Connection Monitor** for end-to-end connectivity checks.
- **IP flow verify** to test whether traffic is allowed or denied by NSGs.
- **Next hop** to see where traffic is routed.
- **Effective security rules** to inspect applied NSG rules.
- **Packet capture** for low-level traffic analysis.
- **NSG flow logs** for traffic logging at the network security group level.
- **Topology** for visualizing network relationships in some scenarios.

**Connection Monitor**

Connection Monitor helps you test connectivity between a source and a destination over time. It is useful for:
- VM-to-VM connectivity.
- VM-to-endpoint connectivity.
- Detecting packet loss, latency, and reachability issues.

Use it when you need continuous monitoring rather than a one-time test. It is especially helpful for hybrid or multi-tier applications.

**NSG flow logs**

NSG flow logs record traffic decisions at the network security group level. They help you answer:
- Which flows were allowed or denied?
- Which source and destination IPs were involved?
- Which ports and protocols were used?

Use flow logs when you need evidence of traffic patterns or security rule behavior. They are often paired with traffic analytics or Log Analytics for analysis.

**Packet capture**

Packet capture is useful when you need deeper inspection of traffic on a VM NIC. It helps diagnose:
- TCP handshake failures.
- Retransmissions.
- Application timeouts.
- Unexpected resets or drops.

Use it sparingly because it can generate large data volumes and operational overhead.

**Effective troubleshooting workflow for networking**

1. Confirm the symptom: unreachable host, slow app, intermittent timeout, or denied traffic.
2. Check **NSG rules** and **effective security rules**.
3. Use **IP flow verify** to test the exact 5-tuple path.
4. Check **next hop** to confirm routing.
5. Use **Connection Monitor** for ongoing reachability.
6. Use **packet capture** if you need packet-level evidence.
7. Review **NSG flow logs** for historical traffic patterns.

A simple CLI example for Network Watcher tools:
```bash
az network watcher show -g NetworkWatcherRG -n NetworkWatcher_eastus
```

```bash
az network watcher test-ip-flow \
  --resource-group rg-demo \
  --vm vm01 \
  --direction Inbound \
  --protocol TCP \
  --local 10.0.0.4:3389 \
  --remote 203.0.113.10:50000
```

```bash
az network watcher show-next-hop \
  --resource-group rg-demo \
  --vm vm01 \
  --source-ip 10.0.0.4 \
  --dest-ip 8.8.8.8
```

**App Service monitoring**

App Service monitoring combines platform metrics, logs, and application telemetry. The exact diagnostic categories and available metrics vary by platform, plan, and feature set, so avoid assuming a single universal list.

Important distinctions:
- **App-level metrics** describe the web app behavior, such as requests, response times, and HTTP status codes.
- **Plan-level metrics** describe the App Service plan, such as CPU and memory usage for the hosting plan.
- **Deployment slot metrics** help you validate staging and production behavior separately.

Common operational signals include:
- Request count.
- Average response time.
- HTTP 4xx and 5xx rates.
- App restarts.
- Availability test results.
- Plan CPU and memory pressure.
- Instance count and autoscale behavior.

**Application Insights**

Application Insights is the preferred tool for application-level telemetry. It helps you monitor:
- Requests.
- Dependencies.
- Exceptions.
- Traces.
- Availability tests.
- Performance bottlenecks.

Use it when you need to understand user experience and application behavior, not just infrastructure health.

A typical App Service monitoring stack:
- **Metrics** for quick health checks.
- **Diagnostic logs** for platform and app logs.
- **Application Insights** for request tracing and exceptions.
- **Log Analytics** for cross-resource queries and alerts.

**Availability tests**

Availability tests check whether your app responds from one or more locations. They are useful for:
- Detecting outages from the user perspective.
- Validating DNS, TLS, and routing.
- Confirming that a deployment did not break public access.

Use availability tests when you need external reachability, not just internal health.

**Deployment slots monitoring**

Deployment slots are useful for safe releases. Monitor them separately because:
- Staging may behave differently from production.
- Slot-specific configuration can affect performance.
- Swap operations can introduce temporary changes in traffic or warm-up behavior.

Operationally, you should:
- Compare metrics between slots before swapping.
- Validate availability tests against the staging slot if appropriate.
- Check logs after a swap for errors or latency spikes.

**Autoscale monitoring**

Autoscale is often driven by metrics such as CPU, memory, or HTTP queue length, depending on the service and plan. For App Service, autoscale is usually configured at the plan level.

Monitor:
- Scale-out events.
- Scale-in events.
- Metric thresholds.
- Instance health after scaling.

A practical concern is that scaling the plan does not always fix application bottlenecks. If response time remains high after scale-out, investigate dependencies, code paths, or downstream services.

**App Service plan vs app-level metrics**

This distinction is important for AZ-104:
- The **App Service plan** is the compute container and is where CPU and memory pressure are often observed.
- The **web app** is the application running on the plan and is where request, response, and error metrics are typically observed.

If you see high CPU, confirm whether it is:
- A plan-level issue affecting all apps in the plan.
- A single app generating load.
- A dependency or code issue causing inefficient processing.

**Practical portal-based setup examples**

For a VM:
1. Open the VM.
2. Go to **Metrics** and chart CPU, network, or disk metrics.
3. Enable **Diagnostic settings** to send guest logs or platform logs to Log Analytics.
4. Open **Insights** to review performance and dependencies.
5. Create an alert rule for CPU or heartbeat loss.

For a storage account:
1. Open the storage account.
2. Review **Metrics** for request and latency trends.
3. Configure **Diagnostic settings** for the relevant storage service logs.
4. Send data to Log Analytics.
5. Build a log or metric alert for failures or throttling.

For a network issue:
1. Open **Network Watcher**.
2. Run **Connection Monitor** for continuous reachability.
3. Use **IP flow verify** to test NSG behavior.
4. Check **next hop** for routing.
5. Review **NSG flow logs** for historical traffic.

For an App Service:
1. Open the web app.
2. Review **Metrics** for requests, response time, and errors.
3. Enable **Application Insights**.
4. Configure **Diagnostic settings** if you need platform logs.
5. Add availability tests and alerts for failures or latency.

**KQL examples for operational monitoring**

Heartbeat presence:
```kusto
Heartbeat
| summarize LastSeen=max(TimeGenerated) by Computer
| where LastSeen < ago(10m)
```

VM performance trend:
```kusto
Perf
| where ObjectName == "Processor" and CounterName == "% Processor Time"
| summarize AvgCPU=avg(CounterValue) by bin(TimeGenerated, 5m), Computer
| order by TimeGenerated desc
```

App request failures, if Application Insights data is available:
```kusto
requests
| where success == false
| summarize FailedRequests=count() by bin(timestamp, 5m), cloud_RoleName
| order by timestamp desc
```

These examples are for investigation and dashboarding. For alerts, ensure the query returns a clear condition that matches the alert evaluation window.

**What to remember for AZ-104**

- **Metrics** are fast and numeric; **logs** are detailed and queryable.
- **Alerts v2** use scope, severity, dimensions, and evaluation windows.
- **Activity Log** tracks control-plane changes.
- **Service Health** tracks Azure platform incidents.
- **Resource Health** shows the state of a specific resource.
- **VM Insights** helps you see guest-level performance beyond platform metrics.
- **Storage monitoring** is service-specific; verify the exact metrics and diagnostic categories.
- **Network Watcher** is essential for connectivity, routing, and NSG troubleshooting.
- **App Service monitoring** should separate app-level behavior from plan-level resource usage.
- **Application Insights** is the best source for request, dependency, and exception telemetry.

**Exam-oriented troubleshooting mindset**

When a scenario appears, identify:
- The resource type.
- The symptom.
- The best signal: metric, log, alert, or health view.
- The correct scope for the alert or query.
- Whether the issue is platform, guest, application, or network related.

A strong answer usually follows this order:
1. Confirm the symptom with the simplest monitoring view.
2. Narrow the scope to the affected resource.
3. Use logs or deeper tools for root cause.
4. Set an alert so the issue is detected earlier next time.
