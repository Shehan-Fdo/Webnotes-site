## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure Monitor overview**

Azure Monitor is the central observability service in Azure. It collects, analyzes, and acts on telemetry from Azure resources, guest operating systems, applications, and custom sources. For AZ-104, the key distinction is between **metrics** and **logs**:

- **Metrics** are numeric time-series values optimized for near-real-time monitoring.
- **Logs** are records of events, traces, and detailed diagnostic data optimized for investigation and correlation.

Use metrics when you need fast detection of a condition, such as CPU spikes or queue depth. Use logs when you need context, root-cause analysis, or a record of what happened before and after an event.

**Metrics: what they are and when to use them**

Metrics are lightweight, structured measurements collected at regular intervals. They are ideal for alerting on current health and for dashboards that show trends over time.

Common characteristics:

- **Low latency**: metrics usually appear faster than logs.
- **Time-series**: each data point is associated with a timestamp.
- **Aggregated**: values are often summarized as average, minimum, maximum, total, or count.
- **Limited retention**: platform metric retention is shorter than log retention, so metrics are not a long-term audit store.

Use metrics for:

- CPU utilization on a virtual machine.
- Disk IOPS or latency.
- Network throughput.
- Storage account availability or transaction count.
- Service health indicators that need quick alerting.

Metric dimensions let you slice a metric by a property such as API name, response code, disk, or instance. Dimensions are important when one metric contains multiple categories of data and you need to isolate the problem.

Example: a storage account transaction metric may be split by response type so you can see whether failures are concentrated in `4xx` or `5xx` responses.

```bash
az monitor metrics list \
  --resource "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storageAccount>" \
  --metric Transactions \
  --interval PT1M \
  --aggregation Total
```

You can also query metrics in the portal by opening a resource, selecting **Metrics**, and choosing the metric namespace, metric name, aggregation, and dimension filters.

**Logs: what they are and when to use them**

Logs are detailed records that help you understand why something happened. They are better for troubleshooting, auditing, and correlation across services.

Important log categories for AZ-104:

- **Platform logs**: generated by Azure infrastructure and services.
  - **Activity Log**: subscription-level record of control-plane operations such as create, update, delete, and role assignment actions.
  - **Resource logs**: emitted by a specific Azure resource, such as a storage account, Key Vault, or network security group.
- **Guest OS logs**: generated inside the operating system of a VM, such as Windows Event Logs or Linux syslog.
- **Application logs**: emitted by an application, often through Application Insights or custom logging.
- **Diagnostic logs**: a common term for logs exported from Azure resources through diagnostic settings; in practice, this often refers to resource logs sent to a destination.

Logs are usually queried with Kusto Query Language (KQL), which lets you filter, summarize, join, and visualize data.

Example: query Azure Activity Log entries for failed operations.

```kusto
AzureActivity
| where ActivityStatusValue =~ "Failed"
| project TimeGenerated, Caller, OperationNameValue, ResourceGroup, ResourceProviderValue, ActivityStatusValue, CorrelationId
| order by TimeGenerated desc
```

> The Activity Log is not stored in Log Analytics by default. It is a platform log in Azure Monitor and can be exported to a Log Analytics workspace, storage account, or event hub through diagnostic settings or activity log export.

**Azure Activity Log: exam-relevant details**

The Activity Log is one of the most important logs for AZ-104 because it records management-plane operations at the subscription scope.

Use it to investigate:

- Who created or deleted a resource.
- Whether a role assignment succeeded.
- Why a deployment failed.
- Which policy or lock blocked a change.

Key points:

- It is subscription-scoped, not resource-scoped.
- It typically retains data for a limited period in the portal.
- Long-term retention requires export to another destination.
- It helps with governance and change tracking, not deep application troubleshooting.

Practical example: if a VM was deleted unexpectedly, check the Activity Log for `Delete` operations and the `Caller` field to identify the actor.

**Resource logs and diagnostic settings**

Resource logs come from individual Azure services and vary by resource type. A storage account, Key Vault, Application Gateway, or VM extension may each expose different categories.

Diagnostic settings are how you route logs and metrics from a resource to destinations such as:

- Log Analytics workspace
- Storage account
- Event Hub
- Partner solution

Typical workflow in the portal:

1. Open the Azure resource.
2. Select **Diagnostic settings**.
3. Add a diagnostic setting.
4. Choose the log categories and metric categories supported by that resource.
5. Select one or more destinations.
6. Save and verify that data starts flowing.

Practical verification steps:

- Check the diagnostic setting status in the portal.
- Query the destination workspace for expected tables or records.
- Confirm the resource actually emits the category you enabled.
- Allow time for ingestion delay before concluding that configuration failed.

Example CLI pattern for enabling diagnostic settings on a resource:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storageAccount>" \
  --workspace "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspace>" \
  --logs '[{"category":"StorageRead","enabled":true},{"category":"StorageWrite","enabled":true},{"category":"StorageDelete","enabled":true}]' \
  --metrics '[{"category":"Transaction","enabled":true}]'
```

> Available categories vary by service. Do not assume every resource supports the same log names or metric categories.

**Metric alerts vs log alerts**

Azure Monitor Alerts are a major AZ-104 topic. The two most important alert types are **metric alerts** and **log alerts**.

| Aspect | Metric alerts | Log alerts |
|---|---|---|
| Data source | Metrics | Log queries in Log Analytics |
| Latency | Faster | Slower, depends on ingestion and query time |
| Best for | Near-real-time threshold monitoring | Complex conditions, correlation, and historical analysis |
| Typical use | CPU > 80%, queue length, availability | Failed requests, specific error patterns, multi-step conditions |
| Query language | Metric conditions | KQL |

Use metric alerts when you need quick action on a numeric threshold. Use log alerts when you need to evaluate a query over records, such as “show all failed requests in the last 5 minutes” or “count errors by region.”

Example metric alert creation pattern:

```bash
az monitor metrics alert create \
  --name vm-high-cpu \
  --resource-group <rg> \
  --scopes "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/<vmName>" \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 2 \
  --description "Alert when VM CPU stays above 80 percent"
```

Example log alert query for failed requests in an Application Insights-connected application:

```kusto
requests
| where success == false
| summarize FailedRequests = count() by bin(TimeGenerated, 5m), cloud_RoleName
| order by TimeGenerated desc
```

> `requests` is an Application Insights table, not a universal Azure table. Use it only when the application telemetry is being sent to Application Insights or a compatible workspace.

**When to prefer each alert type**

Choose **metric alerts** when:

- You need fast detection.
- The condition is a simple threshold.
- The signal is already exposed as a metric.
- You want lower operational overhead.

Choose **log alerts** when:

- You need to count or filter specific events.
- You need to correlate multiple fields or tables.
- The condition depends on text, status codes, or joins.
- You need richer troubleshooting context.

For AZ-104, remember that metric alerts are usually the first choice for infrastructure health, while log alerts are better for detailed investigation and application-level conditions.

**Guest OS logs and VM troubleshooting**

Guest OS logs are collected from inside the virtual machine. They are useful when Azure platform metrics look healthy but the workload still fails.

Examples:

- Windows Event Viewer logs for service crashes or authentication failures.
- Linux `syslog` or `journalctl` output for daemon errors.
- Custom application logs written by your software.

Common troubleshooting pattern:

- Azure metric shows normal VM CPU.
- Application still times out.
- Guest OS logs reveal a service crash, disk issue, or dependency failure.

Useful commands inside a VM:

```bash
# Linux
journalctl -u nginx --since "1 hour ago"
tail -n 100 /var/log/syslog
df -h
```

```powershell
# Windows
Get-WinEvent -LogName System -MaxEvents 50
Get-WinEvent -LogName Application -MaxEvents 50
```

**Metrics, dimensions, and slicing data**

Metric dimensions let you break a metric into meaningful subsets. This is especially useful when one resource serves multiple workloads or endpoints.

Examples of dimension use:

- Split HTTP request metrics by response code.
- Filter storage transactions by API name.
- Separate VM disk metrics by disk name.
- Analyze network metrics by direction or interface.

Why dimensions matter:

- They help you identify whether the issue is isolated or widespread.
- They reduce noise by focusing on the failing subset.
- They make dashboards and alerts more precise.

Practical example: if a web app has high latency, you can filter request duration by endpoint to see whether `/login` is slower than `/home`.

**Retention, granularity, and limitations**

Azure telemetry is not uniform across all services. You should expect differences in what is available and how long it is retained.

Important limitations:

- Not all Azure services expose the same metrics.
- Not all services support the same diagnostic log categories.
- Metric granularity varies by metric and service.
- Retention differs between metrics, logs, and exported destinations.
- Some logs are only available after you enable diagnostic settings.

Exam-oriented reminders:

- Metrics are best for short-term operational monitoring.
- Logs are better for long-term analysis and forensics.
- Activity Log retention in the portal is limited unless you export it.
- Exporting logs early is important if you need compliance or historical troubleshooting.

**Practical use case: Storage account troubleshooting**

A common AZ-104 scenario is a storage account with intermittent upload failures.

Use metrics first:

- Check transaction count, availability, and latency.
- Look for spikes in `4xx` or `5xx` responses.
- Use dimensions to isolate the failing API or response type.

Then use logs:

- Enable diagnostic settings for the storage account.
- Send logs to a Log Analytics workspace.
- Query for failed operations and correlate with time, caller, and request type.

Example KQL pattern:

```kusto
StorageBlobLogs
| where StatusCode >= 400
| summarize Failures = count() by bin(TimeGenerated, 5m), StatusCode, OperationName
| order by TimeGenerated desc
```

If the storage service is not emitting that exact table name in your environment, use the workspace tables created by the diagnostic pipeline for that resource type. Table names vary by service and ingestion mode.

**Practical use case: Virtual machine performance issue**

A VM is reported as slow. Start with metrics:

- CPU percentage
- Available memory, if available through guest monitoring
- Disk read/write latency
- Network in/out

Then move to logs:

- Guest OS logs for service failures.
- Azure Activity Log for recent changes.
- Resource logs from dependent services, such as load balancers or network security groups, if enabled.

A useful troubleshooting sequence is:

1. Check the VM metric chart for spikes.
2. Check the Activity Log for recent changes.
3. Review guest OS logs for application or service errors.
4. Correlate the timing with any deployment or configuration change.

**Practical use case: Network and connectivity issues**

For network troubleshooting, metrics and logs work together.

Use metrics to identify:

- Packet drops
- Throughput changes
- Gateway or firewall health
- Load balancer backend health

Use logs to identify:

- NSG flow-related evidence if enabled through the appropriate service
- Firewall rule matches
- Connection failures
- Route or policy changes

Network Watcher is often used alongside Azure Monitor for deeper network diagnostics. It can help you validate connectivity, inspect effective routes, and troubleshoot security rules.

**Dashboards, workbooks, and visualization**

Azure Monitor Workbooks and dashboards help you combine metrics and logs into a single view.

Use them when you need:

- A shared operational view for a team.
- A troubleshooting workbook that mixes charts and KQL results.
- A dashboard for service health, latency, and error trends.

Workbooks are especially useful because they can:

- Query logs with KQL.
- Display metric charts.
- Add parameters for subscription, resource group, or time range.
- Support interactive troubleshooting.

Example workbook use case:

- Top chart: VM CPU and disk latency metrics.
- Bottom table: recent Activity Log changes.
- Side panel: failed requests from application logs.

**Portal workflow for practical monitoring**

A simple AZ-104-friendly workflow is:

1. Open the resource and inspect **Metrics** for fast signals.
2. Check **Activity Log** for recent control-plane changes.
3. Enable **Diagnostic settings** if deeper logs are needed.
4. Send logs to a **Log Analytics workspace** for querying.
5. Build a **Workbook** or dashboard for repeated investigations.
6. Create the right alert type based on the signal source.

This sequence helps you move from symptom detection to root cause without over-collecting data.

**Common exam traps**

- Confusing the Activity Log with resource logs.
- Assuming all logs are automatically in Log Analytics.
- Using logs when a simple metric alert is enough.
- Using metrics when the problem requires event-level detail.
- Assuming every Azure service exposes the same categories or tables.
- Forgetting that metric dimensions can narrow the problem significantly.
- Ignoring export and retention requirements for audit or compliance.

**Quick comparison summary**

- **Metrics**: fast, numeric, alert-friendly, best for health and thresholds.
- **Logs**: detailed, queryable, slower, best for investigation and correlation.
- **Activity Log**: subscription-level control-plane history.
- **Resource logs**: service-specific diagnostic data.
- **Guest OS logs**: inside the VM, useful for workload-level issues.
- **Workbooks**: combine metrics and logs for operational visibility.

**AZ-104 memory aid**

Think of the workflow as:

- **See it fast** with metrics.
- **Explain it deeply** with logs.
- **Track changes** with the Activity Log.
- **Export and retain** what you need for later.
- **Visualize together** with Workbooks or dashboards.
