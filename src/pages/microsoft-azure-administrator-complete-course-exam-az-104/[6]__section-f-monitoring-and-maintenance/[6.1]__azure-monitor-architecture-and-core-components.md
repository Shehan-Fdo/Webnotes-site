## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure Monitor overview**

Azure Monitor is the central observability platform in Azure. It collects, analyzes, and acts on telemetry from Azure resources, guest operating systems, applications, and custom sources. For AZ-104, you should understand that Azure Monitor is not a single feature; it is a platform made up of data collection, storage, analysis, visualization, and alerting components.

At a high level, Azure Monitor answers three questions:

- What is happening?
- Why is it happening?
- What should you do about it?

The platform uses multiple data types and tools, so exam questions often test whether you can distinguish metrics from logs, platform telemetry from guest telemetry, and monitoring views from alerting or health services.

**Core architecture**

Azure Monitor architecture is usually described in terms of three layers:

- **Monitoring data sources**: where telemetry originates.
- **Data platform**: where telemetry is stored and processed.
- **Consumption and action**: where you view, query, visualize, and respond to telemetry.

Common data sources include:

- Azure resources such as virtual machines, App Service, Storage accounts, SQL databases, and virtual networks.
- Guest operating systems inside VMs, including performance counters and event logs.
- Applications instrumented with Application Insights.
- Custom sources that send data through APIs, agents, or diagnostic settings.
- Azure platform services that emit resource logs and platform metrics.

The data platform is the set of services that receives and stores telemetry:

- **Azure Metrics** stores time-series numeric data optimized for near-real-time analysis.
- **Azure Monitor Logs** stores log and event data in a Log Analytics workspace using the Kusto Query Language (KQL).
- **Activity Log** stores subscription-level control-plane events such as create, update, delete, and policy actions.

Consumption tools include:

- Metrics Explorer
- Logs Explorer
- Workbooks
- Dashboards and portal views
- Alerts and action groups
- Service Health and Resource Health

**Monitoring data sources**

Azure Monitor collects telemetry from several distinct sources, and the source determines the data type and the best tool for analysis.

- **Platform metrics** are emitted by Azure resources automatically.
  - They are numeric and time-based.
  - They are useful for alerting, trend analysis, and quick health checks.
  - Examples include CPU percentage, request count, and disk I/O.
- **Guest metrics** come from inside the operating system or application environment.
  - They require an agent or extension in many scenarios.
  - They include memory usage, process counters, and custom performance counters.
  - They are often stored in Logs rather than native platform metrics.
- **Resource logs** describe operations performed by or on a resource.
  - They include diagnostic information, audit-style events, and service-specific logs.
  - They are commonly routed to Log Analytics, Storage, or Event Hubs.
- **Activity Log** records subscription-level events.
  - It is useful for governance, troubleshooting, and change tracking.
  - It is not the same as resource logs.
- **Application telemetry** comes from code instrumentation.
  - Application Insights captures requests, dependencies, exceptions, traces, and availability data.
  - In workspace-based mode, this telemetry is stored in Log Analytics.

A useful exam distinction is this:

- **Platform metrics** are generated by Azure and are best for fast numeric monitoring.
- **Guest metrics** come from the OS or app and often require agents or workspace ingestion.
- **Resource logs** are detailed event records and are usually queried in Logs Explorer.

**Metrics versus logs**

Metrics and logs solve different problems.

| Aspect | Metrics | Logs |
|---|---|---|
| Data shape | Numeric time series | Structured or semi-structured records |
| Latency | Near real time | Slightly higher latency |
| Best for | Alerting, dashboards, trend lines | Deep investigation, correlation, forensics |
| Query style | Filter, aggregate, chart | KQL queries |
| Storage | Azure Monitor metrics store | Log Analytics workspace |

Use metrics when you need quick operational visibility, such as CPU spikes or request failures. Use logs when you need context, such as which process failed, which user triggered an action, or which dependency caused an exception.

**Log Analytics workspace**

A Log Analytics workspace is the central repository for Azure Monitor Logs. It is where you query data using KQL and where many Azure services send diagnostic and operational logs.

Important workspace concepts:

- A workspace is a logical container for log data.
- Multiple resources can send logs to the same workspace.
- Workspace-based Application Insights stores application telemetry in the same workspace model.
- Retention and cost are important design considerations because log ingestion and storage are billed separately.

Operational considerations:

- **Retention** controls how long data remains available in the workspace.
- **Data export** can send selected tables or data streams to Storage, Event Hubs, or other destinations.
- **Cost** depends on ingestion volume, retention, and any additional export or query-related usage.
- **Workspace design** affects governance, access control, and cross-resource querying.

Example workspace creation with Azure CLI:

```bash
az monitor log-analytics workspace create \
  --resource-group rg-monitoring \
  --workspace-name law-prod-eastus \
  --location eastus
```

Example of querying logs with KQL:

```kusto
Heartbeat
| where TimeGenerated > ago(1h)
| summarize LastSeen=max(TimeGenerated) by Computer
| order by LastSeen desc
```

**Diagnostic settings**

Diagnostic settings control where a resource sends its logs and metrics.

You typically use diagnostic settings to route telemetry to:

- A Log Analytics workspace for analysis and alerting.
- A Storage account for long-term archival.
- An Event Hub for integration with external SIEM or analytics systems.
- Partner solutions, depending on the service.

Why diagnostic settings matter:

- They enable resource logs that are not available through platform metrics alone.
- They let you centralize telemetry across many resources.
- They are often required for compliance, auditing, and troubleshooting.

Practical example for a resource:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Web/sites/app1 \
  --workspace /subscriptions/<subId>/resourceGroups/rg-monitoring/providers/Microsoft.OperationalInsights/workspaces/law-prod-eastus \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true},{"category":"AppServiceConsoleLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

**Retention, export, and cost**

Retention and export are common operational topics in AZ-104-style questions.

- **Retention**
  - Shorter retention reduces storage cost.
  - Longer retention improves historical analysis and compliance support.
  - Some services have default retention behavior that you should verify.
- **Data export**
  - Use export when you need immutable archival, external analytics, or integration with another platform.
  - Export is not a replacement for querying in Log Analytics.
- **Cost**
  - Metrics are generally cheaper and lighter for simple monitoring.
  - Logs can become expensive if you ingest high-volume verbose data without filtering.
  - Diagnostic settings should be scoped carefully to avoid sending unnecessary categories.

A practical rule is to send only the telemetry you need, keep high-value logs in the workspace, and archive only what has a clear business or compliance purpose.

**Application Insights and workspace-based monitoring**

Application Insights is the application performance monitoring component of Azure Monitor. It helps you understand availability, performance, failures, and dependency behavior in your code.

Key Application Insights concepts:

- It captures request rates, response times, exceptions, dependencies, and custom events.
- It supports distributed tracing for multi-tier applications.
- It can be used for availability tests and live metrics.
- It integrates with Azure Monitor alerts and workbooks.

Workspace-based Application Insights is the modern model:

- Telemetry is stored in a Log Analytics workspace.
- You can query application data together with infrastructure logs.
- This improves correlation between app issues and platform issues.
- It simplifies governance because logs and app telemetry share the same workspace model.

Example KQL for Application Insights data in a workspace:

```kusto
requests
| where success == false
| summarize FailedRequests=count() by bin(TimeGenerated, 5m), cloud_RoleName
| order by TimeGenerated desc
```

This model is especially useful when you need to correlate a VM, App Service, or database issue with application failures in one place.

**Metrics Explorer and Logs Explorer**

The Azure portal provides two primary investigation tools:

- **Metrics Explorer**
  - Use it to chart platform metrics quickly.
  - It is ideal for near-real-time operational monitoring.
  - You can filter, split, aggregate, and pin charts to dashboards.
- **Logs Explorer**
  - Use it to query Log Analytics data with KQL.
  - It is ideal for troubleshooting, correlation, and historical analysis.
  - You can join tables, summarize data, and build reusable queries.

Practical difference:

- If you want to know whether CPU spiked in the last 15 minutes, use Metrics Explorer.
- If you want to know which process or request caused the spike, use Logs Explorer.

Example metric query concept in the portal:

- Select a resource.
- Open **Metrics**.
- Choose a metric such as `Percentage CPU`.
- Apply aggregation such as Average or Maximum.
- Split by dimension if needed.

Example log query concept:

```kusto
AzureDiagnostics
| where TimeGenerated > ago(24h)
| where Category == "AppServiceHTTPLogs"
| summarize Requests=count() by bin(TimeGenerated, 1h), ResultType
```

**Dashboards, portal views, and workbooks**

Students often confuse dashboards, portal views, and workbooks. They serve different purposes.

- **Portal views**
  - These are the built-in resource pages and blades in the Azure portal.
  - They are best for quick inspection of a single resource.
  - They often show metrics, logs, alerts, and health in context.
- **Dashboards**
  - These are pinned collections of charts, tiles, and links.
  - They are useful for a shared operational overview.
  - They are better for at-a-glance status than deep analysis.
- **Workbooks**
  - These are interactive, parameterized reports built from metrics, logs, and text.
  - They are best for investigation, operational reporting, and reusable views.
  - They can combine multiple data sources and include filters, charts, and markdown.

Practical guidance:

- Use dashboards for executive or operations summaries.
- Use workbooks for detailed operational analysis and repeatable troubleshooting.
- Use portal views when you are focused on one resource and need immediate context.

**Alerts and alert processing**

Azure Monitor alerts notify you when telemetry matches a condition. Alerts are a core operational control because they turn monitoring into action.

Alert types commonly tested in AZ-104:

- **Metric alerts**
  - Trigger on metric thresholds.
  - Best for fast response to numeric conditions.
- **Log alerts**
  - Trigger on KQL query results.
  - Best for complex conditions and cross-resource logic.
- **Activity Log alerts**
  - Trigger on subscription-level events.
  - Useful for governance and administrative changes.
- **Service Health alerts**
  - Notify you about Azure service incidents, planned maintenance, and advisories.
- **Resource Health alerts**
  - Notify you when a specific resource is unavailable or degraded.

Alert processing concepts:

- **Dynamic thresholds** use machine learning to learn normal behavior and reduce false positives.
- **Smart detection** is commonly associated with Application Insights and helps identify unusual failure patterns or performance anomalies.
- **Action groups** define what happens when an alert fires.
  - They can send email, SMS, push notifications, voice calls, or trigger automation.
  - They can also invoke webhooks, Azure Functions, Logic Apps, or Automation runbooks.

Example metric alert with Azure CLI:

```bash
az monitor metrics alert create \
  --name cpu-high-alert \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm1 \
  --condition "avg Percentage CPU > 80" \
  --description "Alert when VM CPU exceeds 80 percent" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --action-group /subscriptions/<subId>/resourceGroups/rg-monitoring/providers/microsoft.insights/actionGroups/ag-ops
```

Example action group creation:

```bash
az monitor action-group create \
  --name ag-ops \
  --resource-group rg-monitoring \
  --short-name ops \
  --action email admin team@example.com
```

**Service Health and Resource Health**

These services are related to monitoring, but they are not the same as telemetry collection.

- **Service Health**
  - Reports Azure platform incidents, planned maintenance, and health advisories.
  - It is subscription-scoped and helps you understand whether Azure itself is affected.
- **Resource Health**
  - Reports the health of an individual resource.
  - It helps you determine whether a resource is available, degraded, or unavailable.
  - It is useful when the issue may be caused by Azure infrastructure rather than your configuration.

Use these services when you need to distinguish between:

- A problem in your workload.
- A problem in the Azure platform.
- A problem affecting only one resource.

This distinction is important because Azure Monitor telemetry may show symptoms, while Service Health or Resource Health explains the platform-side cause.

**End-to-end monitoring scenario**

A common exam-style scenario is enabling monitoring for a VM or app from source to alert to action.

1. Deploy or identify the resource, such as a virtual machine or App Service.
2. Enable diagnostic settings to send logs and metrics to a Log Analytics workspace.
3. Ensure the guest agent or application instrumentation is installed if you need OS or app telemetry.
4. Verify data in Metrics Explorer and Logs Explorer.
5. Create an alert rule for the condition you care about.
6. Attach an action group to notify or automate response.
7. Validate the alert by generating a test condition or reviewing historical data.
8. Use a workbook or dashboard to track the resource over time.

Example VM monitoring flow:

- Platform metrics show `Percentage CPU`.
- Guest logs in Log Analytics show service failures or memory pressure.
- A log alert detects repeated `EventID` failures.
- An action group sends email and triggers a Logic App.
- Resource Health confirms whether the VM is impacted by Azure infrastructure.

Example App Service flow:

- Application Insights captures request failures and dependency latency.
- Workspace-based telemetry lands in Log Analytics.
- A log alert detects a spike in failed requests.
- An action group notifies the on-call team.
- A workbook correlates app failures with backend dependency errors.

**Practical portal workflow**

A typical Azure portal workflow looks like this:

1. Open the resource.
2. Select **Monitoring**.
3. Review **Metrics** for quick trends.
4. Open **Logs** for KQL-based investigation.
5. Configure **Diagnostic settings** to route telemetry.
6. Create **Alerts** for actionable conditions.
7. Use **Workbooks** or **Dashboards** to summarize the environment.

This workflow is useful because it mirrors how Azure Monitor is used in real operations: collect, store, query, visualize, and respond.

**Common exam distinctions**

- **Azure Monitor** is the overall platform.
- **Log Analytics** is the log storage and query workspace.
- **Application Insights** is application performance monitoring.
- **Activity Log** is subscription-level control-plane history.
- **Resource logs** are emitted by individual resources.
- **Service Health** and **Resource Health** report Azure and resource availability status.
- **Metrics Explorer** is for charts and near-real-time metrics.
- **Logs Explorer** is for KQL queries and deeper analysis.
- **Dashboards** summarize.
- **Workbooks** analyze.

> If a question asks where to query detailed diagnostic records, choose Logs Explorer or Log Analytics.
> If it asks where to view fast numeric trends, choose Metrics Explorer.
> If it asks about Azure platform outages, choose Service Health.
> If it asks about a specific resource’s availability state, choose Resource Health.

**Quick reference commands**

Create a workspace:

```bash
az monitor log-analytics workspace create \
  --resource-group rg-monitoring \
  --workspace-name law-prod-eastus \
  --location eastus
```

Send diagnostics to a workspace:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource <resource-id> \
  --workspace <workspace-id> \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Create a metric alert:

```bash
az monitor metrics alert create \
  --name vm-cpu-alert \
  --resource-group rg-monitoring \
  --scopes <vm-resource-id> \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m
```

Query logs:

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where ActivityStatusValue == "Failed"
| project TimeGenerated, OperationNameValue, ResourceGroup, Caller
| order by TimeGenerated desc
```
