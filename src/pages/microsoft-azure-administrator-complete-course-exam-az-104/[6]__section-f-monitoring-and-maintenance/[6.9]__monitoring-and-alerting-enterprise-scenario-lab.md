## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Enterprise monitoring scenario**

In an enterprise lab, you usually monitor a workload from multiple angles: platform health, guest OS health, application telemetry, and alert response. Azure Monitor is the umbrella service that collects, analyzes, and acts on telemetry from Azure resources, virtual machines, and applications. The goal is not just to “turn on alerts,” but to build a monitoring path that is actionable, low-noise, and easy to troubleshoot.

A practical monitoring design usually includes:

- **Metrics** for near-real-time numeric signals such as CPU, disk, and request rate.
- **Logs** for detailed event and query-based investigation.
- **Alerts** for proactive notification and automation.
- **Workbooks** for interactive troubleshooting and reporting.
- **Action groups** for routing notifications to the right responders.

> In exam scenarios, choose the simplest monitoring feature that meets the requirement, but prefer Azure Monitor workbooks over dashboards when you need interactive investigation.

**Reference architecture for the lab**

A common enterprise topology looks like this:

- A web app or VM-based application emits telemetry.
- Diagnostic settings send platform logs and metrics to a Log Analytics workspace.
- Application Insights captures application-level traces, requests, and dependencies.
- Alerts watch for threshold breaches, service health issues, and log query matches.
- Action groups notify email, SMS, push, voice, or webhook receivers.
- Workbooks summarize health, trends, and incident context.

This layered approach matters because one signal rarely tells the full story. For example, high CPU on a VM may be caused by a deployment, a runaway process, or a downstream dependency failure. Metrics tell you that something is wrong; logs and application telemetry help explain why.

**Core Azure Monitor building blocks**

Azure Monitor is built around a few key data types and services:

- **Metrics**
  - Lightweight numeric values collected at regular intervals.
  - Best for alerting on thresholds and trends.
  - Useful for fast detection of resource pressure.
- **Logs**
  - Structured records stored in Log Analytics.
  - Best for deep investigation and correlation.
  - Queried with Kusto Query Language (**KQL**).
- **Activity log**
  - Subscription-level record of control-plane events.
  - Useful for tracking create, update, delete, and policy-related actions.
- **Diagnostic settings**
  - Route resource logs and metrics to destinations such as Log Analytics, Event Hub, or Storage.
- **Alerts**
  - Trigger notifications or automation when conditions are met.
- **Action groups**
  - Define who gets notified and how.
- **Workbooks**
  - Interactive visualizations and query-driven reports.

**Lab objective: monitor a sample enterprise workload**

A realistic lab usually follows this pattern:

1. Deploy or identify a workload, such as an App Service, VM, or AKS-related component.
2. Enable diagnostic settings for platform logs and metrics.
3. Connect application telemetry to Application Insights.
4. Create a Log Analytics workspace for centralized analysis.
5. Build alerts for performance, availability, and service health.
6. Configure action groups for the operations team.
7. Validate the end-to-end flow by generating a test event.
8. Tune thresholds and reduce alert noise.

**Diagnostic settings and data collection**

Diagnostic settings control where resource telemetry goes. This is one of the most important enterprise monitoring steps because alerts and workbooks depend on the data being available in the right place.

Common destinations include:

- **Log Analytics workspace** for querying and alerting.
- **Storage account** for long-term archival.
- **Event Hub** for integration with SIEM or external systems.

For many Azure resources, you configure diagnostic settings in the Azure portal under the resource’s **Monitoring** section. The exact categories vary by service, but the workflow is similar:

1. Open the resource.
2. Select **Monitoring** > **Diagnostic settings**.
3. Add a diagnostic setting.
4. Choose the log and metric categories you need.
5. Select one or more destinations.
6. Save and verify ingestion.

A typical portal path for a VM, App Service, or Key Vault is:

- Resource page
- **Monitoring**
- **Diagnostic settings**
- **+ Add diagnostic setting**

A practical CLI example for a resource that supports diagnostic settings is shown below. The exact categories and parameters vary by resource type, so treat this as a pattern rather than a universal copy-paste command.

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Web/sites/<appName> \
  --workspace /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<lawName> \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true},{"category":"AppServiceConsoleLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

> Diagnostic category names differ by resource provider. Always confirm the available categories for the specific resource before creating the setting.

Useful validation commands:

```bash
az monitor diagnostic-settings list --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Web/sites/<appName>
az monitor log-analytics workspace show --resource-group <rg> --workspace-name <lawName>
```

**Azure Monitor Agent and data collection rules**

For virtual machines and some hybrid scenarios, the modern approach is the **Azure Monitor Agent** (**AMA**) with **Data Collection Rules** (**DCRs**). This replaces older agent patterns in many scenarios and gives you more control over what data is collected and where it goes.

Use AMA when you need:

- Guest-level performance counters.
- Windows Event Logs or Linux syslog collection.
- Custom log ingestion patterns.
- Centralized control over collection scope.

A DCR defines:

- Which data sources to collect.
- Which streams to send.
- Which destinations receive the data.
- Which machines the rule applies to.

A simplified conceptual example:

```json
{
  "dataSources": {
    "performanceCounters": [
      {
        "name": "cpu",
        "streams": [ "Microsoft-Perf" ],
        "samplingFrequencyInSeconds": 60,
        "counterSpecifiers": [ "\\Processor(_Total)\\% Processor Time" ]
      }
    ]
  },
  "destinations": {
    "logAnalytics": [
      {
        "workspaceResourceId": "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<lawName>"
      }
    ]
  }
}
```

For exam purposes, remember the relationship:

- **AMA** collects the data.
- **DCR** defines what to collect.
- **Log Analytics** stores and queries the data.

**Application Insights for application telemetry**

Application Insights is the application performance monitoring component of Azure Monitor. It is especially useful for web apps and APIs because it captures request rates, response times, failures, dependencies, and traces.

Use it when you need:

- End-to-end request tracing.
- Dependency failure analysis.
- Exception tracking.
- Performance trends by endpoint or operation.
- Correlation between app behavior and infrastructure events.

A common enterprise setup is to connect an App Service to an Application Insights resource and, where appropriate, link it to a Log Analytics workspace for unified querying.

Typical portal flow:

1. Open the web app.
2. Select **Application Insights**.
3. Turn it on or connect an existing resource.
4. Choose the workspace-based configuration if available.
5. Save and verify telemetry ingestion.

A minimal example of application telemetry in code might look like this:

```csharp
using Microsoft.ApplicationInsights;
using Microsoft.ApplicationInsights.Extensibility;

var telemetry = new TelemetryClient(TelemetryConfiguration.CreateDefault());
telemetry.TrackEvent("OrderSubmitted");
telemetry.TrackMetric("CheckoutLatencyMs", 245);
```

In practice, you usually rely on the SDK or auto-instrumentation rather than writing telemetry plumbing manually.

**KQL for investigation**

Kusto Query Language is the primary way to analyze Log Analytics and Application Insights data. Use it to filter, summarize, join, and trend telemetry.

A context-specific example for Application Insights request telemetry:

```kusto
requests
| where timestamp > ago(24h)
| summarize AvgDurationMs = avg(duration), Failures = countif(success == false) by cloud_RoleName
| order by Failures desc
```

This query is appropriate when the workspace contains Application Insights request data. It is not a universal query for every Log Analytics workspace.

A more general Log Analytics example for VM performance data might look like this:

```kusto
Perf
| where TimeGenerated > ago(1h)
| where CounterName == "% Processor Time"
| summarize AvgCPU = avg(CounterValue) by Computer, bin(TimeGenerated, 5m)
| order by AvgCPU desc
```

Useful KQL patterns:

- `where` to narrow the time window or scope.
- `summarize` to aggregate by host, app, or time bin.
- `bin()` to group time series data.
- `extend` to create derived fields.
- `join` to correlate logs from multiple tables.

**Alerts: metric, log, and activity log**

Azure Monitor supports several alert types, and the right choice depends on the signal you are watching.

| Alert type | Best for | Typical use |
|---|---|---|
| Metric alert | Numeric thresholds | CPU, memory, latency, queue depth |
| Log alert | Query-based conditions | Error spikes, failed logins, custom patterns |
| Activity log alert | Control-plane events | Resource changes, service health, policy events |
| Service health alert | Azure platform issues | Outages, planned maintenance, advisories |

Metric alerts are best when the signal is already exposed as a metric and you want fast detection. Log alerts are better when you need richer logic, such as counting failures over time or matching multiple conditions.

A practical metric alert example in the portal:

1. Open the resource or the Azure Monitor **Alerts** page.
2. Select **Create** > **Alert rule**.
3. Choose the target resource.
4. Select a metric such as CPU percentage.
5. Define the threshold and evaluation frequency.
6. Attach an action group.
7. Review and create.

A log alert example using KQL:

```kusto
Perf
| where TimeGenerated > ago(15m)
| where CounterName == "% Processor Time"
| summarize AvgCPU = avg(CounterValue) by Computer
| where AvgCPU > 80
```

This query returns rows when the condition is met. The alert rule then triggers based on the query result count or aggregation logic.

**Service health alerts and activity log alerts**

Service health alerts are important in enterprise operations because they notify you about Azure platform events that may affect your resources. They are distinct from activity log alerts.

Use **service health alerts** for:

- **Service issues** affecting Azure services.
- **Planned maintenance** that may impact availability.
- **Health advisories** that recommend action or awareness.

Use **activity log alerts** for:

- Resource creation, deletion, or configuration changes.
- Subscription-level administrative actions.
- Policy or RBAC-related events.
- Specific control-plane operations on resources.

This distinction matters in exams and in real operations:

- Service health tells you about Azure’s side of the problem.
- Activity log tells you about changes in your subscription.

**Action groups and receiver design**

Action groups define the notification and automation targets for alerts. In enterprise environments, you usually create one or more reusable action groups for operations, security, and application owners.

Common receiver types include:

- **Email**
- **SMS**
- **Voice**
- **Push notification**
- **Webhook**
- **Azure Function**
- **Logic App**
- **ITSM** integrations, where supported

Important operational details:

- Email receivers often require verification before they become active.
- SMS and voice availability can vary by region and country.
- Webhook receivers should expect a structured payload and be able to handle retries.
- Use separate action groups for different response teams to avoid noisy cross-team paging.

A conceptual action group design:

- `ag-ops-critical`
  - Email to on-call team
  - SMS to primary responder
  - Webhook to incident system
- `ag-app-owner`
  - Email to application owners
  - Logic App for ticket creation

A simplified webhook payload example:

```json
{
  "schemaId": "AzureMonitorMetricAlert",
  "data": {
    "essentials": {
      "alertRule": "HighCPU",
      "severity": "Sev2",
      "monitorCondition": "Fired"
    }
  }
}
```

> Webhook consumers should not assume a fixed payload shape across every alert type. Validate the schema and test the integration before relying on it in production.

**Alert processing rules, suppression, and smart grouping**

Enterprise alerting is not just about firing alerts; it is about controlling noise. Azure Monitor provides mechanisms to reduce duplicate notifications and route alerts more intelligently.

Key concepts:

- **Alert processing rules**
  - Modify how alerts are handled after they fire.
  - Useful for suppression during maintenance windows or for routing changes.
- **Suppression**
  - Temporarily prevents notifications from being sent.
  - Useful during planned maintenance, deployments, or known incidents.
- **Smart grouping**
  - Groups related alerts into a single incident-like experience.
  - Helps reduce alert storms when one root cause triggers many symptoms.

Use these features when:

- A deployment causes predictable transient alerts.
- One failing dependency triggers many downstream alerts.
- You need to silence non-actionable alerts during a change window.

A practical enterprise pattern is:

1. Create a maintenance window.
2. Apply an alert processing rule to suppress notifications.
3. Run the change.
4. Re-enable normal alerting.
5. Review any alerts that fired during the window.

**Workbooks and dashboards**

Workbooks are the preferred Azure Monitor tool for interactive analysis. Dashboards are useful for pinned, high-level views, but workbooks are more flexible for troubleshooting because they can combine queries, parameters, charts, and text in one place.

Use **workbooks** when you need:

- Interactive filtering by subscription, resource group, or time range.
- Drill-down from summary to detail.
- Multiple visualizations driven by KQL.
- Troubleshooting views for operations teams.

Use **dashboards** when you need:

- A simple shared overview.
- Pinned tiles for a broad audience.
- A lightweight status board.

A workbook troubleshooting flow might look like this:

1. Add a time range parameter.
2. Add a resource selector parameter.
3. Query CPU, failures, and dependency latency.
4. Visualize trends with charts.
5. Add a table of top failing instances.
6. Link to related logs for drill-down.

Example workbook query for a VM fleet:

```kusto
Perf
| where TimeGenerated > ago(24h)
| where CounterName == "% Processor Time"
| summarize AvgCPU = avg(CounterValue) by Computer
| order by AvgCPU desc
```

A workbook is especially useful in an incident because it can answer multiple questions in one view:

- Which resource is affected?
- When did the issue start?
- Is the problem isolated or widespread?
- Did a deployment or configuration change happen near the same time?

**End-to-end lab scenario**

This scenario ties the pieces together for an enterprise workload.

1. **Prepare the workspace**
   - Create or identify a Log Analytics workspace.
   - Confirm retention and access permissions.
   - Decide which teams need access to logs and workbooks.

2. **Enable telemetry**
   - Turn on diagnostic settings for the target resource.
   - Connect the application to Application Insights.
   - Install Azure Monitor Agent on VMs if guest telemetry is required.
   - Apply a Data Collection Rule for performance counters and event logs.

3. **Create alerting**
   - Add a metric alert for CPU above a threshold.
   - Add a log alert for repeated application failures.
   - Add a service health alert for platform incidents.
   - Add an activity log alert for critical resource changes.

4. **Configure response**
   - Create an action group for the operations team.
   - Add email and webhook receivers.
   - Verify email addresses.
   - Test the webhook endpoint with a sample payload.

5. **Build a workbook**
   - Add summary charts for CPU, failures, and latency.
   - Add a table of top affected resources.
   - Add filters for time range and resource group.

6. **Validate**
   - Generate load or a test failure.
   - Confirm telemetry appears in Log Analytics.
   - Confirm the alert fires.
   - Confirm the action group receives the notification.
   - Confirm the workbook reflects the new data.

7. **Tune**
   - Adjust thresholds if the alert is too sensitive.
   - Increase evaluation windows if the signal is noisy.
   - Suppress alerts during planned maintenance.
   - Review whether smart grouping reduces duplicate incidents.

**Troubleshooting workflow**

When an alert fires, use a consistent investigation path:

- Check the alert details and severity.
- Identify the affected resource and time window.
- Open the related workbook or logs.
- Correlate metrics with application traces and platform events.
- Check for recent deployments, configuration changes, or service health events.
- Decide whether the issue is infrastructure, application, or platform related.

A practical triage sequence:

1. Confirm the alert is real and not a transient spike.
2. Check whether the same issue appears in logs or Application Insights.
3. Review the activity log for recent changes.
4. Check service health for Azure-side incidents.
5. Escalate or suppress based on impact and recurrence.

**Common pitfalls**

- Sending telemetry to the wrong workspace or subscription.
- Creating alerts before the data source is actually ingesting data.
- Using thresholds that are too low and generating alert noise.
- Forgetting to verify email receivers in action groups.
- Assuming every Log Analytics workspace contains Application Insights tables.
- Using dashboards when a workbook would be better for investigation.
- Ignoring service health alerts and focusing only on resource metrics.
- Overlooking alert suppression during maintenance windows.

**Exam-focused takeaways**

- **Azure Monitor** is the umbrella service for metrics, logs, alerts, and visualization.
- **Diagnostic settings** route resource telemetry to destinations such as Log Analytics.
- **AMA** and **DCRs** are the modern pattern for VM and guest data collection.
- **Application Insights** is for application telemetry, not just infrastructure monitoring.
- **Metric alerts** are best for numeric thresholds; **log alerts** are best for query-based logic.
- **Service health alerts** cover Azure platform issues; **activity log alerts** cover subscription events.
- **Action groups** route notifications and automation, and receivers may require verification or special handling.
- **Alert processing rules**, suppression, and smart grouping help reduce noise in enterprise environments.
- **Workbooks** are usually the best choice for interactive troubleshooting and operational reporting.
