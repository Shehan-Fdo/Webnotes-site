## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure Activity Log overview**

The Azure Activity Log is the subscription-level audit trail for control-plane events. It records who did what, when, and where for operations that affect Azure resources and management-plane configuration. Use it to investigate changes, failures, policy actions, and service health events.

The Activity Log is not the same as resource diagnostic logs. It focuses on management operations such as creating a VM, changing a role assignment, or deleting a resource group. For deeper workload telemetry, you typically use resource logs, metrics, or application logs.

- **Scope**: subscription-wide, with filtering by resource group, resource, operation, status, and caller.
- **Retention**: limited in the portal by default; export if you need longer retention or centralized analysis.
- **Use cases**: incident response, change tracking, compliance review, and troubleshooting failed deployments.

**Where to access the Activity Log**

You can open the Activity Log from two common portal contexts, and exam questions often test the difference.

- **Subscription context**: go to the subscription, then select **Activity log**.
  - Path: `Azure portal > Subscriptions > <subscription-name> > Activity log`
  - Best for broad investigations across the entire subscription.
- **Resource context**: open a specific resource, then select **Activity log**.
  - Path: `Azure portal > <resource> > Activity log`
  - Best when you already know the affected resource and want a narrower view.

The resource-level view is filtered to that resource, while the subscription-level view shows all events in the subscription and is better for tracing related changes across multiple resources.

**Activity Log event categories**

Activity Log entries are grouped into categories that help you narrow the investigation.

| Category | What it usually contains | Investigation value |
|---|---|---|
| **Administrative** | Create, update, delete, and role assignment operations | Most common for change tracking and incident review |
| **Policy** | Policy evaluation results, denies, and remediation-related actions | Useful for understanding why a deployment or change was blocked |
| **Service Health** | Azure service incidents, planned maintenance, advisories, and security advisories | Helps distinguish platform issues from user-caused failures |
| **Security** | Security-related alerts or events surfaced by Azure | Less commonly used in day-to-day admin investigations and may not appear as broadly as other categories |
| **Recommendation** | Recommendations related to service health or platform guidance | Less central to audit investigations |

> **Note:** The **Security** category is not the primary place to look for all security incidents. In practice, many security investigations rely more on Microsoft Defender for Cloud, Microsoft Entra logs, or Sentinel, depending on the scenario.

**Key fields to interpret**

Each event includes metadata that helps you determine whether the action was user initiated, platform initiated, or policy driven.

- **Event time**: when Azure recorded the event.
- **Operation name**: the action performed, such as `Microsoft.Compute/virtualMachines/write`.
- **Status**: whether the operation succeeded, failed, or was in progress.
- **Event initiated by**: the caller, such as a user principal, service principal, or managed identity.
- **Resource group**: where the affected resource lives.
- **Correlation ID**: links related events across services and is useful for tracing a single request.
- **Submission timestamp**: when the event was ingested.
- **Substatus**: often provides more detail for failures.

A practical investigation usually starts with `status` and `event initiated by` to separate successful user actions from platform-initiated or denied operations.

**Portal investigation workflow**

A structured workflow helps you move from a broad alert to a specific root cause.

1. Open the **Activity log** at the correct scope.
   - Use the subscription view for unknown or cross-resource issues.
   - Use the resource view when the affected resource is already known.
2. Set the time range to match the incident window.
   - Narrow the window first to reduce noise.
3. Filter by **category**.
   - Start with **Administrative** for change tracking.
   - Add **Policy** for deployment denials or governance issues.
   - Check **Service Health** for platform outages or maintenance.
4. Filter by **status**.
   - Look for `Failed` when troubleshooting broken deployments.
   - Look for `Succeeded` when validating who made a change.
5. Filter by **event initiated by**.
   - Identify the user, service principal, or managed identity that triggered the action.
6. Open the event details.
   - Review operation name, correlation ID, substatus, and JSON details.
7. Pivot to related logs if needed.
   - Use the correlation ID in Log Analytics or other monitoring tools when available.

**Common investigation patterns**

Activity Log investigations usually fall into a few repeatable patterns.

- **Change tracking**
  - Identify who changed a resource.
  - Confirm the exact operation and timestamp.
  - Compare the event with change windows or change tickets.
- **Failure analysis**
  - Find failed operations in the relevant time range.
  - Check whether the failure came from policy, authorization, quota, or platform issues.
- **Policy deny analysis**
  - Look for `Policy` category events with `Denied` or similar failure status.
  - Review the policy assignment, effect, and scope.
- **Service health review**
  - Check whether Azure reported an incident, advisory, or maintenance event.
  - Use this to separate platform problems from configuration problems.

**Policy deny investigation example**

A deployment fails when creating a public IP address. In the Activity Log, you might see a policy-related event like this:

```text
Category: Policy
Operation name: Microsoft.Network/publicIPAddresses/write
Status: Failed
Event initiated by: user@contoso.com
Substatus: RequestDisallowedByPolicy
Message: Resource 'pip-prod-01' was disallowed by policy 'Deny public IP addresses'
Correlation ID: 7f3c2d1a-9b4e-4f2c-8f2d-1a2b3c4d5e6f
```

This tells you the failure is not a permissions issue or a platform outage. Instead, a policy assignment blocked the request. The next step is to inspect the policy assignment scope and effect, then determine whether the resource design must change or the policy needs an exception.

**Service health investigation example**

If users report that a VM deployment is failing in a region, check the **Service Health** category in the Activity Log.

```text
Category: Service Health
Event type: Incident
Status: Active
Title: Azure Compute experiencing issues in East US
Impact: Virtual machine provisioning delays
Recommended action: Retry deployment after service recovery
```

This kind of event helps you confirm that the issue is platform-related. In an exam scenario, this distinction matters because the correct response may be to wait, retry, or move to another region rather than changing RBAC or policy.

**Azure Monitor export and diagnostic settings**

If you need longer retention, centralized analysis, or alerting, export the Activity Log using Azure Monitor diagnostic settings. This is a common exam topic because the portal path is easy to confuse with resource diagnostics.

- **Portal path for subscription-level export**:
  - `Azure portal > Monitor > Activity log > Diagnostic settings`
  - Or: `Azure portal > Subscriptions > <subscription-name> > Activity log > Diagnostic settings`
- **What you can do**:
  - Send Activity Log data to a **Log Analytics workspace**.
  - Archive it to a **storage account**.
  - Stream it to an **event hub** for downstream processing.

A typical configuration sends the log to Log Analytics for KQL-based analysis and to storage for retention.

```bash
az monitor diagnostic-settings create \
  --name send-activity-log \
  --resource /subscriptions/<subscription-id> \
  --workspace /subscriptions/<subscription-id>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspace-name> \
  --logs '[{"category":"Administrative","enabled":true},{"category":"Policy","enabled":true},{"category":"ServiceHealth","enabled":true}]'
```

> **Important:** Diagnostic settings are configured at the subscription scope for the Activity Log. Do not look for this under a single resource’s diagnostic settings when your goal is subscription-wide audit export.

**KQL examples for investigation**

When Activity Log is exported to Log Analytics, you can use Kusto Query Language (KQL) to search and correlate events.

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where ActivityStatusValue == "Failed"
| project TimeGenerated, Caller, OperationNameValue, ResourceGroup, ResourceProviderValue, ActivitySubstatusValue, CorrelationId
| order by TimeGenerated desc
```

This query surfaces failed operations and helps you identify the caller and operation quickly.

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where CategoryValue == "Policy"
| project TimeGenerated, Caller, OperationNameValue, ActivityStatusValue, Properties, CorrelationId
| order by TimeGenerated desc
```

Use this when you suspect a policy deny or governance issue.

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where CategoryValue == "ServiceHealth"
| project TimeGenerated, ActivityStatusValue, Properties, CorrelationId
| order by TimeGenerated desc
```

Use this to confirm whether Azure reported an incident or maintenance event during the outage window.

**CLI and automation examples**

You can query the Activity Log from the Azure CLI for quick checks or automation.

```bash
az monitor activity-log list \
  --offset 24h \
  --status Failed \
  --query "[].{time:eventTimestamp, caller:caller, operation:operationName.value, resource:resourceGroupName, status:status.value}" \
  -o table
```

Filter by resource group:

```bash
az monitor activity-log list \
  --resource-group prod-rg \
  --offset 24h \
  -o table
```

Filter by caller:

```bash
az monitor activity-log list \
  --offset 24h \
  --caller user@contoso.com \
  -o table
```

These commands are useful when you need a quick answer without opening the portal.

**How to distinguish user actions from platform events**

A common exam challenge is deciding whether an event was caused by a person, automation, or Azure itself.

- **User action**
  - `event initiated by` shows a user principal or service principal.
  - The operation often matches a portal, CLI, or ARM deployment action.
- **Automation**
  - The caller may be a service principal or managed identity.
  - The timing may align with a pipeline or scheduled job.
- **Platform event**
  - The caller may be absent or not meaningful for the root cause.
  - Service Health or platform-generated entries explain the issue better than Administrative logs.

If the event is a failed deployment, always check whether the failure is due to policy, authorization, or platform health before assuming a resource misconfiguration.

**Practical exam tips**

- Use the **subscription-level Activity Log** when the question asks about all changes in a subscription.
- Use the **resource-level Activity Log** when the question names a specific resource.
- Check **Administrative** first for most change investigations.
- Check **Policy** when a deployment is denied or blocked.
- Check **Service Health** when the issue looks regional or platform-wide.
- Use **event initiated by** and **status** together to separate successful user actions from failed or platform-driven events.
- Export to **Log Analytics** when you need long-term retention or KQL queries.

**Quick reference: portal paths**

- Subscription Activity Log: `Azure portal > Subscriptions > <subscription> > Activity log`
- Resource Activity Log: `Azure portal > <resource> > Activity log`
- Export configuration: `Azure portal > Monitor > Activity log > Diagnostic settings`
- Log Analytics query area: `Azure portal > Log Analytics workspace > Logs`

**Summary**

The Azure Activity Log is the first place to look for management-plane investigations. Use the correct scope, filter by category and status, and inspect the caller and correlation ID to understand what happened. For longer retention and deeper analysis, export the log through Azure Monitor diagnostic settings into Log Analytics, storage, or Event Hubs.
