## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Diagnostic settings overview**

Azure diagnostic settings control how a supported resource exports platform logs and metrics to monitoring destinations. You use them when you need longer retention, centralized analysis, alerting, compliance evidence, or downstream processing outside the resource itself.

A diagnostic setting is attached to a specific resource, such as a Key Vault, storage account, virtual machine, App Service, or network resource. It defines:
- Which **log categories** to collect
- Which **metrics** to export
- Where to send the data
- Optional destination-specific settings, such as retention or table format choices

Diagnostic settings are part of Azure Monitor. They are not the same as activity logs:
- **Activity log** records subscription-level control-plane events.
- **Resource logs** record data-plane or resource-provider events for a specific resource.
- **Metrics** are time-series measurements, usually near real time.

> Not every Azure resource supports diagnostic settings, and not every supported resource offers all destinations. Availability varies by resource provider, feature, and region.

**What you can send logs to**

Azure Monitor diagnostic settings commonly support these destinations:
- **Log Analytics workspace** for querying with KQL (Kusto Query Language)
- **Storage account** for archival and long-term retention
- **Event Hubs** for streaming to SIEM (Security Information and Event Management), custom apps, or third-party tools

Some scenarios also integrate with partner solutions or service-specific export paths, but for AZ-104 you should know the three core destinations above.

**Log Analytics destination**

Log Analytics is the most useful destination for investigation and alerting because you can query data with KQL, build workbooks, and create alerts.

Key points:
- Data lands in a workspace table structure.
- Some services write to **resource-specific tables**.
- Older or legacy configurations may write to the **AzureDiagnostics** table.
- Retention is governed by the workspace settings and table-level retention policies, not by the diagnostic setting itself.

Practical implications:
- Use Log Analytics when you need search, correlation, and alerting.
- Use resource-specific tables when available for cleaner schemas and easier queries.
- Use `AzureDiagnostics` only when the resource or configuration uses the legacy schema, or when you need to query mixed legacy data.

**Storage account destination**

Storage is best for cheap archival and compliance retention.

Important behavior:
- Diagnostic settings do **not** define retention duration.
- Storage account lifecycle management or manual deletion controls retention.
- Network rules on the storage account can block ingestion if the resource cannot reach the account.
- If the storage account uses firewall restrictions, you must allow the relevant Azure services or otherwise ensure the source can write successfully.

Use storage when:
- You need immutable or long-term archive patterns.
- You want to keep raw logs for audits.
- You do not need interactive querying as the primary use case.

**Event Hubs destination**

Event Hubs is used for streaming logs to external systems.

Important requirements:
- You typically configure an **authorization rule** or shared access policy with send permissions.
- The diagnostic setting must reference the Event Hubs namespace and the correct event hub.
- This destination is common when forwarding logs to a SIEM, custom analytics pipeline, or partner collector.

Use Event Hubs when:
- You need near-real-time streaming.
- You already have a downstream consumer.
- You want to centralize logs outside Azure Monitor.

**Retention and lifecycle behavior**

A common exam trap is assuming the diagnostic setting controls retention. It does not.

Remember:
- **Log Analytics** retention is configured in the workspace and can also be influenced by table-level settings.
- **Storage account** retention is controlled by storage lifecycle policies, archive tiering, or deletion.
- **Event Hubs** is a streaming buffer, not a retention store.

This means you should design retention separately from export.

**Portal configuration workflow**

You usually configure diagnostic settings from the resource blade in the Azure portal.

Typical steps:
1. Open the target resource.
2. Select **Monitoring** or **Monitoring + diagnostics** depending on the resource type.
3. Choose **Diagnostic settings**.
4. Select **+ Add diagnostic setting**.
5. Enter a name for the setting.
6. Choose the log categories and metrics you want.
7. Select one or more destinations:
   - Send to Log Analytics workspace
   - Archive to a storage account
   - Stream to an event hub
8. Save the setting.

Common blade names vary slightly by resource:
- Some resources show **Diagnostic settings** directly under the **Monitoring** section.
- Some services expose a **Logs** or **Metrics** blade that links to diagnostic settings.
- For many resources, you can also start from **Azure Monitor** and then select the resource.

Practical portal tip:
- If you do not see the category you expect, check whether the resource supports that log type.
- If the resource offers both legacy and resource-specific export options, choose the one that matches your query and reporting needs.

**Resource-specific tables versus AzureDiagnostics**

Azure Monitor supports two broad patterns for Log Analytics export:
- **Legacy AzureDiagnostics table**
- **Resource-specific tables**

Resource-specific tables are generally preferred because:
- They use clearer schemas.
- Queries are easier to write.
- Columns are more service-focused.
- They reduce the need to filter a large shared table.

AzureDiagnostics can still be useful when:
- You are working with older configurations.
- A service only exports to the legacy table in your environment.
- You need a single table for multiple resource types.

Example KQL patterns:
```kusto
// Legacy-style query pattern
AzureDiagnostics
| where ResourceProvider == "MICROSOFT.KEYVAULT"
| where TimeGenerated > ago(24h)
| project TimeGenerated, Resource, OperationName, ResultType, ResultDescription
| order by TimeGenerated desc
```

```kusto
// Resource-specific table pattern example
KeyVaultAuditEvents
| where TimeGenerated > ago(24h)
| project TimeGenerated, OperationName, ResultType, CallerIPAddress, Identity
| order by TimeGenerated desc
```

> Table names vary by service and export mode. Always verify the actual table name in your workspace before building production queries.

**Common destination requirements and constraints**

Each destination has practical requirements you should remember.

- **Log Analytics workspace**
  - The workspace must exist and you must have permission to write diagnostic data.
  - Cross-tenant scenarios are not the default and may require additional configuration and permissions.
  - Workspace region and resource region compatibility can matter for some services and governance designs.

- **Storage account**
  - The storage account must be reachable from the source service.
  - Firewall and network rules can block ingestion.
  - Choose a storage account that matches your retention and compliance strategy.

- **Event Hubs**
  - You need an Event Hubs namespace and hub.
  - The authorization rule must allow sending.
  - Namespace capacity and throughput should match expected log volume.

**Azure CLI examples**

Use Azure CLI when you want repeatable automation or quick validation.

Log Analytics destination:
```bash
az monitor diagnostic-settings create \
  --name diag-to-law \
  --resource "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<kvName>" \
  --workspace "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspaceName>" \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Storage account destination:
```bash
az monitor diagnostic-settings create \
  --name diag-to-storage \
  --resource "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storageName>" \
  --storage-account "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<destStorageName>" \
  --logs '[{"category":"StorageRead","enabled":true},{"category":"StorageWrite","enabled":true},{"category":"StorageDelete","enabled":true}]' \
  --metrics '[{"category":"Transaction","enabled":true}]'
```

Event Hubs destination:
```bash
az monitor diagnostic-settings create \
  --name diag-to-eh \
  --resource "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<kvName>" \
  --event-hub "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.EventHub/namespaces/<nsName>/eventhubs/<hubName>" \
  --event-hub-rule "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.EventHub/namespaces/<nsName>/authorizationRules/<ruleName>" \
  --logs '[{"category":"AuditEvent","enabled":true}]'
```

Notes on CLI usage:
- Exact parameter names can vary slightly by Azure CLI version.
- Some services expose different log category names.
- Always confirm the resource ID and destination IDs are correct.

**PowerShell examples**

PowerShell support for diagnostic settings can vary by Az module version. In current environments, you may use `Az.Monitor` cmdlets or object constructors depending on the installed module version.

Example pattern:
```powershell
$resourceId = "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<kvName>"
$workspaceId = "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspaceName>"

$log = New-AzDiagnosticSettingLogSettingsObject -Category "AuditEvent" -Enabled $true
$metric = New-AzDiagnosticSettingMetricSettingsObject -Category "AllMetrics" -Enabled $true

New-AzDiagnosticSetting -Name "diag-to-law" `
  -ResourceId $resourceId `
  -WorkspaceId $workspaceId `
  -Log $log `
  -Metric $metric
```

Use PowerShell carefully:
- Cmdlet availability and parameter names depend on the installed `Az.Monitor` version.
- Verify the module documentation in your environment before using a script in production.
- If a cmdlet is unavailable, use `Get-Command *DiagnosticSetting*` to inspect what your version supports.

**ARM template and Bicep examples**

For infrastructure as code, use ARM templates or Bicep to deploy diagnostic settings consistently.

Bicep example:
```bicep
resource kv 'Microsoft.KeyVault/vaults@2023-07-01' existing = {
  name: kvName
}

resource law 'Microsoft.OperationalInsights/workspaces@2023-09-01' existing = {
  name: workspaceName
}

resource diag 'Microsoft.Insights/diagnosticSettings@2021-05-01-preview' = {
  name: 'diag-to-law'
  scope: kv
  properties: {
    workspaceId: law.id
    logs: [
      {
        category: 'AuditEvent'
        enabled: true
      }
    ]
    metrics: [
      {
        category: 'AllMetrics'
        enabled: true
      }
    ]
  }
}
```

Practical note:
- Use a current stable API version when available in your environment.
- The exact API version can change over time, so verify against current Azure resource provider documentation before production deployment.

**Azure Policy for scale**

Azure Policy is the standard governance tool for deploying diagnostic settings across many resources.

Why it matters:
- You can enforce logging on new resources.
- You can audit resources missing diagnostic settings.
- You can deploy settings automatically using policy remediation.

Common policy patterns:
- **Audit** resources that do not have diagnostic settings.
- **DeployIfNotExists** to create diagnostic settings automatically.
- Use policy initiatives to apply a consistent logging baseline across subscriptions or management groups.

Example use case:
- Require all Key Vaults to send audit logs to a central Log Analytics workspace.
- Require all storage accounts to export read/write/delete logs to a storage archive account.
- Require all production resources to stream logs to Event Hubs for SIEM ingestion.

**End-to-end example: Key Vault logging**

A realistic AZ-104 scenario is enabling Key Vault audit logging.

Typical categories:
- `AuditEvent`

Portal steps:
1. Open the Key Vault.
2. Go to **Monitoring** > **Diagnostic settings**.
3. Select **+ Add diagnostic setting**.
4. Name it `kv-audit-law`.
5. Check `AuditEvent`.
6. Select **Send to Log Analytics workspace**.
7. Choose the workspace.
8. Save.

Verification query:
```kusto
KeyVaultAuditEvents
| where TimeGenerated > ago(24h)
| project TimeGenerated, OperationName, ResultType, CallerIPAddress, Identity
| order by TimeGenerated desc
```

If your environment uses the legacy schema, you may need to query `AzureDiagnostics` instead. The exact table depends on the export mode.

**End-to-end example: Storage account logging**

Storage accounts commonly expose categories such as:
- `StorageRead`
- `StorageWrite`
- `StorageDelete`
- `Transaction`

Use this when you need to audit access patterns or troubleshoot data operations.

Example query pattern:
```kusto
StorageBlobLogs
| where TimeGenerated > ago(24h)
| summarize Count = count() by OperationName, ResultType
| order by Count desc
```

If the service exports to a legacy table in your environment, adjust the query to match the actual table name and schema.

**Troubleshooting checklist**

If logs are not appearing, check the following:
- The diagnostic setting is enabled and saved.
- The selected categories are supported by the resource.
- The destination exists and is correctly referenced.
- The workspace, storage account, or Event Hubs authorization rule has the right permissions.
- Network rules on the storage account are not blocking ingestion.
- You are querying the correct table and time range.
- The resource has had enough time to emit logs after the setting was created.

Common mistakes:
- Assuming diagnostic settings backfill historical logs.
- Querying the wrong table after switching to resource-specific export.
- Expecting metrics and logs to behave the same way.
- Forgetting that some resources only support a subset of destinations.

**Exam-focused takeaways**

- Diagnostic settings export **resource logs and metrics** from a specific Azure resource.
- The main destinations are **Log Analytics**, **Storage**, and **Event Hubs**.
- Retention is handled by the destination, not by the diagnostic setting.
- **Resource-specific tables** are usually easier to query than `AzureDiagnostics`.
- Azure Policy can deploy and enforce diagnostic settings at scale.
- Support varies by resource type, so always verify the available categories and destinations.
