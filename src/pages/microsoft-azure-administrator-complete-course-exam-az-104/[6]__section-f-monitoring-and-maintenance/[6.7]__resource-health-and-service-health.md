## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Overview**

Azure provides two related but distinct health surfaces: **Resource Health** and **Service Health**. In AZ-104, you need to know what each one tells you, where to find it in the portal, and how to use it during troubleshooting.

- **Resource Health** answers: “Is this specific resource healthy right now?”
- **Service Health** answers: “Is Azure experiencing an issue that may affect my subscription?”
- Both are part of Azure Monitor and are commonly used with the **Activity Log** and alerts.

A useful mental model is:

```text
Resource issue? -> check Resource Health
Platform/service issue? -> check Service Health
Need evidence or automation? -> check Activity Log / alerts / Azure Monitor
```

**Resource Health**

Resource Health shows the current and recent health state of an individual Azure resource, such as a virtual machine, SQL database, or App Service. It helps you separate a problem in your workload from a problem in the Azure platform.

**Portal navigation**

- Open the target resource in the Azure portal.
- In the resource menu, select **Resource health**.
- Review the current state, history, and any platform-reported events.

The exact menu placement can vary slightly by resource type and portal updates, but the feature is typically accessed from the resource’s own menu rather than from a separate global page.

**Common health states**

- **Available**: Azure reports the resource is healthy and reachable from the platform perspective.
- **Unavailable**: Azure reports the resource is not healthy or is unreachable.
- **Unknown**: Azure cannot determine the state, often because telemetry is delayed or the resource is in a transitional condition.

You may also see transitional or platform-related wording in the portal, such as:

- **Degraded**
- **Platform initiated**
- **User initiated**
- **Restoring**
- **Rebooting**
- **Migrating**

These labels are context-dependent and can vary by resource type. For exam purposes, focus on the meaning: Azure is distinguishing between a customer-caused event, a platform-caused event, and a state it cannot fully determine yet.

**How to interpret Resource Health**

- Use it to confirm whether Azure sees the resource as healthy.
- Use it to identify whether downtime is caused by Azure infrastructure or by your own configuration.
- Use the event details to see timestamps, reason codes, and whether the issue is platform initiated.

> Resource Health does **not** diagnose guest OS, application, or in-VM service failures. A VM can be **Available** while the operating system, agent, or application is broken.

**Practical examples**

- A VM is unreachable from the internet:
  - Check **Resource Health** first.
  - If it is **Unavailable**, the issue may be platform-related.
  - If it is **Available**, investigate NSGs, routes, DNS, guest OS, firewall, or the application.
- A database connection fails:
  - Resource Health may show **Available** even if the app has a bad connection string or the database firewall blocks access.

**Resource Health alerts**

Resource Health alerts are useful when you want to know when a specific resource becomes unhealthy. They are typically configured in Azure Monitor using alert rules that watch for health state changes.

**Portal-based configuration**

1. Open the resource in the Azure portal.
2. Select **Alerts** or go to **Azure Monitor**.
3. Create a new alert rule.
4. Choose the resource as the scope.
5. Select a signal related to **Resource Health** or a health-related activity log signal, depending on the resource and alerting path.
6. Define the condition for unhealthy states.
7. Add an action group for email, SMS, push, webhook, or automation.
8. Name the rule and create it.

A practical use case is alerting on a production VM or database so you are notified when Azure reports a platform issue before users escalate it.

**Service Health**

Service Health provides subscription-scoped visibility into Azure platform issues, planned maintenance, health advisories, and security advisories that may affect your subscription. It is not limited only to the exact regions or services you currently use; it can also surface broader Azure issues and advisories that are relevant to your subscription.

**Portal navigation**

- Open the Azure portal.
- Search for **Service Health** or go to **Monitor** and then **Service Health**.
- Review the dashboard for active incidents, planned maintenance, advisories, and history.

Service Health is personalized for your subscription context. It highlights events that are relevant to the subscriptions you have access to and the regions/services associated with them, while still allowing broader Azure advisories to appear when they matter to that subscription.

**What Service Health shows**

- **Service issues**: Unplanned Azure incidents affecting one or more services.
- **Planned maintenance**: Scheduled platform work that may affect availability or performance.
- **Health advisories**: Recommendations or notices that may require action.
- **Security advisories**: Security-related notifications that may affect services or configurations.
- **Event history**: Past incidents and advisories for review and auditing.

**How to interpret Service Health**

- It tells you whether Azure has a known platform issue that could affect your workload.
- It does **not** automatically mean your specific resource is down.
- It helps you correlate user-reported outages with Azure-side incidents.

A common exam scenario is this:

- Users report a service outage.
- Resource Health shows the resource as **Available**.
- Service Health shows an active incident in the region.
- Conclusion: the issue is likely Azure platform-related, not a misconfiguration in the resource itself.

**Service Health alerting**

Service Health alerts notify you when Azure posts an incident, maintenance event, or advisory that matches your subscription context.

**Portal-based configuration**

1. Open **Service Health** in the Azure portal.
2. Select **Health alerts** or **Service Health alerts**.
3. Create a new alert.
4. Choose the subscriptions and, if available, the services and regions you want to monitor.
5. Select the event types:
   - Service issues
   - Planned maintenance
   - Health advisories
   - Security advisories
6. Add an action group.
7. Save the alert.

Practical guidance:

- Use broad coverage for production subscriptions.
- Narrow the scope when you only want alerts for critical regions or services.
- Pair alerts with an action group that sends email and triggers automation or ticket creation.

**Resource Health vs. Service Health**

| Feature | Resource Health | Service Health |
|---|---|---|
| Scope | Individual resource | Subscription |
| Main question | Is this resource healthy? | Is Azure having an issue that may affect my subscription? |
| Best for | Isolating a single resource problem | Correlating outages with Azure incidents |
| Portal location | Resource menu | Azure Monitor / Service Health |
| Typical output | Available, Unavailable, Unknown, and related states | Incidents, maintenance, advisories, history |
| Common limitation | Does not diagnose guest OS or app issues | Does not prove a specific resource is down |

**How they work together**

Use both tools in a troubleshooting sequence rather than in isolation.

1. Check **Service Health** for subscription-wide incidents or maintenance.
2. Check **Resource Health** for the specific resource state.
3. Check the **Activity Log** for control-plane operations, failures, or user actions.
4. Check resource-specific diagnostics, metrics, and logs in **Azure Monitor**.
5. If needed, inspect the guest OS, application logs, networking, and identity configuration.

This sequence helps you avoid the common mistake of assuming every outage is either a platform issue or a resource issue. In practice, you often need both views.

**Azure Monitor and Activity Log context**

Azure Monitor is the broader observability platform that includes metrics, logs, alerts, and health-related experiences. The **Activity Log** records subscription-level control-plane events such as create, update, delete, and some platform notifications.

- Use **Activity Log** to see who changed what and when.
- Use **Resource Health** to see whether Azure considers the resource healthy.
- Use **Service Health** to see whether Azure has a known incident or maintenance event.

A practical troubleshooting pattern is:

```bash
# Example: inspect recent activity log events for a resource group
az monitor activity-log list \
  --resource-group rg-prod \
  --offset 7d \
  --query "[].{time:eventTimestamp, status:status.value, operation:operationName.value, caller:caller}" \
  -o table
```

This does not replace the portal health views, but it helps you correlate health events with administrative changes.

**CLI and automation notes**

Azure CLI can help you inspect related telemetry and automate alerting workflows, but the portal remains the clearest place to review Resource Health and Service Health during exam-style troubleshooting.

Useful commands include:

```bash
# List activity log events for a subscription
az monitor activity-log list \
  --offset 24h \
  -o table
```

```bash
# Show details for a specific resource
az resource show \
  --ids /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/<vmName>
```

For alerting, remember that Service Health alert creation in the portal is the most exam-relevant approach. CLI and ARM/Bicep can automate alert deployment, but the exact condition syntax is implementation-specific and can vary by alert type and API version.

**Practical troubleshooting scenarios**

- **VM outage**
  - Check Service Health for regional incidents.
  - Check Resource Health for the VM.
  - If both are healthy, inspect the VM guest OS, NIC, NSG, route table, and boot diagnostics.

- **App Service latency**
  - Check Service Health for platform incidents or maintenance.
  - Check Resource Health for the app service instance.
  - Review metrics, logs, and application telemetry for app-level issues.

- **Database connection failures**
  - Check whether Service Health reports a regional database incident.
  - Check Resource Health for the database resource.
  - Verify firewall rules, private endpoint configuration, DNS, and connection strings.

**Exam focus points**

- **Resource Health** is resource-specific and platform-aware.
- **Service Health** is subscription-scoped and includes incidents, maintenance, and advisories.
- **Available** does not guarantee the application is working.
- **Unavailable** does not always mean the resource is permanently broken; it may reflect a temporary platform issue.
- **Unknown** means Azure cannot confidently determine the state yet.
- Use **Activity Log** and **Azure Monitor** to correlate health events with changes and telemetry.
- Configure alerts so you are notified before users discover the issue.

**Key takeaways**

- Use **Resource Health** for the state of one resource.
- Use **Service Health** for Azure platform events affecting your subscription.
- Use both with **Activity Log** and **Azure Monitor** to build a complete troubleshooting picture.
- Configure alerts for production workloads so you can respond quickly to platform or resource degradation.
