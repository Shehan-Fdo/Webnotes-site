## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Hybrid connectivity overview**

Hybrid connectivity links on-premises networks, branch offices, and other clouds to Azure. In AZ-104, you need to know when to use each option, how traffic flows, and what limits affect design.

The main patterns are:

- **Site-to-Site (S2S) VPN** for encrypted IPsec/IKE tunnels over the internet.
- **Point-to-Site (P2S) VPN** for individual client devices that need secure access.
- **ExpressRoute** for private, dedicated connectivity through a connectivity provider.
- **VNet peering** for Azure-to-Azure connectivity inside Microsoft’s backbone.
- **DNS integration** so private names resolve correctly across on-premises and Azure.

A good design starts with address planning. If your on-premises and Azure address spaces overlap, routing becomes difficult or impossible for many hybrid patterns. Plan non-overlapping CIDR blocks early, especially if you expect future mergers, acquisitions, or multi-region expansion.

**Connectivity choice and when to use it**

Use the simplest option that meets the requirement:

- **VNet peering** when you need low-latency Azure-to-Azure connectivity and do not need encryption.
- **S2S VPN** when you need a cost-effective encrypted tunnel to a branch or datacenter.
- **P2S VPN** when you need secure access for remote users or admins.
- **ExpressRoute** when you need predictable performance, private connectivity, or higher bandwidth.
- **Azure DNS Private Resolver** when you need managed DNS forwarding between on-premises and Azure private zones.

VNet peering is often the first choice for VNet-to-VNet traffic because it is simple and fast. Use VPN between VNets only when you specifically need encryption or when peering is not suitable for the topology.

**Azure VPN Gateway fundamentals**

Azure VPN Gateway provides encrypted connectivity for S2S, P2S, and some VNet-to-VNet scenarios. It is deployed into a dedicated `GatewaySubnet`.

Key design points:

- **GatewaySubnet** must exist before you create the gateway.
- Use a sufficiently large subnet, commonly `/27` or larger, to allow growth and gateway requirements.
- The gateway SKU affects throughput, features, and scaling.
- Active-active mode improves availability by using two gateway instances.
- BGP (Border Gateway Protocol) can automate route exchange with on-premises devices.

Common gateway SKU guidance:

| SKU family | Typical use | Notes |
|---|---|---|
| `VpnGw1` | Small production or lab environments | Lower throughput, fewer features |
| `VpnGw2` | Moderate production workloads | Better throughput and scale |
| `VpnGw3` and above | Higher throughput or more demanding scenarios | Useful when bandwidth or connection count grows |

Practical selection guidance:

- Choose a higher SKU when you need more throughput, more tunnels, or better scale.
- Choose active-active when uptime matters and your on-premises device supports it.
- Choose BGP when you want dynamic route exchange instead of static route maintenance.
- Choose route-based VPN gateways for most modern designs.

> Azure VPN Gateway is not the same as a virtual appliance. If you need custom routing, inspection, or third-party features, you may use an NVA (network virtual appliance) instead.

**Site-to-Site VPN**

S2S VPN connects an on-premises VPN device to an Azure VPN gateway using IPsec/IKE. It is the standard hybrid option for branch-to-Azure connectivity when you want encryption over the public internet.

Typical use cases:

- Branch office access to Azure-hosted apps.
- Small datacenter connectivity.
- Backup path alongside ExpressRoute.
- Transitional connectivity during migration.

Core requirements:

- A compatible on-premises VPN device or firewall.
- A local network gateway in Azure that represents the on-premises address space and VPN device public IP.
- Matching IPsec/IKE parameters on both sides.
- Non-overlapping address spaces.

Example Azure CLI workflow for S2S VPN:

```bash
# Create a resource group
az group create -n rg-hybrid -l eastus

# Create a VNet and GatewaySubnet
az network vnet create \
  -g rg-hybrid \
  -n vnet-hub \
  -l eastus \
  --address-prefixes 10.10.0.0/16 \
  --subnet-name GatewaySubnet \
  --subnet-prefixes 10.10.255.0/27

# Create a workload subnet
az network vnet subnet create \
  -g rg-hybrid \
  --vnet-name vnet-hub \
  -n snet-app \
  --address-prefixes 10.10.1.0/24

# Create the VPN gateway public IP
az network public-ip create \
  -g rg-hybrid \
  -n pip-vpngw \
  -l eastus \
  --sku Standard \
  --allocation-method Static

# Create the VPN gateway
az network vnet-gateway create \
  -g rg-hybrid \
  -n vpngw-hub \
  -l eastus \
  --public-ip-addresses pip-vpngw \
  --vnet vnet-hub \
  --gateway-type Vpn \
  --vpn-type RouteBased \
  --sku VpnGw2
```

After the gateway exists, you define the on-premises side with a local network gateway and create the connection.

```bash
# Represent the on-premises VPN device and prefixes
az network local-gateway create \
  -g rg-hybrid \
  -n lng-onprem \
  --gateway-ip-address 203.0.113.10 \
  --local-address-prefixes 192.168.10.0/24 192.168.20.0/24

# Create the S2S connection
az network vpn-connection create \
  -g rg-hybrid \
  -n conn-onprem \
  --vnet-gateway1 vpngw-hub \
  --local-gateway2 lng-onprem \
  --shared-key 'StrongSharedKeyHere'
```

Important S2S concepts:

- **Shared key** must match on both ends.
- **BGP** can reduce manual route management.
- **Custom IPsec policies** may be required for interoperability.
- **Connection status** should be checked after deployment and after any firewall or routing change.

**Point-to-Site VPN**

P2S VPN lets individual clients connect to Azure from anywhere. It is useful for administrators, developers, and remote workers who need access to private resources without exposing them publicly.

Common authentication methods:

- **Certificate authentication** for smaller deployments or lab environments.
- **Microsoft Entra ID authentication** for user-based access and easier management.
- **RADIUS** for integration with existing identity systems.

P2S client considerations:

- You generate and download a client configuration package from the VPN gateway.
- The client imports the profile and establishes the tunnel.
- The client receives routes to Azure address spaces based on the configuration.
- DNS settings matter if users must resolve private names.

Example P2S configuration flow:

1. Create or verify the VPN gateway.
2. Configure the P2S address pool, authentication method, and DNS servers.
3. Download the client package.
4. Install the client profile on the user device.
5. Test access to private IPs and private DNS names.

P2S is especially useful when:

- You need temporary admin access.
- You want secure access without exposing a full branch network.
- You need a fallback option when site connectivity is unavailable.

**ExpressRoute fundamentals**

ExpressRoute provides private connectivity between your network and Microsoft cloud services through a connectivity provider, colocation facility, or ExpressRoute Direct. Traffic does not traverse the public internet in the normal path.

Use ExpressRoute when you need:

- More predictable latency and bandwidth.
- Private connectivity for enterprise workloads.
- Higher reliability for critical applications.
- A dedicated circuit model instead of internet-based VPN.

Core ExpressRoute concepts:

- **Circuit**: the logical connection you order.
- **Peering**: route exchange for Microsoft services and private Azure resources.
- **Provider**: the carrier or exchange partner that delivers the circuit.
- **Gateway**: the Azure virtual network gateway that connects a VNet to the circuit.

Practical exam distinctions:

- **Private connectivity to VNets** uses ExpressRoute private peering.
- **Microsoft services access** uses Microsoft peering where supported.
- **Public internet access is not the goal** of ExpressRoute.

ExpressRoute tiers and features:

| Feature | Standard | Premium |
|---|---|---|
| Global reach across regions | Limited | Expanded |
| Number of VNets | Lower limit | Higher limit |
| Geographic scope | More constrained | Broader |
| Route scale | Lower | Higher |

Use Premium when you need:

- More VNets connected to the circuit.
- Broader geographic reach.
- More demanding enterprise routing scenarios.

Use Standard when:

- Your design stays within a smaller scope.
- You do not need the extra scale or geographic flexibility.

ExpressRoute Direct is used when you need direct physical connectivity into Microsoft’s edge at high bandwidth. It is typically relevant for large enterprises and specialized network designs.

**ExpressRoute advanced features**

Several features often appear in design questions:

- **FastPath** improves data path performance by sending traffic directly to the VNet after gateway control-plane setup.
- **Global Reach** connects on-premises sites through Microsoft’s network using ExpressRoute circuits.
- **Route filters** control which Microsoft service routes are advertised for Microsoft peering scenarios.
- **BGP** is used for route exchange and path selection.

Use these features carefully:

- Use **FastPath** when you want to reduce gateway data-plane bottlenecks.
- Use **Global Reach** when you need branch-to-branch or datacenter-to-datacenter transit through Microsoft’s backbone.
- Use **route filters** when you need to limit Microsoft service route propagation.

> ExpressRoute is not a replacement for all routing design. You still need correct IP planning, route control, and DNS design.

**VNet peering and transitivity**

VNet peering connects two VNets privately over Microsoft’s backbone. It is fast and simple, but it is not transitive by default.

Important rules:

- Peering is **non-transitive**: if VNet A peers with VNet B, and VNet B peers with VNet C, A does not automatically reach C.
- Address spaces must not overlap.
- You can enable gateway transit in hub-and-spoke designs.
- Use remote gateway and allow gateway transit carefully to avoid routing confusion.

Typical hub-and-spoke pattern:

- Hub VNet contains shared services, firewall, DNS, and gateway.
- Spoke VNets host application workloads.
- Spokes peer to the hub.
- The hub provides centralized connectivity to on-premises or the internet.

This pattern is common because it simplifies governance and centralizes inspection.

**Private endpoints and service endpoints**

Private access to PaaS (Platform as a Service) services is a frequent exam topic. Private endpoints and service endpoints solve different problems.

| Feature | Service endpoint | Private endpoint |
|---|---|---|
| Network path | Service remains public but restricted to VNet identity | Service gets a private IP in your VNet |
| DNS requirement | Usually simpler | Requires private DNS planning |
| Exposure | Public service endpoint still exists | Private access path is preferred |
| Best for | Quick VNet restriction | Strong private access and isolation |

Service endpoints:

- Extend VNet identity to a service.
- Restrict access to selected VNets/subnets.
- Do not assign a private IP to the service.
- Are simpler to configure than private endpoints.

Private endpoints:

- Create a private IP in your subnet for the service.
- Require DNS to resolve the service name to the private IP.
- Are preferred when you want to eliminate public exposure.
- Are common for Storage, Key Vault, SQL, and other PaaS services.

Example private endpoint design for Azure Storage:

1. Create the storage account.
2. Create a private endpoint in the target VNet/subnet.
3. Create or link the private DNS zone, such as `privatelink.blob.core.windows.net`.
4. Ensure the VNet is linked to the private DNS zone.
5. Test name resolution from a VM in the VNet and from on-premises if needed.

Example Azure CLI for a private endpoint and DNS zone link:

```bash
# Create a private DNS zone for blob storage
az network private-dns zone create \
  -g rg-hybrid \
  -n privatelink.blob.core.windows.net

# Link the zone to the VNet
az network private-dns link vnet create \
  -g rg-hybrid \
  -n link-hub \
  -z privatelink.blob.core.windows.net \
  -v /subscriptions/<subId>/resourceGroups/rg-hybrid/providers/Microsoft.Network/virtualNetworks/vnet-hub \
  -e false
```

DNS is critical here. If the private DNS zone is not linked to the VNet, Azure resources in that VNet may not resolve the private endpoint name correctly.

**Azure DNS Private Resolver**

Azure DNS Private Resolver is a managed DNS forwarding service. It helps when you need name resolution between on-premises DNS and Azure private DNS zones without deploying your own DNS forwarder VM.

Core components:

- **Inbound endpoint**: on-premises DNS servers query this endpoint to resolve Azure private names.
- **Outbound endpoint**: Azure can forward DNS queries to on-premises DNS servers.
- **Ruleset**: defines forwarding rules for specific DNS zones.
- **VNet link**: connects the resolver to the VNet where it operates.

When to use it:

- You want managed DNS forwarding instead of custom DNS VMs.
- You need on-premises clients to resolve Azure private endpoint names.
- You need Azure workloads to resolve on-premises internal zones.

When not to use it:

- If a simple custom DNS server already meets the requirement.
- If the environment is too small to justify the extra service.
- If you only need basic Azure-provided DNS inside a single VNet.

Important dependency chain for private DNS resolution:

- The private endpoint must exist.
- The private DNS zone must exist.
- The zone must be linked to the VNet used by the resolver and workloads.
- The resolver must have the correct inbound or outbound endpoint.
- On-premises DNS must forward the relevant zone to the inbound endpoint.

A common pattern is:

- On-premises DNS forwards `privatelink.<service>.core.windows.net` to the Azure DNS Private Resolver inbound endpoint.
- The resolver answers using the private DNS zone linked to the Azure VNet.
- Azure VMs and on-premises clients both resolve the same private name consistently.

**DNS design in hybrid environments**

DNS is often the hidden cause of hybrid failures. Network connectivity may be fine while name resolution fails.

Key principles:

- Azure-provided DNS works well for resources inside a VNet.
- It is not a general recursive DNS service for on-premises clients.
- On-premises DNS usually needs conditional forwarding for Azure private zones.
- Private endpoint names often require zone-specific forwarding.

Common DNS patterns:

- **Azure-provided DNS** for simple VNet-only workloads.
- **Custom DNS servers** in Azure or on-premises for enterprise control.
- **Azure DNS Private Resolver** for managed hybrid forwarding.
- **Conditional forwarders** for private zones such as `privatelink.blob.core.windows.net`.

Troubleshooting tip:

- If a VM can reach a private endpoint by IP but not by name, check the private DNS zone link first.
- If on-premises clients resolve the public IP instead of the private IP, check conditional forwarding and resolver configuration.

**Troubleshooting hybrid connectivity**

Troubleshooting should follow a layered approach: identity, routing, tunnel state, DNS, and service configuration.

Useful checks for VPN:

- Verify the connection status in the portal.
- Check the local network gateway public IP and prefixes.
- Confirm the shared key matches.
- Validate that the on-premises firewall allows IPsec/IKE traffic.
- Review BGP neighbor status if dynamic routing is enabled.

Useful checks for ExpressRoute:

- Confirm the circuit provisioning state.
- Verify peering configuration and provider handoff.
- Check BGP session status.
- Validate advertised prefixes and route tables.
- Confirm the gateway is connected to the circuit.

Useful checks for VNet peering:

- Confirm peering is established in both directions if required.
- Check whether gateway transit is enabled where needed.
- Verify that address spaces do not overlap.
- Ensure NSGs and UDRs do not block traffic.

Useful checks for DNS:

- Test with `nslookup` or `Resolve-DnsName`.
- Compare resolution from Azure VM, on-premises host, and VPN client.
- Confirm private DNS zone links.
- Check forwarding rules and resolver endpoints.

Example diagnostic commands:

```bash
# Check VPN gateway and connection state
az network vpn-connection show -g rg-hybrid -n conn-onprem --query "{status:connectionStatus, ingress:ingressBytesTransferred, egress:egressBytesTransferred}"

# Check ExpressRoute circuit state
az network express-route show -g rg-hybrid -n er-circuit --query "{state:serviceProviderProvisioningState, sku:sku}"

# Test DNS from a Linux VM
nslookup myapp.privatelink.blob.core.windows.net
```

For packet-level or path-level checks, use:

- **Network Watcher** connection troubleshoot.
- **Effective routes** on a VM NIC.
- **IP flow verify** for NSG-related issues.
- **Next hop** to confirm routing decisions.

**Design constraints and caveats**

Several constraints matter in exam scenarios:

- **Overlapping address spaces** block many hybrid designs.
- **VNet peering is non-transitive** unless you deliberately build routing through a hub or gateway.
- **Gateway subnet sizing** should allow future growth.
- **Throughput limits** vary by VPN SKU and ExpressRoute design.
- **DNS forwarding** must match the private zone architecture.
- **Asymmetric routing** can break stateful firewalls and tunnels.
- **UDRs (User Defined Routes)** can override expected paths and cause black holes.

When designing a hub-and-spoke environment:

- Put shared services in the hub.
- Keep spokes simple.
- Centralize DNS and security controls.
- Decide early whether traffic should go through a firewall or directly through peering.

**End-to-end scenario**

A common AZ-104 scenario is a branch office that needs access to an Azure app and a private storage account.

Recommended design:

- Use a hub VNet in Azure.
- Deploy a VPN gateway in the hub.
- Peer spoke VNets to the hub.
- Place the app in a spoke subnet.
- Use a private endpoint for the storage account.
- Link the private DNS zone to the hub VNet.
- Configure on-premises DNS to forward the private zone to Azure DNS Private Resolver.
- Use S2S VPN for branch connectivity.
- Add BGP if the branch router supports it.

Traffic flow:

1. Branch client resolves the app or storage name.
2. DNS forwards private zone queries to Azure if needed.
3. The branch sends traffic through the S2S tunnel.
4. Azure routes traffic through the hub to the spoke or private endpoint.
5. Return traffic follows the same path if routing is symmetric.

If the organization later needs higher bandwidth and more predictable performance, you can replace or supplement the VPN with ExpressRoute while keeping the same hub-and-spoke and DNS principles.

**Practical exam reminders**

- Prefer **VNet peering** for Azure-to-Azure connectivity unless encryption or topology requires VPN.
- Use **S2S VPN** for encrypted branch connectivity over the internet.
- Use **P2S VPN** for individual remote users.
- Use **ExpressRoute** for private enterprise connectivity and higher reliability.
- Use **private endpoints** when you need private IP access to PaaS services.
- Use **Azure DNS Private Resolver** when DNS forwarding becomes a hybrid requirement.
- Check **routes, DNS, and tunnel state** before assuming the service itself is broken.
