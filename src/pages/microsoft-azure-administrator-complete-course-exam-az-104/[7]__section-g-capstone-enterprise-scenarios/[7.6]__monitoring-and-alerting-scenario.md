## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure monitoring overview**

Azure monitoring is the set of services you use to collect, analyze, visualize, and act on telemetry from Azure resources, guest operating systems, applications, and network components. For AZ-104, focus on the core workflow: collect data, store it in the right place, query it, and create alerts that trigger action.

Azure Monitor is the umbrella service. It includes:
- **Metrics** for near real-time numeric data.
- **Logs** for detailed, queryable records.
- **Alerts** for automated notification and remediation.
- **Workbooks** for interactive analysis and reporting.
- **Application Insights** for application performance monitoring.

A useful mental model is:
- Metrics answer “Is something happening right now?”
- Logs answer “What happened, where, and why?”
- Alerts answer “When should I be notified or trigger automation?”

**Azure Monitor data sources**

Azure Monitor can ingest telemetry from multiple layers of the stack:
- **Platform metrics** from Azure resources such as virtual machines, App Service, and storage.
- **Resource logs** from services that support diagnostic settings.
- **Guest OS data** from Windows and Linux machines through the Azure Monitor Agent (AMA).
- **Application telemetry** through Application Insights SDKs or OpenTelemetry-based instrumentation.
- **Activity Log** events for subscription-level control plane operations.

The exam often tests whether you know which signal to use:
- Use **metrics** for fast trend detection and threshold-based alerting.
- Use **logs** for deep investigation and correlation.
- Use **Activity Log** for subscription events such as create, update, delete, and policy actions.
- Use **Application Insights** for application request, dependency, exception, and performance data.

**Log Analytics workspace basics**

A Log Analytics workspace is the central container for Azure Monitor Logs. Multiple Azure resources can send data to a single workspace, which makes it the usual place to centralize operational telemetry.

Key workspace concepts:
- **Scope**: A workspace can collect data from many resources across a subscription or even multiple subscriptions, depending on access and configuration.
- **Retention**: You control how long data stays available for query and compliance needs.
- **Access control**: Permissions are managed with Azure role-based access control (Azure RBAC). Common roles include Log Analytics Reader and Monitoring Reader.
- **Querying**: You use Kusto Query Language (KQL) to search and summarize data.
- **Cost**: Ingestion and retention affect cost, so you should collect only the data you need.

A common design pattern is to send:
- VM guest logs and performance counters to one workspace.
- App service diagnostics to the same workspace.
- Security or operational logs to separate workspaces when access boundaries or retention requirements differ.

> Not every resource supports every destination or every log category. Diagnostic settings vary by service, and some services have service-specific behavior.

**Azure Monitor Agent and Data Collection Rules**

The Azure Monitor Agent (AMA) is the modern agent used to collect guest OS data from Windows and Linux machines. It replaces older approaches in many scenarios and is the preferred choice for new deployments.

Data Collection Rules (DCRs) define:
- What data to collect.
- Which machines or machine groups to apply collection to.
- Where to send the data, such as a Log Analytics workspace or Azure Monitor Metrics.

Why DCRs matter:
- They centralize collection policy.
- They reduce agent sprawl.
- They let you standardize performance counters, event logs, and syslog collection.
- They support scalable onboarding for many VMs.

Typical guest OS collection flow:
1. Install or enable AMA on the VM.
2. Create a DCR that specifies counters, event logs, or syslog facilities.
3. Associate the DCR with the target VM or VM scale set.
4. Verify data in Log Analytics.

Example Azure CLI pattern for associating a DCR with a VM:
```bash
az monitor data-collection rule association create \
  --name vm01-dcr-association \
  --rule-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Insights/dataCollectionRules/<dcrName> \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/<vmName>
```

Example DCR fragment for Windows performance counters:
```json
{
  "dataSources": {
    "performanceCounters": [
      {
        "name": "perfCounterDataSource",
        "streams": [ "Microsoft-Perf" ],
        "samplingFrequencyInSeconds": 60,
        "counterSpecifiers": [
          "\\Processor(_Total)\\% Processor Time",
          "\\Memory\\Available MBytes"
        ]
      }
    ]
  }
}
```

**Diagnostic settings**

Diagnostic settings route resource logs and, for many services, platform metrics to a destination. Common destinations include:
- Log Analytics workspace
- Storage account
- Event Hub

Use diagnostic settings when you need:
- Centralized log retention and search.
- Export to a SIEM or external analytics platform.
- Long-term archival in storage.
- Near-real-time streaming to downstream systems.

Important exam nuance:
- Not all resource types support all destinations.
- Not all log categories or metric categories are available for every service.
- Some services expose diagnostic settings differently, so always verify the resource’s supported categories.

Typical portal workflow:
1. Open the target resource.
2. Select **Diagnostic settings**.
3. Add a new setting.
4. Choose the log and metric categories you need.
5. Select one or more destinations.
6. Save and validate ingestion in the destination workspace or storage account.

Example CLI pattern for creating a diagnostic setting:
```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Web/sites/<appName> \
  --workspace /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspaceName> \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true},{"category":"AppServiceConsoleLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

**Metrics versus logs**

Metrics are best for fast, lightweight monitoring. They are stored as time-series data and are ideal for alerting on thresholds, trends, and availability.

Logs are best for detailed analysis. They support richer queries, joins, and correlation across resources.

Use metrics when:
- You need near real-time alerting.
- You want low-latency dashboards or charts.
- You are tracking CPU, memory, disk, request count, or latency trends.

Use logs when:
- You need to investigate failures.
- You need to correlate events across systems.
- You need to filter by dimensions, categories, or custom fields.

A practical example:
- Metric alert: CPU percentage above 80% for 10 minutes.
- Log query: failed requests by minute, grouped by cloud role or operation name.

**Kusto Query Language basics**

KQL is the query language used in Azure Monitor Logs and Log Analytics. It is central to troubleshooting and log-based alerting.

Common operators and functions:
- `where` filters rows.
- `summarize` aggregates data.
- `bin()` groups time into intervals.
- `project` selects columns.
- `extend` creates calculated columns.
- `join` correlates tables.

Example query for failed requests in Application Insights-style telemetry:
```kusto
requests
| where success == false
| summarize FailedRequests = count() by bin(timestamp, 5m)
| order by timestamp asc
```

If you are querying Log Analytics workspace tables that use `TimeGenerated`, adjust the query accordingly:
```kusto
AzureDiagnostics
| where TimeGenerated > ago(1h)
| summarize Count = count() by bin(TimeGenerated, 5m)
```

That distinction matters because table schemas differ by data source. In exam scenarios, always identify whether you are working with Application Insights telemetry or Log Analytics workspace data.

Useful troubleshooting patterns:
- Start with a narrow time window.
- Filter by resource, operation, or severity.
- Summarize before you join.
- Use `take` or `limit` while exploring large tables.

**Application Insights**

Application Insights is the application performance monitoring component of Azure Monitor. It helps you understand how an application behaves from the user’s perspective and from the dependency perspective.

It commonly captures:
- Requests
- Dependencies
- Exceptions
- Traces
- Custom events and metrics

Why it matters:
- It helps you detect slow requests and failures.
- It shows dependency bottlenecks such as SQL, HTTP, or storage calls.
- It supports distributed tracing across services.

Typical use cases:
- Web apps and APIs.
- Microservices.
- Serverless applications.
- Applications that need end-to-end transaction visibility.

Application Insights can be connected to a Log Analytics workspace, which simplifies querying and centralization. In modern setups, telemetry often flows into workspace-based Application Insights for unified analysis.

**Alerts and alert rules**

Alerts notify you when a condition is met. In Azure Monitor, alert rules evaluate signals and trigger actions.

Main alert types:
- **Metric alerts** for numeric platform metrics.
- **Log alerts** for KQL query results.
- **Activity Log alerts** for subscription-level events.
- **Service Health alerts** for Azure service incidents and planned maintenance.
- **Resource Health alerts** for the health state of a specific resource.

Alert rule components:
- **Signal source**: metric, log, activity log, service health, or resource health.
- **Condition**: the threshold or query logic.
- **Scope**: one resource, many resources, a resource group, or a subscription.
- **Action group**: who or what gets notified.
- **Frequency**: how often the rule evaluates.
- **Severity**: how urgent the alert is.

Alert state and lifecycle:
- An alert can fire when the condition is met.
- It can resolve automatically when the condition clears.
- Some alert types support stateful behavior, where the system tracks active and resolved states.
- Alert processing rules can suppress or route alerts during maintenance windows or known events.

Alert processing rules are important for exam scenarios because they help you:
- Suppress alerts during planned maintenance.
- Reduce noise from repeated notifications.
- Route alerts based on time, scope, or resource tags.

**Action groups**

An action group defines what happens when an alert fires. It is the notification and automation target for alert rules.

Common action group actions:
- Email
- SMS
- Push notification
- Voice call
- Azure Function
- Logic App
- Automation Runbook
- ITSM connector
- Webhook

Use action groups to:
- Notify operators.
- Open tickets.
- Trigger remediation.
- Integrate with external incident management systems.

Example portal workflow:
1. Create an action group.
2. Add notification methods.
3. Add automation actions if needed.
4. Reuse the action group across multiple alert rules.

Example CLI pattern:
```bash
az monitor action-group create \
  --name ag-ops \
  --resource-group <rg> \
  --short-name ops \
  --action email admin1 admin1@example.com
```

**Metric alerts, dimensions, and dynamic thresholds**

Metric alerts are often the best choice for infrastructure monitoring because they are fast and simple.

Important concepts:
- **Dimensions** let you split a metric by a property such as instance, API name, or response code.
- **Multi-resource alerts** let one rule monitor many resources of the same type.
- **Dynamic thresholds** use historical behavior to detect anomalies instead of a fixed number.

When to use dimensions:
- You want to alert only on a specific VM instance.
- You want to separate 4xx and 5xx HTTP responses.
- You want to monitor per-database or per-queue behavior.

When to use dynamic thresholds:
- Baselines vary by time of day or workload.
- A fixed threshold would create too many false positives.
- You want anomaly detection without manually tuning every threshold.

Example metric alert scenario:
- Scope: all VMs in a resource group.
- Condition: CPU percentage greater than 85%.
- Dimension: optional split by VM instance.
- Action: send to an operations action group.

**Log alerts**

Log alerts evaluate a KQL query on a schedule and fire when the query returns matching results or a threshold is exceeded.

Use log alerts when:
- You need complex conditions.
- You need to alert on patterns in logs, not just numeric metrics.
- You need to correlate multiple fields or tables.

Example log alert query:
```kusto
requests
| where success == false
| summarize FailedRequests = count() by bin(timestamp, 5m)
| where FailedRequests > 10
```

A safer pattern is to query the exact schema you are using:
```kusto
AzureDiagnostics
| where TimeGenerated > ago(15m)
| where Category == "AppServiceHTTPLogs"
| summarize FailedRequests = countif(toint(statusCode) >= 500) by bin(TimeGenerated, 5m)
| where FailedRequests > 10
```

Practical guidance:
- Keep queries efficient.
- Use a short evaluation window for near-real-time issues.
- Validate the query in Logs before turning it into an alert.
- Confirm the table schema and timestamp column before saving.

**Activity Log alerts**

Activity Log alerts monitor subscription-level control plane events. They are useful for administrative actions such as:
- Creating or deleting resources.
- Changing role assignments.
- Modifying policies.
- Starting or stopping certain operations.

Use Activity Log alerts when you want to know about:
- Administrative changes.
- Service issues reported in the Activity Log.
- Security-relevant subscription events.

Do not confuse Activity Log alerts with resource logs:
- Activity Log is about Azure control plane events at the subscription level.
- Resource logs are emitted by the resource itself and usually require diagnostic settings.

**Service Health versus Resource Health versus Activity Log**

These three are commonly tested together.

| Signal | Scope | What it tells you | Typical use |
|---|---|---|---|
| Activity Log | Subscription | Administrative and platform events | Track changes and operations |
| Service Health | Azure service / region | Azure platform incidents, planned maintenance, advisories | Know when Azure has a service issue |
| Resource Health | Individual resource | Health state of a specific resource | Determine whether a VM, app, or service is available |

How to think about them:
- Use **Service Health** when Azure has a regional or service-wide problem.
- Use **Resource Health** when you need the status of one resource.
- Use **Activity Log** when you need to audit actions or detect administrative changes.

**Alert processing, suppression, and deduplication**

Alert noise is a real operational problem, so Azure provides mechanisms to reduce it.

Key ideas:
- **Suppression** prevents alerts from notifying during a defined period or condition.
- **Deduplication** reduces repeated notifications for the same issue.
- **Stateful alerts** help avoid repeated firing for one ongoing incident.
- **Alert processing rules** can suppress or route alerts based on time, scope, or resource attributes.

Use these features when:
- You have planned maintenance windows.
- A noisy metric repeatedly crosses a threshold.
- Multiple resources generate the same symptom during a shared outage.

**Workbooks and visualization**

Workbooks provide interactive, parameterized reports over metrics and logs. They are useful for operational views, troubleshooting dashboards, and executive summaries.

Workbooks are helpful because they:
- Combine charts, tables, and text.
- Accept parameters such as subscription, resource group, or time range.
- Query logs and metrics in one place.
- Support repeatable operational views.

Azure portal dashboards can also be used for pinning charts and tiles, but for monitoring analysis, workbooks are usually the more flexible Azure Monitor-centric choice.

**Practical portal workflow for a monitoring scenario**

1. Identify the signal you need.
   - Use metrics for performance and availability.
   - Use logs for investigation.
   - Use Activity Log, Service Health, or Resource Health for platform and administrative events.

2. Centralize data.
   - Create or choose a Log Analytics workspace.
   - Configure diagnostic settings on the resource.
   - Enable AMA and DCRs for guest OS data where needed.

3. Validate ingestion.
   - Open Logs in the workspace.
   - Run a KQL query.
   - Confirm data arrives in the expected table.

4. Create the alert.
   - Choose metric, log, activity log, service health, or resource health.
   - Set the scope and condition.
   - Attach an action group.

5. Reduce noise.
   - Add alert processing rules.
   - Tune thresholds or use dynamic thresholds.
   - Use dimensions to target the right subset of data.

**Example scenario: VM CPU and failed service monitoring**

Suppose you manage a production VM hosting a web app.

Recommended setup:
- Use AMA to collect guest performance counters.
- Send platform metrics to Azure Monitor.
- Send VM logs and application logs to a Log Analytics workspace.
- Create a metric alert for CPU above 85%.
- Create a log alert for repeated application failures.
- Create a Resource Health alert for VM availability.
- Create an action group that emails the operations team and triggers a Logic App.

Example KQL for a custom failure pattern:
```kusto
AppTraces
| where TimeGenerated > ago(30m)
| where SeverityLevel >= 3
| summarize ErrorCount = count() by bin(TimeGenerated, 5m)
| where ErrorCount > 20
```

**Common exam distinctions**

- **Azure Monitor** is the umbrella service; **Application Insights** is for application telemetry.
- **Log Analytics workspace** stores and queries logs; it is not the same as a dashboard.
- **Diagnostic settings** export resource data; they do not replace alert rules.
- **Activity Log** tracks subscription events; it is not the same as resource logs.
- **Service Health** reports Azure service incidents; **Resource Health** reports the state of one resource.
- **AMA + DCR** is the modern guest data collection model.
- **Action groups** define the response; **alert rules** define the condition.

**Troubleshooting tips**

If data is missing:
- Confirm the resource supports the selected diagnostic categories.
- Check that the workspace is in the correct region or accessible scope.
- Verify permissions on the resource and workspace.
- Confirm the agent is installed and the DCR association exists.
- Wait for ingestion latency before assuming failure.

If alerts do not fire:
- Check the scope and time window.
- Validate the query or metric condition.
- Confirm the action group is attached.
- Review alert processing rules for suppression.
- Check whether the alert is stateful and already active.

If you see too many alerts:
- Increase the threshold or evaluation window.
- Use dimensions to narrow the scope.
- Add suppression for maintenance periods.
- Switch from static to dynamic thresholds where appropriate.

**Quick reference**

- **Metrics**: fast, numeric, near real-time.
- **Logs**: detailed, queryable, historical.
- **Activity Log**: subscription control plane events.
- **Service Health**: Azure service incidents and maintenance.
- **Resource Health**: health of a specific resource.
- **AMA**: modern agent for guest OS data.
- **DCR**: defines what guest data to collect and where to send it.
- **Action group**: notification and automation target.
- **Workbook**: interactive monitoring view.
- **Application Insights**: application performance monitoring.
