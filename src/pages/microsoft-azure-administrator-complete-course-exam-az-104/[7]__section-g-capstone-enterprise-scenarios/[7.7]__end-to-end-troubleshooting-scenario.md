## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Troubleshooting mindset**

End-to-end troubleshooting in Azure means you do not jump straight to a fix. You first identify whether the failure is in the **control plane** or the **data plane**, then narrow the scope by service, network path, identity, and configuration. In AZ-104 scenarios, the fastest path is usually: confirm the symptom, verify recent changes, inspect Azure-native diagnostics, and test the simplest possible path from client to resource.

A practical workflow is to ask:

- What is failing?
- Where does it fail?
- Is the failure consistent or intermittent?
- Did anything change recently?
- Is the issue in Azure, the client, or the network between them?

**Control plane vs data plane**

Understanding this distinction is essential because the fix depends on which plane is broken.

- **Control plane** issues affect management operations such as creating a VM, changing a network rule, or reading resource metadata.
- **Data plane** issues affect actual workload traffic such as RDP/SSH, HTTP access, blob reads, or database connections.
- A user may see a resource in the portal but still fail to use it if the data-plane permissions or network path are missing.

Examples:

- You can open a storage account in the portal but still fail to read blobs if you only have management access and not a data-plane role such as `Storage Blob Data Reader`.
- You can manage a VM in Azure but still be unable to connect to it if NSG rules, route tables, or guest OS settings block the traffic.

**A practical triage sequence**

Use a repeatable sequence so you do not miss obvious causes.

1. Confirm the exact error message, timestamp, and affected scope.
2. Check whether the issue is isolated to one user, one VM, one subnet, or one region.
3. Review recent changes in Azure Activity Log, deployment history, and configuration drift.
4. Validate identity and authorization.
5. Validate network reachability.
6. Validate service-specific settings and logs.
7. Apply the smallest safe fix, then retest.

**Azure Portal troubleshooting tools**

The Azure portal includes several built-in blades that are often the quickest way to identify the root cause.

- **Diagnose and solve problems** provides guided troubleshooting for many services, including VMs, networking, storage, and App Service.
- **Resource health** shows whether Azure platform issues are affecting the resource.
- **Activity log** shows control-plane operations such as create, update, delete, and failed deployments.
- **Metrics** and **logs** help you correlate symptoms with spikes, failures, or throttling.
- **Advisor** can surface recommendations for performance, reliability, security, and cost.

Use these tools early when the issue is unclear, because they often reveal whether the problem is platform-related or configuration-related.

**Network troubleshooting workflow**

Network issues are common in AZ-104 scenarios because many symptoms look like application failures but are actually path or policy problems.

Start by validating the path from source to destination.

- Confirm DNS resolution if the client uses a name instead of an IP address.
- Confirm the destination port and protocol.
- Check network security groups (NSGs), user-defined routes (UDRs), Azure Firewall rules, and load balancer configuration.
- Validate whether the target is public, private, or reachable only through a bastion or VPN path.

Useful Azure Network Watcher features include:

- **Connection troubleshoot** to test end-to-end connectivity between a source and destination.
- **IP flow verify** to determine whether an NSG allows or denies a specific flow.
- **Next hop** to see where traffic is routed from a VM.
- **Packet capture** to inspect traffic at the VM level.
- **NSG flow logs** to analyze allowed and denied flows over time.

A common exam pattern is to distinguish between “the VM is running” and “the VM is reachable.” A running VM can still be unreachable because of:

- NSG deny rules
- Guest OS firewall rules
- Incorrect route tables
- Missing public IP or incorrect load balancer rule
- Bastion not configured or not reachable
- DNS resolving to the wrong address

**Azure CLI examples for network checks**

Use CLI commands to validate configuration quickly.

```bash
# View effective NSG rules on a NIC
az network nic list-effective-nsg --resource-group rg-demo --name vm1Nic

# Check next hop from a VM NIC
az network watcher show-next-hop \
  --resource-group rg-demo \
  --vm vm1 \
  --source-ip 10.0.1.4 \
  --dest-ip 8.8.8.8

# Verify whether an NSG allows a specific flow
az network watcher test-ip-flow \
  --resource-group rg-demo \
  --vm vm1 \
  --direction Inbound \
  --protocol TCP \
  --local 10.0.1.4:3389 \
  --remote 203.0.113.25:50000
```

For connectivity testing, use a valid source context and a realistic destination. The exact parameters depend on the scenario, so do not memorize a single universal syntax.

```bash
# Example pattern: test connectivity from a source resource to a destination
az network watcher test-connectivity \
  --source-resource /subscriptions/<subId>/resourceGroups/rg-demo/providers/Microsoft.Compute/virtualMachines/vm1 \
  --dest-address www.microsoft.com \
  --dest-port 443
```

**Bastion and VM access validation**

Azure Bastion is often the correct answer when a VM has no public IP or when you want browser-based RDP/SSH access without exposing the VM directly.

Check these items when Bastion access fails:

- The Bastion host is deployed in the required `AzureBastionSubnet`.
- The subnet size is sufficient for Bastion requirements.
- The VM and Bastion are in compatible virtual network connectivity paths.
- NSG rules allow the required traffic between Bastion and the VM.
- The VM has a supported OS and the guest agent is healthy.
- The user has the right permissions to start a Bastion session.

If you cannot connect through Bastion, validate the path in layers:

1. Confirm Bastion is provisioned and healthy.
2. Confirm the VM is running.
3. Confirm the VM NIC and subnet are reachable from Bastion.
4. Confirm guest OS firewall and login permissions.
5. Confirm the target port is open for RDP or SSH.

**Storage troubleshooting**

Storage problems often split into management-plane visibility and data-plane access.

Common checks include:

- Storage account firewall rules
- Virtual network rules and service endpoints
- Private endpoint configuration and DNS resolution
- Shared access signature (SAS) validity
- Azure role-based access control (Azure RBAC) data-plane roles
- Key-based access if the application uses account keys
- Service health and throttling indicators

A user may be able to see the storage account in the portal but still fail to read blobs if the data-plane role is missing. For blob read access, a role such as `Storage Blob Data Reader` is required; management roles like `Contributor` do not grant blob data access by themselves.

Useful checks:

- Verify whether the storage account allows public network access.
- Verify whether the client is coming from an allowed IP or virtual network.
- Verify whether the application uses the correct endpoint, especially when private endpoints are enabled.
- Verify whether the SAS token is expired, malformed, or missing required permissions.
- Review storage metrics for authentication failures, throttling, or server errors.

Example CLI checks:

```bash
# Show storage account network rules
az storage account show \
  --name mystorageacct \
  --resource-group rg-demo \
  --query networkRuleSet

# List blob containers using Azure AD auth
az storage container list \
  --account-name mystorageacct \
  --auth-mode login
```

If the storage account uses a private endpoint, also validate DNS.

- The client must resolve the storage FQDN to the private IP.
- The private DNS zone must be linked to the correct virtual network.
- Split-brain DNS can cause the portal to work while the application fails, or the reverse.

**VM and guest OS troubleshooting**

When Azure-side checks look healthy, move into the guest OS.

Useful checks include:

- VM power state
- Boot diagnostics
- Serial console access
- OS firewall rules
- Service status
- Disk space
- CPU and memory pressure
- Application logs

Boot diagnostics can help you see startup failures, kernel panics, or login issues. If boot diagnostics is not enabled or not configured correctly for the VM type, the output may be limited.

Example commands:

```bash
# Show VM power state
az vm get-instance-view \
  --resource-group rg-demo \
  --name vm1 \
  --query instanceView.statuses

# Get boot diagnostics information
az vm boot-diagnostics get-boot-log \
  --resource-group rg-demo \
  --name vm1
```

Use the serial console when RDP or SSH is unavailable and you need direct OS-level access. It is especially useful for:

- Resetting local firewall rules
- Checking network configuration
- Inspecting startup scripts
- Recovering from login or service issues

**Load balancer and application reachability**

Load balancer issues often appear as intermittent or partial outages.

For Azure Load Balancer, validate:

- Backend pool membership
- Health probe configuration
- Load balancing rule configuration
- NSG rules on the backend subnet or NIC
- Guest OS service listening on the expected port

Health probes are configured as TCP or HTTP probes, and the backend must respond in the expected way for the probe to mark it healthy. If the probe fails, the load balancer removes the instance from rotation.

Common checks:

- The application is listening on the probe port.
- The probe path returns the expected HTTP status if using an HTTP probe.
- The backend VM allows traffic from the probe source.
- The backend service is not bound only to `localhost`.

**Azure Monitor, logs, and KQL**

Azure Monitor is central to end-to-end troubleshooting because it lets you correlate symptoms across metrics, logs, and alerts.

Use it to:

- Compare the failure time with CPU, memory, disk, and network metrics.
- Search logs for authentication failures, timeouts, or throttling.
- Build alerts for recurring issues.
- Correlate platform events with application behavior.

A simple Log Analytics query for VM performance might look like this:

```kusto
Perf
| where TimeGenerated > ago(1h)
| where ObjectName == "Processor" and CounterName == "% Processor Time"
| summarize AvgCPU = avg(todouble(CounterValue)) by Computer, bin(TimeGenerated, 5m)
| order by TimeGenerated desc
```

If `CounterValue` is stored as a string in your workspace or table variant, convert it with `todouble()` or the appropriate numeric conversion before aggregation.

Useful log sources include:

- `AzureActivity` for control-plane operations
- `Heartbeat` for VM agent presence
- `Perf` for performance counters
- `Event` for Windows event logs
- `Syslog` for Linux system logs
- Resource-specific diagnostic logs for storage, networking, and application services

**Decision points for exam scenarios**

Use these distinctions to choose the right tool quickly.

| Symptom | Likely area | First checks |
|---|---|---|
| Resource visible in portal but unusable | Data plane | RBAC data roles, firewall, SAS, private endpoint |
| Deployment fails | Control plane | Activity log, permissions, policy, quota |
| VM unreachable by RDP/SSH | Network or guest OS | NSG, route, Bastion, firewall, service status |
| Storage access denied | Identity or network | Data-plane role, firewall, DNS, SAS |
| App behind load balancer is down | Backend health | Probe, backend pool, listener, NSG |
| Intermittent connectivity | Routing or inspection | UDR, Azure Firewall, packet capture, flow logs |

**End-to-end walkthrough: VM cannot be reached**

This is a common AZ-104-style scenario.

1. Confirm the VM is running in the portal or with `az vm get-instance-view`.
2. Check whether the VM has a public IP, or whether you must use Bastion, VPN, or a jump host.
3. Validate NSG rules with `az network nic list-effective-nsg` or Network Watcher IP flow verify.
4. Check the route table and next hop if traffic is being sent to a firewall or appliance.
5. Confirm the guest OS firewall allows RDP on `3389` or SSH on `22`.
6. Use boot diagnostics or serial console if the OS is not responding.
7. Retest connectivity from the client or with Network Watcher connection troubleshoot.

If the VM is behind a load balancer, also verify the backend health probe and backend pool membership.

**End-to-end walkthrough: Storage blob access denied**

1. Confirm whether the failure is in the portal, CLI, or application.
2. Check whether the user has a data-plane role such as `Storage Blob Data Reader`.
3. Verify storage firewall rules and private endpoint/DNS configuration.
4. Check whether the SAS token is expired or missing permissions.
5. Review storage metrics and diagnostic logs for authentication failures.
6. Retest with a known-good identity or from an allowed network.

This sequence helps you separate identity problems from network problems, which is a common exam trap.

**Azure Advisor and recommendations**

Azure Advisor is useful when the issue is not a hard failure but a misconfiguration or inefficiency.

Look for recommendations related to:

- High availability
- Security
- Performance
- Cost
- Operational excellence

Advisor is not a replacement for diagnostics, but it can point you toward missing backups, underutilized resources, or risky configurations that contribute to incidents.

**Practical troubleshooting habits**

Good troubleshooting is as much about process as tools.

- Change one variable at a time.
- Record the exact error and timestamp.
- Validate from both the client side and the Azure side.
- Prefer built-in diagnostics before making disruptive changes.
- Recheck after every fix to confirm the symptom is gone.
- If the issue is intermittent, collect logs and metrics before restarting services.

**Common exam pitfalls**

- Assuming a resource is accessible because it appears in the portal.
- Confusing management-plane permissions with data-plane access.
- Ignoring DNS when private endpoints are involved.
- Checking only the Azure side and forgetting guest OS firewalls.
- Treating a load balancer probe failure as an application failure without checking backend health.
- Using a single CLI command as a universal connectivity test without validating the source resource and destination context.

**Quick reference checklist**

- Confirm the symptom and scope.
- Determine control plane or data plane.
- Check Activity Log and Resource Health.
- Validate identity and RBAC.
- Validate network path with Network Watcher.
- Check Bastion, VPN, or jump-host access if applicable.
- Inspect guest OS logs, boot diagnostics, and service status.
- Review storage firewall, private endpoint, and SAS settings when storage is involved.
- Use Azure Monitor and Advisor for correlation and recommendations.

**Key takeaway**

End-to-end troubleshooting in Azure is a layered process. Start broad, narrow systematically, and use the right tool for the right plane. In AZ-104 scenarios, the best answer is usually the one that validates identity, network, and service health in that order rather than guessing at a single root cause.
