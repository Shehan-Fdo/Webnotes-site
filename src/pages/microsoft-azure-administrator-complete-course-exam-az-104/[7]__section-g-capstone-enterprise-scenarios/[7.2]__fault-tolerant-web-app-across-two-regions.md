## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Two-region resilience model**

A fault-tolerant web app across two Azure regions combines compute, traffic routing, data replication, and monitoring so the application can survive a regional outage. The goal is not just uptime; it is controlled recovery with predictable data behavior and a clear failover/failback process.

For AZ-104, focus on the design tradeoffs:
- **Active-active**: both regions serve traffic at the same time.
- **Active-passive**: one region serves traffic while the other stays warm or cold for recovery.
- **Regional redundancy**: protects against a datacenter or zone issue inside one region.
- **Cross-region redundancy**: protects against a full regional outage.

A strong design usually combines:
- **Availability Zones** for intra-region resilience.
- **Geo-redundant data** for cross-region continuity.
- **Global traffic routing** for failover.
- **Monitoring and runbooks** for operational control.

**Resilience layers and when they matter**

Think in layers rather than products. If the app tier is redundant but the database is single-region, the solution is still vulnerable. If traffic can fail over but session state is local, users may lose sessions during a switchover.

- **Compute layer**: App Service, VM Scale Sets, AKS, or VMs deployed in both regions.
- **Traffic layer**: Azure Front Door or Azure Traffic Manager.
- **Data layer**: geo-replicated databases and storage redundancy.
- **State layer**: session, cache, and file handling.
- **Operations layer**: alerts, health probes, and failover procedures.

**Active-active vs active-passive**

Active-active improves availability and can reduce latency by serving users from the nearest healthy region. It also increases complexity because you must handle data synchronization and write conflicts.

Active-passive is simpler and often preferred when:
- The workload is mostly read-heavy.
- You want a clear primary region.
- You need lower operational complexity.
- The data store does not support safe multi-writer patterns.

Active-active is better when:
- You need higher availability and better regional distribution.
- The application is stateless or nearly stateless.
- You can use conflict-free data patterns or a single write region with read replicas.

> Be careful with multi-region writes. If both regions accept writes, you must define conflict resolution, consistency expectations, and user experience during partition events.

**Traffic routing choices: Front Door vs Traffic Manager**

Azure Front Door and Azure Traffic Manager both help route users to healthy endpoints, but they solve different problems.

| Feature | Azure Front Door | Azure Traffic Manager |
|---|---|---|
| Routing type | Layer 7 global HTTP(S) routing | DNS-based routing |
| Health awareness | Fast edge-based health probing | DNS health evaluation |
| Best for | Web apps, APIs, path-based routing, WAF | Simple global failover, endpoint selection |
| TLS termination | Yes | No |
| Path-based routing | Yes | No |
| WAF integration | Yes | No |
| Failover speed | Faster in practice | Depends on DNS caching and TTL |

Use **Azure Front Door** when you need:
- Global HTTP(S) load balancing.
- Path-based routing such as `/api` and `/images`.
- Web Application Firewall (WAF) protection at the edge.
- Faster user redirection after a health change.

Use **Azure Traffic Manager** when you need:
- DNS-based failover.
- Simple active-passive or priority routing.
- Non-HTTP endpoints or a lightweight global routing layer.
- A design where clients can tolerate DNS caching delays.

**DNS failover behavior and TTL**

Traffic Manager returns DNS answers based on the configured routing method, but clients and recursive resolvers cache those answers according to the Time To Live (TTL). That means failover is not instantaneous.

Key implications:
- Lower TTL can reduce failover delay, but increases DNS query volume.
- Cached DNS responses may keep some users on the old endpoint briefly.
- Failover is only as fast as the slowest resolver cache in the path.

For AZ-104 scenarios, remember:
- **Traffic Manager** is often preferred when DNS-based routing is acceptable and you want simple global failover.
- **Front Door** is preferred when you need faster HTTP(S) failover and edge features.

**Regional deployment patterns**

A common two-region web app pattern is:
- Region A: primary app and primary data services.
- Region B: secondary app and replicated data services.
- Global routing: Front Door or Traffic Manager.
- Health checks: endpoint probes and synthetic monitoring.

For regional load balancing inside one region, use:
- **Application Gateway** for Layer 7 regional routing.
- **Azure Load Balancer** for Layer 4 TCP/UDP distribution.

Example comparison:
- Use **Application Gateway** when you need host-based routing, TLS termination, or WAF in a region.
- Use **Load Balancer** when you need simple, high-performance distribution for VMs or VM Scale Sets.

**Example: regional Application Gateway**

A regional Application Gateway can distribute traffic across backend instances in one region while the global layer decides which region receives traffic.

```bash
az network application-gateway create \
  --name agw-eastus \
  --resource-group rg-web-eastus \
  --location eastus \
  --sku WAF_v2 \
  --capacity 2 \
  --vnet-name vnet-eastus \
  --subnet appgw-subnet
```

Typical use:
- Front Door or Traffic Manager sends users to the healthy region.
- Application Gateway balances traffic across app instances in that region.
- WAF policies protect the regional entry point if you are not using Front Door WAF.

**Example: regional Load Balancer**

```bash
az network lb create \
  --resource-group rg-web-eastus \
  --name lb-eastus \
  --sku Standard \
  --frontend-ip-name fe-ip \
  --backend-pool-name be-pool \
  --vnet-name vnet-eastus \
  --subnet app-subnet
```

Use this when:
- Your app tier runs on VMs or VM Scale Sets.
- You need simple health-based distribution.
- You do not need HTTP-aware routing.

**Front Door and WAF design**

Azure Front Door can sit in front of both regions and provide a single global entry point. In a secure design, pair it with a WAF policy to inspect inbound HTTP(S) traffic before it reaches the app.

Common WAF considerations:
- Block common web exploits such as SQL injection and cross-site scripting.
- Use managed rule sets where possible.
- Add custom rules for IP allowlists, geo-filtering, or rate limiting.
- Log WAF events to Log Analytics for investigation.

Practical design:
- Front Door routes to Region A and Region B origins.
- WAF policy protects the frontend.
- Health probes remove unhealthy origins from rotation.
- Backend origin groups point to regional app endpoints.

**Availability Zones and regional redundancy**

Availability Zones protect against datacenter-level failures inside a region. They do not replace cross-region redundancy; they complement it.

Use zones when:
- The region supports zone-redundant services.
- You want higher availability within the primary region.
- You need to reduce the chance that a single datacenter outage causes downtime.

Use two regions when:
- You need recovery from a full regional outage.
- Compliance or business continuity requires geographic separation.

A resilient design often looks like this:
- App tier deployed across multiple zones in each region.
- Data tier replicated across regions.
- Global routing across regions.
- Zone redundancy inside each region.

**Session state handling**

Session state is a common failure point in multi-region designs. If users are pinned to one region and that region fails, local session data may disappear.

Preferred approaches:
- **Stateless app design**: store session data outside the web tier.
- **Distributed session store**: use a shared cache or database-backed session mechanism.
- **Sticky sessions**: use only when necessary and understand the failover cost.

Tradeoffs:
- Sticky sessions simplify short-term state handling but reduce failover flexibility.
- Distributed session stores improve resilience but add latency and operational complexity.
- Stateless design is the most resilient and easiest to scale.

Practical guidance:
- Store authentication state in tokens or centralized identity systems.
- Keep user uploads in shared or replicated storage.
- Avoid in-memory-only session data for critical workflows.

**Data layer options and replication behavior**

The data layer determines whether failover is clean or risky. Different Azure services handle geo-redundancy differently.

Common patterns:
- **Azure SQL Database**: active geo-replication or failover groups.
- **Azure SQL Managed Instance**: failover groups for cross-region disaster recovery.
- **Azure Storage**: geo-redundant options such as GRS, RA-GRS, GZRS, and RA-GZRS where supported.
- **Cosmos DB**: multi-region replication with configurable consistency.
- **Azure Cache for Redis**: use carefully; cache is usually not the system of record.

For exam purposes, remember:
- Geo-redundant storage protects against regional loss, but not all services expose the same read/write behavior.
- Read-access geo-redundant options provide read access to the secondary in some services, but the secondary is not a general-purpose writable primary.
- Storage failover can take time and may not preserve all service semantics exactly as the primary.

**Azure Storage failover considerations**

Azure Storage redundancy is useful for blobs, files, queues, and tables, but behavior differs by service and redundancy mode.

Important points:
- **GRS** replicates data asynchronously to a paired region.
- **RA-GRS** allows read access to the secondary endpoint for some workloads.
- **GZRS** combines zone redundancy in the primary region with geo-replication.
- **RA-GZRS** adds read access to the secondary endpoint where supported.

Operational cautions:
- Secondary endpoints are typically read-only until an account failover occurs.
- Queue and table semantics may not behave like a transactional database.
- Application design should not assume immediate consistency across regions.
- Verify regional support before choosing a redundancy SKU.

Example storage account creation, with the caveat that SKU support depends on region and account type:

```bash
az storage account create \
  --name mystorageacct123 \
  --resource-group rg-web-eastus \
  --location eastus \
  --sku Standard_RAGZRS \
  --kind StorageV2
```

> Confirm that the target region supports the chosen redundancy SKU before deployment. Not every region or account configuration supports every geo-redundant option.

**Database failover groups**

For Azure SQL, failover groups are a common way to provide cross-region disaster recovery. They replicate databases to a secondary server and allow controlled failover.

Typical workflow:
1. Create the primary and secondary SQL servers in different regions.
2. Configure a failover group.
3. Add databases to the group using the supported current Azure CLI or portal workflow.
4. Point the application to the failover group listener endpoint.
5. Test failover and failback procedures.

Example pattern:

```bash
az sql failover-group create \
  --name fog-webapp \
  --partner-server sql-secondary \
  --resource-group rg-sql-primary \
  --server sql-primary \
  --failover-policy Automatic \
  --grace-period 1
```

Then add databases using the supported method for your Azure CLI version or the Azure portal. The exact command surface can vary, so verify the current syntax before scripting production changes.

**Monitoring and alerting for failover decisions**

Failover should be driven by health signals, not guesswork. Use Azure Monitor and Application Insights to detect service degradation and confirm recovery.

Useful signals:
- HTTP 5xx error rate.
- Increased response latency.
- Backend health probe failures.
- Dependency failures to the database or storage.
- Saturation of CPU, memory, or connection pools.
- Synthetic transaction failures from multiple regions.

Application Insights example query:

```kusto
requests
| where timestamp > ago(30m)
| summarize Total=count(), Failed=countif(success == false), FailureRate=100.0 * countif(success == false) / count()
  by bin(timestamp, 5m), cloud_RoleName
| order by timestamp desc
```

If you need to aggregate by role and time, alias the time bin so you can sort on it cleanly:

```kusto
requests
| where timestamp > ago(30m)
| summarize Total=count(), Failed=countif(success == false), FailureRate=100.0 * countif(success == false) / count()
  by cloud_RoleName, TimeBin = bin(timestamp, 5m)
| order by TimeBin desc
```

Alerting guidance:
- Trigger on sustained failure rates, not a single spike.
- Combine availability alerts with dependency alerts.
- Use action groups to notify operators and automation.
- Validate alerts from both regions so you can distinguish local from global issues.

**Failover runbook**

A runbook keeps failover repeatable and reduces human error. It should define who decides, what signals are required, and how to verify success.

1. Confirm the incident scope.
   - Check whether the issue is app-specific, regional, or global.
   - Validate health probes, logs, and synthetic tests.

2. Freeze risky changes.
   - Pause deployments and schema changes.
   - Stop background jobs that could create duplicate writes.

3. Verify data state.
   - Check replication lag.
   - Confirm the secondary database or storage copy is current enough for failover.
   - Identify any in-flight transactions that may be lost.

4. Switch traffic.
   - For Front Door, disable the unhealthy origin or adjust priority.
   - For Traffic Manager, mark the primary endpoint unhealthy or lower its priority.
   - For DNS-based routing, account for TTL delay.

5. Validate the secondary region.
   - Run smoke tests.
   - Confirm authentication, database access, and file access.
   - Check logs for errors after the switch.

6. Communicate status.
   - Notify stakeholders that failover is complete.
   - Record the time of the switch and any data loss window.

7. Plan failback.
   - Do not fail back immediately unless the primary region is stable and data is synchronized.

**Failback after the primary recovers**

Failback is often more dangerous than failover because data may have diverged while the secondary was active. Treat it as a controlled change.

Recommended failback steps:
1. Confirm the primary region is healthy and stable.
2. Re-establish replication from the active region to the recovered region.
3. Validate data consistency and replication lag.
4. Run smoke tests against the primary candidate.
5. Shift traffic gradually.
6. Monitor errors, latency, and user sessions.
7. Keep the previous active region ready as a fallback until confidence is high.

Important:
- If the app accepted writes in the secondary region, reconcile conflicts before switching back.
- If the app is stateless and uses a single-writer database, failback is simpler.
- Document the exact criteria for declaring the primary region ready.

**Storage, queues, and tables in a failover design**

Not all storage services behave the same way during regional recovery.

- **Blob storage**: commonly used for static content and uploads; geo-redundancy helps preserve data.
- **Azure Files**: useful for shared file access, but application design should account for failover behavior and access patterns.
- **Queues**: message duplication or reordering can occur during recovery scenarios; design consumers to be idempotent.
- **Tables**: treat as eventually consistent in geo-recovery planning and avoid assuming transactional cross-region behavior.

Design guidance:
- Make message processing idempotent.
- Use durable checkpoints.
- Avoid relying on a queue as the only source of truth.
- Store critical business state in a database with explicit replication semantics.

**Practical architecture example**

A common AZ-104-friendly design for a two-region web app is:

- **Region A**
  - App Service or VM-based web tier
  - Azure SQL primary
  - Storage account with geo-redundancy
  - Application Insights and Log Analytics

- **Region B**
  - Warm standby app tier
  - Azure SQL secondary
  - Replicated storage
  - Same monitoring stack

- **Global layer**
  - Azure Front Door with WAF, or Traffic Manager for DNS failover
  - Health probes to both regions
  - Alerting and automation for failover

This design works well because:
- Users reach a single global endpoint.
- The app can survive a regional outage.
- Monitoring can detect failures before manual intervention.
- Recovery steps are documented and testable.

**Exam focus points**

For AZ-104, be ready to choose the right service based on the requirement:
- Use **Front Door** for global HTTP(S) routing, WAF, and faster edge-based failover.
- Use **Traffic Manager** for DNS-based global failover and simple endpoint priority routing.
- Use **Application Gateway** for regional Layer 7 load balancing.
- Use **Load Balancer** for regional Layer 4 distribution.
- Use **Availability Zones** for intra-region resilience.
- Use **geo-redundant storage** and **failover groups** for data continuity.
- Use **Application Insights** and **Azure Monitor** to drive failover decisions.

Also remember the tradeoffs:
- Active-active increases complexity and conflict risk.
- Active-passive is simpler but may have slower recovery and lower utilization.
- DNS failover depends on TTL and client caching.
- Session state must be designed explicitly.
- Failback requires data validation, not just traffic redirection.

**Quick decision guide**

- Choose **Front Door** if you need global HTTP(S) routing and WAF.
- Choose **Traffic Manager** if DNS-based failover is enough and you want simple priority routing.
- Choose **Application Gateway** if you need regional HTTP routing and WAF.
- Choose **Load Balancer** if you need regional TCP/UDP distribution.
- Choose **active-active** if the app is stateless or conflict-tolerant.
- Choose **active-passive** if you want simpler operations and clearer data ownership.
