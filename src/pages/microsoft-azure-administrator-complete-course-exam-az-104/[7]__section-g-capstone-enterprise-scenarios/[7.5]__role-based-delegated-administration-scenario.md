## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Delegated administration in Azure**

Delegated administration means you give a user, group, or service principal only the permissions needed to manage a specific scope. In Azure, this is usually done with **Azure role-based access control (Azure RBAC)**, not with Microsoft Entra ID directory roles.

Azure RBAC controls access to Azure resources such as subscriptions, resource groups, and individual resources. Microsoft Entra ID roles control directory-level tasks such as managing users, groups, and app registrations. This distinction is a common AZ-104 exam trap.

- **Azure RBAC** governs resource management permissions.
- **Microsoft Entra ID roles** govern identity and directory administration.
- **Delegation** should usually be assigned to groups, not individual users, for easier lifecycle management.

**Azure RBAC scope hierarchy and inheritance**

Azure permissions are evaluated across a scope hierarchy. A role assignment at a higher scope flows down to lower scopes unless a more specific assignment changes the effective access.

The main scopes are:

- **Management group**
- **Subscription**
- **Resource group**
- **Resource**

A role assigned at a management group can apply to all subscriptions under that group, all resource groups in those subscriptions, and all resources beneath them. A role assigned at a subscription applies to all resource groups and resources in that subscription. A role assigned at a resource group applies only to resources in that group. A role assigned directly to a resource applies only to that resource.

Effective permissions are calculated by combining:

- All applicable role assignments at the current scope and parent scopes
- Any deny assignments, where present
- The actions allowed by each role definition
- Any conditions attached to the assignment, if supported

A simple mental model is: Azure starts at the target resource, walks up the scope chain, and aggregates allowed actions from every inherited assignment that applies.

```text
Management group
  └─ Subscription
      └─ Resource group
          └─ Resource
```

Example:

- You assign **Reader** at the management group.
- You assign **Contributor** at one resource group.
- A user can read everything in the management group hierarchy.
- The same user can also create and modify resources only in that specific resource group.
- They still cannot manage access unless a role grants `Microsoft.Authorization/*/write`.

> A lower-scope assignment does not remove access inherited from a higher scope unless a deny assignment or a more restrictive design changes the outcome.

**Role assignment prerequisites and authorization**

To create or change role assignments, you need the permission `Microsoft.Authorization/roleAssignments/write`. In practice, this permission is included in roles such as:

- **Owner**
- **User Access Administrator**

The **Owner** role can manage resources and access. The **User Access Administrator** role can manage access but not necessarily the resources themselves. This distinction matters in delegated administration scenarios where you want someone to grant access without giving them full control of the workload.

Common exam interpretation:

- If a user can assign roles, they likely have **Owner** or **User Access Administrator** at the relevant scope.
- If they can manage resources but cannot grant access, they may have **Contributor**.
- If they can view resources but not change them, they may have **Reader**.

**Built-in roles versus custom roles**

Azure includes many built-in roles, but built-in roles are not always precise enough for delegated administration. Use a **custom role** when built-in roles grant too much or too little access.

Use custom roles when you need to:

- Allow a help desk team to restart virtual machines but not delete them
- Let a security team read logs and security settings without changing workloads
- Delegate a narrow set of actions for a specific application or resource type
- Avoid giving broad Contributor access when only a few operations are needed

A custom role definition includes:

- `Actions` for allowed management-plane operations
- `NotActions` for exclusions
- `DataActions` for data-plane operations on supported services
- `NotDataActions` for exclusions from data-plane access

Example custom role definition:

```json
{
  "Name": "VM Operator Limited",
  "IsCustom": true,
  "Description": "Can start, stop, and restart virtual machines without deleting them.",
  "Actions": [
    "Microsoft.Compute/virtualMachines/start/action",
    "Microsoft.Compute/virtualMachines/deallocate/action",
    "Microsoft.Compute/virtualMachines/restart/action",
    "Microsoft.Compute/virtualMachines/read"
  ],
  "NotActions": [
    "Microsoft.Compute/virtualMachines/delete"
  ],
  "AssignableScopes": [
    "/subscriptions/00000000-0000-0000-0000-000000000000"
  ]
}
```

Practical guidance:

- Prefer built-in roles when they match the requirement exactly.
- Use custom roles when you need least privilege and the built-in role is too broad.
- Keep custom roles scoped as narrowly as possible in `AssignableScopes`.

**Azure portal role assignment workflow**

The Azure portal is a common exam scenario, so you should know the exact navigation and what each field means.

Typical path:

1. Open the target scope, such as a subscription, resource group, or resource.
2. Select **Access control (IAM)**.
3. Select **+ Add**.
4. Choose **Add role assignment**.
5. Select a role.
6. Select members.
7. Review and assign.

Field-level details you should expect:

- **Role**: the permission set, such as Reader, Contributor, or a custom role.
- **Members**: user, group, service principal, or managed identity.
- **Scope**: the resource container where the assignment applies.
- **Conditions**: optional restrictions for supported scenarios.
- **Review + assign**: final confirmation before the assignment is created.

Portal nuances that matter:

- If you open IAM at a resource group, the assignment applies to that resource group unless you change the scope.
- If you assign at a subscription, the role inherits to all child resource groups and resources.
- If you assign at a resource, the role is limited to that single resource.
- You may need to switch directories if the principal belongs to a different Microsoft Entra tenant.

Example portal reasoning:

- You want a team to manage only one storage account.
- Open the storage account, not the subscription.
- Go to **Access control (IAM)**.
- Assign the role at the storage account scope.
- This avoids granting access to other storage accounts in the subscription.

**Azure CLI and PowerShell examples**

You should be comfortable assigning roles from the command line because AZ-104 often tests operational understanding.

Azure CLI example:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Reader" \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app
```

Assign a role to a group:

```bash
az role assignment create \
  --assignee-object-id 11111111-1111-1111-1111-111111111111 \
  --assignee-principal-type Group \
  --role "Contributor" \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app
```

PowerShell example:

```powershell
New-AzRoleAssignment `
  -ObjectId "11111111-1111-1111-1111-111111111111" `
  -RoleDefinitionName "Reader" `
  -Scope "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app"
```

List role assignments at a scope:

```bash
az role assignment list \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app \
  --output table
```

**Group-based delegation best practices**

Assign roles to groups instead of individual users whenever possible. This makes access easier to manage and reduces the risk of orphaned permissions when people change teams.

Benefits of group-based delegation:

- **Simpler administration** when staff join or leave
- **Consistent access** for team members
- **Better auditability** because the assignment is tied to a team
- **Less operational overhead** during role changes

Recommended pattern:

- Create a Microsoft Entra security group for each job function.
- Assign the Azure RBAC role to the group.
- Add or remove users from the group as responsibilities change.

Example:

- `RG-App-Operators` group gets **Contributor** on `rg-app`.
- Individual engineers are added to the group.
- When an engineer leaves, you remove them from the group instead of editing multiple role assignments.

**Azure RBAC versus Microsoft Entra ID roles**

This distinction is essential for delegated administration questions.

| Area | Azure RBAC | Microsoft Entra ID roles |
|---|---|---|
| Primary purpose | Manage Azure resource access | Manage directory and identity tasks |
| Scope | Management group, subscription, resource group, resource | Tenant-wide directory scope |
| Examples | Reader, Contributor, Owner, User Access Administrator | User Administrator, Groups Administrator, Global Administrator |
| Typical use | VM, storage, network, and app resource permissions | Users, groups, app registrations, tenant settings |

Exam clue examples:

- If the task is to let someone create resource groups or manage VMs, think **Azure RBAC**.
- If the task is to let someone create users or reset passwords, think **Microsoft Entra ID roles**.
- If the task mentions “delegated administration” of Azure resources, it usually means **RBAC**, not directory roles.

**Effective access and troubleshooting**

When access does not behave as expected, verify the effective permissions rather than assuming the latest assignment is the only one that matters.

Use **Access control (IAM) > Check access** in the Azure portal to inspect what a user, group, or service principal can do at a given scope.

Typical troubleshooting flow:

1. Open the target resource, resource group, or subscription.
2. Go to **Access control (IAM)**.
3. Select **Check access**.
4. Search for the principal.
5. Review the roles and scopes that apply.

What to look for:

- Direct assignments at the current scope
- Inherited assignments from parent scopes
- Group memberships that grant additional access
- Conflicting expectations caused by a broader inherited role
- Missing permissions because the assignment was made at the wrong scope

CLI verification can also help:

```bash
az role assignment list --assignee user@contoso.com --all --output table
```

This shows assignments, but you still need to reason about inheritance and scope to determine effective access.

**Role assignment conditions and ABAC**

Azure role assignment conditions are part of **attribute-based access control (ABAC)**. They let you add constraints to a role assignment, such as limiting access to resources with certain tags or properties, where supported.

Use conditions when you need finer control than a plain role assignment provides. For example, you may want a user to read blobs only when a specific tag or resource attribute matches a rule.

Important exam-level guidance:

- Conditions are not universally supported across all Azure services.
- They are more advanced than basic RBAC and are not the first tool you reach for in standard delegation scenarios.
- If the requirement is simple delegated administration, a scope-based role assignment is usually the correct answer.

> Do not assume ABAC is available for every resource type. If the scenario does not explicitly require conditional access to resource attributes, prefer standard RBAC.

**Common delegated administration patterns**

A few patterns appear frequently in real environments and exam questions.

- **Help desk access**
  - Assign a group the minimum role needed to reset or view the relevant resource.
  - Avoid subscription-wide Contributor unless required.

- **Application team access**
  - Assign Contributor or a custom role at the application resource group.
  - Keep production and nonproduction separated by scope.

- **Security operations access**
  - Use Reader or a custom read-only role for monitoring and investigation.
  - Add specific log-reading permissions if needed.

- **Access management delegation**
  - Use User Access Administrator when a team must grant access but should not manage workloads.
  - Use Owner only when both resource and access management are required.

**Practical exam cues**

Look for these phrases and map them to the right concept:

- “Grant access to Azure resources” → Azure RBAC
- “Manage users and groups” → Microsoft Entra ID roles
- “Only this resource group” → scope at resource group
- “All resources in the subscription” → scope at subscription
- “Least privilege” → custom role or narrower built-in role
- “Team-based administration” → assign to a group
- “Can assign roles but not manage resources” → User Access Administrator
- “Can manage resources and access” → Owner

**Quick reference for scope behavior**

- Management group assignments inherit to all child subscriptions.
- Subscription assignments inherit to all child resource groups and resources.
- Resource group assignments inherit to all resources in that group.
- Resource assignments apply only to that resource.
- Effective access is the sum of applicable assignments, not just the most recent one.

**Key takeaways**

Azure delegated administration is about giving the right permissions at the right scope. For AZ-104, focus on the difference between Azure RBAC and Microsoft Entra ID roles, understand inheritance across scopes, and know when to use built-in versus custom roles. Always verify access with **Check access** when troubleshooting, and prefer group-based assignments for maintainable administration.
