## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Scenario overview**

A multi-tier application separates responsibilities across layers such as presentation, application, and data. In Azure, you choose services and network controls based on scale, security, availability, and operational complexity. For AZ-104, you should be able to explain why a tier uses a VM, VM Scale Set (VMSS), App Service, Azure SQL, or a private endpoint, and how the pieces connect securely.

A common exam pattern is a web front end exposed to the internet, an application tier that processes business logic, and a data tier that stores state. The exact shape varies by workload. Some applications go directly from web to database, while others use a dedicated app tier. Do not assume every design must include all three tiers; choose the simplest architecture that satisfies the requirements.

**Core design goals**

- **Security**: minimize public exposure and use private connectivity where possible.
- **Availability**: place critical tiers across fault domains or availability zones when supported.
- **Scalability**: scale stateless tiers horizontally and keep state in managed services.
- **Operability**: centralize logging, monitoring, patching, and backup.
- **Cost control**: avoid overengineering; match service choice to workload needs.

**Typical Azure service choices**

| Tier | Common Azure options | When to use |
|---|---|---|
| Presentation/web | App Service, VMSS, Azure Container Apps, AKS | Use App Service for managed web apps; use VMSS or containers when you need more control or custom runtime behavior. |
| Application/business logic | App Service, VMSS, AKS, Functions | Use App Service or Functions for managed compute; use VMSS or AKS for custom hosting or microservices. |
| Data | Azure SQL Database, Azure SQL Managed Instance, SQL Server on Azure VM, Azure Database for PostgreSQL/MySQL | Use PaaS for reduced admin overhead; use SQL Server on VM when you need OS-level control or legacy compatibility. |

**Choosing the compute platform**

For AZ-104, the main decision is usually between managed PaaS and IaaS.

- **App Service** is a strong choice for web apps and APIs when you want built-in scaling, deployment slots, and less OS administration.
- **Virtual Machines** fit legacy apps, custom agents, or software that needs full OS access.
- **VMSS** is appropriate when you need multiple identical VM instances and automatic scaling.
- **AKS** is usually selected for containerized microservices, but it adds operational complexity and is less common for simple exam scenarios.
- **Azure Functions** work well for event-driven components, background jobs, and lightweight APIs.

**Network segmentation**

A secure multi-tier design uses separate subnets for each tier and controls traffic with Network Security Groups (NSGs). You typically place the web tier in a subnet that can receive internet traffic through a public entry point, while the application and data tiers remain private.

- **NSGs** filter traffic at the subnet or NIC level.
- **Route tables** can force traffic through a firewall or network virtual appliance.
- **Private endpoints** expose supported PaaS services privately inside your virtual network.
- **Service endpoints** are simpler than private endpoints but do not provide the same private IP-based access model.
- **Azure Firewall** or a **network virtual appliance (NVA)** centralizes inspection and egress control.

> Use Azure Firewall or an NVA when you need centralized outbound filtering, threat intelligence, or explicit traffic inspection between tiers. NSGs alone do not provide full firewall capabilities.

**Public entry point options**

For internet-facing web applications, the public entry point is usually one of these:

- **Application Gateway** for HTTP/HTTPS workloads that need Layer 7 features such as TLS termination, path-based routing, host-based routing, and Web Application Firewall (WAF).
- **Azure Load Balancer** for Layer 4 TCP/UDP distribution, such as non-HTTP services or when you need simple port-based load balancing.
- **Front Door** for global HTTP(S) acceleration and edge routing across regions.

For a typical web application, Application Gateway is the more appropriate public entry point because it understands HTTP(S). Azure Load Balancer does not provide URL routing or TLS termination.

**Application Gateway details**

Application Gateway is often used in front of the web tier when you need application-aware routing.

- **Listener**: accepts incoming traffic on a frontend IP and port.
- **Backend pool**: contains the targets, such as VM private IPs or App Service endpoints.
- **Health probe**: checks whether backend instances are healthy.
- **Routing rule**: connects a listener to a backend pool and optional HTTP settings.
- **HTTP settings**: define backend port, protocol, cookie-based affinity, and timeout behavior.

Cookie-based affinity exists, but do not treat it as the main reason to choose Application Gateway. Use it only when a specific application requires session stickiness and you cannot externalize state.

**Load Balancer details**

Azure Load Balancer is useful when you need simple, high-performance distribution at Layer 4.

- It supports TCP and UDP.
- It uses **health probes** to determine backend availability.
- It distributes traffic across backend pool members.
- It does not inspect HTTP paths or host headers.
- It does not terminate TLS.

A basic backend setup includes:

1. Create the backend pool.
2. Add VM NICs or IP addresses to the pool.
3. Configure a health probe on the service port.
4. Create a load-balancing rule that maps frontend port to backend port.
5. Ensure NSG rules allow the probe and application traffic.

Example Azure CLI for a standard load balancer rule:

```bash
az network lb create \
  --resource-group rg-app \
  --name lb-web \
  --sku Standard \
  --frontend-ip-name fe-web \
  --backend-pool-name be-web \
  --vnet-name vnet-app \
  --subnet subnet-web \
  --public-ip-address pip-web

az network lb probe create \
  --resource-group rg-app \
  --lb-name lb-web \
  --name probe-http \
  --protocol tcp \
  --port 80

az network lb rule create \
  --resource-group rg-app \
  --lb-name lb-web \
  --name rule-http \
  --protocol tcp \
  --frontend-port 80 \
  --backend-port 80 \
  --frontend-ip-name fe-web \
  --backend-pool-name be-web \
  --probe-name probe-http
```

**Subnet and NSG design**

A common pattern is one subnet per tier.

- **Web subnet**: allows inbound from the public entry point and outbound to the app tier.
- **App subnet**: allows inbound only from the web subnet and outbound to the data tier or required services.
- **Data subnet**: allows inbound only from the app subnet or private endpoints, depending on the data platform.

Example NSG rules for a web-to-app flow:

```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-app \
  --name Allow-Web-To-App \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.10.1.0/24 \
  --source-port-ranges '*' \
  --destination-address-prefixes 10.10.2.0/24 \
  --destination-port-ranges 8080
```

Use explicit source and destination prefixes whenever possible. Avoid broad `Any` rules unless the workload truly requires them.

**Outbound connectivity**

Private subnets still need controlled outbound access for updates, package downloads, and external APIs.

- **NAT Gateway** provides predictable outbound internet access for resources in a subnet.
- **Azure Firewall** provides centralized outbound filtering and logging.
- **User-defined routes (UDRs)** can force outbound traffic through the firewall or NVA.
- **Private endpoints** reduce the need for public outbound access to Azure PaaS services.

A common design is:

- Web and app subnets use a NAT Gateway for general outbound internet access.
- Sensitive workloads use Azure Firewall for inspection and allow-listing.
- Data tier subnets avoid direct internet access unless required for patching or backup operations.

**Private access to PaaS data services**

Private Link and private endpoints are important for securing data-tier access.

- A **private endpoint** gives a PaaS resource a private IP in your virtual network.
- A **private DNS zone** resolves the service name to the private IP.
- Access is limited to the virtual network and connected networks that can reach the endpoint.
- Not every Azure service supports Private Link, so verify support before designing around it.

For Azure SQL Database, a typical setup includes:

1. Create the SQL server and database.
2. Disable public network access if the design requires private-only access.
3. Create a private endpoint in the application VNet.
4. Link the private DNS zone `privatelink.database.windows.net`.
5. Configure the app to use managed identity or a secure secret store.

Example Azure CLI for a private endpoint to Azure SQL:

```bash
az network private-endpoint create \
  --resource-group rg-app \
  --name pe-sql \
  --vnet-name vnet-app \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Sql/servers/sqlprod \
  --group-id sqlServer \
  --connection-name pe-sql-conn
```

**Data tier choices**

Choose the data platform based on control, compatibility, and administration needs.

- **Azure SQL Database**: best for modern relational apps that fit a managed database model.
- **Azure SQL Managed Instance**: useful when you need near-full SQL Server compatibility with managed service benefits.
- **SQL Server on Azure VM**: best when you need OS access, SQL Agent behavior, custom extensions, or legacy dependencies.
- **Azure Database for PostgreSQL/MySQL**: appropriate for non-SQL Server workloads.

Tradeoffs to remember:

- Azure SQL Database has the least administrative overhead.
- Managed Instance offers more compatibility than Azure SQL Database but usually costs more.
- SQL Server on a VM gives the most control but also the most patching, backup, and HA responsibility.

**Authentication and secrets**

Prefer identity-based access over stored credentials.

- Use **managed identity** for app-to-database authentication when supported.
- Store secrets in **Azure Key Vault** if the application cannot use managed identity directly.
- Rotate secrets and avoid embedding connection strings in code or VM images.

Example connection string pattern for SQL authentication is still common, but managed identity is preferred where supported:

```text
Server=tcp:sqlprod.database.windows.net,1433;Database=appdb;Encrypt=True;TrustServerCertificate=False;
```

**Session state and scaling**

State management affects whether you can scale out safely.

- Keep web and app tiers **stateless** when possible.
- Store session state in a shared cache such as **Azure Cache for Redis** if sticky sessions are not sufficient.
- Use cookie-based affinity only for limited cases where the application cannot externalize session state.
- Prefer token-based authentication for APIs and modern web apps.

Sticky sessions can be acceptable for short-term compatibility, but they reduce load balancing flexibility and complicate failover. For exam scenarios, choose Redis or another shared state store when the requirement is reliable scale-out.

**Availability design**

Availability choices depend on the tier and the service.

- **Availability Sets** protect VMs from single rack failures within a datacenter.
- **Availability Zones** protect workloads from datacenter-level failures within a region.
- **Zone-redundant services** spread instances across zones when supported.
- **Geo-redundancy** protects against regional outages for selected services.

Use this rule of thumb:

- Choose **availability sets** when the region does not support zones or when the workload is legacy VM-based and only needs intra-datacenter fault isolation.
- Choose **availability zones** when the workload is critical and the service supports zone deployment.
- Use **zone-redundant PaaS** for managed services whenever possible.

**Backup and disaster recovery**

A multi-tier application needs backup and recovery planning for the data tier and, in some cases, the compute tier.

- Use **Azure Backup** for VM backups and file-level recovery.
- Use built-in backup and point-in-time restore for Azure SQL Database.
- Use long-term retention when compliance requires it.
- Define **RPO** (Recovery Point Objective) and **RTO** (Recovery Time Objective) before choosing the backup strategy.
- Use **Azure Site Recovery** for VM-based disaster recovery when you need failover to another region.

For data tiers, the backup strategy should match the business impact:

- Low RPO requirements favor frequent backups and transaction log retention.
- Low RTO requirements favor warm standby or active-active patterns.
- Legacy SQL Server on VM often needs both VM-level and SQL-level backup planning.

**Monitoring and troubleshooting**

Monitor each tier separately and correlate signals across the application.

- Use **Azure Monitor** for metrics and alerts.
- Use **Log Analytics** for centralized log queries.
- Use **Application Insights** for application performance and request tracing.
- Use **Network Watcher** for connection troubleshooting and packet-level diagnostics.

Useful troubleshooting tools:

```bash
az network watcher test-connectivity \
  --source-resource /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm-web01 \
  --dest-address 10.10.2.4 \
  --dest-port 8080
```

```bash
az network watcher show-next-hop \
  --resource-group rg-app \
  --vm vm-web01 \
  --source-ip 10.10.1.4 \
  --dest-ip 10.10.3.5
```

For metrics, use the resource-specific command for the service you are checking. For example, VM metrics can be queried with Azure Monitor or the portal, but CLI syntax varies by resource type and API version. In study notes, focus on the concept: verify CPU, memory, disk latency, request rate, backend health, and dependency failures.

**Secure administration**

Private subnets still need admin access. Do not expose management ports broadly.

- Use **Azure Bastion** for browser-based RDP and SSH to VMs without public IPs.
- Use Just-In-Time access when available for temporary elevation.
- Restrict RDP/SSH with NSGs and avoid permanent public management endpoints.
- Use jump hosts only when Bastion is not available or not suitable.

Example Bastion deployment pattern:

- Place Bastion in a dedicated subnet named `AzureBastionSubnet`.
- Keep VMs in private subnets with no public IPs.
- Connect through the Azure portal or supported client workflows.

**End-to-end deployment example**

A practical AZ-104-style design for a small production web app might look like this:

- **Web tier**: App Service or two VMs behind Application Gateway.
- **App tier**: VMSS or App Service API, reachable only from the web tier.
- **Data tier**: Azure SQL Database with private endpoint and private DNS.
- **Admin access**: Azure Bastion for any VMs.
- **Outbound**: NAT Gateway for general egress, or Azure Firewall for inspection.
- **Monitoring**: Application Insights, Log Analytics, and Azure Monitor alerts.

Example flow:

1. Users access the public Application Gateway.
2. Application Gateway routes HTTPS traffic to the web tier.
3. The web tier calls the app tier over a private subnet.
4. The app tier accesses Azure SQL through a private endpoint.
5. Logs and metrics flow to Log Analytics and Application Insights.
6. Administrators connect to private VMs through Azure Bastion.

A simplified Bicep-style sketch:

```bicep
resource vnet 'Microsoft.Network/virtualNetworks@2023-11-01' = {
  name: 'vnet-app'
  location: resourceGroup().location
  properties: {
    addressSpace: {
      addressPrefixes: [
        '10.10.0.0/16'
      ]
    }
    subnets: [
      {
        name: 'subnet-web'
        properties: {
          addressPrefix: '10.10.1.0/24'
          networkSecurityGroup: {
            id: nsgWeb.id
          }
        }
      }
      {
        name: 'subnet-app'
        properties: {
          addressPrefix: '10.10.2.0/24'
          networkSecurityGroup: {
            id: nsgApp.id
          }
        }
      }
      {
        name: 'subnet-private-endpoints'
        properties: {
          addressPrefix: '10.10.3.0/24'
        }
      }
    ]
  }
}
```

This example is intentionally simplified. In a real deployment, you would also add the Application Gateway subnet, Bastion subnet, route tables, private DNS zones, and monitoring resources.

**Exam decision guidance**

Use these rules when choosing services in AZ-104 scenarios:

- Choose **Application Gateway** for internet-facing HTTP(S) apps that need Layer 7 routing or WAF.
- Choose **Load Balancer** for non-HTTP traffic or simple Layer 4 distribution.
- Choose **App Service** when the app can run as a managed web app or API.
- Choose **VMSS** when you need scalable identical VMs.
- Choose **Azure SQL Database** when the app fits a managed relational database.
- Choose **Managed Instance** when SQL compatibility matters more than cost.
- Choose **SQL Server on Azure VM** when legacy requirements force full control.
- Choose **Azure Bastion** for secure admin access to private VMs.
- Choose **Azure Firewall** when you need centralized inspection and egress control.
- Choose **Private Endpoint** when a PaaS service should be reachable privately from your network.

**Common pitfalls**

- Do not place every tier on a public IP.
- Do not assume sticky sessions solve state management.
- Do not use Azure Load Balancer when you need HTTP path routing.
- Do not rely on NSGs alone for full perimeter security.
- Do not expose SQL databases publicly unless the requirement explicitly allows it.
- Do not overuse availability zones if the service or region does not support them.
- Do not choose a VM-based data tier unless the workload truly needs OS-level control.

**Quick reference**

- **Web tier**: public entry point, stateless, horizontally scalable.
- **App tier**: private processing layer, often behind NSGs and possibly a firewall.
- **Data tier**: private access, managed backups, strong authentication.
- **Admin access**: Bastion or jump host, not broad public RDP/SSH.
- **Outbound**: NAT Gateway or Azure Firewall.
- **Monitoring**: Azure Monitor, Log Analytics, Application Insights.
