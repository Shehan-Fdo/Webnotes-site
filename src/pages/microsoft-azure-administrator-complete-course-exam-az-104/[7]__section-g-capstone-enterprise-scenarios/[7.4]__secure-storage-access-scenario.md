## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Secure storage access overview**

Azure Storage security is usually a combination of identity, network, and data-plane controls. In AZ-104 scenarios, you often need to decide whether access should be granted through Microsoft Entra ID, shared keys, SAS (Shared Access Signature), private networking, or a mix of these controls.

The exam commonly tests whether you can choose the right access model for the workload:

- **Identity-based access** for users, groups, and managed identities.
- **Key-based access** for broad administrative access or legacy apps.
- **SAS-based access** for time-bound, scoped delegation.
- **Network restrictions** to prevent public exposure.

A secure design usually follows the principle of least privilege and layers controls so that a valid credential alone is not enough if the network path is blocked.

**Storage account security controls**

A storage account has several important settings that affect exposure and access. You should know where they fit in the portal and what they change.

- **Allow blob public access** controls whether containers can be made anonymously readable.
- **Public network access** controls whether the storage account can be reached from the public internet.
- **Firewall and virtual network rules** restrict which networks can connect.
- **Secure transfer required** forces HTTPS for supported endpoints.
- **Minimum TLS version** limits older protocol versions.
- **Shared key access** can be disabled to force Microsoft Entra ID-based authorization for supported services.

A common exam trap is assuming that turning on authentication alone makes storage private. It does not. If the endpoint is still public and anonymous blob access is allowed, data may still be exposed.

**Authentication and authorization models**

Azure Storage supports multiple ways to authenticate, and the right choice depends on the service and scenario.

- **Microsoft Entra ID** is preferred for Azure Blob and Azure Queue access when supported by the client and service.
- **Storage account keys** grant full access to the account’s data plane and should be treated as highly sensitive secrets.
- **SAS tokens** delegate limited access without sharing the account key.
- **Managed identities** let Azure resources access storage without storing secrets.

For exam purposes, distinguish between authentication and authorization:

- **Authentication** proves who or what is calling.
- **Authorization** determines what that identity can do.

For example, a user may authenticate successfully with Microsoft Entra ID but still be denied because they lack the correct role assignment or share permission.

**Shared Access Signatures**

A SAS is a signed token that grants limited access to storage resources for a defined time and scope. It is one of the most tested storage security concepts in AZ-104.

There are two major SAS types:

| SAS type | Scope | Typical use |
|---|---|---|
| **Service SAS** | A single storage service such as Blob, File, Queue, or Table | Grant access to specific objects in one service |
| **Account SAS** | Multiple services and resource types within the storage account | Delegate broader access across services |

Key SAS concepts:

- **Start and expiry time** define the validity window.
- **Permissions** limit actions such as read, write, delete, list, or add.
- **IP range** can restrict where the SAS can be used from.
- **HTTPS only** should be enforced whenever possible.
- **Stored access policy** can centralize revocation for service SAS in supported services.

A SAS is not the same as a shared key. The SAS is derived from a key or user delegation context, but the token itself is what the client uses.

> Use SAS when you need temporary, scoped access for a partner, application, or download link. Use Microsoft Entra ID when the client can authenticate directly and you want stronger governance.

**User delegation SAS versus account SAS**

For Blob storage, a user delegation SAS is often the preferred modern option because it is signed with Microsoft Entra credentials rather than the storage account key.

- **User delegation SAS**
  - Uses Microsoft Entra ID to obtain a user delegation key.
  - Is limited to Blob storage scenarios.
  - Reduces reliance on account keys.
- **Account SAS**
  - Uses the storage account key.
  - Can target more than one service, depending on the token.
  - Is broader and therefore riskier if leaked.

For AZ-104, remember that user delegation SAS is generally the more secure choice for Blob access when the scenario supports it.

**Practical SAS examples**

A service SAS for a blob container is often used to grant temporary read access to a specific container or blob.

```bash
az storage container generate-sas \
  --account-name mystorageacct \
  --name reports \
  --permissions rl \
  --expiry 2026-05-01T00:00Z \
  --https-only \
  --auth-mode login
```

A SAS URL typically looks like this:

```text
https://mystorageacct.blob.core.windows.net/reports?sv=2020-10-02&ss=b&srt=sco&sp=rl&se=2026-05-01T00:00Z&spr=https&sig=...
```

Important SAS fields:

- `sv` is the storage service version used to sign the token.
- `sp` defines permissions.
- `se` defines expiry.
- `spr=https` enforces HTTPS.
- `sig` is the signature.

For exam prep, focus on the meaning of the fields rather than memorizing a specific `sv` value.

**Storage account keys and key rotation**

Storage account keys provide full data-plane access to the account. Because of that, they are powerful but risky.

- **Key1** and **Key2** support rotation without downtime if you switch clients carefully.
- Regenerate one key at a time.
- Update dependent applications before rotating the key in use.
- Prefer Microsoft Entra ID or SAS where possible to reduce key exposure.

A typical Azure CLI rotation command is:

```bash
az storage account keys renew \
  --account-name mystorageacct \
  --resource-group rg-storage \
  --key primary
```

Operationally, the safe pattern is:

1. Identify which applications use the current key.
2. Regenerate the unused key first.
3. Update clients to the new key.
4. Regenerate the old key after all clients are switched.

> If you rotate the active key before updating clients, applications that depend on it will fail immediately.

**Microsoft Entra ID authorization for storage**

For Blob and Queue access, Microsoft Entra ID can replace shared keys in many scenarios. You assign Azure roles to users, groups, service principals, or managed identities.

Common data-plane roles include:

- **Storage Blob Data Reader** for read-only blob access.
- **Storage Blob Data Contributor** for read/write/delete blob access.
- **Storage Queue Data Contributor** for queue operations.
- **Storage File Data SMB Share Reader** and related roles for Azure Files identity scenarios.

A key exam point is that management-plane roles and data-plane roles are different:

- **Owner**, **Contributor**, and **Reader** manage Azure resources.
- **Storage data roles** control access to the data inside the storage account.

A user with Contributor on the storage account may still be unable to read blobs unless they also have a data role.

**RBAC scope and assignment examples**

You can assign roles at different scopes depending on how broadly access should apply.

- **Subscription scope** affects many resources and is usually too broad for storage data access.
- **Resource group scope** is useful when multiple related storage accounts share the same access pattern.
- **Storage account scope** is the most common choice for a single account.
- **Container or share scope** is useful for tighter control when supported.

Example Azure CLI assignment:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/<subId>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct
```

For portal-based configuration:

1. Open the storage account.
2. Select **Access control (IAM)**.
3. Choose **Add role assignment**.
4. Select the appropriate data role.
5. Choose the user, group, or managed identity.
6. Save and verify the assignment at the correct scope.

**Azure Files access models**

Azure Files is often tested because it supports both SMB (Server Message Block) and REST-based access patterns, and the security model differs from Blob storage.

There are two broad access approaches:

- **Shared key or SAS** for classic file share access.
- **Identity-based access** for SMB access using Microsoft Entra ID, on-premises AD DS (Active Directory Domain Services), or Microsoft Entra Domain Services depending on the scenario.

For SMB access, permissions are layered:

- **Share-level permissions** are granted through Azure role assignments or share access controls.
- **NTFS-style permissions** are enforced inside the file system for SMB clients.
- Effective access is the combination of both.

This means RBAC alone is not the full story for Azure Files. A user may have share access but still be blocked by file/folder ACLs, or vice versa.

**Azure Files identity-based access**

Identity-based access is the preferred enterprise pattern when you need user-level control over SMB shares.

Common scenarios:

- **Microsoft Entra Domain Services** for cloud-only SMB identity integration.
- **AD DS** for hybrid environments with existing domain identities.
- **Microsoft Entra Kerberos** in supported scenarios for modern identity-based SMB access.

Practical considerations:

- The storage account must be configured for the chosen identity source.
- The client must be domain-joined or otherwise able to obtain the required Kerberos tickets or credentials.
- Share permissions and file ACLs must both be configured correctly.

Portal workflow example:

1. Open the storage account.
2. Go to **File shares** and select the share.
3. Configure **Identity-based access** or the relevant authentication method.
4. Assign the appropriate Azure role for share-level access.
5. Set NTFS permissions on folders and files from a domain-joined client.

**Azure Files SMB mount example**

A Linux client can mount an Azure file share over SMB when the environment supports SMB 3.x and the authentication method is configured correctly.

```bash
sudo mkdir -p /mnt/azurefiles
sudo mount -t cifs //mystorageacct.file.core.windows.net/share1 /mnt/azurefiles \
  -o vers=3.0,username=mystorageacct,password='<storage-key>',dir_mode=0777,file_mode=0777,serverino
```

Notes:

- `vers=3.0` ensures SMB 3.x negotiation.
- Key-based mounts use the storage account name as the username.
- Additional options may be required depending on the client OS and security posture.
- For identity-based access, the mount process differs and typically relies on domain authentication rather than a storage key.

**Network access controls**

Storage security is not only about credentials. Network controls often decide whether a request can reach the service at all.

Two common network restriction models are:

- **Service endpoints**
- **Private endpoints**

They solve different problems.

| Feature | Service endpoint | Private endpoint |
|---|---|---|
| Traffic path | Stays on Microsoft backbone after leaving the VNet | Uses a private IP in your VNet |
| Public endpoint | Still exists | Can be disabled for private access patterns |
| DNS requirement | Minimal | Requires private DNS integration |
| Typical use | Restrict access to selected VNets/subnets | Fully private access to storage |

Service endpoints are simpler and are often used when you want to allow a subnet to reach a storage account without exposing it broadly. Private endpoints are more secure and are preferred when you want the storage account reachable only through private IP space.

**Private endpoint setup**

A private endpoint maps a storage service to a private IP address in your virtual network. For Blob, File, Queue, and Table, you can create separate private endpoints depending on the services you need.

Portal steps:

1. Open the storage account.
2. Select **Networking**.
3. Choose **Private endpoint connections** and create a new private endpoint.
4. Select the target service, such as Blob or File.
5. Pick the virtual network and subnet.
6. Configure private DNS integration so the storage FQDN resolves to the private IP.
7. Approve the connection if manual approval is required.

DNS is critical here. Without private DNS, clients may still resolve the public endpoint and fail or bypass the intended private path.

**Service endpoints versus private endpoints**

Service endpoints are useful when you want to allow access from a trusted subnet while still using the public endpoint. Private endpoints are better when you want to eliminate public exposure.

- Use **service endpoints** when:
  - You need a simpler setup.
  - You are comfortable with public endpoint exposure plus subnet restriction.
  - You want to restrict access by VNet/subnet without private DNS complexity.
- Use **private endpoints** when:
  - You need the strongest network isolation.
  - You want private IP-based access.
  - You need to support strict compliance or zero-trust network design.

**Public access and anonymous blob scenarios**

Blob storage can be exposed publicly if container-level public access is allowed. This is controlled by the storage account setting and the container access level.

- **Allow blob public access** must be enabled at the account level for anonymous container/blob access to be possible.
- Container access can then be set to public read access where allowed.
- If the account setting disables public access, anonymous blob access is blocked even if a container is configured for public access.

This is a common exam control because it is easy to overlook. If a scenario says “no anonymous access,” check both the account-level setting and the container-level access level.

**Troubleshooting secure storage access**

When access fails, isolate the problem by checking identity, authorization, and network path in that order.

Common symptoms and likely causes:

- **Authorization failure**
  - Missing RBAC role.
  - Wrong scope.
  - Missing share-level or file-level permission.
- **Authentication failure**
  - Expired SAS.
  - Invalid signature.
  - Wrong storage key.
  - Token acquisition issue for the client identity.
- **Network failure**
  - Storage firewall blocks the client.
  - Private endpoint DNS is misconfigured.
  - Service endpoint is not enabled on the subnet.
- **Protocol failure**
  - SMB version mismatch.
  - Client OS does not support required encryption or authentication method.

A practical troubleshooting sequence:

1. Confirm the client can resolve the correct endpoint.
2. Confirm the network path is allowed.
3. Confirm the identity has the correct role.
4. Confirm the data-plane permission or share ACL.
5. Confirm the token, SAS, or key is valid.

Example CLI checks:

```bash
az storage account show \
  --name mystorageacct \
  --resource-group rg-storage \
  --query "{publicNetworkAccess:publicNetworkAccess,allowBlobPublicAccess:allowBlobPublicAccess}"
```

```bash
az role assignment list \
  --assignee user@contoso.com \
  --scope /subscriptions/<subId>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct \
  -o table
```

**Common exam scenarios**

AZ-104 often presents a scenario and asks you to choose the least disruptive secure fix.

- If an app needs temporary access to one blob container, use a **service SAS** or **user delegation SAS**.
- If a managed identity needs to read blobs, assign a **Storage Blob Data Reader** role.
- If a storage account must not be reachable from the internet, use **private endpoints** and disable public network access where appropriate.
- If a web app needs access from a specific subnet, consider a **service endpoint** or private endpoint depending on the security requirement.
- If anonymous blob access must be prevented, disable **Allow blob public access**.
- If a legacy app uses account keys, plan a migration to **Microsoft Entra ID** or SAS and rotate keys carefully.

**Quick comparison of access choices**

| Requirement | Best fit |
|---|---|
| Temporary delegated access | SAS |
| App or VM access without secrets | Managed identity + RBAC |
| Full administrative data-plane access | Storage account key |
| Private-only network access | Private endpoint |
| Subnet-restricted access with simpler setup | Service endpoint |
| Anonymous public blob access | Blob public access setting and container access level |

**Key takeaways**

Secure storage access in Azure is about combining the right identity model with the right network boundary. For AZ-104, you should be able to explain why a request is allowed or denied based on RBAC, SAS, keys, public access settings, and network configuration.

Focus on these exam-critical distinctions:

- **RBAC** controls who can access data, but only at the correct scope and role type.
- **SAS** delegates limited access and should be time-bound and HTTPS-only.
- **Keys** are powerful and should be rotated carefully.
- **Private endpoints** provide the strongest network isolation.
- **Service endpoints** are a simpler network restriction option.
- **Azure Files** often requires both share-level and file-level permission alignment.
