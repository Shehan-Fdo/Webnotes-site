## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Storage access model overview**

Azure Storage access is controlled by a combination of identity, shared secrets, and network rules. For AZ-104, you should separate three concerns:

- **Authentication**: who or what is requesting access.
- **Authorization**: what that identity can do.
- **Network access**: whether the request can reach the storage endpoint at all.

A storage request can fail even when the identity is valid if the network path is blocked, the wrong endpoint is used, or the data-plane role is assigned at the wrong scope. In practice, you troubleshoot storage by checking these layers in order.

**Storage account access keys**

Storage account keys are shared secrets that grant broad access to the account. They are powerful and simple, but they reduce granularity because anyone with a key can access data according to the service and API used.

- **Key1 and Key2** support key rotation with minimal downtime.
- **Connection strings** often embed the account name and key.
- **Regenerate** one key at a time to avoid breaking all clients.
- **Disable shared key access** when you want to force identity-based access for supported scenarios.

Common CLI examples:

```bash
# Show account keys
az storage account keys list \
  --account-name mystorageacct \
  --resource-group rg-storage

# Regenerate a key
az storage account keys renew \
  --account-name mystorageacct \
  --resource-group rg-storage \
  --key primary
```

Use keys when you need compatibility with legacy applications, quick administrative access, or services that do not support Microsoft Entra ID authentication. Avoid them for long-lived application secrets when a managed identity or SAS is a better fit.

**Microsoft Entra ID and Azure RBAC**

Microsoft Entra ID authentication is the preferred identity model for Azure Storage when the service and protocol support it. Azure Role-Based Access Control (RBAC) then determines what the identity can do.

- **Management-plane roles** control the storage account resource itself.
- **Data-plane roles** control access to blobs, queues, tables, and files where supported.
- Assign data roles at the **storage account scope** or a lower scope such as a container or file share when supported.
- Do not rely on subscription or management group scope for storage data-plane troubleshooting; that is not the normal pattern for data access.

Typical data roles include:

- **Storage Blob Data Reader**: read blob data and metadata.
- **Storage Blob Data Contributor**: read, write, and delete blob data.
- **Storage Queue Data Contributor**: manage queue messages.
- **Storage File Data SMB Share Reader/Contributor/Elevated Contributor**: access Azure Files over SMB with identity-based authorization in supported configurations.

A common exam trap is confusing management roles with data roles:

- **Contributor** can manage the storage account resource.
- **Storage Blob Data Contributor** can access blob data.
- A user may have one and not the other.

Example CLI assignment at the correct scope:

```bash
# Assign blob data access at the storage account scope
az role assignment create \
  --assignee user@contoso.com \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<subId>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct
```

For a container-level scope, use the container resource ID. This is useful when you want access limited to one container rather than the whole account.

**Azure Files authentication differences**

Azure Files supports different authentication methods depending on protocol and configuration. This is an important AZ-104 distinction.

- **SMB**:
  - Supports Microsoft Entra ID-based authorization in supported configurations.
  - Also supports storage account key authentication.
  - Can use Kerberos-based scenarios in domain-integrated environments.
- **REST**:
  - Commonly uses shared key or SAS.
  - Do not assume the same Entra ID model used for Blob applies to REST access for Azure Files.
- **NFS**:
  - Uses network and POSIX-style permissions in supported configurations.
  - Entra ID is not the standard authentication model for NFS access.

For exam purposes, remember that Azure Files is protocol-specific. If a question mentions SMB and identity-based access, think Entra ID or domain integration. If it mentions REST or automation, shared key or SAS is often the expected answer.

**Shared Access Signatures**

A Shared Access Signature (SAS) is a time-limited token that delegates access without exposing the account key to the client. SAS is one of the most tested storage topics in AZ-104.

There are three main SAS types:

- **Service SAS**: scoped to a specific resource such as a blob, container, queue, or file share.
- **Account SAS**: scoped to the storage account and can cover multiple services and resource types.
- **User delegation SAS**: for Blob storage, generated using Microsoft Entra ID and a user delegation key rather than the account key.

Use the right SAS type for the job:

- Use a **service SAS** when you want to grant access to one container, blob, or share.
- Use an **account SAS** when a client needs access across multiple services or resource types in the same account.
- Use a **user delegation SAS** when you want Blob access without exposing the account key and you can authenticate the signer with Entra ID.

Key SAS concepts:

- **Permissions** define allowed operations.
- **Start time** and **expiry time** limit the token lifetime.
- **IP range** can restrict where the token works.
- **HTTPS only** should be used whenever possible.
- **Stored access policies** let you centrally manage and revoke service SAS tokens for containers and file shares.

> A SAS token is not the same as an RBAC role. SAS is a signed delegation token; RBAC is identity-based authorization.

**Service SAS versus account SAS**

This distinction matters in troubleshooting and design.

| Feature | Service SAS | Account SAS |
|---|---|---|
| Scope | One service resource such as a blob, container, queue, or share | Storage account-wide, across supported services |
| Typical use | Share a single container or file share | Allow broader access across services |
| Revocation | Can be tied to a stored access policy for easier revocation | Usually revoked by rotating keys or expiring the token |
| Best for | Least-privilege delegation | Multi-service automation or cross-service workflows |

In AZ-104 scenarios, if a question asks for access to one container only, service SAS is usually the best fit. If it asks for access to blobs and queues in the same account for a short-lived process, account SAS may be more appropriate.

**Stored access policies**

Stored access policies are a management feature for service SAS on containers and file shares. They let you define permissions and validity centrally, then issue SAS tokens that reference the policy.

Benefits:

- You can revoke many SAS tokens by changing or deleting one policy.
- You can update expiry or permissions without reissuing every token.
- You get better operational control than with ad hoc SAS strings.

Example CLI pattern for a container stored access policy:

```bash
az storage container policy create \
  --account-name mystorageacct \
  --container-name data \
  --name readpolicy \
  --permissions r \
  --expiry 2026-12-31T23:59Z \
  --start 2026-01-01T00:00Z
```

> CLI syntax can vary by Azure CLI version. In labs and exams, focus on the concept: a stored access policy is attached to the container or share and referenced by SAS.

**SAS permissions and scope**

SAS permissions are service-specific. Do not memorize one universal letter set and assume it applies everywhere.

- Blob SAS commonly uses permissions such as read, write, delete, list, add, create, update, and process depending on the resource type and operation.
- Queue, file, and table SAS use different permission sets.
- Some letters are not valid for every service or every resource type.

Practical guidance:

- Check the service-specific permission model before generating a SAS.
- Use the minimum permissions required.
- Prefer short expiry times.
- Use stored access policies when you need revocation control.

**Microsoft Entra ID for Azure Storage data access**

When supported, Microsoft Entra ID is the preferred authentication method because it avoids shared secrets and integrates with RBAC.

Typical flow:

1. The user or workload authenticates to Entra ID.
2. Azure RBAC evaluates the assigned data-plane role.
3. The storage service authorizes the request.

Useful CLI examples:

```bash
# Sign in
az login

# Set the subscription
az account set --subscription <subId>

# List blob containers using Entra ID auth
az storage container list \
  --account-name mystorageacct \
  --auth-mode login
```

If `--auth-mode login` fails, check:

- Whether the service supports Entra ID for that protocol.
- Whether the identity has the correct data-plane role.
- Whether the role is assigned at the correct scope.
- Whether network rules or private endpoints block access.

**Azure Files SMB, REST, and NFS authorization**

Azure Files is often tested because its access model differs from Blob storage.

- **SMB**:
  - Supports identity-based access in supported configurations.
  - Commonly used for Windows file shares and domain-integrated environments.
  - Can also use storage account keys.
- **REST**:
  - Often uses shared key or SAS.
  - Useful for automation and service-to-service access.
- **NFS**:
  - Uses network-based access and file system permissions in supported configurations.
  - Not the same Entra ID model as SMB.

When a question asks about Azure Files and identity-based access, verify the protocol first. The correct answer for SMB may be wrong for REST or NFS.

**Network access controls**

Network controls determine whether clients can reach the storage endpoint.

Important settings:

- **Public network access**: enables or disables access over the public endpoint.
- **Storage firewall default action**: controls whether unmatched traffic is allowed or denied when public network access is enabled.
- **Selected networks**: allows only approved IP ranges or virtual networks.
- **Private endpoints**: map the storage service to a private IP in your virtual network.

These settings are related but not identical:

- **Public network access** is an account-level reachability setting.
- **Default action** is the firewall behavior for traffic that reaches the public endpoint.
- A storage account can have public network access enabled but still deny traffic by default.
- A private endpoint can provide access even when public access is restricted.

**Private endpoints and private DNS**

A private endpoint is not complete until name resolution works. For storage, you usually also need a private DNS zone.

Common zones:

- `privatelink.blob.core.windows.net`
- `privatelink.file.core.windows.net`
- `privatelink.queue.core.windows.net`
- `privatelink.table.core.windows.net`

Typical deployment pattern:

1. Create the private endpoint for the storage subresource.
2. Link the private DNS zone to the virtual network.
3. Create or update the DNS zone group on the private endpoint.
4. Verify that the storage FQDN resolves to the private IP.

Example CLI pattern:

```bash
az network private-endpoint create \
  --name pe-mystorageacct \
  --resource-group rg-network \
  --vnet-name vnet-app \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct \
  --group-id blob \
  --connection-name pe-mystorageacct-conn
```

Then add the DNS zone group:

```bash
az network private-endpoint dns-zone-group create \
  --resource-group rg-network \
  --endpoint-name pe-mystorageacct \
  --name default \
  --private-dns-zone privatelink.blob.core.windows.net \
  --zone-name blob-zone
```

> If DNS is missing, the private endpoint may exist but clients still resolve the public storage endpoint and fail.

**Portal configuration workflow**

The Azure portal is commonly used in AZ-104 labs and exam scenarios. Know the basic path for each control.

For storage account access control:

1. Open the storage account.
2. Select **Access control (IAM)** for RBAC assignments.
3. Select **Networking** for firewall, public access, and private endpoint settings.
4. Select the specific service blade, such as **Containers** or **File shares**, for resource-level settings.

For a container SAS:

1. Open the storage account.
2. Open **Containers** and select the container.
3. Use **Shared access tokens** or **Generate SAS** depending on the portal experience.
4. Set permissions, start time, expiry, and allowed protocols.
5. Copy the SAS URL or token and test access.

For a private endpoint:

1. Open the storage account and choose **Private endpoint connections** or create from the networking blade.
2. Select the subresource, such as blob or file.
3. Create or select the private DNS zone.
4. Confirm the connection is approved and DNS is linked.

**Troubleshooting access failures**

A structured troubleshooting approach saves time.

1. Verify the endpoint.
   - Confirm you are using the correct service URL, such as blob versus file.
   - Confirm the client is targeting the private or public endpoint you expect.
2. Verify identity or token.
   - Check the account key, SAS, or Entra ID role.
   - Confirm token expiry and permissions.
3. Verify scope.
   - Ensure the data role is assigned at the storage account, container, or share scope as needed.
4. Verify network access.
   - Check firewall rules, private endpoint status, and DNS resolution.
5. Verify protocol support.
   - Make sure the authentication method matches the service and protocol.

Useful diagnostics:

```bash
# Test blob access with Entra ID
az storage blob list \
  --account-name mystorageacct \
  --container-name data \
  --auth-mode login

# Check DNS resolution from a client
nslookup mystorageacct.blob.core.windows.net
```

If the DNS result points to a public IP when you expect private access, inspect the private DNS zone link and zone group.

**Common exam distinctions**

- **RBAC versus SAS**:
  - RBAC is identity-based and centrally managed.
  - SAS is token-based and time-limited.
- **Service SAS versus account SAS**:
  - Service SAS is narrower.
  - Account SAS is broader.
- **Public network access versus firewall default action**:
  - Public network access controls whether the public endpoint is reachable.
  - Firewall default action controls what happens to traffic that reaches it.
- **Management role versus data role**:
  - Management roles manage the resource.
  - Data roles access the data.
- **SMB versus REST for Azure Files**:
  - SMB can use identity-based authorization in supported configurations.
  - REST commonly uses shared key or SAS.

**Practical design guidance**

Use the least privileged option that meets the requirement:

- Prefer **managed identity** plus Azure RBAC for applications running in Azure.
- Prefer **user delegation SAS** for Blob when you need delegated access without sharing account keys.
- Prefer **service SAS** for single-resource sharing.
- Prefer **private endpoints** when you need to remove public exposure.
- Prefer **stored access policies** when you need revocation control for SAS tokens.

A simple decision rule helps on the exam:

- If the question emphasizes user identity and least privilege, choose Entra ID and RBAC.
- If it emphasizes temporary external sharing, choose SAS.
- If it emphasizes broad legacy compatibility, choose account keys.
- If it emphasizes network isolation, choose private endpoints and DNS.

**Quick reference**

- **Account key**: broad shared secret for the storage account.
- **SAS**: time-limited delegated access token.
- **Service SAS**: scoped to one service resource.
- **Account SAS**: scoped to the storage account across services.
- **User delegation SAS**: Blob SAS signed with Entra ID.
- **RBAC data role**: identity-based authorization for storage data.
- **Private endpoint**: private IP access to storage over your virtual network.
- **Private DNS**: required for name resolution to the private endpoint.
