## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure Files overview**

Azure Files provides fully managed file shares that you access over the network using the Server Message Block (**SMB**) protocol, and in some premium scenarios also Network File System (**NFS**). For AZ-104, focus primarily on SMB file share administration: creating shares, controlling access, securing connectivity, and troubleshooting client access.

Azure Files is useful when you need:
- Shared storage for multiple Windows servers or users
- Lift-and-shift file workloads that expect a file share
- Centralized home directories, application data, or configuration files
- Hybrid access from on-premises and Azure-hosted systems

Unlike a local disk, an Azure file share is a network resource. That means performance, authentication, and connectivity depend on storage account settings, network path, and client configuration.

**Storage account and share types**

Azure Files lives inside a storage account. The account type affects features, performance, and pricing model.

- **StorageV2 accounts** support standard Azure file shares and are the most common choice.
- **FileStorage accounts** are used for premium file shares.
- Standard shares are typically used for general-purpose workloads where cost matters more than low latency.
- Premium shares are provisioned and suited for I/O-intensive workloads.

A share is the unit you create, delete, and manage. A storage account can contain multiple shares, each with its own quota or provisioned size depending on the share type.

> Be careful not to describe standard shares as simply “HDD-based.” For exam purposes, focus on the service tier and performance characteristics rather than oversimplified media assumptions.

**Core administration tasks**

You manage Azure Files at both the storage account level and the share level.

- Create and delete file shares
- Set or adjust share quota
- Configure access control and authentication
- Restrict network access
- Enable snapshots and restore data
- Monitor usage and troubleshoot connectivity
- Integrate with Azure File Sync for hybrid scenarios

A practical admin workflow often looks like this:
1. Create or select a storage account.
2. Create the file share.
3. Decide how clients authenticate.
4. Restrict network access if needed.
5. Mount the share on clients.
6. Monitor usage and adjust quota or performance settings.

**Creating and managing shares**

You can create shares in the Azure portal, Azure CLI, or Azure PowerShell.

Portal workflow:
- Open the storage account.
- Select **File shares**.
- Choose **+ File share**.
- Enter a name and quota.
- Create the share.

Azure CLI example for a standard share:
```bash
az storage share create \
  --account-name mystorageacct \
  --name appshare \
  --quota 100 \
  --auth-mode login
```

Azure PowerShell example:
```powershell
New-AzStorageShare -Name "appshare" -Context $ctx -QuotaGiB 100
```

Important share management actions:
- **Create** a share for a new workload
- **Delete** a share when the workload is retired
- **Resize quota** to allow more data growth
- **List** shares to audit storage usage
- **Set metadata** when you need application-specific tags

For premium shares, think in terms of provisioned capacity rather than a simple “quota only” mindset. The exact management experience differs from standard shares, so always check the share type before changing capacity.

**Authentication and authorization**

Azure Files SMB access can use different authentication models. The choice affects how users connect and what permissions apply.

Common options:
- **Storage account key** authentication
- **Microsoft Entra ID**-based identity authentication for SMB
- **Active Directory Domain Services** or **Microsoft Entra Domain Services** integration for SMB in domain-joined scenarios

With key-based access:
- The storage account key acts like a master secret.
- Access is broad and not user-specific.
- NTFS-style file and folder permissions are not the main control mechanism in the same way they are for identity-based SMB access.

With identity-based SMB access:
- Azure authorization controls whether the identity can access the share.
- File and folder permissions are enforced through NTFS ACLs on the share contents.
- Effective access depends on both share-level authorization and file-system permissions.

A useful mental model is:
- **Share-level access** decides whether you can reach the share.
- **NTFS ACLs** decide what you can do inside the share when identity-based auth is used.

> If you use storage keys, you bypass the identity-based permission model. Do not assume Azure role assignments and NTFS ACLs apply the same way in every access method.

**Azure RBAC and share-level permissions**

For identity-based access, Azure role-based access control (**RBAC**) helps authorize access to the file share resource.

Typical built-in roles include:
- **Storage File Data SMB Share Reader**
- **Storage File Data SMB Share Contributor**
- **Storage File Data SMB Share Elevated Contributor**

These roles are used to grant access at the share level. They do not replace NTFS permissions when identity-based SMB is in use.

Practical interpretation:
- A user may have Azure RBAC permission to access the share but still be blocked by NTFS ACLs on a folder or file.
- A user may have NTFS permissions but still fail to mount the share if they lack the required share-level authorization.
- Key-based access does not use this same RBAC + NTFS combination.

Common exam scenario:
- You want a user to read files in a share.
- You assign a share-level reader role.
- You also ensure the folder ACL grants read access.
- If either layer is missing, access can fail.

**NTFS permissions and inheritance**

When Azure Files is integrated with identity-based SMB authentication, NTFS ACLs behave like they do on Windows file systems.

Key points:
- Permissions can be inherited from parent folders.
- Explicit permissions override inherited behavior where applicable.
- Deny entries can block access even when allow entries exist.
- Ownership and administrative access matter for troubleshooting.

Typical permission patterns:
- **Read** for consumers of shared documents
- **Modify** for application service accounts
- **Full control** for administrators or deployment accounts

Troubleshooting permission issues:
- Check whether the user has the correct Azure role assignment.
- Check whether the share was mounted using the intended identity.
- Review folder inheritance and explicit ACLs.
- Confirm the user is accessing the correct path and not a stale cached mapping.

**Mounting Azure Files on Windows**

Windows clients commonly mount Azure file shares using `net use`. This is the most exam-safe and practical approach for mapped drives.

Example using a storage account key:
```cmd
cmdkey /add:mystorageacct.file.core.windows.net /user:Azure\mystorageacct /pass:<storage-account-key>
net use Z: \\mystorageacct.file.core.windows.net\appshare /persistent:yes
```

If you use identity-based authentication, the exact sign-in flow depends on your domain and client configuration. In many enterprise environments, users access the share with their domain identity after the storage account and client are configured for SMB authentication.

Useful Windows commands:
```cmd
net use
net use Z: /delete
dir Z:\
```

Practical notes:
- Use `net use` for persistent mappings.
- Store credentials carefully; storage account keys are highly privileged.
- If the mapping fails, verify DNS, firewall rules, and SMB port 445 connectivity.

**Mounting Azure Files on Linux**

Linux clients typically mount Azure Files over SMB using `cifs-utils`.

Install the client tools:
```bash
sudo apt-get update
sudo apt-get install -y cifs-utils
```

Create a mount point:
```bash
sudo mkdir -p /mnt/appshare
```

Mount with a storage account key:
```bash
sudo mount -t cifs //mystorageacct.file.core.windows.net/appshare /mnt/appshare \
  -o vers=3.0,username=mystorageacct,password=<storage-account-key>,dir_mode=0777,file_mode=0777,serverino
```

Important Linux considerations:
- Use SMB version 3.0 or later when possible.
- The username format depends on the authentication method; for key-based access, the storage account name is commonly used.
- Persist mounts in `/etc/fstab` only after validating the command manually.
- Identity-based SMB access from Linux is more limited and environment-dependent than Windows domain-joined access.

Example `/etc/fstab` entry:
```fstab
//mystorageacct.file.core.windows.net/appshare /mnt/appshare cifs vers=3.0,username=mystorageacct,password=<storage-account-key>,dir_mode=0777,file_mode=0777,serverino,nofail 0 0
```

**SMB security settings**

Azure Files supports modern SMB security features, and you should know when to use them.

Key security concepts:
- **SMB 3.x** is the preferred protocol family.
- **SMB encryption** protects data in transit.
- **SMB signing** helps protect against tampering.
- Older SMB versions are not appropriate for secure enterprise use.

When to care:
- Use encryption for sensitive data or untrusted networks.
- Prefer private connectivity or VPN/ExpressRoute for production workloads.
- Avoid exposing SMB shares broadly to the public internet unless you have a strong reason and tight controls.

**Networking and firewall controls**

Azure Files access is affected by storage account networking settings.

You can control access with:
- **Public network access** settings
- **Storage account firewall rules**
- **Selected networks**
- **Private endpoints**

Common networking patterns:
- Allow access from all networks for simple testing only
- Restrict access to selected virtual networks and IP ranges
- Use a private endpoint for private, internal-only access

Firewall behavior matters:
- If the storage account blocks the client network, the mount fails even if credentials are correct.
- If you use a private endpoint, DNS must resolve the file endpoint to the private IP address.
- Network rules and authentication are separate checks; both must succeed.

**Private endpoints and DNS**

A private endpoint gives your storage account a private IP address in your virtual network. For Azure Files, you must also configure DNS correctly.

The private DNS zone for Azure Files is:
- `privatelink.file.core.windows.net`

Typical setup:
1. Create the private endpoint for the file service.
2. Link it to the target virtual network.
3. Create or use the private DNS zone `privatelink.file.core.windows.net`.
4. Add the DNS zone group or equivalent DNS records.
5. Verify that `mystorageacct.file.core.windows.net` resolves to the private IP from clients inside the network.

Example CLI pattern:
```bash
az network private-endpoint create \
  --name pe-azurefiles \
  --resource-group rg-storage \
  --vnet-name vnet-prod \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id /subscriptions/<sub>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct \
  --group-ids file \
  --connection-name pe-azurefiles-conn
```

Then link DNS:
```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.file.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.file.core.windows.net \
  --name link-vnet-prod \
  --virtual-network vnet-prod \
  --registration-enabled false
```

> A private endpoint alone is not enough. If DNS is wrong, clients may still try the public endpoint and fail.

**Snapshots and restore operations**

Azure Files snapshots are point-in-time, read-only copies of a file share. They are useful for accidental deletion, corruption, or rollback scenarios.

Snapshot behavior:
- Snapshots are share-level, not account-wide.
- They are read-only.
- They capture the state of the share at the time of creation.
- They are not a separate writable share.

Create a snapshot:
```bash
az storage share snapshot \
  --account-name mystorageacct \
  --name appshare
```

List snapshots:
```bash
az storage share list \
  --account-name mystorageacct \
  --include-snapshot
```

Restore a file from a snapshot:
1. Identify the snapshot timestamp.
2. Browse the snapshot version of the share.
3. Copy the needed file or folder back to the live share.

Example concept using the snapshot URI:
```text
\\mystorageacct.file.core.windows.net\appshare?snapshot=2024-04-26T12:30:00.0000000Z
```

Operationally, you usually restore by copying data from the snapshot back into the active share rather than “mounting” it as a normal writable share.

Common restore scenarios:
- Recover a deleted document
- Roll back a bad application deployment
- Restore a folder after accidental overwrite

**Azure File Sync**

Azure File Sync extends Azure Files into a hybrid file server model. It lets you keep a local Windows Server cache while syncing to an Azure file share.

Core components:
- **Storage sync service** in Azure
- **Sync group**
- **Cloud endpoint** pointing to the Azure file share
- **Server endpoint** on the Windows Server

Typical use cases:
- Branch office file servers
- Centralized cloud backup and tiering
- Consolidating multiple file servers into one cloud-backed share

Cloud tiering:
- Frequently used files stay local.
- Infrequently used files can be tiered to Azure.
- Users still see the full namespace.

Deployment prerequisites:
- Supported Windows Server version
- Azure File Sync agent installed on the server
- Registered server in the storage sync service
- Azure file share created for the cloud endpoint

High-level setup:
1. Create a storage sync service.
2. Create a sync group.
3. Add the Azure file share as the cloud endpoint.
4. Install the Azure File Sync agent on the Windows Server.
5. Register the server.
6. Create a server endpoint and enable cloud tiering if needed.

Example PowerShell installation pattern:
```powershell
Invoke-WebRequest -Uri "https://aka.ms/AFSAgent" -OutFile "AzureFileSyncAgent.msi"
Start-Process msiexec.exe -Wait -ArgumentList '/i AzureFileSyncAgent.msi /quiet'
```

**Monitoring and troubleshooting**

Azure Files troubleshooting usually starts with three questions:
- Can the client resolve the name?
- Can the client reach the network path?
- Does the client have the right permissions?

Useful checks:
- Verify DNS resolution for the storage account name.
- Test TCP 445 connectivity.
- Confirm firewall and private endpoint settings.
- Review share-level authorization and NTFS ACLs.
- Check whether the storage account key or credentials are valid.

Example connectivity test from Windows:
```powershell
Test-NetConnection mystorageacct.file.core.windows.net -Port 445
```

Useful Azure monitoring tools:
- **Metrics** for capacity, transactions, and availability trends
- **Diagnostic settings** to send logs to Log Analytics, Event Hub, or Storage
- **Azure Monitor** alerts for operational thresholds
- **Activity log** for management-plane changes such as share creation or network rule updates

Common troubleshooting scenarios:
- **Access denied**: check RBAC and NTFS permissions
- **Network path not found**: check DNS, firewall, and private endpoint configuration
- **Authentication failure**: verify the auth method and stored credentials
- **Slow performance**: check share type, client SMB version, and network latency

**Exam-oriented comparison**

| Topic | Azure Files SMB | Blob Storage |
|---|---|---|
| Access style | Mounted file share | Object access via API or tools |
| Typical protocol | SMB | HTTPS/REST |
| Best for | Shared file systems | Unstructured objects |
| Permissions | Share-level auth plus NTFS ACLs for identity-based SMB | Azure authorization and application access patterns |
| Client experience | Appears like a network drive | Not a traditional mounted file share |

Use Azure Files when the workload expects a file system. Use Blob Storage when the workload is object-based and does not need SMB semantics.

**Common exam scenarios**

- You need a shared drive for multiple Windows VMs.
  - Choose Azure Files with SMB.
- You need to restrict access to a file share to a private network.
  - Use storage firewall rules or a private endpoint.
- You need to recover a deleted file quickly.
  - Use a snapshot restore workflow.
- You need a branch office file server with cloud-backed storage.
  - Use Azure File Sync.
- You need user-specific file permissions.
  - Use identity-based SMB with Azure RBAC and NTFS ACLs.

**Practical troubleshooting checklist**

1. Confirm the share exists and the name is correct.
2. Verify the storage account type supports the intended share type.
3. Check whether the client uses the correct authentication method.
4. Test DNS resolution for the file endpoint.
5. Test port 445 connectivity.
6. Review storage firewall and private endpoint settings.
7. Validate Azure RBAC role assignments.
8. Validate NTFS permissions if identity-based SMB is used.
9. Check snapshots if the issue is data loss rather than access.
10. Review Azure Monitor metrics and logs for service-side clues.

**Key takeaways**

Azure Files is the Azure service to use when you need a managed SMB file share. For AZ-104, the most important skills are creating and managing shares, understanding authentication models, securing network access, restoring from snapshots, and recognizing when Azure File Sync is the right hybrid solution.

Focus on the interaction between:
- **Authentication** and **authorization**
- **Network access** and **DNS**
- **Share-level permissions** and **NTFS ACLs**
- **Standard** and **premium** share behavior
