## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure Storage network security model**

Azure Storage exposes a public endpoint by default, but you can restrict who can reach it and how. For AZ-104, the key idea is that storage access is controlled at multiple layers:

- **Authentication and authorization** decide *who* can read or write data.
- **Network rules** decide *where* traffic can come from.
- **Private connectivity** moves traffic off the public internet and onto Azure private IP space.
- **DNS** decides whether clients resolve the storage account name to the public endpoint or a private endpoint.

A storage account can be reachable through the public endpoint, through a private endpoint, or both, depending on the configuration. In troubleshooting, always separate “can the client resolve the name?” from “is the network path allowed?” and “does the identity have permission?”

**Storage account public network access**

Storage accounts support public network access settings that affect whether the public endpoint is usable.

- **Enabled from all networks** allows access from any source that can reach the public endpoint, subject to authentication and any higher-level restrictions.
- **Enabled from selected virtual networks and IP addresses** keeps the public endpoint enabled, but the firewall only allows approved IP ranges and selected virtual networks.
- **Disabled** blocks access through the public endpoint. Private endpoints are the intended access path.

> If public network access is disabled, do not expect IP firewall rules, resource instance rules, or trusted Microsoft service exceptions to restore public endpoint access. They do not override the disabled public endpoint in the same way as “selected networks” mode.

This distinction is important in exam scenarios:

- Use **selected networks** when you still need controlled public access for admins, apps, or partners.
- Use **disabled** when you want to force private connectivity only.
- Use **private endpoints** when you need access from a VNet, on-premises network, or peered network without exposing the storage account publicly.

**Firewall and network rule behavior**

Azure Storage firewall rules are evaluated before data plane access is granted. The main rule types are:

- **IP network rules** for public client IP addresses.
- **Virtual network rules** for subnets that have the Microsoft.Storage service endpoint enabled.
- **Resource instance rules** for specific Azure resources that access storage over the public endpoint.
- **Trusted Microsoft services** exceptions for selected platform services.

A practical mental model:

- If the request comes from the internet, the storage firewall checks the source IP.
- If the request comes from an Azure subnet with a service endpoint, the storage firewall checks the subnet rule.
- If the request comes from a supported Azure resource instance, the firewall can allow that specific resource.
- If the request comes through a private endpoint, the public firewall is bypassed for that private path.

**Common Azure CLI patterns**

Use the Azure CLI to inspect and manage storage network settings.

```bash
# Show the storage account network configuration
az storage account show \
  --name mystorageacct \
  --resource-group rg-demo \
  --query "networkRuleSet"

# Allow a public IP range
az storage account network-rule add \
  --account-name mystorageacct \
  --resource-group rg-demo \
  --ip-address 203.0.113.10

# Allow a subnet that already has the Microsoft.Storage service endpoint enabled
az storage account network-rule add \
  --account-name mystorageacct \
  --resource-group rg-demo \
  --subnet /subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/virtualNetworks/vnet1/subnets/subnet1

# Remove a rule
az storage account network-rule remove \
  --account-name mystorageacct \
  --resource-group rg-demo \
  --ip-address 203.0.113.10
```

Use `az storage account update` to change the public network access mode:

```bash
# Allow access only from selected networks
az storage account update \
  --name mystorageacct \
  --resource-group rg-demo \
  --default-action Deny

# Disable public network access
az storage account update \
  --name mystorageacct \
  --resource-group rg-demo \
  --public-network-access Disabled
```

For exam purposes, remember:

- `--default-action Deny` means the firewall blocks by default and only explicit rules allow access.
- `--public-network-access Disabled` is stronger than a deny-by-default firewall because it removes public endpoint access entirely.
- A subnet rule requires the subnet to support the storage service endpoint path.

**Service endpoints versus private endpoints**

Service endpoints and private endpoints both improve security, but they solve different problems.

| Feature | Service endpoint | Private endpoint |
|---|---|---|
| Traffic path | Stays on Azure backbone to the public service endpoint | Uses a private IP in your VNet |
| Public endpoint exposure | Still exists | Can be bypassed for private access |
| DNS change required | Usually no | Yes, typically via private DNS |
| On-premises access | Not ideal | Supported with VPN/ExpressRoute and DNS forwarding |
| Granularity | Subnet-based | Resource/subresource-based |

Use **service endpoints** when you want to restrict access to a subnet but do not need a private IP in your VNet. Use **private endpoints** when you want the storage account to be reachable through a private IP address and private DNS.

**Resource instance rules**

Resource instance rules allow specific Azure resources to access a storage account over the public endpoint without opening broad IP ranges. This is useful when the caller has a stable Azure resource identity but not a fixed public IP.

Common examples include:

- **Azure Synapse Analytics** workspaces accessing storage.
- **Azure Data Factory** managed resources accessing storage.
- **Azure Backup** or other platform services that need controlled storage access.
- **Azure Monitor** or similar services in supported scenarios.

A resource instance rule is narrower than an IP rule because it targets a specific Azure resource, not a network range. Use it when the service supports it and when you want to avoid exposing the storage account to a wider network surface.

**Private endpoints and subresources**

A private endpoint maps a storage subresource to a private IP address in your VNet. Azure Storage commonly exposes these subresources:

- `blob`
- `file`
- `queue`
- `table`
- `web` for static website hosting

You may need multiple private endpoints for one storage account if different workloads use different subresources. For example:

- A web app may need `blob` for object storage.
- A legacy application may need `file` for SMB shares.
- A queue-based worker may need `queue`.

A single private endpoint is tied to one subresource group, so plan access per service. In exam questions, check whether the workload needs Blob, File, Queue, Table, or Web access rather than assuming one endpoint covers everything.

**Private endpoint creation and DNS zone groups**

Creating a private endpoint does not automatically make name resolution work for clients. You usually also need a private DNS zone and a zone group.

```bash
# Create the private endpoint
az network private-endpoint create \
  --name pe-mystorage-blob \
  --resource-group rg-network \
  --vnet-name vnet1 \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct \
  --group-id blob \
  --connection-name peconn-mystorage-blob

# Create or attach the private DNS zone group
az network private-endpoint dns-zone-group create \
  --resource-group rg-network \
  --endpoint-name pe-mystorage-blob \
  --name default \
  --private-dns-zone privatelink.blob.core.windows.net
```

The exact CLI parameters can vary slightly by Azure CLI version, but the concept is stable:

- Create the private endpoint first.
- Associate the correct private DNS zone for the storage subresource.
- Ensure the DNS zone contains the `A` record that maps the storage name to the private IP.

> A private DNS zone group is not strictly required if you manage DNS records elsewhere, but without correct DNS the private endpoint will not be usable by normal name-based clients.

**Private DNS zones for Azure Storage**

Azure Storage private endpoints typically use these private DNS zones:

- `privatelink.blob.core.windows.net`
- `privatelink.file.core.windows.net`
- `privatelink.queue.core.windows.net`
- `privatelink.table.core.windows.net`
- `privatelink.web.core.windows.net`

When the zone is linked to the VNet, clients in that VNet resolve the storage account name to the private IP instead of the public endpoint.

A typical resolution flow is:

1. Client queries `mystorageacct.blob.core.windows.net`.
2. DNS forwards or resolves to the private zone.
3. The private zone returns the private endpoint IP.
4. The client connects directly to that private IP.

**Hybrid DNS and conditional forwarding**

For on-premises or hybrid networks, you often need DNS forwarding so on-prem clients can resolve Azure private endpoint names.

A common pattern is:

- On-prem DNS servers forward `privatelink.blob.core.windows.net` and related zones to Azure DNS or to a DNS server in Azure.
- Azure DNS Private Resolver or custom DNS servers handle the forwarding path.
- Clients use the normal storage FQDN, and DNS returns the private IP.

Example topology:

- On-prem client
- On-prem DNS server
- Site-to-site VPN or ExpressRoute
- Azure DNS forwarder or Azure DNS Private Resolver
- Private DNS zone linked to the VNet
- Storage private endpoint

This matters because private endpoints are only useful if the client can both route to the private IP and resolve the name correctly.

**Custom DNS example**

If you run your own DNS servers, you may need conditional forwarding.

```dns
; Example conditional forwarder concept
privatelink.blob.core.windows.net  -> Azure DNS forwarder IP
privatelink.file.core.windows.net   -> Azure DNS forwarder IP
```

In practice, the exact configuration depends on your DNS platform, but the goal is the same: forward private endpoint zones to a resolver that knows about the Azure private DNS zone.

**Testing name resolution and connectivity**

Always validate both DNS and network reachability.

```bash
# Windows DNS test
nslookup mystorageacct.blob.core.windows.net

# PowerShell DNS test
Resolve-DnsName mystorageacct.blob.core.windows.net

# Test HTTPS connectivity to Blob
Test-NetConnection mystorageacct.blob.core.windows.net -Port 443
```

For Azure Files, test SMB connectivity as well:

```powershell
Test-NetConnection mystorageacct.file.core.windows.net -Port 445
```

Useful checks:

- If DNS returns a public IP when you expect a private IP, the private DNS zone or forwarding path is wrong.
- If DNS returns the private IP but the connection fails, check routing, NSGs, firewalls, and VPN/ExpressRoute connectivity.
- If the connection succeeds but access is denied, check authentication, share permissions, and storage firewall rules.

**Azure Files-specific private access considerations**

Azure Files often introduces extra troubleshooting because it uses SMB (Server Message Block) for Windows file shares and can also use NFS in some configurations.

Key points:

- SMB typically uses **TCP 445**.
- On-prem networks or ISPs may block port 445, which breaks direct Azure Files access.
- Private endpoints do not fix blocked local egress if the client cannot route to the Azure VNet.
- If you use Azure Files over a private endpoint, ensure the client can reach the private IP over VPN or ExpressRoute.

Common pitfalls:

- DNS resolves correctly, but port 445 is blocked on the client network.
- The storage account firewall allows the subnet, but the client is not actually in that subnet.
- The file share is reachable from Azure VMs but not from on-prem because the VPN route is missing.

**Troubleshooting workflow**

Use a consistent sequence when diagnosing storage access issues.

1. Verify the storage account network mode.
2. Check whether the client should use the public endpoint or a private endpoint.
3. Confirm DNS resolution for the storage FQDN.
4. Validate the route to the resolved IP.
5. Check storage firewall rules, subnet rules, or resource instance rules.
6. Confirm identity and data-plane permissions.

A practical command set:

```bash
# Inspect network rules
az storage account show \
  --name mystorageacct \
  --resource-group rg-demo \
  --query "{publicNetworkAccess:publicNetworkAccess, defaultAction:networkRuleSet.defaultAction, bypass:networkRuleSet.bypass}"

# Inspect private endpoints
az network private-endpoint list \
  --resource-group rg-network \
  --query "[].{name:name, subnet:subnet.id, privateLinkServiceConnections:privateLinkServiceConnections[].groupIds}"
```

If you need to verify access from a VM, test with the actual protocol:

- Blob: HTTPS to port 443
- File: SMB to port 445
- Queue/Table: HTTPS to port 443
- Static website: HTTPS to port 443

**Storage firewall exceptions and trusted services**

Azure Storage can allow selected Microsoft services to bypass the firewall when the service is on the trusted services list. This is not the same as opening the storage account to all Azure services.

Use this carefully:

- It reduces friction for platform integrations.
- It does not mean every Azure service can access the account.
- It does not replace private endpoints for strict isolation.

For AZ-104, remember that “trusted Microsoft services” is a convenience feature, not a universal security exception.

**Practical configuration example**

A secure, common pattern for a production storage account is:

- Public network access set to **Disabled**.
- One or more private endpoints created for required subresources.
- Private DNS zones linked to the application VNets.
- On-prem DNS conditional forwarding configured if hybrid access is needed.
- No broad public IP firewall rules unless there is a specific operational need.

Example architecture:

- App VM in `vnet-app` accesses Blob through a private endpoint.
- Admin workstation on-prem resolves the storage name through conditional forwarding.
- Azure Files share is reachable over VPN with SMB on port 445.
- Public endpoint is not used.

**Exam-focused distinctions**

- **Selected networks** means the public endpoint still exists, but only approved sources can use it.
- **Disabled** means the public endpoint is not available.
- **Service endpoints** secure access from a subnet to the public service endpoint.
- **Private endpoints** give the service a private IP in your VNet.
- **Private DNS** is usually required for private endpoint usability.
- **Multiple subresources** may require multiple private endpoints.

**Quick reference**

- Use `az storage account show` to inspect network settings.
- Use `az storage account update` to change public access mode.
- Use `az storage account network-rule add` for IP or subnet rules.
- Use `az network private-endpoint create` for private connectivity.
- Use `az network private-endpoint dns-zone-group create` or equivalent DNS association to make name resolution work.
- Use `nslookup`, `Resolve-DnsName`, and `Test-NetConnection` to validate access.
