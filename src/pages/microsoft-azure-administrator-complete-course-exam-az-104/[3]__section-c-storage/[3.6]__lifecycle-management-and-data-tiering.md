## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Overview**

Azure Storage lifecycle management helps you automatically move blob data between access tiers or delete it when it is no longer needed. This reduces storage cost while keeping frequently used data available with low latency. For AZ-104, focus on how policies are defined, what blob types are supported, and how account settings affect rule evaluation.

The main access tiers are **Hot**, **Cool**, and **Archive**. Hot is optimized for frequent access, Cool for infrequent access, and Archive for long-term retention with the lowest storage cost but the highest retrieval delay and rehydration cost. You choose a tier based on access frequency, retention requirements, and how quickly you may need the data again.

**Core concepts**

- **Hot tier** is for data you read or modify often.
- **Cool tier** is for data you access infrequently but still need online.
- **Archive tier** is for data you rarely access and can tolerate rehydration delays.
- **Lifecycle management** automates tier changes and deletion based on rules.
- **Rehydration** is the process of restoring archived blobs to an online tier.

**Access tiers and when to use them**

Use Hot when the data supports active workloads, such as application content, current logs, or operational files. Use Cool when the data is still needed but access is sporadic, such as monthly reports or older backups that must remain immediately readable. Use Archive when the data is retained for compliance or long-term retention and can wait hours before it becomes readable again.

A practical rule of thumb is to compare storage cost against retrieval cost and delay:

- Hot has the highest storage cost and the lowest access friction.
- Cool lowers storage cost but charges more for reads and has a minimum retention expectation.
- Archive has the lowest storage cost but requires rehydration before reads or copies.

> Archive blobs are not directly readable. You must rehydrate them to Hot or Cool before you can read or copy the content.

**Lifecycle management scope**

Lifecycle management applies to **blob data** in a storage account, including accounts with hierarchical namespace enabled for Azure Data Lake Storage Gen2. In that case, the policy still acts on the underlying blobs, but you should remember that the account is not identical to a standard blob account in every operational detail.

Supported scope is important for exam questions:

- Lifecycle rules primarily target **block blobs**.
- Rules can also target **snapshots** and **versions** when the action and account support it.
- **Append blobs** and **page blobs** have more limited support and are not the primary target for tiering.
- Premium storage scenarios may have different capabilities than standard general-purpose v2 accounts.

**Policy structure and rule behavior**

Azure lifecycle management is defined as a policy containing one or more rules. Each rule has:

- A **name**
- A **type** such as `Lifecycle`
- A **definition** with filters and actions
- Optional filters for prefix, blob type, and blob index tags

Rules evaluate against blobs that match the filter conditions. Actions can include:

- `tierToCool`
- `tierToArchive`
- `delete`
- `tierToCold` is not a standard lifecycle action for Azure Blob Storage
- Snapshot and version actions, where supported, can delete older snapshots or versions

A typical policy file uses the `policy` wrapper and `rules` array. The exact schema matters because Azure validates the structure.

```json
{
  "policy": {
    "rules": [
      {
        "enabled": true,
        "name": "move-to-cool-after-30-days",
        "type": "Lifecycle",
        "definition": {
          "filters": {
            "blobTypes": ["blockBlob"],
            "prefixMatch": ["logs/"]
          },
          "actions": {
            "baseBlob": {
              "tierToCool": {
                "daysAfterModificationGreaterThan": 30
              }
            }
          }
        }
      }
    ]
  }
}
```

**Rule precedence and interaction**

When multiple rules match the same blob, Azure evaluates them independently and applies the resulting actions according to the service’s lifecycle logic. In practice, you should design rules so they do not conflict.

Important planning points:

- More than one rule can match the same blob.
- A blob can qualify for both tiering and deletion if the timing thresholds overlap.
- You should avoid overlapping rules that create ambiguous outcomes.
- Use clear prefixes, tags, or age thresholds to separate workloads.

A common design pattern is:

- Move active data from Hot to Cool after a short inactivity period.
- Move older data from Cool to Archive after a longer period.
- Delete data only after the retention period is complete.

**Account prerequisites and settings**

Lifecycle actions depend on the storage account kind and configuration. For exam purposes, remember that lifecycle management is not just a blob-level feature; some account-level settings affect whether rules can work as intended.

Key prerequisites and considerations:

- Use a storage account that supports blob lifecycle management, typically **StorageV2**.
- Enable **last access time tracking** if you want to base rules on access time instead of modification time.
- Last access time tracking is not enabled by default.
- Access-time-based actions may not update immediately after every read; there can be delay before the service reflects access.
- Some actions and tier transitions are constrained by the blob type and account tier.

A simple example of enabling last access time tracking with Azure CLI:

```bash
az storage account blob-service-properties update \
  --account-name mystorageacct \
  --resource-group rg-storage \
  --enable-last-access-time-tracking true
```

**Creating and managing policies**

You can manage lifecycle policies in the Azure portal, Azure CLI, or ARM/Bicep templates.

In the **Azure portal**, the usual path is:

1. Open the storage account.
2. Select **Data management**.
3. Open **Lifecycle management**.
4. Create or edit a rule in the policy blade.
5. Save the policy and review the JSON view if needed.

Portal validation is useful because it shows the rule builder and the generated JSON. This helps you confirm that filters, actions, and blob types are set correctly before you save.

With Azure CLI, you typically create or update the policy using a JSON file that matches the service schema.

```bash
az storage account management-policy create \
  --account-name mystorageacct \
  --resource-group rg-storage \
  --policy @policy.json
```

If the policy already exists, you may update it instead of creating a new one.

```bash
az storage account management-policy update \
  --account-name mystorageacct \
  --resource-group rg-storage \
  --policy @policy.json
```

A minimal policy file example:

```json
{
  "policy": {
    "rules": [
      {
        "enabled": true,
        "name": "delete-temp-files",
        "type": "Lifecycle",
        "definition": {
          "filters": {
            "blobTypes": ["blockBlob"],
            "prefixMatch": ["temp/"]
          },
          "actions": {
            "baseBlob": {
              "delete": {
                "daysAfterModificationGreaterThan": 14
              }
            }
          }
        }
      }
    ]
  }
}
```

**Practical examples of tiering decisions**

Choose the tier based on how often the data is accessed and how quickly you need it back.

- **Hot**
  - Application assets
  - Current transaction data
  - Frequently queried logs
  - Data used by active analytics jobs

- **Cool**
  - Monthly reports
  - Older backups that still need immediate access
  - Data retained for short-term audit review
  - Content with occasional reads but predictable retention

- **Archive**
  - Compliance archives
  - Long-term legal retention
  - Historical data rarely accessed
  - Backup copies kept only for disaster recovery or audit

A useful exam heuristic is:

- If you need immediate reads, choose Hot or Cool.
- If you can tolerate rehydration delay, choose Archive.
- If the data is temporary, consider deletion instead of tiering.

**Rehydration and retrieval**

When you move a blob out of Archive, Azure rehydrates it to Hot or Cool. Rehydration can take hours depending on priority and service conditions. You can choose a rehydration priority when supported.

```bash
az storage blob set-tier \
  --account-name mystorageacct \
  --container-name archive \
  --name old-report.csv \
  --tier Hot \
  --rehydrate-priority Standard
```

Use this carefully:

- The blob must already be in **Archive** for rehydration to apply.
- Rehydration is not a general tier-change option for any blob.
- You can rehydrate to **Hot** or **Cool**.
- Rehydration priority affects how quickly the blob becomes available.

> Archived blobs cannot be read or copied until rehydration completes.

**Snapshots and versions**

Lifecycle management can also help control storage growth from snapshots and versions. This is useful when applications create many object revisions or when soft-delete and versioning increase retained data.

Typical uses:

- Delete old snapshots after a retention period.
- Delete older versions while keeping the current base blob.
- Tier base blobs while separately cleaning up historical copies.

Be careful with versioning scenarios:

- A rule that targets only base blobs does not automatically remove all versions.
- Snapshot and version cleanup should be planned explicitly.
- Retention requirements may require keeping older versions longer than the base blob.

**Blob index tags and prefix filters**

Lifecycle rules can use filters to target specific data sets. Prefix matching is based on the blob name string, not on real directory semantics.

Important points:

- Blob names are case-sensitive.
- Prefix matching is an exact string prefix match against the blob name.
- Use tags when you need metadata-driven targeting instead of path-like naming.
- Prefixes such as `logs/` are common, but they are just naming conventions.

Example of a rule that targets tagged blobs:

```json
{
  "policy": {
    "rules": [
      {
        "enabled": true,
        "name": "archive-by-tag",
        "type": "Lifecycle",
        "definition": {
          "filters": {
            "blobTypes": ["blockBlob"],
            "blobIndexMatch": [
              {
                "name": "env",
                "op": "==",
                "value": "dev"
              }
            ]
          },
          "actions": {
            "baseBlob": {
              "tierToArchive": {
                "daysAfterModificationGreaterThan": 60
              }
            }
          }
        }
      }
    ]
  }
}
```

**Validation and troubleshooting**

After you create a policy, verify that it is actually taking effect. Lifecycle policies are evaluated by the service on a schedule, so changes are not always immediate.

Ways to validate:

1. Check the policy in the Azure portal under **Lifecycle management**.
2. Use Azure CLI to retrieve the policy and confirm the JSON matches your intent.
3. Inspect blob properties to confirm the access tier changed.
4. Review timestamps and rule conditions to ensure the blob meets the age threshold.
5. Confirm the blob type is supported by the rule.

Useful CLI checks:

```bash
az storage account management-policy show \
  --account-name mystorageacct \
  --resource-group rg-storage
```

```bash
az storage blob show \
  --account-name mystorageacct \
  --container-name logs \
  --name app1.log \
  --query "{tier:properties.accessTier, lastModified:properties.lastModified, archived:properties.archiveStatus}"
```

If a rule does not appear to work, check these common causes:

- The blob does not match the prefix or tag filter.
- The blob type is not supported by the action.
- The age threshold has not been reached.
- Last access time tracking is disabled when the rule depends on access time.
- The blob is already in a tier that makes the action unnecessary.
- The policy JSON does not match the required schema.

**Common exam traps**

- **Policy schema** matters: the JSON must use the correct `policy` wrapper and action structure.
- **Access time** rules require tracking to be enabled and may not update instantly.
- **Archive** is not immediately readable.
- **Prefix** matching is name-based, not folder-based.
- **Snapshots and versions** need explicit planning.
- **Blob type support** is not uniform across block, append, and page blobs.
- **Multiple rules** can overlap, so design them carefully.

**Quick comparison**

| Tier | Best for | Access speed | Cost profile | Notes |
|---|---|---:|---|---|
| Hot | Frequent access | Fastest | Highest storage cost | Best for active workloads |
| Cool | Infrequent access | Fast | Lower storage cost | Good for data still needed online |
| Archive | Rare access | Slowest | Lowest storage cost | Requires rehydration before use |

**Recommended study pattern**

- Memorize the three tiers and their tradeoffs.
- Practice writing a valid lifecycle policy JSON file.
- Know where the policy lives in the portal.
- Understand which blob types and scenarios are supported.
- Be ready to explain why a blob did or did not move tiers.

**Summary**

Lifecycle management is an automation feature that reduces storage cost and enforces retention by moving or deleting blobs based on age, access, tags, or naming patterns. For AZ-104, the most important skills are recognizing the correct policy structure, understanding tier tradeoffs, and knowing the operational limits around access time tracking, archive rehydration, and supported blob types.
