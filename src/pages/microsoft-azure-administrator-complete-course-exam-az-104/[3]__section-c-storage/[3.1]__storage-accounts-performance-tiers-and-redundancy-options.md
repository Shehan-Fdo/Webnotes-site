## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Storage account fundamentals**

An Azure storage account is the top-level namespace for data services such as Blob, File, Queue, and Table storage. In AZ-104, you need to understand how account kind, performance tier, and redundancy settings affect cost, latency, durability, and feature availability.

A storage account is not just a container for data; it also defines the replication model, access tier defaults, and supported workloads. The choices you make at creation time can affect what services and performance options are available later.

- **General-purpose v2 (StorageV2)** is the default recommendation for most workloads.
- **BlobStorage** accounts are legacy and focus on blob data.
- **Premium** storage accounts are optimized for low latency and high IOPS, but they have feature and redundancy limitations.
- **Redundancy** determines how many copies Azure keeps and where those copies live.

**Account kinds and service scope**

Azure storage account kind determines which services and features you can use. For exam purposes, the most important distinction is between general-purpose accounts and specialized premium accounts.

- **StorageV2** supports Blob, File, Queue, and Table services.
- **BlobStorage** supports blob data and is generally considered legacy compared with StorageV2.
- **BlockBlobStorage** is a premium account kind for block blobs.
- **FileStorage** is a premium account kind for Azure Files.

A practical rule is that you choose StorageV2 unless you have a specific performance or feature requirement that points to a premium account. Premium accounts are often selected for workloads that need predictable latency, high IOPS, or low jitter rather than raw throughput alone.

> Premium tiers are usually about performance characteristics, not just capacity or throughput. For Azure Files and Blob workloads, low latency and consistent IOPS can matter more than simple bandwidth numbers.

**Performance tiers and access tiers**

Azure uses the term *performance tier* in two related ways: account performance class and blob access tier. AZ-104 commonly tests both, so keep them separate.

- **Standard** accounts use HDD-backed storage and are suitable for general-purpose workloads.
- **Premium** accounts use SSD-backed or specialized high-performance storage and are suited for latency-sensitive workloads.
- **Hot**, **Cool**, and **Archive** are blob access tiers that influence storage cost and retrieval cost, not the account’s hardware performance tier.

For blob data, the access tier helps you optimize cost based on how often data is read:

- **Hot** for frequently accessed data.
- **Cool** for infrequently accessed data that still needs online access.
- **Archive** for long-term retention with delayed rehydration.

The default access tier is configured at the account level for StorageV2 and BlobStorage accounts. That default applies to new blobs unless you explicitly set a different tier at upload time or later change the blob’s tier. For BlobStorage accounts, the account-level default access tier is especially relevant because it influences how newly created blobs are stored.

Example account creation with a default access tier:

```bash
az storage account create \
  --name mystorageacct01 \
  --resource-group rg-storage \
  --location eastus \
  --sku Standard_LRS \
  --kind StorageV2 \
  --access-tier Hot
```

Example of changing the default access tier after creation:

```bash
az storage account update \
  --name mystorageacct01 \
  --resource-group rg-storage \
  --access-tier Cool
```

Blob-level tier changes are separate from the account default:

```bash
az storage blob set-tier \
  --account-name mystorageacct01 \
  --container-name data \
  --name report.csv \
  --tier Archive
```

**Redundancy options and durability tradeoffs**

Redundancy controls how Azure replicates your data to protect against hardware failure, rack failure, datacenter failure, and regional outage. AZ-104 expects you to know the main options and when to choose each one.

| Option | Scope | Primary use | Tradeoff |
|---|---|---|---|
| **LRS** | Single datacenter | Lowest cost, basic protection | No protection from datacenter outage |
| **ZRS** | Multiple availability zones in one region | Higher availability within a region | Not available for every account/service combination |
| **GRS** | Primary region plus paired secondary region | Regional disaster recovery | Secondary is read-only until failover |
| **RA-GRS** | GRS plus read access to secondary | Read availability during primary outage | More cost and complexity |
| **GZRS** | Zone-redundant primary plus paired secondary region | Stronger availability and disaster recovery | Not supported everywhere |
| **RA-GZRS** | GZRS plus read access to secondary | Highest resilience among common options | Highest cost among standard redundancy choices |

A useful mental model is:

- **LRS** protects against local hardware failure.
- **ZRS** protects against zone-level failure within a region.
- **GRS/GZRS** protect against region-level failure by copying data to a paired region predetermined by Azure.
- **RA-** variants add read access to the secondary endpoint before failover.

The paired region for GRS and GZRS is not chosen arbitrarily by you; Azure uses a predetermined paired region for disaster recovery planning. That matters because replication scope is regional and not simply “any other region.”

> Do not assume redundancy means instant write continuity in the secondary region. In geo-redundant modes, the secondary is normally not writable until failover occurs.

**Replication behavior and failover**

Geo-redundant options replicate data from the primary region to the paired secondary region. The secondary copy is intended for disaster recovery, not for active write operations under normal conditions.

- **GRS** replicates asynchronously to the paired region.
- **RA-GRS** exposes a read-only secondary endpoint.
- **GZRS** combines zone redundancy in the primary region with geo-replication to the paired region.
- **RA-GZRS** adds read access to the secondary endpoint.

Failover behavior is important:

- **Account failover** switches the storage account’s primary endpoint to the secondary region.
- **Service-level failover** is not the same as a general application failover plan; your app must be designed to handle endpoint changes and transient errors.
- During failover, clients may see temporary write unavailability and DNS or endpoint propagation delays.
- After failover, the former secondary becomes the new primary, and the replication direction changes.

A simple topology view:

```text
Primary region
  ├─ LRS: 3 copies in one datacenter scope
  ├─ ZRS: copies across zones in the region
  └─ GRS/GZRS: replicated to paired region

Paired secondary region
  └─ Read-only until failover for RA-GRS/RA-GZRS
```

**Account kinds, redundancy support, and limitations**

Not every redundancy option is available for every account kind or service. This is a common AZ-104 trap.

- **Premium BlockBlobStorage** does not support the same redundancy choices as standard StorageV2 accounts.
- **Premium FileStorage** is designed for performance and has different redundancy support than standard file shares.
- **Azure Files** and **Blob** do not always share the same redundancy availability in premium scenarios.
- **ZRS** is not universally available across all regions, account kinds, and services.
- **Archive** access tier is for blob data only and is not a performance tier for file shares.

Practical exam guidance:

- Use **StorageV2** when you want the broadest feature set and redundancy flexibility.
- Use **premium** only when workload requirements justify it.
- Check the specific service and region before assuming ZRS or geo-redundant support.

**How to choose the right redundancy option**

Choose based on the failure domain you want to survive and the cost you can justify.

- **LRS** when cost is the main concern and the workload can tolerate a datacenter outage.
- **ZRS** when you need higher availability within a region and want to survive a zone failure.
- **GRS** when you need disaster recovery in another region but do not need read access to the secondary.
- **RA-GRS** when read access to the secondary endpoint improves availability or reporting.
- **GZRS/RA-GZRS** when you want zone resilience in the primary region plus geo protection.

For AZ-104, the key distinction is that redundancy is about *where* copies live and *what failure you can survive*, while access tiers are about *how often data is accessed* and *what it costs to keep online*.

**Changing redundancy after creation**

You can often change redundancy after the storage account is created, but the available transitions depend on the current account kind, region, and target option. Some changes are supported directly; others may require recreating the account or using a different account type.

Example of checking the current redundancy setting:

```bash
az storage account show \
  --name mystorageacct01 \
  --resource-group rg-storage \
  --query "sku.name" \
  -o tsv
```

Example of updating the SKU to a different redundancy level when supported:

```bash
az storage account update \
  --name mystorageacct01 \
  --resource-group rg-storage \
  --sku Standard_GRS
```

If you need to verify the account in the portal:

1. Open the storage account.
2. Review **Overview** for the SKU name.
3. Check **Data management** or **Configuration** for access tier settings.
4. Confirm the redundancy option in the account properties or SKU details.

**Access tier management in practice**

Blob access tiers are often used with lifecycle management to reduce cost over time. You can set a default tier at the account level and then override it per blob.

Common patterns:

- Upload new blobs to **Hot** for active workloads.
- Move older blobs to **Cool** using lifecycle rules.
- Archive compliance data that is rarely accessed.
- Rehydrate archived blobs only when needed.

Example lifecycle management policy:

```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-to-cool-after-30-days",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool": {
              "daysAfterModificationGreaterThan": 30
            }
          }
        }
      }
    }
  ]
}
```

Apply a policy with Azure CLI:

```bash
az storage account management-policy create \
  --account-name mystorageacct01 \
  --resource-group rg-storage \
  --policy @policy.json
```

**Protection features related to redundancy**

Redundancy is only one part of a broader data protection strategy. AZ-104 often expects you to recognize adjacent features that protect against accidental deletion, corruption, or malicious changes.

- **Soft delete** helps recover deleted blobs, file shares, or snapshots within a retention window.
- **Versioning** preserves previous blob versions after overwrites or deletes.
- **Snapshots** create point-in-time copies of blobs or file shares.
- **Immutable storage** prevents modification or deletion for compliance scenarios.

These features do not replace redundancy. Instead, they protect against logical data loss, while redundancy protects against infrastructure failure.

A practical strategy is:

- Use **LRS/ZRS/GRS** for infrastructure resilience.
- Use **soft delete** and **versioning** for accidental deletion and overwrite recovery.
- Use **immutable storage** for compliance and retention requirements.

**Verification and troubleshooting**

When validating a storage account, check both the account SKU and the service-specific settings.

Useful CLI checks:

```bash
az storage account show \
  --name mystorageacct01 \
  --resource-group rg-storage \
  --query "{sku:sku.name, kind:kind, accessTier:accessTier, location:primaryLocation}" \
  -o json
```

For blob service properties:

```bash
az storage account blob-service-properties show \
  --account-name mystorageacct01 \
  --resource-group rg-storage
```

For file share performance planning, remember that premium Azure Files is chosen for low latency and high IOPS, while standard file shares are usually chosen for cost efficiency and broader redundancy flexibility.

Common troubleshooting points:

- The selected redundancy option is not supported in the region.
- The account kind does not support the desired tier or redundancy.
- The workload uses premium storage but expects standard-tier features.
- The application assumes the secondary endpoint is writable before failover.
- The default access tier is set, but individual blobs were uploaded with a different tier.

**AZ-104 exam focus**

For exam questions, identify the requirement first:

- If the question emphasizes **lowest cost**, think LRS and standard tiers.
- If it emphasizes **zone failure resilience**, think ZRS or GZRS.
- If it emphasizes **regional disaster recovery**, think GRS or GZRS.
- If it emphasizes **read access to secondary**, think RA-GRS or RA-GZRS.
- If it emphasizes **frequent access and low latency**, think Hot tier or premium storage.
- If it emphasizes **long-term retention**, think Cool or Archive plus retention features.

A strong answer usually combines the right redundancy model with the right access tier and protection features, rather than treating them as interchangeable.

**Quick reference**

- **LRS**: cheapest baseline resilience.
- **ZRS**: zone-level resilience within one region.
- **GRS**: geo-replication to a paired region.
- **RA-GRS**: geo-replication plus read access to secondary.
- **GZRS**: zone redundancy plus geo-replication.
- **RA-GZRS**: zone redundancy plus geo-replication plus read access.
- **Hot/Cool/Archive**: blob access tiers, not account hardware tiers.
- **StorageV2**: default general-purpose account for most scenarios.
- **Premium**: choose for performance-sensitive workloads, especially Azure Files and block blobs.
