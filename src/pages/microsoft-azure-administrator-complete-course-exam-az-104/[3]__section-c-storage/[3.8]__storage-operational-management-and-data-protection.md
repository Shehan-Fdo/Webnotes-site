## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Azure Storage operational management overview**

Azure Storage is a core AZ-104 service because you are expected to secure it, control access to it, monitor it, and recover data from it. In practice, you manage four layers at the same time:

- **Account design**: redundancy, region choice, performance tier, and security defaults.
- **Access control**: Microsoft Entra ID, shared key, and SAS (Shared Access Signature).
- **Network control**: firewall rules, private endpoints, and trusted service access.
- **Data protection**: soft delete, versioning, snapshots, backup, and geo-redundancy.

A good exam answer usually depends on whether the question is about protecting the account, protecting the data, or restoring service after failure.

**Storage account types and common use cases**

Azure Storage is a platform service with multiple data services under one account:

- **Blob storage** for objects, containers, static content, logs, backups, and unstructured data.
- **Azure Files** for SMB (Server Message Block) and NFS (Network File System) file shares.
- **Queue storage** for simple messaging patterns.
- **Table storage** for NoSQL key-value style data.

For AZ-104, blob and files are the most operationally important because they are the services most often secured, backed up, and restored.

**Storage account redundancy and resilience**

Redundancy determines how many copies Azure keeps and where those copies live. Choose redundancy based on cost, availability, and disaster recovery needs.

| Option | Copies | Scope | Typical use |
|---|---:|---|---|
| **LRS** | 3 | Single datacenter | Lowest cost, local resilience |
| **ZRS** | 3 | Multiple availability zones in a region | Higher availability within a region |
| **GRS** | 6 | Primary region + paired secondary region | Regional disaster recovery |
| **GZRS** | 6 | Zone-resilient primary + secondary region | Stronger regional resilience |
| **RA-GRS** | 6 | GRS with read access to secondary | Read-only access during primary outage |
| **RA-GZRS** | 6 | GZRS with read access to secondary | Best resilience with read access |

Key exam points:

- **LRS** protects against local hardware failure, not region loss.
- **ZRS** helps when a zone fails, but it does not replace cross-region disaster recovery.
- **GRS/GZRS** replicate asynchronously to a paired region, so there is always some replication lag.
- **RA-** variants allow read access to the secondary endpoint, which can help with read-only workloads and disaster recovery validation.
- Geo-redundant options do not guarantee zero data loss because replication is asynchronous.

> If the question asks for the best protection against a regional outage, choose a geo-redundant option. If it asks for zone-level resilience within one region, choose ZRS or GZRS.

**Storage account security settings**

Several account-level settings are commonly tested because they reduce exposure before you even configure access rules.

- **Secure transfer required** forces HTTPS and blocks insecure HTTP access for supported endpoints.
- **Minimum TLS version** should be set to a modern version such as TLS 1.2 to reduce protocol downgrade risk.
- **Allow Blob public access** should usually be disabled unless you intentionally host public content.
- **Infrastructure encryption** adds an extra encryption layer for highly sensitive workloads.
- **Hierarchical namespace** enables ADLS Gen2 behavior for blob storage and changes how you manage directories and ACLs.
- **Shared key access** can be disabled in some designs to force Microsoft Entra ID and SAS-based patterns, but verify application compatibility first.

Practical guidance:

- Use **HTTPS-only** access for almost all production storage.
- Disable **public blob access** unless the workload explicitly requires anonymous content delivery.
- Prefer **Microsoft Entra ID** for human and application access when possible.
- Use **shared key** only when legacy tooling or service constraints require it.
- Use **SAS** for scoped, time-limited access when you need delegation without exposing the account key.

**Authorization models: Microsoft Entra ID, shared key, and SAS**

Azure Storage supports multiple authorization models, and AZ-104 expects you to know when each one is appropriate.

- **Microsoft Entra ID**:
  - Uses Azure RBAC for management-plane and data-plane authorization where supported.
  - Is preferred for users, groups, and managed identities.
  - Works well for blob data access and, in supported scenarios, Azure Files SMB access.
- **Shared key**:
  - Uses the storage account access keys.
  - Grants broad access to the account.
  - Is powerful but less granular and harder to govern.
- **SAS**:
  - Grants limited access to a specific resource, time window, and permissions.
  - Can be service SAS, account SAS, or user delegation SAS.
  - Is useful for temporary app access, uploads, downloads, and partner sharing.

Important distinctions:

- **Management plane** access controls the storage account resource itself in Azure Resource Manager.
- **Data plane** access controls the actual blobs, files, queues, and tables.
- A user can have permission to view the storage account in Azure but still be denied access to blob data.
- For blob data, Microsoft Entra ID plus RBAC is the preferred modern model.
- For Azure Files, Microsoft Entra ID-based authentication is supported in specific SMB scenarios, but it is not a universal replacement for Active Directory Domain Services (AD DS) or Microsoft Entra Domain Services.

Example: assign a blob data role to a managed identity

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<accountName>
```

Example: generate a user delegation SAS for blob access

```bash
az storage blob generate-sas \
  --account-name <accountName> \
  --container-name <containerName> \
  --name <blobName> \
  --permissions r \
  --expiry 2026-05-01T00:00Z \
  --auth-mode login \
  --as-user
```

**Access keys and key rotation**

Storage account access keys are root-level secrets. Each account has two keys so you can rotate them with minimal downtime.

Operational best practices:

- Store keys in a secret manager such as Azure Key Vault.
- Prefer connection strings only for legacy applications.
- Rotate keys regularly and after any suspected exposure.
- Regenerate one key at a time, then update dependent applications before rotating the second key.

Typical rotation workflow:

1. Identify which key is currently active in applications.
2. Regenerate the inactive key.
3. Update applications and secrets to use the new key.
4. Validate access.
5. Regenerate the other key during a later maintenance window.

Example:

```bash
az storage account keys list \
  --resource-group <rg> \
  --account-name <accountName>

az storage account keys renew \
  --resource-group <rg> \
  --account-name <accountName> \
  --key primary
```

> Rotating a key is not automatically transparent to applications. If an app still uses the old key, it will fail until you update the secret or connection string.

**Network access control**

Network controls are a major part of storage hardening. You usually combine them with identity-based access.

Common controls:

- **Public network access** can be restricted or disabled.
- **Storage firewall** can allow selected networks only.
- **Virtual network rules** can permit specific subnets.
- **Service endpoints** can keep traffic on the Azure backbone while still using the public endpoint.
- **Private endpoints** expose the storage service through a private IP in your virtual network.
- **Trusted Microsoft services** can be allowed in some scenarios, but do not treat this as a blanket security exception.

Typical secure pattern:

- Disable broad public access.
- Allow only required subnets or private endpoints.
- Use private DNS so clients resolve the storage FQDN to the private IP.
- Use RBAC or SAS for authorization.

Example: allow a subnet through the storage firewall

```bash
az storage account network-rule add \
  --resource-group <rg> \
  --account-name <accountName> \
  --subnet <subnetResourceId>
```

Example: create a private endpoint for blob access

```bash
az network private-endpoint create \
  --name pe-storage-blob \
  --resource-group <rg> \
  --vnet-name <vnetName> \
  --subnet <subnetName> \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<accountName> \
  --group-ids blob \
  --connection-name pe-storage-blob-conn
```

Private DNS validation steps:

1. Create or link the private DNS zone for the service, such as `privatelink.blob.core.windows.net`.
2. Add the virtual network link.
3. Confirm the A record resolves to the private endpoint IP.
4. Test name resolution from a client in the VNet.

Example validation:

```bash
nslookup <accountName>.blob.core.windows.net
```

If DNS is correct, the resolved address should point to the private endpoint path rather than the public endpoint.

**Blob containers, versioning, snapshots, and soft delete**

Blob data protection usually combines multiple features because each one solves a different recovery problem.

- **Versioning** keeps previous versions of blobs after overwrite or delete operations.
- **Snapshots** create point-in-time copies of a blob.
- **Soft delete** retains deleted blobs or containers for a configured retention period.
- **Change feed** records changes for downstream processing and auditing scenarios.

How they differ:

- **Versioning** is best when users overwrite files frequently and you need rollback.
- **Snapshots** are useful for point-in-time recovery of a specific blob.
- **Soft delete** helps recover accidental deletions.
- None of these is a full substitute for a separate backup strategy or geo-redundancy.

Operational caveats:

- Soft delete retention is time-bound; once the retention window expires, recovery is no longer available.
- Snapshots and versions live in the same storage account, so they do not protect you from account deletion, subscription issues, or regional loss by themselves.
- Versioning and snapshots can increase storage consumption.
- If you enable versioning, test restore workflows so users know how to recover the correct version.

Example: enable blob versioning and soft delete

```bash
az storage account blob-service-properties update \
  --account-name <accountName> \
  --resource-group <rg> \
  --enable-versioning true \
  --enable-delete-retention true \
  --delete-retention-days 14 \
  --enable-container-delete-retention true \
  --container-delete-retention-days 14
```

**Azure Files operational management**

Azure Files provides managed file shares that support SMB and, in some cases, NFS. It is commonly used for lift-and-shift file shares, shared application content, and user profile data.

Key management topics:

- **Authentication** depends on the protocol and scenario.
- **SMB** access can use storage keys, AD DS, Microsoft Entra Domain Services, or supported Microsoft Entra ID-based scenarios.
- **NFS** access uses network and export controls rather than SMB-style identity integration.
- **Snapshots** provide point-in-time recovery for file shares.
- **Azure Backup** is the standard enterprise protection model for Azure Files in AZ-104 scenarios.

Important exam distinction:

- **File share snapshots** are quick, local point-in-time copies.
- **Azure Backup** provides policy-based protection, retention, and restore orchestration.
- Snapshots are useful for short-term recovery and operational rollback.
- Backup is better for longer retention, centralized policy, and restore from accidental deletion or corruption.

Restore limitations to remember:

- Restoring a file share snapshot usually returns data within the same share.
- Snapshot-based recovery does not create an independent copy in another account or region.
- Azure Backup restore behavior depends on the selected restore point and target path.
- If the share or account is deleted, snapshot-only protection is not enough.

Example: create a file share snapshot

```bash
az storage share-rm snapshot \
  --resource-group <rg> \
  --storage-account <accountName> \
  --name <shareName>
```

Example: create a file share

```bash
az storage share-rm create \
  --resource-group <rg> \
  --storage-account <accountName> \
  --name appshare \
  --quota 100
```

**Azure Backup for Azure Files**

For Azure Files, the standard AZ-104 answer is usually **Recovery Services vault**. That is the common and broadly supported backup model for file shares.

What Azure Backup gives you:

- Centralized backup policy.
- Retention across daily, weekly, monthly, and yearly schedules.
- Restore to original or alternate location, depending on configuration.
- Protection against accidental deletion and many operational mistakes.

What it does not replace:

- Network security controls.
- Identity and access management.
- Geo-redundant storage design.
- Application-level consistency planning for active workloads.

Operational notes:

- Backup policies should match business recovery objectives.
- Test restores regularly.
- Understand whether you need item-level restore or full share recovery.
- Keep in mind that backup and snapshot retention are different mechanisms.

**Failover and geo-redundant recovery**

Geo-redundant storage helps with regional outages, but you need to understand how failover works.

Key concepts:

- The **primary endpoint** serves read/write traffic during normal operation.
- The **secondary endpoint** is normally read-only when read access is enabled.
- Replication to the secondary region is asynchronous.
- During an account failover, Azure promotes the secondary region to primary.

Operational impact of failover:

- Write access shifts to the new primary region.
- The old primary becomes unavailable as the active write endpoint.
- Some data may be lost if it had not replicated before the outage.
- Applications may need DNS refresh or endpoint validation after failover.

Use failover only when the primary region is unavailable or when disaster recovery procedures require it. It is not a routine maintenance action.

Example:

```bash
az storage account failover \
  --resource-group <rg> \
  --name <accountName>
```

> Account failover is a disruptive action. Treat it as a disaster recovery operation, not a troubleshooting shortcut.

**Monitoring, metrics, and diagnostics**

Monitoring helps you detect throttling, authentication failures, latency spikes, and capacity issues before users report them.

Useful signals to watch:

- **Availability** for service health.
- **Ingress and egress** for traffic patterns.
- **Success server latency** and **end-to-end latency** for performance.
- **Transactions** for workload volume.
- **Authentication errors** for identity or key problems.
- **Throttling** or server-side errors for capacity or request-rate issues.

Common log and alert scenarios:

- Repeated 403 responses can indicate authorization failure, SAS expiry, or firewall denial.
- 404 responses may indicate missing blobs, wrong paths, or deleted objects.
- 409 responses can indicate conflicts, lease issues, or naming collisions.
- 5xx responses can indicate transient service issues or regional problems.
- High latency can indicate client-side network issues, hot partitions, or service pressure.

Example diagnostic settings pattern:

```bash
az monitor diagnostic-settings create \
  --name storage-diag \
  --resource <storageResourceId> \
  --workspace <logAnalyticsWorkspaceResourceId> \
  --logs '[{"category":"StorageRead","enabled":true},{"category":"StorageWrite","enabled":true},{"category":"StorageDelete","enabled":true}]'
```

Use this only as a conceptual pattern; actual category names and availability vary by storage service and diagnostic mode. Always verify the current categories exposed by the resource.

Example alert rule for high transaction errors:

```bash
az monitor metrics alert create \
  --name storage-5xx-alert \
  --resource-group <rg> \
  --scopes <storageResourceId> \
  --condition "avg Transactions > 1000" \
  --description "Alert when storage transaction volume is unusually high"
```

Practical alert ideas:

- Alert on sustained 4xx spikes for authorization or firewall issues.
- Alert on 5xx spikes for service instability.
- Alert on latency above your application threshold.
- Alert on capacity growth if you are approaching quota or cost limits.

**Lifecycle management and data tiering**

Lifecycle management helps you reduce cost by moving data to cooler tiers or deleting stale data automatically.

Typical use cases:

- Move infrequently accessed blobs from **Hot** to **Cool**.
- Move archival data to **Archive** after a retention period.
- Delete temporary or expired data automatically.

Operational cautions:

- Archive tier data must be rehydrated before it can be read.
- Early deletion charges can apply when you move or delete data too soon from some tiers.
- Lifecycle rules should be tested carefully to avoid deleting active data.
- Versioning and snapshots can affect how lifecycle rules behave, so validate the policy against your data model.

Example lifecycle policy pattern:

```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-to-cool",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool": {
              "daysAfterModificationGreaterThan": 30
            }
          }
        }
      }
    }
  ]
}
```

This is a conceptual example. In real deployments, validate the current schema and supported actions for your storage account and API version.

**Practical end-to-end secure storage pattern**

A common AZ-104 scenario is to secure a storage account for application use while preserving recoverability.

Recommended pattern:

- Create the storage account with a secure default configuration.
- Disable public blob access.
- Require HTTPS and a modern TLS version.
- Restrict network access to selected subnets or private endpoints.
- Use Microsoft Entra ID and RBAC for application access where supported.
- Use SAS for temporary delegated access.
- Enable versioning and soft delete for blobs.
- Protect Azure Files with snapshots and Azure Backup.
- Monitor logs and metrics in Log Analytics.

Example architecture flow:

1. Application runs in an Azure VM or App Service with a managed identity.
2. Storage account allows only the application subnet or private endpoint.
3. The managed identity receives `Storage Blob Data Contributor`.
4. Blob versioning and soft delete are enabled.
5. Azure Files share is protected by Recovery Services vault backup.
6. Alerts watch for 403, 5xx, and latency spikes.

**Troubleshooting checklist**

When storage access fails, check the problem in this order:

- **Identity**: Is the user, group, or managed identity assigned the correct data role?
- **Authorization**: Is the SAS expired, malformed, or missing permissions?
- **Network**: Is the client on an allowed subnet, private endpoint, or trusted path?
- **DNS**: Does the storage FQDN resolve to the expected endpoint?
- **Protocol**: Is the client using HTTPS, SMB, or NFS as required?
- **Service state**: Is the account in failover, throttled, or regionally impacted?
- **Data protection**: Is the item recoverable through versioning, snapshot, soft delete, or backup?

Useful commands:

```bash
az storage account show \
  --resource-group <rg> \
  --name <accountName>

az storage account network-rule list \
  --resource-group <rg> \
  --account-name <accountName>

az storage blob list \
  --account-name <accountName> \
  --container-name <containerName> \
  --auth-mode login
```

**Exam focus summary**

For AZ-104, focus on these decision points:

- Choose **RBAC and Microsoft Entra ID** for modern blob access.
- Use **SAS** for scoped, time-limited delegation.
- Use **shared key** only when necessary and protect it carefully.
- Use **private endpoints** for strongest network isolation.
- Use **ZRS** for zone resilience and **GRS/GZRS** for regional resilience.
- Use **versioning, soft delete, and snapshots** for operational recovery.
- Use **Azure Backup** for Azure Files when you need policy-based protection and restore.
- Use **monitoring and alerts** to detect failures before they become outages.
