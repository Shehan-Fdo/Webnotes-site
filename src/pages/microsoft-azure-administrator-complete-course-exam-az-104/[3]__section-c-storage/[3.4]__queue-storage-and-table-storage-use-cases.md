## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Overview**

Azure Queue Storage and Azure Table Storage solve different problems inside the Azure Storage platform. Queue Storage is a lightweight messaging service for decoupling components, while Table Storage is a NoSQL key-value store for structured, schema-less data with fast lookups by key. For AZ-104, you should know when each service fits, what it does well, and where its limits matter.

A useful mental model is:

- **Queue Storage** moves work between components.
- **Table Storage** stores and retrieves data efficiently by key.

**Azure Queue Storage**

Queue Storage is designed for asynchronous communication. One component writes a message, and another component reads and processes it later. This pattern helps you absorb bursts of traffic, isolate failures, and keep user-facing apps responsive.

Use Queue Storage when you need:

- **Decoupling** between a front-end and a back-end worker.
- **Load leveling** during traffic spikes.
- **Simple task handoff** for background jobs.
- **At-least-once delivery** semantics where duplicate processing is acceptable and handled by your app.

Queue messages are small and simple. The service is not meant for complex routing, publish/subscribe fan-out, or guaranteed exactly-once processing. If you need richer messaging features, Azure Service Bus is usually the better fit.

**Queue Storage behavior and design implications**

Queue Storage preserves approximate enqueue order in many common cases, but it does not guarantee strict FIFO (first in, first out) delivery under all conditions. Retries, visibility timeouts, and multiple consumers can change the order in which messages are processed.

Important operational characteristics:

- Messages are retrieved by a consumer and become temporarily invisible.
- If the consumer does not delete the message before the visibility timeout expires, the message can reappear.
- Duplicate processing is possible, so your worker should be idempotent.
- Message size is limited, so store large payloads in Blob Storage and place a reference in the queue message.

A typical pattern is to enqueue a small JSON payload that contains an identifier, a blob URL, or a work item key.

```json
{
  "jobId": "7f3c2c1a-2b7d-4d7f-9f2f-1b2d8f8a1c11",
  "customerId": "C10293",
  "payloadUri": "https://storageaccount.blob.core.windows.net/jobs/job-7f3c2c1a.json",
  "action": "generate-invoice"
}
```

**Queue Storage use cases**

- **Background processing**
  - Image resizing
  - Report generation
  - Email dispatch
  - Invoice creation
- **Burst handling**
  - Accept many requests quickly and process them at a steady rate
- **Cross-component communication**
  - Web app sends tasks to worker roles or containerized workers
- **Retry-friendly workflows**
  - A worker can safely retry a task if the operation is idempotent

**Queue Storage CLI examples**

When using Azure CLI, include an authentication method such as `--auth-mode login`, `--account-key`, or `--connection-string`. The examples below use Azure AD login.

```bash
# Create a queue
az storage queue create \
  --name orders \
  --account-name mystorageacct \
  --auth-mode login

# Add a message
az storage message put \
  --queue-name orders \
  --content '{"orderId":"1001","action":"ship"}' \
  --account-name mystorageacct \
  --auth-mode login

# Peek at messages without removing them
az storage message peek \
  --queue-name orders \
  --num-messages 5 \
  --account-name mystorageacct \
  --auth-mode login

# Receive and delete a message after processing
az storage message get \
  --queue-name orders \
  --num-messages 1 \
  --account-name mystorageacct \
  --auth-mode login

# Delete a specific message using its message ID and pop receipt
az storage message delete \
  --queue-name orders \
  --message-id <message-id> \
  --pop-receipt <pop-receipt> \
  --account-name mystorageacct \
  --auth-mode login
```

> Queue message operations are sensitive to the exact CLI version. If a command name differs in your environment, check `az storage message -h` and `az storage queue -h` for the supported syntax.

**Queue Storage and reliability**

Queue Storage is a good fit when your application can tolerate duplicate delivery and eventual processing. To make it reliable in practice:

- Make workers **idempotent** so repeated processing does not corrupt data.
- Store a processing state or completion marker in a durable system.
- Use a visibility timeout long enough for normal processing, but not so long that failed work is delayed unnecessarily.
- Move poison messages to a dead-letter-like pattern in your own application logic if repeated failures occur.

A common pattern is:

1. Web app writes a job record to Table Storage or a database.
2. Web app enqueues the job ID.
3. Worker reads the queue message.
4. Worker loads the job record, processes it, and updates status.
5. Worker deletes the queue message only after successful completion.

**Azure Table Storage**

Azure Table Storage is a NoSQL key-value store for semi-structured data. It is useful when you need a large number of entities, flexible schema, and fast access by key without the overhead of a relational database.

Use Table Storage when you need:

- **Low-cost storage** for structured or semi-structured data
- **Schema flexibility** where different entities can have different properties
- **Fast point lookups** by partition key and row key
- **Simple metadata storage** for application state, logs, or reference data

Table Storage is not a relational database. It does not provide joins, foreign keys, or complex transactional querying across many entities. It works best when you design around access patterns.

**Core Table Storage concepts**

- **Table**: A collection of entities.
- **Entity**: A row-like record with properties.
- **PartitionKey**: Groups related entities and supports scalability.
- **RowKey**: Uniquely identifies an entity within a partition.
- **Timestamp**: System-managed property used for concurrency and tracking.

The combination of `PartitionKey` and `RowKey` uniquely identifies an entity. Good partition design matters because it affects performance, scalability, and query efficiency.

**Partitioning guidance**

Choose a `PartitionKey` that matches how you read and write data. If you place too many hot writes into one partition, you can create a bottleneck. If you spread data too widely without a query strategy, lookups become inefficient.

Common patterns:

- **Tenant-based partitioning**
  - `PartitionKey = tenantId`
- **Date-based partitioning**
  - `PartitionKey = yyyyMMdd`
- **Workload-based partitioning**
  - `PartitionKey = region` or `deviceType`
- **Entity-group partitioning**
  - `PartitionKey = orderId` for order header and related line items

A good partition key balances:

- Query efficiency
- Write distribution
- Operational simplicity

**Table Storage consistency and concurrency**

For successful operations, Table Storage is generally strongly consistent within the storage account. If you write an entity and then read it back, you should expect to see the latest committed value when the operation succeeds.

Concurrency is typically handled with **ETags** and optimistic concurrency:

- Read the entity and capture its ETag.
- Update the entity using the ETag.
- If another client changed the entity first, the update fails with a concurrency conflict.

This pattern helps prevent accidental overwrites.

```bash
# Create a table
az storage table create \
  --name inventory \
  --account-name mystorageacct \
  --auth-mode login

# Insert an entity
az storage entity insert \
  --table-name inventory \
  --entity PartitionKey=warehouse1 RowKey=item100 Name=Widget Quantity=25 \
  --account-name mystorageacct \
  --auth-mode login

# Query entities by partition key
az storage entity query \
  --table-name inventory \
  --filter "PartitionKey eq 'warehouse1'" \
  --account-name mystorageacct \
  --auth-mode login

# Show a single entity
az storage entity show \
  --table-name inventory \
  --partition-key warehouse1 \
  --row-key item100 \
  --account-name mystorageacct \
  --auth-mode login
```

**Table Storage use cases**

- **Application metadata**
  - User preferences
  - Feature flags
  - Device settings
- **Operational state**
  - Job status
  - Processing checkpoints
  - Audit markers
- **Large-scale simple lookup data**
  - Customer profile summaries
  - Inventory snapshots
  - Session metadata
- **Telemetry and log-style records**
  - Event records with partitioned access patterns

A practical example is an order-processing system:

- Queue Storage holds order IDs waiting to be processed.
- Table Storage stores order status, timestamps, and processing metadata.
- A worker reads the queue, loads the order entity from Table Storage, updates status, and completes the workflow.

**Queue Storage vs Table Storage**

| Aspect | Queue Storage | Table Storage |
|---|---|---|
| Primary purpose | Asynchronous messaging | NoSQL structured storage |
| Best for | Work handoff, background jobs | Key-based lookup, metadata, state |
| Data model | Messages | Entities with properties |
| Access pattern | Producer/consumer | Query by `PartitionKey` and `RowKey` |
| Ordering | Approximate, not strict FIFO | Not a messaging system |
| Delivery semantics | At-least-once | N/A |
| Typical size | Small messages | Larger datasets with simple schema |
| Common risk | Duplicate processing | Poor partition design |

**How to choose in real scenarios**

- Use **Queue Storage** when:
  - You need to decouple a web app from a worker.
  - You want to smooth out spikes in demand.
  - You can process tasks asynchronously.
  - Duplicate processing is acceptable with idempotent logic.

- Use **Table Storage** when:
  - You need cheap, scalable storage for structured data.
  - You mostly read and write by key.
  - You do not need relational joins.
  - You want flexible schema and simple query patterns.

- Use both together when:
  - You need a durable work item record plus asynchronous processing.
  - You want the queue to carry only an identifier and keep the full state in Table Storage.

**End-to-end scenario 1: E-commerce order fulfillment**

A customer places an order on a website.

1. The app writes an order record to Table Storage with `PartitionKey = customerId` and `RowKey = orderId`.
2. The app enqueues the `orderId` in Queue Storage.
3. A worker reads the queue message and loads the order entity.
4. The worker reserves inventory, updates the order status, and records timestamps.
5. The worker deletes the queue message after success.

Why this design works:

- The website stays responsive.
- The worker can scale independently.
- The order record remains available for status checks.
- The queue only carries a small identifier, not the full order payload.

**End-to-end scenario 2: IoT device telemetry staging**

A fleet of devices sends periodic status updates.

- Queue Storage can buffer incoming processing tasks if the system needs to normalize or enrich data before storage.
- Table Storage can store device state snapshots, last-seen timestamps, and lookup-friendly metadata.
- Partitioning by device group or region helps distribute load.

This pattern is useful when you need simple, cost-effective storage and do not require complex analytics queries.

**Practical design tips**

- Keep queue messages small and include references, not large payloads.
- Make queue consumers idempotent.
- Design Table Storage around your read patterns, not around normalized relational design.
- Use `PartitionKey` to group data you query together.
- Use `RowKey` to make each entity unique within a partition.
- Avoid hot partitions by spreading high-volume writes.
- Store binary or large content in Blob Storage instead of Table Storage.
- Use Table Storage for metadata and Queue Storage for orchestration, not as a replacement for a full database or message broker when advanced features are required.

**Common exam traps**

- Queue Storage is not a replacement for Service Bus when you need topics, sessions, dead-letter queues, or advanced messaging guarantees.
- Table Storage is not a relational database and does not support joins.
- Queue Storage does not guarantee strict FIFO ordering.
- Table Storage is not for full-text search or complex ad hoc querying.
- Both services are best understood by their access patterns, not by generic storage terminology.

**Quick comparison checklist**

- If you need to **hand off work**, choose Queue Storage.
- If you need to **store keyed records**, choose Table Storage.
- If you need both, use Queue Storage for the trigger and Table Storage for the state.
- If you need enterprise messaging features, consider Service Bus instead.
- If you need relational queries, consider Azure SQL Database instead.

**Summary**

Queue Storage and Table Storage often appear together in Azure designs because they solve complementary problems. Queue Storage handles asynchronous work distribution, while Table Storage stores lightweight structured data with efficient key-based access. For AZ-104, focus on the service purpose, the data model, the access pattern, and the operational tradeoffs.
