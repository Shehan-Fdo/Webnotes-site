## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Enterprise storage design goals**

Azure Storage in enterprise scenarios is usually about balancing security, availability, cost, and operational simplicity. For AZ-104, you should be able to choose the right storage account kind, lock down network access, configure redundancy, and validate that data protection features match the business requirement.

A strong design starts with the workload:

- **Blob storage** for unstructured objects such as backups, logs, images, and application content.
- **Azure Files** for SMB/NFS-style shared file access.
- **Queue storage** for lightweight messaging between application components.
- **Table storage** for NoSQL key-value style data with simple access patterns.
- **Data Lake Storage Gen2** when analytics workloads need hierarchical namespace and POSIX-like directory semantics.

**Storage account configuration fundamentals**

A storage account is the control plane boundary for most storage services. In enterprise scenarios, you should know which settings are applied at the account level and which are applied per service.

Common account-level settings include:

- **Performance tier**: Standard or Premium, depending on workload latency and throughput needs.
- **Replication**: LRS, ZRS, GRS, GZRS, and read-access variants.
- **Secure transfer required**: Forces HTTPS for data plane access.
- **Minimum TLS version**: Commonly set to TLS 1.2 or later.
- **Public network access**: Can be enabled, disabled, or restricted by firewall rules.
- **Access tier**: Hot, Cool, or Archive for blob data.
- **Hierarchical namespace**: Enables ADLS Gen2 capabilities for blob storage accounts.

> <mark>Do not confuse account kind with capability.</mark> `StorageV2` is the general-purpose choice for most modern scenarios, but `BlobStorage` is still valid for blob-only workloads.

**Choosing the right storage account kind**

Use the account kind that matches the workload and exam scenario:

| Account kind | Typical use | Notes |
|---|---|---|
| `StorageV2` | General-purpose enterprise storage | Preferred default for most AZ-104 scenarios |
| `BlobStorage` | Blob-only workloads | Still valid; useful when only blob features are needed |
| `FileStorage` | Premium Azure Files | Designed for low-latency file shares |
| `BlockBlobStorage` | Premium block blob workloads | Used for high-performance blob scenarios |

For exam questions, look for clues:

- If the scenario mentions blobs, queues, tables, and files together, choose `StorageV2`.
- If the scenario is only about blob data and no other services, `BlobStorage` can still be appropriate.
- If the scenario requires premium file shares, choose `FileStorage`.

**Security baseline for enterprise storage**

A secure storage account usually combines identity, network, and data protection controls.

Recommended baseline:

- **Disable public access** where possible.
- **Require secure transfer**.
- **Set minimum TLS to 1.2**.
- **Use Azure RBAC** for management and data access where supported.
- **Use private endpoints** for private access from virtual networks.
- **Restrict firewall rules** to trusted IPs or subnets when public access must remain enabled.
- **Enable soft delete and versioning** for blob protection.
- **Use encryption with Microsoft-managed keys** unless the business requires customer-managed keys.

Example: create a secure storage account with common hardening settings.

```bash
az storage account create \
  --name staz104lab01 \
  --resource-group rg-storage-lab \
  --location eastus \
  --sku Standard_GRS \
  --kind StorageV2 \
  --https-only true \
  --min-tls-version TLS1_2 \
  --allow-blob-public-access false \
  --default-action Deny
```

This example shows the intent, but in real deployments you often also add explicit network rules or private endpoints after creation.

**Network access, firewall rules, and private endpoints**

Enterprise storage often needs controlled access from specific networks. Azure Storage supports both firewall-based public access restrictions and private endpoints.

Use firewall rules when:

- You need to allow a small set of public IPs.
- You are in a transitional phase before private connectivity is ready.
- The workload is simple and does not require full private DNS integration.

Use private endpoints when:

- You want traffic to stay on the Microsoft backbone through a private IP in your VNet.
- You need strong network isolation for regulated workloads.
- You want to disable public network access after validation.

> Private endpoints are usually the strongest network isolation option, but they only work correctly when DNS, routing, and subnet configuration are also correct.

Common network controls:

- **Public network access**: Enable or disable the public endpoint.
- **Firewall rules**: Allow selected IP ranges or VNets.
- **Service endpoints**: Extend VNet identity to the storage public endpoint.
- **Private endpoints**: Map storage services to private IPs in your VNet.

Example: allow a trusted office IP and a subnet.

```bash
az storage account network-rule add \
  --resource-group rg-storage-lab \
  --account-name staz104lab01 \
  --ip-address 203.0.113.10

az storage account network-rule add \
  --resource-group rg-storage-lab \
  --account-name staz104lab01 \
  --vnet-name vnet-prod \
  --subnet storage-subnet
```

Example: disable public network access after private connectivity is validated.

```bash
az storage account update \
  --resource-group rg-storage-lab \
  --name staz104lab01 \
  --public-network-access Disabled
```

Example: create a private endpoint for blob access.

```bash
az network private-endpoint create \
  --name pe-staz104lab01-blob \
  --resource-group rg-network-lab \
  --vnet-name vnet-prod \
  --subnet private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-storage-lab/providers/Microsoft.Storage/storageAccounts/staz104lab01 \
  --group-id blob \
  --connection-name peconn-staz104lab01-blob
```

In practice, you also need private DNS zone integration so clients resolve the storage FQDN to the private IP.

**Private DNS for storage private endpoints**

Private endpoints are incomplete without DNS. Clients must resolve the storage endpoint name to the private IP address.

Typical private DNS zones:

- `privatelink.blob.core.windows.net`
- `privatelink.file.core.windows.net`
- `privatelink.queue.core.windows.net`
- `privatelink.table.core.windows.net`

Practical flow:

1. Create the private endpoint.
2. Create or link the appropriate private DNS zone.
3. Add the DNS zone group to the private endpoint.
4. Verify name resolution from a client in the VNet.

Validation commands:

```bash
nslookup staz104lab01.blob.core.windows.net
nslookup staz104lab01.file.core.windows.net
```

If DNS is correct, the resolved address should be private when the client is inside the connected network.

**Redundancy and availability choices**

Replication determines how Azure copies data for durability and availability. AZ-104 questions often ask you to match the replication type to the business requirement.

Common options:

- **LRS**: Locally redundant storage; three copies in one datacenter.
- **ZRS**: Zone-redundant storage; copies across availability zones in a region.
- **GRS**: Geo-redundant storage; copies to a paired region.
- **RA-GRS**: Read-access GRS; secondary region is readable.
- **GZRS**: Geo-zone-redundant storage; combines zone and geo redundancy.
- **RA-GZRS**: Read-access GZRS; secondary region is readable.

Use cases:

- **LRS** for dev/test or noncritical workloads with lower cost.
- **ZRS** for regional resilience against zone failure.
- **GRS/RA-GRS** for disaster recovery when cross-region replication is required.
- **GZRS/RA-GZRS** for higher resilience with both zone and geo protection.

Important exam point:

- Replication protects against infrastructure failure.
- It does not replace application-level backup or versioning.
- Failover behavior depends on the replication type and whether Microsoft-initiated or customer-initiated failover is used.

**Blob data protection: soft delete, versioning, and lifecycle**

Blob protection features are common exam topics because they reduce accidental deletion and support recovery.

Key features:

- **Soft delete**: Retains deleted blobs for a retention period.
- **Blob versioning**: Keeps previous versions when blobs are overwritten.
- **Change feed**: Tracks blob changes for downstream processing or audit scenarios.
- **Lifecycle management**: Moves or deletes blobs based on age or access patterns.

Example: enable soft delete and versioning.

```bash
az storage account blob-service-properties update \
  --account-name staz104lab01 \
  --resource-group rg-storage-lab \
  --enable-delete-retention true \
  --delete-retention-days 14 \
  --enable-versioning true
```

Lifecycle management example:

```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-to-cool-and-archive",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool": { "daysAfterModificationGreaterThan": 30 },
            "tierToArchive": { "daysAfterModificationGreaterThan": 90 },
            "delete": { "daysAfterModificationGreaterThan": 365 }
          }
        }
      }
    }
  ]
}
```

Apply it with:

```bash
az storage account management-policy create \
  --account-name staz104lab01 \
  --resource-group rg-storage-lab \
  --policy @policy.json
```

Use lifecycle rules when:

- You need to reduce storage cost automatically.
- You have predictable retention patterns.
- You want to move logs or backups to cooler tiers over time.

**Access tiers and cost optimization**

Blob access tiers help you align cost with usage patterns.

- **Hot**: Frequent access, higher storage cost, lower access cost.
- **Cool**: Infrequent access, lower storage cost, higher access cost.
- **Archive**: Lowest storage cost, offline retrieval with rehydration delay.

Scenario guidance:

- Use **Hot** for active application data.
- Use **Cool** for backups or infrequently accessed content.
- Use **Archive** for long-term retention where retrieval is rare and delay is acceptable.

Remember:

- Archive data cannot be read immediately.
- Rehydration takes time and should be planned.
- Lifecycle policies can automate tier changes, but they do not replace backup strategy.

**Azure Files in enterprise scenarios**

Azure Files provides managed file shares accessible over SMB or NFS depending on the share type and account configuration. It is often used for lift-and-shift applications, shared home directories, and legacy app dependencies.

Common use cases:

- Shared application configuration.
- User profile or home directory storage.
- Lift-and-shift workloads that expect a file share.
- Cross-VM shared data for Windows or Linux clients.

**Azure Files authentication and permissions**

Azure Files access can be controlled with storage keys, SAS, or identity-based authentication depending on the scenario.

Authentication options:

- **Storage account key**: Simple, broad access; suitable for admin tasks and quick mounts.
- **SAS**: Time-bound delegated access.
- **Identity-based access**: Uses Microsoft Entra ID integration with AD DS, Azure AD DS, or Microsoft Entra Kerberos depending on the scenario.

For enterprise design, identity-based access is preferred when you need centralized control and least privilege.

Important distinctions:

- **Share-level permissions** control access to the share.
- **NTFS permissions** control file and folder access inside the share when using identity-based authentication.
- Both layers matter in Windows enterprise scenarios.

Typical configuration flow for identity-based Azure Files:

1. Choose the identity source:
   - **AD DS** for on-premises domain integration.
   - **Azure AD DS** for managed domain services in Azure.
   - **Microsoft Entra Kerberos** for cloud-native identity scenarios where supported.
2. Enable the identity-based authentication method on the storage account.
3. Assign share-level permissions.
4. Apply NTFS permissions on the mounted share.
5. Test access from a domain-joined or identity-enabled client.

> Azure Files identity-based access is not a single switch. You must configure the identity source, permissions, and client authentication path correctly.

Example: create a file share.

```bash
az storage share create \
  --account-name staz104lab01 \
  --name appshare \
  --quota 100
```

Example: mount Azure Files with a storage account key on Windows.

```powershell
$acct = "staz104lab01"
$share = "appshare"
$key = "<storage-account-key>"

cmd.exe /c "net use Z: \\$acct.file.core.windows.net\$share /user:Azure\$acct $key"
```

On some clients, the username format may vary by platform guidance, but the key point is that the password is the storage account key and the UNC path targets the file endpoint.

Linux SMB mount example:

```bash
sudo mount -t cifs //staz104lab01.file.core.windows.net/appshare /mnt/appshare \
  -o vers=3.0,username=staz104lab01,password=<storage-account-key>,dir_mode=0777,file_mode=0777,serverino
```

**Queue storage and Table storage**

Queue and Table storage are often underrepresented in study notes, but they still appear in AZ-104 scenarios.

Queue storage:

- Stores messages for asynchronous processing.
- Helps decouple application components.
- Supports simple work-item patterns and buffering.
- Has a small message size limit compared to service bus messaging.

Use Queue storage when:

- You need lightweight, low-cost message buffering.
- You do not need advanced broker features like topics, sessions, or dead-lettering at the same level as Azure Service Bus.

Table storage:

- Stores schemaless entities with partition and row keys.
- Works well for large-scale key-value access patterns.
- Is not a relational database.
- Is best when you need simple, fast lookups by key.

Practical limitations:

- Query patterns are limited compared to relational databases.
- Design around `PartitionKey` and `RowKey` for performance.
- Avoid ad hoc querying expectations.

Example: create a queue and insert a message.

```bash
az storage queue create \
  --account-name staz104lab01 \
  --name workitems

az storage message put \
  --account-name staz104lab01 \
  --queue-name workitems \
  --content "rebuild-index"
```

**Data Lake Storage Gen2 considerations**

When hierarchical namespace is enabled, the storage account supports ADLS Gen2 semantics.

Why it matters:

- Directory-level operations become more efficient for analytics workloads.
- Access control can be more granular for data lake scenarios.
- It is commonly used with big data and analytics tools.

Use hierarchical namespace when:

- The workload is analytics-oriented.
- You need directory semantics and POSIX-like access patterns.
- The scenario mentions data engineering, lakehouse, or large-scale file analytics.

Do not enable it casually if the workload is just standard blob storage and you do not need the feature, because it changes the account behavior and design assumptions.

**Encryption and customer-managed keys**

Azure Storage encrypts data at rest by default with Microsoft-managed keys. Customer-managed keys (CMK) add control when the organization requires key ownership, rotation policies, or compliance alignment.

CMK requirements typically include:

- A supported storage account type.
- A key in Azure Key Vault or Managed HSM.
- A managed identity for the storage account.
- Permission for the storage identity to access the key.
- Proper key vault networking and access configuration.

Operational impact:

- If the key becomes unavailable or access is revoked, storage operations can fail.
- Key rotation must be planned and tested.
- Key vault access should be monitored carefully.

Example: conceptual CMK configuration flow.

```bash
az storage account update \
  --name staz104lab01 \
  --resource-group rg-storage-lab \
  --assign-identity

az keyvault key create \
  --vault-name kv-storage-lab \
  --name cmk-storage \
  --kty RSA \
  --size 2048
```

Then configure the storage account to use the key URI and identity. The exact command varies by scenario and identity type, but the important exam concept is that `Microsoft.Keyvault` alone is not enough; you must specify the key and identity.

Key rotation guidance:

- Rotate keys according to policy.
- Verify the storage account can still access the new key version.
- Test recovery after rotation in a nonproduction environment first.

**RBAC and data-plane permissions**

Azure role-based access control (RBAC) is central to enterprise storage administration.

Common roles:

- **Storage Account Contributor**: Manage storage account settings, but not necessarily data.
- **Storage Blob Data Reader**: Read blob data.
- **Storage Blob Data Contributor**: Read/write blob data.
- **Storage File Data SMB Share Reader/Contributor/Elevated Contributor**: File share data access roles.
- **Reader**: View resource settings only.

Scope matters:

- Assign management roles at the resource group or storage account scope.
- Assign data roles at the storage account, container, share, or narrower scope when possible.

Example: assign blob data contributor at the storage account scope.

```bash
az role assignment create \
  --assignee <user-object-id> \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<subId>/resourceGroups/rg-storage-lab/providers/Microsoft.Storage/storageAccounts/staz104lab01
```

For least privilege:

- Use the narrowest scope that satisfies the requirement.
- Separate management access from data access.
- Prefer group-based assignments over direct user assignments.

**Monitoring, diagnostics, and troubleshooting**

Enterprise storage needs observability. You should know how to validate access, inspect metrics, and troubleshoot failures.

Useful monitoring tools:

- **Azure Monitor metrics** for capacity, transactions, latency, and availability.
- **Diagnostic settings** to send logs to Log Analytics, Event Hub, or Storage.
- **Activity log** for control-plane changes.
- **Storage analytics logs** where supported and configured.
- **Resource health** for platform-level issues.

Common diagnostic questions:

- Is the client blocked by firewall or private endpoint DNS?
- Is the storage account public access disabled?
- Is the identity missing a data-plane role?
- Is the replication type aligned with the availability requirement?
- Is the client using the correct endpoint and protocol?

Example: send diagnostics to Log Analytics.

```bash
az monitor diagnostic-settings create \
  --name storage-diag \
  --resource /subscriptions/<subId>/resourceGroups/rg-storage-lab/providers/Microsoft.Storage/storageAccounts/staz104lab01 \
  --workspace /subscriptions/<subId>/resourceGroups/rg-monitoring/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --logs '[{"category":"StorageRead","enabled":true},{"category":"StorageWrite","enabled":true},{"category":"StorageDelete","enabled":true}]' \
  --metrics '[{"category":"Transaction","enabled":true}]'
```

Validation commands:

```bash
az storage account show \
  --name staz104lab01 \
  --resource-group rg-storage-lab \
  --query "{httpsOnly:enableHttpsTrafficOnly,minTls:minTlsVersion,publicAccess:publicNetworkAccess,kind:kind,sku:sku.name}"
```

**Backup and recovery patterns**

Backup and recovery in Azure Storage depends on the service and the protection feature in use.

Blob recovery options:

- **Soft delete** for accidental deletion.
- **Versioning** for overwrite recovery.
- **Point-in-time restore** for supported blob scenarios.
- **Replication failover** for regional disaster recovery.

Azure Files recovery options:

- **Azure Backup for Azure Files** for scheduled backup and restore.
- **Share snapshots** for point-in-time recovery in some scenarios.
- **Replication** for availability, not backup.

Important distinction:

- Replication keeps copies available.
- Backup provides restore points and retention control.
- Soft delete and versioning protect against user error, not full disaster recovery by themselves.

Example: Azure Backup for a file share is typically configured through Recovery Services Vault rather than the storage account itself.

**Scenario 1: Secure application data with private access**

A finance application stores invoices in blob containers and must not expose data to the internet.

Recommended design:

- Use `StorageV2`.
- Enable secure transfer and TLS 1.2.
- Disable public network access after testing.
- Create a private endpoint for blob.
- Link private DNS.
- Assign `Storage Blob Data Contributor` to the application identity.
- Enable soft delete and versioning.
- Use lifecycle rules to move old invoices to Cool or Archive.

Why this works:

- It minimizes exposure.
- It supports recovery from accidental deletion.
- It keeps operational access manageable through identity and private networking.

**Scenario 2: Shared departmental file access**

A department needs a shared file location for Windows VMs and a legacy app.

Recommended design:

- Use Azure Files.
- Choose `StorageV2` or `FileStorage` depending on performance needs.
- Use identity-based authentication if the environment supports it.
- Apply share-level permissions and NTFS permissions.
- Use private endpoint if the share must not be reachable publicly.
- Use Azure Backup for the share if point-in-time restore is required.

Why this works:

- SMB access matches the application requirement.
- Identity-based access supports enterprise governance.
- Backup protects against accidental deletion and ransomware-style events.

**Scenario 3: Analytics landing zone**

A data engineering team needs a landing zone for raw files and directory-based processing.

Recommended design:

- Enable hierarchical namespace.
- Use blob storage with ADLS Gen2 semantics.
- Restrict access with RBAC and private endpoints.
- Use lifecycle policies to move cold data to lower-cost tiers.
- Monitor access and transaction metrics.

Why this works:

- Directory semantics improve analytics workflows.
- Private access and RBAC support enterprise security.
- Lifecycle management reduces cost over time.

**Common exam traps**

- Choosing `BlobStorage` as if it were deprecated; it is not deprecated, but `StorageV2` is usually the better default.
- Assuming private endpoint creation is complete without DNS configuration.
- Treating Azure Files key-based mounting as if any username format works universally.
- Assuming `--auth-mode login` alone creates identity-based Azure Files access.
- Thinking CMK is enabled by setting only the key source.
- Confusing replication with backup.
- Using queue storage for advanced messaging requirements that belong to Azure Service Bus.
- Using table storage for relational or complex query workloads.

**Quick validation checklist**

Before you finalize a storage design, verify:

- The account kind matches the workload.
- The replication type matches the availability requirement.
- Public access is restricted appropriately.
- TLS and secure transfer are enforced.
- The correct RBAC roles are assigned at the right scope.
- Private DNS is configured for private endpoints.
- Soft delete, versioning, and lifecycle rules are enabled where needed.
- Backup and recovery are defined separately from replication.
- Monitoring and diagnostics are sending data to a central workspace.

- Pick the correct storage service for the workload.
