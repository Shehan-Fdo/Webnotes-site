## **Course:** Microsoft Azure Administrator Complete Course (Exam AZ-104)

**Blob storage access model**

Azure Blob Storage separates **management-plane** actions from **data-plane** access. Management-plane operations create and configure the storage account, containers, and policies. Data-plane operations read, write, list, and delete blobs and container data.

For AZ-104, you need to know which control grants access in each scenario:

- **Azure RBAC** controls who can access storage data when using Microsoft Entra ID.
- **Shared Access Signatures (SAS)** grant time-bound, scoped access without sharing account keys.
- **Access keys** provide full account-level access and are powerful but broad.
- **Container public access levels** control anonymous access for blob containers only.
- **ACLs** matter when hierarchical namespace is enabled for Azure Data Lake Storage Gen2.

A common exam trap is assuming that a user who can manage a storage account can automatically read blob contents. That is not true unless the user also has the right data-plane permission path.

**Containers and access levels**

A container is a logical grouping of blobs. Its access level determines what anonymous users can do, but only for blob containers that allow public access at the storage account level.

The three container access levels are:

- **Private**: no anonymous access.
- **Blob**: anonymous users can read blob content and blob metadata if they know the blob URL, but they cannot list the container.
- **Container**: anonymous users can read blobs and list the blobs in the container.

This distinction matters:

- If a user has a direct blob URL, `Blob` access may allow read access.
- If a user needs to browse the container contents anonymously, `Container` access is required.
- If public access is disabled at the storage account level, container access levels cannot make data public.

> Public access at the storage account level overrides container-level public access settings. If public access is disabled, anonymous reads and listings are blocked even if a container is set to `Blob` or `Container`.

**Anonymous access behavior**

Anonymous access is often tested through URL-based scenarios.

- A known blob URL can be read anonymously only when the container allows `Blob` or `Container` public access and the blob itself is accessible.
- Listing blobs anonymously requires `Container` access.
- Listing containers at the storage account level is not the same as listing blobs in a container; storage account listing is not exposed as anonymous public browsing in the same way.

Practical interpretation:

- Use `Blob` when you want public file delivery without directory-style browsing.
- Use `Container` when you want a public gallery, index, or browsable set of files.
- Use `Private` for most enterprise workloads.

**Storage account public access controls**

Public access can be blocked at the storage account level. This is a common security hardening step.

Portal path:

1. Open the storage account.
2. Go to **Configuration**.
3. Set **Allow Blob anonymous access** to **Disabled**.
4. Save the change.

Azure CLI example:

```bash
az storage account update \
  --name mystorageacct \
  --resource-group rg-data \
  --allow-blob-public-access false
```

If you later need public access for a specific use case, re-enable the account setting first, then set the container access level as needed.

**Azure RBAC for blob data access**

Azure Role-Based Access Control (RBAC) can grant data-plane permissions when users authenticate with Microsoft Entra ID. This is the preferred enterprise model for most administrative and application scenarios.

Common built-in roles include:

- **Storage Blob Data Reader**: read blob data and container metadata/properties, and perform read/list operations where the scope and permissions allow it.
- **Storage Blob Data Contributor**: read, write, and delete blob data.
- **Storage Blob Data Owner**: full data access plus ability to manage access to blob data in supported scenarios.

Important exam nuance:

- Management roles like **Owner** or **Contributor** on the storage account do not automatically grant blob data access.
- Data roles must be assigned at an appropriate scope, such as the storage account, container, or sometimes a specific blob.
- Scope inheritance applies: a role assigned at the storage account can flow down to containers and blobs unless blocked by other controls.

Example role assignment:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/mystorageacct
```

To verify access, you can inspect role assignments:

```bash
az role assignment list \
  --assignee user@contoso.com \
  --scope /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/mystorageacct \
  -o table
```

In the portal, you can also use **Access control (IAM)** to review assignments and **Check access** to validate whether a user or group has the expected role.

**Entra ID prerequisites and troubleshooting**

For data-plane access with Microsoft Entra ID, the user or service principal must satisfy both identity and authorization requirements.

- The identity must be able to sign in to the tenant.
- The principal must have a data role such as Storage Blob Data Reader.
- The storage account must allow the intended access path.
- The client tool must be authenticated against the correct tenant and subscription.

Storage Explorer and the Azure portal can both be used for blob access, but they behave differently:

- In the portal, you usually need the correct RBAC role and the right subscription context.
- In Storage Explorer, you may need to sign in, select the correct tenant, and refresh the resource tree.
- If you can see the storage account but not the containers or blobs, the issue is often missing data-plane RBAC rather than missing management-plane access.

Useful troubleshooting checks:

- Confirm the user is in the correct Microsoft Entra tenant.
- Confirm the role is assigned at a scope that includes the target container or blob.
- Confirm public access is not being relied on when the account blocks anonymous access.
- Confirm the storage account firewall or networking rules are not blocking the client.

**SAS types and when to use them**

A Shared Access Signature (SAS) is a signed URI or token that grants limited access for a limited time. SAS is useful when you need to delegate access without giving away the account key.

There are three main SAS types:

| SAS type | Typical use | Strengths | Limitations |
|---|---|---|---|
| **Service SAS** | Access to a specific blob, container, queue, or file service resource | Fine-grained scope and permissions | Signed with the account key; revocation is limited unless the key changes |
| **Account SAS** | Access across multiple services in a storage account | Broader service coverage | Broader than service SAS; still key-based |
| **User delegation SAS** | Blob access using Microsoft Entra ID instead of the account key | Stronger security model; no account key exposure | Requires Entra authentication and supported blob service operations |

Use cases:

- Choose **service SAS** when you need a narrow, resource-specific token for a blob or container.
- Choose **account SAS** when a workflow needs access across multiple storage services and you can tolerate broader scope.
- Choose **user delegation SAS** when you want the security benefits of Entra ID and blob service support.

Example service SAS for a blob:

```bash
az storage blob generate-sas \
  --account-name mystorageacct \
  --container-name images \
  --name photo1.jpg \
  --permissions r \
  --expiry 2026-05-01T00:00Z \
  --https-only \
  --auth-mode login \
  --as-user
```

Example container SAS for listing and reading blobs in a container:

```bash
az storage container generate-sas \
  --account-name mystorageacct \
  --name images \
  --permissions rl \
  --expiry 2026-05-01T00:00Z \
  --https-only \
  --auth-mode login \
  --as-user
```

Notes on SAS permissions:

- `r` means read.
- `l` means list.
- `w` means write.
- `d` means delete.
- Permissions are resource-specific, so a permission string must match the resource type you are targeting.

> SAS tokens are time-bound and scope-bound, but they are still bearer tokens. Anyone who has the token can use it until it expires or is otherwise invalidated.

**SAS revocation and security**

Revocation depends on the SAS type.

- **Service SAS** and **account SAS** are tied to the storage account key. Rotating the account key can invalidate them.
- **User delegation SAS** is tied to a user delegation key obtained through Microsoft Entra authorization. It is not revoked by rotating the storage account key.
- For user delegation SAS, practical revocation usually means letting it expire, removing the underlying Entra access, or invalidating the authorization path that allowed the delegation key to be issued.

Security best practices:

- Use the shortest practical expiry time.
- Restrict SAS to HTTPS with `--https-only`.
- Limit permissions to the minimum required.
- Prefer user delegation SAS over account-key-based SAS when possible.
- Avoid embedding SAS tokens in source code or long-lived configuration files.

**Blob-level and container-level scope**

RBAC and SAS both support different scopes, and exam questions often test how scope affects access.

- **Container scope** is common for team access to a dataset.
- **Blob scope** is useful for a single file or object.
- **Storage account scope** is broader and easier to manage, but it grants access to more data.

Practical guidance:

- Use container scope for most shared application data.
- Use blob scope for one-off sharing or tightly controlled distribution.
- Use account scope only when the workload truly needs broad access.

Azure RBAC can be assigned at the blob path level, but that is less common than account or container scope. In many real deployments, teams prefer container-level assignments because they are easier to manage and audit.

**Hierarchical namespace and ACLs**

When hierarchical namespace (HNS) is enabled, the storage account behaves like Azure Data Lake Storage Gen2. In that mode, access control can involve both Azure RBAC and POSIX-like access control lists (ACLs).

Key points:

- RBAC still controls whether a principal can reach the data service.
- ACLs control access to directories and files within the namespace.
- A user may have RBAC permission but still be blocked by ACLs.
- ACL inheritance affects child directories and files unless explicitly overridden.

This is important for troubleshooting because a user may appear to have the right role but still receive authorization failures on a path.

Example ACL concepts:

- `r-x` on a directory allows read and traverse.
- `rw-` on a file allows read and write.
- `---` denies access.

If HNS is enabled, check both:

1. The RBAC assignment on the storage account or container.
2. The ACL on the target directory or file path.

**Portal and CLI access checks**

In the portal, you can validate access by combining IAM review with data browsing:

1. Open the storage account.
2. Select **Access control (IAM)**.
3. Review role assignments for the user, group, or managed identity.
4. Use **Check access** to confirm the effective role.
5. Open **Containers** and test whether the data plane is reachable.

CLI examples for common checks:

```bash
az storage account show \
  --name mystorageacct \
  --resource-group rg-data \
  --query allowBlobPublicAccess
```

```bash
az storage container list \
  --account-name mystorageacct \
  --auth-mode login \
  -o table
```

```bash
az storage blob list \
  --account-name mystorageacct \
  --container-name images \
  --auth-mode login \
  -o table
```

If these commands fail with authorization errors, check:

- The signed-in identity.
- The assigned data role.
- The storage account public access setting.
- Network restrictions.
- ACLs if HNS is enabled.

**Immutability and legal hold**

Blob immutability can prevent modification or deletion even when a user has write permissions. This is relevant in compliance and retention scenarios.

Common controls include:

- **Time-based retention policies**: prevent changes until the retention period ends.
- **Legal hold**: prevents deletion while the hold is active.
- **Immutable blob policies**: enforce WORM (write once, read many) behavior.

Exam implication:

- A user may have `Storage Blob Data Contributor` and still be unable to delete a blob if immutability or legal hold is active.
- These controls are often used for audit, finance, and regulated records.

**Practical access patterns**

A few common patterns help you reason through exam scenarios:

- **Public website assets**: use a container with public blob access only if the storage account allows it, but prefer static website hosting for website content.
- **Internal application data**: use private containers and RBAC or managed identity.
- **Temporary partner access**: use SAS with short expiry and narrow permissions.
- **Data lake workloads**: use HNS with RBAC plus ACLs.

**Static website hosting**

Static website hosting is a separate feature from normal blob container public access. It uses the special `$web` container and a website endpoint.

Important distinctions:

- The website endpoint is public by design when static website hosting is enabled.
- The `$web` container is not managed like a normal public container for anonymous browsing semantics.
- Container public access settings do not control the website endpoint in the same way they control standard blob containers.

Use static website hosting when you want to serve HTML, CSS, JavaScript, and other static assets directly from storage.

Example CLI enablement:

```bash
az storage blob service-properties update \
  --account-name mystorageacct \
  --static-website \
  --index-document index.html \
  --404-document 404.html
```

**Exam-focused comparison**

| Scenario | Best fit | Why |
|---|---|---|
| Internal app reads blobs | RBAC with managed identity | No shared secrets, easy to govern |
| Temporary external download | SAS | Time-limited delegation |
| Public file download by known URL | Container `Blob` access, if allowed | Anonymous read without listing |
| Public browsable file set | Container `Container` access, if allowed | Anonymous read and list |
| Data lake path access | RBAC plus ACLs | HNS requires both layers |
| Compliance-protected records | Immutability or legal hold | Prevents unauthorized change or deletion |

**Troubleshooting checklist**

When blob access fails, work through the problem in this order:

1. Confirm whether the request is anonymous, SAS-based, or Entra ID-based.
2. Check the storage account public access setting.
3. Verify the container access level.
4. Verify RBAC role assignments and scope.
5. Check ACLs if hierarchical namespace is enabled.
6. Check network rules, firewalls, and private endpoints.
7. Check immutability or legal hold if delete or overwrite fails.

This sequence helps you separate identity issues from storage configuration issues and from compliance restrictions.

**Key takeaways**

- Container access levels control anonymous access only when the storage account allows it.
- `Blob` access allows reading known blob URLs; `Container` access also allows listing.
- Azure RBAC is the preferred model for authenticated data access.
- SAS is useful for delegated, time-bound access, with user delegation SAS preferred when possible.
- HNS adds ACLs, which can block access even when RBAC is correct.
- Immutability and legal hold can override normal write and delete permissions.
- Static website hosting is public by design and is not governed the same way as normal container public access.
