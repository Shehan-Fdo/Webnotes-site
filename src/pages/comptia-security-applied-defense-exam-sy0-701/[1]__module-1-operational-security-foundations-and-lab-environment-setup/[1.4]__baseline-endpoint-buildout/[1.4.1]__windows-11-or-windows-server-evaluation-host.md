## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Windows telemetry and hardening context**

Windows 11 and Windows Server environments often need two things at the same time: visibility into activity and a consistent security posture. Sysmon provides detailed telemetry for monitoring and forensics, while security baselines help standardize defensive settings across systems.

Sysmon is a Windows system service and device driver that logs activity to the Windows Event Log. Microsoft describes it as disabled by default, so you must explicitly install and configure it before it produces events. Security baselines, by contrast, give you a structured way to apply and maintain recommended configuration settings across Windows and Windows Server deployments.

- **Sysmon** records system activity for later analysis.
- **Security baselines** help you keep defensive settings consistent.
- Windows and Windows Server are secure out of the box, but many organizations still want more granular control.
- Baselines are useful when you need repeatable hardening across multiple systems.

**Sysmon behavior and value**

Sysmon stays resident after installation and continues across reboots. It writes events to the Windows Event Log and can help you identify malicious or anomalous activity by showing process creations, network connections, and file creation time changes. It does not analyze events by itself, so you still need a collection and analysis path such as Windows Event Collection or SIEM (Security Information and Event Management) agents.

> Sysmon is only useful when it is installed, configured, and monitored. If you do not collect and review its events, you lose most of its defensive value.

- Sysmon runs as a **protected process**, which limits many user-mode interactions.
- It is designed for telemetry, not automatic detection or response.
- You can use its output for security monitoring and forensics.
- It supports downstream analysis through event collection tooling.

**Configuration considerations**

Effective Sysmon deployment depends on choosing a configuration and understanding how filtering works. Microsoft notes that include and exclude rules matter, so you should plan the configuration carefully before relying on the data. A weak or overly broad configuration can create noise, while a narrow one can miss important activity.

```powershell
# Example pattern for installing Sysmon with a configuration file
Sysmon64.exe -i sysmonconfig.xml
```

- Select a configuration that matches your monitoring goals.
- Review include and exclude behavior before deployment.
- Protect the service so it remains reliable.
- Verify that events are reaching the Windows Event Log.

**Security baselines**

Security baselines provide a practical starting point for hardening Windows systems. Microsoft offers baseline guidance for Windows and Windows Server, and the Windows Server 2025 baseline includes over 300 security settings. Baselines can be deployed through PowerShell, Windows Admin Center, and Azure Policy, and OSConfig uses a scenario-based approach to apply desired security measures.

- **Baseline** settings support standardized hardening.
- Windows Server 2025 baseline deployment supports on-premises and Azure Arc-connected devices.
- OSConfig is a security configuration stack for applying baseline scenarios.
- Baselines help you maintain consistency after deployment, not just at initial setup.

**How to think about the exam objective**

For Security+ purposes, the key idea is that evaluation hosts and lab systems should be both observable and hardened. Sysmon gives you detailed host telemetry, and baselines help you establish a secure starting point. Together, they support defensive validation, troubleshooting, and ongoing security management.

1. Identify the host role and decide whether you need telemetry, hardening, or both.
2. Install and configure Sysmon if you need detailed event visibility.
3. Apply an appropriate security baseline to standardize defensive settings.
4. Confirm that logging and baseline controls remain in place over time.

**Quick comparison**

| Area | Sysmon | Security baselines |
|---|---|---|
| Primary purpose | Collect detailed host telemetry | Standardize secure configuration |
| Output | Windows Event Log events | Hardened system settings |
| Default state | Disabled by default | Guidance-based, applied by administrators |
| Best use | Monitoring and forensics | Consistent hardening and maintenance |

**Practical exam takeaway**

If a question asks what to do first for a new Windows host, think about the security goal. If the goal is visibility, Sysmon is the telemetry tool. If the goal is hardening, a baseline is the structured starting point. In real environments, you usually want both.
