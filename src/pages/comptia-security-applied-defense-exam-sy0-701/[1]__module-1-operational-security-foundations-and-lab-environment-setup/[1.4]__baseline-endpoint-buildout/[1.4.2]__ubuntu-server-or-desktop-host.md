## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Ubuntu host security context**

An Ubuntu Server or Desktop host is a common endpoint or server platform where you apply baseline hardening and monitor for suspicious activity. For Security+ purposes, focus on the practical controls that improve visibility and reduce attack surface: secure initial setup, authentication logging, auditing, and ongoing baseline validation.

A secure Ubuntu deployment starts with reducing unnecessary risk. That means you create a non-root user with `sudo` privileges, use SSH carefully, and keep the system aligned with a known-good security baseline. On the desktop side, the same principles apply, but the tools and workflows may be more interactive.

- **Baseline**: a common secure operating state you can compare against over time.
- **Hardening**: removing or disabling unnecessary services, ports, and software.
- **Visibility**: collecting logs that help you detect misuse, brute force attempts, or unexpected privilege use.

**Initial setup and baseline hardening**

When you first build an Ubuntu host, establish a secure starting point before you expose it to regular use. The goal is to make the system predictable, manageable, and easier to audit later.

1. Log in securely for the first time.
2. Create a non-root user and grant only the privileges needed for administration.
3. Set up SSH access with care if you need remote administration.
4. Remove or disable services you do not need.
5. Confirm the host matches your intended security baseline.

> Keep the baseline practical: if a service is not required, do not leave it enabled just because it is installed.

A baseline is not a one-time task. You should check it regularly because configuration drift can reintroduce risk. This is especially important on systems that support applications, remote access, or administrative workflows.

**Authentication and login monitoring**

Ubuntu records privileged authentication activity in classic log files and in the systemd journal. These records help you spot stolen passwords, SSH brute force attempts, and unexpected `sudo` use.

Common sources include:

- `/var/log/auth.log` for privileged authentication events
- `journalctl` for structured systemd journal queries
- `/var/log/wtmp` for login history summaries used by `last`
- `/var/log/lastlog` for last-login information used by `lastlog`

Use these tools together when you need a fuller picture of who accessed the host and when. `auth.log` is especially useful for reviewing authentication failures and privilege escalation attempts, while `last` and `lastlog` help summarize login activity.

```bash
sudo tail -n 50 /var/log/auth.log
journalctl -u ssh
last
lastlog
```

- `auth.log` is useful for reviewing authentication-related events.
- `journalctl` gives you access to systemd-managed records.
- `last` and `lastlog` summarize login history rather than replacing detailed authentication logs.

**System auditing with auditd**

`auditd` is the Linux Audit System userspace daemon. It works with kernel-level auditing to track security-relevant events and write audit records to disk. On Ubuntu, this gives you deeper visibility into system activity than authentication logs alone.

The audit flow is straightforward:

- Applications generate system calls.
- The kernel audit framework evaluates those calls.
- `auditd` records matching events.
- Logs are written under `/var/log/audit`, typically in `audit.log`.

This is useful when you need evidence for compliance or security monitoring. The evidence specifically mentions use cases such as PCI-DSS, HIPAA, and CIS benchmarks, so treat `auditd` as a control for accountability and traceability.

```text
Applications -> Kernel Audit Framework -> auditd -> /var/log/audit/audit.log
```

- **Kernel-level** auditing captures events before they disappear from normal application view.
- **Audit records** support security monitoring and compliance evidence.
- **Log location** matters because you need to know where to look during an investigation.

**Practical monitoring workflow**

A good Ubuntu monitoring workflow combines authentication review, audit logging, and baseline checks. You do not need to treat these as separate silos; they reinforce each other.

1. Review authentication logs for failed logins, unusual `sudo` activity, and SSH anomalies.
2. Query the journal with `journalctl` when you need structured system records.
3. Check `last` and `lastlog` to summarize recent and historical logins.
4. Inspect `auditd` output for security-relevant system activity.
5. Compare the host against your baseline and correct drift.

This workflow helps you move from detection to validation. If you see suspicious authentication behavior, audit records can provide additional context. If the host has drifted from baseline, logging can help you determine whether the change was expected or malicious.

**Desktop host considerations**

Ubuntu Desktop uses the same core Linux security concepts as Ubuntu Server, but the operational focus is different. Desktop hosts often support interactive users, local applications, and accessibility features, so you still need to protect the system while keeping it usable.

- Use the command line when you need precise administrative control.
- Apply the same baseline mindset to desktop services and installed software.
- Monitor authentication and audit logs if the desktop is used for administrative access or sensitive work.

The main exam takeaway is that a desktop host is not exempt from hardening. If it stores credentials, accesses administrative tools, or connects to sensitive systems, you should treat it as part of your security boundary.

**Exam focus reminders**

- Know the purpose of `/var/log/auth.log` and `journalctl`.
- Know that `last` and `lastlog` summarize login history.
- Know that `auditd` provides kernel-level auditing and writes to `/var/log/audit/audit.log`.
- Know that secure baselines include disabling unnecessary services and ports.
- Know that Ubuntu hosts should start with a non-root administrative model.
