## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Network perimeter device overview**

A network perimeter device sits at the boundary between networks and helps control traffic entering or leaving a protected environment. In practice, this often means a firewall appliance or firewall software that enforces security policy at the edge.

For exam purposes, focus on how perimeter devices support segmentation, controlled connectivity, and baseline security. They are not just pass-through devices; they are part of the security design and must be deployed and maintained with the same care as other hardened components.

- **Firewall policy** defines what traffic is allowed or blocked at the boundary.
- **Security baselines** should include the device’s configuration, patch level, and operational settings.
- **Maintenance** matters because perimeter controls only help if they remain in place and current.

**pfSense as a perimeter platform**

pfSense software is installed to a target drive such as an SSD or HDD by booting from installation media and completing the installer. The installer is an online installer, so it requires Internet connectivity to download installation data from Netgate servers.

You should recognize the basic installation flow and the connectivity requirements because they reflect how a perimeter device is brought online in a real environment.

1. Boot from the installation memstick, ISO, or optical disc.
2. Complete the Netgate Installer.
3. Provide a network connection that can reach the Internet.
4. Use a WAN port connected to a live network that supports the required upstream connectivity type.

The installer can use:
- DHCP
- Static IP address
- PPPoE

> If the installer encounters an error while booting or installing from the media, use the documented troubleshooting path for installation issues.

**Installer options and recovery paths**

The pfSense installer presents a license screen first, and you must accept the terms to continue. After that, the welcome menu offers different paths depending on your task.

- **Install** continues the installation process.
- **Rescue Shell** opens a basic shell for advanced preparation, diagnostics, or repairs.
- **Configuration Restore** attempts to restore a previously saved configuration file and apply it to the target installation.

These options matter because perimeter devices often need recovery, repair, or migration support. A configuration restore path is especially useful when you are rebuilding a firewall from known-good settings.

**VLANs for segmentation at the perimeter**

<abbr title="Virtual Local Area Network">VLAN</abbr>s are used to create multiple discrete broadcast domains on a single switch. This lets one physical switching infrastructure behave like multiple isolated segments, which is a common perimeter and internal segmentation strategy.

VLANs support:
- Isolating hosts on specific segments
- Trunking between switches so devices on the same segment do not need to be on the same physical switch
- Multiple VLANs over a single physical port when the device supports trunking

Two requirements must be met to deploy VLANs:
- An <abbr title="Institute of Electrical and Electronics Engineers">IEEE</abbr> 802.1Q VLAN-capable switch
- A network adapter capable of VLAN tagging

> VLANs cannot be used with an unmanaged switch.

**VLAN configuration concepts**

When you create or edit VLANs in pfSense, the key settings describe how tagged traffic is identified and handled.

- **Parent Interface** is the physical interface that carries the VLAN traffic.
- **VLAN Tag Type** identifies the tag protocol identifier.
  - `C-Tag` uses value `0x8100` and is the default for typical 802.1Q VLAN interfaces.
  - `S-Tag` uses value `0x88a8` and is used as the outer tag in double-tagging scenarios such as QinQ.
- **VLAN Tag** is the VLAN identifier, or VID, and ranges from `1` to `4094`.
- **VLAN Priority** is the 802.1p Priority Code Point value and ranges from `0` to `7`.

A practical way to remember the distinction is that the parent interface carries the traffic, the tag identifies the VLAN, and the priority influences how packets are treated within that VLAN.

**Security baseline implications**

Perimeter devices should be deployed with a defined baseline and checked regularly. That baseline includes the security settings on the device, the operating system or firmware state, and any firewall rules that protect the application or network environment.

- Keep the device aligned with the approved configuration.
- Verify that security settings remain in place after deployment.
- Treat patching and configuration review as ongoing tasks, not one-time setup steps.

In exam scenarios, the best answer is often the one that preserves control at the boundary while maintaining a known-good configuration. That is the core role of a network perimeter device.
