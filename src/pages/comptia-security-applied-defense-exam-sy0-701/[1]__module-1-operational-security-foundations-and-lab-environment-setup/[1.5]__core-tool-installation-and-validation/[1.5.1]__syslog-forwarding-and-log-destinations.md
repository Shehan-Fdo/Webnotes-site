## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Syslog forwarding overview**

Syslog forwarding copies log entries from a device to a remote server. On pfSense, the remote logging options under `Status > System Logs` on the `Settings` tab enable this behavior. The main value is that the firewall keeps only a finite amount of local log data, while the remote destination can retain events for longer-term review.

Forwarding logs helps you preserve evidence when local logs roll over, are cleared, or become unavailable because local storage fails. It also supports troubleshooting when you need to review events that happened before a restart or after older entries were cycled out.

- **Retention** is a key reason to forward logs off-device.
- Remote copies improve visibility into events that are no longer on the firewall.
- Centralized logs help with long-term monitoring and incident review.

> Logs sent with this method are delivered in the clear unless the logging path is secured separately.

**Why remote log destinations matter**

A remote syslog server gives you a stable destination for security events from firewalls and similar devices. This matters when policy requires log retention for organizational or legal reasons. In those cases, keeping logs only on the source device is not enough.

Remote logging also helps when you need to analyze blocked traffic and firewall actions in a security monitoring workflow. A SIEM (Security Information and Event Management) platform or log management system can ingest these events for correlation and investigation.

- Use remote logging when local logs are too small for your retention needs.
- Use it when you need to analyze events across multiple systems.
- Use it when you want a central place to search and review firewall activity.

**Log transport and destination considerations**

Syslog commonly uses UDP (User Datagram Protocol) port `514`, while rsyslog often defaults to TCP (Transmission Control Protocol) on port `514`. In practice, the transport and port depend on the logging service and configuration you choose. Some environments also use `6514` for syslog-related traffic.

When you configure a logging server, make sure the service is installed and listening on the expected port. If you use Red Hat Enterprise Linux as a logging server, the rsyslog service must be present on the system that receives logs.

| Item | Practical note |
|---|---|
| UDP `514` | Common default syslog transport and port |
| TCP `514` | Common rsyslog default |
| `6514` | Commonly used for syslog-related secure logging setups |

**Forwarding workflow**

1. Enable remote logging on the source device.
2. Point the device to the remote syslog server.
3. Confirm the server is receiving events.
4. Review the logs in the destination platform for troubleshooting or monitoring.

**Operational benefits**

Forwarding logs to a destination server gives you a better chance of keeping the full event trail. That is especially useful when the firewall restarts, when administrators clear local logs, or when the device loses local storage but remains network-connected.

It also supports security operations by making firewall activity available to analysts who need to determine what was blocked, what was allowed, and what actions the firewall took.

**Key takeaways**

- Local firewall logs are finite, so remote destinations extend visibility.
- Remote syslog supports troubleshooting, monitoring, and retention.
- Logs may be unencrypted in transit unless you secure the transport.
- Destination servers must be configured to receive and retain the forwarded events.
