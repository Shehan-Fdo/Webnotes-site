## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Overview**

Phishing analysis starts with three high-value checks: the URL, the attachment, and the sender. Security teams use these artifacts to decide whether an email is safe, suspicious, or malicious. The goal is to validate what the message claims, compare it to what it actually contains, and enrich the result with reputation data.

A practical workflow focuses on evidence, not assumptions. You inspect headers, extract links, review file behavior or file reputation, and compare sender trust signals against historical context. This helps you prioritize threats early and reduce unnecessary escalation.

**URL analysis**

URL analysis is the process of evaluating a link’s destination and reputation before you interact with it. A malicious URL can be the first step in phishing, malware delivery, or credential theft, so validation is a core defensive task.

Common checks include:

- Comparing visible link text to the actual `href` destination.
- Reviewing the domain, subdomain, and path for suspicious structure.
- Checking the URL against threat intelligence or reputation sources.
- Using sandbox or visualization tools to inspect the destination safely.

When you analyze URLs, focus on whether the link matches the message’s intent and whether the destination has a known malicious history. MITRE D3FEND describes this as **URL Reputation Analysis**, which means analyzing the reputation of a URL to determine risk.

**Attachments**

Attachments can carry malicious content, embedded links, or exploit behavior. In phishing triage, you treat every unexpected attachment as suspicious until you verify its purpose and reputation.

Useful attachment checks include:

- Identifying the file type and whether it matches the sender’s claim.
- Reviewing whether the attachment is a document, archive, or executable.
- Checking the file against malware or threat intelligence services.
- Looking for signs that the attachment is designed to drop additional payloads or redirect the user.

Attachment analysis is especially important early in an investigation because it helps you determine how a threat is delivered and whether it needs immediate containment. If a file is unfamiliar or unexpected, you avoid opening it directly and instead use controlled analysis methods.

**Sender reputation**

Sender reputation helps you judge whether an email source is trustworthy based on historical behavior and observed risk. This includes the sender domain and the mail transfer agent (MTA), which is the system that relays email messages.

Sender MTA reputation analysis considers factors such as:

- How long the MTA has interacted with the enterprise.
- How many sender domains use that MTA.
- How many recipients in the enterprise receive mail from it.
- How many emails and replies have been observed from it.

MITRE D3FEND notes that sender MTA trust can be categorized as:

- unrated
- neutral
- trusted
- suspicious
- malicious

Reputation is contextual, not absolute. A legitimate sender can become less trusted if its domain is spoofed and used for unauthorized mail. That is why you combine reputation with header inspection and message content analysis.

**Email header inspection**

Header inspection gives you the technical trail behind the message. It helps you verify whether the email passed through expected systems and whether authentication checks support the sender’s claim.

Look for these header elements:

- `SPF` (Sender Policy Framework)
- `DKIM` (DomainKeys Identified Mail)
- `DMARC` (Domain-based Message Authentication, Reporting, and Conformance)
- `Received` chain

These fields help you map the message path and identify inconsistencies. A structured review of raw `.eml` files is a common first step in phishing analysis because it exposes the authentication and routing details that the visible email body hides.

**Practical triage workflow**

1. Open the raw message or `.eml` file in a safe analysis environment.
2. Inspect the header chain and authentication results.
3. Extract all embedded URLs and compare displayed text to actual destinations.
4. Check suspicious URLs against reputation or threat intelligence sources.
5. Review attachments for file type, intent, and malicious indicators.
6. Assess sender and MTA reputation using historical and contextual signals.
7. Record findings in a structured triage report for handoff if needed.

This workflow keeps your analysis consistent and supports incident response handoff. It also helps you separate benign messages from those that need containment or deeper investigation.

**Triage mindset**

A strong phishing workflow combines artifact analysis with reputation data. URLs tell you where the message wants you to go, attachments tell you what it wants you to open, and sender reputation tells you how much trust the message deserves. When you evaluate all three together, you improve detection quality and reduce the chance of a dangerous click.

> Treat reputation as one signal, not the final verdict. A trusted-looking sender or link can still be malicious if the surrounding evidence does not match.

**Key terms**

- **Artifact analysis**: Reviewing email components such as headers, URLs, and files to determine risk.
- **Threat intelligence**: External or internal data that helps you identify known malicious indicators.
- **Triage report**: A structured summary of findings used for escalation or handoff.
