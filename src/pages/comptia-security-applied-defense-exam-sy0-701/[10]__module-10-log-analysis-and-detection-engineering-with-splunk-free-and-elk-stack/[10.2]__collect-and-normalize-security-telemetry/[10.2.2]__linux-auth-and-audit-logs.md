## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Linux authentication and audit logging overview**

Linux telemetry is not one-to-one with Windows telemetry. The logging model, file locations, and tools differ, so you need to know where authentication and audit data live and how to interpret them in context.

- **Authentication logs** commonly capture login and privilege activity.
- **Audit logs** record system events that you explicitly watch with audit rules.
- Centralized monitoring tools may ingest both types of logs for security analysis.

On Ubuntu-based systems, `auth.log` is a key source for authentication-related events. On Red Hat Enterprise Linux (RHEL), audit data is stored in `/var/log/audit/audit.log` by default.

**Audit log files and storage**

The Audit system writes entries to `/var/log/audit/audit.log`. If log rotation is enabled, rotated `audit.log` files stay in the same directory. This makes the audit directory the first place to check when you need historical audit evidence.

A practical audit workflow is to define a rule for the file or action you want to monitor, then trigger the activity and review the resulting event.

```bash
-w /etc/ssh/sshd_config -p warx -k sshd_config
cat /etc/ssh/sshd_config
```

- `-w` watches a file path.
- `-p warx` tracks write, attribute change, read, and execute activity.
- `-k sshd_config` assigns a searchable key to the rule.

> If `auditd` is not running, the action will not generate the expected audit event.

**What an audit event shows**

When a watched file is accessed, the audit record can include multiple event types in one sequence. The example event for reading `/etc/ssh/sshd_config` includes syscall, current working directory, and path details.

- `type=SYSCALL` shows the system call context.
- `type=CWD` shows the current working directory.
- `type=PATH` identifies the file involved.
- Fields such as `uid`, `gid`, `pid`, and `comm` help you tie the event to a user and process.

These fields help you answer who did what, from where, and against which file. That is the core value of audit logging during investigations.

**Linux log sources used in security monitoring**

Security monitoring platforms often collect multiple Linux log files to build a fuller picture of activity. The evidence highlights these common sources:

- `auth.log`
- `syslog`
- `daemon.log`
- `dpkg.log`
- `kern.log`
- `audit/audit.log`
- `secure`
- `messages`
- `yum.log`

Different distributions expose different default files, so you should map the log source to the platform before you troubleshoot or forward data.

**How to think about telemetry on Linux**

Linux telemetry is useful for visibility into user, service, system, login, and privilege activity. The main challenge is interpretation: the same security question may require different log sources depending on the distribution and the event type.

- Use authentication logs for login and access-related activity.
- Use audit logs for file access and other watched actions.
- Use centralized collection when you need correlation across hosts or platforms.

**Practical exam takeaways**

1. Identify the right log file for the event you are investigating.
2. Check whether `auditd` is active before expecting audit records.
3. Use audit rules to watch sensitive files such as `/etc/ssh/sshd_config`.
4. Read audit events by correlating syscall, path, process, and user fields.
5. Remember that Linux logging varies by distribution and is not identical to Windows.

**Quick reference**

| Area | Example source | What it helps you see |
|---|---|---|
| Authentication | `auth.log` | Login and privilege activity |
| Audit | `/var/log/audit/audit.log` | Watched file and system events |
| Red Hat audit source | `audit/audit.log` | Audit data on supported systems |

**Why this matters for defense**

Audit and authentication logs give you evidence for detection, investigation, and response. When you know which file records which kind of activity, you can validate suspicious access, confirm process behavior, and support incident analysis with concrete host telemetry.
