## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Windows event logs and Sysmon overview**

Windows event logs provide a record of activity on the system, and Sysmon adds more detailed telemetry for security monitoring and forensics. Together, they help you understand what happened on a host, when it happened, and which process or network action was involved.

Sysmon is an advanced monitoring service from the Sysinternals suite. It records system activity and writes events to the Windows Event Log, but it does not analyze events for you. You use the collected data to investigate behavior, hunt for suspicious activity, and support incident response.

- **Windows Event Log** stores operating system and application events.
- **Sysmon** adds low-level telemetry beyond default Windows logging.
- **Telemetry** includes process execution, network communication, file modification, and configuration change activity.
- **UTC** timestamps help keep event timing consistent across systems.

**Where Sysmon events are stored**

On modern Windows systems, Sysmon events are written to the `Applications and Services Logs -> Microsoft -> Windows -> Sysmon -> Operational` log. This is the location you check when you need to review Sysmon output directly on the host.

Because timestamps are recorded in UTC, you should account for time zone differences when correlating events across endpoints, collectors, and other log sources.

> Sysmon is disabled by default. You must explicitly install and configure it before it produces events.

**What Sysmon helps you see**

Sysmon is useful when you need more detail than standard logging provides. It gives you high-value visibility into how software behaves on a system over time, which makes it useful for both detection and troubleshooting.

Common event categories include:

- **Process execution** to show what ran on the host.
- **Network communication** to show outbound or inbound connections.
- **File modification** to show changes to files.
- **Configuration change** to show changes that affect system behavior.

This level of detail helps you reason about activity in context. For example, a suspicious process launch becomes more useful when paired with network and file events from the same time window.

**Enabling and configuring Sysmon**

You do not get Sysmon telemetry automatically. You must install it and apply a configuration that defines what to collect. Effective deployment depends on choosing the right filters and making sure the service stays protected and reliable.

Key configuration points:

- **Install and configure** Sysmon before expecting events.
- Use **include** and **exclude** filtering carefully.
- Protect the service so logging remains available for analysis.
- Verify that events are being written consistently.

A practical approach is to start with a configuration that captures the telemetry you need, then refine it to reduce noise while preserving high-value events.

**Using Sysmon with Windows Event Forwarding**

In larger environments or labs, you often forward Sysmon data to a collector instead of reviewing each endpoint separately. Windows Event Forwarding (WEF) is a pull-based model where endpoints send events to a collector using WinRM over HTTP or HTTPS.

A typical setup includes:

- A **collector** that receives forwarded events.
- One or more **sources** that generate Sysmon data.
- **Group Policy** to configure forwarding on the sources.
- A central log store such as the **Forwarded Events** log on the collector.

This model keeps logging centralized and low maintenance. It also makes the data easier to search, compare, and export into a SIEM for further analysis.

**How to think about the data during investigations**

Sysmon events are most useful when you correlate them with other logs instead of reading them in isolation. You look for relationships between process activity, network activity, and file changes to build a timeline of behavior.

Useful investigation habits:

- Match events by time window and host.
- Compare process activity with network connections.
- Look for file changes that follow suspicious execution.
- Use the log source and UTC timestamp to align evidence.

**Quick reference**

| Topic | Practical takeaway |
|---|---|
| Sysmon purpose | Adds detailed host telemetry for security monitoring and forensics |
| Default state | Disabled until you install and configure it |
| Event location | `Applications and Services Logs -> Microsoft -> Windows -> Sysmon -> Operational` |
| Time format | UTC |
| WEF role | Centralizes forwarding from endpoints to a collector |

**Exam focus**

For Security+ purposes, remember that Windows logs provide broad system visibility, while Sysmon provides deeper endpoint telemetry. You should know where Sysmon events are stored, that Sysmon must be enabled explicitly, and that forwarding can centralize logs for analysis.
