## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Splunk Free search and dashboard workflow**

Splunk helps you collect, index, and visualize machine data so you can search for security events and turn results into operational insight. In a free-tier lab, you typically use the web interface to explore logs, build searches, and present findings in dashboards for quick review.

- **Search** is the core workflow for finding events in indexed data.
- **Dashboard** views help you organize search results into reusable panels.
- **Splunk Enterprise** and Splunk Cloud Platform both support search and visualization workflows, while Splunkbase provides apps and add-ons.
- The Search and Reporting app is a common place to build and refine queries.

> Use conservative lab assumptions when you practice: the free tier is useful for learning search logic and dashboard layout, but app availability and feature depth can vary by platform and version.

**Getting into the Splunk web UI**

A common lab setup uses Splunk Enterprise on a local machine or VM, then opens the web interface in a browser. Splunk commonly uses port `8000`, so you can reach the UI at `http://localhost:8000` when the service is running.

1. Install Splunk Enterprise or start your existing Splunk instance.
2. Open the web UI in a browser.
3. Confirm that your data source is indexed before you build searches.
4. Run a simple search to verify that events return as expected.

```bash
# Example local access check
curl http://localhost:8000
```

**Building useful searches**

Searches let you filter events, focus on a time range, and identify patterns such as repeated logons or suspicious activity. In security practice, you often start broad, then narrow the query as you learn what the data contains.

- Start with a known data source and a limited time window.
- Look for event codes or fields that indicate success, failure, or other notable activity.
- Refine the search until the result set is small enough to analyze.
- Reuse the final query when you need the same detection logic again.

A practical lab example is Windows logon analysis. Event code `4624` indicates a successful logon, and `4625` indicates a failed logon. Those values are useful when you want to compare normal access with repeated failures that may suggest brute force behavior.

```spl
index=main sourcetype=wineventlog (EventCode=4624 OR EventCode=4625)
| stats count by EventCode, user, src
```

**Turning searches into dashboards**

Dashboards give you a way to present search results as panels instead of one-off query output. This is useful when you want to monitor a scenario, compare counts, or show a summary of suspicious activity.

- Use a dashboard when the same search supports repeated review.
- Keep panels focused on one question each.
- Use labels that make the result understandable without opening the query.
- Prefer simple summaries first, then add detail panels if needed.

A dashboard is especially helpful when you are tracking a lab scenario such as repeated failed logons followed by a successful login. That pattern is easier to review when the search output is visualized in a panel rather than read line by line.

**Practical detection context**

Security-focused searches often look for noisy, repeated attempts that stand out from normal activity. Brute force detection is a common example because it relies on repeated credential guessing and can be expressed with search logic over log data.

- Repeated failures can indicate suspicious access attempts.
- A later success may matter if it follows many failures from the same source.
- Search logic should reflect the behavior you want to detect, not just the raw event count.
- Lab data can be synthetic, but the analysis workflow should still mirror real SOC practice.

**Working with apps and dashboards**

Splunkbase is the place to discover and install apps and add-ons for the Splunk platform. Some apps provide prebuilt dashboards for specific use cases, such as security or cloud monitoring.

- Use Splunkbase when you need additional dashboards or content packs.
- Check compatibility before you install an app.
- Prefer supported apps when you want a stable lab or study environment.
- Treat prebuilt dashboards as starting points, not as a replacement for understanding the underlying search.

**Study takeaways**

Splunk Free is most useful for learning the relationship between data, search, and visualization. If you can index a dataset, write a focused search, and place the result in a dashboard panel, you have the core workflow needed for exam-style analysis and practical SOC tasks.

- Index data first, then search it.
- Use event fields and codes to narrow the result set.
- Convert repeatable searches into dashboard panels.
- Validate what the data shows before you draw conclusions.
