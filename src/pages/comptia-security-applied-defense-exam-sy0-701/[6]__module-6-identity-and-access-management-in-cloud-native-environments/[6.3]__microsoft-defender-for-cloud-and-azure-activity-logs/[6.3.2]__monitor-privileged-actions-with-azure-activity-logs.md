## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Azure Activity Logs for privileged action monitoring**

Azure Monitor activity logs record management operations on Azure resources. These are also called **control plane** operations, which makes them useful for reviewing administrative changes rather than workload data. Common examples include creating a virtual machine, changing a Key Vault access policy, and deployment errors.

The activity log is available by default, so you do not need to enable collection before you start reviewing events. Azure generates the entries, and you cannot change or delete them. That makes the log a reliable source for audit and investigation workflows.

- Use the activity log to review administrative changes after the fact.
- Create alerts to get proactive notification when a relevant event occurs.
- Focus on resource management actions, not application-level activity.

**What the log can and cannot tell you**

Activity logs help you see that a privileged action happened, but they do not always provide full role context for every event. In practice, this matters when a user activates multiple eligible roles through Privileged Identity Management (PIM) during the same time window. You may be able to query actions performed by that user, but the log entry may not clearly identify which role was used for a specific action.

> For audit requirements that need session-style attribution, do not assume the activity log alone will map every action back to a specific activated role.

**How Azure logs support monitoring**

Azure identity and access monitoring often uses multiple log types. Azure Active Directory (Azure AD) provides activity logs, sign-in logs, audit logs, and provisioning logs. Together, these logs help administrators understand user behavior, troubleshoot issues, and support security and compliance goals.

For privileged action monitoring, the key idea is to correlate identity activity with resource management events. That gives you a better picture of who did what and when, especially when you are reviewing elevated access.

- **Activity logs** show management operations on Azure resources.
- **Sign-in logs** help you understand authentication activity.
- **Audit logs** support review of directory and identity changes.
- **Provisioning logs** track identity provisioning activity.

**Operational use cases**

Use Azure Activity Logs when you need evidence of administrative change. They are especially useful for:

- Reviewing who created, updated, or deleted Azure resources.
- Investigating access policy changes, such as Key Vault policy updates.
- Checking deployment failures and related management errors.
- Supporting compliance reviews that require evidence of control plane activity.

The logs are also useful for alerting. If a sensitive resource changes unexpectedly, an alert based on the activity log can help you respond faster than manual review alone.

**Practical monitoring approach**

1. Identify the privileged actions you care about, such as resource creation or access policy changes.
2. Review the activity log for those management operations.
3. Correlate the event time with identity and role activity when available.
4. Forward or centralize the data if auditors need a separate review location.
5. Use alerts for high-value changes so you do not rely only on periodic review.

**Key limitations to remember**

Azure Activity Logs are strong for management visibility, but they are not a complete session recording system.

- They capture control plane operations, not every action inside a workload.
- They may not show a clear role-to-action mapping when multiple privileged roles are active.
- They are best used with other identity and monitoring data for fuller context.

**Exam focus**

For Security+ style questions, remember the practical distinction:

- **Activity logs** = management operations on Azure resources.
- **Alerting** = proactive notification when a logged event matches a condition.
- **Correlation** = needed when you want stronger attribution for privileged user activity.
