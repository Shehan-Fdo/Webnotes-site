## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Access control for cloud resources**

Restricting access to storage and compute resources starts with matching permissions to the job that needs them. In Azure, <abbr title="Role-Based Access Control">RBAC</abbr> uses role assignments to control access to resources for users, groups, service principals, and managed identities.

The practical goal is **least privilege**: give only the access required to perform a task. Avoid broad roles at broad scopes when a narrower assignment will work.

- Assign access at the smallest useful scope.
- Use built-in roles when they fit the need.
- Create custom roles when built-in roles do not meet organizational requirements.
- Review who can access storage and compute resources on a regular basis.

**Azure RBAC fundamentals**

Azure RBAC is the main mechanism for controlling access to Azure resources. Role assignments are the control point, and built-in roles provide predefined permission sets. If those roles are too coarse or too limited, you can define custom roles.

A role assignment is not just a label; it determines what actions a principal can perform at a given scope. That makes scope selection as important as the role itself.

- **Users, groups, service principals, and managed identities** can receive role assignments.
- **Built-in roles** cover common access patterns.
- **Custom roles** help you tailor permissions to specific operational needs.
- **Scope** determines how widely the permission applies.

**Least privilege in practice**

Least privilege means you do not grant unrestricted permissions just because they are convenient. Instead, you grant the minimum access needed for the task and expand only when a real requirement exists.

This approach helps you segregate duties and reduce the impact of mistakes or compromise. It also makes access reviews easier because each assignment has a clearer purpose.

- Grant access only for the actions a user must perform.
- Prefer narrower scopes over subscription-wide access.
- Avoid using broader roles as a shortcut.
- Reassess permissions when job duties change.

> Use the most specific access model that still supports the work. Broader access increases risk without improving control.

**Storage access considerations**

Storage access can be controlled at different levels, but not every level supports the same methods. The evidence indicates that Azure Storage can be configured at the storage account level or container level through RBAC, SAS, and access keys, while file-level access uses SAS only.

Folder-level permission inside a container is not confirmed by the evidence, so treat it cautiously and verify the exact storage service behavior before relying on it.

- Storage account access can be managed with RBAC, SAS, or access keys.
- Container access can also be managed with RBAC, SAS, or access keys.
- File-level access is available through SAS only.
- Do not assume folder-level permissions exist unless the service documentation confirms it.

**Defender for Cloud and security recommendations**

Microsoft Defender for Cloud evaluates resources against built-in and custom security policies, as well as regulatory compliance frameworks. It continuously assesses resources and generates security recommendations when a resource does not meet a defined control.

These recommendations are meant to guide remediation and improve security posture. By default, the Microsoft Cloud Security Benchmark (MCSB) standard is enabled when cloud accounts are onboarded to Defender for Cloud.

- **Security policies** define the standards, controls, and conditions used for assessment.
- **Security recommendations** describe issues and remediation steps.
- **MCSB** is enabled by default for onboarded cloud accounts.
- Recommendations can come from built-in or custom standards.

**Managing recommendations and standards**

Defender for Cloud supports custom security standards and custom recommendations. You can create custom recommendations with a KQL (Kusto Query Language) query and assign them to a custom security standard.

Permissions matter here too. Creating a new security standard requires Owner permissions on the subscription, and creating custom recommendations requires Security Admin permissions. If you are reviewing rather than changing items, Security reader can view recommendations but cannot make changes.

- Use custom recommendations to harden security posture for specific needs.
- Assign custom recommendations to a custom security standard.
- Verify you have the required permissions before making changes.
- Use Security reader for visibility and Security admin for management.

**Review workflow**

A practical review workflow is to inspect recommendations, identify the affected resource, and then apply the smallest change that resolves the issue. Because Defender for Cloud continuously evaluates resources, your changes should be followed by another review to confirm the recommendation clears.

1. Review the recommendation in Defender for Cloud.
2. Identify the resource or workload that triggered it.
3. Apply the least disruptive fix that satisfies the control.
4. Recheck the recommendation status after the change.

**Exam focus**

For Security+ style questions, connect the control to the outcome:

- **Azure RBAC** controls who can access Azure resources.
- **Least privilege** limits exposure by reducing unnecessary permissions.
- **Defender for Cloud** evaluates posture and produces recommendations.
- **Custom standards and recommendations** help adapt controls to organizational needs.
