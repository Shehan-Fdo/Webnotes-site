## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Least-privilege fundamentals**

Least privilege means you grant only the permissions required to complete a task and no additional permissions. In AWS, you express that intent with IAM (Identity and Access Management) policies attached to identities such as users, groups, or roles, or to AWS resources.

A practical least-privilege approach starts with smaller permissions and expands only when a task truly needs it. This is especially important for powerful identities such as the root user, administrative users, and emergency access users, which should not be used for everyday work.

- Grant only the actions needed for the task.
- Scope permissions to the specific resources involved.
- Add conditions when you need to narrow when or how access applies.
- Prefer lesser-privileged test or working accounts while learning and validating access.

**IAM roles and when to use them**

An IAM role is an AWS identity with permission policies, but it is intended to be assumable rather than tied to one person. Roles do not use standard long-term credentials such as a password or access keys. Instead, when you assume a role, you receive temporary security credentials for the role session.

Use roles when you need to delegate access to:
- Users who need temporary access to resources they do not normally use
- Applications or services that need AWS access
- Principals in another AWS account
- Mobile apps that need to use AWS resources

> Use a role when temporary credentials and delegated access fit the use case better than a long-term IAM user.

**Policy structure and evaluation**

IAM policies are JSON documents. The element order in the document does not matter, but the policy must still be valid for the service and use case. Some elements are mutually exclusive, so you cannot combine certain pairs in the same statement.

Common policy concepts include:
- **Action** and `NotAction` are mutually exclusive
- **Principal** and `NotPrincipal` are mutually exclusive
- **Resource** and `NotResource` are mutually exclusive
- `Condition` is optional

AWS evaluates policies to decide whether a request is allowed or denied. The exact behavior depends on the policy type and the request context, but the core idea is consistent: the permissions in the policy determine access.

**Writing tighter policies**

Least-privilege policies are easier to maintain when you start with the narrowest useful scope. For example, if an EC2 instance needs configuration data from S3, allow read access only to the specific bucket that contains that data rather than broad S3 access.

You can also use policy examples and service-specific guidance when writing permissions for a particular AWS service. The details vary by service because each service exposes different actions, resources, and supported conditions.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlySpecificBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": [
        "arn:aws:s3:::example-config-bucket/*"
      ]
    }
  ]
}
```

- Keep the resource scope as specific as possible.
- Use only the actions required for the workflow.
- Add conditions when you need extra control over context.
- Review whether a service supports resource-based policies, identity-based policies, or both.

**Practical setup for least privilege**

When you are setting up AWS access, plan for least privilege from the start. If you are using IAM Identity Center, permissions sets can help you begin with controlled access. If you are working directly with IAM, create at least one additional user or role with reduced permissions for testing and day-to-day scenarios.

1. Identify the task the identity must perform.
2. List only the AWS actions required for that task.
3. Limit the policy to the specific resources involved.
4. Add conditions if the access should apply only in certain contexts.
5. Test the policy and refine it if access is too broad or too narrow.

**Key takeaways**

Least privilege is both a design principle and an operational habit. You reduce risk by avoiding broad permissions, using roles for temporary delegated access, and shaping JSON policies around specific actions, resources, and conditions.

AWS also provides policy references, managed policies for common job functions, and tooling that can help you create baseline policies and then review them as your application changes. The goal is not just to make access work, but to make it work with the smallest practical permission set.
