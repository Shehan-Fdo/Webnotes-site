## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Security Hub and CloudTrail overview**

AWS Security Hub and AWS CloudTrail work together to improve visibility into account activity. Security Hub provides security posture checks, while CloudTrail records actions taken by users, roles, and AWS services. In practice, you use Security Hub to identify misconfigurations and CloudTrail to investigate who did what, when, and from where.

CloudTrail captures API calls as events, including calls from the Security Hub console and calls made through the Security Hub API. If you create a trail, CloudTrail can deliver events continuously to an Amazon S3 bucket. If you do not configure a trail, you can still review recent events in CloudTrail event history.

- **Security Hub** evaluates CloudTrail-related controls.
- **CloudTrail** records API activity for investigation.
- Event history is useful, but it is not a permanent record.

**CloudTrail controls in Security Hub**

Security Hub includes CSPM (Cloud Security Posture Management) controls for CloudTrail. These controls help you check whether logging and protection settings are in place. The controls focus on enabling trails, protecting log data, and making sure the trail is useful for detection and review.

Key CloudTrail-related controls include:

- CloudTrail should be enabled and configured with at least one multi-Region trail that includes read and write management events.
- At least one CloudTrail trail should be enabled.
- CloudTrail log file validation should be enabled.
- CloudTrail trails should be integrated with Amazon CloudWatch Logs.
- The S3 bucket used to store CloudTrail logs should not be publicly accessible.
- S3 bucket access logging should be enabled on the CloudTrail S3 bucket.
- CloudTrail trails should be tagged.
- CloudTrail encryption at rest should be enabled.
- CloudTrail Lake event data stores should be encrypted with customer managed AWS Key Management Service (AWS KMS) keys.

> These controls might not be available in all AWS Regions, so you should verify regional availability before relying on a specific check.

**What CloudTrail gives you for investigations**

CloudTrail helps you reconstruct activity by showing the request details behind an action. From CloudTrail data, you can determine the request made to Security Hub, the source IP address, who made the request, when it happened, and additional details. That makes it useful for incident response, audit support, and troubleshooting suspicious changes.

CloudTrail also helps track privileged activity. In AWS Organizations, the management account or a delegated administrator for IAM can use short-term root access through `sts:AssumeRoot` to perform privileged tasks on member accounts. CloudTrail records the `AssumeRoot` event, which lets you identify actions taken during that short-term session.

**Monitoring workflow**

1. Enable CloudTrail and create at least one trail for ongoing event collection.
2. Configure the trail to capture the management events you need, including read and write activity.
3. Protect the log destination by keeping the S3 bucket private and enabling access logging.
4. Turn on log file validation and encryption to strengthen integrity and confidentiality.
5. Use Security Hub to evaluate CloudTrail-related controls and surface gaps.
6. Review CloudTrail events when you need to trace a Security Hub action or a privileged session.

**Practical monitoring focus**

Security Hub is the control-check layer, and CloudTrail is the evidence layer. Together, they support both prevention-oriented review and detective monitoring. You use Security Hub to spot missing safeguards, then use CloudTrail to confirm activity and investigate the timeline.

- Use Security Hub to find weak logging or storage settings.
- Use CloudTrail to trace API activity and privileged actions.
- Use CloudWatch Logs integration when you want broader operational monitoring.

**Exam takeaways**

- A CloudTrail trail is required for an ongoing record of events.
- CloudTrail event history alone is limited to recent events and is not permanent.
- Security Hub can evaluate CloudTrail configuration through CSPM controls.
- CloudTrail logs can support investigations by showing request details and source information.
- `sts:AssumeRoot` activity is visible in CloudTrail and is relevant for privileged task tracking.
