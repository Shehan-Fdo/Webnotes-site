## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Suricata installation overview**

Suricata is an open source network security monitoring tool that you must install before you can use it. The documentation supports two practical approaches: install from binary packages or build from source. Package-based installation is the quickest path for most systems, while source installation gives you more control over build options and file locations.

- **Binary packages** are the simplest option when your distribution provides them.
- **Source** installation is recommended for users who want more control over the build.
- Verify source distribution files before building when you install from source.

**Package-based installation on Ubuntu**

The quickstart guide assumes a recent Ubuntu release and uses the official stable PPA (Personal Package Archive) for installation. This path also recommends installing `jq` so you can work with Suricata's EVE JSON output more easily.

```bash
sudo apt-get install software-properties-common
sudo add-apt-repository ppa:oisf/suricata-stable
sudo apt update
sudo apt install suricata jq
```

After installation, you can confirm the build options and service state with:

```bash
sudo suricata --build-info
sudo systemctl status suricata
```

- `suricata --build-info` shows the version and compile options.
- `systemctl status suricata` checks whether the service is running.
- `jq` helps format and inspect EVE JSON output.

**Source installation workflow**

Source installation follows a standard build sequence. The documentation shows the basic flow using a release archive, then configuring, compiling, and installing the software.

```bash
tar xzvf suricata-8.0.3.tar.gz
cd suricata-8.0.3
./configure
make
make install
```

This method installs Suricata into `/usr/local/bin/` by default, uses configuration files from `/usr/local/etc/suricata/`, and writes logs to `/usr/local/var/log/suricata`.

- `./configure` prepares the build.
- `make` compiles the source.
- `make install` places the binaries, config, and log paths in their default locations.

**Common configure options**

Suricata supports configure flags that change where it installs and how it is optimized. These options matter when you need portability, a virtual machine-friendly build, or a non-default filesystem layout.

| Option | Effect |
|---|---|
| `--disable-gccmarch-native` | Avoids hardware-specific optimization; useful for portable builds or VMs. |
| `--prefix=/usr/` | Installs the binary into `/usr/bin/` instead of the default `/usr/local/`. |
| `--sysconfdir=/etc` | Places configuration files under `/etc`. |

> Use conservative build options when you need the same package to run across different systems or virtual environments.

**Basic setup after installation**

Once Suricata is installed, the next step is to identify the interfaces and IP addresses it should inspect. This is the practical handoff from installation to monitoring: you confirm the service, then point it at the correct network traffic sources.

1. Identify the interface or interfaces you want Suricata to inspect.
2. Identify the IP address or addresses associated with those interfaces.
3. Confirm the service state with `systemctl status suricata`.
4. Review build details with `suricata --build-info` if you need to verify the installation.

**Key takeaways**

Suricata installation is straightforward, but the method you choose affects control and placement. Package installation is fast and convenient, while source installation gives you explicit control over build behavior and install paths. For exam prep, remember the default source locations, the Ubuntu PPA workflow, and the common configure flags that change portability and filesystem layout.
