## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Snort overview**

Snort is an open source intrusion detection and prevention system (IDS/IPS). It inspects network traffic using rules that match malicious activity and then generates alerts. In inline deployments, it can also stop matching packets.

Snort supports several common uses:

- **Packet sniffing** for observing traffic, similar to `tcpdump`
- **Packet logging** for traffic review and troubleshooting
- **Intrusion detection and prevention** for identifying and blocking suspicious traffic

Snort is also available in packaged form on some platforms, where it can be installed from a package manager and configured through a web interface.

**Detection model and rule sets**

Snort relies on detection signatures called rules. You can create custom rules or enable pre-packaged rule sets, depending on your environment and licensing needs.

Common rule sources mentioned in the documentation include:

- **Snort VRT** rules
- **Snort GPLv2 Community** rules
- **Emerging Threats Open** rules
- **Emerging Threats Pro** rules
- **OpenAppID** detectors and rules for application detection

> Snort rules are the core of detection. If the rules do not match the traffic you care about, Snort will not alert or block as expected.

Snort can also support application detection and filtering through OpenAppID in supported package deployments.

**Installation and initial setup**

A practical Snort deployment starts with installation, version confirmation, and basic network scoping. One common setup flow on Linux is to install Snort as a passive IDS, confirm the version, define `HOME_NET`, create a local rule, and validate the configuration without errors.

Typical setup goals include:

- Confirm Snort is installed and working
- Set `HOME_NET` to match your protected network
- Add at least one local rule for testing
- Validate the configuration before enabling alerts

A package-based deployment may be available from a system package manager, such as through a GUI package manager on supported firewall platforms.

**Configuration concepts**

Snort configuration centers on traffic scope, rule loading, and alert behavior. `HOME_NET` defines the network you want to protect, so it should reflect your internal address space rather than a broad default.

Key configuration ideas:

- **HOME_NET** identifies the protected network
- **Rules** define what traffic Snort should detect
- **Alerts** report matching events
- **Logging** records traffic or detections for later review

Snort can be configured to log detected events, block them, or do both, depending on deployment mode and platform support.

**Operational modes**

Snort is commonly discussed in two monitoring modes:

- **IDS**: passive monitoring that alerts on suspicious traffic
- **IPS**: inline prevention that can block traffic in real time

An IDS helps you observe and investigate activity without stopping it. An IPS adds enforcement, but inline placement introduces operational risk because the device sits in the traffic path.

**Validation and testing**

After installation and configuration, validate the setup before relying on it in production. The evidence emphasizes a simple verification path: confirm the version, ensure `HOME_NET` is correct, create a local rule, and check that configuration validation succeeds.

A basic validation workflow looks like this:

1. Install Snort.
2. Confirm the installed version.
3. Set `HOME_NET` for your environment.
4. Add a local test rule.
5. Validate the configuration.
6. Generate test traffic and confirm the alert appears.

**Practical deployment notes**

Snort documentation and platform guides emphasize that setup details vary by operating system and deployment model. Use the official setup guides and platform-specific documentation when you need exact installation steps.

Helpful reminders:

- **Platform differences** matter for package names and configuration paths
- **Rule selection** affects what Snort detects
- **Testing** confirms that alerts and logging work before production use

**Exam focus**

For Security+ purposes, focus on what Snort does and how it differs from related controls:

- **IDS** detects and alerts
- **IPS** detects and blocks
- **Rules** drive detection
- **Inline placement** enables prevention
- **Passive placement** supports monitoring without blocking

Understanding these relationships helps you answer questions about network defense, alerting, and traffic control.
