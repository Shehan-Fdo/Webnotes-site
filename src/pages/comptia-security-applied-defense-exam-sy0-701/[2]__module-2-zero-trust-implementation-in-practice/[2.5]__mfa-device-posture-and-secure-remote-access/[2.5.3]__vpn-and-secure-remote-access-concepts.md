## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**VPN fundamentals**

A <abbr title="Virtual Private Network">VPN</abbr> creates an encrypted tunnel between a user device and a private network. It protects data in transit and supports remote access when users need to reach internal resources from outside the organization.

- A VPN encrypts traffic between the endpoint and the network.
- After authentication, a VPN often grants broad access to the private network.
- VPNs are commonly used for remote work and protected file sharing.
- VPNs are primarily an access-enabling technology, not a complete security model.

> If a VPN is compromised, the attacker may gain access to more of the network than intended.

**Secure remote access models**

Secure remote access is about balancing accessibility with control. Traditional VPNs emphasize connectivity, while zero-trust approaches emphasize continuous verification and limited access.

- VPNs typically provide an encrypted tunnel into the network.
- Zero-Trust Network Access (<abbr title="Zero-Trust Network Access">ZTNA</abbr>) follows “never trust, always verify.”
- ZTNA validates user identity, device health, and context before granting access.
- ZTNA usually limits access to specific resources based on role.

This difference matters because broad network access can conflict with the principle of least privilege. In practical terms, secure remote access should give users only the access they need, not the entire internal environment.

**Split tunneling**

Split tunneling lets you route some traffic through the VPN while other traffic goes directly to the internet. This can improve performance for services that work better when location is known, while still protecting selected communications.

- **Split tunneling** separates traffic paths by application or device.
- Sensitive traffic can stay encrypted through the VPN.
- Non-sensitive traffic can bypass the VPN for direct internet access.
- The tradeoff is increased security risk if traffic is not carefully controlled.

**VPN vs. ZTNA**

| Aspect | VPN | ZTNA |
|---|---|---|
| Access model | Broad network access after authentication | Limited, role-based access |
| Trust approach | Trusts the authenticated session more broadly | Continuously verifies identity, device, and context |
| Security posture | Protects data in transit with encryption | Applies least privilege more directly |
| Typical use | Remote access into a private network | More granular secure access to specific resources |

Use the comparison to decide whether the goal is network connectivity or tightly controlled application access. For exam purposes, focus on the access scope and verification model, not just the encryption feature.

**Operational considerations**

Remote access design should account for both usability and risk. VPNs remain useful when users need straightforward access to internal resources, but they can expose more of the network than necessary. ZTNA reduces that exposure by limiting access and rechecking trust conditions.

- Choose VPNs when you need encrypted remote connectivity into a private network.
- Choose ZTNA when you need more granular access control.
- Use split tunneling only when the security tradeoff is acceptable.
- Treat remote access as part of a broader defense strategy, not a standalone control.

**Key takeaways**

- A VPN encrypts traffic and creates a tunnel into the private network.
- ZTNA verifies continuously and grants limited access.
- Split tunneling improves flexibility but can increase risk.
- Least privilege is the main design goal for secure remote access.
