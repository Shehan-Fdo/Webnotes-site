## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Endpoint hardening fundamentals**

Endpoint hardening is the process of reducing attack surface by improving the security state of a device or system. In practice, you use patching and other security measures to strengthen the operating system and related controls. Hardening is not a one-time task; it is an ongoing maintenance activity that helps keep internal system components from becoming weak points.

A practical hardening approach focuses on the device itself and the settings that affect its security health. Common areas include:

- **Operating system** patch level and version
- **Security** tools and protections
- **Configuration** settings that reduce exposure
- **Access** controls that limit what the device can do
- **Compliance** status against required rules

> Hardening works best when you treat it as a repeatable process, not a checklist you run once and forget.

**Device posture and what it measures**

Device posture is the security health and configuration state of an endpoint at a given moment. Think of it as a snapshot of whether the device is currently aligned with your security requirements. Posture can change as soon as settings drift, patches fall behind, or protections are disabled.

Posture commonly reflects:

- Operating system settings
- Patch status
- Installed applications
- Firewall configuration
- Enabled protocols
- Open ports
- Password management
- Compliance state
- Group policy application
- Access controls

Modern posture management extends across Windows workstations and servers, Linux distributions, macOS devices, network infrastructure such as switches and routers, and, where applicable, operational technology and industrial control systems.

**Compliance policies and validation**

Compliance policies let you define rules and conditions that managed devices must satisfy. In Microsoft Intune, these policies evaluate device configuration and can mark devices as compliant or non-compliant based on the rules you set. This gives you a clear way to validate whether a device meets your minimum security baseline.

For Windows devices in Intune, compliance settings can include requirements such as:

- **BitLocker** being enabled
- Minimum and maximum operating system versions
- Risk level based on Microsoft Defender for Endpoint
- Device health requirements to help ensure a trusted boot state

When a device meets the policy, it is considered compliant. When it does not, it is marked non-compliant. This status is useful because it turns posture into an enforceable decision point rather than just a report.

**Conditional Access and enforcement**

Compliance results become more powerful when you connect them to Microsoft Entra Conditional Access. Conditional Access can use the current compliance status of a device to decide whether access is allowed. That means only devices that meet your requirements can reach corporate resources.

This model gives you two layers of control:

- **Evaluation** through compliance policies
- **Enforcement** through access rules

The practical benefit is simple: you can block or limit access from devices that do not meet your security baseline, even if the user credentials are valid.

**Validation workflow**

Use a structured workflow when you validate device posture.

1. Define the minimum device requirements.
2. Create a compliance policy for the target platform.
3. Apply the policy to managed devices.
4. Review the compliance result for each device.
5. Connect the result to Conditional Access if you need access enforcement.
6. Remediate devices that are non-compliant and recheck status.

This workflow keeps validation tied to real device state instead of assumptions.

**What to look for during posture review**

When you review a device, focus on the controls that most directly affect trust and access. A device may appear functional while still being out of compliance because of a missing patch, a disabled protection, or a configuration drift.

Useful review points include:

- Current operating system version
- Required security features
- Endpoint protection status
- Firewall and protocol settings
- Policy application status
- Whether the device is compliant or non-compliant

**Practical exam takeaways**

For exam purposes, connect the concepts this way:

- **Hardening** reduces risk by improving device security settings.
- **Device posture** describes the current security state of the endpoint.
- **Compliance policies** define the rules used to evaluate that state.
- **Conditional Access** uses compliance status to control resource access.
- **Validation** confirms that the device meets the required baseline.

These ideas often appear together because posture validation is how you prove that hardening is actually in place and effective.
