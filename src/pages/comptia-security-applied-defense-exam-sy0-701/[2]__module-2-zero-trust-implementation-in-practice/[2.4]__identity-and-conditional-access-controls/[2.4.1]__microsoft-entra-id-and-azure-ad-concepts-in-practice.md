## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Microsoft Entra ID and Azure AD in access control**

Microsoft Entra Conditional Access is a **Zero Trust** policy engine. It uses identity-driven signals to make access decisions and enforce policy. In practice, you treat access as an if-then rule: if a user wants a resource, then they must satisfy the required condition.

- Conditional Access combines signals from users and devices.
- Policies can require an action before access is granted.
- Microsoft Entra ID is the decision point that evaluates the policy outcome.

A practical takeaway is that access control is not only about passwords. The platform can require multifactor authentication (MFA), device compliance, or both, depending on the policy design.

**Conditional Access policy patterns**

Microsoft documents several common policy approaches when organizations are not ready to require full device compliance for everyone. These patterns let you phase in stronger controls without blocking all users at once.

- Require a compliant or Microsoft Entra hybrid joined device for administrators.
- Require a compliant device, Microsoft Entra hybrid joined device, or MFA for all users.
- Block unknown or unsupported device platforms.
- Disable browser persistence when needed for tighter session control.

> Use conservative policy rollout when you are introducing device-based controls. Start with targeted groups or fallback options before enforcing the strictest requirement across all users.

**Device compliance with Microsoft Intune**

Microsoft Intune works with Microsoft Entra to enforce device compliance policies. Intune evaluates whether a device meets minimum configuration requirements, and that compliance state is then used by Conditional Access.

Common compliance checks include:

- Requiring a PIN to unlock.
- Requiring device encryption.
- Requiring a minimum or maximum operating system version.
- Requiring that a device is not jailbroken or rooted.

When the device reports compliance information to Microsoft Entra ID, Conditional Access uses that signal to grant or block access to protected resources. This makes device health part of the access decision instead of a separate after-the-fact check.

**MFA and authentication strength**

MFA is a core control in Microsoft Entra ID. Microsoft notes that using MFA makes an account more than 99.9% less likely to be compromised, which is why it is a common Conditional Access requirement.

Microsoft Entra ID provides built-in authentication strengths:

- **Multifactor authentication strength**: less restrictive.
- **Passwordless MFA strength**: stronger than standard MFA.
- **Phishing-resistant MFA strength**: most restrictive.

You can use a built-in strength or define a custom authentication strength based on your environment. In practice, this lets you align the authentication requirement with the sensitivity of the resource.

**Identity management tasks in Entra ID**

Entra ID is also used for day-to-day identity administration. Lab guidance for identity management shows practical tasks you should expect to perform in the Entra admin center.

Typical tasks include:

- Creating and modifying users.
- Assigning administrative roles.
- Creating and modifying groups.
- Managing license assignments.
- Setting up MFA.
- Configuring Conditional Access policies.

These tasks matter because access control depends on accurate identity data. If users, groups, or roles are misconfigured, Conditional Access decisions may not match the intended security policy.

**How to think about the workflow**

A simple way to connect the concepts is to view them as a chain:

1. You define the identity and device signals you want to trust.
2. Microsoft Entra ID evaluates those signals through Conditional Access.
3. Intune supplies device compliance status when device controls are in use.
4. MFA or compliant device requirements determine whether access is granted.

This workflow helps you separate the control layers:

- **Identity** answers who the user is.
- **Device compliance** answers whether the device meets policy.
- **Conditional Access** decides whether access is allowed.

**Exam focus reminders**

For Security+ style questions, focus on the relationship between the services rather than memorizing product marketing language.

- Microsoft Entra Conditional Access is the policy engine.
- Intune provides device compliance data.
- MFA strengthens authentication decisions.
- Hybrid joined devices can be used as an alternative trust signal in some policies.
- Administrative access often gets stricter requirements than general user access.
