## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Least-privilege access in AWS**

<abbr title="Identity and Access Management">IAM</abbr> is the control plane for who can do what in AWS. The core security idea is **least privilege**: grant only the permissions required to complete a task, and nothing extra. This applies to human users and workloads alike.

AWS guidance emphasizes starting with broader managed permissions when needed, then moving toward tighter policies. For everyday work, avoid relying on highly privileged identities such as the root user, administrative users, or emergency access users unless the task truly requires them.

- Grant access to the specific resource needed, not an entire service.
- Use temporary credentials where possible instead of long-term credentials.
- Review permissions regularly and remove unused users, roles, policies, and credentials.
- Apply conditions to narrow when and how access is allowed.

> Use the root user sparingly and protect it carefully. It is not intended for routine administration.

**Writing restrictive IAM policies**

IAM policies can scope access by action, resource, and condition. A practical least-privilege pattern is to allow only the exact operation needed on the exact resource needed. For example, if an <abbr title="Amazon Elastic Compute Cloud">EC2</abbr> instance needs configuration data from a single <abbr title="Amazon Simple Storage Service">S3</abbr> bucket, allow read access only to that bucket.

Conditions make policies more precise. They help you restrict access based on request context, such as where the request comes from or what network path it uses. AWS also provides tools to help validate and generate policies based on observed access activity.

- Use specific resource ARNs instead of wildcard access when possible.
- Prefer policy conditions to narrow access further.
- Validate policies before deployment to confirm they are both secure and functional.
- Avoid full `*` administrative privileges unless there is a documented need.

**Identity-based and resource-based access**

AWS services do not all support the same IAM features. For example, Security Hub supports identity-based policies, policy actions, policy conditions, temporary credentials, forward access sessions, and service-linked roles, but it does not support resource-based policies, policy resources, ACLs, or service roles.

That means you should check the service’s IAM support before designing access control. If a service does not support a feature you expect, you need to use the supported IAM model instead of assuming a universal pattern.

| IAM feature | Security Hub support |
|---|---|
| Identity-based policies | Yes |
| Resource-based policies | No |
| Policy actions | Yes |
| Policy resources | No |
| Policy condition keys | Yes |
| ACLs | No |
| ABAC | Yes |
| Temporary credentials | Yes |
| Forward access sessions | Yes |
| Service roles | No |
| Service-linked roles | Yes |

**Network restrictions with request-origin controls**

AWS also supports scalable controls that restrict access from unexpected networks. Older condition keys such as `aws:SourceVpc` and `aws:SourceVpce` can work well for smaller environments or tightly scoped workloads, but they can become harder to scale across many VPCs and accounts.

To support broader environments, AWS introduced request-origin condition keys such as `aws:VpceAccount`, `aws:VpceOrgPaths`, and `aws:VpceOrgID`. These help you enforce access based on where the request originates, which is useful when you want consistent controls across multiple accounts.

- Use network-origin conditions to reduce access from untrusted paths.
- Prefer scalable controls when your environment spans many accounts or VPCs.
- Keep the policy model aligned with how your organization is structured.

**Security group restrictions and practical access control**

Security groups are a network-layer restriction that complement IAM. IAM decides whether an identity can perform an action, while network restrictions help decide whether traffic can reach a workload. In practice, you use both to reduce exposure.

A secure design limits access at multiple layers:

- IAM limits what the caller can do.
- Network controls limit where traffic can come from.
- Temporary credentials reduce the value of stolen long-term credentials.
- Regular review removes stale access before it becomes a problem.

**Operational checklist**

1. Start with the smallest set of permissions that still lets the task succeed.
2. Replace broad access with resource-specific permissions.
3. Add conditions to constrain access by context or network origin.
4. Use temporary credentials and roles for workloads and human users where possible.
5. Review policies, keys, and unused identities on a regular schedule.
6. Validate that the target AWS service supports the IAM features you plan to use.

**Exam focus**

For Security+ applied defense scenarios, expect questions that test whether you can:

- Recognize least-privilege IAM as a baseline security control.
- Choose scoped permissions instead of wildcard access.
- Use conditions to restrict access further.
- Distinguish identity-based controls from network restrictions.
- Understand that service IAM support varies and must be checked before implementation.
