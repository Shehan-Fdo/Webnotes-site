## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Lab prerequisites**

A lab built around Zero Trust Architecture (ZTA) depends on identity, device, and access checks rather than implicit network trust. In practice, you should expect every access request to be verified, even when the request comes from inside the network.

- **Zero Trust** assumes no user or device is trusted by default.
- Access decisions rely on signals such as user, device, and location.
- Verification applies to users, devices, and processes, not just logins.
- Remote work, cloud services, and hybrid environments increase the need for this model.

> Do not treat an internal network as automatically safe. Zero Trust replaces that assumption with continuous verification.

**Policy baseline**

A policy baseline gives you a starting point for access control before you expand or customize rules. Conditional Access is the policy engine that combines signals and enforces organizational access decisions using simple if-then logic.

- **Conditional Access** uses identity-driven signals to enforce policy.
- A basic rule can be framed as: if a user wants access, then they must complete an action.
- Common actions include multifactor authentication (MFA) for protected resources.
- Baselines help you start with broad protection and then modify for specific scenarios.

**Planning before you enforce**

Conditional Access is flexible, which makes planning important. Poorly scoped policies can block legitimate access or create unexpected results, so you should define the scope and expected behavior before turning on enforcement.

1. Identify the users, groups, or roles the policy applies to.
2. Identify the applications or resources in scope.
3. Decide which conditions matter for the scenario.
4. Choose the access decision you want to enforce.
5. Test the policy before broad rollout.

**Common policy signals**

Conditional Access policies can use multiple signals to scope decisions. These signals let you tailor access to the situation instead of applying one rule everywhere.

- **User or group membership** helps target specific users, guest users, or roles.
- **Application** scope can cover one, multiple, or all applications.
- **Conditions** can include risk, platform, location, client apps, and devices.
- **Decisions** can block access or grant access with requirements.

**Practical lab baseline**

A practical baseline starts simple and protects the most important access paths first. The goal is to create a secure default that you can expand without losing control of the environment.

- Start with a minimized baseline rather than building every policy from scratch.
- Use broad protection to cover most access paths first.
- Expand or modify the baseline only after you confirm the effect on users and resources.
- Keep the policy set understandable so you can troubleshoot changes quickly.

**Zero Trust behavior in the lab**

Zero Trust changes how you think about access inside the lab. Instead of trusting the network boundary, you verify each request and apply controls based on context.

- Each access attempt should be authenticated or otherwise proven.
- Security checks apply to every device, user, and process.
- MFA is a common control when access needs stronger assurance.
- Continuous monitoring supports ongoing risk management.

**Key takeaways**

The lab prerequisite is not just software access; it is a policy mindset. You prepare the environment by assuming no default trust, then you build a baseline that uses identity and context to control access.

- Verify access instead of assuming trust.
- Use Conditional Access as the policy foundation.
- Start with a baseline, then refine it carefully.
- Test changes before you rely on them in a broader environment.
