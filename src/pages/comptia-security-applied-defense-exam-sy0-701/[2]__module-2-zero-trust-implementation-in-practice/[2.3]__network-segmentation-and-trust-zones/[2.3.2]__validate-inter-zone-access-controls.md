## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**What inter-zone validation means**

Validating inter-zone access controls means checking that traffic between network zones, such as VLANs or secured segments, follows the intended policy. You confirm that allowed flows work and blocked flows stay blocked. This is a practical way to catch rule order mistakes, missing denies, and unexpected exposure before they become incidents.

Zero Trust Network Access (ZTNA) reinforces this mindset by removing implicit trust. Instead of assuming that traffic is safe once it crosses a perimeter, you verify identity, device, and context before access is granted. In practice, that means you test whether segmentation and authorization actually match the policy you designed.

- Verify that each zone only reaches the resources it is supposed to reach.
- Confirm that denied traffic is denied for the right reason.
- Check that rule placement matches the intended chain or evaluation order.
- Treat internal movement as a security control, not just a routing problem.

**How to test the controls**

A practical validation process combines rule review, traffic testing, and log inspection. The evidence emphasizes that administrators often start by drawing the network and walking the rules by hand, then confirm behavior with real traffic. This helps you compare the written policy with the actual firewall behavior.

1. Map the zones and expected flows.
   - List which VLANs, subnets, or segments should communicate.
   - Mark which destinations should remain unreachable.
2. Review the rules in order.
   - Check allow and deny entries carefully.
   - Look for default deny behavior and any internal rules that may affect results.
3. Generate test traffic.
   - Try permitted connections and blocked connections.
   - Include common checks such as ping or application access when they are relevant to the policy.
4. Inspect the logs.
   - Use firewall logs to confirm which rule matched.
   - Look for default deny entries and other logged internal rules.
5. Adjust and retest.
   - Fix rule placement or scope if behavior does not match the design.
   - Repeat the test until the observed result matches the expected result.

**Using firewall logs as evidence**

Firewall logs are one of the clearest ways to validate inter-zone access controls because they show rule hits and blocked attempts. The pfSense documentation notes that the firewall creates log entries for rules configured to log, as well as for internal rules such as default deny rules. That makes logs useful for proving both successful enforcement and intentional blocking.

The GUI is the easiest way to view these entries, but the key point is not the interface. It is whether the log shows the expected rule action for the traffic you tested. If your organization needs a longer retention period, send logs to a syslog server so you keep a permanent record beyond the local log limit.

- Use logs to confirm the exact rule that handled the packet.
- Look for default deny entries when traffic should not pass.
- Retain logs externally if you need longer-term validation evidence.
- Adjust logging preferences if you need more or less detail from internal rules.

> Firewall logs are only useful if you can correlate them with the traffic you generated. Test first, then verify the matching log entry.

**What to look for in a zero trust design**

ZTNA and broader zero trust designs focus on specific, verified access rather than broad network trust. For exam purposes, the important idea is that access should be limited to specific applications or resources based on identity and context. That reduces lateral movement and shrinks the attack surface.

When you validate inter-zone controls in a zero trust environment, you are checking that segmentation is real, not just documented. Secured zones, identity-based controls, and context-aware authorization should all align with the intended policy.

- **Secured zones** should isolate sensitive resources from unnecessary access.
- **Identity-based controls** should require verification before access.
- **Context policies** should influence whether access is granted.
- **Threat scope reduction** should limit how far a compromise can move.

**Common validation mistakes**

A rule can look correct on paper and still fail in practice. The most common issues are rule order, overly broad scope, and missing logging. If a packet is allowed or blocked unexpectedly, the first question is usually whether the rule chain evaluated the traffic the way you assumed.

- A broad allow rule may override a more specific deny rule.
- A default deny may block traffic you expected to permit.
- Missing logs can make troubleshooting slow and ambiguous.
- Testing only one direction can miss asymmetric access problems.

**Exam focus**

For Security+ SY0-701, focus on the relationship between policy, enforcement, and verification. You should be able to recognize that inter-zone access controls are validated by testing actual traffic, checking firewall logs, and confirming that segmentation supports zero trust principles.

- Validate both allowed and denied paths.
- Use logs to prove enforcement.
- Confirm that access is limited to the intended zone or application.
- Prefer practical verification over assumption.
