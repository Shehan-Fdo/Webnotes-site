## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Mixed trust access control**

A mixed trust environment combines users, devices, and resources that do not share the same level of confidence. In this setting, you do not rely on network location alone. Instead, you use identity-driven controls to decide whether access should be allowed, limited, or blocked.

Zero Trust is the core model behind this approach. It assumes breach and verifies every request. That means you treat each access attempt as untrusted until policy confirms otherwise.

- **Identity** is the primary signal for access decisions.
- **Device** posture can influence whether a request is allowed.
- **Policy** determines the required action before access is granted.
- **Least privilege** reduces exposure by limiting access to only what is needed.

**Conditional Access as the policy engine**

Conditional Access uses if-then logic to enforce access rules. If a user wants to access a resource, then they must complete the required action. A common example is requiring multifactor authentication (MFA) before access to an application or service such as Microsoft 365.

This model is practical in mixed trust environments because it lets you apply different controls based on context. You can require stronger verification for sensitive resources while allowing lower-friction access for lower-risk requests.

- **Signals** can include user identity and device-related context.
- **Actions** can include MFA or other policy requirements.
- **Resources** can be protected differently based on sensitivity.
- **Enforcement** happens before access is granted.

**Zero Trust principles to apply**

Zero Trust guidance centers on three core principles: verify explicitly, use least privilege access, and assume breach. These principles help you design access controls that work even when the environment includes both trusted and untrusted elements.

Verify explicitly means you base decisions on available signals rather than assumptions. Use least privilege access to reduce the blast radius of compromise. Assume breach so you design controls that still protect data and systems if an attacker gets a foothold.

- **Verify explicitly** with identity and context.
- **Least privilege** limits what each user or device can reach.
- **Assume breach** keeps you focused on containment and resilience.
- **Continuous verification** supports ongoing trust decisions.

**Practical access restrictions**

In a mixed trust environment, you often combine authentication, device checks, and policy enforcement. This helps you protect sensitive assets while still enabling collaboration.

A useful pattern is to identify the most sensitive assets first, then map how traffic and access should reach them. From there, you apply controls that match the risk of each path and resource.

- Protect sensitive assets first.
- Map access paths before defining policy.
- Apply stronger controls to higher-risk resources.
- Use segmentation and access control to reduce lateral movement.

**Implementation focus**

When you design or evaluate a solution, look for controls that verify users and devices every time they request access. This is especially important where remote work, cloud access, and hybrid environments increase the number of entry points.

You should also remember that device identity matters. If device identity is weak or missing, Zero Trust enforcement becomes harder and unauthorized access risk increases.

1. Identify the resource you want to protect.
2. Define the identity and device signals you will trust.
3. Set the required action, such as MFA, for the access request.
4. Enforce the policy before access is granted.
5. Review whether the control still supports collaboration without overexposing data.

**Exam takeaway**

For Security+ scenarios, the key idea is that access is not granted because a user is “inside” the network. Access is granted because the request satisfies policy. Conditional Access and Zero Trust work together to make that decision based on identity, device, and risk.

> If a scenario describes a mixed trust environment, favor controls that verify each request, enforce least privilege, and require stronger authentication for sensitive access.
