## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Blocked traffic troubleshooting workflow**

When traffic does not reach its destination, start by proving whether the firewall is actually dropping it. In pfSense, the firewall logs are the first place to check because dropped traffic is logged by default, while passed traffic is not. If you do not see a matching red log entry for the traffic, the firewall is less likely to be the cause.

The state table gives you the next verification point. If you attempt the connection and immediately see a state entry for the source or destination, the firewall has passed the traffic. This is especially useful when a pass rule is expected to work but the connection still fails elsewhere.

- Check `Status > System Logs` on the `Firewall` tab first.
- Look for red log entries that match the source, destination, and port.
- If no matching block appears, widen your search beyond the firewall.
- Check `Diagnostics > States` right after reproducing the issue.
- A state entry indicates the firewall accepted the traffic.

> If a rule does not log blocked traffic, absence of a log entry does not prove the traffic was allowed.

**Firewall rule behavior and rule-order issues**

A firewall rule can appear correct and still fail because of rule order or because another rule matches first. In practical troubleshooting, you verify the observed behavior before changing the rule set. If a rule blocks too much or allows too much, test the change in a non-production environment before enabling it in production.

This matters because firewall policy changes can have broad impact. A `deny any` rule placed at the bottom of an ACL (access control list) can still cause unexpected outages if the surrounding policy is not validated first. Conservative testing reduces the chance of making servers unreachable.

- Confirm the rule matches the intended traffic.
- Check whether another rule takes precedence.
- Test policy changes in a non-production environment first.
- Treat broad deny rules as high-risk changes.

**NAT and firewall processing order**

Network Address Translation (NAT) and firewall rules are not evaluated in a single flat list. pfSense processes traffic in a specific order, and that order affects troubleshooting. The documented logical order includes Ethernet rules, outbound NAT rules, inbound NAT rules such as port forwards, dynamically received rules for IPsec and OpenVPN clients, internal automatic rules, and then user-defined rules.

Understanding this order helps you avoid false assumptions. A port forward may exist, but the traffic can still fail if a later rule blocks it or if the packet never reaches the expected processing stage. For more complex setups, floating rules and manual outbound NAT can cause traffic to hit multiple layers in both directions.

- Remember that NAT and firewall processing are separate stages.
- Check port forwards as part of inbound NAT troubleshooting.
- Consider floating rules and manual outbound NAT in complex designs.
- Verify the packet path before assuming the rule is wrong.

**Connectivity checks for outbound failures**

For outbound connectivity problems, start with the WAN interface. On static WANs, confirm the WAN IP address, subnet mask, and gateway. An incorrect address can prevent the ISP from delivering traffic, and an incorrect subnet mask can break access to large portions of the Internet or prevent the firewall from reaching its gateway.

These checks are practical because they isolate basic routing and addressing issues before you spend time on higher-layer troubleshooting. If the WAN configuration is wrong, firewall rules may look fine while connectivity still fails.

- Verify the WAN IP address on `Interfaces > WAN`.
- Confirm the subnet mask is correct for static WANs.
- Check that the gateway exists and uses the correct IP address.
- Re-test connectivity after each correction.

**Troubleshooting approach for misconfigurations**

A reliable troubleshooting sequence is to verify logs, confirm state table behavior, and then inspect rule order and NAT processing. This keeps you focused on evidence instead of assumptions. If the firewall is not logging a block and the state table shows a pass, the issue is likely downstream from the firewall.

Use this approach when you suspect a misconfiguration:
1. Reproduce the traffic problem.
2. Check the firewall logs for a matching block.
3. Check the state table for a matching session.
4. Review NAT and firewall rule order.
5. Validate WAN addressing and gateway settings if the issue is outbound.

This sequence helps you separate blocked traffic from routing, NAT, and configuration errors.
