## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Interface connectivity and routing basics**

To validate connectivity, you check both the physical link and the logical path. A firewall or router can be up as a device while one interface is down, misaddressed, or sending traffic to the wrong gateway. In practice, you confirm the interface status first, then test reachability and route selection.

On pfSense, the interface status page shows the assigned interfaces and their current state. The status can indicate whether an interface is operating normally, associated to a wireless access point, disconnected, or lacking carrier. That makes it a fast first stop when traffic does not behave as expected.

- **Up** means the interface has a link and is operating normally.
- **Associated** applies to wireless interfaces that are connected to an access point.
- **Down** can indicate a dynamic WAN is not connected or has no IP address.
- **No Carrier** usually points to a cabling problem or a device on the other end not responding.

**Physical checks before routing checks**

If the GUI is not reachable from the LAN, start with cabling and link light. A bad cable, a short hand-made cable, or an older direct-connect setup may prevent a usable link. Once both ends show link, verify the client TCP/IP configuration.

If the firewall provides DHCP (Dynamic Host Configuration Protocol), the client should also use DHCP. If DHCP is not in use, assign the client an address in the same subnet as the firewall LAN IP, use the correct subnet mask, and set the firewall LAN IP as the gateway and DNS (Domain Name System) server.

> If the link is not up, routing tests are not meaningful yet. Fix the physical or addressing issue first.

**Using ping and traceroute to validate paths**

`ping` confirms basic reachability, while `traceroute` helps you see the path traffic takes. Together, they help you determine whether traffic is leaving through the expected interface or being redirected to another gateway.

The evidence shows a useful troubleshooting pattern:
- A host may ping and traceroute successfully from one interface but fail from another.
- One LAN may route through a VPN gateway while another LAN routes directly to WAN.
- A traceroute from a device can confirm whether traffic to a specific domain bypasses or uses the VPN.
- A traceroute from the firewall itself may still follow the firewall’s own routing policy, which can differ from the client’s path.

**What to verify when routes look wrong**

When connectivity is inconsistent, compare the interface, gateway, and policy path for each network segment. The same destination can behave differently depending on which interface originates the traffic.

- Check whether the interface is assigned the expected gateway.
- Confirm whether traffic is being policy-routed through a VPN or sent directly to WAN.
- Compare results from the firewall and from a client on the affected LAN.
- Use traceroute to identify the first hop and see where the path diverges.

**Practical troubleshooting flow**

1. Check the interface status page for link state and assignment.
2. Verify cabling and link lights on both the client and firewall.
3. Confirm the client IP settings match the firewall’s addressing model.
4. Test reachability with `ping`.
5. Use `traceroute` to confirm the actual route and first hop.
6. Compare results across interfaces if one network behaves differently from another.

**Key takeaways**

Interface validation is not just about whether the device is powered on. You need to confirm link, addressing, gateway selection, and the actual path traffic takes. `ping` tells you whether a destination responds, while `traceroute` shows whether the traffic follows the expected route.
