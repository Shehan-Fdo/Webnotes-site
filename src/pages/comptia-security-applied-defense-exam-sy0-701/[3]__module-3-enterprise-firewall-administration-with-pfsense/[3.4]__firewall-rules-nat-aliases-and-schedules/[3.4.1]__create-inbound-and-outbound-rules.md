## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**Firewall rule fundamentals**

A firewall sits inline and decides whether traffic is allowed or disallowed. It can make decisions based on port numbers, applications, or both, depending on the firewall type. In practice, you use inbound and outbound rules to control traffic at the network edge, where the outside network meets the inside network.

- **Inbound** rules control traffic entering a network or interface.
- **Outbound** rules control traffic leaving a network or interface.
- **Next-generation firewall** features may inspect applications, not just ports.
- Firewalls can also serve as VPN endpoints or routers in some deployments.

**Rule strategy and default behavior**

A default deny strategy is the best practice for firewall rule design. You permit only the minimum traffic required and let everything else drop. This reduces the number of deny rules you need and keeps the ruleset easier to manage.

In a common two-interface setup, the WAN side uses default deny and the LAN side uses default allow. That means Internet traffic entering from the WAN is denied by default, while traffic leaving from the LAN is permitted by default.

- Use **default deny** to minimize exposure.
- Keep deny rules limited to specific exceptions or special cases.
- Expect different default behavior on different interfaces.
- Treat rule order as important when multiple rules could match the same traffic.

> If a rule appears to be ignored, check whether another rule earlier in the ruleset matches first.

**Creating rules with protocol awareness**

When you create rules, match the rule to the traffic you actually need to support. Some services use multiple related protocols, so a single application may require several ports or protocol types.

For example, voice over IP (VoIP) traffic may involve Real-time Transport Protocol (RTP), Session Initiation Protocol (SIP), H.323, and Secure Real-time Transport Protocol (SRTP). That means the rule set must account for the full communication flow, not just one port.

- **VoIP** commonly uses RTP, SIP, H.323, and SRTP.
- Build rules around the service requirements, not just a single protocol name.
- Use specific destination and port criteria when you want tighter control.
- Allow only the traffic needed for the application to function.

**Logging and verification**

Firewall logs are the first place to check when traffic is blocked unexpectedly. pfSense logs dropped traffic by default and does not log passed traffic unless you enable logging on specific pass or block rules. The firewall log also records default deny and other internal rules when logging is enabled for them.

Use logs to confirm whether the firewall is actually dropping the traffic. If you do not see a red log entry that matches the traffic, the firewall is probably not blocking it. You can also check the state table immediately after attempting a connection to see whether a state exists.

1. Open the firewall logs at `Status > System Logs` on the Firewall tab.
2. Look for a red entry that matches the source, destination, or service.
3. Attempt the connection again and check `Diagnostics > States`.
4. Filter by source or destination to see whether a state was created.
5. If a state exists, the firewall passed the traffic.

- **State table** entries indicate passed traffic.
- Logging behavior for default deny and internal rules can be adjusted in system log settings.
- For long-term retention, send logs to a remote syslog server.

**Troubleshooting blocked or unexpected traffic**

If traffic is not behaving as expected, start with the logs and then confirm the state table. This approach helps you distinguish between a real block and a rule-order or logging issue. If the firewall is not logging a match, the traffic may be passing through another rule or not reaching the firewall at all.

- Check logs before changing rules.
- Verify whether the traffic is blocked, passed, or simply unlogged.
- Use rule order to your advantage when narrowing access.
- Remember that a fail-open design can allow traffic to bypass inspection during a failure, which increases risk.

**Key takeaways**

Inbound and outbound rules work best when you combine clear direction, minimal access, and careful verification. Use default deny as your baseline, create rules that match the actual service requirements, and confirm behavior with logs and the state table.
