## **Course:** CompTIA Security+ Applied Defense (Exam SY0-701)

**NAT and port forwarding overview**

Network Address Translation (NAT) changes packet addressing as traffic crosses a boundary. Inbound NAT, often called **port forwarding**, rewrites the destination of incoming packets and sends them to an internal host. You may also see this described as **Destination NAT (DNAT)** or **Inbound NAT**.

Port forwarding is useful when you need outside clients to reach a service on a private network. Common examples include hosting a web server, exposing remote access services, or supporting applications that require inbound connections from the Internet.

- Port forwarding can target a specific port, a port range, or even a protocol.
- Common protocols include **TCP**, **UDP**, and **SCTP**.
- Some products also allow forwarding of entire protocols such as **GRE** or **ESP**.

**How port forwarding works**

A port forward rule matches traffic arriving on an external interface, rewrites the destination, and forwards the packet to a new internal destination. In pfSense, you configure these rules under `Firewall > NAT > Port Forward`.

When you create a port forward in pfSense, the GUI can also create the associated firewall allow rule. That matters because NAT alone does not guarantee the traffic is permitted through the firewall policy.

- The external address is typically the WAN address.
- The internal target is the server or device on the private network.
- The redirect target port can match the original port or map to a different one.

**Common configuration pattern**

A typical web server forward sends inbound HTTP traffic on port `80` to an internal server such as `192.168.1.100` on port `80`.

```text
Interface: WAN
Protocol: TCP
Destination: WAN address
Destination port range: 80
Redirect target IP: 192.168.1.100
Redirect target port: 80
Filter rule association: Add associated filter rule
```

Other common mappings follow the same pattern:

| Service | External Port | Internal IP | Internal Port |
|---|---:|---|---:|
| HTTP | 80 | `192.168.1.100` | 80 |
| HTTPS | 443 | `192.168.1.100` | 443 |
| SSH | 2222 | `192.168.1.200` | 22 |
| RDP | 3389 | `192.168.1.50` | 3389 |
| Minecraft | 25565 | `192.168.1.30` | 25565 |

**Troubleshooting port forwards**

If a port forward does not work, verify the rule carefully before changing multiple variables. A clean reconfiguration is often faster than patching a broken setup.

1. Recheck the guide and remove incorrect entries if the rule set drifted from the intended design.
2. Test from outside the network, not from a local client on the same LAN.
3. Enable logging on the firewall rule that passes the NAT traffic.
4. Review `Status > System Logs` on the Firewall tab to see whether traffic is permitted or denied.
5. Check the states table under `Diagnostics > States` and filter by source, destination, or port.

> Port forwards on external interfaces do not work from local clients without **NAT reflection**. Test from an external network, a VPN in another region, or a cellular connection.

**Key exam takeaways**

- **Port forwarding** is inbound NAT that exposes an internal service to external clients.
- pfSense uses `Firewall > NAT > Port Forward` for rule creation.
- The GUI can create the matching firewall rule automatically.
- Troubleshooting starts with rule accuracy, external testing, logging, and state inspection.
- NAT reflection affects local testing, so internal tests can give misleading results.

**Quick mental model**

Think of port forwarding as a destination rewrite plus a firewall decision. The NAT rule sends traffic to the right internal host, and the firewall rule decides whether that traffic is allowed to pass.
