## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202) — V15

**Performance problems overview**

Performance problems often appear when a system is slow even though no single resource looks fully saturated. That can happen when work is waiting on another resource, a lock, or a shared component rather than simply using all available CPU or memory. In practice, you look for bottlenecks across compute, storage, network, and synchronization points.

- **Resource contention** happens when multiple processes compete for limited resources.
- A system can feel slow even when CPU, RAM, disk, and GPU usage do not look unusually high.
- Slowdowns may come from timeouts, hung services, or other waiting states rather than raw utilization alone.

**Common contention points**

Resource contention can affect a single PC, a server, or a virtualized host. The same basic idea applies in each case: if several workloads need the same resource at the same time, one or more of them must wait.

- **CPU** contention occurs when too many tasks need processing time at once.
- **Memory** contention appears when workloads compete for available RAM.
- **Disk I/O** contention slows reads and writes when storage becomes a bottleneck.
- **Network** contention limits throughput when bandwidth is shared.
- **Locks** can block progress when one process holds a resource another process needs.
- **Connection pools** can also become a shared limit in multi-process systems.

> When a system is slow but utilization looks low, do not assume the hardware is fine; check for waiting, contention, and hidden bottlenecks.

**Virtualization and shared resources**

In virtualized environments, multiple virtual machines (VMs) share the same physical host resources. If allocation is not balanced, one VM can consume enough CPU, memory, storage, or network bandwidth to slow the others.

- A busy VM can reduce the processing power available to neighboring VMs.
- Shared storage or network paths can create bottlenecks even when the host itself seems capable.
- Poor performance may show up as delays across several VMs, not just one guest.

**Lock contention and thread states**

Lock contention is a specific form of resource contention where threads wait because another thread holds a lock. Windows Performance Analyzer (WPA) can help review this kind of issue using data collected with Windows Performance Recorder (WPR). In this context, thread state vocabulary matters because it helps you interpret what the system is doing.

- **Schedule** a thread means make it the next thread to run on the CPU.
- **Preempt** a thread means remove it from the processor before it finishes or yields.
- **Ready** means the thread can run and is waiting for CPU time.
- **Deferred** means the thread is waiting to run on a specific processor but has not been scheduled yet.
- **Standby** means the thread has been selected to run next on a specific processor.

**Troubleshooting approach**

Use a structured approach when performance is poor. Start by identifying which resource is constrained, then determine whether the issue is local, shared, or caused by waiting on another process.

1. Check whether the slowdown affects one app, one VM, or the whole system.
2. Review CPU, memory, disk, and network activity for obvious bottlenecks.
3. Look for signs of waiting, such as timeouts, hangs, or blocked threads.
4. In virtualized environments, compare resource allocation across VMs and the host.
5. If lock contention is suspected, use tools such as WPA and WPR to inspect thread behavior.

**Practical takeaway**

Low utilization does not always mean good performance. A system can still be slow if it is waiting on locks, shared storage, network bandwidth, or another workload that is consuming the needed resource first. For A+ troubleshooting, focus on the relationship between symptoms and the resource that is actually being contended for.
