## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202) — V15

**Troubleshooting approach**

Use a structured troubleshooting methodology so you do not skip basic checks. A practical sequence is to identify the problem, form a theory of probable cause, test that theory, plan and implement a fix, then verify full functionality. This keeps you focused on evidence instead of guessing.

- **Identify** the symptom first: no connectivity, intermittent connectivity, or a DNS-related failure.
- **Test** the simplest likely cause before changing settings.
- **Verify** the connection again after each change so you know what actually fixed it.

**Check the physical layer first**

If you are on a wired connection, start with the cable and the link light. A lit link indicator usually means the device can communicate with the other end of the cable. If there is no light, suspect a bad cable, loose connection, or a device on the other end that is not active.

> If the physical connection is not stable, higher-level troubleshooting is usually wasted effort until you fix it.

- Confirm the Ethernet cable is fully seated.
- Look for a **link light** on the network port or switch.
- Replace the cable if the link light does not come on.

**Use ping to narrow the fault**

After the physical link looks good, test the local protocol stack and addressing. Start with the loopback address `127.0.0.1` to confirm the operating system’s network stack is responding. Then ping the local IP address to check whether the adapter and local configuration are working.

```bash
ping 127.0.0.1
ipconfig
ping <local-ip-address>
```

- `127.0.0.1` tests the local TCP/IP stack.
- `ipconfig` helps you find the current IP address on Windows.
- A successful ping to the local IP suggests the adapter and local configuration are functioning.

**Move outward in layers**

If local tests pass, continue outward toward the network and internet. This layered approach helps you isolate where communication stops. A failure at one stage tells you the problem is likely at that layer or just beyond it.

- Test the local device first.
- Test the local network next.
- Test internet reachability after local connectivity is confirmed.

**Recognize DNS problems**

A **DNS** (Domain Name System) issue can look like an internet outage even when the connection is otherwise working. DNS translates human-friendly names into IP addresses, so if name resolution fails, web pages may not load even though the network link is up.

- A DNS error can come from local settings, the router, or the provider’s DNS server.
- A system may connect briefly and then stop resolving names correctly.
- If DNS is the issue, trying an alternate DNS server can help isolate the cause.

**Practical troubleshooting clues**

Some symptoms point to the desktop or local configuration rather than the modem or ISP. For example, if another device works on the same cable and modem, the problem is more likely on the original computer. Intermittent failures after a restart can also suggest a local configuration issue rather than a complete service outage.

- Compare the failing device with a known-good device on the same connection.
- Check whether the issue follows the computer or stays with the network hardware.
- Treat repeated DNS failures as a clue to inspect name resolution settings.

**What to remember for the exam**

The A+ troubleshooting pattern is to start simple, verify each layer, and avoid changing multiple things at once. For internet connectivity issues, that usually means checking the cable and link light, testing `127.0.0.1`, checking the local IP, and then considering DNS if the connection exists but websites still fail to load.
