## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202) — V15

**Overview**

Network troubleshooting starts with simple checks that tell you whether the problem is local, upstream, or related to name resolution. In A+ Core 1, you use common tools and commands to verify connectivity, inspect addressing, and narrow down where communication fails.

- **Link light** is a fast physical check for wired connections.
- **Loopback testing** helps confirm the local TCP/IP stack is working.
- **IP configuration** commands show addressing details such as the IPv4 address, subnet mask, and default gateway.
- **Connectivity** tools help you test whether a host can reach another device or service.
- **Routing** and name-resolution commands help you isolate path and DNS issues.

**Core troubleshooting flow**

Start with the simplest evidence and move outward. If a wired device has no link light, the issue may be the cable, port, or adapter. If the local stack responds but external hosts do not, the problem may be with addressing, gateway access, routing, or DNS.

1. Check the physical connection and confirm the link light.
2. Test the local machine with the loopback address `127.0.0.1`.
3. Review the current IP settings with `ipconfig`.
4. Test reachability with `ping`.
5. Trace the path with `tracert` if traffic leaves the local network but still fails.
6. Verify name resolution with `nslookup` when a hostname does not resolve correctly.

**Common Windows commands**

These commands are the most useful starting points for Windows network troubleshooting. They help you inspect configuration, confirm connectivity, and identify where a failure occurs.

| Command | Purpose | Practical use |
|---|---|---|
| `ipconfig` | Displays IP configuration details | Check the IPv4 address, subnet mask, and default gateway |
| `ipconfig /all` | Shows detailed interface information | Review full adapter settings and addressing details |
| `ping` | Tests connectivity to a host | Confirm whether a device responds to ICMP echo requests |
| `tracert` | Traces the route to a destination | Identify where traffic stops along the path |
| `nslookup` | Queries DNS | Check whether a name resolves to the expected address |
| `netsh` | Displays or modifies network configuration | Inspect or adjust interface-related settings |
| `route print` | Displays the routing table | Review how the system chooses network paths |

**Command examples**

Use these commands from Command Prompt to gather quick evidence during troubleshooting.

```cmd
ipconfig /all
ping 127.0.0.1
ping 8.8.8.8
tracert example.com
nslookup example.com
route print
```

`ipconfig /all` is especially useful when you need more than a basic address summary. It gives you a fuller view of the adapter configuration, which helps when a device has an address but still cannot reach the network.

**What each tool tells you**

- `ping` confirms whether a host responds at the network layer.
- `tracert` shows the hop-by-hop path and helps you spot where traffic stops.
- `nslookup` checks whether DNS can translate a name into an IP address.
- `route print` shows the current routing table so you can see how traffic is directed.
- `netsh` supports deeper inspection or modification of network settings.
- `ipconfig` helps you verify whether the device has a valid address, mask, and gateway.

**Using `ipconfig` effectively**

`ipconfig` is one of the first commands to run because it quickly shows whether the device has a usable address configuration. If the address, subnet mask, or default gateway is missing or incorrect, connectivity problems often follow.

- Use `ipconfig` for a quick summary.
- Use `ipconfig /all` when you need detailed adapter information.
- Compare the default gateway with the expected network path.
- Check whether the IPv4 address matches the network you expect to be on.

**Using `ping` and `tracert`**

`ping` helps you confirm basic reachability, while `tracert` helps you see where a path breaks. A successful `ping` to the loopback address shows that the local TCP/IP stack is functioning. A failed `ping` to an external host suggests a problem beyond the local machine.

> If `ping 127.0.0.1` fails, focus on the local TCP/IP stack before troubleshooting the network.

**Name resolution and routing**

A device can have working connectivity but still fail to reach a site by name if DNS is not resolving correctly. In that case, `nslookup` helps you verify the lookup process. If traffic is going to the wrong place or not leaving the host as expected, `route print` helps you inspect the routing table.

- Use `nslookup` when a hostname fails but an IP address may still work.
- Use `route print` when you need to confirm the system’s route selection.
- Use `netsh` when you need a command-line tool for viewing or changing network configuration.

**Practical interpretation**

These tools work best when you use them in sequence instead of randomly. First confirm the local system, then the local network, then the wider path, and finally DNS if the issue involves names rather than addresses. This approach reduces guesswork and helps you isolate the fault faster.

**Key takeaways**

- Start with physical indicators such as the link light.
- Use `ping` to test reachability and `tracert` to follow the path.
- Use `ipconfig` to verify addressing and gateway information.
- Use `nslookup` to check DNS name resolution.
- Use `route print` and `netsh` when routing or configuration details need closer inspection.
