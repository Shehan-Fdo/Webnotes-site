## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202) — V15

**Identity and access management basics**

Identity and access management (IAM) ensures the right people, machines, and software components access the right resources at the right time. The process starts with proving identity, then moves to deciding whether access is allowed.

IAM is a foundation for securing shared resources because it separates *who you are* from *what you can do*. In practice, that means you authenticate first and authorize second.

- **Identity management** creates, stores, and manages identity information.
- **Identity provider (IdP)** software tracks and manages identities.
- **Authentication** verifies that an entity is who it claims to be.
- **Authorization** sets permissions for what an entity can access or do.

**Authentication versus authorization**

Authentication answers the question, “Who are you?” Authorization answers, “What are you allowed to do?” These are related but distinct controls, and confusing them leads to weak access design.

Authentication can use credentials such as usernames and passwords, or other methods like cards, biometrics, and certificates. Authorization then applies rules that determine access to resources, functionality, or data.

| Concept | Purpose | Typical outcome |
|---|---|---|
| Authentication | Proves identity | Access request is trusted or rejected |
| Authorization | Grants permissions | Access is allowed or denied for specific resources |

> Authentication does not determine what files or tasks a user can access. That decision belongs to authorization.

**Shared resource access**

Shared resources are commonly protected by identity-based access so that permissions follow the user or service account rather than relying only on a device or network location. This approach is used to control access more precisely and reduce unnecessary exposure.

Azure Files is one example of identity-based access for shared storage. It supports identity-based authentication over Server Message Block (<abbr title="Server Message Block">SMB</abbr>) for Windows, Linux, and macOS clients. Permissions can be assigned at the share, directory, or file level.

- Identity-based access can be used on-premises or in Azure.
- Permissions can be applied at multiple levels of a file hierarchy.
- Azure Files does not currently support identity-based authentication for Network File System (<abbr title="Network File System">NFS</abbr>) file shares.
- Enabling identity-based authentication on a storage account does not add an extra service charge.

**Why identity-based access matters**

Identity-based access helps you control shared resources more securely than broad, static access methods. It supports better accountability because access is tied to a known identity, which is important for security and compliance.

This model is especially useful when users change roles, devices, or locations. Access can be adjusted by identity and permission rather than by manually reconfiguring every shared resource.

**Practical access model**

1. You identify the user, device, or service.
2. You authenticate the entity with a credential or trusted proof.
3. You authorize the entity for specific resources or actions.
4. You apply permissions at the appropriate scope, such as share, directory, or file.

**Identity baseline concepts**

In Microsoft 365 environments, identity baseline configuration focuses on users, identity security, groups, and identity-related tenant settings. These configurations are part of the core identity services you set up after creating a tenant and configuring domains.

The baseline approach is available to verified Education tenants and is included within the A1 license entitlement. It also depends on the right administrative roles being in place.

- **Identity Admin** manages identity-related settings.
- **IT Admin** handles broader tenant administration.
- **EXO Admin** manages Exchange Online-related tasks.
- **OneDrive Admin** manages OneDrive settings.
- **SharePoint Admin** manages SharePoint settings.

**Key takeaways**

Identity-based access is a practical way to protect shared resources because it combines identity verification with permission control. For A+ purposes, remember the sequence: authenticate first, authorize second.

When you see a shared file service or cloud storage platform, look for where permissions are applied and whether access is tied to an identity, a group, or a service account. That distinction is central to secure resource sharing.
