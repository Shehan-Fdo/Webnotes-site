## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202) — V15

**Backup and restore fundamentals**

Backup and restore means keeping periodic copies of data and applications on a secondary device so you can recover information after loss or damage. This is a core part of disaster recovery because it helps you sustain operations when data is affected by power loss, cyberattack, human error, disaster, or another unplanned event.

A good backup plan is not just about making copies. It also includes deciding how often you back up data, where you store it, and how quickly you need to recover it if something goes wrong.

- **Secondary storage** holds the copied data separate from the original source.
- **Recovery** is the process of returning data or systems to a usable state after loss.
- **Disaster recovery** uses backups as one of its main protections against major disruption.

**Why backups matter**

Backups protect against both accidental and intentional data loss. If the original data is damaged, deleted, or unavailable, a backup gives you a reliable recovery path. This matters for personal devices, business systems, and server environments alike.

In practice, the value of a backup depends on how much data you can afford to lose and how long you can tolerate being without it. That is why backup frequency and storage location are planning decisions, not afterthoughts.

- Backups help you recover from accidental deletion.
- Backups reduce the impact of hardware failure and corruption.
- Backups support continuity after outages and disasters.
- Backups can reduce downtime when restore time matters.

**File system backup and restore**

A file system is the collection of files and file management structures on physical or logical storage. File system backups copy those files so they can be restored later. Oracle Secure Backup documentation notes that file system data can be backed up to tape, showing that backup targets can include removable media as well as other storage types.

Restore operations return the saved files to a usable location. In a practical support workflow, you back up the data first, then restore only the files or folders needed for recovery.

- **File system data** includes files and the structures used to manage them.
- **Restore operations** bring back backed-up files after loss or damage.
- **Tape** is one example of a backup destination used in some environments.

**Windows backup and recovery options**

Windows includes multiple backup and recovery paths, including Windows Backup, File History, OneDrive restore options, and recovery features such as System Restore and Recovery Drive. These tools serve different purposes, so you choose the one that matches the recovery goal.

Windows also provides options for moving files to a new PC and for recovering the operating system when startup or system problems occur. That means backup and restore is not limited to files alone; it can also support broader device recovery.

| Feature | Typical use |
|---|---|
| `Windows Backup` | Back up and restore device data and settings |
| `File History` | Restore previous versions of files |
| `OneDrive restore` | Recover deleted files or folders stored in OneDrive |
| `System Restore` | Roll back system changes affecting Windows |
| `Recovery Drive` | Support recovery when Windows will not start normally |

**Planning a backup strategy**

A backup strategy starts with deciding what data matters most and how much loss is acceptable. IBM describes backup and restore as a practice that depends on the cost of losing access and the cost of replacing or recreating data. That makes planning a business and technical decision at the same time.

You should also think about where the backup lives. Storing copies on a separate device or off-site location improves resilience if the original system is damaged. A dissimilar server or other separate storage target can help reduce the risk that one event affects both the source and the backup.

1. Identify the data and systems you need to protect.
2. Decide how often backups should run.
3. Choose a separate storage target for the copies.
4. Test restore procedures so you know recovery works.
5. Update the plan when data, systems, or risk changes.

**Restore considerations**

A successful backup is only useful if you can restore from it. Restore planning should account for what you need back, how quickly you need it, and whether you are recovering a single file, an application, or an entire system.

Some environments also use technologies that improve availability during backup and restore operations. For example, Microsoft Exchange Server uses Volume Shadow Copy Service (VSS) to help backup and restore applications work with minimal performance impact on active mailbox databases. The key idea is that backup tools should protect data without creating unnecessary disruption.

> A backup that cannot be restored is not a complete recovery solution.

**Key terms to remember**

- **Backup**: a stored copy of data or applications used for recovery.
- **Restore**: the process of returning backed-up data to use.
- **Recovery**: broader return to normal operation after loss or failure.
- **VSS**: Volume Shadow Copy Service, a Windows feature used by some backup applications.
- **File History**: a Windows feature for restoring file versions.

**Exam focus**

For A+ purposes, focus on the relationship between backup, restore, and recovery. You should be able to explain why backups are stored separately, why restore testing matters, and how Windows recovery tools differ from file-level backup tools.

- Backups protect against data loss.
- Restore returns data or systems from a saved copy.
- Recovery tools may address files, settings, or the operating system.
- Planning includes frequency, storage location, and recovery needs.
