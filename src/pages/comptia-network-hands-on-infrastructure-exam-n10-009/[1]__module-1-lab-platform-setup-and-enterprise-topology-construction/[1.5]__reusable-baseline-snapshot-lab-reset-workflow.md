## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Snapshot and baseline workflow**

A reusable baseline snapshot gives you a known-good starting point for lab work. In network labs, snapshots and rollback help you recover after a bad change, such as a wrong routing configuration or an incorrect access control list (ACL). The goal is to preserve a stable topology state so you can test changes without rebuilding everything from scratch.

In GNS3, a snapshot is described as freezing the state of the topology, including nodes, links, interfaces, and disk state. Rollback returns you to that saved state. This makes snapshots useful for repeatable practice, troubleshooting, and fast lab recovery.

- Use a baseline snapshot after you finish a clean lab build.
- Restore the snapshot when a change breaks the lab.
- Keep the baseline simple so you can verify whether a later issue comes from your test change.

**Why this workflow matters**

A reset workflow reduces time spent repairing a damaged lab. Instead of manually undoing every change, you return to a known state and continue testing. That is especially useful when you are practicing configuration changes that can affect reachability or device behavior.

Snapshot-based recovery also supports consistent lab repetition. If you need to practice the same scenario multiple times, a baseline snapshot lets you restart from the same point each time.

- **Baseline** means the clean state you want to return to.
- **Rollback** means restoring that saved state.
- **Repeatability** helps you compare results across attempts.

**Common snapshot and restore issues**

Snapshot restore is not always frictionless. Reported GNS3 issues show that restoring a snapshot can cause the project to exit or the GUI to lock up. One report suggests the problem may involve a remote GNS3 server and path handling between client and server. Another report describes a blank snapshot window and greyed-out menus after closing the snapshot restore dialog.

These reports are reminders to treat snapshot workflows carefully, especially in remote or multi-host setups.

> If the restore process leaves the interface unresponsive, avoid assuming the lab state is valid until you confirm the project is still open and usable.

- Watch for blank restore windows or disabled menus.
- Be cautious when the server runs remotely.
- Verify that the restore action completed before continuing.

**Practical reset workflow**

1. Build the lab to a clean, working state.
2. Save a snapshot that represents your baseline.
3. Make your test changes.
4. If the lab breaks, restore the baseline snapshot.
5. Confirm the topology is back in the expected state before resuming work.

This workflow keeps your practice environment reusable. It also helps you separate the original lab design from the changes you are testing.

**Backup and restore discipline**

Snapshot workflow is only one part of lab recovery. For broader lab protection, EVE-NG guidance emphasizes backing up lab content before reinstalling or moving to a new host. The backup guidance lists folders and data that should be copied for a full content backup, and it warns that restoring to a new EVE instance should be completed before creating the first lab to avoid UID overlap.

That means you should think in layers:

- Use snapshots for quick rollback inside an active lab.
- Use backups for preserving lab content across reinstall or migration.
- Restore content before creating new labs on a fresh system.

**Operational cautions**

EVE-NG rehosting guidance also shows that some lab platforms depend on licensing and internet access during validation. While that is separate from snapshot use, it reinforces a general rule: recovery workflows can depend on platform state, host access, and timing.

For exam prep, focus on the practical habit: save a clean baseline, know how to restore it, and verify the environment after recovery. A good reset workflow saves time and reduces errors when you are iterating through network changes.

**Key takeaways**

- Snapshots preserve a known-good lab state.
- Rollback returns the lab to that state after a failed change.
- Remote or GUI-based restore actions can fail or lock up, so verify recovery.
- Backups protect lab content beyond a single snapshot.
- Restore content before building new labs on a fresh platform when the platform guidance requires it.
