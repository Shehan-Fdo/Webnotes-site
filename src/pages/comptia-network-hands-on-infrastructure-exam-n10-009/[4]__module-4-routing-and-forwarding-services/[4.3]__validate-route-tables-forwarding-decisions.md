## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Routing table fundamentals**

A router forwards traffic by consulting its routing table and choosing the best available path for the destination. The table acts as the device’s map for where packets should go next, and troubleshooting usually starts by checking what the router knows about the network.

Routing tables are not limited to routers. End devices and servers also maintain routing information, but the router’s forwarding decision is the key focus when you validate path selection.

- **Routing table** entries describe known destinations and how to reach them.
- **Next hop** is the next device or address used to continue forwarding.
- **Best path** is selected from the available routes based on the router’s decision process.
- **Metrics** help compare routes when more than one path exists.

**How forwarding decisions are made**

When a packet arrives, the router checks the destination and looks for a matching route. If multiple routes can reach the same network, the router uses its route selection logic to decide which one to install or prefer. In practice, you validate the result by comparing the destination, the route source, and the chosen next hop.

> If the route you expect is missing, the forwarding decision cannot use it, even if the path exists somewhere else in the network.

A default route is a common fallback when no more specific route matches. For example, a route such as `0.0.0.0/0` points traffic toward a general upstream next hop.

```text
ip route 0.0.0.0 0.0.0.0 172.16.30.1
```

**Validate the routing table**

Use the routing table to confirm what the device believes is reachable and which path it will use. In lab-style troubleshooting, `show ip route` is the primary command for viewing route information and checking the selected path.

1. Open the device CLI.
2. Enter privileged EXEC mode if required.
3. Run `show ip route`.
4. Identify the destination network, route source, metric, and next hop.
5. Compare the chosen route with the path you expect.

The goal is not just to see routes listed. You want to confirm that the route with the correct destination and acceptable metric is the one the router actually prefers.

**Best path selection**

Routing decisions depend on the information available in the table. If a routing protocol provides multiple possible paths, the router compares the candidates and chooses the best one according to its selection rules. Metrics are the practical clue you use when comparing routes during study or troubleshooting.

- **Routing protocol** identifies how the route was learned.
- **Metric** indicates the cost or preference used in path comparison.
- **Best path** is the route the router selects for forwarding.
- **Forwarding outcome** should match the selected route in the table.

In packet-tracer-style exercises, you often inspect the table first and then infer the path traffic will take. That process helps you explain why one route wins over another.

**Troubleshooting approach**

If forwarding does not behave as expected, start with the route table before checking deeper protocol details. A missing route, an unexpected default route, or a less-preferred next hop can all change the forwarding decision.

- Confirm the destination is present in the table.
- Check whether a more specific route exists.
- Verify the next hop is reachable.
- Compare metrics when multiple routes exist.
- Look for a default route that may be catching unmatched traffic.

**Key takeaway**

Route validation is about proving that the device has the right route and will forward packets to the right next hop. Once you can read the table and explain the selected path, you can troubleshoot most basic forwarding issues with confidence.
