## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Routing problems at a glance**

Routing issues often show up as unreachable destinations, traffic taking the wrong path, or return traffic arriving on a different interface than the outbound flow. A router uses its routing table to decide the best **next hop**, so any missing route or incorrect route entry can break forwarding.

When you troubleshoot, compare the routing table to the intended network design. That helps you confirm whether the router knows the destination, whether the next hop is valid, and whether traffic is following a symmetric path.

- A **missing route** means the router has no usable entry for the destination prefix.
- An **incorrect next hop** points traffic to the wrong neighbor or an invalid address.
- An **asymmetric path** sends traffic out one path and returns it on another.

**Missing routes**

A missing route prevents the router from forwarding traffic to a specific network because the destination prefix is absent from the routing table. In practical terms, the router may fall back to a default route or drop the packet if no suitable match exists.

Start by verifying that the route is actually missing from the relevant table. In systems with policy routing, the route may exist in a different table than the one you first check.

```bash
ip route show
ip route show 10.20.0.0/24
ip route show table all | grep "10.20.0"
ip route show table local
```

- Check the main routing table first.
- Search for the exact prefix you expect.
- Review all tables if policy routing is in use.
- Check the local table for connected or loopback routes.

> If the route is missing because of a static route problem, confirm that the route was actually installed and that the configuration source added it as expected.

**Incorrect next hops**

An incorrect next hop means the route exists, but the forwarding decision is wrong. Cisco documentation notes that a route can be missing from the routing table when the specified next hop is invalid, so the next-hop address itself is a key troubleshooting point.

You should verify that the next hop is reachable and appropriate for the route type. If the next hop is wrong, traffic may never leave the router correctly, or the route may remain inactive.

- Confirm the next-hop IP address is valid.
- Verify the next hop is reachable from the local router.
- Compare the route entry with the intended topology.
- Check whether the route is inactive because of a next-hop mismatch.

**Asymmetric paths**

Asymmetric routing occurs when the forward and return paths differ. Cisco community material on asymmetric routing highlights this as a real troubleshooting concern, especially when devices expect traffic to follow a consistent path.

This matters because some systems and security controls assume that both directions of a session use the same path. If return traffic arrives from a different interface or device, the session may fail or appear unstable.

- Compare outbound and inbound paths for the same flow.
- Look for devices that may steer return traffic differently.
- Check whether load balancing, failover, or policy routing changes the path.
- Verify that stateful devices can see both directions of the session.

**Troubleshooting workflow**

Use a simple sequence so you do not chase symptoms before confirming the route state.

1. Confirm the destination is actually unreachable.
2. Check the routing table for the expected prefix.
3. Verify the next hop is valid and reachable.
4. Compare the observed path with the intended path.
5. Look for asymmetric forwarding between outbound and return traffic.

**What to compare during analysis**

A routing table gives you the router’s forwarding decision, but the network diagram gives you the intended design. Comparing both helps you spot where the path diverges.

| Item | What you check | Why it matters |
|---|---|---|
| Destination prefix | Is the route present? | Confirms whether the router knows where to send traffic |
| Next hop | Is the address valid and reachable? | Prevents forwarding to an unusable neighbor |
| Interface | Does the route point out the correct interface? | Ensures traffic exits the expected link |
| Return path | Does reply traffic come back the same way? | Helps identify asymmetric routing |

**Practical troubleshooting mindset**

Routing issues are often configuration problems rather than hardware failures. A careful check of the routing table, default gateway behavior, and route validity usually reveals the cause. If the route is present but traffic still fails, focus on the next hop and the path symmetry between devices.
