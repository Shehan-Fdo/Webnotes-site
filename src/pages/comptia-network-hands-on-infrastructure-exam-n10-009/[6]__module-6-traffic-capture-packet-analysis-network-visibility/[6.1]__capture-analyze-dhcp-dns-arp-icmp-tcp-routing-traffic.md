## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Packet capture workflow**

Wireshark is a network protocol analyzer that captures and inspects packets so you can understand traffic flow, isolate issues, and study protocol behavior. It stores captures in `.pcap` or `.pcapng` files and supports filtering to narrow analysis to specific IPs, ports, or protocols.

Capture and analysis usually follow a simple pattern:

1. Start a capture on the correct interface.
2. Generate the traffic you want to observe.
3. Apply display filters to isolate the protocol or conversation.
4. Inspect packet details and follow the exchange across multiple frames.

Capture filters limit what gets recorded, while display filters control what you see after capture. If a filter turns invalid or red, check for syntax mistakes, the wrong filter type, or a protocol that is not present in the capture.

**DHCP traffic**

<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> assigns IP addresses to clients on a network. A common way to recognize DHCP activity is the DORA sequence: Discover, Offer, Request, Acknowledge.

The client begins without a usable address and sends a Discover message as a broadcast. The server responds with an Offer, the client sends a Request, and the server finishes with an Acknowledge.

- Discover starts from `0.0.0.0` and is sent as a broadcast to `255.255.255.255`.
- DHCP uses ports `67` and `68`.
- DORA gives you a quick way to confirm whether address assignment is working.

**DNS traffic**

<abbr title="Domain Name System">DNS</abbr> translates human-readable names into IP addresses. In packet captures, DNS is useful for confirming name resolution and checking whether a client is querying the expected domain.

Useful display filters include:

```wireshark
dns
dns.qry.name == "example.com"
```

- `dns` isolates DNS packets.
- `dns.qry.name` helps you focus on a specific queried name.
- DNS analysis is often the first step when a host can reach an IP address but not a hostname.

**ARP traffic**

<abbr title="Address Resolution Protocol">ARP</abbr> maps an IP address to a local network hardware address. It is especially useful on the local segment because devices need ARP to learn the destination MAC address before sending frames on the LAN.

In captures, ARP often appears when a host is trying to reach a local peer or gateway and does not yet know the target hardware address.

- ARP is local to the broadcast domain.
- Requests are typically broadcast so all devices can see them.
- Replies provide the mapping needed for local delivery.

**ICMP traffic**

<abbr title="Internet Control Message Protocol">ICMP</abbr> supports network diagnostics and error reporting. It is commonly seen in reachability tests and troubleshooting workflows.

When you analyze ICMP, look for request and reply patterns that show whether a destination responds and whether the path behaves as expected.

- ICMP is useful for basic connectivity checks.
- It helps you confirm whether a host is reachable.
- It can also reveal path or error conditions during troubleshooting.

**TCP traffic**

<abbr title="Transmission Control Protocol">TCP</abbr> is connection-oriented, so packet analysis often focuses on the handshake and conversation flow. A common lab task is to inspect the three-way handshake and then filter the packets to view a specific conversation.

The handshake shows how a session begins, and later packets can reveal retransmissions, delays, or application behavior.

```wireshark
tcp
tcp.port == 443
tcp.analysis.retransmission
```

- `tcp` isolates TCP traffic.
- `tcp.port == 443` narrows the view to a specific service port.
- `tcp.analysis.retransmission` helps you spot possible delivery problems.

**Routing traffic**

Routing-related captures help you understand how traffic moves between networks and how devices forward packets toward a destination. Sample capture collections include routing protocol traffic, which makes them useful for studying forwarding behavior and protocol exchanges.

When you inspect routing traffic, focus on the path a packet takes and whether the capture shows traffic crossing network boundaries as expected.

- Routing traffic helps you study packet movement between networks.
- Sample captures can provide protocol-specific examples for analysis.
- Comparing source and destination addresses helps you confirm forwarding behavior.

**Practical analysis tips**

Use filters to reduce noise before you inspect packet details. Start broad, then narrow the view as you identify the protocol or conversation you care about.

- Begin with `ip.addr == X` when you want traffic involving a specific host.
- Use protocol filters such as `dns` or `tcp` to isolate a class of packets.
- Check packet details for addresses, ports, and sequence behavior.
- Compare multiple frames to understand request and response patterns.

**Troubleshooting mindset**

Packet capture is most useful when you connect what you see in the trace to the symptom you are investigating. If a host cannot get an address, start with DHCP. If a name fails, inspect DNS. If local delivery fails, check ARP. If reachability is uncertain, use ICMP. If a session stalls, inspect TCP. If traffic is not taking the expected path, review routing behavior.

> If a filter is invalid, verify that you used the correct syntax for the filter type and that the protocol exists in the capture.
