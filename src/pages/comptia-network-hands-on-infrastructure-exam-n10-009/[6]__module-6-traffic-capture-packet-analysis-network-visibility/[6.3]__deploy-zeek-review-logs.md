## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Zeek overview**

Zeek is a passive, open-source network traffic analyzer that many operators use as a network security monitor (NSM). It sits on a sensor and analyzes traffic in real time without acting as an active defense tool.

Zeek is useful because it produces structured output that supports investigation, troubleshooting, and performance analysis. Its logs are suitable for manual review and for downstream tools such as SIEM platforms.

- Zeek focuses on traffic analysis, not blocking.
- It generates high-fidelity logs from observed network activity.
- It supports both security monitoring and broader network analysis tasks.

**What Zeek logs tell you**

Zeek’s main value is the set of logs it creates. These logs include connection records, application-layer details, DNS requests and replies, and other protocol data. By default, Zeek writes data into tab-separated or JSON log files for post-processing.

The logs help you move from raw packets to usable evidence. Instead of inspecting every packet manually, you review summarized records that show what happened on the network.

- **Connection data** shows observed network sessions.
- **DNS data** shows name resolution activity.
- **Structured output** makes filtering and review easier.

**Deploying Zeek**

Zeek commonly runs in two modes:

1. **Standalone mode** for a single node capturing and analyzing traffic.
2. **Cluster mode** for distributed deployments with manager, proxy, and worker nodes.

For hands-on infrastructure work, standalone mode is the practical starting point because it fits smaller environments and simpler lab setups. Zeek can also be customized through its scripting layer, which reacts to events generated as packets are parsed.

> Zeek is not a signature-based IDS (intrusion detection system) in the same way as tools like Snort or Suricata. Its strength is protocol parsing, metadata extraction, and scriptable analysis.

**Zeek scripting and events**

Zeek’s scripting language is central to how it works. Protocol analyzers generate events as traffic unfolds, and those events feed the scripting engine. That event-driven model drives Zeek’s visible behavior and lets you customize analysis.

This matters during deployment because the logs you review are the result of that analysis pipeline. If you understand that Zeek reacts to events, it is easier to reason about why a log entry appears and how to extend detection later.

- **Events** are generated as protocols are parsed.
- **Scripts** process those events.
- **Custom logic** can change what Zeek records or alerts on.

**Reviewing `conn.log`**

`conn.log` is one of Zeek’s most important logs. It tracks both stateful and stateless traffic, so it is useful for TCP (Transmission Control Protocol) and UDP (User Datagram Protocol) analysis.

When you inspect `conn.log`, focus on the originator and responder fields, protocol details, and the overall session summary. Zeek’s documentation emphasizes that the log is meant for analyst use, not just field-by-field memorization.

- Use `conn.log` to confirm observed connections.
- Review it for both TCP and UDP traffic.
- Treat it as a primary starting point for traffic investigation.

**Reviewing `dns.log`**

`dns.log` is another key source because DNS (Domain Name System) activity often reveals what hosts are trying to reach. Zeek describes `dns.log` as one of its most important data sources.

You can inspect JSON-formatted logs with `jq`, which makes the output easier to read and filter. A typical review workflow is to open the log and inspect query details, response codes, and related metadata.

```bash
jq . -c dns.log
```

A DNS log entry can include fields such as:

- query name
- query type
- response code
- originator and responder addresses
- transport protocol and ports

This makes `dns.log` valuable for both security review and general network troubleshooting.

**Practical review workflow**

1. Start Zeek on the traffic source you want to observe.
2. Confirm that Zeek is generating logs.
3. Open `conn.log` to identify active sessions.
4. Open `dns.log` to review name resolution activity.
5. Use `jq` when the logs are in JSON format and you need compact inspection.

**What to remember for exam prep**

Zeek is best understood as a passive analysis platform that turns traffic into structured logs. Its scripting engine and event model explain how it produces those logs, while `conn.log` and `dns.log` give you immediate visibility into network behavior.

- **Passive monitoring** means Zeek observes traffic without blocking it.
- **Structured logs** support investigation and post-processing.
- **DNS and connection logs** are core review points.
- **Standalone deployment** is the simplest operational mode to start with.
