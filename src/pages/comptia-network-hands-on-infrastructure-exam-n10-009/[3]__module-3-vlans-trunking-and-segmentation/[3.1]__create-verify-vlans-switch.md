## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**VLAN creation and purpose**

A <abbr title="Virtual Local Area Network">VLAN</abbr> separates switch traffic into logical groups so you can segment a network without changing the physical cabling. In practice, you create the VLAN first, then place switch ports into that VLAN so connected devices use the intended broadcast domain.

- Create VLANs in the switch VLAN database before assigning ports.
- Use VLANs to organize traffic into logical segments.
- Keep the configuration goal clear: define the VLAN, then map ports to it.

**Assigning switch ports**

After the VLAN exists, you assign access ports to it. This is the step that makes end devices participate in the VLAN. The source material also distinguishes access ports from trunk ports, which are used when you need VLAN traffic to pass between switches or other network devices.

- Assign an access port to a single VLAN for end-device connectivity.
- Use trunking when multiple VLANs must traverse the link.
- Verify the port mode before troubleshooting VLAN membership.

**Verification commands**

Verification is a core part of VLAN work because a VLAN can exist in the database without appearing where you expect if the port assignment or display context is different. Cisco references `show vlan` as the primary command family for viewing VLAN information, and practical guidance highlights `show vlan brief` as a common summary view.

```bash
show vlan brief
show interfaces trunk
show interfaces switchport
show interfaces status
show spanning-tree
```

- `show vlan brief` gives a compact VLAN summary.
- `show interfaces trunk` helps confirm trunk behavior and VLAN transport.
- `show interfaces switchport` shows port-level switching details.
- `show interfaces status` helps you confirm the physical port state.
- `show spanning-tree` helps you understand how the switch is handling Layer 2 forwarding.

**Troubleshooting VLAN visibility**

If a VLAN does not appear in the output you expect, check whether you are using the right verification command and whether the port is actually assigned to that VLAN. A VLAN may be configured but still not show up in the way you expect if you are looking at the wrong interface context or summary view.

> Be careful not to assume that a missing VLAN in one display means the VLAN was never created. Confirm the VLAN database, the port mode, and the trunk or access configuration before changing anything else.

**Practical workflow**

1. Create the VLAN in the switch VLAN database.
2. Assign the appropriate switchport to the VLAN.
3. Confirm whether the port should be access or trunk.
4. Run `show vlan brief` to verify VLAN presence.
5. Use `show interfaces switchport` or `show interfaces trunk` to confirm port behavior.
6. Check `show interfaces status` and `show spanning-tree` if the result still looks wrong.

**Key takeaways**

VLAN setup is a two-part task: define the VLAN and then place ports into it. Verification matters because the switch can store VLAN information separately from how a port is currently operating. Use the show commands to confirm both the VLAN and the interface behavior before you move on.
