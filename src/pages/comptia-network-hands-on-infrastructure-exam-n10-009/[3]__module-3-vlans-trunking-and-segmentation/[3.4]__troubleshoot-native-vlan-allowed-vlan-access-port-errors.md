## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**VLAN troubleshooting overview**

A <abbr title="Virtual Local Area Network">VLAN</abbr> lets you split a physical switch into separate logical networks. When VLAN settings do not match between connected ports, traffic can fail even though the link is up.

The most common issues in this topic involve:

- **Native VLAN** mismatches on trunk links
- **Allowed VLAN** restrictions that block expected traffic
- **Access port** assignment errors that place a device in the wrong VLAN

A native VLAN mismatch can trigger Cisco warnings such as `%CDP-4-NVLANMISMATCH` or `%CDP-4-NATIVE_VLAN_MISMATCH`. These messages point to a trunk-link configuration problem, not a cabling fault.

**Native VLAN mismatch**

On an <abbr title="Cisco Discovery Protocol">CDP</abbr>-visible trunk, the native VLAN is the VLAN that carries untagged traffic. If the two ends of the trunk use different native VLAN values, the switches detect the mismatch and report it.

- Check both ends of the trunk for the same native VLAN setting.
- Treat the warning as a sign that the trunk configuration needs alignment.
- Verify that the mismatch is not caused by a port that should be an access port instead of a trunk.

> A native VLAN mismatch can still allow the link to come up, but it can break traffic handling and create confusing symptoms.

**Allowed VLAN issues**

A trunk does not always carry every VLAN. The allowed VLAN list defines which VLANs can traverse the trunk. If a VLAN is missing from that list, devices in that VLAN cannot communicate across the trunk even when the physical link is healthy.

- Confirm that the required VLAN is included in the allowed VLAN list.
- Compare both sides of the trunk when troubleshooting asymmetric reachability.
- Use conservative changes: add only the VLANs that should cross the link.

**Access port errors**

An access port belongs to a single VLAN and should connect to an end device, not another switch trunk. If you place a device on the wrong access VLAN, it lands in the wrong broadcast domain and loses expected connectivity.

- Verify that the port mode matches the device type.
- Confirm the access VLAN matches the intended user or device network.
- Recheck the switchport assignment when a host cannot reach peers that should be local.

**Troubleshooting workflow**

1. Identify the symptom.
   - Look for native VLAN mismatch warnings.
   - Note whether only one VLAN is affected or whether the entire link is failing.
2. Check the port role.
   - Decide whether the interface should be an access port or a trunk.
   - Confirm that the configuration matches the physical connection.
3. Compare VLAN settings on both ends.
   - Match the native VLAN on trunk links.
   - Verify the allowed VLAN list.
   - Confirm the access VLAN on host-facing ports.
4. Test connectivity again.
   - Recheck the warning messages.
   - Confirm that traffic now crosses the link as expected.

**Practical troubleshooting mindset**

When a switch port appears to work at Layer 1 but traffic still fails, focus on VLAN membership first. The physical link may be fine while the logical VLAN settings are wrong. That is why native VLAN, allowed VLAN, and access port checks are a core part of switch troubleshooting.

**Key terms**

- **Trunk port**: carries traffic for multiple VLANs.
- **Access port**: carries traffic for one VLAN.
- **Native VLAN**: the VLAN that handles untagged traffic on a trunk.
- **Allowed VLAN list**: the set of VLANs permitted across a trunk.
