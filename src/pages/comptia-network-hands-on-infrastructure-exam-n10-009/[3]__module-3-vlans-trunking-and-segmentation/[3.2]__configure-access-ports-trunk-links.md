## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**VLAN port roles**

A <abbr title="Virtual Local Area Network">VLAN</abbr> separates devices into logical broadcast domains on the same physical switch. When you configure switch ports, you usually choose between an access port for one VLAN and a trunk link for multiple VLANs.

- **Access ports** carry traffic for a single VLAN.
- **Trunk links** carry traffic for multiple VLANs over one physical connection.
- VLANs help you segment traffic without adding more switches.

**Access port behavior**

An access port belongs to one VLAN and is the normal choice for end devices such as workstations, printers, and many phones. The switch treats frames on that port as part of the assigned VLAN.

- Use access mode when the connected device should belong to one VLAN only.
- Keep the configuration simple when the endpoint does not need VLAN tagging.
- Use `switchport mode access` on platforms that support Cisco-style interface commands.

> If a port must carry only one VLAN, do not leave it in a mode that can negotiate trunking.

**Trunk link behavior**

A trunk link transports frames from multiple VLANs between switches or other network devices. Trunks commonly use <abbr title="Institute of Electrical and Electronics Engineers">IEEE</abbr> 802.1Q tagging so the receiving device can identify the VLAN for each frame.

- Trunks let you extend multiple VLANs across a single physical link.
- Tagged frames preserve VLAN identity across the link.
- Trunking is the standard approach for switch-to-switch connectivity in VLAN designs.

**Native VLAN**

The native VLAN is the VLAN associated with untagged traffic on an 802.1Q trunk. In normal trunk operation, frames are tagged, but untagged frames can still appear and must be handled consistently.

- The native VLAN provides a place for untagged frames on the trunk.
- Both ends of the trunk should use the same native VLAN.
- Mismatched native VLAN settings can create confusion and troubleshooting issues.

**Common trunk configuration ideas**

Cisco documentation describes trunking modes and allowed VLAN behavior on trunk ports. In practice, you configure the port as a trunk, then control which VLANs are permitted across it.

- Set the interface to trunk mode when it must carry multiple VLANs.
- Restrict allowed VLANs when you want to limit which VLANs traverse the link.
- Verify that the trunk supports the VLANs you expect to pass.

```bash
interface GigabitEthernet0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 switchport trunk native vlan 99
```

**Allowed VLANs**

A trunk does not always need to carry every VLAN. Limiting the allowed VLAN list reduces unnecessary traffic and helps keep the design intentional.

- Allow only the VLANs that need to cross the trunk.
- Review the allowed list when troubleshooting missing connectivity.
- Keep the native VLAN choice aligned on both ends of the link.

**Practical configuration workflow**

1. Identify whether the port connects to an end device or another switch.
2. Configure access mode for a single-VLAN endpoint.
3. Configure trunk mode for a link that must carry multiple VLANs.
4. Set the native VLAN if the design requires one.
5. Restrict allowed VLANs to the required set.
6. Verify the port mode and VLAN membership after changes.

**Troubleshooting focus**

When VLAN traffic does not pass as expected, check the port role first. Many issues come from a port being in the wrong mode or from trunk settings that do not match on both sides.

- Confirm the interface is in access or trunk mode as intended.
- Check that the VLAN exists and is assigned correctly.
- Compare native VLAN and allowed VLAN settings on both ends of the trunk.
- Look for untagged traffic handling problems on trunk links.

**Key takeaways**

Access ports keep a device in one VLAN. Trunks carry multiple VLANs between network devices using 802.1Q tagging. Native VLAN handling and allowed VLAN lists are the main details to verify when you configure or troubleshoot trunk links.
