## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Why verification matters**

Automation helps you apply changes consistently across many devices, but the run is not complete until you verify the result. In practice, you compare what you intended to deploy with what the device is actually running. This is how you confirm success, catch configuration drift, and avoid assuming a change worked just because the script finished.

Verification is especially important in larger environments where many switches and routers can change at once. A failed run, a manual edit, or an emergency fix can leave the device in a state that no longer matches the standard.

- **Intended state** is the configuration you meant to apply.
- **Running state** is the configuration currently active on the device.
- **Drift** is any difference between those states.
- Verification gives you evidence, not assumptions.

**Compare before you trust the result**

A practical verification workflow checks the device state against a known baseline or desired configuration. Some platforms provide built-in compare functions, while automation tools can compare a running configuration to a file or candidate configuration. The key idea is the same: detect differences before you commit to the change being correct.

For example, configuration comparison tools can show differences in one output, and network automation tools can report whether the device matches the desired state. If a diff appears, you investigate it before treating the run as successful.

```python
import napalm

driver = napalm.get_network_driver('ios')
device = driver(
    hostname='192.168.1.1',
    username='admin',
    password='password',
    optional_args={'secret': 'enablepass'},
)

device.open()
device.load_merge_candidate(filename='/etc/network/configs/router01.cfg')
diff = device.compare_config()

if diff:
    print("Configuration drift detected!")
    print(diff)
else:
    print("Device is in desired state. No drift.")

device.discard_config()
```

- Compare the **running configuration** to the intended configuration.
- Review the diff output for unexpected changes.
- Discard temporary candidate changes when you only want to verify.
- Treat a clean comparison as confirmation, not as a substitute for broader testing.

**Use verification commands and outputs**

Device verification commands help you see what the network is really doing instead of guessing from symptoms. They are useful when troubleshooting after automation, because the issue may be on a switch port, inside a VLAN, or in forwarding logic rather than in the automation tool itself.

Verification output should answer a simple question: did the device accept the change and does its state match what you planned?

- Check the relevant device state after the run.
- Confirm the configuration lines or operational values you expected.
- Look for mismatches between the baseline and the live device.
- Use the output to narrow the problem before you make another change.

**Common verification patterns**

Different platforms expose comparison in different ways, but the workflow stays consistent. Some systems compare baseline and candidate configurations from within a configuration context. Others compare a running configuration against a file or intended template. In both cases, you are validating the result of automation against a known reference.

| Pattern | What you compare | What you learn |
|---|---|---|
| Baseline vs. candidate | Saved reference and proposed changes | Whether the pending change matches intent |
| Running vs. intended file | Live device and desired config file | Whether the device drifted from standard |
| Operational verification | Device state and expected behavior | Whether the network reflects the change |

**Practical verification checklist**

1. Run the automation task.
2. Compare the resulting device state with the intended configuration.
3. Review any diff or mismatch output.
4. Confirm the device is in the expected state.
5. If differences remain, investigate before declaring success.

> Do not assume a completed automation job means the configuration is correct. Always verify the device state after the run.

**What to remember for exam prep**

- Automation is only reliable when you verify the outcome.
- Configuration comparison helps you detect drift quickly.
- Verification commands show the actual network state.
- A successful check means the device matches the intended result, not just that the script executed.
