## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Blocked traffic: start with the logs**

When traffic appears to be blocked, the first check is the firewall log. pfSense logs dropped traffic by default and does not log passed traffic unless a rule is configured to do so. If you do not see a red log entry that matches the traffic, the firewall is not likely dropping it.

- Check `Status > System Logs` on the `Firewall` tab first.
- Look for entries that match the source, destination, and time of the failed connection.
- Remember that a pass rule without logging may leave no visible trace in the log.

A log entry that looks like a block does not always mean the connection is truly denied. Some packets are harmless late arrivals or retransmits that no longer fit the current connection state.

**State table checks**

The state table shows whether the firewall has already accepted a connection. After you attempt the connection, check `Diagnostics > States` and filter by source or destination.

- If a state exists, the firewall has passed the traffic.
- If the rule is a pass rule, the state confirms the packet was allowed.
- If no state exists and the log shows a block, the firewall likely rejected the traffic.

The state table is especially useful when the log and the user report do not agree. It helps you separate a real filtering problem from a timing or state-tracking issue.

**Why legitimate packets can look blocked**

Stateful firewalls track connections, not just individual packets. That means some packets can be logged as blocked even when the session itself was valid.

- A common case is a TCP <abbr title="Transmission Control Protocol">TCP</abbr> `FIN` packet arriving after the firewall has already removed the connection state.
- Lost packets can cause retransmits that arrive after the state is gone.
- Packets that arrive too slowly can fall outside the expected arrival window.
- Web servers may try to reuse connections, which can also trigger harmless blocked-log entries.

> These log entries are usually harmless and do not indicate an actual blocked connection.

**Rule precedence and matching behavior**

Firewall rules are evaluated in order, so rule placement matters. A rule that appears to allow traffic may still not match the packet you are testing if the packet is outside the expected state or does not meet the rule conditions.

- A pass rule for TCP typically allows `TCP SYN` packets to create a state.
- Other TCP flags are expected to match an existing state in the state table.
- Packets with spoofed TCP flags can also appear in blocked logs.
- An `Allow All` style rule does not override state tracking for packets that no longer belong to a valid session.

This is why a packet can be logged under a deny-style message even when a broad allow rule exists. The firewall is enforcing connection state, not just rule intent.

**Practical troubleshooting flow**

1. Check the firewall log for a matching blocked entry.
2. If the log is unclear, attempt the connection again.
3. Check the state table immediately after the attempt.
4. Compare the rule behavior with the packet type and connection state.
5. Treat late `FIN` packets, retransmits, and similar out-of-state traffic as likely harmless unless other evidence shows a real outage.

**Common interpretation guide**

| Observation | Likely meaning |
|---|---|
| Red block log matches the traffic and no state exists | The firewall likely dropped the packet |
| No matching log entry appears | The firewall is not likely blocking the traffic |
| State exists after the attempt | The firewall passed the traffic |
| Blocked log shows a late `TCP FIN` or retransmit | The entry is often harmless and state-related |

**Key takeaway**

Use the log to find the symptom, the state table to confirm whether the firewall passed the session, and rule order to explain why a packet matched or missed a rule. In many cases, a blocked-looking entry is just a normal side effect of stateful filtering.
