## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**pfSense at the network edge**

pfSense is a free network firewall distribution based on the FreeBSD operating system. It includes a web interface for configuring its components, so you do not need UNIX knowledge to manage basic deployment tasks. In practice, you place pfSense at the edge of a network to control traffic between internal systems and external networks.

Because pfSense can provide firewall and routing functions, it is commonly used as the NAT router and firewall for a lab or small environment. When you deploy it in a virtualized lab, make sure it starts before downstream hosts so those systems can reach the network through it.

- pfSense is based on **FreeBSD** and uses a custom kernel.
- It includes third-party packages for additional functionality.
- The web interface is the primary management method.
- It can replace or emulate many commercial firewall deployments in lab and production-style environments.

**Accessing the web configurator**

After a default installation, you access the pfSense web configurator from a browser on a computer connected to the firewall. Use `https://[your LAN IP address]` to reach the interface.

The web UI is where you review menus, configure services, and apply firewall behavior. A stock installation exposes many menus, so it helps to approach configuration by function: interface setup, firewall policy, and any additional services you enable.

- Open a browser from a host on the same network segment as the firewall.
- Connect to `https://[your LAN IP address]`.
- Use the menu system to reach configuration pages after login.
- Expect many options even on a default install.

**Firewall rule behavior**

Firewall rules in pfSense are configured under `Firewall > Rules`. These rules determine how packets are matched and what action the firewall takes. The main actions are pass, block, and reject.

A **pass** rule allows matching traffic through. If state tracking is enabled, pfSense creates a state table entry for the first packet in a connection, which then allows related return traffic. A **block** rule discards matching packets. A **reject** rule also discards matching packets, but for compatible protocols it sends a message back to the source indicating the connection was refused.

> Use `reject` when you want the sender to receive a refusal message; use `block` when you want silent discard behavior.

- **Pass** allows matching packets through.
- **Block** discards matching packets.
- **Reject** discards packets and may notify the source.
- **State tracking** helps return traffic pass for established connections.

**Rule matching and access control**

Firewall rules are one form of access control list, or **ACL**. An ACL is a list of traffic that is allowed or disallowed. You can combine criteria such as source IP address, destination IP address, port number, time of day, or application to shape access decisions.

This matters at the edge because the firewall is not just filtering by address. It can enforce more specific policy based on who is connecting, where they are connecting to, and what kind of traffic they are sending.

- ACLs describe allowed and disallowed traffic.
- Rule criteria can include source and destination IP addresses.
- Port numbers, time of day, and application can also be part of the match.
- Multiple criteria let you build more precise policy.

**Practical deployment focus**

When you deploy pfSense as an edge firewall, think in terms of placement, access, and policy. Placement determines whether pfSense can see all traffic that should be filtered. Access determines how you reach the web configurator. Policy determines what traffic is allowed, blocked, or rejected.

In a lab environment, pfSense often serves as the NAT router and firewall for the segment it protects. That makes it the central control point for traffic entering and leaving the protected network.

- Place pfSense where it can inspect edge traffic.
- Confirm you can reach the web UI from a connected host.
- Build rules that reflect the access you want to permit or deny.
- Verify the firewall starts before dependent systems in virtual labs.

**Quick rule-selection guide**

| Action | Result | Typical use |
|---|---|---|
| `pass` | Allows traffic | Permit approved connections |
| `block` | Silently discards traffic | Drop unwanted traffic without a response |
| `reject` | Discards traffic and may respond | Refuse a connection explicitly |

**Key takeaways**

pfSense gives you a web-managed firewall platform that can sit at the network edge and enforce traffic policy. For exam prep, focus on where you access the interface, how firewall actions differ, and how ACL-style matching supports granular control.
