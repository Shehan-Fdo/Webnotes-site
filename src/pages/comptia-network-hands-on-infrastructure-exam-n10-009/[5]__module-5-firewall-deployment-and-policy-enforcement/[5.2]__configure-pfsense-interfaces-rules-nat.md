## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**pfSense configuration overview**

pfSense uses the `Firewall > NAT` and `Firewall > Rules` areas of the webGUI to control how traffic is translated and filtered. In practice, you use NAT to map addresses between networks and firewall rules to decide which packets are allowed, blocked, or rejected.

Network Address Translation (<abbr title="Network Address Translation">NAT</abbr>) changes network address information in packet headers while traffic is in transit. The most common use is to let a private LAN reach the public Internet with a single public IPv4 address, but pfSense also supports more advanced deployments with multiple public addresses.

- **Outbound NAT** translates source addresses and ports for traffic leaving an interface.
- **Inbound NAT** handles traffic entering a network, such as port forwards.
- **Stateful filtering** lets reply traffic pass automatically after an allow rule creates a state table entry.

**Firewall rule behavior**

Rules on interface tabs are applied per interface and in the inbound direction on that interface. That means LAN traffic is filtered with LAN rules, and Internet-originated traffic is filtered with WAN rules. Because pfSense rules are stateful by default, you usually allow the initiating traffic and let return traffic follow the state entry.

> Outbound rules are generally not required in pfSense because filtering happens on the inbound direction of each interface.

When you configure rules under `Firewall > Rules`, the action determines how matching packets are handled:

- **Pass** allows matching packets through and can create a state table entry.
- **Block** discards matching packets.
- **Reject** discards matching packets and, for compatible protocols, sends a refusal message back to the source.

Floating rules are the main exception to the interface-tab model. They can act on any interface and can apply in the inbound direction, outbound direction, or both.

**Rule order and processing flow**

pfSense processes rules in a specific order, and that order matters when you troubleshoot unexpected behavior. The documented logical sequence includes Ethernet rules, outbound NAT, inbound NAT such as port forwards, dynamically received rules for IPsec and OpenVPN clients, internal automatic rules, and then user-defined firewall rules.

This means a packet may be translated before it reaches a firewall rule, depending on the traffic path and configuration. When you are checking why a packet is allowed or denied, verify both the NAT path and the firewall rule path.

- Check NAT first when translation affects the packet’s source or destination.
- Check interface rules next when the packet is filtered on the inbound side.
- Check floating rules if the traffic does not match the expected interface tab.

**Outbound NAT modes**

Outbound NAT controls how pfSense translates source addresses and ports for traffic leaving an interface. You configure it at `Firewall > NAT` on the `Outbound` tab.

The main modes are:

- **Automatic Outbound NAT**: pfSense creates NAT rules automatically for internal interfaces such as LAN to external interfaces such as WAN.
- **Hybrid Outbound NAT**: pfSense uses manual rules and automatic rules together, which gives extra control without requiring full manual management.
- **Manual Outbound NAT**: pfSense uses only the rules you enter by hand.
- **Automatic NAT rules generation**: pfSense can generate rules automatically, which is useful when you want the default behavior without managing each rule manually.

A practical way to think about the modes is control versus convenience. Automatic is simplest, manual is most explicit, and hybrid sits between them.

| Mode | Behavior | Practical use |
|---|---|---|
| Automatic | Creates rules for common internal-to-external traffic | Best for simple deployments |
| Hybrid | Uses manual rules plus automatic rules for unmatched traffic | Best when you need some custom control |
| Manual | Uses only manually entered rules | Best when you need full control and can maintain the rules carefully |

**NAT and firewall troubleshooting**

When traffic does not behave as expected, confirm the translation path and the filtering path separately. NAT and firewall processing are related, but they are not the same thing.

Use this approach:

1. Identify whether the traffic is leaving the network, entering the network, or staying internal.
2. Check whether NAT changes the source or destination address.
3. Check the relevant interface rule tab for inbound filtering.
4. Review floating rules if the traffic crosses multiple interfaces or uses special handling.
5. Confirm whether the outbound NAT mode matches the intended level of control.

A conservative troubleshooting mindset helps here: if a packet is not matching the rule you expect, the issue may be rule order, interface direction, or NAT translation rather than the rule content itself.

**Exam focus reminders**

For Network+ N10-009, focus on the relationship between NAT, firewall rules, and traffic direction. You should be able to explain why pfSense usually filters on inbound interface rules, how stateful filtering reduces the need for explicit return rules, and how outbound NAT changes source addresses for traffic leaving the network.

- Know where to find NAT settings: `Firewall > NAT`.
- Know where to find firewall rules: `Firewall > Rules`.
- Know that outbound NAT translates traffic leaving an interface.
- Know that rule order affects packet handling.
