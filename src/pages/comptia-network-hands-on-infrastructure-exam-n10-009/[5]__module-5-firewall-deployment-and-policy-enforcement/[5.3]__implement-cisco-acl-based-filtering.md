## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**ACL filtering overview**

An <abbr title="Access Control List">ACL</abbr> is a rule set that allows or denies traffic based on matching criteria. In Cisco environments, you use ACLs to control traffic flow on routers and other devices by matching fields such as source IP address, destination IP address, protocol, and port number.

Extended ACLs give you more precise control than simple address-based filtering. They are useful when you need to permit or block specific application traffic, such as Telnet, SSH, or web access, rather than only entire hosts or networks.

- ACLs can be used to allow or disallow traffic.
- Matching logic depends on rule order and placement.
- Extended ACLs can combine multiple criteria in one statement.
- ACLs appear on routers, firewalls, operating systems, and other network devices.

**Planning the filter**

Before you configure an ACL, identify the traffic you want to match and where the filter should be applied. The lab evidence emphasizes translating requirements into ACL statements that use source, destination, protocol, and port numbers.

A practical lab topology includes multiple LANs, routers, and a web server, with routing already in place through OSPF. That means the ACL work focuses on traffic control, not basic reachability.

- Define the source network or host.
- Define the destination network, host, or server.
- Identify the protocol and port numbers involved.
- Decide whether the ACL should be numbered or named.

**Applying extended ACLs**

Cisco IOS uses ACL statements that are evaluated in order. You place the most specific matches where they can affect the intended traffic path, then apply the ACL to the correct interface and direction.

In the lab context, you configure, apply, and verify both numbered extended ACLs and named extended ACLs. Verification matters because an ACL can be present in the configuration but still not affect traffic if it is attached to the wrong interface or direction.

1. Create the ACL entries that match the required traffic.
2. Apply the ACL to the appropriate interface.
3. Choose the correct direction for the traffic flow.
4. Test the affected connections.
5. Verify the ACL is active on the intended interface.

> If an ACL is applied to the wrong interface or direction, the rule may not filter the traffic you expected.

**Verification and troubleshooting**

Verification is part of the workflow, not an afterthought. You should confirm both connectivity and filtering behavior after the ACL is applied. In the lab, this includes checking that the network still routes correctly while the targeted traffic is blocked or allowed as required.

Useful checks focus on whether the ACL is attached where you expect and whether the matching logic aligns with the requirement. Cisco community guidance also highlights the importance of knowing which interface an access list is applied to.

- Confirm the ACL is applied to the intended interface.
- Confirm the direction matches the traffic path.
- Test allowed traffic and denied traffic separately.
- Recheck rule order if the result is not what you expected.

**Key takeaways**

Extended ACLs are the right tool when you need granular filtering based on more than just source addresses. They let you express policy in terms of protocol and port, which makes them suitable for application-specific restrictions.

For exam prep, focus on three habits:
- Read the traffic requirement carefully.
- Translate the requirement into ordered ACL statements.
- Verify the ACL placement and behavior after deployment.
