## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Enterprise troubleshooting context**

This topic focuses on diagnosing a broken enterprise topology when more than one fault exists at the same time. The lab-style evidence points to a multi-site hub-and-spoke WAN with centralized services at HQ and segmented branch networks. Your job is to isolate issues layer by layer, then restore full connectivity and service access.

The practical value here is not just finding a single bad setting. You need a repeatable process that helps you separate routing problems from infrastructure-service failures and security-policy blocks.

- **Routing** issues can prevent branch networks from reaching each other or HQ services.
- **DHCP** problems can stop hosts from receiving valid IP settings.
- **DNS** failures can make name resolution appear broken even when basic connectivity exists.
- **ACLs** can block specific traffic while leaving other paths functional.

**Topology and lab setup**

The evidence describes a topology-based troubleshooting lab where you cable the network according to a diagram, load devices with supplied scripts, and correct all network errors. One source also shows an addressing table with multiple routers, switches, PCs, and a TFTP server, which reinforces that the environment includes both routed links and local LAN segments.

A practical takeaway is that you should always confirm the physical and logical layout before changing configurations. In a topology with multiple simultaneous faults, a wrong assumption about the path can send you in the wrong direction.

- Verify device roles and interfaces before testing.
- Check whether a host is on a local VLAN, a routed subnet, or a WAN link.
- Confirm default gateways and addressing consistency on end devices.
- Treat the topology diagram as the source of truth for expected connectivity.

**Troubleshooting methodology**

Use a structured troubleshooting flow instead of guessing. The evidence supports a layered approach: identify the problem, establish theories, test them, evaluate results, implement a fix, verify functionality, and document findings. That sequence is especially important when several faults overlap.

1. Identify the symptom and the scope of failure.
2. Test basic connectivity before moving to higher-layer services.
3. Verify routing and reachability between subnets.
4. Check infrastructure services such as DHCP and DNS.
5. Inspect security controls such as ACLs if only specific traffic fails.
6. Apply the smallest safe fix.
7. Re-test the original symptom and related paths.
8. Document what changed and what was verified.

> When multiple faults exist, do not assume the first visible symptom is the root cause. A host that cannot browse by name may have both a routing issue and a DNS issue, so you need to validate each layer independently.

**Common fault patterns in the lab**

The evidence highlights several realistic failure types that can appear together in an enterprise network. These are useful because they force you to distinguish between transport, service, and policy problems.

- **Missing route advertisement** can isolate a branch from the rest of the network.
- **DHCP relay misconfiguration** can prevent clients from receiving addresses.
- **ACL rules** can block DNS traffic to an internal server.
- **Incorrect interface or VLAN settings** can break local segment reachability.
- **Incomplete device loading or cabling** can create failures before routing is even tested.

A good diagnostic habit is to ask whether the failure is universal or selective. Universal failure often points to cabling, interface state, or routing. Selective failure often points to service configuration or filtering.

**How to isolate the fault domain**

Start with the simplest checks and move outward. If a host cannot reach its gateway, focus on local addressing, VLAN membership, and interface status. If it can reach the gateway but not remote subnets, focus on routing. If routing works but address assignment fails, focus on DHCP. If IP connectivity works but names do not resolve, focus on DNS. If only one application or port fails, inspect ACLs.

- **Layer 1/2**: cabling, link state, VLAN placement.
- **Layer 3**: IP addressing, default gateway, routing advertisements.
- **Services**: DHCP scope, relay, DNS reachability.
- **Policy**: ACL direction, match conditions, and affected ports.

**Verification mindset**

After each correction, verify the original symptom and at least one related path. In a multi-fault topology, one fix can expose the next problem, so you should expect the network to improve in stages. That is normal and useful because it confirms you are removing real faults rather than masking them.

Use the addressing plan and topology diagram to compare expected versus actual behavior. If a branch subnet should reach HQ services but cannot, confirm the route path first, then test service access. If a client gets an address but cannot resolve names, test DNS separately instead of revisiting DHCP too early.

**Exam-ready takeaways**

- Use a **layered troubleshooting** process when several symptoms appear together.
- Separate **connectivity** problems from **service** problems.
- Validate the topology, addressing, and device roles before changing configs.
- Expect multiple faults to reveal themselves one at a time after each fix.
- Document the final working state so you can reproduce the resolution.
