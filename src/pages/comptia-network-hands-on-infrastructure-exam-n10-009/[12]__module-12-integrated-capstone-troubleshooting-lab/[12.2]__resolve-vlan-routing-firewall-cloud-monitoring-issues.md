## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Troubleshooting scope**

This topic focuses on resolving connectivity and visibility issues across VLANs, routing, firewall controls, cloud lab environments, and monitoring workflows. In practice, you verify where traffic should flow, confirm the correct interfaces and addresses are in use, and then isolate the layer where the failure appears.

- **VLANs** segment traffic, so a mismatch in VLAN assignment can stop hosts from reaching the expected subnet.
- **Routing** problems often show up when inter-VLAN traffic cannot cross the router or when a default gateway is wrong.
- **Firewall** rules can block traffic even when addressing and switching look correct.
- **Cloud lab** environments may enforce control-network restrictions, so using the wrong interface can trigger a violation.
- **Monitoring** is most useful when it gives actionable context, not just alerts.

**Inter-VLAN routing checks**

Inter-VLAN routing depends on consistent addressing, correct VLAN membership, and router interfaces that match the intended subnets. In the lab example, the router uses subinterfaces such as `G0/0/1.3`, `G0/0/1.4`, and `G0/0/1.13`, each tied to a different VLAN subnet. Switch management interfaces also use a VLAN address and a default gateway that points to the router.

- Confirm each host uses the correct IP address, subnet mask, and default gateway.
- Verify switch management interfaces are assigned to the intended VLAN.
- Check that router subinterfaces map to the correct VLAN IDs and gateway addresses.
- Use the VLAN table to compare access ports, parking-lot ports, and native VLAN placement.

A practical troubleshooting pattern is to compare the live configuration against the addressing plan before changing anything. If a PC cannot reach another subnet, start with the local VLAN, then the router subinterface, then the gateway path.

```text
R1 G0/0/1.3 10.3.0.1/24
R1 G0/0/1.4 10.4.0.1/24
R1 G0/0/1.13 10.13.0.1/24
S1 VLAN 3 10.3.0.11/24 gateway 10.3.0.1
S2 VLAN 3 10.3.0.12/24 gateway 10.3.0.1
PC-A 10.4.0.50/24 gateway 10.4.0.1
PC-B 10.13.0.50/24 gateway 10.13.0.1
```

**Cloud control-network awareness**

Cloud lab environments can include a dedicated control network that must not be used for normal experiment traffic. The documentation emphasizes ensuring that experiment nodes use the correct interfaces and avoiding control-network misuse. If you receive a control-network violation message, the first step is to verify that your nodes are attached to the intended interfaces.

- Use the correct interface for experiment traffic.
- Check for accidental use of the control network.
- Verify interface selection empirically when the topology is unclear.
- Treat a violation email as a sign to re-check node wiring and interface mapping.

> Avoid assuming that a node is on the right network just because it is reachable. In controlled lab environments, the wrong interface can still produce connectivity and policy problems.

**Firewall and access control issues**

Firewall problems usually appear as selective reachability: some services work while others fail. When you troubleshoot, separate addressing issues from policy issues so you do not chase the wrong layer. If routing and VLAN placement are correct but traffic still fails, inspect filtering rules, permitted ports, and any security controls between segments.

- Test whether the failure affects one host, one subnet, or all traffic.
- Compare allowed and blocked flows rather than only checking link status.
- Confirm that the expected path is not being filtered by a security device.
- Re-test after each change so you can identify the rule that fixed the issue.

**Monitoring and remediation**

Monitoring is most effective when it helps you move from alert to action. Modern monitoring tools can provide diagnostics that explain what went wrong and what needs to be corrected. Some platforms also support automated remediation workflows, including scripted actions for common device and performance issues.

- Use monitoring data to narrow the fault domain before making changes.
- Prefer alerts with context over context-free notifications.
- Apply targeted remediation when the issue is repetitive and well understood.
- Recheck service health after remediation to confirm the fix worked.

**Troubleshooting workflow**

1. Identify the symptom and the affected scope.
2. Verify addressing, VLAN membership, and default gateways.
3. Confirm router subinterfaces and switch management VLANs.
4. Check for firewall or policy filtering.
5. Validate cloud lab interface usage if the issue occurs in a controlled environment.
6. Re-test connectivity and monitoring status after each correction.

**Key takeaways**

- Start with the simplest layer: host addressing and VLAN membership.
- Use the routing plan to confirm inter-VLAN paths.
- Treat cloud control-network rules as a hard constraint.
- Use monitoring to guide remediation, not just to record failure.
- Re-verify after every change so you know which fix actually resolved the issue.
