## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Route table fundamentals**

A route table is the traffic controller for a virtual private cloud (VPC). It contains routes that tell network traffic where to go from a subnet or gateway. You use route tables to control which networks your VPC can reach, including other VPCs and on-premises networks.

Each route has two core parts:

- **Destination**: a CIDR block or prefix list.
- **Target**: the next hop, such as an internet gateway, NAT gateway, VPC peering connection, or VPN connection.

Traffic is routed based on destination IP address, so the route table determines the path before packets leave the subnet. When you create a VPC, the platform creates a main route table automatically, and you can add more route tables for finer control.

**Common routing patterns**

Route tables support several network designs. You can use them to build public subnets, private subnets, VPN-only subnets, and isolated subnets. This makes route tables a practical control point for separating workloads by exposure and connectivity.

Typical routing choices include:

- Public subnet traffic to an internet gateway.
- Private subnet traffic to a NAT gateway for outbound internet access.
- Site-to-site or hybrid traffic to a VPN connection.
- Inter-VPC traffic to a VPC peering connection.

These patterns let you shape access without changing the hosts themselves. The route table becomes the policy layer for where traffic is allowed to go.

**Security policy routing with AWS Network Firewall**

AWS Network Firewall uses route table changes to force traffic through firewall endpoints. You modify the VPC route tables so the traffic you want filtered passes through the firewall before it reaches its destination.

A common design is to route traffic between an internet gateway and customer subnets through the firewall endpoint. In that model, incoming traffic from the internet gateway and outgoing traffic from the customer subnets both traverse the firewall path.

> Be careful to avoid asymmetric routing. If return traffic does not follow the expected path, filtering and connectivity can fail.

**Subnet route table verification**

When a wizard configures the VPC, it may set up route tables automatically. If you build the VPC manually, verify the subnet route tables before you rely on the design.

Check these relationships:

- The route table for the subnet that contains the NAT gateway must include a route to an internet gateway.
- Private subnet route tables must point internet traffic to the NAT gateway.
- The route table association must match the subnet role you expect.

A quick verification workflow is:

1. Open the subnet list and select the public subnet.
2. Open the subnet’s Route Table tab.
3. Open the route table ID and confirm the routes.
4. Rename the route table if needed so its purpose is obvious.

**Practical exam takeaways**

Route tables are not just a list of paths; they are how you control reachability in a VPC. For exam scenarios, focus on the relationship between subnet type, next hop, and traffic direction.

- **Destination** tells you where the traffic is going.
- **Target** tells you which network device or connection receives it.
- **Main route table** exists by default for each VPC.
- **Firewall routing** works by steering traffic through firewall endpoints.
- **Private subnet internet access** usually depends on a NAT gateway.
