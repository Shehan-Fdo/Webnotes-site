## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Site-to-site VPN overview**

A site-to-site VPN creates a secure connection between your on-premises equipment and your VPCs. In AWS, this lets a VPC communicate with a local network or remote device that it cannot reach by default. You enable the path by creating the VPN connection and configuring routing so traffic passes through it.

AWS Site-to-Site VPN uses <abbr title="Internet Protocol security">IPsec</abbr> VPN connections. The key idea is that the VPN is not just the tunnel itself; it also depends on routing so packets know how to travel between the two networks.

- A **VPN connection** is the secure connection between your on-premises equipment and your VPCs.
- A **VPN tunnel** is the encrypted link that carries data.
- Routing must direct traffic through the VPN for communication to work.

**Planning the connection**

Before you create the VPN, decide what the target gateway will be. AWS supports several target gateway types during creation, including a virtual private gateway, a transit gateway, a Site-to-Site VPN Concentrator, or `Not associated`.

If you choose `Not associated`, you can assign the target gateway later or use the attachment for AWS Cloud WAN. For the hands-on workflow in the source material, the setup uses a virtual private gateway and assumes you already have a VPC with one or more subnets.

- Use a **virtual private gateway** when following the standard setup flow shown in the source.
- Use `Not associated` only when you intend to attach later or use AWS Cloud WAN.
- Start with an existing VPC and subnets before creating the VPN connection.

**Setup workflow**

The setup process follows a practical sequence. You create the customer gateway, create the target gateway, configure routing, update the security group, create the VPN connection, download the configuration file, and then configure the customer gateway device.

1. Create the customer gateway.
2. Create the target gateway.
3. Configure routing.
4. Update the security group.
5. Create the VPN connection.
6. Download the configuration file.
7. Configure the customer gateway device.

This order matters because the VPN depends on both endpoints and on routing rules that allow traffic to move in both directions.

**Testing the connection**

After you create the VPN and configure the customer gateway, test end-to-end connectivity by launching an instance and pinging it. Use an AMI that responds to ping requests; Amazon Linux is recommended in the source material.

You also need to allow ICMP traffic through the VPC controls and, if applicable, the guest firewall.

- Allow inbound and outbound **ICMP** traffic in the security group or network ACL if traffic is filtered.
- On Windows Server instances, enable inbound ICMPv4 in the Windows firewall.
- Use ping as the basic verification method after setup.

> If you use static routing, make sure the customer gateway device has a static route to your VPC and that the VPN connection has a static route back to the customer gateway device.

**Dynamic versus static routing checks**

The verification steps differ depending on routing mode. With static routing, both sides need routes that point to each other. With dynamic routing, you check that Border Gateway Protocol (BGP) peering is established on the customer gateway device.

- **Static routing** requires matching routes in both directions.
- **Dynamic routing** requires BGP status to be established.
- BGP peering can take about 30 seconds to establish.

**Practical troubleshooting focus**

If ping fails, check the most likely blockers first: routing, security filtering, and endpoint readiness. The instance must respond to ping, the VPC controls must permit ICMP, and the customer gateway must have the correct route or established BGP session.

A simple troubleshooting sequence is to confirm the instance can answer ping, confirm the security group or network ACL allows ICMP, and then confirm the routing state on the customer gateway and VPN connection.
