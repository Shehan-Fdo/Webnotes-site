## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Troubleshooting model for VPN failures**

When a site-to-site VPN does not pass traffic correctly, separate the problem into three areas: tunnel negotiation, routing, and security policy. A tunnel can appear partially healthy while still failing to carry traffic, so you need to verify each layer instead of assuming the connection is fully working.

AWS guidance notes that a VPN is only fully up when both <abbr title="Internet Key Exchange">IKE</abbr> Phase 1 and <abbr title="Internet Protocol Security">IPsec</abbr> Phase 2 are up. For dynamic VPNs, <abbr title="Border Gateway Protocol">BGP</abbr> must also be up. If one phase is down, the overall VPN status is down.

- Check whether the tunnel fails during negotiation or after the tunnel comes up.
- Use logs to identify which phase is failing.
- Confirm whether the issue is one-way traffic, such as ingress without egress.
- Treat selector or policy mismatches as a separate class of failure from basic tunnel establishment.

**Tunnel negotiation checks**

Start with the tunnel state before changing routes or firewall rules. If the VPN cannot establish connectivity, the failure is usually in IKE Phase 1 or IPsec Phase 2. AWS recommends turning on Site-to-Site VPN logs and checking the phase status in the logs or on the customer gateway device.

> If Phase 1 is up but Phase 2 is down, the VPN is still considered down.

Useful checks include:

- Verify that both peers agree on the negotiation parameters.
- Confirm that the tunnel status is not only partially established.
- Review the customer gateway device status alongside cloud-side logs.
- Reset the tunnel or gateway only after you confirm the failure point.

**Routing and redundancy**

Routing problems can make a tunnel look healthy while traffic still fails. AWS Site-to-Site VPN connections use two tunnels, each with a unique public IP address, and both should be configured for redundancy. If one tunnel becomes unavailable, traffic automatically uses the other tunnel for that connection.

This matters because a routing issue may affect only one direction or one tunnel path. In cloud-to-cloud or cloud-to-on-premises designs, you should confirm that traffic is actually taking the expected path and that failover behavior works as intended.

- Expect two tunnels on an AWS Site-to-Site VPN connection.
- Use both tunnels for redundancy.
- Verify that traffic can fail over when one tunnel is unavailable.
- Check whether the problem is limited to one direction of traffic.

**Security policy and selector mismatches**

A common symptom of policy mismatch is that the tunnel comes up, but packets are dropped. In the Azure-to-AWS example, traffic was connecting and receiving ingress but not egress, and troubleshooting indicated a selector mismatch even after the selectors were manually reviewed. That pattern suggests the tunnel negotiation may be successful while the traffic selectors or policy rules still do not match the actual traffic flow.

Policy-based routing and policy-based VPN rules can also affect whether traffic matches the intended path. If traffic does not match the configured policy, it may be sent to a default rule instead of the VPN path.

- Compare the configured selectors with the actual source and destination networks.
- Check whether route-based and policy-based settings align with the design.
- Confirm that the traffic matches the intended security policy.
- Review whether a default policy is catching traffic that should use the tunnel.

**Practical troubleshooting sequence**

1. Check the tunnel logs and identify whether IKE Phase 1 or IPsec Phase 2 is failing.
2. Confirm whether the VPN is fully up or only partially established.
3. Verify routing and failover behavior across both tunnels.
4. Compare traffic selectors, policies, and actual traffic flows.
5. If traffic is still dropped, review the customer gateway and cloud-side configuration together.

**What to remember for exam scenarios**

A tunnel that is “up” does not always mean traffic will pass. The most useful habit is to isolate the failure domain:

- Negotiation failure points to IKE or IPsec.
- Routing failure points to path selection or failover.
- Policy mismatch points to selectors, firewall policy, or route-based versus policy-based design differences.

When you troubleshoot in this order, you avoid changing the wrong layer first and can explain why a VPN may show ingress but still block egress.
