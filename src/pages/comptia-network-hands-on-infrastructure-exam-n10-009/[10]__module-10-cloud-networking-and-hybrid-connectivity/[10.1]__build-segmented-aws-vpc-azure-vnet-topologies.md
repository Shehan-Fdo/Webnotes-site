## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Cloud subnet segmentation overview**

Segmented cloud topologies use route tables to control how traffic moves between subnets and external networks. In both AWS and Azure, the subnet is the unit you manage, and the route table is the policy layer that directs traffic based on destination.

Cloud networking on this topic is less about creating isolated boxes and more about shaping paths. You use route tables to support public, private, VPN-only, and isolated subnet designs, depending on where traffic should go and what should stay local.

- **AWS VPC** uses an implicit router and route tables to direct subnet traffic.
- **Azure VNet** automatically creates a route table for each subnet and applies system routes.
- **Custom routes** let you override default behavior when you need tighter control.

**AWS VPC route table fundamentals**

In Amazon Virtual Private Cloud (VPC), each subnet must be associated with a route table. If you do not explicitly associate one, the subnet uses the main route table by default. A subnet can use only one route table at a time, but multiple subnets can share the same table.

A route entry contains a destination and a target. The destination can be a Classless Inter-Domain Routing (CIDR) block or prefix list, and the target can be an internet gateway, NAT gateway, VPC peering connection, or VPN connection. This makes route tables the main control point for subnet reachability.

- A route table acts as the traffic controller for a VPC.
- The main route table is created when you create the VPC.
- Additional route tables give you more granular control over network paths.
- Routes are matched by destination IP address.

Common AWS routing patterns include:
- Public subnets that send `0.0.0.0/0` to an internet gateway.
- Private subnets that use a NAT gateway for outbound internet access.
- VPN-only subnets that send traffic to a VPN connection.
- Isolated subnets that do not route to external networks.

> `0.0.0.0/0` represents all IPv4 addresses, so it is the default route for internet-bound traffic when the target is an internet gateway.

**Managing AWS subnet associations**

You can inspect subnet routing in the Amazon VPC console by opening the subnet details and checking the Route table tab. This helps you confirm whether the subnet uses the main route table or an explicit association.

When you manage AWS routing, keep the association model in mind:
- A subnet is associated with one route table at a time.
- Multiple subnets can point to the same route table.
- Explicit associations make segmentation easier to verify and maintain.

Use route table changes carefully because they immediately affect traffic flow for the associated subnet. If you are troubleshooting connectivity, start by confirming the subnet’s route table and then review the destination-target pairs.

**Azure VNet route table behavior**

Azure Virtual Network (VNet) routing follows a similar model, but Azure automatically creates a route table for each subnet and adds system default routes. Azure routes outbound traffic from a subnet based on the routes in that subnet’s route table.

You can override some system routes with custom routes. This is useful when you want to steer traffic through a network virtual appliance (NVA) or otherwise change Azure’s default routing between subnets, virtual networks, on-premises networks, and the internet.

- Azure automatically routes traffic between subnets by default.
- Custom routes change that default behavior.
- Route tables help you optimize connectivity and troubleshoot network issues.
- A route table can support more advanced designs, such as traffic inspection through an NVA.

**Azure segmentation use cases**

Azure route tables are especially useful when you want to force traffic through a controlled hop. For example, you can create a route table, add a route, associate it to a subnet, and then route traffic from one subnet to another through an NVA.

This pattern supports layered designs where:
- One subnet hosts workloads.
- Another subnet hosts the NVA.
- A route table directs traffic through the appliance before it reaches its destination.

That approach gives you a practical way to segment traffic without changing the basic subnet layout.

**AWS and Azure comparison**

| Platform | Default behavior | Custom control | Typical segmentation outcome |
|---|---|---|---|
| AWS VPC | Subnets use the main route table unless explicitly associated | Add routes to gateways, peering, or VPN targets | Public, private, VPN-only, isolated subnets |
| Azure VNet | Each subnet gets a system route table automatically | Add custom routes to override defaults | Routed subnet paths, including NVA-based inspection |

**Practical workflow for segmented topologies**

1. Identify the subnet boundaries you want to separate.
2. Decide whether the subnet should reach the internet, another network, or stay isolated.
3. Create or select the route table that matches that intent.
4. Add the required route entries with the correct destination and target.
5. Associate the route table to the subnet.
6. Verify the effective routing from the subnet details or route table view.

**Troubleshooting focus**

When traffic does not follow the expected path, check the route table first. In AWS, confirm the subnet association and the route target. In Azure, confirm whether a custom route overrides the default system route and whether the subnet is associated with the intended table.

A good troubleshooting habit is to validate the destination, target, and association together. If any one of those is wrong, the subnet may still exist and deploy correctly, but traffic will not flow as intended.

**Exam takeaways**

- Route tables control subnet traffic in both AWS and Azure.
- AWS requires explicit subnet-to-route-table association unless the main table is used.
- Azure creates subnet route tables automatically and lets you override system routes.
- Segmented topologies depend on matching route intent to subnet role.
- `0.0.0.0/0` is the key AWS default route for internet access through an internet gateway.
