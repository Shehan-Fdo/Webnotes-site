## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Service reachability and the default gateway**

A host needs an IP address, subnet mask, and default gateway to communicate beyond its local network. The default gateway is the next hop a device uses when the destination is on another network. If the gateway is wrong or unreachable, traffic may fail even when the local address settings look correct.

In the lab-style topology, devices on each subnet use the router interface on that subnet as the gateway. For example, hosts in `192.168.10.0/24` use `192.168.10.1`, and hosts in `192.168.11.0/24` use `192.168.11.1`. Switch management interfaces also point to the router interface in their local subnet.

- **Default gateway** issues often appear as failed access to remote hosts, not local hosts.
- Verify the host IP settings before changing routing or services.
- Match the gateway to the correct subnet interface.
- A device can still reach local neighbors when the gateway is wrong.

**What to verify first**

Start by checking the network documentation and comparing it to the live configuration. The goal is to isolate whether the problem is on the host, the gateway, or the path beyond the gateway.

- Confirm the host has the expected IP address and subnet mask.
- Confirm the default gateway matches the router interface for that subnet.
- Check whether the gateway interface is up and using the correct address.
- Test local connectivity before testing remote connectivity.

A practical troubleshooting flow is to move from the edge inward: host settings, local gateway reachability, then remote service access. This keeps you from changing multiple variables at once.

**Common reachability symptoms**

When a route or gateway is missing, the system may report that the network is unreachable or fail to connect to remote services. These symptoms point to a routing-table or gateway problem rather than an application-only issue.

- `Network is unreachable` usually means the system cannot find a route to the destination network.
- A failed `ping` to a remote address can indicate a missing route or unreachable gateway.
- `ssh` or `curl` failures can reflect the same underlying path problem.
- DNS, DHCP, and ACLs can also block service access even when basic IP settings are correct.

> If local communication works but remote communication fails, focus on the gateway, routing, and any policy controls before assuming the service itself is down.

**Structured troubleshooting approach**

1. Verify the host addressing information against the documentation.
2. Test reachability to the default gateway.
3. Check whether the gateway interface and routing path are correct.
4. Test access to the remote network or service.
5. Document the fix and retest the original failure.

This sequence helps you separate a bad host configuration from a broader routing or service issue. It also gives you a repeatable method for exam labs and real troubleshooting.

**Related infrastructure checks**

Service reachability problems are not always caused by the gateway alone. In multi-service environments, other infrastructure components can interrupt traffic flow.

- **Routing** problems can prevent a branch or host from reaching another network.
- **DHCP** issues can stop a client from receiving valid addressing information.
- **DNS** problems can make a service appear unreachable by name even when the IP path works.
- **ACLs** can block traffic to internal services.

When you troubleshoot, validate each layer in order. First confirm addressing, then routing, then service-specific dependencies. That layered approach is especially useful in segmented networks with centralized services.

**Quick comparison of likely causes**

| Symptom | Likely area to check | Practical focus |
|---|---|---|
| Local hosts reachable, remote hosts fail | Default gateway | Gateway IP, subnet match, interface status |
| `Network is unreachable` | Routing table or gateway | Route presence, next hop reachability |
| IP works, name fails | DNS | Resolver settings and server reachability |
| Address not assigned | DHCP | Lease, relay, and server availability |
| Service blocked selectively | ACL | Permit/deny rules on the path |

**Key takeaways**

- A host needs correct IP settings and a valid default gateway to reach other networks.
- The gateway must belong to the same subnet as the host.
- Troubleshooting works best when you verify documentation, isolate the fault, and test after each change.
- Reachability issues can come from routing, DHCP, DNS, or ACLs, not just the gateway.
