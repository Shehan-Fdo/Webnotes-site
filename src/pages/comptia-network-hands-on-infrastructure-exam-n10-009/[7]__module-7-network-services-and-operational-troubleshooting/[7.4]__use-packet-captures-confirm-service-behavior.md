## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Why packet captures matter**

Packet captures let you verify what a service actually does on the wire instead of relying on assumptions or logs alone. In this topic, the evidence centers on <abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> traffic and Wireshark analysis, where you confirm the client-server exchange, inspect requested options, and compare them with the server’s reply.

DHCP is a client/server protocol used to dynamically assign IP-address parameters and other settings to a client. Because it runs over BOOTP, packet captures are a practical way to observe service behavior during troubleshooting.

- Use a capture file or live capture when you need proof of service flow.
- Filter to the protocol you care about so you can focus on relevant frames.
- Compare client requests with server responses to spot missing or unexpected behavior.

**Filtering and isolating DHCP traffic**

The first step is to narrow the capture to DHCP packets in Wireshark. The lab evidence uses a downloaded capture file and asks you to apply a filter that shows only DHCP traffic. This keeps the analysis focused and makes the handshake easier to follow.

A packet capture is most useful when you can quickly isolate the conversation for a specific client. In the lab, the client is identified by MAC address `00:50:56:00:9f:8e`, which lets you trace one device through the exchange.

- Open the `.pcap` file in Wireshark.
- Apply a display filter for DHCP traffic.
- Identify the client by MAC address when multiple devices appear in the capture.

**Reading the DHCP DORA sequence**

DHCP service behavior is commonly confirmed by the DORA sequence: <mark>Discover</mark>, <mark>Offer</mark>, <mark>Request</mark>, and <mark>Acknowledge</mark>. The capture should show the client asking for configuration, the server offering settings, the client selecting an offer, and the server acknowledging the lease.

This sequence is the clearest sign that the service is functioning normally. If one step is missing, you can narrow the issue to the client, the server, or the path between them.

- `DHCPDISCOVER` shows the client looking for a server.
- `DHCPOFFER` shows the server proposing configuration.
- `DHCPREQUEST` shows the client accepting an offer.
- `DHCPACK` shows the server confirming the lease.

**Inspecting requested and offered options**

The lab specifically points you to DHCP <abbr title="Parameter Request List">PRL</abbr> details in Option 55. That option shows which settings the client asks for in its `DHCPDISCOVER` packet. You then inspect the `DHCPOFFER` packet to see which options the server returns.

This comparison helps you confirm whether the server is supplying the values the client requested. It is also a practical way to verify service behavior when a client receives an address but still lacks some configuration detail.

- Expand the `DHCPDISCOVER` packet.
- Open Option 55 to view the Parameter Request List.
- Compare the requested options with the options in `DHCPOFFER`.

**Using packet capture in troubleshooting workflows**

Packet capture is not limited to DHCP, but DHCP is a clean example of how to validate service behavior. The Meraki documentation shows a workflow where you capture traffic from the relevant switch port or uplink, save the capture as a `.pcap` file, and open it in Wireshark for analysis. That approach is useful when a client cannot obtain IP information and you need to see where the handshake stops.

The practical value is simple: you verify whether traffic reaches the server, whether the server responds, and whether the client completes the exchange. If the capture shows only part of the handshake, you know where to continue troubleshooting.

1. Capture traffic from the interface that carries the client’s DHCP exchange.
2. Save the capture as a `.pcap` file.
3. Open the file in Wireshark.
4. Filter for DHCP and trace the DORA sequence.
5. Check requested options and server replies for mismatches.

> If the client and DHCP server are on the same switch, capture on the client’s port rather than the uplink so you do not miss local traffic.

**What to look for in a healthy capture**

A healthy DHCP capture shows a complete exchange with matching request and response behavior. You should be able to identify the client, follow the four DORA messages, and confirm that the server returns the expected options.

- The client sends a `DHCPDISCOVER`.
- The server replies with a `DHCPOFFER`.
- The client sends a `DHCPREQUEST`.
- The server finishes with a `DHCPACK`.
- Option 55 in the discover packet shows the client’s requested settings.
- The offer packet shows the server’s returned settings.

**Key takeaway**

Packet captures confirm service behavior by showing the actual protocol exchange. For DHCP, that means tracing DORA, checking Option 55, and comparing client requests with server responses in Wireshark.
