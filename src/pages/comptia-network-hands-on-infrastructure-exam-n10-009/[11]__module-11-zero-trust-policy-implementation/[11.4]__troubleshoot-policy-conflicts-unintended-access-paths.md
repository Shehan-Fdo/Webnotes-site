## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Policy conflicts in modern access control**

Modern networks no longer rely on a single perimeter. Users, devices, and applications move across cloud, on-premises, and remote environments, so access control depends on identity, context, and continuous verification rather than location alone. In this model, policy conflicts often appear when different controls do not agree on who should reach a resource or how traffic should flow.

Zero Trust means you do not assume trust based on network position. Instead, you verify each access attempt and limit access to only what is necessary. When policies are inconsistent, you can accidentally create broader access than intended or block legitimate traffic.

- **Zero Trust** evaluates each request in real time.
- **Least privilege** limits access to only required resources.
- **Continuous verification** helps reduce reliance on implicit trust.
- Distributed environments make perimeter-only thinking unreliable.

> <mark>Unintended access paths usually come from policy overlap, missing segmentation, or rules that are broader than the intended trust boundary.</mark>

**How unintended access paths happen**

Microsegmentation divides a network into smaller zones so you can apply security controls more granularly. This helps reduce the attack surface and limit lateral movement, but it also increases the chance of configuration mistakes if you do not map traffic carefully.

A common failure point is assuming that a rule set is enforcing isolation when it is actually allowing a path between zones. Another risk is manual policy creation at scale, which can lead to misconfigurations and rule sprawl. If you do not understand application dependencies and traffic flows, you may create policies that conflict with the intended design.

- **Microsegmentation** creates smaller security boundaries around assets or services.
- **Zone-to-zone policy** must match the intended communication pattern.
- **Manual rules** can become difficult to maintain as environments grow.
- **Application dependencies** must be mapped before you enforce restrictions.

**Troubleshooting approach**

When you troubleshoot policy conflicts, start by comparing the intended access model with the actual traffic path. You want to confirm whether the issue is caused by an overly permissive rule, an overly restrictive rule, or a missing control between segments.

1. Identify the resource that should be protected.
2. Map the traffic flow to and from that resource.
3. Compare the observed path with the intended policy.
4. Check for overlapping rules, broad exceptions, or missing segmentation.
5. Adjust the policy so it matches the least-privilege design.

This process is especially important in distributed environments where users and workloads move across multiple locations. A policy that works in one zone may create an unintended path in another if the same trust assumptions do not apply.

**Practical signs of a policy conflict**

You can often spot a conflict by looking for behavior that does not match the security design. For example, traffic may reach a protected service even though the segment should be isolated, or a legitimate application may fail because a required dependency is blocked.

- **Unexpected reachability** suggests a rule is too broad.
- **Blocked legitimate traffic** suggests a rule is too strict or incomplete.
- **Lateral movement** indicates trust is extending farther than intended.
- **Rule sprawl** makes it harder to see which policy is actually active.

**Why Zero Trust and segmentation matter together**

Zero Trust focuses on verifying identity and privilege for every access attempt. Segmentation and microsegmentation enforce that model by shrinking implicit trust zones and limiting communication paths. Together, they help contain breaches and reduce the chance that an attacker can move laterally after gaining access.

The practical goal is not just to deny access, but to make sure every allowed path is intentional. If a policy conflict creates an unintended path, the environment no longer reflects the least-privilege design you meant to enforce.

**Key takeaways**

- Verify access based on identity, context, and privilege, not location.
- Use segmentation to narrow communication paths between assets.
- Map traffic flows before enforcing restrictive policies.
- Watch for overlap, broad exceptions, and manual rule errors.
- Treat unexpected access as a sign that the policy model and the actual path do not match.
