## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**SDN controller role**

Software-defined networking (SDN) separates the control logic from the hardware that forwards traffic. In a lab, the controller gives you a centralized place to manage network behavior instead of configuring each device independently.

- **Control plane** decisions move into software.
- **Data plane** devices still forward traffic.
- **Management plane** becomes easier to coordinate from one controller.

A controller is useful when you want to model virtualized networking behavior and practice how centralized policy affects the network. OpenDaylight is one example of an SDN controller platform used in lab and OpenStack-related environments.

**OpenDaylight overview**

OpenDaylight is a programmable SDN controller written in Java. The documentation describes it as a platform with separated northbound and southbound interfaces, which supports network virtualization use cases.

- It can act as a drop-in replacement in some OpenStack networking scenarios.
- It supports lab work where you want to explore controller-based networking.
- It is distributed as a Karaf-based installation.

> Use conservative lab assumptions: verify the Java Runtime Environment (JRE), operating system support, and any known issues before you install.

**Installation flow**

The OpenDaylight installation guide organizes setup into a few practical stages. In a lab, you should follow the same sequence so you can isolate problems early.

1. Check the prerequisites for your environment.
2. Download the OpenDaylight distribution.
3. Run the Karaf distribution.
4. Install the required Karaf features.
5. Verify the installation.
6. Remove features or uninstall if you need to reset the lab.

The documentation also notes that some features are simple enough to install directly in the Karaf distribution, while more complex projects may need a dedicated installation guide.

**Karaf distribution basics**

Karaf is the runtime used for OpenDaylight features. In practice, this means you start the distribution, then add the features you need for the lab scenario.

- Start with the default distribution from the OpenDaylight downloads.
- Use Karaf features to add functionality.
- Keep the install minimal until you confirm the controller starts correctly.

```text
OpenDaylight
└── Karaf distribution
    ├── start runtime
    ├── install features
    ├── verify services
    └── uninstall features if needed
```

**Verification and troubleshooting**

After installation, confirm that the controller is running before you build the rest of the lab topology. If something fails, work backward through the install sequence.

- Confirm the runtime starts cleanly.
- Check that the expected features are available.
- Revisit JRE and OS compatibility if startup fails.
- Use uninstall steps to reset the environment when needed.

**Lab takeaways**

This topic is less about memorizing a long command list and more about understanding the controller workflow. You deploy the SDN controller, verify the runtime, and then use it as the centralized control point for virtualized networking exercises.

- SDN centralizes control logic in software.
- OpenDaylight is a Java-based controller platform.
- Karaf features extend the base installation.
- Verification matters before you move on to lab integration.
