## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Overview**

In software-defined networking (SDN), you separate network functions from the physical device and manage them through software. OpenDaylight is a controller platform that exposes APIs and uses southbound interfaces and protocol plugins to communicate with devices and services. For exam prep, focus on the idea that a controller can register and manage real nodes or work with simulated endpoints in a lab so you can test workflows without relying on production hardware.

- **OpenDaylight APIs** provide the northbound interface used to communicate with higher-level services.
- The northbound API is primarily based on **REST** and is rendered through the Model-Driven Service Abstraction Layer.
- **RESTCONF** is used to present REST APIs according to YANG models defined by applications.
- Southbound interfaces and protocol plugins connect the controller to managed devices and services.

**Managed Nodes and Simulated Endpoints**

A managed node is a device or service that the controller can reach and operate on through the southbound layer. A simulated endpoint is a lab-friendly stand-in that lets you practice the same registration and management flow without a live device. The practical goal is the same in both cases: establish a controller-managed target that can be queried, configured, and monitored.

- Use a managed node when you have a real OVSDB-capable device or service to control.
- Use a simulated endpoint when you need a safe test target for learning or validation.
- Registration is part of the controller workflow that makes the endpoint visible to the management plane.
- The controller architecture supports both active and passive connection patterns in OVSDB workflows.

> Keep the distinction practical: the exam focus is on how the controller interacts with endpoints, not on memorizing vendor-specific implementation details.

**Controller Architecture Context**

OpenDaylight’s architecture helps explain why registration matters. The controller sits between applications and the network, translating application intent into device-level actions. In the OVSDB ecosystem, the southbound plugin handles communication with OVSDB nodes, while the controller’s services manage configuration and state.

- **AAA** stands for Authentication, Authorization, and Accounting.
- The Model-Driven Service Abstraction Layer, or **MD-SAL**, supports model-driven operations.
- OVSDB workflows include connection service, network configuration service, and JSON-RPC communication.
- The controller can manage bridges, ports, QoS, and queue-related objects through the OVSDB plugin.

**Typical Registration Workflow**

Use a structured workflow when you bring a node or endpoint under management. The exact steps vary by platform, but the sequence below reflects the practical controller pattern shown in the evidence.

1. Connect the node or endpoint to the controller using the appropriate southbound path.
2. Confirm the controller can identify the target as an OVSDB node or similar managed object.
3. Register the target so it becomes available for configuration and operational queries.
4. Verify that the controller can query the node and apply changes through the management layer.

**What You Should Be Able to Recognize**

For exam purposes, you should recognize the role of each layer in the SDN stack and how registration fits into that stack. The controller does not replace the endpoint; it coordinates it.

- **Northbound** communication serves applications and higher-level services.
- **Southbound** communication reaches devices, nodes, and protocol-specific plugins.
- **MD-SAL** bridges model-driven intent and operational behavior.
- OVSDB support includes both user-initiated and switch-initiated connection workflows.

**Practical Study Takeaways**

When you review this topic, tie the concept of registration to controller visibility and management. If a node is registered, the controller can treat it as part of the managed environment. If an endpoint is simulated, the same management concepts still apply, which makes it useful for hands-on practice and troubleshooting.

- Focus on the relationship between controller, API, and southbound plugin.
- Remember that REST-based northbound access is separate from device-facing communication.
- Treat simulated endpoints as training tools for understanding the same management flow.
- Use the architecture to explain why a node must be registered before it can be managed effectively.
