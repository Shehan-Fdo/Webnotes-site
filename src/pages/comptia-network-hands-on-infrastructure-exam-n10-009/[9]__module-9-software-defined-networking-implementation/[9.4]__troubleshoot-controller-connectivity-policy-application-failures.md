## **Course:** CompTIA Network+ Hands-On Infrastructure (Exam N10-009)

**Controller connectivity and policy failures: what to check first**

In software-defined networking (SDN), the controller centralizes control logic, so a failure can affect multiple devices or policies at once. When connectivity or policy application breaks, start by confirming the controller, the managed device, and the policy path are all healthy before changing configuration.

- **Controller logs** often reveal the first useful clue.
- **Policy scope** matters: a failure may affect one service, one device, or the whole fabric.
- **Feature state** should match the intended controller function before deeper troubleshooting.

**Log-first troubleshooting**

OpenDaylight guidance emphasizes checking logs early. OpenDaylight stores logs in `/var/log/containers/opendaylight`, and the newest log is `karaf.log`. Older logs use incremental names such as `karaf.log.1` and `karaf.log.2`.

When OpenStack networking commands fail, examine the neutron logs first. The `server.log` file in `/var/log/containers/neutron` can include errors about communication with OpenDaylight. If the neutron error comes from interacting with OpenDaylight, you also need the OpenDaylight logs to find the cause.

```text
/var/log/containers/opendaylight/karaf.log
/var/log/containers/neutron/server.log
```

- Check the most recent controller log before older rotated logs.
- Correlate controller errors with the network service log that triggered the failure.
- Look for communication errors, feature load issues, or policy processing messages.

**Validate controller features and plugin state**

For OpenDaylight PCEP (Path Computation Element Protocol), troubleshooting starts with confirming that required features are installed and that local PCE and remote PCC configuration is correct. From the Karaf console, `feature:list -i` shows installed features.

If the issue persists, use `log:tail` or inspect `data/log/karaf.log` to find hints. You can also raise the logger severity to `DEBUG` for deeper detail.

```text
feature:list -i
log:tail
log:set DEBUG org.opendaylight.protocol.pcep
```

- Confirm the required feature set is present before chasing downstream symptoms.
- Verify controller-side and remote-side configuration together.
- Increase logging only when you need more detail for a focused investigation.

**Check for connector and access problems**

A failed NETCONF connector on an OpenDaylight controller points to a controller-to-device communication problem. In practice, you verify that the controller can reach the device and that the connector settings are correct before assuming the policy itself is broken.

OpenDaylight installation troubleshooting also shows that setup failures can come from access restrictions or missing dependencies. For example, if a database setup fails, ownership of `/usr/local/vtn/` may block `db_setup`, and PostgreSQL packages or binaries may be missing.

> If a setup or connector task fails early, check permissions, ownership, and required services before retrying the workflow.

- Confirm directory ownership and access for the service account.
- Verify required database tools and packages are present.
- Treat early setup failures as infrastructure issues until logs show otherwise.

**Policy application failures in ACI**

For Cisco Application Centric Infrastructure (ACI), policy troubleshooting focuses on whether the intended policy is actually programmed and enforced. Security policy and policy-based redirect issues require you to trace the policy path, not just the endpoint symptoms.

Useful checks include:

- Whether the policy object exists and is attached where expected.
- Whether the service graph or redirect path is configured correctly.
- Whether the policy is being applied to the right traffic flow.

Policy failures often look like connectivity problems, but the root cause may be a missing or mismatched policy association rather than a transport outage.

**Practical troubleshooting flow**

1. Confirm the symptom and identify whether it affects one endpoint, one service, or multiple devices.
2. Check the controller logs first, then the related network service logs.
3. Verify required features, plugins, and controller-side configuration.
4. Confirm access, ownership, and dependency health for setup-related failures.
5. Review the policy path and attachment points if the issue is policy-specific.
6. Increase logging only after you narrow the likely failure area.

**What to remember for exam prep**

- SDN controllers centralize control, so one failure can affect many devices.
- Logs are the fastest path to a root cause.
- `karaf.log` is the primary OpenDaylight log to inspect.
- `feature:list -i` helps confirm installed controller features.
- Policy failures can come from incorrect attachment, not just broken connectivity.
- Early setup errors often point to permissions or missing dependencies.
