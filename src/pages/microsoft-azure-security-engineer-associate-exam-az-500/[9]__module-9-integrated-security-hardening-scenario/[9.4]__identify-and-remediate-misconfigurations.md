## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Misconfiguration management in Azure**

Misconfigurations are one of the most common causes of cloud exposure. In Azure, you typically identify them through policy compliance, security posture tools, logs, and service-specific configuration checks, then remediate them by changing settings, enforcing guardrails, or deploying secure defaults.

A strong AZ-500 approach separates the work into four phases:

- **Detection**: find the issue with policy, Defender for Cloud, logs, or service diagnostics.
- **Validation**: confirm the issue is real and understand its blast radius.
- **Containment**: reduce exposure quickly, often by disabling public access or tightening scope.
- **Permanent remediation**: enforce the secure configuration so the issue does not return.

**Common sources of misconfiguration findings**

Azure gives you multiple ways to discover insecure settings. Use more than one source because each catches different classes of problems.

- **Azure Policy** evaluates resources against rules and can mark them compliant or noncompliant.
- **Microsoft Defender for Cloud** surfaces security recommendations and secure score improvements.
- **Activity logs** show who changed what and when.
- **Resource-specific diagnostics** reveal service-level issues such as public endpoints, weak TLS, or missing encryption.
- **Access reviews and identity reports** help identify stale privileged access and risky role assignments.

**Azure Policy fundamentals**

Azure Policy is the primary control for detecting and preventing configuration drift. It evaluates resources against policy definitions and reports compliance at the scope where the assignment exists.

Key concepts:

- **Policy definition**: the rule that describes allowed or disallowed configuration.
- **Initiative**: a collection of related policy definitions.
- **Assignment**: the application of a policy or initiative to a scope.
- **Scope**: management group, subscription, resource group, or resource.
- **Compliance state**: whether a resource matches the policy rule.

Policy evaluation is inherited downward from the assignment scope. If you assign a policy at the management group level, child subscriptions and resource groups inherit that assignment unless excluded. This is important for exam scenarios because a single assignment can govern many resources, but remediation still happens at the affected resource level.

Policy compliance is not always instant. New resources are evaluated during creation, and existing resources are evaluated on a periodic cycle or when you trigger a scan. If you change a policy, allow time for reevaluation or start a manual scan.

**Useful Azure Policy CLI patterns**

Use the Azure CLI to inspect assignments, definitions, and compliance state.

```bash
# List policy assignments at a subscription scope
az policy assignment list --scope /subscriptions/<subscription-id>

# Show a specific assignment
az policy assignment show --name <assignment-name> --scope /subscriptions/<subscription-id>

# List built-in policy definitions
az policy definition list --query "[?policyType=='BuiltIn'].[displayName,name]" -o table

# Trigger a compliance scan
az policy state trigger-scan --subscription <subscription-id>
```

For compliance reporting, prefer the portal or CLI queries that are known to work in your environment. OData-style filters can vary by command version and API behavior, so validate the exact syntax before relying on it in automation.

> Policy state queries are useful for investigation, but exact filter support can differ across CLI versions and API surfaces. Verify commands in your tenant before using them in runbooks.

**Remediation with Azure Policy**

Some policies are only audit controls. Others can enforce or deploy settings.

- **Audit**: reports noncompliance without changing the resource.
- **Deny**: blocks noncompliant creation or update requests.
- **Modify**: changes request properties before deployment.
- **DeployIfNotExists**: deploys a related resource or configuration when the target resource lacks it.

For remediation tasks, the policy assignment usually needs a managed identity and the identity must have sufficient permissions. This is especially important for `deployIfNotExists` and `modify` effects.

Typical remediation flow:

1. Assign the policy or initiative at the correct scope.
2. Enable a managed identity on the assignment if the effect requires deployment or modification.
3. Grant the identity the required role, such as Contributor or a narrower custom role.
4. Trigger a compliance scan.
5. Create a remediation task for existing noncompliant resources.

Example assignment with identity:

```bash
az policy assignment create \
  --name enforce-storage-https \
  --scope /subscriptions/<subscription-id> \
  --policy <policy-definition-id> \
  --identity-system-assigned
```

Example remediation task:

```bash
az policy remediation create \
  --name remediate-storage-https \
  --policy-assignment <assignment-id> \
  --resource-discovery-mode ReEvaluateCompliance
```

Use remediation only when the policy effect supports it and the assignment identity has the right permissions. Otherwise, the task may fail or remain incomplete.

**Defender for Cloud and secure score**

Microsoft Defender for Cloud is a major source of misconfiguration detection. It aggregates security recommendations, secure score, and regulatory compliance views to help you prioritize fixes.

Why it matters:

- **Secure score** gives you a measurable improvement target.
- **Recommendations** identify insecure settings such as open management ports, missing endpoint protection, or weak storage settings.
- **Attack path analysis** can show how multiple misconfigurations combine into a real exposure.
- **Regulatory compliance** maps findings to standards and controls.

Use Defender for Cloud when you need to prioritize remediation by risk, not just by count of findings. A low-effort fix that closes a high-risk exposure should usually come before a cosmetic hardening change.

Common workflow:

1. Open the recommendation.
2. Review affected resources and the remediation steps.
3. Validate whether the finding is a true positive.
4. Apply the fix manually or through policy.
5. Recheck secure score and recommendation status.

**Identity misconfigurations**

Identity issues often create the largest blast radius because they affect access across many services.

Common problems:

- Overly broad role assignments such as `Owner` or `Contributor` at subscription scope.
- Permanent privileged access instead of just-in-time elevation.
- Missing MFA (multifactor authentication) for privileged users.
- Weak Conditional Access policies.
- Stale guest accounts or inactive service principals.
- Excessive app permissions or unmanaged secrets.

Practical remediation patterns:

- Use **Privileged Identity Management (PIM)** for just-in-time role activation.
- Require MFA for privileged roles and sensitive applications.
- Scope role assignments to the smallest practical scope.
- Review service principal credentials and rotate secrets or certificates.
- Remove unused accounts and expired guest access.

Example Azure CLI checks:

```bash
# List role assignments at subscription scope
az role assignment list --scope /subscriptions/<subscription-id> -o table

# Show PIM-related role eligibility is usually done in Entra ID tools or portal
# but you can still inspect active role assignments with Azure CLI.
```

For Conditional Access and PIM, portal-based remediation is often the most direct path because these controls live in Microsoft Entra ID rather than in a single Azure resource group.

**Storage account misconfigurations**

Storage accounts are frequent targets because they often contain sensitive data and are easy to expose accidentally.

Common issues:

- Public blob access enabled when it should be disabled.
- Shared key access left enabled when you want to enforce Microsoft Entra ID authorization.
- HTTPS not enforced.
- Minimum TLS version too low.
- Network access too broad.
- Missing private endpoints for sensitive workloads.
- Unrestricted SAS (shared access signature) usage.

Useful checks:

```bash
az storage account show \
  --name <storage-account> \
  --resource-group <resource-group> \
  --query "{publicAccess:allowBlobPublicAccess, httpsOnly:enableHttpsTrafficOnly, minTls:minimumTlsVersion}" \
  -o json
```

Example hardening:

```bash
az storage account update \
  --name <storage-account> \
  --resource-group <resource-group> \
  --https-only true \
  --min-tls-version TLS1_2
```

For public network access, note that support and exact flags can be version-dependent. In newer Azure CLI and API versions, you may be able to disable public network access directly, but validate the command in your environment before using it in automation.

A secure storage posture usually combines:

- HTTPS only
- TLS 1.2 or later
- Private endpoints for data-plane access
- Network rules that allow only trusted subnets
- Azure AD-based authorization where possible
- Key rotation and SAS governance

**Key Vault misconfigurations**

Key Vault often stores secrets, keys, and certificates, so exposure here is high impact.

Common issues:

- Public network access enabled when private access is required.
- Firewall rules too permissive.
- Soft delete or purge protection not enabled.
- Access policies or RBAC assignments too broad.
- Secrets not rotated.
- Diagnostic logging not enabled.

Example inspection:

```bash
az keyvault show \
  --name <key-vault-name> \
  --resource-group <resource-group> \
  --query "{publicNetworkAccess:properties.publicNetworkAccess, softDelete:properties.enableSoftDelete, purgeProtection:properties.enablePurgeProtection}" \
  -o json
```

Example hardening pattern:

```bash
az keyvault update \
  --name <key-vault-name> \
  --resource-group <resource-group> \
  --public-network-access Disabled
```

> Public network access support depends on the Azure CLI and service API version. If the flag is unavailable, use the portal or ARM/Bicep with a supported API version.

Best practices:

- Prefer **RBAC** or tightly scoped access policies.
- Use private endpoints for sensitive vaults.
- Enable purge protection to reduce accidental or malicious deletion risk.
- Turn on logging and monitor secret access patterns.

**Compute and VM misconfigurations**

Virtual machines and scale sets often drift from secure baselines after deployment.

Common issues:

- Open inbound management ports such as RDP or SSH.
- Missing endpoint protection or vulnerability management.
- Outdated OS patches.
- Weak disk encryption posture.
- Unnecessary public IP addresses.
- Excessive local administrator access.

Remediation options:

- Remove public IPs from management interfaces.
- Use Azure Bastion or JIT access instead of direct exposure.
- Apply update management and patch baselines.
- Enable disk encryption where required by policy.
- Restrict NSG rules to trusted source ranges.

Example network containment:

```bash
az network nsg rule create \
  --resource-group <resource-group> \
  --nsg-name <nsg-name> \
  --name Deny-RDP \
  --priority 100 \
  --direction Inbound \
  --access Deny \
  --protocol Tcp \
  --source-address-prefixes '*' \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 3389
```

Use containment first when a VM is exposed, then follow with a permanent fix such as Bastion, JIT, or a private management network.

**Container and Kubernetes misconfigurations**

Container and Kubernetes environments are common sources of misconfiguration because they combine identity, networking, and workload security.

Common issues:

- Publicly exposed cluster API servers.
- Overly permissive Kubernetes RBAC.
- Pods running as root.
- Privileged containers or hostPath mounts.
- Missing network policies.
- Images pulled from untrusted registries.
- Secrets stored in plain manifests.

Remediation patterns:

- Restrict API server access to trusted networks.
- Use Azure RBAC for Kubernetes where appropriate.
- Apply Pod Security standards or equivalent admission controls.
- Run containers as non-root.
- Limit capabilities and avoid privileged mode.
- Use private registries and image scanning.
- Store secrets in Key Vault and inject them securely.

Example Kubernetes security context:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  containers:
    - name: app
      image: myregistry.azurecr.io/app:1.0
      securityContext:
        runAsNonRoot: true
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
```

For AKS, combine cluster-level policy, namespace controls, and workload-level hardening. A single insecure deployment can bypass otherwise strong platform settings.

**SQL and database security misconfigurations**

Databases often fail security reviews because network access and authentication are too open.

Common issues:

- Public endpoints enabled without need.
- Firewall rules allowing broad IP ranges.
- Weak authentication or local admin use.
- Transparent Data Encryption not enabled where required.
- Auditing and threat detection disabled.
- Missing private endpoints.

Remediation patterns:

- Use private endpoints and disable public access where possible.
- Restrict firewall rules to known application subnets.
- Enable auditing and vulnerability assessment.
- Use Microsoft Entra authentication where supported.
- Enforce encryption at rest and in transit.

When a database is exposed, first contain by narrowing network access, then validate application connectivity, then move to a permanent private-access design.

**Backup and recovery misconfigurations**

Backup settings are often overlooked until a restore is needed.

Common issues:

- Backups disabled or not scheduled.
- Retention too short for recovery objectives.
- Backup vault access too broad.
- Restore testing not performed.
- Immutable or soft-delete protections not enabled where available.

Remediation priorities:

- Enable backups for critical workloads.
- Align retention with business recovery requirements.
- Test restore procedures regularly.
- Protect backup vaults with least privilege and network controls.
- Monitor backup job failures and restore points.

A backup that cannot be restored is a configuration failure, not a protection control.

**Portal-based remediation workflow**

Portal workflows are often the fastest way to fix a live issue.

1. Open the finding in Defender for Cloud or Azure Policy.
2. Review the affected resource and the recommended fix.
3. Check whether the issue is caused by inheritance, a direct setting, or a deployment template.
4. Apply the fix at the right layer:
   - resource setting
   - policy assignment
   - template or Bicep deployment
   - identity or access control
5. Re-evaluate compliance and confirm the finding clears.

Use the portal when you need context, especially for identity, policy inheritance, or cross-service recommendations.

**CLI and IaC remediation patterns**

Infrastructure as Code (IaC) helps prevent recurrence. Use Bicep, ARM templates, or Terraform to encode secure defaults.

Example Bicep for a storage account baseline:

```bicep
resource stg 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: 'mystorageacct'
  location: resourceGroup().location
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
  properties: {
    allowBlobPublicAccess: false
    minimumTlsVersion: 'TLS1_2'
    supportsHttpsTrafficOnly: true
  }
}
```

Use IaC when:

- you need repeatable deployments
- you want drift detection through source control
- you need to standardize secure defaults across environments

Use manual remediation when:

- the issue is urgent and live
- the resource is outside your deployment pipeline
- you need to validate the fix before codifying it

**Detection, validation, containment, and permanent remediation**

A disciplined response model helps you avoid making the problem worse.

- **Detection**: Defender for Cloud flags a public storage account.
- **Validation**: confirm whether the account contains sensitive data and whether public access is actually used.
- **Containment**: disable public access or restrict network rules immediately.
- **Permanent remediation**: move access behind private endpoints and enforce the setting with policy.

Another example:

- **Detection**: Azure Policy reports a VM with an open SSH port.
- **Validation**: check whether the port is required for operations.
- **Containment**: close the NSG rule or restrict it to a jump host.
- **Permanent remediation**: replace direct SSH with Bastion and enforce the rule with policy.

This sequence is exam-relevant because the best answer is not always “fix the setting.” Sometimes you must first stop exposure, then redesign the control.

**How to prioritize fixes**

Prioritize by risk, not by convenience.

- Fix internet-exposed management ports first.
- Fix identity issues with tenant-wide impact next.
- Fix data exposure in storage, Key Vault, and databases early.
- Fix compliance-only findings after high-risk exposures.
- Use secure score and attack path analysis to identify the highest-value changes.

A practical prioritization rule:

- If the issue enables unauthorized access, treat it as urgent.
- If the issue weakens defense-in-depth but does not expose data directly, schedule it next.
- If the issue is a baseline drift with low risk, fold it into the next deployment cycle.

**Exam tips**

- Know the difference between **audit**, **deny**, **modify**, and **deployIfNotExists**.
- Remember that policy compliance can lag behind the actual resource change.
- Expect remediation tasks to require identity and permissions when the policy deploys or modifies resources.
- Use Defender for Cloud for security posture and prioritization, not just policy compliance.
- Be ready to identify whether a fix belongs in the resource, the policy, the identity layer, or the deployment pipeline.
- Treat Azure Blueprints as deprecated for new use; prefer Azure Policy and template-based approaches for modern governance.
