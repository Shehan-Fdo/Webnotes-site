## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Scope and security design goals**

This topic ties together the main control planes you use in Azure security engineering: identity authorization, access conditions, threat protection, secret management, and network filtering. In practice, you rarely use one control alone. You combine them to reduce blast radius, enforce least privilege, protect privileged actions, and inspect traffic at the right layer.

A useful mental model is:

- **RBAC** decides *who can do what* on Azure resources.
- **Conditional Access** decides *under what conditions* a user can sign in or access cloud apps.
- **Defender for Cloud** detects risk, recommends hardening, and raises alerts.
- **Key Vault** stores secrets, keys, and certificates with controlled access.
- **Firewall controls** restrict and inspect network flows between users, apps, and the internet.

**Azure RBAC fundamentals**

Azure Role-Based Access Control (RBAC) is the authorization system for Azure resources. It uses role definitions, role assignments, and scopes to determine effective permissions.

- A **role definition** is the permission set, such as `Reader`, `Contributor`, or `Key Vault Secrets User`.
- A **role assignment** binds a principal to a role at a scope.
- A **scope** can be management group, subscription, resource group, or resource.
- **Effective access** is the result of all inherited assignments plus any deny behavior from higher-level controls.

RBAC is the default model for most Azure resource authorization. Use it to grant the minimum permissions needed for a task, and prefer built-in roles before creating custom roles.

**Common RBAC patterns**

- Assign at the narrowest practical scope.
- Use **management groups** for broad governance and subscription inheritance.
- Use **resource group** scope for teams that manage a workload boundary.
- Use **resource** scope for highly sensitive resources such as a single Key Vault.
- Use **custom roles** when built-in roles are too broad or too narrow.

**Built-in roles and custom roles**

Built-in roles cover common tasks, but exam scenarios often require you to reason about privilege boundaries.

- **Reader**: view-only access.
- **Contributor**: manage resources but not grant access.
- **Owner**: full management plus access delegation.
- **User Access Administrator**: manage role assignments.
- **Key Vault Administrator**: manage Key Vault data-plane and management-plane operations in RBAC mode, depending on the exact role and resource configuration.
- **Key Vault Secrets User**: read secrets only.
- **Key Vault Crypto User**: use cryptographic keys for operations without full key management.

Custom roles are useful when you need a precise permission set, such as allowing a support team to restart VMs but not delete them. A custom role is defined by allowed `actions`, `notActions`, `dataActions`, and `notDataActions`.

```json
{
  "Name": "VM Restart Operator",
  "IsCustom": true,
  "Description": "Can restart virtual machines but not delete them.",
  "Actions": [
    "Microsoft.Compute/virtualMachines/read",
    "Microsoft.Compute/virtualMachines/restart/action"
  ],
  "NotActions": [
    "Microsoft.Compute/virtualMachines/delete"
  ],
  "AssignableScopes": [
    "/subscriptions/00000000-0000-0000-0000-000000000000"
  ]
}
```

**Inspecting RBAC and effective permissions**

You often need to verify what a user can actually do.

- In the portal, use **Access control (IAM)** to view role assignments.
- Use **Check access** to inspect a principal’s effective permissions on a scope.
- Use Azure CLI to list assignments and role definitions.

```bash
az role assignment list --assignee user@contoso.com --all -o table
az role definition list --name "Key Vault Secrets User" -o jsonc
az role assignment list --scope /subscriptions/<subId>/resourceGroups/<rgName> -o table
```

For troubleshooting, remember that permissions can come from multiple scopes. A user may appear overprivileged because of inheritance from a subscription or management group.

**Role assignment vs role definition**

- A **role definition** answers: what permissions exist?
- A **role assignment** answers: who gets those permissions, and where?

This distinction matters in exam questions about governance and delegation. If you need to change what a role can do, edit or create a role definition. If you need to grant access, create a role assignment.

**Conditional Access essentials**

Conditional Access (CA) is Microsoft Entra ID policy enforcement for sign-in and token issuance. It evaluates signals before granting access to cloud apps and resources.

CA is not the same as RBAC:

- **RBAC** authorizes actions after identity is authenticated.
- **Conditional Access** controls whether authentication is allowed and under what conditions.

Use CA when you need to require stronger authentication, restrict access by location or device state, or apply session controls.

**Core Conditional Access building blocks**

A CA policy typically includes:

- **Users or workload identities** to target.
- **Cloud apps or actions** to protect.
- **Conditions** such as device platform, client app, sign-in risk, user risk, or location.
- **Grant controls** such as require MFA or compliant device.
- **Session controls** such as app enforced restrictions or sign-in frequency.

**Named locations and location-based policy**

Named locations let you define trusted or blocked IP ranges and countries/regions.

- Use them to exclude corporate egress IPs from stricter prompts when appropriate.
- Use them to block sign-ins from high-risk geographies.
- Use them carefully; IP-based trust is weaker than device compliance or phishing-resistant MFA.

Example use cases:

- Require MFA outside trusted office networks.
- Block legacy authentication from all locations.
- Require compliant devices only when accessing sensitive SaaS apps.

**Session controls and authentication context**

Session controls shape what happens after sign-in.

- **Sign-in frequency** forces reauthentication after a defined interval.
- **Persistent browser session** controls whether sessions survive browser restarts.
- **App enforced restrictions** can limit download or copy behavior in supported apps.
- **Authentication context** lets you require stronger controls for specific sensitive actions, such as accessing a finance app or privileged portal workflow.

Authentication context is useful when one app contains both normal and highly sensitive operations. You can require step-up authentication only for the sensitive path.

**CA for Azure management vs other cloud apps**

In exam scenarios, distinguish between:

- **Microsoft Azure Management**: protects access to Azure portal, Azure CLI sign-in, Azure PowerShell, and ARM-based management actions.
- **Other cloud apps**: protects SaaS applications such as Microsoft 365 or third-party enterprise apps.

If you want to secure administrative access to Azure resources, target the **Microsoft Azure Management** cloud app. If you want to secure a business app, target that app directly.

**Practical Conditional Access example**

A common secure baseline is:

1. Require MFA for all users.
2. Exclude break-glass accounts from CA policies.
3. Require compliant or hybrid joined devices for privileged apps.
4. Block legacy authentication.
5. Add named locations for trusted corporate networks.
6. Use authentication context for especially sensitive actions.

Portal workflow:

1. Go to **Microsoft Entra ID**.
2. Open **Security** > **Conditional Access**.
3. Create a new policy.
4. Select users, cloud apps, conditions, and grant/session controls.
5. Enable the policy in **Report-only** first when validating impact.

CLI and automation are possible through Microsoft Graph, but portal configuration is often the clearest exam reference.

**Defender for Cloud overview**

Microsoft Defender for Cloud is the cloud security posture management and workload protection service for Azure and hybrid environments. It provides:

- **Recommendations** to improve security posture.
- **Security alerts** for suspicious or malicious activity.
- **Secure score** to measure posture improvement.
- **Regulatory compliance** views for control mapping.
- **Workflow automation** to trigger actions from recommendations or alerts.

Think of it as the service that tells you what to fix and what is actively being attacked.

**Secure score and recommendations**

Secure score helps you prioritize hardening tasks. Recommendations are actionable items such as:

- Enable endpoint protection.
- Turn on vulnerability assessment.
- Restrict public exposure.
- Enable encryption or secure transfer.
- Apply just-in-time access where supported.

Use recommendations to drive remediation work, then verify that the score improves after the change.

**Security alerts and workflow automation**

Alerts indicate suspicious activity, such as:

- Brute-force attempts.
- Malware detection.
- Unusual process execution.
- Exposed management ports.
- Suspicious data access patterns.

Workflow automation can send alerts to:

- Logic Apps
- ITSM systems
- Email or ticketing workflows
- SIEM/SOAR platforms such as Microsoft Sentinel

This is useful when you need to route high-severity alerts to an operations team without manual triage.

**Defender plans and workload coverage**

Defender for Cloud includes workload-specific protections. Exam scenarios may reference:

- **Defender for Servers**: endpoint and server threat protection, vulnerability assessment, and suspicious activity detection.
- **Defender for Storage**: anomalous access and malicious content signals for storage accounts.
- **Defender for SQL**: threat detection and vulnerability assessment for SQL resources.
- **Defender for Containers**: container image and runtime security signals.

The exact feature set depends on the plan and resource type, so avoid assuming every alert or recommendation applies to every workload.

**Regulatory compliance**

The regulatory compliance dashboard maps your environment to frameworks and controls. Use it to:

- Track control coverage.
- Identify gaps against standards.
- Support audit evidence collection.

This is especially useful when a scenario asks how to demonstrate governance rather than just technical hardening.

**Key Vault security model**

Azure Key Vault stores:

- **Secrets**: passwords, connection strings, API keys.
- **Keys**: cryptographic keys for encryption, signing, and wrapping.
- **Certificates**: X.509 certificates and their private keys.

Key Vault is central to secret hygiene because it removes secrets from code, config files, and pipelines.

**RBAC vs access policies in Key Vault**

Key Vault supports two authorization models:

- **Azure RBAC** for unified Azure authorization.
- **Access policies** for vault-specific data-plane permissions.

Azure RBAC is the recommended model for new deployments. Access policies still appear in some scenarios because of migration constraints, feature compatibility, or existing operational patterns.

Important exam nuance:

- Do not assume access policies are only for legacy vaults.
- Do assume Azure RBAC is the preferred starting point unless the scenario explicitly requires access policy behavior.

**Enabling Key Vault RBAC**

To use Azure RBAC for a vault:

1. Open the Key Vault in the portal.
2. Go to **Access configuration**.
3. Set permission model to **Azure role-based access control**.
4. Assign appropriate roles at the vault scope or higher scope.

CLI example:

```bash
az keyvault update \
  --name kv-prod-001 \
  --resource-group rg-security \
  --enable-rbac-authorization true
```

Then assign a role:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<subId>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-prod-001
```

**Key Vault access patterns**

Use the right identity pattern for the workload:

- **Managed identities** for Azure resources such as App Service, Functions, VMs, and Logic Apps.
- **Service principals** for automation and CI/CD when managed identity is not available.
- **User access** for administrators and break-glass workflows.

Managed identities are preferred because they remove credential storage from code and pipelines.

Example: an App Service reading a secret with managed identity

```bash
az webapp identity assign --name app-prod-001 --resource-group rg-app
az role assignment create \
  --assignee <principalId-of-webapp> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<subId>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-prod-001
```

Application code then uses the Azure SDK to request the secret from Key Vault without embedding credentials.

**Key Vault network controls**

Key Vault network settings help restrict where requests can originate.

- **Public network access** can be enabled or disabled depending on the design.
- **Firewall rules** can allow selected IP ranges.
- **Virtual network rules** can allow traffic from selected subnets through service endpoints where supported.
- **Private endpoints** place the vault on a private IP in your virtual network.
- **Trusted Microsoft services** can be allowed in some scenarios, but do not treat this as a blanket bypass for all services.

A private endpoint is usually the strongest network isolation option, but it is not complete by itself. You also need DNS resolution so clients resolve the vault name to the private IP.

```bash
az network private-endpoint create \
  --name pe-kv-prod-001 \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --subnet snet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-prod-001 \
  --group-ids vault \
  --connection-name kv-prod-001-conn
```

Then configure the private DNS zone and link it to the virtual network:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.vaultcore.azure.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.vaultcore.azure.net \
  --name link-vnet-hub \
  --virtual-network vnet-hub \
  --registration-enabled false
```

> A private endpoint without DNS integration often looks “connected” but still fails at name resolution from clients.

**Key Vault soft delete and purge protection**

These features protect against accidental or malicious deletion.

- **Soft delete** retains deleted vault objects for a retention period.
- **Purge protection** prevents permanent deletion until the retention period expires.

Use both for production vaults. This is especially important for secrets and keys that support critical workloads.

Operational implications:

- You can recover deleted items during the retention window.
- Purge protection reduces the risk of destructive admin actions.
- Some governance or compliance scenarios require purge protection to be enabled before production use.

**Key rotation and lifecycle management**

Key rotation reduces exposure if a key is compromised or if policy requires periodic renewal.

- Use rotation policies for keys where supported.
- Rotate secrets on a schedule and update dependent applications.
- Prefer versionless secret references in applications when the platform supports them, so the app can pick up the latest version without code changes.

Example secret retrieval pattern:

```bash
az keyvault secret set \
  --vault-name kv-prod-001 \
  --name SqlPassword \
  --value 'P@ssw0rd-Example!'

az keyvault secret show \
  --vault-name kv-prod-001 \
  --name SqlPassword
```

**Key Vault diagnostics and monitoring**

Enable diagnostic settings to send logs and metrics to Log Analytics, Event Hub, or Storage. Exact categories can vary by resource type and platform changes, so validate the current categories in the portal or with CLI before automating.

Typical monitoring goals:

- Audit secret access.
- Detect denied requests.
- Track key operations.
- Correlate access with identity and source IP.

Example pattern:

```bash
az monitor diagnostic-settings create \
  --name kv-diagnostics \
  --resource /subscriptions/<subId>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-prod-001 \
  --workspace /subscriptions/<subId>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-sec \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Use this as a starting point, but confirm the available categories for the vault and API version in your environment.

**Firewall controls in Azure**

Azure firewalling is layered. Choose the right control for the right layer:

- **NSGs (Network Security Groups)** filter traffic at subnet or NIC level using 5-tuple rules.
- **Azure Firewall** provides centralized stateful filtering, FQDN-based control, threat intelligence, and logging.
- **WAF (Web Application Firewall)** protects HTTP/S applications at Layer 7 against web attacks.
- **Firewall Manager** helps centrally manage Azure Firewall and policy across multiple hubs and secured virtual hubs.

Use NSGs for local segmentation, Azure Firewall for centralized egress and east-west control, and WAF for web application protection.

**When to use each control**

- Use **NSGs** when you need simple subnet or NIC filtering and low operational overhead.
- Use **Azure Firewall** when you need centralized inspection, FQDN filtering, outbound control, and policy consistency.
- Use **WAF** when the workload is a web app exposed over HTTP/S and you need protection against common web exploits.

A common layered design is:

- Internet traffic enters through **WAF**.
- Traffic between subnets or to the internet is filtered by **Azure Firewall**.
- Subnet-level microsegmentation is enforced by **NSGs**.

**Azure Firewall rule types**

Azure Firewall supports different rule collections:

- **Network rules** for IP, port, and protocol-based filtering.
- **Application rules** for FQDN-based outbound HTTP/S filtering.
- **NAT rules** for destination translation and publishing services.

Network rules are appropriate for non-HTTP/S traffic such as TCP and UDP flows. Application rules are appropriate when you want to allow or deny web destinations by domain name.

Example network rule collection:

```json
{
  "name": "net-allow-dns",
  "priority": 100,
  "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
  "action": { "type": "Allow" },
  "rules": [
    {
      "name": "allow-dns",
      "ruleType": "NetworkRule",
      "ipProtocols": [ "UDP" ],
      "sourceAddresses": [ "10.10.1.0/24" ],
      "destinationAddresses": [ "168.63.129.16" ],
      "destinationPorts": [ "53" ]
    }
  ]
}
```

Example application rule collection:

```json
{
  "name": "app-allow-microsoft-update",
  "priority": 200,
  "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
  "action": { "type": "Allow" },
  "rules": [
    {
      "name": "allow-update-sites",
      "ruleType": "ApplicationRule",
      "sourceAddresses": [ "10.10.1.0/24" ],
      "protocols": [
        { "protocolType": "Https", "port": 443 }
      ],
      "targetFqdns": [
        "download.microsoft.com",
        "*.windowsupdate.com"
      ]
    }
  ]
}
```

Use the exact schema expected by the firewall policy API version in your environment. The key exam idea is the distinction between IP/port rules and FQDN-based web rules.

**Azure Firewall Manager**

Azure Firewall Manager centralizes policy and deployment across multiple hubs or secured virtual hubs.

Use it when you need:

- Consistent policy across regions.
- Centralized rule management.
- Hub-and-spoke security architecture.
- Easier governance for multiple firewalls.

This is especially relevant in enterprise designs where each spoke VNet routes outbound traffic through a central inspection point.

**Threat intelligence and logging**

Azure Firewall can use threat intelligence to alert or deny traffic to known malicious destinations.

- **Alert** mode logs suspicious traffic.
- **Deny** mode blocks it.

Logs are typically sent to Log Analytics for analysis. This helps you correlate denied flows with workload behavior and identify misconfigurations or attacks.

**Practical design scenarios**

A few common exam scenarios illustrate how these services work together:

- A developer needs to read a secret from Key Vault:
  - Use a managed identity.
  - Assign `Key Vault Secrets User`.
  - Restrict vault access with private endpoint and DNS.
  - Enable diagnostic logging.

- An admin must access Azure portal only from compliant devices:
  - Use Conditional Access.
  - Target `Microsoft Azure Management`.
  - Require MFA and compliant device.
  - Add named locations for trusted offices if needed.

- A production subscription needs posture improvement:
  - Review Defender for Cloud secure score.
  - Remediate high-impact recommendations first.
  - Enable Defender plans for servers, storage, SQL, or containers as needed.
  - Use workflow automation to route alerts.

- A workload must expose a web app securely:
  - Put WAF in front of the app.
  - Use Azure Firewall for outbound control.
  - Use NSGs for subnet segmentation.
  - Keep Key Vault private and accessed through managed identity.

**Troubleshooting and validation**

When a security control does not behave as expected, validate in layers.

- For **RBAC**, check scope inheritance and role assignments.
- For **Conditional Access**, use report-only mode and sign-in logs.
- For **Defender for Cloud**, inspect recommendations, alerts, and secure score impact.
- For **Key Vault**, verify authorization model, network access, DNS, and private endpoint status.
- For **Firewall**, confirm route tables, rule priority, and whether the traffic matches network or application rules.

Useful commands:

```bash
az role assignment list --assignee <objectId> --all -o table
az keyvault show --name kv-prod-001 --resource-group rg-security --query properties
az network firewall policy rule-collection-group list \
  --firewall-policy-name fwpol-hub \
  --resource-group rg-network -o table
```

**Exam-focused reminders**

- **RBAC** is about authorization on Azure resources; use scopes and effective access to reason about permissions.
- **Conditional Access** is about sign-in conditions; use named locations, MFA, device compliance, and session controls.
- **Defender for Cloud** is about posture and threat detection; use secure score, recommendations, alerts, and workflow automation.
- **Key Vault** should usually use Azure RBAC, managed identities, soft delete, purge protection, and private networking where appropriate.
- **Firewall design** should distinguish NSGs, Azure Firewall, WAF, and Firewall Manager based on the traffic layer and governance need.
