## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Purpose of the sample workload**

A sample workload gives you a controlled environment to validate Azure security controls without risking production systems. In AZ-500 scenarios, you typically deploy a small set of resources that let you test identity, network isolation, logging, encryption, policy, and access review patterns.

The goal is not to build a full application. You want a realistic but minimal topology that supports common security tasks:

- A **virtual network** with segmented subnets
- A **virtual machine** or app tier for access testing
- A **private endpoint** or private access path where applicable
- A **Key Vault** for secrets and keys
- **Monitoring** and diagnostic settings
- **Role-based access control** (RBAC) and managed identity
- Optional **Bastion** for secure administrative access

**Reference architecture for exam practice**

A practical lab layout usually includes:

- A resource group for the workload
- A virtual network with at least two subnets
  - One subnet for compute
  - One subnet for private endpoints or shared services
- A Linux or Windows VM with no public IP
- Azure Bastion for browser-based or native client administration
- Azure Key Vault with access controlled by RBAC or access policies, depending on the scenario
- Log Analytics workspace for centralized logs
- Diagnostic settings on key resources
- Optional storage account for boot diagnostics, artifacts, or test data

This layout helps you practice the security tasks that appear in AZ-500: restricting inbound access, using managed identities, validating logging, and enforcing governance.

**Planning the deployment**

Before you deploy, decide what you are testing. The workload should match the control you want to validate.

- Use a **VM** when you need to test login, disk encryption, NSG rules, or Bastion access.
- Use **Key Vault** when you need to test secret retrieval, RBAC, private access, or audit logging.
- Use **Storage** when you need to test network rules, SAS usage, encryption, or diagnostic logs.
- Use **Policy** when you need to test compliance enforcement and remediation.
- Use **Log Analytics** when you need to verify that telemetry is flowing and queries return expected results.

A good lab design keeps the deployment small and repeatable. You should be able to tear it down and redeploy it quickly.

**Resource group and naming strategy**

Use a dedicated resource group for the sample workload so you can delete everything cleanly after testing. Consistent naming makes it easier to identify dependencies and troubleshoot.

Example naming pattern:

- `rg-az500-lab-eastus`
- `vnet-az500-lab`
- `snet-app`
- `snet-private-endpoints`
- `vm-app01`
- `kv-az500-lab`
- `law-az500-lab`

A simple naming convention reduces mistakes when you assign roles, configure diagnostics, or apply policies.

**Deploying the network foundation**

Start with the virtual network and subnets. Network design matters because many Azure security controls depend on subnet boundaries and private connectivity.

A minimal Bicep example for the network foundation:

```bicep
param location string = resourceGroup().location
param vnetName string = 'vnet-az500-lab'
param appSubnetName string = 'snet-app'
param peSubnetName string = 'snet-private-endpoints'

resource vnet 'Microsoft.Network/virtualNetworks@2023-11-01' = {
  name: vnetName
  location: location
  properties: {
    addressSpace: {
      addressPrefixes: [
        '10.10.0.0/16'
      ]
    }
    subnets: [
      {
        name: appSubnetName
        properties: {
          addressPrefix: '10.10.1.0/24'
        }
      }
      {
        name: peSubnetName
        properties: {
          addressPrefix: '10.10.2.0/24'
          privateEndpointNetworkPolicies: 'Disabled'
        }
      }
    ]
  }
}
```

This snippet is only the network layer. In a real deployment, you add compute, private endpoints, and monitoring resources on top of it.

**Deploying a VM safely**

For security labs, avoid giving the VM a public IP unless the exercise specifically requires it. A safer pattern is to deploy the VM into a private subnet and use Bastion or another controlled access path.

Example Azure CLI pattern for a VM without a public IP:

```bash
az group create \
  --name rg-az500-lab-eastus \
  --location eastus

az network vnet create \
  --resource-group rg-az500-lab-eastus \
  --name vnet-az500-lab \
  --address-prefix 10.10.0.0/16 \
  --subnet-name snet-app \
  --subnet-prefix 10.10.1.0/24

az vm create \
  --resource-group rg-az500-lab-eastus \
  --name vm-app01 \
  --image Ubuntu2204 \
  --size Standard_B2s \
  --vnet-name vnet-az500-lab \
  --subnet snet-app \
  --admin-username azureuser \
  --generate-ssh-keys
```

If you need a Windows VM, use a Windows image and RDP only through Bastion or a jump host:

```bash
az vm create \
  --resource-group rg-az500-lab-eastus \
  --name vm-win01 \
  --image Win2022Datacenter \
  --size Standard_B2s \
  --vnet-name vnet-az500-lab \
  --subnet snet-app \
  --admin-username azureuser \
  --admin-password 'Use-A-Strong-Password-Here!'
```

> Do not assume a VM creation command automatically removes public exposure in every scenario. Verify the resulting NIC and public IP configuration after deployment.

**Using Azure Bastion for secure access**

Azure Bastion lets you connect to VMs over TLS through the Azure portal, and in newer offerings it can also support native client workflows. It is useful when you want to avoid exposing RDP or SSH directly to the internet.

Typical Bastion requirements include:

- A dedicated subnet named `AzureBastionSubnet`
- A public IP for the Bastion host
- A supported SKU and configuration for the features you need

Example network preparation:

```bash
az network vnet subnet create \
  --resource-group rg-az500-lab-eastus \
  --vnet-name vnet-az500-lab \
  --name AzureBastionSubnet \
  --address-prefixes 10.10.3.0/26
```

Example Bastion deployment pattern:

```bash
az network public-ip create \
  --resource-group rg-az500-lab-eastus \
  --name pip-bastion \
  --sku Standard \
  --allocation-method Static

az network bastion create \
  --resource-group rg-az500-lab-eastus \
  --name bastion-az500-lab \
  --vnet-name vnet-az500-lab \
  --public-ip-address pip-bastion \
  --sku Standard
```

Bastion is especially useful in exam labs because it lets you test secure administrative access without opening inbound management ports on the VM.

**Key Vault deployment and access model**

Key Vault is central to many security exercises. You use it to store secrets, keys, and certificates, and to validate access control and audit logging.

Choose the access model based on the scenario:

- Use **RBAC** when you want Azure role assignments to control access.
- Use **access policies** when the lab or legacy design expects policy-based permissions.
- Use **private endpoints** when you want to restrict network access to the vault.

Example Key Vault creation with RBAC authorization:

```bash
az keyvault create \
  --resource-group rg-az500-lab-eastus \
  --name kv-az500-lab \
  --location eastus \
  --enable-rbac-authorization true
```

If you use a managed identity from a VM, assign the role to the identity object ID:

```bash
VM_ID=$(az vm show \
  --resource-group rg-az500-lab-eastus \
  --name vm-app01 \
  --query identity.principalId -o tsv)

az role assignment create \
  --assignee-object-id "$VM_ID" \
  --role "Key Vault Secrets User" \
  --scope "$(az keyvault show \
    --resource-group rg-az500-lab-eastus \
    --name kv-az500-lab \
    --query id -o tsv)"
```

This pattern is reliable because it targets the managed identity directly. In practice, you should confirm the VM has a system-assigned identity enabled before assigning the role.

**Managed identity workflow**

Managed identities let Azure resources authenticate without storing credentials. In a sample workload, this is one of the most important patterns to validate.

Common uses:

- A VM reads a secret from Key Vault
- An app service accesses storage or Key Vault
- A function app retrieves configuration securely

Example enabling a system-assigned identity on a VM:

```bash
az vm identity assign \
  --resource-group rg-az500-lab-eastus \
  --name vm-app01
```

Example retrieving a secret from Key Vault on the VM after role assignment:

```bash
az login --identity

az keyvault secret show \
  --vault-name kv-az500-lab \
  --name AppSecret
```

This validates both identity and authorization. If the command fails, check the role assignment scope, the vault authorization model, and any network restrictions.

**Storage account for secure testing**

A storage account is useful for testing encryption, network access, and diagnostics. Keep the configuration simple and verify the exact diagnostic categories available for the resource and logging mode you choose.

Example storage account creation:

```bash
az storage account create \
  --resource-group rg-az500-lab-eastus \
  --name staz500lab01 \
  --location eastus \
  --sku Standard_LRS \
  --kind StorageV2 \
  --https-only true \
  --min-tls-version TLS1_2
```

For network hardening, prefer private access or selected networks rather than broad public access.

Example of restricting public network access:

```bash
az storage account update \
  --resource-group rg-az500-lab-eastus \
  --name staz500lab01 \
  --default-action Deny
```

When you configure diagnostics, verify the available categories for the specific resource. Storage and other Azure services do not always expose the same category names across all configurations.

**Diagnostic settings and monitoring**

Monitoring is essential because security controls are only useful if you can prove they work. Send logs to Log Analytics so you can query activity, access attempts, and configuration changes.

Example workspace creation:

```bash
az monitor log-analytics workspace create \
  --resource-group rg-az500-lab-eastus \
  --workspace-name law-az500-lab \
  --location eastus
```

Example diagnostic settings for a Key Vault or other resource:

```bash
RESOURCE_ID=$(az keyvault show \
  --resource-group rg-az500-lab-eastus \
  --name kv-az500-lab \
  --query id -o tsv)

WORKSPACE_ID=$(az monitor log-analytics workspace show \
  --resource-group rg-az500-lab-eastus \
  --workspace-name law-az500-lab \
  --query id -o tsv)

az monitor diagnostic-settings create \
  --name diag-to-law \
  --resource "$RESOURCE_ID" \
  --workspace "$WORKSPACE_ID" \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

The exact log categories depend on the resource type. Always check the supported categories before you automate the setting.

**Querying logs in Log Analytics**

Use Kusto Query Language (KQL) to validate that telemetry is arriving. The table name depends on the data source and the diagnostic configuration. Some resources send data to resource-specific tables, while others may still use more generic tables.

Example approach for discovering tables:

```kusto
search "kv-az500-lab"
| summarize count() by $table
| order by count_ desc
```

Example query for recent activity in a resource-specific table, if present:

```kusto
AzureDiagnostics
| where TimeGenerated > ago(24h)
| where ResourceProvider == "MICROSOFT.KEYVAULT"
| summarize count() by OperationName
| order by count_ desc
```

Because table schemas vary, validate the actual table names in your workspace before relying on a query in a lab or exam scenario.

**Policy assignment and compliance**

Azure Policy helps you enforce standards such as tagging, allowed locations, secure transfer, or private access. In a sample workload, policy is useful for testing both compliance and remediation.

A policy assignment usually requires the correct definition name and the correct parameter schema. Built-in policies can vary by cloud, tenant, and version, so confirm the definition before assigning it.

Example workflow:

1. Find the policy definition.
2. Inspect the required parameters.
3. Assign the policy with a complete parameter payload.
4. Check compliance results.

Example discovery commands:

```bash
az policy definition list \
  --query "[?contains(displayName, 'tag')].{name:name, displayName:displayName}" \
  -o table
```

Example assignment pattern for a tag policy that requires both tag name and tag value:

```bash
az policy assignment create \
  --name require-costcenter \
  --display-name "Require CostCenter tag" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>" \
  --scope "/subscriptions/<subscription-id>" \
  --params '{
    "tagName": { "value": "CostCenter" },
    "tagValue": { "value": "AZ500" }
  }'
```

After assignment, review compliance:

```bash
az policy state list \
  --query "[?contains(policyAssignmentName, 'require-costcenter')]" \
  -o table
```

**Network security controls to validate**

A sample workload should let you test how traffic is allowed or denied. Common controls include:

- **Network security groups** on subnets or NICs
- **Private endpoints** for PaaS services
- **Service endpoints** where appropriate
- **Just-in-time** access for VMs if Defender for Cloud is in use
- **No public IP** on management workloads

Example NSG rule to allow SSH only from a trusted source:

```bash
az network nsg rule create \
  --resource-group rg-az500-lab-eastus \
  --nsg-name nsg-app \
  --name AllowSSHFromAdmin \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 203.0.113.10/32 \
  --source-port-ranges '*' \
  --destination-port-ranges 22
```

In a lab, you often pair this with Bastion so you can avoid opening SSH or RDP to the internet.

**Practical validation checklist**

Use the deployed workload to verify that each control behaves as expected.

- Confirm the VM has no public IP if that is the design goal.
- Confirm Bastion can connect to the VM.
- Confirm the VM managed identity can access Key Vault only after role assignment.
- Confirm storage access is blocked or allowed according to network rules.
- Confirm diagnostic logs appear in Log Analytics.
- Confirm policy compliance reflects the deployed configuration.

**Troubleshooting common deployment issues**

Most issues in a sample workload come from dependency order or mismatched configuration.

- **Bastion deployment fails**
  - Check that `AzureBastionSubnet` exists and is large enough.
  - Verify the Bastion SKU supports the features you want.
  - Confirm the public IP is Standard and static.

- **Key Vault access fails**
  - Check whether the vault uses RBAC or access policies.
  - Confirm the managed identity is enabled.
  - Verify the role assignment scope and object ID.
  - Check whether network restrictions block access.

- **Diagnostics do not appear**
  - Confirm the diagnostic setting is attached to the correct resource.
  - Verify the workspace ID.
  - Check the resource’s supported log categories.
  - Allow time for ingestion.

- **Policy compliance is not updating**
  - Confirm the assignment scope.
  - Check whether the policy requires remediation.
  - Re-evaluate compliance after deployment.

**Why this workload matters for AZ-500**

AZ-500 focuses on securing identities, platforms, data, and operations. A sample workload gives you a safe way to practice those skills end to end.

You should be able to explain:

- How to deploy a private workload
- How to secure administrative access
- How to use managed identity instead of secrets
- How to restrict and validate network access
- How to collect and query logs
- How to enforce governance with policy

That combination is what turns isolated Azure features into a secure design.

**Recommended lab sequence**

1. Create the resource group.
2. Deploy the virtual network and subnets.
3. Deploy the VM without a public IP.
4. Add Bastion for secure access.
5. Create Key Vault and enable RBAC or access policies.
6. Assign the VM managed identity the minimum required role.
7. Create the Log Analytics workspace.
8. Configure diagnostic settings on the resources.
9. Apply policy assignments and check compliance.
10. Validate access, logging, and network restrictions.

**Key takeaways**

A sample workload is a controlled Azure environment for testing security controls. Keep it small, private by default, and easy to redeploy. Use managed identity, Bastion, diagnostics, and policy together so you can validate the full security lifecycle rather than a single feature in isolation.
