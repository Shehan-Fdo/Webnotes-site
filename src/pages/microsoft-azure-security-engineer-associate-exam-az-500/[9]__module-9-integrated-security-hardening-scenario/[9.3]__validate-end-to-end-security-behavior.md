## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Purpose of end-to-end validation**

End-to-end security validation means you confirm that a control works from the user or attacker path all the way through the Azure service, network, identity, and logging layers. For AZ-500, you are often asked not just whether a feature is enabled, but whether it actually blocks, allows, logs, or remediates as intended.

A good validation workflow checks four things:

- **Configuration**: the control is deployed and scoped correctly.
- **Path**: traffic or requests follow the expected route.
- **Outcome**: access is allowed, denied, inspected, or remediated as designed.
- **Evidence**: logs, metrics, and alerts show the event.

**Validation mindset**

Use a layered approach. A security feature can appear healthy in the portal while still failing because of DNS, routing, identity scope, or policy assignment issues. Always test the control from the same network, identity, and client type that the real workload uses.

- Validate from the **client side** with a real request.
- Validate from the **service side** with logs and diagnostics.
- Validate from the **control plane** with Azure Resource Manager, Azure Policy, and role assignments.
- Validate from the **data plane** with network traces, DNS resolution, and service responses.

**Identity and access validation**

Identity validation confirms that the right principal can authenticate and authorize, and the wrong principal cannot. In Azure, this usually means checking Microsoft Entra ID authentication, role-based access control (RBAC), conditional access, and privileged access workflows.

Use current Microsoft Graph-based tooling where possible for directory validation. Some `az ad` commands still exist, but Microsoft Graph-based commands and portal views are often the better choice for newer validation tasks.

Common checks include:

- Confirm the signed-in identity:
  ```bash
  az account show --query "{user:user.name, tenantId:tenantId, subscription:id}" -o json
  ```
- Inspect role assignments at a scope:
  ```bash
  az role assignment list --assignee <object-id-or-upn> --scope /subscriptions/<sub-id>/resourceGroups/<rg-name> -o table
  ```
- Check whether a user can activate a privileged role in Privileged Identity Management (PIM) through the portal or Graph-backed views.
- Test with a non-admin account to confirm least privilege.

> Azure CLI can list role assignments, but it does not directly calculate “effective permissions” in a single universal command. Effective access depends on inherited scopes, deny assignments, group membership, PIM activation, and resource-specific authorization behavior.

**Practical RBAC validation pattern**

To validate access, test both an allowed and a denied action.

1. Sign in as the target user or use a service principal.
2. Attempt a read action that should succeed.
3. Attempt a write or delete action that should fail.
4. Review the activity log and authorization failure details.

Example:

```bash
# Read should succeed if the role allows it
az storage account show \
  --name <storage-account> \
  --resource-group <rg-name>

# Write should fail if the role is read-only
az storage account update \
  --name <storage-account> \
  --resource-group <rg-name> \
  --set tags.env=test
```

If the request fails, inspect whether the issue is:

- Missing role assignment
- Wrong scope
- Group membership not yet propagated
- PIM role not activated
- Deny assignment from management group or policy

**Network security validation**

Network validation confirms that traffic follows the intended route and that private access, filtering, and inspection are actually in effect. This is especially important for private endpoints, service endpoints, Azure Firewall, Network Security Groups (NSGs), and Application Gateway Web Application Firewall (WAF).

**Private endpoint validation**

A private endpoint is only validated when the client resolves the service FQDN to the private IP and reaches the service over the private network path. A successful TLS connection alone is not enough.

Check these items:

- DNS resolves the service name to the private endpoint IP.
- The private DNS zone is linked to the correct virtual network.
- Split-horizon DNS or conditional forwarding is configured for hybrid clients.
- The route table and NSG allow the traffic path.
- The service denies public access if that is the design.

Example for a Key Vault private endpoint:

```bash
nslookup <kv-name>.vault.azure.net
```

You want to see the private IP address returned through the private DNS zone, not the public endpoint address. If the name still resolves publicly, check:

- Private DNS zone `privatelink.vaultcore.azure.net`
- Virtual network link
- On-premises DNS forwarding
- Client DNS cache

You can also validate the path with a connection test:

```bash
Test-NetConnection <kv-name>.vault.azure.net -Port 443
```

Use this only as a connectivity check. It does not prove private endpoint usage by itself.

**Storage private endpoint example**

For Storage, the public FQDN often resolves to a private IP when private DNS is configured correctly:

```bash
nslookup <storage-account>.blob.core.windows.net
```

Confirm that the returned address is the private IP from the `privatelink.blob.core.windows.net` zone. If you see the public IP, the most common causes are:

- Missing private DNS zone link
- Incorrect DNS forwarding
- Client using a different DNS server than expected
- Cached public resolution on the client

**Application Gateway WAF validation**

WAF validation should prove that the policy is attached, the rule set is active, and the request is inspected. Do not assume a single payload will always trigger a managed rule, because rule matching depends on the configured rule set, exclusions, request encoding, and request context.

Better validation methods include:

- Use the WAF logs in Log Analytics to confirm rule matches.
- Send known test traffic patterns that align with the enabled managed rules.
- Verify the response code and WAF action in logs.
- Confirm whether the policy is in **Detection** or **Prevention** mode.

Example log query for WAF events:

```kusto
AzureDiagnostics
| where Category == "ApplicationGatewayFirewallLog"
| project TimeGenerated, clientIP_s, requestUri_s, ruleId_s, message_s, action_s
| order by TimeGenerated desc
```

If you need to test blocking behavior, use a controlled request that matches a known rule in your configured policy, then confirm the event in logs rather than relying only on the HTTP response.

**Azure Firewall and NSG validation**

For Azure Firewall, validate both the route and the log evidence:

- Confirm the subnet route table sends traffic to the firewall.
- Confirm the firewall policy or rules allow or deny the expected flow.
- Review firewall logs for the matched rule and action.

Example route table check:

```bash
az network route-table route list \
  --resource-group <rg-name> \
  --route-table-name <rt-name> \
  -o table
```

For NSGs, validate with effective rules and a real packet path:

- Check the NSG association on the subnet or NIC.
- Review effective security rules in the portal.
- Test the port from a client in the same path.

Example connectivity test:

```bash
Test-NetConnection <private-ip> -Port 3389
```

Use the result together with NSG flow logs or Network Watcher connection troubleshooting.

**Logging and monitoring validation**

Security controls are not fully validated until you confirm telemetry. Azure monitoring often involves Azure Monitor, Log Analytics, diagnostic settings, activity logs, and resource logs.

**Diagnostic settings**

Many Azure services require diagnostic settings to send logs to a Log Analytics workspace, storage account, or event hub. Validate that:

- The diagnostic setting exists.
- The correct log categories are enabled.
- The destination workspace is receiving data.
- The retention and access model meet requirements.

Example:

```bash
az monitor diagnostic-settings list \
  --resource <resource-id> \
  -o table
```

**Activity log validation**

The Azure Activity Log records control-plane events such as create, update, delete, and authorization failures. Use it to confirm management actions and policy effects.

Example query in Log Analytics if activity logs are exported:

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where ActivityStatusValue in ("Failed", "Succeeded")
| project TimeGenerated, OperationNameValue, ActivityStatusValue, Caller, ResourceGroup, ResourceProviderValue
| order by TimeGenerated desc
```

**Log alert validation**

Log alerts are query-based alerts that run against a Log Analytics workspace or an alerting scope that supports log queries. You must ensure the data is actually ingested into the workspace before expecting the alert to fire.

Example KQL for a failed sign-in pattern if sign-in logs are ingested into the workspace:

```kusto
SigninLogs
| where ResultType != 0
| summarize FailedCount = count() by UserPrincipalName, bin(TimeGenerated, 15m)
| where FailedCount > 5
```

Validation steps:

1. Confirm the source table exists in the workspace.
2. Run the query manually and verify it returns rows.
3. Create the alert rule against the correct workspace or scope.
4. Trigger the condition with test activity.
5. Confirm the alert fired and the action group executed.

**Metric alert validation**

Metric alerts evaluate numeric platform metrics such as CPU, latency, or request count. The exact Azure CLI syntax can vary by version, so verify against your installed CLI help before using it in production.

A typical validation flow is:

- Identify the metric namespace and metric name.
- Confirm the resource emits the metric.
- Create the alert with a threshold and aggregation.
- Generate load or wait for the metric to cross the threshold.
- Confirm the alert state changes and the action group runs.

Example of checking available metrics:

```bash
az monitor metrics list-definitions \
  --resource <resource-id> \
  -o table
```

Example of querying a metric before creating an alert:

```bash
az monitor metrics list \
  --resource <resource-id> \
  --metric "Percentage CPU" \
  --interval PT1M \
  --aggregation Average
```

**Azure Policy validation**

Azure Policy is often validated by checking whether resources are compliant, denied, or remediated. Understand the difference between policy effects:

- **Deny** blocks noncompliant resource creation or update.
- **Audit** records noncompliance without blocking.
- **Append** adds or modifies fields during deployment.
- **Modify** changes resource properties during deployment.
- **DeployIfNotExists** evaluates whether a required related resource or configuration is missing, then deploys the specified resource or configuration through a remediation path.

The key point is that `DeployIfNotExists` is about detecting noncompliance and then deploying the missing supporting resource or configuration, not simply reacting to a generic condition.

Example policy state check:

```bash
az policy state list \
  --resource-group <rg-name> \
  -o table
```

Example compliance summary:

```bash
az policy assignment list --scope /subscriptions/<sub-id> -o table
az policy state summarize --scope /subscriptions/<sub-id>
```

If you use remediation, confirm that:

- The managed identity for the assignment has the required permissions.
- The remediation task is created.
- The target resource becomes compliant after deployment.

**Practical policy validation example**

If a policy requires diagnostic settings on a storage account, validate it like this:

1. Deploy a storage account without the setting.
2. Confirm the resource is marked noncompliant.
3. Trigger remediation if the policy uses `DeployIfNotExists`.
4. Recheck compliance after the remediation task completes.

**Key Vault validation**

Key Vault validation often combines identity, network, and audit logging checks.

Validate:

- Access policy or RBAC permission to secrets, keys, or certificates.
- Public network access setting.
- Private endpoint resolution if used.
- Diagnostic logs for `SecretGet`, `KeyGet`, `CertificateGet`, and failed authorization events.

Example secret retrieval test:

```bash
az keyvault secret show \
  --vault-name <kv-name> \
  --name <secret-name>
```

If the request fails, determine whether the cause is:

- Missing data-plane permission
- Firewall or private endpoint issue
- DNS resolution problem
- Disabled public access
- Wrong tenant or subscription context

**Entra ID and sign-in validation**

For authentication controls, validate both success and failure paths:

- Successful sign-in from an allowed device or location.
- Blocked sign-in from a disallowed device, location, or risk state.
- MFA (multifactor authentication) challenge when required.
- Conditional Access policy result in sign-in logs.

Useful checks include:

- Sign-in logs in Microsoft Entra admin center
- Conditional Access policy report-only mode
- Authentication methods registration
- Risk-based sign-in outcomes

A practical validation pattern is to test with a pilot user group before broad rollout. Report-only mode is especially useful because it shows the policy impact without enforcing it.

**Service-to-service validation**

Many Azure workloads use managed identities, service principals, or workload identities. Validate these paths separately from human user access.

Check:

- The identity is assigned to the resource.
- The identity has the correct RBAC role or data-plane permission.
- Token acquisition succeeds.
- The downstream service accepts the token.

Example managed identity test from an Azure VM:

```bash
az login --identity
az account get-access-token --resource https://vault.azure.net
```

If token acquisition works but the service call fails, the issue is usually authorization, not authentication.

**Troubleshooting approach**

When validation fails, isolate the layer first, then the control.

- If DNS is wrong, fix name resolution before testing the service.
- If routing is wrong, fix the path before checking the firewall.
- If authorization fails, verify scope and role before changing the application.
- If logs are missing, verify diagnostic settings and workspace ingestion.

A simple triage sequence is:

1. Confirm identity.
2. Confirm DNS.
3. Confirm network path.
4. Confirm service configuration.
5. Confirm logs and alerts.

**Common exam traps**

- A control can be enabled but still ineffective because the client uses the wrong DNS server.
- A successful HTTPS response does not prove private endpoint usage.
- A role assignment at subscription scope may still not grant access if a deny assignment exists.
- A WAF payload test is not guaranteed to match a managed rule.
- A metric alert example may fail if the CLI syntax does not match the installed version.
- A log alert requires the right workspace scope and ingested data.
- Policy remediation requires permissions and a remediation task, not just assignment.

**Quick validation checklist**

- Confirm the control is deployed at the correct scope.
- Test with a real client or identity.
- Verify DNS and routing for private access.
- Check logs, metrics, and activity records.
- Validate both allowed and denied behavior.
- Re-test after remediation or policy changes.

**Reference command patterns**

```bash
# Check current subscription and identity
az account show -o json

# List role assignments for a principal at a scope
az role assignment list --assignee <object-id-or-upn> --scope <scope> -o table

# Inspect policy compliance
az policy state summarize --scope <scope>

# Query a Log Analytics workspace
az monitor log-analytics query \
  --workspace <workspace-id> \
  --analytics-query "AzureActivity | take 5"

# Check DNS resolution from a client
nslookup <service-fqdn>
```

**What to remember for AZ-500**

End-to-end validation is about proving security behavior, not just configuration. The strongest answers usually connect identity, network, policy, and telemetry into one testable flow. If a control does not produce the expected result, use logs and path validation to find the layer where the behavior diverges.
