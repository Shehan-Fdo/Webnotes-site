## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Key rotation and versioning overview**

Azure Key Vault keys are versioned objects. Each time you create a new version, Key Vault keeps the previous versions unless you delete the key object. This matters because many Azure services can reference a key by a versionless identifier and then pick up a newer version according to their own refresh behavior.

A key version is not just a label. It carries its own attributes, such as enabled state, expiration, and activation time. For exam purposes, remember that Key Vault enforces these attributes during cryptographic operations. A disabled key version cannot be used for encrypt, decrypt, sign, or verify operations through Key Vault.

- **Versioned URI** points to one exact key version.
- **Versionless URI** points to the key name only and resolves to the latest version.
- **Enabled** versions can be used for supported operations.
- **Disabled** versions remain stored but cannot be used for cryptographic operations.
- **Expired** versions are not intended for new operations and should be rotated out of active use.

**How Azure Key Vault key versions work**

A key in Key Vault is identified by a name and a version. The version is appended to the key identifier in the URI. A versionless URI is useful when you want a consuming service to follow the latest version without changing the reference every time you rotate.

Example key identifiers:

```text
https://myvault.vault.azure.net/keys/app-key
https://myvault.vault.azure.net/keys/app-key/3f2c1d7a9b4e4b1a8c2d6e7f8a9b0c1d
```

Use a versioned URI when you need deterministic behavior, such as preserving access to a specific historical key version for a workload that must not change unexpectedly. Use a versionless URI when the service supports automatic pickup of new versions and you want rotation to be transparent.

> A versionless key identifier resolves to the latest version, but the consuming service may cache the reference or refresh it on its own schedule.

**Key lifecycle states**

Key versions commonly move through these states:

- **Created**: The version exists in the vault but may not yet be active if a future activation time is set.
- **Enabled**: The version can be used for supported cryptographic operations.
- **Disabled**: The version remains stored but Key Vault blocks cryptographic use.
- **Expired**: The version has passed its expiration time and should not be used for new operations.
- **Deleted**: The key object is removed from active use and may be recoverable only through soft delete at the object level, if enabled.
- **Recovered**: The deleted key object is restored, including its versions.

Important exam distinction:

- **Rotation** creates a new version.
- **Renewal** extends or updates the validity window of an existing version’s attributes.
- **Deletion** removes the key object from active use.
- **Recovery** restores a deleted key object, not an individual version in the usual sense.

**Manual rotation versus automatic rotation policy**

You can rotate keys in two main ways:

1. **Manual rotation**
   - You create a new key version yourself.
   - You use this when you need immediate control, a change window, or a one-off rotation outside policy.
   - In the portal, this is typically done by selecting the key and choosing the option to create a new version.
   - In CLI, you create a new version by running `az keyvault key create` again with the same key name and vault.

2. **Automatic rotation policy**
   - You define a rotation policy that tells Key Vault when to rotate and when to notify.
   - This is useful for predictable compliance-driven rotation.
   - The policy can trigger rotation based on time and can send notifications before or after rotation events.

Manual rotation is the safer answer when a question asks how to immediately create a new version. Automatic rotation is the better answer when the question asks how to reduce operational overhead or enforce periodic rotation.

**Practical manual rotation examples**

Create a new version of an RSA key in the same vault:

```bash
az keyvault key create \
  --vault-name myvault \
  --name app-key \
  --kty RSA \
  --size 2048
```

List versions for a key:

```bash
az keyvault key list-versions \
  --vault-name myvault \
  --name app-key \
  -o table
```

Show the current latest version returned by the key name:

```bash
az keyvault key show \
  --vault-name myvault \
  --name app-key \
  -o jsonc
```

Get a specific version:

```bash
az keyvault key show \
  --vault-name myvault \
  --name app-key \
  --version 3f2c1d7a9b4e4b1a8c2d6e7f8a9b0c1d
```

The `show` command is useful when you want to verify the active version, inspect attributes, or confirm whether the key reference is versioned or versionless.

**How to verify rotation success**

After rotation, verify both the vault state and the consuming service state.

- In Key Vault:
  - Confirm the new version appears in `az keyvault key list-versions`.
  - Check that the latest version is enabled.
  - Confirm the activation and expiration attributes are correct.
- In the portal:
  - Open the key and inspect the versions list.
  - Check which version is marked as current or latest.
  - Review the key’s attributes and rotation policy.
- In the consuming service:
  - Confirm whether the service uses a versionless URI.
  - Check whether the service has refreshed its key reference.
  - Validate that encryption or signing operations use the expected version.

A common troubleshooting pattern is that the vault has the new version, but the service still uses the old one because it is pinned to a versioned URI or has not refreshed yet.

**Rotation policy basics**

A rotation policy defines when Key Vault should rotate a key and what notifications should occur around that event. The exact schema and tooling can vary by API version and client, so be careful with exam questions that show JSON snippets.

Core ideas:

- **Lifetime actions** define what happens as the key approaches a threshold.
- **Time-based triggers** are common for rotation schedules.
- **Notifications** can alert you before or after rotation.
- **Key attributes** such as expiration are separate from the rotation policy itself.

A simplified conceptual example:

```json
{
  "lifetimeActions": [
    {
      "trigger": {
        "timeAfterCreate": "P90D"
      },
      "action": {
        "type": "Rotate"
      }
    }
  ],
  "attributes": {
    "expiryTime": "P180D"
  }
}
```

Use this as a conceptual model only. In practice, the exact payload depends on the CLI, PowerShell module, or REST API version you use.

> Do not confuse key expiration with rotation timing. Expiration limits use of a version; rotation creates a new version.

**CLI and PowerShell management patterns**

Azure CLI is often the clearest way to demonstrate version management in exam prep.

List all versions:

```bash
az keyvault key list-versions \
  --vault-name myvault \
  --name app-key
```

Disable a specific version:

```bash
az keyvault key set-attributes \
  --vault-name myvault \
  --name app-key \
  --version 3f2c1d7a9b4e4b1a8c2d6e7f8a9b0c1d \
  --enabled false
```

Set an expiration on the current version:

```bash
az keyvault key set-attributes \
  --vault-name myvault \
  --name app-key \
  --expires 2027-01-01T00:00:00Z
```

Set an activation time:

```bash
az keyvault key set-attributes \
  --vault-name myvault \
  --name app-key \
  --not-before 2026-05-01T00:00:00Z
```

PowerShell can also manage keys, but the exact cmdlet shape depends on the Az.KeyVault module version. In exam scenarios, focus on the concept: retrieve the key, inspect versions, and update attributes or create a new version as needed.

Example pattern:

```powershell
Get-AzKeyVaultKey -VaultName "myvault" -Name "app-key"
Get-AzKeyVaultKey -VaultName "myvault" -Name "app-key" -Version "3f2c1d7a9b4e4b1a8c2d6e7f8a9b0c1d"
```

**Service behavior after rotation**

Different Azure services handle new key versions differently. This is a common AZ-500 trap.

| Service | Typical behavior with Key Vault keys |
|---|---|
| Azure Storage | Can use customer-managed keys for encryption at rest; versionless references are commonly used so the service can pick up new versions according to its refresh cycle. |
| Azure Disk Encryption | Often requires careful reconfiguration or re-encryption workflows; versioning behavior depends on how the disk encryption set or extension is configured. |
| Azure SQL | Can use customer-managed keys for TDE (Transparent Data Encryption); versionless references are preferred for smoother rotation, but service refresh timing still matters. |

Practical guidance:

- Use a **versionless URI** when the service supports it and you want automatic pickup of new versions.
- Use a **versioned URI** when you need to pin a workload to a specific version.
- Expect **refresh delays** in some services even when the vault has already rotated.
- Reconfigure or reapply the key reference if the service does not automatically move to the new version.

**Troubleshooting rotated keys not taking effect**

When a rotated key is not being used, check the following:

1. Confirm the service is not pinned to a versioned key URI.
2. Verify the new version is enabled and not expired.
3. Check whether the service caches the key reference.
4. Review whether the service requires a manual refresh or reconfiguration.
5. Confirm the key type and service support the rotation model you are using.
6. Validate that the identity used by the service still has access to the vault and key.

Common causes:

- The application stores the full versioned key URI in configuration.
- The service has not refreshed its metadata.
- The new version exists but is disabled or not yet active.
- The workload uses a different vault or key name than expected.
- Access policy or RBAC permissions changed during rotation.

**Permissions and access control**

Key management permissions depend on whether you use the older access policy model or Azure RBAC.

For data-plane key operations, you typically need permissions such as:

- `get`
- `list`
- `create`
- `delete`
- `recover`
- `backup`
- `restore`
- `encrypt`
- `decrypt`
- `sign`
- `verify`
- `wrapKey`
- `unwrapKey`

Rotation policy management is not the same as cryptographic use permissions. In RBAC, use a role that grants the required Key Vault data actions for key management and rotation policy operations. In access policy mode, be careful not to assume that every administrative action is represented as a simple key permission.

**Key export and release limitations**

Key Vault keys are generally non-exportable in normal scenarios. That is a major security feature.

- You do not usually download the private key material from Key Vault.
- You rotate by creating a new version inside the service.
- You keep the key material protected by the vault boundary.
- If a question mentions exporting keys for rotation, treat that as a red flag unless the scenario explicitly involves a supported release workflow or a different product.

This matters because version management is about controlling which version is active, not about moving private key material out of the vault.

**Rotation, expiration, and renewal compared**

| Concept | What it does | Exam clue |
|---|---|---|
| Rotation | Creates a new key version | “Generate a new version” |
| Expiration | Limits when a version should be used | “Set a date after which it should not be used” |
| Renewal | Extends the validity of a version’s attributes | “Extend the lifetime” |
| Deletion | Removes the key object from active use | “Delete the key” |
| Recovery | Restores a deleted key object | “Recover the deleted key” |

Use this distinction to avoid confusing a lifecycle update with a true rotation event.

**Practical workflow for secure key rotation**

1. Review the current key version and its expiration.
2. Decide whether you need manual rotation or a policy-driven rotation.
3. Create a new version in Key Vault.
4. Verify the new version is enabled and active.
5. Update or confirm the consuming service uses a versionless URI if supported.
6. Test encryption, decryption, signing, or verification.
7. Disable or retire the old version only after validating the workload.
8. Monitor for refresh delays or failed operations.

**Exam-focused takeaways**

- Key Vault keys are versioned, and each version has its own attributes.
- A versionless URI resolves to the latest version, but service refresh behavior still matters.
- Disabled versions cannot be used for cryptographic operations through Key Vault.
- Rotation creates a new version; it is not the same as extending expiration.
- Manual rotation is the direct way to create a new version immediately.
- Automatic rotation policies reduce operational effort but depend on supported tooling and service behavior.
- Some Azure services pick up new versions automatically only when configured correctly.
- Key Vault keys are normally non-exportable, so version management happens inside the vault.
