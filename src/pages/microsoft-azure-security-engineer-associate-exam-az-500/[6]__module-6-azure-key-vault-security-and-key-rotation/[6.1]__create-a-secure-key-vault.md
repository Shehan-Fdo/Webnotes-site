## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Azure Key Vault security model**

Azure Key Vault stores and controls access to cryptographic keys, secrets, and certificates. In AZ-500, you need to know how to create a vault that is secure by default and then harden it with identity, authorization, and network controls.

A secure design starts with three questions:

- Who can manage the vault?
- Who can read or use the data plane objects?
- From where can traffic reach the vault?

Key Vault separates **management plane** actions from **data plane** actions:

- Management plane covers creating the vault, changing settings, and assigning roles.
- Data plane covers reading secrets, using keys, and managing certificates.

This distinction matters because a user can have permission to manage the vault resource without being allowed to read secrets inside it.

**Core creation choices**

When you create a vault, you choose settings that affect security posture from the start. The most important are:

- **Subscription and resource group** placement for governance and lifecycle management.
- **Region** for latency, compliance, and disaster recovery planning.
- **SKU** for feature set and cost.
- **Soft delete** and **purge protection** for recovery and deletion safety.
- **Access model** for authorization.
- **Network access** for exposure control.

A typical secure baseline is:

- Enable soft delete.
- Enable purge protection in production.
- Restrict public network access when possible.
- Prefer Azure role-based access control (Azure RBAC) unless you have a specific reason to use access policies.
- Use private connectivity for workloads that do not need public access.

**Create a vault with Azure CLI**

You can create a vault with the Azure CLI and immediately apply secure defaults.

```bash
az keyvault create \
  --name kv-secure-prod-001 \
  --resource-group rg-security-prod \
  --location eastus \
  --sku standard \
  --enable-rbac-authorization true \
  --enable-purge-protection true \
  --public-network-access Disabled
```

Important notes:

- `--enable-rbac-authorization true` uses Azure RBAC for data-plane authorization.
- `--public-network-access Disabled` blocks public endpoint access.
- `--enable-purge-protection true` prevents permanent deletion until the retention period expires.
- `--sku standard` is sufficient for most secret and key scenarios; use `premium` when you need HSM-backed capabilities such as certain key operations.

If you need to inspect the resulting configuration:

```bash
az keyvault show \
  --name kv-secure-prod-001 \
  --resource-group rg-security-prod \
  --query "{name:name, publicNetworkAccess:properties.publicNetworkAccess, enableRbacAuthorization:properties.enableRbacAuthorization, softDeleteRetentionInDays:properties.softDeleteRetentionInDays}"
```

**Create a vault with Bicep**

Infrastructure as Code (IaC) helps you enforce repeatable security settings. Bicep is a common choice for Azure deployments.

```bicep
param vaultName string = 'kv-secure-prod-001'
param location string = resourceGroup().location

resource kv 'Microsoft.KeyVault/vaults@2023-07-01' = {
  name: vaultName
  location: location
  properties: {
    tenantId: subscription().tenantId
    sku: {
      family: 'A'
      name: 'standard'
    }
    enableRbacAuthorization: true
    enablePurgeProtection: true
    publicNetworkAccess: 'Disabled'
    softDeleteRetentionInDays: 90
    networkAcls: {
      bypass: 'None'
      defaultAction: 'Deny'
    }
  }
}
```

Use this pattern when you want:

- Consistent security settings across environments.
- Version-controlled changes.
- Repeatable deployments in CI/CD pipelines.

> If you use access policies instead of Azure RBAC, you must define `accessPolicies` explicitly. If you use Azure RBAC, avoid mixing the two models unless you understand the migration impact.

**Authorization models**

Azure Key Vault supports two authorization approaches:

| Model | Where it applies | Strengths | Considerations |
|---|---|---|---|
| Azure RBAC | Data plane via Azure role assignments | Centralized governance, consistent with Azure IAM | May take time to propagate; role selection must match the operation |
| Access policies | Key Vault-specific data-plane permissions | Fine-grained and familiar for older deployments | Harder to govern at scale; less aligned with modern Azure access management |

For new deployments, Azure RBAC is usually the preferred model because it integrates with Azure governance and simplifies role management.

Common built-in roles include:

- **Key Vault Administrator** for full data-plane management.
- **Key Vault Secrets Officer** for secret management.
- **Key Vault Crypto Officer** for key operations.
- **Key Vault Reader** for read-only metadata access.

Use the least privilege role that matches the task. For example, an application that only reads a secret should not receive administrative permissions.

**Assign access with Azure RBAC**

A common workflow is to create the vault, then assign a role to a user, group, or managed identity.

```bash
az role assignment create \
  --assignee-object-id <object-id> \
  --role "Key Vault Secrets Officer" \
  --scope $(az keyvault show --name kv-secure-prod-001 --resource-group rg-security-prod --query id -o tsv)
```

Practical guidance:

- Prefer **managed identities** for applications running in Azure.
- Prefer **groups** for human access so you can manage membership centrally.
- Use **service principals** only when a managed identity is not available.

**Network security controls**

Network controls reduce exposure even when identity controls are strong. For Key Vault, you should understand the difference between public access, firewall rules, service endpoints, and Private Link.

Key options include:

- **Public network access**: enables or disables access through the public endpoint.
- **Network ACLs**: allow or deny traffic based on IP rules and virtual network rules.
- **Private Endpoint**: gives the vault a private IP in your virtual network through Azure Private Link.
- **Service endpoints**: older model that extends virtual network identity to the service, but does not provide a private IP.

For modern secure designs, Private Link is generally preferred over service endpoints because it keeps traffic on the Microsoft backbone and exposes the vault through a private address in your network.

A hardened network ACL example:

```bash
az keyvault network-rule add \
  --name kv-secure-prod-001 \
  --resource-group rg-security-prod \
  --ip-address 203.0.113.10
```

And to allow a subnet:

```bash
az keyvault network-rule add \
  --name kv-secure-prod-001 \
  --resource-group rg-security-prod \
  --vnet-name vnet-prod \
  --subnet app-subnet
```

If you disable public network access, clients must use private connectivity or another allowed path. This is often the best choice for production workloads.

**Private Endpoint design**

Private Endpoint is the preferred approach when you want to remove public exposure. It maps the vault to a private IP in your virtual network and uses DNS to resolve the vault name to that private address.

Typical design points:

- Place the private endpoint in the same or peered virtual network as the workload.
- Configure private DNS so the vault name resolves correctly.
- Restrict the public endpoint or disable it entirely.
- Validate name resolution from the client subnet.

Example CLI pattern:

```bash
az network private-endpoint create \
  --name pe-kv-secure-prod-001 \
  --resource-group rg-security-prod \
  --vnet-name vnet-prod \
  --subnet private-endpoints \
  --private-connection-resource-id $(az keyvault show --name kv-secure-prod-001 --resource-group rg-security-prod --query id -o tsv) \
  --group-id vault \
  --connection-name kv-conn
```

You also need private DNS integration:

```bash
az network private-dns zone create \
  --resource-group rg-security-prod \
  --name privatelink.vaultcore.azure.net
```

Then link the zone to the virtual network and create the appropriate DNS record through the private endpoint workflow.

**Soft delete and purge protection**

Soft delete keeps deleted vault objects recoverable for a retention period. Purge protection prevents permanent deletion before that retention period ends.

Why this matters:

- It protects against accidental deletion.
- It helps defend against malicious deletion attempts.
- It supports recovery workflows during incident response.

Operational implications:

- You cannot immediately purge protected items.
- Retention settings should align with compliance and recovery requirements.
- In production, purge protection is usually a strong recommendation.

**Secrets, keys, and certificates**

Key Vault can store three main object types:

- **Secrets** for passwords, connection strings, API tokens, and certificates as secret blobs.
- **Keys** for cryptographic operations such as encryption, decryption, signing, and verification.
- **Certificates** for certificate lifecycle management and integration with supported services.

Use the right object type for the job:

- Store application passwords as secrets.
- Use keys when you need cryptographic operations without exposing key material.
- Use certificates when you need certificate lifecycle support and integration with TLS scenarios.

Example secret creation:

```bash
az keyvault secret set \
  --vault-name kv-secure-prod-001 \
  --name SqlAdminPassword \
  --value 'Use-a-strong-random-value'
```

Example key creation:

```bash
az keyvault key create \
  --vault-name kv-secure-prod-001 \
  --name app-signing-key \
  --kty RSA \
  --size 2048
```

**Identity for applications**

Applications should authenticate to Key Vault using Microsoft Entra ID. The most secure pattern is to use a managed identity.

Common patterns:

- **System-assigned managed identity** for a single Azure resource.
- **User-assigned managed identity** when multiple resources share the same identity.
- **Workload identity** in Kubernetes or federated scenarios when supported.

Example: enable a system-assigned identity on a VM and grant it secret access.

```bash
az vm identity assign \
  --resource-group rg-app-prod \
  --name vm-app-prod-01
```

Then assign the Key Vault role to the VM identity object ID.

This avoids storing credentials in code or configuration files.

**Operational validation**

After creation, verify that the vault is actually secure. Do not assume the deployment template enforced everything you intended.

Check these items:

- Public network access is disabled or restricted.
- The correct authorization model is enabled.
- Purge protection is on.
- Soft delete retention is appropriate.
- Private endpoint and DNS are working.
- Only required roles are assigned.

Useful validation commands:

```bash
az keyvault show \
  --name kv-secure-prod-001 \
  --resource-group rg-security-prod \
  --output table
```

```bash
az keyvault private-endpoint-connection list \
  --name kv-secure-prod-001 \
  --resource-group rg-security-prod
```

```bash
az role assignment list \
  --scope $(az keyvault show --name kv-secure-prod-001 --resource-group rg-security-prod --query id -o tsv) \
  --output table
```

**Common secure deployment pattern**

A practical production pattern looks like this:

1. Create the vault with Azure RBAC, purge protection, and public access disabled.
2. Deploy a private endpoint in the application network.
3. Configure private DNS for `privatelink.vaultcore.azure.net`.
4. Assign least-privilege roles to managed identities.
5. Store secrets and keys only after access is verified.
6. Monitor access and configuration drift.

This pattern reduces attack surface and makes access easier to audit.

**Troubleshooting and exam pitfalls**

A few details often matter in exam questions:

- **Azure RBAC vs access policies**: do not assume both are active unless the scenario says so.
- **Public access disabled**: clients on the internet cannot reach the vault, even if they have valid credentials.
- **Network ACLs**: a valid identity still fails if the network path is blocked.
- **Private Endpoint DNS**: name resolution problems often look like authentication failures.
- **Service endpoints**: they are not the same as Private Link and do not give the vault a private IP.
- **Purge protection**: once enabled, it cannot be turned off for the retention period in the same way as a simple toggle.

> Be careful with the `bypass` setting in network ACLs. `AzureServices` does not mean every Microsoft service can always reach the vault; support depends on the specific service and its configuration.

**Quick reference**

- Use **Azure RBAC** for modern authorization.
- Use **Private Endpoint** for private access.
- Disable **public network access** when possible.
- Enable **soft delete** and **purge protection** in production.
- Use **managed identities** for applications.
- Apply **least privilege** to every role assignment.

**Example secure baseline checklist**

- [ ] Vault created in the correct subscription and resource group
- [ ] Azure RBAC enabled
- [ ] Purge protection enabled
- [ ] Soft delete retention configured
- [ ] Public network access disabled or tightly restricted
- [ ] Private endpoint configured
- [ ] Private DNS zone linked
- [ ] Least-privilege roles assigned
- [ ] Secrets and keys tested from approved workloads
