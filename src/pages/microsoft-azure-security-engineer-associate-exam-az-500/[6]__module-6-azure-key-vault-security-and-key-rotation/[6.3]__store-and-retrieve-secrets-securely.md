## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Azure Key Vault is the primary Azure service for storing secrets such as passwords, API keys, connection strings, certificates, and cryptographic keys. For exam purposes, focus on the difference between *secrets*, *keys*, and *certificates*, and on how you control access, audit usage, and integrate retrieval into applications and pipelines.

A secure design keeps secrets out of source code, avoids hardcoding credentials in configuration files, and uses identity-based access wherever possible. In practice, you combine Key Vault with Microsoft Entra ID, managed identities, role-based access control (RBAC), and application-side secret retrieval patterns.

**Core concepts**

<dl>
<dt><strong>Secret</strong></dt>
<dd>A small piece of sensitive data such as a password, token, or connection string.</dd>
<dt><strong>Key</strong></dt>
<dd>A cryptographic key used for encryption, signing, or key wrapping.</dd>
<dt><strong>Certificate</strong></dt>
<dd>A public/private key pair with metadata, often used for TLS (Transport Layer Security) and client authentication.</dd>
<dt><strong>Vault</strong></dt>
<dd>The logical container that stores and governs access to secrets, keys, and certificates.</dd>
</dl>

Key Vault is deployed in a specific Azure region, and you should think about availability and recovery in terms of the vault’s region, redundancy options, and your own backup or replication strategy. Do not assume that storing a secret in Key Vault alone gives you disaster recovery across regions.

**Why use Key Vault**

- **Centralization** keeps sensitive values out of code, build logs, and local config files.
- **Access control** lets you grant narrow permissions to apps, admins, and automation.
- **Auditability** gives you logs for secret access, changes, and administrative actions.
- **Rotation support** helps you replace credentials without redeploying every app.
- **Integration** with Azure services reduces the need for embedded credentials.

A common exam pattern is choosing Key Vault over app settings when the value is sensitive and should be managed independently from the application lifecycle.

**Access models**

Azure Key Vault supports two main authorization approaches:

- **Vault access policy model**
  - You grant permissions directly on the vault.
  - It is simple and familiar for smaller deployments.
  - It can be harder to manage at scale because permissions are vault-specific.

- **Azure RBAC model**
  - You assign Azure roles such as Key Vault Secrets User or Key Vault Administrator.
  - It aligns with broader Azure governance and least privilege.
  - It is usually preferred for new deployments when you want centralized access management.

Use the model that matches your governance design, but avoid mixing approaches without a clear plan. In exam scenarios, the safest answer is often to use the least-privilege role or policy that allows only the required secret operation.

**Common secret lifecycle**

1. Create the secret in Key Vault.
2. Grant an identity permission to read it.
3. Retrieve it at runtime using the app’s identity.
4. Rotate the secret before expiry or compromise.
5. Disable or delete old versions after validation.

This lifecycle matters because secret management is not just storage. You also need controlled retrieval, versioning, rotation, and revocation.

**Creating and managing secrets with Azure CLI**

Use the Azure CLI to create, update, and inspect secrets. The secret value is stored securely in the vault, while metadata such as name, version, and content type help you manage it.

```bash
# Sign in and select the subscription
az login
az account set --subscription "<subscription-id>"

# Create a Key Vault
az keyvault create \
  --name kv-secure-demo \
  --resource-group rg-security \
  --location eastus

# Store a secret
az keyvault secret set \
  --vault-name kv-secure-demo \
  --name DbPassword \
  --value "P@ssw0rd!ChangeMe"

# Read the latest version
az keyvault secret show \
  --vault-name kv-secure-demo \
  --name DbPassword

# List secret versions
az keyvault secret list-versions \
  --vault-name kv-secure-demo \
  --name DbPassword
```

To retrieve a specific version, use the versioned secret identifier returned by Key Vault rather than assuming `--version` is a valid parameter on `show`.

```bash
# Get a specific version using the secret ID
az keyvault secret show \
  --id "https://kv-secure-demo.vault.azure.net/secrets/DbPassword/<version-id>"
```

This distinction is important because versioned retrieval is often used during rollback, validation, or staged rotation.

**Retrieving secrets in PowerShell**

PowerShell is common in administration and automation. Use the Az.KeyVault module and prefer the current secret value access pattern for your module version.

```powershell
Connect-AzAccount
Set-AzContext -Subscription "<subscription-id>"

# Get the latest secret version
$secret = Get-AzKeyVaultSecret -VaultName "kv-secure-demo" -Name "DbPassword"

# Newer usage commonly exposes the plaintext value as SecretValueText
$plainText = $secret.SecretValueText

Write-Host "Secret length: $($plainText.Length)"
```

If you need a `SecureString`, use the secure value directly rather than converting back and forth unnecessarily. For exam prep, remember that the goal is to avoid exposing the secret in logs, transcripts, or debug output.

**Application access patterns**

The most secure pattern is for the application to authenticate with its own identity and retrieve the secret at runtime. Avoid embedding credentials in source code or deployment templates.

Typical options include:

- **Managed identity**
  - Best for Azure-hosted workloads such as App Service, Azure Functions, Virtual Machines, and some container scenarios.
  - Removes the need to store client secrets for the app itself.
  - Works well with Key Vault RBAC or access policies.

- **Service principal**
  - Useful when managed identity is not available.
  - Requires careful handling of client secrets or certificates.
  - Prefer certificate-based authentication over long-lived passwords when possible.

- **Developer identity**
  - Used locally during development with `az login` or Visual Studio sign-in.
  - Should not be used as a production dependency.

A typical runtime flow is:

1. The app authenticates to Microsoft Entra ID.
2. The app requests a token for Key Vault.
3. Key Vault authorizes the request.
4. The app reads the secret and uses it in memory.

**Managed identity example**

For an Azure App Service or Azure Function, you can enable a system-assigned managed identity and grant it access to the vault.

```bash
# Enable managed identity on an App Service
az webapp identity assign \
  --name app-secure-demo \
  --resource-group rg-app

# Get the principal ID
az webapp identity show \
  --name app-secure-demo \
  --resource-group rg-app

# Grant secret read access using RBAC
az role assignment create \
  --assignee-object-id "<principal-id>" \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope "/subscriptions/<sub-id>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-secure-demo"
```

This pattern is preferred because the app never stores a reusable secret for its own authentication.

**App configuration with Key Vault references**

Some Azure services can reference Key Vault secrets directly in configuration. This reduces code changes, but you should understand the refresh behavior for the specific service.

Example App Service setting:

```text
@Microsoft.KeyVault(SecretUri=https://kv-secure-demo.vault.azure.net/secrets/DbPassword/)
```

Use this when you want the platform to resolve the secret reference rather than your code calling the Key Vault API directly. Be careful not to assume continuous automatic refresh in every service or scenario. In practice, refresh timing can depend on the platform, identity, and app lifecycle, and you may need a restart or explicit refresh mechanism.

**Secret rotation**

Rotation is a major security control because it limits the lifetime of exposed credentials. Key Vault supports versioning, which makes rotation easier to manage.

Good rotation practices:

- Create a new secret version before disabling the old one.
- Update dependent applications or services to use the new version.
- Validate the new value in production-like conditions.
- Retire the old version only after confirming the new one works.

Example rotation workflow:

```bash
# Add a new version of the same secret name
az keyvault secret set \
  --vault-name kv-secure-demo \
  --name DbPassword \
  --value "N3wP@ssw0rd!Rotated"
```

Because the secret name stays the same, applications that read the latest version automatically pick up the new value when they refresh or restart, depending on their integration pattern.

**Deletion and recovery**

Key Vault supports soft delete, which means deleted secrets are retained for a recovery period before permanent removal. During that retention window, you can recover the secret unless it has been purged.

Important points:

- **Soft delete** preserves deleted secrets temporarily.
- **Purge** permanently removes the secret.
- **Purge protection** prevents permanent deletion until the retention period expires.

> Soft-deleted secrets are recoverable during the retention window regardless of purge protection. Purge protection only blocks permanent deletion before the retention period ends.

Example recovery commands:

```bash
# Delete a secret
az keyvault secret delete \
  --vault-name kv-secure-demo \
  --name DbPassword

# Recover the deleted secret
az keyvault secret recover \
  --vault-name kv-secure-demo \
  --name DbPassword
```

This behavior is important for both operational recovery and exam questions about accidental deletion.

**Auditing and monitoring**

Use Azure Monitor and Key Vault diagnostic settings to track access and administrative activity. Logging helps you detect suspicious reads, failed authorization attempts, and unexpected secret changes.

Monitor for:

- Secret get/list operations from unexpected identities.
- Repeated authentication failures.
- Secret deletion or purge events.
- Changes to access policies or RBAC assignments.

A practical diagnostic setup sends logs to Log Analytics for querying and alerting.

```bash
az monitor diagnostic-settings create \
  --name kv-diagnostics \
  --resource "/subscriptions/<sub-id>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-secure-demo" \
  --workspace "/subscriptions/<sub-id>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-secure" \
  --logs '[{"category":"AuditEvent","enabled":true}]'
```

Use alerts for high-risk events such as purge attempts or unexpected secret access from nonproduction identities.

**Certificates and keys versus secrets**

Although this topic focuses on secrets, exam questions often mix the three Key Vault object types.

- **Secrets** store arbitrary sensitive strings.
- **Keys** support cryptographic operations and can be used by applications or Azure services.
- **Certificates** combine a key pair and certificate metadata, often with lifecycle management for TLS.

Choose the right object type:

- Use a **secret** for a database password or API token.
- Use a **key** for encryption or signing operations.
- Use a **certificate** for TLS endpoints or client certificate authentication.

**Pipeline and automation considerations**

Automation should authenticate securely and avoid long-lived credentials in pipeline variables. In Azure DevOps or GitHub Actions, prefer federated identity or service connections over embedding secrets in the pipeline definition.

Example pattern for Azure CLI in automation:

```bash
az account show
az keyvault secret show \
  --vault-name kv-secure-demo \
  --name DbPassword \
  --query value -o tsv
```

Use this only after the pipeline identity has been granted the minimum required access. Do not assume `az login --identity` works in every CI/CD environment; it is only valid when the runner itself has a managed identity available, such as certain workloads running inside Azure.

**Common security mistakes**

- Storing secrets in app settings, source control, or build logs.
- Granting broad vault permissions when only read access is needed.
- Using a shared human account instead of workload identity.
- Assuming secret rotation is automatic without testing the application path.
- Forgetting that deleted secrets may still be recoverable during retention.
- Exposing secret values in command output, transcripts, or exception messages.

**Exam-focused decision guide**

Use Key Vault when you need:

- Centralized secret storage with controlled access.
- Runtime retrieval by applications or automation.
- Versioning and rotation support.
- Audit logs for access and change tracking.

Use another approach when:

- The value is not sensitive and belongs in normal configuration.
- The application cannot reach Key Vault and no secure identity pattern exists yet.
- You need a broader secret management platform outside Azure, though Key Vault is still the Azure-native answer for AZ-500 scenarios.

**Practical architecture example**

A secure web app pattern often looks like this:

- The app runs in App Service with a system-assigned managed identity.
- The identity gets `Key Vault Secrets User` on the vault.
- The app stores only the Key Vault reference in configuration.
- The app reads the secret at runtime or through platform resolution.
- Diagnostic logs are enabled for audit and troubleshooting.
- Secret rotation is handled by creating a new version and validating the app.

This design reduces credential sprawl and supports least privilege.

**Quick reference**

| Task | Recommended approach |
|---|---|
| Store a password | Use a Key Vault secret |
| Let an Azure app read a secret | Use managed identity |
| Limit access | Use RBAC or access policies with least privilege |
| Rotate a credential | Create a new secret version |
| Recover accidental deletion | Use soft delete recovery |
| Prevent permanent deletion | Enable purge protection |

**Key takeaways**

- Key Vault is the Azure-native service for secure secret storage and retrieval.
- Use identity-based access, preferably managed identity, instead of embedded credentials.
- Understand secret versioning, soft delete, and purge protection.
- Treat retrieval behavior carefully in platform integrations; do not assume every service refreshes continuously.
- Audit access and changes so you can detect misuse and support incident response.
