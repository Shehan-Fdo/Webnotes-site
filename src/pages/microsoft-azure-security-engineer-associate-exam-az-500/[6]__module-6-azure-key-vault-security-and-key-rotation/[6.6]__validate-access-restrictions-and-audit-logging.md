## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Validating access restrictions means proving that only the intended identities, networks, and devices can reach a resource. Validating audit logging means proving that security-relevant actions are recorded, retained, and queryable. In Azure, you usually verify both together: you test the control, inspect the denial or allow result, and confirm the event appears in logs or alerts.

For AZ-500, focus on three layers:

- **Identity controls**: Microsoft Entra ID authentication, role assignments, conditional access, managed identities, and service principals.
- **Network controls**: public access, private endpoints, firewalls, NSGs, route tables, and service-specific network rules.
- **Observability controls**: diagnostic settings, Log Analytics, Azure Monitor alerts, activity logs, sign-in logs, and resource logs.

**Access restriction validation model**

A good validation flow answers four questions:

1. Can the caller authenticate?
2. Does the caller have authorization?
3. Is the network path allowed?
4. Is the action logged and alertable?

If any answer is unclear, the control is not fully validated.

**Identity and authorization validation**

Azure uses different authorization models depending on the service:

- **Azure RBAC** controls management-plane access and, for some services, data-plane access through data actions.
- **Access policies** are still used by some services, such as Azure Key Vault in policy mode.
- **Microsoft Entra ID** provides authentication, token issuance, and conditional access enforcement.

To validate identity-based access, test with the actual principal that should be allowed or denied. That principal may be:

- A user account
- A group
- A managed identity
- A service principal
- A workload identity federation subject

Use the right token scope for the target service. For example, a Key Vault data-plane call requires a token for the vault endpoint, not just a management-plane ARM token.

```bash
# Sign in as the test identity
az login

# Confirm the current account
az account show

# Check role assignments at scope
az role assignment list \
  --assignee <object-id-or-upn> \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg-name>/providers/Microsoft.KeyVault/vaults/<vault-name> \
  -o table
```

A role assignment alone does not prove access. You must test the actual operation.

- For **read access**, attempt to retrieve a secret, blob, file, or key.
- For **write access**, attempt a create, update, or delete operation.
- For **deny validation**, attempt the same operation from an unauthorized identity and confirm the expected failure.

> A management-plane read such as `az resource show` confirms the ARM resource exists, but it does not validate Key Vault secret, key, or certificate permissions.

**Key Vault validation**

Key Vault is a common exam scenario because it can use either vault access policies or Azure RBAC, depending on configuration. Validation must include both authorization and network path.

Typical checks:

- Confirm whether the vault uses **Azure RBAC** or **vault access policies**.
- Confirm whether public network access is enabled, disabled, or restricted.
- Confirm whether private endpoint access is configured.
- Confirm whether the caller has the correct data-plane permission.

Useful commands:

```bash
# Inspect vault configuration
az keyvault show -n <vault-name> -g <rg-name> -o jsonc

# Test secret access with the current identity
az keyvault secret show \
  --vault-name <vault-name> \
  --name <secret-name>

# List secrets if the identity has list permission
az keyvault secret list --vault-name <vault-name>
```

If the vault uses Azure RBAC, ensure the identity has a data-plane role such as:

- **Key Vault Secrets User**
- **Key Vault Secrets Officer**
- **Key Vault Crypto User**
- **Key Vault Administrator**

If the vault uses access policies, verify the policy grants the required secret, key, or certificate permissions.

Network-restricted vaults require extra validation:

- If public access is disabled, test from a host that reaches the private endpoint.
- Confirm private DNS resolution points the vault name to the private IP.
- Confirm the client is in the correct VNet or connected through VPN/ExpressRoute.

```bash
# Check DNS resolution from a client in the allowed network
nslookup <vault-name>.vault.azure.net

# Check connectivity to the vault endpoint
curl -I https://<vault-name>.vault.azure.net/
```

**Private endpoint and public endpoint validation**

A **private endpoint** gives the service a private IP address in your virtual network. The service itself is still a managed PaaS resource; it is not “VNet-integrated” in the same sense as a VM or app with integrated networking. The private endpoint changes how clients reach the service.

Validate the path by checking:

- The private endpoint connection state is approved.
- Private DNS resolves the service FQDN to the private IP.
- The public endpoint is disabled or blocked if that is the intended design.
- The client can reach the service only from the approved network path.

```bash
# Show private endpoint connections for a resource
az network private-endpoint-connection list \
  --id /subscriptions/<sub-id>/resourceGroups/<rg-name>/providers/Microsoft.KeyVault/vaults/<vault-name> \
  -o table
```

For services that support firewall rules, validate both allow and deny behavior:

- Allow from a trusted IP or subnet.
- Deny from an untrusted IP or subnet.
- Confirm the service logs the denied attempt.

**Network security group and route validation**

For VM and subnet-based restrictions, validate the effective network policy rather than only the configured policy. Effective state reflects inheritance and overrides.

Use the effective NSG and route table commands on the NIC:

```bash
# Show effective NSG rules for a NIC
az network nic show-effective-nsg \
  --resource-group <rg-name> \
  --name <nic-name> \
  -o jsonc

# Show effective routes for a NIC
az network nic show-effective-route-table \
  --resource-group <rg-name> \
  --name <nic-name> \
  -o jsonc
```

What to look for:

- **Deny rules** with higher priority than allow rules.
- **Service tags** such as `AzureCloud`, `Storage`, or `KeyVault` when used in rules.
- **User-defined routes** that send traffic to a firewall or NVA.
- **Next hop** values that explain why traffic is blocked or redirected.

If a VM cannot reach a service, validate in this order:

1. DNS resolution
2. Route selection
3. NSG rules
4. Firewall or proxy rules
5. Service-side network rules

**Microsoft Entra ID sign-in and audit validation**

Microsoft Entra ID provides two log types that matter for validation:

- **Sign-in logs** show authentication attempts, conditional access results, MFA (multifactor authentication) prompts, device state, and token issuance details.
- **Audit logs** show directory changes such as role assignments, app registrations, group membership changes, and policy updates.

Use sign-in logs to validate access restrictions like conditional access. Use audit logs to validate administrative changes and identity lifecycle events.

Common validation questions:

- Was the sign-in blocked by conditional access?
- Did MFA succeed or fail?
- Was the device compliant or hybrid joined?
- Was the user assigned to the correct group?
- Did a role assignment change occur before the access test?

Examples of useful queries in Log Analytics depend on your ingestion setup, but common tables include:

- `SigninLogs`
- `AuditLogs`
- `AADNonInteractiveUserSignInLogs`
- `ServicePrincipalSignInLogs`
- `ManagedIdentitySignInLogs`

```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| where UserPrincipalName == "user@contoso.com"
| project TimeGenerated, AppDisplayName, ConditionalAccessStatus, ResultType, ResultDescription, IPAddress
| order by TimeGenerated desc
```

```kusto
AuditLogs
| where TimeGenerated > ago(24h)
| where OperationName has "Add member to group" or OperationName has "Add app role assignment"
| project TimeGenerated, OperationName, Result, InitiatedBy, TargetResources
| order by TimeGenerated desc
```

Interpretation tips:

- **ConditionalAccessStatus** helps you confirm whether policy applied.
- **ResultType** and **ResultDescription** help distinguish success, failure, and interruption.
- **InitiatedBy** identifies the actor for directory changes.
- **TargetResources** shows the object affected by the change.

**Conditional access validation**

Conditional access is a core access restriction control. Validate it by testing the policy conditions and checking the sign-in result.

Common policy conditions:

- User or group assignment
- Cloud app assignment
- Device platform
- Device state or compliance
- Location
- Client app type
- Risk level
- Authentication strength

Validation approach:

1. Create or identify a policy.
2. Sign in with a test user that matches the policy scope.
3. Attempt access from a controlled device or location.
4. Review the sign-in log for the policy result.
5. Confirm the expected block, grant, or session control.

```kusto
SigninLogs
| where TimeGenerated > ago(8h)
| where UserPrincipalName == "testuser@contoso.com"
| project TimeGenerated, AppDisplayName, ConditionalAccessStatus, ConditionalAccessPolicies, ResultType
| order by TimeGenerated desc
```

If the policy uses MFA, verify:

- The prompt appears.
- The user completes the challenge.
- The log shows the MFA requirement and outcome.

If the policy blocks access, verify:

- The sign-in is denied for the intended reason.
- The policy name appears in the log.
- The block is not caused by a different control such as a disabled account or network outage.

**Managed identities and service principals**

Managed identities and service principals are common for automation and workload access. Validation should prove that the workload can obtain a token and use it only for the intended resource.

For a managed identity:

- Confirm the identity is enabled on the resource.
- Confirm the target resource grants the required role or access policy.
- Test from the workload itself, not from an admin shell.
- Review `ManagedIdentitySignInLogs` when available.

For a service principal:

- Confirm the app registration or enterprise application exists.
- Confirm the secret, certificate, or federated credential is valid.
- Confirm the service principal has the correct role assignment.
- Test token acquisition and the downstream API call.

```bash
# Get a token for a managed identity from an Azure VM or App Service
az login --identity

# Request a token for a specific resource
az account get-access-token --resource https://vault.azure.net

# Use the token in a REST call
TOKEN=$(az account get-access-token --resource https://vault.azure.net --query accessToken -o tsv)
curl -H "Authorization: Bearer $TOKEN" \
  https://<vault-name>.vault.azure.net/secrets/<secret-name>?api-version=7.4
```

If the token is valid but the call fails, check:

- Role assignment scope
- Data-plane permission
- Network restrictions
- Resource-specific firewall rules
- Token audience and API version

**Azure Monitor, diagnostic settings, and log destinations**

To validate audit logging, you must confirm that the right logs are enabled and flowing to the right destination. Azure commonly sends logs to:

- **Log Analytics workspace**
- **Storage account**
- **Event Hub**
- **Partner solution**

Diagnostic settings are the main control for resource logs. Activity logs and Entra logs are configured separately.

Typical destinations:

- **Activity log**: subscription-level control-plane events
- **Resource logs**: service-specific diagnostics
- **Sign-in logs**: Entra authentication events
- **Audit logs**: Entra directory events

Example diagnostic setting workflow:

1. Enable diagnostic settings on the resource.
2. Select the relevant log categories and metrics.
3. Send logs to a Log Analytics workspace.
4. Generate a test event.
5. Query the workspace to confirm ingestion.

```bash
# Create a diagnostic setting for a resource
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource <resource-id> \
  --workspace <workspace-resource-id> \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Common service log destinations and examples:

- **Azure Activity Log**: subscription changes, role assignments, policy changes, resource writes
- **Azure Key Vault logs**: secret, key, and certificate operations
- **Azure Firewall logs**: application, network, and DNS proxy events
- **NSG flow logs**: allowed and denied traffic flows
- **Storage logs**: read, write, delete, and authentication events
- **App Service logs**: HTTP requests, authentication, and platform events

**Azure Firewall and network logging validation**

Azure Firewall logs are often sent to Log Analytics. In modern deployments, resource-specific tables are preferred when configured, though some environments still use `AzureDiagnostics`. Do not assume a single schema across all tenants.

Validate firewall logging by:

- Generating a known allowed request.
- Generating a known denied request.
- Querying the workspace for the relevant table.
- Confirming the rule collection and action.

Example query patterns may vary by schema, so inspect the table names first:

```kusto
search "AzureFirewall"
| summarize count() by $table
```

Then query the specific table available in your workspace. For example, if resource-specific tables are enabled, you may see tables such as:

- `AZFWApplicationRule`
- `AZFWNetworkRule`
- `AZFWDnsProxy`
- `AZFWThreatIntel`

Use the schema that exists in your environment rather than hardcoding field names from another deployment.

**Activity log validation**

The Azure Activity Log records subscription-level control-plane operations. Use it to validate:

- Role assignments
- Policy assignments
- Resource creation, update, and deletion
- Lock changes
- Diagnostic setting changes

Example query:

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where OperationNameValue has_any ("MICROSOFT.AUTHORIZATION/ROLEASSIGNMENTS/WRITE", "MICROSOFT.INSIGHTS/DIAGNOSTICSETTINGS/WRITE")
| project TimeGenerated, Caller, OperationNameValue, ActivityStatusValue, ResourceGroup, ResourceProviderValue
| order by TimeGenerated desc
```

This is useful when you need to prove that a security control was changed and by whom.

**Access Reviews and governance validation**

Access Reviews help validate whether access remains appropriate over time. They are especially useful for group membership, app access, and privileged role assignments.

Use them to:

- Review guest access
- Review privileged group membership
- Review application assignments
- Remove stale access

Validation points:

- The review scope matches the intended users or groups.
- Reviewers receive notifications.
- Decisions are recorded.
- Remediation occurs after the review completes.

Azure Resource Graph can help you inventory resources and confirm whether security settings are present at scale.

```kusto
Resources
| where type =~ "microsoft.keyvault/vaults"
| project name, resourceGroup, location, properties.enableRbacAuthorization
```

Use Resource Graph when you need to confirm configuration drift across many subscriptions or resource groups.

**Alerts and action groups**

Audit logging is only useful if you can detect and respond. Azure Monitor alerts and action groups connect log evidence to response workflows.

Validate alerting by checking:

- The alert rule query or metric condition
- The scope of the alert
- The evaluation frequency and lookback window
- The action group recipients and automation targets
- The alert state after a test event

Common action group targets:

- Email
- SMS
- Push notification
- Voice call
- Azure Function
- Logic App
- Automation Runbook
- ITSM connector
- Webhook

Example log alert pattern:

```kusto
SigninLogs
| where TimeGenerated > ago(15m)
| where ResultType != 0
| where ConditionalAccessStatus == "failure"
```

When you test an alert:

1. Trigger the event.
2. Wait for the evaluation window.
3. Confirm the alert fires.
4. Confirm the action group receives the notification.
5. Confirm the incident or ticket is created if integrated.

**Practical end-to-end validation flow**

A complete validation scenario often looks like this:

1. Configure the restriction.
   - Assign the role or access policy.
   - Apply the network rule or private endpoint.
   - Enable diagnostic settings.
2. Generate a positive test.
   - Use the approved identity.
   - Use the approved network path.
   - Perform the allowed action.
3. Generate a negative test.
   - Use an unauthorized identity or network.
   - Attempt the same action.
   - Confirm the deny behavior.
4. Review logs.
   - Check sign-in logs for identity controls.
   - Check audit logs for directory changes.
   - Check resource logs for service activity.
   - Check activity logs for management-plane changes.
5. Validate alerting.
   - Confirm the alert rule matches the event.
   - Confirm the action group receives the notification.

**Troubleshooting patterns**

If access is denied unexpectedly, check:

- The token audience is correct.
- The identity has the right role or access policy.
- The scope is correct.
- Conditional access is blocking the sign-in.
- The network path is allowed.
- DNS resolves to the expected endpoint.
- The resource firewall allows the source.

If logs are missing, check:

- Diagnostic settings are enabled.
- The correct categories are selected.
- The workspace ingestion is healthy.
- The query targets the correct table.
- The retention period has not expired.
- The event occurred after logging was enabled.

If an alert does not fire, check:

- The query returns the expected event.
- The time window is long enough.
- The alert threshold is correct.
- The action group is attached.
- Suppression or mute rules are not hiding the alert.

**Exam focus**

For AZ-500, be ready to distinguish between:

- **Management-plane validation** and **data-plane validation**
- **Authentication** and **authorization**
- **Public access** and **private endpoint access**
- **Sign-in logs** and **audit logs**
- **Resource logs** and **activity logs**
- **Configured state** and **effective state**

A strong answer usually includes both the control and the evidence that proves it works.
