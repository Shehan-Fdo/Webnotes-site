## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Authorization model overview**

Azure uses different authorization layers depending on whether you are controlling the **management plane** or the **data plane**. This distinction is central to exam questions because the same resource can be managed with Azure RBAC while its data access is controlled by a different model.

- **Management plane** controls the resource itself: create, delete, configure, and assign permissions.
- **Data plane** controls the content inside the service: secrets, blobs, keys, files, messages, and similar payloads.
- **Azure RBAC** is the default authorization system for management-plane access across Azure resources.
- Some services also support Azure RBAC for data-plane access through built-in data roles.
- Other services, especially Azure Key Vault in access policy mode, use service-specific access policies for data-plane permissions.

A practical exam rule is to identify which plane the question is asking about before choosing a role or policy. If the task is “who can manage the vault,” think management-plane RBAC. If the task is “who can read a secret,” think data-plane authorization.

**Azure RBAC fundamentals**

Azure Role-Based Access Control (RBAC) is the Azure-wide authorization framework. It grants permissions by assigning a role definition to a security principal at a scope.

- **Security principal** can be a user, group, service principal, or managed identity.
- **Role definition** is the permission set, such as Reader, Contributor, or a built-in data role.
- **Scope** can be management group, subscription, resource group, resource, or sometimes a child resource.
- **Inheritance** flows from parent scopes to child scopes unless blocked by a deny assignment or a narrower override.

Common built-in role categories include:

- **Management roles** for resource administration, such as Contributor or User Access Administrator.
- **Data roles** for service content access, such as Storage Blob Data Reader or Key Vault Secrets User.
- **Privileged roles** that are often controlled with Microsoft Entra ID Privileged Identity Management (PIM).

A role assignment is effective only when all of these align:

- The principal is correct.
- The role includes the needed action or data action.
- The scope covers the target resource.
- No deny assignment blocks the access.
- The token has refreshed if the principal is using an existing session.

**Management plane versus data plane**

The same Azure service can expose both planes, but the permissions are not interchangeable.

- **Management-plane roles** use `Actions` and `NotActions` in the role definition.
- **Data-plane roles** use `DataActions` and `NotDataActions`.
- A management role can let you configure a storage account without letting you read blobs.
- A data role can let you read blobs without letting you change the storage account settings.

Example of the distinction for Storage:

- `Reader` lets you view the storage account resource.
- `Storage Blob Data Reader` lets you read blob content.
- `Contributor` can manage the account but does not automatically grant blob data access.

This distinction matters in troubleshooting because a user may appear to have “Contributor” and still receive authorization failures when reading data.

**Scope selection and inheritance**

Scope determines where the permission applies and how far it flows.

- **Management group scope** is useful for enterprise-wide governance and standardization.
- **Subscription scope** is common for platform teams and shared services.
- **Resource group scope** is useful for app teams that own a set of resources.
- **Resource scope** is best for least privilege on a single resource.
- **Child-resource scope** is used when a service exposes a subresource with its own authorization boundary.

Use broader scopes when you need consistent access across many resources. Use narrower scopes when you need least privilege and strong separation of duties.

Example scope choices:

| Scope | Typical use | Notes |
|---|---|---|
| Management group | Central security or platform governance | Best for policy and role inheritance across subscriptions |
| Subscription | Shared platform administration | Common for landing zones |
| Resource group | App team access to a workload | Easier lifecycle management |
| Resource | Single service instance access | Strongest least-privilege pattern |

**Role assignment conditions and ABAC**

Azure RBAC supports **attribute-based access control (ABAC)** conditions for some role assignments. Conditions let you restrict access based on resource attributes, request context, or other supported properties.

- Conditions are attached to a role assignment, not the role definition itself.
- They are most commonly used with data-plane scenarios such as Storage.
- They can reduce overbroad access by limiting what a principal can do within an allowed scope.
- They are not available for every role or every resource provider.

A typical use case is allowing a user to read blobs only when the blob name matches a pattern or when the request targets a specific container.

Example conceptually:

```bash
az role assignment create \
  --assignee <user-object-id> \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<account> \
  --condition "@Resource[Microsoft.Storage/storageAccounts/blobServices/containers:name] StringEquals 'finance'"
```

> Condition syntax and supported attributes vary by service and role. Validate the exact provider syntax before using it in production.

**Privileged Identity Management**

Privileged Identity Management (PIM) adds just-in-time access to Azure RBAC and Microsoft Entra ID roles.

- **Eligible assignment** means the user can activate the role when needed.
- **Active assignment** means the role is currently usable.
- Activation may require MFA, justification, approval, or time limits.
- PIM helps reduce standing privilege and is common for admin roles such as Owner, User Access Administrator, or security-related roles.

For exam scenarios, remember that a user may “have” a role in PIM but still not be able to use it until they activate it. If access fails, check whether the assignment is eligible rather than active.

**Azure Key Vault authorization models**

Azure Key Vault is a common exam topic because it supports two different data-plane authorization approaches:

- **Vault access policies**
- **Azure RBAC for Key Vault data-plane access**

The vault uses one data-plane authorization model at a time. However, Azure RBAC can still govern management-plane access to the vault resource regardless of the data-plane model.

This means:

- You can manage the vault resource with Azure RBAC.
- You can authorize secret/key/certificate operations with either access policies or Key Vault data-plane RBAC.
- The chosen data-plane model affects how you grant access to secrets, keys, and certificates.

**Key Vault access policies**

Access policies are vault-scoped permissions that are configured directly on the vault.

- They are evaluated for Key Vault data-plane operations.
- They are not the same as Azure RBAC role assignments.
- They are managed on the vault resource itself.
- They are often used in older deployments or where service-specific permission lists are preferred.

Access policies are separated by object type:

- **Keys** permissions control cryptographic key operations.
- **Secrets** permissions control secret read/write/list/delete operations.
- **Certificates** permissions control certificate lifecycle and related operations.

Common permission groups include:

- For **keys**: get, list, create, import, delete, recover, backup, restore, sign, verify, wrapKey, unwrapKey, encrypt, decrypt, release, rotate, and get rotation policy where supported.
- For **secrets**: get, list, set, delete, recover, backup, restore, purge, and restore-related operations.
- For **certificates**: get, list, create, import, update, delete, recover, backup, restore, manage contacts, get issuers, list issuers, set issuers, and related certificate lifecycle permissions.

The exact permission set varies by Key Vault feature and API version, so avoid memorizing an incomplete list as if it were universal.

Typical portal workflow:

1. Open the Key Vault.
2. Go to **Access policies**.
3. Select **Create** or **Add Access Policy**.
4. Choose the permission template or custom permissions.
5. Select the principal.
6. Save the policy.

Typical CLI workflow:

```bash
az keyvault set-policy \
  --name kv-prod-001 \
  --object-id <principal-object-id> \
  --secret-permissions get list \
  --key-permissions get list \
  --certificate-permissions get list
```

To remove a policy:

```bash
az keyvault delete-policy \
  --name kv-prod-001 \
  --object-id <principal-object-id>
```

Access policy behavior to remember:

- Policies are evaluated at the vault level.
- They are not inherited from resource groups or subscriptions.
- They are simple to reason about for a single vault.
- They can become harder to manage at scale because each vault carries its own policy set.

**Key Vault Azure RBAC model**

When Key Vault uses Azure RBAC for data-plane access, you assign built-in roles that map to secret, key, or certificate operations.

Common examples include:

- **Key Vault Secrets User** for reading secrets.
- **Key Vault Secrets Officer** for managing secrets.
- **Key Vault Crypto User** for cryptographic operations with keys.
- **Key Vault Administrator** for broad Key Vault data-plane administration.

This model is often preferred for centralized governance because it uses standard Azure role assignments, inheritance, and PIM.

Example assignment:

```bash
az role assignment create \
  --assignee <user-or-sp-object-id> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vault-name>
```

Important nuance:

- The role assignment controls data-plane access when the vault is configured for RBAC authorization.
- The vault resource itself can still be managed with Azure RBAC regardless of the data-plane model.
- If a user can open the vault in the portal but cannot read secrets, check the data-plane model and the assigned data role.

**Choosing between access policies and RBAC for Key Vault**

Use the authorization model that matches your operational needs.

- Choose **access policies** when you want vault-local, service-specific permissions and you are maintaining a smaller number of vaults.
- Choose **Azure RBAC** when you want centralized governance, inheritance, PIM support, and consistent enterprise access management.
- Use **RBAC** when you need to manage access across many vaults with the same role structure.
- Use **access policies** when a legacy application expects that model or when the organization has not migrated.

A useful mental model:

- **Management plane**: Azure RBAC always applies.
- **Data plane**: either access policies or Azure RBAC applies, depending on the vault configuration.

**Troubleshooting access failures**

When access fails, work through the problem systematically.

1. Confirm whether the failure is on the management plane or data plane.
2. Verify the authorization model in use.
3. Check the principal identity.
4. Check the scope of the assignment or policy.
5. Check whether the role or policy includes the exact operation.
6. Check for PIM activation status.
7. Wait for propagation and refresh the token.
8. Re-test with a direct CLI call.

Example: a user can see a Key Vault but cannot read a secret.

- If the vault is using access policies, confirm the user has a secret `get` permission.
- If the vault is using RBAC, confirm the user has a data role such as Key Vault Secrets User.
- If the user was just assigned access, sign out and sign back in or refresh the token.
- If the user is eligible in PIM, activate the role first.

Example CLI test for a secret read:

```bash
az keyvault secret show \
  --vault-name kv-prod-001 \
  --name app-secret
```

Expected outcomes:

- Success means the data-plane permission is correct.
- `Forbidden` or `Access denied` usually means the wrong model, missing permission, or stale token.
- `NotFound` can mean the secret does not exist or the caller lacks permission to see it, depending on the service response.

**Propagation delays and token refresh**

Authorization changes are not always immediate.

- Azure RBAC role assignments can take several minutes to propagate.
- Key Vault access policy changes can also take time to become effective.
- Existing access tokens may continue to reflect old claims until refreshed.
- PIM activation may require a new token before the role is usable.

Practical troubleshooting steps:

- Wait a few minutes after making the change.
- Sign out and sign back in.
- Use `az account clear` and authenticate again if needed.
- Re-run the access test from a fresh session.

**Deny assignments and effective access**

A deny assignment blocks access even when an allow assignment exists.

- Deny assignments are less common than allow assignments.
- They are often used by managed services or governance systems.
- They can override inherited permissions in the effective access calculation.
- They are important in enterprise environments where delegated management is used.

If a user has the expected role but still cannot access the resource, check whether a deny assignment is present at a parent scope.

**Azure Lighthouse and delegated access**

Azure Lighthouse enables delegated resource management across tenants.

- A service provider can manage customer resources without being added directly to the customer tenant.
- Delegated access is still governed by Azure RBAC and the delegation definition.
- It is common in managed service provider scenarios.
- It can affect how you interpret effective access because the managing tenant and customer tenant are different.

For exam purposes, remember that Lighthouse changes who is administering the resource, but it does not replace Azure RBAC. It extends delegated management in a controlled way.

**Practical end-to-end troubleshooting example**

Scenario: An application managed identity must read a secret from Key Vault, but the call fails.

1. Identify the identity.
   - Confirm the managed identity object ID used by the app.
2. Check the vault model.
   - Determine whether the vault uses access policies or RBAC for data-plane access.
3. Verify the permission path.
   - If access policies: ensure the identity has `get` on secrets.
   - If RBAC: ensure the identity has a role such as Key Vault Secrets User.
4. Check scope.
   - Confirm the assignment or policy targets the correct vault.
5. Check token freshness.
   - Restart the app or force a new token acquisition.
6. Test with CLI.
   - Use the identity or a privileged operator account to validate the secret read.
7. Review logs.
   - Check Key Vault diagnostic logs and Azure Activity Log for authorization failures.

Example operator validation:

```bash
az keyvault secret show \
  --vault-name kv-prod-001 \
  --name app-secret \
  --query value
```

If the operator can read the secret but the app cannot, the issue is usually identity, scope, or token-related rather than a missing secret.

**Common exam traps**

- Confusing management-plane RBAC with data-plane access.
- Assuming Contributor can read secrets or blobs.
- Treating Key Vault access policies as subscription-level permissions.
- Forgetting that PIM eligible roles are not active until activated.
- Ignoring propagation delay after a role assignment or policy change.
- Using the wrong scope for the required access.
- Assuming a role assignment alone is enough when a deny assignment exists.
- Forgetting that Azure Lighthouse changes delegation context.

**Quick comparison**

| Topic | Azure RBAC | Access policies |
|---|---|---|
| Primary use | Azure-wide authorization | Service-specific data-plane authorization |
| Scope | Management group to resource | Usually resource-specific, such as a Key Vault vault |
| Inheritance | Yes | No, vault-local |
| PIM support | Yes | No |
| ABAC conditions | Supported for some scenarios | No |
| Best for | Centralized governance | Simple vault-local control |

**Key takeaways**

- Separate **management-plane** access from **data-plane** access before choosing a permission model.
- Use Azure RBAC for broad Azure governance and many modern data-plane scenarios.
- Use Key Vault access policies when the vault is configured for that model and you need vault-local permissions.
- Check PIM, scope, propagation delay, and deny assignments when troubleshooting.
- For enterprise scenarios, include management groups, delegated access, and conditional role assignments in your mental model.
