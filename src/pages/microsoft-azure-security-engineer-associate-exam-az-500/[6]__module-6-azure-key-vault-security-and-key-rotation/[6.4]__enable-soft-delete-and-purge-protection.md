## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Azure Key Vault protects cryptographic keys, secrets, and certificates, but protection is not complete unless you also account for deletion scenarios. **Soft delete** keeps deleted objects recoverable for a retention period, while **purge protection** prevents permanent removal until that retention window expires. Together, they reduce the risk of accidental deletion, operational mistakes, and malicious cleanup after compromise.

These features matter in both administration and exam scenarios because Key Vault data is often tied to application availability, encryption, and identity workflows. If a secret or key is deleted, dependent workloads can fail immediately. If an attacker gains limited access, they may try to delete and purge vault contents to hide activity or cause disruption. Soft delete and purge protection help you recover and investigate instead of losing data permanently.

**Core concepts**

<dl>
<dt><strong>Soft delete</strong></dt>
<dd>When you delete a vault object, Azure retains it in a deleted state for a configured retention period instead of removing it immediately.</dd>
<dt><strong>Purge</strong></dt>
<dd>Permanent removal of a deleted object. After purge, recovery is no longer possible.</dd>
<dt><strong>Purge protection</strong></dt>
<dd>A safeguard that blocks purge operations during the retention period, even for users with sufficient permissions.</dd>
<dt><strong>Retention period</strong></dt>
<dd>The number of days Azure keeps deleted objects recoverable. For Key Vault, this is commonly configured between 7 and 90 days depending on the object and service behavior.</dd>
</dl>

**Why you enable both**

Soft delete alone is useful, but it still allows a privileged user to purge the deleted object before the retention period ends. Purge protection closes that gap. In practice:

- **Soft delete** gives you recovery.
- **Purge protection** gives you recovery that cannot be bypassed early.
- Together, they support compliance, incident response, and change control.

> **Important:** Purge protection is a one-way security control for a vault. Once enabled, you cannot disable it on that vault. Plan carefully before turning it on in production.

**How Key Vault deletion works**

When you delete a Key Vault object, the object is moved to a deleted state rather than being removed immediately. The object remains associated with the vault name and can usually be recovered during the retention window. The exact recovery experience depends on the object type:

- **Secrets** can be recovered if still within retention.
- **Keys** can be recovered if still within retention.
- **Certificates** are also protected by soft delete, but certificate recovery behavior can involve the underlying secret and key material depending on how the certificate was created.

The deleted object is not usable while it is in the deleted state. Applications referencing it will fail until you recover it or replace the dependency.

**Operational behavior and exam relevance**

Soft delete and purge protection are often tested in scenarios involving:

- accidental deletion of a production secret
- a compromised admin account attempting to destroy evidence
- a deployment pipeline recreating a vault with the same name
- recovery of deleted keys used for encryption at rest or application signing

Key points to remember:

- Deleted objects are retained for the configured period.
- Purge protection blocks permanent deletion during retention.
- A vault name may remain unavailable while deleted objects still exist, which can affect redeployment.
- Recovery is possible only while the deleted object is still retained.

**Enablement and configuration**

You can enable these settings through the Azure portal, Azure CLI, PowerShell, ARM templates, Bicep, or Azure Policy. The exact syntax varies by resource type and API version, but the intent is the same: ensure the vault retains deleted objects and prevents early purge.

**Azure portal**

In the portal, you typically configure these settings when creating the vault or from the vault’s settings after creation:

- **Soft delete**: enabled by default for modern Key Vault deployments.
- **Purge protection**: must be explicitly enabled.

Use the portal when you want a quick visual check of the current state or when validating a lab environment.

**Azure CLI**

The Azure CLI is useful for automation and repeatable configuration. A typical pattern is to create the vault with soft delete behavior and then enable purge protection.

```bash
az keyvault create \
  --name kv-secure-prod \
  --resource-group rg-security \
  --location eastus \
  --enable-soft-delete true \
  --enable-purge-protection true
```

If the vault already exists, you can update purge protection with a vault-level update command:

```bash
az keyvault update \
  --name kv-secure-prod \
  --resource-group rg-security \
  --enable-purge-protection true
```

To inspect the current settings:

```bash
az keyvault show \
  --name kv-secure-prod \
  --resource-group rg-security \
  --query "{softDelete:properties.enableSoftDelete,purgeProtection:properties.enablePurgeProtection,retentionDays:properties.softDeleteRetentionInDays}"
```

**PowerShell**

PowerShell is common in enterprise automation and remediation scripts.

```powershell
$kv = Get-AzKeyVault -VaultName "kv-secure-prod" -ResourceGroupName "rg-security"
$kv.EnableSoftDelete
$kv.EnablePurgeProtection
$kv.SoftDeleteRetentionInDays
```

To enable purge protection on an existing vault, use the vault update pattern supported by the Az.KeyVault module version in your environment. In practice, you often update the vault object and write it back rather than manipulating deleted objects directly.

**Bicep**

In Bicep, `softDeleteRetentionInDays` and `enablePurgeProtection` are vault-level settings. `sku` is a sibling of `properties`, not nested inside it.

```bicep
resource kv 'Microsoft.KeyVault/vaults@2023-07-01' = {
  name: 'kv-secure-prod'
  location: resourceGroup().location
  properties: {
    tenantId: tenant().tenantId
    sku: {
      family: 'A'
      name: 'standard'
    }
    accessPolicies: []
    enabledForDeployment: false
    enabledForDiskEncryption: false
    enabledForTemplateDeployment: false
    softDeleteRetentionInDays: 90
    enablePurgeProtection: true
  }
}
```

**ARM JSON**

The same schema applies in ARM JSON. Keep `sku` at the resource level and place `softDeleteRetentionInDays` and `enablePurgeProtection` under `properties`.

```json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentTemplate.json#",
  "contentVersion": "1.0.0.0",
  "resources": [
    {
      "type": "Microsoft.KeyVault/vaults",
      "apiVersion": "2023-07-01",
      "name": "kv-secure-prod",
      "location": "[resourceGroup().location]",
      "sku": {
        "family": "A",
        "name": "standard"
      },
      "properties": {
        "tenantId": "[tenant().tenantId]",
        "accessPolicies": [],
        "enabledForDeployment": false,
        "enabledForDiskEncryption": false,
        "enabledForTemplateDeployment": false,
        "softDeleteRetentionInDays": 90,
        "enablePurgeProtection": true
      }
    }
  ]
}
```

**Azure Policy**

Azure Policy is useful when you want to enforce secure defaults across subscriptions or management groups. For Key Vault, you typically audit or deny vaults that do not have purge protection enabled. The policy must evaluate the correct alias for the vault property.

A simplified example pattern looks like this:

```json
{
  "mode": "All",
  "policyRule": {
    "if": {
      "allOf": [
        {
          "field": "type",
          "equals": "Microsoft.KeyVault/vaults"
        },
        {
          "field": "Microsoft.KeyVault/vaults/enablePurgeProtection",
          "notEquals": "true"
        }
      ]
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

Use **audit** first if you need to assess impact before enforcing. Deny policies can block legitimate deployments if your templates or tooling are not aligned.

**Recovery and purge operations**

Soft delete changes the lifecycle of deleted objects. Instead of immediately removing them, Azure stores them in a deleted state. Recovery and purge are separate actions.

Typical recovery flow:

1. Identify the deleted object.
2. Recover it before the retention period expires.
3. Validate that dependent applications can access it again.

Typical purge flow:

1. Confirm the object is no longer needed.
2. Ensure you have authorization and that purge protection does not block the action.
3. Permanently remove the deleted object.

CLI examples for deleted objects vary by object type. For secrets, the deleted-state commands are separate from active secret commands.

```bash
az keyvault secret list-deleted --vault-name kv-secure-prod
az keyvault secret recover --vault-name kv-secure-prod --name app-secret
az keyvault secret purge --vault-name kv-secure-prod --name app-secret
```

For keys and certificates, use the corresponding deleted-state recover and purge commands for that object type.

**Design and governance guidance**

Use these controls as part of a broader vault protection strategy:

- **Enable purge protection** on production vaults that store critical secrets or keys.
- **Use RBAC or access policies carefully** so only trusted administrators can manage vault lifecycle operations.
- **Separate environments** so development testing does not weaken production security posture.
- **Monitor deletion events** with Azure Monitor, activity logs, and alerting.
- **Document retention expectations** so operations teams know when recovery is possible and when it is not.

**Common pitfalls**

- **Assuming soft delete equals purge protection**  
  Soft delete allows recovery, but it does not stop a privileged purge.

- **Forgetting the one-way nature of purge protection**  
  Once enabled, you cannot turn it off on that vault.

- **Using the wrong schema location in templates**  
  `sku` belongs at the resource level for Key Vault, not inside `properties`.

- **Expecting immediate name reuse**  
  A deleted vault or retained deleted objects can block reuse of the same name until cleanup completes.

- **Using active-object commands for deleted objects**  
  Recovery and purge use deleted-state operations, not the same commands you use to manage live secrets.

**Exam tips**

- If a question asks how to prevent permanent deletion of Key Vault data, choose **purge protection**.
- If a question asks how to recover deleted items, choose **soft delete** or recovery of deleted objects.
- If a question asks for the strongest protection against malicious cleanup, choose **both soft delete and purge protection**.
- If a template question involves Key Vault schema, remember that `sku` is not nested under `properties`.

**Quick reference**

| Feature | Purpose | Can recover? | Can permanently delete early? |
|---|---|---:|---:|
| Soft delete | Retains deleted objects for a period | Yes | Yes, if purge is allowed |
| Purge protection | Blocks purge during retention | Yes | No, until retention expires |

**Practical checklist**

- Confirm the vault is a production or critical-security resource.
- Enable soft delete if it is not already active.
- Enable purge protection before the vault goes live.
- Verify retention settings meet your operational and compliance needs.
- Test recovery procedures in a non-production environment.
- Add monitoring for delete, recover, and purge events.

**Troubleshooting notes**

If recovery fails, check whether:

- the retention period has expired
- the object was purged
- the vault name or object name conflicts with a retained deleted resource
- your account lacks the required permissions
- the object type uses a different deleted-state command than expected

If deployment fails, validate:

- the template API version
- the placement of `sku`
- the presence of required vault properties such as `tenantId`
- whether your policy is blocking the deployment
