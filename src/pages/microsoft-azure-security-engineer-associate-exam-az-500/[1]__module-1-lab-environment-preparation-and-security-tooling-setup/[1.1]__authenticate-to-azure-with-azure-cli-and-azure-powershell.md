## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Authentication overview**

Azure CLI and Azure PowerShell both authenticate against Microsoft Entra ID (formerly Azure Active Directory). In practice, you use them to obtain an access token for a user, service principal, or managed identity, then the tool stores or reuses that token for subsequent Azure Resource Manager operations.

The main difference is the interface and object model:

- **Azure CLI** is cross-platform and optimized for shell workflows.
- **Azure PowerShell** uses PowerShell objects and cmdlets, which is often more natural for automation in Windows-centric environments.
- Both tools can target the same subscription, tenant, and cloud environment.

A useful mental model is:

1. Authenticate to Entra ID.
2. Select the correct tenant and subscription.
3. Run management commands against Azure Resource Manager.

**Core sign-in methods**

You can authenticate in several ways depending on the scenario:

- **Interactive user sign-in** for local administration and troubleshooting.
- **Device code sign-in** when a browser is unavailable or constrained.
- **Service principal authentication** for automation and CI/CD pipelines.
- **Managed identity authentication** for Azure-hosted workloads.
- **Certificate-based authentication** when you want stronger non-interactive credential handling than a client secret.

> Use the least-privileged identity that fits the task. For automation, prefer managed identity or certificate-based service principal authentication over long-lived passwords or shared user accounts.

**Azure CLI authentication**

Azure CLI uses the `az login` family of commands. After sign-in, the CLI caches tokens locally so you do not need to reauthenticate for every command.

Common patterns:

```bash
# Interactive browser-based sign-in
az login

# Device code flow
az login --use-device-code

# Sign in to a specific tenant
az login --tenant <tenant-id>

# List subscriptions available to the signed-in identity
az account list -o table

# Select the active subscription
az account set --subscription "<subscription-name-or-id>"

# Show current account context
az account show -o json
```

Important CLI context concepts:

- **Account context** determines which subscription your commands target.
- **Tenant context** matters when the identity has access to multiple directories.
- **Cloud environment** determines whether you are targeting Azure public, Azure Government, or another sovereign cloud.

You can inspect or change the cloud environment with:

```bash
az cloud show
az cloud list -o table
az cloud set --name AzureCloud
```

If you work in a sovereign cloud, set the cloud before logging in so the CLI requests tokens from the correct authority and uses the right endpoints.

**Azure PowerShell authentication**

Azure PowerShell uses `Connect-AzAccount` for sign-in. Like the CLI, it can authenticate interactively or non-interactively and then stores context for later cmdlets.

Common patterns:

```powershell
# Interactive sign-in
Connect-AzAccount

# Sign in to a specific tenant
Connect-AzAccount -Tenant <tenant-id>

# Device code flow
Connect-AzAccount -UseDeviceAuthentication

# View current context
Get-AzContext

# List available subscriptions
Get-AzSubscription

# Set the active subscription
Set-AzContext -SubscriptionId <subscription-id>
```

PowerShell context is especially important because cmdlets often act on the current context implicitly. If you are troubleshooting, always verify the active subscription and tenant before making changes.

**Interactive user authentication**

Interactive sign-in is the simplest method for administrators working at a console or in a secure workstation.

Typical use cases:

- Ad hoc troubleshooting
- Portal-adjacent administration from a terminal
- Learning and validation in labs

Benefits:

- Easy to use
- No secret management required
- Supports MFA (multi-factor authentication) when required by policy

Limitations:

- Not suitable for unattended automation
- Can be blocked by conditional access policies if the environment is not compliant
- Requires user presence

**Device code authentication**

Device code flow is useful when the local machine cannot open a browser or when you are working over SSH.

Example:

```bash
az login --use-device-code
```

```powershell
Connect-AzAccount -UseDeviceAuthentication
```

How it works:

- The tool displays a short code and a URL.
- You open the URL on another device.
- You enter the code and complete sign-in in the browser.
- The original session receives the token after verification.

Use this method when:

- You are on a headless server
- Browser pop-up authentication is unavailable
- You need a fallback for constrained environments

**Service principal authentication**

A service principal is the application identity used by automation to authenticate to Azure. It is the standard choice for scripts, pipelines, and infrastructure automation when managed identity is not available.

CLI example:

```bash
az login --service-principal \
  --username <app-id> \
  --password <client-secret> \
  --tenant <tenant-id>
```

PowerShell example:

```powershell
$secureSecret = ConvertTo-SecureString "<client-secret>" -AsPlainText -Force
$cred = New-Object System.Management.Automation.PSCredential("<app-id>", $secureSecret)
Connect-AzAccount -ServicePrincipal -Tenant <tenant-id> -Credential $cred
```

You can also authenticate with a certificate:

```bash
az login --service-principal \
  --username <app-id> \
  --certificate /path/to/cert.pem \
  --tenant <tenant-id>
```

Why use a service principal:

- It supports non-interactive automation.
- It can be scoped to a subscription, resource group, or specific role assignment.
- It avoids using a human user account in scripts.

Security considerations:

- Prefer certificates over client secrets when possible.
- Store secrets in a secure secret store such as Azure Key Vault.
- Rotate credentials regularly.
- Assign only the roles required for the automation task.

**Managed identity authentication**

Managed identities let Azure resources authenticate without storing credentials in code or configuration. Azure creates and manages the identity lifecycle.

There are two common types:

- **System-assigned managed identity**: tied to one Azure resource and deleted with it.
- **User-assigned managed identity**: created as a standalone resource and can be attached to multiple resources.

CLI examples:

```bash
# Sign in from an Azure resource that has a managed identity
az login --identity

# Use a user-assigned managed identity
az login --identity --username <client-id-or-resource-id>
```

PowerShell examples:

```powershell
Connect-AzAccount -Identity
Connect-AzAccount -Identity -AccountId <client-id-or-resource-id>
```

Use managed identity when:

- The code runs in Azure, such as on a VM, App Service, Function App, or Automation account
- You want to eliminate stored credentials
- You need a strong default for cloud-native automation

A common pattern is to combine managed identity with role-based access control (RBAC) so the workload can access only the resources it needs.

**Tenant, subscription, and context management**

Authentication does not automatically mean you are operating in the right scope. You must verify the tenant and subscription context before making changes.

Key concepts:

- **Tenant** is the Microsoft Entra directory that owns identities and directory objects.
- **Subscription** is the billing and management boundary for Azure resources.
- **Context** is the current working scope used by the CLI or PowerShell session.

CLI context commands:

```bash
az account show
az account list -o table
az account set --subscription <subscription-id>
az account clear
```

PowerShell context commands:

```powershell
Get-AzContext
Get-AzSubscription
Set-AzContext -SubscriptionId <subscription-id>
Clear-AzContext -Force
```

If you work across multiple tenants or subscriptions, explicitly set the context in scripts rather than relying on whatever the interactive session last used.

**Practical verification workflow**

When you sign in, verify identity and scope before running administrative commands.

1. Authenticate with the appropriate method.
2. Confirm the tenant and subscription.
3. Confirm the cloud environment if you are not in Azure public.
4. Run a read-only command first.
5. Make changes only after validation.

Example CLI workflow:

```bash
az login
az account show -o table
az group list -o table
```

Example PowerShell workflow:

```powershell
Connect-AzAccount
Get-AzContext
Get-AzResourceGroup
```

This reduces the risk of modifying the wrong subscription, which is one of the most common operational mistakes in multi-subscription environments.

**Automation examples**

For scripts, avoid interactive authentication. Use a non-interactive identity and make the context explicit.

CLI script example:

```bash
#!/usr/bin/env bash
set -euo pipefail

az login --service-principal \
  --username "$AZURE_CLIENT_ID" \
  --password "$AZURE_CLIENT_SECRET" \
  --tenant "$AZURE_TENANT_ID"

az account set --subscription "$AZURE_SUBSCRIPTION_ID"
az group list -o table
```

PowerShell script example:

```powershell
$ErrorActionPreference = 'Stop'

$secret = ConvertTo-SecureString $env:AZURE_CLIENT_SECRET -AsPlainText -Force
$cred = [PSCredential]::new($env:AZURE_CLIENT_ID, $secret)

Connect-AzAccount -ServicePrincipal -Tenant $env:AZURE_TENANT_ID -Credential $cred
Set-AzContext -SubscriptionId $env:AZURE_SUBSCRIPTION_ID

Get-AzResourceGroup
```

For Azure-hosted workloads, prefer managed identity:

```powershell
Connect-AzAccount -Identity
Set-AzContext -SubscriptionId $env:AZURE_SUBSCRIPTION_ID
```

**Troubleshooting and common errors**

Authentication issues often come from scope, policy, or identity mismatches rather than the login command itself.

Common problems:

- **Wrong tenant**: the identity exists in another directory or the subscription belongs to a different tenant.
- **Wrong subscription**: the session is authenticated, but the active subscription is not the one you expect.
- **Insufficient permissions**: the identity authenticated successfully but lacks RBAC permissions.
- **Conditional Access**: policy blocks the sign-in method or requires compliant devices/MFA.
- **Expired secret or certificate**: service principal credentials are no longer valid.
- **Managed identity not enabled**: the Azure resource has no identity assigned, or the identity lacks role assignments.

Useful checks:

```bash
az account show
az role assignment list --assignee <object-id> -o table
```

```powershell
Get-AzContext
Get-AzRoleAssignment -ObjectId <object-id>
```

If a command fails with authorization errors, distinguish between:

- **Authentication failure**: you could not sign in.
- **Authorization failure**: you signed in, but the identity lacks permission.

That distinction helps you decide whether to fix credentials, tenant selection, or RBAC.

**Security best practices**

- Use **MFA** for human sign-ins.
- Use **managed identity** for Azure workloads whenever possible.
- Use **service principals** only for automation that cannot use managed identity.
- Prefer **certificate-based** authentication over client secrets for long-lived automation.
- Store secrets in **Azure Key Vault** rather than in scripts or source control.
- Scope permissions narrowly with RBAC.
- Review and rotate credentials regularly.
- Clear context when switching between environments to avoid accidental changes.

**Quick comparison**

| Scenario | Recommended method | Why |
|---|---|---|
| Local admin on a workstation | Interactive sign-in | Simple and supports MFA |
| Headless server or SSH session | Device code | No browser required |
| CI/CD pipeline | Service principal or federated identity if available | Non-interactive automation |
| Azure VM or App Service | Managed identity | No stored credentials |
| Multi-tenant administration | Explicit tenant and subscription selection | Prevents scope mistakes |

**Exam focus**

For AZ-500, focus on choosing the right authentication method for the scenario and understanding the security implications.

- Know when to use `az login` versus `Connect-AzAccount`.
- Understand the difference between user sign-in, service principal sign-in, and managed identity.
- Be able to explain tenant, subscription, and context.
- Recognize why managed identity is preferred for Azure-hosted automation.
- Remember that authentication and authorization are separate concerns.
