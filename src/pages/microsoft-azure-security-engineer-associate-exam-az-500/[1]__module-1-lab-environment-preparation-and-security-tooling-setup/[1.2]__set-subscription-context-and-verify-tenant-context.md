## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Subscription and tenant context**

Azure resources are always evaluated in the context of a **subscription** and a **tenant**. The subscription is the billing and management boundary for resources, while the tenant is the Microsoft Entra ID directory that owns identities, groups, and many access decisions. In security work, you often need to confirm both before making changes, because the wrong context can lead to failed deployments, incorrect access reviews, or changes in the wrong environment.

A common AZ-500 task is to authenticate, select the correct subscription, and verify that the tenant matches the directory you intend to manage. This matters especially in organizations with multiple subscriptions, management groups, or cross-tenant administration.

- **Subscription**: the Azure container where resources live and RBAC is applied.
- **Tenant**: the Microsoft Entra ID directory that stores identities and directory-scoped objects.
- **Context**: the active account, subscription, and tenant used by your CLI or PowerShell session.

**Why context verification matters**

Security operations often fail silently when you are signed into the right account but the wrong subscription. For example, a role assignment, policy update, or Key Vault access change may succeed in one subscription and appear missing in another. Verifying context before you act reduces troubleshooting time and prevents accidental changes.

> <mark>Always confirm both subscription and tenant before running security-impacting commands.</mark>

**Azure CLI context management**

Azure CLI uses a current account context that includes the active subscription and tenant. After sign-in, you can list subscriptions and set the one you want to target. This is the most common workflow when you manage multiple subscriptions or tenants.

```bash
az login
az account list --output table
az account set --subscription "Contoso-Prod"
az account show --output jsonc
```

- `az login` authenticates your session.
- `az account list` shows subscriptions available to the signed-in identity.
- `az account set` changes the active subscription for subsequent commands.
- `az account show` confirms the current subscription, tenant, and user context.

If you need to target a specific tenant during sign-in, use the tenant identifier. This is useful when your account has access to multiple directories.

```bash
az login --tenant 00000000-0000-0000-0000-000000000000
az account set --subscription "00000000-0000-0000-0000-000000000000"
```

The CLI stores the active context locally for the session and profile. If you switch subscriptions frequently, always re-check the current context before making changes.

**Useful Azure CLI verification commands**

Use these commands to inspect the active environment and reduce ambiguity:

```bash
az account show --query "{subscription:name, subscriptionId:id, tenantId:tenantId, user:user.name}" --output table
az account list --all --output table
az account tenant list --output table
```

- `tenantId` identifies the Microsoft Entra tenant associated with the current account context.
- `subscriptionId` identifies the active Azure subscription.
- `az account tenant list` helps when your identity can access multiple tenants.

**Azure PowerShell context management**

Azure PowerShell uses a context object that tracks the current account, subscription, and tenant. You typically sign in with `Connect-AzAccount`, then select the subscription with `Set-AzContext`. This is the preferred approach when you automate administrative tasks in PowerShell.

```powershell
Connect-AzAccount
Get-AzSubscription
Set-AzContext -Subscription "Contoso-Prod"
Get-AzContext
```

- `Connect-AzAccount` authenticates to Azure.
- `Get-AzSubscription` lists subscriptions available to the current identity.
- `Set-AzContext` selects the active subscription.
- `Get-AzContext` verifies the current subscription and tenant.

If you need to authenticate against a specific tenant, include the tenant identifier.

```powershell
Connect-AzAccount -Tenant 00000000-0000-0000-0000-000000000000
Set-AzContext -Subscription "00000000-0000-0000-0000-000000000000"
```

PowerShell can also persist context locally, which is convenient but risky if you assume the session is still pointed at the intended subscription. Reconfirm context after reconnecting or after running scripts that may change it.

**Verifying tenant context**

Tenant verification is essential when you work across directories, especially in delegated administration or multi-tenant organizations. A subscription belongs to one tenant, and many directory-scoped operations require the correct tenant context.

Common ways to verify tenant context include:

- Checking the tenant ID in `az account show`.
- Checking the tenant in `Get-AzContext`.
- Confirming the directory in the Azure portal account menu.
- Matching the tenant ID against the expected Microsoft Entra directory.

Example CLI output fields to inspect:

```bash
az account show --query "{name:name, id:id, tenantId:tenantId, user:user.name}" --output jsonc
```

Example PowerShell output fields to inspect:

```powershell
Get-AzContext | Select-Object Name, Subscription, Tenant, Account
```

If the tenant does not match your expectation, stop and re-authenticate. Many access issues are actually context issues.

**Practical workflow for safe administration**

Use a repeatable sequence before making changes in Azure:

1. Sign in to the correct identity.
2. List available subscriptions.
3. Select the intended subscription.
4. Verify the tenant ID.
5. Confirm the active context again before running changes.

Example Azure CLI workflow:

```bash
az login --tenant 00000000-0000-0000-0000-000000000000
az account list --output table
az account set --subscription "Contoso-Prod"
az account show --output table
```

Example PowerShell workflow:

```powershell
Connect-AzAccount -Tenant 00000000-0000-0000-0000-000000000000
Get-AzSubscription | Format-Table Name, Id, TenantId
Set-AzContext -Subscription "Contoso-Prod"
Get-AzContext | Format-List
```

This sequence is especially important before:

- Assigning roles with Azure RBAC.
- Modifying Azure Policy assignments.
- Managing Key Vault access or secrets.
- Creating or deleting security resources.
- Running automation against production subscriptions.

**Common mistakes and troubleshooting**

A frequent mistake is assuming the signed-in account determines the target subscription. In reality, the active subscription context controls where most commands operate. Another common issue is using a subscription name that exists in multiple tenants or environments, which can make the wrong target look correct at first glance.

- **Wrong subscription selected**: commands succeed, but changes appear in the wrong environment.
- **Wrong tenant selected**: directory-scoped operations fail or show incomplete data.
- **Stale context**: a previous session or script left the shell pointed at an unexpected subscription.
- **Ambiguous names**: subscription display names are not unique enough for safe automation.

> If a command affects identity, access, or policy, verify the subscription ID and tenant ID rather than relying on display names alone.

**Automation and scripting considerations**

In scripts, set the context explicitly at the start rather than assuming the user’s current session is correct. This makes automation predictable and safer in shared environments.

```bash
#!/usr/bin/env bash
set -euo pipefail

az login --tenant "$TENANT_ID"
az account set --subscription "$SUBSCRIPTION_ID"

CURRENT_SUB=$(az account show --query id -o tsv)
CURRENT_TENANT=$(az account show --query tenantId -o tsv)

echo "Subscription: $CURRENT_SUB"
echo "Tenant: $CURRENT_TENANT"
```

```powershell
param(
    [Parameter(Mandatory=$true)]
    [string]$TenantId,

    [Parameter(Mandatory=$true)]
    [string]$SubscriptionId
)

Connect-AzAccount -Tenant $TenantId | Out-Null
Set-AzContext -Subscription $SubscriptionId | Out-Null

$ctx = Get-AzContext
$ctx | Select-Object Subscription, Tenant, Account
```

For automation, prefer subscription IDs over names when possible. IDs are unambiguous and reduce the risk of targeting the wrong environment.

**Portal verification**

In the Azure portal, you can confirm context by checking the directory and subscription selectors near the top of the interface. This is useful when you are troubleshooting a command-line issue or validating what a colleague sees in the browser.

- Confirm the active directory in the account or directory switcher.
- Confirm the selected subscription in the subscription filter.
- Reopen the resource blade if you changed context, because portal views may cache prior selections.

**Exam focus**

For AZ-500, expect questions that test whether you can:

- Select the correct subscription before performing security tasks.
- Verify the tenant associated with the current session.
- Distinguish between subscription-scoped and tenant-scoped operations.
- Use Azure CLI or Azure PowerShell to confirm context.
- Avoid accidental changes in the wrong environment.

A strong answer usually includes both the action and the verification step. For example, setting the subscription is not enough unless you also confirm the tenant and active context afterward.

**Quick reference**

| Task | Azure CLI | Azure PowerShell |
|---|---|---|
| Sign in | `az login` | `Connect-AzAccount` |
| Sign in to tenant | `az login --tenant <tenantId>` | `Connect-AzAccount -Tenant <tenantId>` |
| List subscriptions | `az account list` | `Get-AzSubscription` |
| Set subscription | `az account set --subscription <id-or-name>` | `Set-AzContext -Subscription <id-or-name>` |
| Verify context | `az account show` | `Get-AzContext` |

**Key takeaways**

- The active **subscription** determines where most Azure commands apply.
- The **tenant** determines the directory context for identities and directory-scoped operations.
- Always verify both before making security changes.
- Use subscription IDs and tenant IDs for safer automation.
- Recheck context after sign-in, script execution, or environment switching.
