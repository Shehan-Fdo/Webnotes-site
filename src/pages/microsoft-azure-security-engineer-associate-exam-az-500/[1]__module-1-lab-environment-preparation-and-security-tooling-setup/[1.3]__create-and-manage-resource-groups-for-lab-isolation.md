## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Resource groups for lab isolation**

An Azure resource group is a logical container for related resources that share a lifecycle. For lab isolation, you use resource groups to separate environments, reduce blast radius, and apply governance consistently. In AZ-500 scenarios, this matters because you often need to build secure test environments that can be created, audited, and removed without affecting production.

A resource group does not provide network isolation by itself. It is an administrative boundary, not a security boundary. You still need virtual networks, subnets, network security groups, private endpoints, and identity controls to isolate traffic and access.

- Use a dedicated resource group per lab, project, or environment.
- Keep resources with the same lifecycle together.
- Apply tags to support ownership, cost tracking, and cleanup.
- Use Azure role-based access control <abbr title="Role-Based Access Control">RBAC</abbr> at the resource group scope when you want consistent permissions across the lab.

**Why resource groups matter in security labs**

Resource groups help you manage temporary or segmented environments safely. You can grant a team access to only the lab resource group, monitor activity there, and delete the entire group when the lab ends. This reduces the risk of orphaned resources and accidental changes outside the lab.

Common lab isolation goals include:

- Separating student, tester, or incident-response environments from production.
- Limiting who can create, modify, or delete lab assets.
- Applying consistent policy and tagging.
- Simplifying teardown after exercises or investigations.

**Core properties and behavior**

A resource group has a location, but that location stores metadata for the group rather than forcing all resources to live there. Resources inside the group can exist in different regions, although you should usually keep them close when latency, compliance, or cost matters.

Important behaviors:

- A resource can belong to only one resource group at a time.
- Deleting a resource group deletes the resources inside it.
- Moving resources between groups is possible for many resource types, but not all.
- Locks, RBAC assignments, and policies can be scoped to the resource group.

> Deleting a resource group is destructive. Always verify that no shared or production resources are included before removal.

**Designing a lab isolation model**

A practical lab design uses one resource group per isolated unit. For example, you might create separate groups for networking, compute, and shared services only when the lifecycle differs. If the lab is temporary, a single group is often easier to manage and clean up.

Typical patterns:

- **Single lab group**: best for short-lived exercises and simple teardown.
- **Per-team group**: useful when multiple teams need separate permissions.
- **Per-environment group**: useful for dev, test, and validation separation.
- **Per-workload group**: useful when components have different lifecycles or owners.

Example naming pattern:

```text
rg-lab-az500-001
rg-lab-az500-student01
rg-lab-incident-response
```

A consistent naming convention helps you identify purpose, owner, and environment quickly. Pair names with tags such as `Environment=Lab`, `Owner=SecurityTeam`, and `ExpiryDate=2026-05-01`.

**Creating resource groups**

You can create resource groups through the Azure portal, Azure CLI, PowerShell, or infrastructure as code. For repeatable lab isolation, CLI and templates are usually better than manual portal steps.

Azure CLI example:

```bash
az group create \
  --name rg-lab-az500-001 \
  --location eastus \
  --tags Environment=Lab Owner=Webnotes Purpose=AZ500 ExpiryDate=2026-05-01
```

Azure PowerShell example:

```powershell
New-AzResourceGroup `
  -Name "rg-lab-az500-001" `
  -Location "eastus" `
  -Tag @{ Environment="Lab"; Owner="Webnotes"; Purpose="AZ500"; ExpiryDate="2026-05-01" }
```

Bicep example:

```bicep
targetScope = 'subscription'

param rgName string = 'rg-lab-az500-001'
param location string = 'eastus'

resource rg 'Microsoft.Resources/resourceGroups@2021-04-01' = {
  name: rgName
  location: location
  tags: {
    Environment: 'Lab'
    Owner: 'Webnotes'
    Purpose: 'AZ500'
    ExpiryDate: '2026-05-01'
  }
}
```

**Managing access with RBAC**

Use Azure RBAC to control who can work in the lab. Assign roles at the resource group scope when the same permissions should apply to all lab resources. This is simpler than assigning permissions to each resource individually.

Common role choices:

- **Reader**: view-only access for auditors or observers.
- **Contributor**: create and manage resources, but not grant access.
- **Owner**: full control, including access management.
- **User Access Administrator**: manage role assignments without full resource control.

CLI example for a scoped assignment:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role Contributor \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-lab-az500-001
```

Best practices:

- Grant the minimum role needed for the lab task.
- Prefer group-based assignments over individual users.
- Use time-bound access when possible through privileged identity management.
- Review assignments before and after the lab.

**Using Azure Policy and tags**

Azure Policy helps enforce standards in lab resource groups. You can require tags, restrict allowed locations, or deny unsupported resource types. This is useful when you want every lab to follow the same security baseline.

Examples of useful policies:

- Require `Owner` and `Environment` tags.
- Restrict deployments to approved regions.
- Deny public IP creation in sensitive labs.
- Audit resources that do not use approved naming patterns.

Tagging supports operational control:

- `Environment=Lab`
- `Owner=SecurityTeam`
- `CostCenter=SecOps`
- `ExpiryDate=2026-05-01`
- `DataClassification=Internal`

Tags do not enforce security by themselves, but they improve reporting, automation, and cleanup.

**Locks and protection**

Resource locks help prevent accidental deletion or modification. In lab isolation, use locks carefully because they can interfere with teardown automation.

Lock types:

- **CanNotDelete**: users can read and modify, but not delete.
- **ReadOnly**: users can only read; changes are blocked.

Example CLI lock:

```bash
az lock create \
  --name rg-protect-lock \
  --lock-type CanNotDelete \
  --resource-group rg-lab-az500-001
```

Use locks when a lab contains shared or expensive resources. Remove them before automated cleanup if the teardown process must delete the group.

**Moving and reusing resources**

You can move many Azure resources between resource groups, but the move must be supported by the resource type and the move constraints. This is useful when you need to reclassify a lab resource or separate a shared component from a temporary exercise.

Before moving:

- Check whether the resource type supports moves.
- Verify dependencies such as NICs, disks, and public IPs.
- Plan for downtime or configuration updates.
- Reconfirm RBAC and policy scope after the move.

CLI example:

```bash
az resource move \
  --destination-group rg-lab-az500-002 \
  --ids /subscriptions/<subscription-id>/resourceGroups/rg-lab-az500-001/providers/Microsoft.Compute/virtualMachines/vm01
```

**Cleanup and lifecycle management**

Lab isolation is only effective if you can clean up reliably. Deleting the resource group is the fastest way to remove all contained resources, but only after you confirm that nothing important is shared.

Recommended cleanup flow:

1. Review the resource group contents.
2. Remove locks if they block deletion.
3. Export or save any needed artifacts.
4. Delete the resource group.
5. Verify that dependent resources were removed.

CLI example:

```bash
az group delete \
  --name rg-lab-az500-001 \
  --yes \
  --no-wait
```

If deletion fails, common causes include:

- A lock still exists.
- A resource has a dependency outside the group.
- A resource provider is still completing an operation.
- A resource type requires manual cleanup first.

**Practical lab topology**

A simple isolated lab might look like this:

- One resource group for the lab boundary.
- One virtual network with a dedicated subnet.
- One network security group attached to the subnet.
- One or more virtual machines for testing.
- Optional private endpoints or storage accounts for controlled access.

**Common exam and operational pitfalls**

- Treating a resource group as a security boundary instead of a management boundary.
- Mixing shared production resources into a temporary lab group.
- Forgetting to tag resources, which makes cleanup and cost tracking harder.
- Assigning overly broad RBAC roles to lab users.
- Leaving locks in place and then wondering why deletion fails.
- Assuming all resources can be moved between groups without checking dependencies.

**Key takeaways**

Resource groups are the foundation for organizing isolated Azure labs, but they do not replace network or identity security. Use them to group resources with the same lifecycle, apply RBAC and policy consistently, and enable safe teardown. For AZ-500, focus on how resource groups support governance, access control, and operational cleanup in secure lab scenarios.
