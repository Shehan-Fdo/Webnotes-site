## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Route tables and UDR fundamentals**

Azure route tables let you influence how traffic leaves a subnet. A route table contains zero or more routes, and each route can be a system route override or a user-defined route (UDR). You use them when the default Azure routing behavior does not match your security, inspection, or hybrid connectivity design.

A route table is associated at the **subnet** level, not the NIC or VM level. That means every resource in the subnet inherits the subnet’s effective routing behavior unless another Azure feature changes the path. If a subnet has no custom route table, Azure uses only system routes and any applicable BGP (Border Gateway Protocol) routes.

**Why you use UDRs**

- Force spoke traffic through a firewall or NVA (network virtual appliance).
- Override default internet egress with controlled inspection.
- Direct on-premises or cross-premises traffic through a hub.
- Blackhole unwanted destinations by routing them to `None`.
- Keep routing deterministic in hub-and-spoke networks.

**Core route sources**

Azure evaluates multiple route sources in the effective route table:

- **User-defined routes**: Routes you create in a route table.
- **BGP routes**: Routes learned from VPN Gateway or ExpressRoute.
- **System routes**: Azure-provided default routes for local VNet, peering, internet, and platform services.

For a given destination, Azure uses **longest prefix match** first. If multiple routes match the same prefix length, Azure prefers **UDR over BGP over system routes**. Do not assume any extra tie-break rule beyond that documented precedence.

**Common route types and next hops**

| Next hop type | Typical use | Notes |
|---|---|---|
| `VirtualAppliance` | Send traffic to a firewall, proxy, or NVA | Requires a valid next-hop IP address |
| `VirtualNetworkGateway` | Send traffic to a VPN or ExpressRoute gateway | Common in hybrid designs |
| `VnetLocal` | Explicitly route to the local virtual network | System routes already cover local VNet traffic in most cases |
| `Internet` | Force traffic to the public internet | Often used to override a more specific route design |
| `None` | Drop traffic | Useful for deny-by-routing patterns |

> A `None` route is a routing control, not a security boundary by itself. Use it with NSGs (network security groups), Azure Firewall, or other controls.

**Route table limits and scaling considerations**

Azure route tables have practical limits that matter in design and exams. The exact numbers can vary by service updates, but the commonly tested guidance is:

- A route table supports up to **400 routes**.
- A subnet can be associated with **one route table**.
- Route tables are a shared control for all resources in the subnet.
- Large numbers of UDRs can increase operational complexity and make troubleshooting harder.

When you approach scale limits, consider:

- Summarizing prefixes instead of creating many narrow routes.
- Using hub-and-spoke patterns to centralize inspection.
- Reducing route sprawl by designing address spaces carefully.
- Verifying whether BGP can carry some of the routing burden instead of static UDRs.

**How Azure builds the effective route table**

The effective route table is the result of combining route sources for a NIC in a subnet. You typically inspect it on the **NIC** because that is where Azure shows the final effective routes for the attached workload.

- **System routes** are always present.
- **Peering routes** appear when virtual networks are peered.
- **BGP routes** appear when a gateway advertises routes.
- **UDRs** appear when a subnet has an associated route table.

Peering-related routes are system-managed routes that allow traffic between peered VNets. They are not the same as UDRs, and they do not replace BGP routes. In practice, the effective route table can contain all three sources at once, and Azure applies the documented precedence when multiple routes match.

**Association and dissociation behavior**

When you associate a route table to a subnet, Azure immediately applies it to new and existing resources in that subnet. When you remove the association, the subnet falls back to Azure system routes and any applicable BGP routes.

- If you **associate** a route table, its routes become part of the subnet’s effective routing.
- If you **dissociate** a route table, the UDRs stop affecting traffic.
- If no custom route table exists, the subnet uses default Azure routing behavior.

This is important in change control: removing a route table can restore direct internet or peering paths that you previously forced through inspection.

**Creating and managing route tables with Azure CLI**

Use Azure CLI when you want repeatable infrastructure changes. The exact parameter set can vary by CLI version, so verify with `az network route-table -h` if needed.

```bash
# Create a route table
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke1 \
  --location eastus

# Add a UDR that sends a spoke subnet's default traffic to a firewall
az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke1 \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4

# Add a blackhole route for a blocked prefix
az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke1 \
  --name block-partner-net \
  --address-prefix 10.50.0.0/16 \
  --next-hop-type None
```

If you need to disable BGP route propagation, confirm the current CLI syntax for your version. A common pattern is to update the route table property after creation:

```bash
az network route-table update \
  --resource-group rg-network \
  --name rt-spoke1 \
  --disable-bgp-route-propagation true
```

**PowerShell examples**

PowerShell is common in Azure administration and exam scenarios. Use the route table object and subnet configuration carefully, because module syntax can differ slightly by version.

```powershell
# Create a route table
$routeTable = New-AzRouteTable `
  -ResourceGroupName "rg-network" `
  -Location "eastus" `
  -Name "rt-spoke1" `
  -DisableBgpRoutePropagation $true

# Add a route to send default traffic to Azure Firewall
Add-AzRouteConfig `
  -Name "default-to-firewall" `
  -AddressPrefix "0.0.0.0/0" `
  -NextHopType "VirtualAppliance" `
  -NextHopIpAddress "10.0.0.4" `
  -RouteTable $routeTable

# Save the route table
$routeTable | Set-AzRouteTable
```

To associate the route table to a subnet, use the subnet configuration pattern supported by your Az.Network version. One common approach is to set the subnet’s route table ID:

```powershell
$vnet = Get-AzVirtualNetwork -ResourceGroupName "rg-network" -Name "vnet-spoke1"

Set-AzVirtualNetworkSubnetConfig `
  -VirtualNetwork $vnet `
  -Name "workload-subnet" `
  -AddressPrefix "10.1.1.0/24" `
  -RouteTableId $routeTable.Id

$vnet | Set-AzVirtualNetwork
```

**Portal workflow**

The Azure portal is useful for quick validation and troubleshooting. The exact blade labels can change over time, but the route table experience usually includes a setting similar to **Propagate gateway routes** for BGP propagation control.

Typical workflow:

1. Open the route table resource.
2. Add routes under the route table’s routes section.
3. Associate the route table to a subnet.
4. Review the subnet’s effective routes from the NIC or network interface view.
5. Adjust propagation or next-hop settings if traffic does not follow the intended path.

**Hub-and-spoke with Azure Firewall example**

This is one of the most exam-relevant patterns. You place Azure Firewall in the hub and force spoke subnets to send outbound and cross-spoke traffic through it.

**Topology**

- Hub VNet: contains Azure Firewall and gateway resources.
- Spoke VNet: contains application workloads.
- Spoke subnet route table: sends `0.0.0.0/0` and other relevant prefixes to the firewall.
- Firewall rules: allow approved outbound traffic.
- Return path: ensure the hub can route back to the spoke prefixes.

**Example routing design**

- Spoke subnet default route `0.0.0.0/0` -> `VirtualAppliance` at firewall private IP `10.0.0.4`
- On-premises prefix `172.16.0.0/16` -> `VirtualAppliance` at firewall private IP `10.0.0.4`
- Optional blocked prefix `203.0.113.0/24` -> `None`

```bash
az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke1 \
  --name default-to-fw \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke1 \
  --name onprem-to-fw \
  --address-prefix 172.16.0.0/16 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

**Return path considerations**

- The firewall must know how to reach the spoke subnet.
- The hub must have routes back to the spoke address space.
- If the firewall is stateful, return traffic usually follows the established session, but asymmetric routing can still break flows.
- If you use a VPN or ExpressRoute gateway, make sure the gateway and firewall routing do not conflict.

**When to use BGP versus UDR**

BGP is dynamic and useful for hybrid connectivity. UDRs are static and useful for explicit control.

- Use **BGP** when you want routes learned automatically from on-premises or a gateway.
- Use **UDR** when you need to override or force a specific path.
- Use both when you want dynamic reachability plus a security inspection point.

A common pattern is to let BGP advertise on-premises prefixes, then add a UDR for `0.0.0.0/0` to force internet-bound traffic through a firewall.

**Subnet delegation and service exceptions**

Some Azure services and delegated subnets have routing restrictions or special behavior. You should check service documentation before applying aggressive UDRs.

- Delegated subnets may require specific platform routes to function correctly.
- Managed services can ignore or restrict custom routing in some scenarios.
- Private endpoints and service endpoints can change how traffic reaches PaaS services.
- Azure platform dependencies may require access to service IPs or Azure infrastructure addresses.

> Be careful when forcing all traffic through an NVA in a subnet that hosts a delegated service. Some services require direct platform connectivity and may fail if you blackhole or redirect required routes.

**Verification and troubleshooting**

Use the effective route table to confirm what Azure will actually do. The most useful inspection point is the NIC attached to the workload.

**Portal checks**

- Open the NIC.
- Select **Effective routes**.
- Inspect:
  - **Address prefix**
  - **Next hop type**
  - **Next hop IP address**
  - **Source** of the route

Look for unexpected overlaps, such as a broad default route overriding a more specific path you intended to keep local.

**CLI checks**

```bash
az network nic show-effective-route-table \
  --resource-group rg-network \
  --name nic-spoke1
```

**PowerShell checks**

```powershell
Get-AzEffectiveRouteTable `
  -NetworkInterfaceName "nic-spoke1" `
  -ResourceGroupName "rg-network"
```

**Common troubleshooting steps**

1. Confirm the subnet is associated with the intended route table.
2. Check whether BGP propagation is enabled or disabled as expected.
3. Review the effective route table on the NIC.
4. Verify the next-hop IP is reachable and belongs to the correct appliance.
5. Confirm the firewall or NVA allows the traffic.
6. Check for overlapping prefixes that create a more specific match than you expected.

**Common pitfalls**

- **Overlapping prefixes**: A more specific route wins, even if it is not the one you intended.
- **Default route conflicts**: A `0.0.0.0/0` UDR can unintentionally capture internet and hybrid traffic.
- **Disabled propagation**: Turning off BGP propagation can break hybrid connectivity if you still need learned routes.
- **Wrong subnet association**: Associating the route table to the wrong subnet leaves the workload unaffected.
- **Asymmetric routing**: Forward and return paths differ, which can break stateful inspection.
- **Service restrictions**: Some Azure services do not tolerate arbitrary routing changes.

**Exam-oriented distinctions**

- **System routes** are Azure-managed defaults.
- **BGP routes** come from gateways and hybrid connectivity.
- **UDRs** are your explicit overrides.
- **Peering routes** enable VNet-to-VNet reachability and appear in effective routes.
- **Subnet association** is the control point for route tables.
- **NIC effective routes** are the best place to validate the final path.

**Practical design checklist**

- Define the traffic you want to inspect or block.
- Identify the subnet that should inherit the route table.
- Add only the routes you need.
- Use the most specific prefix that matches your design.
- Validate the effective route table on a test NIC.
- Confirm firewall, gateway, and return-path behavior.
- Recheck after peering, gateway, or address-space changes.

**Quick reference**

- Use `VirtualAppliance` for firewalls and NVAs.
- Use `VirtualNetworkGateway` for hybrid gateway paths.
- Use `None` to drop traffic intentionally.
- Use `Internet` only when you want to override a more specific path.
- Use `VnetLocal` when you need an explicit local-VNet route, though Azure already provides local connectivity by default.
