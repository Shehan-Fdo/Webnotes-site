## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Traffic inspection and segmentation in Azure means you validate that packets take the intended path and that only approved flows are allowed. In practice, you combine network controls, routing, DNS, and observability to prove that a workload is isolated correctly and that inspection points actually see the traffic you expect.

For AZ-500, focus on two questions:

- Can the traffic reach the destination?
- If it can, is it inspected, logged, and restricted by the right control?

A good validation workflow checks effective rules and routes first, then tests connectivity, then confirms logs and DNS behavior. That order helps you separate policy problems from path problems.

**Core segmentation controls**

Azure segmentation is usually built from several layers that work together rather than one single feature.

- **Network Security Groups (NSGs)** filter traffic at the subnet or NIC level.
- **User-defined routes (UDRs)** steer traffic through appliances such as Azure Firewall or third-party NVA (network virtual appliance).
- **Azure Firewall** centralizes filtering and logging when traffic is routed through it.
- **Application Gateway with WAF** inspects inbound HTTP/HTTPS traffic at Layer 7.
- **DDoS Protection** helps protect public endpoints from volumetric attacks.
- **Service Endpoints** and **Private Endpoints** change how Azure PaaS services are reached.
- **Azure Bastion** and jump hosts reduce exposure of management ports.

These controls are complementary. For example, an NSG may allow a subnet-to-subnet flow, but a UDR can still force that flow through a firewall, and the firewall can still deny it.

**Network Security Groups**

NSGs are stateful packet filters. They evaluate rules by priority, where a lower number means higher precedence. Azure evaluates custom rules first, then default rules if no custom rule matches.

Important rule behavior:

- **Stateful** means return traffic for an allowed flow is automatically allowed.
- **Priority** determines which matching rule wins.
- **Default rules** still matter when no custom rule applies.
- **Service tags** simplify rules by representing Azure service IP ranges.
- **Application Security Groups (ASGs)** let you group VMs logically and write rules against the group instead of individual IPs.

Example NSG rule set for a web tier:

```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-web \
  --name Allow-HTTP-From-AppGateway \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes AppGatewaySubnet \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 80
```

Use service tags when the source or destination is an Azure platform range that changes over time.

```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-web \
  --name Allow-AzureLoadBalancer \
  --priority 110 \
  --direction Inbound \
  --access Allow \
  --protocol '*' \
  --source-address-prefixes AzureLoadBalancer \
  --destination-address-prefixes '*' \
  --destination-port-ranges '*'
```

Use ASGs when you want segmentation by workload role rather than by subnet size or static IPs.

```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-app \
  --name Allow-Web-To-App \
  --priority 200 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-asgs asg-web \
  --destination-asgs asg-app \
  --destination-port-ranges 443
```

**How to validate NSGs**

Validation usually starts with the effective policy.

- Check the effective NSG rules on the NIC or subnet.
- Confirm that no higher-priority deny blocks the flow.
- Confirm that the source and destination are in the expected ASG or service tag.
- Confirm that the port and protocol match the application.

Useful CLI patterns:

```bash
az network nic list-effective-nsg --resource-group rg-app --name vm1Nic
```

```bash
az network vnet subnet show \
  --resource-group rg-app \
  --vnet-name vnet-spoke \
  --name app-subnet \
  --query networkSecurityGroup
```

If a flow is denied, look for the first matching deny rule and compare it with the intended design. In exam scenarios, remember that a deny can come from a custom rule or from the default deny rule if no allow rule matches.

**Route validation and effective paths**

Routing determines where packets go, which is critical when you inspect traffic through a firewall or NVA. Azure route selection is based on precedence:

1. **User-defined routes** usually override system routes.
2. **BGP routes** from ExpressRoute or VPN can influence path selection.
3. **System routes** apply when no more specific route exists.

More specific prefixes win over less specific prefixes. For example, a `/24` route beats a `/16` route for matching traffic.

Common route sources:

- **System routes** for local VNet, peering, internet, and platform behavior.
- **UDRs** for forced tunneling or service insertion.
- **BGP** for on-premises learned routes in hybrid designs.

Validate the effective route table on a NIC when traffic does not follow the expected inspection path.

```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name vm1Nic
```

A typical hub-and-spoke design uses a UDR in the spoke subnet to send `0.0.0.0/0` or specific prefixes to Azure Firewall in the hub. If the route is missing or overridden, traffic may bypass inspection entirely.

Example UDR association:

```bash
az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name vnet-spoke \
  --name app-subnet \
  --route-table /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-network/providers/Microsoft.Network/routeTables/rt-spoke-egress
```

> If you validate a path only by checking DNS, you can miss routing failures. A name can resolve correctly while the packet still takes the wrong path or is blocked by NSG, UDR, or firewall policy.

**Azure Firewall and inspection points**

Azure Firewall is a centralized stateful firewall service that can inspect traffic only when you route traffic through it. It does not automatically inspect all east-west traffic inside a VNet.

Use Azure Firewall when you need:

- Centralized egress control
- Network and application rule filtering
- FQDN (fully qualified domain name) filtering
- Logging and policy consistency across spokes
- Forced tunneling or controlled internet access

Validation steps:

- Confirm the subnet UDR sends traffic to the firewall.
- Confirm the firewall policy allows the flow.
- Confirm the firewall logs show the connection attempt.
- Confirm return traffic follows the expected path.

Example route to force egress through the firewall:

```bash
az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke-egress \
  --name DefaultToFirewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

Azure Firewall is often paired with:

- **Application Gateway WAF** for inbound web inspection
- **DDoS Protection** for public IP resilience
- **Private Endpoints** for private access to PaaS services

**Application Gateway and WAF**

Application Gateway is a Layer 7 load balancer. When you enable WAF (Web Application Firewall), it can inspect inbound HTTP/HTTPS traffic for common web attacks and policy violations.

Use it when you need:

- TLS termination or end-to-end TLS
- Host-based routing
- Path-based routing
- Web application protection
- Centralized inbound inspection before traffic reaches app subnets

Validation points:

- The listener receives traffic on the expected frontend IP.
- The WAF policy is attached and in the correct mode.
- Backend health is healthy.
- Logs show the request and any WAF action.

A common secure pattern is:

- Internet client
- Public IP on Application Gateway
- WAF policy
- Backend pool in a private subnet
- NSG allows only gateway-to-backend traffic

**DDoS Protection**

DDoS Protection helps defend public endpoints from large-scale attacks. It is not a segmentation tool by itself, but it is relevant when you validate traffic inspection for internet-facing workloads.

Use it when:

- You expose public IPs for critical services
- You need attack mitigation and telemetry
- You want protection for virtual networks with public endpoints

Validation points:

- The protected VNet is associated with the DDoS plan.
- Public endpoints are in scope.
- Alerts and metrics are enabled for visibility.

**Service Endpoints**

Service Endpoints extend VNet identity to supported Azure services while the service still uses its public endpoint. The client keeps its own private IP; the service does not get a private IP in your VNet.

Use service endpoints when you want:

- Simpler private access from a subnet to a PaaS service
- Network-based access control without deploying a private endpoint
- Reduced exposure to the public internet path for supported services

Important nuances:

- The service still exists on a public endpoint.
- Access is restricted by VNet/subnet identity.
- Service endpoint policies can restrict which service resources a subnet can reach.
- You should still validate DNS, NSGs, and service firewall settings.

Example validation idea:

- Confirm the subnet has the service endpoint enabled.
- Confirm the storage account or SQL server firewall allows that subnet.
- Test access from a VM in the subnet.

```bash
az network vnet subnet show \
  --resource-group rg-app \
  --vnet-name vnet-app \
  --name data-subnet \
  --query serviceEndpoints
```

**Private Endpoints and private DNS**

Private Endpoints map a PaaS service to a private IP in your VNet. This is usually the strongest private access pattern for Azure PaaS.

Use Private Endpoints when you need:

- Private IP access to a PaaS service
- Stronger network isolation
- Reduced dependence on public network access
- Fine-grained control over who can reach the service

Validation requires more than checking DNS alone.

- DNS should resolve the service name to the private endpoint IP.
- The client must be in a network path that can reach that private IP.
- NSGs and UDRs must allow the traffic.
- The private endpoint connection must be approved on the service side.
- Private DNS zone links must be correct for the VNet.

Common troubleshooting steps:

1. Run `nslookup` or `Resolve-DnsName` from the client.
2. Confirm the name resolves to the private endpoint address.
3. Confirm the private DNS zone is linked to the VNet.
4. Confirm the private endpoint connection state is approved.
5. Confirm NSGs and routes allow the traffic.

Example DNS check:

```bash
nslookup mystorageaccount.blob.core.windows.net
```

Example Azure CLI inspection:

```bash
az network private-endpoint show \
  --resource-group rg-app \
  --name pe-storage \
  --query "{state:privateLinkServiceConnections[0].privateLinkServiceConnectionState.status, subnet:subnet.id}"
```

> DNS resolving to a private IP is necessary, but it does not prove the client can successfully use the private path. Routing, NSGs, and client network location still determine whether the connection works.

**VNet peering and segmentation**

VNet peering connects VNets privately, but it does not remove the need for explicit segmentation controls.

Key behaviors:

- Peered VNets can communicate privately when peering allows it.
- NSGs still apply to traffic entering or leaving subnets.
- UDRs can still steer traffic through inspection points.
- Peering is not transitive by default.
- Traffic does not automatically pass through a hub unless you route it there.

In hub-and-spoke designs, validate:

- Peering settings allow the intended traffic direction.
- Spoke subnets have UDRs to the hub firewall.
- The hub firewall has rules for spoke-to-spoke or spoke-to-internet flows.
- No asymmetric routing breaks return traffic.

**Azure Bastion and jump hosts**

Azure Bastion and jump hosts help you manage segmentation by removing direct exposure of management ports such as RDP (Remote Desktop Protocol) and SSH (Secure Shell).

Use them when:

- You want to avoid public IPs on VMs
- You need controlled administrative access
- You want to reduce attack surface on management ports

Validation points:

- NSGs deny direct inbound RDP/SSH from the internet.
- Bastion subnet is correctly deployed and named `AzureBastionSubnet`.
- Admin access is only available through Bastion or a hardened jump host.
- Jump host access is itself restricted by NSG and identity controls.

**Network Watcher tools**

Network Watcher provides practical validation tools for path and rule troubleshooting. These tools are especially useful in exam scenarios because they show what Azure thinks will happen to a packet.

Core tools:

- **IP flow verify**: checks whether a specific flow is allowed or denied by NSGs.
- **Next hop**: shows the next hop for a destination from a VM.
- **Connection troubleshoot**: tests end-to-end connectivity and identifies where the path fails.
- **Packet capture**: captures packets for deeper analysis.
- **NSG flow logs**: records flow metadata for NSG-related traffic analysis in environments where the feature is enabled.

Example IP flow verify:

```bash
az network watcher test-ip-flow \
  --resource-group rg-app \
  --vm vm1 \
  --direction Outbound \
  --protocol TCP \
  --local 10.1.1.4:51514 \
  --remote 10.2.1.10:443
```

Example next hop:

```bash
az network watcher show-next-hop \
  --resource-group rg-app \
  --vm vm1 \
  --source-ip 10.1.1.4 \
  --dest-ip 10.2.1.10
```

Example connection troubleshoot:

```bash
az network watcher test-connectivity \
  --source-resource /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm1 \
  --dest-address 10.2.1.10 \
  --dest-port 443
```

Interpretation tips:

- If IP flow verify says deny, focus on NSGs first.
- If next hop shows Internet when you expected a firewall, check UDRs and route precedence.
- If connection troubleshoot fails after the next hop is correct, inspect the firewall, backend service, or DNS.

**NSG flow logs and Traffic Analytics**

NSG flow logs capture flow metadata for traffic that matches NSG processing. In many environments, Azure now favors broader network observability patterns, but NSG flow logs remain relevant for exam study and legacy deployments.

Use them to:

- See allowed and denied flows at the NSG level
- Correlate traffic patterns over time
- Support troubleshooting and forensics

Typical setup flow:

1. Enable Network Watcher in the region.
2. Enable NSG flow logs for the target NSG.
3. Send logs to a storage account.
4. Optionally enable Traffic Analytics for aggregation and visualization.

What to look for:

- Source and destination IPs
- Source and destination ports
- Protocol
- Direction
- Allow or deny decision
- Time window and flow state

Be careful not to over-interpret flow logs. They show network-level metadata, not application success. A flow can be allowed while the application still fails due to TLS, authentication, or backend health issues.

**Practical validation workflow**

A reliable workflow for segmentation validation is:

1. Identify the intended path and inspection points.
2. Check effective NSGs on the source and destination.
3. Check effective routes on the source NIC or subnet.
4. Verify DNS resolution for private or service endpoint scenarios.
5. Test connectivity with Network Watcher or a controlled client.
6. Confirm logs on the firewall, WAF, or NSG flow logs.
7. Fix the first control that blocks the intended flow.

This workflow helps you avoid guessing. It also matches how Azure enforces traffic: policy, route, DNS, and service-side access all matter.

**Common exam scenarios**

- A VM can reach the internet even though a firewall exists.
  - Check whether a UDR actually sends traffic to the firewall.
  - Check whether a more specific route bypasses the default route.

- A subnet can reach a storage account from one VNet but not another.
  - Check service endpoint or private endpoint configuration.
  - Check private DNS zone links and storage firewall rules.
  - Check peering and route settings.

- RDP or SSH works from the internet when it should not.
  - Check NSG inbound rules and public IP exposure.
  - Use Bastion or a jump host instead.

- Web traffic reaches the app but WAF logs show blocks.
  - Check WAF policy mode, exclusions, and rule matches.
  - Confirm backend health and TLS settings.

- Traffic between spokes bypasses the hub firewall.
  - Check peering, UDRs, and route specificity.
  - Remember that peering is not transitive by default.

**Quick comparison**

| Control | Main purpose | Validation focus |
|---|---|---|
| NSG | Subnet/NIC filtering | Effective rules, priorities, service tags, ASGs |
| UDR | Force path selection | Effective routes, next hop, route precedence |
| Azure Firewall | Central inspection | Policy, logs, forced routing |
| Application Gateway WAF | Inbound web inspection | Listener, backend health, WAF logs |
| Service Endpoint | VNet identity to PaaS | Subnet enablement, service firewall, policies |
| Private Endpoint | Private IP to PaaS | DNS, approval state, private DNS zone links |
| Bastion | Secure admin access | No public management ports, Bastion subnet health |
| DDoS Protection | Public endpoint resilience | Plan association, alerts, metrics |

**Key takeaways**

- Validate traffic by checking policy, route, and DNS in that order.
- Use NSGs for subnet and NIC segmentation, but remember default rules still apply.
- Use UDRs to force traffic through inspection points such as Azure Firewall.
- Use Application Gateway WAF for inbound web inspection and Azure Firewall for centralized network control.
- Treat service endpoints and private endpoints as different access models with different validation steps.
- Use Network Watcher tools to prove where traffic is allowed, denied, or rerouted.
- Do not assume DNS success means network success.
