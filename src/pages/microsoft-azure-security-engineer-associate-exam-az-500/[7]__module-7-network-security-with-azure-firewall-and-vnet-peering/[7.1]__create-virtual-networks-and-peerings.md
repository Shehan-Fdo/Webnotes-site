## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Virtual networks and subnets**

An Azure <abbr title="Virtual Network">VNet</abbr> is your private network boundary in Azure. You use it to isolate workloads, control IP addressing, and define how traffic moves between resources. Subnets divide a VNet into smaller segments so you can apply security, routing, and service-specific controls at a finer scope.

A VNet is not just an address container. It is the foundation for:
- Private IP assignment for Azure resources
- Network segmentation by subnet
- Routing control with user-defined routes
- Security enforcement with <abbr title="Network Security Group">NSG</abbr>s
- Connectivity patterns such as peering, VPN, and ExpressRoute

**Address spaces and subnet design**

You define one or more address spaces on the VNet, then carve those ranges into subnets. Plan carefully to avoid overlap with on-premises networks, other VNets, and future expansion.

Common design guidance:
- Use non-overlapping RFC 1918 ranges such as `10.0.0.0/16`, `10.1.0.0/16`, or `172.16.0.0/16`
- Reserve space for growth, shared services, and future spokes
- Keep subnet sizes aligned to workload density and Azure service requirements
- Avoid overly small subnets if you may need private endpoints, appliances, or scale-out workloads later

A subnet is where you usually attach:
- An NSG for traffic filtering
- A route table for custom routing
- Delegation for managed services
- Service endpoints or private endpoints support, depending on the design

**Creating VNets and subnets in the portal**

In the Azure portal, you typically create the VNet first, then add subnets during creation or afterward.

Typical portal flow:
1. Open **Virtual networks** and select **Create**.
2. Choose subscription, resource group, region, and name.
3. Define the VNet address space.
4. Add one or more subnets with their CIDR ranges.
5. Optionally attach NSGs, route tables, or service settings.
6. Review and create.

When you add a subnet later, make sure the new CIDR:
- Falls within the VNet address space
- Does not overlap an existing subnet
- Leaves room for future subnet growth

**Azure CLI examples**

Use the Azure CLI when you want repeatable deployment or automation.

```bash
# Create a resource group
az group create \
  --name rg-networking \
  --location eastus

# Create a VNet with one subnet
az network vnet create \
  --resource-group rg-networking \
  --name vnet-hub \
  --address-prefixes 10.10.0.0/16 \
  --subnet-name snet-shared \
  --subnet-prefixes 10.10.1.0/24

# Add another subnet
az network vnet subnet create \
  --resource-group rg-networking \
  --vnet-name vnet-hub \
  --name snet-firewall \
  --address-prefixes 10.10.2.0/24
```

Useful management commands:
```bash
# Show VNet details
az network vnet show -g rg-networking -n vnet-hub

# List subnets
az network vnet subnet list -g rg-networking --vnet-name vnet-hub

# Update VNet address prefixes
az network vnet update -g rg-networking -n vnet-hub --address-prefixes 10.10.0.0/16 10.11.0.0/16
```

**PowerShell examples**

PowerShell is common in enterprise automation and exam scenarios.

```powershell
# Create a resource group
New-AzResourceGroup -Name rg-networking -Location eastus

# Create a VNet and subnet
$subnet = New-AzVirtualNetworkSubnetConfig -Name snet-shared -AddressPrefix 10.10.1.0/24
$vnet = New-AzVirtualNetwork -ResourceGroupName rg-networking -Location eastus `
  -Name vnet-hub -AddressPrefix 10.10.0.0/16 -Subnet $subnet

# Add a subnet
Add-AzVirtualNetworkSubnetConfig -Name snet-firewall -AddressPrefix 10.10.2.0/24 -VirtualNetwork $vnet
$vnet | Set-AzVirtualNetwork
```

**Subnet-level configuration**

Subnet configuration matters because peering does not replace subnet design. You still control traffic and service placement at the subnet layer.

Common subnet-level settings:
- **NSG association** to allow or deny traffic
- **Route table association** to force traffic through a firewall or NVA
- **Delegation** for services such as Azure App Service Environment or managed databases
- **Private endpoint policies** when using private endpoints
- **Service endpoints** for selected PaaS services

A practical pattern is to separate workloads by function:
- `snet-web` for application front ends
- `snet-app` for business logic
- `snet-data` for databases or private endpoints
- `snet-mgmt` for jump hosts or management tools

This separation makes it easier to apply least privilege, inspect traffic, and troubleshoot routing.

**Network Security Groups and peered traffic**

NSGs filter traffic at the subnet or NIC level. In peered VNets, traffic still passes through NSG evaluation on both sides. Peering does not bypass security rules.

Important NSG behaviors:
- Inbound rules on the destination subnet or NIC must allow the traffic
- Outbound rules on the source subnet or NIC must allow the traffic
- Default rules allow VNet-to-VNet traffic within the same VNet, but peered traffic may still be affected by custom rules
- If you block a port on either side, the connection fails even if peering is healthy

Common rule considerations for peered designs:
- Allow application ports only between required subnets
- Deny broad `Any` to `Any` rules that unintentionally block east-west traffic
- Permit management traffic from a jump subnet to workload subnets
- Allow DNS traffic to the resolver or DNS servers if you use custom DNS
- Consider stateful behavior: return traffic is allowed for established flows, but initial outbound or inbound rules still matter

Example NSG rule idea for a spoke application subnet:
- Allow inbound TCP `443` from hub firewall subnet
- Allow inbound TCP `3389` or `22` only from management subnet
- Deny inbound from `VirtualNetwork` if you want strict segmentation, then add explicit allows

> If you use a hub-spoke model, do not assume peering alone provides segmentation. NSGs and route tables still define what can talk to what.

**VNet peering fundamentals**

VNet peering connects two VNets privately over the Microsoft backbone. Traffic stays on the Azure network and does not traverse the public internet.

Key properties:
- Peering is **bidirectional in effect**, but you configure it on **both** VNets
- Each side has its own peering object
- Settings are not always symmetric
- Address spaces must not overlap
- Peered VNets can be in the same or different regions
- Peering is low latency and high bandwidth compared with VPN-based connectivity

Because each side is configured separately, you can create asymmetric behavior:
- One side can allow gateway transit
- The other side can use remote gateways
- One side can allow forwarded traffic
- The other side can deny it
- One side can use custom DNS settings while the other uses different resolvers

This asymmetry is useful, but it also causes troubleshooting mistakes when only one side is updated.

**Peering settings you must know**

Common peering options include:
- **Allow virtual network access**: permits traffic between peered VNets
- **Allow forwarded traffic**: allows traffic that has been forwarded by an NVA or firewall
- **Allow gateway transit**: lets one VNet share its VPN or ExpressRoute gateway with the peered VNet
- **Use remote gateways**: lets the peered VNet use the other VNet’s gateway

Gateway transit is asymmetric:
- The VNet with the gateway enables **Allow gateway transit**
- The other VNet enables **Use remote gateways**
- You cannot have both sides using remote gateways in the same peering relationship
- This is common in hub-spoke designs where the hub owns connectivity to on-premises networks

**Creating peering in the portal**

Portal steps are straightforward, but both directions still matter.

1. Open the source VNet.
2. Select **Peerings** and choose **Add**.
3. Enter the remote VNet name and subscription.
4. Configure peering options.
5. Save.
6. Repeat from the remote VNet if needed, or verify that Azure created the reciprocal peering depending on the workflow used.

Be careful with cross-subscription or cross-tenant scenarios:
- You need appropriate permissions on both VNets
- The remote VNet must be visible to your account or service principal
- Cross-tenant peering requires consent and correct role assignments in both tenants
- RBAC permissions commonly include network contributor-level rights or equivalent custom permissions

**Azure CLI peering example**

```bash
# Create peering from hub to spoke
az network vnet peering create \
  --resource-group rg-networking \
  --vnet-name vnet-hub \
  --name hub-to-spoke1 \
  --remote-vnet /subscriptions/<subId>/resourceGroups/rg-networking/providers/Microsoft.Network/virtualNetworks/vnet-spoke1 \
  --allow-vnet-access \
  --allow-forwarded-traffic \
  --allow-gateway-transit

# Create reverse peering from spoke to hub
az network vnet peering create \
  --resource-group rg-networking \
  --vnet-name vnet-spoke1 \
  --name spoke1-to-hub \
  --remote-vnet /subscriptions/<subId>/resourceGroups/rg-networking/providers/Microsoft.Network/virtualNetworks/vnet-hub \
  --allow-vnet-access \
  --allow-forwarded-traffic \
  --use-remote-gateways
```

**PowerShell peering example**

```powershell
$hub = Get-AzVirtualNetwork -Name vnet-hub -ResourceGroupName rg-networking
$spoke = Get-AzVirtualNetwork -Name vnet-spoke1 -ResourceGroupName rg-networking

Add-AzVirtualNetworkPeering `
  -Name hub-to-spoke1 `
  -VirtualNetwork $hub `
  -RemoteVirtualNetworkId $spoke.Id `
  -AllowVirtualNetworkAccess `
  -AllowForwardedTraffic `
  -AllowGatewayTransit

Add-AzVirtualNetworkPeering `
  -Name spoke1-to-hub `
  -VirtualNetwork $spoke `
  -RemoteVirtualNetworkId $hub.Id `
  -AllowVirtualNetworkAccess `
  -AllowForwardedTraffic `
  -UseRemoteGateways
```

**Hub-spoke routing design**

Hub-spoke is the most common peering pattern in security-focused Azure designs. The hub hosts shared services such as:
- Azure Firewall
- VPN or ExpressRoute gateway
- DNS services
- Bastion or jump access
- Central monitoring and inspection

Spokes host application workloads. Traffic between spokes usually does not flow directly unless you design for it.

Typical flow:
- Spoke A sends traffic to the hub using a user-defined route
- The hub firewall inspects and forwards the traffic
- The firewall routes to Spoke B if allowed
- Return traffic follows the configured path if routing and NSGs permit it

To enable spoke-to-spoke traffic through the hub:
- Peer each spoke to the hub
- Add UDRs on spoke subnets that point to the firewall or NVA as next hop
- Enable forwarded traffic on the peerings
- Allow the required ports in NSGs and firewall rules
- Ensure the firewall has routes back to the spoke prefixes

Example route table for a spoke subnet:
```bash
az network route-table create \
  -g rg-networking \
  -n rt-spoke1

az network route-table route create \
  -g rg-networking \
  --route-table-name rt-spoke1 \
  -n to-spoke2-via-firewall \
  --address-prefix 10.20.0.0/16 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.10.2.4
```

This pattern is useful when you want:
- Central inspection
- Segmentation between workloads
- Logging and threat detection
- Controlled east-west traffic

**Private DNS and Azure DNS Private Resolver**

DNS is often the hidden dependency in peered networks. Peering connects IP networks, but it does not automatically solve name resolution.

Important DNS concepts:
- Private DNS zones provide private name resolution for Azure resources
- A private DNS zone must be linked to each VNet that needs to resolve records from that zone
- Peered VNets do not automatically inherit private DNS zone links
- Azure DNS Private Resolver helps forward DNS queries between on-premises, hub, and spoke networks without custom VM-based DNS servers

For peered VNets:
- Link the private DNS zone to the hub and any spoke VNets that need resolution
- If you use Azure DNS Private Resolver, place inbound and outbound endpoints in a hub VNet
- Configure DNS forwarding rules so spokes and on-premises clients can resolve private names consistently
- Verify that the VNet’s custom DNS settings point to the resolver or appropriate DNS server

Example private DNS zone link:
```bash
az network private-dns zone create \
  -g rg-networking \
  -n privatelink.database.windows.net

az network private-dns link vnet create \
  -g rg-networking \
  -z privatelink.database.windows.net \
  -n link-hub \
  -v /subscriptions/<subId>/resourceGroups/rg-networking/providers/Microsoft.Network/virtualNetworks/vnet-hub \
  -e false
```

If a spoke needs the same records, create another link for that spoke VNet. Do not rely on peering alone.

**Cross-subscription and cross-tenant considerations**

Peering across subscriptions is common in enterprise environments. Peering across tenants is also possible, but it requires more coordination.

You should verify:
- You have permission to read and create peering on both VNets
- The remote VNet ID is correct
- The remote tenant allows the connection
- RBAC assignments are in place for the identity performing the operation
- Policies or locks do not block network changes

Typical roles and access patterns:
- Network Contributor on both VNets or resource groups
- Owner or Contributor with sufficient network permissions
- Guest access or cross-tenant authorization where required

In exam scenarios, remember that a peering failure may be caused by authorization, not just networking.

**Operational behavior and troubleshooting**

Peering changes are usually quick, but routing and DNS behavior may take time to settle depending on the change and the client.

Operational points:
- Deleting and recreating peering removes the relationship and can interrupt traffic
- Route propagation may take a short time after changes
- Effective routes on NICs may not update instantly in every troubleshooting view
- DNS changes can lag if clients cache old records
- NSG or UDR changes can make peering appear broken when the peering itself is healthy

Troubleshooting checklist:
1. Confirm both peering objects exist and are in the `Connected` state.
2. Verify address spaces do not overlap.
3. Check NSG rules on source and destination subnets or NICs.
4. Review route tables and next hop selection.
5. Confirm gateway transit settings are correct on both sides.
6. Validate DNS resolution if the issue is name-based rather than IP-based.
7. Test with `ping`, `Test-NetConnection`, `curl`, or `nc` depending on protocol support.

Useful commands:
```bash
# Show peering state
az network vnet peering list -g rg-networking --vnet-name vnet-hub -o table

# Check effective routes and NSGs from a VM NIC perspective
az network nic show-effective-route-table -g rg-networking -n nic-spoke1vm
az network nic list-effective-nsg -g rg-networking -n nic-spoke1vm
```

PowerShell route and NSG checks:
```powershell
Get-AzEffectiveRouteTable -NetworkInterfaceName nic-spoke1vm -ResourceGroupName rg-networking
Get-AzEffectiveNetworkSecurityGroup -NetworkInterfaceName nic-spoke1vm -ResourceGroupName rg-networking
```

**Common exam traps**

Watch for these AZ-500-style pitfalls:
- Assuming peering automatically shares DNS zones
- Forgetting that peering must be configured on both sides
- Mixing up `Allow gateway transit` and `Use remote gateways`
- Expecting peered traffic to bypass NSGs
- Overlooking route tables when traffic must traverse a firewall
- Ignoring cross-subscription or cross-tenant permissions
- Using overlapping address spaces, which prevents peering
- Forgetting that spoke-to-spoke traffic usually needs a hub firewall or NVA for inspection

**Quick comparison**

| Topic | What it controls | Why it matters |
|---|---|---|
| VNet address space | Private IP range for the network | Prevents overlap and supports growth |
| Subnet | Smaller segment inside a VNet | Enables segmentation and subnet-level policy |
| NSG | Packet filtering | Controls allowed traffic at subnet or NIC scope |
| Route table | Next hop selection | Forces traffic through firewall or appliance |
| Peering | Private VNet-to-VNet connectivity | Enables low-latency east-west communication |
| Private DNS zone link | Name resolution scope | Makes private names resolve in each VNet |
| Azure DNS Private Resolver | DNS forwarding and resolution | Simplifies hybrid and multi-VNet DNS |

**Practical design summary**

Use VNets to define network boundaries, subnets to segment workloads, NSGs to enforce access, and peering to connect networks privately. In secure Azure designs, peering is only one part of the solution: routing, DNS, and permissions determine whether traffic actually works.

For exam purposes, focus on how these pieces interact:
- Subnets host workloads and policy
- NSGs filter traffic in both directions
- Peering is private and bidirectional in configuration
- Gateway transit is asymmetric
- DNS must be linked or forwarded explicitly
- Hub-spoke routing often requires UDRs and Azure Firewall
- Cross-subscription and cross-tenant peering requires correct authorization
