## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Azure Firewall in a hub network**

Azure Firewall is a managed, stateful network security service that centralizes traffic inspection in a hub-and-spoke design. You place the firewall in a shared hub VNet, then steer spoke traffic through it with user-defined routes (UDRs) and peering settings. This pattern gives you one place to enforce outbound filtering, inbound publishing, and DNS control.

A typical flow looks like this:

```text
Spoke VM -> Spoke subnet UDR -> Hub firewall private IP -> Allowed destination
```

For inbound publishing, the flow is the reverse:

```text
Internet -> Firewall public IP -> DNAT rule -> Private workload in spoke
```

**Why this design matters**

- **Centralized control** reduces rule sprawl across many VNets.
- **Inspection** happens before traffic reaches workloads.
- **Segmentation** is easier because spokes stay isolated from each other unless you explicitly allow traffic.
- **Logging** is consolidated for audit and incident response.

**Prerequisites and design constraints**

Before you deploy, verify the platform requirements and design limits.

- **Dedicated subnet**: Azure Firewall must be deployed into `AzureFirewallSubnet`.
- **Subnet size**: Use a subnet large enough for scale and future growth; `/26` or larger is a common planning choice.
- **Public IP**: Classic hub deployments typically use a Standard public IP for internet-facing SNAT and DNAT.
- **Region support**: Confirm the target region supports the firewall SKU and features you need.
- **Routing design**: Plan UDRs, peering, and gateway integration before deployment.
- **Forced tunneling**: Classic forced tunneling requires `AzureFirewallManagementSubnet` in addition to `AzureFirewallSubnet`.

> Azure Firewall is not a general-purpose VM appliance. You do not install it on a VM or manage the OS. You deploy the managed service into the required subnet.

**Hub-and-spoke topology and traffic path**

The hub contains shared network services:

- Azure Firewall
- VPN gateway or ExpressRoute gateway
- DNS forwarders or Azure DNS Private Resolver, if used
- Shared management or inspection services

Spokes contain application workloads. You usually configure:

- VNet peering between hub and each spoke
- UDRs on spoke subnets to send traffic to the firewall
- Optional gateway transit if spokes use the hub gateway

**Peering settings you must understand**

For hub-spoke firewall scenarios, peering settings are often the exam trap.

- On the **spoke-to-hub** peering:
  - Enable **Allow forwarded traffic** so traffic forwarded by the firewall can traverse peering.
  - Enable **Use remote gateways** if the spoke should use the hub VPN/ExpressRoute gateway.
- On the **hub-to-spoke** peering:
  - Enable **Allow forwarded traffic**.
  - Enable **Allow gateway transit** if the hub gateway should be shared with spokes.

A common pattern is:

| Peering direction | Key settings |
|---|---|
| Hub -> Spoke | Allow forwarded traffic, Allow gateway transit |
| Spoke -> Hub | Allow forwarded traffic, Use remote gateways |

**Routing and route propagation**

Azure Firewall only works as the inspection point if traffic actually reaches it. That means you must control routing carefully.

- Use **UDRs** on spoke subnets to send `0.0.0.0/0` or specific prefixes to the firewall private IP.
- Use **next hop type** `Virtual appliance` for the firewall private IP.
- Consider **disabling BGP route propagation** on subnets when learned routes from VPN/ExpressRoute would bypass the firewall.
- Keep in mind that disabling propagation is a subnet-level choice; it prevents automatically learned routes from taking precedence where you do not want them.

A practical route table example:

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke1 \
  --disable-bgp-route-propagation true

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke1 \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

Associate the route table to the spoke subnet by using the route table resource ID:

```bash
az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name vnet-spoke1 \
  --name app-subnet \
  --route-table "/subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/routeTables/rt-spoke1"
```

**Deployment procedure**

You can deploy Azure Firewall through the portal, CLI, ARM, or Bicep. The steps below show the practical sequence you should know for exam and implementation work.

1. Create or identify the hub VNet.
2. Create the `AzureFirewallSubnet` with the required address space.
3. Create a Standard public IP if you need internet-facing SNAT or DNAT.
4. Deploy Azure Firewall into the hub subnet.
5. Configure firewall policy or classic rules.
6. Peer the hub with each spoke.
7. Add UDRs to spoke subnets.
8. Validate traffic flow and logs.

**Azure CLI example: hub VNet and firewall subnet**

```bash
az network vnet create \
  --resource-group rg-network \
  --name vnet-hub \
  --address-prefixes 10.0.0.0/16 \
  --subnet-name AzureFirewallSubnet \
  --subnet-prefixes 10.0.0.0/26
```

**Create a public IP and deploy the firewall**

```bash
az network public-ip create \
  --resource-group rg-network \
  --name pip-azfw \
  --sku Standard \
  --allocation-method Static

az network firewall create \
  --resource-group rg-network \
  --name azfw-hub \
  --location eastus \
  --sku AZFW_VNet
```

Associate the public IP to the firewall:

```bash
az network firewall ip-config create \
  --resource-group rg-network \
  --firewall-name azfw-hub \
  --name fw-ipconfig \
  --public-ip-address pip-azfw \
  --vnet-name vnet-hub
```

> CLI syntax can vary by Azure CLI version and firewall SKU. Validate the command help in your environment with `az network firewall -h` and `az network firewall ip-config -h`.

**Bicep example**

```bicep
param location string = resourceGroup().location
param firewallName string = 'azfw-hub'
param pipName string = 'pip-azfw'
param vnetName string = 'vnet-hub'

resource pip 'Microsoft.Network/publicIPAddresses@2023-09-01' = {
  name: pipName
  location: location
  sku: {
    name: 'Standard'
  }
  properties: {
    publicIPAllocationMethod: 'Static'
  }
}

resource vnet 'Microsoft.Network/virtualNetworks@2023-09-01' existing = {
  name: vnetName
}

resource firewall 'Microsoft.Network/azureFirewalls@2023-09-01' = {
  name: firewallName
  location: location
  properties: {
    ipConfigurations: [
      {
        name: 'fw-ipconfig'
        properties: {
          subnet: {
            id: '${vnet.id}/subnets/AzureFirewallSubnet'
          }
          publicIPAddress: {
            id: pip.id
          }
        }
      }
    ]
  }
}
```

**Firewall policy and rule processing**

Azure Firewall policy separates configuration from the firewall resource. This is the preferred model for modern deployments.

Rule processing order is important:

1. **DNAT rules** are evaluated first for inbound traffic.
2. **Network rules** are evaluated next for L3/L4 filtering.
3. **Application rules** are evaluated for HTTP/HTTPS and FQDN-based control.

This order matters because a more specific rule type can allow traffic before a broader deny would otherwise apply.

**Network rules**

Use network rules for IP-based allow/deny decisions on TCP, UDP, and ICMP.

```bash
az network firewall policy rule-collection-group collection add-filter-collection \
  --policy-name fw-policy \
  --resource-group rg-network \
  --rule-collection-group-name rcg1 \
  --name net-allow \
  --priority 100 \
  --action Allow
```

For practical exam prep, remember the conceptual shape of a network rule:

- Source addresses or source IP groups
- Destination addresses or destination IP groups
- Destination ports
- Protocols such as TCP or UDP

**Application rules**

Use application rules for FQDN-based control over HTTP, HTTPS, and MSSQL in supported scenarios. These rules are useful when you want to allow specific domains rather than raw IPs.

A conceptual policy snippet:

```json
{
  "ruleType": "ApplicationRule",
  "name": "allow-microsoft-update",
  "sourceAddresses": ["10.1.1.0/24"],
  "protocols": [
    { "protocolType": "Https", "port": 443 }
  ],
  "targetFqdns": [
    "*.windowsupdate.com",
    "*.update.microsoft.com"
  ]
}
```

Use application rules when:

- The destination uses dynamic IPs.
- You need domain-level control.
- You want to inspect outbound web traffic by FQDN.

**DNAT rules**

DNAT (Destination Network Address Translation) rules publish private workloads through the firewall public IP. They are common for jump boxes, web apps, or partner access.

A DNAT rule maps:

- Public IP and port on the firewall
- To a private IP and port in the spoke
- With an allowed source scope when possible

Conceptual example:

```json
{
  "ruleType": "NatRule",
  "name": "dnat-rdp-jumpbox",
  "sourceAddresses": ["203.0.113.10/32"],
  "destinationPorts": ["3389"],
  "translatedAddress": "10.1.2.4",
  "translatedPort": "3389",
  "ipProtocols": ["TCP"]
}
```

> DNAT rules target the firewall’s public IP configuration. Do not assume the rule directly names the public IP in every CLI form; the public IP is attached to the firewall and the rule uses that published endpoint.

**DNS proxy and name resolution**

DNS is a common source of Azure Firewall failures. If you use application rules or FQDN-based filtering, you must plan DNS carefully.

**What DNS proxy does**

- Azure Firewall can act as a DNS proxy for workloads.
- Spokes can send DNS queries to the firewall private IP.
- The firewall forwards queries to upstream DNS servers you configure.

**What must be aligned**

- The firewall must have valid **upstream DNS servers** configured.
- The spoke VNet must use the firewall as its DNS server if you want centralized resolution.
- The firewall must be able to resolve the FQDNs used in application rules.
- If you use private endpoints, ensure private DNS zones are linked and resolvable.

A practical pattern:

```text
Spoke VM DNS -> Firewall private IP -> Upstream DNS -> Public DNS or private resolver
```

If you skip the upstream DNS configuration, DNS proxy can fail even though the firewall is reachable.

**Example DNS settings**

```bash
az network firewall update \
  --resource-group rg-network \
  --name azfw-hub \
  --dns-servers 10.0.1.4 10.0.1.5 \
  --enable-dns-proxy true
```

Then set the spoke VNet DNS server to the firewall private IP:

```bash
az network vnet update \
  --resource-group rg-network \
  --name vnet-spoke1 \
  --dns-servers 10.0.0.4
```

**Forced tunneling**

Forced tunneling sends internet-bound traffic to an on-premises security stack or another next hop instead of allowing direct internet egress from the firewall.

In the classic Azure Firewall model:

- `AzureFirewallSubnet` hosts the firewall data plane.
- `AzureFirewallManagementSubnet` is required for management-plane connectivity when forced tunneling is enabled.
- The management subnet is not optional in that model.

A common forced-tunneling design:

```text
Spoke subnet -> Firewall private IP -> On-premises via VPN/ER -> Internet
```

Use this when:

- Compliance requires all egress to pass through on-premises inspection.
- You need a central security stack outside Azure.
- You want to prevent direct internet egress from Azure workloads.

**Important nuance**

- In a secured virtual hub or Azure Firewall in Virtual WAN, routing behavior is governed by the Virtual WAN design, not the classic VNet deployment model.
- Do not assume the same subnet and management subnet rules apply identically across both models.

**Monitoring, logs, and metrics**

Azure Firewall monitoring is essential for troubleshooting and audit.

**Common log categories**

Depending on deployment mode and diagnostic configuration, you may see:

- **Application rule logs**
- **Network rule logs**
- **DNS proxy logs**
- **Firewall metrics**
- **Threat intelligence logs**
- **NAT/DNAT logs**

Current deployments often use **resource-specific tables** in Log Analytics rather than only `AzureDiagnostics`. Table names and category names vary by configuration, so validate your workspace schema before writing queries.

A safer Kusto Query Language (KQL) pattern is to inspect available tables first:

```kusto
search "Azure Firewall"
| summarize count() by $table
| order by count_ desc
```

Then query the relevant table in your workspace. For example, if your environment uses a resource-specific table, adapt the query to that table name.

**Example diagnostic settings**

```bash
az monitor diagnostic-settings create \
  --name fw-diag \
  --resource "/subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/azureFirewalls/azfw-hub" \
  --workspace "/subscriptions/<subId>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-sec" \
  --logs '[
    {"category":"AzureFirewallApplicationRule","enabled":true},
    {"category":"AzureFirewallNetworkRule","enabled":true},
    {"category":"AzureFirewallDnsProxy","enabled":true},
    {"category":"AzureFirewallNatRule","enabled":true}
  ]' \
  --metrics '[
    {"category":"AllMetrics","enabled":true}
  ]'
```

Category names can differ between classic diagnostics and resource-specific tables, so confirm the exact names in your tenant.

**Useful metrics to watch**

- **Throughput**
- **SNAT port utilization**
- **CPU or capacity-related indicators**
- **Dropped packets**
- **Health state**
- **Rule hit counts**

Alert on:

- Sudden spikes in denied traffic
- High SNAT port usage
- Firewall health degradation
- Unexpected outbound destinations

**Troubleshooting workflow**

When traffic fails, check the path in order.

1. Confirm the spoke subnet has the correct UDR.
2. Confirm the next hop points to the firewall private IP.
3. Confirm peering allows forwarded traffic.
4. Confirm the firewall rule allows the flow.
5. Confirm DNS resolution works for FQDN-based rules.
6. Confirm NSGs do not block the traffic before it reaches the firewall.
7. Review logs and metrics for the denied flow.

**Common failure patterns**

- Traffic bypasses the firewall because the UDR is missing.
- Return traffic fails because peering does not allow forwarded traffic.
- Application rules fail because DNS proxy or upstream DNS is misconfigured.
- Internet egress fails because the firewall has no valid public IP or the rule set is incomplete.
- On-premises routes override the intended firewall path because BGP propagation was not controlled.

**Real-world scenarios**

**1. Secure SaaS egress**

You want only approved Microsoft 365 and update endpoints from a workload subnet.

- Use application rules for FQDNs.
- Send all outbound traffic to the firewall with a default route.
- Log allowed and denied requests for audit.

Outcome:

- Users reach approved SaaS services.
- Unknown destinations are blocked or logged.
- Security teams can review outbound patterns centrally.

**2. Publish an admin jump box**

You need remote RDP access to a management VM in a spoke.

- Create a DNAT rule on the firewall public IP.
- Restrict source IPs to a trusted admin range.
- Keep the VM private with no public IP.

Outcome:

- The VM stays off the internet.
- Admin access is centralized and auditable.
- Exposure is limited to the firewall endpoint.

**3. Hub with VPN gateway and spokes**

You want spokes to use the hub gateway while still forcing inspection through Azure Firewall.

- Enable gateway transit on the hub peering.
- Enable use remote gateways on the spoke peering.
- Add UDRs so spoke traffic still traverses the firewall.
- Disable BGP propagation where learned routes would bypass inspection.

Outcome:

- Spokes reach on-premises networks through the hub.
- Security inspection remains centralized.
- Routing stays predictable.

**Exam focus points**

- Azure Firewall is deployed in a **dedicated subnet** named `AzureFirewallSubnet`.
- Classic forced tunneling requires `AzureFirewallManagementSubnet`.
- Hub-spoke designs depend on **peering settings** and **UDRs**.
- `Allow forwarded traffic` is critical in firewall transit scenarios.
- `Use remote gateways` and `Allow gateway transit` matter when sharing a hub gateway.
- Route propagation can bypass your intended inspection path if you do not control it.
- DNS proxy and upstream DNS must be aligned for FQDN-based rules.
- Rule order matters: **DNAT**, then **network**, then **application**.

**Quick reference**

| Topic | Key point |
|---|---|
| Deployment subnet | `AzureFirewallSubnet` |
| Forced tunneling | Requires `AzureFirewallManagementSubnet` in classic model |
| Spoke routing | UDR to firewall private IP |
| Peering | Allow forwarded traffic on both sides |
| Gateway sharing | Hub allows gateway transit; spoke uses remote gateways |
| DNS | Firewall DNS proxy needs upstream DNS |
| Logging | Use diagnostic settings and validate table schema |
| Rule types | DNAT, network, application |

**Summary**

Azure Firewall in a hub network gives you centralized inspection, simpler governance, and stronger segmentation. The design succeeds only when routing, peering, DNS, and logging are aligned. For AZ-500, focus on the deployment prerequisites, the traffic path, the rule evaluation order, and the routing controls that keep spoke traffic from bypassing the firewall.
