## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Azure network security commonly uses two layers of control: <abbr title="Network Security Group">NSG</abbr>-only filtering and firewall-based control. NSGs are distributed, subnet- or NIC-level packet filters that enforce allow/deny rules close to the workload. Azure Firewall is a centralized, stateful network security service that can inspect and control traffic across multiple spokes, subnets, and virtual networks.

The practical difference is scope and depth. NSG-only control is usually enough for basic segmentation and workload isolation. Firewall-based control is better when you need centralized policy, logging, threat intelligence, application-aware filtering, and consistent egress control.

**Core comparison**

| Capability | NSG-only control | Firewall-based control |
|---|---|---|
| Enforcement point | Subnet or NIC | Centralized hub or dedicated firewall subnet |
| Rule type | 5-tuple network rules | Network, application, and DNAT/SNAT policies |
| Statefulness | Stateful | Stateful |
| Application FQDN filtering | No | Yes, with application rules |
| Central logging | Limited to NSG flow logs and diagnostics | Rich firewall logs and policy analytics |
| East-west segmentation | Basic | Stronger, with forced routing through firewall |
| Internet egress control | Basic, often indirect | Strong, explicit outbound control |
| Threat intelligence | No native TI filtering | Yes, depending on SKU and configuration |
| TLS inspection | No | Supported in specific scenarios and SKUs, with certificate requirements |

**NSG-only control**

An NSG is a layer 4 filter that allows or denies traffic based on source, destination, port, and protocol. It is commonly attached to a subnet, a network interface, or both. Because NSGs are distributed, they scale well and are simple to reason about for local segmentation.

Use NSG-only control when you want to:
- Restrict traffic between subnets in a straightforward way.
- Protect a workload without introducing a centralized inspection path.
- Keep latency low and avoid extra routing complexity.
- Apply the same baseline rules to many subnets or NICs through automation.

NSGs are effective for:
- Denying inbound management ports from the internet.
- Allowing only required ports between application tiers.
- Blocking unnecessary lateral movement within a virtual network.
- Enforcing subnet-level trust boundaries.

A typical NSG rule set might look like this:

```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-web \
  --name Allow-HTTPS-In \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes Internet \
  --source-port-ranges "*" \
  --destination-address-prefixes "*" \
  --destination-port-ranges 443
```

Important NSG characteristics:
- Rules are evaluated by priority, with lower numbers processed first.
- Default rules exist and can affect traffic if you do not override them.
- NSGs do not inspect application content or fully understand URLs or domains.
- NSGs do not provide centralized internet egress governance by themselves.
- NSGs are not a substitute for a perimeter firewall when you need inspection and policy consolidation.

> NSGs are best viewed as distributed access control, not as a full security gateway.

**Firewall-based control**

Azure Firewall provides centralized traffic control and inspection. It is typically deployed in a hub-and-spoke design so that spoke networks route traffic through the firewall for inspection before reaching other spokes or the internet. This design gives you a single policy plane for many workloads.

Use firewall-based control when you need:
- Centralized outbound filtering for all workloads.
- Application-level rules based on FQDNs and URLs.
- Network address translation (<abbr title="Network Address Translation">NAT</abbr>) for inbound publishing.
- Better visibility into traffic patterns and denied flows.
- Consistent policy enforcement across multiple virtual networks.

Azure Firewall commonly supports:
- Network rules for IP, port, and protocol control.
- Application rules for HTTP, HTTPS, and FQDN-based filtering.
- DNAT rules for publishing internal services.
- Threat intelligence-based filtering, depending on configuration.
- Logging to Log Analytics, Storage, or Event Hub.

Example of a simple Azure Firewall policy concept:
- Allow outbound HTTPS only to approved domains.
- Deny all other internet-bound traffic.
- Publish a private web app through a DNAT rule.
- Force all spoke-to-spoke traffic through the firewall for inspection.

Example Azure CLI snippets:

```bash
az network firewall create \
  --resource-group rg-sec \
  --name azfw-hub \
  --location eastus
```

```bash
az network firewall network-rule create \
  --resource-group rg-sec \
  --firewall-name azfw-hub \
  --collection-name Allow-DNS \
  --name Allow-DNS-Out \
  --protocols UDP \
  --source-addresses 10.10.0.0/16 \
  --destination-addresses 168.63.129.16 \
  --destination-ports 53 \
  --action Allow \
  --priority 100
```

```bash
az network firewall application-rule create \
  --resource-group rg-sec \
  --firewall-name azfw-hub \
  --collection-name Allow-Web \
  --name Allow-Microsoft-Update \
  --source-addresses 10.10.0.0/16 \
  --protocols Http=80 Https=443 \
  --target-fqdns download.microsoft.com windowsupdate.microsoft.com \
  --action Allow \
  --priority 200
```

**When NSG-only control is enough**

NSG-only control is often sufficient for:
- Small environments with a few subnets and simple trust boundaries.
- Tier-to-tier restrictions inside a single application.
- Workloads that do not require domain-based egress filtering.
- Scenarios where cost and simplicity matter more than centralized inspection.

A common pattern is:
- Web subnet allows inbound 443 from the internet.
- App subnet allows inbound only from the web subnet.
- Database subnet allows inbound only from the app subnet.
- All other traffic is denied by default.

Example topology:

```text
Internet
   |
[NSG on Web Subnet]
   |
[Web VM / App Service integration]
   |
[NSG on App Subnet]
   |
[Database Subnet]
```

This works well when:
- Traffic paths are predictable.
- You do not need to inspect URLs or FQDNs.
- You can manage rules directly on each subnet or NIC.
- You are comfortable with decentralized policy ownership.

**When firewall-based control is better**

Firewall-based control is the better choice when:
- You need a single outbound policy for many workloads.
- Security teams must own and audit network policy centrally.
- You want to reduce rule sprawl across many NSGs.
- You need to control internet access by domain rather than only by IP.
- You need to publish services through a controlled inbound edge.

A hub-and-spoke design is common:

```text
                 Internet
                     |
                [Azure Firewall]
                     |
        -------------------------------
        |              |              |
     Spoke A        Spoke B        Spoke C
   (Web tier)     (App tier)     (Data tier)
```

In this model:
- Spoke subnets use user-defined routes (<abbr title="User-Defined Route">UDR</abbr>) to send traffic to the firewall.
- The firewall becomes the inspection and egress point.
- NSGs still matter, but they complement the firewall rather than replace it.

**How they work together**

NSGs and Azure Firewall are not mutually exclusive. In mature designs, you use both:
- NSGs provide local subnet or NIC protection.
- Azure Firewall provides centralized inspection and egress governance.
- Route tables steer traffic to the firewall.
- Diagnostic logs from both layers help you troubleshoot and audit.

A common layered approach is:
1. Use NSGs to block obviously unnecessary traffic at the workload boundary.
2. Use UDRs to force traffic through the firewall.
3. Use firewall rules for domain, port, and application control.
4. Use logging to validate policy and detect anomalies.

This layered model improves defense in depth. If one control is misconfigured, the other may still reduce exposure.

**Routing considerations**

Firewall-based control usually requires explicit routing. Without routes, traffic may bypass the firewall and go directly to its destination.

Example UDR for a spoke subnet:

```bash
az network route-table create \
  --resource-group rg-net \
  --name rt-spoke-a

az network route-table route create \
  --resource-group rg-net \
  --route-table-name rt-spoke-a \
  --name DefaultToFirewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

Key routing points:
- `0.0.0.0/0` sends internet-bound traffic to the firewall.
- Specific routes can steer east-west traffic through the firewall.
- Return traffic must follow a valid path to preserve stateful inspection.
- Asymmetric routing can break flows or create confusing troubleshooting symptoms.

**Logging and monitoring**

NSGs and firewalls produce different telemetry.

NSG visibility:
- NSG flow logs can show allowed and denied flows.
- Diagnostic logs help with rule evaluation and troubleshooting.
- Visibility is useful, but it is usually less application-aware.

Firewall visibility:
- Logs can show rule matches, denies, and application destinations.
- You can correlate traffic across many spokes.
- Centralized logging supports security operations and incident response.

Practical monitoring workflow:
- Check NSG flow logs for unexpected subnet-level denies.
- Check firewall logs for denied FQDNs or ports.
- Correlate with route tables if traffic appears to bypass inspection.
- Validate effective security rules on NICs and subnets when troubleshooting.

**Performance and operational tradeoffs**

NSG-only control is lightweight and low overhead. It is usually easier to deploy and maintain, especially in small environments.

Firewall-based control adds:
- More design complexity.
- Additional cost.
- Potential latency from hairpinning traffic through a central point.
- More operational value when policy consistency and inspection are required.

Choose based on:
- Number of workloads and networks.
- Need for centralized governance.
- Compliance and audit requirements.
- Need for domain-based or application-aware filtering.
- Tolerance for routing complexity and cost.

**Practical decision guide**

Use this quick rule of thumb:
- Choose **NSG-only** if you need simple segmentation and local access control.
- Choose **Azure Firewall** if you need centralized egress, FQDN filtering, or inbound publishing.
- Choose **both** if you need defense in depth and enterprise-scale governance.

**Example scenarios**

- A three-tier app in one VNet: NSG-only may be enough.
- A multi-subscription enterprise with shared internet egress: firewall-based control is usually preferred.
- A regulated environment requiring centralized logging and outbound allow lists: firewall-based control is the stronger fit.
- A workload that only needs to block SSH from the internet: an NSG rule is often sufficient.

**Exam focus points**

For AZ-500, be ready to distinguish:
- **NSG** as distributed packet filtering at subnet or NIC scope.
- **Azure Firewall** as centralized, stateful, and policy-driven.
- The need for **UDRs** when forcing traffic through a firewall.
- The difference between network rules and application rules.
- Why NSGs alone do not provide domain-based egress control.

**Summary**

NSG-only control is simple, fast, and effective for local segmentation. Firewall-based control adds centralized inspection, richer logging, and application-aware policy. In real Azure architectures, you often combine both: NSGs for workload-level boundaries and Azure Firewall for enterprise-wide traffic governance.
