## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Azure Firewall rule model**

Azure Firewall is a managed, stateful firewall service that inspects traffic only when that traffic is explicitly routed through it. In practice, you place the firewall in a hub or shared services network and force traffic from spokes, on-premises networks, or internet-bound workloads through it using routes, peering design, or gateway transit patterns. Traffic that does not traverse the firewall, such as intra-VNet traffic that is not forced through the firewall, bypasses inspection.

Azure Firewall supports three rule types:

- **DNAT** rules translate public IP traffic to private IP targets.
- **Network** rules filter by IP address, port, and protocol.
- **Application** rules filter HTTP/HTTPS traffic by FQDN, URL path in some scenarios, and related web properties.

For AZ-500, the most important distinction is that Azure Firewall rules are commonly managed through **Azure Firewall Policy**, not only on the firewall resource itself. Policy-based management is the modern model and is what you should expect in enterprise deployments.

**Firewall Policy versus classic firewall configuration**

Classic Azure Firewall configuration places rules directly on the firewall resource. Firewall Policy centralizes rule management and lets you reuse the same policy across multiple firewalls. This matters because policy inheritance and shared governance are common exam themes.

Key differences:

- **Classic model**
  - Rules are configured on the firewall resource.
  - Less flexible for multi-firewall governance.
  - Common in older deployments.

- **Policy model**
  - Rules live in a firewall policy.
  - One policy can be associated with one or more firewalls.
  - Supports centralized management, rule collection groups, and inheritance patterns.
  - Better suited for enterprise segmentation and delegated administration.

When a firewall uses a policy, the policy becomes the source of truth for rule evaluation. If multiple firewalls share the same policy, they inherit the same rule set unless a different policy is associated.

**Rule hierarchy and evaluation order**

Azure Firewall policy uses a strict hierarchy:

1. **Rule collection group**
2. **Rule collection**
3. **Rule**

Evaluation is not based on a vague “most specific rule wins” model. Instead, Azure Firewall evaluates in this order:

- Rule collection groups are processed by **priority**.
- Within a rule collection group, rule collections are processed by **priority**.
- Within a rule collection, rules are evaluated in the order they are defined.

This is critical for exam accuracy. A broad allow rule does not automatically override a deny rule in another collection group with higher priority. The effective result depends on the exact priority chain and the rule type.

Important behavior:

- **Lower numeric priority value** means higher precedence.
- **Deny** rules stop evaluation when matched.
- **Allow** rules permit traffic only when the packet matches the rule and no earlier deny has already applied.
- Rule type matters: network, application, and DNAT rules are evaluated in their respective contexts.

> Azure Firewall only evaluates traffic that reaches it. If routing does not send the flow through the firewall, no rule can match it.

**Rule collection types**

Azure Firewall policy uses three collection types, each with a distinct purpose.

| Collection type | Purpose | Typical match fields | Common use |
|---|---|---|---|
| DNAT rule collection | Inbound translation from public to private | Public IP, destination port, translated IP/port | Publish internal services |
| Network rule collection | L3/L4 filtering | Source, destination, protocol, ports, IP ranges | East-west and outbound control |
| Application rule collection | L7 web filtering | Source, FQDN, FQDN tags, HTTP/HTTPS, ports | Web access control |

Use the right collection type for the traffic you want to control. If you need to allow TCP 1433 to a SQL server by IP, use a network rule. If you need to allow `https://login.microsoftonline.com`, use an application rule.

**Network rules**

Network rules operate at Layer 3 and Layer 4. They are appropriate when you need to allow or deny traffic based on IP addresses, address ranges, ports, and protocols.

Typical fields:

- **Source addresses**: one or more IPs, CIDR ranges, or address prefixes
- **Destination addresses**: private IPs, public IPs, or address ranges
- **Destination ports**: single ports or ranges such as `1433`, `3389`, or `5000-5010`
- **Protocols**: `TCP`, `UDP`, or `Any` depending on the scenario

Use network rules for:

- Database access such as SQL Server on `1433`
- Remote administration such as RDP on `3389` or SSH on `22`
- Custom application ports
- On-premises to Azure private IP flows over VPN or ExpressRoute

Example network rule collection concept:

```json
{
  "name": "Allow-SQL-From-AppSubnet",
  "priority": 200,
  "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
  "action": {
    "type": "Allow"
  },
  "rules": [
    {
      "name": "Allow-Sql-Tcp",
      "ruleType": "NetworkRule",
      "sourceAddresses": [
        "10.10.1.0/24"
      ],
      "destinationAddresses": [
        "10.20.2.4"
      ],
      "destinationPorts": [
        "1433"
      ],
      "ipProtocols": [
        "TCP"
      ]
    }
  ]
}
```

This is a conceptual policy fragment, not a full ARM template. In real deployments, the rule collection sits inside a rule collection group.

Practical guidance:

- Use the narrowest source range possible.
- Prefer private IP targets for internal services.
- Use port ranges only when the application truly requires them.
- Remember that network rules do not inspect HTTP hostnames or URLs.

**Application rules**

Application rules inspect web traffic at Layer 7. They are designed for HTTP and HTTPS scenarios where you want to control access by domain name rather than by IP address.

Typical fields:

- **Source addresses**
- **Protocol**: HTTP or HTTPS
- **Target FQDNs**: fully qualified domain names such as `www.contoso.com`
- **FQDN tags**: Microsoft-managed tags for common service groups

Use application rules for:

- Browsing control
- SaaS access control
- Microsoft service access patterns
- Domain-based outbound filtering

Example application rule collection concept:

```json
{
  "name": "Allow-Microsoft-Services",
  "priority": 100,
  "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
  "action": {
    "type": "Allow"
  },
  "rules": [
    {
      "name": "Allow-Azure-Portal",
      "ruleType": "ApplicationRule",
      "sourceAddresses": [
        "10.10.1.0/24"
      ],
      "protocols": [
        {
          "protocolType": "Https",
          "port": 443
        }
      ],
      "targetFqdns": [
        "portal.azure.com"
      ]
    }
  ]
}
```

Again, this is a conceptual sketch. In Azure, application rules are usually authored inside a policy rule collection group.

Important application rule concepts:

- **FQDN tags** simplify access to Microsoft service endpoints.
- **Wildcard FQDNs** can be used in some cases, but they are not a substitute for precise design.
- **HTTPS inspection** is limited by TLS behavior and certificate trust requirements.
- **HTTP versus HTTPS** matters because application rules are web-aware, not generic TCP filters.

**FQDN tags and common Microsoft access**

FQDN tags are predefined tags that represent groups of Microsoft service endpoints. They reduce operational overhead when you need to allow access to Azure or Microsoft management services without maintaining a long list of changing domains.

Examples of how they help:

- Allow Azure management traffic for administrators
- Reduce rule sprawl for Microsoft service dependencies
- Simplify maintenance when service endpoints change

Use FQDN tags carefully:

- They are convenient for Microsoft-managed services.
- They are not a general replacement for custom application allow lists.
- You still need to validate whether the tag covers the exact service scope you require.

**DNS behavior and name resolution**

Azure Firewall has its own DNS settings. It can use Azure DNS or custom DNS servers depending on how you configure the firewall policy or firewall resource. This is important because application rules depend on name resolution for FQDN matching.

Practical implications:

- If DNS resolution fails, application rules may not match as expected.
- If your environment uses custom DNS, ensure the firewall can resolve the target FQDNs.
- If you use private endpoints or internal names, make sure the firewall’s DNS path can resolve those names correctly.

Example of setting DNS servers on a firewall resource with Azure CLI:

```bash
az network firewall update \
  --name azfw-hub \
  --resource-group rg-network \
  --dns-servers 10.0.0.4 10.0.0.5
```

Use this carefully in environments where the firewall must resolve internal names or where outbound web filtering depends on custom DNS.

**Creating rules in Azure Portal**

In the Azure Portal, you usually create rules through the firewall policy, not directly on the firewall resource.

Typical workflow:

1. Open the **Firewall Policy** associated with your Azure Firewall.
2. Select **Rule collection groups**.
3. Create or open a rule collection group.
4. Add a rule collection of the correct type:
   - Network
   - Application
   - DNAT
5. Set the collection priority.
6. Add one or more rules.
7. Save and associate the policy with the firewall if needed.

Portal design tips:

- Put broad, shared rules in lower-priority collection groups only when they are truly intended to be evaluated later.
- Keep application and network rules separated for clarity.
- Use descriptive names that reflect source, destination, and intent.

**Azure CLI examples**

Azure Firewall policy rule creation is typically done with the policy and rule collection group commands rather than standalone “create rule” commands. The exact CLI syntax can vary by extension version, so validate with `az network firewall policy rule-collection-group -h` in your environment.

A practical pattern is to create a rule collection group and include the rule collection definitions in the same operation.

Example: create a network rule collection group with one allow rule.

```bash
az network firewall policy rule-collection-group create \
  --policy-name fwpol-hub \
  --resource-group rg-network \
  --name rcg-outbound \
  --priority 200 \
  --rule-collections '[
    {
      "name": "Allow-SQL",
      "priority": 200,
      "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
      "action": { "type": "Allow" },
      "rules": [
        {
          "name": "Allow-Sql-Tcp",
          "ruleType": "NetworkRule",
          "sourceAddresses": ["10.10.1.0/24"],
          "destinationAddresses": ["10.20.2.4"],
          "destinationPorts": ["1433"],
          "ipProtocols": ["TCP"]
        }
      ]
    }
  ]'
```

Example: create an application rule collection group with an allow rule for a web domain.

```bash
az network firewall policy rule-collection-group create \
  --policy-name fwpol-hub \
  --resource-group rg-network \
  --name rcg-web \
  --priority 100 \
  --rule-collections '[
    {
      "name": "Allow-Portal",
      "priority": 100,
      "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
      "action": { "type": "Allow" },
      "rules": [
        {
          "name": "Allow-Azure-Portal",
          "ruleType": "ApplicationRule",
          "sourceAddresses": ["10.10.1.0/24"],
          "protocols": [
            { "protocolType": "Https", "port": 443 }
          ],
          "targetFqdns": ["portal.azure.com"]
        }
      ]
    }
  ]'
```

Example: create a deny network rule collection to block SSH to a subnet.

```bash
az network firewall policy rule-collection-group create \
  --policy-name fwpol-hub \
  --resource-group rg-network \
  --name rcg-deny \
  --priority 50 \
  --rule-collections '[
    {
      "name": "Deny-SSH",
      "priority": 50,
      "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
      "action": { "type": "Deny" },
      "rules": [
        {
          "name": "Block-SSH-To-Servers",
          "ruleType": "NetworkRule",
          "sourceAddresses": ["10.10.1.0/24"],
          "destinationAddresses": ["10.20.2.0/24"],
          "destinationPorts": ["22"],
          "ipProtocols": ["TCP"]
        }
      ]
    }
  ]'
```

Use these examples as patterns, not as copy-paste production templates. Always confirm the current CLI schema in your subscription.

**PowerShell examples**

PowerShell is useful for automation, but you should create meaningful rule collections rather than empty containers.

Example: create a network rule collection object and attach it to a rule collection group.

```powershell
$netRule = New-AzFirewallPolicyNetworkRule `
  -Name "Allow-Sql-Tcp" `
  -SourceAddress "10.10.1.0/24" `
  -DestinationAddress "10.20.2.4" `
  -DestinationPort "1433" `
  -Protocol TCP

$netCollection = New-AzFirewallPolicyFilterRuleCollection `
  -Name "Allow-SQL" `
  -Priority 200 `
  -ActionType Allow `
  -Rule $netRule

New-AzFirewallPolicyRuleCollectionGroup `
  -Name "rcg-outbound" `
  -Priority 200 `
  -ResourceGroupName "rg-network" `
  -FirewallPolicyName "fwpol-hub" `
  -RuleCollection $netCollection
```

Example: create an application rule collection object.

```powershell
$appRule = New-AzFirewallPolicyApplicationRule `
  -Name "Allow-Azure-Portal" `
  -SourceAddress "10.10.1.0/24" `
  -Protocol @(@{ ProtocolType = "Https"; Port = 443 }) `
  -TargetFqdn "portal.azure.com"

$appCollection = New-AzFirewallPolicyFilterRuleCollection `
  -Name "Allow-Portal" `
  -Priority 100 `
  -ActionType Allow `
  -Rule $appRule

New-AzFirewallPolicyRuleCollectionGroup `
  -Name "rcg-web" `
  -Priority 100 `
  -ResourceGroupName "rg-network" `
  -FirewallPolicyName "fwpol-hub" `
  -RuleCollection $appCollection
```

If your module version differs, check the current cmdlet help before using the syntax in automation.

**Practical design patterns**

A common enterprise topology is hub-and-spoke:

- The firewall sits in the hub.
- Spoke VNets route outbound traffic to the firewall.
- On-premises networks reach Azure through VPN or ExpressRoute and use the firewall as a central inspection point.
- Shared firewall policy applies consistent controls across multiple spokes.

Design considerations:

- Use UDRs (user-defined routes) to force traffic to the firewall.
- Ensure return traffic follows a symmetric path when required.
- Separate policy by environment if dev, test, and prod have different trust levels.
- Use rule collection group priorities to separate baseline controls from application-specific exceptions.

Example scenario:

- Spoke subnet `10.10.1.0/24` hosts application servers.
- Database subnet `10.20.2.0/24` hosts SQL.
- You allow TCP `1433` from app to DB with a network rule.
- You allow `portal.azure.com` for admin workstations with an application rule.
- You deny SSH to all server subnets with a higher-precedence deny collection group.

**Troubleshooting rule mismatches**

When traffic does not behave as expected, check the path first, then the policy, then the logs.

Common causes:

- Traffic never reaches the firewall because routing is missing.
- The wrong policy is associated with the firewall.
- A higher-priority deny rule matches before the allow rule.
- DNS resolution fails for an application rule.
- The source or destination address does not match the actual packet flow.
- The protocol or port is wrong.
- The traffic is not HTTP/HTTPS, so an application rule cannot match it.

A practical troubleshooting sequence:

1. Confirm the route table sends traffic to the firewall.
2. Confirm the firewall is associated with the intended policy.
3. Check rule collection group priorities.
4. Check the specific rule fields against the observed flow.
5. Review logs for the matched rule and action.

Example KQL for Azure Monitor logs when using resource-specific tables or diagnostic logs:

```kusto
AzureDiagnostics
| where ResourceType == "AZUREFIREWALLS"
| where Category in ("AzureFirewallNetworkRule", "AzureFirewallApplicationRule", "AzureFirewallNatRule")
| project TimeGenerated, Category, msg_s, action_s, ruleCollectionGroup_s, ruleCollection_s, rule_s, src_ip_s, dst_ip_s, dst_port_s, protocol_s
| order by TimeGenerated desc
```

Depending on your diagnostic configuration, you may also see resource-specific tables instead of `AzureDiagnostics`. Adjust the query to match your workspace schema.

Example of diagnosing a mismatch:

- You expect outbound HTTPS to `portal.azure.com`.
- Logs show the flow is denied by a higher-priority deny collection group.
- The deny rule matches the source subnet and `TCP 443`.
- Resolution: move the allow rule to a higher-priority collection group or narrow the deny scope so it does not block the intended admin subnet.

Useful fields to inspect in logs:

- **ruleCollectionGroup**
- **ruleCollection**
- **rule**
- **action**
- **src_ip**
- **dst_ip**
- **dst_port**
- **protocol**

**Common exam scenarios**

- Allow a subnet to reach a specific SaaS endpoint by FQDN.
- Allow application servers to reach SQL on a private IP and port.
- Deny administrative protocols from user subnets.
- Centralize rules in a policy shared by multiple firewalls.
- Use FQDN tags for Microsoft service access.
- Diagnose why a rule does not match because the traffic bypasses the firewall.

**Key takeaways**

Azure Firewall rule creation is easiest to understand when you separate the traffic type from the management model:

- Use **network rules** for IP, port, and protocol control.
- Use **application rules** for HTTP/HTTPS domain-based control.
- Use **DNAT** for inbound translation.
- Manage enterprise deployments with **Firewall Policy** and rule collection groups.
- Remember the evaluation chain: **rule collection group priority**, then **rule collection priority**, then **rule order**.

If you can explain why a packet matches or does not match a rule, you are ready for AZ-500-style questions on Azure Firewall.
