## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Azure security notifications come from two related but different systems:

- **Defender for Cloud security contacts** control who receives security alerts and how often they are notified.
- **Azure Monitor alert routing** controls where platform, metric, log, and activity log alerts are sent after they fire.

For AZ-500, you need to know both the configuration model and the practical difference between them. Security contacts are about Defender for Cloud alert delivery. Alert routing is about Azure Monitor alert actions, suppression, and notification fan-out.

**Defender for Cloud security contacts**

Security contacts are configured per subscription in Microsoft Defender for Cloud. They define the notification recipients and preferences for security alerts generated by Defender for Cloud.

Common settings include:

- **Security alert emails**: the primary email address used for security notifications.
- **Additional email recipients**: extra addresses that also receive alerts.
- **Alert notification frequency**: how often repeated alerts are sent.
- **Minimum alert severity**: the lowest severity level that triggers email notification.

These settings are often tested because they determine whether a team gets immediate notification or only higher-severity alerts.

> Defender for Cloud security contacts are subscription-scoped. If you need consistent settings across many subscriptions, use Azure Policy to deploy or enforce the configuration rather than expecting a single management-group-wide contact list.

A typical portal path is:

1. Open **Microsoft Defender for Cloud**.
2. Go to **Management** or **Environment settings**.
3. Select the target **subscription**.
4. Open **Security contacts**.
5. Configure email recipients, severity threshold, and notification frequency.
6. Save the settings.

**Practical portal behavior**

- The portal usually lets you specify one or more email recipients.
- Notification frequency is useful when you want to reduce repeated emails for the same issue.
- Severity filtering helps you avoid alert fatigue by sending only high-value alerts to the inbox.

**Example ARM/Bicep-style configuration**

The exact resource type and API version can vary by platform updates, but the concept is to deploy the security contact settings at subscription scope.

```bicep
targetScope = 'subscription'

resource securityContact 'Microsoft.Security/securityContacts@2023-01-01-preview' = {
  name: 'default'
  properties: {
    emails: 'secops@contoso.com;cloudsec@contoso.com'
    alertNotifications: 'On'
    notificationsByRole: 'Off'
    phone: ''
    alertSeverity: 'High'
  }
}
```

Use this pattern as a conceptual reference. In practice, verify the current API version and property names in your tenant before deploying.

**Azure Policy and inheritance**

Security contact settings can be standardized with Azure Policy:

- Use policy to **deployIfNotExists** the security contact configuration.
- Assign the policy at a **management group** to push consistent settings to child subscriptions.
- Remediate existing subscriptions so they inherit the desired baseline.

This matters in exam scenarios because the security contact itself is still subscription-level, but the enforcement mechanism can be management-group-driven.

**Validation and testing**

After configuration, confirm that the settings are effective:

- Check the **Security contacts** blade in the subscription.
- Trigger or simulate a test alert if your environment supports it.
- Verify that the recipient mailbox receives the notification.
- Confirm the severity threshold matches the alert you expect to receive.

If you do not receive mail, check:

- Spam or quarantine filters.
- Whether the alert severity is below the configured threshold.
- Whether the subscription is the one actually generating the alert.
- Whether policy remediation has completed.

**Alert routing in Azure Monitor**

Azure Monitor routes alerts through **action groups**. An action group is a reusable notification and automation container that can be attached to many alert rules.

Action groups can be reused across:

- Metric alerts
- Log alerts
- Activity log alerts
- Service health alerts
- Resource health alerts

A single action group can contain multiple receivers and action types, which makes it a central routing object for operations teams.

Common receiver types include:

- **Email**
- **SMS**
- **Push notifications** in the Azure mobile app
- **Voice**
- **Webhook**
- **Azure Function**
- **Logic App**
- **Automation Runbook**
- **ITSM connector** where supported
- **Event Hub** where supported

> Not every receiver type is equally available in every tenant or scenario. For exam prep, focus on the core receivers first: email, SMS, webhook, Logic App, Azure Function, and Automation Runbook.

**Action group example**

```bash
az monitor action-group create \
  --resource-group rg-monitoring \
  --name ag-secops \
  --short-name secops \
  --action email ops-team ops-team@contoso.com \
  --action webhook ticketing https://contoso.example/webhook
```

This example shows the pattern: create the action group once, then reuse it in multiple alert rules. The exact receiver syntax can vary by CLI version, so always validate with `az monitor action-group create -h` in your environment.

**Metric alert example**

Metric alerts trigger when a metric crosses a threshold.

```bash
az monitor metrics alert create \
  --name cpu-high \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm01 \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --description "CPU above 80 percent" \
  --action ag-secops
```

Use this as a pattern only. In real deployments, confirm the current CLI syntax for metric conditions and action group references. The important exam concept is that the alert rule evaluates a metric over a time window and then invokes the action group.

**Activity log alert example**

Activity log alerts are useful for control-plane events such as:

- Resource creation or deletion
- Role assignment changes
- Policy changes
- Service health events

They are often used for governance and security monitoring because they detect management-plane actions rather than performance metrics.

```bash
az monitor activity-log alert create \
  --name alert-rg-delete \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<sub-id> \
  --condition category=Administrative \
  --action-group ag-secops
```

**Log alert example**

Log alerts use Kusto Query Language (KQL) against Log Analytics data.

```kusto
SecurityEvent
| where EventID == 4625
| summarize FailedLogons = count() by Computer, bin(TimeGenerated, 5m)
| where FailedLogons > 20
```

A log alert is appropriate when you need detection logic that is more expressive than a simple threshold. For example, repeated failed sign-ins, suspicious process creation, or unusual administrative activity.

**Alert processing rules**

Alert processing rules help you reduce noise by modifying alert notifications after the alert is generated.

Use them to:

- Suppress notifications during maintenance windows.
- Apply actions to a defined scope of resources.
- Control when alerts are delivered to receivers.

They are not a replacement for alert rules. The alert still fires; the processing rule changes how notifications are handled.

A practical suppression scenario:

- Scope: all VMs in `rg-dev`
- Condition: any alert from those VMs
- Time window: Saturday 22:00 to Sunday 02:00
- Action: suppress notifications

Conceptually:

```text
Scope: /subscriptions/<sub-id>/resourceGroups/rg-dev
Condition: alert rule matches selected resources
Schedule: weekly maintenance window
Action: suppress notifications
```

This is useful when patching or planned maintenance would otherwise generate expected alerts.

**Alert grouping and smart groups**

Azure Monitor can group related alerts to reduce operational noise.

- **Smart groups** cluster similar alerts so responders can investigate one incident instead of many duplicates.
- Alert grouping is useful when a single root cause triggers multiple downstream alerts.
- Grouping does not stop the original alerts from existing; it helps with triage and incident management.

Use grouping when:

- One outage causes many resource alerts.
- You want to reduce duplicate incident handling.
- You need a cleaner handoff to an operations team.

**Defender for Cloud vs Azure Monitor routing**

These systems overlap in purpose but not in function.

| Area | Defender for Cloud security contacts | Azure Monitor alert routing |
|---|---|---|
| Primary purpose | Notify on Defender for Cloud security alerts | Route alerts from Azure Monitor rules |
| Scope | Subscription | Alert rule and action group scope |
| Main object | Security contact settings | Action groups and alert processing rules |
| Typical use | Security alert emails and thresholds | Multi-channel notification and automation |
| Best for | Defender-generated security findings | Metric, log, activity log, and service alerts |

Use Defender for Cloud contacts when the alert originates from Defender for Cloud. Use Azure Monitor action groups when you need broader alert routing across Azure services.

**Common exam distinctions**

- **Security contacts** are not the same as **action groups**.
- **Continuous export** sends Defender for Cloud data to destinations such as Log Analytics, Event Hub, or a storage account; it is not the same as email notification.
- **Workflow automation** in Defender for Cloud can trigger Logic Apps based on security alerts or recommendations.
- **Alert processing rules** suppress or modify notifications; they do not create alerts.
- **Action groups** can be attached to multiple alert rules and can contain multiple receivers.

**Practical configuration workflow**

1. Define who should receive security notifications.
2. Configure Defender for Cloud security contacts for each subscription.
3. Standardize the settings with Azure Policy if you manage many subscriptions.
4. Create Azure Monitor action groups for operations and incident response.
5. Attach action groups to metric, log, and activity log alerts.
6. Add alert processing rules to suppress expected noise.
7. Validate delivery with a test alert or controlled trigger.

**Troubleshooting**

If alerts are not reaching the right people, check the full path:

- Confirm the alert source:
  - Defender for Cloud security alert
  - Azure Monitor metric alert
  - Azure Monitor log alert
  - Activity log alert
- Confirm the routing object:
  - Security contact
  - Action group
  - Alert processing rule
- Confirm the scope:
  - Correct subscription
  - Correct resource group
  - Correct resource
- Confirm suppression:
  - Maintenance window
  - Processing rule
  - Severity threshold
- Confirm delivery:
  - Email address validity
  - Mail filtering
  - Webhook endpoint health
  - Logic App or Function execution status

For Defender for Cloud findings, also consider:

- **Workflow automation** for response actions.
- **Continuous export** for downstream analytics or SIEM integration.
- **Microsoft Sentinel** if you need centralized security operations and incident correlation.

**Key takeaways**

- Defender for Cloud security contacts are subscription-scoped notification settings.
- Azure Policy can enforce those settings across many subscriptions.
- Azure Monitor uses action groups to route alerts and can reuse them across many rules.
- Alert processing rules suppress or modify notifications, which helps reduce noise.
- Smart groups and alert grouping help responders handle duplicate alerts more efficiently.
- Always verify the alert source before choosing the routing method.
