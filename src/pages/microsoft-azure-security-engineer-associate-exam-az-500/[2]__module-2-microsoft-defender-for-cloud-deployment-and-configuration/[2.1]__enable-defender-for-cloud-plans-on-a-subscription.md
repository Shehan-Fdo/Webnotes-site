## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Microsoft Defender for Cloud helps you assess security posture and enable workload-specific protections across Azure subscriptions. At subscription scope, you typically turn on a plan for a workload area such as servers, containers, databases, storage, or key management so Defender for Cloud can collect signals, generate recommendations, and apply advanced protections where supported.

For AZ-500, focus on three things:

- **Scope**: you enable plans on a subscription, but some settings can be inherited from management groups or constrained by Azure Policy.
- **Tier**: many plans have a **Free** tier and a paid tier such as **Standard** or **Defender** depending on the plan and current product naming.
- **Verification**: you must confirm whether a plan is **Enabled**, **Inherited**, or **Not applicable** before assuming the subscription is protected.

> Plan names and available tiers can change over time. In exam or lab scenarios, use the portal labels shown in the tenant rather than memorizing a single static command syntax.

**Defender for Cloud plan model**

Defender for Cloud uses plan-based onboarding. A plan is not the same as a single feature toggle; it is a subscription-level security offering for a workload area.

Common exam-relevant ideas:

- **Free tier** usually provides basic security posture management and recommendations.
- **Paid tier** enables advanced threat protection, enhanced detections, or workload-specific hardening features where available.
- Some plans are enabled directly on the subscription.
- Some capabilities are governed by policy assignments, resource provider registration, or service-specific onboarding steps.

A practical mental model:

- You select a subscription.
- You choose a plan for a workload area.
- Azure records the plan state.
- Defender for Cloud begins evaluating supported resources in that subscription.

**Common plan areas you may see**

The exact labels vary by portal version and tenant, but exam questions often reference these workload areas:

- **Servers** for virtual machines and servers.
- **App Service** for web apps.
- **Containers** for Kubernetes and container-related protections.
- **Databases** for supported database services.
- **Storage** for storage account protections.
- **Key Vault** for secrets and key protection.
- **Resource Manager** or similar governance-related protections in some environments.

Use the portal labels you see in the subscription. Do not assume every plan is enabled the same way or at the same scope.

**Prerequisites and constraints**

Before enabling a plan, check the subscription context and service constraints.

- The subscription must support the plan you want to enable.
- The subscription type can matter:
  - Pay-as-you-go subscriptions commonly support paid plans.
  - Trial or sponsorship subscriptions may have limitations.
  - Some enterprise agreements or CSP-managed subscriptions can have different billing behavior.
- The target workload must be supported in the region and by the service.
- Azure Policy can enforce or deny plan settings.
- Management group inheritance can make a subscription show an inherited state instead of a directly editable one.

> If a plan is controlled by policy at a higher scope, you may not be able to change it directly at the subscription level until the policy assignment is adjusted.

**Enable a plan in the Azure portal**

This is the most common exam scenario.

1. Sign in to the Azure portal.
2. Open **Microsoft Defender for Cloud**.
3. Select **Environment settings**.
4. Choose the target **subscription**.
5. Open **Defender plans** or the equivalent plan configuration page.
6. Find the workload plan you want to enable.
7. Set the plan to the paid tier or **On** state shown in the portal.
8. Save the change.
9. Wait for the configuration to propagate, then verify the status.

Typical portal indicators:

- **Enabled**: the subscription has the plan turned on directly.
- **Inherited**: the setting comes from a management group or higher-level policy.
- **Not applicable**: the plan cannot be applied to that subscription or workload.
- **Off** or **Free**: the plan is not providing the paid protections.

A typical exam answer often expects you to navigate to **Microsoft Defender for Cloud > Environment settings > subscription > Defender plans**.

**Example: enable a server protection plan**

A common scenario is enabling the server-related plan for a subscription so Defender for Cloud can assess and protect virtual machines.

Expected outcome:

- The subscription shows the server plan as enabled.
- Defender for Cloud begins surfacing recommendations and alerts for supported machines.
- If the subscription already had policy-based onboarding, the portal may show the setting as inherited rather than directly editable.

**CLI example**

Azure CLI command behavior can vary by version, so verify with `az security pricing -h` in your environment before using it in a lab.

```bash
az security pricing -h
az security pricing list --subscription <subscription-id>
az security pricing create --name VirtualMachines --tier Standard --subscription <subscription-id>
```

If your CLI version uses `update` for the same operation, confirm the supported syntax first:

```bash
az security pricing update --name VirtualMachines --tier Standard --subscription <subscription-id>
```

Interpretation:

- `VirtualMachines` is the pricing resource name for the server plan in many tenants.
- `Standard` commonly represents the paid tier for that plan.
- After the command succeeds, the subscription should report the plan as enabled or standard in the portal.

**PowerShell example**

PowerShell cmdlet availability and parameter names can vary by Az module version. Verify with `Get-Command` and module help before using in production.

```powershell
Get-Command *Security*Pricing*
Get-Help Set-AzSecurityPricing -Full
Set-AzSecurityPricing -Name "VirtualMachines" -PricingTier "Standard"
```

Use PowerShell when you want repeatable automation, but always confirm the current module syntax in your environment.

**Verify enablement**

Verification matters as much as configuration.

In the portal, check:

- **Environment settings** for the subscription.
- The plan row for the workload area.
- The current state label:
  - **Enabled** means the subscription is directly configured.
  - **Inherited** means a parent scope controls the setting.
  - **Not applicable** means the plan cannot be used there.
- The recommendations and secure score updates after a short delay.

You can also verify with CLI:

```bash
az security pricing list --subscription <subscription-id> -o table
```

Example output pattern:

```text
Name             PricingTier
---------------   -----------
VirtualMachines   Standard
StorageAccounts   Free
SqlServers        Standard
```

Read the output as follows:

- `Standard` indicates the paid plan is active for that workload area.
- `Free` indicates the subscription is not using the paid plan for that area.
- Missing entries or unexpected values usually mean the plan is not enabled, not supported, or controlled elsewhere.

**Billing and subscription type considerations**

Plan enablement is not just a security decision; it is also tied to subscription eligibility.

- Paid plans generally require a subscription type that supports billing for the service.
- Trial subscriptions may allow limited testing but can expire or restrict some features.
- Some plans are unavailable in certain subscription types or regions.
- If a subscription is under a billing arrangement that does not support the plan, the portal may block enablement or show the plan as unavailable.

For exam scenarios, distinguish between:

- **Unsupported subscription type**: the plan cannot be enabled.
- **Policy restriction**: the plan is technically supported but blocked by governance.
- **Inherited configuration**: the plan is already set at a higher scope.

**Azure Policy and management group inheritance**

Azure Policy often controls Defender for Cloud plan settings at scale.

What this means:

- A management group can apply a policy that enables a plan for all child subscriptions.
- A subscription may show **Inherited** because the setting is not locally editable.
- A deny or modify policy can prevent direct changes at subscription scope.
- If you need a different setting, you usually adjust the policy assignment or scope, not just the subscription blade.

Practical troubleshooting flow:

1. Check whether the plan is inherited.
2. Review management group policy assignments.
3. Confirm whether a deny effect or modify effect is applied.
4. Change the policy at the correct scope if you are authorized.
5. Recheck the subscription state.

**Troubleshooting common blockers**

If a plan will not enable, look for these causes:

- The subscription does not support the plan.
- The plan is already inherited from a parent scope.
- Azure Policy blocks the change.
- The resource provider or service onboarding is incomplete.
- The tenant or region does not expose the plan in the current environment.

Useful checks:

```bash
az account show
az security pricing list --subscription <subscription-id> -o jsonc
az policy assignment list --scope /subscriptions/<subscription-id> -o table
```

If you suspect policy control, inspect assignments at the management group level as well.

**End-to-end exam scenario**

Scenario: You need to enable the server plan on a subscription, but the portal shows the setting as inherited.

1. Open **Microsoft Defender for Cloud**.
2. Go to **Environment settings** and select the subscription.
3. Confirm the plan state is **Inherited**.
4. Check the management group for policy assignments that configure Defender for Cloud plans.
5. If you have permission, update the policy assignment at the management group scope.
6. Return to the subscription and verify the state changes to **Enabled** or the expected paid tier.
7. Confirm the plan appears in `az security pricing list`.

Expected result:

- The subscription reflects the correct plan state.
- Defender for Cloud begins evaluating supported resources.
- Recommendations and alerts appear after processing.

**Practical configuration notes**

When you automate plan enablement, keep these habits:

- Use the portal first to confirm the exact plan name in your tenant.
- Test CLI syntax with `-h` or `--help` before scripting.
- Capture the subscription ID explicitly.
- Validate the result after the change.
- Document whether the setting is direct or inherited.

Example automation pattern:

```bash
SUBSCRIPTION_ID="<subscription-id>"
PLAN_NAME="VirtualMachines"
TIER="Standard"

az security pricing create --name "$PLAN_NAME" --tier "$TIER" --subscription "$SUBSCRIPTION_ID"
az security pricing list --subscription "$SUBSCRIPTION_ID" -o table
```

If your environment requires `update` instead of `create`, substitute the supported command after verifying the CLI version.

**Key exam takeaways**

- Defender for Cloud plans are enabled at subscription scope for specific workload areas.
- The portal path is usually **Microsoft Defender for Cloud > Environment settings > subscription > Defender plans**.
- Common states are **Enabled**, **Inherited**, and **Not applicable**.
- Free and paid tiers are not interchangeable; paid tiers unlock advanced protections where supported.
- Subscription type, policy, and inheritance can all affect whether you can enable a plan.
- Always verify the final state in both the portal and CLI when possible.
