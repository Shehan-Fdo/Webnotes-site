## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Secure Score overview**

Microsoft Defender for Cloud Secure Score is a posture metric that helps you measure how well your Azure environment follows security best practices. It is not a compliance score and it is not a direct measure of breach risk. Instead, it shows how many security recommendations you have addressed compared with the recommendations that apply to your environment.

Secure Score is most commonly reviewed at:
- **Subscription scope**
- **Management group scope**

At these scopes, Defender for Cloud aggregates applicable recommendations and weights them by their potential security impact. A recommendation that affects many resources or reduces a high-risk exposure usually contributes more to the score than a low-impact recommendation.

A useful exam distinction is this:
- **Secure Score** tells you how much of the recommended security baseline you have implemented.
- **Recommendations** tell you what to fix, why it matters, and how to remediate it.

**How Secure Score is calculated**

Secure Score is based on applicable recommendations, but not every recommendation contributes equally. The score reflects the ratio of achieved security controls to total applicable controls, with weighting applied by recommendation impact.

Key points:
- Only recommendations that apply to your environment affect the score.
- Some recommendations may not contribute if they are not applicable to the current resource set.
- Exemptions and dismissals can change how recommendations appear in reporting, but they do not mean the underlying risk disappears.
- A high score does not guarantee that every critical workload is safe; it means you have addressed many of the recommended controls.

> <mark>Exam tip:</mark> Do not confuse Secure Score with Azure Policy compliance. Policy compliance measures whether resources match policy definitions; Secure Score measures security posture improvement based on Defender for Cloud recommendations.

**Portal navigation you should know**

In the Azure portal, you typically review Secure Score and recommendations through Microsoft Defender for Cloud.

Common navigation path:
1. Open the Azure portal.
2. Search for and select **Microsoft Defender for Cloud**.
3. Open **Secure score** to view the overall score and control breakdown.
4. Open **Recommendations** to review individual findings.
5. Drill into a recommendation to see affected resources, remediation steps, and available actions.

Useful UI elements:
- **Overview** or dashboard tiles for posture summary
- **Secure score** blade for score trends and control-level detail
- **Recommendations** blade for triage and remediation
- **Affected resources** list for resource-level investigation
- **Remediate** or **Fix** actions when Defender for Cloud can guide you through a supported change

**Secure Score vs recommendation drill-down**

Use the score view when you need a management summary. Use recommendation details when you need to act.

| View | Best for | What you learn |
|---|---|---|
| Secure Score | Executive or platform overview | Overall posture, trend, and top controls to improve |
| Control drill-down | Prioritization | Which security control contributes most to score |
| Recommendation details | Remediation | Affected resources, risk explanation, and fix guidance |
| Resource view | Triage | Which specific VM, storage account, SQL server, or other asset is noncompliant |

**Recommendation lifecycle and states**

Recommendations move through states as you investigate and remediate them. The exact labels can vary by recommendation type and portal experience, but the exam-relevant concepts are consistent.

Common states and meanings:
- **Unhealthy**: The resource does not meet the recommendation.
- **Healthy**: The resource meets the recommendation.
- **Not applicable**: The recommendation does not apply to the resource or scope.
- **Resolved**: The issue has been fixed and the resource now meets the requirement.
- **Dismissed** or **exempted**: The finding is intentionally suppressed or excluded from reporting for a valid reason.

Important implications:
- A resource can be unhealthy even if the recommendation seems low priority.
- A recommendation can be healthy at one scope and unhealthy at another if the scope changes.
- Exemptions should be used carefully and documented, because they can hide real exposure if overused.

**Severity, priority, and score impact**

Recommendation severity and score impact are related but not identical.

- **Severity** describes how serious the issue is from a security perspective.
- **Score impact** describes how much fixing the issue improves Secure Score.
- **Priority** depends on business context, exposure, and exploitability.

For example:
- A recommendation with moderate severity may have a large score impact if it affects many resources.
- A high-severity issue may have a smaller score impact if it applies to only one resource.
- A recommendation that protects internet-facing assets usually deserves faster action than one affecting an internal-only test system.

When you triage, consider:
- Internet exposure
- Data sensitivity
- Privilege level of the affected resource
- Blast radius if compromised
- Whether the recommendation is a baseline hardening item or a critical vulnerability

**Recommendation categories and examples**

Defender for Cloud recommendations often map to common security domains:
- Identity and access
- Network exposure
- Endpoint protection
- Vulnerability management
- Data protection
- Logging and monitoring
- Configuration hardening

Examples of what you may see:
- Enable just-in-time access for virtual machines
- Restrict network access to a storage account
- Install endpoint protection on virtual machines
- Enable vulnerability assessment on SQL resources
- Turn on diagnostic logs for a service
- Use secure transfer or encryption settings

These recommendations are usually actionable and tied to a specific resource type. The portal often shows:
- Affected resources
- Remediation steps
- Policy or initiative association
- Expected score improvement

**Azure Policy and initiatives**

Defender for Cloud recommendations are often backed by Azure Policy. This matters because policy lets you enforce or audit configuration at scale.

Use policy when you need to:
- Detect noncompliant resources across subscriptions
- Enforce a configuration automatically
- Deploy settings to many resources consistently
- Track compliance over time

Use Defender for Cloud when you need to:
- See security posture in one place
- Prioritize recommendations by score impact
- Investigate affected resources
- Drive remediation workflows

A common pattern is:
- Defender for Cloud identifies a recommendation.
- Azure Policy evaluates compliance.
- An initiative groups related policies for a security baseline.
- You remediate manually or deploy a policy-based fix.

Example initiative use case:
- You assign a security baseline initiative to a subscription.
- Policy audits whether diagnostic settings, encryption, and secure network controls are enabled.
- Defender for Cloud surfaces the resulting recommendations and score changes.

**Practical remediation at scale**

For many recommendations, you should prefer policy-driven remediation over one-off manual fixes.

Typical workflow:
1. Identify the recommendation in Defender for Cloud.
2. Open the related policy assignment or initiative.
3. Review the noncompliant resources.
4. Choose whether to audit, deny, or deploy a fix.
5. Remediate at scale using policy remediation tasks or automation.

Example Azure CLI pattern for policy remediation:
```bash
# List policy assignments in a subscription
az policy assignment list --subscription <subscription-id> -o table

# Show noncompliant resources for a policy assignment
az policy state list --subscription <subscription-id> --query "[?complianceState=='NonCompliant']" -o table

# Start a remediation task for a policy assignment
az policy remediation create \
  --name remediate-security-baseline \
  --policy-assignment <policy-assignment-id> \
  --resource-discovery-mode ReEvaluateCompliance
```

Example PowerShell pattern:
```powershell
# List policy assignments
Get-AzPolicyAssignment -Scope "/subscriptions/<subscription-id>"

# View policy states
Get-AzPolicyState -SubscriptionId "<subscription-id>" |
  Where-Object { $_.ComplianceState -eq "NonCompliant" }

# Start remediation
Start-AzPolicyRemediation `
  -Name "remediate-security-baseline" `
  -PolicyAssignmentId "<policy-assignment-id>"
```

> <mark>Important:</mark> Not every Defender for Cloud recommendation has an automatic fix. Some require manual configuration, application changes, or operational approval.

**CLI and API usage for reporting**

For exam purposes, know that you can retrieve posture and recommendation data through Azure CLI, PowerShell, REST APIs, and Resource Graph patterns. Command names and module support can vary by version, so focus on the workflow rather than memorizing unstable command groups.

Useful approaches:
- Use Azure CLI to inspect security assessments and policy compliance.
- Use PowerShell to query Defender for Cloud and policy state.
- Use REST APIs when you need automation or custom reporting.
- Use Resource Graph for cross-subscription inventory and governance views.

Example Azure CLI pattern for security assessments:
```bash
# Query security assessments at subscription scope
az rest \
  --method get \
  --url "https://management.azure.com/subscriptions/<subscription-id>/providers/Microsoft.Security/assessments?api-version=2020-01-01"
```

Example REST pattern for secure score:
```bash
az rest \
  --method get \
  --url "https://management.azure.com/subscriptions/<subscription-id>/providers/Microsoft.Security/secureScores?api-version=2020-01-01"
```

**Reporting, exports, and governance**

Secure Score and recommendation data are useful beyond the portal. You can export or consume the data for:
- Security dashboards
- Governance reporting
- Executive summaries
- Workbooks and operational dashboards
- Automation pipelines
- Ticketing and remediation tracking

Common reporting patterns:
- Export recommendation data to CSV for analysis.
- Use workbooks to visualize trends over time.
- Send findings to a SIEM or log analytics workspace for correlation.
- Build governance dashboards that track score by subscription or management group.

Example workbook use case:
- Track Secure Score trend by month.
- Show top five recommendations by score impact.
- Break down unhealthy resources by subscription.
- Highlight overdue remediation items.

**How to read recommendation details**

When you open a recommendation, focus on the fields that help you decide what to do next.

Look for:
- **Affected resources**: Which resources need attention
- **Remediation steps**: What to change
- **Impact**: Why the recommendation matters
- **Category**: Which security domain it belongs to
- **Policy link**: Whether Azure Policy can enforce it
- **Score impact**: How much improvement you gain by fixing it

A good triage sequence is:
1. Check whether the resource is internet-facing or high value.
2. Review the recommendation impact and affected count.
3. Determine whether the fix is manual, policy-driven, or automated.
4. Validate the change after remediation.
5. Confirm the recommendation state updates to healthy or resolved.

**Exam scenarios to practice**

Scenario 1: You need the fastest way to understand overall security posture.
- Use **Secure Score**.
- Review the top controls with the largest score impact.
- Prioritize broad, high-value improvements.

Scenario 2: You need to fix a specific VM exposure.
- Use **Recommendations**.
- Open the VM-related recommendation.
- Review affected resources and remediation guidance.

Scenario 3: You need to enforce a baseline across many subscriptions.
- Use **Azure Policy** and an **initiative**.
- Remediate noncompliant resources at scale.
- Track the resulting score improvement in Defender for Cloud.

Scenario 4: You need evidence for governance reporting.
- Export recommendation and score data.
- Use a workbook or dashboard.
- Show trend, status, and remediation progress.

**Common exam traps**

- Secure Score is not the same as compliance score.
- A recommendation can be important even if it does not move the score much.
- A high score does not mean all critical assets are protected.
- Dismissed or exempted findings should not be treated as fixed.
- Policy compliance and Defender for Cloud recommendations are related, but they are not identical views.
- Not every recommendation has a one-click fix.

**Quick reference**

- **Secure Score**: posture metric for Defender for Cloud
- **Recommendation**: actionable security finding
- **Assessment**: evaluation of a resource against a control or recommendation
- **Policy**: enforcement or audit mechanism
- **Initiative**: grouped policy set for a baseline
- **Exemption**: intentional exclusion from evaluation or reporting

**Study takeaway**

Use Secure Score to prioritize where to improve, use recommendations to decide what to fix, and use Azure Policy to enforce changes at scale. For the exam, be ready to explain how score, recommendation state, policy compliance, and remediation workflows fit together in a real Azure security operation.
