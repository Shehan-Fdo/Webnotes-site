## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Azure Policy overview**

Azure Policy is the governance control that evaluates Azure resources against rules you define. You use it to enforce standards, assess drift, and support continuous compliance across subscriptions, management groups, and resource groups.

It is most useful when you need to answer questions such as:

- Are resources deployed with required security settings?
- Can noncompliant resources be blocked at creation time?
- Which resources need remediation?
- How do you prove compliance to auditors?

Azure Policy is not the same as role-based access control (RBAC). RBAC controls *who can do what*; Azure Policy controls *what is allowed* and *what must exist*.

**Core policy building blocks**

A policy solution usually includes three layers:

- **Policy definition**: The rule logic.
- **Policy assignment**: Where the rule applies and with what parameters.
- **Policy initiative**: A collection of related policy definitions, also called a policy set.

A definition can be built-in or custom. Built-in definitions cover common controls such as secure transfer, allowed locations, tagging, and diagnostic settings. Custom definitions let you encode organization-specific requirements.

A policy rule uses JSON and typically includes:

- `if`: The condition to evaluate.
- `then`: The effect to apply when the condition matches.

Example structure:

```json
{
  "mode": "All",
  "policyRule": {
    "if": {
      "field": "type",
      "equals": "Microsoft.Storage/storageAccounts"
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

**Policy rule logic and operators**

Azure Policy supports rich condition logic. You should recognize the common operators and how they combine.

- **Logical operators**
  - `allOf`: All conditions must be true.
  - `anyOf`: At least one condition must be true.
  - `not`: Negates a condition.

- **Comparison operators**
  - `equals` / `notEquals`
  - `in` / `notIn`
  - `like` / `notLike`
  - `contains`
  - `greater` / `greaterOrEquals`
  - `less` / `lessOrEquals`

- **Existence and count**
  - `exists`: Checks whether a field is present.
  - `count`: Evaluates the number of matching child elements or array items.

Example with multiple conditions:

```json
{
  "if": {
    "allOf": [
      {
        "field": "type",
        "equals": "Microsoft.Storage/storageAccounts"
      },
      {
        "field": "Microsoft.Storage/storageAccounts/supportsHttpsTrafficOnly",
        "equals": false
      }
    ]
  },
  "then": {
    "effect": "deny"
  }
}
```

> Use the exact alias supported by the resource provider and policy alias catalog. Alias names vary by resource type and property.

**Policy modes and evaluation scope**

Policy definitions use a `mode` that affects which resource properties are evaluated.

- **`All`**
  - Evaluates all resource types and properties supported by the policy engine.
  - Use this when you need to inspect properties beyond tags and location.
  - Common for many custom policies.

- **`Indexed`**
  - Evaluates resource types that support tags and location.
  - Use this when the policy focuses on tag or location-based checks.
  - Do not assume `Indexed` is always correct for every policy.

In exam scenarios, choose the mode based on the resource and the alias being evaluated. If a policy checks a property outside the tag/location model, `All` is often required.

**Common effects and what they do**

Azure Policy effects determine how the platform responds when a rule matches.

- **`deny`**
  - Blocks the request.
  - Best for preventing insecure or noncompliant deployments.

- **`audit`**
  - Allows the request but marks the resource noncompliant.
  - Best for discovery and phased rollout.

- **`append`**
  - Adds a field to a create or update request if the field is missing.
  - It does not modify existing resources after deployment.
  - Use it for request-time defaults, not for retroactive remediation.

- **`modify`**
  - Changes request properties during create or update.
  - Often used to enforce tags or normalize settings.
  - Requires a managed identity and permissions to write the target resource.

- **`deployIfNotExists`**
  - Checks for a related resource or configuration and deploys a template if the required state is missing.
  - Common for diagnostic settings, monitoring agents, or child resources where supported.
  - Remediation depends on the resource provider and the policy definition.

- **`auditIfNotExists`**
  - Audits when a related resource or configuration is missing.
  - Useful when you want visibility before enforcing remediation.

- **`disabled`**
  - Turns off the policy effect without deleting the assignment.
  - Useful for staged rollouts or temporary exceptions.

**Assignment behavior and parameters**

A policy assignment is where you apply a definition or initiative to a scope such as a management group, subscription, resource group, or resource.

Assignment settings matter in exam questions:

- **Scope**
  - Determines where the policy applies.
  - Child scopes inherit unless excluded.

- **Not scopes**
  - Exclude specific subscriptions, resource groups, or resources from the assignment.
  - Useful for exceptions without removing the main assignment.

- **Enforcement mode**
  - Controls whether the policy is actively enforced.
  - A non-enforcing assignment can still evaluate and report compliance.
  - This is useful for testing or phased adoption.

- **Parameters**
  - Assignment parameters provide the actual values used by the policy.
  - Definition parameters declare what values the policy expects.
  - Do not confuse the two: the definition defines the input contract; the assignment supplies the values.

Example assignment parameter pattern:

```json
{
  "locationList": {
    "value": [
      "eastus",
      "westus2"
    ]
  }
}
```

If a policy definition is parameterized, you must pass the required values at assignment time. In PowerShell or CLI, this is a common source of exam traps.

**Initiatives and aggregated compliance**

A policy initiative groups multiple policy definitions into one governance package. This is useful when you want to enforce a benchmark or a control family rather than a single rule.

Typical initiative uses:

- Security baseline enforcement
- Regulatory compliance mapping
- Tagging and cost governance bundles
- Platform landing zone standards

Initiative behavior to remember:

- Child policy parameters are mapped through the initiative.
- The initiative assignment can expose a simplified parameter surface.
- Compliance is aggregated across the included policies.
- A single noncompliant child policy can make the initiative appear noncompliant at the assignment scope.

In practice, initiatives are often easier to manage than many separate assignments because they centralize governance and reporting.

**Compliance reporting surfaces**

Azure Policy compliance is visible through several surfaces, and you should know how they differ.

- **Azure Portal compliance dashboard**
  - Shows overall compliance by assignment, initiative, and resource.
  - Good for visual review and drill-down.
  - Useful for auditors and operations teams.

- **Policy states**
  - The underlying compliance records for evaluated resources.
  - Show whether a resource is compliant, noncompliant, exempt, conflict, or unknown depending on the surface and timing.

- **Policy insights**
  - Reporting and analytics layer for policy data.
  - Helps you query trends, summarize compliance, and export results.

- **Azure Resource Graph**
  - Useful for querying policy-related resource data at scale.
  - Good for cross-subscription reporting and automation.

- **Azure CLI and PowerShell**
  - Useful for automation, scripting, and CI/CD validation.

Example Azure CLI queries:

```bash
az policy assignment list --scope /subscriptions/<subscription-id>
az policy state list --filter "PolicyAssignmentName eq 'enforce-secure-transfer'"
az policy state summarize --subscription <subscription-id>
```

Example Resource Graph query:

```kusto
PolicyResources
| where type =~ 'microsoft.policyinsights/policystates'
| summarize count() by complianceState
```

> Compliance state names can vary by reporting surface and timing. In practice, focus on the states shown in the portal or query output for the specific API you are using.

**Triggering evaluation and scans**

Azure Policy evaluates on resource creation and update, but compliance reporting is not always immediate. You may need to trigger a scan after assignment changes or when you want faster reporting.

Common approaches:

```bash
az policy state trigger-scan --subscription <subscription-id>
```

For a resource group scope, use the resource-group form supported by your CLI version:

```bash
az policy state trigger-scan --resource-group <resource-group-name>
```

Use scans when:

- You assign a new policy and want updated compliance data.
- You change a policy definition or initiative.
- You need to validate remediation progress.

Remember that scans are asynchronous. The command starts the process; it does not instantly update every compliance record.

**Exemptions and governance exceptions**

Exemptions let you formally document why a resource, resource group, or subscription is excluded from a policy assignment.

Important exemption concepts:

- **Exemption category**
  - Commonly used categories include waiver-style and mitigation-style exceptions.
  - Use the category that matches your governance intent.

- **Expiration**
  - Exemptions can be time-bound.
  - This is important for temporary business exceptions.

- **Justification**
  - Document why the exemption exists.
  - This supports auditability and review.

- **Scope**
  - Exemptions apply to a specific scope and assignment relationship.

Exemptions are better than removing assignments because they preserve governance intent and create an audit trail.

**Remediation and managed identities**

Some policy effects can only correct resources after deployment through remediation tasks. This is especially relevant for `modify` and `deployIfNotExists`.

Key remediation points:

- The policy assignment often needs a **managed identity**.
- The identity is used to perform deployment or update actions.
- The identity must have the required permissions on the target scope.
- For deployment-based remediation, the identity commonly needs rights such as Contributor or a narrower custom role with the required write permissions.

Typical remediation flow:

1. Assign the policy with a managed identity.
2. Ensure the identity has the required role assignments.
3. Create or start a remediation task.
4. Monitor remediation progress and resulting compliance.

Example Azure CLI assignment pattern:

```bash
az policy assignment create \
  --name enforce-tags \
  --scope /subscriptions/<subscription-id> \
  --policy <policy-definition-id> \
  --params '{"tagName":{"value":"CostCenter"},"tagValue":{"value":"1234"}}' \
  --identity-system-assigned
```

Example PowerShell pattern for a parameterized policy:

```powershell
$assignmentParams = @{
  tagName  = @{ value = "CostCenter" }
  tagValue = @{ value = "1234" }
}

New-AzPolicyAssignment `
  -Name "enforce-tags" `
  -Scope "/subscriptions/<subscription-id>" `
  -PolicyDefinition $definition `
  -PolicyParameterObject $assignmentParams `
  -AssignIdentity
```

**Practical custom policy scenario**

A common AZ-500 scenario is enforcing secure configuration on storage accounts.

Example goal:

- Deny storage accounts that do not require secure transfer.
- Audit or remediate resources that are missing required diagnostic settings.
- Report compliance at the subscription or management group level.

Example custom policy definition:

```json
{
  "mode": "All",
  "parameters": {},
  "policyRule": {
    "if": {
      "allOf": [
        {
          "field": "type",
          "equals": "Microsoft.Storage/storageAccounts"
        },
        {
          "field": "Microsoft.Storage/storageAccounts/supportsHttpsTrafficOnly",
          "equals": false
        }
      ]
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

A practical rollout pattern is:

1. Start with `audit` to measure impact.
2. Review compliance data in the portal and query tools.
3. Add exemptions for valid business cases.
4. Switch to `deny` or `modify` when the organization is ready.

**Policy benchmark and initiative example**

A governance benchmark often combines multiple controls into one initiative.

Example initiative contents:

- Require secure transfer on storage accounts.
- Enforce HTTPS-only on App Service.
- Audit missing diagnostic settings.
- Restrict allowed regions.
- Require tags such as `Owner` and `CostCenter`.

This approach helps you manage a landing zone or security baseline with one assignment instead of many.

Example initiative assignment concept:

```json
{
  "properties": {
    "displayName": "Security baseline",
    "policyDefinitionId": "/providers/Microsoft.Authorization/policySetDefinitions/security-baseline",
    "parameters": {
      "allowedLocations": {
        "value": [
          "eastus",
          "westus2"
        ]
      }
    }
  }
}
```

**CLI and PowerShell operations you should recognize**

Azure CLI examples:

```bash
az policy definition list --query "[].{name:name, displayName:displayName}"
az policy assignment list --scope /subscriptions/<subscription-id>
az policy exemption list --scope /subscriptions/<subscription-id>
az policy state summarize --resource-group <resource-group-name>
```

PowerShell examples:

```powershell
Get-AzPolicyDefinition
Get-AzPolicyAssignment -Scope "/subscriptions/<subscription-id>"
Get-AzPolicyExemption -Scope "/subscriptions/<subscription-id>"
Get-AzPolicyStateSummary -SubscriptionId "<subscription-id>"
```

Use these commands to inspect what is assigned, what is exempted, and what is currently noncompliant.

**How to interpret compliance in practice**

When you review compliance, think in terms of lifecycle:

- **Before deployment**
  - A deny policy blocks the request.
  - An audit policy allows the request but records noncompliance.

- **After deployment**
  - Compliance scans identify drift or missing settings.
  - Remediation tasks can correct supported resources.

- **During exception handling**
  - Exemptions explain why a resource is allowed to remain noncompliant.
  - Expiration dates help prevent permanent drift.

A resource can be technically noncompliant but operationally accepted if it is exempted. That distinction matters in governance reporting.

**Common exam distinctions**

- **Azure Policy vs RBAC**
  - Policy enforces rules on resources.
  - RBAC controls access to management operations.

- **Policy vs locks**
  - Policy can sometimes deny delete operations if authored for that purpose.
  - Resource locks are the primary tool for preventing accidental deletion.

- **Audit vs deny**
  - Audit reports.
  - Deny blocks.

- **Append vs modify**
  - Append adds missing request fields.
  - Modify changes request fields and is more flexible.

- **Definition vs assignment**
  - Definition contains the rule.
  - Assignment supplies scope and values.

- **Policy vs initiative**
  - Policy is one rule.
  - Initiative is a bundle of rules.

**End-to-end workflow**

A realistic administration workflow looks like this:

1. Define the control objective, such as requiring secure transfer or approved regions.
2. Create or select a policy definition.
3. Test with `audit` at a limited scope.
4. Assign the policy or initiative with the correct parameters.
5. Add exclusions with `notScopes` or exemptions where justified.
6. Trigger a compliance scan.
7. Review results in the portal, policy states, or query tools.
8. Create remediation tasks where supported.
9. Tighten enforcement to `deny` or `modify` after validation.

This workflow is the safest way to introduce governance without breaking production workloads.

**Quick reference**

- **Use `audit`** when you need visibility first.
- **Use `deny`** when you must block insecure deployments.
- **Use `modify`** when you want to correct request properties.
- **Use `deployIfNotExists`** when a missing related resource can be deployed automatically.
- **Use exemptions** for documented exceptions, not as a replacement for governance.
- **Use initiatives** to manage multiple related controls as one compliance package.
