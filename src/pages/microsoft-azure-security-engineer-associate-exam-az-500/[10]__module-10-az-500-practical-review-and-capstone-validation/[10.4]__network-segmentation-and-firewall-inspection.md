## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Network segmentation overview**

Network segmentation limits lateral movement by separating workloads into trust zones and controlling traffic between them. In Azure, you usually combine subnet design, network security groups (NSGs), application security groups (ASGs), Azure Firewall, route tables, and private endpoints to enforce layered controls.

A good design starts with the principle of least privilege:
- Allow only required flows.
- Prefer explicit deny boundaries between tiers.
- Inspect north-south traffic at centralized choke points.
- Use private access paths for platform services when possible.

Common segmentation patterns include:
- **Hub-and-spoke**: shared security services in a hub VNet, workloads in spoke VNets.
- **Tiered application**: web, app, and data subnets separated by policy.
- **Shared services**: DNS, jump hosts, and inspection tools isolated from workloads.

**Azure virtual network and subnet design**

A virtual network (VNet) is the primary isolation boundary in Azure. Subnets let you apply different controls to different workload tiers. You should design address space carefully because overlapping ranges complicate peering, VPN, and private endpoint routing.

Key design points:
- Keep address spaces non-overlapping across VNets that may peer or connect through VPN/ExpressRoute.
- Reserve room for growth, especially in hub VNets that host shared services.
- Place inspection and routing components in dedicated subnets.
- Use separate subnets for workloads with different trust levels.

A typical hub-and-spoke layout looks like this:
- Hub VNet:
  - Azure Firewall
  - VPN gateway or ExpressRoute gateway
  - DNS forwarders or resolver
  - Bastion or jump access
- Spoke VNets:
  - Application workloads
  - Data services
  - Private endpoints

**Network Security Groups**

An NSG is a stateful packet filter that allows or denies traffic based on 5-tuple and Azure metadata such as source/destination, port, protocol, and direction. NSGs can be associated with a subnet or a network interface card (NIC).

Important behavior:
- **Stateful** means return traffic for an allowed flow is automatically permitted.
- Rules are evaluated by priority, from lowest number to highest precedence.
- Default rules exist and cannot be removed.
- Subnet NSGs and NIC NSGs both apply; the effective result is the most restrictive outcome.

Default rule concepts you should remember:
- Allow VNet-to-VNet traffic inside the same VNet by default.
- Allow Azure load balancer health probes by default.
- Deny all inbound traffic after the allow rules.
- Allow outbound internet traffic by default unless overridden.

Practical evaluation guidance:
- Inbound traffic is checked against the destination NIC and subnet.
- Outbound traffic is checked against the source NIC and subnet.
- If either NSG denies the flow, the traffic is blocked.
- When troubleshooting, inspect the effective security rules on the NIC and subnet.

Example NSG rule creation:
```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-web \
  --name Allow-HTTPS-From-Internet \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes Internet \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 443
```

Example deny rule for lateral movement:
```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-app \
  --name Deny-App-To-Data-All \
  --priority 200 \
  --direction Outbound \
  --access Deny \
  --protocol '*' \
  --source-address-prefixes '*' \
  --source-port-ranges '*' \
  --destination-address-prefixes 10.20.2.0/24 \
  --destination-port-ranges '*'
```

**Application Security Groups**

ASGs let you group NICs logically so NSG rules can target application roles instead of IP addresses. This is useful when IPs change frequently or when you want rules to follow workload identity rather than static addressing.

ASG limitations and scope:
- ASGs are limited to NICs in the same virtual network.
- They do not cross VNet boundaries.
- They are used in NSG source or destination fields.
- They simplify rule management for multi-tier applications.

Example use case:
- `asg-web` contains web server NICs.
- `asg-app` contains application server NICs.
- `asg-data` contains database server NICs.

Example NSG rule using ASGs:
```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-app \
  --name Allow-Web-To-App-443 \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-asgs asg-web \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 443
```

Use ASGs when:
- You manage many similar workloads.
- You want cleaner rule sets than IP-based rules.
- You need role-based segmentation inside one VNet.

Avoid ASGs when:
- Traffic crosses VNets.
- You need to reference non-NIC resources.
- You need broad policy across many regions or subscriptions.

**Service tags and common segmentation patterns**

Service tags simplify NSG and Azure Firewall rule authoring by representing Azure service IP ranges. They reduce maintenance because Microsoft updates the underlying prefixes.

Common service tags include:
- `VirtualNetwork`
- `AzureLoadBalancer`
- `Internet`
- `AzureCloud`
- `Storage`
- `Sql`
- `KeyVault`
- `AzureMonitor`
- `AzureActiveDirectory`

Use service tags when:
- You need to allow access to a Microsoft service without managing IP ranges.
- You want rules that survive backend IP changes.
- You are designing outbound allow lists.

Example NSG rule allowing Azure Monitor:
```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-app \
  --name Allow-AzureMonitor-Out \
  --priority 120 \
  --direction Outbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes '*' \
  --source-port-ranges '*' \
  --destination-address-prefixes AzureMonitor \
  --destination-port-ranges 443
```

**Rule processing and effective security**

NSG rule processing is a frequent exam topic because the final result depends on multiple layers.

Remember this order of thinking:
1. Check whether the traffic is inbound or outbound.
2. Evaluate subnet NSG rules.
3. Evaluate NIC NSG rules.
4. Apply the first matching rule by priority in each NSG.
5. If any applicable NSG denies the flow, the flow is blocked.

Practical implications:
- A subnet-level allow does not override a NIC-level deny.
- A NIC-level allow does not override a subnet-level deny.
- Default rules can unexpectedly permit east-west traffic if you do not add explicit denies.
- You should design explicit deny boundaries between tiers when segmentation matters.

Troubleshooting commands:
```bash
az network nic list-effective-nsg --resource-group rg-app --name nic-web01
az network nic show-effective-route-table --resource-group rg-app --name nic-web01
```

**Azure Firewall architecture**

Azure Firewall is a managed, stateful firewall service used for centralized inspection and policy enforcement. It is commonly deployed in a hub VNet and forced to inspect traffic from spoke VNets through user-defined routes (UDRs).

Deployment requirements:
- Place the firewall in a dedicated subnet named `AzureFirewallSubnet`.
- If you use forced tunneling with a management interface, also use `AzureFirewallManagementSubnet`.
- Size the subnet appropriately for scale and future growth.
- Use hub-and-spoke routing so spokes send traffic to the firewall as the next hop.

When to use Azure Firewall:
- You need centralized outbound control.
- You need DNAT, network rules, and application rules in one service.
- You need logging and policy-based governance across many VNets.
- You need advanced features such as TLS inspection or IDPS in Premium.

Typical hub-and-spoke flow:
- Spoke subnet default route points to Azure Firewall private IP.
- Firewall inspects traffic.
- Firewall forwards allowed traffic to the internet, another spoke, or on-premises.
- Return traffic follows the stateful session.

Example route table to force outbound traffic through the firewall:
```bash
az network route-table route create \
  --resource-group rg-hub \
  --route-table-name rt-spoke \
  --name DefaultToFirewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

**Azure Firewall policy and rule collection groups**

Modern Azure Firewall deployments use **Firewall Policy** rather than classic per-firewall rule configuration. Policy centralizes rules, supports inheritance, and makes it easier to manage multiple firewalls consistently.

Policy concepts:
- **Firewall policy**: the top-level object that stores rule collections and settings.
- **Rule collection group**: a container for related rule collections.
- **Rule collection**: groups rules by type and priority.
- **Inheritance**: child policies can inherit from parent policies, which helps standardize baseline controls.

Rule types:
- **DNAT**: translates public IP traffic to private targets.
- **Network rules**: control TCP, UDP, and ICMP flows by IP, port, and protocol.
- **Application rules**: control HTTP, HTTPS, and FQDN-based web traffic.

Important distinction:
- Application rules are for web traffic and FQDN-based filtering.
- SQL traffic is handled by network rules, not application rules.

Example policy creation:
```bash
az network firewall policy create \
  --resource-group rg-hub \
  --name fwpol-hub \
  --location eastus
```

Example rule collection group creation:
```bash
az network firewall policy rule-collection-group create \
  --resource-group rg-hub \
  --policy-name fwpol-hub \
  --name rcg-base \
  --priority 100
```

Example network rule collection:
```bash
az network firewall policy rule-collection-group collection add-filter-collection \
  --resource-group rg-hub \
  --policy-name fwpol-hub \
  --rule-collection-group-name rcg-base \
  --name Allow-DNS \
  --collection-priority 100 \
  --action Allow \
  --rule-name Allow-DNS-UDP \
  --rule-type NetworkRule \
  --protocols UDP \
  --source-addresses 10.1.0.0/16 \
  --destination-addresses 168.63.129.16 \
  --destination-ports 53
```

> Azure CLI syntax changes over time. Validate command names and parameters against the installed CLI version before using them in production.

**Azure Firewall rule types in practice**

Use the right rule type for the traffic you need to control.

- **DNAT rules**
  - Publish a private service through a firewall public IP.
  - Common for inbound web apps, jump hosts, or partner access.
  - Pair with NSGs so only the firewall can reach the backend.

- **Network rules**
  - Control non-HTTP traffic such as DNS, RDP, SSH, and SQL.
  - Use for private IP-to-IP inspection and outbound allow lists.
  - Useful for on-premises connectivity and east-west control.

- **Application rules**
  - Control outbound web access by FQDN.
  - Support HTTP and HTTPS.
  - Can use URL filtering in the context of application rules, but URL filtering is not a separate standalone firewall mode.

Example application rule:
```bash
az network firewall policy rule-collection-group collection add-filter-collection \
  --resource-group rg-hub \
  --policy-name fwpol-hub \
  --rule-collection-group-name rcg-base \
  --name Allow-Web-Access \
  --collection-priority 200 \
  --action Allow \
  --rule-name Allow-Microsoft-Docs \
  --rule-type ApplicationRule \
  --source-addresses 10.1.1.0/24 \
  --protocols Http=80 Https=443 \
  --target-fqdns learn.microsoft.com docs.microsoft.com
```

**Azure Firewall Standard vs Premium**

Azure Firewall Standard covers core centralized filtering. Azure Firewall Premium adds advanced threat protection and inspection capabilities.

Feature comparison:
| Capability | Standard | Premium |
|---|---:|---:|
| Network rules | Yes | Yes |
| Application rules | Yes | Yes |
| DNAT | Yes | Yes |
| Threat intelligence | Yes | Yes |
| TLS inspection | No | Yes |
| IDPS | No | Yes |
| URL filtering in application rules | Limited/basic web control | Yes, with richer inspection context |
| Web categories | No | Yes |

Exam-relevant guidance:
- Choose **Standard** for basic segmentation and outbound control.
- Choose **Premium** when you need TLS inspection, IDPS, or web category filtering.
- Do not assume Premium is required for every web filtering scenario.

**Forced tunneling and SNAT behavior**

Forced tunneling sends internet-bound traffic to a next hop other than the internet, often an on-premises security stack or another firewall. In Azure Firewall designs, this is important when you want all egress traffic inspected centrally.

SNAT (Source Network Address Translation) behavior matters because Azure Firewall may translate source addresses for outbound traffic.

Key points:
- Azure Firewall performs SNAT for outbound internet traffic by default.
- If traffic is destined for private ranges, SNAT behavior depends on routing and whether the destination is considered private.
- Forced tunneling can change how return traffic flows and may require careful route design.
- Asymmetric routing can break stateful inspection if return traffic bypasses the firewall.

Troubleshooting signs of routing problems:
- Sessions establish but return traffic fails.
- One direction works while the reverse direction is blocked.
- Effective routes show a path that bypasses the firewall.
- Logs show denies even though NSG rules appear correct.

**DNS proxy and FQDN-based filtering**

Azure Firewall DNS proxy helps the firewall resolve domain names on behalf of clients so application rules can match FQDNs consistently. This is especially useful when clients use private DNS or when you want the firewall to control name resolution for outbound web access.

Why DNS proxy matters:
- Application rules depend on domain names.
- If clients resolve names differently than the firewall, rule matching can fail.
- DNS proxy improves consistency for FQDN-based allow lists.

Typical configuration pattern:
- Configure Azure Firewall as the DNS server for spoke workloads.
- Enable DNS proxy on the firewall policy or firewall.
- Forward unresolved queries to an upstream DNS server or Azure-provided DNS as appropriate.
- Ensure clients use the firewall or a resolver path that aligns with policy.

**Private endpoints and private DNS**

Private endpoints give a private IP address to a PaaS service inside your VNet. They reduce exposure by keeping traffic on the Microsoft backbone and avoiding public endpoints.

Important scope details:
- A private endpoint is reachable from the VNet where it exists.
- Peered VNets can reach it if routing and DNS are configured correctly.
- On-premises access requires connectivity into the VNet through VPN or ExpressRoute and correct DNS resolution.
- Private endpoints do not automatically solve name resolution.

Private DNS design:
- Use private DNS zones such as `privatelink.blob.core.windows.net` or service-specific equivalents.
- Link the zone to the VNets that need resolution.
- Configure on-premises DNS forwarding or conditional forwarders if on-prem clients must resolve the private name.
- Use split-horizon DNS when the same service name must resolve differently inside and outside Azure.

Example private DNS zone link:
```bash
az network private-dns zone create \
  --resource-group rg-net \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-net \
  --zone-name privatelink.blob.core.windows.net \
  --name link-spoke1 \
  --virtual-network /subscriptions/<subId>/resourceGroups/rg-net/providers/Microsoft.Network/virtualNetworks/vnet-spoke1 \
  --registration-enabled false
```

**Private endpoint troubleshooting**

When private endpoint access fails, check these layers:
- DNS resolves to the private IP, not the public endpoint.
- The private endpoint NIC exists and is approved.
- The client has network reachability to the VNet.
- NSGs and UDRs do not block the path.
- On-premises DNS forwards the private zone correctly.

Useful validation commands:
```bash
nslookup mystorageaccount.blob.core.windows.net
az network private-endpoint show --resource-group rg-net --name pe-storage01
az network private-dns zone list --resource-group rg-net
```

**NSG flow visibility and traffic analysis**

NSG flow logs are a legacy Network Watcher capability and are not the preferred long-term direction for new designs. For current monitoring strategies, prefer virtual network flow logs and Traffic Analytics where available in your environment.

Use flow visibility to answer:
- Which source and destination pairs are active?
- Which ports are most used?
- Which flows are denied?
- Is traffic reaching the expected subnet or NIC?

Troubleshooting workflow:
1. Check effective security rules.
2. Check effective routes.
3. Review flow logs or traffic analytics.
4. Compare observed traffic with intended segmentation.
5. Adjust NSG, UDR, or firewall policy as needed.

**Common service tag and rule design scenarios**

Service tags often appear in exam questions because they reduce operational overhead.

Typical scenarios:
- Allow Azure platform dependencies:
  - `AzureLoadBalancer`
  - `AzureMonitor`
  - `AzureActiveDirectory`
- Allow outbound internet access only for approved services:
  - `Storage`
  - `KeyVault`
  - `Sql`
- Restrict broad internet exposure:
  - Deny `Internet` except through Azure Firewall or approved proxies

Example outbound allow list pattern:
- Allow `AzureActiveDirectory` on 443 for authentication.
- Allow `KeyVault` on 443 for secret retrieval.
- Allow `Storage` on 443 for application data.
- Deny all other outbound internet traffic.

**Troubleshooting asymmetric routing and blocked flows**

Asymmetric routing happens when traffic leaves one path and returns on another. Stateful devices like Azure Firewall expect both directions of a session to follow a consistent path.

Symptoms:
- Intermittent connectivity.
- SYN packets leave but no SYN-ACK returns.
- Firewall logs show incomplete sessions.
- A peered VNet or on-prem route bypasses the inspection point.

How to diagnose:
- Inspect effective routes on the NIC.
- Verify UDRs on the subnet.
- Confirm return path symmetry through the firewall.
- Check whether peering, VPN, or ExpressRoute introduces a different next hop.
- Validate that NSGs allow the return flow on the expected interface.

**Practical design patterns**

For a three-tier application:
- Put web servers in a web subnet.
- Put app servers in an app subnet.
- Put databases in a data subnet.
- Use NSGs to allow only required tier-to-tier ports.
- Use Azure Firewall for outbound internet inspection and centralized logging.
- Use private endpoints for PaaS dependencies such as storage and Key Vault.

For a secure hub-and-spoke environment:
- Centralize firewall, DNS, and gateway services in the hub.
- Force spoke default routes to the firewall.
- Use ASGs inside each spoke for role-based segmentation.
- Use private DNS zones for private endpoint resolution.
- Log and review denied flows regularly.

**Exam focus summary**

You should be able to explain:
- How NSGs, ASGs, UDRs, and Azure Firewall work together.
- Why subnet and NIC NSGs both matter.
- When to use application rules versus network rules.
- Why Azure Firewall belongs in a hub-and-spoke design.
- How forced tunneling and SNAT affect outbound traffic.
- Why private endpoints require DNS planning.
- Why Premium features matter for TLS inspection and IDPS.
- How to troubleshoot with effective routes, effective security rules, and flow visibility.

**Quick reference**

- **NSG**: stateful packet filter at subnet or NIC scope.
- **ASG**: logical grouping of NICs within one VNet.
- **Azure Firewall**: centralized managed firewall with policy-based control.
- **Firewall Policy**: modern configuration model for Azure Firewall.
- **Private Endpoint**: private IP for PaaS access.
- **DNS Proxy**: helps Azure Firewall resolve FQDNs consistently.
- **Forced Tunneling**: routes internet-bound traffic through a security path.
- **SNAT**: source address translation used by the firewall for outbound flows.
