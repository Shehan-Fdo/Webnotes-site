## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Identity and access hardening overview**

Identity is the primary control plane in Azure security. Hardening focuses on reducing the chance of credential theft, limiting what identities can do, and making privileged access temporary, auditable, and conditional. In practice, you combine Microsoft Entra ID controls, Azure RBAC, Privileged Identity Management (PIM), Conditional Access, governance workflows, and workload identity protections.

A strong design usually follows these principles:

- **Verify explicitly** with strong authentication and risk-aware policies.
- **Use least privilege** with built-in roles first, custom roles when needed.
- **Assume breach** by limiting standing privilege and requiring step-up controls.
- **Continuously govern** guests, privileged users, and app identities.

**Core identity hardening goals**

Identity hardening is not just about sign-in security. It also includes authorization design, lifecycle governance, and reducing attack paths such as legacy authentication, over-permissioned service principals, and unmanaged guest access.

- **Authentication** answers who the user or workload is.
- **Authorization** answers what that identity can do after sign-in.
- **Governance** answers whether the identity should still have access.
- **Session control** answers how access behaves after authentication.

**Microsoft Entra ID authentication methods**

Microsoft Entra ID authentication methods policy controls which methods users can register and use. This is a key hardening area because legacy methods increase phishing and token theft risk.

Common modern methods include:

- **Microsoft Authenticator** with number matching and passwordless options.
- **FIDO2 security keys** for phishing-resistant authentication.
- **Windows Hello for Business** for device-bound sign-in.
- **Temporary Access Pass (TAP)** for secure bootstrap and recovery.
- **Passkeys** where supported in the tenant and client ecosystem.

Legacy methods to reduce or disable where possible:

- **SMS** and **voice call** MFA are weaker than phishing-resistant methods.
- **App passwords** are a legacy workaround for older protocols and should be eliminated by blocking legacy authentication.
- **Security questions** are not suitable for strong enterprise authentication.

A practical hardening approach is to require modern methods for all users, then use Conditional Access to enforce stronger methods for privileged roles and sensitive apps.

> Legacy authentication is a common exam trap. If a protocol cannot do modern MFA, block it rather than compensating with weaker exceptions.

**Blocking legacy authentication**

Legacy authentication includes protocols such as POP, IMAP, SMTP AUTH, and older Office clients that do not support modern authentication. These protocols are attractive to attackers because they often bypass MFA.

Typical controls:

- **Conditional Access** policy that blocks legacy authentication client apps.
- **Authentication methods policy** that removes weak registration options.
- **Per-user MFA** should not be used as the primary control in modern designs because Conditional Access is more flexible and auditable.

Example Conditional Access logic:

1. Target all users.
2. Exclude break-glass accounts.
3. Target legacy authentication client apps.
4. Grant access: block.

**Conditional Access fundamentals**

Conditional Access is the policy engine that evaluates signals before granting access. It is central to identity hardening because it lets you require stronger authentication only when risk, device state, location, or app sensitivity warrants it.

Common signals:

- **User or group**
- **Cloud app**
- **Device platform**
- **Device compliance**
- **Sign-in risk**
- **User risk**
- **Location**
- **Client app type**
- **Authentication strength**

Common grant controls:

- **Require multifactor authentication**
- **Require authentication strength**
- **Require compliant device**
- **Require hybrid Azure AD joined device**
- **Require approved client app**
- **Require app protection policy**
- **Block access**

Common session controls:

- **Sign-in frequency**
- **Persistent browser session**
- **App enforced restrictions**
- **Use Conditional Access App Control**
- **Token protection-related controls where supported**

**Conditional Access session controls**

Session controls shape how access behaves after sign-in. They matter because some threats occur after initial authentication, especially on unmanaged devices or in browser sessions.

Key controls:

- **Sign-in frequency** forces reauthentication after a defined interval. Use it for sensitive apps or privileged users when you want periodic proof of presence.
- **Persistent browser session** controls whether browser sessions survive closing the browser. Disable persistence for high-risk scenarios.
- **App enforced restrictions** can limit download, copy, and print actions in supported apps.
- **Conditional Access App Control** integrates with Microsoft Defender for Cloud Apps to monitor and control sessions in real time.

Practical guidance:

- Use **sign-in frequency** for privileged portals and sensitive administrative tasks.
- Use **persistent browser session** carefully on shared or unmanaged devices.
- Use **app enforced restrictions** for browser-based access to sensitive data on unmanaged endpoints.

**Authentication strength and step-up authentication**

Authentication strength lets you require a specific class of authentication method rather than just “MFA.” This is useful when you want phishing-resistant methods for privileged actions.

Examples:

- Require **phishing-resistant MFA** for Azure portal administration.
- Allow standard MFA for low-risk productivity apps.
- Require stronger methods for finance, HR, or admin portals.

Authentication context enables **step-up authentication**. You can tag a sensitive app, resource, or action so that a user who already signed in must satisfy a stronger policy before accessing that specific resource.

Use cases:

- Accessing a sensitive SharePoint site.
- Opening a privileged Azure management blade.
- Approving a high-impact workflow.
- Accessing confidential line-of-business data.

A common pattern is:

- Baseline access uses normal Conditional Access.
- Sensitive action triggers authentication context.
- The user completes a stronger challenge only when needed.

**Break-glass and emergency access accounts**

Break-glass accounts are emergency access accounts designed to preserve tenant access if normal identity controls fail. They are essential in Azure security design.

Operational requirements:

- Create at least two emergency accounts.
- Use strong, unique passwords stored offline in a secure process.
- Exclude them from Conditional Access policies that could lock out administrators.
- Monitor them aggressively with alerts and sign-in logs.
- Use them only for emergencies and test them periodically.
- Prefer cloud-only accounts so they are not dependent on on-premises identity infrastructure.

Design notes:

- Do not assign day-to-day admin use to break-glass accounts.
- Do not register them in normal MFA flows if that would create a lockout dependency.
- Protect them with compensating controls such as privileged access monitoring and alerting.

> Break-glass accounts are an exception path, not a convenience path. If they are used routinely, the design is wrong.

**Azure RBAC design**

Azure Role-Based Access Control (RBAC) governs access to Azure resources. It is separate from Microsoft Entra ID authentication and is enforced at management plane scope.

Core concepts:

- **Role definition**: the set of allowed actions.
- **Role assignment**: binding a role to a principal at a scope.
- **Scope**: management group, subscription, resource group, or resource.
- **Assignable scopes**: where a custom role can be assigned.

Built-in roles are preferred when they fit the requirement because they are maintained by Microsoft and easier to audit. Use custom roles when built-in roles are too broad or too narrow.

Examples of when to create a custom role:

- A team needs to restart VMs but not delete them.
- A support group needs read-only access to specific resource types.
- A platform team needs limited network configuration rights without subscription-wide owner permissions.

Custom role design tips:

- Start from the smallest set of `Actions` and `DataActions`.
- Avoid wildcard permissions unless necessary.
- Test the role in a nonproduction subscription.
- Document the business justification and scope.

Example custom role definition:

```json
{
  "Name": "VM Restart Operator",
  "IsCustom": true,
  "Description": "Can restart virtual machines without delete permissions.",
  "Actions": [
    "Microsoft.Compute/virtualMachines/read",
    "Microsoft.Compute/virtualMachines/restart/action",
    "Microsoft.Resources/subscriptions/resourceGroups/read"
  ],
  "NotActions": [
    "Microsoft.Compute/virtualMachines/delete"
  ],
  "AssignableScopes": [
    "/subscriptions/00000000-0000-0000-0000-000000000000"
  ]
}
```

**Role assignment examples**

Use Azure CLI to assign a role at the narrowest practical scope.

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Reader" \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-app
```

For service principals or managed identities, you may use the object ID or principal ID depending on the scenario and CLI version.

```bash
az role assignment create \
  --assignee 11111111-1111-1111-1111-111111111111 \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-sec/providers/Microsoft.KeyVault/vaults/kv-prod
```

**Data plane vs management plane**

Azure RBAC can control both management plane and, for some services, data plane access. This distinction is important for exam questions.

- **Management plane**: create, delete, configure Azure resources.
- **Data plane**: read or write the data inside a service, such as secrets, blobs, or keys.

Examples:

- `Reader` is a management-plane role.
- `Key Vault Secrets User` is a data-plane role for secret retrieval when the vault uses Azure RBAC authorization.
- Storage data roles control blob or queue data access, not resource configuration.

For Key Vault, access is governed by one of two mutually exclusive models:

- **Access policies**
- **Azure RBAC authorization** with `enableRbacAuthorization`

If Azure RBAC is enabled on the vault, data-plane access comes from RBAC roles such as `Key Vault Secrets User`. If access policies are used, RBAC data roles do not grant secret access.

**Azure resource locks**

Resource locks protect resources from accidental deletion or modification. They are not an identity control, but they are part of access hardening because they reduce destructive actions even when a user has RBAC permissions.

Lock types:

- **CanNotDelete** prevents deletion but allows read and write.
- **ReadOnly** prevents changes and deletion.

Important differences:

- Locks apply at scope and inherit downward.
- Locks do not replace RBAC.
- Locks do not grant access.
- Locks can block legitimate automation if applied too broadly.

Use locks for:

- Critical resource groups.
- Shared network components.
- Production vaults or storage accounts where deletion would be catastrophic.

**Deny assignments**

Deny assignments explicitly block actions even if a user has an allow role assignment. They are commonly used by managed applications and some platform-managed scenarios.

Key points:

- Deny assignments are not the same as RBAC role assignments.
- They are often created by Azure-managed resources or services.
- They can override allow permissions for specific actions or scopes.

For troubleshooting, do not assume a missing permission is always an RBAC issue. Check for deny assignments, locks, and inherited scope restrictions.

**Privileged Identity Management**

Privileged Identity Management (PIM) reduces standing privilege by making privileged roles eligible instead of permanently active. This is one of the most important AZ-500 hardening topics.

PIM capabilities:

- **Eligible assignments** require activation before use.
- **Just-in-time** access limits the time a role is active.
- **Approval workflows** can require human approval.
- **MFA on activation** can require a fresh challenge.
- **Justification** and **ticket information** improve auditability.
- **Access reviews** help remove stale privilege.

Common privileged roles to manage with PIM:

- Global Administrator
- Privileged Role Administrator
- Security Administrator
- Conditional Access Administrator
- User Administrator
- Subscription Owner
- User Access Administrator

Activation controls to consider:

- Require MFA.
- Require justification.
- Require approval for high-impact roles.
- Limit activation duration.
- Send notifications to security teams.

**Privileged Access Groups**

Privileged Access Groups let you manage group membership through PIM so that users become eligible for group membership rather than permanently assigned.

How they differ from standard group-based role assignment:

- Standard groups are static unless manually changed.
- Privileged Access Groups support eligible membership and activation.
- Role assignment can be made to the group, and users gain access only when their membership is active.

Limitations and considerations:

- They add operational complexity.
- They require careful governance to avoid hidden privilege paths.
- They are best for roles that should be time-bound and reviewed.
- They do not replace the need to review the role assigned to the group.

A common pattern is:

1. Assign an Azure RBAC role to a security group.
2. Make users eligible for membership in that group through PIM.
3. Require activation before the user inherits the role.

**Guest access and external collaboration**

Guest access is a major identity risk because external users can accumulate access over time. Hardening requires both authentication controls and lifecycle governance.

Key controls:

- Restrict who can invite guests.
- Use cross-tenant access settings to control inbound and outbound trust.
- Require MFA from trusted external tenants when appropriate.
- Review guest accounts regularly.
- Remove stale guests and unused app consents.

Cross-tenant access settings let you define trust relationships with other organizations. You can control:

- **Inbound access** to your tenant.
- **Outbound access** from your tenant.
- **Trust settings** for MFA, compliant device claims, and hybrid join claims from trusted tenants.

Practical use cases:

- Trust a partner tenant for MFA claims to reduce friction.
- Block access from high-risk external tenants.
- Apply stricter controls to guest users than to members.

Example portal workflow:

1. Open Microsoft Entra admin center.
2. Go to **External Identities**.
3. Configure **Cross-tenant access settings**.
4. Add an organization or use default settings.
5. Set inbound and outbound trust options.
6. Test with a pilot guest account.

**Tenant restrictions and consent governance**

Tenant restrictions help prevent users from signing into unauthorized tenants from managed devices or networks. This is useful in environments where users should only access approved tenants.

Consent governance reduces the risk of malicious or overbroad app permissions.

Controls to know:

- **Admin consent workflow** for user-requested apps.
- **User consent settings** to limit self-service consent.
- **Publisher verification** to prefer trusted apps.
- **Permission classification** to identify risky scopes.
- **Tenant restrictions** to block access to unauthorized tenants.

Practical guidance:

- Disable broad user consent in enterprise tenants unless there is a strong governance model.
- Use admin consent for high-impact permissions.
- Review enterprise applications and service principals regularly.

**Access reviews and entitlement management**

Access reviews help you periodically verify whether users still need access. Entitlement management helps package access into governed access packages with approvals, expirations, and reviews.

Access reviews are useful for:

- Guest users.
- Privileged role assignments.
- Group membership.
- Application access.

Entitlement management is useful for:

- Onboarding external users.
- Granting access through access packages.
- Enforcing approval and expiration.
- Automating lifecycle cleanup.

Typical governance flow:

1. A user requests access to a package.
2. An approver reviews the request.
3. The user receives time-bound access.
4. An access review later confirms whether access should continue.
5. Expired access is removed automatically.

**Workload identities and secretless authentication**

Non-human identities such as apps, automation, and CI/CD pipelines should avoid long-lived secrets when possible. Modern hardening favors workload identity federation over client secrets.

Preferred approaches:

- **Managed identities** for Azure-hosted workloads.
- **Workload identity federation** for external identity providers such as GitHub Actions or other OIDC-capable systems.
- **Certificate-based authentication** when federation is not available and a secretless path is not possible.

Why this matters:

- Secrets are hard to rotate safely.
- Secrets are often copied into scripts and pipelines.
- Federation uses short-lived tokens and trust assertions instead of stored passwords.

Example federated credential concept:

- A GitHub Actions workflow authenticates to Microsoft Entra ID using OpenID Connect (OIDC).
- Entra validates the external token issuer, subject, and audience.
- The app receives an access token without storing a client secret.

Example Azure CLI app registration setup pattern:

```bash
az ad app create --display-name "ci-cd-deployer"
az ad app federated-credential create \
  --id <app-object-id> \
  --parameters federated-credential.json
```

Example federated credential JSON:

```json
{
  "name": "github-main-branch",
  "issuer": "https://token.actions.githubusercontent.com",
  "subject": "repo:contoso/app:ref:refs/heads/main",
  "audiences": [
    "api://AzureADTokenExchange"
  ]
}
```

**Managed identities**

Managed identities are the preferred way for Azure resources to authenticate to other Azure services without secrets.

Types:

- **System-assigned**: tied to one resource lifecycle.
- **User-assigned**: reusable across multiple resources.

Use managed identities for:

- App Service to Key Vault access.
- Azure Functions to Storage access.
- Virtual machines to Key Vault or ARM access.
- Automation accounts and runbooks where supported.

Best practices:

- Grant only the required data-plane or management-plane role.
- Prefer user-assigned identities when multiple resources need the same identity.
- Remove unused identities and role assignments.

**Authentication and authorization troubleshooting**

When access fails, isolate the layer first.

Checklist:

- Confirm the user or workload can authenticate.
- Confirm the correct tenant is being used.
- Check Conditional Access results.
- Check role assignment scope and inheritance.
- Check for deny assignments or resource locks.
- Check whether the service uses access policies or Azure RBAC.
- Check whether the identity is eligible but not activated in PIM.
- Check whether guest trust or consent settings block the flow.

Useful Azure CLI commands:

```bash
az role assignment list --assignee user@contoso.com --all
az role definition list --name "Reader"
az lock list --resource-group rg-app
az ad signed-in-user show
```

For sign-in troubleshooting, use Microsoft Entra sign-in logs and Conditional Access insights. For authorization troubleshooting, use Azure Activity Log and resource-specific diagnostic logs.

**Exam-relevant design patterns**

Use these patterns to answer AZ-500 scenarios correctly:

- **Privileged admin access**: PIM + phishing-resistant MFA + Conditional Access + access reviews.
- **Guest collaboration**: cross-tenant access settings + access reviews + limited consent.
- **App-to-Azure access**: managed identity or workload identity federation instead of secrets.
- **Sensitive app access**: authentication context + step-up authentication + session controls.
- **Production protection**: RBAC + resource locks + narrow scopes + break-glass accounts.
- **Legacy protocol reduction**: block legacy authentication and remove app passwords.

**Common pitfalls**

- **Confusing RBAC with authentication**: RBAC does not authenticate users; it authorizes actions after sign-in.
- **Using broad Owner access**: Owner is often too permissive for daily administration.
- **Relying on per-user MFA**: Conditional Access is the preferred enterprise control.
- **Ignoring session controls**: MFA alone does not address unmanaged browser sessions.
- **Leaving guests unmanaged**: external users need lifecycle reviews and expiration.
- **Using secrets for automation**: federation or managed identity is usually safer.
- **Forgetting vault authorization mode**: Key Vault access policy and Azure RBAC are different models.

**Quick reference**

| Area | Preferred control | Why it matters |
|---|---|---|
| User sign-in | Conditional Access | Central policy enforcement |
| Privileged roles | PIM | Removes standing privilege |
| External users | Cross-tenant access + access reviews | Limits guest risk |
| Automation | Managed identity or federation | Avoids secrets |
| Sensitive apps | Authentication context | Step-up only when needed |
| Resource protection | RBAC + locks | Limits destructive actions |
| Legacy auth | Block it | Reduces bypass risk |

**Practical portal and CLI examples**

Conditional Access policy concept:

```text
Users: All users
Exclude: Break-glass accounts
Cloud apps: Azure management
Grant: Require phishing-resistant MFA
Session: Sign-in frequency = 8 hours
```

Block legacy authentication with a policy targeting legacy client apps and excluding emergency accounts.

Azure RBAC custom role assignment workflow:

1. Define the minimum actions.
2. Create the custom role.
3. Assign it at the narrowest scope.
4. Validate access with a test principal.
5. Review and refine after operational testing.

PIM activation workflow:

1. Make the user eligible for the role.
2. Require MFA and justification.
3. Set activation duration.
4. Optionally require approval.
5. Review activation logs and alerts.

**Summary**

Identity and access hardening in Azure is a layered design problem. You secure authentication with modern methods and Conditional Access, reduce privilege with Azure RBAC and PIM, govern lifecycle with access reviews and entitlement management, and protect workloads with managed identities or workload identity federation. For AZ-500, focus on choosing the right control for the right layer and on understanding how policies, scopes, and session controls interact.
