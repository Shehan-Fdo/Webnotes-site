## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Key Vault security model**

Azure Key Vault stores and controls access to cryptographic material and application secrets. In exam scenarios, separate the three object types because they behave differently:

- **Secrets** store arbitrary strings such as passwords, API keys, and connection strings.
- **Keys** store cryptographic keys used for encryption, signing, and wrapping operations.
- **Certificates** combine a private key, public certificate, and certificate policy metadata.

The protection model is not just about storage. It includes identity, authorization, network access, logging, and lifecycle controls. A secure design usually combines Microsoft Entra ID authentication, least-privilege authorization, network restrictions, and recovery features such as soft delete and purge protection.

<dl>
<dt><strong>Secret</strong></dt>
<dd>Opaque value managed as a versioned string. Rotation is usually application-driven or automated with custom logic.</dd>
<dt><strong>Key</strong></dt>
<dd>Cryptographic material used by Azure services or applications. Rotation can be policy-driven for supported key types and operations.</dd>
<dt><strong>Certificate</strong></dt>
<dd>Managed certificate object with policy, lifetime actions, and optional issuer integration for renewal.</dd>
</dl>

**Authorization: access policy vs Azure RBAC**

Key Vault supports two authorization models:

- **Vault access policy model**
  - Uses explicit vault-level access policies.
  - Grants data-plane permissions such as `get`, `list`, `set`, `delete`, `recover`, `sign`, and `unwrapKey`.
  - Is simpler for small deployments but less flexible at scale.

- **Azure role-based access control (Azure RBAC)**
  - Uses Azure role assignments for data-plane access.
  - Centralizes management with standard Azure governance tools.
  - Is the preferred model for many modern deployments because it aligns with least privilege and enterprise role management.

When you enable RBAC authorization on a vault, you manage data access with roles such as:

- **Key Vault Secrets User**
- **Key Vault Secrets Officer**
- **Key Vault Crypto User**
- **Key Vault Crypto Officer**
- **Key Vault Certificates Officer**
- **Key Vault Administrator**

Use the narrowest role that satisfies the workload. For example, an app that only reads one secret should not receive administrative permissions.

**Example: create a vault with RBAC authorization**

```bash
az keyvault create \
  --name kv-secure-prod \
  --resource-group rg-security \
  --location eastus \
  --enable-rbac-authorization true \
  --sku standard
```

**Example: assign a data-plane role to a managed identity**

Managed identities are represented by service principals in Microsoft Entra ID. In practice, use the identity's object ID and ensure the principal type matches the identity type you are assigning.

```bash
az role assignment create \
  --assignee-object-id 11111111-2222-3333-4444-555555555555 \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<subId>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-secure-prod
```

> Use the identity's object ID for the managed identity and verify the scope. A role assignment at the vault scope does not automatically grant access to every secret unless the role includes the needed data actions.

**Network protection**

Key Vault network controls determine whether requests can reach the public endpoint, a private endpoint, or both. A private endpoint does not automatically remove the public endpoint. You must configure public network access and firewall settings intentionally.

Key network controls include:

- **Public network access**
  - Can be enabled or disabled depending on the vault configuration.
  - If enabled, the vault can still be reachable over the public endpoint subject to firewall rules.

- **Firewall and virtual network rules**
  - Restrict access by IP address or selected virtual networks.
  - Useful when you need controlled public access.

- **Private Endpoint**
  - Maps the vault to a private IP address in your virtual network.
  - Traffic stays on the Microsoft backbone and does not traverse the public internet.
  - Requires private DNS so clients resolve the vault name to the private IP.

- **Trusted Microsoft services**
  - Some Microsoft services can bypass firewall restrictions when explicitly allowed.
  - This is useful for service integrations, but it is not the same as private access.

A common exam pattern is to combine private endpoint access with public network access disabled. That gives the strongest isolation for workloads that can resolve the private DNS name and reach the VNet.

**Private endpoint and DNS flow**

For Key Vault, the public FQDN is `vault.azure.net`, and the private endpoint uses the `privatelink.vaultcore.azure.net` DNS zone. The client resolves the vault name through a CNAME chain that ultimately points to the private IP.

Typical resolution path:

- `myvault.vault.azure.net`
- CNAME to `myvault.privatelink.vaultcore.azure.net`
- A record in the private DNS zone resolves to the private endpoint IP

**Example: create a private endpoint**

```bash
az network private-endpoint create \
  --name pe-kv-secure-prod \
  --resource-group rg-security \
  --vnet-name vnet-secure \
  --subnet snet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-secure-prod \
  --group-ids vault \
  --connection-name kv-secure-prod-connection
```

**Example: create the private DNS zone and link it**

```bash
az network private-dns zone create \
  --resource-group rg-security \
  --name privatelink.vaultcore.azure.net

az network private-dns link vnet create \
  --resource-group rg-security \
  --zone-name privatelink.vaultcore.azure.net \
  --name link-secure-vnet \
  --virtual-network vnet-secure \
  --registration-enabled false
```

**Example: add a DNS record for the private endpoint**

```bash
az network private-dns record-set a add-record \
  --resource-group rg-security \
  --zone-name privatelink.vaultcore.azure.net \
  --record-set-name myvault \
  --ipv4-address 10.10.2.4
```

**Practical network design**

- Use **private endpoint only** when workloads are inside controlled VNets and can use private DNS.
- Use **firewall rules** when you need limited public access from known IP ranges.
- Use **trusted services** only when the service integration requires it and you understand the bypass implications.
- Disable public network access when the business requirement is exclusive private connectivity.

**Soft delete, purge protection, and recovery**

Soft delete is enabled by default for new Key Vaults and is mandatory in modern Azure behavior. It protects against accidental deletion by retaining deleted objects for a retention period. Purge protection prevents permanent deletion until the retention period expires.

Important recovery behaviors:

- **Soft delete**
  - Applies to secrets, keys, and certificates.
  - Allows recovery of deleted objects during the retention window.
  - Helps protect against accidental or malicious deletion.

- **Purge protection**
  - Prevents immediate permanent deletion.
  - Is especially important for production vaults and compliance scenarios.
  - Works with soft delete to reduce the risk of destructive operations.

- **Vault deletion**
  - A deleted vault can often be recovered during the retention period if soft delete is active.
  - Purge protection blocks irreversible purge until the retention period ends.

> For exam questions, treat soft delete as the baseline protection and purge protection as the stronger safeguard for production and regulated environments.

**Example: check recovery-related properties**

```bash
az keyvault show \
  --name kv-secure-prod \
  --resource-group rg-security \
  --query "{softDeleteRetentionInDays:properties.softDeleteRetentionInDays, enablePurgeProtection:properties.enablePurgeProtection, enableRbacAuthorization:properties.enableRbacAuthorization}"
```

**Keys, secrets, and certificates: lifecycle differences**

Understanding object behavior is critical for AZ-500 questions.

- **Secrets**
  - Support versioning.
  - Do not have native cryptographic rotation semantics.
  - Usually rotate through application logic, automation, or external secret management workflows.
  - Commonly used for passwords and API credentials.

- **Keys**
  - Support versioning.
  - Can be used for encryption, signing, and key wrapping.
  - Rotation can be policy-driven for supported key types and operations.
  - Applications may need to reference the latest version or explicitly handle version changes.

- **Certificates**
  - Support versioning and policy metadata.
  - Can be imported or created/issued through Key Vault.
  - Renewal can be automated when the certificate policy and issuer support it.
  - Some consuming services require versionless references or explicit refresh behavior to pick up renewals.

**Certificate lifecycle: import vs create vs issue**

Key Vault certificates can enter the vault in different ways:

- **Import**
  - You upload an existing certificate, usually as a PFX/PKCS#12 file with the private key.
  - Useful when migrating from another CA or when you already own the certificate.

- **Create**
  - Key Vault generates the key pair and creates a self-signed certificate or a certificate request.
  - Useful for internal testing or when you want Key Vault to own the key material.

- **Issue**
  - Key Vault works with a configured certificate issuer, such as a supported CA integration.
  - Useful for managed renewal workflows where the issuer can renew based on policy.

Certificate renewal is not universally automatic for every Azure service. Some services can consume a Key Vault certificate reference and refresh it, but behavior depends on the service, the reference style, and whether the service supports versionless certificate retrieval.

**Example: create a certificate policy file**

```json
{
  "issuerParameters": {
    "name": "Self"
  },
  "keyProperties": {
    "exportable": true,
    "keyType": "RSA",
    "keySize": 2048,
    "reuseKey": false
  },
  "secretProperties": {
    "contentType": "application/x-pkcs12"
  },
  "x509CertificateProperties": {
    "subject": "CN=app.contoso.com",
    "validityInMonths": 12
  },
  "lifetimeActions": [
    {
      "trigger": {
        "daysBeforeExpiry": 30
      },
      "action": {
        "actionType": "AutoRenew"
      }
    }
  ]
}
```

**Example: create a certificate from policy**

```bash
az keyvault certificate create \
  --vault-name kv-secure-prod \
  --name app-tls-cert \
  --policy @cert-policy.json
```

**Rotation strategies**

Rotation is different for each object type.

- **Secret rotation**
  - Usually requires custom automation.
  - Common pattern: create a new secret version, update the application or downstream service, then retire the old version.
  - Use Azure Automation, Azure Functions, Logic Apps, or CI/CD pipelines when the source system does not support native rotation.

- **Key rotation**
  - Can be policy-driven for supported keys.
  - Useful for cryptographic hygiene and compliance.
  - Applications should be designed to tolerate key version changes.

- **Certificate renewal**
  - Can be automated when the certificate policy and issuer support it.
  - Often involves a renewal trigger before expiry.
  - Consumers may need to reference the certificate in a way that allows refresh.

**Example: add a new secret version**

```bash
az keyvault secret set \
  --vault-name kv-secure-prod \
  --name app-db-password \
  --value "NewStrongPassword!2026"
```

**Example: read the latest secret version**

```bash
az keyvault secret show \
  --vault-name kv-secure-prod \
  --name app-db-password \
  --query value -o tsv
```

**Key rotation policy considerations**

Key rotation policy commands and JSON shapes can vary by Azure CLI version. In exam prep, focus on the concept rather than memorizing a brittle syntax. The important points are:

- Rotation policy defines when a key should rotate.
- Triggers can be time-based, such as a set number of days before expiry.
- Actions can include automatic rotation or notifications.
- Applications should not assume a key version is permanent.

**Practical rotation pattern**

1. Create a key with an expiration date.
2. Define a rotation policy with a pre-expiry trigger.
3. Update the application to use the key by name or a managed reference pattern.
4. Validate that the new version is created before the old one expires.
5. Monitor for failures in dependent services.

**Monitoring, diagnostics, and auditability**

Key Vault diagnostics are essential for detecting unauthorized access, failed operations, and rotation issues. Use Azure Monitor and diagnostic settings to send logs to Log Analytics, Event Hub, or Storage.

Common audit signals include:

- **Secret access events**
  - Detect reads, writes, deletes, and recoveries.
- **Key operations**
  - Detect wrap, unwrap, sign, verify, encrypt, decrypt, and rotation activity.
- **Certificate operations**
  - Detect create, import, renew, delete, and policy changes.
- **Authorization failures**
  - Detect denied requests that may indicate misconfiguration or attack attempts.

Depending on the logging model in your environment, logs may appear in legacy diagnostic tables or resource-specific tables. The exact table names and categories can vary over time, so validate the current Azure Monitor configuration in your tenant.

**Example: enable diagnostic settings to Log Analytics**

```bash
az monitor diagnostic-settings create \
  --name kv-diag \
  --resource /subscriptions/<subId>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-secure-prod \
  --workspace /subscriptions/<subId>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-security \
  --logs '[{"category":"AuditEvent","enabled":true}]'
```

Use logs to answer operational questions such as:

- Who accessed a secret?
- Did a key rotation occur before expiry?
- Was a certificate renewal attempted and did it fail?
- Did a principal receive `403 Forbidden` because of RBAC or firewall restrictions?

**Managed HSM vs Key Vault**

Azure Key Vault and Azure Key Vault Managed HSM solve different problems.

- **Azure Key Vault**
  - Best for secrets, certificates, and general-purpose key management.
  - Supports standard vault features and broad service integration.
  - Suitable for most application and platform security scenarios.

- **Azure Key Vault Managed HSM**
  - Designed for higher-assurance key protection with dedicated HSM-backed control.
  - Focused on key operations rather than secrets and certificates.
  - Appropriate when compliance or cryptographic isolation requires HSM-level control.

Choose Managed HSM when the requirement explicitly calls for hardware-backed key custody with stronger isolation. Choose Key Vault when you need a broader secret, certificate, and key lifecycle platform.

**Common exam scenarios**

- A web app needs to read a database password securely.
  - Use a **secret**.
  - Grant the app a narrow secret-read role.
  - Prefer private endpoint if the app runs in a VNet.

- An application needs to sign tokens or encrypt data.
  - Use a **key**.
  - Plan for versioning and rotation.
  - Monitor for failed cryptographic operations.

- A TLS certificate must renew before expiry.
  - Use a **certificate** with a policy and issuer integration if supported.
  - Confirm the consuming service can refresh the certificate reference.

- A regulated workload must prevent accidental deletion.
  - Enable **purge protection** and rely on soft delete.
  - Restrict delete and purge permissions tightly.

- A workload must never expose Key Vault over the internet.
  - Use a **private endpoint**.
  - Disable public network access.
  - Configure private DNS correctly.

**Troubleshooting checklist**

- Verify the identity has the correct data-plane role or access policy.
- Confirm the vault is using the intended authorization model.
- Check whether public network access is disabled or firewall rules block the client.
- Validate private DNS resolution for `*.vault.azure.net`.
- Confirm the object version exists and has not expired or been deleted.
- Review audit logs for `403`, `404`, and `409` responses.
- Check whether the consuming service supports automatic refresh of certificates or keys.

**Quick comparison**

| Area | Secrets | Keys | Certificates |
|---|---|---|---|
| Primary use | Store sensitive values | Cryptographic operations | TLS and identity certificates |
| Versioning | Yes | Yes | Yes |
| Native rotation | Usually no | Yes, for supported scenarios | Yes, with policy and issuer support |
| Recovery | Soft delete and purge protection | Soft delete and purge protection | Soft delete and purge protection |
| Common consumers | Apps, scripts, services | Apps, Azure services, crypto APIs | Web apps, gateways, services |

**Exam takeaways**

- Use **RBAC** or access policies, but know which model the vault is using.
- Treat **private endpoint** as network isolation, not automatic public endpoint removal.
- Remember that **soft delete** is the baseline and **purge protection** is the stronger safeguard.
- Distinguish **secret rotation**, **key rotation**, and **certificate renewal**.
- Know when to use **Managed HSM** instead of standard Key Vault.
- Validate **DNS**, **network rules**, and **audit logs** when troubleshooting access.
