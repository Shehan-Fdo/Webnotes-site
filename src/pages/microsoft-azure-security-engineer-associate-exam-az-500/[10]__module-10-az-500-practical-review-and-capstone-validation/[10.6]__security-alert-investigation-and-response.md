## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Security alert investigation and response in Azure is about turning telemetry into action. You identify what happened, validate whether it is real, preserve evidence, contain the threat, and then tune detections so the same noise does not keep returning.

For AZ-500, focus on the full workflow across Microsoft Defender, Microsoft Sentinel, Microsoft Entra ID, Azure Monitor, and service-specific logs. The exam often tests which data source to use, which response action is available, and which control-plane or data-plane log contains the evidence.

**Core investigation workflow**

A practical response flow usually follows this pattern:

1. Triage the alert and confirm scope.
2. Collect evidence from the right telemetry source.
3. Correlate related activity across identities, endpoints, cloud resources, and network controls.
4. Contain the threat using the least disruptive action that stops further damage.
5. Preserve evidence before destructive remediation.
6. Tune detections, suppression, and incident grouping to reduce false positives.

This order matters because containment can destroy evidence if you act too early. For example, deleting a malicious VM or rotating a secret before exporting logs may remove the timeline you need for root-cause analysis.

**Telemetry sources you should know**

Different Azure services expose different kinds of evidence. A common exam trap is assuming the Azure Activity Log contains everything. It does not.

- **Azure Activity Log**
  - Records control-plane operations on Azure resources.
  - Useful for changes such as role assignments, policy changes, resource creation, deletion, and configuration updates.
  - Does not record most data-plane access, such as reading a Key Vault secret or downloading a blob.

- **Resource logs**
  - Capture service-specific data-plane or operational events.
  - Examples include Key Vault diagnostic logs, Storage account logs, Azure Firewall logs, and Application Gateway logs.
  - Often sent to Log Analytics, Event Hub, or Storage for retention and analysis.

- **Microsoft Entra ID logs**
  - Include sign-in logs, audit logs, risky users, and risky sign-ins.
  - Essential for identity investigations, conditional access impact analysis, and privilege changes.

- **Microsoft Defender telemetry**
  - Microsoft Defender for Endpoint provides device alerts, device timeline, isolation, live response, and remediation actions.
  - Microsoft Defender for Cloud provides cloud security posture, recommendations, and workload alerts.
  - Microsoft Defender for Cloud Apps helps investigate SaaS activity and shadow IT.
  - Microsoft Defender XDR correlates alerts across identity, endpoint, email, and cloud apps.

- **Microsoft Sentinel data**
  - Centralizes logs, analytics rules, incidents, bookmarks, hunting queries, and automation.
  - Best for cross-domain correlation and incident management.

**Identity investigation with Microsoft Entra ID**

Identity is often the first pivot in an Azure incident. If you see suspicious resource access, start with the user or service principal involved.

Key sources:

- **Sign-in logs**
  - Show interactive and non-interactive sign-ins.
  - Include IP address, device, application, conditional access result, MFA status, and risk signals where available.
  - Use them to confirm whether a sign-in was expected and whether conditional access blocked or allowed it.

- **Audit logs**
  - Show directory changes such as user creation, group membership changes, app registrations, credential resets, and role assignments.
  - Use them to investigate privilege escalation or persistence.

- **Risky users and risky sign-ins**
  - Help prioritize accounts that may be compromised.
  - Useful when you need to decide whether to force password reset, revoke sessions, or require MFA reauthentication.

- **Conditional Access**
  - Important during response because a policy may explain why access succeeded or failed.
  - Check whether a policy enforced MFA, device compliance, location restrictions, or sign-in risk controls.

A practical KQL example for Entra sign-in investigation:

```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| where UserPrincipalName =~ "alice@contoso.com"
| project TimeGenerated, UserPrincipalName, AppDisplayName, IPAddress, Location, ConditionalAccessStatus, ResultType, ResultDescription
| order by TimeGenerated desc
```

For directory changes:

```kusto
AuditLogs
| where TimeGenerated > ago(24h)
| where InitiatedBy.user.userPrincipalName =~ "alice@contoso.com"
| project TimeGenerated, OperationName, Result, TargetResources, InitiatedBy
| order by TimeGenerated desc
```

Use these queries as starting points, not final detections. In real incidents, you usually pivot from one identity to related users, groups, service principals, and devices.

**Microsoft Defender for Endpoint response actions**

Microsoft Defender for Endpoint is a major source of endpoint evidence and containment actions. It is especially relevant when an Azure incident involves a compromised admin workstation or a server used for lateral movement.

Common response actions include:

- **Device isolation**
  - Cuts the device off from the network while preserving Defender connectivity.
  - Use it when you need to stop active attacker movement on a compromised endpoint.
  - This is a Defender for Endpoint action, not a native Azure-only capability.

- **Collect investigation package**
  - Gathers logs and artifacts for analysis.
  - Useful before remediation if you need forensic evidence.

- **Run antivirus scan**
  - Helps detect and remove known malware.

- **Live response**
  - Provides an interactive shell for investigation and limited remediation.
  - Use it when you need to inspect files, processes, registry keys, or network state without full remote desktop access.

- **Restrict app execution / block indicators**
  - Helps prevent known malicious binaries, hashes, IPs, or URLs from being used again.

- **User block/unblock**
  - Often handled through identity controls or integrated response workflows depending on the product and tenant setup.
  - In exam scenarios, think in terms of disabling sign-in, revoking sessions, or blocking access through identity controls.

The **Action center** is where pending response actions are reviewed, approved, and tracked in Microsoft Defender XDR. Exact behavior depends on the tenant and integrated products, but the exam-relevant idea is that some actions require approval and can be monitored centrally.

**Microsoft Defender for Cloud investigation and response**

Microsoft Defender for Cloud helps you investigate cloud workload alerts and prioritize remediation using posture data.

Important concepts:

- **Security alerts**
  - Indicate suspicious or malicious activity in supported Azure resources and workloads.
  - Often link to the affected resource, evidence, and recommended next steps.

- **Recommendations**
  - Show misconfigurations and hardening opportunities.
  - Use them to reduce attack surface after or during incident response.

- **Secure score**
  - Helps prioritize remediation by showing which improvements have the biggest security impact.
  - Useful when you need to decide whether to fix a high-risk exposure immediately or after containment.

- **Alert suppression**
  - Use suppression when an alert is expected, benign, or too noisy to be useful.
  - Suppression reduces repeated alerts from known patterns, but you should document why the alert is suppressed.

- **Dismissal and resolution**
  - Dismiss alerts only when you have confirmed they are false positives or no longer relevant.
  - Resolution should reflect the actual outcome, not just that the alert stopped firing.

A good response pattern is to investigate the alert, check the resource configuration, review related activity logs, and then use recommendations and secure score to harden the environment so the same issue does not recur.

**Microsoft Sentinel incident management**

Microsoft Sentinel is the main Azure-native SIEM and SOAR platform for incident handling. It is where you correlate alerts, manage incidents, and automate response.

Key incident management features:

- **Incidents**
  - Group related alerts into a single case.
  - Help you track the full story instead of treating each alert separately.

- **Incident status**
  - Common states include New, Active, and Closed.
  - Use status changes to reflect triage progress.

- **Ownership**
  - Assign an incident to a responder or team.
  - Ownership improves accountability and avoids duplicate work.

- **Comments**
  - Record findings, decisions, and handoff notes.
  - Comments are useful for auditability and team coordination.

- **Tasks**
  - Break response work into actionable items.
  - Helpful when multiple analysts are working the same incident.

- **Bookmarks**
  - Save important events from logs or hunting queries.
  - Use them to preserve evidence and connect timeline events to an incident.

- **Hunting queries**
  - Support proactive searching for related activity.
  - Use them to confirm whether the incident is isolated or part of a broader campaign.

- **Automation rules**
  - Can change incident status, assign owners, add tags, and run playbooks.
  - Use them to standardize triage and reduce manual effort.

> Automation rules do not “suppress” alerts in the same sense as analytics rule suppression. Suppression is typically handled in the detection rule or alert configuration.

A practical incident workflow in Sentinel:

1. Review the incident summary and related alerts.
2. Open the timeline and inspect entities such as users, hosts, IPs, and resources.
3. Add comments and assign ownership.
4. Create bookmarks for key evidence.
5. Run hunting queries to find related activity.
6. Trigger a playbook if containment or notification is needed.
7. Close the incident only after validation and documentation.

**Microsoft Defender XDR cross-domain investigation**

Microsoft Defender XDR correlates signals across identity, endpoint, email, and cloud apps. This is useful when a single alert is not enough to explain the attack path.

Use it to:

- Correlate a suspicious sign-in with endpoint activity.
- Trace a malicious email attachment to a device execution event.
- Link a cloud app alert to identity risk and endpoint compromise.
- Review the attack timeline in one place instead of jumping between portals.

Important response actions in Defender XDR include:

- **Device isolation**
- **Collect investigation package**
- **Live response**
- **File quarantine**
- **URL or indicator blocking**
- **User actions** such as blocking sign-in or forcing password reset through identity workflows

The **Action center** tracks pending and completed actions, especially when approvals are required or when multiple products contribute actions to the same incident.

**Microsoft Defender for Cloud Apps and cloud app investigation**

Microsoft Defender for Cloud Apps is useful when the incident involves SaaS activity, OAuth apps, or unusual file sharing.

Typical investigation uses:

- Review user activity across cloud apps.
- Identify impossible travel or unusual download behavior.
- Inspect OAuth app consent and risky app permissions.
- Detect shadow IT and unsanctioned app usage.
- Correlate cloud app activity with Entra ID sign-ins and Defender XDR alerts.

This matters in AZ-500 because attackers often move from identity compromise to cloud app abuse before touching Azure resources directly.

**Azure-native logs by service**

Use the right log source for the right question.

| Service | Control-plane evidence | Data-plane evidence |
|---|---|---|
| Azure subscription/resource group | Activity Log | N/A |
| Key Vault | ARM changes in Activity Log | Key Vault diagnostic logs for secret/key/certificate access, depending on configuration |
| Storage account | ARM changes in Activity Log | Storage resource logs for blob/file/queue/table operations, depending on configuration |
| Azure Firewall | Firewall policy/resource changes in Activity Log | Firewall logs and rule hit logs |
| Application Gateway | Configuration changes in Activity Log | Access logs, performance logs, firewall logs if WAF is enabled |
| Azure SQL | Resource changes in Activity Log | SQL auditing and diagnostic logs |
| AKS | Cluster resource changes in Activity Log | Kubernetes control plane and workload logs, depending on setup |

The exam often asks whether an event is a management-plane change or a data-plane access event. If a user changed a firewall rule, that is control plane. If a user read a secret or downloaded a blob, that is data plane.

**Practical CLI examples**

Use Azure CLI to inspect and configure logging and response-related settings. Commands vary by resource type and API version, so treat these as patterns.

Enable diagnostic settings for a resource:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vaultName>" \
  --workspace "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<lawName>" \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Check Azure Activity Log for recent control-plane operations:

```bash
az monitor activity-log list \
  --resource-group <rg> \
  --offset 24h \
  --query "[].{time:eventTimestamp, operation:operationName.value, status:status.value, caller:caller}" \
  -o table
```

List Entra sign-in logs through Microsoft Graph PowerShell or KQL-backed workspaces, depending on your environment. In many exam scenarios, you should know the source conceptually even if the exact tool differs.

Inspect a storage account network configuration:

```bash
az storage account show \
  --name <storageName> \
  --resource-group <rg> \
  --query "{publicNetworkAccess:publicNetworkAccess, defaultAction:networkRuleSet.defaultAction, bypass:networkRuleSet.bypass}"
```

Update a storage account firewall posture:

```bash
az storage account update \
  --name <storageName> \
  --resource-group <rg> \
  --default-action Deny
```

> `publicNetworkAccess` and firewall `defaultAction` are related but not identical controls. Disabling public network access is not the same as setting the firewall default action to Deny.

Inspect Azure Firewall policy and rule collections:

```bash
az network firewall policy rule-collection-group collection list \
  --policy-name <policyName> \
  --resource-group <rg> \
  --rule-collection-group-name <rcgName>
```

**KQL examples for investigation**

Use Kusto Query Language (KQL) to correlate events in Log Analytics or Sentinel.

Find recent Azure resource changes:

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where ActivityStatusValue =~ "Succeeded"
| project TimeGenerated, Caller, OperationNameValue, ResourceGroup, ResourceProviderValue, ActivitySubstatusValue
| order by TimeGenerated desc
```

Find Key Vault access events when diagnostic logs are available:

```kusto
AzureDiagnostics
| where TimeGenerated > ago(24h)
| where ResourceProvider == "MICROSOFT.KEYVAULT"
| where Category == "AuditEvent"
| project TimeGenerated, OperationName, ResultType, identity_claim_appid_g, identity_claim_upn_s, CallerIPAddress
| order by TimeGenerated desc
```

Find storage access events when storage logs are routed to the workspace:

```kusto
StorageBlobLogs
| where TimeGenerated > ago(24h)
| project TimeGenerated, AccountName, OperationName, StatusCode, AuthenticationType, CallerIpAddress
| order by TimeGenerated desc
```

Correlate Sentinel incidents with bookmarks and entities:

```kusto
SecurityIncident
| where TimeGenerated > ago(7d)
| project TimeGenerated, IncidentNumber, Title, Severity, Status, Owner
| order by TimeGenerated desc
```

Use hunting queries to look for related activity after a suspicious sign-in:

```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType == 0
| summarize SignIns=count() by UserPrincipalName, IPAddress, AppDisplayName
| order by SignIns desc
```

For suspicious activity, avoid overfitting to a single app name or one hardcoded indicator. Instead, look for unusual combinations such as new country, new device, unusual time, MFA failures, or impossible travel patterns where available.

**Alert tuning and false-positive reduction**

Alert tuning is part of response because noisy detections waste analyst time and hide real attacks.

Use these controls:

- **Suppression**
  - Suppress known-benign alerts when the pattern is expected and documented.
  - Keep suppression narrow and time-bound where possible.

- **Incident grouping**
  - Group related alerts into one incident to reduce duplication.
  - Tune grouping carefully so unrelated alerts do not collapse into a single case.

- **Analytics rule configuration**
  - Adjust thresholds, entity mappings, and scheduling.
  - Use scheduled rules for recurring patterns and near-real-time rules for faster detection when needed.

- **Exclusions**
  - Exclude known service accounts, maintenance windows, or trusted automation only when justified.
  - Overuse of exclusions creates blind spots.

- **Severity and enrichment**
  - Use severity to prioritize response.
  - Add entity mapping and enrichment so incidents contain useful context.

**Evidence preservation and forensics**

Preserve evidence before you make changes that could erase the attacker’s footprint.

Good preservation steps:

- Export relevant logs from Sentinel, Defender, or Log Analytics.
- Save incident comments, bookmarks, and timelines.
- Capture VM snapshots or disk snapshots when you need disk-level forensics.
- Collect investigation packages from Defender for Endpoint before reimaging a device.
- Preserve copies of suspicious scripts, binaries, and configuration files.
- Record timestamps, user IDs, IP addresses, and resource IDs.

Use snapshots when you need to analyze a compromised VM offline. Use log export when the evidence is primarily in telemetry. If you are unsure, preserve both the logs and the affected system state before remediation.

**Response actions by scenario**

Identity compromise:

- Disable the account or block sign-in.
- Revoke sessions and refresh tokens.
- Reset credentials and enforce MFA.
- Review recent sign-ins, risky sign-ins, and audit logs.
- Check for new app registrations, role assignments, or consent grants.

Compromised endpoint:

- Isolate the device through Defender for Endpoint.
- Collect an investigation package.
- Run a scan and inspect persistence mechanisms.
- Check whether the device accessed Azure portals, management tools, or secrets.

Compromised Azure resource:

- Review Activity Log for recent changes.
- Check resource logs for data-plane access.
- Rotate secrets, keys, or certificates if exposed.
- Restrict network access or disable public exposure where appropriate.
- Validate whether the resource has been used for lateral movement or exfiltration.

Suspicious cloud app activity:

- Review Defender for Cloud Apps activity.
- Revoke risky OAuth consent if needed.
- Disable the app or remove permissions.
- Check Entra sign-ins and audit logs for the user or service principal.

**End-to-end scenario walkthrough**

A realistic AZ-500 scenario might look like this:

1. Defender for Cloud raises an alert on a storage account with unusual access.
2. You check the Activity Log and confirm there was no recent control-plane change.
3. You inspect storage resource logs and see repeated reads from an unfamiliar IP.
4. You pivot to Entra sign-in logs and find the same user signed in from a new location.
5. You review Conditional Access and confirm MFA was satisfied, so the account may still be compromised.
6. You disable the account, revoke sessions, and rotate the storage key or SAS strategy as needed.
7. You preserve logs and create a Sentinel incident with bookmarks and comments.
8. You tune the detection or suppression if the activity was legitimate automation.

Another example:

1. Sentinel creates an incident from multiple alerts tied to a suspicious admin workstation.
2. You assign ownership, add comments, and bookmark the key events.
3. Defender for Endpoint shows PowerShell execution and lateral movement indicators.
4. You isolate the device, collect the investigation package, and block the malicious hash.
5. You check Entra audit logs for new role assignments or app consent.
6. You close the incident only after confirming containment and documenting the timeline.

**Exam focus reminders**

- Know the difference between **control-plane** and **data-plane** logs.
- Know which product provides the response action you want.
- Use Entra sign-in and audit logs for identity investigations.
- Use Sentinel for incident management, bookmarks, hunting, and automation.
- Use Defender for Endpoint for device isolation, live response, and investigation packages.
- Use Defender for Cloud secure score and recommendations to prioritize hardening.
- Use alert suppression and rule tuning to reduce noise without hiding real threats.
- Preserve evidence before destructive remediation whenever possible.
