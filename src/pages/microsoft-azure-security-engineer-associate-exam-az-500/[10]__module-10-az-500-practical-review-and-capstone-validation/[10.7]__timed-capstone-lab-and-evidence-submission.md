## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Timed capstone lab purpose**

A timed capstone lab tests whether you can design, implement, validate, and document Azure security controls under pressure. For AZ-500-style practice, the goal is not only to configure services correctly, but also to prove that the controls work and to capture evidence that is easy to review later.

In a real lab or assessment workflow, you usually need to:

- Interpret the scenario and identify the security outcome.
- Implement the minimum viable change set.
- Validate the result from the Azure control plane and, when relevant, from the data plane.
- Capture screenshots, CLI output, and configuration details.
- Submit evidence in a way that is traceable and reproducible.

A strong submission shows both the *what* and the *why*: what you changed, why it satisfies the requirement, and how you verified it.

**Planning the timed workflow**

Time pressure changes how you work. You should treat the lab like an incident-response checklist: read once, plan once, execute once, verify once. Avoid exploratory clicking unless the task is unclear, because portal navigation can consume time quickly.

A practical approach is:

1. Read all tasks before making changes.
2. Group tasks by dependency, such as identity, network, data protection, and monitoring.
3. Start with foundational settings that other controls depend on.
4. Validate each change immediately.
5. Record evidence as you go, not at the end.

Useful habits for timed work:

- Keep a scratchpad of resource names, IDs, and scopes.
- Use consistent naming so evidence is easy to match to tasks.
- Prefer CLI for repeatable checks and portal for visual confirmation when needed.
- Save command output to files when the lab allows it.

**Common AZ-500 lab domains**

Timed capstone labs often combine multiple security domains. You should expect to move between identity, network, governance, and monitoring controls.

- **Identity and access**
  - Role-based access control (RBAC)
  - Privileged Identity Management (PIM)
  - Conditional Access
  - Managed identities
- **Network security**
  - Network security groups (NSGs)
  - Azure Firewall
  - Private endpoints
  - DNS considerations
- **Data protection**
  - Key Vault
  - Storage account security
  - Encryption settings
  - Secret and key access policies or RBAC
- **Governance and compliance**
  - Azure Policy
  - Resource locks
  - Security recommendations
- **Monitoring and response**
  - Microsoft Defender for Cloud
  - Log Analytics
  - Microsoft Sentinel
  - Diagnostic settings

These areas often overlap. For example, a private endpoint change may require DNS updates, and a Key Vault access change may require both RBAC and network access validation.

**Evidence strategy**

Evidence should prove that the configuration exists and that it behaves as intended. A good evidence set usually includes:

- A before/after comparison when the task involves remediation.
- The exact resource scope, such as subscription, resource group, or resource ID.
- A screenshot or CLI output showing the final setting.
- A validation step that demonstrates the control is effective.
- A timestamp or sequence that shows the work happened within the lab window.

Use evidence that is specific and reproducible. For example, if you enable a diagnostic setting, capture both the setting itself and a log destination check. If you assign a role, capture the assignment and the scope inheritance context.

**CLI-first validation patterns**

Azure CLI is often the fastest way to verify state. Use it to confirm settings, list assignments, and inspect resource properties. When possible, query only the fields you need.

```bash
az account show --output table
az group show --name rg-security-lab --output table
az resource show --ids /subscriptions/<subId>/resourceGroups/rg-security-lab/providers/Microsoft.KeyVault/vaults/kv-secure --query "{name:name, location:location, id:id}" --output jsonc
```

For repeatable evidence, redirect output to a file:

```bash
az role assignment list \
  --scope /subscriptions/<subId>/resourceGroups/rg-security-lab \
  --output jsonc > role-assignments.json

az policy assignment list \
  --scope /subscriptions/<subId> \
  --output table > policy-assignments.txt
```

When you need to inspect a resource, prefer the resource-specific command rather than assuming a generic property path. Azure resource schemas vary by service and API version.

**Identity and access evidence**

Identity tasks often require you to prove both assignment and effective access. A role assignment at a resource group can be inherited by child resources, so do not assume that a resource-level query alone tells the full story.

For RBAC, verify:

- The principal identity, such as a user, group, or service principal.
- The assignment scope, such as subscription, resource group, or resource.
- The role definition, such as Reader, Contributor, or a custom role.
- Whether the assignment is inherited from a parent scope.

Example validation:

```bash
az role assignment list \
  --assignee <user-upn-or-object-id> \
  --all \
  --output table
```

Use `--all` when you need inherited assignments and assignments from parent scopes. If you only query a narrow scope, you may miss effective permissions granted higher in the hierarchy.

For PIM, evidence usually comes from the portal or Microsoft Graph-backed views showing eligible versus active assignments. Capture the activation duration, justification, and MFA (multifactor authentication) requirement if the scenario includes them.

**Key Vault validation**

Key Vault tasks commonly involve access control mode, secret permissions, and network restrictions. Be careful to distinguish control-plane access from data-plane access.

Typical checks include:

- Whether the vault uses Azure RBAC or access policies.
- Whether the intended principal can read secrets or keys.
- Whether public network access is restricted when required.
- Whether private endpoint connectivity is configured correctly.

Useful commands:

```bash
az keyvault show \
  --name kv-secure \
  --query "{name:name, resourceGroup:resourceGroup, enableRbacAuthorization:properties.enableRbacAuthorization}" \
  --output jsonc

az keyvault secret list \
  --vault-name kv-secure \
  --output table
```

If the lab requires network restriction validation, do not rely on a generic `curl -I` test alone. Key Vault and other Azure services may require the correct FQDN, DNS resolution, and service-specific authentication or protocol behavior. For private endpoint scenarios, validate:

- Private DNS zone linkage.
- Name resolution to a private IP.
- Successful access from an allowed network path.
- Failure from a disallowed public path, if the scenario expects it.

Example DNS and connectivity checks:

```bash
nslookup kv-secure.vault.azure.net
az network private-endpoint show \
  --name pe-kv-secure \
  --resource-group rg-security-lab \
  --query "{name:name, privateLinkServiceConnections:privateLinkServiceConnections}" \
  --output jsonc
```

**Storage security validation**

Storage account tasks often involve secure transfer, public access restrictions, shared key access, and network rules. Public access behavior can depend on account configuration and container-level settings, so validate both the account and the container.

Useful checks:

```bash
az storage account show \
  --name stsecurelab \
  --resource-group rg-security-lab \
  --query "{name:name, allowBlobPublicAccess:allowBlobPublicAccess, minimumTlsVersion:minimumTlsVersion, publicNetworkAccess:publicNetworkAccess}" \
  --output jsonc

az storage container show \
  --account-name stsecurelab \
  --name logs \
  --auth-mode login \
  --output jsonc
```

If the task is about blocking anonymous access, confirm:

- The storage account setting that controls blob public access.
- The container access level.
- Whether shared key access is disabled, if required by the scenario.
- Whether network rules or private endpoints are in place.

A secure configuration often pairs storage hardening with diagnostic logging so you can prove access attempts are tracked.

**Network security evidence**

Network tasks usually require you to prove that traffic is allowed or denied as intended. A generic endpoint test is often not enough. You should validate the exact service FQDN, the DNS result, and the path taken.

For example, if you configure a private endpoint:

- Confirm the private endpoint exists.
- Confirm the private DNS zone is linked to the VNet.
- Confirm the service name resolves to a private IP from the test subnet.
- Confirm the service is reachable only from the intended network path.

Example checks:

```bash
az network private-endpoint list \
  --resource-group rg-security-lab \
  --output table

az network private-dns zone list \
  --resource-group rg-security-lab \
  --output table

nslookup <service-name>.privatelink.<service-domain>
```

For NSGs, verify both the rule and the effective behavior:

```bash
az network nsg rule list \
  --resource-group rg-security-lab \
  --nsg-name nsg-app \
  --output table
```

If the lab includes a VM, you can validate connectivity from inside the VM using tools such as `Test-NetConnection` on Windows or `nc`/`curl` on Linux, but choose the tool that matches the protocol and service.

**Policy and compliance evidence**

Azure Policy tasks require careful scope handling. Compliance results are evaluated at the scope where the assignment applies, and inherited scope can affect the outcome. A resource-level check alone may not show the full compliance picture.

Useful commands:

```bash
az policy assignment list \
  --scope /subscriptions/<subId> \
  --output table

az policy state list \
  --scope /subscriptions/<subId> \
  --filter "PolicyAssignmentId eq '/subscriptions/<subId>/providers/Microsoft.Authorization/policyAssignments/<assignmentName>'" \
  --output table
```

If you need to validate a specific resource, include the correct scope and filter rather than assuming a single `--resource` parameter is enough. For evidence, capture:

- The policy assignment.
- The initiative or definition name.
- The compliance state.
- The affected resource or resource group.

A remediation task may also require a policy remediation deployment. In that case, show both the assignment and the remediation result.

**Microsoft Defender for Cloud and Sentinel evidence**

Security monitoring tasks often ask you to enable plans, configure alerts, or route logs. For Defender for Cloud, capture the relevant recommendation or plan status and the resource scope. For Sentinel, evidence usually includes the workspace connection, analytics rule, or incident generation path.

Be careful with command availability because Azure CLI support can vary by extension version. If a Sentinel command is not available in your environment, use the portal or the Azure Resource Manager resource type for validation instead of forcing an outdated command.

Practical validation examples:

```bash
az monitor log-analytics workspace show \
  --resource-group rg-security-lab \
  --workspace-name law-secure \
  --output jsonc

az monitor diagnostic-settings list \
  --resource /subscriptions/<subId>/resourceGroups/rg-security-lab/providers/Microsoft.KeyVault/vaults/kv-secure \
  --output jsonc
```

For Sentinel, confirm:

- The Log Analytics workspace is connected.
- Data connectors are enabled where required.
- Analytics rules exist and are enabled.
- Incidents or alerts are generated when the scenario expects them.

**Evidence capture workflow**

A repeatable capture workflow reduces mistakes under time pressure. Use a consistent naming pattern for files and screenshots.

1. Create a folder for the lab attempt.
2. Save command output for each major task.
3. Capture portal screenshots only after the configuration is confirmed.
4. Annotate each file with the task number or objective.
5. Review the evidence package before submission.

Example folder structure:

```text
lab-evidence/
  01-identity/
  02-network/
  03-data-protection/
  04-policy/
  05-monitoring/
```

Example naming pattern:

```text
task-02-keyvault-rbac-before.json
task-02-keyvault-rbac-after.json
task-03-private-endpoint-dns.png
task-04-policy-compliance.txt
```

If the submission portal accepts a single archive, compress the folder only after you verify that all files open correctly.

**Practical submission checklist**

Before you submit, confirm that each task has proof and that the proof matches the requirement.

- The resource name in the evidence matches the lab prompt.
- The scope in the evidence matches the intended scope.
- The setting is visible in the final state.
- The validation step proves behavior, not just configuration.
- The evidence is readable and complete.
- The submission includes all required files or screenshots.

A quick final review should answer these questions:

- Did you change the right resource?
- Did you validate the right control?
- Did you capture enough context for a reviewer to trust the result?
- Could someone reproduce the result from your evidence?

**Troubleshooting under time pressure**

When a validation fails, isolate the problem by checking identity, network, and service configuration in that order. Many Azure security issues are caused by one of these layers rather than the feature itself.

Common causes include:

- Wrong scope for a role assignment or policy.
- Missing DNS linkage for a private endpoint.
- Conflicting network rules and service endpoints.
- Using the wrong principal when testing access.
- Waiting for propagation after a configuration change.

If a change does not appear immediately, allow for propagation delay and recheck the exact resource state. For example, RBAC and policy changes may take time to become effective.

**High-value lab commands**

These commands are often useful during a timed capstone lab because they help you inspect state quickly.

```bash
az account set --subscription <subId>
az group list --output table
az resource list --resource-group rg-security-lab --output table
az role assignment list --scope /subscriptions/<subId> --output table
az policy assignment list --scope /subscriptions/<subId> --output table
az monitor diagnostic-settings list --resource <resource-id> --output jsonc
```

Use these as starting points, then switch to service-specific commands for the exact control you are validating.

**What a strong submission looks like**

A strong AZ-500 capstone submission is concise, accurate, and traceable. It shows that you can secure Azure resources and prove the result without unnecessary steps.

It usually includes:

- Correct implementation of the requested control.
- Clear evidence of the final configuration.
- Validation that demonstrates the control works as intended.
- Notes or filenames that map directly to the task list.
- No ambiguous screenshots or incomplete command output.

The best evidence packages make it easy for a reviewer to confirm success without guessing. That is the standard to aim for in a timed lab.
