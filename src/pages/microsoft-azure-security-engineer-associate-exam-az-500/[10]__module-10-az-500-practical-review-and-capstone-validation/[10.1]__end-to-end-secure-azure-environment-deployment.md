## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**End-to-end secure deployment overview**

A secure Azure environment is not a single control. You design it as a layered system that starts with governance and identity, then adds network isolation, workload hardening, encryption, and continuous monitoring. For AZ-500, you should think in terms of how controls work together across subscriptions, management groups, resource groups, and individual services.

A practical secure baseline usually includes:

- A clear **landing zone** structure with management groups, subscriptions, and naming/tagging standards.
- Centralized identity and privileged access controls.
- Hub-and-spoke or similarly segmented network design.
- Private access to PaaS services where possible.
- Policy-driven guardrails and continuous compliance monitoring.
- Logging, alerting, and automated response.

**Planning the landing zone**

Before you deploy workloads, define the operating model. This is where many secure designs succeed or fail. If you skip planning, you end up retrofitting controls later, which is harder and riskier.

Key planning items include:

- **Subscription design**
  - Separate platform services from application workloads.
  - Use dedicated subscriptions for shared services, connectivity, identity, and management when appropriate.
  - Isolate production from nonproduction to reduce blast radius.
- **Management group hierarchy**
  - Use management groups to apply policy and RBAC at scale.
  - Place subscriptions under the correct hierarchy before deployment.
  - Keep the hierarchy simple enough to manage but structured enough to enforce standards.
- **Naming and tagging standards**
  - Standardize names for resources, resource groups, and subscriptions.
  - Use tags such as `Environment`, `Owner`, `DataClassification`, and `CostCenter`.
  - Use tags for policy targeting, reporting, and automation.
- **Resource organization**
  - Group resources by application and lifecycle.
  - Keep shared services separate from app-specific resources.
  - Use resource groups as a deployment and lifecycle boundary, not as a security boundary.

A common pattern is to define the platform first, then deploy application landing zones into governed subscriptions. This supports repeatable deployments and policy enforcement.

**Governance and policy-first deployment**

Azure Policy is a core control for secure deployment. It helps you prevent insecure configurations, audit drift, and enforce standards automatically.

Use policy to:

- Deny public IP creation where not allowed.
- Require tags on resources.
- Enforce approved locations.
- Audit or deny insecure storage settings.
- Require private endpoints for sensitive services.
- Deploy diagnostic settings automatically.

Policy concepts to remember:

- **Initiatives** group related policies into a single assignment.
- **Assignments** scope policies to management groups, subscriptions, or resource groups.
- **Effects** determine behavior:
  - `Deny` blocks noncompliant deployments.
  - `Audit` records noncompliance.
  - `DeployIfNotExists` can remediate missing settings.
  - `Modify` can change properties during deployment.

Example policy assignment pattern:

```bash
az policy assignment create \
  --name "deny-public-ip" \
  --display-name "Deny Public IPs" \
  --scope "/subscriptions/<subscription-id>" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>"
```

Use policy as code when possible. In mature environments, policy definitions and assignments are deployed through templates or pipelines so that governance is versioned and repeatable.

> Resource locks are not a substitute for policy. Use locks to prevent accidental deletion or modification, but use policy to enforce configuration standards.

**Identity and privileged access**

Identity is the first security boundary in Azure. Secure deployment starts with strong authentication, least privilege, and controlled elevation.

Core identity controls:

- Use **Microsoft Entra ID** for authentication and authorization.
- Require **MFA** (multifactor authentication) for privileged users.
- Use **Privileged Identity Management (PIM)** for just-in-time elevation.
- Prefer **managed identities** for workloads instead of secrets.
- Use **role-based access control (RBAC)** with the least privilege needed.

Important RBAC guidance:

- Assign roles at the narrowest practical scope.
- Use built-in roles when they fit; create custom roles only when necessary.
- Separate human admin roles from workload identities.
- Review role assignments regularly.

For managed identities, remember that the identity is represented by a service principal in Entra ID. When assigning permissions, you need the identity’s principal/object ID, not the Azure resource ID.

Example: grant a managed identity access to a Key Vault data plane using Azure RBAC:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-principal-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope "/subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vault-name>"
```

Use PIM for roles such as:

- Subscription Owner
- User Access Administrator
- Security Admin
- Key Vault Administrator
- Network Contributor

This reduces standing privilege and improves auditability.

**Network segmentation and traffic control**

A secure Azure deployment usually uses segmentation to limit lateral movement and reduce exposure. The most common enterprise pattern is hub-and-spoke, but the exact topology depends on your organization.

Typical network design goals:

- Keep shared services in a central hub.
- Place workloads in spoke VNets.
- Control east-west traffic with network security groups and routing.
- Force inspection of outbound traffic through Azure Firewall or another security appliance.
- Use private endpoints for PaaS access when possible.

Core network controls:

- **Network Security Groups (NSGs)**
  - Filter traffic at subnet or NIC level.
  - Use them for coarse segmentation and workload isolation.
- **Azure Firewall**
  - Centralized stateful firewall for network and application traffic.
  - Supports network rules, application rules, and DNAT.
- **User-defined routes (UDRs)**
  - Force traffic through the hub or firewall.
  - Commonly used to send outbound internet traffic to Azure Firewall.
- **Private Link / private endpoints**
  - Expose PaaS services privately inside your VNet.
  - Reduce dependence on public endpoints.
- **DDoS Protection**
  - Use for internet-facing workloads that need enhanced protection.

A common forced-tunneling pattern:

1. Spoke subnet routes `0.0.0.0/0` to Azure Firewall in the hub.
2. Firewall inspects outbound traffic.
3. Firewall allows only approved destinations and ports.
4. Private DNS resolves private endpoint names to private IPs.

Example route table entry:

```bash
az network route-table route create \
  --resource-group <rg> \
  --route-table-name <rt-name> \
  --name DefaultToFirewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

**Azure Firewall policy and rule design**

Azure Firewall is often the central egress and inspection control in a secure Azure environment. Use **Firewall Policy** to manage rules consistently across firewalls and regions.

Rule types to know:

- **Application rules**
  - Control outbound HTTP, HTTPS, and MSSQL traffic by FQDN.
  - Useful for web destinations and SaaS endpoints.
- **Network rules**
  - Control traffic by source, destination IP, port, and protocol.
  - Useful for non-HTTP protocols and private IP destinations.
- **DNAT rules**
  - Translate public IP traffic to private resources.
  - Use carefully and only when direct inbound access is required.

Design guidance:

- Prefer application rules for outbound web access.
- Use network rules for private IP-to-IP flows.
- Keep rule collections organized by application or function.
- Log firewall traffic and review denied flows regularly.

Example Azure Firewall policy structure:

```json
{
  "ruleCollectionGroups": [
    {
      "name": "app-egress",
      "priority": 100,
      "ruleCollections": [
        {
          "name": "allow-microsoft-updates",
          "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
          "action": { "type": "Allow" },
          "rules": [
            {
              "name": "allow-windows-update",
              "ruleType": "ApplicationRule",
              "sourceAddresses": ["10.1.0.0/24"],
              "protocols": [{ "protocolType": "Https", "port": 443 }],
              "targetFqdns": ["*.windowsupdate.com"]
            }
          ]
        }
      ]
    }
  ]
}
```

**Secure PaaS hardening**

Many AZ-500 scenarios involve securing platform services rather than only IaaS. The main idea is to remove public exposure, enforce identity-based access, and require encryption.

**Azure Key Vault**

Use Key Vault for secrets, keys, and certificates.

Best practices:

- Use RBAC or access policies consistently based on the vault’s configured authorization model.
- Prefer private endpoints for vault access.
- Disable public network access when the design allows it.
- Enable soft delete and purge protection.
- Use diagnostic logging.
- Restrict who can manage keys versus who can read secrets.

Key Vault DNS behavior is often tested. A client typically uses `kv-secure.vault.azure.net`, which resolves through a CNAME chain to the private endpoint name when private DNS is configured, such as `kv-secure.privatelink.vaultcore.azure.net`, and then to the private IP.

Example private DNS zone link:

```bash
az network private-dns zone create \
  --resource-group <rg> \
  --name privatelink.vaultcore.azure.net

az network private-dns link vnet create \
  --resource-group <rg> \
  --zone-name privatelink.vaultcore.azure.net \
  --name link-spoke1 \
  --virtual-network <vnet-id> \
  --registration-enabled false
```

**App Service**

Secure App Service deployments should reduce public exposure and protect secrets.

Use these controls:

- Require HTTPS only.
- Disable FTP if not needed.
- Use managed identity for secret retrieval.
- Store secrets in Key Vault rather than app settings when possible.
- Restrict inbound access with access restrictions or private endpoints.
- Use TLS certificates for custom domains.

Example App Service Key Vault reference:

```text
@Microsoft.KeyVault(SecretUri=https://kv-secure.vault.azure.net/secrets/app-db-password/)
```

In practice, App Service Key Vault references may require versioned or versionless secret URIs depending on identity permissions and configuration, so validate the exact deployment behavior in your environment.

**Azure SQL Database**

For SQL Database, secure the data plane and the network path.

Use:

- Azure AD authentication where possible.
- Private endpoint access for private connectivity.
- Firewall rules only when public access is required.
- Transparent Data Encryption (TDE) for encryption at rest.
- Auditing and threat detection features.
- Minimal privileged SQL logins.

If you need customer-managed keys, integrate SQL encryption with Key Vault and ensure the key lifecycle is managed carefully.

**Azure Container Registry**

Secure ACR by:

- Disabling public network access when possible.
- Using private endpoints.
- Enabling admin user only when absolutely necessary.
- Using managed identity or service principals with least privilege.
- Restricting image push and pull permissions.
- Scanning images through your security pipeline.

**Encryption at rest and in transit**

Encryption is a baseline requirement, not an optional enhancement. Secure Azure deployments should address both data at rest and data in transit.

At rest:

- Use platform-managed encryption by default where appropriate.
- Use customer-managed keys (CMK) when compliance or control requirements demand it.
- Protect VM disks with disk encryption where required.
- Use TDE for SQL Database.
- Use encryption for storage accounts and managed disks.

In transit:

- Require TLS 1.2 or later where supported.
- Disable insecure protocols and ciphers.
- Use HTTPS-only settings for web apps and storage endpoints.
- Use certificates from trusted sources or Key Vault-managed certificates.
- Prefer private connectivity plus TLS for sensitive services.

Example VM disk encryption pattern:

```bash
az vm encryption enable \
  --resource-group <rg> \
  --name <vm-name> \
  --disk-encryption-keyvault <keyvault-name> \
  --volume-type All
```

For many modern workloads, platform encryption plus private networking and strong identity controls is sufficient. Use additional encryption layers when compliance or data sensitivity requires them.

**Virtual machines and management access**

For IaaS workloads, secure the guest OS, the management plane, and the network path.

Recommended controls:

- Use Azure Bastion for browser-based management access.
- Restrict inbound RDP and SSH.
- Apply NSGs to limit management ports.
- Use Just-In-Time VM access in Defender for Cloud.
- Patch regularly and monitor for vulnerabilities.
- Use disk encryption and secure boot features where supported.

Azure Bastion reduces the need for public RDP or SSH exposure for management access, but it does not eliminate every scenario where direct inbound access might still be required for specialized workflows.

Example NSG rule to deny inbound RDP from the internet:

```bash
az network nsg rule create \
  --resource-group <rg> \
  --nsg-name <nsg-name> \
  --name Deny-RDP-Internet \
  --priority 100 \
  --direction Inbound \
  --access Deny \
  --protocol Tcp \
  --source-address-prefixes Internet \
  --source-port-ranges "*" \
  --destination-address-prefixes "*" \
  --destination-port-ranges 3389
```

**Defender for Cloud deployment**

Microsoft Defender for Cloud is the central security posture management and workload protection service in Azure. It helps you assess risk, prioritize remediation, and protect resources.

Core capabilities:

- **Secure score**
  - Measures your security posture.
  - Higher scores indicate better alignment with recommendations.
- **Regulatory compliance**
  - Maps controls to standards and frameworks.
  - Helps you track compliance gaps.
- **Security recommendations**
  - Prioritized actions to improve posture.
  - Often linked to specific resources and configurations.
- **Just-In-Time VM access**
  - Temporarily opens management ports only when needed.
- **Defender plans**
  - Extend threat protection to servers, SQL, storage, containers, and other services.

Deployment workflow:

1. Enable Defender for Cloud at the subscription or management group scope.
2. Turn on the relevant Defender plans for your workloads.
3. Review secure score recommendations.
4. Remediate high-impact findings first.
5. Configure JIT access for eligible VMs.
6. Monitor compliance and alerts continuously.

JIT VM access example conceptually:

- A user requests access to port 22 or 3389.
- Defender for Cloud opens the port for a limited time.
- The access window expires automatically.

This is useful when you want administrative access without leaving management ports open all the time.

**Microsoft Sentinel deployment and configuration**

Microsoft Sentinel is the SIEM (Security Information and Event Management) and SOAR (Security Orchestration, Automation, and Response) layer for Azure security operations.

Secure deployment steps:

- Create or select a Log Analytics workspace.
- Enable Sentinel on that workspace.
- Connect data sources through data connectors.
- Configure analytics rules for detection.
- Add automation rules for triage and response.
- Use playbooks for automated remediation.

Key Sentinel components:

- **Data connectors**
  - Ingest logs from Azure, Microsoft 365, endpoints, firewalls, and third-party sources.
- **Analytics rules**
  - Detect suspicious activity and generate incidents.
- **Automation rules**
  - Automatically tag, assign, or close incidents based on conditions.
- **Playbooks**
  - Logic Apps workflows that respond to incidents.

Example deployment flow:

1. Send Azure Activity Logs, Key Vault logs, firewall logs, and Entra ID logs to Log Analytics.
2. Enable Sentinel on the workspace.
3. Add analytics rules for privileged role changes, suspicious sign-ins, and denied firewall traffic.
4. Create a playbook to notify the SOC and isolate a compromised resource if needed.

A simple playbook action might send a Teams or email notification, create a ticket, or disable a user account through an approved workflow.

**Logging and monitoring design**

Monitoring should be designed into the environment, not added later. You need logs from identity, network, compute, and PaaS layers.

Common log sources:

- Azure Activity Log
- Entra ID sign-in and audit logs
- Key Vault diagnostic logs
- Azure Firewall logs
- NSG flow logs
- VM guest logs
- App Service logs
- SQL auditing logs
- Storage diagnostic logs

Important guidance:

- Send logs to a central workspace or SIEM.
- Retain logs according to compliance requirements.
- Use alerts for high-risk events, not just volume.
- Correlate identity, network, and resource logs for investigations.

Be careful with log queries. Diagnostic schemas vary by service, category, and destination. For example, Key Vault logs may appear in different tables or resource-specific schemas depending on how diagnostics are configured, so do not assume one universal KQL pattern fits every environment.

**Practical KQL examples**

Use KQL (Kusto Query Language) to investigate security events, but adapt queries to your actual tables and schema.

Example: find failed sign-ins in Entra ID logs:

```kusto
SigninLogs
| where ResultType != 0
| project TimeGenerated, UserPrincipalName, AppDisplayName, IPAddress, ResultDescription
| order by TimeGenerated desc
```

Example: review Azure Activity operations on a resource group:

```kusto
AzureActivity
| where ResourceGroup == "<rg-name>"
| project TimeGenerated, OperationNameValue, ActivityStatusValue, Caller, ResourceProviderValue
| order by TimeGenerated desc
```

Example: inspect firewall denies if your logs are sent to a compatible table:

```kusto
AzureDiagnostics
| where Category == "AzureFirewallNetworkRule"
| where msg_s has "Deny"
| project TimeGenerated, msg_s, SourceIp_s, DestinationIp_s, DestinationPort_s
| order by TimeGenerated desc
```

Use these as patterns, not universal truths. Always validate the table names and fields in your workspace.

**Resource protection and change control**

Secure environments need protection against accidental or unauthorized changes.

Use:

- **Resource locks**
  - `CanNotDelete` prevents deletion.
  - `ReadOnly` prevents most modifications.
- **RBAC separation**
  - Limit who can change network, identity, and security resources.
- **Change control**
  - Use pipelines and approvals for production changes.
- **Backup and recovery**
  - Protect critical data and configuration.

Example lock:

```bash
az lock create \
  --name ProtectProdRG \
  --lock-type CanNotDelete \
  --resource-group <rg>
```

Locks are useful for critical resources such as Key Vaults, firewalls, and production resource groups, but they should not block legitimate operations like patching or controlled updates.

**End-to-end deployment example**

A secure deployment often combines identity, network, policy, and monitoring in one design.

Example architecture flow:

1. Create management groups for platform and application subscriptions.
2. Apply policy to deny public IPs, require tags, and enforce diagnostics.
3. Configure PIM and least-privilege RBAC for admins and operators.
4. Build a hub VNet with Azure Firewall, Bastion, and shared DNS.
5. Build spoke VNets for applications and data services.
6. Add UDRs so spoke traffic routes through the firewall.
7. Deploy private endpoints for Key Vault, SQL Database, and Storage.
8. Configure private DNS zones and VNet links.
9. Enable Defender for Cloud and JIT VM access.
10. Send logs to Log Analytics and onboard Sentinel.
11. Add analytics rules and automation for suspicious activity.
12. Protect critical resources with locks and backup policies.

Example hub-and-spoke routing and private endpoint pattern:

```text
User/Admin -> Azure Bastion -> VM in Spoke
App VM -> UDR -> Azure Firewall -> Approved Internet or Private Endpoint
App Service -> Managed Identity -> Key Vault Private Endpoint
SQL Database -> Private Endpoint -> Private DNS
Logs -> Log Analytics -> Sentinel
```

This pattern reduces public exposure while preserving manageability and observability.

**Common exam distinctions**

AZ-500 questions often test whether you can distinguish similar controls.

Watch for these differences:

- **Azure RBAC vs Kubernetes RBAC**
  - Azure RBAC controls Azure resource access.
  - Kubernetes RBAC controls access inside the cluster.
  - AKS can use Azure RBAC for Kubernetes authorization, but that is distinct from native Kubernetes RBAC.
- **Access policies vs Azure RBAC in Key Vault**
  - They are mutually exclusive for data-plane authorization when the vault is configured for one model or the other.
  - Do not assume both are active for the same data-plane request path.
- **Public network access vs public blob access**
  - Disabling public blob access does not disable all public network access.
  - Network exposure and anonymous data exposure are separate concerns.
- **Firewall rules vs NSGs**
  - NSGs filter subnet/NIC traffic.
  - Azure Firewall provides centralized inspection and logging.
- **Private endpoint vs service endpoint**
  - Private endpoint gives a private IP in your VNet.
  - Service endpoint keeps the service public but restricts access from selected VNets.
- **Defender for Cloud vs Sentinel**
  - Defender for Cloud focuses on posture and workload protection.
  - Sentinel focuses on detection, investigation, and response.

**Deployment checklist**

Use this checklist when designing a secure Azure environment:

- Define management groups, subscriptions, and naming standards.
- Apply policy for location, tags, diagnostics, and public exposure.
- Configure Entra ID, MFA, PIM, and least privilege.
- Build hub-and-spoke networking with firewall and UDRs.
- Use private endpoints and private DNS for sensitive PaaS services.
- Harden App Service, SQL Database, Storage, ACR, and Key Vault.
- Enable encryption at rest and in transit.
- Secure VM access with Bastion and JIT.
- Turn on Defender for Cloud and remediate recommendations.
- Onboard logs to Sentinel and create analytics rules.
- Protect critical resources with locks and backup controls.

**Quick reference commands**

Create a resource group:

```bash
az group create \
  --name rg-secure-prod \
  --location eastus
```

Create a storage account with blob public access disabled:

```bash
az storage account create \
  --name stsecureprod01 \
  --resource-group rg-secure-prod \
  --location eastus \
  --sku Standard_GRS \
  --kind StorageV2 \
  --allow-blob-public-access false
```

Remember that this does not disable public network access by itself. Use network rules or private endpoints for network isolation.

Create a private endpoint for Key Vault:

```bash
az network private-endpoint create \
  --name pe-kv-secure \
  --resource-group rg-secure-prod \
  --vnet-name vnet-spoke1 \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id "/subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vault-name>" \
  --group-id vault
```

**Final exam focus**

For AZ-500, focus on how you would secure a complete environment rather than memorizing isolated features. Expect scenarios that ask you to choose the right combination of governance, identity, network, encryption, monitoring, and response controls.

If a question describes a secure deployment, look for:

- The least-privilege identity model.
- Private connectivity instead of public exposure.
- Policy enforcement before deployment.
- Centralized logging and alerting.
- Workload-specific hardening for PaaS and IaaS.
- Clear separation of management, data, and application planes.
