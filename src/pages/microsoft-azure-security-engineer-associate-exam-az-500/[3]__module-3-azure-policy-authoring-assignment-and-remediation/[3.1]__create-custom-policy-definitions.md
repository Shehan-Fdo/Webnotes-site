## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Azure Policy overview**

Azure Policy is the governance service you use to evaluate and enforce rules across Azure resources. A **custom policy definition** expresses a rule in JSON that Azure Policy evaluates against resource requests and existing resources. You use custom definitions when built-in policies do not match your exact business, security, or compliance requirement.

A policy solution usually has three layers:

- **Policy definition**: the rule logic and parameters.
- **Policy assignment**: where and how the definition is applied, including scope and parameter values.
- **Initiative**: a policy set definition that groups multiple policy definitions under one assignment.

Use a custom definition when you need a precise control such as:

- Denying public IP creation on production workloads.
- Requiring tags like `CostCenter` or `DataClassification`.
- Restricting allowed SKUs for virtual machines or storage accounts.
- Enforcing diagnostic settings to a Log Analytics workspace.
- Deploying missing security configuration with `deployIfNotExists`.

**Definition, assignment, and initiative**

These three objects are commonly tested together because they solve different governance problems.

- **Policy definition**
  - Contains the `policyRule`, optional `parameters`, and metadata.
  - Reusable across many scopes.
  - Can be custom or built-in.
- **Policy assignment**
  - Binds a definition to a scope such as management group, subscription, resource group, or resource.
  - Supplies parameter values.
  - Can include exclusions, enforcement mode, and managed identity for remediation effects.
- **Initiative**
  - Groups multiple definitions into one governance package.
  - Useful when you want one assignment to enforce a baseline such as secure networking, tagging, and diagnostics together.
  - Can include both built-in and custom definitions.

A practical exam rule: if you need one control, use a definition; if you need to apply it somewhere, create an assignment; if you need a bundle of controls, create an initiative.

**Custom policy definition schema**

A policy definition is JSON and must follow the Azure Policy schema. The most important top-level elements are:

- `mode`
- `policyRule`
- `parameters`
- `metadata`

The `policyRule` section is required and contains:

- `if`: the condition to evaluate.
- `then`: the effect to apply when the condition matches.

A minimal structure looks like this:

```json
{
  "mode": "All",
  "parameters": {
    "allowedLocations": {
      "type": "Array",
      "metadata": {
        "displayName": "Allowed locations",
        "description": "The list of allowed Azure regions."
      }
    }
  },
  "policyRule": {
    "if": {
      "not": {
        "field": "location",
        "in": "[parameters('allowedLocations')]"
      }
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

Key schema points:

- **Parameters are declared in the definition** and referenced inside `policyRule` with `[parameters('name')]`.
- **Parameter values are supplied at assignment time**, not inside the definition.
- **Metadata** is where you store display names, descriptions, category, and version information.
- **Policy rules are evaluated as JSON expressions**, not as Azure Resource Manager template expressions.

Common parameter types include:

- `String`
- `Array`
- `Boolean`
- `Integer`
- `Object`

Use parameters when you want one definition to work across multiple teams or environments. For example, a single allowed-locations policy can be assigned to dev, test, and prod with different region lists.

**Mode selection**

The `mode` property controls which resource types and aliases the policy engine evaluates.

- **`All`**
  - Evaluates most resource types.
  - Use this for broad governance controls such as deny, audit, or tag enforcement.
  - Best default for many custom policies.
- **`Indexed`**
  - Focuses on resource types that support tags and location.
  - Commonly used for tag-related policies and some location-based controls.
  - Excludes some resource types, so it is not a universal choice.

Mode matters because some aliases and fields are only available in certain evaluation paths. If you choose the wrong mode, the policy may not evaluate the resource type you expect.

Practical guidance:

- Use `All` when you are unsure and the policy must cover many resource types.
- Use `Indexed` when the policy is specifically about tags or location and you want the narrower evaluation set.
- Verify alias support before authoring a rule that targets a specific property.

> A policy can be syntactically valid but still fail to evaluate as intended if the alias is not available for the selected mode or resource type.

**Common effects**

The `effect` determines what Azure Policy does when the `if` condition matches.

- **`deny`**
  - Blocks noncompliant create or update requests.
  - Use for hard security boundaries such as disallowing public IPs or unapproved regions.
- **`audit`**
  - Allows the request but marks it noncompliant.
  - Use for visibility before enforcement.
- **`auditIfNotExists`**
  - Audits when a related resource or configuration is missing.
  - Useful for checking diagnostics or dependent resources.
- **`deployIfNotExists`**
  - Deploys a missing resource or configuration after evaluation.
  - Common for diagnostic settings, security extensions, or baseline configuration.
- **`modify`**
  - Changes the request before it is committed.
  - Common for adding tags or normalizing properties.
- **`disabled`**
  - Turns off the effect without deleting the definition.

`append` exists in Azure Policy history, but for modern authoring you should generally think in terms of `modify` for request-time property changes. Do not rely on `append` as your primary design pattern.

**When to use each effect**

- Use **deny** when the resource must never exist in an unsafe state.
- Use **audit** when you need discovery first and enforcement later.
- Use **modify** when you want to correct a request automatically, such as adding a missing tag.
- Use **deployIfNotExists** when the policy must create or configure a dependent resource after deployment.
- Use **auditIfNotExists** when you only need to detect missing related configuration.

**Example: allowed locations**

This is a classic custom policy because it is easy to understand and widely used in governance baselines.

```json
{
  "mode": "All",
  "parameters": {
    "listOfAllowedLocations": {
      "type": "Array",
      "metadata": {
        "displayName": "Allowed locations",
        "description": "The list of allowed Azure regions."
      }
    }
  },
  "policyRule": {
    "if": {
      "not": {
        "field": "location",
        "in": "[parameters('listOfAllowedLocations')]"
      }
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

Why this works:

- The policy checks the resource `location`.
- The `not` wrapper inverts the condition.
- The `deny` effect blocks creation outside the approved list.

Typical use cases:

- Keep production workloads in approved regions.
- Satisfy data residency requirements.
- Prevent accidental deployment in unsupported geographies.

**Example: required tag**

Tag enforcement is one of the most common governance patterns.

```json
{
  "mode": "Indexed",
  "parameters": {
    "tagName": {
      "type": "String",
      "metadata": {
        "displayName": "Tag name",
        "description": "The required tag name."
      }
    }
  },
  "policyRule": {
    "if": {
      "field": "[concat('tags[', parameters('tagName'), ']')]",
      "exists": "false"
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

This pattern is useful when you want to require a tag such as `Owner`, `CostCenter`, or `Environment`. In many environments, teams start with `audit` and later move to `deny` after they fix existing resources and update deployment pipelines.

**Example: allowed SKUs**

You can also restrict service tiers or sizes.

```json
{
  "mode": "All",
  "parameters": {
    "allowedSkus": {
      "type": "Array",
      "metadata": {
        "displayName": "Allowed SKUs",
        "description": "Approved SKU names."
      }
    }
  },
  "policyRule": {
    "if": {
      "not": {
        "field": "sku.name",
        "in": "[parameters('allowedSkus')]"
      }
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

Use this to control cost and standardize service tiers. For example, you might allow only `Standard_B2s` and `Standard_D2s_v5` for certain environments.

**Example: deny public IP on a resource type**

A security-focused policy often targets a specific resource property.

```json
{
  "mode": "All",
  "policyRule": {
    "if": {
      "allOf": [
        {
          "field": "type",
          "equals": "Microsoft.Network/networkInterfaces"
        },
        {
          "field": "Microsoft.Network/networkInterfaces/ipConfigurations[*].publicIPAddress.id",
          "exists": "true"
        }
      ]
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

This kind of policy is useful for reducing exposure of workloads that should remain private. In practice, you must confirm the alias exists for the target resource type and API surface.

**Example: modify to add a tag**

`modify` is the preferred effect when you want to add or update request properties automatically.

```json
{
  "mode": "Indexed",
  "parameters": {
    "tagName": {
      "type": "String"
    },
    "tagValue": {
      "type": "String"
    }
  },
  "policyRule": {
    "if": {
      "field": "[concat('tags[', parameters('tagName'), ']')]",
      "exists": "false"
    },
    "then": {
      "effect": "modify",
      "details": {
        "roleDefinitionIds": [
          "/providers/Microsoft.Authorization/roleDefinitions/4a9ae827-6dc8-4573-8ac7-8239d42aa03f"
        ],
        "operations": [
          {
            "operation": "add",
            "field": "[concat('tags[', parameters('tagName'), ']')]",
            "value": "[parameters('tagValue')]"
          }
        ]
      }
    }
  }
}
```

Important points:

- The policy assignment needs a **managed identity**.
- That identity must have the required **role assignments** to make the change.
- The definition itself does not hold the identity; the assignment does.

**deployIfNotExists and remediation**

`deployIfNotExists` is used when Azure Policy should deploy missing configuration after evaluation. This is common for security baselines such as diagnostic settings, monitoring agents, or encryption-related configuration.

A valid `deployIfNotExists` policy usually includes:

- `type`
- `name`
- `existenceCondition`
- `roleDefinitionIds`
- `deployment` template

The policy checks whether the related resource exists. If not, Azure Policy uses the assignment’s managed identity to deploy the missing configuration.

Conceptually:

- The policy evaluates the target resource.
- It checks for a related resource or child configuration.
- If the related item is missing, Azure Policy triggers a deployment.
- A **remediation task** is often needed to apply the policy to existing noncompliant resources.

Use remediation tasks when:

- You assign a `deployIfNotExists` or `modify` policy to resources that already exist.
- You want Azure Policy to fix current noncompliant resources, not just future requests.

Do not assume remediation is universal. Some policies only audit or deny, and some noncompliance states require manual correction.

**Policy assignment and managed identity**

For effects like `modify` and `deployIfNotExists`, the assignment is where identity and permissions matter.

- Create the policy definition first.
- Assign the policy at the correct scope.
- Enable a managed identity on the assignment.
- Grant that identity the required Azure role permissions.
- Create a remediation task if you need to fix existing resources.

This separation is important:

- The **definition** describes what to do.
- The **assignment** decides where to do it and with what identity.
- The **role assignment** gives the identity permission to act.

**Scope and where you can create definitions**

Custom policy definitions can be created at several scopes:

- **Management group**
- **Subscription**
- **Resource group**

Scope choice affects reuse and governance reach.

- Use **management group** scope for enterprise-wide standards.
- Use **subscription** scope for subscription-specific controls.
- Use **resource group** scope for local or project-specific rules.

A higher scope definition can be assigned lower in the hierarchy, but the definition itself should be created where it best matches your governance model.

**Versioning and lifecycle**

Policy definitions evolve over time. Treat them like governed code.

Good practices:

- Store policy JSON in source control.
- Use meaningful names and version metadata.
- Update descriptions when the rule changes.
- Test in a nonproduction subscription before broad rollout.
- Review the impact on existing assignments before tightening a rule.

When you update a definition:

- Existing assignments usually pick up the new logic automatically.
- Compliance state may change after the next evaluation cycle.
- A stricter rule can make previously compliant resources noncompliant.
- A looser rule can reduce enforcement without changing the assignment.

Use metadata to track versioning, for example:

```json
"metadata": {
  "category": "Security",
  "version": "1.2.0",
  "preview": false
}
```

**Authoring workflow**

A practical workflow helps avoid broken policies and exam confusion.

1. Identify the control objective.
2. Choose the right effect: `deny`, `audit`, `modify`, or `deployIfNotExists`.
3. Find the correct alias for the target property.
4. Decide the best `mode`.
5. Define parameters for values that vary by environment.
6. Write and validate the JSON.
7. Create the definition.
8. Assign it at the correct scope.
9. Add a managed identity if the effect requires one.
10. Create remediation tasks when needed.
11. Review compliance results and refine the rule.

**Azure CLI examples**

Use Azure CLI to create and manage policy definitions and assignments. The exact command shape can vary by CLI version, so validate against your installed version and the current help output.

Create a custom policy definition from a JSON file:

```bash
az policy definition create \
  --name allowed-locations \
  --display-name "Allowed locations" \
  --description "Restrict deployments to approved regions" \
  --rules allowed-locations-rule.json \
  --params allowed-locations-params.json \
  --mode All
```

If your CLI version expects a single definition file, use the file-based pattern supported by that version. Always check:

```bash
az policy definition create -h
```

Assign the policy to a subscription:

```bash
az policy assignment create \
  --name assign-allowed-locations \
  --display-name "Assign allowed locations" \
  --scope /subscriptions/<subscription-id> \
  --policy allowed-locations \
  --params '{
    "listOfAllowedLocations": {
      "value": ["eastus", "westus2"]
    }
  }'
```

List policy definitions:

```bash
az policy definition list --query "[?policyType=='Custom'].[name, displayName]" -o table
```

List policy aliases for a namespace:

```bash
az policy alias list --namespace Microsoft.Compute -o table
```

Or inspect provider metadata and filter aliases:

```bash
az provider show --namespace Microsoft.Compute --expand resourceTypes/aliases
```

**PowerShell examples**

PowerShell is common in enterprise automation. Cmdlet parameter shapes can vary by Az module version, so confirm the help text before using a script in production.

Create a policy definition from a JSON file:

```powershell
$definition = Get-Content -Raw .\allowed-locations-definition.json | ConvertFrom-Json
New-AzPolicyDefinition -Name 'allowed-locations' -DisplayName 'Allowed locations' -Policy $definition
```

Assign the policy:

```powershell
New-AzPolicyAssignment `
  -Name 'assign-allowed-locations' `
  -DisplayName 'Assign allowed locations' `
  -Scope '/subscriptions/<subscription-id>' `
  -PolicyDefinition (Get-AzPolicyDefinition -Name 'allowed-locations') `
  -PolicyParameterObject @{
    listOfAllowedLocations = @{
      value = @('eastus', 'westus2')
    }
  }
```

If your module version expects a different input shape, use the cmdlet help and examples from the installed Az.Resources module.

**Bicep and ARM deployment**

You can manage policy definitions as code with Bicep or ARM templates. This is useful for repeatable governance deployment.

A Bicep example that deploys a policy definition resource:

```bicep
resource allowedLocations 'Microsoft.Authorization/policyDefinitions@2021-06-01' = {
  name: 'allowed-locations'
  properties: {
    displayName: 'Allowed locations'
    policyType: 'Custom'
    mode: 'All'
    parameters: {
      listOfAllowedLocations: {
        type: 'Array'
        metadata: {
          displayName: 'Allowed locations'
          description: 'The list of allowed Azure regions.'
        }
      }
    }
    policyRule: {
      if: {
        not: {
          field: 'location'
          in: "[parameters('listOfAllowedLocations')]"
        }
      }
      then: {
        effect: 'deny'
      }
    }
  }
}
```

In practice, many teams store the policy rule as JSON text and inject it into deployment code to avoid syntax mistakes. The important exam concept is that the policy definition is a first-class Azure resource and can be deployed declaratively.

**Validation and troubleshooting**

Policy authoring often fails because of alias, scope, or schema issues rather than logic issues.

Check these first:

- The JSON is valid and matches the expected schema.
- The alias exists for the target resource type.
- The `mode` matches the resource type and field.
- Parameter names match exactly between definition and assignment.
- The assignment scope is correct.
- The managed identity has the required role assignments for `modify` or `deployIfNotExists`.

Common troubleshooting symptoms:

- **Policy never evaluates**
  - Wrong scope.
  - Wrong mode.
  - Unsupported alias.
- **Policy evaluates but does not deny**
  - Condition logic is inverted.
  - The field path is incorrect.
  - The resource type does not expose the alias you used.
- **Remediation fails**
  - Missing managed identity.
  - Missing role assignment.
  - Deployment template is incomplete or invalid.
- **Assignment succeeds but resources remain noncompliant**
  - Existing resources need remediation.
  - The policy is audit-only.
  - The resource provider does not support the targeted property.

**Exam-focused distinctions**

These distinctions are frequently tested:

- A **policy definition** is the reusable rule.
- A **policy assignment** applies the rule to a scope.
- An **initiative** groups multiple definitions.
- **Parameters in the definition** make the rule reusable.
- **Parameters in the assignment** provide environment-specific values.
- **`deny`** blocks creation or update.
- **`audit`** reports noncompliance without blocking.
- **`modify`** changes the request and requires assignment identity and permissions.
- **`deployIfNotExists`** deploys missing configuration and often requires remediation for existing resources.

**Practical governance scenarios**

Use custom policy definitions for real business controls such as:

- Requiring `Environment` and `Owner` tags on all production resources.
- Denying public IP addresses on network interfaces in sensitive subscriptions.
- Allowing only approved VM sizes in cost-controlled subscriptions.
- Enforcing diagnostic settings on Key Vault, Storage, or network resources.
- Restricting resource creation to approved regions for compliance.
- Standardizing naming or metadata through `modify` where appropriate.

These scenarios are valuable because they map directly to how Azure Policy is used in enterprise security operations.

**Key takeaways**

- Build a custom policy definition when built-in policies do not meet your requirement.
- Use `policyRule` with `if` and `then` to express the control.
- Put reusable values in `parameters` and supply them at assignment time.
- Choose `mode` carefully because it affects evaluation and alias availability.
- Prefer `deny` for hard enforcement, `audit` for discovery, `modify` for request correction, and `deployIfNotExists` for missing configuration.
- Remember that the assignment, not the definition, carries the managed identity and scope-specific values.
- Use remediation tasks when you need to fix existing noncompliant resources for supported effects.
