## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Policy initiatives overview**

An Azure Policy initiative, also called a **policy set definition**, groups multiple policy definitions into one assignable unit. You use initiatives when one governance objective requires several related controls, such as enforcing tagging, allowed locations, and secure configuration together.

A policy definition evaluates one rule. An initiative packages many rules so you can assign, parameterize, and report on them as a single governance object. This reduces assignment sprawl and makes compliance reporting easier to manage.

**Core hierarchy**

- **Policy definition**: one rule, one compliance intent.
- **Policy initiative**: a collection of policy definitions with shared metadata and optional shared parameters.
- **Policy assignment**: the act of applying a definition or initiative to a scope.
- **Compliance state**: the evaluation result for assigned resources.

A simple flow looks like this:

```text
Policy definition(s) -> Initiative (policy set definition) -> Assignment -> Evaluation -> Compliance report
```

**When to use an initiative**

Use an initiative when you want to:

- Apply a standard baseline across many subscriptions.
- Bundle related controls for a security or regulatory framework.
- Reuse the same policy collection across environments.
- Simplify assignment by managing one object instead of many.

Use a single policy definition when the requirement is narrow and does not need grouping.

**Scope and placement**

Policy definitions and policy set definitions can be created at **management group** or **subscription** scope. They are not created at resource group or resource scope.

This matters for exam scenarios:

- Create at **management group** scope when you want reuse across multiple subscriptions.
- Create at **subscription** scope when the initiative is specific to one subscription.
- Assign at management group, subscription, resource group, or resource scope depending on the policy type and the effect, but the definition itself still lives at management group or subscription scope.

> A policy assignment can target a narrower scope than the definition, but the definition or initiative itself must be stored at management group or subscription scope.

**Typical initiative structure**

An initiative definition contains metadata and a `properties.policyDefinitions` array. Each entry references a policy definition and can map initiative parameters to child policy parameters.

Common fields include:

- `displayName`
- `description`
- `metadata`
- `parameters`
- `policyDefinitions`

Example initiative JSON:

```json
{
  "properties": {
    "displayName": "Baseline security controls",
    "description": "Groups common security policies for Azure workloads.",
    "metadata": {
      "category": "Security"
    },
    "parameters": {
      "effect": {
        "type": "String",
        "metadata": {
          "displayName": "Effect"
        },
        "allowedValues": [
          "Audit",
          "Deny",
          "Disabled"
        ],
        "defaultValue": "Audit"
      }
    },
    "policyDefinitions": [
      {
        "policyDefinitionId": "/providers/Microsoft.Authorization/policyDefinitions/allowed-locations",
        "parameters": {
          "effect": {
            "value": "[parameters('effect')]"
          }
        }
      },
      {
        "policyDefinitionId": "/providers/Microsoft.Authorization/policyDefinitions/tags-required",
        "parameters": {
          "effect": {
            "value": "[parameters('effect')]"
          }
        }
      }
    ]
  }
}
```

This example shows the important pattern: the initiative can expose a single `effect` parameter and pass it to multiple child policies. In real deployments, the child policy parameter names must match the actual policy definition schema.

**Parameter mapping**

Parameter mapping is one of the most important initiative concepts for AZ-500.

- Initiative parameters let you centralize values used by multiple child policies.
- Child policies can receive the same value through their own parameter names.
- Mapping reduces duplication and keeps assignments consistent.

Common use cases:

- One `effect` parameter controls whether policies audit, deny, or are disabled.
- One `listOfAllowedLocations` parameter feeds multiple location-related policies.
- One `tagName` parameter is reused across tag enforcement policies.

Example of a child policy parameter mapping:

```json
{
  "policyDefinitionId": "/providers/Microsoft.Authorization/policyDefinitions/allowed-locations",
  "parameters": {
    "listOfAllowedLocations": {
      "value": "[parameters('allowedLocations')]"
    }
  }
}
```

**Compliance behavior**

Initiative compliance is aggregated from the compliance of its child policy definitions, but the exact reporting behavior depends on assignment scope, evaluation timing, and whether policies are enabled, exempted, or not applicable.

Practical interpretation:

- A non-compliant child policy can make the initiative appear non-compliant.
- A compliant child policy does not override a non-compliant one.
- Disabled or excluded policies may not contribute the same way as active policies.

For exam prep, remember that initiative compliance is not just a simple yes/no wrapper. It is a roll-up of multiple evaluations.

**Real-world example: security baseline initiative**

A common enterprise pattern is a baseline initiative that includes:

- Allowed locations
- Required tags
- Secure transfer for storage
- Diagnostic settings for key resources
- TLS minimum version requirements

This lets you assign one baseline to a subscription and measure overall posture from one compliance view.

**End-to-end walkthrough**

1. Create or reuse individual policy definitions.
2. Group them into a policy set definition.
3. Expose shared parameters at the initiative level.
4. Map initiative parameters to each child policy.
5. Assign the initiative to a management group or subscription.
6. Review compliance and remediate non-compliant resources.

Example workflow with Azure CLI:

```bash
# Create a policy set definition from a JSON file
az policy set-definition create \
  --name baseline-security-controls \
  --display-name "Baseline security controls" \
  --definitions @initiative.json
```

If your CLI version expects a different payload shape, use the command help and the current Azure CLI documentation for the exact parameter name. The exam focus is the concept: the initiative is created from a JSON definition that contains `policyDefinitions`.

Assigning an initiative:

```bash
az policy assignment create \
  --name baseline-security-assignment \
  --scope /subscriptions/<subscription-id> \
  --policy-set-definition baseline-security-controls \
  --params '{
    "effect": {
      "value": "Audit"
    }
  }'
```

PowerShell example for assignment:

```powershell
New-AzPolicyAssignment `
  -Name "baseline-security-assignment" `
  -Scope "/subscriptions/<subscription-id>" `
  -PolicySetDefinition "/subscriptions/<subscription-id>/providers/Microsoft.Authorization/policySetDefinitions/baseline-security-controls"
```

For parameterized assignments, pass parameters using the cmdlet’s supported parameter structure for your Az module version. The exact syntax can vary, so verify with `Get-Help` before using it in production.

**Azure Portal workflow**

You can also create initiatives in the Azure portal:

- Go to **Policy**.
- Select **Definitions**.
- Choose **+ Initiative definition**.
- Set the scope to management group or subscription.
- Add policy definitions to the initiative.
- Configure initiative parameters and child parameter mappings.
- Save and then assign the initiative.

Portal creation is useful for learning and quick setup, but JSON and automation are better for repeatable governance.

**Common design patterns**

- **Security baseline**: bundle security-related policies for consistent enforcement.
- **Landing zone guardrails**: combine location, SKU, tagging, and diagnostic policies.
- **Regulatory mapping**: group policies that support a framework such as ISO or CIS.
- **Environment-specific initiatives**: separate dev, test, and prod baselines.

**Practical configuration tips**

- Keep initiative names descriptive and stable.
- Use metadata categories to organize definitions.
- Prefer reusable parameters over hardcoded values.
- Test in audit mode before switching to deny.
- Validate child policy parameter names carefully.
- Use management groups for enterprise-wide baselines.

**Troubleshooting and validation**

If an initiative does not behave as expected, check:

- The initiative scope is valid.
- Each child policy definition exists and is accessible.
- Parameter names match exactly.
- The assignment scope includes the target resources.
- The effect is not set to `Disabled`.
- The resource type is in scope for the policy rule.

Useful validation commands:

```bash
az policy set-definition list --query "[].{name:name, displayName:displayName}" -o table
az policy assignment list --scope /subscriptions/<subscription-id> -o table
az policy state list --subscription <subscription-id> -o table
```

**Exam focus points**

- A policy initiative is a **policy set definition**.
- It groups multiple policy definitions under one assignment.
- It can be created at management group or subscription scope.
- It supports parameter mapping from initiative parameters to child policies.
- Compliance is evaluated across the included policies.
- Initiatives simplify governance at scale.

**Quick comparison**

| Item | Purpose | Typical scope |
|---|---|---|
| Policy definition | One rule | Management group or subscription |
| Policy initiative | Group of rules | Management group or subscription |
| Policy assignment | Apply to resources | Management group, subscription, resource group, or resource |

**Mental model for AZ-500**

Think of a policy initiative as a security baseline container. You define the controls once, assign them once, and then monitor many related compliance checks through one governance object.
