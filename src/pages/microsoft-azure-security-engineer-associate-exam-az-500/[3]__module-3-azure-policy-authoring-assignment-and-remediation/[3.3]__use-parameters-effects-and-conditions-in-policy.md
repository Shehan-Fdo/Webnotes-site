## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Azure Policy fundamentals**

Azure Policy evaluates Azure resources against rules you define. You use it to enforce standards, assess compliance, and optionally change or deploy settings when resources do not meet requirements. The core building blocks are:

- **Parameters** to make a definition reusable.
- **Conditions** to target specific resources or request properties.
- **Effects** to decide what happens when a rule matches.

A policy definition is usually written in JSON and then assigned at a scope such as management group, subscription, resource group, or resource. The same definition can behave differently across environments when you pass different parameter values at assignment time.

**Policy parameters**

Parameters let you avoid hardcoding values in a definition. They make one policy reusable for multiple teams, subscriptions, regions, or naming standards.

Common uses include:

- Allowed locations
- Required tag names and values
- SKU restrictions
- Naming prefixes or suffixes
- Thresholds such as maximum allowed size or count

A parameter definition typically includes a type, default value, allowed values, and metadata for display.

```json
{
  "parameters": {
    "allowedLocations": {
      "type": "Array",
      "metadata": {
        "displayName": "Allowed locations",
        "description": "List of Azure regions that are permitted."
      },
      "defaultValue": [
        "eastus",
        "westus2"
      ]
    }
  }
}
```

You reference parameters in the policy rule by using the `parameters()` function.

```json
{
  "if": {
    "field": "location",
    "notIn": "[parameters('allowedLocations')]"
  },
  "then": {
    "effect": "deny"
  }
}
```

Parameters are especially useful when you want one definition to serve multiple business units. For example, a global enterprise may allow `eastus` in one subscription and `westeurope` in another without duplicating the policy logic.

**Conditions and policy logic**

The `if` block defines the condition that must be true for the policy rule to apply. Azure Policy evaluates the resource request or existing resource against fields, aliases, and functions.

Typical condition patterns include:

- `field` comparisons such as `equals`, `notEquals`, `in`, `notIn`, `like`, `match`, and `exists`
- Logical operators such as `allOf`, `anyOf`, and `not`
- Template functions such as `parameters()`, `concat()`, `if()`, `length()`, and `resourceGroup()`

A simple tag enforcement example:

```json
{
  "if": {
    "field": "tags['Environment']",
    "exists": "false"
  },
  "then": {
    "effect": "deny"
  }
}
```

A more realistic condition can combine multiple checks:

```json
{
  "if": {
    "allOf": [
      {
        "field": "type",
        "equals": "Microsoft.Storage/storageAccounts"
      },
      {
        "field": "Microsoft.Storage/storageAccounts/sku.name",
        "notIn": [
          "Standard_LRS",
          "Standard_GRS"
        ]
      }
    ]
  },
  "then": {
    "effect": "audit"
  }
}
```

Use conditions to narrow the policy scope to the exact resources you care about. This reduces false positives and avoids applying a rule to unsupported resource types.

**Alias usage and field targeting**

Azure Policy uses aliases to reference resource properties. An alias maps a policy field expression to a real resource property path. This is important because many Azure resource providers expose nested properties in different shapes.

Examples:

- `location`
- `tags['CostCenter']`
- `Microsoft.Compute/virtualMachines/osProfile.adminUsername`
- `Microsoft.Storage/storageAccounts/networkAcls.defaultAction`

When you write a condition, the `field` value must be a valid alias or resource property name supported by Azure Policy. If you need computed logic, use functions in the value expression, not in the field name.

Example of a valid computed comparison:

```json
{
  "if": {
    "field": "name",
    "equals": "[concat('prod-', parameters('suffix'))]"
  },
  "then": {
    "effect": "deny"
  }
}
```

If a resource provider does not expose an alias for the property you want, the policy may not evaluate that property as expected. In practice, alias availability is a common troubleshooting point when a rule appears correct but never matches.

**Policy effects**

The effect determines what Azure Policy does when the `if` condition evaluates to true. The most common effects in AZ-500 scenarios are:

- **deny**: blocks noncompliant requests
- **audit**: allows the request but records noncompliance
- **append**: adds request properties when the resource provider accepts them in the incoming request
- **modify**: changes supported request properties at evaluation time
- **deployIfNotExists**: deploys a related resource or configuration if the target resource is missing it
- **disabled**: turns the rule off

A quick comparison:

| Effect | Primary purpose | Typical use |
|---|---|---|
| `deny` | Prevent noncompliant creation or update | Block public IPs, disallowed regions |
| `audit` | Report noncompliance without blocking | Visibility before enforcement |
| `append` | Add request properties if supported | Add tags or headers during create/update |
| `modify` | Update supported request properties during evaluation | Enforce tags or property values |
| `deployIfNotExists` | Deploy missing related configuration | Diagnostic settings, monitoring, encryption |
| `disabled` | Disable the rule | Temporary testing or rollback |

> `append` and `modify` are not interchangeable. `append` only works when the resource provider accepts the added request property, while `modify` can update supported fields in the request before the resource is created or updated.

**Deny**

Use `deny` when the safest action is to stop the request. This is common for security guardrails such as:

- Blocking public network access
- Restricting resource locations
- Preventing insecure SKU choices
- Enforcing approved VM sizes

Example:

```json
{
  "if": {
    "field": "location",
    "notIn": "[parameters('allowedLocations')]"
  },
  "then": {
    "effect": "deny"
  }
}
```

Use `deny` carefully in production. If your condition is too broad, you can block legitimate deployments and create operational friction.

**Audit**

Use `audit` when you want visibility before enforcement. It is ideal for:

- Baseline assessments
- Migration projects
- New standards that teams have not adopted yet
- Policies that are difficult to enforce immediately

Example:

```json
{
  "if": {
    "field": "tags['Owner']",
    "exists": "false"
  },
  "then": {
    "effect": "audit"
  }
}
```

Audit is often the first step in a rollout strategy. You can measure impact, fix false positives, and then move to deny or modify later.

**Append**

`append` adds request properties only when the resource provider accepts them in the incoming request. It does not rewrite existing values and it is not a universal defaulting mechanism.

Use it for cases where the request can safely carry extra properties, such as:

- Adding tags during create/update
- Adding a property supported by the resource provider
- Standardizing request metadata

Example:

```json
{
  "if": {
    "field": "tags['Environment']",
    "exists": "false"
  },
  "then": {
    "effect": "append",
    "details": {
      "field": "tags['Environment']",
      "value": "prod"
    }
  }
}
```

In practice, `modify` is usually preferred over `append` for tag enforcement because it is more explicit about changing supported request properties.

**Modify**

`modify` changes supported request properties at evaluation time. It is commonly used to enforce tags, normalize values, or set a property when the request is created or updated.

Important points:

- The policy assignment often needs a managed identity if the effect must write changes.
- The identity needs the right role assignments at the target scope.
- `modify` acts during policy evaluation, not as a separate remediation engine.
- Remediation tasks are used to bring existing resources into compliance after assignment.

Example tag enforcement pattern:

```json
{
  "if": {
    "field": "tags['CostCenter']",
    "exists": "false"
  },
  "then": {
    "effect": "modify",
    "details": {
      "roleDefinitionIds": [
        "/providers/Microsoft.Authorization/roleDefinitions/4a9ae827-6dc8-4573-8ac7-8239d42aa03f"
      ],
      "operations": [
        {
          "operation": "addOrReplace",
          "field": "tags['CostCenter']",
          "value": "12345"
        }
      ]
    }
  }
}
```

The exact role definition depends on the operation and scope. For tag-related policies, a least-privilege role is preferred over broad roles such as Contributor.

**DeployIfNotExists**

`deployIfNotExists` checks whether a related resource or configuration exists. If not, Azure Policy deploys the missing configuration using an embedded template.

This effect is common for security and monitoring controls such as:

- Diagnostic settings
- Log Analytics workspace integration
- Security extensions
- Encryption-related companion resources

A realistic deployment policy needs:

- A valid `existenceCondition`
- A valid `deployment` template
- Correct `roleDefinitionIds`
- A managed identity on the assignment for deployment permissions

Example pattern for diagnostic settings:

```json
{
  "if": {
    "field": "type",
    "equals": "Microsoft.KeyVault/vaults"
  },
  "then": {
    "effect": "deployIfNotExists",
    "details": {
      "type": "Microsoft.Insights/diagnosticSettings",
      "existenceScope": "ResourceGroup",
      "existenceCondition": {
        "allOf": [
          {
            "field": "Microsoft.Insights/diagnosticSettings/workspaceId",
            "exists": "true"
          }
        ]
      },
      "roleDefinitionIds": [
        "/providers/Microsoft.Authorization/roleDefinitions/749f88d5-cbae-40b8-bcfc-e573ddc772fa"
      ],
      "deployment": {
        "properties": {
          "mode": "incremental",
          "template": {
            "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentTemplate.json#",
            "contentVersion": "1.0.0.0",
            "parameters": {
              "resourceName": {
                "type": "string"
              },
              "workspaceId": {
                "type": "string"
              }
            },
            "resources": [
              {
                "type": "Microsoft.Insights/diagnosticSettings",
                "apiVersion": "2021-05-01-preview",
                "name": "[concat(parameters('resourceName'), '/setByPolicy')]",
                "properties": {
                  "workspaceId": "[parameters('workspaceId')]",
                  "logs": [
                    {
                      "category": "AuditEvent",
                      "enabled": true
                    }
                  ],
                  "metrics": [
                    {
                      "category": "AllMetrics",
                      "enabled": true
                    }
                  ]
                }
              }
            ]
          }
        }
      }
    }
  }
}
```

> `existenceScope` supports `ResourceGroup` or `Subscription`, not `Resource`. If you target a resource-level companion configuration, make sure the policy structure matches the supported scope and resource type.

**Disabled**

Use `disabled` when you want the definition present but inactive. This is useful for:

- Testing
- Staged rollouts
- Temporary exceptions
- Safe rollback during troubleshooting

A disabled policy does not evaluate resources for compliance.

**Policy assignment and managed identity**

Some effects need a managed identity on the assignment. This is especially important for:

- `modify`
- `deployIfNotExists`

The identity is used so Azure Policy can perform the required write operations. You also need to grant the identity the minimum role permissions required at the target scope.

Practical guidance:

- Use a system-assigned managed identity when the assignment is tied to one scope.
- Use a user-assigned managed identity when you want reuse across assignments.
- Grant only the permissions needed for the specific effect.
- Verify the identity can write to the target resource group or subscription.

**Compliance states**

Azure Policy compliance reporting helps you understand whether resources meet the rule. Common states include:

- **Compliant**: the resource meets the policy
- **Non-compliant**: the resource violates the policy
- **Conflict**: another policy or assignment conflicts with the result
- **Exempt**: the resource is excluded by exemption
- **Not started / Unknown**: evaluation has not completed or the state is not yet available

Use these states as operational signals, not just report labels. For example, a large number of non-compliant resources after a new assignment may indicate a bad condition, a missing alias, or a rollout that needs remediation.

**Modes and evaluation behavior**

Policy mode controls how Azure Policy evaluates resources. The most common modes are:

- **All**: evaluates all resource types
- **Indexed**: focuses on resource types that support tags and location and have alias support

Use `All` when you need broad coverage across resource types. Use `Indexed` when you are targeting resources that expose the properties you need and you want a more focused evaluation model.

A practical rule of thumb:

- Choose `Indexed` for tag and location governance.
- Choose `All` when you need to evaluate a wider set of resource types or nested resources.

**Real-world scenario: enterprise tagging standard**

A security team wants every production resource to include `Environment=Prod` and `CostCenter`.

1. They start with `audit` to measure the gap.
2. They review noncompliant resources and fix false positives.
3. They switch to `modify` for supported resource types.
4. They use `deny` for critical subscriptions where missing tags are not allowed.

Example policy logic:

```json
{
  "if": {
    "allOf": [
      {
        "field": "tags['Environment']",
        "exists": "false"
      },
      {
        "field": "type",
        "notEquals": "Microsoft.Resources/subscriptions/resourceGroups"
      }
    ]
  },
  "then": {
    "effect": "modify",
    "details": {
      "operations": [
        {
          "operation": "addOrReplace",
          "field": "tags['Environment']",
          "value": "Prod"
        }
      ]
    }
  }
}
```

This approach works well because you can introduce governance gradually instead of blocking every deployment on day one.

**Real-world scenario: diagnostic settings for security logging**

A security operations team needs logs from Key Vault and storage accounts sent to a central Log Analytics workspace.

- They use `deployIfNotExists` because the configuration is a related resource.
- They assign the policy at the subscription or management group scope.
- They grant the policy identity the minimum role needed to create diagnostic settings.
- They trigger remediation for existing resources after assignment.

A typical workflow is:

1. Create or select the policy definition.
2. Assign it with a managed identity.
3. Confirm the target workspace ID is passed as a parameter.
4. Run remediation for existing noncompliant resources.
5. Verify compliance results after the scan completes.

**CLI examples**

Use Azure CLI to assign, inspect, and remediate policy. Set your subscription context first when needed.

```bash
az account set --subscription <subscription-id>
az policy assignment create \
  --name enforce-tags \
  --display-name "Enforce required tags" \
  --scope /subscriptions/<subscription-id> \
  --policy <policy-definition-id> \
  --params '{
    "allowedLocations": {
      "value": ["eastus", "westus2"]
    }
  }'
```

Trigger a compliance scan:

```bash
az policy state trigger-scan
```

Create a remediation task for an assignment:

```bash
az policy remediation create \
  --name remediate-tags \
  --policy-assignment enforce-tags \
  --scope /subscriptions/<subscription-id>
```

Depending on the assignment and effect, you may need additional parameters such as a resource discovery mode or a specific resource group scope. Always verify the assignment name, scope, and identity permissions before running remediation.

**Troubleshooting tips**

If a policy does not behave as expected, check the following:

- The alias exists for the property you are targeting.
- The `field` expression uses a valid alias or resource property name.
- The condition is not too broad or too narrow.
- The effect is supported for the resource type.
- The assignment has the required managed identity and role permissions.
- The policy mode matches the resource types you want to evaluate.
- The remediation task targets the correct scope.

A common troubleshooting pattern is to start with `audit`, confirm the condition matches the intended resources, and then move to enforcement. If a `modify` or `deployIfNotExists` policy fails, inspect the assignment identity and role assignments first, because missing permissions are a frequent cause of failure.

**Exam focus**

For AZ-500, focus on how the pieces work together:

- Parameters make policies reusable and environment-aware.
- Conditions determine when the policy applies.
- Effects determine the response to noncompliance.
- Managed identities and role assignments are required for write-capable effects.
- Remediation is separate from evaluation and is used to fix existing resources.

If you can explain when to use `audit`, `deny`, `modify`, and `deployIfNotExists`, and how parameters and conditions shape the rule, you are covering the most testable part of this topic.
