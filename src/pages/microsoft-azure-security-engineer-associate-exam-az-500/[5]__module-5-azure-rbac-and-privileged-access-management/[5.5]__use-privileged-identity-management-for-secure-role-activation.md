## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Privileged Identity Management overview**

<abbr title="Privileged Identity Management">PIM</abbr> is Microsoft Entra ID’s just-in-time access control service for privileged roles. You use it to reduce standing access by making users eligible for roles instead of permanently active, then requiring activation only when needed.

PIM helps you answer three security questions:

- Who can activate a privileged role?
- What conditions must be met before activation?
- How do you prove and review privileged use afterward?

The main value is risk reduction. If an account is compromised, an attacker cannot immediately use a permanently assigned admin role if that role is only eligible and requires activation with controls such as multi-factor authentication (<abbr title="Multi-Factor Authentication">MFA</abbr>), justification, approval, or time limits.

**Core PIM concepts**

PIM works with both Microsoft Entra roles and Azure resource roles.

- **Eligible assignment** means the user can activate the role when needed.
- **Active assignment** means the role is currently usable without activation.
- **Just-in-time access** means access is granted only for a limited period.
- **Least privilege** means users receive only the access they need, only when they need it.
- **Access reviews** help validate whether privileged access is still required.

A common exam distinction is that PIM is not just a role assignment tool. It is a governance and control layer for privileged access lifecycle management.

**Role types and scope**

PIM can manage:

- **Microsoft Entra roles** such as Global Administrator, Privileged Role Administrator, or Security Administrator.
- **Azure resource roles** such as Owner, Contributor, or User Access Administrator at subscription, resource group, or resource scope.

The scope matters:

- Entra roles govern directory-level permissions.
- Azure resource roles govern Azure Resource Manager permissions.
- A user may need different activation paths depending on whether they are managing identity settings or Azure resources.

> PIM does not replace role-based access control (<abbr title="Role-Based Access Control">RBAC</abbr>); it enhances RBAC with time-bound activation and policy enforcement.

**How secure role activation works**

A typical secure activation flow looks like this:

1. You assign the user as **eligible** for a role.
2. You configure activation requirements for that role.
3. The user requests activation when work begins.
4. PIM evaluates policy requirements.
5. If conditions are met, the role becomes active for a limited duration.
6. PIM logs the activation for audit and review.

Example activation requirements often include:

- MFA verification
- Justification text
- Ticket number or change request reference
- Approval from one or more approvers
- Maximum activation duration
- Notification to security or administrators

**Why use PIM instead of permanent admin access**

Permanent privileged access increases blast radius. PIM reduces exposure by limiting the time window in which elevated permissions exist.

Use PIM when:

- Administrators only need elevated access occasionally.
- You want approval workflows for sensitive roles.
- You need audit trails for compliance.
- You want to enforce MFA before privileged use.
- You want to reduce the number of always-on privileged accounts.

Use permanent assignment only when operational requirements truly demand continuous access and the risk is accepted.

**Common activation controls**

PIM policies let you require one or more controls before activation.

- **MFA** confirms the user is present and reduces token theft risk.
- **Justification** creates an audit trail explaining why access was needed.
- **Approval** adds human review for sensitive roles.
- **Ticketing integration** links activation to a change or incident record.
- **Activation duration** limits how long the role remains active.
- **Notifications** alert stakeholders when privileged access is requested or granted.

A practical policy example:

```text
Role: Azure Subscription Owner
Assignment type: Eligible
Activation requires:
- MFA
- Justification
- Approval by Security Admin group
- Maximum duration: 1 hour
- Notification to security mailbox
```

**Typical Azure portal workflow**

You usually configure PIM in the Microsoft Entra admin center or Azure portal, depending on the role type.

1. Open **Privileged Identity Management**.
2. Select the role category, such as **Microsoft Entra roles** or **Azure resources**.
3. Choose the role and assign users as **eligible**.
4. Configure activation settings for the role.
5. Review notifications, approval settings, and duration limits.
6. Test activation with a non-production account.

The user experience is straightforward:

- The user opens PIM.
- The user selects the eligible role.
- The user clicks **Activate**.
- The user completes required checks.
- The role becomes active for the configured duration.

**Azure CLI and automation examples**

PIM is often managed through portal workflows, but automation is useful for repeatable governance tasks. In practice, role assignment and access governance can be scripted with Microsoft Graph or Azure CLI depending on the scenario.

Example Azure CLI commands for inspecting role assignments:

```bash
az role assignment list --assignee user@contoso.com --all -o table
az role definition list --name "Owner" -o jsonc
```

Example Microsoft Graph PowerShell pattern for privileged role administration:

```powershell
Connect-MgGraph -Scopes "RoleManagement.ReadWrite.Directory","Directory.Read.All"

Get-MgRoleManagementDirectoryRoleAssignmentScheduleRequest
Get-MgRoleManagementDirectoryRoleEligibilitySchedule
```

Example of a policy-oriented JSON concept for activation settings:

```json
{
  "role": "Security Administrator",
  "assignmentType": "Eligible",
  "activationMaximumDuration": "PT1H",
  "requireMfaOnActivation": true,
  "requireJustification": true,
  "requireApproval": true
}
```

The exact API surface and cmdlet names can vary by module version and tenant configuration, so verify against the current Microsoft Graph and Azure tooling in your environment.

**Approval workflow design**

Approval is useful for high-impact roles, but it should be reserved for roles where delay is acceptable. If every activation requires approval, administrators may bypass the process operationally or create excessive friction.

Good approval design:

- Use approvers who understand the business impact.
- Keep approver groups small and available.
- Define backup approvers for off-hours coverage.
- Set clear justification standards.
- Align approval requirements with role sensitivity.

Poor approval design:

- Using the same person as requester and approver.
- Requiring approval for low-risk, high-frequency tasks.
- Leaving no backup approver.
- Allowing indefinite activation windows.

**Access reviews and auditing**

PIM includes review and audit capabilities that help you validate whether privileged access is still needed.

Use access reviews to:

- Reconfirm that eligible assignments are still required.
- Remove stale privileged access.
- Validate group membership for privileged groups.
- Support compliance evidence collection.

Audit logs help you investigate:

- Who activated a role
- When activation occurred
- How long access remained active
- Whether approval was granted
- Whether activation failed and why

For incident response, privileged access logs are important because they show the exact time window in which elevated actions could have occurred.

**Security best practices**

- Prefer **eligible** assignments over permanent active assignments.
- Require **MFA** for all privileged activations.
- Use separate admin accounts for privileged work.
- Limit activation duration to the minimum practical time.
- Require justification for sensitive roles.
- Use approval for high-risk roles only.
- Review privileged assignments regularly.
- Monitor alerts for unusual activation patterns.
- Combine PIM with Conditional Access for stronger control.

A strong operational pattern is to use a dedicated admin account with PIM eligibility, then activate only during maintenance windows.

**Integration with Conditional Access**

Conditional Access can strengthen PIM by enforcing additional conditions before a user can authenticate or activate a role.

Common combinations include:

- Require compliant device for admin sign-in.
- Require MFA for privileged users.
- Block legacy authentication.
- Restrict admin access by location or risk level.
- Require authentication strength for sensitive roles.

This layered approach matters because PIM controls role activation, while Conditional Access controls sign-in conditions.

**Troubleshooting activation issues**

If activation fails, check the following:

- The user is actually **eligible** for the role.
- MFA is completed successfully.
- The user meets approval requirements.
- The activation duration does not exceed policy limits.
- The role is available at the correct scope.
- The user has not exceeded activation frequency or policy constraints.
- The tenant’s licensing and feature configuration support the requested PIM capability.

Common symptoms:

- The **Activate** button is unavailable.
- Activation requests remain pending.
- The role activates but permissions do not appear as expected.
- The user can activate Entra roles but not Azure resource roles, or vice versa.

When troubleshooting, verify both the identity-side role and the Azure resource scope. A user may have the right directory role but still lack the correct subscription or resource group role.

**Exam-focused distinctions**

- **PIM** is for controlling privileged access over time.
- **RBAC** is for assigning permissions to identities at a scope.
- **Eligible** means can activate later.
- **Active** means currently usable.
- **MFA, justification, approval, and duration** are common activation controls.
- **Access reviews** validate whether access should continue.
- **Audit logs** provide evidence of privileged use.

**Practical scenario**

A subscription owner only needs elevated access during monthly network changes. Instead of assigning Owner permanently, you make the user eligible in PIM, require MFA and justification, and limit activation to one hour. If the change is sensitive, you add approval from a security group. After the change, the role expires automatically and the audit log records the activation.

This pattern reduces standing privilege while preserving operational flexibility.

**Quick reference**

- **Use PIM** for just-in-time privileged access.
- **Use eligible assignments** to avoid standing admin rights.
- **Require controls** such as MFA, justification, approval, and time limits.
- **Review and audit** privileged access regularly.
- **Combine with Conditional Access** for stronger identity protection.
