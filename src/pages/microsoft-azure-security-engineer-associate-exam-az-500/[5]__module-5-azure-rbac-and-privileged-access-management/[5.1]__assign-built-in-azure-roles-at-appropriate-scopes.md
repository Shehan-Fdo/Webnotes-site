## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Azure RBAC fundamentals**

Azure <abbr title="Role-Based Access Control">RBAC</abbr> controls who can do what on Azure resources. You assign a role to a security principal at a specific scope, and Azure evaluates that assignment when the principal requests access.

The main idea for exam scenarios is simple: choose the narrowest scope that still lets the user do the job. Narrow scope reduces blast radius and makes access easier to audit.

- **Security principal**: user, group, service principal, or managed identity.
- **Role assignment**: a binding between a principal, a role definition, and a scope.
- **Built-in role**: Microsoft-defined role with a fixed permission set.
- **Custom role**: tenant-specific role you define when built-ins are too broad or too narrow.

**Scope hierarchy and inheritance**

Azure scopes are hierarchical. Permissions assigned at a higher scope flow down to child scopes unless a more specific assignment or a deny assignment changes the effective result.

Common scopes include:

- **Management group**
- **Subscription**
- **Resource group**
- **Resource**

A role assigned at a management group can apply to many subscriptions. A role assigned at a subscription applies to all resource groups and resources in that subscription. A resource-group assignment applies only to resources in that group, and a resource-level assignment applies only to that resource.

> Use the highest scope only when the access requirement truly spans multiple child scopes. Otherwise, prefer resource group or resource scope.

**Built-in role selection strategy**

Built-in roles are the first choice when they match the job. They are easier to manage than custom roles and are widely understood by Azure services and administrators.

Use these principles:

- **Least privilege**: choose the smallest role that satisfies the task.
- **Separation of duties**: avoid combining administrative and data access unless required.
- **Scope alignment**: assign at the scope where the work happens.
- **Group-based access**: assign roles to groups instead of individual users when possible.

Examples of common built-in roles:

- **Owner**: full access to manage resources and assign roles at the assigned scope.
- **Contributor**: manage resources but cannot grant access.
- **Reader**: view resources but cannot change them.
- **User Access Administrator**: manage role assignments at the assigned scope.
- **Virtual Machine Contributor**: manage VMs without broader subscription control.
- **Storage Account Contributor**: manage storage account configuration, not blob data access.
- **Storage Blob Data Reader**: read blob data through Azure AD.
- **Storage Blob Data Contributor**: read and write blob data through Azure AD.
- **Key Vault Administrator**: manage Key Vault data-plane objects and permissions when using Azure RBAC for vault data access.
- **Key Vault Secrets User**: read secret values in the data plane.
- **Key Vault Reader**: view Key Vault resource metadata in the management plane, not secret values.

**Management plane vs data plane**

This distinction is a common exam trap.

- **Management plane**: Azure Resource Manager operations such as creating a vault, changing tags, or viewing resource configuration.
- **Data plane**: operations against the service content itself, such as reading a blob, secret, or key.

A role can apply to one plane and not the other. For example, `Contributor` can manage a storage account resource, but it does not grant blob read/write access. For blob access, you need a storage data role such as `Storage Blob Data Reader` or `Storage Blob Data Contributor`.

For Key Vault, the access model matters:

- **Access policy model**: vault access is controlled by Key Vault access policies.
- **Azure RBAC model**: vault data access is controlled by Azure RBAC roles.

Do not confuse `Key Vault Reader` with secret access. `Key Vault Reader` is for viewing vault resource metadata. To read secrets, you need a data-plane role such as `Key Vault Secrets User` when the vault uses Azure RBAC for data access.

**Appropriate scope examples**

Choose the scope based on where the resource lives and how broadly the access should apply.

- **Management group scope**
  - Use when a team administers many subscriptions under one governance boundary.
  - Good for policy, baseline access, and centralized operations.
  - Example: grant a security operations group `Reader` across all subscriptions in a landing zone management group.

- **Subscription scope**
  - Use when access should cover all resource groups in one subscription.
  - Good for platform teams managing an entire environment.
  - Example: grant `Contributor` to a DevOps team for all resources in a dev subscription.

- **Resource group scope**
  - Use when a team owns a workload boundary.
  - Good default for application teams.
  - Example: grant `Virtual Machine Contributor` to a support team for the `rg-app-prod` resource group.

- **Resource scope**
  - Use when access should be limited to one resource.
  - Good for sensitive or shared services.
  - Example: grant `Storage Blob Data Reader` only on one storage account container path or storage account resource, depending on the access pattern.

**Effective access and inherited permissions**

Azure evaluates all applicable assignments from parent scopes and the current scope. The result is the principal’s effective access.

To verify access, check:

- Direct role assignments at the target scope.
- Inherited role assignments from parent scopes.
- Group membership, including nested group effects where supported by the service.
- Deny assignments, if present.
- PIM <abbr title="Privileged Identity Management">PIM</abbr> activation state for eligible roles.

In the Azure portal, use:

- **Access control (IAM)** on the resource, resource group, subscription, or management group.
- **Check access** to inspect a principal’s effective role assignments.
- **Role assignments** to view direct and inherited assignments.

> If a user says they “have the role” but still cannot access the resource, check the exact scope, the plane of access, and whether the role is active or merely eligible in PIM.

**Deny assignments**

Deny assignments block actions even when a role assignment would otherwise allow them. They are less common than allow assignments but matter in managed scenarios such as Azure Blueprints or certain platform-managed resources.

Key points:

- Deny assignments override matching allow permissions.
- They are not the same as removing a role assignment.
- They can explain why access is denied despite apparently sufficient RBAC roles.

**Built-in role IDs and automation**

Built-in role definition IDs are globally consistent across Azure. Custom role IDs are tenant-specific. In automation, use the role definition resource ID, not just the GUID, when a template or API expects the full identifier.

Examples of built-in role definition IDs are stable enough for scripts and templates, but always verify the exact role name and ID when writing production automation.

Common pattern for a built-in role definition resource ID:

```text
/subscriptions/<subscription-id>/providers/Microsoft.Authorization/roleDefinitions/<role-guid>
```

For management group scope, the role definition can also be referenced with the management group resource path when needed by the deployment context.

**Azure CLI examples**

Use Azure CLI to assign roles at the correct scope.

Assign a role at subscription scope:

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --assignee-principal-type User \
  --role "Reader" \
  --scope /subscriptions/<subscription-id>
```

Assign a role at resource group scope:

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --assignee-principal-type Group \
  --role "Contributor" \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-app-prod
```

Assign a data-plane storage role at storage account scope:

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --assignee-principal-type User \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/stprod01
```

Assign a role at management group scope:

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --assignee-principal-type Group \
  --role "Reader" \
  --scope /providers/Microsoft.Management/managementGroups/mg-platform
```

List role assignments at a scope:

```bash
az role assignment list \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-app-prod \
  --include-inherited \
  -o table
```

Check a principal’s access:

```bash
az role assignment list \
  --assignee <principal-object-id> \
  --include-inherited \
  -o table
```

Find a service principal object ID or app ID before assigning access:

```bash
az ad sp show --id <service-principal-object-id-or-app-id> --query id -o tsv
```

Use the service principal object ID or app ID, not an arbitrary app registration display name, unless you know the name resolves uniquely in your tenant.

**ARM and Bicep examples**

Bicep is usually clearer for role assignments because it makes scope and resource IDs explicit.

Assign a role at resource group scope in Bicep:

```bicep
param principalId string
param roleDefinitionGuid string = 'acdd72a7-3385-48ef-bd42-f606fba81ae7' // Reader

resource rgRoleAssignment 'Microsoft.Authorization/roleAssignments@2022-04-01' = {
  name: guid(resourceGroup().id, principalId, roleDefinitionGuid)
  scope: resourceGroup()
  properties: {
    principalId: principalId
    roleDefinitionId: subscriptionResourceId('Microsoft.Authorization/roleDefinitions', roleDefinitionGuid)
    principalType: 'User'
  }
}
```

Assign a role at management group scope in Bicep:

```bicep
targetScope = 'managementGroup'

param principalId string
param managementGroupId string
param roleDefinitionGuid string = 'acdd72a7-3385-48ef-bd42-f606fba81ae7'

resource mgRoleAssignment 'Microsoft.Authorization/roleAssignments@2022-04-01' = {
  name: guid(managementGroup().id, principalId, roleDefinitionGuid)
  scope: managementGroup(managementGroupId)
  properties: {
    principalId: principalId
    roleDefinitionId: tenantResourceId('Microsoft.Authorization/roleDefinitions', roleDefinitionGuid)
    principalType: 'Group'
  }
}
```

Notes for templates:

- Use the full role definition resource ID expected by the deployment context.
- Use a deterministic `name` value such as `guid(...)` to avoid duplicate assignments.
- Deploy the role assignment at the intended scope rather than relying on ambiguous nested resource behavior.

**Portal workflow**

Use the portal when you need to inspect or validate access quickly.

1. Open the target resource, resource group, subscription, or management group.
2. Select **Access control (IAM)**.
3. Review **Role assignments** for direct and inherited access.
4. Use **Check access** to search for a user, group, or service principal.
5. Confirm the exact scope and role name.
6. If access is missing, check whether the principal is in a group, whether PIM is active, and whether the needed permission is data-plane or management-plane.

**Common exam scenarios**

- A user can view a storage account but cannot read blobs.
  - The user likely has a management-plane role such as `Reader` or `Contributor`.
  - Add a storage data role such as `Storage Blob Data Reader`.

- A user can manage a Key Vault resource but cannot read secrets.
  - The user likely has a management-plane role only.
  - If the vault uses Azure RBAC, assign a data-plane role such as `Key Vault Secrets User`.

- A team needs access to all resources in one workload boundary.
  - Assign at resource group scope, not subscription scope, unless the team truly needs broader access.

- A central operations team needs read-only visibility across many subscriptions.
  - Assign `Reader` at management group scope.

- A user can assign roles but should not manage resources.
  - Use `User Access Administrator`, not `Owner`, if the requirement is access management only.

**Practical decision guide**

When choosing a role and scope, ask:

- What exact action must the principal perform?
- Is the action management-plane or data-plane?
- What is the smallest scope that contains the target resource?
- Can you assign the role to a group instead of a user?
- Does the service use Azure RBAC, access policies, or both?
- Is there an inherited assignment or deny assignment affecting the result?

A good default pattern is:

- Put users in groups.
- Assign built-in roles to groups.
- Scope assignments as narrowly as possible.
- Verify effective access in the portal before closing the change.

**Quick comparison**

| Role type | What it controls | Typical use |
|---|---|---|
| Management-plane role | Resource configuration and visibility | Create, update, delete Azure resources |
| Data-plane role | Service content access | Read blobs, secrets, keys, or files |
| Access policy | Service-specific permission model | Legacy or service-specific access control |
| Deny assignment | Blocks access | Prevents unwanted inherited permissions |

**Key takeaways**

- Azure RBAC is scope-based and inherited.
- Built-in roles are preferred when they fit the task.
- Management-plane access does not automatically grant data-plane access.
- Key Vault and Storage are common exam areas where this distinction matters.
- Use the portal’s **Check access** feature to validate effective permissions.
- Management group scope is useful for centralized governance, but it should not be used when a smaller scope is sufficient.
