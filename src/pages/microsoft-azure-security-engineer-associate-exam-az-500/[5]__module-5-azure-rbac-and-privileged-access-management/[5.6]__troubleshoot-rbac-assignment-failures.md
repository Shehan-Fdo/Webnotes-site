## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Azure role-based access control (RBAC) assignment failures usually come from one of four areas: insufficient permissions, an invalid or unresolved principal, an incorrect scope, or a temporary consistency delay. You troubleshoot by checking the caller’s authorization, verifying the target identity, confirming the scope exists and is correct, and then validating whether the assignment eventually appears after propagation.

RBAC is enforced by the Azure Resource Manager control plane. That means failures often happen before any resource-level action occurs. A role assignment can fail even when the target resource itself is healthy, because the issue is about authorization metadata rather than the workload.

**Common failure categories**

- **Authorization failure**: the caller lacks permission to create role assignments at the target scope.
- **Identity resolution failure**: Azure cannot resolve the assignee to a valid directory object.
- **Scope mismatch**: the assignment is attempted at a scope where the caller cannot assign roles, or the scope string is malformed.
- **Propagation delay**: the assignment succeeds, but access is not immediately effective everywhere.
- **Deny or lock interference**: a deny assignment or management lock blocks the operation.
- **Directory or tenant mismatch**: the principal exists in a different tenant or the wrong directory context is selected.

**How RBAC assignment works**

A role assignment binds three things together:

- a **security principal** such as a user, group, service principal, or managed identity
- a **role definition** such as Reader, Contributor, or a custom role
- a **scope** such as management group, subscription, resource group, or resource

The assignment is stored in Azure Resource Manager and evaluated at request time. If the principal has inherited access through group membership or higher scope inheritance, that access may not be obvious from a single assignment record.

**Check the caller’s permission to assign roles**

To create role assignments, the caller needs permission to write role assignments at the target scope. Common built-in roles that can do this include:

- **Owner**
- **User Access Administrator**
- custom roles that include `Microsoft.Authorization/roleAssignments/write`

A caller with Contributor can manage many resources but usually cannot grant access to others. That distinction is a frequent exam trap.

Use the following commands to inspect the current context and the caller’s effective identity:

```bash
az account show -o jsonc
az ad signed-in-user show -o jsonc
az role assignment list --assignee "$(az ad signed-in-user show --query id -o tsv)" --all -o table
```

The last command shows assignments for the signed-in user, but it does not prove the user can create new assignments at a specific scope. To verify that, inspect the role definitions and the scope where you are trying to assign access.

A more useful check is to review whether the caller has a role that includes role assignment write permissions at or above the target scope:

```bash
az role assignment list \
  --assignee "$(az ad signed-in-user show --query id -o tsv)" \
  --scope /subscriptions/<subscription-id> \
  --include-inherited \
  -o table
```

If you see Owner or User Access Administrator at the subscription or a parent management group, the caller can usually create assignments below that scope unless a deny assignment or lock blocks it.

**Verify the target principal**

The assignee must resolve to a valid Microsoft Entra ID object in the correct tenant. The exact identifier accepted by `az role assignment create --assignee` depends on the principal type and the command path, so do not assume one identifier always works.

Use the most reliable identifier for the principal type:

- **User**: UPN or object ID often works
- **Group**: object ID
- **Service principal**: object ID is safest
- **Managed identity**: use the managed identity’s service principal object ID, not the client ID

A common source of confusion is mixing up:

- **app/client ID**: the application registration identifier
- **object ID**: the directory object for the service principal or managed identity
- **principal ID**: often used loosely, but in Azure RBAC contexts it usually means the directory object ID

Check the principal before assigning:

```bash
az ad user show --id user@contoso.com -o jsonc
az ad group show --group "Finance-Admins" -o jsonc
az ad sp show --id <service-principal-object-id> -o jsonc
```

For managed identities, inspect the identity attached to the resource:

```bash
az resource show \
  --ids /subscriptions/<subscription-id>/resourceGroups/<rg>/providers/Microsoft.ManagedIdentity/userAssignedIdentities/<identity-name> \
  -o jsonc
```

> If the principal was created very recently, Azure AD replication can take time. A valid object may still fail to resolve immediately after creation.

**Understand `PrincipalNotFound`**

`PrincipalNotFound` means Azure could not resolve the assignee to a directory object at the time of the request. Typical causes include:

- the object does not exist in the current tenant
- the wrong identifier was used for the principal type
- the principal was deleted or not yet replicated
- the command was run in the wrong tenant context

Troubleshooting steps:

1. Confirm the current tenant with `az account show`.
2. Confirm the principal exists in that tenant.
3. Use the object ID for service principals and managed identities.
4. Retry after a short delay if the object was created moments ago.

Example:

```bash
az role assignment create \
  --assignee-object-id <object-id> \
  --role Reader \
  --scope /subscriptions/<subscription-id>
```

Using `--assignee-object-id` is often clearer than relying on automatic resolution when you already know the directory object ID.

**Check the scope carefully**

Role assignments are scope-specific. A valid assignment can still fail if the scope string is wrong or if you are targeting a scope that does not exist.

Common scopes:

- management group: `/providers/Microsoft.Management/managementGroups/<mg-id>`
- subscription: `/subscriptions/<subscription-id>`
- resource group: `/subscriptions/<subscription-id>/resourceGroups/<rg-name>`
- resource: full resource ID

Validate the scope before assigning:

```bash
az group show -n <rg-name> -o jsonc
az resource show --ids /subscriptions/<subscription-id>/resourceGroups/<rg-name>/providers/Microsoft.Storage/storageAccounts/<name> -o jsonc
```

If the scope is malformed, the command may fail with a not found or invalid scope error rather than a pure authorization error.

**Use the right command and parameters**

Azure CLI, Azure PowerShell, and the portal all perform the same underlying RBAC operation, but parameter behavior differs.

Azure CLI example:

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --role "Reader" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

Azure PowerShell example:

```powershell
New-AzRoleAssignment `
  -ObjectId "<principal-object-id>" `
  -RoleDefinitionName "Reader" `
  -Scope "/subscriptions/<subscription-id>/resourceGroups/<rg-name>"
```

For service principals and managed identities, use the object ID of the service principal representation in Entra ID. Do not use the application registration client ID unless the specific command documentation says it is accepted for that scenario.

**Diagnose permission-related failures**

When the caller lacks permission, the error often resembles:

- `The client '<id>' with object id '<id>' does not have authorization to perform action 'Microsoft.Authorization/roleAssignments/write' over scope ...`
- `AuthorizationFailed`
- `Forbidden`

What to check:

- whether you are signed in with the expected account
- whether the account has Owner or User Access Administrator at the target scope or above
- whether a custom role includes `Microsoft.Authorization/roleAssignments/write`
- whether a deny assignment blocks the action

Useful commands:

```bash
az role definition list --name "User Access Administrator" -o jsonc
az role assignment list --scope /subscriptions/<subscription-id> --include-inherited -o table
```

If you need to inspect a custom role:

```bash
az role definition list --name "<custom-role-name>" -o jsonc
```

Look for these actions in the role definition:

- `Microsoft.Authorization/roleAssignments/write`
- `Microsoft.Authorization/roleAssignments/delete`
- `Microsoft.Authorization/roleAssignments/read`

**Deny assignments and locks**

A deny assignment can block role assignment creation even when the caller appears to have permission through RBAC. Deny assignments are commonly associated with managed applications, Azure Blueprints in older environments, or other governance controls.

Management locks can also interfere, especially when applied at a scope that prevents changes to authorization resources or related parent resources.

Check for locks:

```bash
az lock list --scope /subscriptions/<subscription-id> -o table
```

Check for deny assignments where applicable in your environment. The exact visibility depends on permissions and tooling, but the key point is that a deny assignment overrides allow-based RBAC.

> Do not assume Azure Policy is the usual cause of a failed role assignment. In practice, RBAC permission gaps, deny assignments, and locks are more common causes.

**Propagation and eventual consistency**

After a successful assignment, access may not be immediately usable everywhere. Azure RBAC is eventually consistent, so there can be a delay before the new permission is honored across all services.

Symptoms:

- the assignment appears in the portal or CLI, but access still fails briefly
- a newly added group member cannot access a resource immediately
- a managed identity still receives `403` after assignment

Mitigation:

- wait a few minutes and retry
- sign out and sign back in if the token was issued before the assignment
- refresh the access token for the principal
- verify group membership has also propagated if the assignment is to a group

**Troubleshoot group-based access**

If you assign a role to a group, the group membership must also be correct. A user may not gain access if:

- the user is not actually in the group
- the membership change has not propagated
- the app is using an old token
- the resource provider caches authorization decisions briefly

Check group membership:

```bash
az ad group member list --group "<group-object-id>" -o table
```

Remember that nested group behavior can vary by service and scenario. Do not assume every Azure service evaluates nested group membership the same way.

**Troubleshoot managed identity assignments**

Managed identities are a common source of assignment errors because the identity has multiple identifiers.

Use these rules:

- use the **object ID** of the managed identity’s service principal for RBAC assignment
- do not confuse it with the resource ID of the Azure resource that owns the identity
- do not confuse it with the client ID used by applications

Example:

```bash
az identity show \
  --name <identity-name> \
  --resource-group <rg-name> \
  --query principalId -o tsv
```

Then assign the role:

```bash
az role assignment create \
  --assignee-object-id "$(az identity show --name <identity-name> --resource-group <rg-name> --query principalId -o tsv)" \
  --role Reader \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

**Troubleshooting workflow**

1. Confirm the exact error message and failing scope.
2. Verify the signed-in account and tenant.
3. Check whether the caller has `roleAssignments/write` at the target scope or above.
4. Confirm the assignee exists in the same tenant.
5. Use the correct identifier for the principal type.
6. Validate the scope string and resource existence.
7. Check for deny assignments or locks.
8. Wait for propagation if the assignment just succeeded.
9. Re-test with a minimal scope such as a test resource group if allowed.

**Practical examples**

Assign Reader to a user at resource group scope:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role Reader \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

Assign Contributor to a service principal using object ID:

```bash
az role assignment create \
  --assignee-object-id <service-principal-object-id> \
  --role Contributor \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

List assignments at a resource group:

```bash
az role assignment list \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name> \
  -o table
```

Inspect the role definition for a custom role:

```bash
az role definition list --name "<custom-role-name>" --query "[0].permissions" -o jsonc
```

**Exam focus**

- **Owner** and **User Access Administrator** can usually create role assignments; **Contributor** usually cannot.
- `PrincipalNotFound` points to identity resolution, not just generic permission failure.
- Managed identities and service principals usually require the service principal object ID for assignment.
- A successful assignment may still need time to propagate.
- Deny assignments and locks can block access even when RBAC appears correct.
- The portal, CLI, and PowerShell all depend on the same underlying authorization model.
