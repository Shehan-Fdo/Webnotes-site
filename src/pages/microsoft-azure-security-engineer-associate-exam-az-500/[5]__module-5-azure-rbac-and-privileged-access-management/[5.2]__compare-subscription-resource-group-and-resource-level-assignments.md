## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Scope-based access control in Azure**

Azure Role-Based Access Control (<abbr title="Role-Based Access Control">RBAC</abbr>) uses scope to decide where a role assignment applies. The same role can be assigned at different levels, but the effective permissions depend on the assignment scope and inheritance. In practice, you choose the narrowest scope that still supports the task to reduce blast radius and improve auditability.

Azure scopes are hierarchical:

- **Management group** scope applies across multiple subscriptions.
- **Subscription** scope applies to all resource groups and resources inside that subscription.
- **Resource group** scope applies to all resources in that group.
- **Resource** scope applies only to one resource instance.

A lower scope inherits permissions from higher scopes. For example, a user with `Reader` at subscription scope can view every resource in that subscription, including all resource groups and resources beneath it.

**Subscription-level assignments**

Subscription scope is the broadest scope commonly used in day-to-day administration. It is useful when a team needs consistent access across an entire subscription, such as platform operators, security auditors, or automation identities that manage many resource groups.

Use subscription scope when:

- You need access across multiple resource groups.
- You want to avoid creating many repeated assignments.
- The role is intended for centralized administration or reporting.

Typical examples include:

- `Owner` for a subscription administrator.
- `Contributor` for a deployment pipeline that creates resources across the subscription.
- `Reader` for auditors or monitoring systems.

A subscription-level assignment is powerful because it automatically covers future resource groups and resources created in that subscription. That convenience also increases risk if the role is too permissive.

```bash
# Assign Reader at subscription scope
az role assignment create \
  --assignee user@contoso.com \
  --role Reader \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000
```

**Resource group-level assignments**

Resource group scope is the most common balance between manageability and least privilege. It is appropriate when a team owns a workload boundary, such as a web app and its supporting database, storage account, and network components.

Use resource group scope when:

- A team manages a single application or environment.
- You want permissions to apply to all resources in that workload boundary.
- You need separation between dev, test, and production groups.

This scope is often preferred for application teams because it reduces the need to assign permissions to each resource individually while still limiting access to one workload.

```bash
# Assign Contributor to a resource group
az role assignment create \
  --assignee app-team@contoso.com \
  --role Contributor \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-web-prod
```

Resource group assignments inherit down to all resources in the group. If a user has `Contributor` on the group, they can manage every resource in that group unless a resource-specific deny or policy restriction applies.

**Resource-level assignments**

Resource scope is the most precise form of assignment. It is used when a user or service principal needs access to one specific resource but should not see or manage the rest of the resource group.

Use resource scope when:

- You need tightly limited access to a single resource.
- A service identity only interacts with one storage account, key vault, or virtual machine.
- You want to separate duties within the same resource group.

Common examples:

- Granting a backup service access to one storage account.
- Allowing a support engineer to restart one virtual machine.
- Giving an application identity access to one Key Vault instance.

```bash
# Assign Storage Blob Data Reader to one storage account
az role assignment create \
  --assignee app@contoso.com \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/stprod01
```

Resource-level assignments are precise, but they can become difficult to manage at scale if you repeat them across many resources. They also require careful tracking because the same resource type may exist in multiple groups with different access needs.

**How inheritance changes effective access**

Azure evaluates access by combining assignments from all applicable scopes. A user may receive permissions from multiple role assignments, and the broadest applicable assignment can make a narrow assignment unnecessary.

Key points:

- **Inheritance** flows from parent scope to child scope.
- **Effective access** is the sum of all applicable role assignments.
- A lower-scope assignment does not override a higher-scope assignment; it only adds permissions.
- To reduce access, remove or narrow the higher-scope assignment.

Example:

- `Reader` at subscription scope lets a user view all resources.
- `Contributor` at one resource group scope lets the same user manage resources only in that group.
- The user still has subscription-wide read access because the subscription assignment remains in effect.

This matters during troubleshooting. If a user can access a resource unexpectedly, check for inherited assignments at subscription or management group scope before assuming the resource-level role is too broad.

**Choosing the right scope**

A practical rule is to assign at the highest scope that matches the operational boundary, but no higher.

- Use **subscription** scope for centralized teams, platform operations, and broad read access.
- Use **resource group** scope for application teams and environment-based ownership.
- Use **resource** scope for special cases and tightly controlled service access.

Consider these tradeoffs:

- **Manageability** improves as scope gets broader because you create fewer assignments.
- **Security** improves as scope gets narrower because you reduce unintended access.
- **Operational fit** depends on how your organization groups resources and responsibilities.

A common pattern is:

- Platform team: subscription scope
- App team: resource group scope
- Automation identity: resource scope for a single dependency

**Practical comparison**

| Scope | Applies to | Best for | Risk level | Typical use |
|---|---|---|---|---|
| Subscription | All resource groups and resources in one subscription | Central administration | Higher | Auditing, platform ops, broad deployments |
| Resource group | All resources in one workload boundary | App or environment ownership | Medium | Dev/test/prod team access |
| Resource | One specific resource | Least privilege access | Lower | Service principals, support tasks, single-resource operations |

**CLI examples for each scope**

You can inspect and manage assignments with Azure CLI. The `--scope` value must match the Azure resource ID format for the target level.

```bash
# Subscription scope
/subscriptions/<subscription-id>

# Resource group scope
/subscriptions/<subscription-id>/resourceGroups/<rg-name>

# Resource scope
/subscriptions/<subscription-id>/resourceGroups/<rg-name>/providers/<resource-provider>/<resource-type>/<resource-name>
```

List role assignments at a subscription:

```bash
az role assignment list \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000 \
  --output table
```

List role assignments at a resource group:

```bash
az role assignment list \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-web-prod \
  --output table
```

List role assignments at a resource:

```bash
az role assignment list \
  --scope /subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-web-prod/providers/Microsoft.Web/sites/app01 \
  --output table
```

**Troubleshooting access issues**

When access does not behave as expected, verify scope first.

1. Check the exact resource ID.
2. Review assignments at the resource, resource group, subscription, and management group levels.
3. Confirm the role definition includes the required actions or data actions.
4. Wait for propagation if the assignment was created recently.
5. Sign out and sign back in if the token may still contain stale claims.

Important considerations:

- **Propagation delay** can make a new assignment appear missing for a short time.
- **Token refresh** may be required before the user sees updated permissions.
- **Data plane** permissions may differ from management plane permissions, especially for storage, Key Vault, and Azure SQL.

> A user may have permission to manage a resource in Azure Resource Manager but still lack data access inside the service. Always distinguish management-plane roles from data-plane roles.

**Real-world design patterns**

A few common designs help keep access clean:

- Separate subscriptions for production and nonproduction when governance requires strong isolation.
- Use one resource group per application or environment to align ownership.
- Assign service principals at resource scope when the workload only needs one dependency.
- Use subscription scope for read-only monitoring and compliance tools.

Example topology:

- Subscription `prod-sub`
  - Resource group `rg-payments-prod`
    - App Service `payments-api`
    - Key Vault `kv-payments`
    - Storage account `stpayments`
  - Resource group `rg-shared-prod`
    - Log Analytics workspace
    - Monitoring resources

In this layout:

- The payments team can receive `Contributor` on `rg-payments-prod`.
- A deployment pipeline can receive `Contributor` on `rg-payments-prod` only.
- A secrets-reading application identity can receive a Key Vault role at the `kv-payments` resource scope.
- Security auditors can receive `Reader` at subscription scope.

**Common mistakes**

- Assigning `Owner` at subscription scope when a resource group scope is sufficient.
- Using resource-level assignments everywhere, which becomes hard to maintain.
- Assuming a resource group assignment blocks broader inherited access.
- Forgetting that a subscription assignment still applies to all child resources.
- Confusing Azure RBAC with Azure Policy or resource locks; they solve different problems.

**Exam focus**

For AZ-500, expect questions that ask you to choose the least-privilege scope for a scenario. The correct answer usually depends on the operational boundary:

- One team owns one workload: choose **resource group** scope.
- One identity needs one resource only: choose **resource** scope.
- Centralized administration or auditing across many groups: choose **subscription** scope.

Remember that scope determines where the role applies, while the role definition determines what actions are allowed. Both must align with the security requirement.
