## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Remove Unnecessary Access**

Removing unnecessary access is a core Azure security practice focused on reducing the number of identities, permissions, and pathways that can be used to reach resources. In Azure, this usually means tightening <abbr title="Role-Based Access Control">RBAC</abbr>, removing stale assignments, limiting privileged roles, and reviewing both human and non-human identities. The goal is to enforce <mark>least privilege</mark>: each principal gets only the access required to do its job, for only as long as needed.

This topic appears across identity, governance, and resource security because excess access often comes from multiple layers:
- Azure role assignments at management group, subscription, resource group, or resource scope
- Microsoft Entra ID users, groups, service principals, and managed identities
- Guest accounts and external collaboration
- Shared keys, connection strings, and legacy authentication paths
- Over-permissive network access that bypasses identity controls

**Why it matters**

Excess access increases the blast radius of compromised credentials, insider misuse, and accidental changes. If you remove unnecessary access, you reduce the number of actions an attacker can perform after initial compromise and make privilege escalation harder.

**Core principles**

- **Least privilege**: grant only the minimum permissions needed.
- **Separation of duties**: avoid giving one identity end-to-end control when multiple roles should be involved.
- **Just-in-time access**: provide elevated access only when required, then remove it automatically.
- **Periodic review**: revalidate access because business needs change over time.
- **Deny by default**: prefer explicit grants over broad inherited permissions.

**Common access sources to review**

- **Azure RBAC** assignments
  - Built-in roles such as Owner, Contributor, and Reader
  - Custom roles with broad `Actions` or `DataActions`
  - Inherited permissions from management groups
- **Microsoft Entra ID** permissions
  - Directory roles such as Global Administrator or Privileged Role Administrator
  - App registrations and enterprise applications
  - Group membership that indirectly grants access
- **Resource-level access**
  - Storage account keys
  - SQL authentication accounts
  - Key Vault access policies or RBAC permissions
  - Shared access signatures and connection strings
- **Operational access**
  - Break-glass accounts
  - Automation accounts
  - Managed identities used by apps and scripts

**Azure RBAC fundamentals**

Azure RBAC controls who can perform management operations on Azure resources. Access is evaluated by combining:
- **Security principal**: user, group, service principal, or managed identity
- **Role definition**: set of allowed actions
- **Scope**: management group, subscription, resource group, or resource
- **Assignment**: binding of principal, role, and scope

A role assignment at a higher scope is inherited by child scopes. This is useful for administration, but it is also a common source of unnecessary access when permissions are granted too high in the hierarchy.

**Typical built-in roles and their risk**

| Role | Typical use | Access risk |
|---|---|---|
| Owner | Full management plus role assignment | Very high; can delegate access |
| Contributor | Manage resources but not access control | High; broad operational power |
| Reader | View-only access | Lower, but still exposes metadata |
| User Access Administrator | Manage role assignments | Very high; can expand access |
| Security Admin | Manage security settings in some contexts | High; should be tightly controlled |

**How to identify unnecessary RBAC access**

Review assignments for:
- Direct user assignments where a group would be better
- Broad roles at subscription or management group scope
- Stale assignments for former employees, contractors, or project teams
- Temporary access that was never removed
- Custom roles with wildcard permissions such as `*`
- Data-plane permissions that exceed application needs

Useful CLI commands:

```bash
# List role assignments at subscription scope
az role assignment list --scope /subscriptions/<subscription-id> -o table

# Show assignments for a specific user
az role assignment list --assignee user@contoso.com -o table

# List role definitions
az role definition list -o table

# Inspect a custom role definition
az role definition list --name "Custom Role Name" -o jsonc
```

To remove an unnecessary assignment:

```bash
az role assignment delete \
  --assignee user@contoso.com \
  --role "Contributor" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

**Group-based access is usually safer**

Assign roles to groups instead of individual users whenever possible. This makes access easier to audit and revoke when people change teams.

Example pattern:
- Create a group for each job function or application team
- Assign the Azure role to the group
- Add or remove users from the group as responsibilities change

```bash
az ad group create --display-name "RG-App-Operators" --mail-nickname RG-App-Operators

az role assignment create \
  --assignee-object-id <group-object-id> \
  --assignee-principal-type Group \
  --role Contributor \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

**Privileged access management**

Privileged access should be tightly controlled because it can change security posture, access control, and billing. Use privileged access workflows where available:
- **Privileged Identity Management** <abbr title="Privileged Identity Management">PIM</abbr> for eligible role activation
- Approval workflows for sensitive roles
- Time-bound activation with justification
- MFA (multi-factor authentication) for elevation
- Access reviews for privileged groups and roles

PIM helps remove unnecessary standing access by making users eligible instead of permanently active. This reduces the number of always-on privileged accounts.

Example operational flow:
1. User is made eligible for a role.
2. User requests activation when needed.
3. Activation requires MFA, justification, and possibly approval.
4. Access expires automatically after the time window.

**Access reviews and entitlement cleanup**

Access reviews help you find and remove access that is no longer needed. They are especially useful for:
- Guest users
- Privileged groups
- Application access
- Teams with frequent membership changes

Review questions should focus on business need:
- Does the user still require this role?
- Is the assignment direct or inherited?
- Can the access be replaced by a group or a narrower role?
- Is the scope broader than necessary?
- Is the access temporary, and if so, has it expired?

**Microsoft Entra ID access reduction**

Azure resource access is only part of the picture. Microsoft Entra ID permissions can also create unnecessary access if directory roles are too broad.

Common cleanup targets:
- Global Administrator assignments
- Privileged Role Administrator assignments
- Application Administrator and Cloud Application Administrator assignments
- Guest users with excessive directory permissions
- Users assigned directly to privileged roles instead of using PIM

Useful checks:
- Review role assignments in Entra ID
- Review privileged identity assignments
- Remove unused enterprise applications and app role assignments
- Restrict who can register applications if not required

**Service principals and managed identities**

Applications often use non-human identities. These identities should be reviewed like user accounts.

Best practices:
- Prefer **managed identities** over stored credentials when an Azure resource can authenticate directly.
- Scope service principal permissions narrowly.
- Remove unused app registrations and enterprise applications.
- Rotate secrets and certificates regularly if they are still required.
- Avoid granting directory-wide permissions to automation identities unless absolutely necessary.

Example: assign a managed identity only the permissions it needs on a storage account:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-principal-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>/providers/Microsoft.Storage/storageAccounts/<storage-name>
```

**Storage and data access**

Data-plane access often persists longer than management-plane access because it is embedded in applications and scripts.

Review and reduce:
- Storage account access keys
- Shared access signatures <abbr title="Shared Access Signature">SAS</abbr>
- SQL logins and passwords
- Key Vault secrets that are no longer used
- Connection strings stored in code or configuration files

Prefer:
- Azure AD authentication where supported
- Managed identities for app-to-resource access
- RBAC-based data access for supported services
- Short-lived SAS tokens instead of account keys when temporary access is required

> Avoid using account keys as a long-term access strategy when Azure AD or managed identity is supported. Keys are powerful and difficult to scope.

**Network access as an access path**

Unnecessary access is not only about identity. Network exposure can bypass intended controls or make sensitive services reachable from too many places.

Review:
- Public IP exposure on virtual machines and PaaS services
- Open network security group <abbr title="Network Security Group">NSG</abbr> rules
- Overly broad firewall rules
- Unrestricted service endpoints or private endpoint exceptions
- Management ports exposed to the internet

Example NSG rule review:

```bash
az network nsg rule list \
  --resource-group <rg-name> \
  --nsg-name <nsg-name> \
  -o table
```

If a rule allows `0.0.0.0/0` to a management port, treat it as a high-priority cleanup item unless there is a documented exception.

**Practical cleanup workflow**

1. Inventory all access paths.
   - List role assignments, group memberships, directory roles, and app permissions.
   - Include inherited permissions from parent scopes.

2. Identify broad or stale access.
   - Look for Owner, Contributor, and User Access Administrator at high scope.
   - Find direct user assignments that should be group-based.
   - Check inactive accounts and unused service principals.

3. Validate business need.
   - Confirm the role, scope, and duration with the resource owner.
   - Determine whether the access is temporary or permanent.

4. Reduce scope or remove access.
   - Replace broad roles with narrower built-in or custom roles.
   - Move from direct assignment to group assignment.
   - Remove expired or unused permissions.

5. Recheck after changes.
   - Confirm the workload still functions.
   - Monitor for denied operations that indicate you removed too much.

**Example: narrowing access with a custom role**

If a team only needs to read and restart a specific app service, a custom role may be better than Contributor.

```json
{
  "Name": "App Service Operator Limited",
  "IsCustom": true,
  "Description": "Can read and restart app services but cannot modify access control.",
  "Actions": [
    "Microsoft.Web/sites/read",
    "Microsoft.Web/sites/restart/action",
    "Microsoft.Web/sites/stop/action",
    "Microsoft.Web/sites/start/action"
  ],
  "NotActions": [
    "Microsoft.Authorization/*/write"
  ],
  "AssignableScopes": [
    "/subscriptions/<subscription-id>"
  ]
}
```

Create the role:

```bash
az role definition create --role-definition app-service-operator-limited.json
```

This approach removes unnecessary access by replacing a broad role with a purpose-built one.

**Common exam-relevant distinctions**

- **Owner vs Contributor**
  - Owner can grant access; Contributor cannot.
  - Both are often broader than necessary.
- **RBAC vs directory roles**
  - RBAC controls Azure resource management.
  - Directory roles control Microsoft Entra ID administration.
- **Direct assignment vs group assignment**
  - Direct assignment is harder to audit at scale.
  - Group assignment simplifies lifecycle management.
- **Eligible vs active privileged access**
  - Eligible access reduces standing privilege.
  - Active access is immediately usable and riskier.
- **Management-plane vs data-plane access**
  - Management-plane access changes resources.
  - Data-plane access reads or writes the data inside them.

**Operational examples**

Check who can manage a resource group:

```bash
az role assignment list \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name> \
  --include-inherited \
  -o table
```

Find users with privileged directory roles in Entra ID:
- Review role assignments in the Entra admin center
- Use PIM to make roles eligible instead of permanent
- Remove inactive or unnecessary privileged role holders

Audit a storage account for excessive access:
- Check RBAC assignments on the storage account
- Review shared keys and SAS usage
- Confirm whether applications can use managed identity instead of secrets

**Troubleshooting access removal**

Removing access can break workloads if the original permission set was undocumented. When that happens:
- Check activity logs and application errors.
- Identify the exact action that failed.
- Add the minimum required permission, not the previous broad role.
- Prefer a group or managed identity if the access is for automation.

**Good governance habits**

- Document every privileged assignment and its business justification.
- Use naming conventions for groups and custom roles.
- Review access on a schedule, not only after incidents.
- Remove temporary access immediately after use.
- Treat external users and service principals as high-risk until validated.

**Key takeaway**

Removing unnecessary access is an ongoing control, not a one-time cleanup. In Azure, you reduce risk by tightening RBAC, minimizing privileged directory roles, using PIM and access reviews, preferring managed identities, and eliminating stale or overly broad permissions across both management and data planes.
