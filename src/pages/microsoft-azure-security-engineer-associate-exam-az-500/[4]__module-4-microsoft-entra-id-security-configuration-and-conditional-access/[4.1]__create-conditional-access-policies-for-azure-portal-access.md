## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Conditional Access for Azure portal access**

Conditional Access is Microsoft Entra ID’s policy engine for controlling access based on identity, device, location, risk, and app context. For Azure portal access, you typically target the Microsoft Azure Management cloud app so you can enforce stronger sign-in requirements before users can manage subscriptions, resource groups, and resources.

The Azure portal is not just a website; it is a management surface for privileged operations. That makes it a high-value target for phishing, token theft, and session hijacking. Conditional Access helps you reduce that risk by requiring stronger authentication, limiting access from untrusted networks, and blocking legacy or unmanaged access paths.

- **Microsoft Entra ID** is the identity platform that evaluates Conditional Access policies.
- **Azure Resource Manager** access is commonly controlled through the Microsoft Azure Management app.
- **MFA** means multi-factor authentication, usually required for portal sign-in.
- **Sign-in risk** and **user risk** can be used if you have Microsoft Entra ID P2 licensing.

**How Conditional Access works**

A Conditional Access policy evaluates signals at sign-in and then applies controls if the conditions match. You define:
- **Assignments**: who the policy applies to and what cloud apps it protects.
- **Conditions**: where, how, and from what device the sign-in occurs.
- **Access controls**: what must happen before access is granted.

A typical Azure portal policy might say: if a user signs in to the Azure portal from outside trusted locations, require MFA and a compliant device. If the sign-in is high risk, block it entirely.

<dl>
<dt><strong>Signal</strong></dt>
<dd>A condition Microsoft Entra ID can evaluate, such as user, device state, location, or risk.</dd>
<dt><strong>Grant control</strong></dt>
<dd>An action that allows access only if requirements are satisfied, such as requiring MFA.</dd>
<dt><strong>Session control</strong></dt>
<dd>A control that limits the session after sign-in, such as sign-in frequency or app enforced restrictions.</dd>
</dl>

**Targeting the Azure portal**

For portal access, the most important cloud app is usually:
- **Microsoft Azure Management**: covers Azure portal and Azure Resource Manager operations.

You may also need to consider related admin surfaces:
- **Microsoft Admin Portals** if you want to protect broader Microsoft 365 administration.
- **All cloud apps** if your security baseline requires consistent controls across SaaS and management apps.

> Be careful when targeting broad scopes. A policy that applies to all cloud apps can affect sign-in flows you did not intend to change, especially during rollout.

**Common policy design patterns**

A secure design usually combines several policies rather than one large rule.

- **Baseline portal protection**
  - Require MFA for all users accessing Microsoft Azure Management.
  - Exclude emergency access accounts.
- **Privileged access protection**
  - Require phishing-resistant authentication methods for administrators.
  - Require compliant or hybrid Azure AD joined devices.
- **Location-based hardening**
  - Block access from countries or regions you do not operate in.
  - Require stronger controls from outside trusted IP ranges.
- **Risk-based response**
  - Block high-risk sign-ins.
  - Require password change or MFA for medium-risk users if licensing supports it.

**Recommended rollout approach**

Use a staged deployment so you do not lock out administrators or break automation.

1. Create the policy in **Report-only** mode.
2. Test with a small pilot group.
3. Review sign-in logs and Conditional Access insights.
4. Switch to **On** when behavior matches expectations.
5. Keep at least one break-glass account excluded from all Conditional Access policies.

**Policy creation in the Azure portal**

You usually create the policy in Microsoft Entra admin center.

1. Open **Microsoft Entra admin center**.
2. Go to **Protection** > **Conditional Access** > **Policies**.
3. Select **New policy**.
4. Name the policy clearly, such as `CA - Azure Portal - Require MFA`.
5. Under **Users**, choose the target users or groups.
6. Under **Cloud apps**, select **Microsoft Azure Management**.
7. Configure conditions such as locations, device platforms, or client apps.
8. Under **Grant**, require the desired controls.
9. Set the policy to **Report-only** first.
10. Save, test, and then enable it.

**Example policy: require MFA for Azure portal**

A common starting point is to require MFA for all users except emergency accounts.

```text
Policy name: CA - Azure Portal - Require MFA
Assignments:
  Users: All users
  Exclude: Break-glass accounts, service accounts used for automation
  Cloud apps: Microsoft Azure Management
Conditions:
  Locations: Any location
  Client apps: Browser, Mobile apps and desktop clients
Grant:
  Require multi-factor authentication
Enable policy: Report-only -> On
```

This policy is effective because it protects the management plane without depending on device compliance. It is also easier to deploy than device-based controls, which may require endpoint management maturity.

**Example policy: require compliant device for admins**

For privileged users, you can add device trust requirements.

```text
Policy name: CA - Azure Portal - Admins - Require Compliant Device
Assignments:
  Users: Directory roles = Global Administrator, Privileged Role Administrator, Subscription Owner
  Cloud apps: Microsoft Azure Management
Conditions:
  Device platforms: Any
  Locations: Any
Grant:
  Require multi-factor authentication
  Require device to be marked as compliant
```

This is useful when administrators should only manage Azure from managed endpoints. It reduces the chance that a stolen password and OTP alone can be used from an unmanaged laptop.

**Example policy: block risky sign-ins**

If your tenant has Entra ID P2, you can use risk signals.

```text
Policy name: CA - Azure Portal - Block High Risk Sign-ins
Assignments:
  Users: All users
  Exclude: Break-glass accounts
  Cloud apps: Microsoft Azure Management
Conditions:
  Sign-in risk: High
Grant:
  Block access
```

This policy is strong because it stops access before the session is established. Use it carefully and validate that your risk engine is producing accurate results.

**Grant controls and their use**

Grant controls define what the user must satisfy to get access.

- **Require multi-factor authentication**
  - Best first control for portal access.
  - Works well for broad deployment.
- **Require authentication strength**
  - Lets you demand a specific method, such as phishing-resistant MFA.
  - Useful for admins and sensitive operations.
- **Require device to be marked as compliant**
  - Depends on Microsoft Intune compliance.
  - Good for managed endpoints.
- **Require hybrid Azure AD joined device**
  - Useful in domain-joined enterprise environments.
- **Require approved client app**
  - More relevant for mobile scenarios than browser-based portal access.
- **Block access**
  - Use for high-risk, unsupported, or disallowed access paths.

When you select multiple grant controls, you can choose:
- **Require all selected controls** for stricter enforcement.
- **Require one of the selected controls** when you want flexibility.

**Session controls for portal sessions**

Session controls do not decide whether access is allowed; they shape the session after sign-in.

- **Sign-in frequency**
  - Forces reauthentication after a set interval.
  - Useful for privileged portal sessions.
- **Persistent browser session**
  - Controls whether the browser keeps the session across restarts.
  - Disable persistence for shared or high-risk devices.
- **App enforced restrictions**
  - More relevant to supported SaaS apps than the Azure portal.
- **Use Conditional Access App Control**
  - Typically used with Microsoft Defender for Cloud Apps for deeper session monitoring.

For Azure portal access, sign-in frequency is often the most practical session control because it reduces the lifetime of stolen sessions.

**Trusted locations and network controls**

Location-based policies help you reduce exposure from unmanaged geographies or networks.

- Define **named locations** for trusted office IP ranges.
- Exclude those locations from stricter controls if appropriate.
- Block access from countries where your organization has no business presence.
- Use caution with VPNs and dynamic IPs because they can cause false positives.

Example named location concept:

```text
Named location: Corporate HQ
Type: IP ranges
IP ranges:
  203.0.113.10/32
  203.0.113.11/32
```

A common pattern is:
- Require MFA everywhere.
- Require compliant device outside trusted locations.
- Block access from high-risk countries.

**Break-glass and emergency access accounts**

Emergency access accounts are critical for tenant recovery. They should be:
- Strongly protected with long, unique passwords.
- Monitored continuously.
- Excluded from Conditional Access policies that could lock out administrators.

Use them sparingly and test them regularly. They are not a substitute for normal admin accounts.

> Do not exclude regular administrator accounts just to avoid policy friction. If you need an exception, document the reason and keep the exception as narrow as possible.

**Service principals and automation**

Azure portal policies usually target human users, but automation can be affected if you include service accounts or app-based sign-ins in broad scopes.

- Prefer **workload identity** or managed identities for automation.
- Avoid using shared user accounts for scripts.
- Exclude non-interactive automation only when necessary and well documented.
- Validate that portal-focused policies do not break deployment pipelines or runbooks that authenticate through user context.

Example Azure CLI sign-in for a managed identity scenario:

```bash
az login --identity
az account show
```

Example interactive sign-in that will be subject to Conditional Access:

```bash
az login
az account list --output table
```

**Troubleshooting and validation**

Use sign-in logs and Conditional Access insights to understand policy behavior.

- Check whether the policy was **applied**, **not applied**, or **report-only**.
- Review the exact control that caused failure.
- Confirm the target app is **Microsoft Azure Management**.
- Verify exclusions for emergency accounts and automation.
- Test from different locations and devices to reproduce the issue.

Useful places to inspect:
- **Sign-in logs** in Microsoft Entra ID.
- **Conditional Access** insights and reporting.
- **What If** tool for simulating policy evaluation.

Example CLI to inspect account context before testing portal access:

```bash
az account show --output jsonc
az ad signed-in-user show
```

If a user cannot access the portal, common causes include:
- MFA registration is incomplete.
- The device is not compliant.
- The sign-in comes from a blocked location.
- The user is in a role or group targeted by a stricter policy.
- A legacy authentication path is being used.

**Best practices**

- Start with **Report-only** mode.
- Protect **Microsoft Azure Management** explicitly.
- Require **MFA** for all users as a baseline.
- Add stronger controls for privileged roles.
- Exclude only true emergency accounts.
- Use **named locations** for trusted networks.
- Prefer **phishing-resistant authentication** for admins when available.
- Monitor sign-in logs after every policy change.
- Keep policies small and purpose-built.

**Exam focus**

For AZ-500, expect questions about:
- Which cloud app to target for Azure portal access.
- How to require MFA or compliant devices.
- How to use report-only mode safely.
- Why break-glass accounts must be excluded.
- How location and risk conditions change access decisions.
- The difference between grant controls and session controls.

**Quick reference**

| Item | Purpose | Typical use |
|---|---|---|
| Microsoft Azure Management | Azure portal and ARM access | Protect management plane sign-ins |
| MFA | Stronger user verification | Baseline portal protection |
| Compliant device | Managed endpoint requirement | Admin and privileged access |
| Named location | Trusted IP or geography | Reduce friction from known networks |
| Report-only | Safe testing mode | Validate before enforcement |
| Block access | Hard deny | High-risk or disallowed sign-ins |

**Practical policy blueprint**

A balanced enterprise setup often looks like this:
- All users accessing the Azure portal must complete MFA.
- Administrators must use compliant devices.
- High-risk sign-ins are blocked.
- Access from untrusted locations triggers stronger controls.
- Emergency accounts are excluded and monitored.

This layered approach gives you defense in depth without making the portal unusable for legitimate administration.
