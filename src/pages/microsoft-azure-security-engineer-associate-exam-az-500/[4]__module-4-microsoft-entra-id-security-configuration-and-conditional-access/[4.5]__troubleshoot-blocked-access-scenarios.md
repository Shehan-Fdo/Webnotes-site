## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Blocked access troubleshooting mindset**

Blocked access in Azure usually falls into one of four buckets: identity, authorization, network, or service configuration. You troubleshoot faster when you identify which layer failed before changing anything.

A practical workflow is to start with the exact error, then determine whether the request reached the service, whether the caller was authenticated, and whether the caller was authorized at the correct scope. Many exam scenarios mix more than one issue, such as a valid user who is blocked by both Azure RBAC and a network rule.

- **Identity** issues involve sign-in, token, tenant, guest access, or stale claims.
- **Authorization** issues involve Azure RBAC, access policies, deny assignments, or Azure Policy.
- **Network** issues involve firewalls, NSGs, UDRs, private endpoints, DNS, or service endpoints.
- **Service configuration** issues involve resource-specific settings such as SQL admin, Key Vault mode, or App Service access restrictions.

**Start with the error and the scope**

The same symptom can mean different things depending on whether the request is management plane or data plane. Management plane actions use Azure Resource Manager (ARM), while data plane actions target the service itself, such as reading a secret, querying a storage blob, or connecting to a SQL database.

Use the scope of the failure to narrow the cause:

- If the portal or CLI cannot create, delete, or modify a resource, inspect ARM authorization, policy, and locks.
- If the resource exists but you cannot read data from it, inspect data-plane permissions, network access, and service-specific settings.
- If one user is blocked but another succeeds, compare role assignments, group membership, tenant context, and token freshness.

**Core troubleshooting sequence**

1. Confirm the exact operation and error text.
2. Check whether the caller is in the correct tenant and subscription.
3. Verify the effective permissions at the relevant scope.
4. Check for deny assignments, Azure Policy effects, and resource locks.
5. Validate network reachability and name resolution.
6. Reauthenticate if permissions changed recently.
7. Review service-specific configuration and logs.

**Azure RBAC and scope inheritance**

Azure Role-Based Access Control (RBAC) is the most common authorization layer for blocked access. Permissions can be inherited from management group, subscription, resource group, or resource scope, so a user may appear to have access at one level while being blocked by a higher-level assignment or deny rule.

Use these checks when RBAC is suspected:

- Review role assignments at the exact scope and at parent scopes.
- Check whether the user receives access through a group, PIM (Privileged Identity Management), or eligible assignment that is not currently active.
- Confirm whether the role includes the needed action or data action.
- Look for deny assignments, which override allow permissions.

Example CLI checks:

```bash
# List role assignments at a scope
az role assignment list --scope /subscriptions/<sub-id> --include-inherited -o table

# Check effective assignments for a principal
az role assignment list --assignee <user-object-id> --all -o table

# View deny assignments when supported by your CLI version/context
az role assignment list --include-deny-assignment -o table
```

> A deny assignment can block access even when a user has an allow role. This is especially important in managed applications, some platform-managed resources, and certain delegated scenarios.

**Management group troubleshooting**

Management group scope is easy to miss because permissions and denies can flow down to many subscriptions. If a user can access one subscription but not another, inspect the management group hierarchy first.

Focus on:

- Inherited role assignments from parent management groups.
- Deny assignments applied at management group scope.
- Azure Policy assignments inherited from management groups.
- Subscription placement under the expected management group.

Useful checks:

```bash
# Show management group hierarchy
az account management-group list -o table

# Inspect role assignments at a management group scope
az role assignment list --scope /providers/Microsoft.Management/managementGroups/<mg-id> --include-inherited -o table
```

If a policy or deny assignment exists at a parent management group, the user may see authorization-like failures even though the subscription-level role assignment looks correct.

**Azure Policy and locks**

Azure Policy can block access or resource changes through deny effects, modify effects, or deployIfNotExists behavior. A policy-driven block often looks like a permissions issue because the operation fails even when the user has RBAC rights.

Check for:

- `deny` effects that prevent creation or update.
- Policy assignments at management group, subscription, or resource group scope.
- Initiative assignments that bundle multiple policies.
- Resource locks such as `CanNotDelete` or `ReadOnly`.

Example checks:

```bash
# List policy assignments at a scope
az policy assignment list --scope /subscriptions/<sub-id> -o table

# List locks on a resource group
az lock list --resource-group <rg-name> -o table
```

**Token freshness and reauthentication**

When you change permissions, the user may still be blocked because their access token contains stale claims. This is common after adding a user to a group, activating a PIM role, or granting a new app permission.

Use these steps:

1. Sign out and sign back in.
2. Refresh the browser session or open a private window.
3. Re-run `az login` if using CLI.
4. If needed, clear cached tokens and retry.

Example:

```bash
az logout
az login
az account show
```

For browser-based access, stale sessions can persist until the token refreshes. If the user was recently granted access, always test again after reauthentication.

**Guest and external user issues**

Guest users often fail access checks for reasons that are not obvious from the Azure RBAC view alone. Cross-tenant access settings, B2B (business-to-business) collaboration restrictions, app consent, and tenant context can all block access.

Check:

- Whether the user is a guest in the correct tenant.
- Whether cross-tenant access settings allow inbound collaboration.
- Whether the app or service principal has consent in the target tenant.
- Whether the user is signing into the correct directory in the portal.

Common symptoms include:

- The user can authenticate but cannot see the subscription.
- The user sees the wrong directory or subscription list.
- The app fails to obtain a token for the target tenant.

**Using Activity Log and correlation IDs**

For management-plane failures, the Azure Activity Log is one of the best tools for tracing the request end to end. It shows the operation, status, caller, and often a correlation ID that helps you connect portal, CLI, and backend events.

Use it to answer:

- Did Azure receive the request?
- Which resource provider rejected it?
- Was the failure due to authorization, policy, or another control-plane issue?

Example CLI query:

```bash
az monitor activity-log list \
  --resource-group <rg-name> \
  --status Failed \
  --offset 24h \
  -o table
```

If you have a correlation ID from the portal or error details, search around that event in the Activity Log. This is especially useful when the error message is generic, such as `AuthorizationFailed`, because the log may reveal the exact denied action or policy.

**Network-layer blocks**

Network issues often look like access problems because the client cannot reach the service endpoint. The difference is that network blocks usually prevent connection establishment, while authorization blocks usually fail after the service is reached.

Validate these layers:

- **NSG**: Network Security Group rules on the subnet or NIC.
- **UDR**: User Defined Route that sends traffic to a firewall or NVA.
- **Service endpoint**: Allows traffic from a subnet to a service over the Azure backbone.
- **Private endpoint**: Exposes the service through a private IP.
- **Private DNS**: Resolves the service name to the private endpoint IP.
- **Firewall**: Service firewall or network ACLs on the target resource.

Practical checks:

```bash
# Check effective NSG rules on a NIC
az network nic list-effective-nsg -g <rg-name> -n <nic-name> -o table

# Show route table associations
az network vnet subnet show -g <rg-name> --vnet-name <vnet-name> -n <subnet-name> -o json

# Test DNS resolution from a VM
nslookup <service-name>.privatelink.database.windows.net
```

A useful pattern is to test from a VM in the same network path as the user or workload. If the VM cannot resolve the private name or cannot reach the private IP, the issue is likely network-related rather than RBAC-related.

**Common network validation examples**

- For a private endpoint, confirm the private DNS zone is linked to the VNet.
- For a service endpoint, confirm the subnet has the endpoint enabled and the service firewall allows that subnet.
- For a UDR, confirm traffic is not being forced through a firewall that blocks the destination.
- For an NSG, confirm outbound rules allow the required port and destination.

Example topology check:

```bash
az network private-endpoint list -g <rg-name> -o table
az network private-dns zone list -g <rg-name> -o table
az network private-dns link vnet list -g <rg-name> -z privatelink.database.windows.net -o table
```

**Storage access troubleshooting**

Azure Storage supports multiple authorization models, and the model in use determines what you should check first. A user may be blocked because they are using Azure AD authentication, a shared key, or a SAS (Shared Access Signature) token, each with different failure modes.

Identify the access method:

- **Azure AD auth** uses RBAC roles such as `Storage Blob Data Reader` or `Storage Blob Data Contributor`.
- **Shared key** uses the account key and bypasses Azure AD RBAC for data access.
- **SAS** uses a signed token with time, permission, and resource constraints.

Common checks:

- Confirm whether the client is using Azure AD, key, or SAS.
- Verify the correct data-plane role if Azure AD is used.
- Check whether the storage account firewall allows the client network.
- Confirm whether hierarchical namespace or container-level permissions affect the operation.

Example CLI:

```bash
# Check storage account network rules
az storage account network-rule list -g <rg-name> -n <storage-account> -o table

# Check blob data-plane role assignments
az role assignment list \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg-name>/providers/Microsoft.Storage/storageAccounts/<storage-account> \
  --include-inherited -o table
```

If a user can list the storage account in ARM but cannot read blobs, the issue is often data-plane RBAC or network access, not management-plane access.

**Key Vault access troubleshooting**

Azure Key Vault has two authorization models: access policy mode and Azure RBAC mode. The troubleshooting path depends on which mode the vault uses.

In access policy mode:

- Secret, key, and certificate permissions are granted separately.
- A user may have secret access but not key or certificate access.
- Portal actions can fail if the user lacks the specific permission for the object type.

In Azure RBAC mode:

- Use built-in roles such as `Key Vault Secrets User`, `Key Vault Reader`, `Key Vault Crypto User`, and related roles.
- Do not assume a generic “reader” role covers all object types.
- Confirm the role is assigned at the vault scope or inherited correctly.

Common portal-specific patterns:

- The portal can show the vault but fail to list secrets.
- The portal can open the vault blade but fail when you click a secret, key, or certificate.
- A user may see `Forbidden` even though they have a management-plane role like Contributor, because that role does not grant data-plane access.

Example checks:

```bash
# Show Key Vault access configuration
az keyvault show -n <vault-name> --query properties.enableRbacAuthorization

# List access policies if the vault uses access policy mode
az keyvault show -n <vault-name> --query properties.accessPolicies

# List RBAC assignments on the vault
az role assignment list --scope /subscriptions/<sub-id>/resourceGroups/<rg-name>/providers/Microsoft.KeyVault/vaults/<vault-name> -o table
```

> In Key Vault, secret, key, and certificate permissions are distinct. A user who can read secrets may still be blocked from cryptographic operations or certificate management.

**SQL access troubleshooting**

Azure SQL access issues often involve a mix of server configuration, authentication, and network access. The error message matters: `Cannot open server requested by the login` usually points to server name, login context, or authentication problems, while firewall or network issues more often appear as connection timeouts or TCP connection failures.

Check these items:

- The server name and fully qualified domain name are correct.
- The user has the right authentication method.
- The Azure AD admin is configured if using Azure AD authentication.
- The contained database user exists in the target database.
- Public network access and firewall rules allow the client, or private endpoint/DNS is configured correctly.

Example SQL troubleshooting flow:

1. Confirm the server name and database name.
2. Verify the login type: SQL authentication or Azure AD authentication.
3. Check whether the Azure AD admin is set on the server.
4. Confirm the user exists as a contained user in the database.
5. Validate firewall rules or private connectivity.

Useful commands:

```bash
az sql server show -g <rg-name> -n <sql-server> -o json
az sql server ad-admin list -g <rg-name> -s <sql-server> -o table
az sql server firewall-rule list -g <rg-name> -s <sql-server> -o table
```

For Azure AD-authenticated access, ensure the user has been created in the database:

```sql
CREATE USER [user@contoso.com] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [user@contoso.com];
```

**App Service, AKS, and other service-specific blocks**

Blocked access is not limited to storage, Key Vault, or SQL. Many Azure services have their own access controls that can look like generic authorization failures.

Examples:

- **App Service** may block requests through access restrictions, authentication settings, or private endpoint configuration.
- **AKS** (Azure Kubernetes Service) may block cluster access through Azure AD integration, Kubernetes RBAC, or network restrictions.
- **Azure Firewall** or application gateways may block traffic before it reaches the target service.
- **Logic Apps**, **Functions**, and **API Management** can fail due to identity, network, or policy settings.

For App Service, check:

- Access restrictions on the app.
- Authentication and authorization settings.
- Private endpoint and DNS resolution.
- Whether the caller is using the correct identity.

For AKS, check:

- Azure AD group membership and cluster role bindings.
- Whether the kubeconfig is current.
- Network access to the API server.
- Whether the cluster uses private API server access.

Example AKS checks:

```bash
az aks show -g <rg-name> -n <aks-name> --query apiServerAccessProfile -o json
az aks get-credentials -g <rg-name> -n <aks-name> --overwrite-existing
kubectl auth can-i get pods -A
```

**Reading common error patterns**

Some errors are broad and require context:

- `AuthorizationFailed` often indicates an ARM authorization failure, but similar authorization-style errors can also appear in service-specific contexts.
- `Forbidden` usually means the request reached the service but the caller lacked permission.
- `ResourceNotFound` can mean the resource does not exist, the caller lacks visibility, or the name is wrong.
- `Timeout` or connection failures usually point to network, DNS, or firewall issues.

Use the error plus the request path to decide whether to inspect RBAC, policy, or network first.

**Practical decision tree**

1. If the request never connects, check DNS, NSG, UDR, firewall, private endpoint, and service endpoint settings.
2. If the request connects but fails immediately with forbidden or unauthorized, check RBAC, access policies, deny assignments, and token freshness.
3. If the request works for one identity but not another, compare tenant, group membership, PIM activation, and role scope.
4. If the request fails only in the portal, compare portal context, browser session, and service-specific blades with CLI output.
5. If the request fails after a recent permission change, force reauthentication and retry.

**Exam-focused reminders**

- Management group scope can override lower-scope expectations through inheritance.
- Azure Policy can block actions even when RBAC looks correct.
- Deny assignments override allow assignments.
- Guest users may need tenant, cross-tenant, and consent checks.
- Storage, Key Vault, and SQL each have distinct data-plane models.
- Network blocks often require DNS validation, not just firewall checks.
- Recent permission changes may not apply until the token refreshes.
