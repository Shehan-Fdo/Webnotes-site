## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Microsoft Defender for Cloud is Azure’s cloud security posture management and cloud workload protection platform. For exam purposes, you need to distinguish three related but different concepts:

- **Security alerts**: detections of suspicious or malicious activity on protected resources.
- **Recommendations**: posture findings that tell you how to reduce risk, such as enabling a control or fixing a configuration.
- **Incidents**: correlated cases, usually created in Microsoft Sentinel or Microsoft Defender XDR when alerting is integrated and analytics rules or correlation logic are in place.

A common exam trap is assuming every alert automatically becomes an incident or that every recommendation is an alert. They are different outputs with different remediation paths.

**Where to find alerts in the Azure portal**

In the Azure portal, you typically review Defender for Cloud alerts from the Defender for Cloud blade rather than from generic Azure Activity Log views.

Typical navigation path:

1. Open the Azure portal.
2. Search for and open **Microsoft Defender for Cloud**.
3. In the left menu, go to **Security alerts** or open the relevant **Environment settings** / subscription context and then view alerts for that scope.
4. Select a specific alert to open its details pane.
5. Review the alert context, affected resource, severity, and recommended response.

Depending on the portal layout and your tenant configuration, alerts may also be surfaced through connected experiences such as Microsoft Sentinel, Microsoft Defender XDR, or partner integrations. The exact entry point can vary, but the alert details and response workflow remain similar.

**Alert severity and exam meaning**

Defender for Cloud uses severity to help you prioritize response. Treat severity as a triage signal, not as the only decision factor.

| Severity | Typical meaning | Exam-oriented response |
|---|---|---|
| High | Strong indication of active compromise or serious malicious behavior | Investigate immediately, contain if needed, and validate scope |
| Medium | Suspicious activity that may indicate attack progression or risky behavior | Investigate promptly and correlate with other telemetry |
| Low | Lower-confidence or lower-impact suspicious activity | Review for context, false positives, or early indicators |
| Informational | Useful context or benign-but-relevant activity | Use for hunting, baselining, or correlation |

Severity does not guarantee impact. A low-severity alert can still matter if it affects a privileged identity, a production subscription, or a sensitive workload.

**Alert, incident, and recommendation differences**

Use this mental model during the exam:

- **Alert**: “Something suspicious happened.”
- **Incident**: “Several signals were correlated into a case.”
- **Recommendation**: “Your configuration increases risk; fix it.”

A security alert may be investigated directly in Defender for Cloud, or it may be forwarded into another system for correlation. Whether it becomes an incident depends on the connected product and rule configuration.

**What to inspect in an alert**

When you open an alert, focus on the fields that help you decide whether the activity is benign, suspicious, or confirmed malicious.

- **Alert name**: identifies the detection logic or threat pattern.
- **Severity**: helps prioritize response.
- **Affected resource**: tells you which VM, storage account, database, subscription, or identity is involved.
- **Subscription and resource group**: helps you determine blast radius and ownership.
- **Time detected**: lets you correlate with change windows, deployments, and sign-in activity.
- **Description and evidence**: explains why the alert fired.
- **Recommended actions**: suggests next steps for investigation or containment.
- **Related entities**: may include IP addresses, users, processes, or resource identifiers.
- **Status or state**: indicates whether the alert is still open, handled, suppressed, or otherwise closed according to the source workflow.

> Alert state names and available actions can differ by alert source and integration. Do not assume one universal state model across all Defender for Cloud-connected alerts.

**Practical portal review workflow**

Use a consistent triage sequence when reviewing a real alert.

1. Open the alert from **Microsoft Defender for Cloud**.
2. Confirm the **scope**: subscription, resource group, and affected resource.
3. Read the **summary** to understand what triggered the detection.
4. Check the **timestamp** and compare it to deployments, maintenance, or user activity.
5. Review the **evidence** and any related entities.
6. Determine whether the activity matches an expected admin task, automation, or known service behavior.
7. Decide whether to investigate further, suppress future noise, or close the alert according to the source workflow.

A useful exam habit is to ask: “Is this a real security signal, or is it expected operational activity?” That question often separates a correct triage answer from a superficial one.

**How to interpret common scenarios**

A few realistic examples help you reason about alerts.

- A high-severity alert on a production virtual machine with an unfamiliar process and an external IP is more likely to require containment.
- A medium-severity alert on a build server during a scheduled patch window may be benign if the evidence matches approved automation.
- A low-severity alert on a test subscription may still matter if the resource has access to production secrets or managed identities.

When in doubt, correlate with:
- Azure Activity Log for control-plane changes
- sign-in logs for identity activity
- workload-specific logs for process, network, or data access evidence
- change management records for approved operations

**Suppression and noise reduction**

Suppression is used to reduce repeated alerts that are expected and verified as benign. In Defender for Cloud, suppression is typically configured from the alert or from alert suppression settings in the Defender for Cloud experience, depending on the alert type and portal layout.

Common suppression conditions can include:
- specific alert name or rule
- resource scope such as subscription or resource group
- resource type
- source or affected entity patterns
- time-based conditions for recurring maintenance windows

Use suppression carefully. Suppressing too broadly can hide real attacks. Prefer narrow scope and documented justification.

**When to dismiss, resolve, or suppress**

Use the following decision rule:

| Action | Use when | Result |
|---|---|---|
| **Dismiss** | The alert is a false positive or no further action is needed in the alert workflow | Closes the alert in the current workflow, if supported |
| **Resolve** | The issue was investigated and remediated, and the alert source supports a resolved state | Marks the alert as handled according to the source system |
| **Suppress** | The alert is expected to recur and is confirmed benign under specific conditions | Prevents repeated noise for matching future alerts |

The exact labels available can vary by alert source and integration. For exam questions, choose the action that best matches the operational intent:
- false positive now → dismiss
- fixed issue after investigation → resolve
- recurring benign pattern → suppress

**Export and integration options**

Defender for Cloud alert export is not automatic in every environment. You usually need to configure the destination and the export path explicitly.

Common integration targets include:
- **Log Analytics workspace**
- **Event Hub**
- **Microsoft Sentinel**
- **Microsoft Defender XDR**
- partner SIEM/SOAR solutions through supported connectors or continuous export

Before relying on exported alerts, verify:
- the relevant subscription or management group is in scope
- continuous export or diagnostic settings are enabled where required
- the destination exists and has permissions
- the connected product is configured to ingest the alert type you expect

A practical pattern is to export alerts to a central workspace or SIEM for correlation, while still using Defender for Cloud for the initial security posture and alert review.

**CLI and query examples for investigation**

Use CLI and KQL (Kusto Query Language) to support investigation, but keep in mind that schemas vary by connector and table.

Azure CLI can help you inspect resource metadata and recent changes, but it is not a universal way to “list Defender for Cloud alerts” unless you are using the specific APIs or connected services that expose them.

```bash
# Show the resource details for an affected VM
az vm show \
  --resource-group rg-prod-app \
  --name vm-app01 \
  --query "{name:name, location:location, provisioningState:provisioningState}" \
  -o json

# Review recent control-plane activity for a resource group
az monitor activity-log list \
  --resource-group rg-prod-app \
  --offset 24h \
  --query "[].{time:eventTimestamp, operation:operationName.value, status:status.value, caller:caller}" \
  -o table
```

For log-based investigation, use the table that actually contains your connected alert data. In many environments, Defender for Cloud alerts may be available through Microsoft Sentinel or Microsoft Defender XDR connectors rather than a single universal Log Analytics table.

```kusto
// Example pattern for investigating security alerts in a connected workspace
SecurityAlert
| where TimeGenerated > ago(24h)
| project TimeGenerated, AlertName, Severity, ProviderName, ProductName, CompromisedEntity
| order by TimeGenerated desc
```

If your environment uses a different table or schema, adapt the query to the connector in use. Do not assume every alert source populates the same fields.

For identity-focused investigation, correlate with sign-in logs:

```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| where UserPrincipalName == "alice@contoso.com"
| project TimeGenerated, AppDisplayName, IPAddress, ResultType, ConditionalAccessStatus
| order by TimeGenerated desc
```

For host investigation, use the table available in your environment and verify that process fields are actually collected before querying them.

```kusto
// Example pattern: only use fields that exist in your collected schema
DeviceProcessEvents
| where TimeGenerated > ago(24h)
| where DeviceName == "vm-app01"
| project TimeGenerated, FileName, ProcessCommandLine, InitiatingProcessFileName
| order by TimeGenerated desc
```

**Microsoft Defender XDR and Sentinel integration**

In connected environments, Defender for Cloud alerts can feed broader detection and response workflows.

- **Microsoft Defender XDR** can unify alerts across endpoints, identities, email, and cloud workloads.
- **Microsoft Sentinel** can ingest alerts and use analytics rules to create incidents, correlate events, and trigger automation.
- Whether an alert becomes an incident depends on the connector and rule configuration; it is not automatic in every setup.

For exam purposes, remember that Defender for Cloud is often the source of the alert, while Sentinel or Defender XDR may be the place where correlation and incident management happen.

**Exam-focused troubleshooting mindset**

If a question asks you what to do with a suspicious alert, think in this order:

- Confirm the affected asset and scope.
- Check severity and evidence.
- Correlate with recent changes and identity activity.
- Decide whether the alert is expected, suspicious, or confirmed malicious.
- Choose the correct action: investigate, resolve, dismiss, or suppress.
- If needed, route the alert into a central monitoring platform for correlation and response.

**Key takeaways**

- Defender for Cloud alerts are security detections, not posture recommendations.
- Portal review starts in **Microsoft Defender for Cloud** and then drills into the alert details.
- Severity guides priority, but context determines response.
- Suppression is for recurring benign noise; use it narrowly.
- Export and incident creation depend on explicit integration and configuration.
- Always validate the actual table, connector, and schema before using KQL in a real environment.
