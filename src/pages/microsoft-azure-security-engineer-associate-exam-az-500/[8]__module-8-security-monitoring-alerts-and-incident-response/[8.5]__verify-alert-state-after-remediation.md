## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Purpose of verification after remediation**

After you fix a security issue, you do not stop at the change itself. You verify that the platform has re-evaluated the resource, that the alert or incident has moved to the expected state, and that the underlying exposure is no longer present. In AZ-500 scenarios, the key is to distinguish between the remediation action and the service-specific state change that proves the fix worked.

Verification usually spans three layers:

- **Alert state**: whether the detection is still firing, resolved, or no longer active.
- **Incident state**: whether related alerts have been grouped and closed in a SIEM/SOAR workflow such as Microsoft Sentinel.
- **Recommendation state**: whether a security control check in Defender for Cloud now shows healthy or not applicable after reassessment.

A common exam trap is assuming that a successful configuration change instantly updates every portal. In practice, each service has its own evaluation cycle and state model.

**Core state model to remember**

| Concept | What it represents | Typical state terms | Where you verify it |
|---|---|---|---|
| **Alert** | A detection condition on a resource or log stream | Fired, resolved, active, inactive | Azure Monitor, Defender for Cloud, Sentinel |
| **Incident** | A grouped security case made from one or more alerts | New, active, closed | Microsoft Sentinel |
| **Recommendation** | A posture or configuration finding | Healthy, unhealthy, not applicable | Microsoft Defender for Cloud |
| **Secure score** | Aggregate security posture metric | Score increases after reassessment | Defender for Cloud |

Use this distinction to answer questions about what changed, where to check it, and how long to wait before concluding the fix failed.

**Defender for Cloud verification**

Defender for Cloud is often used to validate whether a security recommendation is still applicable after you remediate a resource. The service does not usually mark a recommendation as “resolved” in the same way a ticketing system would. Instead, the recommendation becomes healthy, compliant, or no longer applicable after the next assessment cycle.

Typical verification path in the portal:

1. Open **Microsoft Defender for Cloud**.
2. Go to **Recommendations** or **Security posture**.
3. Select the affected subscription, resource group, or resource.
4. Open the recommendation and review the affected resources list.
5. Check whether the resource now appears as healthy, compliant, or removed from the affected set.

Common examples:

- If you enable a missing security feature, the recommendation may remain visible briefly until the next assessment.
- If you remove the underlying exposure, the recommendation may shift to **Not applicable** rather than disappearing immediately.
- If the resource is deleted, the recommendation may disappear after inventory refresh.

> Defender for Cloud reassessment is not always immediate. Expect a delay from several minutes to longer, depending on the control, resource type, and assessment pipeline.

Practical CLI checks for posture-related verification:

```bash
# List security assessments for a resource
az security assessment list --resource-group MyRG --output table

# Show a specific assessment by name
az security assessment show \
  --name <assessment-name> \
  --resource-group MyRG \
  --output jsonc

# List secure score controls
az security secure-score-control list --output table
```

For some resource types, you may also inspect the resource configuration directly to confirm the control is actually in place. For example, if the recommendation was about storage network exposure, verify the storage settings rather than relying only on the recommendation label.

**Azure Monitor alert verification**

Azure Monitor alerts are typically verified by checking whether the alert instance has transitioned from **Fired** to **Resolved**. Azure Monitor does not usually use “closed” as the standard lifecycle term for an alert instance. “Closed” is more common in incident management systems.

Portal navigation:

1. Open **Azure Monitor**.
2. Select **Alerts**.
3. Review **Alert history** or **Alert rules**.
4. Open the alert instance and confirm the current state.
5. Check the signal source, such as metrics or logs, to ensure the condition no longer exists.

For log alerts, the alert may remain fired until the query no longer matches the threshold window. For metric alerts, the metric must fall back below the threshold for the configured evaluation periods.

Example log alert rule for failed sign-ins or suspicious activity:

```kusto
SigninLogs
| where ResultType != 0
| summarize FailedCount = count() by bin(TimeGenerated, 5m), UserPrincipalName
| where FailedCount >= 5
```

Example metric alert concept:

```bash
az monitor metrics alert create \
  --name HighCpuAlert \
  --resource-group MyRG \
  --scopes /subscriptions/<sub-id>/resourceGroups/MyRG/providers/Microsoft.Compute/virtualMachines/MyVM \
  --condition "avg Percentage CPU > 80" \
  --description "CPU above threshold"
```

Verification tips:

- Confirm the alert is no longer **firing** after the fix.
- Recheck the metric or log query window to ensure the condition has cleared.
- Allow for evaluation latency; the alert may take one or more evaluation cycles to resolve.

**Microsoft Sentinel incident verification**

Microsoft Sentinel groups alerts into incidents. After remediation, you often need to verify both the underlying alert and the incident state. An incident can remain open even after the original alert condition is gone if it has not been manually closed or auto-closed by automation.

Portal navigation:

1. Open **Microsoft Sentinel**.
2. Select the workspace.
3. Go to **Incidents**.
4. Open the incident and review the **Alerts** tab.
5. Confirm whether the incident is still active, has been closed, or has been reopened by new evidence.

Sentinel incident lifecycle is useful for exam questions because it separates detection from case management:

- **Alert**: the detection event.
- **Incident**: the investigation container.
- **Closure**: the analyst or automation decision after remediation and validation.

If you use automation, a playbook may close an incident after a successful remediation check. If not, you may need to close it manually after confirming the issue is fixed.

Example KQL for validating a security event after remediation:

```kusto
SecurityEvent
| where EventID == 4728 or EventID == 4732
| where TimeGenerated > ago(24h)
| project TimeGenerated, Computer, Account, Activity
```

For Entra ID role-related events, use the fields that match the audit schema more closely:

```kusto
AuditLogs
| where ActivityDisplayName in ("Add member to role", "Add eligible member to role", "Remove member from role")
| project TimeGenerated, ActivityDisplayName, Result, InitiatedBy, TargetResources
```

This is more reliable for exam prep than loosely filtering on text fragments in `OperationName`.

**State verification workflow**

Use a consistent workflow so you do not confuse remediation with validation.

1. Apply the fix.
2. Confirm the resource configuration directly.
3. Wait for the relevant service to reassess.
4. Check the alert, incident, or recommendation state.
5. Confirm the issue is no longer observable in logs, metrics, or posture data.
6. Document the result and close the case if required.

A practical example for a storage exposure issue:

```bash
# Check storage account network settings
az storage account show \
  --name mystorageacct \
  --resource-group MyRG \
  --query "{publicNetworkAccess:publicNetworkAccess, defaultAction:networkRuleSet.defaultAction, allowBlobPublicAccess:allowBlobPublicAccess}" \
  -o jsonc
```

Interpretation:

- `publicNetworkAccess` controls whether the storage account accepts traffic from public endpoints.
- `networkRuleSet.defaultAction` controls the default network rule behavior, such as `Allow` or `Deny`.
- `allowBlobPublicAccess` controls anonymous blob access and is separate from network access.

Do not treat one setting as proof that all public exposure is disabled.

**Common remediation-to-verification scenarios**

| Scenario | What you fix | What you verify | Expected state change |
|---|---|---|---|
| Missing security control in Defender for Cloud | Enable the control or configuration | Recommendation details and affected resources | Healthy, compliant, or not applicable after reassessment |
| Suspicious activity alert in Azure Monitor | Remove the triggering condition | Alert instance and query results | Fired to resolved |
| Security incident in Sentinel | Contain and remediate the root cause | Incident and related alerts | Active to closed |
| Storage exposure finding | Change network or access settings | Resource configuration and recommendation | Finding no longer applicable or healthy |

**Latency and timing expectations**

Different services update at different speeds, and AZ-500 questions often test whether you know that delay is normal.

- **Azure Monitor**: alert resolution depends on the next evaluation cycle and the alert type.
- **Defender for Cloud**: reassessment can take minutes or longer, especially for posture recommendations.
- **Sentinel**: incidents do not automatically close just because the signal stops; closure may require manual action or automation.
- **Resource configuration checks**: direct CLI or portal inspection is often the fastest way to confirm the fix immediately.

If you need a quick proof point, validate the resource configuration first, then wait for the service state to catch up.

**Practical portal paths to remember**

- **Defender for Cloud**
  - **Microsoft Defender for Cloud** > **Recommendations**
  - **Microsoft Defender for Cloud** > **Security alerts**
  - **Microsoft Defender for Cloud** > **Secure score**
- **Azure Monitor**
  - **Azure Monitor** > **Alerts**
  - **Azure Monitor** > **Metrics**
  - **Azure Monitor** > **Logs**
- **Microsoft Sentinel**
  - **Microsoft Sentinel** > **Incidents**
  - **Microsoft Sentinel** > **Analytics**
  - **Microsoft Sentinel** > **Logs**

These paths matter because the same issue can appear differently depending on the service. A posture issue may show in Defender for Cloud, while the same activity may also generate an alert in Azure Monitor or an incident in Sentinel.

**CLI examples for verification**

Defender for Cloud recommendation and assessment checks:

```bash
az security assessment list --resource-group MyRG --output table
az security assessment show --name <assessment-name> --resource-group MyRG --output jsonc
```

Azure Monitor log query validation:

```kusto
AzureActivity
| where ActivityStatusValue == "Succeeded"
| where OperationNameValue has "Microsoft.KeyVault/vaults"
| project TimeGenerated, OperationNameValue, ResourceGroup, Caller
```

Sentinel incident review via Azure CLI is limited compared with the portal, but you can still inspect workspace data and use KQL to validate whether the triggering activity persists.

Storage verification:

```bash
az storage account show \
  --name mystorageacct \
  --resource-group MyRG \
  --query "{publicNetworkAccess:publicNetworkAccess, allowBlobPublicAccess:allowBlobPublicAccess, defaultAction:networkRuleSet.defaultAction}" \
  -o jsonc
```

Key Vault private endpoint connection verification:

```bash
az keyvault private-endpoint-connection list \
  --vault-name myvault \
  --resource-group MyRG \
  -o table
```

If needed, inspect a specific connection:

```bash
az keyvault private-endpoint-connection show \
  --vault-name myvault \
  --name <connection-name> \
  --resource-group MyRG \
  -o jsonc
```

**Exam-focused distinctions**

- **Alert** is about detection.
- **Incident** is about investigation and response.
- **Recommendation** is about posture and configuration.
- **Secure score** is about aggregate security improvement.

A common AZ-500 pattern is to ask what you should check after remediation. The correct answer depends on the service:

- For a Defender for Cloud recommendation, check the recommendation status and secure score impact.
- For an Azure Monitor alert, check whether the alert instance resolved.
- For a Sentinel incident, check whether the incident was closed and whether related alerts stopped recurring.
- For a resource configuration issue, verify the actual setting directly before waiting for portal reassessment.

**End-to-end remediation examples**

**Example 1: Storage account exposure**

1. You disable public access or tighten network rules.
2. You confirm the storage account settings with `az storage account show`.
3. You wait for Defender for Cloud to reassess the recommendation.
4. You confirm the recommendation is now healthy or not applicable.
5. You verify no new alert is firing in Azure Monitor or Sentinel.

**Example 2: Suspicious sign-in alert**

1. You block the account or reset credentials.
2. You query `SigninLogs` to confirm failed or suspicious attempts stop.
3. You check Azure Monitor alert history for resolution.
4. You review the Sentinel incident and close it if no further evidence appears.
5. You document the remediation and verification time, since log-based alerts may lag.

**Example 3: Key Vault access issue**

1. You correct the access policy, RBAC role, or private endpoint configuration.
2. You verify the effective configuration with CLI or portal.
3. You check the Key Vault private endpoint connection state if private access is involved.
4. You confirm any related alert or recommendation clears after reassessment.

**Key takeaways**

- Verify the **resource configuration** first, then the service state.
- Use the correct lifecycle term for each service: resolved for alerts, closed for incidents, healthy or not applicable for recommendations.
- Expect **reassessment latency** and do not assume instant portal updates.
- Use KQL and CLI to prove the issue is gone, not just hidden.
- Separate public network access, anonymous data access, and security posture controls when validating storage or similar resources.
