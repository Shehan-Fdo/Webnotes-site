## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Incident-response rotation goals**

When a secret, key, or credential may be exposed, your goal is to reduce attacker dwell time, invalidate what you can, and preserve service continuity. In Azure, that usually means you rotate credentials, disable or reassign identities, and verify every dependent workload still authenticates correctly.

A good response plan separates three concerns:

- **Containment**: stop further use of the compromised credential.
- **Recovery**: replace the credential and update dependent systems.
- **Validation**: confirm the old path no longer works and the new path does.

In practice, you often rotate more than one thing at once. For example, if an app registration secret is exposed, you may also need to rotate any Key Vault secret that stores it, update App Service settings, and review logs for misuse.

**What to rotate first**

Prioritize based on blast radius and what the attacker can use immediately.

- **User credentials**: reset passwords, revoke sessions, and enforce MFA if user compromise is suspected.
- **Application credentials**: rotate client secrets, certificates, API keys, and storage access keys.
- **Data-plane keys**: rotate storage account keys, Cosmos DB keys, SQL credentials, and other service keys.
- **Signing material**: rotate cryptographic keys used to sign or encrypt data when compromise affects trust boundaries.

If you suspect active compromise, disable the identity or access path first when possible, then rotate. Rotation alone does not always stop an attacker who already has a valid token or session.

**Identity-first containment**

Managed identities are often the best long-term answer because they remove stored secrets from code and configuration. If a managed identity is compromised through workload access or privilege abuse, you usually respond by reducing its permissions, isolating the workload, or disabling the resource that hosts it rather than “rotating” a secret.

- **System-assigned managed identity**: tied to one Azure resource; deleting the resource removes the identity.
- **User-assigned managed identity**: can be detached from one workload and attached elsewhere; useful for controlled reassignment during recovery.
- **Why it matters**: if you can replace a stored secret with a managed identity, you reduce future rotation burden.

> If an attacker has already obtained a bearer token, rotating the underlying secret does not instantly invalidate that token. You still need to revoke sessions, disable access, or wait for token expiry where applicable.

**Azure Key Vault as the rotation control plane**

Azure Key Vault is the central place to store and rotate secrets, keys, and certificates. Use it to reduce hard-coded credentials and to standardize rotation workflows.

- **Secrets**: passwords, API keys, connection strings, client secrets.
- **Keys**: cryptographic keys for encryption, signing, or wrapping.
- **Certificates**: X.509 certificates for TLS or client authentication.

Key Vault supports versioning, so you can create a new version without deleting the old one immediately. That lets you stage a cutover and test the new value before retiring the old one.

```bash
# Create or update a secret version
az keyvault secret set \
  --vault-name kv-prod \
  --name app-db-password \
  --value 'NewStrongPassword!2026'

# Create a new key version
az keyvault key create \
  --vault-name kv-prod \
  --name app-signing-key \
  --kty RSA \
  --size 2048

# Create a new certificate version from a policy
az keyvault certificate create \
  --vault-name kv-prod \
  --name tls-cert \
  --policy @cert-policy.json
```

For cryptographic keys, rotation means creating a new key version and updating the consumers that reference the key. This is different from secret rotation, where the application often reads a new value and reconnects.

**Secrets versus keys versus certificates**

| Item | Typical use | Rotation pattern | Common risk |
|---|---|---|---|
| Secret | App password, API key, connection string | Create new version, update consumers, retire old version | Stored in code, config, or CI/CD |
| Key | Encryption/signing/wrapping | Create new key version, re-encrypt or rewrap as needed | Old data may still depend on old key version |
| Certificate | TLS, client auth, signing | Issue new cert, deploy, validate chain, retire old cert | Expired certs break service availability |

A key point for the exam is that cryptographic keys are not just “secrets with a different name.” They may protect data that must remain decryptable, so you need a migration plan before disabling the old version.

**Rotating application secrets**

Application secrets usually appear in app registrations, service principals, CI/CD pipelines, or external systems that authenticate to Azure.

Use current Microsoft Entra ID and Microsoft Graph-based approaches where possible. The older `az ad` workflows exist in some environments, but they are not the best choice for new guidance and can vary by CLI version.

Common rotation patterns include:

- Add a new credential first.
- Update the application or pipeline to use the new credential.
- Validate authentication.
- Remove the old credential after cutover.

Example with Microsoft Graph PowerShell:

```powershell
Connect-MgGraph -Scopes "Application.ReadWrite.All"

# Add a new password credential to an app registration
$start = Get-Date
$end = $start.AddYears(1)

Add-MgApplicationPassword -ApplicationId "11111111-2222-3333-4444-555555555555" `
  -PasswordCredential @{
    displayName = "incident-rotation-2026"
    startDateTime = $start
    endDateTime = $end
  }
```

Example with Azure CLI for app registration credential management may vary by version and tenant configuration, so verify the exact command set in your environment before relying on it for automation. For exam prep, focus on the workflow rather than memorizing a brittle command.

**Certificates versus client secrets**

Certificates are often preferred over client secrets because they are harder to exfiltrate and can provide stronger assurance. They are not automatically easier to manage at scale, because you must handle issuance, renewal, and deployment.

Use certificates when:

- The application supports certificate-based authentication.
- You need stronger protection than a shared secret.
- You can automate renewal and deployment.

Use client secrets when:

- The workload cannot use certificates.
- You need a simpler short-term integration.
- You are replacing a compromised secret quickly and will improve the design later.

**Rotating Key Vault secrets in workloads**

Many Azure workloads read secrets from Key Vault directly or through Key Vault references. The rotation pattern depends on how the app consumes the value.

For App Service and Azure Functions, Key Vault references are common:

```text
@Microsoft.KeyVault(SecretUri=https://kv-prod.vault.azure.net/secrets/app-db-password/)
```

A versionless URI lets the platform resolve the latest enabled version when it refreshes the reference, but your application may still cache the value. If the app caches configuration, you must restart or trigger a refresh.

Practical steps:

1. Set the new secret version in Key Vault.
2. Confirm the app identity can read the secret.
3. Refresh the app setting or restart the workload if needed.
4. Validate the app connects with the new value.
5. Disable or delete the old secret version after the cutover window.

For App Service, a restart often forces the app to re-read configuration. For Azure Functions, a host restart or app setting change may be needed depending on the hosting model and caching behavior.

```bash
# Restart an App Service after secret rotation
az webapp restart \
  --resource-group rg-prod \
  --name app-prod

# Restart a Function App
az functionapp restart \
  --resource-group rg-prod \
  --name func-prod
```

**Rotating cryptographic keys in Key Vault**

Key rotation for encryption or signing requires more planning than secret rotation because data may be encrypted with an older key version.

Use this pattern:

1. Create a new key version in Key Vault.
2. Update the application or service to use the new version for new operations.
3. Re-encrypt or rewrap existing data if the design requires it.
4. Keep the old version available until all dependent data is migrated.
5. Disable or retire the old version only after validation.

Example:

```bash
# Create a new RSA key version
az keyvault key create \
  --vault-name kv-prod \
  --name data-protection-key \
  --kty RSA \
  --size 3072
```

If a service uses envelope encryption, the data encryption key is often wrapped by a Key Vault key. In that case, you may only need to rewrap the data keys rather than decrypt and re-encrypt all data.

**Storage account keys and SAS**

Azure Storage has multiple access models, and incident response depends on which one is in use.

- **Storage account keys**: full access to the account; rotate immediately if exposed.
- **Service SAS**: signed with the storage account key; revoking the account key invalidates it.
- **Account SAS**: also signed with the account key; same revocation behavior.
- **User delegation SAS**: signed with a user delegation key obtained through Microsoft Entra ID; it is not revoked by rotating the storage account key.

For account keys, rotate both keys in a controlled sequence so the service stays online.

```bash
# List storage account keys
az storage account keys list \
  --resource-group rg-prod \
  --account-name mystorageacct

# Regenerate the secondary key first
az storage account keys renew \
  --resource-group rg-prod \
  --account-name mystorageacct \
  --key secondary
```

> CLI syntax can differ by Azure CLI version. Verify the exact `az storage account keys` subcommand in your environment before automating rotation.

A safe rotation pattern is:

1. Update applications to use key 2 while key 1 remains active.
2. Regenerate key 1.
3. Switch applications back to key 1 if desired.
4. Regenerate key 2.

For SAS revocation, stored access policies can help with service SAS in some scenarios, but they do not control user delegation SAS the same way. User delegation SAS is constrained by the user delegation key lifetime and the underlying Entra identity state. In incident response, that means you may need to disable the user, revoke sessions, or wait for token and key expiry depending on the compromise.

**Cosmos DB key rotation**

Azure Cosmos DB uses primary and secondary keys for data-plane access. Rotate one key at a time so applications can fail over.

```bash
# Regenerate the primary key
az cosmosdb keys regenerate \
  --resource-group rg-prod \
  --name cosmos-prod \
  --type primary
```

Typical sequence:

1. Update the application to use the secondary key.
2. Regenerate the primary key.
3. Switch the application back if needed.
4. Regenerate the secondary key.

This pattern avoids downtime because one key remains valid while the other is replaced.

**SQL credential rotation**

SQL Server and Azure SQL have different credential models, so be careful not to overgeneralize.

- **SQL Server logins**: can use `ALTER LOGIN` to change passwords.
- **Azure SQL Database**: server-level administration differs from on-premises SQL Server.
- **Contained users**: may require `ALTER USER` rather than `ALTER LOGIN`.

Example for SQL Server:

```sql
ALTER LOGIN app_login
WITH PASSWORD = 'NewStrongPassword!2026'
    OLD_PASSWORD = 'OldStrongPassword!2025';
```

For Azure SQL Database, you often rotate application connection strings or contained database user credentials rather than assuming a server login workflow. In exam scenarios, identify whether the question is about SQL Server, Azure SQL Database, or a managed instance before choosing the command.

**Revoking user sessions and tokens**

If a user account is compromised, password reset alone is not enough. You should revoke sessions so refresh tokens and sign-in sessions are invalidated.

Microsoft Graph supports revoking sign-in sessions:

```http
POST /users/{id}/revokeSignInSessions
```

This forces reauthentication, but it does not guarantee that already-issued access tokens stop working immediately. Access tokens are bearer tokens and usually remain valid until expiry.

Practical response sequence:

1. Disable the account if active abuse is suspected.
2. Reset the password.
3. Revoke sign-in sessions.
4. Review conditional access and MFA status.
5. Investigate sign-in logs and risky sign-in events.

For privileged users, also review role assignments and privileged identity management activation history.

**App-only access and service principals**

App-only access uses application credentials, not user sessions. That means session revocation is not the control you rely on. Instead, you rotate or remove the credential and reduce the app’s permissions.

Good incident-response actions include:

- Remove the exposed client secret or certificate.
- Add a new credential and cut over.
- Disable the service principal if compromise is severe.
- Review app role assignments and API permissions.
- Check downstream systems that may have cached tokens.

Existing access tokens issued to the app may continue until they expire, so credential rotation is necessary but not always instantly sufficient.

**Managed identities in incident response**

Managed identities reduce secret sprawl, but they still need incident-response planning.

Use managed identities when:

- You want to eliminate stored credentials.
- The workload runs in Azure and supports identity-based auth.
- You want simpler rotation because there is no secret to replace.

If a managed identity is abused:

- Remove its role assignments or scope them down.
- Detach a user-assigned managed identity from the workload if needed.
- Isolate or stop the compromised resource.
- Recreate the workload if the host itself is untrusted.

Managed identities are especially useful when you need to rotate other credentials because they can become the replacement authentication method after the incident.

**Practical rotation workflow**

A repeatable incident-response workflow helps you avoid outages.

1. Identify the compromised asset.
2. Determine whether it is a user credential, app credential, storage key, certificate, or cryptographic key.
3. Decide whether to disable first or rotate first.
4. Create the replacement credential or key version.
5. Update every dependent workload, pipeline, or integration.
6. Restart or refresh services that cache configuration.
7. Validate authentication and authorization.
8. Retire the old credential or key after the cutover window.
9. Review logs for post-compromise activity.

**Example response matrix**

| Scenario | Immediate action | Follow-up action |
|---|---|---|
| User password exposed | Reset password, revoke sessions, consider disabling account | Review sign-ins, enforce MFA, check mailbox and app consent |
| App secret exposed | Add new secret, update app, remove old secret | Review app permissions and token usage |
| Storage account key exposed | Regenerate one key, move apps, regenerate the other | Audit storage access and SAS usage |
| Key Vault secret exposed | Create new version, update consumers, retire old version | Check access policies and secret retrieval logs |
| Cryptographic key exposed | Create new key version, rewrap or re-encrypt data | Validate all dependent services before retiring old key |

**Common pitfalls**

- **Assuming rotation is instant**: cached tokens, sessions, and app settings may continue to work until refreshed or expired.
- **Rotating only one layer**: if a secret is stored in Key Vault and copied into App Service settings, both layers may need updates.
- **Deleting old versions too early**: some workloads still depend on the previous key or certificate.
- **Ignoring downstream systems**: CI/CD pipelines, automation accounts, and third-party integrations often hold the same credential.
- **Using the wrong control for the asset**: user session revocation does not replace app secret rotation.

**Exam focus**

For AZ-500, expect questions that ask you to choose the best containment and rotation action for a given compromise.

Remember these distinctions:

- **User compromise**: reset password and revoke sessions.
- **App compromise**: rotate or remove credentials, then update consumers.
- **Storage compromise**: rotate account keys or change SAS strategy.
- **Key Vault compromise**: create new versions and update references.
- **Managed identity design**: prefer it when you can remove stored secrets entirely.

The best answer is usually the one that reduces exposure fastest while preserving service availability and respecting how Azure actually authenticates the workload.
