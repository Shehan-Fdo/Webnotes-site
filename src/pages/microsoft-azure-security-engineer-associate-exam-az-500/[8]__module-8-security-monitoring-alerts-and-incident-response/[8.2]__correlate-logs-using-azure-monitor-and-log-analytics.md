## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Overview**

Correlating logs means connecting events from multiple Azure and Microsoft security data sources to reconstruct what happened, when it happened, and which identities or resources were involved. In AZ-500 scenarios, you use Azure Monitor, Log Analytics, and service-specific logs to investigate suspicious activity, validate controls, and support alerting.

Azure Monitor is the platform for collecting and analyzing telemetry, but security investigations often span more than one portal or data plane. Some logs are queried in Log Analytics, while others are viewed in Microsoft Entra, Microsoft Defender, or a service-specific blade. The exam expects you to understand where data lives, how it is queried, and how to correlate it safely.

**Core concepts**

- **Azure Monitor** collects metrics, logs, and activity data across Azure resources.
- **Log Analytics workspace** stores and queries log data using Kusto Query Language (KQL).
- **Azure Activity Log** records subscription-level control-plane operations such as create, update, delete, and role assignment changes.
- **Resource logs** capture data-plane or service-specific events, such as Key Vault access or network security group flow-related telemetry when enabled.
- **Microsoft Entra logs** include sign-in and audit events that are often essential for identity investigations.
- **Microsoft Sentinel** can use the same Log Analytics data for security operations, hunting, and incident workflows.

**Log Analytics workspace structure and scope**

A Log Analytics workspace is the logical container for log data, retention, access control, and query scope. In practice, you investigate by querying one workspace, multiple workspaces, or a workspace plus other data sources depending on how the environment is designed.

Workspace design affects what you can see:

- **Workspace-based ingestion** centralizes logs for cross-resource correlation.
- **Resource-centric views** let you start from a resource and jump into its logs or diagnostics.
- **Retention settings** determine how far back you can query data.
- **Table availability** depends on which diagnostic settings, agents, or connectors are enabled.
- **Permissions** control whether you can read the workspace, the resource, or both.

Important implications for investigations:

- A log event may exist in a service portal but not yet in Log Analytics if the diagnostic pipeline is not configured.
- Some security data is not stored in the same table as Azure resource logs.
- Query results can be incomplete if the retention window is too short or ingestion is delayed.
- Cross-subscription investigations often require explicit workspace access and cross-workspace queries.

**Common data sources for correlation**

Use multiple sources to build a timeline:

- **Azure Activity Log**
  - Subscription-level operations
  - Useful for resource creation, deletion, policy changes, and role assignments
- **Diagnostic logs**
  - Service-specific logs sent from a resource to a workspace, storage account, or event hub
  - Examples include Key Vault, Azure Firewall, Application Gateway, and many PaaS services
- **Microsoft Entra sign-in logs**
  - Interactive and non-interactive authentication events
  - Useful for identity, MFA, conditional access, and token-related investigations
- **Microsoft Entra audit logs**
  - Directory changes such as user updates, app registrations, group membership changes, and policy changes
- **Defender and security service logs**
  - Alerts and detections may be queried in their own portals or exported for correlation
- **Guest OS logs**
  - Windows Event Logs and Linux Syslog collected by agents or Azure Monitor Agent (AMA)

> Not every security-relevant event lands in the same table or even the same product. For exam scenarios, first identify the source of truth for the event type, then decide whether Log Analytics is the query surface or only one part of the investigation.

**Ingestion, latency, and freshness**

Log correlation depends on time alignment. Events do not always arrive instantly, and different sources have different delays.

- **Ingestion latency** can range from seconds to minutes, and sometimes longer during service delays.
- **Event time** is not always the same as **ingestion time**.
- **Clock skew** on guest systems can distort timelines if the host time is inaccurate.
- **Late-arriving events** can appear after an alert or investigation has already started.

Practical guidance:

- Use a wider time window than the suspected incident window.
- Compare `TimeGenerated` with ingestion-related fields when available.
- Re-run queries after a delay if you suspect delayed telemetry.
- Normalize timestamps to UTC in your reasoning.

**KQL fundamentals for correlation**

Kusto Query Language is the primary query language for Log Analytics. You use it to filter, project, summarize, join, and reshape data.

Common operators and functions:

- `where` filters rows.
- `project` selects columns.
- `extend` creates calculated columns.
- `summarize` aggregates data.
- `join` correlates records across tables.
- `union` combines tables or datasets.
- `sort by` orders results.
- `take` limits output for quick inspection.
- `bin()` groups timestamps into fixed intervals.
- `arg_max()` returns the row with the latest value in a group.
- `make_set()` collects distinct values into a dynamic array.
- `mv-expand` expands arrays or dynamic fields into multiple rows.
- `lookup` is useful for enriching a table with a smaller reference table.

A simple pattern for investigations is:

1. Filter to the relevant time range.
2. Identify the actor, resource, or IP address.
3. Summarize by time or entity.
4. Join to another table for context.
5. Validate the result against the original event details.

**Basic KQL examples**

Filter sign-ins for a specific user:

```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| where UserPrincipalName =~ "alice@contoso.com"
| project TimeGenerated, UserPrincipalName, AppDisplayName, IPAddress, ResultType, ConditionalAccessStatus
| order by TimeGenerated desc
```

Summarize failed sign-ins by IP address:

```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType != 0
| summarize FailedAttempts = count() by IPAddress
| order by FailedAttempts desc
```

Group events into 15-minute buckets:

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| summarize Events = count() by bin(TimeGenerated, 15m), OperationNameValue
| order by TimeGenerated asc
```

**Correlation patterns that matter in AZ-500**

Correlation is usually about linking identity, control plane, and data plane activity.

- **Identity to resource access**
  - Match sign-in events with resource access or audit events to see whether a successful login led to a sensitive action.
- **Control plane to data plane**
  - Match Azure Activity Log changes with resource logs to see whether a configuration change preceded suspicious behavior.
- **Host to cloud**
  - Match guest OS logs with Azure resource logs to determine whether a VM compromise aligns with network or identity events.
- **Alert to evidence**
  - Use an alert as the starting point, then query surrounding logs to confirm scope and impact.

A safer identity correlation approach is to use stable identifiers when possible:

- Prefer object IDs, correlation IDs, or resource IDs over display names.
- Treat user principal names as useful but not always authoritative.
- Expect service principals, managed identities, and apps to appear in audit trails.

Example: correlate sign-in activity with directory audit events using a shared time window and actor context, but avoid assuming every audit event has the same user field shape.

```kusto
let startTime = ago(24h);
let signins =
    SigninLogs
    | where TimeGenerated > startTime
    | project SigninTime = TimeGenerated, UserPrincipalName, IPAddress, AppDisplayName, CorrelationId;
let audits =
    AuditLogs
    | where TimeGenerated > startTime
    | project AuditTime = TimeGenerated, ActivityDisplayName, Result, CorrelationId, InitiatedBy, TargetResources;
signins
| join kind=leftouter audits on CorrelationId
| project SigninTime, UserPrincipalName, IPAddress, AppDisplayName, ActivityDisplayName, Result, AuditTime
| order by SigninTime desc
```

This pattern is useful when the same correlation ID is present, but it is not guaranteed for every scenario. If the join is empty, pivot to time, target resource, or object ID.

**Safer joins and enrichment**

Joins are powerful, but they can also create false confidence if the key is weak.

Use these guidelines:

- Join on stable identifiers when available.
- Use `innerunique` when you want one matching row from the left side and expect duplicates on the right.
- Use `leftouter` when you want to preserve the primary event even if enrichment is missing.
- Use `lookup` when enriching with a small reference set.
- Avoid joining on fields that are inconsistently populated across event types.

Example: enrich failed sign-ins with a small set of known risky IPs.

```kusto
let riskyIPs = datatable(IPAddress:string, RiskLabel:string)
[
  "203.0.113.10", "Known scanner",
  "198.51.100.25", "Suspicious VPN exit"
];
SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType != 0
| lookup kind=leftouter riskyIPs on IPAddress
| project TimeGenerated, UserPrincipalName, IPAddress, RiskLabel, AppDisplayName, ResultType
| order by TimeGenerated desc
```

**Useful investigation functions**

These functions often appear in real investigations and exam-style questions:

- `make_set()` helps you see all distinct values associated with an entity.
- `arg_max()` helps you keep the latest record per user, host, or resource.
- `mv-expand` helps you unpack arrays such as target resources or claims.
- `bin()` helps you spot bursts, periodicity, or attack waves.
- `lookup` helps you add context without a full join explosion.

Example: find the latest event per user.

```kusto
SigninLogs
| where TimeGenerated > ago(7d)
| summarize arg_max(TimeGenerated, *) by UserPrincipalName
| project TimeGenerated, UserPrincipalName, AppDisplayName, IPAddress, ResultType
```

Example: expand target resources from audit events.

```kusto
AuditLogs
| where TimeGenerated > ago(24h)
| mv-expand TargetResources
| project TimeGenerated, ActivityDisplayName, TargetName = tostring(TargetResources.displayName), TargetType = tostring(TargetResources.type)
| order by TimeGenerated desc
```

**Workspace querying techniques**

Log Analytics supports more than single-table queries.

- **Cross-workspace queries** let you investigate multiple workspaces from one query when permissions allow it.
- **Time range controls** in the portal can limit or expand the query window.
- **Saved queries** and favorites help you reuse common hunting patterns.
- **Query packs** and shared queries can standardize investigations across teams.

Example: query across workspaces with a union pattern.

```kusto
union workspace("workspace-a").SigninLogs, workspace("workspace-b").SigninLogs
| where TimeGenerated > ago(24h)
| summarize Failed = countif(ResultType != 0) by UserPrincipalName
| order by Failed desc
```

Use cross-workspace querying when:

- You centralize logs from multiple subscriptions or tenants.
- You need to compare activity across environments.
- You are investigating a distributed incident.

**Azure Activity Log correlation**

The Azure Activity Log is essential for control-plane investigations.

Typical uses:

- Detect who created or deleted a resource.
- Confirm when a role assignment changed.
- Identify policy or lock modifications.
- Trace deployment operations that may have introduced risk.

Example: inspect recent role assignment changes.

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where OperationNameValue has "roleAssignments"
| project TimeGenerated, Caller, OperationNameValue, ActivityStatusValue, ResourceGroup, ResourceProviderValue, CorrelationId
| order by TimeGenerated desc
```

For investigations, correlate Activity Log events with:

- Resource logs to see the effect of the change.
- Sign-in logs to see who authenticated before the change.
- Audit logs to see whether the identity itself was modified.

**Resource logs and diagnostic settings**

Many Azure services emit diagnostic logs only after you configure diagnostic settings. This is a common exam point.

Key ideas:

- Diagnostic settings define where logs go: Log Analytics workspace, storage account, or event hub.
- Supported categories vary by resource type.
- Some resources expose logs, metrics, or both.
- The exact category names are service-specific.

Example: configure diagnostic settings for a resource with Azure CLI.

```bash
az monitor diagnostic-settings create \
  --name kv-diag \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vaultName> \
  --workspace /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspaceName> \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Important caveats:

- The resource ID must be valid.
- Category names must match the target resource’s supported categories.
- Not every resource supports the same metric categories.
- Some services require separate configuration for data-plane logging and platform metrics.

> Do not assume one diagnostic command works unchanged for every Azure service. Always verify the supported categories for the specific resource type.

**Guest OS log correlation**

Guest logs help you connect cloud events to host behavior.

Windows examples:

- Security Event Log for authentication, privilege use, and process creation if enabled.
- System and Application logs for service failures and application issues.

Linux examples:

- `Syslog` for authentication, service, and daemon messages.
- Common fields often include `Facility`, `SeverityLevel`, `HostName`, and `SyslogMessage`.

Example: inspect Linux authentication-related messages without assuming a process-name column exists.

```kusto
Syslog
| where TimeGenerated > ago(24h)
| where SyslogMessage has_any ("sshd", "sudo", "authentication failure", "Accepted password")
| project TimeGenerated, HostName, Facility, SeverityLevel, SyslogMessage
| order by TimeGenerated desc
```

If you need process-level detail, confirm the schema in your workspace first. Column availability can vary by collection method and table version.

**Practical investigation workflow**

A typical security investigation in Azure follows a repeatable pattern.

1. Identify the alert, suspicious event, or user report.
2. Determine the likely source tables and portals.
3. Set a time window that is wider than the suspected incident.
4. Query the primary event source.
5. Correlate with identity, activity, and resource logs.
6. Expand the scope to related users, IPs, hosts, or resources.
7. Validate whether the event is benign, expected, or malicious.
8. Preserve the query and findings for follow-up or incident response.

Example workflow for a suspicious sign-in followed by a privileged change:

```kusto
let startTime = ago(12h);
let suspiciousSignins =
    SigninLogs
    | where TimeGenerated > startTime
    | where ResultType == 0
    | where ConditionalAccessStatus != "success" or RiskLevelDuringSignIn != "none"
    | project SigninTime = TimeGenerated, UserPrincipalName, IPAddress, AppDisplayName, CorrelationId;
let activity =
    AzureActivity
    | where TimeGenerated > startTime
    | project ActivityTime = TimeGenerated, Caller, OperationNameValue, ResourceGroup, ResourceId, CorrelationId;
suspiciousSignins
| join kind=leftouter activity on CorrelationId
| order by SigninTime desc
```

This is a starting point, not a guaranteed one-to-one match. If the correlation ID is absent or different, pivot to user, IP, resource, or time proximity.

**Microsoft Sentinel-style workflow**

If Microsoft Sentinel is present, the same Log Analytics data supports hunting and incident response.

Typical workflow:

- An analytics rule generates an incident from a KQL query.
- You open the incident and inspect the entities.
- You pivot from the entity to related logs.
- You run additional queries to determine scope and impact.
- You document the timeline and close or escalate the incident.

Example hunting query for repeated failed sign-ins from one IP:

```kusto
SigninLogs
| where TimeGenerated > ago(1h)
| where ResultType != 0
| summarize FailedAttempts = count(), Users = make_set(UserPrincipalName, 100) by IPAddress
| where FailedAttempts >= 10
| order by FailedAttempts desc
```

This kind of query can support an alert rule or a hunting workbook. In a Sentinel workflow, you would usually tune thresholds to reduce noise and false positives.

**Alerting model in modern Azure designs**

Azure alerting has evolved. For exam purposes, distinguish between the current model and legacy terminology.

- **Metric alerts** evaluate numeric platform metrics.
- **Log alerts** use scheduled query rules against Log Analytics data.
- **Classic alerts** are retired or deprecated in modern designs.
- **Action groups** define what happens when an alert fires.

Use log alerts when:

- You need to detect patterns in logs rather than numeric thresholds.
- You want to correlate multiple fields or tables.
- You need security detections based on KQL.

Use metric alerts when:

- You need near-real-time threshold monitoring on platform metrics.
- The signal is already exposed as a metric.

**Example alerting pattern**

A log alert might detect:

- Multiple failed sign-ins from the same IP.
- A role assignment change outside a maintenance window.
- Key Vault access denied events from a sensitive vault.
- A burst of firewall denies from a single source.

**Troubleshooting correlation issues**

When a query does not return expected results, check the following:

- The time range is too narrow.
- The data source is not enabled or not sending logs.
- The workspace retention has expired.
- The table schema differs from your assumption.
- The identity field is inconsistent or missing.
- The event arrived late.
- The query uses the wrong join key.

Useful debugging techniques:

- Start with `take 10` or `limit` to inspect raw rows.
- Use `project-away` to simplify noisy columns.
- Use `summarize count()` to confirm data exists before joining.
- Inspect dynamic fields with `tostring()` or `mv-expand`.
- Compare results across adjacent time windows.

**Exam-focused takeaways**

- Know the difference between Azure Activity Log, resource logs, Entra logs, and guest OS logs.
- Understand that Log Analytics is a query and storage layer, not the only place security logs exist.
- Use KQL to filter, summarize, join, and enrich events.
- Prefer stable identifiers over display names for correlation.
- Expect ingestion delay and schema variation.
- Recognize that diagnostic settings are resource-specific.
- Distinguish log alerts from metric alerts and know that classic alerting is retired.
- Be ready to explain how a security investigation moves from alert to evidence to scope.

**Quick reference**

- **`where`**: filter rows
- **`project`**: select columns
- **`summarize`**: aggregate
- **`join`**: correlate tables
- **`bin()`**: bucket time
- **`arg_max()`**: latest record per group
- **`make_set()`**: collect distinct values
- **`mv-expand`**: flatten arrays
- **`lookup`**: enrich with reference data

**Mini checklist for investigations**

- Confirm the source table.
- Expand the time window.
- Identify the actor, IP, host, or resource.
- Correlate with at least one other log source.
- Validate whether the event is expected.
- Save the query for reuse.
