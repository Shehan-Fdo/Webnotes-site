## **Course:** Microsoft Azure Security Engineer Associate (Exam AZ-500)

**Containment goals**

Containment is the incident response phase where you stop or slow attacker activity without destroying evidence or causing unnecessary business impact. In Azure, that usually means reducing network reachability, disabling compromised identities, isolating compute, and preserving logs for later investigation.

The main tradeoff is speed versus stability. You want to cut off the attacker quickly, but you also need to avoid actions that erase forensic data, break recovery paths, or lock out responders.

- **Short-term containment** limits active damage.
- **Long-term containment** gives you time to investigate and remediate safely.
- **Evidence preservation** matters because containment can change logs, memory, and disk state.

**Containment decision points**

Before you act, identify what is compromised and what still needs to stay available. A compromised user account needs a different response than a compromised VM, storage account, or Kubernetes cluster.

Use these questions to choose the right action:

- What is the blast radius?
- Is the attacker using identity, network, or workload access?
- Can you isolate the asset without deleting it?
- Do you need to preserve memory, disk, or audit logs first?
- Will the action affect production traffic or incident responders?

**Identity containment**

Identity is often the fastest path for an attacker, so revoking access early is critical. In Microsoft Entra ID, you can invalidate active sessions, reset credentials, and disable accounts to stop further use.

Common containment actions include:

- **Disable the user account** to block new sign-ins.
- **Revoke sign-in sessions** to force reauthentication.
- **Reset passwords** for compromised users.
- **Remove privileged role assignments** if the account has elevated access.
- **Review and remove malicious app consent** if an application was abused.

A practical Microsoft Graph example for revoking sessions is:

```http
POST https://graph.microsoft.com/v1.0/users/{user-id}/revokeSignInSessions
Authorization: Bearer <access_token>
Content-Type: application/json
```

You can also disable a user with Microsoft Graph:

```http
PATCH https://graph.microsoft.com/v1.0/users/{user-id}
Content-Type: application/json

{
  "accountEnabled": false
}
```

For Azure CLI, identity containment is often easier through Microsoft Graph-backed commands or `az rest` when you need direct API access. For example, to disable a user:

```bash
az rest --method patch \
  --url "https://graph.microsoft.com/v1.0/users/{user-id}" \
  --headers "Content-Type=application/json" \
  --body '{"accountEnabled":false}'
```

> Revoking sessions does not always remove every token immediately in every scenario. Pair it with password reset or account disablement when you need stronger containment.

**Privileged access containment**

If a privileged identity is compromised, containment must happen immediately. Focus on removing the attacker’s ability to escalate or persist.

Typical actions:

- Remove the user from privileged groups.
- Remove active role assignments in Azure RBAC.
- Disable just-in-time access if it is being abused, then re-enable only after review.
- Rotate credentials for service principals, managed identities, or automation accounts if they are exposed.

For Azure role assignments, you can remove access with Azure CLI:

```bash
az role assignment delete \
  --assignee user@contoso.com \
  --role "Contributor" \
  --scope /subscriptions/<subscription-id>
```

For Microsoft Entra role assignments, use the appropriate directory role management path rather than assuming Azure RBAC and Entra roles are interchangeable. They are separate authorization systems.

**Network containment**

Network containment reduces the attacker’s ability to move laterally, exfiltrate data, or reach management endpoints. In Azure, this often means tightening security groups, firewall rules, route tables, and private access paths.

Common actions include:

- Block inbound access with a network security group (NSG).
- Restrict outbound access to stop command-and-control traffic.
- Remove or narrow public IP exposure.
- Use private endpoints where possible.
- Place a workload behind a jump host or Bastion for controlled administration.

A simple NSG rule update example looks like this:

```bash
az network nsg rule update \
  --resource-group rg-sec \
  --nsg-name nsg-app \
  --name DenyInternetRDP \
  --access Deny \
  --direction Inbound \
  --priority 100 \
  --protocol Tcp \
  --destination-port-ranges 3389 \
  --source-address-prefixes Internet
```

If the rule does not already exist, create it instead of updating it. For multiple source prefixes, use a list or comma-separated values depending on the command shape you choose.

For a broader isolation pattern, you can temporarily deny all inbound traffic except from a responder subnet:

```bash
az network nsg rule create \
  --resource-group rg-sec \
  --nsg-name nsg-app \
  --name AllowResponderSubnet \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.10.0.0/24 \
  --destination-port-ranges 22 3389
```

**Workload containment for virtual machines**

For Azure virtual machines, the fastest containment action is often to isolate the VM at the network layer while preserving the disk and memory state for investigation.

Useful actions:

- **Deny inbound traffic** with NSG rules.
- **Restrict outbound traffic** to prevent exfiltration.
- **Stop or deallocate** only when you no longer need live memory evidence.
- **Snapshot disks** before destructive remediation.
- **Use serial console or Bastion** only if you still trust the management path.

A common containment workflow is:

1. Capture the current state with snapshots or disk copies if required.
2. Block network access to and from the VM.
3. Preserve logs and memory evidence if live response is needed.
4. Investigate the host before rebuilding or reimaging.

Example disk snapshot command:

```bash
az snapshot create \
  --resource-group rg-sec \
  --source /subscriptions/<subscription-id>/resourceGroups/rg-app/providers/Microsoft.Compute/disks/osdisk-vm01 \
  --name osdisk-vm01-snap
```

If the VM is actively compromised, you may choose to stop it after evidence capture:

```bash
az vm deallocate \
  --resource-group rg-app \
  --name vm01
```

Stopping a VM can help contain damage, but it also ends live response opportunities. Use it deliberately.

**Azure Bastion and controlled administration**

Azure Bastion provides browser-based RDP and SSH access through the Azure portal without exposing a public IP on the VM. It is useful when you need a controlled management path during containment.

Use Bastion when:

- You need administrative access without opening inbound management ports.
- You want to avoid direct public exposure of RDP or SSH.
- You are working in a locked-down network where jump access is preferred.

Do not assume Bastion is a universal emergency access path. Its availability depends on deployment, SKU, and network design.

**Storage containment**

If a storage account is suspected of exposure or abuse, containment usually focuses on access control and network restrictions rather than deleting the account.

Actions to consider:

- Rotate account keys if shared key access may be compromised.
- Disable public access where appropriate.
- Restrict access with firewall rules or private endpoints.
- Review SAS (shared access signature) tokens and revoke them if they are overexposed.
- Remove unauthorized role assignments on the storage account.

Example of disabling public blob access at the account level:

```bash
az storage account update \
  --resource-group rg-data \
  --name stcontoso01 \
  --allow-blob-public-access false
```

If you suspect a leaked SAS token, remember that SAS tokens are time-bound and permission-bound. Containment may require key rotation or policy changes, not just network filtering.

**Azure SQL containment**

For Azure SQL Database and Azure SQL Managed Instance, containment usually means limiting who can connect and from where. The exact approach depends on whether you use public endpoints, private endpoints, or both.

Common actions:

- Restrict server firewall rules.
- Disable public network access when private connectivity is in place and operationally acceptable.
- Verify that management access remains possible through approved paths.
- Rotate credentials for SQL logins or application secrets if they are exposed.

A firewall rule example:

```bash
az sql server firewall-rule create \
  --resource-group rg-data \
  --server sqlprod01 \
  --name AllowResponderIP \
  --start-ip-address 203.0.113.10 \
  --end-ip-address 203.0.113.10
```

> Disabling public network access is not always the right first move. Use it when private endpoint connectivity and administrative access are already designed and tested.

**Azure Kubernetes Service containment**

For Azure Kubernetes Service (AKS), containment often starts at the workload and node level. You may need to isolate a pod, cordon a node, or restrict cluster access depending on the incident.

Useful actions:

- **Cordon** a node to stop new pods from scheduling there.
- **Drain** a node if you need to move workloads away.
- Restrict cluster API access through approved network paths.
- Rotate cluster credentials if compromise is suspected.
- Review service account tokens and workload identities.

Example node containment:

```bash
kubectl cordon aks-nodepool1-00000000-vmss000000
kubectl drain aks-nodepool1-00000000-vmss000000 --ignore-daemonsets --delete-emptydir-data
```

Use `cordon` when you want to freeze scheduling without evicting workloads immediately. Use `drain` when you need to move workloads off the node, but be aware that it can disrupt running services.

**Azure Firewall and perimeter containment**

Azure Firewall helps centralize traffic control when you need to block malicious destinations or restrict east-west and north-south traffic.

Containment patterns include:

- Deny outbound traffic to known malicious IPs or domains.
- Restrict access to sensitive subnets.
- Force traffic through inspection points.
- Update rule collections to block exfiltration paths.

A simple network rule collection example in Azure Firewall policy uses Azure CLI to create policy rules rather than ad hoc firewall rules on the resource itself:

```bash
az network firewall policy rule-collection-group collection add-filter-collection \
  --policy-name fwpol-prod \
  --resource-group rg-sec \
  --rule-collection-group-name rcg-main \
  --name BlockMaliciousIPs \
  --priority 100 \
  --action Deny \
  --rule-name BlockC2 \
  --rule-type NetworkRule \
  --ip-protocols TCP \
  --source-addresses 10.0.0.0/8 \
  --destination-addresses 198.51.100.25 \
  --destination-ports 443
```

Azure Firewall syntax varies by policy model and CLI version, so verify the exact command shape in your environment before using it in production.

**Containment by resource type**

Different Azure services need different containment levers. The goal is to stop attacker activity with the least destructive action that still works.

| Resource type | Primary containment action | Notes |
|---|---|---|
| User account | Disable account, revoke sessions | Fastest identity containment |
| Service principal | Rotate secrets/certificates, remove role assignments | Check app registrations and automation |
| VM | Isolate network, snapshot disks, deallocate if needed | Preserve evidence before shutdown |
| Storage account | Restrict network access, rotate keys, review SAS | Shared key exposure is high risk |
| SQL Database | Tighten firewall, use private endpoint, rotate credentials | Public access may still be needed temporarily |
| AKS | Cordon/drain nodes, restrict API access | Avoid unnecessary cluster-wide disruption |
| Subscription | Remove high-risk role assignments, lock down policies | Use carefully to avoid blocking responders |

**Preserving evidence during containment**

Containment should not destroy the evidence you need for root cause analysis. If possible, capture logs and state before making disruptive changes.

Preserve:

- Activity logs and diagnostic logs.
- Sign-in logs and audit logs.
- VM disks and memory state.
- Firewall logs and NSG flow logs.
- Application logs and container logs.

Practical preservation steps:

1. Export logs to a secure workspace or storage account.
2. Snapshot disks or export managed disks if the workload is a VM.
3. Record the exact containment actions and timestamps.
4. Keep a chain of custody for artifacts that may be used in a formal investigation.

**Containment workflow example**

A typical Azure incident containment sequence might look like this:

1. Confirm the affected identity, resource, or subscription.
2. Preserve logs and snapshots if needed.
3. Disable compromised identities or revoke sessions.
4. Block malicious network paths with NSGs, firewall rules, or route changes.
5. Isolate the workload by cordoning nodes, deallocating VMs, or restricting access.
6. Validate that attacker activity has stopped.
7. Escalate to eradication and recovery.

**Operational cautions**

Containment can create outages if you apply it too broadly. Be precise and test your assumptions.

- Avoid deleting resources unless you have already captured evidence and confirmed the blast radius.
- Avoid rotating every secret at once unless compromise is widespread.
- Avoid locking yourself out of the subscription or management plane.
- Avoid assuming a single control, such as an NSG rule, will stop all access paths.
- Avoid using public network disablement as a blanket answer when private endpoint design is incomplete.

**Exam focus**

For AZ-500, expect questions that ask you to choose the best immediate containment action for a scenario. The correct answer usually depends on the attack path:

- If the attacker used a user account, revoke sessions and disable the account.
- If the attacker used a VM, isolate the network and preserve the disk.
- If the attacker used exposed network access, tighten NSGs, firewall rules, or public endpoints.
- If the attacker used privileged access, remove role assignments and rotate credentials.
- If the attacker used a container workload, cordon or drain nodes and review cluster access.

The best answer is usually the least destructive action that stops the attack while preserving evidence and keeping recovery options open.
