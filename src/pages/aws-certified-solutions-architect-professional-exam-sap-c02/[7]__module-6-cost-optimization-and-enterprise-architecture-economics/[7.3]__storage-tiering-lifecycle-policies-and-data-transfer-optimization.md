## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Storage tiering strategy**

Storage tiering reduces cost by placing data in the right storage class or service based on access frequency, durability needs, latency, and recovery objectives. For SAP-C02, you should choose the simplest design that satisfies performance and compliance, then automate movement of data as access patterns change.

A good tiering design answers four questions:

- **How often is the data accessed?**
- **How quickly must it be retrieved?**
- **Can the data tolerate AZ or Region loss?**
- **Is the data immutable, versioned, or subject to retention rules?**

Common AWS storage tiers and patterns include:

- **Amazon S3 Standard** for frequently accessed, latency-sensitive object data.
- **S3 Intelligent-Tiering** when access patterns are unknown or change over time.
- **S3 Standard-IA** and **S3 One Zone-IA** for infrequently accessed objects.
- **S3 Glacier Instant Retrieval**, **Flexible Retrieval**, and **Deep Archive** for archival use cases.
- **Amazon EBS** for block storage attached to EC2 instances.
- **Amazon EFS** for shared POSIX file storage with elastic scaling.
- **Amazon FSx** for workload-specific managed file systems.
- **AWS Storage Gateway** for hybrid caching, file sharing, or tape integration.

> **Important:** One Zone-IA is an S3 storage class, not a general-purpose secondary-copy pattern. Use it only when losing the underlying Availability Zone is acceptable.

**Amazon S3 lifecycle policies**

Lifecycle policies automate transitions and expirations for objects so you do not manage storage class changes manually. They are especially useful for logs, backups, media archives, and versioned datasets.

A lifecycle rule can:

- Transition current objects to cheaper storage classes after a time period.
- Expire current objects after a retention period.
- Transition or expire **noncurrent versions** in versioned buckets.
- Remove incomplete multipart uploads.
- Apply only to objects matching a prefix, tags, or size filters.

Lifecycle rules are evaluated against object age and filters, not by manual priority alone. When multiple rules overlap, AWS applies the most specific matching rule set for the object and action. In practice, you should design rules carefully to avoid unintended transitions or expirations.

**Lifecycle rule filtering and precedence**

Use filters to target the right objects:

- **Prefix filters** scope rules to a path-like key pattern such as `logs/`.
- **Tag filters** scope rules to objects with specific metadata such as `class=archive`.
- **Object size filters** help avoid moving tiny objects where transition overhead outweighs savings.
- **Combined filters** let you narrow a rule to a precise subset.

When rules overlap:

- A single object can match multiple rules.
- Transition timing matters; earlier transitions usually win for the same object version.
- Expiration can remove objects after they have already transitioned.
- In versioned buckets, current and noncurrent versions are managed separately.

**Versioning and lifecycle growth**

Versioning changes lifecycle behavior significantly:

- Deleting an object in a versioned bucket creates a delete marker instead of removing the data immediately.
- Old versions continue to consume storage unless you expire them.
- Noncurrent version expiration is critical for controlling cost in buckets with frequent overwrites.
- Lifecycle policies should usually include both current object expiration and noncurrent version cleanup for logs, backups, and generated artifacts.

A practical versioned-bucket lifecycle strategy often includes:

- Transition current objects to IA or Glacier after a short hot period.
- Expire noncurrent versions after a defined retention window.
- Remove expired object delete markers when no longer needed.
- Abort incomplete multipart uploads after a few days.

**Example S3 lifecycle configuration**

```json
{
  "Rules": [
    {
      "ID": "logs-tiering",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER_IR"
        }
      ],
      "Expiration": {
        "Days": 365
      },
      "NoncurrentVersionTransitions": [
        {
          "NoncurrentDays": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "NoncurrentDays": 90,
          "StorageClass": "GLACIER"
        }
      ],
      "NoncurrentVersionExpiration": {
        "NoncurrentDays": 180
      },
      "AbortIncompleteMultipartUpload": {
        "DaysAfterInitiation": 7
      }
    }
  ]
}
```

Apply it with the AWS CLI:

```bash
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-archive-bucket \
  --lifecycle-configuration file://lifecycle.json
```

**S3 lifecycle design considerations**

- **Minimum storage duration charges** can make frequent transitions uneconomical.
- **Retrieval fees** matter for IA and Glacier classes.
- **Small objects** may not benefit from aggressive tiering.
- **Compliance retention** may require object lock instead of simple expiration.
- **Versioned buckets** need explicit noncurrent cleanup to prevent hidden cost growth.

**S3 replication for resilience and locality**

S3 replication copies objects automatically between buckets and is commonly used for disaster recovery, compliance, and data locality. For SAP-C02, distinguish between replication of new objects, existing objects, delete markers, and replica metadata behavior.

Replication requires:

- **Versioning enabled** on both source and destination buckets.
- An IAM role that allows S3 to read from the source and write to the destination.
- A replication configuration on the source bucket.

Common replication patterns:

- **Same-Region Replication (SRR)** for compliance, account separation, or local redundancy.
- **Cross-Region Replication (CRR)** for disaster recovery and geographic isolation.
- **Multi-destination replication** when you need copies in more than one target bucket.

Replication nuances:

- By default, replication applies to **new objects** after the rule is enabled.
- Existing objects require **S3 Batch Replication** or a similar backfill process.
- **Delete markers** can be replicated if configured.
- **Replica modification sync** can propagate metadata or ACL-related changes in supported scenarios.
- For **KMS-encrypted objects**, the destination KMS key policy and IAM permissions must allow encryption and decryption operations.
- If you need object ownership changes or access control adjustments, verify the destination bucket ownership and replication settings.

A minimal replication configuration example:

```json
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "ID": "crr-logs",
      "Status": "Enabled",
      "Priority": 1,
      "Filter": {
        "Prefix": "logs/"
      },
      "DeleteMarkerReplication": {
        "Status": "Enabled"
      },
      "Destination": {
        "Bucket": "arn:aws:s3:::my-destination-bucket",
        "StorageClass": "STANDARD_IA"
      }
    }
  ]
}
```

**S3 Batch Operations and Inventory**

For large-scale management tasks, lifecycle and replication are often paired with inventory and batch tooling.

- **S3 Inventory** provides scheduled reports of object metadata, versions, encryption status, and replication status.
- **S3 Batch Operations** can apply actions to millions or billions of objects, such as copy, restore, tag, ACL update, or Lambda invocation.
- Use **Inventory** to identify candidates for transition, replication, or cleanup.
- Use **Batch Operations** when you need to backfill replication or apply a one-time bulk change.

**Amazon EBS storage optimization**

Amazon EBS is block storage for EC2 and does not provide native tiering across storage classes the way S3 does. Optimization usually comes from choosing the right volume type, using snapshots, and migrating volumes as workload needs change.

Key EBS concepts:

- **gp3** is the general-purpose default for many workloads because you can provision IOPS and throughput independently.
- **io2** is for sustained high IOPS and mission-critical databases.
- **st1** is for throughput-oriented, sequential workloads.
- **sc1** is for cold, infrequently accessed throughput workloads.
- **Snapshots** are incremental backups stored in S3-managed infrastructure.
- **Fast Snapshot Restore (FSR)** pre-warms snapshots in selected AZs to reduce first-read latency after restore.

EBS optimization patterns:

- Use **snapshots** for backup, migration, and cross-Region recovery.
- Use **copy snapshot** to move data between Regions or accounts.
- Use **volume modification** to change size, type, or performance characteristics without replacing the instance.
- Use **FSR** when you need predictable restore performance for critical recovery workflows.
- Use **lifecycle policies for snapshots** through Amazon Data Lifecycle Manager (DLM) to automate retention.

Example DLM snapshot policy:

```json
{
  "Description": "Daily EBS snapshot policy",
  "State": "ENABLED",
  "ExecutionRoleArn": "arn:aws:iam::123456789012:role/AWSDataLifecycleManagerDefaultRole",
  "PolicyDetails": {
    "ResourceTypes": ["VOLUME"],
    "TargetTags": [
      {
        "Key": "Backup",
        "Value": "Daily"
      }
    ],
    "Schedules": [
      {
        "Name": "daily-snapshots",
        "CreateRule": {
          "Interval": 24,
          "IntervalUnit": "HOURS",
          "Times": ["03:00"]
        },
        "RetainRule": {
          "Count": 7
        },
        "CopyTags": true
      }
    ]
  }
}
```

**Amazon EFS tiering and lifecycle**

Amazon EFS supports elastic file storage with lifecycle management that moves infrequently accessed files into lower-cost storage. This is useful for shared application content, home directories, and content repositories.

EFS lifecycle behavior:

- Files transition from **Standard** to **Infrequent Access (IA)** after a configured period of inactivity.
- Files can transition back to Standard when accessed, depending on the lifecycle policy.
- EFS IA reduces storage cost but adds retrieval charges.
- EFS is a good fit when you need shared POSIX semantics and automatic scaling.

CLI example for EFS lifecycle configuration:

```bash
aws efs put-lifecycle-configuration \
  --file-system-id fs-1234567890abcdef0 \
  --lifecycle-policies TransitionToIA=AFTER_30_DAYS,TransitionToPrimaryStorageClass=AFTER_1_ACCESS
```

EFS design notes:

- Use **EFS Standard** for frequently accessed shared data.
- Use **EFS IA** for infrequently accessed files that still need file-system semantics.
- Consider **EFS One Zone** when you can tolerate AZ loss and want lower cost for noncritical data.
- Use **EFS backups** for point-in-time recovery and cross-account protection.

**Amazon FSx workload-specific storage**

FSx provides managed file systems optimized for specific workloads. The exam often tests whether you choose the right FSx flavor rather than generic EFS or S3.

Major FSx options:

- **FSx for Windows File Server**
  - Native Windows file shares with SMB.
  - Integrates with Microsoft Active Directory.
  - Good for Windows applications, user shares, and lift-and-shift file workloads.
- **FSx for Lustre**
  - High-performance parallel file system.
  - Best for HPC, analytics, and workloads that need very high throughput.
  - Integrates well with S3 for data ingestion and export.
- **FSx for NetApp ONTAP**
  - Supports NFS, SMB, and iSCSI.
  - Offers snapshots, cloning, tiering to capacity pool, and enterprise storage features.
  - Good for mixed protocols and enterprise file workloads.
- **FSx for OpenZFS**
  - ZFS-based managed file system with snapshots and clones.
  - Good for Linux-based workloads needing high performance and efficient data management.

FSx exam distinctions:

- Choose **Windows File Server** for Microsoft-native file sharing.
- Choose **Lustre** for compute-heavy, parallel access.
- Choose **ONTAP** when you need snapshots, cloning, and tiering with enterprise features.
- Choose **OpenZFS** when you want ZFS semantics and efficient snapshot-based workflows.

**AWS Storage Gateway**

Storage Gateway bridges on-premises environments with AWS storage. It is often the right answer when you need hybrid integration, local caching, or tape replacement.

Gateway types:

- **File Gateway**
  - Presents NFS or SMB file shares.
  - Stores objects in S3.
  - Useful for backup targets, file shares, and cloud-backed archives.
- **Volume Gateway**
  - Presents iSCSI block volumes.
  - **Cached volumes** keep primary data in S3 and cache frequently accessed data locally.
  - **Stored volumes** keep primary data on-premises and asynchronously back up snapshots to AWS.
- **Tape Gateway**
  - Presents virtual tape libraries to existing backup software.
  - Stores virtual tapes in S3 and archives them to Glacier classes.

Use cases:

- Use **File Gateway** when applications need file shares but you want S3-backed storage.
- Use **Cached Volume Gateway** when you want cloud-backed storage with local performance for hot data.
- Use **Stored Volume Gateway** when you need local primary storage plus cloud backups.
- Use **Tape Gateway** when replacing physical tape infrastructure.

**Data transfer optimization**

Data transfer cost and latency are often as important as storage cost. SAP-C02 scenarios frequently combine storage design with network path selection, hybrid connectivity, and replication strategy.

Key cost concepts:

- **Inter-Region data transfer** usually costs more than same-Region traffic.
- **Cross-AZ data transfer** can add cost when instances, load balancers, or storage endpoints span Availability Zones.
- **Internet egress** is often the most expensive path for large outbound transfers.
- **Private connectivity** can reduce operational risk and improve predictability, but it still has service and port-hour costs.

Optimization principles:

- Keep compute and storage in the **same Region** when latency and compliance allow.
- Keep tightly coupled components in the **same AZ** when resilience requirements permit and the architecture is intentionally single-AZ.
- Use **VPC endpoints** to avoid NAT Gateway charges for private access to AWS services.
- Use **S3 Transfer Acceleration** for long-distance uploads over the public internet when clients are geographically dispersed.
- Use **DataSync** for managed, scheduled, and monitored data movement between on-premises storage and AWS.
- Use **Direct Connect** for consistent hybrid bandwidth and lower jitter than internet VPN in many enterprise designs.
- Use **Site-to-Site VPN** when you need encrypted connectivity quickly or as a backup path to Direct Connect.

**VPC endpoints and private service access**

VPC endpoints reduce exposure to the public internet and can lower data transfer costs associated with NAT traversal.

Common endpoint types:

- **Gateway endpoints** for S3 and DynamoDB.
- **Interface endpoints** for many AWS services via AWS PrivateLink.

Example S3 gateway endpoint policy:

```json
{
  "Statement": [
    {
      "Sid": "AllowSpecificBucketOnly",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": [
        "arn:aws:s3:::my-data-bucket",
        "arn:aws:s3:::my-data-bucket/*"
      ]
    }
  ]
}
```

Practical benefits:

- Reduce NAT Gateway data processing charges for S3 access.
- Keep traffic on the AWS backbone.
- Simplify private access from private subnets.

**DataSync for scheduled transfer and migration**

AWS DataSync automates online data transfer between on-premises storage and AWS services such as S3, EFS, and FSx. It is useful for migrations, recurring sync jobs, and large-scale transfers that need verification and scheduling.

DataSync strengths:

- Incremental transfer after the initial sync.
- Scheduling and task execution control.
- Verification options for data integrity.
- Monitoring through CloudWatch and task history.

Typical workflow:

1. Create source and destination locations.
2. Create a task that maps source to destination.
3. Run the task manually or on a schedule.
4. Monitor transfer metrics and errors.
5. Adjust bandwidth, filters, or schedule as needed.

Example DataSync task creation flow:

```bash
aws datasync create-location-nfs \
  --server-hostname nfs.example.internal \
  --subdirectory /exports/data \
  --on-prem-config AgentArns=arn:aws:datasync:us-east-1:123456789012:agent/agent-0123456789abcdef0 \
  --mount-options Version=NFS4_1
```

```bash
aws datasync create-location-s3 \
  --s3-bucket-arn arn:aws:s3:::my-destination-bucket \
  --s3-config BucketAccessRoleArn=arn:aws:iam::123456789012:role/DataSyncS3AccessRole
```

```bash
aws datasync create-task \
  --source-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-aaaaaaaaaaaaaaaaa \
  --destination-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-bbbbbbbbbbbbbbbbb \
  --name nfs-to-s3-migration
```

> **Note:** Use the actual location ARNs returned by the `create-location-*` commands. The task depends on those exact ARNs.

DataSync is usually a better fit than custom scripts when you need:

- Managed retries and verification.
- Large-scale recurring synchronization.
- Hybrid file transfer with operational visibility.

**Direct Connect and VPN in transfer design**

Direct Connect and VPN are not storage services, but they strongly affect storage migration and replication choices.

Use **Direct Connect** when:

- You need predictable throughput for large migrations.
- You want private connectivity to AWS from a data center.
- You are moving large datasets repeatedly or continuously.

Use **Site-to-Site VPN** when:

- You need quick setup.
- You need encrypted connectivity over the internet.
- You want a backup path alongside Direct Connect.

Design considerations:

- Direct Connect does not automatically encrypt traffic; add encryption at the application layer or use VPN over DX if required.
- VPN performance depends on internet conditions and tunnel design.
- For very large transfers, pair Direct Connect with DataSync or Snowball rather than relying on ad hoc uploads.

**AWS Snow family for offline transfer**

When network transfer is too slow, too expensive, or impractical, use the Snow family.

- **Snowcone** for small edge deployments and limited data movement.
- **Snowball Edge** for larger offline transfer and edge compute.
- **Snowmobile** for extremely large migrations.

Use offline transfer when:

- Bandwidth is insufficient for the migration window.
- Data egress costs are prohibitive.
- You need to seed a large dataset before switching to incremental sync.

**Practical scenario guidance**

- Use **S3 lifecycle** for object archives, logs, and versioned content cleanup.
- Use **S3 replication** for DR, compliance, and multi-Region availability.
- Use **S3 Batch Operations** for backfilling or bulk object actions.
- Use **EBS snapshots and DLM** for EC2 block storage backup and retention.
- Use **EFS lifecycle** for shared file systems with changing access patterns.
- Use **FSx** when the workload needs a specialized file system.
- Use **Storage Gateway** for hybrid file, block, or tape integration.
- Use **DataSync** for managed file migration and recurring sync.
- Use **Direct Connect** or **VPN** to improve hybrid transfer paths.
- Use **Snow** when network transfer is not practical.

**Exam traps and decision cues**

- If the question says **shared POSIX file access**, think **EFS** or **FSx**, not S3.
- If it says **Windows SMB with Active Directory**, think **FSx for Windows File Server**.
- If it says **high-throughput parallel analytics**, think **FSx for Lustre**.
- If it says **cloud-backed file shares or tape replacement**, think **Storage Gateway**.
- If it says **automatic object aging and cleanup**, think **S3 lifecycle**.
- If it says **cross-Region disaster recovery for objects**, think **S3 CRR**.
- If it says **bulk migration with verification and scheduling**, think **DataSync**.
- If it says **offline transfer of massive datasets**, think **Snowball**.

**CLI and configuration reminders**

- Use `aws s3api put-bucket-lifecycle-configuration` for lifecycle rules.
- Use `aws s3api put-bucket-replication` for replication configuration.
- Use `aws efs put-lifecycle-configuration` for EFS lifecycle policies.
- Use `aws datasync` commands to create locations and tasks.
- Use `aws ec2 modify-volume` to change EBS volume characteristics.
- Use `aws dlm create-lifecycle-policy` to automate EBS snapshot retention.

**Final design mindset**

Start with the data’s access pattern, then choose the storage service, then automate movement and retention. For SAP-C02, the best answer is usually the one that minimizes operational overhead while meeting durability, latency, compliance, and cost requirements. If a scenario includes growth, versioning, or repeated transfers, always consider lifecycle cleanup, replication scope, and network path cost together rather than as separate problems.
