## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Lab objective and mindset**

Cost optimization is not just “make it cheaper.” In AWS, you review usage patterns, map spend to architecture, and then choose the least disruptive change that preserves performance, reliability, and operational requirements. For SAP-C02, you are expected to recognize when to use rightsizing, commitments, storage tiering, data transfer reduction, and architectural refactoring.

A good review follows a repeatable sequence:

1. Identify the top cost drivers.
2. Determine whether the spend is steady, spiky, idle, or wasteful.
3. Match the remediation to the workload’s access pattern and risk tolerance.
4. Validate the change with metrics and billing data.

**Core cost optimization framework**

Use a decision flow to avoid overengineering the fix.

```text
High spend found
  ├─ Is the resource idle or overprovisioned?
  │    └─ Rightsize, stop, schedule, or delete
  ├─ Is usage steady and predictable?
  │    └─ Consider Savings Plans or Reserved Instances where applicable
  ├─ Is storage data infrequently accessed?
  │    └─ Move to a lower-cost storage class with the right retrieval tradeoff
  ├─ Is network traffic driving cost?
  │    └─ Reduce NAT, cross-AZ, cross-Region, and internet egress charges
  └─ Is the application architecture causing waste?
       └─ Refactor for managed services, caching, batching, or event-driven patterns
```

**How to review AWS cost data**

Start with the billing and usage tools that show where money is going. The goal is to separate “expensive because it is busy” from “expensive because it is poorly designed.”

- **AWS Cost Explorer** helps you visualize spend by service, linked account, tag, region, and usage type.
- **AWS Cost and Usage Report (CUR)** provides the most detailed line-item data for deeper analysis.
- **AWS Budgets** alerts you when actual or forecasted spend exceeds thresholds.
- **Trusted Advisor** highlights idle or underutilized resources and some cost optimization opportunities.
- **Compute Optimizer** recommends rightsizing for supported resources such as EC2, EBS, Lambda, and ECS in many cases.

A practical review often starts with a monthly trend and then drills into the largest services.

```bash
aws ce get-cost-and-usage \
  --time-period Start=2026-03-01,End=2026-04-01 \
  --granularity MONTHLY \
  --metrics UnblendedCost \
  --group-by Type=DIMENSION,Key=SERVICE
```

If you need to inspect a specific account or tag, group by those dimensions instead of service.

```bash
aws ce get-cost-and-usage \
  --time-period Start=2026-03-01,End=2026-04-01 \
  --granularity DAILY \
  --metrics UnblendedCost \
  --group-by Type=TAG,Key=Environment
```

**Common remediation categories**

**1. Rightsize compute**

Rightsizing means matching provisioned capacity to actual demand. It is usually the first remediation because it is low risk and fast to validate.

- **EC2**: reduce instance size, change family, or move to Graviton if the workload is compatible.
- **ECS/EKS**: reduce task or pod requests and limits, then adjust cluster capacity.
- **Lambda**: tune memory and timeout; higher memory can reduce duration, but only if the workload benefits.
- **RDS/Aurora**: choose a smaller instance class or adjust storage and IOPS settings when overprovisioned.

> EBS volume size cannot be reduced in place. If you need a smaller volume, you typically create a new smaller volume and migrate data, or restore from a snapshot into a smaller target where supported by the workload design.

A common exam pattern is a workload with low CPU and memory utilization but high monthly spend. The best answer is often rightsizing, not commitments.

**2. Commit to steady usage**

Use commitments when usage is predictable and long-lived.

- **Savings Plans** are flexible commitments that apply to eligible compute usage.
  - **Compute Savings Plans** can cover EC2, Lambda, and Fargate usage.
  - **EC2 Instance Savings Plans** are more specific but can provide deeper discounts for a chosen family in a region.
- **Reserved Instances** are more service-specific and are commonly associated with EC2 and some database services.
- Use commitments only after you have confidence that the baseline usage will remain stable.

A useful rule:
- Choose **Savings Plans** when you want flexibility across compute services.
- Choose **Reserved Instances** when the workload is stable and you can commit to a specific service or instance pattern.

Example scenario:
- A production API runs 24/7 on EC2 with stable baseline usage.
- A batch job runs on Fargate during business hours.
- A Compute Savings Plan may be a better fit than a narrowly scoped EC2 commitment because it can absorb both patterns.

**3. Optimize storage**

Storage cost optimization depends on access frequency, retrieval latency, and durability requirements.

- **Amazon S3**:
  - Use lifecycle policies to transition objects based on age and access pattern.
  - Use intelligent tiering when access patterns are unknown or change over time.
  - Use archival classes only when retrieval latency and retrieval cost are acceptable.
- **Amazon EBS**:
  - Choose the right volume type for the workload.
  - gp3 often provides a better cost model than older gp2 patterns because performance is decoupled from size.
  - Delete unattached volumes and old snapshots when no longer needed.
- **Amazon EFS**:
  - Use lifecycle management to move infrequently accessed files to lower-cost storage classes where appropriate.

For S3, do not treat archival tiers as a generic “cheaper is always better” choice. Glacier classes reduce storage cost, but retrieval time and retrieval charges matter. For example, a compliance archive may fit archival storage, while a frequently queried log archive may not.

Example lifecycle policy:

```json
{
  "Rules": [
    {
      "ID": "MoveLogsToIAThenArchive",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER"
        }
      ],
      "Expiration": {
        "Days": 365
      }
    }
  ]
}
```

Use lifecycle transitions only when the access pattern supports them. If objects are still queried often after 30 days, moving them too early can increase total cost.

**4. Reduce network and data transfer cost**

Network cost is often hidden until traffic grows.

- **NAT Gateway** charges can become significant when private subnets send large volumes of traffic to AWS services or the internet.
- **Cross-AZ traffic** can add cost when components in different Availability Zones exchange large amounts of data.
- **Cross-Region replication and transfers** can be expensive for large datasets.
- **Internet egress** is often a major cost driver for public-facing applications.

A common remediation is to keep traffic on the AWS backbone where possible.

- Use **gateway endpoints** for supported services such as Amazon S3 and DynamoDB when private subnets need access without NAT.
- Use **interface endpoints** for supported AWS services when private connectivity is needed.
- Place chatty components in the same AZ when architecture and resilience requirements allow it.
- Cache frequently accessed content with CloudFront or application-level caching.

> VPC endpoints are not a universal replacement for NAT. Only use them for services that support the endpoint type you need, and verify whether a gateway endpoint or interface endpoint is available for the target service.

Example VPC endpoint pattern for S3 access from private subnets:

```text
Private subnet EC2 -> S3 gateway endpoint -> Amazon S3
Private subnet EC2 -> NAT Gateway -> Internet/AWS public endpoint
```

The first path is usually cheaper for S3 access because it avoids NAT data processing charges.

**5. Remove waste**

Waste is the easiest cost to eliminate and often the fastest exam answer.

- Delete unused EBS volumes, snapshots, load balancers, and Elastic IPs.
- Stop or schedule nonproduction instances outside business hours.
- Remove orphaned resources after failed deployments.
- Clean up old AMIs and snapshots that are no longer required.
- Review idle databases, test environments, and forgotten analytics clusters.

A simple operational pattern is to tag resources by owner, environment, and expiration date.

```bash
aws ec2 create-tags \
  --resources i-0123456789abcdef0 \
  --tags Key=Environment,Value=Dev Key=Owner,Value=Platform Key=ExpiresOn,Value=2026-05-31
```

**Practical lab workflow**

Use this sequence when you are asked to review and remediate cost in a scenario.

1. Open Cost Explorer and identify the top service by spend.
2. Drill into usage type, region, and linked account.
3. Check whether the spend is driven by:
   - idle capacity
   - steady baseline usage
   - storage growth
   - data transfer
   - logging volume
4. Validate utilization with CloudWatch metrics, Compute Optimizer, or service-specific dashboards.
5. Choose the least disruptive fix:
   - rightsize
   - commit
   - tier storage
   - reduce network hops
   - delete waste
6. Measure the expected savings and confirm that performance and availability remain acceptable.

**Example end-to-end remediation walkthrough**

Suppose a company sees rising monthly spend in three areas: EC2, NAT Gateway, and S3.

- EC2 instances show low average CPU utilization and are running 24/7.
- NAT Gateway data processing is high because private subnets fetch package updates and S3 objects through NAT.
- S3 stores application logs that are rarely read after 30 days.

A strong remediation plan would be:

- Rightsize the EC2 fleet based on utilization and instance recommendations.
- Replace S3 access through NAT with an S3 gateway endpoint where supported.
- Add an S3 lifecycle policy to transition older logs to a lower-cost storage class that matches retrieval needs.
- Delete obsolete snapshots and unattached volumes discovered during the review.

This sequence usually produces savings faster than a large redesign because it targets the highest-cost, lowest-risk changes first.

**CloudWatch and logging cost control**

Logging can become expensive when applications emit too much data or retain logs too long.

- Reduce log verbosity in the application before logs are sent.
- Use structured logging so you can filter at the source or in the pipeline.
- Set retention policies on log groups.
- Export only the logs you need for long-term analysis.

Example retention configuration:

```bash
aws logs put-retention-policy \
  --log-group-name /aws/lambda/orders-api \
  --retention-in-days 14
```

For high-volume systems, consider whether every debug message belongs in CloudWatch Logs. In many cases, the best savings come from changing the application or log shipper, not from trying to filter after ingestion.

**EBS optimization details**

EBS cost is influenced by volume type, size, provisioned performance, and snapshot retention.

- Use **gp3** when you want a more predictable baseline cost than size-linked performance models.
- Use **io2** only when you need high durability and high IOPS for critical workloads.
- Remove unused snapshots, especially old incremental chains that are no longer needed.
- Review root volumes on terminated instances and data volumes left behind after migrations.

If you need to reduce a volume’s size, plan a migration:
- create a new smaller volume
- copy data
- update mount points or application configuration
- validate the workload
- decommission the old volume

That operational detail matters because EBS does not support shrinking a volume in place.

**Choosing between rightsizing, commitments, and architecture changes**

| Situation | Best first action | Why |
|---|---|---|
| Low utilization on running instances | Rightsize | Fastest low-risk savings |
| Stable 24/7 baseline usage | Savings Plan or Reserved Instance | Discounts reward predictability |
| Large S3 archive with rare access | Lifecycle transition | Storage class matches access pattern |
| High NAT Gateway spend | Add endpoints or redesign traffic flow | Reduces data processing charges |
| Many unused resources | Delete or stop | Eliminates waste immediately |
| High log ingestion cost | Reduce log generation at source | Ingestion cost is incurred before retention |

**Exam cues and traps**

- If the workload is steady and predictable, think commitments.
- If the workload is underutilized, think rightsizing before commitments.
- If the cost is from data transfer, inspect NAT, cross-AZ, and cross-Region paths.
- If the cost is from logs, focus on generation and retention, not retroactive filtering.
- If the cost is from storage, match the storage class to the access pattern and retrieval tolerance.
- If the cost is from EBS, remember that shrinking requires migration, not an in-place resize.

**Quick remediation checklist**

- [ ] Identify the top cost driver in Cost Explorer.
- [ ] Confirm utilization with metrics or recommendations.
- [ ] Check for idle or orphaned resources.
- [ ] Compare steady usage against commitment options.
- [ ] Review storage class, retention, and lifecycle policies.
- [ ] Inspect NAT, endpoint, and cross-AZ traffic.
- [ ] Validate the savings after the change.

**Summary**

Cost optimization in AWS is a balancing act between savings and operational fit. The best SAP-C02 answers usually preserve the workload’s behavior while removing waste, matching capacity to demand, and reducing unnecessary data movement. Start with the simplest safe change, verify the impact, and only then consider deeper architectural refactoring.
