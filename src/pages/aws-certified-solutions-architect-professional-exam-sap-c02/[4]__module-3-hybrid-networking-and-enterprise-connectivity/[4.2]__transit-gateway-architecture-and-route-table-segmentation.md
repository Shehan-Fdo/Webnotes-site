## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Transit Gateway overview**

AWS Transit Gateway (TGW) is a regional network transit hub that connects multiple VPCs, VPNs, AWS Direct Connect gateways, and peering attachments through a centralized routing layer. It reduces the need for full mesh peering and gives you a scalable way to segment traffic between environments such as prod, non-prod, shared services, and inspection.

TGW is not a firewall. It is a routing service. You use route tables, associations, and propagations to control which attachments can send traffic to which destinations. For exam scenarios, the key question is usually not “Can TGW connect them?” but “How do you segment them safely while preserving required reachability?”

**Core TGW architecture**

A TGW has three main concepts:

- **Attachments** connect resources to the TGW.
  - VPC attachments
  - VPN attachments
  - Direct Connect attachments through a Direct Connect gateway
  - Peering attachments to another TGW
- **Route tables** decide where traffic goes after it enters the TGW.
- **Associations and propagations** control which attachment uses which route table and which routes appear in that table.

A practical mental model is:

- The attached VPC route table sends traffic to the TGW attachment.
- The TGW route table decides where the TGW forwards that traffic next.
- The destination VPC route table must also know how to return traffic back through the TGW.

**How traffic flows through TGW**

For a packet to traverse TGW successfully, you need routing on both sides:

- In the source VPC, the subnet route table must point the destination CIDR to the TGW attachment.
- In the TGW, the associated route table must contain a route for the destination CIDR.
- In the destination VPC, the subnet route table must point the source CIDR back to the TGW attachment.

This is why TGW designs often fail in practice: engineers configure the TGW side but forget the VPC route tables.

Example VPC route table entry:

```bash
10.20.0.0/16  ->  tgw-attach-abc123
0.0.0.0/0     ->  nat-0abc123def456
```

That route table tells the VPC to send traffic for `10.20.0.0/16` to TGW, while internet-bound traffic still uses NAT.

**Associations and propagations**

TGW route table behavior is often tested in detail.

- **Association** means an attachment uses a specific TGW route table for traffic entering the TGW from that attachment.
- **Propagation** means routes learned from an attachment are inserted into a TGW route table.

Important distinctions:

- An attachment can be associated with only one TGW route table at a time.
- An attachment can propagate into multiple TGW route tables, depending on design.
- A route table can receive propagated routes from multiple attachments.
- Static routes and propagated routes can coexist in the same TGW route table.

A useful exam-safe way to think about association is:

- “When traffic arrives from this attachment, which TGW route table evaluates it?”

A useful way to think about propagation is:

- “Which destination prefixes should this route table learn from this attachment?”

**Route selection behavior**

TGW route selection follows longest-prefix match first.

- The most specific prefix wins.
- If the same destination prefix exists as both static and propagated routes, the **static route takes precedence**.
- Blackhole routes can intentionally override forwarding for a prefix by denying traffic to that destination.

This matters in segmentation designs because you can use static routes to steer traffic through inspection or to block access to sensitive networks.

Example of route preference:

- `10.10.0.0/16` static route to inspection attachment
- `10.10.0.0/16` propagated route from a spoke VPC

TGW uses the static route for that prefix.

**Blackhole routes**

A **blackhole route** is a TGW route table entry that drops traffic for a destination prefix.

Use cases:

- Intentionally deny access to a CIDR block
- Prevent accidental reachability during migration
- Create a safe default deny posture for a segment

Example:

```bash
aws ec2 create-transit-gateway-route \
  --transit-gateway-route-table-id tgw-rtb-1234567890abcdef0 \
  --destination-cidr-block 10.99.0.0/16 \
  --blackhole
```

This is useful when you want a route table to explicitly reject traffic to a network rather than silently relying on absence of a route.

**Segmentation patterns**

TGW route table segmentation is the main design tool for isolating environments.

Common patterns include:

- **Prod / non-prod separation**
  - Prod attachments propagate only into approved route tables.
  - Non-prod attachments do not learn prod prefixes unless explicitly allowed.
- **Shared services hub**
  - Shared services VPC is reachable from selected spokes.
  - Shared services may advertise its CIDR into multiple spoke route tables.
- **Inspection VPC**
  - Traffic is steered through a security or firewall VPC before reaching sensitive destinations.
- **Default deny with explicit allow**
  - Route tables contain only approved routes and blackholes for blocked prefixes.

A simple segmentation model:

- `rtb-prod` learns prod and shared-services routes.
- `rtb-nonprod` learns non-prod and shared-services routes.
- `rtb-shared` learns prod and non-prod only if return traffic is required from shared services to those networks.
- `rtb-inspect` contains routes to all spoke CIDRs so it can forward inspected traffic to the correct destination.

The important exam point is that reachability is determined by the combination of:
- TGW route table entries
- attachment association
- route propagation
- VPC subnet route tables on both sides

**Shared services design**

Shared services commonly include DNS, AD, logging, CI/CD, or internal APIs.

A typical design:

- Shared services VPC attaches to TGW.
- Spoke VPCs send traffic for shared services CIDRs to TGW.
- The TGW route table associated with each spoke contains the shared services CIDR.
- The shared services VPC subnet route table contains routes back to each spoke CIDR via TGW.

This is not “one-way propagation.” It is a two-way routing requirement.

Practical example:

- Prod VPC route table: `10.100.0.0/16 -> TGW`
- TGW route table for prod: `10.50.0.0/16 -> shared-services attachment`
- Shared services VPC route table: `10.10.0.0/16 -> TGW` and `10.20.0.0/16 -> TGW`

If the return route is missing in shared services, the session fails even if the forward path exists.

**Inspection VPC and appliance mode**

An inspection VPC is often used with AWS Network Firewall, third-party firewalls, IDS/IPS, or traffic analysis appliances.

TGW appliance mode helps preserve flow symmetry for traffic that traverses an appliance attachment. It is useful when the appliance must see both directions of a flow on the same attachment.

Important scope notes:

- Appliance mode does **not** create the routing path by itself.
- You still need correct TGW route table entries.
- You still need correct VPC subnet route tables in the spoke and inspection VPCs.
- Appliance mode helps with symmetric forwarding; it does not replace routing design.

A common pattern is:

- Spoke VPC sends traffic to TGW.
- TGW route table sends traffic to inspection attachment.
- Inspection VPC forwards traffic onward to the destination attachment.
- Return traffic follows the reverse path through the same inspection attachment when designed correctly.

This usually requires:
- TGW routes for spoke CIDRs in the inspection route table
- TGW routes for destination CIDRs in the spoke route table
- VPC routes in the inspection VPC that send relevant traffic back to TGW or to the appliance path

> Appliance mode is a routing aid, not a complete security architecture.

**TGW peering**

TGW peering connects two Transit Gateways, usually across regions, to enable inter-region routing.

Key exam distinctions:

- TGW peering is between TGWs, not directly between VPCs.
- TGW peering is not transitive across multiple TGWs.
- Route propagation does **not** occur across a peering attachment.
- You must add routes manually on both sides of the peering relationship.

This differs from VPC peering in important ways:

| Feature | TGW Peering | VPC Peering |
|---|---|---|
| Connects | TGW to TGW | VPC to VPC |
| Transitive routing | No | No |
| Route propagation across link | No | No |
| Centralized hub support | Yes | No |
| Typical use | Multi-region hub-and-spoke | Simple point-to-point connectivity |

For TGW peering, you usually:

- Create a peering attachment in each region.
- Accept the peering attachment.
- Add static routes in each TGW route table pointing remote CIDRs to the peering attachment.
- Update VPC route tables so local VPCs send remote CIDRs to their local TGW.

Example conceptual route entries:

- Region A TGW route table: `10.200.0.0/16 -> peering attachment`
- Region B TGW route table: `10.10.0.0/16 -> peering attachment`

**Multicast**

TGW multicast is a specialized feature for one-to-many delivery within a TGW domain.

Use cases:

- Market data distribution
- Media or telemetry fan-out
- Legacy multicast workloads that need cloud network support

Important notes:

- Multicast is separate from normal unicast routing.
- It uses multicast domains and group member registrations.
- It is not the default choice for most enterprise segmentation designs.
- For SAP-C02, recognize it as a TGW capability, but do not confuse it with standard route table forwarding.

**Limits and quotas**

TGW design is often constrained by quotas, so architecture choices should account for scale.

Common quota categories include:

- Number of attachments per TGW
- Number of route tables per TGW
- Number of routes per TGW route table
- Number of associations per route table
- Number of propagations per route table

Exact values can vary by region and quota settings, so verify current AWS quotas before final design. For exam purposes, the important point is that TGW is scalable but not unlimited, and route table segmentation can be constrained by route count and attachment count.

Design implications:

- Large CIDR sprawl can exhaust route table entries.
- Over-segmentation can increase operational complexity.
- Summarization can reduce route count when CIDR planning allows it.
- Blackhole routes also consume route entries.

**Practical CLI examples**

Create a TGW route table:

```bash
aws ec2 create-transit-gateway-route-table \
  --transit-gateway-id tgw-0123456789abcdef0
```

Associate an attachment with a route table:

```bash
aws ec2 associate-transit-gateway-route-table \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --transit-gateway-attachment-id tgw-attach-0123456789abcdef0
```

Enable propagation from an attachment into a route table:

```bash
aws ec2 enable-transit-gateway-route-table-propagation \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --transit-gateway-attachment-id tgw-attach-0123456789abcdef0
```

Create a static route to a destination CIDR:

```bash
aws ec2 create-transit-gateway-route \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --destination-cidr-block 10.20.0.0/16 \
  --transit-gateway-attachment-id tgw-attach-0fedcba9876543210
```

Create a blackhole route:

```bash
aws ec2 create-transit-gateway-route \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --destination-cidr-block 10.99.0.0/16 \
  --blackhole
```

Inspect routes:

```bash
aws ec2 search-transit-gateway-routes \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --filters Name=route-search.exact-match,Values=true
```

**CloudFormation example**

A minimal CloudFormation snippet for a TGW route table and attachment association:

```yaml
Resources:
  TransitGateway:
    Type: AWS::EC2::TransitGateway
    Properties:
      Description: Core network hub

  ProdRouteTable:
    Type: AWS::EC2::TransitGatewayRouteTable
    Properties:
      TransitGatewayId: !Ref TransitGateway

  ProdVpcAttachment:
    Type: AWS::EC2::TransitGatewayAttachment
    Properties:
      TransitGatewayId: !Ref TransitGateway
      VpcId: vpc-0123456789abcdef0
      SubnetIds:
        - subnet-aaa111
        - subnet-bbb222

  ProdAssociation:
    Type: AWS::EC2::TransitGatewayRouteTableAssociation
    Properties:
      TransitGatewayAttachmentId: !Ref ProdVpcAttachment
      TransitGatewayRouteTableId: !Ref ProdRouteTable
```

This creates the core objects, but you still need VPC route table entries in the attached VPCs.

**Terraform example**

A simple Terraform pattern:

```hcl
resource "aws_ec2_transit_gateway" "core" {
  description = "Core network hub"
}

resource "aws_ec2_transit_gateway_route_table" "prod" {
  transit_gateway_id = aws_ec2_transit_gateway.core.id
}

resource "aws_ec2_transit_gateway_vpc_attachment" "prod" {
  transit_gateway_id = aws_ec2_transit_gateway.core.id
  vpc_id             = aws_vpc.prod.id
  subnet_ids         = [aws_subnet.a.id, aws_subnet.b.id]
}

resource "aws_ec2_transit_gateway_route_table_association" "prod" {
  transit_gateway_attachment_id  = aws_ec2_transit_gateway_vpc_attachment.prod.id
  transit_gateway_route_table_id  = aws_ec2_transit_gateway_route_table.prod.id
}

resource "aws_ec2_transit_gateway_route_table_propagation" "prod" {
  transit_gateway_attachment_id  = aws_ec2_transit_gateway_vpc_attachment.prod.id
  transit_gateway_route_table_id  = aws_ec2_transit_gateway_route_table.prod.id
}
```

Terraform is useful for repeatable segmentation patterns and for keeping route intent version-controlled.

**Console workflow**

A typical AWS Console workflow is:

1. Create the Transit Gateway.
2. Create TGW route tables for each segment.
3. Create VPC attachments and select the correct subnets.
4. Associate each attachment with the intended TGW route table.
5. Enable propagation only where route learning is required.
6. Add static routes for inspection, shared services, or peering.
7. Update VPC subnet route tables to point destination CIDRs to the TGW attachment.
8. Validate connectivity with reachability tests or instance-level probes.

**Troubleshooting checklist**

When traffic fails, check in this order:

- The source VPC subnet route table points the destination CIDR to TGW.
- The TGW attachment is associated with the correct TGW route table.
- The TGW route table contains a route for the destination CIDR.
- The destination VPC subnet route table contains a return route to the source CIDR.
- No blackhole route is unintentionally matching the destination.
- Appliance mode and firewall policy allow the flow.
- Security groups and network ACLs permit the traffic.
- For peering, routes are configured manually on both TGWs.

Useful commands:

```bash
aws ec2 describe-transit-gateway-route-tables
aws ec2 describe-transit-gateway-route-table-associations
aws ec2 describe-transit-gateway-route-table-propagations
aws ec2 describe-transit-gateway-attachments
```

**Exam-focused design patterns**

- **Hub-and-spoke**
  - Use TGW as the central hub.
  - Attach multiple VPCs and segment them with route tables.
- **Shared services**
  - Centralize common services in one VPC.
  - Propagate or statically route only approved prefixes.
- **Inspection insertion**
  - Force traffic through a firewall VPC using static routes and appliance mode.
- **Deny-by-default**
  - Use blackhole routes and minimal propagation.
- **Multi-region connectivity**
  - Use TGW peering with manual route entries on both sides.

**What to remember for SAP-C02**

- TGW route tables control forwarding after traffic enters TGW.
- VPC route tables must still send traffic to TGW.
- Association is one attachment to one route table.
- Propagation is route learning, not forwarding by itself.
- Static routes beat propagated routes for the same prefix.
- Blackhole routes intentionally deny traffic.
- TGW peering requires manual routes on both sides and does not propagate routes.
- Appliance mode helps symmetry but does not replace routing design.
- Quotas and route scale can shape the architecture.
