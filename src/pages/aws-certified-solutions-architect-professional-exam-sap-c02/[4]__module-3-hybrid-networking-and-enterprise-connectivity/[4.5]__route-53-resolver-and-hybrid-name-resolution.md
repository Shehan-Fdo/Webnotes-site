## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Route 53 Resolver overview**

Amazon Route 53 Resolver is the DNS resolution layer for VPCs. It answers queries for public DNS names, private hosted zones, and hybrid forwarding scenarios. In exam scenarios, you use it to connect AWS workloads to on-premises DNS infrastructure without building custom DNS proxies.

Route 53 Resolver is especially important when you need:
- **Private name resolution** inside a VPC.
- **Conditional forwarding** to on-premises DNS servers.
- **Centralized DNS control** across multiple VPCs or accounts.
- **DNS filtering** with Route 53 Resolver DNS Firewall.

A VPC uses the AmazonProvidedDNS resolver path when DNS support is enabled. The standard link-local resolver endpoint is `169.254.169.253`, and the VPC resolver address is also conceptually the VPC CIDR base plus two. If you disable `enableDnsSupport`, instances lose access to the VPC DNS resolver path, which breaks Route 53 Resolver features, private hosted zone resolution, and interface endpoint private DNS resolution.

**Core DNS resolution flow**

When an instance in a VPC resolves a name, Route 53 Resolver evaluates the request against the VPC’s DNS configuration and associated rules. The practical order depends on the exact namespace and associations, but the key idea is that the resolver chooses the most specific applicable match and then applies the relevant resolution behavior.

Typical resolution sources include:
- **Public DNS** via recursive resolution to the internet.
- **Private hosted zones** associated with the VPC.
- **Resolver rules** associated with the VPC.
- **DNS Firewall** rule groups associated with the VPC.

For exam purposes, avoid oversimplifying overlap behavior. A private hosted zone does not universally “win” over a forwarding rule in every overlap case. Resolution depends on the exact name queried, the associated zones, the associated rules, and the matching specificity.

**Resolver endpoints architecture**

Route 53 Resolver endpoints provide network entry and exit points for hybrid DNS.

- **Inbound endpoints** let on-premises DNS servers forward queries into AWS for private hosted zone resolution.
- **Outbound endpoints** let VPCs forward selected queries to on-premises DNS servers or other DNS resolvers.
- Endpoints are **regional**.
- Each endpoint uses **ENIs** in selected subnets.
- You choose one IP address per subnet for each endpoint direction, and multi-AZ placement improves resilience.
- Security groups and subnet routing must allow DNS traffic to and from the endpoint ENIs.

A common design is:
- Two subnets in different AZs for the endpoint ENIs.
- Security groups allowing UDP/TCP 53 from trusted source networks.
- Network connectivity through **VPN**, **AWS Direct Connect**, or **Transit Gateway** for hybrid reachability.

> Route 53 Resolver endpoints do not create connectivity by themselves. They only provide DNS entry points; the network path between AWS and on-premises must already exist.

**Inbound endpoint use cases**

Use an inbound endpoint when on-premises systems need to resolve AWS private names.

Common pattern:
- On-premises DNS server forwards `corp.aws.example.com` or a private zone suffix to the inbound endpoint IPs.
- The inbound endpoint resolves the query using VPC-associated private hosted zones and AWS DNS behavior.

Example conditional forwarder on a Windows DNS server:
```powershell
Add-DnsServerConditionalForwarderZone `
  -Name "aws.internal.example.com" `
  -MasterServers 10.0.10.10,10.0.20.10 `
  -ReplicationScope Forest
```

Example `BIND` forward zone:
```conf
zone "aws.internal.example.com" {
    type forward;
    forward only;
    forwarders { 10.0.10.10; 10.0.20.10; };
};
```

Use inbound endpoints when:
- On-premises applications need private AWS service names.
- You want centralized DNS resolution for AWS private zones.
- You need hybrid name resolution without duplicating records on-premises.

**Outbound endpoint use cases**

Use an outbound endpoint when workloads in AWS need to resolve on-premises names.

Common pattern:
- A Route 53 Resolver rule forwards `corp.example.com` to on-premises DNS servers.
- The rule is associated with one or more VPCs.
- Queries from those VPCs are sent through the outbound endpoint to the target DNS servers.

Example forwarding rule concept:
- Domain: `corp.example.com`
- Target IPs: `10.1.0.10`, `10.1.0.11`
- Protocol: UDP/TCP 53
- Associated VPCs: application VPCs that need on-premises resolution

A practical `BIND` conditional forwarder on-premises is the mirror image of the AWS setup:
```conf
zone "aws.internal.example.com" {
    type forward;
    forward only;
    forwarders { 172.16.10.10; 172.16.20.10; };
};
```

**Resolver rules and associations**

Resolver rules define how Route 53 Resolver handles queries for a domain namespace. The most common rule type is a **forwarding rule**. AWS also uses system behavior for default resolution, but `RECURSIVE` is not a standard Route 53 Resolver rule type.

Important mechanics:
- A rule is associated with one or more VPCs.
- A VPC can use shared rules from another account through **AWS Resource Access Manager (RAM)**.
- Shared rules still must be associated with the consuming VPC to take effect.
- If multiple rules could apply, the most specific matching namespace is the key factor.
- Rule priority and overlap behavior matter in design and troubleshooting.

Practical implications:
- Use a narrow forwarding rule for a subdomain when you want only that namespace sent on-premises.
- Avoid broad forwarding rules that unintentionally capture names you intended to resolve privately in AWS.
- Validate rule associations in every VPC that needs the behavior.

**Private hosted zones in hybrid designs**

A private hosted zone (PHZ) is a Route 53 hosted zone associated with one or more VPCs. It is the standard way to publish private DNS records for AWS resources.

Use PHZs when:
- You want internal names for EC2, load balancers, RDS, or service endpoints.
- You need split-horizon DNS with different answers for public and private clients.
- You want AWS-native private DNS without external DNS servers.

Example private hosted zone creation:
```bash
aws route53 create-hosted-zone \
  --name aws.internal.example.com \
  --caller-reference "$(date +%s)" \
  --hosted-zone-config PrivateZone=true \
  --vpc VPCRegion=us-east-1,VPCId=vpc-0123456789abcdef0
```

Example associating an additional VPC:
```bash
aws route53 associate-vpc-with-hosted-zone \
  --hosted-zone-id Z1234567890ABC \
  --vpc VPCRegion=us-west-2,VPCId=vpc-0fedcba9876543210
```

Useful inspection command:
```bash
aws route53 list-hosted-zones-by-vpc \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-region us-east-1
```

Important caveats:
- `list-hosted-zones-by-vpc` shows hosted zones associated with that specific VPC, not every private hosted zone in the account.
- Overlapping PHZs and forwarding rules require careful design; do not assume one always overrides the other in every case.
- If you need deterministic behavior, keep namespaces clean and avoid ambiguous overlaps.

**Hybrid name resolution patterns**

Hybrid DNS usually follows one of these patterns:

1. **AWS to on-premises**
   - VPC instances query Route 53 Resolver.
   - A forwarding rule sends `corp.example.com` to on-premises DNS.
   - On-premises DNS answers with internal records.

2. **On-premises to AWS**
   - On-premises DNS forwards `aws.internal.example.com` to an inbound endpoint.
   - Route 53 Resolver answers from private hosted zones.

3. **Centralized DNS hub**
   - A shared services VPC hosts inbound and outbound endpoints.
   - Rules are shared to application VPCs through RAM.
   - Transit Gateway or VPN provides network reachability to on-premises DNS.

4. **Split-horizon DNS**
   - Public names resolve publicly.
   - Private names resolve through PHZs or forwarding rules.
   - Different clients receive different answers based on where they query from.

A common enterprise topology:
- On-premises DNS servers forward AWS private zones to inbound endpoints.
- Application VPCs forward corporate zones to outbound endpoints.
- Shared DNS rules are managed centrally and associated to multiple VPCs.

**DNS Firewall**

Route 53 Resolver DNS Firewall lets you inspect and control DNS queries that pass through Route 53 Resolver in a VPC. It is not a general network firewall; it only affects DNS resolution through the resolver path.

Core concepts:
- **Domain lists** define allowed or blocked domains.
- **Rule groups** contain firewall rules.
- **Associations** attach rule groups to VPCs.
- **Priorities** determine evaluation order when multiple rule groups apply.

Common actions:
- **BLOCK**: Prevent resolution for matching domains.
- **ALLOW**: Permit resolution for matching domains.
- **ALERT**: Log the query and allow resolution.

Practical behavior:
- Rule group priority matters when multiple groups are associated.
- DNS Firewall evaluates queries on the resolver path, not traffic outside Route 53 Resolver.
- It is useful for blocking known malicious domains, enforcing egress DNS policy, and auditing suspicious lookups.

Example conceptual rule group layout:
- Priority 100: block malware domains
- Priority 200: alert on newly registered domains
- Priority 300: allow internal exception list

**Query logging**

Route 53 Resolver query logging records DNS queries that pass through the resolver. This is useful for troubleshooting, auditing, and security investigations.

Supported destinations include:
- **CloudWatch Logs**
- **Amazon S3**

Use query logging when you need:
- Visibility into failed lookups.
- Evidence of unexpected DNS behavior.
- Audit trails for hybrid DNS traffic.

Practical workflow:
1. Create a query log destination.
2. Create a query log configuration.
3. Associate the configuration with one or more VPCs.
4. Review logs in the destination service.

Example CloudWatch Logs destination concept:
```bash
aws logs create-log-group --log-group-name /aws/route53/resolver/querylogs
```

Then create the Route 53 Resolver query log destination and configuration using the Route 53 Resolver APIs or CLI in the documented resource model for your region and CLI version. The exact command shape differs from ordinary CloudWatch log publishing, so validate against the current AWS CLI reference before automation.

**Security and network requirements**

Hybrid DNS depends on both DNS configuration and network reachability.

Checklist:
- Allow UDP 53 and TCP 53 between clients and resolver endpoints.
- Ensure security groups permit DNS traffic to the endpoint ENIs.
- Ensure route tables and network appliances allow the path between AWS and on-premises DNS servers.
- Use Transit Gateway, VPN, or Direct Connect for private connectivity.
- Confirm that on-premises firewalls allow return traffic and DNS responses.

For inbound endpoints:
- On-premises DNS servers must be able to reach the endpoint IPs.
- The endpoint subnet must be reachable from on-premises through the chosen connectivity path.

For outbound endpoints:
- The endpoint must be able to reach target DNS servers on port 53.
- The target DNS servers must accept queries from the endpoint IPs.

**Scaling and resilience**

Route 53 Resolver endpoints are regional and should be deployed for fault tolerance.

Design guidance:
- Place endpoint IPs in at least two AZs.
- Use multiple IPs per endpoint for high availability.
- Size the design for query volume and failover.
- Monitor resolver metrics and query logs for saturation or unusual spikes.

Exam-relevant considerations:
- High query volume may require additional endpoint IPs and careful subnet planning.
- A single-AZ endpoint design creates a DNS dependency on that AZ.
- Shared services DNS should be treated as critical infrastructure.

**Troubleshooting hybrid DNS**

When DNS fails, isolate the failure domain in this order:
1. Client-side resolver configuration.
2. VPC DNS support and DHCP options.
3. PHZ association and record existence.
4. Resolver rule association and namespace match.
5. Endpoint reachability and security groups.
6. On-premises DNS forwarding configuration.
7. Firewall or network ACL restrictions.

Common failure modes:
- Conditional forwarder points to the wrong endpoint IPs.
- The rule exists but is not associated with the VPC.
- The on-premises DNS server cannot reach the inbound endpoint subnet.
- Security groups block UDP/TCP 53.
- A broad forwarding rule captures names that should resolve in a private hosted zone.
- `enableDnsSupport` is disabled in the VPC.
- DNS Firewall blocks the query unexpectedly.

Useful checks:
```bash
dig @169.254.169.253 aws.internal.example.com
dig @10.0.10.10 corp.example.com
nslookup aws.internal.example.com 10.0.10.10
```

If a query works from one VPC but not another, compare:
- VPC association to the PHZ.
- Resolver rule associations.
- Shared rule permissions through RAM.
- DHCP options and custom DNS settings.

**Exam tips**

- Remember that Route 53 Resolver is the DNS engine for VPCs, not just a forwarding feature.
- Distinguish between **inbound** and **outbound** endpoints.
- Know that endpoints are regional and use ENIs in subnets.
- Treat PHZs, resolver rules, and DNS Firewall as separate but interacting layers.
- Be careful with overlap assumptions; specificity and association matter.
- Use RAM for cross-account sharing of resolver rules and hosted zone access patterns where appropriate.
- For hybrid architectures, always verify the network path first, then DNS configuration.

**Practical architecture example**

A retail company runs application workloads in AWS and Active Directory-integrated DNS on-premises.

Design:
- `aws.internal.example.com` is a private hosted zone in Route 53.
- `corp.example.com` is resolved by on-premises DNS.
- Two inbound endpoint IPs are deployed in separate AZs.
- Two outbound endpoint IPs are deployed in separate AZs.
- A forwarding rule for `corp.example.com` is associated with all application VPCs.
- On-premises DNS forwards `aws.internal.example.com` to the inbound endpoint IPs.
- DNS Firewall blocks known malicious domains and alerts on suspicious categories.
- Query logging is enabled to S3 for audit retention.

Why this works:
- AWS workloads resolve corporate names without custom DNS servers.
- On-premises workloads resolve AWS private names without public exposure.
- Multi-AZ endpoints reduce DNS single points of failure.
- Centralized logging and filtering improve security and troubleshooting.

**Key takeaways**

Route 53 Resolver is the foundation of hybrid DNS in AWS. Use private hosted zones for AWS-private names, resolver rules for conditional forwarding, inbound and outbound endpoints for cross-network DNS flow, and DNS Firewall plus query logging for policy and visibility. In exam questions, focus on namespace specificity, VPC associations, endpoint reachability, and the difference between DNS resolution behavior and network connectivity.
