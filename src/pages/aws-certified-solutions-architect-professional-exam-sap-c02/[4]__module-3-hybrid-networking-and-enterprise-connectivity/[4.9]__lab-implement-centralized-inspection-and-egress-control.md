## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Centralized inspection and egress control overview**

A centralized inspection architecture places security controls in a dedicated inspection VPC so multiple spoke VPCs can send traffic through a common enforcement point. In AWS, this pattern usually combines **AWS Transit Gateway (TGW)**, **AWS Network Firewall**, and controlled internet egress through a **NAT Gateway** or other approved path.

This design helps you:
- enforce consistent outbound policy across many VPCs
- inspect east-west and north-south traffic in a single place
- reduce duplicated firewall management in each application VPC
- support compliance requirements for logging, filtering, and segmentation

A typical exam scenario asks you to preserve scalability, keep routing symmetric, and avoid exposing workloads directly to the internet.

**Reference topology**

A practical lab topology often uses:
- one **inspection VPC** with two or more Availability Zones (AZs)
- one **spoke VPC** for workloads
- one **egress path** through NAT Gateway and an internet gateway
- one **Transit Gateway** to connect the VPCs
- one **Network Firewall** endpoint per AZ in dedicated firewall subnets

Example CIDR plan:
- Spoke VPC: `10.10.0.0/16`
- Inspection VPC: `10.20.0.0/16`
- Firewall subnets: `10.20.10.0/24` in AZ-a, `10.20.20.0/24` in AZ-b
- Egress subnets: `10.20.30.0/24` in AZ-a, `10.20.40.0/24` in AZ-b
- Workload subnets: `10.10.1.0/24` in AZ-a, `10.10.2.0/24` in AZ-b

The key design rule is symmetry: traffic from a given AZ should enter and return through the same firewall endpoint path whenever possible.

**Core traffic flow**

The standard centralized egress path is:

1. Workload instance sends traffic to its subnet route table.
2. The spoke subnet route table sends `0.0.0.0/0` to the **Transit Gateway**.
3. The Transit Gateway route table sends the traffic to the inspection VPC attachment.
4. The inspection VPC route table sends the traffic to the **Network Firewall endpoint** in the same AZ.
5. The firewall inspects the traffic and forwards allowed traffic to the egress subnet.
6. The egress subnet route table sends internet-bound traffic to the **NAT Gateway**.
7. The NAT Gateway sends traffic to the internet gateway.

Return traffic follows the reverse path. If you break symmetry, stateful inspection can fail or traffic can become intermittent.

> For stateful inspection, keep the forward and return path aligned within the same AZ whenever possible. Cross-AZ routing can work in some designs, but it increases cost and complexity and can break assumptions during failover.

**AWS Network Firewall concepts**

AWS Network Firewall is a managed, stateful firewall service that supports:
- **stateful rules** for connection-aware inspection
- **stateless rules** for high-speed packet filtering
- **domain list rules** for domain-based allow/deny control
- **logging** for alerts and flow records
- **TLS inspection** in supported scenarios, if your design requires it

Important terms:
- **Firewall policy**: defines rule groups, default actions, and logging behavior
- **Rule group**: reusable set of stateless or stateful rules
- **Firewall**: deployed resource that creates endpoints in selected subnets
- **Firewall endpoint**: managed endpoint used as a route target in the VPC route table

Use stateful rules when you need connection tracking, protocol awareness, or domain-based controls. Use stateless rules when you want simple packet-level filtering or fast prefiltering.

**Firewall policy design**

A firewall policy ties together rule groups and default behavior. In exam scenarios, pay attention to:
- whether the policy is **strict** or **default** stateful
- whether you want default deny or default pass behavior
- whether you need alert-only rules before enforcement
- whether logging is enabled for both flow and alert events

Example policy skeleton:

```json
{
  "FirewallPolicyName": "central-egress-policy",
  "FirewallPolicy": {
    "StatelessDefaultActions": [
      "aws:forward_to_sfe"
    ],
    "StatelessFragmentDefaultActions": [
      "aws:forward_to_sfe"
    ],
    "StatefulRuleGroupReferences": [
      {
        "ResourceArn": "arn:aws:network-firewall:us-east-1:123456789012:stateful-rulegroup/allow-web"
      }
    ],
    "StatefulDefaultActions": [
      "aws:drop_strict"
    ],
    "StreamExceptionPolicy": "DROP"
  }
}
```

Notes:
- `aws:drop_strict` is useful when you want a strict default-deny posture for stateful traffic.
- `aws:forward_to_sfe` sends packets to the stateful engine.
- The exact rule evaluation behavior depends on the policy mode and rule group types, so do not treat default actions as a substitute for explicit allow rules.

**Rule group examples**

A stateful rule group often uses Suricata-compatible rules. Example allow rule group:

```json
{
  "RuleGroupName": "allow-web",
  "Type": "STATEFUL",
  "Capacity": 100,
  "RuleGroup": {
    "RulesSource": {
      "RulesString": "pass tcp any any -> any 80 (msg:\"Allow HTTP\"; sid:1001; rev:1;)\npass tcp any any -> any 443 (msg:\"Allow HTTPS\"; sid:1002; rev:1;)"
    }
  }
}
```

A stateless rule group can block known bad ports before stateful inspection:

```json
{
  "RuleGroupName": "block-common-abuse",
  "Type": "STATELESS",
  "Capacity": 100,
  "RuleGroup": {
    "StatelessRulesAndCustomActions": {
      "StatelessRules": [
        {
          "RuleDefinition": {
            "MatchAttributes": {
              "Protocols": [6],
              "DestinationPorts": [
                { "FromPort": 23, "ToPort": 23 },
                { "FromPort": 445, "ToPort": 445 }
              ]
            },
            "Actions": ["aws:drop"]
          },
          "Priority": 1
        }
      ]
    }
  }
}
```

Use stateless rules for coarse filtering and stateful rules for deeper inspection.

**Firewall creation and subnet placement**

When you create the firewall, you select subnets in the inspection VPC. AWS creates one endpoint per selected subnet/AZ. Those endpoints are the route targets you use in the inspection VPC route tables.

Example CLI pattern:

```bash
aws network-firewall create-firewall \
  --firewall-name central-egress-fw \
  --firewall-policy-arn arn:aws:network-firewall:us-east-1:123456789012:firewall-policy/central-egress-policy \
  --vpc-id vpc-0abc1234def567890 \
  --subnet-mappings SubnetId=subnet-aaa11111 SubnetId=subnet-bbb22222
```

Practical notes:
- choose one firewall subnet per AZ you want to support
- keep firewall subnets separate from egress subnets
- route tables in the inspection VPC must point to the firewall endpoint, not directly to the internet

**Route table design by layer**

A complete design usually needs distinct route tables for each layer.

**1. Spoke workload subnet route table**
- `10.20.0.0/16` → Transit Gateway attachment or TGW route target
- `0.0.0.0/0` → Transit Gateway attachment
- optional internal prefixes → TGW or local routes as needed

**2. Transit Gateway route table**
- spoke VPC CIDR → inspection VPC attachment
- return routes for other spoke CIDRs if you have multiple spokes
- avoid accidental direct internet paths from spokes

**3. Inspection VPC route table for ingress/inspection subnets**
- spoke CIDR(s) → Network Firewall endpoint in the same AZ
- other internal CIDRs → Network Firewall endpoint or local routing as required
- do not send workload traffic directly to NAT from the inspection ingress side

**4. Egress subnet route table**
- `0.0.0.0/0` → NAT Gateway in the same AZ
- internal CIDRs → TGW or local routes as needed
- keep the NAT Gateway in the same AZ as the firewall endpoint when possible to preserve symmetry and reduce cross-AZ data transfer

**5. Public subnet route table**
- `0.0.0.0/0` → Internet Gateway
- this subnet hosts the NAT Gateway, not the workloads

A common mistake is to treat the firewall endpoint like a normal instance ENI. You do not manage the endpoint ENI directly; you route to the managed firewall endpoint target that AWS exposes for the firewall subnet.

**Example route creation workflow**

A realistic sequence is:

1. Create the inspection VPC and subnets.
2. Create the firewall policy and rule groups.
3. Create the firewall in the inspection VPC.
4. Wait for firewall endpoints to become available.
5. Update the inspection VPC route tables to send traffic to the firewall endpoints.
6. Create the NAT Gateway in the public subnet.
7. Update the egress subnet route tables to send `0.0.0.0/0` to the NAT Gateway.
8. Update spoke subnet route tables to send default traffic to the Transit Gateway.
9. Update Transit Gateway route tables to point spoke traffic to the inspection attachment.
10. Validate with test instances and logs.

**Example validation commands**

Use the AWS CLI to inspect the firewall and route tables:

```bash
aws network-firewall describe-firewall \
  --firewall-name central-egress-fw

aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0abc1234def567890

aws ec2 describe-nat-gateways \
  --filter Name=vpc-id,Values=vpc-0abc1234def567890
```

For a test instance in the spoke VPC:

```bash
curl -I https://example.com
nslookup example.com
traceroute 8.8.8.8
```

If DNS resolution fails, check Route 53 Resolver configuration and security groups before assuming the firewall is blocking traffic.

**Route 53 Resolver and DNS control**

Centralized egress is not only about HTTP and HTTPS. DNS can bypass inspection if instances can query external resolvers directly.

Use **Route 53 Resolver** to control DNS flow:
- create inbound endpoints so on-premises or other networks can resolve private hosted zones
- create outbound endpoints and forwarding rules to send selected domains to approved resolvers
- restrict outbound UDP/TCP port 53 to approved DNS paths
- block direct access to public resolvers where policy requires it

Example inbound endpoint CLI pattern:

```bash
aws route53resolver create-resolver-endpoint \
  --creator-request-id inbound-dns-001 \
  --name central-inbound-dns \
  --direction INBOUND \
  --security-group-ids sg-0123456789abcdef0 \
  --ip-addresses SubnetId=subnet-aaa11111,Ip=10.20.10.10 SubnetId=subnet-bbb22222,Ip=10.20.20.10
```

Important:
- use at least two IP addresses in different subnets for high availability
- choose IPs inside the subnet CIDR and not already in use
- security groups must allow DNS traffic from the expected sources

For outbound DNS control, create resolver rules and associate them with the VPCs that should use them.

**DNS Firewall and bypass prevention**

If you need stronger DNS policy enforcement, consider **Route 53 Resolver DNS Firewall**. It lets you:
- block known malicious domains
- allow only approved domains
- log DNS queries for investigation

This is useful when workloads might otherwise bypass network firewall controls by using direct DNS over standard resolvers. DNS Firewall complements Network Firewall; it does not replace it.

**Logging and observability**

Network Firewall logging is essential for troubleshooting and audit. You typically enable:
- **flow logs** for connection metadata
- **alert logs** for rule matches and drops

A logging configuration must point to a valid destination such as CloudWatch Logs, Amazon S3, or Kinesis Data Firehose, using the service’s expected destination structure.

Example CloudWatch Logs destination pattern:

```json
{
  "LogDestinationConfig": [
    {
      "LogType": "ALERT",
      "LogDestinationType": "CloudWatchLogs",
      "LogDestination": {
        "logGroup": "/aws/network-firewall/alerts"
      }
    },
    {
      "LogType": "FLOW",
      "LogDestinationType": "CloudWatchLogs",
      "LogDestination": {
        "logGroup": "/aws/network-firewall/flow"
      }
    }
  ]
}
```

After configuration, validate with CloudWatch Logs:

```bash
aws logs tail /aws/network-firewall/alerts --follow
aws logs tail /aws/network-firewall/flow --follow
```

> CloudWatch Logs only shows firewall events after you configure the destination and grant the required permissions. If logs do not appear, verify the firewall logging configuration, log group names, and IAM permissions first.

**High availability and failover behavior**

Design for AZ failure from the start:
- deploy firewall endpoints in at least two AZs
- deploy NAT Gateways in the same AZs as the firewall endpoints
- keep spoke subnets distributed across the same AZs
- prefer AZ-local routing to avoid cross-AZ hairpinning

If one firewall endpoint fails:
- traffic in that AZ may fail until routes are updated or the service recovers
- you may need automation to shift routes to a healthy AZ in a controlled way
- do not assume automatic failover across AZs for every route table design

If one NAT Gateway fails:
- internet egress from that AZ can fail
- you can fail over to another NAT Gateway, but that may introduce cross-AZ charges and asymmetric routing concerns
- in exam answers, mention both resilience and cost trade-offs

**Symmetric routing considerations**

Stateful inspection depends on seeing both directions of a flow. Keep these principles in mind:
- route outbound and return traffic through the same inspection path
- avoid sending request traffic through one AZ and response traffic through another unless the design explicitly supports it
- ensure TGW route tables and VPC route tables do not create alternate return paths
- verify that security groups and NACLs do not block the return flow

A simple rule of thumb:
- spoke subnet default route points to TGW
- TGW points to inspection VPC
- inspection VPC points to firewall endpoint
- firewall subnet points to NAT Gateway
- NAT Gateway points to IGW

**Practical validation checklist**

Use this checklist after deployment:
- confirm firewall endpoints exist in each selected subnet
- confirm route tables point to the correct next hop in each AZ
- confirm NAT Gateways are in public subnets with IGW access
- confirm spoke instances can resolve DNS through the intended resolver path
- confirm firewall logs show allowed and denied traffic
- confirm blocked destinations are actually blocked
- confirm return traffic follows the same AZ path where required

**Common exam pitfalls**

- confusing the firewall endpoint with a customer-managed ENI
- sending spoke traffic directly to NAT Gateway instead of through inspection
- forgetting the Transit Gateway route table association and propagation model
- using a single AZ for all inspection components
- omitting DNS control and allowing direct resolver bypass
- assuming a firewall policy alone creates routing
- forgetting that logging requires explicit destination configuration
- designing asymmetric routes that break stateful inspection

**When to choose this architecture**

Use centralized inspection and egress control when you need:
- a shared security boundary for many VPCs
- centralized logging and compliance controls
- consistent outbound filtering
- inspection before internet access
- a scalable hub-and-spoke network design

It is less attractive when:
- you have only one small VPC and simple security needs
- latency must be minimized and local egress is acceptable
- the operational overhead of shared inspection outweighs the benefit

**Exam-ready summary**

For SAP-C02, the key ideas are:
- use **Transit Gateway** to centralize routing between spoke VPCs and the inspection VPC
- use **AWS Network Firewall** for stateful and stateless inspection
- route traffic to the **firewall endpoint** in the inspection VPC, not to a manually managed ENI
- send internet-bound traffic from the egress subnet to a **NAT Gateway**
- preserve **symmetric routing** within each AZ
- control DNS with **Route 53 Resolver** and, when needed, **DNS Firewall**
- enable logging so you can prove policy enforcement and troubleshoot failures

**Minimal end-to-end example**

```text
Spoke subnet (10.10.1.0/24)
  0.0.0.0/0 -> Transit Gateway

Transit Gateway route table
  10.10.0.0/16 -> Inspection VPC attachment

Inspection VPC route table
  10.10.0.0/16 -> Network Firewall endpoint in AZ-a

Egress subnet route table
  0.0.0.0/0 -> NAT Gateway in AZ-a

Public subnet route table
  0.0.0.0/0 -> Internet Gateway
```

This pattern gives you centralized control, inspectable traffic, and a clear path for troubleshooting.
