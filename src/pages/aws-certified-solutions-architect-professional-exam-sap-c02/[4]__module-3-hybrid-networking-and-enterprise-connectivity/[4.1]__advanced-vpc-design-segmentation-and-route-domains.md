## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Advanced VPC design: the architectural goal**

A well-designed <abbr title="Virtual Private Cloud">VPC</abbr> is not just a private network container. It is a set of deliberate boundaries that control reachability, inspection, blast radius, and routing behavior across workloads, accounts, and environments. For SAP-C02, you should think beyond “public vs private subnets” and design for segmentation, controlled egress, centralized inspection, and predictable route domains.

The main design questions are:

- Which workloads should share a routing domain?
- Where should traffic be inspected or filtered?
- How do you prevent accidental lateral movement?
- How do you scale the design across multiple accounts and Regions?

A strong answer usually combines subnet segmentation, security groups, network ACLs, route tables, Transit Gateway, and centralized network services.

**Core VPC segmentation concepts**

Segmentation means separating workloads so that only intended traffic can flow between them. In AWS, segmentation is enforced at multiple layers:

- **Security groups** provide stateful instance- or ENI-level filtering.
- **Network ACLs** provide stateless subnet-level filtering.
- **Route tables** define where traffic goes next.
- **VPC boundaries** isolate address space and routing domains.
- **AWS Transit Gateway** connects multiple VPCs while preserving segmentation through attachments and route tables.

A practical design often uses separate subnets for:

- Public-facing load balancers
- Application tiers
- Data tiers
- Shared services
- Inspection appliances
- Management or bastion access

This separation helps you apply different controls to each tier. For example, a database subnet should not have a route to the internet gateway, while an application subnet may need controlled egress through a NAT gateway or inspection VPC.

**Security groups vs network ACLs**

Security groups and network ACLs solve different problems, and you should use them together rather than as substitutes.

| Control | Scope | State | Typical use |
|---|---|---:|---|
| Security group | ENI / instance | Stateful | Allow only required app flows |
| Network ACL | Subnet | Stateless | Coarse subnet guardrails |

Key points:

- **Security groups** are the primary control for most workload traffic.
- **Network ACLs** are useful for explicit deny rules, subnet-level boundaries, and simple guardrails.
- Security groups automatically allow return traffic for permitted connections.
- Network ACLs require explicit inbound and outbound rules for both directions.

A common pattern is to allow application traffic from a load balancer security group to an application security group, then allow database access only from the application security group. This avoids IP-based rules where possible and makes the design more resilient to scaling.

Example security group rules:

```bash
# Allow HTTPS from an ALB security group to app instances
aws ec2 authorize-security-group-ingress \
  --group-id sg-app \
  --protocol tcp \
  --port 443 \
  --source-group sg-alb

# Allow PostgreSQL from app tier to database tier
aws ec2 authorize-security-group-ingress \
  --group-id sg-db \
  --protocol tcp \
  --port 5432 \
  --source-group sg-app
```

**Subnet design and tiering**

Subnet design should reflect both availability and trust boundaries. In a multi-AZ architecture, you usually create matching subnet sets in each Availability Zone.

Typical subnet categories:

- **Public subnets**
  - Route to an internet gateway.
  - Host internet-facing load balancers, NAT gateways, or bastion hosts if used.
- **Private application subnets**
  - No direct internet route.
  - Reach outbound services through NAT or centralized egress.
- **Isolated data subnets**
  - No internet route and often no NAT route.
  - Host databases, caches, or internal services with strict access.
- **Inspection subnets**
  - Host firewalls, IDS/IPS, or traffic analysis appliances.
- **Shared services subnets**
  - Host DNS resolvers, directory services, or internal tooling.

A good subnet plan also reserves address space for growth. Overlapping or too-small CIDR blocks create long-term constraints, especially when you later connect VPCs through Transit Gateway or VPN.

**Route tables as route domains**

A route table is the practical expression of a route domain inside a VPC. Every subnet is associated with exactly one route table at a time, and that route table determines the next hop for traffic leaving the subnet.

Important route targets include:

- `local` for intra-VPC traffic
- Internet gateway for public internet access
- NAT gateway for outbound-only internet access from private subnets
- Transit Gateway for inter-VPC or hybrid routing
- Virtual private gateway for VPN or Direct Connect
- VPC endpoints for private access to AWS services
- Network firewall or appliance ENI for inspection paths

A route domain is the set of destinations reachable through a given routing policy. In practice, you create separate route tables when you want different subnets to have different reachability.

Example route table patterns:

- Public subnet route table
  - `0.0.0.0/0 -> igw-...`
  - `::/0 -> eigw-...` for IPv6 egress if used
- Private app subnet route table
  - `0.0.0.0/0 -> nat-...`
  - Specific prefixes to Transit Gateway for shared services or on-premises
- Isolated subnet route table
  - Only `local` and specific internal routes
  - No default route to internet or NAT

Example CLI inspection:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-1234567890abcdef0
```

**When to split route tables**

You split route tables when traffic policy differs. Common reasons include:

- Different egress requirements for app and data tiers
- Separate routing for inspection versus direct egress
- Distinct on-premises prefixes for different business units
- Controlled access to shared services or partner networks
- Segregation of production and non-production traffic

Avoid creating route tables just because you can. Too many route tables can make operations harder unless each one represents a meaningful policy boundary.

**Centralized egress design**

Centralized egress reduces the number of internet-facing paths and makes inspection easier. Instead of every VPC sending traffic directly to the internet, you can route outbound traffic through a shared egress VPC.

Common centralized egress components:

- NAT gateways for IPv4 outbound internet access
- AWS Network Firewall or third-party appliances for filtering
- Transit Gateway to connect spoke VPCs to the egress VPC
- VPC endpoints to reduce internet dependency for AWS services

A typical flow is:

1. Workload subnet sends `0.0.0.0/0` to Transit Gateway.
2. Transit Gateway routes traffic to an egress VPC attachment.
3. Egress VPC sends traffic through firewall or NAT.
4. Return traffic follows the same path if routing is symmetric.

This design is useful when you need:

- Centralized logging
- Consistent outbound policy
- Reduced public IP exposure
- Shared inspection and compliance controls

> Be careful with asymmetric routing. Stateful appliances and firewalls often require return traffic to follow the same path. If routes differ in each direction, sessions may fail.

**Transit Gateway segmentation and route tables**

<abbr title="AWS Transit Gateway">TGW</abbr> is a hub that connects many VPCs and on-premises networks. It is especially important for route domain design because it lets you segment attachments using separate TGW route tables.

Core concepts:

- **Attachments** connect VPCs, VPNs, or Direct Connect gateways to TGW.
- **TGW route tables** control which attachments can reach which destinations.
- **Propagation** automatically adds routes from an attachment into a TGW route table.
- **Association** determines which TGW route table an attachment uses for its own traffic.

This allows hub-and-spoke segmentation such as:

- Production VPCs can reach shared services and egress.
- Development VPCs can reach shared services but not production.
- Inspection VPCs see traffic from selected spokes.
- On-premises networks can be limited to specific VPCs.

Example conceptual design:

```text
[Prod VPC] ----\
[Dev VPC] -------> [Transit Gateway] ---> [Shared Services VPC]
[On-Prem] ----/              |
                             +------> [Inspection / Egress VPC]
```

A useful pattern is to create separate TGW route tables for:

- Production
- Non-production
- Shared services
- Inspection
- On-premises

Then associate each attachment with the appropriate table and propagate only the routes that should be visible.

Example CLI commands:

```bash
aws ec2 create-transit-gateway-route-table \
  --transit-gateway-id tgw-1234567890abcdef0

aws ec2 associate-transit-gateway-route-table \
  --transit-gateway-route-table-id tgw-rtb-1234567890abcdef0 \
  --transit-gateway-attachment-id tgw-attach-1234567890abcdef0

aws ec2 enable-transit-gateway-route-table-propagation \
  --transit-gateway-route-table-id tgw-rtb-1234567890abcdef0 \
  --transit-gateway-attachment-id tgw-attach-1234567890abcdef0
```

**Inspection VPC patterns**

Inspection VPCs are used when you need to examine, filter, or log traffic centrally. They are common in regulated environments and large enterprises.

Typical inspection components:

- AWS Network Firewall
- Gateway Load Balancer with third-party appliances
- Traffic Mirroring for packet-level analysis
- VPC Flow Logs for metadata-level visibility

Common traffic flow:

- Spoke VPC routes default traffic to TGW.
- TGW sends traffic to inspection VPC.
- Inspection VPC forwards allowed traffic to egress or destination.
- Return traffic is routed back through the same inspection path.

Use inspection VPCs when you need:

- Centralized policy enforcement
- Threat detection
- Compliance logging
- Consistent outbound filtering across many VPCs

Use caution with appliance scaling and failover. If the inspection layer is stateful, you need a design that preserves session continuity and supports AZ failure.

**VPC endpoints and private service access**

VPC endpoints reduce the need for internet access and improve segmentation. They are especially important in isolated or tightly controlled subnets.

Endpoint types:

- **Gateway endpoints** for Amazon S3 and DynamoDB
- **Interface endpoints** for many AWS services via PrivateLink

Benefits:

- Traffic stays on the AWS network.
- You can remove or reduce NAT dependency.
- You can keep workloads in isolated subnets while still reaching AWS services.
- You can apply endpoint policies for additional control.

Example endpoint policy idea:

- Allow only specific S3 buckets.
- Allow only specific AWS service actions.
- Restrict access to a particular VPC or principal where supported.

Example CLI:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-1234567890abcdef0 \
  --service-name com.amazonaws.us-east-1.s3 \
  --route-table-ids rtb-1234567890abcdef0
```

For interface endpoints, remember that you need security group rules on the endpoint ENIs and DNS resolution configured appropriately.

**IPv4, IPv6, and dual-stack considerations**

Advanced VPC designs increasingly use dual-stack networking. IPv6 changes routing and egress assumptions.

Important points:

- IPv6 addresses are globally unique, so you do not use NAT for IPv6 in the same way as IPv4.
- For outbound-only IPv6 internet access, use an egress-only internet gateway.
- Dual-stack subnets may need separate route entries for IPv4 and IPv6.
- Security group and NACL rules must account for both protocols if workloads use both.

Example route table entries:

```text
0.0.0.0/0   -> nat-1234567890abcdef0
::/0        -> eigw-1234567890abcdef0
```

Use IPv6 when you want:

- Large address space
- Reduced NAT dependency
- Modern internet-facing architectures
- Cleaner end-to-end addressing in some environments

**Hybrid connectivity and segmentation**

When you connect AWS to on-premises networks, route domains become even more important. You must control which prefixes are advertised and which subnets can reach corporate networks.

Common hybrid options:

- Site-to-Site VPN
- AWS Direct Connect
- Transit Gateway with VPN and Direct Connect attachments
- Virtual private gateway for simpler single-VPC designs

Design considerations:

- Keep on-premises routes separate from internet routes.
- Advertise only required prefixes.
- Use TGW route tables to prevent unintended transitive access.
- Consider overlapping CIDR risk before connecting networks.

A common enterprise pattern is:

- Production VPCs can reach specific on-premises application services.
- Non-production VPCs can reach only shared tooling or test systems.
- On-premises users can reach bastion or management services, not all private subnets.

**Overlapping CIDR and address planning**

Overlapping CIDR blocks are a major constraint in multi-VPC and hybrid designs. If two networks overlap, routing becomes ambiguous and many connectivity options become difficult or impossible.

Best practices:

- Allocate non-overlapping CIDR ranges early.
- Reserve space for future VPCs, Regions, and environments.
- Plan for mergers, acquisitions, and partner connectivity.
- Use consistent CIDR allocation by environment or business unit.

Example planning approach:

- `10.0.0.0/16` for production
- `10.1.0.0/16` for non-production
- `10.2.0.0/16` for shared services
- `10.3.0.0/16` for inspection and egress

This is not mandatory, but a structured plan makes route domain management much easier.

**Practical route table examples**

A private application subnet route table might look like this:

```text
Destination        Target
10.0.0.0/8         local
0.0.0.0/0          nat-0abc1234def567890
172.16.0.0/12      tgw-1234567890abcdef0
192.168.100.0/24   vpce-0abc1234def567890
```

An isolated database subnet route table might look like this:

```text
Destination        Target
10.0.0.0/8         local
172.16.10.0/24     tgw-1234567890abcdef0
```

The database subnet has no default route, so it cannot reach the internet directly. If it needs patching or package access, you usually provide that through controlled internal services, endpoints, or a management path.

**Operational visibility and troubleshooting**

When route domains fail, the issue is often not the instance itself but the path. You should verify routing, security, and DNS in that order.

Useful tools:

- VPC Flow Logs for accepted and rejected traffic patterns
- Reachability Analyzer for path validation
- `traceroute` or `tracepath` for hop observation where allowed
- `ip route` and `netstat` or `ss` on the instance
- Route table and TGW route table inspection in the console or CLI

Example troubleshooting commands on Linux:

```bash
ip addr show
ip route show
ss -tulpn
curl -I https://example.com
```

Example AWS checks:

```bash
aws ec2 describe-network-interfaces --filters Name=vpc-id,Values=vpc-1234567890abcdef0
aws ec2 describe-vpc-endpoints --filters Name=vpc-id,Values=vpc-1234567890abcdef0
aws ec2 search-transit-gateway-routes \
  --transit-gateway-route-table-id tgw-rtb-1234567890abcdef0 \
  --filters Name=route-search.exact-match,Values=10.1.0.0/16
```

**Design patterns to remember for SAP-C02**

For exam scenarios, look for these patterns:

- **Hub-and-spoke** for centralized connectivity and governance
- **Shared services VPC** for common internal tools and DNS
- **Inspection VPC** for firewalling and logging
- **Egress VPC** for centralized outbound internet access
- **Isolated data tier** for databases and sensitive systems
- **PrivateLink and endpoints** for private AWS service access
- **TGW route tables** for segmentation between environments

When choosing a design, ask:

- Does the workload need internet access?
- Can you replace internet access with endpoints?
- Should traffic be inspected centrally?
- Is the traffic east-west, north-south, or hybrid?
- Do you need separate route domains for compliance or blast-radius control?

**Common exam traps**

- Assuming a security group alone provides subnet isolation.
- Using NAT for traffic that should remain private through VPC endpoints.
- Forgetting that network ACLs are stateless.
- Ignoring asymmetric routing when firewalls or appliances are in the path.
- Overlooking overlapping CIDR blocks before hybrid connectivity.
- Treating Transit Gateway as a security boundary by itself; it is a routing hub, not a firewall.
- Allowing broad `0.0.0.0/0` routes in subnets that should be isolated.

**Quick reference checklist**

- Define subnet tiers by trust and function.
- Use security groups for primary workload filtering.
- Use NACLs for subnet guardrails and explicit denies.
- Separate route tables when reachability differs.
- Prefer VPC endpoints over NAT for AWS service access.
- Use TGW route tables to segment environments and shared services.
- Plan CIDR ranges to avoid overlap.
- Validate return paths for stateful appliances.
- Log and test routes before production rollout.

**Example architecture summary**

A mature enterprise VPC design often includes:

- Public subnets for load balancers
- Private app subnets with NAT or endpoint-based egress
- Isolated data subnets with no internet route
- Shared services VPC for DNS and internal tooling
- Inspection VPC for firewalling
- Transit Gateway for multi-VPC and hybrid connectivity
- VPC endpoints for private AWS service access

This combination gives you controlled segmentation, predictable route domains, and a scalable foundation for multi-account AWS networking.
