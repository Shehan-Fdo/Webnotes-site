## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Hybrid connectivity overview**

AWS Direct Connect provides a private network path from your on-premises environment to AWS. When you combine it with <abbr title="Transit Gateway">TGW</abbr>, you build a scalable hub for multiple VPCs and accounts without creating many one-off VPN or VPC peering links.

A typical design uses:

- **Direct Connect connection** for the physical private circuit.
- **Transit virtual interface (transit VIF)** to exchange routes with a Direct Connect gateway.
- **Direct Connect gateway (DXGW)** as the global Direct Connect routing construct.
- **Transit Gateway** as the regional hub that attaches VPCs and can associate with a DXGW.

A simple topology looks like this:

```text
On-premises routers
   |
   |  BGP over transit VIF
   v
Direct Connect connection
   |
   v
Direct Connect gateway (global)
   |
   |  association
   v
Transit Gateway (regional)
  /   |    \
VPC-A VPC-B VPC-C
```

The key idea is that the DXGW is the bridge between Direct Connect and one or more TGWs, while the TGW distributes routes to attached VPCs in its Region.

**Core concepts and when to use them**

Use TGW with Direct Connect when you need centralized hybrid routing for many VPCs, accounts, or environments. This is common in shared-services, landing-zone, and enterprise network designs.

- **Transit Gateway**:
  - Acts as a regional routing hub.
  - Connects VPCs, VPNs, and Direct Connect integrations.
  - Simplifies transitive routing between many networks.
- **Direct Connect gateway**:
  - Is a global Direct Connect construct.
  - Can associate with TGWs in supported Regions.
  - Does not attach to VPCs directly.
- **Transit VIF**:
  - Carries private routing between on-premises and AWS through DXGW.
  - Is required for TGW integration.
  - Uses BGP to exchange routes.
- **Private VIF**:
  - Is used when connecting a Direct Connect circuit to a single VGW or private AWS network path.
  - Is not the usual choice for TGW-based hub-and-spoke designs.
- **Public VIF**:
  - Is for public AWS service access over Direct Connect.
  - Is unrelated to TGW routing.

> A Direct Connect gateway is not a replacement for TGW. It is the Direct Connect-side routing object that lets your transit VIF reach one or more TGWs.

**Prerequisites and design constraints**

Before you build the lab, confirm the following:

- You have an active Direct Connect connection or a hosted Direct Connect setup that supports transit VIFs.
- Your on-premises edge device supports <abbr title="Border Gateway Protocol">BGP</abbr>.
- You have planned autonomous system numbers (<abbr title="Autonomous System Number">ASN</abbr>s) for both sides.
- Your TGW is in a Region supported by Direct Connect gateway associations.
- Your design accounts for route limits, redundancy, and failover.

Important constraints to remember:

- A transit VIF is required for TGW integration; a private VIF to a VGW is a different pattern.
- A DXGW can associate with TGWs in different Regions, but each association is explicit and subject to AWS support and design constraints.
- Route exchange is controlled by BGP advertisements and the association between the DXGW and TGW; do not assume automatic propagation everywhere.
- TGW route table propagation is configured on TGW attachments, not on the DXGW.
- If you need a route to appear in a TGW route table, the TGW attachment and route table association/propagation settings must be correct.

**BGP planning for the transit VIF**

BGP is the control plane that learns and advertises routes across Direct Connect. Good ASN planning prevents conflicts and makes troubleshooting easier.

Typical planning points:

- Use a unique private ASN for your on-premises edge if possible.
- Choose the Amazon-side ASN for the TGW carefully if you need to align with existing routing policy.
- Advertise only the prefixes you want AWS to learn.
- Summarize routes where appropriate, but avoid over-aggregation that would blackhole traffic.

Example BGP considerations:

- On-premises advertises:
  - `10.10.0.0/16`
  - `10.20.0.0/16`
- AWS advertises:
  - VPC CIDRs attached to TGW
  - Any propagated routes that the TGW learns from attached networks

A practical BGP session on the transit VIF often includes:

- **BGP neighbor IPs** on the Direct Connect link.
- **MD5 authentication** if required by your environment.
- **Hold timer** and **keepalive** values that match your operational standards.
- Route filters on the on-premises router to prevent accidental advertisement of unwanted prefixes.

Example Cisco-style configuration sketch:

```text
router bgp 65010
 neighbor 169.254.100.1 remote-as 64512
 neighbor 169.254.100.1 description AWS Direct Connect transit VIF
 neighbor 169.254.100.1 password <shared-secret>
 address-family ipv4
  network 10.10.0.0 mask 255.255.0.0
  network 10.20.0.0 mask 255.255.0.0
  neighbor 169.254.100.1 activate
 exit-address-family
```

**End-to-end route flow**

A useful mental model is to trace one prefix in each direction.

Example: on-premises subnet `10.10.0.0/16` to VPC `172.31.0.0/16`

1. On-premises router advertises `10.10.0.0/16` over BGP on the transit VIF.
2. The transit VIF sends the route to the DXGW.
3. The DXGW association makes the route available to the associated TGW.
4. The TGW learns the route and, if propagation is enabled on the correct TGW route table, installs it.
5. VPC route tables point traffic for `10.10.0.0/16` to the TGW attachment.
6. Return traffic from the VPC follows the TGW route table back to the DXGW and then to on-premises.

For the reverse path, the VPC CIDR must be advertised or otherwise known to the on-premises router through BGP so return traffic can find the AWS side.

**Lab topology and route tables**

A common lab design uses a hub-and-spoke TGW with one shared-services VPC and one or more workload VPCs.

```text
                    +----------------------+
                    |   On-premises LAN    |
                    | 10.10.0.0/16         |
                    +----------+-----------+
                               |
                               | BGP
                               v
                    +----------------------+
                    | Direct Connect link  |
                    +----------+-----------+
                               |
                               v
                    +----------------------+
                    | DX Gateway (global)  |
                    +----------+-----------+
                               |
                               | association
                               v
                    +----------------------+
                    | TGW (Region A)       |
                    +---+--------------+---+
                        |              |
                        |              |
                 +------+--+      +----+------+
                 | VPC App |      | VPC Shared|
                 |172.31/16 |      |172.32/16  |
                 +---------+      +-----------+
```

Route table responsibilities:

- **On-premises router**
  - Learns AWS VPC CIDRs via BGP.
  - Advertises on-premises prefixes to AWS.
- **TGW route table**
  - Contains routes to VPC attachments and the DXGW-associated path.
  - Uses propagation and static routes as needed.
- **VPC subnet route tables**
  - Send on-premises prefixes to the TGW attachment.
  - Send inter-VPC traffic to the TGW when the TGW is the hub.

> Use the actual on-premises CIDRs in route tables. Do not use a broad supernet unless it truly represents your advertised aggregate.

**AWS CLI workflow**

The exact API shape can vary by CLI version, but the workflow is consistent: create the TGW, create or identify the DXGW, create the transit VIF, associate the DXGW with the TGW, and then verify route exchange.

1. Create the Transit Gateway.

```bash
aws ec2 create-transit-gateway \
  --description "Hybrid hub for Direct Connect" \
  --options AmazonSideAsn=64512,AutoAcceptSharedAttachments=enable,DefaultRouteTableAssociation=disable,DefaultRouteTablePropagation=disable
```

2. Create or identify the Direct Connect gateway.

```bash
aws directconnect create-direct-connect-gateway \
  --direct-connect-gateway-name "dxgw-hybrid"
```

3. Associate the DXGW with the TGW.

```bash
aws directconnect associate-direct-connect-gateway \
  --direct-connect-gateway-id dxgw-xxxxxxxx \
  --gateway-id tgw-xxxxxxxx \
  --add-allowed-prefixes-to-direct-connect-gateway cidr=10.10.0.0/16 cidr=10.20.0.0/16
```

4. Accept the association if your workflow requires explicit acceptance.

```bash
aws directconnect accept-direct-connect-gateway-association-proposal \
  --direct-connect-gateway-id dxgw-xxxxxxxx \
  --proposal-id dxcon-proposal-xxxxxxxx
```

5. Create the transit VIF on the Direct Connect connection.

```bash
aws directconnect create-transit-virtual-interface \
  --connection-id dxcon-xxxxxxxx \
  --new-transit-virtual-interface \
    Asn=65010,AuthKey=sharedsecret,Vlan=101,AddressFamily=ipv4,AmazonAddress=169.254.100.2,CustomerAddress=169.254.100.1,DirectConnectGatewayId=dxgw-xxxxxxxx
```

6. Associate the TGW attachment to the correct TGW route table and enable propagation where needed.

```bash
aws ec2 associate-transit-gateway-route-table \
  --transit-gateway-route-table-id tgw-rtb-xxxxxxxx \
  --transit-gateway-attachment-id tgw-attach-xxxxxxxx

aws ec2 enable-transit-gateway-route-table-propagation \
  --transit-gateway-route-table-id tgw-rtb-xxxxxxxx \
  --transit-gateway-attachment-id tgw-attach-xxxxxxxx
```

> Verify the exact parameter names and accepted enum values in your CLI version before running lab commands. AWS APIs are strict about identifiers and option values.

**Practical route validation**

Validation should confirm three things: the BGP session is up, the routes are exchanged, and the TGW route tables point traffic correctly.

Useful checks on the on-premises router:

```text
show ip bgp summary
show ip bgp
show ip route bgp
```

What you want to see:

- BGP neighbor state is `Established`.
- AWS-advertised VPC prefixes appear in the BGP table.
- Your on-premises prefixes are being advertised to AWS.

Useful AWS checks:

```bash
aws directconnect describe-virtual-interfaces
aws directconnect describe-direct-connect-gateway-associations
aws ec2 search-transit-gateway-routes \
  --transit-gateway-route-table-id tgw-rtb-xxxxxxxx \
  --filters Name=route-search.exact-match,Values=10.10.0.0/16
```

Additional TGW route inspection:

```bash
aws ec2 get-transit-gateway-route-table-associations \
  --transit-gateway-route-table-id tgw-rtb-xxxxxxxx

aws ec2 get-transit-gateway-route-table-propagations \
  --transit-gateway-route-table-id tgw-rtb-xxxxxxxx
```

What to verify:

- The TGW route table contains the on-premises prefixes learned through the DXGW association.
- The VPC attachment is associated with the intended TGW route table.
- Propagation is enabled only where you want learned routes to appear.
- The VPC subnet route table points on-premises prefixes to the TGW attachment.

**Troubleshooting patterns**

If traffic fails, isolate the failure domain in order:

- **Physical or link layer**
  - Confirm the Direct Connect connection is up.
  - Check VLAN and BGP neighbor IP configuration.
- **BGP layer**
  - Confirm the session is established.
  - Check ASN mismatch, authentication, and prefix filters.
- **DXGW association**
  - Confirm the DXGW is associated with the intended TGW.
  - Confirm allowed prefixes are correct.
- **TGW route tables**
  - Confirm association and propagation settings.
  - Confirm the route exists in the correct TGW route table.
- **VPC route tables**
  - Confirm subnet routes send on-premises prefixes to the TGW attachment.
- **Security controls**
  - Confirm security groups, network ACLs, and host firewalls allow the traffic.

Common failure modes:

- On-premises advertises a prefix that is not allowed by the DXGW association.
- The TGW attachment is associated with the wrong route table.
- Propagation is enabled on the wrong TGW route table.
- The VPC subnet route table points to the wrong target.
- Return traffic is missing because AWS prefixes are not advertised back to on-premises.

**High availability and redundancy**

Direct Connect is often deployed in pairs for resilience. A single circuit is a single point of failure, so production designs usually include multiple connections and diverse paths.

Recommended patterns:

- Use at least two Direct Connect connections in separate locations when possible.
- Terminate connections on redundant edge devices.
- Use diverse fiber paths and separate colocation facilities if available.
- Build multiple transit VIFs if your design needs independent failure domains.
- Consider VPN as a backup path for failover if Direct Connect is unavailable.

Failover behavior depends on BGP:

- Lower BGP local preference can make one path primary.
- AS path prepending can make a path less preferred.
- BGP timers influence convergence speed.
- Route withdrawal from one connection should allow the alternate path to take over.

A common hybrid failover design:

- Primary path: Direct Connect to TGW.
- Secondary path: Site-to-Site VPN to TGW.
- On-premises routing prefers Direct Connect.
- If Direct Connect fails, BGP withdraws routes and VPN becomes active.

**Direct Connect gateway limitations and practical implications**

The DXGW is powerful, but it is not unlimited.

Keep these points in mind:

- It is used for private routing between on-premises and AWS, not for public service access.
- It requires a transit VIF for TGW integration.
- It does not replace TGW route tables or VPC route tables.
- Route exchange is still governed by the TGW and VPC attachment model.
- You must design around supported Regions and association rules.
- Prefix limits and route scale can affect large enterprise designs, so summarize carefully and monitor route counts.

**Worked example: route decision path**

Suppose:

- On-premises: `10.10.0.0/16`
- App VPC: `172.31.0.0/16`
- Shared-services VPC: `172.32.0.0/16`

Traffic from `10.10.5.25` to `172.31.8.10` follows this path:

1. The on-premises router sends the packet to the Direct Connect link.
2. BGP has already learned `172.31.0.0/16` from AWS.
3. The packet enters the transit VIF and reaches the DXGW.
4. The DXGW forwards it to the associated TGW.
5. The TGW route table matches `172.31.0.0/16` to the App VPC attachment.
6. The App VPC subnet route table accepts the traffic and delivers it to the instance.

Return traffic from `172.31.8.10` to `10.10.5.25` uses the reverse route learned through TGW and DXGW back to the on-premises router.

**Exam-focused takeaways**

- **TGW** is the regional hub for VPC and hybrid connectivity.
- **DXGW** is the Direct Connect-side global routing construct that links transit VIFs to TGWs.
- **Transit VIF** is required for TGW integration over Direct Connect.
- **BGP** controls route exchange, so ASN planning and prefix advertisement matter.
- **Route tables** must be correct at both the TGW and VPC layers.
- **Redundancy** is essential; design for multiple paths and fast failover.
- **Validation** should include BGP state, learned routes, TGW route table entries, and VPC subnet routes.

**Quick lab checklist**

- Confirm Direct Connect connectivity and transit VIF readiness.
- Create or identify the DXGW.
- Create the TGW with the intended route table behavior.
- Associate the DXGW with the TGW.
- Attach VPCs to the TGW.
- Configure TGW route table association and propagation.
- Update VPC subnet route tables to point on-premises prefixes to the TGW.
- Verify BGP sessions and route exchange.
- Test traffic in both directions.
- Validate failover if a backup path exists.
