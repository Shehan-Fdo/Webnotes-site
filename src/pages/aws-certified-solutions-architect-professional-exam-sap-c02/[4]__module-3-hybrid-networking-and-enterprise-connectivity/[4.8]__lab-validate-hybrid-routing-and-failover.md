## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Lab goal and validation mindset**

This lab focuses on proving that hybrid connectivity behaves the way you expect under normal conditions and during failure. In practice, you validate three things:

- **Reachability** between on-premises and AWS networks.
- **Route selection** across redundant paths.
- **Failover behavior** when a tunnel, connection, or device becomes unavailable.

Hybrid routing usually involves one or more of these AWS components:

- **Site-to-Site VPN** for encrypted IPsec connectivity over the internet.
- **AWS Direct Connect** for private, dedicated connectivity.
- **Transit Gateway (TGW)** as the routing hub between VPCs and hybrid attachments.
- **Border Gateway Protocol (BGP)** for dynamic route exchange.

The key idea is not just “does traffic work,” but “does traffic fail over predictably, and can you prove it with route tables, BGP state, and packet tests.”

**Reference topology**

A common validation setup looks like this:

- On-premises network: `192.168.0.0/16`
- AWS VPC: `10.0.0.0/16`
- Transit Gateway attached to the VPC and hybrid edge
- Two VPN tunnels or a Direct Connect plus VPN backup
- BGP enabled on the hybrid edge device

A simplified flow is:

```text
On-prem router
   |\
   | \__ VPN tunnel 1
   |____ VPN tunnel 2
        \
         Transit Gateway ---- VPC subnets
```

If Direct Connect is present, it often becomes the preferred path for steady-state traffic, with VPN acting as backup.

**Core concepts to validate**

You should verify the following layers separately because a green check at one layer does not guarantee end-to-end success:

- **Layer 3 routing**: routes exist in the correct tables and propagate where expected.
- **BGP adjacency**: peers are established and exchanging prefixes.
- **Path preference**: the intended route is selected based on local preference, AS path, MED, or static route priority.
- **Failover timing**: traffic shifts after a link, tunnel, or BGP session failure.
- **Return path symmetry**: replies come back through a valid route, especially when firewalls or NAT are involved.

**AWS routing components and what to inspect**

**Transit Gateway route tables**

Transit Gateway route tables determine how attachments forward traffic. You typically inspect:

- **Static routes** for explicitly pinned destinations.
- **Propagated routes** learned from VPN, Direct Connect gateway, or VPC attachments.
- **Association** of each attachment to the correct route table.

Useful CLI examples:

```bash
aws ec2 search-transit-gateway-routes \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --filters Name=state,Values=active \
  --query 'Routes[*].{Destination:DestinationCidrBlock,Type:Type,State:State,Attachment:TransitGatewayAttachments[0].TransitGatewayAttachmentId}' \
  --output table
```

```bash
aws ec2 get-transit-gateway-route-table-associations \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0
```

```bash
aws ec2 get-transit-gateway-route-table-propagations \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0
```

What to look for:

- The on-prem prefix appears in the TGW route table.
- The correct attachment is the target for that prefix.
- Propagation is enabled from the expected attachment.
- No more-specific route unexpectedly overrides the intended path.

> `search-transit-gateway-routes` is useful for inspecting TGW route state, but you must use valid filters and interpret the route type carefully. Do not assume a filter name from another command applies here.

**VPC route tables**

Inside the VPC, confirm that subnet route tables point hybrid prefixes to the TGW attachment, not to an internet gateway or NAT gateway.

Example:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'RouteTables[*].Routes[*].{Destination:DestinationCidrBlock,Target:TransitGatewayId,State:State}' \
  --output table
```

Typical route entry:

```text
Destination: 192.168.0.0/16
Target: tgw-0123456789abcdef0
State: active
```

**Site-to-Site VPN**

A single AWS Site-to-Site VPN connection provides **two tunnels** for redundancy. Both tunnels are established independently, and both can carry traffic when routing and device policy allow it. In BGP-based designs, traffic preference depends on routing attributes and your on-premises configuration, not on a simple AWS-side active/standby switch.

Important validation points:

- Each tunnel has its own outside IP addresses.
- Each tunnel has its own inside tunnel CIDRs.
- BGP peers are configured per tunnel.
- Tunnel health can differ even when the VPN connection is “available.”

Example tunnel inspection:

```bash
aws ec2 describe-vpn-connections \
  --vpn-connection-ids vpn-0123456789abcdef0 \
  --query 'VpnConnections[0].VgwTelemetry[*].{OutsideIp:OutsideIpAddress,Status:Status,LastStatusChange:LastStatusChange,AcceptedRouteCount:AcceptedRouteCount}' \
  --output table
```

What to validate:

- `Status` is `UP` for the tunnel you expect to use.
- `AcceptedRouteCount` is nonzero when routes are being learned.
- The tunnel’s BGP session is established on the customer gateway device.

**Direct Connect**

Direct Connect validation focuses on the physical and logical state of the connection and the virtual interface.

Check:

- **Connection state**: the circuit is up.
- **Virtual interface state**: the VIF is available.
- **BGP session**: the peer is established on the customer router.
- **Route advertisement**: prefixes are exchanged in the expected direction.

Example commands:

```bash
aws directconnect describe-connections \
  --connection-id dxcon-abc12345 \
  --query 'connections[0].{Id:connectionId,State:connectionState,Region:region}' \
  --output table
```

```bash
aws directconnect describe-virtual-interfaces \
  --virtual-interface-id dxvif-abc12345 \
  --query 'virtualInterfaces[0].{Id:virtualInterfaceId,State:virtualInterfaceState,Vlan:vlan,Asn:asn,AuthKey:authKey}' \
  --output table
```

For BGP health, rely on the router-side session state and the AWS-side virtual interface state rather than expecting a dedicated `bgpStatus` field in the CLI output.

**BGP validation on the customer gateway**

BGP is the main mechanism that proves hybrid failover is working. On your on-premises router or firewall, inspect:

- Neighbor state
- Received prefixes
- Advertised prefixes
- Best path selection
- Hold timer and keepalive behavior

Examples on common platforms:

```bash
show ip bgp summary
show ip route bgp
show ip bgp neighbors
```

```bash
show bgp summary
show bgp neighbor
show route protocol bgp
```

A healthy session typically shows:

- Neighbor state `Established`
- Prefix counts greater than zero when routes are exchanged
- Stable uptime for the session

If you use link-local BGP peer addresses on VPN or Direct Connect, remember that the exact addresses are assigned per tunnel or virtual interface and are not universal AWS constants. Treat any sample addresses as examples only.

**Route preference and failover behavior**

Failover is only meaningful if you know which path should win before the failure. Common preference patterns include:

- **Primary Direct Connect, backup VPN**
  - Prefer DX using local preference or administrative distance.
  - Keep VPN routes available but less preferred.
- **Dual VPN tunnels**
  - Allow both tunnels to advertise routes.
  - Use BGP attributes to influence which tunnel is preferred.
- **Multiple DX links**
  - Use BGP policy to prefer one connection and keep another as standby.
- **TGW with multiple attachments**
  - Control propagation and route table association to steer traffic.

Typical BGP tools for path control:

- **Local preference**: higher is preferred within the same autonomous system (AS).
- **AS path prepending**: makes a path less attractive to external peers.
- **MED (Multi-Exit Discriminator)**: suggests preferred entry point to a neighbor AS.
- **Route filtering**: prevents unwanted prefixes from being advertised.

Example on a router using policy to prefer DX over VPN:

```text
route-map PREFER-DX permit 10
 set local-preference 200

route-map PREFER-VPN permit 10
 set local-preference 100
```

If your platform uses administrative distance instead of local preference, the same principle applies: make the preferred route more attractive and the backup route less attractive.

**Practical validation workflow**

1. **Confirm baseline routing**
   - Verify TGW route tables, VPC route tables, and on-prem BGP sessions.
   - Confirm the expected prefixes are present on both sides.

2. **Test end-to-end reachability**
   - Ping or trace from on-prem to an EC2 instance in the VPC.
   - Test application ports, not just ICMP, if security policy allows.

3. **Record the active path**
   - Use traceroute or router path inspection to confirm whether traffic uses VPN or Direct Connect.
   - Check BGP best-path output on the customer gateway.

4. **Simulate a failure**
   - Disable one tunnel, shut down a BGP neighbor, or disconnect a DX interface in a controlled lab.
   - Avoid making changes on production paths without a rollback plan.

5. **Observe convergence**
   - Watch for route withdrawal and re-advertisement.
   - Measure how long traffic interruption lasts.

6. **Restore the path**
   - Re-enable the failed component.
   - Confirm the original preferred path returns if your policy is designed that way.

**Example validation commands**

On AWS, inspect route propagation and attachment state:

```bash
aws ec2 describe-transit-gateway-attachments \
  --filters Name=transit-gateway-id,Values=tgw-0123456789abcdef0 \
  --query 'TransitGatewayAttachments[*].{Id:TransitGatewayAttachmentId,ResourceType:ResourceType,State:State}' \
  --output table
```

On the customer router, inspect BGP and routes:

```bash
show ip bgp summary
show ip route 10.0.0.0
show ip route 192.168.0.0
```

On an EC2 instance, verify the return path with traceroute if allowed:

```bash
traceroute 192.168.1.10
```

Or on Linux:

```bash
ip route
ping -c 4 192.168.1.10
```

**Troubleshooting patterns**

**No route in AWS**

If the prefix does not appear in the TGW route table or VPC route table:

- Check whether propagation is enabled.
- Confirm the attachment is associated with the correct TGW route table.
- Verify the on-prem router is advertising the prefix.
- Ensure the prefix is not filtered by a route policy.

**BGP established but traffic fails**

If BGP is up but packets do not flow:

- Check security groups and network ACLs.
- Confirm the return path exists.
- Verify that the destination subnet route table points to the TGW.
- Look for asymmetric routing through a firewall that drops return traffic.

**Tunnel up, no prefixes learned**

If a VPN tunnel is `UP` but routes are missing:

- Confirm BGP is enabled on both ends.
- Check the inside tunnel IP configuration.
- Verify the customer gateway ASN matches the expected peer ASN.
- Inspect route filters on the router.

**Failover does not happen**

If traffic stays on the failed path or does not move as expected:

- Confirm the failed path is actually withdrawn from BGP.
- Check whether the backup route is being suppressed by policy.
- Verify the backup path is present and eligible.
- Review timers; convergence may take longer than expected.

**Design considerations for exam scenarios**

For SAP-C02, you should be able to reason about hybrid routing without relying on memorized commands alone. Focus on these decision points:

- Use **VPN** when you need encrypted connectivity quickly or as a backup to DX.
- Use **Direct Connect** when you need predictable throughput, lower jitter, or private connectivity.
- Use **Transit Gateway** when you need scalable hub-and-spoke routing across multiple VPCs and hybrid attachments.
- Use **BGP** when you need dynamic failover and route exchange.
- Use **static routes** only when the topology is simple or when you intentionally want fixed path selection.

**Common exam traps**

- Assuming a tunnel being “up” means traffic is using it.
- Assuming both VPN tunnels are always active in the same way without considering BGP policy.
- Confusing route propagation with route association in Transit Gateway.
- Treating Direct Connect virtual interface state as the same thing as BGP best-path selection.
- Forgetting that return routing must also be valid for stateful devices and firewalls.

**Useful mental model**

Think of hybrid validation in layers:

- **Physical/logical link**: DX circuit, VPN tunnel, or router interface.
- **Session layer**: BGP neighbor establishment.
- **Routing layer**: prefix advertisement and route table installation.
- **Traffic layer**: actual packets reach the destination and return successfully.
- **Failure layer**: the preferred path disappears and the backup path takes over.

If any layer is broken, the lab is not truly validated.

**Quick checklist**

- [ ] TGW route tables contain the expected on-prem and VPC prefixes.
- [ ] VPC subnet route tables point hybrid traffic to the TGW.
- [ ] VPN tunnels show healthy telemetry and learned routes.
- [ ] Direct Connect connection and VIF are available.
- [ ] BGP neighbors are established on the customer gateway.
- [ ] The preferred path is selected by policy.
- [ ] Failover is tested by removing one path at a time.
- [ ] Traffic returns to the intended primary path after recovery.

**Summary**

Hybrid routing validation is about proving correctness, preference, and resilience. In AWS, that means checking route tables, BGP sessions, and attachment state together, then testing real traffic before and after a controlled failure. For the exam, be precise about what each service controls: TGW steers routes, VPN and Direct Connect provide transport, and BGP determines dynamic path selection.
