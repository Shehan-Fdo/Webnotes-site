## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Overview**

Asymmetric routing happens when traffic takes one path to a destination and a different path back. In AWS, that usually means the forward path is valid, but the return path is blocked, missing, or routed through a different attachment, appliance, or gateway. For SAP-C02, you need to identify where route propagation, route association, and appliance behavior create one-way connectivity.

A good troubleshooting approach is to verify the path hop by hop:

- **Source subnet route table**: confirm the destination prefix points to the expected target.
- **Intermediate gateway or appliance**: confirm it knows how to return traffic.
- **Destination subnet or VPC**: confirm security controls allow the flow.
- **Return path**: confirm the reply uses the same logical path or a compatible one.

> In AWS, many “network down” symptoms are actually return-path problems, not broken forward routing.

**Core concepts**

**Asymmetric routing** means the request and response use different paths. That is not always wrong, but it becomes a problem when stateful devices, security groups, firewalls, NAT devices, or routing domains expect symmetry.

Common AWS components involved:

- **VPC route tables**: determine next hop for traffic leaving a subnet.
- **Route propagation**: automatically inserts routes learned from VPN, Direct Connect, or Transit Gateway attachments.
- **Transit Gateway (TGW)**: central routing hub with route table associations and propagations.
- **Virtual Private Gateway (VGW)**: terminates Site-to-Site VPN and Direct Connect private virtual interfaces in classic designs.
- **NAT Gateway / NAT instance**: translates private subnet traffic to public or shared egress.
- **VPC peering**: non-transitive connectivity between VPCs.
- **Network ACLs (NACLs)** and **security groups**: can block return traffic even when routes are correct.

**How AWS route selection works**

AWS route selection is based on the most specific matching prefix. A `/24` route wins over a `/16` route, and a `/32` host route wins over both. This matters when you have overlapping propagated and static routes, centralized egress, or appliance insertion.

Important behavior to remember:

- **Longest prefix match** decides the route.
- **Static routes** and **propagated routes** can both exist in the same table only when they do not conflict on the exact same destination.
- If a route table contains a more specific route, traffic follows that route even if a broader route points elsewhere.
- In TGW designs, a route may be present in one table but not another because association and propagation are separate controls.

Practical implication:

- A route table can look “correct” at a glance, but a more specific prefix elsewhere can silently override it.
- A missing propagated route often causes return traffic to follow a default route or a different attachment, which creates asymmetric behavior.

**VPC route propagation and common failure modes**

Route propagation is how AWS automatically adds routes learned from attached networks. In practice, this is most visible with VPN, Direct Connect, and TGW.

Typical propagation sources:

- **Site-to-Site VPN** through a VGW or TGW
- **Direct Connect** private connectivity through a VGW or TGW
- **Transit Gateway attachments** between VPCs, VPNs, DX gateways, and peered TGWs

Common issues:

- **Propagation disabled**: the route never appears in the table.
- **Wrong association**: the attachment is connected to the wrong TGW route table.
- **Overlapping CIDRs**: the route exists, but traffic matches a different prefix or is rejected by design.
- **Blackhole route**: the route points to an attachment that no longer exists or is unavailable.
- **Missing return route**: the destination can receive traffic, but cannot send replies back to the source network.

Example VPC route table for centralized egress:

```text
Destination        Target
10.20.0.0/16       tgw-0abc1234
0.0.0.0/0          nat-0def5678
192.168.10.0/24    pcx-0123456789abcdef0
```

This looks fine until you discover that the destination network replies through a different attachment or a default route that bypasses the source path.

**Transit Gateway troubleshooting**

TGW is a frequent source of asymmetric routing because it separates **attachment association** from **route propagation**. An attachment can be associated with one TGW route table and propagate into another. That flexibility is powerful, but it also creates misconfiguration risk.

Key TGW concepts:

- **Association**: which TGW route table an attachment uses for traffic entering TGW.
- **Propagation**: which TGW route tables learn routes from an attachment.
- **Static routes**: manually added TGW routes.
- **Blackhole routes**: explicit drops used to prevent unwanted transit.

Common TGW misconfigurations:

- A VPC attachment is associated with the wrong TGW route table.
- A spoke VPC propagates routes into a table that the hub does not use for forwarding.
- A return route exists in the TGW table, but the destination VPC subnet route table does not point back to TGW.
- Multiple attachments advertise overlapping CIDRs, and the intended path is not the most specific match.
- A blackhole route overrides expected forwarding for a prefix.

Example TGW route table checks:

```bash
aws ec2 describe-transit-gateway-route-tables \
  --filters Name=transit-gateway-id,Values=tgw-0123456789abcdef0

aws ec2 search-transit-gateway-routes \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --filters Name=route-search.exact-match,Values=10.20.0.0/16
```

Useful attachment checks:

```bash
aws ec2 describe-transit-gateway-attachments \
  --filters Name=transit-gateway-id,Values=tgw-0123456789abcdef0

aws ec2 get-transit-gateway-route-table-associations \
  --transit-gateway-route-table-ids tgw-rtb-0123456789abcdef0

aws ec2 get-transit-gateway-route-table-propagations \
  --transit-gateway-route-table-ids tgw-rtb-0123456789abcdef0
```

What to look for:

- The attachment state should be **available**.
- The route table should contain the expected propagated prefixes.
- The association should match the intended traffic domain.
- The route table should not contain an unintended blackhole or more specific override.

**Scenario: TGW spoke-to-spoke traffic fails**

A common exam pattern is two VPCs attached to the same TGW, but traffic only works one way or not at all.

Typical causes:

- The source VPC route table sends traffic to TGW.
- The TGW route table does not have a route to the destination attachment.
- The destination VPC route table lacks the return route to TGW.
- One VPC is associated with a different TGW route table than the other.
- Overlapping CIDRs make the intended route ambiguous or invalid.

Troubleshooting sequence:

1. Confirm both VPC CIDRs do not overlap.
2. Confirm both attachments are **available**.
3. Confirm the source subnet route table points the destination CIDR to TGW.
4. Confirm the TGW route table has a route to the destination attachment.
5. Confirm the destination subnet route table points the source CIDR back to TGW.
6. Confirm security groups and NACLs allow the return ports.

**VPN route propagation and BGP**

For Site-to-Site VPN, route propagation often depends on **BGP** (Border Gateway Protocol). BGP advertises prefixes between AWS and the customer gateway. If BGP is down or the advertised prefixes are wrong, the route may disappear or never propagate.

What to verify:

- **Tunnel state**: both tunnels should be up if you expect redundancy.
- **BGP status**: the session should be established.
- **Advertised prefixes**: the on-premises side should advertise the correct CIDRs.
- **Received routes**: AWS should learn the expected prefixes.
- **Route table propagation**: the learned routes should appear in the intended VPC or TGW table.

Useful commands:

```bash
aws ec2 describe-vpn-connections \
  --vpn-connection-ids vpn-0123456789abcdef0

aws ec2 describe-vpn-connections \
  --vpn-connection-ids vpn-0123456789abcdef0 \
  --query 'VpnConnections[0].VgwTelemetry[*].{OutsideIp:OutsideIpAddress,Status:Status,LastStatusChange:LastStatusChange,AcceptedRouteCount:AcceptedRouteCount}'
```

Interpretation tips:

- **AcceptedRouteCount** helps confirm whether AWS is receiving routes.
- A tunnel can be up while route advertisement is still wrong.
- A route can be present in AWS but not in the customer router if BGP filtering blocks it.

If you use static VPN routes instead of BGP:

- You must add the prefixes manually.
- Missing static routes on either side can look like asymmetric routing.
- Static routing is simpler but less adaptive to failover.

**Direct Connect route propagation**

Direct Connect (DX) introduces similar propagation behavior, but the failure mode often differs from VPN. With DX, routes can propagate through a **virtual private gateway** or a **Transit Gateway**, depending on the design.

Key distinctions:

- **DX private virtual interface** can advertise private prefixes to AWS.
- **BGP** still governs route exchange on the private VIF.
- **VGW propagation** can inject learned routes into VPC route tables.
- **TGW propagation** can inject learned routes into TGW route tables and then onward to attached VPCs.

What to check for DX issues:

- The private VIF is up and BGP is established.
- The advertised prefixes are correct and not filtered.
- The DX gateway association is correct if used.
- The TGW or VGW route table contains the learned prefixes.
- The destination VPC subnet route table points back to the DX path.

Practical troubleshooting commands:

```bash
aws directconnect describe-virtual-interfaces \
  --virtual-interface-id dxvif-0123456789abcdef0

aws directconnect describe-virtual-gateways

aws ec2 describe-vpn-gateways \
  --vpn-gateway-ids vgw-0123456789abcdef0
```

How DX differs from VPN in exam questions:

- DX is often used for higher throughput and more predictable latency.
- VPN is often the backup path.
- If DX works outbound but replies fail, check whether the on-premises router prefers a different return path, such as VPN or internet, causing asymmetry.
- If both DX and VPN advertise the same prefix, route preference and filtering determine which path wins.

**VPC peering and inter-region peering**

VPC peering is non-transitive. That means if VPC A peers with VPC B, and VPC B peers with VPC C, A cannot reach C through B. This is a common source of “route propagation” confusion because peering does not propagate routes automatically across the network the way TGW does.

Important peering behavior:

- You must add routes in both VPC route tables.
- The peering connection must be active.
- Security groups and NACLs must allow traffic.
- Inter-region peering behaves similarly, but latency and routing domains differ.

Common failure patterns:

- One side has the peering route, the other side does not.
- The destination CIDR overlaps with another route, causing the wrong target to win.
- Return traffic exits through a different path, especially when a VPC also has NAT or TGW connectivity.
- Inter-region peering is configured, but the route tables still point to a local gateway or default route.

Example peering routes:

```text
VPC A route table:
10.2.0.0/16    pcx-0aaa1111

VPC B route table:
10.1.0.0/16    pcx-0aaa1111
```

If VPC B also has a default route to NAT, replies to 10.1.0.0/16 still use the peering route because it is more specific. If the route is missing, traffic may go out the NAT path and fail.

**NAT Gateway and NAT instance asymmetry**

NAT devices are a classic source of return-path problems. A private subnet may send traffic to a NAT Gateway in one AZ, but the return traffic or failover path may not match the original source AZ or routing expectation.

Key points:

- **NAT Gateway** is managed and highly available within an AZ, but it is not a cross-AZ routing solution by itself.
- **NAT instance** requires you to manage source/destination checks, routing, and failover.
- Private subnet route tables should point to the intended NAT target.
- Cross-AZ or centralized egress designs can create asymmetric flows if return traffic is forced through a different NAT or firewall path.

Common issues:

- A private subnet in AZ-a routes to a NAT Gateway in AZ-b, and the design does not account for failure or inspection symmetry.
- A firewall appliance expects both directions to traverse the same inspection point.
- A NAT instance has source/destination check enabled, which breaks forwarding.
- Return traffic from the internet reaches a different public IP or gateway than the one used for the outbound flow.

NAT instance check:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].SourceDestCheck'
```

If you use a NAT instance, disable source/destination check:

```bash
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --source-dest-check "{\"Value\": false}"
```

**Security groups and NACLs**

Routing may be correct while stateful or stateless filtering blocks the return path.

Security group behavior:

- **Stateful**: return traffic is automatically allowed if the initial flow is allowed.
- Usually not the cause of asymmetry unless the initial flow is blocked or the traffic enters through an unexpected interface.

NACL behavior:

- **Stateless**: you must allow both inbound and outbound traffic explicitly.
- Commonly causes one-way connectivity when ephemeral ports are not allowed.

Checklist:

- Allow the application port inbound on the destination.
- Allow ephemeral return ports outbound and inbound as needed.
- Confirm NACL rules are ordered correctly.
- Confirm the subnet association is correct.

**Reachability Analyzer and packet-path validation**

AWS Reachability Analyzer is useful for validating the path between a source and destination across VPC components. It helps you identify where the path is blocked, but it is not a substitute for understanding route tables, TGW associations, VPN status, or appliance behavior.

Use it when you need to confirm:

- Whether a route exists between two ENIs, instances, or gateways.
- Whether a security group, NACL, route table, or gateway blocks the path.
- Whether the issue is in the forward path or the return path.

Example CLI:

```bash
aws ec2 start-network-insights-analysis \
  --network-insights-path-id nip-0123456789abcdef0

aws ec2 describe-network-insights-analyses \
  --network-insights-analysis-ids nia-0123456789abcdef0
```

Use it alongside manual checks:

- Route tables
- TGW route tables
- VPN telemetry
- DX BGP status
- Security groups and NACLs

**Troubleshooting decision tree**

1. Determine whether the issue is one-way or complete failure.
2. Identify the source, destination, and intermediate network domains.
3. Check the source subnet route table for the correct next hop.
4. Check the intermediate gateway or TGW route table for the destination prefix.
5. Check the destination subnet route table for the return prefix.
6. Check VPN or DX health if the path crosses on-premises.
7. Check peering routes if the path crosses VPC peering.
8. Check NAT and firewall symmetry if centralized egress or inspection is involved.
9. Check security groups and NACLs for return traffic.
10. Validate with Reachability Analyzer or packet captures where possible.

**Exam-style scenarios**

**Scenario 1: On-premises can reach a VPC, but the VPC cannot reply**

Likely causes:

- Missing return route in the VPC subnet route table.
- Missing propagated route in TGW or VGW.
- On-premises router prefers a different return path.
- NACL blocks ephemeral return ports.

**Scenario 2: Two VPCs attached to TGW cannot communicate**

Likely causes:

- Wrong TGW route table association.
- Missing propagation from one attachment.
- Overlapping CIDRs.
- Blackhole route in TGW.
- Missing subnet route back to TGW.

**Scenario 3: Traffic works until a failover event**

Likely causes:

- VPN or DX failover changes the preferred path.
- NAT or firewall state is not preserved across AZs.
- Route propagation converges, but the return path still points to the old attachment.
- A more specific route remains active on one side only.

**Scenario 4: Inter-region peering shows routes, but traffic is one-way**

Likely causes:

- Missing reciprocal route in the peer VPC.
- Security group or NACL mismatch.
- Overlapping CIDRs.
- A local default route or NAT path is taking the return traffic elsewhere.

```bash
aws ec2 describe-transit-gateway-attachments \
```

```bash
aws ec2 describe-vpn-connections \

```

```bash

```

**Key takeaways**

- **Asymmetric routing** is usually a return-path problem.
- **Propagation** and **association** are separate controls in TGW designs.
- **Longest prefix match** can override the route you expected to use.
- **VPN and Direct Connect** depend on BGP route exchange and health.
- **VPC peering** is non-transitive and requires explicit routes on both sides.
- **NAT and firewall appliances** often create symmetry requirements that AWS routing alone does not guarantee.
- **Security groups** are stateful, but **NACLs** are stateless and commonly break return traffic.

**Quick exam checklist**

- Verify the exact source and destination CIDRs.
- Confirm the route table on both ends.
- Check TGW association and propagation separately.
- Check VPN tunnel state, BGP status, and accepted routes.
- Check DX private VIF health and advertised prefixes.
- Check peering routes in both VPCs.
- Check NAT instance source/destination check and AZ placement.
- Check NACL ephemeral port rules.
- Use Reachability Analyzer to confirm the path, then validate the control plane manually.
