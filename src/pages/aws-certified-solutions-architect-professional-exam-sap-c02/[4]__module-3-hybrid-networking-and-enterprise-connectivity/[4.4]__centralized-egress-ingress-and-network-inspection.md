## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Centralized network design goals**

Centralized egress, ingress, and inspection patterns reduce duplicated controls across accounts and VPCs. In SAP-C02 scenarios, you usually design for one or more of these goals:

- **Centralized control** of outbound internet access, inbound exposure, and traffic inspection.
- **Consistent security policy** across many accounts in an AWS Organizations landing zone.
- **Reduced operational overhead** by sharing inspection and edge services.
- **Clear segmentation** between application VPCs, shared services, inspection VPCs, and egress VPCs.

A common enterprise layout uses a **multi-account landing zone** with:
- Application accounts hosting workload VPCs.
- A shared services account for centralized network services.
- A security account for inspection tooling such as AWS Network Firewall, Gateway Load Balancer appliances, and logging.

**Core architecture patterns**

Centralized networking usually combines three distinct functions:

- **Ingress**: traffic entering from the internet or external networks.
- **Egress**: traffic leaving workloads toward the internet or AWS services.
- **Inspection**: traffic passing through security controls before reaching its destination.

These functions can be separated or combined, but exam answers should distinguish them clearly. For example, an inspection VPC is not automatically the same thing as an egress VPC. If you place NAT Gateway, firewall appliances, and internet gateways in the same design, you must preserve routing symmetry and avoid ambiguous return paths.

**Centralized egress design**

Centralized egress sends outbound traffic from spoke VPCs to a shared egress path. This is useful when you want:
- One place to enforce outbound policy.
- Central logging and monitoring.
- Shared NAT, proxy, or firewall infrastructure.
- Reduced public IP sprawl.

A typical flow is:

1. Workload subnet sends `0.0.0.0/0` to a transit gateway attachment or inspection endpoint.
2. Traffic is inspected by a firewall or appliance chain.
3. Approved traffic exits through a dedicated egress VPC with a NAT Gateway and internet gateway.
4. Return traffic follows the same path back to the source.

> **Important:** Do not assume a NAT Gateway belongs in the inspection VPC. In many designs, NAT lives in a dedicated egress VPC or egress subnet so the inspection path and internet exit path remain distinct and easier to reason about.

**Transit Gateway for centralized routing**

AWS Transit Gateway (TGW) is the common hub for centralized network designs. It connects many VPCs and on-premises networks through a single routing layer.

Key TGW concepts:
- **Attachments** connect VPCs, VPNs, and Direct Connect gateways.
- **Route tables** control which attachments can reach which destinations.
- **Propagation** can automatically add routes from attachments.
- **Association** determines which route table an attachment uses for forwarding.

For SAP-C02, remember:
- Use TGW to centralize routing between spoke VPCs and shared network services.
- Use separate TGW route tables to segment application, inspection, and on-premises traffic.
- Avoid overly broad RFC1918 summaries unless you intentionally want that reachability.

A safer spoke route is usually more specific than `10.0.0.0/8 -> tgw-xxxxxxxx`. Broad private-space routes can unintentionally create transitive access to unrelated networks.

**TGW appliance mode and symmetric routing**

When traffic must pass through stateful appliances, TGW **appliance mode** helps preserve symmetric routing. This matters because many firewalls and inspection appliances require both directions of a flow to traverse the same appliance path.

Use appliance mode when:
- You route traffic through stateful firewalls.
- You need predictable return-path behavior.
- You chain multiple inspection hops and want stable flow symmetry.

Without symmetric routing, return traffic may bypass the appliance and break sessions. This is a common exam trap in centralized inspection scenarios.

**Example TGW routing pattern**

A simplified spoke-to-inspection-to-egress flow might look like this:

```text
Spoke VPC route table:
10.10.0.0/16 -> local
0.0.0.0/0    -> tgw-attach-spoke

TGW inspection route table:
10.10.0.0/16 -> tgw-attach-spoke
0.0.0.0/0    -> tgw-attach-inspection

Inspection VPC route table:
10.10.0.0/16 -> tgw-attach-inspection
0.0.0.0/0    -> nat-gateway-id

Egress VPC route table:
0.0.0.0/0    -> igw-xxxxxxxx
```

This is conceptual, not a universal template. In real deployments, you often need separate route tables for:
- Spoke-to-inspection traffic.
- Inspection-to-egress traffic.
- Return traffic from egress back to inspection and then to spokes.

**AWS Network Firewall**

AWS Network Firewall is a managed, stateful network firewall service that fits centralized inspection designs well. It supports:
- Stateful and stateless rule groups.
- Domain filtering with TLS inspection considerations.
- Logging to Amazon S3, Amazon CloudWatch Logs, or Amazon Kinesis Data Firehose.
- Deployment across multiple Availability Zones.

Use it when you want:
- Managed firewall operations.
- Centralized policy enforcement.
- Integration with TGW or VPC routing.
- Scalable inspection without managing EC2-based appliances.

A minimal firewall policy must be structurally valid. In practice, you define:
- A firewall policy name.
- Stateless default actions.
- Stateless fragment default actions.
- Optional stateless rule group references.
- Optional stateful rule group references.

Example policy skeleton:

```json
{
  "FirewallPolicyName": "central-egress-policy",
  "FirewallPolicy": {
    "StatelessDefaultActions": [
      "aws:forward_to_sfe"
    ],
    "StatelessFragmentDefaultActions": [
      "aws:forward_to_sfe"
    ],
    "StatelessRuleGroupReferences": [
      {
        "ResourceArn": "arn:aws:network-firewall:us-east-1:123456789012:stateless-rulegroup/allow-dns"
      }
    ],
    "StatefulRuleGroupReferences": [
      {
        "ResourceArn": "arn:aws:network-firewall:us-east-1:123456789012:stateful-rulegroup/domain-blocklist"
      }
    ]
  }
}
```

A practical stateless rule group example:

```json
{
  "RuleVariables": {
    "IPSets": {
      "HOME_NET": {
        "Definition": ["10.10.0.0/16", "10.20.0.0/16"]
      }
    }
  },
  "RulesSource": {
    "StatelessRulesAndCustomActions": {
      "StatelessRules": [
        {
          "Priority": 1,
          "RuleDefinition": {
            "MatchAttributes": {
              "Protocols": [6],
              "DestinationPorts": [
                { "FromPort": 443, "ToPort": 443 }
              ],
              "Directions": ["ANY"]
            },
            "Actions": ["aws:forward_to_sfe"]
          }
        },
        {
          "Priority": 2,
          "RuleDefinition": {
            "MatchAttributes": {
              "Protocols": [17],
              "DestinationPorts": [
                { "FromPort": 53, "ToPort": 53 }
              ],
              "Directions": ["ANY"]
            },
            "Actions": ["aws:forward_to_sfe"]
          }
        }
      ]
    }
  }
}
```

This kind of rule group is useful when you want to explicitly allow common outbound protocols before stateful inspection.

**Network Firewall rule design tips**

- Use **stateless rules** for fast classification and coarse allow/deny decisions.
- Use **stateful rules** for domain-based filtering, protocol-aware inspection, and connection tracking.
- Log both accepted and dropped traffic when troubleshooting policy behavior.
- Keep rule groups modular so you can reuse them across accounts and VPCs.

Example stateful domain blocklist concept:

```text
pass tcp any any -> any 443 (msg:"Allow HTTPS"; sid:1001;)
drop tcp any any -> any 443 (msg:"Block known-bad destination"; sid:1002;)
```

For domain-based controls, remember that encrypted traffic limits visibility unless you use TLS inspection or rely on SNI/hostname-based policy features where supported.

**Firewall Manager for centralized policy enforcement**

AWS Firewall Manager helps you centrally manage security policies across multiple accounts and VPCs in AWS Organizations. It is especially relevant when you need consistent enforcement at scale.

Use Firewall Manager to:
- Deploy AWS Network Firewall policies across accounts.
- Manage AWS WAF rules for internet-facing applications.
- Apply security baselines to many VPCs without manual duplication.

This is a strong exam concept because it connects centralized inspection with organizational governance. If a scenario mentions many accounts, delegated administration, and consistent policy rollout, Firewall Manager is often the right answer.

**Gateway Load Balancer and appliance chaining**

Gateway Load Balancer (GWLB) is used when you need to insert third-party or custom appliances into traffic paths at scale. It is common for:
- IDS/IPS appliances.
- Next-generation firewalls.
- Traffic analysis tools.

GWLB works with:
- **GWLB endpoints** in VPCs.
- **Appliance fleets** behind the GWLB.
- **Geneve** encapsulation for traffic steering.

Use GWLB when you need:
- Vendor appliances instead of managed firewall policy.
- Transparent insertion of inspection devices.
- Scalable appliance fleets with load balancing.

A simplified appliance chain looks like this:

```text
Spoke VPC -> GWLB endpoint -> GWLB -> Appliance fleet -> Egress or destination
```

For routing, remember:
- You usually steer specific traffic flows or subnet routes to a GWLB endpoint.
- You do not treat GWLB as a generic internet next hop in the same way as a NAT Gateway.
- Appliance mode or equivalent routing symmetry is critical for stateful devices.

Example conceptual route table entry:

```text
10.10.0.0/16 -> local
0.0.0.0/0    -> vpce-0abc123def4567890
```

This is only a pattern illustration. In real designs, you must ensure the endpoint, appliance, and return path are all aligned.

**PrivateLink for private service access**

AWS PrivateLink lets you expose services privately without traversing the public internet. It is useful for:
- Internal APIs.
- SaaS consumption.
- Shared services across VPCs or accounts.

There are two sides:
- **Service provider** publishes a Network Load Balancer-backed endpoint service.
- **Service consumer** creates an interface endpoint to access the service.

Provider-side example:

```bash
aws ec2 create-vpc-endpoint-service-configuration \
  --network-load-balancer-arns arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/net/my-nlb/1234567890abcdef \
  --acceptance-required
```

Consumer-side example:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Interface \
  --service-name com.amazonaws.vpce.us-east-1.vpce-svc-0123456789abcdef0 \
  --subnet-ids subnet-aaa subnet-bbb \
  --security-group-ids sg-0123456789abcdef0
```

PrivateLink is not a general internet egress solution. It reduces egress by keeping traffic on the AWS backbone for supported services and private endpoints.

**VPC endpoints as egress reduction**

VPC endpoints are a major exam topic because they reduce the need for centralized internet egress for AWS service access.

Types:
- **Gateway endpoints** for Amazon S3 and DynamoDB.
- **Interface endpoints** for many AWS services and partner services.

Why they matter:
- They keep service traffic private.
- They reduce NAT Gateway data processing costs.
- They simplify security by avoiding public internet paths.

Common pattern:
- Use VPC endpoints for AWS APIs and data services.
- Reserve centralized egress for true internet destinations.

Example route table for a gateway endpoint:

```text
pl-xxxxxx (S3 prefix list) -> vpce-gateway-xxxxxxxx
0.0.0.0/0                  -> tgw-xxxxxxxx
```

This lets you send S3 traffic privately while still routing general internet traffic through centralized inspection.

**Ingress patterns: ALB, NLB, CloudFront, and API Gateway**

Ingress design depends on the protocol and application type.

Common options:
- **CloudFront** for global caching, TLS termination, and edge protection.
- **Application Load Balancer (ALB)** for HTTP/HTTPS applications and path-based routing.
- **Network Load Balancer (NLB)** for TCP/UDP/TLS pass-through and static IP needs.
- **API Gateway** for managed APIs, throttling, auth integration, and request-level controls.

Use API Gateway when:
- You expose REST or HTTP APIs.
- You want request validation, throttling, and auth integration.
- You do not need a full web application load balancer.

Use ALB when:
- You need Layer 7 routing.
- You want host/path-based routing.
- You are fronting containers, EC2, or Lambda targets.

Use CloudFront when:
- You need global edge distribution.
- You want to absorb traffic spikes and reduce latency.
- You want to pair with AWS WAF for edge filtering.

A common public ingress stack is:

```text
Internet -> CloudFront -> AWS WAF -> ALB -> application targets
```

For APIs, a common pattern is:

```text
Internet -> API Gateway -> Lambda / ECS / private integration
```

**Comparing ingress options**

| Service | Best for | Key limitation |
|---|---|---|
| CloudFront | Global content delivery and edge protection | Not a direct replacement for all app routing needs |
| ALB | HTTP/HTTPS apps with advanced routing | Not for arbitrary TCP/UDP |
| NLB | High-performance TCP/UDP/TLS | No Layer 7 routing |
| API Gateway | Managed APIs | Not a general-purpose load balancer |

**Direct Connect and Site-to-Site VPN integration**

Centralized network designs often include on-premises connectivity. TGW is the usual hub for both **AWS Direct Connect** and **Site-to-Site VPN**.

Use these when:
- You need hybrid connectivity to corporate networks.
- You want on-premises traffic to traverse centralized inspection.
- You need a single transit layer for multiple VPCs and external networks.

Common design points:
- Attach Direct Connect Gateway or VPN to TGW.
- Use TGW route tables to control which VPCs can reach on-premises networks.
- Decide whether on-premises internet-bound traffic should exit locally or through AWS centralized egress.

For SAP-C02, be ready to reason about:
- Whether on-premises traffic should hairpin through AWS.
- Whether inspection should occur before or after reaching the internet.
- How to avoid asymmetric routing between on-premises, TGW, and firewall appliances.

**Blackhole routes and segmentation**

Blackhole routes are a useful control in centralized designs. They intentionally drop traffic by pointing a route to a non-forwarding target or by removing propagation so no valid path exists.

Use blackhole routes to:
- Prevent accidental transitive access.
- Block specific CIDRs in a TGW route table.
- Enforce segmentation between environments.

Example use cases:
- Block dev VPCs from reaching prod CIDRs.
- Prevent a default route from leaking into a sensitive attachment.
- Stop a route from propagating until a security review is complete.

A blackhole route is often preferable to relying only on security groups when you want network-level segmentation.

**Security groups, NACLs, and inspection layers**

Centralized inspection does not replace native AWS controls.

Use:
- **Security groups** for instance- and ENI-level stateful filtering.
- **Network ACLs (NACLs)** for subnet-level stateless guardrails.
- **Firewall appliances** for centralized policy and advanced inspection.

A layered model is strongest:
- Security groups restrict workload exposure.
- NACLs provide coarse subnet boundaries.
- Centralized firewalls enforce organization-wide policy.

**Practical design example: shared services landing zone**

A common SAP-C02 scenario is a landing zone with:
- Multiple application accounts.
- A shared services account.
- A security account.
- Centralized inspection and egress.

Example flow:
1. Application VPC sends outbound traffic to TGW.
2. TGW forwards traffic to an inspection VPC.
3. AWS Network Firewall or GWLB appliance inspects the flow.
4. Approved traffic goes to an egress VPC with NAT Gateway and internet gateway.
5. AWS service traffic uses VPC endpoints instead of the internet where possible.

This design gives you:
- Centralized logging.
- Consistent policy.
- Lower NAT usage.
- Better control over public exposure.

**Practical CLI examples**

Create a TGW attachment:

```bash
aws ec2 create-transit-gateway-vpc-attachment \
  --transit-gateway-id tgw-0123456789abcdef0 \
  --vpc-id vpc-0123456789abcdef0 \
  --subnet-ids subnet-aaa subnet-bbb
```

Create a route in a VPC route table:

```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 0.0.0.0/0 \
  --transit-gateway-id tgw-0123456789abcdef0
```

Create a gateway VPC endpoint for S3:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.s3 \
  --route-table-ids rtb-0123456789abcdef0 \
  --vpc-endpoint-type Gateway
```

Inspect TGW route tables:

```bash
aws ec2 search-transit-gateway-routes \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --filters Name=state,Values=active
```

**Troubleshooting and exam traps**

- **Asymmetric routing** breaks stateful inspection and is one of the most common design failures.
- **Overly broad routes** can create unintended transitive access across accounts or environments.
- **NAT Gateway placement** matters; do not conflate inspection with internet exit.
- **GWLB endpoint routing** is not the same as ordinary internet routing.
- **PrivateLink** is for private service access, not general outbound internet access.
- **API Gateway** is a public ingress option for APIs and should be considered alongside ALB and CloudFront.
- **Firewall Manager** is the scaling answer when the question emphasizes many accounts and centralized policy.
- **VPC endpoints** often reduce the need for centralized egress and lower cost.

**How to choose the right service**

- Choose **TGW** when you need hub-and-spoke routing across many VPCs and hybrid links.
- Choose **AWS Network Firewall** when you want managed centralized inspection.
- Choose **GWLB** when you need third-party or custom appliances.
- Choose **PrivateLink** when you want private service exposure without internet traversal.
- Choose **VPC endpoints** when AWS service access should stay private.
- Choose **CloudFront, ALB, NLB, or API Gateway** based on ingress protocol and application behavior.
- Choose **Firewall Manager** when you need organization-wide policy deployment.

**Key takeaways**

Centralized egress, ingress, and inspection are about separating concerns:
- Route traffic through the right control points.
- Preserve symmetry for stateful devices.
- Reduce public exposure with endpoints and private connectivity.
- Scale policy with TGW, Network Firewall, GWLB, and Firewall Manager.

For SAP-C02, the best answer is usually the one that balances security, operability, and routing correctness rather than the one that simply “puts everything behind a firewall.”
