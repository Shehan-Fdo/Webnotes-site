## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Connectivity resilience in AWS**

Resilient connectivity is about designing for loss of a path, device, Region, account, or DNS target without losing the application. In SAP-C02 scenarios, you are usually balancing three goals:

- **Availability**: keep traffic flowing during failures.
- **Determinism**: make failover behavior predictable.
- **Operability**: detect, diagnose, and correct issues quickly.

A strong design separates concerns:

- **Routing** decides where packets go.
- **DNS** decides which endpoint clients resolve.
- **Health checks** decide when to stop sending traffic to a target.
- **Observability** confirms what actually happened.

For professional-level questions, you should think beyond “is the link up?” and ask:

- Is the failure in the underlay, overlay, DNS layer, or application layer?
- Is failover automatic, manual, or policy-driven?
- Does the design support transitive routing, centralized inspection, and multi-account governance?
- Can the secondary path actually carry production load?

**Core resilience patterns**

The most common AWS connectivity patterns differ in how they fail over and what they can route.

- **Site-to-Site VPN** provides encrypted IPsec tunnels over the internet and is often the simplest backup path for Direct Connect.
- **AWS Direct Connect** provides private connectivity with predictable latency and bandwidth, but you still need a backup path because physical or provider failures happen.
- **Transit Gateway (TGW)** centralizes routing across VPCs, VPNs, and Direct Connect Gateway associations.
- **AWS Cloud WAN** provides a managed global network fabric for multi-Region and multi-account connectivity, often reducing the complexity of large TGW meshes.
- **VPC peering** is simple and low-latency, but it does not support transitive routing, which limits failover and hub-and-spoke designs.
- **PrivateLink / VPC endpoints** expose services privately without full network adjacency, which is useful for reducing blast radius and avoiding broad routing dependencies.

**Direct Connect failover design**

Direct Connect is usually part of a dual-path design, not a single-path design. You typically pair it with VPN or another DX location so that a loss of one circuit, device, or provider does not isolate workloads.

Key concepts:

- **BGP** (Border Gateway Protocol) is the routing protocol used to exchange routes between your network and AWS.
- **BGP timers** affect how quickly failures are detected.
- **Route preference** is controlled by attributes such as local preference on your side and route priority on AWS-adjacent routing domains.
- **DX Gateway** is used to connect Direct Connect to multiple VPCs across Regions through a Transit Gateway or virtual private gateway, depending on the architecture.

Practical implications:

- If you use a **virtual private gateway (VGW)**, the DX private virtual interface can advertise routes directly to a VPC attachment model.
- If you use a **Direct Connect Gateway** with TGW, the DX path can serve multiple VPCs and Regions through centralized routing.
- If the DX path fails, your backup VPN should already have routes and security controls in place; do not assume failover is automatic unless you have validated route propagation and preference.

A common hybrid pattern is:

- Primary: Direct Connect private VIF
- Secondary: Site-to-Site VPN over the internet
- Routing: BGP with tuned preferences so DX is preferred and VPN is standby

Example BGP configuration conceptually:

```text
neighbor 169.254.100.1 remote-as 7224
neighbor 169.254.100.1 description AWS-DX-PRIMARY
neighbor 169.254.100.1 timers 10 30

route-map PREFER-AWS permit 10
 set local-preference 200
```

Use shorter timers carefully; they improve detection but can increase sensitivity to transient loss. In exam scenarios, the right answer is often “validate failover behavior end to end,” not “assume BGP alone solves it.”

**Transit Gateway, multi-account routing, and centralized inspection**

TGW is the standard hub for many enterprise AWS networks because it supports transitive routing across VPCs, VPNs, and DX Gateway associations. It is especially useful in multi-account environments where you want centralized control.

Typical uses:

- **Shared services** VPCs for DNS, inspection, or egress
- **Spoke VPCs** for application workloads
- **Centralized egress** through NAT gateways or firewall appliances
- **Centralized inspection** with AWS Network Firewall or third-party appliances

Important design points:

- TGW route tables are policy boundaries, not just forwarding tables.
- You often need separate route tables for **spokes**, **inspection**, and **egress**.
- Return paths matter as much as forward paths; asymmetric routing can break stateful firewalls and some application flows.
- In centralized inspection, you must explicitly route traffic from spokes to the inspection VPC and then back to the correct destination CIDRs.

A simplified inspection flow looks like this:

```text
Spoke VPC -> TGW -> Inspection VPC (firewall) -> TGW -> Egress VPC / destination
```

For return traffic, ensure the reverse path is equally explicit:

```text
Destination -> TGW -> Inspection VPC -> TGW -> Spoke VPC
```

> A default route alone is not enough in many inspection designs. You need route symmetry and correct propagation for both directions.

Example TGW route table checks:

```bash
aws ec2 describe-transit-gateway-route-tables \
  --filters Name=transit-gateway-id,Values=tgw-0123456789abcdef0

aws ec2 search-transit-gateway-routes \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --filters Name=state,Values=active
```

Use the route table ID explicitly. In troubleshooting, confirm:

- The attachment exists and is **available**
- The route is **propagated** or **associated** as intended
- The destination CIDR is present in the correct TGW route table
- Security groups and NACLs do not block the flow after routing succeeds

**AWS Cloud WAN for modern multi-Region resilience**

Cloud WAN is a strong option when you need a managed global network across multiple Regions and accounts. It can reduce the operational burden of building and maintaining many TGW peering relationships.

Use Cloud WAN when you want:

- A centralized policy model for global connectivity
- Simplified multi-Region segmentation
- Easier expansion across accounts and Regions
- A managed backbone for branch, on-premises, and VPC connectivity

Compare it with TGW:

| Capability | Transit Gateway | Cloud WAN |
|---|---|---|
| Scope | Regional hub | Global network |
| Best for | Hub-and-spoke and regional centralization | Multi-Region, multi-account global connectivity |
| Policy model | Route tables and attachments | Core network policy and segments |
| Operational complexity | Moderate to high at scale | Lower for global topologies |

In SAP-C02 questions, Cloud WAN is often the better answer when the problem is not just “connect these VPCs,” but “build a resilient global network with consistent policy and failover behavior.”

**VPC peering limitations**

VPC peering is useful for direct VPC-to-VPC communication, but it has important limits that affect resilience.

- It is **non-transitive**: VPC A cannot reach VPC C through VPC B.
- It does not provide a central routing hub.
- It does not scale well for many VPCs or complex failover paths.
- It is not a good fit for centralized inspection or shared egress.

Troubleshooting peering connectivity:

- Confirm the peering connection is **active**
- Add routes in both route tables for the peer CIDRs
- Verify security groups and NACLs
- Check DNS resolution if the application uses private hostnames
- Remember that overlapping CIDRs are not allowed

If you need transitive routing, centralized inspection, or multi-account hub-and-spoke failover, TGW or Cloud WAN is usually the better design.

**PrivateLink and VPC endpoint troubleshooting**

PrivateLink is often used to expose services privately without opening broad network paths. It is common for internal APIs, SaaS integrations, and shared services.

Types to remember:

- **Interface endpoints** for AWS services and PrivateLink-enabled services
- **Gateway endpoints** for S3 and DynamoDB

Troubleshooting checklist:

- Verify the endpoint is in the correct **subnets** and **security groups**
- Confirm the endpoint service accepts connections from the consumer account
- Check DNS resolution for the service name
- Ensure the application uses the private DNS name if required
- Validate that the endpoint policy allows the action

Common failure modes:

- DNS resolves to the public endpoint instead of the private one
- Security group rules block the endpoint ENI
- The service provider has not allowed the consumer principal
- Route tables are correct, but the application is using the wrong hostname

Example interface endpoint creation:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Interface \
  --service-name com.amazonaws.us-east-1.execute-api \
  --subnet-ids subnet-aaa subnet-bbb \
  --security-group-ids sg-0123456789abcdef0
```

**Route 53 failover and health check design**

Route 53 is often the control plane for application failover, but its health checks are only as good as the endpoint you expose. A basic health check can tell you whether a URL returns a response, but it does not automatically prove the entire application stack is healthy.

Important limitations:

- A health check may confirm only the specific endpoint it probes.
- It may not detect database lag, downstream dependency failure, or partial regional degradation.
- It cannot replace application-aware readiness logic.
- It does not validate packet-level network behavior.

Better practice:

- Expose a custom readiness endpoint such as `/healthz` or `/ready`
- Make that endpoint reflect the dependencies that matter for failover
- Use **calculated health checks** when you need to combine multiple signals
- Use alias failover records for AWS targets when appropriate

A practical pattern is:

- Health check 1: application process alive
- Health check 2: database reachable
- Health check 3: critical dependency reachable
- Calculated health check: overall readiness based on the above

Example Route 53 failover record concept:

```text
example.com. 60 IN A <primary-alias-target>
example.com. 60 IN A <secondary-alias-target>
```

In Route 53, alias records are configured in the hosted zone, not as ordinary zone-file A records pointing to hostnames. For AWS resources, alias targets can point to load balancers, CloudFront distributions, API Gateway, and some other AWS endpoints.

Operational guidance:

- Use **primary/secondary failover** when one endpoint should serve traffic and the other should stay idle until failure.
- Use **latency-based routing** when you want the closest healthy Region.
- Use **weighted routing** for gradual shifts, canaries, or active-active patterns.
- Use **geolocation** or **geoproximity** when business or proximity rules matter, but remember geoproximity is configured through Route 53 Traffic Flow policy constructs.

For non-AWS endpoints:

- Health checks can monitor public endpoints if reachable from Route 53 health checkers.
- Failover still depends on DNS TTLs, client caching, and how quickly the target becomes unhealthy.
- If the endpoint is behind another DNS name, validate the full resolution chain.

**Advanced Route 53 patterns**

Use advanced patterns when simple failover is too shallow.

- **Calculated health checks** combine multiple health checks into a single decision.
- **Alias failover** is useful for AWS-managed targets where you want DNS to switch between Regions or load balancers.
- **Multi-value answer** can return multiple healthy records, but it is not a substitute for true failover logic.
- **Traffic Flow** helps you manage complex routing policies, including geoproximity and layered decisions.

When to use each:

- Use calculated health checks when one endpoint should be considered healthy only if several dependencies are healthy.
- Use alias failover when the target is an AWS resource and you want clean integration with Route 53.
- Use weighted routing when you need controlled traffic shifting before a full cutover.

Example of a readiness endpoint:

```python
from flask import Flask, jsonify
import psycopg2

app = Flask(__name__)

@app.get("/ready")
def ready():
    try:
        # Replace with real dependency checks
        return jsonify(status="ok"), 200
    except Exception:
        return jsonify(status="degraded"), 503
```

The key is not the framework; it is that the endpoint reflects the state you want Route 53 to act on.

**Troubleshooting workflow**

A disciplined workflow prevents you from chasing the wrong layer.

1. **Confirm the symptom**
   - Is the issue total outage, partial outage, high latency, or intermittent failure?
   - Is it one client, one Region, one account, or all traffic?

2. **Identify the path**
   - Internet, VPN, Direct Connect, TGW, peering, PrivateLink, or DNS?
   - Forward path and return path may differ.

3. **Check DNS**
   - Resolve the name from the client location.
   - Confirm TTL behavior and cached answers.
   - Verify Route 53 records, health checks, and alias targets.

4. **Check routing**
   - Inspect route tables in the VPC, TGW, and on-premises side.
   - Validate propagation and associations.
   - Look for asymmetric routes.

5. **Check security controls**
   - Security groups
   - Network ACLs
   - Firewall policies
   - Endpoint policies

6. **Check application readiness**
   - Load balancer target health
   - Instance or container health
   - Dependency health
   - Database failover state

7. **Validate with logs and metrics**
   - VPC Flow Logs
   - Route 53 Resolver query logs
   - CloudWatch metrics and alarms
   - Network Firewall logs
   - TGW and Cloud WAN telemetry

**Using VPC Flow Logs correctly**

VPC Flow Logs are useful, but they are not packet capture. They help you see whether traffic was accepted or rejected at the logging point.

Use them to answer questions like:

- Did the ENI see the traffic?
- Was the traffic allowed or rejected?
- Which source and destination addresses were involved?
- Was the traffic on the expected port and protocol?

Do not over-interpret them:

- ACCEPT does not prove the application succeeded.
- REJECT does not always identify the exact upstream cause.
- They do not show full TCP handshake details.
- They do not replace packet capture for asymmetric routing or MTU issues.

Example query workflow with CloudWatch Logs Insights:

```sql
fields @timestamp, srcAddr, dstAddr, srcPort, dstPort, action, protocol
| filter action = "REJECT"
| sort @timestamp desc
| limit 50
```

Use this to narrow the problem, then correlate with route tables, security groups, and application logs.

**Route 53 Resolver logs and DNS troubleshooting**

Route 53 Resolver query logs help you see DNS behavior inside a VPC.

Use them to troubleshoot:

- Wrong private hosted zone resolution
- Unexpected forwarding to on-premises DNS
- Split-horizon DNS issues
- Conditional forwarder mistakes

Common checks:

- Is the query hitting the expected resolver rule?
- Is the private hosted zone associated with the right VPC?
- Are overlapping names causing ambiguous answers?
- Is the client using the VPC resolver or a custom DNS server?

Example association check:

```bash
aws route53resolver list-resolver-rule-associations
aws route53resolver list-resolver-endpoint-ip-addresses --resolver-endpoint-id rslvr-out-0123456789abcdef0
```

**Network Manager and AWS Config for resilience**

AWS Network Manager helps you visualize and manage global network topology, especially when you use TGW, VPN, and Direct Connect at scale.

Use it to:

- View site-to-site connectivity
- Track network health
- Understand global topology
- Correlate failures across Regions and attachments

AWS Config helps you detect drift and misconfiguration.

Useful examples:

- Detect a route table that lost a critical route
- Detect a security group that no longer allows health check traffic
- Detect a TGW attachment or endpoint that changed state
- Audit whether required logging is enabled

In exam scenarios, Config is often the right answer when the question is about continuous compliance or detecting configuration drift rather than immediate packet troubleshooting.

**End-to-end scenario: Direct Connect outage with VPN failover**

A common SAP-C02 scenario is a hybrid application that loses its primary private link.

1. **Observe the symptom**
   - On-premises users cannot reach private AWS services.
   - Internet-based access may still work.

2. **Check the primary path**
   - Verify DX circuit status, BGP session state, and advertised routes.
   - Confirm whether the DX virtual interface is up.
   - Check whether the DX Gateway or VGW still has the expected prefixes.

3. **Check the backup path**
   - Confirm VPN tunnels are up.
   - Verify BGP routes are learned on both sides.
   - Ensure route preference favors DX when healthy and VPN when DX fails.

4. **Validate AWS routing**
   - Check TGW route tables or VGW propagation.
   - Confirm the spoke VPCs still have routes to the shared services VPC.
   - Verify return routes from shared services back to on-premises.

5. **Check DNS and application dependencies**
   - If the app uses private DNS names, confirm resolution still points to reachable endpoints.
   - Validate load balancer target health and backend readiness.

6. **Confirm recovery**
   - Test from an on-premises host.
   - Verify latency and throughput on the backup path.
   - Document whether failover was automatic or required manual route changes.

A good answer in an exam is not just “VPN took over.” It is “the design had validated route preference, symmetric return paths, and tested failover so the backup path could carry production traffic.”

**End-to-end scenario: Multi-Region application failover**

For a multi-Region application, you often combine DNS failover with regional data replication.

Typical pattern:

- Regional ALB or NLB in each Region
- Route 53 failover or latency-based routing
- Health checks against a readiness endpoint
- Data layer replication or promotion strategy
- Optional global accelerator or Cloud WAN for network resilience

Things to verify:

- The secondary Region is actually warm and ready
- The health check reflects application readiness, not just process liveness
- The database promotion process is automated or well-documented
- Client DNS caching will not delay failover beyond acceptable RTO

If the application is stateful, DNS failover alone is not enough. You also need:

- Data replication
- Session strategy
- Idempotent retries
- Clear promotion and rollback steps

**Common exam traps**

- Choosing VPC peering for a hub-and-spoke design that needs transitive routing.
- Assuming Flow Logs prove end-to-end connectivity.
- Using a shallow health check that misses a database outage.
- Forgetting return routes in centralized inspection.
- Treating Cloud WAN and TGW as interchangeable without considering global policy and scale.
- Assuming Direct Connect failover is automatic without BGP and route-preference validation.
- Using PrivateLink when the requirement is broad network connectivity rather than service exposure.
- Expecting Route 53 to fail over instantly without considering TTLs and client caching.

Use these commands to confirm the control plane before you assume the data plane is broken.

**Design guidance for SAP-C02**

When you see a resilience question, choose the answer that:

- Minimizes single points of failure
- Uses the right abstraction for the scale
- Preserves route symmetry where needed
- Validates health at the correct layer
- Supports operational visibility and controlled failover

A strong architecture often combines:

- **TGW or Cloud WAN** for network backbone
- **Direct Connect plus VPN** for hybrid resilience
- **Route 53 health checks and failover** for application traffic steering
- **PrivateLink** for private service exposure
- **Network Manager, Resolver logs, Flow Logs, and Config** for troubleshooting and governance

The best answer is rarely one service alone. It is usually a layered design where routing, DNS, health, and observability all reinforce each other.
