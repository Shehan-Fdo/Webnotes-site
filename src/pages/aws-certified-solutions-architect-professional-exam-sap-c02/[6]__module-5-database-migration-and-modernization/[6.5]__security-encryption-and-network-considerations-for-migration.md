## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Migration security goals**

A secure migration protects data in transit, data at rest, identities, and network paths while minimizing downtime and operational risk. In SAP-C02 scenarios, you often need to choose controls that work during coexistence, cutover, and rollback, not just after the migration is complete.

The main design questions are:

- How do you move data without exposing credentials or sensitive payloads?
- How do you preserve least privilege while temporary migration tooling needs broad access?
- How do you keep private traffic private across accounts, VPCs, Regions, or hybrid links?
- How do you prove compliance and detect misconfiguration during the migration window?

**Identity and access for migration tooling**

Migration often requires temporary elevated access, but that access should be tightly scoped and time-bound. Prefer cross-account role assumption over long-lived IAM user credentials, and use separate roles for orchestration, data copy, and validation.

A common pattern is:

- A source account role grants read-only access to export or snapshot data.
- A destination account role grants write access to import or restore data.
- A central migration account assumes both roles using AWS Security Token Service (STS).

Example trust policy for a destination role that allows a migration account to assume it:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowMigrationAccountAssumeRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/MigrationOrchestrator"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "migration-2026"
        }
      }
    }
  ]
}
```

Practical IAM guidance:

- Use **least privilege** for each migration phase.
- Set session duration only as long as the job needs.
- Use `sts:ExternalId` when a third party or shared automation assumes the role.
- Separate permissions for discovery, copy, restore, and validation.
- Prefer role chaining only when necessary; it can shorten effective session duration.

Example CLI assumption flow:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/DestinationMigrationRole \
  --role-session-name cutover-copy \
  --duration-seconds 3600
```

> If a migration tool needs to create grants on a KMS key, `kms:CreateGrant` may be required. Do not add it by default; grant it only when the workflow actually creates or manages grants.

**Encryption at rest with AWS KMS**

AWS Key Management Service (KMS) protects encryption keys used by many AWS services. For migration, the important idea is that KMS often protects the data key in an envelope encryption model rather than encrypting application data directly.

Use KMS for:

- EBS volumes and snapshots
- RDS databases and snapshots
- S3 objects when the service uses server-side encryption with KMS
- EFS file systems
- AMIs and launch templates that depend on encrypted snapshots
- Application-layer envelope encryption when your app generates a data key and uses KMS to protect that data key

Common migration concerns:

- You must ensure the destination account or Region can use the required KMS key.
- Cross-account snapshot copy requires both snapshot permissions and KMS permissions when encrypted.
- Some services require re-encryption during copy rather than direct reuse of the source key.
- Key policies and grants must allow the destination principal to decrypt, encrypt, or create grants as needed.

Example KMS key policy fragment for a destination role:

```json
{
  "Sid": "AllowDestinationRoleUseOfKey",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::444455556666:role/DestinationMigrationRole"
  },
  "Action": [
    "kms:Decrypt",
    "kms:Encrypt",
    "kms:DescribeKey"
  ],
  "Resource": "*"
}
```

Use `kms:CreateGrant` only when the principal must create grants for AWS services or automation that relies on grants. Many copy and restore workflows do not need it.

**Encrypted backups, snapshots, and AMIs**

Migration plans often fail when teams forget that backups are encrypted too. Treat snapshots and images as protected data, not just the live workload.

Key points:

- **EBS snapshots** inherit encryption state from the source volume.
- **Encrypted AMIs** depend on encrypted snapshots and the KMS key used to create them.
- **RDS snapshots** can be copied across accounts or Regions, but encrypted copies require KMS access in the destination.
- **S3 object ownership** matters when copying between accounts; the destination account should own the objects after migration to avoid access surprises.

Example encrypted EBS snapshot copy:

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --encrypted \
  --kms-key-id arn:aws:kms:us-west-2:444455556666:key/abcd-1234-efgh-5678
```

Example encrypted AMI copy:

```bash
aws ec2 copy-image \
  --source-region us-east-1 \
  --source-image-id ami-0123456789abcdef0 \
  --name migrated-app-ami \
  --encrypted \
  --kms-key-id arn:aws:kms:us-west-2:444455556666:key/abcd-1234-efgh-5678
```

For RDS, ensure the destination account can access the KMS key used for the copied snapshot, then restore from the copied snapshot in the target environment.

**Secrets Manager and Parameter Store migration**

Secrets are often overlooked because they are not part of the application code, but they are critical migration artifacts. You should plan how to move, re-create, or rotate them securely.

Use these patterns:

- **AWS Secrets Manager** for secrets that need rotation, versioning, and managed retrieval.
- **AWS Systems Manager Parameter Store** for configuration values, feature flags, and less sensitive parameters; use SecureString for sensitive values.

Migration guidance:

- Re-create secrets in the destination account rather than exporting plaintext when possible.
- If you must transfer secret values, use a secure, temporary channel and delete intermediate artifacts immediately.
- Update application references to point to the new secret ARN or parameter path.
- Rotate secrets after cutover to invalidate any credentials that may have been exposed during migration.
- Re-test rotation Lambdas, database users, and application permissions after the move.

Example Secrets Manager retrieval during migration validation:

```bash
aws secretsmanager get-secret-value \
  --secret-id arn:aws:secretsmanager:us-west-2:444455556666:secret:prod/db-credentials
```

Example Parameter Store copy approach:

```bash
aws ssm get-parameter \
  --name /prod/app/api-key \
  --with-decryption
```

Then re-create in the destination account:

```bash
aws ssm put-parameter \
  --name /prod/app/api-key \
  --type SecureString \
  --value 'REDACTED' \
  --overwrite
```

Practical considerations:

- Do not hardcode secrets into migration scripts.
- Use short-lived credentials for secret retrieval.
- Rotate database passwords, API keys, and service tokens after cutover.
- Verify that the destination KMS key policy allows the application role to decrypt the secret.

**TLS, certificates, and mTLS**

Transport Layer Security (TLS) protects data in transit. During migration, you may need to preserve trust chains, import certificates, or update endpoints without breaking clients.

Important operational points:

- AWS Certificate Manager (ACM) can import certificates, but imported certificates are not automatically renewed by ACM.
- If you import a certificate, you must manage renewal and re-import before expiration.
- Public ACM certificates are renewed automatically only when ACM manages the certificate and validation remains intact.
- If the application uses mutual TLS (mTLS), you must migrate both server certificates and client trust material.

Migration checklist for TLS:

- Inventory all certificates, private keys, and trust stores.
- Confirm whether the certificate is public, private, imported, or issued by ACM Private CA.
- Update DNS, load balancers, and application endpoints in a controlled sequence.
- Validate certificate chain, SANs (Subject Alternative Names), and hostname matching.
- Reissue or re-import certificates if the destination environment changes the endpoint name.

Example OpenSSL inspection:

```bash
openssl s_client -connect app.example.com:443 -servername app.example.com -showcerts
```

Example certificate import command:

```bash
aws acm import-certificate \
  --certificate fileb://cert.pem \
  --private-key fileb://private-key.pem \
  --certificate-chain fileb://chain.pem
```

For mTLS:

- Ensure the destination load balancer or service supports client certificate validation.
- Migrate the client CA bundle and any revocation controls.
- Test handshake failures intentionally to confirm enforcement.

> Reusing the same private key across environments is usually discouraged. Treat it as an exceptional compatibility workaround only when policy allows and the operational risk is understood.

**Network segmentation and private connectivity**

Migration traffic should follow the same segmentation principles as production traffic. Do not rely on a single firewall rule set alone; combine security groups, network ACLs (NACLs), route tables, and private connectivity options.

Security group design patterns:

- Use **application-tier** security groups that allow inbound only from the load balancer or upstream tier.
- Use **database-tier** security groups that allow inbound only from the application tier.
- Use **migration-only** security groups for temporary replication or transfer hosts.
- Prefer referencing security groups over broad CIDR ranges when traffic stays inside a VPC.

NACL use cases:

- Add coarse subnet-level guardrails for high-risk migration windows.
- Block known-bad ranges or restrict ephemeral ports when required by policy.
- Use NACLs carefully because they are stateless and can break return traffic if misconfigured.

Route table segmentation:

- Separate migration subnets from production subnets when you need controlled data paths.
- Use dedicated route tables for staging, replication, and cutover subnets.
- Remove temporary routes after migration to reduce attack surface.

Private connectivity options:

- **AWS PrivateLink** exposes services privately through interface endpoints without traversing the public internet.
- Use PrivateLink for cross-account service access, partner access, or hybrid scenarios where you want private, service-level connectivity.
- Use VPC endpoints for AWS services such as S3, KMS, Secrets Manager, and Systems Manager to keep control traffic off the internet.

Example PrivateLink service consumer setup:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Interface \
  --service-name com.amazonaws.vpce.us-west-2.vpce-svc-0abc123def4567890 \
  --subnet-ids subnet-aaa subnet-bbb \
  --security-group-ids sg-0123456789abcdef0
```

Example S3 gateway endpoint for private data transfer:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-west-2.s3 \
  --route-table-ids rtb-0123456789abcdef0
```

**Hybrid and cross-account migration paths**

When migrating from on-premises or between accounts, you often need a secure bridge that exists only for the duration of the project.

Common options:

- Site-to-Site VPN for encrypted IPsec connectivity
- AWS Direct Connect for predictable throughput and lower latency
- Transit Gateway for hub-and-spoke routing across multiple VPCs and accounts
- PrivateLink for service-level access without full network adjacency

Design considerations:

- Use separate routing domains for migration traffic when possible.
- Avoid advertising unnecessary on-premises prefixes into the migration VPC.
- Validate asymmetric routing, especially when multiple paths exist.
- Confirm that DNS resolution works across the chosen connectivity model.

**Logging, monitoring, and audit controls**

Migration security is not complete without visibility. You need to detect unauthorized access, unexpected data movement, and misconfigured network controls.

Core services to enable:

- **AWS CloudTrail** for API activity and change tracking
- **VPC Flow Logs** for network traffic visibility at the ENI, subnet, or VPC level
- **AWS Config** for configuration drift and compliance checks
- **Amazon GuardDuty** for threat detection, suspicious API calls, and anomalous network behavior
- **Amazon CloudWatch** for metrics, alarms, and log-based alerting

What to watch during migration:

- Unexpected `AuthorizeSecurityGroupIngress` or `PutBucketPolicy` changes
- KMS access denied errors that indicate missing key policy or grant permissions
- Large data transfers to unknown destinations
- Public exposure of resources that should remain private
- Failed authentication attempts against migration hosts or bastions

Example CloudTrail lookup:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=CopySnapshot
```

Example VPC Flow Logs use case:

- Confirm that replication traffic stays within approved CIDRs.
- Detect unexpected outbound connections from migration instances.
- Validate that PrivateLink traffic uses the intended endpoint ENIs.

Example AWS Config rules to consider:

- Security groups should not allow unrestricted SSH or RDP.
- S3 buckets should not be public.
- EBS volumes should be encrypted.
- RDS instances should be encrypted.
- IAM policies should not grant overly broad access.

**Practical migration workflow**

A secure migration usually follows a repeatable sequence rather than an ad hoc copy operation.

1. Inventory data, secrets, certificates, and dependencies.
2. Create destination IAM roles with least privilege and time-bound access.
3. Prepare KMS keys, key policies, and grants for encrypted resources.
4. Establish private connectivity using VPN, Direct Connect, Transit Gateway, or PrivateLink.
5. Copy or re-create encrypted data, snapshots, AMIs, and secrets.
6. Validate TLS, DNS, and application trust chains in the destination.
7. Enable logging and monitoring before cutover.
8. Rotate secrets and revoke temporary access after cutover.
9. Remove temporary routes, endpoints, and migration roles.

**Scenario patterns for SAP-C02**

Be ready to choose the control that matches the constraint:

- If the requirement is private cross-account service access, choose **PrivateLink**.
- If the requirement is encrypted object or snapshot migration, ensure **KMS** permissions and re-encryption are handled.
- If the requirement is temporary access for automation, use **STS role assumption** with least privilege.
- If the requirement is certificate continuity, plan for **ACM import**, renewal ownership, and trust chain validation.
- If the requirement is auditability, enable **CloudTrail**, **Config**, **Flow Logs**, and **GuardDuty**.

**Common pitfalls**

- Assuming encrypted snapshots can be copied without KMS permissions.
- Forgetting to rotate secrets after migration.
- Leaving temporary security group rules open after cutover.
- Using public endpoints when a VPC endpoint or PrivateLink would keep traffic private.
- Treating imported ACM certificates as automatically renewable.
- Overusing `kms:CreateGrant` when `Decrypt`, `Encrypt`, and `DescribeKey` are sufficient.
- Neglecting object ownership and ACL behavior when copying S3 data across accounts.

**Quick reference**

| Area | What to verify | Why it matters |
|---|---|---|
| IAM | Role trust, session duration, least privilege | Prevents overexposure during temporary access |
| KMS | Key policy, grants, re-encryption path | Required for encrypted copies and restores |
| TLS | Chain, SANs, renewal ownership, mTLS | Prevents cutover failures and trust breaks |
| Network | SGs, NACLs, routes, PrivateLink | Keeps migration traffic private and segmented |
| Secrets | Re-create, rotate, update references | Avoids credential leakage and stale access |
| Logging | CloudTrail, Flow Logs, Config, GuardDuty | Detects misconfigurations and suspicious activity |

**Exam focus**

For SAP-C02, the best answer usually balances security with operational feasibility. Prefer private connectivity, temporary scoped access, and service-native encryption controls. When a scenario mentions encrypted backups, cross-account copies, or private service access, look for the missing dependency: KMS permissions, endpoint configuration, or trust policy alignment.
