## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Lab overview**

A cutover is the controlled switch from a source environment to a target environment after replication, synchronization, or migration is complete. In AWS migration scenarios, your goal is to minimize downtime, preserve data integrity, and verify that the target system is ready to serve production traffic.

This lab topic focuses on two outcomes:

- Performing the final cutover safely.
- Validating that data remains consistent after the switch.

In practice, you usually combine replication health checks, application-level validation, database comparison, and traffic verification. The exact tools vary by workload, but the principles stay the same: stop writes, flush in-flight changes, switch dependencies, and confirm that the target reflects the expected state.

**Cutover strategy**

A good cutover plan reduces uncertainty. You should define the sequence before the maintenance window starts, including who approves each step, what metrics confirm success, and what conditions trigger rollback.

Common cutover patterns include:

- **Big-bang cutover**: You switch all traffic at once after a final sync.
- **Phased cutover**: You move users, services, or regions gradually.
- **Blue/green deployment**: You keep source and target environments side by side and shift traffic when validation passes.
- **Database replication cutover**: You promote the target database after replication lag reaches zero or an acceptable threshold.

For SAP-C02, you should understand why phased approaches are often preferred for critical workloads: they reduce blast radius and make validation easier.

**Pre-cutover checks**

Before you switch traffic, confirm that the target environment is ready and that the source is stable enough to freeze writes.

Typical checks include:

- Replication status is healthy and lag is within tolerance.
- Target capacity matches expected load.
- DNS, load balancers, security groups, and routing rules are prepared.
- Application configuration points to the target endpoints.
- Monitoring, logging, and alerting are enabled on both sides.
- Backups or snapshots exist for rollback.

If the workload uses a database migration service or storage replication, verify that the final sync has completed and that no pending changes remain.

> If you cut over while replication is still lagging, you risk losing the last committed transactions or creating divergent data states.

**Final synchronization**

The final sync is the last opportunity to copy changes from source to target before you stop writes. The exact method depends on the service:

- For databases, you may stop application writes, wait for replication to catch up, and then promote the replica.
- For file systems, you may run a final incremental sync.
- For object storage, you may copy the remaining delta objects and verify object counts or checksums.
- For application state, you may drain sessions and persist transient data.

A practical sequence often looks like this:

1. Put the application into maintenance mode or read-only mode.
2. Stop background jobs, batch processors, and scheduled writers.
3. Wait for replication lag to reach zero.
4. Run the final sync or delta transfer.
5. Verify that source and target counts match.
6. Switch traffic to the target.

**Example: database cutover with replication validation**

```bash
# Check replication lag on a PostgreSQL replica
psql -h target-db.example.com -U admin -d appdb -c "SELECT now() - pg_last_xact_replay_timestamp() AS replication_delay;"

# Confirm the target is ready to accept writes
psql -h target-db.example.com -U admin -d appdb -c "SELECT pg_is_in_recovery();"
```

In a managed AWS database migration, you may also inspect service-specific metrics such as replication lag, transaction logs applied, or task status. The exact command depends on the engine and migration tool, but the validation goal is the same: confirm that the target is current and writable before you redirect clients.

**Traffic switch**

Once the target is synchronized, you redirect application traffic. The mechanism depends on the architecture:

- **Route 53**: Update DNS records or weighted routing.
- **Elastic Load Balancing**: Shift target groups or listener rules.
- **Application configuration**: Change connection strings, endpoints, or secrets.
- **Network routing**: Update routes, NAT paths, or firewall rules if the target is in a different network segment.

DNS-based cutovers are common, but remember that TTL (time to live) affects how quickly clients observe the change. Lower TTL values before the maintenance window if you need faster propagation.

Example Route 53 change pattern:

```json
{
  "Comment": "Cut over app.example.com to the new target",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack-new-alb-123456.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

After the switch, monitor request rates, error rates, latency, and database write behavior. A successful cutover is not complete until the application is stable under real traffic.

**Data consistency validation**

Validation confirms that the target contains the expected data and that the application behaves correctly after the switch. Use multiple layers of checks because no single test proves full consistency.

Common validation methods include:

- **Record counts**: Compare row counts, object counts, or file counts between source and target.
- **Checksums or hashes**: Compare digests for files or exported datasets.
- **Spot queries**: Verify representative business records and recent transactions.
- **Referential integrity checks**: Confirm that related records still link correctly.
- **Application smoke tests**: Exercise login, search, checkout, or write operations.
- **Metric comparison**: Compare error rates, latency, and throughput before and after cutover.

For databases, you often validate both schema and data:

```sql
-- Compare table counts
SELECT COUNT(*) FROM orders;

-- Check recent records
SELECT order_id, status, updated_at
FROM orders
ORDER BY updated_at DESC
LIMIT 10;
```

For file-based migrations, checksum validation is useful:

```bash
# Compare file hashes on source and target
sha256sum /data/export/*.csv > source.hashes
sha256sum /mnt/target/export/*.csv > target.hashes
diff -u source.hashes target.hashes
```

For object storage, you may compare object listings and sizes:

```bash
aws s3 ls s3://source-bucket/data/ --recursive --human-readable --summarize
aws s3 ls s3://target-bucket/data/ --recursive --human-readable --summarize
```

**What consistency means**

Consistency does not always mean byte-for-byte identity across every system. In real migrations, you may accept small differences if they are expected and documented, such as:

- New timestamps generated by the target system.
- Metadata changes caused by re-encryption or re-upload.
- Non-deterministic ordering in exports.
- Application-generated IDs that differ between environments.

What matters is that the target preserves the required business state and that any differences are understood and acceptable.

**Validation checklist**

Use a structured checklist to avoid missing critical checks during the maintenance window.

- Confirm replication is complete or within approved lag.
- Confirm the target accepts writes if the workload requires it.
- Verify schema version compatibility.
- Compare row counts or object counts.
- Validate key business transactions.
- Run application smoke tests.
- Check logs for errors, retries, or timeouts.
- Confirm monitoring alarms are healthy.
- Verify external integrations, such as email, payment, or identity providers.
- Document the result and obtain sign-off.

**Rollback planning**

Rollback is part of cutover design, not an afterthought. If validation fails, you need a fast path back to the source or a safe way to pause traffic.

Rollback options include:

- Repoint DNS or load balancer traffic to the source.
- Re-enable source writes if the source remains authoritative.
- Restore from snapshot or backup if the target is corrupted.
- Re-run replication after fixing the issue.
- Keep the source in read-only standby until the target is proven stable.

> Avoid dual-writer states unless the architecture explicitly supports conflict resolution. Uncontrolled writes to both sides can create irreconcilable divergence.

A rollback decision should be based on objective criteria, such as:

- Missing records above an agreed threshold.
- Failed smoke tests.
- Elevated error rates after cutover.
- Replication or application errors on the target.
- Data integrity checks that do not match expected results.

**Practical AWS services and signals**

In AWS migration labs, you may encounter several services that help with cutover and validation:

- **AWS Database Migration Service (AWS DMS)**: Use task status, latency metrics, and full load plus CDC (change data capture) completion signals.
- **Amazon RDS**: Use replica lag, event logs, and parameter group readiness.
- **Amazon S3**: Use object counts, versioning, and checksum-based verification.
- **AWS Application Load Balancer**: Use target health checks and listener rule changes.
- **Amazon Route 53**: Use weighted or failover routing for controlled traffic shifts.
- **Amazon CloudWatch**: Use dashboards and alarms to observe post-cutover behavior.
- **AWS CloudTrail**: Use API activity logs to confirm that the expected changes occurred.

Example CloudWatch alarm focus areas:

- 5XX error rate.
- Latency p95 or p99.
- CPU and memory saturation.
- Database connections.
- Replication lag.
- Disk queue depth.

**Operational example**

A typical migration cutover for a web application with a database might follow this pattern:

1. Lower DNS TTL well before the window.
2. Confirm the target database replica is fully caught up.
3. Put the application into maintenance mode.
4. Stop background workers and scheduled jobs.
5. Run a final data sync.
6. Validate row counts and recent transactions.
7. Promote the target database.
8. Update application endpoints or DNS records.
9. Re-enable traffic gradually.
10. Monitor logs, metrics, and user-facing errors.
11. Roll back if validation fails or if error rates exceed the threshold.

This sequence works because it separates data freeze, synchronization, traffic shift, and post-cutover verification into distinct phases.

**Common pitfalls**

- Cutting over before replication lag reaches an acceptable level.
- Forgetting to stop asynchronous writers such as queues or cron jobs.
- Validating only counts and not business-critical records.
- Ignoring DNS caching and TTL behavior.
- Failing to test rollback before the maintenance window.
- Overlooking external dependencies like identity providers or payment gateways.
- Assuming the target is correct because the migration tool reported success.

**Exam focus**

For SAP-C02, expect scenario questions that ask you to choose the safest migration approach. The best answer usually:

- Minimizes downtime without sacrificing integrity.
- Uses a controlled final sync.
- Validates data with more than one method.
- Includes a rollback path.
- Uses AWS-native monitoring to confirm success.

When two options seem similar, prefer the one that explicitly verifies consistency and reduces the chance of split-brain or data loss.

**Quick reference**

- **Cutover**: The controlled switch from source to target.
- **Final sync**: The last replication or copy step before traffic moves.
- **Consistency validation**: Checks that confirm the target matches expected business state.
- **Rollback**: The planned return to the source or a safe recovery state.
- **TTL**: DNS cache duration that affects how quickly clients see endpoint changes.
