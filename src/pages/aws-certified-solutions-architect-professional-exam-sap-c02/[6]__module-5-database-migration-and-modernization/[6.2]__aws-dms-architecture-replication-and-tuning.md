## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**AWS DMS overview**

AWS Database Migration Service (<abbr title="Database Migration Service">DMS</abbr>) moves data between homogeneous and heterogeneous databases with minimal downtime. You use it for one-time migrations, ongoing replication, or both together in a full load plus change data capture (<abbr title="Change Data Capture">CDC</abbr>) pattern.

DMS is not a schema design tool. It moves data and, in many cases, table structures, but you still need to plan schema conversion, object compatibility, and cutover sequencing. For exam scenarios, the key question is usually whether you need:
- a lift-and-shift style migration,
- continuous replication for low downtime,
- or a heterogeneous migration that requires schema conversion support from AWS Schema Conversion Tool (<abbr title="AWS Schema Conversion Tool">AWS SCT</abbr>).

**Core architecture**

A DMS deployment has three main parts:
- **Replication instance** or **DMS Serverless** compute
- **Source endpoint**
- **Target endpoint**

The replication component reads from the source, stages and transforms changes as needed, and writes to the target. It also manages task state, cached changes, and checkpointing.

A typical flow looks like this:

```text
Source DB -> Source endpoint -> DMS replication engine -> Target endpoint -> Target DB
```

For heterogeneous migrations, AWS SCT usually converts schema objects first, then DMS migrates the data. For homogeneous migrations, DMS can often handle the move with less conversion work, but you still validate data types, constraints, and application behavior.

**Replication instance versus DMS Serverless**

A **replication instance** is a provisioned compute resource that you size and manage. You choose instance class, storage, and networking, and you monitor CPU, memory, disk, and task throughput.

**DMS Serverless** removes most instance sizing decisions. You define the migration, and AWS scales capacity based on workload. Use it when:
- workload patterns are variable,
- you want less infrastructure management,
- or you prefer a simpler operational model for intermittent migrations.

Exam-relevant differences:
- **Replication instance**
  - Better when you need predictable sizing and direct control.
  - Useful for long-running, steady CDC workloads.
  - Lets you reason about CPU, memory, and storage bottlenecks explicitly.
- **DMS Serverless**
  - Better for bursty or less predictable migration demand.
  - Reduces capacity planning overhead.
  - Still requires endpoint, task, and network design, and you must watch migration behavior rather than instance metrics.

Operationally, both models still depend on source connectivity, target write performance, and task settings. Serverless does not remove schema conversion work, unsupported object limitations, or the need to validate cutover behavior.

**Endpoint architecture and configuration**

Endpoints define how DMS connects to databases and how it behaves with each side of the migration. Endpoint choice matters because source and target settings influence extraction, transformation, and load behavior.

Common endpoint types include:
- **Source endpoint**
  - Reads from the source database.
  - Must support the source engine’s logging or replication features needed for CDC.
- **Target endpoint**
  - Writes to the destination database.
  - Must support the target engine’s load semantics, constraints, and transaction behavior.
- **S3 endpoint**
  - Often used as an intermediate landing zone or target for unload-style workflows.
- **Kinesis / Kafka / other streaming targets**
  - Used when the migration pattern includes event streaming or downstream processing.
- **Document or analytical targets**
  - Used in specialized pipelines where DMS feeds non-relational destinations.

Endpoint-specific settings affect migration behavior in practical ways:
- **Source endpoint settings**
  - Control how DMS reads metadata, logs, and transaction changes.
  - Can influence whether DMS can perform CDC at all.
  - May require supplemental logging, binlog retention, or logical replication configuration depending on the engine.
- **Target endpoint settings**
  - Control how DMS writes rows, handles identity columns, and manages transaction batching.
  - Can affect whether DMS truncates, appends, or upserts data.
  - May determine how DMS handles case sensitivity, schema names, and table preparation.

A few examples of endpoint behavior:
- A source endpoint for MySQL requires binary logs and sufficient retention so DMS can resume CDC after interruptions.
- A PostgreSQL source often needs logical replication settings and replication slots.
- A target endpoint for Amazon Aurora or Amazon RDS may need table preparation and constraint handling tuned to avoid load failures.

**Task types and migration patterns**

DMS tasks define what to move and how to move it. The common task modes are:
- **Migrate existing data**
  - Full load only.
  - Best for one-time moves where CDC is not needed.
- **Migrate existing data and replicate ongoing changes**
  - Full load plus CDC.
  - Most common for low-downtime cutovers.
- **Replicate data changes only**
  - CDC only.
  - Useful when the target is already seeded or when you are building a continuous replication pipeline.

A full load plus CDC migration usually follows this sequence:
1. DMS loads the initial dataset.
2. DMS captures changes that occur during the load.
3. DMS applies cached changes after the load completes.
4. DMS continues CDC until cutover.

This pattern is central to exam questions about minimizing downtime while preserving consistency.

**Schema conversion and AWS SCT boundaries**

AWS SCT handles schema conversion, while DMS handles data movement. That boundary matters most in heterogeneous migrations.

Use AWS SCT when you need to:
- convert database objects between different engines,
- identify incompatible data types,
- estimate conversion effort,
- and generate target-side DDL.

Use DMS when you need to:
- move table data,
- replicate changes,
- and perform the actual migration cutover.

Important boundaries:
- DMS does not fully solve application-level incompatibilities.
- DMS does not automatically convert every stored procedure, trigger, or vendor-specific object.
- DMS may migrate table data even when some schema objects require manual remediation.
- AWS SCT can generate conversion reports, but you still validate the target behavior.

For heterogeneous migrations, the exam often expects you to sequence the work as:
1. Assess source schema with AWS SCT.
2. Convert supported objects.
3. Manually fix unsupported or partially supported objects.
4. Create target schema.
5. Run DMS full load and CDC.
6. Validate application compatibility and performance.

**Unsupported objects, DDL, and schema drift**

DMS is strongest at data movement, not at full database lifecycle replication. It has limitations around DDL replication and schema drift.

Typical limitations and behaviors:
- **DDL replication**
  - DMS does not reliably replicate all schema changes across all engines.
  - Some DDL may be ignored, partially applied, or require manual intervention.
- **Schema drift**
  - If the source schema changes during migration, DMS may continue moving rows but the target can become inconsistent with the source structure.
  - You should freeze schema changes during critical migration windows when possible.
- **Unsupported object types**
  - Stored procedures, functions, triggers, packages, and certain advanced types often need manual conversion or separate deployment.
  - Large objects, partitioning features, and engine-specific extensions may require special handling.

Practical implication:
- If developers alter source tables during CDC, DMS may keep replicating row-level changes, but target table definitions may no longer match the source.
- If a column is added or altered without coordinating the target schema, tasks can fail or data can land incorrectly.
- For exam scenarios, the safest answer is usually to control schema changes through a release freeze or a managed schema deployment process.

**Replication instance sizing and placement**

Replication performance depends heavily on compute, memory, storage, and network placement.

Key sizing considerations:
- **CPU**
  - Affects transformation, parallel load, and CDC apply throughput.
  - High CPU can indicate too many tasks, too much transformation, or insufficient instance size.
- **Memory**
  - Used for caching changes, sorting, and buffering.
  - Low memory can force spill to disk and slow replication.
- **Storage**
  - Used for task logs, cached changes, and spill files.
  - Insufficient disk can cause task failures or severe throttling.
- **Network**
  - Latency and bandwidth between source, replication engine, and target directly affect throughput.

Placement guidance:
- Put the replication component in a VPC with network reachability to both source and target.
- Prefer low-latency connectivity to both endpoints.
- Use private connectivity where possible for stable throughput and security.
- If source and target are in different regions, expect higher latency and tune for larger batches and more buffering.

**DMS Serverless operational considerations**

DMS Serverless is useful when you want less infrastructure management, but you still need to design for migration behavior.

Use it when:
- you have variable or intermittent migration workloads,
- you want to avoid choosing a fixed replication instance size,
- or you are running multiple migrations with different demand profiles.

Be aware of:
- endpoint connectivity and permissions still matter,
- CDC lag can still occur if the source or target is slow,
- schema conversion and unsupported object issues remain unchanged,
- and monitoring shifts from instance sizing to workload behavior and task health.

For exam purposes, Serverless is not a magic fix for poor source logging, target bottlenecks, or schema incompatibility.

**Task settings commonly tested**

Task settings control how DMS loads data, handles errors, and manages CDC behavior. These settings are frequent exam targets because they directly affect cutover success.

Common categories include:
- **Target table preparation mode**
- **Error handling and retry behavior**
- **LOB handling**
- **CDC stop/start behavior**
- **Parallel load and commit behavior**

Target table preparation modes:
- **Do nothing**
  - Use when the target already contains the desired structure and data state.
  - Risky if the target has stale rows or incompatible constraints.
- **Truncate**
  - Clears existing rows before load.
  - Useful when you want a clean reload into existing tables.
- **Drop tables on target**
  - Removes and recreates tables.
  - Useful for fresh migrations when you want DMS to manage table creation.
- **Delete from target**
  - Removes rows without dropping the table.
  - Can be slower than truncate and may interact with triggers or constraints.

Error handling and retry behavior:
- Configure whether DMS stops on table errors or continues.
- Use retry settings to handle transient network or target issues.
- Decide whether to log, skip, or fail on row-level errors based on data quality requirements.
- For critical migrations, fail fast is often safer than silently skipping bad rows.

CDC stop/start options:
- You can stop CDC at a controlled point for cutover.
- You can resume a task from a checkpoint if the source logs are still available.
- You can restart from the beginning of the task if the checkpoint is no longer valid or if the target state is inconsistent.

LOB handling:
- **Full LOB mode**
  - Migrates large objects completely.
  - Safer for correctness, but slower and more memory-intensive.
- **Limited LOB mode**
  - Migrates up to a configured size.
  - Faster, but truncates values beyond the limit.
- **Inline LOB mode**
  - Optimizes smaller LOBs by keeping them in row data.
  - Useful when most LOBs are small and performance matters.

**Practical tuning scenarios**

Tuning is usually about balancing correctness, throughput, and resource usage. The right settings depend on source latency, target capacity, row size, and CDC volume.

Large LOBs:
- Use full LOB mode when data integrity matters more than speed.
- Increase memory and storage to absorb large payloads.
- Consider reducing parallelism if the target cannot keep up with large row writes.
- Example use case: document tables, media metadata, or legacy systems with CLOB/BLOB fields.

High-latency links:
- Increase batch sizes where supported.
- Prefer fewer round trips by using larger commit intervals.
- Place the replication component closer to the source or target if one side is remote.
- Example use case: cross-region migration where source and target are in different AWS Regions.

Heavy CDC workloads:
- Increase instance capacity or use Serverless scaling if the workload is bursty.
- Ensure source logs are retained long enough for lag spikes.
- Tune target write throughput and indexes to reduce apply bottlenecks.
- Example use case: transactional systems with frequent updates and deletes.

A practical configuration example for a large-load, moderate-CDC migration:

```json
{
  "TargetMetadata": {
    "TargetSchema": "",
    "SupportLobs": true,
    "FullLobMode": true,
    "LobChunkSize": 64,
    "LimitedSizeLobMode": false
  },
  "FullLoadSettings": {
    "TargetTablePrepMode": "TRUNCATE_BEFORE_LOAD",
    "MaxFullLoadSubTasks": 8,
    "CommitRate": 10000
  },
  "ChangeProcessingTuning": {
    "BatchApplyEnabled": true,
    "BatchApplyPreserveTransaction": true,
    "MinTransactionSize": 1000,
    "MemoryLimitTotal": 1024,
    "MemoryKeepTime": 60
  },
  "ErrorBehavior": {
    "DataErrorPolicy": "LOG_ERROR",
    "DataTruncationErrorPolicy": "LOG_ERROR",
    "TableErrorPolicy": "STOP_TASK",
    "ApplyErrorDeletePolicy": "IGNORE_RECORD"
  }
}
```

Interpretation:
- `TRUNCATE_BEFORE_LOAD` helps when the target already has tables but should be reloaded cleanly.
- `BatchApplyEnabled` improves CDC throughput when transaction ordering can be preserved.
- `LOG_ERROR` is useful when you want visibility into bad rows without immediately stopping the task.
- `STOP_TASK` on table errors is safer when structural issues indicate a migration design problem.

Another example for a low-latency, high-throughput CDC pipeline:

```json
{
  "ChangeProcessingTuning": {
    "BatchApplyEnabled": true,
    "BatchApplyPreserveTransaction": true,
    "MinTransactionSize": 500,
    "CommitTimeout": 1,
    "MemoryLimitTotal": 2048
  },
  "FullLoadSettings": {
    "MaxFullLoadSubTasks": 16,
    "CommitRate": 5000
  }
}
```

Use this kind of tuning when:
- the target can absorb larger batches,
- transaction ordering matters,
- and you want to reduce per-transaction overhead.

**Monitoring and alarms**

Monitoring is essential because DMS failures often appear first as lag, memory pressure, or disk spill rather than a hard task stop.

Important metrics and signals:
- **Replication lag**
  - Measures how far behind the task is from the source.
  - Rising lag often indicates source throughput, network, or target apply bottlenecks.
- **CPU utilization**
  - High CPU suggests the replication engine is saturated.
  - Common causes include heavy transformations, too many subtasks, or insufficient instance size.
- **Memory utilization**
  - High memory use can lead to spill to disk.
  - Watch for sustained pressure during full load and CDC bursts.
- **Disk usage and swap/spill**
  - DMS may spill cached changes to disk when memory is insufficient.
  - Excessive spill slows replication and can precede task instability.
- **Task throughput**
  - Rows per second, bytes per second, and apply rate help identify whether the source or target is the bottleneck.
- **CDC latency**
  - Useful for understanding whether the task is keeping up with ongoing changes.
- **Full load progress**
  - Helps distinguish initial load issues from CDC issues.

Useful alarms:
- replication lag above a threshold for a sustained period,
- CPU above a high-water mark,
- free storage below a safe floor,
- memory pressure or swap growth,
- task status transitions to failed or stopped unexpectedly.

A practical CloudWatch-style alarm strategy is to alert on:
- lag increasing for more than a few minutes,
- disk usage approaching capacity,
- and CPU or memory saturation during cutover windows.

**Recovery, restart, and checkpoint behavior**

Interrupted tasks do not always restart from zero. DMS uses checkpoints and cached changes, but recovery depends on source log retention and task state.

Key behaviors:
- If a task stops and the source logs are still available, DMS can often resume from the last checkpoint.
- If the task has cached changes in memory or on disk, DMS may replay them after restart.
- If the source log position is no longer available, DMS may require a reload or a new CDC start point.
- If the target is inconsistent after a failure, you may need to reload the table or recreate the target state.

Failure scenarios to understand:
- **Short outage**
  - Task restarts and continues from checkpoint.
  - Best case when source logs are retained and target writes are idempotent enough for the task design.
- **Long outage with expired source logs**
  - CDC cannot resume cleanly.
  - You may need to reload data or re-seed the target.
- **Target write failure**
  - Task may stop or retry depending on error policy.
  - If partial writes occurred, you must validate target consistency before resuming.
- **Schema change during outage**
  - Restart may succeed technically, but the target may no longer match the source schema.
  - You should reconcile schema before continuing.

For exam prep, remember this rule:
- DMS restart success depends on both checkpoint availability and source log retention.
- A task restart is not the same as a guaranteed consistent recovery if the target schema or source logs have drifted.

**Endpoint and task design examples**

A homogeneous migration from Amazon RDS for MySQL to Amazon Aurora MySQL might use:
- source endpoint with binary log access,
- target endpoint with table prep set to truncate or drop/recreate,
- full load plus CDC,
- and batch apply enabled for CDC.

A heterogeneous migration from Oracle to PostgreSQL might use:
- AWS SCT to convert schema objects,
- DMS to move data and CDC,
- manual remediation for unsupported procedures and packages,
- and careful validation of data types, sequences, and application SQL.

A cross-region migration with heavy write traffic might use:
- a replication component placed to minimize latency to the source,
- larger memory and storage capacity,
- CDC tuning for batch apply,
- and alarms on lag and disk spill.

**Common exam decision points**

When you see a migration question, ask:
- Do you need schema conversion, or only data movement?
- Is downtime acceptable, or do you need CDC?
- Are there unsupported objects or schema changes during migration?
- Is the bottleneck source logging, network latency, replication compute, or target write capacity?
- Do you need a fixed replication instance or a more elastic Serverless model?

A good answer usually aligns the tool with the problem:
- **AWS SCT** for schema conversion
- **AWS DMS** for data movement and CDC
- **Replication instance** for explicit capacity control
- **DMS Serverless** for reduced operational overhead
- **Task tuning** for load, LOBs, retries, and CDC throughput

**Quick reference**

- Use **full load** for initial seeding.
- Use **full load plus CDC** for low-downtime cutover.
- Use **CDC only** when the target is already populated.
- Use **AWS SCT** for heterogeneous schema conversion.
- Freeze or tightly control schema changes during migration.
- Monitor lag, CPU, memory, and disk spill.
- Size for the bottleneck you actually have, not just the source database size.
