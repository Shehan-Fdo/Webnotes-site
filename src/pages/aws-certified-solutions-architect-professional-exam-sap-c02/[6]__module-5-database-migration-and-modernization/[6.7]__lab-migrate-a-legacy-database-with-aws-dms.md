## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**AWS Database Migration Service overview**

AWS Database Migration Service (<abbr title="Database Migration Service">DMS</abbr>) helps you move data between source and target databases with minimal downtime. In a legacy migration, you typically use DMS to perform an initial full load and then keep changes flowing with change data capture (<abbr title="Change Data Capture">CDC</abbr>) until cutover.

DMS is useful when you need to:
- Migrate between engines, such as Oracle to PostgreSQL or MySQL to Aurora.
- Keep the source online during migration.
- Validate the target before switching applications.
- Reduce downtime by separating bulk load from ongoing replication.

DMS is not a schema conversion tool by itself. For heterogeneous migrations, you usually pair it with the AWS Schema Conversion Tool (<abbr title="AWS Schema Conversion Tool">SCT</abbr>) to convert schema objects, stored procedures, and engine-specific features.

**Core migration architecture**

A typical DMS migration uses three main components:
- **Source endpoint**: connection details for the legacy database.
- **Target endpoint**: connection details for the destination database.
- **Replication instance**: the managed compute that reads from the source and writes to the target.

A common topology looks like this:

```text
Legacy DB  --->  DMS Replication Instance  --->  Target DB
   |                    |                         |
   |-- full load -------|                         |
   |-- CDC changes ------------------------------>|
```

The replication instance must be able to reach both databases over the network. In practice, that means:
- Correct VPC, subnet, route table, and security group configuration.
- Open database ports on both sides.
- DNS resolution if you use hostnames.
- Proper IAM permissions for DMS to access related AWS resources.

**Planning the migration**

Before you create endpoints or tasks, confirm the migration scope. This avoids rework later and helps you choose the right task settings.

Key planning questions:
- Is this a homogeneous migration or a heterogeneous migration?
- Do you need full load only, or full load plus CDC?
- Can the target accept writes during migration, or must it remain isolated?
- Are there large tables, LOBs (<abbr title="Large Objects">LOBs</abbr>), or unsupported objects?
- Do you need ongoing validation, or only a one-time cutover?

Practical considerations:
- Use SCT for schema conversion when engines differ.
- Make sure the source database supports the required logging or replication settings for CDC.
- Confirm that the target has enough storage, IOPS, and CPU for the load phase.
- Plan for indexes, constraints, triggers, and sequences on the target, because these can affect load performance and data consistency.

**Source and target endpoint setup**

Endpoints define how DMS connects to each database. You specify engine type, server name, port, credentials, and optional SSL/TLS settings.

Example CLI for creating a source endpoint:

```bash
aws dms create-endpoint \
  --endpoint-identifier legacy-source \
  --endpoint-type source \
  --engine-name mysql \
  --server-name legacy-db.example.com \
  --port 3306 \
  --username dms_user \
  --password 'REDACTED' \
  --ssl-mode require
```

Example CLI for creating a target endpoint:

```bash
aws dms create-endpoint \
  --endpoint-identifier target-postgres \
  --endpoint-type target \
  --engine-name postgres \
  --server-name target-db.example.com \
  --port 5432 \
  --username dms_user \
  --password 'REDACTED' \
  --ssl-mode require
```

Important endpoint notes:
- Use `--ssl-mode` when the database requires encrypted transport.
- Store credentials securely, preferably with AWS Secrets Manager where supported in your workflow.
- Match the endpoint engine to the actual database engine and version compatibility.
- For some engines, you may need extra connection attributes for performance or behavior tuning.

**Testing connectivity**

DMS provides an explicit connection test. Use it before creating or starting a task so you can catch network, authentication, and SSL issues early.

CLI example:

```bash
aws dms test-connection \
  --replication-instance-arn arn:aws:dms:us-east-1:123456789012:rep:ABCDEFGHIJKL \
  --endpoint-arn arn:aws:dms:us-east-1:123456789012:endpoint:SOURCEENDPOINTID
```

You can test both source and target endpoints. A successful test confirms that the replication instance can reach the endpoint and authenticate with the provided settings.

Common causes of failure:
- Security group rules do not allow the database port.
- The replication instance is in the wrong subnet or VPC.
- The database user lacks required privileges.
- SSL/TLS settings do not match the server configuration.
- DNS or routing is misconfigured.

> If the test fails, fix connectivity first. A task cannot succeed if the replication instance cannot connect reliably to both endpoints.

**Replication instance sizing and placement**

The replication instance is the compute layer for DMS. Choose its size based on data volume, transformation complexity, LOB usage, and CDC throughput.

Selection guidance:
- Start with a size that can handle the largest expected load phase, not just the steady-state CDC phase.
- Use more memory and CPU for large tables, many indexes, or high transaction rates.
- Use storage that can absorb spikes during full load and change processing.
- Place the instance in a subnet with network access to both databases.

Practical sizing considerations:
- Small lab migrations may work on modest instance classes.
- Production migrations often need larger classes and more storage headroom.
- Instance class availability varies by region and account, so verify supported classes before planning.

DMS encryption and security:
- DMS can encrypt the replication instance storage with AWS Key Management Service (<abbr title="AWS Key Management Service">KMS</abbr>).
- Use SSL/TLS for endpoint connections when the source or target supports it.
- Restrict security groups so only required ports are open.
- Use least-privilege IAM roles for DMS-related operations.

**Task creation and migration modes**

A DMS task defines what to migrate and how to migrate it. The task settings determine whether you run a full load, CDC, or both.

Common migration types:
- **Migrate existing data**: full load only.
- **Migrate existing data and replicate ongoing changes**: full load plus CDC.
- **Replicate data changes only**: CDC only, usually after an initial load done elsewhere.

When you start a task, DMS supports different start types:
- `start-replication`: start the task from the beginning.
- `resume-processing`: continue a stopped or paused task where possible.
- `reload-target`: reload target tables, typically after a reset or data correction.

Example task creation:

```bash
aws dms create-replication-task \
  --replication-task-identifier legacy-migration-task \
  --source-endpoint-arn arn:aws:dms:us-east-1:123456789012:endpoint:SOURCEENDPOINTID \
  --target-endpoint-arn arn:aws:dms:us-east-1:123456789012:endpoint:TARGETENDPOINTID \
  --replication-instance-arn arn:aws:dms:us-east-1:123456789012:rep:ABCDEFGHIJKL \
  --migration-type full-load-and-cdc \
  --table-mappings file://table-mappings.json \
  --replication-task-settings file://task-settings.json
```

Example start command:

```bash
aws dms start-replication-task \
  --replication-task-arn arn:aws:dms:us-east-1:123456789012:task:TASKID \
  --start-replication-task-type start-replication
```

Use `resume-processing` only when the task state supports resumption. If a task failed, completed, or was stopped in a way that discards state, you may need to restart or reload instead of resuming.

**Table mappings**

Table mappings control which schemas and tables DMS includes, excludes, or transforms. This is one of the most important parts of the migration because it determines scope and can reduce unnecessary load.

Example `table-mappings.json`:

```json
{
  "rules": [
    {
      "rule-type": "selection",
      "rule-id": "1",
      "rule-name": "include-all",
      "object-locator": {
        "schema-name": "legacy_app",
        "table-name": "%"
      },
      "rule-action": "include"
    },
    {
      "rule-type": "selection",
      "rule-id": "2",
      "rule-name": "exclude-audit",
      "object-locator": {
        "schema-name": "legacy_app",
        "table-name": "audit_%"
      },
      "rule-action": "exclude"
    }
  ]
}
```

Useful mapping patterns:
- Include specific schemas and tables.
- Exclude audit, staging, or temporary tables.
- Rename schemas or tables when the target naming convention differs.
- Apply transformations when the target engine requires different object names.

Keep mappings simple unless you need transformations. Complex rules are harder to troubleshoot and can create unexpected scope gaps.

**Task settings and load behavior**

Task settings control performance, logging, validation, LOB handling, and target table preparation. These settings often determine whether a migration is smooth or painful.

Example `task-settings.json`:

```json
{
  "TargetMetadata": {
    "TargetSchema": "",
    "SupportLobs": true,
    "FullLobMode": false,
    "LimitedSizeLobMode": true,
    "LobMaxSize": 32
  },
  "FullLoadSettings": {
    "TargetTablePrepMode": "DO_NOTHING",
    "CreatePkAfterFullLoad": true,
    "StopTaskCachedChangesApplied": false,
    "StopTaskCachedChangesNotApplied": false,
    "MaxFullLoadSubTasks": 8,
    "TransactionConsistencyTimeout": 600
  },
  "Logging": {
    "EnableLogging": true
  }
}
```

Important settings to understand:
- **TargetTablePrepMode**
  - `DO_NOTHING`: use when tables already exist and you do not want DMS to drop or truncate them.
  - `DROP_AND_CREATE`: useful in disposable targets or labs.
  - `TRUNCATE_BEFORE_LOAD`: useful when tables exist and you want a clean reload, but it depends on target engine support and table constraints.
- **CreatePkAfterFullLoad**
  - Helps speed up full load by creating primary keys after loading data.
- **LOB handling**
  - Full LOB mode handles large objects more completely but can reduce performance.
  - Limited LOB mode improves performance when you know the maximum size you need.
- **Logging**
  - Enable logging during troubleshooting and initial migration runs.

> `TRUNCATE_BEFORE_LOAD` is not universally safe. Foreign keys, triggers, permissions, and engine-specific limitations can prevent truncation or make it behave differently than expected.

**Monitoring and troubleshooting**

You should monitor both the DMS task and the underlying infrastructure. DMS exposes task status, table state, latency, and error messages, while CloudWatch provides logs and metrics.

Useful CLI commands:

```bash
aws dms describe-replication-tasks \
  --filters Name=replication-task-arn,Values=arn:aws:dms:us-east-1:123456789012:task:TASKID
```

```bash
aws dms describe-table-statistics \
  --replication-task-arn arn:aws:dms:us-east-1:123456789012:task:TASKID
```

```bash
aws logs tail /aws/dms/task/TASKID --follow
```

Monitoring focus areas:
- Task status: running, stopped, failed, or ready.
- Table statistics: loaded, errored, or pending tables.
- CDC latency: how far behind the target is from the source.
- CloudWatch logs: connection errors, transformation issues, and apply failures.

Troubleshooting patterns:
- If the task is stuck in a load phase, check target table locks, indexes, and constraints.
- If CDC lags, inspect source transaction volume and replication instance capacity.
- If tables fail individually, review table mappings and target data type compatibility.
- If the task fails immediately, verify endpoint connectivity and credentials.

**Validation and cutover**

Validation is the step where you confirm that the target contains the expected data and that the application can switch safely.

A practical cutover sequence:
1. Run the full load and let CDC catch up.
2. Validate row counts, key tables, and application-specific checks.
3. Pause writes to the source application during the cutover window.
4. Allow DMS to apply remaining changes until latency reaches zero or an acceptable threshold.
5. Switch the application connection string to the target database.
6. Monitor application behavior and database health after the switch.

Validation methods:
- Compare row counts for critical tables.
- Check sample records and business totals.
- Verify indexes, constraints, and sequences.
- Run application smoke tests against the target.

Use caution with validation assumptions:
- Row counts alone do not prove semantic correctness.
- Some data types, collations, and precision differences can affect results.
- Heterogeneous migrations may require manual checks for stored procedures or application logic.

**Operational best practices**

A successful legacy migration depends on more than just starting a task. You should treat the migration as an operational workflow.

Best practices:
- Test connectivity before task creation.
- Use SCT for schema conversion when engines differ.
- Start with a limited table set in a non-production rehearsal.
- Enable logging during the first run.
- Monitor replication lag and task errors continuously.
- Keep the source available until validation is complete.
- Document rollback steps before cutover.

Common exam-relevant tradeoffs:
- Full load plus CDC minimizes downtime but requires more planning.
- Larger replication instances improve throughput but increase cost.
- Precreating target tables can help control schema, but it also shifts responsibility for constraints and indexes to you.
- Truncating or dropping target tables can simplify reloads, but it is risky in shared environments.

**Example end-to-end workflow**

A typical migration workflow looks like this:
1. Assess the source database and identify schema conversion needs.
2. Create or convert the target schema.
3. Provision the DMS replication instance in the correct VPC and subnet.
4. Create source and target endpoints with secure connectivity.
5. Test both connections.
6. Define table mappings and task settings.
7. Create the replication task.
8. Start the task with the appropriate start type.
9. Monitor logs, table statistics, and latency.
10. Validate the target and cut over the application.
11. Stop or delete the task after the migration is complete.
