## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Lab overview**

A **full load plus CDC** migration uses AWS Database Migration Service (AWS DMS) to copy existing data first, then keep the target synchronized by applying ongoing changes from the source. This pattern is common for production cutovers because it reduces downtime while still giving you a controlled migration window.

Use this approach when you need to:

- Move a live database with minimal interruption.
- Validate the target before cutover while source traffic continues.
- Keep a fallback path until you are ready to switch applications.
- Migrate between engines or versions where native replication is not practical.

A typical migration flow is:

1. Prepare source and target databases.
2. Create the DMS replication resources.
3. Run a task in **full load + CDC** mode.
4. Monitor load progress and replication lag.
5. Quiesce the source, let CDC catch up, and cut over applications.
6. Validate data and decommission the old path after acceptance.

**Why this pattern matters for SAP-C02**

For the exam, you should recognize when AWS DMS is the right tool versus native database replication, backup/restore, or application-level synchronization. Full load plus CDC is especially useful when you need a managed migration service that can handle heterogeneous engines and staged cutover.

**Core AWS DMS components**

AWS DMS uses a few building blocks that you should know before starting the lab:

- **Replication instance**: The managed compute that runs the migration task.
- **Source endpoint**: Connection details for the source database.
- **Target endpoint**: Connection details for the destination database.
- **Replication task**: The migration job that defines what to move and how.
- **Table mappings**: Rules that select schemas, tables, and transformations.

A practical mental model is:

- The replication instance reads from the source.
- It performs the initial bulk copy during full load.
- It then reads ongoing source changes and applies them to the target during CDC, or **change data capture**.

**Reference architecture**

A common lab topology looks like this:

- Source database in one VPC or on-premises network.
- Target database in another VPC, or in the same VPC for a lab.
- DMS replication instance in a subnet with network reachability to both endpoints.
- Security groups and routing that allow database ports and DNS resolution.

```text
Source DB  --->  DMS Replication Instance  --->  Target DB
   |                    |                         |
   |---- full load -----|                         |
   |---- CDC stream ----------------------------->|
```

**Prerequisites and planning**

Before you create the task, confirm the source and target are ready for the migration mode you want.

- **Network access**
  - The replication instance must reach both endpoints on the correct port.
  - Security groups, NACLs, route tables, and DNS must all work together.
- **Credentials and permissions**
  - DMS needs credentials that can read source data and write target data.
  - Some engines require extra permissions for CDC or metadata access.
- **Storage and sizing**
  - The replication instance needs enough storage for cached changes, logs, and task metadata.
  - Larger tables and slower targets require more headroom.
- **Schema compatibility**
  - Verify data types, primary keys, and constraints.
  - Some engine pairs need manual adjustments for unsupported types or behaviors.
- **Change capture readiness**
  - The source must support ongoing change capture in a way DMS can read.
  - For some engines, that means enabling logical replication, transaction log access, or engine-specific CDC settings.

> Do not assume that “CDC enabled” means the task will work automatically. The source engine, version, and configuration determine how DMS reads changes.

**Source and target preparation**

The exact prep steps depend on the database engine, but the goal is always the same: make the source readable for initial load and change capture, and make the target writable and structurally compatible.

**Source preparation checklist**

- Ensure the source has a stable primary key or unique identifier for tables you want to replicate.
- Confirm the source retains enough change history for the migration window.
- Enable the engine-specific settings required for DMS CDC.
- Avoid long-running schema changes during the migration window.
- Reduce write bursts if you want a cleaner cutover.

**Target preparation checklist**

- Create the target database, schema, and required users.
- Decide whether DMS should create tables or whether you will pre-create them.
- Make sure target storage, parameter groups, and maintenance settings are appropriate.
- Disable or defer constraints, triggers, or indexes if they would slow the initial load.
- Re-enable validation controls after the load if your migration plan requires them.

**Engine-specific CDC notes**

Different engines expose changes differently. The exam expects you to know the general pattern, not every implementation detail.

| Source engine | Typical CDC requirement | Notes |
|---|---|---|
| PostgreSQL | Logical replication or equivalent logical decoding path | DMS reads changes from the source’s logical change stream; exact setup depends on version and configuration. |
| MySQL / MariaDB | Binary logs | Binlog format and retention must support the migration window. |
| Oracle | Archived redo logs and supplemental logging | DMS needs enough log history and the right logging settings. |
| SQL Server | Transaction log access and/or CDC-related configuration | DMS can use engine-specific log-based capture paths; do not assume one universal setup. |

**Creating the DMS resources**

You usually create the replication instance first, then endpoints, then the task. In a lab, the console is often the fastest path, but the CLI is useful for repeatability.

**Example: create a replication subnet group**

A replication instance needs a subnet group that references existing subnets. Create that first if it does not already exist.

```bash
aws dms create-replication-subnet-group \
  --replication-subnet-group-identifier lab-dms-subnet-group \
  --replication-subnet-group-description "Lab subnet group for DMS" \
  --subnet-ids subnet-0123456789abcdef0 subnet-0fedcba9876543210
```

**Example: create a replication instance**

```bash
aws dms create-replication-instance \
  --replication-instance-identifier lab-dms-ri \
  --replication-instance-class dms.t3.medium \
  --allocated-storage 50 \
  --vpc-security-group-ids sg-0123456789abcdef0 \
  --replication-subnet-group-identifier lab-dms-subnet-group \
  --publicly-accessible false
```

**Example: create source and target endpoints**

```bash
aws dms create-endpoint \
  --endpoint-identifier source-pg \
  --endpoint-type source \
  --engine-name postgres \
  --username dms_user \
  --password 'REDACTED' \
  --server-name source-db.example.internal \
  --port 5432 \
  --database-name appdb
```

```bash
aws dms create-endpoint \
  --endpoint-identifier target-pg \
  --endpoint-type target \
  --engine-name postgres \
  --username dms_user \
  --password 'REDACTED' \
  --server-name target-db.example.internal \
  --port 5432 \
  --database-name appdb
```

**Creating the migration task**

The task defines the migration type, table mappings, and task settings. For this lab, the key setting is **full-load-and-cdc**.

```bash
aws dms create-replication-task \
  --replication-task-identifier lab-full-load-cdc \
  --source-endpoint-arn arn:aws:dms:us-east-1:123456789012:endpoint:SOURCEARN \
  --target-endpoint-arn arn:aws:dms:us-east-1:123456789012:endpoint:TARGETARN \
  --replication-instance-arn arn:aws:dms:us-east-1:123456789012:rep:RIARN \
  --migration-type full-load-and-cdc \
  --table-mappings file://table-mappings.json \
  --replication-task-settings file://task-settings.json
```

A simple table mapping file might look like this:

```json
{
  "rules": [
    {
      "rule-type": "selection",
      "rule-id": "1",
      "rule-name": "include-all",
      "object-locator": {
        "schema-name": "public",
        "table-name": "%"
      },
      "rule-action": "include"
    }
  ]
}
```

A basic task settings file can control logging, validation, and target behavior:

```json
{
  "Logging": {
    "EnableLogging": true
  },
  "FullLoadSettings": {
    "TargetTablePrepMode": "DO_NOTHING",
    "StopTaskCachedChangesApplied": false,
    "StopTaskCachedChangesNotApplied": false
  },
  "ValidationSettings": {
    "EnableValidation": true,
    "ThreadCount": 5
  }
}
```

**Task settings you should understand**

- **TargetTablePrepMode**
  - `DO_NOTHING`: Use when tables already exist and are ready.
  - `DROP_AND_CREATE`: Rebuild target tables before loading.
  - `TRUNCATE_BEFORE_LOAD`: Clear existing rows but keep table definitions.
- **Validation**
  - Helps confirm source and target row-level consistency.
  - Useful in labs and pre-cutover testing.
- **Logging**
  - Essential for troubleshooting task failures and transformation issues.
- **LOB handling**
  - Large objects can require special settings if the source contains BLOBs or CLOBs.
  - If you see truncation or performance issues, review LOB mode and size limits.

**Starting and monitoring the task**

After creation, start the task and watch the status transitions.

```bash
aws dms start-replication-task \
  --replication-task-arn arn:aws:dms:us-east-1:123456789012:task:LABTASKARN \
  --start-replication-task-type start-replication
```

Common task states include:

- **Creating**
- **Ready**
- **Starting**
- **Running**
- **Stopped**
- **Failed**

You can inspect task status with:

```bash
aws dms describe-replication-tasks \
  --filters Name=replication-task-arn,Values=arn:aws:dms:us-east-1:123456789012:task:LABTASKARN
```

Useful monitoring signals include:

- Full load progress per table.
- CDC latency and source/target lag.
- Task logs for warnings about schema, permissions, or data conversion.
- Table statistics showing rows loaded, inserted, updated, deleted, or errored.

**What you should expect in the console**

During a healthy run, the console usually shows:

- The task status as **Running**.
- Full load tables moving from pending to loaded.
- CDC metrics showing activity if the source is changing.
- No persistent error rows in table statistics.
- Log entries that are informational rather than fatal.

If the task is stuck, look for:

- Endpoint connectivity failures.
- Authentication errors.
- Missing source permissions.
- Unsupported data types or schema mismatches.
- Target constraint violations.

**Real-world migration scenario**

A common production pattern is moving an application database from an on-premises PostgreSQL server to Amazon RDS for PostgreSQL with minimal downtime.

Why teams choose full load plus CDC:

- The database is too large for a short outage window.
- The application cannot tolerate a long maintenance event.
- The team wants to validate the new RDS instance before switching traffic.
- The source must remain available while the target is warmed up.

Typical sequence:

1. Create the RDS target and confirm parameter settings.
2. Configure DMS with source and target endpoints.
3. Run a full load to copy the existing dataset.
4. Keep CDC running while the application continues to write to the source.
5. Freeze writes briefly during cutover.
6. Wait for the source to stop generating changes and for DMS to apply the remaining backlog.
7. Point the application to the target and verify behavior.

**Cutover strategy**

Cutover is the point where you stop treating the source as the live system and switch the application to the target. The safest approach is to combine source quiescence with CDC catch-up.

**Recommended cutover checklist**

1. Confirm the full load has completed for all required tables.
2. Verify CDC is running without persistent errors.
3. Pause or stop application writes to the source.
4. Wait for DMS to apply the remaining cached changes.
5. Confirm there are no unapplied changes and the source is quiesced.
6. Switch application endpoints, connection strings, or DNS.
7. Validate application behavior on the target.
8. Keep the source available for rollback until acceptance is complete.

> A displayed lag value alone is not enough. You want the source to be quiet and the task to show that all cached changes have been applied.

**Validation and reconciliation**

After cutover, compare source and target behavior before you declare success.

- Check row counts for critical tables.
- Compare sample records and business keys.
- Validate application transactions end to end.
- Review DMS validation results if enabled.
- Investigate any rejected rows or transformation mismatches.

If you need a quick SQL sanity check, use simple counts and checksums where appropriate:

```sql
SELECT COUNT(*) FROM public.orders;
SELECT COUNT(*) FROM public.customers;
```

For more targeted validation, compare recent records:

```sql
SELECT id, updated_at
FROM public.orders
ORDER BY updated_at DESC
LIMIT 10;
```

**Troubleshooting patterns**

DMS issues often fall into a few repeatable categories.

- **Connectivity**
  - The replication instance cannot reach the source or target.
  - Check security groups, routing, DNS, and port access.
- **Permissions**
  - The source user lacks read or log access.
  - The target user lacks insert, update, or schema privileges.
- **Schema mismatch**
  - A column type or length differs between source and target.
  - A table lacks a primary key or suitable unique key.
- **Data conversion**
  - Time zones, precision, or encoding cause row errors.
  - Large objects may need special handling.
- **Performance**
  - The replication instance is undersized.
  - The target cannot keep up with the incoming apply rate.
  - Indexes or constraints slow the initial load.

**Example troubleshooting commands**

```bash
aws dms describe-replication-tasks \
  --filters Name=replication-task-arn,Values=arn:aws:dms:us-east-1:123456789012:task:LABTASKARN \
  --query 'ReplicationTasks[0].Status'
```

```bash
aws dms describe-table-statistics \
  --replication-task-arn arn:aws:dms:us-east-1:123456789012:task:LABTASKARN
```

Look for these clues in logs and statistics:

- Tables with zero rows loaded when they should have data.
- Repeated apply errors on a specific table.
- CDC latency that grows steadily instead of stabilizing.
- Validation failures concentrated in one schema or datatype.

**Performance tuning tips**

- Increase replication instance size if CPU or memory is constrained.
- Use a target with enough IOPS and throughput for the apply rate.
- Limit the task to required schemas and tables instead of migrating everything.
- Reduce unnecessary indexes during the initial load, then rebuild them later.
- Tune batch and commit behavior only after you understand the workload.

**When to use full load plus CDC versus other approaches**

| Approach | Best for | Tradeoff |
|---|---|---|
| Full load only | One-time copy with no need to keep source in sync | Requires downtime or a separate sync method later |
| Full load plus CDC | Low-downtime migration with staged cutover | More moving parts and longer task monitoring |
| Native replication | Same-engine or tightly compatible environments | Less flexible across heterogeneous engines |
| Backup and restore | Fast recovery or same-engine lift-and-shift | Usually not enough for near-zero-downtime cutover |

**Exam-focused takeaways**

- **AWS DMS** is the managed service for database migration and ongoing change replication.
- **Full load plus CDC** is the standard pattern for low-downtime cutover.
- The replication instance, endpoints, and task must all be correctly configured.
- Source CDC readiness depends on the engine and version.
- Cutover is safest when the source is quiesced and DMS has applied all cached changes.
- Validation and troubleshooting are part of the migration, not optional extras.

**Quick mental checklist for the lab**

- Source reachable from the replication instance.
- Target ready to accept writes.
- Replication subnet group exists.
- Replication instance created and available.
- Endpoints tested successfully.
- Task set to `full-load-and-cdc`.
- Full load completed.
- CDC running and caught up.
- Source writes paused.
- Application switched and validated.
