## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Schema conversion in AWS migrations**

Schema conversion is the process of adapting database objects and SQL code so a source database can run on a target engine. In AWS exam scenarios, this usually appears when you migrate from one database family to another, such as Oracle to PostgreSQL-compatible Aurora, SQL Server to PostgreSQL, or MySQL to Aurora MySQL.

AWS Schema Conversion Tool (**AWS SCT**) is the primary AWS tool for heterogeneous migrations. It analyzes source schemas, estimates conversion effort, and helps generate target-side DDL and code where possible. It is not a guarantee of semantic equivalence; you still need to validate behavior, performance, and application compatibility after conversion.

- **Heterogeneous migration** means source and target engines differ, such as Oracle to PostgreSQL.
- **Homogeneous migration** means the same engine family, such as MySQL to Amazon RDS for MySQL.
- **Assessment** in SCT means conversion impact analysis, not runtime verification.
- **Target selection** is an architecture decision, not just a database product choice.

**When schema conversion is needed**

You use schema conversion when the target engine does not natively understand the source dialect, object model, or procedural language. This is common when you want to reduce licensing cost, move to a managed service, or standardize on an AWS-native engine.

You may not need heavy schema conversion when:
- You are doing a same-engine migration with minimal change.
- The application already uses portable SQL and avoids engine-specific features.
- You can keep the source engine and move it to managed infrastructure with little or no code change.

> Do not assume every database move requires a full conversion project. If the application depends heavily on engine-specific features, schema conversion may be risky or impractical, and a different migration strategy may be better.

**AWS SCT workflow**

A practical SCT workflow usually follows these steps:

1. Create a project in AWS SCT.
2. Select the source database engine and target database engine.
3. Connect SCT to the source database and, if needed, to the target environment.
4. Run the assessment to identify conversion blockers, warnings, and manual tasks.
5. Review the generated action items and estimate effort.
6. Convert supported schema objects and code.
7. Export the converted DDL or apply it to the target.
8. Validate the target schema, then migrate data with a separate tool or pipeline.

A typical project setup includes:
- **Source engine selection** such as Oracle, SQL Server, MySQL, PostgreSQL, or MariaDB.
- **Target engine selection** such as Amazon Aurora, Amazon RDS, or another supported database.
- **Connection profiles** for source and target endpoints.
- **Rules and mappings** to control how SCT translates objects.

Example AWS CLI-style connection context is not usually created directly in SCT, but the target database itself is often provisioned with standard AWS tooling first:

```bash
aws rds create-db-instance \
  --db-instance-identifier app-target-db \
  --engine postgres \
  --db-instance-class db.m6g.large \
  --allocated-storage 200 \
  --master-username admin \
  --master-user-password 'ChangeMe123!' \
  --multi-az
```

**How SCT assessment reports work**

SCT generates an assessment report that groups objects by conversion status. The report typically highlights:
- Objects that convert automatically.
- Objects that require manual conversion.
- Objects that are not supported.
- Code constructs that need redesign.

Interpret the report as a planning tool:
- Use it to estimate migration effort.
- Use it to identify blockers early.
- Use it to decide whether the target engine is realistic.
- Do not treat it as proof that the migrated application will behave identically.

Common report outputs include:
- **Action items** for unsupported objects or syntax.
- **Conversion statistics** by object type.
- **Severity indicators** for blockers versus warnings.
- **Manual conversion guidance** for procedures, packages, triggers, and views.

**Target selection: Aurora, RDS, RDS Custom, or EC2**

Choosing the target is often the real exam question. The right answer depends on how much operational control you need, how much compatibility you need, and how much AWS-managed functionality you want.

| Target | Best fit | Operational responsibility | Key tradeoffs |
|---|---|---:|---|
| Amazon Aurora | Cloud-native relational workloads needing high availability and read scaling | Low | Best when you want managed HA, fast failover, and read replicas; may require compatibility changes |
| Amazon RDS | Managed standard engine deployment with less admin overhead | Low to medium | Good for lift-and-shift-like moves with minimal app change, but less flexible than EC2 |
| Amazon RDS Custom | Need OS or database-level access with some managed features | Medium | Useful when you need more control than RDS but less than self-managed EC2 |
| Amazon EC2 | Full control, legacy dependencies, or unsupported features | High | Best when the engine or extensions are not supported elsewhere, but you manage everything |

**How to choose in SAP-C02 scenarios**

Use these decision points:

- Choose **Aurora** when you want:
  - High availability with managed failover.
  - Read scaling with reader endpoints.
  - Better cloud-native operational posture.
  - A target that can replace a legacy engine after code and schema changes.

- Choose **RDS** when you want:
  - Managed backups, patching, and Multi-AZ.
  - Minimal operational overhead.
  - A familiar engine family with fewer application changes.
  - A practical target for many lift-and-shift or replatforming scenarios.

- Choose **RDS Custom** when you need:
  - More control over the database host or configuration.
  - Compatibility with features that standard RDS does not expose.
  - A compromise between managed service and self-managed infrastructure.

- Choose **EC2** when you need:
  - Unsupported extensions, agents, or OS-level dependencies.
  - Full control over binaries, file system, and custom tooling.
  - A legacy architecture that cannot be adapted quickly.

**Operational tradeoffs that matter on the exam**

- **HA**: Aurora and RDS provide managed high availability patterns; EC2 requires you to design and operate HA yourself.
- **Read scaling**: Aurora is often preferred when read-heavy scaling matters.
- **Licensing**: Managed targets can simplify licensing, but some commercial engines still require careful license planning.
- **Operational responsibility**: EC2 gives maximum control but also maximum patching, backup, failover, and monitoring burden.
- **Compatibility**: The more engine-specific the source workload, the more likely you need RDS Custom or EC2.

**Migration strategy versus target selection**

Target selection and migration strategy are related but not identical. A workload may be a good fit for a managed target, but not for a direct schema conversion.

Use schema conversion when:
- The application can tolerate SQL and object translation.
- The target engine supports the required data model and procedural logic.
- You can accept some manual remediation.

Avoid schema conversion as the primary strategy when:
- The application depends on many proprietary features.
- The source uses complex packages, jobs, linked objects, or extensions with no target equivalent.
- The business requires exact behavior preservation and the target engine cannot guarantee it.

In those cases, consider:
- Staying on the source engine and moving to managed infrastructure.
- Refactoring the application before migration.
- Redesigning the data access layer.
- Choosing a target that preserves more compatibility, even if it is less cloud-native.

**Oracle migration considerations**

Oracle migrations are a common SAP-C02 topic because Oracle often uses features that do not map cleanly to PostgreSQL-compatible targets.

Important Oracle-specific issues:
- **PL/SQL** often needs manual rewrite or partial translation.
- **Packages** have no direct one-to-one equivalent in PostgreSQL.
- **Sequences** may convert, but application usage patterns still need review.
- **Synonyms** may require redesign or replacement.
- **Materialized views** may need alternative implementation.
- **Scheduler jobs** often need migration to AWS-native scheduling or application logic.
- **DATE** semantics differ from PostgreSQL `DATE` and `TIMESTAMP`; Oracle `DATE` includes time, which is a common trap.
- `SYSDATE` and `CURRENT_TIMESTAMP` are not interchangeable in precision or time zone behavior.

A realistic Oracle-to-Aurora PostgreSQL migration often requires:
- SQL rewrite.
- PL/SQL rewrite.
- Data type mapping review.
- Function and procedure redesign.
- Replacement of Oracle-specific features with PostgreSQL equivalents or application logic.

Example of a simple Oracle sequence and table definition that still needs careful type review:

```sql
CREATE SEQUENCE order_seq START WITH 1 INCREMENT BY 1;

CREATE TABLE orders (
  order_id NUMBER PRIMARY KEY,
  created_at DATE DEFAULT SYSDATE
);
```

A target-side design may need to change both object syntax and semantics:

```sql
CREATE SEQUENCE order_seq START 1 INCREMENT 1;

CREATE TABLE orders (
  order_id BIGINT PRIMARY KEY,
  created_at TIMESTAMP WITHOUT TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

That example is not a trivial conversion because the timestamp behavior and default value semantics differ.

**SQL Server migration considerations**

SQL Server workloads often migrate to Amazon RDS for SQL Server, Amazon EC2, or Aurora PostgreSQL depending on the goal.

Common SQL Server migration issues:
- **T-SQL** syntax differences.
- **SQL Server Agent** jobs need replacement or redesign.
- **CLR** (Common Language Runtime) assemblies are not portable to many targets.
- **Linked servers** often require architectural redesign.
- **SSIS/SSRS/SSAS** dependencies may force a different target or a separate service.
- **Stored procedures** may convert partially, but complex procedural logic often needs manual work.

Target selection guidance:
- Use **RDS for SQL Server** when you want managed SQL Server with minimal application change.
- Use **EC2** when you need unsupported features, custom agents, or OS-level control.
- Use **Aurora PostgreSQL** only when you are willing to rewrite SQL Server-specific logic and accept compatibility gaps.

> A managed SQL Server target is not the same as a pure rehost in the strict sense because the operating model changes. On the exam, focus on the business goal: minimal app change, managed operations, or full compatibility.

**MySQL and MariaDB migration considerations**

MySQL and MariaDB are often easier to migrate than Oracle or SQL Server, but there are still important differences.

Typical patterns:
- **MySQL to Amazon RDS for MySQL** or **Aurora MySQL** is often a low-friction move.
- **MariaDB to Amazon RDS for MariaDB** or **Aurora MySQL** may require feature review.
- Cross-engine moves between MySQL and MariaDB can still expose differences in storage engines, SQL modes, and function behavior.

Things to check:
- Character set and collation.
- `AUTO_INCREMENT` behavior.
- Trigger and stored procedure syntax.
- JSON support differences.
- Replication and binlog settings.
- Engine-specific functions and SQL modes.

Example of a simple MySQL table that may need only modest adjustment:

```sql
CREATE TABLE customers (
  customer_id BIGINT AUTO_INCREMENT PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

Even here, you should validate:
- Default timestamp behavior.
- Time zone handling.
- Index length limits.
- Collation compatibility.

**PostgreSQL migration considerations**

PostgreSQL-to-PostgreSQL migrations are usually homogeneous and often need less schema conversion than heterogeneous migrations. You may still need planning for version differences, extensions, and operational changes.

Common concerns:
- Extension availability on the target.
- `search_path` behavior.
- Sequence ownership and identity columns.
- Function volatility and language support.
- Partitioning syntax differences across versions.

If you are moving PostgreSQL to Aurora PostgreSQL or RDS for PostgreSQL, the main work is often:
- Compatibility validation.
- Extension review.
- Performance testing.
- Cutover planning.

**Unsupported or partially supported scenarios**

Exam questions often test whether you recognize when a target is not appropriate.

Common limitations include:
- Oracle **packages** and advanced PL/SQL constructs.
- SQL Server **Agent**, **CLR**, and **linked servers**.
- PostgreSQL **extensions** that are not available on the chosen managed target.
- Proprietary features that require OS-level access or custom binaries.
- Cross-engine features that depend on external services or local file access.

If the workload depends on these features, you may need:
- RDS Custom.
- EC2.
- Application redesign.
- A different target engine.
- A phased migration with feature replacement.

**Practical interpretation of SCT conversion output**

SCT usually classifies objects into categories such as:
- Automatically converted.
- Converted with warnings.
- Requires manual conversion.
- Not supported.

Use that output to prioritize:
- High-risk objects first, such as packages, procedures, and triggers.
- Shared dependencies next, such as views and foreign keys.
- Data type and default value issues after that.
- Low-risk DDL last.

A practical review checklist:
- Confirm primary keys and foreign keys are preserved.
- Verify triggers still fire in the correct order.
- Check sequence or identity generation.
- Review default values and nullability.
- Validate indexes and partitioning strategy.
- Test application queries against the target.

**Cutover and post-migration validation**

Schema conversion is only one part of the migration. Cutover often exposes issues that the assessment cannot predict.

Important cutover tasks:
- Synchronize sequences or identity values.
- Freeze writes during final sync if needed.
- Reconcile auto-generated keys.
- Rebuild or validate indexes if the target requires it.
- Recreate grants, roles, and permissions.
- Verify triggers, constraints, and foreign keys.
- Run application smoke tests and reconciliation queries.

Example validation queries for a relational target:

```sql
-- Check row counts
SELECT COUNT(*) FROM orders;
SELECT COUNT(*) FROM customers;

-- Check for orphaned rows
SELECT o.order_id
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id
WHERE c.customer_id IS NULL;
```

If you use identity columns or sequences, verify the next generated value before opening the system to writes:

```sql
-- PostgreSQL example
SELECT last_value, is_called FROM order_seq;
```

**AWS-native alternatives and adjacent services**

SCT is not the only migration tool in the AWS ecosystem. Depending on the scenario, you may also use:
- **AWS Database Migration Service (AWS DMS)** for data replication and ongoing sync.
- **AWS DMS Schema Conversion** support for some scenarios, though SCT remains the primary schema tool for heterogeneous migrations.
- **Amazon RDS Blue/Green Deployments** for supported engines and operational cutovers.
- **AWS Backup** and native snapshots for operational protection.

Use SCT for:
- Schema assessment.
- Object translation.
- Heterogeneous conversion planning.

Use DMS for:
- Data movement.
- Change data capture.
- Minimal downtime migration patterns.

**Exam decision cues**

Look for these phrases in questions:
- “Minimize application changes” often points to a managed target with compatible engine family.
- “Reduce operational overhead” often points to Aurora or RDS.
- “Need OS-level access” often points to EC2 or RDS Custom.
- “Unsupported database feature” often points away from standard RDS or Aurora.
- “Cross-engine migration” often implies SCT plus DMS.
- “High read scaling” often points to Aurora.
- “Keep the same engine” often points to homogeneous migration and less schema conversion.

**Quick comparison of migration fit**

| Scenario | Likely best fit | Why |
|---|---|---|
| Oracle to Aurora PostgreSQL with code rewrite | Aurora PostgreSQL | Cloud-native target, but requires substantial conversion |
| SQL Server with minimal app change | RDS for SQL Server | Managed SQL Server with low change |
| Legacy engine with custom OS dependencies | EC2 | Full control and compatibility |
| Need managed service but more control than RDS | RDS Custom | Middle ground |
| MySQL to Aurora MySQL | Aurora MySQL | Managed HA and scaling with low friction |
| PostgreSQL to RDS for PostgreSQL | RDS for PostgreSQL | Homogeneous migration with low conversion effort |

**Key takeaways**

- AWS SCT is primarily for **heterogeneous schema conversion** and assessment.
- SCT helps you estimate effort, but it does not prove runtime correctness.
- Target selection depends on compatibility, HA, read scaling, licensing, and operational responsibility.
- Aurora is often the best cloud-native target when compatibility is acceptable.
- RDS is a strong choice for managed, lower-change migrations.
- RDS Custom and EC2 are the fallback options when you need more control or unsupported features.
- Oracle and SQL Server migrations often require significant manual work beyond automated conversion.
- Cutover success depends on validating sequences, identity columns, constraints, triggers, and application behavior.
