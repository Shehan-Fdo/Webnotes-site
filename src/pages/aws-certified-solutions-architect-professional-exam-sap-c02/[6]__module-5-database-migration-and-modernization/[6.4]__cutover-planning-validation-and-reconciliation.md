## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Cutover planning overview**

Cutover is the controlled transition from a source environment to a target environment. In AWS migration and modernization scenarios, you use cutover planning to reduce downtime, preserve data integrity, and make rollback possible if validation fails.

A strong cutover plan answers four questions:

- What changes at the moment of switchover?
- How do you verify the target is ready?
- How do you confirm users and applications are actually using the target?
- How do you recover if the cutover fails?

For SAP-C02, focus on operational safety, dependency mapping, and validation evidence rather than just “moving traffic.”

**Core cutover goals**

- **Minimize downtime** by pre-staging infrastructure, data, and configuration.
- **Protect consistency** by defining a freeze window and final sync method.
- **Reduce blast radius** by switching one application, tenant, or region at a time when possible.
- **Preserve rollback** with reversible DNS, load balancer, or routing changes.
- **Prove success** with application, infrastructure, and business validation.

**Common cutover patterns**

- **Big-bang cutover**
  - You switch all users and workloads at once.
  - Use when the application cannot run in parallel or the business accepts a short outage.
  - Requires the strongest rollback plan.

- **Phased cutover**
  - You migrate in waves by account, region, business unit, or user cohort.
  - Use when you want to limit risk and validate incrementally.
  - Often paired with feature flags and weighted routing.

- **Blue/green cutover**
  - You keep two environments and shift traffic from blue to green.
  - Use when you need fast rollback and controlled validation.
  - Works well with Route 53, Application Load Balancer, or deployment tooling.

- **Canary cutover**
  - You send a small portion of traffic to the target first.
  - Use when you want to observe error rates, latency, and business KPIs before full release.
  - Best for application changes and some migration transitions.

**Planning inputs you should gather**

- Application dependencies, including upstream and downstream services.
- Data stores, replication lag, and write paths.
- DNS records, TTLs, and resolver behavior.
- Load balancer listeners, target groups, and health checks.
- IAM permissions, secrets, certificates, and KMS keys.
- Monitoring dashboards, alarms, and log locations.
- Business acceptance criteria and rollback triggers.

**Cutover runbook structure**

A runbook should be executable by an on-call engineer under pressure. Keep it time-based and unambiguous.

1. Confirm change window, approvers, and communication channels.
2. Freeze nonessential changes in source and target environments.
3. Verify target health, capacity, and dependencies.
4. Perform final data synchronization.
5. Switch traffic or endpoints.
6. Validate application behavior and data consistency.
7. Monitor for regressions.
8. Roll back if thresholds are exceeded.
9. Document results and reconcile differences.

**Traffic shifting mechanisms**

**DNS-based cutover**

Route 53 is commonly used to move traffic by changing records or using routing policies.

- **Weighted routing**
  - You assign relative weights to records.
  - Use it to gradually shift traffic between environments.
  - Important: this is not a per-request guarantee. DNS responses are probabilistic, and resolver caching can delay or skew the observed distribution.
  - Lower TTL values can help, but they do not eliminate caching effects.

```json
{
  "Comment": "Shift a small portion of traffic to green",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "blue",
        "Weight": 90,
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack.blue-alb-123456.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "green",
        "Weight": 10,
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack.green-alb-abcdef.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

- **Failover routing**
  - You route to a primary endpoint and fail over to a secondary when health checks fail.
  - Use it for disaster recovery or active/passive cutovers.
  - It is not a gradual migration tool.

- **Latency or geolocation routing**
  - You route users based on geography or latency.
  - Use it when regional placement matters.
  - Validate carefully because user experience can vary by resolver location.

> DNS cutovers are often delayed by recursive resolver caching, client-side caching, and application connection reuse. A low TTL helps, but it does not force immediate global propagation.

**Load balancer-based cutover**

For AWS Elastic Load Balancing, the common control points are:

- **Listeners**
- **Rules**
- **Target groups**

You typically cut over by changing listener rules, swapping target groups, or updating weighted forwarding actions on an Application Load Balancer.

```bash
aws elbv2 modify-listener \
  --listener-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:listener/app/my-alb/50dc6c495c0c9188/9f49f7c1c8c9f1a2 \
  --default-actions Type=forward,ForwardConfig='{"TargetGroups":[{"TargetGroupArn":"arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/blue-tg/73e2d6bc24d8a067","Weight":90},{"TargetGroupArn":"arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/green-tg/8c1d2e3f4a5b6c7d","Weight":10}]}'
```

Use load balancer cutover when you want:

- Faster control than DNS
- Health-checked traffic shifting
- Easier rollback by restoring the previous listener action

**Application-level cutover**

Sometimes the safest cutover is not network-based. You may switch:

- Feature flags
- API endpoints
- Database connection strings
- Queue consumers
- Batch job schedules

This is useful when the application can support dual-running or when only a subset of functionality moves first.

**Validation strategy**

Validation should happen before, during, and after cutover. Treat it as a layered process.

**Pre-cutover validation**

Before switching traffic, confirm:

- Infrastructure is deployed correctly.
- Security groups, NACLs, and IAM policies allow required access.
- Certificates are valid and attached.
- Secrets and parameters are present.
- Data replication is current.
- Health checks pass.
- Scaling policies are in place.
- Monitoring and alerting are active.

**Functional validation**

Check that the application behaves correctly in the target environment.

- Log in and complete a representative user journey.
- Create, read, update, and delete test records if safe.
- Verify integrations with downstream systems.
- Confirm background jobs, queues, and scheduled tasks.
- Test error handling and retries.

**Nonfunctional validation**

Validate operational characteristics, not just correctness.

- **Latency**: response times remain within acceptable thresholds.
- **Throughput**: the target handles expected load.
- **Availability**: health checks and failover work.
- **Security**: access controls, encryption, and audit logging remain intact.
- **Cost**: the target does not unexpectedly overprovision.

**Data validation**

Data checks are critical for migrations and cutovers.

- Compare record counts where appropriate.
- Verify recent transactions are present.
- Check replication lag before final sync.
- Validate checksums for files or objects when possible.
- Confirm schema versions and migrations completed successfully.

A practical reconciliation workflow often includes:

```bash
# Example: compare object inventories exported from two systems
sort source-manifest.txt > source.sorted
sort target-manifest.txt > target.sorted
diff -u source.sorted target.sorted
```

Use this only as a coarse comparison. For real reconciliation, you should validate more than filenames or object listings:

- Object metadata
- Versioning state
- Encryption settings
- Checksums or hashes
- Multipart upload behavior
- Application-level record integrity

**Monitoring during cutover**

Use dashboards and alarms to watch for regressions in real time.

Common signals:

- HTTP 5xx error rate
- p95 or p99 latency
- Target health check failures
- CPU, memory, and disk saturation
- Queue depth and consumer lag
- Database connections and replication lag
- Business metrics such as orders, logins, or payments

Example CloudWatch query for an Application Load Balancer metric:

```bash
aws cloudwatch get-metric-statistics \
  --namespace AWS/ApplicationELB \
  --metric-name HTTPCode_Target_5XX_Count \
  --dimensions Name=LoadBalancer,Value=app/my-alb/1234567890abcdef Name=TargetGroup,Value=targetgroup/my-tg/abcdef1234567890 \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T01:00:00Z \
  --period 60 \
  --statistics Sum
```

Use the correct dimensions for the metric you query. Some ALB metrics require both `LoadBalancer` and `TargetGroup`; others use only one dimension.

**Rollback planning**

Rollback must be defined before cutover starts. If you wait until failure, you are already behind.

A rollback plan should specify:

- The exact trigger threshold
- The person authorized to roll back
- The steps to restore the previous state
- The expected data loss or replay window
- How to communicate the rollback decision

Common rollback triggers:

- Elevated 5xx errors
- Authentication failures
- Data corruption or missing writes
- Latency beyond SLA
- Dependency failure in the target environment
- Unexpected cost or scaling behavior

Rollback methods depend on the cutover type:

- **DNS rollback**
  - Restore the previous record set.
  - Expect delayed effect because of caching.

- **Load balancer rollback**
  - Repoint listeners or rules to the previous target group.
  - Usually faster than DNS rollback.

- **Database rollback**
  - Revert to the source database or promote the previous primary.
  - Be careful with write divergence and replication lag.

- **Application rollback**
  - Re-enable the previous version or feature flag.
  - Useful when infrastructure is unchanged.

> If writes occurred on both source and target after cutover, rollback may require conflict resolution or replay. Do not assume a simple endpoint switch restores consistency.

**Reconciliation after cutover**

Reconciliation is the process of proving that source and target states match closely enough for business acceptance.

**What to reconcile**

- User-visible data
- Transaction counts
- File/object inventories
- Configuration values
- Access policies
- Logs and audit trails
- Scheduled jobs and event sources

**How to reconcile safely**

1. Define the reconciliation scope before cutover.
2. Capture a source snapshot or export at a known time.
3. Capture the target state after final sync.
4. Compare using meaningful keys and business rules.
5. Investigate exceptions, then classify them as expected or defects.
6. Obtain sign-off from application and business owners.

**Practical AWS examples**

- **S3 migration**
  - Use S3 Inventory, object versioning, and checksums where possible.
  - Do not rely only on `aws s3 ls` output for validation.
  - Compare manifests, metadata, and application-level counts.

- **Database migration**
  - Compare row counts, key ranges, and replication lag.
  - Validate stored procedures, indexes, and parameter groups.
  - Use application smoke tests after promotion.

- **EC2 or container cutover**
  - Compare AMI or image versions, user data, environment variables, and secrets.
  - Validate autoscaling, health checks, and service discovery.

**Validation and reconciliation checklist**

- Confirm DNS TTLs and cutover timing.
- Verify target health before traffic shift.
- Freeze source writes if required.
- Perform final sync or replication catch-up.
- Shift traffic in controlled increments.
- Watch alarms and logs continuously.
- Validate business transactions end to end.
- Reconcile data and configuration differences.
- Document exceptions and approvals.
- Keep rollback available until acceptance is complete.

**Operational examples**

**Route 53 TTL planning**

Lower TTL values reduce the time resolvers may cache old answers, but they increase query volume.

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch file://weighted-cutover.json
```

Use this to change records, not to prove what external clients currently resolve. To validate live behavior, test from multiple networks or use application telemetry.

**Health check validation**

```bash
curl -fsS https://app.example.com/health
curl -fsS https://app.example.com/ready
```

A health endpoint should reflect real dependency readiness, not just process liveness. If it always returns success, it can hide cutover failures.

**Smoke test example**

```bash
#!/usr/bin/env bash
set -euo pipefail

base_url="https://app.example.com"

curl -fsS "$base_url/health" >/dev/null
curl -fsS "$base_url/login" >/dev/null
curl -fsS "$base_url/api/orders?limit=1" >/dev/null

echo "Smoke tests passed"
```

**Exam focus points**

- **Cutover planning** is about controlled change, not just deployment.
- **Validation** must include infrastructure, application, and business checks.
- **Reconciliation** proves data and state consistency after the move.
- **Rollback** should be designed before traffic shifts.
- **DNS cutover** is affected by caching and is not instantly deterministic.
- **Weighted routing** is probabilistic and best used for gradual exposure.
- **Load balancer cutover** in AWS centers on listeners, rules, and target groups.
- **Reconciliation** requires more than file listings or simple counts when data integrity matters.

**Common pitfalls**

- Assuming DNS changes take effect immediately everywhere.
- Using a single health check as proof of full readiness.
- Comparing only object names instead of metadata and content.
- Forgetting to validate downstream integrations.
- Cutting over without a tested rollback path.
- Allowing source and target writes to diverge without a conflict strategy.
- Treating traffic shift as success before business metrics confirm it.

**Quick mental model**

Think of cutover in three phases:

- **Prepare** the target and freeze the source.
- **Switch** traffic with a reversible mechanism.
- **Prove** correctness through validation and reconciliation.

If any phase is weak, the migration is not complete.
