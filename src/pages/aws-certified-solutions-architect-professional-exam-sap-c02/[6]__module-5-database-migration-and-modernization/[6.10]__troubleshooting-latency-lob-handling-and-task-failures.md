## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Latency troubleshooting: start with the path, not the symptom**

Latency usually comes from one of four places: client/network path, edge or load balancer behavior, application processing, or downstream dependencies such as databases and third-party APIs. The fastest way to isolate the issue is to measure each hop and compare where time is spent.

- **Client-to-edge** latency often points to DNS, routing, or geographic distance.
- **Load balancer** latency can reflect TLS negotiation, connection reuse, target health, or request upload time.
- **Application** latency often comes from CPU saturation, thread starvation, cold starts, or inefficient code paths.
- **Downstream** latency usually shows up as database waits, lock contention, connection pool exhaustion, or slow remote calls.

A practical workflow is to correlate request IDs across logs, metrics, and traces. Use AWS X-Ray or OpenTelemetry where possible, then compare application timing with infrastructure metrics.

```bash
# Example: inspect ALB access logs for timing clues
aws s3 cp s3://my-alb-logs/AWSLogs/123456789012/elasticloadbalancing/us-east-1/2026/04/26/ \
  ./alb-logs/ --recursive

# Example: search for slow requests
zgrep ' 504 ' ./alb-logs/* | head
zgrep ' 500 ' ./alb-logs/* | head
```

**Load balancer timing and what it means**

Application Load Balancer (ALB) access logs are useful because they separate different timing phases. The key fields are often interpreted incorrectly, so read them carefully.

- **request_processing_time** is the time from when the load balancer receives the request until it sends the request to the target.
  - For large uploads, this can include client upload time.
  - It is not just front-end overhead.
- **target_processing_time** is the time the target spends processing the request.
  - High values usually point to application or dependency latency.
- **response_processing_time** is the time from when the target sends the response until the load balancer sends it to the client.
  - Large responses, slow client links, or buffering can affect this.

> Do not assume ALB itself is the payload bottleneck. Large request bodies can increase end-to-end latency, but the more common constraints are downstream target behavior, client timeouts, or application design.

Useful ALB troubleshooting checks:

- Confirm target health and deregistration delay behavior.
- Compare latency across Availability Zones to detect cross-AZ imbalance.
- Check whether HTTP keep-alive and connection reuse are working as expected.
- Review target response codes versus ALB-generated codes such as `502`, `503`, and `504`.

**Network-path latency: isolate DNS, routing, and cross-boundary hops**

When latency appears before the application even starts processing, inspect the network path. This is especially important for multi-VPC, hybrid, and multi-Region architectures.

- **Route 53 / DNS**
  - Slow or inconsistent DNS resolution can add visible delay before connection setup.
  - Check resolver behavior, TTLs, split-horizon DNS, and private hosted zones.
  - Validate that clients are not repeatedly resolving through a distant or overloaded resolver.
- **VPC Flow Logs**
  - Use them to confirm whether traffic is accepted, rejected, or asymmetrically routed.
  - They help identify security group or network ACL issues that manifest as retries and timeouts.
- **NAT Gateway**
  - Can become a bottleneck for internet-bound private subnet traffic.
  - Watch for port exhaustion, connection spikes, and unexpected dependency on a single NAT path.
- **Transit Gateway**
  - Useful for centralized routing, but route propagation and attachment design can create unexpected latency or hairpinning.
- **Cross-AZ and cross-Region traffic**
  - Cross-AZ traffic adds latency and cost.
  - Cross-Region traffic adds more latency and is more sensitive to packet loss and retransmits.

A simple path-checking approach:

```bash
# DNS lookup timing
time dig api.example.com

# Basic connectivity and TLS handshake timing
time curl -o /dev/null -s -w \
'lookup:%{time_namelookup} connect:%{time_connect} appconnect:%{time_appconnect} starttransfer:%{time_starttransfer} total:%{time_total}\n' \
https://api.example.com/health
```

**API Gateway latency and large payload troubleshooting**

API Gateway often appears in exam scenarios where latency is caused by integration design rather than the gateway itself. Distinguish between front-door latency and backend latency.

- **Integration latency** measures time spent calling the backend.
  - High integration latency usually means the backend is slow.
- **Mapping templates** can add transformation overhead.
  - Complex request/response mapping can increase processing time and complicate debugging.
- **Direct-to-S3 upload patterns** are often better for large files.
  - Use presigned URLs or multipart upload when clients need to upload large objects.
  - This avoids pushing large payloads through API Gateway and your application tier.

Service limits and behavior matter in SAP-C02 scenarios:

| Service | Common limit or behavior | Troubleshooting implication |
|---|---:|---|
| API Gateway REST/HTTP API | Payload limits are much smaller than S3 upload patterns; large bodies are often a poor fit | Use presigned S3 uploads or multipart upload for large files |
| Lambda | Invocation payloads are limited; synchronous and asynchronous paths differ | Move large data out of the event and pass references instead |
| SQS | Message body is limited to 256 KB | Store large payloads elsewhere and send a pointer |
| SNS | Message size is limited to 256 KB | Use S3 or another store for large content |
| Step Functions | State input/output and execution history have size constraints | Pass references, not large blobs |

**Lambda latency and failure remediation**

Lambda latency often comes from cold starts, downstream calls, or concurrency limits. The right fix depends on whether the issue is sporadic, sustained, or burst-driven.

- **Reserved concurrency**
  - Guarantees capacity for a function.
  - Also caps the function so it cannot consume all account concurrency.
- **Provisioned concurrency**
  - Pre-initializes execution environments to reduce cold starts.
  - Useful for latency-sensitive synchronous workloads.
- **Timeout tuning**
  - Set the timeout above normal p95/p99 execution time, but below the point where retries become too expensive.
  - A timeout that is too low causes avoidable retries and duplicate work.
- **Memory tuning**
  - More memory also gives more CPU.
  - Increasing memory can reduce duration and sometimes lower cost.
- **Event source mapping settings**
  - Batch size affects throughput and failure blast radius.
  - Maximum batching window trades latency for efficiency.
  - Partial batch response support can reduce reprocessing for stream and queue consumers.

Example Lambda configuration:

```yaml
Resources:
  ProcessorFunction:
    Type: AWS::Lambda::Function
    Properties:
      Runtime: python3.12
      Handler: app.handler
      Timeout: 30
      MemorySize: 1024
      ReservedConcurrentExecutions: 50
      Environment:
        Variables:
          TABLE_NAME: !Ref OrdersTable
```

For stream and queue consumers, tune the event source mapping carefully:

```yaml
Resources:
  QueueMapping:
    Type: AWS::Lambda::EventSourceMapping
    Properties:
      EventSourceArn: !GetAtt WorkQueue.Arn
      FunctionName: !Ref ProcessorFunction
      BatchSize: 10
      MaximumBatchingWindowInSeconds: 5
      FunctionResponseTypes:
        - ReportBatchItemFailures
```

**Task failures in Step Functions and distributed workflows**

Task failures are often not random; they are usually the result of retries, timeouts, bad payload design, or non-idempotent side effects. In Step Functions, understand how the service integration shapes the output.

- With `arn:aws:states:::lambda:invoke`, the Lambda result is wrapped in a response object.
  - The payload is not passed through as a raw Lambda return value unless you reshape it.
  - Use `OutputPath`, `ResultSelector`, or `Parameters` to control what the next state receives.
- Use `Retry` for transient failures such as throttling or temporary network issues.
- Use `Catch` for terminal failures and compensating actions.
- Use `TimeoutSeconds` and `HeartbeatSeconds` to detect hung work.

Example state machine pattern:

```json
{
  "StartAt": "InvokeWorker",
  "States": {
    "InvokeWorker": {
      "Type": "Task",
      "Resource": "arn:aws:states:::lambda:invoke",
      "Parameters": {
        "FunctionName": "arn:aws:lambda:us-east-1:123456789012:function:Worker",
        "Payload.$": "$"
      },
      "ResultSelector": {
        "result.$": "$.Payload"
      },
      "OutputPath": "$.result",
      "Retry": [
        {
          "ErrorEquals": ["Lambda.TooManyRequestsException", "States.Timeout"],
          "IntervalSeconds": 2,
          "MaxAttempts": 3,
          "BackoffRate": 2.0
        }
      ],
      "Catch": [
        {
          "ErrorEquals": ["States.ALL"],
          "Next": "Compensate"
        }
      ],
      "Next": "Done"
    },
    "Compensate": {
      "Type": "Pass",
      "End": true
    },
    "Done": {
      "Type": "Succeed"
    }
  }
}
```

**Idempotency and deduplication: make retries safe**

Retries are only safe when the operation can be repeated without creating duplicate side effects. This is critical for queues, workflows, and task reprocessing.

- **Idempotency** means the same request can be applied multiple times with the same final result.
- **Deduplication** means you detect and ignore repeated work items.
- Use a stable business key such as `orderId`, `paymentId`, or `jobId`.
- Store processed keys in DynamoDB with conditional writes when you need strong duplicate protection.
- Use SQS FIFO queues when ordered, deduplicated processing is required.
- Design compensating transactions for workflows that cannot be made fully idempotent.

Example DynamoDB conditional write:

```python
import boto3
from botocore.exceptions import ClientError

ddb = boto3.client("dynamodb")

def mark_processed(job_id):
    try:
        ddb.put_item(
            TableName="ProcessedJobs",
            Item={"jobId": {"S": job_id}},
            ConditionExpression="attribute_not_exists(jobId)"
        )
        return True
    except ClientError as e:
        if e.response["Error"]["Code"] == "ConditionalCheckFailedException":
            return False
        raise
```

**Large object handling: do not force blobs through the wrong service**

LOB usually means large object, such as images, PDFs, videos, or large JSON/XML payloads. The main design goal is to keep hot paths small and move bulky data to storage that fits the access pattern.

- **S3** is the default choice for large, durable objects.
  - Best for object storage, downloads, archival, and presigned access.
- **EFS** is useful when multiple compute instances need shared file semantics.
  - Good for POSIX-style access, shared libraries, or file-based workflows.
- **DynamoDB** can store metadata and small-to-medium items with low-latency key-value access.
  - Good for lookup-heavy patterns, not for large binary blobs.
- **RDS/Aurora** can store BLOBs, but this is often a poor fit for very large objects.
  - It can increase backup size, replication overhead, and query latency.

A common pattern is to store the object in S3 and the metadata in DynamoDB or Aurora.

```text
Client -> API -> metadata record in DynamoDB
                 -> object upload to S3 using presigned URL
                 -> async processing via SQS / EventBridge / Step Functions
```

Example presigned upload flow:

```bash
# Generate a presigned PUT URL with the SDK or application code
aws s3 presign s3://my-bucket/uploads/report.pdf --expires-in 900
```

> `aws s3 presign` is commonly used for `get-object` style access. For uploads, prefer a presigned `PUT` URL generated by the SDK or a purpose-built application flow.

Example Python presigned PUT:

```python
import boto3

s3 = boto3.client("s3")
url = s3.generate_presigned_url(
    ClientMethod="put_object",
    Params={"Bucket": "my-bucket", "Key": "uploads/report.pdf"},
    ExpiresIn=900
)
print(url)
```

**Database performance troubleshooting: RDS and Aurora**

For relational databases, latency often comes from query plans, locks, connection pressure, or storage bottlenecks. SAP-C02 questions often expect you to choose the right observability tool.

- **Performance Insights**
  - Shows database load and top SQL.
  - Helps identify whether CPU, I/O, or waits dominate.
- **Enhanced Monitoring**
  - Provides OS-level metrics such as CPU, memory, disk, and process activity.
  - Useful when the issue is below the SQL layer.
- **Slow query logs**
  - Show queries exceeding a threshold.
  - Useful for identifying inefficient statements and missing indexes.
- **Lock waits**
  - Indicate blocking transactions or contention on hot rows/tables.
  - Often appear as latency spikes under concurrency.
- **Connection pool exhaustion**
  - Happens when the application opens too many connections or fails to reuse them.
  - Symptoms include timeouts, queueing, and sudden latency under load.

Troubleshooting sequence:

1. Check whether the database is CPU-bound, I/O-bound, or wait-bound.
2. Review top SQL in Performance Insights.
3. Inspect slow query logs for repeated offenders.
4. Look for lock waits and long transactions.
5. Verify application connection pooling and max connection settings.
6. Scale read traffic with read replicas or reduce contention with schema/query changes.

Example MySQL checks:

```sql
SHOW FULL PROCESSLIST;
SHOW ENGINE INNODB STATUS\G
SELECT * FROM information_schema.innodb_trx\G
```

Example PostgreSQL checks:

```sql
SELECT pid, state, wait_event_type, wait_event, query
FROM pg_stat_activity
WHERE state <> 'idle';

SELECT * FROM pg_locks;
```

**Observability: trace the transaction end to end**

Distributed systems fail in ways that are hard to see from a single log line. Use correlation IDs and tracing to follow one request across services.

- **X-Ray trace IDs** let you connect API Gateway, Lambda, ECS, and downstream calls.
- **Request IDs** in logs help correlate a specific invocation or request.
- **Structured logging** makes filtering and aggregation easier.
- **CloudWatch metrics and alarms** show whether the problem is isolated or systemic.

A useful logging pattern is to emit the same correlation ID at every hop.

```python
import json
import logging

logger = logging.getLogger()
logger.setLevel(logging.INFO)

def handler(event, context):
    correlation_id = event.get("headers", {}).get("x-correlation-id", context.aws_request_id)
    logger.info(json.dumps({
        "correlation_id": correlation_id,
        "aws_request_id": context.aws_request_id,
        "path": event.get("path")
    }))
    return {"statusCode": 200, "body": "ok"}
```

**Common failure patterns and what to check first**

- **504 Gateway Timeout**
  - Check backend duration, ALB target response time, and API Gateway integration latency.
  - Verify Lambda timeout and downstream dependency latency.
- **429 Too Many Requests**
  - Check Lambda concurrency, API throttling, SQS consumer scaling, or database connection limits.
- **Duplicate processing**
  - Check retries, at-least-once delivery, and missing idempotency keys.
- **Slow uploads**
  - Check client network, TLS handshake, request body size, and whether the architecture is forcing large payloads through a synchronous path.
- **Intermittent database slowness**
  - Check lock waits, connection pool exhaustion, and burst traffic patterns.

**Practical architecture choices for exam scenarios**

Choose the simplest design that matches the access pattern and failure mode.

- Use **S3 + presigned URLs** for large uploads and downloads.
- Use **SQS** to decouple slow or bursty work from synchronous requests.
- Use **Step Functions** for orchestration, retries, and compensating actions.
- Use **DynamoDB** for low-latency key-value metadata and deduplication state.
- Use **EFS** when multiple compute nodes need shared file access.
- Use **RDS/Aurora** when relational queries and transactions are required, but monitor waits and connection pressure carefully.

A good exam answer usually explains both the symptom and the control point:
- reduce payload size on the synchronous path,
- add buffering or asynchronous processing,
- tune concurrency and timeouts,
- and make retries safe with idempotency.
