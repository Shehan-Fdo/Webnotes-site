## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Troubleshooting mindset for SAP-C02**

Troubleshooting in AWS is usually a process of narrowing the failure domain: identity, network, service control, configuration drift, data plane, or dependency health. For SAP-C02, you are expected to identify the most likely root cause quickly and choose a remediation path that is safe, reversible, and operationally realistic.

A strong workflow is:

1. Confirm the symptom and scope.
2. Check whether the issue is customer-side or an AWS service event.
3. Inspect control-plane evidence such as CloudTrail and AWS Config.
4. Validate permissions, network reachability, and service-specific health.
5. Apply the smallest safe remediation.
6. Verify recovery and prevent recurrence.

**First-line triage tools**

Use AWS-native observability and support signals before changing infrastructure. This reduces guesswork and helps you distinguish transient service issues from misconfiguration.

- **AWS Health Dashboard / Personal Health Dashboard** tells you whether an AWS service event is affecting your account or Region.
- **CloudWatch** provides metrics, alarms, logs, and dashboards for symptom detection.
- **CloudTrail** records API activity and is essential for root-cause analysis of control-plane changes.
- **AWS Config** helps detect configuration drift and shows when a resource changed from a known-good state.
- **Systems Manager** provides operational access and remediation workflows without opening SSH or RDP.

> Use the Health Dashboard first when the failure appears broad or sudden. If AWS reports no incident, focus on your own configuration, permissions, and dependencies.

**AWS Systems Manager for remediation**

AWS Systems Manager (SSM) is a core remediation toolkit in SAP-C02 scenarios. It is especially useful when instances are unreachable, when you need repeatable fixes, or when you want to avoid direct host access.

- **Session Manager** gives shell access to managed instances without inbound SSH or bastions.
- **Run Command** executes commands across fleets for diagnostics or repair.
- **Automation** runs predefined runbooks for common remediation tasks.
- **OpsCenter** centralizes operational issues and links them to incidents, runbooks, and approvals.

Typical uses:

- Restart a failed service on a fleet of EC2 instances.
- Collect logs from isolated instances.
- Patch or reconfigure instances without opening inbound ports.
- Orchestrate a multi-step recovery after a failed deployment.

Example Session Manager access:

```bash
aws ssm start-session --target i-0123456789abcdef0
```

Example Run Command to inspect a service:

```bash
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters 'commands=["systemctl status nginx","journalctl -u nginx --no-pager -n 50"]'
```

Example Automation invocation:

```bash
aws ssm start-automation-execution \
  --document-name "AWS-StopEC2Instance" \
  --parameters InstanceId=i-0123456789abcdef0
```

**IAM troubleshooting and authorization failures**

Many AWS failures are authorization failures disguised as service errors. You should verify both identity-based and resource-based permissions, plus any permission boundaries, SCPs (Service Control Policies), and session policies.

Key tools and techniques:

- **IAM policy simulator** helps you test whether a principal can perform an action on a resource.
- **Access Analyzer** helps identify unintended access and policy issues.
- **CloudTrail** shows the exact denied API call and the principal that made it.
- **STS** (Security Token Service) session context matters when roles are assumed with session policies or external IDs.

Common causes:

- Missing `Allow` for the required action.
- Explicit `Deny` in an SCP, permission boundary, or resource policy.
- Incorrect resource ARN or region mismatch.
- KMS permissions missing for encrypted resources.
- Cross-account access missing trust or resource policy statements.

Example policy simulation:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject kms:Decrypt \
  --resource-arns arn:aws:s3:::my-bucket/path/object.txt arn:aws:kms:us-east-1:123456789012:key/abcd-1234
```

**CloudTrail and AWS Config for root cause and drift**

CloudTrail is your audit trail for who changed what and when. AWS Config is your drift detector for whether the current state matches expected configuration.

Use CloudTrail to answer:

- Which principal changed the security group?
- Was a bucket policy updated before access failures began?
- Did a deployment role call `UpdateStack` or `ModifyDBInstance`?
- Was a KMS key disabled or rotated?

Use AWS Config to answer:

- Did a resource drift from a compliant baseline?
- Was a security group opened too broadly?
- Did a bucket lose versioning or encryption?
- Did a load balancer or target group change unexpectedly?

Practical workflow:

1. Check the symptom time window.
2. Search CloudTrail for API calls around that time.
3. Compare the resource’s current state with AWS Config history.
4. Correlate the change with the first failed request or alarm.
5. Revert the change or apply a compensating fix.

**Network troubleshooting**

Network failures often look like application failures. Separate routing, security, DNS, and service endpoint issues before changing code.

Check these layers:

- **Security groups** allow stateful instance and load balancer traffic.
- **Network ACLs** (NACLs) can block ephemeral return traffic.
- **Route tables** must point to the correct target, NAT gateway, transit gateway, or peering connection.
- **DNS** resolution can fail because of Route 53 records, private hosted zones, or resolver rules.
- **VPC endpoints** can break if endpoint policies or DNS settings are wrong.

Useful commands:

```bash
aws ec2 describe-security-groups --group-ids sg-0123456789abcdef0
aws ec2 describe-route-tables --filters Name=vpc-id,Values=vpc-0123456789abcdef0
aws ec2 describe-network-acls --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

For host-level checks:

```bash
ip addr
ip route
ss -tulpn
curl -v https://example.internal
```

If traffic fails only across subnets or AZs, verify:

- Route propagation.
- NACL symmetry.
- Cross-zone load balancing behavior.
- Private DNS resolution.
- Endpoint and peering policies.

**Elastic Load Balancing troubleshooting**

For Elastic Load Balancing, target health is only one part of the problem. You also need to validate listeners, certificates, TLS negotiation, stickiness, cross-zone behavior, and access logs.

Common failure modes:

- **ALB** listener rules send traffic to the wrong target group.
- **ALB/NLB** certificates are expired, mismatched, or missing the correct Subject Alternative Name (SAN).
- **TLS** negotiation fails because of unsupported ciphers or protocol versions.
- **Stickiness** causes uneven distribution or session persistence issues.
- **Cross-zone load balancing** changes traffic distribution and can expose AZ-specific backend problems.
- **Access logs** reveal 4xx/5xx patterns, target response times, and connection errors.

Checks to perform:

- Confirm listener port, protocol, and default action.
- Validate certificate ARN and expiration.
- Inspect target group health checks and thresholds.
- Review access logs for `elb_status_code` and `target_status_code`.
- Verify whether cross-zone load balancing is enabled where expected.

Example CLI checks:

```bash
aws elbv2 describe-load-balancers
aws elbv2 describe-listeners --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/app/my-alb/50dc6c495c0c9188
aws elbv2 describe-target-health --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/my-tg/6d0ecf831eec9f09
```

**S3 troubleshooting**

Amazon S3 failures often come from policy layers, ownership settings, or object state rather than bucket availability.

Important failure modes:

- **Block Public Access** overrides public bucket or object policies.
- **Bucket ownership controls** can change who owns uploaded objects and affect ACL-based access.
- **Versioning** affects delete markers, rollback behavior, and object retrieval.
- **Lifecycle rules** can expire or transition objects unexpectedly.
- **Replication status** may lag or fail because of permissions, KMS, or destination configuration.
- **Access point policies** can deny access even when the bucket policy appears correct.

Troubleshooting checklist:

- Confirm the request path: bucket, access point, or Multi-Region Access Point.
- Check bucket policy, access point policy, and IAM permissions together.
- Verify Block Public Access at both account and bucket level.
- Inspect object ownership and ACL assumptions.
- Review replication metrics and status for cross-Region copies.
- Confirm the object is not a delete marker or expired by lifecycle.

Example checks:

```bash
aws s3api get-bucket-policy --bucket my-bucket
aws s3api get-public-access-block --bucket my-bucket
aws s3api get-bucket-versioning --bucket my-bucket
aws s3api get-bucket-replication --bucket my-bucket
aws s3api get-bucket-ownership-controls --bucket my-bucket
```

> If access suddenly changes after a migration, suspect ownership controls or Block Public Access before assuming an IAM regression.

**KMS troubleshooting**

AWS Key Management Service (KMS) issues usually appear as decrypt, encrypt, or grant failures on otherwise healthy services.

Common causes:

- The caller lacks `kms:Decrypt`, `kms:Encrypt`, `kms:GenerateDataKey`, or `kms:DescribeKey`.
- The key policy does not allow the principal or service.
- A grant is missing for a service-integrated workflow.
- Key rotation changed the backing key material, exposing stale assumptions in custom code or external systems.
- The key is disabled, pending deletion, or in the wrong Region.

Practical checks:

- Verify the key policy and IAM policy together.
- Confirm the service principal is allowed when AWS-managed integrations are involved.
- Check whether the resource uses a customer managed key or AWS managed key.
- Review CloudTrail for `AccessDeniedException` and KMS API calls.

Example:

```bash
aws kms describe-key --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234
aws kms list-grants --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234
aws kms get-key-policy --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234 --policy-name default
```

**RDS and Aurora troubleshooting**

Amazon RDS and Aurora issues often involve connectivity, parameter groups, subnet groups, failover behavior, read replicas, or connection limits.

Key areas to inspect:

- **DB subnet group** spans at least two AZs for Multi-AZ designs.
- **Parameter groups** can affect max connections, logging, SSL requirements, and engine behavior.
- **Security groups** must allow the client source and port.
- **Failover** can change the writer endpoint and briefly interrupt connections.
- **Read replicas** may lag and are not a substitute for synchronous failover unless the engine supports the required pattern.
- **Connection limits** can be reached due to application pooling issues or spikes.

Troubleshooting steps:

1. Confirm the endpoint type: writer, reader, or custom.
2. Check whether the instance is failing over or rebooting.
3. Verify subnet group and AZ placement.
4. Inspect parameter group changes for connection or timeout settings.
5. Review CloudWatch metrics such as CPU, FreeableMemory, DatabaseConnections, and ReplicaLag.
6. Validate application retry logic and connection pooling.

Example CLI:

```bash
aws rds describe-db-instances --db-instance-identifier mydb
aws rds describe-db-subnet-groups --db-subnet-group-name my-subnet-group
aws rds describe-db-parameter-groups
aws rds describe-events --source-type db-instance --source-identifier mydb
```

Aurora-specific considerations:

- Aurora cluster endpoints abstract the writer and readers, but application caching of endpoints can still cause issues.
- Failover promotes a new writer; clients must reconnect.
- Global Database introduces cross-Region replication lag and failover planning concerns.
- Backtrack and clone features can help recovery in some cases, but they are not substitutes for backups.

**CloudFormation and infrastructure-as-code failures**

Infrastructure-as-code failures are often caused by dependency ordering, missing permissions, invalid parameters, or resource replacement behavior.

What to inspect:

- **Stack events** show the exact resource that failed.
- **Rollback** may hide the original failure if you do not inspect events quickly.
- **Drift detection** reveals manual changes outside CloudFormation.
- **Dependencies** can require explicit `DependsOn` or correct creation order.
- **Change sets** help you preview replacements and deletions before deployment.

Common issues:

- A resource requires a value from another resource that is not yet created.
- A replacement update fails because the old resource cannot be deleted.
- A custom resource times out or returns an error.
- A nested stack fails and propagates rollback to the parent stack.
- A parameter or mapping resolves to an invalid value in one Region.

Example workflow:

1. Open stack events and identify the first failure.
2. Check whether the failure is a create, update, or delete action.
3. Review IAM permissions for the CloudFormation execution role.
4. Inspect drift and compare against the template.
5. Fix the template or dependency chain, then redeploy.

Example commands:

```bash
aws cloudformation describe-stack-events --stack-name my-stack
aws cloudformation detect-stack-drift --stack-name my-stack
aws cloudformation describe-stack-resource-drifts --stack-name my-stack
aws cloudformation create-change-set \
  --stack-name my-stack \
  --change-set-name preview-1 \
  --template-body file://template.yaml
```

**ECS and EKS operational troubleshooting**

Container platform issues often come from capacity, networking, IAM, or scheduling rather than the application image itself.

For Amazon ECS:

- Review **service events** for deployment failures and placement issues.
- Check cluster capacity, task definition compatibility, and target group health.
- Validate IAM roles for task execution and task role permissions.
- Inspect CloudWatch logs for container startup failures.

For Amazon EKS:

- Check **node group health** and cluster autoscaler behavior.
- Validate pod scheduling, taints, tolerations, and resource requests.
- Troubleshoot pod networking with CNI (Container Network Interface) logs and security groups.
- Verify IAM Roles for Service Accounts (IRSA) and OIDC provider configuration.

Useful commands:

```bash
aws ecs describe-services --cluster my-cluster --services my-service
aws ecs describe-tasks --cluster my-cluster --tasks arn:aws:ecs:us-east-1:123456789012:task/my-cluster/1234567890abcdef
kubectl get pods -A
kubectl describe pod my-pod
kubectl get events --sort-by=.metadata.creationTimestamp
```

EKS-specific failure patterns:

- Pods stay in `Pending` because of insufficient CPU, memory, or node selectors.
- Pods fail to pull images because of ECR permissions or network access.
- Service accounts cannot access AWS APIs because IRSA trust or annotation is wrong.
- Nodes join the cluster but fail readiness checks due to CNI, kubelet, or security group issues.

**Storage and data recovery scenarios**

Storage failures often require distinguishing between durability, availability, and access control.

Consider these patterns:

- EBS volume attachment failures can come from AZ mismatch, attachment limits, or stale device mappings.
- EFS mount failures can come from security groups, mount targets, DNS, or IAM authorization.
- FSx issues may involve throughput, client configuration, or AD integration.
- S3 object loss may be recoverable through versioning, replication, or backup, but only if those controls were enabled before the incident.

Practical checks:

- Confirm the resource is in the same AZ where required.
- Verify mount targets exist in the client’s AZ.
- Check file system policies and security groups.
- Review backup and snapshot status before attempting destructive fixes.

**Observability for troubleshooting**

CloudWatch, X-Ray, and logs help you understand symptoms, but they do not replace control-plane evidence.

Use them together:

- **CloudWatch metrics** identify when the issue started.
- **CloudWatch Logs** show application and system errors.
- **X-Ray** traces reveal latency hotspots and downstream failures.
- **CloudTrail** shows who changed the environment.
- **AWS Config** shows whether the environment drifted.

A practical pattern is to correlate:

- Alarm time
- First error log
- Recent API change
- Resource configuration drift
- Service health event

**Exam-style remediation patterns**

These scenarios are common in SAP-C02-style questions:

- An application cannot access S3 after a migration.
  - Check Block Public Access, bucket policy, object ownership, and KMS permissions.
- A database connection fails after a maintenance window.
  - Check failover, endpoint changes, parameter groups, and security groups.
- A deployment rolls back in CloudFormation.
  - Inspect stack events, dependency ordering, and execution role permissions.
- A fleet of instances is unreachable.
  - Use Session Manager, then Run Command to inspect services and logs.
- A load balancer returns 502 or 504 errors.
  - Check target health, listener rules, TLS certificates, and backend timeouts.
- An EKS service stops routing traffic.
  - Check pod readiness, node health, service selectors, and IAM/OIDC configuration.
- A KMS-encrypted workload fails after a key policy change.
  - Check grants, key policy, and decrypt permissions in CloudTrail.

**Safe remediation principles**

Prefer changes that are:

- Reversible.
- Scoped to the failing component.
- Logged and auditable.
- Tested in a lower environment when possible.
- Automated through Systems Manager or infrastructure-as-code.

Avoid:

- Blindly restarting everything.
- Deleting resources before checking drift and dependencies.
- Changing multiple variables at once.
- Assuming the application is broken before checking IAM, network, and service health.

**Quick diagnostic checklist**

1. Check AWS Health Dashboard for service incidents.
2. Review CloudWatch alarms, logs, and metrics.
3. Search CloudTrail for recent changes or denied actions.
4. Compare current state with AWS Config history.
5. Validate IAM, KMS, and resource policies.
6. Test network reachability and DNS.
7. Inspect service-specific health and events.
8. Remediate with the smallest safe change.
9. Verify recovery and add preventive controls.

**Practical command set**

```bash
# CloudTrail lookup
aws cloudtrail lookup-events --lookup-attributes AttributeKey=EventName,AttributeValue=ModifyDBInstance

# AWS Config resource history
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::SecurityGroup \
  --resource-id sg-0123456789abcdef0

# Systems Manager inventory and session
aws ssm describe-instance-information
aws ssm start-session --target i-0123456789abcdef0

# ELB target health
aws elbv2 describe-target-health --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/my-tg/6d0ecf831eec9f09

# RDS events
aws rds describe-events --source-type db-instance --source-identifier mydb
```

**What SAP-C02 expects you to recognize**

- The difference between a service outage and a configuration error.
- The right AWS tool for audit, drift, or live remediation.
- How permissions, encryption, and networking interact across services.
- How to restore service without creating a larger incident.
- When to use automation, and when to stop and investigate first.
