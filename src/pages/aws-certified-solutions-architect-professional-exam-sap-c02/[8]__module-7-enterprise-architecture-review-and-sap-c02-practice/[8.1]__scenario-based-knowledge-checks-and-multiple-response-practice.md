## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**How to approach SAP-C02 scenario questions**

SAP-C02 questions usually describe a business goal, constraints, and one or more hidden trade-offs. Your job is not to find a technically valid answer; it is to find the best answer for the stated requirements, especially around cost, operational overhead, security, resilience, and migration speed.

A strong workflow is:

1. Identify the primary objective.
2. Extract constraints such as compliance, latency, RTO, RPO, budget, and existing tooling.
3. Eliminate answers that solve the wrong problem.
4. Prefer managed services when they meet the requirement.
5. Choose the simplest design that satisfies all constraints.

Common exam signals include:
- “minimize operational overhead” → favor managed services.
- “lowest latency globally” → consider CloudFront or Global Accelerator.
- “private connectivity” → consider VPC endpoints, Direct Connect, VPN, or Transit Gateway.
- “cross-account” → consider IAM roles, resource policies, Organizations, or AWS RAM.
- “near-zero downtime” → consider active-active or warm standby depending on RTO/RPO.

**Multiple-response question strategy**

Multiple-response items are not about guessing a fixed number of answers. The question determines how many answers are correct, and the interface may indicate the count. Read every option carefully and treat each one as independently true or false.

Use this method:
- Eliminate obviously wrong choices first.
- Watch for answers that are technically true but do not satisfy the scenario.
- Look for distractors that solve only part of the problem.
- Compare similar services by their exact use case, not by broad category.

> A common trap is selecting a service that is “good” in general but not the best fit for the workload, security boundary, or recovery target described in the prompt.

**Core decision domains tested in SAP-C02**

SAP-C02 often blends several domains in one scenario. You should be comfortable with the following trade-offs:

- **Identity and access**
  - IAM roles for temporary access and service-to-service permissions.
  - Resource-based policies for direct cross-account access to a resource.
  - Permission boundaries to limit what identities can ever receive.
  - AWS Organizations and service control policies (SCPs) for guardrails across accounts.
- **Networking**
  - VPC endpoints for private service access.
  - NAT Gateway for outbound internet access from private subnets.
  - Transit Gateway for hub-and-spoke routing across many VPCs and on-premises networks.
  - Direct Connect and VPN for hybrid connectivity.
- **Storage**
  - S3 for object storage, lifecycle, replication, and event-driven integration.
  - EBS for block storage attached to a single EC2 instance.
  - EFS for shared POSIX file storage across Linux instances.
  - FSx variants for specialized file workloads.
- **Databases**
  - Aurora for high-performance relational workloads and scaling patterns.
  - RDS for managed relational databases with familiar engines.
  - DynamoDB for key-value and document workloads with predictable access patterns.
- **Resilience**
  - Multi-AZ for synchronous high availability.
  - Read replicas for read scaling and some disaster recovery patterns.
  - Pilot light, warm standby, and active-active for different RTO/RPO targets.
- **Security**
  - KMS for encryption key management.
  - WAF for Layer 7 filtering.
  - Shield for DDoS protection.
  - Secrets Manager or Parameter Store for secret handling.

**IAM and access control scenario patterns**

IAM is one of the most heavily tested areas because many architectures fail due to incorrect trust boundaries rather than missing infrastructure.

Use **IAM roles** when:
- An AWS service needs temporary permissions.
- An application on EC2, Lambda, ECS, or EKS needs to call AWS APIs.
- You want short-lived credentials instead of long-term access keys.

Use **resource-based policies** when:
- Another account, user, or service needs direct access to a resource.
- You want the resource itself to define who can access it.
- You need simpler cross-account sharing for services such as S3, KMS, SNS, SQS, or Lambda.

Use **permission boundaries** when:
- You want to cap the maximum permissions that developers or automation can receive.
- You need delegated IAM administration with guardrails.

Use **SCPs** when:
- You need organization-wide guardrails across member accounts.
- You want to restrict what identities in those accounts can do, regardless of local IAM policy.

Important nuance:
- SCPs do not apply to the management account.
- SCPs set the maximum permissions available in member accounts; they do not grant permissions by themselves.

Example cross-account role assumption:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::111122223333:root" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example permission boundary concept:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:*",
      "Resource": "*"
    }
  ]
}
```

That boundary does not grant access by itself; it only limits what attached identity policies can allow.

**Organizations, landing zones, and cross-account governance**

Professional-level questions often assume a multi-account landing zone rather than a single-account setup. You should recognize when the best answer is to separate workloads by account for blast-radius reduction, billing, and governance.

Common patterns:
- Use a **management account** for organization administration, not for workloads.
- Place shared services in a dedicated account.
- Isolate security tooling in a security account.
- Put logging and archive data in a log archive account.
- Use SCPs to enforce baseline controls such as region restrictions or service allow lists.
- Use AWS RAM to share supported resources across accounts without duplicating them.

Cross-account access patterns:
- Use IAM roles for human or workload access across accounts.
- Use resource policies for direct service access where supported.
- Use AWS Organizations for centralized governance and consolidated billing.
- Use AWS Control Tower when the scenario asks for a governed multi-account landing zone with guardrails and account vending.

Example landing zone decision:
- If the prompt asks for “standardized multi-account governance with guardrails,” choose Organizations plus SCPs, and consider Control Tower if the question emphasizes landing zone automation.
- If the prompt asks for “shared network services across accounts,” consider Transit Gateway, RAM sharing, and centralized networking.

**Networking scenario checks**

Networking questions often hinge on whether traffic is private, hybrid, internet-facing, or transitive across many networks.

**VPC endpoints**
- Use **gateway endpoints** for S3 and DynamoDB.
- Use **interface endpoints** for most other AWS services.
- Gateway endpoints are route-table based and do not use ENIs in your subnets.
- Interface endpoints use Elastic Network Interfaces (ENIs) and support private access to many AWS services.

Choose endpoints when:
- You want to keep traffic off the public internet.
- You want private access from private subnets to AWS services.
- You need to reduce NAT Gateway dependency for AWS API calls.

**NAT Gateway vs NAT instance**
- Use **NAT Gateway** for managed, highly available outbound internet access from private subnets.
- Use a **NAT instance** only when you need custom routing, packet inspection, or legacy control and can accept more operational overhead.

Typical exam clue:
- “Private instances need outbound internet for patching, but no inbound internet access” → NAT Gateway.
- “Need custom firewall rules on the NAT device” → NAT instance may be considered, but AWS Network Firewall is often the better modern answer.

**Transit Gateway**
- Use **Transit Gateway** to simplify hub-and-spoke routing across many VPCs and on-premises networks.
- It reduces route-table sprawl compared with many VPC peering connections.
- It is especially useful when the scenario includes multiple accounts, multiple VPCs, or hybrid connectivity.

**Direct Connect and VPN**
- Use **AWS Direct Connect** for dedicated private connectivity with predictable performance and lower jitter.
- Use **Site-to-Site VPN** for encrypted connectivity over the internet, often as a backup or quick-start hybrid link.
- Use both together when the scenario asks for resilience and private connectivity.

Direct Connect Gateway nuance:
- A Direct Connect Gateway lets you connect a Direct Connect connection to multiple VPCs across Regions through private virtual interfaces, subject to AWS design limits and association rules.
- It is not a universal multi-Region routing solution for every architecture.

**Inbound and outbound Route 53 Resolver endpoints**
- Use **inbound endpoints** so on-premises DNS resolvers can query Route 53 private hosted zones.
- Use **outbound endpoints** so resources in a VPC can resolve on-premises DNS names via forwarding rules.

Example forwarding rule concept:

```bash
aws route53resolver create-resolver-rule \
  --creator-request-id 1234abcd \
  --domain-name corp.example.com \
  --rule-type FORWARD \
  --resolver-endpoint-id rslvr-out-1234567890abcdef \
  --target-ips Ip=10.0.0.10,Port=53 Ip=10.0.0.11,Port=53
```

**AWS Network Firewall**
- Use AWS Network Firewall when you need managed stateful inspection, domain filtering, or centralized egress/ingress controls.
- It is often a better fit than a NAT instance when the requirement is security inspection rather than simple address translation.

**AWS Global Accelerator**
- Use Global Accelerator when the scenario asks for improved global performance, static anycast IPs, or fast failover to healthy endpoints.
- It is useful for TCP and UDP workloads and can outperform DNS-only failover for some latency-sensitive applications.

**Route 53 routing policy traps**

Route 53 questions often test whether you understand the difference between routing intent and actual user location.

- **Geolocation routing** uses the source location of the DNS query, not necessarily the user’s physical location.
- **Latency-based routing** sends users to the Region with the lowest network latency from the resolver’s perspective.
- **Failover routing** is for active-passive health-based routing.
- **Weighted routing** is for traffic splitting and gradual migration.
- **Simple routing** is for a single record set without advanced policy logic.

Practical example:
- If the prompt says “serve different content based on country,” geolocation routing is a candidate.
- If the prompt says “send users to the closest Region,” latency-based routing is usually better.
- If the prompt says “shift 10% of traffic to a new version,” weighted routing fits.

**Storage scenario checks**

Storage questions often require you to distinguish between object, block, and file storage, then refine by performance and sharing needs.

**Amazon S3**
- Use S3 for durable object storage, backups, data lakes, static content, and event-driven workflows.
- Use lifecycle policies to transition objects to cheaper storage classes or expire them.
- Use replication when you need cross-Region or same-Region copy for compliance or DR.

Example lifecycle policy:

```json
{
  "Rules": [
    {
      "ID": "ArchiveAfter30Days",
      "Status": "Enabled",
      "Filter": { "Prefix": "" },
      "Transitions": [
        { "Days": 30, "StorageClass": "STANDARD_IA" },
        { "Days": 90, "StorageClass": "GLACIER" }
      ]
    }
  ]
}
```

Use S3 replication when:
- You need a second copy in another Region for DR.
- You need compliance-driven data residency or separation.
- You need to replicate objects between accounts.

**Amazon EBS**
- Use EBS for block storage attached to a single EC2 instance at a time.
- It is appropriate for boot volumes, databases on EC2, and low-latency block access.
- It is not a shared file system.

**Amazon EFS**
- Use EFS for shared Linux file storage with POSIX semantics across multiple instances.
- It is a strong fit when multiple EC2 instances need concurrent file access.
- It is not the default answer for every shared storage problem; workload performance, protocol, and operating system matter.

**FSx variants**
- **FSx for Lustre** is optimized for high-performance computing and workloads that need very high throughput and low-latency parallel file access.
- **FSx for NetApp ONTAP** is useful when you need enterprise file features, NFS/SMB support, snapshots, and flexible data management.
- **FSx for Windows File Server** is appropriate for Windows-based file sharing and SMB workloads.

Use the FSx family when the prompt mentions:
- HPC or bursty parallel file access → Lustre.
- Enterprise NAS features or mixed protocol needs → NetApp ONTAP.
- Windows file shares → Windows File Server.

**Database scenario checks**

Database questions often test whether you need availability, read scaling, or engine-specific features.

**Aurora vs RDS**
- Use **Aurora** when you need high-performance managed relational storage with strong availability characteristics and scaling features.
- Use **RDS** when the scenario needs a standard managed relational engine and Aurora-specific features are unnecessary.
- Aurora is often a better fit for new cloud-native relational workloads, but not every relational workload should default to Aurora.

**Read replicas vs Multi-AZ**
- Use **read replicas** to scale read traffic and support some DR patterns.
- Use **Multi-AZ** for high availability and automatic failover.
- Multi-AZ is about resilience; read replicas are about read scaling and optional failover support depending on the engine and configuration.

**DynamoDB**
- Use DynamoDB for serverless key-value or document workloads with predictable access patterns and low-latency scale.
- It is a strong fit for session stores, metadata, and high-scale application state.

Practical selection clue:
- If the prompt emphasizes joins, ad hoc queries, or relational constraints, DynamoDB is usually not the right answer.
- If the prompt emphasizes massive scale with simple access patterns, DynamoDB is often a candidate.

**Resilience, RTO, and RPO**

RTO (Recovery Time Objective) is how long you can tolerate downtime. RPO (Recovery Point Objective) is how much data loss you can tolerate.

Use these patterns:
- **Backup and restore** when cost matters more than recovery speed.
- **Pilot light** when you need a minimal core running and can scale up during failover.
- **Warm standby** when you need a smaller but fully functional environment ready to scale quickly.
- **Active-active** when you need the fastest recovery and can support complexity and cost.

Example mapping:
- “Can tolerate hours of downtime” → backup and restore or pilot light.
- “Can tolerate minutes of downtime” → warm standby.
- “Must keep service available during a regional failure” → active-active or multi-Region design.

**Security and edge protection**

**KMS**
- Use AWS Key Management Service (KMS) for encryption key creation, rotation, and access control.
- Use customer managed keys when you need tighter control or auditability.
- Use AWS managed keys when the service-managed default is sufficient.

**AWS WAF**
- Use AWS WAF for Layer 7 filtering against common web exploits, bot patterns, and request-based rules.
- It integrates with Amazon CloudFront, Application Load Balancer, Amazon API Gateway, and AWS AppSync.
- It is not a general network firewall.

**AWS Shield**
- Shield Standard provides baseline DDoS protection.
- Shield Advanced adds enhanced DDoS protections, cost protection, and access to the AWS DDoS Response Team (DRT).

**Secrets handling**
- Use Secrets Manager for rotating secrets and managing credentials.
- Use Systems Manager Parameter Store for configuration values and simpler secret storage needs, especially when rotation is not required.

**CloudFront and edge scenarios**

Use CloudFront when you need:
- Global content caching.
- Lower latency for static and dynamic content.
- Integration with WAF and origin protection.
- TLS termination at the edge.

Example origin protection pattern:
- Put S3 behind CloudFront.
- Restrict S3 bucket access to the CloudFront origin access control or origin access identity pattern, depending on the design.
- Add WAF at CloudFront for request filtering.

**Lambda in scenario questions**

AWS Lambda is best for event-driven tasks, glue logic, and bursty workloads, but you should match it to the full architecture.

Use Lambda when:
- The workload is short-lived and event-driven.
- You want to avoid server management.
- You need automatic scaling for spiky traffic.

Do not overgeneralize Lambda as the answer for every API or backend scenario:
- For a public API, you usually need API Gateway or an Application Load Balancer in front of Lambda.
- For long-running processes, Lambda may not fit due to execution limits.
- For stateful or high-throughput streaming workloads, other services may be better.

Example serverless API pattern:

```text
Client -> Amazon API Gateway -> AWS Lambda -> Amazon DynamoDB
```

**Migration and hybrid scenario checks**

Migration questions often ask for the least disruptive path with acceptable cutover risk.

Consider:
- **AWS Application Migration Service** for lift-and-shift server migration.
- **Database Migration Service (DMS)** for ongoing replication and database migration.
- **Snowball** for large offline data transfer.
- **DataSync** for online file transfer and synchronization.

Hybrid routing nuance:
- Use VPN for quick encrypted connectivity or as backup.
- Use Direct Connect for consistent throughput and private connectivity.
- Use Transit Gateway when many networks need centralized routing.
- Use Direct Connect plus VPN for resilience.

**Observability and operations**

Operational questions often ask how to detect, trace, or respond to issues without adding too much overhead.

Use:
- **CloudWatch metrics and alarms** for thresholds and automated response.
- **CloudWatch Logs** for application and system logs.
- **X-Ray** for distributed tracing when you need request path visibility.
- **CloudTrail** for API auditing and governance.

Example alarm use case:
- Trigger an Auto Scaling policy when CPU utilization stays above a threshold.
- Send an SNS notification when a critical error metric exceeds a limit.

**Worked scenario examples**

**Example 1: Cross-account S3 access**
A data analytics account needs read access to a central S3 bucket in a shared services account.

Best pattern:
- Use an S3 bucket policy to allow the analytics account role.
- Use an IAM role in the analytics account for temporary access.
- Use KMS key policy updates if the bucket objects are encrypted with a customer managed key.

Why:
- Resource policy plus role assumption gives clean cross-account access.
- Long-term access keys are unnecessary.

**Example 2: Private access to AWS services from isolated subnets**
An application in private subnets needs to call S3 and DynamoDB without internet access.

Best pattern:
- Use gateway VPC endpoints for S3 and DynamoDB.
- Avoid NAT Gateway if the only requirement is AWS service access.

Why:
- Gateway endpoints keep traffic private and reduce NAT cost.

**Example 3: Hybrid DNS resolution**
On-premises DNS servers must resolve private hosted zone names in AWS, and EC2 instances must resolve on-premises internal names.

Best pattern:
- Inbound Route 53 Resolver endpoint for on-premises to AWS DNS queries.
- Outbound Route 53 Resolver endpoint with forwarding rules for AWS to on-premises DNS queries.

Why:
- Direction matters, and exam questions often reverse it.

**Example 4: Global low-latency application**
A user-facing application needs static anycast IPs and fast failover across Regions.

Best pattern:
- Use AWS Global Accelerator in front of regional endpoints.
- Optionally combine with Route 53 for DNS-based routing if needed.

Why:
- Global Accelerator improves routing and failover behavior beyond DNS alone.

**Example 5: Shared file storage for Linux application servers**
Multiple Linux instances need concurrent access to the same files.

Best pattern:
- Use EFS if you need shared POSIX file semantics and general-purpose shared access.
- Use FSx for Lustre if the workload is HPC-style and needs very high throughput.
- Use FSx for NetApp ONTAP if enterprise file features or protocol flexibility matter.

Why:
- The best answer depends on performance and workload type, not just “shared file storage.”

**Example 6: Regional failure recovery**
A business can tolerate 15 minutes of downtime and minimal data loss.

Best pattern:
- Warm standby or a highly automated multi-Region design, depending on workload statefulness.
- Use replication and automated failover where possible.

Why:
- Backup and restore is usually too slow for this RTO.
- Active-active may be more expensive than necessary if the workload does not require continuous availability.

**Example 7: Web application protection**
A public application needs protection from SQL injection and common web exploits.

Best pattern:
- Put AWS WAF on CloudFront or Application Load Balancer.
- Add Shield Standard automatically for baseline DDoS protection.
- Consider Shield Advanced if the workload is business-critical and needs DDoS cost protection and DRT support.

**Multiple-response practice questions**

1. A company wants private access from a VPC to S3 and DynamoDB without using the internet. Which options are correct?
   - A. Gateway VPC endpoints
   - B. Interface VPC endpoints
   - C. NAT Gateway
   - D. Route 53 private hosted zones

   Correct: A  
   Why: S3 and DynamoDB use gateway endpoints for private VPC access.

2. A company wants on-premises DNS servers to resolve names in a Route 53 private hosted zone. Which option is correct?
   - A. Outbound Resolver endpoint
   - B. Inbound Resolver endpoint
   - C. NAT instance
   - D. CloudFront distribution

   Correct: B  
   Why: On-premises resolvers query AWS through inbound endpoints.

3. A company needs a managed, highly available outbound internet path for private subnets. Which option is correct?
   - A. NAT Gateway
   - B. Internet Gateway
   - C. Transit Gateway
   - D. VPC peering

   Correct: A  
   Why: Private subnets use NAT Gateway for outbound internet access.

4. A company needs organization-wide restrictions that apply to member accounts. Which options are correct?
   - A. SCPs
   - B. IAM permission boundaries
   - C. AWS Organizations
   - D. Security groups

   Correct: A, C  
   Why: Organizations provides the structure; SCPs provide the guardrails.

5. A workload needs shared Linux file storage across multiple EC2 instances. Which options could be correct depending on workload details?
   - A. EFS
   - B. FSx for Lustre
   - C. EBS
   - D. Instance store

   Correct: A, B  
   Why: EFS fits general shared POSIX access; FSx for Lustre fits high-throughput HPC-style access.

6. A company needs Layer 7 filtering for a public web application. Which options are correct?
   - A. AWS WAF
   - B. Security groups
   - C. Network ACLs
   - D. AWS Shield Standard

   Correct: A  
   Why: WAF inspects HTTP(S) requests at Layer 7.

7. A company needs a dedicated private network link to AWS and wants to connect multiple VPCs across Regions through that link. Which options are correct?
   - A. Direct Connect Gateway
   - B. Site-to-Site VPN only
   - C. Transit Gateway
   - D. Internet Gateway

   Correct: A, C  
   Why: Direct Connect Gateway and Transit Gateway can both be part of the design, but they solve different layers of the hybrid architecture.

8. A company needs a global entry point with static IPs and fast failover to healthy endpoints. Which options are correct?
   - A. AWS Global Accelerator
   - B. Route 53 failover routing
   - C. NAT Gateway
   - D. VPC endpoint

   Correct: A, B  
   Why: Global Accelerator provides static anycast IPs and fast failover; Route 53 failover can also be part of the design.

**Quick elimination checklist**

When you are unsure, ask:
- Is the traffic public or private?
- Is the access cross-account or same-account?
- Is the requirement about availability, read scaling, or both?
- Is the storage object, block, or file?
- Is the workload general-purpose or specialized?
- Is the recovery target measured in minutes, hours, or seconds?
- Does the service need to be managed, or can you accept operational overhead?

**High-value exam traps**

- Choosing EFS when the workload actually needs HPC throughput from FSx for Lustre.
- Choosing Aurora when the prompt only needs a standard managed relational database.
- Choosing read replicas when the question asks for automatic failover.
- Choosing NAT Gateway when the requirement is private AWS service access and VPC endpoints are enough.
- Choosing a resource policy when the scenario needs temporary cross-account access through a role.
- Choosing geolocation routing when the prompt really asks for lowest latency.
- Assuming Lambda alone is a complete API backend without an API front door.
- Forgetting that SCPs do not apply to the management account.

**Final exam mindset**

SAP-C02 rewards precision. The best answer is usually the one that:
- Meets every stated requirement.
- Minimizes unnecessary complexity.
- Uses managed services where appropriate.
- Respects security boundaries and recovery objectives.
- Fits the workload’s actual access pattern and performance profile.

If two answers both seem plausible, compare them against the exact wording of the prompt. The exam often hides the deciding detail in a single phrase such as “cross-account,” “private,” “global,” “low-latency,” “RTO,” or “no internet access.”
