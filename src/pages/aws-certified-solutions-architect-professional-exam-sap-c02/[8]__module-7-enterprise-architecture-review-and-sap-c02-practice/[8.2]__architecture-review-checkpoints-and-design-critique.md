## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Architecture review mindset**

Architecture review is the process of evaluating a design against business goals, operational constraints, security requirements, reliability targets, performance needs, and cost limits. For SAP-C02, you are rarely asked to memorize a single “best” service. Instead, you must critique a proposed design, identify hidden assumptions, and choose the option that best balances tradeoffs.

A strong review asks:
- What is the workload trying to optimize?
- Which failure modes matter most?
- Where are the bottlenecks, single points of failure, or operational risks?
- Which AWS services reduce complexity versus add it?
- What changes when scale, compliance, or recovery objectives become stricter?

Use a repeatable review framework so you do not get distracted by implementation details before validating the fundamentals.

**Core review checkpoints**

Start with the workload’s nonfunctional requirements. These are the constraints that usually drive the architecture choice more than the application code does.

- **Availability**: Define the acceptable downtime and whether the design needs automatic failover, multi-AZ deployment, or multi-Region recovery.
- **Durability**: Check where data is stored, replicated, and backed up, and whether the design can survive accidental deletion or regional disruption.
- **Recovery objectives**: Confirm the recovery time objective (RTO) and recovery point objective (RPO) before choosing backup, replication, or active-active patterns.
- **Security**: Validate identity boundaries, encryption, network segmentation, logging, and least privilege.
- **Performance**: Identify latency-sensitive paths, throughput limits, connection scaling, and caching opportunities.
- **Cost**: Look for overprovisioning, unnecessary data transfer, idle resources, and expensive always-on components.
- **Operability**: Assess how the team deploys, monitors, patches, tests, and recovers the system.

A design can be technically correct and still be a poor fit if it violates one of these checkpoints.

**How to critique an AWS architecture**

A good critique is specific. Do not say a design is “bad”; say which requirement it fails and why. Tie every concern to an observable consequence.

1. Identify the primary business goal.
2. Map the goal to AWS design attributes such as availability, elasticity, isolation, or recovery.
3. Check for hidden coupling between tiers, Regions, accounts, or dependencies.
4. Validate data flow, trust boundaries, and failure domains.
5. Compare the proposed service choice with simpler alternatives.
6. Confirm operational readiness: monitoring, automation, patching, and testing.
7. Revisit cost and complexity after the technical fit is established.

When reviewing a diagram, look for the following red flags:
- A single NAT gateway, load balancer, database, or queue that becomes a bottleneck or failure domain.
- Cross-Region assumptions without explicit replication or failover design.
- Shared credentials or broad IAM permissions.
- Manual recovery steps where automation is required.
- Stateful components placed in the wrong tier for the availability target.
- Data transfer paths that create latency or cost surprises.

**Reliability and failure-domain analysis**

Reliability review focuses on how the system behaves when something fails. In AWS, you should think in terms of failure domains: instance, Availability Zone (AZ), Region, account, and dependency.

- **Instance failure**: Use Auto Scaling, health checks, and stateless application tiers where possible.
- **AZ failure**: Spread critical components across multiple AZs and avoid AZ-local single points of failure.
- **Region failure**: Use backups, replication, pilot light, warm standby, or active-active patterns depending on RTO and RPO.
- **Dependency failure**: Consider what happens if an upstream API, DNS, identity provider, or third-party service becomes unavailable.

For relational databases, remember the distinction:
- **Multi-AZ** is for high availability and automatic failover.
- **Read replicas** are for read scaling and, in some cases, disaster recovery patterns depending on engine and configuration.
- They solve different problems and should not be treated as interchangeable.

A common critique pattern is to ask whether the design can survive the loss of one component without losing the whole service. If not, the architecture likely needs redundancy or a different state-management strategy.

**Security review checkpoints**

Security review is about reducing blast radius and proving control over access, data, and network paths.

- **Identity and access**: Prefer IAM roles over long-lived access keys. Use least privilege and scope permissions to the exact action, resource, and condition.
- **Network segmentation**: Place resources in private subnets when public exposure is unnecessary. Use security groups as stateful instance-level controls and network ACLs only when subnet-level stateless filtering is needed.
- **Encryption**: Verify encryption at rest and in transit. Confirm who manages keys and whether AWS KMS key policies, grants, or rotation settings are required.
- **Logging and detection**: Ensure CloudTrail, CloudWatch Logs, AWS Config, and service-specific logs are enabled where needed.
- **Data protection**: Classify data, restrict exfiltration paths, and review backup access separately from production access.

A practical IAM review often checks whether the policy matches the resource type and action type. For example, `s3:ListBucket` applies to the bucket ARN, while `s3:GetObject` applies to object ARNs. Conditions such as `s3:prefix` are relevant to listing operations, not object retrieval.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListOnlySpecificPrefix",
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::example-bucket",
      "Condition": {
        "StringLike": {
          "s3:prefix": [
            "app/*"
          ]
        }
      }
    },
    {
      "Sid": "ReadObjectsInPrefix",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/app/*"
    }
  ]
}
```

This separation matters because it reflects how AWS evaluates permissions and prevents misleading least-privilege examples.

**Performance review checkpoints**

Performance review asks whether the design can meet latency and throughput targets under expected and peak load.

- **Compute scaling**: Check whether the application tier scales horizontally and whether startup time affects responsiveness.
- **Database bottlenecks**: Look for connection limits, hot partitions, inefficient queries, and missing indexes.
- **Caching**: Use caching when repeated reads dominate and stale data is acceptable within a defined window.
- **Content delivery**: Use Amazon CloudFront for global edge caching and reduced origin load when content is cacheable.
- **Asynchronous processing**: Use queues, streams, or event-driven decoupling when synchronous processing would create latency spikes.
- **Storage access patterns**: Match the storage service to the workload’s access frequency, object size, and consistency needs.

A design critique should ask whether the workload is compute-bound, memory-bound, network-bound, or I/O-bound. The right AWS service often depends on which resource is saturated first.

**Cost and complexity review**

Cost review is not just about the monthly bill. It is also about operational cost, cognitive load, and the risk of overengineering.

- **Overprovisioning**: Check for always-on capacity that could be replaced with autoscaling or serverless options.
- **Data transfer**: Review cross-AZ, cross-Region, and internet egress charges.
- **Managed service fit**: Prefer managed services when they reduce undifferentiated heavy lifting without violating requirements.
- **Architecture sprawl**: Avoid adding services that do not materially improve the design.
- **Lifecycle policies**: Use storage tiering and retention controls to reduce long-term storage cost.

For archival storage, choose the class based on retrieval expectations rather than defaulting to a single “cold” option. S3 lifecycle transitions can move objects to classes such as `STANDARD_IA`, `ONEZONE_IA`, `GLACIER_IR`, `GLACIER`, or `DEEP_ARCHIVE` depending on access pattern and recovery expectations.

```json
{
  "Rules": [
    {
      "ID": "ArchiveLogsAfter30Days",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "GLACIER_IR"
        }
      ],
      "Expiration": {
        "Days": 365
      }
    }
  ]
}
```

Use lifecycle rules when the access pattern is predictable and retention policy is clear. Do not use them as a substitute for backup strategy.

**Availability patterns and when to use them**

Availability design depends on the acceptable outage window and the amount of automation you can support.

| Pattern | Typical use | Strength | Limitation |
|---|---|---|---|
| Single-AZ | Dev/test or noncritical workloads | Lowest cost and simplest | No AZ resilience |
| Multi-AZ | High availability for stateful services | Automatic failover within a Region | Does not provide read scaling by itself |
| Read replicas | Read-heavy workloads and some DR patterns | Offloads reads from primary | Replica lag and engine-specific behavior |
| Backup and restore | Cost-sensitive DR | Simple and inexpensive | Higher RTO |
| Pilot light | Faster DR than backup/restore | Core components prepositioned | More complexity |
| Warm standby | Lower RTO than pilot light | Reduced recovery time | Higher cost |
| Active-active | Very low RTO/RPO targets | Best resilience and global availability | Highest complexity |

Use the simplest pattern that satisfies the objective. If the requirement is only to survive an AZ outage, a multi-AZ design is usually more appropriate than a multi-Region active-active architecture.

**RDS critique examples**

Amazon Relational Database Service (RDS) is often central to architecture review because it combines state, availability, and operational constraints.

- Use **Multi-AZ** when you need automatic failover and higher availability for a relational database.
- Use **read replicas** when you need to scale reads, reduce read pressure on the primary, or support certain recovery patterns.
- Do not treat read replicas as a general-purpose replacement for Multi-AZ.
- Do not assume cross-Region read access is universally available or operationally equivalent across engines and architectures.

A strong critique checks whether the design confuses availability with scaling. If the workload needs more read throughput, a replica may help. If the workload needs automatic failover, Multi-AZ is the more relevant control.

Example operational checks:
- Is the application using the writer endpoint for writes and the correct reader endpoint for reads?
- Is failover tested?
- Are connection pools resilient to endpoint changes?
- Is the application retry logic safe for idempotent and non-idempotent operations?

**S3 critique examples**

Amazon Simple Storage Service (S3) designs often fail review because of access control, lifecycle, or data protection mistakes.

- Use bucket policies, IAM policies, and access points intentionally; do not rely on one control to solve every access problem.
- Separate permissions for listing and object access.
- Use versioning when you need protection against accidental overwrite or deletion.
- Use lifecycle rules for retention and storage-class transitions.
- Use replication when you need copies in another Region or account.

A common review question is whether the design protects against accidental deletion. If not, consider versioning, MFA delete where appropriate, object lock for compliance use cases, and cross-account replication for stronger isolation.

```bash
aws s3api put-bucket-versioning \
  --bucket example-bucket \
  --versioning-configuration Status=Enabled

aws s3api put-bucket-lifecycle-configuration \
  --bucket example-bucket \
  --lifecycle-configuration file://lifecycle.json
```

When reviewing S3 access, check whether the policy matches the intended scope:
- Bucket-level actions such as `s3:ListBucket` use the bucket ARN.
- Object-level actions such as `s3:GetObject` use object ARNs.
- Prefix conditions apply to list operations and should not be mixed into object-read statements as if they controlled the same permission.

**Network design critique**

Network review focuses on reachability, segmentation, and dependency paths.

- **Public subnets** should contain only resources that must accept inbound internet traffic.
- **Private subnets** should host application and data tiers whenever possible.
- **Security groups** should express application trust boundaries.
- **Route tables** should be reviewed for unintended internet exposure or asymmetric routing.
- **NAT gateways** should be evaluated for cost and AZ resilience.
- **Endpoints** can reduce NAT dependency for AWS service access.

A common design flaw is routing all private subnet traffic through a single NAT gateway. That creates both a cost concentration and a failure domain. For higher resilience, place NAT gateways per AZ when the workload justifies the cost, or use VPC endpoints where appropriate.

**Operational excellence review**

Operational excellence means the architecture is observable, automatable, and recoverable by the team that runs it.

- **Monitoring**: Confirm metrics, logs, alarms, and dashboards exist for the critical path.
- **Automation**: Use infrastructure as code, deployment pipelines, and repeatable recovery steps.
- **Testing**: Validate failover, backup restore, scaling, and incident response.
- **Change management**: Review how updates are rolled out and rolled back.
- **Runbooks**: Ensure operators know what to do when alarms fire.

A design that cannot be operated safely at 2 a.m. is not production-ready, even if it looks elegant on paper.

**Design critique workflow**

Use a consistent workflow when answering exam questions or reviewing a proposed architecture.

1. Read the requirement twice and extract the hard constraints.
2. Identify the primary objective: availability, performance, security, cost, or a mix.
3. Mark the failure domains and trust boundaries.
4. Check whether the proposed AWS services match the workload pattern.
5. Eliminate options that violate a hard requirement.
6. Compare the remaining options by tradeoff, not by preference.
7. Choose the simplest design that satisfies the objective.

This workflow helps you avoid common exam traps where a solution is technically possible but operationally inappropriate.

**Practical critique checklist**

Use this checklist when you inspect a diagram or answer choice:

- Does the design meet the stated RTO and RPO?
- Is there a single point of failure in compute, network, storage, or identity?
- Are security boundaries explicit and least privilege enforced?
- Is the data path efficient, or does it cross unnecessary boundaries?
- Are managed services used where they reduce operational burden?
- Is the architecture overbuilt for the requirement?
- Are backups, restore tests, and failover tests defined?
- Are costs acceptable for the expected traffic and retention?

If you cannot answer one of these questions confidently, the design needs more review.

**Common exam traps**

SAP-C02 often tests your ability to reject plausible but incorrect designs.

- Choosing the most resilient architecture when the requirement only asks for cost-effective recovery.
- Using read replicas as if they automatically provide high availability.
- Treating Multi-AZ as a read-scaling feature.
- Ignoring data transfer costs between AZs or Regions.
- Overlooking IAM scope errors in otherwise correct designs.
- Using a complex multi-Region pattern when a simpler single-Region multi-AZ design satisfies the requirement.
- Assuming backups alone provide availability instead of recovery.

The best answer usually aligns with the stated objective and avoids unnecessary complexity.

**Example critique scenario**

A team proposes a web application with an Auto Scaling group across two AZs, an Application Load Balancer, and a single RDS instance in one AZ. The application is stateless, but the database is not.

The critique is straightforward:
- The web tier is resilient to instance failure and AZ loss.
- The database is a single point of failure.
- If the requirement is high availability, the database should use Multi-AZ.
- If the requirement is read scaling, add read replicas separately.
- If the requirement is disaster recovery, add backups and a recovery plan, not just more compute.

This kind of analysis shows that you understand the architecture as a system, not as isolated services.

**CLI and validation examples**

Use AWS CLI commands to verify assumptions during design review or implementation checks.

```bash
aws rds describe-db-instances \
  --db-instance-identifier prod-db

aws rds describe-db-clusters \
  --db-cluster-identifier prod-cluster

aws ec2 describe-security-groups \
  --group-ids sg-0123456789abcdef0

aws cloudtrail describe-trails
```

These commands help you confirm whether the deployed state matches the intended architecture:
- RDS configuration reveals Multi-AZ, replicas, and endpoints.
- Security groups reveal actual network exposure.
- CloudTrail reveals whether audit logging is enabled.

**How to write a strong exam answer**

When you choose an answer, justify it mentally using the requirement, not the service name.

- Prefer the option that directly satisfies the stated objective.
- Reject options that solve a different problem.
- Avoid adding services that increase complexity without improving the requirement.
- Look for the smallest change that closes the gap.
- Remember that AWS managed services often reduce operational burden, but only when they fit the use case.

A good architecture review is disciplined, not imaginative. You are not trying to invent the most advanced design; you are trying to prove that the design is appropriate, resilient, secure, and economical for the stated workload.
