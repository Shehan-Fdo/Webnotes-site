## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Disaster recovery design goals**

Disaster recovery (DR) is about restoring service after a disruptive event while meeting business targets for recovery time and data loss. In AWS architecture questions, you usually choose a pattern by balancing **recovery time objective** (RTO), **recovery point objective** (RPO), cost, operational complexity, and workload statefulness.

- **RTO** is the maximum acceptable downtime.
- **RPO** is the maximum acceptable data loss measured in time.
- **Stateful workloads** usually need more careful data replication and failback planning than stateless workloads.
- **Criticality** drives whether you can tolerate warm standby cost or need active-active behavior.

A good exam answer often starts by identifying the workload type:
- Stateless web tier
- Stateful application tier
- Relational database
- NoSQL database
- File/object storage
- Batch or analytics pipeline
- Legacy monolith with tight coupling

**Core DR patterns**

AWS DR patterns are commonly described as:
- **Backup and restore**
- **Pilot light**
- **Warm standby**
- **Multi-site active-active**

These are not rigid categories; real architectures often mix them. For example, you might use active-active for the stateless front end, warm standby for the application tier, and asynchronous replication for the database.

| Pattern | Typical RTO | Typical RPO | Cost | Complexity | Best fit |
|---|---:|---:|---:|---:|---|
| Backup and restore | High | High | Low | Low | Non-critical workloads, infrequent recovery |
| Pilot light | Moderate | Low to moderate | Low to moderate | Moderate | Core services that can scale up after a disaster |
| Warm standby | Low | Low | Moderate | Moderate to high | Business-critical workloads needing faster recovery |
| Multi-site active-active | Very low | Very low | High | High | Global or mission-critical workloads with continuous availability needs |

**Backup and restore**

This is the simplest and cheapest pattern. You keep backups in durable storage and rebuild the environment after a disaster. It works well when downtime is acceptable and the environment can be recreated from infrastructure as code.

Use this pattern when:
- The workload is not customer-facing or can tolerate long outages.
- Data loss tolerance is measured in hours or days.
- You can automate environment rebuilds with CloudFormation, Terraform, or similar tooling.

Practical AWS building blocks:
- Amazon S3 for backup storage
- AWS Backup for centralized backup policies
- Amazon EBS snapshots for EC2 volumes
- Amazon RDS automated backups and snapshots
- AWS CloudFormation for redeploying infrastructure
- Amazon Machine Images (AMIs) for server rebuilds

Example recovery flow:
1. Recreate networking and security groups.
2. Launch compute from AMIs or templates.
3. Restore databases from snapshots or point-in-time recovery.
4. Rehydrate application configuration and secrets.
5. Validate DNS, health checks, and integrations.

A simple restore-oriented CLI sequence might look like this:

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0123456789abcdef0

aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier app-db-restored \
  --db-snapshot-identifier app-db-snapshot-20260426
```

> Backup and restore is often the right answer when the question emphasizes low cost, infrequent recovery, or the ability to rebuild from code and snapshots.

**Pilot light**

Pilot light keeps the most critical components running in the recovery Region, usually the data layer and minimal supporting services. You then scale out compute and other dependencies during a disaster.

This pattern reduces recovery time compared with backup and restore because the core data is already present. It is useful when:
- You need faster recovery than a full rebuild.
- You can tolerate some provisioning time during failover.
- You want to keep standby cost lower than warm standby.

Typical pilot light components:
- Replicated database or data store
- Minimal networking and IAM setup
- Small or stopped application capacity
- Infrastructure templates ready for rapid expansion

Common AWS services:
- Amazon RDS cross-Region read replica or snapshot restore
- Amazon Aurora Global Database for fast cross-Region database recovery with a single writer
- Amazon DynamoDB global tables for multi-Region data replication when the application can handle multi-writer semantics
- Amazon S3 Cross-Region Replication for object data
- AWS Lambda and API Gateway with preconfigured deployment artifacts

Pilot light is not the same as active-active. In pilot light, the standby Region is usually not serving full production traffic until failover occurs.

**Warm standby**

Warm standby keeps a scaled-down but functional version of the workload running in the secondary Region. It is a stronger recovery posture than pilot light because application components are already deployed and can take traffic quickly.

Use warm standby when:
- You need low RTO but do not need continuous active-active operation.
- You want a smaller but live environment in the recovery Region.
- You can afford to pay for some duplicate capacity.

Typical design:
- Full stack deployed in both Regions
- Primary Region runs at full scale
- Secondary Region runs at reduced scale
- Data replication is continuous or near-continuous
- Traffic shifts through DNS, load balancers, or Global Accelerator

Example failover approach:
- Route 53 health checks detect primary failure.
- DNS records shift to the standby Region.
- Auto Scaling increases capacity in the standby Region.
- Database promotion or writer failover occurs if needed.

A common RDS failover command is:

```bash
aws rds promote-read-replica \
  --db-instance-identifier app-db-replica
```

This creates a standalone DB instance from the replica. It does **not** preserve replication back to the original source, so you must plan for failback separately.

Warm standby is often a strong answer for SAP-C02 when the workload is important, but the question does not require true active-active global availability.

**Multi-site active-active**

Active-active means multiple Regions or sites serve traffic at the same time. This pattern provides the fastest recovery because there is no full cold start during a disaster, but it is also the hardest to design correctly.

Use active-active when:
- The workload is mission-critical and needs very low RTO.
- You can design for distributed traffic management and data consistency.
- The application can tolerate eventual consistency or has a conflict-resolution strategy.

Important distinctions:
- **Active-active for stateless tiers** is much easier than for databases.
- **Active-active across Regions for relational data** is difficult and often not truly multi-writer in the way exam distractors imply.
- **DynamoDB global tables** support multi-Region active-active replication, but you must design for last-writer-wins behavior and conflict handling.
- **Aurora Global Database** is primarily a single-writer, cross-Region replication architecture with fast failover, not true multi-writer active-active across Regions.

Common traffic management options:
- Amazon Route 53 latency-based routing
- Route 53 failover routing
- AWS Global Accelerator
- Application Load Balancer in each Region behind a global routing layer

Example Route 53 failover record set:

```json
{
  "Comment": "Failover routing for app.example.com",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "primary-us-east-1",
        "Failover": "PRIMARY",
        "TTL": 30,
        "ResourceRecords": [
          { "Value": "203.0.113.10" }
        ],
        "HealthCheckId": "12345678-90ab-cdef-1234-567890abcdef"
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "secondary-us-west-2",
        "Failover": "SECONDARY",
        "TTL": 30,
        "ResourceRecords": [
          { "Value": "198.51.100.20" }
        ]
      }
    }
  ]
}
```

If you use the CLI, you typically submit the change batch with `aws route53 change-resource-record-sets` and include the hosted zone ID separately:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch file://failover-records.json
```

**How to compare patterns by workload**

The best DR pattern depends on the workload’s architecture, not just its business importance.

- **Stateless web applications**
  - Often fit warm standby or active-active.
  - Can recover quickly because instances can be replaced.
  - Pair with load balancers, Auto Scaling, and immutable deployments.

- **Relational databases**
  - Usually drive the DR pattern choice.
  - Backup and restore is simplest.
  - Cross-Region read replicas or Aurora Global Database reduce RTO.
  - Promotion and failback steps must be explicit.

- **NoSQL databases**
  - DynamoDB global tables are strong for multi-Region active-active use cases.
  - Design for conflict resolution and item-level consistency behavior.
  - Good for globally distributed applications with low-latency writes.

- **File and object storage**
  - S3 versioning, replication, and lifecycle policies are central.
  - Cross-Region Replication helps with DR, but replication lag still matters.
  - For file systems, consider AWS DataSync, EFS replication, or backup/restore depending on the service.

- **Legacy monoliths**
  - Often favor backup and restore or pilot light.
  - Tight coupling makes active-active expensive and risky.
  - Rehosting may be easier than rearchitecting under time pressure.

**Data layer choices**

Data replication strategy is usually the most important DR decision.

- **Amazon RDS automated backups**
  - Good for point-in-time recovery.
  - Suitable for backup and restore or as part of pilot light.
  - Cross-Region snapshot copy can support regional recovery.

- **Amazon RDS read replicas**
  - Useful for read scaling and DR.
  - Cross-Region replicas can be promoted during disaster recovery.
  - After promotion, the replica becomes standalone and no longer replicates from the source.

- **Amazon Aurora Global Database**
  - Designed for low-latency global reads and fast cross-Region disaster recovery.
  - Primary Region has the writer; secondary Regions are read-only until failover.
  - Appropriate when you need very low RPO and fast recovery, but not multi-writer across Regions.

- **Amazon DynamoDB global tables**
  - Provide multi-Region replication with active-active writes.
  - Best for globally distributed applications that can tolerate eventual consistency and conflict semantics.
  - Not a fit when strict transactional coordination across Regions is required.

- **Amazon S3**
  - Versioning protects against accidental deletion and overwrite.
  - Cross-Region Replication copies objects to another Region.
  - Replication does not automatically copy all bucket settings or IAM policies, so you still need infrastructure management.

**Traffic management and failover**

Traffic shifting is separate from data recovery. You can have a healthy standby database but still fail to route users correctly if DNS, certificates, or health checks are wrong.

- **Route 53 failover routing**
  - Uses health checks to switch between primary and secondary endpoints.
  - Simple and common for regional failover.
  - DNS caching means recovery is not instantaneous for every client.

- **Route 53 latency-based routing**
  - Sends users to the lowest-latency healthy Region.
  - Useful for global applications.
  - Not a disaster recovery mechanism by itself unless combined with health checks and redundant backends.

- **AWS Global Accelerator**
  - Provides static anycast IPs and routes traffic to healthy endpoints.
  - Often improves failover behavior because clients connect to stable IPs rather than waiting on DNS TTL expiration.
  - Client-side connection reuse, TCP session behavior, and application retries still affect perceived recovery.

- **Elastic Load Balancing**
  - Works within a Region.
  - Usually paired with cross-Region routing rather than used alone for DR.

A practical pattern for web applications:
- Global Accelerator or Route 53 directs users to the active Region.
- Application Load Balancer distributes traffic within the Region.
- Auto Scaling replaces failed instances.
- Database replication or promotion handles state.

**Failover and failback**

Failover is only half the problem. Failback is often where architectures fail because teams do not plan how to return to the original Region safely.

Key failover considerations:
- Detect failure with health checks, alarms, or synthetic monitoring.
- Ensure dependencies in the standby Region are ready.
- Promote databases or switch writers only once.
- Update DNS, accelerator endpoints, or routing policies.
- Validate application health before declaring recovery complete.

Key failback considerations:
- Re-establish replication in the original Region.
- Resynchronize data before switching traffic back.
- Avoid split-brain conditions.
- Confirm that the original Region is fully healthy and patched.
- Test the rollback path as part of DR exercises.

> Promoting a replica or switching writers changes topology. You must explicitly design the reverse path; AWS does not automatically restore the original replication relationship in every service.

**Choosing the right pattern in exam scenarios**

Use the workload description to infer the right DR pattern.

- If the question emphasizes **lowest cost**, choose backup and restore or pilot light.
- If it emphasizes **fast recovery without full duplication**, choose pilot light or warm standby.
- If it emphasizes **minimal downtime and global availability**, choose active-active or a managed global data service.
- If it emphasizes **database consistency and single-writer control**, prefer Aurora Global Database or a promoted replica over multi-writer assumptions.
- If it emphasizes **global low-latency writes**, consider DynamoDB global tables.
- If it emphasizes **simple recovery for a legacy app**, choose backup and restore with automation.

A useful mental model:
1. Identify the bottleneck: compute, data, or traffic routing.
2. Determine whether the workload can be rebuilt or must stay warm.
3. Decide whether the data layer can be multi-Region active-active.
4. Choose the lowest-complexity pattern that meets RTO and RPO.

**Practical examples**

A stateless API with RTO under 15 minutes:
- Use warm standby in a second Region.
- Keep the same container image and infrastructure templates in both Regions.
- Replicate configuration and secrets.
- Use Route 53 failover or Global Accelerator.

A customer-facing SaaS platform with global users and low-latency reads:
- Use active-active stateless services.
- Use DynamoDB global tables for distributed state where appropriate.
- Use S3 replication for shared assets.
- Avoid assuming relational multi-writer support unless the service explicitly provides it.

A transactional system with strict writer control:
- Use Aurora Global Database or cross-Region read replicas.
- Keep the secondary Region ready for promotion.
- Plan for application reconnection and failback.
- Do not describe this as true active-active multi-writer across Regions.

A legacy ERP system with large databases and limited modernization budget:
- Use backup and restore or pilot light.
- Automate infrastructure rebuilds.
- Replicate backups to another Region.
- Accept longer recovery time in exchange for lower cost and lower implementation risk.

**Operational testing**

DR plans are only useful if you test them.

- Run game days and regional failover drills.
- Measure actual RTO and RPO, not just theoretical values.
- Validate DNS propagation, certificate trust, and application session behavior.
- Test database promotion, schema compatibility, and application reconnect logic.
- Verify that monitoring and alerting work in the standby Region.

Example validation commands:

```bash
aws route53 test-dns-answer \
  --hosted-zone-id Z1234567890ABC \
  --record-name app.example.com \
  --record-type A

aws rds describe-db-instances \
  --db-instance-identifier app-db-replica
```

**Common exam traps**

- Confusing **pilot light** with active-active.
- Treating **DynamoDB global tables** as a backup-and-restore pattern instead of multi-Region active-active replication.
- Assuming **Aurora Global Database** is multi-writer across Regions.
- Forgetting that **promoted read replicas** become standalone and require a new replication plan.
- Overstating **DNS failover** speed without considering TTL, client caching, and connection reuse.
- Choosing the most advanced pattern when the question asks for the lowest-cost acceptable solution.

**Quick decision guide**

- **Lowest cost, longest recovery:** backup and restore
- **Low cost, faster than rebuild:** pilot light
- **Balanced cost and recovery:** warm standby
- **Fastest recovery, highest complexity:** active-active

**Summary**

DR architecture is a tradeoff between cost, complexity, and recovery objectives. For SAP-C02, the best answer usually matches the workload’s statefulness and the business’s tolerance for downtime and data loss. Stateless tiers can often be active-active, but stateful systems usually need replication, promotion, and explicit failback planning. The strongest answers explain not just how to fail over, but how to recover cleanly and return to normal operation.
