## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Multi-Region design goals**

Multi-Region architectures improve availability, disaster recovery (DR), and sometimes latency by distributing application and data dependencies across AWS Regions. For SAP-C02, you should choose patterns based on recovery objectives rather than defaulting to “active-active” for everything.

- **RTO** (Recovery Time Objective) is how long you can tolerate downtime.
- **RPO** (Recovery Point Objective) is how much data loss you can tolerate.
- Lower RTO usually requires more automation and pre-provisioned capacity.
- Lower RPO usually requires synchronous or near-synchronous replication, which increases complexity and cost.

A good design separates concerns:

- **Traffic management** decides where users go.
- **Application tier** decides where code runs.
- **Data tier** decides where state lives and how it replicates.
- **Shared services** decide how identity, encryption, secrets, and configuration remain usable in every Region.

**Common multi-Region patterns**

The main patterns are active-passive, active-active, pilot light, warm standby, and multi-site active-active. The right choice depends on whether you optimize for cost, recovery speed, or write availability.

| Pattern | Typical use | Strength | Tradeoff |
|---|---|---|---|
| Pilot light | Critical data replicated, minimal compute in standby | Low cost | Slower recovery |
| Warm standby | Reduced-capacity stack always running | Balanced RTO/RPO | Higher cost than pilot light |
| Active-passive | One Region serves traffic, another is ready | Simple operational model | Failover delay and possible DNS propagation delay |
| Active-active | Multiple Regions serve traffic | Best resilience and latency options | Hardest data consistency problem |
| Multi-site active-active | Both Regions fully serve reads/writes | Highest availability | Requires careful conflict handling |

Use active-active only when the application and data model can tolerate distributed writes. If the workload is strongly transactional, active-passive with fast failover is often safer.

**Traffic routing and failover**

Route 53, Global Accelerator, and CloudFront solve different problems. Exam questions often test whether you understand which layer performs health evaluation and how quickly failover occurs.

- **Amazon Route 53** provides DNS-based routing.
  - It supports latency-based routing, weighted routing, geolocation, geoproximity, and failover routing.
  - DNS caching means failover is not immediate for all clients.
  - TTL (time to live) affects how quickly resolvers pick up changes.
- **AWS Global Accelerator** provides anycast static IPs and routes traffic over the AWS global network.
  - It performs endpoint health checks and can fail over faster than DNS-based approaches.
  - It is useful when you need stable IPs and fast regional failover.
  - It supports Application Load Balancers, Network Load Balancers, EC2 instances, and Elastic IPs as endpoints.
- **Amazon CloudFront** is a CDN (content delivery network).
  - It improves edge caching and can fail over between origins.
  - It is not a general application or database failover service.

> Route 53 failover is constrained by DNS behavior. If the client or recursive resolver caches the answer, traffic may continue to the old endpoint until the TTL expires. Global Accelerator avoids that DNS propagation delay for data-plane traffic because clients connect to static anycast IPs.

**Route 53 health checks and failover behavior**

Route 53 health checks are more flexible than simple endpoint pings. You can monitor endpoints directly, combine checks, or tie them to CloudWatch alarms.

- **Endpoint health checks** probe HTTP, HTTPS, or TCP endpoints.
- **Calculated health checks** combine multiple health checks using AND/OR logic.
- **CloudWatch alarm health checks** let Route 53 treat an alarm state as unhealthy.
- **Alias records** can evaluate target health when the target supports it, such as an ALB or CloudFront distribution.

Practical implications:

- Use calculated health checks when a single dependency should not determine failover.
- Use CloudWatch alarm-based checks when application health depends on internal metrics, not just port reachability.
- Use alias records with evaluate target health when you want Route 53 to consider the AWS resource’s health status.
- Do not assume a health check proves full application readiness; it only proves the condition you configured.

Example Route 53 failover record set:

```json
{
  "Comment": "Primary and secondary failover routing",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "primary-us-east-1",
        "Failover": "PRIMARY",
        "AliasTarget": {
          "HostedZoneId": "ALB_HOSTED_ZONE_ID_FOR_REGION",
          "DNSName": "dualstack.app-primary-123456.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "secondary-us-west-2",
        "Failover": "SECONDARY",
        "AliasTarget": {
          "HostedZoneId": "ALB_HOSTED_ZONE_ID_FOR_REGION",
          "DNSName": "dualstack.app-secondary-123456.us-west-2.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

Use the correct Region-specific hosted zone ID for the target resource. Do not hardcode values from memory unless you verify them for the Region and service.

**Global Accelerator versus Route 53**

These services are often compared in SAP-C02 scenarios.

- Use **Route 53** when:
  - You need DNS-based routing policies.
  - You can tolerate DNS caching delays.
  - You want simple failover or latency routing.
- Use **Global Accelerator** when:
  - You need static anycast IPs.
  - You want faster failover at the network edge.
  - You need to preserve client IP visibility at the application layer in a predictable way.
  - You want to route TCP and UDP traffic through AWS edge ingress, including HTTP/HTTPS over TCP listeners.

Global Accelerator endpoint groups are Region-scoped. Health checks are performed against endpoints, and traffic shifts away from unhealthy endpoints without waiting for DNS TTL expiration. That makes it a strong choice for active-active or active-passive designs where fast failover matters.

**CloudFront origin failover**

CloudFront origin failover helps when the origin itself becomes unhealthy, but it only changes which origin serves content.

- Use it for static websites, APIs, or content delivery where origin redundancy matters.
- Pair it with S3, ALB, or custom origins.
- It does not replace application state replication.
- It does not solve database failover.

A common pattern is:

- Route 53 points `www.example.com` to CloudFront.
- CloudFront serves cached content from the edge.
- CloudFront fails over from a primary origin to a secondary origin if the primary returns configured error responses.

For static website hosting, you can combine:

- S3 buckets in multiple Regions.
- S3 Cross-Region Replication (CRR).
- CloudFront origin failover or Route 53 failover.
- Bucket policies that allow CloudFront access through Origin Access Control (OAC).

**Application tier patterns**

The application tier should be stateless whenever possible. Stateless services are easier to duplicate across Regions and easier to fail over.

- Store session state in a shared or replicated data store rather than in-memory instance state.
- Use Auto Scaling groups or container services in each Region.
- Keep configuration externalized through Parameter Store, Secrets Manager, or environment variables.
- Design idempotent APIs so retries after failover do not create duplicate side effects.

For serverless workloads:

- Deploy Lambda functions in each Region.
- Use API Gateway in each Region behind Route 53 or Global Accelerator.
- Replicate event sources and downstream dependencies where needed.
- Prefer asynchronous decoupling for cross-Region workflows.

Example active-active serverless pattern:

- Route 53 latency routing sends users to the nearest Region.
- API Gateway invokes Lambda in that Region.
- Lambda writes to a Region-local DynamoDB table with global tables enabled.
- Events are published to EventBridge for downstream processing in each Region.

**Database design: choosing the right multi-Region model**

Database design is usually the hardest part of multi-Region architecture. The right choice depends on write locality, consistency requirements, and failover expectations.

**Amazon RDS and Multi-AZ**

RDS Multi-AZ is primarily a high availability feature within a Region, not a multi-Region DR solution.

- **Multi-AZ DB instance deployments** provide synchronous standby replication in another Availability Zone.
- **Multi-AZ DB cluster deployments** provide higher availability and faster failover than older instance-based patterns for supported engines.
- They protect against AZ failure, not Region failure.

Use Multi-AZ when:

- You need automatic failover within a Region.
- You want minimal application changes.
- You do not need Region-level isolation.

Do not rely on Multi-AZ alone for Region disaster recovery. For Region loss, you still need cross-Region backups, read replicas, snapshots, or a global database pattern.

**Amazon Aurora Global Database**

Aurora Global Database is a strong choice for low-latency global reads and fast cross-Region disaster recovery.

- One Region is the primary writer Region.
- Secondary Regions receive asynchronous replication.
- Replication lag is typically low, but it is not zero.
- Secondary Regions can serve reads.
- Planned switchover and unplanned failover are managed operations.

Operational points:

- Use **planned switchover** for maintenance or Region migration.
- Use **failover** for disaster scenarios.
- Expect some replication lag, so design for possible data loss within your RPO.
- Consider **write forwarding** when applications in secondary Regions need to send writes to the primary Region without managing cross-Region routing themselves.

Example operational flow:

1. Detect primary Region impairment.
2. Confirm replication lag and data consistency impact.
3. Promote a secondary Region to primary.
4. Update application endpoints or rely on managed routing where supported.
5. Validate write traffic and downstream integrations.
6. Re-establish the old primary as a secondary after recovery.

Aurora Global Database is often a better fit than cross-Region read replicas when you need fast DR and a managed global topology.

**Amazon RDS cross-Region read replicas**

Cross-Region read replicas are useful when you need asynchronous replication and read scaling in another Region.

- They are common for MySQL, MariaDB, PostgreSQL, Oracle, and SQL Server where supported.
- They can be promoted during DR.
- Replication lag means they are not zero-RPO.
- They are simpler than a full global database but usually slower to fail over.

Use them when:

- You need a standby copy in another Region.
- You can tolerate some replication lag.
- You want a lower-cost DR option than a fully managed global database.

**DynamoDB global tables**

DynamoDB global tables support multi-Region, multi-active writes with automatic replication.

- Each Region has a replica table.
- Writes can occur in multiple Regions.
- Replication is asynchronous.
- Last writer wins is the conflict resolution model based on item-level timestamps and replication metadata.

Design guidance:

- Keep items small and access patterns predictable.
- Avoid frequent concurrent writes to the same item from multiple Regions.
- Use idempotent write patterns and version attributes where appropriate.
- Model data so each Region usually owns a subset of writes, even if all Regions can accept writes.

Conflict-handling patterns:

- Use a **single-writer per entity** model when possible.
- Use **region affinity** for user or tenant ownership.
- Use **optimistic locking** with version attributes for application-level conflict detection.
- Use **append-only events** when you need to preserve history and reconcile later.

Example item design for multi-Region writes:

```json
{
  "pk": "ORDER#12345",
  "sk": "META",
  "regionOwner": "us-east-1",
  "status": "PENDING",
  "version": 7,
  "updatedAt": "2026-04-26T12:00:00Z"
}
```

Practical guidance:

- If two Regions can update the same item, define which attributes are authoritative.
- Use conditional writes to prevent accidental overwrites.
- Keep business logic tolerant of eventual consistency across Regions.

Modern CLI syntax for creating a global table depends on the API and AWS CLI version. In practice, you usually create the table and then add replicas, or use the current `create-global-table`/replica management workflow supported by your tooling. Verify the exact syntax in the CLI version you are using.

**S3 multi-Region patterns**

Amazon S3 is often the simplest data layer for multi-Region static content and object replication.

- Use **Cross-Region Replication** to copy objects between buckets.
- Enable versioning on both source and destination buckets.
- Use replication roles and bucket policies to authorize replication.
- Consider replication time control if you need predictable replication latency for supported workloads.

Common patterns:

- Static website content replicated to a secondary Region.
- Backup and archive copies in another Region.
- Media or document storage with Region-local read access.

For static website failover:

- Host content in S3 buckets in two Regions.
- Replicate objects from primary to secondary.
- Put CloudFront in front of both origins or use Route 53 failover.
- Ensure the secondary bucket has the same website content and permissions.

Example replication configuration concept:

```json
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "Status": "Enabled",
      "Priority": 1,
      "DeleteMarkerReplication": { "Status": "Enabled" },
      "Filter": { "Prefix": "" },
      "Destination": {
        "Bucket": "arn:aws:s3:::my-secondary-bucket",
        "StorageClass": "STANDARD"
      }
    }
  ]
}
```

Remember:

- Replication is asynchronous.
- Existing objects are not replicated unless you use S3 Batch Replication or re-copy them.
- Bucket policies and OAC/OAI settings must allow the delivery path you choose.

**Shared dependencies across Regions**

Multi-Region applications fail when shared dependencies are not replicated or are Region-bound.

- **IAM** is global within a partition, so identity policies are not Region-specific, but service-linked roles and trust relationships still matter.
- **KMS multi-Region keys** let you create related keys in multiple Regions with the same key material lineage.
- **Secrets Manager** supports secret replication to other Regions.
- **Systems Manager Parameter Store** can be replicated by automation, but it is not automatically global in the same way as IAM.
- **AWS Certificate Manager** public certificates for CloudFront must be in `us-east-1`, while regional certificates are Region-scoped.

Practical examples:

- Replicate database credentials with Secrets Manager if the application in each Region needs local access.
- Use KMS multi-Region keys when encrypted data must be decryptable in more than one Region with consistent key material lineage.
- Copy configuration parameters during deployment pipelines or with automation if the app depends on Region-local parameters.

Example KMS multi-Region key workflow:

```bash
aws kms create-key --description "Primary multi-Region key"
aws kms replicate-key --key-id <primary-key-id> --replica-region us-west-2
```

Use multi-Region keys when you need encrypted artifacts or services to move between Regions without re-encrypting under unrelated keys.

**Networking across Regions**

Cross-Region networking is important when Regions must communicate privately or when enterprise connectivity spans multiple Regions.

- **Transit Gateway** can connect VPCs within and across Regions.
- **Direct Connect Gateway** extends Direct Connect connectivity to multiple Regions and VPCs.
- **Cloud WAN** provides a managed global network fabric for large-scale multi-Region enterprise connectivity.
- VPC peering is possible across Regions, but it does not scale as well as hub-and-spoke designs.

Choose the right model:

- Use **Transit Gateway** for centralized routing within a smaller multi-Region footprint.
- Use **Direct Connect Gateway** when on-premises connectivity must reach multiple Regions through Direct Connect.
- Use **Cloud WAN** when you need policy-based global network segmentation and centralized control.
- Use **VPC peering** only for simple point-to-point cases.

Example enterprise topology:

- On-premises network connects to Direct Connect.
- Direct Connect Gateway attaches to Transit Gateway in multiple Regions.
- Each Region hosts application and data subnets.
- Cloud WAN or Transit Gateway route tables control segmentation and failover paths.

**Route 53 ARC for advanced failover orchestration**

Route 53 Application Recovery Controller (ARC) helps coordinate recovery for complex multi-Region systems.

- It provides readiness checks and routing control for recovery plans.
- It is useful when multiple dependencies must be validated before failover.
- It helps avoid partial failover where traffic shifts before the target stack is ready.

Use ARC when:

- You have many interdependent resources.
- You need controlled recovery steps.
- You want to reduce human error during regional failover.

ARC is especially relevant for large enterprise DR plans where DNS or load balancer failover alone is not enough.

**Asynchronous messaging and event replication**

Messaging and eventing are often overlooked in DR designs. If your application depends on queues or event buses, you need a Region strategy for them too.

- **SNS** topics are Region-scoped.
- **SQS** queues are Region-scoped.
- **EventBridge** event buses are Region-scoped.
- **Kinesis Data Streams** are Region-scoped.

Common patterns:

- Republish critical events to a secondary Region.
- Use application-level dual publishing for important messages.
- Persist events in a durable store before fan-out if you need replay.
- Rebuild queues after failover rather than trying to “move” them.

Practical considerations:

- SQS does not replicate queue contents across Regions automatically.
- SNS subscriptions are not global.
- EventBridge rules and buses must be recreated in each Region.
- Kinesis DR often requires producer failover and downstream consumer reattachment.

Example event-driven DR pattern:

- Application writes business state to DynamoDB global tables.
- Application publishes an event to EventBridge in the local Region.
- A cross-Region replication process forwards critical events to the secondary Region.
- Consumers in each Region process local events independently.

**End-to-end reference architectures**

Use these reference patterns to reason about exam scenarios.

**Active-passive with fast DNS failover**

- Route 53 failover routing sends traffic to the primary ALB.
- Primary Region runs the full application stack.
- Secondary Region runs minimal warm standby capacity.
- RDS Multi-AZ protects the primary Region from AZ failure.
- Cross-Region read replica or snapshot copy provides DR data.
- Route 53 health checks monitor the primary endpoint.

Best when:

- You want lower cost.
- You can tolerate some failover delay.
- You do not need simultaneous writes in both Regions.

**Active-active with global data**

- Global Accelerator or Route 53 latency routing distributes traffic.
- Each Region runs the full application stack.
- DynamoDB global tables or Aurora Global Database support the data layer.
- Stateless services handle local requests.
- Conflict handling and idempotency are built into the app.

Best when:

- You need low latency for global users.
- You need high availability across Regions.
- Your data model supports distributed writes or a single-writer pattern.

**Static content with regional failover**

- S3 buckets in two Regions store the same content.
- CRR copies objects to the secondary bucket.
- CloudFront serves cached content and can fail over between origins.
- Route 53 or Global Accelerator directs users to the edge entry point.

Best when:

- The workload is mostly static.
- You want simple and cost-effective DR.
- You need fast content delivery with regional resilience.

**Operational runbook considerations**

A DR design is incomplete without operational steps.

1. Define the trigger conditions for failover.
2. Verify health signals from application, database, and dependencies.
3. Promote or enable the secondary Region.
4. Shift traffic using Route 53, Global Accelerator, or CloudFront.
5. Validate writes, reads, and background jobs.
6. Reconcile any data divergence.
7. Plan failback carefully to avoid split-brain or data loss.

Important operational details:

- Planned failover is safer than emergency failover.
- Failback often requires more care than failover because data may have diverged.
- Test recovery regularly with game days and controlled drills.
- Document which services are Region-scoped and which are global.

**Exam distinctions to remember**

- **Multi-AZ** is not the same as multi-Region.
- **DNS failover** is slower than edge or network-level failover because of caching.
- **Aurora Global Database** is for low-latency global reads and fast DR, not synchronous multi-master writes.
- **DynamoDB global tables** support multi-Region writes, but conflict handling still matters.
- **CloudFront origin failover** is for content origin selection, not full application recovery.
- **Global Accelerator** improves failover speed and provides static IPs, but it does not replace data replication.
- **Route 53 ARC** helps orchestrate complex recovery, but it does not replicate your data.
- **Shared services** such as KMS keys, secrets, and parameters must be planned explicitly.

**Practical decision guide**

Use this quick mental model:

- If the problem is **user routing**, think Route 53, Global Accelerator, or CloudFront.
- If the problem is **application statelessness**, think duplicate stacks in each Region.
- If the problem is **database continuity**, think Aurora Global Database, RDS replicas, or DynamoDB global tables.
- If the problem is **shared configuration or encryption**, think Secrets Manager replication, KMS multi-Region keys, and deployment automation.
- If the problem is **enterprise network reachability**, think Transit Gateway, Direct Connect Gateway, or Cloud WAN.

**Common pitfalls**

- Assuming a health check means the whole Region is healthy.
- Using DNS failover and expecting instant cutover for all clients.
- Designing active-active traffic without a conflict strategy.
- Forgetting to replicate secrets, certificates, or KMS keys.
- Treating Multi-AZ as a Region DR solution.
- Ignoring asynchronous systems like queues and event buses.
- Failing to test failback, not just failover.

**Quick reference snippets**

Route 53 health check example:

```bash
aws route53 create-health-check \
  --caller-reference "app-health-2026-04-26" \
  --health-check-config '{
    "IPAddress":"203.0.113.10",
    "Port":443,
    "Type":"HTTPS",
    "ResourcePath":"/health",
    "FullyQualifiedDomainName":"app.example.com",
    "RequestInterval":30,
    "FailureThreshold":3
  }'
```

Global Accelerator accelerator creation example:

```bash
aws globalaccelerator create-accelerator \
  --name app-dr \
  --ip-address-type IPV4 \
  --enabled
```

S3 replication example:

```bash
aws s3api put-bucket-replication \
  --bucket my-primary-bucket \
  --replication-configuration file://replication.json
```

Aurora failover concept:

```bash
aws rds failover-global-cluster \
  --global-cluster-identifier my-global-cluster
```

Use these commands as study references, but always verify the exact syntax for your engine, Region, and CLI version.
