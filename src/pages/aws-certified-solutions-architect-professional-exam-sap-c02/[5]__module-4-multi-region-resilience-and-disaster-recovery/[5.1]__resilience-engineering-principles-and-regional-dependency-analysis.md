## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Resilience engineering mindset**

Resilience is the ability of a system to continue delivering acceptable service when components fail, degrade, or become unavailable. In AWS exam scenarios, you usually design for one of three outcomes:

- Keep serving traffic during a failure.
- Recover quickly with minimal data loss.
- Restore service from backups when live failover is not justified.

A resilient design starts with understanding failure domains:

- **Instance failure**: one EC2 instance, container task, or Lambda execution fails.
- **AZ failure**: one Availability Zone becomes unavailable.
- **Regional failure**: a full Region is impaired or unreachable.
- **Dependency failure**: a supporting service, certificate, secret, DNS record, or third-party endpoint is unavailable.

The exam often tests whether you can distinguish service-level resilience from application-level resilience. Managed services may survive infrastructure failures internally, but your application can still fail if it depends on a single Region, a single certificate, a single KMS key strategy, or a single data replication path.

**Core resilience principles**

Design for resilience by reducing blast radius, removing single points of failure, and making recovery predictable.

- **Automate recovery** so failover does not depend on manual intervention.
- **Decouple components** so one failure does not cascade across the stack.
- **Prefer stateless compute** and externalize session state.
- **Use health checks** at multiple layers, not just one.
- **Test recovery** regularly; untested DR plans are assumptions, not controls.
- **Measure RTO and RPO** before choosing a pattern.

<dl>
<dt><strong>RTO</strong> (Recovery Time Objective)</dt>
<dd>How long you can tolerate service downtime.</dd>
<dt><strong>RPO</strong> (Recovery Point Objective)</dt>
<dd>How much data loss you can tolerate, measured in time.</dd>
</dl>

A low RTO usually pushes you toward warm standby or active-active. A low RPO pushes you toward synchronous replication, continuous replication, or tightly controlled write paths. If both are strict, the architecture becomes more complex and expensive.

**Failure domains and AWS building blocks**

AWS resilience design is easier when you map each layer to its failure domain.

- **Compute**
  - EC2 Auto Scaling across multiple AZs.
  - ECS or EKS services spread across AZs.
  - Lambda is regional, but your function code, layers, and event sources still need regional planning.
- **Network**
  - VPC spans AZs within a Region.
  - Route tables, NAT gateways, interface endpoints, and load balancers can create hidden dependencies if deployed in only one AZ or one Region.
- **Data**
  - RDS Multi-AZ protects against AZ failure, not Region failure.
  - Aurora Global Database supports cross-Region replication for fast disaster recovery.
  - DynamoDB global tables replicate across Regions for multi-Region active-active patterns.
  - S3 is regional, but can be replicated cross-Region with CRR.
- **Identity and access**
  - IAM is a global service.
  - STS (Security Token Service) and service endpoints can introduce regional behavior into authentication flows.
- **Secrets and keys**
  - Secrets Manager, Parameter Store, and KMS keys are regional resources unless you explicitly replicate or recreate them.
- **Edge and DNS**
  - Route 53 health checks and routing policies are often the control plane for failover.
  - CloudFront can absorb global traffic patterns, but origins and certificates still matter.

> Be careful not to treat “AWS service exists globally” as “your application is globally resilient.” The application still depends on where data, certificates, endpoints, and replication targets live.

**Regional dependency analysis**

Regional dependency analysis means identifying every resource that must exist in a Region for the application to function there. This is a common SAP-C02 skill because many architectures fail not from the primary service, but from an overlooked dependency.

Typical hidden dependencies include:

- **ACM certificates**
  - ACM certificates are regional for ALB, API Gateway, and many regional integrations.
  - CloudFront requires ACM certificates in `us-east-1` for viewer-facing TLS.
- **ECR images**
  - Container images must be available in the target Region.
  - Use ECR cross-Region replication or a deployment pipeline that publishes to each Region.
- **Lambda layers and packages**
  - Lambda layers are regional.
  - If you fail over to another Region, the function code may exist but the layer ARN may not.
- **Secrets**
  - Secrets Manager secrets are regional.
  - Replicate or recreate secrets in the failover Region.
- **KMS keys**
  - KMS keys are regional.
  - Encrypted data copied to another Region often needs a destination key and the right grants.
- **VPC endpoints**
  - Interface endpoints are regional and AZ-scoped in placement.
  - Private connectivity patterns must be rebuilt in the failover Region.
- **Third-party integrations**
  - Webhooks, SaaS callbacks, and IP allow lists may be tied to a Region-specific egress path.

A good exam habit is to ask: “If I launch the same stack in another Region, what must be recreated, replicated, or re-pointed?”

**Service resilience versus application resilience**

Some AWS services are resilient by design, but your application still needs a regional strategy.

- **SQS**
  - Queue durability is managed by AWS.
  - Your application still needs a plan for queue location, consumers, dead-letter queues, and cross-Region replay if the Region fails.
- **SNS**
  - Topic delivery is regional.
  - Subscribers, endpoints, and downstream consumers may need regional duplication.
- **EventBridge**
  - Event buses are regional.
  - Cross-Region event routing is possible through custom patterns, but it is not automatic application resilience.
- **API Gateway**
  - APIs are regional or edge-optimized depending on type.
  - Backend dependencies, custom domains, and certificates still need failover planning.

This distinction matters because a service can remain available while the business workflow fails. For example, an SQS queue may still accept messages, but if the consumer fleet or downstream database is unavailable in that Region, the workflow is effectively down.

**Common multi-Region resilience patterns**

Choose the pattern based on RTO, RPO, complexity, and cost.

| Pattern | Typical RTO | Typical RPO | Characteristics |
|---|---:|---:|---|
| Backup and restore | Hours to days | Minutes to hours | Lowest cost, slowest recovery |
| Pilot light | Tens of minutes to hours | Minutes to hours | Core data and minimal services are running |
| Warm standby | Minutes | Near-real-time to minutes | Smaller full stack is always on |
| Active-active | Seconds to minutes | Near-zero to seconds | Highest complexity and cost |

**Backup and restore**

This is the simplest DR pattern. You back up data and rebuild infrastructure in the recovery Region when needed.

Use it when:

- Downtime is acceptable.
- Cost sensitivity is high.
- The application is not latency-sensitive during recovery.
- You can tolerate a longer restore process.

Typical components:

- AWS Backup plans and vaults.
- Cross-Region backup copy.
- Infrastructure as Code (IaC) templates for rebuild.
- Route 53 failover records for cutover.

**Pilot light**

Pilot light keeps the critical data layer and a minimal set of services ready in the secondary Region.

Use it when:

- You need faster recovery than backup and restore.
- You want to minimize standby cost.
- The application can scale up quickly after failover.

Typical components:

- Replicated databases or data stores.
- Minimal compute footprint.
- Pre-created networking, IAM roles, secrets, and DNS.
- Automation to scale out the rest of the stack.

**Warm standby**

Warm standby runs a scaled-down but functional version of the application in the secondary Region.

Use it when:

- You need low RTO.
- You can afford ongoing standby cost.
- You want a simpler failover than active-active.

Typical components:

- Full stack deployed in both Regions.
- Secondary Region runs at reduced capacity.
- Data replication is continuous or near-continuous.
- Traffic shifting is controlled by DNS, load balancers, or application routing.

**Active-active**

Active-active serves traffic from multiple Regions at the same time.

Use it when:

- You need the lowest RTO.
- You need global latency optimization.
- You can design for conflict resolution and data consistency.

Typical components:

- Multi-Region traffic management.
- Multi-writer or conflict-aware data design.
- Global data replication.
- Strong observability and automated rollback.

Active-active is powerful but difficult. The main challenge is not just traffic distribution; it is data consistency, session handling, and operational complexity.

**Reference multi-Region failover architecture**

A practical exam-ready architecture usually combines several layers.

- **Edge and DNS**
  - Route 53 latency-based routing or failover routing.
  - Health checks on application endpoints.
  - CloudFront in front of regional origins when global caching is useful.
- **Network**
  - Separate VPCs in each Region.
  - Subnets across at least two AZs per Region.
  - NAT gateways, interface endpoints, and security groups duplicated per Region.
- **Compute**
  - Auto Scaling groups or ECS services in both Regions.
  - Lambda functions deployed in both Regions.
  - Container images replicated to regional ECR repositories.
- **Data**
  - Aurora Global Database or cross-Region read replicas for some engines.
  - DynamoDB global tables for multi-Region active-active access.
  - S3 Cross-Region Replication for object data.
- **Secrets and configuration**
  - Secrets Manager replication or deployment automation.
  - Parameter Store values recreated in the failover Region.
  - KMS keys created in each Region for encrypted resources.
- **Observability**
  - CloudWatch alarms in each Region.
  - Centralized logs in a durable store such as S3.
  - Health dashboards and synthetic checks.
- **Recovery automation**
  - AWS Lambda, Step Functions, or runbooks to orchestrate failover.
  - IaC to rebuild missing resources.
  - Route 53 or application-level traffic shifting.

A common exam answer is not “use one service,” but “combine DNS failover, replicated data, duplicated compute, and automated recovery.”

**Route 53 failover and traffic shifting**

Route 53 is often the control plane for multi-Region failover.

Common routing policies:

- **Failover routing**
  - Primary and secondary records.
  - Health checks determine when to switch.
- **Latency-based routing**
  - Sends users to the lowest-latency healthy Region.
  - Useful for global applications.
- **Weighted routing**
  - Splits traffic by percentage.
  - Useful for gradual cutovers and testing.
- **Geolocation routing**
  - Routes based on user location.
  - Useful for compliance or regional service boundaries.

Practical example:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch file://failover-records.json
```

Example failover record set:

```json
{
  "Comment": "Failover record for app.example.com",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "primary-us-east-1",
        "Failover": "PRIMARY",
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack.my-alb-123.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "secondary-us-west-2",
        "Failover": "SECONDARY",
        "AliasTarget": {
          "HostedZoneId": "Z1H1FL5HABSF5",
          "DNSName": "dualstack.my-alb-456.us-west-2.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

> ALB alias hosted zone IDs are Region-specific. Always look up the correct value for the exact Region instead of assuming a fixed ID.

**AWS Backup as a cross-Region orchestration option**

AWS Backup is a common exam answer when you need centralized backup policy, cross-Region copy, and restore orchestration.

Key capabilities:

- **Backup plans** define schedules, retention, and lifecycle rules.
- **Backup vaults** store recovery points.
- **Copy actions** move backups to another Region or account.
- **Restore workflows** recover resources into a target Region.

Typical use cases:

- RDS snapshots copied to a DR Region.
- EBS volume backups copied cross-Region.
- EFS backups copied for regional recovery.
- DynamoDB backups protected by backup policies where supported.

Example backup plan concept:

```json
{
  "BackupPlanName": "prod-cross-region-plan",
  "Rules": [
    {
      "RuleName": "daily-backup",
      "TargetBackupVaultName": "primary-vault",
      "ScheduleExpression": "cron(0 5 * * ? *)",
      "StartWindowMinutes": 60,
      "CompletionWindowMinutes": 180,
      "Lifecycle": {
        "DeleteAfterDays": 35
      },
      "CopyActions": [
        {
          "DestinationBackupVaultArn": "arn:aws:backup:us-west-2:123456789012:backup-vault:dr-vault",
          "Lifecycle": {
            "DeleteAfterDays": 35
          }
        }
      ]
    }
  ]
}
```

Restore planning matters as much as backup creation:

- Verify the target Region has the required VPC, subnets, security groups, IAM roles, and KMS keys.
- Confirm the restore target supports the resource type.
- Test application-level recovery, not just snapshot restoration.

**Elastic Disaster Recovery (DRS)**

AWS Elastic Disaster Recovery is designed for low-RTO server recovery and regional failover. It continuously replicates source servers at the block level into a staging area in AWS and can launch recovery instances during a disaster or drill.

Use DRS when:

- You need fast recovery for physical, virtual, or cloud servers.
- You want to minimize downtime without redesigning the application.
- You are migrating legacy workloads with limited native replication options.

Why it matters for SAP-C02:

- It is a strong answer for lift-and-shift DR.
- It can reduce RTO compared with backup and restore.
- It is useful when application refactoring is not feasible.

Operationally, you still need:

- Network design in the recovery Region.
- Security groups and IAM roles.
- Launch templates or recovery settings.
- Runbooks for cutover and failback.

**Data layer resilience choices**

Different data services support different DR patterns.

**Amazon RDS**

RDS resilience depends on engine and configuration.

- **Multi-AZ**
  - Protects against AZ failure.
  - Provides synchronous standby in another AZ.
  - Not a cross-Region DR solution.
- **Read replicas**
  - Useful for read scaling and asynchronous replication.
  - Cross-Region read replicas are available for some engines.
  - They can be promoted, but promotion is not synchronous protection.
- **Snapshots**
  - Useful for backup and restore.
  - Cross-Region copy supports DR, but recovery is slower.

Practical guidance:

- Use Multi-AZ for high availability within a Region.
- Use cross-Region replication or snapshots for disaster recovery.
- Do not confuse read replicas with Multi-AZ.

**Aurora**

Aurora provides stronger DR options than standard RDS for some workloads.

- **Aurora Multi-AZ**
  - Protects within a Region.
  - Uses distributed storage across AZs.
- **Aurora Global Database**
  - Replicates across Regions with low-latency physical replication.
  - Secondary Regions are read-only until promoted.
  - Failover promotes the secondary cluster to primary, and the writer endpoint changes.

Important exam nuance:

- Application endpoints must be updated or abstracted.
- Failover is not just “turn on the secondary Region.”
- You should plan for DNS, connection pooling, and retry behavior.

**DynamoDB**

DynamoDB global tables support multi-Region active-active access.

- Writes replicate across Regions.
- Applications can read and write locally in each Region.
- Conflict resolution is handled by DynamoDB’s replication model.

Use it when:

- You need global low-latency access.
- You want active-active application design.
- Your data model tolerates eventual consistency across Regions.

Avoid assuming global tables are a universal fit:

- They do not solve all transactional consistency problems.
- You still need idempotency and conflict-aware application logic.
- Regional IAM, endpoints, and table deployment still matter.

**S3**

S3 is a regional service, but it supports global access patterns through endpoints, replication, and edge integration.

- Buckets live in a specific Region.
- Virtual-hosted-style access is common.
- S3 Cross-Region Replication copies objects to another Region.
- S3 Transfer Acceleration can improve upload performance for distributed users.

Use S3 replication when:

- You need a secondary copy of objects in another Region.
- You want DR for static assets, backups, or media.
- You need to separate primary and recovery buckets.

**EFS**

Amazon EFS is regional and can be used across multiple AZs within a Region.

For cross-Region recovery:

- Use EFS replication where supported and appropriate for your Region and design.
- If replication is not part of the design, use AWS Backup to restore into the recovery Region.

Exam framing:

- EFS is not a global file system.
- Cross-Region recovery usually requires explicit replication or backup/restore planning.

**Compute and deployment dependencies**

Compute resilience is not just about scaling; it is about placement and redeployment.

- **EC2**
  - Use Auto Scaling groups across multiple AZs.
  - Keep launch templates and AMIs available in each Region.
  - Replicate custom AMIs or rebuild them through pipelines.
- **ECS/EKS**
  - Replicate container images to regional registries.
  - Ensure task definitions, secrets, and IAM roles exist in the failover Region.
  - For EKS, consider cluster add-ons, CNI settings, and ingress controllers.
- **Lambda**
  - Deploy functions, layers, and event sources in each Region.
  - Ensure aliases, versions, and permissions are recreated.
  - Watch for regional dependencies in triggers and destinations.

Example container replication workflow:

```bash
aws ecr create-repository --repository-name app-api --region us-west-2
aws ecr put-replication-configuration --replication-configuration file://replication.json
```

Example replication configuration:

```json
{
  "rules": [
    {
      "destinations": [
        { "region": "us-west-2" },
        { "region": "eu-west-1" }
      ]
    }
  ]
}
```

**Identity, certificates, and secrets**

These are common hidden failure points.

- **IAM**
  - IAM is global.
  - Roles and policies still need to be designed for the services and Regions where they are used.
- **STS**
  - Token issuance and endpoint selection can affect regional behavior.
  - Use the correct STS endpoint strategy for your architecture.
- **ACM**
  - Regional certificates are required for regional services like ALB and API Gateway.
  - CloudFront viewer certificates must be in `us-east-1`.
- **Secrets Manager**
  - Replicate secrets or provision them in each Region.
  - Keep secret names and ARNs consistent where possible.
- **KMS**
  - Create keys in each Region for encrypted resources there.
  - Plan grants and key policies for restore workflows.

A common failure scenario is a healthy secondary Region that cannot start because the certificate, secret, or KMS key was never provisioned there.

**Observability and operational readiness**

Resilience requires visibility.

- **CloudWatch**
  - Use alarms for application health, queue depth, error rates, and latency.
  - Mirror critical alarms in each Region.
- **Logs**
  - Centralize logs in S3 or a log analytics platform.
  - Ensure log delivery itself does not depend on the failing Region.
- **Tracing**
  - Use AWS X-Ray or OpenTelemetry where appropriate.
  - Trace cross-service dependencies to identify bottlenecks during failover.
- **Synthetic checks**
  - Run Route 53 health checks or external probes.
  - Validate the user journey, not just the load balancer.

Operational readiness includes:

- Documented failover and failback runbooks.
- Regular game days and DR drills.
- Clear ownership for DNS, data, and application cutover.
- Measured recovery objectives with evidence from tests.

**Practical failover workflow**

A typical failover process looks like this:

1. Detect the failure using alarms, health checks, or synthetic monitoring.
2. Confirm the scope: AZ issue, Region issue, or dependency issue.
3. Freeze writes if the data layer requires it.
4. Promote or activate the secondary data store if needed.
5. Scale compute in the recovery Region.
6. Update DNS or traffic routing.
7. Validate application health and business transactions.
8. Reconcile data if the architecture is not fully active-active.

Example operational commands:

```bash
aws route53 get-health-check-status --health-check-id abcd1234-5678-90ef-abcd-1234567890ab
aws autoscaling update-auto-scaling-group \
  --auto-scaling-group-name app-dr-asg \
  --desired-capacity 6
aws ecs update-service \
  --cluster dr-cluster \
  --service app-service \
  --desired-count 4
```

**Exam decision guide**

Use the following logic when choosing a resilience pattern:

- If the question emphasizes **lowest cost**, choose backup and restore or pilot light.
- If the question emphasizes **low RTO** without full active-active complexity, choose warm standby.
- If the question emphasizes **global low latency** and **continuous availability**, consider active-active.
- If the question emphasizes **server recovery for legacy workloads**, consider AWS Elastic Disaster Recovery.
- If the question emphasizes **centralized backup policy and cross-Region copies**, consider AWS Backup.
- If the question emphasizes **regional failover for data**, check whether the service supports replication, promotion, or restore in the target Region.

**Common pitfalls**

- Assuming a service-level SLA equals application resilience.
- Forgetting regional copies of certificates, secrets, layers, images, and keys.
- Treating read replicas as equivalent to synchronous HA.
- Using a single Region for DNS, backups, or observability.
- Designing failover without testing failback.
- Ignoring write conflict handling in active-active systems.
- Overlooking that some services are regional even when the control plane feels global.

**Quick comparison of DR strategies**

| Strategy | Best for | Main tradeoff |
|---|---|---|
| Backup and restore | Lowest cost | Slow recovery |
| Pilot light | Moderate RTO with lower cost | Requires scale-up during failover |
| Warm standby | Fast recovery | Ongoing standby cost |
| Active-active | Highest availability and global reach | Complexity and data consistency challenges |

**Key takeaways**

- Start with RTO and RPO, then choose the pattern.
- Map every regional dependency before you declare a design resilient.
- Replicate not just data, but also certificates, secrets, images, and configuration.
- Use AWS Backup for orchestrated backup copies and restores.
- Use DRS for fast server recovery when refactoring is not practical.
- Test failover, failback, and data reconciliation as part of the design.
