## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Active-active multi-Region DR overview**

Active-active multi-Region disaster recovery (DR) means you run workloads in more than one AWS Region at the same time and accept traffic in each Region. The goal is not just recovery, but continuous service during a Regional impairment with minimal user impact.

For SAP-C02, you should distinguish active-active from active-passive:

- **Active-active**: both Regions serve traffic; data and state must be designed for concurrent use.
- **Active-passive**: one Region is warm or standby; failover activates the secondary Region.
- **Multi-Region DR**: broader term that includes both patterns, but the architecture and data consistency requirements differ.

A strong design separates concerns:

- **Traffic routing** decides where users go.
- **Application tier** stays stateless or near-stateless.
- **Data tier** uses a replication model that matches the consistency needs.
- **Shared services** such as identity, secrets, certificates, and observability must work across Regions.

**Reference target architecture**

A common lab topology uses:

- Two Regions, such as `us-east-1` and `us-west-2`
- An application stack in each Region
- A global traffic layer using Route 53 or AWS Global Accelerator
- A replicated data layer using DynamoDB Global Tables, Aurora Global Database, or S3 replication depending on the workload
- Shared or replicated supporting services for secrets, certificates, and logs

A practical mental model is:

- **Edge routing**: send users to the nearest healthy Region.
- **App tier**: scale independently in each Region.
- **State**: keep user sessions and durable data outside the instance fleet.
- **Recovery**: assume one Region can fail without manual rebuild of the surviving Region.

**Design goals and tradeoffs**

Active-active improves availability and can reduce latency, but it increases cost and operational complexity.

- **Availability** improves because one Region can continue serving traffic if the other degrades.
- **Latency** can improve when users are routed to the nearest Region.
- **Cost** rises because you run duplicate capacity and often duplicate data services.
- **Complexity** rises because you must handle replication lag, conflict resolution, and cross-Region observability.
- **Consistency** is harder because not every data store supports safe multi-writer semantics.

> Do not assume active-active automatically means zero downtime or zero data loss. The design still depends on health detection, propagation delays, and the data store’s replication model.

**End-to-end implementation workflow**

Use a repeatable workflow when building the lab.

1. Choose the Regions and define the failure domain.
   - Pick two Regions with low-latency connectivity to your users.
   - Decide whether both Regions are fully active or whether one is traffic-weighted lower.
   - Confirm service availability in both Regions for every required AWS service.

2. Deploy the application stack in both Regions.
   - Use the same AMI, container image, or infrastructure template.
   - Keep configuration externalized through Parameter Store or Secrets Manager.
   - Ensure the app is stateless or uses a shared session strategy.

3. Provision the data layer.
   - Select a replication model that matches the workload.
   - Validate write behavior, read locality, and conflict handling.
   - Test failover and recovery before sending production traffic.

4. Configure global traffic routing.
   - Use Route 53 latency-based routing, failover routing, or weighted routing.
   - Alternatively use Global Accelerator for faster traffic steering and static anycast IPs.
   - Attach health checks or endpoint health where appropriate.

5. Add supporting services.
   - Replicate secrets, certificates, and keys.
   - Centralize logs and metrics.
   - Ensure IAM and automation can operate in both Regions.

6. Validate normal operation.
   - Confirm both Regions answer requests.
   - Check that data writes appear in the expected replica.
   - Verify health checks and alarms.

7. Simulate failure.
   - Stop the app tier in one Region.
   - Block access to the data store or disable an endpoint.
   - Observe routing changes and user impact.

8. Measure recovery.
   - Record failover time, error rate, and data consistency.
   - Tune health check thresholds and traffic weights.
   - Document the operational runbook.

**Traffic routing patterns**

**Route 53 latency-based routing**

Latency-based routing sends users to the Region that should provide the lowest latency, based on AWS measurements. It is useful when you want geographic proximity and can tolerate DNS-based steering behavior.

Typical use cases:

- Global web applications
- Regionally distributed APIs
- User-facing services where latency matters more than strict locality

Key points:

- You create separate records for each Region.
- Each record points to the Regional endpoint.
- You can associate health checks so unhealthy endpoints are removed from answers.
- DNS caching means failover is not instantaneous.

Example hosted zone and record workflow:

```bash
aws route53 create-hosted-zone \
  --name example.com \
  --caller-reference "$(date +%s)"
```

Create health checks for each Regional endpoint:

```bash
aws route53 create-health-check \
  --caller-reference "app-us-east-1" \
  --health-check-config '{
    "IPAddress":"203.0.113.10",
    "Port":443,
    "Type":"HTTPS",
    "ResourcePath":"/health",
    "FullyQualifiedDomainName":"app-use1.example.com",
    "RequestInterval":30,
    "FailureThreshold":3
  }'
```

Create latency records in the hosted zone:

```json
{
  "Comment": "Latency records for active-active app",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "use1",
        "Region": "us-east-1",
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack.alb-use1-1234567890.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "usw2",
        "Region": "us-west-2",
        "AliasTarget": {
          "HostedZoneId": "Z1H1FL5HABSF5",
          "DNSName": "dualstack.alb-usw2-1234567890.us-west-2.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

Important notes:

- The `Region` field is required for latency-based routing records.
- `SetIdentifier` must be unique per record set.
- Alias target hosted zone IDs are service-specific and must match the target type.
- DNS answers can lag behind health changes because of resolver caching.

**Route 53 failover routing**

Failover routing is useful when you want one primary endpoint and one secondary endpoint. It is not the same as active-active, but it is often used in multi-Region DR labs to compare patterns.

- Primary record uses `Failover: PRIMARY`
- Secondary record uses `Failover: SECONDARY`
- Health checks determine when Route 53 returns the secondary record

Use failover routing when:

- You want a clear primary/secondary model
- You need a simpler operational story
- You are testing DR rather than true concurrent active-active serving

**Route 53 weighted routing**

Weighted routing lets you split traffic between Regions by percentage.

- Useful for gradual cutovers
- Useful for canary testing
- Useful for shifting traffic during maintenance

Weighted routing is often combined with health checks and is a practical way to simulate active-active traffic distribution before full cutover.

**AWS Global Accelerator**

Global Accelerator provides static anycast IP addresses and routes traffic to healthy endpoints over the AWS global network. It is often a better fit than DNS when you want faster traffic steering and stable IPs.

Use it when:

- You need static IPs for allowlists or client integrations
- You want faster failover behavior than DNS-based routing typically provides
- You want TCP and UDP acceleration at the edge

Core concepts:

- **Accelerator**: the global entry point
- **Listener**: defines protocol and port
- **Endpoint group**: groups endpoints by Region
- **Endpoint**: ALB, NLB, EC2 instance, or Elastic IP depending on support

Practical creation flow:

```bash
ACCELERATOR_ARN=$(aws globalaccelerator create-accelerator \
  --name lab-accel \
  --ip-address-type IPV4 \
  --enabled \
  --query 'Accelerator.AcceleratorArn' \
  --output text)

aws globalaccelerator create-listener \
  --accelerator-arn "$ACCELERATOR_ARN" \
  --protocol TCP \
  --port-ranges FromPort=443,ToPort=443
```

Capture the listener ARN, then create endpoint groups:

```bash
LISTENER_ARN=$(aws globalaccelerator list-listeners \
  --accelerator-arn "$ACCELERATOR_ARN" \
  --query 'Listeners[0].ListenerArn' \
  --output text)

aws globalaccelerator create-endpoint-group \
  --listener-arn "$LISTENER_ARN" \
  --endpoint-group-region us-east-1 \
  --health-check-protocol HTTPS \
  --health-check-port 443 \
  --health-check-path /health \
  --endpoint-configurations EndpointId=arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/app/alb-use1/50dc6c495c0c9188,Weight=128
```

Operational controls:

- **Traffic dial** adjusts the percentage of traffic sent to a Region.
- **Endpoint weight** influences distribution within an endpoint group.
- **Health checks** remove unhealthy endpoints from consideration.

Example traffic dial adjustment:

```bash
aws globalaccelerator update-endpoint-group \
  --endpoint-group-arn arn:aws:globalaccelerator::123456789012:endpoint-group/abcdef12 \
  --traffic-dial-percentage 50
```

Testing ideas:

- Set one Region’s traffic dial to `0` and confirm traffic shifts.
- Stop the app in one Region and confirm the endpoint is removed.
- Compare failover behavior against Route 53 DNS caching.

> Global Accelerator improves failover responsiveness in many cases, but actual behavior still depends on health checks, endpoint state, and client connection behavior.

**Application tier design**

The app tier should be stateless or nearly stateless.

- Store sessions outside the instance if you need horizontal scaling.
- Use immutable deployments so both Regions run the same version.
- Keep configuration in environment variables, Parameter Store, or Secrets Manager.
- Avoid local disk as the source of truth.

Common patterns:

- **Stateless web tier** behind ALB in each Region
- **Containerized services** on ECS or EKS with identical task definitions or manifests
- **Auto Scaling Groups** with health checks and rolling updates

Example ALB-backed app deployment pattern:

```bash
aws elbv2 create-load-balancer \
  --name app-alb-use1 \
  --subnets subnet-aaa subnet-bbb \
  --security-groups sg-1234567890abcdef0
```

Then register targets and attach a health check path such as `/health`.

**Session management and cache strategy**

Session handling is one of the most common failure points in active-active designs.

Recommended options:

- **Stateless JWTs** for simple authentication flows
- **Shared session store** such as DynamoDB, ElastiCache, or a database-backed session table
- **Sticky sessions** only when necessary, because they reduce failover flexibility

Tradeoffs:

- **ElastiCache** is fast but is usually Region-scoped; cross-Region replication is not a general substitute for a global session store.
- **DynamoDB** works well for session state when low-latency key-value access is enough.
- **Database sessions** are simpler to reason about but can become a bottleneck.

Practical guidance:

- Prefer stateless auth tokens when possible.
- If you must store sessions, design for Region-local reads and a replicated or globally accessible backing store.
- Avoid assuming in-memory session affinity will survive a Regional failover.

**Data layer choices**

Choose the data store based on write pattern, consistency needs, and recovery goals.

| Service | Best fit | Strengths | Caveats |
|---|---|---|---|
| DynamoDB Global Tables | Multi-Region key-value or document workloads | Active-active writes, managed replication, low operational overhead | Conflict handling is service-managed; design carefully for concurrent writes |
| Aurora Global Database | Relational workloads needing low-latency cross-Region reads and DR | Fast cross-Region replication, managed secondary Regions | Only one Region is writable at a time; not active-active multi-writer |
| S3 Cross-Region Replication | Object replication, backups, static assets | Simple object replication, versioning support | Not a transactional database; replication is asynchronous |
| ElastiCache | Regional caching | Low latency within a Region | Not a primary multi-Region system of record |

**DynamoDB Global Tables**

DynamoDB Global Tables are a strong fit for active-active applications that need multi-Region writes with low operational overhead.

Use them when:

- Your access pattern is key-value or document-oriented
- You can model data to avoid hot partitions
- You can tolerate eventual replication across Regions

Important behavior:

- Writes can occur in multiple Regions.
- Replication is asynchronous.
- Conflict resolution is managed by DynamoDB using last-writer-wins semantics based on service-side timestamps and replication behavior.
- You should not rely on application clock values to resolve conflicts.

Example table creation in one Region, then add replicas:

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=OrderId,AttributeType=S \
  --key-schema AttributeName=OrderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1
```

Add replicas after the table exists:

```bash
aws dynamodb update-table \
  --table-name Orders \
  --region us-east-1 \
  --replica-updates '[
    {"Create":{"RegionName":"us-west-2"}}
  ]'
```

Operational tips:

- Use idempotent writes where possible.
- Include version attributes if your application needs optimistic concurrency.
- Design around eventual consistency for cross-Region reads.

**Aurora Global Database**

Aurora Global Database is a good choice for relational workloads that need fast cross-Region replication and disaster recovery, but it is not active-active multi-writer.

Use it when:

- You need SQL and relational constraints
- You want a primary writable Region with secondary read-only Regions
- You need a fast promotion path during Regional disaster recovery

Key points:

- One Region is writable at a time.
- Secondary Regions replicate from the primary with low lag.
- During failover, a secondary Region can be promoted to primary.
- This is a DR pattern, not a concurrent multi-writer active-active pattern.

Example workflow:

1. Create the Aurora cluster in the primary Region.
2. Add a secondary Region as a global database secondary.
3. Validate replication lag.
4. Simulate failover by promoting the secondary Region.

Example CLI sketch:

```bash
aws rds create-global-cluster \
  --global-cluster-identifier lab-global-cluster \
  --engine aurora-mysql
```

Then create or attach regional clusters to the global cluster using the appropriate RDS workflow for your engine and deployment model.

Failover considerations:

- Application connection strings must switch to the new writer endpoint.
- DNS caching and connection pooling can delay recovery.
- Test promotion before relying on it in production.

**S3 replication and static content**

S3 Cross-Region Replication is useful for assets, backups, and object distribution.

Use it for:

- Static website assets
- User uploads
- Backup copies
- Compliance copies

Important details:

- Replication requires versioning on source and destination buckets.
- Replication is asynchronous.
- It is not a substitute for a transactional database.

Example replication configuration concept:

```json
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "Status": "Enabled",
      "Priority": 1,
      "DeleteMarkerReplication": { "Status": "Disabled" },
      "Filter": { "Prefix": "" },
      "Destination": {
        "Bucket": "arn:aws:s3:::example-destination-bucket",
        "StorageClass": "STANDARD"
      }
    }
  ]
}
```

**Supporting services for multi-Region DR**

A complete design includes more than traffic and data.

**Secrets Manager**

- Store secrets centrally and replicate them to the target Region when needed.
- Use separate secret ARNs per Region if your deployment model expects Region-local access.
- Rotate secrets carefully so both Regions stay in sync.

**AWS KMS multi-Region keys**

- Use multi-Region keys when you need the same logical key material in multiple Regions.
- This helps with encrypted data replication and cross-Region restore workflows.
- Remember that key policies and grants still need to be managed per Region.

**ACM certificates**

- ACM certificates are Region-scoped for most services.
- For CloudFront, the certificate must be in `us-east-1`.
- For Regional ALBs and API Gateway endpoints, request or import certificates in each Region.

**IAM and automation**

- IAM is global, but service resources are often Regional.
- Use infrastructure as code so both Regions are built consistently.
- Ensure deployment roles can create and update resources in both Regions.

**SSM Parameter Store**

- Store non-secret configuration in Parameter Store.
- Replicate or redeploy parameters to both Regions.
- Keep Region-specific values isolated when endpoints differ.

**Observability and validation**

You need visibility across both Regions before and after failover.

Recommended observability stack:

- **CloudWatch metrics** for app health, latency, and error rates
- **CloudWatch alarms** in each Region
- **Centralized logs** in S3 or a log analytics platform
- **Synthetic canaries** to test user journeys from multiple locations
- **Route 53 health checks** or Global Accelerator health checks for routing decisions

Useful validation commands:

```bash
curl -I https://app.example.com/health
curl https://app.example.com/api/version
dig app.example.com
nslookup app.example.com
```

For load balancer health:

```bash
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/app-tg/1234567890abcdef
```

For DynamoDB replication checks:

```bash
aws dynamodb get-item \
  --table-name Orders \
  --key '{"OrderId":{"S":"1001"}}' \
  --region us-west-2
```

For Aurora status:

```bash
aws rds describe-global-clusters \
  --global-cluster-identifier lab-global-cluster
```

**Failure injection and test plan**

Use controlled failure tests to prove the design.

1. Validate steady state.
   - Confirm both Regions serve traffic.
   - Confirm data writes replicate as expected.
   - Confirm alarms are green.

2. Disable one app endpoint.
   - Stop the web service or deregister targets in one Region.
   - Expected result: traffic shifts away from the unhealthy endpoint.

3. Simulate a Regional routing failure.
   - Change traffic dial to `0` in Global Accelerator or mark a Route 53 endpoint unhealthy.
   - Expected result: clients reach the surviving Region after routing propagation.

4. Test data-layer behavior.
   - Write data in one Region and read it from the other.
   - Expected result: replication lag is visible but bounded by the service model.

5. Test full Region impairment.
   - Block access to the primary Region’s app tier or data endpoint.
   - Expected result: surviving Region continues serving requests with acceptable degradation.

6. Record recovery metrics.
   - Measure time to detect, time to route, and time to restore.
   - Compare observed behavior with your RTO and RPO targets.

**Expected outcomes by service**

| Service | Expected failover behavior | What to watch |
|---|---|---|
| Route 53 latency routing | DNS answers shift after health and cache effects | Resolver caching, TTL, health check thresholds |
| Route 53 failover routing | Secondary record is returned when primary is unhealthy | Health check accuracy, DNS propagation |
| Global Accelerator | Traffic shifts through the AWS edge network | Endpoint health, traffic dial, client connection persistence |
| DynamoDB Global Tables | Writes continue in surviving Region; replication resumes | Conflict handling, write patterns, hot keys |
| Aurora Global Database | Secondary can be promoted to primary | Promotion time, connection string updates, replication lag |
| S3 CRR | Objects replicate asynchronously | Replication delay, versioning, delete marker behavior |

**Decision guide for data services**

Use this quick guide when choosing the data layer.

- Choose **DynamoDB Global Tables** when you need active-active writes and can model the data as key-value or document access.
- Choose **Aurora Global Database** when you need relational data and can accept one writable Region at a time.
- Choose **S3 CRR** when the data is object-based, such as uploads, backups, or static assets.
- Choose **ElastiCache** only as a cache, not as the system of record.

If your workload needs strong cross-Region transactional consistency, active-active is usually the wrong pattern. In that case, prefer a single-writer design with fast failover.

**Cost and operational tradeoffs**

Active-active is expensive to run continuously.

- You pay for duplicate compute, load balancing, and often duplicate data services.
- You may need duplicate NAT, logging, monitoring, and security controls.
- Replication traffic can add cost and complexity.
- Operational runbooks must cover both Regions equally.

Operationally, you should expect:

- More deployment coordination
- More testing of failover paths
- More careful schema and data migration planning
- More attention to drift between Regions

**Common exam pitfalls**

- Confusing **active-active** with multi-writer support for every AWS database.
- Assuming DNS failover is immediate.
- Treating Aurora Global Database as active-active.
- Using sticky sessions as a substitute for proper state design.
- Forgetting that ACM certificates, KMS keys, and many service resources are Region-scoped.
- Assuming replication eliminates the need for backups.
- Ignoring client-side connection pooling and DNS caching during failover tests.

**Practical lab checklist**

- [ ] Deploy identical app stacks in two Regions.
- [ ] Confirm health endpoints return `200 OK`.
- [ ] Configure Route 53 latency or Global Accelerator routing.
- [ ] Choose a data store that matches the write model.
- [ ] Replicate secrets, certificates, and configuration.
- [ ] Centralize logs and alarms.
- [ ] Test failover by disabling one Region.
- [ ] Measure recovery time and data behavior.
- [ ] Document the runbook and rollback steps.

**Summary**

Active-active multi-Region DR is a design for continuous service, not just recovery. The architecture works best when the application is stateless, the routing layer can steer traffic quickly, and the data layer is chosen with full awareness of consistency and replication limits.

For SAP-C02, focus on:

- When to use Route 53 versus Global Accelerator
- Which data stores truly support multi-Region active-active behavior
- How to validate failover with realistic tests
- The operational and cost impact of running duplicate capacity
