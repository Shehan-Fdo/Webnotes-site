## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Overview**

DNS-based failover and traffic-based failover solve different resilience problems. DNS failover changes which endpoint clients resolve, while traffic-based failover shifts application traffic through an intermediary such as a load balancer, proxy, or routing layer. In AWS, the most common DNS failover tool is Amazon Route 53, paired with health checks and record policies.

Use DNS failover when you need a simple, low-cost way to redirect users away from an unhealthy endpoint. Use traffic-based failover when you need more control over request routing, weighted distribution, or application-level steering. In practice, you often combine both: Route 53 decides which region or stack is active, and a load balancer or application layer distributes traffic inside that stack.

**Core concepts**

- **Route 53 failover routing** uses a primary record and a secondary record.
- **Health checks** determine whether Route 53 should answer with the primary or secondary target.
- **Alias records** let you point DNS names to AWS resources such as Application Load Balancers (ALBs), Network Load Balancers (NLBs), CloudFront distributions, and S3 website endpoints.
- **Traffic-based failover** often means shifting traffic at Layer 7 through an ALB, API Gateway, or application logic rather than relying only on DNS.
- **TTL** (time to live) affects how quickly clients observe a DNS change because resolvers cache answers.

**When to use DNS-based failover**

DNS-based failover is best when you want a straightforward active/passive design.

Typical use cases:

- Regional disaster recovery with a primary Region and a standby Region.
- Blue/green cutover where you want to move users from one environment to another.
- External endpoints that can be reached by DNS name and do not require session-aware routing.
- Simple failover for websites, APIs, or static endpoints where brief DNS caching delay is acceptable.

DNS failover is not instant from the client perspective. Route 53 can change its answer quickly, but recursive resolvers and clients may continue using cached records until TTLs expire.

**Route 53 failover routing policy**

Route 53 failover routing uses two records with the same name and type:

- One record is marked `PRIMARY`.
- One record is marked `SECONDARY`.
- Route 53 returns the primary record when its health check is healthy.
- Route 53 returns the secondary record when the primary is unhealthy or unavailable.

Important behavior:

- Route 53 evaluates the health check status before answering DNS queries.
- Route 53 does not actively move existing client sessions.
- Clients that already cached the primary answer may continue using it until cache expiration.
- If both records are unhealthy, Route 53 behavior depends on configuration and health check state; design for this case explicitly.

**Health checks**

Route 53 health checks can probe endpoints over HTTP, HTTPS, and TCP, or they can be based on calculated health checks and CloudWatch alarms.

For HTTP and HTTPS checks, Route 53 can:

- Verify that the endpoint responds successfully.
- Check for an expected status code.
- Optionally search for a string in the response body.

Do not treat Route 53 health checks as full application validation. They are lightweight reachability and response checks, not end-to-end synthetic tests.

Common health check settings:

- `IPAddress` or `FullyQualifiedDomainName`
- `Port`
- `Type` such as `HTTP`, `HTTPS`, or `TCP`
- `ResourcePath`
- `RequestInterval`
- `FailureThreshold`
- `SearchString`
- `MeasureLatency`

A practical health check often targets a dedicated path such as `/health` or `/status` that returns a fast, deterministic response.

**Example health check JSON**

```json
{
  "CallerReference": "failover-lab-2026-04-26",
  "HealthCheckConfig": {
    "FullyQualifiedDomainName": "app-primary.example.com",
    "Port": 443,
    "Type": "HTTPS",
    "ResourcePath": "/health",
    "RequestInterval": 30,
    "FailureThreshold": 3,
    "SearchString": "OK",
    "EnableSNI": true
  }
}
```

This example uses a single `FullyQualifiedDomainName` key and avoids duplicate JSON fields. In real deployments, you can also health check an ALB DNS name directly if that is the intended endpoint.

**DNS-based failover record example**

Use separate records for the primary and secondary endpoints. If you point to AWS resources with alias records, remember that alias target hosted zone IDs are resource- and Region-specific.

```json
{
  "Comment": "Create failover records",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "primary-us-east-1",
        "Failover": "PRIMARY",
        "AliasTarget": {
          "HostedZoneId": "ZXXXXXXXXXXXXX",
          "DNSName": "alb-primary-123456.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "secondary-us-west-2",
        "Failover": "SECONDARY",
        "AliasTarget": {
          "HostedZoneId": "ZYYYYYYYYYYYYY",
          "DNSName": "alb-secondary-789012.us-west-2.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

Notes on alias records:

- Use the correct hosted zone ID for the specific AWS resource and Region.
- `EvaluateTargetHealth` lets Route 53 consider the target’s health when answering DNS queries.
- Alias records remove the need to manage static IP addresses for many AWS endpoints.

**Traffic-based failover patterns**

Traffic-based failover shifts requests through a routing layer that can make decisions per request, per host, or per path. This is different from DNS failover, which only changes the name resolution answer.

Common patterns:

- **ALB path-based routing** sends `/api` and `/web` to different target groups.
- **Weighted routing** gradually shifts traffic between versions or Regions.
- **Header-based routing** supports canary or internal testing patterns.
- **Application-level failover** retries a secondary backend when the primary backend fails.

A traffic-based design is useful when you need:

- Faster steering than DNS caching allows.
- More granular control than whole-record failover.
- Gradual migration instead of a hard cutover.
- Health-aware routing inside a Region or across services.

**Example topology**

```text
Users
  |
Route 53
  |
ALB in primary Region
  |----------------------|
Target Group A        Target Group B
  |                      |
Primary app            Secondary app
```

In this model, Route 53 may still provide DNS failover at the edge, while the ALB performs traffic distribution inside the active Region.

**ALB health and Route 53 health are different**

This distinction matters in troubleshooting:

- **Route 53 health checks** probe a configured endpoint from Route 53 health checker locations.
- **ALB target health** determines whether the load balancer can send traffic to registered targets.
- **Alias target health evaluation** lets Route 53 consider the AWS resource’s health when the record points to that resource.

These mechanisms are related but not identical. A healthy Route 53 health check does not guarantee that every target behind an ALB is healthy. Likewise, unhealthy targets behind an ALB do not always mean Route 53 health checks will fail unless the checked endpoint reflects that failure.

**Practical AWS CLI workflow**

1. Create a health check for the primary endpoint.
2. Create a health check for the secondary endpoint if needed.
3. Create primary and secondary failover records.
4. Verify DNS answers with `dig` or `nslookup`.
5. Simulate failure by stopping the app or returning an unhealthy status from the health endpoint.
6. Confirm Route 53 changes the DNS answer after health check failure.
7. Restore the primary and confirm the record becomes eligible again.

**Example CLI commands**

```bash
aws route53 create-health-check \
  --caller-reference failover-lab-2026-04-26 \
  --health-check-config '{
    "FullyQualifiedDomainName":"app-primary.example.com",
    "Port":443,
    "Type":"HTTPS",
    "ResourcePath":"/health",
    "RequestInterval":30,
    "FailureThreshold":3,
    "SearchString":"OK",
    "EnableSNI":true
  }'
```

```bash
dig app.example.com
nslookup app.example.com
```

```bash
aws route53 list-health-checks
aws route53 get-health-check --health-check-id <health-check-id>
```

**Troubleshooting**

- **DNS still resolves to the primary**
  - Check whether the primary health check has actually failed.
  - Confirm the health endpoint returns the expected status and body content.
  - Lower TTL only helps future cache refreshes; it does not invalidate all existing caches.

- **Failover seems slow**
  - Remember that recursive resolvers cache answers.
  - Route 53 health check intervals and failure thresholds add detection delay.
  - Client-side DNS caching can extend the observed failover time.

- **ALB target is unhealthy**
  - Inspect target group health checks first.
  - Verify the health check path, port, protocol, and security group rules.
  - Confirm the application returns the expected response quickly enough.

- **Route 53 health check passes but users still fail**
  - The health check may be too shallow.
  - The endpoint may return `200 OK` while downstream dependencies are broken.
  - Consider a dedicated health endpoint that checks critical dependencies only if that is appropriate for your design.

> Route 53 does not “move” active client connections. It answers DNS queries based on current health, and clients may continue using cached records until their DNS cache expires.

**Design guidance for the exam**

- Prefer Route 53 failover routing for simple active/passive DNS resilience.
- Use alias records for AWS endpoints when possible.
- Use health checks that reflect meaningful service availability, not just process liveness.
- Keep TTL low enough to support recovery goals, but not so low that you create unnecessary DNS query volume.
- Distinguish between DNS-level failover and application-level traffic steering.
- Understand that alias target health, Route 53 health checks, and ALB target health solve different layers of the problem.

**Common exam traps**

- Assuming DNS failover is instantaneous for all users.
- Treating a health check as a full synthetic transaction test.
- Using hardcoded alias hosted zone IDs as universal values.
- Confusing ALB target health with Route 53 health check status.
- Forgetting that cached DNS answers can outlive the failover event.

**Quick reference**

- **DNS failover**: changes the DNS answer based on health.
- **Traffic failover**: changes request routing inside the application path.
- **Health check**: lightweight probe used to determine availability.
- **TTL**: cache duration that affects how quickly clients see DNS changes.
- **Alias record**: Route 53 record that points to AWS resources without a CNAME at the zone apex.

**Minimal failover checklist**

- Create primary and secondary endpoints.
- Add a health endpoint on each endpoint.
- Configure Route 53 failover records.
- Set appropriate TTL values.
- Test failure and recovery.
- Verify client behavior with cached DNS in mind.
