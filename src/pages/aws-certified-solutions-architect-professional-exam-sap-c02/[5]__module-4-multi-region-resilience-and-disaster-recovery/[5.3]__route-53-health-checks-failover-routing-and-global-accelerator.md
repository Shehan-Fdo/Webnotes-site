## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Route 53 health checks: purpose and behavior**

Amazon Route 53 is a DNS service that can steer traffic using routing policy and health evaluation. It does not inspect full application state like a proxy or load balancer, but it can test endpoint reachability and basic response behavior.

Route 53 health checks are commonly used to:
- Detect endpoint failure and trigger DNS failover.
- Monitor endpoints from multiple AWS health checker locations.
- Drive CloudWatch alarms for operational alerting or alarm-based failover.
- Check HTTP, HTTPS, TCP, and calculated health conditions.

Key exam points:
- **HTTP/HTTPS checks** validate that the endpoint responds and returns an expected status code.
- **TCP checks** verify that a TCP connection can be established on the configured port.
- **String matching** is only available for HTTP/HTTPS checks, not TCP.
- **Redirects** are not treated as healthy unless the response matches the configured success criteria.
- **4xx/5xx responses** are generally unhealthy unless explicitly allowed by the health check configuration and response matching rules.

A health check can be created against:
- An IP address.
- A domain name.
- A CloudWatch alarm.
- A calculated health check that combines other checks.

**Health check request and response details**

For HTTP/HTTPS health checks, Route 53 sends a request to the configured path and evaluates the response. This is often tested in scenarios involving maintenance pages, redirects, or custom status codes.

Important behavior:
- Route 53 expects a successful response code in the configured range.
- You can configure a fully qualified domain name (FQDN) for host header and certificate validation in HTTPS checks.
- If the endpoint returns a redirect, Route 53 does not automatically follow it as a browser would.
- String matching is limited to the response body for HTTP/HTTPS checks and is not available for TCP checks.
- HTTPS checks validate the certificate chain and hostname behavior according to the health check configuration.

Example CLI for an HTTP health check:

```bash
aws route53 create-health-check \
  --caller-reference hc-web-001 \
  --health-check-config '{
    "IPAddress":"203.0.113.10",
    "Port":80,
    "Type":"HTTP",
    "ResourcePath":"/health",
    "FullyQualifiedDomainName":"app.example.com",
    "RequestInterval":30,
    "FailureThreshold":3,
    "MeasureLatency":true
  }'
```

Example CLI for a TCP health check:

```bash
aws route53 create-health-check \
  --caller-reference hc-db-001 \
  --health-check-config '{
    "IPAddress":"203.0.113.20",
    "Port":5432,
    "Type":"TCP",
    "RequestInterval":30,
    "FailureThreshold":3
  }'
```

**CloudWatch alarm integration**

Route 53 can use a CloudWatch alarm as the basis for health evaluation. This is useful when endpoint health depends on application metrics rather than simple network reachability.

Use this pattern when:
- You need to fail over based on custom application signals.
- You want to combine multiple metrics into one operational alarm.
- You need to trigger DNS changes from a deeper health condition than a single endpoint probe.

Operational pattern:
- Application publishes a metric to CloudWatch.
- CloudWatch alarm transitions to `ALARM`.
- Route 53 health check associated with the alarm becomes unhealthy.
- DNS failover shifts traffic to the secondary record.

Example alarm-driven health check pattern:
```bash
aws cloudwatch put-metric-alarm \
  --alarm-name app-availability-alarm \
  --metric-name HealthyHostCount \
  --namespace AWS/ApplicationELB \
  --statistic Average \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 1 \
  --comparison-operator LessThanThreshold \
  --dimensions Name=LoadBalancer,Value=app/my-alb/50dc6c495c0c9188
```

**Failover routing with Route 53**

Failover routing is a DNS policy that returns a primary record when healthy and a secondary record when the primary is unhealthy. It is a common active/passive design for regional resilience.

Use failover routing when:
- You want a primary region and a standby region.
- You need simple disaster recovery with DNS-based traffic shifting.
- You can tolerate DNS caching delays during failover.

Core requirements:
- You create two records with the same name and type.
- One record is marked `PRIMARY`.
- The other record is marked `SECONDARY`.
- The primary record is associated with a health check or a health-evaluated alias target.
- TTL matters for non-alias records because resolvers cache answers.

Important behavior:
- For non-alias records, DNS caching means failover is not instantaneous.
- Lower TTL values reduce cache duration but do not eliminate resolver caching.
- Alias records do not use TTL in the same way as standard records because Route 53 returns the alias target response.
- If the primary is unhealthy, Route 53 returns the secondary record for failover routing.
- If both records are unhealthy, Route 53 behavior depends on record type and target evaluation; do not assume a guaranteed empty response.

**Failover record example**

Hosted zone context:
- `app.example.com` points to a primary ALB in `us-east-1`.
- `app.example.com` fails over to a secondary ALB in `us-west-2`.

Example JSON-style CLI workflow:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z123456ABCDEFG \
  --change-batch '{
    "Comment":"Primary and secondary failover records",
    "Changes":[
      {
        "Action":"UPSERT",
        "ResourceRecordSet":{
          "Name":"app.example.com",
          "Type":"A",
          "SetIdentifier":"primary-us-east-1",
          "Failover":"PRIMARY",
          "AliasTarget":{
            "HostedZoneId":"Z35SXDOTRQ7X7K",
            "DNSName":"dualstack.app-primary-123456.us-east-1.elb.amazonaws.com",
            "EvaluateTargetHealth":true
          }
        }
      },
      {
        "Action":"UPSERT",
        "ResourceRecordSet":{
          "Name":"app.example.com",
          "Type":"A",
          "SetIdentifier":"secondary-us-west-2",
          "Failover":"SECONDARY",
          "AliasTarget":{
            "HostedZoneId":"Z35SXDOTRQ7X7K",
            "DNSName":"dualstack.app-secondary-654321.us-west-2.elb.amazonaws.com",
            "EvaluateTargetHealth":true
          }
        }
      }
    ]
  }'
```

Notes on this pattern:
- `SetIdentifier` is required for failover records.
- `EvaluateTargetHealth` uses the target’s health semantics where supported.
- For AWS resources such as load balancers, this can be useful, but it is not identical to attaching a separate Route 53 health check.
- Alias target health evaluation is not supported for every AWS resource type.

**Alias records, health evaluation, and limitations**

Alias records are often used with AWS resources because they avoid extra DNS lookup cost and can point to AWS-managed endpoints.

Common alias targets:
- Application Load Balancer (ALB)
- Network Load Balancer (NLB)
- CloudFront distribution
- API Gateway
- S3 website endpoints
- Some other AWS-managed endpoints, depending on record type and service support

Exam-relevant distinctions:
- `EvaluateTargetHealth` is supported only for certain alias targets.
- It does not replace a Route 53 health check in every design.
- For load balancers, health evaluation reflects the load balancer’s own health semantics, not arbitrary application logic.
- You should not assume every alias target supports failover health evaluation.

Practical guidance:
- Use a Route 53 health check when you need explicit endpoint probing.
- Use `EvaluateTargetHealth` when the AWS target’s built-in health semantics are sufficient.
- Use both only when the scenario clearly requires layered health evaluation.

**TTL, caching, and failover timing**

DNS failover is constrained by resolver caching. This is a frequent exam trap.

Key points:
- Non-alias records have TTL values that resolvers cache.
- Lower TTL values reduce the time clients may continue using stale answers.
- Existing cached answers can delay failover even after Route 53 changes its response.
- Alias records behave differently because Route 53 returns the alias target directly, but downstream caching still exists at the resolver and client layers.

Practical implications:
- A 60-second TTL is more responsive than a 300-second TTL.
- DNS failover is not suitable for sub-second or session-level failover.
- If you need near-immediate traffic steering, consider Global Accelerator or a load balancer-based design.

**Route 53 health checks versus failover routing**

Health checks and failover routing are related but not identical.

Use a health check alone when:
- You want monitoring and alerting.
- You need a CloudWatch alarm or operational signal.
- You are not changing DNS answers.

Use failover routing when:
- You want DNS to return a standby endpoint automatically.
- You have a primary/secondary architecture.
- You can accept DNS propagation and caching delays.

Use calculated health checks when:
- You need composite health from multiple endpoints or alarms.
- You want a single failover decision based on several signals.

**Global Accelerator: what it is and what it is not**

AWS Global Accelerator provides static anycast IP addresses and routes traffic over the AWS global network to healthy endpoints. It is designed for performance and fast failover, especially for TCP and UDP workloads.

It is not:
- A CDN.
- A caching layer.
- A DNS replacement.
- A session stickiness service in the same sense as load balancer stickiness.

It is:
- An edge entry point with static IPs.
- A traffic accelerator that reduces dependence on public internet routing.
- A fast failover mechanism for supported endpoints.

Common use cases:
- Multi-region applications needing fast failover.
- TCP or UDP applications that are not a good fit for DNS-only steering.
- Workloads that benefit from static IPs for allowlisting.
- Applications where you want to avoid DNS caching delays.

**Global Accelerator architecture**

Global Accelerator uses:
- **Accelerator**: the top-level resource with static anycast IPs.
- **Listener**: defines protocol and port ranges.
- **Endpoint group**: groups endpoints by region and defines health check behavior and traffic dials.
- **Endpoint**: the actual target resource.

Supported endpoint types commonly include:
- ALB
- NLB
- EC2 instances
- Elastic IP addresses

Important behavior:
- Health checks are configured at the endpoint group level.
- Traffic dials let you shift traffic between regions.
- If all endpoints in a region are unhealthy, Global Accelerator stops sending traffic to that region and uses healthy endpoints in other regions if available.
- If no healthy endpoints exist, traffic cannot be successfully delivered.

Example CLI flow:

```bash
aws globalaccelerator create-accelerator \
  --name prod-accelerator \
  --ip-address-type IPV4 \
  --enabled
```

```bash
aws globalaccelerator create-listener \
  --accelerator-arn arn:aws:globalaccelerator::123456789012:accelerator/abcd1234 \
  --protocol TCP \
  --port-ranges FromPort=80,ToPort=80 FromPort=443,ToPort=443
```

```bash
aws globalaccelerator create-endpoint-group \
  --listener-arn arn:aws:globalaccelerator::123456789012:listener/efgh5678 \
  --endpoint-group-region us-east-1 \
  --health-check-protocol HTTPS \
  --health-check-port 443 \
  --health-check-path /health \
  --traffic-dial-percentage 100 \
  --endpoint-configurations EndpointId=arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/app/prod-alb/50dc6c495c0c9188,Weight=128
```

Notes:
- Listener protocol and port ranges are required.
- Health check settings are configured per endpoint group.
- The endpoint ID must match the supported resource identifier for the endpoint type.
- Health check path is relevant for HTTP/HTTPS health checks, not TCP.

**Global Accelerator health checks and failover**

Global Accelerator performs health checks against endpoints in each endpoint group. This is separate from Route 53 health checks.

Behavior to remember:
- Health checks are regional and endpoint-group specific.
- Traffic is routed to healthy endpoints first.
- Endpoint weights influence distribution among healthy endpoints.
- Traffic dials can reduce or increase traffic to a region.
- If a region becomes unhealthy, Global Accelerator shifts traffic away from it automatically.

This makes Global Accelerator useful when:
- You need faster failover than DNS can provide.
- You want a single static entry point for clients.
- You need to preserve client connectivity across regions without changing DNS records.

**Route 53 latency-based routing versus Global Accelerator**

These services are often confused in exam questions.

| Capability | Route 53 latency-based routing | Global Accelerator |
|---|---|---|
| Decision layer | DNS | Edge network |
| Protocol scope | DNS answers for any hostname | TCP and UDP listeners, with HTTP/HTTPS commonly behind load balancers |
| Failover speed | Limited by DNS caching | Faster, near-real-time path selection |
| Static IPs | No | Yes |
| Caching impact | Yes | No DNS caching dependency for traffic steering |
| Best for | Region selection by DNS latency | Fast failover and global entry point |
| CDN/caching | No | No |

Exam guidance:
- Choose Route 53 latency routing when the question is about DNS-based region selection.
- Choose Global Accelerator when the question emphasizes static IPs, fast failover, or improved global ingress performance.
- Do not confuse Global Accelerator with CloudFront; CloudFront is the CDN.

**Decision matrix**

Use this quick selection guide for exam scenarios:

- **Route 53 health check only**
  - You need monitoring or alarm-driven automation.
  - You do not need DNS failover.
  - You want a simple endpoint probe.

- **Route 53 failover routing**
  - You need active/passive DNS failover.
  - You can tolerate DNS caching delays.
  - You have a primary and standby endpoint.

- **Global Accelerator**
  - You need static anycast IPs.
  - You need fast failover across regions.
  - You are serving TCP or UDP workloads, or you want edge ingress independent of DNS caching.

- **CloudFront**
  - You need caching and content delivery.
  - You are serving HTTP/HTTPS content globally.
  - You want edge caching, not just traffic steering.

**Practical exam traps**

- Do not treat Route 53 as a protocol-aware traffic manager beyond its configured health checks.
- Do not assume TCP health checks can validate application strings.
- Do not assume `EvaluateTargetHealth` is equivalent to a full Route 53 health check.
- Do not assume DNS failover is immediate.
- Do not confuse Global Accelerator with a CDN.
- Do not rely on stickiness as a core Global Accelerator feature.
- Do not assume all AWS targets support alias health evaluation.

**Example architecture patterns**

Active/passive regional web app:
- Route 53 failover records point to primary and secondary ALBs.
- Route 53 health checks monitor `/health`.
- TTL is kept low to reduce cache delay.
- Suitable when cost and simplicity matter more than instant failover.

Multi-region low-latency ingress:
- Global Accelerator fronts regional ALBs or NLBs.
- Health checks run per endpoint group.
- Traffic shifts quickly when a region becomes unhealthy.
- Suitable when static IPs and fast failover are required.

Operational monitoring with DNS failover:
- CloudWatch alarms monitor application metrics.
- Alarm state drives Route 53 health checks.
- Failover occurs only when the application is truly degraded.
- Suitable when simple reachability checks are not enough.

**Summary**

Route 53 health checks provide endpoint probing and alarm integration. Route 53 failover routing uses DNS to return a primary or secondary record based on health, but DNS caching limits how fast clients switch. Global Accelerator provides static anycast IPs and faster regional failover without relying on DNS changes, making it a strong choice for global ingress and TCP/UDP workloads.
