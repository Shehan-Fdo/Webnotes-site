## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Backup governance fundamentals**

Backup governance is the set of controls, policies, and operational checks that ensure backups are created, retained, protected, and recoverable according to business and compliance requirements. In AWS, this usually combines AWS Backup, service-native snapshots, IAM (Identity and Access Management), AWS Organizations, KMS (Key Management Service), and monitoring.

Good governance answers four questions:

- What must be protected?
- How long must it be retained?
- Who can change or delete it?
- How do you prove it can be restored?

A strong design separates policy intent from operational execution. You define backup requirements centrally, apply them consistently, and verify them with restore testing rather than assuming a completed backup equals recoverability.

**Core governance controls**

Use layered controls so a single mistake does not destroy your recovery posture.

- **AWS Backup plans** define schedule, retention, and copy behavior for supported resources.
- **Backup vaults** isolate recovery points and can be encrypted with customer managed KMS keys.
- **Vault access policies** restrict who can read or manage recovery points.
- **Vault Lock** can enforce retention immutability for compliance use cases.
- **AWS Organizations** and delegated administration let a central team manage backup policy across accounts after explicit setup.
- **CloudTrail** records backup and restore API activity for auditability.
- **AWS Config** can help detect drift from required backup settings.
- **KMS key policies** and grants control whether backup data can be encrypted, copied, and restored.

A practical governance model often uses separate accounts:

- **Workload accounts** own the applications and resources.
- **Backup administration account** defines policies and monitors compliance.
- **Backup vault account** stores copies in a more restricted security boundary.
- **DR account or Region** receives cross-account or cross-Region copies for resilience.

**AWS Backup policy design**

AWS Backup is the central service for many AWS workloads, but it does not replace service-specific recovery knowledge. You still need to understand how each resource restores and what metadata or constraints apply.

Typical backup plan elements:

- **Backup rule**
  - Schedule with cron or rate expressions.
  - Start window and completion window.
  - Lifecycle transitions to cold storage where supported.
  - Retention period.
- **Backup selection**
  - Resource tags.
  - Explicit resource ARNs.
  - IAM role used by AWS Backup.
- **Copy actions**
  - Cross-Region or cross-account copy of recovery points.
  - Separate retention in the destination vault.

Example backup plan skeleton:

```json
{
  "BackupPlanName": "prod-daily",
  "Rules": [
    {
      "RuleName": "daily-0300",
      "TargetBackupVaultName": "primary-vault",
      "ScheduleExpression": "cron(0 3 * * ? *)",
      "StartWindowMinutes": 60,
      "CompletionWindowMinutes": 180,
      "Lifecycle": {
        "DeleteAfterDays": 35
      }
    }
  ]
}
```

Create the plan with the CLI:

```bash
aws backup create-backup-plan \
  --backup-plan file://backup-plan.json
```

Then assign resources by tag:

```bash
aws backup create-backup-selection \
  --backup-plan-id 1234abcd-56ef-78gh-90ij-klmnopqrstuv \
  --backup-selection '{
    "SelectionName":"prod-tag-selection",
    "IamRoleArn":"arn:aws:iam::111122223333:role/AWSBackupDefaultServiceRole",
    "ListOfTags":[
      {"ConditionType":"STRINGEQUALS","ConditionKey":"Backup","ConditionValue":"true"}
    ]
  }'
```

**Cross-account and cross-Region copy**

Copying recovery points is a key resilience pattern because it protects against account compromise, Region outage, and accidental deletion in the source account. Use separate vaults and, ideally, separate KMS keys in the destination.

Important considerations:

- The destination vault must allow the source account or backup service to copy into it.
- The destination KMS key must permit encryption for copied recovery points.
- Retention in the destination can differ from the source.
- Copying does not guarantee application consistency unless the source service supports it.

A common pattern is:

- Daily backups in the source Region.
- Immediate copy to a second Region.
- Longer retention in the DR Region.
- Optional copy to a security account for immutability and separation of duties.

**Vault protection and immutability**

Backup vaults are not just storage containers; they are control points. Use them to reduce the chance that an attacker or operator can delete recovery points.

Key features:

- **Vault access policy** limits who can interact with recovery points.
- **Vault Lock** can enforce retention rules.
- **KMS encryption** protects backup content at rest.
- **Separate admin roles** reduce blast radius.

Vault Lock is especially important for regulated environments. In compliance mode, once the changeable period ends, retention settings become immutable. Plan carefully before enabling it because the lock is intentionally hard to reverse.

Example vault lock configuration:

```bash
aws backup put-backup-vault-lock-configuration \
  --backup-vault-name prod-vault \
  --min-retention-days 35 \
  --max-retention-days 365 \
  --changeable-for-days 3
```

Use this when you need WORM-like behavior for backups. The retention bounds must make sense for your policy, and you should test the operational impact before locking production vaults.

> Vault Lock is a governance control, not a substitute for restore testing. Immutable backups that cannot be restored are still a failure.

**Restore testing strategy**

Restore testing proves that backups are usable, not merely present. This is one of the most important SAP-C02 themes because recovery objectives are only meaningful if you validate them.

Test across three dimensions:

- **Technical restore success**
  - Can the service restore the resource?
  - Are dependencies available?
  - Does the restored resource start correctly?
- **Data integrity**
  - Is the data complete and current enough?
  - Are indexes, permissions, and configuration preserved where expected?
- **Operational readiness**
  - Can the team execute the restore under pressure?
  - Are runbooks current?
  - Are access approvals and break-glass steps documented?

AWS Backup supports restore testing workflows for some resource types and can help automate validation. Even when automation exists, you should still perform periodic manual exercises because real incidents expose gaps in permissions, DNS, networking, and application dependencies.

**What to test by service**

Different AWS services restore differently, so your test plan must be resource-specific.

- **Amazon EBS**
  - Restore creates a new volume from a recovery point.
  - Validate attachment, filesystem mount, and application read/write behavior.
- **Amazon RDS**
  - Restore from snapshot creates a new DB instance.
  - Validate parameter groups, option groups, subnet group, security groups, and engine version compatibility.
  - Some restore settings are constrained by the snapshot and engine.
- **Amazon Aurora**
  - Restore creates a new cluster from a snapshot or point-in-time recovery.
  - Validate cluster endpoints, readers, and application connection strings.
- **Amazon DynamoDB**
  - Restore creates a new table from a backup.
  - Validate table data, indexes, throughput or on-demand mode, and application access.
  - Treat encryption and destination configuration as part of the new table setup rather than a generic restore-time choice.
- **Amazon EFS**
  - Restore creates a new file system.
  - Validate mount targets, security groups, POSIX permissions, and application paths.
  - If file-level restore is supported for your workflow, confirm the exact restore parameters for that job type.
- **Amazon S3**
  - AWS Backup supports S3 backups for supported use cases.
  - Validate versioning, object lock, lifecycle, replication, and application access patterns as part of the broader DR design.
- **Amazon EC2**
  - Backups are usually via EBS volumes and AMIs rather than a single “instance restore.”
  - Validate launch templates, user data, IAM instance profile, and network placement.

**Restore job workflow**

AWS Backup restore jobs are service-specific. The metadata required for a restore depends on the resource type, so do not assume one generic metadata shape works everywhere.

Typical workflow:

1. Identify the recovery point.
2. Determine the target resource type and required restore metadata.
3. Choose the destination account, Region, VPC, subnet, or table name as applicable.
4. Start the restore job.
5. Monitor job status and validate the restored resource.
6. Record findings and update the runbook.

Example restore job command pattern:

```bash
aws backup start-restore-job \
  --recovery-point-arn arn:aws:backup:us-east-1:111122223333:recovery-point:abcd1234 \
  --metadata file://restore-metadata.json \
  --iam-role-arn arn:aws:iam::111122223333:role/AWSBackupDefaultServiceRole
```

Example metadata file for a service that needs destination-specific settings:

```json
{
  "targetDBInstanceIdentifier": "restored-prod-db",
  "dbSubnetGroupName": "prod-db-subnets",
  "vpcSecurityGroupIds": "sg-0123456789abcdef0",
  "availabilityZone": "us-east-1a"
}
```

The exact keys vary by service. For example, RDS, EFS, DynamoDB, and EC2-related restores each require different metadata. Always verify the service documentation or test in a nonproduction account before relying on a runbook.

**Disaster recovery runbooks**

A DR runbook is the step-by-step operational guide for restoring service after a failure. It should be executable by someone who is not the original author.

A good runbook includes:

- **Scope**
  - Which application, account, Region, and dependencies it covers.
- **Triggers**
  - What event causes DR activation.
  - Who declares the incident.
- **Roles**
  - Incident commander.
  - Application owner.
  - Network/security approver.
  - Database or platform operator.
- **Prerequisites**
  - Required IAM roles.
  - KMS key access.
  - DNS permissions.
  - Secrets and parameter store access.
- **Recovery sequence**
  - Restore data.
  - Recreate infrastructure.
  - Repoint traffic.
  - Validate application health.
- **Rollback**
  - How to return to the primary site if needed.
- **Evidence**
  - Logs, screenshots, timestamps, and ticket references.

A runbook should be written for the actual failure mode, not just the happy path. For example, a Region outage runbook differs from a single-database corruption runbook.

**Common DR patterns**

Choose the pattern based on recovery time objective (RTO) and recovery point objective (RPO).

| Pattern | RTO | RPO | Notes |
|---|---:|---:|---|
| Backup and restore | High | Moderate to high | Lowest cost, slowest recovery |
| Pilot light | Lower | Low to moderate | Core services are pre-provisioned |
| Warm standby | Low | Low | Smaller active environment in DR Region |
| Multi-site active/active | Very low | Very low | Highest cost and complexity |

Use backup and restore when cost matters more than speed. Use warm standby or active/active when downtime is expensive or unacceptable.

**Runbook example: regional failover**

A Region failover runbook often follows this sequence:

1. Declare the incident and freeze nonessential changes.
2. Confirm the primary Region is unavailable or unsafe.
3. Promote or restore data in the DR Region.
4. Recreate application infrastructure from IaC (Infrastructure as Code).
5. Update DNS, load balancers, or traffic routing.
6. Validate authentication, data access, and business transactions.
7. Monitor for errors and performance regressions.
8. Document the event and schedule a post-incident review.

Example Route 53 failover update:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch file://failover-dns.json
```

Example infrastructure deployment step:

```bash
terraform apply -var-file=dr.tfvars
```

The runbook should specify whether you restore from the latest backup, the latest consistent point, or a known-good snapshot. That decision depends on data loss tolerance and application consistency requirements.

**Governance for compliance and audit**

Compliance teams usually care about proof, not promises. Build evidence into the backup process.

Useful evidence sources:

- **AWS Backup job history** for backup and restore completion.
- **CloudTrail** for API calls such as plan changes, vault updates, and restore requests.
- **AWS Config** for configuration drift and resource compliance.
- **Tagging reports** for backup coverage by application or environment.
- **Restore test reports** with timestamps and success criteria.

Operational controls to document:

- Backup frequency and retention.
- Cross-account and cross-Region copy targets.
- Encryption key ownership and rotation policy.
- Vault access policy and lock settings.
- Restore test cadence and results.
- DR declaration authority and communication path.

**Delegated administration and Organizations**

AWS Backup supports delegated administration through AWS Organizations, which lets a central security or platform team manage backup policies across member accounts after setup. This is useful for standardization, but it is not unlimited central control.

Keep these constraints in mind:

- Delegated admin must be enabled in Organizations.
- Some actions remain account- and Region-scoped.
- Service-specific permissions still apply.
- Resource owners may need local roles for restore validation or application-level checks.

A common operating model is:

- Central team defines backup policies and compliance checks.
- Application teams own restore testing and application validation.
- Security team reviews vault policies, KMS usage, and audit evidence.

**Practical design checklist**

Use this checklist when designing or reviewing a backup and DR solution:

- Define RTO and RPO for each application tier.
- Classify data by criticality and retention requirement.
- Choose the right protection mechanism for each service.
- Separate backup administration from workload administration.
- Encrypt backups with customer managed KMS keys where required.
- Copy critical recovery points to another account or Region.
- Protect vaults with access policies and, where appropriate, Vault Lock.
- Test restores on a schedule.
- Document service-specific restore metadata and dependencies.
- Rehearse failover and failback.
- Capture evidence for audit and compliance.

**Exam focus points**

For SAP-C02, expect questions that test tradeoffs and operational realism.

- **Governance** is about policy, separation of duties, and auditability.
- **Restore testing** is required to prove recoverability.
- **Runbooks** must be specific, current, and executable.
- **Cross-account and cross-Region copies** improve resilience and isolation.
- **Vault Lock** strengthens retention controls but must be planned carefully.
- **Service-specific restore behavior** matters more than generic backup assumptions.
- **Organizations and delegated admin** simplify central management, but they do not remove all local constraints.

**Quick reference commands**

List backup plans:

```bash
aws backup list-backup-plans
```

List recovery points in a vault:

```bash
aws backup list-recovery-points-by-backup-vault \
  --backup-vault-name prod-vault
```

Start a restore job:

```bash
aws backup start-restore-job \
  --recovery-point-arn arn:aws:backup:us-east-1:111122223333:recovery-point:abcd1234 \
  --metadata file://restore-metadata.json \
  --iam-role-arn arn:aws:iam::111122223333:role/AWSBackupDefaultServiceRole
```

Inspect restore jobs:

```bash
aws backup list-restore-jobs
```

Check vault lock settings:

```bash
aws backup describe-backup-vault \
  --backup-vault-name prod-vault
```

**Key takeaway**

Backups are only valuable when they are governed, protected, tested, and operationalized. For SAP-C02, focus on the full lifecycle: define policy, enforce retention, isolate copies, validate restores, and rehearse DR execution with clear runbooks.
