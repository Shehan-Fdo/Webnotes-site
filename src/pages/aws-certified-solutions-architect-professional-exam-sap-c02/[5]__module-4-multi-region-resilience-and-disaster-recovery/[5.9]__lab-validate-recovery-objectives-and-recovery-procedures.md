## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Recovery objectives and why validation matters**

Recovery planning is only useful if you can prove it works under realistic failure conditions. In AWS, you validate not just that backups exist, but that you can restore data, fail over traffic, rebuild infrastructure, and meet business targets for recovery time and data loss.

Two core metrics guide the lab:

- **RTO** (Recovery Time Objective): the maximum acceptable time to restore service.
- **RPO** (Recovery Point Objective): the maximum acceptable amount of data loss, measured in time before the failure.

A good validation exercise checks both technical recovery and user-visible application behavior. That means you test compute, database, storage, DNS, IAM permissions, and automation together instead of treating them as isolated components.

**Recovery strategy selection**

Different AWS services support different recovery patterns. You should choose the pattern that matches the workload criticality, data change rate, and acceptable downtime.

- **Backup and restore** is the simplest pattern and usually has the longest RTO.
- **Pilot light** keeps core data and minimal services ready for scale-up.
- **Warm standby** keeps a scaled-down but functional environment running.
- **Multi-site active/active** reduces RTO and can reduce RPO, but it is more complex and expensive.

For exam scenarios, always map the business requirement to the least complex pattern that still meets the target.

**Prerequisites to validate before you test recovery**

Before you run a recovery drill, confirm that the environment can actually be restored. Many failed exercises are caused by missing permissions or incomplete backup configuration rather than by the recovery procedure itself.

- **Backup retention** is set long enough to cover the recovery window you need.
- **Snapshot schedules** or continuous backups are enabled where appropriate.
- **Restore permissions** exist for the operator role and any automation role.
- **KMS** (Key Management Service) keys allow decrypt and re-encrypt operations during restore.
- **Network dependencies** such as subnets, security groups, route tables, and DNS records are available in the target Region or account.
- **Quotas** are sufficient for launching replacement instances, volumes, or databases.
- **Tags and naming conventions** are consistent so automation can find the right resources.

A practical validation checklist often starts with permissions because restore failures frequently come from missing `iam:PassRole`, `rds:RestoreDBInstanceFromDBSnapshot`, `ec2:RunInstances`, or `backup:StartRestoreJob` access.

**Measuring RTO and RPO in a lab**

You should measure recovery from the user’s perspective, not just from the infrastructure console. Start timing when the failure is detected and stop when the application is usable again.

A simple measurement flow is:

1. Record the failure time and the last confirmed healthy transaction.
2. Trigger the recovery action.
3. Measure the time until the application passes health checks.
4. Measure the time until a real user transaction succeeds.
5. Compare the last committed data point with the failure point to estimate data loss.

> Do not treat “restore to an earlier timestamp” as the RPO itself. RPO is the amount of data you lose relative to the failure event, not just the difference between two timestamps.

A useful validation example is to submit a test order, write a database row, and confirm that the order appears after recovery. That proves more than a simple `/health` endpoint.

**End-to-end multi-tier recovery scenario**

A strong lab exercise ties together compute, database, and DNS failover. Use a simple three-tier application:

- **Web tier** on Amazon EC2 behind an Application Load Balancer.
- **Database tier** on Amazon RDS for MySQL or PostgreSQL.
- **DNS** on Amazon Route 53 with failover routing.

A typical primary/secondary design looks like this:

```text
Users
  -> Route 53 failover record
    -> Primary ALB in us-east-1
      -> EC2 Auto Scaling group
        -> RDS primary database
    -> Secondary ALB in us-west-2
      -> EC2 Auto Scaling group
        -> Restored or replicated database
```

To validate the design, you can simulate a primary Region outage by stopping the web tier, failing over the database, and switching DNS to the secondary stack. The important part is that the application still completes a user transaction after the failover.

**Route 53 failover routing and DNS considerations**

Route 53 failover routing is a common way to direct traffic to a healthy endpoint. You pair it with health checks so DNS can switch when the primary endpoint becomes unhealthy.

Key points to validate:

- **Failover records** use primary and secondary record sets.
- **Health checks** determine whether Route 53 considers the primary endpoint healthy.
- **TTL** (Time To Live) affects how quickly clients see the new record.
- **Low TTL values** reduce failover delay but increase DNS query frequency.
- **Client caching** can still delay cutover even when Route 53 updates immediately.

Example Route 53 record behavior:

```text
Primary A record
- Alias to primary ALB
- Health check enabled
- TTL: 60 seconds

Secondary A record
- Alias to secondary ALB
- Set as failover target
- No health check required if it is the backup target
```

When testing, remember that DNS failover is not instant for every client. Some resolvers cache responses longer than expected, so you should validate from multiple clients or use application-level routing where strict failover timing matters.

**Backup and restore validation**

AWS Backup is the most common service for centralized backup policy management across supported AWS resources. In a lab, validate both the backup plan and the restore path.

What to verify:

- **Backup plan** schedules run on time.
- **Lifecycle rules** move recovery points to cold storage only when appropriate.
- **Vault access policy** allows the restore role to read recovery points.
- **Cross-Region copy** exists if the recovery design requires regional isolation.
- **Cross-account copy** exists if you need blast-radius reduction.
- **Restore testing** is performed regularly, not just backup creation.

A basic AWS Backup restore flow:

```bash
aws backup list-backup-vaults
aws backup list-recovery-points-by-backup-vault --backup-vault-name PrimaryVault
aws backup start-restore-job \
  --recovery-point-arn arn:aws:backup:us-east-1:123456789012:recovery-point:example \
  --metadata file://restore-metadata.json \
  --iam-role-arn arn:aws:iam::123456789012:role/AWSBackupRestoreRole
```

A restore metadata file varies by resource type. For example, an EC2 restore needs instance and network details, while an RDS restore needs database-specific parameters.

**RDS and Aurora recovery validation**

Amazon RDS supports snapshot restore and point-in-time recovery for supported engines. Aurora uses the `rds` API namespace for many operations, even though it is a distinct service family.

Validate these patterns:

- **Snapshot restore** for known-good backups.
- **Point-in-time recovery** for accidental deletes or bad writes.
- **Multi-AZ** failover for instance-level resilience.
- **Aurora failover** within a Region for writer replacement.
- **Aurora Global Database** for cross-Region disaster recovery.

For Aurora, `aws rds failover-db-cluster` forces a failover within the same Region. It does not perform cross-Region recovery. For cross-Region recovery in Aurora Global Database, you promote the secondary cluster or follow the global database recovery procedure.

Example commands:

```bash
aws rds describe-db-snapshots --db-instance-identifier app-db
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier app-db-restore \
  --db-snapshot-identifier app-db-snapshot-2026-04-26

aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier app-db \
  --target-db-instance-identifier app-db-pitr \
  --restore-time 2026-04-26T10:15:00Z
```

For Aurora failover testing:

```bash
aws rds failover-db-cluster --db-cluster-identifier app-aurora-cluster
```

Use this only to test writer promotion inside the same Region. For cross-Region recovery, validate the secondary Region promotion path separately.

**EFS recovery and file-level restoration**

Amazon EFS does not provide a native “restore from backup” command in the same way as RDS. Recovery is usually done by restoring files from AWS Backup or by using EFS replication to another file system, then copying data back if needed.

Practical recovery patterns:

- **AWS Backup restore** to a new EFS file system.
- **EFS replication** to a secondary file system in another Region.
- **Application-level copy back** from the restored or replicated file system.

Example validation approach:

```bash
aws backup start-restore-job \
  --recovery-point-arn arn:aws:backup:us-east-1:123456789012:recovery-point:efs-example \
  --metadata file://efs-restore-metadata.json \
  --iam-role-arn arn:aws:iam::123456789012:role/AWSBackupRestoreRole
```

After restore, mount the recovered file system on a test instance and verify file integrity, permissions, and application startup behavior.

**S3 recovery, versioning, and Object Lock**

Amazon S3 recovery depends heavily on versioning. If versioning is enabled, you can recover prior object versions after overwrite or delete operations. If Object Lock is enabled, retention or legal hold can prevent deletion or overwrite of protected versions, but Object Lock alone does not guarantee recovery unless the needed version still exists and versioning is enabled.

Validate these behaviors:

- **Versioning** preserves prior object versions.
- **Delete markers** can hide objects without removing older versions.
- **Object Lock** protects versions from deletion or overwrite during retention.
- **Replication** can provide a secondary copy in another Region or account.

Useful commands:

```bash
aws s3api put-bucket-versioning \
  --bucket my-app-bucket \
  --versioning-configuration Status=Enabled

aws s3api list-object-versions --bucket my-app-bucket --prefix reports/q1.csv
aws s3api get-object \
  --bucket my-app-bucket \
  --key reports/q1.csv \
  --version-id 3HL4kqtJlcpXroDTDmjVBH40Nrjfkd
```

If you test Object Lock, confirm that the bucket was created with Object Lock enabled and that the retention mode matches the scenario. A locked object is not the same thing as a recoverable backup strategy.

**EC2, EBS, and infrastructure rebuilds**

For compute recovery, you usually restore from AMIs, EBS snapshots, or infrastructure-as-code rather than trying to repair a failed instance in place.

Validate these items:

- **AMI** (Amazon Machine Image) captures the OS and application baseline.
- **EBS** snapshots preserve attached volume data.
- **Launch templates** define repeatable instance configuration.
- **Auto Scaling groups** can replace unhealthy instances automatically.
- **User data** can bootstrap application configuration on launch.

Example commands:

```bash
aws ec2 create-image --instance-id i-0123456789abcdef0 --name app-ami-2026-04-26
aws ec2 describe-images --owners self
aws ec2 create-snapshots --instance-specification InstanceId=i-0123456789abcdef0
```

For recovery testing, terminate one instance in a controlled environment and confirm that the Auto Scaling group replaces it and the application rejoins the load balancer.

**Infrastructure as Code and repeatability**

A recovery procedure is much stronger when you can rebuild the environment from code. Use CloudFormation, AWS CDK, or Terraform to recreate the stack after a failure.

Why this matters:

- **Repeatability** reduces manual error during recovery.
- **Drift detection** helps you find differences between intended and actual state.
- **Version control** gives you a known-good baseline.
- **Automation** shortens RTO and improves consistency.

Example CloudFormation deployment flow:

```bash
aws cloudformation deploy \
  --stack-name app-dr \
  --template-file template.yaml \
  --capabilities CAPABILITY_NAMED_IAM
```

A good lab exercise is to delete the stack in a sandbox account and rebuild it from code, then verify that the application still works with restored data.

**Systems Manager Automation and runbooks**

AWS Systems Manager Automation helps you standardize recovery steps. Use it for repeatable operational workflows such as instance replacement, snapshot restore orchestration, or post-restore validation.

In Automation, use supported Automation actions such as `aws:runCommand` to invoke a Command document. Do not confuse Automation steps with Run Command plugin actions.

Example Automation document pattern:

```yaml
schemaVersion: '0.3'
description: "Restore application host and validate service"
assumeRole: '{{ AutomationAssumeRole }}'
parameters:
  InstanceId:
    type: String
  AutomationAssumeRole:
    type: String
mainSteps:
  - name: RunHealthCheck
    action: aws:runCommand
    inputs:
      DocumentName: AWS-RunShellScript
      InstanceIds:
        - '{{ InstanceId }}'
      Parameters:
        commands:
          - 'curl -fsS http://localhost:8080/health'
```

Use Automation when you want a documented, auditable recovery runbook that operators can execute consistently.

**Elastic Disaster Recovery and when to use it**

AWS Elastic Disaster Recovery is useful when you need low RTO and low RPO for server-based workloads. It continuously replicates source servers to a staging area in AWS and can launch recovery instances quickly.

Use it when:

- You need near-continuous replication for physical, virtual, or cloud servers.
- You want faster failover than snapshot-based recovery.
- You need a generic server recovery option across heterogeneous workloads.

It is especially useful for lift-and-shift disaster recovery scenarios where application refactoring is not practical before the recovery design must be in place.

**Validating permissions and access paths**

Recovery often fails because the operator cannot perform the restore action. Validate permissions for both humans and automation roles.

Common permission areas:

- **Backup** actions such as listing vaults and starting restore jobs.
- **RDS** restore and describe actions.
- **EC2** launch, describe, and snapshot actions.
- **Route 53** record changes if DNS cutover is automated.
- **KMS** decrypt, encrypt, and grant creation.
- **IAM** pass role for services that assume a recovery role.

Example policy fragments:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowRDSRecovery",
      "Effect": "Allow",
      "Action": [
        "rds:DescribeDBSnapshots",
        "rds:RestoreDBInstanceFromDBSnapshot",
        "rds:RestoreDBInstanceToPointInTime",
        "rds:FailoverDBCluster"
      ],
      "Resource": "*"
    },
    {
      "Sid": "AllowBackupRestore",
      "Effect": "Allow",
      "Action": [
        "backup:ListBackupVaults",
        "backup:ListRecoveryPointsByBackupVault",
        "backup:StartRestoreJob"
      ],
      "Resource": "*"
    }
  ]
}
```

For Aurora and other RDS-managed services, use the `rds` namespace rather than a made-up service prefix.

**Testing a full recovery workflow**

A complete recovery test should validate the application, not just the infrastructure. Use a repeatable workflow that includes user-facing transactions.

Suggested procedure:

1. Create a baseline transaction, such as placing an order or writing a record.
2. Trigger a controlled failure in the primary tier.
3. Start the recovery action for compute, database, or both.
4. Switch DNS or load balancer traffic to the recovered environment.
5. Re-run the same transaction and confirm the data is present.
6. Compare the elapsed time to the RTO target.
7. Compare the last successful write to the failure point to estimate RPO.

Example validation script:

```bash
#!/usr/bin/env bash
set -euo pipefail

start=$(date +%s)

curl -fsS https://app.example.com/health
curl -fsS -X POST https://app.example.com/orders \
  -H 'Content-Type: application/json' \
  -d '{"orderId":"dr-test-001","item":"widget","qty":1}'

curl -fsS https://app.example.com/orders/dr-test-001

end=$(date +%s)
echo "Elapsed seconds: $((end - start))"
```

This kind of test proves that the application can accept writes and read them back after recovery.

**Common failure modes during recovery drills**

Recovery tests often fail for predictable reasons. Watch for these issues:

- **Stale DNS** causes clients to keep using the old endpoint.
- **Missing KMS grants** block restore or decryption.
- **Security group rules** do not allow the recovered app to reach the database.
- **Subnet exhaustion** prevents new instances from launching.
- **Parameter group drift** changes database behavior after restore.
- **Application secrets** are not available in the target Region.
- **Backup retention gaps** leave no usable recovery point.

When a drill fails, document the exact step, the error message, and the remediation. That record becomes part of the runbook.

**How to think about exam scenarios**

For SAP-C02, the best answer usually balances resilience, cost, and operational complexity.

- Choose **AWS Backup** for centralized backup governance and restore testing.
- Choose **Aurora Global Database** when you need fast cross-Region database recovery for Aurora workloads.
- Choose **Elastic Disaster Recovery** for fast server recovery with continuous replication.
- Choose **Route 53 failover routing** when DNS-based traffic shifting is acceptable.
- Choose **CloudFormation/CDK/Terraform** when rebuild speed and repeatability matter.
- Choose **Multi-AZ** for high availability within a Region, not for regional disaster recovery.

If a question asks about validating recovery, focus on proving the procedure end to end: data, compute, traffic routing, permissions, and user transaction success.

**Quick reference checklist**

- Confirm backup retention and restore permissions.
- Test restore into a clean environment.
- Validate DNS failover and TTL behavior.
- Measure actual RTO with a real transaction.
- Estimate RPO from the last committed write before failure.
- Rebuild infrastructure from code when possible.
- Document every failure and remediation step.
