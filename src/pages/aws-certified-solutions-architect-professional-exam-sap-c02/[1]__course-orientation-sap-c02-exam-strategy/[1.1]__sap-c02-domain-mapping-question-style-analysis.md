## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**What SAP-C02 is really testing**

SAP-C02 is not a service-recognition exam. It tests whether you can design, migrate, govern, and optimize AWS environments at scale while balancing cost, security, resilience, and operational simplicity. The best answers usually reflect enterprise constraints: multiple accounts, hybrid connectivity, shared services, delegated administration, and long-term operability.

The exam often gives you several valid AWS solutions. Your job is to identify the one that best satisfies the stated constraints with the fewest tradeoffs. That means you must map each question to the blueprint domain first, then filter by hidden requirements such as compliance, latency, blast radius, migration timeline, or organizational boundaries.

**SAP-C02 domain map**

The exam blueprint is commonly organized into four domains. A practical way to study is to map each domain to the AWS design decisions it tends to test.

| Domain | What it emphasizes | Common decision areas |
|---|---|---|
| **Design for Organizational Complexity** | Multi-account governance, identity, network segmentation, shared services | AWS Organizations, SCPs, IAM Identity Center, AWS RAM, Control Tower, Service Catalog, tagging, delegated admin |
| **Design for New Solutions** | Greenfield architecture choices for compute, storage, database, networking, and resilience | VPC design, subnets, routing, ALB/NLB, Auto Scaling, ECS/EKS/Lambda, RDS/Aurora, S3, CloudFront, Route 53, Multi-AZ/Region |
| **Migration Planning** | Assessing, prioritizing, and modernizing workloads | Application portfolio assessment, dependency mapping, 6 Rs, database migration, hybrid connectivity, cutover planning, DMS, MGN |
| **Cost Control** | Cost optimization without breaking reliability or governance | Savings Plans, Reserved Instances, right-sizing, S3 lifecycle, storage classes, Trusted Advisor, Well-Architected, chargeback/showback |

**How to read the blueprint in practice**

Each domain is really a set of subtopics that recur in different forms. For example, networking appears in all four domains, but the angle changes:

- In **organizational complexity**, networking is about account boundaries, shared VPC patterns, centralized inspection, and cross-account access.
- In **new solutions**, networking is about choosing the right topology for availability, latency, and security.
- In **migration planning**, networking is about coexistence, hybrid connectivity, and minimizing cutover risk.
- In **cost control**, networking is about reducing unnecessary data transfer, overprovisioning, and duplicated services.

**Domain 1: Design for Organizational Complexity**

This domain focuses on how you structure AWS for many teams, accounts, and governance requirements. The exam expects you to know how AWS Organizations, IAM, and shared services work together.

**Core subtopics**

- **AWS Organizations**
  - Use organizational units (OUs) to group accounts by environment, business unit, or compliance boundary.
  - Use service control policies (SCPs) to set maximum permissions across accounts.
  - Remember that SCPs do not grant permissions; they only filter what IAM can allow.
  - SCP inheritance applies from the root to OUs and accounts, so a restrictive policy at a parent level affects all children.

- **Identity and access**
  - Use IAM Identity Center for workforce access across multiple accounts.
  - Use IAM roles for cross-account access instead of long-lived IAM users.
  - Use permission boundaries when you need to limit what delegated administrators can create.
  - Use resource-based policies when the service supports them and cross-account access is simpler that way.

- **Shared services and delegation**
  - Use AWS RAM to share resources such as subnets, Transit Gateways, and other supported resources across accounts.
  - Use delegated administrator accounts for services like Security Hub, GuardDuty, and AWS Config where appropriate.
  - Use AWS Service Catalog to provide approved, repeatable infrastructure products to internal teams.
  - Use Control Tower when you need landing zones, guardrails, and account vending at scale.

- **Governance and operations**
  - Use AWS Config for configuration drift detection and compliance rules.
  - Use CloudTrail for audit logging across accounts and Regions.
  - Use AWS Trusted Advisor for best-practice checks in cost, security, fault tolerance, and service limits.
  - Use tagging standards for cost allocation, automation, and ownership.

**Important exam nuance**

SCPs are often tested with inheritance traps. If an SCP denies `ec2:TerminateInstances` at the root, no child account can override that deny with IAM. If a question asks for a way to let one team bypass a restriction, the answer is usually to move that team into a different OU or adjust the SCP structure, not to add a more permissive IAM policy.

**Example: SCP guardrail**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyLeavingOrg",
      "Effect": "Deny",
      "Action": "organizations:LeaveOrganization",
      "Resource": "*"
    },
    {
      "Sid": "DenyDisallowedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

This pattern is common when the organization wants to enforce Region restrictions and prevent account escape. The exam may ask you to choose between IAM policy, SCP, and Config rule; the correct answer depends on whether you need preventive control at the organization level or detective control after deployment.

**Domain 2: Design for New Solutions**

This domain tests whether you can build a secure, resilient, and scalable architecture from scratch. It is the most architecture-heavy domain and often includes networking, compute, storage, database, and edge design in one scenario.

**Networking subtopics**

- **VPC design**
  - Place public subnets for internet-facing load balancers and NAT gateways.
  - Place application and database tiers in private subnets.
  - Use separate subnets per Availability Zone for high availability.
  - Use route tables to control traffic flow between subnets and to the internet.

- **Security groups vs network ACLs**
  - Use security groups as stateful instance- or ENI-level firewalls.
  - Use network ACLs as stateless subnet-level filters.
  - Prefer security groups for most workload controls.
  - Use network ACLs when you need coarse subnet guardrails or explicit deny behavior.

- **Load balancing and ingress**
  - Use Application Load Balancer for HTTP/HTTPS, path-based routing, and host-based routing.
  - Use Network Load Balancer for ultra-low latency, static IPs, and TCP/UDP workloads.
  - Use Gateway Load Balancer for third-party network appliances.
  - Use AWS WAF with CloudFront or ALB for web application filtering.

- **Hybrid connectivity**
  - Use Site-to-Site VPN for quick setup, encryption over the internet, and lower throughput needs.
  - Use Direct Connect for predictable latency, higher bandwidth, and private connectivity.
  - Use Direct Connect plus VPN for resilience and failover.
  - Use Transit Gateway to simplify hub-and-spoke routing across many VPCs and on-premises networks.

- **Private service access**
  - Use VPC endpoints to keep traffic on the AWS network.
  - Use Gateway endpoints for S3 and DynamoDB.
  - Use Interface endpoints for services such as Systems Manager, Secrets Manager, and many API-based services.
  - Use PrivateLink when you need private access to a service or to expose your own service privately across accounts or VPCs.

**Decision example: Transit Gateway vs PrivateLink**

- Choose **Transit Gateway** when you need broad network connectivity between VPCs, accounts, and on-premises networks.
- Choose **PrivateLink** when you want to expose a specific service privately without full network routing between environments.
- Do not use Transit Gateway when the requirement is only to publish one internal API to a consumer account.

**Compute subtopics**

- **EC2**
  - Use Auto Scaling groups for elasticity and self-healing.
  - Use placement groups when workload performance depends on low latency or high throughput.
  - Use Spot Instances for interruption-tolerant workloads.
  - Use Launch Templates for versioned instance configuration.

- **Containers**
  - Use ECS when you want simpler AWS-native orchestration.
  - Use EKS when Kubernetes compatibility or ecosystem requirements matter.
  - Use Fargate when you want serverless container execution and reduced operational overhead.

- **Serverless**
  - Use Lambda for event-driven, short-running, stateless tasks.
  - Use Step Functions for orchestration, retries, branching, and long-running workflows.
  - Use EventBridge for event routing across services and accounts.

**Storage subtopics**

- **S3**
  - Use S3 for durable object storage, static content, backups, and data lakes.
  - Use versioning to protect against accidental overwrite and deletion.
  - Use lifecycle policies to transition objects to cheaper classes.
  - Use replication for cross-Region resilience or compliance.

- **EBS and EFS**
  - Use EBS for block storage attached to a single EC2 instance.
  - Use EFS for shared POSIX file storage across multiple instances.
  - Use EBS snapshots for backup and restore.
  - Use EFS One Zone only when you can tolerate lower resilience in exchange for lower cost.

- **FSx**
  - Use FSx for Windows File Server when you need SMB and Windows integration.
  - Use FSx for Lustre for high-performance compute workloads.
  - Use FSx for NetApp ONTAP when enterprise file features and hybrid integration matter.

**Database subtopics**

- **RDS and Aurora**
  - Use RDS for managed relational databases with familiar engines.
  - Use Aurora when you need higher performance, fast failover, and advanced scaling options.
  - Use Multi-AZ for high availability.
  - Use read replicas for read scaling and cross-Region replication where supported.

- **DynamoDB**
  - Use DynamoDB for key-value and document workloads with predictable access patterns and massive scale.
  - Use on-demand capacity for spiky or unknown traffic.
  - Use global tables for multi-Region active-active designs.

- **Database modernization**
  - Rehost the database when the goal is speed and minimal change.
  - Replatform when you can make limited managed-service changes, such as moving to RDS.
  - Refactor when the application can be redesigned for cloud-native patterns.
  - Re-architect database access when schema, transaction, or scaling constraints block the target state.

**Example: resilient web application topology**

```text
Users
  -> Route 53
  -> CloudFront
  -> AWS WAF
  -> Application Load Balancer
  -> Auto Scaling group in private subnets across 2+ AZs
  -> Aurora Multi-AZ
  -> S3 for static assets and backups
```

This pattern is common because it separates edge protection, ingress, compute, and data tiers. The exam may ask you to improve availability, reduce latency, or lower operational burden. The best answer usually preserves managed services and avoids unnecessary custom failover logic.

**Security and encryption**

- Use KMS for encryption keys and key policy control.
- Use envelope encryption for large data sets and service integrations.
- Use Secrets Manager for rotating secrets and database credentials.
- Use Systems Manager Parameter Store for configuration values and simpler secret storage needs.
- Use IAM roles for workloads instead of embedding credentials.

**Domain 3: Migration Planning**

This domain tests whether you can assess a portfolio, choose the right migration strategy, and reduce risk during transition. It is not just about moving servers; it is about deciding what to move, what to modernize, and what to retire.

**Core subtopics**

- **Application portfolio assessment**
  - Identify business criticality, technical dependencies, compliance requirements, and operational pain points.
  - Classify applications by complexity, coupling, and modernization potential.
  - Determine whether the workload is a good candidate for rehost, replatform, refactor, retire, retain, or replace.

- **Dependency mapping**
  - Map upstream and downstream dependencies before cutover.
  - Include network flows, identity dependencies, database calls, batch jobs, file shares, and external APIs.
  - Use dependency mapping to avoid moving one tier while leaving a hidden dependency behind.

- **The 6 Rs**
  - **Rehost**: lift and shift with minimal change.
  - **Replatform**: make limited optimizations while moving.
  - **Refactor**: redesign for cloud-native architecture.
  - **Repurchase**: move to a SaaS product.
  - **Retire**: decommission unused workloads.
  - **Retain**: keep the workload where it is for now.

- **Migration tooling**
  - Use AWS Application Migration Service (MGN) for server replication and cutover.
  - Use AWS Database Migration Service (DMS) for database migration and ongoing replication.
  - Use Snowball or Snowmobile when network transfer is impractical for large data sets.
  - Use DataSync for online file transfer and ongoing synchronization.

- **Hybrid connectivity**
  - Use VPN for quick, lower-cost connectivity during migration.
  - Use Direct Connect for stable, high-throughput migration pipelines.
  - Use Transit Gateway to centralize routing between on-premises and multiple VPCs.

**Modernization choices**

The exam often asks which migration path is best for a given workload. The right answer depends on time, risk, and target architecture.

- Choose **rehost** when you need speed, the app is stable, and the business wants minimal change.
- Choose **replatform** when a small managed-service change gives meaningful operational benefit, such as moving from self-managed MySQL to RDS.
- Choose **refactor** when the app must scale differently, integrate with events, or eliminate monolithic bottlenecks.
- Choose **replace** when SaaS provides a better fit than rebuilding or migrating.

**Database migration examples**

- Use **DMS** with ongoing replication when you need minimal downtime for a relational database cutover.
- Use **Aurora** as a target when you want managed HA and better scaling than a self-managed database.
- Use **RDS** when the source engine is supported and you want a straightforward managed target.
- Use **DynamoDB** when the access pattern is key-based and the application can be redesigned for NoSQL.

**Example: migration decision flow**

1. Assess the application portfolio and rank workloads by business value and migration complexity.
2. Map dependencies, including identity, DNS, data flows, and batch schedules.
3. Choose the 6 R for each workload.
4. Select the migration tool and target service.
5. Design the cutover, rollback, and validation plan.
6. Execute pilot migrations before large-scale waves.

**Practical cutover pattern**

```bash
# Example: verify replication status during a database migration
aws dms describe-replication-tasks \
  --filters Name=replication-task-id,Values=my-task

# Example: inspect server replication readiness in MGN
aws mgn describe-source-servers
```

**Domain 4: Cost Control**

This domain is about reducing spend while preserving the architecture’s required performance, resilience, and governance. The exam often frames cost as a tradeoff against availability or operational overhead.

**Core subtopics**

- **Compute optimization**
  - Use Savings Plans for steady compute usage.
  - Use Reserved Instances when the workload is predictable and the commitment fits the service.
  - Use Spot Instances for fault-tolerant batch, CI/CD, or stateless workloads.
  - Right-size instances based on utilization metrics, not guesswork.

- **Storage optimization**
  - Use S3 lifecycle policies to transition data to cheaper classes.
  - Use Intelligent-Tiering when access patterns are unknown or change over time.
  - Use EBS volume right-sizing and snapshot lifecycle management.
  - Use EFS and FSx tiers appropriately for performance and cost.

- **Network cost control**
  - Reduce cross-AZ and cross-Region data transfer where possible.
  - Use VPC endpoints to avoid NAT gateway data processing charges for supported services.
  - Use CloudFront to reduce origin load and improve edge delivery.
  - Use centralized egress carefully; it can improve governance but increase data transfer costs if poorly designed.

- **Governance and visibility**
  - Use AWS Cost Explorer to analyze trends.
  - Use AWS Budgets for alerts and threshold-based controls.
  - Use Cost and Usage Reports for detailed billing analysis.
  - Use Trusted Advisor and Well-Architected reviews to identify waste and risk.

**Well-Architected and Trusted Advisor**

The **AWS Well-Architected Framework** is a recurring exam concept because it provides a structured way to evaluate architecture across operational excellence, security, reliability, performance efficiency, cost optimization, and sustainability. When a question asks for the “best” architecture, the answer often aligns with Well-Architected principles: managed services, automation, least privilege, and failure isolation.

**AWS Trusted Advisor** is useful when the question is about identifying idle resources, underutilized instances, open security groups, or service limits. It is not a design tool by itself, but it is a strong operational and cost-optimization signal.

**Question style analysis**

SAP-C02 questions are usually scenario-based and intentionally dense. They often include extra details that are not equally important. Your task is to identify the constraint that drives the answer.

**Common question patterns**

- **Best answer among valid answers**
  - Several options may work technically.
  - Choose the one that best satisfies the stated requirement with the least operational burden.
  - Prefer managed services over self-managed components when the question emphasizes reduced administration.

- **Multi-answer tradeoff traps**
  - Two answers may each solve part of the problem.
  - The correct choice usually covers all requirements, not just the most obvious one.
  - Watch for hidden constraints such as encryption, auditability, cross-account access, or failover.

- **Constraint prioritization**
  - If the question says “lowest operational overhead,” favor managed services.
  - If it says “lowest latency,” favor regional proximity, edge caching, or direct connectivity.
  - If it says “minimal downtime,” favor replication, blue/green, or staged cutover.
  - If it says “least cost,” ensure the answer still meets resilience and performance requirements.

- **Architecture pattern recognition**
  - Many questions are asking you to recognize a standard pattern:
    - multi-account landing zone
    - hub-and-spoke networking
    - active-active or active-passive failover
    - hybrid migration with replication
    - private service exposure with PrivateLink
    - centralized logging and security monitoring

**How to eliminate wrong answers**

- Reject answers that add unnecessary custom code when AWS managed services already solve the problem.
- Reject answers that violate the stated security boundary, such as exposing private services publicly.
- Reject answers that solve the immediate issue but break a hidden requirement like compliance, audit logging, or multi-account governance.
- Reject answers that require manual operations when automation is clearly expected.
- Reject answers that improve one metric while worsening the explicit exam constraint.

**Example: choosing the best networking answer**

A question asks for private connectivity from multiple consumer VPCs to a shared internal API without full network routing.

- **Wrong fit**: VPC peering, because it scales poorly and creates many route relationships.
- **Wrong fit**: Transit Gateway, because it provides broad routing rather than service-level exposure.
- **Best fit**: PrivateLink, because it exposes only the service privately and limits blast radius.

**Example: choosing the best migration answer**

A question asks for a low-risk migration of a stable VM-based application with minimal code changes.

- **Wrong fit**: Refactor, because it adds time and risk.
- **Best fit**: Rehost, because it preserves behavior and accelerates migration.
- **Possible alternative**: Replatform, if the question explicitly wants a managed database or small modernization step.

**Exam wording traps**

- **“Must” vs “should”**
  - “Must” usually indicates a hard constraint.
  - “Should” may allow a more flexible design.

- **“Least operational overhead”**
  - Prefer managed services, automation, and serverless where appropriate.

- **“Highly available”**
  - Look for Multi-AZ, multi-Region, or self-healing designs depending on the failure domain described.

- **“Securely share”**
  - Consider AWS RAM, resource policies, IAM roles, or PrivateLink before building custom access paths.

- **“Centralize”**
  - This often points to Organizations, Control Tower, Transit Gateway, centralized logging, or shared services accounts.

**Practical exam scenarios**

**Scenario 1: Multi-account governance**

You need to prevent developers from creating resources outside approved Regions while still allowing them to deploy freely inside approved accounts.

- Use AWS Organizations with OUs for environment separation.
- Apply SCPs to deny disallowed Regions.
- Use IAM roles and permission sets for access.
- Use AWS Config to detect drift and noncompliance.
- Use Service Catalog for approved templates.

**Scenario 2: Shared internal service**

A security team wants to expose an internal API to several application accounts without opening VPC-wide routing.

- Publish the service behind an NLB or supported endpoint.
- Expose it with PrivateLink.
- Use resource policies and endpoint acceptance controls where applicable.
- Avoid broad peering or Transit Gateway routing if only one service is needed.

**Scenario 3: Hybrid migration**

A legacy application must move to AWS with minimal downtime, but the database is large and the network link is limited.

- Use Direct Connect if available for stable transfer.
- Use DMS for ongoing replication.
- Use MGN for server replication.
- Use Snowball if data transfer volume exceeds practical network capacity.
- Validate cutover and rollback plans before production migration.

**Scenario 4: Cost reduction without performance loss**

A batch workload runs intermittently and can tolerate interruption.

- Use Spot Instances for compute.
- Store outputs in S3 with lifecycle policies.
- Use CloudWatch metrics to right-size the baseline.
- Use Savings Plans only for the steady portion of usage.

**Quick service decision guide**

| Need | Prefer | Avoid when |
|---|---|---|
| Cross-account resource sharing | AWS RAM | You need full network routing |
| Private service exposure | PrivateLink | You need broad VPC-to-VPC routing |
| Hub-and-spoke networking | Transit Gateway | Only one service needs sharing |
| Fast VM migration | MGN | You need major app redesign |
| Database replication | DMS | The target is not a database migration |
| Governance guardrails | SCPs | You only need per-user permissions |
| Approved internal templates | Service Catalog | Teams need unrestricted self-service |
| Cost anomaly detection | Trusted Advisor, Budgets | You need deep architecture redesign |

**Study strategy for SAP-C02**

- Learn the blueprint as a set of decision domains, not as a list of services.
- Practice mapping each scenario to one primary constraint and one secondary constraint.
- Memorize the “best fit” use cases for core services:
  - Organizations, SCPs, RAM, Control Tower, Service Catalog
  - VPC, Transit Gateway, PrivateLink, VPN, Direct Connect
  - ALB, NLB, CloudFront, WAF, Route 53
  - EC2, Auto Scaling, ECS, EKS, Lambda
  - S3, EBS, EFS, FSx
  - RDS, Aurora, DynamoDB, DMS, MGN
  - Cost Explorer, Budgets, Trusted Advisor, Well-Architected
- Train yourself to reject answers that are technically possible but operationally inferior.

**Final exam mindset**

The SAP-C02 exam rewards architectural judgment. The correct answer is often the one that best balances governance, resilience, security, and cost while minimizing unnecessary complexity. If two answers both work, choose the one that is more scalable, more managed, and more aligned with the stated business constraint.
