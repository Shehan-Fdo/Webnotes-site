## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Lab goal and reference architecture**

This lab focuses on building a centralized audit and logging pattern for a multi-account AWS environment. The usual design separates responsibilities:

- A **log archive account** stores immutable or long-retention logs.
- A **security/audit account** reads and analyzes logs.
- **Source accounts** generate service logs and configuration history.
- Optional **processing services** normalize or forward logs for alerting and analytics.

The main services are:

- **AWS CloudTrail** for API activity and management events.
- **AWS Config** for resource configuration history and compliance.
- **Amazon CloudWatch Logs** for application and service log centralization.
- **Amazon S3** for durable log storage and archive retention.

A strong exam answer explains not only how to centralize logs, but also how to preserve integrity, limit access, and verify that delivery works across accounts and Regions.

**Core design principles**

Centralized logging works best when you separate collection, storage, and analysis. You typically want:

- **Write once, read many** storage for audit evidence.
- **Least privilege** access for source accounts and operators.
- **Organization-wide coverage** for CloudTrail and AWS Config where required.
- **Regional consistency** so new accounts and Regions do not become blind spots.
- **Validation and retention controls** so logs are trustworthy and durable.

> CloudTrail and AWS Config solve different problems. CloudTrail records API activity; AWS Config records resource state and configuration changes. They complement each other rather than replace each other.

**Step-by-step lab flow**

A practical lab usually follows this order:

1. Create or identify an AWS Organizations management account.
2. Create a dedicated logging or audit account.
3. Organize source accounts into an OU if you want scoped governance.
4. Configure an organization trail in CloudTrail.
5. Configure AWS Config recorders and delivery channels in each source account and Region.
6. Centralize CloudWatch Logs using subscription filters and a destination target.
7. Store CloudTrail logs in a hardened S3 bucket with the correct bucket policy.
8. Verify delivery, integrity validation, and cross-account visibility.

**AWS Organizations and account layout**

AWS Organizations is the control plane for multi-account governance. For centralized logging, it helps you enforce consistent setup across accounts.

Typical layout:

- **Management account**
  - Owns the organization.
  - Creates delegated admin roles where supported.
- **Log archive account**
  - Stores CloudTrail logs in S3.
  - May also store AWS Config snapshots and CloudWatch Logs exports.
- **Security tooling account**
  - Hosts SIEM, dashboards, or analytics.
  - Reads from centralized log stores.
- **Workload accounts**
  - Generate logs and configuration data.

If you use delegated administration, register the delegated administrator through **AWS Organizations**, not through a service-specific CloudTrail command.

```bash
aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal cloudtrail.amazonaws.com
```

You can also delegate AWS Config or other supported services in a similar way, but the exact service principal must match the service.

**CloudTrail organization trail**

CloudTrail is the primary audit log for AWS API activity. For centralized logging, use an **organization trail** so all accounts in the organization can deliver events to a central S3 bucket.

Key points:

- An organization trail captures events from all accounts in the organization.
- It can be configured for all Regions or a selected set of Regions.
- It should usually include **management events** and, when needed, **data events**.
- Enable **log file validation** to detect tampering.
- Use a dedicated S3 bucket in the log archive account.

A non-organization trail is still useful, but it only covers the account and Regions where it is enabled. It does not automatically include every account in the organization.

**Create the trail**

A common CLI pattern is:

```bash
aws cloudtrail create-trail \
  --name org-audit-trail \
  --s3-bucket-name central-audit-logs-111122223333 \
  --is-organization-trail \
  --is-multi-region-trail \
  --enable-log-file-validation
```

Then start logging:

```bash
aws cloudtrail start-logging --name org-audit-trail
```

If you need to include S3 object-level activity or Lambda invocation activity, add **data event selectors**. Data events are higher volume and higher cost, so scope them carefully.

```bash
aws cloudtrail put-event-selectors \
  --trail-name org-audit-trail \
  --event-selectors '[
    {
      "ReadWriteType": "All",
      "IncludeManagementEvents": true,
      "DataResources": [
        {"Type":"AWS::S3::Object","Values":["arn:aws:s3:::example-bucket/"]},
        {"Type":"AWS::Lambda::Function","Values":["arn:aws:lambda:us-east-1:111122223333:function:*"]}
      ]
    }
  ]'
```

**Verify CloudTrail integrity**

Log file validation creates digest files that let you confirm logs were not altered after delivery.

Useful checks:

- In the console, confirm **Log file validation** is enabled.
- In S3, look for digest files alongside log files.
- Use the CloudTrail validation workflow or CLI tooling to verify the chain of integrity.

A practical validation command pattern is:

```bash
aws cloudtrail validate-logs \
  --trail-arn arn:aws:cloudtrail:us-east-1:111122223333:trail/org-audit-trail \
  --start-time 2026-04-25T00:00:00Z \
  --end-time 2026-04-26T00:00:00Z \
  --s3-bucket central-audit-logs-111122223333
```

If the exact CLI options differ in your environment, use the console or the current AWS CLI reference. The exam focus is the concept: validation proves the log chain is intact.

**S3 bucket policy for CloudTrail delivery**

CloudTrail needs permission to write objects and usually to check the bucket ACL. For organization trails, the bucket policy should allow the CloudTrail service principal to deliver logs into the correct `AWSLogs` prefix structure.

A practical bucket policy includes:

- `s3:GetBucketAcl`
- `s3:PutObject`
- A resource path under `AWSLogs/<account-id-or-org-path>/`
- A condition that restricts writes to the expected AWS account or organization context

Example pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSCloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::central-audit-logs-111122223333",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/org-audit-trail"
        }
      }
    },
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::central-audit-logs-111122223333/AWSLogs/111122223333/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control",
          "AWS:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/org-audit-trail"
        }
      }
    }
  ]
}
```

For organization trails, the exact path and conditions can vary by setup. In practice, you often also add controls for the organization ID and KMS encryption if you use SSE-KMS.

**SSE-KMS for CloudTrail logs**

If you encrypt the bucket with **SSE-KMS** (Server-Side Encryption with AWS Key Management Service), the KMS key policy must allow CloudTrail to use the key for encryption and decryption of delivered objects.

Common requirements:

- Allow the CloudTrail service principal to use the key.
- Allow the log archive account administrators to manage the key.
- Restrict usage to the intended trail or organization.

Example key policy fragment:

```json
{
  "Sid": "AllowCloudTrailUseOfKey",
  "Effect": "Allow",
  "Principal": {
    "Service": "cloudtrail.amazonaws.com"
  },
  "Action": [
    "kms:GenerateDataKey",
    "kms:Decrypt",
    "kms:DescribeKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "AWS:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/org-audit-trail"
    }
  }
}
```

Use SSE-KMS when you need stronger key control, separation of duties, or compliance-driven encryption management.

**AWS Config centralization**

AWS Config records resource configuration changes and evaluates compliance. Centralization usually means two things:

- Each source account and Region runs its own **configuration recorder** and **delivery channel**.
- A central account uses an **aggregator** to read and consolidate those records.

The aggregator does not replace recorders. You still need Config enabled in each account and Region you want to monitor.

**Enable AWS Config in each source account**

A typical setup requires:

- An IAM role for AWS Config to assume.
- A configuration recorder.
- A delivery channel to S3.
- Optional SNS notifications for compliance changes.

Example recorder and delivery channel pattern:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder name=default,roleARN=arn:aws:iam::111122223333:role/aws-config-role \
  --recording-group allSupported=true,includeGlobalResourceTypes=true

aws configservice put-delivery-channel \
  --delivery-channel name=default,s3BucketName=central-config-logs-111122223333
```

Then start recording:

```bash
aws configservice start-configuration-recorder \
  --configuration-recorder-name default
```

In practice, you repeat this in each account and Region that must be covered.

**Create the configuration aggregator**

The aggregator centralizes visibility across accounts and Regions. For organization-wide aggregation, the source is usually the organization itself.

Example pattern:

```bash
aws configservice put-configuration-aggregator \
  --configuration-aggregator name=org-aggregator,organizationAggregationSource="{RoleArn=arn:aws:iam::111122223333:role/AWSConfigRole,AllAwsRegions=true}"
```

The exact role and trust relationship must allow AWS Config to read from the organization. If you use account-based aggregation instead of organization aggregation, the source structure changes accordingly.

**AWS Config aggregator trust and permissions**

The aggregator role needs permissions to read configuration data from source accounts. The source accounts also need AWS Config enabled locally.

Typical permissions include:

- `config:Describe*`
- `config:Get*`
- `organizations:ListAccounts`
- `organizations:DescribeOrganization`

The trust policy should allow the AWS Config service to assume the role.

Example trust policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "config.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

**CloudWatch Logs centralization**

CloudWatch Logs is often used for application logs, Lambda logs, VPC Flow Logs, and service logs. For cross-account centralization, the common pattern is:

- Source account log groups send data through a **subscription filter**.
- A destination in the central account receives the stream.
- The destination forwards to a processing target such as **Kinesis Data Firehose**, **Kinesis Data Streams**, or **Lambda**.

The authorization model is easy to get wrong. The source account typically needs permission to create the subscription filter on its own log group, while the destination account controls access to the destination resource through a resource policy.

**Destination resource policy**

A CloudWatch Logs destination policy authorizes source accounts to use the destination. It is not the same as a generic IAM trust policy for a forwarding role.

Example destination policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSourceAccountsToPutSubscriptionFilters",
      "Effect": "Allow",
      "Principal": {
        "AWS": [
          "arn:aws:iam::444455556666:root",
          "arn:aws:iam::777788889999:root"
        ]
      },
      "Action": "logs:PutSubscriptionFilter",
      "Resource": "arn:aws:logs:us-east-1:111122223333:destination:central-log-destination"
    }
  ]
}
```

The destination then points to a target such as Firehose or Lambda, depending on your design.

**Create a subscription filter in the source account**

Example pattern:

```bash
aws logs put-subscription-filter \
  --log-group-name /aws/lambda/app-prod \
  --filter-name central-forwarding \
  --filter-pattern "" \
  --destination-arn arn:aws:logs:us-east-1:111122223333:destination:central-log-destination
```

Use an empty filter pattern to forward all events, or define a pattern to reduce noise.

**Processing target options**

Choose the target based on what you need to do with the logs:

- **Kinesis Data Firehose**
  - Best for delivery to S3, OpenSearch Service, or third-party destinations.
  - Good for buffering and batching.
- **Kinesis Data Streams**
  - Best when multiple consumers need near-real-time access.
  - Requires more operational management.
- **Lambda**
  - Best for lightweight transformation or routing.
  - Not ideal for very high-volume raw log ingestion.

A common exam-safe design is CloudWatch Logs subscription filter -> Kinesis Data Firehose -> S3.

**Example Firehose delivery path**

```text
Source log group
  -> Subscription filter
  -> CloudWatch Logs destination
  -> Kinesis Data Firehose
  -> S3 log archive bucket
```

This pattern gives you durable storage and optional transformation.

**Practical lab verification**

You should verify each layer independently and then end-to-end.

Checklist:

- CloudTrail:
  - Trail status is `Logging`.
  - S3 bucket receives log files.
  - Digest files appear when validation is enabled.
  - Management events appear in the event history.
- AWS Config:
  - Recorder status is `Recording`.
  - Delivery channel is configured.
  - Aggregator shows accounts and Regions.
  - Compliance items appear centrally.
- CloudWatch Logs:
  - Subscription filter exists on the source log group.
  - Destination policy allows the source account.
  - Target receives events.
  - Logs appear in the central store or processing service.

Useful CLI checks:

```bash
aws cloudtrail get-trail-status --name org-audit-trail
aws cloudtrail describe-trails --include-shadow-trails
aws configservice describe-configuration-recorders
aws configservice describe-configuration-aggregators
aws logs describe-subscription-filters --log-group-name /aws/lambda/app-prod
```

**Troubleshooting patterns**

Common issues usually come from permissions, Region mismatch, or incomplete service setup.

- **CloudTrail logs do not appear**
  - Check the S3 bucket policy.
  - Confirm the trail is started.
  - Confirm the trail is multi-Region if you expect cross-Region coverage.
  - Verify the bucket and trail are in supported Regions.
- **CloudTrail validation fails**
  - Check for missing digest files.
  - Confirm no manual edits were made to delivered objects.
  - Verify the time range and bucket path.
- **AWS Config aggregator is empty**
  - Confirm recorders are running in source accounts.
  - Confirm the delivery channel exists.
  - Confirm the aggregator role trust and permissions.
  - Confirm the source accounts are in the organization or explicitly added.
- **CloudWatch Logs subscription fails**
  - Check the destination resource policy.
  - Confirm the source account has permission to create the subscription.
  - Confirm the target service can accept the stream.
  - Confirm the log group and destination are in compatible Regions.

**Cross-Region replication and retention**

Cross-Region replication is useful when you want resilience against a Region outage or when compliance requires geographically separated copies. It is not a substitute for primary log delivery.

Use it when:

- You need disaster recovery for audit evidence.
- You want a secondary immutable copy in another Region.
- You must satisfy retention or sovereignty requirements with controlled replication.

A common pattern is:

- Primary CloudTrail delivery to a central S3 bucket.
- S3 replication to a secondary bucket in another Region.
- Object lock or retention controls on the archive buckets, if supported by your governance model.

**S3 lifecycle and immutability**

For audit logs, retention matters as much as collection.

Recommended controls:

- Enable **versioning** on the archive bucket.
- Use **lifecycle policies** to transition older logs to cheaper storage.
- Consider **S3 Object Lock** for WORM (Write Once, Read Many) retention where appropriate.
- Restrict delete permissions tightly.
- Separate read access from write access.

Example lifecycle policy pattern:

```json
{
  "Rules": [
    {
      "ID": "ArchiveLogs",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "AWSLogs/"
      },
      "Transitions": [
        {
          "Days": 90,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 365,
          "StorageClass": "GLACIER"
        }
      ],
      "Expiration": {
        "Days": 2555
      }
    }
  ]
}
```

**Exam-focused distinctions**

Know the difference between these concepts:

- **CloudTrail**
  - Records API calls and account activity.
  - Best for who did what and when.
- **AWS Config**
  - Records resource configuration state and changes.
  - Best for what changed and whether it is compliant.
- **CloudWatch Logs**
  - Collects operational logs from services and applications.
  - Best for runtime diagnostics and centralized log processing.
- **S3**
  - Stores logs durably and cheaply.
  - Best for archive, retention, and evidence.

**Common architecture pattern**

A strong centralized logging architecture often looks like this:

```text
AWS Organizations
  ├─ Workload accounts
  │   ├─ CloudTrail management events
  │   ├─ AWS Config recorder + delivery channel
  │   └─ CloudWatch Logs subscription filters
  ├─ Log archive account
  │   ├─ S3 bucket for CloudTrail and Config archives
  │   └─ Optional replication to secondary Region
  └─ Security account
      ├─ AWS Config aggregator
      ├─ Log analytics
      └─ Alerting and investigation tools
```

**What to remember for SAP-C02**

For the exam, focus on the design tradeoffs:

- Use an **organization trail** for broad audit coverage.
- Enable **log file validation** for integrity.
- Use **AWS Config recorders in each account and Region**, then aggregate centrally.
- Use **CloudWatch Logs subscription filters** for operational log forwarding.
- Use **S3 bucket policies and KMS key policies** carefully; most failures are permission-related.
- Use **replication and lifecycle policies** to support resilience and retention.

**Quick validation checklist**

1. Confirm the organization and account structure exists.
2. Confirm CloudTrail is logging and delivering to S3.
3. Confirm CloudTrail log file validation is enabled.
4. Confirm AWS Config is recording in each source account and Region.
5. Confirm the Config aggregator shows all expected accounts.
6. Confirm CloudWatch Logs subscriptions deliver to the central destination.
7. Confirm S3 retention, encryption, and replication settings match the audit requirement.
