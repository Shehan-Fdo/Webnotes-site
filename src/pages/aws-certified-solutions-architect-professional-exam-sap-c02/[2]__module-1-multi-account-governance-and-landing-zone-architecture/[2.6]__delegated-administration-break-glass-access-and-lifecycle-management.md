## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Delegated administration in AWS Organizations**

Delegated administration lets you register an AWS Organizations member account to administer a supported AWS service for the organization. The delegated administrator is the account boundary, not the human or automation principal; IAM identities inside that account perform the work.

Use delegated admin when you want to centralize service operations without giving the management account broad day-to-day access. This pattern reduces blast radius and keeps the management account reserved for organization-level control.

Common exam themes include:
- Separating **organization management** from service administration.
- Using a dedicated security, logging, or tooling account as the delegated admin.
- Understanding that not every AWS service supports delegated administration.
- Recognizing that some services require both Organizations integration and service-specific enablement.

**What delegated admin is and is not**

A delegated administrator account:
- Is an AWS Organizations member account.
- Is registered for a specific supported service.
- Can manage that service across the organization, subject to service permissions and prerequisites.

It is not:
- A replacement for IAM permissions inside the account.
- A universal admin for all AWS services.
- A substitute for the management account when only the management account can perform an action.

A useful mental model is:
- **Organizations** controls who can administer the service.
- **IAM** controls what identities in that delegated account can do.
- **The service** controls which org-wide actions are supported.

**Service prerequisites and limitations**

Delegated administration is service-specific. In practice, you should verify:
- Whether the service supports delegated admin at all.
- Whether the service requires the management account to enable organization integration first.
- Whether the service is regional or global.
- Whether the service needs additional org-level features such as trusted access.

Typical limitations:
- Some services only support delegated admin in certain Regions.
- Some actions remain management-account-only.
- Some services require you to enable organization-wide features before member accounts can be governed.
- Some services have separate concepts for organization configuration, administrator registration, and member enrollment.

> Be careful on exam questions that mix up “can administer the service” with “can perform every org action.” Those are not the same.

**Practical service patterns**

A few common AWS services use delegated administration patterns:
- **Security Hub**: central security administration across accounts and Regions.
- **GuardDuty**: centralized threat detection administration and member management.
- **AWS Config**: organization-wide configuration and aggregator patterns.
- **IAM Identity Center**: identity and access management across accounts and applications.
- **AWS Backup**: centralized backup policies and cross-account governance.
- **CloudTrail** and related governance services may use organization-wide controls, though the exact admin model varies by feature.

For exam purposes, focus less on memorizing every API and more on the pattern:
- Register a delegated admin account.
- Enable trusted access or service integration if required.
- Configure organization-wide settings.
- Onboard member accounts or auto-enroll them where supported.

**Example CLI pattern for delegated admin registration**

The exact command differs by service, but the workflow usually looks like this:

```bash
aws organizations enable-aws-service-access \
  --service-principal securityhub.amazonaws.com

aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal securityhub.amazonaws.com
```

For a service that supports organization-wide configuration, you may then enable org behavior from the delegated admin account using that service’s APIs.

```bash
aws securityhub enable-security-hub
aws securityhub update-organization-configuration \
  --auto-enable true
```

Use the service documentation for the exact API names, but the exam-safe idea is:
- Organizations registers the admin account.
- The service configures org behavior.
- IAM in the delegated account performs the operational work.

**When to choose delegated admin**

Choose delegated administration when:
- You need centralized control for a supported AWS service.
- You want to avoid using the management account for routine operations.
- You need a security or operations team to manage a service across many accounts.
- You want a cleaner separation of duties.

Do not use delegated admin as a workaround when:
- The service does not support it.
- You only need one-off cross-account access to a single resource.
- You need temporary access to a specific account for troubleshooting.

In those cases, a cross-account IAM role is often the better fit.

**Delegated admin vs cross-account role**

| Pattern | Best for | Scope | Typical owner |
|---|---|---|---|
| Delegated admin | Organization-wide service administration | Service-specific, org-aware | Security/operations team |
| Cross-account role | Temporary access to a target account | Account or resource scoped | Admins, automation, support |
| SCPs | Guardrails and restrictions | Organization or OU | Platform/security team |

A cross-account role is usually the right answer when:
- You need to assume a role into a specific account.
- You need temporary operational access.
- The service does not have a delegated admin model.

A delegated admin is usually the right answer when:
- You need a service to manage member accounts centrally.
- The service has native organization integration.
- You want the service itself to understand the organization structure.

**Lifecycle management in AWS identity and access**

Lifecycle management covers how identities are created, changed, disabled, and removed over time. In AWS, this includes:
- Human identities.
- Machine identities.
- Federated identities.
- IAM users and roles.
- IAM Identity Center users and permission sets.
- External identity source synchronization.

The exam often tests whether you understand where the source of truth lives:
- In an external IdP for federated workforce access.
- In IAM Identity Center for AWS access assignments.
- In IAM for long-lived AWS-native identities and roles.
- In AWS Organizations for account lifecycle and guardrails.

**IAM Identity Center lifecycle**

IAM Identity Center is commonly used for workforce access in multi-account environments. It integrates with an identity source such as:
- The built-in Identity Center directory.
- An external identity provider through federation.
- A directory service integration, depending on the setup.

Identity Center does not create generic manually managed roles in target accounts. Instead, it provisions and manages permission-set-based IAM roles in each assigned account. Those roles are created and maintained by Identity Center and are tied to assignments.

Key lifecycle behaviors:
- User and group changes in the source directory can sync into Identity Center.
- Group-based assignments simplify access management.
- Permission sets define the permissions boundary for the generated roles.
- Removing an assignment removes access in the target account.

Practical implications:
- Use groups, not individual users, for most assignments.
- Keep permission sets small and role-focused.
- Treat the source directory as the authoritative lifecycle control point.

Example assignment flow:
1. Create or sync users and groups from the identity source.
2. Define a permission set.
3. Assign the group to one or more AWS accounts.
4. Identity Center provisions the corresponding roles in those accounts.
5. When the assignment is removed, access is revoked through the Identity Center control plane.

**SCIM and directory synchronization**

<abbr title="System for Cross-domain Identity Management">SCIM</abbr> is commonly used to provision and deprovision users and groups from an external identity provider into AWS identity services.

Why it matters:
- It reduces manual user management.
- It keeps group membership aligned with the source directory.
- It supports faster offboarding than manual IAM cleanup.

Important nuance:
- Deprovisioning from the source directory removes the assignment path, but existing sessions may remain valid until they expire.
- Revocation is not always instantaneous for already-issued credentials or sessions.

Practical guidance:
- Use SCIM for automated joiner/mover/leaver workflows.
- Test group sync and deprovisioning end to end.
- Pair deprovisioning with session duration controls and explicit session revocation where supported.

**IAM users, roles, and lifecycle tradeoffs**

IAM users are long-lived identities with credentials that must be managed carefully. They are often discouraged for workforce access because they are harder to govern at scale.

Use IAM users when:
- You need a legacy integration.
- You need a non-human identity that cannot use roles.
- You have a narrow exception with strong controls.

Prefer roles when:
- You need temporary credentials.
- You want federation or service-to-service access.
- You want easier rotation and reduced credential sprawl.

Lifecycle considerations:
- Rotate access keys regularly if IAM users are unavoidable.
- Remove unused users and keys.
- Prefer roles with short-lived credentials over static secrets.
- Use MFA for sensitive human access where applicable.

**Account lifecycle management**

AWS account lifecycle is broader than identity lifecycle. It includes:
- Account creation.
- Account vending.
- Organizational placement.
- Suspension or closure.
- Billing and access implications.

In enterprise environments, account vending is often automated through:
- AWS Control Tower account factory.
- Custom provisioning pipelines.
- AWS Organizations APIs and Service Catalog-based workflows.

Typical account lifecycle flow:
1. Request a new account.
2. Apply baseline guardrails and logging.
3. Attach the account to the correct OU.
4. Assign access through Identity Center or cross-account roles.
5. Monitor usage, billing, and compliance.
6. Close or suspend the account when no longer needed.

When an account is closed:
- Access is removed.
- Resources are eventually terminated according to AWS account closure behavior.
- Billing can continue for some time if there are outstanding charges or retained services.
- Recovery may be limited and time-sensitive.

> Do not assume account closure is an immediate or reversible cleanup mechanism. Plan for data retention, billing, and recovery before closure.

**Break-glass access**

Break-glass access is emergency access used when normal access paths fail. It is designed for rare, high-risk situations such as:
- Identity provider outage.
- Misconfigured federation.
- Lockout from the management account.
- Incident response requiring urgent privileged access.

A good break-glass design is intentionally separate from normal access:
- Separate from SSO/federation dependencies.
- Separate from everyday admin roles.
- Protected by strong controls.
- Tested periodically.

**Break-glass role vs root user**

The root user is the ultimate account identity and should be used only for account-level actions that require it, such as certain billing, support, or account recovery tasks. It is not a substitute for routine emergency administration.

A break-glass role is usually better for emergency operational access because:
- It can be tightly scoped.
- It can require MFA.
- It can be monitored and logged.
- It can be disabled or constrained through policy.

Use the root user only when:
- AWS requires root for a specific account-level action.
- You need account recovery or irreversible account changes.
- You cannot complete the task with a role.

Use a break-glass role when:
- You need emergency admin access to recover services.
- You need to bypass a broken federation path.
- You want a controlled, auditable emergency path.

**Common break-glass implementation patterns**

A practical break-glass setup often includes:
- A dedicated emergency IAM role in each critical account.
- A tightly controlled trust policy.
- MFA requirement for assumption where feasible.
- Very limited permissions, expanded only as needed.
- Separate storage for credentials or access path metadata.
- Alerting on every use.

Example trust policy pattern for a break-glass role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowEmergencyAdmins",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/EmergencyAccessBroker"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

This pattern is only an example. In real designs, you should also consider:
- Who can assume the broker role.
- Whether the broker role itself is protected by MFA and session controls.
- Whether the emergency role should be in a separate account.
- Whether you need approval workflows or just-in-time access.

**Secure storage and rotation for emergency access**

If you use a dedicated emergency IAM user or access path, protect it aggressively:
- Store credentials in a secure vault or password manager.
- Restrict who can retrieve them.
- Rotate credentials on a schedule and after every use.
- Log every retrieval and use.
- Test the recovery path without relying on the normal federation stack.

A more modern pattern is to avoid long-lived emergency IAM user credentials entirely and use:
- A highly restricted emergency role.
- A separate emergency access account.
- Hardware MFA or equivalent strong authentication.
- Out-of-band approval and logging.

**Testing the break-glass path**

You should test the emergency path before you need it:
- Verify that the role can be assumed.
- Verify that MFA and trust conditions work.
- Verify that logging and alerting fire.
- Verify that the path still works if the identity provider is unavailable.
- Verify that the permissions are sufficient for the intended recovery tasks.

A simple test checklist:
1. Simulate a federation outage or disable the normal access path in a nonproduction environment.
2. Assume the emergency role using the documented procedure.
3. Confirm access to the required recovery actions.
4. Confirm CloudTrail or equivalent logging captures the event.
5. Revoke or rotate any temporary credentials used during the test.

**Session duration and exposure control**

For emergency roles, session duration matters. The maximum session duration is set on the role, and the caller can request a duration up to that maximum.

Use shorter sessions when:
- The role is highly privileged.
- The access is emergency-only.
- You want to reduce the window of exposure.

Use longer sessions only when:
- The recovery task is expected to take time.
- Re-authentication would create operational risk.
- You have compensating controls such as monitoring and approval.

Remember:
- Session duration does not revoke already-issued sessions by itself.
- If you need immediate cutoff, you need explicit session revocation or equivalent control where available.

**Organizations guardrails and lifecycle controls**

Service control policies (SCPs) are guardrails, not permissions grants. They define the maximum permissions available in accounts, OUs, or the organization.

Use SCPs to:
- Prevent accidental deletion of critical resources.
- Restrict regions.
- Limit risky actions in production.
- Protect the management account from routine use.

Example SCP pattern to deny disabling CloudTrail in production accounts:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyCloudTrailStop",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail"
      ],
      "Resource": "*"
    }
  ]
}
```

For break-glass scenarios, be careful not to overfit SCPs to a single principal ARN pattern. Principal matching can be brittle. Prefer designs that:
- Use OU placement.
- Use explicit allow/deny boundaries.
- Combine SCPs with role design and MFA.
- Use principal tags or well-structured role naming where appropriate.

**Choosing between delegated admin, cross-account role, and SCPs**

Use this decision logic:
- If you need org-wide administration for a supported service, choose delegated admin.
- If you need temporary access to a specific account, choose a cross-account role.
- If you need to prevent actions across accounts, choose SCPs.
- If you need emergency access when federation fails, choose a break-glass role or controlled emergency account path.

End-to-end scenario examples:
- A security team centrally manages findings across all accounts: use delegated admin for the security service.
- An engineer needs to troubleshoot one workload account for 30 minutes: use a cross-account role.
- The platform team wants to prevent disabling logging in production: use an SCP.
- The identity provider is down and the team must restore access: use break-glass access.

**Exam-style scenario cues**

Look for these clues in questions:
- “Centralized administration across all accounts” usually points to delegated admin.
- “Temporary access to a specific account” usually points to cross-account role assumption.
- “Prevent actions across the organization” usually points to SCPs.
- “Emergency access when SSO is unavailable” usually points to break-glass design.
- “Provisioning and deprovisioning workforce access” usually points to IAM Identity Center plus SCIM or federation.

**Operational best practices**

- Keep the management account minimal and highly protected.
- Use a dedicated security or tooling account for delegated admin where supported.
- Prefer group-based access assignments over individual assignments.
- Automate account vending and baseline guardrails.
- Test emergency access before an incident.
- Rotate and monitor any long-lived credentials.
- Log all privileged actions centrally.
- Review service-specific delegated admin prerequisites before designing the solution.
