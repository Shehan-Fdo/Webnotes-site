## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Lab goal and scope**

AWS Control Tower provides a governed multi-account foundation on top of AWS Organizations. In this lab, you deploy a **landing zone**, which is the baseline environment Control Tower uses to standardize account structure, logging, identity integration, and guardrails.

The practical goal is to understand the setup flow, what Control Tower creates automatically, how to validate success, and how to troubleshoot common failures. For SAP-C02, you should be able to distinguish between the management account, the organization root, the OUs Control Tower creates by default, and the resources that appear after deployment.

**Prerequisites checklist**

Before you start the lab, confirm the environment is suitable for landing zone deployment:

- You are using the **management account** for the AWS Organization.
- AWS Organizations is enabled and you have permission to create and manage organizational resources.
- The root user email for the management account is accessible for verification and recovery.
- You have not already deployed another Control Tower landing zone in the same organization unless the lab explicitly says to use an existing one.
- The home Region you choose is supported by AWS Control Tower.
- You have sufficient IAM permissions to create service-linked roles, CloudFormation resources, and StackSets.

If the account is already part of a complex organization with preexisting SCPs, delegated admin settings, or conflicting IAM Identity Center configuration, expect setup friction. In a lab, the cleanest path is usually a fresh organization or a minimally configured management account.

**Core concepts to know first**

AWS Control Tower sits on top of several AWS services:

- **AWS Organizations** provides the multi-account container.
- **AWS Identity Center** provides centralized user access for the landing zone.
- **AWS CloudFormation StackSets** deploy baseline resources across accounts and Regions.
- **AWS Config** and **AWS CloudTrail** support detective controls and auditability.
- **Service-linked roles** allow Control Tower and related services to operate in your accounts.

A landing zone is not just a console wizard result. It is a governed foundation that creates accounts, OUs, logging, and guardrails so you can scale account vending and policy enforcement consistently.

**Default landing zone structure**

When you deploy a default Control Tower landing zone, Control Tower creates a standard organizational layout. The exact naming can vary slightly by setup choices, but the default pattern is:

- A **Security OU** that contains:
  - the **Log Archive** account
  - the **Audit** account
- A separate **Sandbox OU** for initial workload experimentation

Control Tower does **not** create a Workloads OU by default. You can create additional OUs later, but the default landing zone focuses on security, audit, and a sandbox starting point.

The management account remains the account where you deploy and administer Control Tower. It is not the organization root. The organization root is the top-level container in AWS Organizations, and the management account is simply the account that owns and administers the organization.

**What Control Tower creates during deployment**

During landing zone setup, Control Tower typically creates or configures:

- The **Log Archive** account for centralized log retention.
- The **Audit** account for security visibility and read-only oversight.
- Baseline **organizational units**.
- Required **service-linked roles**.
- **CloudFormation StackSets** and stacks used to apply baseline controls.
- **AWS Config** recorders and delivery channels where needed.
- **CloudTrail** trails or related logging configuration for governance.
- Identity integration with **IAM Identity Center** if you enable it in the setup flow.

> Do not assume you must pre-create the Log Archive and Audit accounts for a standard landing zone. In the default setup, Control Tower creates them as part of deployment.

**Setup wizard flow**

The Control Tower console guides you through a landing zone setup wizard. The exact screens can vary by Region and service updates, but the lab flow usually includes these decisions:

1. Choose the **home Region** for Control Tower.
2. Confirm or configure the **organization email** and account details already associated with the management account.
3. Review the default **organizational structure** Control Tower will create.
4. Select or confirm the **baseline Regions** and any additional governed Regions if the wizard offers them.
5. Configure identity integration options, such as **IAM Identity Center** settings, if prompted.
6. Review the summary and start deployment.

The home Region is the primary Region for Control Tower operations. Additional governed Regions are commonly enabled after the landing zone is created, rather than being treated as a core part of the initial setup in every environment.

**Step-by-step lab procedure**

1. Sign in to the AWS Management Console using the management account.
2. Open **AWS Control Tower**.
3. Start the landing zone setup.
4. Select the home Region.
5. Review the default organizational structure and baseline settings.
6. Confirm identity and access settings if the wizard prompts you.
7. Start deployment and wait for completion.

During deployment, expect the process to take time. Control Tower is provisioning multiple services and cross-account resources, so a successful setup is not immediate.

**Expected success indicators**

After deployment completes, verify the following:

- The Control Tower dashboard shows the landing zone as active or healthy.
- The **Audit** and **Log Archive** accounts exist.
- The default OUs are visible in AWS Organizations.
- Baseline guardrails or controls are enabled.
- CloudFormation-managed resources and StackSets appear in the organization.
- Service-linked roles exist in the management and member accounts where applicable.

Useful verification targets include:

- AWS Organizations console
- AWS Control Tower console
- AWS CloudFormation StackSets console
- AWS IAM console for service-linked roles
- AWS Config console
- AWS CloudTrail console

**Practical verification commands**

CLI support for Control Tower varies by environment and version, so use the console as the primary verification path in a lab. For supporting checks, use AWS CLI commands against Organizations, IAM, CloudFormation, and STS.

```bash
aws sts get-caller-identity
aws organizations describe-organization
aws organizations list-roots
aws organizations list-organizational-units-for-parent --parent-id r-xxxx
aws iam list-roles --query "Roles[?contains(RoleName, 'AWSControlTower') || contains(RoleName, 'AWSServiceRole')].RoleName"
aws cloudformation list-stack-instances --stack-set-name <stack-set-name>
```

Use these commands to confirm:

- You are in the expected account.
- The organization exists and has a root container.
- The expected OUs are present.
- Control Tower-related roles exist.
- StackSets have deployed baseline instances.

If you need to inspect the current identity and account context before troubleshooting, `aws sts get-caller-identity` is the fastest sanity check.

**Common post-deployment artifacts**

A successful landing zone often leaves behind these artifacts:

- **AWSControlTowerExecution** role in enrolled accounts for delegated operations.
- Control Tower-managed **StackSets** for baseline configuration.
- **AWS Config** recorders and delivery channels.
- **CloudTrail** trails or organization-level logging configuration.
- IAM Identity Center assignments and permission sets if enabled.
- Guardrail-related AWS Config rules or SCPs depending on the control type.

These artifacts matter because they prove the landing zone is not just visually present in the console; it is actively enforcing governance.

**Guardrails and controls**

Control Tower guardrails are preventive or detective controls that help enforce governance across accounts and OUs.

- **Preventive guardrails** use **service control policies (SCPs)** to block disallowed actions.
- **Detective guardrails** use AWS Config rules to detect noncompliant resources or settings.

A useful way to think about them:

| Type | Mechanism | Purpose | Example use |
|---|---|---|---|
| Preventive | SCP | Blocks actions before they happen | Prevent disabling CloudTrail in governed accounts |
| Detective | AWS Config | Detects drift or noncompliance | Detect public S3 buckets or unencrypted resources |

Commonly referenced Control Tower controls include:

- Preventive controls that restrict changes to logging or security services.
- Detective controls that check for encryption, logging, or public access issues.
- Strongly recommended baseline controls for auditability and account hygiene.

In a lab, apply guardrails to the OU that contains the accounts you want to govern. Use the **Sandbox OU** for experimentation and the **Security OU** for the central logging and audit accounts. If you create additional OUs later, attach controls based on the workload sensitivity.

**When to use preventive vs detective controls**

- Use **preventive** controls when an action must never occur, such as disabling centralized logging.
- Use **detective** controls when you want visibility and remediation workflows without blocking the action immediately.
- Use both when you need layered governance: block the most dangerous actions and detect the rest.

**Account enrollment workflow**

After the landing zone exists, you can enroll existing accounts into Control Tower governance. This is different from creating new accounts through Account Factory.

| Scenario | Best fit | Notes |
|---|---|---|
| New account creation | Account Factory | Standardized account vending with baseline controls |
| Existing account governance | Account enrollment | Brings an existing account under Control Tower management |

Enrollment is useful when you already have workloads in AWS Organizations and want to bring them under governance without recreating them.

**Common enrollment failures and fixes**

If enrollment fails, check for these issues:

- The account already has conflicting **AWS Config** recorders or delivery channels.
- Existing **CloudTrail** trails conflict with Control Tower-managed logging.
- The account has unsupported or preexisting **SCPs** that block required actions.
- The account is not in a state that Control Tower can manage cleanly.
- IAM Identity Center or delegated admin settings conflict with the landing zone design.

Practical remediation steps:

1. Review the failure message in the Control Tower console.
2. Check the target account for preexisting Config, CloudTrail, or IAM resources.
3. Remove or adjust conflicting resources if the lab permits.
4. Retry enrollment after the account is clean.
5. If the account is heavily customized, consider whether it is better to create a new account through Account Factory instead.

> If an existing account has critical production workloads, do not remove logging or governance resources blindly. Validate the impact before changing Config, CloudTrail, or SCP settings.

**Troubleshooting landing zone deployment**

If the landing zone deployment stalls or fails, use a structured workflow:

1. Confirm you are signed in as the management account.
2. Verify AWS Organizations is healthy and the organization root exists.
3. Check whether the chosen home Region is supported.
4. Review IAM permissions for the deploying principal.
5. Inspect CloudFormation StackSets for failed instances.
6. Check AWS Config and CloudTrail setup for permission or service conflicts.
7. Review the Control Tower console for the first failing step.

Typical symptoms and likely causes:

- **Setup fails early**
  - Missing permissions
  - Organization not properly initialized
  - Unsupported Region selection
- **Setup fails during account creation**
  - Email or account naming conflicts
  - Service quota or organization policy issues
- **Setup fails during baseline deployment**
  - CloudFormation or StackSet permission problems
  - Existing Config or CloudTrail conflicts
- **Setup completes but controls are missing**
  - Region mismatch
  - Delayed propagation
  - Identity Center or StackSet deployment issue

**Useful CLI workflow for troubleshooting**

Use the CLI to confirm the organization and account context, then inspect the baseline resources.

```bash
aws sts get-caller-identity
aws organizations describe-organization
aws organizations list-accounts
aws organizations list-roots
aws iam list-roles --query "Roles[?contains(RoleName, 'AWSControlTower')].RoleName"
```

If StackSets are involved, inspect them in the CloudFormation console first. The console usually gives clearer failure details than the CLI for a first pass. Then use the CLI to confirm whether the expected instances exist.

**Practical lab sequence**

A clean lab sequence helps you avoid confusion between setup, validation, and follow-up tasks.

1. **Mandatory:** Confirm the account is the management account and AWS Organizations is ready.
2. **Mandatory:** Deploy the Control Tower landing zone in the console.
3. **Mandatory:** Wait for the Audit and Log Archive accounts to be created.
4. **Mandatory:** Validate the default OUs and baseline resources.
5. **Mandatory:** Confirm guardrails are active.
6. **Optional:** Enable additional governed Regions after the landing zone is healthy.
7. **Optional:** Create or enroll workload accounts.
8. **Optional:** Attach additional guardrails to workload OUs.

This sequence keeps the lab focused on the landing zone first, then expands into governance and account vending.

**Real-world example: validating the default structure**

After deployment, you might inspect the organization tree and see something like this:

```text
Root
└── Security OU
    ├── Log Archive account
    └── Audit account
└── Sandbox OU
```

This is the default pattern you should expect in a standard landing zone. If you see a different structure, check whether the lab or environment customized the setup.

**Real-world example: checking for Control Tower roles**

A quick IAM role check can confirm that Control Tower has provisioned baseline access paths:

```bash
aws iam list-roles \
  --query "Roles[?contains(RoleName, 'AWSControlTower') || contains(RoleName, 'AWSServiceRole')].[RoleName,Arn]" \
  --output table
```

Look for roles related to Control Tower execution and service-linked operations. Their presence is a strong sign that the landing zone deployment progressed correctly.

**Real-world example: verifying organization context**

```bash
aws organizations describe-organization
aws organizations list-roots
```

Use these commands to confirm that:

- The organization exists.
- The organization has a root container.
- You are not confusing the management account with the root itself.

**Exam and lab takeaways**

- Control Tower creates a governed multi-account baseline, not just a set of accounts.
- The default landing zone includes a **Security OU** with **Log Archive** and **Audit** accounts, plus a **Sandbox OU**.
- The management account administers the organization; it is not the organization root.
- Baseline deployment commonly creates StackSets, service-linked roles, Config, and CloudTrail-related resources.
- Guardrails come in preventive and detective forms, and you choose them based on whether you want to block or detect behavior.
- Existing account enrollment often fails because of preexisting governance or logging resources, so check for conflicts first.

**Quick reference**

- **Landing zone:** The governed AWS Control Tower foundation.
- **Account Factory:** Creates new standardized accounts.
- **Enrollment:** Brings existing accounts under Control Tower governance.
- **Preventive control:** SCP-based block.
- **Detective control:** AWS Config-based detection.
- **Home Region:** Primary Control Tower Region for the landing zone.
