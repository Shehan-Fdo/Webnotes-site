## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Guardrails and control strategy**

In AWS, a **guardrail** is a policy or mechanism that keeps accounts, workloads, and teams within approved boundaries. In multi-account designs, guardrails reduce the chance of accidental drift, unsafe changes, and compliance violations.

A strong architecture usually combines:

- **Preventive controls** to stop bad actions before they happen
- **Detective controls** to identify drift or violations after the fact
- **Corrective controls** to remediate or quarantine noncompliant resources

For SAP-C02, you should be able to choose the right control type for the risk, explain how it works in AWS Organizations, and recognize where a control is effective or limited.

**Preventive vs detective controls**

Preventive controls block or constrain actions before resource state changes. Detective controls observe activity or configuration and alert on violations after they occur.

| Control type | Goal | Typical AWS mechanisms | Strengths | Limitations |
|---|---|---|---|---|
| Preventive | Stop unauthorized or risky actions | Service control policies (SCPs), IAM permission boundaries, session policies, resource policies, service quotas | Reduces blast radius, enforces baseline | Can be bypassed only if the control scope is misunderstood; may be hard to express for every service edge case |
| Detective | Find violations and drift | AWS Config, AWS CloudTrail, Amazon EventBridge, AWS Security Hub, Amazon GuardDuty | Good visibility and auditability | Detection latency; requires alerting, retention, and response workflows |
| Corrective | Fix or isolate issues | AWS Lambda remediation, Systems Manager Automation, AWS Config remediation, auto-scaling quarantine patterns | Can restore compliance quickly | Needs safe automation and rollback planning |

A common exam trap is assuming detective controls are enough. They are not. If a user can create a public S3 bucket, a detective rule may alert later, but the exposure already exists. Use preventive controls when the risk must be blocked.

**Service control policies (SCPs)**

An **SCP** is an AWS Organizations policy that sets the maximum permissions available to principals in accounts that are attached to an organization unit (OU) or the root of the organization. SCPs do not grant permissions by themselves; they define what is allowed to be effective in member accounts.

Important scope rules:

- SCPs apply to **member accounts** in the organization.
- The **management account** is exempt from SCP enforcement.
- SCPs affect the permissions available to principals in member accounts, including the root user in those member accounts.
- SCPs do not grant access; IAM identity policies, resource-based policies, and other authorization layers still matter.
- Service-linked roles and some AWS-managed service operations can be special cases, so always verify service behavior rather than assuming a blanket deny will behave identically everywhere.

For exam purposes, think of SCPs as a **permissions boundary for the account**, not as a replacement for IAM.

**How SCP evaluation works**

AWS evaluates effective permissions by combining multiple layers:

- **SCPs** define the outer boundary for the account
- **IAM identity policies** define what a principal can do inside that boundary
- **Permission boundaries** further limit what an IAM principal can do
- **Session policies** can further reduce permissions for assumed roles or federated sessions
- **Resource-based policies** can grant access directly in some services and scenarios

A useful mental model:

- If an action is denied by an SCP, it is denied in that member account even if IAM allows it.
- If an action is not allowed by the applicable SCP set, IAM cannot expand it beyond the boundary.
- If multiple SCPs apply, the effective permission set is the **intersection** of those SCPs.
- An explicit **Deny** in any applicable SCP wins over any Allow.
- In allow-list mode, you must allow the action in every relevant SCP path that applies to the account.

> **Critical distinction:** SCPs are not the same as IAM permission boundaries or session policies. Permission boundaries limit what a principal can ever receive from IAM policies, while session policies limit a specific session. SCPs limit the account’s maximum effective permissions. They solve different problems and often appear together in troubleshooting questions.

**SCP modes and evaluation patterns**

There are two common SCP design styles:

- **Deny list style**
  - You attach a broad SCP and explicitly deny risky actions.
  - This is easier to start with and often safer for incremental adoption.
- **Allow list style**
  - You attach an SCP that only allows approved services or actions.
  - This is stricter and better for highly governed environments, but it requires careful maintenance.

In allow-list mode, remember:

- An action must be allowed by the SCP path that applies to the account.
- A deny in one SCP cannot be overridden by an allow in another SCP.
- If a parent OU and child OU both have SCPs, the account receives the intersection of both.

Example of a deny-focused SCP that blocks disabling key logging services:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyDisablingSecurityLogging",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail",
        "cloudtrail:UpdateTrail",
        "config:StopConfigurationRecorder",
        "config:DeleteConfigurationRecorder",
        "config:DeleteDeliveryChannel"
      ],
      "Resource": "*"
    }
  ]
}
```

This kind of policy is useful when you want to preserve auditability. For AWS Config, think broadly about the actions that could disable recording or delivery, not just one or two API calls.

**Using conditions in SCPs**

Conditions let you create exceptions or scope guardrails more precisely. Common exam-relevant condition keys include:

- `aws:RequestedRegion`
- `aws:PrincipalArn`
- `aws:ResourceTag/<tag-key>`
- `aws:PrincipalOrgID`
- `aws:PrincipalAccount`

Example: restrict a set of actions to approved regions while leaving room for global-service exceptions in a separate statement or policy design.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "Action": [
        "ec2:*",
        "rds:*",
        "lambda:*"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

Use region restrictions carefully:

- Some AWS services are global or have control-plane behaviors that do not fit a simple region check.
- A blanket `Action: "*"` deny with a region condition can break required global-service calls.
- For exam answers, prefer the idea of a region guardrail with explicit exceptions rather than assuming one universal pattern fits every service.

Example: allow only principals with a specific role path to perform a sensitive action.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenySensitiveActionsExceptBreakGlassRole",
      "Effect": "Deny",
      "Action": [
        "iam:CreateUser",
        "iam:CreateAccessKey",
        "organizations:LeaveOrganization"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotLike": {
          "aws:PrincipalArn": "arn:aws:iam::*:role/BreakGlass/*"
        }
      }
    }
  ]
}
```

This pattern is useful when you need a tightly controlled exception path for emergency access.

**Root user, management account, and member accounts**

A frequent exam trap is confusing the management account with member accounts.

- The **management account** is not subject to SCP enforcement.
- The **root user** in a **member account** is still constrained by SCPs.
- The root user in the management account is not constrained by SCPs because the management account is exempt.

This matters when a question asks how to prevent dangerous actions across the organization. SCPs can restrict root in member accounts, but they do not govern the management account the same way. If the management account is the risk, you need stronger operational controls, separation of duties, and tightly controlled access.

**SCPs vs IAM permission boundaries vs session policies**

These three controls are often tested together.

| Control | Where it applies | What it limits | Common use |
|---|---|---|---|
| SCP | AWS Organizations account boundary | Maximum permissions in member accounts | Organization-wide guardrails |
| Permission boundary | IAM principal | Maximum permissions a role or user can receive from IAM policies | Delegated administration, safe role creation |
| Session policy | Temporary session | Permissions for a specific assumed-role or federated session | Narrowing access for a session or application flow |

Practical differences:

- Use **SCPs** when you want to prevent an entire account from using certain services or actions.
- Use **permission boundaries** when you want to let teams create roles or users without letting them exceed a safe permission ceiling.
- Use **session policies** when you want to reduce permissions for a specific assumed role session without changing the role itself.

Example troubleshooting logic:

1. Check whether the action is denied by an SCP.
2. Check the IAM identity policy.
3. Check the permission boundary on the principal.
4. Check any session policy.
5. Check resource-based policy behavior for the target service.

**Resource-based policies and SCPs**

Resource-based policies can grant access directly to a resource, but they do not override an SCP deny for a principal in a member account. That said, do not oversimplify resource-based authorization:

- Some services allow access through resource policies even when the identity policy is not the primary grant path.
- The exact interaction depends on the service and principal type.
- For exam questions, remember that an SCP deny still blocks the action in the member account.

Examples of resource-based policy use cases:

- S3 bucket policies
- KMS key policies
- SNS topic policies
- SQS queue policies
- Lambda resource policies

If a question involves cross-account access, always consider both the identity side and the resource side.

**Practical SCP examples**

Deny creation of internet-facing resources in a restricted OU:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyPublicS3BucketCreation",
      "Effect": "Deny",
      "Action": [
        "s3:CreateBucket",
        "s3:PutBucketAcl",
        "s3:PutBucketPolicy"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": [
            "public-read",
            "public-read-write",
            "website"
          ]
        }
      }
    }
  ]
}
```

This is conceptually useful, but in practice S3 public access is often better controlled with account-level **S3 Block Public Access**, bucket policies, and preventive governance. Use the right control for the service.

Deny use of non-approved regions for a workload OU:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "RestrictRegions",
      "Effect": "Deny",
      "Action": [
        "ec2:*",
        "rds:*",
        "eks:*",
        "lambda:*"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

Use this pattern when the business has a clear regional footprint and you want to reduce data residency risk. Be careful with global services and shared control-plane dependencies.

Deny changes to security tooling:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ProtectSecurityServices",
      "Effect": "Deny",
      "Action": [
        "guardduty:DeleteDetector",
        "guardduty:UpdateDetector",
        "securityhub:DisableSecurityHub",
        "securityhub:DisassociateFromAdministratorAccount",
        "config:StopConfigurationRecorder",
        "cloudtrail:StopLogging"
      ],
      "Resource": "*"
    }
  ]
}
```

This is a common governance pattern because attackers often try to blind detection before making changes.

**Preventive controls in AWS**

Preventive controls are best when you must stop a risky action from succeeding at all.

Common preventive mechanisms:

- **SCPs**
- **IAM permission boundaries**
- **IAM policies with explicit Deny**
- **Service-specific guardrails**
- **S3 Block Public Access**
- **KMS key policies**
- **VPC security group and network ACL design**
- **AWS Organizations tag policies** for standardization, where applicable

Examples of preventive use cases:

- Prevent creation of public S3 buckets
- Prevent disabling CloudTrail or AWS Config
- Prevent use of unapproved regions
- Prevent developers from creating overly privileged roles
- Prevent deletion of critical backup vaults or log archives

A good preventive control is:

- Easy to reason about
- Hard to bypass accidentally
- Narrow enough to avoid breaking required operations
- Paired with a documented exception process

**Detective controls in AWS**

Detective controls identify violations after they occur. They are essential for visibility, audit, and response.

Common detective mechanisms:

- **AWS Config** rules and conformance packs
- **AWS CloudTrail** event logging
- **Amazon EventBridge** event routing
- **AWS Security Hub** findings aggregation
- **Amazon GuardDuty** threat detection
- **Amazon Inspector** vulnerability findings
- **AWS CloudWatch** alarms and logs
- **AWS Audit Manager** evidence collection

Examples of detective use cases:

- Detect an S3 bucket that becomes public
- Detect an IAM policy that grants wildcard permissions
- Detect a security group that allows `0.0.0.0/0` to port 22
- Detect a root login or unusual API activity
- Detect disabled logging or missing encryption

A detective control is only useful if you also have:

- Alert routing
- Retention
- Ownership
- Remediation playbooks

Without those, you only have delayed awareness.

**AWS Config as a detective control**

AWS Config records resource configuration changes and evaluates them against rules. It is a core detective service for compliance and drift detection.

Typical workflow:

1. AWS Config records resource state changes.
2. A managed or custom rule evaluates the resource.
3. A noncompliant result is generated.
4. EventBridge or Security Hub routes the finding.
5. A remediation action or ticket is created.

Example managed rule use cases:

- `s3-bucket-public-read-prohibited`
- `s3-bucket-public-write-prohibited`
- `encrypted-volumes`
- `restricted-ssh`
- `cloudtrail-enabled`

Example EventBridge pattern for AWS Config compliance change events:

```json
{
  "source": ["aws.config"],
  "detail-type": ["Config Rules Compliance Change"],
  "detail": {
    "configRuleName": ["s3-bucket-public-read-prohibited"],
    "newEvaluationResult": {
      "complianceType": ["NON_COMPLIANT"]
    }
  }
}
```

Use this pattern to trigger automation or notifications when a resource drifts out of compliance.

**AWS CloudTrail as a detective control**

CloudTrail records API activity and is essential for audit and incident response.

Use CloudTrail to:

- Trace who made a change
- Reconstruct the sequence of actions
- Detect suspicious API calls
- Support forensic investigations

Important notes:

- CloudTrail logs are delivered to S3, so protect the destination bucket.
- Use **log file validation** to detect tampering.
- Consider **S3 Object Lock** for stronger immutability requirements.
- Centralize trails in a dedicated logging account when possible.

Example trail creation with the AWS CLI:

```bash
aws cloudtrail create-trail \
  --name org-trail \
  --s3-bucket-name central-cloudtrail-logs \
  --is-multi-region-trail
aws cloudtrail start-logging --name org-trail
```

**GuardDuty, Security Hub, and detective posture**

These services are often used together:

- **GuardDuty** detects threats such as anomalous API activity, credential abuse, and suspicious network behavior.
- **Security Hub** aggregates findings and provides a compliance view.
- **AWS Config** checks configuration compliance.
- **CloudTrail** provides the underlying activity record.

A strong detective posture usually includes:

- Centralized logging
- Cross-account aggregation
- Retention policies
- Automated triage
- Escalation paths

**Control Tower guardrails**

AWS Control Tower provides a higher-level governance framework for multi-account environments. It uses guardrails to help enforce baseline controls.

Common categories:

- **Preventive guardrails**
  - Primarily implemented with SCPs
  - Block actions that violate policy
- **Detective guardrails**
  - Primarily implemented with AWS Config rules and related monitoring
  - Detect and report noncompliance

Control Tower guardrails are often described as:

- **Mandatory**
- **Strongly recommended**
- **Elective**

Use this distinction as a study aid:

- **Mandatory** guardrails are part of the baseline landing zone.
- **Strongly recommended** guardrails are generally expected for secure operations.
- **Elective** guardrails are optional and chosen based on workload needs.

Control Tower is useful when you want standardized account vending, baseline governance, and centralized visibility without building every control from scratch.

**Common exam scenarios**

If the question asks how to stop developers from creating resources in unapproved regions:

- Prefer an **SCP** with region conditions for member accounts.
- Consider exceptions for global services and shared platform operations.

If the question asks how to detect public S3 exposure:

- Use **AWS Config** plus **Security Hub** or **EventBridge**.
- Add a remediation action if the business requires automatic correction.

If the question asks how to prevent disabling CloudTrail:

- Use an **SCP** or explicit IAM deny in the relevant account boundary.
- Protect the log destination with S3 controls and separate administration.

If the question asks how to let teams create roles but not exceed a safe permission ceiling:

- Use a **permission boundary**.
- Optionally combine it with SCPs for organization-wide restrictions.

If the question asks how to centralize governance across many accounts:

- Use **AWS Organizations** and **Control Tower**.
- Apply preventive and detective guardrails at the OU level.

**Troubleshooting access issues**

When access fails, do not assume IAM is the only cause.

Use this checklist:

1. Confirm the principal and account.
2. Check whether an SCP denies the action.
3. Check the IAM identity policy.
4. Check the permission boundary.
5. Check the session policy.
6. Check the resource policy and service-specific authorization model.
7. Check whether the action is regional, global, or service-linked.
8. Check whether a service control or quota is involved.

A practical CLI workflow:

```bash
aws organizations describe-effective-policy \
  --policy-type SERVICE_CONTROL_POLICY \
  --target-id 111122223333
```

You can also inspect IAM policy simulation for identity-side debugging:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::111122223333:role/AppRole \
  --action-names s3:PutBucketPolicy \
  --resource-arns arn:aws:s3:::example-bucket
```

Use simulation carefully: it helps with IAM evaluation, but it does not replace SCP analysis or service-specific behavior checks.

**Design guidance**

Good guardrail design balances security and operability.

- Start with the highest-risk actions.
- Prefer explicit denies for dangerous operations.
- Use allow lists when the environment is stable and tightly governed.
- Add exceptions with conditions rather than broad exclusions.
- Centralize logs and findings so detective controls are actionable.
- Test policies in a nonproduction OU before broad rollout.

A practical governance stack often looks like this:

- **Organizations** for account structure
- **SCPs** for account-level boundaries
- **Permission boundaries** for delegated IAM administration
- **AWS Config** for compliance drift
- **CloudTrail** for auditability
- **GuardDuty** and **Security Hub** for threat and posture visibility
- **Control Tower** for standardized landing zones

**Key takeaways**

- **SCPs** set the maximum permissions for member accounts in AWS Organizations.
- The **management account** is exempt from SCP enforcement.
- SCPs do not grant permissions; they constrain what IAM and other policies can effectively allow.
- Multiple SCPs combine by **intersection**, and any explicit deny wins.
- **Preventive controls** stop bad actions; **detective controls** find them after the fact.
- **AWS Config**, **CloudTrail**, **GuardDuty**, and **Security Hub** are core detective services.
- **Permission boundaries** and **session policies** are not the same as SCPs.
- Control Tower guardrails help standardize governance, but you still need logging, alerting, and remediation.
