## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Cross-account IAM role fundamentals**

Cross-account access in AWS uses an <abbr title="Identity and Access Management">IAM</abbr> role in one account that is assumed by a principal from another account. The key idea is separation of trust and permissions:

- The **trust policy** on the role says who can assume it.
- The **permissions policy** on the role says what the assumed role can do after it is assumed.

This pattern is preferred over sharing long-term access keys because it gives you temporary credentials, centralized control, and easier revocation. In labs and real environments, you usually create a role in the target account and allow a user, role, or AWS service principal in the source account to call `sts:AssumeRole`.

A typical flow looks like this:

1. A principal in Account A requests to assume a role in Account B.
2. AWS Security Token Service (<abbr title="Security Token Service">STS</abbr>) evaluates the role trust policy.
3. If allowed, STS returns temporary credentials.
4. The temporary session uses the role’s permissions in Account B.

**Core trust and permission model**

The trust policy answers the question: “Who can use this role?” The permissions policy answers: “What can this role do once assumed?” Confusing these is a common lab mistake.

- **Trust policy**
  - Attached to the role.
  - Uses the `Principal` element.
  - Often includes `sts:AssumeRole`.
- **Permissions policy**
  - Attached to the role or as an inline policy.
  - Grants access to AWS resources such as S3, DynamoDB, CloudWatch, or EC2.
- **Caller permissions**
  - The source identity also needs permission to call `sts:AssumeRole` on the target role.
  - Without this, the trust policy alone is not enough.

A minimal trust policy for a role in Account B that trusts a role in Account A looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/SourceRole"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

A matching permissions policy might allow read-only access to a bucket:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

**How to design the trust relationship**

Use the narrowest possible trust scope. Prefer a specific role ARN over an entire account unless the lab explicitly requires account-wide trust.

- Trust a **role ARN** when you want a controlled delegation path.
- Trust an **account root ARN** only when you need all identities in that account to assume the role, usually with additional conditions.
- Trust an **AWS service principal** only for service-to-service access, not for human or application cross-account delegation.

A trust policy can also include conditions to reduce risk:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/SourceRole"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalArn": "arn:aws:iam::111122223333:role/SourceRole"
        }
      }
    }
  ]
}
```

In practice, conditions are often used with:

- `aws:PrincipalArn` to constrain the caller.
- `aws:PrincipalOrgID` to trust identities only from a specific AWS Organizations organization.
- `sts:ExternalId` for third-party access patterns.
- `aws:MultiFactorAuthPresent` when you want MFA-based protection for human access.

**Practical CLI workflow**

You can configure and test cross-account role access entirely from the AWS CLI. The exact account IDs and names vary by lab, but the workflow is consistent.

1. Create or update the role trust policy in the target account.
2. Attach the required permissions policy to the role.
3. Ensure the source principal can call `sts:AssumeRole`.
4. Assume the role from the source account.
5. Verify the temporary credentials by calling an API in the target account.

Example role creation:

```bash
aws iam create-role \
  --role-name CrossAccountReadOnlyRole \
  --assume-role-policy-document file://trust-policy.json
```

Attach a permissions policy:

```bash
aws iam put-role-policy \
  --role-name CrossAccountReadOnlyRole \
  --policy-name ReadOnlyS3Access \
  --policy-document file://permissions-policy.json
```

Assume the role from the source account:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::222233334444:role/CrossAccountReadOnlyRole \
  --role-session-name lab-session
```

The response includes temporary credentials:

```json
{
  "Credentials": {
    "AccessKeyId": "ASIA...",
    "SecretAccessKey": "...",
    "SessionToken": "...",
    "Expiration": "2026-04-26T12:34:56Z"
  },
  "AssumedRoleUser": {
    "AssumedRoleId": "AROA...",
    "Arn": "arn:aws:sts::222233334444:assumed-role/CrossAccountReadOnlyRole/lab-session"
  }
}
```

Export the credentials for a shell session:

```bash
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
```

Then verify access:

```bash
aws s3 ls s3://example-bucket
aws sts get-caller-identity
```

`get-caller-identity` is especially useful because it confirms the active principal and account without requiring extra permissions.

**Common lab configuration patterns**

Cross-account labs often test one of these patterns:

- **Read-only access to a shared resource**
  - Example: Account A reads an S3 bucket in Account B.
  - The role in Account B grants only `GetObject` and `ListBucket`.
- **Write access to a central logging or data account**
  - Example: Account A writes objects to a bucket in Account B.
  - The role or bucket policy must allow `PutObject`.
- **Administrative delegation**
  - Example: A platform team role in a management account assumes a role in a workload account.
  - Use tightly scoped permissions and session duration controls.
- **Third-party access**
  - Example: An external vendor assumes a role using `sts:ExternalId`.
  - This reduces the confused deputy risk.

A bucket policy may also be required if the resource itself enforces access. For S3, both IAM permissions and bucket policy can matter:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRoleRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/CrossAccountReadOnlyRole"
      },
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

**Resource-based policies versus role-based access**

Cross-account access can be granted in two broad ways:

| Approach | Where permission lives | Typical use |
|---|---|---|
| Role-based access | IAM role in target account | Temporary credentials, delegated administration |
| Resource-based access | Resource policy on S3, KMS, SNS, SQS, Lambda, etc. | Direct cross-account resource sharing |

Role-based access is the focus of this lab because it is the most general pattern and works across many services. Resource-based policies are still important because some services require them in addition to IAM permissions.

For example:

- **S3** often uses both IAM and bucket policies.
- **KMS** key policies and grants control whether a role can use a key.
- **SNS** and **SQS** commonly rely on resource policies for cross-account access.
- **Lambda** can use resource-based permissions for invocation.

**Troubleshooting access failures**

When cross-account access fails, check the problem in layers:

- **Trust policy**
  - Is the source principal listed correctly?
  - Is the action `sts:AssumeRole` allowed?
- **Caller permissions**
  - Does the source identity have permission to call `sts:AssumeRole` on the target role ARN?
- **Role permissions**
  - Does the assumed role have the needed resource permissions?
- **Resource policy**
  - Does the target resource allow the role principal?
- **Session context**
  - Are you using the temporary credentials from the assumed role session?
- **Explicit deny**
  - Is there a service control policy, permission boundary, or resource policy deny?

Useful diagnostics:

```bash
aws sts get-caller-identity
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::222233334444:role/CrossAccountReadOnlyRole \
  --action-names s3:ListBucket \
  --resource-arns arn:aws:s3:::example-bucket
```

If `AssumeRole` fails, the error often points to one of these issues:

- `AccessDenied` from STS usually means the trust policy or caller permission is wrong.
- `MalformedPolicyDocument` usually means invalid JSON or unsupported policy syntax.
- `Not authorized to perform: sts:AssumeRole` usually means the source identity lacks permission.
- `AccessDenied` on the target service usually means the role lacks resource permissions or the resource policy blocks access.

> If the role assumption succeeds but the service call fails, the trust policy is probably correct and the problem is in the permissions policy or resource policy.

**Security best practices**

Use least privilege at every layer. Cross-account access is powerful, so small mistakes can create broad exposure.

- Trust only the exact principal you need.
- Grant only the actions required for the task.
- Prefer short session durations when possible.
- Use MFA for human operators when appropriate.
- Use `ExternalId` for third-party delegation.
- Monitor role usage with AWS CloudTrail.
- Review and rotate trust relationships when accounts or teams change.

A more restrictive trust policy for a third-party integration may look like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "vendor-12345"
        }
      }
    }
  ]
}
```

**Exam and lab takeaways**

For SAP-C02, you should be able to reason about which policy controls which part of the flow. The most common exam pattern is a scenario where one account needs controlled access to another account’s resources with minimal operational overhead.

Remember these points:

- The **trust policy** allows assumption.
- The **permissions policy** allows actions after assumption.
- The **caller** also needs permission to call `sts:AssumeRole`.
- Some services still require a **resource policy**.
- Temporary credentials are the standard and preferred result of cross-account role assumption.

If a question asks for secure delegation between accounts, the best answer is usually a role in the target account with a tightly scoped trust policy and least-privilege permissions, verified through STS and CloudTrail.

**Quick reference checklist**

- Confirm the target role exists in the destination account.
- Confirm the trust policy names the correct source principal.
- Confirm the source principal can call `sts:AssumeRole`.
- Confirm the role has the required permissions.
- Confirm any resource policy allows the role.
- Test with `aws sts assume-role` and `aws sts get-caller-identity`.
- Inspect CloudTrail if the behavior is unclear.
