## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Centralized logging and security account design**

A centralized logging pattern separates duties across accounts so you can collect, retain, and analyze security telemetry without giving every workload broad access to audit data. In a typical enterprise landing zone, you use a dedicated **log archive** account for immutable storage, a **security tooling** account for detection and response, and one or more **workload** accounts that emit logs.

This pattern improves isolation, reduces blast radius, and makes it easier to apply consistent retention, encryption, and access controls. It also supports compliance because you can prove who generated logs, where they were stored, and how they were protected.

- **Log archive account** stores raw logs with tight write-only ingestion paths.
- **Security account** hosts detection services, dashboards, and incident workflows.
- **Management account** usually owns organization-wide configuration and delegated administration.
- **Workload accounts** produce service logs but should not manage central retention.

**Core design goals**

Centralized logging is not just about collecting everything in one place. You want durable evidence, least privilege, and operational simplicity. The best designs separate ingestion from analysis so that a compromise in a workload account does not let an attacker alter historical audit records.

- **Immutability** for audit evidence, typically with S3 Object Lock where supported and appropriate.
- **Encryption** with AWS Key Management Service (AWS KMS) and tightly scoped key policies.
- **Separation of duties** between administrators, security analysts, and application teams.
- **Organization-wide coverage** using AWS Organizations and delegated administration where supported.
- **Lifecycle management** to move older logs to cheaper storage without losing retention guarantees.

**AWS Organizations and delegated administration**

AWS Organizations lets you standardize logging across accounts and apply service integrations at the organization level. For many security services, you enable trusted access and then register a delegated administrator in the security account. This keeps the management account from becoming the day-to-day operations account.

Common services that fit this model include AWS CloudTrail, AWS Config, Amazon GuardDuty, AWS Security Hub, Amazon Macie, and AWS Security Lake. The exact delegated administrator workflow differs by service, so you should know the pattern rather than memorize one command for all services.

- **Trusted access** allows a service to read organization metadata and manage member-account integration.
- **Delegated administrator** is the account that performs service administration for the organization.
- **Management account** remains the root of the organization but should be minimized for routine security operations.

> Organization-wide services are usually enabled from the management account first, then delegated to the security account. The service-specific API determines whether the admin is registered before or after member enrollment.

**CloudTrail architecture**

AWS CloudTrail records API activity and is the primary control-plane audit source. For enterprise use, you usually create an **organization trail** so events from all member accounts flow to a central S3 bucket. Multi-Region trails are not the default; you must enable them explicitly when you need coverage across Regions.

CloudTrail can also deliver management events to CloudWatch Logs for near-real-time alerting, while S3 remains the durable archive. For data events, you selectively enable high-value resources such as S3 buckets, Lambda functions, or DynamoDB tables because data events can be high volume and costly.

- **Organization trail** captures events from all accounts in the organization.
- **Multi-Region** coverage must be enabled when you need global control-plane visibility.
- **Log file validation** helps detect tampering by verifying digest files.
- **KMS encryption** protects log files at rest in the archive bucket.

A common S3 bucket policy for CloudTrail must allow the service to write objects and read bucket ACLs, and it must scope access to the correct organization or account path. For organization trails, the object prefix usually includes the organization ID structure, not just a single account ID.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSCloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::central-audit-logs",
      "Condition": {
        "StringEquals": {
          "aws:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/OrgTrail"
        }
      }
    },
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::central-audit-logs/AWSLogs/o-abc123xyz/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control",
          "aws:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/OrgTrail"
        }
      }
    }
  ]
}
```

- **Bucket owner full control** ensures the archive account owns delivered objects.
- **SourceArn** helps restrict writes to the expected trail.
- **Organization prefix** prevents misleading examples that only match one account.

For stronger archive protection, prefer **S3 Object Lock** with compliance or governance mode when your retention requirements justify it. MFA Delete exists, but it is operationally awkward and is not the primary modern control for immutable audit archives.

**CloudTrail operational examples**

You often create the trail in the management account or in a delegated administrator account after enabling trusted access. The following examples show the shape of common operations.

```bash
aws organizations enable-aws-service-access \
  --service-principal cloudtrail.amazonaws.com

aws cloudtrail create-trail \
  --name OrgTrail \
  --s3-bucket-name central-audit-logs \
  --is-organization-trail \
  --is-multi-region-trail \
  --enable-log-file-validation \
  --kms-key-id arn:aws:kms:us-east-1:111122223333:key/abcd-1234
```

```bash
aws cloudtrail put-event-selectors \
  --trail-name OrgTrail \
  --event-selectors '[
    {
      "ReadWriteType": "All",
      "IncludeManagementEvents": true,
      "DataResources": [
        {
          "Type": "AWS::S3::Object",
          "Values": ["arn:aws:s3:::critical-bucket/"]
        }
      ]
    }
  ]'
```

- **Management events** are usually enabled broadly.
- **Data events** should be targeted to sensitive resources.
- **Validation** is useful when you need evidence integrity.

**AWS Config in a centralized model**

AWS Config records resource configuration history and evaluates compliance against rules. In a centralized design, you typically aggregate configuration data from multiple accounts and Regions into a single account for visibility and governance. The aggregator is configured with an **OrganizationAggregationSource** when you want organization-wide collection.

AWS Config is not a replacement for CloudTrail. CloudTrail tells you what API call happened; Config tells you what the resource looked like over time. Together they support forensic analysis and drift detection.

- **Configuration recorder** captures resource state changes.
- **Delivery channel** sends snapshots and history to S3.
- **Aggregator** centralizes visibility across accounts and Regions.
- **Config rules** evaluate compliance continuously or on change.

A correct organization aggregation setup uses the organization source structure rather than a simple role-ARN-only pattern.

```bash
aws configservice put-configuration-aggregator \
  --configuration-aggregator-name OrgAggregator \
  --organization-aggregation-source RoleArn=arn:aws:iam::111122223333:role/AWSConfigRole,AllAwsRegions=true
```

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder '{
    "name": "default",
    "roleARN": "arn:aws:iam::111122223333:role/AWSConfigRole",
    "recordingGroup": {
      "allSupported": true,
      "includeGlobalResourceTypes": true
    }
  }'
```

- **AllAwsRegions** is useful when you need broad compliance coverage.
- **Global resource types** matter for IAM and other global services.
- **Recorder role** needs permission to deliver to S3 and publish to Config.

**GuardDuty and Security Hub organization patterns**

Amazon GuardDuty provides threat detection from logs, DNS, VPC Flow Logs, and other signals. Security Hub aggregates findings and normalizes them across services. In an organization, you usually designate a delegated administrator in the security account, then enable member accounts to send findings centrally.

The current model uses delegated administrator registration APIs rather than older one-off enablement commands. The exact CLI syntax can vary by service and API version, so focus on the workflow: enable trusted access, register the admin, then invite or auto-enable members as supported.

- **GuardDuty** centralizes threat detection and can auto-enable member accounts.
- **Security Hub** centralizes findings and standards checks.
- **Delegated admin** should live in the security account, not the workload account.
- **Auto-enable** reduces onboarding friction for new accounts.

Example workflow shape:

```bash
aws organizations enable-aws-service-access \
  --service-principal guardduty.amazonaws.com

aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal guardduty.amazonaws.com
```

```bash
aws organizations enable-aws-service-access \
  --service-principal securityhub.amazonaws.com

aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal securityhub.amazonaws.com
```

After delegation, you enable the service in the security account and configure member auto-enablement or invitations according to the service’s current behavior.

- **Detector** in GuardDuty is the regional resource that processes findings.
- **Hub** in Security Hub is the regional aggregation point.
- **Regional setup** matters because both services are Region-scoped.

**CloudWatch Logs and cross-account delivery**

CloudWatch Logs is useful for near-real-time operational and security alerting. Subscription filters can send log streams to Kinesis Data Streams, Kinesis Data Firehose, or Lambda. Cross-account delivery requires the correct destination policy and permissions; it is not enough to point at a destination ARN without authorizing the source account.

Use CloudWatch Logs when you need fast processing, but keep S3 or another durable archive for long-term retention. Firehose is often used to land logs into S3, OpenSearch Service, or third-party endpoints, but the subscription filter targets the CloudWatch Logs destination model, not a raw Firehose ARN in isolation.

- **Subscription filter** selects matching log events.
- **Destination policy** authorizes cross-account delivery.
- **Firehose** is a delivery path, not a substitute for archive governance.
- **Lambda** is useful for lightweight transformation or alerting.

Example pattern:

```bash
aws logs put-destination \
  --destination-name CentralSecurityDestination \
  --target-arn arn:aws:firehose:us-east-1:111122223333:deliverystream/security-logs \
  --role-arn arn:aws:iam::111122223333:role/CWLtoFirehoseRole
```

```bash
aws logs put-subscription-filter \
  --log-group-name /aws/lambda/app-prod \
  --filter-name SecurityEvents \
  --filter-pattern '"ERROR" "AccessDenied"' \
  --destination-arn arn:aws:logs:us-east-1:111122223333:destination:CentralSecurityDestination
```

> Cross-account log delivery fails most often because the destination policy, IAM role trust, or Region alignment is incomplete.

**Security Lake and data normalization**

Amazon Security Lake centralizes security data in an open format for analytics and investigation. It is valuable for normalized querying across multiple sources, but it does not replace raw immutable archives or every service-native logging path. Treat it as an analytics and correlation layer, not the only system of record.

Security Lake is especially useful when you want to query multiple security sources with a common schema and reduce the effort of building custom ingestion pipelines. You still keep CloudTrail, Config, and service-native archives for compliance and forensic fidelity.

- **Normalized schema** helps analysts query heterogeneous sources.
- **Lake formation-style access patterns** may apply depending on the deployment model.
- **Analytics layer** complements, not replaces, the archive layer.
- **Retention strategy** should still preserve original logs where required.

**S3 archive bucket design**

The central archive bucket is one of the most important controls in the pattern. It should be dedicated to logs, encrypted, versioned, and protected from accidental deletion. You usually deny public access, restrict write access to AWS services and ingestion roles, and allow read access only to security and audit personas.

Recommended controls include:

- **Block Public Access** on every log bucket.
- **Versioning** to preserve object history.
- **SSE-KMS** with a key policy that allows only approved principals.
- **Lifecycle policies** to transition older logs to cheaper storage classes.
- **Object Lock** when immutable retention is required.

A practical bucket policy often includes explicit denies for non-TLS access and broad writes, plus service-specific allows for CloudTrail, Config, and other producers.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyInsecureTransport",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::central-audit-logs",
        "arn:aws:s3:::central-audit-logs/*"
      ],
      "Condition": {
        "Bool": { "aws:SecureTransport": "false" }
      }
    }
  ]
}
```

- **Explicit deny** is a strong guardrail for archive buckets.
- **TLS enforcement** protects data in transit.
- **Least privilege** should apply to both humans and services.

**KMS key policy considerations**

If you encrypt logs with AWS KMS, the key policy must allow the logging services to use the key and allow the archive account to administer it. A common mistake is granting broad IAM permissions but forgetting the key policy, which blocks delivery.

- **Service principal access** is required for delivery encryption.
- **Key rotation** is useful for long-lived audit keys.
- **Separation of duties** should keep key admins distinct from log readers when possible.

Example policy shape:

```json
{
  "Sid": "AllowCloudTrailUseOfKey",
  "Effect": "Allow",
  "Principal": { "Service": "cloudtrail.amazonaws.com" },
  "Action": [
    "kms:GenerateDataKey*",
    "kms:Decrypt",
    "kms:DescribeKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "aws:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/OrgTrail"
    }
  }
}
```

**Operational access and incident response**

Security teams need read access to logs, but they should not have write access to the archive bucket. Use separate roles for analysts, responders, and platform administrators. For incident response, you often combine CloudTrail, Config, GuardDuty, Security Hub, and VPC Flow Logs to reconstruct the timeline.

A practical response workflow is:

1. Review GuardDuty or Security Hub findings for the initial alert.
2. Correlate the event with CloudTrail API activity.
3. Check AWS Config history for resource drift or unauthorized changes.
4. Inspect VPC Flow Logs, ALB logs, or application logs for lateral movement or exfiltration.
5. Preserve relevant evidence in a case bucket or ticket attachment workflow.

- **Correlation** is more valuable than any single log source.
- **Timeline reconstruction** depends on consistent timestamps and retention.
- **Evidence preservation** should avoid modifying original archives.

**Common exam traps**

The exam often tests whether you know the difference between collection, aggregation, and analysis. It also checks whether you understand which account should own which control plane.

- **CloudTrail** is for API auditing, not resource configuration history.
- **Config** is for configuration state, not API call history.
- **GuardDuty** detects threats, but it does not replace audit logs.
- **Security Hub** aggregates findings, but it does not generate them.
- **Security Lake** improves analytics, but it is not the only archive you need.
- **Multi-Region trails** must be enabled explicitly.
- **Delegated administration** is service-specific and not identical across AWS services.

**Reference architecture summary**

A strong enterprise pattern usually looks like this:

- **Management account**
  - Owns the AWS Organization.
  - Enables trusted access for security services.
  - Minimizes day-to-day operations.

- **Security account**
  - Hosts delegated admin for GuardDuty, Security Hub, and similar services.
  - Runs analyst tooling, dashboards, and automation.
  - Receives alerts and findings.

- **Log archive account**
  - Stores CloudTrail, Config, and other raw logs.
  - Uses KMS encryption, versioning, and retention controls.
  - Restricts write access to AWS services and ingestion roles.

- **Workload accounts**
  - Produce logs and findings.
  - Do not control central retention.
  - Use standardized guardrails and service-linked roles.

This separation gives you durable auditability, cleaner permissions, and a design that scales as the organization grows.
