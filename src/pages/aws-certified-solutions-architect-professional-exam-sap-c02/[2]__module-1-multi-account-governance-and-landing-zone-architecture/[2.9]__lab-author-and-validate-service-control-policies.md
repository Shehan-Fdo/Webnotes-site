## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Service Control Policies in AWS Organizations**

Service Control Policies (SCPs) are organization-level guardrails that define the maximum permissions available to accounts in AWS Organizations. They do not grant permissions by themselves; they only limit what identity-based policies, resource-based policies, permission boundaries, and session policies can ultimately allow.

A correct mental model is:

- **SCPs** set the outer boundary for member accounts.
- **IAM policies** and other permission controls grant or further restrict access inside that boundary.
- The **management account** is not affected by SCPs, even though it can sit at the organization root.
- SCPs apply to **member accounts** attached to the root, an organizational unit (OU), or directly to the account.

Use SCPs when you need centralized governance such as:

- Blocking risky services across a business unit.
- Restricting regions for compliance.
- Preventing accidental deletion of critical resources.
- Creating sandbox accounts with tightly controlled service access.

**Core evaluation behavior**

AWS evaluates permissions by combining multiple layers. SCPs are checked first as a guardrail for member accounts, then identity and resource permissions determine whether the action is actually allowed.

A practical way to think about it:

- If an SCP denies an action, the action is blocked regardless of IAM.
- If an SCP does not allow an action in an allow-list model, the action is blocked even if IAM allows it.
- If SCPs allow the action, IAM and other controls still must allow it.

> SCPs never grant access on their own. An allow statement in an SCP only makes the action eligible; it does not authorize the action without matching IAM or resource permissions.

**Organization structure and attachment points**

SCPs can be attached at different levels in the organization hierarchy:

- **Root**
  - Applies to all member accounts in the organization unless overridden by more specific SCPs.
  - Common for baseline guardrails such as region restrictions.
- **OU**
  - Applies to accounts in that OU and inherited child OUs.
  - Useful for department-specific controls.
- **Account**
  - Applies only to one member account.
  - Useful for exceptions or highly specific restrictions.

Important nuances:

- SCPs are inherited downward through the hierarchy.
- An explicit deny in any applicable SCP wins.
- In an allow-list strategy, the effective permissions are the intersection of all applicable SCPs.
- The management account is excluded from SCP enforcement.

**SCP policy types**

There are two common patterns:

- **Deny-list model**
  - Start with broad access and deny only dangerous actions.
  - Easier to adopt incrementally.
  - Safer for existing workloads because it is less likely to break unknown dependencies.
- **Allow-list model**
  - Deny everything except explicitly allowed actions.
  - Stronger governance.
  - Requires careful testing because many AWS services need additional actions for console use, dependent APIs, and service-linked operations.

A deny-list SCP is often the best first step in production. An allow-list SCP is more appropriate for tightly controlled sandbox or regulated environments where you can enumerate required services.

**Authoring SCPs**

SCPs use standard JSON policy syntax with `Version`, `Statement`, `Effect`, `Action` or `NotAction`, and `Resource`. Conditions are supported and are often essential for region controls and service restrictions.

Example deny-list SCP that blocks disabling CloudTrail and tampering with organization guardrails:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyCloudTrailTampering",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:DeleteTrail",
        "cloudtrail:StopLogging",
        "cloudtrail:UpdateTrail"
      ],
      "Resource": "*"
    },
    {
      "Sid": "DenyOrganizationsPolicyChanges",
      "Effect": "Deny",
      "Action": [
        "organizations:DeletePolicy",
        "organizations:DetachPolicy",
        "organizations:UpdatePolicy",
        "organizations:DisablePolicyType"
      ],
      "Resource": "*"
    }
  ]
}
```

This pattern is useful because it protects auditability and governance even if an administrator has broad IAM permissions.

**Region restriction with `aws:RequestedRegion`**

A common SCP use case is limiting where workloads can run. The `aws:RequestedRegion` condition key lets you deny actions outside approved regions.

Example region guardrail:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "NotAction": [
        "iam:*",
        "organizations:*",
        "route53:*",
        "cloudfront:*",
        "support:*",
        "sts:*"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

Use this carefully:

- Some AWS services are global or have control-plane dependencies that do not behave like regional services.
- Console workflows may call additional APIs you did not expect.
- You should test with real workloads before broad rollout.
- The exception list often needs service-specific tuning.

A safer rollout strategy is to start with a small pilot OU and expand after validating common workflows.

**Allow-list SCPs and why they are tricky**

An allow-list SCP can be useful for sandbox accounts, but it is easy to misunderstand. The SCP does not grant permissions by itself, and many AWS actions require more than a single broad `Resource: "*"` statement to work in practice.

Example of a simplified allow-list SCP for a constrained environment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCoreReadOnlyAndBasicCompute",
      "Effect": "Allow",
      "Action": [
        "ec2:Describe*",
        "ec2:RunInstances",
        "ec2:TerminateInstances",
        "s3:ListBucket",
        "s3:GetObject"
      ],
      "Resource": "*"
    }
  ]
}
```

This is intentionally simplified for study purposes. In real deployments:

- `s3:GetObject` and `s3:PutObject` typically require object ARNs, not just bucket-level resources.
- Console access often needs additional `Describe*`, `List*`, and related permissions.
- EC2 instance launch workflows may require permissions for networking, IAM roles, key pairs, security groups, and dependent services.
- You must still provide IAM permissions to the principal.

A better approach is to validate the exact API calls your workload makes and then build the allow-list from observed requirements.

**Practical lab workflow**

A realistic lab usually follows this sequence:

1. Create or choose a test OU in AWS Organizations.
2. Move a non-production member account into that OU.
3. Attach a baseline SCP to the OU.
4. Test access with an IAM role in the member account.
5. Observe denied actions in CloudTrail and the AWS console.
6. Refine the SCP and retest until the guardrail matches the intended behavior.

This workflow helps you avoid applying an untested policy to production accounts.

**Example lab scenario: restrict a sandbox account**

Imagine a sandbox account used by developers. You want to:

- Allow experimentation with EC2 and S3.
- Prevent IAM privilege escalation.
- Prevent disabling logging.
- Restrict usage to approved regions.

A layered approach works well:

- Attach a region restriction SCP to the sandbox OU.
- Attach a deny SCP for IAM and logging tampering.
- Use IAM roles inside the account to grant the actual developer permissions.

This keeps the account usable while reducing blast radius.

**Validation methods**

You should validate SCPs using more than one method.

- **AWS Organizations console**
  - Check which SCPs are attached to the root, OU, and account.
  - Confirm inheritance and effective policy placement.
- **IAM policy simulator**
  - Useful for testing identity policies, but it does not fully model organization SCP inheritance by itself.
- **AWS Organizations policy tools**
  - Use policy validation and effective policy evaluation features where available.
  - These help you reason about the final permission set.
- **CloudTrail**
  - Shows denied API calls and the service that rejected them.
  - Useful for identifying the exact API action that failed.
- **Real workload testing**
  - The most reliable method for catching missing dependent actions and console-only calls.

A good validation loop is:

1. Attempt the intended action.
2. Capture the denied API call in CloudTrail.
3. Identify whether the failure is due to SCP, IAM, permission boundary, or session policy.
4. Update the policy or the IAM role.
5. Retest.

**Troubleshooting denied access**

When access fails, do not assume the SCP is the only cause. Check the full permission chain.

Common failure modes:

- The SCP denies the action explicitly.
- The SCP allow-list does not include a required dependent action.
- IAM does not grant the action even though the SCP allows it.
- A permission boundary blocks the action.
- A session policy from AWS STS (Security Token Service) narrows the permissions.
- A resource-based policy does not allow the principal.
- The action is global or service-linked and needs an exception.

Useful troubleshooting commands:

```bash
aws organizations list-policies --filter SERVICE_CONTROL_POLICY
aws organizations list-targets-for-policy --policy-id p-abc123
aws organizations describe-effective-policy --policy-type SERVICE_CONTROL_POLICY --target-id 111122223333
aws cloudtrail lookup-events --lookup-attributes AttributeKey=EventName,AttributeValue=RunInstances
```

If you are testing from a role, also inspect the role’s identity policy and any permission boundary:

```bash
aws iam get-role --role-name DevSandboxRole
aws iam get-role-policy --role-name DevSandboxRole --policy-name InlinePolicyName
aws iam get-policy-version --policy-arn arn:aws:iam::111122223333:policy/BoundaryPolicy --version-id v1
```

**Expected debugging signals**

Look for these clues:

- `AccessDenied` with an event name that matches the intended action often indicates a policy restriction.
- A console error may hide the underlying API call, so check CloudTrail for the exact denied operation.
- If the action works with CLI but not console, the console may need extra `List*` or `Describe*` permissions.
- If the action works in one region but not another, the region condition may be too strict.
- If a service-linked role fails to create, the SCP may be blocking a required service API.

**Real-world examples of guardrails**

Common SCP guardrails include:

- Blocking root user usage patterns by denying sensitive APIs to all member accounts.
- Preventing deletion of CloudTrail, AWS Config, or Security Hub resources.
- Restricting `ec2:RunInstances` to approved instance families or regions using conditions.
- Denying creation of internet-facing resources in shared accounts.
- Preventing changes to AWS Organizations policies except from a dedicated governance account.

Example deny for IAM privilege escalation primitives:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyPrivilegeEscalationPaths",
      "Effect": "Deny",
      "Action": [
        "iam:CreatePolicyVersion",
        "iam:SetDefaultPolicyVersion",
        "iam:AttachUserPolicy",
        "iam:AttachRolePolicy",
        "iam:PutUserPolicy",
        "iam:PutRolePolicy",
        "iam:PassRole"
      ],
      "Resource": "*"
    }
  ]
}
```

This is a strong guardrail, but it can also break legitimate automation. Always test service roles, deployment pipelines, and break-glass procedures before enforcing it broadly.

**Common exam distinctions**

For SAP-C02, be precise about what SCPs do and do not do:

- SCPs are for **governance**, not direct authorization.
- SCPs affect **member accounts only**.
- SCPs can be attached to the **root, OU, or account**.
- Explicit deny always wins.
- Allow-list SCPs require matching IAM permissions and careful dependency analysis.
- Region restrictions often use `aws:RequestedRegion`.
- Troubleshooting requires checking CloudTrail, IAM, permission boundaries, session policies, and effective policy evaluation.

**Mental model for effective permissions**

A safer conceptual model is:

- AWS Organizations SCPs define the maximum allowed actions for a member account.
- IAM identity policies, resource policies, permission boundaries, and session policies determine what a principal can do within that maximum.
- The final result is the intersection of all applicable controls, with explicit denies taking precedence.

This avoids the misleading idea of a single linear stack and better matches how AWS evaluates access.

**End-to-end validation example**

Suppose you attach a region-restriction SCP to a sandbox OU and then try to launch an EC2 instance in `eu-west-1`.

1. You sign in with a role that has `ec2:RunInstances`.
2. The console attempts several EC2 and networking API calls.
3. The SCP denies the request because `aws:RequestedRegion` is not in the approved list.
4. CloudTrail records the denied event.
5. You confirm the OU attachment and effective policy.
6. You retry in `us-east-1`, where the request succeeds if IAM and dependent permissions are present.

This end-to-end test proves both the guardrail and the exception behavior.

**Best practices**

- Start with a pilot OU before broad rollout.
- Prefer deny-list SCPs for production guardrails.
- Use allow-list SCPs only when you can enumerate dependencies.
- Keep exception lists explicit and documented.
- Validate with real workloads, not just policy text.
- Review CloudTrail regularly for denied events that indicate policy drift.
- Separate governance administration from workload administration.

**Quick reference**

- **Root attachment**: broadest inheritance for member accounts.
- **OU attachment**: best for department or environment boundaries.
- **Account attachment**: best for one-off exceptions.
- **Deny-list**: safer for incremental adoption.
- **Allow-list**: stricter, but more fragile.
- **CloudTrail**: primary source for denied API evidence.
- **Effective policy evaluation**: best for confirming what a member account can actually do.

**Summary**

SCPs are a central AWS Organizations control for enforcing guardrails across member accounts. For exam and lab purposes, focus on the difference between limiting permissions and granting them, the inheritance model, the special treatment of the management account, and the need to validate policies against real API behavior. The most reliable approach is to author a narrow policy, attach it to a test OU, verify the effective policy, and then iterate based on CloudTrail and workload testing.
