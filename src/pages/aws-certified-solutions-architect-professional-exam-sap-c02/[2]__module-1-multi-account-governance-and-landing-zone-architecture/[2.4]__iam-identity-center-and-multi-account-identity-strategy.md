## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**IAM Identity Center in a multi-account strategy**

AWS IAM Identity Center is the preferred workforce access service for centrally managing human access across an AWS Organization. It helps you grant users access to multiple AWS accounts and business applications without creating and managing separate IAM users in every account. In SAP-C02 scenarios, the key design goal is to separate identity management from account-level authorization while keeping access auditable and scalable.

Identity Center is not a universal replacement for IAM users. IAM users can still exist for legacy integrations, break-glass access, or workloads that require long-lived credentials, but for workforce access you usually prefer Identity Center plus federated sign-in.

- **Centralized access** reduces per-account identity sprawl.
- **Permission sets** define what a user can do after sign-in.
- **Assignments** map users or groups to accounts and applications.
- **AWS Organizations** is the control plane for multi-account enablement.

**Core architecture and control points**

Identity Center is enabled in a specific AWS Region, often called the home Region for the instance. You administer the instance in that Region, while end users access the user portal through the Identity Center start URL. The portal is not the same thing as the admin console, and exam questions often test that distinction.

A typical enterprise design looks like this:

- One AWS Organization with multiple OUs (organizational units).
- One Identity Center instance for the organization.
- One identity source for workforce users.
- Permission sets reused across accounts and environments.
- Group-based assignments for scale and least privilege.

> Identity Center is a workforce access solution. It does not manage application identities for every use case, and it does not eliminate the need for IAM users in all edge cases.

**AWS Organizations prerequisites**

Before you can use Identity Center effectively, the AWS Organization must be set up correctly.

- The management account owns the organization.
- Trusted access must be enabled for Identity Center so it can create and manage the required service-linked relationships.
- You typically delegate administration to a member account for separation of duties.
- Accounts must belong to the organization before you can assign permission sets to them.
- OUs help you organize accounts by environment, business unit, or compliance boundary, but assignments are still made to accounts, not directly to OUs.

A practical governance pattern is to use OUs to structure accounts, then apply group-based assignments to the accounts inside those OUs.

Example OU layout:

```text
Root
├── Security
├── Shared-Services
├── Prod
│   ├── Prod-App1
│   └── Prod-App2
└── NonProd
    ├── Dev-App1
    └── Test-App2
```

You then assign different groups and permission sets to the accounts in each OU.

**Identity source options**

Identity Center can use different identity sources depending on your enterprise design.

- **Built-in Identity Center directory**
  - Useful for smaller environments or quick starts.
  - AWS manages the directory inside Identity Center.
- **External identity provider (IdP)**
  - Common for enterprise SSO with Microsoft Entra ID, Okta, Ping, or similar.
  - Supports existing corporate lifecycle and MFA policies.
- **AWS Managed Microsoft AD**
  - Can be used as an identity source in supported configurations.
  - Often relevant when you already operate Microsoft AD and want directory-based integration.
- **AD Connector**
  - Acts as a proxy to an existing on-premises Microsoft Active Directory.
  - Useful when you want to avoid directory synchronization into AWS.

Choose the source based on where your authoritative identities live, how you handle MFA, and whether you need cloud-only, hybrid, or enterprise SSO integration.

**When to use each identity source**

| Identity source | Best fit | Notes |
|---|---|---|
| Built-in directory | Small teams, pilots, simple setups | Fastest to start, less enterprise integration |
| External IdP | Enterprise SSO and lifecycle management | Common choice for large organizations |
| AWS Managed Microsoft AD | AWS-hosted directory-centric designs | Can integrate with Identity Center in supported ways |
| AD Connector | Existing on-prem AD without directory copy | Depends on network connectivity to AD |

**Permission sets and authorization model**

Permission sets are the core abstraction in Identity Center. A permission set is a reusable template that becomes an IAM role in each target account when assigned.

A permission set can include:

- AWS managed policies
- Customer managed policies
- Inline policies
- Permission boundaries in some designs
- Session duration
- Tags for attribute-based access control (ABAC)

The permission set does not directly grant access by itself. Access happens only after you assign a user or group to an account with that permission set.

Example permission set design:

- `ReadOnly-Dev`
- `PowerUser-NonProd`
- `Admin-ProdBreakGlass`
- `Billing-SharedServices`

This lets you reuse the same access model across many accounts without creating separate IAM roles manually in each account.

**How permission sets become IAM roles**

When you assign a permission set to an account, Identity Center provisions an IAM role in that account. The role name typically follows the `AWSReservedSSO_...` pattern.

Important behavior:

- The role trust policy allows Identity Center to assume the role on behalf of the assigned principal.
- The role permissions reflect the permission set definition.
- If you remove the assignment, Identity Center cleans up the corresponding role after deprovisioning.
- If you change the permission set, Identity Center may need to reprovision the target account role so the changes take effect.

Example of the resulting role pattern:

```text
AWSReservedSSO_ReadOnly-Dev_1234567890abcdef
```

This role is managed by Identity Center, so you should not treat it like a hand-crafted IAM role.

> Do not edit `AWSReservedSSO_` roles manually. Identity Center can overwrite or replace them during reprovisioning.

**Session duration and access behavior**

Permission sets include a session duration setting, but the effective session length is still subject to AWS service behavior and role session constraints. In practice, you use session duration to balance usability and security.

- Shorter sessions reduce exposure if credentials are stolen.
- Longer sessions reduce reauthentication friction for operators.
- High-privilege roles often use shorter durations than read-only roles.

Example CLI to create a permission set with a session duration:

```bash
aws sso-admin create-permission-set \
  --instance-arn arn:aws:sso:::instance/ssoins-1234567890abcdef \
  --name ReadOnly-Dev \
  --session-duration PT4H
```

`PT4H` is an ISO 8601 duration value meaning 4 hours.

**Assignments: users, groups, accounts, and applications**

Assignments connect identities to access targets. For AWS account access, you usually assign groups rather than individual users so you can manage membership centrally in the IdP or directory.

Common patterns:

- Assign a `Developers` group to `Dev` accounts with `PowerUser-NonProd`.
- Assign an `Auditors` group to all accounts with `ReadOnlyAccess`.
- Assign a `Finance` group to shared services accounts with billing permissions.
- Assign a `PlatformAdmins` group to production accounts with tightly controlled admin access.

Example assignment strategy:

```text
Group: Developers
  - Dev-App1 -> PowerUser-NonProd
  - Dev-App2 -> PowerUser-NonProd

Group: Auditors
  - Prod-App1 -> ReadOnly-Prod
  - Prod-App2 -> ReadOnly-Prod
  - Shared-Services -> ReadOnly-SharedServices
```

This model scales better than per-user assignments and makes access reviews easier.

**CLI examples for common administration tasks**

Use the IAM Identity Center and SSO Admin APIs for automation and inventory.

List Identity Center instances:

```bash
aws sso-admin list-instances
```

List permission sets in an instance:

```bash
aws sso-admin list-permission-sets \
  --instance-arn arn:aws:sso:::instance/ssoins-1234567890abcdef
```

Describe a permission set:

```bash
aws sso-admin describe-permission-set \
  --instance-arn arn:aws:sso:::instance/ssoins-1234567890abcdef \
  --permission-set-arn arn:aws:sso:::permissionSet/ssoins-1234567890abcdef/ps-1234567890abcdef
```

List assignments for an account and permission set:

```bash
aws sso-admin list-account-assignments \
  --instance-arn arn:aws:sso:::instance/ssoins-1234567890abcdef \
  --account-id 111122223333 \
  --permission-set-arn arn:aws:sso:::permissionSet/ssoins-1234567890abcdef/ps-1234567890abcdef
```

Provision a permission set to an account after changes:

```bash
aws sso-admin provision-permission-set \
  --instance-arn arn:aws:sso:::instance/ssoins-1234567890abcdef \
  --permission-set-arn arn:aws:sso:::permissionSet/ssoins-1234567890abcdef/ps-1234567890abcdef \
  --target-type AWS_ACCOUNT \
  --target-id 111122223333
```

**User portal and admin portal**

The user portal is where end users sign in and launch AWS accounts or assigned applications. The admin console is where administrators configure Identity Center, permission sets, assignments, and identity source settings.

User portal behavior:

- Users see only the accounts and apps assigned to them.
- Users can switch between accounts without separate IAM credentials.
- Users can launch the AWS access portal and then choose a role/account combination.

Admin behavior:

- You manage permission sets and assignments.
- You configure identity source and MFA-related settings through the chosen IdP or directory model.
- You monitor provisioning status and troubleshoot assignment failures.

A user may see multiple tiles in the portal, such as:

- `Prod - ReadOnly`
- `Dev - PowerUser`
- `Finance App`
- `Support Console`

**ABAC and advanced governance**

Attribute-based access control uses identity or resource attributes to make authorization decisions. In Identity Center, you can use session tags and permission set tags to support ABAC patterns.

This is useful when you want access to scale by attributes instead of static role lists.

Examples:

- Department-based access: `department=finance`
- Environment-based access: `env=prod`
- Cost-center-based access: `cost-center=1234`

A common pattern is to pass attributes from the IdP into the session and then use IAM policy conditions to authorize access.

Example IAM policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3AccessForMatchingDepartment",
      "Effect": "Allow",
      "Action": "s3:*",
      "Resource": "arn:aws:s3:::company-data/*",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalTag/department": "finance"
        }
      }
    }
  ]
}
```

Use ABAC when you need flexible, large-scale access control and your identity source can reliably provide attributes.

**Inline policies vs managed policies**

Permission sets can include both inline policies and managed policies.

- **Managed policies**
  - Easier to reuse and audit.
  - Good for standard job functions.
- **Inline policies**
  - Useful for tightly scoped exceptions or custom conditions.
  - Harder to reuse at scale.

A practical design is to use managed policies for baseline access and inline policies for narrow exceptions.

Example permission set policy attachment approach:

```bash
aws sso-admin attach-managed-policy-to-permission-set \
  --instance-arn arn:aws:sso:::instance/ssoins-1234567890abcdef \
  --permission-set-arn arn:aws:sso:::permissionSet/ssoins-1234567890abcdef/ps-1234567890abcdef \
  --managed-policy-arn arn:aws:iam::aws:policy/ReadOnlyAccess
```

**OU-based design patterns**

Although assignments are made to accounts, OUs are still important for governance and automation.

Common patterns:

- Use OUs to separate prod from nonprod.
- Use SCPs to enforce guardrails at the OU level.
- Use Identity Center groups and permission sets for human access.
- Use account vending to place new accounts into the correct OU automatically.

Example scenario:

- `Prod` OU:
  - `PlatformAdmins` get `Admin-ProdBreakGlass`
  - `Auditors` get `ReadOnly-Prod`
- `NonProd` OU:
  - `Developers` get `PowerUser-NonProd`
  - `QA` gets `ReadOnly-NonProd`

This gives you a clean separation between organizational guardrails and user authorization.

**CloudTrail, audit, and visibility**

Identity Center administration and sign-in activity should be part of your audit strategy.

- Use **CloudTrail** to record administrative API calls such as permission set changes and assignments.
- Review sign-in and federation-related events for access investigations.
- Centralize logs in a security account or log archive account.
- Correlate Identity Center events with IAM role assumptions in target accounts.

Useful audit questions:

- Who changed a permission set?
- Which group was assigned to production?
- When was a role provisioned or deprovisioned?
- Which account did a user access from the portal?

CloudTrail helps answer these questions when you need evidence for compliance or incident response.

**Failure, recovery, and migration considerations**

Identity Center designs should account for outages and lifecycle events.

- If the external IdP is unavailable, user sign-in may be impacted even if AWS is healthy.
- If the Identity Center instance is deleted and recreated, assignments and permission set mappings may need to be rebuilt.
- During migrations, preserve access by planning a parallel identity path or staged cutover.
- For break-glass access, keep a tightly controlled fallback mechanism that does not depend on the primary workforce IdP.

Practical resilience patterns:

- Maintain a small number of emergency IAM users or roles for break-glass access.
- Document how to restore assignments and permission sets from infrastructure as code.
- Export configuration state where possible.
- Test IdP outage behavior before production cutover.

> Do not rely on a single identity provider without a recovery plan. Workforce access can fail even when AWS services are healthy.

**Common exam scenarios**

You may see questions that ask you to choose the best identity strategy for a multi-account organization.

Look for these clues:

- Many accounts and many users: choose Identity Center with group-based assignments.
- Need centralized SSO across AWS and SaaS apps: use an external IdP with Identity Center.
- Need environment-specific access: use separate permission sets per environment.
- Need least privilege at scale: use ABAC or group-based role templates.
- Need strong separation of duties: delegate administration and use OU-based governance.

Example decision pattern:

- Developers need access to dev and test accounts only.
- Auditors need read-only access to all accounts.
- Production admins need elevated access only in production and only through a controlled group.

That usually maps to:

- `Developers` group -> `PowerUser-NonProd`
- `Auditors` group -> `ReadOnly-All`
- `ProdAdmins` group -> `Admin-Prod`

**Practical setup workflow**

1. Create or verify the AWS Organization.
2. Enable IAM Identity Center in the chosen home Region.
3. Select the identity source that matches your enterprise directory model.
4. Create groups that reflect job functions or access tiers.
5. Define permission sets for each access pattern.
6. Assign groups to accounts and applications.
7. Test portal access and role switching.
8. Review CloudTrail and provisioning status.
9. Automate changes with IaC or CLI where possible.

**Troubleshooting checklist**

- Confirm the account is in the AWS Organization.
- Confirm Identity Center is enabled in the correct home Region.
- Confirm the identity source is healthy and synchronized.
- Confirm the user is in the expected group.
- Confirm the permission set is provisioned to the target account.
- Confirm the assignment exists for the correct account and permission set.
- Confirm the user is signing in through the correct portal URL.
- Confirm SCPs are not blocking the resulting role actions.

**Key takeaways**

Identity Center is the scalable way to manage workforce access across many AWS accounts. The main design ideas are centralized identity, reusable permission sets, group-based assignments, and organization-wide governance. For SAP-C02, focus on how Identity Center interacts with AWS Organizations, how permission sets become `AWSReservedSSO_` roles, and how to design for auditability, recovery, and least privilege.
