## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Scenario framing**

A regulated global enterprise usually needs a multi-account AWS environment with strong separation of duties, centralized governance, auditable controls, regional data residency, and resilient global delivery. The architecture must support business units, shared services, security operations, and application teams without letting every team build its own isolated cloud stack.

The exam often tests whether you can combine AWS Organizations, AWS Control Tower, identity federation, network segmentation, centralized logging, and disaster recovery into one coherent landing zone. The right answer is rarely a single service; it is usually a pattern that balances governance, operational scale, and exception handling.

**Core landing zone design**

A production enterprise landing zone starts with AWS Organizations and a clear account model. You typically separate accounts by function rather than by application alone.

- **Management account**: keep it for organization administration only.
- **Security account**: centralize detective and response tooling.
- **Log archive account**: store immutable logs and audit evidence.
- **Shared services account**: host DNS, CI/CD, directory connectors, or common tooling.
- **Network account**: own Transit Gateway, inspection, and egress controls.
- **Workload accounts**: isolate applications by environment, region, or business unit.

Use organizational units (OUs) to apply policy boundaries. Common OUs include `Security`, `Infrastructure`, `Sandbox`, `Production`, and `Suspended`. Service control policies (SCPs) set maximum permissions, but they do not grant access; IAM still controls what identities can do inside the boundary.

A practical governance baseline often includes:

- Deny disabling CloudTrail, Config, GuardDuty, Security Hub, or AWS Backup in member accounts.
- Restrict regions to approved geographies.
- Prevent public S3 bucket creation unless explicitly exempted.
- Block root user access keys and require MFA for privileged actions.
- Limit network changes to approved automation roles.

> SCPs are guardrails, not permissions. If you deny an action with an SCP, no identity in the account can bypass it.

**AWS Organizations trusted access and delegated administration**

For centralized security and governance at scale, you usually enable trusted access for AWS services that integrate with Organizations. Trusted access lets a service manage organization-wide resources or settings on your behalf. Delegated administrator designates a member account to administer a service for the organization, which is useful when you do not want the management account to run day-to-day security operations.

Common delegated administrator patterns include:

- **Security Hub**: central security findings aggregation and standards management.
- **GuardDuty**: centralized threat detection across accounts and Regions.
- **AWS Config**: organization-wide configuration recording and conformance packs.
- **AWS Backup**: centralized backup policies and cross-account backup governance.
- **IAM Identity Center**: centralized workforce access management.
- **CloudFormation StackSets**: organization-wide deployment of baseline resources.

A typical sequence is:

1. Enable trusted access for the service in AWS Organizations.
2. Register a delegated administrator account.
3. Turn on organization-wide service configuration.
4. Apply policies, standards, or aggregators from the delegated account.
5. Monitor for member accounts that opt out or drift.

Example AWS CLI commands:

```bash
aws organizations enable-aws-service-access --service-principal securityhub.amazonaws.com
aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal securityhub.amazonaws.com

aws organizations enable-aws-service-access --service-principal guardduty.amazonaws.com
aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal guardduty.amazonaws.com
```

**AWS Control Tower and when to extend it**

AWS Control Tower accelerates landing zone setup by creating a governed multi-account baseline with guardrails, account vending, and preconfigured logging. It is a strong default when you want a standard enterprise foundation quickly.

Control Tower is not a complete replacement for enterprise architecture work. You often extend it when you need:

- **Account Factory for Terraform (AFT)** to provision accounts and customize post-provisioning workflows as code.
- **Custom guardrails** beyond the built-in preventive and detective controls.
- **Existing identity integration** with enterprise directories, custom federation flows, or complex role mapping.
- **Existing network architecture** such as prebuilt Transit Gateway hubs, SD-WAN integration, or centralized inspection.
- **Special compliance workflows** that require custom evidence collection, tagging, or approval steps.

Control Tower limitations to keep in mind:

- It standardizes the landing zone, but it does not design your application network or data platform.
- It may not cover every enterprise exception or bespoke control requirement.
- It works best when your organization can align to its opinionated baseline.
- Deep customization is possible, but you should plan for lifecycle management so upgrades do not break your extensions.

A common pattern is Control Tower for baseline governance plus AFT or StackSets for custom account bootstrapping, network attachments, and workload-specific configuration.

**Identity and access**

Enterprise identity usually relies on federation with an external identity provider (IdP) such as Microsoft Entra ID, Okta, or another SAML 2.0 or OIDC provider. AWS IAM Identity Center is the preferred workforce access layer for many regulated enterprises because it centralizes permission sets, account assignments, and session management.

Key identity concepts:

- **Federation**: users authenticate with the corporate IdP and receive AWS access without separate long-lived IAM users.
- **SCIM provisioning**: System for Cross-domain Identity Management automatically syncs users and groups from the IdP into IAM Identity Center.
- **Permission sets**: reusable role templates that define session duration, managed policies, and inline policies.
- **Attribute-based access control (ABAC)**: use identity or resource tags to scale access control across many accounts.

Practical guidance:

- Use IAM Identity Center for human access.
- Use IAM roles for workloads, automation, and cross-account access.
- Avoid IAM users except for rare break-glass or legacy cases.
- Use short-lived sessions and MFA for privileged roles.
- Map corporate groups to permission sets rather than assigning users individually.

Example SCIM and federation flow:

1. Configure the external IdP as the source of truth.
2. Enable SCIM provisioning to sync users and groups.
3. Configure SAML federation or IAM Identity Center integration.
4. Assign groups to permission sets and target accounts.
5. Audit access through CloudTrail and Identity Center logs.

**Network architecture for a global enterprise**

A regulated enterprise usually centralizes network control while allowing regional workload autonomy. The common hub-and-spoke pattern uses Transit Gateway as the routing core, with separate VPCs for shared services, inspection, egress, and workloads.

Typical components:

- **Transit Gateway (TGW)**: central routing between VPCs, VPN, and Direct Connect.
- **Shared services VPC**: DNS resolvers, directory services, internal tooling.
- **Inspection VPC**: AWS Network Firewall, third-party appliances, or both.
- **Egress VPC**: centralized outbound internet access and NAT.
- **Workload VPCs**: application subnets with limited direct internet exposure.

For centralized inspection and egress, place AWS Network Firewall endpoints in dedicated inspection subnets in each AZ of the inspection VPC. Route traffic from workload VPCs to TGW, then to the inspection VPC, then to the egress VPC or internet gateway. Return traffic must follow symmetric routing where required by the firewall design.

A common routing pattern is:

- Workload subnet route table sends `0.0.0.0/0` to TGW.
- TGW route table sends internet-bound traffic to the inspection attachment.
- Inspection VPC route table sends inspected traffic to the egress attachment.
- Egress VPC route table sends traffic to NAT Gateway or internet gateway.
- Return routes point back through TGW and the firewall path.

Example conceptual route intent:

```text
Workload VPC -> TGW -> Network Firewall -> Egress VPC -> Internet
Internet -> Egress VPC -> Network Firewall -> TGW -> Workload VPC
```

Use AWS Network Firewall when you need stateful inspection, domain filtering, or managed rule groups at the network edge. Use third-party appliances when you need vendor-specific features, but remember to design for scaling, failover, and appliance insertion.

**Global ingress and edge protection**

Internet-facing regulated workloads usually need layered edge controls. AWS WAF protects against common web exploits and supports managed rule groups, rate-based rules, and custom logic. AWS Shield Standard is automatic, while Shield Advanced adds enhanced DDoS protection, cost protection, and response support.

Typical ingress stack:

- **Amazon CloudFront** for global caching and edge termination.
- **AWS WAF** attached to CloudFront, Application Load Balancer, or API Gateway.
- **AWS Shield Advanced** for high-value public endpoints.
- **AWS Certificate Manager (ACM)** for TLS certificates.
- **Route 53** for DNS routing and health checks.

Use CloudFront when you want global performance, origin shielding, and a single edge policy. Use Route 53 latency-based routing or geolocation routing when you need region-aware traffic steering. For regulated workloads, combine edge controls with origin restrictions so users cannot bypass the front door.

**Data residency and regional control**

Data residency is more than denying regions with SCPs. You must consider where data is created, replicated, backed up, cached, logged, and restored. Some AWS services are regional, while others are global or have global control planes that can complicate simple region-deny strategies.

Important controls include:

- Restrict account use to approved Regions with SCPs.
- Configure service-specific regional settings where available.
- Ensure S3 replication destinations stay within approved jurisdictions.
- Review global services such as IAM, Route 53, CloudFront, and Organizations for compliance impact.
- Control where logs and backups are stored, copied, and retained.

For Amazon S3, replication is explicit. You choose source and destination buckets, and cross-Region replication only occurs if configured. You should also consider:

- Bucket policies that prevent unauthorized cross-Region writes.
- Object Lock for immutable retention where required.
- KMS key strategy in each Region for encrypted replication.
- Lifecycle rules that do not accidentally move data to disallowed storage classes or Regions.

For services with global control planes, a region-deny SCP may not be sufficient by itself. You need architecture review, logging, and detective controls to ensure the service does not create compliance gaps.

**Shared services and resource sharing**

AWS Resource Access Manager (AWS RAM) lets you share resources across accounts and, in some cases, OUs. This is useful for shared networking and enterprise platform services.

Common RAM use cases:

- Share Transit Gateway with workload accounts.
- Share Route 53 Resolver rules.
- Share AWS License Manager configurations.
- Share subnets in a VPC with other accounts when appropriate.
- Share custom resources that support centralized governance.

RAM reduces duplication and helps you keep ownership in the platform account while allowing controlled consumption in workload accounts. It is especially useful in landing zone designs where the network team owns the hub and application teams own the spokes.

**Logging, monitoring, and audit evidence**

A regulated enterprise needs centralized, immutable, and queryable logs. CloudTrail records API activity, AWS Config records resource configuration changes, and service logs provide workload-specific evidence. Centralize these into a log archive account and protect them with least privilege and retention controls.

Core logging pattern:

- Organization CloudTrail with all Regions enabled.
- CloudTrail log file validation.
- AWS Config aggregators and organization conformance packs.
- VPC Flow Logs for network visibility.
- ALB, CloudFront, WAF, and application logs for edge and workload analysis.
- S3 Object Lock or restrictive retention for critical audit logs.

CloudTrail Lake can help with centralized log analytics and investigation. It is useful when you need SQL-like queries over event data without exporting everything to a separate analytics stack. For long-term audit workflows, pair CloudTrail Lake with S3-based archival and Athena or OpenSearch where appropriate.

Audit Manager helps collect evidence for control frameworks by pulling from sources such as CloudTrail, Config, Security Hub, and other AWS services. In practice, you map controls to evidence sources, then use automated collection to reduce manual screenshots and spreadsheet-driven audits.

Example evidence workflow:

1. Define the control framework in Audit Manager.
2. Enable evidence collection from CloudTrail and Config.
3. Review automated evidence for completeness.
4. Add manual evidence only where automation cannot prove the control.
5. Export reports for auditors and retain them in the log archive account.

**Backup and disaster recovery**

AWS Backup centralizes backup policy management across accounts and services. In regulated enterprises, you often use backup vaults in a dedicated backup account and copy backups across accounts and Regions for resilience and compliance.

Important AWS Backup capabilities:

- Central backup policies through Organizations.
- Cross-account backup copy.
- Cross-Region backup copy.
- Backup vault lock for immutability.
- Restore testing to validate recovery objectives.

Be aware of service support caveats. Not every resource type supports every backup feature equally, and some services require service-specific planning. Always verify whether the resource supports cross-account copy, cross-Region copy, point-in-time recovery, or restore into a different account.

A practical DR pattern:

- Use **active/passive** for many stateful enterprise systems when cost and complexity matter.
- Use **active/active** only when the application and data layer support conflict-free replication and global traffic management.
- Use Route 53 health checks and failover routing for simpler regional failover.
- Use application-level replication or database-native replication when you need low RPO and controlled consistency.

Comparison guidance:

| Pattern | Best for | Tradeoff |
|---|---|---|
| Active/passive | Most regulated enterprise workloads | Lower cost, slower failover |
| Active/active | Stateless or globally distributed systems | Higher complexity and data consistency challenges |
| Route 53 failover | Regional failover with health checks | DNS-based failover is not instant |
| App-level replication | Databases and stateful services | Requires application-aware design |

For databases, choose the replication model based on consistency and recovery goals. For example, Aurora Global Database can support low-latency cross-Region replication for supported use cases, while other databases may rely on asynchronous replication or backup/restore.

> Do not assume backup equals disaster recovery. Backups protect data; DR restores service continuity.

**Compliance, governance, and detective controls**

Regulated enterprises need preventive, detective, and corrective controls. Preventive controls include SCPs, permission boundaries, and network segmentation. Detective controls include Config rules, Security Hub findings, GuardDuty detections, and CloudTrail alerts. Corrective controls often use automation through Lambda, Systems Manager, or EventBridge.

Useful patterns:

- Use Security Hub as the central findings dashboard.
- Use GuardDuty for threat detection across accounts and Regions.
- Use Config for drift detection and compliance evaluation.
- Use EventBridge to route findings to ticketing or remediation workflows.
- Use Systems Manager Automation for standardized remediation steps.

Example remediation trigger flow:

1. GuardDuty detects suspicious activity.
2. Security Hub aggregates the finding.
3. EventBridge matches the finding pattern.
4. Lambda or Systems Manager runs a containment playbook.
5. The security team receives a ticket and audit trail.

**Hybrid connectivity**

Many global enterprises keep hybrid connectivity for legacy systems, data center dependencies, or regulatory constraints. Common options include Site-to-Site VPN, AWS Direct Connect, and Direct Connect Gateway.

Design considerations:

- Use Direct Connect for predictable throughput and private connectivity.
- Use VPN as backup or for smaller sites.
- Use redundant connections and diverse locations for resilience.
- Terminate hybrid connectivity in a network account and propagate routes carefully through TGW.
- Apply inspection and segmentation before traffic reaches sensitive workloads.

For regulated environments, hybrid connectivity often feeds centralized inspection, DNS resolution, and identity services. Keep routing explicit so you can prove which paths data can take.

**Practical enterprise patterns**

A few patterns appear repeatedly in exam scenarios:

- **Centralized security account**: aggregate GuardDuty, Security Hub, Config, and Audit Manager evidence.
- **Centralized network account**: own TGW, inspection, and egress.
- **Shared services account**: host directory connectors, DNS, and internal platform tooling.
- **Workload isolation**: separate accounts per application or environment to reduce blast radius.
- **Regional autonomy**: deploy workloads in approved Regions with local data stores when residency matters.

Example high-level topology:

```text
[Users]
   |
[CloudFront + WAF + Shield Advanced]
   |
[Regional ALB / API Gateway]
   |
[TGW Hub] ---- [Inspection VPC / Network Firewall] ---- [Egress VPC]
   |
[Workload VPCs in separate accounts]
   |
[Shared services / logging / security accounts]
```

**CLI and configuration examples**

Enable organization CloudTrail and central logging concepts with a trail that writes to a dedicated bucket:

```bash
aws cloudtrail create-trail \
  --name org-trail \
  --s3-bucket-name org-log-archive-bucket \
  --is-multi-region-trail

aws cloudtrail start-logging --name org-trail
```

Create a simple AWS Config recorder and delivery channel in a member account:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder name=default,roleARN=arn:aws:iam::111122223333:role/AWSConfigRole \
  --recording-group allSupported=true,includeGlobalResourceTypes=true

aws configservice put-delivery-channel \
  --delivery-channel name=default,s3BucketName=org-config-bucket
```

**Exam decision points**

When you see a regulated global enterprise scenario, ask these questions:

- Do you need one account or many accounts?
- Which services should be centrally governed through delegated administration?
- Is Control Tower enough, or do you need AFT and custom extensions?
- Where does identity come from, and how are users provisioned?
- How do you centralize logs, evidence, and findings?
- Which Regions are allowed, and how do you enforce residency?
- Do you need centralized inspection, egress, or both?
- Is the DR model active/passive, active/active, or backup/restore?
- Which services support the required backup, replication, or sharing pattern?

The best answer usually favors managed services, centralized governance, and explicit exception handling over ad hoc per-account design. In SAP-C02, the winning architecture is the one that scales operationally while still satisfying compliance, security, and recovery requirements.

**Key takeaways**

- Use AWS Organizations and OUs to define the governance boundary.
- Use delegated administration for centralized security and compliance services.
- Extend Control Tower when enterprise identity, networking, or automation needs exceed the baseline.
- Use IAM Identity Center with federation and SCIM for workforce access.
- Centralize inspection with TGW and AWS Network Firewall when required.
- Protect internet-facing workloads with CloudFront, WAF, and Shield Advanced.
- Treat data residency as a service-by-service design problem, not just an SCP problem.
- Use AWS RAM for controlled cross-account sharing.
- Centralize logs, evidence, and analytics with CloudTrail, Config, CloudTrail Lake, and Audit Manager.
- Validate backup and DR with cross-account copies and restore testing.
