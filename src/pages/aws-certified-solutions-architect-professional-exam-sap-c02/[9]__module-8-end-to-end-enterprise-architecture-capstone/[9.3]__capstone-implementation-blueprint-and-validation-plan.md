## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Capstone purpose and scope**

A capstone implementation blueprint turns a target architecture into an executable plan. For SAP-C02, you are expected to reason about tradeoffs across availability, security, cost, performance, and operational excellence, then validate that the design works under realistic conditions. The blueprint should connect business requirements to AWS services, deployment sequencing, rollback strategy, and measurable acceptance criteria.

A strong capstone plan answers four questions:

- What are you building and why?
- How will you deploy it safely?
- How will you prove it works?
- How will you operate and improve it after launch?

**Blueprint structure**

A practical blueprint usually includes:

- **Requirements summary**
  - Business goals, constraints, and nonfunctional requirements.
  - Recovery objectives such as <abbr title="Recovery Time Objective">RTO</abbr> and <abbr title="Recovery Point Objective">RPO</abbr>.
  - Compliance, data residency, and encryption requirements.

- **Target architecture**
  - Accounts, Regions, and Availability Zones.
  - Network layout, ingress and egress paths, and segmentation.
  - Compute, storage, database, messaging, and identity components.
  - Observability, automation, and governance services.

- **Implementation phases**
  - Foundation and landing zone.
  - Network and identity setup.
  - Data migration and application deployment.
  - Cutover, validation, and stabilization.

- **Validation plan**
  - Functional tests, resilience tests, security checks, and operational checks.
  - Success criteria and rollback triggers.
  - Monitoring and alerting verification.

**Translate requirements into design decisions**

Start by mapping each requirement to a concrete AWS control or service. This keeps the design defensible and makes validation easier.

- High availability requirement may lead to:
  - Multi-AZ deployment for databases and application tiers.
  - Elastic Load Balancing across subnets.
  - Auto Scaling for stateless compute.

- Disaster recovery requirement may lead to:
  - Cross-Region replication for critical data.
  - Backup and restore or pilot light patterns.
  - Route 53 health checks and failover routing.

- Security requirement may lead to:
  - AWS Identity and Access Management <abbr title="Identity and Access Management">IAM</abbr> least privilege.
  - AWS Key Management Service <abbr title="Key Management Service">KMS</abbr> encryption.
  - Private subnets, VPC endpoints, and centralized logging.

- Cost control requirement may lead to:
  - Right-sizing, Savings Plans, and managed services.
  - Lifecycle policies for storage.
  - Event-driven scaling instead of always-on capacity.

**Implementation sequencing**

Sequence matters because many failures come from dependencies being built in the wrong order. Build the foundation first, then layer application components on top.

1. Establish governance and account structure.
   - Create or verify the AWS Organizations structure.
   - Apply service control policies where needed.
   - Set up logging, security, and shared services accounts.

2. Build the network foundation.
   - Create the Virtual Private Cloud <abbr title="Virtual Private Cloud">VPC</abbr>, subnets, route tables, gateways, and endpoints.
   - Design for multi-AZ placement.
   - Validate DNS, routing, and security group boundaries.

3. Configure identity and access.
   - Define roles, permission boundaries, and federation.
   - Use IAM roles for workloads instead of long-lived credentials.
   - Integrate with AWS IAM Identity Center if centralized access is required.

4. Deploy shared platform services.
   - Logging with CloudTrail, AWS Config, and centralized log storage.
   - Monitoring with Amazon CloudWatch.
   - Secrets management with AWS Secrets Manager or Systems Manager Parameter Store.

5. Migrate or deploy data layers.
   - Choose the migration method based on downtime tolerance.
   - Validate schema, replication, and backup settings.
   - Confirm encryption and access controls.

6. Deploy application workloads.
   - Use infrastructure as code for repeatability.
   - Roll out in stages with canary, blue/green, or phased deployment.
   - Keep rollback artifacts ready.

7. Cut over and stabilize.
   - Switch traffic using DNS, load balancer weights, or application routing.
   - Monitor error rates, latency, and saturation.
   - Freeze nonessential changes during the stabilization window.

**Infrastructure as code and repeatability**

Use infrastructure as code <abbr title="Infrastructure as Code">IaC</abbr> to make the blueprint reproducible and auditable. CloudFormation, AWS CDK, or Terraform can express the target state and reduce drift.

Example CloudFormation snippet for a basic multi-AZ application pattern:

```yaml
Resources:
  AppSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Allow app traffic from ALB
      VpcId: vpc-12345678
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 8080
          ToPort: 8080
          SourceSecurityGroupId: sg-alb123456

  AppAutoScalingGroup:
    Type: AWS::AutoScaling::AutoScalingGroup
    Properties:
      VPCZoneIdentifier:
        - subnet-a
        - subnet-b
      MinSize: 2
      MaxSize: 6
      DesiredCapacity: 2
      LaunchTemplate:
        LaunchTemplateId: lt-12345678
        Version: !GetAtt AppLaunchTemplate.LatestVersionNumber
```

Useful CLI checks during implementation:

```bash
aws cloudformation validate-template --template-body file://template.yaml
aws ec2 describe-vpcs
aws ec2 describe-subnets --filters Name=vpc-id,Values=vpc-12345678
aws iam get-role --role-name AppExecutionRole
aws autoscaling describe-auto-scaling-groups
```

**Validation plan design**

Validation is not a single test. It is a set of checks that prove the architecture meets requirements in normal, degraded, and failure conditions.

A complete validation plan should include:

- **Functional validation**
  - Confirm the application serves expected requests.
  - Verify authentication, authorization, and data access.
  - Test integrations with queues, topics, APIs, and databases.

- **Performance validation**
  - Measure latency, throughput, and concurrency.
  - Load test at expected and peak traffic levels.
  - Confirm scaling behavior and bottleneck removal.

- **Resilience validation**
  - Simulate instance failure, AZ impairment, and dependency failure.
  - Verify failover, retry, and backpressure behavior.
  - Confirm recovery objectives are met.

- **Security validation**
  - Check encryption in transit and at rest.
  - Validate least privilege and network segmentation.
  - Review logs, alerts, and detective controls.

- **Operational validation**
  - Confirm alarms, dashboards, runbooks, and on-call procedures.
  - Test backup restore and patching workflows.
  - Verify deployment rollback and change management.

**Validation methods and examples**

Use the right test for the risk you are trying to reduce.

- **Smoke tests**
  - Quick checks after deployment.
  - Confirm the service starts, responds, and reaches dependencies.

- **Integration tests**
  - Validate interactions between components.
  - Useful after database migrations or API changes.

- **Load tests**
  - Confirm capacity and scaling.
  - Use tools such as `k6`, `JMeter`, or `Locust`.

- **Chaos or fault injection**
  - Validate resilience under failure.
  - Use AWS Fault Injection Simulator where appropriate.

- **Backup restore tests**
  - Prove that backups are usable, not just enabled.
  - Measure restore time against RTO.

Example load test command with `k6`:

```bash
k6 run --vus 100 --duration 5m script.js
```

Example health check and failover verification:

```bash
curl -f https://app.example.com/health
aws route53 test-dns-answer --hosted-zone-id Z123456789 --record-name app.example.com --record-type A
```

**Deployment patterns and rollback**

Choose a deployment pattern that matches blast radius and rollback needs.

- **Blue/green**
  - Deploy the new version alongside the old version.
  - Shift traffic after validation.
  - Best when you need fast rollback and minimal downtime.

- **Canary**
  - Send a small percentage of traffic to the new version.
  - Expand only if metrics remain healthy.
  - Best when you want production validation with limited exposure.

- **Rolling**
  - Replace instances gradually.
  - Simpler, but rollback can be slower if state changes are involved.

Rollback should be explicit and rehearsed.

- Keep previous artifacts and AMIs, container images, or Lambda versions.
- Preserve database migration rollback steps when possible.
- Define the exact metric thresholds that trigger rollback.
- Document who approves rollback and how traffic is restored.

> If a change includes an irreversible data migration, treat rollback as a forward-fix plan and validate restore procedures before cutover.

**Data migration and cutover planning**

Data movement is often the highest-risk part of the capstone. Choose the migration approach based on size, downtime, and consistency requirements.

- **AWS Database Migration Service <abbr title="Database Migration Service">DMS</abbr>**
  - Useful for ongoing replication and low-downtime cutover.
  - Validate source and target compatibility.
  - Monitor replication lag closely.

- **AWS DataSync**
  - Useful for file and object transfers at scale.
  - Validate permissions, throughput, and checksum integrity.

- **Snapshot and restore**
  - Useful for simpler migrations or disaster recovery.
  - Validate restore time and application compatibility.

Cutover checklist:

1. Freeze schema changes if required.
2. Confirm replication lag is within tolerance.
3. Run final consistency checks.
4. Switch traffic using DNS, load balancer, or application config.
5. Monitor error rate, latency, and business KPIs.
6. Keep the old environment available until the stabilization window ends.

**Observability and operational readiness**

Observability is part of validation, not an afterthought. You need enough telemetry to know whether the system is healthy and why it is not.

- **Metrics**
  - Track latency, error rate, request volume, CPU, memory, and queue depth.
  - Create alarms on symptoms, not only on infrastructure.

- **Logs**
  - Centralize application, access, and audit logs.
  - Use structured logging where possible.
  - Retain logs according to compliance needs.

- **Traces**
  - Use distributed tracing to follow requests across services.
  - Helpful for microservices and asynchronous workflows.

- **Dashboards**
  - Build a small set of operational dashboards.
  - Include business and technical indicators.

Example CloudWatch alarm concept:

```json
{
  "AlarmName": "High-5XX-Rate",
  "MetricName": "5XXError",
  "Namespace": "AWS/ApplicationELB",
  "Statistic": "Sum",
  "Period": 60,
  "EvaluationPeriods": 3,
  "Threshold": 10,
  "ComparisonOperator": "GreaterThanThreshold"
}
```

**Security and compliance validation**

Security validation should prove that controls are effective, not just configured.

- Verify encryption at rest for EBS, S3, RDS, and backups.
- Verify TLS for client and service-to-service traffic.
- Confirm IAM policies follow least privilege.
- Check that public exposure is intentional and documented.
- Validate audit trails with CloudTrail and AWS Config.
- Review key rotation and secret rotation processes.

Practical checks:

```bash
aws s3api get-bucket-encryption --bucket my-secure-bucket
aws kms describe-key --key-id alias/app-key
aws cloudtrail describe-trails
aws configservice describe-configuration-recorders
```

**Acceptance criteria**

Acceptance criteria make the blueprint testable. They should be measurable and tied to requirements.

Good criteria are specific:

- Application responds to `GET /health` with HTTP 200 within 2 seconds.
- Primary database failover completes within 60 seconds.
- RPO does not exceed 15 minutes for critical data.
- All production logs are centralized and retained for 365 days.
- No public subnet contains a workload that must remain private.

Avoid vague criteria such as:

- System is fast.
- Security is strong.
- Failover works.

**Common SAP-C02 design pitfalls**

- Designing for one Region when the requirement implies regional resilience.
- Using manual steps where automation is needed for repeatability.
- Ignoring data layer dependencies during failover planning.
- Treating backups as equivalent to tested restores.
- Overlooking DNS propagation and client caching during cutover.
- Failing to validate IAM permissions in the target environment.
- Not defining rollback for schema changes.
- Measuring only infrastructure health and missing application-level signals.

**How to present the capstone in an exam answer**

When you write or evaluate a capstone response, structure it like an implementation plan.

- State the requirement and constraint first.
- Name the AWS services and the reason for each choice.
- Explain deployment order and dependencies.
- Include validation steps and success metrics.
- Mention rollback and operational monitoring.
- Call out tradeoffs where relevant.

A concise answer often follows this pattern:

1. Requirement and risk.
2. Proposed architecture.
3. Deployment sequence.
4. Validation and monitoring.
5. Rollback and operational handoff.

**Key takeaways**

A capstone implementation blueprint is a disciplined way to turn architecture into delivery. For SAP-C02, the strongest answers show that you can design for resilience, deploy in the right order, validate with evidence, and operate the system safely after launch. The validation plan is just as important as the architecture because it proves the design meets the business requirement under real conditions.
