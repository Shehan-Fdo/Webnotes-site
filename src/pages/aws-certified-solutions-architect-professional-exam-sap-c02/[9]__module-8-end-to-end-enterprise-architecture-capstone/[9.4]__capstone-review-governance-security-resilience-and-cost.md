## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Capstone framing**

SAP-C02 scenarios usually combine multiple constraints: organizational governance, enterprise security, recovery objectives, and cost pressure. Your job is not to pick the “best” service in isolation, but to choose the design that satisfies the stated requirements with the fewest operational risks.

A strong answer usually shows:
- **Control plane governance** for accounts, permissions, and guardrails.
- **Identity and data protection** for users, workloads, and secrets.
- **Resilience design** aligned to RTO (Recovery Time Objective) and RPO (Recovery Point Objective).
- **Cost controls** that preserve performance and availability.

**Governance and landing zone design**

Governance starts with account structure. Use AWS Organizations to separate blast radius, billing, and policy boundaries. In enterprise designs, you typically place shared services, security tooling, logging, and workload accounts into distinct Organizational Units (OUs) so you can apply different controls at the OU level.

Key account and OU strategy nuances:
- **Management account** should stay minimal and reserved for organization administration.
- **Security OU** often contains audit, log archive, and security tooling accounts.
- **Infrastructure or shared services OU** can host networking, directory, and centralized services.
- **Workload OUs** should separate production, non-production, and regulated workloads.
- **Sandbox OU** is useful for experimentation with tighter spend controls.

Use Service Control Policies (SCPs) to set maximum permissions. SCPs do not grant access; they only limit what identities in member accounts can do. This distinction is common in exam questions.

Example SCP that denies disabling CloudTrail and Config:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyDisablingSecurityServices",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail",
        "config:DeleteConfigurationRecorder",
        "config:StopConfigurationRecorder"
      ],
      "Resource": "*"
    }
  ]
}
```

Governance controls you should recognize:
- **AWS Control Tower** for landing zone automation, account vending, and preventive/detective guardrails.
- **AWS Service Catalog** for approved product portfolios and standardized provisioning.
- **AWS Config** for configuration drift detection and compliance evaluation.
- **Tag policies** for enforcing consistent tag keys and values across accounts.
- **AWS Organizations backup policies** for centralized backup intent in multi-account environments.

Control Tower guardrails are often the fastest answer when the requirement is “prevent users from doing unsafe things across many accounts.” Use Service Catalog when the requirement is “let teams provision only approved infrastructure patterns.”

A practical governance workflow often looks like this:
1. Create accounts through Control Tower or a controlled account vending process.
2. Apply SCPs at the OU level to block disallowed actions.
3. Enforce tag policies for cost allocation and automation.
4. Use Config rules to detect drift and trigger remediation.
5. Route findings to a central security account for review and response.

Example AWS CLI to create a Service Catalog portfolio association:
```bash
aws servicecatalog associate-principal-with-portfolio \
  --portfolio-id port-abc123 \
  --principal-arn arn:aws:iam::123456789012:role/PlatformAdmin \
  --principal-type IAM
```

**Identity, federation, and access control**

Enterprise access design usually centers on AWS IAM Identity Center, formerly AWS Single Sign-On (SSO). Identity Center is the preferred pattern for workforce access because it centralizes user and group management, supports federation, and simplifies permission set assignment across accounts.

Use Identity Center when:
- You need centralized workforce authentication.
- You want to assign role-based access across many AWS accounts.
- You need to integrate with an external identity provider such as Microsoft Entra ID or Okta.

Use federation patterns when:
- Users already authenticate in an enterprise IdP.
- You need short-lived credentials instead of long-term IAM users.
- You want to reduce credential sprawl.

Important access-control concepts:
- **MFA** should be required for privileged access and sensitive operations.
- **Session controls** should limit duration, enforce reauthentication, and reduce standing privilege.
- **Permission sets** in Identity Center map to IAM roles in target accounts.
- **Role chaining** can complicate session duration and should be avoided when possible for human access.
- **IAM users** are generally discouraged for workforce access; prefer federation and Identity Center.

Example trust policy for SAML federation:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:saml-provider/CorpIdP"
      },
      "Action": "sts:AssumeRoleWithSAML",
      "Condition": {
        "StringEquals": {
          "SAML:aud": "https://signin.aws.amazon.com/saml"
        }
      }
    }
  ]
}
```

For machine access, prefer IAM roles with short-lived credentials from EC2, ECS, Lambda, or STS (AWS Security Token Service). Avoid embedding static access keys in code or AMIs.

**Security architecture and data protection**

Security design in SAP-C02 often spans identity, network, application, and data layers. The exam expects you to choose the right control at the right layer rather than overusing one service.

Core security services and patterns:
- **AWS WAF** protects web applications from common exploits and supports managed rules, rate limiting, and custom rules.
- **AWS Shield** provides DDoS protection; Shield Standard is automatic, while Shield Advanced adds enhanced detection and response.
- **AWS Network Firewall** provides stateful inspection and centralized network controls for VPC traffic.
- **AWS Secrets Manager** stores, rotates, and retrieves secrets such as database credentials and API keys.
- **AWS Systems Manager Parameter Store** is useful for configuration values and can store secure strings, often at lower cost than Secrets Manager.
- **Amazon GuardDuty** detects threats using logs and telemetry.
- **AWS Security Hub** aggregates findings and compliance signals.
- **Amazon Inspector** helps identify vulnerabilities in EC2, container images, and Lambda functions.
- **AWS Macie** helps discover and classify sensitive data in S3.

Choose the right secret store:
- Use **Secrets Manager** when you need rotation, secret lifecycle management, or database credential automation.
- Use **Parameter Store** when you need configuration values, hierarchical parameters, or lower-cost secure string storage without built-in rotation.

Example retrieving a secret at runtime:
```bash
aws secretsmanager get-secret-value \
  --secret-id prod/db/master \
  --query SecretString \
  --output text
```

Network security patterns:
- Put **WAF** in front of CloudFront, ALB, or API Gateway for HTTP/S protection.
- Use **Network Firewall** for centralized egress filtering, domain-based controls, and inspection between subnets or VPCs.
- Use security groups for stateful instance-level control and NACLs for coarse subnet-level filtering.
- Prefer private connectivity with VPC endpoints for AWS service access when internet exposure is unnecessary.

> Use WAF for application-layer threats, Shield for DDoS, and Network Firewall for network-layer inspection. They solve different problems and are not interchangeable.

**Encryption and key management**

KMS (AWS Key Management Service) is central to exam scenarios involving encryption at rest. Understand the difference between encryption at rest, in transit, and client-side encryption.

- **Encryption at rest** protects stored data in services like S3, EBS, RDS, and DynamoDB.
- **Encryption in transit** protects data moving over networks using TLS (Transport Layer Security).
- **Client-side encryption** encrypts data before it reaches AWS, which is useful when you need to control the encryption boundary yourself.

KMS key types and behaviors:
- **AWS managed keys** are service-specific and require minimal administration.
- **Customer managed keys** give you control over policy, rotation, and grants.
- **Multi-Region keys** are useful for active/active or disaster recovery designs that need the same logical key material in multiple Regions.
- **Automatic key rotation** is available for customer managed symmetric KMS keys; rotation changes backing key material while preserving the key ID and aliases.
- **Asymmetric keys** are used for signing and verification or encryption use cases that require public/private key pairs.

Multi-Region KMS keys are especially relevant when you replicate encrypted data across Regions and want consistent key identifiers and simpler failover. They do not automatically replicate your encrypted data; they only help with key availability and regional continuity.

Example S3 bucket encryption configuration:
```json
{
  "Rules": [
    {
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "aws:kms",
        "KMSMasterKeyID": "arn:aws:kms:us-east-1:123456789012:key/abcd-1234"
      }
    }
  ]
}
```

Common exam distinctions:
- Use **SSE-S3** when AWS-managed encryption is sufficient and no key control is needed.
- Use **SSE-KMS** when auditability, access control, or key policy control matters.
- Use **client-side encryption** when the application must control plaintext exposure.
- Use **TLS** for data in transit even if the payload is also encrypted at rest.

**Resilience and disaster recovery**

Resilience design should start with RTO and RPO:
- **RTO** is how long you can tolerate service downtime.
- **RPO** is how much data loss you can tolerate.

These objectives drive whether you choose backup-and-restore, pilot light, warm standby, active/passive, or active/active.

Typical mapping:
- **Backup and restore**: lowest cost, highest RTO/RPO.
- **Pilot light**: core components always on, faster recovery.
- **Warm standby**: scaled-down but functional secondary environment.
- **Active/passive**: primary serves traffic, secondary is ready to take over.
- **Active/active**: both sites serve traffic, lowest RTO, usually highest complexity.

Service-specific replication options matter:
- **Aurora Global Database** supports low-latency cross-Region replication and fast failover for Aurora workloads.
- **DynamoDB Global Tables** provide multi-Region active/active replication for DynamoDB.
- **S3 Cross-Region Replication (CRR)** replicates objects asynchronously between buckets.
- **EFS replication** can help with file-system continuity, though recovery design still depends on application behavior.
- **RDS cross-Region read replicas** can support DR, but failover and promotion are more manual than Aurora Global Database.

Route 53 failover is useful, but it has limitations:
- DNS-based failover is not instantaneous because clients and resolvers cache records.
- **TTL** (Time to Live) affects how quickly clients observe changes.
- Health checks can detect endpoint failure, but they do not move existing sessions.
- Stateful applications need session design that survives endpoint changes.

Design implications:
- Store session state in a shared tier such as ElastiCache, DynamoDB, or a database rather than in instance memory.
- Use stateless application tiers behind load balancers when possible.
- Replicate data tier state according to the RPO target.
- Lower TTL values can improve failover responsiveness, but they increase DNS query volume.

Example Route 53 failover record set concept:
```text
Primary record -> ALB in us-east-1
Secondary record -> ALB in us-west-2
Health check -> primary ALB endpoint
TTL -> 60 seconds
```

> DNS failover does not preserve in-memory sessions. If the app cannot tolerate session loss, you need shared session storage or a different architecture.

**Backup strategy and restore validation**

Backups are not complete until you test restores. SAP-C02 often expects you to distinguish between backup creation, backup retention, and restore verification.

AWS Backup can centralize backup policies across supported services and accounts. It is useful for:
- Standardizing backup schedules and retention.
- Applying backup plans across multiple accounts.
- Centralizing audit and compliance reporting.

Important backup concepts:
- **AWS Backup Audit Manager** helps evaluate whether backup plans meet organizational controls.
- **Restore testing** validates that backups are usable, not just present.
- **Lifecycle policies** can move backups to cold storage where supported.
- **Service-native backups** may still be preferable for some workloads when they provide tighter integration or better recovery semantics.

Service-native alternatives and examples:
- **RDS snapshots** and automated backups for relational databases.
- **DynamoDB on-demand backups** and point-in-time recovery.
- **EBS snapshots** for block storage.
- **S3 versioning** and CRR for object durability and recovery.
- **EFS backups** through AWS Backup or native mechanisms depending on the design.

A practical restore-validation workflow:
1. Define the recovery point and recovery procedure for each critical workload.
2. Schedule periodic restore tests into an isolated account or VPC.
3. Verify application startup, data integrity, and permissions.
4. Record actual restore time and compare it to the target RTO.
5. Update runbooks and backup policies based on findings.

Example AWS CLI to start a backup job:
```bash
aws backup start-backup-job \
  --backup-vault-name CentralVault \
  --resource-arn arn:aws:rds:us-east-1:123456789012:db:prod-db \
  --iam-role-arn arn:aws:iam::123456789012:role/AWSBackupDefaultServiceRole
```

**Cost optimization and trade-offs**

Cost optimization in SAP-C02 is rarely about choosing the cheapest service. It is about matching spend to business value while preserving required performance and resilience.

Storage and data cost topics that appear frequently:
- **S3 Intelligent-Tiering** automatically moves objects between access tiers based on usage patterns.
- **S3 Storage Class Analysis** helps identify when lifecycle transitions make sense.
- **EBS gp3** is often cheaper and more flexible than gp2 because you can provision IOPS and throughput independently.
- **EBS io1/io2** are for high-performance workloads that need guaranteed IOPS, but they cost more.
- **Data transfer costs** can dominate architectures that move data across AZs, Regions, or out to the internet.

When to use each storage option:
- Use **gp3** for general-purpose workloads that need predictable performance at lower cost.
- Use **io1/io2** for latency-sensitive or IOPS-intensive databases and transactional systems.
- Use **Intelligent-Tiering** for unknown or changing access patterns.
- Use **Glacier classes** for archival data with infrequent retrieval.

Example lifecycle policy for S3:
```json
{
  "Rules": [
    {
      "ID": "ArchiveOldObjects",
      "Status": "Enabled",
      "Filter": {
        "Prefix": ""
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER"
        }
      ]
    }
  ]
}
```

Cost-aware architecture choices:
- Prefer **managed services** when they reduce operational overhead enough to justify the premium.
- Use **multi-AZ** where availability requirements justify the extra cost.
- Use **cross-Region replication** only when the business requirement needs it.
- Reduce **data transfer** by keeping chatty components in the same AZ or Region when possible.
- Use **VPC endpoints** to avoid NAT Gateway data processing charges for AWS service access.

A common exam trade-off is gp3 versus io2:
- Choose **gp3** when you need lower cost and moderate performance.
- Choose **io2** when the workload requires sustained high IOPS, low latency, and stronger durability characteristics for mission-critical databases.

**Putting it together in scenario design**

The best SAP-C02 answer usually aligns all four dimensions:
- Governance defines who can deploy and what they can deploy.
- Security defines how identities, secrets, and data are protected.
- Resilience defines how the system behaves during failure.
- Cost defines how much redundancy and performance the business can afford.

Example enterprise pattern:
- Use Control Tower to create accounts and apply baseline guardrails.
- Use SCPs and tag policies to enforce organizational standards.
- Use Identity Center with MFA and federation for workforce access.
- Use WAF, Shield, and Network Firewall for layered protection.
- Store secrets in Secrets Manager and configuration in Parameter Store.
- Encrypt data with KMS, using multi-Region keys where DR requires it.
- Replicate critical data with Aurora Global Database, DynamoDB Global Tables, or S3 CRR as appropriate.
- Validate backups with AWS Backup Audit Manager and restore tests.
- Optimize storage with Intelligent-Tiering, lifecycle policies, and the right EBS volume type.

**Exam decision cues**

Use these cues to narrow answers quickly:
- If the question says **prevent**, think SCPs, Control Tower guardrails, WAF rules, or Network Firewall policies.
- If it says **standardize provisioning**, think Service Catalog.
- If it says **centralize workforce login**, think IAM Identity Center and federation.
- If it says **rotate secrets**, think Secrets Manager.
- If it says **low RTO and low RPO**, think active/active or warm standby with replicated data.
- If it says **backup compliance**, think AWS Backup plus Audit Manager and restore testing.
- If it says **reduce storage cost**, think Intelligent-Tiering, lifecycle transitions, or gp3.
- If it says **minimize data transfer**, think architecture locality, endpoints, and replication scope.

**Quick comparison table**

| Requirement | Preferred pattern | Why |
|---|---|---|
| Workforce SSO | IAM Identity Center | Centralized access and account assignment |
| Approved provisioning | Service Catalog | Limits teams to vetted products |
| Org-wide guardrails | SCPs and Control Tower | Preventive controls at scale |
| Secret rotation | Secrets Manager | Built-in rotation and lifecycle |
| Config values | Parameter Store | Lower-cost parameter storage |
| Web DDoS protection | Shield + WAF | Network and application-layer defense |
| Network inspection | Network Firewall | Stateful traffic filtering |
| Fast DR for Aurora | Aurora Global Database | Low-latency cross-Region replication |
| Multi-Region NoSQL | DynamoDB Global Tables | Active/active replication |
| Object replication | S3 CRR | Cross-Region object copies |
| Backup governance | AWS Backup + Audit Manager | Central policy and compliance checks |
| Lower storage cost | Intelligent-Tiering / lifecycle | Automatic tiering and archival |

**Final review mindset**

When you read a scenario, identify the dominant constraint first. Then choose the smallest set of AWS services that satisfy it without creating unnecessary operational complexity. SAP-C02 rewards designs that are secure by default, governable at scale, resilient to the stated failure mode, and cost-aware without being underbuilt.
