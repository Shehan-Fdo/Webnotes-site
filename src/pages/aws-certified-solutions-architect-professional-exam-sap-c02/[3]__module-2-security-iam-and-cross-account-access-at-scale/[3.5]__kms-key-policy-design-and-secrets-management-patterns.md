## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**KMS and secrets management: the exam lens**

AWS Key Management Service (KMS) is the control plane for encryption keys, while AWS Secrets Manager is the managed secret lifecycle service that commonly uses KMS for encryption at rest. In SAP-C02 scenarios, the hard part is usually not “how do I encrypt data?” but “which policy must allow which principal, in which account, for which action, under which condition?”

A strong design separates:
- **Key policy** authorization for the KMS key itself
- **IAM policy** authorization for identities in the account
- **Resource policy** authorization for the secret, when sharing or cross-account access is required
- **Service integration** behavior, such as AWS services creating grants on your behalf

**KMS authorization model**

KMS evaluates multiple layers of authorization. A request succeeds only when the caller is allowed by the KMS key policy and, where applicable, by IAM policy or a grant. For many exam questions, the key distinction is whether you need to modify the key policy, create a grant, or attach an IAM policy.

<dl>
<dt><strong>Key policy</strong></dt>
<dd>The primary policy attached to a KMS key. It controls who can administer and use the key.</dd>
<dt><strong>IAM policy</strong></dt>
<dd>Identity-based permissions for users, roles, and groups. For customer managed keys, IAM permissions work only when the key policy delegates to IAM.</dd>
<dt><strong>Grant</strong></dt>
<dd>A lightweight permission mechanism for temporary or service-specific access to a KMS key.</dd>
</dl>

The standard default key policy includes the root delegation pattern, often called **Enable IAM User Permissions**. This statement allows the account to use IAM policies with the key. Without it, IAM permissions alone do not grant access to a customer managed KMS key.

```json
{
  "Sid": "EnableIAMUserPermissions",
  "Effect": "Allow",
  "Principal": { "AWS": "arn:aws:iam::123456789012:root" },
  "Action": "kms:*",
  "Resource": "*"
}
```

That statement does not mean every principal gets access. It means the account can delegate access through IAM policies. In practice, you still attach least-privilege IAM permissions to the specific role or user.

**Key policy design patterns**

Use key policies to define the trust boundary for the key. For exam scenarios, think in terms of:
- Who administers the key
- Who can use the key
- Which AWS services may use the key on your behalf
- Which conditions must be true for access

A common administration pattern is:
- Account root or a security admin role can manage the key
- Application roles can use the key for specific workloads
- AWS services can use the key only through service-specific conditions

Example key policy fragment for a customer managed key:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableIAMUserPermissions",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::123456789012:root" },
      "Action": "kms:*",
      "Resource": "*"
    },
    {
      "Sid": "AllowKeyAdministration",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::123456789012:role/SecurityAdmin" },
      "Action": [
        "kms:Create*",
        "kms:Describe*",
        "kms:Enable*",
        "kms:Disable*",
        "kms:Put*",
        "kms:Update*",
        "kms:Revoke*",
        "kms:ScheduleKeyDeletion",
        "kms:CancelKeyDeletion",
        "kms:TagResource",
        "kms:UntagResource"
      ],
      "Resource": "*"
    }
  ]
}
```

Use conditions to narrow usage:
- `kms:ViaService` restricts use to a specific AWS service endpoint, such as `secretsmanager.us-east-1.amazonaws.com`
- `kms:CallerAccount` restricts the account making the request
- `kms:EncryptionContext:*` restricts use to matching encryption context values
- `kms:GrantIsForAWSResource` helps ensure a grant is created for an AWS resource, not a human principal
- `kms:MultiRegion` can distinguish multi-Region key behavior in policy logic

Example condition-based usage restriction:

```json
{
  "Sid": "AllowSecretsManagerUseOnly",
  "Effect": "Allow",
  "Principal": { "Service": "secretsmanager.amazonaws.com" },
  "Action": [
    "kms:Encrypt",
    "kms:Decrypt",
    "kms:GenerateDataKey",
    "kms:DescribeKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:ViaService": "secretsmanager.us-east-1.amazonaws.com",
      "kms:CallerAccount": "123456789012"
    }
  }
}
```

**Grants versus key policies**

A **grant** is not a policy replacement; it is a scoped permission record attached to a KMS key. AWS services often create grants automatically when they need to use a key on your behalf. This is a common exam distinction.

Use grants when:
- A service needs temporary or operational access to a key
- You want to avoid editing the key policy for every short-lived use case
- The service creates and manages the grant lifecycle

Use key policy or IAM policy when:
- You need durable, explicit authorization
- You are controlling human or application access
- You need cross-account or cross-service trust boundaries to be clear

A grant is created by `CreateGrant`, and the API returns a **grant token**. The caller can use that token immediately in subsequent KMS requests to avoid waiting for eventual consistency. The token is not a parameter used to create the grant; it is a way to make the newly created grant usable right away.

```bash
aws kms create-grant \
  --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234 \
  --grantee-principal arn:aws:iam::123456789012:role/AppRole \
  --operations Encrypt Decrypt GenerateDataKey
```

Common exam clues:
- If an AWS service needs access to a CMK (customer managed key), think **grant**
- If a human or application role needs access, think **IAM policy plus key policy**
- If the key policy does not delegate to IAM, IAM permissions alone will not work

**KMS condition keys that matter**

Condition keys are often the difference between a secure design and an over-permissive one.

- `kms:ViaService`
  - Restricts use to a specific AWS service
  - Useful for Secrets Manager, EBS, RDS, S3, and other integrated services
- `kms:CallerAccount`
  - Restricts the account making the request
  - Useful in cross-account or service-integrated scenarios
- `kms:EncryptionContext:*`
  - Restricts operations to a matching encryption context
  - Useful when you want the ciphertext to be usable only for a specific application or resource
- `kms:GrantIsForAWSResource`
  - Helps ensure grants are created for AWS resources rather than arbitrary principals
- `kms:MultiRegion`
  - Useful when designing policies for multi-Region keys and replica behavior

Example using encryption context:

```json
{
  "Sid": "RestrictByEncryptionContext",
  "Effect": "Allow",
  "Principal": { "AWS": "arn:aws:iam::123456789012:role/AppRole" },
  "Action": [
    "kms:Encrypt",
    "kms:Decrypt",
    "kms:GenerateDataKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:EncryptionContext:AppName": "Billing"
    }
  }
}
```

This pattern is useful when the application always supplies a known encryption context and you want to prevent reuse outside that context.

**AWS managed keys versus customer managed keys**

AWS managed keys are created and managed by AWS for a service. Customer managed keys are created in your account and give you full policy control.

| Aspect | AWS managed key | Customer managed key |
|---|---|---|
| Policy control | Limited | Full |
| Key policy editing | No | Yes |
| IAM delegation | Service-defined | You define it |
| Cross-account design | Usually limited | Flexible |
| Rotation control | AWS-managed | Configurable |
| Best for | Simple service encryption | Fine-grained access and compliance |

For Secrets Manager:
- AWS managed keys are simpler
- Customer managed keys are preferred when you need explicit control, cross-account patterns, or strict separation of duties

**Secrets Manager fundamentals**

Secrets Manager stores and rotates secrets such as database credentials, API keys, and tokens. It encrypts secret values at rest using KMS. The secret metadata and version staging labels are managed by the service, while the secret value is protected by the selected KMS key.

Typical use cases:
- Database credentials with automatic rotation
- Third-party API keys
- Application secrets shared across workloads
- Secrets that require auditability and controlled access

Example secret creation with a customer managed key:

```bash
aws secretsmanager create-secret \
  --name prod/app/db \
  --secret-string '{"username":"appuser","password":"ExamplePassw0rd!"}' \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234
```

**Secrets Manager and KMS policy interplay**

When Secrets Manager uses a customer managed key, the key policy must allow the service to use the key, and the IAM principal must be allowed to call Secrets Manager APIs. In many designs, the key policy also delegates to IAM through the root statement so that application roles can be authorized cleanly.

A practical pattern is:
- Key policy allows the account to administer the key
- Key policy allows Secrets Manager service use with conditions
- IAM policy allows the application role to call `secretsmanager:GetSecretValue`
- Secret resource policy is added only when sharing or cross-account access is needed

Example IAM policy for an application role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/app/db-*"
    }
  ]
}
```

**Cross-account secret access**

Cross-account access to a secret requires careful separation of responsibilities.

For a consumer account to read a secret in an owner account:
- The consuming principal needs IAM permission in its own account
- The secret’s resource policy must allow that principal or account
- The owning account’s KMS key policy must allow the decrypt path required by the access pattern

A common mistake is to add only a secret resource policy and forget the KMS key policy. If the secret is encrypted with a customer managed key, the decrypt operation still depends on KMS authorization.

Typical pattern:
- Secret resource policy grants `secretsmanager:GetSecretValue` to the consuming role
- KMS key policy allows the relevant principal or service path to decrypt
- Consumer IAM policy allows the role to call Secrets Manager

Example secret resource policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowConsumerAccountRead",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::210987654321:role/ConsumerAppRole" },
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "*"
    }
  ]
}
```

> Cross-account secret sharing is often misunderstood: the secret policy controls access to the secret, but the KMS key policy still controls whether the ciphertext can be decrypted.

If the consumer is expected to decrypt directly through its own path, the KMS key policy must explicitly support that path. If the architecture relies on Secrets Manager service-mediated access, the key policy should authorize the service use pattern accordingly.

**Secrets Manager rotation workflow**

Rotation is usually implemented with a Lambda function that performs the secret lifecycle steps. The service manages staging labels such as `AWSCURRENT`, `AWSPENDING`, and `AWSPREVIOUS` as part of the workflow. Rotation Lambda typically creates or updates the pending version, validates it, and then moves labels using `UpdateSecretVersionStage`.

A practical rotation sequence is:
1. Secrets Manager invokes the rotation Lambda with a rotation step.
2. The Lambda creates or updates the pending secret value.
3. The Lambda tests the new credentials against the target system.
4. The Lambda promotes the new version by moving staging labels.
5. Secrets Manager retains the previous version for rollback and recovery.

Example Lambda permissions often needed for rotation:
- `secretsmanager:GetSecretValue`
- `secretsmanager:PutSecretValue`
- `secretsmanager:UpdateSecretVersionStage`
- `secretsmanager:DescribeSecret`
- `kms:Decrypt` and sometimes `kms:Encrypt` or `kms:GenerateDataKey` if the Lambda handles encrypted material directly
- Target database permissions to create or alter credentials

Example IAM policy fragment for a rotation Lambda:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:PutSecretValue",
        "secretsmanager:UpdateSecretVersionStage",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/app/db-*"
    }
  ]
}
```

Operational safeguards:
- Ensure the Lambda can reach the database network path
- Validate that the target database user has permission to change its own password or create a new credential
- Handle rollback if the new credential fails validation
- Watch for staging label drift if a rotation step fails midway
- Avoid deleting the previous version until the new version is confirmed usable

**Rotation failure modes**

Rotation failures are often caused by permissions or connectivity, not by Secrets Manager itself.

Common failure modes:
- Lambda lacks permission to update the secret version stage
- Lambda cannot connect to the database due to security group or subnet issues
- Database user lacks permission to change its password
- KMS key policy blocks decrypt or generate data key operations
- The rotation function succeeds in one step but fails before label promotion

Exam-oriented troubleshooting approach:
- Check CloudWatch Logs for the rotation Lambda
- Verify the secret’s version staging labels
- Confirm the Lambda execution role can call both Secrets Manager and the target database
- Confirm the KMS key policy allows the service or role path
- Validate that the secret is not stuck in `AWSPENDING`

**Multi-Region secrets and replica keys**

Secrets Manager supports multi-Region replication patterns. In a common design, you create a primary secret in one Region and replicate it to another Region for disaster recovery or latency reduction. Each Region uses its own KMS key, and the replica secret is encrypted with the replica Region’s key.

Design considerations:
- The primary secret uses a primary KMS key
- The replica secret uses a replica or separate KMS key in the target Region
- Policies in each Region must allow the relevant service and principals
- Multi-Region KMS keys can simplify key material consistency, but authorization still needs to be correct in each Region

Example replication CLI:

```bash
aws secretsmanager replicate-secret-to-regions \
  --secret-id arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/app/db-abc123 \
  --add-replica-regions Region=us-west-2,KmsKeyId=arn:aws:kms:us-west-2:123456789012:key/efgh-5678
```

Replica considerations:
- The replica secret is a separate resource in the destination Region
- Access policies may need to be duplicated or adapted
- Application failover should reference the Region-appropriate secret ARN or use a discovery layer

**Secrets Manager versus SSM Parameter Store**

AWS Systems Manager Parameter Store and Secrets Manager both store configuration values, and both can use KMS. The choice depends on lifecycle and operational needs.

| Capability | Secrets Manager | SSM Parameter Store |
|---|---|---|
| Secret rotation | Native rotation workflows | No native secret rotation workflow |
| Secret-specific features | Version staging labels, rotation integration | Simpler parameter model |
| Cost | Higher | Lower, especially for standard parameters |
| Best fit | Credentials and high-value secrets | Configuration values and less dynamic secrets |

Use Secrets Manager when:
- You need automatic rotation
- You need version staging and secret lifecycle management
- You need tighter secret-specific integrations

Use Parameter Store when:
- You need simple configuration storage
- You want lower cost for non-rotating values
- You do not need the full secret lifecycle feature set

Both services can integrate with KMS, but Secrets Manager is usually the stronger SAP-C02 answer for credential rotation and managed secret handling.

**Exam decision guide**

When you see a scenario, ask:
- Is the access for a human, application, or AWS service?
- Is the key customer managed or AWS managed?
- Is the secret shared across accounts?
- Does the service need a grant or a policy?
- Is rotation required?
- Is multi-Region failover part of the design?

Quick mapping:
- **Application role using a CMK**: IAM policy plus key policy delegation
- **AWS service using a CMK**: key policy or service-created grant
- **Cross-account secret read**: secret resource policy plus KMS authorization path
- **Automatic rotation**: Secrets Manager plus Lambda plus target system permissions
- **Disaster recovery**: multi-Region secret replication with Region-specific KMS authorization

**Common pitfalls**

- Assuming IAM permissions alone are enough for a customer managed KMS key
- Forgetting the default root delegation statement in the key policy
- Using a secret resource policy without fixing the KMS key policy
- Treating grants as a replacement for key policy design
- Omitting `kms:ViaService` and allowing broad key use
- Forgetting `UpdateSecretVersionStage` in rotation workflows
- Ignoring database permissions and network access for rotation Lambdas
- Assuming replica secrets automatically inherit all access controls without review

**What to remember for SAP-C02**

KMS policy design is about precise trust boundaries. Secrets Manager is about secure secret lifecycle management, not just storage. The best answer usually combines:
- A correctly delegated KMS key policy
- Least-privilege IAM permissions
- A secret resource policy only when sharing is required
- Rotation Lambda permissions that match the actual workflow
- Region-aware design for replication and failover

If you can explain why a request is allowed, which policy authorizes it, and how the service uses KMS under the hood, you are thinking at the right level for the exam.
