## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Lab objective and security mindset**

This lab focuses on finding and correcting common AWS security misconfigurations before they become incidents. In practice, you review identity, network, storage, logging, and encryption controls, then tighten them using least privilege and defense in depth.

A good remediation workflow is:

- Identify the exposed resource or overly permissive control.
- Confirm the business requirement for the current setting.
- Reduce scope, add conditions, or replace broad access with targeted access.
- Verify the change does not break required workloads.

Security review in AWS is rarely about one setting. You usually need to correlate IAM, security groups, network ACLs, S3 policies, KMS keys, logging, and account-level guardrails.

**Common misconfiguration categories**

The most frequent issues in AWS environments are predictable and repeatable. You should know how to spot them quickly.

- **Overly permissive IAM**: policies with `Action: "*"`, `Resource: "*"`, or broad admin access attached to too many principals.
- **Public network exposure**: security groups allowing `0.0.0.0/0` or `::/0` to sensitive ports such as `22`, `3389`, `3306`, or `5432`.
- **Unrestricted storage access**: S3 buckets with public access, weak bucket policies, or missing Block Public Access settings.
- **Weak encryption posture**: EBS volumes, RDS databases, S3 objects, or Secrets Manager values not protected with customer-managed keys where required.
- **Insufficient logging**: CloudTrail disabled, log delivery broken, or logs not protected from tampering.
- **Over-shared trust relationships**: IAM roles trusted by too many accounts, services, or federated identities.
- **Missing network segmentation**: workloads placed in public subnets without need, or east-west traffic left open between tiers.

**IAM review and remediation**

IAM (Identity and Access Management) is usually the first place to look because excessive permissions create the largest blast radius. Review both identity-based policies and resource-based policies.

Use the AWS CLI to inspect attached policies and effective permissions:

```bash
aws iam list-attached-user-policies --user-name Alice
aws iam list-attached-role-policies --role-name AppRole
aws iam get-policy-version \
  --policy-arn arn:aws:iam::123456789012:policy/ExamplePolicy \
  --version-id v1
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject ec2:TerminateInstances
```

Key remediation patterns:

- Replace wildcard permissions with service- and resource-specific actions.
- Scope access to specific ARNs instead of `*`.
- Add conditions such as `aws:PrincipalArn`, `aws:SourceIp`, `aws:MultiFactorAuthPresent`, or `aws:RequestedRegion` when appropriate.
- Prefer roles over long-term access keys for workloads and automation.
- Remove unused access keys and rotate credentials regularly.
- Use permission boundaries for delegated administration.
- Use service control policies (SCPs) in AWS Organizations to prevent dangerous actions at the account level.

A tighter policy often looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadOnlyS3AccessToOneBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

> If you see `AdministratorAccess` attached to a human user or application role, treat it as a high-priority finding unless there is a documented exception.

**Network exposure review**

Security groups are stateful and are the most common source of accidental exposure. Network ACLs (NACLs) are stateless and can add another layer of control, but they are usually not the primary fix for overly open access.

Review security groups for public ingress:

```bash
aws ec2 describe-security-groups \
  --filters Name=ip-permission.cidr,Values=0.0.0.0/0
```

Typical remediation steps:

- Remove public ingress from administrative ports.
- Restrict access to bastion hosts, VPN ranges, or AWS Systems Manager Session Manager instead of direct SSH/RDP.
- Limit application ports to upstream load balancers or specific security groups.
- Use separate security groups per tier to reduce lateral movement.
- Keep outbound rules as restrictive as the application allows, especially for sensitive environments.

Example of a safer security group pattern:

```json
{
  "IpPermissions": [
    {
      "IpProtocol": "tcp",
      "FromPort": 443,
      "ToPort": 443,
      "UserIdGroupPairs": [
        {
          "GroupId": "sg-alb"
        }
      ]
    }
  ]
}
```

For NACLs, remember the stateless behavior:

- You must allow both inbound and outbound return traffic explicitly.
- Rule order matters because the first matching rule wins.
- NACLs are useful for coarse subnet-level restrictions, but they do not replace security groups.

**S3 bucket security review**

S3 misconfigurations often involve public access, weak bucket policies, or missing encryption and logging. Start by checking Block Public Access and bucket policy status.

Useful CLI checks:

```bash
aws s3api get-public-access-block --bucket example-bucket
aws s3api get-bucket-policy --bucket example-bucket
aws s3api get-bucket-encryption --bucket example-bucket
aws s3api get-bucket-versioning --bucket example-bucket
```

Remediation priorities:

- Enable Block Public Access at the account and bucket level unless public hosting is explicitly required.
- Remove public ACLs and public bucket policies.
- Enforce server-side encryption with S3 managed keys or KMS keys.
- Enable versioning to help recover from accidental deletion or overwrite.
- Use bucket policies to require secure transport with `aws:SecureTransport`.
- Log access using server access logging or CloudTrail data events when needed.

Example bucket policy that denies unencrypted or insecure requests:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyInsecureTransport",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ],
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    }
  ]
}
```

If the bucket stores sensitive data, consider requiring KMS encryption:

```json
{
  "Sid": "DenyIncorrectEncryptionHeader",
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::example-bucket/*",
  "Condition": {
    "StringNotEquals": {
      "s3:x-amz-server-side-encryption": "aws:kms"
    }
  }
}
```

**Encryption and key management**

Encryption is only useful when you understand where keys are stored, who can use them, and how access is audited. Review encryption at rest and in transit separately.

Common checks:

- EBS volumes should use encryption by default where possible.
- RDS should use encrypted storage for sensitive databases.
- S3 should use default encryption and policy enforcement for uploads.
- Secrets should be stored in AWS Secrets Manager or Systems Manager Parameter Store with encryption.
- TLS should be enforced for client connections and service endpoints.

For KMS (Key Management Service), verify:

- Key policies do not unintentionally grant broad access.
- IAM permissions align with the key policy.
- Rotation is enabled where supported and appropriate.
- Cross-account use is explicitly intended and documented.

Example CLI inspection:

```bash
aws kms describe-key --key-id alias/example-key
aws kms get-key-policy --key-id alias/example-key --policy-name default
aws kms list-grants --key-id alias/example-key
```

A common mistake is granting IAM permissions but forgetting that the KMS key policy still controls access. Both layers must allow the operation.

**Logging, detection, and auditability**

If you cannot see activity, you cannot reliably investigate or prove remediation. Logging is a security control, not just an operations feature.

Review these services first:

- **CloudTrail** for API activity and account changes.
- **AWS Config** for configuration drift and compliance rules.
- **CloudWatch Logs** for application and system logs.
- **VPC Flow Logs** for network traffic visibility.
- **S3 access logs or CloudTrail data events** for object-level access.

CLI checks for CloudTrail:

```bash
aws cloudtrail describe-trails
aws cloudtrail get-trail-status --name example-trail
aws cloudtrail get-event-selectors --trail-name example-trail
```

Remediation patterns:

- Enable multi-region CloudTrail for centralized visibility.
- Deliver logs to a dedicated log archive account.
- Protect log buckets with versioning, encryption, and restricted write access.
- Turn on log file validation where available.
- Use AWS Config managed rules to detect drift, such as public S3 buckets or unrestricted security groups.
- Send alerts for high-risk changes, such as IAM policy updates or security group modifications.

A practical detection workflow is:

1. Detect the misconfiguration with Config, Security Hub, or manual review.
2. Confirm the affected resource and scope.
3. Apply the least disruptive fix.
4. Re-test access and verify logs capture the change.

**Secrets and credential hygiene**

Hard-coded credentials are a frequent and avoidable risk. Review code repositories, user data scripts, CI/CD variables, and instance metadata usage.

Best practices:

- Store secrets in Secrets Manager or Parameter Store.
- Use IAM roles for EC2, Lambda, ECS, and EKS workloads instead of embedded keys.
- Disable or rotate unused access keys.
- Use short-lived credentials through federation or role assumption.
- Avoid placing secrets in AMIs, user data, or plaintext environment files.

Example of retrieving a secret at runtime:

```bash
aws secretsmanager get-secret-value \
  --secret-id prod/db/password \
  --query SecretString \
  --output text
```

If an application must read a secret, grant only the exact secret ARN and only the required action.

**Account-level guardrails and governance**

Some misconfigurations are easier to prevent than to fix repeatedly. Use governance controls to reduce the chance of recurrence.

Important guardrails include:

- SCPs to deny risky actions such as disabling CloudTrail or making S3 buckets public.
- AWS Config rules to continuously evaluate resource posture.
- Security Hub to aggregate findings across services.
- IAM Access Analyzer to identify unintended external access.
- Organizations delegated administration for centralized security tooling.

Example SCP concept:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyDisablingCloudTrail",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail"
      ],
      "Resource": "*"
    }
  ]
}
```

Use guardrails carefully. Overly broad denies can break legitimate administration, so test in a non-production OU or account first.

**Practical remediation workflow**

When you encounter a security finding, work through it systematically instead of changing settings ad hoc.

1. Identify the resource type and exposure path.
2. Determine whether the exposure is intentional.
3. Check the effective permissions and trust relationships.
4. Apply the narrowest fix that meets the requirement.
5. Validate with a test principal, test request, or policy simulation.
6. Record the change and add a preventive control if the issue is recurring.

Useful validation commands:

```bash
aws iam simulate-custom-policy \
  --policy-input-list file://policy.json \
  --action-names s3:PutObject s3:GetObject \
  --resource-arns arn:aws:s3:::example-bucket/test.txt

aws ec2 describe-security-group-rules \
  --filters Name=group-id,Values=sg-0123456789abcdef0
```

**Troubleshooting patterns**

Some fixes fail because another layer still allows or denies access.

- If IAM allows access but the request still fails, check the resource policy, KMS key policy, or SCP.
- If a bucket remains public after policy changes, check ACLs and account-level Block Public Access.
- If a port is still reachable, verify both security groups and NACLs, plus any load balancer listeners.
- If logs are missing, confirm the destination bucket policy, KMS permissions, and service delivery configuration.
- If a role cannot access a KMS-encrypted resource, verify both the IAM policy and the key policy.

**Exam-focused takeaways**

For SAP-C02, you should be able to reason about security at multiple layers and choose the least disruptive remediation.

- **Least privilege** is the default answer for IAM issues.
- **Security groups** are the first place to fix accidental network exposure.
- **Block Public Access** is a fast control for S3 public exposure.
- **CloudTrail and Config** are core tools for detection and auditability.
- **KMS key policy plus IAM policy** both matter for encrypted resources.
- **Guardrails** prevent repeat misconfigurations across accounts and OUs.

A strong architecture does not rely on one control. It combines preventive controls, detective controls, and verified remediation so that misconfigurations are both harder to introduce and easier to find.
