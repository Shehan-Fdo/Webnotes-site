## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Security logging and detective control strategy**

In a multi-account AWS environment, you usually separate responsibilities into at least three logical layers:

- **Workload accounts** run applications and generate telemetry.
- **Log archive accounts** store immutable or long-retention logs.
- **Security tooling accounts** host detective services, responders, and analysts.

This separation reduces blast radius, simplifies access control, and makes it easier to prove that logs are protected from accidental or malicious modification. For SAP-C02, you should understand not just which service collects which data, but also how those services behave across Regions, accounts, and organizations.

A strong design usually combines:

- **CloudTrail** for API activity and account governance events.
- **AWS Config** for configuration history and compliance evaluation.
- **GuardDuty** for threat detection.
- **Security Hub** for aggregation and prioritization.
- **Detective** for investigation and relationship analysis.
- **Security Lake** for normalized security data sharing and analytics.
- **CloudWatch Logs**, **S3**, and **Athena** for operational search and retention.

**Multi-account logging architecture**

A common enterprise pattern is:

- Organization management account
  - Owns AWS Organizations and often delegates security administration.
- Log archive account
  - Receives CloudTrail, Config, VPC Flow Logs, WAF logs, Route 53 Resolver logs, and other audit data.
- Security services account
  - Hosts Security Hub, GuardDuty delegated admin, Detective, and sometimes Security Lake delegated admin.
- Workload accounts
  - Produce logs and findings.

This design supports least privilege and makes it easier to apply service control policies (SCPs) that prevent disabling logging or tampering with security services.

Important operational principles:

- Use **SCPs** to deny actions such as `cloudtrail:StopLogging`, `cloudtrail:DeleteTrail`, `config:StopConfigurationRecorder`, and `logs:DeleteLogGroup` where appropriate.
- Use separate KMS keys for log encryption and security tooling where isolation matters.
- Grant read-only access to auditors and responders through tightly scoped IAM roles and cross-account access.

> For exam scenarios, assume that a centralized logging account is preferred unless the question explicitly requires local-only retention or a service limitation prevents centralization.

**AWS CloudTrail fundamentals**

CloudTrail records AWS API activity and is the primary source for answering “who did what, when, and from where?” It is especially important for governance, incident response, and forensic reconstruction.

CloudTrail event categories:

- **Management events**
  - Control plane actions such as `RunInstances`, `CreateUser`, `PutBucketPolicy`, and `AuthorizeSecurityGroupIngress`.
  - Usually enabled by default for many trails.
  - Can be read-only, write-only, or all.
- **Data events**
  - High-volume object-level or resource-level activity such as S3 object access and Lambda function invocation.
  - Must be explicitly enabled for the resources you care about.
  - Often the most expensive category because of volume.
- **Insight events**
  - Detect unusual API activity patterns, such as spikes in `DeleteBucket` or `TerminateInstances`.
  - Useful for anomaly detection, but not a replacement for GuardDuty or Security Hub.
- **Network activity events**
  - Capture network-related activity for supported services and can help with investigations where control-plane context is insufficient.

CloudTrail delivery options:

- **CloudTrail trails**
  - Deliver events to S3 and optionally CloudWatch Logs.
  - Best for durable audit retention and broad compatibility.
- **CloudTrail Lake**
  - A managed event data store for querying CloudTrail events with SQL-like queries.
  - Useful for investigations and ad hoc analysis.
- **Event data stores**
  - Support retention and query patterns distinct from S3-based trails.

Practical guidance:

- Use a trail for long-term audit retention and compliance.
- Use CloudTrail Lake when you need fast investigation queries without building your own query pipeline.
- Use Athena over S3 when you want low-cost ad hoc analysis on archived logs.
- Use OpenSearch or a SIEM when you need near-real-time search, correlation, and dashboards.

**Organization trails and AWS Organizations**

An **organization trail** captures activity across all accounts in an AWS Organization. This is a common SAP-C02 topic because it reduces operational overhead and improves coverage.

Key points:

- Only the **management account** or a properly delegated administration setup can create and manage organization-wide logging, depending on the service and feature.
- Organization trails apply to member accounts automatically.
- You typically enable the trail in all Regions to capture regional API activity consistently.
- Global service events are often delivered to a home Region, but you should understand the trail’s region scope and how it behaves in multi-Region setups.

Operational caveats:

- A member account cannot independently override an organization trail.
- If you use a delegated admin model, verify which account is allowed to administer the service and which account owns the trail or detector.
- Some services are region-specific for administration even when findings or logs are aggregated centrally.

A typical organization trail setup might look like this:

```bash
aws cloudtrail create-trail \
  --name org-audit-trail \
  --s3-bucket-name my-org-log-archive-bucket \
  --is-multi-region-trail \
  --enable-log-file-validation

aws cloudtrail put-event-selectors \
  --trail-name org-audit-trail \
  --event-selectors '[
    {
      "ReadWriteType": "All",
      "IncludeManagementEvents": true,
      "DataResources": [
        {
          "Type": "AWS::S3::Object",
          "Values": ["arn:aws:s3:::my-sensitive-bucket/"]
        }
      ]
    }
  ]'
```

For more granular data-event filtering, use **advanced event selectors** rather than assuming `put-event-selectors` can express every condition you need:

```bash
aws cloudtrail put-advanced-event-selectors \
  --trail-name org-audit-trail \
  --advanced-event-selectors '[
    {
      "Name": "S3ObjectAccess",
      "FieldSelectors": [
        {"Field": "eventCategory", "Equals": ["Data"]},
        {"Field": "resources.type", "Equals": ["AWS::S3::Object"]},
        {"Field": "resources.ARN", "StartsWith": ["arn:aws:s3:::my-sensitive-bucket/"]},
        {"Field": "readOnly", "Equals": ["false"]}
      ]
    }
  ]'
```

Use advanced selectors when you need precise scoping for cost control or compliance. They are especially useful for high-volume data events.

**CloudTrail delivery to a centralized S3 log archive**

A centralized S3 bucket is the standard long-term archive for CloudTrail logs. The bucket policy and KMS key policy must allow CloudTrail to write encrypted log files safely.

Important design considerations:

- The bucket should live in a dedicated log archive account.
- The bucket should block public access.
- Versioning should be enabled.
- Object lock may be appropriate for compliance use cases.
- Server-side encryption should use KMS when required by policy.

A simplified bucket policy pattern for CloudTrail delivery:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSCloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::my-org-log-archive-bucket",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/org-audit-trail"
        }
      }
    },
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-org-log-archive-bucket/AWSLogs/111122223333/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control",
          "AWS:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/org-audit-trail"
        }
      }
    }
  ]
}
```

Notes on this pattern:

- The account ID in the object path should match the trail owner or the organization delivery model you are using.
- For organization trails, ensure the path and conditions align with the actual delivery account and organization setup.
- In real deployments, you may need additional statements for multiple Regions or multiple trails.

A practical KMS key policy for CloudTrail encryption usually needs permissions such as:

- `kms:GenerateDataKey*`
- `kms:DescribeKey`
- `kms:Encrypt`

CloudTrail generally does not need broad decrypt permissions for its service principal in the way a human or application principal might. Keep the policy minimal and aligned to the service’s write path.

Example KMS key policy fragment:

```json
{
  "Sid": "AllowCloudTrailUseOfKey",
  "Effect": "Allow",
  "Principal": { "Service": "cloudtrail.amazonaws.com" },
  "Action": [
    "kms:GenerateDataKey*",
    "kms:DescribeKey",
    "kms:Encrypt"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "AWS:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/org-audit-trail"
    }
  }
}
```

Common failure modes:

- The bucket policy allows S3 writes, but the KMS key policy blocks encryption.
- The KMS key policy is too broad or uses the wrong source condition.
- The bucket path does not match the trail owner or organization delivery structure.
- The trail is multi-Region, but the destination bucket policy only anticipates one Region.

**CloudTrail troubleshooting and investigation use cases**

CloudTrail is often the first place you look when a security event occurs.

Useful investigation patterns:

- Identify the principal that changed a security group.
- Trace who disabled logging or altered a trail.
- Determine whether an IAM role was assumed from an unexpected source.
- Confirm whether an S3 bucket policy or KMS key policy changed before data access.
- Correlate `AssumeRole` activity with subsequent API calls.

Example Athena query against CloudTrail logs in S3:

```sql
SELECT eventtime, useridentity.arn, eventname, sourceipaddress, awsregion
FROM cloudtrail_logs
WHERE eventname = 'ConsoleLogin'
  AND errorcode IS NOT NULL
ORDER BY eventtime DESC
LIMIT 50;
```

For failed console logins, do not rely on a simplistic `responseElements.ConsoleLogin = Failure` pattern. Failed login events often surface through `errorMessage`, `additionalEventData`, or event-specific fields depending on the authentication flow. Validate the exact event structure before building metric filters or alerts.

**AWS Config for configuration history and compliance**

AWS Config records resource configuration changes and evaluates them against rules. It complements CloudTrail by answering “what changed in the resource state?” rather than “who called the API?”

Core capabilities:

- **Configuration recorder**
  - Captures configuration changes for supported resources.
- **Configuration history**
  - Stores point-in-time and change history.
- **Managed rules**
  - AWS-managed compliance checks.
- **Custom rules**
  - Lambda-based or policy-based checks for organization-specific requirements.
- **Conformance packs**
  - Bundles of rules and remediation actions deployed together.
- **Organization conformance packs**
  - Roll out compliance baselines across accounts and Regions.

Typical use cases:

- Detect public S3 buckets.
- Detect unencrypted EBS volumes.
- Detect overly permissive security groups.
- Enforce tagging standards.
- Verify CloudTrail and Config are enabled.

Example recorder setup:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder name=default,roleARN=arn:aws:iam::111122223333:role/aws-config-role,recordingGroup="{allSupported=true,includeGlobalResourceTypes=true}"

aws configservice put-delivery-channel \
  --delivery-channel name=default,s3BucketName=my-org-config-bucket
```

Notes:

- `roleARN` is the expected shorthand key in the CLI example.
- `includeGlobalResourceTypes=true` matters mainly for supported global resource types and should be considered in the context of the Region.
- In organization-wide designs, the exact setup differs between account-based and organization-based aggregation.

Organization aggregation and conformance:

- Use an **organization aggregator** to collect configuration data from multiple accounts and Regions.
- Use **organization conformance packs** to standardize compliance checks.
- Use **remediation actions** to automate fixes where safe.

Example managed rule and remediation workflow:

```bash
aws configservice put-config-rule \
  --config-rule '{
    "ConfigRuleName": "s3-bucket-public-read-prohibited",
    "Source": {
      "Owner": "AWS",
      "SourceIdentifier": "S3_BUCKET_PUBLIC_READ_PROHIBITED"
    }
  }'
```

When to use Config:

- Use it for drift detection and compliance evidence.
- Use it when you need a historical record of resource configuration.
- Use it with Security Hub to surface noncompliance in a security operations workflow.

**Security Hub for centralized findings and standards**

Security Hub aggregates findings from AWS services and partner integrations, then normalizes them into a common format. It is not a log archive; it is a findings aggregation and prioritization service.

Key concepts:

- **Findings**
  - Security issues reported by integrated services.
- **Standards**
  - Preset compliance frameworks such as:
    - AWS Foundational Security Best Practices
    - CIS AWS Foundations Benchmark
    - PCI DSS
  - Availability can vary by Region and account context.
- **Controls**
  - Individual checks within a standard.
- **Insights**
  - Queries and summaries over findings.

Organization-wide Security Hub design:

- Enable Security Hub in each Region you care about.
- Designate a delegated administrator for the organization.
- Configure auto-enable for new accounts where appropriate.
- Aggregate findings centrally for triage and reporting.

Example organization admin setup:

```bash
aws securityhub enable-organization-admin-account \
  --admin-account-id 111122223333
```

Then configure organization behavior in the delegated admin account and Region:

```bash
aws securityhub update-organization-configuration \
  --auto-enable true
```

Important notes:

- Security Hub is **Region-specific** for administration and aggregation.
- Enabling the delegated admin does not automatically enable every standard or every integration.
- You still need to enable standards, configure member account behavior, and verify cross-Region aggregation strategy.

Security Hub is often used with:

- **GuardDuty** for threat findings.
- **Inspector** for vulnerability findings.
- **Config** for compliance findings.
- **IAM Access Analyzer** for external access findings.

**GuardDuty for threat detection**

GuardDuty analyzes signals such as CloudTrail events, VPC Flow Logs, DNS logs, and optional data sources to detect suspicious behavior.

Common exam-relevant features:

- **Delegated administrator**
  - Centralize GuardDuty management across an organization.
- **Auto-enable**
  - Automatically enroll new accounts and optionally new supported data sources.
- **Trusted IP lists**
  - Suppress findings from known safe IP ranges.
- **Threat lists**
  - Flag known malicious IPs or domains.
- **Malware Protection**
  - Helps detect malware in supported storage and compute contexts.
- **EKS protection**
  - Extends detection to Amazon EKS activity.
- **S3 protection**
  - Adds object-level threat detection for S3 access patterns.

Operational pattern:

- Enable GuardDuty in all Regions where workloads run.
- Delegate administration to a security account.
- Turn on auto-enable for new accounts to avoid coverage gaps.
- Review trusted and threat lists carefully to avoid suppressing real attacks.

Example delegated admin workflow:

```bash
aws guardduty create-detector --enable
aws guardduty enable-organization-admin-account \
  --admin-account-id 111122223333
```

The exact API and CLI sequence can vary by feature and service version, so focus on the concept:

- Create or enable the detector in the Region.
- Designate the delegated admin.
- Enable organization auto-enrollment and data source coverage.

Common GuardDuty findings include:

- Unusual API calls from a new geography.
- Credential exfiltration indicators.
- Port scanning or reconnaissance.
- Crypto-mining behavior.
- Suspicious DNS activity.

**Amazon Detective for investigation**

Detective helps you investigate findings by building a graph of related entities, API calls, and behavior over time. It is especially useful after GuardDuty or Security Hub raises an alert.

What Detective needs:

- Data sources such as CloudTrail and VPC Flow Logs, depending on the investigation context.
- An enabled administrator account for the organization or Region.
- Sufficient historical data to make relationships meaningful.

What Detective is good for:

- Following an IAM principal across related API calls.
- Understanding lateral movement or unusual access patterns.
- Investigating whether a GuardDuty finding is isolated or part of a broader campaign.

Limitations and considerations:

- Detective is not a replacement for raw logs.
- It depends on the underlying telemetry being available and enabled.
- Coverage and availability can vary by Region and service integration.

Use Detective when you need to answer:

- Which resources did this principal touch?
- What other accounts or roles were involved?
- Was there a sequence of suspicious actions before the alert?

**Security Lake for normalized security data**

Security Lake centralizes security data in an open format and normalizes it using the Open Cybersecurity Schema Framework (OCSF). It is designed for security analytics and sharing, not as a direct substitute for every raw log archive.

Key points:

- **OCSF** normalization makes it easier to query across sources.
- It supports AWS sources and selected third-party sources.
- It is useful for SIEM, analytics, and cross-tool correlation.
- It is not the same as storing raw CloudTrail or application logs in S3.

Operational differences from S3 log archives:

- S3 log archives preserve raw service logs for compliance and replay.
- Security Lake stores normalized security records for analysis.
- You may use both:
  - S3 for immutable audit retention.
  - Security Lake for analytics and detection pipelines.

Security Lake is a strong fit when you need:

- Centralized security analytics across accounts and Regions.
- Standardized schema for multiple security sources.
- Integration with downstream analytics tools.

**CloudWatch Logs, metric filters, and alerting**

CloudWatch Logs is often used for near-real-time alerting on log streams from CloudTrail, application logs, and service logs.

Common patterns:

- Send CloudTrail to CloudWatch Logs for alerting on specific API activity.
- Use metric filters to count suspicious events.
- Trigger alarms and SNS notifications from those metrics.

Example metric filter for IAM policy changes:

```bash
aws logs put-metric-filter \
  --log-group-name /aws/cloudtrail/org-audit \
  --filter-name IAMPolicyChanges \
  --filter-pattern '{ ($.eventSource = iam.amazonaws.com) && (($.eventName = CreatePolicy) || ($.eventName = PutRolePolicy) || ($.eventName = AttachRolePolicy) || ($.eventName = PutUserPolicy)) }' \
  --metric-transformations metricName=IAMPolicyChanges,metricNamespace=Security,metricValue=1
```

For console login failures, validate the event structure before writing the filter. A robust pattern often checks for `ConsoleLogin` plus failure-related fields rather than assuming a single response field always indicates failure.

Example alarm workflow:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name IAMPolicyChangesAlarm \
  --metric-name IAMPolicyChanges \
  --namespace Security \
  --statistic Sum \
  --period 300 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --evaluation-periods 1 \
  --alarm-actions arn:aws:sns:us-east-1:111122223333:security-alerts
```

**VPC Flow Logs, Route 53 Resolver query logs, and WAF logs**

These logs are often used in incident response and network forensics.

VPC Flow Logs:

- Capture IP traffic metadata for ENIs, subnets, or VPCs.
- Useful for identifying unexpected connections, exfiltration patterns, and lateral movement.
- Can be delivered to CloudWatch Logs or S3.

Current CLI syntax varies by target and resource type, so prefer the documented parameter set for your CLI version. The important exam concept is that you can enable flow logs at the VPC, subnet, or ENI level and choose a destination such as CloudWatch Logs or S3.

Route 53 Resolver query logging:

- Captures DNS query metadata.
- Useful for detecting command-and-control domains or suspicious resolution patterns.
- Can be associated with VPCs and delivered to CloudWatch Logs or S3.

WAF logging:

- Sends web ACL logs to Kinesis Data Firehose, then to S3, OpenSearch, or another supported destination.
- Useful for analyzing blocked requests, rule matches, and attack patterns.

Example WAF logging destination pattern:

```json
{
  "LoggingConfiguration": {
    "ResourceArn": "arn:aws:wafv2:us-east-1:111122223333:regional/webacl/example/12345678-1234-1234-1234-123456789012",
    "LogDestinationConfigs": [
      "arn:aws:firehose:us-east-1:111122223333:deliverystream/waf-logs"
    ]
  }
}
```

**Incident response and forensic workflow**

A good response plan preserves evidence before making disruptive changes.

Typical workflow:

1. Confirm the alert source and scope.
2. Preserve logs and snapshots.
3. Isolate the affected resource if needed.
4. Collect volatile and non-volatile evidence.
5. Analyze the timeline and root cause.
6. Remediate and validate recovery.

Practical actions:

- Create an EBS snapshot before terminating an instance.
- Detach or quarantine security groups rather than deleting resources immediately.
- Use Systems Manager Automation runbooks for repeatable containment.
- Record the instance ID, IAM role, security group, and network path before changes.

Example isolation pattern:

```bash
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --groups sg-quarantine123456
```

Example snapshot creation:

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Forensic snapshot before containment"
```

Systems Manager is useful for controlled response:

- Run automation documents to collect logs.
- Use Session Manager instead of SSH where possible.
- Restrict who can start automation or open sessions.

**SCPs, IAM boundaries, and access control**

In multi-account security operations, prevention matters as much as detection.

Use SCPs to:

- Prevent disabling CloudTrail or Config.
- Prevent deleting log buckets or log groups.
- Restrict regions where security services can be disabled.
- Limit creation of unapproved IAM access keys or overly permissive policies.

Use IAM roles to:

- Separate responder, auditor, and administrator duties.
- Allow read-only access to logs without write access.
- Permit break-glass access with strong controls and monitoring.

A common pattern is:

- Security analysts assume a read-only investigation role.
- Incident responders assume a containment role with limited write permissions.
- Platform administrators manage service configuration but cannot delete archived logs.

**Choosing between CloudTrail Lake, Security Lake, Athena, and OpenSearch**

Use the right tool for the job:

| Service | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| CloudTrail Lake | API event investigation | Managed query experience for CloudTrail | Limited to CloudTrail event data |
| Security Lake | Normalized security analytics | Multi-source OCSF data model | Not a raw log archive |
| Athena | Ad hoc SQL on S3 logs | Low-cost, flexible queries | You manage schema and partitions |
| OpenSearch | Search and dashboards | Fast interactive search | More operational overhead |

Practical selection guidance:

- Choose **CloudTrail Lake** when the question is about AWS API activity and you want fast investigation without building pipelines.
- Choose **Security Lake** when you need normalized security data across multiple sources and tools.
- Choose **Athena** when you already archive logs in S3 and want SQL analysis.
- Choose **OpenSearch** when analysts need interactive search and dashboards.

**Exam-focused troubleshooting scenarios**

When a question asks why a security control is not working, check these common issues:

- Logging is enabled in one Region but not all required Regions.
- The delegated admin is set, but auto-enable is off for new accounts.
- The KMS key policy allows encryption but not the exact service principal path.
- The S3 bucket policy allows writes, but the object prefix or source ARN is wrong.
- Config is recording resources, but the aggregator or conformance pack is not deployed organization-wide.
- GuardDuty findings exist, but Security Hub integration is not enabled in the Region.
- Detective is enabled, but the underlying data sources are missing or incomplete.

When a question asks for the best architecture:

- Use centralized logging for auditability.
- Use delegated administration for security services.
- Use SCPs to enforce guardrails.
- Use multiple detection layers rather than a single service.
- Use raw logs for retention and forensic fidelity, and normalized services for analysis.

**Quick reference: service roles in a mature security operations model**

- **CloudTrail**
  - Records API activity and supports audit trails.
- **AWS Config**
  - Tracks configuration drift and compliance.
- **GuardDuty**
  - Detects threats and suspicious behavior.
- **Security Hub**
  - Aggregates and prioritizes findings.
- **Detective**
  - Investigates relationships and timelines.
- **Security Lake**
  - Normalizes security data for analytics.
- **CloudWatch Logs**
  - Supports near-real-time alerting and metric filters.
- **Athena**
  - Queries archived logs in S3.
- **S3**
  - Stores durable log archives.
- **KMS**
  - Protects log data at rest.

**Key takeaways**

- Centralize logs, but keep raw audit data separate from normalized analytics data.
- Use organization-wide controls to reduce blind spots.
- Understand the difference between trails, findings, configuration history, and normalized security data.
- Validate CLI syntax and service-specific behavior carefully in multi-account scenarios.
- For SAP-C02, expect questions that combine logging, detection, governance, and incident response into one design decision.
