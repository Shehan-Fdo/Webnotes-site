## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Authorization model overview**

AWS authorization is the combination of multiple policy types and guardrails that determine whether a request is allowed. The key exam idea is that AWS does not use a simple fixed sequence like “identity policy first, then resource policy, then boundary.” Instead, the final decision comes from evaluating all applicable permissions and restrictions together, with **explicit deny** always overriding any allow.

At a high level, you should think in terms of:
- **Identity-based policies** attached to IAM users, groups, or roles.
- **Resource-based policies** attached to the target resource.
- **Session policies** attached to an assumed-role or federated session.
- **Permissions boundaries** that cap what an identity can do.
- **Service control policies (SCPs)** in AWS Organizations that cap what accounts and principals can do.

A request is allowed only when the applicable policies collectively permit it and no applicable deny blocks it.

**Identity-based policies**

Identity-based policies are attached to IAM identities and define what those identities can do. They are the most common policy type and are used for day-to-day access control inside an account.

Use identity-based policies when you want to:
- Grant permissions to a user, group, or role.
- Centralize permissions for a workforce or application role.
- Control access to AWS APIs from an assumed role session.

Identity-based policies can be:
- **Managed policies**: reusable policies managed by AWS or your account.
- **Inline policies**: embedded directly in a single user, group, or role.

A simple example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListAndReadBucket",
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket",
        "s3:GetObject"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

Important characteristics:
- They evaluate against the principal making the request.
- They can use **condition keys** such as `aws:PrincipalTag`, `aws:RequestedRegion`, `aws:MultiFactorAuthPresent`, and service-specific keys.
- They are often paired with boundaries or SCPs in enterprise designs.
- They do not by themselves grant access to a resource if another applicable control blocks the request.

**Resource-based policies**

Resource-based policies are attached to the resource itself. They are common for services that need to authorize access directly on the target object, bucket, queue, topic, key, or function.

Typical examples include:
- S3 bucket policies
- SQS queue policies
- SNS topic policies
- KMS key policies
- Lambda function resource policies
- ECR repository policies
- Secrets Manager resource policies

Resource-based policies are especially important for cross-account access because the resource owner can grant access directly to a named principal or service principal.

Key points:
- They can name an IAM user, role, account, or AWS service principal.
- They can allow access without a matching identity-based allow in some services and scenarios.
- They can also contain explicit denies.
- They often use resource-specific condition keys and global keys such as `aws:SourceArn`, `aws:SourceAccount`, `aws:PrincipalArn`, and `aws:PrincipalOrgID`.

A bucket policy example granting a role in another account access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AnalyticsRole"
      },
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::example-bucket/reports/*"
    }
  ]
}
```

A service-principal example for Amazon S3 to invoke Lambda is not typical, but service principals are common in resource policies for event-driven integrations. For example, Lambda permissions often allow `s3.amazonaws.com`, `sns.amazonaws.com`, or `events.amazonaws.com` to invoke a function.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Invoke",
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": "lambda:InvokeFunction",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:ProcessUploads",
      "Condition": {
        "ArnLike": {
          "AWS:SourceArn": "arn:aws:s3:::example-bucket"
        }
      }
    }
  ]
}
```

**Same-account and cross-account behavior**

The exam often tests whether a resource-based policy alone is sufficient or whether the principal also needs an identity-based allow.

For same-account access:
- A resource-based policy can sometimes grant access directly to a user or role in the same account.
- An identity-based policy can also grant access.
- The final decision still respects explicit denies, boundaries, and SCPs.

For cross-account access:
- Some services allow a resource policy to grant access directly to a named IAM user or role in another account.
- Some services require both the resource policy and an identity-based allow in the caller’s account.
- Some services are designed around direct resource-policy authorization, while others rely more heavily on identity-side permissions.

Practical exam rule:
- Do not assume a universal “both sides are always required” pattern.
- Check the service model:
  - S3 bucket policies can directly grant cross-account access to IAM principals.
  - SNS topic policies and SQS queue policies can directly grant access to named principals and service principals.
  - Lambda resource policies can directly grant invocation permissions.
  - KMS key policies are special and require careful handling of the key policy itself.

**KMS authorization**

AWS Key Management Service (KMS) is a common trap because authorization is not just “key policy plus identity policy.” The **key policy** is central.

For IAM permissions to be effective on a KMS key:
- The key policy must either explicitly allow the principal, or
- The key policy must allow the account to use IAM policies for the key, so IAM identity policies can be evaluated.

If the key policy does not permit IAM policy use or the principal directly, then an IAM allow alone does not help.

Common KMS patterns:
- Key policy grants the account root or a key admin role permission to manage the key.
- IAM policies grant application roles permission to use cryptographic operations such as `kms:Encrypt`, `kms:Decrypt`, or `kms:GenerateDataKey`.
- Grants are sometimes used for temporary or service-specific access.

Example key policy pattern that enables IAM policies in the account:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableIAMPolicies",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": "kms:*",
      "Resource": "*"
    }
  ]
}
```

A more realistic KMS use policy for an application role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowAppUseOfKey",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/AppRole"
      },
      "Action": [
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "*"
    }
  ]
}
```

**Session policies**

Session policies are permissions passed when a principal assumes a role or uses federated access. They are attached to the resulting session, not to the role itself.

Use session policies when you want to:
- Further restrict a role session for a specific workload.
- Limit temporary access for a federated user.
- Scope down permissions for a particular automation run.

Important properties:
- Session policies can only **reduce** permissions.
- They cannot grant permissions beyond the role’s identity-based policy.
- The effective permissions of the session are the intersection of the role’s permissions and the session policy, then further constrained by boundaries and SCPs, with explicit denies still winning.

Example STS assume-role call with a session policy:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/DeployRole \
  --role-session-name deploy-20260426 \
  --policy '{
    "Version":"2012-10-17",
    "Statement":[
      {
        "Effect":"Allow",
        "Action":[
          "s3:GetObject",
          "s3:PutObject"
        ],
        "Resource":"arn:aws:s3:::example-artifacts/*"
      }
    ]
  }'
```

Practical notes:
- Session policies are common with role sessions and federated access.
- Session tags can also be used with role sessions and may contribute to the packed policy size limit.
- The packed policy size limit matters when you combine session policy text, managed policy ARNs, and session tags.

If you need to inspect the size impact:
```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/DeployRole \
  --role-session-name test \
  --policy file://session-policy.json \
  --tags Project=Alpha Environment=Dev
```

**Permissions boundaries**

A permissions boundary is an advanced IAM feature that sets the maximum permissions an identity can ever receive from identity-based policies. It does not grant permissions by itself.

Use a boundary when you want to:
- Delegate IAM administration safely.
- Prevent privilege escalation by developers or automation.
- Cap permissions for roles created by a team.

Key behavior:
- The identity’s effective permissions are limited by the boundary.
- A boundary does not affect resource-based policies in the same way as identity-based policies, but it still matters in the overall authorization result.
- Boundaries are especially useful in delegated administration and multi-team environments.

Example boundary:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:*",
        "ec2:Describe*"
      ],
      "Resource": "*"
    }
  ]
}
```

**Service control policies**

Service control policies are organization-level guardrails. They do not grant permissions; they define the maximum permissions available to accounts and principals in those accounts.

Important exam points:
- SCPs apply only in AWS Organizations.
- They restrict what identity-based policies and some resource-based access paths can ultimately do.
- An SCP deny blocks the action even if an identity policy or resource policy allows it.
- An SCP allow does not grant access by itself.

Think of SCPs as a ceiling for the account, not as a permission source.

**How the layers combine**

A useful mental model is:
- Identity-based policies define what the principal can do.
- Resource-based policies define what the resource will accept.
- Session policies narrow a temporary session.
- Boundaries cap the identity’s maximum permissions.
- SCPs cap the account’s maximum permissions.
- Explicit denies override allows everywhere.

For same-account access, a request may be allowed by either:
- An identity-based allow, if no boundary, SCP, or explicit deny blocks it, or
- A resource-based allow that directly authorizes the principal, depending on the service and resource type.

For cross-account access, the resource policy often plays the decisive role for the target resource, but the caller may still need an identity-based allow depending on the service. Do not generalize beyond the service documentation.

**Policy variables and conditions**

Policy variables and conditions are common in exam scenarios because they let you write reusable, context-aware policies.

Examples of useful condition keys:
- `aws:PrincipalArn`
- `aws:PrincipalAccount`
- `aws:PrincipalOrgID`
- `aws:SourceArn`
- `aws:SourceAccount`
- `aws:RequestedRegion`
- `aws:MultiFactorAuthPresent`
- `aws:userid`
- `aws:PrincipalTag/<tag-key>`
- `aws:ResourceTag/<tag-key>`

Policy variables can make policies dynamic. For example, you can scope access to a user’s own home folder in S3:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "HomeFolderAccess",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::example-bucket/home/${aws:username}/*"
    }
  ]
}
```

Common condition patterns:
- Use `aws:SourceArn` and `aws:SourceAccount` to prevent confused deputy problems in service-to-service access.
- Use `aws:PrincipalOrgID` to allow only principals from your organization.
- Use `aws:MultiFactorAuthPresent` to require MFA for sensitive actions.
- Use tag-based conditions for attribute-based access control (ABAC).

Example ABAC-style condition:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "ec2:StartInstances",
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:ResourceTag/Environment": "${aws:PrincipalTag/Environment}"
        }
      }
    }
  ]
}
```

**Trust policy vs resource policy**

A trust policy is a special resource-based policy on an IAM role that controls who can assume the role. It is not the same as a general resource policy on S3, SNS, or SQS, but it uses the same policy language.

Use a trust policy to answer:
- Who can call `sts:AssumeRole`?
- Which external accounts, users, roles, or services can assume this role?

Example trust policy for a service principal:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example trust policy for cross-account role assumption:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "example-external-id"
        }
      }
    }
  ]
}
```

Important distinction:
- A trust policy controls role assumption.
- A resource policy controls access to the resource itself.
- A role session created by `AssumeRole` then uses identity-based permissions plus any session policy limits.

**CLI simulation and troubleshooting**

The IAM policy simulator is useful, but you must understand its limits. A simple `simulate-principal-policy` call does not automatically model every surrounding control unless you provide the relevant inputs.

Example:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject s3:PutObject \
  --resource-arns arn:aws:s3:::example-bucket/data/file1.txt
```

Practical caution:
- This command evaluates the principal’s policies, but it does not fully represent every real-world authorization layer unless you include the relevant context.
- To reason about the final decision, also consider permissions boundaries, SCPs, session policies, and resource-based policies.
- For cross-account or service-invoked access, simulation may not capture the full service-specific behavior.

Useful troubleshooting steps:
1. Check the identity policy attached to the caller.
2. Check the resource policy on the target.
3. Check the role trust policy if the issue is role assumption.
4. Check the session policy if the caller is using STS.
5. Check the permissions boundary and SCPs.
6. Look for explicit denies and condition mismatches.

**Exam-style scenarios**

Scenario 1: Same-account S3 access with a boundary
- A developer role has an identity policy allowing `s3:PutObject`.
- The role has a permissions boundary that allows only `s3:GetObject`.
- Result: `s3:PutObject` is denied because the boundary caps the role’s effective permissions.

Scenario 2: Cross-account SNS publish
- Account A owns an SNS topic.
- The topic policy allows a role in Account B to publish.
- Depending on the service and setup, the role may also need an identity-based allow in Account B.
- Result: do not assume the resource policy alone is universally sufficient; verify the service’s authorization model.

Scenario 3: KMS decrypt for an application role
- The role has an IAM policy allowing `kms:Decrypt`.
- The KMS key policy does not allow the account to use IAM policies and does not directly allow the role.
- Result: decrypt fails because the key policy is the gatekeeper.

Scenario 4: Federated access with session policy
- A federated user assumes a role that can access multiple S3 prefixes.
- The session policy limits access to `arn:aws:s3:::example-bucket/project-a/*`.
- Result: the session can only use the subset allowed by both the role and the session policy.

**Practical design guidance**

Use identity-based policies when:
- You control the principal and want reusable permissions.
- You are granting access inside an account.
- You want to pair permissions with boundaries or ABAC.

Use resource-based policies when:
- You need direct access control on the resource.
- You need cross-account access to a named principal.
- You need to authorize AWS service principals for event-driven integrations.

Use session policies when:
- You need temporary least-privilege scoping.
- You want to narrow a role session for a specific task or tenant.
- You are working with federated identities or STS-based automation.

Use SCPs and boundaries when:
- You need guardrails that prevent privilege escalation.
- You want to enforce organization-wide or team-wide limits.

**Quick comparison**

| Policy type | Attached to | Grants access | Limits access | Common use |
|---|---|---:|---:|---|
| Identity-based | User, group, role | Yes | No | Day-to-day permissions |
| Resource-based | Resource | Yes | No | Cross-account and service access |
| Session policy | STS session | No | Yes | Temporary least privilege |
| Permissions boundary | Identity | No | Yes | Delegated admin guardrail |
| SCP | Organization/account | No | Yes | Org-wide guardrail |

**Key takeaways**

- AWS authorization is additive only in the sense that multiple allows may contribute; explicit denies and guardrails still win.
- Resource-based policies can directly authorize named principals or service principals, but the exact cross-account behavior is service-specific.
- KMS is special: the key policy is central, and IAM permissions alone are not enough unless the key policy permits IAM use.
- Session policies only reduce permissions and are common with STS and federated access.
- Conditions, tags, and policy variables are essential for real-world least-privilege designs and exam scenarios.
