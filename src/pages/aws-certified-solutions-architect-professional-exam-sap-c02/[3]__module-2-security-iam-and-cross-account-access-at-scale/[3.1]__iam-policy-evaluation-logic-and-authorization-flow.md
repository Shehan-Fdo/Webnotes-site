## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**IAM authorization model overview**

AWS evaluates a request by combining the caller’s identity, the requested action, the target resource, and the applicable policy types. The exam often tests whether you can identify which policy type grants access, which policy type only limits access, and which policy type can override everything with an explicit deny.

At a high level, authorization is not a single policy check. It is a sequence of checks across identity-based policies, resource-based policies, permissions boundaries, session policies, service control policies (SCPs), and service-specific authorization layers such as KMS key policies or S3 object ownership rules.

- **Identity-based policies** attach to IAM users, groups, and roles.
- **Resource-based policies** attach to the resource itself, such as an S3 bucket policy, KMS key policy, SNS topic policy, or Lambda resource policy.
- **Permissions boundaries** cap the maximum permissions a user or role can ever receive.
- **SCPs** cap the maximum permissions available in an AWS Organizations account or OU.
- **Session policies** further restrict temporary credentials from STS (AWS Security Token Service).

**Core evaluation rule**

AWS authorization is effectively an intersection of allowed permissions, followed by explicit deny checks. If any applicable policy explicitly denies the request, the request fails. If no policy allows the request, the result is an implicit deny.

A useful mental model is:

1. Start with the request context.
2. Evaluate explicit denies across all applicable policy layers.
3. Evaluate allows in identity-based and resource-based policies.
4. Apply boundaries, SCPs, and session policies as upper bounds.
5. Apply service-specific rules that may further restrict access.

> **Important:** Do not assume a resource policy always replaces identity permissions. For some services and principals, a resource policy can grant direct access; for others, the caller still needs identity-based permission or a trust/session path. Always consider the service and principal type.

**Policy types and what they do**

| Policy type | Can grant access? | Can limit access? | Typical use |
|---|---:|---:|---|
| Identity-based policy | Yes | Yes, indirectly via explicit deny | Attach permissions to IAM users, groups, roles |
| Resource-based policy | Yes | Yes, via explicit deny | Allow cross-account or service principals |
| Permissions boundary | No | Yes | Set maximum permissions for a user or role |
| SCP | No | Yes | Set account/OU permission guardrails |
| Session policy | No | Yes | Restrict temporary STS credentials |
| KMS key policy | Yes | Yes | Authorize use of a KMS key |
| ACL | Limited | Limited | Legacy or object-level access control in some services |

**Identity-based policy evaluation**

Identity-based policies are the first place you usually look when a principal is denied. They answer: “What is this user or role allowed to do in general?”

For an IAM user, the effective permissions are the union of:
- Direct user policies
- Group policies
- Any permissions boundary attached to the user
- Any SCPs that apply to the account
- Any session policy if the user is using temporary credentials through federation or STS

For an IAM role session, the effective permissions are the union of:
- The role’s identity-based policies
- Any permissions boundary attached to the role
- Any session policy attached when the role was assumed
- Any SCPs that apply to the account

A role session is not the same as the role identity. When you assume a role, AWS evaluates the resulting session principal, and resource policies may reference the role ARN or the assumed-role session ARN depending on the service and policy type.

**Assumed-role sessions and federated identities**

When a principal assumes a role through STS, the session inherits the role’s permissions but can be further restricted by a session policy. This matters in cross-account access and in troubleshooting because the resource policy may allow the role, but the session policy can still block the action.

Federated identities follow a similar pattern:
- **SAML** federation uses an identity provider assertion to call `AssumeRoleWithSAML`.
- **OIDC** federation uses a web identity token to call `AssumeRoleWithWebIdentity`.
- The trust policy on the role must allow the federation principal.
- The resulting session permissions are the intersection of the role permissions and any session policy or session tags-based conditions.

Example trust policy for OIDC:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
        }
      }
    }
  ]
}
```

**Resource-based policy evaluation**

Resource-based policies are attached to the target resource and can directly authorize access for:
- IAM users and roles in the same or another account
- Assumed-role sessions, depending on the service and policy syntax
- AWS service principals such as `lambda.amazonaws.com`, `sns.amazonaws.com`, or `events.amazonaws.com`

Resource policies are especially important for cross-account access because they let the resource owner grant access without needing to manage the caller’s identity policy in the resource account.

Common examples:
- S3 bucket policies
- KMS key policies
- SNS topic policies
- SQS queue policies
- Lambda function resource policies
- Secrets Manager resource policies

A resource policy can allow access even when the caller has no matching identity-based allow, but this is not universal. Some services still require the caller to have identity permissions, and some operations require both sides of the authorization path to be correct.

**Principal, NotPrincipal, and explicit deny**

Resource-based policies use `Principal` to specify who can access the resource. `NotPrincipal` is the inverse and is often used in deny statements, but it is easy to misuse.

- **Principal** identifies the allowed caller.
- **NotPrincipal** identifies everyone except the listed principal.
- **Explicit deny** in a resource policy overrides any allow from identity policies, boundaries, or SCPs.

Example bucket policy allowing a specific role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadFromAnalyticsRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AnalyticsReadRole"
      },
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::example-data-bucket/*"
    }
  ]
}
```

Example explicit deny using `NotPrincipal`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyEveryoneExceptBreakGlassRole",
      "Effect": "Deny",
      "NotPrincipal": {
        "AWS": "arn:aws:iam::111122223333:role/BreakGlassAdmin"
      },
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::example-data-bucket",
        "arn:aws:s3:::example-data-bucket/*"
      ]
    }
  ]
}
```

Use `NotPrincipal` carefully. It can create surprising results when combined with assumed-role sessions, service principals, or future principals you did not intend to exclude.

**Permissions boundaries**

A permissions boundary is an upper bound on what a user or role can do. It does not grant permissions by itself. It only limits the maximum permissions that identity-based policies can provide.

Think of it as: “Even if the identity policy says allow, the boundary must also allow.”

Important points:
- Boundaries apply to IAM users and roles, not to groups.
- Boundaries do not affect resource-based policies directly in the same way they affect identity permissions, but the effective access of the principal is still constrained by the boundary.
- A boundary is not a substitute for an allow policy.

Example boundary that limits a role to read-only S3 access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlyS3",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-data-bucket",
        "arn:aws:s3:::example-data-bucket/*"
      ]
    }
  ]
}
```

This example is only an upper bound. The role still needs identity policies that request those actions, and the resource ARNs must match the action type correctly:
- `s3:ListBucket` uses the bucket ARN
- `s3:GetObject` uses the object ARN

**SCPs in AWS Organizations**

SCPs define the maximum permissions available to accounts, OUs, and the root of the organization. They do not grant permissions. They only filter what identity policies and some resource-based access paths can ultimately do.

Key exam points:
- SCPs apply to IAM users and roles in member accounts.
- SCPs also affect access through assumed roles and federated sessions in those accounts.
- An SCP deny can block an action even if a resource policy allows it, because the principal’s effective permissions are still constrained by the organization guardrail.
- SCPs do not affect the management account in the same way they affect member accounts.

A common trap is assuming a resource policy alone can bypass an SCP. It cannot if the principal’s effective permissions are outside the SCP boundary.

**Session policies**

Session policies are optional inline policies passed when assuming a role or federating into a session. They reduce the permissions of the resulting session.

Use cases:
- Temporary least privilege for automation
- Narrowing access for a specific workload
- Restricting a federated session to a subset of the role’s permissions

Example `assume-role` with a session policy:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/AppRole \
  --role-session-name app-session \
  --policy '{
    "Version":"2012-10-17",
    "Statement":[
      {
        "Effect":"Allow",
        "Action":["s3:GetObject"],
        "Resource":["arn:aws:s3:::example-data-bucket/*"]
      }
    ]
  }'
```

The session can never exceed the intersection of:
- Role permissions
- Session policy
- Permissions boundary
- SCPs

**KMS authorization flow**

AWS Key Management Service (KMS) is a frequent SAP-C02 trap because authorization depends on more than one document.

The key policy is the primary authorization document for a KMS key, but IAM policies are not sufficient by themselves unless the key policy allows the account to use IAM permissions. In practice:
- The key policy must allow the caller directly, or
- The key policy must delegate to the account so IAM policies can authorize use

Common pattern:
- Key policy includes the account root principal or a statement that enables IAM policies in the account.
- IAM identity policy then grants `kms:Encrypt`, `kms:Decrypt`, `kms:GenerateDataKey`, or other actions.

Example key policy fragment that enables IAM policies in the account:

```json
{
  "Sid": "EnableIAMUserPermissions",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:root"
  },
  "Action": "kms:*",
  "Resource": "*"
}
```

KMS also supports **grants**, which are useful for temporary or service-integrated permissions. Grants can authorize specific operations without editing the key policy every time. In many service integrations, the service creates or uses grants on your behalf.

Practical KMS troubleshooting:
- Check the key policy first.
- Check whether IAM permissions are enabled by the key policy.
- Check grants if the service uses them.
- Check SCPs and session policies.
- Check encryption context conditions if present.

**S3 authorization edge cases**

S3 is another common exam area because authorization can involve multiple layers:
- IAM identity policy
- Bucket policy
- Object ACLs
- Object ownership settings
- SCPs and boundaries

Important nuances:
- `s3:ListBucket` applies to the bucket ARN.
- `s3:GetObject` and `s3:PutObject` apply to object ARNs.
- Bucket policies are often the cleanest way to grant cross-account access.
- Object ACLs are legacy and less preferred, but they still appear in exam scenarios.
- Object Ownership settings can disable ACL-based access patterns.

Cross-account direct access usually requires:
- An identity-based allow in the caller’s account
- A bucket policy that allows the caller principal
- No explicit deny in any applicable layer

Example cross-account bucket policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::444455556666:role/ExternalReader"
      },
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::example-data-bucket/*"
    }
  ]
}
```

**Lambda, SNS, and SQS cross-account patterns**

These services commonly use resource policies to authorize service-to-service or cross-account access.

- **Lambda** resource policies allow invocation by principals or services.
- **SNS** topic policies allow publish or subscribe permissions.
- **SQS** queue policies allow send or receive permissions from other accounts or services.

Typical patterns:
- A service principal such as `sns.amazonaws.com` or `events.amazonaws.com` is allowed in the resource policy.
- The calling service or principal must also satisfy any identity or trust requirements relevant to the integration.
- Cross-account invocation often requires both a resource policy and a caller-side permission or trust path.

Example Lambda permission for an S3 bucket to invoke a function:

```bash
aws lambda add-permission \
  --function-name ProcessUploads \
  --statement-id AllowS3Invoke \
  --action lambda:InvokeFunction \
  --principal s3.amazonaws.com \
  --source-arn arn:aws:s3:::example-upload-bucket
```

**Conditions and evaluation nuances**

Conditions are often the deciding factor in exam questions. A statement that looks like an allow may not match because the condition fails.

Common condition operators:
- `StringEquals`
- `StringLike`
- `ArnEquals`
- `Bool`
- `NumericEquals`
- `IpAddress`
- `DateGreaterThan`
- `ForAllValues`
- `ForAnyValue`
- `IfExists`

Important behavior:
- If a condition key is missing and the operator does not use `IfExists`, the statement usually does not match.
- `ForAllValues` requires every value in the request context to satisfy the condition.
- `ForAnyValue` requires at least one value to satisfy the condition.
- `IfExists` makes the condition tolerant of missing keys, which can be useful but also dangerous if misunderstood.

Example with `IfExists`:

```json
{
  "Effect": "Allow",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::example-data-bucket/*",
  "Condition": {
    "StringEqualsIfExists": {
      "s3:x-amz-server-side-encryption": "aws:kms"
    }
  }
}
```

This allows the request if the key is absent or if it equals `aws:kms`. That can be useful for optional headers, but it is not a strict enforcement control.

**Evaluation order for a typical request**

A practical end-to-end sequence helps with troubleshooting and exam reasoning:

1. Identify the principal type.
   - IAM user
   - IAM role
   - Assumed-role session
   - Federated session
   - AWS service principal

2. Identify the target resource and service.
   - Some services use resource policies heavily.
   - Some rely primarily on identity policies.
   - Some require both.

3. Check explicit denies.
   - Identity policy deny
   - Resource policy deny
   - Boundary deny by omission
   - SCP deny
   - Session policy deny
   - Service-specific deny

4. Check identity-based allows.
   - User, group, or role policy
   - Session policy intersection

5. Check resource-based allows.
   - Bucket policy, key policy, topic policy, queue policy, function policy

6. Apply boundaries and SCPs.
   - These are ceilings, not grants.

7. Apply service-specific authorization rules.
   - KMS key policy and grants
   - S3 object ownership and ACLs
   - Lambda invoke permissions
   - Trust policy for role assumption

**Worked authorization trace**

Scenario:
- An IAM role in Account A tries to read an S3 object in Account B.
- The role has an identity policy allowing `s3:GetObject`.
- The role has a permissions boundary allowing only `s3:GetObject` and `s3:ListBucket`.
- Account A is in an OU with an SCP that allows S3 read actions.
- The bucket policy in Account B allows the role ARN.
- A session policy attached at assume time denies `s3:GetObject` on one prefix.

Trace:
1. The principal is an assumed-role session.
2. The identity policy allows `s3:GetObject`.
3. The bucket policy allows the role principal.
4. The permissions boundary does not block the action.
5. The SCP does not block the action.
6. The session policy denies the specific prefix.
7. The request fails with an explicit deny.

This is a classic example of why the most restrictive applicable policy wins.

**AccessDenied vs implicit deny**

These two outcomes matter in troubleshooting and exam questions.

- **AccessDenied** usually means an explicit deny or a service-level authorization failure.
- **Implicit deny** means no applicable allow was found.

In practice, AWS often returns `AccessDenied` for both cases, so you must infer the cause from the policy set and the service behavior.

Useful debugging approach:
- Start with `aws iam simulate-principal-policy` for identity policy evaluation.
- Use `aws iam simulate-custom-policy` for isolated policy testing.
- Check CloudTrail for the exact denied action and resource.
- Inspect service-specific logs or error messages.

Example simulation:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::example-data-bucket/report.csv
```

Simulation limitations:
- It does not perfectly model every service-specific rule.
- It may not fully capture SCP effects unless you include the right context.
- It cannot always reproduce cross-account resource-policy behavior exactly as the live service does.
- It is a policy reasoning tool, not a full service emulator.

**Troubleshooting workflow**

1. Confirm the exact API action from CloudTrail.
2. Identify the principal ARN and whether it is a session ARN.
3. Check identity policies for an allow.
4. Check resource policies for an allow or deny.
5. Check permissions boundaries and session policies.
6. Check SCPs for the account or OU.
7. Check service-specific controls such as KMS key policy, S3 ACL/object ownership, or Lambda permission statements.
8. Re-test with policy simulation and a minimal reproduction.

**Common SAP-C02 traps**

- Assuming a resource policy always bypasses identity policy requirements.
- Forgetting that a permissions boundary is an upper bound, not a grant.
- Mixing bucket ARNs and object ARNs in S3 policy examples.
- Treating KMS IAM permissions as sufficient without key policy support.
- Ignoring session policies on assumed roles.
- Forgetting that SCPs can block access even when a resource policy allows it.
- Misreading `ImplicitDeny` as a service outage rather than a missing allow.
- Overlooking `IfExists` and missing condition keys.
- Confusing the role ARN with the assumed-role session ARN in resource policies.

**Quick revision checklist**

- **Explicit deny** always wins.
- **Implicit deny** means no allow matched.
- **Identity policies** grant permissions to principals.
- **Resource policies** grant access to resources and can support cross-account access.
- **Boundaries** and **SCPs** limit maximum permissions.
- **Session policies** narrow temporary credentials.
- **KMS** requires key policy awareness, not just IAM policy.
- **S3** requires correct ARN types and may involve object ownership rules.
- **Federation** depends on trust policy plus session permissions.

**Practical command set**

Inspect a role’s attached policies:

```bash
aws iam list-attached-role-policies --role-name AppRole
aws iam list-role-policies --role-name AppRole
```

Inspect a bucket policy:

```bash
aws s3api get-bucket-policy --bucket example-data-bucket
```

Inspect a KMS key policy:

```bash
aws kms get-key-policy \
  --key-id arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ef-abcd-1234567890ab \
  --policy-name default
```

Inspect an SCP attachment in Organizations:

```bash
aws organizations list-policies-for-target \
  --target-id ou-abcd-12345678 \
  --filter SERVICE_CONTROL_POLICY
```

Use the exact target ID for the account or OU you are investigating. The command shows policies attached to that target; it does not by itself explain the full effective permission path.

**Mental model for exam questions**

When you see an authorization question, ask:
- Who is the principal?
- Is the principal a user, role, session, or service?
- Which policy types apply?
- Is there any explicit deny?
- Is the resource policy granting access directly, or only complementing identity permissions?
- Are boundaries, SCPs, or session policies narrowing the result?
- Does the service have special rules such as KMS key policy, S3 object ownership, or Lambda invoke permissions?

If you answer those in order, you can usually derive the correct outcome without guessing.
