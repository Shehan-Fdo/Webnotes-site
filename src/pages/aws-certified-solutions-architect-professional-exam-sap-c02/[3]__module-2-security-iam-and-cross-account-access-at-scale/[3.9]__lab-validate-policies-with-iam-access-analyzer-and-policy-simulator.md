## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Purpose of the lab**

This lab focuses on two complementary tools for access control review in AWS: **IAM Access Analyzer** and the **IAM Policy Simulator**. You use them to catch policy mistakes before they become outages or security gaps.

Access Analyzer helps you inspect policy documents for findings such as overly broad access, missing conditions, or unintended external exposure. The Policy Simulator helps you test how AWS evaluates a specific request against identity-based policies, resource-based policies, permissions boundaries, and session policies. Together, they reduce guesswork when you design or troubleshoot authorization.

**What each tool is best for**

Use the tools for different questions:

- **IAM Access Analyzer** is best when you want to review a policy document for potential issues before deployment.
- **IAM Policy Simulator** is best when you want to test whether a specific principal can perform a specific action on a specific resource under a given set of policies.
- **Access Analyzer findings** are static analysis results; they do not prove that a request will succeed at runtime.
- **Policy simulation** is a model of authorization logic; it does not perfectly reproduce every service-specific edge case.

A good exam answer often depends on choosing the right tool for the question being asked.

**IAM policy evaluation model**

AWS authorization is usually the result of multiple policy layers working together. You should think in terms of the final decision path:

- An **identity-based policy** attached to a user, group, or role can allow or deny actions.
- A **resource-based policy** attached to the target resource can allow or deny access directly.
- A **permissions boundary** limits the maximum permissions a principal can receive.
- A **session policy** can further restrict temporary credentials.
- **Service control policies** (SCPs) in AWS Organizations can set account-level guardrails.
- A request is allowed only when the applicable policies permit it and no explicit deny overrides it.

This layered model matters because a policy may look correct in isolation but still fail due to another control.

**IAM Access Analyzer overview**

IAM Access Analyzer uses automated reasoning to identify access risks in IAM and resource-based policy documents. It is especially useful for spotting unintended access before you deploy a policy.

Common uses include:

- Reviewing a new policy for overly broad permissions.
- Checking whether a resource policy exposes access outside your account.
- Finding statements that allow public or cross-account access.
- Detecting missing or weak conditions in sensitive permissions.

Access Analyzer is not a universal validator for every AWS resource type. It analyzes supported policy documents and surfaces findings based on policy semantics and supported resource-policy contexts.

**Policy validation findings**

When Access Analyzer validates a policy, it can surface findings such as:

- **Security warnings** for risky access patterns.
- **Errors** for malformed or invalid policy elements.
- **Suggestions** for improvements, such as adding conditions or narrowing scope.

Typical issues include:

- Wildcard actions like `s3:*` where a narrower action set is safer.
- Wildcard resources like `"Resource": "*"` where a specific ARN is possible.
- Missing conditions on sensitive actions.
- Public or cross-account access in resource policies.
- Unsupported or ineffective policy structure for the intended use.

Treat findings as guidance, not as a substitute for understanding the business requirement.

**CLI validation workflow**

A common workflow is to validate a policy document before attaching it. For resource policies, you often need to provide the resource type context so the analyzer can interpret the policy correctly.

```bash
aws accessanalyzer validate-policy \
  --policy-document file://bucket-policy.json \
  --policy-type RESOURCE_POLICY \
  --validate-policy-resource-type AWS::S3::Bucket
```

For an identity policy, the command may look like this:

```bash
aws accessanalyzer validate-policy \
  --policy-document file://role-policy.json \
  --policy-type IDENTITY_POLICY
```

Use the output to review findings and decide whether to tighten the policy before deployment.

> The exact supported resource types and validation behavior depend on the policy type and AWS service. Do not assume every resource policy is validated the same way.

**Example resource policy review**

A bucket policy that grants public read access is often flagged as risky, even if another control later blocks actual access.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicRead",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

Why this matters:

- The policy is publicly readable by design.
- Access Analyzer may flag it as public exposure.
- For Amazon S3, **Block Public Access** can prevent public access from taking effect, but the policy can still be reported as public.
- You should verify both the policy intent and the effective service controls.

**IAM Policy Simulator overview**

The IAM Policy Simulator evaluates how AWS authorization logic applies to a specific request. It is useful when you need to answer questions like:

- Can this role call `ec2:RunInstances`?
- Does this user have access to `s3:GetObject` on this object ARN?
- Is a permissions boundary blocking an otherwise allowed action?
- Does a resource policy grant access even if the identity policy is missing an allow?

The simulator is powerful, but it is still a simulation. Some services, condition keys, and authorization behaviors are not modeled exactly the same as live AWS enforcement.

**When to use the simulator**

Use the simulator when you need to test a concrete access path:

- A user cannot access a resource and you need to isolate the cause.
- You want to compare the effect of an identity policy versus a permissions boundary.
- You need to confirm whether a resource policy grants access to a principal.
- You are validating least-privilege changes before rollout.

Use the simulator carefully when the service has complex authorization logic, service-linked roles, or condition keys that may not be fully represented.

**CLI simulation workflow**

The simulator can evaluate a principal policy against one or more actions and resources. Make sure each action is tested against the correct resource ARN.

For example, `s3:ListBucket` applies to the bucket ARN, while `s3:GetObject` applies to the object ARN.

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:ListBucket s3:GetObject \
  --resource-arns arn:aws:s3:::example-bucket arn:aws:s3:::example-bucket/* \
  --context-entries ContextKeyName=s3:prefix,ContextKeyType=string,ContextKeyValues=reports/
```

Practical notes:

- Use the bucket ARN for bucket-level actions.
- Use the object ARN pattern for object-level actions.
- Add context entries when the policy depends on conditions such as prefixes, tags, source IP, or MFA.
- Review both the decision and the matched statement to understand why AWS allowed or denied the request.

**Interpreting simulator output**

The simulator returns decisions such as:

- `allowed`
- `explicitDeny`
- `implicitDeny`

Interpretation:

- **Allowed** means the simulated request has at least one applicable allow and no explicit deny.
- **Explicit deny** means a deny statement overrides any allow.
- **Implicit deny** means no statement allowed the request.

A common troubleshooting pattern is to start with an implicit deny and then identify which policy layer is missing the allow.

**Identity policy versus resource policy**

Identity policies and resource policies solve different problems:

- **Identity policy**: attached to the caller, defines what the principal can do.
- **Resource policy**: attached to the target resource, defines who can access that resource.

Examples:

- An IAM role policy can allow `s3:GetObject`.
- An S3 bucket policy can allow a specific role from another account.
- A KMS key policy can allow a principal to use the key, but the caller may still need identity-based permission depending on the access path.

When you simulate access, always verify both sides of the authorization relationship.

**Cross-account access and trust policies**

For role assumption, the trust policy and the caller’s permissions both matter.

A role trust policy might allow a principal from another account:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::123456789012:root" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Important points:

- The trust policy says who is trusted to assume the role.
- The caller also needs permission to call `sts:AssumeRole` in its own identity policy in many scenarios.
- For same-account access, the caller generally needs an identity-based allow plus the trust relationship.
- For cross-account access, both the caller’s permissions and the role trust policy are involved.

Do not assume the trust policy alone is sufficient.

**Permissions boundaries and session policies**

The simulator is especially useful when access is unexpectedly restricted by higher-order controls.

- A **permissions boundary** caps the maximum permissions a principal can receive.
- A **session policy** can reduce permissions for temporary credentials obtained through STS (AWS Security Token Service).
- An allow in the identity policy does not override a boundary or session restriction.

Example pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "iam:PassRole",
      "Resource": "arn:aws:iam::123456789012:role/AppExecutionRole"
    }
  ]
}
```

If a permissions boundary omits `iam:PassRole`, the simulator should show the effective denial even though the identity policy allows it.

**Conditions you should test**

Many real-world policies depend on conditions. If you do not simulate the condition context, you may draw the wrong conclusion.

Common condition keys to test include:

- `aws:MultiFactorAuthPresent`
- `aws:SourceIp`
- `aws:PrincipalTag/*`
- `aws:RequestTag/*`
- `aws:ResourceTag/*`
- Service-specific keys such as `s3:prefix`

Example with MFA:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "iam:ChangePassword",
      "Resource": "*",
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

If you simulate without the MFA context, the request may appear denied even though the policy is correct for MFA-authenticated sessions.

**Practical validation pattern**

A reliable workflow is:

1. Write or update the policy.
2. Validate the policy document with Access Analyzer.
3. Review warnings, errors, and suggestions.
4. Simulate the exact request path with the Policy Simulator.
5. Use the correct resource ARN for each action.
6. Add condition context values that match the real request.
7. Confirm the result against the intended access model.
8. Deploy only after the policy behaves as expected.

This sequence helps you catch both syntax-level issues and authorization logic issues.

**Example troubleshooting scenario**

Suppose a developer says a role can list an S3 bucket but cannot read objects.

You might find:

- `s3:ListBucket` is allowed on `arn:aws:s3:::example-bucket`.
- `s3:GetObject` is missing on `arn:aws:s3:::example-bucket/*`.
- The bucket policy allows the role, but the identity policy does not.
- A permissions boundary blocks object read access.
- A condition on the bucket policy requires a specific VPC endpoint or source IP.

The simulator helps isolate which layer is responsible.

**Common exam traps**

Watch for these mistakes in SAP-C02 scenarios:

- Assuming a policy validator proves runtime access.
- Using the wrong ARN shape for the action being tested.
- Forgetting that explicit deny overrides allow.
- Ignoring permissions boundaries or SCPs.
- Treating a trust policy as the only requirement for role assumption.
- Assuming a public S3 policy always results in public access despite Block Public Access settings.
- Expecting the simulator to perfectly model every service-specific authorization nuance.

**Useful CLI examples**

Validate an identity policy:

```bash
aws accessanalyzer validate-policy \
  --policy-document file://policy.json \
  --policy-type IDENTITY_POLICY
```

Simulate a principal policy for an EC2 action:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/DevRole \
  --action-names ec2:StartInstances \
  --resource-arns arn:aws:ec2:us-east-1:123456789012:instance/i-0123456789abcdef0
```

Simulate access with a tag condition:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:user/Alice \
  --action-names ec2:TerminateInstances \
  --resource-arns arn:aws:ec2:us-east-1:123456789012:instance/i-0123456789abcdef0 \
  --context-entries ContextKeyName=aws:ResourceTag/Environment,ContextKeyType=string,ContextKeyValues=Dev
```

**Best practices**

- Validate policies before deployment.
- Simulate the exact action and resource combination you expect in production.
- Include condition context when policies depend on tags, MFA, IP ranges, or prefixes.
- Review both identity and resource policies for cross-account access.
- Use explicit denies sparingly and intentionally.
- Keep resource ARNs as specific as possible.
- Re-test after any policy, boundary, or SCP change.

**Quick comparison**

| Tool | Best use | Strength | Limitation |
|---|---|---|---|
| IAM Access Analyzer | Static policy review | Finds risky or malformed policy patterns | Does not prove runtime authorization |
| IAM Policy Simulator | Request-level authorization testing | Shows effective allow/deny behavior | Does not perfectly model every service edge case |

**Key takeaways**

- Access Analyzer helps you validate policy documents and identify risky access patterns.
- The Policy Simulator helps you test how AWS evaluates a specific request.
- Use the correct ARN for each action, especially for services like S3.
- Cross-account role assumption involves both trust policy and caller permissions.
- Effective authorization can be blocked by boundaries, session policies, or SCPs even when a policy appears to allow access.
- For S3, Block Public Access can prevent public exposure, but Access Analyzer may still flag the policy as public.
