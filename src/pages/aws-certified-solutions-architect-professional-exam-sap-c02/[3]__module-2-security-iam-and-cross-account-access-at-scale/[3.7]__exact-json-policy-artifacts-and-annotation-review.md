## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Policy artifact review: what you are actually validating**

On SAP-C02, you often need to inspect exact JSON policy artifacts rather than rely on a high-level summary. The exam expects you to reason about how AWS evaluates identity policies, resource policies, trust policies, SCPs, permission boundaries, session policies, and service-specific controls.

A useful mental model is:

- **Identity-based policies** grant permissions to IAM users, groups, and roles.
- **Resource-based policies** attach to resources and can grant access directly to principals.
- **Trust policies** define who can assume a role.
- **SCPs** (Service Control Policies) set organization-level guardrails.
- **Permission boundaries** cap the maximum permissions an identity can receive.
- **Session policies** further restrict temporary credentials.

The key exam skill is not memorizing one linear order. It is understanding which policy types can grant, which can only restrict, and which are evaluated together to produce the final authorization decision.

**Core JSON grammar and policy structure**

Most AWS policies use a common JSON shape with `Version` and `Statement`. Each statement can contain one or more of the following:

- `Effect`
- `Action` or `NotAction`
- `Resource` or `NotResource`
- `Principal` or `NotPrincipal`
- `Condition`
- `Sid`

A minimal identity policy looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowListBuckets",
      "Effect": "Allow",
      "Action": "s3:ListAllMyBuckets",
      "Resource": "*"
    }
  ]
}
```

Important grammar details:

- `Action` and `Resource` are the most common positive-match fields.
- `NotAction` means “all actions except these,” but it is only meaningful in the context of the statement’s `Resource` scope and `Effect`.
- `NotResource` means “all resources except these,” and it is often used in deny statements.
- `Principal` appears in resource-based policies and trust policies, not in standard identity-based permission policies.
- `NotPrincipal` is supported in some resource-based policies, but it is easy to misuse and can create broad unintended access or denial patterns.
- `Condition` narrows when a statement applies; it does not replace `Action` or `Resource`.

> Be careful with `NotAction` and `NotResource`. They are powerful, but they are also easy to overgrant or overdeny if you do not fully understand the scope of the statement.

**How multiple statements combine inside one policy**

Within a single policy document, AWS evaluates all statements that match the request context.

- Any matching **explicit Deny** wins over matching allows.
- Multiple matching **Allow** statements are additive.
- A statement that does not match the request has no effect.
- A `Deny` statement can be very broad if it uses wildcards or `NotAction`.

Example with both allow and deny:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Read",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    },
    {
      "Sid": "DenySensitivePrefix",
      "Effect": "Deny",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/private/*"
    }
  ]
}
```

In this example:

- Reads are allowed generally.
- Reads under `private/` are denied.
- The deny statement overrides the allow statement for matching objects.

**Identity policies, resource policies, and trust policies**

These three policy families are often confused, but they serve different purposes.

- **Identity policy**
  - Attached to a user, group, or role.
  - Answers: “What can this identity do?”
- **Resource policy**
  - Attached to a resource such as an S3 bucket, KMS key, SNS topic, SQS queue, ECR repository, Secrets Manager secret, EventBridge bus, API Gateway API, or some Lambda functions.
  - Answers: “Who can access this resource?”
- **Trust policy**
  - Attached to an IAM role.
  - Answers: “Who can assume this role?”

A trust policy is not the same as a permission policy. It does not grant the role permissions to call AWS APIs after assumption; it only controls role assumption.

Example trust policy for a cross-account role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountAssumeRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalArn": "arn:aws:iam::111122223333:role/ExternalAuditor"
        }
      }
    }
  ]
}
```

Practical notes:

- `sts:AssumeRole` is the key action in a role trust policy.
- `Principal` identifies the caller or caller set allowed to assume the role.
- Conditions can narrow trust to a specific role, external ID, source account, or MFA requirement.
- For service principals, the `Principal` may be a service such as `ec2.amazonaws.com` or `lambda.amazonaws.com`.

**Trust policy patterns you should recognize**

Common trust policy scenarios on SAP-C02 include:

- **Cross-account access**
  - One account’s role trusts another account’s role or account root.
- **Service-linked roles**
  - AWS creates and manages the trust relationship for a service.
- **Federation**
  - A SAML or OIDC identity provider assumes a role through `sts:AssumeRoleWithSAML` or `sts:AssumeRoleWithWebIdentity`.
- **MFA-gated assumption**
  - The trust policy requires `aws:MultiFactorAuthPresent` to be `true`.

Example MFA condition in a trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "RequireMFA",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

**Policy evaluation: the accurate mental model**

AWS authorization is not a single linear checklist. A better model is:

- Some policy types act as **filters on maximum permissions**:
  - SCPs
  - Permission boundaries
  - Session policies
- Some policy types can **grant permissions**:
  - Identity policies
  - Resource policies
  - Some trust policies for role assumption
- Some policy types can **deny** and override matching allows:
  - Explicit deny statements in identity, resource, SCP, boundary, or session policy contexts

A request is allowed only when the relevant policy layers permit it and no applicable explicit deny blocks it.

Key exam-safe points:

- An SCP does not grant permissions; it limits what accounts and principals can do within the organization.
- A permission boundary does not grant permissions; it caps what an identity policy can grant.
- A session policy restricts the temporary session created by STS (Security Token Service).
- A resource policy can sometimes grant access even when the principal lacks an identity-based allow, depending on the service and scenario.
- A resource policy cannot bypass an SCP or a permission boundary when those controls apply to the principal’s effective permissions.

> Do not describe AWS authorization as “SCP first, then boundary, then identity, then resource” in a rigid sequence. That oversimplifies how AWS combines policy types and can lead to wrong exam answers.

**SCPs in AWS Organizations**

SCPs apply at the organization root, OU (organizational unit), or account level. They define the maximum available permissions for accounts and principals in those accounts.

Important SCP behaviors:

- SCPs are inherited from parent containers to child OUs and accounts.
- An SCP can be used as:
  - a deny list, where you allow everything except explicit denies, or
  - a whitelist model, where only explicitly allowed actions are available, depending on how the SCPs are structured in the organization.
- SCPs do not grant permissions by themselves.
- The management account has special behavior and is not governed the same way as member accounts for all operations.

Example SCP deny:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyLeavingOrg",
      "Effect": "Deny",
      "Action": "organizations:LeaveOrganization",
      "Resource": "*"
    }
  ]
}
```

Exam implications:

- If a user has an IAM allow but the SCP denies the action, the action fails.
- If an SCP does not allow an action in a whitelist-style setup, the action is unavailable even if IAM allows it.
- SCPs are common in multi-account governance questions.

**Permission boundaries**

A permission boundary is an advanced IAM feature that sets the maximum permissions an IAM user or role can receive from identity-based policies.

Use cases:

- Delegating IAM administration safely.
- Allowing teams to create roles/users without exceeding a controlled permission set.
- Preventing privilege escalation through overly broad identity policies.

Example boundary policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "BoundaryAllowReadOnly",
      "Effect": "Allow",
      "Action": [
        "ec2:Describe*",
        "s3:Get*",
        "s3:List*"
      ],
      "Resource": "*"
    }
  ]
}
```

Practical notes:

- The boundary does not grant access by itself.
- The identity policy must still allow the action.
- The final permission is the intersection of the identity policy and the boundary.
- A boundary is often used with delegated admin patterns.

**Session policies and STS**

Session policies are passed when you call STS to create temporary credentials. They further restrict the permissions of the session.

Example with `aws sts assume-role`:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/ReadOnlyOps \
  --role-session-name audit-session
```

You can also pass a session policy to narrow the session:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/ReadOnlyOps \
  --role-session-name audit-session \
  --policy '{
    "Version":"2012-10-17",
    "Statement":[
      {
        "Effect":"Allow",
        "Action":["s3:ListBucket"],
        "Resource":"arn:aws:s3:::example-bucket"
      }
    ]
  }'
```

Important:

- Session policies restrict the session further.
- They do not expand permissions beyond the role’s base permissions.
- They are useful for temporary least-privilege access.

**Condition keys and namespaces**

Condition keys are critical in exam scenarios because they often determine whether a policy is safe enough or too broad.

Common global condition keys include:

- `aws:PrincipalArn`
- `aws:SourceAccount`
- `aws:SourceArn`
- `aws:SourceVpce`
- `aws:RequestedRegion`
- `aws:MultiFactorAuthPresent`
- `aws:PrincipalOrgID`

Examples of use:

- `aws:PrincipalArn`
  - Restricts access to a specific IAM principal ARN.
- `aws:SourceAccount`
  - Helps prevent confused deputy problems in service-to-service access.
- `aws:SourceArn`
  - Narrows a service invocation to a specific source resource.
- `aws:SourceVpce`
  - Restricts access to traffic coming through a specific VPC endpoint.
- `aws:RequestedRegion`
  - Limits actions to a region.
- `aws:MultiFactorAuthPresent`
  - Requires MFA for sensitive operations.
- `aws:PrincipalOrgID`
  - Allows only principals from your AWS Organization.

Example bucket policy using multiple global keys:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowFromSpecificVPCEAndOrg",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*",
      "Condition": {
        "StringEquals": {
          "aws:SourceVpce": "vpce-0123456789abcdef0",
          "aws:PrincipalOrgID": "o-abcd1234"
        }
      }
    }
  ]
}
```

**Service-specific condition keys**

Many services define their own condition key namespaces. These are often tested indirectly through policy snippets.

Examples:

- `s3:prefix`
- `s3:delimiter`
- `kms:ViaService`
- `kms:EncryptionContext:context-key`
- `secretsmanager:ResourceTag/tag-key`
- `events:source`
- `apigateway:Request/Path`
- `ecr:ResourceTag/tag-key`

Use service-specific keys when the service exposes a control that global keys cannot express.

**KMS key policies and grants**

AWS Key Management Service (KMS) is a frequent source of exam confusion because access can involve key policies, IAM policies, and grants.

Important nuances:

- A KMS key policy is always central to authorization for the key.
- Many accounts use the default key policy pattern that enables IAM policies in the account to control access, often by allowing the account root and delegating through IAM.
- In some cases, a key policy alone can authorize access.
- In other cases, IAM permissions are also needed, especially when the key policy delegates to IAM.
- AWS services may use grants or service-integrated permissions, which can change the exact access path.

Example of a common default-style key policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableIAMUserPermissions",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": "kms:*",
      "Resource": "*"
    }
  ]
}
```

Example KMS usage with a service condition:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowUseViaS3Only",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/AppRole"
      },
      "Action": [
        "kms:Encrypt",
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "kms:ViaService": "s3.us-east-1.amazonaws.com"
        }
      }
    }
  ]
}
```

Practical exam guidance:

- Check whether the key policy delegates to IAM.
- Check whether the service uses a grant or a service principal path.
- Check whether the request is direct KMS API usage or indirect through another AWS service.
- Do not assume every KMS scenario requires a direct IAM allow in the same way.

**Policy size, formatting, and attachment limits**

Real-world policy artifacts have limits that can matter in architecture questions.

Common constraints to remember:

- JSON must be syntactically valid.
- Duplicate keys are not well-defined in practice and can produce ambiguous behavior in policy tooling and parsers.
- Policies have size limits that vary by policy type and service.
- IAM managed policies, inline policies, trust policies, SCPs, and resource policies each have their own size and attachment constraints.
- Large organizations often hit limits when they overuse explicit statements instead of reusable patterns.

Practical implications:

- Prefer groups, roles, and managed policies for reuse.
- Use conditions and wildcards carefully to reduce policy sprawl.
- Split responsibilities across multiple policies when a single document becomes too large.
- Validate policy size before deployment in automation pipelines.

**Managed policies vs inline policies**

AWS supports multiple policy artifact types for IAM permissions.

| Type | Reusable | Attached to multiple identities | Typical use |
|---|---:|---:|---|
| AWS managed policy | Yes | Yes | Common baseline permissions maintained by AWS |
| Customer managed policy | Yes | Yes | Organization-specific reusable permissions |
| Inline policy | No | No | Identity-specific one-off permissions |

Key differences:

- **AWS managed policies**
  - Created and maintained by AWS.
  - Useful for standard job functions.
- **Customer managed policies**
  - Created by you.
  - Best for reusable least-privilege patterns.
- **Inline policies**
  - Embedded directly in a user, group, or role.
  - Useful for tightly scoped, non-reusable permissions.
  - Harder to audit at scale.

Exam tip:

- If a question asks for centralized governance and reuse, prefer customer managed policies.
- If a question asks for a one-off permission tied to a single role, inline policies may be acceptable.
- If a question asks for AWS-maintained baseline permissions, consider AWS managed policies.

**Annotation workflow for policy review**

When you review a policy artifact, annotate it line by line so you can explain intent, scope, and risk.

A practical workflow:

1. Identify the policy type.
2. Identify the principal, if any.
3. Identify the action set.
4. Identify the resource scope.
5. Check for `Condition` keys.
6. Look for `Allow` and `Deny` interactions.
7. Check for wildcard risk.
8. Check whether the policy grants, restricts, or only delegates trust.
9. Compare the policy’s intent to the service’s actual authorization model.

Example annotated identity policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlyS3Inventory",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::inventory-bucket",
        "arn:aws:s3:::inventory-bucket/*"
      ]
    }
  ]
}
```

Annotation:

- `Version`: policy language version.
- `Sid`: human-readable statement identifier.
- `Effect`: grants access.
- `Action`: read operations only.
- `Resource`: bucket and objects, which is required because `ListBucket` targets the bucket ARN while `GetObject` targets object ARNs.

**Worked example: cross-account role assumption**

Suppose Account A needs to let Account B read an S3 bucket through a role.

Trust policy on the role in Account A:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "TrustAccountBRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::222233334444:role/ReaderRole"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Permissions policy attached to the role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::shared-data",
        "arn:aws:s3:::shared-data/*"
      ]
    }
  ]
}
```

What to verify:

- The trust policy allows the external principal to assume the role.
- The role permissions allow the S3 actions.
- If the bucket has a bucket policy, it must also permit access if the service path requires it.
- If an SCP or permission boundary blocks the action, the request still fails.

**Worked example: resource policy with confused deputy protection**

A common pattern is to allow a service to access a resource only when the request comes from a specific source account and source ARN.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowServiceFromSpecificAccount",
      "Effect": "Allow",
      "Principal": {
        "Service": "events.amazonaws.com"
      },
      "Action": "sqs:SendMessage",
      "Resource": "arn:aws:sqs:us-east-1:123456789012:my-queue",
      "Condition": {
        "StringEquals": {
          "aws:SourceAccount": "123456789012"
        },
        "ArnLike": {
          "aws:SourceArn": "arn:aws:events:us-east-1:123456789012:rule/MyRule"
        }
      }
    }
  ]
}
```

Why this matters:

- `aws:SourceAccount` reduces the risk of another account abusing the permission.
- `aws:SourceArn` ties the permission to a specific EventBridge rule.
- This pattern is common in service-to-service integrations.

**CLI tools for policy inspection and validation**

Use AWS CLI and IAM tooling to inspect artifacts and test assumptions.

Useful commands:

```bash
aws iam get-policy --policy-arn arn:aws:iam::aws:policy/ReadOnlyAccess
aws iam get-policy-version --policy-arn arn:aws:iam::123456789012:policy/CustomReadOnly --version-id v1
aws iam get-role --role-name AppRole
aws iam list-attached-role-policies --role-name AppRole
aws iam list-role-policies --role-name AppRole
aws organizations list-policies --filter SERVICE_CONTROL_POLICY
aws organizations describe-policy --policy-id p-abc123
aws sts get-caller-identity
```

Policy simulation can help with identity policy reasoning:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject s3:DeleteObject \
  --resource-arns arn:aws:s3:::example-bucket/private/file.txt
```

Important caveats:

- Simulation is useful for identity-based policy evaluation.
- It does not fully replace service-specific authorization behavior.
- It is not a complete end-to-end checker for every combination of SCPs, boundaries, resource policies, and service integrations.

For policy syntax and some semantic checks, you can use:

```bash
aws accessanalyzer validate-policy --policy-document file://policy.json --policy-type IDENTITY_POLICY
```

Use it to catch common issues in identity and resource policies, but do not treat it as a universal validator for SCPs, permission boundaries, or every service-specific authorization path.

**Common exam traps**

- Confusing a trust policy with a permissions policy.
- Assuming a resource policy always requires a matching identity policy.
- Assuming a resource policy can override an SCP or permission boundary.
- Treating `Allow` as sufficient even when an explicit `Deny` exists elsewhere.
- Forgetting that `ListBucket` and `GetObject` use different resource ARNs.
- Using `Principal` in an identity policy where it is not valid.
- Ignoring `Condition` keys that narrow or completely block access.
- Overlooking that some services require both service-side resource policy support and IAM permissions.
- Assuming all JSON parsers treat duplicate keys the same way; AWS policy behavior can become ambiguous, so avoid duplicates entirely.

**Quick comparison of policy types**

| Policy type | Attached to | Grants access | Restricts access | Key exam use |
|---|---|---:|---:|---|
| Identity policy | User, group, role | Yes | Yes, via deny | What can this principal do? |
| Resource policy | Resource | Yes | Yes, via deny | Who can access this resource? |
| Trust policy | Role | No, for API use | Yes | Who can assume this role? |
| SCP | Root, OU, account | No | Yes | What is the org maximum? |
| Permission boundary | User, role | No | Yes | What is the identity maximum? |
| Session policy | STS session | No | Yes | What is the temporary session maximum? |

**Practical review checklist**

Before you answer a policy question, check:

- Which policy type is shown?
- Is the policy granting access or only allowing assumption?
- Are there any explicit denies?
- Are `Action` and `Resource` correctly paired?
- Are `Principal` and `NotPrincipal` used in the right policy type?
- Are there conditions that depend on source account, VPC endpoint, region, or MFA?
- Is the access cross-account or service-integrated?
- Could an SCP or permission boundary still block the request?
- Does the service use a resource policy, a grant, or both?
- Is the policy artifact syntactically valid and within size constraints?

**Why annotation matters on SAP-C02**

The exam often gives you a policy snippet and asks what happens next. Annotation helps you translate JSON into authorization behavior quickly.

A strong annotation should tell you:

- who is trusted,
- what action is allowed or denied,
- which resource is in scope,
- which conditions narrow the request,
- and what other policy layers might still block the call.

If you can explain those five points from the JSON alone, you are usually ready for the policy-heavy questions on the exam.
