## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Cross-account access overview**

Cross-account access in AWS usually means a principal in one account uses <abbr title="Security Token Service">STS</abbr> to assume a role in another account. The target account owns the role, and the source account owns the caller identity. This pattern is preferred over sharing long-term credentials because it gives you temporary credentials, centralized control, and easier revocation.

The core idea is simple:

- The **trust policy** on the target role says who can assume it.
- The **identity policy** on the caller says whether it can call `sts:AssumeRole`.
- The resulting **session** gets temporary permissions from the role, then is further constrained by any session policy and permissions boundary that apply.

For SAP-C02, you should be able to reason about which policy authorizes which step, especially when access spans multiple accounts and services such as Amazon S3, AWS KMS, and AWS Organizations.

**Trust policy fundamentals**

A trust policy is a resource-based policy attached to an IAM role. It does not grant permissions to perform AWS actions directly; it only controls who can obtain the role session.

A minimal trust policy for a role in Account B that allows a role in Account A to assume it looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowAccountARoleToAssume",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/SourceAppRole"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Important points:

- **Principal** identifies the trusted caller.
- **Action** is usually `sts:AssumeRole` for IAM roles.
- **Condition** often narrows trust with `aws:PrincipalArn`, `aws:PrincipalOrgID`, `sts:ExternalId`, or MFA-related keys.
- A trust policy can trust an entire account root ARN, but that is broader than trusting a specific role and is usually paired with conditions.

> Be careful with `arn:aws:iam::<account-id>:root`. It does not mean the root user alone; it generally represents principals in that account that can satisfy the trust relationship, subject to their own permissions and any conditions you add.

**Identity policy versus resource policy**

Cross-account access often works through either an identity-based policy, a resource-based policy, or both. For `sts:AssumeRole`, the caller typically needs an identity-based permission allowing the assume-role call, while the target role’s trust policy must allow the caller.

For other services, the authorization model can differ:

- **S3** can authorize through bucket policies, access point policies, or identity policies.
- **KMS** requires both IAM authorization and key policy or grant authorization for many operations.
- **Lambda**, **SNS**, and **SQS** also use resource policies for cross-account access.

A useful exam rule:

- If the action is “assume a role,” think **trust policy + caller permission**.
- If the action is “access a resource directly,” think **resource policy and/or identity policy**, depending on the service.

**Implementing a basic cross-account role assumption**

A common lab flow is:

1. Create a role in the target account.
2. Add a trust policy that names the source principal.
3. Attach permissions to the target role for the resources it should access.
4. Allow the source principal to call `sts:AssumeRole`.
5. Test with the AWS CLI using temporary credentials.

Example target role permissions policy in Account B for S3 read access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadTargetBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::target-bucket",
        "arn:aws:s3:::target-bucket/*"
      ]
    }
  ]
}
```

Example source-side permission for an IAM role in Account A:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowAssumeTargetRole",
      "Effect": "Allow",
      "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::444455556666:role/TargetReadRole"
    }
  ]
}
```

Then assume the role:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/TargetReadRole \
  --role-session-name cross-account-lab
```

The response includes temporary credentials:

```json
{
  "Credentials": {
    "AccessKeyId": "ASIA...",
    "SecretAccessKey": "...",
    "SessionToken": "...",
    "Expiration": "2026-04-26T18:30:00Z"
  }
}
```

Export them into a shell session:

```bash
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
aws sts get-caller-identity
```

**Using the AWS CLI to verify access**

After assuming the role, verify the effective identity and test the target service.

```bash
aws sts get-caller-identity
aws s3 ls s3://target-bucket
aws s3 cp ./report.csv s3://target-bucket/reports/report.csv
```

If access fails, check the failure point in order:

- The source principal lacks `sts:AssumeRole`.
- The target role trust policy does not trust the source principal.
- The target role permissions do not allow the downstream service action.
- A permissions boundary, session policy, SCP, or resource policy blocks the request.
- The service requires additional permissions, such as KMS permissions for encrypted S3 objects.

**S3 cross-account access patterns**

For Amazon S3, cross-account access can be granted by a bucket policy, by assuming a role that has S3 permissions, or by both. In many production designs, you use both for least privilege and clearer audit boundaries.

A bucket policy can grant access directly to a role from another account:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSourceRoleRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/SourceAppRole"
      },
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::target-bucket",
        "arn:aws:s3:::target-bucket/*"
      ]
    }
  ]
}
```

Key S3 points:

- A resource-based policy can authorize access even when the caller’s identity policy is minimal, but the exact authorization path depends on the action and service behavior.
- For least privilege, you often still attach an identity policy to the caller or use an assumed role so you can scope access tightly and audit the session.
- Bucket policies are especially useful when the bucket owner wants to control access centrally.

If the objects are encrypted with SSE-KMS, S3 access alone is not enough.

**KMS permissions with SSE-KMS**

When Amazon S3 objects use <abbr title="Server-Side Encryption with AWS Key Management Service">SSE-KMS</abbr>, the caller needs permission to use the KMS key in addition to S3 permissions. The exact KMS actions depend on the operation:

- Reading encrypted objects commonly requires `kms:Decrypt` and often `kms:DescribeKey`.
- Writing encrypted objects commonly requires `kms:Encrypt`, `kms:GenerateDataKey*`, and often `kms:DescribeKey`.
- Listing buckets does not usually require KMS permissions, but object retrieval does.

Example KMS key policy statement allowing a cross-account role to use the key:

```json
{
  "Sid": "AllowCrossAccountUseForS3",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::111122223333:role/SourceAppRole"
  },
  "Action": [
    "kms:Decrypt",
    "kms:Encrypt",
    "kms:GenerateDataKey*",
    "kms:DescribeKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:ViaService": "s3.us-east-1.amazonaws.com"
    }
  }
}
```

Practical guidance:

- Use `kms:ViaService` to limit key use to S3 when appropriate.
- Keep the key policy and IAM policy aligned.
- Remember that KMS authorization is often the hidden cause of `AccessDenied` on encrypted S3 objects.

**Session policies and permissions boundaries**

A session policy is an optional policy passed when assuming a role. It can only reduce the permissions available to the session. The effective permissions are the intersection of:

- the role’s identity-based permissions,
- the session policy,
- any permissions boundary on the role,
- and any applicable SCPs or resource policies.

This matters in advanced SAP-C02 scenarios because a role may appear to have access, but the session still fails due to a boundary or session policy.

Example assume-role call with a session policy:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/TargetReadRole \
  --role-session-name cross-account-lab \
  --policy '{
    "Version":"2012-10-17",
    "Statement":[
      {
        "Effect":"Allow",
        "Action":"s3:GetObject",
        "Resource":"arn:aws:s3:::target-bucket/reports/*"
      }
    ]
  }'
```

Use session policies when you want:

- temporary, task-specific restriction,
- narrower access than the base role,
- safer delegation for automation or break-glass workflows.

**External ID and confused deputy protection**

When a third party or another account assumes your role, use `sts:ExternalId` to reduce confused deputy risk. This is especially important when a SaaS provider or shared automation account assumes roles in many customer accounts.

Example trust policy condition:

```json
{
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::111122223333:role/PartnerAccessRole"
  },
  "Action": "sts:AssumeRole",
  "Condition": {
    "StringEquals": {
      "sts:ExternalId": "customer-12345"
    }
  }
}
```

Use `ExternalId` when:

- a third party manages access on your behalf,
- multiple customers share the same external principal,
- you need a unique token to bind the trust relationship to your account.

**Trust policy conditions for tighter control**

You can reduce trust scope with conditions instead of trusting a broad principal alone.

Common condition keys:

- `aws:PrincipalArn` to match a specific role or user pattern.
- `aws:PrincipalOrgID` to trust principals only from your AWS Organizations organization.
- `aws:MultiFactorAuthPresent` to require MFA for human access.
- `sts:ExternalId` for third-party delegation.
- `aws:SourceArn` and `aws:SourceAccount` for service principals in service-to-service trust.

Example trust policy that allows only principals from a specific organization:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOrgPrincipals",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalOrgID": "o-abcd1234"
        }
      }
    }
  ]
}
```

This pattern is useful when you want to trust many accounts in your organization without enumerating each one.

**Cross-account access for IAM users, roles, and federated identities**

The source principal may be an IAM user, IAM role, or federated identity. The mechanism to grant `sts:AssumeRole` differs slightly:

- **IAM user**: attach an identity policy directly to the user.
- **IAM role**: attach the permission to the role that the workload assumes.
- **Federated identity**: map the federated session to a role or use identity center / federation flow, then allow that role to assume the target role.

Example IAM user policy:

```bash
aws iam put-user-policy \
  --user-name Alice \
  --policy-name AllowAssumeTargetRole \
  --policy-document file://assume-role-policy.json
```

If the caller is not an IAM user, use the appropriate attachment mechanism:

- `aws iam put-role-policy` for an IAM role inline policy,
- `aws iam attach-role-policy` for a managed policy,
- federation or identity center configuration for external identities.

**Troubleshooting access denied errors**

When cross-account access fails, isolate the layer that denied the request.

1. Confirm the caller identity with `aws sts get-caller-identity`.
2. Verify the source principal has `sts:AssumeRole`.
3. Check the target role trust policy principal and conditions.
4. Confirm the target role permissions allow the downstream action.
5. Check for SCPs, permissions boundaries, session policies, and resource policies.
6. For encrypted resources, verify KMS permissions and key policy or grants.

Useful diagnostics:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::111122223333:role/SourceAppRole \
  --action-names sts:AssumeRole \
  --resource-arns arn:aws:iam::444455556666:role/TargetReadRole
```

For role sessions, CloudTrail is often the best source of truth. Look for:

- `AssumeRole`
- `AccessDenied`
- the `principalId` and `sessionContext`
- the target resource ARN

**CloudTrail and audit considerations**

Cross-account role assumption is highly auditable. CloudTrail records both the assume-role event and the downstream API calls made by the temporary session.

What to look for:

- The `AssumeRole` event in the target account.
- The `userIdentity.type` value, often `AssumedRole`.
- The session name, which helps identify the workload or operator.
- The source IP and MFA context when relevant.

Good session naming improves traceability:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/TargetReadRole \
  --role-session-name alice-lab-20260426
```

**Common lab design patterns**

- **Hub-and-spoke access**: a central security or tooling account assumes roles in workload accounts.
- **Third-party delegation**: an external account assumes a role with `ExternalId`.
- **Organization-wide trust**: a role trusts principals from the same AWS Organizations org using `aws:PrincipalOrgID`.
- **Service-to-service access**: a service principal assumes or uses a role with `aws:SourceArn` and `aws:SourceAccount` conditions.

**Exam-focused distinctions**

- **Trust policy** answers: “Who can assume this role?”
- **Permissions policy** answers: “What can the role do after assumption?”
- **Session policy** answers: “What subset of the role’s permissions should this session get?”
- **Permissions boundary** answers: “What is the maximum this role can ever do?”
- **SCP** answers: “What is the maximum any principal in this account can do?”

**Practical checklist for the lab**

1. Create the target role in the destination account.
2. Add a trust policy for the source principal.
3. Attach least-privilege permissions to the target role.
4. Grant the source principal permission to call `sts:AssumeRole`.
5. Test the role assumption with the AWS CLI.
6. Validate downstream access to S3, KMS, or another target service.
7. Review CloudTrail for the assume-role event and any denied calls.
8. Tighten conditions with `ExternalId`, `PrincipalOrgID`, or MFA if needed.

**Key takeaways**

Cross-account access is usually a two-sided authorization problem: the caller must be allowed to ask for the role, and the target role must trust the caller. After assumption, the session is constrained by the role’s permissions and any additional limits such as session policies, permissions boundaries, SCPs, and resource policies.

For SAP-C02, pay special attention to S3 and KMS interactions, because many “role assumption” problems are actually downstream service authorization problems. End of Notes
