## **Course:** AWS Certified Solutions Architect – Professional (Exam SAP-C02)

**Enterprise least privilege model**

At enterprise scale, least privilege is not a single policy type. You combine identity-based policies, resource-based policies, permission boundaries, session policies, and AWS Organizations service control policies (SCPs) to shape what a principal can do. The exam often tests whether you can choose the right control for the right layer.

A useful mental model is:

- **Identity policies** grant permissions to an IAM user, group, or role.
- **Resource policies** grant access to a resource, such as an S3 bucket or KMS key.
- **Permission boundaries** cap the maximum permissions that an IAM user or role can receive from identity-based policies.
- **Session policies** further restrict permissions for a temporary session.
- **SCPs** set the maximum permissions available in an account or organizational unit (OU) within AWS Organizations.

The effective permission is the result of the applicable policies together, with explicit denies overriding allows. Do not think of this as a simple fixed sequence; think of it as an intersection of constraints.

**IAM policy types and how they interact**

Identity-based policies are attached to IAM users, groups, and roles. They answer: “What can this principal do if nothing else restricts it?”

Resource-based policies are attached to the target resource. They answer: “Who can access this resource, and under what conditions?”

Permission boundaries are attached to IAM users and IAM roles, not groups. They are especially useful when you delegate IAM administration to teams but want to prevent privilege escalation. A boundary does not grant permissions by itself; it only limits what identity-based policies can grant.

Session policies are passed when a role is assumed, or when temporary credentials are issued through federation. They are useful when you want to narrow a session for a specific task without changing the role’s base policy.

SCPs apply only in AWS Organizations and only to accounts that are members of the organization. They do not apply to the management account. SCPs do not grant permissions; they only define the outer ceiling for what identities in member accounts can do.

**Permission boundaries in practice**

Permission boundaries are a common exam topic because they solve a specific delegation problem: you want developers or platform teams to create and manage roles, but you do not want them to create roles that can exceed a safe permission envelope.

Typical use cases:

- Delegating IAM role creation to application teams.
- Preventing self-service administrators from attaching `AdministratorAccess`.
- Allowing a team to create roles for Lambda, ECS, or EC2 while restricting the role to approved services and actions.
- Enforcing a standard maximum permission set across many accounts.

A boundary is evaluated with the identity policy. If the identity policy allows `s3:PutObject` but the boundary does not, the action is denied. If the boundary allows it but the identity policy does not, the action is still denied.

For IAM users, boundaries are useful when the user directly performs actions with long-term credentials. For IAM roles, boundaries are useful when the role is assumed by humans, applications, or AWS services. In both cases, the boundary constrains permissions granted through identity-based policies.

> A permission boundary does not replace a trust policy. A trust policy controls who can assume a role; a boundary controls what the role can do after it is assumed.

**Creating roles with a boundary**

A common exam trap is confusing `iam:CreateRole` with `iam:PassRole`.

- Use `iam:CreateRole` with the `iam:PermissionsBoundary` condition key to require that any newly created role includes an approved boundary.
- Use `iam:PassRole` when a principal needs to pass an existing role to a service such as Lambda, EC2, or ECS.

Example policy that allows role creation only when a specific boundary is attached:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCreateRoleOnlyWithBoundary",
      "Effect": "Allow",
      "Action": "iam:CreateRole",
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "iam:PermissionsBoundary": "arn:aws:iam::123456789012:policy/DeveloperBoundary"
        }
      }
    }
  ]
}
```

If you also want the principal to attach the boundary after creation, you must explicitly allow the relevant IAM actions, such as `iam:PutRolePermissionsBoundary`, and still constrain them with conditions.

Example boundary policy for a developer role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCommonAppActions",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    },
    {
      "Sid": "DenyPrivilegeEscalation",
      "Effect": "Deny",
      "Action": [
        "iam:*",
        "organizations:*",
        "kms:PutKeyPolicy"
      ],
      "Resource": "*"
    }
  ]
}
```

This is only an example pattern. In real designs, you usually scope resources more tightly and avoid broad `Resource: "*"` unless the action requires it.

**`iam:PassRole` and privilege escalation**

`iam:PassRole` is one of the most important SAP-C02 escalation topics. It does not let a user assume a role. It lets a user delegate an existing role to a service.

That means the danger is indirect:

- If a user can create or modify a Lambda function and can pass a highly privileged role to Lambda, the function may run with that role’s permissions.
- If a user can launch EC2 instances and pass an admin role to EC2, the instance profile may expose powerful credentials to code running on the instance.
- If a user can update an ECS task definition and pass a privileged task role, the container can inherit those permissions.

Best practice:

- Restrict `iam:PassRole` to specific role ARNs.
- Restrict it to specific services using conditions such as `iam:PassedToService`.
- Combine it with boundaries and SCPs so the user cannot create a new privileged role and then pass it.

Example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowPassOnlyAppExecutionRoleToLambda",
      "Effect": "Allow",
      "Action": "iam:PassRole",
      "Resource": "arn:aws:iam::123456789012:role/AppLambdaExecutionRole",
      "Condition": {
        "StringEquals": {
          "iam:PassedToService": "lambda.amazonaws.com"
        }
      }
    }
  ]
}
```

**SCPs and AWS Organizations**

SCPs are organization-level guardrails. They are attached to the root, OUs, or accounts in AWS Organizations. They define the maximum permissions that identities in member accounts can receive.

Important exam distinctions:

- SCPs apply only to accounts that are part of an organization.
- SCPs do not apply to the management account.
- SCPs do not grant access; they only filter what is possible.
- SCPs affect IAM users and roles in member accounts, including service-linked roles in those member accounts, subject to AWS service behavior and SCP support.
- SCPs do not replace IAM policies, resource policies, or KMS key policies.

The default behavior in Organizations is often misunderstood. If an account is in an organization and has the default `FullAWSAccess` SCP attached at the root, then SCPs do not further restrict permissions unless you add more restrictive SCPs. If you remove or replace that allow-all SCP, you can easily end up with no effective permissions for affected accounts or OUs.

**Deny-list and allow-list SCP strategies**

A deny-list strategy starts with broad access and blocks specific actions. This is easier to adopt incrementally.

Example deny SCP that blocks leaving the organization and disabling CloudTrail:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyLeavingOrgAndDisablingLogging",
      "Effect": "Deny",
      "Action": [
        "organizations:LeaveOrganization",
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail"
      ],
      "Resource": "*"
    }
  ]
}
```

An allow-list strategy is stricter. You explicitly allow only approved actions, and everything else is denied by omission when the SCP set does not include broader allows.

Example allow-list SCP:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOnlyCoreReadAndApprovedOps",
      "Effect": "Allow",
      "Action": [
        "ec2:Describe*",
        "s3:Get*",
        "s3:List*",
        "cloudwatch:Get*",
        "cloudwatch:List*"
      ],
      "Resource": "*"
    }
  ]
}
```

Use allow-list SCPs when you want a tightly controlled environment, such as a sandbox OU or a regulated workload OU. Use deny-list SCPs when you need broad baseline access but want to block a small set of risky actions.

**Region restriction SCPs**

A common enterprise pattern is to restrict workloads to approved AWS Regions using `aws:RequestedRegion`. This is often implemented with `NotAction` so that you deny all actions outside approved Regions while exempting global services that do not support regional endpoints in the same way.

Example pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "NotAction": [
        "iam:*",
        "route53:*",
        "cloudfront:*",
        "organizations:*",
        "support:*",
        "sts:*"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

This pattern is useful, but the exact exception list depends on the services you use. Global services, service-linked operations, and cross-Region control-plane workflows may require additional handling. Do not memorize one exception list as universally correct.

**Policy evaluation and edge cases**

The exam often tests whether you understand why a request is denied even when one policy seems to allow it.

Think in terms of all applicable controls:

- Identity-based policy must allow the action.
- Permission boundary must also allow the action.
- Session policy must also allow the action.
- SCP must also allow the action.
- Resource policy must allow the action when the resource is protected by one.
- Explicit deny in any applicable policy wins.

Cross-account access is especially important:

- If an IAM role in Account A accesses an S3 bucket in Account B, the role’s identity policy must allow the action, the bucket policy must allow it, and any boundary, session policy, or SCP constraints must still permit it.
- If a resource policy grants access directly to a role session or user, that grant can be effective even when the principal’s identity policy is narrower. However, the boundary still constrains what the principal can do through its identity-based permissions, and SCPs still constrain the account-level ceiling.
- For KMS, the key policy is often the deciding resource policy. A missing key policy permission can look like an IAM issue even when the IAM policy is correct.

A practical exam rule: when troubleshooting, do not assume the first policy you inspect is the blocker. Check the whole permission path.

**Troubleshooting workflow**

Use a structured approach instead of a fixed policy order.

1. Confirm the principal and session type.
   - Is it an IAM user, role, assumed-role session, federated session, or service principal?
   - Is a session policy in play?

2. Check the requested action and resource ARN.
   - Many failures are caused by a resource ARN mismatch or an action that does not support resource-level permissions.

3. Inspect identity-based policies.
   - Look for missing allows or explicit denies.

4. Inspect permission boundaries.
   - Verify that the boundary includes the action and resource pattern.

5. Inspect SCPs.
   - Confirm the account is in an organization and the OU/root SCPs permit the action.

6. Inspect resource policies and service-specific policies.
   - Examples include S3 bucket policies, KMS key policies, SNS topic policies, SQS queue policies, and Secrets Manager resource policies.

7. Check condition keys.
   - Common issues involve `aws:RequestedRegion`, `aws:PrincipalArn`, `aws:SourceVpce`, `aws:SourceIp`, `kms:ViaService`, and `iam:PassedToService`.

8. Validate service-specific dependencies.
   - Some actions require additional permissions on related resources, such as KMS decrypt permissions for encrypted data or `iam:PassRole` for service execution.

**AWS IAM Access Analyzer and policy simulation**

AWS IAM Access Analyzer helps identify unintended access, especially from resource policies. It is useful for spotting public or cross-account exposure and for validating whether a policy grants access beyond your intended scope.

The IAM policy simulator is useful for testing identity policies and boundaries, but you must understand its limits. It does not evaluate SCPs, and it needs the right context to model conditions accurately.

Example simulation for an identity policy with a boundary:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/DeveloperRole \
  --action-names s3:PutObject \
  --resource-arns arn:aws:s3:::my-app-bucket/* \
  --permissions-boundary-policy-input-list file://boundary.json \
  --context-entries ContextKeyName=aws:RequestedRegion,ContextKeyType=string,ContextKeyValues=us-west-2
```

Use additional context entries when testing condition-based policies. For example:

- `aws:PrincipalArn`
- `aws:SourceIp`
- `aws:SourceVpce`
- `aws:RequestedRegion`
- `iam:PassedToService`

For KMS, also test service context such as `kms:ViaService` when relevant.

**Control Tower guardrails versus SCPs**

AWS Control Tower uses preventive and detective guardrails to help govern multi-account environments. Preventive guardrails are typically implemented with SCPs under the hood, while detective guardrails identify noncompliant resources or configurations after the fact.

Use this distinction:

- **SCPs** are the raw Organizations control mechanism.
- **Control Tower guardrails** are higher-level governance abstractions that may deploy SCPs and AWS Config rules for you.

On the exam, if the question asks for organization-wide prevention of an action, SCPs are the direct answer. If the question asks for a managed landing zone governance framework with built-in guardrails, Control Tower is often the better fit.

**Choosing the right control**

| Requirement | Best fit | Why |
|---|---|---|
| Limit what teams can create in IAM | Permission boundary | Caps delegated IAM administration |
| Prevent risky actions across an OU | SCP | Applies at org level to member accounts |
| Restrict a temporary session | Session policy | Narrows assumed-role permissions |
| Control access to a specific resource | Resource policy | Grants or denies access at the resource |
| Delegate service access through a role | `iam:PassRole` | Lets a principal hand a role to a service |

**Exam-style scenarios**

- A platform team needs to create application roles, but those roles must never exceed a predefined permission set.
  - Use a permission boundary and require it at role creation with `iam:CreateRole` plus `iam:PermissionsBoundary`.

- Security wants to prevent any account in a production OU from disabling CloudTrail or leaving the organization.
  - Use an SCP with explicit denies.

- A contractor should access only one S3 bucket in another account.
  - Use a bucket policy to grant access. If the contractor assumes a role, add a trust policy for the role. If the contractor uses an IAM user or federated principal directly, no trust policy is involved.

- A developer can deploy Lambda functions but must not attach an admin role to them.
  - Restrict `iam:PassRole` to approved execution roles and use a boundary to prevent creation of privileged roles.

- A workload must run only in approved Regions.
  - Use an SCP with `aws:RequestedRegion`, and verify global service exceptions.

**Common pitfalls**

- Confusing `iam:PassRole` with role assumption.
- Assuming permission boundaries apply to IAM groups.
- Treating SCPs as if they grant permissions.
- Forgetting that the management account is not governed by SCPs.
- Assuming a resource policy is always blocked by a boundary in every access path.
- Using the policy simulator as if it evaluates SCPs.
- Forgetting service-specific policies such as KMS key policies or VPC endpoint policies.
- Overlooking condition keys that silently deny access.

**Practical design patterns**

For enterprise landing zones, a strong pattern is:

- Use SCPs for coarse organizational guardrails.
- Use permission boundaries for delegated IAM administration.
- Use resource policies for cross-account and service-specific access.
- Use session policies for temporary task scoping.
- Use IAM Access Analyzer and simulation tools to validate the design.

A layered model gives you defense in depth. It also reduces the blast radius when one control is misconfigured.

**Quick reference**

- **Permission boundary**: maximum permissions for an IAM user or role.
- **SCP**: maximum permissions for accounts in AWS Organizations.
- **Trust policy**: who can assume a role.
- **`iam:PassRole`**: who can hand a role to a service.
- **Resource policy**: who can access a resource directly.
- **Session policy**: temporary restriction on an assumed role or federated session.
