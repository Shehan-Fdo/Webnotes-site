## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Designing for access patterns**

DynamoDB table design starts with the questions your application must answer, not with normalized entities. You model the table around the reads and writes you need most often, then choose keys and indexes that make those operations efficient.

The core exam idea is simple: DynamoDB is not a relational database. You do not join tables at query time, and you do not filter your way out of a poor key design. If an attribute is not part of a key or index, you generally cannot query it efficiently.

- Start from concrete access patterns such as “get one order,” “list orders for a customer,” or “find all open tickets for a tenant.”
- Design for the dominant read paths first, then verify that writes remain practical.
- Prefer a small number of well-chosen tables over many normalized tables.

**Primary key design**

Every DynamoDB table needs a primary key. It can be either:
- **Simple key**: partition key only
- **Composite key**: partition key plus sort key

The partition key determines item distribution across partitions. The sort key lets you store multiple related items under the same partition key and query them by range or prefix.

A good key design gives you:
- Fast point lookups
- Efficient range queries
- Natural grouping of related items
- Predictable scaling behavior

A poor key design creates hot partitions, forces scans, or makes common queries impossible without extra indexes.

**GetItem versus Query**

Use `GetItem` when you know the full primary key for exactly one item. On a composite-key table, that means you must provide both the partition key and the sort key.

Use `Query` when you know the partition key and want one or more items from that partition, optionally narrowed by sort key conditions.

- `GetItem` is the fastest way to fetch a single item by exact key.
- `Query` is the right choice for one-to-many access within a partition.
- `Scan` reads every item and is usually a design smell for production access paths.

Example of a composite-key table:

```bash
aws dynamodb get-item \
  --table-name Orders \
  --key '{
    "PK": {"S": "CUST#1001"},
    "SK": {"S": "ORDER#2026-04-07#A100"}
  }'
```

Example `Query` for all orders for one customer:

```bash
aws dynamodb query \
  --table-name Orders \
  --key-condition-expression "PK = :pk AND begins_with(SK, :prefix)" \
  --expression-attribute-values '{
    ":pk": {"S": "CUST#1001"},
    ":prefix": {"S": "ORDER#"}
  }'
```

If you only know `PK` and not `SK`, `GetItem` is not enough on a composite-key table. If you only know a non-key attribute such as `Status`, DynamoDB cannot query it directly unless you model an index for it.

**Single-table modeling and item collections**

Single-table design stores multiple entity types in one table. You distinguish items with key prefixes and type attributes, then use the sort key to group related records.

A common pattern is an item collection: all items that share the same partition key. This lets you model a business aggregate such as a customer, order, ticket, or tenant.

Example item collection:

```text
PK = CUST#1001
SK = PROFILE
SK = ORDER#2026-04-07#A100
SK = ORDER#2026-04-08#A101
SK = ADDRESS#HOME
```

This design supports:
- Fetching the customer profile with `GetItem`
- Listing all customer orders with `Query`
- Fetching a specific child item by exact sort key

Use item type prefixes such as `PROFILE`, `ORDER#`, or `ADDRESS#` to keep multiple access patterns in one collection.

**Deriving a table from access patterns**

A reliable exam approach is to translate requirements into key design.

1. List the access patterns in plain language.
2. Mark which ones are point lookups, range queries, or alternate lookups.
3. Decide whether the base table can satisfy each pattern.
4. Add a GSI (Global Secondary Index) only when the base table cannot.
5. Check write cost, consistency needs, and item size constraints.

Example requirements:
- Get a customer by customer ID
- List all orders for a customer, newest first
- Find all open orders across all customers
- Fetch an order by order ID

Possible design:
- Base table PK: `CUST#<customerId>`
- Base table SK: `TYPE#<timestamp>#<orderId>`
- GSI1 PK: `STATUS#<status>`
- GSI1 SK: `CREATED_AT#<timestamp>#<orderId>`
- Optional GSI2 PK: `ORDER#<orderId>` if order lookup by ID is common and order IDs are not already unique in the base key

This is the exam mindset: one table can support multiple access patterns if you choose keys carefully.

**Global Secondary Indexes**

A GSI lets you query the same data using an alternate partition key and optional sort key. GSIs are the main tool for access patterns that the base table cannot satisfy.

Use a GSI when:
- You need to query by a different attribute
- You need a different sort order
- You need a sparse view of only some items
- You need to support an alternate lookup path without duplicating the whole table

Important GSI properties:
- You define the index key schema separately from the base table.
- You can project only selected attributes into the index.
- Queries on a GSI are eventually consistent.
- A GSI can be sparse, meaning only items that contain the index key attributes appear in the index.

Sparse index example:
- Base table stores all orders
- Only open orders include `OpenStatus = "OPEN"`
- GSI on `OpenStatus` returns only open orders

This is useful when you want a selective access path without maintaining a separate table.

Example table creation with a GSI:

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions \
      AttributeName=PK,AttributeType=S \
      AttributeName=SK,AttributeType=S \
      AttributeName=Status,AttributeType=S \
      AttributeName=CreatedAt,AttributeType=S \
  --key-schema \
      AttributeName=PK,KeyType=HASH \
      AttributeName=SK,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST \
  --global-secondary-indexes '[
    {
      "IndexName": "StatusCreatedAtIndex",
      "KeySchema": [
        {"AttributeName": "Status", "KeyType": "HASH"},
        {"AttributeName": "CreatedAt", "KeyType": "RANGE"}
      ],
      "Projection": {
        "ProjectionType": "INCLUDE",
        "NonKeyAttributes": ["OrderId", "CustomerId", "Total"]
      }
    }
  ]'
```

Example query against the GSI:

```bash
aws dynamodb query \
  --table-name Orders \
  --index-name StatusCreatedAtIndex \
  --key-condition-expression "Status = :status AND begins_with(CreatedAt, :prefix)" \
  --expression-attribute-values '{
    ":status": {"S": "OPEN"},
    ":prefix": {"S": "2026-04"}
  }'
```

> A GSI cannot rescue a design that lacks the right key attributes. If the attribute is not modeled into the index key, you still cannot query it efficiently.

**Local Secondary Indexes**

An LSI (Local Secondary Index) uses the same partition key as the base table but a different sort key. LSIs are useful when you need another way to sort or query items within the same partition.

Key constraints:
- You must define LSIs when you create the table.
- You cannot add an LSI later.
- The LSI shares the base table partition key.
- LSIs support strongly consistent reads.
- Each partition key value has an item collection limit of 10 GB across the base table and all LSIs.

That 10 GB item collection limit matters when one partition key can accumulate many items, such as a very active tenant or customer.

Example use case:
- Base table PK: `TENANT#123`
- Base table SK: `ORDER#<createdAt>#<orderId>`
- LSI sort key: `STATUS`

This lets you query all items for one tenant and sort or filter them by status without changing the partition key.

**GSI versus LSI**

| Feature | GSI | LSI |
|---|---|---|
| Partition key | Different from base table | Same as base table |
| Sort key | Optional | Different from base table |
| Creation time | Can be added later | Must exist at table creation |
| Consistency | Eventually consistent only | Can be strongly consistent |
| Item collection limit | No shared 10 GB item collection limit | Shares the 10 GB item collection limit per partition key |
| Best for | Alternate access patterns | Alternate sort order within one partition |

Choose a GSI for alternate lookup paths. Choose an LSI when you need another sort order on the same partition key and you can accept the table-creation-time constraint.

**Many-to-many relationships and adjacency lists**

DynamoDB handles many-to-many relationships with duplication and key design, not joins. A common pattern is the adjacency list, where each relationship is stored as an item.

Example: students and courses
- Student item: `PK = STUDENT#1`, `SK = PROFILE`
- Course item: `PK = COURSE#9`, `SK = PROFILE`
- Enrollment edge: `PK = STUDENT#1`, `SK = COURSE#9`
- Reverse enrollment edge: `PK = COURSE#9`, `SK = STUDENT#1`

This supports:
- Listing all courses for a student
- Listing all students in a course

You often duplicate relationship items in both directions when both traversals are important. That is normal in DynamoDB because it trades write complexity for fast reads.

**Overloading indexes**

You can overload a single GSI or LSI to support multiple access patterns by using prefixes in the index keys.

Example GSI key values:
- `PK = STATUS#OPEN`
- `PK = STATUS#CLOSED`
- `PK = TYPE#INVOICE`
- `PK = TYPE#REFUND`

This lets one index serve multiple query shapes. The sort key can further distinguish entities or time ranges.

Benefits:
- Fewer indexes to manage
- Lower write amplification than creating many separate indexes
- Flexible modeling for related access patterns

Tradeoff:
- You must carefully namespace key values to avoid collisions
- Query logic becomes more dependent on consistent prefixes

**Hot partitions and write sharding**

A hot partition happens when too many reads or writes target the same partition key. DynamoDB distributes data by partition key, so a skewed key can create throttling even if the table has plenty of overall capacity.

Common causes:
- A single tenant receives most traffic
- A timestamp-only partition key concentrates writes
- A popular “latest items” query hits one key repeatedly

One mitigation is write sharding. You add a shard suffix to spread writes across multiple partition key values.

Example write-sharded keys:
- `PK = ORDER#2026-04-07#00`
- `PK = ORDER#2026-04-07#01`
- `PK = ORDER#2026-04-07#02`

To read the full logical set, your application queries each shard and merges the results.

```python
import boto3

ddb = boto3.client("dynamodb")
shards = ["00", "01", "02", "03"]

items = []
for shard in shards:
    resp = ddb.query(
        TableName="Orders",
        KeyConditionExpression="PK = :pk",
        ExpressionAttributeValues={
            ":pk": {"S": f"ORDER#2026-04-07#{shard}"}
        }
    )
    items.extend(resp["Items"])
```

Write sharding improves distribution, but it adds application complexity because you must know the shard set and reconstruct results across shards.

**Querying by non-key attributes**

DynamoDB does not support efficient ad hoc queries on arbitrary attributes. You can:
- Use `Scan`, which reads the whole table or index
- Add a GSI or LSI that models the attribute as a key
- Denormalize the attribute into a key-friendly form

For exam purposes, the rule is:
- If you need to query it often, make it a key.
- If you only need to filter a small result set after a key-based query, use `FilterExpression`.
- Do not use `FilterExpression` as a substitute for a bad key design.

Example of a filter after a key-based query:

```bash
aws dynamodb query \
  --table-name Orders \
  --key-condition-expression "PK = :pk" \
  --filter-expression "#st = :status" \
  --expression-attribute-names '{"#st":"Status"}' \
  --expression-attribute-values '{
    ":pk": {"S": "CUST#1001"},
    ":status": {"S": "OPEN"}
  }'
```

This still reads all items in the partition that match `PK = :pk`, then filters them client-side on the service side. It does not reduce the read cost as much as a key-based query on a purpose-built index.

**Consistency and read behavior**

Base table `GetItem` and `Query` can use strongly consistent reads in many cases. GSIs are eventually consistent only. LSIs can support strongly consistent reads because they share the base table partition key.

Use strong consistency when:
- You must read your own write immediately
- You are reading critical state such as a lock or workflow checkpoint
- Your application cannot tolerate stale reads

Use eventual consistency when:
- Slightly stale data is acceptable
- You need the flexibility of a GSI
- You are reading derived views or search-like access paths

**Practical design checklist**

Before you finalize a DynamoDB design, verify:
- Each required access pattern maps to a `GetItem`, `Query`, or index query
- No common path depends on `Scan`
- The partition key avoids obvious hot spots
- The sort key supports the needed ordering or prefix lookup
- Any GSI is sparse only when that is intentional
- Any LSI is created at table creation time and stays within the 10 GB item collection limit
- You understand the write amplification from duplicating data into indexes
- You know whether eventual consistency is acceptable

**Exam traps to avoid**

- Assuming you can query any attribute without an index
- Using `FilterExpression` as if it were a key condition
- Forgetting that `GetItem` on a composite-key table needs both key values
- Designing an LSI after the table already exists
- Ignoring the 10 GB item collection limit for a single partition key
- Treating a GSI as strongly consistent
- Overusing `Scan` for production access patterns
- Designing a timestamp-only partition key that creates a hot partition

**Mental model for the exam**

Think in terms of access paths:
- What do you need to fetch?
- What key do you know at read time?
- Can the base table answer it?
- If not, which index makes it efficient?

If you can answer those four questions, you can usually derive the correct DynamoDB table design.
