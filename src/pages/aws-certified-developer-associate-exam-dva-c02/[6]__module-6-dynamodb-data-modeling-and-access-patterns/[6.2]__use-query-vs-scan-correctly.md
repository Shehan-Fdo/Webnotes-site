## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Core idea**

Amazon DynamoDB is an <abbr title="NoSQL database service">NoSQL</abbr> database built around access patterns. The exam expects you to choose the most efficient read operation for the question being asked:

- Use **Query** when you know the partition key value and want a targeted read.
- Use **Scan** when you must inspect items without a key-based access path.
- Prefer redesigning the table or adding an index over relying on Scan for important production access patterns.

A good mental model is: **Query is selective; Scan is exhaustive**. Query reads items for one partition key value on a table or index and can further narrow results with sort key conditions. Scan reads every item in the table or index and filters afterward.

**Query: the preferred read path**

Query is the primary way to retrieve data in DynamoDB when your access pattern is known. You must provide the partition key value, and you may optionally constrain the sort key. Query can run against:

- A **table**
- A **local secondary index** (<abbr title="Local Secondary Index">LSI</abbr>)
- A **global secondary index** (<abbr title="Global Secondary Index">GSI</abbr>)

Use Query when you want predictable performance and lower read cost than scanning the full dataset. It is especially useful for:

- Fetching all orders for a customer
- Fetching all log events for a device within a time range
- Fetching items with a sort-key prefix such as `begins_with`

Query supports common sort key conditions:

- `=`
- `<`
- `<=`
- `>`
- `>=`
- `BETWEEN`
- `begins_with`

A typical Query with a prefix match on the sort key looks like this:

```javascript
import { DynamoDBClient } from "@aws-sdk/client-dynamodb";
import { DynamoDBDocumentClient, QueryCommand } from "@aws-sdk/lib-dynamodb";

const client = new DynamoDBClient({});
const ddb = DynamoDBDocumentClient.from(client);

const result = await ddb.send(
  new QueryCommand({
    TableName: "Orders",
    KeyConditionExpression: "#pk = :customerId AND begins_with(#sk, :prefix)",
    ExpressionAttributeNames: {
      "#pk": "customerId",
      "#sk": "orderId"
    },
    ExpressionAttributeValues: {
      ":customerId": "CUST#123",
      ":prefix": "ORDER#2026-04"
    },
    ScanIndexForward: false,
    Limit: 20
  })
);

console.log(result.Items);
```

Key points for exam prep:

- **Query requires the partition key value** for the table or index.
- **Query does not search the whole table**.
- **Query can return many items** from the same partition key value, so cost still depends on the amount of data read.
- **Query can be strongly consistent on tables and LSIs**, but **not on GSIs**.
- **Query reads up to 1 MB per page** before pagination is required.

**Query on tables and indexes**

Query behaves slightly differently depending on where you run it:

- On a **table**, you query using the table’s primary key.
- On an **LSI**, you still query by the table’s partition key, but you can use the LSI’s alternate sort key.
- On a **GSI**, you query by the GSI’s partition key and optional sort key.

This matters because the index defines the access path. If your application needs to find items by a non-key attribute, the best solution is often to model that attribute into a GSI and Query the index.

Example of querying a GSI:

```javascript
const result = await ddb.send(
  new QueryCommand({
    TableName: "Orders",
    IndexName: "StatusCreatedAtIndex",
    KeyConditionExpression: "#status = :status AND #createdAt >= :start",
    ExpressionAttributeNames: {
      "#status": "status",
      "#createdAt": "createdAt"
    },
    ExpressionAttributeValues: {
      ":status": "PENDING",
      ":start": "2026-04-01T00:00:00Z"
    }
  })
);
```

Use this pattern when the access pattern is stable and important. For exam questions, if you need to retrieve items by a field that is not part of the primary key, the best answer is often **add a GSI and Query it**, not Scan the table.

**Scan: the exhaustive fallback**

Scan reads every item in the table or index and then applies any filter expression afterward. It is the least efficient read pattern in DynamoDB because it consumes read capacity for the items read, not just the items returned.

Use Scan when:

- You are doing ad hoc analysis
- You are migrating or auditing data
- You do not have a key-based access pattern available
- You are building a one-time administrative tool

Scan can run against:

- A **table**
- An **LSI**
- A **GSI**

Important distinction:

- A Scan on an index is still a **scan of that index**, not a targeted query.
- A Scan on a GSI does **not** become a Query just because the index exists.

A basic Scan example:

```javascript
import { ScanCommand } from "@aws-sdk/lib-dynamodb";

const result = await ddb.send(
  new ScanCommand({
    TableName: "Orders",
    FilterExpression: "#status = :status",
    ExpressionAttributeNames: {
      "#status": "status"
    },
    ExpressionAttributeValues: {
      ":status": "PENDING"
    },
    Limit: 25
  })
);

console.log(result.Items);
```

This is a common trap: `FilterExpression` does not reduce the items read, only the items returned. DynamoDB still scans the underlying data first.

**Query vs Scan: how to choose**

Choose the operation based on whether you can target the data with a key.

| Situation | Best choice | Why |
|---|---|---|
| You know the partition key value | Query | Reads only the matching partition key scope |
| You need a sort-key range or prefix | Query | Supports efficient key conditions |
| You need all items in a table or index | Scan | Exhaustive read is required |
| You need to search by a non-key attribute | Usually add a GSI and Query | Better design than scanning repeatedly |
| You are doing one-time maintenance or export | Scan | Simpler for ad hoc work |

A useful exam rule:

- If the access pattern matters and will be used repeatedly, **model for Query**.
- If the task is temporary or exploratory, **Scan** may be acceptable.

> If a question suggests “search the whole table for a status value,” the better production answer is usually to create a GSI on `status` and Query it, not to rely on Scan.

**Pagination and the 1 MB page limit**

Both Query and Scan return results in pages. DynamoDB limits a single response page to **1 MB** of data before pagination continues. This is a frequent exam and real-world detail.

When a response is truncated, DynamoDB returns `LastEvaluatedKey`. You pass that value back as `ExclusiveStartKey` to continue from where the previous page ended.

Query pagination example:

```javascript
let lastKey;

do {
  const page = await ddb.send(
    new QueryCommand({
      TableName: "Orders",
      KeyConditionExpression: "#pk = :customerId",
      ExpressionAttributeNames: {
        "#pk": "customerId"
      },
      ExpressionAttributeValues: {
        ":customerId": "CUST#123"
      },
      ExclusiveStartKey: lastKey,
      Limit: 50
    })
  );

  console.log(page.Items);
  lastKey = page.LastEvaluatedKey;
} while (lastKey);
```

Scan pagination example:

```javascript
let lastKey;

do {
  const page = await ddb.send(
    new ScanCommand({
      TableName: "Orders",
      FilterExpression: "#status = :status",
      ExpressionAttributeNames: {
        "#status": "status"
      },
      ExpressionAttributeValues: {
        ":status": "PENDING"
      },
      ExclusiveStartKey: lastKey,
      Limit: 100
    })
  );

  console.log(page.Items);
  lastKey = page.LastEvaluatedKey;
} while (lastKey);
```

Practical notes:

- `Limit` controls how many items DynamoDB evaluates in a request, not necessarily how many items are returned after filtering.
- `LastEvaluatedKey` is the marker for the next page.
- Pagination is required even for Query when the matching data exceeds 1 MB.

**Consistency behavior**

Consistency is another important difference:

- **Query on a table** can use **strongly consistent reads** if you set `ConsistentRead: true`.
- **Query on an LSI** can also use strongly consistent reads.
- **Query on a GSI** does **not** support strongly consistent reads.
- **Scan on a table** can use strongly consistent reads.
- **Scan on a GSI** does **not** support strongly consistent reads.

Example:

```javascript
const result = await ddb.send(
  new QueryCommand({
    TableName: "Orders",
    KeyConditionExpression: "#pk = :customerId",
    ExpressionAttributeNames: {
      "#pk": "customerId"
    },
    ExpressionAttributeValues: {
      ":customerId": "CUST#123"
    },
    ConsistentRead: true
  })
);
```

Use strong consistency only when the application truly needs the latest committed data. It increases read cost and is not available on GSIs.

**FilterExpression vs KeyConditionExpression**

This distinction is a common exam topic.

- **KeyConditionExpression** narrows the items DynamoDB reads at the key level.
- **FilterExpression** removes items after DynamoDB has already read them.

That means:

- A Query with a good key condition is efficient.
- A Query with a broad key condition and a filter can still read a lot of data.
- A Scan with a filter still reads the whole table or index.

Example of an inefficient pattern:

```javascript
const result = await ddb.send(
  new QueryCommand({
    TableName: "Orders",
    KeyConditionExpression: "#pk = :customerId",
    FilterExpression: "#status = :status",
    ExpressionAttributeNames: {
      "#pk": "customerId",
      "#status": "status"
    },
    ExpressionAttributeValues: {
      ":customerId": "CUST#123",
      ":status": "PENDING"
    }
  })
);
```

This is valid, but if `customerId` has many items and only a few are `PENDING`, you still read the full partition scope first. If `status` is the real access pattern, model for it with an index.

**Designing for Query**

DynamoDB works best when you design around access patterns before you write code. For exam questions, think in terms of “how will the application retrieve data?”

Common design strategies:

- Put the most selective and frequently used lookup in the **partition key**.
- Use the **sort key** for ranges, ordering, and prefixes.
- Add a **GSI** when you need a second query path.
- Use **composite keys** to support multiple entity types in one table.

Example single-table style key design:

- `PK = CUSTOMER#123`
- `SK = ORDER#2026-04-01#A100`

This lets you Query:

- All items for a customer
- All orders for a date range
- All orders with a prefix such as `ORDER#2026-04`

A Query with descending order:

```javascript
const result = await ddb.send(
  new QueryCommand({
    TableName: "AppData",
    KeyConditionExpression: "#pk = :pk AND #sk BETWEEN :start AND :end",
    ExpressionAttributeNames: {
      "#pk": "PK",
      "#sk": "SK"
    },
    ExpressionAttributeValues: {
      ":pk": "CUSTOMER#123",
      ":start": "ORDER#2026-04-01",
      ":end": "ORDER#2026-04-30"
    },
    ScanIndexForward: false
  })
);
```

**When Scan is acceptable**

Scan is not “wrong” in every case. It is appropriate when:

- You need a one-time full dataset pass
- You are building a backfill or migration script
- You are validating data quality
- You are exploring a small table where performance is not a concern

Still, be cautious:

- Scan cost grows with table size.
- Scan can create hot read pressure on large tables.
- Scan is usually a poor choice for user-facing requests.

**Common exam traps**

- Thinking a `FilterExpression` makes Scan efficient.
- Forgetting that Query requires the partition key value.
- Assuming Query can search arbitrary attributes without an index.
- Forgetting that Query on a GSI cannot be strongly consistent.
- Ignoring the 1 MB page limit and `LastEvaluatedKey`.
- Choosing Scan when the correct answer is to add a GSI and Query it.
- Forgetting that Scan can also run on an index, not just a table.

**Quick decision checklist**

Use this checklist during exam questions:

1. Do you know the partition key value?
   - Yes: use Query.
   - No: consider whether a GSI is needed.
2. Do you need a range or prefix on the sort key?
   - Yes: use Query with a sort key condition.
3. Are you reading everything or doing ad hoc analysis?
   - Yes: use Scan.
4. Is the access pattern important and repeated?
   - Yes: redesign for Query, usually with a GSI.
5. Do you need strong consistency?
   - Use a table or LSI, not a GSI.

**Practical CLI examples**

Query a table:

```bash
aws dynamodb query \
  --table-name Orders \
  --key-condition-expression "#pk = :pk" \
  --expression-attribute-names '{"#pk":"customerId"}' \
  --expression-attribute-values '{":pk":{"S":"CUST#123"}}'
```

Scan a table with a filter:

```bash
aws dynamodb scan \
  --table-name Orders \
  --filter-expression "#status = :status" \
  --expression-attribute-names '{"#status":"status"}' \
  --expression-attribute-values '{":status":{"S":"PENDING"}}'
```

Paginate a Scan from the CLI by using the returned `LastEvaluatedKey` as `--exclusive-start-key` in the next request. This is useful for scripts that process large tables in batches.

**Summary**

Query is the correct choice when you can target data by key. It is efficient, index-aware, and the preferred DynamoDB access pattern. Scan is the fallback when you must inspect all items or when no key-based path exists. For production systems, the best answer is often not “use Scan,” but “model the data so you can Query it.”
