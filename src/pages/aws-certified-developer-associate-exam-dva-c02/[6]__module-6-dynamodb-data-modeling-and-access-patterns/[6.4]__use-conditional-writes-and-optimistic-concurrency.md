## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Conditional writes: the core idea**

Conditional writes let you update data only when the item still matches the state you expect. In Amazon DynamoDB, this is the primary tool for preventing accidental overwrites when multiple clients read and write the same item.

This matters because distributed systems rarely have a single writer. Two users, Lambda functions, or services can read the same record, make different changes, and then write back. Without a condition, the last write wins and earlier changes can be lost. With a condition, the write succeeds only if the item still satisfies your rule.

Common use cases include:

- Preventing duplicate creation of an item
- Ensuring a balance never goes below zero
- Updating a record only if a version number still matches
- Enforcing business rules such as status transitions

**How conditional writes work in DynamoDB**

A conditional write attaches a **ConditionExpression** to a write operation. DynamoDB evaluates the condition atomically with the write. If the condition is true, the write succeeds. If it is false, DynamoDB rejects the write with a conditional check failure.

Typical write operations that support conditions include:

- `PutItem`
- `UpdateItem`
- `DeleteItem`
- `ConditionCheck` in transactions

A condition can reference:

- Attribute existence or absence
- Attribute values
- Comparisons such as `=`, `<>`, `<`, `<=`, `>`, `>=`
- Functions such as `attribute_exists()`, `attribute_not_exists()`, `begins_with()`, `contains()`, `size()`

Example: only create an item if it does not already exist.

```bash
aws dynamodb put-item \
  --table-name Orders \
  --item '{
    "OrderId": {"S": "A1001"},
    "Status": {"S": "PENDING"}
  }' \
  --condition-expression "attribute_not_exists(OrderId)"
```

If another client already created `OrderId = A1001`, this write fails instead of overwriting the existing item.

**Optimistic concurrency control**

Optimistic concurrency control assumes conflicts are uncommon, so you proceed without locking the item. Before writing, you include a version check to confirm that nobody else changed the item since you last read it.

The usual pattern is:

- Read the item and its `version` attribute
- Modify the item in your application
- Write the update only if `version` still equals the value you read
- Increment `version` on success

This prevents lost updates because each writer must prove it is working from the latest known version.

Example update with a version check:

```bash
aws dynamodb update-item \
  --table-name Inventory \
  --key '{"Sku":{"S":"SKU-123"}}' \
  --update-expression "SET Quantity = :q, version = version + :inc" \
  --condition-expression "version = :expected" \
  --expression-attribute-values '{
    ":q": {"N": "42"},
    ":inc": {"N": "1"},
    ":expected": {"N": "7"}
  }'
```

If the stored `version` is no longer `7`, the update fails. Your application then rereads the item, reapplies the change, and retries if appropriate.

**Why version attributes are useful**

A version attribute gives you a simple, explicit concurrency token. It is easy to reason about and works well for most application-level updates.

Benefits:

- Detects stale reads before overwriting newer data
- Works without locks, which keeps throughput high
- Fits well with stateless services and serverless workloads
- Makes retries safe when combined with idempotent logic

A common schema pattern is:

```json
{
  "PK": "USER#123",
  "SK": "PROFILE",
  "email": "user@example.com",
  "displayName": "Asha",
  "version": 12
}
```

Each successful update increments `version` to `13`, `14`, and so on.

**Practical patterns**

Use conditional writes when you need to enforce invariants at write time.

- Create-only semantics:
  - Use `attribute_not_exists(PK)` to ensure a record is inserted once
- State transitions:
  - Allow `PENDING` to become `APPROVED`, but not `APPROVED` back to `PENDING`
- Inventory control:
  - Update only if `Quantity >= :requested`
- Idempotency:
  - Store a request token and reject duplicates with `attribute_not_exists(requestId)`

Example: decrement stock only if enough inventory remains.

```bash
aws dynamodb update-item \
  --table-name Inventory \
  --key '{"Sku":{"S":"SKU-123"}}' \
  --update-expression "SET Quantity = Quantity - :n" \
  --condition-expression "Quantity >= :n" \
  --expression-attribute-values '{":n":{"N":"3"}}'
```

This is safer than reading the quantity in one request and writing it in another, because the condition and the update are evaluated atomically.

**Transactions versus conditional writes**

Conditional writes protect a single item or a single write operation. If you need to coordinate multiple items, use DynamoDB transactions.

Key differences:

| Capability | Conditional write | Transaction |
|---|---|---|
| Single-item atomicity | Yes | Yes |
| Multiple items | No | Yes |
| Prevent lost updates | Yes | Yes |
| All-or-nothing across items | No | Yes |
| Higher complexity | Lower | Higher |

Use conditional writes when one item contains the state you need to protect. Use transactions when several items must change together.

**Error handling and retries**

A failed condition is not a transient infrastructure error. It means the item no longer matches your expectation.

In application code, handle this separately from throttling or network failures:

- Retry only after rereading the item
- Recompute the update from the latest state
- Avoid blind retries of the same stale write
- Surface a conflict to the user when manual resolution is needed

Example Python-style logic:

```python
import boto3
from botocore.exceptions import ClientError

dynamodb = boto3.client("dynamodb")

try:
    dynamodb.update_item(
        TableName="Inventory",
        Key={"Sku": {"S": "SKU-123"}},
        UpdateExpression="SET Quantity = Quantity - :n, version = version + :one",
        ConditionExpression="version = :expected AND Quantity >= :n",
        ExpressionAttributeValues={
            ":n": {"N": "3"},
            ":one": {"N": "1"},
            ":expected": {"N": "7"},
        },
    )
except ClientError as e:
    code = e.response["Error"]["Code"]
    if code == "ConditionalCheckFailedException":
        print("Item changed or condition not met; reread and retry if appropriate.")
    else:
        raise
```

**Design considerations**

Conditional writes are powerful, but they work best when your data model supports them.

- Keep the data needed for the invariant in the same item when possible
- Use a version attribute for general concurrency control
- Prefer atomic counters or guarded updates over read-modify-write in application code
- Make retries idempotent when the client may repeat requests
- Expect occasional conflicts in hot partitions and design for them

If many writers contend for the same item, conflicts increase. In that case, you may need to reduce write contention, shard the workload, or redesign the access pattern.

**Common exam angles**

For the exam, focus on recognizing when conditional writes solve a concurrency problem.

You should associate these ideas:

- **Optimistic concurrency** means no locking, but you verify the item version before writing
- **Conditional write** means the write happens only if the condition is true
- **ConditionalCheckFailedException** means the condition was not met, not that DynamoDB is unavailable
- **Lost update prevention** is a classic use case for version checks
- **Atomicity** applies to the condition and the write together on the same item

A typical scenario might describe two Lambda functions updating the same DynamoDB record. The correct solution is usually a conditional update with a version attribute, not a lock service.

**CLI and SDK examples**

A create-if-not-exists pattern:

```bash
aws dynamodb put-item \
  --table-name Users \
  --item '{
    "UserId": {"S": "u-100"},
    "Name": {"S": "Mina"}
  }' \
  --condition-expression "attribute_not_exists(UserId)"
```

A guarded status transition:

```bash
aws dynamodb update-item \
  --table-name Tickets \
  --key '{"TicketId":{"S":"T-900"}}' \
  --update-expression "SET #s = :new" \
  --condition-expression "#s = :old" \
  --expression-attribute-names '{"#s":"Status"}' \
  --expression-attribute-values '{
    ":old":{"S":"OPEN"},
    ":new":{"S":"IN_PROGRESS"}
  }'
```

A version-based update in JavaScript (AWS SDK v3 style):

```javascript
import { DynamoDBClient, UpdateItemCommand } from "@aws-sdk/client-dynamodb";

const client = new DynamoDBClient({});

await client.send(new UpdateItemCommand({
  TableName: "Profiles",
  Key: {
    UserId: { S: "u-100" }
  },
  UpdateExpression: "SET displayName = :name, version = version + :one",
  ConditionExpression: "version = :expected",
  ExpressionAttributeValues: {
    ":name": { S: "Mina Patel" },
    ":one": { N: "1" },
    ":expected": { N: "12" }
  }
}));
```

**Troubleshooting checklist**

If a conditional write fails, check the following:

- The condition references the correct attribute name
- The expected version is current
- The item exists if you used `attribute_exists()`
- The item does not exist if you used `attribute_not_exists()`
- The data type in the condition matches the stored type
- Another writer updated the item between read and write

If you see repeated conflicts, the workload may be too contended for a simple optimistic approach. In that case, consider redesigning the item model or reducing concurrent writes to the same key.

**Key takeaway**

Conditional writes and optimistic concurrency are DynamoDB’s standard tools for safe concurrent updates. Use them to enforce invariants, prevent lost updates, and make your application resilient without introducing locks.
