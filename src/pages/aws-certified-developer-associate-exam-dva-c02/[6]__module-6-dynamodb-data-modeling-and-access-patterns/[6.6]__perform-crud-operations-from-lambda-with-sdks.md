**CRUD from Lambda: the core idea**

AWS Lambda functions often act as the application layer that creates, reads, updates, and deletes data in AWS services. You typically use an AWS SDK (Software Development Kit) inside the function to call services such as Amazon DynamoDB, Amazon S3, Amazon RDS, Amazon SQS, or Amazon OpenSearch Service.

The exam expects you to understand not just the API calls, but also the operational constraints of Lambda:
- short execution time and statelessness
- IAM (Identity and Access Management) permissions attached to the execution role
- retries and idempotency
- pagination and throttling
- network placement for private resources

**Lambda execution model and SDK usage**

A Lambda function runs your code in a managed runtime or container image. Your code uses the SDK to make API calls to AWS services. The functionâ€™s execution role provides temporary credentials automatically through the runtime environment.

Common SDK choices:
- **AWS SDK for JavaScript v3** for Node.js
- **boto3** for Python
- AWS SDK for Java, .NET, Go, or other supported runtimes

Use the SDK when you need direct service APIs. Use service integrations when you want Lambda to be triggered by or connected to another AWS service without custom code. For CRUD, SDK calls are the normal pattern because you need application-specific logic.

**IAM permissions for CRUD**

Your Lambda execution role must allow the exact actions the function performs. Keep permissions as narrow as possible.

Typical examples:
- DynamoDB: `dynamodb:GetItem`, `dynamodb:PutItem`, `dynamodb:UpdateItem`, `dynamodb:DeleteItem`, `dynamodb:Query`, `dynamodb:Scan`
- S3: `s3:GetObject`, `s3:PutObject`, `s3:DeleteObject`
- RDS: database permissions are enforced by the database engine, not IAM alone; IAM may still be used for authentication in some setups
- Secrets Manager: `secretsmanager:GetSecretValue` for retrieving credentials or API keys

For DynamoDB, remember that permissions are resource-scoped:
- table-level operations usually reference the table ARN
- index queries require the specific index ARN(s) used by the function
- least privilege means granting only the table and index resources the code actually touches

Example IAM policy for a Lambda function that reads and writes one DynamoDB table and queries one specific GSI (Global Secondary Index):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DynamoDbTableAccess",
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:PutItem",
        "dynamodb:UpdateItem",
        "dynamodb:DeleteItem",
        "dynamodb:Query"
      ],
      "Resource": [
        "arn:aws:dynamodb:us-east-1:123456789012:table/Customers",
        "arn:aws:dynamodb:us-east-1:123456789012:table/Customers/index/CustomerIdIndex"
      ]
    }
  ]
}
```

> If your code queries multiple indexes, include each specific index ARN. Avoid assuming a table ARN alone covers index access for `Query`.

**Choosing the right SDK client**

Many AWS SDKs offer two styles:
- **low-level client**: uses service-native request shapes
- **document client / resource wrapper**: converts native language types into service types

For DynamoDB:
- the low-level client uses `AttributeValue` shapes such as `{ S: "value" }`, `{ N: "123" }`, `{ BOOL: true }`
- the document client lets you use native values such as strings, numbers, arrays, and objects

Use the document client when you want simpler code for application data. Use the low-level client when you need precise control over marshaling or service-specific features.

**DynamoDB CRUD from Lambda**

DynamoDB is a common choice for Lambda-backed applications because it is serverless, low-latency, and scales automatically.

Typical operations:
- **Create** with `PutItem`
- **Read** with `GetItem` or `Query`
- **Update** with `UpdateItem`
- **Delete** with `DeleteItem`

`Query` is preferred over `Scan` when you know the partition key or can use an index. `Scan` reads the whole table or index and is more expensive and slower.

Example using AWS SDK for JavaScript v3 with the DynamoDB document client:

```javascript
import { DynamoDBClient } from "@aws-sdk/client-dynamodb";
import {
  DynamoDBDocumentClient,
  PutCommand,
  GetCommand,
  UpdateCommand,
  DeleteCommand,
  QueryCommand
} from "@aws-sdk/lib-dynamodb";

const client = new DynamoDBClient({});
const ddb = DynamoDBDocumentClient.from(client);

const TABLE_NAME = process.env.TABLE_NAME;

export const handler = async (event) => {
  const { operation, customerId, name } = event;

  if (operation === "create") {
    await ddb.send(new PutCommand({
      TableName: TABLE_NAME,
      Item: {
        customerId,
        name,
        updatedAt: new Date().toISOString()
      }
    }));
    return { statusCode: 201, body: "Created" };
  }

  if (operation === "read") {
    const result = await ddb.send(new GetCommand({
      TableName: TABLE_NAME,
      Key: { customerId }
    }));
    return { statusCode: 200, body: JSON.stringify(result.Item ?? null) };
  }

  if (operation === "update") {
    await ddb.send(new UpdateCommand({
      TableName: TABLE_NAME,
      Key: { customerId },
      UpdateExpression: "SET #n = :name, updatedAt = :ts",
      ExpressionAttributeNames: { "#n": "name" },
      ExpressionAttributeValues: {
        ":name": name,
        ":ts": new Date().toISOString()
      }
    }));
    return { statusCode: 200, body: "Updated" };
  }

  if (operation === "delete") {
    await ddb.send(new DeleteCommand({
      TableName: TABLE_NAME,
      Key: { customerId }
    }));
    return { statusCode: 200, body: "Deleted" };
  }

  if (operation === "queryByName") {
    const result = await ddb.send(new QueryCommand({
      TableName: TABLE_NAME,
      IndexName: "CustomerNameIndex",
      KeyConditionExpression: "#n = :name",
      ExpressionAttributeNames: { "#n": "name" },
      ExpressionAttributeValues: {
        ":name": name
      }
    }));
    return { statusCode: 200, body: JSON.stringify(result.Items ?? []) };
  }

  return { statusCode: 400, body: "Unsupported operation" };
};
```

Key DynamoDB points:
- `PutItem` replaces the item if the key already exists unless you add a condition
- `UpdateItem` is usually better than read-modify-write for partial updates
- `ConditionExpression` helps prevent accidental overwrites
- `Query` requires a partition key equality condition and optional sort key conditions
- `Scan` should be avoided for large tables in Lambda because it can consume time and read capacity quickly

**Low-level DynamoDB client example**

If you use `@aws-sdk/client-dynamodb`, you must send typed attribute values:

```javascript
import { DynamoDBClient, QueryCommand } from "@aws-sdk/client-dynamodb";

const client = new DynamoDBClient({});

export const handler = async () => {
  const result = await client.send(new QueryCommand({
    TableName: "Customers",
    IndexName: "CustomerIdIndex",
    KeyConditionExpression: "customerId = :cid",
    ExpressionAttributeValues: {
      ":cid": { S: "cust-123" }
    }
  }));

  return {
    statusCode: 200,
    body: JSON.stringify(result.Items ?? [])
  };
};
```

Use the document client if you want to avoid manual marshaling. Use the low-level client if you need exact control over the wire format.

**Pagination and large result sets**

Lambda has limited runtime, so you should handle paginated responses carefully. Many AWS APIs return a continuation token:
- DynamoDB: `LastEvaluatedKey`
- S3: `ContinuationToken`
- many list APIs: `NextToken`

For DynamoDB `Query`, continue until `LastEvaluatedKey` is absent or until you reach your own processing limit.

Example pagination loop:

```javascript
import { DynamoDBDocumentClient, QueryCommand } from "@aws-sdk/lib-dynamodb";

export const handler = async () => {
  const items = [];
  let ExclusiveStartKey;

  do {
    const result = await ddb.send(new QueryCommand({
      TableName: process.env.TABLE_NAME,
      IndexName: "CustomerNameIndex",
      KeyConditionExpression: "#n = :name",
      ExpressionAttributeNames: { "#n": "name" },
      ExpressionAttributeValues: { ":name": "Alice" },
      ExclusiveStartKey
    }));

    items.push(...(result.Items ?? []));
    ExclusiveStartKey = result.LastEvaluatedKey;
  } while (ExclusiveStartKey && items.length < 100);

  return {
    statusCode: 200,
    body: JSON.stringify(items)
  };
};
```

Practical guidance:
- cap the number of items you process in one invocation
- store the continuation token if the workflow must resume later
- avoid unbounded loops in Lambda
- prefer asynchronous fan-out if the dataset is large

**Idempotency and retries**

Lambda may retry invocations depending on the trigger and failure mode. Your CRUD code should be safe to run more than once.

Use idempotency when:
- the same event can be delivered multiple times
- a write must not create duplicates
- downstream retries are possible

Common techniques:
- use a deterministic primary key
- use `ConditionExpression` to prevent duplicate inserts
- store a request ID or idempotency token
- make updates commutative when possible

Example conditional write:

```javascript
await ddb.send(new PutCommand({
  TableName: process.env.TABLE_NAME,
  Item: {
    orderId: "order-123",
    status: "PENDING"
  },
  ConditionExpression: "attribute_not_exists(orderId)"
}));
```

If the item already exists, DynamoDB throws a conditional check failure instead of overwriting it.

**S3 CRUD from Lambda**

S3 is object storage, so CRUD maps to object operations:
- create: `PutObject`
- read: `GetObject`
- update: usually another `PutObject` to overwrite the object
- delete: `DeleteObject`

Example with AWS SDK for JavaScript v3:

```javascript
import { S3Client, PutObjectCommand, GetObjectCommand, DeleteObjectCommand } from "@aws-sdk/client-s3";

const s3 = new S3Client({});

export const handler = async () => {
  await s3.send(new PutObjectCommand({
    Bucket: process.env.BUCKET_NAME,
    Key: "reports/daily.json",
    Body: JSON.stringify({ generatedAt: new Date().toISOString() }),
    ContentType: "application/json"
  }));

  const result = await s3.send(new GetObjectCommand({
    Bucket: process.env.BUCKET_NAME,
    Key: "reports/daily.json"
  }));

  await s3.send(new DeleteObjectCommand({
    Bucket: process.env.BUCKET_NAME,
    Key: "reports/daily.json"
  }));

  return { statusCode: 200, body: "S3 operations complete" };
};
```

Remember:
- `GetObject` returns a stream in Node.js
- overwriting an object is the usual â€śupdateâ€ť pattern
- versioning changes delete semantics because a delete marker may be created instead of permanent removal

**RDS and relational databases from Lambda**

Lambda can connect to Amazon RDS or Amazon Aurora using a database driver or ORM (Object-Relational Mapper). In this case, CRUD is performed with SQL statements rather than AWS service APIs.

Typical patterns:
- use a connection pool carefully, because Lambda may reuse execution environments
- keep connections outside the handler when possible to benefit from reuse
- place Lambda in the same VPC (Virtual Private Cloud) as the database if the database is private
- use Secrets Manager or Systems Manager Parameter Store for credentials
- consider RDS Proxy to reduce connection storms

Example with PostgreSQL in Node.js:

```javascript
import { Client } from "pg";

let client;

export const handler = async () => {
  if (!client) {
    client = new Client({
      host: process.env.DB_HOST,
      port: 5432,
      user: process.env.DB_USER,
      password: process.env.DB_PASSWORD,
      database: process.env.DB_NAME,
      ssl: { rejectUnauthorized: true }
    });
    await client.connect();
  }

  await client.query(
    "INSERT INTO customers (customer_id, name) VALUES ($1, $2) ON CONFLICT (customer_id) DO UPDATE SET name = EXCLUDED.name",
    ["cust-123", "Alice"]
  );

  const result = await client.query(
    "SELECT customer_id, name FROM customers WHERE customer_id = $1",
    ["cust-123"]
  );

  return {
    statusCode: 200,
    body: JSON.stringify(result.rows)
  };
};
```

Use TLS (Transport Layer Security) correctly and validate certificates in production. Avoid disabling certificate validation except for controlled local testing.

**Python and boto3 in Lambda**

Python Lambda runtimes commonly include `boto3`, but you should not rely on a specific bundled version for application behavior. Package dependencies when you need version consistency or newer features.

Example:

```python
import os
import json
import boto3

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table(os.environ["TABLE_NAME"])

def handler(event, context):
    table.put_item(
        Item={
            "customerId": event["customerId"],
            "name": event["name"]
        },
        ConditionExpression="attribute_not_exists(customerId)"
    )

    response = table.get_item(
        Key={"customerId": event["customerId"]}
    )

    return {
        "statusCode": 200,
        "body": json.dumps(response.get("Item"))
    }
```

If you need a specific `boto3` or `botocore` version, include it in your deployment package or use a Lambda layer.

**Environment variables and configuration**

Use environment variables for non-secret configuration such as table names, bucket names, and index names. Keep secrets in Secrets Manager or Parameter Store.

Example configuration:
- `TABLE_NAME=Customers`
- `BUCKET_NAME=my-app-reports`
- `INDEX_NAME=CustomerNameIndex`

Example retrieval pattern:

```javascript
const tableName = process.env.TABLE_NAME;
if (!tableName) {
  throw new Error("TABLE_NAME is required");
}
```

This keeps code portable across environments such as dev, test, and prod.

**Error handling and retries**

AWS SDKs already retry some transient failures, but you still need application-level handling.

Handle:
- throttling exceptions
- conditional check failures
- validation errors
- timeouts
- network errors

Practical guidance:
- set reasonable SDK timeouts
- log enough context to diagnose failures
- do not log secrets or full credentials
- distinguish retryable from non-retryable errors

Example retry-aware pattern:

```javascript
try {
  await ddb.send(command);
} catch (err) {
  if (err.name === "ConditionalCheckFailedException") {
    return { statusCode: 409, body: "Duplicate request" };
  }
  throw err;
}
```

**Networking and VPC considerations**

If Lambda accesses a public AWS API endpoint, it can usually do so without VPC attachment. If it must access private resources such as an RDS instance in private subnets, place the function in the VPC.

Important effects of VPC attachment:
- Lambda needs subnet and security group configuration
- the function may require a NAT gateway for outbound internet access
- private service access may need VPC endpoints
- cold starts can be affected by networking setup, though this is less of a concern than it used to be

Use VPC endpoints when you want private access to services like DynamoDB or S3 without traversing the public internet.

**Practical deployment checklist**

1. Create or identify the target data store.
2. Attach an execution role with least-privilege permissions.
3. Set environment variables for resource names and configuration.
4. Package the correct SDK or driver dependencies.
5. Test create, read, update, and delete paths separately.
6. Verify pagination, retries, and error handling.
7. Confirm network access if the target is private.
8. Review logs in Amazon CloudWatch for failures and latency.

**Common exam traps**

- Assuming `Scan` is equivalent to `Query`
- Using a table ARN alone for index-specific DynamoDB access
- Forgetting that low-level DynamoDB clients require typed attribute values
- Disabling TLS certificate validation in production database code
- Relying on a specific bundled `boto3` version in Lambda
- Ignoring idempotency when Lambda retries are possible
- Forgetting that S3 object updates are usually overwrites, not in-place edits

**Quick comparison of CRUD patterns**

| Service | Create | Read | Update | Delete |
|---|---|---|---|---|
| DynamoDB | `PutItem` | `GetItem` / `Query` | `UpdateItem` | `DeleteItem` |
| S3 | `PutObject` | `GetObject` | overwrite with `PutObject` | `DeleteObject` |
| RDS | `INSERT` | `SELECT` | `UPDATE` | `DELETE` |

**What to remember for the exam**

- Lambda uses SDKs or drivers to perform CRUD against AWS services and databases.
- The execution role must allow the exact actions and resources used by the function.
- DynamoDB `Query` is efficient when you know the partition key or use an index.
- Low-level DynamoDB clients require `AttributeValue` types; document clients use native types.
- Pagination, retries, and idempotency matter because Lambda is short-lived and may retry.
- Private databases often require VPC configuration and secure credential handling.
- Use TLS correctly and avoid unsafe production shortcuts.
