## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Idempotency fundamentals**

Idempotency means that repeating the same request produces the same intended outcome without creating duplicate side effects. In distributed systems, this matters because clients, SDKs, API Gateway, Lambda, and downstream services may retry requests after timeouts, throttling, or transient failures.

For AWS developer scenarios, idempotency is most often implemented when you need to protect writes such as:

- Creating orders
- Submitting payments
- Starting workflows
- Provisioning resources
- Writing events that must not be duplicated

A practical design usually stores a request identifier and the result of the first successful processing. If the same request arrives again, you return the original result instead of executing the operation again.

**Why idempotency matters in AWS**

AWS services often favor at-least-once delivery or retry behavior. That means your code must tolerate duplicate invocations.

- **Lambda retries** can happen when an event source or integration replays a request.
- **Client retries** can occur when a network timeout hides a successful write.
- **Asynchronous processing** can deliver the same message more than once.
- **Partial failures** can leave a system unsure whether the first attempt completed.

Idempotency is not the same as deduplication in a queue. Deduplication reduces duplicates in transit; idempotency ensures your business operation remains correct even if duplicates still occur.

**Common idempotency patterns**

The most common pattern is to use a unique idempotency key supplied by the caller or derived from the request.

- **Client-generated key**: The caller sends a UUID or business key such as `order-12345`.
- **Request hash**: The system hashes the request payload and uses the hash as the key.
- **Natural key**: A domain identifier such as `paymentId` or `reservationId`.

A robust implementation stores a record with:

- The idempotency key
- Request status such as `IN_PROGRESS`, `COMPLETED`, or `FAILED`
- The response payload or a pointer to it
- A timestamp for expiration

If the same key is seen again:

- Return the stored response if processing already completed
- Reject or wait if another request is still in progress
- Allow a new attempt if the record has expired

**DynamoDB as an idempotency store**

Amazon DynamoDB is a common choice because it supports low-latency key-value access, conditional writes, and TTL-based expiration.

A typical table design uses the idempotency key as the partition key.

```json
{
  "idempotencyKey": "order-12345",
  "status": "COMPLETED",
  "responseCode": 201,
  "responseBody": "{\"orderId\":\"o-789\",\"status\":\"created\"}",
  "createdAt": 1714147200,
  "expiresAt": 1714233600
}
```

Important DynamoDB behaviors:

- **Conditional writes** let you create the record only if it does not already exist.
- **Strongly consistent reads** can help when you need to verify the latest state immediately after a write.
- **TTL** removes expired items asynchronously, which is useful for cleanup but not immediate.

A common write flow is:

1. Attempt a conditional `PutItem` with `attribute_not_exists(idempotencyKey)`.
2. If the write succeeds, process the request.
3. Update the item with the final status and response.
4. If the write fails because the key already exists, read the stored result and return it.

Example using the AWS CLI:

```bash
aws dynamodb put-item \
  --table-name IdempotencyTable \
  --item '{
    "idempotencyKey": {"S": "order-12345"},
    "status": {"S": "IN_PROGRESS"},
    "createdAt": {"N": "1714147200"},
    "expiresAt": {"N": "1714233600"}
  }' \
  --condition-expression "attribute_not_exists(idempotencyKey)"
```

**Handling in-progress requests**

A duplicate request may arrive while the first request is still running. You need a strategy for that state.

- Return a `409 Conflict` or similar response if the operation is still in progress.
- Poll the record until completion if the client can tolerate waiting.
- Use a short lock timeout so abandoned in-progress records do not block forever.

A practical pattern is to store a lease or lock expiration separate from the business TTL. That prevents a crashed worker from holding the key indefinitely.

**TTL in DynamoDB**

TTL stands for **Time to Live**. In DynamoDB, TTL marks an item for automatic deletion after a specified epoch time in a designated attribute, commonly `expiresAt`.

TTL is useful for:

- Cleaning up idempotency records after the retry window ends
- Removing temporary session or lock records
- Expiring cached results
- Reducing storage cost for short-lived data

TTL is not immediate. DynamoDB deletes expired items asynchronously, so an expired item may remain visible for some time after the expiration timestamp.

Example item with TTL:

```json
{
  "idempotencyKey": "order-12345",
  "status": "COMPLETED",
  "responseCode": 201,
  "expiresAt": 1714233600
}
```

Key TTL considerations:

- The TTL attribute must be a Unix epoch time in seconds.
- Expiration is best-effort, not a precise timer.
- You should not rely on TTL for security-sensitive deletion deadlines.
- Expired items may still be returned by reads until DynamoDB removes them.

**When to use TTL with idempotency**

Use TTL to define how long you want to remember a request.

- Short TTLs work for retry windows such as minutes or hours.
- Longer TTLs work when clients may retry much later.
- The TTL should match the maximum period during which a duplicate request could reasonably arrive.

If the TTL is too short:

- A late retry may be treated as a brand-new request
- Duplicate side effects can occur

If the TTL is too long:

- Storage costs increase
- Old keys may block legitimate replays longer than necessary

**Example architecture**

A common serverless pattern is:

- API Gateway receives a request
- Lambda validates the payload and extracts an idempotency key
- Lambda writes an `IN_PROGRESS` record to DynamoDB with a conditional write
- Lambda performs the business action
- Lambda updates the record to `COMPLETED` and stores the response
- TTL eventually removes the record after the retry window

This pattern is especially useful for order submission, payment initiation, and workflow start endpoints.

```text
Client -> API Gateway -> Lambda -> DynamoDB (idempotency record)
                           |
                           +-> SQS / Step Functions / downstream service
```

**Practical Lambda example**

A simplified Python example shows the core idea.

```python
import json
import time
import boto3
from botocore.exceptions import ClientError

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("IdempotencyTable")

def handler(event, context):
    key = event["headers"].get("Idempotency-Key")
    if not key:
        return {"statusCode": 400, "body": "Missing Idempotency-Key"}

    now = int(time.time())
    ttl = now + 86400

    try:
        table.put_item(
            Item={
                "idempotencyKey": key,
                "status": "IN_PROGRESS",
                "createdAt": now,
                "expiresAt": ttl
            },
            ConditionExpression="attribute_not_exists(idempotencyKey)"
        )
    except ClientError as e:
        if e.response["Error"]["Code"] == "ConditionalCheckFailedException":
            existing = table.get_item(Key={"idempotencyKey": key}).get("Item")
            if existing and existing.get("status") == "COMPLETED":
                return {
                    "statusCode": existing.get("responseCode", 200),
                    "body": existing.get("responseBody", "{}")
                }
            return {"statusCode": 409, "body": "Request already in progress"}
        raise

    # Perform the business action here
    result = {"orderId": "o-789", "status": "created"}

    table.update_item(
        Key={"idempotencyKey": key},
        UpdateExpression="SET #s = :s, responseCode = :c, responseBody = :b",
        ExpressionAttributeNames={"#s": "status"},
        ExpressionAttributeValues={
            ":s": "COMPLETED",
            ":c": 201,
            ":b": json.dumps(result)
        }
    )

    return {"statusCode": 201, "body": json.dumps(result)}
```

This example shows the core mechanics, but production code should also handle:

- Validation failures before reserving the key
- Cleanup of abandoned `IN_PROGRESS` records
- Safe serialization of large responses
- Retries around transient DynamoDB errors

**Conditional writes and race conditions**

Conditional writes are the main tool for preventing two workers from claiming the same idempotency key at the same time.

- `attribute_not_exists(pk)` ensures only the first writer succeeds.
- A second writer receives a conditional check failure.
- This avoids a race where both workers think they own the request.

If you need to update state safely, use a conditional update based on the current status.

```bash
aws dynamodb update-item \
  --table-name IdempotencyTable \
  --key '{"idempotencyKey":{"S":"order-12345"}}' \
  --update-expression "SET #s = :completed" \
  --condition-expression "#s = :inprogress" \
  --expression-attribute-names '{"#s":"status"}' \
  --expression-attribute-values '{":completed":{"S":"COMPLETED"},":inprogress":{"S":"IN_PROGRESS"}}'
```

**Idempotency in event-driven systems**

Event-driven architectures often need idempotency because the same event can be delivered more than once.

- **SQS** standard queues can deliver duplicates.
- **SNS** fanout can trigger multiple consumers, each of which must be idempotent.
- **EventBridge** targets may retry deliveries.
- **Lambda event sources** may replay records after failures.

For these systems, use the event ID, message ID, or a business key as the idempotency key. Store the processing result before acknowledging success when possible.

**TTL versus application-level expiration**

TTL is convenient, but it is not a substitute for application logic.

- Use application-level checks when you must reject expired requests immediately.
- Use TTL to clean up records after they are no longer needed.
- Combine both when you need deterministic behavior and eventual cleanup.

For example, your Lambda can reject a request if `expiresAt < now`, even if DynamoDB has not deleted the item yet.

**Operational considerations**

Idempotency stores need monitoring and careful sizing.

- Watch for hot partitions if many requests use the same key pattern.
- Ensure the partition key has enough cardinality.
- Track conditional check failures to detect duplicate traffic or client retry storms.
- Monitor TTL deletion lag if cleanup timing matters.
- Use CloudWatch metrics and logs to observe request outcomes.

A few practical tips:

- Keep the idempotency record small to reduce read and write cost.
- Store only the response data you need to replay.
- Use a separate table if idempotency traffic is very different from your main domain data.
- Choose a TTL that matches the business retry horizon, not just the technical retry interval.

**Exam-focused distinctions**

For exam scenarios, distinguish these ideas carefully:

- **Idempotency**: repeated requests produce the same effect.
- **Deduplication**: duplicate messages are filtered out, usually by the transport or queue.
- **TTL**: data expires after a time and is removed later.
- **Conditional write**: atomic protection against duplicate creation.
- **At-least-once delivery**: duplicates are possible, so your code must be safe.

If a question asks how to prevent duplicate order creation in Lambda, the best answer is usually a DynamoDB-backed idempotency key with conditional writes and TTL-based cleanup.

**Troubleshooting patterns**

If duplicates still appear, check the following:

- The client is generating a stable idempotency key for retries.
- The key is included on every retry attempt.
- The conditional write is actually enforced.
- The business action is not executed before the reservation succeeds.
- The TTL is long enough to cover the retry window.
- The response replay logic returns the stored result correctly.

If expired records are still present, remember that DynamoDB TTL deletion is asynchronous. That is expected behavior.

**Quick reference**

- Use **idempotency** to make retries safe.
- Use **DynamoDB conditional writes** to reserve a request key atomically.
- Use **TTL** to expire idempotency records after the retry window.
- Store the **status** and response so duplicates can be answered consistently.
- Treat TTL as eventual cleanup, not immediate deletion.
