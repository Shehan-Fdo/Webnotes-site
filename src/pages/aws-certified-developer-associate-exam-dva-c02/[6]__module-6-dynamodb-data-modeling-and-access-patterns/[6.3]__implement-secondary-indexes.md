## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Secondary indexes in DynamoDB**

Amazon DynamoDB uses a primary key for direct item access, but many applications need alternate access patterns. Secondary indexes let you query the same data by different attributes without scanning the base table. For exam purposes, focus on the difference between **local secondary indexes** and **global secondary indexes**, because their creation rules, capacity behavior, and query limits are different.

A secondary index stores a projection of table items. You query the index with `Query`, not `Scan`, when you want efficient access by the index key. The index can include only selected attributes, which reduces storage and can improve read efficiency depending on the projection type.

**Core concepts**

- **Primary key design** drives access patterns first; indexes are a secondary tool when one table must support multiple query shapes.
- **Query** on an index is efficient because DynamoDB uses the index key rather than reading the whole table.
- **Projection** controls which non-key attributes are copied into the index.
- **Sparse index** behavior is common with GSIs: only items that contain the indexed attribute appear in the index.

**Local secondary indexes (LSIs)**

An LSI uses the same partition key as the base table but a different sort key. This means you can change the sort order or query by an alternate range attribute while staying within the same partition key value.

LSIs have important constraints that are often tested:

- You can create an LSI only when the table uses a **composite primary key**.
- You must define the LSI at table creation time; you cannot add an LSI later.
- An LSI shares the table’s partition key, so it is constrained by the base table’s partition key distribution.
- An LSI shares the table’s provisioned throughput with the base table.
- The maximum item collection size for a given partition key is **10 GB** across the table and all LSIs.

Because LSIs are tied to the base table partition key, they are useful when you need multiple sort-order views of the same logical item collection. They are not a good fit when you need a new partitioning strategy.

**Example LSI use case**

Suppose you store orders by `CustomerId` and want to query by `OrderDate` or `Status` within each customer.

```json
{
  "TableName": "Orders",
  "KeySchema": [
    { "AttributeName": "CustomerId", "KeyType": "HASH" },
    { "AttributeName": "OrderId", "KeyType": "RANGE" }
  ],
  "AttributeDefinitions": [
    { "AttributeName": "CustomerId", "AttributeType": "S" },
    { "AttributeName": "OrderId", "AttributeType": "S" },
    { "AttributeName": "OrderDate", "AttributeType": "S" }
  ],
  "LocalSecondaryIndexes": [
    {
      "IndexName": "CustomerOrderDateIndex",
      "KeySchema": [
        { "AttributeName": "CustomerId", "KeyType": "HASH" },
        { "AttributeName": "OrderDate", "KeyType": "RANGE" }
      ],
      "Projection": {
        "ProjectionType": "INCLUDE",
        "NonKeyAttributes": ["Status", "TotalAmount"]
      }
    }
  ]
}
```

You would query the LSI when you need all orders for one customer sorted by date. You would not use it to distribute writes across a different partition key, because the partition key remains `CustomerId`.

**Global secondary indexes (GSIs)**

A GSI uses a different partition key and optionally a different sort key from the base table. This makes GSIs the main tool for new access patterns that do not match the table’s primary key.

GSIs have several exam-relevant behaviors:

- You can add a GSI after table creation.
- DynamoDB backfills the index when you create it on an existing table.
- During backfill, the index is populated from existing table data and may take time to become fully usable for all items.
- A GSI has its own capacity settings in **provisioned** mode, so it can become a separate throttling point from the base table.
- In **on-demand** mode, the GSI scales independently, but it still has its own performance characteristics and can throttle if access patterns are uneven or hot.
- GSI key attributes must be **top-level scalar attributes**; they cannot be nested document attributes or sets.

Because a GSI can use a different partition key, it can solve access patterns that would otherwise require a scan. It is also the right choice when you need to query by an attribute that is not part of the table key schema.

**Example GSI use case**

If you store support tickets by `TicketId` but need to look them up by `AssigneeId`, create a GSI on `AssigneeId`.

```json
{
  "TableName": "Tickets",
  "AttributeDefinitions": [
    { "AttributeName": "TicketId", "AttributeType": "S" },
    { "AttributeName": "AssigneeId", "AttributeType": "S" },
    { "AttributeName": "CreatedAt", "AttributeType": "S" }
  ],
  "KeySchema": [
    { "AttributeName": "TicketId", "KeyType": "HASH" }
  ],
  "GlobalSecondaryIndexes": [
    {
      "IndexName": "AssigneeCreatedAtIndex",
      "KeySchema": [
        { "AttributeName": "AssigneeId", "KeyType": "HASH" },
        { "AttributeName": "CreatedAt", "KeyType": "RANGE" }
      ],
      "Projection": {
        "ProjectionType": "KEYS_ONLY"
      },
      "ProvisionedThroughput": {
        "ReadCapacityUnits": 5,
        "WriteCapacityUnits": 5
      }
    }
  ]
}
```

This index lets you query all tickets assigned to one person and sort them by creation time.

**Sparse indexes**

A sparse index contains only items that have the indexed attribute. This is a common and powerful GSI pattern because it reduces index size and focuses the index on a subset of items.

Use a sparse GSI when only some items need to be queried by a particular access pattern.

- If every item has `Status`, a full index on `Status` may be appropriate.
- If only open incidents have `EscalationOwner`, a sparse GSI on `EscalationOwner` includes only those open incidents.
- If you want to find “active” records, you can write only active items with an attribute such as `ActivePartitionKey` or `GSI1PK`.

**Sparse GSI example**

Imagine a `Users` table where only premium users have a `BillingAccountId`. A sparse GSI on `BillingAccountId` lets you query only premium users without indexing every user.

```json
{
  "TableName": "Users",
  "Item": {
    "UserId": "u-123",
    "Email": "user@example.com",
    "BillingAccountId": "ba-789",
    "Plan": "premium"
  }
}
```

Items without `BillingAccountId` do not appear in the index. This is useful when the query target is a subset of the table and you want to avoid a larger, noisier index.

**Projection types**

Projection determines which attributes are copied into the index.

| Projection type | What it includes | When to use it |
|---|---|---|
| `KEYS_ONLY` | Only the table and index keys | Smallest index footprint; use when you only need keys to fetch the base item |
| `INCLUDE` | Keys plus selected non-key attributes | Good balance when queries need a few extra fields |
| `ALL` | All table attributes | Simplest, but largest storage and write cost |

Projection affects index size and read behavior, but it does not change the need to design the key schema around the query pattern.

**Capacity and throttling behavior**

Capacity behavior is a frequent exam topic because LSIs and GSIs behave differently.

- **LSIs** share the base table’s provisioned throughput.
- **GSIs** have their own throughput in provisioned mode.
- A GSI can throttle even when the base table is healthy.
- A hot partition in the base table can still affect LSI performance because the LSI uses the same partition key distribution.
- Writes to the base table must also update any affected indexes, so index design increases write cost.

In practice, if a GSI is heavily queried, you may need to provision it separately or redesign the key to spread traffic more evenly. If an LSI is overloaded, the fix is usually at the table design level, because the LSI cannot change partitioning.

**Querying indexes**

You query a secondary index with `Query` and specify the index name. You can use key conditions on the index key attributes, and you can add a filter expression for non-key attributes.

```bash
aws dynamodb query \
  --table-name Tickets \
  --index-name AssigneeCreatedAtIndex \
  --key-condition-expression "AssigneeId = :a and CreatedAt >= :d" \
  --expression-attribute-values '{":a":{"S":"agent-42"},":d":{"S":"2026-01-01T00:00:00Z"}}'
```

A `FilterExpression` is applied after DynamoDB reads the matching items from the table or index. It does **not** reduce read capacity consumption for those items, so it is not a substitute for a good key design.

> `FilterExpression` removes items from the response after the read. It does not make the query cheaper in read capacity units.

**When to use `FilterExpression`**

- Use it for small post-read refinements when the key condition already narrows the result set well.
- Avoid it when you are trying to compensate for a poor index design.
- Prefer a GSI or LSI when the attribute should be part of the access pattern.

**LSI versus GSI**

| Feature | LSI | GSI |
|---|---|---|
| Partition key | Same as base table | Different from base table |
| Sort key | Different from base table | Optional, different from base table |
| Creation time | Only at table creation | Can be added later |
| Backfill | No separate backfill after creation | Backfills existing data when added |
| Throughput | Shares table throughput | Own throughput in provisioned mode |
| Item collection limit | 10 GB per partition key | No LSI-style item collection limit |
| Best use | Alternate sort order within one partition key | New access pattern or new partitioning strategy |

**Practical design patterns**

- Use an **LSI** when you need multiple ways to sort items for the same partition key, such as by date, status, or priority within one customer.
- Use a **GSI** when you need to query by a different entity, such as `Email`, `AssigneeId`, `Category`, or `Status`.
- Use a **sparse GSI** when only a subset of items should be queryable through the index.
- Use `KEYS_ONLY` when the index is only a lookup path to the base item.
- Use `INCLUDE` when you want to avoid fetching the base table for a few frequently read attributes.

**Operational considerations**

When you add a GSI to an existing table, DynamoDB builds it in the background. During this time, the index may not immediately reflect all historical items. Plan for this when deploying schema changes to production.

For write-heavy workloads, remember that each indexed attribute update adds write overhead. If you update an attribute that is part of a GSI key, DynamoDB must update the index entry as well. This can increase latency and cost.

If you need to query nested data, consider whether you can denormalize the needed value into a top-level scalar attribute. Since GSI keys must be top-level scalar attributes, nested document paths are not valid index keys.

**CLI example: create a table with an LSI**

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions \
      AttributeName=CustomerId,AttributeType=S \
      AttributeName=OrderId,AttributeType=S \
      AttributeName=OrderDate,AttributeType=S \
  --key-schema \
      AttributeName=CustomerId,KeyType=HASH \
      AttributeName=OrderId,KeyType=RANGE \
  --local-secondary-indexes '[
    {
      "IndexName": "CustomerOrderDateIndex",
      "KeySchema": [
        {"AttributeName":"CustomerId","KeyType":"HASH"},
        {"AttributeName":"OrderDate","KeyType":"RANGE"}
      ],
      "Projection": {"ProjectionType":"ALL"}
    }
  ]' \
  --provisioned-throughput ReadCapacityUnits=5,WriteCapacityUnits=5
```

**CLI example: add a GSI to an existing table**

```bash
aws dynamodb update-table \
  --table-name Tickets \
  --attribute-definitions \
      AttributeName=AssigneeId,AttributeType=S \
      AttributeName=CreatedAt,AttributeType=S \
  --global-secondary-index-updates '[
    {
      "Create": {
        "IndexName": "AssigneeCreatedAtIndex",
        "KeySchema": [
          {"AttributeName":"AssigneeId","KeyType":"HASH"},
          {"AttributeName":"CreatedAt","KeyType":"RANGE"}
        ],
        "Projection": {"ProjectionType":"KEYS_ONLY"},
        "ProvisionedThroughput": {
          "ReadCapacityUnits": 5,
          "WriteCapacityUnits": 5
        }
      }
    }
  ]'
```

**Exam-focused takeaways**

- **LSIs** are for alternate sort keys on the same partition key and must be created with the table.
- **GSIs** are for new access patterns and can be added later.
- **Sparse GSIs** are a common pattern when only some items should appear in the index.
- **FilterExpression** does not reduce read capacity because it is applied after the read.
- **GSI keys** must be top-level scalar attributes.
- **LSIs** are limited by the 10 GB item collection size per partition key and share table throughput.

**Quick decision guide**

- Need a different partition key? Use a **GSI**.
- Need a different sort key under the same partition key? Use an **LSI**.
- Need to query only a subset of items? Consider a **sparse GSI**.
- Need to add the index after launch? Use a **GSI**.
- Need to keep all access within one partition key and stay under 10 GB? An **LSI** may fit.
