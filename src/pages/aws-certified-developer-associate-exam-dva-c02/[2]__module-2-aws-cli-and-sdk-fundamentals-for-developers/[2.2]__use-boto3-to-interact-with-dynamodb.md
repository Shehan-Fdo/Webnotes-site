## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**DynamoDB access patterns with boto3**

`boto3` is the AWS SDK for Python. For DynamoDB, you usually choose between the **resource** interface and the **client** interface:

- **Resource API** is higher level and more Pythonic.
- **Client API** is lower level and closer to the raw service model.
- For exam prep, know both, but prefer the resource API for readability unless you need explicit request/response control.

A practical mental model helps:

- Use `GetItem` when you know the full primary key.
- Use `Query` when you know the partition key and want matching sort key values.
- Use `Scan` when you must inspect the whole table or index.
- Use `BatchGetItem` for many known keys across one or more tables.
- Use `BatchWriteItem` for bulk puts and deletes.

**Resource vs client**

The resource interface uses native Python types and condition helpers. The client interface uses DynamoDB’s typed attribute format and is useful when you need exact control over payloads.

```python
import boto3

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("Orders")

client = boto3.client("dynamodb")
```

With the resource API, you pass normal Python values:

```python
table.put_item(
    Item={
        "UserId": "u123",
        "OrderId": "o456",
        "Status": "PENDING",
        "Total": 42.50
    }
)
```

With the client API, you use DynamoDB attribute maps:

```python
client.put_item(
    TableName="Orders",
    Item={
        "UserId": {"S": "u123"},
        "OrderId": {"S": "o456"},
        "Status": {"S": "PENDING"},
        "Total": {"N": "42.5"}
    }
)
```

Use the client when you need:

- Fine-grained control over expressions and low-level response fields.
- Direct access to service-specific shapes.
- Consistent handling of typed attributes in generic tooling.

Use the resource when you want:

- Cleaner application code.
- Easier condition expressions with `boto3.dynamodb.conditions`.
- Less boilerplate for common CRUD operations.

**Table design and key access**

DynamoDB performance depends on access patterns. Your code should match the table key schema.

- A **partition key** identifies the item group.
- A **sort key** orders items within the partition.
- `Query` can use equality on the partition key and comparison operators on the sort key.
- `begins_with` is valid only on the sort key in a key condition expression.

Example with the resource API:

```python
from boto3.dynamodb.conditions import Key

response = table.query(
    KeyConditionExpression=Key("UserId").eq("u123") & Key("OrderId").begins_with("o")
)
```

This pattern is only valid if `UserId` is the partition key and `OrderId` is the sort key. Do not use `begins_with` on a non-key attribute in `KeyConditionExpression`; for non-key attributes, use `FilterExpression` instead.

**GetItem and PutItem**

`GetItem` is the direct single-item read when you know the full primary key. It returns at most one item.

```python
response = table.get_item(
    Key={
        "UserId": "u123",
        "OrderId": "o456"
    }
)

item = response.get("Item")
```

`PutItem` creates or replaces an item with the same primary key. If you omit a condition, it can overwrite an existing item.

```python
table.put_item(
    Item={
        "UserId": "u123",
        "OrderId": "o456",
        "Status": "PENDING"
    }
)
```

To prevent accidental overwrite, add a condition:

```python
from boto3.dynamodb.conditions import Attr

table.put_item(
    Item={
        "UserId": "u123",
        "OrderId": "o456",
        "Status": "PENDING"
    },
    ConditionExpression=Attr("OrderId").not_exists()
)
```

That condition is useful for idempotent create flows where you want the write to fail if the item already exists.

**UpdateItem patterns**

`UpdateItem` is the preferred way to modify part of an item without rewriting the whole record. It supports `SET`, `REMOVE`, `ADD`, and `DELETE` update actions.

```python
from boto3.dynamodb.conditions import Attr

table.update_item(
    Key={
        "UserId": "u123",
        "OrderId": "o456"
    },
    UpdateExpression="SET #st = :new_status, UpdatedAt = :ts",
    ExpressionAttributeNames={
        "#st": "Status"
    },
    ExpressionAttributeValues={
        ":new_status": "SHIPPED",
        ":ts": "2026-04-26T12:00:00Z"
    },
    ConditionExpression=Attr("Status").eq("PENDING")
)
```

Use conditional updates when you need optimistic concurrency or state transitions.

Common update actions:

- `SET` assigns or replaces an attribute value.
- `REMOVE` deletes an attribute.
- `ADD` increments numbers or adds elements to sets.
- `DELETE` removes elements from sets.

Example increment:

```python
table.update_item(
    Key={"UserId": "u123", "OrderId": "o456"},
    UpdateExpression="ADD RetryCount :inc",
    ExpressionAttributeValues={":inc": 1}
)
```

**DeleteItem and conditional deletes**

`DeleteItem` removes an item by key. Add a condition when deletion should only happen in a specific state.

```python
table.delete_item(
    Key={
        "UserId": "u123",
        "OrderId": "o456"
    },
    ConditionExpression=Attr("Status").eq("CANCELLED")
)
```

This is useful when you want to prevent deleting active records or enforce workflow rules.

**Query vs Scan**

`Query` is the primary read pattern for DynamoDB. It is efficient because it targets a single partition key value and can optionally narrow by sort key.

```python
from boto3.dynamodb.conditions import Key

response = table.query(
    KeyConditionExpression=Key("UserId").eq("u123"),
    FilterExpression=Attr("Status").eq("SHIPPED")
)
```

Important distinction:

- `KeyConditionExpression` determines which items DynamoDB reads from the key index.
- `FilterExpression` is applied after the read, so it does not reduce read capacity for the items already evaluated.
- `ProjectionExpression` reduces the response payload, not the amount of data DynamoDB reads from the table or index.

`Scan` reads every item in the table or index, so it is expensive at scale.

```python
response = table.scan(
    FilterExpression=Attr("Status").eq("SHIPPED"),
    ProjectionExpression="UserId, OrderId, Status"
)
```

Use `Scan` for:

- Administrative tasks.
- Small tables.
- One-time migrations or audits.

Avoid `Scan` for:

- Hot paths.
- User-facing requests.
- Large production tables unless you have no better access pattern.

**Pagination and `LastEvaluatedKey`**

DynamoDB returns results in pages. A response may include `LastEvaluatedKey`, which means more data is available.

`Limit` controls how many items DynamoDB evaluates before applying any filter expression. The number of items returned can be lower if a filter removes some of them.

```python
response = table.query(
    KeyConditionExpression=Key("UserId").eq("u123"),
    Limit=10
)

items = response.get("Items", [])
last_key = response.get("LastEvaluatedKey")
```

To fetch all pages, loop until `LastEvaluatedKey` is absent:

```python
from boto3.dynamodb.conditions import Key

items = []
kwargs = {
    "KeyConditionExpression": Key("UserId").eq("u123")
}

while True:
    response = table.query(**kwargs)
    items.extend(response.get("Items", []))

    last_key = response.get("LastEvaluatedKey")
    if not last_key:
        break

    kwargs["ExclusiveStartKey"] = last_key
```

For scans, the pattern is the same:

```python
items = []
kwargs = {"ProjectionExpression": "UserId, OrderId, Status"}

while True:
    response = table.scan(**kwargs)
    items.extend(response.get("Items", []))

    last_key = response.get("LastEvaluatedKey")
    if not last_key:
        break

    kwargs["ExclusiveStartKey"] = last_key
```

**BatchGetItem**

Use `BatchGetItem` when you know many exact keys and want to fetch them in one call. The request can span multiple tables, and DynamoDB may return unprocessed keys that you must retry.

Important exam points:

- Items can be returned in any order.
- Duplicate keys in the request are not allowed.
- You must retry `UnprocessedKeys` until the request completes.

Low-level client example:

```python
response = client.batch_get_item(
    RequestItems={
        "Orders": {
            "Keys": [
                {"UserId": {"S": "u123"}, "OrderId": {"S": "o1"}},
                {"UserId": {"S": "u123"}, "OrderId": {"S": "o2"}}
            ],
            "ProjectionExpression": "UserId, OrderId, Status"
        }
    }
)

items = response["Responses"].get("Orders", [])
unprocessed = response.get("UnprocessedKeys", {})
```

A retry loop should keep sending only the unprocessed keys:

```python
request_items = {
    "Orders": {
        "Keys": [
            {"UserId": {"S": "u123"}, "OrderId": {"S": "o1"}},
            {"UserId": {"S": "u123"}, "OrderId": {"S": "o2"}}
        ]
    }
}

while request_items:
    response = client.batch_get_item(RequestItems=request_items)
    # process response["Responses"]

    request_items = response.get("UnprocessedKeys", {})
```

**BatchWriteItem**

Use `BatchWriteItem` for bulk inserts and deletes. Each table maps to a list of wrapped write requests.

```python
client.batch_write_item(
    RequestItems={
        "Orders": [
            {
                "PutRequest": {
                    "Item": {
                        "UserId": {"S": "u123"},
                        "OrderId": {"S": "o789"},
                        "Status": {"S": "PENDING"}
                    }
                }
            },
            {
                "DeleteRequest": {
                    "Key": {
                        "UserId": {"S": "u123"},
                        "OrderId": {"S": "o456"}
                    }
                }
            }
        ]
    }
)
```

Key constraints:

- Each request item must be wrapped exactly as `PutRequest` or `DeleteRequest`.
- `BatchWriteItem` does not support conditional writes.
- You must handle `UnprocessedItems` and retry them.
- The operation is not transactional.

Use it for:

- Bulk imports.
- Cleanup jobs.
- Backfills where occasional retries are acceptable.

Do not use it when you need:

- All-or-nothing behavior.
- Conditional logic per item.
- Strong consistency across multiple writes.

**Conditional writes and optimistic locking**

Conditional expressions help you protect data integrity. They are common in concurrent applications.

Typical patterns:

- Prevent duplicate creation.
- Update only if the item is in an expected state.
- Implement optimistic locking with a version attribute.

Example version check:

```python
table.update_item(
    Key={"UserId": "u123", "OrderId": "o456"},
    UpdateExpression="SET Version = :next, Status = :status",
    ConditionExpression=Attr("Version").eq(3),
    ExpressionAttributeValues={
        ":next": 4,
        ":status": "SHIPPED"
    }
)
```

If the condition fails, DynamoDB raises `ConditionalCheckFailedException`. Treat that as an application-level conflict, not a retryable throughput error.

**Global Secondary Indexes and Local Secondary Indexes**

A **Global Secondary Index** (GSI) can use a different partition key and sort key from the base table. A **Local Secondary Index** (LSI) shares the base table partition key but uses a different sort key.

Practical differences:

| Feature | GSI | LSI |
|---|---|---|
| Partition key | Different from base table allowed | Same as base table |
| Sort key | Different from base table allowed | Different from base table |
| Query access | Query by index key schema | Query by table partition key only |
| Throughput model | Separate from base table | Shares base table throughput |
| Creation time | Can be added after table creation | Must be defined at table creation |

Use a GSI when your application needs a new access pattern that the base table does not support.

Example query against a GSI:

```python
response = table.query(
    IndexName="StatusIndex",
    KeyConditionExpression=Key("Status").eq("SHIPPED")
)
```

You can still project or filter on non-key attributes, but the key condition itself must use the index key attributes.

**Error handling and retries**

DynamoDB operations can fail for different reasons, and your code should distinguish them.

Common exceptions:

- `ProvisionedThroughputExceededException` indicates capacity pressure.
- `ThrottlingException` indicates throttling at the service or account level.
- `ConditionalCheckFailedException` indicates a business rule or concurrency conflict.
- `ResourceNotFoundException` often means the table or index name is wrong.
- `ValidationException` often means the request shape or expression is invalid.

Example retry pattern with exponential backoff:

```python
import time
import random
import botocore

def retryable_write():
    delay = 0.1
    for attempt in range(5):
        try:
            table.put_item(
                Item={"UserId": "u123", "OrderId": "o999", "Status": "NEW"},
                ConditionExpression=Attr("OrderId").not_exists()
            )
            return
        except botocore.exceptions.ClientError as e:
            code = e.response["Error"]["Code"]

            if code in {
                "ProvisionedThroughputExceededException",
                "ThrottlingException"
            }:
                time.sleep(delay + random.random() * delay)
                delay *= 2
                continue

            if code == "ConditionalCheckFailedException":
                raise

            raise
```

Use retry logic for throttling, but do not blindly retry conditional failures. A condition failure usually means the request should branch or stop.

**Practical CLI and debugging workflow**

Even when your application uses boto3, the AWS CLI is useful for verifying table behavior and request shapes.

```bash
aws dynamodb get-item \
  --table-name Orders \
  --key '{"UserId":{"S":"u123"},"OrderId":{"S":"o456"}}'
```

Query with a key condition:

```bash
aws dynamodb query \
  --table-name Orders \
  --key-condition-expression "UserId = :u" \
  --expression-attribute-values '{":u":{"S":"u123"}}'
```

Scan with a filter:

```bash
aws dynamodb scan \
  --table-name Orders \
  --filter-expression "Status = :s" \
  --expression-attribute-values '{":s":{"S":"SHIPPED"}}'
```

Use the CLI to confirm:

- The table name and index name are correct.
- The key schema matches your code.
- Your expressions are syntactically valid.
- Pagination and filtering behave as expected.

**Common exam traps**

Watch for these distinctions:

- `Query` is not the same as `Scan`.
- `FilterExpression` does not replace a key condition.
- `ProjectionExpression` reduces payload size, not necessarily read capacity.
- `Limit` applies before filtering.
- `BatchGetItem` is for reads only.
- `BatchWriteItem` is not transactional and does not support conditions.
- `begins_with` belongs in a sort key condition, not on arbitrary attributes.
- Low-level client calls require typed attribute values.
- Resource API calls accept native Python types.

**End-to-end application flow**

A typical DynamoDB-backed API often follows this sequence:

1. Read the item with `GetItem` or `Query`.
2. Validate the current state in application code.
3. Apply a conditional `UpdateItem` or `PutItem`.
4. Handle `ConditionalCheckFailedException` as a conflict.
5. Retry only throttling-related failures with backoff.
6. Use `BatchGetItem` or `BatchWriteItem` for bulk operations.

This pattern keeps your application safe under concurrency and efficient under load.

**Quick reference**

- **Single known item**: `get_item`
- **Many known items**: `batch_get_item`
- **Partition key plus optional sort key logic**: `query`
- **Full table or index traversal**: `scan`
- **Partial modification**: `update_item`
- **Bulk insert/delete**: `batch_write_item`
- **Concurrency protection**: `ConditionExpression`

**Troubleshooting scenarios**

If a query returns no items:

- Confirm the partition key value is exact.
- Check whether you are querying the table or the correct index.
- Verify the sort key condition is valid for the schema.
- Remember that filters do not create matches; they only remove already-read items.

If a write fails unexpectedly:

- Check for a condition failure.
- Confirm the key attributes are present.
- Verify attribute names do not conflict with reserved words.
- Inspect `ValidationException` details for malformed expressions.

If you see throttling:

- Reduce request rate.
- Add exponential backoff with jitter.
- Revisit partition key distribution to avoid hot partitions.
- Consider whether your access pattern needs a GSI or a different data model.
