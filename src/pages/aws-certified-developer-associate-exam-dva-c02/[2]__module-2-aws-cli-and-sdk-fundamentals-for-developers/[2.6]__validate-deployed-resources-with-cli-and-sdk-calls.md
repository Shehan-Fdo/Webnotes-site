## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Purpose of validation**

After you deploy infrastructure or application code, you need to confirm that the resource exists, is configured as expected, and is actually usable. In AWS, that usually means combining:

- **AWS CLI** calls for quick inspection and scripting
- **AWS SDK** calls for application-level checks and automation
- **Waiters** for asynchronous state transitions
- **Service-specific verification** for the resource type you deployed

Validation is not just “does the API return something.” You often need to confirm the resource is in the right state, in the right region, with the right permissions, and with the expected downstream dependencies.

**What you should verify**

A good post-deployment check usually answers these questions:

- Did the resource get created?
- Is it in the expected region and account?
- Is it in a usable state?
- Does its configuration match what you deployed?
- Can the caller access it with the intended permissions?
- Does the resource interact correctly with dependent services?

For the exam, pay attention to the difference between:

- **Existence checks**: the resource is present
- **Configuration checks**: the resource has the expected settings
- **Operational checks**: the resource is ready to serve traffic or process requests
- **Authorization checks**: the caller can use the resource as intended

**AWS CLI validation patterns**

The AWS CLI is useful for fast, repeatable checks in scripts and pipelines. You typically use `describe-*`, `get-*`, `list-*`, and `head-*` style commands, but each service behaves differently.

Common patterns:

- `describe-*` often returns detailed configuration and status
- `get-*` often returns a single resource or a focused view
- `list-*` confirms presence among collections
- `wait` blocks until a resource reaches a target state
- `head-*` can confirm existence in some services, but not always authorization or full readiness

Example: verify a Lambda function exists and inspect its configuration.

```bash
aws lambda get-function --function-name my-function
aws lambda get-function-configuration --function-name my-function
```

Use `get-function` when you want configuration plus code location metadata, and `get-function-configuration` when you want the direct configuration view. For deployment validation, the configuration call is often the clearer check.

Example: verify an S3 bucket exists and inspect objects.

```bash
aws s3api head-bucket --bucket my-bucket
aws s3api list-objects-v2 --bucket my-bucket --max-items 10
aws s3api get-bucket-location --bucket my-bucket
```

`head-bucket` can return `403 Forbidden` if the bucket exists but your caller lacks permission, so it does not prove that the bucket name is wrong or that you have access. If you need region confirmation, use `get-bucket-location` or equivalent.

**Waiters and asynchronous readiness**

Many AWS operations are eventually consistent or asynchronous. A resource may be created before it is fully usable. Waiters help you pause until AWS reports a target state.

Examples include:

- Lambda function state transitions
- EC2 instance lifecycle changes
- CodeDeploy deployment completion
- RDS instance availability
- CloudFormation stack completion

Example waiter usage for Lambda:

```bash
aws lambda wait function-active --function-name my-function
aws lambda get-function-configuration --function-name my-function
```

For code or configuration updates, be careful: a generic active-state waiter may not fully guarantee that the latest update is propagated in every workflow. In some cases, a more specific waiter or a follow-up configuration check is more appropriate.

A practical pattern is:

1. Wait for the service to report a stable state.
2. Query the resource configuration.
3. Validate a functional behavior, such as invoking the function or reading a test object.

**AWS SDK validation patterns**

The SDK is the right choice when validation belongs inside application code, deployment automation, or tests. The same principles apply: check existence, state, and behavior.

Typical SDK use cases:

- Integration tests after deployment
- Health checks in CI/CD pipelines
- Application startup validation
- Custom deployment verification scripts

Example in Python with `boto3` for Lambda:

```python
import boto3

client = boto3.client("lambda")

response = client.get_function_configuration(FunctionName="my-function")
print(response["State"])
print(response["LastUpdateStatus"])
print(response["Runtime"])
```

This is useful because you can inspect both the current state and the last update status. For deployment validation, `State` and `LastUpdateStatus` are often more informative than a simple existence check.

Example in Python for DynamoDB table validation:

```python
import boto3
from botocore.exceptions import ClientError

dynamodb = boto3.client("dynamodb")

try:
    response = dynamodb.describe_table(TableName="Orders")
    table = response["Table"]
    print(table["TableStatus"])
    print(table["ItemCount"])
except ClientError as e:
    if e.response["Error"]["Code"] == "ResourceNotFoundException":
        print("Table does not exist")
    else:
        raise
```

Using `ClientError` with error-code inspection is the most broadly reliable pattern across AWS SDK calls. Service-specific exception classes are not always the best general-purpose approach.

**Practical validation by service**

Different services need different checks. The exam expects you to choose the right API for the job.

**Lambda**

For Lambda, validate:

- Function existence
- Runtime, handler, memory, timeout, and environment variables
- Code update status
- Invocation success

Useful commands:

```bash
aws lambda get-function-configuration --function-name my-function
aws lambda invoke --function-name my-function --payload '{}' response.json
cat response.json
```

A deployment is not fully validated until you confirm the function can be invoked successfully, especially if the change affects permissions, environment variables, or downstream integrations.

**S3**

For S3, validate:

- Bucket existence
- Object placement
- Object metadata or content
- Region, if region matters to the workflow

Useful commands:

```bash
aws s3api head-bucket --bucket my-bucket
aws s3api get-bucket-location --bucket my-bucket
aws s3api list-objects-v2 --bucket my-bucket --prefix uploads/ --max-items 20
aws s3api head-object --bucket my-bucket --key uploads/app.zip
```

`head-object` is often better than `list-objects-v2` when you need to confirm a specific artifact exists. `head-bucket` alone does not tell you the bucket region.

**DynamoDB**

For DynamoDB, validate:

- Table existence
- Table status
- Billing mode and key schema
- Item presence, if relevant

Useful commands:

```bash
aws dynamodb describe-table --table-name Orders
aws dynamodb get-item \
  --table-name Orders \
  --key '{"OrderId":{"S":"123"}}'
```

`describe-table` is the primary configuration and status check. If you need to confirm data flow, `get-item` or a targeted query is more meaningful than only checking the table definition.

**API Gateway**

For API Gateway, validate:

- API exists
- Stage exists
- Deployment is associated with the stage
- Route or method configuration is correct
- Endpoint responds as expected

Useful commands vary by API type, but the general pattern is:

```bash
aws apigateway get-rest-api --rest-api-id abc123
aws apigateway get-stages --rest-api-id abc123
```

For HTTP APIs and REST APIs, make sure you use the correct service namespace and resource model. A deployed API is not validated until you test the endpoint itself.

**EventBridge**

For EventBridge, validate:

- Event bus exists
- Rule exists on the correct bus
- Targets are attached
- Events match the rule pattern

Useful commands:

```bash
aws events describe-event-bus --name my-bus
aws events describe-rule --name my-rule --event-bus-name my-bus
aws events list-targets-by-rule --rule my-rule --event-bus-name my-bus
```

If the rule is not on the default bus, include the correct `--event-bus-name`. That detail matters because the same rule name can exist in different buses. A test event is useful, but it only proves end-to-end delivery if the event pattern, bus, and target permissions are all correct.

**CodeDeploy**

For CodeDeploy, validate:

- Deployment exists
- Deployment status is successful or in progress
- Deployment group is configured correctly
- Target instances or targets are registered

Useful commands:

```bash
aws deploy get-deployment --deployment-id d-ABC123
aws deploy get-deployment-group \
  --application-name MyApp \
  --deployment-group-name MyGroup
aws deploy list-deployment-instances --deployment-id d-ABC123
```

These calls help you inspect deployment progress and target membership, but they do not by themselves prove application health. Final health usually depends on lifecycle events, load balancer health checks, and the underlying compute service.

**CloudFormation as a validation layer**

If your deployment is managed by CloudFormation, stack status is often the first thing to check.

Useful commands:

```bash
aws cloudformation describe-stacks --stack-name my-stack
aws cloudformation describe-stack-events --stack-name my-stack
```

A stack reaching `CREATE_COMPLETE` or `UPDATE_COMPLETE` is a strong signal, but you should still validate critical downstream resources if the application depends on them. CloudFormation success does not always mean the application is functionally correct.

**IAM and authorization validation**

Sometimes the deployment is correct, but the caller lacks permission. In that case, validation must include IAM.

Useful commands:

```bash
aws sts get-caller-identity
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/MyRole \
  --action-names s3:GetObject lambda:InvokeFunction \
  --resource-arns arn:aws:s3:::my-bucket/my-key arn:aws:lambda:us-east-1:123456789012:function:my-function
```

Use policy simulation to reason about permissions, but do not treat it as a perfect runtime proof for every resource-based policy scenario. It is a simulation, not a live end-to-end request.

A useful workflow is:

1. Confirm the caller identity with `sts get-caller-identity`.
2. Simulate the intended permissions.
3. Perform the real API call or invocation.
4. Inspect the actual response and error codes.

**Pagination and large result sets**

When validating resources at scale, you may need to page through results. The AWS CLI has pagination controls that are easy to confuse.

Important distinctions:

- `--max-items` limits how many items the CLI returns to you
- `--starting-token` resumes a paginated CLI result
- `--page-size` controls the size of each API call made by the CLI paginator

Example:

```bash
aws ec2 describe-instances --page-size 20 --max-items 50
```

This does not mean the service itself only supports 20-item pages. It means the CLI paginator makes smaller underlying requests and then returns up to 50 items to you.

**A practical validation workflow**

A reliable deployment validation sequence often looks like this:

1. Confirm the deployment completed successfully in the orchestrator.
2. Wait for the target resource to reach a stable state.
3. Query the resource configuration.
4. Confirm the resource is in the expected region and account.
5. Test a real operation against the resource.
6. Check logs or metrics if the operation fails.

Example shell script for Lambda validation:

```bash
#!/usr/bin/env bash
set -euo pipefail

FUNCTION_NAME="my-function"

aws lambda wait function-active --function-name "$FUNCTION_NAME"
aws lambda get-function-configuration --function-name "$FUNCTION_NAME"

aws lambda invoke \
  --function-name "$FUNCTION_NAME" \
  --payload '{}' \
  response.json >/dev/null

cat response.json
```

This pattern is useful, but remember that a waiter plus configuration check is not always enough for code-update workflows. If the deployment changes code, you may also want to verify the function’s behavior with a real invocation and inspect the response or logs.

**Common pitfalls**

- **Assuming existence means readiness**
  - A resource can exist before it is usable.
- **Using the wrong region**
  - Many AWS resources are regional, and a successful call in one region does not validate another.
- **Confusing permission errors with missing resources**
  - `403` can mean the resource exists but access is denied.
- **Relying only on a waiter**
  - Waiters confirm a state transition, not necessarily business correctness.
- **Skipping functional tests**
  - A configuration check does not prove the resource works end to end.
- **Ignoring service-specific semantics**
  - The right validation command depends on the service and the resource type.

**Exam focus**

For DVA-C02, expect questions that ask you to choose the best way to verify a deployment outcome. The correct answer usually depends on whether you need:

- A configuration read
- A status check
- A readiness wait
- A functional invocation
- A permission simulation
- A paginated inventory scan

When in doubt, prefer the most direct service-specific API that answers the question you are actually asking. For example:

- Use `describe-table` for DynamoDB status
- Use `get-function-configuration` for Lambda configuration
- Use `head-object` for a specific S3 object
- Use `describe-stack-events` for CloudFormation troubleshooting
- Use a real invocation or request to confirm end-to-end behavior

**Quick reference**

| Need | Better choice | Why |
|---|---|---|
| Confirm a resource exists | `describe-*`, `get-*`, `head-*` | Direct resource lookup |
| Confirm readiness | `wait` + status query | Handles asynchronous state |
| Confirm behavior | Real invocation/request | Proves runtime functionality |
| Confirm permissions | `sts get-caller-identity`, IAM simulation | Checks identity and policy logic |
| Scan many resources | CLI pagination | Handles large inventories |

**Mental model**

Think of validation in layers:

- **Layer 1: Inventory**
  - The resource exists.
- **Layer 2: State**
  - The resource is in the expected status.
- **Layer 3: Configuration**
  - The settings match the deployment.
- **Layer 4: Behavior**
  - The resource performs the intended action.
- **Layer 5: Access**
  - The caller can use it with the intended permissions.

If you validate only one layer, you may miss the real problem. The strongest deployment checks combine multiple layers with the smallest number of API calls needed to prove correctness.
