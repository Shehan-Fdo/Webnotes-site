## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Lambda interaction with the AWS SDK**

The AWS SDK for JavaScript/TypeScript lets you manage and invoke <abbr title="AWS Lambda">Lambda</abbr> functions from Node.js applications, scripts, and serverless backends. In practice, you use it to trigger functions synchronously for request/response workflows, asynchronously for event-driven processing, and to inspect or manage function metadata when needed.

For exam purposes, focus on the difference between:
- **Invocation mode**: synchronous vs asynchronous
- **Payload shape**: JSON serialization and deserialization
- **Client version**: AWS SDK for JavaScript v3 is modular and preferred in modern code
- **Error handling**: service exceptions, function errors, and transport failures

**AWS SDK v3 basics**

The AWS SDK for JavaScript v3 uses service clients and command objects. For Lambda, you typically create a `LambdaClient` and send commands such as `InvokeCommand`. This pattern is common in both JavaScript and TypeScript.

```ts
import { LambdaClient, InvokeCommand } from "@aws-sdk/client-lambda";

const client = new LambdaClient({ region: "us-east-1" });

const command = new InvokeCommand({
  FunctionName: "my-function",
  Payload: new TextEncoder().encode(JSON.stringify({ message: "hello" })),
});

const response = await client.send(command);

const payloadText = response.Payload
  ? new TextDecoder().decode(response.Payload)
  : "";

console.log("StatusCode:", response.StatusCode);
console.log("Payload:", payloadText);
```

Key points:
- **Modular imports** reduce bundle size compared with monolithic SDK usage.
- **`Payload`** is a `Uint8Array` in v3, so you encode input and decode output.
- **`FunctionName`** can be a name, full ARN, or partial ARN depending on the use case and permissions.

**Synchronous invocation**

Synchronous invocation is the default pattern when you need an immediate response from Lambda. The caller waits for the function to finish and receives the result in the response payload.

Use synchronous invocation when:
- You need a direct answer from the function
- The caller must react to success or failure immediately
- You are building APIs, validation steps, or orchestration logic

Typical response fields include:
- **`StatusCode`**: HTTP-like status for the invoke request
- **`Payload`**: function response body
- **`FunctionError`**: indicates the function itself failed, even if the SDK call succeeded
- **`ExecutedVersion`**: the version that actually ran

```ts
import { LambdaClient, InvokeCommand } from "@aws-sdk/client-lambda";

const client = new LambdaClient({ region: "us-east-1" });

const response = await client.send(
  new InvokeCommand({
    FunctionName: "order-processor",
    InvocationType: "RequestResponse",
    Payload: new TextEncoder().encode(JSON.stringify({
      orderId: "12345",
      action: "validate",
    })),
  })
);

if (response.FunctionError) {
  const errorBody = response.Payload
    ? new TextDecoder().decode(response.Payload)
    : "";
  throw new Error(`Lambda failed: ${errorBody}`);
}
```

Important behavior:
- A successful SDK call does **not** guarantee the Lambda function succeeded.
- If the function throws an exception, Lambda may still return `StatusCode: 200` with `FunctionError` set.
- You should inspect both the response metadata and the payload.

**Asynchronous invocation**

Asynchronous invocation sends the event and returns immediately. Lambda queues the event and processes it later. This is useful for fire-and-forget workflows, background jobs, and decoupled event processing.

Use asynchronous invocation when:
- The caller should not wait for completion
- You want to absorb bursts with Lambda’s internal queueing
- You are triggering downstream processing from a user action or event

```ts
import { LambdaClient, InvokeCommand } from "@aws-sdk/client-lambda";

const client = new LambdaClient({ region: "us-east-1" });

await client.send(
  new InvokeCommand({
    FunctionName: "thumbnail-generator",
    InvocationType: "Event",
    Payload: new TextEncoder().encode(JSON.stringify({
      bucket: "uploads-bucket",
      key: "images/photo.png",
    })),
  })
);
```

Notes:
- **`InvocationType: "Event"`** means asynchronous execution.
- The response usually confirms acceptance, not completion.
- Retries and failure handling happen on the Lambda side, so you should design idempotent handlers.

> Asynchronous invocation is not a substitute for durable workflow orchestration. If you need step-by-step state, use services such as Step Functions rather than relying only on Lambda retries.

**Request payloads and response decoding**

Lambda expects the request payload to be bytes. In JavaScript and TypeScript, you usually serialize an object with `JSON.stringify()` and encode it with `TextEncoder`. On the way back, decode the `Uint8Array` with `TextDecoder` and parse JSON if the function returns JSON.

```ts
const input = { userId: "u-100", includeProfile: true };
const payload = new TextEncoder().encode(JSON.stringify(input));

const response = await client.send(
  new InvokeCommand({
    FunctionName: "get-user-profile",
    Payload: payload,
  })
);

const raw = response.Payload ? new TextDecoder().decode(response.Payload) : "{}";
const data = JSON.parse(raw);

console.log(data);
```

Practical guidance:
- Keep payloads small; Lambda invoke payloads have size limits.
- Validate input before invoking to avoid sending malformed events.
- Treat the response as untrusted data and parse defensively.

**TypeScript typing patterns**

TypeScript improves safety around Lambda invocation by making request and response shapes explicit. The SDK still returns runtime data, so you often define your own interfaces for the function contract.

```ts
interface GetUserProfileRequest {
  userId: string;
  includeProfile?: boolean;
}

interface GetUserProfileResponse {
  userId: string;
  name: string;
  profile?: {
    title: string;
    bio: string;
  };
}

const request: GetUserProfileRequest = {
  userId: "u-100",
  includeProfile: true,
};

const response = await client.send(
  new InvokeCommand({
    FunctionName: "get-user-profile",
    Payload: new TextEncoder().encode(JSON.stringify(request)),
  })
);

const body = response.Payload
  ? JSON.parse(new TextDecoder().decode(response.Payload)) as GetUserProfileResponse
  : null;
```

Why this matters:
- **Interfaces** document the contract between caller and function.
- Type assertions help, but they do not validate runtime data.
- For stronger guarantees, pair TypeScript with schema validation such as Zod or JSON Schema.

**Error handling and retries**

Lambda invocation can fail in several ways:
- **Client-side errors**: invalid parameters, missing region, credential issues
- **Service-side errors**: throttling, access denied, function not found
- **Function errors**: the code inside Lambda throws or returns an error response

A robust caller distinguishes these cases.

```ts
import {
  LambdaClient,
  InvokeCommand,
  ResourceNotFoundException,
  AccessDeniedException,
  TooManyRequestsException,
} from "@aws-sdk/client-lambda";

try {
  const response = await client.send(
    new InvokeCommand({
      FunctionName: "billing-job",
      Payload: new TextEncoder().encode(JSON.stringify({ invoiceId: "inv-1" })),
    })
  );

  if (response.FunctionError) {
    const details = response.Payload
      ? new TextDecoder().decode(response.Payload)
      : "Unknown function error";
    console.error("Function failed:", details);
  }
} catch (error) {
  if (error instanceof TooManyRequestsException) {
    console.error("Throttled by Lambda");
  } else if (error instanceof ResourceNotFoundException) {
    console.error("Function not found");
  } else if (error instanceof AccessDeniedException) {
    console.error("Missing permission to invoke function");
  } else {
    console.error("Unexpected invoke failure:", error);
  }
}
```

Best practices:
- Retry transient failures with exponential backoff and jitter.
- Avoid retrying non-transient validation errors.
- Make handlers idempotent so duplicate invocations do not corrupt state.

**Credentials and permissions**

The SDK uses the standard AWS credential provider chain. In local development, credentials may come from environment variables, shared config files, or an assumed role. In AWS compute environments, the SDK usually picks up the attached IAM role automatically.

Common permission needed:
- `lambda:InvokeFunction`

Example IAM policy for invoking a specific function:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "lambda:InvokeFunction",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:my-function"
    }
  ]
}
```

Operational notes:
- Use the least privilege resource ARN possible.
- If you invoke aliases or versions, include those ARNs when required.
- Cross-account invocation requires both IAM permission and a resource-based policy on the Lambda function.

**Client configuration**

You usually configure region explicitly and let credentials resolve automatically. You can also tune retries, timeouts, and custom endpoints for local testing.

```ts
import { LambdaClient } from "@aws-sdk/client-lambda";

const client = new LambdaClient({
  region: "us-east-1",
  maxAttempts: 3,
});
```

Useful configuration ideas:
- Set the correct **region** to avoid `ResourceNotFoundException` caused by looking in the wrong region.
- Increase retries for transient network issues.
- Use custom endpoints only for local emulators or specialized environments.

**Local development and testing**

For local testing, you can mock the SDK or point it at a local endpoint if you use an emulator. In unit tests, mocking the client is usually simpler and more reliable than invoking real AWS resources.

Example with a mocked client pattern:

```ts
import { LambdaClient, InvokeCommand } from "@aws-sdk/client-lambda";

const client = new LambdaClient({ region: "us-east-1" });

// In tests, replace client.send with a stub or use a mocking library.
```

Practical testing strategy:
- Unit test serialization, deserialization, and error handling.
- Integration test against a real Lambda function in a non-production account.
- Avoid hard-coding credentials in test code.

**CLI parity for troubleshooting**

When debugging SDK behavior, compare it with the AWS CLI. The CLI helps confirm whether the issue is in your code, IAM policy, or Lambda configuration.

```bash
aws lambda invoke \
  --function-name my-function \
  --payload '{"message":"hello"}' \
  response.json

cat response.json
```

If the CLI works but the SDK fails, check:
- Region mismatch
- Credential source
- Payload encoding
- IAM permissions in the runtime environment

**Common exam themes**

The exam often expects you to choose the right invocation pattern and recognize how the SDK behaves.

- **Use `InvokeCommand`** to call Lambda from Node.js or TypeScript.
- **Encode and decode payloads** because the SDK uses binary buffers for request and response bodies.
- **Check `FunctionError`** to detect function-level failures.
- **Use asynchronous invocation** for decoupled event processing.
- **Grant `lambda:InvokeFunction`** for permission to call the function.
- **Prefer SDK v3** for modern modular applications.

**Quick comparison**

| Topic | Synchronous (`RequestResponse`) | Asynchronous (`Event`) |
|---|---|---|
| Caller waits | Yes | No |
| Immediate result | Yes | No |
| Typical use | APIs, orchestration | Background jobs, event fan-out |
| Error visibility | Direct in response | Handled by Lambda retry model |
| Payload handling | Request and response | Request only |

**Practical workflow example**

A common pattern is a web API that invokes Lambda to perform a business action and returns the result to the client.

1. Your application receives an HTTP request.
2. You validate the input and build a JSON payload.
3. You invoke Lambda synchronously with `InvokeCommand`.
4. You inspect `StatusCode` and `FunctionError`.
5. You decode the response and return it to the caller.

```ts
import { LambdaClient, InvokeCommand } from "@aws-sdk/client-lambda";

const client = new LambdaClient({ region: "us-east-1" });

export async function run() {
  const payload = new TextEncoder().encode(JSON.stringify({
    action: "calculateQuote",
    customerId: "c-42",
  }));

  const result = await client.send(
    new InvokeCommand({
      FunctionName: "quote-service",
      InvocationType: "RequestResponse",
      Payload: payload,
    })
  );

  if (result.FunctionError) {
    throw new Error(
      result.Payload ? new TextDecoder().decode(result.Payload) : "Lambda error"
    );
  }

  return result.Payload ? JSON.parse(new TextDecoder().decode(result.Payload)) : null;
}
```

**Key takeaways**

- Use the AWS SDK v3 `LambdaClient` and `InvokeCommand` for modern JavaScript and TypeScript code.
- Serialize request objects with `JSON.stringify()` and `TextEncoder`.
- Decode `Payload` with `TextDecoder` and parse JSON carefully.
- Distinguish between SDK-level failures and Lambda function errors.
- Choose synchronous invocation for immediate responses and asynchronous invocation for fire-and-forget processing.
- Ensure IAM permissions and region settings match the target function.
