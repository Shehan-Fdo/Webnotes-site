## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**AWS CLI fundamentals**

The AWS Command Line Interface (<abbr title="Command Line Interface">CLI</abbr>) lets you create, inspect, and automate AWS resources from a terminal. For Developer Associate tasks, you should know how to authenticate, choose a Region, control output, and query resource metadata quickly.

The CLI is especially useful when you need repeatable workflows, fast inspection, or scripting across services such as Amazon S3, Amazon EC2, AWS Lambda, and Amazon DynamoDB. It also helps you verify what your application or deployment pipeline actually created.

- **Profiles** isolate credentials and settings for different accounts or roles.
- **Regions** determine where most resources are created and queried.
- **Output formats** control how results are displayed for humans or scripts.
- **Queries** reduce large JSON responses to only the fields you need.

**Configuration and identity**

Before you create anything, confirm which identity and Region the CLI will use. The CLI reads credentials and settings from environment variables, shared config files, and named profiles. In practice, you often use `aws configure` for a default profile and `--profile` for alternate accounts.

```bash
aws configure
aws configure list
aws sts get-caller-identity
aws configure set region us-east-1
aws configure set output json
```

Typical configuration files are stored in:

- `~/.aws/credentials`
- `~/.aws/config`

A minimal profile setup looks like this:

```ini
# ~/.aws/credentials
[default]
aws_access_key_id = AKIAEXAMPLE
aws_secret_access_key = exampleSecretKey

[dev]
aws_access_key_id = AKIAEXAMPLEDEV
aws_secret_access_key = exampleSecretKeyDev
```

```ini
# ~/.aws/config
[default]
region = us-east-1
output = json

[profile dev]
region = us-west-2
output = table
```

Use `aws sts get-caller-identity` to verify the active principal before running destructive commands. This is one of the fastest ways to catch a wrong profile or assumed role.

> Be careful with environment variables such as `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`, and `AWS_DEFAULT_REGION`. They override shared config in many shells and can cause confusing results if you forget they are set.

**Creating resources from the CLI**

Most AWS services expose `create-*` commands. The exact parameters vary by service, but the workflow is consistent: define required inputs, run the command, and capture the returned identifiers for later inspection.

For example, you can create an S3 bucket, a DynamoDB table, or a Lambda function from the terminal.

```bash
aws s3api create-bucket \
  --bucket my-dev-notes-bucket-123456 \
  --region us-east-1
```

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=OrderId,AttributeType=S \
  --key-schema AttributeName=OrderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

```bash
aws lambda create-function \
  --function-name hello-cli \
  --runtime python3.12 \
  --role arn:aws:iam::123456789012:role/lambda-exec-role \
  --handler app.handler \
  --zip-file fileb://function.zip
```

When creating resources, pay attention to these common patterns:

- **Idempotency** is not guaranteed for every create call, so repeated execution can fail or create duplicates.
- **Required parameters** differ by service and often include names, Regions, IAM roles, or networking settings.
- **Eventually consistent** services may not show a new resource immediately in a follow-up list or describe call.
- **Waiters** can help you pause until a resource reaches a usable state.

For example, after creating an EC2 instance, you may wait until it is running:

```bash
aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --count 1 \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=cli-demo}]'
```

```bash
aws ec2 wait instance-running --instance-ids i-0123456789abcdef0
```

**Inspecting resources with list and describe**

Inspection usually starts with `list-*` or `describe-*` commands. `list-*` returns collections, while `describe-*` returns detailed metadata for one or more resources. The response is usually JSON, which is ideal for automation and filtering.

```bash
aws s3api list-buckets
aws ec2 describe-instances
aws lambda list-functions
aws dynamodb describe-table --table-name Orders
```

Use `--query` with <abbr title="JavaScript Object Notation">JSON</abbr> output to extract only the fields you need. This is one of the most important CLI skills for exam scenarios and real-world scripting.

```bash
aws ec2 describe-instances \
  --query 'Reservations[].Instances[].{Id:InstanceId,State:State.Name,Type:InstanceType}' \
  --output table
```

```bash
aws dynamodb describe-table \
  --table-name Orders \
  --query 'Table.{Name:TableName,Status:TableStatus,Arn:TableArn}' \
  --output json
```

Useful inspection commands include:

- `aws s3api list-buckets` to confirm bucket names and creation time.
- `aws ec2 describe-instances` to inspect instance state, tags, and networking.
- `aws iam get-role` to inspect trust policy and role metadata.
- `aws lambda get-function` to inspect configuration and code location.
- `aws dynamodb describe-table` to inspect key schema, billing mode, and status.

**Filtering and formatting output**

The CLI supports multiple output formats. Choose the one that matches your use case:

| Output | Best for | Notes |
|---|---|---|
| `json` | Automation and scripting | Default in many setups; best for `--query` |
| `table` | Human-readable inspection | Good for quick terminal review |
| `text` | Simple scripting | Compact, but less structured |
| `yaml` / `yaml-stream` | Readable structured output | Useful for some inspection workflows |

Examples:

```bash
aws s3api list-buckets --output table
aws ec2 describe-instances --output json
aws lambda list-functions --output text
```

The `--query` option uses JMESPath, a query language for JSON. It lets you select, rename, sort, and filter fields without external tools.

```bash
aws ec2 describe-instances \
  --query 'Reservations[].Instances[?State.Name==`running`].[InstanceId,Tags[?Key==`Name`].Value|[0]]' \
  --output table
```

You can also combine `--query` with `--output text` for shell-friendly values:

```bash
aws s3api list-buckets \
  --query 'Buckets[].Name' \
  --output text
```

**Practical inspection patterns**

A common workflow is to create a resource, capture its identifier, and then inspect it with a second command. This is especially useful when a service returns an ARN (<abbr title="Amazon Resource Name">ARN</abbr>) or ID that you need for later steps.

```bash
TABLE_ARN=$(aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=OrderId,AttributeType=S \
  --key-schema AttributeName=OrderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --query 'TableDescription.TableArn' \
  --output text)

aws dynamodb describe-table \
  --table-name Orders \
  --query 'Table.{Name:TableName,Status:TableStatus,Arn:TableArn}' \
  --output table
```

For S3, remember that bucket names are global, while most other resources are Region-scoped. That means a bucket name must be unique across all AWS accounts, not just within your account.

```bash
aws s3api create-bucket \
  --bucket my-unique-bucket-name-123456 \
  --create-bucket-configuration LocationConstraint=us-west-2 \
  --region us-west-2
```

> `us-east-1` is a special case for S3 bucket creation: the `--create-bucket-configuration` parameter is not used the same way as in other Regions.

**Working with profiles and roles**

Named profiles help you switch between accounts, environments, or assumed roles. This is common in development, testing, and CI/CD pipelines.

```bash
aws sts get-caller-identity --profile dev
aws ec2 describe-instances --profile dev --region us-west-2
```

If you assume a role, the CLI can use temporary credentials. These credentials expire, so scripts should handle refresh or re-authentication.

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/ReadOnlyRole \
  --role-session-name cli-session
```

A typical role-based workflow is:

1. Assume the role or authenticate through your SSO or federation flow.
2. Export or store the temporary credentials.
3. Run create or describe commands with the intended profile.
4. Verify the identity again with `aws sts get-caller-identity`.

**Safe scripting and troubleshooting**

When automating resource creation, always validate inputs and inspect outputs. A small typo in a name, Region, or role ARN can create resources in the wrong place or fail with permissions errors.

Common troubleshooting checks:

- Confirm the active identity with `aws sts get-caller-identity`.
- Confirm the Region with `aws configure list` or `aws configure get region`.
- Confirm permissions with the relevant IAM policy and service action.
- Confirm the resource exists in the same Region where you created it.
- Confirm the command syntax with `aws <service> help` or `aws <service> <operation> help`.

Useful shell patterns include:

```bash
set -euo pipefail

aws sts get-caller-identity
aws dynamodb describe-table --table-name Orders --output json
```

You can also use `--dry-run` where supported, especially with EC2, to validate permissions without making changes.

```bash
aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --count 1 \
  --dry-run
```

**Exam-focused takeaways**

For the exam, focus on matching the command type to the task:

- Use **create** commands to provision resources.
- Use **list** commands to enumerate collections.
- Use **describe** or **get** commands to inspect one resource in detail.
- Use **query** and `--output` to make responses script-friendly.
- Use **profiles** and `--region` to avoid acting on the wrong account or Region.
- Use **waiters** when a resource must reach a ready state before the next step.

A strong CLI workflow is: authenticate, verify identity, create the resource, inspect the result, and then automate the next step with the returned ID or ARN.
