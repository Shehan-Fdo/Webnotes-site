## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why environment variables matter**

AWS SDKs use a standard credential provider chain to discover authentication data. Environment variables are one of the fastest ways to supply credentials and region settings without hardcoding secrets in source code. This is especially useful for local development, CI/CD pipelines, containers, and temporary troubleshooting sessions.

Environment variables are process-scoped, so they affect only the shell session or runtime that inherits them. That makes them convenient for short-lived access, but you should still treat them as sensitive data.

- **Access key ID** and secret access key are commonly provided through environment variables.
- **Session tokens** are required when you use temporary credentials from AWS STS (Security Token Service).
- **Region** can also be set through environment variables to avoid ambiguous SDK behavior.
- **Profiles** are often better for long-term local development, but environment variables are useful when you need explicit, ephemeral control.

**Core authentication variables**

The most important variables for AWS SDK authentication are part of the standard AWS environment variable set. SDKs in multiple languages recognize these names consistently.

| Variable | Purpose | Example |
|---|---|---|
| `AWS_ACCESS_KEY_ID` | Access key for an IAM user or temporary credentials | `AKIA...` |
| `AWS_SECRET_ACCESS_KEY` | Secret key paired with the access key | `wJalrXUtnFEMI/K7MDENG/bPxRfiCY...` |
| `AWS_SESSION_TOKEN` | Temporary session token for STS credentials | `IQoJb3JpZ2luX2VjE...` |
| `AWS_REGION` | Preferred AWS Region for SDK calls | `us-east-1` |
| `AWS_DEFAULT_REGION` | Fallback region used by some tools and SDKs | `us-west-2` |

The SDK typically uses these variables when no explicit credentials are passed in code. If both environment variables and code-based credentials exist, the explicit code configuration usually wins.

> Do not store long-term credentials in source code or commit them to a repository. Use environment variables only for local, temporary, or controlled runtime contexts.

**How the credential chain works**

AWS SDKs generally follow a credential provider chain. The exact order can vary by SDK and version, but the idea is the same: the SDK checks multiple sources until it finds valid credentials.

Common sources include:

- Environment variables
- Shared credentials files such as `~/.aws/credentials`
- Shared config files such as `~/.aws/config`
- IAM role credentials from the runtime environment, such as an EC2 instance profile or an ECS task role
- Web identity or SSO-based providers, depending on the SDK and configuration

Environment variables are often checked early because they are simple and explicit. This makes them useful for overriding other sources during testing.

**Setting variables in common shells**

You usually export variables in your shell before running the application or CLI command. The syntax depends on the operating system and shell.

```bash
export AWS_ACCESS_KEY_ID="AKIAEXAMPLE"
export AWS_SECRET_ACCESS_KEY="exampleSecretKeyValue"
export AWS_SESSION_TOKEN="exampleSessionTokenValue"
export AWS_REGION="us-east-1"
```

For a single command, you can set them inline:

```bash
AWS_ACCESS_KEY_ID="AKIAEXAMPLE" \
AWS_SECRET_ACCESS_KEY="exampleSecretKeyValue" \
AWS_SESSION_TOKEN="exampleSessionTokenValue" \
AWS_REGION="us-east-1" \
python app.py
```

On Windows PowerShell:

```powershell
$env:AWS_ACCESS_KEY_ID="AKIAEXAMPLE"
$env:AWS_SECRET_ACCESS_KEY="exampleSecretKeyValue"
$env:AWS_SESSION_TOKEN="exampleSessionTokenValue"
$env:AWS_REGION="us-east-1"
python .\app.py
```

On Windows Command Prompt:

```cmd
set AWS_ACCESS_KEY_ID=AKIAEXAMPLE
set AWS_SECRET_ACCESS_KEY=exampleSecretKeyValue
set AWS_SESSION_TOKEN=exampleSessionTokenValue
set AWS_REGION=us-east-1
python app.py
```

**Using environment variables with the AWS CLI**

The AWS Command Line Interface (CLI) uses the same environment variables as the SDKs. This makes the CLI a useful way to validate that your environment is configured correctly before you debug application code.

```bash
aws sts get-caller-identity
aws s3 ls
aws dynamodb list-tables --region us-east-1
```

If the credentials are valid, `aws sts get-caller-identity` returns the authenticated principal. This is one of the best quick checks for confirming that the environment variables are being read correctly.

**Practical workflow for local development**

A common local workflow is to load temporary credentials into your shell, run the application, and then clear the variables when finished. This reduces the chance of accidentally reusing stale credentials.

1. Obtain credentials from a secure source, such as AWS IAM Identity Center, STS, or a controlled secrets workflow.
2. Export the variables in the shell session.
3. Verify access with `aws sts get-caller-identity`.
4. Run your application or tests.
5. Unset the variables when you are done.

```bash
unset AWS_ACCESS_KEY_ID
unset AWS_SECRET_ACCESS_KEY
unset AWS_SESSION_TOKEN
unset AWS_REGION
```

**Temporary credentials and session tokens**

When you assume a role or use federated access, AWS returns temporary credentials. These credentials usually include three parts:

- Access key ID
- Secret access key
- Session token

All three are required. If you omit `AWS_SESSION_TOKEN`, authentication may fail even if the access key and secret key are correct.

This is a common exam and troubleshooting point: long-term IAM user credentials do not use a session token, but temporary credentials do.

**Region configuration and why it matters**

Authentication and region selection are related but distinct. Credentials prove who you are; the region tells the SDK where to send requests.

If you do not set a region, the SDK may:

- Fail with a region resolution error
- Use a default from another configuration source
- Send requests to an unexpected region

Use `AWS_REGION` or `AWS_DEFAULT_REGION` to make behavior predictable.

```bash
export AWS_REGION="us-west-2"
aws ec2 describe-instances
```

For multi-region applications, you may still override the region in code or per command, but environment variables are a good default for local testing.

**Example application behavior**

In many SDKs, you do not need to manually read environment variables in your application code. The SDK does it for you.

```python
import boto3

s3 = boto3.client("s3")
response = s3.list_buckets()
print(response)
```

If the shell has valid AWS environment variables, `boto3` can authenticate automatically. This is one reason environment variables are preferred over embedding credentials in code.

**Common mistakes and troubleshooting**

A few issues appear frequently when configuring SDK authentication with environment variables:

- **Wrong variable names**: The SDK will not recognize custom names unless your code maps them manually.
- **Missing session token**: Temporary credentials require `AWS_SESSION_TOKEN`.
- **Expired credentials**: STS credentials expire, so you must refresh them.
- **Incorrect region**: Authentication may succeed, but the request can still fail if the region is wrong.
- **Shell scope confusion**: Variables set in one terminal do not automatically appear in another.
- **Whitespace or quoting issues**: Copying values with extra spaces or line breaks can break authentication.

A useful debugging sequence is:

1. Print the variables in the current shell.
2. Confirm the region.
3. Run `aws sts get-caller-identity`.
4. Check whether the application is using a different credential source.
5. Review SDK debug logs if needed.

```bash
env | grep '^AWS_'
aws configure list
aws sts get-caller-identity --debug
```

**Security best practices**

Environment variables are better than hardcoding secrets, but they are not a perfect secret store. They can still be exposed through process inspection, shell history, crash dumps, or misconfigured logging.

Use these practices:

- Prefer temporary credentials over long-term access keys.
- Rotate credentials regularly if you must use IAM user keys.
- Avoid printing secrets in logs or debug output.
- Use IAM roles in AWS compute environments when possible.
- Keep credentials out of `.env` files that are committed to source control.
- Limit permissions with the principle of least privilege.

> If you are running on AWS infrastructure, an IAM role is usually safer than manually injecting access keys through environment variables.

**`.env` files and local tooling**

Some local development tools load environment variables from a `.env` file. This can be convenient, but it is not an AWS feature by itself. The application or framework must explicitly load the file.

Example `.env` file:

```dotenv
AWS_ACCESS_KEY_ID=AKIAEXAMPLE
AWS_SECRET_ACCESS_KEY=exampleSecretKeyValue
AWS_SESSION_TOKEN=exampleSessionTokenValue
AWS_REGION=us-east-1
```

Example Node.js usage with `dotenv`:

```javascript
require("dotenv").config();
const { S3Client, ListBucketsCommand } = require("@aws-sdk/client-s3");

const client = new S3Client({});
const response = await client.send(new ListBucketsCommand({}));
console.log(response);
```

Use `.env` files carefully:

- Keep them out of version control.
- Restrict file permissions.
- Prefer secret managers or role-based credentials for shared environments.

**How this appears in exam scenarios**

For exam questions, focus on the intent of environment variables:

- They provide a quick, externalized way to supply credentials and region settings.
- They are commonly used for local development and automation.
- They are part of the SDK credential provider chain.
- They should not be used to permanently store secrets in application code.

If a question asks how to make an SDK authenticate without changing code, environment variables are often the best answer.

**Minimal verification checklist**

- Confirm the shell has `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY`.
- Add `AWS_SESSION_TOKEN` if the credentials are temporary.
- Set `AWS_REGION` or `AWS_DEFAULT_REGION`.
- Run `aws sts get-caller-identity`.
- Launch the application in the same environment session.
