## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Pagination fundamentals**

Many AWS APIs return results in pages rather than a single response. Pagination prevents oversized payloads, reduces latency spikes, and helps services scale by limiting the amount of data returned per request. In practice, you request a page, inspect the response for a continuation token, and keep requesting until no more pages remain.

Common pagination markers include:

- **NextToken** in many AWS service APIs
- **Marker** in older or service-specific APIs
- **ContinuationToken** in some object listing APIs
- SDK-managed iterators that hide tokens from your code

You should design code to process each page incrementally instead of loading all results into memory. This matters when listing large numbers of resources such as S3 objects, DynamoDB items, CloudWatch logs, or IAM entities.

**Why pagination matters in applications**

Pagination is not just a UI concern. It affects reliability, memory use, and cost:

- **Lower memory pressure** by processing one page at a time
- **Better responsiveness** by starting work before the full dataset is retrieved
- **Safer retries** because page boundaries give you smaller units of work
- **Reduced timeout risk** in Lambda, containers, and long-running jobs

When you need a complete dataset, use a loop that continues until the service stops returning a continuation token. When you only need a subset, stop early to save time and API calls.

**Manual pagination pattern**

A typical manual loop looks like this:

```python
import boto3

client = boto3.client("ec2")

token = None
while True:
    params = {"MaxResults": 100}
    if token:
        params["NextToken"] = token

    response = client.describe_instances(**params)

    for reservation in response.get("Reservations", []):
        for instance in reservation.get("Instances", []):
            print(instance["InstanceId"])

    token = response.get("NextToken")
    if not token:
        break
```

This pattern is useful when you need custom filtering, early exit logic, or page-by-page side effects. The important rule is to treat each page as a unit of work and to persist progress if the job must survive failures.

**SDK paginators**

Most AWS SDKs provide paginators that simplify iteration and reduce token-handling bugs. Use them when you want to focus on business logic rather than request plumbing.

Example with Python `boto3`:

```python
import boto3

client = boto3.client("s3")
paginator = client.get_paginator("list_objects_v2")

for page in paginator.paginate(Bucket="my-bucket", Prefix="logs/"):
    for obj in page.get("Contents", []):
        print(obj["Key"])
```

SDK paginators are especially useful when:

- You need to iterate through many pages
- You want consistent handling across services
- You want to avoid off-by-one token mistakes
- You want cleaner code in Lambda or microservices

> Be careful not to assume every API uses the same token field name or the same page size semantics. Always check the service API model or SDK documentation for the exact pagination contract.

**Pagination in AWS CLI**

The AWS CLI also paginates by default for many commands. It may automatically fetch multiple pages and display a combined result. You can control this behavior when needed.

Useful CLI patterns:

```bash
aws s3api list-objects-v2 --bucket my-bucket --prefix logs/ --max-items 50
aws ec2 describe-instances --max-results 5
aws iam list-users --max-items 10
```

Key CLI concepts:

- `--max-results` or service-specific equivalents limit the page size sent to the service
- `--max-items` limits the total number of items returned by the CLI
- `--starting-token` resumes a paginated CLI operation
- `--no-paginate` disables automatic pagination in some commands

Use `--starting-token` when you need to resume a CLI-driven workflow after interruption. Use `--no-paginate` only when you intentionally want a single service call and understand the result size.

**Retry fundamentals**

Retries handle transient failures that can succeed if attempted again. AWS applications should expect occasional network issues, throttling, and temporary service unavailability. A good retry strategy improves resilience without creating retry storms.

Typical retryable conditions include:

- **Throttling** responses such as `ThrottlingException` or `TooManyRequestsException`
- **Transient network errors** such as connection resets or timeouts
- **Service-side 5xx errors** such as `InternalServerError` or `ServiceUnavailable`
- **Request rate limits** enforced by AWS APIs

Retries should be selective. Do not blindly retry every error, especially when the request is not idempotent or when the error indicates a permanent problem.

**Idempotency and safe retries**

An operation is **idempotent** if repeating it produces the same end state. Idempotency is critical when retries may cause duplicate side effects.

Examples:

- Safe to retry:
  - `Describe*` and `List*` calls
  - `PutItem` with a deterministic primary key when you intend to overwrite
  - `UpdateItem` with conditional logic that prevents duplicate state transitions
- Risky to retry without safeguards:
  - `RunInstances` if you do not use client tokens
  - `SendMessage` if duplicate delivery is harmful
  - Payment or order creation workflows without deduplication

For operations that support it, use a **client token** or request identifier to make retries safe. Many AWS APIs accept an idempotency token so the service can recognize duplicate requests.

Example pattern:

```python
import boto3
import uuid

client = boto3.client("ec2")

response = client.run_instances(
    ImageId="ami-1234567890abcdef0",
    InstanceType="t3.micro",
    MinCount=1,
    MaxCount=1,
    ClientToken=str(uuid.uuid4())
)
```

**Retry strategies**

A robust retry strategy usually includes:

- **Exponential backoff** to increase wait time between attempts
- **Jitter** to spread retries and avoid synchronized bursts
- **Maximum attempts** to prevent infinite retry loops
- **Timeouts** so requests fail fast enough for the caller to recover

A common backoff pattern is:

1. Try the request.
2. If the error is retryable, wait briefly.
3. Increase the delay on each attempt.
4. Stop after a bounded number of retries.

Jitter matters because many clients may fail at the same time. If they all retry on the same schedule, they can overload the service again. Randomized delay reduces this thundering herd effect.

Example pseudo-logic:

```python
import random
import time

base = 0.2
max_delay = 5.0

for attempt in range(1, 6):
    try:
        do_request()
        break
    except RetryableError:
        delay = min(max_delay, base * (2 ** (attempt - 1)))
        delay = random.uniform(0, delay)
        time.sleep(delay)
```

**AWS SDK retry behavior**

AWS SDKs include built-in retry logic, and you should understand how to tune it rather than reimplementing it from scratch. Modern SDKs often support standard retry modes that handle throttling and transient failures automatically.

Common concepts include:

- **Standard mode** for general-purpose retries with backoff
- **Adaptive mode** in some SDKs for more dynamic throttling-aware behavior
- **Legacy mode** in older clients with different retry characteristics
- Configurable **max attempts** or retry count

Example with Python `boto3`/`botocore` configuration:

```python
import boto3
from botocore.config import Config

config = Config(
    retries={
        "max_attempts": 8,
        "mode": "standard"
    }
)

client = boto3.client("dynamodb", config=config)
```

Example with AWS CLI configuration:

```bash
aws configure set retry_mode standard
aws configure set max_attempts 8
```

Use SDK retries as the first line of defense. Add application-level retries only when you need custom behavior, such as compensating actions, workflow orchestration, or special handling for specific exceptions.

**Service errors and how to interpret them**

AWS service errors generally fall into a few categories:

- **Client errors (4xx)** indicate a problem with the request, permissions, or resource state
- **Server errors (5xx)** indicate a service-side issue that may succeed later
- **Throttling errors** indicate rate limiting and usually merit retry with backoff

Common examples:

- `AccessDeniedException` or `AccessDenied`:
  - Usually means IAM permissions are missing or a resource policy denies access
  - Fix the policy rather than retrying
- `ValidationException`:
  - Usually means malformed input, invalid parameter values, or schema mismatch
  - Retry is not useful unless your code can correct the input
- `ResourceNotFoundException`:
  - Usually means the resource does not exist or is not yet visible due to eventual consistency
  - Retry only if the resource was just created and propagation delay is plausible
- `ConditionalCheckFailedException`:
  - Usually means a DynamoDB condition did not match
  - This is often a business-rule failure, not a transient error
- `ThrottlingException`:
  - Usually means you exceeded a service quota or request rate
  - Retry with backoff and consider reducing concurrency
- `InternalServerError`:
  - Usually means a transient service issue
  - Retry is appropriate within limits

**Eventual consistency considerations**

Some AWS services are eventually consistent. A resource may be created successfully but not immediately visible to a subsequent read. This can look like a missing resource or a temporary failure.

Practical implications:

- After creating a resource, a follow-up `Describe` or `Get` may briefly return not found
- A retry with backoff may succeed after propagation
- You should not confuse propagation delay with a permanent failure

This is common in workflows that create then immediately use resources, such as:

- Creating an IAM role and then attaching it to a Lambda function
- Creating an S3 bucket policy and then reading it back
- Creating a DynamoDB table and then writing to it immediately

When the service supports waiters, use them instead of hand-rolled polling. Waiters encapsulate the correct polling interval and terminal states.

Example:

```python
import boto3

client = boto3.client("ec2")
waiter = client.get_waiter("instance_running")
waiter.wait(InstanceIds=["i-1234567890abcdef0"])
```

**Handling errors in code**

You should catch specific exceptions where possible and avoid broad exception handling that hides bugs. Handle retryable and non-retryable errors differently.

Example in Python:

```python
import boto3
from botocore.exceptions import ClientError

client = boto3.client("sqs")

try:
    response = client.receive_message(
        QueueUrl="https://sqs.us-east-1.amazonaws.com/123456789012/my-queue",
        MaxNumberOfMessages=10,
        WaitTimeSeconds=20
    )
except ClientError as e:
    code = e.response["Error"]["Code"]

    if code in ("ThrottlingException", "TooManyRequestsException"):
        # retry with backoff
        raise
    elif code == "AccessDenied":
        # fix IAM or resource policy
        raise
    else:
        # log and fail fast or route to dead-letter handling
        raise
```

Good error handling practices:

- Log the error code, request ID, and relevant resource identifiers
- Preserve the original exception when rethrowing
- Separate retryable from terminal failures
- Use structured logs so you can search by error code and request ID

**Dead-letter and fallback patterns**

Retries are not the only resilience tool. If a request keeps failing, you may need a fallback path.

Useful patterns include:

- **Dead-letter queues (DLQs)** for asynchronous processing failures
- **Circuit breakers** to stop repeated calls to a failing dependency
- **Fallback reads** from cached or replicated data
- **Compensating actions** in workflows that partially succeed

A DLQ is especially useful when a message or job repeatedly fails after retries. Instead of blocking the main queue, the failed item is moved aside for later inspection.

**Practical CLI troubleshooting**

When diagnosing AWS API failures, use the CLI with `--debug` to inspect request IDs, retries, and response codes.

```bash
aws s3api list-objects-v2 --bucket my-bucket --prefix logs/ --debug
```

Useful troubleshooting steps:

1. Confirm the IAM principal has the required permissions.
2. Check whether the resource exists in the expected Region.
3. Verify whether the request is hitting a quota or rate limit.
4. Inspect the error code and request ID in debug output.
5. Retry only if the error is transient or propagation-related.

**Exam-focused distinctions**

For DVA-C02, you should be able to choose the right behavior based on the error type and workload pattern:

- Use **pagination** when an API returns large result sets
- Use **SDK paginators** when you want simpler, safer iteration
- Use **exponential backoff with jitter** for throttling and transient failures
- Use **idempotency tokens** for retry-safe create operations
- Use **waiters** or polling for eventual consistency after resource creation
- Do not retry **authorization**, **validation**, or other permanent client errors
- Reduce concurrency when throttling persists instead of only increasing retries

**Quick comparison**

| Situation | Best response | Why |
|---|---|---|
| Large list response | Paginate | Prevents oversized responses and memory issues |
| Throttling error | Retry with backoff and jitter | Temporary rate limiting may clear |
| Access denied | Fix IAM or resource policy | Retry will not change permissions |
| Resource not found immediately after create | Wait and retry cautiously | May be eventual consistency |
| Duplicate create risk | Use idempotency token | Prevents duplicate side effects |

**Implementation checklist**

- Use paginators for list-heavy workflows
- Process each page incrementally
- Set sensible timeouts and retry limits
- Retry only transient and throttling errors
- Add jitter to avoid synchronized retries
- Use idempotency tokens for unsafe create operations
- Log request IDs and error codes
- Prefer waiters for post-create polling when available

**Common pitfalls**

- Fetching all pages into memory when streaming would be safer
- Retrying non-idempotent operations without deduplication
- Treating `AccessDenied` as transient
- Ignoring eventual consistency after resource creation
- Overriding SDK retries with custom loops that amplify traffic
- Using fixed retry intervals without jitter

A resilient AWS client balances throughput, correctness, and service protection. Pagination keeps data retrieval manageable, retries absorb transient failures, and careful error classification prevents wasted attempts on permanent failures.
