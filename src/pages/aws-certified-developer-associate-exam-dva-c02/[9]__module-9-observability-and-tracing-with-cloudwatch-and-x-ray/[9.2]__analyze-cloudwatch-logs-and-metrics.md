## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**CloudWatch observability model**

Amazon CloudWatch gives you a practical path from raw telemetry to actionable troubleshooting. You usually start with **metrics** for fast trend detection, use **logs** for detailed context, and then connect the two with alarms, metric filters, and queries.

A useful mental model is:

```text
Application emits logs/metrics
        ↓
CloudWatch stores metrics and log events
        ↓
Dashboards and alarms surface anomalies
        ↓
Logs Insights and metric filters help you investigate root cause
```

For the exam, focus on choosing the right tool for the job:

- **Metrics** are numeric time-series data, ideal for alarms and dashboards.
- **Logs** are event records, ideal for debugging and forensic analysis.
- **Alarms** turn metric conditions into notifications or automated actions.
- **Insights queries** help you search and aggregate logs interactively.
- **Metric filters** convert log patterns into custom metrics when you need alerting on log content.

**CloudWatch metrics fundamentals**

CloudWatch metrics are stored as time-series data points with dimensions. A dimension is a name/value pair that identifies a specific metric series, such as `ApiName`, `Stage`, `FunctionName`, or `TableName`.

Key metric concepts:

- **Namespace** groups related metrics, such as `AWS/Lambda`, `AWS/ApiGateway`, or `AWS/DynamoDB`.
- **Dimensions** narrow a metric to a specific resource or slice of traffic.
- **Statistic** determines how CloudWatch aggregates data, such as `Average`, `Sum`, `Minimum`, `Maximum`, or percentile values.
- **Period** is the aggregation interval used in graphs and alarms.
- **Resolution** determines how often data points are published.

Standard and high-resolution metrics differ in how frequently they are published:

- **Standard resolution** metrics are typically published at 1-minute granularity.
- **High-resolution** metrics can be published at 1-second granularity and are useful for near-real-time monitoring.
- High-resolution metrics cost more and are only worth using when you need faster detection or tighter automation.

Retention is based on metric age, with older data rolled up into coarser aggregates over time. In practice, this means you can still view historical trends, but not always at the original fine-grained resolution.

| Metric type | Typical use | Notes |
|---|---|---|
| Standard resolution | General monitoring | Lower cost, common default |
| High-resolution | Fast alerting and detailed telemetry | Higher cost, shorter period alarms possible |

**Metric retention and aggregation**

CloudWatch does not keep all metric data at the same granularity forever. Older data is retained in aggregated form, so you should design dashboards and alarms around the resolution you actually need.

Important implications:

- Recent data is more detailed than older data.
- Long-term trend analysis works well, but exact second-by-second history does not.
- Alarm evaluation uses the metric period you configure, so choose a period that matches the signal you care about.
- For cost and clarity, avoid high-resolution metrics unless you need them.

**Logs: structure, ingestion, and retention**

CloudWatch Logs stores log events in log groups and log streams.

- A **log group** is the container for related logs, often one per application, service, or environment.
- A **log stream** is an ordered sequence of events from a single source, such as one Lambda execution environment or one EC2 instance.
- A **log event** is a single timestamped record.

You should configure **retention** on log groups so logs expire automatically instead of growing forever. This is both a cost-control and compliance practice.

Common retention actions:

```bash
aws logs put-retention-policy \
  --log-group-name /aws/lambda/my-function \
  --retention-in-days 14
```

Useful operational commands:

```bash
aws logs describe-log-groups
aws logs describe-log-streams --log-group-name /aws/lambda/my-function
aws logs tail /aws/lambda/my-function --follow
```

The `tail` command is useful for live troubleshooting because it streams recent log events without requiring you to open the console.

**CloudWatch Logs Insights**

CloudWatch Logs Insights is the interactive query tool for analyzing log data in CloudWatch Logs. It is especially useful when you need to search, filter, aggregate, and visualize logs quickly during troubleshooting.

Use it when you want to answer questions like:

- Which requests are failing most often?
- What is the latency distribution?
- Which error messages appear most frequently?
- Which users, resources, or endpoints generate the most traffic?

A Logs Insights query typically uses:

- `fields` to select fields for display
- `filter` to narrow results
- `sort` to order results
- `limit` to cap output
- `stats` to aggregate data
- `parse` to extract values from unstructured or semi-structured logs

Example query for Lambda errors:

```sql
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 20
```

Example query for request counts by status code, assuming the log format exposes a `status` field:

```sql
fields status
| stats count(*) as requests by status
| sort requests desc
```

If your logs are not already structured, you often need `parse` to extract fields first:

```sql
fields @timestamp, @message
| parse @message '"status":*,' as status
| parse @message '"latency":*,' as latency
| stats count(*) as requests, avg(latency) as avgLatency by status
| sort requests desc
```

Common Logs Insights functions and patterns:

- `stats count(*)` for event counts
- `stats avg(field)` for averages
- `stats min(field)`, `max(field)`, `sum(field)`
- `stats pct(field, 95)` for percentile analysis
- `filter field like /regex/` for pattern matching
- `parse` with glob-style or regex-style extraction
- `bin(5m)` to group time-series data into 5-minute buckets

Example latency trend query:

```sql
fields @timestamp, latency
| filter ispresent(latency)
| stats avg(latency) as avgLatency, pct(latency, 95) as p95 by bin(5m)
| sort bin(5m) asc
```

Practical exam nuance:

- Logs Insights is excellent for ad hoc investigation and root-cause analysis.
- It is not a replacement for alarms because it is interactive and query-based.
- If you need a metric that can alarm automatically, use a metric filter or publish a custom metric.

**Metric filters: turning logs into metrics**

Metric filters scan log events for patterns and increment a CloudWatch metric when a match occurs. This is the bridge between logs and metrics.

Use metric filters when:

- You want to alarm on a log pattern such as `ERROR`, `AccessDenied`, or `Task timed out`.
- You need a count of specific events over time.
- You want a lightweight signal without running repeated log queries.

Example metric filter for Lambda errors:

```bash
aws logs put-metric-filter \
  --log-group-name /aws/lambda/my-function \
  --filter-name LambdaErrorCount \
  --filter-pattern '"ERROR"' \
  --metric-transformations \
      metricName=LambdaErrorCount,metricNamespace=Custom/App,metricValue=1
```

Example filter pattern ideas:

- Exact text match: `"ERROR"`
- JSON field match: `{ $.status = 5* }`
- Multiple terms: `"AccessDenied" "DynamoDB"`
- Exclusion patterns for noise reduction

A metric filter is often the right choice when you need a simple alarm on a log event count, such as:

- number of application errors
- number of failed authentications
- number of throttling events
- number of timeout messages

**Log subscription filters and destinations**

Subscription filters stream matching log events to another service in near real time. This is different from metric filters: subscription filters forward log data, while metric filters create metrics.

Common destinations:

- **Kinesis Data Streams** for custom real-time processing
- **Kinesis Data Firehose** for delivery to S3, OpenSearch Service, or other destinations
- **AWS Lambda** for custom transformation or alerting logic

Use subscription filters when you need:

- centralized log processing
- security analytics
- custom enrichment or routing
- archival outside CloudWatch Logs
- near-real-time pipelines

Example subscription filter to Lambda:

```bash
aws logs put-subscription-filter \
  --log-group-name /aws/lambda/my-function \
  --filter-name ForwardErrors \
  --filter-pattern '"ERROR"' \
  --destination-arn arn:aws:lambda:us-east-1:123456789012:function:ProcessLogs
```

Important operational notes:

- The destination must allow CloudWatch Logs to invoke it.
- Subscription filters are commonly used for cross-account log centralization.
- You usually choose Firehose when you want durable delivery to storage or analytics systems.

**Alarms: standard, composite, and anomaly detection**

CloudWatch alarms evaluate metrics and change state when thresholds are crossed.

Alarm states:

- **OK** means the metric is within the expected range.
- **ALARM** means the threshold condition is met.
- **INSUFFICIENT_DATA** means CloudWatch does not have enough data to evaluate the alarm.

Standard alarm configuration includes:

- metric name and namespace
- dimensions
- statistic
- period
- evaluation periods
- threshold
- comparison operator
- missing data treatment

Example alarm for Lambda errors with dimensions:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name MyFunctionErrors \
  --namespace AWS/Lambda \
  --metric-name Errors \
  --dimensions Name=FunctionName,Value=my-function \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --treat-missing-data notBreaching \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:alerts
```

Why dimensions matter:

- Without the correct dimensions, you may alarm on the wrong metric series or on no useful series at all.
- Many AWS service metrics are only meaningful when scoped to the right resource, stage, or table.

**Composite alarms** combine multiple alarms using Boolean logic. Use them to reduce noise and alert only when several conditions are true.

Example use cases:

- page only when both error rate and latency are high
- suppress alerts during maintenance by combining with a maintenance-state alarm
- reduce duplicate notifications from many resource-level alarms

**Anomaly detection** uses a learned baseline to detect unusual metric behavior. It is useful when a fixed threshold is hard to define, such as traffic that varies by time of day.

Use anomaly detection when:

- normal behavior changes by season or hour
- you want adaptive alerting
- a static threshold creates too many false positives

**Dashboards and visualization**

CloudWatch dashboards help you visualize metrics, alarms, and operational status in one place. They are useful for both day-to-day operations and incident response.

Dashboard widgets commonly include:

- metric widgets
- text widgets
- alarm widgets
- log query widgets in supported environments and configurations

Dashboards are especially useful when you want to compare multiple signals side by side, such as:

- Lambda errors and duration
- API Gateway 4XX/5XX errors and latency
- DynamoDB throttles and consumed capacity
- ECS or EC2 CPU and memory trends

Example dashboard widget snippet:

```json
{
  "type": "metric",
  "properties": {
    "metrics": [
      [ "AWS/Lambda", "Errors", "FunctionName", "my-function" ],
      [ ".", "Duration", ".", "." ]
    ],
    "period": 60,
    "stat": "Sum",
    "region": "us-east-1",
    "title": "Lambda Errors and Duration"
  }
}
```

Cross-account and cross-region considerations:

- Dashboards can be shared and viewed across accounts with the right permissions.
- You can build operational views that span multiple regions, but metric availability and permissions must be configured correctly.
- For centralized operations, use cross-account observability features where available, rather than manually copying data.

**CloudWatch Agent vs CloudWatch Logs Agent**

The **CloudWatch Agent** is the modern agent for collecting logs, metrics, and system-level telemetry from EC2 instances and on-premises servers.

The older **CloudWatch Logs Agent** is legacy and focused mainly on log collection. In exam scenarios, prefer the CloudWatch Agent unless the question explicitly references the older agent or a legacy setup.

CloudWatch Agent advantages:

- collects both metrics and logs
- supports more configuration options
- is the recommended choice for new deployments

Typical use cases:

- EC2 memory and disk metrics, which are not available by default
- custom application logs from servers
- unified telemetry collection

Example agent configuration fragment:

```json
{
  "metrics": {
    "metrics_collected": {
      "mem": {
        "measurement": [ "mem_used_percent" ]
      },
      "disk": {
        "measurement": [ "used_percent" ],
        "resources": [ "*" ]
      }
    }
  },
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/messages",
            "log_group_name": "/ec2/system",
            "log_stream_name": "{instance_id}"
          }
        ]
      }
    }
  }
}
```

**Practical troubleshooting workflows**

A good troubleshooting workflow starts with a metric signal, then drills into logs for context.

1. Check the alarm or dashboard for the symptom.
2. Identify the affected resource and time window.
3. Open Logs Insights for the relevant log group.
4. Filter for errors, timeouts, or throttles.
5. Parse structured fields if needed.
6. Correlate the log findings with the metric spike.
7. Decide whether to fix code, scale capacity, or adjust configuration.

Example: Lambda timeout investigation

- Alarm shows increased `Errors` and `Duration`.
- Logs Insights reveals repeated `Task timed out after 3.00 seconds`.
- You inspect downstream calls and find a slow dependency.
- You increase timeout, optimize the dependency, or add retries with backoff.

Example query:

```sql
fields @timestamp, @message
| filter @message like /Task timed out/
| sort @timestamp desc
| limit 50
```

Example: API Gateway 5XX investigation

- Metric alarm shows a spike in `5XXError`.
- Logs show backend integration failures or malformed responses.
- You correlate request IDs from API Gateway access logs with Lambda or application logs.
- You confirm whether the issue is in the API layer, integration layer, or downstream service.

Example: DynamoDB throttling investigation

- Metric alarm shows `ThrottledRequests`.
- Logs or application telemetry show retry storms.
- You check whether the table needs more capacity, better partition key distribution, or adaptive retry logic.

**Choosing the right CloudWatch feature**

| Need | Best fit | Why |
|---|---|---|
| Alert on a numeric threshold | Alarm on a metric | Fast, automatic, low overhead |
| Count log events matching a pattern | Metric filter | Converts logs into metrics |
| Investigate a recent incident | Logs Insights | Flexible search and aggregation |
| Stream logs to another system | Subscription filter | Near-real-time forwarding |
| Reduce alert noise from multiple alarms | Composite alarm | Boolean logic across alarms |
| Detect unusual traffic patterns | Anomaly detection | Adaptive baseline |

**Exam-relevant distinctions**

- **Logs Insights** is for interactive analysis, not automatic alerting.
- **Metric filters** are for turning log patterns into metrics.
- **Subscription filters** are for forwarding logs to another destination.
- **Alarms** evaluate metrics, not raw logs.
- **Dimensions** matter when targeting AWS service metrics.
- **High-resolution metrics** are useful for fast detection but cost more.
- **CloudWatch Agent** is the preferred modern agent for servers.
- **Retention policies** should be set on log groups to control cost and compliance.

**Common CLI patterns**

Create a log group retention policy:

```bash
aws logs put-retention-policy \
  --log-group-name /aws/lambda/my-function \
  --retention-in-days 30
```

Create a metric filter:

```bash
aws logs put-metric-filter \
  --log-group-name /aws/lambda/my-function \
  --filter-name ErrorCount \
  --filter-pattern '"ERROR"' \
  --metric-transformations metricName=ErrorCount,metricNamespace=Custom/App,metricValue=1
```

Create an alarm on a custom metric:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name AppErrorAlarm \
  --namespace Custom/App \
  --metric-name ErrorCount \
  --statistic Sum \
  --period 300 \
  --evaluation-periods 1 \
  --threshold 5 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:alerts
```

Run a Logs Insights query from the CLI:

```bash
aws logs start-query \
  --log-group-name /aws/lambda/my-function \
  --start-time 1714080000 \
  --end-time 1714083600 \
  --query-string 'fields @timestamp, @message | filter @message like /ERROR/ | sort @timestamp desc | limit 20'
```

**Quick decision guide**

- Use **metrics** when you need trend lines, thresholds, and alarms.
- Use **logs** when you need detailed context or root cause.
- Use **Logs Insights** when you need to search and summarize logs quickly.
- Use **metric filters** when log content should drive alerting.
- Use **subscription filters** when logs must flow to another system.
- Use **composite alarms** when you want fewer, smarter notifications.
