## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Structured logging in Lambda**

Structured logging means you emit logs as machine-readable records, usually JSON, instead of free-form text. In AWS Lambda, this makes logs easier to search, filter, correlate, and analyze in Amazon CloudWatch Logs. It is especially useful when you need to trace a request across services, isolate failures, or build metrics and alerts from log data.

Lambda already captures anything written to `stdout` and `stderr`, so your main task is to format log entries consistently and include useful context such as request IDs, function names, severity, and business identifiers.

- **JSON logs** are easier to query than plain text.
- **Correlation IDs** help you trace one request across multiple services.
- **Severity levels** let you separate informational events from errors.
- **Consistent fields** make CloudWatch Logs Insights queries reliable.

**Why structured logging matters**

Free-form logs are hard to search at scale because each developer may format messages differently. Structured logs solve this by using predictable keys. That consistency helps you answer questions like:

- Which requests failed in the last 15 minutes?
- Which customer IDs are producing repeated errors?
- How long did a specific workflow take?
- Which Lambda invocation produced a downstream timeout?

Structured logging also supports operational automation. For example, you can create CloudWatch Logs Insights queries, metric filters, and alarms based on fields such as `level`, `service`, `requestId`, or `errorType`.

**Lambda logging basics**

Lambda sends runtime output to CloudWatch Logs automatically. In most runtimes, anything you print is captured. The practical difference is whether you print plain text or structured records.

A typical structured log entry includes:

- **timestamp** for event ordering
- **level** such as `INFO`, `WARN`, or `ERROR`
- **message** describing the event
- **requestId** from the Lambda invocation context
- **service** or function name
- **correlationId** or trace identifier
- **additional fields** relevant to the business event

Example JSON log line:

```json
{
  "timestamp": "2026-04-26T12:34:56.789Z",
  "level": "INFO",
  "service": "orders-api",
  "function": "CreateOrderHandler",
  "requestId": "c1f2d3e4-5678-90ab-cdef-1234567890ab",
  "correlationId": "a9b8c7d6e5f4",
  "message": "Order created successfully",
  "orderId": "ord-1042",
  "customerId": "cust-7781",
  "durationMs": 42
}
```

**What to include in each log event**

A good structured log balances usefulness and cost. Include fields that help you diagnose issues without exposing sensitive data.

- **requestId** from the Lambda context for invocation-level tracing
- **correlationId** from an upstream API Gateway request, event payload, or custom header
- **level** to classify severity
- **message** as a human-readable summary
- **service** and **function** to identify the source
- **operation** or **action** to describe what the code is doing
- **durationMs** for performance analysis
- **errorName** and **errorMessage** for failures
- **stackTrace** only when needed for debugging

Avoid logging:

- Secrets, passwords, tokens, or API keys
- Full personal data unless required and approved
- Large payloads that increase cost and noise
- Repeated duplicate messages at high volume

> Do not log sensitive data just because it is available in the event object. Redact or omit fields that are not needed for troubleshooting.

**Common Lambda logging patterns**

You can implement structured logging in several ways depending on runtime and tooling.

- **Manual JSON logging**: build log objects yourself and print them as JSON.
- **Logging libraries**: use a library that formats logs consistently and supports context injection.
- **Embedded metrics or tracing correlation**: combine logs with metrics and traces for deeper observability.

Manual logging is simple and portable. Libraries reduce boilerplate and help standardize fields across functions.

**Node.js example**

In Node.js, you can write JSON to `console.log()` and include the Lambda request ID from the context.

```javascript
export const handler = async (event, context) => {
  const log = {
    timestamp: new Date().toISOString(),
    level: "INFO",
    service: "orders-api",
    function: context.functionName,
    requestId: context.awsRequestId,
    message: "Processing order",
    orderId: event.orderId
  };

  console.log(JSON.stringify(log));

  try {
    // business logic here
    console.log(JSON.stringify({
      timestamp: new Date().toISOString(),
      level: "INFO",
      service: "orders-api",
      function: context.functionName,
      requestId: context.awsRequestId,
      message: "Order processed successfully",
      orderId: event.orderId
    }));

    return { statusCode: 200, body: "ok" };
  } catch (err) {
    console.error(JSON.stringify({
      timestamp: new Date().toISOString(),
      level: "ERROR",
      service: "orders-api",
      function: context.functionName,
      requestId: context.awsRequestId,
      message: "Order processing failed",
      errorName: err.name,
      errorMessage: err.message
    }));
    throw err;
  }
};
```

**Python example**

Python’s standard `logging` module can emit JSON if you format records yourself or use a helper library.

```python
import json
import logging
from datetime import datetime

logger = logging.getLogger()
logger.setLevel(logging.INFO)

def log_event(level, message, context, **fields):
    record = {
        "timestamp": datetime.utcnow().isoformat() + "Z",
        "level": level,
        "service": "orders-api",
        "function": context.function_name,
        "requestId": context.aws_request_id,
        "message": message,
        **fields
    }
    print(json.dumps(record))

def lambda_handler(event, context):
    log_event("INFO", "Processing order", context, orderId=event.get("orderId"))

    try:
        # business logic here
        log_event("INFO", "Order processed successfully", context, orderId=event.get("orderId"))
        return {"statusCode": 200, "body": "ok"}
    except Exception as exc:
        log_event(
            "ERROR",
            "Order processing failed",
            context,
            orderId=event.get("orderId"),
            errorName=type(exc).__name__,
            errorMessage=str(exc)
        )
        raise
```

**Java example**

In Java, you often use a logging framework and emit JSON through a formatter or structured logger.

```java
import com.fasterxml.jackson.databind.ObjectMapper;
import java.time.Instant;
import java.util.HashMap;
import java.util.Map;

public class Handler {
    private static final ObjectMapper mapper = new ObjectMapper();

    public String handleRequest(Map<String, Object> event, Context context) throws Exception {
        Map<String, Object> log = new HashMap<>();
        log.put("timestamp", Instant.now().toString());
        log.put("level", "INFO");
        log.put("service", "orders-api");
        log.put("function", context.getFunctionName());
        log.put("requestId", context.getAwsRequestId());
        log.put("message", "Processing order");
        log.put("orderId", event.get("orderId"));

        System.out.println(mapper.writeValueAsString(log));
        return "ok";
    }
}
```

**CloudWatch Logs Insights with structured logs**

CloudWatch Logs Insights works best when your logs are structured and fields are consistent. If your log lines are JSON, Insights can parse them automatically or with `parse`.

Example query:

```sql
fields @timestamp, level, message, requestId, orderId
| filter level = "ERROR"
| sort @timestamp desc
| limit 20
```

Another useful query for latency analysis:

```sql
fields @timestamp, message, durationMs, function
| filter level = "INFO" and ispresent(durationMs)
| stats avg(durationMs), max(durationMs), min(durationMs) by function
```

If your logs are not automatically parsed, you can extract fields manually:

```sql
fields @timestamp, @message
| parse @message '"level":"*"' as level
| parse @message '"requestId":"*"' as requestId
| filter level = "ERROR"
| sort @timestamp desc
```

**Correlation and tracing**

Structured logs become much more powerful when you include identifiers that connect events across systems.

- **requestId** identifies a single Lambda invocation.
- **correlationId** links multiple services in one business transaction.
- **traceId** can connect logs to AWS X-Ray or other distributed tracing systems.

If API Gateway invokes Lambda, you can pass a correlation ID in a header such as `X-Correlation-Id` and copy it into every log line. If the request enters through an event source like SQS or EventBridge, you may need to generate or propagate the ID yourself.

Example propagation pattern:

```json
{
  "level": "INFO",
  "message": "Received event",
  "requestId": "lambda-request-id",
  "correlationId": "client-correlation-id",
  "source": "api-gateway"
}
```

**Using AWS Lambda Powertools**

AWS Lambda Powertools is a common choice for structured logging, tracing, and metrics. It reduces boilerplate and standardizes log output.

Python example:

```python
from aws_lambda_powertools import Logger

logger = Logger(service="orders-api")

@logger.inject_lambda_context
def lambda_handler(event, context):
    logger.info("Processing order", extra={"orderId": event.get("orderId")})
    return {"statusCode": 200, "body": "ok"}
```

Node.js example:

```javascript
import { Logger } from "@aws-lambda-powertools/logger";

const logger = new Logger({ serviceName: "orders-api" });

export const handler = async (event, context) => {
  logger.addContext(context);
  logger.info("Processing order", { orderId: event.orderId });
  return { statusCode: 200, body: "ok" };
};
```

Powertools is useful when you want:

- Automatic injection of Lambda context fields
- Consistent JSON formatting
- Easier correlation across logs, metrics, and traces
- Less custom logging code in each function

**Configuration and deployment considerations**

Structured logging is mostly an application concern, but a few platform settings matter.

- **Log retention**: set CloudWatch log group retention to control cost.
- **Log level control**: use environment variables such as `LOG_LEVEL` to reduce noise in production.
- **Sampling**: log more detail for errors than for successful requests.
- **Cold start visibility**: log initialization events separately if needed.

Example environment variable usage:

```bash
LOG_LEVEL=INFO
SERVICE_NAME=orders-api
```

Example SAM template snippet:

```yaml
Resources:
  OrdersFunction:
    Type: AWS::Serverless::Function
    Properties:
      Handler: app.lambda_handler
      Runtime: python3.12
      Environment:
        Variables:
          LOG_LEVEL: INFO
          SERVICE_NAME: orders-api
      Policies:
        - AWSLambdaBasicExecutionRole
```

**Operational best practices**

- **Keep field names stable** across all functions.
- **Use lowercase or consistent casing** for keys such as `level`, `message`, and `requestId`.
- **Log at the right level**:
  - `DEBUG` for detailed diagnostics
  - `INFO` for normal business events
  - `WARN` for recoverable issues
  - `ERROR` for failures
- **Prefer one event per log line** so queries remain simple.
- **Redact sensitive values** before logging.
- **Include business identifiers** such as `orderId` or `userId` when they help troubleshooting.
- **Avoid logging entire event payloads** unless the payload is small and safe.

**Troubleshooting workflow**

A practical workflow for debugging Lambda with structured logs is:

1. Find the failing invocation in CloudWatch Logs.
2. Filter by `level = "ERROR"` or by `requestId`.
3. Inspect the surrounding `INFO` events for context.
4. Check downstream service calls and latency fields.
5. Correlate with metrics, traces, or DLQ messages if the failure is asynchronous.

This approach is faster than reading unstructured text because you can search by exact fields instead of scanning paragraphs of log output.

**Common exam-relevant distinctions**

- **Structured logging** improves searchability and analysis; it is not the same as tracing.
- **CloudWatch Logs** stores the output; it does not automatically make logs structured.
- **JSON formatting** is a common implementation choice, but the key idea is consistent key-value records.
- **Correlation IDs** help across services, while **request IDs** identify a single Lambda invocation.
- **Powertools** is a convenience library, not a requirement.

**Quick reference**

| Concept | Purpose | Example |
|---|---|---|
| `requestId` | Identify one Lambda invocation | `context.awsRequestId` |
| `correlationId` | Link related requests across services | `X-Correlation-Id` header |
| `level` | Classify severity | `INFO`, `ERROR` |
| `message` | Human-readable summary | `Order created successfully` |
| `durationMs` | Measure performance | `42` |
| `stackTrace` | Debug failures | Exception trace output |

**Key takeaways**

Structured logging in Lambda gives you logs that are easier to query, filter, and automate against. Use consistent JSON fields, include invocation and correlation identifiers, and avoid sensitive data. For exam purposes, remember that Lambda sends logs to CloudWatch, and structured logs make CloudWatch Logs Insights and operational troubleshooting much more effective.
