## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**What trace interpretation tells you**

AWS X-Ray helps you understand how a request moves through distributed systems. You use it to identify where latency is introduced, which downstream dependency failed, and whether a slowdown comes from your code, an AWS service call, or a retry pattern.

For the DVA-C02 exam, focus on three ideas:

- **Trace**: the end-to-end record of a request as it crosses services.
- **Segment**: the portion of the trace produced by one service or application component.
- **Subsegment**: a smaller timed unit inside a segment, often used for downstream calls such as HTTP requests, SDK calls, or database access.

A trace is only useful if you can connect the timing data to the architecture. That means reading the service map, understanding parent-child relationships, and recognizing when a dependency is synchronous, asynchronous, sampled, or not instrumented at all.

**X-Ray architecture and prerequisites**

X-Ray does not magically see every call. You need supported services, active tracing, and instrumentation in the application or runtime where required. In practice, that often means:

- Enabling **active tracing** on supported AWS services such as Lambda or API Gateway.
- Instrumenting application code with the X-Ray SDK or OpenTelemetry-to-X-Ray path where applicable.
- Patching supported clients so downstream calls create subsegments.
- Running the X-Ray daemon or using the managed integration path required by the runtime.

> Do not assume every AWS API call appears automatically as a subsegment. If the SDK/client is not instrumented or patched, the call may not be visible in the trace.

A common exam trap is assuming X-Ray can infer all dependencies from CloudTrail or service logs. It cannot. X-Ray shows what was traced, not every action that occurred.

**Trace structure and correlation**

X-Ray correlates work across services using a trace ID and propagated trace context. The incoming request carries a trace header, and each downstream hop can continue the same trace while creating its own segment or subsegment.

Typical header propagation uses the `X-Amzn-Trace-Id` header for HTTP-based calls. The header carries the trace identity and parent context so downstream services can attach their work to the same trace.

Example header:

```http
X-Amzn-Trace-Id: Root=1-65f1c2ab-3d2f4a7b9c1d2e3f4a5b6c7d;Parent=53995c3f42cd8ad8;Sampled=1
```

Key points:

- **Root** identifies the trace.
- **Parent** identifies the upstream segment or subsegment.
- **Sampled** indicates whether the request is traced.

This matters when you inspect multi-service flows:

- **API Gateway to Lambda**: the request context is propagated so the Lambda invocation appears as part of the same trace when tracing is enabled.
- **Lambda to HTTP service**: the SDK or HTTP client must be instrumented to create a downstream subsegment.
- **Lambda to SQS/SNS**: the trace context may not behave like a synchronous HTTP chain; you often need to reason about message propagation and separate traces.
- **ECS/EC2 service-to-service calls**: you need instrumentation on both sides or at least on the caller to see the outbound dependency.

**Segments, subsegments, and what they mean**

A segment represents a service’s view of the request. A subsegment represents a timed dependency or internal operation inside that service.

Common subsegment types include:

- **HTTP client calls**
- **AWS SDK calls**
- **SQL/database calls**
- **Custom code blocks** you annotate manually

A useful mental model:

- The **segment** answers: “How long did this service take?”
- The **subsegment** answers: “Which part of that time was spent on a dependency or internal step?”

Example of a custom subsegment in Python:

```python
from aws_xray_sdk.core import xray_recorder
from aws_xray_sdk.core import patch_all

patch_all()

def handler(event, context):
    xray_recorder.begin_subsegment("validate_input")
    try:
        # validation logic
        pass
    finally:
        xray_recorder.end_subsegment()
```

Use custom subsegments when you need visibility into code paths that are not automatically traced, such as validation, transformation, or retry loops.

**Annotations versus metadata**

X-Ray lets you attach extra data to traces in two different ways:

- **Annotations** are indexed and searchable.
- **Metadata** is not indexed and is used for detailed context.

Use annotations for fields you want to query later, such as:

- `customerTier`
- `orderStatus`
- `tenantId`
- `environment`

Use metadata for richer debugging details, such as:

- full request payloads
- response bodies
- stack traces
- internal debug objects

Example:

```python
from aws_xray_sdk.core import xray_recorder

subsegment = xray_recorder.current_subsegment()
subsegment.put_annotation("tenantId", "t-123")
subsegment.put_metadata("payload", {"items": 3, "source": "mobile"}, "debug")
```

Practical rule:

- If you need to **search** for it, use an annotation.
- If you only need to **inspect** it during investigation, use metadata.

**Sampling and why traces may be missing**

X-Ray does not record every request by default. Sampling reduces cost and overhead by tracing only a subset of requests.

Sampling behavior uses two main ideas:

- **Reservoir**: a guaranteed number of traces per second for a service or rule.
- **Fixed rate**: a percentage of additional requests sampled after the reservoir is used.

This means:

- Low traffic services may still get traces because the reservoir ensures a baseline.
- High traffic services may only show a fraction of requests.
- A missing trace does not always mean the request failed or was not executed.

Sampling rules are important when interpreting latency and error patterns. If a rare failure is not sampled, you may not see it in X-Ray even though it happened.

Example of a sampling rule document:

```json
{
  "version": 2,
  "rules": [
    {
      "description": "Trace API requests",
      "service_name": "*",
      "http_method": "*",
      "url_path": "/api/*",
      "fixed_rate": 0.05,
      "reservoir_size": 1
    }
  ],
  "default": {
    "fixed_rate": 0.01,
    "reservoir_size": 1
  }
}
```

Interpretation tips:

- Use a higher sampling rate temporarily during incident analysis.
- Remember that sampling can hide rare downstream failures.
- Compare X-Ray with logs and metrics when you need full coverage.

**Reading the service map**

The service map gives you a topology view of traced services and their dependencies. It is useful for spotting bottlenecks and failure hotspots quickly.

Look for these indicators:

- **Node color** often reflects health or error conditions.
- **Edge thickness** can indicate traffic volume.
- **Latency** shows how long requests spend in a node or across an edge.
- **Fault rate** shows server-side failures.
- **Error rate** shows non-fault errors, often 4xx-class or application-level errors depending on context.
- **Throttle indicators** may appear when a service is being rate-limited.

Use the service map to answer questions like:

- Which dependency is slow?
- Which service is returning errors?
- Is the issue isolated to one path or spread across the system?
- Is the bottleneck in the caller, the callee, or the network path between them?

A practical interpretation pattern:

- High **latency** with low errors often suggests slow downstream processing, retries, or queueing.
- High **fault rate** suggests server-side failure in the service or dependency.
- High **error rate** with low faults may indicate validation issues, bad requests, or client-side problems.
- A thick edge with normal latency may simply mean high traffic, not a problem.

**Trace timeline and latency analysis**

The trace detail view shows a timeline of segments and subsegments. This is where you identify where time is spent.

When you inspect a trace, ask:

1. Which segment started the request?
2. Which subsegment consumed the most time?
3. Did the caller wait on a downstream dependency?
4. Did retries or backoff inflate the total duration?
5. Is the delay inside the service or outside it?

Common latency patterns:

- **Long parent segment, short subsegments**: internal application work is slow.
- **Short parent segment, long downstream subsegment**: dependency latency dominates.
- **Multiple repeated subsegments**: retries are likely.
- **Gaps between segments**: asynchronous handoff, queueing, or missing instrumentation.

Retries and exponential backoff are especially important. A request can look slow without a hard failure because the client retries a transient error before succeeding. In traces, this often appears as repeated downstream calls or a long subsegment with multiple attempts.

> A successful response does not mean the path was healthy. Retries can hide transient failures while still increasing end-to-end latency.

**Synchronous versus asynchronous dependencies**

Trace interpretation changes when the architecture is event-driven.

For synchronous calls:

- The caller waits for the callee.
- Latency is directly visible in the parent request.
- Parent-child timing is easier to interpret.

For asynchronous flows:

- The producer and consumer are decoupled.
- The original request may finish before the downstream work starts.
- You may need to correlate traces using message IDs, annotations, logs, or business identifiers.

Common event-driven patterns:

- **Lambda to SQS**: the Lambda invocation may enqueue a message and return quickly; the actual processing happens later in a separate consumer trace.
- **Lambda to SNS**: publishing is fast, but subscribers process independently.
- **SQS to Lambda**: the queue introduces delay, so end-to-end latency includes queue wait time, not just function execution.
- **EventBridge-driven workflows**: causality is distributed across events rather than a single synchronous chain.

How to interpret these flows:

- Do not expect a single continuous timeline across all asynchronous hops.
- Look for shared annotations or business IDs to connect producer and consumer traces.
- Use CloudWatch metrics and logs alongside X-Ray to understand queue depth, age of oldest message, and consumer lag.

**Common AWS trace patterns for exam prep**

These patterns show up often in architecture questions.

**API Gateway to Lambda to DynamoDB**

- API Gateway receives the request and passes trace context to Lambda when tracing is enabled.
- Lambda executes business logic.
- The DynamoDB call appears as a downstream subsegment if the SDK is instrumented.
- High latency may come from Lambda cold start, business logic, or DynamoDB throttling.

What to check:

- API Gateway latency versus integration latency
- Lambda duration and cold start indicators
- DynamoDB throttles, retries, and consumed capacity

**Lambda to SQS**

- Lambda enqueues a message and returns.
- The enqueue operation may appear as an SDK subsegment.
- The consumer trace is separate and may occur later.
- If the queue grows, the issue is often downstream processing capacity, not the producer.

What to check:

- SQS queue depth
- message age
- consumer concurrency
- dead-letter queue activity

**ECS or EC2 service to service call**

- The caller creates an outbound HTTP subsegment if instrumented.
- The callee creates its own segment if it is instrumented.
- A slow edge often indicates network latency, downstream slowness, or retries.

What to check:

- caller-side timeout settings
- connection pooling
- DNS resolution issues
- downstream service saturation

**API Gateway to Lambda to external HTTP API**

- API Gateway starts the trace.
- Lambda handles the request.
- The external call appears only if the HTTP client is instrumented.
- If the client is not patched, the external dependency may be invisible even though it is the real bottleneck.

**Interpreting downstream dependency failures**

When a dependency fails, X-Ray can help you distinguish among several cases:

- **Fault**: a server-side failure, often 5xx-class or an unhandled exception.
- **Error**: a non-fault problem, often 4xx-class or application-level rejection, depending on the integration.
- **Throttle**: the service is rate-limiting requests.
- **Timeout**: the caller waited too long for a response.

Be careful with broad mappings. The exact classification can vary by service integration and instrumentation path. For exam purposes, focus on the general meaning rather than assuming every 4xx or 5xx response maps identically in every service.

Useful interpretation clues:

- Repeated throttles often mean you need backoff, retries, or higher service limits.
- Timeouts often mean the caller timeout is too short, the dependency is slow, or the network path is unstable.
- Faults often point to code defects, unavailable dependencies, or unhandled exceptions.
- Errors often point to invalid input, authorization issues, or request validation failures.

**CLI commands you should recognize**

X-Ray CLI commands are useful for trace discovery and retrieval, but remember their scope.

List trace summaries:

```bash
aws xray get-trace-summaries \
  --start-time 2026-04-25T00:00:00Z \
  --end-time 2026-04-26T00:00:00Z
```

This returns **trace summaries**, not full trace documents. Use it to find candidate traces by time window, service, response time, or error characteristics.

Retrieve full trace documents by trace ID:

```bash
aws xray batch-get-traces \
  --trace-ids 1-65f1c2ab-3d2f4a7b9c1d2e3f4a5b6c7d \
  --trace-ids 1-65f1c2ac-abcdef0123456789abcdef01
```

X-Ray trace IDs follow the `1-8hex-24hex` format. Use valid IDs from summaries or the console.

Get service graph data:

```bash
aws xray get-service-graph \
  --start-time 2026-04-25T00:00:00Z \
  --end-time 2026-04-26T00:00:00Z
```

This helps you inspect service relationships programmatically.

**Practical investigation workflow**

1. Start with the **service map** to identify the slow or failing node.
2. Open a representative **trace** from that node.
3. Inspect the **timeline** for the longest segment or subsegment.
4. Check whether the delay is in the caller, the dependency, or a retry loop.
5. Compare with **CloudWatch metrics** such as latency, errors, throttles, queue depth, and concurrency.
6. Use **logs** to confirm the exact exception, request ID, or downstream response.
7. If needed, adjust **sampling** temporarily to capture more examples.

**How to reason about missing or partial traces**

A trace may be incomplete for several reasons:

- Sampling excluded the request.
- A service is not supported or not actively traced.
- The client library was not instrumented.
- The request crossed an asynchronous boundary.
- A downstream service did not propagate trace context.

When a trace looks broken, do not immediately assume the application failed. First check whether the architecture supports end-to-end tracing for that path.

**Exam-focused interpretation cues**

- If a question asks for the best tool to visualize dependencies, think **service map**.
- If it asks where time was spent in a request, think **trace timeline** and **subsegments**.
- If it asks how to search for a business-specific request, think **annotations**.
- If it asks why traces are missing, think **sampling** and **instrumentation**.
- If it asks about asynchronous messaging, remember that causality is not always a single continuous trace.

**Quick comparison**

| Concept | What it tells you | Searchable | Typical use |
|---|---|---:|---|
| Trace | End-to-end request path | No | Investigate one request |
| Segment | Service-level work | No | Measure a service’s contribution |
| Subsegment | Downstream call or internal step | No | Find the slow dependency |
| Annotation | Indexed trace attribute | Yes | Filter and search traces |
| Metadata | Extra diagnostic context | No | Store rich debug details |

**Practical examples of what to look for**

- A Lambda function with a long duration and a short DynamoDB subsegment likely spends time in application logic or retries.
- An API Gateway request with a fast Lambda segment but a slow external HTTP subsegment likely waits on the external API.
- An SQS consumer with long gaps before invocation may be delayed by queue backlog rather than code execution.
- An ECS service with repeated outbound HTTP subsegments may be retrying a flaky downstream service.

**Troubleshooting mindset**

Use X-Ray as a correlation tool, not a standalone truth source. The best investigations combine:

- **X-Ray** for request flow and timing
- **CloudWatch metrics** for aggregate health
- **CloudWatch Logs** for error details
- **Service-specific metrics** for throttling, queue depth, and concurrency

That combination helps you distinguish between:

- slow code
- slow dependencies
- missing instrumentation
- sampling gaps
- asynchronous delay
- retry amplification
