## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Observability model: why correlation matters**

Correlating **logs**, **metrics**, and **traces** lets you move from “something is wrong” to “this request failed because this downstream call timed out.” In AWS, each signal answers a different question:

- **Metrics** tell you *how much* and *how often*.
- **Logs** tell you *what happened*.
- **Traces** tell you *where time was spent* across services.

A strong troubleshooting workflow uses all three together. You typically start with a metric alarm, inspect logs for context, and then use traces to follow the request path through distributed systems.

**Core concepts and signal roles**

<dl>
<dt><strong>Metrics</strong></dt>
<dd>Time-series values such as latency, error count, throttles, CPU, or queue depth. Use them for alerting, dashboards, and trend analysis.</dd>
<dt><strong>Logs</strong></dt>
<dd>Event records with timestamps and context. Use them to capture request IDs, error messages, stack traces, and business events.</dd>
<dt><strong>Traces</strong></dt>
<dd>End-to-end request journeys made of spans. Use them to identify service boundaries, latency contributors, and dependency failures.</dd>
</dl>

In AWS, the most common observability stack for application correlation is:

- **Amazon CloudWatch** for metrics, logs, alarms, and dashboards.
- **AWS X-Ray** for distributed tracing.
- **AWS Distro for OpenTelemetry (ADOT)** or the OpenTelemetry SDKs/Collectors for instrumenting applications and exporting telemetry.

**How correlation works in practice**

Correlation depends on shared identifiers and consistent timestamps. The most useful identifiers are:

- **Request ID** or **correlation ID** in application logs.
- **Trace ID** and **span ID** in tracing systems.
- **Dimensions** in metrics, such as service name, environment, or API route.

A typical flow looks like this:

1. A CloudWatch alarm fires on elevated `5XXError` or `Latency`.
2. You inspect application logs for the same time window and filter by request ID or error code.
3. You open the corresponding trace in X-Ray to see which downstream call consumed time or failed.
4. You compare the trace with metrics from the service, database, or queue to confirm whether the issue is load, throttling, retries, or dependency latency.

**CloudWatch metrics: what to watch**

CloudWatch metrics are the first signal for detection. They are best for identifying patterns, thresholds, and anomalies.

Common application and infrastructure metrics include:

- **Latency** for API Gateway, Application Load Balancer, Lambda, and custom application timing.
- **Error rates** such as `4XXError`, `5XXError`, or function errors.
- **Throughput** such as requests per second, invocations, or messages processed.
- **Resource saturation** such as CPU, memory, disk, or concurrency.
- **Queue and stream health** such as `ApproximateNumberOfMessagesVisible` for SQS or iterator age for Kinesis.

Use metrics when you need:

- Fast alerting with alarms.
- Trend analysis over time.
- A low-cardinality signal that scales well.
- A dashboard view of service health.

Example CloudWatch alarm for Lambda errors:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "OrdersApi-LambdaErrors" \
  --namespace "AWS/Lambda" \
  --metric-name "Errors" \
  --dimensions Name=FunctionName,Value=orders-api-handler \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --treat-missing-data notBreaching
```

**CloudWatch logs: what to include**

Logs provide the context that metrics lack. To make logs useful for correlation, include structured fields that can be searched consistently.

Recommended log fields:

- **timestamp**
- **level**
- **service**
- **environment**
- **requestId**
- **traceId**
- **spanId**
- **userId** or tenant identifier, if appropriate
- **operation**
- **statusCode**
- **latencyMs**
- **errorMessage**

Structured JSON logs are easier to query than free-form text. For example:

```json
{
  "timestamp": "2026-04-26T12:01:22.123Z",
  "level": "ERROR",
  "service": "orders-api",
  "environment": "prod",
  "requestId": "c2f1d8a9-7d5a-4d2f-9f1d-8b4c2f0c1e11",
  "traceId": "1-662c8f2a-3f0b2c1d4e5f678901234567",
  "operation": "CreateOrder",
  "statusCode": 502,
  "latencyMs": 3012,
  "errorMessage": "Downstream payment service timed out"
}
```

Use CloudWatch Logs Insights to search and aggregate logs quickly:

```sql
fields @timestamp, level, service, requestId, traceId, errorMessage
| filter service = "orders-api" and level = "ERROR"
| sort @timestamp desc
| limit 20
```

You can also group by error type or route to identify hotspots:

```sql
fields @timestamp, statusCode, operation
| filter service = "orders-api"
| stats count(*) as errors by statusCode, operation
| sort errors desc
```

**AWS X-Ray traces: how to read them**

X-Ray helps you understand distributed request flow across services such as API Gateway, Lambda, ECS, EC2, DynamoDB, SQS, and downstream HTTP calls.

A trace is made of:

- **Trace ID**: the identifier for the whole request path.
- **Segments**: data from a service handling the request.
- **Subsegments**: downstream calls made by that service.
- **Annotations**: indexed key-value pairs for filtering.
- **Metadata**: additional unindexed data for debugging.

Use X-Ray when you need to answer:

- Which service added the most latency?
- Did the failure happen in my code or a downstream dependency?
- Are retries or cold starts increasing response time?
- Is a specific route or tenant behaving differently?

Example of adding annotations in code so you can filter traces:

```python
from aws_xray_sdk.core import xray_recorder

segment = xray_recorder.current_segment()
segment.put_annotation("service", "orders-api")
segment.put_annotation("operation", "CreateOrder")
segment.put_annotation("environment", "prod")
```

Annotations are useful because they are searchable in X-Ray, unlike arbitrary metadata.

**OpenTelemetry and ADOT**

OpenTelemetry is an open standard for generating traces, metrics, and logs. In AWS, ADOT packages OpenTelemetry components for AWS environments.

Use OpenTelemetry when you want:

- Vendor-neutral instrumentation.
- Consistent telemetry across multiple languages and services.
- Export to AWS services such as X-Ray and CloudWatch, or to other backends.

A common pattern is:

- Application emits telemetry using OpenTelemetry SDK.
- OpenTelemetry Collector receives telemetry.
- Collector exports traces to X-Ray and metrics/logs to the appropriate destination.

Example collector configuration that exports traces to X-Ray:

```yaml
receivers:
  otlp:
    protocols:
      grpc:
      http:

processors:
  batch:

exporters:
  awsxray: {}

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [awsxray]
```

**Practical correlation workflow**

When an incident occurs, use a repeatable sequence.

1. **Start with the metric**
   - Check the alarm, dashboard, or anomaly.
   - Identify the time window and affected dimension, such as function name, API route, or queue.

2. **Find matching logs**
   - Filter by the same time window.
   - Search for request IDs, trace IDs, error messages, or status codes.
   - Look for retries, timeouts, throttling, or dependency failures.

3. **Open the trace**
   - Use the trace ID from the log or request header.
   - Inspect the critical path and slowest spans.
   - Compare service timings with the metric spike.

4. **Confirm the root cause**
   - Determine whether the issue is application code, downstream service latency, resource exhaustion, or configuration.
   - Validate with a second signal before changing production settings.

**Example: API latency investigation**

Suppose an API Gateway endpoint becomes slow.

- **Metric**: `Latency` rises from 120 ms to 3 s.
- **Log**: application logs show repeated `Downstream payment service timed out`.
- **Trace**: X-Ray shows the payment call span taking 2.8 s, with retries.
- **Conclusion**: the bottleneck is the downstream dependency, not the API handler itself.

This pattern is common in microservices because the visible symptom often appears at the edge service, while the real issue is deeper in the call chain.

**Example: Lambda error investigation**

A Lambda function starts failing after a deployment.

- **Metric**: `Errors` increases and `Duration` approaches the timeout.
- **Log**: stack trace shows a missing environment variable.
- **Trace**: the trace ends early with an exception in the initialization path.
- **Conclusion**: the deployment introduced a configuration issue.

For Lambda, also watch:

- **Cold starts** if latency spikes only on first invocation.
- **Concurrency** if throttling or queue buildup appears.
- **Timeouts** if downstream calls exceed the configured function timeout.

**Correlation patterns by AWS service**

| Service | Useful metric | Useful log clue | Useful trace clue |
|---|---|---|---|
| API Gateway | `Latency`, `5XXError`, `4XXError` | request path, status code, integration error | upstream-to-downstream timing |
| Lambda | `Errors`, `Duration`, `Throttles` | stack trace, request ID, timeout | cold start, downstream span timing |
| ECS/EKS | CPU, memory, task restarts | container logs, exit code | service-to-service call timing |
| DynamoDB | throttles, consumed capacity | conditional check failures | client retries, hot partition symptoms |
| SQS | queue depth, age of oldest message | consumer failures, message IDs | processing delay across consumers |
| Kinesis | iterator age, read throttles | shard processing errors | lag between ingestion and processing |

**Designing for correlation**

To make correlation reliable, design telemetry into the application from the start.

- **Propagate correlation IDs** across service boundaries.
- **Log in structured JSON** rather than plain text where possible.
- **Use consistent naming** for services, routes, and environments.
- **Add trace annotations** for searchable business context.
- **Emit custom metrics** for business events such as checkout failures or order completion time.
- **Keep timestamps in UTC** and use ISO 8601 format.

Example middleware pattern for propagating a request ID:

```javascript
app.use((req, res, next) => {
  const requestId = req.headers['x-request-id'] || crypto.randomUUID();
  req.requestId = requestId;
  res.setHeader('x-request-id', requestId);
  next();
});
```

Then include the same ID in logs and, when possible, in trace annotations.

**Custom metrics: when logs are not enough**

Logs are detailed but expensive to query at scale. Custom metrics are useful when you need a compact signal for a business event.

Examples:

- Order checkout failures
- Payment authorization latency
- Cache hit ratio
- Messages sent to a dead-letter queue
- Number of retries per request

Example using the CloudWatch Embedded Metric Format (EMF):

```json
{
  "_aws": {
    "Timestamp": 1714132882123,
    "CloudWatchMetrics": [
      {
        "Namespace": "OrdersService",
        "Dimensions": [["Service", "Environment"]],
        "Metrics": [
          { "Name": "CheckoutFailures", "Unit": "Count" },
          { "Name": "CheckoutLatency", "Unit": "Milliseconds" }
        ]
      }
    ]
  },
  "Service": "orders-api",
  "Environment": "prod",
  "CheckoutFailures": 1,
  "CheckoutLatency": 3012
}
```

Use custom metrics when you need alarms or dashboards for a business KPI, not just raw infrastructure health.

**Troubleshooting checklist**

- Confirm the time window and affected environment.
- Check the top-level metric first.
- Search logs for the same request ID or trace ID.
- Look for retries, throttling, timeouts, and dependency errors.
- Inspect the trace for the slowest span and failure point.
- Compare with related service metrics, such as database latency or queue depth.
- Verify whether the issue is isolated to one route, tenant, or deployment version.

**Common exam-relevant distinctions**

- **Metrics** are best for alerting and trends.
- **Logs** are best for detailed context and debugging.
- **Traces** are best for distributed latency analysis.
- **Annotations** in X-Ray are searchable; **metadata** is not.
- **Structured logs** improve correlation far more than unstructured text.
- **Correlation IDs** help connect signals across services.
- **OpenTelemetry** is the standard instrumentation approach; **ADOT** is AWS’s distribution.

**Operational best practices**

- Use alarms on symptoms that matter, not on every raw metric.
- Keep log retention aligned with incident response and compliance needs.
- Sample traces carefully if traffic is high, but preserve enough data for debugging.
- Instrument critical paths first: authentication, checkout, payment, persistence, and async consumers.
- Validate observability in non-production before release.
- Test failure scenarios so you know which signals appear when a dependency breaks.

**Quick reference: what to use first**

- **Need an alert?** Use metrics.
- **Need request context?** Use logs.
- **Need end-to-end latency breakdown?** Use traces.
- **Need to connect all three?** Use a shared request ID or trace ID and consistent timestamps.

**Minimal end-to-end example**

A request enters API Gateway, invokes Lambda, writes to DynamoDB, and publishes to SQS.

- API Gateway metric shows elevated latency.
- Lambda logs show retries on DynamoDB writes.
- X-Ray trace shows the DynamoDB subsegment as the slowest span.
- SQS queue depth rises because downstream processing is delayed.

That combined view tells you the issue is not just “the API is slow”; it is a downstream persistence bottleneck that cascades into asynchronous processing.
