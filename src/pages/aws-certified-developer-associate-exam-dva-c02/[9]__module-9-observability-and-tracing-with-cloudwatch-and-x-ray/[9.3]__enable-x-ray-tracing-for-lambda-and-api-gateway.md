## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Overview**

AWS X-Ray helps you trace requests as they move through distributed applications. For DVA-C02, the key idea is that you enable tracing on the services that support it, then use the X-Ray console, service map, and trace details to understand latency, errors, and downstream dependencies.

For this topic, focus on two common integration points:

- **AWS Lambda** can emit trace data when tracing is active.
- **API Gateway REST APIs** support X-Ray tracing at the stage level.
- **HTTP APIs** are not the same as REST APIs for X-Ray tracing; for exam purposes, know that X-Ray tracing is associated with REST API stages.

**What X-Ray gives you**

X-Ray breaks a request into segments and subsegments so you can see where time is spent. In a typical serverless path, API Gateway may appear as the front-door service, Lambda appears as the compute segment, and downstream SDK calls can appear as subsegments if the code is instrumented.

- **Segments** represent the service handling the request.
- **Subsegments** represent work done inside that service, such as SDK calls or database access.
- **Trace IDs** correlate the request across services.
- **Sampling** controls how many requests are traced, so not every request necessarily appears.

> X-Ray visibility is regional. The traced resources and the X-Ray console view must be in the same AWS Region for you to see the trace there.

**Enable tracing on Lambda**

Lambda tracing is configured per function, and the setting is typically **Active** or **PassThrough**.

- **Active** tells Lambda to create and propagate trace data for the function.
- **PassThrough** lets Lambda continue an incoming trace header without forcing the function to generate its own trace segment.
- The deployed **version** or **alias** matters when you update tracing on a published function; make sure the traffic target is the version or alias you expect.

You can enable tracing in the console, with the CLI, or through infrastructure as code.

```bash
aws lambda update-function-configuration \
  --function-name my-function \
  --tracing-config Mode=Active
```

If you deploy with an alias, verify the alias points to the version with tracing enabled.

```bash
aws lambda publish-version --function-name my-function
aws lambda update-alias \
  --function-name my-function \
  --name prod \
  --function-version 12
```

A common deployment pattern is to enable tracing on the published version, then shift the alias to that version so production traffic uses the traced code path.

**Lambda runtime instrumentation**

Lambda can emit the service-level trace, but your code must be instrumented if you want deeper visibility into downstream calls. Use the AWS X-Ray SDK or OpenTelemetry where appropriate, and remember that async context propagation matters.

For Node.js, do not assume `getSegment()` always returns a value in an async handler unless the runtime and SDK context are correctly patched.

```javascript
const AWSXRay = require('aws-xray-sdk-core');
const AWS = AWSXRay.captureAWS(require('aws-sdk'));

exports.handler = async (event) => {
  const segment = AWSXRay.getSegment();
  if (segment) {
    const subsegment = segment.addNewSubsegment('work');
    try {
      const ddb = new AWS.DynamoDB.DocumentClient();
      await ddb.get({
        TableName: 'Orders',
        Key: { id: '123' }
      }).promise();
    } finally {
      subsegment.close();
    }
  }

  return {
    statusCode: 200,
    body: JSON.stringify({ ok: true })
  };
};
```

Use this pattern carefully: the SDK must be patched and the execution context must preserve the current segment. If you only need service-level tracing for the exam, know that enabling Lambda tracing is often enough to see the function in the trace map.

**Enable tracing on API Gateway**

For API Gateway, X-Ray tracing is enabled on the **stage** for a **REST API**. This is the exam-relevant distinction.

- Enable tracing on the stage, not on the individual method.
- The stage must be deployed after the setting changes.
- API Gateway then participates in the trace for requests that are sampled and supported.

Example CLI update for a REST API stage:

```bash
aws apigateway update-stage \
  --rest-api-id a1b2c3d4e5 \
  --stage-name prod \
  --patch-operations op=replace,path=/tracingEnabled,value=true
```

If you use infrastructure as code, the setting is typically expressed on the stage resource.

```yaml
Resources:
  ApiStage:
    Type: AWS::ApiGateway::Stage
    Properties:
      RestApiId: !Ref MyRestApi
      StageName: prod
      DeploymentId: !Ref ApiDeployment
      TracingEnabled: true
```

For exam prep, remember the contrast:

| API type | X-Ray tracing support | Key point |
|---|---:|---|
| REST API | Yes | Enable tracing on the stage |
| HTTP API | Not the same as REST API tracing | Do not assume the same X-Ray behavior |

**How the trace appears**

When tracing is working, you can usually see:

- A service map entry for API Gateway and Lambda.
- A trace detail page showing request timing.
- Downstream calls as subsegments if the code is instrumented.
- Error or fault annotations when the request fails.

A simple request flow looks like this:

```text
Client -> API Gateway REST API stage -> Lambda function -> DynamoDB
```

If the Lambda function calls another AWS service with the X-Ray SDK or supported instrumentation, that downstream call can appear as a subsegment under the Lambda segment.

**Verify in the console**

Use the X-Ray console to confirm that tracing is active and producing data.

1. Open the **X-Ray** console in the same Region as the resources.
2. Check the **Service map** for API Gateway and Lambda nodes.
3. Open **Traces** and inspect a recent request.
4. Confirm the trace shows the expected latency and downstream calls.
5. If nothing appears, check sampling, stage deployment, and Lambda tracing mode.

You can also verify the Lambda configuration directly:

```bash
aws lambda get-function-configuration \
  --function-name my-function \
  --query 'TracingConfig'
```

And inspect the API Gateway stage:

```bash
aws apigateway get-stage \
  --rest-api-id a1b2c3d4e5 \
  --stage-name prod \
  --query 'tracingEnabled'
```

**Troubleshooting checklist**

If traces do not appear, check the most common causes first.

- **Region mismatch**: the X-Ray console is regional, so inspect the same Region where the API and Lambda run.
- **Sampling**: low traffic or sampling rules can mean no trace is recorded for a given request.
- **Stage not deployed**: API Gateway stage changes require deployment to take effect.
- **Wrong API type**: REST API tracing is the exam-relevant feature; do not assume HTTP APIs behave the same way.
- **Lambda tracing mode**: confirm the function is set to **Active** when you expect Lambda to create trace data.
- **Alias/version mismatch**: traffic may still point to an older version or alias without the tracing setting you changed.
- **Missing instrumentation**: service-level tracing may work, but downstream subsegments require SDK instrumentation.
- **Permissions and service availability**: the region must support X-Ray, and the service must be available for trace collection there.

> If you enable tracing but still see no service map entry, first confirm that the request path is actually being invoked and that the sampled request reaches the traced Lambda version or alias.

**Practical deployment example**

A common serverless setup uses API Gateway as the front door and Lambda as the backend. You enable tracing on both sides, deploy the API stage, and then generate traffic.

```bash
aws lambda update-function-configuration \
  --function-name orders-handler \
  --tracing-config Mode=Active

aws apigateway update-stage \
  --rest-api-id a1b2c3d4e5 \
  --stage-name prod \
  --patch-operations op=replace,path=/tracingEnabled,value=true
```

Then invoke the API and inspect the trace:

```bash
curl https://a1b2c3d4e5.execute-api.us-east-1.amazonaws.com/prod/orders
```

In the X-Ray console, look for:

- The API Gateway segment at the edge of the request.
- The Lambda segment for the backend execution.
- Any subsegments for SDK calls such as DynamoDB, S3, or SNS.

**Exam focus**

For DVA-C02, the most testable facts are:

- **Lambda tracing** is enabled with the function tracing configuration.
- **API Gateway REST API tracing** is enabled on the stage.
- **HTTP APIs** are not interchangeable with REST APIs for X-Ray tracing questions.
- **X-Ray traces are regional**, so you inspect them in the same Region as the traced resources.
- **Alias and version targeting** matters when tracing is enabled on a deployed Lambda function.
- **Sampling** means tracing is not guaranteed for every request.

**Quick reference**

- Use `aws lambda update-function-configuration --tracing-config Mode=Active` to enable Lambda tracing.
- Use `aws apigateway update-stage ... tracingEnabled=true` for REST API stages.
- Verify with `aws lambda get-function-configuration` and `aws apigateway get-stage`.
- Confirm traces in the X-Ray **service map** and **trace list**.
- Instrument code if you need downstream subsegments, not just the service-level trace.
