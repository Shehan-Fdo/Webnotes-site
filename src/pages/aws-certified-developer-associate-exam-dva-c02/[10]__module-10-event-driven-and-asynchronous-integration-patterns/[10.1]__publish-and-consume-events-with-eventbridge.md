## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Amazon EventBridge overview**

Amazon EventBridge is a serverless event bus service that helps you route events from AWS services, SaaS applications, and your own applications to targets such as Lambda, SQS, SNS, Step Functions, Kinesis, API destinations, and other event buses. It is designed for loose coupling: producers emit events without knowing who consumes them.

Use EventBridge when you want:
- **Event routing** based on content, not just topic subscription.
- **Fan-out** to multiple targets with independent rules.
- **Centralized event governance** across accounts and applications.
- **Scheduled events** with cron or rate expressions.
- **Event replay** for recovery or testing.

EventBridge is not a general message queue. It does not guarantee FIFO ordering, and it is not a substitute for SQS when you need buffering, backpressure handling, or strict queue semantics.

**Core concepts**

EventBridge uses a few terms that appear frequently in exam questions:

- **Event**: A JSON document describing something that happened.
- **Event bus**: The routing layer that receives events and evaluates rules.
- **Rule**: A pattern that matches events and sends matching events to one or more targets.
- **Target**: The destination that receives matched events.
- **Archive**: A stored copy of events on an event bus for later replay.
- **Replay**: Re-delivery of archived events to an event bus over a selected time range.
- **Pipe**: A managed point-to-point integration that moves and optionally transforms events from a source to a target with filtering and enrichment.

A typical event contains fields such as `source`, `detail-type`, `detail`, `resources`, `account`, `region`, and `time`.

```json
{
  "version": "0",
  "id": "c1f2f0f0-1234-5678-90ab-abcdef123456",
  "detail-type": "OrderCreated",
  "source": "com.example.orders",
  "account": "123456789012",
  "time": "2026-04-26T12:00:00Z",
  "region": "us-east-1",
  "resources": [],
  "detail": {
    "orderId": "A1001",
    "amount": 149.99,
    "priority": "high"
  }
}
```

**Event buses and permissions**

EventBridge has multiple bus types:

- **Default bus**: Receives events from AWS services and custom applications when they are explicitly published with permission.
- **Custom bus**: A bus you create for your application or domain.
- **Partner bus**: Used for SaaS partner integrations.

Two permission questions often appear separately:

- **Who can put events on a bus?**
  - Controlled by IAM permissions and, for cross-account publishing, the event bus resource policy.
  - The caller must be allowed to call `events:PutEvents` on the target bus.
- **Who can create or manage rules on a bus?**
  - Controlled by IAM permissions such as `events:PutRule`, `events:PutTargets`, `events:DeleteRule`, and related actions.
  - These are management permissions, not publishing permissions.

> A bus does not automatically accept custom events just because it is the default bus. The caller still needs the required permissions.

A common pattern is to allow one account or application role to publish events to a central bus, while a separate operations role manages rules and targets.

**Publishing events**

You publish custom events with `PutEvents`. This is the primary API for application-generated events.

```bash
aws events put-events \
  --entries '[
    {
      "Source": "com.example.orders",
      "DetailType": "OrderCreated",
      "Detail": "{\"orderId\":\"A1001\",\"amount\":149.99,\"priority\":\"high\"}",
      "EventBusName": "orders-bus"
    }
  ]'
```

Important publishing details:
- `Source` is usually your application namespace, such as `com.example.orders`.
- `DetailType` is a short label for the event category.
- `Detail` must be a JSON string.
- `EventBusName` can be a bus name or ARN.

Use custom events when your application needs to notify multiple downstream consumers without direct coupling.

**AWS service events**

Many AWS services emit events to EventBridge automatically. This is useful for operational automation and event-driven workflows.

Common examples:
- **EC2** instance state-change notifications.
- **EBS** volume notifications.
- **CodeCommit** repository events.
- **CloudTrail management events** delivered to EventBridge.
- **ECS** task and service events.
- **AWS Health** events.
- **S3** bucket events when EventBridge notifications are enabled on the bucket.

For S3, you enable EventBridge notifications on the bucket, then route those events with rules. This is different from legacy S3 notification destinations.

Example S3 event flow:
1. An object is uploaded to an S3 bucket.
2. S3 emits an event to EventBridge because EventBridge notifications are enabled.
3. A rule matches the event pattern.
4. A target such as Lambda or SQS processes the event.

Example EC2 state-change flow:
1. An EC2 instance changes state.
2. EC2 publishes the state-change event to EventBridge.
3. A rule matches `running` or `stopped`.
4. A target triggers automation, such as a Lambda function or Step Functions workflow.

**Event pattern matching**

EventBridge rules use event patterns to match incoming events. Matching is structural and exact where specified. You match on fields such as `source`, `detail-type`, and nested `detail` properties.

Example rule pattern for EC2 stopped instances:

```json
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["stopped"]
  }
}
```

Example nested pattern for an application event:

```json
{
  "source": ["com.example.orders"],
  "detail-type": ["OrderCreated"],
  "detail": {
    "priority": ["high"],
    "amount": [{ "numeric": [">=", 100] }]
  }
}
```

Pattern matching notes:
- Arrays in the event pattern mean “match any of these values.”
- Nested objects are matched by path.
- Missing fields do not match unless you explicitly use `exists`.
- EventBridge supports operators such as:
  - `prefix`
  - `numeric`
  - `exists`
  - `anything-but`
  - `cidr`
  - `wildcard`
- Matching is not SQL-like; it is JSON pattern matching.

Examples of operator use:

```json
{
  "detail": {
    "filename": [{ "prefix": "reports/" }],
    "status": [{ "anything-but": ["FAILED", "CANCELLED"] }]
  }
}
```

```json
{
  "detail": {
    "ipAddress": [{ "cidr": "10.0.0.0/8" }]
  }
}
```

Practical limitations to remember:
- Pattern matching is exact for the fields you specify.
- You cannot use arbitrary expressions or joins.
- Arrays in the event payload are matched by element, not by complex query logic.
- If you need transformation before routing, consider a Pipe or target-side processing.

**Rules, targets, and fan-out**

A single event can match multiple rules. Each rule can send the event to one or more targets. This gives you fan-out without the producer knowing the consumers.

Common targets:
- **Lambda** for custom code and lightweight processing.
- **SQS** for buffering and decoupling.
- **SNS** for pub/sub fan-out to subscribers.
- **Step Functions** for orchestration.
- **Kinesis Data Streams** for streaming pipelines.
- **API destination** for calling external HTTP endpoints.
- **Another event bus** for event routing across domains or accounts.

Example rule that sends EC2 stop events to Lambda:

```bash
aws events put-rule \
  --name ec2-stopped-rule \
  --event-pattern '{
    "source": ["aws.ec2"],
    "detail-type": ["EC2 Instance State-change Notification"],
    "detail": { "state": ["stopped"] }
  }' \
  --event-bus-name default
```

Attach a Lambda target:

```bash
aws events put-targets \
  --rule ec2-stopped-rule \
  --event-bus-name default \
  --targets "Id"="1","Arn"="arn:aws:lambda:us-east-1:123456789012:function:HandleEc2Stop"
```

For Lambda targets, you must also grant EventBridge permission to invoke the function:

```bash
aws lambda add-permission \
  --function-name HandleEc2Stop \
  --statement-id AllowEventBridgeInvoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:123456789012:rule/ec2-stopped-rule
```

**Target permissions and roles**

Target permissions are a common exam trap. EventBridge can invoke some targets directly, but others require an IAM role that EventBridge assumes.

Use this mental model:
- **Lambda**: EventBridge needs permission to invoke the function.
- **SQS**: The queue policy must allow EventBridge to send messages.
- **SNS**: The topic policy must allow EventBridge to publish.
- **Step Functions**: EventBridge typically needs an IAM role that allows `states:StartExecution`.
- **Kinesis**: EventBridge needs permissions to put records.
- **Cross-account or cross-service targets**: Often require both a resource policy and an IAM role.

Example SQS target policy statement:

```json
{
  "Sid": "AllowEventBridgeSendMessage",
  "Effect": "Allow",
  "Principal": { "Service": "events.amazonaws.com" },
  "Action": "sqs:SendMessage",
  "Resource": "arn:aws:sqs:us-east-1:123456789012:orders-queue"
}
```

Example Step Functions target role trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "events.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example Step Functions permissions policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "states:StartExecution",
      "Resource": "arn:aws:states:us-east-1:123456789012:stateMachine:OrderWorkflow"
    }
  ]
}
```

**Retry behavior and failure handling**

EventBridge retries target delivery for many target types when delivery fails due to transient issues. The exact behavior depends on the target and integration, but the exam-relevant idea is that EventBridge is not “fire once and forget” in the sense of guaranteed success.

Key points:
- EventBridge can retry failed deliveries to supported targets.
- You can configure a **dead-letter queue** (DLQ) for failed target invocations.
- A DLQ is usually an SQS queue.
- Retries and DLQs are configured on the target side, not on the event producer.

Example target with DLQ and retry policy in CloudFormation:

```yaml
Resources:
  EventRule:
    Type: AWS::Events::Rule
    Properties:
      EventPattern:
        source:
          - com.example.orders
      Targets:
        - Id: LambdaTarget
          Arn: !GetAtt OrderHandler.Arn
          DeadLetterConfig:
            Arn: !GetAtt EventBridgeDLQ.Arn
          RetryPolicy:
            MaximumRetryAttempts: 3
            MaximumEventAgeInSeconds: 3600
```

Use a DLQ when you need to inspect events that could not be delivered after retries. This is especially useful for operational visibility and recovery.

**Archives and replay**

Archives let you store events from an event bus for later analysis or recovery. You create an archive on an event bus, and it captures events according to the archive configuration.

Important archive facts:
- Archives are associated with an **event bus**, not a rule.
- You can filter which events are archived.
- You can replay archived events back to an event bus.
- Replayed events are delivered as new events and can match rules again.
- Replay is useful for backfilling downstream systems or testing new rules.

Example archive creation:

```bash
aws events create-archive \
  --archive-name orders-archive \
  --event-source-arn arn:aws:events:us-east-1:123456789012:event-bus/orders-bus \
  --event-pattern '{
    "source": ["com.example.orders"]
  }'
```

Example replay:

```bash
aws events start-replay \
  --replay-name orders-replay-2026-04 \
  --event-source-arn arn:aws:events:us-east-1:123456789012:archive/orders-archive \
  --destination arn:aws:events:us-east-1:123456789012:event-bus/orders-bus \
  --event-start-time 2026-04-01T00:00:00Z \
  --event-end-time 2026-04-02T00:00:00Z
```

Replay considerations:
- Replayed events can trigger the same rules again.
- Downstream consumers should be idempotent.
- Replay is not a replacement for durable queueing.

**Global endpoints and failover**

EventBridge global endpoints help with disaster recovery and regional failover for event ingestion. They are relevant when you need a resilient event entry point across Regions.

Use global endpoints when:
- You want a primary and secondary Region.
- You need failover for event publishing.
- You want to reduce the operational burden of custom failover logic.

High-level behavior:
- You publish to a global endpoint.
- EventBridge routes to the primary Region under normal conditions.
- If health checks indicate failure, traffic can fail over to the secondary Region.

This is a reliability feature, not a routing pattern feature. It is useful in exam scenarios involving multi-Region event ingestion.

**EventBridge Pipes**

EventBridge Pipes is related to EventBridge but solves a different problem from rules.

Use Pipes when you need:
- A direct source-to-target integration.
- Filtering before delivery.
- Optional enrichment with a Lambda function or API call.
- Simpler point-to-point movement from sources such as SQS, Kinesis, DynamoDB Streams, or MSK to targets.

Use rules when you need:
- Event bus-based pub/sub.
- Multiple consumers.
- Centralized event routing from many producers.

Quick distinction:
- **Rules**: bus-centric, many-to-many routing.
- **Pipes**: source-to-target integration, often one-to-one.

Example use case:
- SQS queue contains order messages.
- A Pipe filters only high-priority orders.
- The Pipe enriches the message.
- The Pipe sends the result to Step Functions.

**EventBridge versus SNS versus SQS**

These services overlap in exam questions, but they solve different problems.

| Service | Best for | Delivery model | Filtering | Typical use |
|---|---|---|---|---|
| EventBridge | Event routing and integration | Event bus to multiple targets | Rich JSON event patterns | Domain events, AWS service events, automation |
| SNS | Simple pub/sub notifications | Topic to subscribers | Basic subscription filtering | Broadcast alerts, mobile push, email, fan-out |
| SQS | Durable buffering and decoupling | Queue to one consumer group | No native content routing like EventBridge | Work queues, backpressure, retries, async processing |

How to choose:
- Use **EventBridge** when the event content determines routing.
- Use **SNS** when you want to broadcast a notification to many subscribers with simple filtering.
- Use **SQS** when you need durable buffering and one consumer processes each message.

A common architecture is EventBridge -> SQS -> worker fleet when you want routing plus buffering.

**Cross-account and cross-Region routing**

EventBridge supports cross-account event delivery and event bus-to-event bus routing. This is useful for centralized logging, security, and shared services.

Important distinctions:
- **Publishing to another account’s bus** requires permission on the destination bus for `events:PutEvents`.
- **Creating rules in another account** requires management permissions in that account.
- These are separate concerns and should not be conflated.

For cross-account event bus routing:
- The destination bus must allow the source account or role to put events.
- The source side may need a rule that targets the destination bus.
- The target bus policy must explicitly permit the action.

Example bus policy statement:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSourceAccountPutEvents",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::111122223333:root" },
      "Action": "events:PutEvents",
      "Resource": "arn:aws:events:us-east-1:444455556666:event-bus/central-bus"
    }
  ]
}
```

Cross-Region routing is also possible by targeting a bus in another Region, but you should pay attention to latency, permissions, and operational design.

**Practical end-to-end example**

A realistic workflow is to process an S3 upload event and trigger downstream automation.

1. You enable EventBridge notifications on the S3 bucket.
2. S3 emits an object-created event to EventBridge.
3. A rule matches objects uploaded to a specific prefix.
4. The rule sends the event to a Lambda function.
5. Lambda validates the object and starts a Step Functions workflow.

Example event pattern:

```json
{
  "source": ["aws.s3"],
  "detail-type": ["Object Created"],
  "detail": {
    "bucket": {
      "name": ["my-upload-bucket"]
    },
    "object": {
      "key": [{ "prefix": "incoming/" }]
    }
  }
}
```

Example Lambda handler snippet:

```python
import json

def lambda_handler(event, context):
    detail = event.get("detail", {})
    bucket = detail.get("bucket", {}).get("name")
    key = detail.get("object", {}).get("key")

    print(f"Processing s3://{bucket}/{key}")
    return {
        "statusCode": 200,
        "body": json.dumps({"message": "accepted"})
    }
```

This pattern is common in exam scenarios because it combines:
- AWS service event source
- Event pattern filtering
- Lambda target invocation
- Downstream orchestration

**Common exam traps**

- Confusing **EventBridge** with SNS or SQS.
- Assuming the default bus automatically accepts custom events without permission.
- Treating rule creation permissions as the same as event publishing permissions.
- Forgetting that some targets need an IAM role or resource policy.
- Assuming EventBridge is the mechanism for Lambda async failure handling.
- Overstating CloudTrail coverage; management events are common, but not every CloudTrail data event is routed the same way.
- Describing archives as rule-scoped instead of bus-scoped.
- Forgetting that replay creates new deliveries that can trigger rules again.
- Using EventBridge when a queue is the better fit for buffering and backpressure.

**Quick decision guide**

- Use **EventBridge rules** for event-driven routing from AWS services or custom applications.
- Use **EventBridge Pipes** for direct source-to-target movement with filtering and enrichment.
- Use **SNS** for broadcast notifications.
- Use **SQS** for durable work queues.
- Use **Step Functions** when the event should start or continue a workflow.
- Use **Lambda** when the event needs code-based processing.

**CLI and troubleshooting tips**

Useful commands:
- `aws events put-events` to publish custom events.
- `aws events put-rule` to create a rule.
- `aws events put-targets` to attach targets.
- `aws events list-rules` to inspect rules.
- `aws events list-targets-by-rule` to verify target attachment.
- `aws events test-event-pattern` to validate a pattern against sample JSON.

Example pattern test:

```bash
aws events test-event-pattern \
  --event-pattern '{
    "source": ["aws.ec2"],
    "detail-type": ["EC2 Instance State-change Notification"],
    "detail": { "state": ["stopped"] }
  }' \
  --event '{
    "source": "aws.ec2",
    "detail-type": "EC2 Instance State-change Notification",
    "detail": { "state": "stopped" }
  }'
```

Troubleshooting checklist:
- Confirm the event is actually being published to the correct bus.
- Verify the rule pattern matches the real event shape.
- Check target permissions and resource policies.
- Inspect DLQs for failed deliveries.
- Confirm the target Region and account are correct.
- Use CloudWatch metrics and logs for delivery failures and invocation errors.

**Exam takeaway**

EventBridge is the AWS service for content-based event routing across AWS services, custom applications, and SaaS integrations. For DVA-C02, focus on:
- event buses, rules, targets, and permissions
- pattern matching and nested JSON fields
- target-specific permissions and DLQs
- archives, replay, and global endpoints
- when to choose EventBridge versus SNS, SQS, or Pipes
