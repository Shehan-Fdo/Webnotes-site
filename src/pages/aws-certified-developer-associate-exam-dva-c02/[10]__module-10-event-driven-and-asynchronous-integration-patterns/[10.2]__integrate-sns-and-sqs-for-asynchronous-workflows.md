## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why SNS and SQS together**

Amazon Simple Notification Service (SNS) and Amazon Simple Queue Service (SQS) solve different parts of the same problem: decoupling producers from consumers. SNS is a pub/sub service that pushes messages to multiple subscribers, while SQS is a durable queue that lets consumers pull messages at their own pace.

In asynchronous architectures, you commonly use SNS as the event distribution layer and SQS as the buffering layer. This gives you fanout, isolation between consumers, and better fault tolerance when downstream systems are slow or temporarily unavailable.

- **SNS** is best when one event must reach multiple targets.
- **SQS** is best when you need durable buffering and controlled consumption.
- Together, they support event-driven workflows where producers do not wait for consumers.

**Core fanout pattern**

A common pattern is:

1. A producer publishes an event to an SNS topic.
2. SNS delivers a copy of the message to each subscribed SQS queue.
3. Independent consumers poll their own queues and process messages at their own speed.

This pattern is useful when different services need the same event but should not block each other. For example, an order-created event can trigger billing, shipping, analytics, and email workflows without coupling those systems together.

```text
Producer -> SNS topic -> SQS queue A -> Consumer A
                     -> SQS queue B -> Consumer B
                     -> SQS queue C -> Consumer C
```

- Each queue gets its own copy of the message.
- A slow consumer does not affect the others.
- You can scale consumers independently.

**When to choose SNS, SQS, or both**

| Service | Primary role | Typical use |
|---|---|---|
| SNS | Broadcast | Send one event to many subscribers |
| SQS | Buffer | Decouple producer and consumer timing |
| SNS + SQS | Fanout with durability | Distribute events to multiple independent workers |

Use SNS alone when subscribers can process messages directly and you do not need queue semantics. Use SQS alone when a single consumer group needs buffering. Use both when you need pub/sub plus durable, pull-based processing.

**Subscription and delivery basics**

SNS supports multiple subscription protocols, including SQS. When you subscribe an SQS queue to an SNS topic, SNS sends notifications to the queue as messages. The queue then stores the message until a consumer receives and deletes it.

A subscription can be configured with a filter policy so that only matching messages are delivered to a given queue. This is a key fanout technique because it lets you route different event types to different consumers without changing the publisher.

```json
{
  "eventType": ["OrderCreated", "OrderUpdated"],
  "region": ["us-east-1"]
}
```

- Use filtering to reduce unnecessary downstream processing.
- Keep publishers simple by embedding routing attributes in the message or message attributes.
- Prefer message attributes for filterable metadata when the payload should remain focused on business data.

**Raw message delivery**

By default, SNS wraps the original payload in an SNS envelope when delivering to SQS. If you enable raw message delivery, SQS receives the original message body without the SNS metadata wrapper.

This is useful when:
- Your consumer expects the original payload format.
- You want simpler parsing logic.
- You do not need SNS envelope fields such as topic ARN or subscription metadata.

Example subscription configuration with raw delivery:

```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:orders-topic \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:billing-queue \
  --attributes RawMessageDelivery=true
```

If you leave raw delivery disabled, your consumer must parse the SNS envelope and then extract the `Message` field.

**Message filtering for targeted fanout**

SNS message filtering is one of the most exam-relevant features in SNS-to-SQS designs. It lets you send the same event stream to multiple queues while each queue receives only the messages it cares about.

Example use case:
- `billing-queue` receives only `payment` events.
- `shipping-queue` receives only `shipment` events.
- `analytics-queue` receives all events.

```json
{
  "eventType": ["payment"]
}
```

Practical guidance:
- Use message attributes when you want clean routing rules.
- Keep filter policies small and explicit.
- Test filters carefully, because a mismatched policy can silently prevent delivery.

**SQS consumer behavior**

SQS is pull-based, so consumers poll the queue rather than receiving pushed callbacks. In asynchronous workflows, consumers usually run as workers, containers, Lambda functions, or application services that continuously poll for messages.

A consumer flow typically looks like this:

1. Receive a message from SQS.
2. Process the work.
3. Delete the message only after successful processing.

If processing fails and the message is not deleted, SQS makes it visible again after the visibility timeout expires.

**Visibility timeout**

The visibility timeout is the period during which a received message stays hidden from other consumers. It gives the current consumer time to process the message without duplicate concurrent processing.

Why it matters:
- If the timeout is too short, another consumer may receive the message before processing finishes.
- If the timeout is too long, failed messages take longer to retry.
- You should set it longer than your typical processing time, with room for retries or transient delays.

Example:

```bash
aws sqs set-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/billing-queue \
  --attributes VisibilityTimeout=60
```

You can also extend visibility during long-running processing with `ChangeMessageVisibility`.

```bash
aws sqs change-message-visibility \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/billing-queue \
  --receipt-handle AQEB123... \
  --visibility-timeout 120
```

**Long polling**

Long polling reduces empty responses and lowers cost by waiting for messages to arrive before returning from `ReceiveMessage`. It is commonly used in asynchronous consumers because it improves efficiency and reduces unnecessary API calls.

Example:

```bash
aws sqs receive-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/billing-queue \
  --wait-time-seconds 20 \
  --max-number-of-messages 10
```

Use long polling when:
- You want fewer empty receives.
- Your consumer can wait briefly for messages.
- You are building a steady worker process.

**Retries, failures, and dead-letter queues**

SNS and SQS handle failure at different layers.

For SNS-to-SQS delivery:
- SNS attempts to deliver the notification to the subscribed queue.
- Delivery failures are usually caused by permission issues, queue policy problems, KMS encryption issues, or temporary service-side problems.
- SNS retries delivery according to its delivery retry behavior for the subscription protocol, but SQS subscriptions are generally reliable when permissions and encryption are configured correctly.

For message processing:
- Once the message is in SQS, the consumer is responsible for processing it.
- If processing fails repeatedly, SQS redrive policy can move the message to a dead-letter queue (DLQ) after the configured receive count is exceeded.

This distinction is important:
- SNS delivery failure is about getting the message into the queue.
- SQS DLQ behavior is about handling messages that reached the queue but could not be processed successfully.

Example redrive policy:

```json
{
  "deadLetterTargetArn": "arn:aws:sqs:us-east-1:123456789012:billing-dlq",
  "maxReceiveCount": "5"
}
```

Practical guidance:
- Use a DLQ for poison messages that repeatedly fail processing.
- Monitor DLQ depth and age.
- Reprocess DLQ messages only after fixing the underlying bug or dependency issue.

> A DLQ does not replace good retry logic. It is the safety net after retries are exhausted.

**FIFO topics and queues**

If you need strict ordering and exactly-once processing semantics within a message group, use FIFO (First-In, First-Out) SNS topics and SQS queues together.

Key points:
- FIFO SNS topics can publish to FIFO SQS queues.
- Ordering is preserved within a message group.
- Deduplication prevents duplicate processing within the deduplication window.

For SNS FIFO topics:
- You must provide a `MessageGroupId`.
- You may provide a `MessageDeduplicationId`, unless content-based deduplication is enabled on the topic.
- If content-based deduplication is enabled, SNS can derive the deduplication ID from the message body.

Example publish command:

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:orders.fifo \
  --message '{"orderId":"123","status":"CREATED"}' \
  --message-group-id order-123 \
  --message-deduplication-id evt-001
```

Example with content-based deduplication enabled:
- You still provide `MessageGroupId`.
- You do not need to always provide `MessageDeduplicationId` if the topic is configured to compute it from content.

For SQS FIFO queues:
- Use the same message group strategy to preserve ordering.
- Be careful with consumer parallelism, because messages in the same group are processed in order.

**Encryption with KMS**

SNS and SQS support server-side encryption using AWS Key Management Service (KMS). This protects message data at rest.

Important exam point:
- The main permissions concern is the KMS key policy or grants that allow the AWS service to use the key.
- In most cases, the application principal does not need direct KMS permissions just to publish to an encrypted SNS topic or receive from an encrypted SQS queue.
- If you manage the key yourself, ensure the service principal and the relevant AWS service integration are allowed to use the key.

Typical service-side access is controlled through:
- KMS key policy
- Grants
- Service integration permissions

Example SQS queue encryption configuration:

```bash
aws sqs set-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/billing-queue \
  --attributes KmsMasterKeyId=alias/aws/sqs
```

Example SNS topic encryption configuration:

```bash
aws sns set-topic-attributes \
  --topic-arn arn:aws:sns:us-east-1:123456789012:orders-topic \
  --attribute-name KmsMasterKeyId \
  --attribute-value alias/aws/sns
```

Practical guidance:
- Use AWS managed keys when you want simpler setup.
- Use customer managed keys when you need tighter control, auditing, or cross-account governance.
- Check key policy first when encrypted delivery fails.

**IAM and resource policies**

SNS-to-SQS integration requires both IAM and resource-based permissions to line up.

For the SQS queue policy, allow the SNS topic to send messages to the queue. A typical queue policy includes:
- `sqs:SendMessage`
- A condition that restricts access to the specific SNS topic ARN

Example queue policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSNSToSendMessage",
      "Effect": "Allow",
      "Principal": {
        "Service": "sns.amazonaws.com"
      },
      "Action": "sqs:SendMessage",
      "Resource": "arn:aws:sqs:us-east-1:123456789012:billing-queue",
      "Condition": {
        "ArnEquals": {
          "aws:SourceArn": "arn:aws:sns:us-east-1:123456789012:orders-topic"
        }
      }
    }
  ]
}
```

For the publisher:
- The publishing identity needs permission to call `sns:Publish`.
- The consumer identity needs permission to call SQS receive and delete actions such as `sqs:ReceiveMessage`, `sqs:DeleteMessage`, and often `sqs:GetQueueAttributes`.

**Practical topology examples**

A simple order-processing workflow often uses one topic and multiple queues:

- `orders-topic`
  - `billing-queue`
  - `shipping-queue`
  - `analytics-queue`

This design works well when:
- Each downstream system has different processing speed.
- You want to isolate failures.
- You need independent scaling.

A filtered fanout version might route:
- `payment` events to billing
- `shipment` events to logistics
- `customer` events to CRM

```json
{
  "eventType": ["shipment"]
}
```

**Operational considerations**

Monitor both the transport layer and the processing layer.

Useful metrics and signals include:
- SNS delivery failures
- SQS queue depth
- Approximate age of oldest message
- Number of messages received but not deleted
- DLQ message count
- Consumer error rate

Operational best practices:
- Set alarms on queue age, not just queue length.
- Use idempotent consumers because duplicates can happen.
- Keep message payloads small and store large blobs in S3 if needed.
- Include correlation IDs for tracing across services.

**Idempotency and duplicate handling**

Even with retries and deduplication, you should design consumers to be idempotent. That means processing the same message more than once does not create incorrect side effects.

Common techniques:
- Store processed message IDs in a durable store.
- Use conditional writes in DynamoDB.
- Make updates naturally idempotent, such as setting state rather than incrementing blindly.

Example pattern:
- Message contains `eventId`.
- Consumer checks whether `eventId` was already processed.
- If yes, it skips side effects.
- If no, it processes and records completion.

**Common exam traps**

- Confusing SNS push delivery with SQS pull consumption.
- Assuming SQS messages are deleted automatically after receipt.
- Forgetting that visibility timeout controls reappearance after failed processing.
- Treating DLQs as an SNS feature rather than an SQS redrive feature.
- Ignoring queue policies when SNS cannot deliver to SQS.
- Overlooking KMS key policy requirements for encrypted topics and queues.
- Assuming raw message delivery is always enabled by default.
- Forgetting that FIFO ordering is preserved only within a message group.

**Quick configuration checklist**

1. Create the SNS topic.
2. Create the SQS queue or queues.
3. Attach a queue policy that allows the SNS topic to send messages.
4. Subscribe each queue to the topic.
5. Add filter policies if you need selective routing.
6. Enable raw message delivery if your consumer expects the original payload.
7. Configure visibility timeout and long polling on the queue.
8. Add a DLQ and redrive policy for failed processing.
9. Enable encryption if required and verify KMS permissions.
10. Test publish, delivery, consumption, and failure handling end to end.

**Minimal CLI example**

```bash
# Create topic
aws sns create-topic --name orders-topic

# Create queue
aws sqs create-queue --queue-name billing-queue

# Subscribe queue to topic
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:orders-topic \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:billing-queue

# Publish a message
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:orders-topic \
  --message '{"orderId":"123","eventType":"OrderCreated"}'
```

**Key takeaways**

SNS and SQS are often used together to build resilient asynchronous workflows. SNS handles fanout and event distribution, while SQS provides buffering, retry control, and consumer decoupling. For exam purposes, focus on queue policies, filtering, raw delivery, visibility timeout, DLQs, FIFO behavior, and encryption permissions.
