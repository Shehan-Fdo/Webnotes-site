## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Retry, failure handling, and DLQ fundamentals**

Resilient AWS solutions assume that transient failures happen. Network blips, throttling, downstream outages, and malformed payloads all require different responses. For the DVA-C02 exam, you should know when to retry, when to stop, and when to isolate failed work in a dead-letter queue (<abbr title="Dead-Letter Queue">DLQ</abbr>).

A good design separates:
- **Transient failures**, which may succeed on a later attempt.
- **Permanent failures**, which should be quarantined or surfaced for manual review.
- **Poison messages**, which repeatedly fail and can block progress in queues or workflows.

**Retry strategy design**

Retries help with temporary issues, but uncontrolled retries can amplify load and increase cost. Use bounded retries with backoff and jitter so many clients do not retry at the same time.

Common retry patterns:
- **Immediate retry** for very short-lived failures, such as a brief network timeout.
- **Exponential backoff** to increase wait time between attempts.
- **Jitter** to randomize retry timing and reduce retry storms.
- **Circuit breaking** to stop calling a failing dependency until it recovers.

A practical exponential backoff pattern in application code looks like this:

```python
import random
import time

def retry(operation, attempts=5, base_delay=0.2):
    for attempt in range(1, attempts + 1):
        try:
            return operation()
        except Exception as e:
            if attempt == attempts:
                raise
            delay = base_delay * (2 ** (attempt - 1))
            delay = delay + random.uniform(0, delay * 0.2)
            time.sleep(delay)
```

Use retries when:
- The failure is likely temporary.
- The operation is safe to repeat or is idempotent.
- The downstream service documents retryable errors.

Avoid aggressive retries when:
- The request is malformed.
- The operation is not idempotent and cannot tolerate duplicates.
- The dependency is already under heavy load.

**Idempotency and duplicate protection**

Retries can create duplicate side effects unless your code is idempotent. <abbr title="Application Programming Interface">API</abbr> calls, event handlers, and queue consumers should be designed so repeated processing does not corrupt state.

Techniques include:
- **Idempotency keys** to deduplicate repeated requests.
- **Conditional writes** in DynamoDB using expressions such as `attribute_not_exists(...)`.
- **State checks** before performing an action.
- **Deduplication windows** where supported by the service.

Example DynamoDB conditional write:

```python
import boto3

ddb = boto3.client("dynamodb")

ddb.put_item(
    TableName="Orders",
    Item={
        "orderId": {"S": "123"},
        "status": {"S": "CREATED"}
    },
    ConditionExpression="attribute_not_exists(orderId)"
)
```

This prevents a retry from creating the same order twice if the first write already succeeded.

**AWS SDK retries**

Many AWS SDKs include built-in retry logic. You should know that SDK retries are usually the first line of defense for transient AWS service errors such as throttling or short network failures.

Typical retryable conditions include:
- Throttling responses.
- Transient 5xx service errors.
- Connection timeouts.
- Temporary DNS or transport issues.

For the AWS CLI and SDKs, retry behavior can often be tuned through configuration. For example, the AWS CLI supports retry mode settings:

```bash
aws configure set retry_mode standard
aws configure set max_attempts 5
```

Retry modes vary by SDK, but the general idea is the same:
- **Standard** mode balances resilience and latency.
- **Adaptive** mode can reduce pressure during throttling.
- **Legacy** mode may exist in some tools but is less preferred.

**SQS retry behavior and visibility timeout**

<abbr title="Amazon Simple Queue Service">Amazon SQS</abbr> uses message visibility timeout to control retries. When a consumer receives a message, the message becomes invisible for the visibility timeout period. If the consumer does not delete the message before the timeout expires, SQS makes it visible again for another attempt.

Key points:
- **Visibility timeout** is not a retry counter; it is a lock period.
- A message can be processed more than once if the consumer fails to delete it.
- You should set the visibility timeout longer than the expected processing time.
- You can extend visibility during long-running processing with `ChangeMessageVisibility`.

Example:

```bash
aws sqs receive-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders \
  --max-number-of-messages 1 \
  --wait-time-seconds 20
```

If processing may take longer than the initial timeout, extend it:

```bash
aws sqs change-message-visibility \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders \
  --receipt-handle AQEB... \
  --visibility-timeout 120
```

Use a DLQ when:
- A message repeatedly fails processing.
- You want to isolate poison messages.
- You need to inspect failed payloads later.

**Dead-letter queues in SQS**

A DLQ is a separate queue that receives messages after they exceed the source queue’s `maxReceiveCount`. This prevents a bad message from blocking the rest of the queue indefinitely.

Important SQS rules:
- A **standard queue** can use only a **standard DLQ**.
- A **FIFO queue** can use only a **FIFO DLQ**.
- The source queue and DLQ must be in the same AWS account and Region.
- The DLQ should usually have a longer retention period than the source queue.

For standard queues:
- Message order is best effort, so moving a message to a DLQ does not preserve ordering guarantees.
- The original enqueue timestamp is preserved when the message moves to the DLQ.
- Retention is based on the original enqueue time in the source queue.

For FIFO queues:
- Use a FIFO DLQ to preserve FIFO semantics.
- The original enqueue timestamp is also preserved when the message moves to the DLQ.
- Retention is still based on the time the message was first enqueued in the source queue, not the time it entered the DLQ.

Example redrive policy:

```json
{
  "deadLetterTargetArn": "arn:aws:sqs:us-east-1:123456789012:orders-dlq",
  "maxReceiveCount": 5
}
```

Example queue creation with a DLQ:

```bash
aws sqs create-queue --queue-name orders-dlq.fifo \
  --attributes FifoQueue=true,ContentBasedDeduplication=true

aws sqs create-queue --queue-name orders.fifo \
  --attributes FifoQueue=true,ContentBasedDeduplication=true,RedrivePolicy='{"deadLetterTargetArn":"arn:aws:sqs:us-east-1:123456789012:orders-dlq.fifo","maxReceiveCount":"5"}'
```

Operationally, DLQs help you:
- Inspect malformed messages.
- Reprocess messages after fixing code or data.
- Separate application bugs from transient infrastructure issues.

**SNS subscription redrive and failure handling**

<abbr title="Amazon Simple Notification Service">Amazon SNS</abbr> supports subscription-level DLQs through a `RedrivePolicy`. This is useful when delivery to a subscription target fails after retries.

Supported subscription types for DLQs include:
- **SQS**
- **Lambda**
- **HTTP**
- **HTTPS**

Not supported:
- Email
- SMS

SNS retries delivery based on the subscription protocol and target behavior. If delivery continues to fail, SNS can send the undelivered notification to the configured DLQ.

Example subscription with a redrive policy:

```json
{
  "deadLetterTargetArn": "arn:aws:sqs:us-east-1:123456789012:sns-subscription-dlq"
}
```

Example CLI flow:

```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:orders \
  --protocol https \
  --notification-endpoint https://example.com/sns

aws sns set-subscription-attributes \
  --subscription-arn arn:aws:sns:us-east-1:123456789012:... \
  --attribute-name RedrivePolicy \
  --attribute-value '{"deadLetterTargetArn":"arn:aws:sqs:us-east-1:123456789012:sns-subscription-dlq"}'
```

Use SNS DLQs when:
- A subscriber endpoint is intermittently unavailable.
- You need to preserve failed notifications for later analysis.
- You want to avoid losing messages after repeated delivery failures.

**Lambda asynchronous invocation retries and destinations**

<abbr title="AWS Lambda">AWS Lambda</abbr> handles failures differently for synchronous and asynchronous invocations.

For **synchronous** invocations:
- The caller receives the error directly.
- Your client or upstream service must decide whether to retry.

For **asynchronous** invocations:
- Lambda queues the event and retries automatically on failure.
- By default, Lambda retries asynchronous invocations twice after the initial attempt.
- You can configure maximum event age and retry attempts.

Example configuration:

```bash
aws lambda put-function-event-invoke-config \
  --function-name process-order \
  --maximum-retry-attempts 2 \
  --maximum-event-age-in-seconds 3600
```

Lambda asynchronous invocation can also use **destinations** to route the result of a successful or failed invocation. Destinations can target:
- SQS
- SNS
- Lambda
- EventBridge

Use destinations when you want to:
- Capture success or failure outcomes.
- Trigger follow-up workflows.
- Avoid writing custom failure-handling code for every event source.

A failure destination is often more flexible than a DLQ because it can carry richer context about the invocation result. A DLQ is simpler and is focused on preserving the original event payload.

> Lambda destinations are for asynchronous invocation outcomes. They are not the same as source-event DLQs for services like SQS or SNS.

**Step Functions retry and catch behavior**

<abbr title="AWS Step Functions">AWS Step Functions</abbr> lets you define retry and catch logic directly in the state machine. This is useful when you want workflow-level control over transient and terminal failures.

Use `Retry` to handle temporary errors and `Catch` to route failures to alternate states.

A current best-practice Lambda task uses the Lambda service integration:

```json
{
  "Comment": "Order processing workflow",
  "StartAt": "InvokeWorker",
  "States": {
    "InvokeWorker": {
      "Type": "Task",
      "Resource": "arn:aws:states:::lambda:invoke",
      "Parameters": {
        "FunctionName": "arn:aws:lambda:us-east-1:123456789012:function:process-order",
        "Payload.$": "$"
      },
      "Retry": [
        {
          "ErrorEquals": [
            "Lambda.ServiceException",
            "Lambda.AWSLambdaException",
            "Lambda.SdkClientException",
            "States.Timeout"
          ],
          "IntervalSeconds": 2,
          "MaxAttempts": 3,
          "BackoffRate": 2.0
        }
      ],
      "Catch": [
        {
          "ErrorEquals": [
            "States.ALL"
          ],
          "ResultPath": "$.error",
          "Next": "SendToFailureHandler"
        }
      ],
      "End": true
    },
    "SendToFailureHandler": {
      "Type": "Pass",
      "End": true
    }
  }
}
```

Important Step Functions notes:
- `States.ALL` is the broad catch-all error matcher.
- `States.Timeout` is a common built-in error for timeouts.
- `Lambda.ServiceException`, `Lambda.AWSLambdaException`, and `Lambda.SdkClientException` are common Lambda integration errors.
- `Retry` applies before `Catch` when both are present.
- `Catch` can route the workflow to a compensating action, alerting step, or failure queue.

Other built-in Step Functions errors you should recognize:
- `States.TaskFailed`
- `States.HeartbeatTimeout`
- `States.ResultPathMatchFailure`
- `States.ParameterPathFailure`

Use Step Functions retries when:
- The failure is transient and workflow-local.
- You want centralized retry policy instead of custom code.
- You need different handling for different error classes.

**Choosing between retries, DLQs, and catches**

These mechanisms solve different problems.

| Mechanism | Best for | Typical scope | Outcome |
|---|---|---|---|
| Retry | Transient failures | Client, SDK, workflow, queue consumer | Another attempt |
| DLQ | Poison messages and repeated failures | Queue or subscription | Isolated failed payload |
| Catch | Workflow branching on failure | Step Functions | Alternate state path |
| Destination | Async invocation outcome handling | Lambda async | Routed success/failure event |

A practical rule:
- Retry first for transient issues.
- Use DLQs when repeated failures need quarantine.
- Use Catch when a workflow can recover or compensate.
- Use destinations when you need post-invocation routing for Lambda async events.

**Operational patterns and troubleshooting**

When a message or invocation fails repeatedly, inspect the failure mode before changing retry counts.

Checklist:
- Confirm whether the error is transient or permanent.
- Verify idempotency before increasing retries.
- Check whether the visibility timeout is too short.
- Review DLQ depth and age metrics.
- Inspect logs for payload validation errors.
- Confirm IAM permissions for the caller, Lambda function, or Step Functions state machine.

Useful CloudWatch metrics and signals:
- **SQS**: `ApproximateNumberOfMessagesVisible`, `ApproximateAgeOfOldestMessage`
- **Lambda**: `Errors`, `Throttles`, `DeadLetterErrors`, `DestinationDeliveryFailures`
- **SNS**: delivery failure metrics and subscription logs where enabled
- **Step Functions**: execution failures, retries, and state-level error history

Example CloudWatch Logs insight query for Lambda failures:

```sql
fields @timestamp, @message
| filter @message like /ERROR|Exception|Task timed out/
| sort @timestamp desc
| limit 20
```

**Exam-focused distinctions**

Know these distinctions for DVA-C02:
- **SQS redrive policy** moves messages to a DLQ after `maxReceiveCount`.
- **Visibility timeout** controls when an unacknowledged message becomes visible again.
- **SNS subscription DLQs** are configured with `RedrivePolicy` and support SQS, Lambda, HTTP, and HTTPS subscriptions.
- **Lambda async retries** happen automatically unless you configure event invoke settings.
- **Step Functions** uses `Retry` and `Catch` with built-in error names such as `States.ALL` and `States.Timeout`.
- **Idempotency** is essential whenever retries can cause duplicate side effects.

**Practical design examples**

A common resilient architecture looks like this:
- API Gateway receives a request.
- Lambda validates and writes a record idempotently.
- Lambda publishes an event to SNS or SQS.
- A consumer processes the event with bounded retries.
- Failed messages move to a DLQ for later inspection.
- A Step Functions workflow handles compensating actions if a business process fails.

Another pattern for long-running work:
- SQS buffers requests.
- Lambda polls the queue.
- Visibility timeout is set longer than average processing time.
- The consumer extends visibility for long jobs.
- Poison messages go to a DLQ after repeated failures.

These patterns reduce data loss, prevent infinite retry loops, and make failures observable.
