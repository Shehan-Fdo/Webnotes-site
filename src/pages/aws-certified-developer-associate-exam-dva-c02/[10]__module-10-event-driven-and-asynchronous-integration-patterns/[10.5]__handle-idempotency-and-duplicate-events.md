## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Idempotency fundamentals**

Idempotency means that repeating the same request produces the same intended outcome without creating extra side effects. In AWS application design, this matters because many services deliver messages with **at-least-once** semantics, and clients or downstream systems may retry after timeouts, throttling, or transient failures.

A practical idempotent design lets you safely handle:
- Client retries after network timeouts
- Lambda retries from asynchronous invocations
- Duplicate messages from Amazon SQS or Amazon SNS fanout paths
- Replayed events from event sources or manual reprocessing

The goal is not to prevent every duplicate from arriving. The goal is to make duplicates harmless.

**Why duplicates happen**

Duplicates are normal in distributed systems. You should expect them when:
- A producer does not know whether the consumer completed work
- A consumer times out after performing the side effect
- A queue redelivers a message after visibility timeout expires
- An event source retries delivery after an error
- A batch partially succeeds and is retried

In AWS, this is especially important for serverless and event-driven architectures because the platform favors reliability over exactly-once delivery.

**Core design patterns**

Use one or more of these patterns depending on the workflow:

- **Client-generated idempotency key**
  - The caller sends a unique token with the request.
  - The service stores the token and the result of the first successful processing.
  - Repeated requests with the same token return the same result.

- **Natural business key**
  - Use a domain identifier such as `orderId`, `paymentId`, or `shipmentId`.
  - This works when the business object itself is unique and stable.

- **Deduplication store**
  - Persist processed message IDs in DynamoDB, Amazon Aurora, or another durable store.
  - Check before processing, then write a completion marker after success.
  - Use a conditional write to avoid race conditions.

- **State transition guard**
  - Only allow valid transitions, such as `PENDING -> PAID -> SHIPPED`.
  - Ignore repeated events that would not change state.

- **Inbox/outbox pattern**
  - Store incoming event IDs in an inbox table to prevent reprocessing.
  - Store outgoing events in an outbox table so side effects are published reliably after the transaction.

**DynamoDB as an idempotency store**

DynamoDB is a common choice because it supports low-latency reads and conditional writes. A typical approach is to create a table keyed by an idempotency token or message ID.

Example table item:
```json
{
  "idempotencyKey": "abc-123",
  "status": "COMPLETED",
  "responsePayload": {
    "orderId": "o-789",
    "result": "accepted"
  },
  "expiresAt": 1714147200
}
```

A common flow is:
1. Receive request or event.
2. Attempt a conditional put with `attribute_not_exists(idempotencyKey)`.
3. If the write succeeds, process the business action.
4. Store the final result and status.
5. If the key already exists, return the stored result or skip processing.

Example AWS CLI conditional write:
```bash
aws dynamodb put-item \
  --table-name IdempotencyKeys \
  --item '{
    "idempotencyKey": {"S": "abc-123"},
    "status": {"S": "IN_PROGRESS"},
    "expiresAt": {"N": "1714147200"}
  }' \
  --condition-expression "attribute_not_exists(idempotencyKey)"
```

This pattern prevents two workers from processing the same logical request at the same time.

**Lambda and retries**

AWS Lambda may invoke your function more than once for the same event, depending on the trigger type and failure mode.

- **Asynchronous invocations**
  - Lambda retries automatically on failure.
  - You must design handlers to tolerate duplicate execution.

- **SQS event source mappings**
  - Messages can be delivered more than once.
  - If the function fails, the message becomes visible again after the visibility timeout.

- **Kinesis and DynamoDB Streams**
  - Records are processed in order per shard or partition key, but retries can still occur.
  - A failed batch may be retried, so your handler should avoid repeating side effects.

A safe Lambda handler usually:
- Extracts a stable event identifier
- Checks whether the event was already processed
- Performs the side effect only once
- Records success before returning

Example Node.js-style pseudocode:
```javascript
export const handler = async (event) => {
  const key = event.idempotencyKey || event.messageId;

  const acquired = await putIfAbsent(key, "IN_PROGRESS");
  if (!acquired) {
    return { statusCode: 200, body: "Duplicate ignored" };
  }

  try {
    const result = await processOrder(event);
    await markComplete(key, result);
    return result;
  } catch (err) {
    await markFailed(key, err.message);
    throw err;
  }
};
```

**SQS duplicate handling**

Amazon SQS is designed for reliable delivery, not exactly-once processing. Standard queues can deliver duplicates, and even FIFO queues only provide deduplication within a limited window.

Important SQS behaviors:
- **Standard queues**
  - At-least-once delivery
  - Best-effort ordering
  - Duplicate messages are possible

- **FIFO queues**
  - Exactly-once processing is not guaranteed end-to-end
  - Deduplication is based on a deduplication ID within the deduplication interval
  - Ordering is preserved per message group

For FIFO queues, use:
- `MessageDeduplicationId` to reduce duplicate sends
- `MessageGroupId` to preserve ordering for related messages

Example send command:
```bash
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders.fifo \
  --message-body '{"orderId":"o-789","action":"create"}' \
  --message-group-id "order-o-789" \
  --message-deduplication-id "create-o-789"
```

Even with FIFO, your consumer should still be idempotent because retries and downstream failures can still cause repeated processing.

**SNS, EventBridge, and fanout**

Amazon SNS and Amazon EventBridge are common event sources in decoupled architectures. They improve scalability, but they also increase the chance that the same logical event reaches multiple consumers or is retried.

Use these practices:
- Include a stable event ID in the payload
- Treat the event as immutable
- Make each consumer idempotent independently
- Do not assume one consumer’s success means another consumer succeeded

A fanout topology often looks like this:
- Producer publishes one event
- SNS or EventBridge distributes it to multiple targets
- Each target maintains its own deduplication state
- Each target can retry without affecting the others

**Database constraints as protection**

Sometimes the simplest idempotency control is a database constraint.

Examples:
- Unique constraint on `orderId`
- Conditional update on a status field
- Transactional insert that fails if the record already exists

This is useful when the business rule itself defines uniqueness. For example, a payment record should not be created twice for the same external transaction reference.

Example SQL pattern:
```sql
INSERT INTO payments (payment_id, order_id, amount, status)
VALUES ('pay-123', 'o-789', 49.99, 'PENDING')
ON CONFLICT (payment_id) DO NOTHING;
```

If the insert is ignored, you know the request was already handled or is in progress.

**Transactional outbox and inbox**

The **outbox pattern** helps when you must update a database and publish an event without losing consistency between them.

Typical flow:
1. Write the business record and an outbox record in the same transaction.
2. A separate publisher reads the outbox table.
3. The publisher sends the event to SNS, EventBridge, or SQS.
4. Mark the outbox row as sent.

The **inbox pattern** helps consumers avoid duplicate processing:
1. Receive event.
2. Check whether the event ID exists in the inbox table.
3. If not, insert the event ID and process it.
4. If yes, skip processing.

These patterns are especially useful when you need stronger operational guarantees than a simple in-memory cache can provide.

**Visibility timeout and partial failure**

For SQS consumers, the visibility timeout is a major source of duplicates if it is too short. If processing takes longer than the timeout, the message can reappear and be processed again.

You should:
- Set the visibility timeout longer than the expected processing time
- Extend visibility when processing long-running tasks
- Make processing idempotent anyway
- Use a dead-letter queue (DLQ) for poison messages

A DLQ does not solve duplicates by itself, but it helps isolate messages that repeatedly fail.

**Practical implementation checklist**

1. Choose a stable idempotency key.
   - Prefer a business identifier when available.
   - Otherwise use a client-generated token or message ID.

2. Persist processing state durably.
   - Use DynamoDB, Aurora, or another durable store.
   - Avoid relying only on memory or local files.

3. Guard the first write.
   - Use conditional writes or unique constraints.
   - Prevent two workers from claiming the same request.

4. Store the result when possible.
   - Return the same response for repeated requests.
   - This improves client behavior and reduces ambiguity.

5. Make side effects repeat-safe.
   - Avoid double-charging, double-creating, or double-sending.
   - Check current state before mutating.

6. Tune retry and timeout settings.
   - Align Lambda timeout, SQS visibility timeout, and downstream timeouts.
   - Reduce unnecessary retries caused by premature timeouts.

**Example architecture**

A common order-processing flow might be:
- API Gateway receives `POST /orders`
- Lambda generates or receives an idempotency key
- Lambda writes a record to DynamoDB with a conditional put
- Lambda publishes an event to EventBridge
- A fulfillment Lambda consumes the event
- Fulfillment Lambda checks its inbox table before reserving inventory
- A payment service uses the same order ID to prevent duplicate charges

This design allows retries at each layer without creating duplicate orders.

**Common pitfalls**

- **Using only memory**
  - A Lambda execution environment can be recycled at any time.
  - In-memory deduplication is not durable.

- **Assuming FIFO means exactly once**
  - FIFO reduces duplicates in a limited window.
  - It does not remove the need for idempotent consumers.

- **Writing side effects before recording the key**
  - If the process crashes after the side effect but before the record, the duplicate will repeat the action.

- **Using random IDs for every retry**
  - A new ID makes each retry look like a new request.
  - Reuse the same idempotency token for the same logical operation.

- **Ignoring downstream retries**
  - Even if your handler is idempotent, a downstream API call may not be.
  - Apply the same pattern at each boundary.

**Exam focus**

For the AWS Certified Developer – Associate exam, expect questions that ask you to:
- Prevent duplicate processing in Lambda consumers
- Choose between SQS standard and FIFO behavior
- Use DynamoDB conditional writes for deduplication
- Design retry-safe event-driven workflows
- Handle at-least-once delivery from AWS services

When you see a scenario with retries, timeouts, or duplicate messages, think:
- What is the stable identifier?
- Where do I store the deduplication state?
- How do I prevent the side effect from happening twice?

**Quick reference**

- **Idempotency**: repeated execution has the same effect as one execution
- **At-least-once delivery**: messages may arrive more than once
- **Deduplication**: detecting and ignoring repeats
- **Conditional write**: atomic insert/update based on a condition
- **Inbox/outbox**: durable patterns for safe event consumption and publication
