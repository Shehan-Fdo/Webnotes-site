## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**AWS CDK workflow overview**

The AWS Cloud Development Kit (AWS CDK) turns code into AWS CloudFormation templates. The core lifecycle is `synth`, `diff`, `deploy`, and `destroy`. You use these commands to preview infrastructure, validate changes, release stacks, and clean up resources.

This workflow matters because it gives you a repeatable path from local code to cloud resources. In practice, you should treat infrastructure changes like application releases: review the generated template, compare it to what is already deployed, deploy intentionally, and destroy only when you are sure resources are no longer needed.

- **`synth`** generates the CloudFormation template and asset manifests.
- **`diff`** compares the synthesized template with the deployed stack.
- **`deploy`** creates or updates the stack in AWS.
- **`destroy`** removes the stack and its managed resources.

**CDK app structure and bootstrap context**

A CDK app is usually written in TypeScript, JavaScript, Python, Java, or C#. The app defines one or more stacks, and each stack maps to a CloudFormation stack. Before deployment, the target AWS environment often needs to be bootstrapped so CDK can store deployment assets such as Lambda code, Docker images, and file assets.

A typical project layout looks like this:

```text
my-cdk-app/
├── bin/
│   └── app.ts
├── lib/
│   └── my-stack.ts
├── cdk.json
├── package.json
└── tsconfig.json
```

A simple stack definition in TypeScript might look like this:

```typescript
import * as cdk from 'aws-cdk-lib';
import { Construct } from 'constructs';
import * as s3 from 'aws-cdk-lib/aws-s3';

export class MyStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    new s3.Bucket(this, 'AppBucket', {
      versioned: true,
      encryption: s3.BucketEncryption.S3_MANAGED,
    });
  }
}
```

**`synth`: generate the CloudFormation output**

`cdk synth` runs the app and produces the CloudFormation template that AWS will use to create or update resources. It also resolves context values, applies constructs, and packages assets.

Use `synth` when you want to inspect the final infrastructure definition before deployment. This is the best place to catch logical errors, unexpected defaults, or missing permissions in the generated template.

Common uses:

- Validate that the stack compiles and the app can resolve context.
- Inspect the generated CloudFormation YAML or JSON.
- Confirm that assets are staged correctly.
- Produce output for review in CI/CD pipelines.

Example command:

```bash
cdk synth
```

You can synthesize a specific stack:

```bash
cdk synth MyStack
```

Useful output patterns:

```bash
cdk.out/
├── MyStack.template.json
├── manifest.json
└── asset.<hash>/
```

Key points:

- **Context resolution** happens during synthesis, including lookups such as VPCs, AMIs, or availability zones.
- **Assets** are prepared for later upload, but not necessarily deployed yet.
- **CloudFormation** is the final target format, even though you write higher-level constructs.

> If synthesis fails, deployment cannot proceed. Fix code errors, missing context, or invalid construct properties before moving on.

**`diff`: compare desired and deployed state**

`cdk diff` compares the synthesized template with the currently deployed CloudFormation stack. It shows what will change if you deploy now. This is one of the most important safety checks in the CDK workflow.

Use `diff` when you want to understand the impact of a change before applying it. It helps you spot resource replacements, deletions, and property updates that may cause downtime.

Example command:

```bash
cdk diff
```

Compare a specific stack:

```bash
cdk diff MyStack
```

What to look for:

- **Additions**: new resources that will be created.
- **Modifications**: property changes on existing resources.
- **Replacements**: changes that force a resource to be recreated.
- **Deletions**: resources removed from the template.

A diff may show output similar to this:

```text
Stack MyStack
Resources
[~] AWS::S3::Bucket AppBucket AppBucket1234
 └─ [~] VersioningConfiguration
     └─ [~] Status
         ├─ [-] Suspended
         └─ [+] Enabled
```

Practical interpretation:

- A bucket versioning change is usually safe, but you still verify whether the update is in-place.
- A replacement on stateful resources such as databases, queues, or buckets can be risky.
- A deletion in the diff may mean data loss if the resource is not retained.

You can also use diff in automation to gate deployments. For example, a pipeline can fail if the diff contains unexpected replacements or deletions.

**`deploy`: create or update the stack**

`cdk deploy` uploads assets, creates the CloudFormation change set, and executes it. This is the command that actually provisions or updates AWS resources.

Use `deploy` after you have reviewed the synth output and diff. In development, you may deploy frequently. In production, you should deploy through controlled pipelines and approvals.

Example command:

```bash
cdk deploy
```

Deploy a specific stack:

```bash
cdk deploy MyStack
```

Deploy with approval prompts for security-sensitive changes:

```bash
cdk deploy --require-approval broadening
```

Common deployment behaviors:

- **Asset publishing** uploads Lambda bundles, container images, and other files to bootstrap resources.
- **Change sets** let CloudFormation preview the update before execution.
- **Rollback** occurs automatically if the stack update fails, unless you explicitly change rollback behavior in a controlled workflow.

A typical deployment flow is:

1. Run `cdk synth` to validate the generated template.
2. Run `cdk diff` to review the changes.
3. Run `cdk deploy` to apply the update.
4. Verify the deployed resources and application behavior.

Example with multiple stacks:

```bash
cdk deploy NetworkStack AppStack
```

Practical deployment considerations:

- **Bootstrap** the environment first if assets are involved.
- **Permissions** must allow CloudFormation, IAM, S3, ECR, and other services used by the stack.
- **Environment targeting** matters when your app uses explicit account and region values.

Example environment binding in TypeScript:

```typescript
new MyStack(app, 'MyStack', {
  env: {
    account: process.env.CDK_DEFAULT_ACCOUNT,
    region: process.env.CDK_DEFAULT_REGION,
  },
});
```

**`destroy`: remove the stack and clean up resources**

`cdk destroy` deletes the CloudFormation stack and the resources it manages. Use it for temporary environments, test stacks, and cleanup after experiments.

Example command:

```bash
cdk destroy
```

Destroy a specific stack:

```bash
cdk destroy MyStack
```

Important behavior:

- CloudFormation deletes resources in dependency order.
- Some resources may fail to delete if they are not empty or if deletion protection is enabled.
- Retained resources are not removed if the template or resource policy marks them for retention.

Examples of common cleanup blockers:

- **S3 buckets** with objects still inside.
- **RDS** databases with deletion protection enabled.
- **Security groups** or network resources still referenced by other stacks.
- **IAM** resources with dependencies or service-linked constraints.

If a bucket must be removed, empty it first:

```bash
aws s3 rm s3://my-bucket-name --recursive
cdk destroy MyStack
```

> Destroy is irreversible for the stack resources that are actually deleted. Always confirm that you do not need the data, logs, or backups before running it.

**Bootstrap and asset publishing**

Many CDK deployments rely on a bootstrap stack in the target account and region. The bootstrap stack provides the staging resources needed for file assets, Docker assets, and deployment roles.

Typical bootstrap command:

```bash
cdk bootstrap aws://123456789012/us-east-1
```

Why bootstrap matters:

- **File assets** such as Lambda zip files need an S3 staging bucket.
- **Docker assets** need an ECR repository for image publishing.
- **Deployment roles** help CDK and CloudFormation perform actions securely.

If you see asset publishing failures, check:

- The target account and region.
- Whether the bootstrap stack exists.
- Whether your credentials can assume the required roles.
- Whether the asset path or Docker build context is valid.

**Context lookups and reproducibility**

CDK can perform environment lookups during synthesis, such as fetching VPC IDs, subnet IDs, or availability zones. These lookups are stored in `cdk.context.json` to make future syntheses more stable.

Example lookup:

```typescript
const vpc = ec2.Vpc.fromLookup(this, 'DefaultVpc', {
  isDefault: true,
});
```

Why this matters:

- **Reproducibility** improves when context is committed and managed carefully.
- **Stale context** can cause diffs that do not reflect the current environment.
- **Environment-specific values** can make a stack behave differently across accounts or regions.

If the environment changes and the lookup becomes stale, clear or refresh context:

```bash
cdk context --clear
cdk synth
```

**Practical command sequence**

A safe local workflow usually follows this pattern:

```bash
npm install
cdk bootstrap aws://123456789012/us-east-1
cdk synth
cdk diff
cdk deploy
```

For teardown:

```bash
cdk destroy
```

If you are working with multiple stacks, target them explicitly:

```bash
cdk synth NetworkStack
cdk diff NetworkStack AppStack
cdk deploy NetworkStack AppStack
cdk destroy AppStack
```

**Deployment safety and change management**

The CDK workflow is not just about commands; it is about controlling risk. You should review changes before deployment, especially when a change can replace stateful resources or broaden permissions.

Watch for these high-risk changes:

- **IAM policy** updates that grant broader access.
- **Resource replacement** on databases, buckets, or queues.
- **Security group** changes that open new network paths.
- **Deletion** of resources that hold application state.

Helpful practices:

- Use separate stacks for networking, data, and application layers.
- Keep stateful resources isolated from frequently changing application code.
- Review diffs in pull requests or CI/CD jobs.
- Use approval prompts for sensitive changes.
- Tag resources consistently for ownership and cost tracking.

**Troubleshooting common issues**

If `synth` fails:

- Check TypeScript or language runtime errors.
- Verify construct properties and required parameters.
- Confirm that context lookups can resolve in the current environment.

If `diff` shows unexpected changes:

- Inspect `cdk.context.json` for stale values.
- Check whether a construct ID changed, which can cause replacement.
- Confirm that defaults or feature flags changed between CDK versions.

If `deploy` fails:

- Review CloudFormation events in the AWS console.
- Check IAM permissions for the deploying principal.
- Verify that bootstrap resources exist and are accessible.
- Look for service limits, naming conflicts, or invalid resource properties.

If `destroy` fails:

- Empty S3 buckets and remove dependent objects.
- Disable deletion protection only if you are certain.
- Check for retained resources or cross-stack references.

**Exam focus**

For exam purposes, understand the purpose of each command and the order in which you use them.

- **`synth`** produces the CloudFormation template from CDK code.
- **`diff`** shows the delta between the synthesized template and the deployed stack.
- **`deploy`** applies the stack to AWS through CloudFormation.
- **`destroy`** deletes the stack and managed resources.
- **Bootstrap** is often required before deploying assets.
- **Context** values can affect synthesis and reproducibility.

A good mental model is:

- Write infrastructure as code.
- Synthesize to see the final template.
- Diff to review impact.
- Deploy to apply changes.
- Destroy to clean up when finished.
