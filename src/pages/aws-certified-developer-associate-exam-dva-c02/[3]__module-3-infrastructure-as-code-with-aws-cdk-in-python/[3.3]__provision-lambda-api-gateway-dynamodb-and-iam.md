## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Provisioning overview**

This topic ties together the common serverless application stack: **AWS Lambda** for compute, **Amazon API Gateway** for HTTP front doors, **Amazon DynamoDB** for persistence, and **AWS Identity and Access Management (IAM)** for authorization. On the exam, you are often asked to choose the correct combination of permissions, deployment artifacts, and service settings rather than just identify a service by name.

A good mental model is:

- API Gateway receives the request and handles API concerns.
- Lambda runs the business logic.
- DynamoDB stores application state.
- IAM controls who can invoke what and what each component can access.

**Lambda provisioning and packaging**

Lambda is an event-driven compute service. You package code, configure runtime settings, attach an execution role, and then connect triggers or invoke it directly. For exam scenarios, pay attention to how the function is deployed and how it is updated over time.

Common provisioning choices include:

- **Zip package** deployment for most application code.
- **Container image** deployment when you need custom OS libraries or a larger dependency footprint.
- **Lambda layers** for shared dependencies, custom runtimes, or reusable libraries across multiple functions.

A typical zip-based deployment flow looks like this:

```bash
zip -r function.zip app.py requirements.txt
aws lambda create-function \
  --function-name orders-handler \
  --runtime python3.12 \
  --handler app.lambda_handler \
  --role arn:aws:iam::123456789012:role/lambda-exec-role \
  --zip-file fileb://function.zip
```

For updates, you usually publish a new version and shift traffic with an alias:

```bash
aws lambda update-function-code \
  --function-name orders-handler \
  --zip-file fileb://function.zip

aws lambda publish-version \
  --function-name orders-handler

aws lambda create-alias \
  --function-name orders-handler \
  --name prod \
  --function-version 12
```

Important exam points:

- **Versions** are immutable snapshots of code and configuration.
- **Aliases** are mutable pointers to versions and are commonly used for `dev`, `test`, and `prod`.
- Environment-specific configuration often lives in **environment variables**, while traffic routing and release control often use aliases.
- If API Gateway invokes a Lambda alias, the invoke permission must cover the **alias-qualified ARN** or otherwise explicitly allow that alias target.

> A permission on the unqualified function ARN does not automatically guarantee invoke access to every alias in every setup. Be precise about the ARN used in the permission statement.

**Lambda execution roles and IAM basics**

Lambda uses an **execution role** to call AWS services on your behalf. The role has two distinct IAM parts:

- A **trust policy** that allows the Lambda service principal to assume the role.
- A **permissions policy** that grants the role access to resources such as DynamoDB, CloudWatch Logs, or KMS.

Example trust policy for a Lambda execution role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example permissions policy for logging and DynamoDB access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:PutItem",
        "dynamodb:UpdateItem",
        "dynamodb:Query"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders"
    }
  ]
}
```

Use least privilege:

- Scope DynamoDB permissions to the table or index ARN when possible.
- Add `kms:Decrypt` only when the function reads encrypted data that requires it.
- Avoid broad `*` permissions unless the question explicitly calls for a quick prototype.

**Lambda invocation patterns and event sources**

Lambda can be invoked in different ways, and the invocation model affects retries and error handling.

- **Synchronous invocation**: the caller waits for a response, such as API Gateway calling Lambda.
- **Asynchronous invocation**: the service queues the event and Lambda processes it later, such as some event sources and direct async calls.
- **Event source mapping**: Lambda polls supported sources like Amazon SQS, Amazon Kinesis Data Streams, and DynamoDB Streams.

Practical implications:

- Synchronous callers receive immediate success or failure.
- Asynchronous invocations can retry automatically and may send failed events to a dead-letter queue or destination, depending on configuration.
- Poll-based sources use batch processing and retry behavior tied to the source mapping.

Example event source mapping:

```bash
aws lambda create-event-source-mapping \
  --function-name orders-stream-processor \
  --event-source-arn arn:aws:dynamodb:us-east-1:123456789012:table/Orders/stream/2026-04-26T00:00:00.000 \
  --starting-position LATEST \
  --batch-size 100
```

Exam-relevant distinctions:

- **DynamoDB Streams** are commonly used to react to table changes.
- **SQS** is used for decoupling and buffering.
- **Retries** and **dead-letter queues (DLQs)** matter when you need reliable processing.
- For API-driven workloads, Lambda is usually invoked synchronously through API Gateway.

**API Gateway provisioning and integration**

API Gateway provides the HTTP interface for your Lambda-backed application. The exam often tests how you connect routes, deploy stages, and control access.

Core API Gateway concepts:

- **Resources and methods** in REST APIs define paths and verbs.
- **Routes** in HTTP APIs define request paths and methods more directly.
- **Integrations** connect API Gateway to Lambda, HTTP backends, or other AWS services.
- **Stages** represent deployed environments such as `dev`, `test`, and `prod`.
- **Deployments** capture a snapshot of API configuration for a stage.

A typical REST API flow is:

1. Create the API and resource path.
2. Add a method such as `GET` or `POST`.
3. Configure the Lambda integration.
4. Grant API Gateway permission to invoke the Lambda function or alias.
5. Deploy the API to a stage.

Example Lambda permission for API Gateway:

```bash
aws lambda add-permission \
  --function-name orders-handler:prod \
  --statement-id apigw-invoke-prod \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn arn:aws:execute-api:us-east-1:123456789012:abc123/prod/POST/orders
```

Notes on this permission:

- Use the alias-qualified function name when the stage should invoke a specific alias.
- Restrict `--source-arn` to the API, stage, method, and resource when possible.
- If you change stages or aliases, verify the permission still matches the actual invoke path.

**Stages, deployments, and stage variables**

Stages are a major exam topic because they connect deployment management with environment-specific behavior.

Use stages to:

- Separate environments such as `dev`, `qa`, and `prod`.
- Apply stage-level throttling and logging settings.
- Map custom domains to a specific stage.
- Store **stage variables** for environment-specific values.

Stage variables are useful when you want the same API configuration to point to different Lambda aliases or backend endpoints.

Example stage variable usage:

- `lambdaAlias = dev`
- `backendUrl = https://api.example.internal`

In a Lambda integration, you can reference a stage variable in the integration URI or use it in mapping logic depending on the API type and configuration model. A common pattern is to pair:

- API Gateway stage `dev` with Lambda alias `dev`
- API Gateway stage `prod` with Lambda alias `prod`

This keeps deployment promotion predictable and reduces the risk of sending production traffic to the wrong version.

**Custom domains, routing, and release patterns**

Custom domains let you present a stable API hostname while changing the underlying stage or API version.

Typical uses:

- `api.example.com` points to a production stage.
- `dev-api.example.com` points to a development stage.
- Base path mappings route `/v1` or `/orders` to a specific API/stage.

This is useful when you need:

- Versioned public endpoints.
- Blue/green or canary-style release workflows.
- A stable client URL while you update Lambda aliases behind the scenes.

**API Gateway security, throttling, and quotas**

API Gateway can enforce request controls at multiple levels.

Key concepts:

- **Throttling** limits request rate and burst capacity.
- **Usage plans** associate API keys with throttling and quota limits.
- **API keys** identify clients for metering and quota enforcement.
- **Resource policies** can restrict access by source IP, VPC endpoint, or AWS account.

Important exam distinctions:

- Usage plans and API keys are primarily associated with **REST APIs**.
- HTTP APIs have different feature coverage and do not use the same usage plan model as REST APIs.
- A resource policy with only an `Allow` statement does not automatically create a deny-by-default posture unless the API is configured and accessed in a way that enforces that policy. In practice, resource policies are often combined with explicit `Deny` statements or network/account conditions.

Example REST API usage plan conceptually:

```bash
aws apigateway create-usage-plan \
  --name GoldPlan \
  --throttle burstLimit=50,rateLimit=20 \
  --quota limit=100000,period=MONTH
```

Use these controls when:

- You need to protect a backend from client spikes.
- You want to meter third-party consumers.
- You need per-client quotas for a public API.

**Request validation, CORS, and binary media**

These features often appear in practical provisioning questions.

- **Request validation** checks parameters and request bodies before the request reaches Lambda.
- **CORS** (Cross-Origin Resource Sharing) allows browser-based clients from other origins to call your API.
- **Binary media types** let API Gateway pass binary payloads such as images or PDFs correctly.

Why they matter:

- Validation reduces unnecessary Lambda invocations and improves error handling.
- CORS is required for many single-page applications that call APIs from the browser.
- Binary support is important when your API returns files or accepts uploads.

Example CORS response headers:

```json
{
  "Access-Control-Allow-Origin": "https://app.example.com",
  "Access-Control-Allow-Methods": "GET,POST,OPTIONS",
  "Access-Control-Allow-Headers": "Content-Type,Authorization"
}
```

For binary payloads, ensure the API configuration and integration both support the content type you need.

**DynamoDB table provisioning and data modeling**

DynamoDB is a managed NoSQL database designed for predictable low-latency access. The exam focuses heavily on table design and access patterns.

Core table concepts:

- **Partition key** determines item distribution.
- **Sort key** enables multiple items per partition and ordered access within a partition.
- **Primary key** can be simple or composite.
- **Items** are JSON-like documents.
- **Attributes** are the fields inside items.

Example table creation with a composite key:

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions \
      AttributeName=CustomerId,AttributeType=S \
      AttributeName=OrderId,AttributeType=S \
  --key-schema \
      AttributeName=CustomerId,KeyType=HASH \
      AttributeName=OrderId,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST
```

Why a sort key matters:

- It lets you store multiple orders for one customer.
- It supports queries like “all orders for customer X” or “orders between two timestamps” if the sort key is designed appropriately.
- It helps avoid scans, which are expensive and slow at scale.

**Indexes, projections, and access patterns**

Secondary indexes are a common exam topic because they solve query access patterns without changing the base table key.

Types:

- **Global Secondary Index (GSI)**: different partition key and optional sort key from the base table.
- **Local Secondary Index (LSI)**: same partition key as the base table, different sort key.

Operational implications:

- GSIs are created with their own throughput model and can be queried independently.
- LSIs must be created at table creation time and share the base table partition key.
- Index **projections** determine which attributes are copied into the index.

Projection types:

- `KEYS_ONLY`
- `INCLUDE`
- `ALL`

Use projections carefully:

- `KEYS_ONLY` minimizes storage but may require extra reads from the base table.
- `INCLUDE` balances storage and query convenience.
- `ALL` is simplest but can increase storage and write amplification.

Example GSI creation concept:

```bash
aws dynamodb update-table \
  --table-name Orders \
  --attribute-definitions AttributeName=Status,AttributeType=S \
  --global-secondary-index-updates '[
    {
      "Create": {
        "IndexName": "StatusIndex",
        "KeySchema": [
          {"AttributeName": "Status", "KeyType": "HASH"},
          {"AttributeName": "OrderId", "KeyType": "RANGE"}
        ],
        "Projection": {
          "ProjectionType": "INCLUDE",
          "NonKeyAttributes": ["CustomerId", "Total"]
        }
      }
    }
  ]'
```

Exam guidance:

- Use a GSI when your query pattern needs a different partition key.
- Use an LSI when you need an alternate sort order under the same partition key and can define it at table creation.
- Design around access patterns first, not around normalized relational modeling.

**Provisioned capacity, on-demand mode, and autoscaling**

DynamoDB supports two main capacity modes:

- **On-demand**: automatically scales for unpredictable workloads.
- **Provisioned**: you specify read and write capacity units.

Use cases:

- On-demand is good for spiky or unknown traffic.
- Provisioned is good when traffic is predictable and you want cost control.

Autoscaling is relevant in provisioned mode:

- It adjusts capacity based on utilization targets.
- It helps avoid manual scaling during traffic growth.
- It does not behave the same as on-demand mode, which removes the need to set capacity units manually.

Exam tip:

- If the question emphasizes unpredictable traffic and minimal operational management, choose on-demand.
- If it emphasizes predictable load and cost optimization with scaling policies, choose provisioned with autoscaling.

**DynamoDB writes, transactions, and conditional updates**

DynamoDB supports atomic operations and transactions. This is important when you need consistency across items or need to prevent invalid state.

Useful operations:

- `PutItem`
- `UpdateItem`
- `DeleteItem`
- `TransactWriteItems`
- `TransactGetItems`

For inventory or balance updates, use a **condition expression** to prevent invalid decrements.

Example safe inventory decrement:

```bash
aws dynamodb update-item \
  --table-name Inventory \
  --key '{"Sku":{"S":"SKU-123"}}' \
  --update-expression "SET quantity = quantity - :one" \
  --condition-expression "quantity >= :one" \
  --expression-attribute-values '{":one":{"N":"1"}}'
```

Why the condition matters:

- It prevents quantity from going negative.
- It makes the update fail atomically if stock is insufficient.
- It is the correct pattern for exam questions about safe decrements.

Transactional example:

```bash
aws dynamodb transact-write-items \
  --transact-items '[
    {
      "Update": {
        "TableName": "Inventory",
        "Key": {"Sku": {"S": "SKU-123"}},
        "UpdateExpression": "SET quantity = quantity - :one",
        "ConditionExpression": "quantity >= :one",
        "ExpressionAttributeValues": {
          ":one": {"N": "1"}
        }
      }
    },
    {
      "Put": {
        "TableName": "Orders",
        "Item": {
          "OrderId": {"S": "ORD-1001"},
          "Sku": {"S": "SKU-123"},
          "Quantity": {"N": "1"}
        }
      }
    }
  ]'
```

Use transactions when:

- You must update multiple items atomically.
- You need all-or-nothing behavior across related records.
- You want to avoid partial writes during order creation or inventory reservation.

**DynamoDB streams and event-driven processing**

DynamoDB Streams capture item-level changes and are often used to trigger Lambda functions.

Common use cases:

- Audit trails.
- Materialized views.
- Search index updates.
- Event-driven workflows after table changes.

Stream records can include:

- `INSERT`
- `MODIFY`
- `REMOVE`

Pair streams with Lambda when you need near-real-time reactions to data changes. Be aware that stream processing is asynchronous and should be designed to handle retries and idempotency.

**IAM for API Gateway, Lambda, and DynamoDB**

IAM is the glue that makes the stack work securely.

Key distinctions:

- A **permissions policy** says what actions are allowed.
- A **trust policy** says who can assume the role.
- A **resource-based policy** is attached to the resource itself, such as a Lambda function permission statement.

Common relationships:

- Lambda execution role trusts `lambda.amazonaws.com`.
- API Gateway needs permission to invoke Lambda.
- A user or CI/CD role needs permission to create APIs, deploy stages, and manage Lambda/DynamoDB resources.
- DynamoDB access is usually granted through the Lambda execution role, not directly to the API Gateway service.

Example IAM policy for a deployment role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "lambda:CreateFunction",
        "lambda:UpdateFunctionCode",
        "lambda:PublishVersion",
        "lambda:CreateAlias",
        "apigateway:POST",
        "apigateway:PUT",
        "apigateway:PATCH",
        "dynamodb:CreateTable",
        "iam:PassRole"
      ],
      "Resource": "*"
    }
  ]
}
```

Important exam note:

- `iam:PassRole` is required when a principal creates or updates a service resource and passes an IAM role to it.
- The principal must be allowed to pass that specific role to the target service.
- A deployment pipeline often needs both service permissions and `iam:PassRole`.

**End-to-end serverless provisioning pattern**

A common exam scenario is a REST API that creates and reads orders.

A practical architecture is:

- API Gateway `POST /orders` invokes Lambda `orders-handler:prod`.
- Lambda writes to DynamoDB `Orders`.
- Lambda execution role allows `dynamodb:PutItem` and CloudWatch Logs.
- API Gateway stage `prod` uses throttling and a custom domain.
- A separate `dev` stage points to `orders-handler:dev`.

Example request flow:

1. Client calls `https://api.example.com/orders`.
2. API Gateway authenticates or authorizes the request.
3. API Gateway invokes the correct Lambda alias.
4. Lambda validates input and writes to DynamoDB.
5. Lambda returns a JSON response to API Gateway.
6. API Gateway maps the response back to the client.

Example Lambda handler sketch:

```python
import json
import os
import boto3

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table(os.environ["ORDERS_TABLE"])

def lambda_handler(event, context):
    body = json.loads(event["body"])
    order_id = body["orderId"]

    table.put_item(
        Item={
            "OrderId": order_id,
            "CustomerId": body["customerId"],
            "Total": body["total"]
        }
    )

    return {
        "statusCode": 201,
        "headers": {
            "Content-Type": "application/json"
        },
        "body": json.dumps({"message": "created", "orderId": order_id})
    }
```

**Deployment and configuration strategy**

Use environment-specific separation to reduce risk:

- **Lambda aliases** for code version routing.
- **Environment variables** for table names, log levels, and feature flags.
- **API Gateway stages** for request routing and throttling.
- **Stage variables** when the API integration needs stage-specific backend values.

A practical release pattern is:

1. Deploy a new Lambda version.
2. Update the `dev` alias and test.
3. Promote the same version to `prod` alias.
4. Update API Gateway stage mappings if needed.
5. Verify permissions still match the alias-qualified ARN.

**Troubleshooting and exam traps**

Watch for these common mistakes:

- Granting API Gateway invoke permission only on the wrong ARN, especially when aliases are involved.
- Using a Lambda execution role without a trust policy for `lambda.amazonaws.com`.
- Choosing a scan when the access pattern can be satisfied by a query or index.
- Forgetting the condition expression on inventory decrements.
- Assuming usage plans apply equally to all API Gateway types.
- Confusing stage deployment with code deployment; they are related but not the same.
- Using a GSI when an LSI is required, or vice versa.
- Forgetting that LSIs must be defined at table creation time.
- Omitting `iam:PassRole` in deployment automation.

**Quick comparison**

| Topic | Best fit | Key exam clue |
|---|---|---|
| Lambda version | Immutable release snapshot | Use for controlled deployments |
| Lambda alias | Mutable pointer to a version | Use for `dev`/`prod` routing |
| API Gateway stage | Deployed API environment | Controls throttling and stage variables |
| DynamoDB on-demand | Unpredictable traffic | Minimal capacity management |
| DynamoDB provisioned | Predictable traffic | Can use autoscaling |
| GSI | Different partition key access pattern | Can be added after table creation |
| LSI | Alternate sort key under same partition key | Must exist at table creation |
| IAM trust policy | Who can assume the role | Needed for Lambda execution roles |
| IAM permissions policy | What the role can do | Scope to least privilege |

**Key takeaways**

- Design DynamoDB around access patterns, not normalization.
- Use Lambda aliases and API Gateway stages to separate environments cleanly.
- Grant invoke permissions on the correct Lambda ARN, especially when aliases are used.
- Use IAM trust policies for role assumption and permissions policies for service access.
- Add condition expressions to protect critical DynamoDB updates.
- Choose on-demand or provisioned capacity based on traffic predictability and operational needs.
