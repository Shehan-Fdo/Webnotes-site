## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**What a CDK Python application is**

The AWS Cloud Development Kit (<abbr title="Cloud Development Kit">CDK</abbr>) lets you define AWS infrastructure in Python and synthesize it into AWS CloudFormation templates. A Python CDK app usually includes a virtual environment, a CDK app entry point, one or more stacks, and the `cdk.json` file that tells the CDK CLI how to run your app.

You use this workflow when you want infrastructure to live in version control, be reviewed like application code, and be deployed consistently across environments. For exam prep, focus on the relationship between the Python source, the synthesized CloudFormation output, and the `cdk` command-line interface.

**Core project structure**

A typical Python CDK project contains a small set of files and directories that work together:

- `app.py` or another entry file
- `cdk.json`
- `requirements.txt`
- `.venv/`
- one or more stack modules, such as `my_cdk_app/my_cdk_app_stack.py`
- the `cdk.out/` synthesis output directory

The exact layout can vary depending on the CDK version and project template, but the purpose stays the same: your Python code defines constructs, and the CLI synthesizes them into deployable artifacts.

**Initialize the project**

You normally create a new CDK app with the CDK CLI. The CLI scaffolds the Python project, creates the virtual environment instructions, and adds starter files.

```bash
mkdir my-cdk-app
cd my-cdk-app
cdk init app --language python
```

After initialization, you typically install dependencies inside the project’s virtual environment. The generated instructions may differ by platform, so use the interpreter path that exists on your machine.

```bash
# macOS/Linux example
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt

# Windows example
py -m venv .venv
.venv\Scripts\activate
python -m pip install -r requirements.txt
```

If the template already created `.venv`, activate that environment instead of creating a second one. The important point is that you run CDK and Python commands in the same environment so dependency versions stay consistent.

**Understand `cdk.json` and the app entry point**

The `cdk.json` file tells the CDK CLI how to execute your app. In Python projects, the `app` field often points to the virtual environment interpreter and the entry script.

```json
{
  "app": ".venv/bin/python app.py"
}
```

On Windows, the path is different:

```json
{
  "app": ".venv\\Scripts\\python.exe app.py"
}
```

Some templates use `python3` or `python`, but you should not assume that a specific command exists on every platform. The safest approach is to use the interpreter from the virtual environment.

The app entry file usually creates the CDK application object and instantiates one or more stacks.

```python
#!/usr/bin/env python3
import aws_cdk as cdk

from my_cdk_app.my_cdk_app_stack import MyCdkAppStack

app = cdk.App()
MyCdkAppStack(app, "MyCdkAppStack")

app.synth()
```

The construct ID `"MyCdkAppStack"` is the logical identifier for the stack in the CDK app. It is not automatically the deployed CloudFormation stack name unless you explicitly set `stack_name` in the stack props or override the name during deployment.

**Create a stack**

A stack is the unit of deployment in AWS CloudFormation. In CDK, you define it as a class that extends `Stack`.

```python
from aws_cdk import Stack
from constructs import Construct

class MyCdkAppStack(Stack):
    def __init__(self, scope: Construct, construct_id: str, **kwargs) -> None:
        super().__init__(scope, construct_id, **kwargs)
```

You usually place resources inside the stack constructor. This keeps related infrastructure grouped together and makes the deployment boundary clear.

A common pattern is to pass environment information when you want the stack to target a specific AWS account and Region.

```python
MyCdkAppStack(
    app,
    "MyCdkAppStack",
    env=cdk.Environment(account="123456789012", region="us-east-1"),
)
```

Use `env` when you want explicit account/Region targeting. If you omit it, the stack can remain environment-agnostic, which is useful for reusable templates and some multi-environment workflows.

**Add a simple resource**

A minimal example often includes an Amazon S3 bucket or an AWS Lambda function. The exact resource is less important than understanding how constructs compose.

```python
from aws_cdk import Stack, aws_s3 as s3
from constructs import Construct

class MyCdkAppStack(Stack):
    def __init__(self, scope: Construct, construct_id: str, **kwargs) -> None:
        super().__init__(scope, construct_id, **kwargs)

        s3.Bucket(
            self,
            "AppBucket",
            versioned=True,
        )
```

This example creates a bucket construct inside the stack. CDK translates the construct tree into a CloudFormation template during synthesis.

For Lambda, prefer a runtime that matches your CDK library version and exam scope. If you need a broadly compatible example, use a runtime that is known to be supported by your installed `aws-cdk-lib` version rather than assuming the newest runtime constant is available.

```python
from aws_cdk import Stack, aws_lambda as _lambda
from constructs import Construct

class MyCdkAppStack(Stack):
    def __init__(self, scope: Construct, construct_id: str, **kwargs) -> None:
        super().__init__(scope, construct_id, **kwargs)

        _lambda.Function(
            self,
            "HelloFunction",
            runtime=_lambda.Runtime.PYTHON_3_11,
            handler="app.handler",
            code=_lambda.Code.from_asset("lambda"),
        )
```

**Install dependencies**

Python CDK apps rely on `aws-cdk-lib` and `constructs`. Your project may also include additional libraries for testing or application logic.

```bash
python -m pip install aws-cdk-lib constructs
python -m pip freeze > requirements.txt
```

If you use a generated template, `requirements.txt` may already contain the needed packages. Keep the file aligned with the environment you actually use for synthesis and deployment.

**Synthesize the app**

Synthesis converts your Python code into CloudFormation templates and supporting assets in `cdk.out/`.

```bash
cdk synth
```

This command is one of the most important in CDK development because it validates that your code can be turned into infrastructure. It also helps you inspect the generated template before deployment.

You can view the synthesized template directly:

```bash
cdk synth > template.yaml
```

Use synthesis when you want to verify resource properties, logical IDs, and asset packaging behavior without deploying anything.

**Compare changes with `cdk diff`**

`cdk diff` shows the difference between the current synthesized output and the deployed CloudFormation stack, and it can also be used in workflows that compare against a template file or are influenced by context and feature flags.

```bash
cdk diff
```

This is useful before deployment because it helps you spot unintended changes, such as resource replacement, property drift, or new permissions.

> `cdk diff` is a review tool, not a deployment tool. It helps you understand the impact of your code changes before you run `cdk deploy`.

**Deploy the stack**

Deployment sends the synthesized CloudFormation template to AWS and creates or updates the stack.

```bash
cdk deploy
```

If your app contains multiple stacks, you can deploy a specific one:

```bash
cdk deploy MyCdkAppStack
```

During deployment, the CDK CLI may prompt you to approve IAM-related changes or confirm resource creation. This is expected when the template introduces new permissions or sensitive resources.

**Bootstrap the environment when needed**

Before deploying assets such as Lambda code, Docker images, or file assets, the target account and Region often need CDK bootstrap resources. Bootstrapping creates the supporting S3 bucket, ECR repository, and roles used by the CDK deployment process.

```bash
cdk bootstrap aws://123456789012/us-east-1
```

You usually bootstrap once per account and Region combination, though you may need to repeat it for additional environments.

**Practical workflow**

A reliable CDK Python workflow looks like this:

1. Create the project with `cdk init app --language python`.
2. Activate the virtual environment.
3. Install or verify dependencies.
4. Define stacks and constructs in Python.
5. Run `cdk synth` to validate the template.
6. Run `cdk diff` to inspect changes.
7. Run `cdk deploy` to publish the stack.

This sequence reduces deployment surprises and keeps your infrastructure changes reviewable.

**Common configuration patterns**

Use stack props to control deployment behavior and environment targeting.

```python
MyCdkAppStack(
    app,
    "MyCdkAppStack",
    stack_name="my-cdk-app-prod",
    description="Production stack for the CDK Python app",
    env=cdk.Environment(account="123456789012", region="us-east-1"),
)
```

Use asset directories for local code or static files that CDK packages during synthesis and deployment.

```python
_lambda.Code.from_asset("lambda")
```

Use context values when you need environment-specific configuration that should be resolved during synthesis.

```bash
cdk synth -c stage=dev
```

And in Python:

```python
stage = app.node.try_get_context("stage") or "dev"
```

Context is useful for non-secret configuration such as environment names, feature toggles, or lookup parameters. Do not use it for secrets.

**Troubleshooting basics**

If synthesis fails, check the following first:

- Your virtual environment is active.
- `aws-cdk-lib` and `constructs` are installed in that environment.
- The app command in `cdk.json` points to a valid interpreter path.
- Your Python version is compatible with the installed CDK libraries.
- The stack code imports the correct modules and construct classes.

If deployment fails, inspect the CloudFormation events and the CDK CLI output. Common causes include missing bootstrap resources, insufficient IAM permissions, invalid resource names, or Region/account mismatches.

**Exam focus**

For the exam, know these concepts well:

- **CDK app**: the Python entry point that defines the application.
- **Stack**: the deployable CloudFormation unit.
- **Construct**: the building block used to define resources.
- **Synthesis**: conversion from code to CloudFormation.
- **Bootstrap**: prerequisite setup for asset-based deployments.
- **Diff**: preview of changes before deployment.
- **Deploy**: creation or update of AWS resources.

You should also understand why CDK is useful: it gives you reusable infrastructure definitions, supports higher-level abstractions, and integrates infrastructure changes into normal software development workflows.

**Quick reference**

| Command | Purpose |
|---|---|
| `cdk init app --language python` | Create a new Python CDK app |
| `cdk synth` | Generate CloudFormation output |
| `cdk diff` | Review changes before deployment |
| `cdk deploy` | Deploy the stack to AWS |
| `cdk bootstrap` | Prepare an account and Region for CDK assets |

**Minimal end-to-end example**

```python
#!/usr/bin/env python3
import aws_cdk as cdk
from aws_cdk import Stack, aws_s3 as s3
from constructs import Construct

class DemoStack(Stack):
    def __init__(self, scope: Construct, construct_id: str, **kwargs) -> None:
        super().__init__(scope, construct_id, **kwargs)

        s3.Bucket(self, "DemoBucket", versioned=True)

app = cdk.App()
DemoStack(app, "DemoStack")
app.synth()
```

This example shows the essential flow: define the app, create a stack, add a construct, and synthesize. Once you understand that flow, you can expand the app with parameters, multiple stacks, environment targeting, and reusable constructs.
