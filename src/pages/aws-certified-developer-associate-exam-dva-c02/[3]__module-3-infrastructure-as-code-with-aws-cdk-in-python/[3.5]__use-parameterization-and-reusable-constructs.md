## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Overview**

CloudFormation parameterization and reusable constructs help you avoid hardcoding values, reduce duplication, and promote consistent infrastructure across environments. In practice, you use parameters for values that vary by deployment, mappings for static lookups, conditions for optional resources, and reusable constructs such as nested stacks, modules, and macros for shared patterns.

For the exam, focus on choosing the right abstraction:
- Use **parameters** for values supplied at deploy time.
- Use **mappings** for fixed key-value lookups inside the template.
- Use **conditions** to create environment-specific behavior.
- Use **nested stacks** or **modules** to reuse infrastructure patterns.
- Use **macros** when you need template transformation before deployment.

**Why parameterization matters**

Hardcoded values make templates brittle. Parameterization lets you deploy the same template to dev, test, and prod with different inputs while keeping the template source controlled and reviewable. It also improves portability across accounts and Regions.

Common benefits include:
- Reduced duplication across templates
- Safer environment promotion
- Easier review and change management
- Better support for CI/CD pipelines
- Fewer manual edits before deployment

**CloudFormation parameters**

Parameters let you pass values into a template at stack creation or update time. They are the most common way to make a template reusable.

Typical parameter types include:
- `String`
- `Number`
- `CommaDelimitedList`
- AWS-specific types such as `AWS::EC2::VPC::Id`, `AWS::EC2::Subnet::Id`, and `AWS::SSM::Parameter::Value<String>`

A parameter definition can include:
- `Default`
- `AllowedValues`
- `AllowedPattern`
- `MinLength` and `MaxLength`
- `MinValue` and `MaxValue`
- `ConstraintDescription`
- `NoEcho`

Example:

```yaml
Parameters:
  EnvironmentName:
    Type: String
    Default: dev
    AllowedValues:
      - dev
      - test
      - prod
    Description: Deployment environment name

  InstanceType:
    Type: String
    Default: t3.micro
    AllowedValues:
      - t3.micro
      - t3.small
      - t3.medium

  VpcId:
    Type: AWS::EC2::VPC::Id
    Description: Existing VPC for the application
```

Use parameters when:
- The value changes per deployment.
- You want the deployer or pipeline to supply the value.
- You need validation at stack creation time.

> Do not use parameters for secrets unless you understand the exposure model. CloudFormation parameters are not a secret store. Prefer AWS Secrets Manager or AWS Systems Manager Parameter Store for sensitive values.

**Parameter validation and constraints**

Validation helps catch bad inputs before resources are created. This is especially useful in exam scenarios where a template must reject invalid environment names, CIDR blocks, or instance types.

Example with pattern and constraints:

```yaml
Parameters:
  AppName:
    Type: String
    AllowedPattern: '^[a-z][a-z0-9-]{2,20}$'
    ConstraintDescription: Must start with a lowercase letter and contain only lowercase letters, numbers, and hyphens.
```

Use `NoEcho: true` only to mask parameter values in stack events and the console. It does not make the value a secure secret in every context, so avoid treating it as a full security boundary.

**Pseudo parameters**

Pseudo parameters are predefined values that CloudFormation resolves automatically. They help you write portable templates without hardcoding account or Region data.

Common pseudo parameters:
- `AWS::AccountId`
- `AWS::Region`
- `AWS::Partition`
- `AWS::StackName`
- `AWS::NoValue`

Example:

```yaml
Resources:
  LogBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub '${AWS::StackName}-${AWS::AccountId}-${AWS::Region}-logs'
```

Use `AWS::NoValue` in conditions to remove a property when a condition is false.

```yaml
Conditions:
  UseSnapshot: !Equals [!Ref UseDBSnapshot, 'true']

Resources:
  DBInstance:
    Type: AWS::RDS::DBInstance
    Properties:
      DBSnapshotIdentifier: !If
        - UseSnapshot
        - !Ref DBSnapshotName
        - !Ref AWS::NoValue
```

**Mappings**

Mappings provide static lookup tables inside a template. They are ideal when you need a fixed relationship such as Region-to-AMI or environment-to-size.

Example:

```yaml
Mappings:
  RegionMap:
    us-east-1:
      AMI: ami-0123456789abcdef0
    us-west-2:
      AMI: ami-0fedcba9876543210
```

Reference a mapping with `Fn::FindInMap` or `!FindInMap`:

```yaml
Resources:
  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !FindInMap [RegionMap, !Ref AWS::Region, AMI]
      InstanceType: t3.micro
```

Use mappings when:
- The values are fixed and known at template authoring time.
- You want a simple lookup without external dependencies.
- You need Region-specific or environment-specific constants.

Avoid mappings when the data changes frequently. In that case, use parameters, SSM Parameter Store, or a pipeline-generated value.

**Conditions**

Conditions let you create resources or properties only when a logical expression evaluates to true. They are useful for optional features, environment-specific resources, and cost control.

Example:

```yaml
Conditions:
  IsProd: !Equals [!Ref EnvironmentName, prod]

Resources:
  ProdAlarm:
    Type: AWS::CloudWatch::Alarm
    Condition: IsProd
    Properties:
      AlarmName: !Sub '${AWS::StackName}-high-cpu'
```

You can also use conditions with `Fn::If` to set property values dynamically.

Common uses:
- Create extra alarms in production only
- Attach larger instance types in prod
- Enable encryption or backups conditionally
- Choose between new and existing resources

**Intrinsic functions for reuse**

Intrinsic functions are the glue that makes templates reusable.

Key functions:
- `!Ref` returns a parameter value, logical ID, or resource-specific value.
- `!Sub` performs string substitution.
- `!Join` concatenates strings.
- `!FindInMap` retrieves mapping values.
- `!If` selects between two values.
- `!ImportValue` imports an exported value from another stack.
- `!GetAtt` retrieves resource attributes.

Example:

```yaml
Outputs:
  ApiUrl:
    Value: !Sub 'https://${ApiGateway}.execute-api.${AWS::Region}.amazonaws.com/prod/'
```

Use `!Sub` when you need readable string interpolation. Use `!Join` when you need explicit concatenation or when working with lists.

**Reusable constructs: nested stacks**

Nested stacks let you split a large CloudFormation template into smaller templates. The parent stack manages the nested stack as a resource of type `AWS::CloudFormation::Stack`, and updates are coordinated through the parent stack operation.

This is useful when:
- A template becomes too large or hard to maintain
- You want to reuse a common infrastructure component
- You want clearer separation of concerns across template files

Example parent template:

```yaml
Resources:
  NetworkStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-bucket/network.yaml
      Parameters:
        EnvironmentName: !Ref EnvironmentName
```

Important behavior:
- The nested stack is still part of the parent stack lifecycle.
- You update it through the parent stack, not as an independent top-level stack in the same design.
- Outputs from the nested stack can be referenced by the parent stack.

Example nested stack output usage:

```yaml
Outputs:
  PublicSubnetId:
    Value: !Ref PublicSubnet
    Export:
      Name: !Sub '${AWS::StackName}-PublicSubnetId'
```

**Cross-stack references**

Cross-stack references let one stack export a value and another stack import it. This is useful for sharing stable infrastructure values such as VPC IDs, subnet IDs, or security group IDs.

Producer stack:

```yaml
Outputs:
  SharedVpcId:
    Value: !Ref VPC
    Export:
      Name: SharedVpcId
```

Consumer stack:

```yaml
Resources:
  AppSubnetGroup:
    Type: AWS::RDS::DBSubnetGroup
    Properties:
      DBSubnetGroupDescription: App subnets
      SubnetIds:
        - !ImportValue SharedVpcId
```

Use cross-stack references when:
- You need to share values between independently managed stacks.
- The exported value is stable and unlikely to change often.
- You want to reduce duplication without merging templates.

Be careful:
- Export names must match exactly.
- You cannot easily delete or modify an export while another stack imports it.
- Cross-stack references create coupling, so use them for stable shared infrastructure, not fast-changing application data.

**Reusable constructs: modules**

CloudFormation modules package resource patterns into reusable building blocks. They help standardize common infrastructure and reduce template repetition while keeping the CloudFormation model.

Use modules when:
- You want a reusable pattern with a defined interface
- You need consistency across teams or accounts
- You want to encapsulate a common resource set without writing a full nested stack

Modules are different from nested stacks:
- Nested stacks are separate templates managed as stack resources.
- Modules are reusable template fragments with a schema-like interface.

**Macros and transforms**

Macros let CloudFormation transform a template before it is processed. A macro can add, modify, or generate template content. Common built-in transforms include `AWS::Serverless` for AWS SAM and `AWS::Include` for inserting template fragments.

Example transform declaration:

```yaml
Transform:
  - AWS::Serverless-2016-10-31
```

`AWS::Include` inserts the contents of an S3 object into the template at the location of the transform. The included object must be valid fragment content for insertion, not an arbitrary standalone file.

Example:

```yaml
Transform:
  - Name: AWS::Include
    Parameters:
      Location: s3://my-bucket/fragments/common-tags.yaml
```

Use macros when:
- You need template generation or expansion
- You want to reuse fragments across templates
- You are using a framework such as AWS SAM

Be aware:
- Macros run before CloudFormation creates resources.
- A macro can change the final template shape, so debugging may require inspecting the transformed output.
- `AWS::Include` is for inserting fragments, not for importing stack outputs.

**AWS SAM as a reusable abstraction**

AWS Serverless Application Model (SAM) is a CloudFormation transform that simplifies serverless resource definitions. It is a common exam example of reusable constructs because it reduces boilerplate for Lambda, API Gateway, and event sources.

Example:

```yaml
Transform: AWS::Serverless-2016-10-31

Resources:
  HelloFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: src/
      Handler: app.lambda_handler
      Runtime: python3.12
      Events:
        ApiEvent:
          Type: Api
          Properties:
            Path: /hello
            Method: get
```

SAM is useful when:
- You want concise serverless templates
- You need repeatable patterns for Lambda-based applications
- You want a deployment workflow built around CloudFormation

**SSM Parameter Store for externalized configuration**

For values that change more often than template code, AWS Systems Manager Parameter Store is often a better fit than hardcoding or mapping. You can reference SSM parameters directly in CloudFormation using parameter types or dynamic references.

Example parameter type:

```yaml
Parameters:
  DbPassword:
    Type: AWS::SSM::Parameter::Value<String>
    Default: /myapp/prod/db/password
```

Dynamic reference example:

```yaml
Resources:
  MyFunction:
    Type: AWS::Lambda::Function
    Properties:
      Environment:
        Variables:
          LOG_LEVEL: !Ref LogLevel
          DB_HOST: '{{resolve:ssm:/myapp/prod/db/host}}'
```

Use SSM Parameter Store when:
- You want centralized configuration
- You need environment-specific values without editing templates
- You want to separate configuration from infrastructure code

For secrets, prefer Secrets Manager or secure SSM parameters with appropriate access controls.

**Practical CLI examples**

Deploy a stack with parameter overrides:

```bash
aws cloudformation deploy \
  --stack-name myapp-dev \
  --template-file template.yaml \
  --capabilities CAPABILITY_NAMED_IAM \
  --parameter-overrides \
    EnvironmentName=dev \
    InstanceType=t3.micro
```

Create a change set to review parameter-driven changes before execution:

```bash
aws cloudformation create-change-set \
  --stack-name myapp-prod \
  --template-body file://template.yaml \
  --change-set-name prod-preview \
  --parameters ParameterKey=EnvironmentName,ParameterValue=prod
```

List exports for cross-stack reference troubleshooting:

```bash
aws cloudformation list-exports
```

Describe a stack to inspect resolved parameters and outputs:

```bash
aws cloudformation describe-stacks --stack-name myapp-dev
```

**Design patterns you should recognize**

A few common patterns appear often in exam questions and real deployments:
- Use a single template with parameters for environment differences.
- Use mappings for Region-specific AMIs or fixed size tables.
- Use conditions to turn optional resources on or off.
- Use nested stacks to split a large architecture into manageable pieces.
- Use exports and imports to share stable infrastructure values.
- Use macros or SAM to reduce boilerplate and standardize patterns.

**Choosing the right tool**

| Need | Best fit | Why |
|---|---|---|
| Different values per deployment | Parameters | Supplied at stack create/update time |
| Static lookup table | Mappings | Fast, local, deterministic |
| Optional resources or properties | Conditions | Cleanly include or exclude logic |
| Large reusable template sections | Nested stacks | Breaks templates into manageable units |
| Shared stable values across stacks | Cross-stack references | Explicit producer/consumer contract |
| Template generation or expansion | Macros | Preprocessing before deployment |
| Serverless boilerplate reduction | SAM | Higher-level CloudFormation abstraction |

**Common pitfalls**

- Do not confuse **parameters** with secrets storage.
- Do not use mappings for data that changes frequently.
- Do not assume nested stacks are independently managed like separate application services; they are still controlled through the parent stack.
- Do not mismatch export and import names in cross-stack references.
- Do not treat `AWS::Include` as a generic file import mechanism; it inserts template fragments.
- Do not forget required organizational setup when using service-managed StackSets in real environments.
- Do not overuse conditions when a simpler parameter default or separate template would be clearer.

**StackSets and multi-account reuse**

StackSets extend CloudFormation across multiple accounts and Regions. They are useful when you need to deploy the same baseline infrastructure repeatedly.

Two permission models matter:
- **Self-managed** StackSets use IAM roles you create and manage.
- **Service-managed** StackSets integrate with AWS Organizations and typically target organizational units or accounts in the organization.

Example CLI pattern:

```bash
aws cloudformation create-stack-set \
  --stack-set-name baseline-network \
  --template-body file://network-baseline.yaml \
  --permission-model SERVICE_MANAGED
```

In practice, service-managed StackSets usually require organization integration and the right admin or delegated admin setup before you can target accounts or OUs successfully.

**Real-world example: environment-aware application stack**

This example combines parameters, mappings, and conditions.

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Environment-aware app stack

Parameters:
  EnvironmentName:
    Type: String
    AllowedValues: [dev, test, prod]
  InstanceType:
    Type: String
    Default: t3.micro

Mappings:
  EnvConfig:
    dev:
      DesiredCapacity: 1
    test:
      DesiredCapacity: 2
    prod:
      DesiredCapacity: 3

Conditions:
  IsProd: !Equals [!Ref EnvironmentName, prod]

Resources:
  AppInstance:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: !Ref InstanceType
      Tags:
        - Key: Environment
          Value: !Ref EnvironmentName

  ProdOnlyAlarm:
    Type: AWS::CloudWatch::Alarm
    Condition: IsProd
    Properties:
      AlarmName: !Sub '${AWS::StackName}-cpu-high'
      ComparisonOperator: GreaterThanThreshold
      EvaluationPeriods: 2
      Threshold: 80
      MetricName: CPUUtilization
      Namespace: AWS/EC2
      Period: 60
      Statistic: Average
```

This pattern is useful because it keeps one template reusable across environments while still allowing production-specific behavior.

**Exam focus**

For DVA-C02, expect questions that ask you to:
- Choose between parameters, mappings, and conditions
- Identify when to use nested stacks versus cross-stack references
- Recognize the purpose of transforms and macros
- Understand how to reuse infrastructure safely across environments
- Avoid hardcoding values that should be externalized

**Quick recall**

- **Parameters**: runtime inputs to a stack
- **Mappings**: static lookup tables
- **Conditions**: conditional resource or property logic
- **Nested stacks**: reusable subtemplates managed by a parent stack
- **Cross-stack references**: export/import stable values
- **Macros/Transforms**: preprocess templates
- **SAM**: serverless-focused CloudFormation transform
