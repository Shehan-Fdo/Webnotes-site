## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**AWS CDK mental model**

The AWS Cloud Development Kit (CDK) models infrastructure as a tree of **constructs** that is rooted in an **App**. You define resources in code, synthesize them into AWS CloudFormation templates, and then deploy those templates to an AWS account and Region.

At exam level, you should distinguish between:

- **Synthesis time**: when CDK code runs and generates CloudFormation.
- **Deployment time**: when CloudFormation creates or updates real AWS resources.
- **Resolution time**: when tokens, references, and lookups are resolved either during synthesis or deployment depending on the value type.

A useful way to think about CDK is:

- The **App** is the root of the construct tree.
- **Stacks** are deployment units inside the app.
- **Constructs** are reusable building blocks inside stacks or other constructs.
- **Environment configuration** determines whether values are fixed, looked up, or resolved later.

**App, stack, and construct hierarchy**

The **App construct** is the top-level scope in a CDK application. It owns the construct tree and acts as the root from which all other constructs are created. In exam terms, the app is where you instantiate one or more stacks and optionally apply global context, tags, and synthesis behavior.

A typical structure looks like this:

```ts
import * as cdk from 'aws-cdk-lib';
import { MyStack } from '../lib/my-stack';

const app = new cdk.App();

new MyStack(app, 'DevStack', {
  env: { account: '111111111111', region: 'us-east-1' },
});

new MyStack(app, 'ProdStack', {
  env: { account: '222222222222', region: 'us-east-1' },
});

app.synth();
```

Key exam points:

- **App** is the root scope; every construct has a parent scope except the app itself.
- **Stacks** are instantiated from the app and become separate CloudFormation templates.
- **Nested constructs** can be used to organize resources, but they do not create separate deployments unless they are stacks.
- The construct tree determines how IDs, paths, and logical IDs are derived.

**Constructs and scope/id behavior**

A **construct** is an abstraction that represents one or more AWS resources. Constructs are created with two important inputs:

- **scope**: the parent construct that contains it.
- **id**: a unique identifier within that scope.

The combination of scope and id matters because it affects:

- **Uniqueness** within the construct tree.
- The construct **path** from the app root.
- Generated CloudFormation **logical IDs** for resources.
- The stability of resource names across refactors.

Example:

```ts
const bucket = new s3.Bucket(this, 'ArtifactsBucket');
```

Here:

- `this` is the scope, usually a stack or another construct.
- `'ArtifactsBucket'` is the id unique within that scope.
- CDK uses the construct path to generate a logical ID such as `ArtifactsBucket...` in the synthesized template.

Important exam nuance:

- Changing a construct’s **id** or moving it to a different **scope** can change the generated logical ID.
- A logical ID change can cause CloudFormation to replace a resource if the resource cannot be updated in place.
- Stable construct paths help avoid unintended replacements.

**Logical IDs and CloudFormation naming**

CloudFormation uses **logical IDs** to track resources inside a stack template. CDK generates these IDs from the construct tree path, then applies hashing to keep them unique and deterministic.

Practical implications:

- A resource’s logical ID is not the same as its physical name.
- If you rename a construct or reorganize the tree, the logical ID may change.
- Some resources allow explicit physical names, but that does not eliminate logical ID behavior.

Example of a resource with an explicit physical name:

```ts
new s3.Bucket(this, 'LogsBucket', {
  bucketName: 'my-company-logs-bucket',
});
```

Use explicit names carefully:

- They can simplify cross-account references and human readability.
- They can also create deployment conflicts if the name must be globally unique or already exists.
- They do not replace the need for stable logical IDs.

**Stacks as deployment boundaries**

A **stack** is the unit of deployment in AWS CDK and CloudFormation. Each stack synthesizes to one CloudFormation template and is deployed independently.

Use multiple stacks when you need:

- **Failure isolation** between parts of the system.
- Separate deployment lifecycles.
- Different environments or accounts.
- Clear dependency boundaries.
- Smaller templates that are easier to reason about.

Use a single stack when:

- Resources are tightly coupled.
- You want simpler deployment and rollback behavior.
- Cross-stack communication would add unnecessary complexity.

Exam-relevant tradeoff:

- More stacks can improve modularity and blast-radius control.
- Too many stacks can increase dependency management overhead and deployment order complexity.

**Stack properties and operational controls**

Stacks support properties that matter in real deployments and exam scenarios.

Common stack-level concerns include:

- **Termination protection**: prevents accidental stack deletion.
- **Tags**: apply metadata for cost allocation, governance, or automation.
- **Dependencies**: control deployment order between stacks.
- **Permissions boundaries**: constrain IAM permissions created by the stack, when supported by the synthesizer or deployment model.
- **Stack synthesizer**: controls how assets and bootstrap resources are handled.

Example:

```ts
const stack = new cdk.Stack(app, 'AppStack', {
  terminationProtection: true,
  tags: {
    Project: 'Payments',
    Owner: 'PlatformTeam',
  },
});
```

Notes:

- **Termination protection** is useful for production stacks that should not be deleted casually.
- **Tags** propagate to supported resources and help with governance.
- **Dependencies** are important when one stack consumes outputs from another.
- **Permissions boundaries** are often used in controlled enterprise environments to limit IAM privilege escalation.

**Synthesis, deployment, and the synthesizer**

The **synthesizer** determines how CDK prepares assets and CloudFormation templates for deployment. This is especially important for files, Docker images, and other assets that must be uploaded before CloudFormation can reference them.

Typical flow:

1. CDK synthesizes the app into CloudFormation templates and asset manifests.
2. Assets are published to bootstrap resources such as an S3 bucket or ECR repository.
3. CloudFormation deploys the stack using the generated template.

Example CLI flow:

```bash
cdk bootstrap aws://111111111111/us-east-1
cdk synth
cdk deploy
```

Why bootstrap matters:

- CDK often needs pre-created resources to store assets.
- The bootstrap stack provides those resources and associated roles.
- Without bootstrapping, asset publishing may fail for stacks that include files or container images.

Exam distinction:

- **Synthesis-time resolution** happens when CDK can determine a value while generating the template.
- **Deployment-time resolution** happens when CloudFormation resolves tokens such as `Ref`, `Fn::GetAtt`, or parameters during deployment.

Example of a deployment-time token:

```ts
const table = new dynamodb.Table(this, 'Table', {
  partitionKey: { name: 'pk', type: dynamodb.AttributeType.STRING },
});

new cdk.CfnOutput(this, 'TableName', {
  value: table.tableName,
});
```

If the table name is not explicitly set, CDK may pass a token that CloudFormation resolves later.

**Environment configuration: account and Region**

CDK stacks can be **environment-specific** or **environment-agnostic**.

An environment-specific stack sets:

- `account`
- `region`

Example:

```ts
new MyStack(app, 'UsEast1Stack', {
  env: { account: '111111111111', region: 'us-east-1' },
});
```

An environment-agnostic stack omits `env` and can be synthesized without binding to a specific account or Region.

Tradeoffs:

- **Environment-specific stacks** support features that require knowing the target environment, such as lookups and some imported resources.
- **Environment-agnostic stacks** are more portable but cannot always perform environment-dependent operations at synthesis time.

**Environment lookups and context lookups**

A **lookup** asks AWS for information during synthesis and stores the result in CDK context. This is useful for discovering existing infrastructure such as VPCs, hosted zones, or AMIs.

Example:

```ts
const vpc = ec2.Vpc.fromLookup(this, 'Vpc', {
  vpcId: 'vpc-0123456789abcdef0',
});
```

Important behavior:

- The lookup result is cached in `cdk.context.json`.
- Re-synthesizing uses the cached value unless you clear or refresh context.
- Lookups generally require a concrete account and Region because CDK must know where to query.

Common exam limitation:

- Environment-agnostic stacks cannot reliably use lookups that depend on a specific account or Region.
- Imported resources that rely on environment-specific identifiers may also require explicit `env`.

Useful commands:

```bash
cdk context
cdk synth
cdk context --clear
```

**Context, CloudFormation parameters, and runtime configuration**

These three configuration mechanisms solve different problems.

- **CDK context**: synthesis-time configuration for the CDK app.
- **CloudFormation parameters**: deployment-time inputs supplied when creating or updating a stack.
- **Runtime configuration**: values read by the application after deployment, often from SSM Parameter Store, Secrets Manager, environment variables, or AppConfig.

Use them like this:

- Use **context** when the CDK code needs a value to decide what resources to create.
- Use **parameters** when the template should stay reusable and the deployer should provide a value.
- Use **runtime configuration** when the application needs a value after deployment without changing infrastructure.

Example CDK context in `cdk.json`:

```json
{
  "context": {
    "stage": "dev",
    "enableAlarms": true
  }
}
```

Example CloudFormation parameter:

```ts
const envName = new cdk.CfnParameter(this, 'EnvName', {
  type: 'String',
  default: 'dev',
});

new cdk.CfnOutput(this, 'Environment', {
  value: envName.valueAsString,
});
```

Example runtime configuration with SSM:

```ts
const param = ssm.StringParameter.fromStringParameterName(
  this,
  'ConfigParam',
  '/myapp/api/url'
);
```

Practical rule:

- If the value changes how infrastructure is built, prefer **context** or code.
- If the value changes per deployment, prefer **parameters**.
- If the value changes while the app runs, prefer **runtime configuration**.

**Cross-stack communication**

When one stack needs data from another, use explicit and safe patterns.

Common approaches:

- **References**: pass construct references when stacks are in the same app and same environment.
- **Outputs and imports**: export a value from one stack and import it in another.
- **SSM Parameter Store**: publish a value in one stack and read it in another or at runtime.
- **Event-driven decoupling**: avoid direct stack coupling when possible.

Example using stack outputs:

```ts
new cdk.CfnOutput(this, 'BucketName', {
  value: bucket.bucketName,
  exportName: 'SharedBucketName',
});
```

Example importing in another stack:

```ts
const bucketName = cdk.Fn.importValue('SharedBucketName');
```

Safer pattern in the same app:

```ts
const producer = new ProducerStack(app, 'Producer');
new ConsumerStack(app, 'Consumer', {
  bucket: producer.bucket,
});
```

Why this matters:

- Direct references are convenient but can create deployment ordering constraints.
- Exports/imports are explicit but can make refactoring harder because exports are part of the stack contract.
- SSM-based decoupling can reduce tight coupling between stacks and applications.

> Avoid circular dependencies between stacks. If Stack A depends on Stack B and Stack B depends on Stack A, deployment will fail or require redesign.

**Stack dependencies and deployment order**

Use stack dependencies when one stack must exist before another.

Example:

```ts
consumer.addDependency(producer);
```

This tells CDK and CloudFormation to deploy the producer stack first.

When to use dependencies:

- A consumer stack imports values from a producer stack.
- A stack needs a resource created by another stack before it can succeed.
- You want deterministic deployment order in a multi-stack app.

When to avoid them:

- If the dependency is only for application logic, prefer runtime discovery or configuration.
- If the dependency creates a cycle, refactor to shared services or parameterized runtime config.

**Environment-agnostic stacks and their limits**

Environment-agnostic stacks are useful for reusable libraries and templates, but they have constraints.

Limitations include:

- No reliable account-specific or Region-specific lookups.
- Harder use of imported resources that require concrete environment identifiers.
- Some asset publishing and bootstrap behaviors depend on knowing the target environment.
- Cross-stack references across environments are not supported in the same way as same-environment references.

Use environment-agnostic stacks when:

- You want a reusable construct library.
- You are building templates for multiple accounts or Regions.
- You do not need synthesis-time discovery of existing infrastructure.

Use environment-specific stacks when:

- You need VPC lookups, hosted zone lookups, or other environment-bound discovery.
- You deploy assets that require bootstrap resources in a known environment.
- You need predictable account/Region targeting.

**Practical stack design patterns**

A common production layout separates concerns by deployment boundary:

- **Networking stack**: VPC, subnets, security groups.
- **Data stack**: databases, queues, buckets.
- **Application stack**: Lambda, API Gateway, ECS, or Step Functions.
- **Observability stack**: alarms, dashboards, log subscriptions.

This pattern helps with:

- Independent updates.
- Clear ownership.
- Reduced blast radius.
- Easier rollback of isolated components.

Example topology:

```text
App
├── NetworkStack
├── DataStack
├── ServiceStack
└── MonitoringStack
```

Typical dependency direction:

- `ServiceStack` depends on `NetworkStack` and `DataStack`.
- `MonitoringStack` may depend on outputs from all stacks or consume metrics indirectly.

**Construct authoring and reuse**

When you create your own construct, keep it focused on a single responsibility. A construct should encapsulate a pattern, not an entire application.

Good construct characteristics:

- Reusable across stacks.
- Accepts configuration through props.
- Exposes useful outputs or references.
- Avoids hidden environment assumptions unless documented.

Example custom construct:

```ts
interface QueueWorkerProps {
  queue: sqs.IQueue;
}

class QueueWorker extends Construct {
  constructor(scope: Construct, id: string, props: QueueWorkerProps) {
    super(scope, id);

    new lambda.Function(this, 'Handler', {
      runtime: lambda.Runtime.NODEJS_20_X,
      handler: 'index.handler',
      code: lambda.Code.fromAsset('lambda'),
      environment: {
        QUEUE_URL: props.queue.queueUrl,
      },
    });
  }
}
```

This pattern is useful because:

- The construct can be reused in multiple stacks.
- The caller controls the environment and dependencies.
- The construct remains testable and composable.

**Exam checklist**

- Know that the **App** is the root of the construct tree.
- Know that **stacks** are the deployment unit and synthesize to CloudFormation templates.
- Understand how **scope** and **id** determine construct uniqueness and influence logical IDs.
- Distinguish **synthesis-time** from **deployment-time** resolution.
- Know why **bootstrap** is required for asset publishing.
- Understand when to use **environment-specific** versus **environment-agnostic** stacks.
- Recognize the limits of **lookups** and imported resources in agnostic stacks.
- Know the difference between **context**, **CloudFormation parameters**, and **runtime configuration**.
- Understand safe **cross-stack communication** patterns and deployment dependencies.
- Remember stack-level controls such as **termination protection**, **tags**, and **dependencies**.


Use `cdk diff` to inspect changes before deployment, especially when construct renames or scope changes might alter logical IDs and trigger replacements.
