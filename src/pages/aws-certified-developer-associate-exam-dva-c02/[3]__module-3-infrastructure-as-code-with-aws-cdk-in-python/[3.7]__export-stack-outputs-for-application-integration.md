## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**CloudFormation exports and imports**

AWS CloudFormation lets you expose selected stack outputs so other stacks can reuse them without hardcoding values. This is a common pattern for application integration when one stack creates shared infrastructure and another stack consumes it.

A typical flow is:

```text
Producer stack -> Exported output -> Consumer stack
```

Use this pattern when you want to separate concerns, reduce duplication, and keep infrastructure references consistent across environments.

- **Producer stack** creates a resource and publishes a value such as a VPC ID, subnet ID, security group ID, or database endpoint.
- **Consumer stack** imports that value and uses it in a resource definition.
- **Outputs** are the bridge between stacks, but only for values that are stable and safe to share.

**Why this matters for the exam**

CloudFormation exports often appear in questions about multi-stack architectures, shared networking, and decoupled deployments. You need to know when to use exports, when not to use them, and what restrictions apply.

**Core concepts**

CloudFormation outputs are declared in the `Outputs` section of a template. To make an output available to other stacks, you assign it an `Export` name.

```yaml
Outputs:
  SharedVpcId:
    Description: VPC ID used by application stacks
    Value: !Ref SharedVpc
    Export:
      Name: SharedVpcId
```

A consumer stack imports the value with `Fn::ImportValue` or the short form `!ImportValue`.

```yaml
Resources:
  AppSubnetGroup:
    Type: AWS::RDS::DBSubnetGroup
    Properties:
      DBSubnetGroupDescription: Subnets for the application database
      SubnetIds:
        - !ImportValue PrivateSubnetA
        - !ImportValue PrivateSubnetB
```

- **Export name** is the identifier other stacks reference.
- **Import value** resolves at deployment time, not runtime.
- **Stack output** is only exported if you explicitly define the export name.

**Real-world application examples**

Exports are useful anywhere one stack owns shared infrastructure and another stack consumes it.

- **RDS subnet groups**
  - A networking stack exports private subnet IDs.
  - A database stack imports those subnet IDs to create an `AWS::RDS::DBSubnetGroup`.
- **ECS services**
  - A shared network stack exports a security group ID and subnet IDs.
  - An application stack imports them for an `AWS::ECS::Service` or load balancer target group.
- **Lambda environment configuration**
  - A data stack exports an SQS queue URL or DynamoDB table name.
  - A compute stack imports the value into Lambda environment variables.

Example with Lambda environment variables:

```yaml
Resources:
  AppFunction:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: orders-worker
      Runtime: nodejs20.x
      Handler: index.handler
      Role: arn:aws:iam::123456789012:role/lambda-exec-role
      Code:
        ZipFile: |
          exports.handler = async () => ({ statusCode: 200 });
      Environment:
        Variables:
          TABLE_NAME: !ImportValue OrdersTableName
          QUEUE_URL: !ImportValue OrdersQueueUrl
```

**Producer and consumer stack pattern**

A clean design is to keep foundational resources in one stack and application resources in another.

```text
[Network Stack]
  - VPC
  - Private subnets
  - Security groups
  - Exports: VpcId, PrivateSubnetA, PrivateSubnetB

        |
        v

[App Stack]
  - RDS DB subnet group
  - ECS service
  - Lambda function
  - Imports: VpcId, PrivateSubnetA, PrivateSubnetB
```

This pattern helps you update application resources without recreating shared infrastructure.

**Important scope and naming rules**

CloudFormation exports are **Region-scoped** and **account-scoped**. A stack can only import values exported in the same AWS account and the same Region.

- You cannot import an export from another Region.
- You cannot import an export from another account.
- Export names must be unique within an account and Region.
- Export names are case-sensitive in practice, so treat them consistently.

> An export name cannot be reused by another stack in the same account and Region while it is in use.

**Dependency rules and cycles**

`Fn::ImportValue` has an important limitation for exam scenarios: you cannot use it with values that depend on resources created in the same template if that creates a dependency cycle.

- The imported value must come from a stack that already exists or is otherwise independently resolvable.
- You cannot create a circular dependency where Stack A imports from Stack B while Stack B also depends on Stack A.
- You also cannot build an export from a resource attribute that itself depends on an import from the same template in a way that forms a cycle.

Example of a bad pattern:

```yaml
Resources:
  MySecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: App SG
      VpcId: !ImportValue SharedVpcId
```

If the stack that exports `SharedVpcId` also depends on `MySecurityGroup`, you create a cycle and deployment fails.

**How to think about it for the exam**

- If a value is produced by a separate foundational stack, `Fn::ImportValue` is appropriate.
- If the value is created in the same stack and then immediately imported back into that same dependency chain, suspect a cycle.
- If the question mentions circular dependencies, the correct answer is usually to refactor the stacks or pass values another way.

**Export immutability and renaming limitations**

Export names are effectively immutable while they are in use.

- You cannot directly rename an export that has active consumers.
- You cannot change the export name if another stack imports it.
- To rename, you must first remove all imports or migrate consumers to a new export name.

A safe migration approach is:

1. Add a new export name alongside the old one.
2. Update consumer stacks to import the new name.
3. Verify all consumers are migrated.
4. Remove the old export.

This is a common exam trap because the obvious answer “rename the export” is not allowed when consumers still depend on it.

**Inspecting exports and imports**

Use the AWS CLI to inspect exported values and understand stack relationships.

List exports in the current account and Region:

```bash
aws cloudformation list-exports
```

Describe a stack and review its outputs:

```bash
aws cloudformation describe-stacks \
  --stack-name network-stack \
  --query "Stacks[0].Outputs"
```

Check stack events when a deployment fails:

```bash
aws cloudformation describe-stack-events \
  --stack-name app-stack \
  --query "StackEvents[?ResourceStatus=='CREATE_FAILED' || ResourceStatus=='UPDATE_FAILED'].[Timestamp,LogicalResourceId,ResourceStatusReason]"
```

List imports for a specific export name:

```bash
aws cloudformation list-imports \
  --export-name SharedVpcId
```

- `list-exports` shows what is available to consume.
- `list-imports` shows which stacks currently depend on an export.
- `describe-stacks` helps you confirm the producer stack output.
- `describe-stack-events` helps you troubleshoot failed imports, missing values, or dependency issues.

**Practical troubleshooting workflow**

When a consumer stack fails to deploy, check the problem in this order:

1. Confirm the export exists with `list-exports`.
2. Confirm the export name matches exactly in the consumer template.
3. Check whether the export is in the same account and Region.
4. Review `describe-stack-events` for dependency or validation errors.
5. Use `list-imports` to see whether the export is already in use by other stacks.

Common failure messages often point to one of these issues:

- Export name not found.
- Export name already in use.
- Cannot delete export because it is imported by another stack.
- Circular dependency detected.

**When to use CloudFormation exports vs SSM Parameter Store**

Both CloudFormation exports and AWS Systems Manager (SSM) Parameter Store can share values across components, but they fit different use cases.

| Feature | CloudFormation exports | SSM Parameter Store |
|---|---|---|
| Best for | Stack-to-stack infrastructure wiring | Application config and runtime parameters |
| Scope | Same account and Region | Can be shared more flexibly with IAM controls |
| Lifecycle | Tied to CloudFormation stacks | Independent of stack lifecycle |
| Typical values | VPC IDs, subnet IDs, security groups, ARNs | Feature flags, AMI IDs, endpoints, config strings |
| Consumer type | CloudFormation templates | Apps, Lambda, ECS, EC2, CI/CD pipelines |

Use **CloudFormation exports** when:

- The value is an infrastructure dependency between stacks.
- You want CloudFormation to manage the relationship.
- The consumer is another CloudFormation stack.

Use **SSM Parameter Store** when:

- The value is application configuration or operational data.
- Multiple non-CloudFormation consumers need the value.
- You want easier runtime retrieval or centralized config management.

> For exam questions, choose CloudFormation exports for tightly coupled infrastructure outputs and SSM Parameter Store for broader configuration sharing.

**End-to-end example**

A networking stack creates a VPC and private subnets, then exports the subnet IDs. An application stack imports those subnet IDs to create an RDS subnet group and deploy an ECS service into the same network.

Producer stack:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Shared network stack

Resources:
  SharedVpc:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.0.0.0/16

  PrivateSubnetA:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref SharedVpc
      CidrBlock: 10.0.1.0/24
      AvailabilityZone: us-east-1a

  PrivateSubnetB:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref SharedVpc
      CidrBlock: 10.0.2.0/24
      AvailabilityZone: us-east-1b

Outputs:
  VpcId:
    Value: !Ref SharedVpc
    Export:
      Name: SharedVpcId

  PrivateSubnetAId:
    Value: !Ref PrivateSubnetA
    Export:
      Name: PrivateSubnetA

  PrivateSubnetBId:
    Value: !Ref PrivateSubnetB
    Export:
      Name: PrivateSubnetB
```

Consumer stack:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Application stack

Resources:
  AppDbSubnetGroup:
    Type: AWS::RDS::DBSubnetGroup
    Properties:
      DBSubnetGroupDescription: Subnets for app database
      SubnetIds:
        - !ImportValue PrivateSubnetA
        - !ImportValue PrivateSubnetB

  AppServiceSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: App service security group
      VpcId: !ImportValue SharedVpcId

  AppFunction:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: app-worker
      Runtime: python3.12
      Handler: index.handler
      Role: arn:aws:iam::123456789012:role/lambda-exec-role
      Code:
        ZipFile: |
          def handler(event, context):
              return {"statusCode": 200}
      Environment:
        Variables:
          VPC_ID: !ImportValue SharedVpcId
```

This design keeps the network stack reusable and lets the application stack consume only the values it needs.

**Exam-style scenario callouts**

- If a question asks how to share a VPC ID between stacks, use a CloudFormation output export and `Fn::ImportValue`.
- If a question asks how to avoid hardcoding subnet IDs in an ECS or RDS template, import the subnet exports.
- If a question asks why an export cannot be renamed, remember that active imports block the change.
- If a question mentions a dependency cycle, the correct fix is to break the circular reference by redesigning the stack boundaries.

**Key takeaways**

- CloudFormation exports let one stack publish values for another stack to consume.
- `Fn::ImportValue` resolves exported values at deployment time.
- Exports are limited to the same account and Region.
- You cannot create dependency cycles with imported values.
- Export names cannot be renamed directly while consumers still use them.
- `list-exports`, `list-imports`, `describe-stacks`, and `describe-stack-events` are useful for troubleshooting.
- Use SSM Parameter Store when the value is more like application configuration than stack wiring.
