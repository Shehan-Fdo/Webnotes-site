## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Observability overview**

AWS observability usually combines three layers:

- **Logs** for event detail and forensic troubleshooting.
- **Metrics** for time-series monitoring and alerting.
- **Tracing** for request flow analysis across distributed services.

CloudWatch and X-Ray are the core AWS-native services for these layers. In practice, you instrument your application to emit structured logs, publish business or operational metrics, and propagate trace context so you can follow a request end to end.

**CloudWatch Logs fundamentals**

CloudWatch Logs stores application, system, and audit logs in **log groups** and **log streams**.

- A **log group** is the container for related logs, often one per application, environment, or service.
- A **log stream** is an ordered sequence of log events from a single source, such as one EC2 instance, Lambda execution environment, or container task.
- A **log event** is an individual timestamped record.

Common use cases include:

- Debugging application failures.
- Auditing security or operational events.
- Searching for patterns during incident response.
- Feeding logs into metric filters or subscription filters.

A practical naming strategy helps you manage retention and access control. For example, you might separate logs by environment:

```text
/app/orders/dev
/app/orders/test
/app/orders/prod
```

You can set retention on log groups so logs expire automatically instead of growing indefinitely.

```bash
aws logs put-retention-policy \
  --log-group-name /app/orders/prod \
  --retention-in-days 30
```

**Sending logs to CloudWatch Logs**

How you deliver logs depends on the compute platform.

- **AWS Lambda** automatically sends `stdout` and `stderr` to CloudWatch Logs.
- **Amazon EC2** instances usually need the CloudWatch agent or a log forwarder.
- **Amazon ECS** and **Amazon EKS** commonly use the `awslogs` driver, FireLens, or a sidecar/daemonset approach.
- **On-premises or hybrid workloads** can also ship logs with the CloudWatch agent if network access and permissions are in place.

For EC2, the CloudWatch agent can collect files and system metrics. A minimal log collection configuration looks like this:

```json
{
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/app/app.log",
            "log_group_name": "/app/orders/prod",
            "log_stream_name": "{instance_id}",
            "timezone": "UTC"
          }
        ]
      }
    }
  }
}
```

A typical Linux install flow is:

```bash
sudo yum install -y amazon-cloudwatch-agent
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-config-wizard
sudo systemctl enable amazon-cloudwatch-agent
sudo systemctl start amazon-cloudwatch-agent
```

**CloudWatch Logs permissions**

To publish logs, the runtime needs IAM (Identity and Access Management) permissions.

- Lambda execution roles need permissions such as `logs:CreateLogGroup`, `logs:CreateLogStream`, and `logs:PutLogEvents`.
- EC2 instances usually use an instance profile with permissions for the CloudWatch agent.
- ECS tasks may use a task execution role or task role depending on the logging setup.

A common least-privilege policy for writing to a specific log group is:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "WriteAppLogs",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/app/orders/prod:*"
    }
  ]
}
```

> If your application cannot create the log group automatically, create it ahead of time and attach retention and access controls before deployment.

**CloudWatch Logs Insights**

CloudWatch Logs Insights lets you query logs interactively with a purpose-built query language.

Use it when you need to:

- Find errors across many log streams.
- Aggregate counts by status code, request path, or user.
- Investigate latency spikes or exception bursts.

Example query:

```sql
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 20
```

Another useful pattern is extracting structured fields from JSON logs:

```sql
fields @timestamp, level, service, requestId, message
| filter level = "error"
| stats count() by service
```

Structured JSON logs are easier to query than free-form text. If you control the application format, emit JSON with stable keys such as `level`, `service`, `requestId`, and `durationMs`.

**CloudWatch Metrics fundamentals**

CloudWatch Metrics stores time-series data points for monitoring and alarms.

- A **metric** is identified by a namespace, name, and dimensions.
- A **dimension** is a key-value pair that adds context, such as `InstanceId`, `FunctionName`, or `ServiceName`.
- A **data point** is a single measurement at a specific time.

CloudWatch provides:

- **AWS service metrics** automatically emitted by many services.
- **Custom metrics** that your application publishes.
- **High-resolution metrics** for finer-grained monitoring when needed.

Use metrics when you need alerting, dashboards, or trend analysis. Logs are better for detail; metrics are better for fast detection and visualization.

**Publishing custom metrics**

Custom metrics are useful for business or application signals that AWS does not emit by default.

Examples:

- Orders created per minute.
- Payment failures.
- Queue processing latency.
- Cache hit ratio.
- Domain-specific SLA indicators.

You can publish metrics with the AWS CLI:

```bash
aws cloudwatch put-metric-data \
  --namespace "MyApp/Orders" \
  --metric-data '[
    {
      "MetricName": "OrdersCreated",
      "Dimensions": [
        {"Name": "Environment", "Value": "prod"},
        {"Name": "Service", "Value": "checkout"}
      ],
      "Value": 1,
      "Unit": "Count"
    }
  ]'
```

In application code, you often publish a metric when a business event occurs. For example, a checkout service can increment `OrdersCreated` after a successful payment.

A Python example using `boto3`:

```python
import boto3

cloudwatch = boto3.client("cloudwatch")

cloudwatch.put_metric_data(
    Namespace="MyApp/Orders",
    MetricData=[
        {
            "MetricName": "CheckoutLatency",
            "Dimensions": [
                {"Name": "Environment", "Value": "prod"},
                {"Name": "Service", "Value": "checkout"}
            ],
            "Value": 248.7,
            "Unit": "Milliseconds"
        }
    ]
)
```

**Metric design best practices**

Good metric design keeps dimensions useful and cost manageable.

- Keep metric names stable and descriptive.
- Use dimensions that help you slice data meaningfully.
- Avoid high-cardinality dimensions such as `requestId` or `userId` in metrics.
- Use logs or traces for per-request detail instead of metrics.
- Prefer counts, rates, latency, and error percentages for operational monitoring.

High-cardinality dimensions can create too many unique time series and make metrics expensive or hard to use.

**CloudWatch alarms and dashboards**

Alarms evaluate metrics against thresholds and can trigger actions.

Common alarm actions include:

- Sending notifications through Amazon SNS.
- Stopping, terminating, or recovering EC2 instances.
- Triggering Auto Scaling policies.

Example alarm creation:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name HighCheckoutErrors \
  --namespace "MyApp/Orders" \
  --metric-name CheckoutErrors \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 10 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=Environment,Value=prod Name=Service,Value=checkout \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:alerts
```

Dashboards help you visualize multiple metrics together. A useful dashboard combines:

- Request volume.
- Error rate.
- Latency percentiles.
- Queue depth.
- Downstream dependency health.

**CloudWatch Logs to Metrics**

Metric filters turn log patterns into metrics. This is useful when your application already emits logs but not metrics.

Examples:

- Count `ERROR` lines.
- Count `5xx` responses.
- Track specific exception types.
- Measure occurrences of security-relevant events.

Example metric filter pattern:

```bash
aws logs put-metric-filter \
  --log-group-name /app/orders/prod \
  --filter-name ErrorCount \
  --filter-pattern '"ERROR"' \
  --metric-transformations \
      metricName=ErrorCount,metricNamespace=MyApp/Orders,metricValue=1
```

Use metric filters when:

- You cannot easily change application code.
- You need a quick operational signal from existing logs.
- You want alarms based on log content.

**AWS X-Ray fundamentals**

AWS X-Ray helps you trace requests through distributed systems.

It is especially useful when your application includes:

- Multiple microservices.
- API Gateway, Lambda, and downstream AWS services.
- Complex latency or error propagation.
- Asynchronous or fan-out workflows.

X-Ray concepts:

- A **trace** represents a single end-to-end request path.
- A **segment** is data from one service or component.
- A **subsegment** is a downstream call within a segment, such as an SDK call to DynamoDB or S3.
- A **trace ID** correlates all parts of the request.
- A **sampling rule** controls how many requests are traced.

X-Ray gives you a service map, latency breakdowns, and error analysis. It is most valuable when logs tell you what happened in one component, but tracing shows where time was spent across components.

**Instrumenting applications for X-Ray**

You usually need two things:

- The X-Ray daemon or service integration, depending on the compute platform.
- Application instrumentation using the AWS X-Ray SDK or OpenTelemetry with X-Ray-compatible export, where supported.

For Lambda, X-Ray tracing can be enabled on the function configuration. For EC2 or containers, you typically run the X-Ray daemon and configure the SDK to send trace data to it.

Example Python instrumentation:

```python
from aws_xray_sdk.core import xray_recorder
from aws_xray_sdk.core import patch_all

patch_all()

xray_recorder.configure(service="orders-api")

def handler(event, context):
    subsegment = xray_recorder.begin_subsegment("call-dynamodb")
    try:
        # downstream call here
        pass
    finally:
        xray_recorder.end_subsegment()
```

A more realistic pattern is to wrap downstream SDK calls so X-Ray automatically records timing and errors.

**X-Ray and downstream AWS services**

X-Ray can trace calls to supported AWS services such as:

- Amazon DynamoDB
- Amazon S3
- Amazon SNS
- Amazon SQS
- Amazon API Gateway
- AWS Lambda

This helps you identify whether latency comes from your code or from a dependency.

For example:

- A Lambda function may be fast in its own code but slow because DynamoDB throttles.
- An API may return 500 errors because a downstream service times out.
- A queue consumer may spend most of its time waiting on an external API.

**Sampling and cost control**

Tracing every request can be expensive or unnecessary in high-volume systems. Sampling reduces overhead while still giving you representative traces.

Use sampling when:

- Traffic volume is high.
- You need a statistically useful view of performance.
- You want to reduce trace ingestion costs.

Keep in mind:

- Sampling is not the same as logging every error.
- You may want to trace all errors while sampling successful requests.
- Sampling rules should reflect production traffic patterns.

**Logs, metrics, and traces together**

The strongest observability setups correlate all three signals.

A practical workflow is:

1. Use **metrics** to detect that something is wrong.
2. Use **logs** to inspect the exact error or context.
3. Use **traces** to identify the slow or failing dependency.

Example scenario:

- CloudWatch alarm detects elevated `5xx` errors.
- Logs Insights shows repeated `TimeoutError` messages.
- X-Ray reveals that the latency is concentrated in a DynamoDB call or an external HTTP dependency.

This layered approach reduces mean time to resolution because each tool answers a different question.

**Practical deployment patterns**

For Lambda-based applications:

- Enable CloudWatch Logs automatically through the execution role.
- Emit structured JSON logs.
- Publish custom metrics for business events.
- Enable X-Ray tracing for request-level visibility.

For EC2-based applications:

- Install the CloudWatch agent for logs and metrics.
- Use an instance profile with least-privilege permissions.
- Run the X-Ray daemon if you need tracing.
- Standardize log paths and retention.

For container-based applications:

- Send logs to CloudWatch Logs using the logging driver or FireLens.
- Emit metrics from the application or a sidecar.
- Use X-Ray or OpenTelemetry instrumentation for traces.

**Troubleshooting checklist**

When observability data is missing, check the following:

- The IAM role has permission to write logs, metrics, or traces.
- The log group exists and retention is configured.
- The application is writing to the expected stream or file path.
- The CloudWatch agent or daemon is running.
- The region matches the resource configuration.
- Sampling is not excluding the trace you expect to see.
- The application emits structured data that your queries can parse.

A quick verification flow for logs on Linux EC2:

```bash
sudo systemctl status amazon-cloudwatch-agent
sudo tail -n 50 /var/log/app/app.log
aws logs describe-log-streams --log-group-name /app/orders/prod
```

**Exam focus**

For the exam, focus on choosing the right observability tool for the problem:

- Use **CloudWatch Logs** for detailed event records and troubleshooting.
- Use **CloudWatch Metrics** for alarms, dashboards, and trend monitoring.
- Use **metric filters** when you need metrics from existing logs.
- Use **X-Ray** for distributed tracing and latency analysis.
- Use **IAM permissions** correctly so the service can publish telemetry.
- Use **structured logs** and meaningful metric dimensions to keep analysis efficient.

A strong answer usually connects the symptom to the right signal:

- “Need to count errors from log text” → metric filter.
- “Need to alert on latency” → custom metric and alarm.
- “Need to find where a request slows down across services” → X-Ray.
- “Need to retain and search application events” → CloudWatch Logs.
