## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**CodeBuild test-and-package workflow**

AWS CodeBuild runs your commands inside a managed build environment and is commonly used to validate code, produce deployable artifacts, and hand those artifacts to Amazon S3, AWS CodePipeline, AWS CodeDeploy, or CloudFormation. For exam purposes, the key idea is that you separate validation from packaging so that tests run first and packaging happens only after the code is considered buildable.

A typical flow is:

- **Source checkout** from GitHub, CodeCommit, Bitbucket, S3, or CodePipeline.
- **Test execution** in `pre_build` or `build`.
- **Packaging** in `build` or `post_build`.
- **Artifact upload** after the build phases, based on the `artifacts` section of the buildspec and the project configuration.

> Artifact upload is not strictly tied to a successful `post_build` command in every case. The build status and artifact upload behavior depend on the build outcome and configuration, so do not assume a failed `post_build` always prevents artifact handling.

**Buildspec selection and project configuration**

CodeBuild can use a buildspec in two common ways:

- **Inline buildspec** defined in the CodeBuild project.
- **Repository buildspec file** such as `buildspec.yml` in the source root or a specified path.

The project configuration determines which buildspec is used, and that affects where you place test and package commands. If you store the buildspec in the repository, the build follows the versioned file in source control. If you define it inline, the project controls the workflow centrally.

Common project settings that matter in exam scenarios:

- **Source provider**: GitHub, CodeCommit, S3, or CodePipeline.
- **Buildspec source**: inline or file in the repository.
- **Environment image**: managed image or custom image with the required runtime tools.
- **Privileged mode**: required for Docker builds.
- **Artifacts destination**: S3 or pipeline output.
- **Cache**: local or S3 cache for dependency reuse.

**Build phases and where to put tests**

CodeBuild buildspec version `0.2` uses phases such as `install`, `pre_build`, `build`, and `post_build`. You usually place dependency installation in `install`, tests in `pre_build` or `build`, and packaging in `post_build` after tests succeed.

- **install**
  - Install runtime dependencies, package managers, and build tools.
  - Use this for `npm ci`, `pip install`, `mvn -v`, or Docker setup checks.
- **pre_build**
  - Run fast validation and unit tests early.
  - Fail fast before expensive packaging work begins.
- **build**
  - Compile, bundle, or create the deployable output.
  - Use this for application packaging, image builds, or static site generation.
- **post_build**
  - Perform final artifact preparation, rename outputs, generate manifests, or publish metadata.

A common exam pattern is to run tests before packaging so that broken code does not produce a deployable artifact.

**Example buildspec for test then package**

```yaml
version: 0.2

phases:
  install:
    runtime-versions:
      java: corretto17
    commands:
      - echo "Installing dependencies"
      - mvn -v
  pre_build:
    commands:
      - echo "Running unit tests"
      - mvn test
  build:
    commands:
      - echo "Packaging application"
      - mvn package -DskipTests
  post_build:
    commands:
      - echo "Build completed on $(date)"
      - ls -la target

artifacts:
  files:
    - target/*.jar
  discard-paths: yes
```

This pattern is useful when you want tests to run in one lifecycle invocation and packaging to happen in a separate invocation. `mvn package -DskipTests` skips test execution, but it is still a separate Maven lifecycle run, so use it intentionally rather than as a shortcut for a single combined command.

**Environment variables and build context**

CodeBuild exposes environment variables that help you locate source files, name artifacts, and tag outputs with the commit identity.

Common variables:

- `CODEBUILD_SRC_DIR`
  - Root directory of the primary source checkout.
- `CODEBUILD_RESOLVED_SOURCE_VERSION`
  - Resolved commit ID, version, or source revision.
- `CODEBUILD_BUILD_ID`
  - Unique build identifier.
- `CODEBUILD_BUILD_NUMBER`
  - Incrementing build number for the project.
- `CODEBUILD_SOURCE_VERSION`
  - Source version requested by the build trigger.

These are useful when you want artifact names to include the commit SHA or when your buildspec must reference files relative to the checked-out source.

```bash
echo "Source root: $CODEBUILD_SRC_DIR"
echo "Commit: $CODEBUILD_RESOLVED_SOURCE_VERSION"
mkdir -p dist
cp -r "$CODEBUILD_SRC_DIR"/app dist/
zip -r "artifact-${CODEBUILD_RESOLVED_SOURCE_VERSION}.zip" dist
```

Use `CODEBUILD_SRC_DIR` instead of hardcoding paths so the build works across different project layouts and source providers.

**Artifacts and handoff to downstream services**

CodeBuild artifacts are the files you want to preserve after the build. They are commonly uploaded to Amazon S3 or passed to CodePipeline as output artifacts. In a pipeline, the build stage usually produces an artifact that a later deploy stage consumes.

Important artifact behaviors:

- **Primary artifacts** are defined under `artifacts`.
- **Secondary artifacts** are defined under `artifacts.secondary-artifacts`.
- **Paths are relative** to the build root unless you set `base-directory`.
- **Encryption** can be applied through the S3 bucket and KMS (AWS Key Management Service) settings.
- **Pipeline handoff** often uses the CodePipeline artifact store, which is typically S3-backed and may be encrypted with KMS.

A build artifact is not just a zip file; it is the deployment package that downstream services expect. For example:

- **Lambda** expects a zip archive containing function code and dependencies.
- **CodeDeploy** may expect an application revision bundle with an `appspec.yml`.
- **CloudFormation** may consume packaged templates and nested artifacts.
- **Static web apps** often need a zipped `dist/` or `build/` directory.

**Primary artifact example**

```yaml
artifacts:
  files:
    - '**/*'
  base-directory: dist
  discard-paths: yes
```

This uploads the contents of `dist/` and strips directory paths. If you omit `base-directory`, CodeBuild collects files relative to the build root, which may still be useful but can package the wrong paths if you are not careful.

**Secondary artifacts syntax**

Secondary artifacts belong under the `artifacts` section in buildspec version `0.2`, not as a top-level key.

```yaml
artifacts:
  files:
    - target/app.jar
  secondary-artifacts:
    docs:
      files:
        - docs/**/* 
      base-directory: docs
      discard-paths: yes
    reports:
      files:
        - reports/**/*.xml
      base-directory: reports
      discard-paths: no
```

Use secondary artifacts when one build produces multiple deployable outputs, such as an application bundle and a documentation bundle, or a backend package and a frontend package.

**Reports and test result publishing**

CodeBuild can publish test reports to CodeBuild Reports if the project is configured to use reports. The buildspec can define report file locations, but the report group and project settings must support report collection.

Typical use cases:

- JUnit XML from Java, Maven, or Gradle tests.
- NUnit or other supported test formats.
- Multiple test files aggregated into one report group.

Example report configuration:

```yaml
reports:
  junit-reports:
    files:
      - '**/surefire-reports/*.xml'
    base-directory: target
    file-format: JUNITXML
```

This tells CodeBuild where to find test result files. The report group is associated with the project or created through the service configuration, so the buildspec alone is not the entire setup.

**Caching for faster test and package workflows**

Caching reduces repeated dependency downloads and speeds up builds, especially when you run tests and packaging on every commit.

Two common cache types:

- **Local cache**
  - Stored on the build host.
  - Useful for Docker layer caching and dependency reuse on the same host.
- **S3 cache**
  - Shared across builds through an S3 bucket.
  - Useful for package managers and build tools that can reuse downloaded dependencies.

Example cache configuration:

```yaml
cache:
  paths:
    - '/root/.m2/**/*'
    - 'node_modules/**/*'
```

For Maven, caching `~/.m2` can significantly reduce build time. For Node.js, caching `node_modules` may help, although many teams prefer reinstalling from lockfiles for reproducibility.

**Docker image builds**

If you build container images in CodeBuild, you must enable **privileged mode** on the project and use an image that supports Docker commands.

Example:

```yaml
phases:
  pre_build:
    commands:
      - aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin 123456789012.dkr.ecr.us-east-1.amazonaws.com
  build:
    commands:
      - docker build -t myapp:latest .
      - docker tag myapp:latest 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:latest
      - docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:latest
```

> `docker build` requires privileged mode in the CodeBuild project. Without it, Docker daemon access fails.

**`run-as` and command execution context**

The `run-as` setting lets you run commands as a specific user, but that user must exist in the build image. Do not assume a user like `codebuild` is present in every managed or custom image.

Example:

```yaml
phases:
  build:
    run-as: root
    commands:
      - chown -R 1000:1000 /tmp/build-output
      - ls -la /tmp/build-output
```

Use `run-as` when you need to control file ownership or execute commands with a non-default user. Verify the user exists in the image before relying on it.

**Failure handling with `on-failure`**

`on-failure` controls how CodeBuild reacts when a command fails in a phase. In buildspec v0.2, it can support behaviors such as aborting, continuing, or retrying in supported contexts.

Practical guidance:

- Use **ABORT** when a failed command should stop the build immediately.
- Use **CONTINUE** when a non-critical step should not fail the entire build.
- Use retry behavior carefully for transient network or package repository issues.

Example:

```yaml
phases:
  install:
    on-failure: ABORT
    commands:
      - npm ci
```

Treat `on-failure` as a phase-level control, not just a generic command wrapper.

**Static web app packaging example**

A common non-Lambda scenario is building a frontend app and packaging the compiled output for S3 or CodePipeline deployment.

```yaml
version: 0.2

phases:
  install:
    commands:
      - npm ci
  pre_build:
    commands:
      - npm test
  build:
    commands:
      - npm run build
  post_build:
    commands:
      - echo "Packaging static site"
      - cd dist
      - zip -r ../site.zip .

artifacts:
  files:
    - site.zip
```

This pattern is useful when the deploy stage expects a single archive or when you want to preserve the compiled site as a discrete artifact.

**CodeBuild project configuration checklist**

Before you rely on a buildspec, make sure the project is configured correctly.

1. Select the correct **source provider** and branch or revision.
2. Choose whether the buildspec comes from the **repository** or is **inline** in the project.
3. Pick an environment image with the required runtime tools.
4. Enable **privileged mode** if you build Docker images.
5. Configure **artifacts** to match the downstream service expectation.
6. Enable **reports** if you want test results published.
7. Add **cache** settings if dependency installation is a bottleneck.

**End-to-end example for test, package, report, and pipeline handoff**

This example shows a realistic buildspec for a Java application that runs tests, packages a JAR, publishes test reports, and produces an artifact for downstream deployment.

```yaml
version: 0.2

phases:
  install:
    runtime-versions:
      java: corretto17
    commands:
      - echo "Preparing build environment"
      - mvn -v
  pre_build:
    commands:
      - echo "Running tests before packaging"
      - mvn test
  build:
    commands:
      - echo "Creating deployable package"
      - mvn package -DskipTests
  post_build:
    commands:
      - echo "Staging artifact"
      - mkdir -p output
      - cp target/*.jar output/
      - ls -la output

reports:
  unit-tests:
    files:
      - '**/surefire-reports/*.xml'
    base-directory: target
    file-format: JUNITXML

artifacts:
  files:
    - '**/*'
  base-directory: output
  discard-paths: yes
```

Why this works well:

- Tests run before packaging, so failures stop the build early.
- The packaged JAR is copied into a clean output directory.
- Test results are published separately from the deployable artifact.
- The artifact structure is simple for CodePipeline or S3 consumption.

**Exam-focused distinctions**

- **Test first, package second** is the safest default for CI/CD pipelines.
- **Buildspec location** matters because inline and repository-based buildspecs are managed differently.
- **Artifact paths** must match the actual output directory structure.
- **Reports** require both buildspec configuration and project/report support.
- **Docker builds** require privileged mode.
- **Caching** improves speed but does not change build correctness.
- **Environment variables** help you avoid hardcoded paths and names.

**Common mistakes to avoid**

- Putting packaging before tests when the pipeline should fail fast.
- Assuming `post_build` must succeed for artifacts to exist in every configuration.
- Defining `secondary-artifacts` at the wrong level in the buildspec.
- Using `base-directory` values that do not match the real output folder.
- Forgetting privileged mode for Docker builds.
- Assuming a user named `codebuild` exists in every image.
- Treating report configuration as complete without enabling report collection in the project.
- Hardcoding source paths instead of using `CODEBUILD_SRC_DIR`.

**Quick reference**

- **Use `pre_build`** for tests you want to run before packaging.
- **Use `build`** for compilation and artifact creation.
- **Use `post_build`** for final staging and metadata.
- **Use `artifacts`** for deployable outputs.
- **Use `reports`** for test result publishing.
- **Use `cache`** for faster repeated builds.
- **Use `CODEBUILD_SRC_DIR`** and related variables for portable paths.
