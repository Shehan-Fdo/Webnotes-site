## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Pipeline-driven CDK deployments**

AWS Cloud Development Kit (AWS CDK) lets you define infrastructure in code and deploy it through a continuous delivery pipeline instead of manual `cdk deploy` runs. This approach is useful when you want repeatable releases, promotion across environments, and controlled approvals for production changes.

A CDK pipeline typically follows a simple flow:

- **Source**: retrieve application code from a repository.
- **Build/Synth**: run tests and generate CloudFormation templates with `cdk synth`.
- **Deploy**: apply stacks to one or more environments in sequence.
- **Approve**: pause for manual review when needed before production.

The key idea is that the pipeline deploys the synthesized CloudFormation output, not your source code directly. That separation improves traceability and makes the release artifact deterministic.

**Core CDK pipeline concepts**

CDK pipelines are usually built with the `aws-cdk-lib/pipelines` module. The pipeline itself is an AWS CDK stack that creates and manages the underlying AWS CodePipeline resources. In practice, you define the pipeline in code, then let it deploy the rest of your application stacks.

Important terms:

- **Synth**: the process of converting CDK constructs into CloudFormation templates.
- **Self-mutation**: the pipeline updates itself when the pipeline definition changes.
- **Wave**: a group of stages that can run in parallel.
- **Stage**: a deployment target such as dev, test, or prod.
- **Step**: an action in the pipeline, such as a shell command or approval gate.

A common pattern is to keep the pipeline stack separate from the application stacks it deploys. That separation reduces blast radius and makes it easier to bootstrap and manage permissions.

**Typical deployment topology**

A CDK pipeline often uses a dedicated tooling account and deploys application stacks into one or more target accounts and Regions.

- **Tooling account**
  - Hosts the pipeline, artifact buckets, and CodeBuild projects.
  - Often owns the CDK bootstrap resources.
- **Target accounts**
  - Receive application stacks for dev, staging, and production.
  - Use cross-account IAM roles created by CDK bootstrap.
- **Regions**
  - Can be single-Region or multi-Region depending on the application.

This model is especially useful when you need environment isolation, separate billing boundaries, or stricter production controls.

**Bootstrapping for pipeline deployments**

Before a CDK pipeline can deploy to an account and Region, you usually bootstrap that environment. Bootstrapping creates the resources CDK needs, such as:

- S3 buckets for file assets and templates
- IAM roles for deployment and lookup
- ECR repositories for container assets when needed
- KMS keys in some bootstrap configurations

A typical bootstrap command looks like this:

```bash
cdk bootstrap aws://123456789012/us-east-1
```

For modern CDK pipelines, you often bootstrap with a trusted execution role model so the pipeline can deploy cross-account safely. If you use multiple accounts, bootstrap each target environment.

> If the target environment is not bootstrapped correctly, pipeline deployment commonly fails during asset publishing or stack deployment.

**How the pipeline is assembled**

A CDK pipeline is usually defined with a source action, a synth step, and one or more deployment stages. The synth step runs in CodeBuild and produces the CloudFormation templates and asset manifests that the pipeline later deploys.

Example TypeScript pipeline definition:

```ts
import * as cdk from 'aws-cdk-lib';
import * as pipelines from 'aws-cdk-lib/pipelines';
import { Construct } from 'constructs';
import { MyAppStage } from './my-app-stage';

export class PipelineStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    const pipeline = new pipelines.CodePipeline(this, 'Pipeline', {
      synth: new pipelines.ShellStep('Synth', {
        input: pipelines.CodePipelineSource.gitHub('my-org/my-repo', 'main'),
        commands: [
          'npm ci',
          'npm run build',
          'npx cdk synth',
        ],
      }),
    });

    pipeline.addStage(new MyAppStage(this, 'Dev', {
      env: { account: '111111111111', region: 'us-east-1' },
    }));

    pipeline.addStage(new MyAppStage(this, 'Prod', {
      env: { account: '222222222222', region: 'us-east-1' },
    }), {
      pre: [
        new pipelines.ManualApprovalStep('ApproveProd'),
      ],
    });
  }
}
```

What this does:

- Pulls source from GitHub.
- Installs dependencies and synthesizes the app.
- Deploys a dev stage first.
- Requires manual approval before production.

**Source and synth steps**

The source step defines where your code comes from. Common sources include:

- GitHub
- CodeCommit
- Bitbucket
- Amazon S3, in some custom workflows

The synth step is where you validate and package the application. This step often includes:

- Dependency installation
- Unit tests
- Linting
- CDK synthesis
- Optional security scans

A practical synth command sequence might be:

```bash
npm ci
npm test
npm run lint
npx cdk synth
```

This is important because failures here stop bad changes before they reach deployment.

**Self-mutation behavior**

CDK pipelines can update themselves when the pipeline definition changes. This is called **self-mutation**. It means that if you change the pipeline stack code, the pipeline can deploy the updated pipeline configuration automatically.

Why it matters:

- You do not need to manually redeploy the pipeline for every pipeline change.
- The pipeline stays aligned with the source definition.
- Changes to stages, approvals, or steps can be rolled out through the same delivery mechanism.

Self-mutation is powerful, but it also means pipeline changes should be reviewed carefully. A broken pipeline definition can affect future deployments until corrected.

**Stages, waves, and approvals**

A **stage** represents a deployment target. You usually model each environment as a stage so you can promote the same artifact through multiple environments.

A **wave** groups stages that can deploy in parallel. Use waves when environments are independent and do not need strict sequential promotion.

Common patterns:

- **Sequential promotion**
  - Dev deploys first.
  - Test deploys after dev succeeds.
  - Prod deploys after approval.
- **Parallel regional rollout**
  - Multiple Regions deploy in a wave.
  - Useful for global applications.

Approvals are often inserted before production or other sensitive environments. A manual approval step is appropriate when:

- You need human review of test results.
- You want a release manager to sign off.
- Compliance requires a gate before production.

**Practical pipeline example with pre- and post-steps**

You can attach steps before or after a stage deployment.

```ts
pipeline.addStage(new MyAppStage(this, 'Test', {
  env: { account: '111111111111', region: 'us-east-1' },
}), {
  pre: [
    new pipelines.ShellStep('IntegrationTests', {
      commands: [
        'npm ci',
        'npm run test:integration',
      ],
    }),
  ],
  post: [
    new pipelines.ShellStep('SmokeTests', {
      commands: [
        'curl -f https://test.example.com/health',
      ],
    }),
  ],
});
```

Use pre-steps for validation before deployment and post-steps for verification after deployment.

**Asset publishing and deployment mechanics**

CDK applications often include assets such as:

- Lambda function bundles
- Docker images
- Static files for S3
- CloudFormation templates

The pipeline publishes these assets to the bootstrap resources before deploying stacks. This is why bootstrapping and permissions are critical.

For container-based workloads, CDK may build and push Docker images to Amazon Elastic Container Registry (Amazon ECR). For Lambda, it may upload zipped code bundles to S3.

A simplified asset flow looks like this:

1. CDK synthesizes the app and records asset metadata.
2. The pipeline publishes file and image assets.
3. CloudFormation deploys the stack using those asset references.

**Permissions and cross-account access**

Cross-account deployment requires IAM roles that allow the pipeline account to assume deployment roles in target accounts. CDK bootstrap usually creates these roles.

Key permission considerations:

- The pipeline execution role needs permission to read source, run builds, and publish assets.
- Target accounts need deploy roles that CloudFormation can assume.
- Lookup roles may be required for context lookups such as VPC or hosted zone queries.

Keep permissions as narrow as possible. Overly broad roles make the pipeline easier to break and harder to secure.

**Common deployment commands**

Even when using pipelines, you still use CDK commands locally for development and troubleshooting.

```bash
# Synthesize templates locally
npx cdk synth

# Diff local changes against deployed stacks
npx cdk diff

# Bootstrap an environment
npx cdk bootstrap aws://123456789012/us-east-1

# Deploy a stack manually for testing
npx cdk deploy MyStack
```

Use `cdk diff` to understand what the pipeline will eventually apply. Use `cdk synth` to catch template issues before committing.

**Troubleshooting pipeline failures**

Pipeline failures usually fall into a few categories:

- **Synth failures**
  - Missing dependencies
  - TypeScript or build errors
  - Failing tests
- **Bootstrap or permission failures**
  - Missing roles
  - Incorrect trust relationships
  - Unbootstrapped target account
- **Deployment failures**
  - CloudFormation rollback
  - Resource conflicts
  - Invalid parameters or quotas
- **Asset publishing failures**
  - ECR push issues
  - S3 access denied
  - Large asset or network problems

A practical troubleshooting sequence is:

1. Check the failed CodeBuild or CodePipeline action logs.
2. Confirm the environment is bootstrapped.
3. Run `npx cdk synth` locally.
4. Run `npx cdk diff` to inspect the change set.
5. Review CloudFormation events for the target stack.

**Example pipeline structure in practice**

A well-structured CDK pipeline often looks like this:

- **Pipeline stack**
  - Defines the CodePipeline resource.
  - Owns source, synth, and deployment orchestration.
- **Application stages**
  - Group related stacks for each environment.
  - Reuse the same constructs with different environment values.
- **Validation steps**
  - Run tests before deployment.
  - Run smoke tests after deployment.
- **Approval gates**
  - Protect production or regulated environments.

This structure keeps your infrastructure delivery repeatable and auditable.

**Exam focus points**

For exam purposes, focus on the following ideas:

- CDK pipelines automate deployment of CDK apps through CodePipeline.
- The pipeline deploys synthesized CloudFormation templates.
- Bootstrapping is required for asset publishing and cross-account deployment.
- Self-mutation allows the pipeline to update its own definition.
- Stages, waves, and approval steps control promotion flow.
- Pre- and post-steps support validation and verification.
- Cross-account deployments rely on IAM roles created by bootstrap.

```ts
const pipeline = new pipelines.CodePipeline(this, 'Pipeline', {

    input: pipelines.CodePipelineSource.gitHub('my-org/my-repo', 'main'),
    commands: ['npm ci', 'npm test', 'npx cdk synth'],
  }),
});

pipeline.addStage(new MyAppStage(this, 'Dev', {
  env: { account: '111111111111', region: 'us-east-1' },
}));

pipeline.addStage(new MyAppStage(this, 'Prod', {
  env: { account: '222222222222', region: 'us-east-1' },

  pre: [new pipelines.ManualApprovalStep('PromoteToProd')],
});
```

This pattern captures the essential exam-ready workflow: source, synth, deploy dev, approve, deploy prod.
