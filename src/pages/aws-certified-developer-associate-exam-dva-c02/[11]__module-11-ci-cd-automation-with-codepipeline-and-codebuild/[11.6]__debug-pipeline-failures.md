## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Pipeline failure triage**

When a delivery pipeline fails, you first identify whether the problem is in the **source**, **build**, **deploy**, or **pipeline orchestration** layer. That distinction matters because the fix is usually different for each stage. A source failure often means the pipeline never received a valid revision. A build failure usually means the application compiled or tested incorrectly. A deploy failure often means the target service rejected the artifact or deployment configuration.

Use the pipeline execution view as your starting point. It shows the stage and action that failed, the failure message, and whether the failure is retriable. In AWS CodePipeline, action-level diagnostics are often more useful than the overall pipeline status because they tell you which action failed, whether the action retried, and what the last successful execution looked like.

- **CodePipeline** execution history shows the failed action, timestamps, and revision details.
- **Action-level** failure messages help you separate a bad artifact from a bad pipeline configuration.
- **Retry behavior** depends on the action type and failure cause; some transient failures can be retried, while configuration errors usually require a fix before rerun.
- **Orchestration issues** affect the pipeline itself, such as missing permissions, invalid artifact paths, or a misconfigured action provider.

> If the pipeline fails before an action starts, suspect orchestration, permissions, or artifact wiring. If the action starts and then fails, inspect the service-specific logs and output artifacts.

**CodePipeline diagnostics and execution history**

CodePipeline gives you a stage-by-stage execution trail. Use it to determine whether the failure happened while fetching the source, running the build, or deploying. The execution history also helps you compare the current run with a previous successful run.

Useful checks include:

- The failed stage and action name.
- The action provider, such as S3, GitHub, CodeBuild, or CodeDeploy.
- The input artifact and output artifact names.
- The revision ID or commit ID associated with the source.
- The retry count and whether the action was automatically retried.
- The execution ID for correlating logs across services.

A practical CLI workflow is to inspect the pipeline and the execution details:

```bash
aws codepipeline get-pipeline --name my-pipeline

aws codepipeline list-pipeline-executions --pipeline-name my-pipeline

aws codepipeline get-pipeline-execution \
  --pipeline-name my-pipeline \
  --pipeline-execution-id 01234567-89ab-cdef-0123-456789abcdef
```

If you need to inspect a specific action failure, use the stage execution details:

```bash
aws codepipeline list-action-executions --pipeline-name my-pipeline
```

Look for:

- `status` values such as `Failed`, `InProgress`, or `Succeeded`.
- `errorDetails` or failure messages that point to permissions, artifact issues, or provider errors.
- `externalExecutionId`, which can map to a CodeBuild build ID or a CodeDeploy deployment ID.

**Source-stage failures**

Source-stage failures are common and often easy to misread as pipeline-wide issues. The source action is responsible for producing a valid artifact, so if it fails, later stages never receive input.

Common source-stage causes include:

- **S3 source artifact issues**
  - The object key is wrong or the object does not exist.
  - The source artifact bucket versioning is disabled when versioned revisions are expected.
  - The object is encrypted with a KMS key that the pipeline role cannot use.
  - The artifact format does not match what the source action expects.

- **Git connection or authorization problems**
  - The CodeStar Connections connection is not authorized or is in a pending state.
  - The OAuth token or connection permission is revoked.
  - The pipeline role lacks permission to read the repository.
  - The repository branch or commit reference is invalid.

- **Webhook or trigger misconfiguration**
  - The webhook is not created or is disabled.
  - The branch filter does not match the pushed branch.
  - EventBridge rules do not match the source event pattern.
  - The source action is configured for polling when webhook-based triggering is expected, or vice versa.

A failed S3 source revision often looks like a missing object or a bad key path. For example:

```bash
aws s3 ls s3://my-source-bucket/releases/app.zip
aws s3api head-object --bucket my-source-bucket --key releases/app.zip
```

If the object exists but the pipeline still fails, check permissions and encryption:

```bash
aws s3api get-bucket-versioning --bucket my-source-bucket
aws kms describe-key --key-id alias/my-artifact-key
```

A common permission-related source failure is a trust or access policy problem. For example, the pipeline service role may not trust CodePipeline, or the source provider role may not allow the required read actions. In practice, you verify:

- The IAM role trust policy includes the correct service principal.
- The role has `s3:GetObject`, `s3:GetObjectVersion`, or repository read permissions as needed.
- The artifact bucket policy allows the pipeline service role to read the object.

**Build-stage failures in CodeBuild**

CodeBuild failures are usually the most information-rich because the build logs show the exact command that failed. The key is to separate environment setup failures from application test failures.

Common build-stage causes include:

- **Dependency installation failures**
  - Package manager cannot reach the internet or a private repository.
  - `npm install`, `pip install`, `mvn package`, or `gradle build` fails because of a bad lockfile or incompatible version.
  - The build image lacks required tools or runtime versions.
  - A proxy, VPC, or security group blocks outbound access.

- **Test failures**
  - Unit tests fail because of code defects.
  - Integration tests fail because a dependency service is unavailable.
  - Test data or environment setup is incomplete.
  - The build passes compilation but fails during the test phase.

- **Environment variable or Parameter Store resolution errors**
  - A required environment variable is missing.
  - A value from AWS Systems Manager Parameter Store cannot be decrypted or read.
  - A Secrets Manager secret is missing or the build role lacks permission.
  - The buildspec references a variable name that does not exist.

- **Artifact or packaging issues**
  - The output path in `artifacts` is wrong.
  - The build produces no files in the expected directory.
  - The artifact is too large or contains excluded files.

A typical `buildspec.yml` can help you isolate the failing phase:

```yaml
version: 0.2

phases:
  install:
    runtime-versions:
      nodejs: 20
    commands:
      - npm ci
  pre_build:
    commands:
      - npm test -- --runInBand
  build:
    commands:
      - npm run build
artifacts:
  files:
    - '**/*'
  base-directory: dist
```

If dependency installation fails, inspect the install phase logs first. If tests fail after dependencies install successfully, the issue is usually application logic or test environment setup rather than infrastructure.

Useful commands inside the build log or local reproduction:

```bash
npm ci
npm test
aws ssm get-parameter --name /myapp/config/db-url --with-decryption
aws secretsmanager get-secret-value --secret-id myapp/prod/db
```

For environment variable issues, check both the buildspec and the project configuration:

- Build environment variables defined in the project.
- Variables injected from Parameter Store.
- Variables injected from Secrets Manager.
- Reserved variables provided by CodeBuild, such as build IDs and source versions.

If the build uses batch builds, remember that one child build can fail while others succeed. Inspect the batch summary and the individual child build logs. If reports are enabled, use them to see which tests failed even when the build container exits after test execution.

```bash
aws codebuild batch-get-builds --ids my-project:build-id
aws codebuild batch-get-reports --report-arns arn:aws:codebuild:us-east-1:123456789012:report/my-report
```

When report groups are configured, test results may be available even if the build fails after tests run. That is useful when the container exits with a packaging error after the test suite already produced a report.

**Deploy-stage failures in CodeDeploy**

CodeDeploy failures depend on the compute platform, and the failure pattern changes between EC2/on-premises, Lambda, and ECS deployments. The deployment artifact format and lifecycle hooks are different for each target.

For EC2 or on-premises deployments, the `AppSpec` file defines files, hooks, and deployment behavior. For Lambda and ECS deployments, the `AppSpec` file is still central, but the structure and validation rules differ.

- **Lambda deployments**
  - The AppSpec references the Lambda function and alias.
  - CodeDeploy shifts traffic between versions using the alias.
  - Lifecycle hooks such as `BeforeAllowTraffic` and `AfterAllowTraffic` can validate the new version.
  - Failures often occur when the function version, alias, or permissions are incorrect.

- **ECS deployments**
  - The AppSpec references the ECS service and container name/port mapping.
  - CodeDeploy validates the target group and listener configuration.
  - Traffic shifting depends on the load balancer and target group health checks.
  - Failures often occur when the task definition, container port, or target group mapping is wrong.

For ECS blue/green deployments, CodeDeploy validates that the new task set becomes healthy before shifting traffic. If the target group health checks fail, the deployment can stop before full traffic cutover. If lifecycle hooks fail, the deployment may stop even if the tasks start successfully.

A simplified ECS AppSpec example:

```yaml
version: 0.0
Resources:
  - TargetService:
      Type: AWS::ECS::Service
      Properties:
        TaskDefinition: arn:aws:ecs:us-east-1:123456789012:task-definition/my-task:42
        LoadBalancerInfo:
          ContainerName: app
          ContainerPort: 8080
Hooks:
  - BeforeAllowTraffic: "validate-before-traffic"
  - AfterAllowTraffic: "validate-after-traffic"
```

A simplified Lambda AppSpec example:

```yaml
version: 0.0
Resources:
  - myLambdaFunction:
      Type: AWS::Lambda::Function
      Properties:
        Name: my-function
        Alias: live
Hooks:
  - BeforeAllowTraffic: "preTrafficHook"
  - AfterAllowTraffic: "postTrafficHook"
```

When a deployment fails, inspect the deployment ID and lifecycle event details:

```bash
aws deploy get-deployment --deployment-id d-ABCDEFGHI
aws deploy get-deployment-target --deployment-id d-ABCDEFGHI --target-id target-id
aws deploy list-deployment-instances --deployment-id d-ABCDEFGHI
```

Look for:

- Hook failures in `BeforeInstall`, `AfterInstall`, `BeforeAllowTraffic`, or `AfterAllowTraffic`.
- Target group health check failures.
- Incorrect alias or task definition references.
- IAM permission issues for the deployment service role.

**Rollback and redeployment behavior**

Rollback behavior is important because it determines whether a failed deployment leaves the system in a safe state. In CodeDeploy, automatic rollback can restore the previous revision when a deployment fails or when alarms trigger, depending on configuration. If automatic rollback is disabled, the failed revision may remain partially applied until you manually redeploy or roll back.

In CodePipeline, a failed stage does not automatically mean the entire pipeline is rolled back. Instead, the pipeline execution stops, and you typically rerun the pipeline after fixing the issue. If the pipeline is configured to trigger on new source revisions, a new commit or artifact can start a fresh execution.

Key points:

- **Automatic rollback enabled**
  - CodeDeploy can shift traffic back to the previous version.
  - The service may restore the last known good revision.
  - You still need to inspect the root cause before redeploying.

- **Automatic rollback disabled**
  - The failed deployment may remain in a failed or partially shifted state.
  - You may need to manually redeploy a known good revision.
  - For ECS or Lambda, verify the active task set or alias before retrying.

- **CodePipeline rerun behavior**
  - A failed execution can often be retried from the failed action or restarted from the beginning.
  - If the source revision is unchanged, the same defect will fail again.
  - If the failure was transient, a rerun may succeed without code changes.

**Distinguishing action failures from orchestration issues**

A common exam trap is confusing a service failure with a pipeline configuration failure. Use the following rule of thumb:

- If the action starts and logs a service-specific error, the issue is likely in the source, build, or deploy service.
- If the action never starts, the issue is likely in permissions, artifact wiring, or pipeline configuration.
- If the pipeline cannot resolve the next action, suspect a malformed stage definition or missing artifact.

Examples of orchestration issues include:

- The output artifact name does not match the next action’s input artifact.
- The pipeline role cannot assume the action role.
- The action configuration references a deleted resource.
- The artifact bucket policy blocks access.
- The region or account reference is incorrect.

A trust policy problem between CodePipeline and CodeBuild can look like this in IAM terms:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "codebuild.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

If the trust relationship is wrong, CodeBuild may never start or may fail to access the resources it needs. Always verify both the trust policy and the attached permissions policy.

**Notifications and faster detection**

Notifications help you detect failures before users report them. In AWS, you can use Amazon EventBridge to react to pipeline state changes, CodeBuild build failures, and CodeDeploy deployment events. You can also route notifications to Amazon SNS for email or downstream automation.

Common patterns:

- EventBridge rule on `CodePipeline Pipeline Execution State Change`.
- EventBridge rule on `CodeBuild Build State Change`.
- EventBridge rule on `CodeDeploy Deployment State-change Notification`.
- SNS topic for human-readable alerts.
- Lambda function for automated triage or ticket creation.

A simple EventBridge pattern for pipeline failures:

```json
{
  "source": ["aws.codepipeline"],
  "detail-type": ["CodePipeline Pipeline Execution State Change"],
  "detail": {
    "state": ["FAILED"]
  }
}
```

This helps you capture failures quickly and correlate them with the execution ID before logs age out or the next deployment overwrites the context.

**Practical troubleshooting workflow**

1. Identify the failed stage and action in CodePipeline.
2. Check whether the failure is source, build, deploy, or orchestration related.
3. Open the service-specific logs:
   - CodePipeline execution history for action metadata.
   - CodeBuild logs in CloudWatch Logs.
   - CodeDeploy deployment and lifecycle event details.
4. Verify permissions, artifact paths, and environment configuration.
5. Reproduce the issue locally when possible.
6. Fix the root cause, then rerun the pipeline or redeploy the revision.

**High-value exam signals**

- **Source failures** often involve S3 object issues, Git authorization, or trigger misconfiguration.
- **Build failures** often involve dependency installation, tests, environment variables, or Parameter Store access.
- **Deploy failures** often involve AppSpec errors, lifecycle hook failures, target group health checks, or alias/task definition mismatches.
- **Rollback behavior** depends on whether automatic rollback is enabled in CodeDeploy.
- **Notifications** through EventBridge or SNS reduce mean time to detect and triage failures.

**Quick reference**

| Area | What to inspect | Common failure clues |
|---|---|---|
| Source | Execution history, revision ID, artifact bucket, webhook/trigger | Missing S3 object, bad branch, revoked connection |
| Build | CloudWatch Logs, buildspec, env vars, reports | Install failure, test failure, missing secret/parameter |
| Deploy | Deployment ID, lifecycle hooks, target health | AppSpec mismatch, hook failure, traffic shift validation |
| Orchestration | IAM roles, artifact names, action configuration | Trust policy issue, bad input/output mapping, missing permissions |
