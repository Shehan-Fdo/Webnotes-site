## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Promotion pipelines: the core idea**

Promotion means moving the same build output through environments such as dev, test, staging, and prod with increasing controls. The exam often tests whether you preserve artifact integrity while changing only the deployment target and environment-specific configuration.

A good promotion design keeps the artifact immutable and uses separate deployment steps or parameters per environment. You typically build once, then deploy many times.

- **Build once** to reduce drift between environments.
- **Promote the same artifact** rather than rebuilding for each stage.
- **Gate production** with automated checks and manual approval when needed.
- **Separate application config** from the artifact so environment values do not require a rebuild.

**Common AWS services in a promotion flow**

AWS CodePipeline orchestrates stages, CodeBuild produces artifacts, and deployment targets consume them. Depending on the workload, you may also use Amazon S3, Amazon Elastic Container Registry (ECR), AWS CodeArtifact, AWS Lambda, AWS Signer, and AWS Key Management Service (KMS).

- **CodePipeline** manages stage transitions and approvals.
- **CodeBuild** compiles, tests, packages, and can sign or publish artifacts.
- **S3** stores versioned build outputs and deployment bundles.
- **ECR** stores container images for promotion across environments.
- **CodeArtifact** stores package artifacts such as npm, Maven, Python, or NuGet packages.
- **Signer** provides AWS-native signing for code and artifacts.

**Artifact immutability and traceability**

An artifact should be traceable from source commit to build output to deployed version. In practice, you want a unique build identifier, versioned storage, and a way to verify that the deployed object is the same one that passed validation.

S3 versioning is a common safeguard because it preserves prior object versions for rollback and audit. Object immutability can be strengthened with S3 Object Lock when compliance or retention requirements apply, though Object Lock is not always required for standard exam scenarios.

- **Versioning** helps you recover previous artifact versions.
- **Object Lock** can prevent deletion or overwrite during a retention period.
- **Build metadata** should include commit ID, build number, and artifact hash.
- **Rollback** is easier when the exact prior artifact remains available.

> A common exam trap is confusing artifact storage encryption with application payload encryption. Encrypting the CodePipeline artifact store protects the pipeline’s intermediate files, while application-level encryption protects the contents of the package or data the application uses.

**CodePipeline artifact store and encryption**

CodePipeline uses an artifact store, usually an S3 bucket, to hold input and output artifacts between actions. You can encrypt this artifact store with AWS KMS.

For stronger control, use a customer-managed KMS key rather than the default AWS-managed key. This is especially relevant when you need explicit key policy control, cross-account access, or auditability.

Example pipeline artifact store configuration:

```json
{
  "artifactStore": {
    "type": "S3",
    "location": "my-pipeline-artifacts-123456789012-us-east-1",
    "encryptionKey": {
      "id": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555",
      "type": "KMS"
    }
  }
}
```

In the console, you configure this by selecting the artifact bucket and choosing a KMS key for encryption. The key must allow CodePipeline and the relevant AWS principals to use it.

- **Artifact store encryption** protects files exchanged between pipeline actions.
- **Customer-managed KMS keys** give you tighter policy control.
- **Application artifact encryption** is separate and may be handled inside the package, container image, or application data layer.
- **Do not assume** artifact store encryption automatically encrypts the application’s internal secrets or payloads.

**Cross-account artifact store requirements**

Cross-account promotion is common when dev, test, and prod live in different AWS accounts. CodePipeline can orchestrate this, but the artifact storage and IAM trust relationships must be designed carefully.

A common pattern is to use a separate artifact bucket in the target account or a dedicated shared services account, with policies that allow the pipeline role and target action roles to read and write as needed. The exact setup depends on where the pipeline runs and which account owns the action.

Key requirements include:

- **S3 bucket policy** that grants the pipeline role access to the artifact bucket.
- **KMS key policy** that allows encryption and decryption by the pipeline and target roles.
- **IAM role trust policy** that permits CodePipeline or the target service to assume the action role.
- **Cross-account action role** in the target account for deploy actions such as CloudFormation, CodeDeploy, or ECS.

A typical trust relationship for a target account role looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "codepipeline.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

A bucket policy may need to allow the pipeline account to access objects:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowPipelineReadWrite",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/CodePipelineServiceRole"
      },
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:GetObjectVersion",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-cross-account-artifacts",
        "arn:aws:s3:::my-cross-account-artifacts/*"
      ]
    }
  ]
}
```

- **Separate artifact buckets** reduce coupling between accounts.
- **Cross-account KMS permissions** are often the missing piece in failed pipelines.
- **Action roles** should be scoped to the minimum permissions required.
- **Bucket and key policies** must align with the pipeline’s assumed roles.

**Approval actions in CodePipeline**

Manual approval actions are used when a human must review a release before it proceeds. They are common before production deployment, especially after automated tests and security checks pass.

In the console, you add an approval action as a stage action. In JSON, it is represented as an action provider of `Manual` in the `Approval` category. You can include a custom message and an SNS topic for notifications.

Example approval action:

```json
{
  "name": "ApproveProd",
  "actions": [
    {
      "name": "ManualApproval",
      "actionTypeId": {
        "category": "Approval",
        "owner": "AWS",
        "provider": "Manual",
        "version": "1"
      },
      "runOrder": 1,
      "configuration": {
        "CustomData": "Review test results, change request, and security scan output before approving.",
        "NotificationArn": "arn:aws:sns:us-east-1:123456789012:PipelineApprovals"
      }
    }
  ]
}
```

Approval actions have a timeout window. If no one approves before expiration, the action fails and the pipeline stops. The approval status is visible in the CodePipeline UI, where you can approve or reject directly from the stage view.

- **CustomData** gives reviewers context.
- **NotificationArn** sends approval requests to SNS subscribers.
- **Timeout/expiration** prevents approvals from lingering indefinitely.
- **UI status** shows pending, approved, rejected, or expired states.

A practical design pattern is to place automated checks before the manual approval. For example, a Lambda function or CodeBuild job can run policy validation, integration tests, or security scans, and only then allow the approval stage to appear.

- **Lambda checks** can validate release metadata or query external systems.
- **CodeBuild checks** can run tests, linting, and vulnerability scans.
- **Manual approval** becomes the final human gate, not the first line of defense.

**Automated checks before approval**

You can use Lambda or CodeBuild as a pre-approval gate to reduce human review burden. This is useful when the approval should only happen if the artifact meets objective criteria.

Examples of automated checks:

- **Unit and integration tests** in CodeBuild.
- **Static analysis** or dependency scanning.
- **Change-set validation** for CloudFormation deployments.
- **Policy checks** against release metadata or environment rules.

A simple Lambda-based validation step might inspect a manifest and fail the stage if the artifact hash does not match the expected value:

```python
import json

def lambda_handler(event, context):
    manifest = event.get("artifact", {})
    expected = manifest.get("sha256")
    actual = manifest.get("verifiedSha256")
    if expected != actual:
        raise Exception("Artifact integrity check failed")
    return {"status": "ok"}
```

- **Automated gates** reduce approval fatigue.
- **Human approval** is still useful for business or risk decisions.
- **Fail fast** before production if validation is deterministic.

**Secure artifact patterns for S3**

S3 is often used for build outputs, deployment bundles, and release manifests. Secure promotion depends on both access control and object lifecycle design.

Recommended patterns:

- **Enable versioning** on the artifact bucket.
- **Use SSE-KMS** (server-side encryption with KMS) for sensitive artifacts.
- **Restrict write access** to the build role only.
- **Restrict read access** to deployment roles and pipeline roles.
- **Use object tags or prefixes** to separate environments or release channels.
- **Consider Object Lock** for regulated or high-assurance release archives.

Example bucket encryption configuration:

```json
{
  "Rules": [
    {
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "aws:kms",
        "KMSMasterKeyID": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555"
      }
    }
  ]
}
```

- **Versioning** supports rollback and audit.
- **SSE-KMS** gives key-level control and CloudTrail visibility.
- **Object Lock** helps prevent tampering after release.
- **Least privilege** matters more than broad bucket access.

**Application artifact encryption versus pipeline encryption**

These are related but different concerns.

- **Pipeline artifact store encryption** protects the files moving between pipeline stages.
- **Application artifact encryption** protects the payload itself, such as a ZIP file, JAR, container image, or package.
- **Data encryption at runtime** protects secrets, databases, and user data after deployment.

For example, a ZIP file stored in S3 may be encrypted with SSE-KMS, but the application inside the ZIP may still need its own secret management strategy using AWS Secrets Manager or AWS Systems Manager Parameter Store.

> If a question asks about securing the pipeline, think artifact store, IAM, KMS, and approvals. If it asks about securing the application, think secrets, runtime encryption, and application-specific controls.

**AWS Signer and integrity verification**

AWS Signer is an AWS-native signing service used to sign code artifacts and verify integrity before deployment. It is especially relevant when you need to prove that an artifact has not been altered after signing.

Common uses include:

- **Lambda deployment packages**
- **Container images**
- **Code artifacts** that require trusted provenance

A typical flow is:

1. Build the artifact in CodeBuild.
2. Sign the artifact with AWS Signer.
3. Store or publish the signed artifact.
4. Verify the signature before deployment or at deployment time.

This is stronger than a generic checksum because the signature is tied to a trusted signing profile and key material.

- **Checksums** detect accidental corruption.
- **Signatures** provide provenance and tamper evidence.
- **Signer** is the AWS-native service to remember for exam questions.

**CodeArtifact as a promotion target and source**

AWS CodeArtifact is useful when your application is package-based rather than image-based. It acts as a managed package repository for npm, Maven, PyPI, NuGet, and other ecosystems.

In promotion workflows, CodeArtifact can serve as:

- **A source** for dependencies during build.
- **A target** for publishing internal packages after validation.
- **A promotion boundary** between dev and prod repositories or domains.

A common pattern is to publish a package to a development repository, run tests, and then promote the same package version to a production repository or a separate repository in another account.

Example npm configuration for CodeArtifact:

```bash
aws codeartifact login --tool npm --domain my-domain --domain-owner 123456789012 --repository my-repo
npm publish
```

Example Maven settings often point to a repository endpoint with temporary auth token usage:

```bash
aws codeartifact login --tool maven --domain my-domain --domain-owner 123456789012 --repository my-repo
mvn deploy
```

- **Package promotion** is common for libraries and shared services.
- **Repository separation** can mirror environment boundaries.
- **Immutable package versions** help prevent accidental overwrite.
- **Dependency resolution** should be controlled to avoid pulling unapproved versions.

**Container artifact promotion with ECR**

For containerized applications, Amazon ECR is the usual artifact store. Promotion often means moving the same image digest across environments rather than rebuilding the image.

Useful patterns include:

- **Tagging by digest** for immutability.
- **Promoting tags** such as `dev`, `test`, and `prod` to point to approved images.
- **ECR replication** for cross-region or cross-account distribution.
- **Image scanning** before promotion when security policy requires it.

Example image push and tag flow:

```bash
docker build -t myapp:1.2.3 .
docker tag myapp:1.2.3 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:1.2.3
docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:1.2.3
```

If you need cross-region promotion, ECR replication can automatically copy images to another region or account. This is often preferable to rebuilding in each region because it preserves the exact image digest.

- **Image digest** is the strongest identifier for a container artifact.
- **Replication** supports multi-region release patterns.
- **Scanning** can be used as an automated gate before approval.
- **Tags alone** are mutable, so do not rely on them as the only source of truth.

**Practical pipeline topology**

A typical secure promotion pipeline might look like this:

1. Source stage pulls from CodeCommit, GitHub, or S3.
2. Build stage compiles and tests in CodeBuild.
3. Package stage publishes to S3, ECR, or CodeArtifact.
4. Validation stage runs automated checks in Lambda or CodeBuild.
5. Manual approval stage waits for human review.
6. Deploy stage promotes to staging or production.

Example conceptual flow:

```text
Source -> Build -> Test -> Sign -> Store -> Validate -> Approve -> Deploy
```

- **Sign** before distribution when integrity matters.
- **Store** in immutable or versioned repositories.
- **Validate** with automated policy checks.
- **Approve** only after objective criteria pass.

**Exam-oriented distinctions**

The exam often asks you to choose the best control for a specific risk. Focus on the control that matches the problem.

| Scenario | Best-fit control |
|---|---|
| Need human review before prod | Manual approval action in CodePipeline |
| Need artifact integrity and provenance | AWS Signer |
| Need pipeline file encryption | CodePipeline artifact store with KMS |
| Need rollback of prior build output | S3 versioning |
| Need package promotion | CodeArtifact |
| Need container promotion across regions | ECR replication |
| Need cross-account deployment | Separate artifact bucket, IAM trust, KMS policy, action role |

- **Manual approval** is for business or operational judgment.
- **Signer** is for tamper evidence and trusted provenance.
- **Versioning** is for rollback and traceability.
- **Replication** is for distributing the same container artifact.

**Troubleshooting and common pitfalls**

Many pipeline failures come from permissions or encryption mismatches rather than the deployment logic itself.

- **KMS access denied** usually means the key policy does not allow the pipeline role or target role to use the key.
- **S3 access denied** often means the bucket policy or object ACL is too restrictive.
- **Cross-account action failure** often means the action role trust policy is incomplete.
- **Approval never completes** if the action expires before a reviewer responds.
- **Unexpected rebuilds** can introduce drift if you do not promote the same artifact.

> If a question mentions “same artifact across environments,” avoid answers that rebuild per environment unless the prompt explicitly requires environment-specific compilation.

**Quick reference commands**

Use these commands when working with artifact repositories and promotion flows:

```bash
# Inspect an ECR image digest
aws ecr describe-images --repository-name myapp --image-ids imageTag=1.2.3

# Upload a versioned artifact to S3
aws s3 cp build/output.zip s3://my-artifacts/releases/output-1.2.3.zip

# Enable CodeArtifact login for npm
aws codeartifact login --tool npm --domain my-domain --domain-owner 123456789012 --repository my-repo

# Start a CodePipeline execution
aws codepipeline start-pipeline-execution --name my-pipeline
```

**What to remember for the exam**

- **Promotion** means moving the same verified artifact through stages.
- **Approvals** in CodePipeline can include SNS notifications, custom messages, and expiration behavior.
- **Artifact store encryption** is separate from application payload encryption.
- **Cross-account pipelines** need aligned S3, KMS, and IAM trust policies.
- **AWS Signer** is the AWS-native integrity and signing service to recognize.
- **CodeArtifact** is the package promotion answer for dependency-based applications.
- **ECR replication** is the key container promotion pattern across regions or accounts.
- **S3 versioning** and, when needed, Object Lock support rollback and traceability.
