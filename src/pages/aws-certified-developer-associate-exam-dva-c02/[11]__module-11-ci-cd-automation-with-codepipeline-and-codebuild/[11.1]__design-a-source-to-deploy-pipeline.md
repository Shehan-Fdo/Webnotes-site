## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Pipeline design goals**

A source-to-deploy pipeline automates the path from code commit to production release. On AWS, the core services are **CodePipeline** for orchestration, **CodeBuild** for build and test, **CodeDeploy** for application deployment, and **CloudFormation** for infrastructure delivery. In exam scenarios, you usually choose the service combination that best matches the target platform, release strategy, and security boundary.

A good design balances:
- Fast feedback for developers
- Repeatable deployments
- Least-privilege access
- Artifact integrity and encryption
- Clear rollback behavior

**Core pipeline stages and action categories**

CodePipeline organizes work into stages, and each stage contains one or more actions. The action category tells you what kind of work the action performs, while the provider tells you which AWS service or external integration executes it.

Common action categories include:
- **Source**: Retrieves code or artifacts from a repository or connection
- **Build**: Compiles code, runs tests, packages output
- **Test**: Runs validation or quality checks, often with a separate tool
- **Deploy**: Releases to EC2, ECS, Lambda, Elastic Beanstalk, or CloudFormation
- **Approval**: Pauses the pipeline for manual review
- **Invoke**: Calls a Lambda function or other integrated action for custom automation

Typical providers you should recognize:
- **Source**: CodeCommit, CodeStar Connections, Amazon S3, GitHub via Connections
- **Build/Test**: CodeBuild
- **Deploy**: CodeDeploy, CloudFormation, Elastic Beanstalk, Amazon ECS, AWS Lambda
- **Approval**: Manual approval action
- **Invoke**: Lambda

A stage can contain multiple actions, but actions only run in parallel when they share the same `runOrder` and do not depend on each other’s output. If one action needs the output of another, give it a higher `runOrder`.

**Common end-to-end patterns**

The exam often tests which pipeline pattern fits a workload.

- **CodeCommit → CodeBuild → CodeDeploy**
  - Best for application code that deploys to EC2 or Lambda through CodeDeploy.
  - Use when you need build, test, package, and controlled rollout.

- **GitHub/CodeStar Connections → CodeBuild → CloudFormation**
  - Best for infrastructure as code.
  - Use when the pipeline should update stacks, nested stacks, or change sets.

- **CodeCommit → CodeBuild → Amazon ECS**
  - Best for containerized applications.
  - Build a container image, push to Amazon ECR, then update the ECS service.

- **Source → CodeBuild → Elastic Beanstalk**
  - Best for simpler web app deployments where Beanstalk manages the environment.

- **Source → CodeBuild → Manual approval → Deploy**
  - Best for production promotion with human review.

**CodePipeline behavior and execution modes**

CodePipeline supports different execution behaviors that matter in real deployments and exam questions.

- **Queued**
  - New executions wait until the current execution finishes.
  - Useful when you want strict serialization.

- **Superseded**
  - A newer execution can replace an older in-progress execution.
  - Useful for fast-moving branches where only the latest commit matters.

- **Parallel**
  - Multiple executions can run at the same time.
  - Useful when each commit must be tested independently.

Pipeline updates can affect executions. If you change the pipeline definition while executions are running, AWS may continue the existing execution with the old structure or apply the updated structure depending on the change and execution state. In practice, design for predictable promotion and avoid frequent structural changes during critical releases.

**Source actions and repository integration**

Source is the first stage in most pipelines. Your choice depends on where code lives and how you want to trigger the pipeline.

- **CodeCommit**
  - Native AWS Git repository service.
  - Works well with IAM and EventBridge triggers.
  - Good for fully AWS-native workflows.

- **CodeStar Connections**
  - Integrates with GitHub, GitHub Enterprise, Bitbucket, and similar providers through a connection.
  - Preferred for modern external source integrations.
  - Supports webhook-style change detection through the connection.

- **Amazon S3**
  - Useful for packaged artifacts, templates, or manually uploaded source bundles.
  - Common in infrastructure pipelines or when source is generated elsewhere.

Practical CodeCommit trigger example:
```bash
aws codecommit put-repository-triggers \
  --repository-name my-app \
  --triggers '[{
    "name":"pipeline-trigger",
    "destinationArn":"arn:aws:events:us-east-1:123456789012:rule/CodeCommitPushRule",
    "branches":["main"],
    "events":["all"]
  }]'
```

EventBridge is often used to react to repository events and start downstream automation. For example, a push to `main` can trigger a pipeline execution or a custom Lambda function.

CodeStar Connections example for a source action:
```json
{
  "name": "Source",
  "actionTypeId": {
    "category": "Source",
    "owner": "AWS",
    "provider": "CodeStarSourceConnection",
    "version": "1"
  },
  "outputArtifacts": [
    { "name": "SourceOutput" }
  ],
  "configuration": {
    "ConnectionArn": "arn:aws:codestar-connections:us-east-1:123456789012:connection/abc123",
    "FullRepositoryId": "my-org/my-repo",
    "BranchName": "main",
    "DetectChanges": "true"
  },
  "runOrder": 1
}
```

**Build and test with CodeBuild**

CodeBuild compiles code, runs tests, creates deployable artifacts, and can publish reports. It is the standard build engine in AWS pipelines.

A buildspec defines the build lifecycle. The most important sections are:
- `version`
- `phases`
- `artifacts`
- `cache`
- `reports`
- `env`

Example buildspec:
```yaml
version: 0.2

env:
  variables:
    APP_NAME: my-app
  exported-variables:
    - IMAGE_TAG

phases:
  install:
    runtime-versions:
      nodejs: 20
    commands:
      - npm ci
  pre_build:
    commands:
      - npm test
      - IMAGE_TAG=$(echo $CODEBUILD_RESOLVED_SOURCE_VERSION | cut -c 1-7)
  build:
    commands:
      - npm run build
      - docker build -t my-app:$IMAGE_TAG .
  post_build:
    commands:
      - docker tag my-app:$IMAGE_TAG 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:$IMAGE_TAG
      - docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:$IMAGE_TAG

artifacts:
  files:
    - '**/*'
  base-directory: dist

reports:
  junit-reports:
    files:
      - 'test-results/*.xml'
    file-format: JUNITXML

cache:
  paths:
    - 'node_modules/**/*'
```

Key CodeBuild concepts:
- **Phases** run in order and let you separate setup, validation, build, and packaging.
- **Exported variables** can pass values to later CodePipeline actions.
- **Reports** publish test results and code coverage.
- **Cache** speeds up repeated builds by reusing dependencies.
- **Artifacts** are the files passed to later stages.

A CodeBuild project used in CodePipeline commonly uses `CODEPIPELINE` as the artifact type. The project still needs a valid source configuration and service role. A minimal project definition is not enough for production use.

Example project skeleton:
```json
{
  "name": "my-build-project",
  "serviceRole": "arn:aws:iam::123456789012:role/CodeBuildServiceRole",
  "artifacts": {
    "type": "CODEPIPELINE"
  },
  "environment": {
    "type": "LINUX_CONTAINER",
    "image": "aws/codebuild/standard:7.0",
    "computeType": "BUILD_GENERAL1_SMALL",
    "privilegedMode": true
  },
  "source": {
    "type": "CODEPIPELINE",
    "buildspec": "buildspec.yml"
  }
}
```

**Artifacts, encryption, and S3 stores**

CodePipeline uses an artifact store, usually an S3 bucket, to move files between stages. This is a common exam topic because it touches security, cross-region delivery, and cross-account access.

Important artifact store considerations:
- Enable **S3 bucket versioning** for reliability and traceability.
- Use **SSE-KMS** (server-side encryption with AWS Key Management Service) when you need stronger control over artifact access.
- Ensure the bucket policy allows the pipeline role and downstream service roles to read and write artifacts.
- For cross-region pipelines, define a separate artifact store per region.
- For cross-account pipelines, the bucket and KMS key policy must allow the consuming account roles to use them.

A practical S3 bucket policy and KMS key policy often need to allow:
- CodePipeline service role
- CodeBuild service role
- CodeDeploy or CloudFormation execution role
- Cross-account action roles, if used

> In cross-account designs, do not assume the pipeline can automatically use every target role. You usually need explicit trust policies, artifact bucket permissions, and KMS key permissions on both sides.

**IAM and least privilege**

Each service in the pipeline should have its own service role.

- **CodePipeline service role**
  - Reads source artifacts
  - Writes and reads from artifact stores
  - Invokes downstream actions
  - Needs permission to pass roles where required

- **CodeBuild service role**
  - Reads source artifacts
  - Writes build artifacts
  - Pulls from ECR if building containers
  - Writes logs to CloudWatch Logs
  - Accesses Secrets Manager or Parameter Store if needed

- **CodeDeploy service role**
  - Reads deployment bundles
  - Updates target instances, Lambda aliases, or ECS services
  - Interacts with Auto Scaling and CloudWatch alarms for rollback

- **CloudFormation execution role**
  - Creates and updates stack resources
  - Must be scoped to the resources the pipeline manages

Use least privilege:
- Restrict S3 access to specific buckets and prefixes.
- Restrict KMS permissions to required keys.
- Restrict `iam:PassRole` to specific roles and services.
- Use separate roles for build, deploy, and approval automation.

**Deployment targets and service-specific behavior**

**CodeDeploy for EC2 and on-premises**

CodeDeploy supports in-place and blue/green deployments for EC2 and on-premises instances.

- **In-place deployment**
  - Updates instances directly.
  - Simpler and cheaper.
  - Suitable when brief downtime is acceptable or the app can tolerate rolling updates.

- **Blue/green deployment**
  - Deploys to a replacement environment.
  - Shifts traffic after validation.
  - Better for safer releases and rollback.

CodeDeploy uses an AppSpec file and lifecycle hooks. For EC2, the AppSpec file maps files and hook scripts.

Example EC2 AppSpec:
```yaml
version: 0.0
os: linux
files:
  - source: /
    destination: /var/www/myapp
hooks:
  BeforeInstall:
    - location: scripts/before_install.sh
      timeout: 300
      runas: root
  AfterInstall:
    - location: scripts/after_install.sh
      timeout: 300
      runas: root
  ApplicationStart:
    - location: scripts/start_app.sh
      timeout: 300
      runas: root
  ValidateService:
    - location: scripts/validate.sh
      timeout: 300
      runas: root
```

Lifecycle order matters:
1. `BeforeInstall`
2. `AfterInstall`
3. `ApplicationStart`
4. `ValidateService`

For blue/green deployments, CodeDeploy can shift traffic through an Elastic Load Balancer and use alarms to trigger rollback.

**CodeDeploy for Lambda**

Lambda deployments use aliases and traffic shifting. The AppSpec file is different from EC2 and does not use CloudFormation-style resource blocks.

Example Lambda AppSpec:
```yaml
version: 0.0
Resources:
  - myLambdaFunction:
      Type: AWS::Lambda::Function
      Properties:
        Name: my-function
        Alias: live
        CurrentVersion: 1
        TargetVersion: 2
Hooks:
  BeforeAllowTraffic: preTrafficHook
  AfterAllowTraffic: postTrafficHook
```

Common predefined deployment configurations:
- `CodeDeployDefault.LambdaAllAtOnce`
- `CodeDeployDefault.LambdaCanary10Percent5Minutes`
- `CodeDeployDefault.LambdaCanary10Percent10Minutes`
- `CodeDeployDefault.LambdaLinear10PercentEvery1Minute`
- `CodeDeployDefault.LambdaLinear10PercentEvery2Minutes`

Use canary when you want a small percentage of traffic to validate the new version before full rollout. Use linear when you want gradual traffic shifting over time.

**CodeDeploy for ECS**

For Amazon ECS, CodeDeploy supports blue/green deployments with task sets and load balancer traffic shifting.

Key ideas:
- The new task set starts alongside the old one.
- Traffic shifts from the original target group to the replacement target group.
- Hooks and alarms can validate the deployment.
- Rollback can occur if health checks or CloudWatch alarms fail.

**Elastic Beanstalk integration**

Elastic Beanstalk can be a deployment target in CodePipeline when you want managed application environments.

Use it when:
- You want AWS to manage EC2, load balancing, and scaling details.
- You need a simpler deployment target than hand-managed EC2 instances.
- You are deploying web applications with standard environment settings.

A pipeline can package application source, build it, and then deploy the artifact to a Beanstalk environment.

**CloudFormation in pipelines**

CloudFormation is the standard choice for infrastructure as code in a pipeline. It can create or update stacks, nested stacks, and change sets.

Common CloudFormation pipeline patterns:
- **Create or update stack**
  - Directly applies the template.
  - Simple, but less reviewable.

- **Create change set**
  - Generates a proposed update.
  - Lets you inspect changes before execution.
  - Often paired with a manual approval action.

Useful CloudFormation action details:
- Pass template files and parameter files as artifacts.
- Use `Capabilities` such as `CAPABILITY_IAM` or `CAPABILITY_NAMED_IAM` when the template creates IAM resources.
- Use stack parameters or parameter overrides for environment-specific values.
- Nested stacks help break large deployments into reusable pieces.

Example deploy command for a stack update:
```bash
aws cloudformation deploy \
  --stack-name my-app-stack \
  --template-file template.yaml \
  --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM \
  --parameter-overrides Environment=prod ImageTag=v1.2.3
```

If your template requires parameters, include `--parameter-overrides`. Omitting them can cause failures or unintended defaults.

**Manual approval actions**

Approval actions are used to pause a pipeline before production deployment or another sensitive step.

Use approval when:
- A human must review test results, change sets, or security findings
- You need a release gate between staging and production
- Compliance requires explicit sign-off

Limitations:
- Approval is manual, not automated.
- It does not run code.
- It is best placed after validation and before production deploy.

A common pattern is:
1. Source
2. Build
3. Test
4. Manual approval
5. Deploy to production

**Cross-account and cross-region design**

Cross-account pipelines are common in enterprise AWS environments. The source account may host the pipeline, while build or deploy targets live in separate accounts.

Typical mechanics:
- The pipeline account stores the main pipeline.
- The artifact bucket and KMS key must allow access from target accounts.
- Action roles in the target account must trust the pipeline account or the pipeline service.
- The deploy action often assumes a role in the target account with permissions to update the target resource.

Cross-region pipelines add another layer:
- Each region needs its own artifact store.
- Artifacts may need replication or separate packaging.
- The pipeline must reference the correct regional action providers and stores.

Practical trust policy pattern for a target role:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "codepipeline.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

In real designs, you often scope this further with `aws:SourceArn` and account conditions.

**Security and operational best practices**

- Use separate branches or environments for dev, test, and prod.
- Protect production with manual approval and change sets.
- Encrypt artifacts with KMS.
- Store secrets in Secrets Manager or Systems Manager Parameter Store, not in source.
- Use CloudWatch Logs for build diagnostics.
- Use CloudWatch alarms for deployment rollback triggers.
- Tag resources for cost allocation and ownership.
- Keep pipeline definitions in version control.

Operationally, you should also:
- Monitor failed executions and action retries.
- Review artifact retention and bucket lifecycle policies.
- Validate IAM permissions after every major pipeline change.
- Test rollback paths before production rollout.

**Practical pipeline examples**

**Example 1: CodeCommit to CodeDeploy for EC2**

Flow:
1. Developer pushes to CodeCommit.
2. CodePipeline starts on the `main` branch.
3. CodeBuild runs unit tests and packages the app.
4. CodeDeploy copies the revision to EC2 instances.
5. Lifecycle hooks validate the service before traffic is considered healthy.

Use this when you need controlled application rollout on EC2.

**Example 2: GitHub to CodeBuild to CloudFormation**

Flow:
1. GitHub push triggers CodeStar Connections.
2. CodeBuild validates templates and runs tests.
3. CloudFormation creates a change set.
4. Manual approval reviews the change set.
5. CloudFormation executes the stack update.

Use this when infrastructure changes must be reviewed before deployment.

**Example 3: Source to CodeBuild to ECS**

Flow:
1. Source action retrieves application code.
2. CodeBuild builds a container image and pushes it to ECR.
3. CodePipeline updates the ECS service with the new image tag.
4. ECS performs a rolling or blue/green deployment depending on configuration.

Use this for containerized microservices.

**Example 4: Source to CodeBuild to Elastic Beanstalk**

Flow:
1. Source action retrieves the application bundle.
2. CodeBuild runs tests and packages the artifact.
3. Elastic Beanstalk deploys the application version to the environment.

Use this for managed web applications with minimal infrastructure management.

**How to choose the right AWS service**

| Need | Best fit | Why |
|---|---|---|
| Orchestrate multi-step release | CodePipeline | Native pipeline control and stage gating |
| Compile and test code | CodeBuild | Managed build environment and reports |
| Deploy to EC2/Lambda/ECS | CodeDeploy | Traffic shifting, hooks, rollback |
| Deploy infrastructure | CloudFormation | Declarative stack updates and change sets |
| Managed web app hosting | Elastic Beanstalk | Simplified environment management |
| External Git provider integration | CodeStar Connections | Secure source integration |

**Exam tips**

- Choose **CodePipeline** when the question asks for orchestration across multiple AWS services.
- Choose **CodeBuild** when the question asks for build, test, packaging, or container image creation.
- Choose **CodeDeploy** when the question asks for controlled application rollout, hooks, or traffic shifting.
- Choose **CloudFormation** when the question asks for infrastructure deployment or change sets.
- Choose **CodeStar Connections** when the source is GitHub or another external Git provider.
- Look for **manual approval** when the scenario requires human review before production.
- Remember that **artifact encryption** and **cross-account permissions** are often the hidden constraint in advanced questions.

**Troubleshooting signals**

- Build fails early: check buildspec syntax, runtime versions, and service role permissions.
- Deploy fails on EC2: check AppSpec paths, hook scripts, instance profile, and CodeDeploy agent.
- CloudFormation action fails: check parameters, capabilities, and stack events.
- Source action does not trigger: check branch filters, connection status, and EventBridge rules.
- Cross-account access fails: check bucket policy, KMS key policy, and role trust relationships.

**Quick reference**

- **CodePipeline**: orchestration and stage control
- **CodeBuild**: build, test, package, reports
- **CodeDeploy**: application deployment and traffic shifting
- **CloudFormation**: infrastructure as code
- **CodeStar Connections**: external source integration
- **EventBridge**: event-driven triggers and automation
- **S3 artifact store**: pipeline artifact transport and encryption
