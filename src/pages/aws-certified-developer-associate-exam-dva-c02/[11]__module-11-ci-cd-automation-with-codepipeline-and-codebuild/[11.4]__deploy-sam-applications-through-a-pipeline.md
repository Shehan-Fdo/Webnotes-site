## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Pipeline deployment overview**

AWS Serverless Application Model (<abbr title="AWS Serverless Application Model">SAM</abbr>) is a framework for building and deploying serverless applications with CloudFormation-compatible templates. In a pipeline, SAM usually fits into a standard continuous integration and continuous delivery (<abbr title="Continuous Integration / Continuous Delivery">CI/CD</abbr>) flow:

- **Source**: code is committed to a repository such as CodeCommit, GitHub, or Bitbucket.
- **Build**: `sam build` compiles dependencies and prepares deployment artifacts.
- **Package**: artifacts are uploaded to Amazon S3 and template references are rewritten.
- **Deploy**: CloudFormation creates or updates the stack, often through a change set.
- **Approve**: optional manual approval gates production promotion.

For exam purposes, remember that `sam deploy` is a convenience wrapper around CloudFormation stack updates and change sets. It does not replace the need to prepare artifacts correctly, and source-based projects typically require a prior `sam build`.

**Core SAM commands in a pipeline**

Use the right command for the right stage. The distinction matters because pipeline failures often come from mixing local developer workflows with automated delivery workflows.

- `sam build`
  - Resolves dependencies and creates a build output directory, usually `.aws-sam/build`.
  - Use it before packaging or deploying source-based applications.
  - Run it in the build stage, not in the deploy stage, when you want deterministic artifacts.
- `sam package`
  - Uploads local artifacts to S3 and rewrites the template to point to those uploaded artifacts.
  - Useful when you want an explicit packaging step before deployment.
  - Often paired with `aws cloudformation deploy` or `aws cloudformation package`-style workflows.
- `sam deploy`
  - Deploys the application by creating or updating a CloudFormation stack.
  - Can upload artifacts for you when used with options such as `--resolve-s3` and `--resolve-image-repos`.
  - Commonly used in CI/CD with `--no-confirm-changeset` and `--no-fail-on-empty-changeset`.

A practical local-to-pipeline sequence looks like this:

```bash
sam build
sam deploy \
  --stack-name my-sam-app-dev \
  --region us-east-1 \
  --capabilities CAPABILITY_IAM \
  --no-confirm-changeset \
  --no-fail-on-empty-changeset \
  --resolve-s3
```

> `CAPABILITY_IAM` is required when the stack creates or modifies IAM resources. Use `CAPABILITY_NAMED_IAM` only when those IAM resources have explicit custom names.

**How SAM fits into CodeBuild and CodePipeline**

A common AWS-native pipeline uses AWS CodePipeline for orchestration and AWS CodeBuild for build and deploy actions. CodePipeline passes artifacts between stages using an artifact store in Amazon S3. The build stage produces a packaged template and deployment artifacts; later stages consume those outputs.

A typical flow is:

1. Source stage retrieves the repository revision and emits a source artifact.
2. Build stage runs `sam build` and then packages the application.
3. Deploy stage consumes the packaged template and deploys to a target stack.
4. Approval stage optionally pauses before production deployment.

A minimal `buildspec.yml` for SAM packaging might look like this:

```yaml
version: 0.2

phases:
  install:
    runtime-versions:
      python: 3.11
    commands:
      - python --version
      - pip install --upgrade pip
      - pip install aws-sam-cli
  build:
    commands:
      - sam build
      - sam package --s3-bucket "$ARTIFACT_BUCKET" --output-template-file packaged.yaml
artifacts:
  files:
    - packaged.yaml
    - template.yaml
```

In practice, many teams use a CodeBuild image that already includes SAM CLI rather than installing it every run. That approach is usually more repeatable and easier to govern.

**End-to-end pipeline example**

A full pipeline usually separates build and deploy concerns. The build stage creates immutable artifacts, and the deploy stage references those artifacts rather than rebuilding from source.

Example CodePipeline structure:

```yaml
Resources:
  Pipeline:
    Type: AWS::CodePipeline::Pipeline
    Properties:
      RoleArn: arn:aws:iam::123456789012:role/CodePipelineServiceRole
      ArtifactStore:
        Type: S3
        Location: my-pipeline-artifacts-bucket
      Stages:
        - Name: Source
          Actions:
            - Name: SourceAction
              ActionTypeId:
                Category: Source
                Owner: AWS
                Provider: CodeCommit
                Version: '1'
              OutputArtifacts:
                - Name: SourceOutput
              Configuration:
                RepositoryName: my-sam-repo
                BranchName: main

        - Name: Build
          Actions:
            - Name: BuildAction
              ActionTypeId:
                Category: Build
                Owner: AWS
                Provider: CodeBuild
                Version: '1'
              InputArtifacts:
                - Name: SourceOutput
              OutputArtifacts:
                - Name: BuildOutput
              Configuration:
                ProjectName: my-sam-build-project

        - Name: Approve
          Actions:
            - Name: ManualApproval
              ActionTypeId:
                Category: Approval
                Owner: AWS
                Provider: Manual
                Version: '1'
              RunOrder: 1

        - Name: Deploy
          Actions:
            - Name: DeployAction
              ActionTypeId:
                Category: Deploy
                Owner: AWS
                Provider: CloudFormation
                Version: '1'
              InputArtifacts:
                - Name: BuildOutput
              Configuration:
                ActionMode: CREATE_UPDATE
                StackName: my-sam-app-prod
                TemplatePath: BuildOutput::packaged.yaml
                Capabilities: CAPABILITY_IAM
```

This pattern is important because the deploy action consumes the packaged template from the pipeline artifact store. The deploy stage does not need access to the original source repository if the build stage already produced the final template and artifacts.

**Artifact handling and packaging**

SAM applications often include Lambda function code, layers, and container images. The pipeline must make those artifacts available to CloudFormation in a way that the deploy stage can reference.

- **S3-based artifacts**
  - `sam package` uploads ZIP files and rewrites the template with S3 object references.
  - The deploy stage uses the packaged template, not the raw source tree.
- **Container image artifacts**
  - Image-based SAM resources use Amazon Elastic Container Registry (<abbr title="Amazon Elastic Container Registry">ECR</abbr>).
  - The pipeline must build the image and push it to ECR before deployment, or use SAM options that handle image repository resolution.
  - The deploy role needs permission to access the repository and update the stack.

Example image-based workflow:

```bash
sam build --use-container
sam deploy \
  --stack-name my-image-app \
  --capabilities CAPABILITY_IAM \
  --resolve-image-repos \
  --no-confirm-changeset \
  --no-fail-on-empty-changeset
```

If you manage image repositories explicitly, create them ahead of time or let SAM resolve them during deployment when supported by your workflow.

**Change sets and approvals**

CloudFormation change sets are a key exam topic because they let you preview stack changes before execution. SAM deploy uses CloudFormation under the hood, so change sets fit naturally into a pipeline approval model.

- A change set shows what will be added, modified, or removed.
- A manual approval action can review the change set before production deployment.
- This is useful when you want to catch unintended IAM, networking, or resource replacement changes.

A common pattern is:

1. Build and package the application.
2. Create a change set for the target stack.
3. Pause for approval.
4. Execute the change set after approval.

For example, a CloudFormation deploy action in CodePipeline can create and execute a change set automatically, while a manual approval stage sits between build and deploy. If you need more control, you can also use a separate script step to create the change set and inspect it before execution.

**Cross-account deployment**

Cross-account delivery is common when you deploy from a central tooling account into dev, test, and prod accounts. The key idea is role assumption and trust relationships.

- The **pipeline role** in the tooling account needs permission to invoke CodeBuild, read/write artifacts, and assume deployment roles.
- The **deployment role** in the target account trusts the pipeline account or a specific pipeline role.
- The deployment role needs permissions for CloudFormation, S3, IAM, Lambda, API Gateway, and any other resources the stack manages.

A simplified trust relationship looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111111111111:role/CodePipelineServiceRole"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

In exam scenarios, look for:
- Separate accounts for source, build, and deploy.
- A deployment role in the target account.
- An artifact bucket that is accessible to the pipeline and deployment actions.
- KMS permissions if artifacts are encrypted.

**Safe deployment strategies**

Serverless applications often need safer rollout patterns than a direct all-at-once update. SAM supports deployment preferences for Lambda functions through AWS CodeDeploy integration.

- **Canary** deployments shift a small percentage of traffic first, then complete the rollout if health checks pass.
- **Linear** deployments shift traffic in increments over time.
- **All-at-once** deployments move traffic immediately and are simpler but riskier.

Example SAM function configuration:

```yaml
Resources:
  MyFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: src/
      Handler: app.handler
      Runtime: python3.11
      AutoPublishAlias: live
      DeploymentPreference:
        Type: Canary10Percent5Minutes
        Alarms:
          - !Ref FunctionErrorAlarm
```

This is especially relevant in pipelines because the deployment stage can succeed while traffic shifting continues afterward. If alarms trigger, CodeDeploy can roll back automatically.

**Bootstrap and prerequisites**

A pipeline cannot deploy successfully unless the required infrastructure exists.

- The artifact bucket must exist and be accessible.
- ECR repositories must exist or be resolvable for image-based apps.
- IAM roles must allow CodePipeline, CodeBuild, CloudFormation, and SAM-related actions.
- KMS keys, if used, must allow encryption and decryption across the pipeline stages.
- The target stack name and region must match the deployment environment.

For guided local deployment, `sam deploy --guided` stores preferences in `samconfig.toml`. That is useful for developer setup, but it is not the normal choice for non-interactive pipelines. In CI/CD, prefer explicit parameters and environment variables.

Example `samconfig.toml`:

```toml
version = 0.1

[default.deploy.parameters]
stack_name = "my-sam-app-dev"
region = "us-east-1"
capabilities = "CAPABILITY_IAM"
resolve_s3 = true
confirm_changeset = false
fail_on_empty_changeset = false
```

**When to use `sam build`, `sam package`, and `sam deploy`**

Use the command that matches the stage and artifact model.

| Command | Best use | Pipeline note |
|---|---|---|
| `sam build` | Prepare source code and dependencies | Usually runs in CodeBuild |
| `sam package` | Upload artifacts and rewrite template | Useful when you want explicit artifact promotion |
| `sam deploy` | Create or update the stack | Often used with `--no-confirm-changeset` in automation |

A practical rule:
- Use `sam build` for source-based projects.
- Use `sam package` when you want a distinct packaging artifact for later deployment.
- Use `sam deploy` when you want SAM to handle the stack update flow directly, especially with `--resolve-s3` or `--resolve-image-repos`.

**Common exam pitfalls**

- **Guided mode** is for interactive setup, not automated pipelines.
- **CAPABILITY_IAM** covers IAM resources; **CAPABILITY_NAMED_IAM** is only needed when those IAM resources have explicit names.
- **Packaging** and **deployment** are related but not identical; a pipeline may separate them for control and approvals.
- **Image-based apps** need ECR handling, but SAM can help resolve repositories in the deployment flow.
- **Change sets** are useful when you want visibility before execution, especially for production.
- **Environment isolation** is safest when you use separate stacks and, often, separate accounts for different stages.

**Practical deployment checklist**

1. Build the application with `sam build`.
2. Package artifacts to S3 or ECR as needed.
3. Store the packaged template as a pipeline artifact.
4. Deploy with CloudFormation or `sam deploy` using explicit non-interactive flags.
5. Add manual approval before production.
6. Use change sets and alarms for safer releases.
7. Verify IAM capabilities and cross-account trust before promoting.

**Quick reference**

- **Source stage**: fetches code and starts the pipeline.
- **Build stage**: runs `sam build` and packages artifacts.
- **Deploy stage**: updates the CloudFormation stack.
- **Approval stage**: gates production promotion.
- **Artifact store**: holds packaged templates and build outputs.
- **Change set**: previews stack changes before execution.
- **DeploymentPreference**: enables canary or linear traffic shifting for Lambda.
