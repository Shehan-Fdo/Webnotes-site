## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Lambda execution roles: the core idea**

An AWS Lambda function does not use your IAM user credentials. Instead, the Lambda service assumes an **execution role** on your behalf and receives temporary security credentials through AWS Security Token Service (STS). Those credentials are short-lived and scoped by the role’s permissions, which is why execution roles are central to least-privilege design.

For the exam, separate three concepts clearly:

- **Trust policy**: who can assume the role.
- **Permissions policy**: what the role can do after it is assumed.
- **Resource-based policy**: who can invoke the Lambda function or access the function from another account or service.

A common exam trap is mixing up the role that Lambda assumes with the permissions needed by the service that triggers Lambda.

**Execution role and trust policy**

The execution role is an IAM role attached to the function. Its trust policy must allow the Lambda service principal to assume it.

Typical trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Key points:

- **STS** issues temporary credentials when Lambda assumes the role.
- The trust policy answers: “Who can assume this role?”
- The permissions policy answers: “What can the function do once assumed?”
- You usually do not grant users direct access to the execution role unless they are deploying or managing the function.

**Baseline permissions for logging**

The standard managed policy for CloudWatch Logs access is **AWSLambdaBasicExecutionRole**. This is the baseline policy you should expect in many exam scenarios because Lambda commonly needs to create log streams and write log events.

It typically covers:

- `logs:CreateLogGroup`
- `logs:CreateLogStream`
- `logs:PutLogEvents`

Use it when the function needs basic logging to CloudWatch Logs. If the function cannot write logs, debugging becomes much harder, and exam questions often hinge on recognizing this missing permission.

A common attachment pattern is:

- **AWSLambdaBasicExecutionRole** for logs
- Additional custom permissions for application-specific AWS API calls
- Optional VPC-related permissions when the function runs in a VPC

**Least privilege and permissions boundaries**

An execution role should be as narrow as possible. Give the function only the actions it needs, on only the resources it needs.

A **permissions boundary** is different from the execution role policy:

- The **execution role policy** grants the role its effective permissions.
- The **permissions boundary** sets the maximum permissions the role can ever have.
- The boundary does not grant permissions by itself; it limits them.

This distinction matters in delegated administration and least-privilege designs. For example, a platform team may allow developers to create Lambda roles, but attach a boundary that prevents them from granting broad access such as `s3:*` or `iam:*`.

Practical difference:

- If the execution role policy allows `dynamodb:GetItem`, the function can call it.
- If a permissions boundary denies or omits that action, the role cannot exceed the boundary even if the attached policy is broader.

Example boundary concept:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:*",
        "dynamodb:GetItem",
        "dynamodb:PutItem"
      ],
      "Resource": "*"
    }
  ]
}
```

Use boundaries when:

- Developers can create or update roles, but you want guardrails.
- You need to cap permissions in multi-team environments.
- You want to prevent privilege escalation through role attachment.

**Common managed policy combinations**

In exam scenarios, a Lambda function often needs a small set of managed policies or equivalent custom permissions.

Typical combinations:

- **AWSLambdaBasicExecutionRole** for CloudWatch Logs
- **AWSLambdaVPCAccessExecutionRole** for VPC networking support
- A custom policy for the target service, such as DynamoDB, S3, SQS, KMS, or Secrets Manager

The VPC policy is important when the function must create and manage elastic network interfaces in a VPC. In practice, the function also needs the logging policy, so the common pair is:

- **AWSLambdaBasicExecutionRole**
- **AWSLambdaVPCAccessExecutionRole**

If the function accesses encrypted resources or environment variables, add KMS permissions as needed.

**VPC-enabled functions**

When a Lambda function runs inside a VPC, it needs permissions to manage network interfaces. The service uses those permissions to attach the function to subnets and security groups.

Typical permissions include:

- `ec2:CreateNetworkInterface`
- `ec2:DescribeNetworkInterfaces`
- `ec2:DeleteNetworkInterface`

The managed policy **AWSLambdaVPCAccessExecutionRole** is the usual exam answer for this requirement.

Important operational note:

- VPC-enabled Lambda functions often need both the VPC access policy and the basic logging policy.
- If the function also calls AWS services, you still need those service-specific permissions in the execution role.

**Encrypted environment variables, Secrets Manager, and KMS**

If your function uses encrypted environment variables, AWS Key Management Service (KMS), or encrypted data in another service, the execution role may need additional permissions.

Common cases:

- **Encrypted environment variables**: Lambda decrypts them at runtime, so the role may need `kms:Decrypt` on the relevant key.
- **Secrets Manager**: the function may need `secretsmanager:GetSecretValue`.
- **Encrypted S3 objects or DynamoDB data**: the function may need KMS permissions if the data is encrypted with a customer managed key.

Example KMS permission pattern:

```json
{
  "Effect": "Allow",
  "Action": [
    "kms:Decrypt",
    "kms:DescribeKey"
  ],
  "Resource": "arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ab-cdef-1234567890ab"
}
```

Be careful to scope KMS permissions to the specific key whenever possible. Broad KMS access is a common least-privilege mistake.

**Resource-based policies versus caller permissions**

Lambda uses two different authorization paths depending on the integration.

1. **The invoking service or principal needs permission to call Lambda**
2. **Lambda may need a resource-based policy to allow that invocation**

Use this mental model:

- If a service directly invokes Lambda, the service or calling principal must be allowed to invoke the function.
- If the invocation crosses account boundaries or comes from certain AWS services, Lambda often needs a resource-based policy on the function itself.

This is a frequent exam topic because the correct answer depends on the trigger type.

**When Lambda needs a resource-based policy**

A resource-based policy on the Lambda function is commonly required for:

- **Cross-account invocation**
- Some AWS service integrations that invoke Lambda directly
- Allowing a service principal such as `sns.amazonaws.com` or `events.amazonaws.com` to invoke the function

The policy is attached to the function, not the role. It says who can invoke the function.

Example Lambda permission for EventBridge:

```bash
aws lambda add-permission \
  --function-name process-events \
  --statement-id AllowEventBridgeInvoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:123456789012:rule/MyRule
```

Example Lambda permission for SNS:

```bash
aws lambda add-permission \
  --function-name process-sns-message \
  --statement-id AllowSNSInvoke \
  --action lambda:InvokeFunction \
  --principal sns.amazonaws.com \
  --source-arn arn:aws:sns:us-east-1:123456789012:MyTopic
```

**When the invoking service needs permission**

Some services do not invoke Lambda directly in the same way. Instead, the service configuration itself must be allowed to target the function, and the service integration manages the call.

For exam purposes, remember:

- **API Gateway** integrates with Lambda through an integration setup and requires permission for API Gateway to invoke the function.
- **SNS** and **EventBridge** commonly require a Lambda resource-based policy statement allowing the service principal.
- **S3** event notifications also require permission for S3 to invoke Lambda, usually via a function policy statement.

The key is to identify whether the question is asking about:

- permissions on the **Lambda execution role**, or
- permissions for the **invoker** to call the Lambda function.

**API Gateway invocation example**

API Gateway is a common trigger on the exam. To let API Gateway invoke Lambda, you grant invoke permission on the function.

Example:

```bash
aws lambda add-permission \
  --function-name my-api-handler \
  --statement-id apigw-invoke \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn arn:aws:execute-api:us-east-1:123456789012:abc123/prod/GET/orders
```

What to remember:

- API Gateway needs permission to invoke the function.
- The Lambda execution role does not need permission to be invoked.
- The function still needs its own execution role permissions for downstream AWS calls.

**SNS invocation example**

SNS is another common trigger. If an SNS topic invokes Lambda, the function must allow the SNS service principal.

Example:

```bash
aws lambda add-permission \
  --function-name process-topic-message \
  --statement-id sns-invoke \
  --action lambda:InvokeFunction \
  --principal sns.amazonaws.com \
  --source-arn arn:aws:sns:us-east-1:123456789012:orders-topic
```

Typical exam clues:

- A topic publishes messages and Lambda processes them.
- The function is not being called by your code directly.
- The fix is often a resource-based policy on the Lambda function, not a change to the execution role.

**Cross-account invocation and confused deputy protection**

Cross-account Lambda invocation requires a resource-based policy on the function that explicitly allows the other account or service principal.

To reduce the risk of the **confused deputy** problem, use:

- `aws:SourceArn`
- `aws:SourceAccount`

These conditions restrict which source can invoke the function.

Why this matters:

- Without source restrictions, a service principal might be able to invoke the function from an unintended source.
- With source restrictions, you bind the permission to a specific topic, rule, bucket, or API.

Example policy statement concept:

```json
{
  "Sid": "AllowSpecificSNSTopic",
  "Effect": "Allow",
  "Principal": {
    "Service": "sns.amazonaws.com"
  },
  "Action": "lambda:InvokeFunction",
  "Resource": "arn:aws:lambda:us-east-1:123456789012:function:process-topic-message",
  "Condition": {
    "ArnLike": {
      "AWS:SourceArn": "arn:aws:sns:us-east-1:123456789012:orders-topic"
    },
    "StringEquals": {
      "AWS:SourceAccount": "123456789012"
    }
  }
}
```

Use these conditions whenever the service supports them, especially in cross-account or multi-tenant designs.

**Exam decision guide**

Use this quick reasoning pattern:

- If Lambda needs to call AWS APIs, update the **execution role**.
- If a service needs to invoke Lambda, update the **Lambda resource-based policy** or the service integration permission.
- If the function runs in a VPC, add **AWSLambdaVPCAccessExecutionRole**.
- If the function needs logs, add **AWSLambdaBasicExecutionRole**.
- If the function reads secrets or encrypted data, add the relevant **KMS** or **Secrets Manager** permissions.

A concise exam-ready summary:

- **Logging**: `AWSLambdaBasicExecutionRole`
- **VPC networking**: `AWSLambdaVPCAccessExecutionRole`
- **Service access**: custom execution role permissions
- **Invocation permission**: Lambda resource-based policy
- **Least privilege guardrail**: permissions boundary

**Practical IAM design example**

Suppose a Lambda function reads from DynamoDB, writes logs, decrypts a secret, and runs in a VPC.

A realistic execution role might include:

- `AWSLambdaBasicExecutionRole`
- `AWSLambdaVPCAccessExecutionRole`
- `dynamodb:GetItem`
- `secretsmanager:GetSecretValue`
- `kms:Decrypt`

Example custom policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders"
    },
    {
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue"
      ],
      "Resource": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/orders/*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ab-cdef-1234567890ab"
    }
  ]
}
```

This design keeps the role narrow while covering the function’s runtime needs.

**Troubleshooting patterns**

If a Lambda function fails, check the failure mode before changing permissions.

- If logs are missing, verify **AWSLambdaBasicExecutionRole**.
- If the function cannot attach to a VPC, verify **AWSLambdaVPCAccessExecutionRole**.
- If the function gets `AccessDenied` when calling AWS APIs, inspect the execution role policy.
- If a trigger cannot invoke the function, inspect the Lambda resource-based policy and the service principal.
- If cross-account access fails, confirm `aws:SourceArn` and `aws:SourceAccount` conditions.

> If the question is about “who can call Lambda,” think resource-based policy. If it is about “what Lambda can do,” think execution role.

**Quick reference**

| Need | Where to configure | Typical answer |
|---|---|---|
| CloudWatch Logs | Execution role | `AWSLambdaBasicExecutionRole` |
| VPC access | Execution role | `AWSLambdaVPCAccessExecutionRole` |
| DynamoDB/S3/SQS/Secrets Manager access | Execution role | Custom permissions |
| API Gateway/SNS/EventBridge invoke Lambda | Lambda resource policy | `lambda:InvokeFunction` permission |
| Cross-account invoke | Lambda resource policy | `aws:SourceArn` and `aws:SourceAccount` |
| Guardrail on role creation | Permissions boundary | Maximum allowed permissions |

**CLI workflow for role setup**

A typical setup sequence looks like this:

1. Create or choose an IAM role with the Lambda trust policy.
2. Attach **AWSLambdaBasicExecutionRole**.
3. Attach **AWSLambdaVPCAccessExecutionRole** if the function uses a VPC.
4. Add custom permissions for downstream services.
5. Add a Lambda resource-based policy for the trigger if needed.

Example role attachment commands:

```bash
aws iam attach-role-policy \
  --role-name my-lambda-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

aws iam attach-role-policy \
  --role-name my-lambda-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaVPCAccessExecutionRole
```

**What to memorize for the exam**

- Lambda assumes the execution role using **STS** and temporary credentials.
- The trust policy must allow `lambda.amazonaws.com`.
- **AWSLambdaBasicExecutionRole** is the standard logging baseline.
- **AWSLambdaVPCAccessExecutionRole** is the standard VPC baseline.
- Execution role policies control what the function can do.
- Permissions boundaries cap what the role can ever do.
- Resource-based policies control who can invoke the function.
- Use `aws:SourceArn` and `aws:SourceAccount` for safer cross-account and service-based invocations.
