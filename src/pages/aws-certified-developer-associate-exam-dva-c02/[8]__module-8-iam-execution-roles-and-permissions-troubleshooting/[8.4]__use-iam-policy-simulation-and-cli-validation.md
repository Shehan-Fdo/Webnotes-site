## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**IAM policy simulation: what it is and when to use it**

IAM policy simulation helps you test whether a request is allowed or denied before you deploy or troubleshoot access. It is most useful when you need to reason about identity-based policies, explicit denies, and condition keys without guessing from the console error alone.

Use the simulator when you want to:

- Validate a policy change before attaching it to a user, role, or group.
- Troubleshoot `AccessDenied` errors by checking which statement blocks access.
- Compare expected access across multiple actions or resources.
- Test condition logic such as tags, regions, transport security, or source IP.

The simulator is not a full replacement for live authorization. It is a diagnostic tool, and its results depend on the policies and context you provide.

**Core simulator commands**

The AWS CLI provides two main commands for policy simulation:

- `simulate-principal-policy` for a specific IAM principal such as a user or role.
- `simulate-custom-policy` for standalone policy documents that are not attached to a principal.

A practical mental model is:

- Use `simulate-principal-policy` when you want to ask, “What would this user or role be allowed to do?”
- Use `simulate-custom-policy` when you want to ask, “What would this policy document allow or deny if evaluated?”

Example principal simulation:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:user/DevUser \
  --action-names s3:GetObject s3:PutObject \
  --resource-arns arn:aws:s3:::my-app-bucket/* \
  --caller-arn arn:aws:iam::123456789012:user/DevUser
```

Example custom policy simulation:

```bash
aws iam simulate-custom-policy \
  --policy-input-list file://policy.json \
  --action-names dynamodb:GetItem dynamodb:PutItem \
  --resource-arns arn:aws:dynamodb:us-east-1:123456789012:table/Orders
```

**How to choose the right command**

`simulate-principal-policy` is the exam-relevant default when you are validating a real IAM identity. It evaluates the policies attached to the specified principal and lets you test multiple actions and resources in one call.

`simulate-custom-policy` is useful when you are reviewing a policy document before attachment or when you want to isolate the effect of a policy statement. It does not magically reproduce every live authorization path; you still need to supply the relevant context and understand service-specific limits.

A quick comparison:

| Command | Best use | Typical input |
|---|---|---|
| `simulate-principal-policy` | Validate a user or role | Principal ARN plus actions/resources |
| `simulate-custom-policy` | Test a policy document | JSON policy file plus actions/resources |

**Supplying multiple policies and contexts**

Real authorization often depends on more than one policy document. You may need to evaluate:

- Identity-based policies attached to the principal.
- Inline policies and managed policies.
- Additional context such as tags or request attributes.
- Optional policy documents that represent extra constraints you want to test.

For exam purposes, remember that the simulator is strongest for identity-policy reasoning. It is not a perfect mirror of every live authorization layer, especially when service-specific behavior or resource-based policy evaluation is involved.

A clean way to test a custom policy file is to keep the JSON readable and focused:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadOnlyOrders",
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:Query"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders"
    }
  ]
}
```

If you need to test several actions at once, pass them together:

```bash
aws iam simulate-custom-policy \
  --policy-input-list file://policy.json \
  --action-names dynamodb:GetItem dynamodb:Query dynamodb:PutItem \
  --resource-arns arn:aws:dynamodb:us-east-1:123456789012:table/Orders
```

This helps you see which actions are allowed and which are denied under the same policy.

**Reading simulator output**

The most important output field is the decision for each action-resource pair:

- `allowed`
- `explicitDeny`
- `implicitDeny`

Interpretation matters:

- `allowed` means at least one applicable statement permits the action and no explicit deny overrides it.
- `explicitDeny` means a matching deny statement blocks the request.
- `implicitDeny` means no statement allowed the request.

For troubleshooting, distinguish these carefully:

- An **explicit deny** always wins.
- An **implicit deny** usually means the policy is missing an allow, the resource ARN is wrong, or the condition did not match.

Example reasoning:

- `s3:GetObject` returns `explicitDeny` because a deny statement blocks access to a sensitive prefix.
- `s3:PutObject` returns `implicitDeny` because no allow statement covers that action.

That difference is often the key to exam questions.

**Mapping results back to policy statements**

When a simulation fails, work backward from the decision to the statement logic:

- Check the `Action` element first.
- Check the `Resource` or `NotResource` scope.
- Check the `Condition` block.
- Check whether a deny statement matches more broadly than the allow.

A useful troubleshooting pattern is:

1. Identify the denied action and resource.
2. Find all statements that mention that action.
3. Compare the resource ARN pattern to the request ARN.
4. Evaluate each condition key against the request context.
5. Look for any explicit deny that overrides the allow.

This is especially important when policies use wildcards, tag conditions, or region restrictions.

**Condition keys you should know for exam scenarios**

IAM simulation is especially valuable when testing condition-based access control. Common keys include:

- `aws:RequestedRegion` to restrict actions to a region.
- `aws:SecureTransport` to require HTTPS.
- `aws:SourceIp` to restrict by client IP.
- `aws:SourceVpce` to restrict access through a specific VPC endpoint.
- `aws:PrincipalArn` to match the calling principal.
- `aws:PrincipalTag/<TagKey>` for principal tag-based access.
- `aws:ResourceTag/<TagKey>` for resource tag-based access.
- `aws:MultiFactorAuthPresent` for MFA-based access decisions.

Example policy using region and transport conditions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOnlyInUsEast1OverTLS",
      "Effect": "Allow",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-app-bucket/*",
      "Condition": {
        "StringEquals": {
          "aws:RequestedRegion": "us-east-1"
        },
        "Bool": {
          "aws:SecureTransport": "true"
        }
      }
    }
  ]
}
```

Example policy using tags:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowTaggedPrincipalsToReadTaggedObjects",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::project-data/*",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalTag/Department": "Finance",
          "aws:ResourceTag/Classification": "Internal"
        }
      }
    }
  ]
}
```

Tag-based access control is a common exam theme because it scales better than hardcoding ARNs for every user or resource.

**Simulating tag-based access**

To validate tag-based policies, make sure the simulation includes the same tag context that the real request would have. If the principal or resource tags are missing from the simulation, the result may show an implicit deny even though the live request would succeed.

Practical guidance:

- Use principal tags when the policy depends on attributes of the caller.
- Use resource tags when the policy depends on the target resource.
- Verify that tag keys and values match exactly, including case.

A typical troubleshooting pattern is:

- The policy allows `ec2:StartInstances` only when `aws:PrincipalTag/Team = DevOps`.
- The simulation returns `implicitDeny`.
- You discover the role has no `Team=DevOps` session tag or principal tag.

That tells you the policy logic is fine, but the request context is incomplete.

**Cross-account access reasoning**

Cross-account access often involves both identity-side permissions and a resource-side grant, but the exact requirement depends on the service and access pattern. Do not assume every service needs the same combination.

For exam questions, reason like this:

- Does the target service support resource-based policies?
- Is the caller in another account?
- Is the access granted by the resource policy alone, or does the caller also need an identity policy?
- Is the request using a role session, a federated session, or a direct IAM principal?

Examples:

- **S3** bucket policies can grant cross-account access directly for supported actions.
- **KMS** key policies and grants are central to cross-account use, and identity permissions may also matter depending on the operation.
- **Lambda** resource policies can allow invocation from another account.

When validating cross-account access, simulate the identity policy first, then inspect the resource policy separately if the service supports one. The simulator is not a universal replacement for live cross-account authorization testing.

**Console versus CLI**

The IAM Policy Simulator exists in both the AWS console and the CLI.

Use the console when you want:

- A visual view of allowed and denied actions.
- Quick manual testing during learning or troubleshooting.
- Easier exploration of statement effects.

Use the CLI when you want:

- Repeatable validation in scripts or CI/CD.
- Bulk testing across many actions or resources.
- Faster troubleshooting from a terminal.
- Automation around policy review before deployment.

For exam prep, know both, but the CLI is especially useful when you need deterministic, repeatable checks.

**Troubleshooting explicit deny versus implicit deny**

A strong exam skill is identifying why a request fails.

Scenario 1: explicit deny

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowRead",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    },
    {
      "Sid": "DenySecretPrefix",
      "Effect": "Deny",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/secret/*"
    }
  ]
}
```

If you simulate `s3:GetObject` on `arn:aws:s3:::my-bucket/secret/file.txt`, the result should be `explicitDeny`.

Scenario 2: implicit deny

If you simulate `s3:DeleteObject` with the same policy, the result is `implicitDeny` because no allow statement covers that action.

The exam often asks you to choose the best fix:

- Add an allow when the result is implicit deny.
- Remove or narrow a deny when the result is explicit deny.

**Practical CLI examples**

Simulate multiple actions against one resource:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject s3:PutObject s3:DeleteObject \
  --resource-arns arn:aws:s3:::my-app-bucket/* \
  --caller-arn arn:aws:iam::123456789012:role/AppRole
```

Simulate a region-restricted action:

```bash
aws iam simulate-custom-policy \
  --policy-input-list file://region-policy.json \
  --action-names ec2:RunInstances \
  --resource-arns "*" \
  --context-entries ContextKeyName=aws:RequestedRegion,ContextKeyType=string,ContextKeyValues=us-east-1
```

Simulate HTTPS enforcement:

```bash
aws iam simulate-custom-policy \
  --policy-input-list file://tls-policy.json \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::my-secure-bucket/* \
  --context-entries ContextKeyName=aws:SecureTransport,ContextKeyType=boolean,ContextKeyValues=true
```

> Be careful with CLI syntax for `--context-entries`. Depending on shell and CLI version, you may need quoted structures or JSON-style input. Treat the example as a pattern, not a copy-paste guarantee.

**Temporary credentials and session context**

Temporary credentials can change the effective authorization context. This matters for:

- Assumed roles.
- Federated sessions.
- Session tags.
- MFA-based conditions.

The simulator can help you reason about these cases, but you must supply the right context. Do not assume a role ARN alone represents the full session behavior.

Important limitations to remember:

- A role session is not identical to the role identity itself.
- Session tags may affect tag-based conditions.
- MFA-related conditions depend on authenticated session context.
- Service-linked roles and some service-invoked actions may not be modeled the same way as direct IAM calls.

**Resource-based policy validation**

Resource-based policies are important in AWS, but simulator support is not a universal substitute for live service behavior. Treat resource-policy simulation as service-dependent and limited.

Use the simulator to validate the identity-side logic first, then confirm the resource policy in the service that owns the resource. This is especially relevant for services such as:

- S3 bucket policies
- KMS key policies
- Lambda function policies
- SNS topic policies
- SQS queue policies

For exam questions, focus on the authorization model of the specific service rather than assuming one simulator workflow covers everything.

**Exam-style scenarios**

Scenario 1: A developer can list an S3 bucket but cannot read objects.

- Likely cause: missing `s3:GetObject` allow on the object ARN.
- Simulator clue: `s3:ListBucket` may be `allowed`, while `s3:GetObject` is `implicitDeny`.

Scenario 2: A request fails only when using a noncompliant network path.

- Likely cause: `aws:SecureTransport` or `aws:SourceVpce` condition mismatch.
- Simulator clue: the action is denied only when the condition context is absent or false.

Scenario 3: A role works in one region but not another.

- Likely cause: `aws:RequestedRegion` restriction.
- Simulator clue: same action returns `allowed` in one region and `implicitDeny` in another.

Scenario 4: A tagged principal can access one resource but not another.

- Likely cause: `aws:PrincipalTag` and `aws:ResourceTag` mismatch.
- Simulator clue: the action is allowed only when both tags match the policy condition.

**How to study this topic for DVA-C02**

Focus on the decision logic, not just the command syntax.

- Know when to use `simulate-principal-policy` versus `simulate-custom-policy`.
- Understand `allowed`, `explicitDeny`, and `implicitDeny`.
- Practice reading `Condition` blocks with common keys.
- Be able to explain why a request fails from the policy structure.
- Remember that the simulator is a diagnostic aid, not a perfect replica of every AWS authorization path.

A good exam answer usually identifies the missing allow, the overriding deny, or the condition mismatch quickly and accurately.
