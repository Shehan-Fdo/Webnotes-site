## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Least-privilege design goals**

Least privilege means you grant only the actions, resources, and conditions that a workload needs to function. For exam scenarios, you usually start from the service API calls the application makes, then map each call to the narrowest possible IAM permissions.

A good policy answers three questions:

- **What action** does the code call?
- **Which resource** does that action target?
- **Under what conditions** should the action be allowed?

For AWS developer workloads, the most common trio is:

- **DynamoDB** for application data
- **CloudWatch Logs** for runtime logging
- **X-Ray** for distributed tracing

The exam often tests whether you can avoid overbroad permissions such as `dynamodb:*`, `logs:*`, or `xray:*` when a smaller set is enough.

**IAM policy building blocks**

An IAM policy statement uses `Effect`, `Action`, `Resource`, and optional `Condition`. Least privilege usually means:

- Prefer specific actions over wildcards.
- Prefer table, index, log group, or trace resource ARNs over `*`.
- Use conditions when resource scoping alone is not enough.
- Separate permissions by function when different components need different access.

Example structure:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOrdersTable",
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:Query"
      ],
      "Resource": [
        "arn:aws:dynamodb:us-east-1:123456789012:table/Orders",
        "arn:aws:dynamodb:us-east-1:123456789012:table/Orders/index/*"
      ]
    }
  ]
}
```

**DynamoDB permissions**

DynamoDB is a common exam focus because different API calls require different resource scopes. Some actions target the table, some target indexes, and some are effectively table-level operations that do not support fine-grained resource scoping in the same way.

Key patterns:

- **Item-level reads and writes** often use the table ARN.
- **Query on a global secondary index (GSI)** requires the index ARN in the `Resource` element.
- **Scan** typically uses the table ARN.
- **Index access** needs the index ARN, not just the table ARN.
- **Batch and transaction APIs** may require permissions on multiple tables or broader resource scope depending on the operation.

Common ARN patterns:

```text
arn:aws:dynamodb:us-east-1:123456789012:table/Orders
arn:aws:dynamodb:us-east-1:123456789012:table/Orders/index/ByCustomerId
arn:aws:dynamodb:us-east-1:123456789012:table/Orders/index/*
```

Important exam nuance:

- `dynamodb:Query` against a **table** can use the table ARN.
- `dynamodb:Query` against a **GSI** should use the index ARN.
- If your code can query multiple indexes, you can scope to `table/<name>/index/*`, but that is broader than a single index ARN.

A practical read policy for one table and one index:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOrderData",
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:Query",
        "dynamodb:Scan"
      ],
      "Resource": [
        "arn:aws:dynamodb:us-east-1:123456789012:table/Orders",
        "arn:aws:dynamodb:us-east-1:123456789012:table/Orders/index/ByCustomerId"
      ]
    }
  ]
}
```

**DynamoDB resource-level nuances**

Several DynamoDB actions are commonly tested because they do not behave like simple table reads:

- **`BatchGetItem`**
  - Reads from multiple tables in one call.
  - Resource scoping is less straightforward because the request can span tables.
  - In exam questions, expect broader permissions or multiple table ARNs depending on the exact access pattern.

- **`BatchWriteItem`**
  - Writes to multiple tables in one call.
  - Often needs permissions on each target table.
  - Use only the tables the application actually writes to.

- **`TransactGetItems`**
  - Reads multiple items atomically across one or more tables.
  - Scope permissions to every table involved.

- **`TransactWriteItems`**
  - Performs atomic writes across one or more tables.
  - Scope permissions to every table involved.

- **`ConditionCheckItem`**
  - Used in transactions to verify item state without writing.
  - Requires read-style permissions on the relevant table or item scope.

A transaction policy often looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "TransactionalAccess",
      "Effect": "Allow",
      "Action": [
        "dynamodb:TransactGetItems",
        "dynamodb:TransactWriteItems",
        "dynamodb:ConditionCheckItem"
      ],
      "Resource": [
        "arn:aws:dynamodb:us-east-1:123456789012:table/Orders",
        "arn:aws:dynamodb:us-east-1:123456789012:table/OrderAudit"
      ]
    }
  ]
}
```

**DynamoDB conditions and fine-grained access**

DynamoDB supports condition keys that help narrow access beyond the ARN. These are especially useful when multiple tenants, environments, or item patterns share the same table.

Common uses:

- Restrict access to a specific partition key pattern.
- Limit access by leading key values in `Query`.
- Separate access by environment tags or principal tags.
- Control access to indexes or attributes when the application design supports it.

Examples of condition-based approaches:

- `dynamodb:LeadingKeys` to restrict partition key values.
- `dynamodb:Attributes` to limit returned attributes.
- `dynamodb:Select` to control query style.
- `aws:PrincipalTag` or `aws:ResourceTag` for tag-based access control.

Example with a leading key condition:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "TenantScopedRead",
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:Query"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders",
      "Condition": {
        "ForAllValues:StringEquals": {
          "dynamodb:LeadingKeys": [
            "${aws:PrincipalTag/TenantId}"
          ]
        }
      }
    }
  ]
}
```

Use conditions when:

- One table stores data for multiple tenants.
- You want to prevent cross-tenant reads even if the table ARN is shared.
- You need to enforce attribute-level or key-level restrictions.

> Be careful with transaction and batch APIs: a condition that works for single-item access may not fully express the access pattern for multi-item operations.

**CloudWatch Logs permissions**

CloudWatch Logs permissions are often needed by Lambda functions, ECS tasks, EC2 instances, and custom applications. The most common exam pattern is a Lambda execution role that writes logs to a log group and log stream.

The standard baseline for Lambda logging is usually:

- `logs:CreateLogGroup`
- `logs:CreateLogStream`
- `logs:PutLogEvents`

In many real deployments, AWS-managed Lambda execution policies already include the common logging permissions. For custom least-privilege roles, you should scope them to the specific log group when possible.

CloudWatch Logs ARN patterns:

```text
arn:aws:logs:us-east-1:123456789012:log-group:/aws/lambda/MyFunction
arn:aws:logs:us-east-1:123456789012:log-group:/aws/lambda/MyFunction:log-stream:*
```

Important distinctions:

- **Log group ARN** identifies the container for streams.
- **Log stream ARN** identifies the individual stream.
- Some actions apply to the log group, while others apply to streams within it.
- Wildcards belong at the end of the resource pattern when you need multiple streams.

A least-privilege Lambda logging policy often looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "WriteLambdaLogs",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/lambda/MyFunction:log-stream:*"
    }
  ]
}
```

If the log group does not already exist and your deployment does not create it ahead of time, you may also need:

```json
{
  "Sid": "CreateLogGroupIfMissing",
  "Effect": "Allow",
  "Action": "logs:CreateLogGroup",
  "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/lambda/MyFunction"
}
```

Practical guidance:

- Include `logs:CreateLogGroup` only when the function or deployment process must create the group.
- Omit it when infrastructure provisioning creates the log group ahead of time.
- Scope `CreateLogStream` and `PutLogEvents` to the specific log group’s streams.

Example CLI to inspect log groups and streams:

```bash
aws logs describe-log-groups --log-group-name-prefix /aws/lambda/MyFunction
aws logs describe-log-streams --log-group-name /aws/lambda/MyFunction
```

Current behavior note:

- CloudWatch Logs sequence-token handling has changed over time.
- Do not memorize outdated examples that rely on a fixed token value.
- For exam purposes, focus on the permission model and resource scoping rather than a specific token workflow.

**CloudWatch Logs conditions and access control**

CloudWatch Logs supports IAM conditions in some scenarios, which can help you constrain access further. Common patterns include:

- Restricting access by log group name or ARN pattern.
- Using tag-based controls with `aws:ResourceTag`.
- Using principal tags to separate environments or teams.

Example with a tag condition:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "TaggedLogGroupAccess",
      "Effect": "Allow",
      "Action": [
        "logs:DescribeLogStreams",
        "logs:GetLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/lambda/*",
      "Condition": {
        "StringEquals": {
          "aws:ResourceTag/Environment": "dev"
        }
      }
    }
  ]
}
```

Use conditions when:

- Multiple applications share a logging account.
- You want to separate dev, test, and prod log access.
- You need read access only to tagged log groups.

**X-Ray permissions**

AWS X-Ray helps you trace requests across services. The exact IAM permissions depend on how tracing is integrated.

Two common paths exist:

- **Application-managed tracing**
  - Your code uses the X-Ray SDK or emits trace data directly.
  - The application role needs permissions to send trace data and possibly read trace data for diagnostics.

- **Service-managed tracing**
  - A service such as Lambda can publish trace segments with active tracing enabled.
  - The service integration may handle some tracing behavior, but your application may still need permissions if it uses the SDK or reads trace data.

Common X-Ray actions:

- `xray:PutTraceSegments`
- `xray:PutTelemetryRecords`
- `xray:GetSamplingRules`
- `xray:GetSamplingTargets`
- `xray:GetSamplingStatisticSummaries`
- `xray:BatchGetTraces`
- `xray:GetTraceSummaries`
- `xray:PutResourcePolicy`

For most application instrumentation, the write-side permissions are the key ones:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "WriteXRayData",
      "Effect": "Allow",
      "Action": [
        "xray:PutTraceSegments",
        "xray:PutTelemetryRecords"
      ],
      "Resource": "*"
    }
  ]
}
```

Why `Resource` is often `*` for X-Ray:

- Many X-Ray actions are not resource-scoped in the same way as DynamoDB tables or CloudWatch log groups.
- The exam often expects you to know that some services rely on action-level permissions with `Resource: "*"`.
- If a service supports resource-level control, use it; otherwise, keep the action list minimal.

Sampling-related permissions matter when:

- You use the X-Ray SDK and want the application to request sampling rules.
- You run a distributed system that needs dynamic sampling behavior.
- You troubleshoot trace volume and want the service to query sampling configuration.

Example with read and write tracing permissions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "XRayInstrumentation",
      "Effect": "Allow",
      "Action": [
        "xray:PutTraceSegments",
        "xray:PutTelemetryRecords",
        "xray:GetSamplingRules",
        "xray:GetSamplingTargets",
        "xray:GetSamplingStatisticSummaries"
      ],
      "Resource": "*"
    }
  ]
}
```

**Lambda, ECS, and managed integrations**

The exam may ask how tracing permissions differ by compute service.

For **Lambda**:

- If you enable active tracing, Lambda can integrate with X-Ray at the service level.
- If your function also uses the X-Ray SDK, the execution role may need the X-Ray write permissions.
- If you only rely on Lambda-managed tracing and do not emit traces from code, the permission needs may be smaller than a fully instrumented application.

For **ECS** or other container workloads:

- The task role usually needs the application permissions.
- If you run the X-Ray daemon or sidecar, the containerized app may send trace data through that component.
- The exact setup depends on whether the service or the application is responsible for emitting trace segments.

For **EC2**:

- The instance profile or application role needs the permissions if the app sends trace data directly.
- You may also need the X-Ray daemon installed and configured.

Practical takeaway:

- Do not assume every traced workload needs the same X-Ray policy.
- Match the policy to the integration path: service-managed tracing, SDK-based tracing, or daemon/sidecar-based tracing.

**End-to-end least-privilege example**

A common exam scenario is a Lambda function that:

- Reads and writes items in one DynamoDB table.
- Writes logs to one CloudWatch log group.
- Sends trace data to X-Ray.

A combined execution role policy might look like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DynamoDBAccess",
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:PutItem",
        "dynamodb:UpdateItem",
        "dynamodb:Query"
      ],
      "Resource": [
        "arn:aws:dynamodb:us-east-1:123456789012:table/Orders",
        "arn:aws:dynamodb:us-east-1:123456789012:table/Orders/index/ByCustomerId"
      ]
    },
    {
      "Sid": "CloudWatchLogsAccess",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/lambda/OrderProcessor:log-stream:*"
    },
    {
      "Sid": "XRayAccess",
      "Effect": "Allow",
      "Action": [
        "xray:PutTraceSegments",
        "xray:PutTelemetryRecords"
      ],
      "Resource": "*"
    }
  ]
}
```

If the deployment does not pre-create the log group, add a separate statement for `logs:CreateLogGroup`. If the function queries only one GSI, scope the DynamoDB `Query` permission to that index ARN instead of using the table wildcard.

**CLI examples for validation**

You can use the AWS CLI to test whether your policy matches the intended resources.

Check DynamoDB table metadata:

```bash
aws dynamodb describe-table --table-name Orders
```

Query a specific index:

```bash
aws dynamodb query \
  --table-name Orders \
  --index-name ByCustomerId \
  --key-condition-expression "CustomerId = :cid" \
  --expression-attribute-values '{":cid":{"S":"cust-123"}}'
```

Inspect log group resources:

```bash
aws logs describe-log-groups --log-group-name-prefix /aws/lambda/OrderProcessor
aws logs describe-log-streams --log-group-name /aws/lambda/OrderProcessor
```

Send a test trace from an instrumented app only if your runtime and SDK support it; otherwise, verify that the role includes the minimal X-Ray write actions.

**Exam traps and how to avoid them**

- **Using `*` when a resource ARN is available**
  - Prefer table, index, or log group ARNs whenever the service supports them.

- **Confusing table ARNs with index ARNs**
  - `Query` on a GSI needs the index ARN.
  - A table ARN alone can be insufficient for index access.

- **Overgranting CloudWatch Logs**
  - `logs:*` is broader than needed.
  - Lambda usually needs only create stream and put events, plus create group if the group is not pre-provisioned.

- **Assuming outdated sequence-token behavior**
  - Focus on permissions and resource scope, not a fixed token example.

- **Treating all X-Ray integrations the same**
  - Service-managed tracing and SDK-based tracing can have different IAM needs.

- **Ignoring transaction and batch APIs in DynamoDB**
  - These APIs often require permissions on every table involved, not just one.

**Quick reference**

| Service | Common least-privilege actions | Typical resource scope |
|---|---|---|
| DynamoDB | `GetItem`, `PutItem`, `UpdateItem`, `DeleteItem`, `Query`, `Scan` | Table ARN and, for `Query` on GSIs, index ARN |
| DynamoDB transactions | `TransactGetItems`, `TransactWriteItems`, `ConditionCheckItem` | Every involved table ARN |
| CloudWatch Logs | `CreateLogGroup`, `CreateLogStream`, `PutLogEvents`, `DescribeLogStreams` | Log group ARN and log stream ARN pattern |
| X-Ray | `PutTraceSegments`, `PutTelemetryRecords`, sampling read actions | Often `*` for action-only permissions |

**How to reason in the exam**

When you see a permissions question, work from the application outward:

1. Identify the exact AWS API calls.
2. Determine whether each call is read, write, or control-plane access.
3. Map each call to the narrowest resource ARN.
4. Add conditions only when they improve isolation without breaking the workload.
5. Avoid broad managed policies unless the question explicitly asks for a quick default.

A strong answer usually shows that you understand both the service behavior and the IAM scoping model. For DynamoDB, that means table versus index permissions. For CloudWatch Logs, that means log group versus log stream ARNs. For X-Ray, that means matching permissions to the tracing integration path.
