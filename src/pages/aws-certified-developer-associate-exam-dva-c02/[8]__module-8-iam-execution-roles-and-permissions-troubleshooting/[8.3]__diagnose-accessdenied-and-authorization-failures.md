## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**AccessDenied troubleshooting mindset**

`AccessDenied` usually means AWS evaluated the request and found no path to allow it, or an explicit deny blocked it. The fastest way to diagnose the issue is to identify **who** made the call, **what** action was attempted, **which resource** was targeted, and **which policy layer** denied it.

In AWS, authorization is not just one policy. A request can be affected by:

- **Identity-based policies** attached to users, groups, or roles
- **Resource-based policies** on the target resource
- **Permissions boundaries** on the principal
- **Service control policies** (SCPs) in AWS Organizations
- **Session policies** from role assumption or federation
- **VPC endpoint policies** for private access paths
- **KMS key policies** and grants for encrypted resources

A useful mental model is: AWS allows a request only when there is an applicable allow and no applicable explicit deny.

**Common error patterns**

Not every authorization failure looks identical. The exact message often hints at the layer involved.

- `AccessDenied` or `AccessDeniedException`
  - Generic authorization failure from many services
- `UnauthorizedOperation`
  - Common in EC2 when the caller lacks permission
- `User is not authorized to perform: <action> on resource: <arn>`
  - Direct IAM denial with action and resource context
- `KMS.AccessDeniedException`
  - Often indicates missing KMS permissions, key policy access, or grant issues
- `403 Forbidden`
  - Common in S3, API Gateway, CloudFront, and signed URL scenarios
- `The security token included in the request is invalid`
  - Often a credential, session, or signing problem rather than pure authorization

> A request can fail even when the IAM policy looks correct if another layer, such as a permissions boundary or KMS key policy, blocks it.

**How AWS evaluates authorization**

AWS evaluates permissions by combining multiple policy types. The practical rule is simple: an explicit deny wins, and otherwise the request must be allowed by all relevant controls.

- **Identity policy evaluation**
  - Checks whether the principal has an allow for the action and resource
- **Resource policy evaluation**
  - Checks whether the resource trusts the principal
- **Boundary and organization controls**
  - Check whether the principal is allowed to use the permission at all
- **Service-specific checks**
  - Some services add their own authorization rules, such as S3 object ownership or KMS encryption context

A request can be allowed in IAM but still fail because the service adds extra constraints. For example, S3 access may fail if the bucket policy denies the principal, or if the object is owned by another account and ACLs are involved.

**Step-by-step diagnosis workflow**

Use a repeatable process so you do not guess at the cause.

1. Identify the exact API action.
   - Read the error message carefully.
   - Map the failure to an API action such as `s3:GetObject`, `dynamodb:PutItem`, or `lambda:InvokeFunction`.

2. Identify the principal.
   - Determine whether the caller is an IAM user, IAM role, assumed role session, federated user, Lambda execution role, EC2 instance profile, or service principal.
   - If the call is made by an application, inspect the runtime credentials source.

3. Identify the target resource.
   - Confirm the full ARN when possible.
   - Check whether the resource is in the same account and Region.

4. Check identity-based permissions.
   - Review attached policies, inline policies, and group membership.
   - Verify the action and resource match exactly.

5. Check for explicit denies.
   - Look for deny statements in identity policies, resource policies, SCPs, permissions boundaries, session policies, and endpoint policies.

6. Check service-specific authorization.
   - For encrypted data, inspect KMS permissions.
   - For S3, inspect bucket policy, object ownership, and ACL behavior if relevant.
   - For API Gateway, inspect resource policy and authorizer behavior.

7. Reproduce with the AWS CLI.
   - Use the same identity and region.
   - Confirm whether the failure is consistent.

8. Inspect logs and traces.
   - Use CloudTrail to see the denied API call and the principal that made it.
   - Use service logs where available.

**Useful CLI commands**

The AWS CLI helps you confirm the caller identity and reproduce the failing action.

```bash
aws sts get-caller-identity
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/MyAppRole \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::my-bucket/example.txt
```

For role sessions, inspect the assumed role identity and session context:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/MyAppRole \
  --role-session-name debug-session
```

If the problem is with S3, test the exact object path:

```bash
aws s3api head-object \
  --bucket my-bucket \
  --key example.txt
```

For DynamoDB, verify table-level permissions and resource ARN matching:

```bash
aws dynamodb put-item \
  --table-name Orders \
  --item '{"OrderId":{"S":"123"},"Status":{"S":"NEW"}}'
```

**IAM policy logic to verify**

Authorization failures often come from a mismatch between what you intended and what the policy actually matches.

- **Action names**
  - `s3:GetObject` is not the same as `s3:ListBucket`
  - `dynamodb:PutItem` is not the same as `dynamodb:UpdateItem`
- **Resource ARNs**
  - Many actions require a resource ARN, not `*`
  - Some actions support only `*` as the resource
- **Condition keys**
  - Conditions can silently block access if the request context does not match
- **Wildcards**
  - Overly broad wildcards can still fail if the service requires a different resource type or ARN format

Example identity policy for read access to one S3 object prefix:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadAppPrefix",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/app/*"
    },
    {
      "Sid": "ListBucketForPrefix",
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket"
      ],
      "Resource": "arn:aws:s3:::my-bucket",
      "Condition": {
        "StringLike": {
          "s3:prefix": [
            "app/*"
          ]
        }
      }
    }
  ]
}
```

This policy allows object reads under `app/` and bucket listing only for that prefix. If the application tries to read `arn:aws:s3:::my-bucket/private/file.txt`, the request fails because the resource does not match.

**Explicit deny sources**

An explicit deny is the most important thing to search for because it overrides any allow.

- **Identity policy deny**
  - A deny statement attached to the user, group, or role blocks the action
- **Resource policy deny**
  - A bucket policy, queue policy, or key policy can deny access
- **SCP deny**
  - An organization-level SCP can block actions across accounts
- **Permissions boundary deny**
  - The boundary limits what the identity can ever do
- **Session policy deny**
  - Temporary credentials can be further restricted at session time
- **Endpoint policy deny**
  - A VPC endpoint can block requests even when IAM allows them

A common exam trap is assuming that an IAM role policy is enough. If the role has permission but the permissions boundary omits the action, the request still fails.

**Resource-based policy examples**

Resource policies are common for cross-account access and service integrations.

S3 bucket policy example allowing a role from another account:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AnalyticsRole"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/reports/*"
    }
  ]
}
```

If this still fails, check:

- The caller’s identity policy in the source account
- The bucket policy in the destination account
- Object ownership and ACL behavior if the object was uploaded by another account
- Whether the request is using the correct Region and endpoint

**KMS-related authorization failures**

AWS Key Management Service (KMS) often causes confusing `AccessDenied` errors because access requires more than one permission check.

You typically need:

- Permission to use the AWS service action, such as `s3:GetObject`
- Permission to use the KMS key, such as `kms:Decrypt`
- A key policy or grant that allows the principal
- Sometimes an encryption context match

Example IAM policy for decrypting with a specific key:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowDecryptForAppKey",
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt",
        "kms:DescribeKey"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ef-abcd-1234567890ab"
    }
  ]
}
```

If an S3 object is encrypted with KMS, the caller may need both S3 permission and KMS permission. A missing `kms:Decrypt` permission can surface as an S3 access failure.

**S3-specific pitfalls**

S3 authorization failures are common because access can involve IAM, bucket policies, object ACLs, ownership, and KMS.

- **`s3:ListBucket` vs `s3:GetObject`**
  - Listing a bucket and reading an object are different permissions
- **Bucket ARN vs object ARN**
  - Bucket-level actions use `arn:aws:s3:::bucket-name`
  - Object-level actions use `arn:aws:s3:::bucket-name/key`
- **Prefix conditions**
  - A policy may allow only a specific prefix
- **Object ownership**
  - Cross-account uploads can create ownership issues if not managed correctly
- **Block Public Access**
  - Public access settings can override permissive ACLs or policies

Example of a common mismatch:

```bash
aws s3 ls s3://my-bucket/private/
aws s3 cp s3://my-bucket/private/report.csv .
```

The first command needs `s3:ListBucket`; the second needs `s3:GetObject`. A policy that grants only one of them produces a partial failure.

**Lambda and API Gateway authorization**

Serverless applications often fail because the execution role or invoke permissions are incomplete.

For Lambda, check:

- The function’s execution role permissions
- The caller’s permission to invoke the function
- Resource-based permissions on the function for cross-account or service invocation

Example invoke permission for API Gateway:

```bash
aws lambda add-permission \
  --function-name MyFunction \
  --statement-id apigw-invoke \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn arn:aws:execute-api:us-east-1:123456789012:abc123/*/GET/myresource
```

If API Gateway returns `403`, verify:

- The method authorization type
- The Lambda authorizer or IAM auth configuration
- The resource policy on the API
- The caller’s SigV4 signing if IAM auth is enabled

**DynamoDB authorization checks**

DynamoDB permissions are usually straightforward, but failures still happen when the resource ARN or action is wrong.

- Table-level actions often require the table ARN
- Some operations also need index ARNs
- Streams require separate permissions
- PartiQL and batch operations may need multiple actions

Example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:PutItem",
        "dynamodb:UpdateItem"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders"
    }
  ]
}
```

If the application queries a global secondary index, you may also need the index ARN in the `Resource` list.

**CloudTrail as the source of truth**

CloudTrail records the API call, the principal, the source IP, the request parameters, and the error code. It is one of the best tools for diagnosing authorization failures.

Look for:

- `errorCode`
- `errorMessage`
- `userIdentity`
- `eventSource`
- `eventName`
- `requestParameters`
- `resources`

Example CloudTrail event fields to inspect:

```json
{
  "eventSource": "s3.amazonaws.com",
  "eventName": "GetObject",
  "errorCode": "AccessDenied",
  "userIdentity": {
    "type": "AssumedRole",
    "arn": "arn:aws:sts::123456789012:assumed-role/MyAppRole/debug-session"
  }
}
```

This tells you the request came from an assumed role session, which means you should inspect the role policy, session policy, permissions boundary, and any resource policy on the bucket.

**IAM policy simulator and Access Analyzer**

Use AWS tools to validate your reasoning before changing production access.

- **IAM policy simulator**
  - Tests whether a principal can perform an action on a resource
  - Useful for identity policies and some context-based checks
- **Access Analyzer**
  - Helps identify unintended public or cross-account access
  - Useful for reviewing resource policies and external access paths

These tools are especially helpful when the policy is large or when multiple conditions interact.

**Practical troubleshooting checklist**

1. Confirm the exact error text and service.
2. Identify the principal and whether it is temporary credentials.
3. Verify the action name and resource ARN.
4. Search for explicit denies in all policy layers.
5. Check service-specific requirements such as KMS, ACLs, or endpoint policies.
6. Reproduce with the AWS CLI using the same identity.
7. Inspect CloudTrail for the denied event.
8. Use the IAM policy simulator to validate the expected result.
9. Fix the narrowest policy layer possible.
10. Retest and confirm the request succeeds without broadening access unnecessarily.

**Exam-focused distinctions**

The exam often tests whether you can tell similar failures apart.

| Symptom | Likely cause | What to check first |
|---|---|---|
| `AccessDenied` on S3 object read | Missing `s3:GetObject` or bucket policy deny | Object ARN and bucket policy |
| `AccessDenied` on S3 bucket listing | Missing `s3:ListBucket` | Bucket ARN and prefix condition |
| `UnauthorizedOperation` in EC2 | Missing EC2 permission | Identity policy and SCP |
| `KMS.AccessDeniedException` | Missing KMS permission or key policy access | `kms:Decrypt`, key policy, grant |
| `403 Forbidden` from API Gateway | Auth config, resource policy, or signing issue | IAM auth, resource policy, SigV4 |
| Works in console but not app | Different role, session policy, or endpoint path | Runtime credentials and CloudTrail |

**Real-world debugging example**

Suppose a Lambda function reads an encrypted object from S3 and fails with `AccessDenied`.

The likely checks are:

- The Lambda execution role has `s3:GetObject`
- The bucket policy allows the role or account
- The object key matches the allowed prefix
- The KMS key policy allows the role to decrypt
- The role is not restricted by a permissions boundary
- The request is going through the expected VPC endpoint, if applicable

A minimal execution role policy might look like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/data/*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ef-abcd-1234567890ab"
    }
  ]
}
```

If the error persists, the next most likely cause is a bucket policy deny or a missing KMS key policy statement.

**Key takeaways**

- **AccessDenied** means authorization failed somewhere in the evaluation chain.
- An explicit deny overrides every allow.
- Always verify the action, resource ARN, principal, and policy layer.
- KMS, S3, API Gateway, and cross-account access are frequent failure points.
- CloudTrail and the IAM policy simulator are the fastest tools for diagnosis.
- Fix the narrowest policy or trust relationship that blocks the request.
