## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why generated roles fail**

AWS Cloud Development Kit (CDK) and AWS Serverless Application Model (SAM) often create IAM roles for you. That convenience is useful, but it also hides the exact trust policy, inline permissions, and logical IDs that CloudFormation manages. When a function or service starts returning `AccessDenied`, the root cause is often one of three things:

- The generated role does not include the action you need.
- The wrong role is attached, overridden, or replaced.
- A different permission boundary exists at the account, stack, or resource level.

For exam purposes, you should separate three permission paths:

- **Execution role permissions**: what the Lambda function, ECS task, or other service can do at runtime.
- **Invoke permissions**: who or what can call the function or service.
- **Resource policies**: permissions attached to the target resource, such as S3 bucket policies or API Gateway resource policies.

A common mistake is to fix the wrong layer. For example, `AccessDenied` inside Lambda execution usually means the function role is missing permissions, while `403 Forbidden` from API Gateway may mean the caller lacks invoke permission or the API resource policy blocks access.

**How CDK generates and manages roles**

CDK synthesizes CloudFormation templates from your code. If you do not provide a role explicitly, many constructs create a default role with permissions inferred from helper methods such as `grantRead`, `grantWrite`, or `grantReadWrite`.

Typical CDK patterns include:

- Using a construct-provided default role.
- Supplying a custom role with the `role` property.
- Adding permissions with `grant*` helpers.
- Applying a permissions boundary to constrain future permissions.

Example Lambda with an inferred execution role:

```ts
import * as lambda from 'aws-cdk-lib/aws-lambda';
import * as s3 from 'aws-cdk-lib/aws-s3';

const bucket = new s3.Bucket(this, 'DataBucket');

const fn = new lambda.Function(this, 'ProcessorFn', {
  runtime: lambda.Runtime.NODEJS_20_X,
  handler: 'index.handler',
  code: lambda.Code.fromAsset('lambda'),
});

bucket.grantRead(fn);
```

This helper adds the minimum permissions needed for the specific grant operation, but it is still your responsibility to verify whether the function truly needs read-only, write-only, or broader access. A helper is not automatically safer than a custom policy; it is simply a convenience that reduces boilerplate.

**CDK role customization options**

CDK gives you several ways to control generated roles. Use the least invasive option that still meets the requirement.

- **Default generated role**
  - Best when the construct can infer permissions cleanly.
  - Fastest path for simple single-account workloads.
- **`role` property override**
  - Best when you need a pre-created role, cross-account trust, or strict governance.
  - Replaces the generated execution role for that construct.
- **Inline policy or grant helpers**
  - Best when you want CDK to manage the role but still add targeted permissions.
- **Permissions boundary**
  - Best when your organization requires a hard upper limit on what the role can ever receive.

Example custom role in CDK:

```ts
import * as iam from 'aws-cdk-lib/aws-iam';
import * as lambda from 'aws-cdk-lib/aws-lambda';

const role = new iam.Role(this, 'FnRole', {
  assumedBy: new iam.ServicePrincipal('lambda.amazonaws.com'),
  managedPolicies: [
    iam.ManagedPolicy.fromAwsManagedPolicyName('service-role/AWSLambdaBasicExecutionRole'),
  ],
});

role.addToPolicy(new iam.PolicyStatement({
  actions: ['dynamodb:GetItem', 'dynamodb:PutItem'],
  resources: ['arn:aws:dynamodb:us-east-1:123456789012:table/Orders'],
}));

new lambda.Function(this, 'Fn', {
  runtime: lambda.Runtime.NODEJS_20_X,
  handler: 'index.handler',
  code: lambda.Code.fromAsset('lambda'),
  role,
});
```

If you supply `role`, CDK uses that role instead of generating a new one for the function. That means your custom role must already trust the service principal and include the required permissions.

**SAM generated roles and policy templates**

SAM can generate execution roles when you omit `Role` on a resource such as `AWS::Serverless::Function`. In that case, SAM expands policy templates and policy statements into the generated role during transform.

Important exam distinction:

- If you use `Policies`, SAM uses them only when it is generating the execution role.
- If you specify `Role`, SAM does not generate or attach execution-role permissions from `Policies` for that function.
- Policy templates such as `S3ReadPolicy` or `DynamoDBCrudPolicy` are transformed into IAM statements, not attached as managed policies.

Example SAM function with generated role:

```yaml
Resources:
  ProcessorFunction:
    Type: AWS::Serverless::Function
    Properties:
      Runtime: nodejs20.x
      Handler: app.handler
      CodeUri: src/
      Policies:
        - S3ReadPolicy:
            BucketName: !Ref DataBucket
        - DynamoDBCrudPolicy:
            TableName: !Ref OrdersTable
```

Example SAM function with a custom role:

```yaml
Resources:
  ProcessorRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              Service: lambda.amazonaws.com
            Action: sts:AssumeRole
      ManagedPolicyArns:
        - arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

  ProcessorFunction:
    Type: AWS::Serverless::Function
    Properties:
      Runtime: nodejs20.x
      Handler: app.handler
      CodeUri: src/
      Role: !GetAtt ProcessorRole.Arn
      Policies:
        - S3ReadPolicy:
            BucketName: !Ref DataBucket
```

In this case, `Policies` are not used to build the function’s execution role because `Role` is explicitly set. This is a common exam trap.

**CDK versus SAM role behavior**

| Topic | CDK | SAM |
|---|---|---|
| Default role generation | Common across many constructs | Common for serverless resources when `Role` is omitted |
| Permission helpers | `grantRead`, `grantWrite`, `grantReadWrite`, `addToRolePolicy` | `Policies`, policy templates, and explicit `Role` |
| Custom role override | `role` property on constructs | `Role` property on resources |
| Policy expansion | CDK synthesizes IAM resources directly | SAM transform expands templates into IAM statements |
| Common confusion | Helper grants may be broader than needed | `Policies` are ignored for execution-role generation when `Role` is set |

**How to inspect what was generated**

Do not assume the source code is the final IAM shape. Inspect the synthesized or transformed template.

For CDK:

```bash
cdk synth
```

This shows the CloudFormation template CDK generates. If you need to inspect the deployed stack, use CloudFormation or the AWS CLI after deployment.

For SAM:

```bash
sam build
```

Then inspect the transformed template in `.aws-sam/build/template.yaml`. `sam build --debug` is useful for verbose diagnostics, but it is not the primary way to inspect the transformed template.

Useful inspection commands:

```bash
aws cloudformation get-template \
  --stack-name my-stack \
  --template-stage Processed
```

```bash
aws iam get-role --role-name MyGeneratedRole
aws iam list-role-policies --role-name MyGeneratedRole
aws iam get-role-policy --role-name MyGeneratedRole --policy-name inline-policy-name
```

If the role is generated by CloudFormation, the final role name may be derived from the stack and logical ID. If the construct path changes, the logical ID can change too, which may create a new role on redeploy.

**Common failure symptoms and the likely fix**

- `AccessDeniedException` from Lambda code
  - The execution role lacks the required API action.
  - Fix by adding the missing permission to the function role or the resource grant.
- `User is not authorized to perform: lambda:InvokeFunction`
  - The caller lacks invoke permission.
  - Fix by adding a Lambda resource policy or the correct IAM permission to the caller.
- `403 Forbidden` from API Gateway
  - The API resource policy, authorizer, or invoke permission is wrong.
  - Fix the API Gateway resource policy or caller authorization path.
- `The role defined for the function cannot be assumed by Lambda`
  - The trust policy is wrong.
  - Fix the `Principal` to `lambda.amazonaws.com` for Lambda execution roles.
- `CloudFormation cannot update a stack because a resource with the same name already exists`
  - A role name or logical ID change caused replacement.
  - Fix naming, retain policies, or remove hard-coded names if replacement is not intended.

**Troubleshooting workflow**

1. Identify the failing layer.
   - Runtime access failure points to the execution role.
   - Invocation failure points to caller permissions or resource policy.
2. Inspect the synthesized or transformed template.
   - Use `cdk synth` or `sam build` and review the generated IAM resources.
3. Compare the intended permissions with the actual statements.
   - Check actions, resources, and conditions.
4. Check trust policy and boundaries.
   - Ensure the service principal can assume the role.
   - Ensure a permissions boundary is not blocking the action.
5. Redeploy and verify.
   - Confirm the stack update changed the generated role as expected.

**Role drift, logical IDs, and redeploy behavior**

Generated roles can appear to “drift” when the template changes, but the real issue is often a change in logical ID or construct path.

- In CDK, changing a construct ID or moving a construct in the tree can change the logical ID.
- In SAM, transform output can change when resource properties change, which can alter generated IAM resources.
- CloudFormation updates in place when possible, but it replaces resources when immutable properties change or when the logical identity changes.

Practical implications:

- If the role name is auto-generated, redeploying after a logical ID change may create a new role.
- If you hard-code a role name, replacement can fail because the old name still exists.
- If you need stable identity, keep construct paths stable and avoid unnecessary renames.

**Permissions boundaries and governance**

A permissions boundary limits the maximum permissions a role can receive, even if your template grants more. This is useful in enterprise environments where developers can create roles but cannot exceed a policy ceiling.

Example CDK permissions boundary:

```ts
const role = new iam.Role(this, 'BoundedRole', {
  assumedBy: new iam.ServicePrincipal('lambda.amazonaws.com'),
  permissionsBoundary: iam.ManagedPolicy.fromManagedPolicyArn(
    this,
    'Boundary',
    'arn:aws:iam::123456789012:policy/DeveloperBoundary',
  ),
});
```

If a permission is missing even though the template looks correct, check whether a boundary, SCP (Service Control Policy), or session policy is blocking it.

**Cross-account assumptions and trust policy fixes**

When a generated role must be assumed across accounts, the trust policy must explicitly allow the external principal or role. This is not the same as granting permissions on the target resource.

Example trust policy for cross-account assumption:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/ExternalCallerRole"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Use cross-account roles when one account needs to act in another account. Do not confuse this with Lambda execution permissions, which still require the service principal trust relationship.

**End-to-end example: fixing a broken Lambda role**

Suppose a Lambda function reads from S3 and writes to DynamoDB, but it fails with `AccessDeniedException` on `dynamodb:PutItem`.

1. Inspect the generated role.
   - In CDK, run `cdk synth` and locate the Lambda role policy.
   - In SAM, run `sam build` and inspect `.aws-sam/build/template.yaml`.
2. Confirm the missing action.
   - The role may include `dynamodb:GetItem` but not `dynamodb:PutItem`.
3. Add the narrowest fix.
   - In CDK, add `table.grantWriteData(fn)` or a scoped `addToRolePolicy`.
   - In SAM, add a policy template or explicit policy statement.
4. Redeploy and retest.
   - Confirm the function now succeeds without broadening unrelated permissions.

Example CDK fix:

```ts
table.grantWriteData(fn);
```

Example SAM fix:

```yaml
Policies:
  - Statement:
      - Effect: Allow
        Action:
          - dynamodb:PutItem
        Resource: !GetAtt OrdersTable.Arn
```

**Exam tips**

- Treat generated roles as implementation details you must verify, not trust blindly.
- Remember that `Policies` in SAM are only used when SAM generates the role.
- Remember that `Role` overrides generated execution-role permissions in SAM.
- Remember that CDK helper grants are convenience methods, not a guarantee of least privilege.
- Inspect synthesized output before assuming the template matches your intent.
- Separate execution-role problems from invoke-permission problems before changing IAM.

**Quick reference**

- Use `cdk synth` to inspect CDK output.
- Use `sam build` and inspect `.aws-sam/build/template.yaml` for SAM output.
- Use `Role` in SAM or `role` in CDK when you need a fully custom execution role.
- Use policy templates and grant helpers when you want concise, maintainable permissions.
- Check trust policies, permissions boundaries, and logical ID changes when a role behaves unexpectedly.
