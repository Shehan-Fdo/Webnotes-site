## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why validation matters**

When an AWS call fails, you need to determine whether the problem is identity, permissions, region, endpoint resolution, request signing, payload shape, or service state. The exam often tests whether you can choose the fastest reliable way to prove what is wrong.

A good troubleshooting flow is:

- Confirm the caller identity.
- Confirm the region and endpoint.
- Confirm the credentials source.
- Confirm permissions and policy boundaries.
- Confirm the resource state and eventual consistency.
- Reproduce the request with the simplest tool that still preserves the failure.

**Core validation tools**

You typically validate AWS behavior with three layers of tooling:

- **AWS CLI** for fast, repeatable checks from a shell.
- **AWS SDKs** for application-level behavior and credential/region resolution.
- **Raw HTTP requests** for endpoint, signing, and protocol-level debugging.

Each layer answers a different question:

- CLI tells you whether the environment and IAM permissions work.
- SDK tells you whether your application code resolves configuration correctly.
- HTTP tells you whether the service endpoint, request shape, and signing are correct.

**Identity and configuration resolution**

Before you debug a service call, verify which identity and region the tool is actually using. Many failures come from the wrong profile, stale environment variables, or an unexpected role assumption path.

Common AWS CLI and SDK configuration sources include:

- Environment variables such as `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`, `AWS_REGION`, and `AWS_DEFAULT_REGION`.
- Shared credentials file at `~/.aws/credentials`.
- Shared config file at `~/.aws/config`.
- IAM role credentials from an EC2 instance profile, ECS task role, or Lambda execution role.
- AWS IAM Identity Center (SSO) cached credentials.
- Explicit profile selection with `--profile` or SDK profile settings.

Typical precedence is:

1. Explicit parameters in code or CLI flags.
2. Environment variables.
3. Shared config and credentials files.
4. Role-based credentials from the runtime environment.
5. SSO or other provider-specific sources, depending on the SDK and setup.

Use these commands to inspect the active identity and region:

```bash
aws sts get-caller-identity
aws configure list
aws configure list-profiles
aws sts get-caller-identity --profile dev
aws s3 ls --region us-east-1
```

`aws configure list` is especially useful because it shows where each setting came from. If the wrong account or region is being used, check for:

- A profile mismatch.
- An exported environment variable overriding the profile.
- A default region in `~/.aws/config`.
- A role assumption chain that changed the effective identity.
- SSO login expiration.

> If a CLI or SDK call uses the wrong identity, fix configuration first. Debugging permissions before confirming identity often wastes time.

**CLI validation patterns**

The AWS CLI is the fastest way to validate access because it gives you a direct, human-readable response. Use the smallest command that exercises the same permission path as the application.

Common patterns:

- `aws sts get-caller-identity` confirms the caller identity without requiring service-specific permissions.
- `aws <service> help` confirms the CLI is installed and the command exists.
- `aws <service> <operation> --region <region>` validates region-specific behavior.
- `--debug` shows endpoint resolution, signing details, retries, and credential provider selection.

Example:

```bash
aws sts get-caller-identity --debug
aws dynamodb list-tables --region us-west-2
aws sqs get-queue-url --queue-name orders --region us-east-1
aws sns list-topics --region us-east-1
aws apigateway get-rest-apis --region us-east-1
aws logs describe-log-groups --region us-east-1
```

Use `--debug` when you need to see:

- Which credential provider won.
- Which region the SDK or CLI selected.
- Which endpoint was resolved.
- Whether the request was signed.
- Whether retries or redirects occurred.

**Interpreting common CLI outcomes**

A single command can reveal several classes of problems, but the response is not always perfectly specific.

- `AccessDenied` or `UnauthorizedOperation` usually means IAM denied the action, but it can also involve a permissions boundary, session policy, SCP, or resource-based policy.
- `ResourceNotFoundException` often means the resource name, region, or account is wrong.
- `InvalidSignatureException` or signature mismatch errors usually indicate bad signing, clock skew, or the wrong region/endpoint.
- `ThrottlingException` means the request is valid but rate-limited.
- `ValidationException` usually means the request shape or parameter values are invalid.

For S3, `head-bucket` and `ls` can help, but they are not perfect truth machines:

```bash
aws s3 ls s3://my-bucket
aws s3api head-bucket --bucket my-bucket
aws s3api get-bucket-location --bucket my-bucket
```

Use them carefully:

- `aws s3 ls s3://my-bucket` is useful for a quick basic check.
- `aws s3api head-bucket` can return ambiguous 403/404-style outcomes depending on permissions and service behavior.
- `get-bucket-location` helps confirm region when you suspect a region mismatch.

For region-sensitive services, always test with the intended region explicitly:

```bash
aws dynamodb describe-table --table-name Orders --region us-east-1
aws lambda get-function --function-name process-order --region us-east-1
aws sqs get-queue-attributes --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders --attribute-names All
```

**SDK validation patterns**

SDKs are the best way to validate how application code resolves credentials, region, retries, and endpoint settings. They are also the best way to reproduce a production issue in a controlled script.

SDK troubleshooting often focuses on:

- Credential provider chain resolution.
- Region resolution.
- Retry behavior.
- Serialization and deserialization.
- Endpoint overrides.
- Timeouts and connection settings.

A minimal Python example:

```python
import boto3

session = boto3.Session(profile_name="dev", region_name="us-east-1")
sts = session.client("sts")
print(sts.get_caller_identity())
```

A minimal Node.js example:

```javascript
import { STSClient, GetCallerIdentityCommand } from "@aws-sdk/client-sts";

const client = new STSClient({ region: "us-east-1" });
const response = await client.send(new GetCallerIdentityCommand({}));
console.log(response);
```

A minimal Java example:

```java
StsClient sts = StsClient.builder()
    .region(Region.US_EAST_1)
    .build();

System.out.println(sts.getCallerIdentity());
```

Use SDKs when you need to verify:

- The application is loading the expected profile or role.
- The region is set correctly in code.
- The app can reach the correct endpoint.
- The request fails only in the application, not in the shell.
- Retries or timeouts are masking the original error.

**Credential provider chain troubleshooting**

If the SDK uses the wrong identity, inspect the provider chain in the runtime environment. Common sources include:

- Explicit credentials in code.
- Environment variables.
- Shared config and credentials files.
- Container or instance metadata credentials.
- Web identity or role assumption flows.
- SSO cached sessions.

Practical checks:

```bash
env | grep '^AWS_'
cat ~/.aws/config
cat ~/.aws/credentials
aws sts get-caller-identity --profile dev
aws sso login --profile dev
```

In code, avoid hardcoding credentials. Prefer the default provider chain or an explicit role/profile only when needed. If you must override, do it intentionally and document why.

**Region and endpoint validation**

Many AWS errors are really endpoint problems. A request can fail because the client is pointed at the wrong region, the wrong partition, or a custom endpoint that does not match the service.

Validate region explicitly:

```bash
aws s3api list-buckets --region us-east-1
aws dynamodb list-tables --region eu-west-1
aws lambda list-functions --region ap-southeast-2
```

When debugging endpoint resolution, check:

- The configured region.
- Any custom endpoint override.
- Whether the service is global or regional.
- Whether the request is using the correct partition, such as `aws`, `aws-cn`, or `aws-us-gov`.

Some services are global or have special endpoint behavior, so do not assume every service behaves the same way. For example, IAM and Route 53 are global in many contexts, while S3, DynamoDB, Lambda, SQS, SNS, and CloudWatch are region-scoped for most operations.

If you use raw HTTP, confirm the host matches the service and region:

```bash
curl -I https://s3.us-east-1.amazonaws.com/
curl -I https://dynamodb.us-east-1.amazonaws.com/
curl -I https://lambda.us-east-1.amazonaws.com/
```

These checks are useful for connectivity and DNS validation, but they do not prove authorization.

**Raw HTTP validation and SigV4**

Many AWS service APIs require <abbr title="Signature Version 4">SigV4</abbr> signing. A plain `curl` request is often not enough unless the endpoint is public or the operation supports anonymous access. For authenticated calls, you need the correct headers, canonical request, and signature.

Use raw HTTP when you need to validate:

- DNS and network reachability.
- TLS certificate and proxy behavior.
- Endpoint correctness.
- Request signing details.
- Header canonicalization and payload hashing.

A signed request example using `curl` with an external signer or proxy is often easier than hand-building SigV4. One common pattern is to use a signing helper such as `aws4` or a local proxy that signs requests before forwarding them.

Example conceptual flow:

```bash
# Pseudocode-style example: sign the request before sending it
curl -X POST "https://dynamodb.us-east-1.amazonaws.com/" \
  -H "Content-Type: application/x-amz-json-1.0" \
  -H "X-Amz-Target: DynamoDB_20120810.ListTables" \
  -H "Authorization: AWS4-HMAC-SHA256 Credential=..., SignedHeaders=..., Signature=..." \
  -H "X-Amz-Date: 20260426T120000Z" \
  -d '{}'
```

For a real validation workflow, use the AWS CLI or SDK to generate the request first, then compare the endpoint, headers, and region with your raw HTTP attempt. If the CLI works but `curl` fails, the issue is usually signing, headers, or payload formatting.

**Practical service validation examples**

Different services expose different failure modes, so choose a command that matches the resource you are testing.

- **S3**: validate bucket access, region, and object permissions.
  ```bash
  aws s3 ls s3://my-bucket
  aws s3api get-object --bucket my-bucket --key path/to/file.txt /tmp/file.txt
  aws s3api head-object --bucket my-bucket --key path/to/file.txt
  ```
- **DynamoDB**: validate table existence, read/write permissions, and region.
  ```bash
  aws dynamodb describe-table --table-name Orders --region us-east-1
  aws dynamodb put-item --table-name Orders --item '{"id":{"S":"123"}}' --region us-east-1
  ```
- **Lambda**: validate function existence, invoke permissions, and alias/version targeting.
  ```bash
  aws lambda get-function --function-name process-order --region us-east-1
  aws lambda invoke --function-name process-order --payload '{}' response.json --region us-east-1
  ```
- **SQS**: validate queue URL resolution and message permissions.
  ```bash
  aws sqs get-queue-url --queue-name orders --region us-east-1
  aws sqs send-message --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders --message-body "test"
  ```
- **SNS**: validate topic access and publish permissions.
  ```bash
  aws sns list-topics --region us-east-1
  aws sns publish --topic-arn arn:aws:sns:us-east-1:123456789012:orders --message "test"
  ```
- **API Gateway**: validate API existence and stage/route behavior.
  ```bash
  aws apigateway get-rest-apis --region us-east-1
  curl -i https://abc123.execute-api.us-east-1.amazonaws.com/prod/health
  ```
- **CloudWatch Logs**: validate log group access and region.
  ```bash
  aws logs describe-log-groups --region us-east-1
  aws logs filter-log-events --log-group-name /aws/lambda/process-order --region us-east-1
  ```

**Permissions troubleshooting beyond IAM identity**

A valid identity does not guarantee access. When a request is denied, check the full authorization path.

Possible blockers include:

- **Identity-based policies** attached to the user or role.
- **Resource-based policies** on the target resource, such as Lambda permissions, S3 bucket policies, SNS topic policies, or SQS queue policies.
- **Permissions boundaries** that cap what an identity can do.
- **Service control policies** in AWS Organizations that deny actions at the account or OU level.
- **Session policies** applied during role assumption.
- **KMS key policies** when the resource is encrypted with AWS Key Management Service.

A useful exam mindset is:

- If the identity is correct but access is denied, do not stop at the IAM policy attached to the role.
- Check whether the resource itself allows the caller.
- Check whether an organization-level policy blocks the action.
- Check whether encryption introduces an additional permission requirement.

Example: an S3 `GetObject` call may fail because the bucket policy denies access, the object is encrypted with a KMS key you cannot use, or an SCP blocks the action even though the role policy allows it.

**Eventual consistency, waiters, and polling**

Some AWS resources are not immediately available after creation or update. Validation may require waiting for the resource to reach the expected state.

Use waiters when the SDK or CLI provides them:

```bash
aws ec2 wait instance-running --instance-ids i-0123456789abcdef0
aws rds wait db-instance-available --db-instance-identifier mydb
aws lambda wait function-active --function-name process-order
```

If no waiter exists, poll the resource state with backoff:

```bash
for i in 1 2 3 4 5; do
  aws dynamodb describe-table --table-name Orders --region us-east-1 && break
  sleep $((i * 2))
done
```

Waiters are useful because they encode service-specific retry logic and reduce brittle custom loops. Polling is still useful when you need custom conditions or when a waiter is unavailable.

**Pagination and partial results**

Some list operations return only a subset of results. If you validate with a paginated API and ignore the next token, you may think the resource is missing when it is simply on another page.

Use pagination-aware commands:

```bash
aws s3api list-objects-v2 --bucket my-bucket --max-items 10
aws dynamodb scan --table-name Orders --limit 10
aws logs describe-log-groups --limit 50
aws sqs list-queues --max-results 10
```

In SDKs, use paginators rather than manual loops when possible. This is especially important when validating large environments or confirming that a resource exists among many similar resources.

**Practical troubleshooting workflow**

1. Confirm the caller identity with `aws sts get-caller-identity`.
2. Confirm the region with `aws configure list` or an explicit `--region`.
3. Reproduce the issue with the smallest CLI command that touches the same resource.
4. Add `--debug` to inspect endpoint resolution and signing.
5. Check identity-based policies, resource-based policies, permissions boundaries, SCPs, and KMS permissions.
6. If the resource was just created or updated, wait or poll for eventual consistency.
7. If CLI works but the app fails, compare SDK configuration, environment variables, and code-level region or credential overrides.
8. If HTTP fails but CLI/SDK works, inspect SigV4 headers, host, path, and payload formatting.

**Exam-oriented takeaways**

- Use `sts get-caller-identity` to confirm who the tool is acting as.
- Use explicit regions to avoid false negatives from region mismatch.
- Use `--debug` to inspect credential resolution and endpoint selection.
- Remember that access errors can come from more than one policy layer.
- Use waiters or polling for resources that are not immediately consistent.
- Use paginated reads when validating large result sets.
- Use raw HTTP to isolate endpoint and signing issues, but do not assume plain `curl` is sufficient for authenticated AWS APIs.

**Quick reference**

| Problem | Best first check | Why |
|---|---|---|
| Wrong account or role | `aws sts get-caller-identity` | Confirms the effective identity |
| Wrong region | `aws configure list` or explicit `--region` | Region mismatch is a common cause of false failures |
| CLI works, app fails | SDK config and provider chain | The app may resolve credentials differently |
| CLI fails with access denied | IAM, resource policy, boundary, SCP, KMS | Denial can happen at multiple layers |
| Resource not found | Region, account, name, pagination | The resource may exist elsewhere or on another page |
| Raw HTTP fails | SigV4, headers, endpoint, payload | HTTP requests are sensitive to signing and formatting |
