## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Endpoint design and integration choices**

API Gateway is the front door for serverless APIs and a common integration layer for Lambda, AWS services, and HTTP backends. For exam scenarios, you need to choose the right API type and integration pattern based on features, latency, and operational overhead.

- **REST APIs** support resources, methods, request/response models, mapping templates, usage plans, API keys, and richer transformation features.
- **HTTP APIs** are lighter weight, lower cost, and simpler to configure, but they expose fewer advanced features.
- **WebSocket APIs** are for bidirectional, stateful communication and are not the default choice for standard CRUD endpoints.
- Use **Lambda proxy integration** when you want API Gateway to pass the request through with minimal transformation and let Lambda handle most of the logic.
- Use non-proxy integrations when you need request/response mapping, legacy backend adaptation, or fine-grained transformation.

A typical REST API request path is not “API Gateway -> stage -> Lambda” as separate hops. The stage is a deployment and routing context, while the request is matched to a resource and method, then forwarded to the integration.

**API Gateway configuration concepts**

For REST APIs, the exam often tests how resources, methods, integrations, and deployments fit together. You define a resource such as `/orders`, attach methods like `GET` or `POST`, configure the integration, and then deploy the API to a stage.

- **Resource**: A path component such as `/users/{id}`.
- **Method**: The HTTP verb attached to a resource, such as `GET`, `POST`, or `DELETE`.
- **Integration**: The backend target, such as Lambda, an HTTP endpoint, or an AWS service.
- **Deployment**: A snapshot of the API configuration.
- **Stage**: A named deployment environment such as `dev`, `test`, or `prod`.

Common REST API authoring features include:

- **Method request** settings for headers, query strings, path parameters, and authorization.
- **Integration request** settings for mapping and backend invocation.
- **Method response** settings for status codes and response models.
- **Integration response** settings for mapping backend responses to method responses.

These features matter when you need to transform payloads, return different status codes, or adapt a backend that does not speak HTTP in the shape your clients expect.

**Authorization and endpoint security**

API Gateway supports multiple authorization patterns, and the right choice depends on who calls the API and how much control you need.

- **IAM authorization** is best when AWS principals call the API and you want SigV4-signed requests with fine-grained IAM policy control.
- **Cognito user pools authorizer** is useful when end users authenticate with Cognito and you want JWT-based user authentication.
- **Lambda authorizers** are useful when you need custom authorization logic, token introspection, or integration with an external identity system.
- **No auth** is acceptable only for public endpoints that are intentionally open, usually with other controls such as throttling or WAF.

Practical selection guidance:

- Use **IAM auth** for service-to-service access inside AWS.
- Use **Cognito** when the API is user-facing and you want managed user sign-in.
- Use a **Lambda authorizer** when the token format or policy logic is custom.
- Combine authorization with **resource policies** when you need to restrict access by source VPC endpoint, account, or IP range.

Example IAM-signed request:

```bash
aws apigateway test-invoke-method \
  --rest-api-id abc123 \
  --resource-id xyz789 \
  --http-method GET \
  --path-with-query-string "/orders/42"
```

Example Lambda authorizer flow:

```json
{
  "type": "TOKEN",
  "identitySource": "method.request.header.Authorization",
  "authorizerUri": "arn:aws:apigateway:us-east-1:lambda:path/2015-03-31/functions/arn:aws:lambda:us-east-1:123456789012:function:AuthFn/invocations",
  "authorizerResultTtlInSeconds": 300
}
```

**CORS, custom domains, and usage plans**

Cross-Origin Resource Sharing (CORS) is required when browsers call your API from a different origin. API Gateway can return the necessary `Access-Control-Allow-*` headers, but you must configure both preflight and actual responses correctly.

- Add an `OPTIONS` method for preflight requests when needed.
- Return `Access-Control-Allow-Origin` with a specific origin or `*` when appropriate.
- Include `Access-Control-Allow-Headers` and `Access-Control-Allow-Methods` that match the browser request.
- Remember that credentials and wildcard origins do not mix in the usual browser rules.

Example CORS response headers:

```http
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET,POST,OPTIONS
Access-Control-Allow-Headers: Content-Type,Authorization
Access-Control-Allow-Credentials: true
```

Custom domains let you present a stable API hostname and map it to one or more APIs.

- For REST APIs, you use **base path mappings**.
- For HTTP APIs, you use **API mappings**.
- The mapping routes a custom domain path to an API and stage; it is not an extra runtime hop.

Usage plans and API keys are REST API features used for throttling and quota management for consumers.

- Use them when you need per-client limits or metering.
- Do not treat API keys as strong authentication; they are identifiers for usage control, not user identity.
- Pair them with authorization if the API must be protected.

**Lambda integration patterns**

Lambda is the most common backend for API endpoints in exam scenarios. The key distinction is whether API Gateway uses proxy or non-proxy integration.

- In **proxy integration**, API Gateway forwards the request context and payload to Lambda and expects Lambda to return a structured response.
- In **non-proxy integration**, API Gateway can transform the request and response with mapping templates and integration responses.

A typical Lambda proxy response looks like this:

```json
{
  "statusCode": 200,
  "headers": {
    "Content-Type": "application/json"
  },
  "body": "{\"message\":\"ok\"}",
  "isBase64Encoded": false
}
```

A minimal Node.js handler:

```javascript
export const handler = async (event) => {
  const id = event.pathParameters?.id;
  return {
    statusCode: 200,
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ id, message: "retrieved" })
  };
};
```

Important behavior to remember:

- Unhandled Lambda errors in proxy integrations commonly surface as `502 Bad Gateway` from API Gateway.
- If the Lambda integration times out, API Gateway commonly returns `504 Gateway Timeout`.
- API Gateway can reject requests before Lambda runs if authorization, validation, throttling, or request mapping fails.
- Lambda timeout and API Gateway integration timeout are separate limits.

Example of a timeout-aware Lambda setting:

```yaml
Resources:
  ApiHandler:
    Type: AWS::Lambda::Function
    Properties:
      Timeout: 10
      MemorySize: 256
```

**Request and response transformation**

Non-proxy integrations are useful when you need to reshape data between client and backend. This is common with legacy HTTP services or when you want to hide backend details.

- **Mapping templates** use Velocity Template Language (VTL) to transform requests and responses.
- **Models** define expected payload shapes for validation and documentation.
- **Request validation** can enforce required parameters or body schemas before the backend is invoked.

Example request mapping template:

```vtl
{
  "orderId": "$input.params('id')",
  "body": $input.json('$'),
  "requestId": "$context.requestId"
}
```

Example response mapping idea:

- Map backend `404` to API Gateway `404`.
- Map backend validation errors to `400`.
- Map backend throttling or dependency failures to appropriate 5xx responses.

When to use these features:

- Use **models** when clients need a contract and you want API Gateway to validate input.
- Use **mapping templates** when the backend expects a different shape than the client sends.
- Use **integration responses** when the backend status codes do not align with your public API contract.

**DynamoDB as the primary persistence layer**

Amazon DynamoDB is the most common persistence choice for serverless APIs because it scales without server management and integrates well with Lambda. It is a good fit when you know your access patterns and can model them up front.

- Design around **access patterns**, not around normalization.
- Use a **partition key** to distribute data and a **sort key** to model multiple items per partition.
- Use **secondary indexes** when you need alternate query patterns.

Core read/write operations:

- `GetItem` for a single item by primary key.
- `PutItem` to create or replace an item.
- `UpdateItem` for partial updates.
- `DeleteItem` for removal.
- `Query` for efficient key-based lookups.
- `Scan` only when you cannot avoid reading the whole table.

Example AWS SDK for JavaScript v3 query:

```javascript
import { DynamoDBClient } from "@aws-sdk/client-dynamodb";
import { DynamoDBDocumentClient, QueryCommand } from "@aws-sdk/lib-dynamodb";

const client = DynamoDBDocumentClient.from(new DynamoDBClient({}));

const result = await client.send(new QueryCommand({
  TableName: "Orders",
  KeyConditionExpression: "pk = :pk",
  ExpressionAttributeValues: {
    ":pk": "USER#123"
  }
}));
```

**Consistency, conditional writes, and optimistic locking**

DynamoDB reads are usually eventually consistent by default, but you can request strongly consistent reads on base table and local secondary index reads. Strong consistency is useful when you need the latest committed value immediately after a write.

- Use **eventually consistent reads** for higher throughput and lower cost when slight staleness is acceptable.
- Use **strongly consistent reads** when correctness depends on reading the latest write from the base table.
- Strongly consistent reads are not available for global secondary indexes.

Conditional writes are essential for concurrency control and business rules.

- Use `ConditionExpression` to prevent overwrites.
- Use conditions to enforce uniqueness, state transitions, or inventory limits.
- Use conditions with `PutItem`, `UpdateItem`, `DeleteItem`, and transactional writes.

Example conditional update:

```javascript
await client.send(new UpdateCommand({
  TableName: "Inventory",
  Key: { pk: "SKU#100", sk: "META" },
  UpdateExpression: "SET quantity = quantity - :n",
  ConditionExpression: "quantity >= :n",
  ExpressionAttributeValues: {
    ":n": 2
  }
}));
```

Optimistic locking uses a version attribute to detect concurrent updates.

- Read the item with a version number.
- Update only if the version matches.
- Increment the version on success.
- Retry on conditional failure if the business logic allows it.

Example versioned update:

```javascript
UpdateExpression: "SET #v = #v + :one, status = :s",
ConditionExpression: "#v = :expected",
ExpressionAttributeNames: {
  "#v": "version"
},
ExpressionAttributeValues: {
  ":one": 1,
  ":expected": 3,
  ":s": "PAID"
}
```

**Batch and transactional operations**

Batch and transactional APIs solve different problems.

- Use **BatchGetItem** when you need multiple items by key and can tolerate partial failures or retries.
- Use **BatchWriteItem** for bulk puts and deletes, but remember it does not support conditional logic.
- Use **TransactWriteItems** when multiple writes must succeed or fail together.
- Use **TransactGetItems** when you need an all-or-nothing read set.

Transactional writes are useful for:

- Maintaining inventory counts and order records together.
- Creating a record only if another record does not already exist.
- Updating multiple related items atomically.

Example transaction pattern:

```javascript
import { TransactWriteCommand } from "@aws-sdk/lib-dynamodb";

await client.send(new TransactWriteCommand({
  TransactItems: [
    {
      Update: {
        TableName: "Inventory",
        Key: { pk: "SKU#100", sk: "META" },
        UpdateExpression: "SET quantity = quantity - :n",
        ConditionExpression: "quantity >= :n",
        ExpressionAttributeValues: { ":n": 1 }
      }
    },
    {
      Put: {
        TableName: "Orders",
        Item: {
          pk: "ORDER#9001",
          sk: "META",
          sku: "SKU#100",
          quantity: 1
        },
        ConditionExpression: "attribute_not_exists(pk)"
      }
    }
  ]
}));
```

**DynamoDB capacity modes and scaling**

DynamoDB offers two main capacity modes, and exam questions often ask you to choose between them based on traffic predictability.

- **On-demand capacity** is best for unpredictable or spiky traffic.
- **Provisioned capacity** is best when you can estimate traffic and want tighter cost control.
- **Auto scaling** adjusts provisioned capacity based on utilization targets.

Behavior to remember:

- On-demand automatically handles sudden traffic changes, but you still can be throttled if you exceed service limits or hit hot partitions.
- Provisioned tables can throttle when read or write capacity is insufficient.
- Auto scaling helps provisioned tables adapt, but it is not instant and does not eliminate poor key design.

Practical guidance:

- Choose **on-demand** for new workloads, variable traffic, or uncertain sizing.
- Choose **provisioned** when traffic is predictable and you want to optimize cost.
- Use **adaptive capacity** and good partition key design to reduce hot partition risk.

**Indexes: GSI and LSI**

Secondary indexes let you query the same data by different keys. The exam often tests when to use a Global Secondary Index versus a Local Secondary Index.

| Feature | GSI | LSI |
|---|---|---|
| Partition key | Can differ from base table | Must match base table partition key |
| Sort key | Optional and can differ | Can differ from base table sort key |
| Capacity | Separate read/write capacity in provisioned mode | Shares base table provisioned throughput |
| Consistency | Eventually consistent only | Can be strongly consistent |
| Creation time | Can be added after table creation | Must be defined at table creation |

Use a **GSI** when you need a new access pattern with a different partition key. Use an **LSI** when you need alternate sorting within the same partition key and want strong consistency.

Example table creation with a GSI:

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions \
    AttributeName=pk,AttributeType=S \
    AttributeName=sk,AttributeType=S \
    AttributeName=gsi1pk,AttributeType=S \
    AttributeName=gsi1sk,AttributeType=S \
  --key-schema \
    AttributeName=pk,KeyType=HASH \
    AttributeName=sk,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST \
  --global-secondary-indexes '[
    {
      "IndexName": "GSI1",
      "KeySchema": [
        {"AttributeName": "gsi1pk", "KeyType": "HASH"},
        {"AttributeName": "gsi1sk", "KeyType": "RANGE"}
      ],
      "Projection": {"ProjectionType": "ALL"}
    }
  ]'
```

**RDS and Aurora for relational persistence**

Use Amazon RDS or Aurora when your data is relational, you need SQL joins, or your application depends on transactional relational modeling. This is common when the access pattern is not naturally key-value or when you need complex reporting.

- **RDS** is the managed relational database service for engines such as MySQL, PostgreSQL, MariaDB, Oracle, and SQL Server.
- **Aurora** is AWS’s cloud-optimized relational database with higher performance and better scaling characteristics than standard RDS for many workloads.
- **Aurora Serverless** is useful when you want relational semantics with more elastic capacity behavior.

Availability and scaling concepts:

- **Multi-AZ** provides high availability through synchronous replication and automatic failover.
- **Read replicas** help with read scaling and can also support disaster recovery patterns.
- **Aurora replicas** are typically faster to promote and are part of Aurora’s architecture.
- Use **Multi-AZ** for availability, not for read scaling.
- Use **read replicas** when read traffic is high and eventual consistency is acceptable.

RDS Proxy is useful when Lambda creates many short-lived connections.

- It helps with connection pooling and reduces database connection storms.
- It is especially valuable for bursty Lambda workloads.
- It is supported only for specific engines and configurations, so verify compatibility before choosing it.

Example connection string pattern:

```text
postgresql://app_user:password@db-proxy.example.us-east-1.rds.amazonaws.com:5432/appdb
```

**S3 for object persistence and file delivery**

Amazon S3 is a strong fit for blobs, uploads, downloads, and static assets. It is not a relational database, but it is often part of an API architecture for file handling.

Common patterns:

- Store uploaded files in S3 and metadata in DynamoDB or RDS.
- Return a **pre-signed URL** so clients upload directly to S3 without sending large files through Lambda.
- Return a pre-signed download URL for temporary access to private objects.
- Use **multipart upload** for large files to improve reliability and resume capability.

Example pre-signed upload flow:

```javascript
import { S3Client, PutObjectCommand } from "@aws-sdk/client-s3";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";

const s3 = new S3Client({});
const command = new PutObjectCommand({
  Bucket: "my-upload-bucket",
  Key: "uploads/user-123/photo.png",
  ContentType: "image/png"
});

const url = await getSignedUrl(s3, command, { expiresIn: 900 });
```

Important S3 details:

- Set the correct **Content-Type** so browsers and downstream consumers interpret the object correctly.
- Use **metadata** for application-specific attributes that should travel with the object.
- Use bucket policies and IAM to control access.
- Prefer direct-to-S3 uploads for large payloads to avoid Lambda memory and timeout pressure.

**Observability and troubleshooting**

You need to trace failures across API Gateway, Lambda, and the data store to diagnose endpoint issues quickly. The exam often expects you to know which logs and metrics reveal the problem.

- **API Gateway access logs** show request metadata, status codes, latency, and request IDs.
- **API Gateway execution logs** are more detailed and help with request/response mapping and integration issues.
- **Lambda logs** in CloudWatch Logs show application exceptions, timeouts, and custom diagnostics.
- **X-Ray** helps trace a request end-to-end across services when enabled.

Useful troubleshooting signals:

- `4xx` errors often indicate client input, auth, or validation problems.
- `5xx` errors often indicate backend failures, mapping issues, or timeouts.
- `429 Too Many Requests` indicates throttling.
- `502` from API Gateway often indicates malformed Lambda proxy output or unhandled Lambda failure.
- `504` often indicates an integration timeout.

Example structured logging in Lambda:

```javascript
console.log(JSON.stringify({
  requestId: event.requestContext?.requestId,
  path: event.path,
  user: event.requestContext?.authorizer?.claims?.sub ?? null
}));
```

Example X-Ray tracing in Lambda:

```javascript
import AWSXRay from "aws-xray-sdk-core";
import { DynamoDBClient } from "@aws-sdk/client-dynamodb";

const ddb = AWSXRay.captureAWSv3Client(new DynamoDBClient({}));
```

Practical debugging approach:

1. Check the API Gateway status code and request ID.
2. Review API Gateway logs for mapping, auth, or throttling issues.
3. Review Lambda logs for exceptions or timeouts.
4. Check DynamoDB or RDS metrics for throttling, latency, or connection pressure.
5. Use X-Ray traces to identify the slow or failing segment.

**Common exam scenarios**

These patterns show up frequently in DVA-C02-style questions:

- A browser client needs a secure API with user sign-in: use API Gateway with Cognito or a Lambda authorizer, plus CORS.
- A service-to-service API inside AWS needs fine-grained access control: use IAM authorization.
- A high-traffic endpoint with simple CRUD and variable load: use API Gateway, Lambda, and DynamoDB on-demand.
- A relational workload with joins and transactions: use RDS or Aurora, often with RDS Proxy for Lambda.
- Large file uploads: use S3 pre-signed URLs and store metadata separately.
- A need for alternate query patterns on the same data: add a GSI or LSI based on the access pattern.

**Practical deployment examples**

A simple Lambda-backed REST API in CloudFormation:

```yaml
Resources:
  OrdersApi:
    Type: AWS::ApiGateway::RestApi
    Properties:
      Name: OrdersApi

  OrdersResource:
    Type: AWS::ApiGateway::Resource
    Properties:
      RestApiId: !Ref OrdersApi
      ParentId: !GetAtt OrdersApi.RootResourceId
      PathPart: orders

  GetOrdersMethod:
    Type: AWS::ApiGateway::Method
    Properties:
      RestApiId: !Ref OrdersApi
      ResourceId: !Ref OrdersResource
      HttpMethod: GET
      AuthorizationType: AWS_IAM
      Integration:
        Type: AWS_PROXY
        IntegrationHttpMethod: POST
        Uri: !Sub arn:aws:apigateway:${AWS::Region}:lambda:path/2015-03-31/functions/${OrdersFunction.Arn}/invocations
```

A basic CORS-enabled HTTP API example:

```yaml
Resources:
  HttpApi:
    Type: AWS::ApiGatewayV2::Api
    Properties:
      Name: PublicApi
      ProtocolType: HTTP
      CorsConfiguration:
        AllowOrigins:
          - https://app.example.com
        AllowMethods:
          - GET
          - POST
          - OPTIONS
        AllowHeaders:
          - Content-Type
          - Authorization
```

**Key takeaways**

API Gateway gives you the endpoint layer, Lambda gives you compute, and DynamoDB, RDS/Aurora, or S3 provide persistence depending on the data model. For the exam, focus on choosing the right API type, securing the endpoint correctly, and matching the persistence layer to the access pattern.

- Use **REST API** features when you need models, mapping, usage plans, or API keys.
- Use **HTTP API** when you want a simpler, cheaper API with fewer advanced features.
- Use **Lambda proxy** for straightforward serverless backends.
- Use **DynamoDB** for scalable key-value and document access patterns.
- Use **RDS/Aurora** for relational data and SQL requirements.
- Use **S3** for objects, uploads, and downloads.
- Use **logs and X-Ray** to trace failures across the request path.
