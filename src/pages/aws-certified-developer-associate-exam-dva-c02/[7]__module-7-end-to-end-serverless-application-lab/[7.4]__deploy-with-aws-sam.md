## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**AWS SAM deployment overview**

AWS Serverless Application Model (<abbr title="Serverless Application Model">SAM</abbr>) is an open-source framework for building and deploying serverless applications on AWS. It extends AWS CloudFormation with a simplified syntax for defining Lambda functions, API Gateway, DynamoDB, Step Functions, and related resources.

Use SAM when you want a repeatable workflow for:

- Defining infrastructure and application code in one template.
- Building and packaging Lambda artifacts consistently.
- Deploying with CloudFormation-managed stacks.
- Testing locally before pushing changes to AWS.

A SAM application typically follows this lifecycle:

1. Write a `template.yaml` with `AWS::Serverless::*` resources.
2. Run `sam build` to prepare artifacts.
3. Test locally with `sam local` commands when useful.
4. Deploy with `sam deploy` or package first with `sam package`.
5. Clean up with `sam delete` when the stack is no longer needed.

**Core SAM template concepts**

A SAM template is still a CloudFormation template, but it uses the `Transform: AWS::Serverless-2016-10-31` transform to expand shorthand resources into standard CloudFormation resources at deploy time.

Common resource types include:

- `AWS::Serverless::Function` for Lambda functions.
- `AWS::Serverless::Api` for API Gateway REST APIs.
- `AWS::Serverless::HttpApi` for API Gateway HTTP APIs.
- `AWS::Serverless::SimpleTable` for lightweight DynamoDB tables.
- `AWS::Serverless::StateMachine` for Step Functions workflows.

A minimal function example:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Transform: AWS::Serverless-2016-10-31
Description: Simple SAM function

Globals:
  Function:
    Timeout: 10
    MemorySize: 256

Resources:
  HelloFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: src/
      Handler: app.lambda_handler
      Runtime: python3.12
      Events:
        ApiEvent:
          Type: Api
          Properties:
            Path: /hello
            Method: get
```

SAM expands this into underlying CloudFormation resources such as Lambda, IAM roles, log groups, and API Gateway integrations.

**Build and deploy workflow**

`sam build` prepares your application for deployment by resolving dependencies and staging artifacts into `.aws-sam/build`. This step is important because the build output is what SAM deploys, not your raw source tree.

Typical commands:

```bash
sam build
sam deploy
```

For many projects, you build locally and then deploy the generated artifacts. This separation helps you catch packaging issues before CloudFormation creates or updates the stack.

Important deployment variants:

- **`sam deploy`**: Deploys the built application directly from the SAM template and build artifacts.
- **`sam package`**: Uploads local artifacts to Amazon S3 and produces a packaged template with remote artifact locations.
- **`sam deploy` with packaged template**: Useful in CI/CD pipelines where packaging and deployment are separate stages.

Example packaging flow:

```bash
sam build
sam package \
  --s3-bucket my-artifact-bucket \
  --output-template-file packaged.yaml
sam deploy \
  --template-file packaged.yaml \
  --stack-name my-sam-app
```

Use `sam package` when you need a deployable template artifact for later approval, promotion, or cross-account deployment. Use direct `sam deploy` when you want a simpler workflow and do not need a separate packaged template file.

**Guided deployment and `samconfig.toml`**

`sam deploy --guided` is the most common first-time deployment path. It prompts you for stack settings and then writes a `samconfig.toml` file so future deployments can run non-interactively.

Typical guided prompts include:

- Stack name.
- AWS Region.
- Confirm changes before deploy.
- Allow SAM CLI to create roles.
- S3 bucket selection or creation.
- Parameter overrides.
- Save arguments to configuration file.

Example:

```bash
sam deploy --guided
```

The generated `samconfig.toml` stores deployment defaults for later runs:

```toml
version = 0.1

[default.deploy.parameters]
stack_name = "my-sam-app"
region = "us-east-1"
confirm_changeset = true
capabilities = "CAPABILITY_IAM"
resolve_s3 = true
parameter_overrides = "Stage=dev"
```

Parameter precedence matters:

- **CLI flags** override values in `samconfig.toml`.
- **`samconfig.toml`** overrides defaults inferred by SAM.
- **Environment-specific config sections** can be selected with `--config-env`.
- **AWS credentials/profile settings** still come from your shell environment or `--profile`.

Example with a non-default config environment:

```bash
sam deploy --config-env prod
```

This is useful when you maintain separate settings for `dev`, `test`, and `prod` without duplicating templates.

**S3 artifact handling**

SAM uploads deployment artifacts to Amazon S3. You must provide a bucket or let SAM manage one for you.

Key options:

- **`--s3-bucket`**: Use an existing bucket that you manage.
- **`--resolve-s3`**: Let SAM create or reuse a managed deployment bucket for the current account and region.

Practical differences:

- Use `--s3-bucket` when your organization requires a preapproved artifact bucket, encryption policy, or bucket lifecycle controls.
- Use `--resolve-s3` for quick starts and first deployments when you do not want to manually provision the bucket.
- `--resolve-s3` is convenient, but the bucket it uses is still an S3 artifact store; you should treat it as deployment infrastructure, not application storage.

Example:

```bash
sam deploy \
  --stack-name my-sam-app \
  --resolve-s3 \
  --capabilities CAPABILITY_IAM
```

If you already have a bucket:

```bash
sam deploy \
  --stack-name my-sam-app \
  --s3-bucket my-company-sam-artifacts \
  --capabilities CAPABILITY_IAM
```

**CloudFormation change sets and stack updates**

SAM deploys through AWS CloudFormation. When you run `sam deploy`, SAM prepares the template and artifacts, then creates a CloudFormation change set if you request confirmation or if the workflow requires it. CloudFormation is the service that executes the stack update.

This matters because:

- You can inspect proposed resource changes before execution.
- SAM-generated resources appear in the change set as expanded CloudFormation resources.
- Rollback behavior is governed by CloudFormation, not SAM alone.

Useful flags:

- `--confirm-changeset` to review changes before execution.
- `--no-confirm-changeset` for unattended deployments.
- `--disable-rollback` only when you intentionally want to preserve failed stack state for debugging.

Example:

```bash
sam deploy \
  --stack-name my-sam-app \
  --resolve-s3 \
  --capabilities CAPABILITY_IAM \
  --confirm-changeset
```

When a deployment fails, CloudFormation may roll back the stack to the previous stable state. If nested or transformed resources are involved, the rollback still follows CloudFormation semantics, so inspect the stack events in the CloudFormation console or with the CLI.

**Capabilities and permissions**

Some deployments require CloudFormation capabilities to acknowledge IAM or macro-related changes.

Common capabilities:

- **`CAPABILITY_IAM`**: Required when the template creates or updates IAM resources.
- **`CAPABILITY_NAMED_IAM`**: Required when IAM resources have explicit names.
- **`CAPABILITY_AUTO_EXPAND`**: Relevant when a template uses macros or transforms that CloudFormation must expand during change set creation.

Example:

```bash
sam deploy \
  --stack-name my-sam-app \
  --resolve-s3 \
  --capabilities CAPABILITY_IAM
```

Do not assume every SAM deployment needs all capabilities. Use the minimum set that matches the resources in your template.

**Parameter overrides and environment-specific deployments**

`--parameter-overrides` lets you inject environment-specific values without editing the template. This is common for stage names, feature flags, bucket names, and table names.

Example:

```bash
sam deploy \
  --stack-name my-sam-app-dev \
  --resolve-s3 \
  --capabilities CAPABILITY_IAM \
  --parameter-overrides Stage=dev LogLevel=debug
```

Use parameter overrides when:

- You deploy the same template to multiple environments.
- You want to keep secrets and environment-specific values out of source-controlled template defaults.
- You need to vary resource names or application behavior by stage.

A corresponding template parameter example:

```yaml
Parameters:
  Stage:
    Type: String
    Default: dev
  LogLevel:
    Type: String
    Default: info
```

**Local testing with `sam local`**

The `sam local` commands help you test Lambda handlers and API integrations before deployment. They run your function in a local container that approximates the Lambda runtime.

Common commands:

- `sam local invoke` to run a single function with an event payload.
- `sam local start-api` to emulate API Gateway locally.
- `sam local start-lambda` to expose a local Lambda endpoint for integration testing.

Example invocation:

```bash
sam build
sam local invoke HelloFunction -e events/hello.json
```

Example local API server:

```bash
sam local start-api
```

Use local testing to catch handler errors, dependency issues, and event-shape mismatches early. It does not fully replace AWS testing because IAM, networking, managed service integrations, and some runtime differences still exist in the cloud.

**Container image-based Lambda deployments**

SAM also supports Lambda functions packaged as container images. This is useful when your function needs custom OS libraries, larger dependencies, or a custom runtime environment.

Template example:

```yaml
Resources:
  ImageFunction:
    Type: AWS::Serverless::Function
    Properties:
      PackageType: Image
      ImageUri: 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-function:latest
```

Build and deploy flow:

```bash
sam build
sam deploy \
  --stack-name my-image-app \
  --resolve-s3 \
  --capabilities CAPABILITY_IAM
```

For image-based functions, the image is stored in Amazon Elastic Container Registry (<abbr title="Elastic Container Registry">ECR</abbr>). In practice, you build and push the image as part of your workflow, then deploy the stack so Lambda points to the image URI. In many pipelines, `sam build` prepares metadata and `sam deploy` updates the stack, while the image push step happens separately or through SAM-managed image packaging depending on the workflow.

**`sam sync` for rapid iteration**

`sam sync` helps you push certain local changes faster during development. It is best understood as an iterative development tool, not a full replacement for deployment.

Use it when:

- You want quick feedback during active development.
- You are changing function code frequently.
- You are using `--watch` to detect changes automatically.

Example:

```bash
sam sync --watch
```

Important limitations:

- Code changes are the primary target for sync workflows.
- Infrastructure or template changes often still require a full `sam deploy`.
- Behavior can vary by resource type and SAM CLI version.

Treat `sam sync` as a productivity tool for development, not as the authoritative production deployment path.

**Validation and inspection**

`sam validate` checks that your template is structurally valid for SAM/CloudFormation usage, but it is not a guarantee that deployment will succeed. It can catch syntax and some template issues early, yet it does not replace deployment testing or CloudFormation event inspection.

Example:

```bash
sam validate
```

Use it as a quick preflight check, especially in CI, but rely on:

- `sam build` for packaging correctness.
- `sam local` for runtime behavior.
- `sam deploy --confirm-changeset` for deployment review.
- CloudFormation events for failure diagnosis.

**Cleanup and teardown**

`sam delete` removes the CloudFormation stack created by SAM and can also clean up related artifacts, but the exact behavior depends on how the stack and buckets were configured.

Example:

```bash
sam delete --stack-name my-sam-app
```

Be aware of these caveats:

- You may be prompted to confirm deletion.
- Artifact buckets and uploaded objects may require explicit cleanup.
- Retained resources or custom deletion policies can leave resources behind.
- If the stack contains resources with `DeletionPolicy: Retain`, those resources are intentionally preserved.

Use `sam delete` carefully in shared accounts or environments with compliance requirements.

**Practical deployment patterns**

A common development-to-production flow looks like this:

1. Write or update the SAM template and application code.
2. Run `sam build`.
3. Test locally with `sam local invoke` or `sam local start-api`.
4. Deploy to a dev stack with `sam deploy --guided`.
5. Reuse `samconfig.toml` for repeat deployments.
6. Promote the same template to higher environments with parameter overrides or separate config environments.

Example CI-friendly deployment:

```bash
sam build
sam deploy \
  --stack-name my-sam-app-prod \
  --config-env prod \
  --no-confirm-changeset \
  --capabilities CAPABILITY_IAM \
  --parameter-overrides Stage=prod
```

**Exam focus**

- SAM is a CloudFormation-based deployment framework for serverless applications.
- `sam build` prepares artifacts; `sam deploy` updates the stack through CloudFormation.
- `sam deploy --guided` creates `samconfig.toml` for repeatable deployments.
- `--resolve-s3` is convenient for first-time or simple deployments; `--s3-bucket` is preferred when you manage the artifact bucket yourself.
- `sam package` is useful when you need a packaged template for a separate deployment stage.
- `sam local invoke` and `sam local start-api` support local testing before deployment.
- `--parameter-overrides` is a standard way to pass environment-specific values.
- `CAPABILITY_IAM` is required only when IAM resources are created or updated.
- `CAPABILITY_AUTO_EXPAND` is relevant when CloudFormation must expand transforms or macros during change set processing.
- `sam sync` accelerates iterative development, but it does not replace full deployments for all infrastructure changes.
- `sam delete` can remove the stack, but retained resources and artifact cleanup may require extra attention.
