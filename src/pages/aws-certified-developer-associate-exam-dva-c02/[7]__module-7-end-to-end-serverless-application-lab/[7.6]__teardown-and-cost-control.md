## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why teardown and cost control matter**

AWS charges often continue after you delete the “main” resource because supporting services, data transfer, logs, snapshots, and shared networking can remain billable. In exam scenarios, you are usually expected to identify the hidden cost drivers, stop ongoing spend quickly, and verify that cleanup is complete.

A good teardown strategy is not just “delete everything.” You need to understand which resources are still attached, which services bill independently, and which account-level controls help you detect drift before the bill arrives.

**Core cost-control mindset**

Think in layers:

- **Resource layer**: EC2 instances, Lambda functions, RDS databases, EBS volumes, S3 buckets, NAT Gateways, and load balancers.
- **Data layer**: snapshots, backups, object versions, log retention, and transferred bytes.
- **Account layer**: budgets, alerts, tags, consolidated billing, and anomaly detection.
- **Organization layer**: payer account visibility, linked account charge aggregation, and cost allocation across accounts.

This layered view is important because deleting an application stack may still leave behind storage, logs, or network paths that continue to generate charges.

**AWS Organizations and consolidated billing**

In multi-account environments, AWS Organizations uses a **management account** (historically called the payer account) to centralize billing for member accounts. Charges from linked accounts roll up into the organization’s bill, which makes it easier to see total spend and apply shared controls.

Key exam points:

- **Consolidated billing** aggregates charges across linked accounts into one bill.
- The **management account** pays the bill, but each linked account still has its own resources and usage.
- Cost visibility can be broken down by account, service, region, and tag when reporting is configured correctly.
- Shared discounts and commitments may apply across accounts, depending on the pricing model.

This matters during teardown because a resource in one account can still affect the organization’s total spend, and a cleanup task may require checking multiple linked accounts rather than only the one where the app was deployed.

**Cost allocation across accounts and tags**

AWS cost reporting becomes much more useful when you can attribute spend to the right team, project, or environment. Two common mechanisms are account-based allocation and tag-based allocation.

- **Account-based allocation**:
  - Each linked account can represent a team, workload, or environment.
  - This is simple and reliable for broad charge separation.
- **Tag-based allocation**:
  - You tag resources with keys like `Environment`, `Project`, or `Owner`.
  - Tags do not automatically appear in billing reports unless you activate them as **cost allocation tags** in the Billing console.
  - After activation, AWS can include those tags in Cost Explorer and cost and usage reports.

Example tagging pattern:

```bash
aws ec2 create-tags \
  --resources i-0123456789abcdef0 vol-0123456789abcdef0 \
  --tags Key=Project,Value=webapp Key=Environment,Value=dev Key=Owner,Value=team-a
```

Important nuance:

- A tag existing on a resource is not enough for billing visibility.
- You must activate the tag as a **cost allocation tag** before it appears in cost reports.
- Activation is not retroactive in the sense of magically fixing all historical reporting expectations; plan tagging early for best results.

**Budgets, alerts, and anomaly detection**

AWS Budgets helps you set thresholds and receive alerts when spend, usage, or commitment metrics exceed expectations. It is one of the first tools to check when you want to catch a runaway deployment or a failed teardown.

Use Budgets to monitor:

- **Actual cost** against a monthly or daily threshold.
- **Forecasted cost** when spend is trending upward.
- **Usage-based budgets** for services billed by quantity.
- **RI coverage** and **RI utilization** if you use Reserved Instances.
- **Savings Plans coverage** and utilization where applicable.

AWS Cost Anomaly Detection adds a proactive layer by identifying unusual spend patterns automatically. It is especially useful when a deployment creates a sudden spike in data transfer, NAT Gateway usage, or log ingestion that might not be obvious from a simple budget threshold.

Practical exam takeaway:

- Use **Budgets** for threshold-based control.
- Use **Cost Anomaly Detection** for unexpected behavior.
- Use **Cost Explorer** to investigate the cause after an alert.

Example budget notification workflow:

```json
{
  "BudgetName": "dev-monthly-budget",
  "BudgetLimit": {
    "Amount": "50",
    "Unit": "USD"
  },
  "TimeUnit": "MONTHLY",
  "BudgetType": "COST",
  "CostFilters": {
    "TagKeyValue": ["Environment$dev"]
  }
}
```

**Cost Explorer and investigation workflow**

Cost Explorer helps you analyze where money is going by service, linked account, region, usage type, and tag. It is the main tool for answering, “What is still costing money after teardown?”

Use it to:

- Identify the top services by spend.
- Compare current spend to previous periods.
- Filter by account, region, or tag.
- Spot unusual usage types such as data transfer or NAT Gateway processing.

A practical investigation flow is:

1. Open **Cost Explorer** and review the highest-cost services.
2. Group by **linked account** if you use Organizations.
3. Filter by **region** to find resources left in an unexpected region.
4. Group by **usage type** to expose hidden charges like data transfer or log ingestion.
5. Check whether tags are activated and applied correctly.

If a stack is “deleted” but spend continues, Cost Explorer often reveals the real issue: a storage artifact, a network gateway, or a logging service still generating usage.

**Trusted Advisor cost checks**

AWS Trusted Advisor includes cost-related checks that help you find idle or underutilized resources. This is especially useful when you need a quick cleanup pass before shutting down a lab or development environment.

Common cost-related findings include:

- Idle or underutilized EC2 instances.
- Unattached EBS volumes.
- Underutilized load balancers.
- Reserved Instance optimization opportunities.
- Potential savings from rightsizing or commitment changes.

Trusted Advisor is not a replacement for teardown verification, but it is a strong discovery tool when you want to find resources that are easy to miss manually.

**Common teardown cost traps**

Some resources continue billing even after the “main” application is gone. These are frequent exam traps because they are easy to overlook.

- **NAT Gateway**:
  - Charges accrue for hourly usage and data processing.
  - Deleting instances does not stop NAT Gateway charges if the gateway remains.
- **EBS volumes and snapshots**:
  - Terminating an EC2 instance does not always delete attached EBS volumes.
  - Snapshots remain billable in S3-backed storage until deleted.
- **CloudWatch Logs**:
  - Log ingestion and log storage both cost money.
  - Applications that keep writing logs can continue generating spend even after compute is removed.
- **Data transfer**:
  - Deleting resources does not erase charges already incurred for outbound data transfer.
  - Cross-AZ, cross-region, and internet egress charges can surprise you.
- **Elastic Load Balancers**:
  - Load balancers bill while provisioned, even if traffic is low.
- **RDS backups and snapshots**:
  - Automated backups and manual snapshots can persist after database deletion, depending on settings.
- **S3 buckets**:
  - Storage, requests, lifecycle transitions, and versioned objects can continue to cost money.
- **Elastic IP addresses**:
  - Unassociated Elastic IPs can incur charges.
- **CloudWatch metrics and alarms**:
  - Custom metrics and alarms can contribute to ongoing cost.

> Deleting an application does not necessarily delete its dependencies. In teardown questions, always ask what was created automatically, what was shared, and what persists independently.

**Savings Plans vs Reserved Instances**

Both **Savings Plans** and **Reserved Instances (RIs)** reduce compute cost, but they behave differently and are often tested as a comparison.

- **Savings Plans**:
  - Generally more flexible than RIs.
  - Commit to a spend amount over time rather than a specific instance family in the same rigid way.
  - Better when your compute usage changes frequently.
- **Reserved Instances**:
  - More specific to instance attributes and can be less flexible.
  - Useful when you know your steady-state usage.
  - Coverage and utilization are separate metrics you should monitor.

Exam nuance:

- **Coverage** tells you how much of your usage is discounted.
- **Utilization** tells you how much of your commitment you actually used.
- A high commitment with low utilization can still waste money.
- Budgets can be used to monitor commitment-related metrics, not just raw spend.

**Practical teardown workflow for a sample stack**

A full cleanup should follow the dependency chain so you do not leave behind billable artifacts.

Example stack:
- Application Load Balancer
- Auto Scaling group with EC2 instances
- EBS volumes
- RDS database
- CloudWatch log group
- NAT Gateway
- S3 deployment bucket
- Route 53 record
- IAM roles and policies

Recommended teardown order:

1. Disable traffic first.
   - Remove Route 53 records or shift traffic away.
   - Deregister targets from the load balancer.
2. Stop compute scaling.
   - Delete the Auto Scaling group.
   - Terminate EC2 instances.
3. Remove persistent storage.
   - Delete unused EBS volumes.
   - Delete manual snapshots if they are no longer needed.
4. Remove managed data services.
   - Delete the RDS instance and confirm backup retention settings.
   - Empty and delete S3 buckets, including versioned objects if enabled.
5. Remove networking cost sources.
   - Delete the NAT Gateway.
   - Release unused Elastic IPs.
   - Delete load balancers and target groups.
6. Remove observability artifacts.
   - Delete CloudWatch log groups if retention is no longer needed.
   - Remove custom metrics or alarms if appropriate.
7. Clean up IAM and supporting resources.
   - Delete unused roles, policies, and instance profiles.
   - Remove leftover security groups, launch templates, and key pairs if they are no longer needed.

Example CLI cleanup commands:

```bash
aws elbv2 delete-load-balancer --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/app/demo/50dc6c495c0c9188

aws autoscaling delete-auto-scaling-group \
  --auto-scaling-group-name demo-asg \
  --force-delete

aws ec2 delete-nat-gateway --nat-gateway-id nat-0123456789abcdef0

aws ec2 release-address --allocation-id eipalloc-0123456789abcdef0

aws logs delete-log-group --log-group-name /aws/lambda/demo-function

aws s3 rm s3://demo-deploy-bucket --recursive
aws s3api delete-bucket --bucket demo-deploy-bucket
```

For RDS, you often need to decide whether to retain final snapshots:

```bash
aws rds delete-db-instance \
  --db-instance-identifier demo-db \
  --skip-final-snapshot
```

Use `--skip-final-snapshot` only when you are sure the data is disposable. In real environments, a final snapshot is often safer, but it continues to cost money until deleted.

**End-to-end alert-to-resolution example**

Suppose a budget alert fires because monthly spend suddenly jumps after a demo deployment.

1. **Alert arrives**
   - Budgets sends an email or SNS notification.
   - Cost Anomaly Detection flags unusual spend in `us-east-1`.
2. **Investigate**
   - Open Cost Explorer and group by service.
   - You see high charges from `NAT Gateway`, `CloudWatch Logs`, and `DataTransfer-Out-Bytes`.
3. **Trace the source**
   - Check the linked account and environment tag.
   - Confirm the demo stack is still running in a development account.
4. **Teardown**
   - Delete the load balancer, Auto Scaling group, and EC2 instances.
   - Remove the NAT Gateway and release the Elastic IP.
   - Empty the S3 bucket and delete log groups.
   - Delete snapshots and unused EBS volumes.
5. **Verify**
   - Review the Billing console and Cost Explorer over the next billing period.
   - Confirm no residual spend appears for the deleted resources.
   - Check that the budget trend returns to baseline.

This workflow is a strong exam pattern because it combines detection, investigation, cleanup, and verification.

**How to verify teardown completion**

Do not assume deletion is complete just because the console shows no running instances. Verification should happen in billing and usage tools.

Check:

- **Billing console** for current charges and linked account totals.
- **Cost Explorer** for service-level spend after cleanup.
- **Budgets** for whether alerts stop firing.
- **Trusted Advisor** for idle or unattached resources.
- **CloudWatch Logs** and S3 for lingering storage growth.

A useful habit is to review the next billing period after teardown. Some charges are delayed or appear after usage is finalized, so a clean console today does not always mean a zero-cost bill tomorrow.

**CLI and console habits that reduce surprises**

Use automation where possible so teardown is repeatable and less error-prone.

- Tag every resource at creation time.
- Use infrastructure as code, such as CloudFormation or Terraform, so you can destroy stacks consistently.
- Prefer short-lived environments for labs and demos.
- Set budgets before deployment, not after.
- Review region usage because forgotten resources in another region are a common cause of residual spend.

Example CloudFormation cleanup:

```bash
aws cloudformation delete-stack --stack-name demo-stack
aws cloudformation wait stack-delete-complete --stack-name demo-stack
```

Even after stack deletion, verify that the template did not create resources outside the stack, such as manually created snapshots, S3 objects, or retained log groups.

**Exam-focused distinctions**

- **Deleting a stack** is not the same as deleting every billable artifact.
- **Stopping an EC2 instance** does not delete its EBS volume unless configured to do so.
- **Removing a database** may leave snapshots and backups behind.
- **Deleting a resource** does not reverse charges already incurred.
- **Budget alerts** detect threshold breaches; **anomaly detection** finds unusual patterns.
- **Savings Plans** are generally more flexible than **Reserved Instances**.
- **Cost allocation tags** must be activated before they appear in billing reports.

**Quick teardown checklist**

- Confirm traffic is drained.
- Delete compute and scaling resources.
- Remove load balancers, NAT Gateways, and Elastic IPs.
- Delete unused volumes, snapshots, and backups.
- Empty and delete S3 buckets.
- Remove CloudWatch logs, alarms, and custom metrics if no longer needed.
- Check Budgets, Cost Explorer, and Trusted Advisor.
- Verify charges in the Billing console over the next billing cycle.
- Confirm tags are activated for future cost attribution.

**Mental model for the exam**

When you see a teardown or cost-control question, ask:

- What is still running?
- What is still stored?
- What is still logging?
- What is still transferring data?
- What is shared across accounts?
- What is visible in billing reports?
- What needs to be activated before it shows up in cost analysis?

If you can answer those questions, you can usually identify the hidden charge and the correct control to stop it.
