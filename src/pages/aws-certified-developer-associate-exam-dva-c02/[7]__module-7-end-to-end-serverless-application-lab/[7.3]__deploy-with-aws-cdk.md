## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**AWS CDK deployment model**

AWS Cloud Development Kit (CDK) lets you define infrastructure in code, then synthesize that code into an AWS CloudFormation template. The deployment flow is usually:

1. Write constructs in your preferred language.
2. Run `cdk synth` to generate CloudFormation.
3. Review changes with `cdk diff`.
4. Deploy with `cdk deploy`.

CDK is not a separate provisioning engine. It is a higher-level framework that produces CloudFormation, and CloudFormation performs the actual resource creation and updates.

- **Synthesis** turns your app into a CloudFormation template and supporting assets.
- **Deployment** sends the synthesized template and assets to AWS through CloudFormation.
- **Context lookups** and environment resolution happen during synthesis, not at runtime inside AWS.

A typical app structure looks like this:

```bash
my-cdk-app/
├── bin/
│   └── app.ts
├── lib/
│   └── stack.ts
├── cdk.json
├── cdk.context.json
└── package.json
```

**How CDK determines account and Region**

CDK CLI resolves the target environment from your AWS credentials and configuration. In practice, it uses the active profile, environment variables, and the AWS SDK credential chain. Common inputs include:

- `AWS_PROFILE`
- `AWS_DEFAULT_REGION`
- `AWS_REGION`
- `~/.aws/config`
- `~/.aws/credentials`

If you do not explicitly set an environment on the stack, CDK may use the current credentials and Region during synthesis and deployment. For exam purposes, remember that `CDK_DEFAULT_ACCOUNT` and `CDK_DEFAULT_REGION` are values the CLI makes available to your app; they are not a substitute for authenticating to AWS.

```bash
export AWS_PROFILE=dev
export AWS_REGION=us-east-1
cdk synth
cdk deploy
```

**Synthesis versus deployment**

It helps to separate what CDK does locally from what AWS does remotely.

- **Local synthesis**
  - Resolves context values.
  - Bundles assets when needed.
  - Produces CloudFormation templates.
  - Writes or updates `cdk.context.json` for cached lookups.

- **CloudFormation deployment**
  - Creates or updates stacks.
  - Applies resource changes.
  - Manages rollback behavior when updates fail.
  - Exposes stack outputs after completion.

This distinction matters in troubleshooting. If a lookup value is stale, the issue often appears during synthesis or diff, not during the CloudFormation update itself.

**Pre-deployment review with `cdk diff`**

Use `cdk diff` to compare the current synthesized template with the deployed stack. This is one of the most important CDK commands for safe deployments and exam scenarios.

```bash
cdk diff
cdk diff MyStack
cdk diff --context env=dev
```

`cdk diff` helps you answer questions such as:

- What resources will be added, modified, or removed?
- Did a property change force replacement?
- Did a context lookup change the generated template?
- Are there security-sensitive changes that require approval?

A diff is especially useful before deploying to production because it surfaces destructive changes early.

> `cdk diff` compares templates; it does not deploy anything.

**Deploying stacks with `cdk deploy`**

`cdk deploy` synthesizes the app if needed, uploads required assets, and asks CloudFormation to apply the stack update. It may prompt for confirmation depending on the change set and approval settings.

```bash
cdk deploy
cdk deploy MyStack
cdk deploy StackA StackB
```

Common deployment behavior:

- CDK synthesizes the app before deployment if the output is not current.
- Assets such as Lambda bundles or Docker images are published as needed.
- CloudFormation performs the stack update.
- The CLI can wait for completion and stream progress.

For automation, you usually disable prompts and make the command non-interactive.

```bash
cdk deploy --require-approval never --ci
```

**Approval settings and security prompts**

The `--require-approval` flag controls whether CDK prompts you before deploying security-sensitive changes. The documented values are:

- `never`
- `any-change`
- `broadening`

Use these carefully:

- `never` disables approval prompts.
- `any-change` prompts for any stack change.
- `broadening` prompts when changes broaden permissions or network exposure, such as IAM policy expansion or security group ingress changes.

This flag is important in exam questions because it is about security review, not generic stack metadata changes.

```bash
cdk deploy --require-approval broadening
```

**Outputs and downstream consumption**

CloudFormation stack outputs are values exported by the stack after deployment. In CDK, you define them with `CfnOutput`. Outputs are useful for:

- URLs
- ARNs
- bucket names
- API endpoints
- resource identifiers for later automation

```ts
import * as cdk from 'aws-cdk-lib';
import { Construct } from 'constructs';

export class AppStack extends cdk.Stack {
  constructor(scope: Construct, id: string) {
    super(scope, id);

    new cdk.CfnOutput(this, 'ApiUrl', {
      value: 'https://example.execute-api.us-east-1.amazonaws.com/prod/',
    });
  }
}
```

After deployment, you can view outputs in the CloudFormation console, with the AWS CLI, or by capturing them during deployment.

```bash
aws cloudformation describe-stacks \
  --stack-name AppStack \
  --query "Stacks[0].Outputs"
```

**Capturing outputs with `--outputs-file`**

Use `cdk deploy --outputs-file` when you want deployment outputs written to a local file for scripts or CI/CD pipelines.

```bash
cdk deploy AppStack --outputs-file outputs.json
cat outputs.json
```

This is useful when a downstream step needs values such as an API endpoint or bucket name without manually querying CloudFormation.

A typical pattern is:

1. Deploy the stack.
2. Read the outputs file.
3. Pass the values to a test, integration job, or another deployment step.

**Multi-stack deployments and dependencies**

When your app contains multiple stacks, CDK can infer dependencies from references between stacks. If Stack B consumes an output or resource from Stack A, CDK usually deploys Stack A first.

```ts
const network = new NetworkStack(app, 'NetworkStack');
const appStack = new AppStack(app, 'AppStack', {
  vpc: network.vpc,
});
```

Deployment behavior to remember:

- CDK deploys dependent stacks in the correct order when dependencies are declared through references.
- If you deploy multiple stacks explicitly, CDK may include dependencies unless you use `--exclusively`.
- `--exclusively` limits deployment to the named stacks only.

```bash
cdk deploy AppStack --exclusively
```

Use `--exclusively` when you want to avoid deploying upstream dependencies in the same command, but be careful: if the target stack truly depends on another stack, excluding dependencies can break the deployment workflow.

**Bootstrapping and environment preparation**

Before CDK can deploy assets to an account and Region, the environment often needs to be bootstrapped. Bootstrapping creates the supporting resources CDK uses for file assets, Docker assets, and deployment roles.

Typical bootstrap command:

```bash
cdk bootstrap aws://123456789012/us-east-1
```

Bootstrapping matters because CDK may need:

- an S3 bucket for file assets
- an ECR repository for container assets
- IAM roles for deployment and lookup
- metadata that identifies the bootstrap stack version

If deployment fails with bootstrap-related errors, check whether:

- the target account and Region were bootstrapped
- the bootstrap stack version is old
- the bootstrap qualifier does not match the app’s expected qualifier
- the deployment role permissions are insufficient

A common exam scenario is a deployment that works in one account but fails in another because the target environment was never bootstrapped or was bootstrapped with a different configuration.

> If the app uses newer CDK features or asset publishing behavior, you may need to re-bootstrap the environment with a compatible bootstrap stack version.

**Context lookups and stale cached values**

CDK uses context to cache environment-specific values that are resolved during synthesis, such as availability zones, VPC lookups, or hosted zone lookups. These values are stored in `cdk.context.json`.

Example of a lookup-driven construct:

```ts
import * as ec2 from 'aws-cdk-lib/aws-ec2';

const vpc = ec2.Vpc.fromLookup(this, 'ImportedVpc', {
  vpcId: 'vpc-0123456789abcdef0',
});
```

Workflow to manage context:

1. Run `cdk synth` or `cdk deploy` to trigger the lookup.
2. Inspect `cdk.context.json` if the result seems stale.
3. Clear or refresh context when the environment changes.

Useful commands:

```bash
cat cdk.context.json
cdk context --clear
cdk synth -v
```

Important note:

- `cdk context` is primarily for managing context values, not for displaying them as a general query command.
- If a lookup result is stale, deleting the relevant entry from `cdk.context.json` or clearing context forces CDK to resolve it again.

Stale context can cause confusing diffs because the synthesized template may still reflect an old VPC, subnet, or availability zone selection.

**Deployment assets and publishing**

CDK can package and publish assets automatically during deployment. This includes:

- Lambda function bundles
- Docker image assets
- static files for S3
- other file-based artifacts referenced by the template

You usually do not publish these manually unless you are troubleshooting or using a specialized pipeline. CDK handles asset staging and publishing as part of the deployment workflow.

Example Lambda asset deployment:

```ts
import * as lambda from 'aws-cdk-lib/aws-lambda';

new lambda.Function(this, 'Fn', {
  runtime: lambda.Runtime.NODEJS_20_X,
  handler: 'index.handler',
  code: lambda.Code.fromAsset('lambda'),
});
```

During `cdk deploy`, CDK stages the asset, uploads it to the bootstrap bucket or ECR repository, and then CloudFormation references the published asset.

**Rollback behavior and failure handling**

CloudFormation controls rollback when a stack update fails. CDK exposes flags that influence how the deployment behaves, but the actual rollback is still a CloudFormation concern.

Useful flags:

- `--rollback` to enable rollback behavior where supported
- `--no-rollback` to keep failed resources for debugging in some workflows
- `--force` to proceed with deployment actions even when CDK would otherwise stop for certain checks

Example:

```bash
cdk deploy MyStack --no-rollback
```

Use rollback-related flags carefully. Keeping failed resources can help with debugging, but it can also leave partially created infrastructure behind and increase cleanup work.

**Hotswap for rapid iteration**

Hotswap is a development-time optimization that updates certain resource changes directly without a full CloudFormation deployment. It is useful when you want faster feedback during local development.

```bash
cdk deploy --hotswap
```

Use hotswap only when you understand its limitations:

- It supports only a subset of resource changes.
- It is intended for rapid iteration, not production deployment.
- It may not update all stack resources or CloudFormation metadata.
- It is not a replacement for a full deploy when you need an authoritative stack state.

A practical rule is:

- Use hotswap for quick local testing.
- Use normal `cdk deploy` for production, shared environments, and final validation.

**Common deployment workflow**

A safe, exam-friendly workflow is:

1. Confirm the target account and Region.
2. Ensure the environment is bootstrapped.
3. Run `cdk synth` to validate synthesis.
4. Run `cdk diff` to review the changes.
5. Deploy with the appropriate approval and CI flags.
6. Capture outputs if another step needs them.
7. Verify the deployed stack in CloudFormation or with the AWS CLI.

Example:

```bash
export AWS_PROFILE=dev
export AWS_REGION=us-east-1

cdk synth
cdk diff MyStack
cdk deploy MyStack --require-approval broadening --outputs-file outputs.json
```

**Troubleshooting deployment issues**

Common problems and what they usually mean:

- **Access denied**
  - The current AWS identity lacks permissions for CloudFormation, S3 asset upload, ECR publishing, or IAM role assumption.
- **Bootstrap missing**
  - The target account and Region were not bootstrapped.
- **Stale lookup values**
  - `cdk.context.json` contains cached data that no longer matches the environment.
- **Approval prompt blocks automation**
  - Add `--require-approval never` in CI/CD when appropriate.
- **Dependency ordering issues**
  - A stack depends on another stack that was not deployed or was excluded with `--exclusively`.

A useful debugging command is:

```bash
cdk deploy MyStack -v
```

Verbose output helps you see synthesis steps, asset publishing, and CloudFormation interactions.

**Exam focus points**

- **`cdk diff`** is the pre-deployment review command.
- **`cdk deploy`** uses CloudFormation to apply the synthesized template.
- **Bootstrapping** is required for asset publishing and deployment roles in many environments.
- **Context caching** can make deployments appear inconsistent until you refresh `cdk.context.json`.
- **Outputs** are often consumed by later automation steps and can be captured with `--outputs-file`.
- **`--exclusively`** affects multi-stack deployment behavior.
- **`--require-approval`** protects against security-sensitive changes.
- **Hotswap** is for fast development, not full production fidelity.

**Practical command reference**

```bash
# Synthesize the app
cdk synth

# Compare local changes to deployed state
cdk diff

# Deploy a single stack
cdk deploy MyStack

# Deploy without interactive approval
cdk deploy MyStack --require-approval never --ci

# Capture outputs for downstream steps
cdk deploy MyStack --outputs-file outputs.json

# Deploy only the named stack, not dependencies
cdk deploy MyStack --exclusively

# Clear cached context values
cdk context --clear

# Show verbose synthesis/deploy details
cdk deploy -v
```
