## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Application skeleton overview**

An application skeleton is the minimal, production-minded foundation you build before adding business features. In AWS, this usually means defining the runtime, deployment path, configuration strategy, IAM permissions, logging, and data access patterns early so the application can scale without major redesign.

For exam purposes, focus on how the skeleton supports:
- **Separation of concerns** between compute, configuration, and persistence
- **Repeatable deployment** through infrastructure as code
- **Least privilege** access for application components
- **Observability** through logs, metrics, and tracing
- **Data model fit** for the chosen access patterns

A good skeleton reduces later rework. If you choose the wrong data model or overcouple the app to a single database shape, you often pay for it in performance, cost, and operational complexity.

**Typical AWS application skeleton components**

A practical AWS application skeleton often includes:
- A compute layer such as AWS Lambda, Amazon ECS, or Amazon EC2
- An API layer such as Amazon API Gateway or an Application Load Balancer
- A persistence layer such as Amazon DynamoDB, Amazon RDS, or Amazon S3
- A configuration and secrets layer such as AWS Systems Manager Parameter Store or AWS Secrets Manager
- An identity and permissions model using AWS Identity and Access Management (IAM)
- Logging and monitoring with Amazon CloudWatch and AWS X-Ray

You usually define these pieces before implementing business logic so that the application can be deployed, tested, and secured from the start.

**Choosing the right data model**

The data model should match how the application reads and writes data, not just how the data looks conceptually. In AWS, this is especially important because managed services reward access-pattern-driven design.

Use these questions to guide the model:
- What are the primary read patterns?
- What are the write frequency and consistency requirements?
- Do you need relational joins, transactions, or flexible schema?
- Is the workload predictable or highly variable?
- Do you need global distribution or low-latency access?

For example:
- Use **Amazon DynamoDB** when you need single-digit millisecond performance, predictable key-based access, and flexible schema design.
- Use **Amazon RDS** when you need relational structure, SQL queries, joins, and familiar transactional behavior.
- Use **Amazon S3** when the data is object-based, such as documents, images, backups, or static assets.

**Access patterns first, schema second**

With DynamoDB, you design around access patterns. That means you identify the queries the application must support, then shape the table keys and indexes to serve them efficiently.

Common design principles:
- Model around the most frequent and latency-sensitive queries
- Use a partition key that distributes traffic well
- Use a sort key to support grouping and range queries
- Add secondary indexes only when necessary
- Avoid scanning large tables in production paths
- Denormalize when it improves read efficiency and reduces round trips

Example access patterns for an order system:
- Get all orders for a customer
- Get a specific order by order ID
- List orders by status and date
- Update order status quickly

A DynamoDB design might use:
- `PK = CUSTOMER#<customerId>`
- `SK = ORDER#<orderDate>#<orderId>`

This supports efficient retrieval of all orders for a customer and ordered listing by date.

**DynamoDB skeleton example**

A simple table design for an application skeleton might look like this:

```json
{
  "TableName": "Orders",
  "AttributeDefinitions": [
    { "AttributeName": "PK", "AttributeType": "S" },
    { "AttributeName": "SK", "AttributeType": "S" }
  ],
  "KeySchema": [
    { "AttributeName": "PK", "KeyType": "HASH" },
    { "AttributeName": "SK", "KeyType": "RANGE" }
  ],
  "BillingMode": "PAY_PER_REQUEST"
}
```

This design is useful when:
- You want to avoid capacity planning early in development
- Traffic is variable or hard to predict
- You are building a new system and want fast iteration

A sample item might be:

```json
{
  "PK": "CUSTOMER#123",
  "SK": "ORDER#2026-04-26#A1001",
  "orderId": "A1001",
  "status": "PENDING",
  "total": 89.50,
  "createdAt": "2026-04-26T10:15:00Z"
}
```

This item shape supports efficient retrieval by customer and time.

**Relational model considerations**

Use a relational model when the application needs:
- Strong normalization
- Complex joins
- Multi-row transactions
- Ad hoc SQL reporting
- Referential integrity

In Amazon RDS, you typically define tables with primary keys, foreign keys, indexes, and transaction boundaries. This is a strong fit for systems like:
- Order management with many related entities
- Financial workflows
- Applications that already depend on SQL tooling

Example schema sketch:

```sql
CREATE TABLE customers (
  customer_id BIGINT PRIMARY KEY,
  email VARCHAR(255) NOT NULL UNIQUE,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE orders (
  order_id BIGINT PRIMARY KEY,
  customer_id BIGINT NOT NULL,
  status VARCHAR(32) NOT NULL,
  total DECIMAL(10,2) NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_orders_customer
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);
```

This model is appropriate when you need to query orders by customer and enforce referential integrity at the database layer.

**When to use S3 as part of the data model**

Amazon S3 is not a database, but it is often part of the application data model for large objects and immutable artifacts.

Use S3 for:
- Uploaded files
- Media assets
- Build artifacts
- Logs and archives
- Exported reports

A common pattern is to store metadata in DynamoDB or RDS and the actual object in S3.

Example:
- DynamoDB stores `documentId`, owner, status, and S3 key
- S3 stores the file content at `s3://bucket/documents/<documentId>.pdf`

This separation keeps the database small and efficient while allowing durable object storage.

**Application skeleton with API, compute, and persistence**

A common serverless skeleton looks like this:

```text
Client -> API Gateway -> Lambda -> DynamoDB
                     -> Lambda -> S3
                     -> Lambda -> Secrets Manager
```

Why this pattern is common:
- API Gateway handles request routing and throttling
- Lambda provides event-driven compute without server management
- DynamoDB provides scalable key-value/document storage
- S3 stores large binary or unstructured content
- Secrets Manager stores credentials securely

A container-based skeleton may look like this:

```text
Client -> Application Load Balancer -> ECS Service -> RDS
                                         -> ECS Service -> ElastiCache
```

This is useful when:
- You need long-running services
- You want more control over runtime dependencies
- You need relational persistence or in-memory caching

**Configuration and environment management**

Your skeleton should separate code from configuration. Do not hardcode environment-specific values such as table names, bucket names, or secret values.

Use:
- Environment variables for non-sensitive runtime settings
- Parameter Store for configuration values
- Secrets Manager for credentials and API keys

Example Lambda environment variables:

```bash
TABLE_NAME=Orders
BUCKET_NAME=app-documents-prod
LOG_LEVEL=INFO
```

Example AWS CLI retrieval from Parameter Store:

```bash
aws ssm get-parameter \
  --name "/myapp/prod/table-name" \
  --with-decryption
```

Example Secrets Manager retrieval:

```bash
aws secretsmanager get-secret-value \
  --secret-id prod/db-credentials
```

Keep sensitive values out of source control and avoid embedding them in deployment templates unless they are securely referenced.

**IAM for application access**

The application skeleton should include IAM permissions from the beginning. Each component should have only the permissions it needs.

Common examples:
- Lambda execution role can read one DynamoDB table and one secret
- ECS task role can write logs and access a specific S3 bucket
- CI/CD role can deploy infrastructure but not read production data

Example IAM policy for a Lambda function reading a single table:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:PutItem",
        "dynamodb:UpdateItem",
        "dynamodb:Query"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders"
    }
  ]
}
```

This is better than granting broad access to all tables or all AWS resources.

**Infrastructure as code for repeatability**

Use infrastructure as code (IaC) to define the skeleton consistently across environments. Common tools include AWS CloudFormation, AWS CDK, and Terraform.

Benefits:
- Repeatable deployments
- Version-controlled infrastructure
- Easier rollback and review
- Consistent dev, test, and prod environments

Example CloudFormation snippet:

```yaml
Resources:
  OrdersTable:
    Type: AWS::DynamoDB::Table
    Properties:
      TableName: Orders
      BillingMode: PAY_PER_REQUEST
      AttributeDefinitions:
        - AttributeName: PK
          AttributeType: S
        - AttributeName: SK
          AttributeType: S
      KeySchema:
        - AttributeName: PK
          KeyType: HASH
        - AttributeName: SK
          KeyType: RANGE
```

This lets you create the same table in every environment without manual setup.

**Observability in the skeleton**

Add logging and tracing before production traffic arrives. If you wait until later, debugging becomes much harder.

Include:
- Structured logs with request IDs
- Metrics for latency, errors, and throttles
- Distributed tracing for multi-service requests

Example Python Lambda logging pattern:

```python
import json
import logging

logger = logging.getLogger()
logger.setLevel(logging.INFO)

def handler(event, context):
    logger.info(json.dumps({
        "requestId": context.aws_request_id,
        "path": event.get("path"),
        "message": "request received"
    }))
    return {"statusCode": 200, "body": "ok"}
```

This makes it easier to correlate logs with API requests and downstream calls.

**Data modeling tradeoffs**

Different data stores optimize for different access patterns. Choose based on the workload, not preference.

| Service | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| DynamoDB | Key-based access, high scale, low latency | Fast, serverless, flexible schema | Requires access-pattern design, limited joins |
| RDS | Relational data, SQL, transactions | Familiar model, joins, constraints | More operational overhead, scaling can be harder |
| S3 | Objects and large files | Durable, cheap, simple | Not queryable like a database |
| ElastiCache | Hot data, caching | Very low latency | Volatile, not primary storage |

Use caching only when you have a clear read bottleneck or repeated expensive lookups. Do not use cache as a substitute for a poorly designed primary data model.

**Practical DynamoDB design techniques**

To build a strong data model, you often combine multiple entity types in one table. This is a common single-table design approach.

Example entity layout:
- Customer profile item
- Order item
- Order line item
- Status index item

Possible keys:
- `PK = CUSTOMER#123`
- `SK = PROFILE`
- `SK = ORDER#2026-04-26#A1001`
- `SK = ORDER#2026-04-26#A1001#LINE#1`

This can reduce the number of queries needed to assemble a page or workflow.

Use secondary indexes when you need alternate access paths:
- Global Secondary Index (GSI) for querying by a different partition key
- Local Secondary Index (LSI) for alternate sort key access within the same partition key

Be careful with index design:
- Every index adds write cost
- Poorly chosen keys can create hot partitions
- Over-indexing increases complexity and cost

**Example API and data flow**

A simple create-order flow might be:

1. Client sends `POST /orders`
2. API Gateway forwards the request to Lambda
3. Lambda validates input and writes the order to DynamoDB
4. Lambda stores a receipt or attachment in S3 if needed
5. Lambda returns a response with the created order ID

Example request payload:

```json
{
  "customerId": "123",
  "items": [
    { "sku": "ABC-1", "qty": 2 },
    { "sku": "XYZ-9", "qty": 1 }
  ]
}
```

Example response:

```json
{
  "orderId": "A1001",
  "status": "PENDING",
  "createdAt": "2026-04-26T10:15:00Z"
}
```

This flow is simple, testable, and easy to extend.

**Common exam pitfalls**

Watch for these mistakes:
- Designing a database around tables instead of access patterns
- Using scans where queries or indexes are required
- Hardcoding environment values in code
- Granting overly broad IAM permissions
- Storing large binary objects in a database instead of S3
- Choosing RDS when the workload is better suited to DynamoDB, or vice versa
- Ignoring observability until after deployment

> If a question asks for the best data model, start by identifying the access pattern, consistency needs, and scaling profile before choosing the service.

**Quick decision guide**

- Use **DynamoDB** when you need serverless scale, predictable access patterns, and low-latency key lookups.
- Use **RDS** when you need SQL, joins, and relational integrity.
- Use **S3** when the data is an object or file.
- Use **Secrets Manager** for credentials and secrets.
- Use **Parameter Store** for non-secret configuration.
- Use **IAM roles** to give each component only the permissions it needs.

**Minimal deployment skeleton example**

A small but realistic serverless stack might include:
- API Gateway endpoint
- Lambda function
- DynamoDB table
- S3 bucket for uploads
- IAM role for the function
- CloudWatch log group

Example AWS CLI commands to inspect the deployed resources:

```bash
aws lambda get-function --function-name order-service
aws dynamodb describe-table --table-name Orders
aws s3 ls s3://app-documents-prod
aws logs describe-log-groups --log-group-name-prefix /aws/lambda/order-service
```

These commands help you verify that the skeleton exists and is wired correctly before you add more features.

**Key takeaway**

Build the skeleton around deployment, security, observability, and data access from the start. Then choose the data model that best matches how the application reads and writes data. In AWS, that usually means designing for access patterns first and implementation details second.
