## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why this setup matters**

A TypeScript environment for AWS gives you static typing, better refactoring safety, and clearer contracts for Lambda handlers, SDK calls, and event payloads. For exam prep, focus on how the local development toolchain maps to AWS runtime behavior: you write TypeScript, compile to JavaScript, package dependencies correctly, and test in a way that resembles the deployed Lambda environment.

In AWS development, the main goal is not just “make TypeScript work,” but make it work in a way that is predictable in Lambda, SAM, and CI/CD pipelines. That means you need a repeatable project layout, a build step, and a clear understanding of runtime-specific values such as `AWS_REGION` and handler entry points.

**Core project components**

A practical AWS TypeScript project usually includes:

- **Node.js** runtime for local development and Lambda compatibility.
- **TypeScript** compiler for type checking and transpilation.
- **AWS SDK for JavaScript v3** for modular AWS service access.
- **Build tooling** such as `tsc`, `esbuild`, or a framework-specific bundler.
- **Local test tools** for unit tests and Lambda-style invocation.
- **Infrastructure definition** with AWS SAM, CDK, or CloudFormation when you deploy.

A minimal project often looks like this:

```text
project/
├─ src/
│  ├─ handler.ts
│  └─ lib/
├─ test/
├─ dist/
├─ package.json
├─ tsconfig.json
└─ template.yaml
```

**Node.js and TypeScript prerequisites**

You should install a current LTS version of Node.js and use a package manager consistently. For AWS Lambda, Node.js versions must match supported runtimes, so avoid assuming the latest local Node version is always deployable.

Common setup commands:

```bash
node -v
npm -v
npm init -y
npm install --save-dev typescript @types/node
npx tsc --init
```

Useful package manager options:

- `npm` is the default and simplest choice for exam scenarios.
- `pnpm` can speed up installs and reduce disk usage.
- `yarn` is still common in older projects, but many AWS examples now use `npm`.

**TypeScript configuration**

Your `tsconfig.json` controls how TypeScript compiles source files into JavaScript. For AWS Lambda, you usually want a configuration that targets the Node.js runtime you deploy to and emits CommonJS or ES modules consistently with your packaging strategy.

Example `tsconfig.json` for a Node.js Lambda project:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "CommonJS",
    "moduleResolution": "node",
    "outDir": "dist",
    "rootDir": "src",
    "strict": true,
    "esModuleInterop": true,
    "resolveJsonModule": true,
    "sourceMap": true,
    "types": ["node"]
  },
  "include": ["src/**/*.ts", "test/**/*.ts"],
  "exclude": ["node_modules", "dist"]
}
```

Key settings to understand:

- **`target`** controls the JavaScript syntax emitted by the compiler.
- **`module`** must match how your runtime loads code and how your bundler packages it.
- **`outDir`** separates compiled output from source files.
- **`rootDir`** keeps source structure predictable.
- **`strict`** improves correctness by enabling stronger type checks.
- **`sourceMap`** helps debug compiled code locally and in logs.

If you use ES modules, you must align `package.json`, file extensions, and Lambda runtime behavior carefully. For exam purposes, know that module format mismatches are a common source of deployment errors.

**Package configuration**

Your `package.json` should define scripts for build, test, and local execution. This keeps your workflow repeatable and makes CI/CD simpler.

Example:

```json
{
  "name": "aws-typescript-lambda",
  "version": "1.0.0",
  "private": true,
  "type": "commonjs",
  "scripts": {
    "build": "tsc",
    "clean": "rm -rf dist",
    "test": "node --test",
    "lint": "eslint .",
    "start": "node dist/handler.js"
  },
  "devDependencies": {
    "@types/node": "^22.0.0",
    "typescript": "^5.0.0"
  }
}
```

Practical notes:

- Use **`build`** to compile TypeScript before packaging or deployment.
- Use **`clean`** to remove stale artifacts that can hide build problems.
- Use **`start`** only for local smoke tests, not as a substitute for Lambda invocation.
- Add **`lint`** and test scripts early so code quality checks run before deployment.

**AWS SDK v3 usage**

The AWS SDK for JavaScript v3 is modular, so you import only the clients and commands you need. This reduces bundle size and improves cold-start characteristics compared with older monolithic patterns.

Install a service client:

```bash
npm install @aws-sdk/client-s3
```

Basic example:

```ts
import { S3Client, PutObjectCommand } from "@aws-sdk/client-s3";

const client = new S3Client({ region: process.env.AWS_REGION });

export async function uploadExample() {
  await client.send(
    new PutObjectCommand({
      Bucket: "example-bucket",
      Key: "hello.txt",
      Body: "hello from TypeScript"
    })
  );
}
```

Important AWS SDK v3 concepts:

- **Clients** represent a service, such as `S3Client` or `DynamoDBClient`.
- **Commands** represent operations, such as `PutObjectCommand`.
- **`send()`** executes the command and returns a promise.
- **Region** should usually come from `process.env.AWS_REGION` in Lambda.
- **Credentials** are resolved by the default provider chain in AWS environments.

> In Lambda, do not assume shared config files are the normal source of region settings. The runtime typically provides `AWS_REGION`, and the SDK uses the environment and execution role behavior expected in AWS.

**Credentials and region resolution**

In local development, you often authenticate with an AWS profile. In Lambda, you usually rely on the execution role rather than hard-coded credentials.

Typical local profile usage:

```ts
import { S3Client } from "@aws-sdk/client-s3";
import { fromIni } from "@aws-sdk/credential-providers";

const client = new S3Client({
  region: "us-east-1",
  credentials: fromIni({ profile: "dev" })
});
```

Important clarification:

- **`fromIni()`** returns a credential provider function, not static credentials.
- The SDK calls that provider when it needs credentials.
- This pattern is useful locally, but you should not bake profile-based credentials into deployed Lambda code.

For Lambda, prefer:

```ts
import { S3Client } from "@aws-sdk/client-s3";

const client = new S3Client({
  region: process.env.AWS_REGION
});
```

**Lambda handler structure**

A TypeScript Lambda handler should export a function with a signature that matches the event source. For API Gateway HTTP APIs, `APIGatewayProxyHandlerV2` is a common type.

Example:

```ts
import type { APIGatewayProxyHandlerV2 } from "aws-lambda";

export const main: APIGatewayProxyHandlerV2 = async (event) => {
  return {
    statusCode: 200,
    headers: { "content-type": "application/json" },
    body: JSON.stringify({
      message: "Hello from TypeScript",
      path: event.rawPath
    })
  };
};
```

Why this matters:

- The type helps you access event fields safely.
- The handler return shape must match the event source integration.
- Strong typing reduces mistakes when you parse headers, query strings, or request bodies.

Install Lambda types when needed:

```bash
npm install --save-dev @types/aws-lambda
```

**Build and packaging workflow**

For AWS deployment, you usually compile TypeScript into a separate output directory and package only the compiled JavaScript plus runtime dependencies.

A common workflow is:

1. Write source in `src/*.ts`.
2. Run `tsc` or a bundler to emit JavaScript into `dist/`.
3. Copy or bundle runtime dependencies.
4. Deploy the build artifact to Lambda, SAM, or another AWS service.

Example build commands:

```bash
npm run build
ls -R dist
```

If you use plain `tsc`, remember:

- TypeScript does not bundle dependencies.
- You must include `node_modules` or package dependencies separately.
- The Lambda handler path must point to the compiled file, not the `.ts` source.

**AWS SAM integration**

AWS Serverless Application Model (SAM) is a common way to define and deploy Lambda applications. In a TypeScript project, you typically build first and then point the template at the compiled artifact layout.

Example `template.yaml`:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Transform: AWS::Serverless-2016-10-31
Description: TypeScript Lambda example

Resources:
  HelloFunction:
    Type: AWS::Serverless::Function
    Properties:
      Runtime: nodejs22.x
      Handler: dist/handler.main
      CodeUri: .
      MemorySize: 128
      Timeout: 10
      Environment:
        Variables:
          NODE_OPTIONS: --enable-source-maps
```

Practical SAM notes:

- The handler path must match the built artifact structure.
- If your build process outputs to `dist/`, the deployed package must contain that directory.
- `CodeUri` points to the source or build context, but the final artifact layout determines what Lambda loads.
- For TypeScript projects, many teams use a SAM build hook, a Makefile, or a bundler so the deployed artifact is consistent.

A more robust approach is to use a build step that copies or bundles the handler into the expected location before deployment.

**Local testing and invocation**

You should test both pure functions and Lambda-shaped handlers. Unit tests are best for business logic, while integration-style tests validate event parsing and response formatting.

Example local smoke test using the compiled module:

```bash
node -e "import('./dist/handler.js').then(async (m) => {
  const event = { rawPath: '/test' };
  const context = {
    awsRequestId: 'local-test',
    functionName: 'hello',
    functionVersion: '$LATEST',
    memoryLimitInMB: '128',
    logGroupName: '/aws/lambda/hello',
    logStreamName: 'local',
    callbackWaitsForEmptyEventLoop: false
  };
  const result = await m.main(event, context, () => {});
  console.log(result);
})"
```

Use this carefully:

- This is a smoke test, not a full Lambda emulator.
- The handler contract still expects a realistic event and context shape.
- For deeper testing, use a framework such as SAM CLI or a dedicated test harness.

A better unit-test pattern is to isolate logic:

```ts
export function buildResponse(path: string) {
  return {
    statusCode: 200,
    body: JSON.stringify({ path })
  };
}
```

Then test `buildResponse()` directly without Lambda scaffolding.

**Testing tools**

For TypeScript AWS projects, common test tools include:

- **Node test runner** for lightweight built-in testing.
- **Jest** for mature mocking and snapshot support.
- **Vitest** for fast modern testing with TypeScript support.

Example with Node’s built-in test runner:

```ts
import test from "node:test";
import assert from "node:assert/strict";
import { buildResponse } from "../src/lib/buildResponse";

test("buildResponse returns path", () => {
  const result = buildResponse("/hello");
  assert.equal(result.statusCode, 200);
  assert.match(result.body, /\/hello/);
});
```

Testing guidance:

- Mock AWS SDK calls when you want fast unit tests.
- Use integration tests sparingly because they can incur cost and require credentials.
- Keep tests deterministic by avoiding real network calls unless the test is explicitly an integration test.

**Environment variables and configuration**

AWS applications often rely on environment variables for runtime configuration. In Lambda, the runtime injects values such as `AWS_REGION`, and you can add your own application settings.

Example:

```ts
const tableName = process.env.TABLE_NAME;
if (!tableName) {
  throw new Error("TABLE_NAME is required");
}
```

Best practices:

- Validate required variables at startup.
- Keep secrets in AWS Secrets Manager or AWS Systems Manager Parameter Store rather than plain environment variables when possible.
- Use environment variables for non-sensitive configuration such as table names, log levels, and feature flags.

**Source maps and debugging**

Source maps help you debug TypeScript more effectively by mapping runtime stack traces back to original source lines. This is especially useful in Lambda logs and local debugging sessions.

Enable source maps in `tsconfig.json`:

```json
{
  "compilerOptions": {
    "sourceMap": true
  }
}
```

Then enable Node.js source map support in Lambda or local runtime:

```yaml
Environment:
  Variables:
    NODE_OPTIONS: --enable-source-maps
```

Why this helps:

- Stack traces point to `.ts` files instead of compiled `.js`.
- You can diagnose runtime errors faster.
- It improves observability during development and in production logs.

**Bundling versus plain compilation**

You can either compile with `tsc` or bundle with a tool such as `esbuild`.

Comparison:

| Approach | Strengths | Tradeoffs |
|---|---|---|
| `tsc` only | Simple, clear output, good for learning | Does not bundle dependencies |
| `esbuild` | Fast, small bundles, good cold-start profile | Requires bundler-specific config |
| SAM build hooks | Integrates with deployment workflow | More moving parts |

Use bundling when:

- You want smaller deployment packages.
- You want to reduce startup overhead.
- You prefer a single-file artifact for Lambda.

Use plain compilation when:

- You want transparent output for debugging.
- Your project is small.
- You are learning the deployment flow.

**Common project scripts**

A practical `package.json` script set might look like this:

```json
{
  "scripts": {
    "clean": "rm -rf dist",
    "typecheck": "tsc --noEmit",
    "build": "npm run clean && tsc",
    "test": "node --test",
    "package": "npm run build && zip -r function.zip dist node_modules package.json"
  }
}
```

Notes on packaging:

- `typecheck` catches errors without generating output.
- `package` is useful for manual Lambda uploads.
- If you bundle dependencies, you may not need to include `node_modules`.

**Practical deployment checklist**

1. Install Node.js LTS and initialize the project.
2. Add TypeScript, Node types, and AWS SDK v3 clients.
3. Configure `tsconfig.json` for your runtime and output directory.
4. Write handlers in `src/` and keep business logic separate from Lambda glue code.
5. Build the project and verify the compiled output in `dist/`.
6. Test locally with realistic events and context objects.
7. Package the compiled artifact and dependencies correctly.
8. Deploy with SAM, CDK, or your chosen AWS workflow.
9. Confirm runtime variables such as `AWS_REGION` and application settings are available in Lambda.

**Exam-focused reminders**

- **Lambda region** usually comes from `AWS_REGION`, not from local shared config files.
- **Credential providers** such as `fromIni()` are functions that resolve credentials when called by the SDK.
- **Handler paths** must match the built artifact, not the TypeScript source file.
- **API Gateway event types** help you write safer handlers and reduce parsing mistakes.
- **Build output** and deployment packaging are separate concerns; both must be correct for a successful deployment.

**Troubleshooting patterns**

If deployment fails, check these first:

- The compiled file exists in the package at the path referenced by the handler.
- The runtime version matches your local assumptions.
- The module format matches the Lambda runtime and `package.json` settings.
- Required environment variables are present.
- AWS SDK clients are using the correct region and credentials source.

If local tests pass but Lambda fails, the issue is often packaging or runtime mismatch rather than business logic. Inspect the deployed artifact layout, the handler string, and the runtime logs before changing application code.
