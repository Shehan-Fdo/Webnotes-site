## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**AWS CLI identity and region basics**

The AWS Command Line Interface (AWS CLI) uses a layered configuration model so you can separate identities, environments, and target Regions. In practice, you often configure one default identity for day-to-day work and additional named profiles for specific roles, accounts, or environments such as `dev`, `test`, and `prod`.

A profile can define:
- **Credentials** such as access key ID and secret access key.
- **Region** such as `us-east-1` or `eu-west-1`.
- **Output format** such as `json`, `yaml`, or `text`.
- **Advanced providers** such as `role_arn`, `source_profile`, `credential_process`, or SSO/IAM Identity Center settings.

The CLI and SDKs do not always resolve settings the same way, so you should know both the file layout and the precedence rules. This matters for troubleshooting `AccessDenied`, `InvalidClientTokenId`, and “wrong Region” errors.

**Where AWS CLI settings live**

By default, the AWS CLI reads two files in your home directory:

- `~/.aws/credentials`
- `~/.aws/config`

The files serve different purposes:
- `credentials` stores long-term or generated secret material.
- `config` stores non-secret settings such as Region, output format, role assumptions, and SSO profile metadata.

A typical split looks like this:

```ini
# ~/.aws/credentials
[default]
aws_access_key_id = AKIAEXAMPLE
aws_secret_access_key = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY

[dev]
aws_access_key_id = AKIADEVEXAMPLE
aws_secret_access_key = devSecretExampleKey
```

```ini
# ~/.aws/config
[default]
region = us-east-1
output = json

[profile dev]
region = us-west-2
output = yaml
```

Note the naming difference:
- In `credentials`, the profile name is written as `[dev]`.
- In `config`, non-default profiles use `[profile dev]`.

**How the CLI resolves credentials**

The AWS CLI follows a provider chain. The exact chain can vary by command and version, but the exam-relevant order is:

1. **Command-line options**
   - Example: `aws s3 ls --profile dev --region us-west-2`
   - Explicit flags override everything else for that invocation.

2. **Environment variables**
   - Example: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`, `AWS_REGION`, `AWS_DEFAULT_REGION`, `AWS_PROFILE`
   - These are useful for ephemeral shells, CI/CD jobs, and containerized tasks.

3. **Shared configuration and credentials files**
   - `~/.aws/credentials`
   - `~/.aws/config`
   - Named profiles, role profiles, SSO profiles, and `credential_process` entries live here.

4. **Container and instance metadata sources**
   - ECS task role credentials
   - EC2 instance profile credentials from the Instance Metadata Service (IMDS)
   - These are common in AWS-hosted compute and are used when no higher-priority source is present.

For exam purposes, remember this practical rule:
- If you pass a value on the command line, it wins.
- If not, environment variables win over shared files.
- If neither is present, the CLI falls back to profile-based config and then metadata-based sources when available.

> Be careful when both `AWS_PROFILE` and explicit `--profile` are present. The command-line option takes precedence for that command.

**Creating and inspecting profiles**

You can create a profile interactively with `aws configure` or target a specific profile with `--profile`.

```bash
aws configure --profile dev
```

This prompts you for:
- AWS Access Key ID
- AWS Secret Access Key
- Default region name
- Default output format

The command writes values into the appropriate files. After configuration, inspect the raw files to verify what was written:

```bash
cat ~/.aws/credentials
cat ~/.aws/config
```

You can also edit the files directly when you need advanced settings that `aws configure` does not prompt for.

A practical profile example:

```ini
# ~/.aws/config
[profile dev]
region = us-west-2
output = json
role_arn = arn:aws:iam::123456789012:role/DeveloperRole
source_profile = base
```

```ini
# ~/.aws/credentials
[base]
aws_access_key_id = AKIABASEEXAMPLE
aws_secret_access_key = baseSecretExampleKey
```

In this pattern:
- `base` holds the long-term credentials.
- `dev` assumes a role using `source_profile`.
- The CLI uses the base profile to obtain temporary credentials for the role.

**Region configuration and why it matters**

The Region determines where the CLI sends API requests. Many AWS services are regional, so a wrong Region can make resources appear missing even when they exist.

Common symptoms of a Region mismatch:
- `ResourceNotFoundException`
- `NoSuchBucket`
- Empty `describe` results
- Requests going to the wrong endpoint

You can set the Region in several ways:
- Per command: `aws ec2 describe-instances --region us-west-2`
- Per profile: `region = us-west-2`
- Per shell session: `export AWS_REGION=us-west-2`

The most important distinction is between `AWS_REGION` and `AWS_DEFAULT_REGION`:
- **`AWS_REGION`** is the more explicit Region variable and is widely honored by the AWS CLI and SDKs.
- **`AWS_DEFAULT_REGION`** is also supported by the CLI and many SDKs, and is often used as a fallback or legacy-compatible setting.

For troubleshooting, set one Region source consistently. If both are present, behavior can differ by toolchain, shell, or SDK language binding, so avoid relying on mixed values.

Example:

```bash
export AWS_REGION=us-west-2
export AWS_DEFAULT_REGION=us-west-2
aws sts get-caller-identity
```

If you need a one-off override:

```bash
aws dynamodb list-tables --region eu-central-1
```

**Environment variables you should know**

Environment variables are convenient for automation, but they are easy to forget because they override file-based settings.

Common variables:
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `AWS_SESSION_TOKEN`
- `AWS_PROFILE`
- `AWS_REGION`
- `AWS_DEFAULT_REGION`
- `AWS_SHARED_CREDENTIALS_FILE`
- `AWS_CONFIG_FILE`
- `AWS_CA_BUNDLE`
- `AWS_PAGER`

Temporary session credentials require all three security values:
- Access key ID
- Secret access key
- Session token

Example:

```bash
export AWS_ACCESS_KEY_ID=ASIAEXAMPLE
export AWS_SECRET_ACCESS_KEY=exampleSecretKey
export AWS_SESSION_TOKEN=exampleSessionToken
export AWS_REGION=us-east-1
```

This is common when you obtain temporary credentials from STS (AWS Security Token Service), SSO, or an external credential broker.

**Custom file locations**

`AWS_SHARED_CREDENTIALS_FILE` and `AWS_CONFIG_FILE` let you point the CLI at non-default locations. This is useful when:
- You want to isolate credentials for different projects.
- You run multiple AWS toolchains on the same machine.
- You need to mount credentials into containers without using the home directory.
- You want to test with a controlled config in CI.

Example:

```bash
export AWS_SHARED_CREDENTIALS_FILE=/tmp/aws/credentials
export AWS_CONFIG_FILE=/tmp/aws/config
aws configure --profile sandbox
```

This keeps the AWS files separate from your normal `~/.aws` directory. It is especially useful in build agents and ephemeral development environments.

**SSO and IAM Identity Center profiles**

AWS CLI v2 supports AWS IAM Identity Center, formerly AWS Single Sign-On (SSO), as a first-class profile type. This is the preferred approach in many organizations because it avoids long-term access keys for human users.

Typical flow:
1. Create or update an SSO profile with `aws configure sso`.
2. Authenticate with `aws sso login --profile <name>`.
3. Use the profile with AWS CLI commands.

Example:

```bash
aws configure sso --profile dev-sso
aws sso login --profile dev-sso
aws sts get-caller-identity --profile dev-sso
```

An SSO profile stores metadata in `~/.aws/config`, not static secret keys. The CLI then exchanges the cached SSO session for temporary role credentials.

A simplified SSO profile looks like this:

```ini
[profile dev-sso]
sso_start_url = https://my-sso-portal.awsapps.com/start
sso_region = us-east-1
sso_account_id = 123456789012
sso_role_name = DeveloperAccess
region = us-west-2
output = json
```

Important behavior:
- `aws configure sso` helps you set up the profile interactively.
- `aws sso login` creates or refreshes the local SSO session cache.
- The CLI uses cached tokens until they expire, then you log in again.

**`credential_process` for external credential providers**

`credential_process` lets a profile call an external program that returns credentials in a standard JSON format. This is useful when credentials come from:
- A corporate identity broker
- A custom vault or secrets system
- A local helper that exchanges one token for another
- A non-AWS SSO integration

Example profile:

```ini
[profile brokered]
region = us-east-1
credential_process = /usr/local/bin/get-aws-creds --profile brokered
```

The helper must output JSON similar to:

```json
{
  "Version": 1,
  "AccessKeyId": "ASIAEXAMPLE",
  "SecretAccessKey": "exampleSecretKey",
  "SessionToken": "exampleSessionToken",
  "Expiration": "2026-04-26T18:00:00Z"
}
```

Use `credential_process` when you want the AWS CLI and SDKs to consume credentials without storing them directly in AWS config files. It is especially useful in enterprise environments.

**Role-based profiles and chaining**

A common pattern is to keep a base profile and assume roles into target accounts.

```ini
[profile base]
region = us-east-1

[profile prod-admin]
role_arn = arn:aws:iam::999999999999:role/AdminRole
source_profile = base
region = us-west-2
```

This works well when:
- You need separate access boundaries.
- You want to avoid copying static keys into multiple profiles.
- You manage multiple AWS accounts from one workstation.

You can test the active identity with:

```bash
aws sts get-caller-identity --profile prod-admin
```

If the role assumption fails, check:
- The base profile credentials
- Trust policy on the target role
- Permission to call `sts:AssumeRole`
- Correct account ID and role name
- Region only if the target service call is regional

**AWS CLI vs SDK behavior**

The AWS CLI and SDKs share the same general credential model, but the details can differ by language and runtime.

Key differences to remember:
- The CLI often honors `AWS_PROFILE` and `--profile` directly.
- SDKs usually rely on their language-specific default credential provider chain.
- Some SDKs prefer `AWS_REGION`, while others also accept `AWS_DEFAULT_REGION`.
- Some SDKs may require explicit region configuration in code if no environment variable is set.
- SSO support and token caching behavior can vary by SDK version.

Practical troubleshooting approach:
- Verify the CLI works first with `aws sts get-caller-identity`.
- Confirm the same profile and Region are visible to the application runtime.
- Check whether the SDK reads the same shared files or expects explicit configuration.
- Compare local shell variables with the environment used by the app, container, or test runner.

Example in Python:

```python
import boto3

session = boto3.Session(profile_name="dev")
client = session.client("s3", region_name="us-west-2")
print(client.list_buckets())
```

Example in Node.js:

```javascript
import { S3Client, ListBucketsCommand } from "@aws-sdk/client-s3";

const client = new S3Client({ region: process.env.AWS_REGION });
const response = await client.send(new ListBucketsCommand({}));
console.log(response);
```

Example in Java:

```java
S3Client s3 = S3Client.builder()
    .region(Region.of(System.getenv("AWS_REGION")))
    .build();
```

These examples show a common exam theme: the application may fail even when the CLI works if the app does not load the same profile or Region settings.

**Troubleshooting checklist**

When AWS authentication or Region selection fails, check the problem in this order:

1. Confirm the active identity.
   - Run `aws sts get-caller-identity`.
   - Add `--profile <name>` if needed.

2. Confirm the Region.
   - Run `aws configure list`.
   - Check `AWS_REGION`, `AWS_DEFAULT_REGION`, and profile `region`.

3. Confirm which credential source is winning.
   - Look for environment variables.
   - Check shared files.
   - Check role profiles, SSO profiles, or `credential_process`.

4. Confirm the files being read.
   - Inspect `AWS_SHARED_CREDENTIALS_FILE` and `AWS_CONFIG_FILE`.
   - Verify the expected profile names and section headers.

5. Confirm temporary credentials are still valid.
   - Session tokens expire.
   - SSO sessions expire.
   - Role credentials expire.

Useful inspection commands:

```bash
aws configure list
aws configure list-profiles
aws sts get-caller-identity --profile dev
env | grep '^AWS_'
```

`aws configure list` is especially helpful because it shows where each setting came from, which makes precedence issues easier to diagnose.

**Practical examples**

Create a new profile and verify the files:

```bash
aws configure --profile sandbox
cat ~/.aws/credentials
cat ~/.aws/config
```

Use a profile for a single command:

```bash
aws ec2 describe-regions --profile sandbox --region us-east-1
```

Set a shell-wide default Region for a session:

```bash
export AWS_REGION=ap-southeast-2
aws s3 ls
```

Use a custom config location in CI:

```bash
export AWS_SHARED_CREDENTIALS_FILE=$PWD/.aws/credentials
export AWS_CONFIG_FILE=$PWD/.aws/config
aws sts get-caller-identity --profile ci
```

Use SSO for a developer profile:

```bash
aws configure sso --profile team-dev
aws sso login --profile team-dev
aws s3 ls --profile team-dev
```

Use an external credential helper:

```ini
[profile vault]
region = us-east-1
credential_process = /opt/bin/aws-creds-from-vault --role developer
```

**Exam-ready takeaways**

- **Profiles** let you separate identities, Regions, and output formats.
- **Credentials** usually live in `~/.aws/credentials`; non-secret settings live in `~/.aws/config`.
- **Command-line options** override environment variables and shared files.
- **Environment variables** are powerful but can silently override profile settings.
- **`AWS_REGION`** and **`AWS_DEFAULT_REGION`** are both important; keep them consistent.
- **SSO/IAM Identity Center** in AWS CLI v2 uses `aws configure sso` and `aws sso login`.
- **`credential_process`** is the advanced hook for external credential providers.
- **Custom file paths** via `AWS_SHARED_CREDENTIALS_FILE` and `AWS_CONFIG_FILE` are useful for isolation and automation.
- **SDKs** may need explicit profile or Region configuration even when the CLI works.
