## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**AWS CLI v2 overview**

The <abbr title="Command Line Interface">CLI</abbr> lets you manage AWS services from a terminal, script, or automation pipeline. AWS CLI v2 is the current major version and is the preferred choice for most developer and operations workflows because it includes newer features, improved defaults, and broader support for modern authentication and output handling.

Use the CLI when you need to:

- Automate repetitive AWS tasks.
- Test APIs quickly without writing application code.
- Inspect resources during debugging.
- Integrate AWS operations into build and deployment scripts.

A clean installation matters because many AWS tasks depend on the exact CLI version, profile configuration, and credential resolution order.

**What AWS CLI v2 changes**

AWS CLI v2 is distributed as a standalone installer and does not require Python on the host machine. That makes installation simpler and more predictable than older v1 setups.

Key practical differences:

- **Bundled runtime**: v2 includes its own runtime dependencies.
- **Modern installer**: installation is usually a package, archive, or MSI-based process.
- **Version verification**: you should confirm the binary path and version after install.
- **Script compatibility**: most common commands are the same, but some output and auth behaviors differ from v1.

> If you already have AWS CLI v1 installed, verify which binary is first in your `PATH` before assuming the upgrade succeeded.

**Installation methods by platform**

Choose the installer that matches your operating system and package management style. The goal is to place the `aws` executable on your `PATH` and confirm that the shell resolves the expected binary.

**Linux installation**

On Linux, AWS CLI v2 is commonly installed from the bundled `.zip` archive.

Typical flow:

1. Download the installer archive.
2. Unzip it.
3. Run the install script with elevated privileges.
4. Verify the installed version.

Example:

```bash
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install
aws --version
```

For ARM-based Linux systems, use the ARM package instead of the x86_64 package:

```bash
curl "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install
aws --version
```

Useful Linux checks:

```bash
which aws
ls -l "$(which aws)"
aws --version
```

**macOS installation**

On macOS, you can install AWS CLI v2 using the package installer or a package manager if your environment standardizes on one.

Package installer flow:

```bash
curl "https://awscli.amazonaws.com/AWSCLIV2.pkg" -o "AWSCLIV2.pkg"
sudo installer -pkg AWSCLIV2.pkg -target /
aws --version
```

If you use Homebrew, the package manager may install a managed copy of the CLI. That is convenient, but you should still verify the resolved binary:

```bash
brew install awscli
which aws
aws --version
```

**Windows installation**

On Windows, the standard approach is the MSI installer.

Typical steps:

1. Download the `.msi` installer.
2. Run the installer with administrative privileges if required.
3. Open a new terminal session.
4. Verify the version.

PowerShell example:

```powershell
msiexec.exe /i https://awscli.amazonaws.com/AWSCLIV2.msi
aws --version
```

Windows verification commands:

```powershell
where.exe aws
Get-Command aws
aws --version
```

**Verifying the installation**

Version verification is not just a formality. It confirms that:

- The shell can find the `aws` executable.
- The installed binary is AWS CLI v2.
- The binary is the one you expect, not an older copy from another toolchain.

Run:

```bash
aws --version
```

A typical v2 response looks like:

```text
aws-cli/2.x.x Python/3.x.x Linux/5.x.x exe/x86_64 prompt/off
```

On macOS or Windows, the platform details differ, but the `aws-cli/2` prefix should still appear.

Additional checks:

```bash
aws help
aws sts get-caller-identity
```

The `aws help` command confirms the CLI is functional. The `sts get-caller-identity` command confirms that credentials are configured and that the CLI can reach AWS STS (<abbr title="Security Token Service">STS</abbr>).

> `aws --version` only confirms the binary is installed. It does not prove your credentials, region, or permissions are correct.

**PATH and binary resolution**

The most common installation issue is not the installer itself, but shell resolution. Your terminal uses the first matching `aws` executable found in `PATH`.

Check the resolved binary:

```bash
which aws
```

On Windows PowerShell:

```powershell
Get-Command aws
```

If you see an unexpected path, you may have:

- An older AWS CLI v1 installation earlier in `PATH`.
- A package-manager-managed install that differs from the manual install.
- Multiple shells with different environment settings.

Practical troubleshooting steps:

- Open a new terminal after installation.
- Confirm the installer completed without errors.
- Remove or reorder conflicting `aws` binaries if needed.
- Re-run `aws --version` after adjusting `PATH`.

**Common verification patterns**

Use these checks in order when validating a new setup:

1. Confirm the binary exists:
   ```bash
   which aws
   ```
2. Confirm the version:
   ```bash
   aws --version
   ```
3. Confirm the CLI responds to help:
   ```bash
   aws help
   ```
4. Confirm AWS access with credentials:
   ```bash
   aws sts get-caller-identity
   ```

This sequence separates installation problems from authentication problems.

**Configuration after install**

After installation, you usually configure credentials and a default region. The CLI reads settings from shared configuration files and environment variables.

Common setup command:

```bash
aws configure
```

This prompts for:

- AWS Access Key ID
- AWS Secret Access Key
- Default region name
- Default output format

Example profile-based configuration:

```bash
aws configure --profile dev
aws sts get-caller-identity --profile dev
```

Useful environment variables:

```bash
export AWS_PROFILE=dev
export AWS_REGION=us-east-1
```

On Windows PowerShell:

```powershell
$env:AWS_PROFILE = "dev"
$env:AWS_REGION = "us-east-1"
```

**Practical examples**

List S3 buckets after installation and configuration:

```bash
aws s3 ls
```

Inspect your caller identity:

```bash
aws sts get-caller-identity
```

Query EC2 instances in a region:

```bash
aws ec2 describe-instances --region us-east-1
```

These commands are useful because they validate different layers:

- `aws s3 ls` checks service access and permissions.
- `aws sts get-caller-identity` checks identity resolution.
- `aws ec2 describe-instances` checks regional API access.

**Troubleshooting installation issues**

If `aws` is not recognized, the issue is usually one of the following:

- The installer did not complete successfully.
- The binary directory is not in `PATH`.
- You need to restart the terminal session.
- Another installation shadows the new one.

If `aws --version` shows v1 instead of v2:

- Check `which aws` or `Get-Command aws`.
- Remove the older binary or adjust `PATH`.
- Reinstall v2 if necessary.

If the CLI runs but AWS commands fail:

- Verify credentials with `aws configure list`.
- Check the active profile.
- Confirm the region is set.
- Ensure the IAM <abbr title="Identity and Access Management">IAM</abbr> permissions allow the requested action.

Helpful diagnostic command:

```bash
aws configure list
```

This shows where the CLI is sourcing credentials, region, and output settings.

**Why this matters for exam prep**

For AWS Certified Developer – Associate, you should know how to:

- Install AWS CLI v2 on common operating systems.
- Verify the installed version and binary path.
- Distinguish installation problems from credential problems.
- Use the CLI for quick validation of AWS access.

The exam may not ask you to memorize every installer flag, but it does expect you to understand how the CLI fits into development and troubleshooting workflows.

**Quick reference**

- **Install**: use the platform-appropriate AWS CLI v2 package or archive.
- **Verify**: run `aws --version`.
- **Resolve path**: use `which aws` or `Get-Command aws`.
- **Test access**: run `aws sts get-caller-identity`.
- **Configure**: use `aws configure` or profiles.
