## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why these tools matter**

AWS Cloud Development Kit (AWS CDK) and AWS Serverless Application Model (AWS SAM) CLI are common developer tools for building, testing, and deploying AWS workloads from code. CDK helps you define infrastructure as code using familiar programming languages, while SAM focuses on serverless application packaging, local testing, and deployment workflows.

For exam prep, you should know not only how to install them, but also how they fit into a typical development lifecycle:

- **CDK v2** is used to define AWS resources in code and synthesize them into CloudFormation templates.
- **SAM CLI** is used to build, test, and deploy serverless applications, especially Lambda-based stacks.
- Both tools depend on a working AWS CLI-style environment, valid AWS credentials, and a supported runtime on your machine.

**Prerequisites and environment checks**

Before installing either tool, confirm that your workstation is ready. Most installation issues come from missing runtimes, PATH problems, or version mismatches.

- **Node.js** is required for AWS CDK CLI usage and for many CDK app projects.
- **Python**, **Java**, **Go**, **.NET**, or **TypeScript** may be needed depending on your CDK app language.
- **Docker** is often required for SAM local testing and for some CDK asset workflows.
- **AWS credentials** should already be configured with `aws configure`, environment variables, or an IAM Identity Center profile.

A quick verification sequence helps you catch problems early:

```bash
node -v
npm -v
python3 --version
docker --version
aws --version
```

If a command fails, fix the runtime or PATH issue before installing the AWS tools. On Windows, also confirm that the installer added the binaries to your user PATH and that you opened a new terminal session after installation.

**AWS CDK v2 installation**

AWS CDK v2 uses a single major version line, which reduces confusion compared with older v1-era package naming. The CLI is commonly installed globally with npm. Use a version-pinned install when you want to avoid accidental major-version drift.

```bash
npm install -g aws-cdk@2
cdk --version
```

If you prefer a project-local workflow, you can install the CLI globally and keep the library dependencies in your app’s `package.json`. A typical CDK app also needs the language-specific libraries for the AWS constructs you use.

```bash
mkdir my-cdk-app
cd my-cdk-app
cdk init app --language typescript
npm install
```

For TypeScript projects, the generated app usually includes dependencies such as `aws-cdk-lib` and `constructs`. The CLI version and library version should stay compatible, so avoid mixing very old project templates with a much newer CLI.

**CDK prerequisites and version guidance**

CDK depends on a supported Node.js runtime. In practice, use an active LTS release rather than an outdated version. If `cdk` installs but does not run, the most common causes are:

- **PATH issues** where the global npm binary directory is not visible in your shell.
- **Permission issues** when installing globally without the right npm prefix or admin rights.
- **Version mismatch** between the CDK CLI and the app libraries.
- **Unsupported Node.js version** that breaks the CLI or your app dependencies.

Useful checks:

```bash
npm prefix -g
npm root -g
which cdk
where cdk
```

On Unix-like systems, `which cdk` should point to the global npm bin directory. On Windows, `where cdk` should resolve to the installed executable.

If global installs are problematic, you can use `npx` with a pinned version:

```bash
npx aws-cdk@2 --version
```

This is helpful in CI/CD pipelines or when you want to avoid a machine-wide global install.

**CDK bootstrap and environment setup**

`cdk bootstrap` prepares an AWS environment for CDK deployments. It creates the resources that CDK uses to publish assets and support modern deployments, including the default bootstrap stack in the target account and Region.

You usually need bootstrapping when:

- You deploy stacks that use file assets, Lambda code assets, container assets, or other published artifacts.
- You use the default modern CDK deployment workflow.
- You deploy to a new account or Region that has not been bootstrapped yet.

A typical bootstrap command looks like this:

```bash
cdk bootstrap aws://123456789012/us-east-1
```

Bootstrapping commonly creates:

- An S3 bucket for file assets.
- An ECR repository for container assets when needed.
- IAM roles used by deployment and asset publishing workflows.
- A CloudFormation bootstrap stack that CDK references during deployment.

> If you see deployment failures mentioning missing bootstrap resources or asset publishing roles, the target account/Region likely needs `cdk bootstrap`.

You can inspect the synthesized template before deploying:

```bash
cdk synth
```

And compare the synthesized CloudFormation template with the deployed stack template:

```bash
cdk diff
```

`cdk diff` does **not** compare live resources directly; it compares what CDK synthesized locally against the currently deployed CloudFormation template. That distinction matters when drift exists between the template and actual resources.

**CDK install, upgrade, and uninstall workflow**

When you need to upgrade CDK, keep the CLI and libraries aligned. A safe approach is to update the CLI intentionally and then update project dependencies.

```bash
npm install -g aws-cdk@2
npm update aws-cdk-lib constructs
```

If you use a project-local CLI version, update the dev dependency instead of the global package. This is often easier to reproduce in teams and CI.

To remove a global install:

```bash
npm uninstall -g aws-cdk
```

If you suspect a broken global install, reinstall after clearing the npm cache only if needed:

```bash
npm cache verify
npm install -g aws-cdk@2
```

**AWS SAM CLI installation**

AWS SAM CLI is available through several platform-specific installation methods. Choose the method that matches your operating system and team workflow.

Common installation paths include:

- **macOS**: Homebrew or the official package installer.
- **Windows**: MSI installer.
- **Linux**: Package manager, RPM/DEB package, or the official install instructions for your distribution.

Examples:

```bash
# macOS with Homebrew
brew tap aws/tap
brew install aws-sam-cli

# macOS upgrade
brew upgrade aws-sam-cli
```

```powershell
# Windows PowerShell example after MSI install
sam --version
```

```bash
# Linux package-based install example varies by distro
sam --version
```

Because Linux distributions differ, follow the package format that matches your system. If you install from a downloaded package, make sure the binary directory is on your PATH.

**SAM CLI prerequisites and verification**

SAM CLI relies on a working local runtime environment. In many workflows, you also need Docker for local invocation and API emulation.

Check the installation:

```bash
sam --version
sam --help
```

If `sam` is not found:

- Confirm the installer completed successfully.
- Open a new terminal session so PATH changes take effect.
- Verify the install location is included in your shell PATH.
- On macOS and Linux, check file permissions if the binary exists but cannot run.

A practical environment check:

```bash
docker ps
sam --version
aws sts get-caller-identity
```

If Docker is unavailable, `sam local invoke` and `sam local start-api` may fail or behave unexpectedly because local emulation often depends on container execution.

**SAM project workflow and configuration**

SAM projects usually follow a build-and-deploy flow. A common sequence is:

1. Build the application with `sam build`.
2. Test locally with `sam local invoke` or `sam local start-api`.
3. Deploy with `sam deploy --guided` for first-time setup.
4. Reuse the generated configuration in later deployments.

A minimal example:

```bash
sam init
cd my-sam-app
sam build
sam local start-api
sam deploy --guided
```

`sam local invoke` often requires a function logical ID or an event file, so do not assume it runs with no arguments in real projects.

```bash
sam local invoke HelloWorldFunction -e events/event.json
```

The guided deployment flow usually creates or updates `samconfig.toml`, which stores deployment settings such as stack name, Region, and parameter overrides. This file helps you avoid retyping the same options every time.

Example `samconfig.toml`:

```toml
version = 0.1

[default.deploy.parameters]
stack_name = "my-sam-app"
region = "us-east-1"
confirm_changeset = true
capabilities = "CAPABILITY_IAM"
resolve_s3 = true
```

This file is important for repeatable deployments and team consistency. If you change Regions or accounts, review the stored values before deploying.

**SAM deployment details**

`sam deploy --guided` is useful for first-time deployments because it prompts you for stack name, Region, capabilities, and artifact handling. It also writes deployment preferences to `samconfig.toml`.

Typical deployment flow:

```bash
sam build
sam deploy --guided
```

You often need `CAPABILITY_IAM` or `CAPABILITY_NAMED_IAM` when your template creates IAM resources. If you omit required capabilities, deployment fails with a CloudFormation permission error.

Useful deployment flags:

```bash
sam deploy --stack-name my-sam-app --capabilities CAPABILITY_IAM
sam deploy --no-confirm-changeset
sam deploy --resolve-s3
```

**CDK vs SAM: when to use each**

Both tools can deploy AWS infrastructure, but they solve slightly different problems.

| Tool | Best fit | Strengths | Typical workflow |
|---|---|---|---|
| CDK v2 | General infrastructure and application stacks | Rich abstractions, multi-service architectures, reusable constructs | `cdk synth`, `cdk diff`, `cdk deploy` |
| SAM CLI | Serverless applications | Fast Lambda-centric packaging, local testing, guided deployment | `sam build`, `sam local invoke`, `sam deploy --guided` |

Use **CDK** when you want to model broader infrastructure, compose reusable constructs, or manage complex environments. Use **SAM** when your project is primarily serverless and you want a streamlined Lambda-focused developer experience.

A realistic example:

- You choose **CDK** for an application that includes API Gateway, Lambda, DynamoDB, SQS, and custom networking.
- You choose **SAM** for a smaller serverless API where you want quick local testing and simple deployment settings.

**Troubleshooting common installation problems**

Most setup failures are easy to diagnose once you know the usual patterns.

- **`command not found`** means the binary is not on PATH or the install failed.
- **Permission denied** often means the global npm prefix or installer location is not writable.
- **Version mismatch** can happen when the CLI is newer than the libraries or when Node.js is too old.
- **Docker errors** usually affect SAM local workflows rather than deployment itself.
- **AWS credential errors** indicate that the CLI can run, but it cannot authenticate to AWS.

Examples of useful diagnostics:

```bash
echo $PATH
npm config get prefix
cdk doctor
sam --debug --version
aws configure list
```

`cdk doctor` can help identify environment issues in CDK setups. For SAM, `--debug` often reveals installer or runtime problems more clearly.

If you need to clean up a broken SAM installation, uninstall it using the method that matches how you installed it:

```bash
# Homebrew
brew uninstall aws-sam-cli

# Example package removal on Linux varies by distro
sudo apt remove aws-sam-cli
```

Then reinstall using the recommended method for your platform.

**Practical end-to-end setup flow**

A simple developer workstation setup often looks like this:

1. Install Node.js LTS and verify `node` and `npm`.
2. Install AWS CDK v2 with `npm install -g aws-cdk@2`.
3. Install AWS SAM CLI using the platform-specific installer.
4. Verify `cdk --version` and `sam --version`.
5. Configure AWS credentials with `aws configure` or an IAM Identity Center profile.
6. Bootstrap the target account and Region with `cdk bootstrap` if you plan to deploy CDK stacks.
7. Run `sam build` and `sam deploy --guided` for SAM projects, or `cdk synth` and `cdk deploy` for CDK projects.

A quick visual flow:

```text
Install runtime -> Install tools -> Verify PATH -> Configure AWS credentials
        -> Bootstrap CDK environment -> Build/test -> Deploy
```

**Exam-focused takeaways**

- **CDK v2** is installed commonly with `npm install -g aws-cdk@2`.
- **SAM CLI** is installed with a platform-specific installer or package manager.
- **Bootstrap** is required for many CDK deployments because it creates the resources CDK uses for asset publishing and deployment support.
- **`cdk diff`** compares synthesized templates with deployed CloudFormation templates, not live resources.
- **`sam build`** usually comes before `sam deploy --guided`.
- **`samconfig.toml`** stores guided deployment settings for repeatable SAM deployments.
- **PATH, Node.js, Docker, and permissions** are the most common setup problems.
