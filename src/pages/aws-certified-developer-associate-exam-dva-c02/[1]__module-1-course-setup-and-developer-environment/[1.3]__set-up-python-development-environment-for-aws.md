## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why this setup matters**

A reliable Python development environment helps you write, test, and deploy AWS-integrated code with fewer surprises. For AWS development, you typically need three things working together: a local Python runtime, isolated dependencies, and a secure way to authenticate to AWS services.

The goal is not just to “install Python.” You want a repeatable setup that supports the AWS SDK for Python, local testing, and safe credential handling across projects.

**Core components of a Python AWS environment**

A practical setup usually includes:

- **Python interpreter** for running application code and scripts.
- **Virtual environment** to isolate package versions per project.
- **AWS SDK for Python (`boto3`)** for service calls.
- **AWS CLI** for configuration, verification, and operational tasks.
- **Credential provider chain** so code can authenticate without hardcoding secrets.
- **Local tooling** such as linters, formatters, and test frameworks.

A clean environment reduces dependency conflicts, especially when different projects require different versions of `boto3`, `botocore`, or supporting libraries.

**Install Python and verify the runtime**

Use a supported Python version for your project and keep it consistent across your team and CI/CD pipeline. On many systems, you can verify the installed version with:

```bash
python3 --version
pip3 --version
```

If multiple Python versions exist, be explicit about which interpreter you use. On Unix-like systems, `python3` is safer than `python` because `python` may point to Python 2 or another interpreter depending on the machine.

A typical project layout starts with a dedicated directory:

```bash
mkdir my-aws-app
cd my-aws-app
```

**Create and activate a virtual environment**

A virtual environment isolates dependencies from the system Python installation. This is the standard approach for Python projects because it prevents package conflicts and makes builds reproducible.

Create one with the built-in `venv` module:

```bash
python3 -m venv .venv
```

Activate it:

```bash
source .venv/bin/activate
```

On Windows PowerShell:

```powershell
.venv\Scripts\Activate.ps1
```

After activation, confirm that `python` and `pip` point to the environment:

```bash
which python
which pip
```

On Windows:

```powershell
Get-Command python
Get-Command pip
```

When you finish working, deactivate it:

```bash
deactivate
```

> Keep the virtual environment out of version control. Add `.venv/` to `.gitignore`.

**Install AWS SDK and common development packages**

For AWS development in Python, `boto3` is the primary SDK. It uses `botocore` under the hood for low-level service communication.

Install the SDK and useful tooling:

```bash
pip install boto3
pip install pytest black flake8 mypy
```

Common package roles:

- **`boto3`**: AWS service access from Python.
- **`pytest`**: test runner for unit and integration tests.
- **`black`**: opinionated code formatter.
- **`flake8`**: style and lint checks.
- **`mypy`**: static type checking for typed Python code.

If you want reproducible installs, pin dependencies in `requirements.txt`:

```txt
boto3==1.34.0
pytest==8.2.0
black==24.4.2
flake8==7.0.0
mypy==1.10.0
```

Then install from the file:

```bash
pip install -r requirements.txt
```

For larger projects, you may also use `pip-tools`, Poetry, or another dependency manager. The key exam concept is isolation and repeatability, not a specific tool.

**Configure AWS credentials securely**

Your code should authenticate using the AWS credential provider chain rather than embedding access keys in source code. The AWS SDK and CLI can read credentials from several sources, including environment variables, shared config files, IAM roles, and AWS SSO/Identity Center in supported workflows.

The most common local setup uses the AWS CLI:

```bash
aws configure
```

This typically stores values in:

- `~/.aws/credentials`
- `~/.aws/config`

Example profile configuration:

```ini
[default]
region = us-east-1
output = json

[dev]
region = us-west-2
output = json
```

Example credentials file:

```ini
[default]
aws_access_key_id = AKIAEXAMPLE
aws_secret_access_key = exampleSecretKey
```

Prefer named profiles for multiple environments:

```bash
aws configure --profile dev
```

Then use the profile in your shell:

```bash
export AWS_PROFILE=dev
```

Or in code:

```python
import boto3

session = boto3.Session(profile_name="dev")
s3 = session.client("s3")
print(s3.list_buckets())
```

> Do not hardcode long-term access keys in application code, notebooks, or committed configuration files.

**Understand the credential chain**

When `boto3` needs credentials, it searches multiple locations in a defined order. In practice, this means you can run the same code locally, in CI, and on AWS infrastructure with different authentication sources.

Common sources include:

- Environment variables such as `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY`
- Shared credentials and config files under `~/.aws/`
- IAM role credentials from EC2, ECS, or Lambda
- AWS SSO/Identity Center sessions
- Web identity or federated identity flows in supported environments

This matters because you should design code to be environment-agnostic. Your application should ask the SDK for credentials, not manage secrets itself.

**Use environment variables for quick local testing**

Environment variables are useful for temporary testing or CI jobs. Example:

```bash
export AWS_ACCESS_KEY_ID=AKIAEXAMPLE
export AWS_SECRET_ACCESS_KEY=exampleSecretKey
export AWS_REGION=us-east-1
```

For session-based credentials, also set:

```bash
export AWS_SESSION_TOKEN=exampleSessionToken
```

This approach is convenient, but it is not ideal for long-lived local development because shell history, process listings, and accidental exposure can create risk.

**Test the setup with the AWS CLI**

Before writing application code, verify that the CLI can authenticate and reach AWS:

```bash
aws sts get-caller-identity
```

This command returns the account, user, and ARN associated with the current credentials. It is one of the fastest ways to confirm that your environment is configured correctly.

You can also test service access:

```bash
aws s3 ls
aws lambda list-functions
```

If these commands fail, check:

- The active profile
- The selected region
- Whether credentials are expired
- Whether the IAM permissions are sufficient

**Write a minimal boto3 test script**

A small script helps confirm that Python, `boto3`, and credentials all work together.

```python
import boto3

def main():
    sts = boto3.client("sts")
    identity = sts.get_caller_identity()
    print(identity)

if __name__ == "__main__":
    main()
```

Run it with:

```bash
python app.py
```

If you use a named profile:

```python
import boto3

session = boto3.Session(profile_name="dev", region_name="us-west-2")
s3 = session.client("s3")
print(s3.list_buckets())
```

This pattern is useful when you want explicit control over region and profile selection during development.

**Manage regions intentionally**

AWS services are regional, so your environment should set a default region. If you omit it, some SDK calls may fail or behave unexpectedly depending on the service and configuration.

You can set the region in several ways:

- In `~/.aws/config`
- With `AWS_REGION` or `AWS_DEFAULT_REGION`
- In `boto3.Session(region_name="...")`
- Per client call in some SDK patterns

Example:

```python
import boto3

dynamodb = boto3.resource("dynamodb", region_name="us-east-1")
```

Use a consistent region in local development to avoid accidentally creating resources in the wrong place.

**Use a project structure that scales**

A simple but maintainable layout might look like this:

```text
my-aws-app/
├── .venv/
├── src/
│   └── app.py
├── tests/
│   └── test_app.py
├── requirements.txt
├── pyproject.toml
└── .gitignore
```

This structure separates application code from tests and keeps environment-specific files out of source control.

A minimal `pyproject.toml` for tooling might include formatter and linter settings:

```toml
[tool.black]
line-length = 88

[tool.pytest.ini_options]
testpaths = ["tests"]
```

**Add local testing and mocking**

You should test AWS-integrated code without always calling live services. Unit tests often mock `boto3` clients, while integration tests may use a sandbox AWS account.

Example with `unittest.mock`:

```python
from unittest.mock import Mock, patch
import boto3

@patch("boto3.client")
def test_list_buckets(mock_client):
    s3 = Mock()
    s3.list_buckets.return_value = {"Buckets": []}
    mock_client.return_value = s3

    client = boto3.client("s3")
    result = client.list_buckets()

    assert result == {"Buckets": []}
```

For more realistic local AWS testing, you may use tools such as:

- **Moto** for mocking many AWS services in Python tests
- **LocalStack** for local emulation of AWS APIs

These tools are helpful when you want faster feedback and fewer calls to real AWS resources. They are not perfect substitutes for AWS, so validate critical paths in a real account as well.

**Handle secrets and configuration safely**

Separate secrets from code. Use environment variables, AWS profiles, or a secrets manager service for runtime values.

Good practices include:

- Store secrets in AWS Secrets Manager or AWS Systems Manager Parameter Store when appropriate.
- Keep `.env` files out of version control.
- Use different profiles or roles for dev, test, and prod.
- Rotate access keys if they are used at all.
- Prefer temporary credentials over long-term credentials.

Example `.gitignore` entries:

```gitignore
.venv/
__pycache__/
.env
.aws/
```

Be careful with `.aws/`; many teams do not commit it, but you should not blindly ignore it in every repository if you intentionally store non-secret config artifacts elsewhere. The important point is to avoid committing credentials.

**Use IAM roles when running on AWS**

When your Python code runs on AWS compute services, prefer IAM roles instead of static credentials.

Typical examples:

- **Lambda** uses an execution role.
- **EC2** uses an instance profile.
- **ECS** tasks use a task role.
- **EKS** workloads can use IAM roles for service accounts.

This approach removes the need to distribute access keys and improves security posture. Your local development environment may use profiles, but production should usually rely on role-based access.

**Troubleshoot common setup issues**

Common problems and likely causes:

- **`ModuleNotFoundError: No module named 'boto3'`**
  - The virtual environment is not activated.
  - `boto3` is installed in a different interpreter.

- **`Unable to locate credentials`**
  - No profile, environment variables, or role is available.
  - The AWS CLI was not configured correctly.

- **`ExpiredToken`**
  - Temporary credentials expired.
  - Re-authenticate or refresh the session.

- **`AccessDenied`**
  - The IAM principal lacks permission for the action.
  - The policy may allow the service but not the specific resource.

- **Wrong region errors**
  - The SDK is pointed at the wrong region.
  - The resource exists in a different region.

A quick diagnostic sequence is often:

```bash
which python
which pip
python --version
pip show boto3
aws configure list
aws sts get-caller-identity
```

**Practical workflow for a new project**

1. Create a project directory.
2. Create and activate a virtual environment.
3. Install `boto3` and development tools.
4. Configure an AWS profile with the CLI.
5. Verify identity with `aws sts get-caller-identity`.
6. Write a minimal `boto3` script.
7. Add tests and mock AWS calls where appropriate.
8. Commit only source code and configuration, not credentials.

This workflow gives you a repeatable baseline for building AWS applications in Python.

**Exam-focused takeaways**

For the AWS Certified Developer – Associate exam, focus on these ideas:

- **Isolation**: use virtual environments to avoid dependency conflicts.
- **Authentication**: rely on the AWS credential provider chain, not hardcoded keys.
- **Verification**: use `aws sts get-caller-identity` and small `boto3` scripts to confirm setup.
- **Security**: prefer IAM roles and temporary credentials over static secrets.
- **Repeatability**: pin dependencies and keep environment configuration consistent.

A well-built Python AWS environment supports faster development, safer testing, and fewer deployment surprises.
