## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Purpose of environment validation**

Environment validation is the process of proving that your local workstation, credentials, SDKs, and deployment tools can actually talk to AWS and to any local emulators you rely on. For exam prep, think of it as a fast, low-risk way to separate toolchain problems from application problems.

A good smoke test answers four questions:

- Can you authenticate?
- Can you reach the intended region?
- Can your SDK or CLI make a minimal API call?
- Can your build and deploy tools run end to end?

This matters because many failures are not caused by code. They come from missing credentials, stale shells, wrong regions, Docker not running, or an incomplete bootstrap/setup step.

**Common prerequisites to verify first**

Before you troubleshoot AWS-specific behavior, confirm the local environment itself is healthy.

- **AWS CLI** is installed and on your `PATH`.
- **SDK runtime** is installed for the language you use, such as Python, Node.js, Java, or .NET.
- **Docker** is running if you use AWS SAM CLI local commands or local emulators that depend on containers.
- **Shell profile** changes have been reloaded after installation.
- **Network access** is available to AWS endpoints, proxies, or private endpoints if your environment uses them.

Typical checks:

```bash
aws --version
python3 --version
node --version
java -version
docker version
```

If a command is not found, the issue is often installation or `PATH`, not AWS configuration.

> If you just installed a tool and the shell still cannot find it, reload the profile or open a new terminal session. On macOS and Linux that may mean sourcing `~/.zshrc`, `~/.bashrc`, or the relevant profile file.

**AWS CLI validation**

The AWS CLI is the fastest way to confirm credentials and region settings. Start with a non-destructive identity call.

```bash
aws sts get-caller-identity
```

This returns the account, ARN, and user or role identity that the CLI is using. It is one of the best first smoke tests because it validates:

- credential resolution
- basic network connectivity
- STS access
- the active AWS account context

If you need to confirm the active region, inspect configuration rather than guessing:

```bash
aws configure list
aws configure list-profiles
aws configure get region
```

You can also test a specific profile explicitly:

```bash
aws sts get-caller-identity --profile dev
aws sts get-caller-identity --region us-east-1
```

**Credential resolution and profiles**

AWS tools use a shared configuration model. Credentials may come from environment variables, shared config files, named profiles, SSO, role assumption, or `credential_process`. The important exam idea is not that one tool has a special feature, but that the same credential chain can be exercised in different ways.

Useful files and settings:

- `~/.aws/credentials`
- `~/.aws/config`
- `AWS_PROFILE`
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `AWS_SESSION_TOKEN`
- `AWS_DEFAULT_REGION`

Example profile configuration:

```ini
[profile dev]
region = us-west-2
output = json

[profile dev-admin]
role_arn = arn:aws:iam::123456789012:role/AdminRole
source_profile = dev
region = us-west-2
```

Example environment variable usage:

```bash
export AWS_PROFILE=dev
export AWS_DEFAULT_REGION=us-west-2
aws sts get-caller-identity
```

For troubleshooting, prefer explicit profile selection over guessing which credentials are active.

**Region validation**

Region mistakes are a common source of false failures. Validate the region in the same command you use for the test.

- Use `--region` when you want a one-off override.
- Use profile configuration when the region should be stable for that profile.
- Check the region used by the service you are testing, especially for regional services like Lambda, DynamoDB, S3, and CloudWatch Logs.

Example:

```bash
aws lambda list-functions --region us-east-1
aws dynamodb list-tables --region us-west-2
```

If a resource is “missing,” confirm that you are querying the correct region before assuming it was deleted.

**IAM permission debugging**

When authentication works but an action fails, the next question is usually authorization. A useful tool is IAM policy simulation.

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:user/dev-user \
  --action-names lambda:InvokeFunction s3:PutObject \
  --resource-arns arn:aws:lambda:us-west-2:123456789012:function:my-function
```

Use simulation to isolate whether the failure is caused by:

- missing identity-based permissions
- an explicit deny
- a permissions boundary
- a session policy
- a resource-based policy mismatch

This is especially helpful when a role can list resources but cannot invoke, write, or assume another role.

**Language SDK smoke tests**

SDK smoke tests should be minimal, deterministic, and focused on one successful API call. The goal is to prove that the runtime, credentials, and region are wired correctly.

**Python example**

Use `boto3` and call STS. Avoid treating raw credential object fields as proof of usable credentials; a real API call is more reliable.

```python
import boto3

session = boto3.Session(profile_name="dev", region_name="us-west-2")
sts = session.client("sts")

response = sts.get_caller_identity()
print(response["Arn"])
print(response["Account"])
```

If you need to inspect resolved credentials for debugging, force resolution carefully:

```python
creds = session.get_credentials()
frozen = creds.get_frozen_credentials() if creds else None
print(frozen.access_key[:4] + "..." if frozen else "no credentials")
```

**Node.js example**

```javascript
import { STSClient, GetCallerIdentityCommand } from "@aws-sdk/client-sts";

const client = new STSClient({ region: "us-west-2" });
const response = await client.send(new GetCallerIdentityCommand({}));

console.log(response.Arn);
console.log(response.Account);
```

**Java example**

A minimal Java smoke test can use the AWS SDK for Java v2 and STS.

```java
import software.amazon.awssdk.regions.Region;
import software.amazon.awssdk.services.sts.StsClient;
import software.amazon.awssdk.services.sts.model.GetCallerIdentityResponse;

public class SmokeTest {
    public static void main(String[] args) {
        try (StsClient sts = StsClient.builder()
                .region(Region.US_WEST_2)
                .build()) {
            GetCallerIdentityResponse response = sts.getCallerIdentity();
            System.out.println(response.arn());
            System.out.println(response.account());
        }
    }
}
```

A successful SDK smoke test proves more than compilation. It confirms the runtime can load dependencies, resolve credentials, and reach AWS endpoints.

**AWS SAM CLI validation**

AWS SAM CLI is useful for local validation before deployment. Validate both the template and the runtime behavior.

Start with template validation:

```bash
sam validate
```

Then build the application:

```bash
sam build
```

If your function depends on native libraries or a specific runtime image, `sam build` helps catch packaging issues early.

Run a local invocation to validate handler logic and event shape:

```bash
sam local invoke MyFunction --event events/event.json
```

For API-backed functions, start a local API server:

```bash
sam local start-api
```

This is useful when you want to validate routing, request mapping, and handler behavior without deploying to AWS.

Practical SAM smoke-test workflow:

1. Validate the template.
2. Build the application.
3. Invoke the function locally with a known event.
4. Start the local API and send a request.
5. Compare local output with expected AWS behavior.

Example event file:

```json
{
  "resource": "/hello",
  "path": "/hello",
  "httpMethod": "GET",
  "headers": {
    "Content-Type": "application/json"
  },
  "body": null,
  "isBase64Encoded": false
}
```

> `sam local invoke` and `sam local start-api` depend on Docker for many setups. If Docker is not running, local validation may fail even when your template is correct.

**AWS CDK validation**

AWS CDK validation is not just `cdk synth`. You should also confirm that the bootstrap environment exists and that the generated changes match your intent.

Useful commands:

```bash
cdk synth
cdk diff
cdk bootstrap
```

What each command tells you:

- `cdk synth` confirms your app can compile and produce a CloudFormation template.
- `cdk diff` shows the expected changes before deployment.
- `cdk bootstrap` prepares the target account and region with the resources CDK needs for assets and deployments.

A practical CDK validation sequence:

1. Confirm the target account and region.
2. Run `cdk synth` to catch synthesis errors.
3. Run `cdk diff` to inspect the change set.
4. Verify the environment is bootstrapped if assets or modern synthesis are used.
5. Deploy only after the diff matches expectations.

Example bootstrap check:

```bash
cdk bootstrap aws://123456789012/us-west-2
cdk diff
```

If `cdk diff` shows unexpected resource replacement, stop and review logical IDs, environment-specific values, and asset paths before deploying.

**Lambda invoke smoke tests**

A direct Lambda invoke is a strong post-build or post-deploy smoke test because it validates permissions, function configuration, and runtime behavior.

With AWS CLI v2, when you pass literal JSON on the command line, include `--cli-binary-format raw-in-base64-out` for many workflows.

```bash
aws lambda invoke \
  --function-name my-function \
  --cli-binary-format raw-in-base64-out \
  --payload '{"name":"test"}' \
  response.json
cat response.json
```

If the function expects a specific event schema, use a payload that matches the real trigger. For example, an API Gateway event should resemble the API Gateway request shape, not a custom JSON object.

For safer validation:

- use a read-only or no-op code path
- send a unique test marker in the payload
- write results to a temporary file
- inspect the response before repeating the test

**CloudWatch Logs validation**

CloudWatch Logs is often the fastest way to confirm that a deployed function actually ran. The `aws logs tail` command is available in AWS CLI v2.

```bash
aws logs tail /aws/lambda/my-function --follow --region us-west-2
```

Use it to watch invocation output while you trigger the function from another terminal.

If you are using AWS CLI v1 or a constrained environment, confirm command availability before relying on it. Do not assume every CLI installation supports the same subcommands.

**Local emulators and service substitutes**

Local emulators help you validate code paths without incurring AWS costs or waiting for cloud resources. They are especially useful for unit-like integration smoke tests.

Common examples:

- **LocalStack** for multiple AWS service APIs
- **DynamoDB Local** for table operations and query behavior
- **Amazon S3-compatible local endpoints** for object upload/download testing

Example DynamoDB Local endpoint usage:

```bash
aws dynamodb list-tables \
  --endpoint-url http://localhost:8000 \
  --region us-west-2
```

Example Python client against a local endpoint:

```python
import boto3

ddb = boto3.client(
    "dynamodb",
    region_name="us-west-2",
    endpoint_url="http://localhost:8000"
)

print(ddb.list_tables())
```

Use local emulators when you want fast feedback on request formatting, retries, and basic integration logic. Do not treat them as perfect substitutes for AWS-managed behavior.

**Post-deploy smoke tests**

After deployment, run a small set of safe checks that prove the service is reachable and functioning.

Good smoke tests are:

- fast
- repeatable
- low cost
- non-destructive
- easy to clean up

Examples of safe checks:

- invoke a Lambda function with a test payload
- call a health endpoint on API Gateway or an Application Load Balancer
- read a known item from DynamoDB
- list a small, bounded set of resources
- confirm a CloudWatch log entry was written

Example API check:

```bash
curl -sS https://api.example.com/health
```

Example DynamoDB read:

```bash
aws dynamodb get-item \
  --table-name smoke-test-table \
  --key '{"pk":{"S":"healthcheck"}}' \
  --region us-west-2
```

**Making smoke tests idempotent and safe**

Smoke tests should not create permanent clutter or destroy shared data. Design them so repeated runs are harmless.

- Use unique prefixes or timestamps for temporary resources.
- Prefer read-only checks when possible.
- If you must write data, clean it up in the same test flow.
- Avoid deleting shared tables, queues, or buckets.
- Use a dedicated test account or isolated environment for destructive validation.
- Tag temporary resources so they can be found and removed later.

Example pattern for a safe write test:

```bash
TEST_ID="smoke-$(date +%s)"
aws s3 cp test.txt "s3://my-smoke-bucket/$TEST_ID/test.txt"
aws s3 rm "s3://my-smoke-bucket/$TEST_ID/test.txt"
```

**Troubleshooting workflow**

When a smoke test fails, isolate the layer that failed instead of changing multiple variables at once.

1. Confirm the command syntax.
2. Confirm the tool version.
3. Confirm the active profile and region.
4. Confirm credentials with `aws sts get-caller-identity`.
5. Confirm the required permission with IAM simulation.
6. Confirm the runtime or container is available.
7. Retry with a minimal request.

Common failure patterns:

- `AccessDeniedException` usually indicates IAM or resource policy issues.
- `Unable to locate credentials` usually indicates profile, environment variable, or SSO setup problems.
- `The config profile could not be found` usually indicates a profile name mismatch.
- `Cannot connect to the Docker daemon` usually indicates Docker is not running or the user lacks access.
- `ResourceNotFoundException` often means the wrong region or account.

**Practical validation checklist**

Use this as a quick pre-deploy checklist.

- Confirm CLI and SDK versions.
- Confirm shell profile reload after installation.
- Confirm `aws sts get-caller-identity` succeeds.
- Confirm the intended region is active.
- Confirm `sam validate` and `sam build` succeed if using SAM.
- Confirm `cdk synth` and `cdk diff` succeed if using CDK.
- Confirm Docker is running for local SAM or emulator workflows.
- Confirm a minimal SDK call succeeds in your language.
- Confirm a safe post-deploy smoke test exists.
- Confirm cleanup steps for any temporary data.

**Exam focus**

For DVA-C02, expect questions that test your ability to choose the right validation method for the situation.

- Use `sts get-caller-identity` to validate credentials.
- Use `sam build`, `sam local invoke`, and `sam local start-api` to validate serverless code locally.
- Use `cdk synth`, `cdk diff`, and `cdk bootstrap` to validate infrastructure-as-code readiness.
- Use `simulate-principal-policy` to debug authorization.
- Use local emulators when you need fast, isolated feedback.
- Use safe, idempotent smoke tests after deployment.

The key idea is to validate early, validate cheaply, and validate the exact layer that is most likely to fail.
