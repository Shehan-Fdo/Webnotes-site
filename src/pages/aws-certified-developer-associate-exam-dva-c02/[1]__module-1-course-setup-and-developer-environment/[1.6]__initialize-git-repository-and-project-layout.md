## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why repository initialization matters**

A clean Git repository gives you a repeatable starting point for AWS application development. For exam scenarios, you should recognize how repository structure supports local development, CI/CD, infrastructure as code, and safe collaboration.

In practice, you usually initialize a repo when you start a new Lambda, API, or infrastructure project. The layout you choose affects how easily you can package code, manage dependencies, and keep generated artifacts out of version control.

**Core Git initialization workflow**

A new repository typically starts with `git init`, followed by a branch naming decision and an initial commit. If your Git version supports it, you can create the default branch directly with `git init -b main`. If not, initialize normally and rename the branch afterward.

```bash
mkdir my-serverless-app
cd my-serverless-app
git init
git branch -M main
git status
```

- `git init` creates the `.git` metadata directory.
- `git branch -M main` renames the current branch to `main` even if `main` does not yet exist.
- `git init -b main` is convenient, but it is not available in all Git versions.
- `git status` confirms the repository state before you add files.

> Use `git init -b main` only when you know the installed Git version supports it. The safer, broadly compatible pattern is `git init` followed by `git branch -M main`.

**Typical AWS project layout**

For AWS developer projects, the repository should separate application code, infrastructure definitions, tests, and deployment helpers. This makes it easier to reason about what gets deployed and what stays local.

A common layout for a serverless application looks like this:

```text
my-serverless-app/
├── src/
│   └── app.py
├── tests/
│   └── test_app.py
├── template.yaml
├── README.md
├── .gitignore
└── requirements.txt
```

For a CDK-based project, you often see a structure like this:

```text
my-cdk-app/
├── app/
│   └── stack.ts
├── bin/
│   └── my-cdk-app.ts
├── test/
├── cdk.json
├── package.json
├── tsconfig.json
└── README.md
```

- **Source code** belongs in a dedicated directory such as `src/`, `app/`, or `lambda/`.
- **Infrastructure as code** files such as `template.yaml`, `cdk.json`, or stack definitions should be committed.
- **Tests** should live in a separate directory so they are easy to run in CI.
- **Documentation** such as `README.md` helps collaborators understand setup and deployment.
- **Dependency manifests** like `package.json`, `requirements.txt`, or `pom.xml` should be versioned.

**What to commit and what to ignore**

You should commit files that define the project and make builds reproducible. You should ignore files that are generated locally, contain secrets, or can be recreated from source.

A practical `.gitignore` for AWS and Node.js tooling might include:

```gitignore
# Dependencies
node_modules/

# Build output
dist/
build/
cdk.out/
.aws-sam/

# Python cache
__pycache__/
*.pyc
.venv/

# Environment files
.env

# OS/editor files
.DS_Store
.idea/
.vscode/
```

- `node_modules/` is generated by package managers and should not be committed.
- `dist/`, `build/`, `cdk.out/`, and `.aws-sam/` are build artifacts that can be regenerated.
- `.env` often contains local secrets or environment-specific values.
- `package-lock.json` is usually committed in Node.js projects because it improves reproducible installs.
- `requirements.txt`, `package.json`, and lockfiles are part of the project contract and should generally be tracked.

> Do not ignore lockfiles such as `package-lock.json` in typical AWS/SAM/CDK projects unless you have a specific reason. They help keep dependency resolution consistent across environments.

**Branching and first commit**

After initialization, create a first commit that captures the project skeleton. This gives you a stable baseline for future changes and makes it easier to review diffs.

```bash
git add .
git commit -m "Initial project structure"
```

If you are starting a new branch-based workflow, you may also create a feature branch after the initial commit:

```bash
git checkout -b feature/add-lambda-handler
```

- The first commit should include only the intended project scaffold.
- Feature branches help isolate work before merging into `main`.
- A small, clean initial commit makes later history easier to understand.

**Connecting to a remote repository**

Once the local repository is ready, add a remote and push the initial branch.

```bash
git remote add origin https://github.com/example/my-serverless-app.git
git push -u origin main
```

- `git remote add origin ...` registers the remote repository.
- `git push -u origin main` pushes the branch and sets upstream tracking.
- If the remote already contains commits, you usually reconcile history first by cloning, or by fetching and integrating changes before pushing.
- `git pull` fetches remote changes and then integrates them, usually by merge unless your configuration or command options request a rebase.

**Scenario: starting a Lambda project from scratch**

Suppose you are building an AWS Lambda function with SAM. You want a repository that supports local testing, deployment packaging, and team collaboration.

A sensible starting point is:

```text
lambda-orders/
├── src/
│   └── handler.py
├── tests/
│   └── test_handler.py
├── template.yaml
├── requirements.txt
├── .gitignore
└── README.md
```

You would then:

1. Initialize Git in the project directory.
2. Create or rename the default branch to `main`.
3. Add the SAM template and application source.
4. Add a `.gitignore` that excludes `.aws-sam/`, virtual environments, and caches.
5. Commit the scaffold.
6. Push to a remote repository for collaboration and CI/CD.

This layout works well because the SAM template defines deployment behavior, the source directory contains the Lambda handler, and the tests can run independently before deployment.

**Common mistakes and troubleshooting**

Repository setup issues often come from mixing generated files with source files or from starting with an unclear layout.

- Committing build artifacts makes diffs noisy and can hide real changes.
- Ignoring lockfiles can cause dependency drift between local machines and CI.
- Placing application code at the repository root can make larger projects harder to navigate.
- Forgetting to set the upstream branch can make the first push less convenient.
- Assuming `git pull` always merges can lead to surprises if rebase is configured.

If a remote already has commits, do not force a simplistic workflow. You can:

- clone the remote and work from that baseline,
- fetch and merge or rebase local work,
- or integrate the remote history before your first push.

**Exam-focused takeaways**

For DVA-C02, focus on the relationship between Git setup and AWS delivery workflows.

- Use `git init` to start version control in a new project directory.
- Use `git branch -M main` when you need a compatible way to name the default branch `main`.
- Commit source, tests, templates, and dependency manifests.
- Ignore generated artifacts, caches, and local environment files.
- Keep lockfiles when they improve reproducibility.
- Structure the repository so deployment tooling and application code are easy to separate.

A well-organized repository reduces deployment mistakes, supports automation, and makes it easier to reason about changes during development and code review.
