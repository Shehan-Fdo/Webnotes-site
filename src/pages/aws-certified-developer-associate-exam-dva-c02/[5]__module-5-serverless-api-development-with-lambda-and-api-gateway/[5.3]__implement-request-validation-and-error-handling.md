## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Request validation and error handling overview**

Request validation is the process of rejecting bad input before it reaches business logic or downstream services. In AWS application designs, you usually validate at multiple layers:

- **Edge validation** in Amazon API Gateway to catch malformed requests early.
- **Application validation** in AWS Lambda or containers to enforce business rules.
- **Downstream validation** in services such as DynamoDB, S3, or Step Functions when the request depends on resource state.

This layered approach reduces cost, improves latency, and produces clearer client errors. It also helps you separate syntax problems from business-rule failures, which makes troubleshooting easier.

A good error-handling strategy returns consistent responses, logs enough context for operators, and avoids leaking internal implementation details to callers.

**Where validation belongs**

Use the earliest practical layer for each rule:

- **API shape checks** belong in API Gateway when you can validate required parameters, headers, and payload structure.
- **Business rules** belong in the application layer because they often depend on data lookups or domain logic.
- **Authorization-related checks** should be handled by IAM, Lambda authorizers, Cognito, or resource policies rather than custom validation code.
- **Idempotency checks** belong in the application or persistence layer when repeated requests must not create duplicate side effects.

A useful mental model is:

- Reject obvious bad input as soon as possible.
- Transform and normalize valid input before business logic.
- Map internal failures to stable external error responses.

**API Gateway request validation**

API Gateway can validate request parameters and body content before invoking a backend. This is useful when you want to reduce Lambda invocations for invalid requests.

Common validation capabilities include:

- Required query string parameters.
- Required path parameters.
- Required headers.
- Request body validation against a JSON Schema model.

For REST APIs, you can attach a request validator and models to methods. A typical flow is:

1. Define a model for the expected JSON payload.
2. Attach the model to the method request.
3. Enable request validation for body, parameters, or both.
4. Return a `400 Bad Request` when validation fails.

Example OpenAPI fragment for a REST API model:

```yaml
openapi: 3.0.1
info:
  title: Orders API
  version: 1.0.0
paths:
  /orders:
    post:
      x-amazon-apigateway-request-validator: Validate body, query string parameters, and headers
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/OrderRequest'
      responses:
        '200':
          description: OK
components:
  schemas:
    OrderRequest:
      type: object
      required:
        - customerId
        - items
      properties:
        customerId:
          type: string
        items:
          type: array
          minItems: 1
          items:
            type: object
            required:
              - sku
              - quantity
            properties:
              sku:
                type: string
              quantity:
                type: integer
                minimum: 1
```

For a deployed REST API, you can also use the AWS CLI to inspect or manage validators:

```bash
aws apigateway get-request-validators --rest-api-id a1b2c3d4
aws apigateway create-request-validator \
  --rest-api-id a1b2c3d4 \
  --name "ValidateBodyAndParams" \
  --validate-request-body \
  --validate-request-parameters
```

Validation at the gateway is especially helpful when:

- You want to protect Lambda concurrency from malformed traffic.
- You need predictable client errors for contract violations.
- You want to enforce a schema before transformation or integration mapping.

> API Gateway validation is not a substitute for application validation. Always re-check critical business rules in the backend.

**Lambda input validation**

Lambda functions should validate all untrusted input, even if API Gateway already validated the request. Gateway validation can be bypassed by direct invocation, alternate integrations, or future changes in the API contract.

Typical Lambda validation tasks include:

- Checking required fields and types.
- Enforcing length, range, and format constraints.
- Normalizing values such as email addresses or case-sensitive identifiers.
- Rejecting unsupported enum values.
- Verifying that referenced resources exist and are in the expected state.

A simple Node.js example:

```javascript
export const handler = async (event) => {
  const body = typeof event.body === 'string' ? JSON.parse(event.body) : event.body;

  if (!body?.customerId) {
    return {
      statusCode: 400,
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        message: 'customerId is required'
      })
    };
  }

  if (!Array.isArray(body.items) || body.items.length === 0) {
    return {
      statusCode: 400,
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        message: 'items must contain at least one item'
      })
    };
  }

  return {
    statusCode: 200,
    body: JSON.stringify({ ok: true })
  };
};
```

For more complex schemas, use a validation library such as Ajv for JSON Schema or a language-native validator. The goal is to keep validation rules explicit and testable.

Practical validation patterns:

- Validate once at the boundary, then work with typed or normalized data internally.
- Separate syntactic validation from business validation.
- Return field-specific messages when the client can correct the request.
- Log the raw event carefully, redacting secrets and personal data.

**Error categories and response mapping**

A strong error strategy distinguishes between client errors and server errors.

Common categories:

- **400 Bad Request**: malformed JSON, missing required fields, invalid types, or schema violations.
- **401 Unauthorized**: missing or invalid authentication.
- **403 Forbidden**: authenticated but not allowed.
- **404 Not Found**: resource does not exist or is intentionally hidden.
- **409 Conflict**: duplicate resource, version conflict, or state conflict.
- **422 Unprocessable Entity**: syntactically valid request that fails business validation, when your API uses this convention.
- **429 Too Many Requests**: throttling or rate limiting.
- **500 Internal Server Error**: unexpected application failure.
- **502 Bad Gateway**: upstream integration failure or malformed proxy response.
- **503 Service Unavailable**: temporary dependency outage or maintenance.

Use stable response shapes so clients can parse errors consistently. A common pattern is:

```json
{
  "message": "Validation failed",
  "requestId": "abc-123",
  "errors": [
    {
      "field": "items[0].quantity",
      "reason": "must be greater than 0"
    }
  ]
}
```

This structure helps clients display actionable feedback and helps operators correlate failures with logs and traces.

**Handling errors in Lambda**

In Lambda, the way you signal an error depends on the integration type.

For proxy integrations:

- Return a structured HTTP response when you want to control the status code.
- Throw an exception when you want the platform or integration mapping to treat it as a failure.

For non-proxy integrations:

- Use integration response mappings to translate backend errors into HTTP responses.
- Match error patterns carefully so the correct status code is returned.

Example of a controlled error response in Python:

```python
import json

def handler(event, context):
    try:
        body = json.loads(event.get("body") or "{}")
    except json.JSONDecodeError:
        return {
            "statusCode": 400,
            "headers": {"Content-Type": "application/json"},
            "body": json.dumps({"message": "Invalid JSON"})
        }

    if "orderId" not in body:
        return {
            "statusCode": 400,
            "headers": {"Content-Type": "application/json"},
            "body": json.dumps({"message": "orderId is required"})
        }

    try:
        # business logic here
        return {
            "statusCode": 200,
            "body": json.dumps({"status": "accepted"})
        }
    except Exception as exc:
        print(f"Unhandled error: {exc}")
        return {
            "statusCode": 500,
            "headers": {"Content-Type": "application/json"},
            "body": json.dumps({"message": "Internal server error"})
        }
```

Best practices for Lambda error handling:

- Catch expected exceptions and convert them to client-friendly responses.
- Let unexpected exceptions fail fast so alarms and retries can work.
- Include a correlation identifier such as the API Gateway request ID or AWS X-Ray trace ID.
- Avoid returning stack traces to clients.

**API Gateway error mapping**

API Gateway can transform backend errors into client responses. This is important when your backend returns generic failures but you need consistent HTTP semantics.

For REST APIs with Lambda non-proxy integrations, you can use integration response selection patterns. For example, a backend error string might be mapped to `400` or `409`.

Example mapping idea:

- Backend returns `ValidationError: missing customerId`
- API Gateway maps it to `400 Bad Request`
- Client receives a JSON error body with a clear message

For Lambda proxy integrations, the Lambda function itself should return the final HTTP status code and body. In that case, API Gateway mostly passes the response through.

Example proxy response:

```json
{
  "statusCode": 409,
  "headers": {
    "Content-Type": "application/json"
  },
  "body": "{\"message\":\"Order already exists\"}"
}
```

**Validation in event-driven workflows**

When Lambda is triggered by SQS, SNS, EventBridge, or Kinesis, there is no direct HTTP response to the caller. Error handling changes from client response mapping to retry and failure management.

Key patterns:

- Validate message structure before processing.
- Use dead-letter queues (DLQs) for messages that repeatedly fail.
- Make handlers idempotent so retries do not duplicate side effects.
- Distinguish transient failures from permanent validation failures.

For SQS event source mappings:

- A failed batch is retried according to the queue and Lambda retry behavior.
- Partial batch response can reduce reprocessing when only some records fail.
- Invalid messages should usually be sent to a DLQ after retries are exhausted.

For EventBridge:

- Validate event detail fields before acting on them.
- Use schema registry or application-level checks when event contracts matter.
- Treat malformed events as permanent failures unless the producer can be fixed quickly.

For Kinesis:

- A single bad record can block shard progress if not handled carefully.
- Validate each record independently.
- Consider isolating poison-pill records and logging enough context to replay them later.

**Idempotency and duplicate protection**

Retries are normal in distributed systems. Validation and error handling should account for duplicate requests and repeated events.

Use idempotency when:

- Clients may retry after timeouts.
- API Gateway or Lambda retries can occur.
- Event sources deliver at-least-once.

Common approaches:

- Use a client-provided idempotency key.
- Store request fingerprints in DynamoDB with a TTL.
- Check for existing resource state before creating a new one.
- Make writes conditional.

Example DynamoDB conditional write concept:

```bash
aws dynamodb put-item \
  --table-name Orders \
  --item '{
    "orderId": {"S": "o-123"},
    "status": {"S": "PENDING"}
  }' \
  --condition-expression "attribute_not_exists(orderId)"
```

If the item already exists, DynamoDB rejects the write, which you can map to a `409 Conflict`.

**Logging, metrics, and tracing**

Good error handling is incomplete without observability. You need enough telemetry to diagnose failures without exposing sensitive data.

Use:

- **Structured logs** with request IDs, user IDs, and error codes.
- **CloudWatch Metrics** for validation failures, dependency errors, and throttles.
- **AWS X-Ray** or distributed tracing to follow a request across services.
- **CloudWatch Logs Insights** to search for recurring validation patterns.

Example structured log entry:

```json
{
  "level": "ERROR",
  "requestId": "abc-123",
  "errorType": "ValidationError",
  "message": "customerId is required",
  "path": "/orders"
}
```

Useful metrics to track:

- Count of `400` responses.
- Count of schema validation failures.
- Count of downstream timeouts.
- Count of retries and DLQ deliveries.
- Latency percentiles for successful versus failed requests.


A simple SAM template fragment for an API-backed Lambda:

```yaml
Resources:
  OrdersFunction:
    Type: AWS::Serverless::Function
    Properties:
      Handler: app.handler
      Runtime: nodejs20.x
      Events:
        CreateOrder:
```

If you use a custom error envelope, keep it consistent across endpoints:

```json
{
  "errorCode": "VALIDATION_ERROR",
  "message": "items must contain at least one item",
  "requestId": "abc-123"
}
```

**Testing strategies**

Test validation and error handling at multiple levels:

1. Unit test field validation and business rules.
2. Integration test API Gateway request validation and Lambda responses.
3. End-to-end test malformed requests, duplicate requests, and dependency failures.
4. Load test throttling and retry behavior to confirm the system fails predictably.

Focus on edge cases:

- Empty body.
- Invalid JSON.
- Missing required headers.
- Wrong content type.
- Duplicate submission.
- Downstream timeout.
- Partial batch failure in event-driven consumers.

**Exam-focused takeaways**

For DVA-C02, remember these practical distinctions:

- API Gateway can reject invalid requests before Lambda runs.
- Lambda must still validate input because gateway validation is not sufficient by itself.
- Proxy integrations usually require the function to return the final HTTP response.
- Non-proxy integrations often rely on mapping templates and integration responses.
- Event-driven systems need DLQs, retries, and idempotency more than HTTP status codes.
- Clear, consistent error payloads improve client behavior and operational support.

**Common pitfalls**

- Assuming API Gateway validation replaces backend validation.
- Returning generic `500` for all failures, including client mistakes.
- Logging secrets, tokens, or full personal data in error logs.
- Forgetting idempotency when retries can duplicate writes.
- Letting poison-pill messages block event processing without a DLQ.
- Mixing transport errors with business-rule errors in the same response.

**Quick reference**

- Use `400` for invalid input.
- Use `409` for conflicts and duplicates.
- Use `500` for unexpected server failures.
- Validate early, validate again at the boundary, and fail safely.
- Prefer deterministic, structured error responses over ad hoc strings.
