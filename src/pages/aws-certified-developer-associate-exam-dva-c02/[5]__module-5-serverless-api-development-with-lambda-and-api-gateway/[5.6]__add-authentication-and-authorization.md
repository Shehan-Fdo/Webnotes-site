## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Authentication vs. authorization**

Authentication answers *who you are*; authorization answers *what you can do*. In AWS, these concerns often overlap because a request may be authenticated by SigV4, a JWT, or a federated identity, then authorized by IAM, a resource policy, a key policy, or a service-specific policy model.

For exam questions, identify the control plane first:

- **Identity-based policies** attach to IAM users, groups, or roles.
- **Resource-based policies** attach to the resource itself, such as an S3 bucket, Lambda function, or KMS key.
- **Trust policies** define who can assume a role.
- **Session policies** and **permissions boundaries** further constrain permissions.
- **Service control policies** (SCPs) in AWS Organizations set account-level guardrails.

A request is allowed only when the effective authorization result permits it and no explicit deny applies. Do not think of AWS as evaluating a single linear list; multiple policy types combine to produce the final decision.

**IAM policy evaluation model**

AWS authorization is a combination problem, not a simple sequence. The practical exam rule is:

- An **explicit deny** always wins.
- If no deny exists, AWS checks whether any applicable allow remains after all constraints are applied.
- **Identity policies** can allow actions, but permissions boundaries and SCPs can still block them.
- **Resource policies** can grant access directly to a principal, sometimes even when the principal has no identity-based allow for that resource.
- **Session policies** reduce the permissions available to an assumed role session.
- **Permissions boundaries** cap the maximum permissions a principal can ever use.

A useful mental model is:

| Policy type | What it does | Common exam clue |
|---|---|---|
| Identity-based policy | Grants permissions to a principal | “Attach policy to role/user/group” |
| Resource-based policy | Grants access to a resource | “Allow another account or service to access this resource” |
| Trust policy | Allows role assumption | “Who can assume this role?” |
| Permissions boundary | Sets maximum permissions | “Limit what this role/user can ever do” |
| Session policy | Narrows a temporary session | “Assume role but restrict this session” |
| SCP | Sets account guardrails | “Organization-wide deny/allow boundary” |

Important edge cases:

- A **resource-based policy** can authorize a principal even if the principal has no identity-based allow, as long as no other boundary blocks it.
- A **permissions boundary** does not grant permissions; it only limits them.
- An **SCP** does not grant permissions; it only limits what accounts can use.
- A **session policy** cannot elevate permissions beyond the role’s base permissions.
- An **explicit deny** in any applicable policy type overrides allows.

**IAM roles, trust policies, and role assumption**

A role has two distinct parts:

- A **trust policy** that answers who may assume the role.
- Permission policies that answer what the role can do after assumption.

This distinction is a common exam trap. If a Lambda function cannot assume a role, the problem is often the trust policy, not the permission policy.

Example trust policy for Lambda:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example trust policy for a cross-account role assumption:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/DeveloperRole"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Use roles when you want temporary credentials and a clear trust boundary. This is the most common pattern for:

- **Lambda execution roles**
- **EC2 instance profiles**
- **ECS task roles**
- **Cross-account access**
- **Federated access from external identity providers**

**Common AWS service roles**

Different compute services use IAM roles in different ways:

- **Lambda execution role**
  - Used by the function at runtime.
  - Needs permissions for logs, data access, and downstream APIs.
  - Trust policy usually allows `lambda.amazonaws.com`.

- **EC2 instance profile**
  - Delivers temporary credentials to the instance metadata service.
  - The application retrieves credentials from IMDSv2.
  - Trust policy usually allows `ec2.amazonaws.com`.

- **ECS task role**
  - Used by containers to call AWS APIs.
  - Prefer a task role over embedding static credentials in images.
  - Trust policy usually allows `ecs-tasks.amazonaws.com`.

- **Service-linked role**
  - A role that a service uses to call other AWS services on your behalf.
  - AWS may create it automatically for some services or features, but not always at enablement time.
  - It is service-specific and managed by AWS, so you usually do not edit the trust relationship manually.

Example EC2 instance profile trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

**Cross-account access patterns**

Cross-account access is a frequent exam topic. The right pattern depends on whether you need temporary credentials, centralized control, or direct resource sharing.

Use an **assumed role in the target account** when:

- You need strong auditability and centralized control.
- The target account should own the permissions.
- You want temporary credentials with a clear trust policy.
- The access spans multiple services or requires broad permissions.

Use a **resource-based policy** when:

- The service supports it and the access is naturally resource-centric.
- You want to grant access directly to a principal or account.
- You need a simpler pattern for a specific resource, such as an S3 bucket or Lambda function.

Do not treat resource policies as universally “cleaner.” Many real-world cross-account designs still use roles, and some services do not support the same resource-policy model.

Examples of resource-based access:

- **S3 bucket policy** for cross-account read/write to objects.
- **Lambda resource policy** to allow another account or service to invoke a function.
- **KMS key policy** to allow another account to use a key.

Example S3 bucket policy allowing another account to read objects:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadFromPartnerAccount",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::444455556666:root"
      },
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

Example role assumption flow:

1. A principal in Account A calls `sts:AssumeRole` on a role in Account B.
2. The role’s trust policy in Account B allows that principal.
3. The role’s permission policies define what the temporary session can do.
4. Optional session policies and boundaries further restrict the session.

**Amazon Cognito user pools and identity pools**

Cognito separates user authentication from AWS credential vending.

- **User pools** handle user sign-up, sign-in, password policies, MFA, and hosted UI.
- **Identity pools** federate identities from supported providers and exchange them for temporary AWS credentials.

A common exam mistake is to assume a user pool token alone directly becomes AWS credentials. In practice, identity pools are the component that vends AWS credentials after federation.

User pool configuration details that matter:

- **App clients** define how applications authenticate.
- **Client secret** is appropriate for confidential clients, not public browser/mobile apps.
- **Callback URLs** and **logout URLs** must match the application endpoints exactly.
- **Hosted UI** can simplify OAuth 2.0 and OpenID Connect (OIDC) flows.
- **MFA** and password policies are configured in the user pool.
- **Groups** can map users to roles or influence claims used by downstream authorization logic.

Common user pool app client settings:

- Enable the correct OAuth flows for your app type.
- Configure scopes such as `openid`, `email`, and `profile` when using OIDC.
- Set callback and sign-out URLs for the application domain.
- Use the hosted UI when you want a managed sign-in experience.

Example app client conceptually configured for hosted UI:

```text
App client: web-app-client
OAuth flow: Authorization code grant
Scopes: openid email profile
Callback URL: https://app.example.com/callback
Logout URL: https://app.example.com/logout
```

Identity pool use cases:

- Grant temporary AWS credentials to authenticated users.
- Grant limited guest access to unauthenticated users when enabled.
- Map federated identities to IAM roles based on provider, group, or rules.

Common flow:

1. The user authenticates with a supported identity provider.
2. The application receives a token from that provider.
3. The identity pool validates the identity and maps it to an IAM role.
4. AWS STS issues temporary credentials for the mapped role.

**API Gateway authorization**

API Gateway supports multiple authorization models, and the exact setup differs between REST APIs and HTTP APIs.

For **HTTP APIs**:

- Use **JWT authorizers** for OIDC-compatible tokens, including Cognito user pool tokens.
- Use **IAM authorization** when callers sign requests with SigV4.
- Use **Lambda authorizers** when you need custom token validation or custom claims logic.

For **REST APIs**:

- Use **Cognito user pool authorizers** for user pool tokens.
- Use **Lambda authorizers** for custom authorization logic.
- Use **IAM authorization** for SigV4-signed requests.

Practical selection guide:

- Use **JWT authorizers** when the token issuer is trusted and the claims are enough.
- Use **Lambda authorizers** when you need custom logic, external lookups, or nonstandard tokens.
- Use **IAM auth** when AWS principals or temporary credentials should call the API.
- Use **Cognito authorizers** in REST APIs when you want native user pool integration.

Example HTTP API JWT authorizer configuration:

```yaml
Authorizer:
  JwtConfiguration:
    Audience:
      - 1234567890abcdef
    Issuer: https://cognito-idp.us-east-1.amazonaws.com/us-east-1_AbCdEf123
  IdentitySource:
    - $request.header.Authorization
  Name: CognitoJwtAuthorizer
  AuthorizerType: JWT
```

Example REST API Cognito authorizer concept:

```text
Type: Cognito User Pool Authorizer
Identity source: method.request.header.Authorization
User pool: us-east-1_AbCdEf123
```

Example IAM-signed request with the AWS CLI:

```bash
aws apigateway test-invoke-method \
  --rest-api-id a1b2c3d4e5 \
  --resource-id abc123 \
  --http-method GET \
  --path-with-query-string "/items"
```

For real client calls, you typically use SDKs or signed HTTP requests. If you use `curl`, you must sign the request with SigV4 or place it behind a service that handles signing.

**Lambda authorizers**

Lambda authorizers let you implement custom authorization logic before API Gateway forwards the request.

Use them when:

- You need to validate a custom token format.
- You need to call an external authorization service.
- You need to derive permissions from custom claims or tenant metadata.

Be careful about the API type:

- **REST API Lambda authorizers** commonly return an IAM policy document plus context.
- **HTTP API Lambda authorizers** use a different response shape and can return simple responses or IAM policy-like structures depending on the configuration.

Do not assume one response format works for every API Gateway type.

Example REST API authorizer response shape:

```json
{
  "principalId": "user-123",
  "policyDocument": {
    "Version": "2012-10-17",
    "Statement": [
      {
        "Action": "execute-api:Invoke",
        "Effect": "Allow",
        "Resource": "arn:aws:execute-api:us-east-1:123456789012:api-id/stage/GET/items"
      }
    ]
  },
  "context": {
    "tenantId": "t-001"
  }
}
```

**KMS permissions model**

AWS Key Management Service (KMS) authorization is layered and often tested in scenario questions.

The main pieces are:

- **Key policy**: the primary resource policy for the KMS key.
- **IAM policy**: can grant KMS permissions if the key policy allows IAM policy use for the principal or otherwise explicitly permits the principal.
- **Grants**: temporary, scoped permissions often used by AWS services or applications.
- **AWS managed keys** and **customer managed keys**: differ in how much control you have.

A practical rule: KMS access usually requires both the key-side authorization and the caller-side authorization to line up, but the exact mechanism depends on the key policy design.

Default key policy behavior matters:

- A customer managed key starts with a key policy that gives the account root administrative control.
- You can then delegate permissions through IAM or direct key policy statements.
- If the key policy does not allow the relevant principal or IAM delegation path, IAM permissions alone are not enough.

Example key policy statement allowing a role to use a key:

```json
{
  "Sid": "AllowUseOfKeyForAppRole",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:role/AppRole"
  },
  "Action": [
    "kms:Encrypt",
    "kms:Decrypt",
    "kms:GenerateDataKey"
  ],
  "Resource": "*"
}
```

Example IAM policy for a role using KMS:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ef-ghij-klmnopqrstuv"
    }
  ]
}
```

Grants are useful when:

- An AWS service needs short-lived access to a key.
- You want to delegate a narrow set of operations without changing the key policy.
- You need a temporary permission path for a specific grantee.

Example grant creation:

```bash
aws kms create-grant \
  --key-id arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ef-ghij-klmnopqrstuv \
  --grantee-principal arn:aws:iam::123456789012:role/AppRole \
  --operations Encrypt Decrypt GenerateDataKey \
  --name app-role-grant
```

Remember:

- Grants do not replace key policy authorization.
- Grants are scoped and revocable.
- Some services create and manage grants behind the scenes.

**Presigned URLs and temporary access**

Presigned URLs let you delegate time-limited access to an S3 object or another signed AWS request.

Use them when:

- You want a client to upload or download without AWS credentials.
- You need temporary access for a browser or external user.
- You want to avoid exposing long-lived credentials.

For uploads, prefer a **presigned PUT** generated by an SDK or a **presigned POST** for browser-based form uploads. The AWS CLI `s3 presign` command is commonly used for GET-style URLs and is not the best example for uploads.

Example SDK-style presigned PUT in Python:

```python
import boto3

s3 = boto3.client("s3")
url = s3.generate_presigned_url(
    ClientMethod="put_object",
    Params={
        "Bucket": "my-bucket",
        "Key": "uploads/report.pdf",
        "ContentType": "application/pdf"
    },
    ExpiresIn=3600
)

print(url)
```

Example presigned POST:

```python
import boto3

s3 = boto3.client("s3")
post = s3.generate_presigned_post(
    Bucket="my-bucket",
    Key="uploads/report.pdf",
    Fields={"Content-Type": "application/pdf"},
    Conditions=[{"Content-Type": "application/pdf"}],
    ExpiresIn=3600
)

print(post)
```

**Troubleshooting authorization failures**

Error messages often reveal which layer is failing.

- **AccessDenied**
  - Usually means authorization failed.
  - Check identity policies, resource policies, boundaries, SCPs, and key policies.
  - Also check whether the principal is the one you expect.

- **Unauthorized**
  - Often means authentication failed or the token is invalid.
  - Common with API Gateway, Cognito, and custom auth flows.

- **Invalid token**
  - Usually means the JWT is expired, malformed, signed by the wrong issuer, or missing required claims.

- **Signature mismatch**
  - Often means SigV4 signing is incorrect.
  - Check region, service name, canonical request, headers, and clock skew.

- **KMS access denied**
  - Check both the key policy and the caller’s IAM permissions.
  - Confirm the correct key ARN and region.

- **AssumeRole denied**
  - Check the role trust policy first.
  - Then check the caller’s permission to call `sts:AssumeRole`.

Useful CLI checks:

```bash
aws sts get-caller-identity
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject kms:Decrypt
```

**Practical exam patterns**

When you see a scenario, map it to the right control:

- **Need a backend service to call AWS APIs**
  - Use an execution role, task role, or instance profile.

- **Need a user to sign in to a web app**
  - Use Cognito user pools, often with hosted UI and OAuth 2.0.

- **Need temporary AWS credentials for federated users**
  - Use Cognito identity pools or STS federation.

- **Need another account to access a resource**
  - Consider a resource policy or a cross-account role, depending on the service and control requirements.

- **Need to protect a KMS key**
  - Use key policy plus IAM permissions, and grants when appropriate.

- **Need custom API access control**
  - Use API Gateway authorizers, Lambda authorizers, or IAM auth depending on the token and caller type.

**Quick comparison of common authorization choices**

| Scenario | Best fit | Why |
|---|---|---|
| Lambda needs S3 access | Execution role | Temporary credentials, least privilege |
| EC2 app needs DynamoDB access | Instance profile | No static keys on the instance |
| Container needs Secrets Manager access | ECS task role | Per-task permissions |
| External partner needs S3 object access | Bucket policy or cross-account role | Depends on service and control needs |
| Web users sign in to app | Cognito user pool | Managed authentication |
| Web users need AWS credentials | Cognito identity pool | Federated AWS credentials |
| API caller has JWT | JWT authorizer | Validates issuer and claims |
| API caller is an AWS principal | IAM auth | SigV4 and IAM permissions |
| Custom token logic | Lambda authorizer | Flexible authorization logic |

**Key exam takeaways**

- Authentication and authorization are separate, but AWS often combines them in one request flow.
- Trust policies control who can assume a role; permission policies control what the role can do.
- Resource policies can be powerful, but they are not a universal replacement for roles.
- KMS authorization is layered and often requires both key-side and caller-side permissions.
- Cognito user pools authenticate users; identity pools vend AWS credentials.
- API Gateway authorization differs between REST APIs and HTTP APIs.
- Troubleshooting starts by identifying whether the failure is authentication, authorization, or signing.
