## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Configuration injection overview**

Environment variables let you separate deployment-specific values from application code. In AWS, this is a common way to pass non-secret settings such as log levels, feature flags, bucket names, table names, and service endpoints.

Use environment variables when you want:
- Different values per environment, such as `dev`, `test`, and `prod`.
- Simple runtime configuration without rebuilding the application.
- A standard interface that works across Lambda, containers, and local development.

Environment variables are best for small, flat values. They are not a replacement for structured configuration files when you need nested data, validation, or large payloads.

**Why this matters on AWS**

AWS services often expect you to inject configuration at deploy time rather than hard-code it. This supports immutable builds and makes promotion across environments safer.

Common examples include:
- Lambda function settings such as `TABLE_NAME` or `LOG_LEVEL`.
- ECS task definitions with container environment variables.
- EC2 user data or systemd service overrides for startup-time configuration.
- CI/CD pipelines that pass values into build and deploy steps.

> Do not store secrets in plain environment variables unless you understand the security tradeoffs. For sensitive values, prefer AWS Secrets Manager or AWS Systems Manager Parameter Store with encryption.

**Core concepts**

<dl>
<dt><strong>Environment variable</strong></dt>
<dd>A key-value pair available to a process at runtime, such as <code>LOG_LEVEL=info</code>.</dd>
<dt><strong>Configuration injection</strong></dt>
<dd>Supplying settings from the deployment environment instead of embedding them in source code.</dd>
<dt><strong>Immutable artifact</strong></dt>
<dd>A build output that stays the same across environments while configuration changes externally.</dd>
</dl>

Environment variables are usually read once at startup and cached in memory by the application. If you change the variable in the platform, the running process may not see the update until it is restarted or redeployed.

**AWS Lambda environment variables**

Lambda supports environment variables directly in the function configuration. Your code reads them at runtime using the language’s standard process environment APIs.

Typical uses:
- Database table names
- S3 bucket names
- Stage names
- Feature flags
- Third-party API base URLs

Example in Node.js:
```javascript
export const handler = async () => {
  const tableName = process.env.TABLE_NAME;
  const logLevel = process.env.LOG_LEVEL || "info";

  console.log(`Using table ${tableName} with log level ${logLevel}`);
  return { statusCode: 200, body: "ok" };
};
```

Example in Python:
```python
import os

def lambda_handler(event, context):
    table_name = os.environ["TABLE_NAME"]
    log_level = os.getenv("LOG_LEVEL", "info")
    print(f"Using table {table_name} with log level {log_level}")
    return {"statusCode": 200, "body": "ok"}
```

Key Lambda behaviors:
- Variables are configured per function version or alias deployment workflow, depending on how you manage updates.
- Values are limited in size at the function configuration level, so keep them small.
- Lambda encrypts environment variables at rest with AWS-managed encryption, and you can use a customer managed key for additional control.
- Updating environment variables typically requires a new configuration update and may trigger a new deployment state.

**Practical AWS CLI examples**

You can set Lambda environment variables with the AWS CLI:

```bash
aws lambda update-function-configuration \
  --function-name my-function \
  --environment "Variables={TABLE_NAME=Orders,LOG_LEVEL=debug,STAGE=dev}"
```

To inspect them:
```bash
aws lambda get-function-configuration \
  --function-name my-function
```

If you need to update only one variable, fetch the current map first, merge the change, and send the full environment object back. Lambda updates replace the environment variable set rather than patching a single key in place.

**Containers and task definitions**

For Amazon Elastic Container Service (Amazon ECS), environment variables are commonly defined in the task definition. This keeps the container image portable across environments.

Example task definition snippet:
```json
{
  "containerDefinitions": [
    {
      "name": "app",
      "image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:latest",
      "environment": [
        { "name": "LOG_LEVEL", "value": "info" },
        { "name": "TABLE_NAME", "value": "Orders" }
      ]
    }
  ]
}
```

For Docker locally, you can inject variables with:
```bash
docker run -e LOG_LEVEL=debug -e TABLE_NAME=Orders myapp:latest
```

Or use an environment file:
```bash
docker run --env-file .env myapp:latest
```

Container notes:
- Use environment variables for non-sensitive runtime settings.
- Use secrets mechanisms for credentials, tokens, and private keys.
- Keep the image generic so the same artifact can run in multiple environments.

**Secrets versus configuration**

A common exam distinction is whether a value is configuration or secret.

Use environment variables for:
- Non-sensitive settings
- Feature toggles
- Resource identifiers
- Timeouts and tuning values

Use Secrets Manager or Parameter Store for:
- Passwords
- API keys
- OAuth tokens
- Private certificates
- Database credentials

If you inject a secret into an environment variable, the application can read it easily, but the value may be exposed through logs, crash dumps, process inspection, or misconfigured diagnostics. That is why managed secret stores are preferred.

**AWS Systems Manager Parameter Store integration**

Parameter Store is often used to centralize configuration values. You can retrieve parameters at deploy time or runtime and then inject them into environment variables.

Example with the AWS CLI:
```bash
aws ssm get-parameter \
  --name /myapp/dev/table-name \
  --with-decryption
```

A deployment pipeline might:
1. Read a parameter from Parameter Store.
2. Pass the value into a Lambda environment variable or ECS task definition.
3. Deploy the updated service.

This pattern keeps configuration external while still letting the application consume simple environment variables.

**Application startup pattern**

A robust application usually validates required environment variables at startup. This fails fast if deployment configuration is incomplete.

Example in Node.js:
```javascript
const requiredVars = ["TABLE_NAME", "AWS_REGION"];

for (const name of requiredVars) {
  if (!process.env[name]) {
    throw new Error(`Missing required environment variable: ${name}`);
  }
}
```

Example in Python:
```python
import os

required = ["TABLE_NAME", "AWS_REGION"]

missing = [name for name in required if not os.getenv(name)]
if missing:
    raise RuntimeError(f"Missing required environment variables: {', '.join(missing)}")
```

This approach is useful because:
- Misconfiguration is detected immediately.
- Failures happen during startup instead of during traffic.
- You avoid partial runtime behavior caused by missing values.

**Naming and design conventions**

Good environment variable names are:
- Uppercase
- Descriptive
- Stable across environments
- Free of application-specific secrets in the name itself

Examples:
- `TABLE_NAME`
- `QUEUE_URL`
- `LOG_LEVEL`
- `FEATURE_X_ENABLED`
- `API_BASE_URL`

Avoid:
- Ambiguous names like `VALUE1`
- Deeply nested JSON blobs unless absolutely necessary
- Overloading one variable with multiple meanings

If you need structured configuration, consider:
- Multiple variables for separate fields
- A JSON document in Parameter Store or AppConfig
- A config file generated during deployment

**Local development and parity**

You should mirror production configuration patterns locally as closely as possible. This reduces surprises when code moves to AWS.

Common local approaches:
- `.env` files for developer machines
- Shell exports such as `export LOG_LEVEL=debug`
- Docker Compose environment sections
- LocalStack or mocked AWS endpoints for integration testing

Example `.env` file:
```bash
TABLE_NAME=Orders
LOG_LEVEL=debug
STAGE=local
```

Example shell usage:
```bash
set -a
source .env
set +a
python app.py
```

Keep `.env` files out of source control if they contain secrets or machine-specific values.

**CI/CD usage**

Pipelines often inject environment variables into build and deploy steps. This is useful for:
- Selecting the target account or region
- Choosing the deployment stage
- Passing artifact metadata
- Toggling test behavior

Example GitHub Actions step:
```yaml
- name: Deploy
  env:
    AWS_REGION: us-east-1
    STAGE: prod
    TABLE_NAME: Orders
  run: |
    aws lambda update-function-configuration \
      --function-name my-function \
      --environment "Variables={TABLE_NAME=$TABLE_NAME,STAGE=$STAGE}"
```

In AWS CodeBuild, environment variables can be defined in the build project or injected from the buildspec and parameter sources.

**Security and operational considerations**

Environment variables are convenient, but they have limits.

Important considerations:
- They are visible to the process and often to child processes.
- They may appear in diagnostic output if you print the full environment.
- They are not ideal for large or complex configuration payloads.
- They can be changed only through deployment or process restart in many systems.

Best practices:
- Store only non-sensitive values in plain environment variables.
- Use encryption and managed secret stores for sensitive data.
- Validate required values at startup.
- Document each variable and its expected format.
- Keep names consistent across environments.

**Troubleshooting checklist**

If your application cannot read a variable:
1. Confirm the variable name matches exactly, including case.
2. Verify the deployment updated the correct function, task, or container.
3. Check whether the process needs a restart to pick up changes.
4. Inspect the runtime logs for startup validation errors.
5. Confirm the value was not overwritten by another deployment layer.

Common mistakes:
- Using lowercase names in one place and uppercase in another.
- Updating the wrong environment or alias.
- Storing a secret in a variable and then exposing it in logs.
- Assuming a running process automatically refreshes configuration.

**Exam focus**

For DVA-C02, remember the practical distinction:
- Environment variables are for simple runtime configuration.
- Secrets Manager and Parameter Store are for sensitive or centrally managed values.
- Lambda and ECS both support environment-variable injection.
- Good cloud-native design keeps code and configuration separate.

A strong answer usually emphasizes portability, deployment-time injection, and secure handling of secrets.
