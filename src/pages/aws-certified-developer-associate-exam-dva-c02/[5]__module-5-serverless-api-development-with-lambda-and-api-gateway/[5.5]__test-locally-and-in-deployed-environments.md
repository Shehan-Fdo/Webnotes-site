## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Why this topic matters**

Testing strategy is a recurring exam theme because AWS applications often behave differently in local emulation than they do in real AWS environments. For DVA-C02, you need to know when to use local tools for fast feedback, when to deploy to AWS for realistic validation, and how to test asynchronous and event-driven systems safely.

Local testing is best for developer speed and repeatability. Deployed testing is best for validating IAM (Identity and Access Management), networking, managed service integrations, event delivery semantics, and production-like behavior that local emulators cannot fully reproduce.

**Local testing with AWS SAM and related tools**

AWS SAM (Serverless Application Model) is the primary local-development tool for Lambda-centric serverless applications. It lets you build, invoke, and emulate API Gateway, Lambda, and some event sources on your workstation.

Common SAM commands:

```bash
sam build
sam local invoke MyFunction -e events/event.json
sam local start-api
sam local start-lambda
sam validate
```

- `sam build` packages dependencies and prepares artifacts for local or deployed use.
- `sam local invoke` runs a Lambda function with a sample event payload.
- `sam local start-api` emulates API Gateway routes backed by Lambda.
- `sam local start-lambda` exposes a local Lambda endpoint for integration-style testing.
- `sam validate` checks template structure before deployment.

A typical SAM template for local and deployed testing might look like this:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Transform: AWS::Serverless-2016-10-31
Resources:
  OrdersFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: src/
      Handler: app.handler
      Runtime: nodejs20.x
      Events:
        ApiEvent:
          Type: Api
          Properties:
            Path: /orders
            Method: post
```

Local testing is useful when you want to:

- Verify handler logic with known inputs.
- Debug serialization, validation, and error handling quickly.
- Run unit tests and integration tests without waiting for cloud deployment.
- Reproduce common API request shapes or event payloads.

> Local emulation is not a full substitute for AWS. It can hide IAM permission issues, service integration differences, and event delivery behavior that only appear in deployed environments.

**CloudFormation, CDK, and deployment-oriented validation**

Infrastructure as Code (IaC) testing is part of the workflow because your application behavior depends on the resources you define. CloudFormation and the AWS Cloud Development Kit (CDK) support validation before and during deployment.

Useful CloudFormation and CDK checks:

```bash
aws cloudformation validate-template --template-body file://template.yaml
aws cloudformation create-change-set \
  --stack-name my-stack \
  --change-set-name test-change-set \
  --template-body file://template.yaml \
  --capabilities CAPABILITY_IAM
aws cloudformation describe-change-set \
  --stack-name my-stack \
  --change-set-name test-change-set
cdk synth
cdk diff
cdk deploy
```

- `validate-template` checks syntax and basic template structure.
- Change sets let you review proposed resource changes before execution.
- `cdk synth` generates the CloudFormation template from CDK code.
- `cdk diff` shows what will change in AWS before deployment.
- `cdk deploy` applies the stack and is often paired with post-deploy smoke tests.

For exam purposes, change sets are important when you need to inspect the impact of a deployment before it touches live resources. This is especially useful for IAM changes, replacement-sensitive resources, and production stacks.

**SAM Accelerate and faster deployed feedback loops**

SAM Accelerate (`sam sync`) reduces the time between code changes and deployed validation. It is useful when local emulation is insufficient but full redeploys are too slow.

```bash
sam sync --stack-name my-stack --watch
sam sync --stack-name my-stack --code
```

- `--watch` continuously syncs changes during development.
- `--code` updates function code quickly without a full stack redeploy.
- It is helpful for iterative testing against real AWS services.

Use SAM Accelerate when you need to test:

- Real IAM permissions.
- Real API Gateway, EventBridge, S3, SQS, or DynamoDB integrations.
- Environment-specific behavior such as VPC access, KMS (Key Management Service) encryption, or resource policies.

**When local emulation falls short**

Some AWS services and event patterns are difficult to reproduce accurately on a laptop. The exam often expects you to recognize these limitations.

Key service-specific limitations:

- **S3 event notifications**
  - Local emulators may not fully reproduce bucket notification timing, prefix/suffix filtering, or duplicate delivery behavior.
  - Real S3 events are asynchronous and can arrive more than once.
  - Test object-created workflows in AWS when you need confidence in actual event wiring.

- **EventBridge**
  - Event pattern matching, bus permissions, and cross-account delivery are best validated in AWS.
  - Local testing can confirm payload shape, but not full routing behavior or resource policy enforcement.
  - Use deployed tests for rules that depend on exact event patterns or targets.

- **SQS/Lambda event source mappings**
  - Local emulation does not fully model polling, batch size, visibility timeout, partial batch failure, or redrive behavior.
  - In AWS, Lambda polls SQS asynchronously and retries according to queue and function settings.
  - Validate dead-letter queue (DLQ) behavior, batch failure handling, and message reprocessing in a deployed environment.

- **SNS, Step Functions, and asynchronous chains**
  - Fan-out, retries, and service-to-service permissions are more realistic in AWS.
  - Local tools may not capture timing, concurrency, or retry semantics accurately.

- **DynamoDB streams**
  - Stream ordering, shard behavior, and retry semantics are better validated in AWS.
  - Local tests can verify handler logic, but not the full stream delivery model.

**Testing asynchronous workflows in deployed environments**

Asynchronous systems need tests that account for retries, duplicates, and eventual consistency. A successful request does not always mean downstream work is complete.

Important concepts:

- **Idempotency**
  - Design handlers so repeated events do not create duplicate side effects.
  - Use idempotency keys, conditional writes, or deduplication tables when appropriate.
  - This matters because many AWS event sources can deliver events more than once.

- **Duplicate event handling**
  - S3, EventBridge, SNS, and stream-based workflows can produce repeated deliveries.
  - Your code should tolerate reprocessing without corrupting state.

- **Eventual consistency**
  - Some AWS services do not reflect changes immediately.
  - Tests should poll for expected state rather than assume immediate availability.

- **Retries and backoff**
  - AWS services may retry failed deliveries automatically.
  - Your tests should verify that retry logic does not create duplicate records or repeated side effects.

A practical deployed test pattern for an async workflow:

1. Deploy a test stack with isolated resources.
2. Trigger the workflow with a known input.
3. Wait for downstream completion using polling or a callback mechanism.
4. Verify final state in DynamoDB, S3, CloudWatch Logs, or another durable store.
5. Confirm duplicate delivery does not change the final result.

Example of a simple idempotent write pattern in Lambda:

```python
import boto3
from botocore.exceptions import ClientError

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("Orders")

def handler(event, context):
    order_id = event["orderId"]
    try:
        table.put_item(
            Item={"orderId": order_id, "status": "PROCESSED"},
            ConditionExpression="attribute_not_exists(orderId)"
        )
    except ClientError as e:
        if e.response["Error"]["Code"] == "ConditionalCheckFailedException":
            return {"status": "duplicate_ignored"}
        raise
    return {"status": "ok"}
```

**Testing APIs and Lambda functions**

For synchronous APIs, you can combine local and deployed tests.

Local checks are good for:

- Request validation.
- Response formatting.
- Error mapping.
- Basic business logic.

Deployed checks are better for:

- IAM authorization and resource-based policies.
- API Gateway integration behavior.
- Lambda environment variables and secrets access.
- VPC connectivity to private resources.

Example smoke test after deployment:

```bash
API_URL="https://abc123.execute-api.us-east-1.amazonaws.com/prod/orders"
curl -s -X POST "$API_URL" \
  -H "Content-Type: application/json" \
  -d '{"orderId":"123","amount":42}'
```

If the endpoint depends on authentication, test the full auth path rather than bypassing it. For example, validate Cognito, IAM authorization, or custom authorizers in AWS because local emulators may not fully reproduce token validation and policy evaluation.

**Testing S3, EventBridge, and SQS workflows**

These services are common in exam scenarios because they connect loosely coupled components.

- **S3**
  - Use deployed tests to confirm bucket notifications, object key filters, and permissions.
  - Verify that the Lambda execution role can read the object and write any downstream result.
  - Expect duplicate or delayed notifications in real systems.

- **EventBridge**
  - Test event patterns against real buses and targets.
  - Confirm the event shape matches the rule pattern exactly.
  - Validate cross-account or cross-region routing in AWS, not locally.

- **SQS**
  - Test queue visibility timeout, DLQ redrive, and batch failure behavior in AWS.
  - Confirm Lambda event source mapping batch size and partial batch response handling.
  - Use message attributes and deduplication features where appropriate.

Example SQS event source mapping in CloudFormation:

```yaml
Resources:
  QueueProcessor:
    Type: AWS::Lambda::Function
    Properties:
      Handler: index.handler
      Runtime: python3.12
      Role: arn:aws:iam::123456789012:role/lambda-exec
      Code:
        ZipFile: |
          def handler(event, context):
              return {"ok": True}

  OrdersQueue:
    Type: AWS::SQS::Queue

  QueueMapping:
    Type: AWS::Lambda::EventSourceMapping
    Properties:
      EventSourceArn: !GetAtt OrdersQueue.Arn
      FunctionName: !Ref QueueProcessor
      BatchSize: 10
      Enabled: true
```

**CI/CD testing with AWS-native services**

AWS-native pipelines often run tests at multiple stages. This is important for exam questions about automated validation and safe deployment.

Common services:

- **AWS CodeBuild**
  - Runs unit tests, linting, packaging, and integration tests.
  - Can deploy ephemeral test stacks and run smoke tests against them.

- **AWS CodePipeline**
  - Orchestrates source, build, test, and deploy stages.
  - Can gate production deployment on successful validation.

A typical pipeline flow:

1. Source change enters CodePipeline.
2. CodeBuild runs unit tests and builds artifacts.
3. A test stack is deployed to a nonproduction environment.
4. Smoke tests and integration tests run against the deployed stack.
5. Approval or automated checks allow promotion to production.

Example `buildspec.yml`:

```yaml
version: 0.2
phases:
  install:
    runtime-versions:
      python: 3.12
  build:
    commands:
      - pip install -r requirements.txt
      - pytest
      - sam build
      - sam deploy --stack-name my-test-stack --no-confirm-changeset --capabilities CAPABILITY_IAM
      - ./scripts/smoke-test.sh
```

This pattern is useful when you need confidence that the deployed environment works end to end, not just that the code compiles.

**Choosing local versus deployed testing**

Use the right environment based on what you are trying to prove.

| Scenario | Best choice | Why |
|---|---|---|
| Pure business logic in a Lambda handler | Local | Fast feedback and easy repeatability |
| API request/response formatting | Local | Simple to emulate and debug |
| IAM permissions to S3, DynamoDB, or KMS | Deployed | Real policy evaluation only happens in AWS |
| S3 event notifications | Deployed | Local emulation is incomplete |
| EventBridge routing and target permissions | Deployed | Real event bus behavior matters |
| SQS/Lambda retry and DLQ behavior | Deployed | Polling and redelivery semantics are AWS-specific |
| Infrastructure changes before release | Change set / `cdk diff` | Safer review before execution |
| Iterative testing against real AWS services | SAM Accelerate | Faster than full redeploys |

A good exam answer often combines both approaches: test logic locally first, then validate integrations in AWS before release.

**Practical deployment and validation workflow**

A reliable workflow for serverless development usually looks like this:

1. Write unit tests for handler logic and edge cases.
2. Run `sam build` and local invocation tests with sample events.
3. Use `sam validate`, `cdk synth`, or `validate-template` to catch template issues.
4. Review a CloudFormation change set or `cdk diff`.
5. Deploy to a test stack or sandbox account.
6. Run smoke tests against real endpoints and event sources.
7. Verify logs, metrics, and downstream state.
8. Test failure paths, retries, and duplicate event handling.
9. Promote only after the deployed behavior matches expectations.

**Observability during testing**

Testing is stronger when you inspect runtime evidence, not just return values.

Useful tools:

- **CloudWatch Logs** for function output and stack diagnostics.
- **CloudWatch Metrics** for errors, throttles, and duration.
- **X-Ray** for tracing distributed requests and latency bottlenecks.
- **AWS CLI** for querying deployed state and confirming side effects.

Example log inspection:

```bash
aws logs tail /aws/lambda/MyFunction --follow
```

Example state verification:

```bash
aws dynamodb get-item \
  --table-name Orders \
  --key '{"orderId":{"S":"123"}}'
```

**Exam-oriented takeaways**

- Use local tools for fast feedback on code, templates, and simple request handling.
- Use deployed tests for IAM, networking, managed service integrations, and event-driven behavior.
- Expect local emulation gaps for S3 notifications, EventBridge routing, and SQS/Lambda event source mappings.
- Design asynchronous workflows for idempotency, retries, duplicates, and eventual consistency.
- Use change sets, `cdk diff`, and SAM Accelerate to reduce deployment risk and speed up validation.
- In CI/CD, use CodeBuild and CodePipeline to automate unit tests, integration tests, and smoke tests before promotion.
