## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**API Gateway and Lambda integration model**

API Gateway acts as the front door for HTTP requests and can invoke Lambda as a backend without exposing Lambda directly to clients. In exam terms, you need to know the difference between the API layer, the integration layer, and the Lambda permission that allows invocation.

A typical flow is:

- Client sends an HTTP request to API Gateway.
- API Gateway authenticates and authorizes the request if configured.
- API Gateway transforms or forwards the request to Lambda.
- Lambda returns a response that API Gateway maps back to the client.

The key idea is that authorization for the client and permission for API Gateway to invoke Lambda are separate concerns. A client may be allowed to call the API, but API Gateway still needs an IAM resource policy permission on the Lambda function to invoke it.

**Integration types you should know**

API Gateway supports two common Lambda integration styles:

- **Lambda proxy integration**
  - API Gateway forwards the request to Lambda with minimal transformation.
  - Lambda receives request context, headers, query strings, path parameters, and body in a standard event shape.
  - Lambda must return a correctly formatted response object.
- **Non-proxy integration**
  - API Gateway uses mapping templates to transform the request and response.
  - You control the payload shape more explicitly.
  - This is more flexible for legacy backends, but more complex to maintain.

For exam prep, proxy integration is usually the default mental model because it is simpler and faster to configure. Non-proxy integration matters when you need custom request/response shaping, selective header mapping, or legacy compatibility.

**REST API vs HTTP API**

API Gateway has two major API types, and the Lambda integration details differ:

- **REST APIs**
  - Support Lambda proxy and non-proxy integrations.
  - Support stage-level caching when explicitly enabled.
  - Use REST-style resource/method configuration.
  - Use binary media types configuration for binary payload handling.
- **HTTP APIs**
  - Are simpler and lower cost for many use cases.
  - Support Lambda proxy-style integrations with route-based configuration.
  - Use API mappings and route settings rather than REST-style base path mappings and method resources.
  - Handle binary payloads differently from REST APIs; do not assume REST API binary media type configuration applies.

Use REST APIs when you need advanced API Gateway features such as request validation, usage plans, API keys, or more mature mapping behavior. Use HTTP APIs when you want a simpler, cheaper, lower-latency API surface and your requirements fit the newer model.

**Lambda proxy request and response shape**

With Lambda proxy integration, API Gateway sends a structured event to Lambda. The exact shape varies slightly between REST API and HTTP API, but the core idea is the same: Lambda receives request metadata and the body, and returns a structured response.

Example REST API proxy event fields:

```json
{
  "resource": "/orders",
  "path": "/orders",
  "httpMethod": "POST",
  "headers": {
    "Content-Type": "application/json"
  },
  "queryStringParameters": {
    "debug": "true"
  },
  "pathParameters": {
    "orderId": "123"
  },
  "body": "{\"item\":\"book\"}",
  "isBase64Encoded": false
}
```

A valid proxy response must include status and body fields:

```json
{
  "statusCode": 200,
  "headers": {
    "Content-Type": "application/json"
  },
  "body": "{\"message\":\"ok\"}",
  "isBase64Encoded": false
}
```

If the response is malformed, API Gateway commonly returns a `502 Bad Gateway`. This is a frequent exam point: the Lambda function may run successfully, but if the response format is invalid, the client still sees a gateway error.

**Error handling and status code mapping**

Error handling differs between proxy and non-proxy integrations.

- **Proxy integration**
  - Lambda must return a valid proxy response object.
  - If Lambda throws an exception or returns an invalid payload, API Gateway often surfaces a `502`.
  - To return application errors cleanly, you usually catch exceptions in Lambda and return a structured response with the desired `statusCode`.
- **Non-proxy integration**
  - API Gateway can map backend errors using integration response selection patterns.
  - You can map specific Lambda error strings or regex patterns to different HTTP status codes.
  - This is useful when you want API Gateway to translate backend failures into client-friendly responses.

A practical Lambda proxy error pattern in Python:

```python
import json

def lambda_handler(event, context):
    try:
        if event.get("queryStringParameters", {}).get("fail") == "true":
            return {
                "statusCode": 400,
                "headers": {"Content-Type": "application/json"},
                "body": json.dumps({"message": "Bad request"})
            }

        return {
            "statusCode": 200,
            "headers": {"Content-Type": "application/json"},
            "body": json.dumps({"message": "Success"})
        }
    except Exception as e:
        return {
            "statusCode": 500,
            "headers": {"Content-Type": "application/json"},
            "body": json.dumps({"message": "Internal error", "detail": str(e)})
        }
```

For non-proxy integrations, you may use integration response mappings to convert backend errors into HTTP responses. In exam scenarios, remember that API Gateway does not automatically infer your desired client status code unless you explicitly design for it.

**Lambda permission for API Gateway**

API Gateway cannot invoke Lambda unless the function policy allows it. This permission is separate from client authorization.

You typically add a resource-based permission with `lambda:AddPermission` or the equivalent CloudFormation/SAM/CDK configuration.

Example CLI permission for a REST API method:

```bash
aws lambda add-permission \
  --function-name my-function \
  --statement-id apigw-rest-invoke \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn "arn:aws:execute-api:us-east-1:123456789012:abc123/prod/GET/orders"
```

Example CLI permission for an HTTP API route:

```bash
aws lambda add-permission \
  --function-name my-function \
  --statement-id apigw-http-invoke \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn "arn:aws:execute-api:us-east-1:123456789012:abc123/*/*/orders"
```

Important permission nuances:

- **Source ARN scope** should match the API, stage, method, and resource pattern you intend to allow.
- For REST APIs, the ARN commonly includes the stage and method.
- For HTTP APIs, route patterns are different and often use wildcard matching for stage and route.
- If the permission is too narrow, API Gateway receives `500` or `403`-style invocation failures depending on the failure point.

**Request and response mapping templates**

Non-proxy integrations use Velocity Template Language (VTL) mapping templates to transform requests and responses.

Use mapping templates when you need to:

- reshape incoming JSON
- inject static values
- rename fields
- pass only selected headers or query parameters
- translate legacy payload formats

Example request mapping template:

```vtl
{
  "orderId": "$input.params('orderId')",
  "customerId": "$input.params('customerId')",
  "body": $input.json('$'),
  "requestId": "$context.requestId"
}
```

Key exam-relevant behaviors:

- Mapping can be based on `Content-Type`.
- If no template matches, API Gateway uses passthrough behavior based on configuration.
- Passthrough can be set to allow, never, or when no templates match.
- Unmapped requests may be rejected or forwarded depending on the passthrough setting.
- VTL is powerful but brittle; proxy integration is often preferred unless you need transformation.

A common pattern is to map only a subset of request data to reduce coupling between client payloads and backend implementation.

**Binary payload handling**

Binary support is a common source of confusion.

For **REST APIs**:

- You configure binary media types on the API.
- API Gateway uses the request `Content-Type` and client `Accept` headers to decide whether to treat payloads as binary.
- Lambda proxy integrations may receive the body as base64-encoded text when the payload is treated as binary.
- Lambda must set `isBase64Encoded` correctly in the response when returning binary data.

For **HTTP APIs**:

- Binary handling is configured differently from REST APIs.
- You still use `isBase64Encoded` in Lambda proxy responses.
- Do not assume REST API binary media type settings apply to HTTP APIs.

Example Lambda response for an image:

```json
{
  "statusCode": 200,
  "headers": {
    "Content-Type": "image/png"
  },
  "body": "iVBORw0KGgoAAAANSUhEUgAA...",
  "isBase64Encoded": true
}
```

Practical guidance:

- Use `isBase64Encoded: true` when the body is base64 text representing binary content.
- Ensure the `Content-Type` matches the actual media type.
- For REST APIs, verify the client `Accept` header if binary responses are not behaving as expected.
- If the binary configuration is wrong, clients may receive corrupted output or plain text instead of the intended file.

**Deployment lifecycle and stages**

API Gateway changes are not always live immediately. You often need to deploy changes to a stage.

For REST APIs:

- You create or update resources, methods, integrations, and mappings.
- You deploy the API to a stage.
- The stage points to a specific deployment snapshot.

For HTTP APIs:

- Route and integration changes are generally applied through the API configuration model, but stage behavior still matters for deployment and routing settings.

Important concepts:

- **Stage**: a named environment such as `dev`, `test`, or `prod`.
- **Deployment**: a snapshot of API configuration.
- **Redeploy**: required after many REST API changes before clients see them.

Stage variables can be used to parameterize integrations. In REST APIs, you can reference stage variables in integration URIs or mapping logic. This is useful when the same API stage should point to different Lambda aliases or backend targets.

Example stage-variable-style Lambda URI pattern:

```text
arn:aws:apigateway:us-east-1:lambda:path/2015-03-31/functions/arn:aws:lambda:us-east-1:123456789012:function:${stageVariables.lambdaAlias}/invocations
```

Use stage variables carefully:

- They help avoid hardcoding environment-specific values.
- They are useful for blue/green or alias-based deployments.
- They can make troubleshooting harder if the effective target is not obvious.

**Custom domains and mappings**

API Gateway can expose a custom domain name instead of the default execute-api hostname.

For REST APIs:

- You use base path mappings to connect a custom domain to an API stage.
- The base path can route traffic to a specific stage.

For HTTP APIs:

- The terminology is API mappings rather than REST-style base path mappings.
- The mapping still connects a custom domain to an API and stage, but the configuration model differs.

Exam takeaway:

- Know that custom domains can map to stages.
- Do not assume the REST API terminology applies unchanged to HTTP APIs.

**Security and authorization**

API Gateway can protect Lambda-backed endpoints with several authorization mechanisms:

- **IAM authorization**
  - Clients sign requests with AWS credentials.
  - Useful for service-to-service access.
- **Cognito user pools**
  - Supported for both REST APIs and HTTP APIs.
  - Useful for user authentication with JWT-based identity.
- **Lambda authorizers**
  - Custom authorization logic in Lambda.
  - Can be used when built-in options are not enough.
- **Resource policies**
  - Restrict who can call the API based on source IP, VPC endpoint, or AWS account conditions.

Important distinction:

- Client authorization controls who can call the API.
- Lambda invocation permission controls whether API Gateway can invoke the function.

A client may authenticate successfully, but if the Lambda permission is missing, the request still fails at the integration layer.

**HTTP API setup example with Lambda**

A minimal HTTP API setup usually includes a route, an integration, and permission for Lambda invocation.

Example AWS CLI flow:

```bash
# Create the HTTP API
aws apigatewayv2 create-api \
  --name orders-api \
  --protocol-type HTTP

# Create the Lambda integration
aws apigatewayv2 create-integration \
  --api-id a1b2c3d4 \
  --integration-type AWS_PROXY \
  --integration-uri arn:aws:lambda:us-east-1:123456789012:function:orders-handler \
  --payload-format-version 2.0

# Create a route
aws apigatewayv2 create-route \
  --api-id a1b2c3d4 \
  --route-key "POST /orders" \
  --target "integrations/xyz789"

# Create a stage
aws apigatewayv2 create-stage \
  --api-id a1b2c3d4 \
  --stage-name prod \
  --auto-deploy
```

Then add Lambda permission:

```bash
aws lambda add-permission \
  --function-name orders-handler \
  --statement-id apigw-http-orders \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn "arn:aws:execute-api:us-east-1:123456789012:a1b2c3d4/*/POST/orders"
```

This example shows the core pieces, but in real deployments you also manage logging, throttling, and authorization.

**REST API setup example with Lambda**

A REST API requires resources, methods, integrations, deployment, and permission.

Example CLI sequence:

```bash
# Create the REST API
aws apigateway create-rest-api --name orders-rest-api

# Get the root resource ID, then create a resource
aws apigateway create-resource \
  --rest-api-id abc123 \
  --parent-id root-resource-id \
  --path-part orders

# Add a method
aws apigateway put-method \
  --rest-api-id abc123 \
  --resource-id resource-id \
  --http-method GET \
  --authorization-type NONE

# Attach Lambda integration
aws apigateway put-integration \
  --rest-api-id abc123 \
  --resource-id resource-id \
  --http-method GET \
  --type AWS_PROXY \
  --integration-http-method POST \
  --uri "arn:aws:apigateway:us-east-1:lambda:path/2015-03-31/functions/arn:aws:lambda:us-east-1:123456789012:function:orders-handler/invocations"

# Deploy the API
aws apigateway create-deployment \
  --rest-api-id abc123 \
  --stage-name prod
```

Then add permission:

```bash
aws lambda add-permission \
  --function-name orders-handler \
  --statement-id apigw-rest-orders \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn "arn:aws:execute-api:us-east-1:123456789012:abc123/prod/GET/orders"
```

Note the distinction between:

- the client-facing method, such as `GET`
- the integration HTTP method, which is often `POST` for Lambda invocations

That difference is easy to miss on the exam.

**CloudFormation example**

A deployable REST API stack needs more than just the API and Lambda function. You typically need the method, integration, permission, deployment, and stage.

```yaml
Resources:
  OrdersFunction:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: orders-handler
      Runtime: python3.12
      Handler: index.lambda_handler
      Role: arn:aws:iam::123456789012:role/lambda-exec-role
      Code:
        ZipFile: |
          def lambda_handler(event, context):
              return {
                  "statusCode": 200,
                  "headers": {"Content-Type": "application/json"},
                  "body": "{\"message\":\"ok\"}",
                  "isBase64Encoded": False
              }

  OrdersApi:
    Type: AWS::ApiGateway::RestApi
    Properties:
      Name: orders-api

  OrdersResource:
    Type: AWS::ApiGateway::Resource
    Properties:
      RestApiId: !Ref OrdersApi
      ParentId: !GetAtt OrdersApi.RootResourceId
      PathPart: orders

  OrdersMethod:
    Type: AWS::ApiGateway::Method
    Properties:
      RestApiId: !Ref OrdersApi
      ResourceId: !Ref OrdersResource
      HttpMethod: GET
      AuthorizationType: NONE
      Integration:
        Type: AWS_PROXY
        IntegrationHttpMethod: POST
        Uri: !Sub >
          arn:aws:apigateway:${AWS::Region}:lambda:path/2015-03-31/functions/${OrdersFunction.Arn}/invocations

  OrdersPermission:
    Type: AWS::Lambda::Permission
    Properties:
      FunctionName: !Ref OrdersFunction
      Action: lambda:InvokeFunction
      Principal: apigateway.amazonaws.com
      SourceArn: !Sub >
        arn:aws:execute-api:${AWS::Region}:${AWS::AccountId}:${OrdersApi}/*/GET/orders

  OrdersDeployment:
    Type: AWS::ApiGateway::Deployment
    DependsOn: OrdersMethod
    Properties:
      RestApiId: !Ref OrdersApi
      StageName: prod
```

This pattern is more realistic for exam study because it shows the full invocation chain.

**Troubleshooting checklist**

When an API Gateway to Lambda integration fails, check the following:

- **Permission**
  - Does Lambda allow `apigateway.amazonaws.com` to invoke it?
  - Does the `SourceArn` match the API, stage, method, and route?
- **Integration type**
  - Are you using proxy or non-proxy intentionally?
  - Does the Lambda response match the expected format?
- **Deployment**
  - Did you redeploy the REST API after changing methods or integrations?
- **Mapping**
  - Do request or response templates match the `Content-Type`?
  - Is passthrough behavior blocking unmapped requests?
- **Binary handling**
  - Is the API configured for binary payloads correctly?
  - Is `isBase64Encoded` set appropriately?
- **Authorization**
  - Is the client authorized to call the API?
  - Is that separate from Lambda invocation permission?
- **Timeouts**
  - Is the Lambda function timeout shorter than the API Gateway integration timeout?
  - If the client times out, Lambda may still continue running until its own timeout or completion, but the client will not wait indefinitely.

**Timeout behavior**

API Gateway has an integration timeout, and Lambda has its own function timeout. These are related but not identical.

- If API Gateway times out first, the client receives a timeout error.
- Lambda may continue running after the client disconnects if the invocation is still active and Lambda has not timed out.
- If Lambda itself times out, the invocation fails regardless of API Gateway settings.

For exam purposes, remember that the client request lifecycle and the backend execution lifecycle are not the same thing.

**Common exam traps**

- Assuming API Gateway caching is automatic for Lambda integrations.
- Assuming binary payloads always use the same behavior across REST APIs and HTTP APIs.
- Forgetting that a valid Lambda response format is required for proxy integrations.
- Confusing client authorization with Lambda invocation permission.
- Forgetting to redeploy REST API changes.
- Using a too-broad or too-narrow `SourceArn`.
- Treating custom domain mappings as identical across REST APIs and HTTP APIs.
- Assuming Cognito user pools are only for REST APIs.

**Quick comparison**

| Topic | REST API | HTTP API |
|---|---|---|
| Lambda integration style | Proxy and non-proxy | Proxy-style integration |
| Deployment model | Explicit deployments to stages | Simpler route-based model with stages |
| Binary handling | Binary media types | Different configuration model |
| Custom domain mapping | Base path mapping | API mapping |
| Advanced mapping features | Strong support | More limited |
| Typical use case | Feature-rich APIs | Simpler, lower-cost APIs |

**Practical design guidance**

Choose proxy integration when you want to move quickly and keep Lambda responsible for response formatting. Choose non-proxy integration when you need API Gateway to transform payloads or normalize legacy backend contracts.

Use REST APIs when you need advanced API Gateway features or legacy compatibility. Use HTTP APIs when you want a simpler architecture and your requirements fit the newer feature set.

For production systems, combine:

- least-privilege Lambda permissions
- explicit authorization
- structured error handling
- stage-based deployments
- logging and tracing

That combination gives you a secure and debuggable API surface.
