## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Lambda handlers for API requests**

When you design an AWS Lambda handler behind Amazon API Gateway, you are translating an HTTP request into application logic and then translating the result back into an HTTP response. For the exam, focus on the contract between API Gateway and Lambda: the event payload, the response shape, status codes, headers, query strings, path parameters, and authorizer context.

A good handler is predictable, stateless, and defensive. It should validate input, normalize headers, return the correct status code, and avoid leaking implementation details. In practice, you often build a small routing layer inside the function or use one Lambda per route when the API is simple.

- **API Gateway** is the front door that maps HTTP requests to Lambda events.
- **Lambda proxy integration** passes the request details to your function with minimal mapping.
- **Event version** matters because REST APIs and HTTP APIs use different payload formats.
- **Response shape** must match the integration type or API Gateway returns an error.

**REST API and HTTP API event models**

API Gateway has two relevant Lambda proxy event formats for the exam: the older REST API payload format and the newer HTTP API payload format. The exact shape depends on the API type and the configured payload version, not just the word “Lambda” in general.

For REST APIs, you commonly see the v1.0 event shape. For HTTP APIs, you commonly see the v2.0 event shape. The practical difference is that v2.0 simplifies the request object and changes how cookies and some request metadata are represented.

<dl>
<dt><strong>REST API v1</strong></dt>
<dd>Includes fields such as <code>httpMethod</code>, <code>path</code>, <code>headers</code>, <code>multiValueHeaders</code>, <code>queryStringParameters</code>, and <code>requestContext</code>.</dd>
<dt><strong>HTTP API v2</strong></dt>
<dd>Includes fields such as <code>version</code>, <code>routeKey</code>, <code>rawPath</code>, <code>rawQueryString</code>, <code>cookies</code>, <code>headers</code>, and <code>requestContext</code>.</dd>
</dl>

A common exam trap is assuming the API type alone determines everything. In reality, the integration payload version and authorizer configuration also affect the event structure you receive.

```json
{
  "version": "2.0",
  "routeKey": "GET /orders/{id}",
  "rawPath": "/orders/123",
  "rawQueryString": "expand=items",
  "headers": {
    "accept": "application/json",
    "x-request-id": "abc-123"
  },
  "cookies": [
    "session=eyJhbGciOi...",
    "theme=dark"
  ],
  "requestContext": {
    "http": {
      "method": "GET",
      "path": "/orders/123"
    },
    "requestId": "request-id"
  },
  "pathParameters": {
    "id": "123"
  },
  "queryStringParameters": {
    "expand": "items"
  }
}
```

- Use `pathParameters` for route variables such as `/orders/{id}`.
- Use `queryStringParameters` for filters, pagination, and optional modifiers.
- Use `headers` for request metadata, but treat header names as case-insensitive.
- Use `cookies` in HTTP API v2 instead of trying to parse multiple `Set-Cookie` values from headers.

**Writing a robust handler**

Your handler should separate concerns:
- Parse and normalize the incoming request.
- Validate required fields.
- Call business logic.
- Map domain outcomes to HTTP responses.

This structure makes the code easier to test and reduces the chance of returning malformed responses.

```javascript
export const handler = async (event) => {
  const method = event.requestContext?.http?.method ?? event.httpMethod;
  const headers = normalizeHeaders(event.headers);
  const id = event.pathParameters?.id;

  if (method === 'GET' && id) {
    return ok({ id, message: 'Fetched successfully' });
  }

  return response(400, { message: 'Bad request' });
};

function normalizeHeaders(headers = {}) {
  return Object.fromEntries(
    Object.entries(headers).map(([k, v]) => [k.toLowerCase(), v])
  );
}

function ok(body) {
  return response(200, body);
}

function response(statusCode, body, extraHeaders = {}) {
  return {
    statusCode,
    headers: {
      'content-type': 'application/json',
      ...extraHeaders
    },
    body: JSON.stringify(body)
  };
}
```

- Normalize headers when you depend on values such as `authorization`, `content-type`, or `idempotency-key`.
- Return JSON consistently unless your API explicitly serves another media type.
- Keep business logic outside the handler when possible so you can unit test it without AWS context.

**Response formats and status codes**

API Gateway expects a specific response object from Lambda proxy integrations. The exact fields differ slightly by payload version, but the core idea is the same: return a status code, headers, and a serialized body.

For REST API v1, you can return `multiValueHeaders` when you need repeated header keys. For HTTP API v2, use the `cookies` array for multiple cookies rather than repeating `Set-Cookie` in `headers`.

> Do not assume `Set-Cookie` in `headers` is the correct way to return multiple cookies for HTTP API v2. Use the `cookies` field for that payload format.

```javascript
// HTTP API v2 response example
export const handler = async () => {
  return {
    statusCode: 200,
    headers: {
      'content-type': 'application/json'
    },
    cookies: [
      'session=abc123; HttpOnly; Secure; SameSite=Lax',
      'theme=dark; Path=/'
    ],
    body: JSON.stringify({ ok: true })
  };
};
```

For REST API v1, a response can include `multiValueHeaders`:

```javascript
export const handler = async () => {
  return {
    statusCode: 200,
    headers: {
      'content-type': 'application/json'
    },
    multiValueHeaders: {
      'set-cookie': [
        'session=abc123; HttpOnly; Secure; SameSite=Lax',
        'theme=dark; Path=/'
      ]
    },
    body: JSON.stringify({ ok: true })
  };
};
```

- Use `200 OK` for successful reads and updates when a response body is returned.
- Use `201 Created` when you create a new resource.
- Use `204 No Content` when you intentionally return no body.
- Use `400 Bad Request` for validation failures.
- Use `401 Unauthorized` when authentication is missing or invalid.
- Use `403 Forbidden` when the caller is authenticated but not allowed.
- Use `404 Not Found` when the resource does not exist.
- Use `409 Conflict` for duplicate or state-conflict conditions.
- Use `500 Internal Server Error` for unexpected failures.

> A `204 No Content` response should not include a response body. Avoid adding `Content-Type` unless you have a specific reason and know the client behavior you want.

**Path, query, and body handling**

API Gateway can pass route parameters, query strings, and request bodies separately. Your handler should treat each source according to its purpose.

- `pathParameters` identify the resource.
- `queryStringParameters` shape the view of the resource.
- `body` carries the payload for create or update operations.

For JSON requests, parse the body carefully and handle invalid JSON explicitly.

```javascript
export const handler = async (event) => {
  let payload;

  try {
    payload = event.body ? JSON.parse(event.body) : {};
  } catch {
    return {
      statusCode: 400,
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify({ message: 'Invalid JSON body' })
    };
  }

  if (!payload.name) {
    return {
      statusCode: 400,
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify({ message: 'name is required' })
    };
  }

  return {
    statusCode: 201,
    headers: { 'content-type': 'application/json' },
    body: JSON.stringify({ id: '123', name: payload.name })
  };
};
```

- Check `event.isBase64Encoded` when the API may send binary data.
- Decode the body before parsing if the payload is base64-encoded.
- Validate content type when your function accepts more than one media type.

**Idempotency and safe retries**

API clients and gateways may retry requests. For operations that create resources or trigger side effects, idempotency helps prevent duplicates. An idempotency key lets you recognize repeated requests and return the same result safely.

A practical pattern is to read a client-provided key from a header, normalize the header name, and store the result in DynamoDB or another persistence layer.

```javascript
export const handler = async (event) => {
  const headers = normalizeHeaders(event.headers);
  const idempotencyKey = headers['idempotency-key'];

  if (!idempotencyKey) {
    return {
      statusCode: 400,
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify({ message: 'Missing Idempotency-Key header' })
    };
  }

  // Check persistence for an existing result before creating a new one.
  return {
    statusCode: 201,
    headers: { 'content-type': 'application/json' },
    body: JSON.stringify({ message: 'Created', idempotencyKey })
  };
};
```

- Treat header names as case-insensitive.
- Store the request outcome alongside the idempotency key when you need true replay safety.
- Use idempotency for POST-like operations that must not create duplicates.

**Authorizers and request context**

API Gateway can attach identity and authorization data to the event. The exact location depends on the API type and authorizer type.

For REST APIs:
- IAM authorization and custom Lambda authorizers populate data under `requestContext.authorizer`.
- Cognito authorizers can expose claims under `requestContext.authorizer.claims`.

For HTTP APIs:
- JWT authorizers expose claims in the request context according to the HTTP API v2 format.
- Lambda authorizer context is available in the request context for the configured payload format, but the exact structure depends on the authorizer type and configuration.

Be careful not to overgeneralize. Not every authorizer populates `claims`, and custom Lambda authorizers do not behave the same as Cognito or JWT authorizers.

```javascript
export const handler = async (event) => {
  const claims =
    event.requestContext?.authorizer?.claims ??
    event.requestContext?.authorizer?.jwt?.claims ??
    {};

  const userId = claims.sub || claims.username || 'unknown';

  return {
    statusCode: 200,
    headers: { 'content-type': 'application/json' },
    body: JSON.stringify({ userId })
  };
};
```

- Use optional chaining because authorizer data may be absent.
- Check the authorizer type before relying on a specific field.
- Avoid assuming every request includes identity claims.

**CORS and browser clients**

If your API is called from a browser, Cross-Origin Resource Sharing (CORS) matters. API Gateway can handle preflight requests and response headers, but your Lambda response still needs to be compatible with browser expectations.

Typical CORS headers include:
- `Access-Control-Allow-Origin`
- `Access-Control-Allow-Headers`
- `Access-Control-Allow-Methods`
- `Access-Control-Allow-Credentials` when cookies or credentials are involved

For HTTP APIs, CORS configuration is often set at the API level. For REST APIs, you may need to configure method responses, integration responses, and OPTIONS routes depending on the setup.

```javascript
export const handler = async () => {
  return {
    statusCode: 200,
    headers: {
      'content-type': 'application/json',
      'access-control-allow-origin': 'https://example.com',
      'access-control-allow-credentials': 'true'
    },
    body: JSON.stringify({ ok: true })
  };
};
```

- Match `Access-Control-Allow-Origin` to the calling origin when credentials are used.
- Return the correct headers on preflight `OPTIONS` requests.
- Remember that browser failures can be caused by CORS even when Lambda succeeds.

**Binary data and content handling**

If your API returns images, PDFs, or other binary content, API Gateway and Lambda must agree on binary handling. The response may need to be base64-encoded, and the API must be configured to treat the media type as binary.

```javascript
export const handler = async () => {
  const imageBytes = Buffer.from('...binary data...');
  return {
    statusCode: 200,
    headers: {
      'content-type': 'image/png'
    },
    isBase64Encoded: true,
    body: imageBytes.toString('base64')
  };
};
```

- Set `isBase64Encoded` when returning binary content.
- Configure binary media types or equivalent API settings as required.
- Ensure the `Content-Type` matches the actual payload.

**Error handling and mapping failures**

A handler should convert internal failures into safe HTTP responses. Do not leak stack traces or raw database errors to clients. Instead, log the detailed error and return a controlled message.

```javascript
export const handler = async () => {
  try {
    throw new Error('Database timeout');
  } catch (err) {
    console.error('Request failed', err);
    return {
      statusCode: 500,
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify({ message: 'Internal server error' })
    };
  }
};
```

- Log enough detail for troubleshooting.
- Return stable error messages to clients.
- Map known domain errors to specific status codes when possible.

**Testing Lambda handlers**

Test your handler at two levels:
- Unit test the pure business logic with plain JavaScript objects.
- Integration test the Lambda event shape that API Gateway sends.

A REST API event and an HTTP API event are not interchangeable. If you test one, you should not assume the other behaves the same way.

```javascript
// Example Jest-style test
import { handler } from './index.js';

test('returns 200 for a valid GET request', async () => {
  const event = {
    version: '2.0',
    requestContext: { http: { method: 'GET', path: '/orders/123' } },
    pathParameters: { id: '123' },
    headers: { accept: 'application/json' }
  };

  const response = await handler(event);

  expect(response.statusCode).toBe(200);
});
```

For REST API authorizer testing, a Cognito-backed event may include claims such as:

```json
{
  "requestContext": {
    "authorizer": {
      "claims": {
        "sub": "user-123",
        "email": "user@example.com"
      }
    }
  }
}
```

For custom Lambda authorizers, do not assume `claims` exists unless your authorizer explicitly adds them.

> Invoking Lambda directly with the AWS CLI tests the function payload only. It does not test API Gateway request mapping, CORS behavior, authorizer injection, or binary media type handling.

```bash
aws lambda invoke \
  --function-name my-api-handler \
  --payload file://event.json \
  response.json
```

**Local development and debugging**

Local tools help you iterate quickly, but they should mirror the deployed event format you expect. Use sample events that match your API type and payload version.

- Use `sam local invoke` or `sam local start-api` when you want a closer API Gateway simulation.
- Use `aws lambda invoke` when you want to test the Lambda function in isolation.
- Use `curl` against a deployed API when you need to verify the full request path.

```bash
sam local start-api
curl -i http://127.0.0.1:3000/orders/123
```

A practical debugging workflow is:
1. Reproduce the issue with the same API type and payload version.
2. Inspect the raw event in CloudWatch Logs.
3. Confirm the response object matches the expected schema.
4. Verify headers, cookies, and status codes at the API Gateway boundary.

**Exam-focused takeaways**

For DVA-C02, the most important ideas are the request/response contract, the difference between REST API v1 and HTTP API v2 payloads, and the way authorizers and CORS affect the event. You should be able to recognize the correct event fields, choose the right status code, and return a response that API Gateway can transform into a valid HTTP response.

- Know where to read path parameters, query strings, headers, and body content.
- Know how REST API v1 differs from HTTP API v2.
- Know when to use `multiValueHeaders` versus `cookies`.
- Know that direct Lambda invocation is not the same as API Gateway testing.
- Know that authorizer context varies by authorizer type and API configuration.
