## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Strong typing in AWS infrastructure definitions**

Strong typing means your tooling understands the expected shape, type, and allowed values of a resource definition before you deploy it. In AWS, this shows up most clearly in <abbr title="Infrastructure as Code">IaC</abbr> tools such as AWS CloudFormation, AWS Cloud Development Kit (<abbr title="AWS Cloud Development Kit">CDK</abbr>), and AWS Serverless Application Model (<abbr title="AWS SAM">SAM</abbr>).

The exam expects you to understand that strong typing reduces runtime surprises, but it does not eliminate all deployment-time failures. Some issues are caught early by your editor, compiler, or template validator; others are only discovered when CloudFormation creates or updates the stack, or when the target AWS service applies its own constraints.

- **Compile-time safety** comes from languages and libraries that model AWS resources with classes, interfaces, enums, and typed properties.
- **Template-time validation** comes from CloudFormation checking template structure and some property constraints.
- **Deploy-time validation** still happens when AWS services enforce semantic rules that are not fully captured by the template schema.

**CloudFormation as the schema source of truth**

CloudFormation resource type documentation is the authoritative reference for each resource’s properties, return values, update behavior, and allowed values. When you work with strong typing, you should always compare your template or construct against the resource type docs.

A resource definition is not just “valid YAML or JSON.” It must also satisfy the resource schema:

- Required properties must be present.
- Property types must match the schema, such as string, number, list, map, or boolean.
- Some properties accept only specific patterns or enumerated values.
- Some constraints are enforced by CloudFormation, while others are enforced later by the target service.

For example, a CloudFormation template can be syntactically valid but still fail during stack creation if a property value violates a service-specific rule.

```yaml
Resources:
  MyTopic:
    Type: AWS::SNS::Topic
    Properties:
      TopicName: my-app-topic
```

If you supply a value that is the correct type but invalid for the service, the template may pass initial checks and still fail later. That distinction matters for exam questions about validation stages.

**Validation stages you should distinguish**

CloudFormation-related validation happens in layers, and the exam may ask where a problem is detected.

1. **Template parsing and basic validation**
   - CloudFormation checks that the template structure is valid.
   - It verifies that sections such as `Resources`, `Parameters`, `Mappings`, `Conditions`, and `Outputs` are shaped correctly.
   - It catches obvious syntax and schema issues.

2. **Change set creation**
   - CloudFormation can evaluate a proposed update before execution.
   - Change sets show what will be added, modified, or removed.
   - They help you inspect the impact of a deployment, but they do not guarantee success.

3. **Stack operation**
   - During create or update, CloudFormation resolves intrinsic functions and sends requests to AWS services.
   - Some errors appear only here because the final resolved value is invalid in context.
   - The target service may reject a value that CloudFormation could not fully validate.

> Do not assume that “valid template” means “successful deployment.” CloudFormation validation is helpful, but it is not a complete substitute for service-level rules.

**CloudFormation typing basics**

CloudFormation uses a declarative model with typed sections and intrinsic functions. Strong typing matters in several places:

- **Parameters** define input types such as `String`, `Number`, `List<Number>`, and `CommaDelimitedList`.
- **Mappings** are key-value lookup tables with string-like keys and values.
- **Conditions** evaluate to booleans and control whether resources or properties are included.
- **Outputs** expose values such as ARNs, IDs, or URLs for cross-stack use.

Intrinsic functions return values with specific shapes:

- `Ref` returns a resource-specific value, often a name or ID.
- `Fn::GetAtt` returns an attribute from a resource, such as an ARN.
- `Fn::Sub` returns a string after variable substitution.
- `Fn::Join` returns a string.
- `Fn::If` returns one of two values, and both branches must be compatible with the expected property type.

Example:

```yaml
Parameters:
  EnvironmentName:
    Type: String

Resources:
  MyBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub '${EnvironmentName}-assets'

Outputs:
  BucketArn:
    Value: !GetAtt MyBucket.Arn
```

Here, `!Sub` produces a string, and `!GetAtt` returns the bucket ARN, which is suitable for export or cross-stack reference.

**Why strong typing helps**

Strong typing catches mistakes earlier and makes templates easier to reason about.

- It reduces accidental property name typos.
- It helps you pass the right shape of data into resources.
- It improves editor autocomplete and inline documentation.
- It makes refactoring safer when resource names or outputs change.
- It helps you build reusable stacks and constructs with predictable interfaces.

In practice, strong typing is especially useful when you compose resources across stacks. For example, exporting a typed output such as a bucket ARN or queue URL makes it easier for another stack to consume the value correctly.

```yaml
Outputs:
  QueueUrl:
    Value: !Ref MyQueue
    Export:
      Name: MyQueueUrl
```

A consuming stack can import the value and use it where a string URL is expected.

**Common CloudFormation typing pitfalls**

Some failures are simple type mismatches, while others are semantic mismatches that only appear later.

- A property expects a string, but you provide a number.
- A property expects a list, but you provide a scalar.
- A property expects a specific format, but the value violates service rules.
- A resource is missing a required property.
- A conditional branch returns a value of the wrong shape.

Examples:

```yaml
Resources:
  BadBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: 12345
```

`BucketName` expects a string. An unquoted numeric scalar is parsed as a number in YAML, so this is a type mismatch.

```yaml
Resources:
  BadLambda:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: my-function
      Runtime: python3.12
      Handler: index.handler
```

This is incomplete because `Code` is required. CloudFormation can detect the missing required property before deployment.

```yaml
Resources:
  MyTopic:
    Type: AWS::SNS::Topic
    Properties:
      TopicName: !Sub '${AWS::StackName}-topic'
```

This is valid if the resulting name satisfies SNS topic naming rules. The important point is that a resolved string can still fail if it violates service constraints.

**CDK strong typing**

The AWS CDK uses programming language types to model AWS resources. This gives you compile-time help before synthesis.

CDK resources are usually represented by:

- **L1 constructs**: direct CloudFormation mappings, such as `CfnBucket`
- **L2 constructs**: higher-level, opinionated constructs with safer defaults
- **L3 constructs**: patterns that combine multiple resources into a reusable architecture

Type safety appears in several ways:

- Constructors require the correct property interface.
- Enums restrict values to known options.
- Methods and attributes return typed values.
- IDEs can autocomplete valid properties and methods.

Example in TypeScript:

```ts
import * as s3 from 'aws-cdk-lib/aws-s3';
import { Stack, StackProps } from 'aws-cdk-lib';
import { Construct } from 'constructs';

export class StorageStack extends Stack {
  constructor(scope: Construct, id: string, props?: StackProps) {
    super(scope, id, props);

    const bucket = new s3.Bucket(this, 'AssetsBucket', {
      versioned: true,
      encryption: s3.BucketEncryption.S3_MANAGED,
    });

    bucket.addLifecycleRule({
      prefix: 'tmp/',
      expiration: cdk.Duration.days(30),
    });
  }
}
```

This is strongly typed because:

- `versioned` expects a boolean.
- `encryption` expects a value from the `BucketEncryption` enum.
- `addLifecycleRule` expects a structured object with known fields.

If you pass the wrong type, TypeScript flags it before deployment.

**CDK tokens and unresolved values**

CDK often works with <abbr title="CloudFormation intrinsic values that are not resolved until synthesis or deployment">tokens</abbr>. A token is a placeholder for a value that is not fully known when you write the code.

Important distinction:

- Some values are resolved during **synthesis** into CloudFormation template expressions.
- Some values remain unresolved placeholders in the synthesized template and are only resolved during **deployment** by CloudFormation.
- The CDK type system still treats many of these as strings, numbers, or lists, even though the concrete value is not yet known.

Example:

```ts
const bucketArn = bucket.bucketArn;
```

`bucketArn` is typed as a string in TypeScript, but at synthesis time it may become a CloudFormation expression such as `Fn::GetAtt`. That is still useful because the consuming code can treat it as the correct type.

This is why CDK is powerful for cross-stack references:

- You can pass a bucket ARN to a Lambda permission.
- You can pass a queue URL to an event source mapping.
- You can pass a table name to an environment variable.

Example:

```ts
new lambda.Function(this, 'Fn', {
  runtime: lambda.Runtime.NODEJS_20_X,
  handler: 'index.handler',
  code: lambda.Code.fromAsset('lambda'),
  environment: {
    BUCKET_ARN: bucket.bucketArn,
  },
});
```

The environment variable value is ultimately a string, but it can be supplied by a token that resolves to a string.

**CDK compile-time safety beyond simple properties**

CDK gives you more than property typing. It also helps you model valid AWS usage patterns.

- **Interfaces** define required and optional configuration fields.
- **Enums** prevent invalid string literals.
- **Constructs** encapsulate best practices and reduce low-level mistakes.
- **Method signatures** guide you toward valid combinations of inputs.

Examples of safer patterns:

- `lambda.Runtime.NODEJS_20_X` instead of a raw string like `'nodejs20.x'`
- `s3.BucketEncryption.S3_MANAGED` instead of a custom string
- `ec2.InstanceType.of(ec2.InstanceClass.T3, ec2.InstanceSize.MICRO)` instead of hand-built instance type strings

These patterns reduce the chance of typos and invalid combinations.

**SAM and strong typing**

AWS SAM is a transform on top of CloudFormation, so it inherits CloudFormation’s schema rules after translation. SAM adds shorthand syntax for serverless resources, but the resulting template still must satisfy CloudFormation and the target service.

SAM helps with strong typing in two ways:

- It provides a more concise resource model for common serverless patterns.
- It still relies on CloudFormation property validation after the transform expands the template.

Example:

```yaml
Resources:
  HelloFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: src/
      Handler: app.handler
      Runtime: nodejs20.x
      Environment:
        Variables:
          TABLE_NAME: !Ref OrdersTable
```

The environment variable value is a string after resolution, but it can be supplied by an intrinsic function. SAM does not require literal strings only.

A common SAM pitfall is assuming the shorthand removes all CloudFormation rules. It does not. If the translated template contains a missing required property or an invalid value, deployment still fails.

Example of a typing pitfall:

```yaml
Resources:
  BadFunction:
    Type: AWS::Serverless::Function
    Properties:
      Handler: index.handler
      Runtime: python3.12
```

This is incomplete because `CodeUri` or an equivalent code location is required. SAM may accept the shorthand structure, but the transformed CloudFormation resource still needs valid code configuration.

**Mappings, conditions, and outputs as typed structures**

These sections are often overlooked, but they matter for exam questions about strong typing.

- **Mappings** are lookup tables used to select values based on keys such as region or environment.
- **Conditions** return boolean-like results and control resource creation or property inclusion.
- **Outputs** expose values for humans, scripts, and other stacks.

Example:

```yaml
Mappings:
  EnvMap:
    dev:
      InstanceType: t3.micro
    prod:
      InstanceType: t3.small

Conditions:
  IsProd: !Equals [!Ref EnvironmentName, prod]

Outputs:
  AppUrl:
    Value: !Sub 'https://${LoadBalancer.DNSName}'
```

Use these structures carefully:

- Mapping values should match the expected downstream type.
- Condition branches must be compatible with the property they control.
- Outputs should return values that are meaningful to consumers, such as ARNs, URLs, or names.

**Cross-stack references and exported values**

Strong typing becomes especially valuable when one stack depends on another.

- Export a value with a clear semantic meaning, such as a queue URL or bucket ARN.
- Import it in another stack using the correct expected type.
- Avoid guessing at formats; use the documented return value of the resource.

Example:

```yaml
Outputs:
  TableName:
    Value: !Ref OrdersTable
    Export:
      Name: OrdersTableName
```

A consuming stack can import the value and use it in an environment variable, IAM policy, or event source configuration.

This matters because cross-stack references are often string-based, but the string must represent the right thing:

- ARN for permissions
- Name for resource lookup
- URL for queue consumers
- ID for network or resource association

**Practical CLI workflow**

Use AWS CLI commands to validate and inspect templates, but remember their limits.

```bash
aws cloudformation validate-template --template-body file://template.yaml
aws cloudformation create-change-set \
  --stack-name my-stack \
  --change-set-name preview-1 \
  --template-body file://template.yaml \
  --capabilities CAPABILITY_IAM
aws cloudformation deploy \
  --stack-name my-stack \
  --template-file template.yaml \
  --capabilities CAPABILITY_NAMED_IAM
```

What these commands do:

- `validate-template` checks basic template structure and some template-level issues.
- `create-change-set` previews the update and can surface some problems before execution.
- `deploy` performs the actual stack operation and may reveal service-level validation failures.

> `validate-template` does not perform full resource schema validation or deploy-time service validation. Always confirm resource-specific rules in the CloudFormation documentation.

**How to think about exam questions**

When the exam asks about strong typing, look for the best answer that reduces errors earlier in the development lifecycle.

- Choose CDK when you want language-level typing, IDE support, and reusable constructs.
- Choose CloudFormation when you want declarative infrastructure with documented resource schemas.
- Choose SAM when you want concise serverless definitions that still translate into CloudFormation.
- Use resource documentation when you need the exact property type, allowed values, or return value.

A good mental model is:

- **Language type system** catches mistakes before synthesis.
- **CloudFormation schema** catches many template issues before or during stack operation.
- **AWS service validation** catches the remaining semantic constraints during deployment.

**Quick reference**

| Area | What strong typing gives you | Common failure mode |
|---|---|---|
| CloudFormation parameters | Correct input shape | Wrong type or missing required value |
| CloudFormation intrinsic functions | Predictable return types | Resolved value violates service rules |
| CDK constructs | Compile-time safety and autocomplete | Invalid combination of properties |
| SAM resources | Concise serverless definitions | Translation still produces invalid CloudFormation |
| Cross-stack references | Safer sharing of outputs | Wrong semantic value, such as name vs ARN |

**Key takeaways**

- Strong typing reduces infrastructure mistakes before deployment.
- CloudFormation resource documentation is the source of truth for schemas and allowed values.
- CDK adds compile-time safety through typed constructs, enums, and interfaces.
- SAM simplifies serverless definitions but still depends on CloudFormation validation after translation.
- Not all errors are caught early; some appear only during stack creation or at the target service.
