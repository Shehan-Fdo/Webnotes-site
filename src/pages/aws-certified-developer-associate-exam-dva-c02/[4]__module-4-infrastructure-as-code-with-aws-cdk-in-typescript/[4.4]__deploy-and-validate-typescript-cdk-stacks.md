## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**CDK deployment workflow**

The AWS Cloud Development Kit (CDK) lets you define cloud infrastructure in code and synthesize it into AWS CloudFormation templates. In TypeScript, you typically work in a project that contains an app, one or more stacks, and supporting constructs. Deployment is not just “push code”; you first synthesize, inspect, and validate the generated template, then deploy it to an AWS account and Region.

A reliable workflow is:

- write or update constructs in TypeScript
- run `cdk synth` to generate CloudFormation
- run `cdk diff` to review changes
- run `cdk deploy` to create or update stacks
- validate the deployed resources and application behavior

This sequence matters because CDK abstractions can hide low-level CloudFormation details. Validation helps you catch unintended replacements, missing permissions, bad environment settings, and resource naming issues before they affect production.

**Project structure and core CDK concepts**

A TypeScript CDK app usually includes these parts:

- **App**: the entry point that instantiates stacks
- **Stack**: a deployable unit that maps to a CloudFormation stack
- **Construct**: a reusable building block for one or more AWS resources
- **Environment**: the target AWS account and Region for deployment

A minimal `bin` entry point often looks like this:

```typescript
#!/usr/bin/env node
import * as cdk from 'aws-cdk-lib';
import { MyAppStack } from '../lib/my-app-stack';

const app = new cdk.App();

new MyAppStack(app, 'MyAppStack', {
  env: {
    account: process.env.CDK_DEFAULT_ACCOUNT,
    region: process.env.CDK_DEFAULT_REGION,
  },
});
```

The `env` property is important when you want deterministic deployments to a specific account and Region. If you omit it, CDK can synthesize environment-agnostic templates, which is useful in some cases but can limit features that require concrete account or Region values.

**Bootstrap requirements**

Before deploying a CDK app to an environment, you often need to bootstrap that environment. Bootstrapping creates supporting resources such as an S3 bucket for assets and roles used by the CDK deployment process.

Typical bootstrap command:

```bash
cdk bootstrap aws://123456789012/us-east-1
```

Use bootstrapping when:

- you deploy assets such as Lambda bundles, Docker images, or file assets
- you deploy to a new account or Region
- your stack uses modern CDK asset publishing features

If bootstrapping is missing, deployment can fail with asset publishing or permissions errors. In exam scenarios, always consider whether the target environment has been bootstrapped.

**Synthesizing and inspecting templates**

`cdk synth` converts your TypeScript app into a CloudFormation template. This is the first validation checkpoint because it confirms that your code compiles and that CDK can resolve the construct tree.

```bash
npm run build
cdk synth
```

Common reasons to synthesize before deployment:

- verify that the stack compiles successfully
- inspect generated logical IDs and resource properties
- confirm that parameters, mappings, and conditions render as expected
- catch invalid references early

You can also direct output to a file for review:

```bash
cdk synth > template.yaml
```

When reviewing the synthesized template, look for:

- unexpected resource replacements
- incorrect IAM policies
- wrong bucket names, ARNs, or environment values
- missing dependencies between resources

**Diffing changes before deployment**

`cdk diff` compares your current CDK app with the deployed stack and shows what will change. This is one of the most useful validation tools because it reveals whether a change is additive, destructive, or replacement-based.

```bash
cdk diff MyAppStack
```

Use `cdk diff` to identify:

- resource additions and deletions
- property changes that trigger replacement
- IAM policy changes
- changes to security groups, buckets, or databases

A diff is especially important when working with stateful resources. For example, changing a database identifier or certain storage properties can cause replacement, which may lead to data loss if you do not plan carefully.

> Treat any replacement of stateful resources as a high-risk change and verify backup, migration, or retention settings before deploying.

**Deploying stacks**

`cdk deploy` synthesizes the app if needed and then creates or updates the CloudFormation stack. You can deploy a single stack or multiple stacks.

```bash
cdk deploy MyAppStack
```

Useful deployment options include:

```bash
cdk deploy --all
cdk deploy MyAppStack --require-approval never
cdk deploy MyAppStack --outputs-file outputs.json
```

Common deployment behaviors:

- CDK uploads assets to the bootstrap bucket or asset store
- CloudFormation executes the change set
- stack events show progress and failures
- outputs can be captured for downstream validation

Use `--require-approval` carefully. CDK may prompt for approval when a deployment introduces security-sensitive changes, such as IAM policy broadening or security group exposure. In automated pipelines, you may suppress prompts only when you have separate review controls.

**Validating deployment success**

Deployment success means more than “the command exited cleanly.” You should validate both infrastructure state and application behavior.

Validation checks can include:

- confirming the stack status is `CREATE_COMPLETE` or `UPDATE_COMPLETE`
- checking CloudFormation events for failures or rollbacks
- verifying outputs such as URLs, ARNs, or resource names
- testing the application endpoint or API behavior
- confirming logs, metrics, and alarms behave as expected

Example CloudFormation inspection commands:

```bash
aws cloudformation describe-stacks --stack-name MyAppStack
aws cloudformation describe-stack-events --stack-name MyAppStack
```

If the stack enters `ROLLBACK_COMPLETE` or `UPDATE_ROLLBACK_COMPLETE`, inspect the failure reason before retrying. Some failed stacks require deletion or manual cleanup before redeployment.

**Validating with AWS CLI and service-specific checks**

After deployment, use the AWS CLI to confirm that the expected resources exist and are configured correctly. The exact checks depend on the services in the stack.

Examples:

```bash
aws s3 ls
aws lambda get-function --function-name MyFunction
aws apigateway get-rest-apis
aws logs describe-log-groups
```

For an API-backed stack, you might validate:

- the API endpoint returns the expected status code
- the Lambda integration logs show successful invocation
- the IAM role has the minimum required permissions
- the deployed stage matches the intended environment

For an S3-backed stack, you might validate:

- the bucket exists in the correct Region
- bucket encryption is enabled if required
- public access settings match the design
- event notifications or lifecycle rules are present

**Practical validation patterns**

A good validation strategy combines infrastructure checks and functional tests. Infrastructure checks confirm that AWS resources exist and are configured correctly. Functional tests confirm that the application behaves as intended.

Example validation flow for a Lambda and API Gateway stack:

```bash
cdk diff MyApiStack
cdk deploy MyApiStack
aws cloudformation describe-stacks --stack-name MyApiStack
curl -i https://abc123.execute-api.us-east-1.amazonaws.com/prod/health
aws logs tail /aws/lambda/MyFunction --follow
```

This pattern helps you detect:

- deployment drift between code and cloud state
- broken integrations
- missing environment variables
- permission issues that only appear at runtime

**Context-specific deployment concerns**

TypeScript CDK stacks often include environment variables, asset bundling, and cross-stack references. These can affect deployment and validation.

- **Environment variables**: ensure values are available during synthesis or deployment when needed
- **Asset bundling**: confirm build output is generated correctly before `cdk deploy`
- **Cross-stack references**: verify dependencies and export/import names
- **Logical IDs**: understand that construct path changes can alter resource identities

A common Lambda asset workflow looks like this:

```bash
npm run build
cdk synth
cdk deploy
```

If your Lambda code is bundled during synthesis, a build failure can prevent deployment even when the infrastructure code is valid. Validate both the application artifact and the stack definition.

**IAM and security validation**

CDK often generates IAM roles and policies automatically. You should validate these carefully because overly broad permissions are a common risk.

Look for:

- wildcard actions such as `*`
- wildcard resources where a narrower ARN is possible
- unintended trust relationships
- missing permissions that cause runtime failures

Example policy snippet:

```typescript
myFunction.addToRolePolicy(new iam.PolicyStatement({
  actions: ['dynamodb:GetItem', 'dynamodb:PutItem'],
  resources: [table.tableArn],
}));
```

This is preferable to broad permissions because it limits access to the specific table. During validation, confirm that the deployed role contains only the permissions required by the workload.

**Handling failures and rollbacks**

CloudFormation manages deployment failures by rolling back changes unless rollback is disabled in a specific workflow. When a deployment fails, the stack events are the best source of truth.

Typical failure causes:

- invalid IAM permissions
- resource name conflicts
- unsupported property values
- dependency ordering issues
- missing bootstrap resources
- quota limits

Troubleshooting approach:

1. Check the latest stack event for the failing resource.
2. Read the exact error message and identify the AWS service involved.
3. Fix the CDK code or environment issue.
4. Run `cdk diff` again to confirm the intended change.
5. Redeploy after the issue is resolved.

**Useful commands for day-to-day work**

```bash
cdk ls
cdk synth
cdk diff
cdk deploy
cdk destroy
```

Additional commands that help with validation:

```bash
cdk doctor
cdk context
aws cloudformation list-stacks
aws cloudformation describe-stack-resources --stack-name MyAppStack
```

Use `cdk doctor` when you suspect local environment issues such as missing dependencies, incorrect credentials, or CDK configuration problems. Use `cdk context` when you need to inspect cached lookups that may affect synthesis.

**Validation in CI/CD pipelines**

In a pipeline, you usually automate the same checks you would run manually. A typical sequence is:

1. install dependencies
2. run tests and linting
3. build TypeScript
4. run `cdk synth`
5. run `cdk diff` against the target environment if permitted
6. deploy with controlled approvals
7. run post-deployment smoke tests

Example pipeline-oriented commands:

```bash
npm ci
npm test
npm run build
cdk synth
cdk deploy MyAppStack --require-approval never
```

In automated environments, make sure the pipeline has:

- AWS credentials with the correct permissions
- access to the target account and Region
- bootstrap resources in place
- a clear rollback or notification strategy

**Common exam-relevant distinctions**

- **Synthesis** happens locally and produces CloudFormation.
- **Deployment** happens in AWS through CloudFormation.
- **Validation** confirms both the template and the live resources.
- **Diffing** helps you understand impact before changes are applied.
- **Bootstrapping** prepares the environment for asset-based deployments.

These distinctions matter because exam questions often ask which step identifies a problem before deployment, which step creates AWS resources, or which step verifies the deployed application.

**Quick reference checklist**

- confirm AWS credentials and target account/Region
- bootstrap the environment if assets are involved
- run tests and build TypeScript
- run `cdk synth`
- inspect `cdk diff`
- deploy the stack
- verify CloudFormation status and stack events
- test the application endpoint or service behavior
- review logs, metrics, and IAM permissions

**Key takeaways**

TypeScript CDK deployment is safest when you treat it as a controlled workflow rather than a single command. Synthesis, diffing, deployment, and validation each answer a different question about your infrastructure. Use CloudFormation events, AWS CLI checks, and application smoke tests together to confirm that the stack is both correctly provisioned and functionally ready.
