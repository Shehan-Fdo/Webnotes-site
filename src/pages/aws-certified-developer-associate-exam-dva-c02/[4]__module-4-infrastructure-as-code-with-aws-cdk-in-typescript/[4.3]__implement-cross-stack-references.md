## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Cross-stack references in CloudFormation**

<abbr title="AWS CloudFormation">CloudFormation</abbr> cross-stack references let one stack consume values created by another stack without hardcoding identifiers. You typically use them to share infrastructure primitives such as VPC IDs, subnet IDs, security group IDs, bucket names, or API endpoints across independently managed stacks.

The core pattern is:

- The producing stack exports a value with `Outputs` and `Export`.
- The consuming stack imports that value with `Fn::ImportValue` or the `!ImportValue` shorthand.
- CloudFormation tracks the dependency so the export cannot be removed while it is still in use.

This is useful when you want separate lifecycle boundaries, such as networking in one stack and application resources in another.

**How exports and imports work**

A stack can publish values through its outputs. An exported output becomes globally available within the same AWS Region and account. Another stack can import that value by name.

```yaml
Outputs:
  VpcId:
    Description: VPC identifier for application stacks
    Value: !Ref AppVpc
    Export:
      Name: !Sub '${AWS::StackName}-VpcId'
```

A consumer stack can then reference it:

```yaml
Resources:
  AppSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: App SG
      VpcId: !ImportValue network-stack-VpcId
```

Key points:

- **Exports are regional**: the producer and consumer must be in the same AWS Region.
- **Exports are account-scoped**: cross-account import is not supported by CloudFormation exports.
- **Export names must be unique** within an account and Region.
- **Imported values are resolved at deployment time**, not at template parse time.

**When to use cross-stack references**

Use cross-stack references when you need a clean separation of concerns:

- A shared networking stack creates the VPC, subnets, route tables, and shared security groups.
- An application stack creates compute, load balancers, and databases that need those network identifiers.
- A data stack publishes a bucket name or KMS key ARN for multiple application stacks.

This pattern is especially helpful when:

- Different teams own different stacks.
- You want to update application resources without touching foundational infrastructure.
- You need to reduce duplication of shared resource definitions.

Avoid overusing exports for everything. If many stacks depend on a single export, you create a strong coupling that makes refactoring harder.

**Important limitations**

CloudFormation enforces dependency rules around exports:

- You cannot delete or modify an exported output if another stack imports it.
- You must remove the import from all consuming stacks before changing the export name or deleting the export.
- Export names should be stable and intentionally versioned if you expect evolution.

> If you need to change a widely used export, create a new export name first, update consumers to import the new value, and only then remove the old export.

**Practical export naming strategies**

Good export names are predictable and unique. Common patterns include:

- `${AWS::StackName}-VpcId`
- `network-prod-VpcId`
- `shared-${AWS::Region}-SubnetAId`

Use `Fn::Sub` to build names dynamically when appropriate:

```yaml
Outputs:
  PublicSubnetA:
    Value: !Ref PublicSubnetA
    Export:
      Name: !Sub '${AWS::StackName}:PublicSubnetA'
```

This helps avoid collisions and makes the source stack obvious.

**Example: network stack and application stack**

A common architecture is a foundational network stack plus one or more application stacks.

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Network stack

Resources:
  AppVpc:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.0.0.0/16

  PublicSubnetA:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref AppVpc
      CidrBlock: 10.0.1.0/24
      AvailabilityZone: !Select [0, !GetAZs '']

Outputs:
  VpcId:
    Value: !Ref AppVpc
    Export:
      Name: network-prod-VpcId

  PublicSubnetAId:
    Value: !Ref PublicSubnetA
    Export:
      Name: network-prod-PublicSubnetAId
```

Consumer stack:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Application stack

Resources:
  AppSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: App security group
      VpcId: !ImportValue network-prod-VpcId

  AppInstance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: ami-0123456789abcdef0
      InstanceType: t3.micro
      SubnetId: !ImportValue network-prod-PublicSubnetAId
      SecurityGroupIds:
        - !Ref AppSecurityGroup
```

This pattern keeps the network stack reusable while allowing the application stack to evolve independently.

**CLI workflow**

You can deploy stacks and inspect exports with the AWS CLI.

```bash
aws cloudformation deploy \
  --stack-name network-prod \
  --template-file network.yaml \
  --capabilities CAPABILITY_NAMED_IAM

aws cloudformation list-exports

aws cloudformation deploy \
  --stack-name app-prod \
  --template-file app.yaml \
  --capabilities CAPABILITY_NAMED_IAM
```

Useful inspection commands:

```bash
aws cloudformation describe-stacks --stack-name network-prod
aws cloudformation list-imports --export-name network-prod-VpcId
```

These commands help you verify which values are exported and which stacks depend on them.

**Dependency behavior and update order**

CloudFormation manages dependency ordering automatically when a stack imports an export. The producer stack cannot be updated in a way that breaks the consumer.

Practical implications:

- Updating the exported resource may trigger updates in consuming stacks if the imported value changes.
- Renaming an export is a breaking change.
- Deleting a producer stack usually fails if its exports are still imported.

If you need to replace a resource, consider a migration plan:

1. Create a new resource in the producer stack.
2. Export the new value under a new name.
3. Update consumers to import the new export.
4. Remove the old import references.
5. Delete the old export after all consumers are migrated.

**Alternatives to cross-stack references**

Cross-stack references are not always the best choice. Other options include:

- **Parameters**: pass values manually or through automation when stacks are deployed together.
- **SSM Parameter Store**: store shared values centrally when you need looser coupling or cross-account patterns.
- **AWS Secrets Manager**: use for sensitive values, not for infrastructure IDs.
- **Nested stacks**: use when you want a parent stack to manage child stacks with tighter orchestration.

A quick comparison:

| Approach | Best for | Notes |
|---|---|---|
| Cross-stack references | Shared infrastructure values in same account/Region | Strong dependency tracking |
| Parameters | Simple one-time inputs | No automatic linkage |
| SSM Parameter Store | Shared runtime configuration | More flexible, can be cross-stack and cross-service |
| Nested stacks | Hierarchical CloudFormation design | Parent-child lifecycle management |

**Troubleshooting common issues**

Common errors usually come from naming, scope, or dependency problems.

- **Export name already exists**: another stack in the same account and Region already uses that export name.
- **No export named ... found**: the consumer references a name that does not exist or is in a different Region/account.
- **Export cannot be deleted as it is in use**: one or more stacks still import the value.
- **Circular dependency**: two stacks depend on each other through exports and imports; break the cycle by redesigning the shared boundary.

A practical debugging sequence:

1. Run `aws cloudformation list-exports` to confirm the export exists.
2. Run `aws cloudformation list-imports --export-name <name>` to identify consumers.
3. Check that both stacks are in the same Region and account.
4. Verify the export name matches exactly, including case and punctuation.
5. Remove or update imports before changing the export.

**Best practices**

- Keep exports focused on stable infrastructure identifiers.
- Use clear, versioned export names when change is likely.
- Export only what consumers truly need.
- Prefer one-way dependencies from foundational stacks to application stacks.
- Document the contract between stacks so teams know which exports are stable.
- Use `Fn::Sub` for readable export names and `Fn::ImportValue` for explicit consumption.

**Exam-focused takeaways**

For the exam, remember these high-value facts:

- Cross-stack references use `Outputs` with `Export` and `Fn::ImportValue`.
- They work only within the same AWS account and Region.
- Export names must be unique and cannot be changed while in use.
- They are ideal for sharing VPC, subnet, and security group identifiers across stacks.
- If a change is required, migrate consumers to a new export name before removing the old one.

**Quick reference**

```yaml
Outputs:
  SharedValue:
    Value: !Ref SomeResource
    Export:
      Name: shared-stack-SharedValue
```

```yaml
Resources:
  ConsumerResource:
    Type: AWS::Some::Resource
    Properties:
      SharedId: !ImportValue shared-stack-SharedValue
```
