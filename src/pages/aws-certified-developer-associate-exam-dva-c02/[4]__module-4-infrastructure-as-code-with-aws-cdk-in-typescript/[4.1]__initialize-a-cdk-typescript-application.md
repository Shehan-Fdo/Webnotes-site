## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**What an AWS CDK TypeScript app is**

The AWS Cloud Development Kit (<abbr title="Cloud Development Kit">CDK</abbr>) lets you define cloud infrastructure in familiar programming languages. In a TypeScript app, you write constructs, stacks, and app logic in TypeScript, then synthesize them into AWS CloudFormation templates.

This topic focuses on the initial project setup: creating the app, understanding the generated files, installing dependencies, and preparing the project for iterative infrastructure development.

- Use CDK when you want infrastructure as code with programming-language features such as loops, conditionals, and reusable abstractions.
- Use TypeScript when you want strong typing, broad CDK support, and a common developer workflow with Node.js tooling.
- The initial app structure is designed to separate reusable infrastructure code from deployment configuration.

**Prerequisites and local tooling**

Before you initialize a CDK app, make sure your workstation has the required tools. CDK TypeScript projects depend on Node.js and npm, and you also need AWS credentials configured for deployment.

Typical prerequisites include:

- **Node.js** and **npm** for package management and the CDK CLI.
- **AWS CLI** for credential configuration and account access.
- An AWS account and an IAM identity with permissions to create the resources you plan to deploy.
- A supported TypeScript editor or IDE for code completion and diagnostics.

A common setup flow looks like this:

```bash
node --version
npm --version
aws --version
cdk --version
```

If the CDK CLI is not installed globally, install it with:

```bash
npm install -g aws-cdk
```

> If you use multiple AWS accounts or profiles, verify the active profile before deploying. A wrong profile can synthesize correctly but deploy to the wrong account or region.

**Create the project directory and initialize the app**

You usually start by creating a new directory and running the CDK app initializer. The CDK CLI provides a TypeScript project template that includes the standard files needed for a working app.

```bash
mkdir my-cdk-app
cd my-cdk-app
cdk init app --language typescript
```

The `cdk init` command creates a project scaffold and installs the baseline dependencies. The `app` template is the standard starting point for a CDK application that defines one or more stacks.

After initialization, you typically see files such as:

- `bin/` for the application entry point.
- `lib/` for stack definitions and constructs.
- `package.json` for scripts and dependencies.
- `tsconfig.json` for TypeScript compiler settings.
- `cdk.json` for CDK app execution settings.
- `README.md` for project notes and usage guidance.

**Generated project structure**

The generated structure separates the application bootstrap from the infrastructure definition. This makes it easier to grow from a single stack into a multi-stack application.

A typical TypeScript CDK layout resembles:

```text
my-cdk-app/
├── bin/
│   └── my-cdk-app.ts
├── lib/
│   └── my-cdk-app-stack.ts
├── test/
├── cdk.json
├── package.json
├── tsconfig.json
└── README.md
```

Key files and their roles:

- **`bin/<app-name>.ts`** is the entry point that creates the CDK `App` and instantiates stacks.
- **`lib/<stack-name>.ts`** contains the stack class where you define AWS resources.
- **`cdk.json`** tells the CDK CLI how to run the app, usually through `npx ts-node`.
- **`package.json`** stores scripts such as `build`, `watch`, `test`, and `cdk`.
- **`tsconfig.json`** configures TypeScript compilation and module resolution.

The separation matters because the `bin` file handles application wiring, while the `lib` file holds reusable infrastructure logic.

**Core CDK concepts in the starter app**

A CDK app is built around a few core abstractions:

- **App**: the top-level container for one or more stacks.
- **Stack**: a deployable unit that maps to a CloudFormation stack.
- **Construct**: the basic building block used to define AWS resources and higher-level patterns.
- **Environment**: the target AWS account and region for deployment.

In the starter app, you usually create an `App` instance and pass it into one or more stack classes.

```ts
import * as cdk from 'aws-cdk-lib';
import { MyCdkAppStack } from '../lib/my-cdk-app-stack';

const app = new cdk.App();

new MyCdkAppStack(app, 'MyCdkAppStack', {
  /* If you want to target a specific account/region, set env here */
  // env: { account: process.env.CDK_DEFAULT_ACCOUNT, region: process.env.CDK_DEFAULT_REGION },
});
```

This pattern gives you a clean entry point for adding additional stacks later.

**Dependencies and package management**

A TypeScript CDK app uses npm dependencies for both runtime and development tooling. The initializer typically adds the CDK libraries and TypeScript support packages.

Common packages include:

- `aws-cdk-lib` for the CDK v2 library.
- `constructs` for construct base classes.
- `typescript` for compilation.
- `ts-node` for running TypeScript directly during synthesis.
- `@types/node` for Node.js type definitions.

A typical `package.json` may include scripts like:

```json
{
  "scripts": {
    "build": "tsc",
    "watch": "tsc -w",
    "test": "jest",
    "cdk": "cdk"
  }
}
```

Practical implications:

- `npm run build` checks that your TypeScript compiles cleanly.
- `npm run watch` helps during iterative development by recompiling on file changes.
- `npx cdk synth` generates the CloudFormation template from your TypeScript code.
- `npx cdk deploy` deploys the synthesized stack to AWS.

**TypeScript configuration basics**

The `tsconfig.json` file controls how TypeScript compiles your CDK code. CDK starter projects usually target a modern Node.js runtime and enable settings that support decorators and module interoperability where needed.

A simplified example:

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "commonjs",
    "strict": true,
    "esModuleInterop": true,
    "outDir": "dist",
    "rootDir": "."
  }
}
```

Important settings to understand:

- **`strict`** improves type safety and catches mistakes early.
- **`esModuleInterop`** helps with importing CommonJS and ES modules.
- **`outDir`** defines where compiled JavaScript is written if you build locally.
- **`rootDir`** helps TypeScript understand the source layout.

In many CDK workflows, you synthesize directly from TypeScript using `ts-node`, so compilation output is less important than type checking and editor support.

**CDK app execution with `cdk.json`**

The `cdk.json` file tells the CDK CLI how to execute your app. In a TypeScript project, it commonly uses `npx ts-node` so the CLI can run the TypeScript entry point without a separate manual build step.

Example:

```json
{
  "app": "npx ts-node --prefer-ts-exts bin/my-cdk-app.ts"
}
```

Why this matters:

- The CDK CLI reads `cdk.json` to know which file to execute.
- `ts-node` runs TypeScript directly.
- `--prefer-ts-exts` helps resolution when both `.ts` and compiled files exist.

**Initialize, synthesize, and deploy workflow**

A typical workflow after initialization is straightforward:

1. Install dependencies if the template did not do so automatically.
2. Review the generated stack and app entry point.
3. Synthesize the template to validate the app.
4. Bootstrap the environment if required.
5. Deploy the stack.

Example commands:

```bash
npm install
npm run build
npx cdk synth
npx cdk bootstrap
npx cdk deploy
```

Use `cdk bootstrap` when the target account and region do not yet have the CDK bootstrap resources. These resources support asset publishing and deployment operations.

**Example stack definition**

A starter stack often begins with a simple resource so you can verify the deployment path end to end. For example, you might create an Amazon S3 bucket or an Amazon DynamoDB table.

```ts
import * as cdk from 'aws-cdk-lib';
import { Construct } from 'constructs';
import * as s3 from 'aws-cdk-lib/aws-s3';

export class MyCdkAppStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    new s3.Bucket(this, 'DemoBucket', {
      versioned: true,
      removalPolicy: cdk.RemovalPolicy.DESTROY,
      autoDeleteObjects: true,
    });
  }
}
```

This example shows several common CDK patterns:

- **Construct inheritance** is used to define a stack class.
- **Resource properties** are expressed in TypeScript objects.
- **Removal policy** controls what happens when the stack is deleted.
- **Auto-delete behavior** is useful for non-production test environments.

> Be careful with destructive settings such as `RemovalPolicy.DESTROY` and `autoDeleteObjects: true`. They are useful for learning and ephemeral environments, but they are usually inappropriate for production data.

**Environment targeting and account/region selection**

You can let CDK infer the target account and region from your AWS CLI profile, or you can set them explicitly in stack props. Explicit environment settings are useful when you want predictable deployments or when your app must synthesize environment-specific resources.

Example:

```ts
new MyCdkAppStack(app, 'MyCdkAppStack', {
  env: {
    account: process.env.CDK_DEFAULT_ACCOUNT,
    region: process.env.CDK_DEFAULT_REGION,
  },
});
```

Use environment variables when you want the app to follow the active AWS CLI profile. Use hardcoded values only when you intentionally target a fixed environment.

**Common CLI commands you should know**

The CDK CLI is central to the development loop. These commands are especially important during initialization:

```bash
cdk init app --language typescript
cdk synth
cdk diff
cdk deploy
cdk destroy
cdk list
```

What each command does:

- **`cdk synth`** converts your TypeScript app into a CloudFormation template.
- **`cdk diff`** compares the current synthesized template with the deployed stack.
- **`cdk deploy`** creates or updates the stack in AWS.
- **`cdk destroy`** removes the deployed stack and its resources, subject to deletion policies.
- **`cdk list`** shows the stacks in the app.

**Testing and validation**

CDK projects often include a `test/` directory for unit tests. Testing infrastructure code helps you catch regressions before deployment and verify that constructs synthesize as expected.

Common testing approaches include:

- Snapshot tests against synthesized templates.
- Assertions on resource counts and properties.
- Validation of stack outputs and dependencies.

A simple Jest-based test might look like:

```ts
import * as cdk from 'aws-cdk-lib';
import { Template } from 'aws-cdk-lib/assertions';
import { MyCdkAppStack } from '../lib/my-cdk-app-stack';

test('creates a bucket', () => {
  const app = new cdk.App();
  const stack = new MyCdkAppStack(app, 'TestStack');
  const template = Template.fromStack(stack);

  template.resourceCountIs('AWS::S3::Bucket', 1);
});
```

Testing is especially valuable when you refactor constructs or add conditional logic.

**Practical development tips**

A few habits make CDK TypeScript projects easier to maintain:

- Keep reusable infrastructure logic in `lib/` and app wiring in `bin/`.
- Use descriptive construct IDs so synthesized resources are easier to identify.
- Prefer small, composable constructs over large monolithic stacks.
- Run `cdk diff` before every deployment to review changes.
- Use environment-specific configuration for values that differ between dev, test, and prod.

A simple parameterization pattern can help you avoid hardcoding values:

```ts
const bucketName = process.env.BUCKET_NAME ?? 'my-default-bucket-name';
```

This is useful for local experimentation, but for production you usually prefer CDK context, stack props, or configuration files over ad hoc environment variables.

**Troubleshooting initialization issues**

Initialization problems usually come from missing tooling, mismatched versions, or AWS credential issues.

Common symptoms and causes:

- `cdk: command not found` means the CDK CLI is not installed or not on your `PATH`.
- TypeScript compilation errors often indicate missing dependencies or incorrect imports.
- `Unable to resolve AWS account to use` usually means credentials or default profile settings are missing.
- Bootstrap-related deployment failures often mean the target environment has not been prepared with `cdk bootstrap`.

Useful checks:

```bash
aws sts get-caller-identity
cdk doctor
npm ls aws-cdk-lib constructs
```

`cdk doctor` helps identify environment and configuration issues in the local CDK setup.

**Exam focus**

For exam purposes, understand the purpose of each generated file and the overall workflow from initialization to deployment.

You should be able to explain:

- Why `cdk init app --language typescript` is used.
- What `bin/` and `lib/` contain.
- How `cdk.json` controls app execution.
- Why `aws-cdk-lib` and `constructs` are required.
- When to run `cdk bootstrap`.
- How `cdk synth`, `cdk diff`, and `cdk deploy` fit into the development cycle.

A strong mental model is:

- You write TypeScript infrastructure code.
- CDK synthesizes it into CloudFormation.
- CloudFormation performs the actual AWS resource provisioning.

**Summary**

Initializing a CDK TypeScript application gives you a structured starting point for infrastructure as code. The generated project separates app entry logic from stack definitions, uses familiar Node.js tooling, and supports a fast synth-deploy feedback loop.

Focus on the scaffold, the core files, the CLI workflow, and the relationship between TypeScript code and synthesized CloudFormation. Those are the foundations you need before building more advanced CDK solutions.
