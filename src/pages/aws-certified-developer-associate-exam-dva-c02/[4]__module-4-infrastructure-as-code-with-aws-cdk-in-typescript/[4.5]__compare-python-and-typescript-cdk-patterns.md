## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Overview**

AWS Cloud Development Kit (<abbr title="Cloud Development Kit">CDK</abbr>) lets you define cloud infrastructure with familiar programming languages. For the exam, you should understand that Python and TypeScript CDK apps share the same underlying model, but the developer experience differs in project layout, typing, dependency management, and how you express constructs and properties.

The most important idea is that CDK code is still a programming application, not just a declarative template. You write code, instantiate constructs, and then run `cdk synth` to produce AWS CloudFormation templates. The language you choose affects ergonomics, not the resulting infrastructure model.

**Core similarity: the construct model**

Both Python and TypeScript use the same CDK concepts:

- **App** is the root of the CDK application.
- **Stack** groups resources that deploy together.
- **Construct** is the basic building block for reusable infrastructure.
- **L1 constructs** map closely to CloudFormation resources.
- **L2 constructs** provide higher-level abstractions with sensible defaults.
- **L3 patterns** combine multiple resources into opinionated solutions.

In both languages, you typically import from the same CDK libraries and compose resources in code. The main difference is how each language expresses types, imports, and runtime behavior.

**Project structure and bootstrap patterns**

A standard CDK project includes a few common files, but the exact layout varies by language and template version. For Python, you commonly see:

- `app.py`
- a package directory such as `my_cdk_app/`
- `requirements.txt`
- `README.md`
- `cdk.json`
- `.venv/` for a local virtual environment

For TypeScript, you commonly see:

- `bin/` with the app entry point
- `lib/` with stack definitions
- `package.json`
- `package-lock.json` or `yarn.lock`
- `tsconfig.json`
- `cdk.json`
- `README.md`

A CDK app usually defines the command to run in `cdk.json`. That command must be runnable for `cdk synth`, `cdk diff`, and `cdk deploy` to work.

Example `cdk.json` patterns:

```json
{
  "app": "python3 app.py"
}
```

```json
{
  "app": "npx ts-node bin/my-cdk-app.ts"
}
```

Some projects use a build step instead:

```json
{
  "app": "npm run build && npx cdk synth"
}
```

That pattern is not universal; many TypeScript templates run the app directly with `ts-node`, while others compile first. The exam may expect you to recognize both styles.

**TypeScript CDK patterns**

TypeScript is the most common CDK language because CDK itself is written in TypeScript and the type system is strong at design time. You usually get excellent editor support, autocomplete, and compile-time feedback.

A typical stack file looks like this:

```typescript
import { Stack, StackProps, Duration } from 'aws-cdk-lib';
import { Construct } from 'constructs';
import * as lambda from 'aws-cdk-lib/aws-lambda';

export class MyStack extends Stack {
  constructor(scope: Construct, id: string, props?: StackProps) {
    super(scope, id, props);

    new lambda.Function(this, 'Handler', {
      runtime: lambda.Runtime.NODEJS_20_X,
      handler: 'index.handler',
      code: lambda.Code.fromAsset('lambda'),
      timeout: Duration.seconds(10),
    });
  }
}
```

Key TypeScript characteristics:

- **Static typing** catches many mistakes before synthesis.
- **Interfaces** and optional properties make construct props explicit.
- **Enums** and union types help constrain valid values.
- **Generics** and type inference improve reuse in helper functions.
- **Compilation** may be required if your app uses `tsc` instead of `ts-node`.

TypeScript is often a good fit when you want strong IDE support and predictable refactoring. It is also useful when you want to model reusable infrastructure libraries with clear interfaces.

Common TypeScript workflow:

1. Install dependencies with `npm install`, `yarn`, or `pnpm`.
2. Run the app through `npx ts-node` or compile with `npm run build`.
3. Use `cdk synth` to generate the template.
4. Use `cdk diff` to inspect changes.
5. Use `cdk deploy` to apply the stack.

Example commands:

```bash
npm install
npx cdk synth
npx cdk diff
npx cdk deploy
```

If the app entry point in `cdk.json` depends on compiled output, you may need:

```bash
npm run build
npx cdk synth
```

**Python CDK patterns**

Python CDK apps are popular for developers who prefer Python syntax and ecosystem tools. The CDK Python binding uses generated classes and type hints, so you still get structure and validation, but the experience differs from TypeScript.

A typical Python stack file looks like this:

```python
from aws_cdk import Stack, Duration
from constructs import Construct
from aws_cdk import aws_lambda as lambda_

class MyStack(Stack):
    def __init__(self, scope: Construct, construct_id: str, **kwargs) -> None:
        super().__init__(scope, construct_id, **kwargs)

        lambda_.Function(
            self,
            "Handler",
            runtime=lambda_.Runtime.NODEJS_20_X,
            handler="index.handler",
            code=lambda_.Code.from_asset("lambda"),
            timeout=Duration.seconds(10),
        )
```

Key Python characteristics:

- **Dynamic runtime** means some errors appear when code executes, not only at compile time.
- **Type hints** improve editor support and can catch many mistakes early.
- **Keyword arguments** are common, but many CDK classes still expect typed property objects or specific named parameters depending on the construct.
- **Readable syntax** can make complex infrastructure easier to scan.

Python CDK apps often include a virtual environment. A common setup uses `.venv` and a local install of dependencies:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cdk synth
cdk deploy
```

On Windows, activation is typically:

```powershell
.venv\Scripts\activate.bat
```

Some projects use `uv`, `poetry`, or another dependency manager instead of plain `pip`. For exam prep, know that the exact install command depends on the project template, but the CDK workflow still centers on `synth`, `diff`, and `deploy`.

**Typing and validation differences**

TypeScript generally provides stronger compile-time guarantees. Python provides flexibility, but that flexibility can shift some mistakes to runtime.

Practical differences:

- **TypeScript** can reject invalid property names or incompatible values during compilation or editor analysis.
- **Python** can still catch many issues immediately when the construct is instantiated, especially when the generated bindings expose typed signatures.
- **Both languages** can surface deployment-time issues if the problem is only visible after synthesis or during CloudFormation deployment.

This means the contrast is not “TypeScript is safe and Python is unsafe.” Instead, TypeScript tends to catch more issues earlier through static analysis, while Python relies more on runtime execution and type-aware tooling.

Examples of issues you may catch early:

- Missing required construct arguments.
- Invalid enum values.
- Wrong property types.
- Incorrect import paths.
- Misnamed stack or construct parameters.

Examples of issues that may still appear later:

- Invalid resource combinations that pass local checks but fail CloudFormation validation.
- IAM permission problems.
- Missing assets or file paths.
- Environment-specific deployment failures.

**Construct props and language style**

CDK constructs often accept a scope, an identifier, and a props object or keyword arguments. The exact style depends on the language binding and the construct level.

TypeScript commonly uses a typed props object:

```typescript
new s3.Bucket(this, 'Bucket', {
  versioned: true,
  encryption: s3.BucketEncryption.S3_MANAGED,
});
```

Python often uses named parameters that mirror the same underlying props:

```python
s3.Bucket(
    self,
    "Bucket",
    versioned=True,
    encryption=s3.BucketEncryption.S3_MANAGED,
)
```

For many higher-level constructs, the Python binding still maps to a generated typed signature rather than arbitrary free-form keywords. That means you should not assume any keyword is accepted just because Python syntax allows it.

When you write reusable code, keep the following in mind:

- **TypeScript** favors explicit interfaces and object literals.
- **Python** favors readable named arguments and class-based composition.
- **Both** benefit from helper functions for repeated patterns.

**Imports and module organization**

TypeScript CDK projects usually import from `aws-cdk-lib` and `constructs`:

```typescript
import { Stack } from 'aws-cdk-lib';
import { Construct } from 'constructs';
```

Python CDK projects usually import from `aws_cdk` and `constructs`:

```python
from aws_cdk import Stack
from constructs import Construct
```

For service modules, TypeScript often uses namespace-style imports:

```typescript
import * as s3 from 'aws-cdk-lib/aws-s3';
```

Python often uses aliased module imports:

```python
from aws_cdk import aws_s3 as s3
```

This difference matters when you read examples or translate snippets between languages. The resource names and properties are usually the same conceptually, but the import syntax changes.

**Synthesis, deployment, and app execution**

The CDK CLI runs your app to produce a CloudFormation template. That means your language runtime must be available and your app entry point must execute successfully.

Typical workflow:

1. Ensure dependencies are installed.
2. Confirm the app command in `cdk.json` is correct.
3. Run `cdk synth` to generate the template.
4. Run `cdk diff` to compare against deployed state.
5. Run `cdk deploy` to create or update resources.

Example TypeScript app command:

```json
{
  "app": "npx ts-node bin/my-cdk-app.ts"
}
```

Example Python app command:

```json
{
  "app": "python3 app.py"
}
```

If the app command fails, synthesis fails. This is why language-specific runtime setup is part of the CDK workflow, not an optional extra.

**Practical Lambda example**

A simple Lambda function stack looks similar in both languages. The main differences are syntax and imports.

TypeScript:

```typescript
import { Stack, Duration } from 'aws-cdk-lib';
import { Construct } from 'constructs';
import * as lambda from 'aws-cdk-lib/aws-lambda';

export class LambdaStack extends Stack {
  constructor(scope: Construct, id: string) {
    super(scope, id);

    new lambda.Function(this, 'Fn', {
      runtime: lambda.Runtime.NODEJS_20_X,
      handler: 'index.handler',
      code: lambda.Code.fromAsset('lambda'),
      timeout: Duration.seconds(15),
    });
  }
}
```

Python:

```python
from aws_cdk import Stack, Duration
from constructs import Construct
from aws_cdk import aws_lambda as lambda_

class LambdaStack(Stack):
    def __init__(self, scope: Construct, construct_id: str, **kwargs) -> None:
        super().__init__(scope, construct_id, **kwargs)

        lambda_.Function(
            self,
            "Fn",
            runtime=lambda_.Runtime.NODEJS_20_X,
            handler="index.handler",
            code=lambda_.Code.from_asset("lambda"),
            timeout=Duration.seconds(15),
        )
```

Use a runtime constant that exists in the CDK version you are studying. Runtime enums evolve over time, so avoid memorizing a single constant as if it were universal across all versions.

**When to prefer each language**

Choose TypeScript when you want:

- Strong compile-time feedback.
- Tight integration with CDK examples and ecosystem tooling.
- Clear interfaces for reusable infrastructure libraries.
- Familiarity with JavaScript tooling such as `npm`, `ts-node`, and `tsc`.

Choose Python when you want:

- Python syntax and ecosystem alignment.
- Readable imperative code for infrastructure logic.
- Easy integration with Python-based automation or data tooling.
- A concise style for many common CDK patterns.

For the exam, the key is not to memorize a “best” language. You should understand how the same CDK concepts appear in both and how the workflow changes with the runtime.

**Exam-focused comparison**

| Area | TypeScript | Python |
|---|---|---|
| Typing | Strong static typing | Dynamic runtime with type hints |
| Common imports | `aws-cdk-lib`, `constructs` | `aws_cdk`, `constructs` |
| App execution | Often `ts-node` or compiled JS | Often `python3 app.py` in a venv |
| Error detection | Many issues caught at compile time | Many issues caught at runtime or by type hints |
| Syntax style | Object literals and interfaces | Named parameters and classes |
| Tooling | `npm`, `yarn`, `pnpm`, `tsc` | `pip`, `.venv`, `poetry`, `uv` |

**Troubleshooting tips**

- Check `cdk.json` first if `cdk synth` fails to start the app.
- Verify your virtual environment or Node.js toolchain is active.
- Confirm the construct import path matches the language binding.
- Inspect type errors before assuming the issue is with AWS.
- Use `cdk diff` to understand whether a change is logical or accidental.

> If the app cannot run locally, CDK cannot synthesize the stack. Fix the runtime and dependency setup before debugging the infrastructure itself.

**Quick translation mindset**

When moving between Python and TypeScript CDK examples, translate the structure first and the syntax second:

- `Stack` and `Construct` stay conceptually the same.
- Service modules map to the same AWS services.
- Props names usually stay aligned.
- Import style, typing style, and project tooling change.

A good habit is to compare the construct signature, not just the code shape. That helps you avoid copying syntax that looks valid but does not match the target language binding.
