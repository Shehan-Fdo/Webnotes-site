## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Production-style serverless architecture**

A production serverless application is more than a Lambda function behind an API. You design for deployment safety, observability, scaling behavior, browser integration, asynchronous workflows, and operational controls. For DVA-C02, the important pattern is to connect the pieces into a maintainable system rather than treating each service in isolation.

A typical end-to-end stack includes:

- **Frontend** hosted on Amazon S3 with Amazon CloudFront for global delivery
- **API layer** using Amazon API Gateway to expose HTTP endpoints
- **Compute** with AWS Lambda for business logic
- **Data** in Amazon DynamoDB for low-latency key-value access
- **Async integration** with Amazon EventBridge, Amazon SQS, or AWS Step Functions
- **Auth** with Amazon Cognito or IAM-based authorization
- **Observability** with Amazon CloudWatch, AWS X-Ray, and structured logs
- **Delivery** through AWS SAM, AWS CodePipeline, and AWS CodeBuild

The exam often tests whether you choose the right service for the right integration style. Synchronous request/response, asynchronous event routing, scheduled jobs, and long-running orchestration each have different tradeoffs.

**Reference architecture and request flow**

A production-style flow usually looks like this:

1. A user opens the SPA (single-page application) from CloudFront.
2. The browser calls API Gateway over HTTPS.
3. API Gateway authenticates the request and invokes Lambda.
4. Lambda reads or writes DynamoDB.
5. Lambda emits domain events to EventBridge for downstream consumers.
6. CloudWatch and X-Ray capture metrics, logs, and traces.
7. CodePipeline promotes changes through environments with controlled deployments.

A simple topology:

```text
Browser -> CloudFront -> S3 (SPA)
                 |
                 v
            API Gateway -> Lambda -> DynamoDB
                               |
                               v
                         EventBridge bus
                               |
               +---------------+----------------+
               v                                v
          Step Functions                    SQS / SNS
               |
               v
         Lambda / other tasks
```

**API Gateway in production**

API Gateway is the front door for serverless APIs. In production, you care about more than routing. You manage stages, deployments, throttling, custom domains, and browser access patterns.

Key concepts:

- **Deployment** is a snapshot of the API configuration.
- **Stage** is a named environment such as `dev`, `test`, or `prod`.
- **Stage variables** let you parameterize backend values per stage, such as Lambda aliases or downstream endpoints.
- **Custom domains** let you present a stable URL like `api.example.com`.
- **Usage plans** and API keys help meter and throttle usage for selected API types.
- **Throttling** protects backends from bursts and helps enforce quotas.

Practical stage and deployment example with AWS CLI:

```bash
aws apigateway create-deployment \
  --rest-api-id a1b2c3d4 \
  --stage-name prod \
  --description "Initial production deployment"
```

Stage variables can point to different Lambda aliases:

```json
{
  "lambdaAlias": "prod"
}
```

You can then reference the alias in integration settings so the same API stage can target a specific versioned backend. This is useful when you want to promote code independently from the API configuration.

For browser-based SPAs, CORS (Cross-Origin Resource Sharing) is essential. If your frontend is on `https://app.example.com` and API Gateway is on `https://api.example.com`, the API must return the correct CORS headers.

Typical CORS response headers:

```http
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Headers: Content-Type,Authorization,X-Amz-Date,X-Api-Key,X-Amz-Security-Token
Access-Control-Allow-Methods: GET,POST,PUT,DELETE,OPTIONS
Access-Control-Allow-Credentials: true
```

Important production considerations:

- **Preflight** requests use `OPTIONS` and must succeed before the browser sends the real request.
- **Credentials** require a specific origin; you cannot use `*` with credentialed requests.
- **Throttling** should be set to protect Lambda and DynamoDB from spikes.
- **Usage plans** are useful when you expose partner or public APIs with API keys.
- **Custom domains** simplify client configuration and certificate management.

> If you forget CORS on a browser-facing API, the backend may work in Postman but fail in the browser.

**Lambda runtime, scaling, and concurrency**

Lambda is the compute engine for most serverless backends. In production, you need to understand how concurrency affects latency, cost, and throttling.

Important concurrency controls:

- **Reserved concurrency** sets aside capacity for a function and caps its maximum concurrent executions.
- **Provisioned concurrency** keeps execution environments initialized to reduce cold starts.
- **Account concurrency limits** apply across all Lambda functions in the account and Region.
- **Burst traffic** can cause throttling if concurrency is exhausted.

Use reserved concurrency when you want to protect downstream systems or guarantee capacity for a critical function. Use provisioned concurrency when latency matters and you need predictable startup performance.

Example CLI commands:

```bash
aws lambda put-function-concurrency \
  --function-name order-handler \
  --reserved-concurrent-executions 20
```

```bash
aws lambda put-provisioned-concurrency-config \
  --function-name order-handler \
  --qualifier prod \
  --provisioned-concurrent-executions 10
```

Operational implications:

- **Reserved concurrency** can starve other functions if set too aggressively.
- **Provisioned concurrency** adds cost but reduces cold starts.
- **Throttling** at Lambda may surface as 429 or 5xx responses upstream.
- **Async invocations** may retry automatically, so design idempotent handlers.

A practical pattern is to publish versions and route traffic through aliases:

```bash
aws lambda publish-version --function-name order-handler
aws lambda update-alias \
  --function-name order-handler \
  --name prod \
  --function-version 12
```

This supports safe promotion and rollback without changing the API integration target.

**Data layer with DynamoDB**

DynamoDB is the common choice for serverless persistence because it scales without server management and integrates well with Lambda. The exam expects you to understand access patterns, not just table creation.

Design principles:

- Model around **queries**, not joins.
- Use a **partition key** that distributes load well.
- Add a **sort key** when you need ordered items or multiple item types.
- Use **GSIs** (Global Secondary Indexes) for alternate access patterns.
- Prefer **conditional writes** for optimistic concurrency and idempotency.

Example table creation:

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=orderId,AttributeType=S \
  --key-schema AttributeName=orderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

Example conditional write in application code:

```python
import boto3
dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("Orders")

table.put_item(
    Item={"orderId": "123", "status": "PENDING"},
    ConditionExpression="attribute_not_exists(orderId)"
)
```

Why this matters:

- **Conditional writes** prevent duplicate creation in retried workflows.
- **On-demand capacity** simplifies unpredictable traffic patterns.
- **GSIs** support lookups like `customerId` or `status` without scanning.
- **Streams** can trigger downstream processing when items change.

**EventBridge for event routing and decoupling**

Amazon EventBridge is the right choice when you want to publish domain events and let multiple consumers react independently. It is especially useful for decoupling services, integrating SaaS or AWS services, and scheduling serverless tasks.

Use EventBridge when:

- You want to **decouple** producers from consumers.
- You need **fan-out** to multiple targets without direct service coupling.
- You want to route events based on content using rules.
- You need **scheduled** or cron-like triggers for serverless jobs.
- You want a central event bus for domain events such as `OrderCreated` or `PaymentFailed`.

Example event flow:

- Lambda writes an order to DynamoDB.
- Lambda publishes an event to EventBridge.
- EventBridge routes the event to:
  - a fulfillment Lambda
  - an analytics pipeline
  - a notification workflow

Example event publication:

```python
import boto3, json
events = boto3.client("events")

events.put_events(
    Entries=[
        {
            "Source": "app.orders",
            "DetailType": "OrderCreated",
            "Detail": json.dumps({"orderId": "123", "customerId": "c-9"}),
            "EventBusName": "default"
        }
    ]
)
```

Example rule pattern:

```json
{
  "source": ["app.orders"],
  "detail-type": ["OrderCreated"]
}
```

Scheduled integration example:

```json
{
  "source": ["aws.events"],
  "detail-type": ["Scheduled Event"]
}
```

Practical uses:

- **Nightly cleanup** jobs
- **Periodic reconciliation** tasks
- **Event-driven workflows** without direct Lambda-to-Lambda calls
- **Cross-service integration** where producers should not know consumers

EventBridge is often preferable to direct invocation when you want loose coupling and future extensibility. If you need guaranteed ordered processing or explicit buffering, SQS may be a better fit. If you need orchestration with branching and retries, Step Functions is stronger.

**Asynchronous messaging and orchestration**

Not every workflow should be synchronous. Production systems often combine EventBridge, SQS, SNS, and Step Functions.

Choose based on behavior:

- **SQS** for buffering, backpressure, and reliable queue-based processing
- **SNS** for simple pub/sub fan-out to multiple subscribers
- **EventBridge** for event routing, filtering, and SaaS/AWS integration
- **Step Functions** for multi-step orchestration, retries, branching, and human-readable workflow state

A common pattern:

- API Gateway accepts a request.
- Lambda validates and stores the request.
- Lambda sends a message to SQS or emits an EventBridge event.
- A worker Lambda processes the message asynchronously.
- Step Functions coordinates multi-step business logic when the workflow becomes complex.

Step Functions is especially useful when you need:

- explicit state transitions
- retries with backoff
- parallel branches
- error handling and compensation
- long-running workflows that should not live inside one Lambda invocation

**Observability with CloudWatch and X-Ray**

Production serverless systems need logs, metrics, alarms, and traces. CloudWatch gives you the operational baseline, while X-Ray helps you understand request paths across services.

CloudWatch responsibilities:

- **Logs** from Lambda, API Gateway, and other services
- **Metrics** such as invocation count, errors, duration, throttles, and iterator age
- **Alarms** for thresholds like elevated 5xx errors or Lambda throttling
- **Dashboards** for service health and business KPIs

X-Ray responsibilities:

- **Tracing** a request across API Gateway, Lambda, DynamoDB, and downstream calls
- **Service maps** showing latency and error hotspots
- **Subsegments** for SDK calls inside Lambda
- **Annotations** for searchable metadata such as `orderId` or `tenantId`

To enable tracing for Lambda, you typically turn on active tracing and ensure the function has permission to write trace data. For API Gateway, enable X-Ray tracing on the stage so the request can be correlated from the edge to the function.

Example Lambda handler with trace-friendly logging:

```python
import json
import logging

logger = logging.getLogger()
logger.setLevel(logging.INFO)

def lambda_handler(event, context):
    logger.info(json.dumps({
        "message": "request received",
        "requestId": context.aws_request_id,
        "path": event.get("path")
    }))
    return {
        "statusCode": 200,
        "headers": {"Content-Type": "application/json"},
        "body": json.dumps({"ok": True})
    }
```

What to look for in traces:

- **High latency at API Gateway** may indicate auth, integration, or backend delay.
- **Long Lambda duration** may point to inefficient code or downstream calls.
- **DynamoDB subsegment errors** may indicate throttling or bad keys.
- **Missing segments** often mean tracing is not enabled consistently across services.

Useful CloudWatch Logs Insights example:

```sql
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 20
```

**CI/CD with SAM, CodeBuild, and CodePipeline**

A production serverless app should be built and deployed automatically. AWS SAM simplifies packaging and deployment, while CodeBuild and CodePipeline provide repeatable delivery.

A common pipeline flow:

1. Source code is pushed to a repository.
2. CodePipeline starts on commit.
3. CodeBuild runs tests, linting, and packaging.
4. SAM builds the application artifacts.
5. CloudFormation deploys to a test or staging environment.
6. Automated checks run.
7. The pipeline promotes to production with approval or automated gates.

Example SAM build and deploy commands:

```bash
sam build
sam deploy --guided
```

Example buildspec for CodeBuild:

```yaml
version: 0.2
phases:
  install:
    runtime-versions:
      python: 3.12
  build:
    commands:
      - pip install -r requirements.txt
      - pytest
      - sam build
      - sam package --s3-bucket my-artifact-bucket --output-template-file packaged.yaml
artifacts:
  files:
    - packaged.yaml
```

Why this matters:

- **CodeBuild** gives you isolated, repeatable builds.
- **CodePipeline** automates promotion across environments.
- **SAM** keeps infrastructure and application code together.
- **Automated tests** reduce deployment risk and catch regressions early.

For production, use separate environments and parameterize values such as table names, API endpoints, and feature flags. Keep secrets out of source control and inject them through environment variables, AWS Systems Manager Parameter Store, or AWS Secrets Manager.

**Safe deployments and rollback strategy**

Serverless deployments should be reversible. Lambda versions, aliases, and deployment strategies help you reduce blast radius.

Common safe deployment techniques:

- **Versioned Lambda aliases** for controlled promotion
- **Canary deployments** to shift a small percentage of traffic first
- **Linear deployments** to gradually increase traffic
- **Rollback alarms** to revert on elevated errors or latency
- **Immutable artifacts** so the same build is promoted through environments

Example alias-based promotion:

```bash
aws lambda update-alias \
  --function-name order-handler \
  --name prod \
  --function-version 13
```

If you use CodeDeploy with Lambda, you can shift traffic gradually and attach CloudWatch alarms to trigger rollback. This is useful when a new version changes performance characteristics or downstream behavior.

**Frontend integration and browser concerns**

A production SPA usually needs a stable hosting and API integration story. S3 and CloudFront are the standard pair for static hosting, while API Gateway handles backend calls.

Important frontend concerns:

- **CORS** must allow the browser origin.
- **Authentication tokens** should be stored carefully; prefer secure browser patterns and short-lived tokens.
- **Custom domains** make frontend and API URLs predictable.
- **Cache behavior** should be tuned so static assets can be cached aggressively while API responses are not cached unless intended.

Example CloudFront + S3 pattern:

- `app.example.com` points to CloudFront
- CloudFront serves the SPA from S3
- `api.example.com` points to API Gateway
- The SPA calls `api.example.com` with `fetch()`

Example browser request:

```javascript
fetch("https://api.example.com/orders", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": `Bearer ${token}`
  },
  body: JSON.stringify({ itemId: "sku-1", quantity: 2 })
});
```

If the API uses Cognito authorizers or JWT-based auth, ensure the browser sends the token consistently and that preflight requests are allowed.

**Authentication and authorization**

Serverless apps often use Amazon Cognito for user authentication or IAM for service-to-service access. API Gateway can integrate with both.

Common patterns:

- **Cognito User Pools** for end-user sign-in and JWT validation
- **IAM authorization** for AWS-native callers
- **Lambda authorizers** when you need custom token logic
- **Resource policies** for restricting access by source or account

Choose Cognito when you need user sign-up, sign-in, and token issuance. Choose IAM when the caller is another AWS service or a trusted AWS principal. Use Lambda authorizers only when built-in options do not fit the security model.

**End-to-end deployment example**

A production-style stack can be expressed in one SAM template with frontend, API, auth, data, and async components. The exact implementation varies, but the structure is consistent.

Example skeleton:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Transform: AWS::Serverless-2016-10-31

Parameters:
  StageName:
    Type: String
    Default: prod

Resources:
  OrdersTable:
    Type: AWS::DynamoDB::Table
    Properties:
      BillingMode: PAY_PER_REQUEST
      AttributeDefinitions:
        - AttributeName: orderId
          AttributeType: S
      KeySchema:
        - AttributeName: orderId
          KeyType: HASH

  OrdersFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: src/orders/
      Handler: app.lambda_handler
      Runtime: python3.12
      Timeout: 10
      MemorySize: 256
      Environment:
        Variables:
          TABLE_NAME: !Ref OrdersTable
      Policies:
        - DynamoDBCrudPolicy:
            TableName: !Ref OrdersTable
      Events:
        ApiEvent:
          Type: Api
          Properties:
            Path: /orders
            Method: post

  EventsBus:
    Type: AWS::Events::EventBus
    Properties:
      Name: app-bus

  OrderCreatedRule:
    Type: AWS::Events::Rule
    Properties:
      EventBusName: !Ref EventsBus
      EventPattern:
        source:
          - app.orders
        detail-type:
          - OrderCreated
      Targets:
        - Arn: !GetAtt FulfillmentFunction.Arn
          Id: FulfillmentTarget

  FulfillmentFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: src/fulfillment/
      Handler: app.lambda_handler
      Runtime: python3.12

Outputs:
  ApiUrl:
    Value: !Sub "https://${ServerlessRestApi}.execute-api.${AWS::Region}.amazonaws.com/${StageName}/"
```

This pattern shows how the pieces fit together:

- API Gateway exposes the API.
- Lambda handles business logic.
- DynamoDB stores application state.
- EventBridge routes domain events.
- SAM manages the infrastructure as code.

**Operational checklist for exam scenarios**

When you see a serverless architecture question, check these areas:

- **Integration type**
  - Synchronous request/response
  - Asynchronous event routing
  - Scheduled execution
  - Long-running orchestration

- **Deployment safety**
  - Lambda versions and aliases
  - Staged API deployments
  - Canary or linear rollout
  - Rollback alarms

- **Performance**
  - Reserved concurrency
  - Provisioned concurrency
  - API Gateway throttling
  - DynamoDB capacity mode and access patterns

- **Browser integration**
  - CORS headers
  - Custom domains
  - Auth token handling
  - Cache behavior

- **Observability**
  - CloudWatch logs and metrics
  - X-Ray traces across services
  - Alarms on errors, latency, and throttles

- **Delivery**
  - SAM build and deploy
  - CodeBuild test/build steps
  - CodePipeline promotion flow
  - Environment-specific parameters

**Common exam traps**

- Choosing direct Lambda-to-Lambda calls when EventBridge or SQS is a better decoupling choice.
- Forgetting that browser clients need CORS, especially for `OPTIONS` preflight.
- Using one Lambda version for all traffic without aliases or rollback strategy.
- Ignoring throttling and concurrency limits when a function is behind a bursty API.
- Treating CloudWatch logs as enough observability without tracing the full request path.
- Using scans in DynamoDB instead of modeling the access pattern.
- Deploying manually when the scenario clearly calls for CI/CD automation.

**Practical deployment commands**

A minimal local-to-cloud workflow with SAM:

```bash
sam validate
sam build
sam deploy --stack-name serverless-capstone --guided
```

If you need to inspect logs during testing:

```bash
sam logs -n OrdersFunction --stack-name serverless-capstone --tail
```

If you need to invoke a function directly:

```bash
aws lambda invoke \
  --function-name order-handler \
  --payload '{"orderId":"123"}' \
  response.json
```

**Summary of what to remember**

A production-style serverless application combines API Gateway, Lambda, data storage, async messaging, observability, and automated delivery. For DVA-C02, focus on choosing the right service for each integration pattern and on the operational details that make the system reliable in production.

The highest-value ideas are:

- **API Gateway** manages stages, deployments, throttling, custom domains, and CORS.
- **Lambda** needs concurrency planning, versioning, and alias-based promotion.
- **EventBridge** is ideal for decoupled event routing and scheduled integrations.
- **CloudWatch and X-Ray** provide logs, metrics, alarms, and end-to-end traces.
- **SAM, CodeBuild, and CodePipeline** enable repeatable build-test-deploy automation.
- **DynamoDB** should be modeled around access patterns and idempotent writes.
