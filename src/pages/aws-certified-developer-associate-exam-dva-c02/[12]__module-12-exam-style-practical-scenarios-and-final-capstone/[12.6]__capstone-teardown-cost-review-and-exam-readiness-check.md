## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Capstone purpose and final review mindset**

A capstone teardown is the last operational step after hands-on labs or a project build. You remove resources, confirm that nothing continues to incur charges, and use the teardown as a final validation of your AWS service knowledge. For the DVA-C02 exam, this matters because many questions test whether you understand the lifecycle of serverless, container, storage, and monitoring resources, not just how to create them.

The goal is to leave the environment in a clean, auditable state while reinforcing the services and patterns you used during the build.

- **Teardown** means deleting or disabling resources you no longer need.
- **Cost review** means checking what is still running, what is billable, and what may have hidden charges.
- **Exam readiness** means confirming you can explain service behavior, tradeoffs, and troubleshooting steps without relying on notes.

**Teardown strategy and order of operations**

You should remove resources in an order that avoids dependency errors. Start from the top of the stack, where user-facing and ephemeral resources usually depend on lower-level infrastructure.

1. Delete or scale down application-facing resources first.
   - Remove load balancers, API Gateway stages, CloudFront distributions, and application autoscaling targets if they are no longer needed.
   - Stop or delete compute resources such as EC2 instances, ECS tasks, EKS workloads, Lambda event sources, and batch jobs.
2. Remove data and integration resources next.
   - Delete queues, topics, streams, and event rules after confirming no active producers or consumers depend on them.
   - Clean up temporary S3 buckets, DynamoDB tables, RDS snapshots, and EFS file systems only after you confirm you no longer need the data.
3. Delete networking and security resources last.
   - Remove NAT gateways, Elastic IPs, VPC endpoints, security groups, route tables, and custom VPCs after dependent resources are gone.
   - Release reserved capacity or dedicated resources if you created them for the lab.
4. Verify that no orphaned resources remain.
   - Check for unattached EBS volumes, idle load balancers, stale snapshots, unused IAM roles, and log groups with retention you no longer need.

> Be careful with shared or persistent resources. Some items, such as CloudWatch log groups, snapshots, and S3 buckets with versioning, can continue to generate storage charges even after the main application is deleted.

**Common teardown targets by service**

Different AWS services require different cleanup actions. Some resources disappear automatically when their parent is deleted, while others persist and must be removed manually.

- **Lambda**
  - Delete event source mappings if they were created separately.
  - Remove aliases, versions, and provisioned concurrency if they are no longer needed.
  - Confirm that CloudWatch log groups are not left behind unintentionally.
- **API Gateway**
  - Delete stages, deployments, custom domains, and usage plans if they were created for the lab.
  - Remove Lambda permissions that were added to allow invocation from API Gateway.
- **S3**
  - Empty versioned buckets before deletion.
  - Remove multipart uploads if they are incomplete and still consuming storage.
- **DynamoDB**
  - Delete tables only after confirming that streams, backups, and global tables are not needed.
  - Remember that on-demand tables and provisioned tables both stop billing only after deletion.
- **RDS**
  - Decide whether to create a final snapshot before deletion.
  - Delete read replicas and automated backups if they are no longer required.
- **ECS and EKS**
  - Scale services to zero, delete services, then delete clusters or node groups.
  - Remove task definitions, load balancers, and target groups if they were created only for the lab.
- **CloudFormation**
  - Delete the stack rather than deleting individual resources when the stack owns the environment.
  - Review stack policies and termination protection before teardown.

**Practical teardown commands**

Use the AWS CLI (Command Line Interface) when you want repeatable cleanup or when the console is too slow for many resources. Always confirm the target resource before deletion.

```bash
aws s3 rm s3://my-lab-bucket --recursive
aws s3api delete-bucket --bucket my-lab-bucket

aws dynamodb delete-table --table-name MyLabTable

aws lambda delete-function --function-name my-lab-function

aws apigateway delete-rest-api --rest-api-id a1b2c3d4e5

aws cloudformation delete-stack --stack-name my-lab-stack
```

For versioned S3 buckets, you must remove versions and delete markers before deleting the bucket.

```bash
aws s3api list-object-versions --bucket my-lab-bucket
aws s3api delete-objects --bucket my-lab-bucket --delete file://delete.json
```

A simple `delete.json` file can look like this:

```json
{
  "Objects": [
    { "Key": "app.zip", "VersionId": "111111" },
    { "Key": "logs/2026/04/01.txt", "VersionId": "222222" }
  ],
  "Quiet": true
}
```

**Cost review and billing awareness**

Cost review is not just about deleting resources. You also need to understand which services bill by time, request, storage, or data transfer. This helps you identify the most common sources of surprise charges in a lab environment.

- **Compute charges**
  - EC2 bills while instances run, even if the workload is idle.
  - Lambda bills by requests and duration, so frequent invocations or long-running functions can add up.
  - ECS on EC2 bills for the underlying instances; ECS on Fargate bills for vCPU and memory while tasks run.
- **Storage charges**
  - S3 bills for stored objects, storage class, requests, lifecycle transitions, and retrieval in some classes.
  - EBS bills for provisioned volume size, not just used data.
  - RDS bills for allocated storage, backups, and snapshots.
- **Network charges**
  - NAT gateways can be expensive because they bill per hour and per GB processed.
  - Data transfer out to the internet is usually billable.
  - Cross-AZ traffic can also create charges in some architectures.
- **Monitoring and logging charges**
  - CloudWatch Logs bill for ingestion, storage, and some query activity.
  - High-volume application logs can become a hidden cost if retention is not set.
- **Managed service extras**
  - Snapshots, backups, and replicas often persist after the primary resource is deleted.
  - Provisioned concurrency, reserved capacity, and dedicated endpoints can continue billing until removed.

**How to review costs effectively**

Use a layered approach: first identify the biggest spenders, then drill into service-level details, then confirm whether the spend is expected.

1. Open the Billing and Cost Management console.
2. Review the current month’s spend and compare it with the previous period.
3. Check the Cost Explorer for service-level and daily trends.
4. Filter by linked account, region, or tag if your lab uses cost allocation tags.
5. Inspect usage types for clues such as NAT gateway hours, data transfer, or storage retention.
6. Confirm that any remaining spend matches a known resource.

A useful CLI check for billing-related metadata is limited, but you can still inspect resource inventories and tags to correlate costs with active resources.

```bash
aws resourcegroupstaggingapi get-resources \
  --tag-filters Key=Project,Values=lab \
  --output table
```

If you use tags consistently, cost allocation becomes much easier. A simple tagging strategy helps you identify lab resources quickly.

```json
{
  "Project": "dva-c02-lab",
  "Owner": "student",
  "Environment": "training",
  "Expires": "2026-04-26"
}
```

**Hidden cost patterns to watch for**

Some resources are easy to forget because they are not the main application component.

- **NAT gateways** remain billable even when no traffic flows.
- **Elastic IPs** can incur charges when allocated but unused.
- **CloudWatch log groups** continue storing logs until retention expires or you delete them.
- **EBS volumes** remain billable after instance termination unless configured for deletion on termination.
- **RDS snapshots** and automated backups can outlive the database.
- **S3 versioning** can keep old object versions and delete markers.
- **Interface VPC endpoints** bill hourly and by data processed.
- **Provisioned concurrency** for Lambda continues until you remove it.

**Exam readiness check: what you should be able to explain**

The final readiness check is about service behavior, not memorization alone. You should be able to reason through architecture, permissions, deployment, and troubleshooting scenarios.

- **Identity and access**
  - Explain the difference between IAM users, roles, and policies.
  - Describe when to use resource-based policies versus identity-based policies.
  - Understand least privilege and how to scope permissions for Lambda, API Gateway, S3, DynamoDB, and EventBridge.
- **Application deployment**
  - Compare ZIP deployment packages, container images, and infrastructure as code.
  - Explain how CloudFormation, SAM, or CDK manage lifecycle and rollback.
  - Know how aliases and versions support safe Lambda deployments.
- **Event-driven design**
  - Distinguish between synchronous and asynchronous invocation.
  - Explain retries, dead-letter queues (DLQs), and idempotency.
  - Understand EventBridge rules, SQS queues, SNS topics, and stream processing with DynamoDB Streams or Kinesis.
- **Observability**
  - Use CloudWatch metrics, logs, and alarms to diagnose failures.
  - Know how X-Ray tracing helps identify latency and downstream errors.
  - Understand how to correlate request IDs across services.
- **Data and storage**
  - Choose the right storage service for durability, access pattern, and consistency needs.
  - Explain S3 event notifications, DynamoDB capacity modes, and RDS backup behavior.
- **Networking**
  - Understand VPC basics, security groups, route tables, subnets, and NAT gateways.
  - Know when a Lambda function needs VPC access and what that changes operationally.
- **Security**
  - Recognize common secrets-management patterns with AWS Secrets Manager and Systems Manager Parameter Store.
  - Understand encryption at rest and in transit.
  - Know how KMS (Key Management Service) affects access and service integration.

**High-value exam scenarios to rehearse**

You should practice explaining the “why” behind each choice. These are common patterns that show up in developer-focused AWS questions.

- A Lambda function cannot access an RDS database in a private subnet.
  - Check VPC configuration, security groups, subnet routing, and whether the function has the right IAM permissions.
- An S3 upload triggers a function multiple times.
  - Consider event filtering, duplicate notifications, and idempotent processing.
- A deployment fails after a new Lambda version is published.
  - Review alias routing, permissions, environment variables, and downstream dependencies.
- A queue grows while consumers appear healthy.
  - Inspect visibility timeout, batch size, error handling, and dead-letter queue settings.
- Costs are unexpectedly high after a lab.
  - Look for NAT gateways, log retention, snapshots, provisioned concurrency, and idle compute.

**Quick self-check checklist**

Use this checklist before you consider the capstone complete.

- [ ] All lab resources are deleted or intentionally retained.
- [ ] S3 buckets are empty, including versions and delete markers.
- [ ] Lambda functions, aliases, and event source mappings are removed if no longer needed.
- [ ] API Gateway stages, custom domains, and usage plans are deleted.
- [ ] DynamoDB tables, streams, and backups are reviewed.
- [ ] RDS instances, replicas, snapshots, and automated backups are accounted for.
- [ ] ECS/EKS services are scaled down and clusters are removed if appropriate.
- [ ] NAT gateways, Elastic IPs, and interface endpoints are deleted if unused.
- [ ] CloudWatch log retention is set appropriately or logs are removed.
- [ ] Billing and Cost Explorer show no unexpected active spend.
- [ ] You can explain the architecture and tradeoffs without notes.

**Practical exam-day habits**

On exam day, read each scenario for the service boundary first. Many wrong answers are technically possible but operationally inefficient or too broad. Look for the smallest change that solves the problem while preserving security and reliability.

- Prefer managed services when the question emphasizes reduced operational overhead.
- Prefer event-driven decoupling when the question emphasizes scalability or resilience.
- Prefer IAM roles over long-lived credentials.
- Prefer least privilege and scoped resource policies.
- Prefer idempotent processing for retries and duplicate events.
- Prefer CloudWatch and X-Ray when the question asks for diagnosis or observability.

**Final readiness summary**

If you can tear down a lab cleanly, explain where costs come from, and reason through common AWS developer scenarios, you are in good shape for DVA-C02. The exam rewards practical judgment: choosing the right service, the right permission model, and the right operational control for the situation.
