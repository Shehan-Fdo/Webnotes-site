## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Scenario overview**

AWS X-Ray helps you trace requests across distributed systems so you can identify latency, errors, and downstream dependencies. In exam scenarios, visibility problems usually come from one of four places: tracing is not enabled, sampling is too aggressive, the application is not instrumented correctly, or IAM permissions block the daemon, service, or integration from publishing trace data.

A useful mental model is that the application generates trace metadata, the X-Ray daemon or service integration forwards it, and X-Ray stores and visualizes the segments and subsegments. If any link in that chain is broken, the service map and trace list can look incomplete or empty.

**Core X-Ray concepts**

X-Ray uses a few terms that often appear in troubleshooting questions:

- **Segment**: the top-level record for a request handled by a service.
- **Subsegment**: a child record that captures downstream calls such as SDK requests or database access.
- **Trace**: the full end-to-end path across services.
- **Service map**: the visual graph of traced services and dependencies.
- **Sampling**: the mechanism that limits how many requests are traced to control cost and overhead.

On the exam, distinguish between missing traces and missing permissions. A service can be healthy and still appear absent from the service map if it is not sampled, not actively traced, or not sending data in the expected format.

**How X-Ray data flows**

In many self-managed environments, your application sends trace data to the X-Ray daemon over UDP on `127.0.0.1:2000`, and the daemon uses the instance role or task role to call the X-Ray service API. This distinction matters:

- The **application** usually does not need direct permission to call `xray:PutTraceSegments` if it only talks to the local daemon.
- The **daemon or runtime integration** needs permission to publish trace data to AWS.
- In containerized or EC2-based setups, the **instance profile** or **task role** is commonly the identity that matters.

> Do not assume the application role itself must always have `AWSXRayDaemonWriteAccess`. In many architectures, the daemon uses the role to publish trace data, while the app only emits UDP packets locally.

A typical flow looks like this:

```text
App code -> local X-Ray daemon -> AWS X-Ray service -> service map / traces
```

**IAM permissions to check**

When traces do not appear, verify the permissions attached to the role used by the daemon or managed integration. Common permissions include:

- `xray:PutTraceSegments`
- `xray:PutTelemetryRecords`
- `xray:GetSamplingRules`
- `xray:GetSamplingTargets`
- `xray:GetSamplingStatisticSummaries`

The AWS managed policy `AWSXRayDaemonWriteAccess` is often used for the daemon or runtime component that publishes trace data. It is not always the correct policy to attach to the application role itself.

Practical checks:

```bash
aws iam list-attached-role-policies --role-name MyInstanceRole
aws iam get-role --role-name MyInstanceRole
```

If you need to inspect whether the role can publish traces, simulate the permission:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/MyInstanceRole \
  --action-names xray:PutTraceSegments xray:PutTelemetryRecords
```

**Active tracing and service configuration**

X-Ray visibility also depends on whether the service is configured to trace requests.

- **EC2 / ECS / EKS / custom hosts**: you usually need application instrumentation plus a daemon or sidecar.
- **Lambda**: X-Ray integration is built in when active tracing is enabled; you do not install the daemon manually.
- **API Gateway REST APIs**: tracing is enabled at the stage level.
- **Other AWS services**: some services support X-Ray natively, but the exact setup varies.

For exam questions, confirm whether the service is actually configured for tracing before blaming IAM.

Example: enable active tracing for a Lambda function:

```bash
aws lambda update-function-configuration \
  --function-name my-function \
  --tracing-config Mode=Active
```

Example: enable tracing on an API Gateway REST API stage:

```bash
aws apigateway update-stage \
  --rest-api-id a1b2c3d4e5 \
  --stage-name prod \
  --patch-operations op=replace,path=/tracingEnabled,value=true
```

> The API Gateway command above applies to REST APIs. HTTP APIs use different tracing and stage configuration behavior, so do not reuse this command blindly.

**Instrumentation patterns**

Instrumentation is the code or middleware that creates segments and subsegments. The exact approach depends on the runtime.

For Node.js, simply calling `openSegment()` and `closeSegment()` is not enough in every app. You need the X-Ray SDK configured correctly, and the request lifecycle must be supported by the middleware or framework integration.

A more realistic Express setup looks like this:

```javascript
const express = require('express');
const AWSXRay = require('aws-xray-sdk');

const app = express();

AWSXRay.config([AWSXRay.plugins.EC2Plugin]);
AWSXRay.middleware.enableDynamicNaming('*.example.com');

app.use(AWSXRay.express.openSegment('myApp'));

app.get('/orders/:id', (req, res) => {
  const segment = AWSXRay.getSegment();
  const subsegment = segment.addNewSubsegment('dynamodb-call');

  // Simulate downstream work
  subsegment.close();
  res.json({ ok: true });
});

app.use(AWSXRay.express.closeSegment());

app.listen(3000);
```

Key points:

- **Middleware order** matters in Express.
- The request must pass through the X-Ray middleware to create and close segments correctly.
- You often need additional plugins or context propagation for AWS SDK calls.
- If you use async frameworks or custom servers, verify that the SDK supports the request lifecycle you are using.

For Python, Java, .NET, and other runtimes, the same principle applies: the SDK or agent must wrap the request path so trace context survives across calls.

**Daemon and runtime deployment patterns**

In EC2 or container environments, the daemon is often deployed as a sidecar or local process.

Example Docker Compose pattern:

```yaml
services:
  app:
    image: my-app:latest
    environment:
      AWS_XRAY_DAEMON_ADDRESS: xray-daemon:2000
    depends_on:
      - xray-daemon

  xray-daemon:
    image: amazon/aws-xray-daemon
    command: ["-o"]
    ports:
      - "2000:2000/udp"
```

Notes:

- The app sends trace data to the daemon address.
- The daemon needs network access to AWS X-Ray endpoints.
- The role attached to the host, task, or pod identity is what usually authorizes publishing.

In ECS, you commonly use a task role for the application task and a sidecar daemon or an AWS-managed integration depending on the platform. In EKS, you may use a DaemonSet or sidecar plus IAM Roles for Service Accounts (IRSA) when appropriate.

**Troubleshooting checklist**

Use a layered approach so you do not fix the wrong component.

1. Confirm that tracing is enabled for the service.
2. Confirm that the application is instrumented correctly.
3. Confirm that the daemon or managed integration can reach X-Ray.
4. Confirm that the role used to publish traces has the required permissions.
5. Confirm that sampling is not excluding most requests.
6. Confirm that the region is correct and matches the X-Ray console you are viewing.

Common symptoms and likely causes:

- **Empty service map**
  - Tracing is disabled.
  - Sampling is too low.
  - The daemon cannot publish segments.
  - The app is instrumented incorrectly.
  - You are looking in the wrong region.
- **Traces exist but are incomplete**
  - Subsegments are not created for downstream calls.
  - Context is lost across threads, async tasks, or middleware boundaries.
  - SDK instrumentation is partial.
- **Permission errors in logs**
  - The daemon or runtime role lacks `xray:PutTraceSegments` or related actions.
  - The wrong role is attached to the instance, task, or service account.

**Sampling behavior**

Sampling reduces overhead by tracing only a subset of requests. This is normal and expected.

- The default sampling rules may trace only a fraction of requests.
- High traffic services may show fewer traces than you expect.
- A low trace count does not automatically mean a failure.

If you need to verify whether sampling is the issue, inspect the active rules:

```bash
aws xray get-sampling-rules
aws xray get-sampling-targets
aws xray get-sampling-statistics
```

If the service is not appearing at all, sampling is only one possible cause. Also check tracing enablement, permissions, and instrumentation.

**CLI and configuration examples**

Enable tracing in a Lambda deployment workflow:

```bash
aws lambda update-function-configuration \
  --function-name orders-api \
  --tracing-config Mode=Active
```

Check whether a role has X-Ray permissions:

```bash
aws iam list-attached-role-policies --role-name orders-task-role
```

Inspect recent traces:

```bash
aws xray get-trace-summaries \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T23:59:59Z
```

Example environment variable for a local daemon address:

```bash
export AWS_XRAY_DAEMON_ADDRESS=127.0.0.1:2000
```

**Exam traps and how to avoid them**

- **Do not confuse the app role with the daemon role**: the app may only send UDP locally.
- **Do not assume the daemon is always required**: Lambda and some managed services integrate without a manually installed daemon.
- **Do not reuse REST API commands for HTTP APIs**: API Gateway tracing configuration differs by API type.
- **Do not blame IAM first**: missing traces can also come from sampling or disabled active tracing.
- **Do not assume `openSegment()` alone is enough**: middleware and lifecycle support are required for correct context handling.

**Quick decision guide**

If you see no traces:

- Check active tracing first.
- Check the region.
- Check instrumentation and middleware.
- Check the daemon or sidecar path.
- Check the publishing role permissions.
- Check sampling rules.

If you see partial traces:

- Check subsegment creation.
- Check async context propagation.
- Check SDK instrumentation for downstream services.

If you see permission errors:

- Check the role used by the daemon, task, or service integration.
- Verify `xray:PutTraceSegments` and related actions.
- Confirm the correct managed policy or custom policy is attached.

**Practical mental model**

Think of X-Ray troubleshooting as three layers:

- **Collection**: the app and SDK create trace data.
- **Transport**: the daemon or managed integration forwards it.
- **Authorization and storage**: IAM allows publishing and X-Ray stores the data.

When a service is missing from the graph, ask which layer failed. That approach is usually faster and more accurate than guessing at a single permission problem.
