## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Scenario overview**

A broken CI/CD pipeline usually fails in one of four places: source, build, deploy, or permissions. Your job is to identify the failing service, inspect the execution history, and correct the smallest change that restores the flow. In AWS, that often means checking CodePipeline stage state, CodeBuild logs, CodeDeploy lifecycle events, and the IAM roles that connect them.

A good repair workflow starts with the pipeline execution record, then moves outward to the service that owns the failure. Do not guess from the symptom alone. Use the service-specific logs and event history to confirm whether the issue is caused by a bad artifact, a missing permission, an invalid buildspec, a deployment hook failure, or an environment mismatch.

**Core troubleshooting flow**

1. Check the pipeline execution status in CodePipeline.
2. Identify the failed action and its error message.
3. Open the owning service logs or event history.
4. Verify IAM permissions, artifact paths, and environment configuration.
5. Re-run only after you confirm the root cause.

**CodePipeline failure analysis**

CodePipeline is the orchestration layer, so it tells you where the failure occurred but not always why. The most useful starting points are the pipeline execution details and the action execution history.

- **Pipeline state** shows which stage is blocked and whether the failure is in source, build, approval, or deploy.
- **Action execution history** is often the best place to find per-action error messages and timestamps.
- **Execution details** help you correlate a failed action with the artifact revision that triggered it.
- **CloudWatch Events / EventBridge** can emit pipeline state changes for alerting and automation.

Useful CLI commands:

```bash
aws codepipeline get-pipeline-state --name my-pipeline
aws codepipeline get-pipeline-execution \
  --pipeline-name my-pipeline \
  --pipeline-execution-id 01234567-89ab-cdef-0123-456789abcdef
```

For action-level inspection, use `list-action-executions` with a JSON filter structure rather than a bare key-value string:

```bash
aws codepipeline list-action-executions \
  --pipeline-name my-pipeline \
  --filter '{"pipelineExecutionId":"01234567-89ab-cdef-0123-456789abcdef"}'
```

When you inspect failures, look for:

- A stage stuck in `Failed`, `Stopped`, or `InProgress` longer than expected.
- A source revision that never produced the expected artifact.
- A build action that succeeded but emitted the wrong output artifact name.
- A deploy action that cannot find the revision bundle or target environment.

> Not every failed action is retryable in place. Retry availability depends on the action type, the failure state, and whether the underlying issue has been corrected.

**Source stage issues**

Source failures usually come from repository access, webhook configuration, or artifact format problems. The source action must produce a valid revision and artifact that downstream stages can consume.

Common causes include:

- The source connection is revoked or expired.
- The repository branch name changed.
- The webhook or polling trigger is misconfigured.
- The source output artifact name does not match what later stages expect.
- The source provider cannot access the repository due to IAM or connection issues.

Practical checks:

- Confirm the branch and repository path in the source action configuration.
- Verify the source artifact is created and passed to the next stage.
- Check whether the pipeline expects a ZIP archive, a source bundle, or a specific folder structure.
- If using CodeStar Connections, confirm the connection is still `AVAILABLE`.

Example source action considerations:

```json
{
  "name": "Source",
  "actions": [
    {
      "name": "SourceFromGit",
      "actionTypeId": {
        "category": "Source",
        "owner": "AWS",
        "provider": "CodeStarSourceConnection",
        "version": "1"
      },
      "outputArtifacts": [
        { "name": "SourceOutput" }
      ]
    }
  ]
}
```

**Build stage issues with CodeBuild**

CodeBuild failures are often caused by the buildspec, environment image, IAM role, or artifact packaging. The build role must be able to read input artifacts, write logs, and access any external services the build uses.

Typical build failure categories:

- **Buildspec syntax** errors or missing phases.
- **Dependency installation** failures in `install` or `pre_build`.
- **Test failures** in the `build` or `post_build` phase.
- **Artifact path** mismatches, especially when the output directory is wrong.
- **Environment variable** issues, including missing secrets or parameter values.
- **Permission** problems for S3, ECR, Parameter Store, Secrets Manager, or KMS.

A minimal buildspec example:

```yaml
version: 0.2

phases:
  install:
    runtime-versions:
      python: 3.11
    commands:
      - pip install -r requirements.txt
  build:
    commands:
      - pytest
artifacts:
  files:
    - '**/*'
  base-directory: dist
```

When the build needs configuration values, scope permissions carefully. For example, access to Systems Manager Parameter Store and Secrets Manager should usually be limited to the specific parameter or secret ARN, not `*`. If the parameter or secret is encrypted with a customer managed AWS Key Management Service (AWS KMS) key, the role also needs `kms:Decrypt` for that key.

Example IAM policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadBuildConfig",
      "Effect": "Allow",
      "Action": [
        "ssm:GetParameters",
        "secretsmanager:GetSecretValue"
      ],
      "Resource": [
        "arn:aws:ssm:us-east-1:123456789012:parameter/app/prod/*",
        "arn:aws:secretsmanager:us-east-1:123456789012:secret:app/prod/*"
      ]
    },
    {
      "Sid": "DecryptIfNeeded",
      "Effect": "Allow",
      "Action": "kms:Decrypt",
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555"
    }
  ]
}
```

For container image builds and pushes to Amazon Elastic Container Registry (Amazon ECR), the build role needs the full push workflow permissions, not just read access. A common push flow requires:

- `ecr:GetAuthorizationToken`
- `ecr:BatchCheckLayerAvailability`
- `ecr:InitiateLayerUpload`
- `ecr:UploadLayerPart`
- `ecr:CompleteLayerUpload`
- `ecr:PutImage`

Depending on the workflow, `ecr:BatchGetImage` may also be needed for image existence checks or promotion logic.

Example push-oriented policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "ecr:GetAuthorizationToken",
      "Resource": "*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "ecr:BatchCheckLayerAvailability",
        "ecr:InitiateLayerUpload",
        "ecr:UploadLayerPart",
        "ecr:CompleteLayerUpload",
        "ecr:PutImage"
      ],
      "Resource": "arn:aws:ecr:us-east-1:123456789012:repository/my-app"
    }
  ]
}
```

Useful CodeBuild diagnostics:

```bash
aws codebuild batch-get-builds --ids my-build-id
aws logs tail /aws/codebuild/my-project --follow
```

Inside the build container, verify the artifact output before the `artifacts` phase completes:

```bash
pwd
ls -la
find dist -maxdepth 2 -type f
```

**Deploy stage issues with CodeDeploy**

CodeDeploy failures depend on the deployment target type: EC2/on-premises, Lambda, or Amazon ECS. The deployment bundle, AppSpec file, and lifecycle hooks must match the target type exactly.

For EC2/on-premises deployments:

- The instance profile needs permission to access the deployment bundle source used by the revision, commonly Amazon S3.
- The CodeDeploy agent must be installed and running on the instance.
- The AppSpec file must be valid and placed at the root of the revision bundle.
- Lifecycle hooks must reference scripts that exist and are executable.
- The target instance must be correctly tagged or included in the deployment group.

A simple EC2 AppSpec example:

```yaml
version: 0.0
os: linux
files:
  - source: /
    destination: /var/www/myapp
hooks:
  BeforeInstall:
    - location: scripts/before_install.sh
      timeout: 300
      runas: root
  AfterInstall:
    - location: scripts/after_install.sh
      timeout: 300
      runas: root
```

For Lambda deployments, CodeDeploy does not deploy directly to `$LATEST`. It shifts traffic between published versions using an alias that already points to the current version. CodeDeploy manages the alias routing configuration during the deployment, so the alias and version setup must be correct before the deployment starts.

A Lambda deployment AppSpec uses lifecycle event names mapped to Lambda function names in a specific structure. A simplified but more accurate example is:

```yaml
version: 0.0
Resources:
  - myLambdaFunction:
      Type: AWS::Lambda::Function
      Properties:
        Name: my-function
        Alias: live
        CurrentVersion: 12
        TargetVersion: 13
Hooks:
  BeforeAllowTraffic: preTrafficHook
  AfterAllowTraffic: postTrafficHook
```

For Amazon ECS deployments, CodeDeploy updates the task set and shifts traffic through the load balancer. Common issues include:

- Incorrect task definition revision.
- Missing container port mapping.
- Target group health check failures.
- IAM permissions for ECS, Elastic Load Balancing, or CodeDeploy service roles.

**IAM and service role repair**

Many pipeline failures are permission failures disguised as service errors. When a stage cannot read an artifact, write logs, or invoke a downstream service, the first thing to check is the role attached to that action or service.

Common roles involved:

- **CodePipeline service role** for orchestration and artifact access.
- **CodeBuild service role** for build-time AWS API calls.
- **CodeDeploy service role** for deployment coordination.
- **EC2 instance profile** for on-instance deployment actions and bundle retrieval.

Troubleshooting approach:

- Confirm the trust policy allows the correct service principal.
- Confirm the permissions policy includes the exact actions needed.
- Scope resources to the specific bucket, repository, secret, or key when possible.
- Check for explicit denies in SCPs, permission boundaries, or session policies.

Example trust policy for CodeBuild:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "codebuild.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example S3 artifact access pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:GetObjectVersion",
        "s3:PutObject"
      ],
      "Resource": [
        "arn:aws:s3:::my-artifact-bucket/*"
      ]
    }
  ]
}
```

If the pipeline uses encrypted artifacts, the role that reads or writes them may also need KMS permissions for the specific key.

**Artifact and packaging problems**

A pipeline can fail even when every service is healthy if the artifact contract is broken. The artifact contract includes file names, directory structure, compression format, and the expected output artifact name.

Watch for these mismatches:

- The build produces `build/` but the deploy stage expects `dist/`.
- The AppSpec file is not at the root of the bundle.
- The output artifact name in CodeBuild does not match the input artifact name in CodePipeline.
- The deployment bundle contains nested directories that break script paths.
- The ZIP file is created with the wrong root folder, causing CodeDeploy to look in the wrong place.

Practical validation steps:

```bash
unzip -l artifact.zip
tar -tf artifact.tar.gz
aws s3 ls s3://my-artifact-bucket/pipelines/my-app/
```

If the artifact is generated by CodeBuild, confirm the `artifacts` section matches the actual output directory:

```yaml
artifacts:
  files:
    - '**/*'
  base-directory: output
```

**EventBridge and notifications**

EventBridge helps you detect failures quickly and automate responses. You can route pipeline state changes to SNS, Lambda, or incident tooling.

Common uses:

- Notify on stage failure.
- Trigger a diagnostic Lambda function.
- Open a ticket when a deployment fails.
- Capture execution metadata for audit and trend analysis.

Example event pattern concept:

```json
{
  "source": ["aws.codepipeline"],
  "detail-type": ["CodePipeline Pipeline Execution State Change"],
  "detail": {
    "state": ["FAILED"]
  }
}
```

**Practical repair examples**

A build fails because CodeBuild cannot read a secret. You inspect the build logs, confirm the secret ARN, and update the build role to allow `secretsmanager:GetSecretValue` on that secret. If the secret uses a customer managed KMS key, you also add `kms:Decrypt` for that key.

A deploy fails because CodeDeploy cannot find the AppSpec file. You inspect the artifact ZIP and discover the file is nested under an extra top-level folder. You change the packaging step so `appspec.yml` sits at the root of the revision bundle.

A pipeline fails after a successful build because the deploy stage cannot access the artifact bucket. You check the CodePipeline service role and add the missing `s3:GetObject` permission for the artifact path, then verify the bucket policy does not block the role.

**Exam-focused distinctions**

- **CodePipeline** tells you where the failure is.
- **CodeBuild** tells you whether the build environment, commands, or permissions are wrong.
- **CodeDeploy** tells you whether the revision, hooks, or target environment are wrong.
- **IAM** is often the hidden root cause when a service reports access failure.
- **Artifacts** must match the expected structure exactly, or downstream stages fail even if the code is correct.

**Quick checklist for troubleshooting**

- Confirm the failing stage and action.
- Read the service-specific logs.
- Verify the artifact name and path.
- Check the service role trust policy.
- Scope permissions to the exact resource.
- Confirm KMS access only when encryption uses a customer managed key.
- Validate deployment-specific files such as `buildspec.yml` and `appspec.yml`.
- Re-run the pipeline only after the root cause is fixed.

**Key takeaways**

A CI/CD repair scenario is mostly a process of narrowing scope. Start with the pipeline execution, then move to the service that owns the failure, and finally inspect IAM and artifact contracts. In exam questions, the correct answer is often the one that fixes the smallest broken link in the chain rather than the one that broadly “rebuilds the pipeline.”
