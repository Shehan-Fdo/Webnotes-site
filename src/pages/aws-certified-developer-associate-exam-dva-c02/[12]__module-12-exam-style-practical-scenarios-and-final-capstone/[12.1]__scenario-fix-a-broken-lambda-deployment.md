## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Scenario overview**

A broken AWS Lambda deployment usually fails in one of four places: packaging, permissions, runtime configuration, or integration wiring. In exam scenarios, you are expected to identify the failure domain quickly, then apply the smallest fix that restores invocation and preserves least privilege.

The most common symptoms are:

- `AccessDeniedException` during deploy or invoke
- `Handler not found` or `Runtime.ImportModuleError`
- `InvalidParameterValueException` for runtime, role, or event source settings
- timeouts, memory errors, or missing environment variables
- function updates that succeed but the code still behaves as if the old version is running

**What to check first**

Start with the Lambda function configuration and recent deployment changes. A deployment can be “broken” even when the code uploaded successfully if the handler name, runtime, role trust policy, or artifact structure is wrong.

- Confirm the function’s **runtime** matches the code package.
- Confirm the **handler** string points to the correct file and exported symbol.
- Confirm the execution **role** can be assumed by Lambda.
- Confirm the deployment artifact contains the expected files at the package root.
- Confirm any event source mapping or trigger still points to the correct function version or alias.

**Lambda deployment anatomy**

A Lambda deployment is more than code upload. The function depends on a valid execution role, a compatible runtime, a correctly structured artifact, and any required environment or network settings.

- **Execution role**: the IAM role Lambda assumes to call AWS APIs and write logs.
- **Handler**: the entry point Lambda invokes, such as `app.lambda_handler`.
- **Runtime**: the language environment, such as Python, Node.js, Java, or .NET.
- **Artifact**: the ZIP file or container image that contains your code and dependencies.
- **Trigger**: the event source, such as API Gateway, S3, EventBridge, or SQS.

If any one of these is wrong, the function may deploy but fail at runtime.

**Common failure modes and fixes**

**1. IAM role or trust policy problems**

Lambda must be able to assume the execution role. The role trust policy should allow the Lambda service principal.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Typical symptoms:

- function creation fails with a role assumption error
- invocation works, but CloudWatch Logs are missing because the role lacks logging permissions
- the function cannot access S3, DynamoDB, SQS, or Secrets Manager

Minimum logging permissions usually include:

```json
{
  "Effect": "Allow",
  "Action": [
    "logs:CreateLogGroup",
    "logs:CreateLogStream",
    "logs:PutLogEvents"
  ],
  "Resource": "*"
}
```

Practical fix pattern:

- verify the trust relationship includes `lambda.amazonaws.com`
- attach only the permissions the function needs
- check whether the function uses a customer-managed KMS key, VPC resources, or cross-service access that requires extra permissions

> If the role is wrong, the function may fail before your code runs. That makes IAM one of the first places to inspect.

**2. Handler mismatch**

A handler mismatch is one of the most common deployment errors. Lambda looks for a specific file and function name.

Examples:

- Python: `app.lambda_handler`
- Node.js: `index.handler`
- Java: `com.example.Handler::handleRequest`

For Python, the file `app.py` must contain a function named `lambda_handler`:

```python
def lambda_handler(event, context):
    return {
        "statusCode": 200,
        "body": "ok"
    }
```

If the handler is configured as `main.handler` but the file is `app.py`, Lambda returns an error because it cannot find the entry point.

Checklist:

- confirm the handler string matches the file name
- confirm the function name is exported correctly
- confirm the deployment package root is not nested one directory too deep

**3. Packaging and dependency issues**

A ZIP deployment often fails because the package structure is wrong. Lambda expects your code and dependencies at the root of the archive unless you intentionally package them differently.

For Python, a common mistake is zipping the parent folder instead of its contents:

```bash
# Correct: package contents at the root of the ZIP
cd my-function
zip -r ../function.zip .

aws lambda update-function-code \
  --function-name my-function \
  --zip-file fileb://function.zip
```

For Node.js, dependencies must be installed before packaging:

```bash
npm install
zip -r function.zip index.js node_modules package.json
```

Common packaging problems:

- missing dependency modules
- compiled libraries built for the wrong architecture
- ZIP contains an extra top-level folder
- container image exceeds size or uses an unsupported base image
- native extensions are built on a different OS or CPU architecture than the Lambda runtime

Useful checks:

- inspect the ZIP contents with `unzip -l function.zip`
- verify the runtime architecture: `x86_64` or `arm64`
- ensure dependencies match the runtime version

**4. Runtime incompatibility**

The runtime must match the code and dependencies. If you deploy code written for a newer language version than the function runtime supports, the function may fail at import or initialization.

Examples:

- Python code uses syntax not supported by the configured Python runtime
- Node.js code depends on a module version incompatible with the runtime
- Java function uses a classpath layout that does not match the packaging method

Practical fix:

- update the runtime if the code requires a newer supported version
- rebuild dependencies for the target runtime
- avoid relying on deprecated runtimes in production

**5. Environment variable and configuration errors**

A function can deploy successfully but fail when it reads missing or malformed environment variables.

Typical examples:

- database endpoint not set
- secret name misspelled
- feature flag missing
- region or bucket name incorrect

Use environment variables for configuration, but validate them at startup:

```python
import os

TABLE_NAME = os.environ["TABLE_NAME"]
```

If the variable is missing, the function fails immediately, which is often better than silently using the wrong resource.

Best practices:

- keep secrets in AWS Secrets Manager or Systems Manager Parameter Store
- store non-secret configuration in environment variables
- avoid hardcoding account IDs, regions, and resource names
- use aliases or stages to separate dev, test, and prod values

**6. Timeout, memory, and initialization issues**

A deployment may appear broken when the function is actually timing out or running out of memory.

Symptoms:

- `Task timed out after X.XX seconds`
- `Runtime exited with error`
- slow cold starts
- intermittent failures under load

Fixes:

- increase the timeout if the workload is legitimately long-running
- increase memory to get more CPU share and faster execution
- reduce initialization work in the global scope
- reuse SDK clients outside the handler when appropriate

Example of efficient client reuse in Python:

```python
import boto3

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("Orders")

def lambda_handler(event, context):
    return table.get_item(Key={"id": event["id"]})
```

This reduces per-invocation setup overhead.

**7. VPC and network access problems**

If the function runs in a VPC, it may lose internet access unless you provide a NAT gateway or only access private resources. This often looks like a broken deployment when the real issue is network reachability.

Common symptoms:

- cannot reach public APIs
- cannot download dependencies at runtime
- cannot connect to RDS, ElastiCache, or private endpoints
- function hangs until timeout

Checklist:

- confirm the subnet route table has the correct routes
- confirm security groups allow the required traffic
- confirm the function has access to NAT if it needs outbound internet
- confirm the target service is reachable from the selected subnets

**8. Event source mapping and trigger issues**

Sometimes the function code is fine, but the trigger is misconfigured.

Examples:

- S3 event notification points to the wrong function ARN
- API Gateway integration uses an outdated stage or deployment
- EventBridge rule targets an alias that no longer exists
- SQS event source mapping is disabled or points to the wrong queue
- DynamoDB Streams mapping is behind or failing

Useful commands:

```bash
aws lambda list-event-source-mappings --function-name my-function
aws lambda get-function-configuration --function-name my-function
aws lambda get-policy --function-name my-function
```

For API Gateway, verify the integration and deployment stage. For S3, verify the bucket notification configuration. For EventBridge, verify the rule target ARN.

**9. Versioning and alias mistakes**

A deployment can succeed but traffic may still go to an older version if the alias was not updated.

Common patterns:

- `$LATEST` updated, but the alias still points to an older published version
- CodeDeploy shifted traffic only partially
- a canary deployment is still in progress
- the trigger points to a versioned ARN instead of the alias

Practical guidance:

- use aliases for stable environments
- publish a version after each release
- update the alias only after validation
- confirm the trigger references the intended alias or version

**10. Permissions for downstream services**

The function may start correctly but fail when it calls another AWS service.

Examples:

- reading from S3 requires `s3:GetObject`
- writing to DynamoDB requires `dynamodb:PutItem`
- publishing to SNS requires `sns:Publish`
- sending to SQS requires `sqs:SendMessage`
- decrypting with KMS requires `kms:Decrypt`

A minimal IAM policy should grant only the exact actions and resources required.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "dynamodb:PutItem",
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders"
    }
  ]
}
```

**Debugging workflow**

Use a consistent sequence so you do not waste time chasing symptoms.

1. Check the latest CloudWatch Logs for the exact error message.
2. Verify the handler, runtime, and package structure.
3. Verify the execution role trust policy and permissions.
4. Verify environment variables and secrets references.
5. Verify network access if the function is in a VPC.
6. Verify trigger configuration, event source mappings, and aliases.
7. Redeploy the smallest possible fix and retest.

CloudWatch Logs are often the fastest path to the root cause:

```bash
aws logs tail /aws/lambda/my-function --follow
```

If the function never writes logs, suspect the execution role or a failure before the handler starts.

**Deployment commands and practical examples**

A typical ZIP-based update flow looks like this:

```bash
aws lambda update-function-code \
  --function-name my-function \
  --zip-file fileb://function.zip

aws lambda update-function-configuration \
  --function-name my-function \
  --handler app.lambda_handler \
  --runtime python3.12 \
  --timeout 10 \
  --memory-size 256
```

To inspect the current configuration:

```bash
aws lambda get-function-configuration --function-name my-function
```

To test the function directly:

```bash
aws lambda invoke \
  --function-name my-function \
  --payload '{"id":"123"}' \
  response.json
```

For container image deployments, verify the image URI and repository permissions:

```bash
aws lambda update-function-code \
  --function-name my-function \
  --image-uri 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-function:latest
```

**Container image-specific checks**

If the function uses a container image, the deployment can fail because of image build or entrypoint issues rather than Lambda code issues.

Check:

- the image contains the correct runtime and dependencies
- the Dockerfile uses a supported base image
- the `CMD` or `ENTRYPOINT` matches the handler expectations
- the image is pushed to Amazon Elastic Container Registry (ECR) in the correct region
- the Lambda execution role can pull from ECR

Example Dockerfile pattern:

```dockerfile
FROM public.ecr.aws/lambda/python:3.12

COPY app.py ${LAMBDA_TASK_ROOT}
COPY requirements.txt .
RUN pip install -r requirements.txt -t ${LAMBDA_TASK_ROOT}

CMD ["app.lambda_handler"]
```

**Observability and rollback**

When a deployment breaks production, you need a fast rollback path.

- use aliases to shift traffic back to the last known good version
- keep previous versions published for quick recovery
- monitor `Errors`, `Throttles`, `Duration`, and `IteratorAge` where relevant
- inspect `InitDuration` for cold-start-related issues
- use structured logs so you can correlate failures with request IDs

Example rollback approach:

1. Identify the last healthy published version.
2. Point the alias back to that version.
3. Confirm the trigger uses the alias, not `$LATEST`.
4. Investigate the broken release offline.

**Exam-focused decision cues**

When you see a broken Lambda deployment scenario, map the clue to the likely fix:

- `Handler not found` → handler string or package layout
- `Unable to import module` → missing dependency or wrong ZIP structure
- `AccessDenied` on logs or AWS API calls → IAM permissions
- function times out after deployment → timeout, memory, or VPC access
- trigger stopped working after release → alias, version, or event source mapping
- code works locally but not in Lambda → runtime mismatch, packaging, or environment variables

**Quick reference checklist**

- **Code entry point**: handler name and file path match
- **Artifact layout**: dependencies are packaged correctly
- **Runtime**: language version is compatible
- **IAM**: trust policy and permissions are correct
- **Config**: environment variables and secrets exist
- **Network**: VPC routing and security groups allow access
- **Trigger**: event source points to the right ARN, alias, or version
- **Observability**: CloudWatch Logs show the exact failure

**Key takeaway**

A broken Lambda deployment is usually not a single “Lambda problem.” It is a mismatch between code, configuration, permissions, and integration. The fastest fix comes from isolating which layer failed, then correcting only that layer with the least disruptive change.
