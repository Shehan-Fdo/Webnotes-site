## **Course:** AWS Certified Developer – Associate (Exam DVA-C02)

**Scenario overview**

This scenario focuses on repairing a direct integration between Amazon API Gateway and Amazon DynamoDB. In exam terms, you need to identify where the request or response path is broken: authorization, request mapping, integration configuration, or response mapping. The key is to understand how API Gateway transforms HTTP requests into DynamoDB actions and how DynamoDB returns data.

A working design usually follows this flow:

- Client sends an HTTP request to API Gateway.
- API Gateway uses an **AWS service integration** to call DynamoDB.
- API Gateway assumes an IAM role and uses its temporary credentials to sign the downstream request.
- DynamoDB performs the action and returns a service response.
- API Gateway maps that response back to the client format.

**Core integration model**

For a REST API, API Gateway can integrate directly with DynamoDB without Lambda. This is useful when you want lower latency, fewer moving parts, and simpler operational overhead. It is also a common exam pattern because the failure often comes from a small configuration mismatch rather than a code bug.

Important concepts to keep straight:

- **Service integration** means API Gateway calls another AWS service directly.
- **IAM role assumption** means API Gateway assumes a role that has permission to call DynamoDB.
- **Mapping templates** translate between client JSON and DynamoDB’s action-specific request format.
- **Integration responses** translate DynamoDB output or errors into HTTP responses.

**IAM permissions and trust**

API Gateway does not call DynamoDB as the end user. Instead, it assumes an IAM role that you configure on the integration. That role must trust API Gateway and must allow the required DynamoDB actions.

Typical trust policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "apigateway.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Typical permissions policy for a `PutItem` integration:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:PutItem"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders"
    }
  ]
}
```

Key troubleshooting points:

- The trust policy must allow `apigateway.amazonaws.com`.
- The permissions policy must include the exact DynamoDB action used by the integration.
- The resource ARN must match the table, and if needed, the index or stream ARN.
- If the role is wrong, API Gateway may return a 500-level integration error even though the client request looks valid.

> For direct service integrations, API Gateway assumes the IAM role and signs the downstream request with the role’s temporary credentials. It does not forward the caller’s IAM identity unless you explicitly design for that pattern.

**DynamoDB action mapping**

DynamoDB service integrations are action-specific. The request body that API Gateway sends to DynamoDB must match the target action’s expected shape.

Common actions in this scenario:

- `PutItem` for creating or replacing an item.
- `GetItem` for reading a single item by primary key.
- `UpdateItem` for modifying attributes.
- `DeleteItem` for removing an item.
- `Query` for partition-key-based retrieval.
- `Scan` for full table reads, usually avoided in production.

For `PutItem`, the request must include the item in DynamoDB JSON format, where each attribute is typed:

```json
{
  "TableName": "Orders",
  "Item": {
    "orderId": { "S": "123" },
    "status": { "S": "NEW" },
    "amount": { "N": "49.99" }
  }
}
```

For `GetItem`, the request uses the key only:

```json
{
  "TableName": "Orders",
  "Key": {
    "orderId": { "S": "123" }
  }
}
```

**Mapping templates and Velocity Template Language**

API Gateway REST APIs commonly use Velocity Template Language (VTL) mapping templates to transform incoming JSON into DynamoDB’s request format. This is where many scenario failures occur.

A request mapping template for `PutItem` might look like this:

```vtl
{
  "TableName": "Orders",
  "Item": {
    "orderId": { "S": "$input.path('$.orderId')" },
    "status": { "S": "$input.path('$.status')" },
    "amount": { "N": "$input.path('$.amount')" }
  }
}
```

Practical notes:

- Use `$input.path()` to extract JSON fields from the request body.
- Use `$util.escapeJavaScript()` when you need to safely embed strings.
- Convert numbers to strings for DynamoDB numeric attributes.
- Ensure the template output is valid JSON.

A safer example for string fields:

```vtl
{
  "TableName": "Orders",
  "Item": {
    "orderId": { "S": "$util.escapeJavaScript($input.path('$.orderId'))" },
    "status": { "S": "$util.escapeJavaScript($input.path('$.status'))" }
  }
}
```

**Passthrough behavior and content types**

For REST API non-proxy integrations, API Gateway uses the request’s `Content-Type` header to choose a mapping template. If no template matches, the behavior depends on `passthroughBehavior`.

Common settings:

- `WHEN_NO_MATCH` allows the request to pass through when no template matches.
- `WHEN_NO_TEMPLATES` passes through only if no templates are defined at all.
- `NEVER` rejects requests that do not match a template.

This matters because a client sending `application/json` may receive a `415 Unsupported Media Type` if no matching template exists and passthrough is not allowed.

Example integration configuration idea:

```yaml
passthroughBehavior: NEVER
requestTemplates:
  application/json: |
    {
      "TableName": "Orders",
      "Item": {
        "orderId": { "S": "$input.path('$.orderId')" }
      }
    }
```

Troubleshooting guidance:

- Confirm the client sends the expected `Content-Type`.
- Confirm the integration has a template for that exact media type.
- Check whether passthrough is intentionally enabled.
- If you see `415`, suspect template mismatch or `NEVER` passthrough behavior.

**DynamoDB response handling**

A common exam trap is assuming `PutItem` returns the inserted item. It does not. By default, `PutItem` returns only metadata such as consumed capacity if requested, not the new item attributes.

If you need the written item back, you usually design the API differently:

- Return the request payload from API Gateway if that is acceptable.
- Use `GetItem` after `PutItem` if you need to read the stored item.
- Use `UpdateItem` with `ReturnValues` if you need old or updated attributes from the update operation.

For `PutItem`, `ReturnValues` can request the old item attributes, not the newly written item. That distinction is important.

Example of a response mapping template for a `GetItem` integration:

```vtl
#if($input.path('$.Item'))
{
  "orderId": "$input.path('$.Item.orderId.S')",
  "status": "$input.path('$.Item.status.S')",
  "amount": "$input.path('$.Item.amount.N')"
}
#else
{
  "message": "Item not found"
}
#end
```

For `PutItem`, a response template often returns a simple success message:

```vtl
{
  "message": "Order created successfully"
}
```

**Integration responses and error mapping**

API Gateway can map backend errors to HTTP status codes. This is useful when DynamoDB returns a service error or when the request template fails.

Typical mappings:

- `ConditionalCheckFailedException` to `409 Conflict` when a conditional write fails.
- `ValidationException` to `400 Bad Request` for malformed input.
- `AccessDeniedException` to `403 Forbidden` or `500` depending on where the failure occurs.
- `ResourceNotFoundException` to `404 Not Found` when the table or resource is missing.

Example integration response selection pattern:

```json
"selectionPattern": ".*ConditionalCheckFailedException.*"
```

Use this when you want API Gateway to route a specific backend error to a custom client response.

**Conditional writes and idempotency**

DynamoDB conditional expressions help prevent accidental overwrites and support idempotent APIs. This is especially useful when the client may retry requests.

Example `PutItem` with a condition:

```json
{
  "TableName": "Orders",
  "Item": {
    "orderId": { "S": "123" },
    "status": { "S": "NEW" }
  },
  "ConditionExpression": "attribute_not_exists(orderId)"
}
```

Why this matters:

- It prevents duplicate creation.
- It supports safe retries.
- It lets you return a conflict-style response when the item already exists.

**REST API versus HTTP API considerations**

Do not overgeneralize API Gateway behavior across API types.

- **REST APIs** support mapping templates, integration responses, and resource policies in the classic way used in this scenario.
- **HTTP APIs** have a different feature set and generally use simpler integrations.
- Resource policy behavior and support differ by API type, so always confirm which API type the scenario describes.

For exam questions, if you see VTL templates, integration responses, or detailed passthrough behavior, you are almost certainly dealing with a REST API scenario.

**Resource policies and access control**

Resource policies can restrict who can invoke an API, but the exact behavior depends on the API type. For REST APIs, you can use resource policies to limit access by:

- AWS account
- IAM principal
- Source IP address
- VPC endpoint

Example REST API resource policy concept:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "execute-api:Invoke",
      "Resource": "arn:aws:execute-api:us-east-1:123456789012:abc123/*/POST/orders",
      "Condition": {
        "StringEquals": {
          "aws:SourceVpce": "vpce-0123456789abcdef0"
        }
      }
    }
  ]
}
```

Use this carefully:

- It is powerful for private or restricted APIs.
- It can block requests before they reach the integration.
- It is not the same as the IAM role used by API Gateway to call DynamoDB.

**Deployment, stage, and live configuration**

A deployed API stage points to a specific deployment. If you are troubleshooting a live issue, you need to verify the stage’s active deployment rather than assuming the latest configuration is live.

Useful CLI workflow:

```bash
aws apigateway get-stage \
  --rest-api-id abc123 \
  --stage-name prod
```

Look for the `deploymentId` in the output, then inspect that deployment:

```bash
aws apigateway get-deployment \
  --rest-api-id abc123 \
  --deployment-id xyz789
```

Why this matters:

- Editing an API does not automatically change the live stage.
- A stage can still point to an older deployment.
- The active deployment is what clients actually use.

**Practical troubleshooting checklist**

When a direct API Gateway to DynamoDB integration fails, check the layers in order:

1. Confirm the API type is REST API if you expect mapping templates and integration responses.
2. Verify the method request and integration request are both configured.
3. Check the IAM role trust policy for `apigateway.amazonaws.com`.
4. Check the IAM permissions for the exact DynamoDB action and table ARN.
5. Confirm the request mapping template matches the incoming `Content-Type`.
6. Verify `passthroughBehavior` is appropriate for unmatched content types.
7. Inspect the DynamoDB action shape for the correct keys and attribute types.
8. Review integration response mappings for backend errors.
9. Confirm the stage points to the intended deployment.

**Common exam patterns**

These are the patterns most likely to appear in DVA-C02 questions:

- A `PutItem` integration fails because the request template uses the wrong DynamoDB JSON shape.
- A request returns `415 Unsupported Media Type` because the content type does not match any template and passthrough is disabled.
- The API returns `500` because the assumed role lacks `dynamodb:PutItem`.
- The stage still uses an old deployment, so a fix appears not to work.
- A response template incorrectly expects the inserted item from `PutItem`, but the action does not return it.
- A resource policy blocks invocation even though the integration role is correct.

**Example end-to-end repair flow**

If you are asked to repair a broken order-creation endpoint, a good solution often looks like this:

1. Verify the method uses a REST API AWS service integration to DynamoDB.
2. Confirm the integration role trusts API Gateway and allows `dynamodb:PutItem`.
3. Fix the request mapping template so it emits valid DynamoDB `Item` JSON.
4. Ensure the client sends `application/json` and the template matches that media type.
5. Set `passthroughBehavior` intentionally, usually `NEVER` for strict APIs.
6. Return a simple success response or perform a follow-up `GetItem` if the client needs stored data.
7. Redeploy the API and confirm the stage points to the new deployment.

**Useful CLI and configuration examples**

Inspect a REST API integration:

```bash
aws apigateway get-integration \
  --rest-api-id abc123 \
  --resource-id r1 \
  --http-method POST
```

Inspect the stage deployment:

```bash
aws apigateway get-stage \
  --rest-api-id abc123 \
  --stage-name prod
```

Example of a strict request template for a create operation:

```vtl
#set($inputRoot = $input.path('$'))
{
  "TableName": "Orders",
  "Item": {
    "orderId": { "S": "$util.escapeJavaScript($inputRoot.orderId)" },
    "customerId": { "S": "$util.escapeJavaScript($inputRoot.customerId)" },
    "status": { "S": "NEW" }
  },
  "ConditionExpression": "attribute_not_exists(orderId)"
}
```

**Key takeaways**

- API Gateway assumes an IAM role to call DynamoDB in a direct service integration.
- Mapping templates must match both the client payload and the DynamoDB action shape.
- `PutItem` does not return the inserted item by default.
- `passthroughBehavior` controls what happens when no template matches the request content type.
- REST API features such as VTL templates, integration responses, and resource policies are central to this scenario.
- Always verify the active stage deployment before concluding a fix failed.
