## **Course:** 200-301 CCNA

**DNS overview**

- **DNS** stands for Domain Name System.
- It translates human-readable names into IP addresses.
- It helps you access services with names like `www.example.com` instead of numeric addresses.
- It works for devices on the Internet and local networks.
- It reduces the need to remember long IP address values.

**How DNS is organized**

- DNS uses a **hierarchical** structure.
- A **recursive resolver** handles the user query first.
- A **root server** points to the correct top-level domain.
- A **TLD server** handles the top-level domain, such as `.com` or `.org`.
- An **authoritative server** provides the final answer for the domain.

**DNS lookup flow**

- You enter a hostname in a browser or application.
- The recursive resolver receives the request.
- The resolver checks for an answer and follows the DNS hierarchy if needed.
- The root server directs the resolver toward the right TLD.
- The TLD server directs the resolver toward the authoritative server.
- The authoritative server returns the IP address.
- The resolver gives the result back to the client.

**Key DNS roles**

- The **recursive resolver** does the lookup work on your behalf.
- The **authoritative server** owns the DNS records for the domain.
- Cached answers can be outdated.
- For an address lookup, the authoritative source is the server that owns the record.

**Why DNS matters**

- DNS makes web access easier for users.
- DNS supports network communication by mapping names to addresses.
- DNS is a core IP service in networking.
- DNS is important for troubleshooting and certification study.

**Common DNS record types**

- `A` records map a name to an IPv4 address.
- `AAAA` records map a name to an IPv6 address.
- `CNAME` records point one name to another name.
- `MX` records support email routing.
- `NS` records identify name servers.
- `TXT` records store text data for verification or policy use.
- `SOA` records define zone authority details.

**Practical takeaways**

- Use DNS when you want to reach a service by name.
- Check the resolver path when name resolution fails.
- Verify authoritative records when cached data looks wrong.
- Remember that DNS answers can depend on the server that responds.
