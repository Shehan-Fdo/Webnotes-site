## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Why verification habits matter**

On the RHCSA exam, you are graded on results, not intent. A change that looks correct in a config file is not enough unless you verify that the system actually behaves as required. Good verification habits help you catch typos, missing dependencies, incorrect SELinux contexts, bad `fstab` entries, and services that fail only after a reboot.

The best workflow is simple:

- Make one change.
- Verify the change immediately.
- Test persistence where relevant.
- Re-check after a reboot or service restart when the task depends on it.

A strong verification routine reduces guesswork and saves time during exam tasks.

**Core verification mindset**

Treat every task as two separate questions:

- Did the configuration apply now?
- Will it still work after restart or reboot?

That distinction matters for:

- **Services** managed by `systemd`
- **Storage** mounted through `/etc/fstab`
- **Users** and home directories created from skeleton files
- **Firewall** rules that must survive reboots
- **SELinux** labels and booleans
- **Scheduled jobs** that must run later
- **Containers** that must restart or expose ports correctly

A practical exam habit is to verify from the outside as well as from the system itself. For example, if you configure a web server, check the service status locally and also test access with `curl` or a browser-like request.

**General verification workflow**

1. Confirm the configuration syntax or state.
2. Check the active runtime state.
3. Test the feature with a real command or access attempt.
4. Reboot or restart the relevant component if persistence matters.
5. Verify again after the restart.

Useful commands for this pattern include:

```bash
systemctl status <service>
systemctl is-enabled <service>
systemctl is-active <service>
journalctl -u <service> --no-pager
```

For network-facing tasks, also test the service directly:

```bash
curl -I http://localhost
ss -tulpn
```

For file-based tasks, inspect both the file and the mounted result:

```bash
ls -l /path/to/file
mount | grep /mountpoint
findmnt /mountpoint
```

**Service verification**

For `systemd` services, verify three things: the unit is enabled if persistence is required, the service is active now, and the service actually performs its job.

- Use `systemctl status <name>` to see the current state and recent log lines.
- Use `systemctl is-enabled <name>` to confirm boot-time behavior.
- Use `systemctl is-active <name>` to confirm runtime behavior.
- Use `journalctl -u <name>` when the service fails or behaves unexpectedly.

Example:

```bash
systemctl enable --now httpd
systemctl status httpd
systemctl is-enabled httpd
curl -I http://localhost
```

If a service depends on a file, port, or SELinux setting, verify those dependencies too. A service can be active but still unusable from the client side.

**Persistence checks after reboot**

For exam tasks, reboot verification is often the difference between a passing and failing configuration. After you make a persistent change, confirm it survives a restart.

Common checks:

- `systemctl is-enabled <service>` after reboot
- `systemctl status <service>` after reboot
- `curl`, `ss`, or `ps` to confirm the service still works
- `mount` or `findmnt` to confirm storage remounts
- `firewall-cmd --list-all` to confirm firewall rules remain
- `getenforce` and `getsebool -a` or `getsebool <boolean>` to confirm SELinux state

Example reboot verification sequence:

```bash
reboot
```

After logging back in:

```bash
systemctl is-active httpd
systemctl is-enabled httpd
curl -I http://localhost
findmnt /webdata
firewall-cmd --list-services
getenforce
```

> If a task depends on persistence, do not stop at a successful immediate test. Reboot or restart the relevant component and verify again.

**Storage verification**

Storage tasks on RHCSA often require more than creating a filesystem. You should verify device identity, mount behavior, and persistence across reboot.

Check the following:

- The block device or logical volume exists.
- The filesystem is present and mounted at the correct mount point.
- The mount uses the expected options.
- The `/etc/fstab` entry is correct and uses stable identifiers such as `UUID=`.
- The mount survives a reboot.

Useful commands:

```bash
lsblk -f
blkid
findmnt
mount | grep /data
cat /etc/fstab
```

Example `/etc/fstab` entry using a UUID:

```fstab
UUID=3f2a1c2d-7b4e-4d1f-9c2a-8f1d2e3a4b5c /data xfs defaults,noatime 0 0
```

After editing `/etc/fstab`, always test it before rebooting:

```bash
mount -a
findmnt /data
```

If `mount -a` returns no error, that is a good sign, but still confirm the mount is actually present. A typo in the mount point or options can still cause problems later.

For LVM, verify both the logical volume and its activation state:

```bash
lvs
vgs
pvs
lsblk
```

After reboot, confirm the volume is active and mounted:

```bash
lvs
lsblk -f
findmnt /data
```

If the logical volume does not appear, check whether the volume group is available and whether the system activated it automatically. In exam scenarios, this is a common place to lose points if you only verify the filesystem and not the underlying LVM state.

**Practical storage scenario**

Suppose you create `/dev/vgdata/lvshare`, format it as `xfs`, and mount it at `/share`.

Verify in order:

1. The logical volume exists.
2. The filesystem is created.
3. The mount point is present.
4. The mount is active.
5. `/etc/fstab` contains the correct UUID entry.
6. The system still mounts it after reboot.

Example commands:

```bash
lvs /dev/vgdata/lvshare
blkid /dev/vgdata/lvshare
mkdir -p /share
mount /dev/vgdata/lvshare /share
findmnt /share
```

Then add the persistent entry and test it:

```bash
UUID=$(blkid -s UUID -o value /dev/vgdata/lvshare)
echo "UUID=$UUID /share xfs defaults 0 0" >> /etc/fstab
mount -a
findmnt /share
```

After reboot:

```bash
lvs
findmnt /share
```

**User and group verification**

When you create users, verify more than just the account name. RHCSA tasks often involve home directories, skeleton files, password aging, and group membership.

Check:

- The user exists in `/etc/passwd`
- The home directory exists
- The home directory contains expected skeleton files
- The shell is correct
- Password expiration settings are correct
- Supplementary groups are assigned correctly

Useful commands:

```bash
id alice
getent passwd alice
getent group developers
ls -la /home/alice
chage -l alice
```

If you create a user with a home directory, verify that the skeleton files were copied from `/etc/skel`:

```bash
useradd -m alice
ls -la /home/alice
ls -la /etc/skel
```

Typical files to check include hidden shell startup files such as `.bashrc`, `.bash_profile`, or `.bash_logout`, depending on the distribution defaults.

For password expiration behavior, verify with `chage`:

```bash
chage -l alice
```

If you set aging values, confirm the output reflects the intended policy:

```bash
chage -M 30 -m 1 -W 7 alice
chage -l alice
```

For exam-style validation, also test login behavior when appropriate. If the task requires a user to change a password at next login, confirm the account state rather than assuming the command worked.

**User creation example**

```bash
useradd -m -c "Alice Admin" -G wheel,developers alice
echo 'RedHat123!' | passwd --stdin alice
chage -M 45 -m 1 -W 7 alice
id alice
ls -la /home/alice
chage -l alice
```

**Password and account state checks**

Account-related tasks often fail because the user exists but is locked, expired, or missing a home directory.

Verify:

- `passwd -S <user>` for password status
- `chage -l <user>` for aging details
- `id <user>` for group membership
- `getent passwd <user>` for account presence

Example:

```bash
passwd -S alice
chage -l alice
id alice
```

**SELinux verification**

SELinux is a frequent source of hidden failures. A service may be configured correctly but still blocked by labels, booleans, or context mismatches.

Verify:

- Current mode with `getenforce`
- Persistent mode in `/etc/selinux/config`
- File contexts with `ls -Z`
- Boolean settings with `getsebool`
- AVC denials in logs when something fails

Useful commands:

```bash
getenforce
sestatus
ls -Z /var/www/html
getsebool httpd_enable_homedirs
ausearch -m AVC -ts recent
journalctl -t setroubleshoot --no-pager
```

If you change file labels, verify the label is correct after the change:

```bash
semanage fcontext -a -t httpd_sys_content_t '/webdata(/.*)?'
restorecon -Rv /webdata
ls -Zd /webdata
ls -Z /webdata/index.html
```

A common exam habit is to verify both the label and the service access path. For example, if Apache serves content from a non-default directory, confirm the directory label and then test the page with `curl`.

```bash
curl http://localhost
```

**Firewall verification**

Firewall tasks should be checked at both the rule level and the service level. Do not assume a rule is active just because you added it.

Verify:

- The firewall service is running
- The rule is present in the active zone
- The rule is permanent if required
- The service or port is reachable

Useful commands:

```bash
systemctl status firewalld
firewall-cmd --get-active-zones
firewall-cmd --list-all
firewall-cmd --list-services
firewall-cmd --list-ports
```

Example:

```bash
firewall-cmd --permanent --add-service=http
firewall-cmd --reload
firewall-cmd --list-services
```

Then test the service locally:

```bash
curl -I http://localhost
```

If you open a custom port, verify both the rule and the listener:

```bash
firewall-cmd --permanent --add-port=8080/tcp
firewall-cmd --reload
ss -tulpn | grep 8080
firewall-cmd --list-ports
```

After reboot, confirm the rule remains in place:

```bash
firewall-cmd --list-all
```

**Scheduled task verification**

Scheduled jobs are easy to configure incorrectly because the syntax may be valid but the timing, user context, or environment may be wrong. You should verify both the schedule definition and the execution history.

For `cron`, check:

- The crontab entry exists for the correct user
- The cron service is running
- The job output or side effect appears at the expected time

Useful commands:

```bash
crontab -l
crontab -u alice -l
systemctl status crond
journalctl -u crond --no-pager
```

Example cron entry:

```cron
*/5 * * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```

To verify a cron job, do not rely only on `crontab -l`. Confirm that the command actually runs by checking the log file or the expected output file.

For `at`, verify the queue and the `atd` service:

```bash
atq
systemctl status atd
```

Example:

```bash
echo "touch /tmp/at-test" | at now + 2 minutes
atq
ls -l /tmp/at-test
```

If the task is time-sensitive, check the system clock and timezone as well:

```bash
timedatectl
date
```

**Container verification**

If your course scope includes containers, verify both the container runtime state and the exposed application behavior. With Podman, a container can exist, be stopped, or be running with incorrect port mappings.

Check:

- Container status
- Restart policy
- Port mappings
- Volume mounts
- Logs
- Whether the service inside the container is reachable

Useful commands:

```bash
podman ps
podman ps -a
podman inspect <container>
podman logs <container>
podman port <container>
```

Example:

```bash
podman run -d --name webtest -p 8080:80 --restart=always registry.access.redhat.com/ubi9/httpd-24
podman ps
podman port webtest
curl -I http://localhost:8080
```

If you use a persistent container service, verify it after reboot or after restarting the user session or system service that manages it. Also confirm the container is still mapped to the expected port and that the application responds.

**Network and hostname verification**

When you change network settings, verify both the configuration and the live state.

Check:

- IP address assignment
- Default route
- DNS resolution
- Hostname
- Interface state

Useful commands:

```bash
ip addr show
ip route
nmcli device status
nmcli connection show
hostnamectl
getent hosts server1
```

If a task depends on name resolution, test it directly:

```bash
ping -c 1 server1.example.com
getent hosts server1.example.com
```

For interface changes, confirm the connection profile is active and persistent:

```bash
nmcli connection show --active
nmcli connection show "<connection-name>"
```

**Exam-style verification checklist**

Use this checklist after completing any RHCSA task:

- Confirm the object exists.
- Confirm the runtime state is correct.
- Confirm the configuration file is correct.
- Confirm the service or feature works now.
- Confirm it survives restart or reboot if required.
- Confirm related dependencies such as SELinux, firewall, and permissions.

A compact command set for final validation:

```bash
systemctl status <service>
systemctl is-enabled <service>
findmnt <mountpoint>
lsblk -f
getenforce
firewall-cmd --list-all
id <user>
chage -l <user>
crontab -l
atq
podman ps
```

**Common verification mistakes**

- Checking only the config file and not the live service.
- Forgetting to test after reboot.
- Using a device path in `/etc/fstab` instead of a stable `UUID=`.
- Verifying a mount point without confirming the underlying LVM activation.
- Assuming SELinux is not involved because the service starts.
- Adding a firewall rule without reloading or checking the active zone.
- Creating a user without confirming the home directory or skeleton files.
- Checking `crontab -l` but not confirming the job actually runs.
- Verifying a container exists but not that it is running or reachable.

**Practical final pass routine**

Before you move on from a task, run a final pass that matches the requirement:

1. Check the service or object state.
2. Test the user-facing behavior.
3. Confirm persistence settings.
4. Reboot or restart if the task requires persistence.
5. Re-check the result after the restart.

Example final pass for a web service backed by storage and SELinux:

```bash
systemctl is-active httpd
systemctl is-enabled httpd
findmnt /webdata
ls -Zd /webdata
firewall-cmd --list-services
curl -I http://localhost
```

This habit turns verification into a repeatable process instead of a last-minute guess. On the RHCSA exam, that discipline is often what separates a partial configuration from a fully correct one.
