## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Why RHCSA tasks fail in practice**

RHCSA tasks usually fail for predictable reasons: you configure the right feature in the wrong place, forget persistence, or verify the wrong layer. The exam rewards careful reading, exact command usage, and validation after every change. Most failures are not caused by lack of knowledge; they come from small mismatches between the requested state and the system state you actually created.

A useful mental model is to check four layers every time:

- **Syntax**: did the command run without error?
- **State**: did the setting take effect now?
- **Persistence**: will it survive reboot?
- **Policy**: is something like SELinux, firewalling, or permissions blocking it?

> If a service “works” only after manual intervention, it is usually not complete for exam purposes.

**Common exam failure patterns**

Many RHCSA mistakes repeat across topics because the same habits cause them.

- **Reading too quickly**: you miss details like hostname, mount point, port number, user name, or file path.
- **Changing the wrong target**: you edit the wrong config file, interface, unit, or filesystem.
- **Forgetting persistence**: you use `ip addr add`, `mount`, or `systemctl start` without the matching persistent configuration.
- **Skipping verification**: you assume the change worked instead of checking with `ip a`, `ss -tulpen`, `systemctl status`, `lsblk`, or `getenforce`.
- **Ignoring SELinux**: a service can be correctly configured but still blocked by labels or booleans.
- **Ignoring the firewall**: the service listens locally but is unreachable from the network.
- **Using the wrong user context**: you run a command as the wrong user or forget `sudo`.
- **Not checking file ownership and mode**: permissions look close but are not exact enough.
- **Breaking existing state**: you overwrite a working config instead of making the minimal required change.

**Command-line mistakes that cost points**

Small CLI errors often produce a valid-looking result that is still wrong.

- **Wrong option order or typo**: many commands accept similar-looking flags, so verify output carefully.
- **Missing quotes**: paths or values with spaces break shell commands.
- **Using relative paths**: exam tasks are safer when you use absolute paths like `/etc/ssh/sshd_config`.
- **Confusing `>` and `>>`**: one overwrites, the other appends.
- **Forgetting `sudo`**: some commands partially work, then fail on protected files.
- **Using the wrong shell expansion**: globbing can affect multiple files unexpectedly.
- **Editing the wrong file with the right content**: for example, changing `/etc/hosts` when the task requires DNS or vice versa.

Practical examples:

```bash
# Check the exact command result and exit status
useradd alice
echo $?

# Verify a file was written as intended
cat /etc/hosts
ls -l /etc/hosts

# Confirm a service is enabled and active
systemctl is-enabled sshd
systemctl is-active sshd
```

**Permissions and ownership mistakes**

File access tasks are a frequent source of partial credit loss because the content may be correct while access is not.

- **Wrong owner**: the file exists, but the required user or group does not own it.
- **Wrong mode**: `640` vs `644` vs `600` matters.
- **Wrong directory permissions**: a file can be readable, but the parent directory blocks traversal.
- **Forgetting group membership**: you add a user to a group but do not re-login or refresh the session.
- **Using ACLs when standard permissions were expected**: access control lists (ACLs) can solve the problem, but they may not match the task intent.
- **Not preserving setgid or sticky behavior**: shared directories often need special bits.

Useful checks:

```bash
ls -l /path/to/file
namei -l /path/to/file
getfacl /path/to/file
id alice
```

**Service management mistakes**

Service tasks often fail because the service is started but not enabled, or the configuration is changed without a reload.

- **Starting without enabling**: `systemctl start` affects only the current boot.
- **Editing config without reload/restart**: many daemons do not reread configuration automatically.
- **Restarting the wrong unit**: the package name and service name are not always identical.
- **Forgetting socket activation behavior**: some services may appear inactive until triggered.
- **Not checking logs**: `journalctl` often shows the real reason for failure.

Typical workflow:

```bash
systemctl status nginx
systemctl enable --now nginx
systemctl restart nginx
journalctl -u nginx --no-pager -n 50
```

**Networking mistakes**

Networking tasks are especially prone to “looks right, but does not persist” failures.

- **Configuring only the runtime state**: `ip addr add` and `ip route add` disappear after reboot.
- **Editing the wrong connection profile**: NetworkManager uses connection profiles, not just interface names.
- **Forgetting to bring the connection up**: a profile can be saved but inactive.
- **Wrong gateway or prefix length**: a single bit error breaks routing.
- **DNS misconfiguration**: the host can ping by IP but not resolve names.
- **Firewall mismatch**: the service listens, but the port is blocked.
- **Interface naming assumptions**: names like `ens160` or `enp0s3` vary by system.

Useful commands:

```bash
nmcli device status
nmcli connection show
nmcli connection modify "System eth0" ipv4.addresses 192.0.2.10/24
nmcli connection modify "System eth0" ipv4.gateway 192.0.2.1
nmcli connection modify "System eth0" ipv4.dns 192.0.2.53
nmcli connection up "System eth0"

ip addr show
ip route show
ss -tulpen
```

**Storage and filesystem mistakes**

Storage tasks often fail because the device, filesystem, mount point, or fstab entry is slightly wrong.

- **Mounting the wrong device**: `/dev/sdb1` vs `/dev/sdb`.
- **Forgetting to format**: a partition exists but has no filesystem.
- **Mounting to a non-empty directory**: existing files become hidden and create confusion.
- **Wrong UUID or device path in `/etc/fstab`**: device names can change across boots.
- **Missing mount options**: `defaults`, `noexec`, `nodev`, or `nosuid` may be required.
- **Not testing `fstab`**: a bad entry can break boot or fail silently until reboot.
- **Confusing swap and data partitions**: each has a different setup path.

Safe workflow:

```bash
lsblk -f
blkid
mkfs.xfs /dev/sdb1
mkdir -p /mnt/data
mount /dev/sdb1 /mnt/data
df -hT /mnt/data
```

Before rebooting, always validate `/etc/fstab`:

```bash
mount -a
echo $?
```

> A failed `mount -a` is a warning sign that the system may not boot cleanly with your storage changes.

**SELinux mistakes**

Security-Enhanced Linux (SELinux) is a common hidden blocker. A service can be correctly installed, configured, and running, yet still fail because the security context is wrong.

- **Disabling SELinux instead of fixing it**: this may not satisfy the task and is usually a bad habit.
- **Wrong file context**: web content, SSH keys, or custom service files may need correct labels.
- **Missing boolean**: some services require a SELinux boolean to allow access.
- **Assuming permissive mode is permanent**: it may only be temporary.
- **Not checking audit logs**: AVC (Access Vector Cache) denials explain many failures.

Useful commands:

```bash
getenforce
sestatus
ls -Z /var/www/html
restorecon -Rv /var/www/html
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
restorecon -Rv /srv/web
getsebool -a | grep httpd
ausearch -m avc -ts recent
```

**Firewall mistakes**

Firewall issues are easy to miss because local testing can succeed while remote access fails.

- **Opening the wrong zone**: the interface may not be in the zone you edited.
- **Opening the wrong service or port**: `http` is not the same as `https`.
- **Forgetting permanent rules**: runtime changes vanish after reload or reboot.
- **Not reloading the firewall**: permanent changes do not always apply immediately.
- **Testing only from localhost**: remote connectivity is the real test.

Example:

```bash
firewall-cmd --get-active-zones
firewall-cmd --zone=public --add-service=http
firewall-cmd --zone=public --add-service=http --permanent
firewall-cmd --reload
firewall-cmd --list-all
```

**User and group management mistakes**

Account tasks often fail because the account exists but is not usable in the intended way.

- **Forgetting to set a password**: the user exists but cannot log in.
- **Confusing primary and supplementary groups**: group ownership and login group are different concepts.
- **Not setting the shell**: a user may be created with `/sbin/nologin` or another restricted shell.
- **Not locking or unlocking intentionally**: account state can override password correctness.
- **Not checking `/etc/skel` expectations**: home directory contents may differ from what you expect.

Verification commands:

```bash
id alice
getent passwd alice
getent group developers
passwd -S alice
chage -l alice
```

**Shell and text-processing mistakes**

Many exam tasks require editing files or extracting data. These fail when you use the right tool in the wrong way.

- **Using `sed` without testing**: a regex can replace more than intended.
- **Appending duplicate lines**: repeated entries in config files can cause ambiguity.
- **Breaking syntax with bad indentation or quoting**: especially in shell scripts and YAML-like config formats.
- **Editing with the wrong line endings**: rare on Linux, but copied files can still cause issues.
- **Not preserving comments or structure when required**: some configs are sensitive to ordering.

Safer editing pattern:

```bash
cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak
vi /etc/ssh/sshd_config
sshd -t
systemctl restart sshd
```

**A practical troubleshooting sequence**

When a task fails, use a repeatable sequence instead of guessing.

1. **Read the requirement again** and identify the exact target state.
2. **Check the current state** with a command that proves what exists now.
3. **Change one thing at a time** so you know what fixed or broke the issue.
4. **Validate the service or feature** from the user’s perspective, not just locally.
5. **Check logs and policy** if the result is still wrong.
6. **Make it persistent** and test after a reboot or connection reload when relevant.

Example workflow for a web service:

```bash
rpm -q httpd
systemctl status httpd
ss -tulpen | grep :80
getenforce
firewall-cmd --list-services
ls -Z /var/www/html
journalctl -u httpd --no-pager -n 50
```

**High-value verification habits**

Good verification prevents most exam mistakes.

- **Use `systemctl is-enabled` and `is-active`** to distinguish persistence from runtime state.
- **Use `mount`, `findmnt`, and `df -hT`** to confirm storage behavior.
- **Use `ip a`, `ip route`, and `nmcli`** to confirm both runtime and persistent networking.
- **Use `ss -tulpen`** to confirm listening sockets.
- **Use `getenforce`, `ls -Z`, and `ausearch`** to confirm SELinux behavior.
- **Use `test -f`, `test -d`, and `echo $?`** when scripting or validating file presence.

**Exam-time habits that reduce failure**

- **Work from the requirement, not memory**: the task wording is the source of truth.
- **Prefer minimal changes**: avoid rewriting configs unless necessary.
- **Verify after each major step**: do not wait until the end.
- **Keep a mental checklist**: permissions, ownership, SELinux, firewall, service state, persistence.
- **Use the right tool for the layer**: `systemctl` for services, `nmcli` for NetworkManager, `firewall-cmd` for firewalld, `semanage` and `restorecon` for SELinux labels.
- **Reboot or reload when persistence matters**: if the task requires it, prove it survives.

**Common “looks correct but fails” scenarios**

- A service is active, but the port is blocked by the firewall.
- A file has the right content, but the wrong SELinux label prevents access.
- A user exists, but the password is locked or the shell is invalid.
- A network address is assigned, but only in the current session.
- A mount works now, but `/etc/fstab` contains a typo and fails on reboot.
- A directory is shared, but the group or setgid bit is missing.
- A config file is edited, but the daemon was never restarted.

**Final mindset**

RHCSA success depends on disciplined execution. Treat every task as a state-management problem: identify the desired state, apply the smallest correct change, and prove the result with the right verification command. If something fails, assume the issue is in the layer you have not checked yet rather than repeating the same command.
