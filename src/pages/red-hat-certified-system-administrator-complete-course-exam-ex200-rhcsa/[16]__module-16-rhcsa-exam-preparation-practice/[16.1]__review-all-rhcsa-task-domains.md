## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**RHCSA task-domain overview**

The RHCSA exam tests whether you can administer a RHEL system from the command line under time pressure. You are expected to configure a system so it works after reboot, verify your changes, and recover when something breaks. The exam is hands-on, so the best preparation is to practice end-to-end tasks rather than memorize isolated commands.

The major domains usually include:
- Essential command-line operations
- User and group administration
- File permissions and access control
- Service management with `systemd`
- Storage and filesystem management
- Networking
- Security controls such as SELinux and firewalld
- Software management and repositories
- Containers with Podman
- Boot, rescue, and troubleshooting workflows

**How to approach exam tasks**

You should think in terms of outcomes:
- Make the change.
- Verify the change.
- Persist the change across reboot.
- Troubleshoot if the first attempt fails.

A good habit is to confirm the current state before editing anything:
```bash
hostnamectl
ip a
lsblk
findmnt
systemctl status sshd
getenforce
firewall-cmd --state
```

**Command-line fundamentals**

You need to move quickly through the shell, inspect files, and manipulate text. Common tasks include locating files, filtering output, redirecting streams, and using archives.

Key tools:
- `man` and `info` for documentation
- `grep`, `cut`, `sort`, `uniq`, `wc`, `head`, `tail`
- `find`, `locate`, `xargs`
- `tar`, `gzip`, `bzip2`, `xz`
- `cp`, `mv`, `rm`, `mkdir`, `touch`, `ln`
- `chmod`, `chown`, `chgrp`, `umask`

Practical examples:
```bash
find /etc -name "*.conf" -type f
grep -R "Listen" /etc/httpd
tar -C /etc -czf /root/etc-backup.tgz myapp
```

Use `-C` with `tar` when you want clean relative paths in the archive. That avoids the common problem of storing absolute paths.

**Text processing and shell behavior**

You should understand redirection and pipelines:
- `>` overwrites a file
- `>>` appends
- `2>` redirects standard error
- `|` passes standard output to the next command
- `tee` writes to a file and the terminal

Example:
```bash
journalctl -u sshd | grep -i failed | tee /root/sshd-failures.txt
```

Shell expansion matters:
- `*` matches multiple characters
- `?` matches one character
- `{a,b}` expands alternatives
- `~` expands to the home directory

**Users, groups, and authentication**

You must create local users and groups, set passwords, and manage account properties. The exam often checks whether you can create a user with the correct supplementary groups, shell, and home directory.

Typical workflow:
1. Create the group if needed.
2. Create the user.
3. Set the password.
4. Verify `/etc/passwd`, `/etc/shadow`, and `/etc/group`.

Examples:
```bash
groupadd developers
useradd -m -G developers -s /usr/bin/bash alice
passwd alice
id alice
```

Use the shell path that exists on the system, usually listed in `/etc/shells`:
```bash
cat /etc/shells
```

Useful account commands:
- `usermod` to change group membership, shell, home, or comment
- `groupmod` to rename a group
- `userdel` and `groupdel` to remove accounts
- `chage` to inspect or adjust password aging

Example:
```bash
usermod -aG developers alice
chage -l alice
```

> Be careful with supplementary groups: `-aG` appends groups. If you omit `-a`, you can accidentally replace the user’s existing supplementary groups.

**File permissions and ACLs**

You need to control access with traditional permissions and access control lists (ACLs, Access Control Lists). Standard permissions are the first layer; ACLs are useful when multiple users or groups need exceptions.

Core commands:
```bash
chmod 640 report.txt
chown alice:developers report.txt
setfacl -m u:bob:rw report.txt
getfacl report.txt
```

Important concepts:
- `rwx` on files and directories behaves differently
- Directory execute permission controls traversal
- The setgid bit on a directory makes new files inherit the directory group
- The sticky bit on shared directories prevents users from deleting files they do not own

Examples:
```bash
chmod g+s /srv/project
chmod +t /srv/shared
```

Use ACLs when ownership and group permissions are not enough:
```bash
setfacl -m g:qa:rx /srv/app
setfacl -x u:bob /srv/app
```

**SELinux basics**

SELinux (Security-Enhanced Linux) is a mandatory access control system. On RHCSA tasks, the most common issue is not the service itself but the file context. If a service cannot read content, check labels before disabling SELinux.

Inspect status:
```bash
getenforce
sestatus
ls -Z /var/www/html
```

Common fix pattern:
1. Put the file in the correct path.
2. Restore the default context.
3. Verify access.

Examples:
```bash
restorecon -Rv /var/www/html
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
restorecon -Rv /srv/web
```

If a service needs a boolean, verify the exact requirement before changing it:
```bash
getsebool -a | grep httpd
setsebool -P httpd_can_network_connect on
```

> Prefer correct labeling over disabling SELinux. Temporary permissive mode is a troubleshooting step, not a solution.

**Service management with systemd**

You must manage services and timers with `systemd`. Know how to start, stop, enable, disable, and inspect units.

Examples:
```bash
systemctl status sshd
systemctl start httpd
systemctl enable --now chronyd
systemctl disable --now postfix
systemctl list-units --type=service
```

Useful troubleshooting commands:
```bash
journalctl -u httpd
journalctl -xe
systemctl cat sshd
systemctl daemon-reload
```

Understand the difference between:
- `start` and `enable`
- runtime state and boot-time persistence
- `restart` and `reload`

If a service supports reload, use it when you want to apply configuration without dropping connections:
```bash
systemctl reload nginx
```

**Networking with NetworkManager**

RHCSA networking tasks usually involve configuring a static IP address, gateway, DNS, hostname, and connection persistence. NetworkManager stores settings in connection profiles, so you should modify the profile rather than only changing the live interface.

Inspect current state:
```bash
nmcli device status
nmcli connection show
ip addr
ip route
```

Set a static address:
```bash
nmcli connection modify "System eth0" ipv4.addresses 192.0.2.10/24
nmcli connection modify "System eth0" ipv4.gateway 192.0.2.1
nmcli connection modify "System eth0" ipv4.method manual
nmcli connection up "System eth0"
```

Set DNS and search domains persistently:
```bash
nmcli connection modify "System eth0" ipv4.dns 192.0.2.53
nmcli connection modify "System eth0" +ipv4.dns 192.0.2.54
nmcli connection modify "System eth0" ipv4.dns-search example.com
nmcli connection up "System eth0"
```

Set the hostname:
```bash
hostnamectl set-hostname server1.example.com
```

Verify persistence by checking the connection profile and rebooting if time allows:
```bash
nmcli connection show "System eth0"
cat /etc/resolv.conf
```

**Networking troubleshooting mindset**

If networking fails, check in this order:
- Link and device state
- IP address and route
- DNS settings
- Firewall rules
- Service binding

Useful commands:
```bash
ping -c 3 192.0.2.1
ping -c 3 example.com
ss -tulpn
```

**Firewall management with firewalld**

You need to open services and ports with `firewalld`. Remember that runtime changes are not enough if the task requires persistence.

Examples:
```bash
firewall-cmd --get-active-zones
firewall-cmd --add-service=http
firewall-cmd --permanent --add-service=http
firewall-cmd --reload
firewall-cmd --list-all
```

Open a custom port:
```bash
firewall-cmd --permanent --add-port=8080/tcp
firewall-cmd --reload
```

If a service still fails after opening the firewall, verify:
- The service is running
- The service listens on the expected port
- SELinux allows the access
- The correct zone is active on the interface

**Package management and repositories**

You should manage packages with `dnf`, including installing, removing, querying, and updating software. RHCSA tasks often require enabling a local repository from installation media or a mounted ISO.

Common commands:
```bash
dnf install httpd
dnf remove telnet
dnf list installed
dnf info podman
dnf provides */semanage
```

Create a local repository from mounted installation media:
```bash
mkdir -p /mnt/rhel
mount /dev/sr0 /mnt/rhel
```

Typical repository files for RHEL DVD media:
```ini
# /etc/yum.repos.d/baseos.repo
[BaseOS]
name=BaseOS
baseurl=file:///mnt/rhel/BaseOS
enabled=1
gpgcheck=1
gpgkey=file:///mnt/rhel/RPM-GPG-KEY-redhat-release

# /etc/yum.repos.d/appstream.repo
[AppStream]
name=AppStream
baseurl=file:///mnt/rhel/AppStream
enabled=1
gpgcheck=1
gpgkey=file:///mnt/rhel/RPM-GPG-KEY-redhat-release
```

Import and verify the GPG key:
```bash
rpm --import /mnt/rhel/RPM-GPG-KEY-redhat-release
rpm -qa gpg-pubkey\*
```

Module streams matter for packages delivered as application streams:
```bash
dnf module list nodejs
dnf module enable nodejs:18
dnf install nodejs
```

Understand the difference between:
- `dnf install` for packages
- `dnf module enable` for stream selection
- `dnf module reset` when you need to change streams

**Storage fundamentals**

Storage is one of the most important RHCSA domains. You should be able to identify disks, partition them, create filesystems, mount them persistently, and extend storage when needed.

Inspect storage:
```bash
lsblk
blkid
fdisk -l
parted -l
df -hT
```

Create a partition and filesystem:
```bash
parted /dev/sdb mklabel gpt
parted /dev/sdb mkpart primary 1MiB 100%
mkfs.xfs /dev/sdb1
mkdir -p /data
mount /dev/sdb1 /data
```

Make the mount persistent with UUID:
```bash
blkid /dev/sdb1
```

Example `/etc/fstab` entry:
```fstab
UUID=1234-ABCD  /data  xfs  defaults  0  0
```

Test `fstab` safely:
```bash
mount -a
findmnt /data
```

> Always test `/etc/fstab` before rebooting. A bad entry can drop you into emergency mode.

**LVM and filesystem growth**

Logical Volume Manager (LVM) is common on RHEL systems. You should know how to create physical volumes, volume groups, and logical volumes, then grow them safely.

Example:
```bash
pvcreate /dev/sdb2
vgcreate vgdata /dev/sdb2
lvcreate -n lvapp -L 2G vgdata
mkfs.xfs /dev/vgdata/lvapp
mount /dev/vgdata/lvapp /app
```

Extend an LVM-backed XFS filesystem:
```bash
lvextend -r -L +1G /dev/vgdata/lvapp
```

If you extend without `-r`, grow the filesystem separately:
```bash
lvextend -L +1G /dev/vgdata/lvapp
xfs_growfs /app
```

For ext4, use `resize2fs` after extending the logical volume:
```bash
lvextend -L +1G /dev/vgdata/lvapp
resize2fs /dev/vgdata/lvapp
```

**Non-LVM resizing workflows**

You may also need to handle partitions directly. The exact workflow depends on the filesystem and partition table.

General pattern:
1. Expand the underlying disk or partition.
2. Re-read the partition table if needed.
3. Grow the filesystem with the correct tool.

Examples:
- XFS: `xfs_growfs /mountpoint`
- ext4: `resize2fs /dev/partition`

If the partition is not LVM-backed, be careful to confirm the device path before resizing.

**Swap management**

You should create and activate swap, and make it persistent.

Example:
```bash
mkswap /dev/sdb3
swapon /dev/sdb3
swapon --show
```

Persistent `/etc/fstab` entry:
```fstab
UUID=5678-EFGH  none  swap  defaults  0  0
```

**Boot process and recovery**

RHCSA includes recovery tasks such as fixing a broken boot, resetting a root password, repairing `/etc/fstab`, and relabeling SELinux contexts. You should know how to use GRUB2 and emergency or rescue modes.

Common recovery goals:
- Reset the root password
- Fix a bad `fstab`
- Recover from a failed boot after a storage change
- Relabel filesystems after a context problem

To edit a boot entry in GRUB2:
1. Reboot and interrupt the menu.
2. Highlight the kernel entry.
3. Press `e`.
4. Append the needed kernel arguments.
5. Boot with <kbd>Ctrl</kbd> + <kbd>X</kbd> or <kbd>F10</kbd>.

Useful temporary kernel arguments:
- `rd.break` to drop into an early emergency shell
- `enforcing=0` to boot with SELinux permissive for troubleshooting
- `selinux=0` only if you need SELinux fully disabled for a specific recovery case

Reset root password from a break shell:
```bash
mount -o remount,rw /sysroot
chroot /sysroot
passwd root
touch /.autorelabel
exit
reboot
```

If `/etc/fstab` is broken, mount the root filesystem, edit the file, and test before rebooting:
```bash
mount -o remount,rw /sysroot
chroot /sysroot
vi /etc/fstab
mount -a
```

If the system drops to emergency mode, inspect logs and mounts:
```bash
journalctl -xb
lsblk
cat /etc/fstab
```

**GRUB2 password recovery and bootloader awareness**

You should understand that GRUB2 can be protected with a password and that bootloader changes may be required in secured environments. If you forget the GRUB password, recovery usually requires authorized console access and administrative procedures specific to the environment. On the exam, the practical skill is often to edit boot parameters, not to bypass security controls.

**Containers with Podman**

RHCSA container tasks focus on running and managing containers without a daemon. Podman is rootless-capable and uses OCI (Open Container Initiative) images.

Inspect images and containers:
```bash
podman images
podman ps -a
podman inspect myweb
```

Pull and run an image that is available in your environment:
```bash
podman pull registry.access.redhat.com/ubi9/nginx-120
podman run -d --name myweb -p 8080:8080 registry.access.redhat.com/ubi9/nginx-120
```

If a registry requires authentication:
```bash
podman login registry.example.com
```

Use volumes for persistent data:
```bash
podman volume create webdata
podman run -d --name myweb -v webdata:/usr/share/nginx/html:Z -p 8080:8080 registry.access.redhat.com/ubi9/nginx-120
```

Important container concepts:
- `-v` mounts persistent storage
- `:Z` relabels content for SELinux when needed
- `--rm` removes a container when it exits
- `--name` makes management easier
- `-p` publishes ports

Example with a bind mount:
```bash
mkdir -p /srv/web
echo "Hello from Podman" > /srv/web/index.html
podman run -d --name webtest -p 8080:8080 -v /srv/web:/usr/share/nginx/html:Z registry.access.redhat.com/ubi9/nginx-120
```

**Container persistence and systemd integration**

For exam purposes, you may need a container to start at boot. A common approach is to create a systemd user service or a system service that starts the container. Podman also supports generated units, though newer workflows may prefer Quadlet on newer RHEL releases.

Example generated unit workflow:
```bash
podman run -d --name myweb -p 8080:8080 registry.access.redhat.com/ubi9/nginx-120
podman generate systemd --name myweb --files
```

Then place the unit where systemd can use it and enable it:
```bash
systemctl --user daemon-reload
systemctl --user enable --now container-myweb.service
```

If you use rootless containers, remember:
- The user must have a valid login session or lingering enabled if the service should run without login
- `loginctl enable-linger alice` can help for persistent user services
- SELinux labeling still matters for mounted content

**Practical exam scenarios**

These scenarios combine multiple domains the way the exam often does.

1. Configure a web service with persistent access
   - Install `httpd`
   - Place content in the correct document root
   - Fix SELinux labels with `restorecon` or `semanage fcontext`
   - Open the firewall permanently
   - Enable and start the service
   - Verify with `curl`

   Example:
   ```bash
   dnf install -y httpd
   echo "RHCSA" > /var/www/html/index.html
   restorecon -Rv /var/www/html
   firewall-cmd --permanent --add-service=http
   firewall-cmd --reload
   systemctl enable --now httpd
   curl http://localhost
   ```

2. Add a new filesystem and mount it persistently
   - Identify the disk
   - Partition it
   - Create the filesystem
   - Mount it
   - Add the UUID to `/etc/fstab`
   - Test with `mount -a`

   Example:
   ```bash
   lsblk
   parted /dev/sdb mklabel gpt
   parted /dev/sdb mkpart primary 1MiB 100%
   mkfs.xfs /dev/sdb1
   mkdir -p /data
   blkid /dev/sdb1
   echo "UUID=... /data xfs defaults 0 0" >> /etc/fstab
   mount -a
   ```

3. Recover from a bad boot configuration
   - Use `rd.break` or rescue mode
   - Mount the root filesystem read-write
   - Fix `/etc/fstab` or reset the root password
   - Relabel if needed
   - Reboot and verify

4. Configure a container with persistent content
   - Pull a valid image from an accessible registry
   - Create a volume or bind mount
   - Publish the port
   - Verify the service with `curl`
   - Add a systemd unit if persistence is required

**Verification habits**

Verification is part of the task, not an optional extra. After every change, check the live state and the persistent configuration.

Useful checks:
```bash
id alice
getfacl /srv/project
systemctl is-enabled httpd
nmcli connection show "System eth0"
firewall-cmd --list-all
ls -Z /var/www/html
mount | grep /data
```

**Common exam pitfalls**

- Changing only the runtime firewall and forgetting `--permanent`
- Editing `/etc/resolv.conf` directly instead of the NetworkManager profile
- Using the wrong SELinux fix when a label change is enough
- Forgetting `-a` with `usermod -aG`
- Mounting storage but not adding `/etc/fstab`
- Extending an XFS filesystem without running `xfs_growfs` or using `lvextend -r`
- Using a container image that is not available in the exam environment
- Skipping verification after a reboot-sensitive change

**Final preparation strategy**

Practice each domain as a repeatable workflow:
- Read the task carefully.
- Identify the target object: user, service, mount, port, or container.
- Make the smallest correct change.
- Verify immediately.
- Persist the change if the task requires it.
- Recheck after a reboot when possible.

If you can complete tasks without relying on a GUI and can recover from mistakes calmly, you are preparing in the right way for EX200.
