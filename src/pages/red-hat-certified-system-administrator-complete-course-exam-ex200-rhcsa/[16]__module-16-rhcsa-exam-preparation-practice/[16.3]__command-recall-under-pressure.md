## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**What this topic is for**

RHCSA tasks are time-limited and hands-on, so success depends on recalling the right command family quickly and using it in the right order. You are rarely asked to explain theory; you are asked to make a system work, verify it, and recover when the first attempt fails.

The best exam strategy is to memorize command patterns, not isolated commands. That means knowing how to inspect state, change state, verify the result, and troubleshoot with logs or status output.

**Core command workflow**

Most RHCSA tasks follow the same loop:

1. Inspect the current state.
2. Make the smallest safe change.
3. Verify the result.
4. Recheck logs or configuration if the result is wrong.

Common inspection commands include:

```bash
hostnamectl
ip addr
ip route
ss -tulpen
systemctl status sshd
journalctl -u sshd -b
lsblk
blkid
mount
df -h
getenforce
id
```

Use this pattern under pressure:

- **Inspect first** before editing files or restarting services.
- **Change one thing at a time** so you can identify what fixed the issue.
- **Verify with a command that proves the service is actually working**, not just enabled.

**Shell basics you must recall quickly**

Redirection and pipes are used constantly in RHCSA tasks, especially when you need to capture output, append logs, or chain commands.

```bash
command > file.txt
command >> file.txt
command 2> errors.txt
command | less
command | tee output.txt
grep 'pattern' file.txt
sort file.txt | uniq -c
```

Key behaviors:

- `>` overwrites a file.
- `>>` appends to a file.
- `|` sends standard output to the next command.
- `tee` writes output to a file and still prints it to the terminal.
- `2>` captures standard error separately from standard output.

Useful text-processing commands:

```bash
cut -d: -f1 /etc/passwd
awk -F: '{print $1, $3}' /etc/passwd
sed -n '1,5p' file.txt
grep -R "Listen" /etc/httpd
```

**File search and location commands**

You often need to find a file, a binary, or a package-provided path quickly.

```bash
find /etc -name '*.conf'
find /var -type f -mtime -1
locate httpd.conf
which python3
whereis sshd
```

Use them like this:

- **find** searches the filesystem in real time and is best when you know a directory tree.
- **locate** is fast but depends on an updated database.
- **which** shows the binary that will run from your `PATH`.
- **whereis** shows binary, source, and man page locations.

If `locate` returns nothing, update the database first:

```bash
updatedb
locate fstab
```

**Archive and compression commands**

RHCSA tasks may require you to bundle files, extract backups, or move data between systems. Know the common archive tools and when to use them.

```bash
tar -cvf backup.tar /etc/ssh
tar -xvf backup.tar
tar -czvf backup.tar.gz /var/log
tar -xzvf backup.tar.gz
gzip file.txt
gunzip file.txt.gz
bzip2 file.txt
bunzip2 file.txt.bz2
xz file.txt
unxz file.txt.xz
```

Practical notes:

- `tar` is the main archive tool.
- `gzip`, `bzip2`, and `xz` compress single files.
- `tar` is often combined with compression flags for directories and backups.
- On the exam, read the filename extension carefully so you choose the right extraction command.

**Service management and logs**

Systemd is central to RHCSA. You should know how to start, stop, enable, disable, and inspect services.

```bash
systemctl status httpd
systemctl start httpd
systemctl stop httpd
systemctl restart httpd
systemctl enable httpd
systemctl disable httpd
systemctl is-enabled httpd
systemctl is-active httpd
```

Important distinctions:

- `start` affects the current boot only.
- `enable` makes the service start at boot.
- `restart` stops and starts the service.
- `reload` asks a service to re-read configuration without a full restart, if supported.

For troubleshooting, use service-specific logs rather than generic logs when possible:

```bash
journalctl -u httpd -b
journalctl -xeu httpd
journalctl -u sshd --since today
```

Use `journalctl -xeu service` when a service fails to start, because it combines recent errors with the unit’s logs. `journalctl -b` limits output to the current boot.

> `systemctl daemon-reload` is for reloading systemd unit definitions after you change a unit file, not for ordinary application configuration changes.

**Networking recall**

Networking tasks often require you to inspect the current state, modify a connection profile, and then bring the profile back up or reapply it.

```bash
ip addr show
ip link show
ip route show
nmcli device status
nmcli connection show
nmcli connection show "System eth0"
hostname
hostnamectl
```

Common `nmcli` patterns:

```bash
nmcli connection add type ethernet ifname eth0 con-name office ipv4.addresses 192.0.2.10/24 ipv4.gateway 192.0.2.1 ipv4.dns 192.0.2.53 ipv4.method manual
nmcli connection modify office ipv4.addresses 192.0.2.20/24
nmcli connection modify office ipv4.gateway 192.0.2.1
nmcli connection modify office ipv4.dns "192.0.2.53 192.0.2.54"
nmcli connection modify office ipv4.method manual
nmcli connection down office
nmcli connection up office
```

When you change a live profile:

- Use `nmcli connection down/up` if the change affects the profile and you can safely bounce it.
- Use `nmcli device reapply eth0` when you want NetworkManager to push compatible changes without a full disconnect.
- Some changes, such as IP address or gateway updates, may require a reconnect to take effect.

DNS troubleshooting commands:

```bash
cat /etc/resolv.conf
nslookup example.com
dig example.com
dig @192.0.2.53 example.com
```

Be careful with `/etc/resolv.conf`:

- On RHEL 8/9 it is often managed by NetworkManager.
- It may be a symlink or dynamically generated file.
- For persistent DNS changes, prefer `nmcli` rather than editing the file directly.

**Firewall commands**

Firewalld is commonly used in RHCSA tasks. Know how to inspect zones and open services or ports.

```bash
firewall-cmd --state
firewall-cmd --get-active-zones
firewall-cmd --list-all
firewall-cmd --permanent --add-service=http
firewall-cmd --permanent --add-port=8080/tcp
firewall-cmd --reload
```

Useful habits:

- Add permanent rules when the change must survive reboot.
- Reload after permanent changes so the runtime configuration matches the saved configuration.
- Verify with `--list-all` or `--list-services`.

**SELinux recall**

SELinux is a frequent source of exam points because the service may be configured correctly but still blocked by policy. You need to inspect labels, booleans, ports, and mode.

```bash
getenforce
setenforce 0
setenforce 1
ls -Z
ps -Z
semanage fcontext -l
semanage port -l
getsebool -a
```

Important details:

- `setenforce` changes runtime mode only.
- It works only when SELinux is enabled at boot and currently in enforcing or permissive mode.
- If SELinux is disabled in the kernel boot parameters, `setenforce` cannot turn it on.

Common file-label workflow:

```bash
mkdir -p /web
cp index.html /web/
semanage fcontext -a -t httpd_sys_content_t '/web(/.*)?'
restorecon -Rv /web
```

If `semanage` is missing, install the package that provides it:

```bash
dnf install -y policycoreutils-python-utils
```

Common troubleshooting pattern:

- Check the file context with `ls -Z`.
- Check the process context with `ps -Z`.
- Check booleans with `getsebool -a | grep httpd`.
- Check denials in the journal or audit logs if access still fails.

Example boolean change:

```bash
setsebool -P httpd_can_network_connect on
```

Use `-P` to make the boolean persistent across reboots.

**Storage and filesystem recall**

Storage tasks often require you to identify disks, partition them, create filesystems, mount them, and make the mount persistent.

```bash
lsblk
blkid
fdisk -l
parted -l
mkfs.xfs /dev/sdb1
mkfs.ext4 /dev/sdb1
mount /dev/sdb1 /mnt/data
umount /mnt/data
df -h
```

RHCSA commonly uses XFS on RHEL, so prefer `mkfs.xfs` unless the task explicitly asks for another filesystem.

Persistent mount example:

```bash
UUID=1234-ABCD  /data  xfs  defaults  0 0
```

Workflow reminder:

1. Create or identify the block device.
2. Make the filesystem.
3. Create the mount point.
4. Mount it.
5. Add the `fstab` entry.
6. Test with `mount -a`.

If `mount -a` fails, fix the entry before rebooting.

**LVM recall**

Logical Volume Manager (LVM) is a major RHCSA topic and should be memorized as a command family, not as isolated commands.

```bash
pvcreate /dev/sdb
vgcreate vgdata /dev/sdb
lvcreate -n lvapp -L 2G vgdata
lvs
vgs
pvs
mkfs.xfs /dev/vgdata/lvapp
mount /dev/vgdata/lvapp /app
```

To extend storage:

```bash
lvextend -L +1G /dev/vgdata/lvapp
xfs_growfs /app
```

Key points:

- Use `pvcreate` to initialize a physical volume.
- Use `vgcreate` to build a volume group.
- Use `lvcreate` to create a logical volume.
- Use `lvs`, `vgs`, and `pvs` to verify the layout.
- For XFS, grow the filesystem with `xfs_growfs` after extending the LV.

**User, group, and permission commands**

User and permission tasks are common and usually straightforward if you remember the command families.

```bash
useradd alice
usermod -aG wheel alice
passwd alice
groupadd devops
id alice
chage -l alice
```

Permissions and ownership:

```bash
chmod 640 file.txt
chown alice:devops file.txt
chgrp devops file.txt
umask
```

Special permission examples:

```bash
chmod u+s /usr/local/bin/tool
chmod g+s /shared
chmod +t /shared
```

Use these carefully:

- `setuid` on executables runs the program with the file owner’s privileges.
- `setgid` on directories helps new files inherit the group.
- The sticky bit on shared directories prevents users from deleting files they do not own.

**Software management**

Package management is a frequent exam task, especially when a service or tool is missing.

```bash
dnf search httpd
dnf info httpd
dnf install -y httpd
dnf remove -y httpd
dnf update -y
dnf list installed
dnf provides */semanage
```

Repository and module recall:

```bash
dnf repolist
dnf module list
dnf module enable nodejs:18
dnf module reset nodejs
```

Local RPM installation:

```bash
dnf install -y ./package.rpm
rpm -qf /usr/sbin/sshd
rpm -ql httpd
rpm -ivh package.rpm
rpm -Uvh package.rpm
```

Practical notes:

- Use `dnf` for normal package management.
- Use `rpm` for direct package queries or local file installation when required.
- If a command is missing, identify the package with `dnf provides`.

**Boot and recovery recall**

Some RHCSA workflows involve boot troubleshooting, kernel parameters, or rescue mode. You do not need to memorize every bootloader detail, but you should know the core tools.

```bash
grubby --default-kernel
grubby --info=ALL
grubby --update-kernel=ALL --args="selinux=0"
grubby --update-kernel=ALL --remove-args="selinux=0"
```

Useful concepts:

- `grubby` manages kernel arguments on systems that use it.
- Inspect current kernel arguments before changing them.
- Temporary boot edits are often made from the GRUB menu during recovery.
- Rescue mode is used when the system cannot boot normally.
- Emergency mode gives a minimal environment for fixing critical issues.

If you need to inspect the current booted kernel:

```bash
uname -r
cat /proc/cmdline
```

**Scheduling commands**

Cron and one-time jobs are common recall items.

```bash
crontab -e
crontab -l
systemctl status crond
at 14:30
atq
atrm 1
```

Cron examples:

```bash
0 2 * * * /usr/local/bin/backup.sh
```

Use `at` for one-time execution and `cron` for recurring jobs. Remember that scheduled jobs often fail because of missing paths or environment variables, so use full paths in scripts.

**Process and job control**

You may need to identify, stop, or prioritize processes.

```bash
ps aux
ps -ef
top
pgrep sshd
pkill -HUP sshd
kill -15 1234
kill -9 1234
nice -n 10 command
renice 5 -p 1234
```

Guidance:

- Prefer `kill -15` first because it allows graceful shutdown.
- Use `kill -9` only when a process ignores normal termination.
- `pgrep` and `pkill` are faster than manual `ps | grep` workflows.

**Practical troubleshooting patterns**

When a task fails, use the smallest set of commands that proves the failure point.

Service not reachable:

```bash
systemctl status httpd
journalctl -xeu httpd
ss -tulpen
firewall-cmd --list-all
getenforce
ls -Z /var/www/html
```

Network change not applied:

```bash
nmcli connection show office
nmcli device status
ip addr show
nmcli device reapply eth0
nmcli connection down office && nmcli connection up office
```

Mount not persistent:

```bash
mount -a
cat /etc/fstab
blkid
lsblk -f
```

Permission issue:

```bash
ls -l
ls -Z
id
getfacl file.txt
```

If the problem is still unclear, narrow the scope:

- Check the service log.
- Check the firewall.
- Check SELinux.
- Check the file label or port label.
- Check whether the configuration file syntax is valid.

**How to study this topic efficiently**

Do not memorize commands in isolation. Practice them in short scenarios so your brain links the task to the command family.

1. Create a file-sharing task and solve it with permissions, SELinux, and firewall changes.
2. Create a mount task and verify it with `mount`, `df`, and `/etc/fstab`.
3. Break a service intentionally and recover it with `systemctl`, `journalctl`, and `ss`.
4. Change a NetworkManager profile and confirm the result with `ip addr` and `nmcli`.
5. Install a missing package, then use `dnf provides` or `rpm -qf` to confirm where a command came from.

The exam rewards speed, but speed comes from a stable mental map: inspect, change, verify, troubleshoot. If you can recall the right command family under pressure, you can usually recover the rest.
