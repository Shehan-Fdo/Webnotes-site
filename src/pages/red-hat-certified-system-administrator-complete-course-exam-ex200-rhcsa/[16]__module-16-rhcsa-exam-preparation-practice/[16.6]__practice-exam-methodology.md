## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**What this topic is really about**

The RHCSA exam is not a theory test; it is a timed, hands-on administration exercise. Your methodology matters as much as your technical knowledge because you must complete tasks, verify them, and recover quickly when something does not work on the first attempt.

A strong exam method helps you:
- avoid wasting time on low-value troubleshooting
- complete easy tasks first and bank points early
- verify each change before moving on
- keep your system in a known-good state

**How to approach the exam**

Treat the exam like a controlled change window. Read every task carefully, identify dependencies, and decide whether a task is safe to complete immediately or should wait until prerequisite work is done.

A practical workflow is:
1. Read all tasks once before making changes.
2. Mark tasks that depend on storage, networking, users, or services.
3. Complete quick wins first.
4. Verify each task as soon as you finish it.
5. Return to difficult tasks with a clean mental model of the system.

This approach reduces rework. Many RHCSA tasks are independent, but some build on earlier configuration, such as storage mounts, SELinux labels, firewall rules, and service enablement.

**Time management strategy**

Time pressure is one of the biggest exam risks. You do not need to solve tasks in the order presented. You need to maximize points within the time limit.

Use a triage model:
- **Immediate tasks** are short, familiar, and low-risk.
- **Dependent tasks** require another task to be completed first.
- **Troublesome tasks** are unfamiliar or likely to require debugging.

A good rule is to start with tasks you can finish in a few minutes, then move to medium-complexity items, and leave the most uncertain tasks for a second pass. This keeps momentum and prevents one difficult problem from consuming the entire exam.

**Initial system survey**

Before changing anything, inspect the environment so you know what you are working with. This is especially useful when the exam system differs slightly from your practice lab.

Useful commands:
```bash
hostnamectl
ip a
ip r
nmcli device status
lsblk
df -hT
getenforce
systemctl --failed
```

What to look for:
- active network interfaces and IP addresses
- available disks, partitions, and mount points
- current SELinux mode
- failed services that may affect later tasks
- filesystem types already in use

If a task depends on a hostname, interface, or disk name, confirm the exact value before editing configuration.

**Task reading and dependency mapping**

Many exam mistakes come from acting too early. Read the wording carefully and identify:
- the target object
- the required state
- the verification method
- any implied dependency

Examples:
- If a task asks for a service to be reachable on a port, you may need the service, firewall, and SELinux configuration.
- If a task asks for a persistent mount, you need the filesystem, mount point, and `/etc/fstab` entry.
- If a task asks for a user with a specific shell and group membership, you need to create the group before assigning it if the group does not already exist.

A simple note-taking pattern helps:
- object: `user`, `group`, `service`, `mount`, `firewall`, `SELinux`
- action: `create`, `modify`, `enable`, `start`, `allow`, `persist`
- verification: `id`, `systemctl status`, `ss`, `mount`, `ls -Z`, `firewall-cmd`

**Command discipline**

Use commands that are explicit and repeatable. Avoid guessing when a command can show you the current state first.

Examples:
```bash
id student1
getent passwd student1
getent group developers
systemctl status sshd.service
firewall-cmd --list-all
semanage port -l | grep http_port_t
```

Good exam habits:
- prefer `systemctl status` before restarting a service
- inspect current firewall and SELinux settings before changing them
- confirm existing users, groups, and mounts before creating duplicates
- use `man` pages when a command syntax is uncertain

> Do not assume a command succeeded just because it returned to the prompt. Verify the result.

**Verification methodology**

Verification is part of the task, not an optional final step. The exam rewards correct state, not just command execution.

Use verification that matches the task:
- for services, check `systemctl status` and `systemctl is-enabled`
- for listening sockets, use `ss -ltnp` or `ss -lunp` rather than a narrow grep pattern
- for mounts, use `findmnt`, `mount`, and `df -hT`
- for SELinux, use `getenforce`, `ls -Z`, and `semanage` queries
- for users and groups, use `id`, `getent passwd`, and `getent group`

Examples:
```bash
systemctl is-active sshd.service
systemctl is-enabled sshd.service
ss -ltnp
findmnt /data
ls -Z /var/www/html
id alice
```

A narrow command like `ss -tulpn | grep :80` can miss relevant details or match the wrong process. Prefer a broader socket listing and then inspect the exact line you need.

**Service management workflow**

For service tasks, think in terms of state and persistence:
- start the service now
- enable it for boot if required
- confirm it is active
- confirm it is enabled

Typical commands:
```bash
systemctl enable --now sshd.service
systemctl status sshd.service
systemctl is-enabled sshd.service
```

Use the full unit name when you want to be explicit. On RHCSA systems, `sshd.service` is the SSH daemon unit you should expect to manage.

If a service fails:
1. inspect the status output
2. read recent logs with `journalctl -xeu <service>`
3. check configuration syntax
4. confirm dependencies such as ports, files, or permissions
5. restart after fixing the root cause

Example:
```bash
journalctl -xeu httpd
apachectl configtest
systemctl restart httpd
```

**Networking methodology**

Networking tasks often require both persistent configuration and immediate activation. Use `nmcli` for most RHCSA network changes because it is reliable and scriptable.

Common workflow:
```bash
nmcli con show
nmcli dev status
nmcli con mod "System eth0" ipv4.addresses 192.0.2.10/24
nmcli con mod "System eth0" ipv4.gateway 192.0.2.1
nmcli con mod "System eth0" ipv4.dns 192.0.2.53
nmcli con mod "System eth0" ipv4.method manual
nmcli con up "System eth0"
```

Method tips:
- identify the correct connection profile name first
- modify the profile, not just the live interface
- bring the connection up after changes
- verify with `ip a`, `ip r`, and `nmcli dev show`

If DNS is part of the task, confirm name resolution with a real lookup:
```bash
getent hosts example.internal
```

**Storage methodology**

Storage tasks are high-value and often require careful sequencing. On RHCSA, XFS is the default filesystem you are most likely to use for many tasks, and it cannot be shrunk. Plan before formatting.

Typical workflow:
1. identify the correct disk or partition with `lsblk`
2. create the partition if needed
3. create the filesystem
4. create the mount point
5. mount it immediately
6. add the persistent entry in `/etc/fstab`
7. verify with `findmnt` and `df -hT`

Example:
```bash
lsblk
mkfs.xfs /dev/vdb1
mkdir -p /data
mount /dev/vdb1 /data
blkid /dev/vdb1
```

Persistent mount example:
```fstab
UUID=12345678-1234-1234-1234-123456789abc  /data  xfs  defaults  0  0
```

Useful checks:
```bash
findmnt /data
mount -a
df -hT /data
```

If a mount fails after editing `/etc/fstab`, do not keep guessing. Read the error, confirm the UUID or device path, and check filesystem type and mount options.

**SELinux methodology**

SELinux is often the hidden cause of a service or access problem. When a task fails despite correct permissions, check SELinux before changing unrelated settings.

Useful commands:
```bash
getenforce
ls -Z /var/www/html
ausearch -m avc -ts recent
semanage fcontext -l | grep /var/www
restorecon -Rv /var/www/html
```

For port-related tasks, inspect the current mapping before changing it:
```bash
semanage port -l | grep http_port_t
```

If you need to allow a service on a non-default port, first determine whether the port is already mapped and to what type. Then choose the correct action:
- add a new mapping if the port is unmapped
- modify an existing mapping if the port is already assigned
- remove the old mapping if it conflicts with the required type

Example workflow:
```bash
semanage port -l | grep 8080
semanage port -a -t http_port_t -p tcp 8080
# if already mapped to the wrong type, inspect first, then modify or delete as appropriate
```

For file context issues, use persistent labeling rather than temporary fixes:
```bash
semanage fcontext -a -t httpd_sys_content_t '/web(/.*)?'
restorecon -Rv /web
```

**Firewall methodology**

Firewall tasks are usually straightforward, but they must be verified. Open the correct service or port and ensure the change is persistent.

Examples:
```bash
firewall-cmd --get-active-zones
firewall-cmd --zone=public --add-service=http --permanent
firewall-cmd --zone=public --add-port=8080/tcp --permanent
firewall-cmd --reload
firewall-cmd --list-all
```

Guidelines:
- use services when a predefined service exists
- use ports when the service is nonstandard
- reload after permanent changes
- verify the active zone before assuming the rule applied where you expected

**User, group, and permission methodology**

Account and permission tasks are usually easy points if you work carefully. Create the identity, assign membership, and verify the result.

Examples:
```bash
groupadd developers
useradd -m -G developers alice
passwd alice
id alice
```

For file permissions:
```bash
chown alice:developers /shared/report.txt
chmod 0640 /shared/report.txt
```

For default access control, remember that ownership and mode bits are only part of the picture. If the task involves shared directories, set the correct group ownership and consider the setgid bit:
```bash
chmod 2775 /shared
```

**Archiving and text processing methodology**

These tasks reward precision and speed. Use the right tool for the job and avoid overcomplicating simple transformations.

Examples:
```bash
tar -czf backup.tgz /etc/ssh
tar -xzf backup.tgz -C /restore
grep -R "Listen" /etc/httpd
sed -i 's/old/new/g' file.txt
awk -F: '{print $1}' /etc/passwd
```

Practical habits:
- test a command on a small file before editing a critical config
- keep backups of files you modify
- use redirection carefully so you do not overwrite the wrong file

**Container methodology**

Container tasks may involve running a container, exposing a port, or ensuring it starts after reboot. Focus on the task requirement rather than a single tool.

Examples:
```bash
podman pull registry.access.redhat.com/ubi9/httpd-24
podman run -d --name web -p 8080:8080 registry.access.redhat.com/ubi9/httpd-24
podman ps
podman logs web
```

For persistence, use the method requested by the task or the environment standard. Depending on the scenario, that may involve systemd integration or a Quadlet file rather than relying only on a manual `podman run` command.

**Troubleshooting loop**

When something fails, use a short, repeatable loop:
1. identify the symptom
2. check the most likely layer first
3. inspect logs or status output
4. change one thing
5. verify again

This prevents random edits that create new problems. For example, if a web service is unreachable:
- confirm the service is running
- confirm the port is listening
- confirm the firewall allows it
- confirm SELinux is not blocking it
- confirm the content path and permissions are correct

**Exam-day habits**

Small habits improve your score:
- keep a scratchpad of task numbers and status
- save time by using tab completion and command history
- avoid rebooting unless a task requires it or you have exhausted simpler checks
- do not spend too long on one task before moving on
- re-read the exact wording before declaring a task complete

A useful mental model is: configure, verify, persist, and re-verify. If you cannot prove the state, the task is not done.

**Practical mini-checklist**

Use this quick checklist during the exam:
- read all tasks first
- identify dependencies
- complete easy tasks early
- verify every change
- inspect logs before guessing
- use persistent configuration where required
- confirm SELinux, firewall, and service state
- return for a final review of unfinished items

**Common pitfalls to avoid**

- using the wrong service unit name
- editing only the live network state instead of the connection profile
- forgetting to make a mount persistent
- assuming a port is open without checking firewall and SELinux
- relying on a narrow verification command that misses the real issue
- formatting storage before confirming the correct device
- skipping verification after a successful-looking command

**Final exam mindset**

The RHCSA exam rewards calm execution. You do not need to be perfect on the first command; you need to be systematic. Read carefully, change deliberately, verify immediately, and move on once the task is proven complete.
