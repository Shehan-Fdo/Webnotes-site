## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Purpose of the final readiness checklist**

The RHCSA exam rewards consistent execution under time pressure. Your final review should focus on verifying that you can complete common administration tasks from memory, recover from mistakes, and confirm results quickly. The goal is not to memorize trivia; it is to prove that you can work safely on a live RHEL system.

Use this checklist to identify weak spots in the core domains:

- **Local storage**: partitions, filesystems, mounts, swap, and persistence.
- **Users and permissions**: accounts, groups, sudo, ACLs, and file ownership.
- **Service management**: `systemd`, boot targets, and service troubleshooting.
- **Networking**: interface configuration, hostname, name resolution, and connectivity checks.
- **Security**: firewalld, SELinux, and basic access control.
- **Software and automation**: package management, archives, shell scripting, and scheduled tasks.
- **Troubleshooting**: logs, boot issues, and recovery workflows.

A strong final review is hands-on. Read less, do more, and verify every task with commands that prove the system state.

**What you should be able to do without hesitation**

You should be able to complete these tasks on a fresh RHEL system:

- Log in locally and remotely with the correct user context.
- Switch to `root` or use `sudo` appropriately.
- Inspect disks with `lsblk`, `blkid`, and `fdisk -l`.
- Create and mount filesystems persistently with `/etc/fstab`.
- Manage users, groups, passwords, and account aging.
- Set permissions with `chmod`, `chown`, `chgrp`, and ACLs.
- Start, stop, enable, disable, and inspect services with `systemctl`.
- Configure networking with `nmcli` and verify with `ip` and `ping`.
- Manage firewall rules with `firewall-cmd`.
- Check SELinux mode and troubleshoot denials with audit tools.
- Install and remove packages with `dnf`.
- Create and run simple shell scripts.
- Schedule recurring jobs with `cron` and one-time jobs with `at`.
- Read logs with `journalctl` and identify the cause of failures.

**Core command review**

The exam expects you to know the common tools and what each one proves.

```bash
# Identity and privilege
whoami
id
sudo -l
su -

# Storage inspection
lsblk
blkid
df -h
mount
findmnt

# Service management
systemctl status sshd
systemctl enable --now httpd
systemctl restart crond
systemctl list-units --type=service

# Networking
ip addr
ip route
nmcli device status
nmcli connection show

# Logs
journalctl -xe
journalctl -u sshd
journalctl --since "1 hour ago"

# SELinux and firewall
getenforce
sestatus
ausearch -m AVC,USER_AVC -ts recent
firewall-cmd --list-all
```

Use these commands as validation tools, not just as diagnostics. If you create a configuration, immediately prove that it works.

**Storage readiness checklist**

Storage tasks are common exam material because they combine planning, formatting, mounting, and persistence. You should be able to create a usable filesystem from an unused block device and make it survive a reboot.

- Identify the target device with `lsblk` and `blkid`.
- Create a partition with `fdisk`, `parted`, or `gdisk` when needed.
- Format the partition with the correct filesystem type, usually `XFS` on modern RHEL.
- Create a mount point with `mkdir -p`.
- Mount the filesystem and verify it with `findmnt` or `df -h`.
- Add a persistent entry to `/etc/fstab`.
- Test the `fstab` entry with `mount -a` before rebooting.
- Know how to create and activate swap if the task requires it.

Example workflow:

```bash
lsblk
sudo parted /dev/sdb --script mklabel gpt mkpart data xfs 1MiB 100%
sudo mkfs.xfs /dev/sdb1
sudo mkdir -p /data
sudo mount /dev/sdb1 /data
sudo blkid /dev/sdb1
```

Persistent mount example:

```fstab
UUID=1234abcd-5678-ef90-1234-abcdef567890 /data xfs defaults 0 0
```

Then validate:

```bash
sudo mount -a
findmnt /data
```

If `mount -a` fails, fix the entry before moving on. A broken `fstab` line can prevent a successful boot or cost you time during the exam.

**Permissions, ownership, and access control**

You should be comfortable with both basic UNIX permissions and more advanced access control lists (ACLs). Many exam tasks require you to grant access without changing the primary owner or group structure.

- Use `chmod` for standard mode changes.
- Use `chown` to change owner and group.
- Use `chgrp` when only the group changes.
- Use `umask` to control default permissions.
- Use `getfacl` and `setfacl` for ACLs.
- Use the setgid bit on shared directories when group inheritance matters.

Example:

```bash
sudo mkdir -p /shared/projects
sudo chgrp developers /shared/projects
sudo chmod 2775 /shared/projects
sudo setfacl -m u:alice:rwx /shared/projects
sudo getfacl /shared/projects
```

The `2` in `2775` sets the setgid bit, which helps new files inherit the directory group. This is useful for collaborative directories where group ownership must remain consistent.

**Users, groups, and sudo**

You should be able to create local users and groups, set passwords, and grant limited administrative access. Know how to verify the result instead of assuming the command worked.

- Create users with `useradd`.
- Set passwords with `passwd`.
- Create groups with `groupadd`.
- Add users to supplementary groups with `usermod -aG`.
- Inspect account details with `id`, `getent passwd`, and `getent group`.
- Configure sudo access through `/etc/sudoers` or `/etc/sudoers.d/` using `visudo`.

Example:

```bash
sudo groupadd ops
sudo useradd -m -G ops dev1
sudo passwd dev1
id dev1
```

Sudo rule example:

```sudoers
dev1 ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart httpd
```

Use `visudo` or `visudo -f /etc/sudoers.d/dev1` so syntax errors do not lock you out. Always verify with `sudo -l` from the target account.

**Service management and boot targets**

`systemd` is central to RHCSA. You should know how to inspect service state, enable services at boot, and troubleshoot failures. Be precise about the difference between starting a service now and enabling it for future boots.

- `systemctl start` activates a service immediately.
- `systemctl enable` creates boot-time persistence.
- `systemctl enable --now` does both in one step.
- `systemctl status` shows current state and recent log output.
- `systemctl disable` prevents automatic startup.
- `systemctl mask` blocks activation entirely.

Example:

```bash
sudo systemctl enable --now crond
sudo systemctl status crond
sudo systemctl restart crond
```

Boot target example:

```bash
sudo systemctl get-default
sudo systemctl set-default multi-user.target
```

If a service fails, inspect its logs with:

```bash
journalctl -u crond --since "today"
```

Use `journalctl` because many current RHEL systems rely on the journal rather than `/var/log/messages`. Do not assume a traditional text log file exists.

**Networking readiness**

You should be able to configure and verify basic network settings without a GUI. `nmcli` is the primary tool for persistent configuration on RHEL.

- Check interface state with `nmcli device status`.
- View active connections with `nmcli connection show`.
- Modify a connection profile with `nmcli connection modify`.
- Bring a connection up with `nmcli connection up`.
- Verify addressing with `ip addr`.
- Verify routing with `ip route`.
- Test connectivity with `ping`, `curl`, or `dig` when available.

Example static configuration:

```bash
sudo nmcli connection modify "System eth0" \
  ipv4.addresses 192.0.2.50/24 \
  ipv4.gateway 192.0.2.1 \
  ipv4.dns 192.0.2.10 \
  ipv4.method manual

sudo nmcli connection up "System eth0"
ip addr show
ip route
```

Hostname example:

```bash
sudo hostnamectl set-hostname server1.example.com
hostnamectl
```

If name resolution is part of the task, check `/etc/hosts` and DNS settings carefully. A correct IP address is not enough if the hostname or resolver configuration is wrong.

**Firewall and SELinux**

Security tasks often fail because the service is correct but access is blocked. You should verify both `firewalld` and SELinux when troubleshooting.

- Check firewall zones with `firewall-cmd --get-active-zones`.
- List allowed services and ports with `firewall-cmd --list-all`.
- Add permanent rules with `--permanent`, then reload.
- Check SELinux mode with `getenforce` and `sestatus`.
- Investigate denials with audit logs and `ausearch`.

Example firewall rule:

```bash
sudo firewall-cmd --permanent --add-service=http
sudo firewall-cmd --reload
sudo firewall-cmd --list-services
```

SELinux troubleshooting example:

```bash
getenforce
sestatus
sudo ausearch -m AVC,USER_AVC -ts recent
sudo journalctl -t setroubleshoot --since "1 hour ago"
```

> On a standard RHCSA system, AVC denials are more reliably checked with `ausearch -m AVC,USER_AVC -ts recent` and `journalctl` for the affected service. Do not depend on `/var/log/messages` or a `setroubleshoot` tag being present.

If you need to adjust SELinux context, use `semanage fcontext` and `restorecon` rather than disabling SELinux.

```bash
sudo semanage fcontext -a -t httpd_sys_content_t "/webdata(/.*)?"
sudo restorecon -Rv /webdata
ls -Zd /webdata
```

**Package management and software validation**

You should be able to install, remove, and query packages with `dnf`. Know how to confirm that a package is present and that the expected binary or service exists.

- Search packages with `dnf search` or `dnf provides`.
- Install with `dnf install`.
- Remove with `dnf remove`.
- Query installed packages with `rpm -q`.
- Inspect package contents with `rpm -ql`.

Example:

```bash
sudo dnf install -y httpd
rpm -q httpd
rpm -ql httpd | head
```

If a task asks you to provide a service, verify both the package and the service unit. Installing software does not automatically start it.

**Scheduling tasks**

You should know how to schedule recurring and one-time jobs. Use the correct tool for the timing requirement.

- Use `cron` for recurring jobs.
- Use `at` for one-time jobs.
- Edit user crontabs with `crontab -e`.
- Inspect scheduled jobs with `crontab -l` and `atq`.
- Ensure the `crond` service is enabled and running.

Example crontab entry:

```cron
# Run a backup script every day at 2:30 AM
30 2 * * * /usr/local/bin/backup.sh
```

Example one-time job:

```bash
echo "systemctl restart httpd" | at 23:00
atq
```

Remember that cron jobs run with a minimal environment. Use full paths in scripts and scheduled commands.

**Shell scripting and command-line habits**

The exam may require simple shell scripts or command sequences. Your scripts should be readable, executable, and safe.

- Start scripts with a shebang such as `#!/bin/bash`.
- Use `set -e` carefully if you want the script to stop on errors.
- Quote variables to avoid word splitting.
- Test scripts with `bash -n` and by running them directly.
- Make scripts executable with `chmod +x`.

Example:

```bash
#!/bin/bash
set -u

backup_dir="/backup"
mkdir -p "$backup_dir"
tar -czf "$backup_dir/home-$(date +%F).tgz" /home
```

Validate with:

```bash
bash -n backup.sh
chmod +x backup.sh
./backup.sh
```

**Troubleshooting workflow**

A reliable troubleshooting process saves time. When something does not work, isolate the layer that failed instead of changing multiple things at once.

1. Confirm the task requirement and expected end state.
2. Check the current system state with the right inspection command.
3. Make one change.
4. Re-test immediately.
5. Read logs or audit output if the change does not work.
6. Revert or correct the change before moving on.

Useful checks:

```bash
systemctl status <service>
journalctl -u <service> --since "10 minutes ago"
ip addr
ip route
mount | grep /data
ls -Z /webdata
```

For boot or login issues, think in layers:

- Is the service installed?
- Is it enabled?
- Is it running?
- Is the firewall open?
- Is SELinux allowing access?
- Is the file or directory context correct?
- Is the network path correct?

**Exam-day execution strategy**

The exam is time-limited, so your workflow matters as much as your knowledge. You should work in a predictable order and verify each task before leaving it.

1. Read all tasks first and identify dependencies.
2. Start with tasks that unlock others, such as storage, users, and networking.
3. Use `man`, `--help`, and tab completion only when needed; do not waste time browsing.
4. Verify each task with a command that proves success.
5. Keep notes of what you changed so you can recover quickly if needed.
6. Recheck the final state before submitting.

Practical habits:

- Use `sudo -i` when you need a root shell for many tasks.
- Prefer `systemctl enable --now` when a service must persist and run immediately.
- Prefer `mount -a` after editing `/etc/fstab`.
- Prefer `restorecon` after changing SELinux file contexts.
- Prefer `journalctl` over legacy log file assumptions.

**Quick self-test checklist**

Use this as a final pass before the exam:

- Can you create a user, set a password, and add the user to a group?
- Can you grant sudo access safely with `visudo`?
- Can you create an XFS filesystem and mount it persistently?
- Can you configure a static IP address with `nmcli`?
- Can you enable and verify a service with `systemctl`?
- Can you open a firewall port or service with `firewall-cmd`?
- Can you identify and fix an SELinux denial?
- Can you schedule a cron job and a one-time `at` job?
- Can you read `journalctl` output and identify the failing component?
- Can you explain the difference between a temporary change and a persistent one?

**Final mindset**

The best final readiness strategy is repetition with verification. Build a task, break it, fix it, and confirm it again. If you can perform the core RHCSA tasks from a clean system without hesitation, you are ready for the exam.
