## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**What scenario-based mock tasks test**

RHCSA scenarios combine multiple skills into one task, so you must identify the goal, choose the right tool, and verify the result. The exam usually rewards a clean workflow more than memorized commands.

- Read the task carefully and note the target state, not just the tool name.
- Verify after every change with commands such as `lsblk`, `mount`, `ss`, `systemctl`, `getenforce`, and `firewall-cmd`.
- Prefer persistent configuration when the task says the change must survive reboot.
- Use temporary changes only when you need to test before committing.

A good exam habit is to translate each scenario into three questions:
1. What is broken or missing?
2. What is the required final state?
3. How do I prove it works?

**General troubleshooting workflow**

Start with the simplest checks before changing configuration. Many RHCSA failures come from a wrong path, wrong device name, wrong label, or a service that is not enabled.

- Check the service state with `systemctl status <unit>`.
- Check logs with `journalctl -xe` or `journalctl -u <unit>`.
- Check file ownership and permissions with `ls -l`, `getfacl`, and `namei -l`.
- Check SELinux context with `ls -Z` and policy state with `getenforce`.
- Check network state with `ip addr`, `ip route`, `nmcli`, and `ss -tulpn`.

> If a service fails to start, confirm the executable path and permissions in the unit file before assuming SELinux or firewall problems.

**Storage scenarios**

Storage tasks often require you to create partitions, LVM components, swap, and mounts that persist across reboot. Always identify the correct device first; on exam systems the new disk may be `/dev/vdb`, `/dev/vdc`, or a partition such as `/dev/vdb1`.

- Use `lsblk`, `blkid`, and `fdisk -l` to inspect disks.
- Use `parted` or `fdisk` to create partitions.
- Use `pvcreate`, `vgcreate`, and `lvcreate` for LVM.
- Use `mkfs.xfs` or `mkfs.ext4` before mounting a new filesystem.
- Use `mkswap` and `swapon` for swap.
- Use UUIDs or labels in `/etc/fstab` when the task asks for stable mounting.

Example workflow for a new data volume:

```bash
lsblk
fdisk /dev/vdb
pvcreate /dev/vdb1
vgcreate vgdata /dev/vdb1
lvcreate -n lvshare -L 2G vgdata
mkfs.xfs /dev/vgdata/lvshare
mkdir -p /srv/share
mount /dev/vgdata/lvshare /srv/share
blkid /dev/vgdata/lvshare
```

Then add a persistent mount entry:

```fstab
UUID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx /srv/share xfs defaults 0 0
```

- Create the filesystem before adding the `fstab` entry.
- Mount only on an empty directory, because mounting hides existing files.
- For ext4 growth, extend the logical volume first, then grow the filesystem while it is mounted for normal online expansion.

Example of extending an XFS filesystem:

```bash
lvextend -L +1G /dev/vgdata/lvshare
xfs_growfs /srv/share
```

Example of extending ext4:

```bash
lvextend -L +1G /dev/vgdata/lvshare
resize2fs /dev/vgdata/lvshare
```

For swap, remember that the swap area does not use a normal filesystem:

```bash
mkswap /dev/vdb2
swapon /dev/vdb2
echo 'UUID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx none swap defaults 0 0' >> /etc/fstab
```

**Mounting by UUID or LABEL**

Using UUID or LABEL avoids breakage if device names change.

- Find identifiers with `blkid`.
- Use `UUID=` for unique device identity.
- Use `LABEL=` when the task explicitly asks for a named mount source.

Example:

```bash
blkid /dev/vdb1
echo 'LABEL=DATA /data xfs defaults 0 0' >> /etc/fstab
mount -a
```

**Networking scenarios**

Networking tasks often combine IP configuration, hostname changes, name resolution, and service access. On RHCSA, `nmcli` is the primary tool for persistent network configuration.

- Use `nmcli device status` to identify the active interface.
- Use `nmcli connection show` to inspect profiles.
- Use `nmcli connection modify` to set persistent values.
- Use `nmcli connection up <name>` to activate changes.
- Use `hostnamectl` to set the system hostname.
- Use `/etc/hosts` for local name resolution when DNS is not available.

Example static IPv4 configuration:

```bash
nmcli connection show
nmcli connection modify "System eth0" \
  ipv4.addresses 192.0.2.20/24 \
  ipv4.gateway 192.0.2.1 \
  ipv4.dns 192.0.2.10,192.0.2.11 \
  ipv4.method manual
nmcli connection up "System eth0"
```

- Use a comma-separated DNS list for clarity.
- Confirm the profile is attached to the intended interface.
- Verify with `ip addr show`, `ip route`, and `resolvectl status` if available.

Example hostname and hosts file setup:

```bash
hostnamectl set-hostname server1.example.com
cat >> /etc/hosts <<'EOF'
192.0.2.20 server1.example.com server1
192.0.2.30 server2.example.com server2
EOF
```

If the task asks for a profile to survive interface renaming, ensure the connection profile matches the device you intend to manage and verify the active profile after activation.

**Firewall scenarios**

Firewall tasks usually require opening a service or port permanently and verifying the active zone.

- Check the active zone with `firewall-cmd --get-active-zones`.
- Add services with `--permanent` when the change must survive reboot.
- Reload after permanent changes with `firewall-cmd --reload`.
- Use port rules when no predefined service exists.

Example:

```bash
firewall-cmd --get-active-zones
firewall-cmd --permanent --add-service=http
firewall-cmd --permanent --add-port=8080/tcp
firewall-cmd --reload
firewall-cmd --list-all
```

- Use `--permanent` for persistent changes.
- Use `--runtime-to-permanent` only when you intentionally tested a runtime rule and want to copy it into permanent config.
- Verify the service name exists before adding it.

**SELinux scenarios**

SELinux tasks often involve a service that works when disabled but fails when enforcing. The correct fix is usually a label, boolean, or policy adjustment, not disabling SELinux.

- Check mode with `getenforce`.
- Inspect labels with `ls -Z`.
- Restore default labels with `restorecon -Rv`.
- Use `semanage fcontext` for persistent label changes.
- Use `setsebool -P` for persistent boolean changes.

Example of fixing a web directory label:

```bash
mkdir -p /web/content
semanage fcontext -a -t httpd_sys_content_t '/web/content(/.*)?'
restorecon -Rv /web/content
```

Example of allowing a web server to read content in a nonstandard location:

```bash
getsebool -a | grep httpd
setsebool -P httpd_read_user_content on
```

- Choose the boolean that matches the actual access pattern.
- Do not use unrelated booleans as a generic fix.
- If the service writes content, confirm whether the target type should allow write access.

When troubleshooting, check audit logs:

```bash
ausearch -m AVC,USER_AVC -ts recent
```

**Service management scenarios**

Service tasks often ask you to start, enable, disable, mask, or troubleshoot a unit. You should know the difference between runtime state and boot-time state.

- `start` affects the current session.
- `enable` makes the service start at boot.
- `mask` prevents manual or automatic start.
- `disable` stops boot-time activation but does not block manual start.

Example:

```bash
systemctl status httpd
systemctl enable --now httpd
systemctl disable --now telnet.socket
systemctl mask cups
systemctl unmask cups
```

If a service fails, inspect the unit and its dependencies:

```bash
systemctl cat httpd
systemctl list-dependencies httpd
journalctl -u httpd --no-pager
```

- Check `ExecStart` for a wrong path.
- Check file permissions on the executable.
- Check whether the service needs a directory, socket, or port that is missing.
- Check whether SELinux or the firewall blocks the service after the binary itself is confirmed valid.

**User and group scenarios**

User tasks often combine account creation, group membership, password aging, and home directory setup. Know how to create users with the correct primary group and skeleton files.

- Use `useradd` for new accounts.
- Use `groupadd` for shared groups.
- Use `usermod -aG` to add supplementary groups.
- Use `passwd` to set passwords.
- Use `chage` to inspect or change password aging.
- Use `userdel -r` carefully when removing accounts and home directories.

Example:

```bash
groupadd developers
useradd -m -k /etc/skel -g developers -G wheel alice
passwd alice
chage -l alice
```

- `-m` creates the home directory.
- `-k` copies skeleton files from the specified directory.
- `-g` sets the primary group.
- `-G` adds supplementary groups.

Password aging example:

```bash
chage -M 90 -m 7 -W 14 alice
```

If you unlock an account, confirm that it is not also expired or disabled by aging rules.

```bash
passwd -S alice
usermod -U alice
```

**ACL and permission scenarios**

Access Control Lists (ACLs) let you grant access beyond standard owner/group/other permissions. They are common in shared-directory tasks.

- Use `getfacl` to inspect ACLs.
- Use `setfacl` to grant or remove entries.
- Use the default ACL on a directory to control inheritance for new files and directories.

Example:

```bash
mkdir -p /shared/project
setfacl -m u:alice:rwx /shared/project
setfacl -m g:developers:rwx /shared/project
setfacl -d -m g:developers:rwx /shared/project
getfacl /shared/project
```

- Default ACLs apply to newly created items inside the directory.
- Standard permissions and umask still matter.
- Setgid on a directory helps new files and subdirectories inherit the directory group.

Example of setgid collaboration directory:

```bash
chmod 2775 /shared/project
chgrp developers /shared/project
```

**File search, text processing, and archive scenarios**

RHCSA tasks may ask you to find files by name, size, owner, or content, then package or transform them. These tasks are often quick points if you know the right filters.

- Use `find` for file discovery.
- Use `grep` for content matching.
- Use `tar`, `gzip`, `bzip2`, and `xz` for archives and compression.
- Use `sed`, `awk`, `cut`, `sort`, `uniq`, and `wc` for text processing.

Examples:

```bash
find /var/log -type f -name '*.log'
find /home -type f -user alice -size +10M
grep -R "ERROR" /var/log/httpd
tar -czf /tmp/logs.tgz /var/log/httpd
```

Text processing example:

```bash
awk -F: '$3 >= 1000 {print $1, $3}' /etc/passwd | sort
```

- Use `find -exec` when you need to act on matched files.
- Use `grep -n` to include line numbers.
- Use `tar -tf` to inspect archive contents before extraction.

**Shell scripting and automation scenarios**

Basic shell scripting tasks usually require a small script that checks conditions, loops over files, or performs repeated actions. Keep scripts simple and test them before submission.

- Use a shebang such as `#!/bin/bash`.
- Quote variables to avoid word splitting.
- Use exit codes to signal success or failure.
- Use `chmod +x` to make scripts executable.

Example:

```bash
#!/bin/bash
for file in /var/log/*.log; do
  [ -f "$file" ] || continue
  gzip -k "$file"
done
```

Example of a simple conditional:

```bash
#!/bin/bash
if id alice &>/dev/null; then
  echo "alice exists"
else
  useradd alice
fi
```

- Test scripts with `bash -n script.sh` for syntax.
- Run them with `bash script.sh` before making them executable.
- Use `set -e` carefully; it can stop a script early if a command fails.

**Recovery scenarios**

Recovery tasks are high-value because they test your ability to repair a system that does not boot normally. The exact steps depend on whether you are in emergency mode, rescue mode, or an initramfs shell.

- In many boot-failure cases, the root filesystem is mounted read-only or under `/sysroot`.
- In rescue or initramfs environments, you often need to remount the root filesystem read-write before editing files.
- Use `chroot /sysroot` when the installed system must be repaired from the rescue environment.
- Use `systemctl default` only after the underlying issue is fixed and you want to continue booting.

Example workflow for fixing `/etc/fstab` from an initramfs or rescue shell:

```bash
mount -o remount,rw /sysroot
chroot /sysroot
vi /etc/fstab
exit
reboot
```

If the root password is forgotten, a common recovery path is to edit the GRUB boot entry, boot into a rescue target, and reset the password from the installed system.

Typical steps:

1. Reboot and interrupt GRUB.
2. Edit the kernel line and add `rd.break` or a rescue-related boot target.
3. Boot into the emergency shell.
4. Remount the installed system read-write if needed.
5. `chroot` into `/sysroot`.
6. Reset the password with `passwd root`.
7. Relabel if required, then reboot.

Example:

```bash
mount -o remount,rw /sysroot
chroot /sysroot
passwd root
touch /.autorelabel
exit
reboot
```

For GRUB configuration recovery, remember the output path depends on firmware mode.

- BIOS systems commonly use `/boot/grub2/grub.cfg`.
- UEFI systems commonly use `/boot/efi/EFI/redhat/grub.cfg`.

Example:

```bash
grub2-mkconfig -o /boot/grub2/grub.cfg
```

Example UEFI path:

```bash
grub2-mkconfig -o /boot/efi/EFI/redhat/grub.cfg
```

**Mixed-failure scenarios**

Some exam tasks combine multiple problems, such as a web service that fails because of permissions, SELinux labels, and firewall rules at the same time. Solve them in a logical order.

- Confirm the service binary and unit file first.
- Confirm the content path and ownership.
- Confirm SELinux labels and booleans.
- Confirm the firewall allows the required port or service.
- Confirm the service is enabled if the task requires boot persistence.

Example web server recovery flow:

```bash
systemctl status httpd
ls -ld /srv/www /srv/www/html
ls -Z /srv/www/html
semanage fcontext -a -t httpd_sys_content_t '/srv/www/html(/.*)?'
restorecon -Rv /srv/www/html
firewall-cmd --permanent --add-service=http
firewall-cmd --reload
systemctl enable --now httpd
```

- Fix the most basic failure first.
- Re-test after each change.
- Avoid changing multiple variables at once unless the task is clearly simple.

**Exam execution strategy**

A scenario-based exam rewards calm sequencing. You should work from identification to implementation to verification.

1. Read the task and underline the required end state.
2. Inspect the current system state with the least invasive commands.
3. Make the smallest correct change.
4. Verify the result immediately.
5. Persist the configuration if the task requires reboot survival.
6. Move to the next task only after the current one is confirmed.

Useful verification commands:

```bash
lsblk
blkid
ip addr
nmcli connection show --active
systemctl is-enabled httpd
firewall-cmd --list-all
getfacl /shared/project
getenforce
```

**Common exam pitfalls**

- Using the wrong device name for LVM or swap.
- Forgetting `--permanent` for firewall changes that must survive reboot.
- Editing a file but not reloading or restarting the affected service.
- Fixing SELinux by disabling it instead of correcting labels or booleans.
- Mounting a filesystem on a directory that already contains important data.
- Forgetting to enable a service after starting it.
- Assuming a network change is persistent when it only affects the runtime profile.
- Skipping verification after each change.

**Quick reference mindset**

Scenario tasks are not about speed alone; they are about reliable diagnosis. If you can identify the subsystem involved, choose the correct persistent fix, and prove the result, you are working like an RHCSA.

- Storage: identify, partition, format, mount, persist.
- Networking: profile, address, route, hostname, resolution.
- Security: permissions, ACLs, SELinux, firewall.
- Services: start, enable, mask, troubleshoot.
- Users: create, group, password, aging, lock/unlock.
- Recovery: boot mode, remount, chroot, repair, reboot.
