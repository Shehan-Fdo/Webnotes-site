## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**SELinux troubleshooting mindset**

Security-Enhanced Linux (**SELinux**) is a mandatory access control system that can block actions even when UNIX permissions look correct. In practice, you troubleshoot by identifying the denial, confirming the policy decision, and then choosing the least risky fix.

The key exam skill is not disabling SELinux. It is understanding *why* a service is denied and correcting the label, boolean, or policy context so the service works while SELinux stays enforcing.

- Check whether the issue is really SELinux before changing anything else.
- Use audit logs to find the exact denial and the source context.
- Prefer label fixes and booleans over broad policy changes.
- Use permissive mode only as a temporary diagnostic step.

**How SELinux denials happen**

SELinux decisions depend on the process context, file context, port context, and policy rules. A service can fail when one of those contexts does not match what the policy expects.

Common causes include:

- A file has the wrong **security context** after being copied or created in the wrong directory.
- A service tries to bind to a port that is not labeled for that service.
- A daemon needs a feature controlled by an SELinux **boolean**.
- A custom application writes to a path that policy does not allow.

A useful mental model is: UNIX permissions answer “can the user access the file,” while SELinux answers “is this process allowed to do this action on this object.”

**Confirming SELinux status**

Before troubleshooting denials, verify that SELinux is enabled and enforcing or permissive.

```bash
getenforce
sestatus
```

Typical outputs:

- `Enforcing` means SELinux blocks denied actions and logs them.
- `Permissive` means SELinux logs denials but does not block them.
- `Disabled` means SELinux is off; this is uncommon on RHCSA systems and not the normal troubleshooting target.

You can also inspect the current mode in `/etc/selinux/config`, but runtime state matters more during troubleshooting.

```bash
cat /etc/selinux/config
```

> If a service works in permissive mode but fails in enforcing mode, SELinux is likely involved.

**Finding denials in audit logs**

SELinux denials are usually recorded by the audit subsystem. The most important tools are `ausearch`, `audit2why`, and `audit2allow`.

Search recent AVC denials:

```bash
sudo ausearch -m AVC,USER_AVC -ts recent
```

Search by service or time window:

```bash
sudo ausearch -m AVC -ts today
sudo ausearch -m AVC -ts 10:00:00 -te 10:30:00
```

Read the denial details from the audit log:

```bash
sudo grep AVC /var/log/audit/audit.log
```

A denial record often includes:

- `scontext`: source context of the process
- `tcontext`: target context of the file, port, or object
- `tclass`: object class, such as `file`, `dir`, or `tcp_socket`
- `denied` permission: what action was blocked, such as `read`, `write`, or `name_bind`

Example AVC message:

```text
avc:  denied  { write } for  pid=1234 comm="httpd" name="index.html" dev="sda1" ino=123456
scontext=system_u:system_r:httpd_t:s0
tcontext=unconfined_u:object_r:default_t:s0
tclass=file permissive=0
```

This tells you that `httpd` tried to write a file labeled `default_t`, which is usually not allowed.

**Interpreting the denial**

The fastest way to understand a denial is to identify the process type and the target type.

- **Source context**: the domain of the running process, such as `httpd_t`, `sshd_t`, or `named_t`
- **Target context**: the label on the file, directory, or port
- **Class**: the object type, such as file or socket
- **Permission**: the action denied

Use `ls -Z` to inspect file labels:

```bash
ls -Z /var/www/html
ls -Z /srv/myapp
```

Use `ps -eZ` to inspect process labels:

```bash
ps -eZ | grep httpd
```

Use `ss -ltnZ` to inspect port labels:

```bash
ss -ltnZ
```

If the target label does not match the service’s expected type, SELinux may deny access even when ownership and mode bits are correct.

**Using audit2why for explanation**

`audit2why` explains why a denial occurred and often suggests the right fix.

```bash
sudo ausearch -m AVC -ts recent | audit2why
```

This is useful when you want to know whether the denial is caused by:

- a mislabeled file
- a missing boolean
- a policy limitation that requires a different design

`audit2why` is diagnostic. It does not change policy.

**Using audit2allow carefully**

`audit2allow` can generate a custom policy module from denial records.

```bash
sudo ausearch -m AVC -ts recent | audit2allow -M myapp_local
sudo semodule -i myapp_local.pp
```

Use this only when you understand the denial and no standard fix exists. For RHCSA-level administration, prefer standard SELinux management first:

- restore correct labels
- move content to the proper directory
- enable the required boolean
- adjust port labels if needed

> Custom policy modules can weaken security if you allow too much. Use them only when you can justify the exception.

**Fixing mislabeled files and directories**

The most common SELinux fix is restoring the correct file context.

View expected contexts with `semanage fcontext`:

```bash
sudo semanage fcontext -l | grep httpd
```

Apply a persistent label rule for a custom directory:

```bash
sudo semanage fcontext -a -t httpd_sys_content_t '/webdata(/.*)?'
sudo restorecon -Rv /webdata
```

If a web server needs write access to a directory, use the appropriate writable type rather than `httpd_sys_content_t`:

```bash
sudo semanage fcontext -a -t httpd_sys_rw_content_t '/webdata/uploads(/.*)?'
sudo restorecon -Rv /webdata/uploads
```

Important points:

- `semanage fcontext -a` adds a persistent mapping.
- `restorecon` applies the label based on policy mappings.
- `chcon` changes labels immediately but is usually temporary and can be overwritten by `restorecon`.

Example of a temporary label change:

```bash
sudo chcon -t httpd_sys_content_t /webdata/index.html
```

Use `chcon` for quick testing, then make the change persistent with `semanage fcontext` if it solves the problem.

**Working with SELinux booleans**

Booleans toggle specific policy behavior without writing custom policy. They are common in service troubleshooting.

List booleans related to a service:

```bash
getsebool -a | grep httpd
```

View a specific boolean:

```bash
getsebool httpd_enable_homedirs
```

Enable a boolean persistently:

```bash
sudo setsebool -P httpd_enable_homedirs on
```

Common use cases include:

- allowing web servers to read user home directories
- allowing FTP or NFS-related access depending on service policy
- enabling network connections for certain daemons

Use booleans when the denial matches a documented policy feature. This is safer than creating a custom module.

**Troubleshooting port denials**

A service may fail because it binds to a port not allowed by SELinux policy.

Check the port label:

```bash
semanage port -l | grep http
```

If your service must use a nonstandard port, add it to the correct SELinux port type.

Example for a web service on port 8080:

```bash
sudo semanage port -a -t http_port_t -p tcp 8080
```

If the port already exists under another type, modify it instead:

```bash
sudo semanage port -m -t http_port_t -p tcp 8080
```

Then restart the service and verify it listens:

```bash
sudo systemctl restart httpd
ss -ltnp | grep 8080
```

**Temporary permissive mode for diagnosis**

You can set SELinux permissive mode temporarily to confirm that SELinux is the cause of a failure.

```bash
sudo setenforce 0
```

Return to enforcing mode after testing:

```bash
sudo setenforce 1
```

This is useful when:

- the service fails and you need to isolate the cause quickly
- you want to compare behavior between enforcing and permissive modes

Do not leave the system permissive unless the task explicitly requires it.

**Common troubleshooting workflow**

1. Reproduce the failure.
2. Confirm SELinux mode with `getenforce`.
3. Inspect audit logs with `ausearch`.
4. Identify the source and target contexts.
5. Decide whether the fix is a label, boolean, port mapping, or policy exception.
6. Apply the smallest safe change.
7. Re-test the service.
8. Verify the denial no longer appears in the audit log.

**Practical example: Apache cannot read custom content**

Suppose Apache serves content from `/webdata`, but requests return `403 Forbidden`.

Check the file label:

```bash
ls -Zd /webdata /webdata/index.html
```

If the label is `default_t`, Apache will likely be denied. Fix it persistently:

```bash
sudo semanage fcontext -a -t httpd_sys_content_t '/webdata(/.*)?'
sudo restorecon -Rv /webdata
sudo systemctl restart httpd
```

If Apache must write uploads, label only the writable subdirectory accordingly:

```bash
sudo semanage fcontext -a -t httpd_sys_rw_content_t '/webdata/uploads(/.*)?'
sudo restorecon -Rv /webdata/uploads
```

This keeps the rest of the site read-only from SELinux’s perspective.

**Practical example: Apache on a nonstandard port**

If `httpd` is configured to listen on `8080`, but SELinux blocks the bind, add the port to the correct type.

```bash
sudo semanage port -a -t http_port_t -p tcp 8080
sudo systemctl restart httpd
```

Verify the service and port:

```bash
systemctl status httpd
ss -ltnp | grep 8080
```

**Practical example: Boolean-based fix**

If Apache must serve files from user home directories, a boolean may be required.

```bash
getsebool httpd_enable_homedirs
sudo setsebool -P httpd_enable_homedirs on
```

Then verify access again. If the denial disappears, the boolean was the correct fix.

**Useful commands summary**

| Task | Command |
|---|---|
| Check mode | `getenforce` |
| View SELinux status | `sestatus` |
| Search recent denials | `ausearch -m AVC,USER_AVC -ts recent` |
| Explain denials | `audit2why` |
| Generate policy module | `audit2allow -M name` |
| Inspect file labels | `ls -Z` |
| Inspect process labels | `ps -eZ` |
| Inspect port labels | `ss -ltnZ` |
| Restore labels | `restorecon -Rv path` |
| Add persistent file context | `semanage fcontext -a ...` |
| Add persistent port label | `semanage port -a ...` |
| Change boolean persistently | `setsebool -P ...` |

**Exam-focused best practices**

- Prefer `restorecon` and `semanage fcontext` for file labeling problems.
- Prefer `setsebool -P` for documented policy toggles.
- Prefer `semanage port` for nonstandard service ports.
- Use `chcon` only as a temporary test.
- Use `audit2allow` only when standard fixes do not solve the issue.
- Verify the fix by repeating the original action and checking that the denial no longer appears.

**Key takeaway**

SELinux denials are usually solved by matching the service to the correct label, port type, or boolean. The safest workflow is to read the audit message, understand the mismatch, and apply the smallest persistent fix that preserves enforcement.
