## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**File permissions and ownership overview**

Linux access control starts with three layers: the file owner, the group owner, and the permission bits applied to each class. On Red Hat Enterprise Linux, you use `chmod` to change permissions, `chown` to change ownership, and `chgrp` to change the group owner.

These commands are central to system administration because they control who can read, modify, or execute files and directories. In practice, you use them to secure configuration files, share project directories, and prevent accidental changes.

- **Owner** is the user account that controls the file.
- **Group** is the group identity used for shared access.
- **Others** means every user who is neither the owner nor in the group.
- **Permissions** are usually represented as `r`, `w`, and `x`.

A quick view of permissions looks like this:

```bash
$ ls -l /etc/hosts
-rw-r--r--. 1 root root 220 Apr 26 10:00 /etc/hosts
```

In that output:
- `-rw-r--r--` is the permission string.
- The first `root` is the owner.
- The second `root` is the group.

**Understanding permission bits**

Each file or directory has three permission classes:
- **User** permissions apply to the owner.
- **Group** permissions apply to the file’s group.
- **Other** permissions apply to everyone else.

Each class can have:
- **Read (`r`)**: view file contents or list directory contents.
- **Write (`w`)**: modify file contents or create/delete entries in a directory.
- **Execute (`x`)**: run a file as a program or enter a directory.

For directories, the meaning changes slightly:
- `r` lets you list names in the directory.
- `w` lets you create, remove, or rename entries.
- `x` lets you access files and traverse the directory.

> A directory with `r` but without `x` can show names, but you cannot access the files inside it.

**Using `chmod` to change permissions**

`chmod` changes the mode bits on files and directories. You can use symbolic mode or numeric mode.

Symbolic mode is readable and useful when changing one class at a time:
```bash
chmod u+x script.sh
chmod g-w report.txt
chmod o-rwx secret.txt
chmod ug+rw shared.txt
```

Numeric mode is compact and common in exams:
- `r = 4`
- `w = 2`
- `x = 1`

Add the values for each class:
- `7 = rwx`
- `6 = rw-`
- `5 = r-x`
- `4 = r--`

Examples:
```bash
chmod 644 file.txt
chmod 755 script.sh
chmod 700 private.key
chmod 2775 shared-project/
```

Common meanings:
- `644` for regular files that should be readable by others but writable only by the owner.
- `755` for executable scripts and programs.
- `700` for private files or directories.
- `2775` for collaborative directories with setgid behavior.

**Practical `chmod` examples**

Use `chmod` when you need to:
- Make a script executable.
- Restrict access to sensitive data.
- Allow a team to collaborate in a shared directory.
- Remove accidental write access from non-owners.

Example workflow for a script:
```bash
chmod 755 backup.sh
./backup.sh
```

If a script is not executable, you can still run it with an interpreter:
```bash
bash backup.sh
```

But for normal execution, the execute bit should be set.

**Using `chown` to change ownership**

`chown` changes the user owner, and optionally the group owner. This is useful when files are created by the wrong account or when you need to hand off responsibility for a directory tree.

Basic syntax:
```bash
chown user file
chown user:group file
chown :group file
```

Examples:
```bash
chown alice report.txt
chown alice:developers report.txt
chown :developers report.txt
```

You can apply changes recursively with `-R`:
```bash
chown -R alice:developers /srv/project
```

Use recursive ownership changes carefully because they affect every file and subdirectory under the target path.

**Using `chgrp` to change group ownership**

`chgrp` changes only the group owner. It is often used when multiple users need access to the same files through a shared group.

Examples:
```bash
chgrp developers report.txt
chgrp -R developers /srv/project
```

`chown :group file` and `chgrp group file` achieve the same group change. `chown` is more flexible because it can change both owner and group in one command.

**When to use each command**

- Use **`chmod`** when permissions are wrong but ownership is correct.
- Use **`chown`** when the user owner is wrong or both owner and group need adjustment.
- Use **`chgrp`** when only the group owner needs to change.

A common exam scenario is a directory that must be writable by a team but owned by a specific user. In that case, you often combine ownership changes with permission changes.

**Shared directory example**

Suppose you want `/srv/shared` to support collaboration:
- Owner: `alice`
- Group: `developers`
- Permissions: owner and group can write, others can read only

You might configure it like this:
```bash
mkdir -p /srv/shared
chown alice:developers /srv/shared
chmod 2775 /srv/shared
```

The leading `2` sets the **setgid** bit on the directory. That causes new files and subdirectories created inside it to inherit the directory’s group, which is useful for team collaboration.

Verify the result:
```bash
ls -ld /srv/shared
```

Expected pattern:
```bash
drwxrwsr-x. 2 alice developers 4096 Apr 26 10:00 /srv/shared
```

Notice the `s` in the group execute position. That indicates setgid is active on the directory.

**Recursive changes and verification**

When you change ownership or permissions recursively, verify the result immediately.

Useful commands:
```bash
ls -l file.txt
ls -ld /srv/project
stat file.txt
namei -l /srv/project/subdir/file.txt
```

`stat` gives detailed metadata, including numeric mode and ownership:
```bash
stat report.txt
```

`namei -l` is helpful when access fails because of a parent directory permission problem. It shows permissions on every path component.

**Common permission patterns**

| Mode | Meaning | Typical use |
|---|---|---|
| `600` | owner read/write only | private keys, secrets |
| `644` | owner read/write, others read | text files, web content |
| `700` | owner full access only | private directories |
| `750` | owner full, group read/execute | admin tools, shared scripts |
| `755` | owner full, others read/execute | executable programs |
| `775` | owner and group full, others read/execute | team directories |

Use the least permissive mode that still supports the task.

**Practical troubleshooting**

If a user cannot access a file, check these in order:
1. The file owner and group.
2. The permission bits on the file.
3. The permission bits on every parent directory.
4. Whether the user belongs to the expected group.
5. Whether ACLs (Access Control Lists) or SELinux are also affecting access.

Example checks:
```bash
id alice
ls -ld /srv/shared
ls -l /srv/shared/report.txt
getfacl /srv/shared/report.txt
```

If the file looks correct but access still fails, SELinux may be involved. Permission bits are necessary, but not always sufficient.

> `chmod` does not grant access if SELinux policy denies it.

**Special permission notes**

Some permission bits have behavior beyond the basic `rwx` model:
- **setuid** on executables runs the program with the file owner’s effective user ID.
- **setgid** on executables runs with the file group’s effective group ID.
- **setgid** on directories makes new files inherit the directory group.
- **sticky bit** on directories prevents users from deleting files they do not own, even if the directory is writable.

Examples:
```bash
chmod 4755 /usr/local/bin/tool
chmod 2755 /srv/shared
chmod 1777 /tmp
```

For RHCSA, you should recognize these modes and understand their effect, even if the topic focus is `chmod`, `chown`, and `chgrp`.

**Exam-oriented command patterns**

You should be comfortable with these forms:
```bash
chmod u=rw,g=r,o= file.txt
chmod -R g+w /srv/project
chown root:root /etc/important.conf
chown :wheel admin-script.sh
chgrp developers /srv/teamfile
```

A few practical reminders:
- Use `-R` only when you intend to affect the full tree.
- Use `:` in `chown user:group` to change both owner and group.
- Use `:` alone in `chown :group` to change only the group.
- Use `ls -l` and `stat` to confirm changes.

**Quick reference**

- `chmod` changes permissions.
- `chown` changes user owner and optionally group owner.
- `chgrp` changes group owner only.
- `r` means read, `w` means write, `x` means execute.
- Directories use `x` for traversal and `w` for entry modification.
- Numeric modes are often faster on the exam.
- Recursive changes are powerful and risky.

**Mini lab example**

A simple practice task:
- Create a directory for a team.
- Make `alice` the owner.
- Make `developers` the group.
- Allow the owner and group to write.
- Ensure new files inherit the group.

Commands:
```bash
mkdir -p /srv/labshare
chown alice:developers /srv/labshare
chmod 2775 /srv/labshare
touch /srv/labshare/testfile
ls -l /srv/labshare/testfile
```

If the directory is configured correctly, `testfile` should inherit the `developers` group.

**Key takeaways**

File permissions and ownership are foundational to Linux administration. `chmod` controls access, `chown` assigns responsibility, and `chgrp` supports shared access. On the RHCSA exam, you should be able to read permission strings, apply numeric and symbolic modes, and verify the result with standard inspection commands.
