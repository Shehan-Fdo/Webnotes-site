## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**ACL fundamentals**

An <abbr title="Access Control List">ACL</abbr> extends traditional Linux owner/group/other permissions with per-user and per-group entries. Use ACLs when standard mode bits are too coarse, such as when one user needs read access to a shared directory without changing the owning group.

The base permission model still applies first:

- **Owner** permissions apply to the file owner.
- **Group** permissions apply to the owning group, but ACLs can refine this.
- **Other** permissions remain the fallback for everyone else.

ACLs are most useful on shared directories, project trees, and lab systems where multiple users need different access levels without changing ownership.

> ACL support depends on the filesystem and mount options. Before troubleshooting permissions, verify that the target filesystem supports ACLs and is mounted with ACL capability when needed.

**Prerequisites and filesystem verification**

On modern Red Hat systems, ACL support is commonly available on local filesystems such as XFS and ext4, but you should still verify the target mount. If ACLs are not behaving as expected, check both the filesystem type and the mount options.

Useful checks:

```bash
findmnt -no TARGET,FSTYPE,OPTIONS /shared
mount | grep ' /shared '
```

If you need to confirm ACL capability on a mounted filesystem, inspect the mount options and test with `getfacl`/`setfacl`. If the filesystem or mount does not support ACLs, `setfacl` may fail or the ACL may not persist.

Typical verification workflow:

```bash
touch /shared/testfile
setfacl -m u:alice:r-- /shared/testfile
getfacl /shared/testfile
```

If the ACL appears and survives a remount or reboot, the filesystem is handling ACLs correctly.

**Reading ACLs with `getfacl`**

`getfacl` shows the full permission model, including named users, named groups, the mask, and default ACLs on directories. This is the command you use to verify what is actually effective.

Example output:

```bash
getfacl /shared/project
```

```text
# file: shared/project
# owner: root
# group: devs
user::rwx
user:alice:rwx
group::r-x
group:qa:r--
mask::rwx
other::---
default:user::rwx
default:user:alice:rwx
default:group::r-x
default:mask::rwx
default:other::---
```

Key points:

- **Named user** entries grant permissions to specific users.
- **Named group** entries grant permissions to specific groups.
- **mask** limits the effective permissions of named users, named groups, and the owning group.
- **default** entries apply only to directories and are inherited by new files and subdirectories.

`getfacl` often prints `effective:` comments when the mask reduces a permission entry. For example:

```text
user:alice:rwx    #effective:r-x
group:qa:r--      #effective:r--
```

This means the ACL entry requests more access than the mask allows. In practice, the mask is the ceiling for those entries. If a user seems to have less access than expected, check the mask first.

**Setting ACLs with `setfacl`**

Use `setfacl` to add, modify, or remove ACL entries. The most common forms are:

```bash
setfacl -m u:alice:rwx /shared/project
setfacl -m g:qa:r-- /shared/project/report.txt
setfacl -x u:alice /shared/project/report.txt
```

Common options:

- `-m` modifies or adds ACL entries.
- `-x` removes specific ACL entries.
- `-b` removes all access ACL entries.
- `-k` removes default ACLs from a directory.
- `-R` applies changes recursively.
- `-d` sets default ACL entries on directories.

A practical rule:

- Use **access ACLs** on existing files and directories to change current access.
- Use **default ACLs** on directories to control inheritance for new content.

**Mask behavior and troubleshooting**

The ACL **mask** is automatically recalculated from the union of named user and named group permissions when you modify ACLs, unless you explicitly set it. This is a common exam troubleshooting point because the mask can silently reduce access.

Example:

```bash
setfacl -m u:alice:rwx,g:qa:r-- /shared/project
getfacl /shared/project
```

If the resulting mask becomes `rwx`, both entries can be effective at their full requested permissions. If the mask is narrower, `getfacl` will show `effective:` values that reflect the reduction.

Important interpretation rules:

- The mask affects the **owning group** entry and all **named user/group** entries.
- The owner entry `user::` is not limited by the mask.
- If a user cannot write even though the ACL entry says `rw-`, check whether the mask is `r--` or similar.

You can also set the mask explicitly:

```bash
setfacl -m m::r-x /shared/project
```

Use explicit mask control carefully, because it can override the automatic recalculation you may expect.

**Default ACL inheritance on directories**

Default ACLs are the mechanism that makes permissions propagate to newly created files and subdirectories. They are stored on directories only.

A default ACL on a directory affects:

- New files created inside the directory.
- New subdirectories created inside the directory.
- The inherited default ACL on those new subdirectories, if present.

A default ACL does not retroactively change existing files. For existing content, you must apply access ACLs directly, often with recursion.

Example of setting a default ACL:

```bash
setfacl -m d:u:alice:rwx,d:g:qa:r-x,d:o:--- /shared/project
getfacl /shared/project
```

This means new items created under `/shared/project` inherit those permissions, subject to the creator’s umask and the filesystem’s ACL rules.

**Recursive behavior with `-R`**

`setfacl -R` is important, but its behavior depends on whether you are changing access ACLs or default ACLs.

- For **access ACLs**, `-R` applies the ACL change to existing files and directories in the tree.
- For **default ACLs**, only directories can store default ACLs, so recursion applies the default ACL to directories in the tree, not to regular files.

This distinction matters in exam labs:

- Use `-R` when you need to fix permissions on an existing tree.
- Use default ACLs on the top-level directory when you want future content to inherit permissions.
- Combine both when you need to correct current content and ensure future inheritance.

Example:

```bash
setfacl -R -m u:alice:rwx /shared/project
setfacl -R -m d:u:alice:rwx /shared/project
```

The first command changes current access throughout the tree. The second command ensures directories in the tree carry the default ACL for future inheritance.

> Be careful: recursive ACL changes can affect large trees quickly. In an exam, confirm the target path before running `-R`.

**Step-by-step lab: create, inherit, and verify**

1. Create a shared directory and confirm its current permissions.
2. Add a default ACL so a second user inherits access.
3. Create a new file as that second user.
4. Verify the inherited ACL and effective permissions.

Example lab:

```bash
mkdir -p /srv/share/lab
chown root:devs /srv/share/lab
chmod 2770 /srv/share/lab
setfacl -m d:u:alice:rwx,d:g:devs:rwx,d:o:--- /srv/share/lab
getfacl /srv/share/lab
```

Now create content as another user:

```bash
su - alice
touch /srv/share/lab/alice.txt
echo "test" > /srv/share/lab/alice.txt
getfacl /srv/share/lab/alice.txt
```

What to verify:

- The new file inherits the ACL from the parent directory.
- The file’s access ACL reflects the inherited entries.
- The mask may reduce the effective permissions shown by `getfacl`.

If the file is created with less access than expected, inspect:

- The parent directory’s default ACL.
- The file’s mask.
- The creator’s `umask`.

**Removing ACLs and restoring a clean state**

Exam tasks often require you to clean up after testing. Use `setfacl` to remove ACLs and return to standard mode bits.

Remove all access ACL entries from a file or directory:

```bash
setfacl -b /srv/share/lab/alice.txt
```

Remove all default ACL entries from a directory:

```bash
setfacl -k /srv/share/lab
```

Remove ACLs recursively from a tree:

```bash
setfacl -R -b /srv/share/lab
find /srv/share/lab -type d -exec setfacl -k {} +
```

A practical cleanup sequence for a directory tree is:

```bash
setfacl -R -b /srv/share/lab
find /srv/share/lab -type d -exec setfacl -k {} +
```

This clears access ACLs from files and directories, then removes default ACLs from directories so the tree returns to a clean baseline.

**Practical troubleshooting workflow**

When ACLs do not behave as expected, work through the problem in this order:

1. Confirm the path and filesystem.
2. Check the current ACL with `getfacl`.
3. Look for a restrictive `mask`.
4. Verify whether you set an access ACL or a default ACL.
5. Confirm the file was created after the default ACL was applied.
6. Check the creator’s `umask` if inheritance seems narrower than expected.

Useful commands:

```bash
namei -l /srv/share/lab/alice.txt
getfacl /srv/share/lab
getfacl /srv/share/lab/alice.txt
ls -ld /srv/share/lab /srv/share/lab/alice.txt
```

**Common exam patterns**

- Grant one user access to a shared directory without changing ownership.
- Allow a group to collaborate on a directory while preserving stricter `other` permissions.
- Set default ACLs so new files inherit the correct access automatically.
- Apply recursive ACLs to an existing project tree.
- Remove ACLs cleanly after testing or when reverting to standard permissions.

**Quick reference**

| Task | Command |
|---|---|
| View ACLs | `getfacl path` |
| Add or modify ACL | `setfacl -m u:alice:rwx path` |
| Remove one ACL entry | `setfacl -x u:alice path` |
| Remove all access ACLs | `setfacl -b path` |
| Remove default ACLs on a directory | `setfacl -k dir` |
| Apply recursively | `setfacl -R -m ... path` |
| Set default ACL | `setfacl -m d:u:alice:rwx dir` |

**Exam-ready reminders**

- **Default ACLs** are for inheritance on directories.
- **Access ACLs** are for current permissions on files and directories.
- The **mask** can make permissions look correct in the ACL entry but still limit actual access.
- `getfacl` is your verification tool; do not assume the ACL is effective until you inspect it.
- Use recursive cleanup when you need to restore a lab environment quickly.
