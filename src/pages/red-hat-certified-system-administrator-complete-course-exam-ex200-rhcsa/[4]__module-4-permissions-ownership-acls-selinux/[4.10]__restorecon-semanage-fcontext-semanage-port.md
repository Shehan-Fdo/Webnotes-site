## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**SELinux context management overview**

SELinux (Security-Enhanced Linux) uses labels to decide whether a process can access a file, directory, or network port. On RHCSA, you often fix access problems by correcting labels rather than disabling SELinux. The three tools in this topic work together:

- `restorecon` reapplies the default SELinux context from policy.
- `semanage fcontext` defines persistent file-context rules for paths.
- `semanage port` defines persistent SELinux labels for network ports.

The key idea is persistence. `chcon` changes a label immediately, but the change is usually temporary and can be lost when contexts are restored. `semanage` changes policy mappings so the correct label survives relabeling and reboot.

**SELinux context basics**

A file or directory label usually has four fields:

- **user**: SELinux user identity
- **role**: SELinux role
- **type**: the most important field for access control
- **level**: MLS/MCS level, less common in RHCSA troubleshooting

You will most often work with the **type** field. For example, Apache content typically needs `httpd_sys_content_t`, while writable web content may need `httpd_sys_rw_content_t`.

Use `ls -Z` to inspect labels:

```bash
ls -Z /var/www/html
ls -Zd /var/www /srv/web
```

Use `ps -eZ` to inspect process labels:

```bash
ps -eZ | grep httpd
```

**restorecon: restore default labels**

`restorecon` resets a file or directory to the SELinux context defined by policy for that path. It does not invent a new rule; it applies the current default mapping.

Use it when:

- A file has the wrong label after a copy, move, or manual `chcon`.
- You want to revert a path to the policy default.
- You have already created a persistent mapping with `semanage fcontext` and want to apply it now.

Common options:

- `-v` shows what changed.
- `-R` applies recursively.
- `-n` performs a dry run.
- `-F` forces a reset even if the current label appears correct.

Examples:

```bash
restorecon -v /var/www/html/index.html
restorecon -Rv /var/www/html
restorecon -nRv /srv/web
```

A recursive restore is common after defining a new file-context rule. If you create a new web root under `/srv/web`, you typically define the mapping first, then apply it with `restorecon`.

> `restorecon` only works as expected if policy already knows the correct label for that path. If the path is nonstandard, you usually need `semanage fcontext` first.

**semanage fcontext: persistent file-context mappings**

`semanage fcontext` manages the file-context database used by SELinux policy. This is the correct tool when a nonstandard path should permanently receive a specific type.

Typical use cases:

- Hosting web content outside `/var/www`
- Storing shared content in a custom directory
- Restoring correct labels after moving application data
- Preparing a directory tree for a service that expects a specific SELinux type

The most common pattern is:

1. Add a file-context rule with `semanage fcontext -a`.
2. Apply it with `restorecon -Rv`.

Example for a custom Apache document root:

```bash
mkdir -p /srv/web/site1
echo "Hello RHCSA" > /srv/web/site1/index.html

semanage fcontext -a -t httpd_sys_content_t '/srv/web/site1(/.*)?'
restorecon -Rv /srv/web/site1
ls -Zd /srv/web/site1 /srv/web/site1/index.html
```

The regular expression `(/.*)?` means the rule applies to the directory and everything under it. This is important because SELinux file contexts are usually assigned by path pattern, not by a single file only.

If the path already has a rule, inspect it before changing anything:

```bash
semanage fcontext -l | grep '/srv/web/site1'
```

If you need to replace an existing mapping, use `-m` instead of `-a`:

```bash
semanage fcontext -m -t httpd_sys_content_t '/srv/web/site1(/.*)?'
```

If you need to remove a custom rule:

```bash
semanage fcontext -d '/srv/web/site1(/.*)?'
```

**Important file-context patterns**

- Use a path regex that matches the directory and its contents when needed.
- Quote the pattern so the shell does not expand special characters.
- Match the narrowest path that solves the problem.
- Use the correct type for the service domain; arbitrary type names are not valid unless they exist in policy.

Common Apache-related types:

- `httpd_sys_content_t` for read-only web content
- `httpd_sys_rw_content_t` for content Apache must write
- `httpd_log_t` for log files
- `httpd_sys_script_exec_t` for CGI or script execution in some setups

Example for writable web content:

```bash
mkdir -p /srv/web/uploads
semanage fcontext -a -t httpd_sys_rw_content_t '/srv/web/uploads(/.*)?'
restorecon -Rv /srv/web/uploads
```

**chcon versus semanage fcontext**

`chcon` changes the label directly on the filesystem. It is useful for quick testing, but it is not persistent across relabeling.

```bash
chcon -t httpd_sys_content_t /srv/web/site1/index.html
ls -Z /srv/web/site1/index.html
```

Use `semanage fcontext` when you need the label to survive:

- `restorecon`
- filesystem relabels
- package updates that reset contexts
- reboot and policy refresh

A practical rule: use `chcon` to test, then use `semanage fcontext` for the final fix.

**semanage port: persistent SELinux port labeling**

`semanage port` manages which TCP or UDP ports are allowed for a given SELinux port type. This matters when a service listens on a nondefault port.

Use it when:

- A service is configured to listen on a custom port.
- SELinux blocks the service even though the firewall allows the port.
- You need a persistent policy change for a service port.

First inspect existing labels:

```bash
semanage port -l | grep http_port_t
semanage port -l | grep ssh_port_t
```

For Apache, the port type is usually `http_port_t`. Before adding a new port, check whether it is already included in the existing mapping. Many common ports, including 8080 on some systems, may already be labeled `http_port_t`. In that case, `-a` would fail because the mapping already exists.

Example workflow for a custom Apache port:

1. Check the current SELinux port mapping.
2. Configure Apache to listen on the port.
3. Add the port to the correct SELinux type only if it is not already present.
4. Reload or restart the service.

```bash
semanage port -l | grep http_port_t
```

If the port is not already mapped to `http_port_t`, add it:

```bash
semanage port -a -t http_port_t -p tcp 8081
```

If the port is already part of another range for the same type, you may need to modify the existing range rather than add a duplicate. Be careful with ranges: `semanage port -m` changes an existing mapping entry, so you must specify the correct range or entry that policy already uses.

Example for a service that uses a custom port already defined in policy:

```bash
semanage port -l | grep myservice
```

If the service type exists in policy and the port is not assigned, add the mapping with the real SELinux type for that service:

```bash
semanage port -a -t <existing_service_port_type> -p tcp <port>
```

Replace `<existing_service_port_type>` with a valid policy type, not an invented name. The type must already exist in the loaded SELinux policy.

**Practical Apache example**

Suppose Apache should listen on TCP 8081 and serve content from `/srv/web/site1`.

```bash
dnf install -y httpd policycoreutils-python-utils

mkdir -p /srv/web/site1
echo "Custom site" > /srv/web/site1/index.html

semanage fcontext -a -t httpd_sys_content_t '/srv/web/site1(/.*)?'
restorecon -Rv /srv/web/site1

semanage port -a -t http_port_t -p tcp 8081

grep -n '^Listen' /etc/httpd/conf/httpd.conf
echo 'Listen 8081' > /etc/httpd/conf.d/listen8081.conf

systemctl enable --now httpd
systemctl restart httpd
```

If Apache still fails, check:

```bash
ss -tlnp | grep httpd
ausearch -m AVC -ts recent
journalctl -xe
```

An AVC (Access Vector Cache) denial often indicates a label problem, not a service problem. Read the denial carefully before changing policy.

**Troubleshooting workflow**

When a service cannot access a file or port, use a structured approach:

1. Confirm the service is running.
2. Check the SELinux mode with `getenforce`.
3. Inspect labels with `ls -Z` or `semanage port -l`.
4. Compare the current label to the expected policy type.
5. Fix the persistent mapping with `semanage fcontext` or `semanage port`.
6. Apply file changes with `restorecon`.
7. Retest and review AVC logs if needed.

Useful commands:

```bash
getenforce
ls -Z /path/to/content
semanage fcontext -l | grep '/path/to/content'
semanage port -l | grep 8081
ausearch -m AVC -ts recent
```

**Common exam patterns**

- A web directory under `/srv` is inaccessible until you label it with `httpd_sys_content_t`.
- A writable upload directory needs `httpd_sys_rw_content_t`, not the read-only content type.
- A service on a nondefault port may need a `semanage port` change even when the firewall is open.
- `restorecon` is the final step after defining a persistent file-context rule.
- `chcon` alone is not enough for a durable fix.

**Quick reference**

| Task | Tool | Typical action |
|---|---|---|
| Reset a mislabeled file to policy default | `restorecon` | `restorecon -Rv /path` |
| Make a custom directory keep a label | `semanage fcontext` | `semanage fcontext -a -t TYPE '/path(/.*)?'` |
| Allow a service on a custom port | `semanage port` | `semanage port -a -t http_port_t -p tcp 8081` |
| Inspect current labels | `ls -Z`, `semanage fcontext -l`, `semanage port -l` | Verify before changing |

**Key takeaways**

- `restorecon` applies the policy default label to a path.
- `semanage fcontext` creates persistent file-context rules for nonstandard paths.
- `semanage port` creates persistent SELinux port mappings for services on custom ports.
- Always verify the current mapping before adding a new one.
- Use the real policy type for the service; do not invent type names.
- For port ranges, modify the existing range carefully instead of assuming a single-port entry.
