## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

Linux permissions are not only about the mode bits you see in `ls -l`. New files and directories are shaped by a combination of the process `umask`, the parent directory’s special bits, and any Access Control Lists (ACLs) applied to the parent. For RHCSA, you need to know how these controls interact so you can build shared directories that behave predictably.

The key idea is that inheritance is not automatic in the same way for every mechanism:

- `umask` affects the permissions a process requests for newly created objects.
- `setgid` on a directory influences group ownership of new entries and can help keep a team directory consistent.
- Default ACLs can define inherited permissions for new files and subdirectories.
- The sticky bit controls deletion behavior in shared writable directories, not inheritance.

**Permission model refresher**

Traditional Unix permissions use three classes:

- **user**: the file owner
- **group**: the owning group
- **other**: everyone else

For directories, the execute bit means “can traverse,” and the write bit means “can create, delete, or rename entries” if you also have execute permission. This is why a directory can be writable even when files inside it are not.

A common exam trap is assuming that group write on a directory is enough for collaboration. It is not, unless the directory’s group ownership and effective permissions are set correctly.

**`umask` and default creation modes**

`umask` is a per-shell or per-process mask that removes permissions from the default mode requested by applications. It does not add permissions; it only subtracts them.

Typical defaults:

- regular files are usually requested as `666`
- directories are usually requested as `777`

The final mode is computed by subtracting the `umask` from the requested mode.

Example:

```bash
umask 0022
# file: 666 -> 644
# dir : 777 -> 755
```

Check the current value with:

```bash
umask
umask -S
```

Set it temporarily in the current shell:

```bash
umask 0002
```

Common meanings:

- `0022`: private by default; group and other lose write
- `0002`: collaborative; group keeps write, other loses write
- `0077`: highly restrictive; only the owner gets access

> `umask` affects what the application asks for, but ACLs and directory inheritance can still change the final result.

**`setgid` on directories**

The set-group-ID bit on a directory causes new files and subdirectories created inside it to inherit the directory’s group ownership. This is very useful for shared project directories because it keeps group ownership consistent without requiring users to manually change groups.

Inspect it with:

```bash
ls -ld /shared/team
```

You will see an `s` in the group execute position when setgid is active:

```bash
drwxrwsr-x 2 root devteam 4096 Apr 26 10:00 /shared/team
```

Set it with:

```bash
chmod 2775 /shared/team
```

The leading `2` sets the setgid bit.

Important points:

- **setgid does not grant write access** by itself.
- It only affects group inheritance and directory behavior.
- You still need the directory’s group mode or ACLs to allow collaboration.

This is why shared directories often use `setgid` together with default ACLs.

**Sticky bit for shared writable directories**

The sticky bit is commonly used on directories like `/tmp`. It allows users to create files in a shared writable directory but prevents them from deleting or renaming files they do not own.

Inspect it with:

```bash
ls -ld /tmp
```

You will see a `t` in the other execute position when sticky bit is set:

```bash
drwxrwxrwt 10 root root 4096 Apr 26 10:00 /tmp
```

Set it with:

```bash
chmod 1777 /shared/dropbox
```

Use the sticky bit when many users need write access but should not be able to remove each other’s files.

**ACLs and inheritance**

ACLs provide more granular control than traditional mode bits. They are especially useful when you need a shared directory to grant specific rights to a group while preserving a restrictive `umask` for users.

Two ACL concepts matter here:

- **access ACL**: applies to the current file or directory
- **default ACL**: inherited by new files and subdirectories created inside a directory

View ACLs with:

```bash
getfacl /shared/team
```

Set ACLs with:

```bash
setfacl -m g:devteam:rwx /shared/team
setfacl -d -m g:devteam:rwx /shared/team
```

The `-d` option sets a default ACL.

**Default ACL inheritance rules**

When a new object is created inside a directory with a default ACL, the parent’s default ACL is used to build the child’s initial ACL.

Key behavior:

- New **files** inherit the default ACL entries, but the file’s requested mode and `umask` still matter.
- New **directories** inherit the default ACL entries and also inherit default ACLs for their own future children.
- The ACL **mask** can reduce the effective rights of named users and groups.

This is one of the most important RHCSA details: the ACL entry may show `rwx`, but the **effective** permission can be lower because of the mask.

**ACL mask and effective permissions**

The ACL mask is the maximum effective permission for:

- named users
- named groups
- the owning group

It does not limit the owner entry or `other`.

Example ACL:

```bash
getfacl /shared/team
```

```text
# file: shared/team
# owner: root
# group: devteam
user::rwx
group::rwx
group:qa:rwx
mask::r-x
other::---
default:user::rwx
default:group::rwx
default:group:qa:rwx
default:mask::r-x
default:other::---
```

In this example:

- `group:qa:rwx` is present
- `mask::r-x` limits the effective rights to read and execute
- the effective permission for `group:qa` becomes `r-x`

You can see this in `getfacl` output, which often prints effective rights in comments:

```text
group:qa:rwx	#effective:r-x
```

Adjust the mask with:

```bash
setfacl -m m:rwx /shared/team
```

Or reduce it:

```bash
setfacl -m m:rx /shared/team
```

If you change a named group entry and the effective rights do not seem to change, inspect the mask first. This is a common troubleshooting step.

**How `umask`, default ACLs, and `setgid` interact**

In a shared directory, these mechanisms work together:

- `setgid` keeps group ownership consistent.
- default ACLs define what permissions new entries should receive.
- `umask` can still remove permissions from the requested mode, especially for files.

A practical pattern for a team directory is:

- directory owned by `root:devteam`
- mode `2775`
- default ACL granting group write
- users use `umask 0002`

This combination helps ensure that new files are group-writable and new subdirectories remain collaborative.

**Exam-focused example: shared directory with setgid and default ACL**

Suppose you configure:

```bash
mkdir -p /srv/project
chown root:devteam /srv/project
chmod 2775 /srv/project
setfacl -m g:devteam:rwx /srv/project
setfacl -d -m g:devteam:rwx /srv/project
setfacl -d -m o::--- /srv/project
```

Now a user with `umask 0002` creates:

```bash
touch /srv/project/report.txt
mkdir /srv/project/docs
```

Expected results:

- `report.txt`
  - owner: the creating user
  - group: `devteam` because of `setgid`
  - mode: typically `664` if the application requests `666` and `umask 0002` applies
  - ACL: group `devteam` remains writable if the ACL and mask allow it

- `docs`
  - owner: the creating user
  - group: `devteam`
  - mode: typically `2775` or `775`-style directory permissions depending on the creation path and ACLs
  - default ACLs are inherited by `docs` so future children also follow the same policy

A more exam-safe way to think about it:

- the file gets group ownership from `setgid`
- the file’s mode is reduced by `umask`
- the directory’s default ACL can restore or preserve group access
- the ACL mask may still cap the effective rights

If the directory has a default ACL of `group:devteam:rwx` and `mask:rwx`, then a new file often ends up effectively group-writable even when the user’s `umask` is restrictive, because the ACL can supply the group access that mode bits alone would remove.

**Why default ACLs are often needed with `umask`**

A restrictive `umask` such as `0022` is common on multiuser systems. That is good for privacy, but it can break collaboration in shared directories because new files become group-unwritable.

Default ACLs solve this by making the directory itself responsible for inherited collaboration rights.

Example:

```bash
umask 0022
touch /shared/team/file1
```

Without a default ACL, the file is usually `644`, so the group cannot edit it.

With a default ACL and a suitable mask, the file can still be group-writable even though the user’s `umask` is `0022`.

**Inspecting permissions and inheritance**

Use these commands to verify what is happening:

```bash
ls -ld /shared/team
ls -l /shared/team
getfacl /shared/team
stat /shared/team
```

What to look for:

- `ls -ld` shows the directory mode, group, and special bits
- `getfacl` shows access and default ACLs, including the mask
- `stat` shows numeric mode and ownership details

If a file is not inheriting as expected, check:

1. the parent directory’s group and `setgid`
2. the parent directory’s default ACLs
3. the ACL mask
4. the user’s `umask`

**Troubleshooting a broken shared directory**

A common RHCSA scenario is a team directory where users can create files, but others cannot edit them.

Step-by-step fix:

1. Confirm the directory ownership and mode.
   ```bash
   ls -ld /srv/team
   ```

2. Ensure the directory uses the correct group and setgid.
   ```bash
   chown root:devteam /srv/team
   chmod 2775 /srv/team
   ```

3. Add a default ACL for the team group.
   ```bash
   setfacl -m g:devteam:rwx /srv/team
   setfacl -d -m g:devteam:rwx /srv/team
   ```

4. Check the ACL mask.
   ```bash
   getfacl /srv/team
   ```
   If needed, widen it:
   ```bash
   setfacl -m m:rwx /srv/team
   setfacl -d -m m:rwx /srv/team
   ```

5. Ask users to use a collaborative `umask`.
   ```bash
   umask 0002
   ```

This sequence is often enough to restore expected shared-write behavior.

**Removing or resetting permissions when reverting**

When troubleshooting or returning a directory to normal behavior, you may need to remove special inheritance settings.

Remove ACLs:

```bash
setfacl -b /srv/team
setfacl -k /srv/team
```

- `-b` removes all access ACL entries
- `-k` removes default ACLs

Remove setgid:

```bash
chmod g-s /srv/team
```

Remove sticky bit:

```bash
chmod -t /srv/team
```

Reset `umask` in the current shell:

```bash
umask 0022
```

If you want to restore a directory to a plain mode-bit model, clear default ACLs first, then remove setgid if it is no longer needed.

> Be careful: removing ACLs can immediately change access for existing users and groups. Verify with `getfacl` before and after.

**Choosing the right tool**

| Need | Best tool | Notes |
|---|---|---|
| Keep new files private by default | `umask` | Simple per-user control |
| Keep group ownership consistent in a shared directory | `setgid` on directory | Does not grant write access by itself |
| Prevent users from deleting each other’s files | sticky bit | Common on dropbox-style directories |
| Grant inherited collaboration rights | default ACLs | Most flexible for shared team spaces |

A practical decision flow:

1. Start with `umask` for user-wide defaults.
2. Use `setgid` on shared directories to preserve group ownership.
3. Add default ACLs when mode bits alone are not enough.
4. Add sticky bit only when deletion protection is needed.

**Practical RHCSA examples**

Create a shared directory for a team:

```bash
groupadd devteam
mkdir -p /srv/dev
chown root:devteam /srv/dev
chmod 2775 /srv/dev
setfacl -m g:devteam:rwx /srv/dev
setfacl -d -m g:devteam:rwx /srv/dev
setfacl -d -m o::--- /srv/dev
```

Verify:

```bash
getfacl /srv/dev
ls -ld /srv/dev
```

Create a collaborative dropbox where users can add files but not delete others’ files:

```bash
mkdir -p /srv/dropbox
chmod 1777 /srv/dropbox
```

If you need both collaboration and controlled deletion, combine sticky bit with group ownership and default ACLs carefully.

**Common exam mistakes**

- Assuming `setgid` grants write access.
- Forgetting that the ACL mask can silently reduce effective rights.
- Checking only `ls -l` and ignoring `getfacl`.
- Using `umask` alone for a shared directory and then wondering why group members cannot edit files.
- Forgetting that default ACLs apply to new objects, not existing ones.
- Removing ACLs but leaving `setgid` in place, or vice versa, and not fully resetting the directory.

**Quick verification checklist**

- `ls -ld` shows the expected group and special bits.
- `getfacl` shows the expected access and default ACLs.
- The ACL mask allows the intended effective rights.
- New files and directories inherit the expected group ownership.
- `umask` is appropriate for the user workflow.
- Sticky bit is present only when deletion protection is required.

**Summary**

For RHCSA, think of directory inheritance as layered behavior:

- `umask` shapes the initial mode request.
- `setgid` preserves group ownership in shared directories.
- default ACLs define inherited permissions for new files and subdirectories.
- the ACL mask controls the effective rights of named users and groups.
- sticky bit protects shared writable directories from unwanted deletion.

If a shared directory behaves strangely, inspect all four layers before changing anything else. That habit saves time on the exam and in real administration.
