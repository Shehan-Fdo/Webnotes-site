**Linux permission model overview**

Linux file permissions control who can read, write, or execute a file or directory. On RHCSA, you need to understand both the meaning of permissions and how to change them quickly and accurately with `chmod`.

Each file or directory has three permission classes:

- **User**: the file owner
- **Group**: users in the file’s group
- **Other**: everyone else

Each class can have three basic permissions:

- **Read** (`r`)
- **Write** (`w`)
- **Execute** (`x`)

You inspect permissions with commands such as:

```bash
ls -l
stat file.txt
```

Example output:

```bash
-rw-r--r--. 1 alice dev 120 Apr 26 10:00 file.txt
drwxr-x---. 2 alice dev 4096 Apr 26 10:05 projects
```

The first character shows the file type:

- `-` regular file
- `d` directory
- `l` symbolic link

The next nine characters are grouped as user, group, and other permissions.

**Numeric mode basics**

Numeric mode uses octal values to represent permissions. This is the fastest way to set a complete permission pattern when you already know the desired result.

Permission values are:

- **Read** = `4`
- **Write** = `2`
- **Execute** = `1`

Add the values for each class:

- `7` = `4 + 2 + 1` = `rwx`
- `6` = `4 + 2` = `rw-`
- `5` = `4 + 1` = `r-x`
- `4` = `r--`
- `3` = `2 + 1` = `-wx`
- `2` = `-w-`
- `1` = `--x`
- `0` = `---`

A common full permission string is `755`:

- user: `rwx`
- group: `r-x`
- other: `r-x`

Another common value is `644`:

- user: `rw-`
- group: `r--`
- other: `r--`

Use numeric mode when you want to replace the entire permission set in one command.

```bash
chmod 644 report.txt
chmod 755 /usr/local/bin/backup.sh
chmod 700 ~/.ssh
```

**Symbolic mode basics**

Symbolic mode changes permissions by describing *who* gets *what*.

The general form is:

```bash
chmod [ugoa][+-=][rwx] file
```

Where:

- **u** = user
- **g** = group
- **o** = other
- **a** = all

Operators:

- `+` add permission
- `-` remove permission
- `=` set exactly

Examples:

```bash
chmod u+x script.sh
chmod g-w shared.txt
chmod o=r notes.txt
chmod a+r public.txt
```

Symbolic mode is useful when you want to make a small change without disturbing the rest of the permission set.

**Numeric vs symbolic mode**

Use numeric mode when you know the exact final permission set. Use symbolic mode when you want to modify one class or one permission without recalculating the whole mode.

| Mode | Best use case | Example |
|---|---|---|
| Numeric | Set exact permissions quickly | `chmod 640 file.txt` |
| Symbolic | Adjust specific bits safely | `chmod g+w file.txt` |

Examples of equivalent results:

```bash
chmod 755 app.sh
chmod u=rwx,g=rx,o=rx app.sh
```

Both commands produce the same permission pattern.

**Directory permissions and why execute matters**

Directories use permissions differently from regular files:

- **Read** allows listing names in the directory
- **Write** allows creating, deleting, or renaming entries
- **Execute** allows entering the directory and accessing metadata for items inside

A directory without execute permission is effectively inaccessible, even if it has read permission.

Examples:

```bash
chmod 755 /srv/data
chmod 750 /srv/secure
chmod 700 /root/private
```

Practical interpretation:

- `755` for shared read access with owner control
- `750` for group access but no access for others
- `700` for private directories

> A directory needs `x` for traversal. Without it, users cannot `cd` into the directory or access files by path.

**Common RHCSA permission patterns**

These patterns appear often in administration tasks:

- `644` for regular text files that should be readable by everyone but writable only by the owner
- `600` for private files such as credentials or keys
- `700` for private directories
- `755` for executable scripts and shared directories
- `750` for group-shared administrative directories

Examples:

```bash
chmod 600 ~/.ssh/id_rsa
chmod 644 /etc/example.conf
chmod 755 /usr/local/bin/cleanup.sh
chmod 750 /srv/team
```

**Special permission bits**

Linux also supports three special bits that extend standard permissions:

- **setuid** (`u+s`)
- **setgid** (`g+s`)
- **sticky bit** (`+t`)

These are important on RHCSA because they affect how programs and directories behave.

**setuid**

When set on an executable file, the process runs with the file owner’s effective user ID. This is often used for privileged utilities.

Numeric representation adds a leading digit:

- `4` for setuid
- `2` for setgid
- `1` for sticky bit

Example:

```bash
chmod 4755 /path/to/program
```

Symbolic equivalent:

```bash
chmod u+s /path/to/program
```

**setgid**

On executable files, setgid makes the process run with the file group’s effective group ID. On directories, it causes new files and subdirectories to inherit the directory’s group, which is useful for shared project areas.

Examples:

```bash
chmod 2750 /srv/project
chmod g+s /srv/project
```

**sticky bit**

On directories, the sticky bit prevents users from deleting or renaming files they do not own, even if the directory is writable. This is common on shared writable directories such as `/tmp`.

Examples:

```bash
chmod 1777 /tmp
chmod +t /shared/dropbox
```

**Reading special bits in `ls -l`**

Special bits appear in the execute positions:

- `s` or `S` for setuid/setgid
- `t` or `T` for sticky bit

Examples:

- `rws` means execute is set and the special bit is active
- `rwS` means special bit is set but execute is not
- `rwt` means sticky bit and execute are set
- `rwT` means sticky bit is set but execute is not

**Practical examples**

Set a script to be executable by the owner and readable by everyone:

```bash
chmod 755 deploy.sh
```

Make a private SSH key readable only by the owner:

```bash
chmod 600 ~/.ssh/id_ed25519
```

Allow a group to collaborate in a directory while preserving group ownership on new files:

```bash
chmod 2770 /srv/teamshare
chgrp developers /srv/teamshare
```

Create a world-writable temporary directory with sticky bit protection:

```bash
chmod 1777 /srv/public-drop
```

**Using `chmod` safely**

Before changing permissions, inspect the current state:

```bash
ls -l file.txt
stat file.txt
```

After changing permissions, verify the result:

```bash
ls -l file.txt
stat -c '%A %a %n' file.txt
```

Useful `stat` format example:

```bash
stat -c 'Permissions: %A  Octal: %a  Name: %n' file.txt
```

This helps you confirm both the symbolic and numeric view.

**Common mistakes**

- Using `chmod 777` when a narrower permission set is appropriate
- Forgetting that directories need execute permission for access
- Applying execute permission to data files that do not need it
- Confusing file permissions with ownership; `chmod` does not change owner or group
- Assuming setgid on a file behaves the same as setgid on a directory

> Avoid `chmod 777` unless you have a specific, justified reason. It often creates unnecessary security exposure.

**Practical troubleshooting workflow**

When permissions do not behave as expected, check the following in order:

1. Confirm ownership with `ls -l`.
2. Confirm the permission bits with `stat` or `ls -l`.
3. Check directory traversal permissions on every parent directory.
4. Verify whether ACLs (Access Control Lists) or SELinux are also affecting access.
5. Adjust only the minimum permission needed.

Example diagnostic commands:

```bash
namei -l /srv/teamshare/report.txt
getfacl /srv/teamshare/report.txt
ls -Zd /srv/teamshare/report.txt
```

`namei -l` is especially useful because it shows permissions on each path component.

**Quick reference**

| Symbolic | Numeric | Meaning |
|---|---:|---|
| `rwx` | `7` | read, write, execute |
| `rw-` | `6` | read, write |
| `r-x` | `5` | read, execute |
| `r--` | `4` | read only |
| `-wx` | `3` | write, execute |
| `-w-` | `2` | write only |
| `--x` | `1` | execute only |
| `---` | `0` | no permissions |

**Exam-focused takeaways**

- Use **numeric mode** for exact permission sets.
- Use **symbolic mode** for targeted changes.
- Remember that **directories require execute permission** for traversal.
- Know the special bits: **setuid**, **setgid**, and **sticky bit**.
- Verify results with `ls -l`, `stat`, and `namei -l`.
