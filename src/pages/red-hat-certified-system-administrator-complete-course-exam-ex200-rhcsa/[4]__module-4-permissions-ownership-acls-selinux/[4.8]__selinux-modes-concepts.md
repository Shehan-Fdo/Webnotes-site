## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**SELinux overview**

<abbr title="Security-Enhanced Linux">SELinux</abbr> is a mandatory access control system that adds policy enforcement on top of standard Unix permissions. It limits what a process can do even when file ownership and mode bits would otherwise allow access. On RHCSA systems, you usually work with SELinux in enforcing mode and focus on labels, booleans, ports, and troubleshooting.

The core idea is simple: access decisions depend on the relationship between the subject, the object, and the policy. A process runs in a domain, files and directories have security contexts, and the policy decides whether that domain may access that label on that class of object.

- **Subject**: the active process requesting access
- **Object**: the resource being accessed, such as a file, port, or socket
- **Policy**: the rule set that defines allowed interactions

**SELinux modes and policy types**

SELinux has operational modes and policy types. The mode controls enforcement behavior, while the policy type controls the rule set in use.

- **Enforcing**: SELinux checks policy and blocks denied actions
- **Permissive**: SELinux logs denials but does not block them
- **Disabled**: SELinux is turned off; labels are not enforced

Common policy types include:

- **targeted**: the standard Red Hat policy; most system services run confined, while user sessions are less restricted
- **minimum**: a lighter policy set than targeted; less common in RHCSA environments
- **mls**: <abbr title="Multi-Level Security">MLS</abbr> policy for strict sensitivity levels and categories; uncommon for RHCSA, but important to recognize

Check the current mode and policy with:

```bash
getenforce
sestatus
```

Example output concepts to recognize:

```bash
$ getenforce
Enforcing

$ sestatus
SELinux status:                 enabled
Current mode:                   enforcing
Mode from config file:          enforcing
Policy from config file:        targeted
```

> If SELinux is disabled, labels may still exist on disk, but policy enforcement is not active until SELinux is enabled again and the system relabels if needed.

**Security contexts and labels**

SELinux labels are called security contexts. A typical context has four fields:

```text
user:role:type:level
```

For RHCSA, the most important field is usually the **type**. Type enforcement is the main mechanism you use when troubleshooting services.

Example labels:

```bash
ls -Z /var/www/html
ps -eZ | grep httpd
id -Z
```

Typical context examples:

- `system_u:object_r:httpd_sys_content_t:s0` for web content
- `system_u:system_r:httpd_t:s0` for the Apache process domain
- `unconfined_u:unconfined_r:unconfined_t:s0-s0:c0.c1023` for an unconfined user session

The label fields mean:

- **user**: SELinux user identity
- **role**: allowed role for the subject
- **type**: primary access control label for most RHCSA tasks
- **level**: sensitivity and category range, mainly relevant to MLS/MCS

**Process domains and domain transitions**

A process type is not just a label you read; it is the domain the process runs in. When a service starts, SELinux may transition the process from one domain to another based on policy. This is a core concept for understanding why a daemon can read some files and not others.

For example:

- `systemd` starts a service
- policy allows the executable to transition into a confined domain
- the running process receives a domain such as `httpd_t`, `smbd_t`, or `ftpd_t`

This matters because the process domain determines what the service can access. If Apache runs in `httpd_t`, it can only read content labeled for HTTP access unless policy or booleans allow more.

Useful checks:

```bash
ps -eZ | grep -E 'httpd|smb|vsftpd'
```

You can compare the executable label and the running process domain to understand transitions:

```bash
ls -Z /usr/sbin/httpd
ps -eZ | grep httpd
```

**File contexts and default labeling**

SELinux uses file context rules to assign labels to files and directories. The system stores default mappings and applies them when files are created or relabeled.

Inspect contexts with:

```bash
ls -Z /path/to/file
matchpathcon /var/www/html/index.html
semanage fcontext -l | grep /var/www
```

Common file types you should recognize:

- **`httpd_sys_content_t`**: static web content readable by Apache
- **`httpd_sys_rw_content_t`**: web content Apache may write to
- **`samba_share_t`**: content shared by Samba
- **`public_content_t`**: content intended for broad read access
- **`public_content_rw_t`**: content intended for broad read/write access
- **`ftpd_...` types**: FTP-related content labels, depending on use case

When you create a directory tree for a service, the label often does not match the service automatically. You usually fix this with `semanage fcontext` and `restorecon`, not with `chcon` alone.

Example for Apache content:

```bash
sudo mkdir -p /srv/web/site
sudo semanage fcontext -a -t httpd_sys_content_t '/srv/web/site(/.*)?'
sudo restorecon -Rv /srv/web/site
ls -Zd /srv/web/site
```

**Temporary changes vs persistent changes**

Use `chcon` for a quick, temporary label change. Use `semanage fcontext` for a persistent rule.

```bash
sudo chcon -t httpd_sys_content_t /srv/web/site/index.html
sudo semanage fcontext -a -t httpd_sys_content_t '/srv/web/site(/.*)?'
sudo restorecon -Rv /srv/web/site
```

Why this matters:

- `chcon` changes the label on the file directly
- `restorecon` resets labels to the policy default
- `semanage fcontext` changes the default mapping so future relabels preserve your intended context

**Equivalence mappings for mirrored directory trees**

When you mirror a directory structure and want SELinux to treat one path like another, use file context equivalence mappings with `semanage fcontext -e`. This is useful when you want a custom path to inherit the same labeling rules as an existing path.

Example:

```bash
sudo semanage fcontext -a -e /var/www /srv/www
sudo restorecon -Rv /srv/www
```

This tells SELinux to apply the same context rules used for `/var/www` to `/srv/www`. It is especially helpful when migrating content or building a parallel directory tree.

**Booleans**

SELinux booleans are on/off switches that adjust policy behavior without changing the policy itself. They are common in RHCSA troubleshooting because they enable specific service capabilities.

List booleans:

```bash
getsebool -a
```

Check one boolean:

```bash
getsebool httpd_enable_homedirs
```

Set a boolean temporarily:

```bash
sudo setsebool httpd_enable_homedirs on
```

Set it persistently:

```bash
sudo setsebool -P httpd_enable_homedirs on
```

Exam-practical examples:

- **Apache reading home directories**
  - `httpd_enable_homedirs`
- **Apache making outbound network connections**
  - `httpd_can_network_connect`
- **Samba exporting home directories**
  - `samba_enable_home_dirs`
- **FTP access to home directories or writable content**
  - FTP-related booleans may vary by scenario and installed services

A practical workflow is:

1. Identify the service and the denied action.
2. Check whether a boolean exists for that behavior.
3. Enable it temporarily first if you are testing.
4. Make it persistent only after confirming it solves the problem.

Example:

```bash
sudo setsebool -P httpd_can_network_connect on
getsebool httpd_can_network_connect
```

**Ports and SELinux**

SELinux also controls which ports a service may bind to. A service can fail even when the port is free if the port type is not allowed.

Inspect port mappings:

```bash
semanage port -l | grep http
semanage port -l | grep samba
```

If you move Apache to a nonstandard port, you may need to label that port for HTTP:

```bash
sudo semanage port -a -t http_port_t -p tcp 8080
```

If the port already exists under another type, modify it instead:

```bash
sudo semanage port -m -t http_port_t -p tcp 8080
```

This is common when a service listens on a custom port but SELinux still expects the default port type.

**Troubleshooting denials**

When access fails, do not assume Unix permissions are the only issue. SELinux denials often appear in audit logs.

Useful commands:

```bash
sudo ausearch -m AVC,USER_AVC -ts recent
sudo journalctl -t setroubleshoot --since "10 min ago"
sudo tail -f /var/log/audit/audit.log
```

An <abbr title="Access Vector Cache">AVC</abbr> denial usually includes the source context, target context, class, and permission. Read those fields to identify the exact mismatch.

Example denial pattern:

```text
avc:  denied  { read } for  pid=1234 comm="httpd" name="index.html"
scontext=system_u:system_r:httpd_t:s0
tcontext=unconfined_u:object_r:default_t:s0
tclass=file
```

Interpretation:

- `scontext` shows the process domain
- `tcontext` shows the file label
- `tclass` shows the object type, such as `file`, `dir`, or `tcp_socket`
- the denied permission tells you what the process tried to do

**Using sealert for actionable fixes**

`sealert` can summarize audit events and suggest a likely fix. This is useful when you need to move quickly during troubleshooting.

Example:

```bash
sudo sealert -a /var/log/audit/audit.log
```

Typical output may point you toward one of these fixes:

- relabel the file or directory with `restorecon`
- add a persistent file context rule with `semanage fcontext`
- enable a boolean with `setsebool -P`
- adjust a port label with `semanage port`

Treat `sealert` as a guide, then verify the suggested change rather than applying it blindly.

**Relabeling and recovery**

If labels are badly damaged or you need a full relabel, SELinux can rebuild contexts at boot. A common recovery method is to create `/.autorelabel` and reboot.

```bash
sudo touch /.autorelabel
sudo reboot
```

On the next boot, the system performs a full filesystem relabel. This is useful after major policy changes, restoring from backups, or fixing widespread label corruption.

You can also relabel specific paths without rebooting:

```bash
sudo restorecon -Rv /var/www /srv/www
```

Use `restorecon` for targeted repairs and `/.autorelabel` for broad recovery.

**Common service scenarios**

Apache, Samba, and FTP are the most exam-relevant services for SELinux practice.

- **Apache**
  - static content under `/var/www/html` usually works with default labels
  - custom document roots often need `semanage fcontext` and `restorecon`
  - booleans may be needed for home directories or outbound connections

- **Samba**
  - shared directories need the correct Samba file type
  - booleans may be needed for home directory sharing or writable shares
  - verify both file labels and share configuration

- **FTP**
  - writable directories often need a writable public or FTP-related label
  - booleans may be required for access to user home directories or anonymous content
  - confirm the daemon domain and the directory label together

Example Apache workflow:

```bash
sudo mkdir -p /srv/www/site
echo "Hello SELinux" | sudo tee /srv/www/site/index.html
sudo semanage fcontext -a -t httpd_sys_content_t '/srv/www/site(/.*)?'
sudo restorecon -Rv /srv/www/site
sudo systemctl restart httpd
curl http://localhost
```

Example Samba-style label workflow:

```bash
sudo mkdir -p /srv/samba/share
sudo semanage fcontext -a -t samba_share_t '/srv/samba/share(/.*)?'
sudo restorecon -Rv /srv/samba/share
ls -Zd /srv/samba/share
```

**Verification checklist**

Before you declare a SELinux issue solved, verify all relevant layers:

- The service is running
- The process domain is correct
- The file or directory label matches the service
- The port label matches the listening port
- Any required boolean is enabled
- The audit log no longer shows new denials

Useful commands:

```bash
getenforce
sestatus
ls -Z /path
ps -eZ
getsebool -a | grep httpd
semanage fcontext -l
semanage port -l
```

**Practical mental model**

Think of SELinux as a label-based gatekeeper:

- A process starts in a domain
- The domain can only access objects with allowed labels
- Booleans open or close specific policy paths
- Ports are labeled too, not just files
- Relabeling restores policy-consistent state when labels drift

If you remember only one troubleshooting pattern, use this:

1. Confirm the service and its process domain.
2. Check the target file or port label.
3. Review AVC denials.
4. Apply the smallest persistent fix that matches the policy model.
