## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**SELinux overview**

Security-Enhanced Linux (<abbr title="Security-Enhanced Linux">SELinux</abbr>) is a mandatory access control system that adds policy-based restrictions on top of standard Linux permissions. On RHCSA, you are expected to recognize when SELinux is the cause of an access problem, inspect labels, and fix the label or policy in a safe, persistent way.

SELinux decisions depend on the label attached to the subject and object involved in an action:

- A **process** has a security context.
- A **file** or directory has a security context.
- A **port** can also have a security context.
- The SELinux policy compares those contexts to decide whether access is allowed.

A common exam pattern is: permissions look correct, but access still fails. In that case, you check SELinux mode, inspect labels, review audit logs, and restore or define the correct context.

**SELinux modes and why they matter**

SELinux runs in one of three modes:

- **Enforcing**: policy is active and denied actions are blocked and logged.
- **Permissive**: policy is not enforced, but denials are still logged.
- **Disabled**: SELinux is off; labels are not used for enforcement.

Use these commands to inspect the current mode:

```bash
getenforce
sestatus
```

Typical output examples:

```bash
$ getenforce
Enforcing
```

```bash
$ sestatus
SELinux status:                 enabled
Current mode:                   enforcing
Mode from config file:          enforcing
Policy version:                 targeted
```

Practical impact:

- In **enforcing** mode, a mislabeled file can break a service immediately.
- In **permissive** mode, the service may appear to work, but audit logs still reveal the denial.
- In **disabled** mode, label problems are hidden during troubleshooting, which can create surprises after re-enabling SELinux.

> If you change `/etc/selinux/config`, the new setting usually takes effect after reboot. Use `setenforce` only for temporary runtime changes.

Temporary mode changes:

```bash
setenforce 0   # permissive
setenforce 1   # enforcing
```

**SELinux context structure**

A full SELinux context usually appears as `user:role:type:level`.

Example:

```bash
system_u:object_r:httpd_sys_content_t:s0
```

Each field has a distinct purpose:

- **user**: SELinux user identity, such as `system_u`, `unconfined_u`, or `user_u`.
- **role**: SELinux role, such as `object_r` for files or `system_r` for processes.
- **type**: the most important field for everyday access control on RHCSA.
- **level**: MLS/MCS <abbr title="Multi-Level Security">MLS</abbr>/<abbr title="Multi-Category Security">MCS</abbr> sensitivity or category range, such as `s0` or `s0:c123,c456`.

Why each field matters:

- The **type** field is the main label used by targeted policy on typical RHEL systems.
- The **role** field matters more for process transitions and role-based policy decisions.
- The **user** field matters when SELinux users are mapped to Linux users or when you inspect full contexts.
- The **level** field matters when categories or sensitivity ranges are in use, especially with containers or MLS-style labeling.

A few examples:

- A web file might be labeled `httpd_sys_content_t`.
- A web process often runs as `httpd_t`.
- A port used by HTTP service may be labeled `http_port_t`.

**File contexts, process contexts, and port contexts**

SELinux uses different context types for different objects.

- **File context**: controls access to files and directories.
- **Process context**: controls what a running program can do.
- **Port context**: controls which services may bind to network ports.

Examples:

```bash
ls -Z /var/www/html
ps -eZ | grep httpd
semanage port -l | grep http_port_t
```

Interpretation:

- `ls -Z` shows file labels on directories and files.
- `ps -eZ` shows process labels.
- `semanage port -l` shows which port types are allowed for which TCP/UDP ports.

Concrete example:

- Apache reads content from `/var/www/html` only if the files are labeled with a type allowed by the `httpd_t` process.
- If Apache listens on a nonstandard port, that port must be labeled with `http_port_t` or an equivalent allowed port type.

**Inspecting labels**

Use `ls -Z` to inspect file labels:

```bash
ls -Z /var/www
ls -Zd /var/www/html
```

Use `ps -Z` or `ps -eZ` to inspect process labels:

```bash
ps -Z -C httpd
ps -eZ | head
```

Use `id -Z` to inspect your current shell context:

```bash
id -Z
```

Use `semanage fcontext -l` to inspect persistent file context rules:

```bash
semanage fcontext -l | grep /var/www
```

Use `semanage port -l` to inspect port mappings:

```bash
semanage port -l | grep http
```

**File context management workflow**

File contexts are the most common RHCSA SELinux task. You usually do not change the label directly with `chcon` unless you need a temporary test. Instead, you define or adjust the persistent rule and then apply it.

Common tools:

- `restorecon`: restores the default label based on policy and local file context rules.
- `semanage fcontext`: manages persistent file context mappings.
- `chcon`: changes a label directly on a file, but the change is not persistent across relabeling.

Typical workflow:

1. Check the current label.
2. Check the expected label pattern.
3. Add or modify a persistent rule.
4. Apply the label with `restorecon`.
5. Verify the result.

Example:

```bash
ls -Zd /webdata
semanage fcontext -l | grep /webdata
semanage fcontext -a -t httpd_sys_content_t '/webdata(/.*)?'
restorecon -Rv /webdata
ls -Zd /webdata
```

The regex-style path in `semanage fcontext` matters:

- `'/webdata(/.*)?'` matches the directory and everything under it.
- Without the recursive pattern, only the exact path may be matched.

**Adding, modifying, and deleting file context rules**

Use `semanage fcontext -a` to add a new persistent mapping:

```bash
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
```

Use `semanage fcontext -m` to modify an existing mapping:

```bash
semanage fcontext -m -t httpd_sys_content_t '/srv/www(/.*)?'
```

Use `semanage fcontext -d` to delete a mapping:

```bash
semanage fcontext -d '/srv/www(/.*)?'
```

Verify the rule list:

```bash
semanage fcontext -l | grep '/srv/www\|/srv/web'
```

Important behavior:

- `-a` fails if a matching rule already exists.
- `-m` is useful when you need to change the type or pattern of an existing rule.
- `-d` removes the persistent rule, but it does not automatically relabel files until you run `restorecon` or relabel the system.

**restorecon behavior and forced relabeling**

`restorecon` resets labels to the policy default or to the local custom mapping defined by `semanage fcontext`.

Basic usage:

```bash
restorecon -v /var/www/html/index.html
restorecon -Rv /var/www/html
```

Useful options:

- `-R`: recurse through directories.
- `-v`: show what changed.
- `-F`: force a reset even if the current label appears acceptable.

Use `-F` when you need to override a label that may be considered equivalent or when a file has a label that you want to replace explicitly with the policy-defined label.

Example:

```bash
restorecon -RFv /srv/web
```

When a forced relabel may be needed:

- A file was copied from another location and inherited an unexpected label.
- A directory has a label that looks close enough but is not the exact expected type.
- You changed local file context rules and want to ensure all files are reset cleanly.

For a broader relabel, you may need to trigger a full filesystem relabel at boot by creating `/.autorelabel` and rebooting, especially after major label corruption or policy changes.

```bash
touch /.autorelabel
reboot
```

**`chcon` versus `semanage fcontext`**

Use `chcon` for temporary testing and `semanage fcontext` for permanent policy-aligned fixes.

Examples:

```bash
chcon -t httpd_sys_content_t /srv/web/index.html
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
restorecon -Rv /srv/web
```

Key distinction:

- `chcon` changes the label on the file immediately.
- `semanage fcontext` changes the rule that `restorecon` uses to set labels persistently.

If you use only `chcon`, the label can disappear after relabeling, package updates, or `restorecon`.

**Equivalence and special cases**

SELinux supports file context equivalence, which lets one path inherit the labeling rules of another path. This is useful when you want a custom directory tree to behave like a standard one.

Example:

```bash
semanage fcontext -a -e /var/www /srv/www
restorecon -Rv /srv/www
```

This tells SELinux to treat `/srv/www` as equivalent to `/var/www` for labeling purposes.

Common special cases:

- **Home directories**: user home content may need labels such as `user_home_t`, `user_home_dir_t`, or service-specific types if a daemon must read from them.
- **Mount points**: a mounted filesystem may not inherit the labels you expect from the parent directory.
- **Nonstandard filesystems**: some filesystems may not support SELinux labels normally, or may require mount options and policy-aware handling.
- **NFS and removable media**: labeling behavior can differ, so you often need to verify support and policy expectations rather than assume standard local-disk behavior.

Practical caution:

> If a directory is a mount point, relabeling the parent path may not affect the mounted filesystem contents the way you expect. Always verify the actual mounted path with `mount`, `findmnt`, and `ls -Z`.

Useful checks:

```bash
findmnt /srv/web
mount | grep /srv/web
ls -Zd /srv/web
```

**Troubleshooting denials**

When access fails, inspect the audit log first. SELinux denials are usually recorded in `/var/log/audit/audit.log` or the journal.

Useful commands:

```bash
ausearch -m AVC,USER_AVC -ts recent
grep AVC /var/log/audit/audit.log
journalctl -t setroubleshoot --since "10 min ago"
```

If `setroubleshoot` is installed, `sealert` can summarize likely causes:

```bash
sealert -a /var/log/audit/audit.log
```

You can also use `audit2why` to interpret why a denial happened:

```bash
ausearch -m AVC -ts recent | audit2why
```

Typical interpretation:

- The denial may indicate a mislabeled file.
- The denial may show that a process type is not allowed to access a file type.
- The denial may reveal a port type mismatch for a service binding to a nonstandard port.

Example denial workflow:

1. Reproduce the failure.
2. Check the current SELinux mode with `getenforce`.
3. Inspect the target file label with `ls -Z`.
4. Review AVC messages with `ausearch` or `grep AVC`.
5. Determine whether the fix is a label correction, a port mapping, or a policy adjustment.
6. Apply the least invasive persistent fix.
7. Re-test the service.

**Practical file labeling example**

Suppose you want Apache to serve content from `/srv/web`.

1. Create the directory and content:

```bash
mkdir -p /srv/web
echo "Hello from SELinux" > /srv/web/index.html
```

2. Check the current label:

```bash
ls -Zd /srv/web /srv/web/index.html
```

3. Add a persistent file context rule:

```bash
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
```

4. Apply the label:

```bash
restorecon -Rv /srv/web
```

5. Verify:

```bash
ls -Zd /srv/web /srv/web/index.html
```

If Apache still cannot read the content, check the process context and audit logs before changing anything else.

**Practical process context example**

A process context helps you understand what domain a service runs in.

```bash
ps -eZ | grep httpd
```

Example output might show `httpd_t` for Apache worker processes. That matters because the policy allows `httpd_t` to access only certain file types and ports.

If a service is running in an unexpected domain, the issue may be service configuration, not file labeling.

**Practical port context example**

If Apache must listen on port `8080`, the port type must allow it.

Check current port mappings:

```bash
semanage port -l | grep http_port_t
```

Add a port if needed:

```bash
semanage port -a -t http_port_t -p tcp 8080
```

If the port already exists under another type, modify it instead:

```bash
semanage port -m -t http_port_t -p tcp 8080
```

Verify:

```bash
semanage port -l | grep 8080
```

This is a common exam distinction:

- File labels control what content a service can read or write.
- Port labels control what network ports a service can bind to.
- Process labels control what the running service domain is allowed to do.

**Common RHCSA troubleshooting patterns**

- A file copied into `/var/www/html` has the wrong label, so Apache returns `403 Forbidden`.
- A custom web directory works after `chcon`, but breaks again after `restorecon` because the rule was never made persistent.
- A service listens on a nondefault port, but SELinux blocks the bind until the port is labeled correctly.
- A directory under a home path needs a custom rule or equivalence mapping before a daemon can access it.
- A mounted filesystem does not inherit the expected labels, so you must inspect the mount and apply the correct context strategy.

**Quick command reference**

```bash

sestatus
setenforce 0
setenforce 1

```

**Exam strategy**

- Start by checking the SELinux mode.
- Confirm whether the problem is file, process, or port related.
- Prefer persistent fixes with `semanage` plus `restorecon`.
- Use `chcon` only when you need a temporary test or quick validation.
- Read AVC denials carefully before making changes.
- Verify the final state with `ls -Z`, `ps -Z`, `semanage fcontext -l`, or `semanage port -l`.

**Key takeaways**

SELinux labels are not just metadata; they are part of the access decision. For RHCSA, you need to know how to inspect the current mode, interpret the full context structure, distinguish file/process/port labels, and apply persistent fixes with `semanage` and `restorecon`. Good troubleshooting starts with audit logs and ends with a verified label change, not a guess.
