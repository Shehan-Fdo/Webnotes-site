## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

Linux file permissions include three special bits that change how execution and directory access behave: **setuid**, **setgid**, and the **sticky bit**. These are common RHCSA topics because they affect privilege handling, shared directories, and secure multi-user workflows.

The key idea is that these bits do not replace normal read/write/execute permissions. Instead, they modify how the kernel interprets access for a file or directory. You should know how to identify them, set them, verify them, and understand the security implications.

**Special permission bits at a glance**

| Bit | Applies to | Effect |
|---|---|---|
| setuid | Executable files | Runs the program with the file owner’s effective user ID |
| setgid | Executable files | Runs the program with the file group’s effective group ID |
| setgid | Directories | New files inherit the directory’s group; subdirectories often inherit setgid too |
| sticky bit | Directories | Only file owner, directory owner, or root can delete/rename entries |

**How to read them in `ls -l` output**

Special bits appear in the execute field:

- `s` means the special bit is set and execute is also set.
- `S` means the special bit is set but execute is not set.
- `t` means sticky bit is set and execute is also set.
- `T` means sticky bit is set but execute is not set.

Examples:

- `-rwsr-xr-x` indicates **setuid** on an executable.
- `-rwxr-sr-x` indicates **setgid** on an executable.
- `drwxrwxrwt` indicates the **sticky bit** on a directory such as `/tmp`.

**Numeric and symbolic modes**

You can set special bits with either symbolic or numeric notation.

```bash
chmod u+s file
chmod g+s directory
chmod +t directory
chmod 4755 program
chmod 2755 shared_dir
chmod 1777 /tmp
```

The leading digit in numeric mode represents special bits:

- `4` = setuid
- `2` = setgid
- `1` = sticky bit

Combine them as needed:

- `4755` = setuid + `755`
- `2755` = setgid + `755`
- `1777` = sticky bit + `777`
- `6755` = setuid + setgid + `755`

**setuid**

**setuid** on an executable makes the process run with the file owner’s effective user ID, not the user who launched it. This is useful when a normal user needs a controlled action that requires elevated privileges.

A classic example is `passwd`, which must update protected system files. On many systems, it is owned by `root` and has setuid enabled.

```bash
ls -l /usr/bin/passwd
-rwsr-xr-x. 1 root root ... /usr/bin/passwd
```

Practical implications:

- The program may access files the invoking user cannot access directly.
- The effective user ID changes during execution, but the real user ID remains the caller.
- Poorly written setuid programs can create serious security risks.

> Be careful with setuid binaries. Any bug in a privileged executable can become a privilege-escalation path.

Useful checks:

```bash
find / -perm -4000 -type f 2>/dev/null
```

This finds files with the setuid bit set. On RHCSA systems, you should be able to recognize why a file is listed and whether it is expected.

**setgid on files**

**setgid** on an executable works like setuid, but for the effective group ID. The program runs with the file’s group privileges.

Example:

```bash
ls -l /usr/bin/write
-rwxr-sr-x. 1 root tty ... /usr/bin/write
```

This is less common than setuid, but the concept is the same:

- The process gains the file group’s effective group ID.
- Access decisions may change based on group membership.
- It is useful for controlled group-based access.

You can search for setgid files with:

```bash
find / -perm -2000 -type f 2>/dev/null
```

**setgid on directories**

On directories, **setgid** is especially important for collaboration. New files and subdirectories created inside a setgid directory inherit the directory’s group instead of the creator’s primary group.

This is useful for shared project directories where all members should work under the same group ownership.

Example setup:

```bash
groupadd projectx
mkdir /srv/projectx
chown root:projectx /srv/projectx
chmod 2775 /srv/projectx
```

What this does:

- `root:projectx` sets the directory owner and group.
- `2775` enables setgid and gives group write access.
- Files created inside inherit group `projectx`.

Verify:

```bash
ls -ld /srv/projectx
drwxrwsr-x. 2 root projectx ... /srv/projectx
```

Typical workflow:

1. Create a shared group.
2. Assign the directory to that group.
3. Enable setgid on the directory.
4. Ensure group members have write access.

If you also need to control deletion in a shared directory, combine setgid with the sticky bit only when the use case requires it.

**Sticky bit**

The **sticky bit** is mainly used on directories. It prevents users from deleting or renaming files they do not own, even if the directory is writable by many users.

The most common example is `/tmp`:

```bash
ls -ld /tmp
drwxrwxrwt. 10 root root ... /tmp
```

Why it matters:

- Without sticky bit, any user with write access to a shared directory could remove other users’ files.
- With sticky bit, deletion is restricted to the file owner, directory owner, or `root`.

This is essential for world-writable shared directories.

Set it with:

```bash
chmod +t /shared
chmod 1777 /shared
```

Search for sticky directories:

```bash
find / -type d -perm -1000 2>/dev/null
```

**Common administrative examples**

A shared team directory with controlled group inheritance:

```bash
groupadd devteam
mkdir -p /srv/devteam
chown root:devteam /srv/devteam
chmod 2775 /srv/devteam
```

A public drop area where users can create files but not delete each other’s content:

```bash
mkdir -p /srv/dropbox
chmod 1777 /srv/dropbox
```

A privileged helper binary:

```bash
ls -l /usr/bin/passwd
-rwsr-xr-x. 1 root root ... /usr/bin/passwd
```

**How to verify special bits**

Use `ls -l` for a quick visual check and `stat` for precise details.

```bash
stat /usr/bin/passwd
stat /srv/projectx
```

Example output fields to notice:

- Access mode
- Owner and group
- File type
- Special bits in the mode string

For a more targeted view:

```bash
stat -c '%A %a %U %G %n' /srv/projectx
```

This prints:

- symbolic permissions
- numeric permissions
- owner
- group
- file name

**How to set and remove special bits**

Symbolic methods are often easiest to read:

```bash
chmod u+s /path/to/file
chmod g+s /path/to/file-or-dir
chmod +t /path/to/dir
chmod u-s /path/to/file
chmod g-s /path/to/file-or-dir
chmod -t /path/to/dir
```

Numeric methods are compact and common in exams:

```bash
chmod 4755 /usr/local/bin/tool
chmod 2750 /srv/shared
chmod 1777 /tmp
```

Remember:

- `chmod 755 file` removes special bits unless you include the leading digit.
- `chmod 0755 file` is different from `chmod 4755 file`.
- The leading digit is required to preserve or set special permissions in numeric form.

**Security considerations**

Special bits are powerful and should be used sparingly.

- **setuid** increases risk because it grants elevated execution context.
- **setgid** on executables is less common but still sensitive.
- **sticky bit** is a directory protection mechanism, not a privilege escalation tool.
- World-writable directories should almost always use sticky bit if multiple users share them.

Best practices:

- Prefer group-based access control before special bits.
- Use setgid directories for collaboration instead of making files globally writable.
- Audit setuid binaries regularly.
- Avoid placing custom setuid programs on systems unless absolutely necessary.

**Troubleshooting and exam tips**

If a permission string looks unusual, inspect the execute positions:

- `s` or `S` in the user execute slot points to setuid.
- `s` or `S` in the group execute slot points to setgid.
- `t` or `T` in the others execute slot points to sticky bit.

If a shared directory is not inheriting the expected group:

- Confirm the directory has setgid enabled.
- Confirm the parent directory permissions allow creation.
- Confirm the user is a member of the target group.
- Confirm the user’s umask is not removing group write access.

Example check:

```bash
id alice
ls -ld /srv/projectx
umask
```

If users can delete each other’s files in a shared directory:

- Check whether sticky bit is set.
- Check whether the directory is world-writable.
- Confirm ownership and permissions.

**Quick reference**

- **setuid**: `chmod u+s file` or `chmod 4xxx file`
- **setgid on file**: `chmod g+s file` or `chmod 2xxx file`
- **setgid on directory**: `chmod g+s dir` or `chmod 2xxx dir`
- **sticky bit**: `chmod +t dir` or `chmod 1xxx dir`

**Practical RHCSA mindset**

On the exam, you are usually expected to:

- recognize special bits in `ls -l`
- apply the correct permission mode with `chmod`
- create a shared directory with group inheritance
- secure a shared writable directory with sticky bit
- verify the result with `ls -ld`, `stat`, or `find`

If you can explain what each bit changes and when to use it, you can solve most related tasks quickly and safely.
