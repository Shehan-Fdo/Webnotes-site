**SELinux context changes with `chcon`**

`chcon` changes the SELinux security context of a file, directory, or other labeled object. In practice, you use it when you need a quick, immediate label adjustment to test whether an SELinux denial is caused by an incorrect context.

SELinux labels usually include four fields:

- **user**: SELinux user identity
- **role**: SELinux role
- **type**: the most important field for most file access decisions
- **level**: MLS/MCS sensitivity level, less commonly changed in RHCSA scenarios

A typical file context looks like this:

```bash
$ ls -Z /var/www/html/index.html
-rw-r--r--. root root system_u:object_r:httpd_sys_content_t:s0 index.html
```

If a web server cannot read a file because the type is wrong, `chcon` can help you verify the fix quickly.

**Common use cases**

Use `chcon` when you need a temporary or diagnostic label change.

- **Troubleshooting**: test whether SELinux labeling is the cause of an access problem.
- **Short-lived changes**: adjust a label for a quick validation before making a permanent policy-based change.
- **Lab work**: experiment with SELinux contexts during practice without editing policy.

Example: changing a directory to a web content type:

```bash
sudo chcon -t httpd_sys_content_t /srv/web
sudo chcon -R -t httpd_sys_content_t /srv/web
```

After the change, verify the label:

```bash
ls -Zd /srv/web
```

If the service now works, the original issue was likely the file type label.

**Why `chcon` is limited**

`chcon` is not the preferred way to make permanent SELinux labeling changes. Its main limitation is that the change is often not persistent across relabeling or restoration operations.

- **Not policy-based**: it changes the label directly on the file, but does not update the systemâ€™s file context rules.
- **Can be overwritten**: `restorecon` may reset the label to match the configured policy.
- **May not survive relabels**: a full filesystem relabel can revert the change.
- **Easy to lose track of**: ad hoc labels can create confusion later if you forget they were manually applied.

> If you need the label to survive future relabeling, use `semanage fcontext` to define the rule and then apply it with `restorecon`.

**Persistent alternative: `semanage fcontext` + `restorecon`**

The standard workflow for permanent SELinux file labeling is:

1. Add a file context rule with `semanage fcontext`.
2. Apply the rule with `restorecon`.

Example for a custom web directory:

```bash
sudo semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
sudo restorecon -Rv /srv/web
```

This approach updates the policy mapping so the label is reapplied consistently.

**When to choose each tool**

| Tool | Best for | Persistence | Typical use |
|---|---|---:|---|
| `chcon` | Quick test or temporary fix | No | Diagnose SELinux labeling issues |
| `semanage fcontext` + `restorecon` | Permanent label management | Yes | Production-ready SELinux configuration |

**Practical troubleshooting workflow**

When a service fails and SELinux is suspected, follow a simple process:

1. Check the current label with `ls -Z` or `ls -Zd`.
2. Compare it to the expected type for the service.
3. Use `chcon` to test a temporary fix.
4. If the test works, replace the temporary change with a persistent `semanage fcontext` rule.
5. Apply the rule with `restorecon`.

Example for Apache content outside the default document root:

```bash
sudo mkdir -p /srv/web
echo "test" | sudo tee /srv/web/index.html
sudo chcon -R -t httpd_sys_content_t /srv/web
curl http://localhost/
```

If access succeeds after the `chcon`, make the change permanent:

```bash
sudo semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
sudo restorecon -Rv /srv/web
```

**Useful options and patterns**

- **`-t`**: set the SELinux type, the most common file-label adjustment.
- **`-R`**: apply changes recursively to directories and their contents.
- **`-u`**: set the SELinux user field, used less often in RHCSA file-label tasks.
- **`-r`**: set the SELinux level, usually not needed for standard administration tasks.
- **`-l`**: list valid SELinux context components for a type or object class, depending on usage and platform support.

Example of recursive type change:

```bash
sudo chcon -R -t samba_share_t /srv/samba/share
```

**Verification commands**

Always verify the result after changing labels.

```bash
ls -Z /path/to/object
matchpathcon /path/to/object
```

- `ls -Z` shows the current label on the file.
- `matchpathcon` shows the label expected from policy mappings.

If `ls -Z` and `matchpathcon` disagree, the file may have a manual label override that `restorecon` can normalize.

**Limitations to remember for the exam**

- `chcon` is useful for diagnosis, but not for durable configuration.
- A manual label change can be lost after `restorecon` or relabeling.
- The correct production fix is usually a file context rule, not a one-off label.
- SELinux denials may also be caused by booleans, permissions, or service configuration, so labeling is only one part of troubleshooting.

**Example scenario**

A custom Apache site lives in `/srv/web`, but Apache cannot read it because the directory has the wrong type.

```bash
$ ls -Zd /srv/web
drwxr-xr-x. root root unconfined_u:object_r:default_t:s0 /srv/web
```

Temporary test:

```bash
sudo chcon -R -t httpd_sys_content_t /srv/web
```

Persistent fix:

```bash
sudo semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
sudo restorecon -Rv /srv/web
```

This workflow confirms the cause quickly and then converts the temporary workaround into a policy-backed solution.

**Key takeaway**

Use `chcon` to prove a labeling problem and restore service access quickly during troubleshooting. Use `semanage fcontext` plus `restorecon` when you need the label to remain correct after future relabels.
