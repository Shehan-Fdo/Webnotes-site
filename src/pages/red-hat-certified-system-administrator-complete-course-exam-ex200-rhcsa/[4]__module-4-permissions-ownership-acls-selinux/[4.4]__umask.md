## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**What `umask` does**

`umask` is the *user file-creation mode mask*. It controls the default permissions assigned to new files and directories by subtracting permissions from the system’s base defaults.

- New files and directories do not inherit permissions directly from `umask`; they start from a default and then have bits removed.
- It affects permissions only at creation time, not existing files.
- It is a shell setting, so different sessions can have different values.
- It is important for security because it prevents newly created files from being too open.

On Linux, the usual base defaults are:

- Files: `666` (`rw-rw-rw-`)
- Directories: `777` (`rwxrwxrwx`)

The `umask` removes permissions from those defaults.

**How the calculation works**

Think of `umask` as a subtraction mask applied to the default mode.

- For files, execute permission is typically not granted by default, even if the base mode is `666`.
- For directories, execute permission matters because it allows traversal.

Example:

- `umask 022`
  - Files become `644` (`666 - 022`)
  - Directories become `755` (`777 - 022`)
- `umask 077`
  - Files become `600`
  - Directories become `700`

A practical way to remember it:

- Lower `umask` values mean more permissive defaults.
- Higher `umask` values mean more restrictive defaults.

**Check and set `umask`**

You can view the current value with:

```bash
umask
```

You can set it for the current shell session with:

```bash
umask 022
```

To verify the effect, create a file and a directory:

```bash
umask 027
touch testfile
mkdir testdir
ls -ld testfile testdir
```

Expected result:

- `testfile` is typically `rw-r-----` (`640`)
- `testdir` is typically `rwxr-x---` (`750`)

**Common `umask` values**

| `umask` | Files | Directories | Typical use |
|---|---:|---:|---|
| `022` | `644` | `755` | Common default for general multi-user systems |
| `027` | `640` | `750` | More secure default for shared systems |
| `077` | `600` | `700` | Private files and directories |
| `002` | `664` | `775` | Collaborative environments |

**Why files and directories differ**

Directories need the execute bit to be accessible. Without execute permission on a directory, you cannot `cd` into it or access files inside it, even if you can list the directory contents in some cases.

- File permissions control reading, writing, and execution of the file itself.
- Directory permissions control listing, creating, deleting, and traversing entries.

This is why the same `umask` produces different final modes for files and directories.

**Symbolic and numeric forms**

`umask` is commonly shown in numeric form, but symbolic permission logic helps you understand the result.

- Base file mode: `666`
- Base directory mode: `777`
- Mask removes bits from those bases

Example:

```bash
umask 002
```

This removes:
- no permissions from owner
- no permissions from group
- write permission from others

Result:
- files: `664`
- directories: `775`

**Special cases and practical behavior**

Some programs do not rely solely on `umask`.

- Applications may create files with explicit permissions using system calls such as `open()` or `mkdir()`.
- Some services override or manage permissions after creation.
- `umask` still matters for most shell-created files and many standard utilities.

Examples of commands affected by `umask`:

```bash
touch report.txt
mkdir project
cp source.txt copy.txt
install -m 755 script.sh /usr/local/bin/script.sh
```

Notes:

- `touch` creates a file using the current `umask`.
- `mkdir` creates a directory using the current `umask`.
- `cp` may preserve source permissions unless told otherwise.
- `install -m` sets explicit permissions and can bypass the default effect you expect from `umask`.

**Where `umask` is configured**

You can set `umask` in shell startup files so it applies automatically.

Common locations:

- `~/.bashrc` for interactive non-login shells
- `~/.bash_profile` or `~/.profile` for login shells
- `/etc/profile`
- `/etc/bashrc` or `/etc/profile.d/*.sh` for system-wide defaults

Example user-level configuration:

```bash
# ~/.bashrc
umask 027
```

Example system-wide configuration:

```bash
# /etc/profile.d/umask.sh
umask 027
```

> Be careful when changing system-wide defaults. A more restrictive `umask` can break workflows that expect group write access, while a more permissive one can expose sensitive data.

**How to inspect the current shell environment**

Use these commands to confirm what is happening in your session:

```bash
umask
echo $SHELL
ps -p $$
```

If the value is not what you expect, check which startup file your shell reads and whether another script overrides it later.

**Relationship to permissions and security**

`umask` is not a replacement for proper ownership and ACLs (Access Control Lists). It is a first-line control for default permissions.

Use it to:

- reduce accidental exposure of new files
- enforce safer defaults for shared systems
- support collaboration without making files world-writable

Use other tools when you need finer control:

- `chmod` for changing permissions after creation
- `chown` for changing ownership
- `setfacl` and `getfacl` for per-user or per-group access control

**Practical examples**

Create a private workspace:

```bash
umask 077
mkdir secrets
touch secrets/passwords.txt
ls -ld secrets secrets/passwords.txt
```

Expected behavior:

- `secrets` is `700`
- `passwords.txt` is `600`

Create a shared project directory:

```bash
umask 002
mkdir team-project
touch team-project/notes.txt
ls -ld team-project team-project/notes.txt
```

Expected behavior:

- `team-project` is `775`
- `notes.txt` is `664`

**Troubleshooting checklist**

If permissions look wrong, check these points:

1. Confirm the current `umask` with `umask`.
2. Verify whether the file was created by a shell command or by an application that sets permissions explicitly.
3. Check whether a parent directory has a restrictive mode or ACL.
4. Inspect startup files if the value changes between sessions.
5. Remember that `cp -p` and similar tools may preserve source metadata.

**Exam focus**

For RHCSA, you should be able to:

- explain what `umask` does
- calculate resulting file and directory permissions
- set `umask` for the current session
- make a persistent `umask` change in shell startup files
- recognize when `umask` is not the only factor affecting permissions

A quick mental model helps on the exam:

- `umask 022` is a common default
- `umask 077` is for private data
- directories gain execute bits by default, files usually do not

**Quick reference**

```bash
# View current umask
umask

# Set a restrictive session default
umask 077

# Set a collaborative session default
umask 002

# Verify results
touch file1
mkdir dir1
ls -ld file1 dir1
```
