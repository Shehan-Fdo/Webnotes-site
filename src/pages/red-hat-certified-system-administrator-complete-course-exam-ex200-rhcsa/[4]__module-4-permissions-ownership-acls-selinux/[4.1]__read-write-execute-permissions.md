## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Permission model overview**

Linux permissions control who can read, modify, or traverse files and directories. On RHCSA, you must understand both the symbolic view (`rwxr-x---`) and the numeric view (`750`) because exam tasks often require quick interpretation and precise changes.

Each file or directory has:
- **Owner user**
- **Group owner**
- **Other users**
- Optional **ACLs** (Access Control Lists) for finer-grained access

The basic permission bits are:
- **Read (`r`)**: view file contents or list directory contents
- **Write (`w`)**: modify file contents or create/remove entries in a directory
- **Execute (`x`)**: run a file as a program or traverse a directory

A common exam mistake is treating file and directory permissions the same way. They are related, but the meaning changes depending on object type.

**File permissions**

For regular files, permissions control access to the file contents.

- **Read** allows you to open and inspect the file.
- **Write** allows you to change the file contents.
- **Execute** allows the kernel to run the file as a program if it is a binary or a script with a valid interpreter line.

For scripts, execute permission alone is not always enough. The script file also needs read permission so the interpreter can read the script contents.

```bash
# Binary executable: execute permission is enough to run it
ls -l /usr/bin/ls

# Script example: needs execute permission to launch,
# and usually read permission so the interpreter can read the file
ls -l /usr/local/bin/backup.sh
head -n 1 /usr/local/bin/backup.sh
```

Practical implications:
- A binary can often run with `x` even if `r` is not set.
- A shell script typically needs `r` and `x` for the user running it.
- Without `x`, the file cannot be executed directly, even if it is readable.

**Directory permissions**

Directory permissions control access to the directory entries, not the file contents themselves.

- **Read** lets you list filenames in the directory.
- **Write** lets you create, delete, or rename entries in the directory.
- **Execute** lets you enter the directory and access files inside it if you know their names.

This is one of the most important RHCSA concepts:
- To **delete or rename a file in a directory**, you need **write + execute on the directory**, not on the file itself.
- To **access a file by name** inside a directory, you need **execute on the directory**.
- To **list directory contents**, you need **read + execute** on the directory.

> If you can read a directory but not execute it, you may see filenames but still be unable to access them by path.

**Symbolic and numeric modes**

Permissions are commonly represented in two forms.

- **Symbolic mode** uses letters such as `u+rwx`, `g-w`, or `o=rx`.
- **Numeric mode** uses octal values such as `755`, `640`, or `700`.

Common mappings:
- `r = 4`
- `w = 2`
- `x = 1`

Examples:
- `7 = rwx`
- `6 = rw-`
- `5 = r-x`
- `4 = r--`

```bash
# View permissions
ls -l file.txt

# Add execute permission for the owner
chmod u+x script.sh

# Remove write permission from group and others
chmod go-w report.txt

# Set exact permissions
chmod 640 secrets.txt

# Set directory permissions
chmod 750 /srv/projects
```

**Ownership and access checks**

Linux checks access in this order:
1. If you are the **owner**, owner permissions apply.
2. Otherwise, if you are in the file’s **group**, group permissions apply.
3. Otherwise, **other** permissions apply.

This matters when troubleshooting because a user may be denied even though the file appears to have the right mode bits. The effective access depends on ownership, group membership, and any ACLs.

Use these commands to inspect ownership and permissions:
```bash
ls -l /path/to/file
stat /path/to/file
id username
```

Example output interpretation:
- `-rwxr----- 1 alice dev 1234 Apr 26 10:00 app.sh`
  - Owner `alice` has `rwx`
  - Group `dev` has `r--`
  - Others have no access

**Root, sudo, and permission checks**

Permission checks behave differently for privileged users.

- **Root** can generally bypass standard file permission checks.
- **sudo** runs a command with elevated privileges, so the command may succeed even when the invoking user would normally be denied.
- Non-root users are constrained by standard Unix permission bits and ACLs.

This distinction is important during troubleshooting:
- A command that works with `sudo` may fail without it because the user lacks the required permissions.
- If a task says “as a regular user,” do not assume root-like access.
- Some operations still depend on directory permissions even when the file itself is readable.

Example:
```bash
# Fails for a non-root user without permission
cat /root/secret.txt

# Succeeds with elevated privileges
sudo cat /root/secret.txt
```

**Common permission scenarios**

A few patterns appear often in RHCSA tasks.

- **Read a file**: need `r` on the file.
- **Edit a file**: need `w` on the file, and often `r` as well for the editor to load it.
- **Run a script**: need `x` on the script; usually `r` too.
- **Enter a directory**: need `x` on the directory.
- **List a directory**: need `r` and `x` on the directory.
- **Create a file in a directory**: need `w` and `x` on the directory.
- **Delete or rename a file in a directory**: need `w` and `x` on the directory.

A useful mental model:
- File permissions control the file.
- Directory permissions control the directory entries.

**Practical command examples**

Use `chmod` to change permissions, `chown` to change ownership, and `chgrp` to change group ownership.

```bash
# Make a script executable by the owner
chmod u+x deploy.sh

# Give owner full access, group read-only, others no access
chmod 740 confidential.txt

# Allow everyone to read and execute a directory, but only owner to write
chmod 755 /usr/local/bin

# Change owner and group
chown alice:dev /srv/app/config.ini

# Change only the group
chgrp wheel /srv/app/config.ini
```

You can also use symbolic expressions to make targeted changes:
```bash
# Add read permission for group
chmod g+r notes.txt

# Remove execute from others
chmod o-x public-script.sh

# Set exact owner/group/other permissions
chmod u=rw,g=r,o= /etc/myapp.conf
```

**Special bits to recognize**

RHCSA permission questions may also involve special mode bits.

- **Setuid (`s`)**: executable runs with the file owner’s privileges
- **Setgid (`s`)**: executable runs with the file group’s privileges; on directories, new files inherit the directory group
- **Sticky bit (`t`)**: on directories, only file owners or root can delete their own files

Examples:
```bash
chmod u+s /usr/local/bin/tool
chmod g+s /srv/shared
chmod +t /tmp/shared
```

These are not the same as basic `rwx`, but they often appear in permission troubleshooting.

**ACLs and extended access**

Access Control Lists provide more granular permissions than the traditional owner/group/other model. Use ACLs when you need to grant access to specific users or groups without changing the main mode bits.

View ACLs with:
```bash
getfacl /path/to/file
```

Set ACLs with:
```bash
setfacl -m u:bob:rw file.txt
setfacl -m g:qa:rX project/
setfacl -x u:bob file.txt
```

Important ACL concepts:
- The **ACL mask** limits the effective permissions for named users, named groups, and the owning group.
- If permissions appear unexpectedly limited, check the mask.
- The mask can make an ACL entry look permissive while the effective access is reduced.

Example:
```bash
setfacl -m u:bob:rw report.txt
setfacl -m m:r report.txt
getfacl report.txt
```

If `bob` is granted `rw` but the mask is `r--`, the effective permission becomes read-only. This is a common troubleshooting point.

Default ACLs apply to new files and directories created inside a directory.

```bash
# Set a default ACL so new files inherit access
setfacl -m d:g:dev:rwX /srv/project
getfacl /srv/project
```

Notes on default ACL behavior:
- Default ACLs affect newly created items, not existing ones.
- The inherited permissions are still filtered by the process `umask`.
- The ACL mask may also reduce effective access, so verify both the default entry and the mask.

**Troubleshooting workflow**

When a permission issue appears, check the problem systematically.

1. Identify the object type: file or directory.
2. Check ownership and mode bits with `ls -l` or `stat`.
3. Check whether the user is the owner or in the correct group with `id`.
4. Inspect ACLs with `getfacl`.
5. Confirm whether the user is root or using `sudo`.
6. Verify directory traversal permissions on every parent directory in the path.

Useful commands:
```bash
namei -l /srv/app/data/report.txt
ls -ld /srv /srv/app /srv/app/data
getfacl /srv/app/data/report.txt
```

`namei -l` is especially helpful because it shows permissions on each path component. A user may have access to the target file but still fail because one parent directory lacks execute permission.

**Real-world examples**

Example 1: Allow a team to edit files in a shared directory.
```bash
chown alice:dev /srv/team
chmod 2770 /srv/team
setfacl -m g:dev:rwx /srv/team
setfacl -m d:g:dev:rwx /srv/team
```
- `2` sets the setgid bit so new files inherit the group.
- `770` restricts access to owner and group.
- The default ACL ensures new files remain group-writable.

Example 2: Make a script runnable by users but not editable by others.
```bash
chown root:root /usr/local/bin/cleanup.sh
chmod 755 /usr/local/bin/cleanup.sh
```
- Users can execute the script.
- Only root can modify it.

Example 3: Restrict a private directory.
```bash
chmod 700 /home/alice/private
```
- Only the owner can read, write, and enter the directory.
- Other users cannot list or traverse it.

**Exam tips**

- Remember that **directory write** controls create, delete, and rename operations.
- Remember that **directory execute** controls traversal and access by path.
- Check ACL **mask** values when effective permissions do not match the entry you set.
- Use `sudo` only when the task requires elevated privileges.
- Verify every directory in the path, not just the final file.
- For scripts, distinguish between **execute permission** and the interpreter’s need to **read** the script contents.

**Quick reference**

- `r` on file: read contents
- `w` on file: modify contents
- `x` on file: execute program
- `r` on directory: list names
- `w` on directory: create/delete/rename entries
- `x` on directory: traverse/access entries
