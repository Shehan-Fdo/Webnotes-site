## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

On RHEL, you usually control network access with **firewalld**, which manages firewall rules dynamically and persists them when needed. For RHCSA, you should know how to open and close ports safely, verify the active firewall state, and understand the difference between allowing a service and allowing a raw port.

A port is a numbered endpoint used by a network service. Opening a port means permitting traffic to reach that endpoint through the firewall. Closing a port blocks that traffic. In practice, you should prefer service-based rules when a predefined service exists, because they are easier to read and less error-prone than manually specifying port numbers.

**Core firewall concepts**

`firewalld` uses **zones** to apply different trust levels to network interfaces and sources. A zone can allow a set of services, ports, and protocols. The default zone is often `public`, but you should always verify the active configuration before making changes.

- **Runtime configuration** affects the running firewall immediately.
- **Permanent configuration** survives reboot after you reload or restart the firewall.
- **Services** are named presets such as `ssh`, `http`, or `https`.
- **Ports** are explicit numbers such as `8080/tcp` or `53/udp`.
- **Protocols** can also be allowed directly when needed.

A common exam mistake is changing only the runtime firewall and forgetting to make the rule permanent. Another is opening the wrong protocol, such as `tcp` when the application uses `udp`.

**Inspecting the current firewall state**

Before changing anything, check which zone is active and what it currently allows.

```bash
sudo firewall-cmd --state
sudo firewall-cmd --get-default-zone
sudo firewall-cmd --get-active-zones
sudo firewall-cmd --list-all
sudo firewall-cmd --list-all --zone=public
```

Useful checks include:

- `--state` confirms whether `firewalld` is running.
- `--get-active-zones` shows which interfaces belong to which zones.
- `--list-all` displays services, ports, and rich rules for a zone.
- `--list-ports` shows only raw port openings.
- `--list-services` shows allowed service names.

If you need to inspect the permanent configuration, add `--permanent` to the query.

```bash
sudo firewall-cmd --permanent --list-all --zone=public
```

**Opening a port**

To open a port, add a rule for the correct zone. Use `--add-port` with the port number and protocol.

```bash
sudo firewall-cmd --zone=public --add-port=8080/tcp
```

This changes the runtime configuration only. To make it persistent:

```bash
sudo firewall-cmd --permanent --zone=public --add-port=8080/tcp
sudo firewall-cmd --reload
```

Key points:

- Always specify the protocol, usually `tcp` or `udp`.
- Use the correct zone for the interface or source.
- Reload after permanent changes so the runtime and permanent rules match.

If you need to open a range of ports, use a hyphen.

```bash
sudo firewall-cmd --permanent --zone=public --add-port=6000-6010/tcp
sudo firewall-cmd --reload
```

**Closing a port**

To close a port, remove the rule from the zone.

```bash
sudo firewall-cmd --zone=public --remove-port=8080/tcp
```

For a persistent removal:

```bash
sudo firewall-cmd --permanent --zone=public --remove-port=8080/tcp
sudo firewall-cmd --reload
```

If a port remains open after removal, check whether:

- the rule was removed from the wrong zone
- the change was made only at runtime or only permanently
- a service rule still allows the same traffic
- a rich rule or direct rule is also permitting access

**Opening and closing services**

When a service definition exists, prefer it over raw ports. For example, `http` usually maps to TCP port 80, and `https` to TCP port 443.

```bash
sudo firewall-cmd --zone=public --add-service=http
sudo firewall-cmd --permanent --zone=public --add-service=http
sudo firewall-cmd --reload
```

To close it:

```bash
sudo firewall-cmd --zone=public --remove-service=http
sudo firewall-cmd --permanent --zone=public --remove-service=http
sudo firewall-cmd --reload
```

Service-based rules are useful because they are easier to understand and may include multiple ports or protocols if the service requires them.

**Runtime versus permanent changes**

`firewall-cmd` defaults to runtime changes unless you add `--permanent`. This distinction matters during troubleshooting and on the exam.

| Change type | Command behavior | Survives reboot |
|---|---|---|
| Runtime | Immediate effect | No |
| Permanent | Stored in config | Yes, after reload |

A safe workflow is:

1. Make the change at runtime.
2. Test the application.
3. Add the same change permanently.
4. Reload and verify again.

This approach reduces the chance of locking yourself out or leaving an incorrect rule in place.

**Verifying your changes**

After opening or closing a port, confirm the result.

```bash
sudo firewall-cmd --zone=public --list-ports
sudo firewall-cmd --zone=public --list-services
sudo firewall-cmd --zone=public --query-port=8080/tcp
sudo firewall-cmd --zone=public --query-service=http
```

`--query-port` and `--query-service` return whether the rule exists. This is especially useful in scripts and during exam validation.

You can also test connectivity from another host with tools such as `nc`, `curl`, or `telnet` if installed.

```bash
nc -vz server.example.com 8080
curl -I http://server.example.com
```

If the firewall is correct but the connection still fails, the service may not be listening on the expected port.

```bash
sudo ss -tuln
sudo ss -tulpn
```

`ss` helps you confirm whether the application is actually bound to the port you opened.

**Zones and interfaces**

A port rule only applies within the zone that protects the relevant interface or source. If you open a port in `public` but the interface is in `internal`, the rule may not affect traffic.

```bash
sudo firewall-cmd --get-active-zones
sudo firewall-cmd --zone=public --list-all
sudo firewall-cmd --zone=internal --list-all
```

If needed, assign an interface to a zone:

```bash
sudo firewall-cmd --permanent --zone=public --add-interface=ens160
sudo firewall-cmd --reload
```

Be careful when changing zone assignments on a remote system. A wrong zone can block access unexpectedly.

> <mark>Critical</mark>: If you are connected over SSH, confirm that the zone for your management interface still allows `ssh` before reloading firewall changes.

**Rich rules**

Rich rules provide more precise control than simple port openings. Use them when you need to restrict by source address, destination, protocol, or logging behavior.

Example: allow SSH only from a specific subnet.

```bash
sudo firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" source address="192.0.2.0/24" service name="ssh" accept'
sudo firewall-cmd --reload
```

Example: allow a port only from one host.

```bash
sudo firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" source address="192.0.2.50" port port="8080" protocol="tcp" accept'
sudo firewall-cmd --reload
```

Rich rules are powerful, but for RHCSA you should first be comfortable with basic service and port rules.

**Common practical scenarios**

- **Web server**
  - Open `http` and `https` services.
  - Verify the web server is listening on `80/tcp` and `443/tcp`.
- **Custom application**
  - Open the exact port the app uses, such as `8080/tcp`.
  - Confirm the app binds to that port with `ss`.
- **Temporary troubleshooting**
  - Add a runtime rule first.
  - Test connectivity.
  - Make it permanent only after validation.
- **Removing access**
  - Remove the service or port from the correct zone.
  - Reload if the change is permanent.

**Troubleshooting checklist**

If a port appears closed or traffic still fails, check the following in order:

1. Confirm `firewalld` is running.
2. Confirm the correct zone is active for the interface.
3. Confirm the rule exists in the same zone.
4. Confirm the rule is runtime, permanent, or both as intended.
5. Confirm the application is listening on the expected port and protocol.
6. Confirm no other firewall layer or network device is blocking traffic.

Useful commands:

```bash
sudo systemctl status firewalld
sudo firewall-cmd --get-active-zones
sudo firewall-cmd --zone=public --list-all
sudo firewall-cmd --zone=public --query-port=8080/tcp
sudo ss -tuln
```

**Exam tips**

- Prefer `firewall-cmd` over editing configuration files directly.
- Use `--permanent` for persistent changes and `--reload` to apply them.
- Match the protocol exactly.
- Verify the active zone before changing rules.
- Use service names when available; use ports when the service is custom.
- Remember that closing a port means removing the allow rule, not adding a deny rule in most RHCSA tasks.

**Quick reference**

```bash
# Check firewall status
sudo firewall-cmd --state

# List active zones
sudo firewall-cmd --get-active-zones

# Open a port temporarily
sudo firewall-cmd --zone=public --add-port=8080/tcp

# Open a port permanently
sudo firewall-cmd --permanent --zone=public --add-port=8080/tcp
sudo firewall-cmd --reload

# Close a port temporarily
sudo firewall-cmd --zone=public --remove-port=8080/tcp

# Close a port permanently
sudo firewall-cmd --permanent --zone=public --remove-port=8080/tcp
sudo firewall-cmd --reload

# Open a service
sudo firewall-cmd --permanent --zone=public --add-service=http
sudo firewall-cmd --reload

# Verify a rule
sudo firewall-cmd --zone=public --query-port=8080/tcp
```
