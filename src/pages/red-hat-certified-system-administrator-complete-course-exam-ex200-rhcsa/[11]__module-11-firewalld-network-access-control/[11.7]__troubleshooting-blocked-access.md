## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

Blocked access usually comes from one of four layers: the service is not running, the service is not listening on the expected address or port, the firewall is blocking traffic, or SELinux is denying access after the connection reaches the host. On RHCSA, you need to identify which layer is failing before you change anything.

A good troubleshooting mindset is to work from the outside in:

- Confirm the client can reach the host.
- Confirm the service is listening on the expected port and address.
- Confirm the firewall allows the traffic in the correct zone.
- Confirm SELinux labels and booleans match the service’s access pattern.
- Confirm the application itself is configured to allow the request.

**Common symptoms and what they usually mean**

Not every “can’t connect” problem is a firewall problem. Different failures point to different layers:

- **Connection refused** often means nothing is listening on that port, or the service is bound only to localhost.
- **Connection timed out** often means a firewall, routing, or network path issue.
- **Permission denied** or application-specific denial often points to SELinux, file permissions, or service ACLs.
- **Works on IPv4 but not IPv6** often means dual-stack binding or firewall rules differ between stacks.

> A timeout is not the same as an SELinux denial. SELinux usually blocks access after the connection reaches the service or when the service tries to read or write files, so you often see application errors, AVC denials, or connection failures rather than a generic network timeout.

**Step-by-step troubleshooting workflow**

Use a repeatable sequence so you do not guess at the fix.

1. Verify the service is active.
   ```bash
   systemctl status httpd
   systemctl status sshd
   systemctl status nfs-server
   systemctl status smb
   ```
2. Verify the service is listening.
   ```bash
   ss -tulpn
   ss -tulpn | grep -E ':(22|80|443|2049|445)\b'
   ```
3. Verify the host address and route.
   ```bash
   ip addr show
   ip route
   ping -c 4 192.0.2.10
   traceroute 192.0.2.10
   ```
4. Verify the firewall zone on the active interface.
   ```bash
   firewall-cmd --get-active-zones
   firewall-cmd --zone=public --list-all
   firewall-cmd --get-default-zone
   ```
5. Verify SELinux status and denials.
   ```bash
   getenforce
   sestatus
   ausearch -m AVC,USER_AVC -ts recent
   journalctl -t setroubleshoot --since "10 min ago"
   ```
6. Test from the client with the right protocol.
   ```bash
   curl -I http://server.example.com
   nc -vz server.example.com 22
   ssh -v user@server.example.com
   ```

**Service state and binding**

A service can be running but still unreachable if it listens only on `127.0.0.1` or `::1`. That is a binding problem, not a firewall problem.

Check the listening address with `ss`:

```bash
ss -tulpn | grep httpd
ss -tulpn | grep sshd
```

Typical examples:

- `127.0.0.1:80` means only local connections work.
- `0.0.0.0:80` means IPv4 on all interfaces.
- `[::]:80` means IPv6 on all interfaces, and often IPv4 too if dual-stack is enabled.
- A service bound to one interface may be unreachable from another network even if the firewall is open.

For services that use configuration files, verify the listen directives:

```bash
grep -E '^(Listen|Address|Port)' /etc/httpd/conf/httpd.conf /etc/ssh/sshd_config
```

For SSH, common settings include:

```bash
Port 22
ListenAddress 0.0.0.0
ListenAddress ::
```

For Apache, the listen directive may look like:

```apache
Listen 80
```

If you change a listen setting, restart the service:

```bash
systemctl restart httpd
systemctl restart sshd
```

**Firewall troubleshooting with firewalld**

On current RHEL systems, `firewalld` is the standard host firewall. The most common exam mistake is opening a service in the wrong zone.

First identify the active interface and its zone:

```bash
firewall-cmd --get-active-zones
nmcli device status
ip addr show
```

Then confirm the service is allowed in that zone:

```bash
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-ports
```

If the active interface is in a different zone, change that zone instead of the default zone:

```bash
firewall-cmd --zone=internal --add-service=http
firewall-cmd --zone=internal --add-service=http --permanent
firewall-cmd --reload
```

Useful checks:

- **Runtime vs permanent**: a change without `--permanent` disappears after reload or reboot.
- **Service vs port**: use the predefined service when possible, such as `ssh`, `http`, `https`, `nfs`, or `samba`.
- **Custom ports**: use `--add-port=8080/tcp` when the service uses a nonstandard port.
- **Rich rules**: use them when you need source-based or more specific access control.

Example custom port rule:

```bash
firewall-cmd --zone=public --add-port=8080/tcp --permanent
firewall-cmd --reload
```

Example source-based access:

```bash
firewall-cmd --zone=internal --add-source=192.0.2.0/24 --add-service=ssh --permanent
firewall-cmd --reload
```

If you suspect the wrong zone is attached to the interface, inspect NetworkManager and firewalld together:

```bash
nmcli connection show
nmcli connection show "System eth0" | grep -i zone
firewall-cmd --get-active-zones
```

**IPv6 and dual-stack issues**

A service may appear open on IPv4 but fail on IPv6, or the reverse. This often happens when:

- The service listens only on one address family.
- The firewall allows the port in one family but not the other.
- The client resolves the hostname to the other stack first.

Check both stacks:

```bash
ss -tulpn -4
ss -tulpn -6
getent hosts server.example.com
```

Test explicitly:

```bash
curl -4 -I http://server.example.com
curl -6 -I http://server.example.com
nc -4 -vz server.example.com 80
nc -6 -vz server.example.com 80
```

If IPv6 is not intended, do not disable it casually during exam work unless the task allows it. Instead, verify the service binds correctly and the firewall zone permits the traffic.

**SELinux troubleshooting workflow**

SELinux is a mandatory access control system. It does not replace the firewall; it controls what a process can do after it is already running and, in many cases, after the network connection is established.

A practical workflow is:

1. Confirm SELinux is enforcing or permissive.
   ```bash
   getenforce
   sestatus
   ```
2. Reproduce the failure.
3. Search for recent AVC denials.
   ```bash
   ausearch -m AVC,USER_AVC -ts recent
   journalctl -t setroubleshoot --since "15 min ago"
   ```
4. Identify the denied process, target file, and action.
5. Fix the label, boolean, or policy setting.
6. Retest.

Common SELinux tools:

```bash
ls -Z /var/www/html
ps -eZ | grep httpd
matchpathcon /var/www/html/index.html
semanage fcontext -l | grep httpd
```

If a service cannot read or write a file, the file context is often the issue. For example, Apache content should usually be labeled for web access:

```bash
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
restorecon -Rv /srv/web
```

For writable web content, use a writable type when appropriate:

```bash
semanage fcontext -a -t httpd_sys_rw_content_t '/srv/web/uploads(/.*)?'
restorecon -Rv /srv/web/uploads
```

For Samba shares, the file context matters more than a blanket boolean in many exam scenarios:

```bash
semanage fcontext -a -t samba_share_t '/srv/samba/share(/.*)?'
restorecon -Rv /srv/samba/share
```

For NFS exports, use the correct context for the intended access pattern:

```bash
semanage fcontext -a -t public_content_t '/srv/nfs/share(/.*)?'
restorecon -Rv /srv/nfs/share
```

If the export must be writable by NFS clients and the scenario calls for SELinux labeling, use the writable type:

```bash
semanage fcontext -a -t public_content_rw_t '/srv/nfs/share(/.*)?'
restorecon -Rv /srv/nfs/share
```

> Do not assume a boolean is the fix for every SELinux problem. Many access issues are solved by the correct file context, not by turning on broad policy exceptions.

**Useful SELinux booleans**

Booleans are runtime policy switches. Use them when the service needs a supported exception and the exam scenario clearly points to that need.

Common examples:

```bash
getsebool -a | grep httpd
getsebool -a | grep samba
getsebool -a | grep nfs
```

Examples of when booleans may help:

- Allowing Apache to make outbound network connections:
  ```bash
  setsebool -P httpd_can_network_connect on
  ```
- Allowing Apache to serve content from home directories:
  ```bash
  setsebool -P httpd_enable_homedirs on
  ```
- Allowing Samba to export home directories or broader content in specific scenarios:
  ```bash
  setsebool -P samba_enable_home_dirs on
  ```

Use booleans only when they match the problem. If the file label is wrong, fix the label first.

**Practical blocked-access scenarios**

**SSH blocked**

If SSH fails, check the service, port, and firewall first:

```bash
systemctl status sshd
ss -tulpn | grep :22
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-services
```

Typical fixes:

- Start and enable the service:
  ```bash
  systemctl enable --now sshd
  ```
- Open the SSH service in the correct zone:
  ```bash
  firewall-cmd --zone=public --add-service=ssh --permanent
  firewall-cmd --reload
  ```
- Confirm `sshd_config` is not bound only to localhost.

If SSH is configured on a nonstandard port, open that port and update the client:

```bash
firewall-cmd --zone=public --add-port=2222/tcp --permanent
firewall-cmd --reload
ssh -p 2222 user@server.example.com
```

**HTTP blocked**

For Apache, verify the daemon, listen port, firewall, and SELinux labels:

```bash
systemctl status httpd
ss -tulpn | grep :80
firewall-cmd --zone=public --add-service=http --permanent
firewall-cmd --reload
ls -Z /var/www/html
```

If content is stored outside the default document root, label it correctly:

```bash
semanage fcontext -a -t httpd_sys_content_t '/srv/www(/.*)?'
restorecon -Rv /srv/www
```

If the site needs write access, use the writable type for the writable path only.

**NFS blocked**

For NFS, the server must export the directory, the firewall must allow NFS-related services, and SELinux must permit the export path.

Check:

```bash
systemctl status nfs-server
exportfs -v
ss -tulpn | grep 2049
firewall-cmd --zone=public --add-service=nfs --add-service=mountd --add-service=rpc-bind --permanent
firewall-cmd --reload
```

If the export path is mislabeled, fix it:

```bash
semanage fcontext -a -t public_content_t '/srv/nfs/share(/.*)?'
restorecon -Rv /srv/nfs/share
```

For writable exports, use the writable label when required by the scenario:

```bash
semanage fcontext -a -t public_content_rw_t '/srv/nfs/share(/.*)?'
restorecon -Rv /srv/nfs/share
```

**Samba blocked**

For Samba, verify the daemon, firewall, and file context:

```bash
systemctl status smb nmb
ss -tulpn | grep -E ':(445|139)\b'
firewall-cmd --zone=public --add-service=samba --permanent
firewall-cmd --reload
ls -Z /srv/samba/share
```

Label the share correctly:

```bash
semanage fcontext -a -t samba_share_t '/srv/samba/share(/.*)?'
restorecon -Rv /srv/samba/share
```

If users can connect but cannot read or write files, check both SELinux and Unix permissions:

```bash
ls -ldZ /srv/samba/share
getfacl /srv/samba/share
```

**Legacy host-based access controls**

Older systems may use TCP wrappers, which control access through `/etc/hosts.allow` and `/etc/hosts.deny`. This is mostly a legacy concept, but you should recognize it if a service behaves as if the network is open yet access is still denied.

Example:

```bash
cat /etc/hosts.allow
cat /etc/hosts.deny
```

Typical pattern:

```text
sshd: 192.0.2.0/255.255.255.0
ALL: ALL
```

If TCP wrappers are in use, the allow file is checked before the deny file. Many modern RHEL services no longer rely on TCP wrappers, so treat this as a legacy troubleshooting clue rather than a default fix.

**Practical command set for exam use**

Use these commands often:

```bash
systemctl status <service>
systemctl enable --now <service>
ss -tulpn
firewall-cmd --get-active-zones
firewall-cmd --zone=<zone> --list-all
firewall-cmd --zone=<zone> --add-service=<service> --permanent
firewall-cmd --reload
getenforce
ausearch -m AVC,USER_AVC -ts recent
restorecon -Rv <path>
semanage fcontext -a -t <type> '<path>(/.*)?'
```

**Decision guide**

Use this quick mapping to narrow the cause:

- **Service down**: start and enable the daemon.
- **Wrong bind address**: change `ListenAddress`, `Listen`, or equivalent.
- **Firewall zone mismatch**: open the service in the zone attached to the active interface.
- **SELinux denial**: inspect AVC logs, then fix the file context or boolean.
- **Wrong file permissions**: adjust ownership, mode, or ACLs.
- **IPv4/IPv6 mismatch**: test both stacks and confirm dual-stack listening.

**Troubleshooting mindset**

The fastest path is to prove each layer in order instead of changing multiple settings at once. If you open the firewall before checking the service bind address, you may hide the real problem. If you disable SELinux instead of fixing the label, you may pass a test but miss the correct exam solution.

For RHCSA, the expected answer is usually the smallest correct change:

- Open the right service in the right firewalld zone.
- Label the content correctly with `semanage fcontext` and `restorecon`.
- Enable only the boolean that matches the scenario.
- Bind the service to the correct address family and interface.
