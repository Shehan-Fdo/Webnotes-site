## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Firewall state management overview**

`firewalld` is the dynamic firewall service used on modern Red Hat Enterprise Linux systems. Instead of editing low-level packet filter rules directly, you manage zones, services, ports, and rich rules through `firewall-cmd` and then verify that the running state matches what you intended.

For RHCSA-level work, you need to know two related tasks:

- **Reloading** the firewall so configuration changes are applied cleanly.
- **Verifying** the active firewall state so you can confirm the system is actually enforcing the expected policy.

A common exam mistake is assuming that a configuration change is active just because it was saved. In `firewalld`, runtime and permanent configuration are related but not identical, so you must check both when troubleshooting.

**Core concepts**

`firewalld` uses zones to group trust levels and apply rules to interfaces or sources. Services are predefined rule sets for common applications such as SSH, HTTP, or HTTPS. Ports and rich rules provide more specific control when a service definition is not enough.

Key terms:

- **Runtime configuration**: The currently active firewall state in memory.
- **Permanent configuration**: The saved configuration that loads after reboot or reload.
- **Reload**: Re-reads permanent configuration and applies it to the running firewall.
- **Zone**: A named policy set that controls traffic handling.
- **Service**: A predefined firewall rule bundle for an application.

A reload is useful when you want the running firewall to reflect saved changes without restarting the service. It is also safer than stopping the firewall, because it preserves firewall protection while applying updates.

**Checking service status before and after changes**

Before you troubleshoot rules, confirm that `firewalld` is running and enabled.

```bash
systemctl status firewalld
systemctl is-active firewalld
systemctl is-enabled firewalld
```

Typical outcomes:

- `active` means the firewall daemon is running.
- `enabled` means it starts automatically at boot.
- `inactive` or `disabled` usually means your rules are not being enforced as expected.

If the service is not running, start and enable it:

```bash
sudo systemctl enable --now firewalld
```

If you change firewall settings and want to ensure the daemon is still healthy, you can also check its logs:

```bash
journalctl -u firewalld --since "10 minutes ago"
```

**Reloading firewall configuration**

Use `firewall-cmd --reload` to apply permanent configuration to the runtime firewall.

```bash
sudo firewall-cmd --reload
```

This is the standard action after you make permanent changes such as:

- Adding a service to a zone with `--permanent`
- Opening a port permanently
- Adding a rich rule permanently
- Changing zone bindings for interfaces or sources

Example workflow:

```bash
sudo firewall-cmd --permanent --zone=public --add-service=http
sudo firewall-cmd --reload
```

Why this matters:

- The first command saves the change.
- The second command makes the running firewall use the saved configuration.
- Without the reload, the change may not affect current traffic.

> `--reload` can interrupt some transient runtime-only settings because it replaces the active runtime state with the permanent configuration.

If you only want to apply a change immediately for testing, you can omit `--permanent` and modify runtime state directly:

```bash
sudo firewall-cmd --zone=public --add-port=8080/tcp
```

That change is active right away, but it disappears after reload or reboot unless you also save it permanently.

**Verifying active firewall state**

Verification is as important as configuration. You should confirm the zone, services, ports, and interface bindings that are actually active.

Common verification commands:

```bash
sudo firewall-cmd --state
sudo firewall-cmd --get-default-zone
sudo firewall-cmd --get-active-zones
sudo firewall-cmd --list-all
sudo firewall-cmd --list-all-zones
```

What each command tells you:

- `--state` confirms whether `firewalld` is running.
- `--get-default-zone` shows the default zone used when no other zone applies.
- `--get-active-zones` shows which zones are currently bound to interfaces or sources.
- `--list-all` shows the rules for the default or specified zone.
- `--list-all-zones` gives a full view of every zone and its settings.

Example output interpretation:

- If `eth0` appears under `public`, traffic on that interface uses the `public` zone.
- If `ssh` is listed under services, SSH is allowed in that zone.
- If `8080/tcp` appears under ports, that port is open in that zone.

To inspect a specific zone:

```bash
sudo firewall-cmd --zone=public --list-all
```

To inspect permanent settings rather than runtime settings:

```bash
sudo firewall-cmd --permanent --zone=public --list-all
```

This distinction is critical when a rule seems to “disappear” after reload. In that case, you likely changed runtime state only.

**Runtime versus permanent state**

The most important troubleshooting skill is knowing whether a rule exists only in memory or is saved on disk.

| State | Applies immediately | Survives reload | Survives reboot |
|---|---:|---:|---:|
| Runtime | Yes | No | No |
| Permanent | No, until reload or restart | Yes | Yes |

Typical patterns:

- Use runtime changes for quick testing.
- Use permanent changes for real configuration.
- Reload after permanent changes to make them active now.

Example:

```bash
sudo firewall-cmd --add-service=https
sudo firewall-cmd --list-services
sudo firewall-cmd --reload
sudo firewall-cmd --list-services
```

If you did not use `--permanent`, the service may vanish after reload. That behavior is expected.

**Practical verification workflow**

When you add or modify firewall access, verify in a consistent order:

1. Confirm the firewall daemon is running.
2. Confirm the correct zone is active on the target interface.
3. Confirm the rule exists in runtime state.
4. Confirm the rule exists in permanent state if you need persistence.
5. Test connectivity from the client side.

Example:

```bash
systemctl is-active firewalld
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-ports
firewall-cmd --permanent --zone=public --list-services
```

This sequence helps you isolate whether the issue is:

- The firewall service is down.
- The interface is in the wrong zone.
- The rule was added to the wrong zone.
- The rule was added only at runtime.
- The client is testing the wrong protocol or port.

**Common configuration examples**

Allow SSH permanently in the `public` zone:

```bash
sudo firewall-cmd --permanent --zone=public --add-service=ssh
sudo firewall-cmd --reload
sudo firewall-cmd --zone=public --list-services
```

Open a custom TCP port permanently:

```bash
sudo firewall-cmd --permanent --zone=public --add-port=8443/tcp
sudo firewall-cmd --reload
sudo firewall-cmd --zone=public --list-ports
```

Allow HTTP and HTTPS together:

```bash
sudo firewall-cmd --permanent --zone=public --add-service=http
sudo firewall-cmd --permanent --zone=public --add-service=https
sudo firewall-cmd --reload
sudo firewall-cmd --zone=public --list-services
```

Remove a rule and verify it is gone:

```bash
sudo firewall-cmd --permanent --zone=public --remove-service=http
sudo firewall-cmd --reload
sudo firewall-cmd --zone=public --list-services
```

**Troubleshooting firewall state**

If a service is still unreachable after reloading, check these likely causes:

- The service is allowed in the wrong zone.
- The interface is attached to a different zone than expected.
- The port number or protocol is incorrect.
- The application is not listening on the expected address.
- Another firewall layer or network device is blocking traffic.

Useful commands for troubleshooting:

```bash
ip addr show
ss -tuln
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-all
```

`ss -tuln` shows listening TCP and UDP sockets. This helps you confirm whether the application is actually accepting connections on the port you opened.

Example: if you opened `8443/tcp` but `ss` shows the service listening only on `127.0.0.1:8443`, remote clients still cannot connect. The firewall is not the only factor.

**Interface and zone verification**

A firewall rule is only effective in the zone that applies to the traffic. That is why interface binding matters.

```bash
firewall-cmd --get-active-zones
```

Example output conceptually looks like this:

```text
public
  interfaces: eth0
internal
  interfaces: eth1
```

This means:

- Traffic arriving on `eth0` uses `public`.
- Traffic arriving on `eth1` uses `internal`.

If you need to move an interface to another zone, do it permanently and then reload:

```bash
sudo firewall-cmd --permanent --zone=internal --change-interface=eth1
sudo firewall-cmd --reload
sudo firewall-cmd --get-active-zones
```

**Useful verification commands at a glance**

- `firewall-cmd --state`
- `firewall-cmd --get-default-zone`
- `firewall-cmd --get-active-zones`
- `firewall-cmd --list-all`
- `firewall-cmd --permanent --list-all`
- `firewall-cmd --list-services`
- `firewall-cmd --list-ports`
- `firewall-cmd --query-service=http`
- `firewall-cmd --query-port=8443/tcp`

The `--query-*` options are helpful when you want a yes/no answer instead of a full listing.

Example:

```bash
sudo firewall-cmd --zone=public --query-service=ssh
sudo firewall-cmd --zone=public --query-port=8443/tcp
```

**Exam-focused habits**

For RHCSA tasks, work methodically:

- Make the change in the correct zone.
- Decide whether it must be runtime, permanent, or both.
- Reload when you need permanent changes to take effect immediately.
- Verify the active state after every change.
- Test from the client side when possible.

A reliable pattern is:

```bash
sudo firewall-cmd --permanent --zone=public --add-service=http
sudo firewall-cmd --reload
sudo firewall-cmd --zone=public --list-all
```

If the result is not what you expect, do not guess. Check the active zone, the permanent zone, and the listening service.

**Summary**

Reloading applies saved firewall configuration to the running system, and verification confirms that the intended rules are actually active. On RHCSA systems, you should always distinguish between runtime and permanent state, confirm the active zone, and validate both firewall settings and application listeners. That workflow prevents most firewall-related access problems and is essential for exam success.
