## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Service enablement on RHEL**

On Red Hat Enterprise Linux, “enabling a service” usually means making it start automatically at boot and ensuring it is allowed through the host firewall. For exam tasks, you often need to combine three layers:

- **systemd** to manage the service process
- **firewalld** to permit network access
- **SELinux** to allow the service to serve content or accept connections correctly

SSH and HTTP are common examples because they represent two different service patterns:

- **SSH** is a remote administration service that is usually already installed and enabled.
- **HTTP** is a web service that may need package installation, service activation, firewall rules, and content placement.

Understanding the full path from package to reachable service is more important than memorizing one command.

**systemd basics for services**

`systemd` is the service manager on modern RHEL systems. It controls daemons, boot targets, sockets, and dependencies. For RHCSA-level work, you should know how to inspect, start, stop, enable, and verify services.

Common commands:

```bash
systemctl status sshd
systemctl start sshd
systemctl stop sshd
systemctl restart sshd
systemctl enable sshd
systemctl disable sshd
systemctl is-enabled sshd
systemctl is-active sshd
```

Key points:

- **`start`** affects the current boot session only.
- **`enable`** creates the boot-time association so the service starts automatically.
- **`restart`** is useful after configuration changes.
- **`reload`** applies supported configuration changes without fully stopping the daemon, but not every service supports it.
- **`status`** is your first troubleshooting command because it shows active state, recent logs, and the unit file path.

A practical workflow is:

1. Check whether the package is installed.
2. Start the service.
3. Enable it at boot.
4. Confirm it is active.
5. Open the firewall if remote access is required.
6. Test from another host.

**SSH service enablement**

SSH, or **Secure Shell**, provides encrypted remote login and command execution. On RHEL, the server daemon is `sshd`. In many installations it is already present, but you still need to verify that it is running and reachable.

Typical verification:

```bash
rpm -q openssh-server
systemctl status sshd
ss -tulpn | grep sshd
```

If the server package is missing, install it:

```bash
sudo dnf install -y openssh-server
```

Then enable and start the daemon:

```bash
sudo systemctl enable --now sshd
```

This single command both starts the service immediately and enables it for future boots.

**SSH configuration considerations**

The main server configuration file is `/etc/ssh/sshd_config`. Common exam-safe changes include:

- Changing the listening port
- Disabling root login
- Restricting authentication methods
- Limiting allowed users or groups

Example:

```conf
Port 22
PermitRootLogin no
PasswordAuthentication yes
AllowUsers student admin
```

After editing the file, validate and apply the change:

```bash
sudo sshd -t
sudo systemctl restart sshd
```

Important details:

- Use `sshd -t` to test syntax before restarting.
- If you change the port, update the firewall and any SELinux port labeling if needed.
- If you disable password authentication, make sure key-based authentication is already working or you may lock yourself out.

> <mark>Critical</mark>: Always keep an existing session open while testing SSH changes. If the new configuration fails, you can recover without losing access.

**SSH client-side usage**

To connect to a remote host:

```bash
ssh user@server.example.com
```

If SSH listens on a non-default port:

```bash
ssh -p 2222 user@server.example.com
```

Useful client options:

- `-i` to specify a private key
- `-v` for verbose troubleshooting
- `-L` for local port forwarding
- `-X` for X11 forwarding, if enabled and needed

Example key-based login:

```bash
ssh-keygen -t ed25519
ssh-copy-id user@server.example.com
ssh user@server.example.com
```

For RHCSA, you should understand that SSH access depends on all of the following:

- The daemon is running
- The firewall allows the port
- The server is listening on the expected address and port
- Authentication is valid
- SELinux is not blocking the chosen port or file context

**HTTP service enablement**

HTTP, or **Hypertext Transfer Protocol**, is commonly provided by the Apache web server package `httpd` on RHEL. A basic web service requires the package, the daemon, firewall access, and content in the correct document root.

Install Apache:

```bash
sudo dnf install -y httpd
```

Enable and start it:

```bash
sudo systemctl enable --now httpd
```

Verify it is listening:

```bash
systemctl status httpd
ss -tulpn | grep :80
```

The default document root is usually `/var/www/html`. A simple test page:

```bash
echo "Hello from RHEL" | sudo tee /var/www/html/index.html
```

Then test locally:

```bash
curl http://localhost
```

If the service is working locally but not remotely, the issue is often firewall or SELinux related.

**firewalld and service access**

`firewalld` controls network access by zone and service definition. For SSH and HTTP, you usually allow the predefined service names rather than opening raw ports.

Check the active zone:

```bash
sudo firewall-cmd --get-active-zones
```

List allowed services:

```bash
sudo firewall-cmd --zone=public --list-services
```

Allow SSH and HTTP permanently:

```bash
sudo firewall-cmd --permanent --add-service=ssh
sudo firewall-cmd --permanent --add-service=http
sudo firewall-cmd --reload
```

If you changed SSH to a custom port, you may need a port rule instead of the default service definition:

```bash
sudo firewall-cmd --permanent --add-port=2222/tcp
sudo firewall-cmd --reload
```

Practical guidance:

- Use `--permanent` for persistent changes.
- Use `--reload` to apply permanent changes without rebooting.
- Use `--add-service=http` for standard web traffic on port 80.
- Use `--add-service=https` if you later enable TLS on port 443.

**SELinux and web content**

SELinux, or **Security-Enhanced Linux**, can block access even when the service and firewall are correct. For HTTP, the most common issue is incorrect file context on web content.

Check contexts:

```bash
ls -Z /var/www/html
```

The default web content type should typically be `httpd_sys_content_t` for static files. If you place content in a nonstandard directory, Apache may not be allowed to read it.

Example of a custom document root:

```bash
sudo mkdir -p /srv/web/site
echo "Custom site" | sudo tee /srv/web/site/index.html
```

Label the directory for web access:

```bash
sudo semanage fcontext -a -t httpd_sys_content_t "/srv/web/site(/.*)?"
sudo restorecon -Rv /srv/web/site
```

If `semanage` is unavailable, install the policy tools package:

```bash
sudo dnf install -y policycoreutils-python-utils
```

For temporary troubleshooting, you can inspect denials:

```bash
sudo ausearch -m AVC,USER_AVC -ts recent
sudo journalctl -t setroubleshoot
```

Avoid disabling SELinux as a routine fix. On the exam, the expected solution is usually to correct the context or boolean, not to weaken security.

**Common HTTP booleans and use cases**

Some Apache features require SELinux booleans. A common example is allowing Apache to make outbound network connections, such as when it proxies to an application backend.

View booleans:

```bash
getsebool -a | grep httpd
```

Enable a boolean persistently:

```bash
sudo setsebool -P httpd_can_network_connect on
```

This is useful when Apache needs to connect to another service, but it is not required for a simple static website.

**Troubleshooting workflow**

When a service is not reachable, use a layered approach instead of guessing.

1. Confirm the service is installed.
2. Confirm the daemon is active.
3. Confirm it is enabled at boot.
4. Confirm the port is listening.
5. Confirm the firewall allows the traffic.
6. Confirm SELinux is not blocking access.
7. Confirm the client is using the correct hostname, port, and protocol.

Useful commands:

```bash
systemctl status sshd httpd
ss -tulpn
firewall-cmd --list-all
getenforce
journalctl -u httpd -b
journalctl -u sshd -b
```

For HTTP, test from the local machine first:

```bash
curl -I http://localhost
```

Then test from another host:

```bash
curl -I http://server.example.com
```

For SSH, test connectivity and authentication:

```bash
ssh -v user@server.example.com
```

If the port is closed, check whether the service is listening on the expected interface:

```bash
ss -tulpn | grep -E ':(22|80)\b'
```

**Typical exam scenarios**

You may be asked to complete tasks such as:

- Install and enable `sshd`
- Allow SSH through the firewall
- Change SSH to a nondefault port and make it reachable
- Install `httpd`, create a web page, and make it accessible
- Fix a web server blocked by SELinux
- Verify that a service starts automatically after reboot

A reliable sequence for HTTP setup is:

1. Install `httpd`.
2. Create or copy content into `/var/www/html`.
3. Enable and start `httpd`.
4. Open the firewall service `http`.
5. Test with `curl` locally and remotely.
6. Check SELinux labels if access fails.

Example end-to-end setup:

```bash
sudo dnf install -y httpd
echo "RHCSA web test" | sudo tee /var/www/html/index.html
sudo systemctl enable --now httpd
sudo firewall-cmd --permanent --add-service=http
sudo firewall-cmd --reload
curl http://localhost
```

**Service management best practices**

Use the service name, not the process name, when managing daemons with `systemctl`. For example, manage `sshd` and `httpd`, not `ssh` or `apache` unless the unit name actually matches.

Keep these habits:

- Verify with `systemctl status` after every change.
- Use `ss -tulpn` to confirm listening sockets.
- Use `firewall-cmd` for persistent network access.
- Use `restorecon` after creating files in labeled directories.
- Use `semanage fcontext` for permanent SELinux labeling rules.

**Quick reference**

| Task | Command |
|---|---|
| Start SSH now | `systemctl start sshd` |
| Enable SSH at boot | `systemctl enable sshd` |
| Start and enable HTTP | `systemctl enable --now httpd` |
| Allow HTTP in firewall | `firewall-cmd --permanent --add-service=http` |
| Reload firewall | `firewall-cmd --reload` |
| Test Apache locally | `curl http://localhost` |
| Check SELinux mode | `getenforce` |
| Fix file context | `restorecon -Rv /var/www/html` |

**What to remember for RHCSA**

The exam usually rewards practical, verifiable configuration. For SSH and HTTP, that means you should be able to:

- Install the correct package
- Enable and start the service
- Open the correct firewall rule
- Confirm the service is listening
- Fix SELinux labeling when needed
- Test from the local host and a remote client

If you can explain why a service is not reachable, you can usually fix it quickly.
