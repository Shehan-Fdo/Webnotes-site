## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**firewalld change model**

`firewalld` separates configuration into **runtime** and **permanent** state. Runtime changes affect the running firewall immediately, while permanent changes are stored on disk and applied after a reload or reboot.

This distinction matters in administration and on the RHCSA exam because you often need to decide whether a change should be temporary for testing or persistent for long-term policy.

- **Runtime** changes are active now and usually disappear after `firewalld` reloads or the system reboots.
- **Permanent** changes are written to configuration files under `/etc/firewalld/` and survive restarts.
- `firewall-cmd` can target either state with `--runtime-to-permanent`, `--permanent`, or a reload workflow.

**Core concepts**

`firewalld` uses zones, services, ports, protocols, and rich rules to control traffic. The same object can exist in both runtime and permanent configuration, but the two views are not automatically identical unless you synchronize them.

A practical way to think about it:

- Runtime is the live working copy.
- Permanent is the saved source of truth.
- Reloading `firewalld` replaces runtime with permanent configuration.

<dl>
<dt><strong>Zone</strong></dt>
<dd>A named trust level that groups rules, such as `public`, `home`, or `internal`.</dd>
<dt><strong>Service</strong></dt>
<dd>A predefined set of ports and protocols, such as `ssh` or `http`.</dd>
<dt><strong>Rich rule</strong></dt>
<dd>A more expressive rule format for matching source, service, port, logging, and actions.</dd>
</dl>

**Inspecting current firewall state**

Before changing anything, check both runtime and permanent settings so you know what is actually active and what will survive a reload.

```bash
firewall-cmd --get-active-zones
firewall-cmd --list-all
firewall-cmd --list-all --permanent
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-services --permanent
```

Useful inspection commands:

- `--list-all` shows services, ports, protocols, masquerading, and rich rules for the selected zone.
- `--get-active-zones` shows which interfaces are assigned to which zones.
- `--list-services` is a quick way to verify allowed services.
- `--permanent` switches the query to saved configuration instead of runtime state.

**Making runtime changes**

Runtime changes are ideal when you want to test access without committing the policy. They take effect immediately and are useful during troubleshooting.

```bash
firewall-cmd --zone=public --add-service=http
firewall-cmd --zone=public --add-port=8080/tcp
firewall-cmd --zone=public --remove-service=http
```

Common runtime operations:

- Add a service with `--add-service=NAME`.
- Add a port with `--add-port=PORT/PROTO`.
- Remove a rule with `--remove-service=NAME` or `--remove-port=PORT/PROTO`.
- Change the default zone with `--set-default-zone=ZONE`.

If you only make runtime changes, they are not guaranteed to persist after a reload.

> A runtime-only change is useful for validation, but it is easy to lose if you forget to save it.

**Making permanent changes**

Permanent changes are what you use for intended policy. Add `--permanent` to write the change to the saved configuration.

```bash
firewall-cmd --permanent --zone=public --add-service=http
firewall-cmd --permanent --zone=public --add-port=8080/tcp
firewall-cmd --permanent --zone=public --remove-service=http
```

Important behavior:

- `--permanent` updates the configuration on disk, not the live firewall.
- The running firewall does not change until you reload or otherwise synchronize runtime state.
- Permanent changes are the correct choice for long-term access rules.

**Synchronizing runtime and permanent state**

After changing permanent configuration, you usually need to apply it to the running firewall. The standard method is `firewall-cmd --reload`.

```bash
firewall-cmd --reload
```

What reload does:

- Re-reads permanent configuration.
- Replaces runtime rules with the saved configuration.
- Applies the new policy without restarting the `firewalld` service.

You can also copy runtime changes into permanent storage when you have tested a rule and want to keep it.

```bash
firewall-cmd --runtime-to-permanent
```

Use this when:

- You tested a runtime rule and confirmed it works.
- You want to preserve the current live configuration.
- You do not want to retype the rule as a permanent command.

**Typical workflow patterns**

1. Test a rule at runtime.
2. Verify the application works.
3. Save the working rule permanently.
4. Reload or synchronize if needed.

Example workflow:

```bash
firewall-cmd --zone=public --add-service=http
curl http://server.example.com
firewall-cmd --runtime-to-permanent
```

Alternative workflow:

```bash
firewall-cmd --permanent --zone=public --add-service=http
firewall-cmd --reload
```

Choose the first approach when you want to validate behavior before saving. Choose the second when you already know the rule is correct and want a clean persistent change.

**Services versus ports**

Whenever possible, prefer services over raw ports because services are easier to read and maintain. A service definition may include multiple ports and protocols, which reduces the chance of missing a required rule.

- **Service** rules are higher level and more descriptive.
- **Port** rules are useful when no predefined service exists or when the application uses a custom port.
- If you use a custom application port, document it clearly so future administrators understand why it is open.

Examples:

```bash
firewall-cmd --permanent --zone=public --add-service=ssh
firewall-cmd --permanent --zone=public --add-port=8443/tcp
firewall-cmd --reload
```

**Verifying changes**

Always verify both the live and saved state after making changes. This prevents confusion when a rule appears to work temporarily but is not persistent.

```bash
firewall-cmd --zone=public --list-all
firewall-cmd --zone=public --list-all --permanent
firewall-cmd --query-service=http
firewall-cmd --query-port=8080/tcp
```

Verification tips:

- Use `--query-service=NAME` to check whether a service is enabled.
- Use `--query-port=PORT/PROTO` to check whether a port is open.
- Compare runtime and permanent output if behavior seems inconsistent.

**Reload versus restart**

A reload is usually enough for firewall policy changes. It is less disruptive than restarting the service.

- `firewall-cmd --reload` applies permanent configuration to runtime.
- `systemctl restart firewalld` restarts the daemon and may be unnecessary for normal rule changes.
- Prefer reload for routine administration unless troubleshooting requires a restart.

```bash
firewall-cmd --reload
systemctl status firewalld
```

**Common exam-relevant scenarios**

You may need to open SSH, HTTP, or a custom application port while keeping the system secure.

Example: allow web access permanently on the default zone.

```bash
firewall-cmd --permanent --zone=public --add-service=http
firewall-cmd --permanent --zone=public --add-service=https
firewall-cmd --reload
```

Example: temporarily test a custom service port.

```bash
firewall-cmd --zone=public --add-port=9000/tcp
ss -tuln | grep 9000
firewall-cmd --runtime-to-permanent
```

Example: remove a rule from both runtime and permanent state.

```bash
firewall-cmd --zone=public --remove-port=9000/tcp
firewall-cmd --permanent --zone=public --remove-port=9000/tcp
firewall-cmd --reload
```

**Troubleshooting checklist**

If a rule does not behave as expected, check these items in order:

1. Confirm the correct zone is active on the interface.
2. Verify whether the rule exists in runtime, permanent, or both.
3. Reload after permanent changes.
4. Confirm the service is actually listening on the expected port.
5. Check for SELinux or application-level restrictions if the firewall is correct.

Useful commands:

```bash
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-all
ss -tuln
getenforce
```

**Configuration file locations**

Permanent firewalld configuration is stored under `/etc/firewalld/`. Zone definitions and custom rules are typically managed there, though `firewall-cmd` is the preferred interface for most administration tasks.

- `/etc/firewalld/zones/` contains zone configuration files.
- `/etc/firewalld/services/` contains custom service definitions.
- Manual edits are possible, but command-line management is safer and less error-prone.

**Practical guidance**

Use runtime changes when you are testing or troubleshooting. Use permanent changes when the policy should survive reboot or reload. After any permanent edit, reload the firewall and verify the result.

A reliable admin habit is:

- make the smallest change needed,
- verify it immediately,
- save it only after confirming it works,
- recheck both runtime and permanent views.
