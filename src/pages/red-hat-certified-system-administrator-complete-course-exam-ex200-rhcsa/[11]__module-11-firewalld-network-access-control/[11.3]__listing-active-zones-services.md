## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

`firewalld` manages host-based firewall policy on RHEL by organizing rules into **zones** and attaching network interfaces or source addresses to those zones. When you need to confirm what traffic is allowed, the first task is to identify the active zone(s) and the services enabled in them.

A zone is a trust profile. For example, a public-facing interface usually belongs in a restrictive zone, while a trusted internal network may use a more permissive one. Services are predefined rule bundles such as `ssh`, `http`, or `cockpit` that open the ports and protocols required by common applications.

**Why this matters**

- You verify the current firewall state before troubleshooting connectivity.
- You confirm whether a service is allowed without manually checking every port.
- You avoid changing the wrong zone, which is a common administration mistake.
- You can compare runtime state with persistent configuration to understand what survives a reboot.

**Core concepts**

- **Runtime configuration** is the active firewall state in memory.
- **Permanent configuration** is stored on disk and loaded at boot or when reloaded.
- **Active zone** means a zone currently bound to at least one interface or source.
- **Service** means a named firewall rule set defined by `firewalld`.

**Listing active zones**

The most useful command for exam and day-to-day administration is:

```bash
firewall-cmd --get-active-zones
```

Typical output may look like this:

```bash
public
  interfaces: enp1s0
internal
  interfaces: enp2s0
```

This tells you which zones are active and which interfaces belong to each zone. If a zone has no interface or source assigned, it is not active.

You can also inspect the default zone:

```bash
firewall-cmd --get-default-zone
```

The default zone is the fallback zone used when an interface is not explicitly assigned elsewhere.

**Inspecting services in an active zone**

To list services allowed in the current runtime configuration for a specific zone, use:

```bash
firewall-cmd --zone=public --list-services
```

Example output:

```bash
ssh dhcpv6-client
```

This means the `public` zone currently allows SSH and DHCPv6 client traffic.

If you want to see more than services, inspect all runtime settings for the zone:

```bash
firewall-cmd --zone=public --list-all
```

Example output:

```bash
public (active)
  target: default
  interfaces: enp1s0
  services: ssh dhcpv6-client
  ports: 8080/tcp
  protocols: icmp
  forward: no
  masquerade: no
  rich rules:
```

This is often the fastest way to understand the effective firewall policy for a zone.

**Useful inspection commands**

- `firewall-cmd --get-active-zones` shows active zones and their interfaces.
- `firewall-cmd --get-default-zone` shows the fallback zone.
- `firewall-cmd --zone=<zone> --list-services` shows allowed services in a zone.
- `firewall-cmd --zone=<zone> --list-ports` shows explicitly opened ports.
- `firewall-cmd --zone=<zone> --list-all` shows the full runtime view.
- `firewall-cmd --list-all-zones` shows all zones and their settings.

**Runtime versus permanent state**

A common source of confusion is that a service may appear in the runtime configuration but not in the permanent configuration, or vice versa. Use both views when validating changes.

```bash
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-services --permanent
```

If the outputs differ, the runtime and permanent states are not aligned. A `--reload` loads permanent settings into runtime, while `--runtime-to-permanent` copies runtime settings to disk.

> Be careful: if you add a rule without `--permanent`, it may disappear after a reboot or reload depending on how you applied it.

**Practical workflow**

1. Identify the active zone.
2. Check which services are enabled in that zone.
3. Confirm the interface is attached to the expected zone.
4. Compare runtime and permanent settings if behavior is inconsistent.
5. Test connectivity from the client side after making changes.

Example workflow:

```bash
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-all
firewall-cmd --zone=public --list-all --permanent
```

**Adding and verifying a service**

If you need to allow a service, add it to the correct zone and then verify it.

```bash
firewall-cmd --zone=public --add-service=http
firewall-cmd --zone=public --list-services
```

For a persistent change:

```bash
firewall-cmd --zone=public --add-service=http --permanent
firewall-cmd --reload
firewall-cmd --zone=public --list-services
```

This pattern is important because `--add-service` without `--permanent` only affects the current runtime session.

**Common services you may encounter**

- `ssh` for remote shell access.
- `http` for web traffic on port 80/tcp.
- `https` for secure web traffic on port 443/tcp.
- `cockpit` for the web-based administration interface.
- `dhcpv6-client` for IPv6 address assignment support.

These are service names, not necessarily port numbers you type manually. `firewalld` maps them to the correct ports and protocols.

**Troubleshooting checklist**

If a service is not reachable, check the following:

- The correct interface is in the expected zone.
- The service is enabled in that zone.
- The rule exists in runtime and, if needed, permanent configuration.
- The application is actually listening on the expected port.
- SELinux is not blocking the service.
- The client is testing the correct IP address and protocol.

A useful command to confirm listening sockets is:

```bash
ss -tuln
```

For example, if `http` is allowed but no process is listening on port 80, the firewall is not the problem.

**Example scenario**

Suppose `enp1s0` is in the `public` zone and you need to allow web access:

```bash
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --add-service=http --permanent
firewall-cmd --reload
firewall-cmd --zone=public --list-services
```

If the interface is in the wrong zone, move it to the correct one instead of opening extra services everywhere.

**Zone and service inspection summary**

- Use `--get-active-zones` to find where traffic is being filtered.
- Use `--list-services` to see what named access is allowed.
- Use `--list-all` when you need the full picture.
- Always distinguish runtime from permanent state.
- Verify the listening application after firewall changes.

**Quick reference**

```bash
firewall-cmd --get-active-zones
firewall-cmd --get-default-zone
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-all
firewall-cmd --zone=public --list-services --permanent
firewall-cmd --reload
ss -tuln
```

**Exam focus**

For RHCSA tasks, you should be able to:

- Identify the active zone for an interface.
- List services enabled in that zone.
- Determine whether a rule is runtime-only or permanent.
- Verify that a firewall change matches the intended network access.
- Troubleshoot access issues by checking both `firewalld` and the service itself.
