## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Firewalld overview**

`firewalld` is the dynamic firewall manager used on modern RHEL systems. It controls packet filtering through the kernel firewall backend while letting you change rules without rebuilding the entire firewall state. For RHCSA tasks, you usually manage access by assigning interfaces or sources to zones, then allowing services, ports, or rich rules in the correct zone.

Key ideas to remember:

- **Runtime** changes affect the running firewall immediately.
- **Permanent** changes survive reboot but do not affect the running firewall until you reload or also apply the change at runtime.
- **Zones** define trust levels and rule sets for different network contexts.
- **Services** are named collections of ports and protocols.
- **Rich rules** give you fine-grained control when simple service or port rules are not enough.

**Service management and basic verification**

Before troubleshooting firewall behavior, confirm that the daemon is running and enabled. On RHEL, `firewalld` is typically managed with `systemctl`.

```bash
systemctl status firewalld
systemctl enable --now firewalld
systemctl restart firewalld
systemctl disable firewalld
```

Useful checks:

- `systemctl status firewalld` shows whether the service is active.
- `systemctl is-enabled firewalld` confirms whether it starts at boot.
- `firewall-cmd --state` reports whether `firewalld` is running.
- `firewall-cmd --reload` loads permanent configuration into the runtime firewall.
- `firewall-cmd --runtime-to-permanent` copies runtime settings into permanent configuration.

> If `firewalld` is stopped, `firewall-cmd` may fail or report that the daemon is not running. Start the service before making firewall changes.

**Zones and trust levels**

A **zone** is a named policy profile that determines how much trust you place in traffic from a network connection, interface, or source address. You do not usually open a port globally; instead, you open it in the zone that applies to the traffic.

Common zone behavior:

- **drop** silently discards packets.
- **block** rejects packets rather than silently dropping them.
- **public** is for untrusted networks and allows only selected incoming connections.
- **trusted** allows all traffic and should be used carefully.
- Other zones such as `home`, `work`, `internal`, `dmz`, and `external` provide different trust assumptions.

A practical mental model:

- Use `public` for a laptop on a shared network.
- Use `internal` or `home` for a trusted LAN.
- Use `dmz` for a server that should expose only a few services.
- Use `drop` when you want no response to unsolicited traffic.
- Use `block` when you want traffic rejected rather than silently ignored.

**Default zone and zone assignment**

The **default zone** is the zone used when a connection, interface, or source is not explicitly bound to another zone. This matters because a service opened in the wrong zone may not be reachable.

View and change the default zone:

```bash
firewall-cmd --get-default-zone
firewall-cmd --set-default-zone=public
```

List available zones and inspect one:

```bash
firewall-cmd --get-zones
firewall-cmd --list-all-zones
firewall-cmd --zone=public --list-all
```

To see which zones are active and what interfaces or sources they contain:

```bash
firewall-cmd --get-active-zones
```

Typical output may show an interface bound to a zone, such as `ens160` in `public`. If multiple zones are active, the interface or source binding determines which zone’s rules apply.

**Changing the active zone for an interface or source**

You often need to move a network interface or source network into a different zone. This is important when a server has multiple NICs or when you want to trust one subnet more than another.

Assign an interface to a zone:

```bash
firewall-cmd --zone=internal --change-interface=ens160
```

Assign a source network to a zone:

```bash
firewall-cmd --zone=trusted --add-source=192.168.50.0/24
```

Make the change permanent if needed:

```bash
firewall-cmd --permanent --zone=internal --change-interface=ens160
firewall-cmd --permanent --zone=trusted --add-source=192.168.50.0/24
firewall-cmd --reload
```

Important behavior:

- An interface bound to a zone uses that zone’s rules for traffic arriving on that interface.
- A source bound to a zone applies to traffic from that source network.
- If both interface and source are relevant, source-based matching can affect how traffic is classified.
- Always verify the effective zone with `--get-active-zones` and zone listings after changes.

**Services**

A **service** in firewalld is a named definition that maps to one or more ports, protocols, and sometimes helper settings. Services are easier to manage than raw ports because they are human-readable and standardized.

Examples of common services:

- `ssh`
- `http`
- `https`
- `cockpit`
- `dns`
- `smtp`

List available service definitions:

```bash
firewall-cmd --get-services
```

Allow a service in the current runtime zone:

```bash
firewall-cmd --zone=public --add-service=http
firewall-cmd --zone=public --add-service=ssh
```

Make it permanent:

```bash
firewall-cmd --permanent --zone=public --add-service=http
firewall-cmd --permanent --zone=public --add-service=ssh
firewall-cmd --reload
```

Remove a service:

```bash
firewall-cmd --zone=public --remove-service=http
firewall-cmd --permanent --zone=public --remove-service=http
firewall-cmd --reload
```

Why services are preferred when available:

- They reduce mistakes compared with manually entering ports.
- They make intent clearer during troubleshooting.
- They survive package updates better when the service definition changes.

**Ports**

Use a **port** rule when no service definition exists or when the application uses a nonstandard port. You must specify the port number and protocol.

Open a TCP port:

```bash
firewall-cmd --zone=public --add-port=8080/tcp
```

Open a UDP port:

```bash
firewall-cmd --zone=public --add-port=161/udp
```

Make the port permanent:

```bash
firewall-cmd --permanent --zone=public --add-port=8080/tcp
firewall-cmd --reload
```

Remove a port:

```bash
firewall-cmd --zone=public --remove-port=8080/tcp
```

Practical guidance:

- Use `tcp` for most web, SSH, and database services.
- Use `udp` for protocols such as DNS queries or SNMP.
- Always include the protocol; `8080` alone is incomplete.
- Prefer a service rule when one exists, because it is easier to read and maintain.

**Runtime versus permanent configuration**

Firewalld keeps two views of configuration:

- **Runtime**: active now, lost on reboot unless saved.
- **Permanent**: stored on disk, loaded at boot or after reload.

Examples:

```bash
firewall-cmd --zone=public --add-service=http
firewall-cmd --permanent --zone=public --add-service=http
firewall-cmd --reload
```

Interpretation:

- If you add only the permanent rule, it appears after `--reload` or reboot.
- If you add only the runtime rule, it works immediately but disappears after reboot.
- If you need both immediate effect and persistence, apply the rule in both contexts or add it permanently and reload.

Verify both views:

```bash
firewall-cmd --zone=public --list-services
firewall-cmd --permanent --zone=public --list-services
```

**Rich rules**

A **rich rule** gives you more control than a simple service or port rule. You can match on source, destination, family, protocol, port, service, logging, auditing, and action. Use rich rules when you need exceptions, logging, rate limiting, or source-specific access.

General structure:

```text
rule [family="ipv4|ipv6"] [source address="..."] [destination address="..."] [service name="..."] [port port="..." protocol="tcp|udp"] [log [prefix="..."] [level="..."] [limit value="..."]] [audit] [accept|reject|drop]
```

Common elements:

- **family** limits the rule to `ipv4` or `ipv6`.
- **source** matches a client address or network.
- **service** matches a named service.
- **port** matches a specific port and protocol.
- **log** writes a log message when the rule matches.
- **audit** records an audit event.
- **accept**, **reject**, and **drop** define the final action.
- **limit** rate-limits logging or matching behavior.

Examples:

Allow SSH only from one subnet:

```bash
firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" source address="192.168.50.0/24" service name="ssh" accept'
```

Allow HTTP and log matches before accepting:

```bash
firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" service name="http" log prefix="FW-HTTP " level="info" accept'
```

Reject a specific port from a subnet:

```bash
firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" source address="10.10.10.0/24" port port="8443" protocol="tcp" reject'
```

Drop traffic silently from a noisy host:

```bash
firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" source address="203.0.113.25" drop'
```

Rate-limit logging:

```bash
firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" service name="ssh" log prefix="SSH " level="info" limit value="1/m" accept'
```

Notes on rich rules:

- `log` and the final action are separate parts of the rule.
- `reject` sends a rejection response; `drop` silently discards traffic.
- Use `limit` to avoid flooding logs.
- Keep rules specific so you do not accidentally expose more than intended.

**Practical workflow for opening access**

When you need to expose a service, follow a repeatable process.

1. Identify the correct zone for the interface or source.
2. Add the service or port to that zone.
3. Decide whether the change must be permanent.
4. Reload if you changed only permanent configuration.
5. Verify from the server and from a client.

Example: enable web access on `public`:

```bash
firewall-cmd --zone=public --add-service=http
firewall-cmd --permanent --zone=public --add-service=http
firewall-cmd --reload
firewall-cmd --zone=public --list-services
```

Example: open a custom application port:

```bash
firewall-cmd --zone=public --add-port=5000/tcp
firewall-cmd --permanent --zone=public --add-port=5000/tcp
firewall-cmd --reload
```

**Verification from the server side**

Use `firewall-cmd` to confirm the active configuration:

```bash
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-all
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-ports
firewall-cmd --zone=public --list-rich-rules
```

Useful troubleshooting commands:

```bash
ss -tuln
ip addr show
journalctl -u firewalld --since "10 minutes ago"
```

Interpretation tips:

- `ss -tuln` confirms whether the application is listening on the expected port.
- `ip addr show` helps you confirm which interface has the address you expect.
- `journalctl -u firewalld` can reveal syntax errors or reload issues.

**Verification from a client perspective**

Firewall changes are only useful if the service is reachable from the intended client. Test connectivity from another host on the network.

Examples:

```bash
nc -vz server.example.com 80
curl -I http://server.example.com
ssh user@server.example.com
```

For UDP services, use tools appropriate to the protocol and service behavior. For example, DNS may require `dig`:

```bash
dig @server.example.com example.com
```

If a connection fails:

- Confirm the service is running on the server.
- Confirm the correct zone contains the rule.
- Confirm the interface or source is bound to that zone.
- Confirm the rule is present in runtime, not only permanent configuration.
- Confirm the client is testing the correct IP address and protocol.

**Common exam patterns**

RHCSA firewall tasks usually involve one of these patterns:

- Allow a standard service such as `ssh`, `http`, or `https`.
- Open a custom port for an application.
- Bind an interface or source to the correct zone.
- Make a change permanent.
- Remove an unnecessary rule.
- Verify the active configuration after a change.

A good exam habit is to check the current state before editing:

```bash
firewall-cmd --get-default-zone
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-all
```

Then make the smallest correct change and verify again.

**Quick comparison**

| Concept | What it controls | Typical use |
|---|---|---|
| Zone | Trust level and rule set | Separate policies for different networks |
| Service | Named port/protocol definition | Open standard applications like SSH or HTTP |
| Port | Specific TCP/UDP port | Open custom or nonstandard applications |
| Rich rule | Fine-grained matching and actions | Source-based access, logging, reject/drop behavior |

**Troubleshooting checklist**

- Confirm `firewalld` is running with `systemctl status firewalld`.
- Confirm the correct zone with `--get-active-zones`.
- Confirm the default zone with `--get-default-zone`.
- Confirm the rule exists in the intended zone.
- Confirm whether the rule is runtime, permanent, or both.
- Confirm the application is listening with `ss -tuln`.
- Confirm the client is testing the right host, port, and protocol.

**Exam-ready reminders**

- Use `--permanent` for persistence.
- Use `--reload` after permanent-only changes.
- Prefer services over raw ports when possible.
- Use rich rules for source-specific or logging requirements.
- Remember that `drop` and `block` are not the same: `drop` silently discards traffic, while `block` rejects it.
