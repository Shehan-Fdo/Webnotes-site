## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Structured troubleshooting mindset**

Troubleshooting on RHCSA is less about memorizing fixes and more about following a repeatable process. You want to identify the symptom, narrow the scope, verify the cause, apply the smallest safe change, and confirm the result. This approach reduces guesswork and helps you work under exam pressure.

A good workflow usually looks like this:

1. Confirm the problem and its impact.
2. Check the most likely layer first.
3. Gather evidence before changing anything.
4. Fix one variable at a time.
5. Verify the service, boot path, mount, or network path end to end.

A useful mental model is to separate failures into layers:

| Layer | Typical question | Common tools |
|---|---|---|
| Hardware / virtualization | Is the device visible and healthy? | `dmesg`, `lsblk`, `lspci`, `ip link`, `smartctl` |
| Boot / kernel | Does the system reach the expected target? | GRUB2, `rd.break`, `journalctl -b`, `systemctl` |
| Storage / filesystem | Is the disk mounted correctly and writable? | `lsblk`, `blkid`, `mount`, `fsck`, `xfs_repair` |
| Service / process | Is the daemon running and listening? | `systemctl`, `ss`, `journalctl -u` |
| Network | Is the interface up and reachable? | `nmcli`, `ip`, `ping`, `ss`, `firewall-cmd` |
| Security policy | Is SELinux or firewall blocking access? | `getenforce`, `ausearch`, `sealert`, `firewall-cmd` |

**Start with the simplest checks**

Before you assume a complex root cause, verify the basics:

- **Power and cabling**: confirm the machine is on, connected, and not in a failed virtual hardware state.
- **Disk health**: check whether the device appears in `lsblk` and whether the kernel reports I/O errors.
- **Memory and CPU symptoms**: look for kernel messages, OOM kills, or repeated crashes.
- **NIC status**: confirm the interface is present, carrier is detected, and the link is up.
- **Recent changes**: ask what changed last, because most outages follow a change in config, package, or policy.

Useful quick commands:

```bash
dmesg -T | tail -n 50
lsblk -f
ip link show
ip addr show
journalctl -b -p warning
```

**Hardware-level troubleshooting basics**

RHCSA problems often begin below the service layer. If the system cannot see the device, no amount of service tuning will help.

- **Disk visibility**
  - Use `lsblk`, `blkid`, and `fdisk -l` to confirm the disk and partitions exist.
  - Check `dmesg` for I/O errors, timeouts, or controller resets.
  - On virtual machines, verify the disk is attached to the VM and the controller type is correct.

- **Disk health**
  - For SATA/SAS/NVMe devices, `smartctl` can reveal failing media if installed.
  - Repeated read/write errors in `dmesg` often indicate hardware or virtualization issues rather than filesystem corruption.

- **Memory symptoms**
  - Kernel logs may show out-of-memory kills or machine check exceptions.
  - Unexpected application exits can be caused by memory pressure rather than the application itself.

- **NIC and link status**
  - `ip link show` tells you whether the interface is administratively up.
  - `ethtool <iface>` can show link detection and speed if available.
  - If the interface exists but has no carrier, check cabling, switch port, or virtual network attachment.

Example:

```bash
ip link show ens160
ethtool ens160
dmesg -T | grep -iE 'error|fail|link|timeout'
```

> If the kernel cannot see the device or reports repeated hardware errors, fix the hardware or virtual attachment first. Do not chase service configuration until the platform is stable.

**A practical troubleshooting decision tree**

Use this sequence when you are unsure where to begin:

1. Identify the symptom.
   - Boot failure
   - Mount failure
   - Service not starting
   - Network unreachable
   - Access denied
2. Determine the layer.
   - Hardware, boot, filesystem, service, network, or security policy
3. Check logs and status.
   - `journalctl`, `systemctl status`, `dmesg`, `ss`, `nmcli`, `ausearch`
4. Test the smallest likely fix.
   - Correct a config line, mount option, firewall rule, or SELinux label
5. Verify from the user’s perspective.
   - Reboot, remount, reconnect, or retry the client action

**Boot troubleshooting and recovery**

Boot issues on RHCSA usually involve GRUB2, kernel arguments, initramfs, root filesystem mounting, or a broken `/etc/fstab`. Your goal is to get the system to a shell, repair the cause, and restore a clean boot.

Common boot symptoms:

- The system drops to the emergency shell.
- The machine hangs during boot.
- The root filesystem is not found.
- A wrong UUID or mount option prevents startup.
- A password reset or SELinux relabel is needed after recovery.

**GRUB2 and bootloader checks**

On RHEL systems, the GRUB2 configuration path depends on the boot mode and layout:

- BIOS systems commonly use `/boot/grub2/grub.cfg`
- UEFI systems commonly use `/boot/efi/EFI/<vendor>/grub.cfg`

You should confirm the actual layout before regenerating configuration.

Useful commands:

```bash
ls /boot
ls /boot/efi/EFI
grub2-editenv list
```

To regenerate GRUB2 configuration, use the correct target for the system:

```bash
# BIOS example
grub2-mkconfig -o /boot/grub2/grub.cfg

# UEFI example; vendor directory may be redhat, centos, rocky, etc.
grub2-mkconfig -o /boot/efi/EFI/redhat/grub.cfg
```

If you need to edit a boot entry temporarily, use the GRUB menu and append kernel arguments such as `rd.break` or `systemd.unit=rescue.target`.

**Recovering with `rd.break`**

`rd.break` drops you into the initramfs environment before the real root filesystem is fully mounted. This is useful for password recovery, SELinux relabeling, and fixing root-level issues.

Typical recovery flow:

1. At the GRUB menu, edit the kernel line and append `rd.break`.
2. Boot into the emergency shell.
3. Remount the real root filesystem read-write.
4. Chroot into `/sysroot`.
5. Make the required fix.
6. Relabel if needed.
7. Exit and reboot.

Example:

```bash
mount -o remount,rw /sysroot
chroot /sysroot
passwd root
touch /.autorelabel
exit
reboot
```

The `touch /.autorelabel` step is important when you change files that SELinux labels must re-evaluate, especially after password recovery or manual file restoration. On the next boot, SELinux relabels the filesystem.

> For root password recovery on SELinux-enabled systems, do not rely on `restorecon` alone for `/etc/shadow`. Use the standard recovery flow and trigger a full relabel when appropriate.

**Emergency mode and rescue mode**

- **Rescue mode** is useful when you need a minimal environment with the root filesystem mounted.
- **Emergency mode** gives you a very small shell for critical repair when normal boot cannot continue.

You can also use `systemd.unit=rescue.target` or `systemd.unit=emergency.target` from the GRUB kernel line when needed.

**Checking `/etc/fstab` and UUID issues**

A bad `/etc/fstab` entry is a common cause of boot failure. Typical problems include:

- Wrong UUID
- Missing device
- Invalid mount options
- Filesystem type mismatch
- Network mount dependency without proper boot handling

Workflow:

1. Boot to rescue or emergency mode.
2. Inspect `/etc/fstab`.
3. Compare UUIDs with `blkid`.
4. Test mounts before rebooting.
5. Fix the file and verify cleanly.

Example:

```bash
blkid
cat /etc/fstab
mount -a
```

If `mount -a` returns no errors, the entries are syntactically and semantically valid enough for a test mount. If it fails, correct the issue before rebooting.

**Boot repair checklist**

- Confirm the correct kernel and initramfs are present.
- Verify the root filesystem UUID in GRUB and `/etc/fstab`.
- Check whether the system uses BIOS or UEFI.
- Use `journalctl -b` after boot to inspect the previous boot’s errors.
- Trigger `touch /.autorelabel` after SELinux-sensitive recovery changes.
- Reboot only after `mount -a` succeeds and the root cause is fixed.

**Filesystem and storage troubleshooting**

Storage problems usually fall into one of three categories:

- The device is missing or misidentified.
- The filesystem is corrupted or inconsistent.
- The filesystem is healthy but mounted incorrectly or with the wrong permissions/options.

Start by identifying the device and filesystem type:

```bash
lsblk -f
blkid
mount | column -t
```

**When to use `fsck` and when to use `xfs_repair`**

- Use `fsck` for filesystems that support it, such as ext4.
- Use `xfs_repair` for XFS filesystems.
- Do not run repair tools on a mounted filesystem unless the tool explicitly supports it and the situation is safe.
- For XFS, repair is typically done on an unmounted filesystem.

Examples:

```bash
fsck -f /dev/sdb1
xfs_repair /dev/sdb1
```

If an ext filesystem is dirty, `fsck` can often repair metadata inconsistencies. If XFS reports corruption, `xfs_repair` is the standard tool. If the filesystem is mounted and in use, unmount it first or boot into rescue mode.

**Mount troubleshooting**

If a filesystem does not appear where expected:

- Check whether the mount point exists.
- Verify the device name or UUID.
- Confirm the filesystem type.
- Inspect mount options.
- Review `journalctl -b` for mount errors.

Example:

```bash
mkdir -p /data
mount /dev/sdb1 /data
findmnt /data
```

If the mount works manually but not at boot, the issue is usually `/etc/fstab`, dependency ordering, or a missing device at boot time.

**Permissions versus filesystem versus mount issues**

A directory can appear empty or inaccessible for several different reasons:

- The filesystem is not mounted.
- The mount point is wrong.
- Permissions block access.
- SELinux blocks access.
- The service is writing to a different path than you expect.

Check in this order:

1. `findmnt <path>`
2. `ls -ld <path>`
3. `getenforce`
4. `ausearch -m avc -ts recent`
5. Service logs

**Service troubleshooting**

When a service fails, you want to know whether the problem is:

- The unit is disabled or masked.
- The configuration file is invalid.
- The service cannot bind to its port.
- A dependency is missing.
- SELinux or firewall blocks access.
- The service is running, but the client is connecting to the wrong place.

Start with systemd:

```bash
systemctl status httpd
journalctl -u httpd -b
systemctl is-enabled httpd
systemctl is-active httpd
```

**Inspecting unit overrides and masking**

Systemd drop-ins and masks can change service behavior in ways that are easy to miss.

- Use `systemctl cat <unit>` to see the full unit plus drop-ins.
- Use `systemctl edit <unit>` to create a drop-in override.
- Use `systemctl mask <unit>` to prevent a service from starting.
- Use `systemctl unmask <unit>` to restore normal behavior.

Examples:

```bash
systemctl cat sshd
systemctl mask telnet.socket
systemctl unmask telnet.socket
systemctl edit httpd
```

If a service behaves unexpectedly, inspect overrides before changing the main unit file. A drop-in may be setting a different environment variable, restart policy, or dependency.

**Service binding versus firewall exposure**

A service can be running and still unreachable if it is not listening on the expected address or if the firewall blocks the port.

Check the listening socket:

```bash
ss -tulpn | grep -E ':80|:443'
```

Check the firewall:

```bash
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-ports
```

If the service is listening only on `127.0.0.1`, remote clients cannot connect. If the service listens on `0.0.0.0` or the host IP but the firewall blocks the port, you must open the correct service or port in the active zone.

Example fix:

```bash
firewall-cmd --permanent --add-service=http
firewall-cmd --reload
```

**SELinux troubleshooting**

SELinux issues often look like permissions problems, but the fix is different. A file may have the correct UNIX permissions and still be denied by policy.

Use this decision path:

1. Confirm the service or command fails.
2. Check whether SELinux is enforcing.
3. Look for AVC (Access Vector Cache) denials.
4. Compare file labels and expected contexts.
5. Fix the label, boolean, or policy-related setting.

Useful commands:

```bash
getenforce
ausearch -m avc -ts recent
journalctl -b | grep -i avc
ls -Z /var/www/html
```

If `setroubleshoot` is installed, it may provide human-readable hints, but it is not guaranteed to be present. `ausearch` and `journalctl` are more reliable starting points.

**Common SELinux fixes**

- Restore default labels with `restorecon`.
- Change persistent labels with `semanage fcontext` followed by `restorecon`.
- Adjust booleans when a service needs a policy exception.

Examples:

```bash
restorecon -Rv /var/www/html
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
restorecon -Rv /srv/web
getsebool -a | grep httpd
setsebool -P httpd_can_network_connect on
```

**Permissions versus SELinux versus firewall**

A common RHCSA exam trap is confusing these three layers. Use this comparison:

| Symptom | Likely cause | Best first check |
|---|---|---|
| File exists but access is denied locally | UNIX permissions or ownership | `ls -l` |
| Service works locally but not remotely | Firewall or bind address | `ss`, `firewall-cmd` |
| Service fails with “permission denied” despite correct UNIX permissions | SELinux | `ausearch -m avc -ts recent` |
| Web content not served from custom directory | SELinux label or Apache config | `ls -Z`, `restorecon` |

**Network troubleshooting**

Network issues on RHCSA usually involve interface state, IP configuration, routing, DNS, or firewall rules. Work from the local host outward.

**Interface and connection checks**

Use NetworkManager tools first:

```bash
nmcli device status
nmcli connection show
nmcli connection up "System eth0"
nmcli connection down "System eth0"
```

If the connection profile name differs, use the actual profile name shown by `nmcli connection show`. Do not assume a specific profile name.

Check addressing and routes:

```bash
ip addr show
ip route show
```

**DNS troubleshooting**

DNS problems often appear as “host unreachable” or “name not known” even when the network is otherwise fine.

Check resolver configuration:

```bash
cat /etc/resolv.conf
nmcli device show ens160 | grep -i dns
getent hosts example.com
```

Use `getent` because it tests the system resolver path, not just a single tool’s behavior.

**Connectivity testing order**

1. Ping the local interface or gateway.
2. Ping a known IP address.
3. Test name resolution.
4. Test the service port.
5. Check firewall and service binding.

Example:

```bash
ping -c 3 192.168.1.1
ping -c 3 8.8.8.8
getent hosts redhat.com
ss -tulpn | grep :22
```

**Firewall troubleshooting**

The firewall can block access even when the service is healthy.

Check the active zone and allowed services:

```bash
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-all
```

Open the correct service or port in the active zone, then reload:

```bash
firewall-cmd --permanent --add-service=ssh
firewall-cmd --reload
```

**End-to-end network example**

Suppose a web server is reachable locally but not from another host.

1. Confirm the service is running:
   ```bash
   systemctl status httpd
   ```
2. Confirm it is listening on the expected address:
   ```bash
   ss -tulpn | grep :80
   ```
3. Confirm the firewall allows HTTP:
   ```bash
   firewall-cmd --list-services
   ```
4. Confirm SELinux labels are correct:
   ```bash
   ls -Z /var/www/html
   ```
5. Test from a remote client:
   ```bash
   curl http://server.example.com
   ```

If the service listens on `127.0.0.1`, change the application bind address. If it listens correctly but remote access fails, open the firewall. If the firewall is open but content still fails, inspect SELinux and application logs.

**A structured exam workflow**

When you face an unknown problem in the exam, use this repeatable sequence:

1. Read the symptom carefully.
2. Identify the affected layer.
3. Check status and logs.
4. Compare expected versus actual configuration.
5. Make the smallest safe change.
6. Verify the fix.
7. Re-test after reboot if persistence matters.

This workflow is especially useful when multiple issues exist at once. For example, a service may fail because the filesystem is not mounted, the config file is wrong, and SELinux labels are missing. Fixing only one layer may not fully resolve the symptom.

**Scenario 1: Root password recovery with SELinux**

A system boots, but you cannot log in as root.

1. Edit the GRUB entry and append `rd.break`.
2. Boot into the initramfs shell.
3. Remount the real root filesystem read-write:
   ```bash
   mount -o remount,rw /sysroot
   ```
4. Chroot into the installed system:
   ```bash
   chroot /sysroot
   ```
5. Reset the root password:
   ```bash
   passwd root
   ```
6. Trigger a relabel:
   ```bash
   touch /.autorelabel
   ```
7. Exit and reboot.

This works because you modify the installed system, not the temporary initramfs environment, and you ensure SELinux relabeling happens on the next boot.

**Scenario 2: Service fails after a storage change**

A database service stops after a new disk is added.

1. Check whether the new filesystem is mounted:
   ```bash
   findmnt /var/lib/mysql
   ```
2. Inspect `/etc/fstab` for the new UUID:
   ```bash
   blkid
   cat /etc/fstab
   ```
3. Test the mount:
   ```bash
   mount -a
   ```
4. Check service logs:
   ```bash
   journalctl -u mariadb -b
   ```
5. Verify permissions and labels:
   ```bash
   ls -ldZ /var/lib/mysql
   ```

If the mount is missing, the service may be writing to an empty directory on the root filesystem instead of the intended data volume.

**Scenario 3: Web content works locally but not remotely**

1. Confirm Apache is running:
   ```bash
   systemctl status httpd
   ```
2. Confirm the socket is listening on the network:
   ```bash
   ss -tulpn | grep :80
   ```
3. Check the firewall zone:
   ```bash
   firewall-cmd --get-active-zones
   firewall-cmd --zone=public --list-services
   ```
4. Check SELinux labels:
   ```bash
   ls -Z /var/www/html
   ausearch -m avc -ts recent
   ```
5. Fix the missing layer:
   - Open the firewall if the port is blocked.
   - Correct the bind address if the service listens only on localhost.
   - Restore labels if SELinux denies access.

**Scenario 4: SSH access fails after a config change**

1. Verify the daemon is active:
   ```bash
   systemctl status sshd
   ```
2. Check whether it is listening:
   ```bash
   ss -tulpn | grep :22
   ```
3. Confirm the firewall allows SSH:
   ```bash
   firewall-cmd --list-services
   ```
4. Review recent logs:
   ```bash
   journalctl -u sshd -b
   ```
5. If you edited `/etc/ssh/sshd_config`, validate syntax before restarting:
   ```bash
   sshd -t
   ```

**Practical troubleshooting habits**

- Keep a terminal open for logs while you test changes.
- Change one thing at a time.
- Reproduce the failure after each fix.
- Save working commands for later reuse.
- Verify persistence when the exam expects a permanent solution.

**Useful command set for RHCSA troubleshooting**

```bash
# System and boot
journalctl -b
systemctl status <unit>
systemctl cat <unit>
systemctl is-enabled <unit>

# Storage
lsblk -f
blkid
mount
findmnt
mount -a
fsck -f /dev/sdXn
xfs_repair /dev/sdXn

# Network
nmcli device status
nmcli connection show
ip addr
ip route
ss -tulpn
ping -c 3 <host>

# Security
getenforce
ls -Z
ausearch -m avc -ts recent
restorecon -Rv <path>
firewall-cmd --get-active-zones
firewall-cmd --list-all
```

**What to remember for the exam**

- Start with the layer closest to the symptom.
- Confirm hardware visibility before changing software.
- Use `journalctl`, `systemctl`, `nmcli`, `ss`, `firewall-cmd`, and SELinux tools as your core toolkit.
- For boot recovery, remember `mount -o remount,rw /sysroot`, `chroot /sysroot`, and `touch /.autorelabel` when needed.
- For filesystem repair, choose the correct tool for the filesystem type.
- For service access problems, distinguish between bind address, firewall, permissions, and SELinux.

A disciplined workflow is often faster than guessing. On RHCSA, the best troubleshooting skill is not knowing every possible failure; it is knowing how to prove which layer is broken.
