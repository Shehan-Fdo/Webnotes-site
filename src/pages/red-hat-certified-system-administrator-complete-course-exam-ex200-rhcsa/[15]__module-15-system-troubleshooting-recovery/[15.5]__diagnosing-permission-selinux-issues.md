## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

Permission problems on RHEL usually come from one of four layers: traditional Unix mode bits, ownership, ACLs (Access Control Lists), and SELinux. You diagnose them in that order because the simplest layer often explains the failure fastest. A service can look “broken” even when the file exists and the user can read it, because a parent directory lacks execute permission, an ACL mask removes access, or SELinux blocks the operation.

The key exam skill is not memorizing one command. It is building a repeatable workflow:
- Identify the exact user, process, file, and action.
- Check path traversal and basic mode bits.
- Inspect ACLs and default ACL inheritance.
- Verify SELinux context, booleans, and audit denials.
- Apply the smallest persistent fix that matches the cause.

**Unix Permissions and Ownership**

Traditional permissions control access through user, group, and other mode bits. You inspect them with `ls -l` and `stat`, and you change them with `chmod`, `chown`, and `chgrp`.

```bash
ls -l /srv/app/config.ini
stat /srv/app/config.ini
```

Typical output shows:
- **Owner**: the user who owns the file.
- **Group**: the group associated with the file.
- **Mode bits**: read, write, and execute permissions for user, group, and other.

Use `stat` when you need precise numeric and symbolic details, especially for troubleshooting scripts or verifying a change:
```bash
stat -c '%A %a %U %G %n' /srv/app/config.ini
```

Important points:
- **Read** on a file allows viewing contents.
- **Write** on a file allows modifying contents.
- **Execute** on a file allows running it as a program or script.
- **Execute** on a directory means traversal; without it, you cannot access entries inside even if you know the filename.

> A common mistake is checking the target file only. If any parent directory lacks execute permission, access fails before the file is even reached.

**Directory Traversal Troubleshooting**

When a path fails, check every directory in the path, not just the final object. A user needs execute permission on each directory component to traverse it.

```bash
namei -l /srv/app/private/report.txt
```

`namei -l` is especially useful because it shows permissions on each path element. If `namei` is unavailable, inspect each directory manually:
```bash
ls -ld /srv /srv/app /srv/app/private
```

Troubleshooting pattern:
1. Confirm the file exists.
2. Check the final file permissions.
3. Check every parent directory for execute permission.
4. Check ownership and group membership.
5. Check ACLs and SELinux if Unix permissions look correct.

**ACLs: Access Control Lists**

ACLs provide finer-grained access than standard mode bits. They are common when one user needs access without changing the file’s group ownership.

Use `getfacl` to inspect ACLs:
```bash
getfacl /srv/shared/project.txt
```

Use `setfacl` to grant access:
```bash
setfacl -m u:alice:rw /srv/shared/project.txt
setfacl -m g:devteam:rwx /srv/shared
```

Key ACL concepts:
- **Access ACL** applies to a specific file or directory.
- **Default ACL** on a directory is inherited by new files and subdirectories.
- **Mask** limits the effective permissions for named users and groups.

Example of a default ACL on a directory:
```bash
setfacl -m d:u:alice:rwx /srv/shared
setfacl -m d:g:devteam:rwx /srv/shared
```

If inheritance seems broken, inspect the default ACL and the mask:
```bash
getfacl /srv/shared
```

A common real-world issue is that a user appears to have been granted access, but the ACL mask reduces the effective rights. For example, a named user may show `rw-`, while the mask is `r--`, so write access still fails.

**ACL Mask and Effective Permissions**

The ACL mask is often the hidden cause of “permission denied” even when the ACL entry looks correct. The mask affects named users, named groups, and the owning group.

Example:
```bash
# file: /srv/shared/report.txt
# owner: root
# group: devteam
user::rw-
user:alice:rw-
group::r--
mask::r--
other::---
```

In this case, `alice` does not effectively have write access because the mask is `r--`.

To fix it, adjust the ACL so the mask allows the needed rights:
```bash
setfacl -m m::rw /srv/shared/report.txt
```

If you want to remove ACLs and return to mode-bit-only behavior:
```bash
setfacl -b /srv/shared/report.txt
setfacl -k /srv/shared
```

- `-b` removes access ACL entries.
- `-k` removes default ACLs from a directory.

**SELinux Fundamentals**

SELinux (Security-Enhanced Linux) adds mandatory access control on top of Unix permissions. Even when mode bits and ACLs allow access, SELinux can still deny the action.

Check SELinux status:
```bash
getenforce
sestatus
```

Common modes:
- **Enforcing**: SELinux policy is active and blocks unauthorized actions.
- **Permissive**: SELinux logs denials but does not block them.
- **Disabled**: SELinux is off.

For exam work, you usually leave SELinux enabled and fix the label, boolean, or policy issue rather than disabling it.

**SELinux Contexts and Labels**

SELinux uses security contexts with four fields:
- **User**
- **Role**
- **Type**
- **Level**

The **type** is the most important field for everyday administration. You inspect labels with `ls -Z` or `ls -lZ`.

```bash
ls -Z /srv/web/index.html
ls -lZ /srv/web/index.html
```

Use `ls -Z` when you only need the context. Use `ls -lZ` when you also need ownership and mode bits. That distinction matters during troubleshooting because a file can have the right Unix permissions but the wrong SELinux type.

Example:
```bash
-rw-r--r--. root root unconfined_u:object_r:default_t:s0 /srv/web/index.html
```

A file in a web content directory should usually have a web content type such as `httpd_sys_content_t`, not `default_t`. The exact expected type depends on the service and path.

**Temporary vs Persistent SELinux Label Changes**

Use `chcon` for a temporary label change. It is useful for quick testing, but it is not persistent across relabeling or some restore operations.

```bash
chcon -t httpd_sys_content_t /srv/web/index.html
```

Use `semanage fcontext` plus `restorecon` for a persistent fix. `semanage fcontext` defines the file-context mapping, and `restorecon` applies the label from those file-context rules.

```bash
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
restorecon -Rv /srv/web
```

Important distinction:
- `chcon` changes the label directly on the file.
- `semanage fcontext` changes the rule used to label matching paths.
- `restorecon` restores labels according to those rules.

> Do not rely on `chcon` for a permanent solution. It is easy to lose after relabeling or policy-driven restoration.

**SELinux Booleans**

Booleans toggle specific policy behavior without changing labels. They are useful when a service is blocked by a policy feature that is intentionally optional.

List booleans:
```bash
getsebool -a | grep httpd
semanage boolean -l | grep httpd
```

Inspect one boolean:
```bash
getsebool httpd_enable_homedirs
```

Set a boolean persistently:
```bash
setsebool -P httpd_enable_homedirs on
```

Use booleans when the service needs a policy exception that is already supported by SELinux. For example, a web server may need access to home directories or network connections depending on the role.

**Reading SELinux Denials**

When SELinux blocks access, the audit log records an AVC (Access Vector Cache) denial. The most direct tools are `ausearch`, `sealert`, and the audit logs under `/var/log/audit/audit.log`.

Search recent denials:
```bash
ausearch -m AVC,USER_AVC -ts recent
```

Review the audit log directly:
```bash
grep AVC /var/log/audit/audit.log
```

If `setroubleshoot` is installed, you may also inspect its messages, but it is not the primary source for AVC records.

Use `sealert` when available:
```bash
sealert -a /var/log/audit/audit.log
```

Interpretation tips:
- Look for the **source context** and **target context**.
- Identify the **denied permission** such as `read`, `write`, `name_connect`, or `append`.
- Check whether the denial is caused by a wrong file type, missing boolean, or an unsupported action.

**audit2why and audit2allow**

`audit2why` explains why a denial happened. `audit2allow` can generate policy rules from denials, but you should use it carefully.

```bash
ausearch -m AVC,USER_AVC -ts recent | audit2why
ausearch -m AVC,USER_AVC -ts recent | audit2allow -M localfix
```

Then review the generated policy before installing it:
```bash
cat localfix.te
semodule -i localfix.pp
```

Use these tools as diagnostic aids, not as the first fix. For RHCSA-style tasks, the expected solution is usually to correct labels, booleans, or file placement rather than create custom policy.

**End-to-End SELinux Example: Web Content Write Denial**

Suppose a web application must write uploads into `/srv/web/uploads`, but the service gets denied.

1. Check the file and directory labels:
```bash
ls -ldZ /srv/web /srv/web/uploads
```

2. Inspect recent AVC denials:
```bash
ausearch -m AVC,USER_AVC -ts recent
```

3. Confirm the service context and the denied action. If the denial shows the web service trying to write to a directory labeled for read-only content, the type is likely wrong.

4. Apply the correct persistent label:
```bash
semanage fcontext -a -t httpd_sys_rw_content_t '/srv/web/uploads(/.*)?'
restorecon -Rv /srv/web/uploads
```

5. If the service also needs a policy exception, inspect booleans:
```bash
getsebool -a | grep httpd
```

This workflow solves the root cause without weakening security.

**End-to-End SELinux Example: Non-Web Service File Creation Denial**

A custom service running as `myapp_t` may fail to create files in `/var/lib/myapp/data`.

Troubleshooting steps:
- Verify the directory label with `ls -ldZ`.
- Check the audit log for `create`, `write`, or `add_name` denials.
- Confirm whether the service expects a specific type such as `myapp_var_lib_t`.
- Apply a persistent file-context rule if the path is correct.

Example:
```bash
semanage fcontext -a -t myapp_var_lib_t '/var/lib/myapp(/.*)?'
restorecon -Rv /var/lib/myapp
```

If the service still fails, inspect whether the process domain itself is correct and whether the application is running under the expected systemd unit.

**ACL and SELinux Together**

ACLs and SELinux are independent. A user can be allowed by ACLs and still denied by SELinux, or vice versa. That is why you should not stop after one layer looks correct.

Example troubleshooting sequence:
- `ls -l` confirms mode bits.
- `getfacl` confirms ACLs.
- `ls -Z` confirms SELinux labels.
- `ausearch` confirms whether SELinux denied the action.

If a user cannot read a file:
```bash
ls -l /srv/shared/report.txt
getfacl /srv/shared/report.txt
ls -Z /srv/shared/report.txt
ausearch -m AVC,USER_AVC -ts recent
```

**Common Real-World Failure Patterns**

- **Missing execute on a directory**: the user can see the file name in documentation but cannot traverse the path.
- **ACL mask too restrictive**: the ACL entry looks correct, but effective permissions are reduced.
- **Wrong SELinux type**: the file is in the right place but labeled for a different service.
- **Boolean not enabled**: the service is allowed by policy only when a feature flag is on.
- **Temporary label fix only**: `chcon` works until the next relabel, then the issue returns.

**Practical Troubleshooting Workflow**

1. Reproduce the failure with the exact user or service.
2. Check path traversal with `namei -l`.
3. Check mode bits and ownership with `ls -l` and `stat`.
4. Check ACLs with `getfacl`.
5. Check SELinux mode with `getenforce` and labels with `ls -Z`.
6. Search AVC denials with `ausearch -m AVC,USER_AVC -ts recent`.
7. Apply the smallest persistent fix:
   - `chmod`, `chown`, or `chgrp` for Unix permissions.
   - `setfacl` for ACLs.
   - `semanage fcontext` plus `restorecon` for SELinux labels.
   - `setsebool -P` for supported policy toggles.

**Quick Command Reference**

```bash
# Basic permissions
ls -l
stat -c '%A %a %U %G %n' file
chmod 640 file
chown user:group file

# Path traversal
namei -l /path/to/file

# ACLs
getfacl file
setfacl -m u:alice:rw file
setfacl -m d:g:devteam:rwx /shared
setfacl -b file

# SELinux
getenforce
sestatus
ls -Z file
ls -lZ file
chcon -t httpd_sys_content_t file
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
restorecon -Rv /srv/web
getsebool -a | grep httpd
setsebool -P httpd_enable_homedirs on

# Audit and denials
ausearch -m AVC,USER_AVC -ts recent
grep AVC /var/log/audit/audit.log
audit2why
audit2allow -M localfix
```

**Exam Tips**

- Start with the simplest layer first: mode bits, then ACLs, then SELinux.
- Use `ls -Z` and `ls -lZ` deliberately; do not assume one replaces the other.
- Prefer persistent SELinux fixes with `semanage fcontext` and `setsebool -P`.
- Treat `chcon` as a temporary diagnostic tool.
- Verify the parent directories when a path fails unexpectedly.
- Remember that SELinux denials come from audit logs, not from ordinary permission errors.

**Summary**

Permission troubleshooting on RHCSA is about disciplined layering. Unix permissions control basic access, ACLs refine it, and SELinux enforces mandatory policy. The fastest path to a correct fix is to identify which layer is blocking the action, confirm it with the right command, and apply the smallest persistent change that matches the cause.
