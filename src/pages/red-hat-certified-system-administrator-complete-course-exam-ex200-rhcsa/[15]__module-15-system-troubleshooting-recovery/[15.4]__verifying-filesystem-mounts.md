**Filesystem mount verification overview**

A mounted filesystem is only useful if you can prove it is attached where you expect, with the options you expect, and that it survives reboot when required. On RHCSA systems, verification is a routine administrative task after creating partitions, formatting filesystems, editing `/etc/fstab`, or attaching network and removable storage.

Mount verification answers three practical questions:

- Is the filesystem mounted now?
- Is it mounted on the correct mount point with the correct type and options?
- Will it mount again after reboot without manual intervention?

**Core commands for checking mounts**

The most useful tools are `findmnt`, `mount`, `df`, `lsblk`, and `blkid`. Each shows a different view of the same storage state.

- `findmnt` is the best command for a tree-style view of mount relationships.
- `mount` shows active mounts and can also test or perform mounts.
- `df` shows mounted filesystems and space usage.
- `lsblk` shows block devices, partitions, and mount points.
- `blkid` shows UUIDs and filesystem types, which are important for `/etc/fstab`.

A quick verification workflow often looks like this:

```bash
findmnt
df -hT
lsblk -f
blkid
```

**Using `findmnt` effectively**

`findmnt` reads mount information from the kernel and presents it in a structured format. It is especially useful when you want to verify a specific mount point or compare source, target, filesystem type, and options.

Common checks:

```bash
findmnt /data
findmnt -t xfs
findmnt -o TARGET,SOURCE,FSTYPE,OPTIONS
```

Typical output fields include:

- **TARGET**: the mount point path
- **SOURCE**: the device, UUID, or remote source
- **FSTYPE**: filesystem type such as `xfs` or `ext4`
- **OPTIONS**: mount options such as `rw`, `relatime`, `noexec`, or `nodev`

Use `findmnt` when you need to confirm that a mount is attached exactly where expected, especially after editing `/etc/fstab`.

**Using `mount` to inspect active mounts**

The `mount` command without arguments prints the current mount table. This is useful for a quick check, but the output is less structured than `findmnt`.

```bash
mount | grep ' /data '
```

You can also verify a specific device or mount point:

```bash
mount /dev/sdb1 /mnt
mount | grep /mnt
```

Important points:

- `mount` shows what is currently active, not what is configured for the next boot.
- A successful manual mount does not guarantee `/etc/fstab` is correct.
- If a mount fails, the error message often points to a bad device name, missing directory, unsupported filesystem, or invalid option.

> A filesystem can be mounted manually and still fail at boot if `/etc/fstab` contains the wrong UUID, mount point, or option syntax.

**Checking usage with `df`**

`df` reports mounted filesystems and their space usage. It is useful for confirming that the filesystem is mounted and that the expected capacity is visible.

```bash
df -hT
df -hT /data
```

Useful flags:

- `-h` shows human-readable sizes
- `-T` includes filesystem type

`df` is best for answering:

- Is the filesystem mounted?
- How much space is available?
- Does the filesystem type match expectations?

It is not the best tool for checking mount options or mount source details.

**Inspecting device and filesystem identity with `lsblk` and `blkid`**

`lsblk` shows the relationship between disks, partitions, and mount points. The `-f` option adds filesystem type, label, and UUID information.

```bash
lsblk
lsblk -f
```

Example output fields to verify:

- **NAME**: device name such as `sdb1`
- **FSTYPE**: filesystem type
- **LABEL**: filesystem label, if present
- **UUID**: unique identifier used in `/etc/fstab`
- **MOUNTPOINTS**: current mount location

`blkid` is especially useful when preparing `/etc/fstab` entries:

```bash
blkid /dev/sdb1
```

Example:

```bash
/dev/sdb1: UUID="3f2d1c7a-9c2a-4d5f-8f1a-2b7d9c1e4a11" TYPE="xfs"
```

Use UUIDs in `/etc/fstab` when possible because they are more stable than device names like `/dev/sdb1`, which can change across boots.

**Verifying `/etc/fstab` entries**

Persistent mounts depend on correct `/etc/fstab` syntax. A valid entry typically includes the device identifier, mount point, filesystem type, mount options, dump field, and fsck pass field.

Example:

```fstab
UUID=3f2d1c7a-9c2a-4d5f-8f1a-2b7d9c1e4a11  /data  xfs  defaults  0  0
```

Verify the file carefully:

- The mount point directory exists.
- The filesystem type matches the actual filesystem.
- The UUID or label matches the real device.
- The options are valid for that filesystem.
- The final two numeric fields are set appropriately.

A common RHCSA-safe validation method is to test the entry without rebooting:

```bash
mount -a
findmnt /data
```

If `mount -a` returns no error, the syntax is usually acceptable, but you should still confirm the mount with `findmnt` or `df`.

**Practical verification procedure**

When you create or modify a mount, verify it in this order:

1. Confirm the device and filesystem identity.
   ```bash
   lsblk -f
   blkid /dev/sdb1
   ```
2. Confirm the mount point exists.
   ```bash
   ls -ld /data
   ```
3. Mount the filesystem or reload mounts from `/etc/fstab`.
   ```bash
   mount /data
   # or
   mount -a
   ```
4. Verify the active mount.
   ```bash
   findmnt /data
   df -hT /data
   ```
5. Confirm persistence by checking `/etc/fstab` syntax and source values.
   ```bash
   cat /etc/fstab
   ```

**Mount options to verify**

Mount options affect behavior and security. Always confirm that the active options match the intended configuration.

Common options include:

- `defaults`: standard read-write behavior with typical defaults
- `ro`: read-only
- `rw`: read-write
- `noexec`: prevents execution of binaries from the mount
- `nodev`: ignores device files on the mount
- `nosuid`: ignores setuid and setgid bits
- `relatime`: updates access times in a reduced way

Check options with:

```bash
findmnt -o TARGET,SOURCE,FSTYPE,OPTIONS /data
```

This is important when you are hardening a mount or troubleshooting why a program cannot run from a filesystem.

**Troubleshooting failed mounts**

When a mount does not appear as expected, isolate the cause systematically.

Common causes:

- The mount point directory does not exist.
- The device name is wrong or the disk is missing.
- The filesystem type in `/etc/fstab` is incorrect.
- The UUID in `/etc/fstab` does not match the device.
- The mount options contain a typo.
- The filesystem is corrupted or needs repair.
- The device is already mounted elsewhere.

Useful commands:

```bash
journalctl -b | grep -i mount
dmesg | tail -n 20
mount -av
findmnt -R /data
```

`mount -av` is especially helpful because it attempts all entries in `/etc/fstab` and prints verbose status. If one entry fails, the output often identifies the problem line.

> Be careful when editing `/etc/fstab`: a bad entry can delay boot or drop the system into emergency mode if it is required for startup.

**Verifying remote and special filesystems**

Not all mounts are local disks. NFS, CIFS, and pseudo-filesystems such as `proc`, `sysfs`, and `tmpfs` also appear in mount checks.

Examples:

```bash
findmnt -t nfs
findmnt -t tmpfs
df -hT | grep -E 'nfs|tmpfs'
```

For remote mounts, verify:

- The server name or export path is correct.
- Network connectivity exists.
- Required services are reachable.
- Authentication details, if any, are valid.

A remote mount may be configured correctly but still fail because the network is unavailable at boot.

**Useful verification patterns**

A few command combinations cover most exam and admin scenarios:

```bash
findmnt /data
lsblk -f
blkid
mount -a
df -hT /data
```

For a mount that must persist across reboot, verify both the live state and the configuration state:

- Live state: `findmnt`, `mount`, `df`
- Configuration state: `/etc/fstab`, `blkid`, `lsblk -f`

**Example `/etc/fstab` workflow**

Suppose you add a new XFS filesystem on `/dev/sdb1` and want it mounted at `/data`.

```bash
mkdir -p /data
blkid /dev/sdb1
```

Add the entry:

```fstab
UUID=3f2d1c7a-9c2a-4d5f-8f1a-2b7d9c1e4a11  /data  xfs  defaults  0  0
```

Test it:

```bash
mount -a
findmnt /data
df -hT /data
```

If the mount fails, inspect the exact line in `/etc/fstab`, compare the UUID, and check the journal for the error message.

**Exam-focused takeaways**

For RHCSA tasks, you should be able to:

- Identify mounted filesystems quickly.
- Confirm mount points, filesystem types, and options.
- Validate `/etc/fstab` entries before reboot.
- Use UUIDs and filesystem labels appropriately.
- Troubleshoot mount failures with `mount -a`, `findmnt`, and logs.

The safest habit is to verify both the active mount and the persistent configuration every time you change storage.
