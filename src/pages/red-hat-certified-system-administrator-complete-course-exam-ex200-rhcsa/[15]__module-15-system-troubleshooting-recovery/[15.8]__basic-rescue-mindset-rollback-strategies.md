## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Rescue mindset: stabilize first, then repair**

A good rescue workflow starts with containment. Before you change anything, identify the failure domain, preserve evidence when possible, and choose the least invasive fix that restores access. On RHCSA systems, that usually means checking boot targets, storage visibility, mount state, SELinux context, and package integrity before you attempt deeper recovery.

The practical goal is to get the system back to a known-good state without making the problem worse. That means you should prefer reversible changes, verify each step, and avoid “shotgun” fixes that alter multiple variables at once.

- **Assess scope**: determine whether the issue is boot, authentication, storage, networking, or package-related.
- **Minimize change**: edit the smallest number of files and services needed to regain control.
- **Verify early**: test mounts, services, and configuration syntax before rebooting.
- **Document state**: note what you changed so you can undo it if needed.

**Common rescue entry points**

You often recover a system from one of these states:

- **Single-user or emergency mode**: the system boots, but normal multi-user services are unavailable.
- **rd.break / initramfs shell**: you need to repair the root filesystem, reset credentials, or fix boot issues before the real root mounts.
- **Live media or rescue environment**: the installed system will not boot far enough to provide a shell.
- **Chroot into the installed system**: you need to run tools as if you were inside the broken OS.

A rescue shell is not the same as a fully booted system. Device names, mount points, SELinux state, and network availability may differ, so always confirm your environment before applying fixes.

**Boot and access recovery workflow**

When the system fails to boot normally, work from the outside in:

1. Identify whether the failure occurs before the kernel loads, during initramfs, during mount of the root filesystem, or after `systemd` starts.
2. Check whether the root filesystem is accessible and writable.
3. Mount the installed system under a temporary mount point if needed.
4. Chroot into the installed system when you need to run package, bootloader, or configuration tools against the real root.
5. Rebuild or repair only the components that are actually broken.
6. Exit, unmount cleanly, and reboot.

A typical rescue mount sequence from a live environment looks like this:

```bash
lsblk -f
mount /dev/mapper/rhel-root /mnt/sysroot
mount /dev/sda1 /mnt/sysroot/boot
mount /dev/sda2 /mnt/sysroot/boot/efi
for i in /dev /proc /sys /run; do mount --bind $i /mnt/sysroot$i; done
chroot /mnt/sysroot
```

If the system uses LVM, encrypted storage, or separate `/boot` and EFI partitions, you must activate and mount those components before chrooting.

**Password reset and SELinux relabel flow**

A common RHCSA recovery task is resetting the root password from `rd.break`. The key idea is that you are repairing the installed system, not the temporary initramfs shell.

1. Boot with `rd.break` appended to the kernel command line.
2. Remount the real root filesystem read-write.
3. Chroot into the installed system.
4. Change the password.
5. Create `/.autorelabel` inside the chrooted installed system if SELinux relabeling is needed.
6. Exit, reboot, and allow relabeling to complete.

Example:

```bash
mount -o remount,rw /sysroot
chroot /sysroot
passwd root
touch /.autorelabel
exit
reboot
```

> `/.autorelabel` must be created in the installed system’s root filesystem after you chroot into `/sysroot`. If you skip the chroot, you may place the file in the wrong root and the relabel will not happen as intended.

**Bootloader recovery concepts**

Bootloader repair depends on firmware type.

- **BIOS systems**: GRUB is typically installed to the disk MBR or BIOS boot area, and `grub2-install` is a common recovery step.
- **UEFI systems**: GRUB is installed to the EFI System Partition, and recovery usually involves mounting the EFI partition, reinstalling the GRUB EFI package if needed, and regenerating configuration.

For UEFI, the repair path is usually closer to this:

```bash
mount /dev/sda2 /boot/efi
chroot /mnt/sysroot
dnf reinstall grub2-efi-x64 shim-x64
grub2-mkconfig -o /boot/efi/EFI/redhat/grub.cfg
```

On some systems, the exact EFI directory name may differ. Verify the vendor path under `/boot/efi/EFI/` before writing configuration files.

For BIOS-style recovery, the flow is different:

```bash
chroot /mnt/sysroot
grub2-install /dev/sda
grub2-mkconfig -o /boot/grub2/grub.cfg
```

Use the correct method for the firmware mode. A BIOS recovery command is not a universal fix for UEFI systems.

**Rollback strategies for packages and configuration**

Rollback is about returning to a previous known state, but it is not magic. Package history and filesystem snapshots are different tools with different guarantees.

- **Package manager history**: useful when a recent `dnf` transaction introduced a problem.
- **Configuration backups**: useful when a manual edit broke a service.
- **Filesystem snapshots**: useful when your storage stack supports them and you need a fast revert.
- **Immutable backups**: useful when you need a clean restore point after a major change.

`dnf history` lets you inspect and reverse package transactions:

```bash
dnf history
dnf history info <ID>
dnf history undo <ID>
dnf history rollback <ID>
```

Use these carefully:

- `dnf history undo <ID>` attempts to reverse a specific transaction.
- `dnf history rollback <ID>` tries to return the system to the state before the chosen transaction ID, but later changes, dependency shifts, or repository differences can affect the result.
- Neither command guarantees a perfect byte-for-byte return to the exact prior package set.

A practical example:

```bash
dnf history
dnf history info 42
dnf history rollback 42
```

If rollback fails, inspect the transaction details and resolve dependency conflicts manually. In exam scenarios, the important skill is understanding that package rollback is best-effort and may require follow-up verification.

**Configuration rollback habits**

For text-based configuration, the safest rollback strategy is to keep a known-good copy before editing.

- Copy the original file before changes:
  ```bash
  cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak
  ```
- Validate syntax before restart:
  ```bash
  sshd -t
  ```
- Restore the backup if validation fails:
  ```bash
  cp /etc/ssh/sshd_config.bak /etc/ssh/sshd_config
  systemctl restart sshd
  ```

For services managed by `systemd`, always check the unit status after a change:

```bash
systemctl status sshd
journalctl -u sshd -b
```

This reduces the chance that a bad edit survives until the next reboot.

**Filesystem and mount recovery**

Mount problems are common because a bad `/etc/fstab` entry can prevent normal boot or leave a system partially available.

Start by checking the current mount table and the filesystem UUIDs:

```bash
lsblk -f
blkid
cat /etc/fstab
mount -a
```

`mount -a` is useful because it validates the entries that can be mounted in the current environment, but it does not guarantee that every mount will succeed at boot. Network timing, device discovery, and systemd mount ordering can still cause boot-time failures.

If `mount -a` reports no errors, that is a good sign, but you should still verify:

- the device exists at boot time,
- the UUID or label is correct,
- the filesystem type matches,
- any network dependency is available,
- the mount point exists and has the right permissions.

A safer workflow for `/etc/fstab` edits is:

1. Back up `/etc/fstab`.
2. Edit one line at a time.
3. Test with `mount -a`.
4. Confirm the mount is present with `findmnt`.
5. Reboot only after the current environment behaves as expected.

Example:

```bash
cp /etc/fstab /etc/fstab.bak
vi /etc/fstab
mount -a
findmnt /data
```

If a bad `fstab` entry blocks boot, you may need to boot into rescue mode, comment out the offending line, and regenerate the boot environment if required.

**SELinux recovery mindset**

SELinux issues often look like application failures, but the fix may be a context problem rather than a service problem.

Useful checks:

```bash
getenforce
ls -Z /var/www/html
ausearch -m AVC -ts recent
journalctl -t setroubleshoot --since "1 hour ago"
```

If you change file locations or restore files from backup, contexts may be wrong. In that case, relabeling or restoring default contexts is often safer than disabling SELinux.

Common repair commands:

```bash
restorecon -Rv /var/www/html
touch /.autorelabel
```

Use `restorecon` for targeted fixes and `/.autorelabel` when a broader relabel is appropriate after a major recovery.

**Service recovery and verification**

After any repair, verify the service path end to end.

- Check syntax before restart when the tool supports it.
- Restart the service only after the configuration is valid.
- Confirm the service is active and listening if applicable.
- Review logs for the exact failure reason.

Examples:

```bash
nginx -t
systemctl restart nginx
systemctl status nginx
ss -tulpn | grep :80
journalctl -u nginx -b
```

For SSH, web, and storage-related services, the same pattern applies: validate, restart, inspect logs, and confirm the expected behavior.

**Practical rescue checklist**

Use this checklist when you face an unfamiliar failure:

1. Identify the failure stage.
2. Preserve the current state with backups or notes.
3. Confirm storage visibility with `lsblk`, `blkid`, and `findmnt`.
4. Check logs with `journalctl` and service-specific tools.
5. Repair the smallest broken component first.
6. Validate configuration syntax before restarting services.
7. Use `dnf history` only when the issue clearly follows a package transaction.
8. Reboot only after the system is stable in the current session.

**What to remember for the exam**

- **Rescue mode** is for controlled repair, not random experimentation.
- **Chroot** lets you operate on the installed system from a rescue environment.
- **UEFI and BIOS** boot repairs are different; choose the correct bootloader method.
- **Rollback** is best-effort unless you have snapshots or backups.
- **`mount -a`** checks current mount entries, but it does not prove boot success.
- **SELinux relabeling** is often required after password resets or major file restoration.

A disciplined rescue approach is one of the most valuable RHCSA skills because it combines diagnosis, safe change management, and verification. The exam rewards candidates who can restore service without introducing new problems.
