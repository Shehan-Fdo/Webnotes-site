## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Boot targets and recovery overview**

On Red Hat Enterprise Linux (RHEL), `systemd` controls boot targets, service startup, and recovery modes. For RHCSA tasks, you need to know how to inspect the current target, change the default target persistently, switch targets in a running session, and recover a system that fails to boot or starts the wrong services.

A **target** is a `systemd` synchronization point that groups units. Common targets map to familiar runlevels:

- `multi-user.target` for text-mode, multi-user operation
- `graphical.target` for graphical login
- `rescue.target` for single-user-style maintenance with more services than emergency mode
- `emergency.target` for the most minimal maintenance shell

Use target management when you need to change the system’s normal boot behavior. Use recovery modes when the system cannot boot normally or you need to repair configuration from a minimal environment.

**Inspecting the current and default target**

The default target is what the system tries to reach at boot. The currently active target may be different if you manually switched modes after boot.

```bash
systemctl get-default
systemctl list-units --type=target
systemctl status default.target
```

Useful checks:

- `systemctl get-default` shows the persistent default target.
- `systemctl list-units --type=target` shows active targets in the current session.
- `systemctl status default.target` helps confirm what `default.target` points to.
- `systemctl is-system-running` gives a quick health summary of the booted system.

To verify the current boot state more precisely, inspect the active target:

```bash
systemctl list-units --type=target --state=active
```

**Changing the default boot target persistently**

Use `systemctl set-default` to change the target used on the next boot and all future boots until changed again.

```bash
sudo systemctl set-default multi-user.target
sudo systemctl set-default graphical.target
```

This command updates the `default.target` symlink, typically under `/etc/systemd/system/`. It is the correct persistent method for changing the normal boot target.

To confirm the change:

```bash
systemctl get-default
readlink -f /etc/systemd/system/default.target
```

If you need to manage the target manually, you can also create or adjust the symlink yourself, but `systemctl set-default` is preferred because it is explicit and less error-prone.

> Do not confuse the default target with a one-time boot override. `systemctl set-default` changes persistent boot behavior; it does not affect only the current boot.

**Switching targets in a running system**

If the system is already booted and you want to move to another target immediately, use `systemctl isolate`.

```bash
sudo systemctl isolate multi-user.target
sudo systemctl isolate graphical.target
```

`isolate` stops units that are not part of the target and starts the units required by that target. This is useful for maintenance, but it can interrupt services and user sessions.

Common use cases:

- Move from graphical mode to text mode for troubleshooting.
- Enter rescue-like operation from a running system without rebooting.
- Test whether a target transition works before making it the default.

**Temporary boot-time target changes with GRUB**

When the system cannot boot normally, you may need to edit the kernel command line in GRUB to boot once into a different target. This is a temporary change unless you also update the persistent default target.

At the GRUB menu:

1. Highlight the desired kernel entry.
2. Press `e` to edit.
3. Find the line that starts with `linux` or `linuxefi`.
4. Append a boot argument such as `systemd.unit=rescue.target` or `systemd.unit=emergency.target`.
5. Boot with <kbd>Ctrl</kbd> + <kbd>x</kbd> or <kbd>F10</kbd>.

Example kernel line:

```text
linux /vmlinuz-5.x.x-... root=/dev/mapper/rhel-root ro crashkernel=auto quiet systemd.unit=rescue.target
```

Use the correct target for the situation:

- `systemd.unit=rescue.target` for a maintenance shell with more services available.
- `systemd.unit=emergency.target` for the smallest possible environment.

> `systemd.unit=` is a boot-time kernel argument. It changes only that boot unless you make a separate persistent configuration change.

**When to use rescue vs emergency**

`rescue.target` starts a minimal multi-user environment and mounts local filesystems needed for maintenance. It is usually the first choice when you need to repair the system but still want access to more tools and mounted filesystems.

`emergency.target` starts the most minimal environment. It typically gives you a root shell with the root filesystem mounted read-only, and you may need to remount it read-write before making changes.

Typical workflow in emergency mode:

```bash
mount -o remount,rw /
```

Then inspect and repair the system:

```bash
ls /etc
cat /etc/fstab
journalctl -xb
```

Use emergency mode when:

- Boot fails very early.
- `fstab` errors prevent normal startup.
- You need the smallest possible environment for repair.

Use rescue mode when:

- You need a shell with more services available.
- You want local filesystems mounted for easier troubleshooting.
- You need to repair a service or configuration without full normal boot.

**Recovering from a failed boot target or bad default target**

If the system boots into the wrong target or fails after a target change, first determine whether the issue is persistent or temporary.

1. Check the current default target:
   ```bash
   systemctl get-default
   ```
2. Check the active target and boot state:
   ```bash
   systemctl list-units --type=target --state=active
   systemctl is-system-running
   ```
3. If the system is already up, switch to a working target:
   ```bash
   sudo systemctl isolate multi-user.target
   ```
4. If the system is not booting correctly, edit GRUB and boot once with:
   ```text
   systemd.unit=rescue.target
   ```
5. After repair, set the correct persistent default:
   ```bash
   sudo systemctl set-default multi-user.target
   ```

If you need to change the persistent default from a rescue shell, use `systemctl set-default` after the system is mounted and available. If you are in a chroot environment during recovery, make sure you are operating on the installed system’s root, not the live environment.

**GRUB and persistent boot configuration**

For RHCSA-level recovery, distinguish between:

- **Persistent target selection** with `systemctl set-default`
- **One-time boot override** with `systemd.unit=...`
- **Bootloader configuration changes** with GRUB tools

If you change GRUB defaults or kernel arguments permanently, regenerate the GRUB configuration when required by the platform.

Common RHEL 8/9 BIOS path:

```bash
sudo grub2-mkconfig -o /boot/grub2/grub.cfg
```

Common UEFI path:

```bash
sudo grub2-mkconfig -o /boot/efi/EFI/redhat/grub.cfg
```

Use the correct output path for the system firmware type. If you only need a one-time boot into rescue or emergency mode, you do not need to regenerate GRUB.

**Service recovery workflow**

When a service fails, do not restart it blindly. First identify the failure, inspect logs, fix the root cause, then restart and verify.

A practical workflow:

1. Check service status:
   ```bash
   systemctl status sshd.service
   ```
2. Inspect recent logs:
   ```bash
   journalctl -xeu sshd.service
   ```
3. Validate configuration if the service supports it:
   ```bash
   sshd -t
   ```
4. Fix the configuration error.
5. Restart the service:
   ```bash
   sudo systemctl restart sshd.service
   ```
6. Confirm it is active:
   ```bash
   systemctl is-active sshd.service
   ```

This pattern applies to many services, not just SSH. The exact validation command depends on the daemon.

**Example: recovering SSH after a configuration mistake**

If `sshd` fails because of a bad config file, you can diagnose and repair it safely.

```bash
sudo sshd -t
```

If the command reports an error, edit `/etc/ssh/sshd_config` and correct the issue. Then test again:

```bash
sudo sshd -t
sudo systemctl restart sshd.service
sudo systemctl status sshd.service
```

If the service still fails, inspect logs:

```bash
journalctl -xeu sshd.service
```

Common SSH recovery checks:

- Confirm `PermitRootLogin`, `PasswordAuthentication`, and `Port` values are valid.
- Ensure the firewall allows the chosen port.
- Verify host keys exist under `/etc/ssh/`.
- Check SELinux denials if the service starts but connections fail.

**Example: recovering a web or custom service**

For a service that reads a configuration file, use the service’s own test mode if available.

```bash
sudo systemctl status httpd.service
sudo journalctl -xeu httpd.service
sudo apachectl configtest
sudo systemctl restart httpd.service
```

For a custom unit, inspect the unit file and logs:

```bash
systemctl cat myapp.service
journalctl -u myapp.service --since today
```

Then correct the underlying issue, such as:

- wrong file path
- missing permissions
- bad syntax in the config file
- port already in use
- dependency not started

**Working with rescue and emergency shells**

In recovery shells, your filesystem state matters.

- In `rescue.target`, local filesystems are usually mounted, so you can often edit files directly.
- In `emergency.target`, the root filesystem is commonly mounted read-only, so remount it read-write before editing:
  ```bash
  mount -o remount,rw /
  ```

If the system uses separate filesystems such as `/var`, `/home`, or `/boot`, verify they are mounted before changing files there:

```bash
mount -a
findmnt
```

If `mount -a` reports errors, inspect `/etc/fstab` carefully. A bad `fstab` entry is a common reason for boot failure.

**Root password recovery with `rd.break`**

If you need to reset the root password on a RHEL system, use the initramfs break method.

1. At the GRUB menu, edit the kernel line.
2. Append:
   ```text
   rd.break
   ```
3. Boot into the break shell.
4. Remount the installed system root read-write:
   ```bash
   mount -o remount,rw /sysroot
   ```
5. Chroot into the installed system:
   ```bash
   chroot /sysroot
   ```
6. Set a new root password:
   ```bash
   passwd root
   ```
7. If SELinux relabeling is needed because file contexts may have been affected during recovery, create the relabel flag inside the installed system root:
   ```bash
   touch /.autorelabel
   ```
8. Exit the chroot and continue boot:
   ```bash
   exit
   exit
   ```

Important details:

- Create `/.autorelabel` **inside the chroot** so it exists in the installed system.
- A root password reset alone does not normally require a full relabel.
- Use the relabel flag when recovery actions may have changed SELinux labels or when the system needs a clean relabel on next boot.

**Verifying and documenting the fix**

After recovery, verify both the target and the service state.

```bash
systemctl get-default
systemctl is-active sshd.service
systemctl is-enabled sshd.service
systemctl list-units --type=service --state=failed
```

Useful verification habits:

- Confirm the default target matches the intended boot mode.
- Confirm the service is active and enabled if it should start at boot.
- Check for failed units after repair.
- Reboot once if needed to confirm the persistent fix works.

**Quick reference commands**

```bash
systemctl get-default
systemctl set-default multi-user.target
systemctl set-default graphical.target
systemctl isolate rescue.target
systemctl isolate multi-user.target
systemctl status sshd.service
journalctl -xeu sshd.service
systemctl is-active sshd.service
systemctl is-enabled sshd.service
```

**Exam-focused distinctions**

- Use `systemctl set-default` for a persistent target change.
- Use `systemctl isolate` for a live session change.
- Use `systemd.unit=...` in GRUB for a one-time boot into rescue or emergency mode.
- Use `grub2-mkconfig` only when you need to regenerate bootloader configuration.
- Use `rd.break` for password recovery and low-level initramfs repair.
- Use `journalctl` and service-specific validation commands before restarting a failed service.

**Common mistakes to avoid**

- Adding `systemd.unit=` to a persistent kernel argument when you only meant to change the default target.
- Restarting a service without checking `systemctl status` and logs first.
- Forgetting to remount `/` read-write in emergency mode before editing files.
- Creating `/.autorelabel` in the wrong root during password recovery.
- Confusing a temporary boot override with a persistent target change.
