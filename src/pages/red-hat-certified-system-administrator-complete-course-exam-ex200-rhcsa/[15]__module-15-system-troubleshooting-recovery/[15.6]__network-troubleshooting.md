## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Network troubleshooting workflow**

Network problems are easiest to solve when you work from the bottom up: link, addressing, routing, name resolution, and finally application reachability. On RHCSA systems, most day-to-day issues involve NetworkManager, incorrect static settings, DHCP failures, DNS misconfiguration, or firewall and SELinux policy blocking traffic.

A practical workflow is:

1. Confirm the interface exists and is up.
2. Verify the IP address, route, and DNS settings.
3. Test local, remote, and name-based connectivity.
4. Check logs for NetworkManager, DHCP, and resolver errors.
5. Inspect firewall and SELinux only after basic network checks pass.

**Core commands for first-pass diagnosis**

Use these commands early because they quickly separate link, IP, and name-resolution problems.

```bash
ip link show
ip addr show
ip route show
nmcli device status
nmcli connection show
resolvectl status
```

- `ip link show` reveals interface flags such as `UP` and `LOWER_UP`.
- `ip addr show` confirms assigned IPv4 and IPv6 addresses.
- `ip route show` shows the default route and connected networks.
- `nmcli device status` tells you whether NetworkManager sees the device as connected, disconnected, or unmanaged.
- `resolvectl status` is useful on systems using `systemd-resolved`; on many RHEL systems, `/etc/resolv.conf` is still the primary resolver file managed by NetworkManager.

**Interface state and carrier**

An interface can be administratively up but still have no physical carrier. That distinction matters when a cable is unplugged, a virtual NIC is disconnected, or a switch port is down.

```bash
ip link show ens160
ethtool ens160
cat /sys/class/net/ens160/carrier
```

- `state UP` in `ip link` means the interface is enabled.
- `LOWER_UP` usually indicates the link layer reports carrier.
- `ethtool` can show negotiated speed, duplex, and link detection.
- `/sys/class/net/<iface>/carrier` returns `1` for carrier present and `0` for no carrier.

> Do not assume `UP` means the network is usable. You still need an address, a route, and working name resolution.

**NetworkManager basics**

RHCSA systems typically use **NetworkManager** to manage interfaces and profiles. A profile stores the intended configuration, while the device is the actual NIC.

Useful commands:

```bash
nmcli device status
nmcli connection show
nmcli connection show "System ens160"
nmcli connection up "System ens160"
nmcli connection down "System ens160"
```

- `nmcli device status` shows whether a device is connected and which profile is active.
- `nmcli connection show` lists saved profiles.
- `nmcli connection show "<name>"` displays the full profile configuration.
- `nmcli connection up` activates a profile.
- `nmcli connection down` deactivates it.

Common profile files are stored under:

```bash
/etc/NetworkManager/system-connections/
```

- These files are the persistent connection profiles used by NetworkManager.
- They are root-owned and usually readable only by root.
- If you inspect them directly, remember they may contain secrets such as Wi-Fi keys or VPN credentials.

Legacy systems may also contain references to:

```bash
/etc/sysconfig/network-scripts/
```

- On modern RHEL releases, this directory is legacy awareness rather than the primary management path.
- For RHCSA troubleshooting, focus on NetworkManager profiles first.

**Inspecting and editing connection profiles**

When a profile is wrong, inspect it before changing anything.

```bash
nmcli connection show "System ens160"
```

Look for:

- `ipv4.method`
- `ipv4.addresses`
- `ipv4.gateway`
- `ipv4.dns`
- `ipv4.never-default`
- `ipv6.method`
- `connection.interface-name`

A safe troubleshooting pattern is to modify the existing profile carefully or create a new one for the target interface rather than trying to force a profile meant for another device.

Example static IPv4 configuration:

```bash
nmcli connection modify "System ens160" \
  ipv4.method manual \
  ipv4.addresses 192.0.2.10/24 \
  ipv4.gateway 192.0.2.1 \
  ipv4.dns "192.0.2.53 192.0.2.54" \
  ipv4.never-default no \
  ipv6.method ignore
nmcli connection up "System ens160"
```

- `ipv4.gateway` adds a gateway for IPv4, but the final routing behavior still depends on the full profile and route metrics.
- `ipv4.never-default yes` prevents the profile from installing a default route.
- `ipv6.method ignore` disables IPv6 on that profile when you want to isolate IPv4 troubleshooting.

**DHCP troubleshooting**

DHCP issues usually fall into one of three categories: the client never sends a request, the server does not answer, or the lease is received but not applied correctly.

Check whether the interface is using DHCP:

```bash
nmcli connection show "System ens160" | grep -E 'ipv4.method|ipv6.method'
nmcli device show ens160
```

- `ipv4.method auto` means DHCP is expected.
- `ipv4.method manual` means static addressing is configured.
- `nmcli device show` can display the current lease-derived address, gateway, and DNS information.

Useful log checks:

```bash
journalctl -u NetworkManager -b
journalctl -b | grep -i dhcp
```

- NetworkManager logs often show DHCP discover, offer, request, and ack activity.
- If the interface never gets a lease, check cabling, VLAN assignment, switch port state, and firewall rules on the DHCP path.
- If a lease exists but the address is wrong, verify that the profile is not mixing static and DHCP settings.

You can also renew by bouncing the connection:

```bash
nmcli connection down "System ens160"
nmcli connection up "System ens160"
```

If you need a lower-level test, `dhclient` may be available on some systems, but NetworkManager remains the primary tool on RHCSA hosts.

**Routing and default gateway checks**

A host may have a valid address but still fail to reach other networks because the default route is missing or incorrect.

```bash
ip route show
ip route get 8.8.8.8
```

- `ip route show` should display a `default via <gateway>` line when the host needs off-subnet access.
- `ip route get <destination>` shows which interface and gateway the kernel would use.
- If multiple interfaces exist, route metrics determine which default route wins.

Common routing problems:

- No default route on a host that needs external access.
- Wrong gateway address.
- Competing default routes from multiple active profiles.
- `ipv4.never-default` unintentionally set to `yes`.

If you need to add a route temporarily for testing:

```bash
ip route add 198.51.100.0/24 via 192.0.2.1 dev ens160
```

For persistent configuration, use NetworkManager rather than temporary `ip route` changes.

**DNS troubleshooting**

DNS problems often look like “the network is down” even when IP connectivity is fine. Separate name resolution from transport by testing both numeric and named destinations.

```bash
ping -c 3 8.8.8.8
ping -c 3 example.com
getent hosts example.com
getent ahosts example.com
```

- Numeric ping tests basic IP reachability.
- `getent hosts` checks name resolution through the Name Service Switch (NSS) path.
- `getent ahosts` shows address-family results and may reflect `/etc/hosts`, resolver order, and IPv4/IPv6 preferences.
- If numeric ping works but names fail, focus on DNS configuration.

Inspect resolver settings:

```bash
cat /etc/resolv.conf
nmcli device show ens160 | grep -i dns
```

- NetworkManager often writes DNS servers and search domains into `/etc/resolv.conf`.
- Search domains can make short hostnames resolve unexpectedly or fail if the wrong domain is applied.
- Multiple resolvers are tried in order, but behavior can vary with resolver implementation and system configuration.

Common DNS issues:

- Wrong DNS server in the profile.
- Missing search domain for short hostnames.
- `/etc/hosts` overriding or masking DNS results.
- IPv4 works but IPv6 names fail, or the reverse, because only one address family is configured correctly.

Example DNS profile settings:

```bash
nmcli connection modify "System ens160" \
  ipv4.dns "192.0.2.53 192.0.2.54" \
  ipv4.dns-search "lab.example.com"
nmcli connection up "System ens160"
```

**Hostname and local identity problems**

A broken hostname setup can cause confusion in logs, prompts, and service discovery. Verify the system identity with the standard tools.

```bash
hostnamectl
cat /etc/hostname
cat /etc/hosts
```

- `hostnamectl` shows the static hostname and current transient hostname.
- `/etc/hostname` stores the persistent hostname.
- `/etc/hosts` should map the local hostname to the correct local address when required.

A common pattern is:

```bash
127.0.0.1   localhost
192.0.2.10  server1.example.com server1
```

Use this carefully:

- Do not map the hostname to the wrong interface address.
- Ensure the hostname resolves consistently for local services and administrative tools.
- If the system uses a static hostname, set it with `hostnamectl set-hostname server1.example.com`.

**IPv6 troubleshooting**

IPv6 can fail independently of IPv4. On RHCSA systems, you should know the difference between static IPv6, DHCPv6, and SLAAC (Stateless Address Autoconfiguration).

Check IPv6 state:

```bash
ip -6 addr show
ip -6 route show
nmcli connection show "System ens160" | grep -E 'ipv6.method|ipv6.addresses|ipv6.gateway'
```

- `ipv6.method auto` usually allows SLAAC and may also use DHCPv6 depending on network behavior.
- `ipv6.method manual` uses static IPv6 settings.
- `ipv6.method ignore` disables IPv6 on that profile.

Common IPv6 pitfalls:

- The interface has a link-local address only (`fe80::/10`) and no global address.
- Router advertisements are blocked, so SLAAC never completes.
- A static IPv6 address is configured without a matching gateway or prefix.
- DNS returns AAAA records, but the host has no working IPv6 route.

Example static IPv6 configuration:

```bash
nmcli connection modify "System ens160" \
  ipv6.method manual \
  ipv6.addresses 2001:db8:10::10/64 \
  ipv6.gateway 2001:db8:10::1 \
  ipv6.dns "2001:db8:10::53"
nmcli connection up "System ens160"
```

**Firewall and SELinux checks**

If IP connectivity works but a service still fails, the local firewall or SELinux may be blocking the traffic.

Firewall checks:

```bash
firewall-cmd --state
firewall-cmd --list-all
firewall-cmd --get-active-zones
```

- Confirm the service port is allowed in the active zone.
- Verify the interface is assigned to the expected zone.
- Remember that ICMP ping success does not prove TCP service access.

SELinux checks:

```bash
getenforce
ausearch -m AVC -ts recent
journalctl -t setroubleshoot -b
```

- `getenforce` shows whether SELinux is enforcing.
- AVC (Access Vector Cache) denials often explain why a service cannot bind or connect.
- Use `semanage port` to label nonstandard service ports when required.

Example port labeling:

```bash
semanage port -l | grep http
semanage port -a -t http_port_t -p tcp 8080
```

If the port already exists under another type, modify it instead:

```bash
semanage port -m -t http_port_t -p tcp 8080
```

> On some systems, `semanage` comes from `policycoreutils-python-utils`. If the command is missing, install the package before troubleshooting SELinux port labels.

**Logs and journal analysis**

Logs are essential when the configuration looks correct but activation still fails.

```bash
journalctl -u NetworkManager -b
journalctl -b | grep -iE 'networkmanager|dhcp|dns|link|carrier'
```

Look for:

- Profile activation failures.
- Authentication or permission errors.
- DHCP timeouts.
- Device rename or driver issues.
- DNS update failures.

If a connection fails to activate, the journal often tells you whether the problem is:

- A missing device.
- A bad profile key.
- A duplicate IP address.
- A failed DHCP exchange.
- A route conflict.

**Packet capture and traffic verification**

Packet capture helps when you need to prove whether traffic leaves the host or returns.

```bash
tcpdump -i ens160 -n
tcpdump -i ens160 -n port 53
tcpdump -i ens160 -n host 192.0.2.53
```

Interpretation examples:

- DNS queries leave but no replies return: check resolver settings, firewall rules, upstream DNS reachability, and local firewall filtering of replies.
- ARP requests repeat with no reply on IPv4 networks: check the gateway, VLAN, switch port, or duplicate addressing.
- ICMP echo requests leave but no echo replies return: the remote host may be down, filtered, or on the wrong route.

A simple capture for DHCP:

```bash
tcpdump -i ens160 -n port 67 or port 68
```

- You should see discover, offer, request, and ack traffic during lease acquisition.
- If you only see discover packets, the server or network path is not responding.

**Application-layer verification**

Ping is useful, but it does not prove that a service works. Always test the actual protocol when possible.

```bash
curl -I http://192.0.2.20
curl -vk https://192.0.2.20
nc -vz 192.0.2.20 22
nc -vz 192.0.2.20 80
```

- `curl` verifies HTTP or HTTPS reachability and can reveal TLS or proxy issues.
- `nc -vz` checks whether a TCP port accepts connections.
- A service can fail even when ping succeeds because the application port is closed, filtered, or misconfigured.

**Common RHCSA troubleshooting scenarios**

- The interface is `UP`, but there is no carrier.
  - Check cable, virtual NIC attachment, switch port, and `ethtool`.
- The host has an IP address but cannot reach the internet.
  - Check the default route, gateway, and DNS separately.
- The host can ping by IP but not by name.
  - Check `/etc/resolv.conf`, NetworkManager DNS settings, and search domains.
- The host resolves names but cannot reach a service.
  - Check firewall, SELinux, and the service port.
- IPv4 works but IPv6 fails.
  - Check `ipv6.method`, router advertisements, and IPv6 routes.
- DHCP never assigns an address.
  - Check NetworkManager logs, switch connectivity, and DHCP traffic with `tcpdump`.

**Practical troubleshooting sequence**

1. Confirm the device is present and linked.
2. Confirm the profile is active and correctly bound to the device.
3. Confirm the interface has the expected IPv4 and IPv6 addresses.
4. Confirm the default route exists and points to the correct gateway.
5. Confirm DNS servers and search domains are correct.
6. Test numeric connectivity, then name resolution, then application ports.
7. Check logs, firewall, and SELinux if the network path looks correct.

**Useful reference commands**

```bash
nmcli device status
nmcli connection show
ip addr show

hostnamectl
cat /etc/hosts
cat /etc/resolv.conf

firewall-cmd --list-all

```

**Exam-focused habits**

- Prefer `nmcli` for persistent network changes.
- Use `ip` for immediate state inspection.
- Use `journalctl` when activation or DHCP fails.
- Use `curl` or `nc` when the network is up but the service still fails.
- Keep IPv4 and IPv6 troubleshooting separate until you know which family is broken.
