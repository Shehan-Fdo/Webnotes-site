## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Service status basics**

On modern Red Hat Enterprise Linux systems, `systemd` manages services, sockets, mounts, timers, and other units. When you check service status, you are usually verifying whether a unit is loaded, active, enabled, failed, or masked. This is the first step in troubleshooting because it tells you whether the problem is with startup, runtime behavior, or dependency resolution.

A service can be:
- **active**: currently running or otherwise in use
- **inactive**: not running
- **failed**: started but exited with an error or crashed
- **enabled**: starts automatically at boot
- **disabled**: does not start automatically
- **masked**: intentionally blocked from starting

Use `systemctl` for most service checks. `service` and `chkconfig` may still appear on older systems, but `systemctl` is the standard for RHCSA-level administration.

```bash
systemctl status sshd
systemctl is-active sshd
systemctl is-enabled sshd
systemctl is-failed sshd
```

`systemctl status` gives the most useful summary because it shows:
- the unit state
- the main process ID
- recent log lines
- the reason for the last failure, if any

**Reading `systemctl status` output**

A typical status check includes the unit name, load state, active state, and recent journal entries. You use this output to decide whether the service is running, whether it started at boot, and whether it has encountered errors.

```bash
systemctl status firewalld
```

Key fields to interpret:
- **Loaded**: whether the unit file is available and parsed
- **Active**: current runtime state
- **Sub**: more specific state, such as `running`, `dead`, or `exited`
- **Main PID**: the primary process for the service
- **Docs**: documentation references from the unit file

If a service is not running, `systemctl status` often shows the last few log messages. This is useful because many failures are visible immediately without opening the full journal.

> A service can be enabled and still fail at runtime. Boot-time enablement does not guarantee a healthy process after startup.

**Checking whether a service is running or enabled**

For quick checks, use the dedicated `systemctl` queries. These are ideal in scripts and during exam tasks because they return a simple answer.

```bash
systemctl is-active httpd
systemctl is-enabled httpd
systemctl is-failed httpd
```

Common return values:
- `active` or `inactive` for `is-active`
- `enabled`, `disabled`, `static`, or `masked` for `is-enabled`
- `failed` or `unknown` for `is-failed`

Use `systemctl list-unit-files` when you want to inspect install-time state across many services.

```bash
systemctl list-unit-files --type=service
```

**Starting, stopping, and restarting services**

When troubleshooting, you often need to restart a service after changing configuration. A restart stops and starts the service again, which is useful after config edits but can briefly interrupt users. A reload asks the service to re-read configuration without a full stop, but only works if the service supports it.

```bash
systemctl restart sshd
systemctl reload nginx
systemctl try-reload-or-restart httpd
```

Use these carefully:
- **restart**: safest when you are unsure whether reload is supported
- **reload**: preferred when the service supports live config reload
- **try-reload-or-restart**: attempts reload first, then restarts if needed

**Understanding dependencies**

`systemd` units can depend on other units. Dependencies determine startup order and whether a service can start successfully. If a required dependency is missing or failed, the service may not start.

Important dependency types include:
- **Requires=**: the unit cannot function without the dependency
- **Wants=**: a soft dependency; startup continues even if the dependency fails
- **After=**: controls ordering, not requirement
- **Before=**: inverse ordering relationship
- **BindsTo=**: stronger runtime coupling than `Requires=`
- **PartOf=**: ties stop/restart behavior to another unit

A common mistake is assuming `After=` means “depends on.” It does not. `After=` only changes order. If the dependency is not also required or wanted, the service may still start without it.

**Inspecting dependencies with `systemctl`**

Use dependency views to understand what a service needs and what depends on it.

```bash
systemctl list-dependencies sshd
systemctl list-dependencies --reverse network.target
```

Useful options:
- `--reverse`: shows units that depend on the selected unit
- `--all`: includes inactive dependencies
- `--before` and `--after`: filter by ordering relationships

You can also inspect the unit file directly:

```bash
systemctl cat sshd
systemctl show sshd
```

`systemctl show` is especially useful because it exposes machine-readable properties such as:
- `ActiveState`
- `SubState`
- `UnitFileState`
- `FragmentPath`
- `After`
- `Requires`
- `Wants`

Example:

```bash
systemctl show -p ActiveState -p SubState -p UnitFileState sshd
```

**Practical dependency troubleshooting**

If a service fails to start, check whether one of its dependencies is missing, inactive, or misconfigured. This is common with services that require:
- a network interface
- a database backend
- a mounted filesystem
- a socket or timer
- a specific target such as `network-online.target`

A service may appear healthy in isolation but still fail because a required mount is unavailable or a network target has not been reached yet.

Example workflow:
1. Check the service status.
2. Inspect recent logs.
3. Review dependencies.
4. Verify the dependent unit is active.
5. Restart the service after fixing the root cause.

```bash
systemctl status mariadb
systemctl list-dependencies mariadb
journalctl -u mariadb -b
```

**Using the journal for service failures**

`systemctl status` shows only a small slice of logs. For deeper troubleshooting, use `journalctl`. This is often necessary when the failure is caused by configuration syntax, permission issues, missing files, or dependency timeouts.

```bash
journalctl -u httpd
journalctl -u httpd -b
journalctl -xeu httpd
```

Useful patterns:
- `-u <unit>`: filter by service
- `-b`: show only the current boot
- `-xeu`: show extended, explanatory, and unit-specific messages

If a service fails repeatedly, the journal may show:
- permission denied errors
- port binding conflicts
- missing configuration files
- SELinux denials
- dependency timeout messages

**Common service states and what they mean**

| State | Meaning | Typical action |
|---|---|---|
| `active (running)` | Service is up and processing requests | Verify functionality |
| `inactive (dead)` | Service is stopped | Start it if needed |
| `failed` | Service exited with an error | Check logs and dependencies |
| `activating` | Service is starting | Wait or inspect for delays |
| `deactivating` | Service is stopping | Confirm shutdown behavior |
| `masked` | Service is blocked from starting | Unmask before starting |

**Working with targets and boot order**

Targets are synchronization points in `systemd`. They help group services and define boot stages. A service may depend on a target such as `multi-user.target` or `network.target`. If the target is not reached, dependent services may not start as expected.

```bash
systemctl get-default
systemctl list-dependencies multi-user.target
systemctl list-dependencies network-online.target
```

This is useful when a service starts manually but not automatically at boot. In that case, the issue may be ordering or target membership rather than the service itself.

**Exam-focused troubleshooting pattern**

When a service is not behaving as expected, use a repeatable sequence.

1. Check the unit state.
   ```bash
   systemctl status <service>
   ```
2. Confirm whether it is enabled.
   ```bash
   systemctl is-enabled <service>
   ```
3. Review logs for the current boot.
   ```bash
   journalctl -u <service> -b
   ```
4. Inspect dependencies.
   ```bash
   systemctl list-dependencies <service>
   ```
5. Verify related units, mounts, or network state.
   ```bash
   systemctl status network-online.target
   systemctl status <mount-point>.mount
   ```
6. Fix the root cause and restart the service.
   ```bash
   systemctl restart <service>
   ```

**Practical examples**

Check whether the SSH daemon is running and enabled:

```bash
systemctl status sshd
systemctl is-active sshd
systemctl is-enabled sshd
```

Inspect a web server and its dependencies:

```bash
systemctl status httpd
systemctl list-dependencies httpd
journalctl -u httpd -b
```

Check whether a service is blocked by masking:

```bash
systemctl is-enabled cups
systemctl mask cups
systemctl unmask cups
```

Verify a service after configuration changes:

```bash
apachectl configtest
systemctl restart httpd
systemctl status httpd
```

**Configuration and unit-file awareness**

Sometimes the issue is not the service itself but the unit definition or an override. Use `systemctl cat` to view the full unit file, including drop-in overrides.

```bash
systemctl cat sshd
systemctl cat httpd
```

If you need to inspect the unit file path and related metadata:

```bash
systemctl show -p FragmentPath -p DropInPaths httpd
```

This helps you determine whether a vendor unit file, a local override, or a generated unit is controlling behavior.

**Best practices**

- Check `systemctl status` first because it gives immediate context.
- Use `journalctl -u` when the status output is not enough.
- Distinguish between ordering and dependency requirements.
- Verify enablement separately from runtime state.
- Restart only after confirming the underlying issue is fixed.
- Remember that a service can be enabled, inactive, or failed at the same time.

**Quick reference**

```bash
systemctl status <service>
systemctl is-active <service>
systemctl is-enabled <service>
systemctl list-dependencies <service>
systemctl cat <service>
journalctl -u <service> -b
```

These commands cover most RHCSA service-status and dependency checks. If you can read their output confidently, you can diagnose many startup and runtime issues quickly and accurately.
