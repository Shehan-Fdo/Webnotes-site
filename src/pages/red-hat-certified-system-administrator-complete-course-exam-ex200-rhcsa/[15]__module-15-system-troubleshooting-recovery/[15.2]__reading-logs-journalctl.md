## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

`journalctl` is the primary tool for reading logs collected by `systemd-journald`. On modern RHEL systems, it gives you a unified view of kernel messages, boot logs, service output, and system events. For RHCSA work, you use it to troubleshoot failed services, inspect boot issues, and correlate events by time, unit, priority, or boot session.

The journal is different from traditional text logs in `/var/log` because it is structured, queryable, and often binary-backed. That structure makes filtering fast and precise, especially when you need to answer questions like “What failed during the last boot?” or “What did this service print right before it crashed?”

**What the journal stores**

`systemd-journald` collects messages from several sources:

- Kernel messages
- Messages from system services managed by `systemd`
- Standard output and standard error from services
- Syslog-compatible messages from applications
- Boot and session metadata

Because entries are indexed, you can filter by fields such as `_SYSTEMD_UNIT`, `_PID`, `_UID`, `_BOOT_ID`, and `PRIORITY`.

**Basic usage**

The simplest invocation shows the full journal, newest entries last by default:

```bash
journalctl
```

Useful display options include:

```bash
journalctl -n 20
journalctl -f
journalctl --no-pager
journalctl -r
```

- `-n 20` shows the last 20 lines or entries.
- `-f` follows new log entries in real time.
- `--no-pager` prints directly to the terminal.
- `-r` reverses the order so the newest entries appear first.

`journalctl -f` is especially useful while restarting a service or reproducing a problem, because you can watch log messages appear as they happen.

**Filtering by boot and time**

Boot filtering is one of the most important troubleshooting skills. Each boot has its own journal context, and you can inspect the current or previous boot quickly.

```bash
journalctl -b
journalctl -b -1
journalctl --list-boots
```

- `-b` shows the current boot.
- `-b -1` shows the previous boot.
- `--list-boots` displays boot IDs and relative indexes.

Time-based filtering helps narrow a problem to a specific window:

```bash
journalctl --since "2026-04-26 08:00:00" --until "2026-04-26 09:00:00"
journalctl --since today
journalctl --since "1 hour ago"
journalctl --since yesterday
```

Use time filters when you know approximately when the failure occurred, such as after a reboot, package update, or configuration change.

**Filtering by service or unit**

For service troubleshooting, filter by the systemd unit name. This is one of the most common RHCSA tasks.

```bash
journalctl -u sshd
journalctl -u sshd.service
journalctl -u crond --since today
journalctl -u NetworkManager -b
```

- `-u` targets a specific unit.
- You can omit `.service` in many cases.
- Combine unit filters with time or boot filters for better precision.

A practical workflow is:

```bash
systemctl status sshd
journalctl -u sshd -b --no-pager
```

`systemctl status` gives a quick summary and recent log lines, while `journalctl` provides the full history for that unit.

**Filtering by priority**

Journal entries have severity levels. Filtering by priority helps you focus on warnings and errors.

Priority levels are:

- `0` emergency
- `1` alert
- `2` critical
- `3` error
- `4` warning
- `5` notice
- `6` informational
- `7` debug

Examples:

```bash
journalctl -p err
journalctl -p warning
journalctl -p 3
journalctl -p 3..0
```

- `-p err` shows error and more severe messages.
- Numeric values are also accepted.
- Range syntax can be used to include multiple severities.

For troubleshooting, start with `warning` or `err`, then widen the scope if needed.

**Kernel and boot messages**

Kernel logs are often essential when diagnosing hardware, driver, storage, or boot issues.

```bash
journalctl -k
journalctl -k -b
journalctl -k -b -1
```

- `-k` shows only kernel messages.
- Combine it with boot filters to inspect the current or previous boot.

If a system fails early in boot, kernel messages may reveal filesystem errors, missing devices, or initramfs problems before services even start.

**Following logs in real time**

When you are actively testing a change, follow the journal live:

```bash
journalctl -f
journalctl -u httpd -f
journalctl -f -p warning
```

This is useful when you:

- Restart a service and want immediate feedback
- Trigger a login attempt or network event
- Watch for repeated errors during a test

If you are debugging a service, pair live logs with a second terminal running the command or action that triggers the issue.

**Output formatting and readability**

Several options improve readability and make logs easier to scan:

```bash
journalctl -o short
journalctl -o short-iso
journalctl -o verbose
journalctl -o cat
```

- `short` is a compact default-style view.
- `short-iso` includes ISO-like timestamps.
- `verbose` shows all journal fields.
- `cat` prints only the message text.

`short-iso` is often the best choice when you need clear timestamps for incident timelines.

**Searching and matching**

You can search within the output using standard tools, but `journalctl` also supports field-based filtering. For example:

```bash
journalctl _PID=1234
journalctl _UID=1000
journalctl _COMM=sshd
journalctl MESSAGE_ID=fc2e22bc6ee647b6b90729ab34a250b1
```

Common fields include:

- `_PID` for process ID
- `_UID` for user ID
- `_COMM` for command name
- `_EXE` for executable path
- `_SYSTEMD_UNIT` for the unit name

Field filters are powerful because they match structured metadata rather than just text.

**Practical troubleshooting workflow**

A reliable workflow for service problems is:

1. Check the service state.
2. Read recent logs for the unit.
3. Narrow by boot, time, or priority if needed.
4. Correlate with kernel messages when the issue looks system-level.

Example:

```bash
systemctl status nginx
journalctl -u nginx -b --no-pager
journalctl -u nginx --since "10 minutes ago"
journalctl -p err --since today
```

If the service fails to start, look for:

- Syntax or configuration errors
- Permission denials
- Port conflicts
- Missing files or directories
- Dependency failures from other units

**Common real-world examples**

Inspect SSH login failures:

```bash
journalctl -u sshd --since today
```

Check NetworkManager activity:

```bash
journalctl -u NetworkManager -b
```

Review boot problems:

```bash
journalctl -b -p err
journalctl -k -b
```

Track a service while restarting it:

```bash
journalctl -u httpd -f
systemctl restart httpd
```

These patterns are common in exam scenarios because they help you move from symptom to cause quickly.

**Persistent versus volatile journals**

The journal may be stored in memory or on disk depending on system configuration. If persistent storage is enabled, logs survive reboots. If not, older logs may disappear after restart.

Typical persistent storage location:

```bash
/var/log/journal
```

If that directory exists and journald is configured for persistence, you can inspect logs across boots. If not, only the current boot or recent volatile data may be available.

**Configuration notes**

The journald configuration file is:

```bash
/etc/systemd/journald.conf
```

A common setting for persistent storage is:

```ini
[Journal]
Storage=persistent
```

After changing configuration, restart the service:

```bash
systemctl restart systemd-journald
```

Be careful when editing journald settings on a production system, because log retention and disk usage can change significantly.

> If the journal is not persistent, older boot logs may be unavailable after a reboot. Always verify with `journalctl --list-boots` before assuming history exists.

**Useful command combinations**

A few combinations cover most RHCSA troubleshooting needs:

```bash
journalctl -b -u sshd -p err
journalctl --since "1 hour ago" -u crond
journalctl -k -b -p warning
journalctl -u NetworkManager -f
```

These combinations let you narrow by:

- Boot
- Unit
- Time window
- Severity
- Kernel versus service output

**When to use `journalctl` instead of text logs**

Use `journalctl` when you need:

- Structured filtering
- Boot-specific investigation
- Service output captured by `systemd`
- Real-time follow mode
- Kernel and user-space logs in one place

Use traditional log files when an application writes only to files under `/var/log` or when a vendor tool explicitly requires a text log.

**Exam tips**

For RHCSA tasks, remember these high-value patterns:

- `journalctl -u <unit>` for service logs
- `journalctl -b` for current boot
- `journalctl -b -1` for previous boot
- `journalctl -p err` for errors
- `journalctl -f` for live troubleshooting
- `journalctl -k` for kernel messages

If you are unsure where to start, check `systemctl status <unit>` first, then use `journalctl` to expand the investigation.
