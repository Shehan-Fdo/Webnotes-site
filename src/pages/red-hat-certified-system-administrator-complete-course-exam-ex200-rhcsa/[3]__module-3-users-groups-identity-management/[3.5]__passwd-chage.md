## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

`passwd` and `chage` manage password state, aging, and account expiration on Linux systems. For RHCSA, you need to know how to force a password change, set aging policy, inspect current settings, and troubleshoot why a user cannot log in.

These tools affect different parts of authentication:

- **Password expiration** controls whether the current password is still valid.
- **Account expiration** controls whether the account itself is allowed to log in after a date.
- **Password locking** disables password-based authentication, but other authentication methods may still work.

That distinction matters on RHEL systems, especially when SSH keys, smart cards, or other non-password methods are in use.

**`passwd` basics**

`passwd` changes a user password and can also manipulate password status. As root, you can manage other users; as a regular user, you can usually change only your own password.

Common usage:

```bash
passwd
passwd alice
```

- `passwd` with no arguments changes your own password.
- `passwd alice` changes `alice`’s password when run by root.
- The command updates password-related fields in `/etc/shadow`.

Useful options:

- `-l` locks the password by prefixing the encrypted password field with `!` in `/etc/shadow`.
- `-u` unlocks the password by removing the lock marker.
- `-e` expires the password immediately so the user must change it at next login.
- `-S` shows password status in a compact form.

Example:

```bash
sudo passwd -l alice
sudo passwd -u alice
sudo passwd -e alice
sudo passwd -S alice
```

> `passwd -l` and `passwd -u` affect the **password**, not the entire account. A user may still authenticate with non-password methods if those are configured.

**Password locking versus account locking**

A locked password means the password hash is not accepted for login. This is not the same as disabling the account entirely.

- **Password lock**
  - Prevents password-based logins.
  - Does not necessarily block SSH public key authentication.
  - Is useful when you want to stop password use without deleting the account.

- **Account expiration**
  - Prevents logins after a specified date.
  - Is enforced by account policy, not by the password hash alone.
  - Is more appropriate when the account should stop being usable altogether.

- **Password expiration**
  - Forces the user to change the password at next login.
  - Does not disable the account.
  - Is often used for initial provisioning or compliance.

A practical check:

```bash
sudo passwd -S alice
sudo grep '^alice:' /etc/shadow
```

Typical `passwd -S` output includes status codes such as:

- `P` for usable password
- `L` for locked password
- `NP` for no password set

**`chage` for aging policy**

`chage` manages password aging and account expiration. It is the main tool for viewing and setting when passwords must be changed and when an account expires.

View current settings:

```bash
chage -l alice
```

This shows:

- Last password change date
- Minimum password age
- Maximum password age
- Warning period before expiration
- Inactive period after expiration
- Account expiration date

Set aging values:

```bash
sudo chage -m 7 -M 90 -W 14 -I 30 alice
```

This means:

- Minimum age: 7 days before another password change is allowed
- Maximum age: 90 days before the password expires
- Warning: 14 days of warning before expiration
- Inactive: 30 days after expiration before the account becomes inaccessible

Set account expiration:

```bash
sudo chage -E 2026-12-31 alice
```

Clear account expiration on RHEL:

```bash
sudo chage -E -1 alice
```

Force password change at next login:

```bash
sudo chage -d 0 alice
```

`chage -d 0` sets the last password change field to `0` in `/etc/shadow`, which the system interprets as “password must be changed at next login.”

**How `passwd -e` and `chage -d 0` differ**

Both commands force a password change at next login, but they do it through different interfaces.

- `passwd -e`
  - Expires the password immediately.
  - Is a direct `passwd` action.
- `chage -d 0`
  - Sets the last change date to `0` in `/etc/shadow`.
  - Is often preferred when you are already managing aging policy with `chage`.

In practice, both are valid for RHCSA tasks that require a user to change the password on first login.

**System-wide defaults and inheritance**

New users often inherit password aging defaults from system policy files. On RHEL, the most relevant source is typically `/etc/login.defs`, and user creation tools may also apply defaults from skeleton or profile settings.

Check the aging defaults:

```bash
grep -E 'PASS_MAX_DAYS|PASS_MIN_DAYS|PASS_WARN_AGE' /etc/login.defs
```

Typical settings might look like:

```bash
PASS_MAX_DAYS   90
PASS_MIN_DAYS   7
PASS_WARN_AGE   14
```

Important RHCSA nuance:

- These defaults affect **new accounts** created after the policy is in place.
- Existing accounts do not automatically change unless you update them with `chage` or another management tool.
- `useradd` may also use defaults from `/etc/default/useradd` and related configuration, depending on the system.

If you need to verify what a new user will inherit, inspect both the login defaults and the user creation defaults:

```bash
useradd -D
cat /etc/login.defs
```

**Practical examples**

Create a user, set a password, and force a change at first login:

```bash
sudo useradd bob
sudo passwd bob
sudo chage -d 0 bob
```

Set a password to expire every 60 days with a 10-day warning:

```bash
sudo chage -M 60 -W 10 bob
```

Lock the password for a maintenance hold:

```bash
sudo passwd -l bob
```

Unlock it later:

```bash
sudo passwd -u bob
```

Expire the account on a specific date:

```bash
sudo chage -E 2026-09-30 bob
```

Inspect the result:

```bash
sudo chage -l bob
sudo passwd -S bob
```

**Troubleshooting login failures**

When a user cannot log in, check whether the issue is password expiration, password locking, or account expiration.

Common symptoms:

- The user is prompted to change the password immediately after login.
  - The password has expired.
  - Check `chage -l username` or `passwd -S username`.

- The login fails even though the password is correct.
  - The password may be locked.
  - Check `/etc/shadow` for a leading `!` in the password field.

- The account is rejected regardless of password.
  - The account may be expired.
  - Check `chage -l username` for the account expiration date.

Useful diagnostics:

```bash
sudo chage -l alice
sudo passwd -S alice
sudo grep '^alice:' /etc/shadow
```

Example `/etc/shadow` patterns:

- `alice:!$6$...` indicates a locked password.
- `alice:$6$...` indicates a usable password hash.
- A `0` in the last-change field indicates the password must be changed at next login.
- An account expiration date field set to a valid date means the account stops working after that date.

If a user reports “I can still log in with SSH keys,” remember that password locking does not necessarily disable key-based access. That is expected behavior, not a bug.

**Exam-oriented workflow**

For RHCSA tasks, use a repeatable workflow:

1. Inspect the current state with `chage -l` and `passwd -S`.
2. Decide whether you need password expiration, account expiration, or password locking.
3. Apply the change with the appropriate command.
4. Verify the result immediately.
5. Test with a login attempt if the task allows it.

Example:

```bash
sudo chage -l alice
sudo passwd -S alice
sudo chage -M 45 -W 7 alice
sudo chage -d 0 alice
sudo chage -l alice
```

**Key takeaways**

- `passwd` is primarily for password changes and password status.
- `chage` is the main tool for aging policy and account expiration.
- Password locking is not the same as account disabling.
- `chage -d 0` and `passwd -e` both force a password change at next login.
- `chage -E -1` clears account expiration on RHEL.
- New users inherit defaults from system policy, so check `/etc/login.defs` and related user defaults when troubleshooting.
