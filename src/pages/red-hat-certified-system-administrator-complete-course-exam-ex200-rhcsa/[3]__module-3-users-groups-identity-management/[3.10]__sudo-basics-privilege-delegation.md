## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**sudo fundamentals**

`sudo` lets you run commands with elevated privileges without logging in as `root`. On RHEL, this is the standard way to delegate administrative tasks while keeping accountability through per-user authorization and logging.

The main idea is **least privilege**: you grant only the commands a user needs, instead of giving full root access. This reduces risk and makes auditing easier.

- `sudo` authenticates the invoking user, not `root`.
- Authorization comes from `/etc/sudoers` and files in `/etc/sudoers.d/`.
- `sudo` can allow full root access, specific commands, or command patterns.
- `sudo` is usually preferred over direct `su -` for routine administration.

**How sudo works**

When you run `sudo`, the system checks whether your user is allowed to execute the requested command as another user, usually `root`. The rule can be broad or narrow:

- A rule may allow all commands for a user or group.
- A rule may allow only a specific command path.
- A rule may include arguments, wildcards, or omit arguments entirely.
- Matching is based on the command path and the rule pattern, so the exact behavior depends on how the rule is written.

This means you should be precise when testing. If a rule allows `/usr/bin/journalctl -u httpd --no-pager`, then test with the full path and the same arguments.

**Core files and tools**

The main configuration file is `/etc/sudoers`. Do not edit it directly with a normal text editor unless you know what you are doing. Use `visudo`, which validates syntax before saving.

Common locations:

- `/etc/sudoers` for the main policy
- `/etc/sudoers.d/` for drop-in policy files
- `/var/log/secure` or journald for authentication and sudo-related messages, depending on system logging configuration

Useful commands:

```bash
sudo -l
sudo -v
sudo -k
sudo -i
sudo -s
```

- `sudo -l` lists allowed commands for the current user.
- `sudo -v` refreshes the cached authentication timestamp.
- `sudo -k` invalidates the cached timestamp.
- `sudo -i` starts a login shell as the target user, usually `root`.
- `sudo -s` starts a shell with the target user’s environment more lightly modified than `-i`.

**Editing sudoers safely**

Always use `visudo` because it checks syntax and helps prevent lockout.

```bash
sudo visudo
```

To edit a drop-in file:

```bash
sudo visudo -f /etc/sudoers.d/admins
```

A bad sudoers file can break privilege escalation for everyone, so syntax validation matters.

> Never edit `/etc/sudoers` with a plain editor unless you have a recovery plan.

**Common sudoers syntax**

A sudoers rule has the general shape:

```text
user host_list = (runas_list) command_list
```

Examples:

```text
alice ALL=(root) /usr/bin/systemctl restart httpd
%wheel ALL=(ALL) ALL
```

Key parts:

- `user` can be a username or a group prefixed with `%`.
- `host_list` often uses `ALL` on single-system lab environments.
- `runas_list` defines which user the command runs as.
- `command_list` defines the permitted command or commands.

The `%wheel ALL=(ALL) ALL` line is often present on RHEL systems but may be commented out by default. If it is commented, it does nothing until enabled.

**Privilege delegation with groups**

Group-based delegation is common because it scales better than per-user rules.

Example:

```text
%admins ALL=(root) /usr/bin/systemctl restart httpd
```

This allows members of `admins` to restart Apache without granting full root access.

Practical steps:

1. Create the group if needed.
2. Add users to the group.
3. Add a sudoers rule for the group.
4. Validate with `visudo`.
5. Test with `sudo -l` and the exact command path.

Example commands:

```bash
sudo groupadd admins
sudo usermod -aG admins alice
sudo visudo -f /etc/sudoers.d/admins
```

**Command-specific delegation**

Command-specific rules are the most useful for RHCSA-style least-privilege tasks. You can allow a single administrative action instead of all root access.

Examples:

```text
alice ALL=(root) /usr/bin/systemctl restart httpd
alice ALL=(root) /usr/bin/systemctl status httpd
```

You can also allow a command with arguments:

```text
bob ALL=(root) /usr/bin/journalctl -u httpd --no-pager
```

Be careful with argument matching:

- If you include arguments, the rule is more restrictive.
- If you omit arguments, the user may run the command with different arguments, depending on the rule.
- Wildcards can broaden access, so use them only when necessary.

**Command-specific Defaults**

Some sudo settings apply only to a specific command. The syntax uses `Defaults!` with no space before the command path.

Example:

```text
Defaults!/usr/bin/systemctl log_output
```

This kind of setting is useful when you want special behavior for one command without changing global sudo behavior.

Other common `Defaults` settings include:

```text
Defaults logfile="/var/log/sudo.log"
Defaults timestamp_timeout=5
Defaults lecture=always
```

- `logfile` writes sudo events to a dedicated file if supported by the build and policy.
- `timestamp_timeout` controls how long authentication is cached.
- `lecture` controls whether sudo displays a warning message.

**Logging and auditing**

`sudo` activity is typically logged through the system logging stack. On many RHEL systems, messages may appear in `/var/log/secure` or in the journal, but exact visibility depends on the logging configuration.

Useful checks:

```bash
sudo tail -f /var/log/secure
journalctl -f
```

If your system uses journald and the `sudo` tag is present, you may filter for it, but this is configuration-dependent:

```bash
journalctl | grep sudo
```

For exam and lab work, focus on knowing that sudo actions are logged and that the exact viewing method depends on the host’s logging setup.

**Testing and verification**

After you change sudoers, verify the policy before relying on it.

1. Check syntax with `visudo`.
2. List permissions with `sudo -l`.
3. Test the exact allowed command.
4. Confirm the command runs as the intended target user.

Example:

```bash
sudo -l
sudo /usr/bin/systemctl restart httpd
sudo /usr/bin/journalctl -u httpd --no-pager
```

If the rule is path-specific, use the full path in your test. Do not assume `sudo systemctl ...` will match a rule written for `/usr/bin/systemctl`.

**Environment and shell behavior**

`sudo` can preserve or reset parts of the environment depending on policy. This matters when commands depend on variables like `PATH`, `HOME`, or locale settings.

Common patterns:

- `sudo -i` gives you a root login shell and root’s environment.
- `sudo -s` gives you a shell with elevated privileges but less of a login-shell reset.
- `sudo` may reset `PATH` to a secure default.

If a command is not found under `sudo`, use the full path:

```bash
sudo /usr/bin/systemctl status httpd
```

This avoids ambiguity and matches sudoers rules more reliably.

**Password prompts and timestamp caching**

By default, sudo may cache authentication for a short period. This reduces repeated password prompts during a maintenance session.

Useful behavior:

- The first `sudo` command prompts for your password.
- Subsequent commands may not prompt again until the timeout expires.
- `sudo -k` forces a new password prompt next time.
- `sudo -v` refreshes the cached timestamp without running a command.

This is convenient for admins, but it also means unattended terminals should be locked or closed when not in use.

**Common delegation patterns**

Typical RHCSA-relevant use cases include:

- restarting a service
- checking service status
- reading logs for a specific service
- managing a small set of administrative commands
- allowing a support user to perform one operational task without full root access

Examples:

```text
support ALL=(root) /usr/bin/systemctl restart nginx
support ALL=(root) /usr/bin/systemctl status nginx
support ALL=(root) /usr/bin/journalctl -u nginx --no-pager
```

This approach is better than granting `ALL` because it limits the blast radius if the account is compromised.

**Troubleshooting sudo rules**

If a rule does not work, check these items:

- The command path matches the sudoers entry.
- The user is in the expected group.
- The sudoers file has valid syntax.
- The file permissions are correct.
- The command arguments match the rule.
- The rule is not overridden by another entry.

Useful diagnostics:

```bash
sudo -l
sudo visudo -c
id alice
ls -l /etc/sudoers /etc/sudoers.d/
```

`visudo -c` checks sudoers syntax without editing:

```bash
sudo visudo -c
```

If a drop-in file is ignored, confirm it has safe permissions and valid syntax. Sudo is intentionally strict about policy file integrity.

**Security best practices**

- Grant the smallest command set that solves the task.
- Prefer group-based rules for maintainability.
- Use full command paths in sudoers.
- Avoid broad wildcards unless required.
- Review and remove temporary access after the task is complete.
- Keep sudoers changes in `/etc/sudoers.d/` when possible for clarity.

A good sudo policy is easy to read, easy to audit, and hard to misuse.

**Quick reference**

| Task | Command |
|---|---|
| Validate sudoers syntax | `sudo visudo -c` |
| Edit main policy safely | `sudo visudo` |
| Edit drop-in file | `sudo visudo -f /etc/sudoers.d/name` |
| List allowed commands | `sudo -l` |
| Clear cached auth | `sudo -k` |
| Refresh cached auth | `sudo -v` |
| Start root login shell | `sudo -i` |

**Practical example**

A support user needs to restart Apache and view its status, but nothing else.

```text
support ALL=(root) /usr/bin/systemctl restart httpd
support ALL=(root) /usr/bin/systemctl status httpd
```

Then test:

```bash
sudo -l
sudo /usr/bin/systemctl status httpd
sudo /usr/bin/systemctl restart httpd
```

If the command fails, check whether the binary path is correct on the system and whether the rule was saved in a valid sudoers file.

**Key takeaways**

- `sudo` delegates privilege without sharing the root password.
- `visudo` is the safe way to manage sudoers.
- Rules can be broad or highly specific.
- Full command paths improve reliability and clarity.
- Logging and exact command matching depend on policy and system configuration.
- Least privilege is the goal for both security and exam readiness.
