## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Group management overview**

Linux groups control shared access to files, directories, and administrative tasks. On Red Hat Enterprise Linux, you manage local groups with `groupadd`, `groupmod`, and `groupdel`. These tools update the local account databases, primarily `/etc/group` and `/etc/gshadow`, and they are central to RHCSA-level user administration.

A group can be a **primary group** for a user or a **supplementary group** used for shared access. You use group management when you need to:

- Create a new collaboration group for a project.
- Rename a group without changing its numeric identity.
- Change a group’s numeric GID to match policy.
- Remove an unused group after verifying it is no longer assigned.

**Group database basics**

Linux stores group information in text databases that are queried through the Name Service Switch (`nsswitch.conf`) stack. For local groups, the key files are:

- `/etc/group` for group names, passwords placeholder, GIDs, and member lists.
- `/etc/gshadow` for secure group administration data.
- `/etc/passwd` for each user’s primary GID field.
- `/etc/login.defs` for default values such as the GID range.

A group entry in `/etc/group` looks like this:

```text
developers:x:1050:alice,bob
```

Field meaning:

- `developers` is the group name.
- `x` indicates the password is stored in `/etc/gshadow`.
- `1050` is the numeric GID.
- `alice,bob` are supplementary members.

**Creating groups with `groupadd`**

Use `groupadd` to create a new local group. The most common form is simple and explicit:

```bash
sudo groupadd developers
```

This creates a group using the next available GID from the system’s configured range. On many systems, that range is defined in `/etc/login.defs` with values such as `GID_MIN` and `GID_MAX`.

Common options:

- `-g GID` sets the numeric GID explicitly.
- `-r` creates a system group, usually from the system GID range.
- `-f` forces success if the group already exists, but it does not broadly resolve all GID conflict cases.

Examples:

```bash
sudo groupadd -g 1050 developers
sudo groupadd -r backup
sudo groupadd -f developers
```

Practical notes:

- Use `-g` when you must match an existing policy or migrate data that depends on a specific GID.
- Use `-r` for service or daemon groups that should live in the system range.
- Use `-f` only when you want idempotent behavior for an already existing group name.

> `groupadd -f` mainly suppresses the error when the named group already exists. It is not a general-purpose fix for all duplicate or GID-collision scenarios.

**Checking group creation**

After creating a group, verify it with `getent`, `grep`, or `id`:

```bash
getent group developers
id alice
grep '^developers:' /etc/group
```

Prefer `getent group` because it respects the system’s name service configuration and works whether the group is local or provided by another identity source.

**Modifying groups with `groupmod`**

Use `groupmod` to change an existing group’s name or GID. This command updates the group database entries; it does not rename files on disk or change file ownership metadata automatically.

Common options:

- `-n NEWNAME` renames the group.
- `-g GID` changes the numeric GID.
- `-o` allows a non-unique GID when used with `-g`, if the system policy permits it.

Examples:

```bash
sudo groupmod -n devteam developers
sudo groupmod -g 2000 devteam
sudo groupmod -g 2000 -o devteam
```

Important behavior:

- Renaming a group changes the name in `/etc/group` and `/etc/gshadow`.
- Existing files owned by the old group keep their numeric GID until you change ownership.
- Users’ supplementary group membership is updated through the group database, but file metadata remains unchanged.

**Renaming a group safely**

A rename is often used when a team changes names but keeps the same access identity. The numeric GID is what matters for file ownership, so a rename alone usually preserves access to existing files.

Example:

```bash
sudo groupmod -n devteam developers
getent group devteam
```

After the rename, verify:

- The new name exists.
- The old name no longer resolves.
- Users still have the expected access through the same GID.

**Changing a group’s GID**

Changing the GID is more disruptive than renaming because file ownership stores the numeric GID, not the group name. If you change a group’s GID, files and directories that were owned by the old GID will not automatically follow.

Typical workflow:

1. Identify the current GID before changing it.
2. Change the group’s GID with `groupmod`.
3. Find files still owned by the old numeric GID.
4. Update ownership to the new group.

Example:

```bash
getent group devteam
sudo groupmod -g 2000 devteam
```

To remediate file ownership, use the old numeric GID value that existed before the change:

```bash
sudo find /srv/dev -gid 1050 -exec chgrp devteam {} +
```

This is the reliable approach because `-gid 1050` matches the historical numeric ownership, not the current group name.

A safer full remediation sequence looks like this:

```bash
oldgid=1050
newgroup=devteam

sudo groupmod -g 2000 "$newgroup"
sudo find /srv/dev -gid "$oldgid" -exec chgrp "$newgroup" {} +
```

Why this matters:

- `find -group NAME` depends on current name resolution and is not a reliable post-change remediation method.
- Numeric GID matching targets the actual ownership stored on disk.
- You should capture the old GID before the change, because once the group is modified, the old name may no longer resolve as expected.

> Do not rely on `find -group oldname` after a rename or GID change. Use the old numeric GID value instead.

**Deleting groups with `groupdel`**

Use `groupdel` to remove a local group when it is no longer needed.

```bash
sudo groupdel developers
```

Before deleting a group, verify that no users still depend on it as a primary or supplementary group. Deleting a group that is still assigned can leave users with broken group references or unexpected access behavior.

Useful checks:

```bash
getent group developers
getent passwd | awk -F: '$4 == 1050 {print $1, $4}'
id alice
```

The `getent passwd` check inspects the primary GID field numerically. This is more reliable than searching `/etc/passwd` for a group name, because `/etc/passwd` stores the primary group as a number, not a group name.

If you need to find users whose primary group is a specific GID:

```bash
gid=1050
getent passwd | awk -F: -v gid="$gid" '$4 == gid {print $1}'
```

If you need to find supplementary membership:

```bash
getent group developers
```

**Safe deletion workflow**

Use a deliberate sequence before removing a group:

1. Confirm the group exists.
2. Identify users with the group as primary or supplementary membership.
3. Reassign users or files if needed.
4. Delete the group.
5. Verify the group is gone.

Example:

```bash
getent group developers
getent passwd | awk -F: '$4 == 1050 {print $1}'
sudo groupdel developers
getent group developers
```

If the group is still in use, reassign users first:

```bash
sudo usermod -g devteam alice
sudo usermod -aG devteam bob
```

**Common exam-relevant scenarios**

Group administration on RHCSA often appears in practical tasks such as:

- Creating a shared project group.
- Renaming a group to match a new department name.
- Changing a GID to align with a standard.
- Cleaning up obsolete groups after migration.

A typical shared directory setup might look like this:

```bash
sudo groupadd projectx
sudo mkdir -p /srv/projectx
sudo chgrp projectx /srv/projectx
sudo chmod 2770 /srv/projectx
```

The `2` in `2770` sets the setgid bit on the directory so new files inherit the directory’s group. This is often useful for collaborative workspaces.

Verify the result:

```bash
ls -ld /srv/projectx
```

Expected behavior:

- The directory group is `projectx`.
- New files created inside inherit the group when setgid is set.
- Access is controlled by group permissions.

**Configuration files and defaults**

`groupadd` uses system defaults from `/etc/login.defs` when you do not specify a GID. Relevant settings may include:

- `GID_MIN`
- `GID_MAX`
- `SYS_GID_MIN`
- `SYS_GID_MAX`

Example inspection:

```bash
grep -E '^(GID_MIN|GID_MAX|SYS_GID_MIN|SYS_GID_MAX)' /etc/login.defs
```

These values help determine whether a group is treated as a regular group or a system group. Exact ranges can vary by distribution and policy.

**Troubleshooting and verification**

When a group operation does not behave as expected, check the following:

- The group name is unique.
- The GID is not already in use unless you intentionally allow it.
- The group exists in the local database or identity source you expect.
- File ownership still reflects the correct numeric GID after a rename or GID change.

Useful commands:

```bash
getent group
getent group devteam
id alice
ls -ln /srv/dev
```

Use `ls -n` or `ls -ln` when you want to inspect numeric ownership directly. This is especially helpful after GID changes because it shows the stored numeric IDs instead of resolving names.

**Quick command reference**

| Task | Command |
|---|---|
| Create a group | `groupadd developers` |
| Create with explicit GID | `groupadd -g 1050 developers` |
| Create a system group | `groupadd -r backup` |
| Rename a group | `groupmod -n devteam developers` |
| Change a GID | `groupmod -g 2000 devteam` |
| Delete a group | `groupdel devteam` |
| Verify group | `getent group devteam` |
| Find files by old GID | `find /srv/dev -gid 1050 -exec chgrp devteam {} +` |

**Key takeaways**

- `groupadd` creates local groups and can assign either automatic or explicit GIDs.
- `groupmod -n` renames the group in the account databases, not on-disk files.
- `groupmod -g` changes the numeric identity and requires file ownership remediation.
- `groupdel` should only be used after you confirm the group is no longer needed.
- Numeric GID checks are the most reliable way to handle post-change ownership cleanup.
