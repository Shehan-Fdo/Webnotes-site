## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

Linux user and group identity data is stored in four core text databases: `/etc/passwd`, `/etc/shadow`, `/etc/group`, and `/etc/gshadow`. On modern Red Hat systems, these files work together with Name Service Switch (`nsswitch.conf`) and Pluggable Authentication Modules (PAM) to resolve identities, authenticate users, and manage group membership.

For RHCSA, you need to understand what each file contains, how they relate, and which tools safely modify them. You should also know how to inspect them directly, because troubleshooting often requires checking the underlying records rather than relying only on higher-level commands.

**What each file stores**

- `/etc/passwd` stores account identity and login metadata.
- `/etc/shadow` stores password hashes and password aging data.
- `/etc/group` stores group names and membership lists.
- `/etc/gshadow` stores secure group administration data and, when used, group passwords.

These files are plain text, but you should not edit them casually. Use account-management commands such as `useradd`, `usermod`, `userdel`, `groupadd`, `groupmod`, `groupdel`, `passwd`, and `chage` so that related records stay consistent.

**`/etc/passwd` structure**

Each line in `/etc/passwd` represents one user account and contains seven colon-separated fields:

```text
name:password:UID:GID:GECOS:home:shell
```

- **name** is the login name.
- **password** is usually `x` on systems that use shadow passwords.
- **UID** is the numeric user ID.
- **GID** is the numeric primary group ID.
- **GECOS** is comment data, often the full name or contact info.
- **home** is the user’s home directory.
- **shell** is the login shell.

Example:

```text
alice:x:1001:1001:Alice Smith:/home/alice:/bin/bash
```

Important points:

- The file is world-readable because many programs need to map names to IDs.
- The password field is not the real password on shadow-enabled systems; it is typically `x`.
- System accounts often have low UIDs and may use shells such as `/usr/sbin/nologin` or `/bin/false`.
- The primary group is stored as a numeric GID, not a group name.

Useful inspection commands:

```bash
getent passwd alice
grep '^alice:' /etc/passwd
id alice
```

`getent passwd alice` is preferred when you want NSS-aware lookup, because it can include local files and other configured identity sources. `grep` against `/etc/passwd` is useful when you specifically want the local file contents.

**`/etc/shadow` structure**

`/etc/shadow` stores sensitive authentication data and is readable only by root and privileged tools. Each line contains nine colon-separated fields:

```text
name:password:lastchg:min:max:warn:inactive:expire:reserved
```

- **name** matches the login name in `/etc/passwd`.
- **password** contains the password hash or a lock marker.
- **lastchg** is the date of last password change, in days since the Unix epoch.
- **min** is the minimum number of days before a password can be changed again.
- **max** is the maximum password age.
- **warn** is the number of days before expiration when warnings begin.
- **inactive** is the number of days after expiration before the account is disabled.
- **expire** is the absolute account expiration date, in days since the epoch.
- **reserved** is unused.

Example:

```text
alice:$6$rounds=5000$saltsalt$hashedvalue...:19800:7:90:14:30:20000:
```

Common password field values:

- A valid hash, often starting with `$6$` for SHA-512 on many systems.
- `!` or `!!` to indicate a locked or not-yet-initialized password.
- `*` in some service accounts to prevent password-based login.

> Do not assume `passwd -u` always restores the original hash exactly. If multiple lock markers are present, the result can vary. Verify the final shadow entry after unlocking.

Useful inspection commands:

```bash
sudo grep '^alice:' /etc/shadow
sudo passwd -S alice
sudo chage -l alice
```

`passwd -S` gives a concise status summary, while `chage -l` shows aging information in a human-readable form.

**Password aging and account expiration**

Password aging controls how long a password remains valid and when the user must change it. Account expiration controls whether the account itself can still be used.

- **Password aging** affects authentication with a password.
- **Account expiration** disables the account after a date or inactivity period.
- **Inactive period** can disable logins after the password expires and remains unchanged.

Typical administration commands:

```bash
sudo chage -M 90 -m 7 -W 14 alice
sudo chage -E 2026-12-31 alice
sudo chage -d 0 alice
```

What these do:

- `-M 90` sets maximum password age to 90 days.
- `-m 7` prevents password changes for the first 7 days after a change.
- `-W 14` warns the user 14 days before expiration.
- `-E 2026-12-31` sets an account expiration date.
- `-d 0` forces a password change at next login.

You can also inspect the raw shadow aging fields, but `chage` is safer and clearer for administration.

**`/etc/group` structure**

`/etc/group` stores group definitions. Each line contains four colon-separated fields:

```text
group_name:password:GID:user_list
```

- **group_name** is the group name.
- **password** is usually `x` when shadow group support is used.
- **GID** is the numeric group ID.
- **user_list** is a comma-separated list of supplementary members.

Example:

```text
developers:x:1002:alice,bob
```

Important points:

- A user’s **primary group** is stored in `/etc/passwd` as the numeric GID.
- Supplementary group membership is listed in `/etc/group`.
- The `user_list` field is not the only source of membership; a user can also belong to a group through their primary GID.

Useful inspection commands:

```bash
getent group developers
grep '^developers:' /etc/group
id alice
groups alice
```

`id` is especially useful because it shows both the primary group and supplementary groups as the system sees them.

**`/etc/gshadow` structure**

`/etc/gshadow` is the secure companion to `/etc/group`. It stores group administration data and, if used, group passwords. On systems with shadow group support, `groupadd` typically creates both `/etc/group` and `/etc/gshadow` entries by default.

Each line contains four colon-separated fields:

```text
group_name:password:administrators:members
```

- **group_name** matches the group in `/etc/group`.
- **password** can hold a group password hash or lock marker.
- **administrators** lists users allowed to administer the group.
- **members** lists users who are members for group password access.

Example:

```text
developers:!:alice:alice,bob
```

In practice:

- Group passwords are rarely used on modern systems.
- The file is root-readable only.
- It matters most when you manage group administrators or use tools that update secure group metadata.

**How the files relate to each other**

The four files form a coordinated identity database:

- `/etc/passwd` maps usernames to UIDs, primary GIDs, home directories, and shells.
- `/etc/shadow` stores password hashes and aging for the same usernames.
- `/etc/group` maps group names to GIDs and supplementary members.
- `/etc/gshadow` stores secure group administration data for the same groups.

A login flow is conceptually:

1. The system resolves the username to a UID and primary group from identity sources.
2. Authentication checks the password hash and aging data if password login is used.
3. After authentication, the system determines group membership for the session and access control.

This is a useful mental model, but real systems may involve NSS, PAM, SSSD, LDAP, and cached identity data, so the exact sequence can vary.

**Safe administration commands**

Use account tools instead of editing files directly whenever possible.

```bash
sudo useradd alice
sudo usermod -aG developers alice
sudo usermod -s /usr/sbin/nologin backupsvc
sudo userdel -r olduser
sudo groupadd developers
sudo groupmod -n devteam developers
sudo groupdel oldgroup
sudo passwd alice
sudo passwd -l alice
sudo passwd -u alice
```

Notes:

- `useradd` creates the account and usually a matching primary group depending on system defaults.
- `usermod -aG` appends supplementary groups without removing existing ones.
- `-s /usr/sbin/nologin` is a common way to prevent interactive login for service accounts.
- `userdel -r` removes the home directory and mail spool in addition to the account.
- `passwd -l` locks the password by marking the shadow entry so password login is disabled.
- `passwd -u` unlocks the password, but you should verify the resulting shadow entry afterward.

> Avoid editing `/etc/passwd`, `/etc/shadow`, `/etc/group`, or `/etc/gshadow` by hand unless you are recovering a broken system and understand the consequences.

**Practical examples**

Create a service account with no interactive shell:

```bash
sudo useradd -r -s /usr/sbin/nologin backupsvc
sudo passwd -l backupsvc
```

Create a shared project group and add users:

```bash
sudo groupadd projectx
sudo usermod -aG projectx alice
sudo usermod -aG projectx bob
getent group projectx
```

Force a password change at next login:

```bash
sudo chage -d 0 alice
```

Set password aging policy:

```bash
sudo chage -M 60 -m 1 -W 7 alice
```

Inspect local identity records:

```bash
grep '^alice:' /etc/passwd
sudo grep '^alice:' /etc/shadow
grep '^projectx:' /etc/group
sudo grep '^projectx:' /etc/gshadow
```

**Troubleshooting and verification**

When a user cannot log in, check the account in layers:

- Confirm the account exists in `/etc/passwd` or through `getent passwd`.
- Confirm the password is valid and not locked in `/etc/shadow`.
- Confirm the shell is valid and not set to a non-login shell unintentionally.
- Confirm the home directory exists and has correct ownership.
- Confirm group membership if access to files or commands depends on it.

Useful commands:

```bash
id alice
passwd -S alice
chage -l alice
ls -ld /home/alice
getent passwd alice
sudo grep '^alice:' /etc/shadow
```

If a user can authenticate but cannot access resources, check group membership and file permissions:

```bash
id alice
getent group developers
ls -l /path/to/resource
```

If a service account should never log in interactively, verify the shell and password state:

```bash
getent passwd backupsvc
sudo passwd -S backupsvc
```

**Common exam points**

- Know the field order in each file.
- Know which file is public and which is protected.
- Know that `/etc/passwd` uses `x` when shadow passwords are enabled.
- Know that `/etc/shadow` stores hashes and aging data.
- Know that `/etc/group` stores supplementary membership.
- Know that `/etc/gshadow` supports secure group administration.
- Know how to inspect entries with `getent`, `grep`, `id`, `passwd`, and `chage`.
- Know how to lock and unlock accounts safely.
- Know how to add users to supplementary groups without removing existing memberships.

**Quick reference**

| File | Purpose | Typical access | Key fields |
|---|---|---|---|
| `/etc/passwd` | User identity and login metadata | World-readable | name, UID, GID, home, shell |
| `/etc/shadow` | Password hashes and aging | Root-only | hash, last change, min/max age |
| `/etc/group` | Group definitions and members | World-readable | group name, GID, member list |
| `/etc/gshadow` | Secure group admin data | Root-only | group password, admins, members |

**Mental model for RHCSA**

Think of the files in pairs:

- `/etc/passwd` and `/etc/shadow` describe users.
- `/etc/group` and `/etc/gshadow` describe groups.

Then remember the split between identity and secret data:

- Public identity data lives in `/etc/passwd` and `/etc/group`.
- Sensitive authentication and administration data lives in `/etc/shadow` and `/etc/gshadow`.

That separation is why Linux can let many programs read identity information while protecting password hashes and administrative controls.
