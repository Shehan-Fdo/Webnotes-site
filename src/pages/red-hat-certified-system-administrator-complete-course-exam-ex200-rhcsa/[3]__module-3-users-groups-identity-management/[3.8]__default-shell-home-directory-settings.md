## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

When you create local users on RHEL, the system applies defaults for the user’s login shell, home directory location, and initial account skeleton. These defaults matter because they determine whether a user can log in interactively, where their files live, and what baseline files appear in the new home directory.

For RHCSA, you should know both the command-line options that override defaults for a single user and the configuration files that define system-wide behavior. The safest exam approach is to understand what `useradd` does by default, how to create a home directory explicitly, and how to verify the resulting account settings.

**Default behavior of `useradd`**

`useradd` creates a new local account, but it does not always create a home directory unless you request it or the system is configured to do so. On standard RHEL systems, the common expectation is:

- The account gets a login shell from the system default.
- The home directory path is derived from the base home directory and the username.
- The home directory is not created unless you use `-m` or the system default is configured accordingly.
- The account skeleton files are copied from `/etc/skel` only when the home directory is created.

A typical manual creation looks like this:

```bash
sudo useradd -m alice
sudo passwd alice
```

This creates `/home/alice` by default, copies files from `/etc/skel`, and then sets a password so the account can authenticate.

If you omit `-m`, the user entry may exist in `/etc/passwd`, but the home directory is not created by default:

```bash
sudo useradd bob
getent passwd bob
ls -ld /home/bob
```

That distinction is important on the exam: account creation and home directory creation are related, but not identical.

**Default shell settings**

The login shell is the program started when a user logs in interactively. On RHEL, the conventional non-login shell for disabled accounts is `/sbin/nologin`.

Common shell choices include:

- `/bin/bash` for normal interactive users
- `/sbin/nologin` for accounts that should not provide shell access
- `/bin/false` for accounts that should also fail login, though `/sbin/nologin` is the expected RHEL answer in most admin tasks

A user with a shell like `/sbin/nologin` can still exist for services or special accounts, but cannot open an interactive shell session.

Example of creating a user with a disabled interactive shell:

```bash
sudo useradd -m -s /sbin/nologin svcbackup
sudo passwd svcbackup
```

If you need to inspect the shell assigned to an account, check `/etc/passwd` or use `getent`:

```bash
getent passwd svcbackup
```

The shell is the last field in the passwd entry.

**Home directory location and creation**

The home directory path is usually built from a base directory plus the username. On RHEL, the default base is commonly `/home`, so a user named `alice` gets `/home/alice`.

You can override the home directory for a single user with `-d`:

```bash
sudo useradd -m -d /srv/users/alice alice
```

This creates the account with `/srv/users/alice` as the home directory. If you want the directory created and populated with skeleton files, keep `-m`.

Key points:

- `-d` sets the path.
- `-m` creates the directory if it does not already exist.
- Without `-m`, the path can be recorded in the account database without creating the directory.
- If the directory already exists, ownership and permissions still need to be correct.

A practical verification sequence:

```bash
getent passwd alice
ls -ld /srv/users/alice
```

**Skeleton files and initial user environment**

When `useradd -m` creates a home directory, it copies default startup files from `/etc/skel`. These files provide the initial shell environment for new users.

Typical contents may include:

- `.bashrc`
- `.bash_profile`
- `.bash_logout`

The exact files depend on the system configuration. If you customize `/etc/skel`, new users created afterward inherit those changes.

Example:

```bash
sudo cp /etc/skel/.bashrc /etc/skel/.bashrc.backup
sudo vi /etc/skel/.bashrc
```

Use this carefully: changes to `/etc/skel` affect only future accounts, not existing ones.

**System-wide defaults for new users**

RHEL stores default `useradd` settings in `/etc/default/useradd`. This file is the exam-safe place to review and adjust the standard defaults for new local users.

View the current defaults with:

```bash
useradd -D
cat /etc/default/useradd
```

Typical values may include:

- `GROUP`
- `HOME`
- `SHELL`
- `SKEL`
- `INACTIVE`
- `EXPIRE`

A representative file might look like this:

```ini
GROUP=100
HOME=/home
INACTIVE=-1
EXPIRE=
SHELL=/bin/bash
SKEL=/etc/skel
CREATE_MAIL_SPOOL=yes
```

Important exam note:

> Do not assume `useradd -D` is the universal method to change defaults on every system. For RHCSA study purposes, treat `/etc/default/useradd` as the primary configuration source and use `useradd -D` mainly to display or confirm defaults.

If you need to change the default shell for future users, edit the file directly:

```bash
sudo vi /etc/default/useradd
```

Then set:

```ini
SHELL=/bin/bash
HOME=/home
SKEL=/etc/skel
```

This affects future `useradd` operations, not existing users.

**Login definitions and related configuration**

The file `/etc/login.defs` influences several account-creation behaviors. For home directories, the most relevant setting is `HOME_MODE`, which can define the permissions applied when a home directory is created.

Example conceptually:

```ini
HOME_MODE 0750
```

This helps standardize the mode of newly created home directories when the system applies that setting. In practice, the final permissions can also be affected by the creation process and the user’s umask when files are created inside the home directory.

Other settings in `/etc/login.defs` may affect account aging, UID ranges, and password behavior, but for this topic focus on:

- `HOME_MODE` for home directory permissions
- UID/GID allocation ranges
- password aging defaults

Do not confuse `HOME_MODE` with the user’s shell or with the permissions of files created later inside the home directory.

**Practical account creation examples**

Create a standard interactive user with a home directory:

```bash
sudo useradd -m -s /bin/bash devops1
sudo passwd devops1
```

Create a service account without interactive login:

```bash
sudo useradd -r -s /sbin/nologin -M backupsvc
```

Here:

- `-r` creates a system account
- `-s /sbin/nologin` blocks interactive login
- `-M` prevents home directory creation

Create a user with a custom home path:

```bash
sudo useradd -m -d /data/home/jane -s /bin/bash jane
sudo passwd jane
```

Verify the result:

```bash
getent passwd jane
ls -ld /data/home/jane
```

**Verification and troubleshooting**

After creating a user, verify three things:

1. The account exists in the identity database.
2. The shell is what you intended.
3. The home directory exists and has the correct ownership and mode.

Useful commands:

```bash
getent passwd alice
id alice
ls -ld /home/alice
```

If the home directory is missing, check whether you used `-m` or whether the system default is configured to create homes automatically.

If the shell is wrong, inspect `/etc/default/useradd` and the explicit `-s` option used during account creation.

If the home directory exists but permissions are wrong, fix ownership and mode:

```bash
sudo chown -R alice:alice /home/alice
sudo chmod 750 /home/alice
```

Use `chmod` carefully; the correct mode depends on your security policy and the intended access model.

**Common RHCSA patterns**

You should be comfortable with these patterns:

- Create a user with a home directory and Bash shell.
- Create a user with a custom home directory.
- Create a non-login service account.
- Confirm system defaults for future users.
- Distinguish between account creation and home directory creation.

A quick reference workflow:

```bash
sudo useradd -m -s /bin/bash user1
sudo passwd user1
getent passwd user1
ls -ld /home/user1
```

For a disabled account:

```bash
sudo useradd -m -s /sbin/nologin user2
sudo passwd user2
getent passwd user2
```

**What to remember for the exam**

- `useradd` creates the account entry; `-m` creates the home directory.
- `/etc/default/useradd` controls defaults for future users.
- `useradd -D` is best treated as a display/check command unless you know the exact supported options on that system.
- `/etc/login.defs` can influence home directory mode through `HOME_MODE`.
- `/sbin/nologin` is the standard RHEL choice for a non-interactive shell.
- `/etc/skel` supplies initial files for new home directories.

**Mini checklist**

- Confirm the shell with `getent passwd <user>`.
- Confirm the home directory with `ls -ld <home>`.
- Use `-m` when you want the directory created.
- Use `-d` when you want a custom home path.
- Edit `/etc/default/useradd` for future default changes.
- Use `/sbin/nologin` for accounts that should not log in interactively.
