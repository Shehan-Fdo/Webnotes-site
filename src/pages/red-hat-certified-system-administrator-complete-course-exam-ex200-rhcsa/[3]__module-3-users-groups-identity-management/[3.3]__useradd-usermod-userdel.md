## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

`useradd`, `usermod`, and `userdel` are the core shadow-utils commands for managing local user accounts on RHEL-based systems. In RHCSA tasks, you use them to create accounts, adjust account properties, and remove accounts cleanly.

These commands affect local identity data stored in files such as `/etc/passwd`, `/etc/shadow`, `/etc/group`, and `/etc/gshadow`. They do not manage directory services like LDAP or Active Directory.

- `useradd` creates a new local account.
- `usermod` changes an existing account.
- `userdel` removes an account and, optionally, its home directory and mail spool.
- `passwd` is often used alongside them to set or lock passwords.

**Local account files and what they store**

Understanding the account database helps you predict command behavior and troubleshoot issues.

- `/etc/passwd` stores user name, UID, GID, home directory, and login shell.
- `/etc/shadow` stores password hashes and password aging data.
- `/etc/group` stores group membership.
- `/etc/gshadow` stores secure group administration data.
- `/etc/default/useradd` and `/etc/login.defs` provide defaults for new accounts on many systems, but exact behavior can vary by distribution and version.

A typical account entry looks like this:

```text
alice:x:1001:1001:Alice Smith:/home/alice:/bin/bash
```

The fields are:

- `alice` is the login name.
- `x` indicates the password hash is in `/etc/shadow`.
- `1001` is the UID.
- `1001` is the primary GID.
- `/home/alice` is the home directory.
- `/bin/bash` is the login shell.

**Creating users with `useradd`**

`useradd` creates a new account and, depending on defaults and options, may create a home directory and copy skeleton files from `/etc/skel`.

Common usage:

```bash
sudo useradd alice
sudo passwd alice
```

This creates the account and then sets a password so the user can log in.

Useful options include:

- `-m` creates the home directory if it is not created by default.
- `-d /path` sets a custom home directory.
- `-s /bin/bash` sets the login shell.
- `-u 1050` sets a specific UID.
- `-g group` sets the primary group.
- `-G group1,group2` adds supplementary groups.
- `-c "Full Name"` sets the comment field, often used for the real name.
- `-r` creates a system account with a lower UID range.
- `-M` prevents home directory creation.
- `-k /etc/skel-custom` copies skeleton files from a custom directory when used with `-m`.

Example:

```bash
sudo useradd -m -d /home/alice -s /bin/bash -c "Alice Smith" alice
sudo passwd alice
```

This creates `alice` with a home directory, Bash shell, and comment field.

**How defaults influence `useradd`**

Defaults are useful when you want consistent account creation across a system.

- `useradd -D` displays the current defaults.
- `useradd -D -s /bin/bash` changes the default shell for future accounts.
- `useradd -D -b /home` changes the default base home directory.
- `useradd -D -f 30` sets the password inactivity period default in some environments.

Example:

```bash
useradd -D
```

Typical output may include the default home base, shell, skeleton directory, and group behavior.

> Be careful with defaults: they affect future accounts, not existing ones.

**Home directory creation and skeleton files**

When a home directory is created, `useradd` can populate it from `/etc/skel`.

- Files in `/etc/skel` are copied into the new home directory.
- This is useful for default shell profiles, aliases, and startup scripts.
- The permissions of copied files are influenced by the system defaults and the source file modes.

Example:

```bash
ls -la /etc/skel
```

Common files include:

- `.bashrc`
- `.bash_profile`
- `.bash_logout`

If you need a custom starting environment, place carefully chosen files in `/etc/skel` before creating the account.

**Passwords and password hashes**

`useradd` does not usually set a usable password by itself. You normally set one with `passwd`.

```bash
sudo passwd alice
```

For automation, you may see `usermod -p`, but it expects a full encrypted password hash string exactly as generated by a password hashing tool. If you provide a placeholder or malformed value, authentication will fail.

Example of generating a hash:

```bash
openssl passwd -6
```

Then use the resulting hash with care:

```bash
sudo usermod -p '$6$rounds=5000$...real-generated-hash...' alice
```

- The hash must be complete and valid.
- The command does not encrypt a plain-text password for you.
- For exam practice, prefer `passwd` unless the task explicitly requires a precomputed hash.

**Modifying users with `usermod`**

`usermod` changes account attributes after creation. It is commonly used for renaming accounts, moving home directories, changing shells, and adjusting group membership.

Common examples:

```bash
sudo usermod -s /bin/zsh alice
sudo usermod -d /home/alice.new alice
sudo usermod -aG wheel alice
```

Key options:

- `-l newname` renames the login name.
- `-d /new/home` changes the home directory path.
- `-m` moves the contents of the old home directory to the new location when used with `-d`.
- `-s /bin/bash` changes the login shell.
- `-g group` changes the primary group.
- `-G group1,group2` sets supplementary groups.
- `-a` appends groups when used with `-G`; without `-a`, the listed groups replace the current supplementary group list.
- `-c "New Comment"` changes the comment field.
- `-L` locks the password.
- `-U` unlocks the password.
- `-e YYYY-MM-DD` sets an account expiration date.
- `-f DAYS` sets the number of inactive days after password expiration before the account is disabled.

**Renaming an account safely**

Renaming a user changes the login name in account databases, but it does not automatically fix every related path or running session.

```bash
sudo usermod -l a.smith alice
```

After this:

- The account name changes in `/etc/passwd` and related files.
- Existing processes and open sessions may continue under the old login context until they end.
- File ownerships on disk still reference the numeric UID, not the name, so ownership usually remains intact if the UID does not change.
- Any scripts, cron jobs, mail spools, or application configs that reference the old name may need manual updates.

If you also need to move the home directory:

```bash
sudo usermod -d /home/a.smith -m a.smith
```

This changes the home directory path and moves the contents from the old home to the new one.

**Changing group membership**

Supplementary groups control access to shared resources such as admin tools, shared project directories, and device permissions.

Add a user to additional groups without removing existing ones:

```bash
sudo usermod -aG wheel,developers alice
```

Important behavior:

- `-a` means append.
- `-G` without `-a` replaces the supplementary group list.
- Group membership changes usually take effect in new logins, not in already-open shells.

Verify membership with:

```bash
id alice
groups alice
```

**Changing the primary group**

The primary group is the default group for new files created by the user.

```bash
sudo usermod -g developers alice
```

Use this when you want a user’s default file ownership to align with a project or team group.

**Locking and unlocking accounts**

Locking an account prevents password-based login without deleting the account.

```bash
sudo usermod -L alice
sudo usermod -U alice
```

You can also use `passwd -l alice` and `passwd -u alice` on many systems. For RHCSA, know that locking disables password authentication while preserving the account.

**Expiring accounts**

Account expiration is useful for temporary users, contractors, or lab accounts.

```bash
sudo usermod -e 2026-12-31 alice
```

This sets a date after which the account cannot be used for login.

To remove expiration:

```bash
sudo usermod -e '' alice
```

**Deleting users with `userdel`**

`userdel` removes a local account from the system databases.

Basic usage:

```bash
sudo userdel alice
```

This removes the account entry, but it does not necessarily remove the home directory or mail spool unless you ask for it.

Common options:

- `-r` removes the home directory and mail spool.
- `-f` forces removal in some situations, but its exact behavior can vary by implementation and should not be relied on as a routine exam tool.
- `-Z` may remove SELinux user mapping on some systems, but this is not a primary RHCSA focus.

Example:

```bash
sudo userdel -r alice
```

This is the usual choice when you want to remove the account and its home directory.

> Use `userdel -f` cautiously. Its behavior is implementation-dependent and is not a substitute for understanding whether the account is logged in or whether files still exist.

**What `userdel -r` removes**

When you use `-r`, the command typically removes:

- The user’s home directory.
- The user’s mail spool.
- The account entry from local databases.

It does not automatically remove every file owned by that UID elsewhere on the system.

To find remaining files owned by a deleted UID:

```bash
sudo find / -xdev -uid 1001
```

This is useful when you need to clean up project files, application data, or stray ownerships after account removal.

**Practical workflow examples**

Create a new user, set a password, and add to `wheel`:

```bash
sudo useradd -m -s /bin/bash alice
sudo passwd alice
sudo usermod -aG wheel alice
```

Rename a user and move the home directory:

```bash
sudo usermod -l a.smith alice
sudo usermod -d /home/a.smith -m a.smith
```

Remove a temporary account and its home:

```bash
sudo userdel -r tempuser
```

**Verification commands**

Always verify the result after account changes.

```bash
getent passwd alice
id alice
ls -ld /home/alice
sudo grep '^alice:' /etc/shadow
```

Useful checks:

- `getent passwd USER` confirms the account entry.
- `id USER` shows UID, primary group, and supplementary groups.
- `ls -ld /home/USER` confirms home directory ownership and existence.
- `grep` on `/etc/shadow` helps confirm password status when needed.

**Common pitfalls**

- `useradd` creates the account, but it does not always set a usable password.
- `usermod -G` without `-a` replaces supplementary groups instead of adding to them.
- Renaming a user does not automatically update every file path, script, or session.
- `userdel -r` removes the home directory, but not all files owned by the UID elsewhere.
- `usermod -p` requires a valid encrypted hash, not plain text and not a placeholder.
- Group membership changes usually require a new login session to take effect.

**Exam-focused tips**

RHCSA tasks often ask you to create, modify, and remove users with exact properties. Read the prompt carefully and verify each requirement.

- Use `useradd` for initial creation.
- Use `usermod` for later adjustments.
- Use `userdel -r` when the task explicitly says to remove the home directory.
- Use `id`, `getent`, and `ls -ld` to confirm your work.
- Prefer commands that make the requested change directly instead of editing files manually unless the task requires it.

**Quick reference**

| Task | Command |
|---|---|
| Create user | `useradd alice` |
| Create user with home and shell | `useradd -m -s /bin/bash alice` |
| Set password | `passwd alice` |
| Add supplementary groups | `usermod -aG wheel,developers alice` |
| Change login shell | `usermod -s /bin/bash alice` |
| Rename user | `usermod -l a.smith alice` |
| Move home directory | `usermod -d /home/a.smith -m a.smith` |
| Lock account | `usermod -L alice` |
| Unlock account | `usermod -U alice` |
| Delete user | `userdel alice` |
| Delete user and home | `userdel -r alice` |

**Troubleshooting notes**

If a user cannot log in, check the account state in this order:

1. Confirm the account exists with `getent passwd USER`.
2. Check whether the password is locked with `passwd -S USER` or by inspecting `/etc/shadow`.
3. Verify the shell is valid and listed in `/etc/shells`.
4. Confirm the home directory exists and has the correct ownership.
5. Check group membership if access to resources is failing.

If a rename or move does not behave as expected, remember that open sessions, running processes, and external references may still point to the old identity or path until you update them manually.
