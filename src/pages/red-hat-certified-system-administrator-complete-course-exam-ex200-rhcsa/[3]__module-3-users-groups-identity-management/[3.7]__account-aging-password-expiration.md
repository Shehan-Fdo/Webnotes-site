## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Account aging overview**

Account aging controls how long a password remains valid, when users must change it, and when an account becomes inactive after expiration. On Red Hat Enterprise Linux, these settings help enforce security policy without manually tracking every user.

The core idea is simple:

- **Password aging** governs password lifetime.
- **Password expiration** forces a change after the lifetime ends.
- **Account inactivity** can disable logins after a grace period.
- **Password aging data** is stored in `/etc/shadow`.

A typical RHCSA task is to inspect or modify these values for a user account and verify the result with `chage` or `passwd`.

**Shadow file and aging fields**

Linux stores password-related metadata in `/etc/shadow`, not in `/etc/passwd`. The shadow entry includes aging fields that define when a password was last changed and how long it remains valid.

A shadow line conceptually looks like this:

```text
username:password:lastchg:min:max:warn:inactive:expire:reserved
```

Key fields:

- **lastchg**: Days since the Unix epoch when the password was last changed.
- **min**: Minimum number of days before the password can be changed again.
- **max**: Maximum number of days the password is valid.
- **warn**: Number of days before expiration to warn the user.
- **inactive**: Days after password expiration before the account is disabled.
- **expire**: Absolute account expiration date.
- **reserved**: Unused.

You usually do not edit `/etc/shadow` directly. Use `chage`, `passwd`, or user management tools so the system updates the file safely.

**Inspecting password aging**

Use `chage` to view aging settings for a user. This is the most exam-relevant inspection tool.

```bash
chage -l alice
```

Typical output includes:

- **Last password change**
- **Password expires**
- **Password inactive**
- **Account expires**
- **Minimum number of days between password change**
- **Maximum number of days between password change**
- **Number of days of warning before password expires**

You can also inspect the shadow entry directly, but that is less readable:

```bash
sudo grep '^alice:' /etc/shadow
```

Use `passwd -S` for a compact status view:

```bash
passwd -S alice
```

This shows whether the password is set, locked, or expired, plus aging information in a terse format.

**Setting password aging with `chage`**

`chage` is the primary tool for changing aging policy on a per-user basis.

Common options:

- `-m` sets the minimum days between password changes.
- `-M` sets the maximum days a password remains valid.
- `-W` sets warning days before expiration.
- `-I` sets inactive days after expiration before lockout.
- `-E` sets an absolute account expiration date.
- `-d` sets the last password change date.
- `-l` lists current settings.

Example: require password changes every 90 days, warn 7 days before expiry, and allow 14 inactive days after expiration.

```bash
sudo chage -M 90 -W 7 -I 14 alice
```

Example: force a password change at next login by setting the last change date to the epoch.

```bash
sudo chage -d 0 alice
```

That makes the password appear immediately expired, so the user must change it on next authentication.

Example: set an account expiration date.

```bash
sudo chage -E 2026-12-31 alice
```

Use account expiration when you want the entire account to stop working on a specific date, such as for contractors or temporary access.

**Using `passwd` for expiration and locking**

`passwd` can also manage some account state.

Lock a password:

```bash
sudo passwd -l alice
```

Unlock a password:

```bash
sudo passwd -u alice
```

Locking a password prevents password-based login, but it does not necessarily disable all access methods. For example, key-based SSH or other authentication paths may still work depending on configuration.

Force password expiration for a user:

```bash
sudo passwd -e alice
```

This is similar in effect to setting the last change date to force a change at next login.

Check status:

```bash
passwd -S alice
```

**Default aging policy for new users**

System-wide defaults influence new accounts. On Red Hat systems, these defaults are commonly controlled through `/etc/login.defs` and the `useradd` defaults.

Important settings include:

- `PASS_MAX_DAYS`
- `PASS_MIN_DAYS`
- `PASS_WARN_AGE`

Example excerpt:

```conf
PASS_MAX_DAYS   90
PASS_MIN_DAYS   7
PASS_WARN_AGE   7
```

These values affect newly created accounts, but they do not automatically rewrite existing users. For existing accounts, use `chage` or a scripted policy update.

You can inspect defaults for `useradd` with:

```bash
useradd -D
```

This shows the default home directory, shell, skeleton directory, and related account creation settings. Aging defaults are often managed separately through login policy files.

**Practical user lifecycle examples**

Aging settings are useful in real administration scenarios:

- **Employee accounts**: Set a 90-day password rotation and warning period.
- **Temporary contractors**: Set an account expiration date with `-E`.
- **Training labs**: Force a password reset on first login with `-d 0`.
- **Dormant accounts**: Use inactivity settings to disable accounts after expiration.

Example workflow for a new contractor account:

```bash
sudo useradd contractor1
sudo passwd contractor1
sudo chage -M 60 -W 10 -I 7 -E 2026-09-30 contractor1
```

This creates the account, sets the password, and applies a limited lifetime with warnings and inactivity lockout.

**Understanding expiration behavior**

Password expiration and account expiration are related but different:

- **Password expiration** means the user must change the password before continuing.
- **Account expiration** means the account cannot be used after the specified date.
- **Inactive period** applies after password expiration and before the account is fully disabled.

A user may still log in if the password is expired but the account is not, provided the system allows a password change during authentication. If the account itself is expired, login fails regardless of password status.

> Be careful not to confuse password lockout with account expiration. A locked password blocks password authentication, while an expired account blocks the account entirely.

**Troubleshooting and verification**

When a user reports login problems, verify the aging state before changing anything else.

1. Check the account status.
   ```bash
   passwd -S alice
   chage -l alice
   ```
2. Confirm whether the password is expired, locked, or inactive.
3. Review `/etc/shadow` only if you need to confirm the raw stored values.
4. If needed, reset the aging policy.
   ```bash
   sudo chage -M 90 -W 7 -I 14 alice
   sudo chage -d $(date +%F) alice
   ```

If a user cannot log in after a policy change, check these common causes:

- The account is locked with `passwd -l`.
- The account expiration date has passed.
- The password expired and the user cannot complete a change.
- The inactive period elapsed after expiration.
- The user is subject to a separate authentication policy such as PAM or directory services.

**Command reference**

| Task | Command |
|---|---|
| View aging details | `chage -l user` |
| Set max password age | `chage -M 90 user` |
| Set warning period | `chage -W 7 user` |
| Set inactivity period | `chage -I 14 user` |
| Set account expiration | `chage -E 2026-12-31 user` |
| Force password change | `chage -d 0 user` |
| Lock password | `passwd -l user` |
| Unlock password | `passwd -u user` |
| Show password status | `passwd -S user` |

**Exam-focused reminders**

- Use `chage` for aging policy changes.
- Use `passwd` for password lock and status checks.
- Remember that `/etc/shadow` stores the aging metadata.
- Know the difference between password expiration, inactivity, and account expiration.
- Verify changes with `chage -l` after applying them.

**Quick practical example**

If you need to configure a user so the password expires every 45 days, warns 5 days before expiration, and disables the account 10 days after expiration:

```bash
sudo chage -M 45 -W 5 -I 10 username
sudo chage -l username
```

This is the kind of task you should be able to complete quickly and verify confidently during RHCSA work.
