## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Group fundamentals**

Linux uses groups to simplify access control. Instead of assigning permissions to each user individually, you assign ownership and permissions to a group, then place users into that group. This is especially important on RHCSA systems where you must manage shared directories, service accounts, and collaborative workflows.

A user can belong to:
- a **primary group**, which is the default group associated with new files the user creates
- one or more **supplementary groups**, which grant additional access beyond the primary group

The distinction matters because group membership affects both file creation and authorization checks. A user may have access through a supplementary group even when that group is not the user’s default group.

**Primary group behavior**

A user’s primary group is stored in `/etc/passwd` as a numeric group ID (GID). The group name itself is resolved through `/etc/group` or directory services such as LDAP if configured.

Typical examples:
- A newly created user often gets a private primary group with the same name as the user
- Shared accounts may use a project or team group as the primary group
- File creation uses the user’s effective group ID, so new files usually inherit the primary group unless special directory settings override it

You can inspect a user’s primary group with:
```bash
id alice
getent passwd alice
```

Example output pattern:
```bash
uid=1001(alice) gid=1001(alice) groups=1001(alice),1005(projectx),10(wheel)
alice:x:1001:1001:Alice Example:/home/alice:/bin/bash
```

Here:
- `gid=1001(alice)` is the primary group
- `groups=...` includes both the primary group and supplementary groups
- `projectx` and `wheel` are additional groups available to the user

**Supplementary group behavior**

Supplementary groups provide extra access without changing the user’s default group. They are commonly used for:
- shared project directories
- administrative access such as `wheel`
- application-specific access like `docker`, `libvirt`, or `storage` when present on the system

A user can be listed as a member of a supplementary group in `/etc/group`, but that does not mean the group is the user’s primary group. Also, a user may have a primary group that is not listed in the member field of `/etc/group` because primary membership is defined by the user’s GID, not by the group member list.

Check group membership with:
```bash
getent group projectx
id alice
groups alice
```

Example:
```bash
projectx:x:1005:alice,bob,carol
```

This means `alice`, `bob`, and `carol` are explicitly listed as members of `projectx`. It does not by itself describe their primary groups.

**Creating users with group assignments**

When creating a user, you can set the primary group and supplementary groups at the same time.

Common `useradd` patterns:
```bash
# Create a user with a specific primary group
useradd -g projectx alice

# Create a user and add supplementary groups
useradd -G wheel,accounting eric

# Create a user with a home directory and custom primary group
useradd -m -g developers maya
```

Important details:
- `-g` sets the **primary group**
- `-G` sets **supplementary groups**
- `useradd` does not append to existing supplementary groups; it defines the initial set at account creation
- If you want to add groups later, use `usermod -aG`

Example:
```bash
usermod -aG wheel,accounting eric
```

> Be careful with `usermod -G` without `-a`: it replaces the user’s supplementary group list instead of appending to it.

**Modifying group membership**

Use `usermod` to change a user’s group relationships after creation.

Common commands:
```bash
# Change the primary group
usermod -g projectx alice

# Add supplementary groups without removing existing ones
usermod -aG wheel,accounting eric

# Replace all supplementary groups
usermod -G developers,qa eric
```

Practical implications:
- Changing the primary group affects the default group for newly created files after the change
- Adding a supplementary group grants access only where that group has permissions
- Replacing supplementary groups can accidentally remove access, so verify carefully before applying it on production systems

**Creating and managing groups**

Groups are managed with `groupadd`, `groupmod`, and `groupdel`.

Examples:
```bash
groupadd projectx
groupadd -g 2005 accounting
groupmod -n finance accounting
groupdel oldteam
```

Useful checks:
```bash
getent group projectx
getent group finance
```

On RHCSA systems, `getent` is preferred because it queries the system’s name service switch configuration and works with local files or centralized identity sources.

**File ownership and group access**

Every file has an owner user and an owner group. Group permissions determine what members of that group can do with the file.

Example:
```bash
ls -l report.txt
```

Output pattern:
```bash
-rw-rw---- 1 alice projectx 4096 Apr 26 10:15 report.txt
```

Interpretation:
- `alice` owns the file
- `projectx` is the owning group
- members of `projectx` can read and write because group permissions are `rw-`

To change group ownership:
```bash
chgrp projectx report.txt
# or
chown :projectx report.txt
```

To change both owner and group:
```bash
chown alice:projectx report.txt
```

**Default group inheritance and shared directories**

A shared directory often uses a group owner and the setgid bit so that new files inherit the directory’s group instead of the creator’s primary group.

Example:
```bash
mkdir /srv/projectx
chgrp projectx /srv/projectx
chmod 2770 /srv/projectx
```

What this does:
- `2` sets the setgid bit on the directory
- new files and subdirectories inherit the `projectx` group
- only the owner and group can access the directory if permissions are `770`

Verify with:
```bash
ls -ld /srv/projectx
```

Example output:
```bash
drwxrws--- 2 root projectx 4096 Apr 26 10:20 /srv/projectx
```

The `s` in the group execute position indicates setgid on the directory.

**Login sessions and group changes**

Group membership changes do not always appear immediately in an existing shell session. In practice:
- new supplementary group membership is usually recognized at the next login
- the current session may still use the old group list
- `newgrp` can start a subshell with a different primary group for testing or temporary access

Examples:
```bash
id
newgrp projectx
id
```

This distinction is important during troubleshooting. If you add a user to `wheel` and they still cannot use `sudo`, they may need to log out and back in before the new group membership is applied to their session.

**Common RHCSA verification workflow**

When you troubleshoot group access, verify identity in layers:
1. Confirm the user exists.
2. Confirm the group exists.
3. Confirm the user is listed in the correct supplementary groups.
4. Confirm the file or directory permissions allow the intended access.
5. Confirm the user has started a new session after membership changes.

Useful commands:
```bash
id alice
groups alice
getent passwd alice
getent group projectx
ls -ld /srv/projectx
```

**Practical examples**

Create a shared project group and assign users:
```bash
groupadd projectx
useradd -m -g projectx alice
useradd -m -G projectx bob
usermod -aG projectx carol
```

Check membership:
```bash
id alice
id bob
id carol
getent group projectx
```

Create a collaborative directory:
```bash
mkdir -p /srv/projectx
chgrp projectx /srv/projectx
chmod 2770 /srv/projectx
```

Test access:
```bash
su - alice
touch /srv/projectx/alice-file
ls -l /srv/projectx/alice-file
```

If the directory has setgid enabled, the new file should inherit the `projectx` group.

**Troubleshooting tips**

- If a user cannot access a shared resource, check both the group membership and the permission bits.
- If `id` shows the expected group but access still fails, verify the directory or file group owner and mode.
- If you recently changed group membership, start a new login session or use `newgrp` to test the updated group context.
- If you used `usermod -G` without `-a`, you may have overwritten existing supplementary groups.
- If a group appears in `getent group` with members listed, that list reflects explicit supplementary membership in the group database; it does not fully describe primary group relationships.

**Exam-focused summary**

For RHCSA, you should be able to:
- distinguish primary from supplementary groups
- create users with `useradd -g` and `useradd -G`
- modify memberships with `usermod -g`, `usermod -aG`, and `usermod -G`
- inspect membership with `id`, `groups`, `getent passwd`, and `getent group`
- manage group ownership with `chgrp` and `chown`
- use setgid on shared directories to preserve group inheritance

Mastering these basics helps you build secure shared access patterns and troubleshoot permission problems quickly.
