## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Identity, groups, and directory lookups**

Linux authorization often depends on who you are and which groups you belong to. On Red Hat systems, you verify that identity with `id`, inspect group membership with `groups`, and query account databases with `getent`. These commands are essential when troubleshooting file access, sudo behavior, shared directories, and service accounts.

At the RHCSA level, you should be able to read their output quickly and understand the difference between:
- the current effective identity of a process,
- the groups assigned to a user,
- and the account information stored in local or network identity sources.

**Core concepts**

`id` reports numeric and name-based identity information for a user. It shows the user ID (`uid`), primary group ID (`gid`), and supplementary groups. This is the most complete quick check when you need to know what permissions a shell session likely has.

`groups` focuses on group membership only. It is simpler than `id` and useful when you want a fast answer to “which groups does this user belong to?”

`getent` means “get entries” and queries system databases through the Name Service Switch (`NSS`). That matters because the system may use local files, LDAP, SSSD, or other identity sources. `getent` gives you a consistent way to ask the system for users, groups, hosts, services, and more.

**`id` command**

Use `id` to inspect the identity of the current user or another account. It is especially useful when you need to confirm whether a user has the expected primary group and supplementary groups.

```bash
id
id alice
id -u alice
id -g alice
id -G alice
id -nG alice
```

Common output fields:
- `uid=`: user ID number and username
- `gid=`: primary group ID number and group name
- `groups=`: all groups the user belongs to, including the primary group
- `context=`: SELinux context, if SELinux is enabled and the command includes it in the environment

Useful options:
- `-u` shows only the numeric user ID.
- `-g` shows only the numeric primary group ID.
- `-G` shows all group IDs.
- `-n` converts IDs to names when used with `-u`, `-g`, or `-G`.
- `-r` shows real IDs instead of effective IDs on systems where that distinction matters.

Practical interpretation:
- If a user cannot read a file, compare the file’s ownership and mode bits against the output of `id`.
- If a user should access a shared directory, confirm that the expected group appears in `id` output.
- If a command runs with elevated privileges, the effective identity may differ from the login identity.

Example:
```bash
$ id alice
uid=1001(alice) gid=1001(alice) groups=1001(alice),10(wheel),1005(projects)
```

This tells you:
- `alice` has a primary group named `alice`
- `alice` is also in `wheel`, which often grants sudo access on Red Hat systems
- `alice` is in `projects`, which may control access to shared files

**`groups` command**

Use `groups` when you only need group membership. It is shorter than `id` and often easier to scan in scripts or quick checks.

```bash
groups
groups alice
groups root
```

Typical output:
```bash
alice : alice wheel projects
```

Important behavior:
- With no argument, `groups` shows the groups for the current user.
- With a username, it shows that user’s groups.
- The primary group is included in the list.
- Output is human-friendly, but `id -nG` is often more flexible for scripting.

When to prefer `groups`:
- You want a quick membership check during troubleshooting.
- You are teaching or documenting group-based access.
- You do not need numeric IDs.

When to prefer `id`:
- You need both user and group information.
- You need numeric IDs for file ownership or ACL troubleshooting.
- You want a more complete identity snapshot.

**`getent` and NSS lookups**

`getent` queries databases through NSS, so it reflects the system’s configured identity sources rather than only local files. This is important on enterprise systems where users and groups may come from `/etc/passwd`, `/etc/group`, LDAP, or SSSD.

Common databases:
- `passwd` for user accounts
- `group` for groups
- `shadow` for password aging data, usually restricted
- `hosts` for hostname resolution
- `services` for service names and ports

Examples:
```bash
getent passwd alice
getent group wheel
getent passwd
getent group
```

Why `getent` is valuable:
- It confirms whether the system can resolve an account through its configured identity stack.
- It helps distinguish between “the account does not exist” and “the account exists but the application is not reading the right source.”
- It works consistently across local and centralized identity systems.

Example `passwd` entry:
```bash
alice:x:1001:1001:Alice Example:/home/alice:/bin/bash
```

Field meanings:
- username
- password placeholder
- UID
- GID
- GECOS/comment field
- home directory
- login shell

Example `group` entry:
```bash
wheel:x:10:alice,bob
```

Field meanings:
- group name
- password placeholder
- GID
- comma-separated member list

**Local files versus centralized identity**

On a standalone system, `getent passwd alice` may read from `/etc/passwd`. In a domain-joined or centrally managed environment, the same command may return data from SSSD or another directory service.

This distinction matters because:
- `id` and `groups` show the identity as the system currently resolves it.
- `getent` shows whether the NSS layer can retrieve the account.
- A user may exist in a directory service but still fail to log in if the system is offline, misconfigured, or not caching the identity correctly.

A practical troubleshooting pattern:
1. Check whether the account resolves with `getent passwd username`.
2. Check group membership with `id username`.
3. Compare the result with file ownership and permissions.
4. Verify the relevant service if the lookup fails.

**Relationship to file permissions**

Linux file access depends on user, group, and other permissions. These commands help you map identity to access.

Example:
```bash
ls -l /srv/share/report.txt
-rw-rw----. 1 root projects 2048 Apr 26 10:00 /srv/share/report.txt
```

Interpretation:
- Owner is `root`
- Group is `projects`
- Members of `projects` can read and write
- Other users have no access

Check whether a user can access it:
```bash
id alice
```

If `projects` appears in the group list, `alice` can access the file according to the mode bits.

**Practical troubleshooting workflow**

Use a repeatable sequence when a user reports access problems.

1. Confirm the account exists.
   ```bash
   getent passwd alice
   ```
2. Confirm group membership.
   ```bash
   id alice
   groups alice
   ```
3. Inspect the target file or directory.
   ```bash
   ls -ld /srv/share
   ls -l /srv/share/report.txt
   ```
4. Compare permissions with identity.
5. If needed, check whether the user needs to log out and back in for new group membership to take effect.

> Group membership changes do not always apply to an existing login session immediately. A user may need a new shell session, a re-login, or a service restart depending on the context.

**Useful command patterns**

Check the current user:
```bash
id
groups
getent passwd "$(whoami)"
```

Check another user:
```bash
id bob
groups bob
getent passwd bob
getent group projects
```

Check numeric IDs only:
```bash
id -u bob
id -g bob
id -G bob
```

Check names instead of numbers:
```bash
id -un bob
id -gn bob
id -nG bob
```

**Scripting and automation use cases**

These commands are common in shell scripts because they are simple and reliable.

Example: verify that a user belongs to a required group.
```bash
#!/bin/bash
user="$1"
required_group="wheel"

if id -nG "$user" | grep -qw "$required_group"; then
  echo "$user has access"
else
  echo "$user does not have access"
fi
```

Example: confirm that an account exists before proceeding.
```bash
#!/bin/bash
user="$1"

if getent passwd "$user" >/dev/null; then
  echo "Account exists"
else
  echo "Account not found"
fi
```

Example: extract the primary group name.
```bash
id -gn alice
```

Example: extract all group names as a space-separated list.
```bash
id -nG alice
```

**Common exam-relevant pitfalls**

- `groups` shows membership, but it does not show numeric IDs.
- `id` includes the primary group in the group list, which can be confusing if you expect only supplementary groups.
- `getent` may return data from sources beyond local files, so it is the best first check in mixed environments.
- A user added to a group may need to start a new session before the change is visible.
- If a command works for `root` but not for a normal user, compare `id` output with the file’s ownership and mode bits before changing permissions.

**Quick comparison**

| Command | Best use | Typical output |
|---|---|---|
| `id` | Full identity check | UID, GID, and groups |
| `groups` | Fast group membership check | Group names only |
| `getent` | NSS database lookup | Account or group records |

**What to remember for RHCSA**

- Use `id` to verify the current or another user’s identity.
- Use `groups` for a quick group membership check.
- Use `getent` to confirm what the system can resolve through NSS.
- Match identity information to file ownership and permissions when troubleshooting access.
- Remember that group changes may not appear in an existing session until the user logs in again.
