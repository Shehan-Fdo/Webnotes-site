## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**User and group fundamentals**

Linux access control starts with identities. A user account identifies a person or service, while groups let you assign shared access to files, directories, and administrative privileges. On RHEL, you manage these identities with tools such as `useradd`, `usermod`, `userdel`, `groupadd`, `groupmod`, and `groupdel`.

A practical RHCSA mindset is to think in terms of:
- **Identity**
  - Who owns files and processes.
  - Which UID (user ID) and GID (group ID) the system uses.
- **Authorization**
  - What the user can read, write, execute, or administer.
- **Session state**
  - What the current shell knows about group membership and privilege changes.

**Core account databases**

RHEL stores account information across several files. You rarely edit these directly, but you must know what each file does.

| File | Purpose | Notes |
|---|---|---|
| `/etc/passwd` | User account metadata | Username, UID, GID, home directory, shell |
| `/etc/shadow` | Password hashes and aging data | Root-readable only |
| `/etc/group` | Group definitions | Group name, GID, members |
| `/etc/gshadow` | Group password/admin data | Less commonly edited directly |
| `/etc/login.defs` | Login policy defaults | UID/GID ranges, password aging defaults |
| `/etc/default/useradd` | `useradd` defaults | Default shell, home base, skeleton directory |
| `/etc/skel` | Skeleton home content | Files copied into new home directories |

A typical `/etc/passwd` entry looks like this:

```text
alice:x:1001:1001:Alice Example:/home/alice:/bin/bash
```

Interpretation:
- `alice` is the username.
- `x` means the password hash is stored in `/etc/shadow`.
- `1001` is the UID.
- `1001` is the primary GID.
- `/home/alice` is the home directory.
- `/bin/bash` is the login shell.

A typical `/etc/group` entry looks like this:

```text
appteam:x:2001:alice,bob
```

Interpretation:
- `appteam` is the group name.
- `2001` is the GID.
- `alice,bob` are supplementary members.

**UIDs, GIDs, and account ranges**

You should understand the difference between regular users and system accounts. The exact ranges can vary by distribution policy, but RHEL commonly uses separate ranges for system and regular accounts.

- **Regular user accounts**
  - Typically receive UIDs in a higher range, often starting around `1000`.
  - Intended for human users and long-lived application identities.
- **System accounts**
  - Typically use lower UIDs and GIDs.
  - Intended for daemons and services such as `sshd`, `chronyd`, or web services.

To inspect the configured ranges, check `/etc/login.defs`:

```bash
grep -E '^(UID_MIN|UID_MAX|GID_MIN|GID_MAX|SYS_UID_MIN|SYS_UID_MAX|SYS_GID_MIN|SYS_GID_MAX)' /etc/login.defs
```

To see actual account IDs on the system:

```bash
getent passwd
getent group
id alice
```

Practical interpretation:
- If a user has a UID in the regular-user range, it is usually a human login account.
- If a service account has a low UID, it is usually a system account.
- Do not assume the exact numeric cutoff without checking the system policy.

**Primary groups and supplementary groups**

Every user has one **primary group** and zero or more **supplementary groups**.

- **Primary group**
  - Stored as the GID field in `/etc/passwd`.
  - Used as the default group owner for newly created files.
- **Supplementary groups**
  - Listed in `/etc/group`.
  - Provide additional access without changing the primary group.

Example:

```bash
id alice
```

Possible output:

```text
uid=1001(alice) gid=1001(alice) groups=1001(alice),2001(appteam),10(wheel)
```

This means:
- `alice` belongs to her primary group `alice`.
- She is also a member of `appteam`.
- She can use `wheel`-based administrative access if sudo policy allows it.

> On RHEL, whether a newly created user gets a private primary group by default depends on system policy, especially `USERGROUPS_ENAB` in `/etc/login.defs` and the distro’s default user creation behavior. Do not assume it is universal without checking the host.

**How `useradd` builds a new account**

`useradd` creates the account entry and can also create the home directory if you ask it to. It reads defaults from multiple places:
- `/etc/default/useradd` for defaults such as shell and home base.
- `/etc/skel` for files copied into the new home directory.
- `/etc/login.defs` for policy such as UID/GID ranges and password aging defaults.

Check the current defaults:

```bash
useradd -D
cat /etc/default/useradd
ls -la /etc/skel
```

A common account creation example:

```bash
sudo useradd -m -d /srv/alice -s /bin/bash -c "Alice Example" alice
sudo passwd alice
```

What this does:
- `-m` creates the home directory if it does not already exist.
- `-d /srv/alice` sets the home path.
- `-s /bin/bash` sets the login shell.
- `-c` sets the comment field, often used for the full name.

Important behavior:
- `-d` sets the home directory path, but it does not create the directory by itself.
- If the target directory already exists, `useradd` may use it depending on the state and options.
- If you want a predictable new home directory, pair `-d` with `-m`.

A more service-oriented account example:

```bash
sudo useradd -r -s /sbin/nologin -d /var/lib/myapp myapp
```

This creates a system account:
- `-r` requests a system account.
- `-s /sbin/nologin` prevents interactive login.
- `-d /var/lib/myapp` records a home path for the service.

> Use the `nologin` path that exists on the system. On many RHEL systems `/sbin/nologin` and `/usr/sbin/nologin` are equivalent or linked, but exam-safe practice is to verify the actual path with `command -v nologin` or `ls -l /sbin/nologin /usr/sbin/nologin`.

**Skeleton files and home directory initialization**

When `useradd -m` creates a home directory, it copies files from `/etc/skel` into that directory. This is how new users get initial shell configuration such as `.bashrc`, `.bash_profile`, or other site-specific defaults.

Example:

```bash
ls -la /etc/skel
```

If you place a file in `/etc/skel`, every future user created with home-directory initialization receives a copy.

Example workflow:

```bash
echo 'export HISTTIMEFORMAT="%F %T "' | sudo tee -a /etc/skel/.bashrc
sudo useradd -m bob
sudo ls -la /home/bob
```

Use this carefully:
- Changes affect only future accounts, not existing ones.
- It is useful for standardizing shell behavior across new users.

**Managing existing users with `usermod`**

`usermod` changes account properties after creation. Use it when you need to adjust group membership, shell, home directory, or account name.

Common tasks:
- Add a user to supplementary groups.
- Change the primary group.
- Move the home directory.
- Lock or unlock login access.
- Rename an account.

Examples:

```bash
sudo usermod -aG wheel alice
sudo usermod -g appteam alice
sudo usermod -d /srv/alice -m alice
sudo usermod -s /sbin/nologin myapp
```

Key points:
- `-aG` appends supplementary groups; without `-a`, you may replace the user’s current supplementary group list.
- `-g` changes the primary group.
- `-d` changes the home directory path.
- `-m` moves the existing home contents to the new location.
- `-s` changes the login shell.

A practical caution:
- Group membership changes do not always affect the current shell immediately.
- The user may need to log out and back in.
- In some cases, `newgrp` can refresh the active group in the current shell.

**Refreshing group membership in a session**

When you add a user to a new group, the kernel and shell do not always update the current session’s group list automatically. This matters during RHCSA tasks because a user may still be denied access until the session refreshes.

Ways to refresh:
1. Log out and log back in.
2. Start a new shell session with `su - alice`.
3. Use `newgrp` to switch the active group in the current shell.

Example:

```bash
sudo usermod -aG appteam alice
su - alice
id
```

If the new group still does not appear in the current shell, use:

```bash
newgrp appteam
id
```

What `newgrp` does:
- Starts a new shell with the specified group as the effective primary group for that session.
- Is useful when you need newly granted group access immediately.
- Does not permanently change the account’s group membership; it only affects the current shell session.

**Group management with `groupadd`, `groupmod`, and `groupdel`**

Groups are first-class objects. You create them when you need shared access or a controlled administrative boundary.

Examples:

```bash
sudo groupadd appteam
sudo groupmod -n webteam appteam
sudo groupdel oldteam
```

Notes:
- `groupadd` creates a new group.
- `groupmod` renames or changes a group’s GID.
- `groupdel` removes a group if it is not in use as a primary group for an account.

If you need to inspect group membership:

```bash
getent group appteam
```

**Direct group membership management with `groupmems`**

`groupmems` is useful when you want to manage membership in a group without editing the group file manually. It is especially relevant for troubleshooting or when you want to add or remove users from a group in a controlled way.

Examples:

```bash
sudo groupmems -g appteam -a alice
sudo groupmems -g appteam -d alice
sudo groupmems -g appteam -l
```

Use cases:
- Add a user to a group when you want a focused group-management tool.
- Remove a user from a group without changing unrelated group data.
- List current members for verification.

Be aware:
- Administrative permissions and group administration policy can affect whether `groupmems` works as expected.
- On many systems, direct `usermod -aG` is more common in exam workflows.

**Password and login control**

A user account can exist without being able to log in interactively. This is common for service accounts and locked accounts.

Useful commands:

```bash
sudo passwd alice
sudo passwd -l alice
sudo passwd -u alice
sudo usermod -L alice
sudo usermod -U alice
```

Interpretation:
- `passwd -l` locks the password by disabling password-based login.
- `passwd -u` unlocks it.
- `usermod -L` and `-U` also lock and unlock the account password state.

For a service account, combine a disabled login shell with a locked password if you want to prevent interactive access:

```bash
sudo useradd -r -s /sbin/nologin -M backupsvc
sudo passwd -l backupsvc
```

This is useful when:
- The account runs a daemon or scheduled task.
- You want the identity to exist for file ownership or service execution.
- You do not want shell login access.

**User deletion and cleanup**

`userdel` removes an account, but it may refuse or require extra cleanup if the user is active or has leftover resources.

Examples:

```bash
sudo userdel alice
sudo userdel -r alice
```

What to know:
- `userdel` removes the account entry.
- `-r` also removes the home directory and mail spool.
- If the user is logged in or has running processes, deletion may fail or be blocked.
- Cron jobs, mail spools, and other user-owned files can remain and may need manual cleanup.

Troubleshooting example:

```bash
ps -u alice
sudo pkill -u alice
sudo userdel -r alice
```

If deletion still fails:
- Check for lingering processes.
- Check for mounted filesystems or open files.
- Inspect scheduled jobs and mail spool locations.
- Verify whether the account is still referenced by services or file ownership.

**Sudo and administrative access**

`sudo` lets approved users run commands as root or another user. On RHCSA systems, membership in `wheel` is a common pattern, but actual access depends on sudo policy.

Verify sudo rights for a specific user:

```bash
sudo -l -U alice
```

If you are already logged in as `alice`, you can also run:

```bash
sudo -l
```

Common administrative check:
- `sudo -l -U alice` shows what `alice` may run according to policy.
- `sudo visudo` edits sudoers safely, but it does not by itself confirm that a specific user has access.

Example sudoers fragment:

```text
%wheel ALL=(ALL) ALL
```

This means:
- Any user in the `wheel` group can run commands as any user, including root, if they authenticate successfully.

A more limited example:

```text
alice ALL=(root) /usr/bin/systemctl restart httpd
```

This allows `alice` to run only the specified command as root.

**Primary vs supplementary group behavior in practice**

File ownership and access often depend on the primary group at file creation time. This is why group planning matters.

Example scenario:
- `alice` belongs to `appteam`.
- Her primary group is `alice`.
- She creates a file in `/srv/shared`.

The file may be owned like this:

```text
-rw-r--r-- 1 alice alice 0 Apr 26 10:00 report.txt
```

If you want shared collaboration, you may instead:
- Set the directory group to `appteam`.
- Enable group inheritance with the setgid bit.
- Ensure users are members of the shared group.

Example:

```bash
sudo mkdir -p /srv/shared
sudo chgrp appteam /srv/shared
sudo chmod 2775 /srv/shared
```

The `2` in `2775` sets the setgid bit on the directory, so new files inherit the directory’s group.

**Practical permission workflow**

A common RHCSA task is to create a shared directory for a team.

1. Create the group.
2. Add users to the group.
3. Create the directory.
4. Set ownership and permissions.
5. Verify access from a user session.

Example:

```bash
sudo groupadd projectx
sudo usermod -aG projectx alice
sudo usermod -aG projectx bob
sudo mkdir -p /srv/projectx
sudo chown root:projectx /srv/projectx
sudo chmod 2770 /srv/projectx
```

Verification:

```bash
id alice
ls -ld /srv/projectx
su - alice
touch /srv/projectx/testfile
```

If access fails:
- Confirm the user has refreshed group membership.
- Confirm the directory group is correct.
- Confirm the permissions allow group write access.
- Confirm no ACLs or SELinux rules are blocking access.

**Troubleshooting account and group issues**

When something does not work, inspect the identity state first.

Useful commands:

```bash
id
groups
getent passwd alice
getent group appteam
ls -ld /home/alice
```

Common problems:
- The user was added to a group, but the session was not refreshed.
- The home directory exists but has the wrong owner or permissions.
- The login shell is set to `nologin`.
- The account is locked.
- The user is not in `wheel`, so sudo fails.
- The group exists, but the directory permissions do not allow access.

A realistic troubleshooting case:
- You add `alice` to `appteam`.
- She still cannot write to `/srv/projectx`.
- You check `id alice` and see the new group in the account database.
- She is still in an old shell session.
- You log out and back in, or use `newgrp appteam`, and the access works.

**Quick comparison**

| Command | Main purpose | Typical exam use |
|---|---|---|
| `useradd` | Create users | New local accounts and service accounts |
| `usermod` | Modify users | Add groups, change shell, move home |
| `userdel` | Remove users | Clean up unused accounts |
| `groupadd` | Create groups | Shared access and role separation |
| `groupmod` | Modify groups | Rename or adjust GID |
| `groupdel` | Remove groups | Cleanup unused groups |
| `groupmems` | Manage group members | Direct membership troubleshooting |
| `newgrp` | Switch active group in session | Immediate access after group changes |

**Exam-focused reminders**

- Check the current defaults before creating accounts with `useradd -D`.
- Use `/etc/skel` to understand what new home directories inherit.
- Do not assume a user’s group membership is active in the current shell.
- Use `sudo -l -U username` to inspect sudo rights for a specific user.
- Use `passwd -l` or `usermod -L` to disable password login when needed.
- Use `userdel -r` carefully because it can leave or require cleanup for active processes, mail, or cron-related data.
- Verify numeric IDs with `id`, `getent passwd`, and `getent group` when troubleshooting.

**Mini reference for common account states**

- **Interactive user**
  - Normal shell such as `/bin/bash`
  - Home directory created
  - Password set
  - May belong to `wheel` or other groups
- **Locked user**
  - Password disabled
  - May still own files and run processes
  - Useful for temporary suspension
- **Service account**
  - Often uses `nologin`
  - May have no home directory or a service-specific home
  - Used by daemons, timers, or application services

**Useful command examples**

```bash
# Create a regular user with a home directory and shell
sudo useradd -m -s /bin/bash alice
sudo passwd alice

# Add the user to supplementary groups
sudo usermod -aG wheel,appteam alice

# Verify identity and group membership
id alice
getent group appteam

# Create a system account for a service
sudo useradd -r -s /sbin/nologin -M myservice

# Lock and unlock an account
sudo passwd -l alice
sudo passwd -u alice

# Remove an account and its home directory
sudo userdel -r alice
```
