**Environment basics for scheduled jobs**

Scheduled jobs often fail for reasons that are not obvious from the command itself. The most common cause is a mismatch between the environment you have in an interactive shell and the much smaller environment available when the job runs later.

For RHCSA-level administration, you should understand how `cron` and `at` differ in their handling of:

- **Environment variables** such as `PATH`, `HOME`, `SHELL`, and custom variables
- **Working directory** at execution time
- **Shell interpretation** of commands, quoting, and redirection
- **User context** and permissions
- **Absolute paths** versus relying on shell lookup

A job that works when you type it manually may fail in a scheduler because the scheduler does not start from your interactive shell profile. The safest approach is to write jobs so they are self-contained and explicit.

**Cron environment behavior**

`cron` runs jobs with a minimal, predictable environment. It does not load your interactive shell startup files like `~/.bashrc` or `~/.bash_profile` unless you explicitly arrange for that behavior in the command or script.

Typical cron environment characteristics:

- `PATH` is often limited, so commands like `python`, `rsync`, or `myscript.sh` may not be found unless you use full paths or set `PATH` in the crontab.
- `HOME` is usually set to the user’s home directory.
- `SHELL` is commonly `/bin/sh` unless overridden in the crontab.
- The current working directory is not guaranteed to be your home directory.

A practical cron entry should use absolute paths and explicit redirection:

```cron
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

# Run a backup script every day at 02:15
15 2 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```

Key points:

- **Absolute paths** reduce ambiguity and avoid `PATH` issues.
- **Explicit `PATH`** in the crontab helps when you must call multiple utilities.
- **Redirection** should be deliberate so you can capture both standard output and standard error.
- **Shell choice** matters if your script uses Bash-specific syntax.

> If you rely on shell features such as arrays, `[[ ... ]]`, or brace expansion, make sure the job runs under a shell that supports them, or place the logic in a script with a proper shebang such as `#!/bin/bash`.

**Cron and startup files**

Cron does not behave like an interactive login session. That means environment setup from shell profiles is usually absent.

Common implications:

- Variables exported in your terminal session are not automatically available to cron.
- Aliases are not available.
- Functions defined in your shell are not available.
- `cd` behavior is not automatic; scripts should set their own working directory if needed.

If a job depends on a specific directory, make that explicit in the script:

```bash
#!/bin/bash
set -e

cd /var/lib/app || exit 1
/usr/bin/tar -czf /backup/app-$(date +%F).tar.gz .
```

This pattern is safer than assuming cron will start in `/var/lib/app`.

**At jobs and execution context**

`at` schedules a one-time job. Like cron, it does not guarantee a full interactive environment later. The command you submit is stored and executed by a shell at the scheduled time, but you should not assume that your current shell state will be preserved in a reliable or complete way.

Use `at` when you need a one-off task, such as:

- A delayed reboot
- A one-time cleanup
- A temporary maintenance action
- A deferred script execution

Example submission:

```bash
echo '/usr/local/bin/generate-report.sh' | at 02:00
```

If the script needs data from your current session, pass it explicitly through the command line, a file, or exported variables written into a wrapper script. Do not assume that unexported shell variables will be available later.

A safer pattern is to create a small wrapper script:

```bash
#!/bin/bash
export REPORT_DIR=/srv/reports
/usr/local/bin/generate-report.sh "$REPORT_DIR"
```

Then schedule the wrapper:

```bash
echo '/usr/local/bin/run-report-wrapper.sh' | at 02:00
```

This keeps the scheduled command deterministic and easier to troubleshoot.

**Working directory and relative paths**

Relative paths are a frequent source of failures in scheduled jobs. The scheduler may start the job from a directory you do not expect, so commands like `./backup.sh` or `logs/output.txt` can break.

Prefer one of these approaches:

- Use absolute paths everywhere
- Change directory at the start of the script
- Write output to a known absolute location

Example:

```bash
#!/bin/bash
cd /opt/app || exit 1
./bin/process-data --input /data/input.csv > /var/log/process-data.log 2>&1
```

This works because the script explicitly sets the directory before using a relative executable path.

**Shell, quoting, and redirection**

Scheduled jobs are often executed through a shell, so shell parsing rules still apply. That means quoting and redirection must be correct at submission time.

Important behaviors:

- Single quotes prevent variable expansion at submission time.
- Double quotes allow expansion by the shell that processes the command.
- Redirection applies to the command or command group it is attached to.
- Semicolons separate commands, but they do not group output redirection.

A common mistake is to assume a variable will expand later inside a quoted `at` command. If you submit:

```bash
echo '/usr/local/bin/generate-report.sh "$REPORT_DIR"' | at 02:00
```

the literal text `"$REPORT_DIR"` is stored in the job, so the variable is not expanded by the shell that created the job. If you need the current value, either expand it before submission or place the logic in a script that sets or exports the variable itself.

For troubleshooting output, group commands when you want one redirection target:

```bash
* * * * * {
  /usr/bin/date
  /usr/bin/env
  /usr/bin/pwd
} >> /tmp/cron-debug.log 2>&1
```

This is clearer than attaching redirection to only the first command.

**Practical debugging approach**

When a scheduled job fails, compare the scheduler environment to your interactive shell. A good debugging method is to log the environment and working directory from inside the job.

Example cron debug entry:

```cron
* * * * * {
  /usr/bin/date
  /usr/bin/env
  /usr/bin/pwd
} >> /tmp/cron-debug.log 2>&1
```

This helps you verify:

- Which variables are present
- Which shell-related values are set
- What the current directory is
- Whether the command is running at all

You can also test a script manually with a reduced environment to mimic scheduler conditions:

```bash
env -i PATH=/usr/bin:/bin HOME=/home/student /bin/bash /usr/local/bin/test-job.sh
```

This is useful because it removes most inherited variables and exposes hidden dependencies.

**Cron-specific configuration options**

Crontabs can define environment variables at the top of the file. This is useful when multiple jobs share the same settings.

Example:

```cron
SHELL=/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin
MAILTO=admin@example.com

0 3 * * * /usr/local/bin/cleanup.sh
30 3 * * * /usr/local/bin/report.sh
```

Useful variables include:

- **SHELL** to choose the command interpreter
- **PATH** to control command lookup
- **MAILTO** to receive job output by email if mail is configured

Be careful not to overuse environment variables in crontabs. If a job depends on many settings, a dedicated script is usually easier to maintain and test.

**User crontab versus system crontab**

User crontabs and system crontabs differ in execution context.

User crontab characteristics:

- Runs as the owning user
- Uses that user’s permissions
- Usually stored and edited with `crontab -e`

System crontab characteristics:

- Can specify the user field explicitly
- Often includes a different format
- May be used for system maintenance tasks

Example system crontab line:

```cron
0 1 * * * root /usr/local/sbin/system-maintenance.sh
```

Because system jobs may run with different privileges, environment assumptions become even more important. A root job may have a different `PATH` and different access to files than a regular user job.

**At job inspection and verification**

You can list pending `at` jobs with:

```bash
atq
```

You can inspect a job with:

```bash
at -c <jobid>
```

This shows the stored job content as it will be executed. Use it to confirm the command text, but do not treat it as proof of the original interactive shell environment. It shows what was queued, not every detail of the shell session that submitted it.

If you need to remove a queued job:

```bash
atrm <jobid>
```

**Best practices for exam and real-world use**

Use these habits to avoid environment-related failures:

- **Use absolute paths** for commands, scripts, and output files.
- **Set `PATH` explicitly** in crontabs when needed.
- **Write self-contained scripts** instead of long inline commands.
- **Set the working directory** inside the script if relative paths are required.
- **Export required variables** inside the script or wrapper.
- **Test with a minimal environment** before relying on a scheduled job.
- **Capture output and errors** to a log file for troubleshooting.
- **Avoid interactive-only features** such as aliases and shell functions.

A robust scheduled script often looks like this:

```bash
#!/bin/bash
set -euo pipefail

PATH=/usr/local/bin:/usr/bin:/bin
export PATH

cd /opt/app || exit 1

/usr/bin/python3 /opt/app/scripts/report.py \
  --output /var/tmp/report.txt >> /var/log/report.log 2>&1
```

This script is resilient because it defines its own environment, uses explicit paths, and handles failure early.

**Common failure patterns**

Watch for these recurring issues:

- A command works in your terminal but not in cron because the binary is not in cron’s `PATH`.
- A script uses `./script.sh` but cron starts in a different directory.
- A job depends on `~/.bashrc`, but cron does not source it.
- A variable is referenced in a scheduled command but was never exported or written into the script.
- Output redirection is attached to only one command in a multi-command line.
- A shell feature is used under `/bin/sh` even though the script assumes Bash.

**Exam focus**

For EX200, expect to recognize why a scheduled job fails and how to fix it quickly. The safest answer is usually one of these:

- Add absolute paths
- Set `PATH` in the crontab
- Move logic into a script with a proper shebang
- Avoid relying on interactive shell state
- Verify the stored job or log output

If a scheduled task behaves differently from a manual test, assume the environment is the first thing to check.
