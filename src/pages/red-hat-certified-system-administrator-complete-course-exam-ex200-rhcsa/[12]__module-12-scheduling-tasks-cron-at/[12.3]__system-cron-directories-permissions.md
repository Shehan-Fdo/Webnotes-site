## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Cron scheduling on RHEL**

`cron` is a time-based job scheduler that runs commands at fixed times, intervals, or dates. On Red Hat Enterprise Linux (RHEL), you use it for recurring maintenance tasks such as log cleanup, backups, report generation, and package housekeeping.

For RHCSA exam purposes, you need to know both the **user crontab** model and the **system cron** model. The key difference is who owns the schedule and where the job definition lives.

- **User crontab**: per-user schedules managed with `crontab -e`
- **System cron**: centrally managed schedules in `/etc/crontab`, `/etc/cron.d/`, and periodic directories
- **Permissions**: determine who can create crontabs and which files cron will execute

> `cron` does not provide a full interactive shell environment. Jobs should use absolute paths and should not depend on aliases, shell startup files, or a rich `PATH` unless you define it explicitly.

**System cron file locations and purpose**

RHEL uses several system-managed locations for recurring jobs. Each path has a distinct purpose, and exam questions often test whether you know where to place a job and what syntax it requires.

- `/etc/crontab`
  - Main system crontab file
  - Supports an extra **user** field in each job line
  - Commonly used for system-wide schedules
- `/etc/cron.d/`
  - Drop-in directory for additional system cron files
  - Each file uses the same syntax as `/etc/crontab`, including the user field
  - Useful for package-managed or application-specific schedules
- `/etc/cron.hourly/`
  - Scripts run once per hour by the system’s periodic job mechanism
  - Intended for executable scripts, not raw crontab syntax
- `/etc/cron.daily/`
  - Scripts run once per day
- `/etc/cron.weekly/`
  - Scripts run once per week
- `/etc/cron.monthly/`
  - Scripts run once per month

These periodic directories are typically invoked by a system scheduler chain. On many RHEL systems, `anacron` or a systemd timer-based mechanism ensures missed jobs run later if the machine was off at the scheduled time. The exact implementation can vary by version and package set, so focus on the behavior: periodic scripts are executed automatically according to their interval.

**Syntax differences: user crontab vs system cron**

A common exam trap is using the wrong syntax in the wrong file.

- In a **user crontab**, you do **not** include a user field
- In `/etc/crontab` and `/etc/cron.d/*`, you **must** include a user field
- In `/etc/cron.hourly/`, `/etc/cron.daily/`, `/etc/cron.weekly/`, and `/etc/cron.monthly/`, you place executable scripts, not cron schedule lines

Example of a user crontab entry:

```cron
# minute hour day-of-month month day-of-week command
15 2 * * * /home/alice/bin/backup.sh
```

Example of a system cron entry in `/etc/crontab` or `/etc/cron.d/backup`:

```cron
# minute hour day-of-month month day-of-week user command
15 2 * * * root /usr/local/sbin/backup.sh
```

> If you place `root` in a user crontab, the line is invalid. If you omit the user field in `/etc/crontab` or `/etc/cron.d/`, the line is also invalid.

**Permissions and ownership expectations**

Cron is strict about file ownership and permissions because it treats these files as trusted automation inputs. Incorrect modes can cause jobs to be ignored or rejected.

- `/etc/crontab`
  - Typically owned by `root:root`
  - Should be writable only by root
  - Common mode: `0644`
- Files in `/etc/cron.d/`
  - Typically owned by `root:root`
  - Should be writable only by root
  - Common mode: `0644`
- Scripts in `/etc/cron.hourly/`, `/etc/cron.daily/`, `/etc/cron.weekly/`, `/etc/cron.monthly/`
  - Usually owned by `root:root`
  - Must be executable
  - Common mode: `0755`
- User crontabs
  - Stored under `/var/spool/cron/` on RHEL
  - Managed by `crontab`
  - Not edited directly in normal administration

Practical examples:

```bash
sudo chown root:root /etc/cron.d/cleanup
sudo chmod 0644 /etc/cron.d/cleanup

sudo chown root:root /etc/cron.daily/logrotate
sudo chmod 0755 /etc/cron.daily/logrotate
```

For scripts in periodic directories, the execute bit matters. A script with mode `0644` will usually be skipped because cron can read it but cannot run it.

**Access control with `cron.allow` and `cron.deny`**

RHEL uses access-control files to decide who may create user crontabs with `crontab -e`.

- `/etc/cron.allow`
  - If this file exists, only users listed in it may use `crontab`
  - It takes precedence over `cron.deny`
- `/etc/cron.deny`
  - Used only if `/etc/cron.allow` does not exist
  - Users listed in it are denied access
- If neither file exists
  - Only `root` may use `crontab`

This precedence is important for the exam.

Example:

```bash
cat /etc/cron.allow
alice
bob
```

In that case, only `alice` and `bob` can run `crontab -e`. Everyone else is blocked, even if they are not listed in `/etc/cron.deny`.

**Managing user crontabs**

You manage per-user schedules with the `crontab` command.

Common commands:

```bash
crontab -e      # edit your crontab
crontab -l      # list your crontab
crontab -r      # remove your crontab
crontab -u bob -l   # root can inspect another user's crontab
```

A user crontab entry should use absolute paths and should be careful with shell syntax:

```cron
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin

0 3 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```

The `PATH` line helps cron find commands, but it does not make the job behave like an interactive login shell. If your script depends on environment variables, shell functions, or profile files, define them inside the script or source them explicitly.

**Periodic directory behavior**

The periodic directories are a simple way to run scripts at fixed intervals without writing explicit cron timing lines.

- `/etc/cron.hourly/`
  - Good for frequent housekeeping tasks
  - Example use: cache cleanup, short log rotation checks
- `/etc/cron.daily/`
  - Good for daily maintenance
  - Example use: database vacuum jobs, report generation, temporary file cleanup
- `/etc/cron.weekly/`
  - Good for weekly tasks
  - Example use: deeper cleanup, weekly audits, archive rotation
- `/etc/cron.monthly/`
  - Good for monthly tasks
  - Example use: monthly summaries, retention enforcement

A script placed in one of these directories should be self-contained and executable:

```bash
#!/bin/bash
/usr/bin/find /tmp -type f -mtime +7 -delete
```

Save it as `/etc/cron.daily/cleanup-tmp`, then set:

```bash
sudo chown root:root /etc/cron.daily/cleanup-tmp
sudo chmod 0755 /etc/cron.daily/cleanup-tmp
```

**RHEL scheduling chain and `anacron`**

On RHEL, periodic jobs may be handled by `cron`, `anacron`, or systemd timer integration depending on the release and installed packages. For exam study, the important concept is that periodic jobs are designed to run even if the system was not powered on at the exact scheduled time.

`anacron` is especially relevant for systems that are not always on. It helps ensure daily, weekly, and monthly jobs run after a missed interval. You do not need to memorize every implementation detail, but you should understand why periodic directories are useful on intermittently powered systems.

Typical configuration files you may encounter:

- `/etc/anacrontab`
- `/etc/cron.daily/`
- `/etc/cron.weekly/`
- `/etc/cron.monthly/`

Example `anacrontab`-style entry:

```cron
1   5   cron.daily   run-parts /etc/cron.daily
```

This means the daily jobs run with a delay after boot if they were missed.

**Real-world examples**

Example 1: system-wide cleanup job in `/etc/cron.d/cleanup`

```cron
# minute hour day-of-month month day-of-week user command
10 2 * * * root /usr/local/sbin/cleanup-temp.sh
```

This is valid in `/etc/cron.d/` because it includes the user field. It would be invalid in a user crontab.

Example 2: executable daily script

```bash
#!/bin/bash
/usr/bin/find /var/tmp -type f -mtime +3 -delete
```

```bash
sudo install -o root -g root -m 0755 cleanup-var-tmp.sh /etc/cron.daily/cleanup-var-tmp
```

Example 3: user crontab with output logging

```cron
MAILTO=admin@example.com
0 1 * * 1 /home/alice/bin/weekly-report.sh >/home/alice/weekly-report.log 2>&1
```

**Troubleshooting cron jobs**

When a cron job fails, check the schedule syntax, permissions, environment, and logs in that order.

1. Verify the file type and syntax
   - User crontab: no user field
   - `/etc/crontab` and `/etc/cron.d/`: user field required
   - Periodic directories: executable scripts only
2. Verify ownership and mode
   - System cron files should usually be `root:root` and `0644`
   - Periodic scripts should usually be `root:root` and `0755`
3. Verify the command path
   - Use absolute paths such as `/usr/bin/rsync`
   - Do not rely on `which` to validate a cron command
4. Verify script executability and interpreter line
   - Ensure the script starts with a valid shebang such as `#!/bin/bash`
5. Check logs

Common log locations and commands:

```bash
sudo journalctl -u crond
sudo grep CRON /var/log/cron
sudo tail -f /var/log/cron
```

Depending on the system configuration, you may also see messages in the system journal:

```bash
sudo journalctl -xe
```

Common failure clues:

- `bad minute`
  - The time field syntax is invalid
- `user unknown`
  - The user field in `/etc/crontab` or `/etc/cron.d/` is wrong
- `permission denied`
  - The script is not executable or the target file is not accessible
- `No such file or directory`
  - The command path is wrong or the script references a missing interpreter
- `CRON[pid]: (user) CMD (...)`
  - Indicates the job was accepted and started
- `CRON[pid]: (user) ERROR (...)`
  - Indicates runtime failure or execution problem

**Exam-focused checklist**

- Know the difference between user crontabs and system cron files
- Remember that `/etc/crontab` and `/etc/cron.d/` require a user field
- Remember that `/etc/cron.hourly/`, `/etc/cron.daily/`, `/etc/cron.weekly/`, and `/etc/cron.monthly/` contain executable scripts
- Know the default access rule:
  - `cron.allow` exists: only listed users may use `crontab`
  - otherwise `cron.deny` applies
  - if neither exists: only `root` can use `crontab`
- Use `root:root` ownership and safe permissions
- Use absolute paths and explicit environment settings in jobs
- Check `/var/log/cron` or `journalctl -u crond` when troubleshooting

**Quick reference**

| Location | Purpose | Syntax | Typical mode |
|---|---|---|---|
| `crontab -e` | Per-user schedule | No user field | Managed by `crontab` |
| `/etc/crontab` | System-wide schedule | Includes user field | `0644` |
| `/etc/cron.d/` | Drop-in system schedules | Includes user field | `0644` |
| `/etc/cron.hourly/` | Hourly scripts | Executable script | `0755` |
| `/etc/cron.daily/` | Daily scripts | Executable script | `0755` |
| `/etc/cron.weekly/` | Weekly scripts | Executable script | `0755` |
| `/etc/cron.monthly/` | Monthly scripts | Executable script | `0755` |

**Practical administration pattern**

A reliable cron deployment usually follows this pattern:

1. Write a script that is self-contained and uses absolute paths.
2. Test the script manually before scheduling it.
3. Place it in the correct cron location.
4. Set ownership and permissions correctly.
5. Confirm the job appears in the expected crontab or directory.
6. Check logs after the scheduled time.

This approach reduces failures caused by environment differences, missing permissions, and syntax mistakes.
