**Cron user job management**

`crontab` manages per-user cron tables, which are schedules of commands executed by the `crond` daemon at specific times. On Red Hat Enterprise Linux, you use `crontab -e` to edit your own cron table, `crontab -l` to display it, and `crontab -r` to remove it.

Cron is useful for recurring maintenance tasks such as log cleanup, backups, report generation, and periodic health checks. It is best for jobs that need to run on a fixed schedule and do not require interactive input.

- **Per-user scope** means each user can have a separate cron table.
- **`crond`** is the daemon that reads scheduled jobs and runs them in the background.
- **Non-interactive execution** means commands must run without prompts or terminal input.

A cron entry uses five time fields followed by the command to run:

```cron
# minute hour day-of-month month day-of-week command
30 2 * * 1 /usr/local/bin/backup.sh
```

**Using `crontab -e`**

`crontab -e` opens your current user's cron table in the default editor. If no crontab exists yet, the command creates one. This is the standard way to add or modify scheduled jobs for your account.

Typical workflow:

1. Run `crontab -e`.
2. Add one or more cron lines.
3. Save and exit the editor.
4. Verify the result with `crontab -l`.

Example cron entries:

```cron
# Run a script every day at 03:15
15 3 * * * /home/student/bin/daily-report.sh

# Run every 10 minutes
*/10 * * * * /usr/local/bin/check-service.sh

# Run at 01:00 on Sundays
0 1 * * 0 /usr/local/sbin/weekly-maintenance.sh
```

Important practical points:

- Use full paths for commands and scripts.
- Set executable permissions on scripts when needed.
- Redirect output if you want logs or want to avoid mail delivery.
- Keep commands simple and test them manually before scheduling.

Example with logging:

```cron
0 4 * * * /usr/local/bin/cleanup.sh >> /var/log/cleanup.log 2>&1
```

> Cron jobs run with a minimal environment. Do not assume your interactive shell variables, aliases, or PATH settings are available.

**Using `crontab -l`**

`crontab -l` lists the current user's cron table to standard output. Use it to confirm what is scheduled, review changes, or troubleshoot unexpected job behavior.

Common uses:

- Verify that a new entry was saved correctly.
- Check whether a job still exists after editing.
- Compare the active schedule with what you intended to configure.

Example:

```bash
crontab -l
```

If no crontab exists, the command typically reports that there is no crontab for the user. That is normal and means the user has not created any scheduled jobs yet.

You can also combine listing with filtering when reviewing large tables:

```bash
crontab -l | grep backup
```

**Using `crontab -r`**

`crontab -r` removes the current user's entire crontab. This is a destructive action because it deletes all scheduled jobs for that user at once.

Use it only when you intend to clear the schedule completely.

```bash
crontab -r
```

Because it removes everything, verify before and after:

```bash
crontab -l
crontab -r
crontab -l
```

> `crontab -r` does not ask for confirmation on many systems. Treat it as irreversible unless you have a backup copy of the crontab.

A safer pattern is to back up the current table first:

```bash
crontab -l > ~/crontab.backup
crontab -r
```

**Cron syntax essentials**

A cron line has this structure:

```cron
* * * * * command
```

Field meanings:

- **Minute**: `0-59`
- **Hour**: `0-23`
- **Day of month**: `1-31`
- **Month**: `1-12`
- **Day of week**: `0-7` where both `0` and `7` usually mean Sunday

Useful patterns:

- `*` means every valid value.
- `*/n` means every `n` units.
- Commas separate multiple values.
- Hyphens define ranges.

Examples:

```cron
# Every hour at minute 0
0 * * * * /usr/local/bin/hourly-task.sh

# At 08:00 and 20:00 every day
0 8,20 * * * /usr/local/bin/twice-daily.sh

# Every weekday at 18:30
30 18 * * 1-5 /usr/local/bin/weekday-job.sh
```

**Environment and reliability**

Cron jobs often fail because the command works interactively but not in cron. The most common causes are missing paths, permissions, or environment assumptions.

Best practices:

- Use absolute paths for binaries and files.
- Set `PATH` explicitly if needed.
- Redirect both standard output and standard error.
- Make scripts executable with `chmod +x`.
- Test the command as the target user before scheduling it.

Example of a safer cron script header:

```bash
#!/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

/usr/bin/date >> /tmp/cron-test.log
```

**Troubleshooting cron jobs**

If a job does not run, check these areas:

- The `crond` service is active.
- The cron syntax is valid.
- The command path is correct.
- The user has permission to run the command.
- Output is being captured for review.

Useful commands:

```bash
systemctl status crond
journalctl -u crond
crontab -l
```

If you need to inspect logs on systems that write cron activity to system logs, look for entries related to `CRON` or `crond` in the journal or log files.

**Common exam-relevant distinctions**

`crontab` commands manage a user’s personal cron table, not system-wide schedules. That is different from files in `/etc/cron.d/`, `/etc/crontab`, or scripts in `/etc/cron.hourly/`, which are system-managed scheduling mechanisms.

- **`crontab -e`** edits the current user’s table.
- **`crontab -l`** displays the current user’s table.
- **`crontab -r`** deletes the current user’s table.
- **System cron files** are managed separately and usually require administrative access.

**Practical examples**

Create a backup job for the current user:

```bash
crontab -e
```

Add:

```cron
0 2 * * * /home/student/bin/backup-home.sh >> /home/student/backup.log 2>&1
```

Verify:

```bash
crontab -l
```

Remove all scheduled jobs for the current user:

```bash
crontab -r
```

Confirm removal:

```bash
crontab -l
```

**Quick reference**

| Command | Purpose | Risk level |
|---|---|---|
| `crontab -e` | Edit the current user’s cron table | Low |
| `crontab -l` | List the current user’s cron table | None |
| `crontab -r` | Remove the current user’s cron table | High |

**Key exam habits**

- Use `crontab -e` for user-level recurring tasks.
- Use `crontab -l` immediately after editing to verify the saved schedule.
- Treat `crontab -r` as destructive and back up first if needed.
- Write cron commands so they work without an interactive shell.
