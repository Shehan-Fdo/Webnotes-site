## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**What `at` does**

`at` schedules a command or script to run once at a specific time in the future. Use it when you need a delayed action that does not repeat, such as a one-time cleanup, a reminder, or a maintenance command after hours.

Unlike recurring schedulers such as `cron`, `at` is event-based and then exits. That makes it useful for ad hoc administration tasks where you know the exact execution time but do not want to maintain a recurring schedule.

- **One-time execution** is the core use case.
- **Interactive input** lets you build the job command line after launching `at`.
- **Deferred execution** is handled by the `atd` daemon, which must be running for jobs to execute.

**Basic syntax and job creation**

You start an `at` session with a time specification, then provide the commands to run. End input with <kbd>Ctrl</kbd> + <kbd>D</kbd>.

```bash
at 23:30
at now + 10 minutes
at 5pm tomorrow
at 2026-04-27 08:00
```

Inside the `at` prompt, you can type one or more shell commands:

```bash
at now + 1 hour
at> /usr/local/bin/backup-home.sh
at> logger "Backup job started"
at> <Ctrl-D>
```

A practical pattern is to place multiple commands in a shell script and schedule that script, especially when the task is more than a few simple commands.

```bash
cat <<'EOF' > /usr/local/bin/cleanup-temp.sh
#!/bin/bash
rm -rf /tmp/app-cache/*
logger "Temporary cache cleaned"
EOF

chmod +x /usr/local/bin/cleanup-temp.sh
at now + 30 minutes
at> /usr/local/bin/cleanup-temp.sh
at> <Ctrl-D>
```

**Time specifications**

`at` accepts many human-readable time formats. The exact parsing can vary slightly by implementation, but common forms are consistent across Linux systems.

- Absolute times:
  - `HH:MM`
  - `midnight`
  - `noon`
  - `teatime` in some implementations
- Relative times:
  - `now + 5 minutes`
  - `now + 2 hours`
  - `now + 1 day`
- Date-qualified times:
  - `10:00 tomorrow`
  - `08:00 04/27/2026`

Use absolute times when you need precision and relative times when you want a quick delay from the current moment.

> If the time expression is ambiguous, `at` may reject it or interpret it unexpectedly. Verify the scheduled time with `atq` after submission.

**Managing the at daemon**

`at` depends on the `atd` service to run queued jobs. On systemd-based RHEL systems, you typically manage it with `systemctl`.

```bash
systemctl status atd
systemctl enable --now atd
```

If `atd` is not active, jobs can still be queued, but they will not execute until the daemon is running.

**Viewing queued jobs**

Use `atq` to list pending jobs. It shows the job ID, scheduled time, queue, and owner.

```bash
atq
```

Example output:

```text
5   Tue Apr 27 08:00:00 2026 a root
6   Tue Apr 27 09:30:00 2026 a alice
```

- The **job ID** is used for removal and inspection.
- The **queue** is a letter that groups jobs for internal scheduling behavior.
- The **owner** is the user who submitted the job.

Queue letters are not a simple universal priority ranking. Their behavior is implementation-specific, so treat them as scheduling categories rather than a general “lower letter means higher priority” rule.

**Inspecting a job**

Use `at -c` to display the job’s generated shell wrapper and environment setup as stored for execution. This is useful when you want to confirm what command `at` recorded.

```bash
at -c 5
```

This output is best used for inspection and troubleshooting. It shows the script wrapper and the environment-related setup that `at` will use, but it is not a perfect snapshot of the live runtime environment.

**Removing a job**

Use `atrm` to delete a queued job by ID.

```bash
atrm 5
```

You can also remove multiple jobs in one command:

```bash
atrm 5 6 7
```

After removal, confirm with `atq` that the job no longer appears.

**Permissions and access control**

Access to `at` is controlled by `/etc/at.allow` and `/etc/at.deny`.

- If `/etc/at.allow` exists, only users listed in that file may use `at`.
- If `/etc/at.allow` does not exist and `/etc/at.deny` exists, users not listed in `at.deny` may use `at`.
- If neither file exists, access is typically allowed to all users on many Linux systems, including common RHEL behavior.

This is an important exam detail: do not assume a root-only default when both files are absent.

Example `/etc/at.allow`:

```text
root
alice
```

Example `/etc/at.deny`:

```text
bob
carol
```

> File ownership and permissions matter. These control files are normally managed by root, and incorrect edits can unintentionally block or allow scheduling.

**Output handling and mail**

By default, `at` captures standard output and standard error from the job and attempts to mail the results to the submitting user. The exact delivery depends on local mail configuration and system setup.

- If a job produces output, that output is typically mailed.
- If a job produces no visible output, mail behavior may still depend on the system’s mail configuration and job messages.
- Redirect output explicitly when you want predictable logging.

Examples:

```bash
at now + 15 minutes
at> /usr/local/bin/report.sh > /var/log/report.log 2>&1
at> <Ctrl-D>
```

Or discard output intentionally:

```bash
at now + 15 minutes
at> /usr/local/bin/quiet-task.sh >/dev/null 2>&1
at> <Ctrl-D>
```

For troubleshooting, redirect to a file instead of relying only on mail.

**Environment and shell behavior**

`at` runs commands in a non-interactive shell. That means you should not rely on interactive shell features, aliases, or terminal state.

- Use absolute paths for commands.
- Set required environment variables inside the job or script.
- Avoid commands that require a TTY unless you explicitly provide one.

Example:

```bash
at now + 20 minutes
at> PATH=/usr/local/bin:/usr/bin:/bin
at> export PATH
at> /usr/local/bin/maintenance.sh
at> <Ctrl-D>
```

A script is usually more reliable than typing a long command sequence directly into `at`, especially when the task depends on environment variables or multiple steps.

**Practical examples**

A delayed package cache cleanup:

```bash
at now + 1 hour
at> dnf clean all
at> <Ctrl-D>
```

A one-time service restart after a maintenance window:

```bash
at 02:00 tomorrow
at> systemctl restart httpd
at> <Ctrl-D>
```

A reminder message to the console or log:

```bash
at now + 10 minutes
at> logger "Check backup status on server01"
at> <Ctrl-D>
```

A safer file-based task:

```bash
cat <<'EOF' > /usr/local/bin/check-disk.sh
#!/bin/bash
df -h > /var/tmp/disk-check.txt
EOF
chmod +x /usr/local/bin/check-disk.sh

at 18:00
at> /usr/local/bin/check-disk.sh
at> <Ctrl-D>
```

**Troubleshooting**

If a job does not run, check the following in order:

1. Confirm `atd` is active.
2. Verify the job is still queued with `atq`.
3. Check access control files in `/etc/at.allow` and `/etc/at.deny`.
4. Inspect the job with `at -c <jobid>`.
5. Review mail or redirected output for errors.
6. Confirm the command path and permissions are correct.

Common causes of failure include:

- The command is not in the job’s `PATH`.
- The script is not executable.
- The user lacks permission to access the target file or service.
- The system mail subsystem is not configured, so output is not delivered as expected.
- The scheduled time was entered incorrectly.

**`at` versus `cron`**

Use `at` when you need a single execution. Use `cron` when you need repetition.

| Feature | `at` | `cron` |
|---|---|---|
| Execution pattern | One-time | Repeating |
| Best for | Delayed ad hoc tasks | Scheduled recurring tasks |
| Job storage | Queue until run | Cron table entries |
| Typical use | Maintenance after reboot window, one-off reminders | Daily backups, log rotation, periodic checks |

**Exam-focused reminders**

- Know how to create a job with `at`.
- Know how to list jobs with `atq`.
- Know how to remove jobs with `atrm`.
- Know how to inspect a job with `at -c`.
- Know that `atd` must be running.
- Know the access-control file behavior for `/etc/at.allow` and `/etc/at.deny`.
- Use absolute paths and explicit redirection for reliable jobs.

**Quick reference**

```bash
# Schedule a job
at now + 30 minutes

# List queued jobs
atq

# Inspect a job
at -c 5

# Remove a job
atrm 5

# Start the daemon
systemctl enable --now atd
```
