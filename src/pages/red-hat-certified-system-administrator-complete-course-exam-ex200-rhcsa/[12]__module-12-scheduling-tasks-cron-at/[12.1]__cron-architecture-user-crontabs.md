**Cron architecture on RHEL**

`cron` is the classic time-based job scheduler on Red Hat Enterprise Linux (RHEL). You use it for recurring tasks such as log cleanup, backups, report generation, and maintenance scripts. It is best for jobs that should run at a specific minute, hour, day, or weekday, and it runs commands without requiring an interactive session.

On modern RHEL systems, the scheduling stack usually includes:

- **crond**: the daemon that reads schedules and launches jobs
- **crontab**: the command used to install and manage per-user crontab files
- **anacron**: a companion scheduler for jobs that should run even if the system was off at the scheduled time

A useful mental model is:

- `crond` checks schedules every minute
- `crontab` installs user-specific schedules
- `anacron` fills gaps for missed periodic jobs on systems that are not always on

**User crontab files and where they live**

Per-user crontabs are not edited directly in normal administration. They are managed by the `crontab` command and stored in the spool area.

- Per-user crontab location on RHEL: `/var/spool/cron/USERNAME`
- These files are owned and maintained by `crontab`
- You should not edit them with a text editor unless you know the implications, because `crontab` handles validation and installation

Typical workflow:

```bash
crontab -e
crontab -l
crontab -r
```

- `crontab -e` opens the current user’s crontab in an editor
- `crontab -l` lists the current user’s installed jobs
- `crontab -r` removes the current user’s crontab

> Do not manually manage `/var/spool/cron/USERNAME` unless you are intentionally troubleshooting. The supported interface is `crontab`.

**Crontab file format**

A user crontab line has five time fields followed by the command to run:

```text
* * * * * command
- - - - -
| | | | |
| | | | +----- day of week (0-7, where 0 and 7 are Sunday)
| | | +------- month (1-12)
| | +--------- day of month (1-31)
| +----------- hour (0-23)
+------------- minute (0-59)
```

Example:

```cron
15 2 * * 1 /usr/local/bin/backup.sh
```

This runs at 02:15 every Monday.

**Important cron syntax behavior**

Cron uses **OR logic** between the day-of-month and day-of-week fields in standard Vixie/cron behavior, which is a common exam trap.

- If both fields are restricted, the job runs when either field matches
- Example: `0 4 1 * 1` can run on the 1st of the month and also on Mondays, not only when both conditions are true

Example:

```cron
0 4 1 * 1 /usr/local/bin/report.sh
```

If you need strict “both conditions must match” behavior, you usually handle that inside the script itself.

**Special strings for common schedules**

Cron supports shorthand strings that replace the five time fields:

- `@reboot` — run once at startup
- `@hourly` — run once per hour
- `@daily` — run once per day
- `@weekly` — run once per week
- `@monthly` — run once per month
- `@yearly` or `@annually` — run once per year

Examples:

```cron
@reboot /usr/local/bin/startup-check.sh
@daily /usr/local/bin/daily-maintenance.sh
@weekly /usr/local/bin/rotate-reports.sh
```

These are useful when the schedule is simple and readability matters.

**Environment and execution context**

Cron jobs run with a minimal environment. That means your script should not assume the same PATH, shell settings, or interactive variables that you have in a login shell.

Common best practices:

- Use absolute paths for commands and scripts
- Set `PATH` explicitly if needed
- Redirect output to a log file or use mail if configured
- Make scripts executable and test them manually first

Example:

```cron
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin
MAILTO=admin@example.com
0 3 * * * /usr/local/bin/cleanup.sh >> /var/log/cleanup.log 2>&1
```

**Managing another user’s crontab as root**

As `root`, you can create or inspect a crontab for another user. This is a common administrative and RHCSA-style task.

Create or edit a crontab for user `alice`:

```bash
sudo crontab -u alice -e
```

List that user’s crontab:

```bash
sudo crontab -u alice -l
```

Verify the spool file exists:

```bash
sudo ls -l /var/spool/cron/alice
sudo cat /var/spool/cron/alice
```

A practical example:

```bash
sudo crontab -u alice -e
```

Add:

```cron
0 1 * * * /home/alice/bin/backup-home.sh
```

Then verify:

```bash
sudo crontab -u alice -l
sudo ls -l /var/spool/cron/alice
```

This confirms both the installed schedule and the underlying managed file.

**Cron access control files**

RHEL uses access control files to determine who may use `crontab`.

- `/etc/cron.allow`
- `/etc/cron.deny`

Behavior is typically:

- If `/etc/cron.allow` exists, only users listed there may use `crontab`
- If `/etc/cron.allow` does not exist and `/etc/cron.deny` exists, users listed in `cron.deny` are blocked
- If neither file exists, behavior depends on the distribution defaults, but on RHEL you should know the allow/deny model and verify local policy

Example:

```bash
sudo cat /etc/cron.allow
sudo cat /etc/cron.deny
```

Use these files carefully because they control who can schedule jobs at all.

**System cron directories versus user crontabs**

RHEL supports both per-user crontabs and system-wide cron locations. They serve different purposes.

- User crontabs: managed by `crontab`, stored in `/var/spool/cron/USERNAME`
- System cron jobs: often placed in `/etc/crontab` or `/etc/cron.d/`
- Periodic scripts: commonly found in `/etc/cron.hourly/`, `/etc/cron.daily/`, `/etc/cron.weekly/`, and `/etc/cron.monthly/`

A key distinction is that system cron files usually include a user field, while user crontabs do not.

Example system crontab line:

```cron
15 2 * * * root /usr/local/sbin/system-task.sh
```

In `/etc/crontab`, the `root` field tells cron which account runs the command.

**Cron and anacron on RHEL**

`cron` and `anacron` solve related but different problems.

- `cron` runs jobs at exact times while the system is up
- `anacron` runs periodic jobs that may have been missed while the machine was off
- `anacron` is especially useful for laptops, desktops, and intermittently powered systems
- On servers that stay on continuously, `cron` alone may be sufficient for many tasks

Typical RHEL periodic job flow:

- `crond` handles minute-level scheduling and user crontabs
- `anacron` ensures daily, weekly, and monthly maintenance jobs still run after downtime
- The periodic scripts in `/etc/cron.daily/`, `/etc/cron.weekly/`, and similar directories are often triggered through `anacron` or related system mechanisms, depending on the RHEL version and configuration

Common configuration files:

- `/etc/anacrontab` — controls anacron timing and commands
- `/etc/crontab` — system-wide cron schedule
- `/etc/cron.d/` — drop-in cron definitions

Example `/etc/anacrontab` style entries:

```cron
SHELL=/bin/sh
PATH=/sbin:/bin:/usr/sbin:/usr/bin
1   5   cron.daily   run-parts /etc/cron.daily
7   25  cron.weekly  run-parts /etc/cron.weekly
@monthly 45 cron.monthly run-parts /etc/cron.monthly
```

The exact format can vary by distribution and package version, so verify the local file before making assumptions.

**Logging and troubleshooting**

When cron jobs do not run, check both the scheduler and the job output.

Useful checks:

```bash
systemctl status crond
journalctl -u crond
```

On RHEL systems using `rsyslog`, cron activity is commonly written to `/var/log/cron`:

```bash
sudo tail -f /var/log/cron
```

Also inspect the job’s own output destination:

- Redirect stdout and stderr to a file
- Use `MAILTO` if mail delivery is configured
- Confirm the script exits successfully

Example troubleshooting pattern:

```cron
* * * * * /usr/local/bin/test-job.sh >> /tmp/test-job.log 2>&1
```

Then verify:

```bash
cat /tmp/test-job.log
journalctl -u crond --since "10 minutes ago"
sudo tail -n 50 /var/log/cron
```

**Practical examples**

A simple daily cleanup job for the current user:

```cron
0 2 * * * /home/student/bin/cleanup.sh >> /home/student/cleanup.log 2>&1
```

A reboot-time job:

```cron
@reboot /usr/local/bin/start-services-check.sh
```

A weekly root-maintained job for another user:

```bash
sudo crontab -u bob -e
```

Add:

```cron
30 1 * * 0 /home/bob/bin/weekly-sync.sh >> /home/bob/weekly-sync.log 2>&1
```

Verify:

```bash
sudo crontab -u bob -l
sudo ls -l /var/spool/cron/bob
```

A system-wide job in `/etc/cron.d/backup`:

```cron
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin

15 3 * * * root /usr/local/sbin/backup-all.sh
```

This is useful when you want centralized administration rather than per-user ownership.

**Common exam and admin pitfalls**

- **Permissions**: the script must be executable and readable by the account running it
- **Paths**: cron does not inherit your interactive shell PATH reliably
- **Time fields**: remember the minute field comes first
- **Day matching**: day-of-month and day-of-week use OR logic in standard cron behavior
- **Access control**: `cron.allow` and `cron.deny` can block `crontab` usage
- **Missed jobs**: use `anacron` for periodic tasks that should run after downtime
- **Logging**: check `/var/log/cron` and `journalctl` when troubleshooting

**Quick verification checklist**

1. Confirm the daemon is running:
   ```bash
   systemctl status crond
   ```
2. Confirm the crontab is installed:
   ```bash
   crontab -l
   ```
3. Confirm the underlying spool file exists:
   ```bash
   ls -l /var/spool/cron/USERNAME
   ```
4. Confirm the job command works manually:
   ```bash
   /full/path/to/script.sh
   ```
5. Confirm logs show execution:
   ```bash
   journalctl -u crond
   tail -f /var/log/cron
   ```

**Key takeaways**

Cron is simple, reliable, and heavily used in RHEL administration. For RHCSA-level work, you should know how to create and verify user crontabs, understand the spool file location, recognize access control files, and troubleshoot with logs. You should also distinguish cron from anacron so you can choose the right mechanism for always-on versus intermittently powered systems.
