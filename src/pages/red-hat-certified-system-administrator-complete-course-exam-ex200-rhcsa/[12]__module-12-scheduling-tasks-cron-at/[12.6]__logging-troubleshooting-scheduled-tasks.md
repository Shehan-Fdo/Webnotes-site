## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

Scheduled tasks on RHEL usually fail for a small set of reasons: the daemon is not running, the job is written in the wrong file format, permissions block execution, the command works manually but not in the scheduled environment, or security controls such as SELinux prevent access. Good troubleshooting starts by confirming the scheduler, then checking logs, then validating the job definition and the target command.

For exam prep, you should know how to troubleshoot both **cron** and **at** jobs, including system-wide cron files, per-user crontabs, periodic directories, and access-control files for `at`.

**How cron logging works on RHEL**

Cron activity is commonly visible in the system journal, and on many RHEL systems it may also be written to `/var/log/cron` through `rsyslog` depending on logging configuration. Do not assume only one location exists. When a job does not run, check both the journal and traditional log files if they are present.

Useful commands:

```bash
systemctl status crond
journalctl -u crond
grep CRON /var/log/cron
grep crond /var/log/messages
```

- **`journalctl -u crond`** shows service startup, failures, and job execution messages when the daemon logs to the journal.
- **`/var/log/cron`** is a traditional location you may still encounter on RHEL systems with `rsyslog` configured for cron messages.
- **`/var/log/messages`** may also contain related entries on some systems, depending on logging rules.

If `crond` is not active, scheduled jobs will not run.

```bash
systemctl enable --now crond
systemctl is-enabled crond
systemctl is-active crond
```

**Cron job syntax and file format**

Cron syntax errors are a common cause of silent failure. The format depends on where the job is defined.

- **User crontab** entries do not include a username field.
- **`/etc/crontab`** and files in **`/etc/cron.d/`** require a username field.
- Comments begin with `#`.
- Environment variables such as `SHELL`, `PATH`, and `MAILTO` can affect execution.

Examples:

```cron
# User crontab: minute hour day-of-month month day-of-week command
15 2 * * * /usr/local/bin/backup.sh

# /etc/crontab or /etc/cron.d/example: includes user field
15 2 * * * root /usr/local/bin/backup.sh
```

Common syntax problems:

- Missing the user field in `/etc/crontab` or `/etc/cron.d/*`.
- Extra spaces or tabs in the wrong place.
- Invalid time fields such as `61` for minutes.
- Using `%` in a command without escaping it.
- Forgetting that cron uses a minimal environment, not your interactive shell profile.

> A job can look correct in a terminal and still fail in cron because the shell environment is different.

**Verify the command outside cron first**

A reliable troubleshooting method is to run the command manually as the same user and compare the result with the scheduled execution. This helps separate command failure from scheduling failure.

```bash
sudo -u alice /usr/local/bin/report.sh
env -i PATH=/usr/bin:/bin /usr/local/bin/report.sh
```

- **Manual success** but cron failure often points to environment, permissions, or path issues.
- **Manual failure** means the command itself needs fixing before you troubleshoot scheduling.
- Use absolute paths for commands, scripts, and output files.

If the script depends on environment variables, define them explicitly in the crontab or inside the script.

```cron
PATH=/usr/sbin:/usr/bin:/sbin:/bin
MAILTO=admin@example.com
0 3 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```

**Troubleshooting user crontabs**

User crontabs are managed with `crontab -e`, `crontab -l`, and `crontab -r`. If a user’s job is missing or not running, verify the correct user owns the crontab and that the file was installed successfully.

```bash
crontab -u alice -l
crontab -u alice -e
ls -l /var/spool/cron/
```

Things to check:

- The crontab exists for the intended user.
- The user account is not locked or expired.
- The script is executable if it is meant to be run directly.
- File permissions allow the user to read the script and write any output files.
- The job does not rely on shell features that are unavailable in `/bin/sh`.

**Troubleshooting system crontabs and `/etc/cron.d/`**

System crontabs are useful when you need a job to run as a specific account without placing it in a user’s personal crontab. They are also a common source of formatting mistakes.

```bash
cat /etc/crontab
ls -l /etc/cron.d/
```

Check these details:

- The file includes the required user field.
- The file name in `/etc/cron.d/` contains no invalid characters.
- The file permissions are readable by `root` and typically not writable by regular users.
- The command path is absolute.
- The target user has permission to run the command and access the files it needs.

Example:

```cron
# /etc/cron.d/reporting
SHELL=/bin/bash
PATH=/usr/sbin:/usr/bin:/sbin:/bin

0 1 * * * root /usr/local/bin/generate-report.sh
```

**Periodic cron directories**

RHEL systems may also use periodic directories such as `/etc/cron.hourly/`, `/etc/cron.daily/`, `/etc/cron.weekly/`, and `/etc/cron.monthly/`. These are typically executed by `crond` and, on some systems, by `anacron` for missed runs after downtime.

Troubleshoot them by checking:

- The script is present in the correct directory.
- The script is executable.
- The script has a valid shebang such as `#!/bin/bash`.
- The directory and script permissions are correct.
- `anacron` is installed and enabled where periodic catch-up execution is expected.

```bash
ls -l /etc/cron.daily/
chmod +x /etc/cron.daily/cleanup
head -n 1 /etc/cron.daily/cleanup
systemctl status anacron
```

If a daily job never runs, confirm whether the system relies on `anacron` for missed periodic tasks and whether the machine was powered off during the scheduled window.

**`at` job troubleshooting**

The `at` command schedules one-time jobs. Its access control is often the first thing to check when a user reports permission errors.

Access-control behavior on RHEL:

- If `/etc/at.allow` exists, only users listed in that file may use `at`.
- If `/etc/at.allow` does not exist and `/etc/at.deny` exists, users not listed in `at.deny` may use `at`.
- If neither file exists, only `root` may use `at`.
- If both files exist, `at.allow` takes precedence and `at.deny` is ignored.

Check the files directly:

```bash
ls -l /etc/at.allow /etc/at.deny
cat /etc/at.allow
cat /etc/at.deny
```

Schedule and inspect jobs:

```bash
atq
at -c 1
atrm 1
```

A practical `at` example should use a simple command that behaves predictably in a noninteractive shell:

```bash
echo '/usr/bin/touch /tmp/at-test' | at now + 10 minutes
```

Troubleshooting tips:

- Use absolute paths in the command.
- Avoid relying on interactive shell aliases or functions.
- Confirm the user has permission to read the command and write the target file.
- Check the job output and mail delivery if the command produces output.

**SELinux and file context issues**

SELinux can block scheduled tasks even when the command and permissions look correct. This is common when a job writes to a nonstandard path or executes a script from a location with the wrong context.

Check for denials:

```bash
ausearch -m AVC,USER_AVC -ts recent
journalctl | grep -i avc
```

Inspect contexts:

```bash
ls -Z /usr/local/bin/backup.sh
ls -Z /var/www/html/
```

If a script or directory has the wrong label, restore the default context:

```bash
restorecon -Rv /usr/local/bin/backup.sh
restorecon -Rv /var/www/html/
```

If you need to diagnose a denial, compare the scheduled job’s target path with the expected SELinux type. For example, a cron job writing to a web content directory may fail if the directory is not labeled for that service.

> Do not disable SELinux as a first response. Identify the denial and fix the label or policy issue.

**Service configuration and daemon behavior**

Sometimes `crond` is running, but jobs still do not execute because the daemon configuration or environment is wrong. On RHEL, review service-related settings and any local configuration that affects cron behavior.

Useful checks:

```bash
systemctl cat crond
grep -v '^#' /etc/sysconfig/crond 2>/dev/null
ps -ef | grep [c]rond
```

Look for:

- Custom daemon options in the systemd unit or drop-in files.
- Environment settings that affect logging or mail delivery.
- Unexpected restart loops or failures in the service status output.
- Time synchronization problems that make jobs appear to run late or not at all.

If the system clock is wrong, scheduled jobs may seem broken even though cron is functioning.

```bash
timedatectl
```

**Troubleshooting workflow**

1. Confirm the scheduler is active with `systemctl status crond` or `systemctl status anacron` where relevant.
2. Check logs in `journalctl -u crond` and, if present, `/var/log/cron` or `/var/log/messages`.
3. Verify the job syntax in the correct file type: user crontab, `/etc/crontab`, or `/etc/cron.d/*`.
4. Run the command manually with a minimal environment and compare the result.
5. Check permissions, executable bits, and file ownership.
6. Review `at.allow` and `at.deny` if the issue involves `at`.
7. Inspect SELinux denials and restore file contexts if needed.
8. Confirm periodic directories and `anacron` behavior for hourly, daily, weekly, or monthly jobs.

**Common failure patterns**

- **Wrong file format**: a system crontab entry missing the user field.
- **Bad path**: the command works in your shell but not in cron because `PATH` is too short.
- **Non-executable script**: the file exists but lacks `chmod +x`.
- **Permission mismatch**: the scheduled user cannot read the script or write the output file.
- **SELinux denial**: the job is blocked by labeling or policy.
- **Access-control failure**: `at` is denied because of `at.allow` or `at.deny`.
- **Missed periodic run**: the machine was off and `anacron` is not configured to catch up.

**Practical examples**

Compare manual execution with scheduled execution:

```bash
sudo -u alice /usr/local/bin/cleanup.sh
sudo -u alice env -i PATH=/usr/bin:/bin /usr/local/bin/cleanup.sh
```

Check a cron entry and its logs:

```bash
crontab -l
journalctl -u crond --since "today"
grep CRON /var/log/cron
```

Inspect a system cron file:

```bash
cat /etc/cron.d/backup
chmod 644 /etc/cron.d/backup
```

Check `at` permissions and queue:

```bash
ls -l /etc/at.allow /etc/at.deny
atq
at -c 2
```

**Key exam reminders**

- Use absolute paths in scheduled jobs.
- Know the difference between user crontabs and system cron files.
- Remember that `/etc/crontab` and `/etc/cron.d/*` require a user field.
- Check both the journal and traditional cron logs when troubleshooting.
- Understand `at.allow` and `at.deny` precedence exactly.
- Verify executable bits, SELinux labels, and periodic directory behavior.
