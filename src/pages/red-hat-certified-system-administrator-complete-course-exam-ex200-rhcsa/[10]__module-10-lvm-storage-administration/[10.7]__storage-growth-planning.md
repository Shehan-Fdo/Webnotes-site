## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Storage growth planning**

Storage growth planning is the process of expanding capacity without breaking data access, bootability, or filesystem integrity. On RHCSA systems, you usually grow storage in layers:

- **Disk**: the virtual or physical device increases in size.
- **Partition**: the partition table is updated to use the new space.
- **LVM**: the physical volume, volume group, and logical volume are extended.
- **Filesystem**: the filesystem is grown to consume the larger block device.

The key exam skill is knowing which layer is limiting growth and which tools operate on that layer. You do not “grow storage” with one command in every case; you verify the stack first, then expand the correct component in the correct order.

**Core planning concepts**

Before changing anything, identify the current layout and the mount path that users actually consume. A filesystem may sit directly on a partition, on an LVM logical volume, or on another block abstraction such as RAID. Growth planning starts with mapping the path from disk to mount point.

Useful inspection commands:

```bash
lsblk
lsblk -f
blkid
df -hT
findmnt
pvs
vgs
lvs
```

- `lsblk` shows the block-device tree and helps you see parent-child relationships.
- `lsblk -f` adds filesystem type, UUID, and mount point.
- `df -hT` shows used and available space from the filesystem perspective.
- `findmnt` confirms the exact mount source and target.
- `pvs`, `vgs`, and `lvs` show LVM capacity at each layer.

A practical planning rule is to expand the smallest constrained layer first, then verify the next layer can use the new space. For example, if an XFS filesystem is on an LVM logical volume, you must extend the logical volume before the filesystem can grow.

**Storage stack patterns**

Common RHCSA storage layouts include:

| Layout | Growth path | Notes |
|---|---|---|
| Partition → ext4 | grow partition, then grow filesystem | ext4 can grow online; shrinking requires unmounting and `e2fsck` first |
| Partition → XFS | grow partition, then grow filesystem | XFS grows online but does not shrink |
| LVM LV → ext4 | extend LV, then grow filesystem | `lvextend -r` can automate filesystem growth |
| LVM LV → XFS | extend LV, then grow filesystem | `xfs_growfs` runs on the mounted filesystem |
| Whole disk → LVM PV → VG → LV | expand disk, then PV, then LV, then filesystem | only use a whole disk as a PV when it is dedicated and unused |

> A whole disk used as a physical volume must be empty and dedicated to LVM. If it contains a partition table or data, `pvcreate` can destroy existing contents.

**Planning with partitions**

If storage is partitioned, you may need to enlarge the partition before the filesystem can see more space. This is common when a VM disk grows and the last partition should consume the new free space.

Typical tools:

- `parted`
- `fdisk`
- `sgdisk`
- `partprobe`
- `udevadm settle`

Example workflow for a partition-backed filesystem:

```bash
lsblk /dev/vda
sudo parted /dev/vda print
sudo parted /dev/vda resizepart 2 100%
sudo partprobe /dev/vda
lsblk /dev/vda
```

Important planning details:

- `resizepart` changes the partition table end point; it does not resize the filesystem.
- The kernel may not immediately reread the new table if the partition is busy.
- Some partition changes require a reboot or a device rescan before the new size is visible.
- Not every partition can be safely resized live, especially if the kernel or partitioning tool cannot update the in-use table cleanly.

If the partition is the last partition on the disk and there is free space after it, growth is usually straightforward. If another partition follows it, you cannot simply extend into occupied space.

**LVM growth planning**

LVM is the most flexible growth model on RHCSA systems. It separates storage into layers:

- **PV**: physical volume, usually a partition or whole disk initialized for LVM.
- **VG**: volume group, a pool of extents from one or more PVs.
- **LV**: logical volume, the allocatable block device presented to a filesystem.

A common growth sequence is:

1. Increase the underlying disk or add a new disk.
2. Expand or create the PV.
3. Extend the VG capacity.
4. Extend the LV.
5. Grow the filesystem.

Example with a new disk added to an existing VG:

```bash
sudo pvcreate /dev/vdb
sudo vgextend rhel /dev/vdb
sudo lvextend -L +10G /dev/rhel/home
sudo xfs_growfs /home
```

If you want the LV to consume all free space in the VG:

```bash
sudo lvextend -l +100%FREE /dev/rhel/home
sudo xfs_growfs /home
```

You can also let LVM resize the filesystem for you when supported:

```bash
sudo lvextend -r -L +5G /dev/rhel/home
```

- `-r` asks LVM to resize the filesystem after extending the LV.
- This is convenient, but you should still verify the filesystem type and mount state.
- It is safest when you understand exactly what filesystem is on the LV.

**Filesystem-specific behavior**

Different filesystems have different growth and shrink rules. This matters for exam planning because the wrong assumption can lead to data loss or a failed task.

| Filesystem | Grow online? | Shrink online? | Shrink offline? | Notes |
|---|---:|---:|---:|---|
| XFS | Yes | No | No | Grow with `xfs_growfs` on the mounted filesystem |
| ext4 | Yes | No | Yes | Shrink requires unmounting and `e2fsck` before `resize2fs` |
| ext3 | Yes | No | Yes | Shrink requires unmounting and `e2fsck` before `resize2fs` |
| ext2 | Yes | No | Yes | Shrink requires unmounting and `e2fsck` before `resize2fs` |

Key points:

- **XFS** grows only while mounted and cannot shrink.
- **ext4/ext3/ext2** can grow while mounted in many cases, but shrinking requires the filesystem to be unmounted first.
- Before shrinking ext filesystems, run `e2fsck` to check and repair consistency.
- `resize2fs` is the tool for ext-family resizing; it does not replace filesystem checking.

Example shrink sequence for an ext4 filesystem:

```bash
sudo umount /dev/rhel/data
sudo e2fsck -f /dev/rhel/data
sudo resize2fs /dev/rhel/data 20G
sudo lvreduce -L 20G /dev/rhel/data
sudo mount /dev/rhel/data /data
```

> Shrinking is risky because the filesystem size must be reduced before the block device is reduced, and the filesystem must be clean. Always verify the target size and backup data first.

**XFS growth workflow**

XFS is common on RHEL systems and is designed for online growth. The filesystem grows through its mount point, not by directly targeting the raw block device.

Example:

```bash
sudo lvextend -L +5G /dev/rhel/var
sudo xfs_growfs /var
```

Or with a partition-backed XFS filesystem:

```bash
sudo parted /dev/vda resizepart 3 100%
sudo partprobe /dev/vda
sudo xfs_growfs /mnt/data
```

Important conditions:

- The underlying block device must already be larger before `xfs_growfs` can use the new space.
- The XFS filesystem must be mounted.
- You run `xfs_growfs` against the mount point, not the device path.

**ext filesystem growth workflow**

For ext4, growth is usually simpler than shrink because it can often happen while mounted.

Example:

```bash
sudo lvextend -L +2G /dev/rhel/logs
sudo resize2fs /dev/rhel/logs
```

If you use `lvextend -r`, the filesystem resize is handled automatically:

```bash
sudo lvextend -r -L +2G /dev/rhel/logs
```

Planning notes:

- ext filesystems can grow after the block device grows.
- Shrinking is a separate operation and requires a clean, unmounted filesystem.
- `e2fsck` is part of the shrink workflow, not an optional extra.

**Choosing the right growth method**

Use the simplest safe method that matches the layout:

- If the filesystem is on an **LVM LV**, prefer `lvextend` and then grow the filesystem.
- If the filesystem is on a **partition**, resize the partition first, then grow the filesystem.
- If the filesystem is **XFS**, plan for online growth only.
- If you may need to **shrink later**, ext4 is more flexible than XFS.

A quick decision guide:

1. Identify the mount point with `findmnt`.
2. Identify the block device with `lsblk -f`.
3. Determine whether the filesystem is on a partition or LVM.
4. Check the filesystem type.
5. Expand the correct lower layer first.
6. Grow the filesystem last.

**Practical examples**

Grow an LVM-backed XFS home directory:

```bash
sudo vgs
sudo lvs
sudo lvextend -L +10G /dev/rhel/home
sudo xfs_growfs /home
df -hT /home
```

Add a new disk to LVM and extend `/var`:

```bash
sudo pvcreate /dev/vdb
sudo vgextend rhel /dev/vdb
sudo lvextend -l +100%FREE /dev/rhel/var
sudo xfs_growfs /var
```

Grow a partition-backed ext4 filesystem:

```bash
sudo parted /dev/vda print
sudo parted /dev/vda resizepart 2 100%
sudo partprobe /dev/vda
sudo resize2fs /dev/vda2
df -hT /data
```

**Verification and troubleshooting**

After any growth operation, verify both the block layer and the filesystem layer.

```bash
lsblk
df -hT
mount | grep -E ' /data | /home | /var '
pvs
vgs
lvs
```

Common issues:

- The filesystem does not show new space because you resized the LV but forgot to grow the filesystem.
- The partition table changed, but the kernel has not reread it yet.
- `xfs_growfs` fails because you targeted the device instead of the mount point.
- `resize2fs` fails because the filesystem is not mounted for growth or not unmounted and checked for shrink.
- `pvcreate` fails or destroys data because the target disk was not empty or dedicated.

**Exam-focused reminders**

- **XFS**: grow only, online, via mount point.
- **ext4**: grow online; shrink offline with `e2fsck` first.
- **LVM**: extend the LV before the filesystem.
- **Partitions**: update the partition table before filesystem growth.
- **Verification**: always confirm with `lsblk` and `df -hT` after changes.

**Safe planning mindset**

Storage growth is less about memorizing one command and more about understanding dependencies. You should always ask:

- What is the current filesystem type?
- Is the filesystem on a partition or an LV?
- Does the lower layer already have free space?
- Can the filesystem grow online?
- Is a reboot or reread needed for the kernel to see the new size?

That sequence prevents the most common RHCSA mistakes and helps you choose the correct tool quickly under exam pressure.
