## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Filesystem resizing overview**

Resizing storage on RHEL usually means changing one or more layers in the storage stack:

- **Disk**: the virtual or physical block device grows first.
- **Partition**: the partition is expanded to use the new space.
- **Volume manager**: LVM logical volume is extended if present.
- **Filesystem**: the filesystem is grown or, in limited cases, shrunk.

A useful mental model is:

```text
Disk -> Partition -> LVM PV -> VG -> LV -> Filesystem -> Mount point
```

You must resize in the correct order. In practice, growth is common and safe when done carefully; shrinking is more constrained and depends on the filesystem type.

**Core rules for RHCSA labs**

- **XFS** can grow online, but it cannot shrink.
- **ext4** can grow online and can shrink, but shrinking usually requires the filesystem to be unmounted.
- **LVM** can grow and shrink logical volumes, but filesystem support and mount state determine whether the full operation is possible.
- **Partition tables** must be updated before the filesystem can use new space on a partition-backed device.
- **Root filesystems** require special handling because they are in use during normal operation.

> For exam tasks, always identify the filesystem type first with `lsblk -f`, `blkid`, or `df -T` before choosing a resize method.

**Inspecting the current layout**

Before changing anything, confirm what sits on top of what.

```bash
lsblk -f
df -Th
blkid
pvs
vgs
lvs
```

Useful checks:

- `lsblk -f` shows device hierarchy, filesystem type, and mount points.
- `df -Th` shows mounted filesystems and their types.
- `pvs`, `vgs`, and `lvs` show LVM capacity and allocation.
- `blkid` helps confirm UUIDs and filesystem signatures.

If you need a more detailed LVM view, target the specific object:

```bash
vgdisplay vgdata
lvdisplay /dev/vgdata/lvhome
pvdisplay /dev/sdb2
```

**Resizing a partition-backed filesystem**

When a filesystem lives directly on a partition, you usually expand the partition first, then grow the filesystem.

Typical workflow:

1. Increase the virtual disk size in the hypervisor or cloud platform.
2. Re-read the partition layout or expand the partition.
3. Grow the filesystem to fill the new space.
4. Verify the new size with `df -h`.

For many RHCSA-style labs, `growpart` is the fastest tool when available.

```bash
growpart /dev/sda 2
```

This expands partition 2 on `/dev/sda` to the end of the disk. After that, grow the filesystem.

For ext4:

```bash
resize2fs /dev/sda2
```

For XFS:

```bash
mount | grep /data
xfs_growfs /data
```

XFS growth uses the mount point, not the block device. The filesystem must be mounted.

**When to use `growpart` vs `parted`**

- Use **`growpart`** when you want a quick, script-friendly way to extend the last partition into free space.
- Use **`parted`** when you need manual control, such as deleting and recreating a partition with the same start sector or adjusting a non-final partition in a controlled lab scenario.
- Use **`partprobe`** or a reboot if the kernel does not immediately notice the new partition table.

Example with `parted`:

```bash
parted /dev/sda
(parted) print
(parted) resizepart 2 100%
(parted) quit
partprobe /dev/sda
```

If the kernel still uses the old table, you may need to reboot. On exam systems, a reboot is sometimes the simplest safe way to ensure the new partition map is active.

**Growing a filesystem on a direct block device**

If the filesystem is not on LVM, the order is still:

1. Expand the underlying disk or partition.
2. Expand the filesystem.

For ext4:

```bash
lsblk -f
growpart /dev/vda 1
resize2fs /dev/vda1
df -h /mnt/data
```

For XFS:

```bash
lsblk -f
growpart /dev/vda 1
mount /dev/vda1 /mnt/data
xfs_growfs /mnt/data
df -h /mnt/data
```

If the filesystem is already mounted, `xfs_growfs` can run online. If it is not mounted, mount it first.

**LVM growth workflow**

LVM is common in RHCSA labs because it makes resizing easier. The usual growth path is:

1. Increase the disk or partition that backs the physical volume.
2. Extend the physical volume if needed.
3. Extend the logical volume.
4. Grow the filesystem.

If the PV is on a partition, expand the partition first, then run `pvresize`.

```bash
growpart /dev/sdb 2
pvresize /dev/sdb2
lvextend -l +100%FREE /dev/vgdata/lvhome
```

Then grow the filesystem.

For XFS:

```bash
xfs_growfs /home
```

For ext4:

```bash
resize2fs /dev/vgdata/lvhome
```

A common shortcut is to extend the LV and filesystem together:

```bash
lvextend -r -l +100%FREE /dev/vgdata/lvhome
```

The `-r` option asks LVM to resize the filesystem after extending the LV. This is convenient, but you should still know the manual steps for the exam.

**LVM growth example**

Suppose `/home` is on `/dev/vgdata/lvhome` and the VG has free space.

```bash
lvs
df -Th /home
lvextend -L +2G /dev/vgdata/lvhome
xfs_growfs /home
df -h /home
```

If the filesystem is ext4 instead:

```bash
lvextend -L +2G /dev/vgdata/lvhome
resize2fs /dev/vgdata/lvhome
```

**Shrinking filesystems**

Shrinking is where many mistakes happen.

- **XFS cannot shrink.**
- **ext4 can shrink**, but the filesystem must be unmounted for the shrink operation.
- If the filesystem is on LVM, shrink the filesystem first, then reduce the LV.
- If the filesystem is on a partition, shrink the filesystem first, then reduce the partition.

> Never reduce the block device before reducing the filesystem. Doing so risks data loss.

**ext4 shrink workflow on LVM**

A safe ext4 shrink sequence is:

1. Unmount the filesystem.
2. Check the filesystem.
3. Shrink the filesystem.
4. Reduce the logical volume.
5. Mount and verify.

Example:

```bash
umount /home
e2fsck -f /dev/vgdata/lvhome
resize2fs /dev/vgdata/lvhome 8G
lvreduce -L 8G /dev/vgdata/lvhome
mount /home
df -h /home
```

Notes:

- `e2fsck -f` is important before shrinking ext4.
- `resize2fs` can shrink the filesystem to a target size.
- `lvreduce` should happen only after the filesystem is safely smaller than the LV.

If you want to reduce by a relative amount, calculate carefully. In exam settings, explicit target sizes are safer than relative shrink values.

**ext4 shrink workflow on a partition**

If ext4 sits directly on a partition, the sequence is similar:

1. Unmount the filesystem.
2. Check the filesystem.
3. Shrink the filesystem.
4. Shrink the partition.
5. Remount and verify.

Example:

```bash
umount /dev/sda3
e2fsck -f /dev/sda3
resize2fs /dev/sda3 10G
parted /dev/sda
(parted) resizepart 3 12GiB
(parted) quit
partprobe /dev/sda
mount /dev/sda3 /data
df -h /data
```

The partition must remain slightly larger than the filesystem until the filesystem shrink is complete.

**XFS growth workflow**

XFS is the default filesystem on many RHEL systems and is very common in RHCSA tasks.

Key properties:

- It grows online.
- It does not shrink.
- It uses the mount point for growth commands.

Example:

```bash
lvextend -L +5G /dev/vgdata/lvvar
xfs_growfs /var
```

If the filesystem is on a partition:

```bash
growpart /dev/sda 4
mount /dev/sda4 /var
xfs_growfs /var
```

You can verify the result with:

```bash
df -h /var
xfs_info /var
```

**Filesystem health checks**

Before and after resizing, validate the filesystem when appropriate.

For ext4:

```bash
e2fsck -f /dev/vgdata/lvhome
```

For XFS, use `xfs_repair` only when the filesystem is unmounted and you suspect corruption.

```bash
umount /data
xfs_repair /dev/vgdata/lvdata
```

Important points:

- `xfs_repair` is for repair, not routine resizing.
- Do not run `xfs_repair` on a mounted XFS filesystem.
- For normal XFS growth, `xfs_growfs` is the correct tool.

**Root filesystem resizing**

The root filesystem is special because it is mounted during normal operation.

For growth:

- If the root filesystem is on XFS, you can often grow it online if the underlying disk, partition, or LV has already been expanded.
- If the root filesystem is on ext4, growth is also possible online in many cases after the block device is expanded.

For shrink:

- In-place shrinking of the root filesystem is generally not practical for RHCSA workflows.
- For XFS root, shrinking is not supported.
- For ext4 root, you typically boot into rescue mode or use alternate media, unmount the root filesystem, and then perform the shrink steps.

A practical rescue-mode approach is:

1. Boot into rescue or emergency mode.
2. Activate LVM if needed.
3. Mount the installed system somewhere, if the rescue environment has not already done so.
4. Run filesystem checks and resize commands on the offline root filesystem.
5. Reboot and verify.

Example for an ext4 root LV in rescue mode:

```bash
lvm vgchange -ay
mount /dev/vg0/root /mnt/sysroot
e2fsck -f /dev/vg0/root
resize2fs /dev/vg0/root 15G
lvreduce -L 15G /dev/vg0/root
reboot
```

If the rescue environment already mounted the installed system, adjust the mount points accordingly. The key requirement is that the filesystem you shrink must be offline.

**Practical RHCSA command patterns**

Common commands you should recognize:

```bash
lsblk -f
df -Th
growpart /dev/sda 2
pvresize /dev/sda2
lvextend -r -L +5G /dev/vgdata/lvhome
resize2fs /dev/vgdata/lvhome
xfs_growfs /home
e2fsck -f /dev/vgdata/lvhome
partprobe /dev/sda
```

Useful verification commands:

```bash
mount | grep /home
findmnt /home
df -h /home
lvs -a -o +devices
```

**Decision guide**

| Situation | Recommended action |
|---|---|
| XFS on LVM needs more space | Extend LV, then run `xfs_growfs` |
| ext4 on LVM needs more space | Extend LV, then run `resize2fs` |
| ext4 on LVM needs less space | Unmount, `e2fsck`, shrink filesystem, then `lvreduce` |
| XFS needs less space | Migrate data to a new smaller filesystem |
| Partition-backed filesystem needs more space | Expand partition, then grow filesystem |
| Partition-backed ext4 needs less space | Shrink filesystem, then shrink partition |

**RHCSA-style practice scenarios**

1. You are told `/home` on `/dev/vgdata/lvhome` must grow by 2 GiB.
   - Check the filesystem type.
   - Extend the LV.
   - Grow the filesystem with the correct tool.
   - Verify with `df -h`.

2. You are told `/data` is ext4 on `/dev/sdb3` and must be reduced from 20 GiB to 12 GiB.
   - Unmount `/data`.
   - Run `e2fsck -f`.
   - Shrink the filesystem to 12 GiB.
   - Shrink the partition to match.
   - Remount and verify.

3. You are told the root filesystem is XFS and the disk was expanded.
   - Extend the partition or LV that backs `/`.
   - Grow the XFS filesystem online if possible.
   - Confirm with `df -h /`.

**Troubleshooting tips**

- If `resize2fs` complains, confirm the filesystem is unmounted for shrink operations.
- If `xfs_growfs` fails, confirm the filesystem is mounted and that you are using the mount point.
- If the kernel does not see a new partition size, run `partprobe` or reboot.
- If `pvresize` shows no change, verify that the underlying partition or disk actually grew.
- If `lvextend -r` fails, perform the filesystem resize manually after extending the LV.

**Exam strategy**

- Identify the filesystem type first.
- Determine whether the task is growth or shrink.
- Resize from the bottom up for growth: disk or partition, then PV, then LV, then filesystem.
- Resize from the top down for shrink: filesystem first, then LV or partition.
- Prefer the simplest safe tool that fits the scenario.
- Verify every change with `lsblk`, `df -h`, and the relevant LVM commands.
