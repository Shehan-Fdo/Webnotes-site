## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**LVM physical volumes: what these commands do**

Logical Volume Manager (<abbr title="Logical Volume Manager">LVM</abbr>) uses **physical volumes** (<abbr title="Physical Volume">PV</abbr>s) as the storage layer underneath volume groups and logical volumes. On RHCSA tasks, you usually prepare a disk or partition, initialize it with `pvcreate`, verify it with `pvs` or `pvdisplay`, then add it to a volume group with `vgcreate` or `vgextend`.

A PV is not just “a disk with a label.” LVM writes its own metadata structures to the device so it can identify the PV, track the volume group membership, and manage extents. That metadata is what makes LVM flexible, but it also means you must verify the target device carefully before initializing it.

- **PV metadata** includes an LVM label and metadata areas that store identifiers and allocation information.
- **Device discovery** matters because `pvcreate` writes to the selected block device and can destroy existing filesystem or partition data.
- **Verification** with `lsblk`, `blkid`, `pvs`, and `pvdisplay` helps confirm you are working on the intended device.

> `pvcreate` is destructive to the target device. Always confirm the device path before you initialize it.

**Device discovery before you create a PV**

Before you run `pvcreate`, identify the correct disk or partition. On exam systems, you often start with a newly attached virtual disk such as `/dev/sdb` or a partition such as `/dev/sdb1`.

Use `lsblk` to see the block device tree and current mount points:

```bash
lsblk -f
```

Use `blkid` to inspect existing filesystem signatures:

```bash
blkid /dev/sdb /dev/sdb1
```

Typical interpretation:

- `lsblk` shows the device hierarchy, size, and whether a partition already exists.
- `blkid` shows filesystem types and UUIDs if a signature is present.
- If a device already contains a filesystem or old LVM metadata, you should investigate before reusing it.

A practical workflow is:

1. Confirm the target device with `lsblk`.
2. Check for existing signatures with `blkid`.
3. Create a partition if your design calls for one.
4. Run `pvcreate` on the final block device.
5. Verify the result with `pvs` and `pvdisplay`.

**Whole disk or partition: what to use on RHCSA**

For RHCSA, both whole disks and partitions can be used as PVs, but the safest exam habit is to follow the task requirements and your site’s conventions.

- Use a **whole disk** when the task explicitly says to use the entire disk for LVM.
- Use a **partition** when you want clearer device management or when the task expects a partition table.
- Use a partition table when you need to separate storage roles or when the environment already uses partition-based layouts.

A common exam pattern is to create a single partition that spans the disk and then initialize that partition as a PV:

```bash
parted /dev/sdb --script mklabel gpt mkpart primary 1MiB 100%
pvcreate /dev/sdb1
```

Why the `1MiB` start offset matters:

- It aligns the partition for better compatibility and performance.
- It avoids edge cases with boot sectors and legacy alignment issues.
- It keeps the layout clean for later troubleshooting.

If the task says to use the whole disk, you can initialize the disk directly:

```bash
pvcreate /dev/sdb
```

**`pvcreate`: initialize a device for LVM**

`pvcreate` writes LVM metadata to a block device so LVM can treat it as a PV. It does not create a filesystem. It prepares the device for inclusion in a volume group.

Basic syntax:

```bash
pvcreate /dev/sdb1
```

Useful options:

- `-v` increases verbosity.
- `-ff` forces initialization in some conflict cases, but use it carefully.
- `-y` answers yes to prompts when you are certain the device is correct.

Example:

```bash
pvcreate -v /dev/sdb1
```

What happens conceptually:

- LVM writes an identifying label to the device.
- LVM stores metadata area information.
- The PV becomes available for volume group creation or extension.

> Do not use `-ff` as a routine cleanup shortcut. If a device already contains signatures, first determine whether `pvremove`, `wipefs`, or a different target device is the correct fix.

If you need to remove LVM metadata from a device, use `pvremove` rather than forcing `pvcreate` blindly:

```bash
pvremove /dev/sdb1
```

If stale filesystem signatures remain and you are intentionally repurposing the device, `wipefs` may be appropriate:

```bash
wipefs -a /dev/sdb1
```

Use these tools only when you are sure the data is no longer needed.

**`pvs`: quick PV inventory**

`pvs` provides a concise, tabular view of physical volumes. It is the fastest command for checking whether a device has been initialized and whether it belongs to a volume group.

Basic use:

```bash
pvs
```

Commonly useful fields include:

- `PV` — the device path, such as `/dev/sdb1`
- `VG` — the volume group name if the PV belongs to one
- `Fmt` — the metadata format, usually `lvm2`
- `Attr` — PV attribute flags used by LVM
- `PSize` — total size of the PV
- `PFree` — free extents or free space on the PV

Example output:

```bash
pvs
  PV         VG     Fmt  Attr PSize  PFree
  /dev/sdb1  vgdata lvm2  a--  20.00g 20.00g
```

How to read it:

- A populated `VG` column means the PV is already part of that volume group.
- A blank `VG` column usually means the PV is not assigned to a VG yet.
- `PFree` helps you see whether the PV has unallocated space available.
- `Attr` is an LVM-specific flag field; interpret it as metadata state, not as a generic “available” or “active” label.

For exam work, `pvs` is especially useful after each step:

- after `pvcreate`, to confirm the PV exists
- after `vgextend`, to confirm the PV joined the VG
- after troubleshooting, to compare expected and actual device membership

You can also request more detail with selected columns:

```bash
pvs -o pv_name,vg_name,pv_fmt,pv_attr,pv_size,pv_free
```

This is helpful when you want a clean, explicit view of the PV state.

**`pvdisplay`: detailed PV inspection**

`pvdisplay` shows a more verbose view of a physical volume. Use it when you need to inspect size, allocation, metadata, and extent mapping.

Basic use:

```bash
pvdisplay /dev/sdb1
```

Typical information includes:

- PV name and VG name
- PV size
- PE size, where PE means **physical extent**
- Total number of extents
- Allocatable status
- Free and allocated extents

Example output:

```bash
pvdisplay /dev/sdb1
  --- Physical volume ---
  PV Name               /dev/sdb1
  VG Name               vgdata
  PV Size               20.00 GiB / not usable 4.00 MiB
  Allocatable           yes
  PE Size               4.00 MiB
  Total PE              5119
  Free PE               5119
  Allocated PE          0
```

How to interpret the key fields:

- **PV Size** shows the usable size of the physical volume.
- **not usable** space is typically reserved for metadata and alignment.
- **PE Size** is the allocation unit used by LVM inside the VG.
- **Free PE** and **Allocated PE** tell you how much of the PV is still available for logical volumes.
- **Allocatable** indicates whether LVM can allocate extents from this PV at the moment.

The `Allocatable` field is useful for troubleshooting, but do not reduce it to a simple yes/no administrative toggle. It reflects whether the PV can currently contribute extents to LVM allocation decisions.

**Extent mapping with `pvdisplay -m`**

The `-m` option shows how extents map to logical volumes. This is valuable when you need to understand where data lives or why a VG is fragmented across multiple PVs.

Example:

```bash
pvdisplay -m /dev/sdb1
```

You may see mappings such as:

```bash
  --- Physical Segments ---
  Physical extent 0 to 255:
    Logical volume     /dev/vgdata/lvapp
    Logical extents    0 to 255
  Physical extent 256 to 511:
    FREE
```

Why this matters:

- It shows which PV extents are consumed by which logical volume.
- It helps you confirm whether a specific PV is carrying a particular LV segment.
- It is useful when planning migrations, resizing, or troubleshooting uneven allocation.

**Practical workflow: create, verify, then extend a VG**

A common RHCSA sequence is to create a PV, verify it, and then add it to a volume group.

1. Confirm the device exists:

```bash
lsblk -f /dev/sdb
blkid /dev/sdb /dev/sdb1
```

2. Create the partition if needed:

```bash
parted /dev/sdb --script mklabel gpt mkpart primary 1MiB 100%
```

3. Initialize the PV:

```bash
pvcreate /dev/sdb1
```

4. Verify the PV:

```bash
pvs /dev/sdb1
pvdisplay /dev/sdb1
```

5. Add it to a VG:

```bash
vgextend vgdata /dev/sdb1
```

6. Verify again:

```bash
pvs /dev/sdb1
pvdisplay /dev/sdb1
```

Expected change after `vgextend`:

- `VG` now shows `vgdata`
- `PFree` may still equal the full size if no logical volumes have been created yet
- `pvdisplay` shows the PV as part of the VG and reports the VG name

**Example before and after VG membership**

Before `vgextend`:

```bash
pvs /dev/sdb1
  PV         VG Fmt  Attr PSize  PFree
  /dev/sdb1     lvm2 a--  20.00g 20.00g
```

After `vgextend vgdata /dev/sdb1`:

```bash
pvs /dev/sdb1
  PV         VG     Fmt  Attr PSize  PFree
  /dev/sdb1  vgdata lvm2 a--  20.00g 20.00g
```

The exact `Attr` flags can vary by state and LVM version, so focus on the device, VG membership, and free space rather than memorizing a single character pattern.

**Troubleshooting and verification patterns**

When a PV does not behave as expected, use a layered check:

- `lsblk` confirms the kernel sees the device and partition layout.
- `blkid` confirms whether filesystem signatures still exist.
- `pvs` confirms whether LVM recognizes the PV.
- `pvdisplay` confirms detailed PV state and VG membership.

Common situations:

- The device appears in `lsblk` but not in `pvs`.
  - The device may not have been initialized with `pvcreate`.
  - You may have created a PV on a different path than expected.
- `blkid` shows an old filesystem signature.
  - The device may still contain non-LVM metadata.
  - You may need to remove stale signatures before reusing the device.
- `pvdisplay` shows the PV but not the expected VG.
  - The PV may not have been added to the VG yet.
  - You may be looking at the wrong partition or disk.

A useful verification sequence after any storage change is:

```bash
lsblk -f
blkid
pvs
pvdisplay
```

**Exam-oriented command examples**

Create a new PV on a partition:

```bash
parted /dev/sdb --script mklabel gpt mkpart primary 1MiB 100%
pvcreate /dev/sdb1
pvs /dev/sdb1
```

Inspect detailed PV information:

```bash
pvdisplay /dev/sdb1
```

Show extent mapping:

```bash
pvdisplay -m /dev/sdb1
```

Prepare a disk directly as a PV:

```bash
pvcreate /dev/sdc
pvs /dev/sdc
```

Remove LVM metadata when repurposing a device:

```bash
pvremove /dev/sdb1
wipefs -a /dev/sdb1
```

**Best practices for RHCSA tasks**

- Verify the target device with `lsblk` before writing anything.
- Prefer a partition when the task or environment expects one.
- Use `pvcreate` only on the final intended block device.
- Confirm the result with `pvs` and `pvdisplay`.
- Use `pvdisplay -m` when you need to understand extent placement.
- Avoid forcing metadata changes unless you have confirmed the device is safe to reuse.

**Quick mental model**

Think of the LVM stack like this:

- **PV** = the raw storage object initialized for LVM
- **VG** = the pool built from one or more PVs
- **LV** = the logical volume carved from the VG

`pvcreate` prepares the base layer, `pvs` gives you a fast inventory, and `pvdisplay` gives you the detailed state needed for verification and troubleshooting.
