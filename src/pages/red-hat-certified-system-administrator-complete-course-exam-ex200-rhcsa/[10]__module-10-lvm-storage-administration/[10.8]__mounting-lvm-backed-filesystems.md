## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**LVM-backed filesystem overview**

<abbr title="Logical Volume Manager">LVM</abbr> lets you present storage as flexible logical volumes instead of fixed partitions. For RHCSA tasks, you usually create or identify a physical volume, build a volume group, create a logical volume, put a filesystem on it, and mount it either temporarily or persistently.

The key exam skill is not just creating storage, but making it usable after reboot. That means you must understand the full path from block device to mount point:

- **Physical volume**: the storage object LVM can use, such as a whole disk or a partition
- **Volume group**: pooled storage built from one or more physical volumes
- **Logical volume**: the block device you format and mount
- **Filesystem**: the structure that stores files, such as XFS or ext4
- **Mount point**: the directory where the filesystem becomes accessible

On current RHEL systems, **XFS** is the default filesystem for many installations. For RHCSA, the important point is to choose the filesystem that matches the task and to mount it correctly, not to assume one filesystem is universally best for every case.

**Typical workflow**

A common RHCSA workflow looks like this:

1. Identify the storage device with `lsblk`, `blkid`, or `fdisk -l`.
2. Create or confirm the LVM structure.
3. Create a filesystem on the logical volume.
4. Create a mount point directory.
5. Mount the filesystem with `mount`.
6. Verify the mount.
7. Add a persistent entry to `/etc/fstab`.
8. Test the persistent configuration safely.

Example device discovery:

```bash
lsblk
blkid
sudo pvs
sudo vgs
sudo lvs
```

**LVM building blocks**

LVM separates storage management into layers so you can resize and reorganize more easily than with traditional partitions alone. In exam scenarios, you may be given:

- a whole disk intended for LVM use
- a partition already marked for LVM
- an existing logical volume that only needs formatting and mounting

If you are creating the LVM stack yourself, the commands usually follow this order:

```bash
sudo pvcreate /dev/sdb
sudo vgcreate vgdata /dev/sdb
sudo lvcreate -n lvapp -L 5G vgdata
```

If the disk is already partitioned for LVM, use the partition path instead, such as `/dev/sdb1`. The important exam habit is to confirm the actual device name before running destructive commands.

> Be careful not to format or initialize the wrong device. Always verify with `lsblk` and `pvs` before creating a physical volume.

**Creating the filesystem**

After the logical volume exists, create a filesystem on it. On RHEL, XFS is commonly used for new filesystems, while ext4 may also appear in exam tasks or legacy systems.

```bash
sudo mkfs.xfs /dev/vgdata/lvapp
```

For ext4:

```bash
sudo mkfs.ext4 /dev/vgdata/lvapp
```

Important filesystem behavior:

- **XFS** grows online with `xfs_growfs` after the filesystem is mounted.
- **ext4** can grow online with `resize2fs` after the block device is expanded and the filesystem is mounted.
- **ext4 shrinking** is not an online operation and requires offline handling; do not assume you can shrink it while mounted.
- **XFS shrinking** is not supported.

**Mounting temporarily**

A temporary mount is useful for testing or immediate access. You mount the logical volume to a directory that already exists.

```bash
sudo mkdir -p /data
sudo mount /dev/vgdata/lvapp /data
```

Verify the result:

```bash
findmnt /data
df -hT /data
mount | grep /data
```

The `findmnt` command is especially useful because it shows the source device, target mount point, and filesystem type in a readable format.

**Persistent mounting with `/etc/fstab`**

A mount that must survive reboot belongs in `/etc/fstab`. Use the filesystem UUID or the logical volume path. UUIDs are often more robust because device names can change.

Get the UUID:

```bash
blkid /dev/vgdata/lvapp
```

Example `/etc/fstab` entry:

```fstab
UUID=12345678-1234-1234-1234-123456789abc  /data  xfs  defaults  0  0
```

You can also use the device path:

```fstab
/dev/vgdata/lvapp  /data  xfs  defaults  0  0
```

UUID-based entries are usually safer for exam and production use because they are less dependent on device enumeration order.

**`/etc/fstab` field meanings**

Each line in `/etc/fstab` has six fields:

| Field | Meaning | Example |
|---|---|---|
| 1 | Device, UUID, or label | `UUID=...` |
| 2 | Mount point | `/data` |
| 3 | Filesystem type | `xfs` |
| 4 | Mount options | `defaults` |
| 5 | Dump flag | `0` |
| 6 | Filesystem check order | `0` for XFS |

For RHCSA, remember:

- **XFS** uses `0` in the sixth field because `fsck` does not check XFS in the same way as ext filesystems.
- ext4 commonly uses `1` or `2` depending on the root filesystem and other mounts, but follow the task requirements.

**Mount options you should know**

Mount options control behavior at mount time. Common options include:

- `defaults` for standard read-write behavior
- `ro` for read-only access
- `noexec` to prevent execution of binaries from the mount
- `nosuid` to ignore setuid/setgid bits
- `nodev` to ignore device files on the mount
- `relatime` as a common timestamp update policy

Example:

```fstab
UUID=12345678-1234-1234-1234-123456789abc  /data  xfs  defaults,noexec,nodev  0  0
```

Use restrictive options when the task calls for them, especially on shared or user-writable data mounts.

**Safe testing of `/etc/fstab`**

After editing `/etc/fstab`, test it without rebooting. A syntax error can prevent the system from mounting filesystems correctly during boot.

1. Save the file.
2. Run `sudo mount -a`.
3. Check for errors.
4. Verify the mount with `findmnt` or `df -hT`.

Example:

```bash
sudo mount -a
findmnt /data
```

If `mount -a` returns an error, inspect `/etc/fstab` for typos, wrong UUIDs, wrong filesystem types, or missing mount points.

**Troubleshooting mount failures**

When a mount fails, work from the device upward:

- Confirm the logical volume exists with `lvs`
- Confirm the filesystem exists with `blkid`
- Confirm the mount point directory exists
- Confirm the `/etc/fstab` entry is correct
- Check recent boot logs if the failure happened during startup

Useful commands:

```bash
sudo lvs
sudo blkid
sudo ls -ld /data
sudo mount -av
journalctl -xb
```

`journalctl -xb` is often the most useful starting point for boot-related mount problems because it shows messages from the current boot. Use `journalctl -xe` when you want a broader view of recent system events.

Common causes of failure:

- wrong UUID in `/etc/fstab`
- typo in the filesystem type
- missing mount point directory
- filesystem corruption
- trying to mount a device that has no filesystem yet
- using the wrong device path, such as a whole disk instead of the intended partition or logical volume

> If `/etc/fstab` is wrong, the system may still boot, but the mount can fail and drop you into emergency mode depending on the filesystem and boot dependency.

**Resizing LVM-backed filesystems**

One advantage of LVM is that you can expand storage more easily than with fixed partitions. The general sequence is:

1. Extend the logical volume.
2. Grow the filesystem to match.
3. Verify the new size.

Example for XFS:

```bash
sudo lvextend -L +2G /dev/vgdata/lvapp
sudo xfs_growfs /data
```

Example for ext4:

```bash
sudo lvextend -L +2G /dev/vgdata/lvapp
sudo resize2fs /dev/vgdata/lvapp
```

Notes on resizing:

- **XFS** grows only while mounted.
- **ext4** can grow online in many cases after the block device is extended.
- Shrinking is much more constrained and is not a routine RHCSA task for XFS.
- Always verify the filesystem type before choosing the resize tool.

Check the result:

```bash
df -hT /data
lvs
```

**Practical exam workflow**

A realistic RHCSA task may ask you to make a new 2 GiB filesystem available at `/srv/projects` and persist it across reboots. A clean approach is:

1. Identify the storage device or partition.
2. Create the LVM objects if needed.
3. Create the filesystem.
4. Create the mount point.
5. Mount it.
6. Add the `/etc/fstab` entry.
7. Test with `mount -a`.
8. Verify with `findmnt`.

Example:

```bash
sudo pvcreate /dev/sdb1
sudo vgcreate vgproj /dev/sdb1
sudo lvcreate -n lvproj -L 2G vgproj
sudo mkfs.xfs /dev/vgproj/lvproj
sudo mkdir -p /srv/projects
sudo mount /dev/vgproj/lvproj /srv/projects
sudo blkid /dev/vgproj/lvproj
```

Then add the UUID to `/etc/fstab`:

```fstab
UUID=12345678-1234-1234-1234-123456789abc  /srv/projects  xfs  defaults  0  0
```

Test it:

```bash
sudo umount /srv/projects
sudo mount -a
findmnt /srv/projects
```

**Verification commands**

Use these commands to confirm each layer:

- `pvs` to inspect physical volumes
- `vgs` to inspect volume groups
- `lvs` to inspect logical volumes
- `lsblk` to view the block-device tree
- `blkid` to show filesystem UUIDs and types
- `findmnt` to confirm active mounts
- `df -hT` to show mounted filesystem usage and type

A quick inspection sequence:

```bash
lsblk -f
pvs
vgs
lvs
findmnt
```

**Common mistakes to avoid**

- Formatting the wrong device
- Mounting a filesystem without creating the mount point
- Forgetting to add the `/etc/fstab` entry
- Using the wrong filesystem type in `/etc/fstab`
- Forgetting to test with `mount -a`
- Assuming a mount is persistent because it works right now
- Using `fsck` expectations on XFS
- Trying to shrink XFS

**Exam-focused reminders**

For RHCSA, you should be able to:

- create or recognize LVM storage components
- format a logical volume with the correct filesystem
- mount it manually
- make the mount persistent with `/etc/fstab`
- verify the mount and troubleshoot failures
- extend an LVM-backed filesystem when required

The exam rewards careful verification. If a task says the filesystem must be available after reboot, do not stop at `mount`; always confirm the persistent configuration.
