## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**LVM resizing overview**

Logical Volume Manager (**LVM**) lets you resize storage more flexibly than fixed partitions. On RHCSA, you are expected to know how to extend a logical volume, grow the filesystem on it, and reduce it only when the filesystem and workflow allow it.

The key idea is that the logical volume and the filesystem are separate layers. You must resize both, and the order matters. Extending is usually straightforward. Reducing is more constrained because you must shrink the filesystem first, then reduce the logical volume, and only for filesystems that support shrinking.

- **PV**: physical volume, the disk or partition added to LVM.
- **VG**: volume group, the storage pool built from one or more PVs.
- **LV**: logical volume, the allocatable block device created from a VG.
- **Filesystem**: the data structure on top of the LV, such as `xfs` or `ext4`.

**Inspecting current layout**

Before changing size, verify what exists and how much free space is available in the volume group. This helps you choose whether to extend by a fixed amount or consume all remaining space.

```bash
pvs
vgs
lvs
lsblk
df -hT
```

Useful fields to check:

- `pvs` shows physical volumes and their free space.
- `vgs` shows total and free extents in each volume group.
- `lvs` shows logical volume names, sizes, and attributes.
- `df -hT` shows mounted filesystems and filesystem types.
- `lsblk` helps you map devices, mount points, and layers.

A practical habit is to confirm the filesystem type before planning a resize. `xfs` and `ext4` behave differently when shrinking.

**Extending a logical volume**

Extending is the common case. You can grow the LV first, then grow the filesystem. If the filesystem supports online growth and the volume is mounted, you can often complete the change without unmounting.

Typical extension commands:

```bash
lvextend -L +2G /dev/rhel/home
```

```bash
lvextend -l +100%FREE /dev/rhel/home
```

- `-L +2G` adds 2 GiB to the current LV size.
- `-l +100%FREE` consumes all remaining free extents in the VG.
- Use `+` when you want to add space relative to the current size.
- Use an absolute size like `-L 6G` only when you want the final LV size to be exactly 6 GiB and the current LV is smaller than that value.

> `lvextend -L 6G /dev/rhel/home` does not mean “add 6G.” It means “set the final size to 6G.” If the LV is already larger than 6G, the command fails.

After extending the LV, grow the filesystem.

For `xfs`, use `xfs_growfs` on the mount point:

```bash
xfs_growfs /home
```

For `ext4`, use `resize2fs` on the block device:

```bash
resize2fs /dev/rhel/home
```

A common RHCSA-safe pattern is to use the `-r` option with `lvextend`, which attempts to resize the filesystem automatically after extending the LV:

```bash
lvextend -r -L +2G /dev/rhel/home
```

This is convenient, but you should still understand the manual sequence because exam tasks may require explicit steps or troubleshooting.

**Filesystem-specific behavior**

`xfs` is a modern filesystem commonly used on RHEL. It can grow while mounted, but it cannot shrink. That makes it a good choice for systems where online expansion matters more than reduction.

`ext4` can usually be grown while mounted if the kernel and filesystem support it. It can also be shrunk, but only offline, meaning the filesystem must be unmounted first.

| Filesystem | Grow online | Shrink | Notes |
|---|---:|---:|---|
| `xfs` | Yes | No | Grow with `xfs_growfs` on the mount point |
| `ext4` | Usually yes | Yes, offline only | Shrink with `resize2fs` after unmounting |

**Growing an XFS filesystem**

For `xfs`, the LV and filesystem growth are separate steps unless you use `lvextend -r`.

Example workflow:

```bash
lvextend -L +5G /dev/rhel/data
xfs_growfs /data
```

Important points:

- `xfs_growfs` works on the mounted filesystem, not the raw block device.
- The mount point must be active.
- The filesystem grows to fill the newly available LV space.

You can verify the result with:

```bash
df -hT /data
lvs /dev/rhel/data
```

**Growing an ext4 filesystem**

For `ext4`, you can usually grow the filesystem after extending the LV, even while mounted.

Example workflow:

```bash
lvextend -L +1G /dev/rhel/home
resize2fs /dev/rhel/home
```

If you want a single-step approach and the environment supports it:

```bash
lvextend -r -L +1G /dev/rhel/home
```

Verification:

```bash
df -hT /home
lvs /dev/rhel/home
```

**Reducing a logical volume safely**

Reducing is more delicate than extending because data loss is possible if you shrink in the wrong order or shrink a filesystem that does not support it.

The safe general sequence for shrinkable filesystems is:

1. Back up important data.
2. Unmount the filesystem.
3. Check and repair the filesystem if needed.
4. Shrink the filesystem to a size smaller than the target LV.
5. Reduce the logical volume to the target size.
6. Remount and verify.

> `xfs` cannot be shrunk. If the LV contains `xfs`, you must migrate data elsewhere, recreate the LV smaller, and restore the data.

**Reducing ext4 volumes**

`ext4` supports shrinking, but only offline. You must unmount the filesystem before shrinking it.

A safe example workflow:

```bash
umount /dev/rhel/home
e2fsck -f /dev/rhel/home
resize2fs /dev/rhel/home 3G
lvreduce -L 3G /dev/rhel/home
mount /home
```

Important safety notes:

- Shrink the filesystem to a size **at or below** the final LV size, but leave practical headroom when possible.
- Verify the filesystem shrink completed successfully before running `lvreduce`.
- Do not assume matching numbers are always enough in every situation; the filesystem must fit comfortably inside the new LV size.
- `e2fsck -f` is important because shrinking a filesystem requires a clean filesystem check.

A more conservative approach is to shrink the filesystem slightly smaller than the target LV, then reduce the LV to the final size:

```bash
resize2fs /dev/rhel/home 2900M
lvreduce -L 3G /dev/rhel/home
```

This gives you some margin for metadata and avoids cutting the filesystem too close to the limit.

**Reducing with `lvreduce`**

`lvreduce` changes the LV size downward. It is destructive if the filesystem is still larger than the new LV. Use it only after the filesystem has been safely shrunk.

Example:

```bash
lvreduce -L -1G /dev/rhel/home
```

or with an absolute final size:

```bash
lvreduce -L 3G /dev/rhel/home
```

- `-L -1G` subtracts 1 GiB from the current LV size.
- `-L 3G` sets the final LV size to 3 GiB.
- Use `-y` only when you are certain and want to skip confirmation prompts.

**Using `lvresize` as a combined tool**

`lvresize` can grow or shrink an LV, and with `-r` it can also attempt filesystem resizing. It is useful, but on exams you should still know the manual commands.

Examples:

```bash
lvresize -r -L +2G /dev/rhel/data
```

```bash
lvresize -r -L 3G /dev/rhel/home
```

The `-r` option is convenient for supported filesystems, but you should not rely on it blindly for shrink operations. Always confirm filesystem type and support first.

**Verification after resizing**

After any resize, verify both the block device and the mounted filesystem.

```bash
lvs
lsblk
df -hT
```

What to confirm:

- The LV size matches the intended target.
- The filesystem type is still correct.
- The mount point shows the expected usable space.
- The data is accessible and intact.

If the filesystem does not reflect the new size, you likely resized only the LV or used the wrong filesystem command.

**Common RHCSA command patterns**

Extend an LV and grow an XFS filesystem:

```bash
lvextend -L +2G /dev/rhel/data
xfs_growfs /data
```

Extend an LV and grow an ext4 filesystem:

```bash
lvextend -L +2G /dev/rhel/home
resize2fs /dev/rhel/home
```

Shrink an ext4 filesystem and LV:

```bash
umount /home
e2fsck -f /dev/rhel/home
resize2fs /dev/rhel/home 3G
lvreduce -L 3G /dev/rhel/home
mount /home
```

Extend using all free space in the VG:

```bash
lvextend -l +100%FREE /dev/rhel/data
xfs_growfs /data
```

**Troubleshooting and best practices**

- Use `pvs`, `vgs`, and `lvs` to identify where free space exists before resizing.
- Confirm the mount point and filesystem type before choosing the resize command.
- Prefer `+` syntax for growth when you want to add space rather than set a final size.
- Remember that `xfs` grows online but does not shrink.
- Remember that `ext4` shrink operations require unmounting and a filesystem check.
- If a command fails, re-check whether you used the device path or the mount point correctly.
- Keep backups before any shrink operation.

**Practical exam mindset**

On RHCSA, the safest approach is to think in layers:

- First, decide whether you are changing the LV, the filesystem, or both.
- Second, identify the filesystem type.
- Third, choose the correct order of operations.
- Fourth, verify the result with `df`, `lvs`, and `lsblk`.

That workflow reduces mistakes and helps you recover quickly if a command does not behave as expected.
