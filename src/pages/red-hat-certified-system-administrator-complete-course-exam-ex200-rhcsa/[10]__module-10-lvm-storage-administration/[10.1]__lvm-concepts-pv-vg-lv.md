## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**LVM overview**

<abbr title="Logical Volume Manager">LVM</abbr> gives you a flexible storage layer between physical disks and mounted filesystems. Instead of binding a filesystem directly to one partition, you group storage into reusable pools and carve out logical volumes as needed.

This design helps you:
- grow filesystems with less downtime,
- combine multiple disks into one storage pool,
- separate data by purpose,
- take snapshots for short-term recovery tasks.

At a high level, LVM uses three core objects:
- **PV**: physical volume
- **VG**: volume group
- **LV**: logical volume

A useful mental model is:
- PVs provide raw storage blocks.
- VGs pool those blocks together.
- LVs consume space from the pool and appear as block devices you can format and mount.

**Core object relationships**

| Object | Full name | Role | Typical command family |
|---|---|---|---|
| PV | Physical Volume | Initializes a disk or partition for LVM use | `pvcreate`, `pvs`, `pvdisplay` |
| VG | Volume Group | Pools one or more PVs into shared storage | `vgcreate`, `vgs`, `vgdisplay` |
| LV | Logical Volume | Allocates usable block storage from a VG | `lvcreate`, `lvs`, `lvdisplay` |

The relationship is hierarchical:
- One or more PVs form one VG.
- One VG can contain many LVs.
- An LV is what you usually format with `xfs` or `ext4`, then mount.

**Physical volumes**

A **PV** is a disk or partition that LVM has initialized with metadata so it can participate in a volume group. You can use whole disks or partitions.

Common examples:
- `/dev/sdb`
- `/dev/sdc1`

You do not need a special partition flag for `pvcreate` to work, but marking the partition type as Linux LVM can help humans and tools identify intent. On GPT disks, that usually means setting the partition type to `Linux LVM`.

Typical workflow:
1. Create a partition if you are not using the whole disk.
2. Initialize it with `pvcreate`.
3. Verify it with `pvs` or `pvdisplay`.

```bash
sudo pvcreate /dev/sdb1
sudo pvs
sudo pvdisplay /dev/sdb1
```

Important PV facts:
- LVM stores metadata on the PV itself.
- The same VG metadata is written as copies onto the PVs that belong to that VG.
- A PV can belong to only one VG at a time.
- You can add more PVs later to expand capacity.

**Volume groups**

A **VG** is a storage pool built from one or more PVs. It abstracts the underlying disks so you can allocate space from a shared pool rather than from a single device.

Why use a VG:
- You can grow storage without redesigning the filesystem layout.
- You can spread one logical storage pool across multiple disks.
- You can isolate workloads by using separate VGs for different purposes.

Design choice guidance:
- Use **one VG** when you want flexible shared capacity across related data.
- Use **separate VGs** when you want administrative isolation, different backup policies, or clearer failure boundaries.
- Be cautious about putting critical unrelated workloads into one VG if you want to limit blast radius.

Create a VG:
```bash
sudo vgcreate vg_data /dev/sdb1 /dev/sdc1
sudo vgs
sudo vgdisplay vg_data
```

Useful VG options and concepts:
- `-s` sets the **physical extent** size at VG creation time.
- `-n` names the VG.
- `vgs` shows summary information.
- `vgextend` adds more PVs to an existing VG.
- `vgreduce` removes a PV from a VG after data has been moved away.

Example with a custom extent size:
```bash
sudo vgcreate -s 16M vg_data /dev/sdb1 /dev/sdc1
```

**Extents and allocation**

LVM divides storage into fixed-size chunks called **physical extents** (PEs). LVs are built from matching **logical extents** (LEs). In practice, PE and LE sizes usually match one-to-one within a VG, but the PE size is chosen when the VG is created and is not globally fixed.

Why extents matter:
- They determine allocation granularity.
- They affect how precisely you can size an LV.
- They influence snapshot and thin provisioning behavior indirectly through space management.

Practical implications:
- Smaller extents give finer allocation control.
- Larger extents can simplify management for large volumes.
- You usually think in extents when using `-l` with `lvcreate`.

Examples:
- `-L 10G` allocates by size.
- `-l 100%FREE` allocates all remaining free extents.
- `-l 50%VG` allocates half of the VG’s extents.

```bash
sudo lvcreate -n lv_app -L 10G vg_data
sudo lvcreate -n lv_logs -l 100%FREE vg_data
```

**Logical volumes**

An **LV** is the block device you actually format and mount. It behaves like a disk partition from the filesystem’s point of view, but it can be resized more flexibly.

Common LV tasks:
- create a new LV,
- extend an existing LV,
- reduce an LV only when the filesystem supports it and you have a safe procedure,
- snapshot an LV for short-term consistency tasks.

Create and format an LV:
```bash
sudo lvcreate -n lv_home -L 20G vg_data
sudo mkfs.xfs /dev/vg_data/lv_home
sudo mkdir -p /home
sudo mount /dev/vg_data/lv_home /home
```

Useful display commands:
```bash
sudo lvs
sudo lvdisplay /dev/vg_data/lv_home
sudo lsblk
```

Naming conventions:
- VG names and LV names should be descriptive.
- Device paths usually appear as `/dev/<vgname>/<lvname>`.
- You can also see them under `/dev/mapper/`.

**Common LVM command patterns**

Use these commands often when inspecting or managing LVM:

```bash
sudo pvs
sudo vgs
sudo lvs
sudo pvdisplay
sudo vgdisplay
sudo lvdisplay
```

Creation and expansion examples:
```bash
sudo pvcreate /dev/sdd1
sudo vgextend vg_data /dev/sdd1
sudo lvextend -L +5G /dev/vg_data/lv_home
sudo xfs_growfs /home
```

For ext4, grow the filesystem after extending the LV:
```bash
sudo resize2fs /dev/vg_data/lv_home
```

For XFS, grow the mounted filesystem with `xfs_growfs`:
```bash
sudo xfs_growfs /mountpoint
```

> XFS can grow online but cannot shrink. Plan carefully before choosing a filesystem and layout.

**Metadata backup and restore**

LVM metadata is critical because it describes how PVs, VGs, and LVs fit together. Red Hat systems commonly keep automatic metadata backups under `/etc/lvm/backup/` and archive copies under `/etc/lvm/archive/`.

Useful commands:
- `vgcfgbackup` saves current VG metadata.
- `vgcfgrestore` restores VG metadata from a backup file.

Example:
```bash
sudo vgcfgbackup vg_data
sudo ls /etc/lvm/backup/
sudo ls /etc/lvm/archive/
```

Restore is a recovery tool, not a routine operation:
```bash
sudo vgcfgrestore vg_data
```

Use metadata backup and restore when:
- a VG configuration becomes corrupted,
- you need to recover after accidental metadata damage,
- you are troubleshooting a failed LVM import or device issue.

**Snapshots**

An LVM snapshot captures a point-in-time view of an LV. Classic snapshots are useful for short-term consistency tasks such as making a backup while the source volume stays online.

Important caveats:
- Classic snapshots are not a full backup.
- They have capacity limits and can fill up.
- Performance can degrade as snapshot activity increases.
- They are usually best used briefly, then removed.
- On modern systems, thin snapshots are a separate feature from classic snapshots.

Typical snapshot workflow:
1. Create the snapshot with enough space for expected changes.
2. Mount it read-only if you only need to read data.
3. Use it for backup or verification.
4. Remove it when finished.

Example:
```bash
sudo lvcreate -s -n lv_home_snap -L 2G /dev/vg_data/lv_home
sudo mkdir -p /mnt/snap
sudo mount -o ro /dev/vg_data/lv_home_snap /mnt/snap
```

After use, remove the snapshot:
```bash
sudo umount /mnt/snap
sudo lvremove /dev/vg_data/lv_home_snap
```

Snapshot planning tips:
- Size the snapshot based on expected write activity during its lifetime.
- Keep the snapshot window short.
- Monitor free space in the VG and snapshot usage.
- Do not treat snapshots as long-term archival storage.

**Safe removal and cleanup**

Remove LVM components in the correct order to avoid data loss and busy-device errors.

Typical removal sequence:
1. Unmount the filesystem.
2. Remove any `/etc/fstab` entry if the mount should not return at boot.
3. Remove the LV.
4. Remove the VG if it is no longer needed.
5. Remove the PV label if the disk should return to non-LVM use.

Examples:
```bash
sudo umount /home
sudo lvremove /dev/vg_data/lv_home
sudo vgremove vg_data
sudo pvremove /dev/sdb1
```

If the LV is still mounted, `lvremove` usually fails. Always verify with:
```bash
mount | grep vg_data
lsblk
```

**Boot-time and `/etc/fstab` troubleshooting**

LVM-backed filesystems often fail at boot because of a bad `/etc/fstab` entry, a missing VG, or a renamed device path. When this happens, the system may drop into emergency mode.

Common recovery checks:
- Confirm the VG and LV exist with `vgs` and `lvs`.
- Confirm the filesystem UUID with `blkid`.
- Check `/etc/fstab` for typos.
- Use the correct device path or UUID.

Example `/etc/fstab` entry:
```fstab
UUID=1234-ABCD  /data  xfs  defaults  0 0
```

Troubleshooting steps:
1. Boot into emergency mode or rescue mode if needed.
2. Run `lvm pvscan`, `vgscan`, and `lvscan` if volumes are not visible.
3. Activate the VG if necessary with `vgchange -ay`.
4. Fix `/etc/fstab` or mount options.
5. Test with `mount -a` before rebooting.

Useful discovery commands:
```bash
sudo pvscan
sudo vgscan
sudo lvscan
sudo vgchange -ay
sudo blkid
```

> If `/etc/fstab` is wrong, the system may still boot if you comment out the bad line or use `nofail` for noncritical mounts.

**Practical design choices**

LVM is most useful when you need flexibility, but that flexibility should be intentional.

Good reasons to use multiple PVs in one VG:
- you want one shared storage pool,
- you expect to grow capacity over time,
- you want to allocate space dynamically among related workloads.

Good reasons to use separate VGs:
- you want isolation between applications,
- you want different snapshot or backup policies,
- you want to keep system and data storage separate,
- you want to reduce the impact of one workload consuming all free space.

A simple example layout:
- `vg_root` for operating system volumes,
- `vg_data` for application data,
- `vg_backup` for temporary backup staging or scratch space.

This separation makes troubleshooting easier and helps prevent one service from exhausting all available storage.

**Quick reference**

- **PV**: initialized disk or partition used by LVM.
- **VG**: pool of storage built from one or more PVs.
- **LV**: allocatable block device created from a VG.
- **PE/LE**: allocation units used by LVM; PE size is set per VG.
- **Snapshot**: short-term point-in-time copy, not a backup replacement.
- **Metadata backup**: use `vgcfgbackup` and `vgcfgrestore` for recovery.

**Minimal end-to-end example**

```bash
sudo pvcreate /dev/sdb1
sudo vgcreate vg_data /dev/sdb1
sudo lvcreate -n lv_share -L 5G vg_data
sudo mkfs.xfs /dev/vg_data/lv_share
sudo mkdir -p /srv/share
sudo mount /dev/vg_data/lv_share /srv/share
echo '/dev/vg_data/lv_share /srv/share xfs defaults 0 0' | sudo tee -a /etc/fstab
```

This sequence shows the full lifecycle:
- initialize the PV,
- create the VG,
- allocate an LV,
- create a filesystem,
- mount it,
- persist it across reboots.
