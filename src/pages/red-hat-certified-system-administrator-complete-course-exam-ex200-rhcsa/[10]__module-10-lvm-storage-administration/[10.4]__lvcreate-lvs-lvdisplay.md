**Logical Volume Manager overview**

The <abbr title="Logical Volume Manager">LVM</abbr> layer gives you flexible storage management by separating filesystems from physical disks. You use it when you need easier resizing, better abstraction, or a cleaner way to manage multiple disks.

LVM organizes storage in three main layers:

- **Physical volumes** (`PV`): disks or partitions initialized for LVM.
- **Volume groups** (`VG`): storage pools built from one or more physical volumes.
- **Logical volumes** (`LV`): block devices carved from a volume group and formatted with a filesystem.

This model lets you grow storage without redesigning the whole layout. In RHCSA tasks, you commonly create a logical volume, format it, mount it, and verify it with LVM display commands.

**Core commands and their purpose**

The three commands in this topic cover creation and inspection:

- `lvcreate` creates a new logical volume inside a volume group.
- `lvs` shows a compact summary of logical volumes.
- `lvdisplay` shows detailed logical volume information.

You often pair them with related commands such as `pvs`, `vgs`, `pvcreate`, `vgcreate`, `mkfs`, and `mount`.

**Creating a logical volume with `lvcreate`**

`lvcreate` allocates space from a volume group and creates a block device under `/dev/<vgname>/<lvname>` and usually also under `/dev/mapper/<vgname>-<lvname>`.

Common syntax:

```bash
lvcreate -L <size> -n <lvname> <vgname>
```

Example:

```bash
lvcreate -L 10G -n data vg0
```

This creates a 10 GiB logical volume named `data` in the volume group `vg0`.

You can also create an LV by percentage of free space:

```bash
lvcreate -l 100%FREE -n backup vg0
```

Useful options:

- `-L` sets an absolute size such as `5G` or `500M`.
- `-l` sets extents, often with `100%FREE` or `50%VG`.
- `-n` names the logical volume.
- `-s` creates a snapshot, if supported and needed.
- `-t` performs a test run without making changes.

A practical workflow is to create the LV, format it, and mount it:

```bash
lvcreate -L 2G -n appdata vg0
mkfs.xfs /dev/vg0/appdata
mkdir -p /srv/appdata
mount /dev/vg0/appdata /srv/appdata
```

> If you create the LV but forget to format it, the device exists but cannot store filesystems yet.

**Understanding size units and extents**

LVM internally allocates storage in <abbr title="Physical Extent">PE</abbr>s, or physical extents. When you use `-L`, LVM converts the requested size into extents. When you use `-l`, you work directly with extents.

Why this matters:

- `-L` is easier for human-readable sizing.
- `-l` is more precise when you want to consume all remaining space or a specific extent count.
- Extent-based allocation avoids ambiguity when you need exact VG usage.

Example:

```bash
lvcreate -l 100%FREE -n logs vg0
```

This is a common exam-friendly pattern because it uses all remaining free space in the volume group.

**Inspecting logical volumes with `lvs`**

`lvs` provides a concise, script-friendly summary of logical volumes. It is often the fastest way to confirm that an LV exists and to check its size, attributes, and volume group.

Example:

```bash
lvs
```

Typical output fields include:

- `LV`: logical volume name.
- `VG`: volume group name.
- `Attr`: attribute flags describing state and type.
- `LSize`: logical volume size.
- `Pool`, `Origin`, or `Data%`: shown for special LV types such as thin volumes or snapshots.

You can request more detail with custom columns:

```bash
lvs -o lv_name,vg_name,lv_size,lv_attr,devices
```

This is useful when you want to see where the LV lives and how it is mapped.

Common uses:

- Verify that creation succeeded.
- Check whether the LV size matches the requested size.
- Confirm whether an LV is active.
- Inspect snapshot or thin-provisioning metadata when relevant.

**Reading `lvs` attributes**

The `Attr` field is compact but important. It encodes LV type and state. You do not need to memorize every flag for RHCSA, but you should recognize that it tells you whether the LV is active, open, a snapshot, thin, or read-only.

Practical interpretation:

- An active LV can usually be mounted.
- A suspended or inactive LV may indicate a problem or an administrative state.
- Snapshot-related flags indicate special storage behavior and space usage.

If you are troubleshooting, `lvs` is often the first command to run because it gives a quick status overview.

**Inspecting detailed metadata with `lvdisplay`**

`lvdisplay` shows verbose information about one or more logical volumes. It is more detailed than `lvs` and is useful when you need exact paths, allocation details, or LV metadata.

Example:

```bash
lvdisplay
```

Specific LV:

```bash
lvdisplay /dev/vg0/data
```

Typical information includes:

- LV path
- LV name
- VG name
- LV size
- Current LE count
- Allocation policy
- Read/write status
- Block device path

This command is especially helpful when you need to confirm the exact device node before formatting or mounting.

**Comparing `lvs` and `lvdisplay`**

| Command | Best for | Output style |
|---|---|---|
| `lvs` | Quick checks, scripting, compact summaries | Short, tabular |
| `lvdisplay` | Detailed inspection and troubleshooting | Verbose, descriptive |

Use `lvs` when you want a fast overview. Use `lvdisplay` when you need full metadata or are diagnosing a storage issue.

**Typical RHCSA workflow**

A common storage task follows this sequence:

1. Confirm the volume group has free space.
2. Create the logical volume.
3. Verify the LV with `lvs` or `lvdisplay`.
4. Create a filesystem.
5. Mount the filesystem.
6. Make the mount persistent with `/etc/fstab` if required.

Example:

```bash
vgs
lvcreate -L 4G -n webdata vg0
lvs
mkfs.ext4 /dev/vg0/webdata
mkdir -p /webdata
mount /dev/vg0/webdata /webdata
```

If persistence is required, add an `fstab` entry using the UUID:

```bash
blkid /dev/vg0/webdata
```

Then add a line such as:

```fstab
UUID=1234-ABCD  /webdata  ext4  defaults  0 0
```

**Practical examples of common creation patterns**

Create a fixed-size LV:

```bash
lvcreate -L 1G -n swaplv vg0
```

Create an LV using all free space:

```bash
lvcreate -l 100%FREE -n data vg0
```

Create multiple LVs from one VG:

```bash
lvcreate -L 5G -n home vg0
lvcreate -L 2G -n var vg0
lvcreate -L 1G -n tmp vg0
```

These patterns are useful when you want to divide a storage pool into purpose-specific volumes.

**Troubleshooting and verification**

If `lvcreate` fails, check the following:

- The volume group exists.
- The VG has enough free extents.
- The LV name is not already in use.
- You typed the VG name correctly.
- The requested size is valid.

Useful verification commands:

```bash
pvs
vgs
lvs
lvdisplay /dev/vg0/data
```

If the LV exists but cannot be mounted, verify:

- The filesystem was created.
- The mount point exists.
- The device path is correct.
- The LV is active.

Example checks:

```bash
lsblk
blkid
mount | grep /webdata
```

**Naming and device paths**

LVM creates predictable device paths:

- `/dev/<vgname>/<lvname>`
- `/dev/mapper/<vgname>-<lvname>`

Example:

- VG: `vg0`
- LV: `data`
- Paths:
  - `/dev/vg0/data`
  - `/dev/mapper/vg0-data`

Use the `/dev/vg0/data` form in most administrative commands because it is easy to read and commonly used in documentation.

**Exam-focused reminders**

- Create the LV before formatting it.
- Verify the LV with `lvs` after creation.
- Use `lvdisplay` when you need detailed metadata.
- Remember that `lvcreate` only allocates block storage; it does not create a filesystem.
- Use `-L` for simple size requests and `-l 100%FREE` when you want all remaining space.

**Quick reference**

```bash
# Create a 10G LV named data in vg0
lvcreate -L 10G -n data vg0

# Create an LV using all free space
lvcreate -l 100%FREE -n backup vg0

# Show all logical volumes
lvs

# Show detailed information for one LV
lvdisplay /dev/vg0/data
```

**Mental model for the exam**

Think of `lvcreate` as the allocation step, `lvs` as the quick status check, and `lvdisplay` as the detailed report. If you can create the LV, confirm it exists, and identify its device path, you have the core skill needed for RHCSA storage tasks.
