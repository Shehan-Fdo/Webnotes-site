**LVM volume groups: what these commands do**

Logical Volume Manager (LVM) lets you pool physical storage into flexible storage containers. A **volume group** (VG) sits between physical volumes (PVs) and logical volumes (LVs), and these commands help you create and inspect that layer.

- `vgcreate` creates a new volume group from one or more PVs.
- `vgs` gives a compact, script-friendly summary of volume groups.
- `vgdisplay` shows a detailed, human-readable view of a volume group.

For RHCSA tasks, you usually use these commands after preparing disks or partitions with `pvcreate`. The exam often expects you to create a VG with the correct name, verify its size, and confirm that it is active and ready for logical volume creation.

**`vgcreate` syntax and usage**

The common syntax is:

```bash
vgcreate [options] VG_NAME PV1 [PV2 ...]
```

The positional order matters:

1. Put the **volume group name** first.
2. Follow it with one or more **physical volumes**.

Examples:

```bash
vgcreate vgdata /dev/sdb1
vgcreate vgapps /dev/sdb1 /dev/sdc1
vgcreate -s 32M vgbackup /dev/sdd1 /dev/sde1
```

- Use multiple PVs in one command when you want the VG to span more than one disk or partition.
- Use `-s` to set the **physical extent size** at creation time.
- If you omit `-s`, LVM uses the default extent size, which is commonly `4 MiB` on modern RHEL systems, but you should verify it rather than assume it.

> `vgcreate` fails if the PVs are not initialized with `pvcreate`, if the VG name already exists, or if the devices are in use by another LVM structure.

**Extent size and why it matters**

A volume group is divided into fixed-size chunks called **physical extents** (PEs). Logical volumes are allocated in these units, so extent size affects flexibility and maximum addressable size.

- Smaller extents give finer allocation granularity.
- Larger extents reduce metadata overhead but make allocation less granular.
- For exam work, the default is usually fine unless the task explicitly asks for a specific size.

To verify the chosen extent size after creation, use `vgdisplay` or `vgs`:

```bash
vgdisplay vgdata
vgs -o vg_name,vg_size,vg_free,vg_extent_size,vg_extent_count,vg_free_count
```

Look for:

- `PE Size` in `vgdisplay`
- `VG Extent Size` in `vgs`

That confirms the actual extent size used by the VG.

**Creating a volume group step by step**

A typical RHCSA workflow is:

1. Identify the target device or partition.
2. Initialize it as a PV.
3. Create the VG.
4. Verify the result.
5. Create an LV from the VG.

Example:

```bash
pvcreate /dev/sdb1 /dev/sdc1
vgcreate vgdata /dev/sdb1 /dev/sdc1
vgs vgdata
vgdisplay vgdata
```

This creates a VG named `vgdata` that spans two PVs. The verification commands confirm that the VG exists, how much space it contains, and how much remains free.

**How to read `vgs` output**

`vgs` is the quick status command. It is useful in the exam because it shows the most important fields in a compact table.

Example:

```bash
vgs
```

Typical output columns include:

- `VG` — the volume group name.
- `#PV` — number of physical volumes in the VG.
- `#LV` — number of logical volumes currently created.
- `#SN` — number of snapshots, if any.
- `Attr` — VG attributes.
- `VSize` — total VG size.
- `VFree` — free space remaining.

Example interpretation:

```bash
VG     #PV #LV #SN Attr   VSize   VFree
vgdata   2   0   0 wz--n-  39.99g  39.99g
```

- `#PV 2` means the VG spans two physical volumes.
- `#LV 0` means no logical volumes exist yet.
- `VFree` equals `VSize` because nothing has been allocated.
- `Attr` tells you the VG state.

**Understanding `VG Attr` flags for troubleshooting**

The `Attr` field in `vgs` and `vgdisplay` is important when you need to determine whether a VG is usable.

Common flags you should recognize:

- `w` — writable.
- `r` — read-only.
- `z` — resizable.
- `x` — exported.
- `p` — partial.
- `c` — clustered.
- `n` — normal, not clustered.
- `a` — active.
- `s` — shared.

A typical active, normal VG often appears as something like `wz--n-` or similar, depending on system state and LVM version.

Practical exam interpretation:

- `a` indicates the VG is active and available for use.
- `p` indicates partial availability, often because one or more PVs are missing or failed.
- `x` indicates exported, which is not the normal state for local use.
- `r` indicates read-only, which can block changes.
- `w` indicates the VG can be written to.

If a VG is not behaving as expected, check the attribute string before assuming the problem is with the LV or filesystem.

**How to read `vgdisplay` output**

`vgdisplay` provides a detailed report and is useful when you need exact values.

Example:

```bash
vgdisplay vgdata
```

Important fields include:

- `VG Name` — the VG name.
- `VG Size` — total size of the VG.
- `PE Size` — size of each physical extent.
- `Total PE` — total number of extents in the VG.
- `Alloc PE / Size` — extents and size already allocated.
- `Free  PE / Size` — extents and size still available.
- `VG UUID` — unique identifier.
- `VG Status` — usually `resizable` if the VG can grow.
- `VG Access` — typically `read/write`.
- `VG Attr` — attribute flags, similar to `vgs`.

A useful exam habit is to compare `VG Size`, `Alloc PE`, and `Free PE` to confirm whether the VG has enough room for the requested LV.

**`vgs` versus `vgdisplay`**

| Command | Best use | Output style |
|---|---|---|
| `vgs` | Quick checks, scripting, exam verification | Compact table |
| `vgdisplay` | Detailed inspection and troubleshooting | Verbose report |

Use `vgs` when you want a fast answer. Use `vgdisplay` when you need to confirm extent size, allocation, or status details.

**Renaming and removing a volume group**

Even though the topic centers on creation and inspection, exam scenarios may require cleanup or correction.

Rename a VG with `vgrename`:

```bash
vgrename vgold vgnew
```

- Use this when the VG name is wrong or must match a task requirement.
- Update any references in `/etc/fstab`, mount units, or scripts if the VG name is used there.

Remove a VG with `vgremove`:

```bash
vgremove vgdata
```

- You must remove any LVs first, or use the appropriate cleanup order.
- If the VG is in use, unmount filesystems and deactivate or remove dependent LVs before removal.

A safe cleanup sequence is often:

```bash
lvremove /dev/vgdata/lvapp
vgremove vgdata
```

**End-to-end RHCSA-style scenario**

Suppose the exam asks you to create a volume group named `vgfiles` from two new partitions and verify that it is ready for later logical volume creation.

1. Confirm the devices are initialized as PVs.
2. Create the VG with both PVs in one command.
3. Verify the VG summary.
4. Inspect detailed output to confirm extent size and free space.
5. Interpret the attribute flags to ensure the VG is active and usable.

Example:

```bash
pvcreate /dev/sdb1 /dev/sdc1
vgcreate vgfiles /dev/sdb1 /dev/sdc1
vgs vgfiles
vgdisplay vgfiles
```

Expected reasoning:

- `vgcreate vgfiles /dev/sdb1 /dev/sdc1` creates one VG from two PVs.
- `vgs vgfiles` confirms the VG exists and shows total and free space.
- `vgdisplay vgfiles` confirms the `PE Size`, `Total PE`, and `Free PE`.
- If `VG Attr` shows a partial or read-only state, you should troubleshoot before creating LVs.

If the task later asks for a rename:

```bash
vgrename vgfiles vgarchive
```

If the task asks for cleanup:

```bash
vgremove vgarchive
```

**Practical verification commands**

Use these commands to confirm the VG state during the exam:

```bash
pvs
vgs
vgdisplay vgfiles
lvs
```

- `pvs` confirms the underlying physical volumes.
- `vgs` confirms the volume group summary.
- `vgdisplay` confirms detailed VG properties.
- `lvs` confirms whether any logical volumes already exist.

**Common mistakes to avoid**

- Creating the VG before running `pvcreate` on the devices.
- Reversing the `vgcreate` argument order.
- Forgetting that multiple PVs can be listed in one `vgcreate` command.
- Assuming the default extent size without verifying it.
- Ignoring `VG Attr` when the VG appears present but is not fully usable.
- Forgetting cleanup commands such as `vgrename` and `vgremove` when a task requires correction.

**Exam-focused takeaway**

For RHCSA, you should be able to create a VG from one or more PVs, verify it with both `vgs` and `vgdisplay`, interpret the attribute flags, and adjust or remove the VG when needed. The key is not just knowing the commands, but reading their output quickly and correctly under exam pressure.
