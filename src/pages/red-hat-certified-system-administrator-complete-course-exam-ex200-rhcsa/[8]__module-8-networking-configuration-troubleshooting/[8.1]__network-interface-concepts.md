## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Network interface fundamentals**

A network interface is the point where a system sends and receives traffic. On RHEL 8/9, you usually manage interfaces through **NetworkManager**, which tracks both the physical device and the connection profile that configures it. For RHCSA tasks, the key idea is that the device is the hardware or virtual NIC, while the profile is the saved configuration that NetworkManager applies to that device.

A simple mental model helps:

```text
Device (eth0, ens160, enp1s0)
  -> NetworkManager
    -> Connection profile (IP, DNS, MTU, VLAN, bond, bridge)
      -> Active state on the interface
```

- **Device**: the actual interface seen by the kernel.
- **Profile**: the persistent settings stored by NetworkManager.
- **State**: whether the profile is active, inactive, or unmanaged.

**Interface naming on RHEL**

RHEL uses **predictable network interface names** so names stay stable across reboots and hardware changes when possible. Instead of old names like `eth0`, you often see names such as `ens192`, `enp1s0`, or `eno1`.

Common naming patterns:

- `eno1`: onboard Ethernet
- `ens1`: hot-plug slot or embedded device
- `enp2s0`: PCI bus path-based name
- `wlp3s0`: wireless interface

These names come from systemd/udev naming rules and are designed to reduce ambiguity. If hardware changes, verify the name before editing a profile.

Useful checks:

```bash
ip link show
nmcli device status
ls /sys/class/net
```

> If you rename or replace hardware, do not assume the old interface name still exists. Always confirm the current name before applying changes.

**NetworkManager on RHEL 8/9**

NetworkManager is the standard network management service on modern RHEL systems. It manages interface activation, IP configuration, DNS settings, and many advanced connection types such as VLANs, bonds, and bridges.

Service control and status checks are important for troubleshooting:

```bash
systemctl status NetworkManager
systemctl restart NetworkManager
systemctl enable NetworkManager
```

If networking behaves unexpectedly, check whether the service is running and whether it has reloaded the current profiles.

**Devices, profiles, and persistence**

NetworkManager stores connection profiles persistently, usually under:

```bash
/etc/NetworkManager/system-connections/
```

These files are typically root-owned and contain the saved settings for each connection. Editing them directly is possible, but `nmcli` is safer and less error-prone for exam tasks.

Useful commands:

```bash
nmcli connection show
nmcli connection show --active
nmcli device status
nmcli device show ens192
```

- `nmcli connection show` lists saved profiles.
- `nmcli device status` shows whether a device is connected, disconnected, or unmanaged.
- `nmcli device show` reveals current device properties such as IP addresses, gateway, DNS, and MTU.

**Managed and unmanaged devices**

A device can be **unmanaged**, meaning NetworkManager ignores it. This can happen because of configuration, a conflicting service, or explicit settings.

Check and change management state:

```bash
nmcli device status
nmcli device set ens192 managed yes
nmcli device set ens192 managed no
```

Use this when a device exists but NetworkManager refuses to control it. On exam systems, unmanaged status is a common reason a connection profile appears correct but never activates.

**Connection activation and deactivation**

A profile can exist without being active. Activation applies the saved settings to the device.

```bash
nmcli connection up ens192
nmcli connection down ens192
nmcli device connect ens192
nmcli device disconnect ens192
```

- `connection up` activates a named profile.
- `device connect` asks NetworkManager to bring the device online using an available profile.
- `device disconnect` drops the active connection.

If multiple profiles exist for one device, confirm which one is active:

```bash
nmcli connection show --active
```

**IPv4 configuration concepts**

IPv4 settings are commonly configured with DHCP or static addressing.

- **DHCP**: the server assigns IP settings automatically.
- **Static**: you manually set address, gateway, and DNS.

For DHCP, `ipv4.method auto` requests an address from a DHCP server:

```bash
nmcli connection modify ens192 ipv4.method auto
nmcli connection down ens192
nmcli connection up ens192
```

To renew a lease, bounce the connection or the device. In practice, bringing the connection down and back up is the most exam-friendly approach.

Static IPv4 example:

```bash
nmcli connection modify ens192 \
  ipv4.method manual \
  ipv4.addresses 192.0.2.10/24 \
  ipv4.gateway 192.0.2.1 \
  ipv4.dns 192.0.2.53
nmcli connection up ens192
```

Key points:

- `ipv4.method manual` disables DHCP for that profile.
- `ipv4.addresses` can include one or more addresses.
- `ipv4.gateway` sets the default route.
- `ipv4.dns` sets resolver servers for that connection.

**IPv6 configuration concepts**

IPv6 can be configured automatically or manually. The most common automatic mode is **SLAAC** (Stateless Address Autoconfiguration), where the host learns an address from router advertisements.

Important distinction:

- `ipv6.method auto` enables automatic IPv6 configuration, primarily through router advertisements/SLAAC.
- DHCPv6 behavior depends on network support and additional settings; do not assume it is always used.

Examples:

```bash
nmcli connection modify ens192 ipv6.method auto
nmcli connection modify ens192 ipv6.method disabled
```

To disable IPv6 on a connection, use `ipv6.method disabled`. This is useful when an environment requires IPv4-only connectivity or when IPv6 causes routing or DNS confusion during troubleshooting.

Manual IPv6 example:

```bash
nmcli connection modify ens192 \
  ipv6.method manual \
  ipv6.addresses 2001:db8:10::10/64 \
  ipv6.gateway 2001:db8:10::1 \
  ipv6.dns 2001:db8:10::53
```

**DNS and name resolution**

NetworkManager can manage DNS settings for a connection and update resolver configuration accordingly. On many RHEL systems, `/etc/resolv.conf` is a generated or managed file, so do not treat it as the only source of truth.

Check DNS-related settings with:

```bash
nmcli device show ens192 | grep -E 'IP4.DNS|IP6.DNS|GENERAL.CONNECTION'
cat /etc/resolv.conf
```

If name resolution fails, verify:

- the active profile has the expected DNS servers
- the connection is up
- the system is using the intended resolver configuration
- the gateway and routing are correct

A common exam issue is that IP connectivity works but DNS does not. In that case, test with both an IP address and a hostname:

```bash
ping -c 3 192.0.2.1
ping -c 3 example.com
```

If the first works and the second fails, focus on DNS rather than the interface itself.

**MTU and link settings**

The **MTU** (Maximum Transmission Unit) controls the largest packet size the interface sends without fragmentation. Standard Ethernet MTU is usually 1500, while jumbo frames often use 9000 when the entire path supports it.

Set MTU on a profile with the Ethernet property:

```bash
nmcli connection modify ens192 ethernet.mtu 9000
nmcli connection up ens192
```

Use jumbo frames only when every device on the path supports the same MTU. A mismatch can cause intermittent connectivity that is hard to diagnose.

**Common interface types**

RHCSA tasks may involve more than a plain Ethernet NIC. You should understand the purpose of each type and when to use it.

| Type | Purpose | Typical use |
|---|---|---|
| Ethernet | Basic wired interface | Standard host connectivity |
| VLAN | Logical subinterface on a tagged network | Separate traffic on one physical link |
| Bond | Combine multiple NICs for redundancy or throughput | High availability or link aggregation |
| Bridge | Connect multiple interfaces at Layer 2 | Virtualization, containers, shared L2 segments |

**VLAN concepts**

A VLAN (Virtual LAN) separates traffic logically on the same physical network. The parent interface carries tagged frames, and the VLAN interface represents one VLAN ID.

Example:

```bash
nmcli connection add type vlan con-name vlan10 ifname ens192.10 dev ens192 id 10 \
  ipv4.method manual ipv4.addresses 192.0.2.20/24 ipv4.gateway 192.0.2.1
```

Use VLANs when a host must participate in multiple network segments over one physical NIC.

**Bond concepts**

A bond combines two or more physical interfaces into one logical interface. This can provide redundancy or, depending on mode and switch support, increased throughput.

Typical bond setup:

```bash
nmcli connection add type bond con-name bond0 ifname bond0 mode active-backup
nmcli connection add type ethernet con-name bond0-slave1 ifname ens192 master bond0 slave-type bond
nmcli connection add type ethernet con-name bond0-slave2 ifname ens193 master bond0 slave-type bond
nmcli connection up bond0
```

Notes:

- `active-backup` is common because it is simple and does not require special switch configuration.
- Slaves should be created as Ethernet profiles with `slave-type bond`.
- The bond profile owns the IP configuration, not the slave profiles.

**Bridge concepts**

A bridge connects interfaces at Layer 2 and is often used for virtualization or when a host must act like a switch segment. The bridge itself gets the IP address; the attached ports do not.

Typical bridge setup:

```bash
nmcli connection add type bridge con-name br0 ifname br0
nmcli connection add type ethernet con-name br0-port1 ifname ens192 master br0 slave-type bridge
nmcli connection up br0
```

Use bridges when you need guest VMs or containers to appear directly on the physical network. The bridge acts as the logical network interface, while the physical NIC becomes a bridge port.

**Practical troubleshooting workflow**

When an interface fails, work from the device outward: hardware, link, profile, IP, routing, DNS.

1. Check whether the device exists and is up.
   ```bash
   ip link show
   nmcli device status
   ```
2. Check whether NetworkManager manages it.
   ```bash
   nmcli device status
   nmcli device set ens192 managed yes
   ```
3. Check the active profile and its settings.
   ```bash
   nmcli connection show --active
   nmcli connection show ens192
   ```
4. Restart the connection or service if needed.
   ```bash
   nmcli connection down ens192
   nmcli connection up ens192
   systemctl restart NetworkManager
   ```
5. Verify addressing, routes, and DNS.
   ```bash
   ip addr show ens192
   ip route
   cat /etc/resolv.conf
   ```

A useful end-to-end failure scenario:

- Symptom: `ping 192.0.2.1` fails.
- Check: `nmcli device status` shows the device disconnected.
- Fix: bring the profile up with `nmcli connection up ens192`.
- Verify: `ip addr show` now shows the expected address.
- If DNS still fails, inspect `nmcli device show` and `/etc/resolv.conf`.

Another common scenario:

- Symptom: the interface has an IP address, but the host cannot reach the network.
- Check: `ip route` shows no default gateway.
- Fix: add `ipv4.gateway` to the profile and reactivate it.
- Verify: `ping` the gateway, then test external connectivity.

**Interface naming, persistence, and hardware changes**

Predictable names reduce confusion, but hardware replacement can still change the interface name. After a NIC swap or VM reconfiguration, verify:

- the current device name
- whether the old profile still matches the new device
- whether the profile is bound to the correct interface

Useful commands:

```bash
nmcli connection show
nmcli connection modify "Wired connection 1" connection.interface-name ens192
nmcli connection reload
```

If a profile is tied to a specific interface name, a hardware change can prevent it from activating until you update the binding.

**Quick reference**

- Use `nmcli` for most configuration tasks.
- Use `ip` to inspect live kernel state.
- Use `systemctl` to verify NetworkManager service health.
- Use `nmcli device status` to detect unmanaged or disconnected interfaces.
- Use `nmcli connection show --active` to confirm which profile is actually in use.
- Use `nmcli connection modify` to change persistent settings.
- Use `nmcli connection up/down` to apply changes.

**Exam-focused reminders**

- Prefer persistent profile changes over temporary `ip` commands unless the task explicitly asks for a temporary change.
- Confirm the active profile after every change.
- Remember that DNS, routes, and interface state are separate troubleshooting layers.
- Use the correct slave type for bonds and bridges so the configuration survives activation.
- Verify the exact interface name before editing a profile.
