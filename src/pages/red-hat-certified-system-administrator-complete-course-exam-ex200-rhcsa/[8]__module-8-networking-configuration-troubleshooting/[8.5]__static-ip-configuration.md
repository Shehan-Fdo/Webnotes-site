## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Static IP configuration overview**

A static IP address is manually assigned and does not change unless you edit the connection profile. On RHEL, you usually manage this with **NetworkManager**, which stores settings in connection profiles rather than directly on the interface.

Use static addressing when a host must keep a predictable address for services such as SSH, DNS, web servers, monitoring, or firewall rules. In RHCSA tasks, you are typically expected to configure the network so it survives reboot and remains active after changes.

- **NetworkManager** is the service that manages interfaces and connection profiles on modern RHEL.
- A **connection profile** contains the IP method, address, gateway, DNS, and autoconnect behavior.
- Static configuration is usually applied to the profile, not to the device alone.
- Verify both the active device state and the saved profile state.

**Core concepts and required values**

To configure a static IPv4 address, you usually need these values:

| Setting | Purpose | Example |
|---|---|---|
| `address` | Host IP and prefix | `192.168.1.50/24` |
| `gateway` | Default route to other networks | `192.168.1.1` |
| `dns` | Name resolution server | `8.8.8.8` |
| `method` | Address assignment mode | `manual` |

For IPv6, you may also need:

| Setting | Purpose | Example |
|---|---|---|
| `ipv6.addresses` | Static IPv6 address and prefix | `2001:db8:1::50/64` |
| `ipv6.gateway` | IPv6 default gateway | `2001:db8:1::1` |
| `ipv6.dns` | IPv6 DNS server | `2001:4860:4860::8888` |
| `ipv6.method` | IPv6 assignment mode | `manual` |

If the environment does not require IPv6, you may leave it enabled with no static address, or disable it on the profile if the task explicitly asks for that. Be careful: disabling IPv6 is not the same as configuring IPv4.

**Identify the active interface and profile**

Before changing anything, inspect the current state so you know which profile to edit.

```bash
nmcli device status
nmcli connection show
nmcli connection show "System eth0"
nmcli connection show "Wired connection 1"
```

Useful fields to inspect:

- **GENERAL.DEVICE** shows the interface name.
- **connection.id** shows the profile name.
- **ipv4.method** and **ipv6.method** show how each stack is configured.
- **ipv4.addresses**, **ipv4.gateway**, and **ipv4.dns** show current values.
- **connection.autoconnect** shows whether the profile comes up automatically.

You can also inspect a profile in a more structured way:

```bash
nmcli -f connection.id,connection.interface-name,connection.autoconnect,ipv4.method,ipv4.addresses,ipv4.gateway,ipv4.dns,ipv6.method,ipv6.addresses,ipv6.gateway connection show "System eth0"
```

**Configure static IPv4 with nmcli**

`nmcli` is the primary RHCSA tool for network configuration. The safest exam approach is to modify the existing profile, then bring the connection up or reapply it.

```bash
nmcli connection modify "System eth0" \
  ipv4.addresses 192.168.1.50/24 \
  ipv4.gateway 192.168.1.1 \
  ipv4.dns 8.8.8.8 \
  ipv4.method manual \
  connection.autoconnect yes
```

Apply the change:

```bash
nmcli connection up "System eth0"
```

If the connection is already active and you only changed a few properties, `nmcli device reapply eth0` can often apply the updated profile without a full disconnect. Use it when you want to minimize disruption.

```bash
nmcli device reapply eth0
```

> `nmcli connection down` is not required just to make a static IP change take effect, and it can interrupt connectivity unnecessarily.

**Create a new static profile with nmcli**

If no suitable profile exists, create one and bind it to the interface.

```bash
nmcli connection add \
  type ethernet \
  con-name static-eth0 \
  ifname eth0 \
  ipv4.addresses 192.168.1.50/24 \
  ipv4.gateway 192.168.1.1 \
  ipv4.dns 8.8.8.8 \
  ipv4.method manual \
  connection.autoconnect yes
```

Then activate it:

```bash
nmcli connection up static-eth0
```

This approach is useful when the existing profile is broken, misnamed, or tied to the wrong interface.

**Configure static IPv6**

Static IPv6 uses the same profile model. You can configure IPv4 and IPv6 together or separately.

```bash
nmcli connection modify "System eth0" \
  ipv6.addresses 2001:db8:1::50/64 \
  ipv6.gateway 2001:db8:1::1 \
  ipv6.dns 2001:4860:4860::8888 \
  ipv6.method manual \
  connection.autoconnect yes
```

If the task requires disabling IPv6 on the profile, set the method to `ignore`:

```bash
nmcli connection modify "System eth0" ipv6.method ignore
```

Use `ignore` only when the requirement is explicit. Otherwise, leave IPv6 available.

**Verify the configuration**

Verification is part of the task, not an optional extra. Check both the profile and the live interface.

```bash
nmcli connection show "System eth0"
ip addr show eth0
ip route
resolvectl status
```

Look for:

- The expected static address with the correct prefix.
- The default route pointing to the configured gateway.
- DNS servers matching the profile.
- `connection.autoconnect: yes` if persistence is required.
- The interface state showing `UP`.

If you need a compact view of the active address:

```bash
ip -4 addr show eth0
ip -6 addr show eth0
```

To confirm the host can reach the gateway and resolve names:

```bash
ping -c 4 192.168.1.1
ping -c 4 example.com
```

**Using nmtui and the GUI**

For RHCSA, `nmcli` is the most direct and scriptable method, but you may also use `nmtui` or the graphical network settings if the environment provides them.

`nmtui` is a text-based interface that is useful when you want a guided menu without memorizing every `nmcli` option.

```bash
nmtui
```

Typical path:

1. Select **Edit a connection**.
2. Choose the target interface profile.
3. Set IPv4 to **Manual**.
4. Enter the address, prefix, gateway, and DNS.
5. Enable autoconnect if needed.
6. Save and exit.
7. Reactivate the connection if prompted.

In the GUI, open the network settings for the interface, switch IPv4 to manual, enter the values, and save. The GUI is useful for visual confirmation, but exam tasks are usually faster with `nmcli`.

**Common verification and troubleshooting workflow**

A practical troubleshooting sequence helps when the host has the wrong address or no connectivity.

1. Check the active device and profile.
   ```bash
   nmcli device status
   nmcli connection show
   ```
2. Inspect the saved settings.
   ```bash
   nmcli connection show "System eth0"
   ```
3. Correct the profile values.
   ```bash
   nmcli connection modify "System eth0" ipv4.addresses 192.168.1.50/24 ipv4.gateway 192.168.1.1 ipv4.dns 8.8.8.8 ipv4.method manual
   ```
4. Apply the profile.
   ```bash
   nmcli connection up "System eth0"
   ```
5. Recheck the live interface and routes.
   ```bash
   ip addr show eth0
   ip route
   ```
6. Test reachability and DNS.
   ```bash
   ping -c 4 192.168.1.1
   ping -c 4 example.com
   ```

A common misconfiguration is using the wrong prefix length. For example, `192.168.1.50/24` is not the same as `192.168.1.50/16`. A wrong prefix can make the host believe local peers are remote, which breaks routing and name resolution in subtle ways.

**Practical example: fix a broken static profile**

Suppose the host should use `192.168.10.25/24`, but it was configured with the wrong gateway and no DNS.

```bash
nmcli connection show "Office LAN"
nmcli connection modify "Office LAN" \
  ipv4.addresses 192.168.10.25/24 \
  ipv4.gateway 192.168.10.1 \
  ipv4.dns "1.1.1.1 8.8.8.8" \
  ipv4.method manual \
  connection.autoconnect yes
nmcli connection up "Office LAN"
```

Then verify:

```bash
nmcli connection show "Office LAN"
ip route
resolvectl status
```

If the interface already had an active lease or stale state, `nmcli device reapply eth0` may be enough after the profile update. If not, bringing the connection up again usually refreshes the live configuration.

**Autoconnect behavior**

Autoconnect ensures the profile comes up automatically after reboot or link events. This is important for servers and for exam tasks that require persistence.

```bash
nmcli connection modify "System eth0" connection.autoconnect yes
nmcli connection show "System eth0" | grep autoconnect
```

If multiple profiles exist for the same device, autoconnect can determine which one wins at boot. Make sure the intended profile is enabled and the old one is disabled or removed if it causes conflicts.

**Legacy file-based configuration recognition**

Modern RHEL 8 and 9 use NetworkManager keyfiles under `/etc/NetworkManager/system-connections/`. Older `network-scripts` files under `/etc/sysconfig/network-scripts/` are historical and should not be your primary exam workflow.

If you encounter a keyfile on disk, it may look like this:

```ini
[connection]
id=static-eth0
type=ethernet
interface-name=eth0
autoconnect=true

[ipv4]
method=manual
addresses=192.168.1.50/24;192.168.1.1
dns=8.8.8.8;

[ipv6]
method=ignore
```

After editing a keyfile manually, reload or reapply it with NetworkManager:

```bash
nmcli connection reload
nmcli connection up static-eth0
```

If the file permissions are wrong, NetworkManager may ignore it. Keyfiles typically need restrictive permissions, and NetworkManager may warn if the file is too open.

> Do not rely on deprecated `network-scripts` as a practical RHCSA method on current RHEL releases. Treat them as recognition-only material if they appear in older systems or legacy documentation.

**Quick comparison for review**

| Tool | Best use | Notes |
|---|---|---|
| `nmcli` | Primary exam method | Fast, scriptable, precise |
| `nmtui` | Guided terminal setup | Good for interactive changes |
| GUI | Visual confirmation | Useful when a desktop is available |
| Keyfile edit | Troubleshooting or direct file inspection | Reload with NetworkManager after changes |

**Key takeaways**

- Set static IPs on the **connection profile**, not just the device.
- Use `ipv4.method manual` for static IPv4 and `ipv6.method manual` for static IPv6.
- Include address, gateway, DNS, and autoconnect when persistence matters.
- Verify with `nmcli connection show`, `ip addr`, `ip route`, and DNS checks.
- Use `nmcli connection up` or `nmcli device reapply` to apply changes; a down/up cycle is not inherently required.
- Prefer `nmcli` for RHCSA, with `nmtui` and GUI as valid alternatives when available.
