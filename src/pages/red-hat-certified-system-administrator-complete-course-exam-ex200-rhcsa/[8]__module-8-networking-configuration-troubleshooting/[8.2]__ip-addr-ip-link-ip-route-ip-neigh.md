## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

The `ip` suite from **iproute2** is the modern Linux networking toolkit you use to inspect and temporarily change interfaces, addresses, routes, and neighbors. On RHCSA, you need to read its output quickly, identify the active device, and understand which changes are only temporary versus which belong in a persistent **NetworkManager** connection profile.

These commands are especially important when you troubleshoot connectivity, verify a gateway, confirm a static address, or inspect ARP/neighbor cache behavior. In practice, you often use `ip` first to diagnose the live state, then use `nmcli` to make the configuration persistent.

**Core idea: temporary vs persistent configuration**

`ip` changes the running kernel networking state immediately, but those changes usually do not survive a reboot or a NetworkManager reconnect. For exam tasks, this distinction matters.

- **Temporary changes** with `ip`:
  - affect the live interface, route table, or neighbor table
  - disappear after reboot or when NetworkManager reapplies the connection
  - are useful for testing, troubleshooting, and emergency recovery
- **Persistent changes** with `nmcli`:
  - update the NetworkManager connection profile
  - survive reboot and reconnection
  - are the correct choice for permanent IP addresses, gateways, DNS, and static routes

A common workflow is:

1. Inspect the current state with `ip`.
2. Confirm the correct interface name with `ip link`.
3. Test a temporary address or route if needed.
4. Make the same change persistent with `nmcli`.

> If NetworkManager manages the interface, a manual `ip addr add` or `ip route add` is usually overwritten later unless you also update the connection profile.

**`ip` command family at a glance**

The `ip` command is a single entry point with subcommands for different networking objects.

- `ip addr` or `ip address`:
  - show, add, delete, or replace IP addresses on interfaces
- `ip link`:
  - show and change link-layer interface state and properties
- `ip route`:
  - inspect and modify the routing table
- `ip neigh`:
  - inspect and manage the neighbor table, which covers **ARP** (Address Resolution Protocol) for IPv4 and **Neighbor Discovery** for IPv6

**`ip link`: identify and control interfaces**

Use `ip link` to see which interfaces exist, whether they are up, and what their device names are. This is often the first step before any configuration because predictable names such as `ens160`, `enp0s3`, or `eth0` must be matched to the correct physical or virtual NIC.

Typical output includes the interface index, flags, MAC address, and state.

```bash
ip link show
```

Common uses:

- confirm the correct device before changing it
- bring an interface up or down
- inspect the MAC address and link state
- verify whether the kernel sees carrier on the device

Useful command patterns:

```bash
ip link show
ip link show dev ens160
ip link set ens160 up
ip link set ens160 down
```

Key points:

- **Predictable interface names** are common on modern RHEL systems, so do not assume `eth0`.
- `ip link` shows the device name you must use in `ip addr`, `ip route`, and `nmcli`.
- `state UP` means the interface is administratively enabled; it does not guarantee network reachability.
- `LOWER_UP` usually indicates the link has carrier.

You can also inspect the MAC address and MTU:

```bash
ip link show dev ens160
```

Example output fields to recognize:

- `link/ether`:
  - Ethernet MAC address
- `mtu`:
  - maximum transmission unit
- `qdisc`:
  - queuing discipline
- `state`:
  - administrative or operational state

**`ip addr`: inspect and manage IP addresses**

`ip addr` shows IPv4 and IPv6 addresses assigned to interfaces. It is the fastest way to confirm whether an interface has the expected address, prefix length, and scope.

```bash
ip addr show
ip addr show dev ens160
```

What to look for:

- `inet`:
  - IPv4 address and prefix, such as `192.0.2.10/24`
- `inet6`:
  - IPv6 address and prefix
- `scope global`:
  - usable beyond the local host
- `scope link`:
  - only valid on the local link
- `secondary` or multiple `inet` lines:
  - more than one address on the same interface

Common command forms:

```bash
ip addr show
ip addr show dev ens160
ip addr add 192.0.2.10/24 dev ens160
ip addr del 192.0.2.10/24 dev ens160
ip addr replace 192.0.2.10/24 dev ens160
```

How the address subcommands differ:

- `add`:
  - adds a new address
  - fails if the exact address already exists
- `del`:
  - removes a specific address
- `replace`:
  - adds the address if missing, or updates it if present
  - useful for idempotent scripts and quick corrections

Practical examples:

```bash
# Add a temporary IPv4 address
ip addr add 192.0.2.50/24 dev ens160

# Remove it later
ip addr del 192.0.2.50/24 dev ens160

# Replace the address with a corrected value
ip addr replace 192.0.2.60/24 dev ens160
```

You can also flush addresses, but use caution:

```bash
ip addr flush dev ens160
```

> `ip addr flush` removes addresses from the live interface. On a managed system, NetworkManager may reapply the profile afterward, so this is not a persistence method.

**Persistent address configuration with NetworkManager**

For RHCSA, if you need a permanent address, configure the connection profile with `nmcli`, not `ip`.

Example pattern:

```bash
nmcli con show
nmcli con mod "System ens160" ipv4.addresses 192.0.2.10/24
nmcli con mod "System ens160" ipv4.gateway 192.0.2.1
nmcli con mod "System ens160" ipv4.method manual
nmcli con up "System ens160"
```

Important distinctions:

- `ip addr add 192.0.2.10/24 dev ens160`
  - changes the running interface only
- `nmcli con mod ... ipv4.addresses 192.0.2.10/24`
  - stores the address in the connection profile
- `nmcli con up ...`
  - activates the profile and applies the persistent settings

If you need to remove a persistent address, update the profile and reactivate it:

```bash
nmcli con mod "System ens160" -ipv4.addresses 192.0.2.10/24
nmcli con up "System ens160"
```

**`ip route`: inspect and manage routing**

The routing table determines where packets go after the kernel chooses an outgoing interface. `ip route` is essential for checking the default gateway, connected networks, and static routes.

```bash
ip route show
ip route show table main
```

Typical route types you should recognize:

- **connected route**:
  - automatically created when an interface has an address
  - represents the directly attached subnet, such as `192.0.2.0/24 dev ens160`
- **host route**:
  - route to a single IP address, usually `/32` for IPv4 or `/128` for IPv6
- **default route**:
  - used when no more specific route matches
  - shown as `default via 192.0.2.1 dev ens160`

Example output patterns:

```bash
default via 192.0.2.1 dev ens160
192.0.2.0/24 dev ens160 proto kernel scope link src 192.0.2.10
```

Interpretation:

- `default via 192.0.2.1 dev ens160`
  - packets to unknown destinations go to the gateway `192.0.2.1`
- `proto kernel`
  - route was created automatically by the kernel
- `scope link`
  - destination is directly reachable on the local network
- `src 192.0.2.10`
  - preferred source address for that route

Common command forms:

```bash
ip route show
ip route add 198.51.100.0/24 via 192.0.2.1 dev ens160
ip route del 198.51.100.0/24 via 192.0.2.1 dev ens160
ip route replace 198.51.100.0/24 via 192.0.2.1 dev ens160
ip route add default via 192.0.2.1 dev ens160
ip route del default
```

Practical examples:

```bash
# Add a temporary static route
ip route add 203.0.113.0/24 via 192.0.2.1 dev ens160

# Replace it if the gateway changes
ip route replace 203.0.113.0/24 via 192.0.2.254 dev ens160

# Remove the route
ip route del 203.0.113.0/24 via 192.0.2.254 dev ens160
```

**Route selection and troubleshooting**

When traffic fails, do not guess. Use `ip route get` to see which route the kernel would choose for a destination.

```bash
ip route get 8.8.8.8
ip route get 203.0.113.25
```

This is useful because it shows:

- the selected outgoing interface
- the next hop gateway
- the source address the kernel will use
- whether a more specific route overrides the default route

Example interpretation:

```bash
ip route get 8.8.8.8
```

Possible output:

```bash
8.8.8.8 via 192.0.2.1 dev ens160 src 192.0.2.10 uid 0
```

This means:

- traffic to `8.8.8.8` leaves through `ens160`
- the next hop is `192.0.2.1`
- the source address is `192.0.2.10`

Persistent routing with NetworkManager:

```bash
nmcli con mod "System ens160" ipv4.routes "203.0.113.0/24 192.0.2.1"
nmcli con mod "System ens160" ipv4.gateway 192.0.2.1
nmcli con mod "System ens160" ipv4.method manual
nmcli con up "System ens160"
```

For a default gateway, set the gateway in the profile rather than using a temporary `ip route add default`:

```bash
nmcli con mod "System ens160" ipv4.gateway 192.0.2.1
nmcli con up "System ens160"
```

If you need to remove a persistent route, delete it from the profile:

```bash
nmcli con mod "System ens160" -ipv4.routes "203.0.113.0/24 192.0.2.1"
nmcli con up "System ens160"
```

**`ip neigh`: inspect ARP and IPv6 neighbor discovery**

The neighbor table maps Layer 3 addresses to Layer 2 hardware addresses. On IPv4, this is primarily **ARP**. On IPv6, it uses **Neighbor Discovery Protocol**.

```bash
ip neigh show
ip neigh show dev ens160
```

Why it matters:

- it helps you confirm whether the host has learned the MAC address for a gateway or peer
- it helps diagnose local-link connectivity problems
- it can reveal stale or failed neighbor entries

Typical states include:

- `REACHABLE`
- `STALE`
- `DELAY`
- `PROBE`
- `FAILED`
- `INCOMPLETE`

Example output:

```bash
192.0.2.1 dev ens160 lladdr 00:11:22:33:44:55 REACHABLE
```

Interpretation:

- `192.0.2.1` is the neighbor IP
- `lladdr` is the link-layer address, usually the MAC
- `REACHABLE` means the entry is currently valid

Common command forms:

```bash
ip neigh show
ip neigh flush dev ens160
ip neigh flush all
```

Use `flush` carefully:

- it clears cached neighbor entries
- it can force the system to relearn MAC addresses
- it is useful when troubleshooting stale ARP/ND information

**Neighbor troubleshooting workflow**

If you can ping the local gateway but not a remote host, check the neighbor table and route selection together.

1. Verify the interface is up.
2. Verify the address is present.
3. Verify the default route exists.
4. Check the neighbor entry for the gateway.
5. Use `ip route get` to confirm the chosen path.

Example:

```bash
ip link show dev ens160
ip addr show dev ens160
ip route show
ip neigh show dev ens160
ip route get 8.8.8.8
```

If the gateway entry is stuck in `FAILED` or `INCOMPLETE`, the issue may be:

- wrong VLAN or switch port
- incorrect subnet mask
- gateway unreachable on the local segment
- duplicate IP or ARP conflict
- interface down or disconnected cable

**Command reference summary**

| Command | Purpose | Common use |
|---|---|---|
| `ip link show` | Display interfaces and link state | Confirm the correct NIC name and whether it is up |
| `ip link set dev <iface> up` | Enable an interface | Bring a NIC online |
| `ip link set dev <iface> down` | Disable an interface | Temporarily stop traffic |
| `ip addr show` | Display IP addresses | Verify IPv4/IPv6 assignment |
| `ip addr add <addr>/<prefix> dev <iface>` | Add a temporary address | Quick test or emergency fix |
| `ip addr del <addr>/<prefix> dev <iface>` | Remove a temporary address | Undo a test change |
| `ip addr replace <addr>/<prefix> dev <iface>` | Add or update an address | Idempotent change |
| `ip route show` | Display routing table | Check default gateway and static routes |
| `ip route add <prefix> via <gw> dev <iface>` | Add a temporary route | Test reachability to a network |
| `ip route del <prefix> via <gw> dev <iface>` | Remove a route | Undo a test route |
| `ip route replace <prefix> via <gw> dev <iface>` | Add or update a route | Safer scripted change |
| `ip route get <dest>` | Show route selection for a destination | Troubleshoot path choice |
| `ip neigh show` | Display neighbor cache | Check ARP/ND resolution |
| `ip neigh flush dev <iface>` | Clear neighbor entries | Force relearning |

**RHCSA-style tasks and examples**

Add a temporary address for testing:

```bash
ip addr add 192.0.2.20/24 dev ens160
ip addr show dev ens160
```

Remove the temporary address:

```bash
ip addr del 192.0.2.20/24 dev ens160
```

Bring an interface up after confirming the correct device name:

```bash
ip link show
ip link set ens160 up
```

Add a temporary default gateway:

```bash
ip route add default via 192.0.2.1 dev ens160
ip route show
```

Remove the default gateway:

```bash
ip route del default via 192.0.2.1 dev ens160
```

Add a temporary static route to a remote subnet:

```bash
ip route add 10.10.20.0/24 via 192.0.2.1 dev ens160
ip route get 10.10.20.15
```

Flush stale neighbor entries on the interface:

```bash
ip neigh flush dev ens160
ip neigh show dev ens160
```

Make the same configuration persistent with NetworkManager:

```bash
nmcli con mod "System ens160" ipv4.addresses 192.0.2.10/24
nmcli con mod "System ens160" ipv4.gateway 192.0.2.1
nmcli con mod "System ens160" ipv4.routes "10.10.20.0/24 192.0.2.1"
nmcli con mod "System ens160" ipv4.method manual
nmcli con up "System ens160"
```

**Practical troubleshooting checklist**

When a host cannot reach the network, use a layered approach.

- **Layer 1/2**:
  - `ip link show dev <iface>` to confirm the device is present and up
  - verify carrier and correct interface naming
- **Layer 3 address**:
  - `ip addr show dev <iface>` to confirm the IP and prefix
  - check for the correct subnet mask
- **Routing**:
  - `ip route show` to confirm the default route and static routes
  - `ip route get <destination>` to see the selected path
- **Neighbor resolution**:
  - `ip neigh show dev <iface>` to confirm the gateway MAC is known
- **Persistence**:
  - if the live fix works, update the NetworkManager profile with `nmcli`

**Common exam pitfalls**

- Using `ip` to make a change and forgetting that it is temporary
- Configuring the wrong interface because the device name was not verified first
- Adding a route without the correct gateway or device
- Confusing a connected route with a manually added static route
- Checking only `ping` output instead of verifying `ip route get`
- Assuming `ip neigh` is only for IPv4; it also reflects IPv6 neighbor discovery

**Mental model to remember**

- `ip link` answers: which interface, and is it up?
- `ip addr` answers: what IP addresses are assigned?
- `ip route` answers: where does traffic go?
- `ip neigh` answers: what MAC address corresponds to the next hop or peer?

Use `ip` to understand the live system, then use `nmcli` to make the configuration persistent.
