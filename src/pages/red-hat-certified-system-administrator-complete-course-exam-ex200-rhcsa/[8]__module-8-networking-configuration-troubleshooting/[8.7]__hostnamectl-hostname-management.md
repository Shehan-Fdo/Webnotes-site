## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Hostname fundamentals**

A hostname identifies a Linux system on a network and in local administration tools. On modern Red Hat systems, `hostnamectl` is the preferred interface because it integrates with `systemd` and writes persistent settings correctly.

There are three hostname forms you should understand:

- **Static hostname**: the persistent name stored in `/etc/hostname`.
- **Transient hostname**: the runtime name currently active in the kernel; it can be set by DHCP or NetworkManager.
- **Pretty hostname**: a human-friendly label that may include spaces and special characters, mainly for display.

The command `hostname` usually shows the current runtime hostname. `hostnamectl` shows and manages all hostname types, which makes it better for troubleshooting and exam tasks.

| Type | Purpose | Persistence | Typical source |
|---|---|---:|---|
| Static | Canonical system name | Yes | `/etc/hostname` via `hostnamectl set-hostname` |
| Transient | Current runtime name | No | Kernel, DHCP, NetworkManager |
| Pretty | Display label | Yes | `hostnamectl set-hostname --pretty` |

**How `hostnamectl` works**

`hostnamectl` is the systemd tool for querying and changing host identity. When you set a hostname with `hostnamectl set-hostname`, it updates the static hostname in `/etc/hostname` and also applies the runtime hostname immediately.

That relationship matters:

- `hostnamectl` changes the persistent static hostname and the live runtime hostname together.
- `hostname` shows the current runtime value, so it may differ from the static value if something changed it later.
- `hostnamectl --static` shows the persistent name from `/etc/hostname`.
- `hostnamectl --transient` shows the current kernel/runtime hostname.
- `hostnamectl --pretty` shows the display name, if one is configured.

> If NetworkManager, DHCP, or cloud-init overwrites the transient hostname, the system may still have the correct static hostname in `/etc/hostname`, but `hostname` can temporarily show a different value.

**Core commands**

Use `hostnamectl` for most hostname administration tasks.

```bash
hostnamectl
hostnamectl status
hostnamectl set-hostname server1.example.com
hostnamectl set-hostname --static server1.example.com
hostnamectl set-hostname --pretty "RHCSA Lab Server"
hostnamectl set-hostname --transient tempname
```

Useful verification commands:

```bash
hostname
hostname -f
hostnamectl --static
hostnamectl --transient
hostnamectl --pretty
cat /etc/hostname
```

**Setting a hostname correctly**

For RHCSA-style administration, you usually want a persistent static hostname and correct name resolution. A typical change sequence is:

1. Set the hostname with `hostnamectl`.
2. Verify `/etc/hostname` contains the new static name.
3. Update `/etc/hosts` so the short name and FQDN resolve properly.
4. Confirm the runtime hostname and name resolution with validation commands.

Example:

```bash
sudo hostnamectl set-hostname server1.example.com
cat /etc/hostname
hostnamectl
hostname
hostname -f
```

Expected behavior:

- `cat /etc/hostname` shows `server1.example.com`.
- `hostnamectl` reports the static hostname as `server1.example.com`.
- `hostname` usually shows `server1.example.com`.
- `hostname -f` returns the FQDN only if name resolution is correct.

**Why `/etc/hosts` matters**

Hostname resolution is not only about the local hostname setting. Many tools use name lookup to map the system name to an IP address. If `/etc/hosts` is wrong or incomplete, `hostname -f` may return only the short name, or commands may resolve the wrong address.

A common RHCSA-friendly pattern is to map both the loopback and the system IP correctly.

Example `/etc/hosts`:

```ini
127.0.0.1   localhost localhost.localdomain
192.0.2.10  server1.example.com server1
```

This ensures:

- `server1.example.com` resolves to the correct address.
- `server1` resolves to the same host.
- `hostname -f` can return the FQDN when the resolver sees the full mapping.

If the system IP changes, update `/etc/hosts` to match the new address. For example, if the host moves from `192.0.2.10` to `192.0.2.20`, change the entry accordingly:

```ini
192.0.2.20  server1.example.com server1
```

If you keep the old IP in `/etc/hosts`, tools may resolve the hostname to the wrong address, which can break local services, SSH trust, or lab validation.

**Transient hostname and network services**

The transient hostname is the live hostname in the kernel. It can be changed by network services, especially when DHCP provides a hostname option or when NetworkManager applies connection settings.

This means:

- A static hostname in `/etc/hostname` is persistent.
- A transient hostname may be overwritten after reboot or network reconnect.
- In lab environments, you should verify both the static and transient values if the hostname appears to “change back.”

NetworkManager can also manage hostname behavior through connection profiles. In some environments, DHCP or connection settings may supply a hostname that overrides the transient value. If you need to control this in a network context, inspect the active connection and hostname-related settings.

Useful checks:

```bash
nmcli general hostname
nmcli connection show
nmcli device show
```

You can set the system hostname through NetworkManager as well:

```bash
sudo nmcli general hostname server1.example.com
```

This updates the system hostname through the network management stack and is useful when you are working in a network-admin context. Still, for exam consistency, `hostnamectl` is the primary tool you should know.

If DHCP keeps replacing the transient hostname, ensure the static hostname is correct and review the active connection profile. In some cases, you may need to adjust the connection’s hostname behavior or disable hostname updates from DHCP, depending on the environment.

**Validation and troubleshooting**

A good validation sequence checks three layers:

- The configured static hostname.
- The current runtime hostname.
- The name resolution path.

Example:

```bash
hostnamectl --static
hostnamectl --transient
hostnamectl --pretty
hostname
hostname -f
getent hosts server1.example.com
getent hosts server1
cat /etc/hostname
cat /etc/hosts
```

Common failure scenarios:

- `hostname -f` returns only `server1`
  - `/etc/hosts` likely lacks the FQDN or has the short name listed before the FQDN in a way that confuses resolution.
  - Fix the host entry so both names map to the correct IP.

- `hostname` shows an unexpected name after reboot
  - A network service or DHCP may have overwritten the transient hostname.
  - Recheck `hostnamectl --static` and the active NetworkManager connection.

- `hostnamectl` shows the correct static hostname, but applications still resolve the old address
  - `/etc/hosts` may still point to the old IP.
  - Update the mapping and re-test with `getent hosts`.

- `hostnamectl` changes the name, but `hostname -f` still fails
  - The hostname is set, but DNS or `/etc/hosts` does not provide a valid forward lookup.
  - Add a correct local mapping or ensure DNS is configured.

**Exam-style end-to-end scenario**

You are asked to rename a server to `web01.example.com`, ensure the short name `web01` resolves locally, and verify the change.

1. Set the static hostname.
2. Confirm the runtime hostname updates.
3. Edit `/etc/hosts` to map the correct IP.
4. Verify short and full name resolution.

Example session:

```bash
sudo hostnamectl set-hostname web01.example.com
hostnamectl --static
hostnamectl --transient
cat /etc/hostname
```

Expected output pattern:

```text
web01.example.com
web01.example.com
web01.example.com
```

Then update `/etc/hosts`:

```ini
127.0.0.1   localhost localhost.localdomain
192.0.2.50  web01.example.com web01
```

Verify:

```bash
hostname
hostname -f
getent hosts web01
getent hosts web01.example.com
```

Expected result:

- `hostname` returns `web01.example.com`.
- `hostname -f` returns `web01.example.com`.
- `getent hosts web01` and `getent hosts web01.example.com` both resolve to `192.0.2.50`.

**Practical troubleshooting notes**

If you need to inspect the hostname configuration file directly, remember that the persistent static hostname lives in `/etc/hostname`. `hostnamectl` writes this file for you, so manual editing is usually unnecessary unless you are recovering from a broken configuration.

Use manual editing only when needed:

```bash
sudo vi /etc/hostname
```

After manual changes, refresh or verify the runtime hostname:

```bash
sudo hostnamectl set-hostname web01.example.com
```

This is safer than relying on ad hoc commands because it keeps the static and runtime values aligned.

**Quick reference**

- Use `hostnamectl` to set and inspect hostnames.
- Use `hostnamectl --static` to confirm persistence.
- Use `hostnamectl --transient` to confirm the live kernel value.
- Use `hostnamectl --pretty` for display-only labels.
- Use `hostname -f` to test fully qualified name resolution.
- Use `/etc/hosts` to ensure the short name and FQDN resolve correctly.
- Use `nmcli general hostname` when hostname management is tied to NetworkManager workflows.

**Key takeaway**

For RHCSA tasks, the safest approach is to set the hostname with `hostnamectl`, verify `/etc/hostname`, and then fix `/etc/hosts` so both the short name and FQDN resolve to the correct IP. If the hostname appears to revert, check whether NetworkManager or DHCP is overriding the transient hostname.
