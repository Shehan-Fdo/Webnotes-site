## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

`nmcli` is the command-line client for **NetworkManager**, the service that manages network devices and connection profiles on modern RHEL systems. For RHCSA tasks, you use `nmcli` to inspect devices, create and modify persistent profiles, activate or deactivate connections, and troubleshoot why a link is not coming up.

A key exam concept is the difference between a **device** and a **connection profile**:

- A **device** is the physical or virtual interface, such as `ens160`, `eth0`, or `bond0`.
- A **connection profile** is the saved configuration, such as IP addressing, DNS, gateway, VLAN ID, or bond settings.
- A profile can exist without being active.
- A device can have multiple profiles available, but only one active profile at a time per device in typical setups.

**Core mental model**

NetworkManager decides which profile to activate based on the profile settings, device availability, interface name binding, and matching rules. That means you should not assume the profile name alone is enough in every case. If a profile is not tied to a specific interface, you may need to specify the device explicitly when activating it.

**Inspecting the current state**

Start by checking both the overall NetworkManager state and the device/profile inventory.

```bash
nmcli general status
nmcli device status
nmcli connection show
nmcli connection show --active
```

Useful output fields include:

- **GENERAL.STATE** from `nmcli general status`
- **DEVICE**, **TYPE**, **STATE**, and **CONNECTION** from `nmcli device status`
- **UUID**, **TYPE**, and **DEVICE** from `nmcli connection show`
- Active profile details from `nmcli connection show --active`

To inspect one profile in detail:

```bash
nmcli connection show "office-net"
nmcli connection show uuid 12345678-1234-1234-1234-123456789abc
```

If you need the live state of a device, compare the profile with the active device:

```bash
nmcli device show ens160
nmcli connection show --active
```

**Listing and understanding connection profiles**

`nmcli connection show` lists saved profiles. This is the persistent configuration stored by NetworkManager, usually under `/etc/NetworkManager/system-connections/` on RHEL.

Common profile fields include:

- `connection.id` — human-readable profile name
- `connection.uuid` — unique identifier
- `connection.type` — `ethernet`, `wifi`, `bridge`, `bond`, `vlan`, and others
- `connection.interface-name` — optional binding to a specific interface
- `ipv4.method` and `ipv6.method` — addressing mode
- `ipv4.addresses`, `ipv4.gateway`, `ipv4.dns`, `ipv4.dns-search`
- `ipv4.route-metric` — route preference
- `autoconnect` — whether the profile should activate automatically

To list available connection types supported by your installed NetworkManager version, use help and tab completion:

```bash
nmcli connection add help
nmcli connection add type help
```

Depending on version, the exact help output varies, but on RHCSA systems you should be familiar with common types such as:

- `ethernet`
- `bridge`
- `bond`
- `vlan`
- `team` on some systems, though bridge/bond are more common exam targets

**Creating a new connection profile**

You can create a profile without immediately activating it. This is useful when you want to prepare configuration first and bring it up later on the correct device.

Example: create an Ethernet profile with static IPv4 settings but do not bind it yet.

```bash
nmcli connection add \
  type ethernet \
  con-name office-net \
  ifname "" \
  ipv4.method manual \
  ipv4.addresses 192.0.2.10/24 \
  ipv4.gateway 192.0.2.1 \
  ipv4.dns 192.0.2.53,192.0.2.54 \
  ipv4.dns-search example.com \
  autoconnect yes
```

Notes:

- `con-name` sets the profile name.
- `ifname ""` creates a generic profile that is not tied to a specific interface name.
- `ipv4.method manual` enables static IPv4 configuration.
- `ipv4.dns` accepts a list; comma-separated values are a reliable form.
- `ipv4.dns-search` sets search domains used for name resolution.
- `autoconnect yes` allows NetworkManager to bring the profile up automatically when a matching device appears.

If you want to create a profile and bind it to a specific interface, set `ifname`:

```bash
nmcli connection add \
  type ethernet \
  con-name server1-eth0 \
  ifname ens160 \
  ipv4.method auto
```

For a DHCP-based profile, `ipv4.method auto` requests an address from a DHCP server.

**Activating a profile**

To activate a profile, use `nmcli connection up`. In practice, activation may depend on whether the profile is already bound to a device or whether NetworkManager can match it to one.

```bash
nmcli connection up office-net
```

If the profile is generic or multiple devices could match, specify the interface:

```bash
nmcli connection up office-net ifname ens160
```

You can also activate by UUID:

```bash
nmcli connection up 12345678-1234-1234-1234-123456789abc
```

If a profile is already associated with a device, `nmcli connection up` usually activates it there. If not, provide the device name to avoid ambiguity.

To bring a device up using an eligible profile, you can also use:

```bash
nmcli device connect ens160
```

This asks NetworkManager to connect the device and activate a suitable profile if one exists. It does not fix an unmanaged device by itself.

**Deactivating and deleting profiles**

Deactivation and deletion are different operations:

- **Deactivate** stops the active connection on a device.
- **Delete** removes the saved profile from NetworkManager.

Deactivate a profile:

```bash
nmcli connection down office-net
```

Delete a saved profile:

```bash
nmcli connection delete office-net
```

Delete by UUID if needed:

```bash
nmcli connection delete uuid 12345678-1234-1234-1234-123456789abc
```

Use deletion when a profile is obsolete, misconfigured, or should not remain on the system. Use deactivation when you only need to stop the current session.

**Modifying connection properties**

`nmcli connection modify` is the main tool for persistent changes. It updates the profile, not just the live session.

Examples:

```bash
nmcli connection modify office-net ipv4.method manual
nmcli connection modify office-net ipv4.addresses 192.0.2.10/24
nmcli connection modify office-net ipv4.gateway 192.0.2.1
nmcli connection modify office-net ipv4.dns 192.0.2.53,192.0.2.54
nmcli connection modify office-net ipv4.dns-search example.com
nmcli connection modify office-net ipv4.route-metric 100
```

Common properties to know for RHCSA-style tasks:

- `ipv4.method` — `auto`, `manual`, `disabled`, `shared`
- `ipv4.addresses` — one or more addresses with prefix length
- `ipv4.gateway` — default gateway
- `ipv4.dns` — DNS servers
- `ipv4.dns-search` — search domains
- `ipv4.route-metric` — lower values are preferred
- `connection.autoconnect` — `yes` or `no`
- `connection.interface-name` — binds or unbinds a profile from a device

To change the profile name:

```bash
nmcli connection modify office-net connection.id office-prod
```

To change the autoconnect behavior:

```bash
nmcli connection modify office-prod connection.autoconnect yes
```

To clear a property, use the appropriate empty value syntax for the property and shell quoting. For example, to remove an interface binding, you may need:

```bash
nmcli connection modify office-prod connection.interface-name ""
```

Behavior can vary slightly by version and shell quoting, so verify the result with `nmcli connection show`.

> Be careful when editing live network settings remotely. A bad change can disconnect your session. If possible, keep a second access path open or use console access.

**DNS, routes, and search domains**

RHCSA tasks often require more than just an IP address. You should know how to set name resolution and route preference.

Set DNS servers:

```bash
nmcli connection modify office-net ipv4.dns 192.0.2.53,192.0.2.54
```

Set DNS search domains:

```bash
nmcli connection modify office-net ipv4.dns-search example.com,lab.example.com
```

Set a route metric:

```bash
nmcli connection modify office-net ipv4.route-metric 50
```

Why route metric matters:

- Lower metrics are preferred when multiple routes exist.
- It helps control which interface becomes the primary path.
- It is common in multi-homed systems or lab environments.

If you need a static route, NetworkManager supports route properties, though exact syntax can vary by version and route complexity. For exam purposes, focus on the common properties above unless a task explicitly asks for static routes.

**Creating common profile types**

RHCSA environments often use more than Ethernet. You should recognize how to create and inspect bridge, bond, and VLAN profiles.

Bridge example:

```bash
nmcli connection add type bridge con-name br0 ifname br0 ipv4.method auto
nmcli connection add type ethernet con-name br0-port1 ifname ens160 master br0
```

Bond example:

```bash
nmcli connection add type bond con-name bond0 ifname bond0 mode active-backup
nmcli connection add type ethernet con-name bond0-slave1 ifname ens160 master bond0
nmcli connection add type ethernet con-name bond0-slave2 ifname ens192 master bond0
```

VLAN example:

```bash
nmcli connection add type vlan con-name vlan10 ifname ens160.10 dev ens160 id 10 ipv4.method auto
```

Practical notes:

- A **bridge** is used to connect multiple interfaces at Layer 2.
- A **bond** combines interfaces for redundancy or throughput.
- A **VLAN** tags traffic with an 802.1Q VLAN ID.
- The exact slave/master syntax can vary slightly by NetworkManager version, so verify with `nmcli connection show`.

**Editing profiles interactively**

`nmcli connection edit` opens an interactive editor for a profile. It is useful when you want to inspect or adjust settings without memorizing every property name.

```bash
nmcli connection edit office-net
```

Inside the editor, you can inspect and modify properties, then save changes according to the editor prompts and version-specific commands.

Because interactive behavior varies across NetworkManager releases, prefer `nmcli connection modify` for exam reliability unless the task specifically asks for interactive editing.

**Reloading profiles after external file edits**

NetworkManager stores persistent profiles in `/etc/NetworkManager/system-connections/`. Direct file editing is generally discouraged, but you may encounter it in troubleshooting or legacy workflows.

Important points:

- You need **root** privileges to read or modify these files safely.
- Files in this directory are typically owned by `root` and protected with restrictive permissions.
- NetworkManager expects secure permissions; overly permissive files may be ignored.

Typical permissions look like this:

```bash
ls -l /etc/NetworkManager/system-connections/
```

If you edit a file externally, reload NetworkManager’s connection data:

```bash
nmcli connection reload
```

Then verify the profile and, if needed, reapply or reactivate it:

```bash
nmcli connection show
nmcli connection up office-net
```

On some systems, `nmcli connection load` or a disconnect/reconnect cycle may be needed depending on what changed. For exam work, remember the general sequence: edit securely, reload, verify, then activate or reapply.

**Troubleshooting connection problems**

When a connection does not come up, check the problem from the top down:

1. Confirm NetworkManager is running.
2. Confirm the device exists and is recognized.
3. Confirm a profile is available and correctly configured.
4. Confirm the profile is active on the intended device.
5. Check IP, gateway, DNS, and route settings.

Useful commands:

```bash
systemctl status NetworkManager
nmcli general status
nmcli device status
nmcli connection show
nmcli connection show --active
nmcli device show ens160
```

If a profile fails to activate, inspect the detailed error:

```bash
nmcli connection up office-net
journalctl -u NetworkManager -b
```

Common causes include:

- Wrong interface name
- Duplicate or conflicting IP configuration
- Missing gateway or DNS settings
- A profile bound to a different device
- A device that is down, disconnected, or unmanaged
- A typo in the profile name or UUID

To verify the live state of a specific profile:

```bash
nmcli connection show office-net
nmcli connection show office-net --active
```

To verify the live state of a device:

```bash
nmcli device show ens160
```

**Unmanaged devices**

An unmanaged device is not controlled by NetworkManager. `nmcli device connect` does not automatically fix the root cause if the device is marked unmanaged by configuration.

Common recovery checks:

- Confirm the interface is not blocked by NetworkManager configuration.
- Check `/etc/NetworkManager/NetworkManager.conf` and drop-in files under `/etc/NetworkManager/conf.d/`.
- Look for `unmanaged-devices` settings.
- Verify that the interface is not intentionally excluded by a rule or a plugin.
- Restart NetworkManager only after correcting the configuration.

Example checks:

```bash
grep -R "unmanaged-devices" /etc/NetworkManager /usr/lib/NetworkManager 2>/dev/null
nmcli device status
```

If you change NetworkManager configuration files, reload or restart the service as appropriate:

```bash
nmcli general reload
systemctl restart NetworkManager
```

Use restart carefully on remote systems.

**Practical workflow for RHCSA tasks**

A reliable exam workflow is:

1. Identify the target device and current state.
2. Inspect the existing profile or create a new one.
3. Modify the required properties.
4. Activate the profile on the correct device.
5. Verify the live configuration.

Example workflow:

```bash
nmcli device status
nmcli connection show
nmcli connection add type ethernet con-name lab-net ifname ens160 ipv4.method auto
nmcli connection modify lab-net ipv4.dns 192.0.2.53,192.0.2.54 ipv4.dns-search lab.example.com
nmcli connection up lab-net ifname ens160
nmcli connection show --active
nmcli device show ens160
```

For a static configuration:

```bash
nmcli connection modify lab-net ipv4.method manual
nmcli connection modify lab-net ipv4.addresses 192.0.2.20/24
nmcli connection modify lab-net ipv4.gateway 192.0.2.1
nmcli connection modify lab-net ipv4.route-metric 100
nmcli connection up lab-net ifname ens160
```

**Verification checklist**

After changes, confirm all of the following:

- The profile exists in `nmcli connection show`
- The correct device is listed in `nmcli device status`
- The profile is active in `nmcli connection show --active`
- The device has the expected IP address in `nmcli device show <device>`
- DNS and gateway values match the task requirements
- The connection survives a down/up cycle if persistence matters

**Quick reference commands**

```bash
nmcli general status
nmcli device status
nmcli connection show
nmcli connection show --active
nmcli connection show <name-or-uuid>
nmcli connection add type ethernet con-name <name> ifname <iface> ipv4.method auto
nmcli connection modify <name> ipv4.addresses 192.0.2.10/24
nmcli connection modify <name> ipv4.gateway 192.0.2.1
nmcli connection modify <name> ipv4.dns 192.0.2.53,192.0.2.54
nmcli connection modify <name> ipv4.dns-search example.com
nmcli connection up <name> ifname <iface>
nmcli connection down <name>
nmcli connection delete <name>
nmcli connection reload
nmcli device show <iface>
```

**Exam tips**

- Use `nmcli` for persistent changes instead of editing files unless the task explicitly requires file editing.
- Verify both the profile and the device state after every change.
- Remember that activation can depend on interface matching, not just the profile name.
- Use `nmcli connection delete` when the task asks you to remove a profile.
- Prefer `nmcli connection modify` for predictable results across RHCSA environments.
