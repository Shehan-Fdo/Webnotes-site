## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

`nmtui` is the text-based user interface for **NetworkManager**, the service that manages network connections on Red Hat Enterprise Linux (RHEL). You use it when you want interactive, menu-driven configuration in a terminal without editing files by hand.

It is especially useful on servers where:
- You have console access but no graphical desktop.
- You need to configure networking quickly during installation, recovery, or exam tasks.
- You want to avoid syntax mistakes in manual configuration files.

`nmtui` does not replace command-line tools such as `nmcli`; it complements them. For RHCSA-level work, you should understand both, but `nmtui` is often the fastest way to create or modify a connection under time pressure.

**What nmtui manages**

`nmtui` edits NetworkManager connection profiles, not the kernel directly. A connection profile stores settings such as:
- Interface name or MAC address binding
- IPv4 and IPv6 addressing
- Gateway and DNS servers
- Autoconnect behavior
- Connection name and type

When you activate a profile, NetworkManager applies those settings to the interface.

**Prerequisites and service state**

Before using `nmtui`, confirm that NetworkManager is running. On modern RHEL systems, it usually is.

```bash
systemctl status NetworkManager
```

If needed, enable and start it:

```bash
sudo systemctl enable --now NetworkManager
```

Useful checks:
- `nmcli general status` shows overall NetworkManager state.
- `nmcli device status` shows interfaces and whether they are connected.
- `ip addr` confirms the resulting IP configuration after activation.

> If NetworkManager is not managing the interface, `nmtui` changes may not take effect as expected. Always verify device state after saving.

**Launching nmtui**

Start the interface from a terminal or console:

```bash
nmtui
```

You typically see a menu with options such as:
- Edit a connection
- Activate a connection
- Set system hostname
- Quit

Navigation is keyboard-based:
- <kbd>Tab</kbd> moves between controls.
- Arrow keys move through lists.
- <kbd>Enter</kbd> selects an item.
- <kbd>Esc</kbd> backs out of a dialog.
- <kbd>Space</kbd> toggles checkboxes.

**Common workflow**

A typical interactive configuration flow is:
1. Open `nmtui`.
2. Choose **Edit a connection**.
3. Select the target interface or existing profile.
4. Set IPv4 and/or IPv6 values.
5. Save the profile.
6. Activate the connection.
7. Verify with `ip addr`, `ip route`, and `nmcli`.

**Creating or editing a connection**

In **Edit a connection**, you can create a new profile or modify an existing one. This is where you define the network behavior for an interface.

Common fields include:
- **Profile name**: Human-readable connection name.
- **Device**: The interface to bind to, such as `ens160` or `enp1s0`.
- **IPv4 configuration**: Automatic (DHCP) or manual.
- **IPv6 configuration**: Automatic, manual, ignore, or disabled depending on version and policy.
- **DNS servers**: Resolver addresses, often separated by commas or spaces depending on the dialog.
- **Search domains**: Optional domain suffixes.
- **Routes**: Static route entries if needed.
- **Autoconnect**: Whether the profile comes up automatically at boot or when the device appears.

**IPv4 configuration modes**

The most common IPv4 modes are:
- **Automatic (DHCP)**: The system requests an address, gateway, and DNS from a DHCP server.
- **Manual**: You specify the address, prefix length, gateway, and DNS.
- **Disabled/Ignore**: The interface does not use IPv4.

For manual configuration, you usually provide:
- IP address, such as `192.168.1.50`
- Prefix length, such as `24`
- Gateway, such as `192.168.1.1`
- DNS servers, such as `8.8.8.8` and `1.1.1.1`

Example topology:
- Host: `server1`
- Interface: `ens160`
- Address: `192.168.10.20/24`
- Gateway: `192.168.10.1`
- DNS: `192.168.10.10`

**IPv6 configuration modes**

IPv6 may be:
- **Automatic**: Uses router advertisements and/or DHCPv6 depending on network design.
- **Manual**: You assign a static IPv6 address and gateway.
- **Ignore/Disabled**: IPv6 is not used on that connection.

Example static IPv6 values:
- Address: `2001:db8:10::20/64`
- Gateway: `2001:db8:10::1`
- DNS: `2001:db8:10::53`

**Practical example: static IPv4 setup**

Suppose you need to configure `ens160` with a static address. In `nmtui`, you would:
1. Select the connection for `ens160`.
2. Change IPv4 method to **Manual**.
3. Enter the address and prefix.
4. Enter the gateway.
5. Add DNS servers.
6. Save and exit.

Equivalent `nmcli` example for comparison:

```bash
nmcli connection modify ens160 \
  ipv4.addresses 192.168.10.20/24 \
  ipv4.gateway 192.168.10.1 \
  ipv4.dns "192.168.10.10 8.8.8.8" \
  ipv4.method manual
nmcli connection up ens160
```

This comparison helps you understand what `nmtui` is doing behind the scenes.

**Activating a connection**

After saving changes, the profile may not be active yet. Use **Activate a connection** in `nmtui` to bring it up.

Activation matters because:
- Saved settings do not always apply to the live interface immediately.
- A connection can exist but remain down.
- Multiple profiles may exist for one device, and only one is active at a time.

If activation fails, check:
- Whether the interface is physically connected.
- Whether the profile is bound to the correct device.
- Whether the IP settings conflict with the network.
- Whether another profile is already controlling the interface.

**Hostname configuration**

`nmtui` also includes **Set system hostname**. This changes the system hostname persistently.

Use it when:
- The exam task asks you to set a specific host identity.
- You want the hostname to match the role of the machine.
- You need a quick interactive method instead of `hostnamectl`.

Equivalent command:

```bash
sudo hostnamectl set-hostname server1.example.com
```

**Verification commands**

After using `nmtui`, verify the result with command-line tools. This is important because the interface may accept input even when the network is not actually reachable.

Recommended checks:
- `ip addr show ens160` to confirm the assigned address.
- `ip route` to confirm the default gateway.
- `nmcli connection show --active` to confirm the active profile.
- `nmcli device show ens160` to inspect detailed device properties.
- `ping -c 4 <gateway>` to test local reachability.
- `ping -c 4 8.8.8.8` to test external routing.
- `getent hosts example.com` to test DNS resolution.

Example verification session:

```bash
ip addr show ens160
ip route
nmcli connection show --active
nmcli device show ens160 | grep -E 'IP4|GENERAL.STATE|GENERAL.CONNECTION'
```

**Connection profile files**

NetworkManager stores profiles under:

```bash
/etc/NetworkManager/system-connections/
```

These files are usually keyfile format and are protected with restrictive permissions. You normally do not edit them directly during routine RHCSA tasks unless required.

A profile may contain settings similar to:

```ini
[connection]
id=ens160
type=ethernet
interface-name=ens160

[ipv4]
method=manual
addresses=192.168.10.20/24;
gateway=192.168.10.1
dns=192.168.10.10;8.8.8.8;

[ipv6]
method=ignore
```

`nmtui` is useful because it updates these settings safely without requiring you to remember the exact file syntax.

**When to use nmtui vs nmcli**

| Tool | Best use | Strength |
|---|---|---|
| `nmtui` | Interactive setup on a console | Fast, low syntax risk |
| `nmcli` | Scripting and repeatable automation | Precise and scriptable |
| Manual files | Rare troubleshooting or special cases | Direct control |

Use `nmtui` when you need speed and clarity. Use `nmcli` when you need reproducibility or when the exam asks for command-line configuration.

**Troubleshooting tips**

If the interface does not come up:
- Confirm the device name with `nmcli device status`.
- Make sure the profile is attached to the correct NIC.
- Check for typos in IP address, prefix, or gateway.
- Ensure the network actually uses the addressing scheme you entered.
- Verify that the interface is not blocked by another profile or stale configuration.

If DNS fails but IP connectivity works:
- Check the DNS server entries in the profile.
- Confirm `/etc/resolv.conf` is being managed by NetworkManager.
- Test name resolution with `getent hosts`.

If the wrong profile activates:
- Review autoconnect settings.
- Rename profiles clearly to match their purpose.
- Remove unused or conflicting profiles if necessary.

**Exam-focused workflow**

For RHCSA tasks, a reliable approach is:
1. Identify the interface name with `nmcli device status`.
2. Open `nmtui`.
3. Edit or create the correct connection.
4. Set the required IPv4/IPv6 values.
5. Save the profile.
6. Activate the connection.
7. Verify with `ip addr`, `ip route`, and `ping`.
8. Set the hostname if the task requires it.

This workflow minimizes mistakes and gives you a clear path from configuration to validation.

**Key takeaways**

- `nmtui` is the interactive front end for NetworkManager.
- It is ideal for console-based network setup on RHEL.
- It edits connection profiles, not raw interface state.
- Always verify changes after saving and activating.
- Pair `nmtui` with `nmcli` and `ip` for complete validation.
