## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**DNS fundamentals**

<abbr title="Domain Name System">DNS</abbr> translates hostnames into IP addresses and other records. On RHEL, DNS configuration is usually part of network configuration, but you should also understand how the resolver on the host consumes that configuration.

A typical lookup path is:

- Application asks the system resolver for a name.
- The resolver consults `/etc/nsswitch.conf`.
- DNS settings come from `/etc/resolv.conf` or from a resolver service integration.
- A DNS server returns the record set.

Common record types you should recognize:

- **A**: IPv4 address
- **AAAA**: IPv6 address
- **PTR**: reverse lookup from IP to name
- **CNAME**: alias to another name
- **MX**: mail exchanger
- **NS**: authoritative name server
- **TXT**: text data, often used for verification and policy

**Resolver configuration on RHEL**

On modern RHEL systems, `/etc/resolv.conf` may be a static file, a symlink, or a file generated by a network management component. The exact behavior depends on how the system is configured. Do not assume one fixed ownership model.

Inspect the current resolver state with:

```bash
cat /etc/resolv.conf
ls -l /etc/resolv.conf
```

A typical resolver file contains:

```conf
nameserver 192.0.2.53
nameserver 192.0.2.54
search example.com corp.example.com
options timeout:2 attempts:3
```

Key points:

- **nameserver** entries are queried in order.
- **search** domains help resolve short hostnames.
- **options** tune resolver behavior, such as retry timing.
- The first reachable server is not always the only one used; behavior depends on the resolver library and server responses.

If you manage DNS through NetworkManager, the active connection profile often supplies DNS servers and search domains. The runtime view can include merged data from multiple sources, so compare the profile and the live state when troubleshooting.

Useful commands:

```bash
nmcli connection show
nmcli connection show "System eth0"
nmcli device show eth0
```

Look for fields such as:

- `IP4.DNS`
- `IP4.DOMAIN`
- `IP6.DNS`
- `IP6.DOMAIN`
- `GENERAL.CONNECTION`

To set DNS servers on a connection profile:

```bash
nmcli connection modify "System eth0" ipv4.dns "192.0.2.53 192.0.2.54"
nmcli connection modify "System eth0" ipv4.ignore-auto-dns yes
nmcli connection modify "System eth0" ipv4.dns-search "example.com corp.example.com"
nmcli connection up "System eth0"
```

For IPv6, use the matching `ipv6.*` properties:

```bash
nmcli connection modify "System eth0" ipv6.dns "2001:db8::53"
nmcli connection modify "System eth0" ipv6.ignore-auto-dns yes
nmcli connection modify "System eth0" ipv6.dns-search "example.com"
```

When you need a route-only domain, prefix the domain with `~` in the NetworkManager profile:

```bash
nmcli connection modify "System eth0" ipv4.dns-search "~corp.example.com"
```

This tells NetworkManager to treat the domain as route-only when supported by the DNS plugin and resolver integration. Use it when you want queries for that suffix to go to a specific connection's DNS server without adding it as a general search suffix.

> Be careful: route-only behavior depends on the DNS backend and system integration. Verify the effective resolver behavior after changes.

**Editing connection profiles directly**

NetworkManager stores connection profiles under `/etc/NetworkManager/system-connections/`. You normally edit them with `nmcli`, but you may inspect them for troubleshooting.

Example profile fragment:

```ini
[ipv4]
method=auto
dns=192.0.2.53;192.0.2.54;
ignore-auto-dns=true
dns-search=example.com;corp.example.com;
```

After manual edits, reload or reconnect the profile:

```bash
nmcli connection reload
nmcli connection down "System eth0"
nmcli connection up "System eth0"
```

**Testing DNS resolution**

Use tools that show the query path and response details.

Preferred tools:

- **dig**: detailed DNS query tool
- **host**: concise lookup tool
- `getent hosts`: uses the system name service switch path

Examples:

```bash
dig example.com
dig A example.com
dig AAAA example.com
dig -x 192.0.2.10
host example.com
getent hosts example.com
```

For reverse lookups, use `dig -x` or the corresponding `in-addr.arpa` / `ip6.arpa` name. For example:

```bash
dig -x 192.0.2.10
```

If you want to inspect a specific server directly, add `@server`:

```bash
dig @192.0.2.53 example.com
dig @192.0.2.53 -x 192.0.2.10
```

Useful `dig` options:

- `+short` for compact output
- `+trace` to follow delegation from root to authoritative servers
- `+nocmd` and `+noall +answer` for focused output

Examples:

```bash
dig +short example.com
dig +trace example.com
dig +noall +answer example.com
```

A CNAME often appears in the answer section even when you query `A` or `AAAA`, because the resolver follows the alias chain. That is normal and useful when diagnosing name indirection.

**Troubleshooting workflow**

A practical DNS troubleshooting sequence is:

1. Confirm the interface is up and has the expected IP configuration.
2. Check the active DNS servers and search domains.
3. Test resolution with `getent`, `dig`, and `host`.
4. Compare short-name and FQDN behavior.
5. Test against a known server directly with `dig @server`.
6. Verify reverse lookups if name-to-IP mapping matters.
7. Check firewall and reachability if queries time out.

Example checks:

```bash
ip addr show
ip route
nmcli device status
nmcli device show eth0
getent hosts server1
dig server1.example.com
dig -x 192.0.2.10
```

Common failure patterns:

- **NXDOMAIN**: the name does not exist in DNS
- **SERVFAIL**: the server failed to answer correctly
- **timeout**: network path, firewall, or server availability issue
- **wrong answer**: stale cache, wrong server, or incorrect search domain

If short names fail but FQDNs work, inspect the search list and the resolver order. If only one server fails, test each server directly.

**NetworkManager and DNS behavior**

NetworkManager can receive DNS information from DHCP, static profile settings, VPN connections, and manual overrides. The effective DNS state may be a combination of these sources.

Important settings:

- `ipv4.ignore-auto-dns yes` prevents DHCP-provided DNS from being used.
- `ipv4.dns` sets static IPv4 DNS servers.
- `ipv4.dns-search` sets search domains.
- `ipv6.ignore-auto-dns yes` prevents IPv6-provided DNS from being used.
- `ipv6.dns` sets static IPv6 DNS servers.

Example static configuration:

```bash
nmcli connection modify "System eth0" ipv4.method manual
nmcli connection modify "System eth0" ipv4.addresses 192.0.2.20/24
nmcli connection modify "System eth0" ipv4.gateway 192.0.2.1
nmcli connection modify "System eth0" ipv4.dns "192.0.2.53 192.0.2.54"
nmcli connection modify "System eth0" ipv4.dns-search "example.com"
nmcli connection up "System eth0"
```

If you use DHCP but want to keep the address while overriding DNS:

```bash
nmcli connection modify "System eth0" ipv4.ignore-auto-dns yes
nmcli connection modify "System eth0" ipv4.dns "192.0.2.53"
nmcli connection up "System eth0"
```

**`/etc/nsswitch.conf` and name service order**

The Name Service Switch <abbr title="Name Service Switch">NSS</abbr> file controls which databases are consulted and in what order. For host lookups, the `hosts:` line is especially important.

Example:

```conf
hosts: files dns myhostname
```

Meaning:

- `files` checks `/etc/hosts`
- `dns` queries DNS
- `myhostname` can resolve the local host name through system services

If `/etc/hosts` contains an entry, it can override DNS for that name:

```conf
192.0.2.20 server1.example.com server1
```

Use `/etc/hosts` for local overrides, lab environments, or temporary testing. Avoid using it as a substitute for proper DNS in production unless you have a clear reason.

**`/etc/hosts` versus DNS**

Use the right tool for the job:

- **/etc/hosts**
  - Fast local override
  - Useful for bootstrapping or isolated systems
  - Not scalable for many hosts
- **DNS**
  - Centralized, scalable, and authoritative
  - Supports service discovery and reverse lookups
  - Preferred for enterprise environments

If a hostname resolves unexpectedly, check `/etc/hosts` before assuming DNS is broken.

**Legacy tools and modern practice**

`nslookup` still exists on many systems, but it is a legacy tool and its behavior can differ from the system resolver path. Prefer `dig`, `host`, and `getent` for troubleshooting on modern RHEL systems.

Examples:

```bash
nslookup example.com
host example.com
getent ahosts example.com
```

Use `getent` when you want to test the same resolution path applications use through NSS.

**Practical lab examples**

Set a static DNS server and verify it:

```bash
nmcli connection modify "System eth0" ipv4.dns "192.0.2.53"
nmcli connection modify "System eth0" ipv4.ignore-auto-dns yes
nmcli connection up "System eth0"
cat /etc/resolv.conf
dig @192.0.2.53 example.com
```

Add a search domain and test short-name resolution:

```bash
nmcli connection modify "System eth0" ipv4.dns-search "example.com"
nmcli connection up "System eth0"
getent hosts server1
dig server1
```

Check reverse DNS:

```bash
dig -x 192.0.2.20
host 192.0.2.20
```

Inspect the resolver path used by the system:

```bash
grep '^hosts:' /etc/nsswitch.conf
cat /etc/resolv.conf
```

**Troubleshooting checklist**

- Confirm the interface has link and an IP address.
- Confirm the correct connection profile is active.
- Confirm DNS servers are present in the live configuration.
- Confirm `/etc/nsswitch.conf` includes `dns` for host lookups.
- Confirm `/etc/hosts` is not masking the DNS name.
- Confirm the DNS server is reachable on port 53 over UDP and, if needed, TCP.
- Confirm the record exists and is spelled correctly.
- Confirm search domains are appropriate for short-name lookups.

**Exam-focused reminders**

- Know how to inspect and modify DNS settings with `nmcli`.
- Know how to verify resolver configuration with `/etc/resolv.conf`.
- Know how to test lookups with `dig`, `host`, and `getent`.
- Know how to use `dig -x` for reverse lookups.
- Know how `/etc/hosts` and `/etc/nsswitch.conf` affect name resolution.
- Know that DNS behavior can vary depending on how the system integrates NetworkManager and the resolver.
