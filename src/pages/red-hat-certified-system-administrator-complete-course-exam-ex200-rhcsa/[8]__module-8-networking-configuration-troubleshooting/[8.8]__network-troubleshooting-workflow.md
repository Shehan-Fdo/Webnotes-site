## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Network troubleshooting mindset**

A reliable troubleshooting workflow matters more than memorizing isolated commands. On RHEL, you usually work from the bottom of the stack upward: link, addressing, routing, name resolution, firewall policy, and then the service itself. This reduces guesswork and helps you prove where the failure occurs.

A good workflow also avoids false conclusions. For example, a host can answer `ping` but still fail to reach a TCP service, and a link can be “up” without a valid NetworkManager profile being active. On the exam, you want to verify each layer with the right tool before changing anything.

- Start with the symptom and scope.
- Confirm whether the issue is local, remote, or service-specific.
- Change one variable at a time so you can verify the effect.

**Layered troubleshooting approach**

Use a repeatable sequence so you do not skip important checks. The exact order can vary, but this progression is practical for RHCSA tasks:

1. Verify the interface state and IP configuration.
2. Check the route to the destination.
3. Test DNS if you are using hostnames.
4. Confirm the service is listening on the expected address and port.
5. Inspect firewall rules and active zones.
6. Review SELinux only when the service and firewall look correct.
7. Re-test after each change.

This workflow is especially useful when a system has multiple interfaces, multiple zones, or both IPv4 and IPv6 connectivity.

**Interface and link checks**

Begin with the local device state. `ip` is the primary tool for interface and routing inspection on current RHEL systems.

```bash
ip link show
ip addr show
ip route show
ip -br addr
```

What to look for:

- **Link state**: `UP` means the interface is enabled at the kernel level, but it does not guarantee NetworkManager has an active connection profile.
- **Addressing**: confirm the expected IPv4 or IPv6 address, prefix length, and scope.
- **Routes**: verify the default gateway and any static routes needed for the destination.

If the interface is administratively down, you can bring the link up:

```bash
ip link set dev enp1s0 up
```

That only enables the device. If NetworkManager manages the interface, you often also need to activate or reconnect the profile:

```bash
nmcli device status
nmcli device connect enp1s0
nmcli connection show
nmcli connection up "System enp1s0"
```

Use `nmcli` when you need to inspect or activate NetworkManager-managed connections. It is the correct tool for persistent configuration and for understanding which profile is attached to which device.

- `nmcli device status` shows whether a device is connected, disconnected, or unmanaged.
- `nmcli connection show` lists saved profiles.
- `nmcli connection show --active` shows active profiles.
- `nmcli device show enp1s0` displays device-specific details such as IP, gateway, DNS, and connection association.

**Addressing and route validation**

A host can have a valid address but still fail to reach a destination because the route is missing or wrong. Check the local routing table and the path to the target.

```bash
ip route
ip route get 192.0.2.50
ip route get 8.8.8.8
```

Useful checks:

- Confirm the default route points to the correct gateway.
- Confirm the source address chosen by the kernel matches the expected interface.
- Confirm there is no conflicting route with a more specific prefix.
- For IPv6, use `ip -6 route` and `ip -6 route get <address>`.

If you need to add a temporary route for testing, use `ip route add`, but remember that it is not persistent unless you configure it through NetworkManager or the appropriate network profile.

```bash
ip route add 192.0.2.0/24 via 192.0.2.1 dev enp1s0
```

For persistent changes, prefer `nmcli`:

```bash
nmcli connection modify "System enp1s0" ipv4.gateway 192.0.2.1
nmcli connection modify "System enp1s0" ipv4.routes "192.0.2.0/24 192.0.2.1"
nmcli connection up "System enp1s0"
```

**DNS troubleshooting**

If you can reach an IP address but not a hostname, the problem is often name resolution rather than connectivity. Check `/etc/resolv.conf`, the active NetworkManager profile, and the resolver behavior.

```bash
cat /etc/resolv.conf
nmcli device show enp1s0 | grep -i dns
getent hosts server.example.com
```

Why these checks matter:

- `getent hosts` tests the system resolver path, not just a single application.
- `resolv.conf` may be managed by NetworkManager or another resolver component.
- DNS failures can look like network failures when the real issue is name lookup.

If you need to set DNS through NetworkManager, configure it on the connection profile:

```bash
nmcli connection modify "System enp1s0" ipv4.dns "192.0.2.53 192.0.2.54"
nmcli connection modify "System enp1s0" ipv4.ignore-auto-dns yes
nmcli connection up "System enp1s0"
```

For IPv6 DNS, use `ipv6.dns` and `ipv6.ignore-auto-dns` as needed.

**Connectivity testing**

Use the right test for the layer you are validating. `ping` confirms basic IP reachability, but it does not prove that a TCP or UDP service is available.

```bash
ping -c 4 192.0.2.10
ping -c 4 server.example.com
traceroute 192.0.2.10
tracepath 192.0.2.10
```

Practical interpretation:

- If `ping` fails to an IP address, check link, address, route, gateway, and upstream filtering.
- If `ping` works but a service fails, focus on the service port, firewall, SELinux, and binding address.
- If a hostname fails but the IP works, focus on DNS.

`traceroute` helps identify where packets stop along the path. `tracepath` is also commonly available on RHEL and can be useful when installed. Use whichever is present on the system.

For port-level testing, use tools that actually test the service protocol or at least the socket:

```bash
nc -vz 192.0.2.10 22
nc -vz 192.0.2.10 80
curl -I http://192.0.2.10
```

- `nc -vz` checks whether a TCP port is reachable.
- `curl -I` verifies HTTP response headers.
- A successful `ping` does not mean the port is open.

**Service binding and listening sockets**

If the network path is fine, verify that the service is listening on the expected address and port. A service may be running but bound only to `127.0.0.1`, which makes it unreachable from other hosts.

```bash
ss -tulpen
ss -tulpn | grep ':80'
ss -tulpn | grep ':22'
```

What to confirm:

- The service process is present.
- The port is listening.
- The bind address matches the intended access pattern.
- The protocol is correct, especially for services that support both TCP and UDP.

Examples of common binding issues:

- A web server listens only on localhost.
- A database listens only on a private interface.
- A daemon listens on IPv6 only, while the client uses IPv4.

If needed, inspect the service unit and configuration:

```bash
systemctl status httpd
systemctl status sshd
grep -R "Listen" /etc/httpd/ /etc/ssh/ 2>/dev/null
```

**Firewall troubleshooting with firewalld**

On RHEL, `firewalld` is the standard firewall manager. Troubleshooting requires checking the active zone, not just the default zone. A common mistake is to run a generic listing and assume it reflects the interface you are testing.

```bash
firewall-cmd --state
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-all
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-ports
```

Why this matters:

- An interface may belong to a zone other than `public`.
- Runtime and permanent settings can differ.
- A service may be allowed in one zone but blocked in another.

To inspect the zone attached to an interface:

```bash
firewall-cmd --get-zone-of-interface=enp1s0
```

To allow a service temporarily:

```bash
firewall-cmd --zone=public --add-service=http
```

To make it persistent:

```bash
firewall-cmd --zone=public --add-service=http --permanent
firewall-cmd --reload
```

For a custom port:

```bash
firewall-cmd --zone=public --add-port=8080/tcp --permanent
firewall-cmd --reload
```

A practical firewall workflow is:

1. Identify the active zone for the interface.
2. Check whether the required service or port is allowed in that zone.
3. Add the rule temporarily to test.
4. Make it permanent only after confirming the fix.

**SELinux as a network access factor**

SELinux can block network-facing services even when the firewall and routing are correct. Treat it as a later-stage check after you have confirmed the service is listening and the firewall allows the traffic.

Useful commands:

```bash
getenforce
sestatus
ausearch -m AVC,USER_AVC -ts recent
journalctl -t setroubleshoot --since "1 hour ago"
```

Important caution:

> `sealert` is useful when installed, but it is not guaranteed to be present on a minimal RHCSA system. Do not depend on it as your only SELinux check.

Common SELinux-related network issues include:

- A service is denied access to a nonstandard port.
- A daemon is blocked from binding to a port type it is not allowed to use.
- A file context problem prevents a service from reading its configuration or content.

Examples of SELinux-aware fixes:

```bash
semanage port -l | grep http
semanage port -a -t http_port_t -p tcp 8080
restorecon -Rv /var/www/html
```

If you need to test whether SELinux is the cause, you can temporarily set permissive mode for diagnosis:

```bash
setenforce 0
```

Return to enforcing mode after testing:

```bash
setenforce 1
```

Use this only as a diagnostic step, not as a permanent solution.

**A practical troubleshooting sequence**

When a remote service is unreachable, follow a disciplined order so you do not chase symptoms.

1. Confirm the local interface is up and has the expected address.
2. Confirm the route to the destination exists.
3. Test name resolution if you are using a hostname.
4. Test basic reachability with `ping`.
5. Test the service port with `nc`, `curl`, or a protocol-specific client.
6. Confirm the service is listening on the correct address and port.
7. Check the active firewalld zone and allowed service or port.
8. Review SELinux denials if the issue persists.
9. Re-test after each change.

Example workflow for a web service:

```bash
ip addr show enp1s0
ip route
getent hosts web1.example.com
ping -c 4 web1.example.com
nc -vz web1.example.com 80
curl -I http://web1.example.com
ss -tulpn | grep ':80'
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-services
getenforce
```

**Common failure patterns**

Some failures repeat often enough that you should recognize them quickly.

- **No IP address**
  - The interface is up but not configured.
  - The DHCP lease failed.
  - The NetworkManager profile is disconnected.

- **Wrong default gateway**
  - Local traffic works, but remote traffic fails.
  - `ip route get` shows the wrong source or next hop.

- **DNS failure only**
  - IP-based access works.
  - Hostname-based access fails.
  - Resolver configuration is wrong or unreachable.

- **Service listens locally only**
  - `ss` shows `127.0.0.1:PORT`.
  - Remote clients cannot connect even though the service is running.

- **Firewall zone mismatch**
  - The interface is in a different zone than expected.
  - The needed service is allowed in one zone but not the active one.

- **SELinux denial**
  - The service starts but cannot access a port, file, or resource.
  - Audit logs show AVC denials.

**Useful command set for the exam**

Keep a small toolkit ready and use it consistently.

```bash
ip addr show
ip route show
ip route get <destination>
nmcli device status
nmcli connection show --active
nmcli device show <interface>
getent hosts <name>
ss -tulpn
firewall-cmd --get-active-zones
firewall-cmd --zone=<zone> --list-all
getenforce
ausearch -m AVC,USER_AVC -ts recent
```

If you need to make a persistent network change, prefer `nmcli` over temporary `ip` changes. If you need to validate a service, prefer a port-aware test over `ping`. If you need to inspect firewall behavior, always check the active zone.

**Exam-focused habits**

The RHCSA exam rewards calm, verifiable troubleshooting. You should avoid broad changes and instead prove each layer before moving on.

- Use `ip` for immediate state.
- Use `nmcli` for NetworkManager-managed configuration.
- Use `ss` for listening sockets.
- Use `firewall-cmd` with the active zone.
- Use SELinux checks only after network and firewall checks.
- Re-test after every fix so you know which change solved the problem.

A disciplined workflow saves time and prevents accidental regressions. It also helps you explain your reasoning if you need to revisit a system after making multiple changes.
