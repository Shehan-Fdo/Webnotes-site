## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

The RHCSA exam expects you to edit configuration files quickly and safely, often on a remote system with limited time. `vi` and `vim` are the standard editors you should know because they are available on nearly every Red Hat system. The core workflow is simple: enter the right mode, make changes, verify them, then save or quit without losing work.

`vi` is a modal editor, which means keys do different things depending on the current mode. Most mistakes come from forgetting which mode you are in. The most important habit is to return to Normal mode before issuing commands like save, quit, search, or replace.

- **Normal mode** is for navigation and editor commands.
- **Insert mode** is for typing text.
- **Command-line mode** is for save, quit, search, and substitution commands.
- **Visual mode** is for selecting text, though it is less essential for RHCSA than the other modes.

**Getting into and out of modes**

You usually start in Normal mode. Press `i` to insert before the cursor, `a` to append after the cursor, or `o` to open a new line below. Press `<Esc>` to return to Normal mode from Insert or Visual mode.

A reliable habit is to tap `<Esc>` a few times before running a command. This reduces the chance of accidentally typing a command into the file instead of into the editor command line.

- `i` enters Insert mode at the cursor.
- `I` enters Insert mode at the beginning of the line.
- `a` appends after the cursor.
- `A` appends at the end of the line.
- `o` opens a new line below.
- `O` opens a new line above.
- `<Esc>` returns to Normal mode.

**Saving files**

Saving writes your changes to disk. In `vi`, saving is usually done from Normal mode with `:` commands. You should know the difference between saving, saving under a new name, and saving only if the file changed.

```vim
:w
:w filename
:wq
:x
```

- `:w` writes the current file.
- `:w filename` writes the buffer to a new file name.
- `:wq` writes and quits.
- `:x` writes only if needed, then quits.
- `ZZ` in Normal mode saves and quits if the file changed.

`ZZ` is convenient, but `:wq` is more explicit and easier to remember under pressure. If you are editing a system file, saving may require elevated privileges. In that case, you often open the file with `sudo`:

```bash
sudo vi /etc/ssh/sshd_config
```

If you forgot to use `sudo` and the file is read-only, you can sometimes write through `sudo tee`, but for the exam it is usually better to reopen the file correctly with the needed privileges.

> If you see `E45: 'readonly' option is set`, you are likely editing a file without permission or opened it in read-only mode. Check your privileges before continuing.

**Quitting without saving**

Quitting safely matters when you make a mistake or want to abandon changes. The editor protects you from accidental data loss by warning when the buffer has unsaved changes.

```vim
:q
:q!
```

- `:q` quits only if there are no unsaved changes.
- `:q!` quits and discards unsaved changes.
- `ZQ` quits without saving, similar to `:q!`.

Use `:q!` when you want to abandon a bad edit and start over. Use it carefully, because it discards all unsaved changes in the current buffer.

**Search basics**

Searching helps you find configuration lines quickly, especially in large files. In Normal mode, `/` searches forward and `?` searches backward.

```vim
/PermitRootLogin
?PasswordAuthentication
n
N
```

- `/pattern` searches forward for `pattern`.
- `?pattern` searches backward for `pattern`.
- `n` repeats the last search in the same direction.
- `N` repeats the last search in the opposite direction.

Searches are case-sensitive by default in many setups, but Vim can be configured to ignore case. For exam work, assume exact matching unless you know the file content and search behavior. If you need a literal search for special characters, escape them with `\`.

Common search habits:

- Search for the exact directive name, such as `PermitRootLogin`.
- Use `n` to jump through repeated matches.
- Use `N` when you overshoot the target.
- Search before editing to avoid changing the wrong line.

**Search options and useful patterns**

Vim supports regular expressions in search patterns. You do not need advanced regex for most RHCSA tasks, but basic awareness helps when matching variants.

```vim
/^#\?PermitRootLogin
/^[[:space:]]*ListenAddress
```

- `^` matches the start of a line.
- `$` matches the end of a line.
- `#\?` matches an optional `#`, which is useful for commented directives.
- `[[:space:]]*` matches optional leading whitespace.

These patterns are useful when a configuration file contains both commented examples and active settings. For example, you may want to find a directive whether it is commented or not, then decide whether to uncomment and change it.

**Replacing text**

Replacement is one of the most important `vi` skills for RHCSA because many configuration changes involve editing repeated directives. The substitution command uses the form `:s/old/new/`.

```vim
:s/old/new/
:%s/old/new/g
:%s/old/new/gc
```

- `:s/old/new/` replaces the first match on the current line.
- `:%s/old/new/g` replaces all matches in the file.
- `:%s/old/new/gc` replaces all matches in the file and asks for confirmation.

The `%` range means the whole file. The `g` flag means replace all matches on each line, not just the first one. The `c` flag is useful when you want to review each change before accepting it.

A practical example for a configuration file:

```vim
:%s/SELINUX=enforcing/SELINUX=permissive/g
```

If you need to change only one occurrence on a line, omit `g`. If you need to confirm each replacement, add `c`.

**Replacing with confirmation**

Interactive replacement is safer when the file contains similar values or when you are not fully sure of the impact.

```vim
:%s/ListenAddress 0.0.0.0/ListenAddress 192.0.2.10/gc
```

During confirmation, Vim prompts you with options such as:

- `y` to replace this match.
- `n` to skip this match.
- `a` to replace all remaining matches.
- `q` to quit the substitution.
- `l` to replace this match and quit.
- `<Esc>` to stop the command.

Use confirmation when editing files with repeated settings, especially if the same word appears in comments, examples, or multiple service blocks.

**Working with commented configuration lines**

Many Red Hat configuration files include commented defaults. A common task is to uncomment a line and modify its value. You can do this manually or with substitution.

Example:

```vim
#PermitRootLogin yes
```

To uncomment and change it, you might use:

```vim
:%s/^#PermitRootLogin yes/PermitRootLogin no/
```

This approach is precise and avoids accidental changes elsewhere. If the line may have leading spaces, use a pattern that allows whitespace:

```vim
:%s/^[[:space:]]*#\?PermitRootLogin .*/PermitRootLogin no/
```

This is especially useful when configuration files vary slightly between systems or when indentation is inconsistent.

**Practical command-line workflow**

A typical RHCSA editing session follows a repeatable pattern:

1. Open the file with the correct privileges.
2. Search for the directive you need.
3. Edit the line or use substitution.
4. Verify the result.
5. Save and quit.

Example:

```bash
sudo vi /etc/ssh/sshd_config
```

Inside `vi`:

```vim
/PermitRootLogin
:%s/^#PermitRootLogin.*/PermitRootLogin no/
:wq
```

This workflow is efficient because it minimizes scrolling and reduces the chance of editing the wrong section.

**Useful navigation while searching**

Search is faster when combined with movement commands. After finding a match, you can move around the file without leaving Normal mode.

- `gg` goes to the top of the file.
- `G` goes to the bottom of the file.
- `:set number` shows line numbers.
- `:set nonumber` hides line numbers.
- `:<line>` jumps to a specific line number.

Example:

```vim
:set number
:42
```

Line numbers are helpful when instructions or logs reference a specific line. They also make it easier to confirm that a substitution affected the intended area.

**Undo and recovery**

When editing under time pressure, you need a quick way to recover from mistakes. Undo is often faster than quitting and reopening the file.

- `u` undoes the last change.
- `Ctrl` + `r` redoes an undone change.
- `U` restores the current line in some Vim versions, though `u` is the more reliable habit to build.

If a substitution goes wrong, you can often undo it immediately. If the file becomes confusing, `:q!` lets you exit without saving and restart cleanly.

**Common exam-safe habits**

These habits reduce errors and save time during the exam:

- Check mode before typing commands.
- Search before editing.
- Use precise substitutions instead of manual repeated edits.
- Confirm replacements when the file contains similar text.
- Save only after verifying the final content.
- Quit without saving if the edit is clearly wrong.

**Quick reference**

```vim
i        " insert before cursor
a        " append after cursor
<Esc>    " return to Normal mode
:w       " save
:wq      " save and quit
:x       " save if changed and quit
:q       " quit if unchanged
:q!      " quit without saving
/word    " search forward
?word    " search backward
n        " next match
N        " previous match
:s/a/b/  " replace first match on current line
:%s/a/b/g " replace all matches in file
```

**Troubleshooting**

If commands do not behave as expected, the issue is usually one of these:

- You are in Insert mode instead of Normal mode.
- You typed a command into the file instead of the command line.
- The search pattern is too broad or too narrow.
- The substitution matched more text than intended.
- You do not have permission to save the file.

When in doubt, press `<Esc>`, search again, and make one change at a time. That approach is slower than a bulk replace, but it is safer when editing critical system configuration.
