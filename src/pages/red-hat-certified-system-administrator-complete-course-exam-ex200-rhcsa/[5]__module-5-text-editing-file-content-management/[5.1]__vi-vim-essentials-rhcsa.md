## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Why vi/vim matters on RHCSA**

`vi` and `vim` are the safest editors to rely on during the exam because they are commonly available on minimal Red Hat systems. You should be able to edit configuration files quickly without depending on a graphical desktop or extra packages.

`vim` is the enhanced version of `vi`, but the core workflow is the same:
- **Normal mode** for navigation and commands
- **Insert mode** for typing text
- **Command-line mode** for saving, searching, substituting, and setting options
- **Visual mode** for selecting text blocks

> On exam systems, do not assume your personal shell aliases, editor settings, or `~/.vimrc` customizations exist. Use the editor as it is installed on the system.

**Modes and the mental model**

The most important habit is knowing which mode you are in. Most mistakes happen when you try to type a command while still in insert mode, or when you try to edit text while in normal mode.

- **Normal mode** is the default after opening a file.
- **Insert mode** accepts typed text.
- **Command-line mode** starts with `:` and is used for editor commands.
- **Visual mode** selects text for operations like delete, yank, or change.

Common mode transitions:
- Press `i` to insert before the cursor.
- Press `a` to append after the cursor.
- Press `o` to open a new line below.
- Press `O` to open a new line above.
- Press `<Esc>` to return to normal mode.

**Opening files efficiently**

You can open files directly at a useful location, which saves time during the exam.

```bash
vim /etc/ssh/sshd_config
vim +42 /etc/fstab
vim +/PermitRootLogin /etc/ssh/sshd_config
vim -R /var/log/messages
```

Useful patterns:
- `vim +42 file` opens at line 42.
- `vim +/pattern file` opens at the first match of `pattern`.
- `vim -R file` opens in read-only mode, which is useful for inspection.
- `view file` is another read-only entry point on many systems.

**Navigation in normal mode**

You should move quickly without reaching for arrow keys all the time. The exam rewards speed and accuracy.

- `h`, `j`, `k`, `l` move left, down, up, right.
- `w` moves to the next word.
- `b` moves to the previous word.
- `e` moves to the end of the current word.
- `0` moves to the start of the line.
- `^` moves to the first nonblank character.
- `$` moves to the end of the line.
- `gg` moves to the first line.
- `G` moves to the last line.
- `:42` jumps to line 42.
- `:set number` shows line numbers.
- `:set nonumber` hides line numbers.

Practical use:
- Turn on line numbers when a task references a line number in a config file.
- Turn them off again if they distract you.
- Use `:set number` and `:set nonumber` as temporary session settings, not permanent assumptions.

**Searching and jumping**

Search is one of the fastest ways to find configuration keys.

- `/pattern` searches forward.
- `?pattern` searches backward.
- `n` repeats the search in the same direction.
- `N` repeats the search in the opposite direction.
- `*` searches forward for the word under the cursor.
- `#` searches backward for the word under the cursor.

Examples:
```vim
/ListenAddress
?PermitRootLogin
```

You can combine search with opening a file:
```bash
vim +/SELINUX /etc/selinux/config
```

This is useful when you know the setting name but not the line number.

**Editing text quickly**

In normal mode, editing commands are often built from an operator plus a motion. This is a core `vi`/`vim` skill.

- `x` deletes the character under the cursor.
- `dd` deletes the current line.
- `dw` deletes from the cursor to the start of the next word.
- `d$` deletes from the cursor to the end of the line.
- `cw` changes a word.
- `cc` changes the whole line.
- `yy` yanks the current line.
- `p` pastes after the cursor.
- `P` pastes before the cursor.

Operator + motion examples:
- `d2w` deletes two words.
- `c$` changes from the cursor to the end of the line.
- `y}` yanks to the next paragraph boundary.
- `dG` deletes from the cursor to the end of the file.
- `ygg` yanks from the cursor to the start of the file.

This pattern matters because it lets you edit precisely without switching into insert mode too often.

**Line ranges and multi-line operations**

RHCSA tasks often involve editing several related lines in a config file. `vim` supports line ranges in command-line mode and in normal-mode operations.

Useful range forms:
- `:1,10d` deletes lines 1 through 10.
- `:5,8y` yanks lines 5 through 8.
- `:3,7m 20` moves lines 3 through 7 after line 20.
- `:12,15co 30` copies lines 12 through 15 after line 30.
- `:.,$d` deletes from the current line to the end of the file.

Examples:
```vim
:10,20y
:25,30d
:5,8m 40
:12,14co $
```

Notes:
- `m` means move.
- `co` means copy.
- `.` means the current line.
- `$` means the last line.

These commands are especially useful when you need to reorder blocks in configuration files or duplicate a stanza before modifying it.

**Visual mode for block edits**

Visual mode is helpful when the edit is easier to see than to describe with motions.

- `v` starts character-wise visual selection.
- `V` starts line-wise visual selection.
- `<Ctrl> + v` starts block-wise visual selection.

Typical uses:
- Select several lines and then `d` to delete them.
- Select a block and then `y` to yank it.
- Select text and then `c` to replace it.

Practical scenario:
- You may need to duplicate a block of configuration lines, paste it elsewhere, and then adjust a few values.
- Line-wise visual mode is often the fastest way to handle multi-line config sections.

**Substitution and global replacement**

Search-and-replace is one of the most important exam skills because configuration files often require repeated value changes.

Basic substitution syntax:
```vim
:s/old/new/
```

Common forms:
- `:s/old/new/` replaces the first match on the current line.
- `:s/old/new/g` replaces all matches on the current line.
- `:%s/old/new/g` replaces all matches in the entire file.
- `:%s/old/new/gc` replaces all matches in the entire file with confirmation.

Examples:
```vim
:%s/localhost/127.0.0.1/g
:10,20s/yes/no/g
:%s/^#//gc
```

Important flags:
- `g` means replace all matches on each line.
- `c` means confirm each replacement.
- `i` may be available for case-insensitive matching depending on settings, but do not rely on it unless you know the session behavior.

Practical exam use:
- Remove comment markers from a block of settings.
- Replace a hostname or IP address across a file.
- Update repeated values in a service config without editing each line manually.

> Be careful with `:%s/.../.../g` on large files. Review the pattern first with a search, or use `c` to confirm replacements interactively.

**Registers, yank behavior, and clipboard awareness**

`vim` stores copied or deleted text in registers. Understanding this helps when you need to paste the right content repeatedly.

- `"` introduces a register name in many commands.
- `yy`, `dd`, and `p` use the unnamed register by default.
- Small deletes may overwrite the unnamed register.
- Named registers let you preserve text intentionally.

Examples:
```vim
"ayy
"ap
"bdd
"bp
```

Practical points:
- If you yank a line and then delete something else, the most recent delete may replace what you expected to paste.
- Use named registers when you need to keep one snippet safe while editing another.
- You can inspect registers with `:registers` or `:reg`.

Example inspection:
```vim
:registers
```

Clipboard behavior depends on how `vim` was built and whether the system has GUI clipboard support. On many exam systems, you should assume you are working inside the editor’s own registers, not the desktop clipboard.

**Undo, redo, and recovery during editing**

Mistakes are normal. You should recover quickly instead of restarting the edit.

- `u` undoes the last change.
- `<Ctrl> + r` redoes an undone change.
- `U` may restore the current line in some `vi` variants, but `u` is the safer habit.

Useful recovery workflow:
1. Make a change.
2. If it is wrong, press `u`.
3. If you undo too far, press `<Ctrl> + r`.
4. Re-run the search or substitution with a safer pattern.

**Saving, quitting, and write safety**

You should know the save and exit commands cold.

- `:w` saves the file.
- `:q` quits if there are no unsaved changes.
- `:q!` quits without saving.
- `:wq` saves and quits.
- `ZZ` saves and quits from normal mode if the file changed.
- `:x` saves and quits, similar to `:wq`.

Safer write options:
- `:w filename` writes the current buffer to a new file.
- `:saveas filename` saves the buffer under a new name in many `vim` builds.

Practical recovery workflow before risky edits:
```bash
cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak
vim /etc/ssh/sshd_config
```

Inside `vim`, you can also preserve a copy:
```vim
:w /tmp/sshd_config.backup
:saveas /tmp/sshd_config.test
```

This is useful when you want to test a change without losing the original file.

**Temporary settings that help during the exam**

Session options can make editing easier, especially in dense config files.

- `:set number` shows line numbers.
- `:set nonumber` hides line numbers.
- `:set paste` reduces auto-indenting and formatting surprises when pasting text.
- `:set nopaste` restores normal paste behavior.
- `:set list` shows tabs and trailing spaces.
- `:set nolist` hides those markers.

Example:
```vim
:set number
:set paste
```

Why this matters:
- `paste` helps when inserting blocks copied from another terminal or notes so `vim` does not reindent them unexpectedly.
- `number` helps when a task references a specific line or when you need to verify a range.

**Configuration files and defaults**

`vim` behavior can come from several places:
- System-wide defaults, often under `/etc/`
- User settings in `~/.vimrc`
- Runtime defaults if no custom configuration exists

On exam systems:
- Do not assume `~/.vimrc` exists.
- Do not assume syntax highlighting, line numbers, or custom key mappings are enabled.
- Verify behavior in the current session with `:set` and by testing commands directly.

If you need to inspect current settings:
```vim
:set
:set number?
:set paste?
```

The `?` form shows the current value of a specific option.

**Practical editing workflow for RHCSA tasks**

A reliable workflow reduces mistakes when editing service configs.

1. Open the file directly at the relevant location if possible.
   ```bash
   vim +/ListenAddress /etc/ssh/sshd_config
   ```
2. Turn on line numbers if they help.
   ```vim
   :set number
   ```
3. Search for the setting you need.
   ```vim
   /PermitRootLogin
   ```
4. Edit with a precise operator or substitution.
   ```vim
   :%s/^#PermitRootLogin.*/PermitRootLogin no/
   ```
5. Save and verify.
   ```vim
   :w
   ```
6. Quit only after confirming the file looks correct.
   ```vim
   :q
   ```

**Practical examples**

Example 1: uncomment and change a setting
```vim
:%s/^#\?PasswordAuthentication.*/PasswordAuthentication no/
```

Example 2: replace a hostname throughout a file
```vim
:%s/server01/server02/g
```

Example 3: copy a block and move it lower in the file
```vim
:20,28co 45
```

Example 4: delete a range of lines
```vim
:15,18d
```

Example 5: inspect registers after copying
```vim
:registers
```

**Common exam pitfalls**

- **Mode confusion**: you type `:w` while still in insert mode and nothing happens.
- **Overwriting the wrong text**: a delete replaces the unnamed register before you paste.
- **Overusing search-and-replace**: a broad `:%s` can change more than intended.
- **Assuming custom settings**: line numbers, syntax highlighting, or mappings may not exist.
- **Skipping backups**: a bad save can waste time if you have no fallback copy.

**Quick command reference**

```vim
i        " insert before cursor
a        " append after cursor
o        " open line below
<Esc>    " return to normal mode
/word    " search forward
?word    " search backward
n        " next match
N        " previous match
dd       " delete line
yy       " yank line
p        " paste after cursor
u        " undo
Ctrl-r   " redo
:w       " save
:q!      " quit without saving
:%s/a/b/g " replace all in file
:set number
:set paste
:registers
```

**What to remember under pressure**

`vi` and `vim` are about controlled editing, not memorizing every feature. For RHCSA, focus on fast navigation, precise line edits, safe substitutions, and recovery habits. If you can search, replace, copy, move, and save confidently, you can handle most configuration tasks without losing time.
