## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Why `sed` and `awk` matter**

`sed` and `awk` are core Unix text-processing tools that help you inspect, transform, and summarize command output without opening an editor. In RHCSA-style administration, you use them to clean config files, extract fields from logs, generate quick reports, and automate repetitive checks.

- `sed` is best for stream editing: substitutions, deletions, insertions, and line-oriented transformations.
- `awk` is best for structured text: field extraction, arithmetic, counters, and summaries.
- Both tools work well in pipelines, so you can combine them with `grep`, `sort`, `uniq`, `cut`, `tr`, and shell redirection.

**Core mental model**

`sed` processes input as a stream of lines and applies editing commands to each line. `awk` reads input as records, splits each record into fields, and runs pattern-action rules. Understanding those two models helps you choose the right tool quickly.

- `sed` focuses on *pattern space* and *hold space*.
- `awk` focuses on *records*, *fields*, and *actions*.
- Both tools are usually used non-interactively in scripts and one-liners.

**`sed` basics**

`sed` stands for *stream editor*. It reads input line by line, applies commands, and writes output. It is ideal when you need simple, repeatable transformations.

Common uses include:

- replacing text in config files
- deleting unwanted lines
- printing selected lines
- inserting or appending text
- stopping early after a match

A basic substitution uses `s/pattern/replacement/flags`:

```bash
sed 's/old/new/' file.txt
sed 's/old/new/g' file.txt
```

- Without `g`, `sed` replaces only the first match on each line.
- With `g`, it replaces all matches on the line.
- Use different delimiters when the pattern contains `/`, such as `s#/var/www#/srv/www#g`.

**Useful `sed` commands**

`sed` supports a small but powerful command set for administration tasks.

```bash
sed 'd' file.txt          # delete all lines
sed '2d' file.txt         # delete line 2
sed '1,5d' file.txt       # delete lines 1 through 5
sed -n '5p' file.txt      # print only line 5
sed 's/^/# /' file.txt    # comment every line
```

- `d` deletes matching lines from the output stream.
- `p` prints selected lines; it is most useful with `-n`.
- `-n` suppresses automatic printing so you control output explicitly.
- Anchors like `^` and `$` help match line starts and ends.

**Editing files in place**

GNU `sed` on RHEL commonly supports `-i` for in-place editing. This is useful for configuration files, but you should still be careful and make backups.

```bash
sed -i 's/SELINUX=enforcing/SELINUX=permissive/' /etc/selinux/config
sed -i.bak 's/^#Port 22/Port 2222/' /etc/ssh/sshd_config
```

- `-i` edits the file directly.
- `-i.bak` creates a backup file on GNU `sed` with the suffix `.bak`.
- Backup suffix behavior is implementation-dependent across `sed` variants, so do not assume identical behavior on every Unix.

> Be careful with in-place edits on critical files. Test the command on a copy first when possible.

**`sed` addressing**

Addresses tell `sed` which lines to act on.

```bash
sed -n '1,10p' file.txt
sed '/^#/d' file.txt
sed '/^$/d' file.txt
sed '/error/ s/^/ALERT: /' messages.log
```

- Numeric addresses target specific line numbers.
- Regex addresses target matching lines.
- Address ranges let you operate on blocks of text.
- You can combine an address with a command for selective edits.

**Pattern space and hold space**

`sed` works primarily with the *pattern space*, which is the current line being processed. The *hold space* is a secondary buffer you can use to store and retrieve text across commands.

Typical uses include:

- swapping lines
- building multi-line transformations
- preserving a previous line while processing the current one

Example of exchanging pattern and hold space:

```bash
sed -n '
1h
1!{
  x
  p
  x
}
$ p
' file.txt
```

This is advanced, but the key idea is simple: the pattern space holds the current working line, and the hold space stores temporary data between commands.

**Early exit and transliteration**

Two useful `sed` features for administration are `q`/`Q` and `y///`.

```bash
sed -n '/^UUID=/ { p; q; }' /etc/fstab
sed 'y/abc/ABC/' file.txt
```

- `q` quits after printing the current pattern space if printing is enabled.
- `Q` quits without printing the current pattern space.
- `y///` transliterates characters one-for-one, which is useful for simple character mapping.
- `y///` is not a regex replacement; it maps characters positionally.

**Practical `sed` examples**

Use `sed` when you need quick line-oriented edits.

```bash
# Remove comments and blank lines
sed '/^#/d; /^$/d' /etc/ssh/sshd_config

# Replace a port number
sed 's/^Port 22$/Port 2222/' /etc/ssh/sshd_config

# Print only lines containing a keyword
sed -n '/Failed/p' /var/log/secure

# Insert a line before a match
sed '/^PermitRootLogin/a AllowUsers admin' /etc/ssh/sshd_config
```

- `a` appends text after the matched line.
- `i` inserts text before the matched line.
- `c` changes the matched line entirely.
- These commands are useful for controlled config-file edits.

**`awk` basics**

`awk` is a pattern scanning and processing language. It is especially useful when output is column-like or record-based.

A simple `awk` program has the form:

```bash
awk 'pattern { action }' file
```

If you omit the pattern, the action runs for every record.

```bash
awk '{ print $1, $3 }' file.txt
```

- `$0` is the entire record.
- `$1`, `$2`, `$3`, and so on are fields.
- `print` writes output with the default output separator, usually a space.
- `awk` is excellent for summarizing command output.

**Field and record separators**

By default, `awk` splits records on newline and fields on runs of whitespace. You can change both behaviors.

```bash
awk -F: '{ print $1, $3 }' /etc/passwd
awk 'BEGIN { FS=":" } { print $1, $3 }' /etc/passwd
awk 'BEGIN { RS=""; FS="\n" } { print $1 }' file.txt
```

- `FS` is the field separator.
- `RS` is the record separator.
- `OFS` is the output field separator.
- `ORS` is the output record separator.

`RS` matters when records are not line-based. For example, setting `RS=""` makes blank-line-separated paragraphs become records, which is useful for some reports and multi-line blocks.

**`awk` arithmetic and reporting**

`awk` can do arithmetic directly, which makes it ideal for totals, averages, and counters.

```bash
awk '{ sum += $3 } END { print sum }' numbers.txt
awk '{ sum += $3; count++ } END { if (count > 0) print sum / count }' numbers.txt
awk '$3 > 1000 { print $1, $3 }' /etc/passwd
```

For `/etc/passwd`, use the correct field separator:

```bash
awk -F: '$3 > 1000 { print $1, $3 }' /etc/passwd
```

- `sum += $3` accumulates a running total.
- `count++` increments a counter.
- `END` runs after all input is processed.
- Always set `-F:` for colon-separated files like `/etc/passwd`.

**Counters, totals, and averages**

These are common exam-style tasks because they turn raw output into useful summaries.

```bash
# Count lines containing "error"
awk '/error/ { count++ } END { print count }' /var/log/messages

# Sum the second column
awk '{ total += $2 } END { print total }' data.txt

# Average the second column
awk '{ total += $2; n++ } END { if (n > 0) print total / n }' data.txt
```

- Use `END` for final reporting.
- Guard division by zero when calculating averages.
- Use `NR` for the current record number and `NF` for the number of fields in the current record.

**Associative arrays**

Associative arrays are one of `awk`’s most practical features. They let you count occurrences, group values, and build summaries by key.

```bash
awk '{ count[$1]++ } END { for (k in count) print k, count[k] }' access.log
```

Example: count logins by username from a simple file.

```bash
awk '{ users[$1]++ } END { for (u in users) print u, users[u] }' users.txt
```

- Array keys are strings.
- This is useful for frequency counts and grouping.
- Output order is not guaranteed unless you sort it afterward.

To sort the result:

```bash
awk '{ count[$1]++ } END { for (k in count) print k, count[k] }' access.log | sort
```

**Practical `awk` examples for administration**

Use `awk` to extract and summarize real admin data.

```bash
# Show usernames and shells from /etc/passwd
awk -F: '{ print $1, $7 }' /etc/passwd

# Show users with UID >= 1000
awk -F: '$3 >= 1000 { print $1, $3 }' /etc/passwd

# Summarize disk usage from machine-friendly output
df -P | awk 'NR==1 || $5 ~ /%/ { gsub(/%/, "", $5); if (NR==1 || $5+0 > 80) print $1, $5, $6 }'
```

For disk usage, machine-friendly output is safer than trying to parse human-oriented formatting. `df -P` gives a predictable layout, and stripping `%` explicitly is clearer than relying on implicit numeric coercion.

```bash
# Count failed SSH attempts in a log
awk '/Failed password/ { fail++ } END { print fail }' /var/log/secure
```

- Prefer predictable input formats when possible.
- Use `awk` for summaries, not just extraction.
- Sort or post-process output when you need stable ordering.

**`awk` pattern rules and control flow**

`awk` can run different actions for different patterns.

```bash
awk '
/^#/ { next }
/^$/ { next }
{ print }
' file.txt
```

- `next` skips to the next record.
- Pattern-action blocks can be stacked.
- `BEGIN` runs before input starts.
- `END` runs after input ends.

You can also use `if`, `else`, loops, and functions inside actions.

```bash
awk '
{
  if ($3 > 1000) {
    print $1, $3
  }
}
' /etc/passwd
```

**Quoting pitfalls in the shell**

Quoting is a common source of mistakes when you embed `sed` and `awk` in shell commands.

- Use single quotes around most `sed` and `awk` programs so the shell does not expand `$1`, `*`, or backslashes.
- Use double quotes only when you intentionally want shell variable expansion.
- Escape embedded single quotes carefully if you need them inside a single-quoted script.
- Remember that the shell, `sed`, and `awk` each have their own parsing rules.

Examples:

```bash
name="alice"
awk -v user="$name" '$1 == user { print $0 }' /etc/passwd
sed "s/PLACEHOLDER/$name/" file.txt
```

- `-v` passes shell variables into `awk` safely.
- Double quotes allow shell expansion, but they also expose you to unwanted interpretation if the replacement text contains special characters.
- For arbitrary replacement text, consider escaping the value first.

> Do not assume shell quoting and regex quoting are the same thing. They solve different parsing problems.

**Portability notes**

RHEL commonly uses GNU `sed` and GNU `awk` or `gawk`, but portability still matters when you write scripts.

- GNU `sed` supports `-i` and some extensions that may differ on other systems.
- GNU `awk` supports features that may not exist in older or smaller `awk` implementations.
- POSIX-compliant syntax is safer when you want scripts to run broadly.
- For exam tasks on RHEL, GNU behavior is usually what you see, but you should still write clear, conservative commands.

**Common mistakes to avoid**

- Using `sed` when `awk` is better for field-based summaries.
- Forgetting `-F:` for colon-separated files.
- Assuming `df -h` is easy to parse reliably in scripts.
- Overusing regular expressions when a simple field comparison is enough.
- Editing critical files in place without a backup.
- Forgetting that `awk` field numbers start at `$1`, not `$0`.
- Assuming output order from associative arrays is sorted.

**When to choose each tool**

| Task | Better tool | Why |
|---|---|---|
| Replace text in a config file | `sed` | Line-oriented substitution is simple and fast |
| Delete blank or comment lines | `sed` | Direct line filtering |
| Extract columns from structured output | `awk` | Field-aware processing |
| Count or total values | `awk` | Built-in arithmetic |
| Group values by key | `awk` | Associative arrays |
| Make a quick one-line edit | `sed` | Concise stream editing |

**Exam-oriented workflow**

When you face a text-processing task, work through it in a predictable order.

1. Identify the input format.
2. Decide whether the task is line-based or field-based.
3. Choose `sed` for edits or `awk` for summaries.
4. Test the command on sample output first.
5. Add sorting, filtering, or backups if needed.

Example workflow for a log summary:

```bash
grep 'Failed password' /var/log/secure | awk '{ count[$11]++ } END { for (u in count) print u, count[u] }' | sort
```

This kind of pipeline is useful because it narrows the data first, then summarizes it.

**Practice tasks**

Use these exercises to build exam-ready speed.

- Remove comments and blank lines from a configuration file with `sed`.
- Replace a service port in a config file and create a backup.
- Extract usernames and shells from `/etc/passwd` with `awk`.
- Count how many users have UID 1000 or greater.
- Summarize failed login attempts by username from a log file.
- Compute the average of a numeric column from command output.
- Convert lowercase text to uppercase with `sed y///`.
- Print only the first matching line from a file and stop processing early.

**Quick reference**

- `sed 's/old/new/g' file` replaces text globally on each line.
- `sed -n '/pattern/p' file` prints only matching lines.
- `sed -i.bak '...' file` edits in place and keeps a backup on GNU `sed`.
- `awk -F: '{ print $1, $3 }' /etc/passwd` extracts fields from colon-separated data.
- `awk '{ sum += $2 } END { print sum }' file` totals a numeric column.
- `awk '{ count[$1]++ } END { for (k in count) print k, count[k] }' file` counts occurrences by key.
- `RS` changes record splitting; `FS` changes field splitting.

**Summary**

`sed` is your fast tool for stream editing, while `awk` is your flexible tool for structured text analysis. For RHCSA tasks, focus on practical transformations: remove noise, extract fields, count values, and generate concise reports. If you understand line processing, field splitting, arithmetic, and associative arrays, you can solve most exam-style text-processing problems efficiently.
