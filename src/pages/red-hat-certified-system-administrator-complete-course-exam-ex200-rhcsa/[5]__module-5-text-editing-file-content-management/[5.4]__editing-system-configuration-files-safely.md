## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Why safe editing matters**

On RHEL, many tasks are completed by changing text-based configuration files. A small typo can prevent a service from starting, lock you out of remote access, or break boot-time behavior. Safe editing is not just about knowing `vi`; it is about making changes in a controlled way, validating them, and recovering quickly if something fails.

For RHCSA-level work, you should be comfortable editing files directly, preserving permissions and labels, and checking service behavior after every change. You also need to know when a safer tool exists, such as `visudo` for sudoers-related files.

**Core workflow for safe edits**

A reliable workflow reduces risk and makes troubleshooting easier:

1. Identify the exact file or drop-in directory that controls the setting.
2. Check the current state before editing.
3. Make a backup or copy with preserved metadata when appropriate.
4. Edit with a safe tool.
5. Validate syntax or service behavior.
6. Reload or restart only after validation.
7. Inspect logs if the service fails.

This workflow matters because many RHEL services use layered configuration. The effective setting may come from the main file, a file in `/etc/<service>.d/`, or a vendor default overridden by a drop-in. Do not assume the first file you open is the only place that matters.

**Finding the right configuration source**

Before editing, confirm where the setting is actually defined. On modern RHEL systems, configuration may be split across multiple files.

Common examples:

- `sshd` may read `/etc/ssh/sshd_config` and drop-ins under `/etc/ssh/sshd_config.d/`
- `sudo` uses `/etc/sudoers` and files in `/etc/sudoers.d/`
- `systemd` services often use unit drop-ins under `/etc/systemd/system/<unit>.d/`
- Network and firewall settings may be managed by dedicated tools rather than direct file edits

Useful commands:

```bash
systemctl cat sshd
systemctl cat firewalld
grep -R --line-number --include='*.conf' 'PermitRootLogin' /etc/ssh
grep -R --line-number --include='*' 'Defaults' /etc/sudoers /etc/sudoers.d
```

For SSH, remember that `sshd_config` can include other files. If you only inspect the main file, you may miss the effective setting.

**Using `vi` safely**

`vi` is the standard editor you should know for RHCSA. You do not need advanced customization, but you should be fast with basic navigation and edits.

Common actions:

- Open a file: `vi /etc/ssh/sshd_config`
- Search forward: `/PermitRootLogin`
- Search backward: `?PasswordAuthentication`
- Jump to a line: `:42`
- Delete a line: `dd`
- Copy a line: `yy`
- Paste below: `p`
- Replace a word under the cursor: `cw`
- Save and quit: `:wq`
- Quit without saving: `:q!`

A few practical examples:

```vim
:set number
/ListenAddress
dd
oListenAddress 192.0.2.10
Esc
:wq
```

```vim
:%s/^#PermitRootLogin.*/PermitRootLogin no/
:wq
```

If you are editing quickly under pressure, line numbers and search are often faster than scrolling. Use `:set number` temporarily if it helps you land on the right line.

**Safe editing habits**

Good habits prevent accidental damage:

- Make one change at a time when possible.
- Keep a terminal open with a second session for validation.
- Avoid editing the same file from multiple sessions at once.
- Save only after you have confirmed the change is correct.
- Prefer a temporary copy when testing a complex edit.

Concurrent edits can overwrite each other. If automation, another admin, or a config management tool may touch the file, check before you edit and coordinate changes.

**Backups, copies, and metadata**

Before changing an important file, create a backup. A simple copy is often enough for exam work, but be aware of metadata.

Examples:

```bash
cp -a /etc/ssh/sshd_config /root/sshd_config.bak
cp --preserve=mode,ownership,timestamps /etc/sudoers /root/sudoers.bak
```

`cp -a` is useful because it preserves many attributes, but it does not guarantee that the destination path has the correct SELinux context. If you replace or restore a file in `/etc`, verify labels and run `restorecon` when needed.

Extended attributes and ACLs can matter in some environments. If you need to preserve them during a copy, consider:

```bash
getfacl /etc/somefile
getfattr -d /etc/somefile
cp -a /etc/somefile /root/
```

If you restore a file into its original location, check whether permissions, ownership, ACLs, and SELinux labels still match expectations.

> If a file has the wrong SELinux label, the service may fail even when the text content is correct. Use `restorecon -v /path/to/file` to reset the label to the default for that path.

**Editing sudoers safely**

Never edit `/etc/sudoers` with a normal editor. Use `visudo`, which validates syntax before saving and helps prevent lockout.

Examples:

```bash
visudo
visudo -f /etc/sudoers.d/admins
```

A typical safe entry:

```sudoers
%wheel ALL=(ALL) ALL
```

A common drop-in file might look like:

```sudoers
alice ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart httpd
```

`visudo` is important because a syntax error in sudoers can break privilege escalation for all administrators. If you use `/etc/sudoers.d/`, keep file permissions restrictive and syntax valid.

**Working with service configuration files**

Many services require a restart or reload after editing. The exact action depends on the service and the type of change.

Examples:

```bash
systemctl restart sshd
systemctl reload httpd
systemctl restart firewalld
```

Use reload when the service supports it and the change does not require a full restart. If you are unsure, check the service documentation or use `systemctl status <unit>` after the change.

A practical SSH example:

```bash
cp -a /etc/ssh/sshd_config /root/sshd_config.bak
vi /etc/ssh/sshd_config
sshd -t
systemctl restart sshd
```

If the configuration includes drop-ins, validate the effective configuration rather than only the main file when possible.

**Validation before restart**

Validation catches syntax errors before they become outages. Many services provide a built-in test mode.

Examples:

```bash
sshd -t
nginx -t
httpd -t
visudo -c
systemd-analyze verify /etc/systemd/system/myapp.service
```

For SSH, `sshd -t` checks syntax, but it does not guarantee the service will behave as expected after restart. It is still a strong first check.

For systemd unit files, `systemd-analyze verify` can catch malformed directives and dependency issues. For sudoers, `visudo -c` is the standard syntax check.

**Troubleshooting after a failed edit**

If a service fails after a change, use a systematic workflow instead of guessing.

1. Check the service state:
   ```bash
   systemctl status sshd
   systemctl status httpd
   ```
2. Review recent logs:
   ```bash
   journalctl -u sshd -b --no-pager
   journalctl -u httpd -b --no-pager
   ```
3. Inspect service-specific logs if they exist:
   - `/var/log/secure` for authentication-related issues
   - `/var/log/messages` for general system messages
   - application-specific logs under `/var/log/`
4. Re-check the edited file for typos, missing quotes, or duplicate directives.
5. Confirm SELinux context and file permissions:
   ```bash
   ls -lZ /etc/ssh/sshd_config
   restorecon -v /etc/ssh/sshd_config
   ```

`systemctl status` gives a quick summary, but it may not show the full cause. `journalctl` often reveals the actual parser error, permission issue, or dependency failure. For SSH problems, the cause may also involve PAM, SELinux, or socket activation, so do not stop at the first log line you see.

**Common RHEL troubleshooting examples**

A few realistic scenarios help you recognize patterns:

- A typo in `sshd_config` prevents SSH from starting:
  - Fix the syntax
  - Run `sshd -t`
  - Restart `sshd`
- A sudoers change breaks privilege escalation:
  - Boot into a root shell or use an existing root session
  - Repair the file with `visudo`
  - Validate with `visudo -c`
- A copied config file works on content but fails due to labeling:
  - Check `ls -Z`
  - Run `restorecon`
- A service still uses old settings after editing:
  - Confirm whether it needs `reload` or `restart`
  - Check for drop-in files or included configs

**Handling permissions, ownership, and labels**

Configuration files usually need the correct owner, mode, and SELinux label. The exact mode depends on the file, but the important point is to preserve or restore the expected metadata.

Examples:

```bash
chown root:root /etc/ssh/sshd_config
chmod 600 /etc/ssh/sshd_config
restorecon -v /etc/ssh/sshd_config
```

Do not assume that a file is functional just because the text is correct. A wrong label or overly permissive mode can cause the service to refuse the file or create a security issue.

**Immutable files and rescue situations**

Sometimes a file cannot be edited because it is immutable or the system is in a restricted state.

Check for immutability:

```bash
lsattr /etc/ssh/sshd_config
```

If the immutable bit is set, remove it before editing:

```bash
chattr -i /etc/ssh/sshd_config
vi /etc/ssh/sshd_config
chattr +i /etc/ssh/sshd_config
```

In rescue or emergency mode, you may need to mount the root filesystem read-write before editing:

```bash
mount -o remount,rw /
vi /etc/fstab
```

This is useful when a boot issue requires a direct fix. After editing critical files such as `/etc/fstab`, validate carefully before rebooting.

**Editing with minimal risk in practice**

A good real-world pattern is to stage, validate, and then apply:

```bash
cp -a /etc/httpd/conf/httpd.conf /root/httpd.conf.bak
vi /etc/httpd/conf/httpd.conf
httpd -t
systemctl reload httpd
journalctl -u httpd -b --no-pager | tail -n 20
```

Another example for SSH hardening:

```bash
cp -a /etc/ssh/sshd_config /root/sshd_config.bak
vi /etc/ssh/sshd_config
sshd -t
systemctl restart sshd
```

If you are working remotely, be especially careful with SSH changes. Keep an existing session open until you confirm the new configuration works, so you do not lock yourself out.

**Quick reference for exam use**

- Use `vi` for direct file edits.
- Use `visudo` for sudoers files.
- Back up before changing important configs.
- Validate syntax before restarting services.
- Check `systemctl status` and `journalctl` after failures.
- Remember drop-in directories and included files.
- Restore SELinux labels with `restorecon` when needed.
- Preserve metadata when copying files, including ACLs and extended attributes where relevant.
- Avoid concurrent edits and coordinate with automation.

**Practical line-edit examples**

These are common quick edits you should be able to perform in `vi`:

```vim
/PermitRootLogin
cwPermitRootLogin no
:wq
```

```vim
/ListenAddress
oListenAddress 192.0.2.20
Esc
:wq
```

```vim
:%s/^#\?PasswordAuthentication.*/PasswordAuthentication no/
:wq
```

These patterns are useful because they let you make precise changes without rewriting the whole file. Always re-check the file after saving, especially when using substitutions.

**Key takeaway**

Safe configuration editing on RHEL is a process, not a single command. Find the real source of the setting, edit carefully, preserve metadata, validate syntax, and confirm the service behavior with logs and status checks. That discipline prevents outages and is exactly the kind of operational habit RHCSA expects.
