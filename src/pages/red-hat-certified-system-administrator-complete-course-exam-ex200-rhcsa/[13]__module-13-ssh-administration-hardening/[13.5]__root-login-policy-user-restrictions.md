## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Root access policy: what you control and why**

On RHEL-based systems, “root access” is not one setting. You usually control it through a combination of SSH policy, password state, `sudo`, PAM (Pluggable Authentication Modules), and local account restrictions. For RHCSA tasks, the goal is often to allow administration while reducing direct root logins.

The most important distinction is between:

- **Logging in as root**
- **Switching to root after logging in as a normal user**
- **Granting limited administrative rights with `sudo`**

These are not equivalent. A user may be blocked from direct root SSH login but still be able to run administrative commands with `sudo`, or use `su` if they know the root password.

**Root account behavior and password locking**

The root account is a special local account. You can disable direct password-based use of root without deleting the account. This is commonly done by locking the password hash.

Typical commands:

```bash
sudo passwd -l root
sudo passwd -u root
sudo passwd -S root
```

- `passwd -l root` locks the root password by prepending a lock marker in `/etc/shadow`.
- `passwd -u root` unlocks the password if a valid hash still exists.
- `passwd -S root` shows the password status.

Important nuance:

- A leading `!` in the password field commonly indicates a locked password.
- `!!` often appears for an account that has no usable password set yet.
- These markers are implementation details in `/etc/shadow`, not a universal account-disable mechanism.

> Locking the password does not necessarily disable every way to become root. If another path exists, such as `sudo` or console access with a different authentication method, root-level actions may still be possible.

**Direct root login versus `su` and `sudo`**

Direct root login means authenticating as `root` at a console, TTY, or SSH session. This is controlled by the login service and, for SSH, by `sshd_config`.

`su` and `su -` behave differently:

- `su` switches to another user, usually root, while keeping much of the current environment.
- `su -` starts a login shell for the target user and loads that user’s login environment.
- Both typically require the target user’s password, which for root means the root password unless PAM or policy changes the behavior.

`sudo` is different:

- You authenticate as your own user.
- `sudo` checks authorization rules in `/etc/sudoers` or `/etc/sudoers.d/*`.
- You can grant narrow privileges without sharing the root password.

Practical comparison:

```bash
su
su -
sudo id
sudo -l
```

Use `su -` when you need a root login shell with root’s environment. Use `sudo` when you want auditable, limited privilege escalation. For exam tasks, `sudo` is usually the safer and more controllable method.

**SSH root login policy**

SSH is the most common place where root login restrictions are enforced. The main setting is `PermitRootLogin` in `/etc/ssh/sshd_config`.

Common values:

- `yes` allows root login.
- `no` denies root login.
- `prohibit-password` allows root login only with public key authentication, not passwords.
- `without-password` is the older OpenSSH spelling you may still see in legacy materials; it is effectively the same idea as `prohibit-password` on many systems.
- `forced-commands-only` is a specialized mode for key-based automation.

Example:

```conf
# /etc/ssh/sshd_config
PermitRootLogin no
PasswordAuthentication yes
```

If you want root to be able to use SSH keys but not passwords, use:

```conf
PermitRootLogin prohibit-password
```

Older equivalent you may encounter:

```conf
PermitRootLogin without-password
```

After changing SSH policy, validate and reload:

```bash
sudo sshd -t
sudo systemctl reload sshd
```

If `sshd -t` reports no output, the configuration syntax is valid. Always test before reloading to avoid locking yourself out.

**Local console and TTY restrictions**

Not all restrictions are SSH-specific. You may need to control who can log in on a local console or virtual terminal. This is often handled through PAM and login service policy rather than SSH settings.

Useful places to inspect:

- `/etc/pam.d/login` for local console logins
- `/etc/pam.d/sshd` for SSH logins
- `/etc/pam.d/su` for `su`
- `/etc/pam.d/system-auth` and `/etc/pam.d/password-auth` for shared authentication behavior

A common pattern is to use PAM account rules to deny access based on user, group, origin, or time.

Example `pam_access.so` usage:

```conf
# /etc/pam.d/login
account required pam_access.so
```

Then define rules in `/etc/security/access.conf`:

```conf
- : root : tty1
- : alice : tty2
+ : ALL : ALL
```

This example denies `root` on `tty1` and denies `alice` on `tty2`, while allowing everyone else. The exact matching behavior depends on the PAM module and the source field format, so test carefully.

A more realistic use is to restrict a group from local login while still allowing SSH or `sudo`:

```conf
# /etc/security/access.conf
- : (restricted) : ALL
+ : ALL : ALL
```

This denies members of the `restricted` group from services that consult `pam_access.so`.

**PAM account lockout and denial mechanisms**

PAM can deny access at multiple stages:

- **auth** modules validate identity, such as passwords or MFA.
- **account** modules enforce policy, such as access rules, expiration, or time restrictions.
- **password** modules manage password changes.
- **session** modules apply session setup rules.

For user restrictions, the account phase is especially important. A user can have a valid password but still be denied by PAM account rules.

Common inspection points:

```bash
cat /etc/pam.d/sshd
cat /etc/pam.d/login
cat /etc/pam.d/su
grep -R "pam_access.so\|pam_faillock.so" /etc/pam.d /etc/security
```

`pam_faillock.so` is used on many RHEL systems to lock accounts after repeated failed logins. It is separate from `passwd -l` and from SSH policy.

Typical commands to inspect lockout state:

```bash
faillock --user alice
faillock --user root
```

Depending on system configuration, you may also see `pam_faillock` settings in `system-auth` and `password-auth`. The exact thresholds and unlock behavior are controlled there.

**How to tell whether an account is disabled, locked, or merely restricted**

This distinction matters in troubleshooting and exam scenarios.

Check the password status:

```bash
sudo passwd -S alice
sudo grep '^alice:' /etc/shadow
```

Interpretation guide:

- A locked password means password authentication is blocked, but other access paths may still exist.
- An expired account may be denied even if the password is correct.
- A shell set to `/sbin/nologin` or `/bin/false` prevents interactive login.
- A PAM rule may deny access only for certain services, hosts, or times.

Check the account shell:

```bash
getent passwd alice
```

Example output:

```text
alice:x:1001:1001:Alice User:/home/alice:/bin/bash
```

If the shell is `/sbin/nologin`, the user cannot get an interactive shell even if authentication succeeds.

Check account expiration:

```bash
chage -l alice
```

Useful indicators:

- Password expiration date
- Account expiration date
- Inactive period after password expiry

**Disabling a user account in practical terms**

There is no single universal “disable user” command that covers every scenario. In practice, you combine controls based on the goal.

Common approaches:

- Lock the password with `passwd -l user`
- Expire the account with `usermod -L` or `chage`
- Set the shell to `nologin`
- Remove the user from allowed groups
- Add PAM access restrictions
- Disable SSH password login for root or all users if policy requires it

Examples:

```bash
sudo passwd -l alice
sudo usermod -s /sbin/nologin alice
sudo chage -E 0 alice
```

These do different things:

- `passwd -l alice` blocks password-based authentication.
- `usermod -s /sbin/nologin alice` blocks interactive shell login.
- `chage -E 0 alice` expires the account immediately.

> Do not assume one control replaces another. A locked password does not necessarily stop key-based SSH, `sudo`, or service accounts from being used in other ways.

**Root password versus root login policy**

You can disable the root password itself while still allowing administrative access through other methods. This is a distinct control from SSH policy.

Examples:

- Lock root’s password to prevent password-based login.
- Set `PermitRootLogin no` to block SSH root login.
- Use `sudo` for administrative tasks.
- Keep console access available for emergency recovery if your environment requires it.

A common secure pattern is:

```conf
# /etc/ssh/sshd_config
PermitRootLogin no
PasswordAuthentication yes
```

Then grant admin rights to a named user:

```bash
sudo usermod -aG wheel adminuser
sudo visudo
```

In `/etc/sudoers`, ensure the wheel group is enabled:

```conf
%wheel ALL=(ALL) ALL
```

This lets `adminuser` authenticate as themselves and escalate with `sudo`, while root SSH login remains blocked.

**`sudo` and `su` policy differences**

`sudo` and `su` are often confused, but they solve different problems.

`sudo`:

- Uses the invoking user’s identity.
- Can be limited to specific commands.
- Produces better auditability.
- Does not require sharing the root password.

`su`:

- Requires the target account’s password unless PAM or policy changes it.
- Gives a shell as the target user.
- Is less granular than `sudo`.

Examples:

```bash
sudo -l
sudo systemctl restart sshd
su -
```

If `su` is restricted, check `/etc/pam.d/su` and group membership. Some systems allow only members of a privileged group, often `wheel`, to use `su` to root.

Example PAM restriction pattern:

```conf
# /etc/pam.d/su
auth required pam_wheel.so use_uid
```

This means only users in the wheel group can `su` to root, depending on the exact configuration.

**Practical troubleshooting workflow**

When a user cannot log in, determine which layer is blocking access.

1. Check whether the account exists.
   ```bash
   getent passwd alice
   ```
2. Check password status and expiration.
   ```bash
   passwd -S alice
   chage -l alice
   ```
3. Check the login shell.
   ```bash
   getent passwd alice
   ```
4. Check SSH policy if the failure is remote.
   ```bash
   sudo sshd -T | grep -i permitrootlogin
   sudo grep -E '^(PermitRootLogin|PasswordAuthentication|PubkeyAuthentication)' /etc/ssh/sshd_config
   ```
5. Check PAM rules for the service.
   ```bash
   sudo cat /etc/pam.d/sshd
   sudo cat /etc/pam.d/login
   ```
6. Review logs.
   ```bash
   sudo journalctl -u sshd
   sudo journalctl -t sshd
   sudo journalctl -t sudo
   sudo tail -n 50 /var/log/secure
   ```

Logging location depends on system configuration. On RHEL, authentication messages may appear in `journald`, `/var/log/secure`, or both.

**Verifying whether a user is truly disabled**

A user is truly disabled only if the relevant access path is blocked. Check all of these:

- Password status with `passwd -S`
- Account expiration with `chage -l`
- Shell with `getent passwd`
- Group membership with `id user`
- SSH policy with `sshd -T`
- PAM restrictions in `/etc/pam.d/*`
- `sudo` permissions with `sudo -l -U user`

Example:

```bash
id alice
passwd -S alice
chage -l alice
sudo -l -U alice
```

If `alice` has a locked password but can still run `sudo`, the account is not fully disabled. If `alice` cannot SSH but can log in locally, the restriction is service-specific, not global.

**Common exam-relevant controls versus advanced controls**

For RHCSA, prioritize the controls you are most likely to use in hands-on tasks.

Most likely in RHCSA tasks:

- `passwd -l` and `passwd -u`
- `chage`
- `usermod -s /sbin/nologin`
- `sudo` and `/etc/sudoers`
- `PermitRootLogin` in `sshd_config`
- Basic PAM awareness and log inspection

Less common or more advanced:

- Fine-grained `pam_access.so` rules
- `pam_faillock` tuning
- Complex `su` restrictions
- Time-based or host-based PAM policy
- Key-only root SSH access with `prohibit-password`

**Quick reference: root and user restriction commands**

```bash
# Lock and unlock passwords
passwd -l root
passwd -u root
passwd -l alice
passwd -u alice

# Inspect status
passwd -S root
chage -l alice
getent passwd alice
id alice

# SSH root policy
grep -E '^(PermitRootLogin|PasswordAuthentication|PubkeyAuthentication)' /etc/ssh/sshd_config
sshd -t
systemctl reload sshd

# PAM and access rules
cat /etc/pam.d/sshd
cat /etc/pam.d/login
cat /etc/security/access.conf

# sudo checks
visudo
sudo -l
sudo -l -U alice
```

**Exam strategy**

When you see a task about restricting root or user access, identify the exact login path first. Then choose the narrowest control that satisfies the requirement.

- Use `sshd_config` for SSH-specific root restrictions.
- Use `passwd -l` for password locking.
- Use `chage` for expiration-based disablement.
- Use `nologin` for interactive shell denial.
- Use PAM when the restriction must apply to a service, host, or class of users.
- Use `sudo` when the requirement is controlled administrative access rather than full root login.

The safest exam habit is to verify after every change. Test the intended login path, inspect logs, and confirm that you did not block your own administrative access.
