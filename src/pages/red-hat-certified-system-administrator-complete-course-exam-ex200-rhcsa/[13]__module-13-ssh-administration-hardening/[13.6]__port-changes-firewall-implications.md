**Port changes in Linux systems**

Changing a service port is rarely a single-step task. On RHEL-based systems, you usually need to coordinate three layers:

- The **service** must listen on the new port.
- **SELinux** must allow the service type to use that port.
- **firewalld** must permit traffic to reach it.

If any one layer is missed, the service may appear to be running but still be unreachable from the network. For RHCSA tasks, you should think in terms of end-to-end connectivity, not just configuration syntax.

**How the layers interact**

A port change often fails for reasons that are easy to confuse:

- The application is still bound to the old port.
- SELinux blocks the new port because it is not labeled for the service type.
- The firewall allows the port only in the wrong zone.
- The firewall change was made only at runtime and was not saved permanently.
- The interface is attached to a different zone than the one you edited.

A good troubleshooting habit is to verify each layer in order: service, SELinux, then firewall.

**Service configuration first**

Before changing SELinux or firewall rules, make sure the application is actually configured to listen on the new port.

Examples:

```bash
ss -tulpn
```

```bash
grep -R "Listen" /etc/httpd/ /etc/nginx/ 2>/dev/null
```

For Apache, a port change usually requires updating the listen directive and any virtual host definitions. For example:

```apache
Listen 8080
<VirtualHost *:8080>
    DocumentRoot /var/www/html
</VirtualHost>
```

For Nginx, you would update the `listen` directive in the server block:

```nginx
server {
    listen 8080;
    server_name example.com;
    root /usr/share/nginx/html;
}
```

After changing the application config, reload or restart the service as appropriate:

```bash
systemctl reload httpd
systemctl restart nginx
```

Use `reload` when the service supports it and the change is minor. Use `restart` when the service requires a full restart to pick up the new port.

**SELinux port labeling**

SELinux controls which service domains may bind to which ports. A port must be labeled with the correct SELinux type for the service to use it.

Check current labels with:

```bash
semanage port -l | grep -E 'http_port_t|ssh_port_t|smtp_port_t'
```

Common examples:

- `http_port_t` for web services
- `ssh_port_t` for SSH
- `smtp_port_t` for mail services

If a port is already assigned to the correct type, do not add it again. First check the existing mapping.

```bash
semanage port -l | grep ':8080'
```

If the port is already mapped to the desired type, no change is needed. If it is mapped to a different type, you usually need to modify the existing mapping rather than add a duplicate.

Use these actions carefully:

- `-a` adds a new port mapping when the port is not already defined for that SELinux type/protocol.
- `-m` modifies an existing mapping for a type/protocol pair.
- `-d` deletes a mapping when it is no longer needed.

Example workflow for a web service on port 8080:

```bash
semanage port -l | grep http_port_t | grep 8080
```

If `8080` is not listed for `http_port_t`, add it:

```bash
semanage port -a -t http_port_t -p tcp 8080
```

If `8080` is already defined for another type and you need to move it to `http_port_t`, first inspect the current mapping and then modify the existing definition if appropriate:

```bash
semanage port -m -t http_port_t -p tcp 8080
```

> Use `-m` only when the port/protocol mapping already exists and you are changing its SELinux type or adjusting an existing definition. If the port is not defined for that type/protocol, `-m` is not the right tool.

For SSH on a nonstandard port, verify whether the port is already labeled before adding it:

```bash
semanage port -l | grep ssh_port_t
```

If you need SSH on `2222/tcp` and it is not already present, add it:

```bash
semanage port -a -t ssh_port_t -p tcp 2222
```

If `2222/tcp` is already mapped to another type, resolve the conflict by inspecting the current label and then changing or removing the conflicting mapping as needed.

**Firewall basics: runtime vs permanent**

`firewalld` manages access by zone. A common exam mistake is to open a port in the wrong zone or to change only the runtime configuration.

Key ideas:

- **Runtime** changes take effect immediately but are lost after reload or reboot unless saved permanently.
- **Permanent** changes survive reboot but do not affect the running firewall until reloaded.
- `--runtime-to-permanent` copies the current runtime configuration into permanent storage.

Check the active zone and interface assignment first:

```bash
firewall-cmd --get-active-zones
```

Example output might show:

```text
public
  interfaces: enp1s0
internal
  interfaces: enp2s0
```

This matters because you must edit the zone that actually applies to the interface receiving traffic.

Inspect a zone:

```bash
firewall-cmd --zone=public --list-all
```

If multiple zones are active, identify which interface or source network is relevant before making changes. Do not assume `public` is the correct zone.

**Opening standard services vs custom ports**

Use `--add-service` for services that firewalld already knows by name, and `--add-port` for custom or nonstandard ports.

Examples:

```bash
firewall-cmd --zone=public --add-service=http
firewall-cmd --zone=public --add-port=8080/tcp
```

Use `--add-service` when the service definition matches the application’s standard port, such as `http`, `https`, `ssh`, or `cockpit`. Use `--add-port` when the application listens on a nondefault port or when no service definition exists.

To make the change permanent:

```bash
firewall-cmd --zone=public --add-port=8080/tcp --permanent
```

Then reload firewalld if you changed permanent settings:

```bash
firewall-cmd --reload
```

If you changed only runtime settings and want them saved, copy them to permanent storage:

```bash
firewall-cmd --runtime-to-permanent
```

**End-to-end example: web service moved to 8080**

A complete port change for a web server usually includes all three layers.

1. Update the application to listen on the new port.
2. Label the port for SELinux if needed.
3. Open the port in the correct firewalld zone.

Example for Apache:

```apache
Listen 8080
<VirtualHost *:8080>
    DocumentRoot /var/www/html
</VirtualHost>
```

Then adjust SELinux if `8080/tcp` is not already allowed for `http_port_t`:

```bash
semanage port -l | grep http_port_t | grep 8080
```

If needed:

```bash
semanage port -a -t http_port_t -p tcp 8080
```

Open the firewall in the correct zone:

```bash
firewall-cmd --get-active-zones
firewall-cmd --zone=public --add-port=8080/tcp --permanent
firewall-cmd --reload
```

Verify the service is listening:

```bash
ss -tulpn | grep 8080
```

Verify firewall access:

```bash
firewall-cmd --zone=public --list-ports
```

**Troubleshooting workflow**

When a connection fails after a port change, use a layered diagnostic approach.

1. Confirm the service is listening on the expected port.
   ```bash
   ss -tulpn | grep 8080
   systemctl status httpd
   ```
2. Confirm SELinux allows the port.
   ```bash
   semanage port -l | grep 8080
   getenforce
   ausearch -m AVC -ts recent
   ```
3. Confirm the firewall allows the port in the correct zone.
   ```bash
   firewall-cmd --get-active-zones
   firewall-cmd --zone=public --list-all
   ```
4. Test locally and remotely.
   ```bash
   curl -I http://localhost:8080
   curl -I http://server.example.com:8080
   ```
5. Review logs if the service still fails.
   ```bash
   journalctl -u httpd --since "10 minutes ago"
   ```

If SELinux is enforcing and you see AVC (Access Vector Cache) denials, do not disable SELinux as a first response. Fix the port label or the service context instead.

**Choosing the correct firewalld zone**

When multiple interfaces or sources are present, the correct zone depends on where the traffic enters the host.

Useful commands:

```bash
firewall-cmd --get-active-zones
firewall-cmd --get-default-zone
nmcli device status
ip addr show
```

Practical approach:

- Match the interface to its zone using `--get-active-zones`.
- If traffic comes from a specific network, check whether a source-based zone applies.
- If you are unsure, inspect the zone bound to the interface that owns the target IP address.

If needed, assign an interface to a zone:

```bash
firewall-cmd --zone=internal --change-interface=enp2s0 --permanent
firewall-cmd --reload
```

Be careful: changing the zone can affect other allowed services on that interface.

**Common command patterns**

```bash
# Show listening sockets
ss -tulpn

# List SELinux port mappings
semanage port -l

# Add a custom TCP port to a service type
semanage port -a -t http_port_t -p tcp 8080

# Open a custom port in the active zone
firewall-cmd --zone=public --add-port=8080/tcp --permanent

# Open a standard service in the active zone
firewall-cmd --zone=public --add-service=http --permanent

# Reload firewalld after permanent changes
firewall-cmd --reload

# Check active zones
firewall-cmd --get-active-zones
```

**Practical troubleshooting scenario**

Suppose users cannot reach a web app after you move it from port 80 to 8080.

Symptoms:

- `curl http://localhost:8080` works on the server.
- Remote clients time out.
- `ss -tulpn` shows the service listening on `8080`.
- SELinux is enforcing.

Likely causes and fixes:

- The firewall does not allow `8080/tcp` in the correct zone.
- The service is listening only on `127.0.0.1` instead of `0.0.0.0` or the server IP.
- SELinux does not allow the service type to bind to `8080`.

A clean fix sequence is:

```bash
ss -tulpn | grep 8080
semanage port -l | grep http_port_t | grep 8080
firewall-cmd --get-active-zones
firewall-cmd --zone=public --add-port=8080/tcp --permanent
firewall-cmd --reload
```

If SELinux still blocks access, add or correct the port label:

```bash
semanage port -a -t http_port_t -p tcp 8080
```

Then retest from another host:

```bash
curl -I http://server.example.com:8080
```

**Best practices for RHCSA tasks**

- Verify the service listens on the intended port before changing firewall rules.
- Check whether the port already exists in SELinux before using `semanage port -a`.
- Use `semanage port -m` only when modifying an existing mapping.
- Open standard services with `--add-service` and custom ports with `--add-port`.
- Confirm the active firewalld zone before making changes.
- Save permanent firewall changes when the task must survive reboot.
- Reload firewalld after permanent changes; do not reload unnecessarily if you only need a runtime test.
- Prefer targeted fixes over disabling SELinux or stopping the firewall.
