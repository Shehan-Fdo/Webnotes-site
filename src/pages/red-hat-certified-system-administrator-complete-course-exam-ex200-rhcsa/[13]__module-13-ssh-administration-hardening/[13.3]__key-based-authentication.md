## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

Key-based authentication lets you log in to SSH (Secure Shell) without typing a password each time. You authenticate with a private key on the client and a matching public key on the server. For RHCSA, you should know how to generate keys, install the public key on a remote system, verify permissions, and troubleshoot common failures.

This method is preferred over password authentication because it is more resistant to brute-force attacks, supports automation, and can be combined with passphrases for stronger protection. It is also the foundation for many administration workflows, including Ansible, Git access, and secure remote maintenance.

**How SSH key authentication works**

SSH key authentication uses asymmetric cryptography:

- The **private key** stays on the client and must be protected.
- The **public key** is copied to the server.
- During login, the server proves that the client holds the private key without receiving it.

The server checks the user’s authorized public keys, usually in `~/.ssh/authorized_keys`, and compares the offered key against the stored entries. If the key matches and the account is allowed to log in, SSH grants access.

A typical flow looks like this:

1. You generate a key pair on the client.
2. You copy the public key to the target account on the server.
3. You ensure the server-side permissions and SELinux context are correct.
4. You test login and confirm that SSH no longer prompts for a password.

**Generating SSH key pairs**

On RHEL, `ssh-keygen` is the standard tool for creating key pairs. The most common modern choice is `ed25519`, which is fast and secure for general use.

```bash
ssh-keygen -t ed25519 -C "admin@workstation"
```

If you need broader compatibility with older systems, you may use RSA with a strong key size:

```bash
ssh-keygen -t rsa -b 4096 -C "admin@workstation"
```

During generation, you are typically prompted for:

- The file location for the key pair, such as `~/.ssh/id_ed25519`
- A passphrase, which protects the private key if it is stolen
- Optional overwrite confirmation if the file already exists

Common key files:

- `~/.ssh/id_ed25519` for the private key
- `~/.ssh/id_ed25519.pub` for the public key
- `~/.ssh/id_rsa` and `~/.ssh/id_rsa.pub` for RSA

A passphrase is strongly recommended for interactive use. If you need repeated noninteractive access, you can use an SSH agent to cache the decrypted private key for the session.

**Deploying the public key to a remote account**

The safest exam-relevant approach is to create the remote `.ssh` directory and `authorized_keys` file explicitly, then set ownership and permissions correctly. Do not rely on a single append command to create everything in the right state.

Example manual deployment:

```bash
ssh user@server1 'mkdir -p ~/.ssh && touch ~/.ssh/authorized_keys && chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys'
cat ~/.ssh/id_ed25519.pub | ssh user@server1 'cat >> ~/.ssh/authorized_keys'
ssh user@server1 'chown -R user:user ~/.ssh'
```

A more controlled version creates the file first and then appends the key:

```bash
ssh user@server1 'umask 077; mkdir -p ~/.ssh; : > ~/.ssh/authorized_keys; chmod 700 ~/.ssh; chmod 600 ~/.ssh/authorized_keys'
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server1
```

`ssh-copy-id` is convenient because it appends the public key to the correct file and handles the common workflow for you. It still depends on the remote account being reachable with an initial authentication method, usually a password.

> Be careful with permissions and ownership. SSH may ignore keys if the home directory, `.ssh` directory, or `authorized_keys` file is writable by the wrong user or group.

**Required permissions and ownership**

SSH is strict about file and directory security. If permissions are too open, key authentication can fail even when the key itself is correct.

Typical safe settings:

- Home directory: owned by the user, not writable by others
- `~/.ssh`: mode `700`
- `~/.ssh/authorized_keys`: mode `600`
- Private key on the client: mode `600` or stricter

Example remediation on the server:

```bash
sudo chown user:user /home/user
sudo chmod 755 /home/user
sudo mkdir -p /home/user/.ssh
sudo chown -R user:user /home/user/.ssh
sudo chmod 700 /home/user/.ssh
sudo chmod 600 /home/user/.ssh/authorized_keys
```

If the home directory itself has incorrect ownership or permissions, fix that first. SSH checks the path from the home directory down to the key file, so a bad parent directory can break authentication even if `authorized_keys` is correct.

**SELinux considerations**

On RHEL, SELinux can also affect SSH key authentication if file labels are wrong. In normal cases, files created in a user’s home directory inherit the correct context, but copied or restored files may not.

Useful checks:

```bash
ls -ldZ /home/user /home/user/.ssh /home/user/.ssh/authorized_keys
restorecon -Rv /home/user/.ssh
```

If you move or restore home directories, `restorecon` is often the safest way to reset labels to the expected SELinux context.

**Server-side SSH configuration**

The SSH daemon reads `/etc/ssh/sshd_config` and related drop-in configuration files on newer systems. Key authentication is usually enabled by default, but you should know the relevant directives.

Common settings include:

```conf
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
PasswordAuthentication yes
KbdInteractiveAuthentication yes
```

Important notes:

- `AuthorizedKeysFile` may be overridden or expanded by system policy, so do not assume a single universal path on every RHEL installation.
- `PasswordAuthentication` controls password logins separately from key logins.
- `KbdInteractiveAuthentication` is the modern name for keyboard-interactive authentication on current OpenSSH versions.
- `ChallengeResponseAuthentication` is deprecated and may be removed or ignored on newer releases.

After changing SSH configuration, validate and reload the daemon:

```bash
sudo sshd -t
sudo systemctl reload sshd
```

Use `reload` when possible so you do not interrupt existing sessions. If validation fails, fix the configuration before reloading.

**Client-side SSH configuration**

You can simplify repeated connections with `~/.ssh/config`. This is useful when you manage multiple hosts or use different keys for different environments.

Example:

```conf
Host server1
    HostName 192.0.2.10
    User user
    IdentityFile ~/.ssh/id_ed25519
    IdentitiesOnly yes
```

Benefits:

- You avoid typing long command lines.
- You can select a specific key per host.
- `IdentitiesOnly yes` prevents SSH from offering too many keys and confusing authentication.

Test the connection with verbose output when troubleshooting:

```bash
ssh -v user@server1
```

The `-v` flag shows which keys are offered and why authentication succeeds or fails.

**Using ssh-agent**

An SSH agent stores decrypted private keys in memory for the current session. This is useful when your private key has a passphrase and you want to avoid retyping it repeatedly.

Typical workflow:

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
ssh-add -l
```

Notes:

- `ssh-agent` is a convenience tool, not a replacement for key protection.
- If the agent is not running or the key is not loaded, SSH may still prompt for the passphrase.
- On desktop environments, the agent may start automatically.

**Troubleshooting key-based authentication**

When key login fails, check the problem in layers: client key, server account, permissions, SELinux, and SSH configuration.

Common checks:

```bash
ssh -vvv user@server1
```

Look for messages about:

- Offered keys not being accepted
- Permission denied
- Bad ownership or modes on directories/files
- Missing or unreadable `authorized_keys`
- SELinux denials
- Disabled public key authentication

Server-side logs are often the fastest way to identify the issue:

```bash
sudo journalctl -u sshd
sudo journalctl -xe
```

On some systems, authentication messages may also appear in `/var/log/secure`.

Typical causes and fixes:

- **Wrong permissions**
  - Fix the home directory, `.ssh`, and `authorized_keys` modes.
- **Wrong ownership**
  - Ensure the user owns their home directory and `.ssh` content.
- **Incorrect key file**
  - Confirm the public key in `authorized_keys` matches the client’s private key.
- **SELinux label mismatch**
  - Run `restorecon -Rv /home/user/.ssh`.
- **SSH daemon settings**
  - Confirm `PubkeyAuthentication yes` and that the correct `AuthorizedKeysFile` path is in use.
- **Account restrictions**
  - Check shell access, account lock status, and any `AllowUsers` or `DenyUsers` rules.

**Practical recovery pattern**

If a user’s SSH key login is broken, repair the account carefully:

```bash
sudo chown user:user /home/user
sudo chmod 755 /home/user
sudo mkdir -p /home/user/.ssh
sudo chown user:user /home/user/.ssh
sudo chmod 700 /home/user/.ssh
sudo touch /home/user/.ssh/authorized_keys
sudo chown user:user /home/user/.ssh/authorized_keys
sudo chmod 600 /home/user/.ssh/authorized_keys
sudo restorecon -Rv /home/user/.ssh
```

Then re-add the public key:

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@server1 'cat >> ~/.ssh/authorized_keys'
```

If you want a more reliable one-step deployment, use `ssh-copy-id` after the directory and file exist with correct permissions.

**Disabling password authentication after verification**

Once key-based login works, you may choose to reduce attack surface by disabling password authentication for SSH. Do this only after confirming that you have a working key-based login and a recovery path.

Example:

```conf
PasswordAuthentication no
KbdInteractiveAuthentication no
PubkeyAuthentication yes
```

Then validate and reload:

```bash
sudo sshd -t
sudo systemctl reload sshd
```

> Never disable password or console access until you have verified that at least one administrative login method still works. Otherwise, you can lock yourself out.

**Real-world usage examples**

Interactive login with a specific key:

```bash
ssh -i ~/.ssh/id_ed25519 user@server1
```

Copy a key to a host:

```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server1
```

Check the loaded keys in your agent:

```bash
ssh-add -l
```

Inspect the public key fingerprint:

```bash
ssh-keygen -lf ~/.ssh/id_ed25519.pub
```

Compare the remote authorized keys file:

```bash
ssh user@server1 'cat ~/.ssh/authorized_keys'
```

**Exam focus points**

For RHCSA, be ready to:

- Generate an SSH key pair with `ssh-keygen`
- Install a public key on a remote account
- Set correct permissions on `.ssh` and `authorized_keys`
- Verify ownership of the home directory and SSH files
- Use `ssh -v` or `ssh -vvv` to troubleshoot
- Validate `sshd` configuration with `sshd -t`
- Reload `sshd` safely after changes
- Recognize the difference between password, keyboard-interactive, and public key authentication

**Quick reference**

| Item | Typical value |
|---|---|
| Private key | `~/.ssh/id_ed25519` |
| Public key | `~/.ssh/id_ed25519.pub` |
| Server key file | `~/.ssh/authorized_keys` |
| `.ssh` directory mode | `700` |
| `authorized_keys` mode | `600` |
| Home directory ownership | User-owned |
| SSH daemon config | `/etc/ssh/sshd_config` |

**Summary**

Key-based authentication is a core SSH skill and a frequent administration pattern on RHEL. The key ideas are simple: generate a key pair, place the public key in the correct server-side file, and ensure permissions, ownership, and SELinux labels are correct. Most failures come from file mode problems, wrong ownership, or configuration mismatches rather than from the cryptography itself.
