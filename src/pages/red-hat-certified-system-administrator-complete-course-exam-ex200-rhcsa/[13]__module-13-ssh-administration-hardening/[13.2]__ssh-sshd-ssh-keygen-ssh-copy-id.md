## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**SSH fundamentals**

<abbr title="Secure Shell">SSH</abbr> is the standard encrypted protocol for remote login and command execution on Linux systems. On RHEL, you typically use the `ssh` client to connect to a remote host and `sshd` to accept incoming connections on the server.

SSH protects traffic in transit by encrypting the session, which makes it the default choice for administration over untrusted networks. It also supports authentication with passwords or, preferably, public keys.

- `ssh` is the client program you run from your workstation or another server.
- `sshd` is the daemon that listens for connections on the managed host.
- `ssh-keygen` creates key pairs for public key authentication.
- `ssh-copy-id` installs your public key on a remote account.

A typical connection looks like this:

```bash
ssh user@server.example.com
```

You can specify a nondefault port with `-p`:

```bash
ssh -p 2222 user@server.example.com
```

You can also run a single remote command:

```bash
ssh user@server.example.com 'hostname -f'
```

**Client-side usage and common options**

The `ssh` client reads configuration from `/etc/ssh/ssh_config` for system defaults and `~/.ssh/config` for per-user settings. User-specific settings are useful when you connect to many systems and want to avoid repeating hostnames, usernames, ports, or identity files.

Common options include:

- `-i` to specify a private key file.
- `-l` to specify the login name.
- `-p` to specify the port.
- `-v`, `-vv`, and `-vvv` for increasing debug output.
- `-X` or `-Y` for X11 forwarding when enabled and needed.

Example `~/.ssh/config` entry:

```sshconfig
Host web1
    HostName web1.example.com
    User devops
    Port 22
    IdentityFile ~/.ssh/id_ed25519
```

With that file in place, you can connect with:

```bash
ssh web1
```

SSH also supports local and remote port forwarding, which is useful for tunneling services through an encrypted channel. For RHCSA-level administration, the most important skill is recognizing the syntax and knowing when a tunnel is appropriate.

```bash
ssh -L 8080:127.0.0.1:80 user@server.example.com
```

That command forwards local port `8080` to port `80` on the remote host from the remote host's perspective.

**Key-based authentication with ssh-keygen**

Public key authentication is the preferred method for routine administration because it avoids sending passwords over the network and can be combined with passphrases for stronger protection.

`ssh-keygen` creates a key pair:

- a **private key** that stays on your local system and must be protected
- a **public key** that you copy to remote systems

Generate an Ed25519 key pair, which is a modern default choice on current systems:

```bash
ssh-keygen -t ed25519 -C "admin@example.com"
```

If you need RSA for compatibility, you can generate it explicitly:

```bash
ssh-keygen -t rsa -b 4096 -C "admin@example.com"
```

During generation, you are prompted for:

- the file path, usually `~/.ssh/id_ed25519` or `~/.ssh/id_rsa`
- a passphrase, which is strongly recommended

Common key files:

- `~/.ssh/id_ed25519`
- `~/.ssh/id_ed25519.pub`
- `~/.ssh/id_rsa`
- `~/.ssh/id_rsa.pub`

You can inspect a public key with:

```bash
cat ~/.ssh/id_ed25519.pub
```

You can view the fingerprint of a key pair with:

```bash
ssh-keygen -lf ~/.ssh/id_ed25519.pub
```

If you need to change the passphrase on an existing private key, use:

```bash
ssh-keygen -p -f ~/.ssh/id_ed25519
```

> Keep private keys readable only by you. A private key should normally have permissions `600`, and the `~/.ssh` directory should normally be `700`.

**Installing keys with ssh-copy-id**

`ssh-copy-id` is the simplest way to place your public key into a remote account's `~/.ssh/authorized_keys` file. It appends the key if it is not already present and helps set up the correct directory and file permissions.

Basic usage:

```bash
ssh-copy-id user@server.example.com
```

If you use a specific key:

```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server.example.com
```

After installation, test passwordless login:

```bash
ssh user@server.example.com
```

What `ssh-copy-id` does conceptually:

- creates `~/.ssh` on the remote host if needed
- appends your public key to `~/.ssh/authorized_keys`
- preserves existing authorized keys
- relies on an initial authentication method, usually a password

A manual equivalent looks like this:

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@server.example.com 'umask 077; mkdir -p ~/.ssh; cat >> ~/.ssh/authorized_keys'
```

**Server-side configuration with sshd**

`sshd` is the OpenSSH server daemon. On RHEL, it is managed by `systemd` and usually runs as `sshd.service`.

Check status:

```bash
systemctl status sshd
```

Start and enable it:

```bash
sudo systemctl enable --now sshd
```

Restart after configuration changes:

```bash
sudo systemctl restart sshd
```

The main server configuration file is `/etc/ssh/sshd_config`. Common settings include:

- `Port` to change the listening port
- `PermitRootLogin` to control direct root access
- `PasswordAuthentication` to allow or deny password logins
- `PubkeyAuthentication` to enable public key authentication
- `AllowUsers` or `AllowGroups` to restrict who may log in
- `X11Forwarding` to permit X11 forwarding if required

Example hardened configuration:

```sshconfig
Port 22
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers alice bob
```

After editing `sshd_config`, validate the syntax before restarting:

```bash
sudo sshd -t
```

If the test succeeds, reload or restart the daemon:

```bash
sudo systemctl restart sshd
```

If you change the SSH port, remember to update firewall rules and SELinux policy as needed. On RHEL, a common firewall adjustment is:

```bash
sudo firewall-cmd --permanent --add-port=2222/tcp
sudo firewall-cmd --reload
```

**Authentication flow and authorized keys**

When a user connects with a key, `sshd` checks the user's home directory and the `~/.ssh/authorized_keys` file for a matching public key. The private key never leaves the client machine.

Important file and directory permissions:

- `~/.ssh` should usually be `700`
- `~/.ssh/authorized_keys` should usually be `600`
- the user's home directory must not be writable by others in a way that breaks SSH security checks

Example permission repair:

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```

If SSH key login fails, common causes include:

- wrong file permissions
- incorrect username
- wrong key file selected on the client
- `sshd_config` disables public key authentication
- SELinux or firewall issues after a port change
- the remote account has a shell or home directory problem

Use verbose client output to troubleshoot:

```bash
ssh -vvv user@server.example.com
```

**Practical administration workflow**

A common RHCSA workflow is to create a key pair, install the public key on the server, and then harden the server to prefer keys over passwords.

1. Generate a key pair on your workstation.
2. Copy the public key to the remote account.
3. Confirm passwordless login works.
4. Edit `/etc/ssh/sshd_config` to disable password authentication if appropriate.
5. Validate the configuration with `sshd -t`.
6. Restart `sshd` and test again in a second session before closing your current one.

Example sequence:

```bash
ssh-keygen -t ed25519 -C "alice@admin"
ssh-copy-id -i ~/.ssh/id_ed25519.pub alice@server.example.com
ssh alice@server.example.com
sudo sshd -t
sudo systemctl restart sshd
```

> Always keep one active session open when changing SSH settings. If you disable the wrong authentication method or lock yourself out, the open session may be your only recovery path.

**Useful files and directories**

- `/etc/ssh/sshd_config` — server configuration
- `/etc/ssh/ssh_config` — system-wide client configuration
- `~/.ssh/config` — per-user client configuration
- `~/.ssh/authorized_keys` — allowed public keys on the server
- `~/.ssh/id_ed25519` — private key
- `~/.ssh/id_ed25519.pub` — public key

**Troubleshooting checklist**

When SSH does not work, verify the problem in layers:

- network reachability with `ping` or `nc`
- service status with `systemctl status sshd`
- listening port with `ss -tlnp | grep sshd`
- server syntax with `sshd -t`
- firewall rules with `firewall-cmd --list-all`
- authentication details with `ssh -vvv`

Example port check:

```bash
ss -tlnp | grep sshd
```

Example firewall inspection:

```bash
sudo firewall-cmd --list-services
sudo firewall-cmd --list-ports
```

If SELinux is enforcing and you move SSH to a nondefault port, you may need to label the port appropriately rather than only opening the firewall. On RHEL, use `semanage port` for persistent SELinux port changes when required.

```bash
sudo semanage port -a -t ssh_port_t -p tcp 2222
```

**Exam-focused takeaways**

SSH is central to remote administration on RHEL, so you should be comfortable with both client and server tasks. For RHCSA, focus on the ability to:

- generate and manage SSH key pairs
- install keys with `ssh-copy-id`
- connect with `ssh` using common options
- configure `sshd` safely
- validate changes before restarting services
- troubleshoot permissions, ports, and authentication failures

The safest pattern is to test key-based access first, then harden the server gradually while preserving a recovery path.
