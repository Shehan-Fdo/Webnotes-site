## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**SSH overview**

<abbr title="Secure Shell">SSH</abbr> provides encrypted remote access, command execution, file transfer, and tunneling. On RHEL-based systems, the OpenSSH suite is the standard implementation. You use the client tools to connect outward and the server daemon (`sshd`) to accept inbound connections.

SSH matters in RHCSA because it is the default secure administration path. You should know how to connect, authenticate, transfer files, troubleshoot host key issues, and harden access with server-side policy.

- **Client tools** include `ssh`, `scp`, `sftp`, `ssh-agent`, and `ssh-add`.
- **Server daemon** is `sshd`, managed by `systemctl`.
- **Configuration files** live in `/etc/ssh/` for system-wide settings and `~/.ssh/` for per-user client settings.
- **Keys** are preferred over passwords for routine administration.

**Core SSH connection flow**

When you connect, the client first verifies the server identity using the server host key. If the host key is unknown, SSH prompts you to trust it and stores the fingerprint in `~/.ssh/known_hosts`. After that, the client authenticates the user with a password, public key, or another allowed method.

A typical connection looks like this:

```bash
ssh student@server1.example.com
```

Useful client options include:

```bash
ssh -p 2222 student@server1.example.com
ssh -i ~/.ssh/id_ed25519 student@server1.example.com
ssh -o StrictHostKeyChecking=accept-new student@server1.example.com
ssh -o UserKnownHostsFile=/tmp/known_hosts-test student@server1.example.com
```

- `-p` changes the TCP port.
- `-i` selects a private key file.
- `-o` sets a one-off client option.
- `StrictHostKeyChecking=accept-new` can reduce prompts in controlled environments, but use it carefully.

**Client-side configuration**

Per-user SSH client settings belong in `~/.ssh/config`. This file lets you define host aliases, default usernames, ports, identity files, and other options for specific systems or groups of systems.

Example:

```sshconfig
Host server1
    HostName server1.example.com
    User student
    Port 22
    IdentityFile ~/.ssh/id_ed25519

Host lab-*
    User admin
    ServerAliveInterval 30
    ServerAliveCountMax 3
```

Common client-side settings:

- **Host** defines a matching alias or pattern.
- **HostName** sets the real destination name or IP address.
- **User** sets the login name.
- **Port** sets a nondefault SSH port.
- **IdentityFile** points to a private key.
- **ServerAliveInterval** helps keep long sessions active.
- **ForwardAgent** enables agent forwarding for that host when needed.

Permissions matter for client files:

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/config
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
```

SSH may ignore insecure files, so verify ownership and permissions when authentication fails unexpectedly.

**Public key authentication**

Public key authentication is the most important SSH login method for exam work. You generate a key pair on the client, place the public key on the server, and keep the private key secret on the client.

Generate a modern key pair:

```bash
ssh-keygen -t ed25519 -C "student@server1"
```

If you need compatibility with older environments, RSA may still appear, but Ed25519 is the preferred modern choice when supported.

Copy the public key to the server:

```bash
ssh-copy-id student@server1.example.com
```

If `ssh-copy-id` is unavailable, you can install the key manually in a safe way:

```bash
cat ~/.ssh/id_ed25519.pub | ssh student@server1.example.com '
  umask 077 &&
  mkdir -p ~/.ssh &&
  touch ~/.ssh/authorized_keys &&
  chmod 700 ~/.ssh &&
  chmod 600 ~/.ssh/authorized_keys &&
  cat >> ~/.ssh/authorized_keys
'
```

- **`authorized_keys`** must exist and be writable only by the target user.
- **Home directory permissions** must not allow group or other write access.
- A home directory mode like `755` is commonly acceptable if ownership is correct and the directory is not writable by others.

> If SSH key login fails, check ownership and permissions first. A correct key file with wrong permissions is still rejected.

Verify the key-based login:

```bash
ssh -i ~/.ssh/id_ed25519 student@server1.example.com
```

**SSH agent usage**

`ssh-agent` stores decrypted private keys in memory so you do not retype passphrases repeatedly. `ssh-add` loads a key into the agent.

Example session:

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
ssh-add -l
ssh student@server1.example.com
```

- **ssh-agent** is useful for repeated logins during a work session.
- **ssh-add** loads keys into the agent.
- **ssh-add -l** lists loaded identities.

Agent forwarding allows a remote host to use your local agent for a second hop without copying private keys to the remote system.

```bash
ssh -A jumpbox.example.com
```

Use agent forwarding only when you trust the intermediate host, because it extends your authentication capability to that session.

**Server-side SSH configuration**

The SSH server daemon is `sshd`. Its main configuration file is `/etc/ssh/sshd_config`, and additional drop-in files may exist under `/etc/ssh/sshd_config.d/` on current RHEL releases.

Check the active configuration:

```bash
sshd -t
sshd -T | less
```

- **`sshd -t`** validates syntax.
- **`sshd -T`** prints the effective configuration after defaults and includes.

Common server-side directives:

```sshconfig
Port 22
PermitRootLogin no
PasswordAuthentication yes
PubkeyAuthentication yes
AllowUsers student admin
AllowGroups wheel sshadmins
DenyUsers testuser
DenyGroups blocked
```

Notes on current OpenSSH behavior:

- **PasswordAuthentication** controls password logins.
- **KbdInteractiveAuthentication** is the current directive name for keyboard-interactive authentication on modern OpenSSH releases.
- **ChallengeResponseAuthentication** is older and may be absent or deprecated on newer systems.
- **PermitRootLogin** is often set to `no` in hardened environments.
- **AllowUsers** and **AllowGroups** are useful for restricting who may connect.
- **DenyUsers** and **DenyGroups** provide explicit blocks.

A practical hardening example:

```sshconfig
PermitRootLogin no
PasswordAuthentication no
KbdInteractiveAuthentication no
PubkeyAuthentication yes
AllowUsers student
```

> If you disable password and keyboard-interactive authentication, confirm that key-based login works before ending your session.

Apply configuration changes carefully:

```bash
sudo sshd -t
sudo systemctl reload sshd
```

Use `reload` only for changes that `sshd` can re-read safely. Some changes require a full restart:

```bash
sudo systemctl restart sshd
```

A restart is often the safer choice after major authentication or port changes, especially during lab work where you need predictable behavior.

**Firewall and SELinux considerations**

SSH usually listens on TCP port 22. If you change the port, you must update the firewall and SELinux policy.

Check listening ports:

```bash
ss -tlnp | grep sshd
```

Allow the default service through firewalld:

```bash
sudo firewall-cmd --add-service=ssh
sudo firewall-cmd --permanent --add-service=ssh
```

If you move SSH to port 2222, open that port:

```bash
sudo firewall-cmd --permanent --add-port=2222/tcp
sudo firewall-cmd --reload
```

SELinux can also block nondefault SSH ports. Label the port with the SSH type:

```bash
sudo semanage port -l | grep ssh
sudo semanage port -a -t ssh_port_t -p tcp 2222
```

If the port already exists in the policy, modify it instead:

```bash
sudo semanage port -m -t ssh_port_t -p tcp 2222
```

- **`-a`** adds a new mapping.
- **`-m`** modifies an existing mapping.
- The default port 22 is typically already labeled `ssh_port_t`.

**Host keys and known_hosts**

The server presents a host key to prove its identity. The client stores the fingerprint in `~/.ssh/known_hosts`. If the host key changes unexpectedly, SSH warns you about a possible man-in-the-middle attack.

Common causes of host key warnings:

- The server was rebuilt or reinstalled.
- The SSH host keys were regenerated.
- The server IP now points to a different machine.
- DNS or load balancer routing changed.

Inspect the stored entry:

```bash
ssh-keygen -F server1.example.com
```

Remove an old entry:

```bash
ssh-keygen -R server1.example.com
ssh-keygen -R 192.0.2.10
```

If a server is rebuilt and missing host keys, regenerate them on the server:

```bash
sudo ssh-keygen -A
sudo systemctl restart sshd
```

- **`ssh-keygen -A`** creates any missing default host keys.
- This is useful after image rebuilds or incomplete installations.

**File transfer with scp and sftp**

`scp` copies files over SSH. `sftp` provides an interactive file-transfer session with commands similar to FTP but encrypted.

Examples:

```bash
scp report.txt student@server1.example.com:/tmp/
scp -r project/ student@server1.example.com:/home/student/
scp student@server1.example.com:/var/log/messages .
```

Interactive `sftp`:

```bash
sftp student@server1.example.com
```

Inside `sftp`:

```text
pwd
lpwd
ls
lls
cd /tmp
lcd /home/student/downloads
get report.txt
put backup.tar.gz
bye
```

- **scp** is convenient for quick copies.
- **sftp** is better when you need an interactive remote file browser.
- Use `-P` with `scp` for a nondefault port, but `-p` with `sftp` is a different option set, so check the command syntax carefully.

**Tunneling and port forwarding**

SSH can forward ports securely. This is useful for accessing services that are only reachable from a remote network or for testing local services through an encrypted channel.

Local forwarding sends a local port to a remote destination:

```bash
ssh -L 8080:127.0.0.1:80 student@server1.example.com
```

Remote forwarding exposes a remote port back to your local machine:

```bash
ssh -R 9090:127.0.0.1:3000 student@server1.example.com
```

Dynamic forwarding creates a SOCKS proxy:

```bash
ssh -D 1080 student@server1.example.com
```

- **-L** is common for reaching internal web apps.
- **-R** is useful for reverse access during troubleshooting.
- **-D** creates a proxy for applications that support SOCKS.

**Troubleshooting workflow**

Start with the simplest checks and move outward from the client to the server.

1. Confirm name resolution and reachability.
   ```bash
   ping -c 1 server1.example.com
   getent hosts server1.example.com
   ```
2. Verify the SSH service is running on the server.
   ```bash
   sudo systemctl status sshd
   sudo ss -tlnp | grep sshd
   ```
3. Check the server configuration syntax.
   ```bash
   sudo sshd -t
   ```
4. Review firewall and SELinux settings.
   ```bash
   sudo firewall-cmd --list-all
   sudo semanage port -l | grep ssh
   ```
5. Inspect client-side debug output.
   ```bash
   ssh -vvv student@server1.example.com
   ```
6. Check authentication files and permissions.
   ```bash
   ls -ld ~ ~/.ssh ~/.ssh/authorized_keys
   ls -l ~/.ssh
   ```

Useful debug clues:

- **Permission denied (publickey)** usually means the key is missing, wrong, or unreadable.
- **Host key verification failed** usually means the stored host key does not match the server.
- **Connection refused** often means `sshd` is not listening or a firewall blocks the port.
- **Connection timed out** often means routing, firewall, or DNS problems.

**Practical server access control patterns**

Use access-control directives to reduce exposure without breaking administration workflows.

Example: allow only a small admin group:

```sshconfig
AllowGroups sshadmins
PasswordAuthentication no
PubkeyAuthentication yes
```

Example: block a temporary account:

```sshconfig
DenyUsers tempuser
```

Example: permit only specific users:

```sshconfig
AllowUsers student root
```

These controls are evaluated by `sshd` before or during authentication, so they are effective for limiting who can attempt a login.

**Exam-oriented reminders**

- Know the difference between client configuration and server configuration.
- Know where to place public keys: `~/.ssh/authorized_keys` on the server side.
- Know how to validate `sshd_config` before restarting the service.
- Know how to recover from host key mismatches safely.
- Know how to open a nondefault SSH port in both firewalld and SELinux.
- Know the current directive name `KbdInteractiveAuthentication` instead of relying on older terminology.
- Know that not every `sshd` change is safely applied with reload.

**Quick reference**

```bash
# Client
ssh student@server1.example.com
ssh -i ~/.ssh/id_ed25519 student@server1.example.com
ssh-copy-id student@server1.example.com
scp file.txt student@server1.example.com:/tmp/
sftp student@server1.example.com

# Agent
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# Server
sudo sshd -t
sudo systemctl restart sshd
sudo firewall-cmd --permanent --add-service=ssh
sudo semanage port -a -t ssh_port_t -p tcp 2222
sudo ssh-keygen -A
```
