**SSH host key verification**

When you connect to a remote system with <abbr title="Secure Shell">SSH</abbr>, the client verifies the server’s identity using the server’s host key. This protects you from man-in-the-middle attacks by ensuring you are talking to the same host you connected to before.

The first time you connect, SSH usually uses <mark>trust on first use</mark> (TOFU): it asks you to accept the host key and stores it in `~/.ssh/known_hosts`. On later connections, the client compares the presented key to the stored key.

- The host key belongs to the **server**, not the user.
- The client stores trusted keys in `~/.ssh/known_hosts` for a regular user and `/etc/ssh/ssh_known_hosts` for system-wide trust.
- A mismatch warning means the server key changed or the connection may be under attack.

Common host key algorithms include `ed25519`, `ecdsa`, and `rsa`. Modern systems often prefer `ed25519` because it is fast and compact, but you may still encounter RSA on older systems.

**How verification works**

SSH checks the server’s public host key against the local known-hosts database. If the key matches, the connection proceeds silently. If it does not match, SSH stops and warns you.

Typical warning patterns include:

```text
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@    WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!     @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
```

This warning is important because it can indicate:

- The server was rebuilt or reinstalled.
- The SSH host keys were regenerated.
- DNS or IP address now points to a different machine.
- A potential <abbr title="man-in-the-middle">MITM</abbr> attack is occurring.

**Known hosts file behavior**

The `known_hosts` file stores hostnames, IP addresses, and the corresponding public host keys. It may also contain hashed hostnames if hashing is enabled.

Useful locations:

- `~/.ssh/known_hosts` for the current user
- `/etc/ssh/ssh_known_hosts` for all users
- `/etc/ssh/ssh_known_hosts2` on some older systems, though it is rarely used now

You can inspect entries with:

```bash
ssh-keygen -F server1.example.com
ssh-keygen -F 192.0.2.10
```

If you need to remove a stale entry, use:

```bash
ssh-keygen -R server1.example.com
ssh-keygen -R 192.0.2.10
```

This is safer than manually editing the file because it updates the correct hashed or plain-text entry.

**First connection and acceptance**

On first contact, SSH prompts you to confirm the fingerprint. The fingerprint is a short representation of the host key, used for human verification.

Example prompt:

```text
The authenticity of host 'server1.example.com (192.0.2.10)' can't be established.
ED25519 key fingerprint is SHA256:abc123...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

Best practice is to verify the fingerprint out-of-band when possible, such as through console access, a trusted admin, or documented provisioning output.

**Troubleshooting a changed host key**

If the host key changed unexpectedly, do not blindly accept the new key. First determine whether the change is legitimate.

1. Confirm the server identity through a trusted channel.
2. Check whether the system was reinstalled or SSH keys were regenerated.
3. Remove the stale entry from `known_hosts`.
4. Reconnect and verify the new fingerprint.

Example workflow:

```bash
ssh-keygen -F server1.example.com
ssh-keygen -R server1.example.com
ssh user@server1.example.com
```

If you manage the server, you can inspect its current host keys:

```bash
sudo ls -l /etc/ssh/ssh_host_*
sudo ssh-keygen -lf /etc/ssh/ssh_host_ed25519_key.pub
sudo ssh-keygen -lf /etc/ssh/ssh_host_rsa_key.pub
```

The `-l` option prints the fingerprint so you can compare it with what the client expects.

**Regenerating server host keys**

If the server’s host keys are missing or corrupted, SSH may fail to start or may generate new keys on first boot depending on the distribution and configuration. On Red Hat-based systems, host keys are typically managed under `/etc/ssh/`.

If regeneration is required, you may use:

```bash
sudo ssh-keygen -A
sudo systemctl restart sshd
```

`ssh-keygen -A` generates all missing host keys for the system. Use it carefully, because new keys will trigger host key mismatch warnings for existing clients.

**Client-side troubleshooting commands**

When SSH fails, increase verbosity to see what is happening during host key verification:

```bash
ssh -v user@server1.example.com
ssh -vv user@server1.example.com
ssh -vvv user@server1.example.com
```

The extra output helps you identify:

- Which key type the client is checking
- Which `known_hosts` file is being used
- Whether the mismatch is due to a changed key, a hostname mismatch, or a DNS issue

You can also test name resolution and connectivity:

```bash
getent hosts server1.example.com
ping -c 1 server1.example.com
nc -vz server1.example.com 22
```

If DNS resolves to the wrong address, SSH may appear to have a host key problem when the real issue is name resolution.

**Server-side troubleshooting commands**

On the server, verify that the SSH daemon is running and listening:

```bash
sudo systemctl status sshd
sudo ss -tlnp | grep :22
```

Check logs for authentication or key-related issues:

```bash
sudo journalctl -u sshd
sudo journalctl -xe
```

If SELinux or firewall rules block access, the connection may fail before host key verification completes. Confirm the service is reachable:

```bash
sudo firewall-cmd --list-services
sudo firewall-cmd --add-service=ssh --permanent
sudo firewall-cmd --reload
```

**Host key fingerprints and verification**

Fingerprints let you compare a host key without reading the full key. Common fingerprint formats include SHA-256 and older MD5 output.

Examples:

```bash
ssh-keygen -lf /etc/ssh/ssh_host_ed25519_key.pub
ssh-keygen -E sha256 -lf /etc/ssh/ssh_host_rsa_key.pub
```

If you need to see the raw public host key, use:

```bash
sudo cat /etc/ssh/ssh_host_ed25519_key.pub
```

Use fingerprints for verification, not the private key files. Private host keys must remain protected and readable only by root.

**Managing known_hosts entries**

The `known_hosts` file may contain multiple entries for the same host if the hostname and IP address were both used, or if the host key changed over time. This can create confusing warnings.

Useful commands:

```bash
ssh-keygen -F server1.example.com
ssh-keygen -F 192.0.2.10
ssh-keygen -R server1.example.com
```

If you need to inspect the file directly:

```bash
cat ~/.ssh/known_hosts
```

Be careful when editing manually, especially if the file uses hashed hostnames. `ssh-keygen -R` is the preferred method.

**Practical workflow for a mismatch warning**

When SSH reports a changed host key, use a disciplined process:

1. Stop and treat the warning as potentially serious.
2. Verify whether the server was rebuilt, replaced, or readdressed.
3. Compare the server’s current fingerprint with a trusted source.
4. Remove the stale entry only after confirmation.
5. Reconnect and confirm the new key is accepted.

Example:

```bash
ssh-keygen -F server1.example.com
ssh-keygen -R server1.example.com
ssh user@server1.example.com
```

If the host is managed by automation, update the provisioning source so future deployments publish the correct host key information.

**Common causes of host key problems**

- The server was reinstalled and generated new SSH host keys.
- The IP address now points to a different machine.
- A load balancer or NAT device changed the backend target unexpectedly.
- The `known_hosts` file contains stale or duplicated entries.
- DNS is resolving to the wrong host.
- The server’s SSH configuration changed key algorithms.

**Security considerations**

Host key verification is a core SSH defense. Disabling it weakens security and should be avoided except in controlled, temporary testing scenarios.

> Avoid using `StrictHostKeyChecking=no` as a routine workaround. It suppresses verification and can hide a real attack or misconfiguration.

If you must automate first-time connections, prefer a controlled provisioning process that preloads trusted host keys rather than blindly accepting unknown ones.

**Useful SSH client options**

Some options affect host key behavior:

```bash
ssh -o StrictHostKeyChecking=yes user@server1.example.com
ssh -o UserKnownHostsFile=/tmp/test_known_hosts user@server1.example.com
ssh -o GlobalKnownHostsFile=/etc/ssh/ssh_known_hosts user@server1.example.com
```

These are useful for testing, automation, or isolating a problem from your normal user profile.

**Example troubleshooting checklist**

- Confirm the hostname and IP address are correct.
- Check whether the server was rebuilt or rekeyed.
- Inspect `~/.ssh/known_hosts` for stale entries.
- Compare fingerprints with a trusted source.
- Use `ssh -vvv` to see detailed client-side diagnostics.
- Verify `sshd` is running and reachable on port `22`.
- Check firewall and DNS if the connection target seems wrong.

**Key takeaways**

Host key verification protects you from impersonation and MITM attacks. Most problems come from legitimate server changes, stale client records, or incorrect name resolution. The safest fix is to verify the server identity first, then remove the outdated known-hosts entry and reconnect.
