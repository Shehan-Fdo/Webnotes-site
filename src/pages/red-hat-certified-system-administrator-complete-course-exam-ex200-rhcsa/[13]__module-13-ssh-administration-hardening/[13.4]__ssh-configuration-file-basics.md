## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**SSH configuration overview**

<abbr title="Secure Shell">SSH</abbr> is the standard remote administration protocol on Linux. It provides encrypted access for shell sessions, file transfer, and tunneling. On Red Hat Enterprise Linux, you usually work with two configuration layers:

- **Client configuration** controls how your system connects to remote hosts.
- **Server configuration** controls how the SSH daemon accepts and secures incoming connections.

The main files are:

- `/etc/ssh/ssh_config` for system-wide client defaults
- `~/.ssh/config` for per-user client overrides
- `/etc/ssh/sshd_config` for server behavior
- `/etc/ssh/ssh_host_*` for host keys used by the server

Understanding where a setting belongs matters because SSH applies configuration in a specific order. User-level client settings override system-wide client settings, and command-line options override both. On the server side, `sshd` reads its daemon configuration and then enforces authentication, access, and session rules.

**Client-side configuration basics**

The SSH client reads settings from multiple places. This lets you define defaults for all connections and then specialize behavior for individual hosts.

Common client options include:

- `Host` to match one or more remote systems
- `HostName` to define the real destination name or IP address
- `User` to set the login name
- `Port` to use a nondefault port
- `IdentityFile` to specify a private key
- `ForwardAgent` to control agent forwarding
- `StrictHostKeyChecking` to control host key verification behavior

A simple per-user client configuration looks like this:

```sshconfig
Host web1
    HostName 192.0.2.10
    User admin
    Port 22
    IdentityFile ~/.ssh/id_ed25519

Host backup
    HostName backup.example.com
    User backupuser
    Port 2222
```

With this file in place, you can connect using:

```bash
ssh web1
ssh backup
```

This is useful when you want short host aliases, consistent usernames, or a nonstandard port without typing long command lines every time.

**Server-side configuration basics**

The SSH server daemon is `sshd`. Its main configuration file is `/etc/ssh/sshd_config`. This file controls how the server listens, authenticates users, and restricts access.

Common server directives include:

- `Port` to change the listening port
- `PermitRootLogin` to allow or deny direct root login
- `PasswordAuthentication` to enable or disable password logins
- `PubkeyAuthentication` to allow public key authentication
- `AllowUsers` to restrict which users may log in
- `AllowGroups` to restrict which groups may log in
- `X11Forwarding` to permit or deny X11 forwarding
- `ClientAliveInterval` and `ClientAliveCountMax` to manage idle sessions

A minimal hardening-oriented example:

```sshconfig
Port 22
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers alice bob
X11Forwarding no
ClientAliveInterval 300
ClientAliveCountMax 2
```

This configuration reduces attack surface by disabling password logins and direct root access. It also limits access to named users and disconnects stale sessions more aggressively.

> Be careful when disabling password authentication. If you do not have a working key-based login before reloading `sshd`, you can lock yourself out.

**How SSH configuration precedence works**

SSH does not treat all settings equally. It evaluates configuration in layers, and the most specific applicable setting wins.

For the client:

1. Command-line options have the highest priority.
2. `~/.ssh/config` applies next.
3. `/etc/ssh/ssh_config` provides system defaults.

For the server:

1. `sshd` reads `/etc/ssh/sshd_config`.
2. Match-based blocks can override settings for specific users, groups, or addresses.
3. Runtime behavior still depends on authentication method, keys, and file permissions.

This means you should place broad defaults in system files and use user-specific or host-specific overrides only when needed.

**Useful client commands**

You often need to test connectivity, inspect verbose negotiation, or verify which configuration SSH is using.

```bash
ssh -v user@server.example.com
ssh -vvv user@server.example.com
ssh -p 2222 user@server.example.com
ssh -i ~/.ssh/id_ed25519 user@server.example.com
```

Useful diagnostic commands include:

- `ssh -v` for basic debug output
- `ssh -vvv` for detailed protocol negotiation
- `ssh -G host` to print the final effective client configuration for a host
- `ssh -T host` to test without requesting a pseudo-terminal

Example:

```bash
ssh -G web1 | less
```

This is helpful when you want to confirm which `User`, `Port`, or `IdentityFile` SSH will actually use after all config files are processed.

**Editing and validating server configuration**

When you change `sshd_config`, validate before restarting or reloading the service. This avoids service outages caused by syntax errors.

Typical workflow:

1. Edit `/etc/ssh/sshd_config`.
2. Validate the file.
3. Reload or restart the daemon.
4. Test access from another session.

Commands:

```bash
sudo sshd -t
sudo systemctl reload sshd
sudo systemctl restart sshd
sudo systemctl status sshd
```

Notes:

- `sshd -t` checks syntax and reports errors.
- `reload` is preferred when the daemon supports it and you only changed configuration.
- `restart` is more disruptive but ensures the service fully reinitializes.

**Host keys and known hosts**

SSH uses cryptographic host keys to identify servers. The server stores its host keys under `/etc/ssh/`, commonly as:

- `/etc/ssh/ssh_host_ed25519_key`
- `/etc/ssh/ssh_host_rsa_key`
- corresponding `.pub` public key files

Clients store trusted server fingerprints in `~/.ssh/known_hosts`. This protects you from man-in-the-middle attacks by warning when a server identity changes unexpectedly.

If a host key changes legitimately, you may need to remove the old entry:

```bash
ssh-keygen -R server.example.com
ssh-keygen -R 192.0.2.10
```

Then reconnect and verify the new fingerprint before accepting it.

**Authentication methods**

SSH supports multiple authentication methods, but the most common on RHCSA systems are password and public key authentication.

Public key authentication uses a key pair:

- **Private key** stays on the client and must be protected.
- **Public key** is copied to the server, usually in `~/.ssh/authorized_keys`.

Typical setup:

```bash
ssh-keygen -t ed25519
ssh-copy-id user@server.example.com
```

If `ssh-copy-id` is unavailable, you can append the public key manually:

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@server.example.com 'mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys'
```

Key permissions matter:

- `~/.ssh` should usually be `700`
- `~/.ssh/authorized_keys` should usually be `600`
- private keys should be readable only by the owner

If permissions are too open, `sshd` may refuse the key.

**Common security-related directives**

Several `sshd_config` settings are especially relevant for exam prep and real administration.

- `PermitRootLogin no` prevents direct root SSH logins.
- `PasswordAuthentication no` forces key-based access when combined with public key setup.
- `PubkeyAuthentication yes` enables key authentication.
- `AllowUsers` limits login to specific accounts.
- `AllowGroups` limits login to members of specific groups.
- `MaxAuthTries` reduces brute-force attempts.
- `LoginGraceTime` limits how long a client can take to authenticate.
- `ClientAliveInterval` and `ClientAliveCountMax` help disconnect idle sessions.

Example access control block:

```sshconfig
AllowUsers alice@192.0.2.* bob
AllowGroups wheel sshadmins
```

Use these controls carefully. Overly restrictive rules can block legitimate administrative access, especially if you rely on multiple accounts or jump hosts.

**Match blocks for conditional configuration**

`Match` blocks let you apply settings only to certain users, groups, or addresses. This is useful when you need exceptions without weakening the global policy.

Example:

```sshconfig
PasswordAuthentication no
PermitRootLogin no

Match User backup
    PasswordAuthentication yes
    ForceCommand /usr/local/bin/backup-shell

Match Address 192.0.2.0/24
    X11Forwarding no
```

Use `Match` blocks for controlled exceptions, such as backup accounts or restricted network ranges. Keep the global baseline secure and narrow the exception as much as possible.

**Practical troubleshooting checklist**

When SSH does not work, check the problem in layers.

1. Confirm network reachability with `ping` or `nc`.
2. Verify the service is running on the server.
3. Confirm the port is open in the firewall.
4. Check `sshd_config` syntax with `sshd -t`.
5. Review authentication logs.
6. Test with verbose client output.

Useful commands:

```bash
sudo systemctl status sshd
sudo ss -tlnp | grep sshd
sudo firewall-cmd --list-services
sudo journalctl -u sshd
ssh -vvv user@server.example.com
```

On Red Hat systems, authentication failures often appear in the journal. If you need to inspect recent SSH-related messages:

```bash
sudo journalctl -u sshd --since "10 minutes ago"
```

If SELinux is enforcing and access still fails, check for denials, though SSH configuration issues are more common than SELinux problems for basic connectivity.

**File permissions and ownership**

SSH is strict about permissions because weak permissions can expose private keys or allow tampering.

Important rules:

- `~/.ssh` should be owned by the user.
- `authorized_keys` should not be writable by others.
- private keys should not be group- or world-readable.
- system configuration files should be owned by `root`.

Example permission repair:

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
chmod 600 ~/.ssh/id_ed25519
chown -R alice:alice ~/.ssh
```

If SSH refuses a key, permissions are one of the first things to verify.

**Real-world example: secure admin access**

A common administrative pattern is to allow only key-based access for named admins and disable root login.

Server configuration:

```sshconfig
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowGroups wheel
```

Client configuration for a jump host:

```sshconfig
Host prod-jump
    HostName jump.example.com
    User admin
    IdentityFile ~/.ssh/id_ed25519
    Port 22
```

Workflow:

- You connect to the jump host using your key.
- You use that host to reach internal systems.
- You keep root access disabled and use `sudo` for privilege escalation.

This pattern improves auditability and reduces the risk of direct root compromise.

**Exam-focused takeaways**

For RHCSA, you should be able to:

- identify the difference between client and server SSH configuration
- edit `~/.ssh/config` and `/etc/ssh/sshd_config`
- validate `sshd` syntax before applying changes
- use `ssh -v` and `ssh -G` for troubleshooting
- configure key-based authentication
- understand host key verification and `known_hosts`
- apply basic access restrictions with `AllowUsers`, `AllowGroups`, and `Match`

A strong mental model is:

- client config decides how you connect
- server config decides whether you are allowed in
- keys and permissions decide whether authentication succeeds
