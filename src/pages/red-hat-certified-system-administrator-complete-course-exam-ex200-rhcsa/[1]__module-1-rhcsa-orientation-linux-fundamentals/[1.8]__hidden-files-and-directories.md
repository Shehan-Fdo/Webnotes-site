**Hidden files and directories overview**

In Linux, a file or directory is “hidden” when its name begins with a dot (`.`). This is a naming convention, not a filesystem security feature. Hidden entries still exist in the filesystem and can be accessed directly if you know the path.

This matters in RHCSA work because many user and application settings live in dotfiles under home directories, such as shell startup files, SSH configuration, and desktop or tool-specific config directories.

- **Dotfile**: a regular file whose name starts with `.` such as `.bashrc` or `.gitconfig`
- **Hidden directory**: a directory whose name starts with `.` such as `.ssh` or `.config`
- Hidden entries are not protected by default; they are only omitted from normal directory listings
- You often inspect hidden files when troubleshooting user environments, login behavior, or permissions

**Listing hidden files and directories**

The most common way to show hidden entries is with `ls -a`. The `-a` option includes `.` and `..` as well as dotfiles and dot-directories. If you want a long listing with permissions and ownership, use `ls -la`.

```bash
ls -a
ls -la /home/alice
```

Use `ls -A` when you want hidden entries but do not want `.` and `..` included. This is often more practical than `-a` for everyday inspection.

```bash
ls -A /home/alice
```

- `ls` alone shows only non-hidden entries
- `ls -a` shows all entries, including `.` and `..`
- `ls -A` shows hidden entries except `.` and `..`
- `ls -ld .ssh` shows details about the directory itself, not its contents

A useful RHCSA habit is to inspect a user’s home directory for startup files before changing shell behavior. For example, if a user’s prompt, aliases, or environment variables are not behaving as expected, check `.bashrc`, `.bash_profile`, and related files.

```bash
ls -la /home/alice
cat /home/alice/.bashrc
cat /home/alice/.bash_profile
```

**Creating hidden files and directories**

You create a hidden file or directory by naming it with a leading dot. The shell does not treat it specially beyond the naming convention.

```bash
touch ~/.myfile
mkdir ~/.myconfig
```

- `touch` creates an empty hidden file or updates timestamps if the file already exists
- `mkdir` creates a hidden directory when the name begins with `.`
- Hidden names are common for per-user configuration and application state

A practical example is creating a private configuration directory for a tool under a user’s home directory.

```bash
mkdir -p ~/.config/myapp
touch ~/.config/myapp/settings.conf
```

**Shell globbing and hidden entries**

Shell globbing does not match dotfiles by default. For example, `*` matches visible names but not names beginning with `.`. This is why `cp * /tmp/dir/` does not copy hidden files.

```bash
echo *
echo .*
```

Use caution with `.*` in shell patterns because it can match `.` and `..` depending on the shell and context. For safe interactive work, prefer `ls -A` for listing and explicit paths for access.

> Avoid using `.*` as a general-purpose copy or delete pattern. It can include `.` and `..`, which is dangerous if combined with recursive commands.

If you need to copy a directory tree and include dotfiles, use a tool that preserves hidden entries naturally, such as `cp -a` on the directory itself.

```bash
cp -a /etc/skel/. /tmp/testhome/
```

That pattern copies the contents of `/etc/skel`, including hidden files, into the destination. It is safer and more complete than trying to stitch together shell globs.

- `cp -a source/. dest/` copies all contents, including dotfiles
- `cp -a` preserves attributes as much as possible
- Copying the directory contents directly is more reliable than separate visible and hidden glob patterns

**Finding hidden files and directories**

`find` is useful when you need to search recursively for dotfiles or dot-directories. Because hidden names are just names, `find` can match them with `-name` or `-path`.

```bash
find ~ -type f -name '.*'
find ~ -type d -name '.*'
```

These commands search by filename only, so they are useful for locating dotfiles and dot-directories under a tree. For more precise searches, use `-path` when you need to match a full relative path pattern.

```bash
find ~ -path '*/.*'
```

- `-type f` limits results to regular files
- `-type d` limits results to directories
- `-name '.*'` matches names that begin with a dot
- `-path '*/.*'` is often better when you want to match hidden path components recursively

If you want to exclude `.` and `..` from shell expansion, do not rely on `.*` in the shell. Use `find` or `ls -A` instead.

**Permissions and common hidden configuration files**

Many hidden files are configuration files, so permissions matter. Some are readable by the user only, while others are world-readable if they are intended to be shared. SSH-related files are especially important because incorrect permissions can cause login failures.

Typical SSH-related examples include:

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/config
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
```

- `~/.ssh` is commonly restricted so only the owner can access it
- Private keys and sensitive client config files are usually `600`
- Public keys are commonly `644`
- Exact permission checks can vary by file type and SSH client/server expectations

> Do not assume one permission set fits every hidden file. Apply stricter permissions to secrets and use the least access needed for each file.

A common administrative task is fixing a user’s SSH setup after copying files or restoring a home directory. If SSH refuses to use a key, inspect ownership and permissions first.

```bash
ls -ld ~/.ssh
ls -l ~/.ssh
```

**Shell startup files and user environment**

Hidden files in home directories often control shell behavior. For Bash, common files include `.bash_profile`, `.bashrc`, and sometimes `.profile`. These files can define aliases, environment variables, prompt settings, and startup logic.

```bash
cat ~/.bashrc
cat ~/.bash_profile
```

A typical pattern is to place interactive shell settings in `.bashrc` and login-shell settings in `.bash_profile`. On many systems, `.bash_profile` sources `.bashrc` so both login and interactive shells get the same environment.

```bash
# Example ~/.bash_profile
if [ -f ~/.bashrc ]; then
    . ~/.bashrc
fi
```

- `.bashrc` is commonly used for aliases and interactive shell settings
- `.bash_profile` is commonly used for login-shell initialization
- `.profile` may be used by other shells or as a fallback
- Editing these files affects only the user whose home directory contains them

A practical RHCSA workflow is to inspect these files when a user reports missing aliases, incorrect `PATH`, or unexpected prompt behavior.

1. Check the user’s home directory with `ls -la`.
2. Inspect `.bashrc` and `.bash_profile`.
3. Edit the relevant file with `vi` or `vim`.
4. Reload the shell configuration with `source ~/.bashrc` or by opening a new session.

```bash
source ~/.bashrc
```

**Common admin examples**

Hidden files are often involved in backup, restore, and user provisioning tasks. For example, when preparing a new home directory from `/etc/skel`, you want the user’s default dotfiles copied into place.

```bash
cp -a /etc/skel/. /home/newuser/
chown -R newuser:newuser /home/newuser
```

This approach copies the contents of `/etc/skel`, including hidden files, and then assigns ownership to the new user.

Another common task is checking whether a user has custom shell settings that affect troubleshooting.

```bash
sudo -u alice ls -la /home/alice
sudo -u alice cat /home/alice/.bashrc
```

- Use `sudo -u` to inspect files as a specific user when needed
- Hidden files often explain differences between users on the same system
- Ownership and permissions are as important as the file contents

**Practical cautions**

Hidden files are not a security boundary. If a user has permission to read a directory, they can access hidden entries inside it by name. The dot only affects default display behavior in tools like `ls`.

- Hidden files are visible to `ls -a` and `find`
- They are accessible by direct path if permissions allow it
- Do not rely on dot-prefix naming to protect secrets
- Use proper permissions, ownership, and access controls instead

For exam readiness, remember the difference between “hidden from casual listing” and “protected from access.” The first is naming convention; the second is filesystem permissions.

**Quick reference**

| Task | Command |
|---|---|
| Show hidden entries | `ls -a` |
| Long listing with hidden entries | `ls -la` |
| Show hidden entries without `.` and `..` | `ls -A` |
| Create hidden file | `touch .filename` |
| Create hidden directory | `mkdir .dirname` |
| Search for hidden names | `find ~ -name '.*'` |
| Copy directory contents including dotfiles | `cp -a source/. dest/` |

**Exam tips**

- Use `ls -la` when you need to inspect a home directory for startup files or permissions
- Use `find` for recursive searches instead of shell globbing when you need reliable results
- Use `cp -a source/. dest/` to include hidden contents safely
- Check ownership and permissions on `.ssh` and shell startup files when troubleshooting user access
- Remember that hidden files are ordinary files with a dot-prefixed name
