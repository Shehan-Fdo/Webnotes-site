## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**RHCSA EX200 at a glance**

The Red Hat Certified System Administrator (RHCSA) exam, EX200, is a hands-on performance-based test. You do not answer multiple-choice questions; you complete real administration tasks on a live system. The exam measures whether you can configure, troubleshoot, and maintain a RHEL environment under time pressure.

The core expectation is simple: you must be able to perform common Linux administration tasks reliably, without internet access, and with minimal guidance. That means you should know not only *what* to do, but also *how to verify* that it worked.

- **Performance-based** means you must produce working system changes, not explain them.
- **Task-oriented** means each objective is usually independent and practical.
- **Verification-focused** means you should check your work with commands, not assume success.

**What the exam is really testing**

RHCSA is less about memorizing isolated commands and more about demonstrating operational competence. You are expected to understand how system components fit together: storage, users, services, networking, security, and basic automation.

In practice, that means you should be comfortable with:

- Navigating the shell efficiently.
- Reading documentation with `man`, `info`, and `--help`.
- Editing configuration files safely.
- Using `systemctl`, `journalctl`, `ip`, `nmcli`, `firewall-cmd`, `semanage`, `restorecon`, `tar`, `rsync`, `dnf`, `useradd`, `passwd`, `chmod`, `chown`, `sudo`, `mount`, `lsblk`, `blkid`, `lvm`, and `podman`.
- Confirming persistence after reboot when the task requires it.

> The exam rewards correct outcomes, not elegant shortcuts. If a task asks for a persistent configuration, you must make it survive reboot and verify that behavior.

**Exam environment and constraints**

The exam is delivered on a live RHEL system, typically in a controlled lab environment. You should expect:

- No internet access.
- No external notes or browser access.
- A limited set of systems and accounts.
- Tasks that may depend on earlier configuration being correct.
- A need to work quickly but carefully.

You are usually given a standard user account and a `root`-equivalent path through `sudo` or direct root access depending on the exam setup. You should know how to elevate privileges safely and how to avoid breaking the environment.

**How to approach tasks**

A good RHCSA workflow is consistent:

1. Read the task carefully.
2. Identify the target host, user, service, or path.
3. Check the current state before changing anything.
4. Make the smallest correct change.
5. Verify the result.
6. Persist the configuration if required.
7. Move on.

This workflow matters because many exam tasks are designed to be solved in multiple ways, but only one way is efficient and reliable under exam pressure.

**Useful verification habits**

Verification is one of the biggest differentiators between passing and failing. You should confirm both immediate behavior and persistence.

Examples:

```bash
# Check service state
systemctl status sshd
systemctl is-enabled sshd

# Inspect logs
journalctl -u sshd --no-pager
journalctl -xe --no-pager

# Check network configuration
ip addr show
nmcli device status
nmcli connection show

# Confirm storage
lsblk
blkid
mount | grep /data
cat /etc/fstab
```

Use verification commands after every meaningful change. If a task involves a service, confirm it is active and enabled if required. If it involves storage, confirm the filesystem is mounted and survives reboot. If it involves networking, confirm the connection profile is persistent, not just temporarily active.

**Time management strategy**

The exam is time-constrained, so you should avoid overengineering. Focus on the highest-value tasks first and keep a steady pace.

- Start with tasks you know well to build momentum.
- Skip a difficult task temporarily if it blocks progress.
- Return later with a fresh perspective.
- Avoid spending too long on a single misconfiguration.
- Use documentation efficiently instead of guessing.

A practical approach is to keep a mental split between:

- **Fast wins**: tasks you can complete quickly and verify immediately.
- **Risk tasks**: storage, networking, SELinux, and boot persistence changes that need careful validation.

**Common RHCSA topic areas**

The exam objectives typically cover a broad set of system administration skills. Exact weighting can vary by exam version, but the following areas are central:

- **Essential commands**
  - File manipulation, text processing, archiving, permissions, and shell usage.
- **Operation of running systems**
  - Boot process, services, logs, process management, and scheduling.
- **User and group management**
  - Accounts, passwords, sudo access, and default settings.
- **Networking**
  - IP configuration, hostname management, and connectivity checks.
- **Service configuration**
  - Enabling and managing common services.
- **Storage management**
  - Partitions, LVM, mount points, swap, and filesystem creation.
- **Security**
  - Firewalld, SELinux, and access controls.
- **Containers**
  - Basic Podman usage and container lifecycle management.

**Command-line mindset**

You should be comfortable using the shell as your primary interface. RHCSA expects you to know how to combine commands and inspect output quickly.

Examples of practical shell habits:

```bash
# Find files and inspect content
find /etc -name '*.conf'
grep -R "Listen" /etc/httpd

# Compare configuration before and after
diff -u /etc/ssh/sshd_config{.bak,}

# Inspect command syntax
man nmcli
man semanage
dnf info podman
```

When in doubt, read the manual page for the exact tool on the system. RHEL tools often have subtle differences in syntax or behavior across versions, and the exam expects you to adapt.

**Networking expectations**

You should know how to configure and verify network settings using `nmcli` and related tools. On modern RHEL systems, NetworkManager is the standard management layer.

Typical tasks include:

- Setting a static IPv4 address.
- Configuring a default gateway and DNS server.
- Renaming or activating a connection profile.
- Setting the hostname.
- Verifying connectivity with `ping`, `ip route`, and `nmcli`.

A safe static configuration pattern is:

```bash
nmcli con mod "System eth0" ipv4.method manual
nmcli con mod "System eth0" ipv4.addresses 192.0.2.10/24
nmcli con mod "System eth0" ipv4.gateway 192.0.2.1
nmcli con mod "System eth0" ipv4.dns 192.0.2.53
nmcli con mod "System eth0" connection.autoconnect yes
nmcli con up "System eth0"
```

Then verify:

```bash
nmcli con show "System eth0"
ip addr show
ip route
cat /etc/resolv.conf
```

The key idea is persistence. A temporary `ip addr add` change may work immediately, but it is not enough if the task requires a lasting configuration.

**Storage expectations**

Storage tasks are common and often require careful sequencing. You should understand how to identify disks, create partitions or logical volumes, format filesystems, mount them, and make them persistent.

Typical workflow:

1. Identify available block devices with `lsblk` and `blkid`.
2. Create the required partition or logical volume.
3. Create the filesystem.
4. Mount it to the correct directory.
5. Add a persistent mount entry.
6. Verify after mounting and after reboot if possible.

Example LVM workflow:

```bash
pvcreate /dev/sdb
vgcreate vgdata /dev/sdb
lvcreate -n lvdata -L 2G vgdata
mkfs.xfs /dev/vgdata/lvdata
mkdir -p /data
mount /dev/vgdata/lvdata /data
```

For persistence, prefer UUID-based entries in `/etc/fstab`:

```bash
blkid /dev/vgdata/lvdata
```

Then add a line like:

```fstab
UUID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx /data xfs defaults 0 0
```

Verify with:

```bash
mount -a
findmnt /data
```

> Avoid blindly appending to `/etc/fstab`. Duplicate or incorrect entries can prevent boot or break mounts. Edit carefully and validate with `mount -a`.

**Filesystem choice and formatting**

RHCSA tasks may specify a filesystem type. XFS is common on RHEL, but you should not assume it is always correct. Use the filesystem requested by the task.

- Use `mkfs.xfs` when XFS is required.
- Use `mkfs.ext4` when ext4 is required.
- Confirm the filesystem type with `lsblk -f` or `blkid`.

Example:

```bash
mkfs.ext4 /dev/vgdata/lvbackup
```

**User, group, and sudo expectations**

You should be able to create users and groups, set passwords, manage supplementary groups, and grant limited administrative access with sudo.

Common tasks include:

- Creating local users and groups.
- Setting password aging or account expiration.
- Configuring default shell or home directory behavior.
- Granting sudo privileges through `/etc/sudoers.d/`.

Example:

```bash
groupadd developers
useradd -m -G developers alice
passwd alice
```

For sudo, use a drop-in file and validate it:

```bash
echo 'alice ALL=(ALL) ALL' > /etc/sudoers.d/alice
chmod 440 /etc/sudoers.d/alice
chown root:root /etc/sudoers.d/alice
visudo -cf /etc/sudoers.d/alice
```

The file must have the correct ownership and permissions. Validation with `visudo -cf` helps catch syntax errors before they become a problem.

**Service management expectations**

You should know how to manage systemd services with `systemctl`. That includes starting, stopping, enabling, disabling, masking, and checking status.

Examples:

```bash
systemctl status chronyd
systemctl enable --now chronyd
systemctl restart sshd
systemctl is-enabled firewalld
```

When troubleshooting, inspect logs with `journalctl`:

```bash
journalctl -u chronyd --no-pager
journalctl -xe --no-pager
```

Use the exact unit name on the system. Not every service name is identical across environments, and some tasks depend on whether the package is installed.

**SELinux and firewall expectations**

Security tasks often involve SELinux and `firewalld`. You should understand the difference between temporary and persistent changes.

For SELinux, common actions include:

- Checking current mode with `getenforce`.
- Inspecting file contexts with `ls -Z`.
- Restoring default contexts with `restorecon`.
- Setting persistent file context rules with `semanage fcontext`.

Example:

```bash
semanage fcontext -a -t httpd_sys_content_t '/web(/.*)?'
restorecon -Rv /web
ls -Z /web
```

For firewall rules, use `firewall-cmd`:

```bash
firewall-cmd --add-service=http
firewall-cmd --add-service=http --permanent
firewall-cmd --reload
firewall-cmd --list-services
```

Remember that a runtime-only firewall change disappears after reload unless you also make it permanent.

**Container expectations**

RHCSA may include basic container tasks using Podman. You should know how to search, pull, run, inspect, and manage containers.

Examples:

```bash
podman search httpd
podman pull registry.access.redhat.com/ubi9/httpd-24
podman run -d --name web -p 8080:8080 registry.access.redhat.com/ubi9/httpd-24
podman ps
podman logs web
podman exec -it web /bin/bash
```

For a useful service container, make sure the container actually serves content or runs the intended process. A container that starts and exits immediately is not a successful service deployment.

A more practical example with a mounted content directory:

```bash
mkdir -p ~/webcontent
echo 'RHCSA container test' > ~/webcontent/index.html
podman run -d --name web -p 8080:8080 -v ~/webcontent:/var/www/html:Z registry.access.redhat.com/ubi9/httpd-24
```

Then verify with:

```bash
curl http://localhost:8080
podman ps
```

**Documentation and troubleshooting habits**

You should treat documentation as part of the exam workflow. If a command fails, do not guess repeatedly. Check the syntax, inspect the current state, and read the relevant logs.

Useful habits:

- Use `man <command>` before changing unfamiliar settings.
- Use `systemctl status` and `journalctl` for service failures.
- Use `ip addr`, `ip route`, and `nmcli` for network issues.
- Use `lsblk`, `blkid`, `findmnt`, and `mount` for storage issues.
- Use `getenforce`, `ls -Z`, and `ausearch` when SELinux is involved.

**Practical exam mindset**

The best exam strategy is disciplined and repeatable. You should think like an administrator who must leave the system in a known-good state.

- Make one change at a time.
- Verify immediately.
- Prefer persistent configuration over temporary fixes.
- Keep notes of task names and completed items.
- Do not assume defaults match the task requirements.
- Recheck after reboot when the task depends on persistence.

**What success looks like**

A strong RHCSA candidate can:

- Read a task and translate it into exact system changes.
- Use the correct tool for the job.
- Verify the result without external help.
- Recover from mistakes quickly.
- Keep configurations persistent and clean.

The exam is designed to measure real operational readiness. If you can build and validate a working system configuration from the command line, you are aligned with the exam’s expectations.

**Quick reference checklist**

- Confirm the task requirements before changing anything.
- Use `man` pages for exact syntax.
- Prefer persistent settings for networking, storage, services, and security.
- Validate with `systemctl`, `journalctl`, `nmcli`, `ip`, `lsblk`, `findmnt`, `firewall-cmd`, and `getenforce`.
- Use `visudo -cf` for sudoers changes.
- Use `mount -a` and `findmnt` to test `/etc/fstab`.
- Use `curl` or `podman logs` to confirm container behavior.
- Keep your workflow calm, methodical, and verifiable.
