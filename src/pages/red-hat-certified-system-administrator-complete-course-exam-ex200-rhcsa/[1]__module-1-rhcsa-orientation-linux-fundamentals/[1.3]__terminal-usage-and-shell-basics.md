## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Terminal fundamentals**

The terminal is your primary interface for RHCSA tasks. You use it to inspect the system, edit files, manage processes, and verify changes quickly. On the exam, speed matters, but correctness matters more: you should know how to navigate the shell, understand what the shell expands before execution, and choose the right command for the job.

A **shell** is the command interpreter that reads your input, expands patterns and variables, and launches commands. On RHEL, `/bin/bash` is the default interactive shell for many users, so these notes focus on Bash behavior unless stated otherwise.

- **Prompt**: the visible cue that the shell is ready for input.
- **Command line**: the text you type before pressing <kbd>Enter</kbd>.
- **Exit status**: the numeric result of the last command, available in `$?`.

```bash
echo $?
printf 'Last exit status: %s\n' "$?"
```

**Shell types and startup behavior**

You should distinguish between login, interactive, and non-interactive shells because they read different startup files. This matters when environment variables, aliases, or functions appear to “work in one terminal but not another.”

- **Login shell**: starts when you log in directly, such as on a console or via SSH with a login session.
  - Bash typically reads `/etc/profile`, then one of `~/.bash_profile`, `~/.bash_login`, or `~/.profile`.
- **Interactive non-login shell**: starts when you open a new terminal inside an existing session.
  - Bash typically reads `~/.bashrc`.
- **Non-interactive shell**: runs scripts or one-off commands.
  - Bash usually reads the `BASH_ENV` file if set, but not your interactive startup files by default.

Practical examples:

```bash
# Login shell example
bash -l

# Interactive shell example
bash

# Non-interactive shell example
bash -c 'echo "hello from script mode"'
```

For exam troubleshooting, remember that environment changes in `~/.bashrc` may not affect login shells unless you source the file or chain it from `~/.bash_profile`.

> If you change shell startup files during the exam, test the effect in a new shell session or with `source ~/.bashrc` rather than assuming the current shell has reloaded them.

**Command resolution and builtins**

When you type a command, Bash resolves it in a specific order. Understanding this helps you debug “command not found” issues and avoid confusion between aliases, functions, builtins, and external binaries.

Typical resolution order in Bash:

1. Alias
2. Function
3. Shell builtin
4. External command found in `$PATH`

Useful inspection commands:

```bash
type ls
type cd
type -a echo
command -v grep
which grep
```

Use `type` or `command -v` when you want shell-aware resolution. `which` may work on many systems, but it is not the best choice for aliases, functions, or builtins because it is often an external utility and its behavior can vary.

Common builtins you should recognize:

- `cd`
- `echo`
- `printf`
- `export`
- `unset`
- `read`
- `alias`
- `history`
- `test` / `[`
- `source` / `.`

Why builtins matter:

- They run inside the shell process, so they can change the current shell state.
- `cd` must be a builtin because an external program cannot change the parent shell’s working directory.
- `export` marks variables for inheritance by child processes.

```bash
pwd
cd /etc
pwd

MYVAR='RHCSA'
export MYVAR
bash -c 'echo "$MYVAR"'
```

**Variables, quoting, and expansion**

Shell expansion happens before command execution. You should know the common expansion types and the order in which Bash applies them, because many exam mistakes come from quoting errors rather than command syntax.

Common expansions include:

- **Tilde expansion**: `~` becomes a home directory.
- **Parameter expansion**: `$VAR` or `${VAR}` becomes a variable value.
- **Command substitution**: `$(command)` inserts command output.
- **Arithmetic expansion**: `$((expression))` evaluates math.
- **Brace expansion**: `{a,b}` or `{1..5}` generates text patterns.
- **Filename expansion**: globbing with `*`, `?`, and `[]`.

Example:

```bash
USER_NAME='alice'
echo ~
echo "$USER_NAME"
echo "Today is $(date +%F)"
echo $((2 + 3 * 4))
echo file{1,2,3}.txt
```

Expansion order is important in practice:

- Tilde expansion happens early and usually only when unquoted at the start of a word.
- Variable and command substitution occur before word splitting and pathname expansion.
- Quoting suppresses many expansions.
- Single quotes preserve literal text.
- Double quotes allow variable and command substitution but suppress most word splitting and globbing.

Examples:

```bash
name='world'
echo 'Hello $name'
echo "Hello $name"

files="*.txt"
echo $files
echo "$files"
```

Use quotes by default when variables may contain spaces or wildcard characters.

> Do not assume brace expansion is the same as globbing. Brace expansion creates text before pathname matching happens.

**Globbing and file matching**

Globbing is the shell’s filename pattern matching. It is not regular expressions. This distinction is common on RHCSA-style tasks.

Patterns:

- `*` matches any string, including empty.
- `?` matches exactly one character.
- `[abc]` matches one character from a set.
- `[a-z]` matches one character in a range.

Examples:

```bash
ls /etc/*.conf
ls /var/log/maillog?
ls file[1-3].txt
```

Useful options and shell settings:

- `nullglob` makes unmatched globs expand to nothing.
- `failglob` makes unmatched globs produce an error.
- `set -f` disables pathname expansion.

```bash
shopt -s nullglob
files=(*.log)
printf '%s\n' "${files[@]}"
```

**Redirection and file descriptors**

Redirection is one of the most important shell skills for the exam. You use it to capture output, suppress noise, combine streams, and feed input from files.

Standard file descriptors:

- `0` = standard input (`stdin`)
- `1` = standard output (`stdout`)
- `2` = standard error (`stderr`)

Common redirection patterns:

```bash
command > out.txt
command >> out.txt
command < input.txt
command 2> error.txt
command > out.txt 2> error.txt
command > out.txt 2>&1
command &> all.txt
```

Key ideas:

- `>` overwrites a file.
- `>>` appends to a file.
- `2>` redirects error output.
- `2>&1` duplicates file descriptor 2 to wherever file descriptor 1 currently points.
- `&>` redirects both stdout and stderr in Bash.

Order matters:

```bash
# Correct: both stdout and stderr go to all.txt
command > all.txt 2>&1

# Different behavior: stderr still goes to the terminal
command 2>&1 > all.txt
```

That second form first sends stderr to the current stdout target, then changes stdout. The result is often not what you intended.

Practical examples:

```bash
grep 'root' /etc/passwd > matches.txt
grep 'root' /etc/passwd 2> errors.txt
find /root -name '*.conf' > found.txt 2>&1
```

You can also discard output:

```bash
command > /dev/null
command 2> /dev/null
command > /dev/null 2>&1
```

Input redirection and here documents are also useful:

```bash
wc -l < /etc/passwd

cat <<'EOF'
This text is literal.
$HOME is not expanded here.
EOF
```

**Pipes and command chaining**

A pipe sends the stdout of one command to the stdin of another. This is a core shell workflow for filtering and transforming data.

```bash
ps aux | grep sshd
dmesg | tail
cat /etc/passwd | cut -d: -f1
```

Prefer direct command input when possible, but pipes are excellent for composing small tools.

Common chaining operators:

- `;` runs the next command regardless of success.
- `&&` runs the next command only if the previous succeeds.
- `||` runs the next command only if the previous fails.

Examples:

```bash
mkdir -p /tmp/demo && cd /tmp/demo
make test || echo "tests failed"
command1 ; command2
```

For exam work, `&&` is especially useful when you want to avoid continuing after a failure.

**Command substitution and practical variable use**

Command substitution lets you capture command output into a variable or another command line. Use `$(...)` rather than backticks for readability and nesting.

```bash
today=$(date +%F)
echo "Today is $today"

count=$(wc -l < /etc/passwd)
echo "Users listed: $count"
```

Variable assignment rules:

- No spaces around `=`.
- Use quotes when values may contain spaces.
- Export only when child processes need the variable.

```bash
PROJECT='RHCSA lab'
export PROJECT
printf 'Project: %s\n' "$PROJECT"
```

Common exam pattern:

```bash
backup_dir="/backup/$(date +%F)"
mkdir -p "$backup_dir"
cp /etc/hosts "$backup_dir/"
```

**Process management and job control**

Job control helps you manage long-running commands in the terminal. You should know the difference between jobs and processes.

- A **process** is a running program with a process ID, or **PID**.
- A **job** is a shell-managed pipeline or command started in the current shell.

Useful commands:

```bash
jobs
bg
fg
ps -ef
ps aux
ps -o pid,ppid,stat,cmd -p 1234
```

Start a job in the background:

```bash
sleep 300 &
jobs
```

Bring it back to the foreground:

```bash
fg %1
```

Suspend a foreground job with <kbd>Ctrl</kbd> + <kbd>Z</kbd>, then resume it in the background:

```bash
bg %1
```

Signals and termination:

```bash
kill 1234
kill -TERM 1234
kill -9 1234
pkill sshd
```

Use `kill` first with the default `SIGTERM` signal. Reserve `SIGKILL` (`-9`) for cases where a process does not respond, because it prevents cleanup.

Managing work after logout:

- Background jobs tied to your shell usually stop when the shell exits.
- `nohup` can help a command continue after logout by ignoring hangup signals.
- `disown` removes a job from the shell’s job table.
- `tmux` or `screen` is often the safest way to keep an interactive session alive.

Examples:

```bash
nohup long_task > long_task.log 2>&1 &
disown
```

> `nohup` and `disown` help, but `tmux` is usually the most reliable way to preserve an interactive troubleshooting session across disconnects.

**Finding and inspecting processes**

You often need to identify a process before you can stop or troubleshoot it.

```bash
ps -ef | grep nginx
pgrep sshd
pidof sshd
top
```

Useful fields in `ps` output:

- `PID`: process ID
- `PPID`: parent process ID
- `TTY`: controlling terminal
- `STAT`: process state
- `CMD`: command name and arguments

Example:

```bash
ps -o pid,ppid,user,stat,cmd -C bash
```

**Shell history and command recall**

History speeds up repetitive work and helps you recover long commands.

```bash
history
history 20
!42
!! 
!grep
```

Useful shortcuts:

- <kbd>Ctrl</kbd> + <kbd>R</kbd>: reverse incremental search
- <kbd>Ctrl</kbd> + <kbd>A</kbd>: move to start of line
- <kbd>Ctrl</kbd> + <kbd>E</kbd>: move to end of line
- <kbd>Ctrl</kbd> + <kbd>U</kbd>: cut from cursor to start
- <kbd>Ctrl</kbd> + <kbd>K</kbd>: cut from cursor to end
- <kbd>Ctrl</kbd> + <kbd>W</kbd>: delete previous word

History expansion can be powerful, but use it carefully in exam environments because it can surprise you if the previous command was not what you expected.

**Terminal navigation and filesystem basics**

You should be fluent with the commands that show where you are and what exists around you.

```bash
pwd
ls
ls -l
ls -a
cd /etc
cd -
```

Useful `ls` options:

- `-l` long listing
- `-a` include hidden files
- `-h` human-readable sizes
- `-R` recursive listing

Examples:

```bash
ls -lah /var/log
find /etc -maxdepth 1 -type f
```

**Text viewing and quick inspection**

Before editing, inspect files safely. This reduces mistakes and helps you understand file format and content.

```bash
cat /etc/hosts
less /etc/services
head -n 5 /etc/passwd
tail -n 20 /var/log/messages
```

`less` is especially useful because it supports search and navigation:

- `/pattern` searches forward
- `?pattern` searches backward
- `n` repeats the search
- `N` reverses the search
- `g` jumps to the start
- `G` jumps to the end

For log monitoring, `tail -f` follows appended data. On GNU coreutils systems, `tail -F` follows by filename and can be more useful across log rotation events, but that behavior is GNU-specific and may not be identical on every platform.

```bash
tail -f /var/log/messages
tail -F /var/log/messages
```

**Editing with vi/vim**

`vi` or `vim` is the most important editor to know for RHCSA. You do not need to be a power user, but you should be fast with basic navigation, search, and edits.

Modes:

- **Normal mode**: navigation and commands
- **Insert mode**: text entry
- **Command-line mode**: save, quit, search, and substitute

Core workflow:

1. Open the file.
2. Navigate to the target line.
3. Enter insert mode only when needed.
4. Save and quit cleanly.

```bash
vi /etc/hosts
```

Essential commands:

- `i` insert before cursor
- `a` append after cursor
- `o` open a new line below
- `Esc` return to normal mode
- `:w` write/save
- `:q` quit
- `:wq` save and quit
- `:q!` quit without saving

Navigation and speed:

- `gg` go to first line
- `G` go to last line
- `:set number` show line numbers
- `:set nonumber` hide line numbers
- `:42` jump to line 42
- `/pattern` search forward
- `n` next match
- `N` previous match

Copy, cut, and paste:

- `yy` copy current line
- `dd` cut current line
- `p` paste after cursor
- `P` paste before cursor
- `3yy` copy three lines
- `3dd` cut three lines

Useful exam pattern:

```vim
:set number
/ListenAddress
dd
oListenAddress 192.0.2.10
:wq
```

If you need to replace text quickly:

```vim
:%s/old/new/g
```

**Practical shell examples**

These examples combine several basics you will use repeatedly.

```bash
# Show the current user and working directory
printf 'User: %s\n' "$USER"
printf 'Directory: %s\n' "$PWD"

# Create a variable and use it safely
backup_name="backup-$(date +%F)"
mkdir -p "/tmp/$backup_name"

# Capture command output
kernel=$(uname -r)
echo "Running kernel: $kernel"

# Filter output and save errors separately
grep 'sshd' /var/log/messages > sshd.log 2> sshd.err

# Compare two command outputs in Bash
diff <(ls /etc | sort) <(ls /usr | sort)
```

The `diff <(...)` example uses Bash process substitution, so treat it as Bash-specific rather than generic POSIX shell syntax.

**Troubleshooting workflow for the exam**

When a shell task fails, use a consistent approach instead of guessing.

1. Check the exact command you typed.
2. Inspect quoting, redirection, and variable expansion.
3. Verify the current shell and environment.
4. Confirm the file or process exists.
5. Use `type`, `command -v`, `ps`, `ls`, or `stat` as appropriate.
6. Re-run the command with simpler pieces if needed.

Examples:

```bash
type -a python
command -v sshd
ps -ef | grep '[s]shd'
ls -l /path/to/file
stat /path/to/file
```

**Exam-focused reminders**

- Use `printf` when you need predictable formatting; `echo` is fine for simple output but can vary with options and escape handling.
- Quote variables unless you intentionally want word splitting or glob expansion.
- Prefer `type` or `command -v` over `which` when checking command resolution.
- Use `ps`, `pgrep`, and `kill` to manage processes by PID.
- Use `jobs`, `bg`, `fg`, and `nohup` for shell job control.
- Use `less`, `head`, `tail`, and `vi` efficiently to inspect and edit files under time pressure.
