## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Redirection fundamentals**

Shell redirection controls where command input comes from and where output goes. In Linux, every process starts with three standard file descriptors:

- `0` = **stdin** (standard input)
- `1` = **stdout** (standard output)
- `2` = **stderr** (standard error)

You use redirection when you want to save output to a file, feed data into a command, separate normal output from errors, or automate workflows without interactive typing. This is a core RHCSA skill because many administration tasks depend on capturing logs, filtering command output, and building repeatable command pipelines.

Common redirection operators:

- `>` redirects stdout and overwrites the target file
- `>>` redirects stdout and appends to the target file
- `<` redirects stdin from a file
- `2>` redirects stderr
- `2>>` appends stderr
- `&>` redirects both stdout and stderr in Bash
- `|` sends stdout from one command into stdin of another

A simple example:

```bash
echo "server1" > hosts.txt
cat < hosts.txt
```

Use `>` when you want a clean replacement of file contents. Use `>>` when you want to preserve existing data and add new lines. Be careful with overwrite redirection because the shell truncates the file immediately before the command writes anything.

> <mark>Important:</mark> `>` replaces file contents. If you need to preserve data, use `>>` or back up the file first.

**Standard output and standard error**

Most commands write normal results to stdout and problems to stderr. This distinction matters because you can redirect them independently.

Example:

```bash
ls /etc/passwd /no/such/file
```

- The listing of `/etc/passwd` goes to stdout
- The error for `/no/such/file` goes to stderr

To save only errors:

```bash
ls /etc/passwd /no/such/file 2> errors.log
```

To save both streams separately:

```bash
ls /etc/passwd /no/such/file > output.log 2> errors.log
```

To combine both into one file in Bash:

```bash
ls /etc/passwd /no/such/file &> all.log
```

If you need to append both streams, use:

```bash
ls /etc/passwd /no/such/file >> output.log 2>> errors.log
```

A useful troubleshooting pattern is to redirect stderr to stdout and then pipe the combined stream into another command:

```bash
command 2>&1 | less
```

This sends errors to the same stream as normal output, then passes everything to `less` for review. The order matters: `2>&1` must usually appear before the pipe if you want both streams in the pipeline.

**Pipes and command chaining**

A pipe connects the stdout of one command to the stdin of another. Pipes let you build small, focused commands that work together.

Example:

```bash
ps aux | grep sshd
```

Here, `ps aux` produces a process list, and `grep sshd` filters it. Pipes are powerful because they reduce the need for temporary files and make command lines easier to compose.

Common pipeline patterns:

- `command | less` for paging long output
- `command | grep pattern` for filtering text
- `command | sort | uniq` for organizing and deduplicating data
- `command | wc -l` for counting lines
- `command | head` or `command | tail` for sampling output

Example with multiple stages:

```bash
journalctl -u sshd | grep -i fail | tail -n 10
```

This shows recent SSH daemon failures by filtering case-insensitively and then displaying the last 10 matches.

Pipes only pass stdout by default. If a command writes important information to stderr, it will not flow through the pipe unless you redirect it first.

**Filtering text with grep, sort, uniq, and wc**

These utilities are frequently used with redirection and pipes.

`grep` searches for matching lines:

```bash
grep "error" /var/log/messages
grep -i "error" /var/log/messages
grep -v "error" /var/log/messages
```

- `-i` ignores case
- `-v` inverts the match

`sort` orders lines:

```bash
sort users.txt
sort -r users.txt
```

`uniq` removes adjacent duplicate lines, so it usually follows `sort`:

```bash
sort users.txt | uniq
sort users.txt | uniq -c
```

`wc` counts lines, words, and bytes:

```bash
wc -l /etc/passwd
grep -i "failed" auth.log | wc -l
```

A practical example for finding repeated entries:

```bash
cut -d: -f1 /etc/passwd | sort | uniq -c | sort -nr
```

This extracts usernames, counts duplicates, and sorts by highest count first. Even if duplicates are unlikely in `/etc/passwd`, the pattern is useful for logs and reports.

**Input redirection and here documents**

Input redirection feeds a file into a command that expects stdin.

Example:

```bash
wc -l < /etc/passwd
```

This is similar to `wc -l /etc/passwd`, but it demonstrates stdin redirection explicitly.

A here document lets you send multiple lines of input directly from the shell:

```bash
cat <<EOF
line one
line two
EOF
```

Here documents are useful for quick configuration snippets, scripted input, and test data. In shell scripts, quote the delimiter if you want to prevent variable expansion:

```bash
cat <<'EOF'
$HOME is not expanded here
EOF
```

A here string sends a single string into stdin:

```bash
grep root <<< "root:x:0:0:root:/root:/bin/bash"
```

**File descriptor control**

File descriptors are numeric handles for open files and streams. You can redirect them individually.

Examples:

```bash
command > output.txt 2> error.txt
command 1> output.txt 2> error.txt
```

These are equivalent because `1` is stdout.

To discard output, redirect it to `/dev/null`:

```bash
command > /dev/null
command > /dev/null 2>&1
```

This is useful when you want to suppress normal output or silence a command in scripts.

To send stdout to one file and stderr to another:

```bash
find /etc -name passwd > matches.txt 2> errors.txt
```

To append stderr while overwriting stdout:

```bash
find /etc -name passwd > matches.txt 2>> errors.txt
```

**Command substitution and pipelines**

Command substitution captures the output of a command and inserts it into another command. It is not the same as a pipe, but it often appears in the same workflows.

Example:

```bash
echo "Files: $(ls /etc | wc -l)"
```

Use command substitution when you need a command’s result as an argument. Use a pipe when you want to stream output into another command.

Compare:

- Pipe: `ls | wc -l`
- Substitution: `echo "Count: $(ls | wc -l)"`

For scripts, prefer safer and more predictable tools than parsing `ls` output when possible. For example, use `find` for file discovery.

**Practical administration examples**

Redirection and pipes are common in real system administration tasks.

Log review:

```bash
journalctl -u sshd | grep -i "failed"
```

Save a report:

```bash
journalctl -u sshd | grep -i "failed" > ssh-failures.log
```

Count failed logins:

```bash
grep -i "failed password" /var/log/secure | wc -l
```

Find large files and sort them:

```bash
find /var -type f -printf '%s %p\n' 2>/dev/null | sort -nr | head
```

This command suppresses permission errors, prints file sizes and paths, sorts numerically in reverse order, and shows the largest entries.

Extract usernames from `/etc/passwd`:

```bash
cut -d: -f1 /etc/passwd
```

Combine with sorting and counting:

```bash
cut -d: -f1 /etc/passwd | sort | uniq
```

**Useful shell patterns**

A few patterns appear often in RHCSA tasks:

- `command > file 2>&1` to capture all output in one file
- `command | tee file` to display output and save it at the same time
- `command | tee -a file` to append while displaying
- `command 2>/dev/null` to hide errors
- `command | grep -E 'pattern1|pattern2'` for extended matching

Example with `tee`:

```bash
dmesg | tee dmesg.txt
```

`tee` is valuable when you want to inspect output interactively and keep a copy for later.

Example with append mode:

```bash
echo "backup completed" | tee -a /var/log/backup.log
```

**Troubleshooting redirection issues**

If a command behaves unexpectedly, check these common causes:

- You redirected the wrong stream
- You used `>` when you meant `>>`
- You placed `2>&1` in the wrong position
- The shell expanded variables or wildcards before the command ran
- The target file is not writable by your user

A frequent mistake is assuming pipes carry stderr. They do not unless you redirect stderr into stdout first.

Example of correct combined piping:

```bash
command 2>&1 | grep -i error
```

Example of incorrect intent if you want both streams filtered:

```bash
command | grep -i error
```

This misses stderr output.

Another common issue is using `uniq` without sorting first. `uniq` only removes adjacent duplicates, so it is most effective after `sort`.

**Quick reference**

| Operator | Meaning | Typical use |
|---|---|---|
| `>` | Redirect stdout, overwrite | Save fresh output |
| `>>` | Redirect stdout, append | Add to a log |
| `<` | Redirect stdin from file | Feed input from a file |
| `2>` | Redirect stderr | Save errors separately |
| `2>>` | Append stderr | Grow an error log |
| `2>&1` | Send stderr to stdout | Combine streams |
| `|` | Pipe stdout to stdin | Build command chains |
| `tee` | Copy stdin to file and stdout | View and save output |

**Exam-focused takeaways**

For EX200, you should be comfortable doing the following without hesitation:

- Redirect command output to files
- Append output instead of overwriting it
- Separate stdout and stderr
- Combine streams when needed
- Build pipelines with `grep`, `sort`, `uniq`, `wc`, `head`, `tail`, and `tee`
- Recognize when a command needs stdin versus when it produces stdout

Practice with real commands until the behavior is automatic. The exam often rewards precise stream handling more than memorized theory.
