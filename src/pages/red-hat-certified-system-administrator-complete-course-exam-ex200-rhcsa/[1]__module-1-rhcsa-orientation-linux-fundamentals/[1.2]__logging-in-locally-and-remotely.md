## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Local and remote access overview**

RHEL systems support both direct console access and remote access over the network. For RHCSA tasks, you need to know how to log in locally, switch users safely, and recover access when normal login fails. The exam often tests whether you can identify the right access path quickly, especially when SSH is unavailable or a system is misconfigured.

Local access is useful when:
- You are at the machine or using a virtual console.
- Network services are down.
- You need recovery access after a bad SSH or firewall change.

Remote access is useful when:
- You administer the system from another host.
- You need to verify SSH service, firewall, and authentication settings.
- You want to perform routine administration without physical access.

**Local console access and virtual terminals**

On RHEL, the graphical login and text consoles are exposed through virtual terminals (VTs). The exact key mapping can vary by version and desktop/session type, so do not memorize a single fixed GUI key as universal. In practice, you should know how to move between text and graphical sessions using the console switch keys available on the system.

Common console concepts:
- **TTY** means a text terminal, usually a virtual console such as `tty1`, `tty2`, and so on.
- **VT** means virtual terminal, the kernel-managed console sessions.
- The graphical session may be on a different VT depending on the environment and display manager.

Useful habits:
- Use the console when SSH is unavailable.
- Confirm which VT is active rather than assuming a fixed mapping.
- Return to the login screen or desktop using the system’s console switch keys.

> On modern RHEL systems, the graphical session is not reliably tied to one fixed function key across all environments. For exam scenarios, focus on the concept of switching between VTs rather than memorizing a single universal GUI key.

**Logging in locally**

A local login uses the system console, a physical keyboard and display, or a VM console. After authentication, you receive a shell based on the account’s configured shell.

Typical local login flow:
1. Select the console or graphical login screen.
2. Enter the username and password.
3. Verify the shell prompt or desktop session.
4. Use `whoami`, `id`, or `pwd` to confirm identity and context.

Example:
```bash
whoami
id
pwd
```

A local login is often the fastest way to recover from network misconfiguration because it bypasses SSH, firewalld, and routing issues.

**Switching users after login**

After logging in, you often need elevated privileges. RHCSA tasks commonly require you to understand the difference between switching to a root login shell and running a single command as root.

- `su` starts a shell as another user, usually root.
- `su -` starts a login shell for the target user and loads that user’s environment.
- `sudo` runs a command as another user, usually root, based on policy.
- `sudo -i` starts an interactive root login shell.
- `sudo -s` starts a shell with root privileges but does not fully simulate a login shell.

Examples:
```bash
su -
```

```bash
sudo -i
```

```bash
sudo systemctl status sshd
```

Practical differences:
- Use `su -` when you want root’s login environment, including root’s `PATH` and shell initialization files.
- Use `sudo -i` when your account is allowed to use `sudo` and you want an interactive root session.
- Use `sudo` for one-off administrative commands.

**Remote login with SSH**

Secure Shell (SSH) is the standard remote access method on RHEL. It encrypts the session and supports password or key-based authentication.

Basic client syntax:
```bash
ssh user@server.example.com
```

If the SSH server listens on a nondefault port:
```bash
ssh -p 2222 user@server.example.com
```

Useful client options:
```bash
ssh -v user@server.example.com
ssh -i ~/.ssh/id_ed25519 user@server.example.com
```

- `-v` enables verbose output and helps you diagnose authentication and connection problems.
- `-i` specifies a private key file.

When you connect for the first time, SSH may prompt you to trust the host key fingerprint. Verify it through a trusted channel before accepting it in production environments.

**SSH server basics on the managed host**

For remote login to work, the server side must be ready. On RHEL, the OpenSSH server package provides the `sshd` daemon.

Check installation:
```bash
rpm -q openssh-server
```

Check service state:
```bash
systemctl status sshd
```

Enable and start the service:
```bash
sudo systemctl enable --now sshd
```

If the service is already installed but not running:
```bash
sudo systemctl start sshd
```

If you change configuration that requires a full reread, restart the service:
```bash
sudo systemctl restart sshd
```

> Do not assume `reload` is enough for every `sshd_config` change. Some settings require a restart, and a reload may not apply all changes as expected.

Validate configuration before restarting:
```bash
sudo sshd -t
```

This syntax check is valuable because it catches many configuration errors before you lock yourself out.

**Firewall access for SSH**

A common cause of remote login failure is a blocked firewall. On RHEL, `firewalld` often controls access to SSH.

Check the active firewall rules:
```bash
sudo firewall-cmd --list-all
```

Allow SSH through the firewall:
```bash
sudo firewall-cmd --permanent --add-service=ssh
sudo firewall-cmd --reload
```

Verify the service is allowed:
```bash
sudo firewall-cmd --query-service=ssh
```

If SSH uses a custom port, you must allow that port instead of, or in addition to, the default service:
```bash
sudo firewall-cmd --permanent --add-port=2222/tcp
sudo firewall-cmd --reload
```

**SSH key-based authentication**

Key-based authentication is preferred for secure and convenient access. The client keeps a private key, and the server stores the matching public key in the user’s `~/.ssh/authorized_keys` file.

Generate a key pair:
```bash
ssh-keygen -t ed25519
```

Copy the public key to the server:
```bash
ssh-copy-id user@server.example.com
```

Typical server-side location:
```bash
~/.ssh/authorized_keys
```

Permissions matter:
- `~/.ssh` should usually be `700`.
- `authorized_keys` should usually be `600`.
- The user’s home directory and `.ssh` path must be accessible without overly permissive ownership or modes.

Example permission fix:
```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
chown -R user:user ~/.ssh
```

If SELinux labels are wrong, fix the specific SSH files or directory rather than relabeling the entire home directory:
```bash
restorecon -Rv ~/.ssh
```

For a single file:
```bash
restorecon -v ~/.ssh/authorized_keys
```

**Troubleshooting SSH login failures**

When remote login fails, use a structured approach. Do not rely on `ping` as your primary test because ICMP may be blocked even when SSH works.

A better checklist:
1. Confirm the host is reachable by name or IP.
2. Test the SSH port directly.
3. Verify `sshd` is installed, enabled, and running.
4. Check firewall rules.
5. Check authentication, permissions, and SELinux labels.
6. Review logs if needed.

Port test examples:
```bash
ssh -v user@server.example.com
```

```bash
nc -vz server.example.com 22
```

If `nc` is unavailable, use SSH verbose mode first. It often reveals whether the failure is network, key exchange, authentication, or permission related.

Common failure points:
- `sshd` is not running.
- Port 22 is blocked by `firewalld` or an external firewall.
- The user account is locked or has an invalid shell.
- `authorized_keys` has incorrect permissions.
- SELinux labels on `.ssh` content are incorrect.
- The server is listening on a nondefault port.

Useful server-side checks:
```bash
systemctl is-active sshd
ss -tlnp | grep sshd
journalctl -u sshd --since "10 minutes ago"
```

**Local recovery workflow for RHCSA**

If you cannot log in remotely, use the local console or VM console to recover access. This is a core RHCSA skill because many exam scenarios intentionally break remote access.

Recovery options:
- Use the root console if you still have physical or virtual console access.
- Boot into rescue mode or emergency mode if normal boot or login fails.
- Use single-user style recovery when the system is otherwise inaccessible.

Typical recovery paths:
- **Rescue mode**: boots a minimal environment for repair tasks.
- **Emergency mode**: provides a minimal shell with the root filesystem often mounted read-only or minimally mounted.
- **Single-user style access**: used to repair configuration, reset passwords, or fix boot issues.

In a recovery shell, you may need to remount the root filesystem read-write:
```bash
mount -o remount,rw /
```

Then inspect and repair the problem:
```bash
vi /etc/ssh/sshd_config
systemctl restart sshd
```

If you need to reset a password from recovery mode:
```bash
passwd username
```

If SELinux relabeling is required after a repair, use the appropriate targeted relabel or a full relabel only when necessary.

> Use recovery mode carefully. A broad relabel or broad filesystem change can take longer than needed and may hide the real issue you are supposed to fix.

**Using `su` and `sudo` in real sessions**

You should be comfortable moving between normal user and administrative contexts during both local and remote sessions.

Examples:
```bash
su -
```

```bash
sudo -i
```

```bash
sudo systemctl restart sshd
```

Practical guidance:
- Use `su -` when you are already on the system and know the root password.
- Use `sudo` when your account is configured for delegated administration.
- Use `sudo -i` when you need a root shell that behaves like a login session.
- Prefer `sudo` for single commands to reduce accidental changes.

Environment differences matter:
- `su -` loads root’s login environment.
- `sudo` usually preserves only a limited environment.
- `sudo -i` gives you a root login shell and is often the closest to a direct root login experience.

**Exam-focused command checklist**

Memorize this compact workflow for login and access problems:

1. Check local access first if remote login fails.
2. Verify SSH service:
   ```bash
   systemctl status sshd
   ```
3. Confirm SSH is listening:
   ```bash
   ss -tlnp | grep :22
   ```
4. Check firewall access:
   ```bash
   firewall-cmd --list-services
   firewall-cmd --query-service=ssh
   ```
5. Test the SSH connection with verbose output:
   ```bash
   ssh -v user@host
   ```
6. Validate SSH configuration:
   ```bash
   sshd -t
   ```
7. Restart SSH if needed:
   ```bash
   systemctl restart sshd
   ```
8. Fix permissions and labels in `~/.ssh` if key login fails:
   ```bash
   chmod 700 ~/.ssh
   chmod 600 ~/.ssh/authorized_keys
   restorecon -Rv ~/.ssh
   ```

**Practical examples**

Remote login with a key:
```bash
ssh -i ~/.ssh/id_ed25519 devops@192.0.2.10
```

Switch to root after login:
```bash
sudo -i
```

Check which user and shell context you are in:
```bash
whoami
echo $SHELL
id
```

Allow SSH through the firewall and enable the service:
```bash
sudo systemctl enable --now sshd
sudo firewall-cmd --permanent --add-service=ssh
sudo firewall-cmd --reload
```

Fix a broken SSH configuration safely:
```bash
sudo sshd -t
sudo systemctl restart sshd
```

**Key takeaways**

- Local console access is your fallback when remote access fails.
- SSH is the standard remote login method, but it depends on service, firewall, and authentication being correct.
- Use `ssh -v` and direct port checks instead of relying on `ping`.
- Know the difference between `su -`, `sudo`, and `sudo -i`.
- For RHCSA recovery, be ready to use the root console, rescue mode, or emergency mode to regain access.
- When fixing SSH key issues, target the `.ssh` directory and related files, not the entire home directory.
