## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Why text processing matters**

On RHCSA, you often need to inspect logs, extract fields, count events, and verify configuration output quickly from the shell. These tools let you transform plain text without writing scripts, which is exactly the kind of practical workflow the exam expects.

The core idea is simple:

- `grep` filters lines by pattern.
- `cut` extracts columns or character ranges.
- `sort` orders lines for comparison or grouping.
- `uniq` collapses adjacent duplicates and counts repeats.
- `tee` copies output to both the screen and a file.

Used together, they form a fast pipeline for troubleshooting and reporting.

**`grep`: find matching lines**

`grep` searches text line by line and prints lines that match a pattern. It is one of the most important RHCSA tools because logs, configs, and command output are usually line-oriented.

Common uses:

- Search for a literal string.
- Use regular expressions for flexible matching.
- Ignore case when the exact capitalization is unknown.
- Show line numbers for easier follow-up.

Examples:

```bash
grep 'sshd' /var/log/secure
grep -i 'failed' /var/log/secure
grep -n 'PermitRootLogin' /etc/ssh/sshd_config
grep -v '^#' /etc/fstab
```

Useful options:

- `-i` ignores case.
- `-n` prints line numbers.
- `-v` inverts the match and shows nonmatching lines.
- `-r` or `-R` searches recursively through directories.
- `-E` enables extended regular expressions.
- `-F` treats the pattern as a fixed string, which is useful when you do not need regex syntax.
- `-w` matches whole words.
- `-c` counts matching lines instead of printing them.
- `-o` prints only the matching part of each line.

Regular expression basics for the exam:

- `.` matches any single character.
- `*` repeats the previous token zero or more times.
- `^` anchors the match to the start of a line.
- `$` anchors the match to the end of a line.
- `[]` defines a character class.
- `grep -E` lets you use extended regex operators like `+`, `?`, and `{m,n}` without escaping them.

Examples:

```bash
grep -E '^[0-9]{3}' file.txt
grep -E '^(error|warning)' /var/log/messages
grep -E 'sshd|sudo' /var/log/secure
```

> In basic `grep`, `{}` are not special unless escaped, so `grep '^[0-9]{3}' file.txt` matches the literal characters `{3}` rather than three digits. Use `grep -E` for `{3}` style repetition.

Practical RHCSA note:

- On RHEL systems, authentication logs are commonly in `/var/log/secure`.
- For systemd-managed services and many modern troubleshooting tasks, `journalctl` is often the better source than a flat log file.

Examples:

```bash
journalctl -u sshd
journalctl -p err
journalctl --since today | grep -i failed
```

**`cut`: extract fields or characters**

`cut` is useful when output is consistently structured. It can extract delimited fields or fixed character positions.

Two main modes:

- Field extraction with `-d` and `-f`
- Character extraction with `-c`

Examples:

```bash
cut -d: -f1 /etc/passwd
cut -d: -f1,3 /etc/passwd
cut -c1-10 file.txt
```

Important behavior:

- `cut` works best with a stable delimiter such as `:` or `,`.
- It is less reliable with variable spacing because repeated spaces can create empty fields.
- It does not understand quoted CSV rules or complex log formats.

Good use cases:

- `/etc/passwd` and `/etc/group` use `:` delimiters.
- Simple CSV-like data with a consistent delimiter.
- Fixed-width text where character positions are known.

Examples:

```bash
cut -d: -f1,3 /etc/passwd
cut -d, -f2 sales.csv
cut -c1-8 report.txt
```

Avoid using `cut` when the data is not truly columnar. For example, many log files are better handled with `grep`, `awk`, or `journalctl` filters than with `cut`.

**`sort`: order lines for comparison and grouping**

`sort` arranges lines in a predictable order. This is often a required step before using `uniq`, because `uniq` only removes adjacent duplicates.

Common options:

- `-n` sorts numerically.
- `-r` reverses the order.
- `-u` removes duplicate lines after sorting.
- `-k` sorts by a specific field.
- `-t` sets the field delimiter.
- `-f` ignores case.
- `-h` performs human-numeric sorting for values like `2K`, `10M`, and `1G`.

Examples:

```bash
sort names.txt
sort -n numbers.txt
sort -r names.txt
sort -t: -k3,3 /etc/passwd
sort -u users.txt
```

Field-based sorting example:

```bash
sort -t: -k3,3 /etc/passwd
```

This sorts by the third colon-delimited field, which is the UID in `/etc/passwd`.

Practical notes:

- Use `-n` for plain numbers; otherwise, `sort` compares text lexicographically.
- Use `-k` with `-t` when sorting structured records.
- `-h` is helpful for human-readable sizes, but it is not a general replacement for numeric sorting.

Example of numeric vs text sorting:

```bash
printf "2\n10\n1\n" | sort
printf "2\n10\n1\n" | sort -n
```

**`uniq`: collapse adjacent duplicates and count repeats**

`uniq` removes or summarizes duplicate lines, but only when duplicates are adjacent. That is why you usually run `sort` first.

Common options:

- `-c` prefixes each line with the number of occurrences.
- `-d` shows only duplicate lines.
- `-u` shows only unique lines.
- `-i` ignores case when comparing lines.

Examples:

```bash
sort names.txt | uniq
sort names.txt | uniq -c
sort names.txt | uniq -d
sort names.txt | uniq -u
```

Counting repeated values is a common exam pattern:

```bash
cut -d: -f1 /etc/passwd | sort | uniq -c
```

This counts how many times each username appears in the extracted list, which is usually `1` for `/etc/passwd` but demonstrates the standard pipeline pattern.

If you want the most frequent entries first:

```bash
cut -d: -f1 /etc/passwd | sort | uniq -c | sort -nr
```

Why the order matters:

- `sort` groups identical lines together.
- `uniq` then sees adjacent duplicates and can count or remove them.
- Without sorting, `uniq` only compares neighboring lines and misses separated duplicates.

**`tee`: split output to screen and file**

`tee` reads from standard input and writes the same output to standard output and one or more files. It is useful when you want to inspect a pipeline interactively while also saving a record.

Basic example:

```bash
ls -l /etc | tee /tmp/etc-listing.txt
```

Append instead of overwrite:

```bash
echo "new entry" | tee -a /tmp/log.txt
```

Common RHCSA uses:

- Save command output while still seeing it on screen.
- Capture troubleshooting output for later review.
- Feed a pipeline and preserve intermediate results.

Example with a pipeline:

```bash
journalctl -u sshd | grep -i failed | tee /tmp/sshd-failures.txt
```

Useful options:

- `-a` appends to the file instead of truncating it.
- `-i` ignores interrupts, though this is rarely needed for exam tasks.

`tee` is especially helpful when a command produces output you want to both inspect and reuse. It is also useful in scripts when you need logging without losing terminal visibility.

**Common pipelines and exam-style patterns**

These tools are often combined. The exam usually tests whether you can choose the right tool for the structure of the data.

Examples:

```bash
grep -i 'error' /var/log/secure | sort | uniq -c
cut -d: -f1 /etc/passwd | sort | uniq
journalctl -u sshd | grep -i 'failed' | tee /tmp/failed-ssh.txt
```

A realistic workflow:

1. Filter the relevant lines with `grep`.
2. Extract the needed field with `cut` if the data is delimiter-based.
3. Sort the results so duplicates are adjacent.
4. Use `uniq -c` to count repeated values.
5. Save the final output with `tee` if you need a copy.

Example counting usernames from `/etc/passwd`:

```bash
cut -d: -f1 /etc/passwd | sort | uniq -c | sort -nr
```

Example extracting and saving SSH-related failures from the system journal:

```bash
journalctl -u sshd --since today | grep -i failed | tee /tmp/sshd-failures.txt
```

**Choosing the right tool**

| Task | Best tool | Why |
|---|---|---|
| Find lines containing a pattern | `grep` | Fast line filtering |
| Extract a known field | `cut` | Simple delimiter-based parsing |
| Order output | `sort` | Groups or ranks lines |
| Count duplicates | `uniq -c` | Summarizes adjacent repeated lines |
| Save and display output | `tee` | Duplicates standard output |

**Practical cautions**

- `grep` matches lines, not files, unless you use recursive options.
- `cut` is not a general parser; it works best with simple, consistent delimiters.
- `uniq` does not find all duplicates unless the input is sorted first.
- `sort -n` is for numbers; plain `sort` is lexical.
- `tee` writes to files immediately, so be careful not to overwrite important output accidentally.

> When working with logs on RHEL, prefer the correct source for the system: `/var/log/secure` for many authentication events and `journalctl` for systemd journal data.

**Quick command reference**

```bash
grep -i 'pattern' file
grep -E 'regex' file
cut -d: -f1 file
cut -c1-10 file
sort -n file
sort -t: -k3,3 file
uniq -c
tee output.txt
tee -a output.txt
```

**Exam mindset**

For RHCSA, focus on recognizing the structure of the data before choosing a command:

- Use `grep` when you need to filter by content.
- Use `cut` when the data has a reliable delimiter or fixed width.
- Use `sort` before `uniq` when you need counts or deduplication.
- Use `tee` when you need to preserve output while continuing to work in the terminal.

If a task feels awkward with `cut`, the data may not be suitable for it. In that case, reconsider the source or use a more appropriate text-processing tool.
