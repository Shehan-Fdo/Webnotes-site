## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Path fundamentals**

A path tells Linux where a file or directory lives in the filesystem hierarchy. You use paths constantly with commands such as `cd`, `ls`, `cp`, `mv`, `rm`, `cat`, `touch`, and `mkdir`. Understanding the difference between **absolute** and **relative** paths helps you navigate quickly and avoid operating on the wrong file.

Linux uses a single rooted directory tree that starts at `/`. Every path is resolved from either that root or from your current working directory, depending on the path type you provide.

- **Absolute path**: starts at `/` and names the full location.
- **Relative path**: starts from your current working directory.
- `.` means the current directory.
- `..` means the parent directory.
- `~` expands to your home directory in most shells.

**Absolute paths**

An absolute path gives the complete location of a file or directory, beginning at the root directory `/`. It does not depend on where you currently are in the filesystem, so it is predictable and safe for scripts and administrative tasks.

Examples:

```bash
/etc/ssh/sshd_config
/home/student/Documents/report.txt
/usr/bin/ls
```

Use absolute paths when you need reliability, especially in automation, system administration, and troubleshooting. If a script runs from an unknown directory, absolute paths prevent ambiguity.

- **Predictable**: the same path always points to the same location.
- **Script-friendly**: avoids dependence on the current directory.
- **Safer for administration**: reduces the chance of modifying the wrong file.

Example with commands:

```bash
cat /etc/fstab
cp /var/log/messages /tmp/messages.backup
ls -l /usr/lib/systemd/system/
```

**Relative paths**

A relative path is interpreted from your current working directory, which you can check with `pwd`. Relative paths are shorter and convenient during interactive work, especially when you are already inside the relevant directory tree.

Examples:

```bash
Documents/report.txt
./script.sh
../shared/config.ini
```

Relative paths often use `.` and `..` to move within the tree:

- `.` refers to the current directory.
- `./file.txt` means a file in the current directory.
- `..` refers to the parent directory.
- `../file.txt` means a file one level up.

Example workflow:

```bash
pwd
/home/student/projects

ls
app  docs

cd docs
cat notes.txt

cd ..
cp docs/notes.txt app/
```

Relative paths are useful when you are exploring a project tree or working inside a directory structure where repeated full paths would be cumbersome.

**How the shell resolves paths**

The shell resolves a path by checking whether it begins with `/`.

- If it begins with `/`, the shell treats it as absolute.
- If it does not, the shell treats it as relative to the current directory.

This matters for commands that accept file arguments and for shell redirection.

```bash
echo "hello" > output.txt
echo "hello" > /tmp/output.txt
```

In the first command, `output.txt` is created in the current directory. In the second, the file is created in `/tmp` regardless of where you are.

**Current working directory**

Your current working directory is the directory the shell uses as the starting point for relative paths. Check it with `pwd`.

```bash
pwd
```

Example:

```bash
cd /etc
pwd
/etc

cat hosts
```

Here, `cat hosts` works because `hosts` is resolved as `/etc/hosts` from the current directory.

If you move to another directory, the same relative path may refer to a different file or fail entirely.

```bash
cd /var
cat hosts
```

This fails unless `/var/hosts` exists, because the shell now looks in `/var`.

**Common path shortcuts**

Linux shells support several useful path shortcuts and symbols.

- `~` expands to your home directory, such as `/home/student`.
- `~/Downloads` points to a directory under your home directory.
- `.` refers to the current directory.
- `..` refers to the parent directory.
- `-` in `cd -` returns to the previous directory.

Examples:

```bash
cd ~
cd ~/Documents
ls ./scripts
ls ../shared
cd -
```

These shortcuts are especially useful when moving between nearby directories or when you want to avoid typing long absolute paths.

**Practical comparison**

| Feature | Absolute path | Relative path |
|---|---|---|
| Starts with `/` | Yes | No |
| Depends on current directory | No | Yes |
| Best for scripts | Yes | Sometimes |
| Best for quick interactive work | Sometimes | Yes |
| Risk of ambiguity | Low | Higher |

Use absolute paths when correctness matters more than brevity. Use relative paths when you are already in the right location and want faster typing.

**Real-world command examples**

Administrative tasks often mix both path types. For example, you may navigate with relative paths but use absolute paths for critical files.

```bash
cd /etc
ls ssh
sudo cp ssh/sshd_config /root/sshd_config.backup
```

Here, `ssh/sshd_config` is relative to `/etc`, while `/root/sshd_config.backup` is absolute.

Another example:

```bash
mkdir -p /tmp/lab/config
cd /tmp/lab
touch config/app.conf
```

The directory is created with an absolute path, then the file is created with a relative path from `/tmp/lab`.

**Why path choice matters**

Choosing the wrong path type can cause subtle errors:

- A relative path may point to the wrong file if you are in the wrong directory.
- A script that assumes a specific working directory may fail when run from elsewhere.
- Administrative commands can affect unintended files if you misread the current location.

> Be careful with destructive commands such as `rm`, `mv`, and `cp`. Always confirm the path before you run them, especially when using relative paths.

**Troubleshooting path issues**

If a command fails with “No such file or directory,” check these items:

1. Confirm your current directory with `pwd`.
2. Verify the path spelling and case.
3. Decide whether the command expects an absolute or relative path.
4. Use `ls` to inspect the target directory.
5. Expand shortcuts mentally or explicitly, such as `~` to your home directory.

Examples:

```bash
pwd
ls -l
ls -l /etc/ssh
ls -l ./ssh
```

If `./ssh` fails but `/etc/ssh` exists, the issue is likely that you are not in `/etc`.

**Practical shell patterns**

You can combine path types in everyday work:

```bash
cd /var/log
grep "error" messages
cp messages ~/Desktop/messages.log
```

You can also use relative paths in scripts when the script intentionally operates within a known directory structure, but absolute paths are usually more robust for system-level automation.

Example script snippet:

```bash
#!/bin/bash

LOG_DIR="/var/log"
BACKUP_DIR="/root/log-backups"

mkdir -p "$BACKUP_DIR"
cp "$LOG_DIR/messages" "$BACKUP_DIR/messages.$(date +%F)"
```

This script uses absolute paths to avoid dependence on where it is launched from.

**Key takeaways**

- **Absolute paths** begin at `/` and are independent of the current directory.
- **Relative paths** depend on the current working directory.
- `pwd` shows where relative paths start.
- `.` and `..` help you move within the directory tree.
- Use absolute paths for reliability and scripts; use relative paths for convenience during interactive work.
