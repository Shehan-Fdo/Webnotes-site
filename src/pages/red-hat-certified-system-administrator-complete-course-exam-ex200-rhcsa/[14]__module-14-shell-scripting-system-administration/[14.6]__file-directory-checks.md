## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**File and directory checks: what you verify and why**

File and directory checks are the basic validation steps you use to confirm that a path exists, has the expected type, and is accessible with the right ownership and permissions. On RHCSA tasks, these checks help you avoid destructive mistakes, confirm configuration changes, and troubleshoot why a command cannot read, write, or execute a file.

You usually verify:
- **Existence** of a path before acting on it.
- **Type** of object: regular file, directory, symbolic link, device, socket, or FIFO.
- **Permissions** and access mode for the current user.
- **Ownership** and group membership.
- **Metadata** such as size, timestamps, and link count.
- **Content state** when you need to confirm a file was created or updated correctly.

A good workflow is to inspect first, change second. That habit matters on the exam because many tasks require you to preserve existing data and make only the minimum necessary change.

**Core commands for checking files and directories**

The most common tools are `ls`, `stat`, `file`, `test`, `find`, `pwd`, `cd`, `readlink`, and `namei`. Each answers a different question.

- `ls -l` shows permissions, ownership, size, and timestamps.
- `ls -ld` shows details about a directory itself instead of its contents.
- `stat` gives detailed metadata, including inode number and access times.
- `file` identifies the file type by content, not just by name.
- `test` or `[ ]` checks conditions in scripts and shell commands.
- `find` locates files and can filter by type, size, ownership, or permissions.
- `pwd` confirms your current working directory.
- `readlink -f` resolves a symbolic link to its final target.
- `namei -l` traces each path component and its permissions.

Example inspection commands:

```bash
pwd
ls -ld /etc /etc/ssh/sshd_config
stat /etc/ssh/sshd_config
file /etc/ssh/sshd_config
readlink -f /etc/alternatives/editor
namei -l /var/log/messages
```

**Understanding file types**

Linux treats many objects as files, but the type determines how you interact with them. Checking the type prevents mistakes such as trying to edit a directory or execute a data file.

Common types you should recognize:
- **Regular file**: stores text or binary data.
- **Directory**: contains directory entries for files and subdirectories.
- **Symbolic link**: points to another path.
- **Block device**: buffered device such as a disk.
- **Character device**: unbuffered device such as a terminal.
- **FIFO**: named pipe for process communication.
- **Socket**: endpoint for interprocess communication.

Use `ls -l` and `stat` to inspect type indicators:
- `-` regular file
- `d` directory
- `l` symbolic link
- `b` block device
- `c` character device
- `p` FIFO
- `s` socket

Example:

```bash
ls -l /dev/null /tmp /etc/passwd /bin/sh
stat /dev/null
file /bin/sh
```

`file` is especially useful when a filename is misleading. A file named `backup.txt` may actually contain compressed data or a script. The command reads signatures and reports the likely type.

**Permissions and access checks**

Permissions determine whether a user can read, write, or execute a file, or list and traverse a directory. On RHCSA, you often need to confirm whether a failure is caused by permissions rather than by a missing file.

Permission bits:
- **r** read
- **w** write
- **x** execute for files, traverse for directories

For directories:
- `r` lets you list names if `x` also allows traversal.
- `w` lets you create, delete, or rename entries if `x` also allows traversal.
- `x` lets you enter or access entries by name.

For files:
- `r` lets you read content.
- `w` lets you modify content.
- `x` lets you run the file as a program or script.

Check permissions with:

```bash
ls -ld /var/log
ls -l /usr/bin/grep
stat -c '%A %a %U %G %n' /etc/passwd
```

A practical way to test access is with shell conditionals:

```bash
test -r /etc/passwd && echo "readable"
test -w /etc/hosts && echo "writable"
test -x /usr/bin/bash && echo "executable"
```

Or with the bracket form:

```bash
[ -d /etc/ssh ] && echo "directory exists"
[ -f /etc/ssh/sshd_config ] && echo "regular file exists"
```

**Ownership and group checks**

Ownership matters because many administrative tasks depend on the correct user and group. A file may exist and have the right permissions, but still fail a service check if the owner is wrong.

Check ownership with:
- `ls -l`
- `stat`
- `find -user`
- `find -group`

Example:

```bash
ls -l /etc/ssh/sshd_config
stat -c '%U %G %n' /etc/ssh/sshd_config
find /var/www -user apache -o -group apache
```

When you troubleshoot access, compare:
- The file owner and group.
- The current user and supplementary groups.
- The directory permissions on every parent directory.

A file can be readable, but still inaccessible if one parent directory lacks execute permission.

**Path validation and parent directory traversal**

A path is only usable if every component is accessible. This is why a file can appear correct in `ls -l`, yet still fail when opened by a program.

Use `namei -l` to inspect each path element:

```bash
namei -l /var/lib/pgsql/data/postgresql.conf
```

This is useful when:
- A nested directory blocks access.
- A symbolic link points somewhere unexpected.
- A parent directory has restrictive permissions.
- A service account cannot reach a file even though the file itself looks correct.

You can also verify the current location before using relative paths:

```bash
pwd
cd /etc/ssh
ls -l sshd_config
```

Relative paths depend on the current working directory, so confirm it before running commands that create, copy, or edit files.

**Timestamps and change tracking**

Timestamps help you confirm whether a file changed when expected. Linux commonly tracks:
- **atime**: last access time
- **mtime**: last modification time
- **ctime**: last metadata change time

Use `stat` to inspect them:

```bash
stat /etc/hosts
```

Typical uses:
- Confirm a configuration file was edited recently.
- Check whether a backup file is newer than the original.
- Determine whether permissions or ownership changed even if content did not.

Be careful: `ctime` is not creation time. It changes when metadata changes, such as permissions, ownership, or link count.

**Links and inode checks**

Files can have multiple names. Understanding links helps you avoid editing the wrong object or deleting the only remaining reference.

Types of links:
- **Hard link**: another directory entry pointing to the same inode.
- **Symbolic link**: a separate file that stores a path to another file.

Check link count and inode with:

```bash
ls -li /etc/passwd
stat /etc/passwd
ls -l /usr/bin/python3
readlink /usr/bin/python3
```

Why this matters:
- A hard link keeps data alive until the last link is removed.
- A symbolic link can break if the target moves or is deleted.
- Two names with the same inode are the same underlying file.

Use `find` to locate links when needed:

```bash
find / -samefile /etc/passwd 2>/dev/null
find / -type l -lname '*sshd*' 2>/dev/null
```

**Directory checks and safe navigation**

Directory checks are not just about existence. You also need to confirm that the directory is the correct one and that you can safely work inside it.

Useful commands:
- `ls -ld /path` to inspect the directory itself.
- `cd /path` to verify it is accessible.
- `pwd` to confirm where you are.
- `find /path -maxdepth 1` to inspect contents without deep recursion.

Example:

```bash
ls -ld /etc /etc/ssh
cd /etc/ssh && pwd
find /var/log -maxdepth 1 -type f
```

When you are about to create or modify files, check:
- The directory exists.
- You have write permission in the directory.
- The directory is not a symlink to an unexpected location.
- The filesystem is mounted read-only if writes fail unexpectedly.

**Using `find` for verification**

`find` is one of the most useful verification tools because it can search by many attributes. It is especially helpful when you need to confirm that a file exists somewhere in a tree or that a directory contains the expected object type.

Common patterns:

```bash
find /etc -type f -name 'sshd_config'
find /var/log -type d -name 'httpd'
find /home -user student
find /tmp -perm -002
find /var -size +10M
```

Use `find` when:
- You do not know the exact location.
- You need to verify ownership or permissions across many files.
- You want to confirm that a directory tree contains only the expected types.

For safer searches on large trees, limit depth:

```bash
find /etc -maxdepth 2 -type f -name '*.conf'
```

**Shell tests for scripts and quick checks**

The `test` command and its `[ ]` form are standard shell tools for checking file conditions in scripts and one-liners. They are common in exam tasks where you need to branch based on whether a file or directory exists.

Common file tests:
- `-e` exists
- `-f` regular file
- `-d` directory
- `-L` symbolic link
- `-r` readable
- `-w` writable
- `-x` executable
- `-s` non-empty

Examples:

```bash
if [ -f /etc/ssh/sshd_config ]; then
  echo "config exists"
fi

if [ -d /var/www/html ] && [ -w /var/www/html ]; then
  echo "web root is writable"
fi
```

Use these checks in scripts to avoid errors before file operations.

**Practical troubleshooting workflow**

When a file or directory behaves unexpectedly, use a consistent sequence:
1. Confirm the path exists.
2. Confirm the object type.
3. Check permissions on the file and every parent directory.
4. Check ownership and group.
5. Inspect timestamps and recent changes.
6. Resolve symbolic links if the path is indirect.
7. Verify the current working directory if using relative paths.

Example workflow:

```bash
pwd
ls -ld /opt/app /opt/app/config
namei -l /opt/app/config/settings.ini
stat /opt/app/config/settings.ini
file /opt/app/config/settings.ini
```

This approach helps you distinguish between:
- Missing file
- Wrong file type
- Permission denial
- Broken symlink
- Incorrect path
- Unexpected metadata change

> If a command fails, do not assume the file is missing. On Linux, access failures are often caused by a parent directory permission, a broken link, or a wrong working directory.

**Real-world examples**

Checking whether a service configuration file is present and readable:

```bash
[ -f /etc/ssh/sshd_config ] && [ -r /etc/ssh/sshd_config ] && echo "ready"
```

Verifying a directory before copying files into it:

```bash
if [ -d /backup ] && [ -w /backup ]; then
  cp /etc/hosts /backup/
fi
```

Finding recently modified files in a directory:

```bash
find /etc -type f -mtime -1
```

Confirming a symlink target:

```bash
ls -l /etc/localtime
readlink -f /etc/localtime
```

Inspecting a file before editing it:

```bash
stat /etc/fstab
sed -n '1,20p' /etc/fstab
```

**Common exam pitfalls**

- Checking only the file and ignoring parent directory permissions.
- Using `ls -l` on a directory when you meant `ls -ld`.
- Assuming a filename tells you the file type.
- Forgetting that a symlink may point elsewhere.
- Confusing `ctime` with creation time.
- Using relative paths without confirming `pwd`.
- Treating a readable file as usable without checking execute permission on directories in the path.

**Quick reference**

- `ls -l`: file details
- `ls -ld`: directory details
- `stat`: metadata and timestamps
- `file`: content-based type detection
- `find`: search and verification
- `test` / `[ ]`: conditional checks
- `namei -l`: path component inspection
- `readlink -f`: resolve final target
- `pwd`: current directory

**Summary**

File and directory checks are a foundational RHCSA skill because they let you verify state before making changes. Use the right tool for the question you are asking: `ls` for quick inspection, `stat` for metadata, `file` for type, `namei` for path traversal, and `find` for broader verification. When you combine these checks with permission and ownership awareness, you can troubleshoot safely and efficiently.
