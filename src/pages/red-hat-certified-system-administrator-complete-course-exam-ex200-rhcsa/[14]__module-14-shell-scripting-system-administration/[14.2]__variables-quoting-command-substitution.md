**Shell variables and why they matter**

Bash variables let you store values for reuse in scripts, one-liners, and interactive sessions. On RHCSA, you use them to make commands readable, reduce repetition, and pass data between shell constructs such as loops, conditionals, and command substitutions.

A variable assignment has no spaces around `=`:

```bash
name="alice"
count=3
path=/usr/local/bin
```

- `name="alice"` stores a string value.
- `count=3` stores a numeric-looking value, but Bash still treats it as a string unless you use arithmetic expansion.
- `path=/usr/local/bin` is a plain assignment and does not run a command.

To read a variable, prefix it with `$`:

```bash
echo "$name"
echo "$count"
```

Use quotes when expanding variables unless you specifically want word splitting or pathname expansion.

**Variable scope and environment**

A shell variable exists only in the current shell unless you export it. Exporting makes it available to child processes started from that shell.

```bash
project="rhce"
export project
bash -c 'echo "$project"'
```

- `project="rhce"` creates a shell variable.
- `export project` marks it for inheritance by child processes.
- `bash -c 'echo "$project"'` shows that the child shell receives the exported value.

You can inspect exported variables with tools such as `env` or `printenv`:

```bash
env | grep '^project='
printenv project
```

Common environment variables include:

- `HOME` for your home directory
- `PATH` for command lookup directories
- `USER` for the current login name
- `SHELL` for the login shell
- `PWD` for the current working directory

> Be careful when changing `PATH`. A bad `PATH` can hide commands or make the shell unable to find basic utilities.

**Positional parameters and special parameters**

Bash scripts receive arguments through positional parameters. These are essential for RHCSA-style scripting tasks.

- `$0` is the script name or shell name.
- `$1` through `$9` are the first nine arguments.
- `${10}` and above require braces.
- `$#` is the number of positional parameters.
- `$*` expands to all positional parameters as a single word in many contexts.
- `$@` expands to all positional parameters as separate words.
- `$?` is the exit status of the last command.
- `$$` is the current shell process ID.
- `$!` is the process ID of the most recent background job.

Example script behavior:

```bash
#!/bin/bash
echo "script: $0"
echo "first arg: $1"
echo "arg count: $#"
echo "all args with \$*: $*"
echo "all args with \$@: $@"
```

When iterating over arguments, prefer `"$@"` because it preserves each argument as its own item:

```bash
for item in "$@"; do
  echo "[$item]"
done
```

- `"$*"` joins arguments into one string using the first character of `IFS` (Internal Field Separator).
- `"$@"` preserves argument boundaries, which is usually what you want in scripts.

**Quoting rules**

Quoting controls how Bash interprets special characters, spaces, and expansions. This is one of the most important safety topics in shell scripting.

Use **double quotes** to preserve spaces while still allowing variable and command expansion:

```bash
file="/var/log/messages"
echo "Log file: $file"
```

Use **single quotes** to prevent all expansion inside the quotes:

```bash
echo '$HOME is not expanded here'
```

Use **backslash escaping** for a single special character:

```bash
echo \$HOME
echo "He said \"hello\""
```

Practical effects of quoting:

- Unquoted variables can undergo word splitting on `IFS` characters and pathname expansion if they contain glob characters such as `*`, `?`, or `[]`.
- Double quotes prevent splitting and globbing, but still allow `$var`, `$(command)`, and backslash escapes for certain characters.
- Single quotes are the safest way to preserve literal text.

Example of why quoting matters:

```bash
name="Alice Smith"
echo $name
echo "$name"
```

- `echo $name` may split the value into two words.
- `echo "$name"` preserves the full string as one argument.

A more dangerous example involves globbing:

```bash
pattern='*'
echo $pattern
echo "$pattern"
```

- `echo $pattern` may expand to filenames in the current directory.
- `echo "$pattern"` prints a literal `*`.

**Command substitution**

Command substitution captures the output of a command and inserts it into another command or assignment. Use `$(...)` in modern Bash.

```bash
today=$(date +%F)
echo "Today is $today"
```

- `$(date +%F)` runs `date` and captures its standard output.
- `today=...` stores the result for later use.

You can nest command substitutions cleanly with `$(...)`:

```bash
archive="backup-$(hostname)-$(date +%Y%m%d).tar.gz"
echo "$archive"
```

Avoid legacy backticks unless you must support very old shells. Backticks are harder to read and awkward to nest:

```bash
# Prefer this
user_home=$(getent passwd "$USER" | cut -d: -f6)

# Avoid this style when possible
user_home=`getent passwd "$USER" | cut -d: -f6`
```

Common pitfalls:

- Command substitution strips trailing newlines from the captured output.
- If the command produces multiple lines, the result is still subject to word splitting if you do not quote the expansion.
- Nested backticks require escaping and are difficult to maintain.

Example with nested substitution:

```bash
stamp=$(date +%Y%m%d)
logfile="/var/log/app-${stamp}-$(hostname -s).log"
```

**Parameter expansion and useful forms**

Parameter expansion is the shell’s built-in way to transform variable values without external commands. This is faster and more reliable than piping to `sed` or `awk` for simple cases.

Basic forms:

```bash
name="rhcsa"
echo "${name}"
echo "${name:-default}"
echo "${name:=default}"
echo "${name:+alt}"
echo "${name:?name is required}"
```

- `${name}` expands the variable.
- `${name:-default}` uses `default` if `name` is unset or empty.
- `${name:=default}` assigns `default` if `name` is unset or empty.
- `${name:+alt}` expands to `alt` only if `name` is set and not empty.
- `${name:?message}` prints an error and exits the shell or script if `name` is unset or empty.

Length and slicing are also useful:

```bash
text="system"
echo "${#text}"
echo "${text:0:3}"
```

- `${#text}` returns the string length.
- `${text:0:3}` extracts a substring.

Pattern-based substitution can replace text inline:

```bash
path="/var/log/httpd/access.log"
echo "${path/log/httpd/nginx}"
echo "${path%.log}"
echo "${path##*/}"
```

- `${var/pat/repl}` replaces the first match.
- `${var//pat/repl}` replaces all matches.
- `${var%.log}` removes the shortest matching suffix.
- `${var##*/}` removes the longest matching prefix, often used to get a basename.

Indirect expansion lets you reference the value of a variable whose name is stored in another variable:

```bash
varname="HOSTNAME"
echo "${!varname}"
```

This is useful in advanced scripts, but use it carefully because it can make code harder to read.

**Readonly variables**

Use `readonly` when a value should not change after assignment. This helps protect important script constants and environment values.

```bash
readonly role="admin"
echo "$role"
```

You can also mark existing variables as readonly:

```bash
path="/srv/data"
readonly path
```

After a variable is readonly, attempts to change or unset it fail.

- `readonly` is useful for constants in scripts.
- It helps prevent accidental reassignment.
- It is not a security boundary; it only affects the current shell and its children.

**Practical RHCSA scripting examples**

Variables are often used in loops, conditionals, and argument handling. These patterns are common in exam tasks.

Loop over files safely:

```bash
for file in /etc/*.conf; do
  [ -e "$file" ] || continue
  echo "Checking $file"
done
```

- Quoting protects each filename.
- The `[ -e "$file" ]` check avoids issues when the glob does not match.

Use variables in conditionals:

```bash
service_name="sshd"

if systemctl is-active --quiet "$service_name"; then
  echo "$service_name is running"
else
  echo "$service_name is not running"
fi
```

Use script arguments:

```bash
#!/bin/bash
user="$1"

if [ -z "$user" ]; then
  echo "Usage: $0 USERNAME"
  exit 1
fi

id "$user"
```

- `$1` reads the first argument.
- `[ -z "$user" ]` checks for an empty string.
- `$0` helps print a usage line that names the script.

Process multiple arguments:

```bash
#!/bin/bash
for host in "$@"; do
  ping -c 1 "$host" >/dev/null && echo "$host reachable"
done
```

This pattern preserves spaces in hostnames or labels if they appear in arguments.

**Safe and unsafe expansion patterns**

A good rule is to quote variable expansions unless you intentionally need splitting or globbing.

Safe patterns:

```bash
cp "$src" "$dest"
mkdir -p "$dir"
grep "$pattern" "$file"
```

Unsafe patterns:

```bash
cp $src $dest
mkdir -p $dir
grep $pattern $file
```

Why unsafe patterns fail:

- Spaces in values become multiple arguments.
- Wildcards can expand to filenames.
- Empty variables can remove required arguments.
- Unexpected values can change command behavior.

> Never rely on unquoted expansion when the value may contain spaces, tabs, newlines, or glob characters.

**Exam-focused reminders**

For RHCSA, you should be comfortable with these shell behaviors:

- Assign variables without spaces around `=`.
- Quote expansions unless you need splitting.
- Use `"$@"` for argument forwarding and loops.
- Use `$(...)` instead of backticks.
- Export variables when child processes need them.
- Recognize special parameters such as `$0`, `$#`, `$?`, and `$$`.
- Use parameter expansion for simple string manipulation instead of external commands.

A compact example that combines several concepts:

```bash
#!/bin/bash
readonly prefix="backup"
target_dir="${1:-/tmp}"
stamp=$(date +%F)
outfile="${prefix}-${stamp}.txt"

echo "Writing to $target_dir/$outfile"
printf '%s\n' "created by $USER" > "$target_dir/$outfile"
```

This script shows default values, command substitution, readonly data, and safe quoting in one place.
