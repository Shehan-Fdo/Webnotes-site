## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Control flow in Bash**

Bash control flow lets you make scripts respond to conditions and repeat work safely. For RHCSA, you should understand how `if`, `case`, `for`, `while`, and `until` behave, how exit status drives decisions, and how to write readable scripts that handle common admin tasks.

In shell, most decisions depend on the exit status of a command:
- `0` means success.
- Nonzero means failure or a specific error condition.

That model is central to `if`, loops, and short-circuit operators such as `&&` and `||`.

**Exit status and command evaluation**

A shell command can be used as a condition because `if`, `while`, and `until` test the exit status of the command they run. This is not the same as a separate “test mode” for arbitrary command substitution.

- `if command; then` runs `command` and checks its exit status.
- `if $(command); then` is not a valid conditional form; the shell performs command substitution first, then tries to execute the resulting text as a command name.
- External commands still run as separate processes; shell builtins and functions run in the current shell process.

Common ways to test conditions:
- `test expression`
- `[ expression ]`
- `[[ expression ]]` in Bash
- command exit status from utilities such as `grep`, `systemctl`, or `id`

Examples:
```bash
if id alice >/dev/null 2>&1; then
  echo "alice exists"
else
  echo "alice does not exist"
fi
```

```bash
if [ -f /etc/fstab ]; then
  echo "fstab exists"
fi
```

```bash
if [[ $USER == root ]]; then
  echo "You are root"
fi
```

**The `if` statement**

Use `if` when you need a binary or multi-branch decision based on a condition. The shell evaluates the condition list, then executes the first matching branch.

Basic form:
```bash
if command; then
  commands
elif other_command; then
  commands
else
  commands
fi
```

Key points:
- `then` starts the body for a successful condition.
- `elif` adds another condition without nesting.
- `else` handles the fallback case.
- `fi` closes the block.

Practical examples:
```bash
if systemctl is-active --quiet sshd; then
  echo "sshd is running"
else
  echo "sshd is not running"
fi
```

```bash
if [ -r /etc/ssh/sshd_config ]; then
  grep '^PermitRootLogin' /etc/ssh/sshd_config
elif [ -e /etc/ssh/sshd_config ]; then
  echo "Config exists but is not readable"
else
  echo "Config file missing"
fi
```

Use `if` when:
- You need to test a file, variable, or command result.
- You want multiple conditions with different actions.
- You need compound logic with `&&` and `||`.

Compound tests:
```bash
if [ -f /etc/passwd ] && [ -r /etc/passwd ]; then
  echo "Readable passwd file"
fi
```

> Be careful with `[` and `]`: they are commands, not punctuation. You need spaces around them.

**The `case` statement**

Use `case` when one value must be matched against several patterns. It is cleaner than a long `if/elif` chain when you compare the same variable repeatedly.

Basic form:
```bash
case "$1" in
  start)
    systemctl start sshd
    ;;
  stop)
    systemctl stop sshd
    ;;
  restart|reload)
    systemctl restart sshd
    ;;
  *)
    echo "Usage: $0 {start|stop|restart|reload}"
    exit 1
    ;;
esac
```

Important behavior:
- Patterns use shell glob syntax, not regular expressions.
- Each branch usually ends with `;;`.
- `*` is the default catch-all pattern.
- Quote the variable being matched to avoid word splitting and glob expansion.

Pattern examples:
- `a*` matches values starting with `a`
- `?.txt` matches one character followed by `.txt`
- `[0-9])` matches a single digit
- `foo|bar)` matches either `foo` or `bar`

Bash-specific branch terminators:
- `;&` continues execution into the next branch body without re-testing the pattern.
- `;;&` continues to the next pattern test.

These are Bash extensions, not POSIX shell syntax, so use them only when you know the script runs under Bash.

Practical admin example:
```bash
read -r action

case "$action" in
  enable)
    systemctl enable --now sshd
    ;;
  disable)
    systemctl disable --now sshd
    ;;
  status)
    systemctl status sshd
    ;;
  *)
    echo "Unknown action: $action"
    ;;
esac
```

**Loops overview**

Loops repeat commands while a condition remains true or until a condition becomes true. They are useful for:
- Processing lists of files or users
- Polling for readiness
- Repeating a task across multiple hosts or items
- Reading command output line by line

Main loop types:
- `for` for iterating over a list
- `while` for repeating while a condition succeeds
- `until` for repeating until a condition succeeds

**The `for` loop**

Use `for` when you already have a list of items. Bash assigns each item to a variable in turn.

Basic form:
```bash
for name in alice bob carol; do
  echo "User: $name"
done
```

File iteration example:
```bash
for file in /etc/sysconfig/*.conf; do
  [ -e "$file" ] || continue
  echo "Found: $file"
done
```

Safer line-by-line processing:
```bash
while IFS= read -r line; do
  echo "$line"
done < /etc/passwd
```

Notes:
- `IFS` is the Internal Field Separator; setting `IFS=` preserves leading and trailing whitespace.
- `read -r` prevents backslash escapes from being interpreted.
- Avoid `for x in $(command)` for data with spaces or newlines because command substitution splits on whitespace.

Better alternatives:
```bash
while IFS= read -r user; do
  echo "Processing $user"
done < <(cut -d: -f1 /etc/passwd)
```

```bash
mapfile -t users < <(cut -d: -f1 /etc/passwd)
for user in "${users[@]}"; do
  echo "$user"
done
```

**The `while` loop**

Use `while` when you want to repeat as long as a condition remains successful. The condition is checked before each iteration, so the body may run zero times.

Basic form:
```bash
while command; do
  commands
done
```

Typical uses:
- Reading input until EOF (end of file)
- Polling for a service or file
- Repeating until a counter reaches a limit

Counter example:
```bash
count=1
while [ "$count" -le 5 ]; do
  echo "Attempt $count"
  count=$((count + 1))
done
```

Polling example:
```bash
while ! systemctl is-active --quiet NetworkManager; do
  echo "Waiting for NetworkManager..."
  sleep 2
done
```

This pattern waits until the service becomes active. Use `while ! ...` when you want to continue until success.

Reading from a file:
```bash
while IFS= read -r line; do
  echo "$line"
done < /etc/hosts
```

> `while` checks the condition before the first iteration. If the condition is already false, the loop body does not run.

**The `until` loop**

Use `until` when you want to repeat until a condition becomes true. It is the inverse of `while`.

Basic form:
```bash
until command; do
  commands
done
```

Practical readiness check:
```bash
until systemctl is-active --quiet NetworkManager; do
  echo "Waiting for NetworkManager..."
  sleep 2
done
```

This is often clearer than `while ! command; do ... done` when you think in terms of “wait until ready.”

Another example:
```bash
count=1
until [ "$count" -gt 5 ]; do
  echo "Count: $count"
  count=$((count + 1))
done
```

Use `until` when:
- You want to wait for a service, file, or condition to become available.
- You want the loop to continue while the test fails.

**Loop control: `break` and `continue`**

Use `break` to exit a loop early and `continue` to skip to the next iteration.

Examples:
```bash
for user in alice bob carol; do
  [ "$user" = "bob" ] && break
  echo "$user"
done
```

```bash
for file in /var/log/*.log; do
  [ -f "$file" ] || continue
  echo "Processing $file"
done
```

Nested loops can take a numeric argument:
```bash
break 2
continue 2
```

That exits or skips two levels of nested loops.

**Practical scripting patterns**

A small admin script often combines `case` with loops and tests.

Example: service action wrapper
```bash
#!/bin/bash

service_name=sshd

case "${1:-}" in
  start)
    systemctl start "$service_name"
    ;;
  stop)
    systemctl stop "$service_name"
    ;;
  restart)
    systemctl restart "$service_name"
    ;;
  status)
    systemctl status "$service_name"
    ;;
  *)
    echo "Usage: $0 {start|stop|restart|status}"
    exit 1
    ;;
esac
```

Example: retry until a file appears
```bash
#!/bin/bash

until [ -f /tmp/ready.flag ]; do
  echo "Waiting for /tmp/ready.flag"
  sleep 1
done

echo "File is ready"
```

Example: process users from a file
```bash
#!/bin/bash

while IFS=: read -r user _ uid _ _ home shell; do
  printf '%s -> %s\n' "$user" "$shell"
done < /etc/passwd
```

**Quoting and safety**

Quoting matters in control flow because unquoted variables can split into multiple words or expand as globs.

Good:
```bash
if [ -n "$name" ]; then
  echo "$name"
fi
```

Risky:
```bash
if [ -n $name ]; then
  echo $name
fi
```

Guidelines:
- Quote variables in tests and command arguments.
- Use `[[ ... ]]` in Bash when you need safer string matching and pattern tests.
- Use `"$@"` to preserve argument boundaries in scripts.
- Prefer explicit tests over relying on ambiguous word splitting.

**Common test operators**

The `test` command and `[ ... ]` support many useful checks.

File tests:
- `-e file` exists
- `-f file` regular file
- `-d file` directory
- `-r file` readable
- `-w file` writable
- `-x file` executable
- `-s file` nonempty

String tests:
- `-z string` empty
- `-n string` nonempty
- `string1 = string2` equal
- `string1 != string2` not equal

Numeric tests:
- `-eq` equal
- `-ne` not equal
- `-lt` less than
- `-le` less than or equal
- `-gt` greater than
- `-ge` greater than or equal

Examples:
```bash
if [ -s /var/log/messages ]; then
  echo "Log file has content"
fi
```

```bash
if [ "$uid" -ge 1000 ]; then
  echo "Likely a regular user"
fi
```

**Bash vs POSIX shell**

RHCSA scripting is usually Bash-focused, but it helps to know what is portable.

- `if`, `case`, `for`, `while`, and `until` are POSIX shell constructs.
- `[[ ... ]]`, arrays, `mapfile`, `;&`, and `;;&` are Bash-specific.
- `(( ... ))` arithmetic evaluation is Bash and other modern shells, but not POSIX `sh` syntax in the same form.

If you write scripts for `/bin/bash`, Bash features are acceptable. If portability matters, prefer POSIX-compatible syntax.

**Troubleshooting and exam tips**

When a script misbehaves, check these first:
- Missing spaces around `[ ... ]`
- Unquoted variables causing word splitting
- Using `=` instead of `==` in the wrong context
- Forgetting `fi`, `done`, or `esac`
- Using `for x in $(command)` on data that contains spaces
- Confusing `while` and `until` logic

Useful debugging tools:
```bash
bash -x script.sh
```

```bash
set -x
# script body
set +x
```

You can also validate syntax without running the script:
```bash
bash -n script.sh
```

For RHCSA, focus on writing scripts that are:
- Readable
- Predictable
- Safe with quoting
- Correct in exit-status logic

**Quick reference**

- `if` chooses among conditions.
- `case` matches one value against patterns.
- `for` iterates over a list.
- `while` repeats while a condition succeeds.
- `until` repeats until a condition succeeds.
- `break` exits a loop.
- `continue` skips to the next iteration.
