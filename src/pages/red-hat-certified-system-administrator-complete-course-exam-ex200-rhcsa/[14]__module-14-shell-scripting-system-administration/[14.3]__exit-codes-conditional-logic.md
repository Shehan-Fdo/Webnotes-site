**Exit codes: the shell’s success/failure signal**

Every command in Linux returns an <abbr title="exit status">exit code</abbr> to the shell. You use that code to decide whether the previous command succeeded, failed, or needs special handling. In shell scripting and one-liners, exit codes are the foundation of conditional logic.

- `0` means success.
- `1` through `255` mean some kind of failure or special condition.
- The shell stores the most recent exit code in `$?`.
- Many tools use exit codes consistently, but the exact meaning of nonzero values is command-specific.

A practical habit is to check `$?` only immediately after the command you care about. Any later command overwrites it.

```bash
ls /etc/passwd
echo $?

grep 'nologin' /etc/passwd
echo $?
```

> A nonzero exit code does not always mean “broken.” Some commands use specific nonzero values to indicate expected states, such as “not found” or “no matches.”

**Inspecting and using `$?`**

You can use `$?` in interactive work, scripts, and troubleshooting. It is especially useful when a command prints little output but still needs verification.

- `0` indicates the command completed successfully.
- `2` is commonly used for incorrect usage or syntax errors, but this is not universal.
- `127` often means the shell could not find the command.
- `126` often means the command was found but could not be executed.
- `130` commonly indicates interruption by `Ctrl` + `C`.

Examples:

```bash
cp /etc/hosts /tmp/
echo $?

cp /does/not/exist /tmp/
echo $?
```

If you need to preserve the exit code in a script, store it before running another command:

```bash
some_command
status=$?
echo "Exit status: $status"
```

**Conditional execution with `&&` and `||`**

The shell evaluates `&&` and `||` based on the exit status of the command before them. This lets you chain commands without writing a full `if` block.

- `command1 && command2` runs `command2` only if `command1` succeeds.
- `command1 || command2` runs `command2` only if `command1` fails.
- You can combine them for simple success/failure flows.

Examples:

```bash
mkdir -p /srv/app && cd /srv/app
```

```bash
grep -q '^root:' /etc/passwd && echo "root exists"
```

```bash
id alice || useradd alice
```

Use these carefully when commands have side effects. A failed command may still partially change state, so chaining is not a substitute for proper error handling in complex scripts.

**Command grouping and precedence**

The shell evaluates `&&` before `||` in many practical expressions, but you should not rely on implicit readability. Use parentheses or braces to make intent clear.

- `{ ...; }` groups commands in the current shell.
- `( ... )` runs commands in a subshell.
- Grouping helps when you want multiple commands to act as one success/failure unit.

Examples:

```bash
{ mkdir -p /backup && cp /etc/hosts /backup/; } || echo "backup failed"
```

```bash
( cd /var/log && tar -czf /tmp/logs.tgz . )
```

Use a subshell when you want directory changes or variable changes isolated from the current shell. Use braces when you want to stay in the current shell.

**Testing conditions with `test` and `[ ]`**

The `test` command evaluates expressions and returns an exit code. The bracket form `[ ]` is a shell builtin syntax for `test` and is widely used in scripts.

- `test expression` and `[ expression ]` are equivalent in most shell usage.
- Spaces are required after `[` and before `]`.
- Missing spaces cause syntax errors.

Examples:

```bash
test -f /etc/passwd
echo $?

[ -f /etc/passwd ]
echo $?
```

Common file tests:

- `-e path` exists
- `-f path` is a regular file
- `-d path` is a directory
- `-r path` is readable
- `-w path` is writable
- `-x path` is executable
- `-s path` exists and is not empty

Common string tests:

- `-z string` is empty
- `-n string` is nonempty
- `string1 = string2` are equal
- `string1 != string2` are not equal

Common numeric tests:

- `num1 -eq num2` equal
- `num1 -ne num2` not equal
- `num1 -lt num2` less than
- `num1 -le num2` less than or equal
- `num1 -gt num2` greater than
- `num1 -ge num2` greater than or equal

Examples:

```bash
[ -d /etc ] && echo "directory exists"
[ -s /var/log/messages ] && echo "log has content"
[ "$USER" = "root" ] && echo "running as root"
[ 10 -gt 5 ] && echo "true"
```

> Always quote variables in string tests unless you intentionally want word splitting or glob expansion. Unquoted empty variables can break `[ ]` expressions.

**`[[ ]]` and safer conditional expressions**

In Bash, `[[ ]]` is a more capable conditional syntax than `[ ]`. It reduces common quoting problems and supports pattern matching and more flexible comparisons.

- `[[ ]]` is a Bash keyword, not the external `test` command.
- It handles empty variables more safely.
- It supports `==` pattern matching in many cases.
- It is not POSIX shell syntax, so use it only when Bash is guaranteed.

Examples:

```bash
if [[ -n $HOSTNAME && $HOSTNAME == server* ]]; then
  echo "Matches server pattern"
fi
```

Use `[[ ]]` for Bash scripts when you want safer string handling. Use `[ ]` when you need portability to `/bin/sh` or POSIX-compliant scripts.

**`if`, `elif`, and `else`**

The `if` statement gives you structured branching when simple chaining is not enough. It evaluates a command or test and chooses a block based on the exit code.

Basic form:

```bash
if command; then
  commands
elif other_command; then
  commands
else
  commands
fi
```

Example:

```bash
if [ -f /etc/ssh/sshd_config ]; then
  echo "SSH config exists"
else
  echo "SSH config missing"
fi
```

Use `if` when:

- You need multiple branches.
- You want readable logic.
- You need to run several commands for each outcome.
- You want to test more than one condition in sequence.

A more realistic example:

```bash
if id alice >/dev/null 2>&1; then
  echo "User exists"
else
  useradd alice
fi
```

**Combining tests with `-a`, `-o`, and shell logic**

Older `test` syntax supports logical AND and OR with `-a` and `-o`, but these can be confusing and are easy to misread. Prefer shell operators `&&` and `||` or use `[[ ]]` with `&&` and `||` inside the expression.

Examples:

```bash
[ -f /etc/passwd ] && [ -r /etc/passwd ] && echo "readable file"
```

```bash
if [[ -f /etc/passwd && -r /etc/passwd ]]; then
  echo "readable file"
fi
```

Avoid writing complex `test` expressions with `-a` and `-o` unless you understand precedence very well. Clear logic is more important than compact syntax.

**Practical RHCSA-style examples**

You often use exit codes and conditionals in administration tasks such as user management, service checks, and file validation.

- Check whether a service is active before restarting it.
- Create a directory only if it does not already exist.
- Verify a file before copying or archiving it.
- Run fallback commands when a primary action fails.

Examples:

```bash
systemctl is-active sshd >/dev/null 2>&1 && echo "sshd is running"
```

```bash
[ -d /backup ] || mkdir -p /backup
```

```bash
cp /etc/fstab /backup/fstab.$(date +%F) && echo "Backup complete" || echo "Backup failed"
```

```bash
rpm -q httpd >/dev/null 2>&1 || dnf install -y httpd
```

These patterns are common in exam tasks because they let you make decisions quickly without writing full scripts.

**Exit codes in pipelines**

A pipeline sends the output of one command into another. By default, the exit code of a pipeline is the exit code of the last command, which can hide earlier failures.

Example:

```bash
grep 'root' /etc/passwd | wc -l
echo $?
```

If `grep` fails but `wc` succeeds, the pipeline may still return success. In Bash, `set -o pipefail` makes the pipeline fail if any command in the pipeline fails.

```bash
set -o pipefail
grep 'pattern' missingfile | wc -l
echo $?
```

Use `pipefail` in scripts when you need reliable failure detection across pipelines. This is especially useful for automation and validation steps.

**`set -e` and cautious error handling**

`set -e` tells the shell to exit when a command returns a nonzero status. It can reduce boilerplate in simple scripts, but it can also behave unexpectedly in compound commands, conditionals, and pipelines.

- It is useful for short, linear scripts.
- It can make debugging harder if you do not understand its exceptions.
- It does not replace explicit checks for commands where failure is expected.

Example:

```bash
set -e

mkdir -p /tmp/demo
cp /etc/hosts /tmp/demo/
```

For exam work and real administration, prefer explicit `if` statements when failure handling matters. Use `set -e` only when you understand the control flow and the commands involved.

**Exit status patterns you should recognize**

Some commands and shell situations commonly produce recognizable exit codes.

- `0`: success
- `1`: general failure
- `2`: misuse of shell builtins or incorrect arguments
- `126`: found but not executable
- `127`: command not found
- `128+n`: terminated by signal `n`

Examples:

```bash
nosuchcommand
echo $?
```

```bash
chmod -x /tmp/script.sh
/tmp/script.sh
echo $?
```

Understanding these patterns helps you troubleshoot quickly when a command fails in a script or during an exam task.

**Best practices for reliable conditional logic**

Good shell logic is explicit, readable, and conservative.

- Quote variables in tests: `[ "$var" = "value" ]`
- Prefer `[[ ]]` in Bash for safer string handling.
- Use `if` for multi-step decisions.
- Use `&&` and `||` for short, obvious flows.
- Check command success with exit codes, not just printed output.
- Redirect noisy output to `/dev/null` when you only care about success or failure.
- Use `pipefail` when pipeline failures matter.
- Avoid ambiguous `test` expressions with `-a` and `-o`.

A compact but readable pattern:

```bash
if rpm -q chrony >/dev/null 2>&1; then
  systemctl enable --now chronyd
else
  dnf install -y chrony && systemctl enable --now chronyd
fi
```

**Troubleshooting workflow**

When a conditional does not behave as expected, verify each part separately.

1. Run the command alone and inspect its exit code with `echo $?`.
2. Test the condition independently with `test`, `[ ]`, or `[[ ]]`.
3. Add `set -x` to trace execution in a script.
4. Check for quoting issues, missing spaces, or wrong operator choice.
5. Confirm whether the command’s nonzero exit code is expected behavior.

Example debug session:

```bash
set -x
[ -f "$file" ] && echo "exists"
set +x
```

Tracing helps you see how the shell evaluates your logic and where a failure occurs.

**Summary of when to use each construct**

- Use `$?` when you need the last command’s status.
- Use `&&` and `||` for short conditional chains.
- Use `if` for readable branching and multiple actions.
- Use `[ ]` for POSIX-compatible tests.
- Use `[[ ]]` for safer Bash-specific tests.
- Use `pipefail` when pipeline failures must be detected.
- Use `set -e` only when you understand its limits.

These tools let you build reliable command sequences, validate system state, and automate routine RHCSA tasks with confidence.
