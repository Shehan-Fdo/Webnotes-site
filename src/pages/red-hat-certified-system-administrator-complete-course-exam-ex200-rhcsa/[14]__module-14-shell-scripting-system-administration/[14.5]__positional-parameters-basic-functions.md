## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

Positional parameters and functions are core Bash features that help you write scripts that are easier to read, reuse, and debug. Positional parameters let a script receive input from the command line, while functions let you group repeated logic into named blocks. For RHCSA-style tasks, you often use both to make shell automation concise and reliable.

In practice, you use positional parameters when a script needs arguments such as filenames, usernames, or service names. You use functions when the same sequence of commands appears more than once, or when you want to isolate a task like validation, cleanup, or reporting.

**Positional Parameters**

Positional parameters are the numbered variables Bash assigns to script arguments and function arguments. They are referenced as `$1`, `$2`, `$3`, and so on. The special variable `$0` holds the script name or shell name, depending on how the script is invoked.

Common parameter variables include:

- `$0` for the script name
- `$1` through `$9` for the first nine arguments
- `${10}`, `${11}`, and so on for arguments beyond nine
- `$#` for the number of positional parameters
- `$*` for all parameters as a single word when quoted behavior is not desired
- `$@` for all parameters as separate words when quoted
- `$?` for the exit status of the last command
- `$$` for the current shell process ID
- `$!` for the process ID of the most recent background job

A script that uses positional parameters should validate that the expected arguments are present before continuing. This prevents confusing failures later in the script.

```bash
#!/usr/bin/env bash

if [ "$#" -lt 2 ]; then
  echo "Usage: $0 SOURCE DESTINATION"
  exit 1
fi

cp "$1" "$2"
echo "Copied $1 to $2"
```

In this example, `$#` checks the argument count, and `$1` and `$2` refer to the source and destination values. Quoting the variables protects paths that contain spaces.

**Shifting Positional Parameters**

The `shift` builtin removes the first positional parameter and moves the rest down by one. This is useful when you want to process arguments one at a time in a loop.

```bash
#!/usr/bin/env bash

while [ "$#" -gt 0 ]; do
  echo "Processing: $1"
  shift
done
```

After each `shift`, the old `$2` becomes the new `$1`. This pattern is common in scripts that accept a variable number of arguments.

You can also shift by more than one position:

```bash
shift 2
```

Use this only when you know enough arguments exist. If you shift past the available parameters, Bash reports an error.

> Be careful when using `shift` in scripts that must preserve the original argument list. Once shifted, the original positions are gone unless you saved them first.

**Quoting `$@` and `$*`**

The difference between `$@` and `$*` matters when arguments contain spaces.

- `"$@"` expands to separate quoted arguments
- `"$*"` expands to one combined string using the first character of `IFS` (Internal Field Separator)

For most scripting tasks, `"$@"` is the safer choice because it preserves argument boundaries.

```bash
#!/usr/bin/env bash

for arg in "$@"; do
  echo "Arg: $arg"
done
```

This loop prints each argument exactly as passed, even if an argument contains spaces.

**Basic Functions**

A Bash function is a named block of commands that you can call like a command. Functions improve readability and reduce duplication. They are especially useful for repeated checks, logging, cleanup, and error handling.

You can define a function in either of these forms:

```bash
myfunc() {
  echo "Hello from a function"
}
```

or:

```bash
function myfunc {
  echo "Hello from a function"
}
```

The first form is more portable and commonly preferred in shell scripts.

A function is called by name:

```bash
myfunc
```

Functions can return an exit status using `return`. They do not return arbitrary strings the way some programming languages do; instead, they typically communicate through output, variables, or exit codes.

```bash
check_root() {
  if [ "$EUID" -ne 0 ]; then
    echo "Run as root"
    return 1
  fi
  return 0
}
```

Here, `return 1` signals failure and `return 0` signals success. The caller can test the result with `if check_root; then ...`.

**Function Scope and Arguments**

Functions can accept positional parameters just like scripts. Inside a function, `$1`, `$2`, and so on refer to the function’s arguments, not the script’s original arguments.

```bash
greet_user() {
  echo "Hello, $1"
}

greet_user alice
```

If you need the script’s original arguments inside a function, save them before calling the function or pass them explicitly.

Local variables help prevent accidental overwriting of global values. Use `local` inside functions for variables that should not escape the function scope.

```bash
count_files() {
  local dir="$1"
  local count
  count=$(find "$dir" -maxdepth 1 -type f | wc -l)
  echo "$count"
}
```

Using `local` keeps `dir` and `count` limited to the function. This reduces side effects and makes scripts easier to maintain.

**Practical Function Patterns**

Functions are often used for repeated tasks such as logging, validation, and cleanup.

```bash
log() {
  echo "[$(date +%F\ %T)] $*"
}

die() {
  echo "Error: $*" >&2
  exit 1
}

require_file() {
  [ -f "$1" ] || die "Missing file: $1"
}
```

These patterns are useful because they centralize behavior:

- `log` standardizes messages
- `die` prints an error and exits
- `require_file` validates input before continuing

A script can then read more clearly:

```bash
#!/usr/bin/env bash

require_file "$1"
log "Starting copy"
cp "$1" /tmp/
log "Copy complete"
```

**Using Functions with Command Substitution and Pipelines**

Functions can be used in command substitution, but you should be careful about output versus exit status. If a function prints text, that text becomes the substitution result. If a function only sets a return code, capture it with conditionals.

```bash
is_valid_user() {
  id "$1" >/dev/null 2>&1
}

if is_valid_user alice; then
  echo "User exists"
else
  echo "User does not exist"
fi
```

When a function is part of a pipeline, remember that each command may run in a subshell depending on the shell and context. This can affect variable changes made inside the function. For exam tasks, prefer simple, direct function calls when you need to preserve state.

**Debugging and Inspection**

When scripts behave unexpectedly, inspect the arguments and function flow. Bash provides useful debugging options:

```bash
#!/usr/bin/env bash
set -x

echo "Script name: $0"
echo "First arg: $1"
echo "Arg count: $#"
```

`set -x` prints commands as they execute, which helps you see how parameters expand. You can disable tracing with `set +x` after the problematic section.

You can also print arguments safely for debugging:

```bash
printf 'arg[%d]=%q\n' 1 "$1"
printf 'arg[%d]=%q\n' 2 "$2"
```

`%q` shows shell-escaped output, which is useful when spaces or special characters are involved.

**Common Exam-Relevant Uses**

In RHCSA-style scripting tasks, positional parameters and functions often appear in small automation jobs such as:

- copying files to a target directory passed on the command line
- creating users from a list of names
- checking whether a service name or path was provided
- wrapping repeated `systemctl` or `dnf` actions in a helper function
- validating input before making system changes

A simple service helper might look like this:

```bash
#!/usr/bin/env bash

restart_service() {
  systemctl restart "$1" || return 1
  systemctl is-active --quiet "$1"
}

if [ "$#" -ne 1 ]; then
  echo "Usage: $0 SERVICE"
  exit 1
fi

if restart_service "$1"; then
  echo "$1 restarted successfully"
else
  echo "Failed to restart $1"
  exit 1
fi
```

This pattern combines argument validation, a function, and exit-status checking.

**Best Practices**

Use these habits to keep scripts reliable:

- Quote parameter expansions unless you explicitly need word splitting
- Validate argument count with `$#` before using `$1`, `$2`, and so on
- Prefer `"$@"` over `$*` when forwarding arguments
- Use `local` for function variables that should not leak
- Return status codes from functions and test them with `if`
- Keep functions small and focused on one task
- Use descriptive names for both functions and parameters
- Avoid relying on global variables when a parameter will do

**Troubleshooting**

If a script behaves incorrectly, check these common issues:

- Missing quotes around parameters can split paths with spaces
- Using `$*` instead of `"$@"` can merge arguments unexpectedly
- Forgetting to validate `$#` can cause empty or unset parameter use
- Shifting too many times can discard needed arguments
- Assuming a function can `return` text instead of an exit status can lead to design mistakes
- Reusing variable names without `local` can overwrite values outside the function

A quick diagnostic pattern is to print the current parameters before and after a `shift`:

```bash
#!/usr/bin/env bash

echo "Before shift: count=$#, first=$1"
shift
echo "After shift: count=$#, first=$1"
```

This helps you confirm how Bash reassigns positional parameters.

**Summary**

Positional parameters let scripts accept and process command-line input, while functions let you package repeated logic into reusable units. Together, they make shell scripts more maintainable and easier to test. For exam work, focus on correct quoting, argument validation, function return codes, and safe use of `shift` and `"$@"`.
