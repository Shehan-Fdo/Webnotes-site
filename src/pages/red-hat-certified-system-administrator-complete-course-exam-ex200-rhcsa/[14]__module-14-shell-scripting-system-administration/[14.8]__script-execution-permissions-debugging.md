## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Script execution fundamentals**

A script is just a text file until the kernel and shell agree on how to execute it. On Linux, execution depends on file permissions, the interpreter line, and the way you invoke the file. For RHCSA-level work, you need to know how to make a script runnable, how to diagnose “permission denied” and “bad interpreter” errors, and how to trace what the script is actually doing.

The most common execution paths are:

- Run it directly as an executable file, such as `./backup.sh`.
- Pass it to an interpreter, such as `bash backup.sh`.
- Use a shebang line so the kernel selects the interpreter automatically.

A script can be readable but still not executable. That distinction matters because the shell can read a file as input, but the kernel requires execute permission to launch it as a program.

**Permissions, ownership, and execute bits**

Linux file mode bits control who can read, write, or execute a file. For scripts, the execute bit is the key permission that allows direct execution.

- `r` means read the file contents.
- `w` means modify the file contents.
- `x` means execute the file as a program.

Typical checks:

```bash
ls -l myscript.sh
stat myscript.sh
```

Example output might show:

```bash
-rw-r--r--. 1 user user 42 Apr 26 10:00 myscript.sh
```

That file is readable, but not executable. To make it executable for the owner:

```bash
chmod u+x myscript.sh
```

To make it executable for everyone who should run it:

```bash
chmod 755 myscript.sh
```

Use `chmod` carefully:

- `chmod 700 script.sh` restricts execution to the owner.
- `chmod 750 script.sh` allows the owner and group to execute.
- `chmod 755 script.sh` allows all users to execute, but only the owner to modify it.

A common troubleshooting pattern is:

1. Check whether the file has `x` permission.
2. Check whether you are in the correct directory.
3. Check whether the filesystem is mounted with `noexec`.
4. Check whether the script has a valid shebang and line endings.

> A file with execute permission can still fail to run if the filesystem is mounted with `noexec` or if the interpreter path is invalid.

**Shebangs and interpreter selection**

A shebang is the first line of a script and tells the kernel which interpreter to use. It usually looks like:

```bash
#!/bin/bash
```

or:

```bash
#!/usr/bin/env bash
```

Use a shebang when you want direct execution such as `./script.sh`. Without it, the kernel does not know which interpreter should process the file.

Key points:

- `#!/bin/bash` uses a fixed path to Bash.
- `#!/usr/bin/env bash` searches `PATH` for `bash`, which can improve portability.
- The shebang must be the first line with no leading spaces.
- The interpreter path must exist and be executable.

If the shebang is wrong, you may see:

- `bad interpreter: No such file or directory`
- `cannot execute: required file not found`

A practical example:

```bash
#!/bin/bash
echo "Hello from Bash"
```

Make it executable and run it:

```bash
chmod +x hello.sh
./hello.sh
```

If you run the same file with `bash hello.sh`, the execute bit is not required because Bash reads the file directly. That is useful for testing, but it does not replace proper permissions when you want direct execution.

**Direct execution versus interpreter invocation**

These two methods behave differently:

| Method | Needs `x` bit | Uses shebang | Typical use |
|---|---:|---:|---|
| `./script.sh` | Yes | Yes | Normal execution |
| `bash script.sh` | No | No | Testing or forcing Bash |
| `sh script.sh` | No | No | POSIX shell execution, if compatible |

This distinction matters when debugging:

- If `bash script.sh` works but `./script.sh` fails, the issue is often permissions, shebang, or filesystem execution policy.
- If `sh script.sh` fails but `bash script.sh` works, the script may use Bash-specific syntax not supported by `/bin/sh`.

**Common execution errors and what they mean**

Several errors point to specific problems:

- `Permission denied`
  - The file lacks execute permission.
  - The directory path may not allow traversal.
  - The filesystem may be mounted `noexec`.
- `No such file or directory`
  - The script path is wrong.
  - The shebang interpreter path does not exist.
  - The file may have Windows line endings that corrupt the shebang.
- `bad interpreter`
  - The interpreter in the shebang is missing or invalid.
- `command not found`
  - The shell cannot find the command in `PATH`.
  - You may need `./script.sh` instead of `script.sh`.

Useful checks:

```bash
file hello.sh
head -n 1 hello.sh
which bash
echo "$PATH"
mount | grep noexec
```

The `file` command can reveal line endings and file type. If a script was edited on Windows, you may see `CRLF` line terminators, which can break the shebang.

To fix line endings:

```bash
sed -i 's/\r$//' hello.sh
```

Or, if available:

```bash
dos2unix hello.sh
```

**PATH, current directory, and command lookup**

When you type a command without a path, the shell searches directories listed in `PATH`. The current directory is usually not searched unless you explicitly include `.` in `PATH`, which is uncommon and discouraged.

Examples:

```bash
./myscript.sh
myscript.sh
```

These are not equivalent. The first runs a file in the current directory. The second searches `PATH` for a command named `myscript.sh`.

For RHCSA tasks, remember:

- Use `./script.sh` for a local script.
- Use an absolute path like `/home/user/bin/script.sh` when clarity matters.
- Do not assume the current directory is in `PATH`.

You can inspect `PATH` with:

```bash
echo "$PATH"
```

And temporarily add a directory:

```bash
export PATH="$HOME/bin:$PATH"
```

**Filesystem mount options and `noexec`**

A filesystem mounted with `noexec` prevents direct execution of binaries and scripts from that mount point. This is a common cause of `Permission denied` even when the file has `x` permission.

Check mount options:

```bash
mount | grep noexec
findmnt -no TARGET,OPTIONS /home
```

If a directory is on a `noexec` mount, you can still often run a script by passing it to an interpreter:

```bash
bash /home/user/script.sh
```

That works because Bash reads the file as data rather than asking the kernel to execute it directly.

**Debugging with `bash -x` and `set -x`**

Tracing helps you see each command as the shell executes it. This is one of the most useful debugging techniques for shell scripts.

Run a script with tracing:

```bash
bash -x script.sh
```

Or enable tracing inside the script:

```bash
#!/bin/bash
set -x

name="Alice"
echo "Hello, $name"
```

To stop tracing:

```bash
set +x
```

Tracing is useful when:

- A variable is empty unexpectedly.
- A conditional branch is not behaving as expected.
- A command substitution returns a surprising value.
- A loop is iterating over the wrong items.

For more controlled debugging, combine tracing with error handling:

```bash
#!/bin/bash
set -x
set -euo pipefail
```

Meaning:

- `-e` exits on command failure.
- `-u` treats unset variables as errors.
- `pipefail` returns the failure status of the first failing command in a pipeline.

> Be careful with `set -euo pipefail` in scripts you do not fully understand. It improves safety, but it can also make scripts exit earlier than expected during debugging.

**Using `bash -n` for syntax checking**

Before running a script, you can check its syntax without executing it:

```bash
bash -n script.sh
```

This catches many syntax errors, such as:

- Missing `fi`
- Missing `done`
- Broken quoting
- Invalid function syntax

`bash -n` does not detect logic errors. A script can be syntactically valid and still behave incorrectly, so combine syntax checking with tracing.

**Exit codes and failure detection**

Every command returns an exit status. By convention:

- `0` means success.
- Nonzero means failure.

Check the last command’s exit code:

```bash
echo $?
```

Use this when debugging scripts that appear to run but do not produce the expected result. In shell scripts, you often test exit codes with conditionals:

```bash
if cp source.txt /tmp/; then
  echo "Copy succeeded"
else
  echo "Copy failed"
fi
```

For pipelines, remember that the exit status of the pipeline may hide earlier failures unless `pipefail` is enabled.

**Practical debugging workflow**

A reliable workflow helps you isolate problems quickly.

1. Verify the file exists and is the one you expect.
   ```bash
   ls -l ./script.sh
   file ./script.sh
   ```
2. Check permissions and ownership.
   ```bash
   stat ./script.sh
   ```
3. Inspect the shebang and first few lines.
   ```bash
   head -n 5 ./script.sh
   ```
4. Test syntax.
   ```bash
   bash -n ./script.sh
   ```
5. Trace execution.
   ```bash
   bash -x ./script.sh
   ```
6. Check for line-ending problems if the script still fails.
   ```bash
   file ./script.sh
   sed -i 's/\r$//' ./script.sh
   ```

This sequence narrows the problem from file-level issues to shell-level logic issues.

**Example script with safe execution and debugging hooks**

```bash
#!/bin/bash
set -euo pipefail

name="${1:-world}"

if [[ -z "$name" ]]; then
  echo "Usage: $0 <name>" >&2
  exit 1
fi

echo "Hello, $name"
```

Run it:

```bash
chmod +x greet.sh
./greet.sh Alice
```

Debug it with tracing:

```bash
bash -x ./greet.sh Alice
```

This example shows several good practices:

- A valid shebang.
- Strict mode for safer execution.
- Parameter expansion with a default value.
- A usage check with a nonzero exit code.

**Working with script locations and relative paths**

Scripts often fail because they assume the current working directory is a specific place. Relative paths are resolved from the directory where you run the script, not from the script’s own location.

Example problem:

```bash
cat data/input.txt
```

If the script is launched from another directory, that path may not exist. A more robust approach is to derive the script directory:

```bash
#!/bin/bash
script_dir="$(cd -- "$(dirname -- "${BASH_SOURCE[0]}")" && pwd)"
cat "$script_dir/data/input.txt"
```

This pattern is especially useful when a script must access files relative to itself.

**When to use `sh`, `bash`, or direct execution**

Choose the invocation method based on your goal:

- Use `./script.sh` when the script is executable and has a correct shebang.
- Use `bash script.sh` when you want to test or bypass execute permission.
- Use `sh script.sh` only when the script is POSIX-compatible and you want `/bin/sh` behavior.

Be aware that `sh` on many systems is not Bash. Bash-specific features such as arrays, `[[ ... ]]`, or `source` may fail under `sh`.

**Quick reference commands**

```bash
chmod +x script.sh
bash -n script.sh
bash -x script.sh
head -n 1 script.sh
file script.sh
stat script.sh
echo $?
```

**Exam-focused takeaways**

For RHCSA tasks, you should be able to:

- Make a script executable with `chmod +x`.
- Explain why `./script.sh` differs from `bash script.sh`.
- Recognize and fix a bad shebang.
- Diagnose `Permission denied` versus `No such file or directory`.
- Use `bash -n` and `bash -x` to validate and trace scripts.
- Identify `noexec` mount issues and work around them when appropriate.
- Correct Windows line endings that break shell scripts.

A strong troubleshooting habit is to separate permission problems from interpreter problems and then from script logic problems. That approach saves time and prevents unnecessary changes.
