## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Script structure basics**

A shell script is a plain text file containing commands that the shell can execute. On Linux, you typically use scripts to automate repetitive administration tasks such as user creation, service checks, log collection, and file maintenance.

A well-structured script is easier to read, debug, and reuse. At minimum, a script usually includes:

- A **shebang** line to define the interpreter
- Comments that explain intent or assumptions
- Commands arranged in a predictable order
- Optional variables, tests, and control flow

A simple script can be as small as a few lines, but good structure matters because scripts often run unattended and may be executed by different users or systems.

```bash
#!/bin/bash
# Display system information

hostname
date
uname -r
```

**Shebang line**

The **shebang** is the first line of a script and starts with `#!`. It tells the operating system which interpreter should run the file.

Common examples:

```bash
#!/bin/bash
#!/usr/bin/env bash
#!/bin/sh
#!/usr/bin/python3
```

Use the shebang when you want the script to run consistently with a specific interpreter, regardless of the user’s current shell. This is especially important when the script uses features that are not portable across shells.

- `#!/bin/bash` runs the script with Bash directly.
- `#!/bin/sh` requests the system shell, which may be a more limited POSIX-compatible shell.
- `#!/usr/bin/env bash` finds `bash` through the `PATH`, which can help when Bash is not in a fixed location.

> If a script has no shebang and you run it as `./script.sh`, the kernel cannot know which interpreter to use and execution may fail.

**How script execution works**

When you execute a script file directly, Linux checks the file header. If the file begins with a valid shebang, the kernel launches the interpreter named there and passes the script path as an argument.

If you run a script with an explicit interpreter, the shebang is less important because you are already choosing the interpreter yourself.

```bash
bash myscript.sh
sh myscript.sh
python3 myscript.py
```

This distinction matters:

- `./myscript.sh` uses the shebang
- `bash myscript.sh` uses Bash even if the shebang says something else
- `sh myscript.sh` may break Bash-specific syntax

**File permissions and execution**

A script must usually be readable and executable to run directly. The execute bit tells Linux that the file is allowed to be launched as a program.

Check permissions with:

```bash
ls -l myscript.sh
```

Typical permission changes:

```bash
chmod +x myscript.sh
chmod 755 myscript.sh
```

- `chmod +x` adds execute permission
- `chmod 755` gives the owner write access and everyone else read/execute access
- `chmod 700` is common for private administrative scripts

If you do not want to make a script executable, you can still run it by passing it to an interpreter:

```bash
bash myscript.sh
```

**Recommended script layout**

A practical shell script often follows this order:

1. Shebang
2. Comment block or purpose statement
3. Variable definitions
4. Function definitions
5. Main logic
6. Exit status

Example:

```bash
#!/bin/bash
# Backup /etc to /backup and report success

SOURCE="/etc"
DEST="/backup"
DATE=$(date +%F)
ARCHIVE="etc-backup-$DATE.tar.gz"

mkdir -p "$DEST"
tar -czf "$DEST/$ARCHIVE" "$SOURCE"

if [ $? -eq 0 ]; then
  echo "Backup created: $DEST/$ARCHIVE"
else
  echo "Backup failed" >&2
  exit 1
fi
```

This structure helps you separate configuration from logic and makes later maintenance easier.

**Comments and readability**

Comments explain why the script exists or why a specific command is used. They should add value, not repeat the command text.

Good uses of comments:

- Documenting assumptions
- Explaining non-obvious logic
- Marking sections of a longer script
- Warning about destructive actions

```bash
#!/bin/bash
# Remove old temporary files older than 7 days
find /tmp -type f -mtime +7 -delete
```

Avoid over-commenting obvious commands. Clear variable names and simple structure often reduce the need for many comments.

**Variables and quoting**

Variables make scripts flexible and easier to maintain. Use uppercase names for environment-style values and quote expansions to avoid word splitting and globbing issues.

```bash
#!/bin/bash
BACKUP_DIR="/var/backups"
FILE_NAME="system.log"

cp "/var/log/$FILE_NAME" "$BACKUP_DIR/"
```

Important habits:

- Quote variables unless you intentionally want splitting
- Use `"$var"` instead of `$var` in most command arguments
- Prefer descriptive names such as `SOURCE_DIR` or `TARGET_USER`

**Exit status and error handling**

Shell commands return an exit status, where `0` usually means success and nonzero means failure. Scripts should check critical commands and exit appropriately when something fails.

Example:

```bash
#!/bin/bash
mkdir -p /backup || exit 1
cp /etc/hosts /backup/hosts || exit 1
echo "Done"
```

You can also inspect `$?`, which stores the exit status of the last command:

```bash
command
echo $?
```

For more robust scripts, use `set -e` carefully to stop on errors, but understand that it can make debugging harder if you do not know where the failure occurred.

```bash
#!/bin/bash
set -e

mkdir -p /backup
cp /etc/hosts /backup/
```

**Choosing the right interpreter**

The shebang should match the syntax used in the script.

- Use Bash for arrays, `[[ ... ]]`, brace expansion, and Bash-specific features
- Use `/bin/sh` for portable POSIX shell syntax
- Use Python or another language when shell is awkward for the task

Examples of Bash-specific syntax:

```bash
#!/bin/bash
names=("alice" "bob" "carol")
echo "${names[0]}"
```

This may fail under `sh`, so the shebang must match the language features you use.

**Common script file issues**

A script may fail even when the commands inside are correct. Common causes include:

- Missing execute permission
- Wrong shebang path
- Windows line endings (`CRLF`) instead of Unix line endings
- Syntax incompatible with the chosen shell
- Running the script from the wrong directory and using relative paths incorrectly

Check the file type and line endings with:

```bash
file myscript.sh
```

If line endings are wrong, convert them with tools such as `dos2unix` if available:

```bash
dos2unix myscript.sh
```

**Testing and debugging**

You should test scripts before relying on them in production or on exam tasks. A few simple tools help you validate behavior.

Run syntax checks for Bash scripts:

```bash
bash -n myscript.sh
```

Trace execution to see each command as it runs:

```bash
bash -x myscript.sh
```

You can also add tracing inside the script:

```bash
#!/bin/bash
set -x

echo "Starting task"
id
hostname
set +x
echo "Tracing disabled"
```

These tools help you identify quoting mistakes, bad paths, and unexpected variable values.

**Practical examples**

A minimal administrative script:

```bash
#!/bin/bash
echo "Current user: $(whoami)"
echo "Current directory: $(pwd)"
```

A script that checks whether a service is active:

```bash
#!/bin/bash
SERVICE="sshd"

if systemctl is-active --quiet "$SERVICE"; then
  echo "$SERVICE is running"
else
  echo "$SERVICE is not running"
fi
```

A script that uses a function for clarity:

```bash
#!/bin/bash

backup_file() {
  local src="$1"
  local dest="$2"
  cp "$src" "$dest"
}

backup_file /etc/hosts /tmp/hosts.backup
```

**Exam-focused takeaways**

For RHCSA-style tasks, you should be able to:

- Recognize the purpose of the shebang
- Add the correct interpreter line to a script
- Make a script executable with `chmod +x`
- Run scripts directly or through an interpreter
- Identify why a script fails due to shell mismatch or permissions
- Use basic debugging commands such as `bash -n` and `bash -x`

A script is not just a list of commands. The shebang, permissions, quoting, and exit handling determine whether it behaves reliably in real system administration work.
