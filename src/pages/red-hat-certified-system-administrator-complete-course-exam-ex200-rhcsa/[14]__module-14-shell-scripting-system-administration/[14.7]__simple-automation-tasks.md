## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Automation overview**

Simple automation on RHEL usually means scheduling commands or scripts to run later, repeatedly, or at boot. For RHCSA, you should know how to use `cron` for recurring jobs, `at` for one-time jobs, and shell scripts for reusable tasks. You also need to know where to look when jobs do not run as expected.

Automation is most useful when a task is:
- repetitive, such as log cleanup or backups
- time-based, such as a report every morning
- deferred, such as a one-time maintenance action after hours
- tied to boot or periodic system maintenance

**`cron` fundamentals**

`cron` runs commands on a schedule. On RHEL, the `crond` service reads user crontabs and system cron files. Each entry has five time fields followed by the command:

```text
* * * * * command
- - - - -
| | | | |
| | | | +---- day of week (0-7, Sunday = 0 or 7)
| | | +------ month (1-12)
| | +-------- day of month (1-31)
| +---------- hour (0-23)
+------------ minute (0-59)
```

Use `crontab -e` to edit your own jobs and `crontab -l` to list them.

```bash
crontab -e
crontab -l
crontab -r
```

- `crontab -e` opens your personal crontab in the default editor.
- `crontab -l` shows the current user’s scheduled jobs.
- `crontab -r` removes the current user’s crontab, so use it carefully.

A practical example that runs a cleanup script every day at 02:15:

```cron
15 2 * * * /usr/local/bin/cleanup-temp.sh >> /var/log/cleanup-temp.log 2>&1
```

This example:
- runs at 02:15 every day
- uses an absolute path for the script
- appends standard output and standard error to a log file
- avoids relying on an interactive shell environment

> Cron jobs run with a minimal environment. Use absolute paths for commands, scripts, and files, and set any needed variables inside the script.

**Cron special strings**

Cron supports shorthand schedules that are useful for common recurring tasks:

```cron
@reboot /usr/local/bin/startup-check.sh
@hourly /usr/local/bin/hourly-task.sh
@daily /usr/local/bin/daily-task.sh
@weekly /usr/local/bin/weekly-task.sh
@monthly /usr/local/bin/monthly-task.sh
@yearly /usr/local/bin/yearly-task.sh
```

- `@reboot` runs once when the system starts.
- `@hourly`, `@daily`, `@weekly`, `@monthly`, and `@yearly` are easier to read than numeric fields.
- These are still subject to the system’s `cron` and `anacron` behavior.

**System crontabs and periodic directories**

RHEL also uses system-wide cron files and directories. These are commonly used for administrative tasks and package-managed jobs.

Typical locations include:
- `/etc/crontab`
- `/etc/cron.d/`
- `/etc/cron.hourly/`
- `/etc/cron.daily/`
- `/etc/cron.weekly/`
- `/etc/cron.monthly/`

System crontab entries include a user field:

```cron
# minute hour day month weekday user command
30 4 * * * root /usr/local/sbin/report-health.sh
```

- The extra `user` field tells the system which account runs the command.
- Files in `/etc/cron.d/` use the same format as `/etc/crontab`.
- Scripts in the periodic directories run according to the system’s `cron` and `anacron` configuration.

`anacron` matters on systems that are not always running. If a daily or weekly job is missed because the machine was off, `anacron` can run it later when the system comes back up, depending on configuration.

**`at` for one-time jobs**

Use `at` when you need a command to run once at a specific time. This is ideal for deferred maintenance, temporary changes, or a single reminder task.

Start an interactive `at` session with a time specification:

```bash
at 23:30
```

Then enter the commands to run, one per line, and finish with <kbd>Ctrl</kbd> + <kbd>D</kbd>.

Example session:

```bash
$ at 23:30
at> /usr/local/bin/cleanup-temp.sh >> /var/log/cleanup-temp.log 2>&1
at> echo "Cleanup completed at $(date)" >> /var/log/cleanup-temp.log
at> <Ctrl-D>
job 5 at Sat Apr 26 23:30:00 2026
```

You can also submit a command non-interactively:

```bash
echo '/usr/local/bin/cleanup-temp.sh >> /var/log/cleanup-temp.log 2>&1' | at 23:30
```

`at` typically executes commands through `/bin/sh`, so do not assume your interactive shell aliases, functions, or environment are available. Use absolute paths and explicit redirection.

Useful `at` commands:
- `atq` lists queued jobs
- `at -c <jobid>` shows the exact commands and environment snapshot for a queued job
- `atrm <jobid>` removes a queued job

Example inspection workflow:

```bash
atq
at -c 5
atrm 5
```

A practical deferred maintenance example:
1. Schedule a one-time log rotation or cleanup task for after business hours.
2. Redirect output to a file so you can verify success later.
3. Check the log after the scheduled time and confirm the expected changes.

Example:

```bash
echo '/usr/local/sbin/rotate-app-logs.sh >> /var/log/rotate-app-logs.log 2>&1' | at 01:00
atq
```

After execution, verify:
- the log file contains the command output
- the target files changed as expected
- no errors were written to stderr

**Access control for `cron` and `at`**

RHEL can restrict who may use `cron` and `at` through allow and deny files.

Common files:
- `/etc/cron.allow`
- `/etc/cron.deny`
- `/etc/at.allow`
- `/etc/at.deny`

Important behavior:
- If an `allow` file exists, only users listed there may use the service.
- If no `allow` file exists and a `deny` file exists, users listed in `deny` are blocked.
- If neither file exists, access is generally limited to `root` on RHEL for exam purposes.

Example `cron.allow`:

```text
alice
bob
```

Example `at.allow`:

```text
alice
bob
```

> For RHCSA study, remember that `allow` takes precedence over `deny` for access decisions, and `root` is always able to schedule jobs.

**Troubleshooting scheduled jobs**

When a job does not run, check the schedule, permissions, command path, and logs. Most failures come from missing absolute paths, incorrect permissions, or environment assumptions.

Useful checks:
- `crontab -l` to confirm the entry exists
- `atq` to confirm the job is queued
- `at -c <jobid>` to inspect the exact `at` job content
- `systemctl status crond` to confirm the daemon is active
- `journalctl -u crond` to inspect cron-related messages

On RHEL, cron and at messages may also appear in:
- `/var/log/cron`
- the system journal via `journalctl`

Examples:

```bash
systemctl status crond
journalctl -u crond --since "today"
grep CRON /var/log/cron
```

If a cron job fails silently, check:
- the command uses full paths
- the script is executable: `chmod +x /path/to/script`
- the script has a valid shebang such as `#!/bin/bash`
- the user has permission to read/write the referenced files
- output is redirected to a log for debugging

**Shell scripts as automation tools**

A shell script makes automation easier to reuse, test, and version. For RHCSA, you should be able to create a script, make it executable, and schedule it.

Example script:

```bash
#!/bin/bash
set -euo pipefail

LOGFILE=/var/log/disk-check.log
echo "Disk check started at $(date)" >> "$LOGFILE"
df -h >> "$LOGFILE"
echo "Disk check finished at $(date)" >> "$LOGFILE"
```

Make it executable:

```bash
chmod 750 /usr/local/sbin/disk-check.sh
```

Run it manually first:

```bash
/usr/local/sbin/disk-check.sh
```

Then schedule it with `cron`:

```cron
0 6 * * 1-5 /usr/local/sbin/disk-check.sh
```

Good script practices for automation:
- use a shebang such as `#!/bin/bash`
- use `set -euo pipefail` when appropriate to fail fast
- use absolute paths
- log both start and finish times
- redirect output to a file or logging system
- test manually before scheduling

**Practical exam workflow**

When asked to automate a task, follow a simple sequence:
1. Write or identify the command or script.
2. Test it manually.
3. Choose `cron` for recurring jobs or `at` for one-time jobs.
4. Use absolute paths and explicit redirection.
5. Verify the schedule with `crontab -l`, `atq`, or `at -c`.
6. Check logs after execution.

**Common pitfalls**

- Using relative paths in scheduled jobs
- Forgetting that cron runs with a minimal environment
- Assuming aliases or shell functions are available in `at`
- Forgetting to make scripts executable
- Misreading the time format in cron entries
- Ignoring `anacron` behavior on systems that are not always on
- Not checking `/var/log/cron` or `journalctl -u crond` when troubleshooting

**Quick reference**

| Task | Tool | Typical command |
|---|---|---|
| List recurring jobs | `cron` | `crontab -l` |
| Edit recurring jobs | `cron` | `crontab -e` |
| Queue one-time job | `at` | `at 23:30` |
| List queued one-time jobs | `at` | `atq` |
| Inspect queued job | `at` | `at -c <jobid>` |
| Remove queued job | `at` | `atrm <jobid>` |
| Check cron service | `crond` | `systemctl status crond` |
| View cron logs | `journalctl` | `journalctl -u crond` |

**Key takeaways**

`cron` is for recurring schedules, `at` is for one-time execution, and shell scripts make both easier to manage. For RHCSA, focus on correct syntax, absolute paths, output redirection, and verification. If a job fails, inspect the schedule, permissions, and logs before changing the command.
