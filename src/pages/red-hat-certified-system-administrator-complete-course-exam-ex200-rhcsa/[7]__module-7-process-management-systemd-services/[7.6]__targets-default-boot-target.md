## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Targets and boot flow**

On Red Hat Enterprise Linux, <abbr title="system and service manager">systemd</abbr> uses **targets** to group system states and service sets. A target is similar to a runlevel conceptually, but it is more flexible because it can pull in multiple units, dependencies, and services.

You use targets to control what kind of environment the system boots into, such as a multi-user text login or a graphical desktop. For RHCSA, you should know how to inspect the current target, change the default target, and switch targets during a running session.

Common target names include:

- `multi-user.target` for a non-graphical, multi-user system
- `graphical.target` for a graphical login environment
- `rescue.target` for single-user maintenance with minimal services
- `emergency.target` for the most minimal recovery shell

A target is a unit file managed by `systemd`, so you interact with it using `systemctl`.

**Targets versus runlevels**

Traditional SysV init used runlevels such as `3` and `5`. systemd preserves compatibility by mapping runlevels to targets, but you should think in terms of targets rather than runlevels.

| SysV runlevel | systemd target | Typical use |
|---|---|---|
| `0` | `poweroff.target` | Shut down the system |
| `1` | `rescue.target` | Maintenance mode |
| `3` | `multi-user.target` | Text-based multi-user system |
| `5` | `graphical.target` | Graphical desktop |
| `6` | `reboot.target` | Reboot the system |

This mapping matters because many older references still mention runlevels. On RHCSA, use the target names directly unless a question explicitly references a runlevel.

**Inspecting the current target**

To see the target currently active on the running system, use:

```bash
systemctl get-default
systemctl list-units --type=target
systemctl isolate multi-user.target
```

Useful commands:

- `systemctl get-default` shows the default target used at boot
- `systemctl list-units --type=target` shows active targets
- `systemctl status` gives a broader view of system state
- `systemctl isolate <target>` switches immediately to a target and stops units not required by it

`systemctl get-default` does not show the currently active target; it shows the target that systemd will boot into by default.

To identify the active target, you can use:

```bash
systemctl list-units --type=target --state=active
```

**Default boot target**

The **default boot target** is the target systemd starts after boot unless overridden by kernel parameters or manual intervention. On a typical server, the default is often `multi-user.target`. On a workstation, it is often `graphical.target`.

To check the default:

```bash
systemctl get-default
```

Typical output:

```bash
multi-user.target
```

To set the default target persistently, use:

```bash
systemctl set-default multi-user.target
```

Or switch back to a graphical boot:

```bash
systemctl set-default graphical.target
```

This changes the symbolic link `/etc/systemd/system/default.target` to point to the chosen target.

> `systemctl set-default` changes future boots only. It does not change the current session.

You can verify the link with:

```bash
ls -l /etc/systemd/system/default.target
```

Example output:

```bash
lrwxrwxrwx. 1 root root 37 Apr 26 10:00 /etc/systemd/system/default.target -> /usr/lib/systemd/system/multi-user.target
```

**Changing the current target**

If you need to switch the running system to another target without rebooting, use `systemctl isolate`.

Examples:

```bash
systemctl isolate multi-user.target
systemctl isolate graphical.target
systemctl isolate rescue.target
```

This is useful when you need to move from a graphical environment to a text-only environment or enter maintenance mode.

Important behavior:

- `isolate` starts the requested target and its dependencies
- `isolate` stops units that are not part of the new target
- `isolate` requires the target to allow isolation
- `graphical.target` and `multi-user.target` are commonly isolatable
- `emergency.target` is for recovery and may leave you with very limited services

**Practical use cases**

You may need to change targets for troubleshooting, resource reduction, or recovery.

Examples:

- Boot into `multi-user.target` when you want a server without a GUI
- Boot into `graphical.target` when a desktop environment is required
- Use `rescue.target` when you need local administrative access with basic services
- Use `emergency.target` when the system has serious boot problems and you need a minimal shell

A common RHCSA task is to make a system boot into text mode by default:

```bash
systemctl set-default multi-user.target
```

If the system is currently in graphical mode and you want to switch now:

```bash
systemctl isolate multi-user.target
```

**Rescue and emergency targets**

`rescue.target` provides a minimal environment with essential mounts and services. It is intended for maintenance tasks such as fixing configuration files, resetting passwords, or repairing boot issues.

`emergency.target` is even more minimal. It usually provides only a root shell and the root filesystem mounted read-only or minimally mounted, depending on the failure state and boot path.

Use these targets when normal boot is not possible or when you need to perform low-level recovery.

Typical recovery workflow:

1. Boot into `rescue.target` if the system can still start basic services.
2. Use `emergency.target` if the boot process fails earlier or more severely.
3. Remount filesystems read-write if needed before editing files.

Example remount command in a recovery shell:

```bash
mount -o remount,rw /
```

**Working with target dependencies**

Targets are not just labels; they pull in other units. For example, `graphical.target` usually includes `multi-user.target` plus display manager services and graphical session components.

You can inspect what a target wants with:

```bash
systemctl list-dependencies graphical.target
systemctl list-dependencies multi-user.target
```

This helps you understand why certain services start or stop when you switch targets.

**Boot-time troubleshooting**

If a system boots into the wrong target, check the default target and the bootloader configuration.

Useful checks:

```bash
systemctl get-default
ls -l /etc/systemd/system/default.target
```

If the system still boots into a different mode, consider whether a kernel command-line override is in effect. For example, `systemd.unit=multi-user.target` can override the default target for a single boot.

Example GRUB edit at boot time:

- Press <kbd>e</kbd> on the GRUB menu
- Append `systemd.unit=multi-user.target` to the kernel line
- Boot with the modified entry

This is useful for temporary recovery without changing the persistent default.

**Commands you should know**

```bash
systemctl get-default
systemctl set-default multi-user.target
systemctl set-default graphical.target
systemctl isolate multi-user.target
systemctl isolate graphical.target
systemctl list-units --type=target
systemctl list-dependencies graphical.target
```

**Exam-focused reminders**

- Use `systemctl get-default` to check the boot target.
- Use `systemctl set-default` to make a persistent change.
- Use `systemctl isolate` to switch the current session.
- Remember that `default.target` is a symbolic link managed by systemd.
- Know the common mapping between `multi-user.target` and text mode, and `graphical.target` and GUI mode.

**Example workflow**

If you need to configure a server to boot without a GUI and verify the result:

1. Set the default target:
   ```bash
   systemctl set-default multi-user.target
   ```
2. Confirm the change:
   ```bash
   systemctl get-default
   ```
3. Reboot if you need to test the boot path:
   ```bash
   reboot
   ```
4. After boot, verify the active target:
   ```bash
   systemctl list-units --type=target --state=active
   ```

This workflow is common in administration tasks because it separates the persistent boot setting from the current runtime state.
