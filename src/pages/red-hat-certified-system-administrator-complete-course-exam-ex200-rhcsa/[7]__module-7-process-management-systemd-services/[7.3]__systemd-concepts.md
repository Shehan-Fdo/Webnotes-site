## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**systemd overview**

`systemd` is the init system and service manager used by modern Red Hat Enterprise Linux systems. It starts and supervises services, mounts filesystems, manages sockets and timers, tracks boot targets, and collects logs through `journald`.

It replaces older SysV-style startup with dependency-based parallel activation. For RHCSA tasks, you mainly use `systemctl`, `journalctl`, and a few boot-time tools to inspect and control the system.

- **PID 1** is the first process started by the kernel; on systemd systems, it is `systemd`.
- **Unit** is the generic object systemd manages, such as a service, mount, socket, timer, or target.
- **Target** groups units into a boot state, similar to a runlevel but more flexible.

**Unit types and what they do**

A unit file describes how systemd should manage a resource. The most common unit types for RHCSA are:

- **Service units** (`.service`) manage daemons such as `sshd` or `httpd`.
- **Target units** (`.target`) group units into a system state.
- **Socket units** (`.socket`) listen on a port or socket and activate a service on demand.
- **Timer units** (`.timer`) trigger jobs on a schedule.
- **Mount units** (`.mount`) represent mounted filesystems.
- **Path units** (`.path`) watch filesystem paths and trigger actions.

You usually do not create complex custom units on RHCSA, but you must understand how to inspect and control them.

**Unit file locations, search order, and override precedence**

systemd searches unit files in multiple locations. The order matters because higher-precedence locations override lower-precedence ones.

- `/etc/systemd/system/` contains administrator overrides and custom units.
- `/run/systemd/system/` contains runtime-generated units and transient overrides.
- `/usr/lib/systemd/system/` contains vendor-provided unit files from packages.

A unit in `/etc/systemd/system/` overrides the same-named unit in `/usr/lib/systemd/system/`. The `/run/systemd/system/` tree sits between them and is typically used for temporary runtime state.

This precedence is important when a package update changes a vendor unit but your system still uses a local override. If you edit the wrong copy, your change may not take effect.

- Use `systemctl cat <unit>` to view the effective unit and any drop-ins.
- Use `systemctl show <unit>` to inspect properties.
- Use `systemctl status <unit>` to see the active source and runtime state.

```bash
systemctl cat sshd.service
systemctl status sshd.service
systemctl show -p FragmentPath -p DropInPaths sshd.service
```

> If a unit exists in multiple locations, systemd uses the highest-precedence version and then applies drop-in snippets from override directories.

**Common systemctl commands**

`systemctl` is the main control tool for services and targets. RHCSA tasks often require you to start, stop, enable, disable, and inspect units.

- `systemctl start <unit>` starts a unit now.
- `systemctl stop <unit>` stops a unit now.
- `systemctl restart <unit>` stops and starts a unit.
- `systemctl reload <unit>` asks a service to re-read configuration without a full restart, if supported.
- `systemctl try-restart <unit>` restarts only if the unit is already running.
- `systemctl reload-or-restart <unit>` reloads if supported, otherwise restarts.
- `systemctl enable <unit>` makes a unit start at boot.
- `systemctl disable <unit>` prevents boot-time start.
- `systemctl mask <unit>` blocks manual or automatic start by linking it to `/dev/null`.
- `systemctl unmask <unit>` removes the mask.
- `systemctl status <unit>` shows current state and recent log lines.
- `systemctl is-active <unit>` checks whether a unit is running.
- `systemctl is-enabled <unit>` checks whether a unit is enabled at boot.
- `systemctl list-units` shows active units.
- `systemctl list-unit-files` shows installed unit files and their enablement state.
- `systemctl list-jobs` shows queued start/stop jobs.

```bash
systemctl start sshd
systemctl enable sshd
systemctl status sshd
systemctl is-active sshd
systemctl is-enabled sshd
systemctl list-jobs
```

**Reload versus daemon-reload**

This is a common exam confusion point.

- `systemctl reload <unit>` tells a running service to reload its own configuration, if the service supports reload.
- `systemctl daemon-reload` tells systemd itself to re-scan unit files and rebuild its dependency tree.

Use `daemon-reload` after you change a unit file, create a new unit, or modify a drop-in. Use `reload` when the service can apply configuration changes without restarting.

Examples:

```bash
# After editing /etc/systemd/system/myapp.service
systemctl daemon-reload
systemctl restart myapp.service

# If the service supports live config reload
systemctl reload nginx.service
```

> `daemon-reload` does not restart services. It only makes systemd aware of unit-file changes.

**Enablement, static units, and presets**

Not every unit can be enabled. Some units are **static**, meaning they have no `[Install]` section and are intended to be started only as dependencies or by another trigger.

- Use `systemctl list-unit-files` to see whether a unit is `enabled`, `disabled`, `static`, `masked`, or `indirect`.
- Use `systemctl is-enabled <unit>` to confirm whether a unit can be enabled.
- If a unit is static, `systemctl enable` usually fails because there is no install target to create symlinks for.

```bash
systemctl list-unit-files | grep '^crond'
systemctl is-enabled crond.service
```

systemd presets provide vendor or site policy for default enablement. Preset files usually live in:

- `/usr/lib/systemd/system-preset/`
- `/etc/systemd/system-preset/`

`systemctl preset <unit>` applies preset policy to a single unit. `systemctl preset-all` applies preset rules to all installed units. This is useful after package installation or when you want to align the system with preset policy.

- `preset-all` does not mean “enable everything.”
- It follows the first matching preset rule and may enable or disable units accordingly.

```bash
systemctl preset-all
systemctl preset sshd.service
```

**Targets and boot states**

Targets are systemd’s way of grouping units into a boot state. They are more flexible than old runlevels because they can pull in multiple services and dependencies.

Common targets include:

- `multi-user.target` for a non-graphical multi-user system.
- `graphical.target` for a graphical login environment.
- `rescue.target` for single-user maintenance with basic services.
- `emergency.target` for the most minimal recovery shell.

Useful target commands:

- `systemctl get-default` shows the default boot target.
- `systemctl set-default <target>` changes the default target.
- `systemctl isolate <target>` switches immediately to a target and stops units not required by it.

```bash
systemctl get-default
systemctl set-default multi-user.target
systemctl isolate rescue.target
```

**Rescue versus emergency**

These two modes are often tested because they sound similar but behave differently.

- **rescue.target** starts a minimal system with basic services and mounts, usually enough for local maintenance.
- **emergency.target** starts the most minimal environment possible, often with only the root filesystem mounted read-only or minimally mounted, and provides a root shell for repair.

Use `rescue.target` when you need a maintenance shell with some services available. Use `emergency.target` when normal boot is too broken and you need the smallest possible environment.

```bash
systemctl isolate emergency.target
systemctl isolate rescue.target
```

> On many systems, entering `emergency.target` requires the root password or console access, depending on boot and security settings.

**Boot-time troubleshooting and jobs**

systemd schedules work as jobs. If a unit is stuck starting or stopping, a job may still be active.

- `systemctl list-jobs` shows queued jobs and their states.
- `systemctl cancel <job>` is not commonly used in RHCSA tasks, but job inspection helps explain delays.
- `systemctl status <unit>` often shows whether a job is pending or failed.

If a service hangs during startup, check dependencies, socket activation, and logs before forcing restarts.

```bash
systemctl list-jobs
systemctl status sshd
journalctl -u sshd --since "10 minutes ago"
```

**Service management and dependency behavior**

A service may depend on other units, sockets, mounts, or targets. Starting one unit can pull in others automatically.

- `Requires=` means the dependent unit must also start.
- `Wants=` is a weaker dependency and does not fail the parent if the child fails.
- `After=` controls ordering, not dependency.
- `Before=` is the inverse ordering directive.

These relationships matter when a service starts too early, fails because a filesystem is not mounted, or waits on a network dependency.

```ini
[Unit]
Description=Example App
After=network-online.target
Wants=network-online.target

[Service]
ExecStart=/usr/local/bin/example-app
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

**Editing and creating unit files**

When you create or modify a unit file, place custom files in `/etc/systemd/system/`. Avoid editing vendor files in `/usr/lib/systemd/system/` because package updates can overwrite them.

A common workflow is:

1. Create or edit the unit file in `/etc/systemd/system/`.
2. Run `systemctl daemon-reload`.
3. Start or restart the unit.
4. Enable it if it should start at boot.

```bash
vi /etc/systemd/system/myapp.service
systemctl daemon-reload
systemctl enable --now myapp.service
```

A minimal service unit might look like this:

```ini
[Unit]
Description=My App Service
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/myapp --config /etc/myapp.conf
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

**Socket activation**

Socket activation lets systemd listen on a port or UNIX socket and start the service only when traffic arrives. This reduces idle resource use and can improve startup behavior.

A socket-activated service usually has both a `.socket` and a `.service` unit. systemd starts the service when the socket receives a connection.

Example socket unit:

```ini
[Unit]
Description=Example Echo Socket

[Socket]
ListenStream=12345

[Install]
WantedBy=sockets.target
```

Example service unit:

```ini
[Unit]
Description=Example Echo Service
Requires=example-echo.socket

[Service]
ExecStart=/usr/local/bin/example-echo
StandardInput=socket
```

Useful inspection commands:

```bash
systemctl status example-echo.socket
ss -ltnp | grep 12345
systemctl list-sockets
```

- `systemctl list-sockets` shows active socket units and the ports or paths they listen on.
- `ss -ltnp` helps confirm which process or socket is bound to a port.

**Journald and log inspection**

`systemd-journald` collects logs from the kernel, services, and user processes. `journalctl` is the main tool for reading those logs.

Common commands:

- `journalctl` shows the full journal.
- `journalctl -u <unit>` filters by unit.
- `journalctl -b` shows logs from the current boot.
- `journalctl -b -1` shows the previous boot.
- `journalctl -p err` filters by priority.
- `journalctl --since "1 hour ago"` limits by time.
- `journalctl -f` follows new log entries live.

```bash
journalctl -u sshd
journalctl -b
journalctl -p warning --since today
journalctl -f
```

**Persistent versus volatile journal storage**

By default, journal storage may be volatile or persistent depending on system configuration and whether `/var/log/journal` exists.

- **Volatile** storage keeps logs in `/run/log/journal/`, which is temporary.
- **Persistent** storage keeps logs in `/var/log/journal/`, which survives reboot.

You can control this in `/etc/systemd/journald.conf` with the `Storage=` setting:

- `Storage=auto` uses persistent storage if `/var/log/journal` exists, otherwise volatile.
- `Storage=persistent` forces persistent storage when possible.
- `Storage=volatile` keeps logs in memory-backed storage.
- `Storage=none` disables storage.

```ini
# /etc/systemd/journald.conf
[Journal]
Storage=persistent
```

After changing journald configuration, restart the service:

```bash
systemctl restart systemd-journald
```

To verify storage behavior, inspect the journal paths:

```bash
ls -ld /var/log/journal /run/log/journal
journalctl --disk-usage
```

**Timers as scheduled alternatives to cron**

Timers are systemd’s scheduling mechanism. They can replace cron jobs and are often easier to manage as units.

- A `.timer` unit defines when to run.
- A `.service` unit defines what to run.
- `OnBootSec=`, `OnUnitActiveSec=`, and `OnCalendar=` are common timer directives.

```ini
[Unit]
Description=Daily cleanup timer

[Timer]
OnCalendar=daily
Persistent=true

[Install]
WantedBy=timers.target
```

Inspect timers with:

```bash
systemctl list-timers
systemctl status cleanup.timer
```

**Practical RHCSA workflow**

When troubleshooting a service or boot issue, use a repeatable sequence:

1. Check whether the unit is active and enabled.
2. Inspect recent logs for the unit.
3. Confirm the unit file location and any overrides.
4. Reload systemd if you changed unit files.
5. Restart or reload the service as appropriate.
6. Check jobs and dependencies if the unit hangs.

```bash
systemctl status httpd
systemctl is-enabled httpd
systemctl cat httpd
journalctl -u httpd --since "30 minutes ago"
systemctl daemon-reload
systemctl restart httpd
systemctl list-jobs
```

**Key exam distinctions**

- `reload` affects the service; `daemon-reload` affects systemd’s unit cache.
- `restart` always restarts; `try-restart` only restarts if already running.
- `reload-or-restart` prefers reload when supported.
- `enable` creates boot-time symlinks; `start` only affects the current session.
- `static` units cannot usually be enabled.
- `rescue.target` is less restrictive than `emergency.target`.
- `/etc/systemd/system/` overrides `/usr/lib/systemd/system/`.
- `/run/systemd/system/` holds runtime-generated state and transient units.

**Quick reference commands**

```bash
systemctl status <unit>
systemctl enable --now <unit>
systemctl disable --now <unit>
systemctl daemon-reload
systemctl reload <unit>
systemctl restart <unit>
systemctl try-restart <unit>
systemctl reload-or-restart <unit>
systemctl list-unit-files
systemctl list-jobs
journalctl -u <unit>
journalctl -b
systemctl set-default multi-user.target
systemctl isolate rescue.target
```
