**Process management overview**

Linux processes are running instances of programs. As an administrator, you inspect them to understand system load, identify runaway tasks, and stop or reprioritize work safely. The core tools in this topic fall into three groups:

- **Inspection**: `ps`, `top`
- **Selection by name or attributes**: `pgrep`, `pkill`
- **Signal and priority control**: `kill`, `nice`, `renice`

A process has a **PID** (process ID), a numeric identifier used by the kernel. Many administrative tasks require you to find the PID first, then act on it with a signal or priority change.

**`ps`: snapshot of processes**

`ps` shows a point-in-time view of processes. It does not update continuously, so it is best when you want a precise snapshot or need to script output.

Common uses:

- `ps -ef` shows all processes in a full-format listing.
- `ps aux` shows all processes with BSD-style output.
- `ps -u username` filters processes owned by a user.
- `ps -p PID` shows a specific process.
- `ps -o` customizes output columns.

Examples:

```bash
ps -ef
ps aux
ps -u alice
ps -p 1234 -o pid,ppid,ni,pri,stat,cmd
```

Useful columns include:

- `PID`: process ID
- `PPID`: parent process ID
- `TTY`: controlling terminal
- `STAT`: process state
- `NI`: nice value
- `CMD`: command name and arguments

The `STAT` field often includes state codes such as:

- `R`: running
- `S`: sleeping
- `D`: uninterruptible sleep
- `T`: stopped
- `Z`: zombie

A practical `ps` pattern is to combine it with `grep`, though `pgrep` is usually cleaner for process matching:

```bash
ps -ef | grep httpd
```

> Be careful: `grep` can match itself in the output. Use `pgrep` when you want name-based matching without that noise.

**`top`: live process and resource monitor**

`top` provides a continuously updating view of CPU, memory, load average, and process activity. It is ideal when you need to identify what is consuming resources right now.

Typical information shown:

- System uptime and load average
- Task counts
- CPU usage breakdown
- Memory and swap usage
- Per-process CPU and memory usage

Useful interactive actions in `top`:

- `P`: sort by CPU usage
- `M`: sort by memory usage
- `k`: kill a process by PID
- `r`: renice a process by PID
- `q`: quit

Example:

```bash
top
```

In practice, use `top` to answer questions like:

- Which process is using the most CPU?
- Is memory pressure increasing?
- Is a service stuck or thrashing?
- Which PID should you target with `kill` or `renice`?

**`pgrep`: find processes by name or attributes**

`pgrep` searches the process table and prints matching PIDs. It is more reliable than parsing `ps` output with text tools.

Common examples:

```bash
pgrep sshd
pgrep -u alice bash
pgrep -l httpd
pgrep -a nginx
```

Useful options:

- `-u`: match by user
- `-l`: show PID and process name
- `-a`: show PID and full command line
- `-f`: match against the full command line, not just the executable name

Example with full command-line matching:

```bash
pgrep -f 'python3 .*app.py'
```

`pgrep` is especially useful when multiple instances of a service are running and you need all matching PIDs.

**`pkill`: send signals by name or attributes**

`pkill` is the signal-sending counterpart to `pgrep`. Instead of printing PIDs, it sends a signal to matching processes.

Common examples:

```bash
pkill httpd
pkill -u alice bash
pkill -f 'python3 .*app.py'
```

By default, `pkill` sends `SIGTERM` (signal 15), which asks a process to exit cleanly. You can choose another signal with `-signal` or `-<signal>` syntax:

```bash
pkill -HUP sshd
pkill -9 firefox
```

Use `pkill` when you want to stop all matching processes without manually collecting PIDs. It is powerful, so verify your match criteria before using it on production systems.

> Use `pkill` carefully with `-f`, because broad command-line matches can terminate more processes than intended.

**`kill`: send signals to a PID**

`kill` sends a signal to one or more specific PIDs. Despite the name, it does not always terminate a process; it delivers a signal, and the process decides how to respond.

Basic usage:

```bash
kill 1234
kill -TERM 1234
kill -HUP 1234
kill -9 1234
```

Common signals:

- `SIGTERM` (`15`): polite request to terminate
- `SIGHUP` (`1`): often used to reload configuration
- `SIGKILL` (`9`): immediate termination by the kernel, cannot be caught or ignored
- `SIGSTOP`: stop a process immediately
- `SIGCONT`: continue a stopped process

Signal names and numbers are both accepted in many shells:

```bash
kill -15 1234
kill -TERM 1234
```

A good workflow is:

1. Try `SIGTERM` first.
2. If the process ignores it or hangs, investigate.
3. Use `SIGKILL` only when necessary.

Example troubleshooting sequence:

```bash
ps -p 1234 -o pid,ppid,stat,ni,cmd
kill -TERM 1234
sleep 2
kill -KILL 1234
```

**`nice`: start a process with adjusted priority**

`nice` launches a command with a modified scheduling priority. In Linux, the **nice value** ranges from `-20` to `19`:

- Lower values mean higher priority.
- Higher values mean lower priority.
- Ordinary users can usually increase the nice value, making a process less aggressive.
- Lowering the nice value typically requires elevated privileges.

Examples:

```bash
nice -n 10 tar -czf backup.tar.gz /home/alice
nice -n 19 ionice -c3 rsync -a /data/ /backup/
```

Use `nice` for background jobs that should yield CPU to interactive work, such as:

- backups
- large file compression
- batch data processing
- long-running maintenance tasks

A process started with a higher nice value is less likely to interfere with user-facing workloads.

**`renice`: change priority of a running process**

`renice` adjusts the nice value of an existing process. This is useful when a job is already running and you want to reduce or increase its CPU priority.

Examples:

```bash
renice 10 -p 1234
renice 15 -u alice
renice 5 -g developers
```

Common targets:

- `-p`: PID
- `-u`: user
- `-g`: process group

Interpretation:

- Increasing the nice value lowers priority.
- Decreasing the nice value raises priority and may require root.

Example workflow:

```bash
ps -p 1234 -o pid,ni,cmd
renice 15 -p 1234
ps -p 1234 -o pid,ni,cmd
```

If a process is consuming too much CPU but should keep running, `renice` is often safer than killing it.

**Signals and when to use them**

Signals are asynchronous notifications sent to processes. The most important ones for RHCSA-level administration are:

- `SIGTERM`: request graceful shutdown
- `SIGHUP`: request reload or reinitialization
- `SIGKILL`: force termination
- `SIGSTOP`: pause execution
- `SIGCONT`: resume execution

Typical use cases:

- Reload a daemon after config changes with `SIGHUP` if supported.
- Stop a misbehaving process with `SIGTERM` first.
- Force termination only if the process does not exit.
- Pause and resume jobs during maintenance with `SIGSTOP` and `SIGCONT`.

Examples:

```bash
kill -HUP 2345
kill -STOP 2345
kill -CONT 2345
```

**Practical process-control workflow**

A reliable administrative workflow is:

1. Identify the process with `top`, `ps`, or `pgrep`.
2. Confirm the command, owner, and PID.
3. Decide whether to stop it or reprioritize it.
4. Use `kill`, `pkill`, or `renice` with the least disruptive option first.
5. Verify the result.

Example:

```bash
pgrep -a httpd
ps -p 2345 -o pid,ppid,ni,stat,cmd
renice 10 -p 2345
kill -TERM 2345
```

If the process is part of a service managed by `systemd`, prefer service management commands when appropriate, because they preserve service state and dependencies better than killing individual processes.

**Common exam and admin patterns**

- Use `ps -ef` or `ps aux` to inspect all processes quickly.
- Use `top` to identify current resource hogs.
- Use `pgrep -a` to find matching processes without manual filtering.
- Use `pkill` to terminate all matching processes by name.
- Use `kill` when you already know the PID.
- Use `nice` for new long-running jobs that should be less urgent.
- Use `renice` for jobs already running.

**Troubleshooting tips**

- If `kill` appears ineffective, check whether the process is in uninterruptible sleep (`D` state); such processes may not respond until the kernel operation completes.
- If a process becomes a zombie (`Z`), the process has already exited but its parent has not reaped it yet. Killing the zombie does not help; investigate the parent process.
- If `pgrep` returns multiple PIDs, confirm whether that is expected before using `pkill`.
- If you need exact matching, prefer `pgrep -x` for the process name or `pgrep -f` for the full command line, depending on the situation.

**Quick reference**

| Tool | Purpose | Typical use |
|---|---|---|
| `ps` | Snapshot of processes | Inspect PID, owner, state, command |
| `top` | Live monitoring | Find CPU/memory hotspots |
| `pgrep` | Find matching PIDs | Search by name, user, or command line |
| `pkill` | Signal matching processes | Stop or reload processes by pattern |
| `kill` | Signal specific PIDs | Terminate, stop, continue, or reload |
| `nice` | Start with priority adjustment | Launch background jobs politely |
| `renice` | Change running priority | Reduce or raise CPU priority |

**Minimal command set to remember**

```bash
ps -ef
top
pgrep -a processname
pkill processname
kill -TERM PID
nice -n 10 command
renice 15 -p PID
```
