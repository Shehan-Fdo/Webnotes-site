## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Process fundamentals**

A Linux process is a running instance of a program. On RHEL, process management is part of daily administration because you need to identify runaway tasks, inspect resource usage, and stop or restart services safely. Systemd also tracks services in **cgroups** (control groups), which means it can manage the full process tree for a unit instead of only a single PID.

Common process states include:

- **Running** or runnable: actively executing or waiting for CPU time.
- **Sleeping**: waiting for an event, I/O, or a timer.
- **Stopped**: paused by a signal such as `SIGSTOP`.
- **Zombie**: exited, but the parent has not reaped the exit status yet.

Useful process-identification commands:

```bash
ps aux
ps -ef
ps -o pid,ppid,user,ni,pri,stat,cmd -p 1234
pgrep sshd
pgrep -a sshd
pidof sshd
```

- `ps` shows a snapshot of processes.
- `pgrep` searches by name and can print matching PIDs.
- `pidof` returns the PID(s) of a named daemon.
- `ps -o` lets you choose columns for troubleshooting.

**Signals and process control**

Signals are software interrupts sent to processes. You use them to request termination, reload configuration, stop, continue, or inspect behavior. The signal number and name can vary by platform, so check the mapping when needed.

```bash
kill -l
kill -TERM 1234
kill -HUP 1234
kill -KILL 1234
pkill -u alice sshd
```

- `kill -l` lists signal names and numbers.
- `SIGTERM` asks a process to exit cleanly.
- `SIGHUP` often means “reload configuration,” but only if the application supports it.
- `SIGKILL` forces termination and cannot be trapped.
- `pkill` matches by name or other criteria; use it carefully because it can affect multiple processes.

> Avoid using broad process-kill commands on services with active sessions unless you understand the impact. For example, terminating `sshd` can disconnect users and may stop all daemon instances.

**Priorities and CPU scheduling**

Linux uses process priority and niceness to influence CPU scheduling. Lower nice values mean higher priority. On RHEL, you typically adjust niceness with `nice` when starting a command or `renice` for an existing process.

```bash
nice -n 10 tar -czf backup.tgz /data
renice 5 -p 1234
renice -5 -p 1234
```

- `nice` sets the initial priority for a new command.
- `renice` changes the priority of a running process.
- Negative nice values require appropriate privileges.
- Use priority changes to reduce impact from batch jobs or to give important tasks more CPU preference.

**Monitoring process activity**

For exam tasks, you should be comfortable reading live process and resource data. `top` is the classic interactive monitor; `htop` may not be installed by default.

```bash
top
ps -eo pid,ppid,cmd,%cpu,%mem --sort=-%cpu | head
ps -eo pid,ppid,cmd,%mem --sort=-%mem | head
```

- `top` helps you identify CPU or memory spikes quickly.
- `ps --sort` is useful for one-shot reports.
- `head` limits output to the most relevant entries.

**systemd service model**

On modern RHEL systems, **systemd** is the init system and service manager. It starts the system, manages services, mounts, sockets, timers, and targets, and tracks processes in unit cgroups. That cgroup tracking is important because stopping a service usually stops the entire process tree associated with that unit.

Common unit types include:

- `.service` for daemons and one-shot jobs.
- `.socket` for socket activation.
- `.timer` for scheduled activation.
- `.target` for grouping system states and boot milestones.

Useful service commands:

```bash
systemctl status sshd
systemctl start sshd
systemctl stop sshd
systemctl restart sshd
systemctl reload sshd
systemctl enable sshd
systemctl disable sshd
systemctl is-enabled sshd
systemctl is-active sshd
```

- `restart` stops and starts the service.
- `reload` asks the service to re-read configuration without a full restart, but only if the service supports it.
- `enable` creates boot-time activation links.
- `disable` removes boot-time activation links.
- `is-active` and `is-enabled` are useful in troubleshooting and exam validation.

**Unit file locations and overrides**

Systemd reads unit files from multiple locations. Vendor-provided units usually live under `/usr/lib/systemd/system/`, while administrator changes belong under `/etc/systemd/system/`.

Typical layout:

- `/usr/lib/systemd/system/` — packaged vendor unit files.
- `/etc/systemd/system/` — administrator overrides, custom units, and symlinks created by `enable`.
- `/run/systemd/system/` — runtime-generated units and transient state.

Use `systemctl cat` to inspect the effective unit source and any drop-ins:

```bash
systemctl cat sshd
systemctl cat example.service
```

Use `systemctl edit` to create a drop-in override safely:

```bash
systemctl edit sshd
```

This opens an editor and writes a drop-in under `/etc/systemd/system/sshd.service.d/`. Drop-ins are preferred when you want to change a small part of a vendor unit without copying the whole file.

Example drop-in:

```ini
[Service]
Restart=always
RestartSec=5
```

After editing unit files or drop-ins, reload systemd’s configuration:

```bash
systemctl daemon-reload
```

**Service troubleshooting workflow**

When a service fails, verify the unit definition, logs, and runtime state in a structured way.

1. Check the current status.
2. Inspect the unit file and drop-ins.
3. Validate syntax.
4. Review logs.
5. Reload systemd if you changed unit files.
6. Restart or re-enable the service if needed.

Commands:

```bash
systemctl status example.service
systemctl cat example.service
systemd-analyze verify /etc/systemd/system/example.service
journalctl -u example.service
journalctl -xe
```

- `systemd-analyze verify` checks unit syntax and common dependency issues.
- `journalctl -u` filters logs for a specific unit.
- `journalctl -xe` shows recent high-priority messages and hints.

A minimal custom service example:

```ini
[Unit]
Description=Example App
After=network.target

[Service]
ExecStart=/usr/local/bin/example-app
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

Notes on this example:

- `After=network.target` orders startup after the network stack is brought up, but it does **not** guarantee that the network is fully configured.
- If you need to wait for actual connectivity, `network-online.target` is the usual ordering target, and it typically requires a wait service such as `NetworkManager-wait-online.service` or `systemd-networkd-wait-online.service` to delay boot until the network is ready.
- Do not assume every daemon should use `network-online.target`; use it only when the application truly depends on network availability at start time.

> Do not change `Type=` or add `PIDFile=` unless the application actually daemonizes and writes the PID file in the expected location. An incorrect service type can make systemd track the process incorrectly.

**Boot targets and runlevels**

Systemd targets replace traditional SysV runlevels, but RHEL keeps familiar aliases for compatibility. Understanding the mapping helps when you need to change boot behavior or recover a system.

Common aliases:

- `runlevel3.target` and `multi-user.target` are commonly associated with text-mode multi-user operation.
- `runlevel5.target` and `graphical.target` are associated with graphical boot.
- `rescue.target` is similar to single-user rescue mode.
- `emergency.target` provides the most minimal boot environment.

Practical commands:

```bash
systemctl get-default
systemctl set-default multi-user.target
systemctl isolate rescue.target
systemctl isolate emergency.target
```

- `get-default` shows the default boot target.
- `set-default` changes the default target for future boots.
- `isolate` switches to a target immediately and stops units not required by that target.

Recovery differences:

- `rescue.target` starts a minimal system with basic services and a root shell.
- `emergency.target` starts the most minimal environment, typically with the root filesystem mounted read-only and only essential functionality available.
- Use `emergency.target` when you need the smallest possible environment for repair.

**Timers**

Systemd timers replace many cron use cases. A timer unit triggers a service unit based on time events. Timers are useful for scheduled maintenance, periodic jobs, and delayed startup tasks.

Timer types:

- **OnCalendar** for calendar-based schedules.
- **OnBootSec** for a delay after boot.
- **OnActiveSec** for a delay after the timer becomes active.
- **OnStartupSec** for a delay after the manager starts.

Example timer and service pair:

```ini
# /etc/systemd/system/backup.service
[Unit]
Description=Backup job

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
```

```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Run backup daily

[Timer]
OnCalendar=daily
Persistent=true
Unit=backup.service

[Install]
WantedBy=timers.target
```

Important timer details:

- `Persistent=true` helps systemd catch up missed calendar-based runs after downtime.
- It is not a universal requirement for every timer behavior.
- `OnBootSec`, `OnActiveSec`, and `OnStartupSec` are relative timers and do not depend on calendar catch-up in the same way.
- `Unit=` is optional if the timer name matches the service name.

Enable and inspect timers:

```bash
systemctl enable --now backup.timer
systemctl list-timers --all
systemctl status backup.timer
```

- `systemctl list-timers --all` shows active and inactive timers, last run time, and next run time.
- Use it to confirm whether a timer is scheduled correctly.
- If a timer does not fire, check both the timer unit and the service unit logs.

**Socket activation**

Socket activation lets systemd listen on a socket and start the service only when traffic arrives. This reduces boot-time overhead and can improve on-demand service startup.

Example socket unit:

```ini
# /etc/systemd/system/example.socket
[Unit]
Description=Example socket

[Socket]
ListenStream=8080
Accept=no

[Install]
WantedBy=sockets.target
```

Related options:

- `ListenStream=` for TCP stream sockets.
- `ListenDatagram=` for UDP datagram sockets.
- `Accept=` controls whether systemd spawns one service per connection or a single shared service.
- `SocketUser=` and `SocketGroup=` can control ownership in some cases.

Typical workflow:

```bash
systemctl enable --now example.socket
systemctl status example.socket
systemctl list-sockets
```

- Start the socket unit, not necessarily the service unit.
- The service starts when a connection arrives if socket activation is configured correctly.
- Use `systemctl list-sockets` to see active listening sockets managed by systemd.

**Practical service administration examples**

Restarting a service after configuration changes:

```bash
systemctl restart sshd
```

Reloading a service that supports configuration reload:

```bash
systemctl reload sshd
```

Checking whether a service is enabled and active:

```bash
systemctl is-enabled crond
systemctl is-active crond
```

Finding the process tree for a service:

```bash
systemctl status crond
ps -ef | grep [c]rond
```

Inspecting the cgroup for a unit:

```bash
systemd-cgls
systemctl status sshd
```

- `systemd-cgls` shows the cgroup tree and helps you understand which processes belong to which service.
- This is useful when a service spawns helper processes or when you need to confirm that systemd cleaned up child processes.

**Troubleshooting checklist**

When a process or service misbehaves, use a repeatable workflow:

1. Identify the process or unit with `ps`, `pgrep`, `pidof`, or `systemctl status`.
2. Check logs with `journalctl -u UNIT`.
3. Inspect the unit source with `systemctl cat UNIT`.
4. Validate unit syntax with `systemd-analyze verify`.
5. Confirm dependencies and ordering, especially around network and mount availability.
6. Reload systemd with `systemctl daemon-reload` after unit edits.
7. Restart or re-enable the unit if the change affects runtime or boot behavior.

**Exam-focused reminders**

- Know the difference between `network.target` and `network-online.target`.
- Know where vendor units and admin overrides live.
- Know how to use `systemctl edit` and `systemctl cat`.
- Know how to verify timers with `systemctl list-timers --all`.
- Know the runlevel-to-target mapping and how to change the default target.
- Know how to identify processes with `pgrep`, `pidof`, and `ps`.
- Know the common signals and what `kill -l` shows.
- Know that systemd manages service cgroups, not just single PIDs.
