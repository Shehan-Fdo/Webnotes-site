## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**systemd service management overview**

`systemctl` is the primary command-line tool for managing **systemd** units on modern Red Hat systems. A unit can be a service, mount, socket, timer, target, or other managed object, but for RHCSA service control you focus mainly on `*.service` units.

The key idea is that service management has two separate concerns:

- **Runtime state**: whether the service is running right now.
- **Boot-time behavior**: whether the service starts automatically at boot.

That distinction matters because `start` and `stop` affect the current session, while `enable` and `disable` affect future boots. `mask` goes one step further by making a unit impossible to start until you explicitly unmask it.

**Checking service state with `systemctl status`**

`systemctl status` is the fastest way to inspect a unit. It shows whether the service is active, inactive, failed, or loading, along with recent log lines and the main process ID when available.

```bash
systemctl status sshd.service
systemctl status sshd
```

Useful output fields include:

- **Loaded**: whether the unit file is found and whether it is enabled, disabled, or masked.
- **Active**: current runtime state such as `active (running)` or `inactive (dead)`.
- **Main PID**: the primary process ID for the service, if one exists.
- **Docs**: documentation references from the unit file.
- **Recent log lines**: last journal entries related to the unit.

You can use the shorter unit name when the suffix is obvious. For services, `systemctl status sshd` usually resolves to `sshd.service`.

> If a service fails to start, `status` often gives the first useful clue. Check the `Active:` line and the recent journal messages before changing configuration.

**Starting and stopping services**

`start` launches a service immediately in the current boot session. `stop` terminates it now. These commands do not change whether the service starts at the next boot.

```bash
sudo systemctl start httpd
sudo systemctl stop httpd
```

Typical use cases:

- Start a service after installing or configuring it.
- Stop a service before maintenance or troubleshooting.
- Test a configuration change without rebooting.

A service may fail to start if:

- The configuration file has syntax errors.
- The required port is already in use.
- A dependency is missing.
- The service is disabled by policy or masked.

**Restarting services**

`restart` stops and then starts the service again. Use it when you want to apply configuration changes that require a full process reload and the service does not support a graceful reload command.

```bash
sudo systemctl restart sshd
sudo systemctl restart firewalld
```

Restart is common after editing configuration files such as:

- `/etc/ssh/sshd_config`
- `/etc/httpd/conf/httpd.conf`
- `/etc/chrony.conf`

A restart briefly interrupts service availability. For services that support it, `reload` may be less disruptive, but that is a separate command and not the focus here.

**Enabling and disabling services at boot**

`enable` creates the necessary symlinks so the service starts automatically when the system reaches the relevant boot target. `disable` removes those symlinks so the service does not start automatically.

```bash
sudo systemctl enable sshd
sudo systemctl disable sshd
```

Important points:

- `enable` does **not** start the service immediately.
- `disable` does **not** stop a service that is already running.
- To affect both boot-time and current runtime state, combine commands.

Example workflow:

```bash
sudo systemctl enable --now sshd
sudo systemctl disable --now bluetooth
```

The `--now` option is useful when you want a single command to change both states at once:

- `enable --now` enables the service and starts it now.
- `disable --now` disables the service and stops it now.

**Masking and unmasking services**

`mask` is stronger than `disable`. It links the unit file to `/dev/null`, which prevents the service from being started manually or indirectly by dependencies. This is useful when you want to ensure a service cannot run at all.

```bash
sudo systemctl mask telnet.socket
sudo systemctl mask cups
```

`unmask` removes that restriction and restores the unit so it can be started again.

```bash
sudo systemctl unmask cups
```

Use `mask` when:

- You need to block a service that should never run on the system.
- A conflicting service must be prevented from starting.
- You want to enforce a hard administrative restriction.

Use `disable` when:

- You only want to prevent automatic startup at boot.
- Manual start should still remain possible.

> A masked unit cannot be started until it is unmasked. If a dependency tries to activate it, systemd blocks the request.

**Practical comparison of the main actions**

| Command | Current session | Next boot | Typical purpose |
|---|---:|---:|---|
| `start` | Starts now | No change | Bring a service up immediately |
| `stop` | Stops now | No change | Shut a service down immediately |
| `restart` | Restarts now | No change | Apply changes by cycling the service |
| `enable` | No change | Starts automatically | Persist service startup across boots |
| `disable` | No change | Does not start automatically | Prevent automatic boot startup |
| `mask` | No change unless already running | Cannot start | Hard block service activation |
| `unmask` | No change | Restores ability to start | Remove a mask so the unit can be used again |

**Common RHCSA command patterns**

Use these patterns to manage services efficiently:

```bash
# Check whether a service is running and enabled
systemctl status chronyd

# Start and enable a service for future boots
sudo systemctl enable --now chronyd

# Stop a service and prevent it from starting at boot
sudo systemctl disable --now avahi-daemon

# Restart after editing configuration
sudo systemctl restart sshd

# Block a service completely
sudo systemctl mask cups

# Restore a masked service
sudo systemctl unmask cups
sudo systemctl start cups
```

**Verifying results**

After any change, verify both runtime and boot-time state. This is a common exam habit and helps you avoid false assumptions.

```bash
systemctl status sshd
systemctl is-enabled sshd
systemctl is-active sshd
```

These checks answer different questions:

- `is-active` tells you whether the service is running now.
- `is-enabled` tells you whether it starts at boot.
- `status` gives the full summary and recent logs.

Example outputs you may see conceptually:

- `active` or `active (running)` for a healthy service.
- `inactive` or `dead` for a stopped service.
- `enabled` for boot startup.
- `disabled` for no automatic startup.
- `masked` for a blocked unit.

**Troubleshooting workflow**

When a service does not behave as expected, use a simple sequence:

1. Check the unit state with `systemctl status <unit>`.
2. Confirm whether it is enabled with `systemctl is-enabled <unit>`.
3. Try starting or restarting it with `systemctl start <unit>` or `systemctl restart <unit>`.
4. Review the log output shown by `status` if the start fails.
5. If the unit is masked, run `systemctl unmask <unit>` before trying again.

A practical example:

```bash
sudo systemctl status httpd
sudo systemctl restart httpd
sudo systemctl status httpd
```

If the service still fails, the issue is usually in configuration, permissions, port conflicts, or missing dependencies rather than in `systemctl` itself.

**Service management examples**

A few realistic administration scenarios help connect the commands to daily work:

- You install a web server and want it available after reboot:
  - `sudo systemctl enable --now httpd`
- You are decommissioning an insecure legacy service:
  - `sudo systemctl disable --now telnet.socket`
  - `sudo systemctl mask telnet.socket`
- You edit SSH configuration and need the daemon to pick up changes:
  - `sudo systemctl restart sshd`
- You want to inspect why a database service is failing:
  - `systemctl status mariadb`

**Key exam reminders**

- `start` and `stop` affect the current boot only.
- `enable` and `disable` affect future boots only.
- `restart` is a stop/start cycle and may interrupt users.
- `mask` is stronger than `disable` and prevents activation.
- `status` is your first diagnostic command.
- Use `sudo` when your user does not have administrative privileges.

**Quick reference**

```bash
systemctl status <unit>
systemctl start <unit>
systemctl stop <unit>
systemctl restart <unit>
systemctl enable <unit>
systemctl disable <unit>
systemctl mask <unit>
systemctl unmask <unit>
systemctl is-active <unit>
systemctl is-enabled <unit>
```
