## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**What `journalctl` does**

`journalctl` reads logs from the systemd journal, which is the central logging system used by `systemd`. It is the primary tool for reviewing service failures, boot messages, kernel events, and authentication-related entries on modern RHEL systems.

The journal stores structured records, so you can filter by service, boot, priority, executable, process ID, transport, and many other fields. This makes it more precise than plain text log files when you need to isolate a problem quickly during exam troubleshooting.

- Use `journalctl` when a service starts, fails, or behaves unexpectedly.
- Use it to inspect the current boot, previous boots, or kernel messages.
- Use field-based filters when you need to narrow logs beyond a unit name.

**Core viewing commands**

The most common workflow is to start with a unit, then narrow by boot or severity if needed. On RHCSA systems, `httpd.service` is a more exam-relevant example than `nginx.service`.

```bash
journalctl -u httpd.service
journalctl -u httpd.service -b
journalctl -u httpd.service -b -1
journalctl -k
journalctl --no-pager
```

- `-u` filters by systemd unit name.
- `-b` shows logs from the current boot.
- `-b -1` shows the previous boot.
- `-k` shows kernel messages only.
- `--no-pager` prints directly to the terminal, which is useful in exam environments.

`journalctl -u httpd.service -b` is often the first command you should try when a service fails after boot. If the service was restarted multiple times, the journal can show the sequence of failures, warnings, and startup messages that explain the root cause.

**Output formatting options**

`journalctl` supports multiple output formats. The default human-readable format is useful for interactive review, but structured formats help when you need more detail or machine-readable output.

```bash
journalctl -u httpd.service -o short
journalctl -u httpd.service -o verbose
journalctl -u httpd.service -o json
```

- `-o short` is concise and readable.
- `-o verbose` shows all available journal fields for each entry.
- `-o json` is useful when you need structured output for parsing or inspection.

`-o short` is a common choice when you want timestamps and messages without extra noise. `-o verbose` is better when you need to inspect metadata such as `_PID`, `_COMM`, `_EXE`, or `_TRANSPORT`.

**Filtering by priority and severity**

The journal stores a priority value for each entry. You can filter by severity using `-p` with a single level or a range. Lower numbers are more severe.

Common priority levels:

- `0` emerg
- `1` alert
- `2` crit
- `3` err
- `4` warning
- `5` notice
- `6` info
- `7` debug

Examples:

```bash
journalctl -p err
journalctl -p warning..emerg
journalctl -u httpd.service -p 3..0
```

- `journalctl -p err` shows error-level and more severe messages.
- `warning..emerg` includes warning, error, critical, alert, and emergency messages.
- `3..0` is useful when you want only serious failures.

Use severity filtering when a service produces too much normal output and you need to focus on failure-related entries. This is especially helpful when a service logs verbose startup information.

**Filtering by journal fields**

The journal is structured, so you can query by fields instead of only by text. This is one of the most important concepts for troubleshooting.

Common fields:

- `_SYSTEMD_UNIT` identifies the unit that produced or is associated with the entry.
- `_PID` identifies the process ID.
- `_COMM` identifies the command name.
- `_EXE` identifies the executable path.
- `_TRANSPORT` identifies how the message entered the journal, such as `kernel`, `syslog`, `stdout`, `stderr`, or `audit`.

Examples:

```bash
journalctl _SYSTEMD_UNIT=httpd.service
journalctl _PID=1234
journalctl _COMM=httpd
journalctl _EXE=/usr/sbin/httpd
journalctl _TRANSPORT=kernel
```

- `_PID=1234` is useful when you know the exact process that emitted the message.
- `_COMM=httpd` helps when multiple processes are involved and you want the command name.
- `_EXE=/usr/sbin/httpd` is useful when you need to confirm the binary that generated the entry.
- `_TRANSPORT=kernel` is the correct way to isolate kernel-originated journal entries.

> `_TRANSPORT` is the entry transport, not a source path. It tells you how the message reached the journal.

A practical process-focused query might look like this:

```bash
journalctl _COMM=sshd
journalctl _EXE=/usr/sbin/sshd
journalctl _PID=2145 -o verbose
```

This is useful when a service spawns helper processes or when you need to trace a specific daemon instance.

**Kernel logs and service troubleshooting**

Kernel messages often explain hardware, driver, filesystem, or boot-time issues that affect services. `journalctl -k` is the direct way to view them, while `_TRANSPORT=kernel` is the field-based equivalent.

```bash
journalctl -k
journalctl _TRANSPORT=kernel
journalctl -k -p warning..emerg
```

- `-k` is the fastest way to inspect kernel logs.
- `_TRANSPORT=kernel` is useful when you want to combine kernel filtering with other field queries.
- Severity filtering can reduce noise when the kernel emits many informational messages.

Use kernel logs when a service fails because of missing devices, mount problems, SELinux denials, or boot-time dependency issues. The journal often shows the underlying cause before the service failure becomes obvious.

**Boot selection and time-based review**

You can inspect logs from the current boot or earlier boots. This matters when a service worked before rebooting or when the failure only appears after startup.

```bash
journalctl -b
journalctl -b -1
journalctl -b -2
journalctl --list-boots
```

- `-b` shows the current boot.
- `-b -1` shows the previous boot.
- `--list-boots` displays boot IDs and relative indexes.

Time-based filtering is also useful when you know approximately when the issue occurred.

```bash
journalctl --since "2026-04-26 08:00:00"
journalctl --since "1 hour ago"
journalctl --until "10 minutes ago"
journalctl --since "today" --until "now"
```

Use time filters to reduce the search space before applying more specific unit or field filters.

**Persistent versus volatile journal storage**

The journal can be stored in two modes:

- **Volatile storage** keeps logs in memory and under `/run/log/journal`, so entries are lost after reboot.
- **Persistent storage** keeps logs under `/var/log/journal`, so older boots remain available after restart.

On RHEL systems, journald typically uses persistent storage if `/var/log/journal` exists; otherwise it falls back to volatile storage. That means persistence is not automatic on every system.

To enable persistent storage, you can use either directory-based persistence or configuration-based persistence.

```bash
sudo mkdir -p /var/log/journal
sudo systemctl restart systemd-journald
```

You can also set the configuration explicitly:

```ini
# /etc/systemd/journald.conf
[Journal]
Storage=persistent
```

Then verify the effective setting:

```bash
grep -E '^\s*Storage=' /etc/systemd/journald.conf
journalctl --disk-usage
journalctl --list-boots
```

- Creating `/var/log/journal` encourages persistent storage on systems configured to use it.
- Restarting `systemd-journald` often applies the change, but a reboot may be needed to confirm that older boots are retained.
- `Storage=persistent` is the explicit configuration method and is clearer for long-term setup.
- `journalctl --list-boots` helps verify whether previous boots are now available.

> If the directory is created after the system has already booted, the current session may still behave like a volatile journal until journald restarts or the system reboots.

**Practical troubleshooting workflow**

When a service fails, use a repeatable sequence so you do not miss the cause.

1. Check the service state.
   ```bash
   systemctl status httpd.service
   ```
2. Review the current boot logs for that unit.
   ```bash
   journalctl -u httpd.service -b --no-pager
   ```
3. Narrow by severity if the output is noisy.
   ```bash
   journalctl -u httpd.service -b -p warning..emerg --no-pager
   ```
4. Inspect kernel messages if the service depends on storage, networking, or hardware.
   ```bash
   journalctl -k --since "10 minutes ago" --no-pager
   ```
5. If the issue happened before reboot, inspect the previous boot.
   ```bash
   journalctl -u httpd.service -b -1 --no-pager
   ```

This workflow helps you move from broad status checks to precise log evidence without guessing.

**Reading journal entries effectively**

A journal entry contains metadata and the message body. The message is only part of the record. Metadata can reveal which unit, process, or transport produced the entry.

Useful fields to recognize:

- `_SYSTEMD_UNIT` for the unit name.
- `_PID` for the process ID.
- `_COMM` for the command name.
- `_EXE` for the executable path.
- `_TRANSPORT` for the entry source mechanism.
- `PRIORITY` for severity.

Example verbose output:

```bash
journalctl -u httpd.service -n 1 -o verbose
```

This can show fields such as:

```text
_SYSTEMD_UNIT=httpd.service
_COMM=httpd
_EXE=/usr/sbin/httpd
_TRANSPORT=stdout
PRIORITY=3
```

Interpret the transport carefully:

- `stdout` or `stderr` usually means the message came from a service process writing to standard output or error.
- `syslog` means the message entered through the syslog interface.
- `kernel` means the kernel generated the message.
- `audit` means the audit subsystem generated the message.

**Useful command patterns for RHCSA**

These patterns are practical for exam tasks and troubleshooting.

```bash
journalctl -u sshd.service -b --no-pager
journalctl -u httpd.service -p err --no-pager
journalctl -k -p warning..emerg --no-pager
journalctl _COMM=sshd --since "today"
journalctl _EXE=/usr/sbin/httpd -o short
journalctl --list-boots
```

- Use `sshd.service` when checking remote access problems.
- Use `httpd.service` when validating web service startup and configuration.
- Use `--no-pager` when you need uninterrupted output in a terminal or script.
- Use `-o short` for quick human review and `-o verbose` when you need metadata.

**Common exam mistakes**

- Confusing `-k` with a unit filter; `-k` shows kernel logs only.
- Assuming persistent logs exist without checking `/var/log/journal`.
- Using text search when a field query would be more precise.
- Forgetting `--no-pager` when output opens in a pager and slows review.
- Treating `_TRANSPORT` as a file path instead of a journal metadata field.
- Using the wrong service name for the task; on RHCSA, `httpd.service` is often more relevant than `nginx.service`.

**Quick reference**

| Task | Command |
|---|---|
| View current boot logs for a service | `journalctl -u httpd.service -b` |
| View previous boot logs | `journalctl -b -1` |
| View kernel messages | `journalctl -k` |
| Show only errors and worse | `journalctl -p err` |
| Show logs without paging | `journalctl --no-pager` |
| Show verbose metadata | `journalctl -o verbose` |
| Show structured JSON output | `journalctl -o json` |
| List recorded boots | `journalctl --list-boots` |
| Query by process name | `journalctl _COMM=httpd` |
| Query by executable | `journalctl _EXE=/usr/sbin/httpd` |

**Key takeaways**

`journalctl` is the central tool for reading systemd logs on RHEL. You use it to inspect services, boots, kernel events, and structured metadata. For RHCSA tasks, the most important habits are to filter by unit, narrow by boot, use severity levels, and switch to field-based queries when the problem is not obvious from the service name alone.
