## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Service troubleshooting workflow**

A reliable troubleshooting workflow helps you isolate whether a problem is caused by the service itself, its configuration, its dependencies, the network, or access controls such as SELinux. On RHEL, most services are managed by `systemd`, so your first checks usually involve `systemctl`, `journalctl`, and service-specific validation commands.

A good workflow is iterative: confirm the symptom, inspect the unit state, read logs, validate configuration, test connectivity, and then restart or reload only after you understand the failure mode. This reduces guesswork and avoids masking the root cause.

- Start with the unit state and recent logs.
- Confirm whether the service is enabled, active, failed, or waiting on a socket.
- Check whether the issue is local to the host or caused by a dependency such as DNS, storage, or the network.

**Initial triage with systemd**

`systemd` tracks services as units. The most useful first commands are `systemctl status`, `systemctl is-active`, and `systemctl is-enabled`. These tell you whether the service is running now, whether it starts at boot, and whether it has recently failed.

```bash
systemctl status httpd
systemctl is-active httpd
systemctl is-enabled httpd
systemctl list-units --type=service --state=failed
```

Use `systemctl status` to see the last few log lines, the main process ID, and the unit file path. If the service is in a failed state, `systemctl reset-failed <unit>` clears the failure marker after you fix the underlying issue.

- **active** means the service is currently running.
- **failed** means the unit exited unsuccessfully or could not start.
- **inactive** means it is stopped and not currently running.
- **enabled** means it starts at boot; it does not guarantee it is running now.

> If a service is socket-activated, the service unit may not appear active until a client connects. Check the socket unit first with `systemctl status <name>.socket` and verify the listening port with `ss -ltnp`.

**Read logs before changing anything**

Logs usually reveal the exact failure. `journalctl` is the primary tool for systemd-managed services, and it is often faster than guessing from symptoms alone. Filter by unit, boot, or time window to narrow the output.

```bash
journalctl -u httpd
journalctl -u httpd -b
journalctl -u httpd --since "10 minutes ago"
journalctl -xe
```

For persistent problems, compare the current boot with the previous boot if the service failed during startup:

```bash
journalctl -u httpd -b -1
```

If the service writes to a traditional log file, inspect that too. Some daemons still use `/var/log/messages`, `/var/log/secure`, or application-specific logs under `/var/log/`.

- Use `journalctl -u <unit>` for systemd unit messages.
- Use `journalctl -xe` for high-priority recent errors and context.
- Use `journalctl -b` to focus on the current boot.
- Use `journalctl -b -1` when the issue happened before the last reboot.

**Validate the configuration**

Many services fail because of a syntax error, a missing include file, or an invalid path. Validate the configuration before restarting the daemon. The exact test command depends on the service.

```bash
httpd -t
nginx -t
sshd -t
named-checkconf
named-checkzone example.com /var/named/example.com.zone
```

Use the service’s native validation command when available. For example, `httpd -t` checks Apache syntax, `nginx -t` checks NGINX configuration, and `sshd -t` validates the SSH daemon configuration. `named-checkconf` and `named-checkzone` are useful for BIND DNS configuration and zone files.

- Prefer a service-specific syntax check over a blind restart.
- Validate included files and referenced paths, not just the main config file.
- Re-run the test after every change until it passes cleanly.

A simple Apache example:

```bash
apachectl configtest
```

A simple SSH example:

```bash
sshd -t
```

A simple BIND example:

```bash
named-checkconf /etc/named.conf
named-checkzone example.com /var/named/example.com.zone
```

**Check ports, sockets, and listeners**

A service can be running but still unreachable if it is not listening on the expected address or port. Use `ss` to confirm the socket state and bound interfaces.

```bash
ss -ltnp
ss -lunp
ss -ltnp '( sport = :80 )'
```

Interpret the output carefully:

- `LISTEN` means a TCP service is accepting connections.
- `UNCONN` or a bound UDP socket may still be valid for datagram services.
- `127.0.0.1:port` means the service is only reachable locally.
- `0.0.0.0:port` or `[::]:port` means it listens on all interfaces for that protocol family.

If the service is socket-activated, inspect the socket unit and the listening socket together:

```bash
systemctl status cups.socket
systemctl status cups.service
ss -ltnp | grep ':631'
```

This distinction matters because the socket unit can be active and listening even when the service has not yet spawned.

**Confirm dependencies and prerequisites**

Services often fail because something they depend on is missing or unavailable. Common dependencies include DNS resolution, mounted filesystems, certificates, database backends, and network reachability.

- Check name resolution with `getent hosts <name>`.
- Check mounts with `findmnt` or `mount`.
- Check disk space with `df -h`.
- Check whether the required port is reachable with `ss`, `nc`, or `curl`.
- Check whether the service account can read its files and directories.

Examples:

```bash
getent hosts db01.example.com
findmnt /var/lib/postgresql
df -h /var
curl -I http://localhost
nc -vz 127.0.0.1 5432
```

If a daemon depends on a database or remote API, verify that the backend is reachable before focusing on the front-end service. A service may start but still fail health checks if its dependency is down.

**Inspect permissions and ownership**

Incorrect file ownership or mode bits are a common cause of startup failures. Services often need read access to configuration files, write access to state directories, and execute access on parent directories.

```bash
ls -l /etc/httpd/conf/httpd.conf
ls -ld /var/lib/myapp /var/log/myapp
namei -l /var/lib/myapp/data.db
```

Use `namei -l` to inspect every directory in a path. This helps you find a missing execute bit on a parent directory, which can block access even when the target file looks correct.

- Configuration files usually need to be readable by the daemon or root.
- Runtime directories often need to be writable by the service user.
- Parent directories must allow traversal with the execute bit.

**Check SELinux when access looks correct but the service still fails**

Security-Enhanced Linux (SELinux) can block access even when Unix permissions look fine. On RHEL, AVC (Access Vector Cache) denials are best checked through the audit subsystem.

```bash
ausearch -m AVC,USER_AVC -ts recent
journalctl -t audit --since "10 minutes ago"
sealert -a /var/log/audit/audit.log
```

If you suspect SELinux, look for denied file access, port binding, or network access. Common clues include `permission denied` in logs even though ownership and mode bits are correct.

- Use `ausearch` to find recent denials.
- Use `sealert` when `setroubleshoot` is installed and you want a human-readable summary.
- Use `getenforce` to confirm whether SELinux is enforcing, permissive, or disabled.

```bash
getenforce
sestatus
```

> Do not disable SELinux as a first response. Fix the label, policy, or boolean when possible.

Useful SELinux checks and fixes:

```bash
ls -Z /var/www/html
restorecon -Rv /var/www/html
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
restorecon -Rv /srv/web
```

If a service needs a non-default port, verify the SELinux port label:

```bash
semanage port -l | grep http_port_t
semanage port -a -t http_port_t -p tcp 8080
```

**Use service-specific diagnostics**

Some daemons provide their own health checks or debug modes. These are often more precise than generic system tools.

- `httpd -t` or `apachectl configtest` for Apache syntax.
- `nginx -t` for NGINX syntax.
- `sshd -t` for SSH daemon configuration.
- `named-checkconf` and `named-checkzone` for BIND.
- `postfix check` for Postfix configuration.
- `smbcontrol` and Samba logs for Samba issues, when applicable.

When a service has a built-in test mode, use it before restarting. This avoids repeated failed starts and makes the logs easier to interpret.

**Restart, reload, or re-enable correctly**

Choose the right action for the change you made.

- Use `systemctl restart <unit>` when the daemon must re-read everything or when it is stuck.
- Use `systemctl reload <unit>` when the service supports reloading configuration without dropping connections.
- Use `systemctl daemon-reload` after changing unit files, drop-ins, or dependencies.
- Use `systemctl enable <unit>` when you want the service to start at boot.

Examples:

```bash
systemctl restart httpd
systemctl reload httpd
systemctl daemon-reload
systemctl enable --now sshd
```

If a service fails repeatedly, inspect the logs before restarting it again. Repeated restarts can hide the original error and create noise.

**A practical troubleshooting sequence**

1. Confirm the symptom with `systemctl status <unit>`.
2. Read recent logs with `journalctl -u <unit> -b`.
3. Validate the configuration with the service’s native test command.
4. Check ports and listeners with `ss -ltnp`.
5. Verify dependencies such as DNS, mounts, and backend services.
6. Check file ownership, permissions, and SELinux labels.
7. Fix the root cause.
8. Reload or restart the service.
9. Confirm the service is active and reachable.

This sequence works well in exam scenarios because it is fast, repeatable, and evidence-driven.

**Common failure patterns**

- **Syntax error in config**
  - The daemon exits immediately.
  - Logs usually point to a line number or file path.
  - Fix the syntax and retest before restarting.

- **Port already in use**
  - Another process is bound to the same port.
  - Use `ss -ltnp` to identify the conflicting process.
  - Stop or reconfigure the conflicting service.

- **Wrong bind address**
  - The service listens only on `127.0.0.1`.
  - Remote clients cannot connect.
  - Update the listen directive and restart.

- **Missing file or directory**
  - The daemon cannot open a config, key, socket, or data file.
  - Check paths, ownership, and parent directory permissions.
  - Recreate the file or correct the path.

- **SELinux denial**
  - Unix permissions look correct, but access still fails.
  - Check audit logs and labels.
  - Restore labels or adjust policy.

- **Dependency failure**
  - The service starts but cannot reach a database, DNS server, or mount.
  - Verify the dependency first.
  - Fix the upstream service or network path.

**Useful command set for exam practice**

```bash
systemctl status <unit>
systemctl is-active <unit>
systemctl is-enabled <unit>
journalctl -u <unit> -b
journalctl -xe
ss -ltnp
ss -lunp
ausearch -m AVC,USER_AVC -ts recent
getenforce
restorecon -Rv <path>
systemctl restart <unit>
systemctl reload <unit>
```

**Troubleshooting mindset**

Treat service troubleshooting as a sequence of evidence checks, not a guessing game. Start with the unit state, then move through logs, configuration validation, sockets, dependencies, permissions, and SELinux. On RHCSA tasks, this workflow helps you solve problems quickly and explain your reasoning clearly.
