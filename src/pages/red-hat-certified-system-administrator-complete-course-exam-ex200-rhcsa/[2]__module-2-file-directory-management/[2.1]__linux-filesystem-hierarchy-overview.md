**Filesystem hierarchy fundamentals**

Linux uses a single rooted directory tree that starts at `/`. Unlike drive-letter systems, every file and device is attached somewhere under this tree, either on the root filesystem or on a mounted filesystem. On Red Hat Enterprise Linux, the layout follows the <abbr title="Filesystem Hierarchy Standard">FHS</abbr>, which defines where common types of data should live so tools, administrators, and services can predict paths consistently.

The hierarchy matters because it affects booting, package management, backups, permissions, and troubleshooting. If you know what belongs in each directory, you can quickly identify what is safe to edit, what is generated at runtime, and what should be preserved across reinstalls.

- `/` is the top of the tree and the starting point for all absolute paths.
- Mount points attach additional filesystems into the directory tree.
- The FHS separates static system files, variable data, logs, and user data.
- Red Hat systems use this layout to keep administration predictable across hosts.

**Core directory roles**

The most important directories are not just names; they represent categories of data with different lifecycle expectations. Some directories contain essential boot and system binaries, while others store logs, temporary files, or user home directories. Understanding the purpose of each location helps you place files correctly and avoid breaking system behavior.

| Directory | Purpose | Notes |
|---|---|---|
| `/bin` | Essential user commands | On modern systems, often merged into `/usr/bin` via symlinks or filesystem layout changes. |
| `/sbin` | Essential system administration commands | Often merged into `/usr/sbin` on newer distributions. |
| `/boot` | Bootloader files, kernel images, initramfs | Critical for system startup. |
| `/dev` | Device nodes | Represents hardware and pseudo-devices such as terminals and disks. |
| `/etc` | System-wide configuration | Text-based configuration files for the local host. |
| `/home` | Regular user home directories | Each user typically gets a subdirectory here. |
| `/root` | Superuser home directory | Separate from `/home` for the `root` account. |
| `/lib`, `/lib64` | Essential shared libraries | Needed by binaries in `/bin` and `/sbin`. |
| `/media` | Removable media mount points | Common for USB drives and optical media. |
| `/mnt` | Temporary mount point | Used by administrators for ad hoc mounts. |
| `/opt` | Optional add-on software | Often used by third-party applications. |
| `/proc` | Virtual process and kernel information | Not stored on disk; generated by the kernel. |
| `/run` | Volatile runtime data | Cleared at boot; used for PID files and sockets. |
| `/srv` | Service data | Data served by system services, when used. |
| `/sys` | Kernel and device information | Exposes sysfs for hardware and kernel objects. |
| `/tmp` | Temporary files | Usually world-writable with sticky bit protection. |
| `/usr` | Userland applications and read-only shareable data | Contains most installed software and libraries. |
| `/var` | Variable data | Logs, spools, caches, databases, and mail queues. |

**Why `/usr` and `/var` are important**

A common mistake is assuming all software lives in `/bin` or `/etc`. In practice, most installed applications and libraries are under `/usr`, while changing data belongs under `/var`. This separation helps systems remain stable because program files can be treated as mostly static, while logs and caches can grow and change independently.

- `/usr/bin` contains most user commands installed by packages.
- `/usr/sbin` contains many administrative tools.
- `/usr/lib` and `/usr/lib64` contain shared libraries and support files.
- `/var/log` stores log files generated by services and the kernel.
- `/var/lib` stores persistent state for applications and services.
- `/var/cache` stores cache data that can often be regenerated.
- `/var/spool` stores queued work such as print jobs or mail queues.

A practical example is a web server: the executable may live in `/usr/sbin`, configuration in `/etc`, web content in `/var/www` or `/srv/www` depending on policy, and logs in `/var/log`. The exact path can vary by package and distribution conventions, but the data type should still match the directoryâ€™s purpose.

**Configuration, state, and logs**

Red Hat systems rely heavily on text configuration in `/etc`. These files are usually edited by administrators and are intended to define system behavior. By contrast, `/var` contains data that changes during normal operation, and `/run` contains transient runtime information that should not be treated as persistent.

- `/etc` holds host-specific configuration such as network settings, service configs, and authentication policy.
- `/var` holds data that survives reboots and changes over time.
- `/run` holds runtime state that is recreated at boot.
- `/tmp` is for temporary scratch data and should not be used for long-term storage.

> Do not store important data in `/tmp` or `/run`; both are volatile or cleanup-prone and may be cleared automatically.

**Mount points and filesystem layout**

A mount point is an empty directory where another filesystem becomes accessible. This is how Linux can combine multiple disks, partitions, LVM volumes, network filesystems, and removable media into one tree. The directory name itself does not indicate the storage device; the mounted filesystem does.

Common examples include mounting a separate disk on `/home`, a data volume on `/var`, or a USB drive under `/mnt/usb`. You can inspect mounted filesystems with `findmnt`, `mount`, `df`, or `lsblk`.

```bash
findmnt
lsblk -f
df -hT
mount | column -t
```

If you need to mount a filesystem manually, you typically create a mount point and then mount the device there:

```bash
sudo mkdir -p /mnt/data
sudo mount /dev/sdb1 /mnt/data
```

To make the mount persistent, you add an entry to `/etc/fstab` using the filesystem UUID or device path. UUIDs are preferred because device names can change across boots.

```fstab
UUID=1234-ABCD  /mnt/data  xfs  defaults  0  0
```

**Important special-purpose directories**

Some directories are easy to overlook but are essential for system behavior.

- `/dev` contains device files such as `/dev/sda`, `/dev/null`, and `/dev/tty`.
- `/proc` exposes process and kernel information, such as `/proc/cpuinfo` and `/proc/meminfo`.
- `/sys` exposes kernel objects and device attributes through sysfs.
- `/run` often contains PID files, sockets, and service state used by systemd.
- `/boot` contains the kernel and bootloader artifacts needed to start the machine.

These are not ordinary directories backed by regular files on disk. They are virtual or special filesystems that the kernel populates dynamically.

**User and administrator home locations**

User data is typically stored under `/home`, while the `root` account uses `/root`. This separation is intentional: it keeps the superuserâ€™s files isolated from regular users and simplifies recovery if user home directories are on a separate filesystem.

- `/home/alice` is the normal home directory for user `alice`.
- `/root` is the administrative home directory for the superuser.
- Hidden configuration files in home directories begin with a dot, such as `.bashrc` or `.ssh/`.

A quick inspection of home directory contents might look like this:

```bash
ls -la /home/alice
ls -la /root
```

**Temporary storage and cleanup behavior**

Temporary directories are designed for short-lived data. `/tmp` is commonly writable by all users, but the sticky bit prevents users from deleting each otherâ€™s files. Some systems also use `/var/tmp` for temporary data that should survive reboots longer than `/tmp`.

- `/tmp` is for short-lived temporary files.
- `/var/tmp` is for temporary files that may persist across reboots.
- Cleanup policies may remove old files automatically.
- Applications should not rely on temporary directories for durable storage.

You can inspect permissions on `/tmp` with:

```bash
ls -ld /tmp /var/tmp
```

A typical secure `/tmp` directory shows the sticky bit, often as `drwxrwxrwt`.

**Practical command examples**

Use path inspection commands to understand where files live and how the hierarchy is mounted.

```bash
pwd
cd /etc
ls
stat /etc/hosts
file /bin/ls
readlink -f /bin
```

Useful commands for filesystem hierarchy work include:

- `pwd` to show your current directory.
- `cd` to move through the tree.
- `ls -l` and `ls -la` to inspect contents and hidden files.
- `stat` to view metadata such as permissions, ownership, and timestamps.
- `file` to identify file type.
- `readlink -f` to resolve symlinks and canonical paths.

**Common RHCSA exam expectations**

For RHCSA-level tasks, you should be able to identify the purpose of major directories, choose appropriate locations for files, and recognize mount points and runtime data. You are also expected to understand the difference between configuration, persistent state, and temporary data.

- Put host-specific configuration in `/etc`.
- Put variable logs and service state in `/var`.
- Put user files in `/home`.
- Put temporary scratch data in `/tmp` or `/var/tmp` depending on lifespan.
- Use `/mnt` for temporary administrator mounts.
- Use `/media` for removable media.

**Quick mental model**

A simple way to remember the hierarchy is:

- `/etc` = configuration
- `/usr` = installed software
- `/var` = changing data
- `/home` = user data
- `/boot` = startup files
- `/proc`, `/sys`, `/dev` = kernel and device interfaces
- `/tmp` = temporary files

That model helps you decide where a file belongs and whether it should persist, be regenerated, or be treated as system-critical.
