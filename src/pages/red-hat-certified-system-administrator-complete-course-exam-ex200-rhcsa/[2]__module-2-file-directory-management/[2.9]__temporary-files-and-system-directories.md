**Temporary storage on RHEL**

Temporary files are used for short-lived data such as caches, lock files, sockets, PID files, and application scratch space. On RHEL, the most important locations are `/tmp`, `/var/tmp`, `/run`, and `/dev/shm`. You should know which ones persist across reboot, which ones are memory-backed, and which ones are cleaned automatically.

A practical RHCSA mindset is to choose the directory based on lifetime and visibility:

- Use `/tmp` for short-term temporary files that do not need to survive reboot.
- Use `/var/tmp` for temporary data that should survive reboot for a longer period.
- Use `/run` for runtime state created after boot, especially by services.
- Use `/dev/shm` for shared memory and in-memory temporary data.

**Core directory behavior**

`/tmp` and `/var/tmp` are both typically world-writable with the sticky bit set, usually mode `1777`. The sticky bit means users can create files in the directory, but they can only delete or rename files they own. This protects shared temporary directories from accidental or malicious cleanup by other users.

Verify permissions with:

```bash
ls -ld /tmp /var/tmp
```

Typical output looks like:

```bash
drwxrwxrwt. 10 root root 4096 Apr 26 10:00 /tmp
drwxrwxrwt.  3 root root 4096 Apr 26 10:00 /var/tmp
```

The trailing `t` indicates the sticky bit. The owner is usually `root`, and the group is usually `root`, but the key expectation is the `1777` mode.

**Persistence and cleanup differences**

`/tmp` is intended for temporary data that can be removed automatically. On many RHEL systems, `systemd-tmpfiles` cleans `/tmp` on a relatively aggressive schedule, often based on age and boot-time policies. In contrast, `/var/tmp` is preserved longer and is cleaned less aggressively, commonly on a much longer retention schedule. The exact timing can vary by system policy, but the exam-relevant distinction is:

- `/tmp` is short-lived and commonly cleaned sooner.
- `/var/tmp` is longer-lived and commonly retained across reboots and for extended periods.

Because of this, do not store important data in `/tmp`. If an application needs temporary data to survive reboot, `/var/tmp` is the safer choice.

> Do not assume files in `/tmp` remain available after reboot or after a cleanup job runs. If the data matters, store it elsewhere.

**`tmpfs` and memory-backed temporary storage**

On many RHEL configurations, `/tmp` and `/run` may be mounted as `tmpfs`, which is a temporary filesystem stored in RAM and swap rather than on a persistent disk. This has two important effects:

- Data disappears when the system reboots.
- Space usage consumes memory, so large files can reduce available RAM.

If `/tmp` is on `tmpfs`, it is fast but limited by memory policy and mount size. If `/run` is on `tmpfs`, it is ideal for runtime state because it is automatically cleared at boot and does not require disk I/O.

Check mount type and options with:

```bash
findmnt /tmp /run /dev/shm
```

Or:

```bash
mount | grep -E ' /tmp | /run | /dev/shm '
```

Example output may show `tmpfs`:

```bash
tmpfs on /run type tmpfs (rw,nosuid,nodev,relatime,size=1638400k,mode=755)
tmpfs on /dev/shm type tmpfs (rw,nosuid,nodev,relatime,size=655360k,mode=1777)
```

If a directory is on `tmpfs`, remember that file creation affects available memory, not just disk space. This matters when troubleshooting services that fail because runtime files or shared memory allocations exhaust the tmpfs size.

**`/run` for runtime state**

`/run` is a volatile runtime directory used for files that should exist only while the system is running. It replaces older use of `/var/run`, which is commonly a symlink to `/run` on modern systems. Services often place PID files, sockets, and transient state here.

Common examples include:

- PID files for daemons
- Unix domain sockets
- Lock files
- Service-generated runtime data

You can inspect `/var/run` and confirm the relationship:

```bash
ls -ld /var/run /run
readlink -f /var/run
```

A typical result shows `/var/run` resolving to `/run`.

**Systemd runtime directories**

`systemd` can create and manage runtime directories automatically for services. This is useful when a daemon needs a writable directory under `/run` with the correct ownership and permissions.

Example unit file snippet:

```ini
[Service]
User=appuser
Group=appuser
RuntimeDirectory=appservice
RuntimeDirectoryMode=0750
ExecStart=/usr/local/bin/appservice
```

This configuration creates `/run/appservice` at service start and removes it when the service stops. The directory is owned by the service user and group, which reduces manual setup and permission mistakes.

Verify the directory after starting the service:

```bash
systemctl daemon-reload
systemctl start appservice
ls -ld /run/appservice
systemctl status appservice
```

If you need multiple runtime directories, you can list them space-separated:

```ini
RuntimeDirectory=appservice appservice-cache
```

Use `RuntimeDirectory=` when the service needs a clean, boot-scoped location for sockets, PID files, or transient state. This is preferable to creating directories manually in startup scripts.

**`/dev/shm` and shared memory**

`/dev/shm` is another common temporary in-memory location on RHEL systems. It is a `tmpfs` mount used for POSIX shared memory and other applications that need fast, memory-backed temporary storage.

Typical characteristics:

- It is volatile and cleared on reboot.
- It is memory-backed, so it consumes RAM.
- It is usually world-writable with sticky bit mode `1777`.

Check it with:

```bash
ls -ld /dev/shm
findmnt /dev/shm
```

Example:

```bash
drwxrwxrwt. 2 root root 40 Apr 26 10:00 /dev/shm
```

Applications may use `/dev/shm` for inter-process communication or temporary data that benefits from RAM speed. Be careful not to confuse it with persistent storage.

**Ownership and permissions expectations**

For shared temporary directories, the standard expectation is:

- `root:root` ownership
- mode `1777`
- sticky bit set

This applies most commonly to `/tmp`, `/var/tmp`, and `/dev/shm`. The sticky bit is essential because it prevents one user from deleting another user’s files in a shared writable directory.

Check permissions and ownership with:

```bash
stat -c '%A %a %U %G %n' /tmp /var/tmp /dev/shm
```

If permissions are wrong, applications may fail or the system may be less secure. For example, a missing sticky bit on `/tmp` is a serious security issue.

**Managing temporary files with `systemd-tmpfiles`**

RHEL uses `systemd-tmpfiles` to create, clean, and manage temporary files and directories based on configuration rules. This is the mechanism behind many automatic cleanup behaviors.

Useful commands:

```bash
systemd-tmpfiles --clean
systemd-tmpfiles --create
systemd-tmpfiles --remove
```

You can inspect tmpfiles configuration in:

- `/usr/lib/tmpfiles.d/`
- `/etc/tmpfiles.d/`

A simple rule example:

```conf
d /run/myapp 0750 appuser appuser -
```

This creates `/run/myapp` with the specified mode and ownership. The last field controls age-based cleanup behavior; `-` means no automatic age cleanup for that rule.

For troubleshooting, review logs and configuration:

```bash
journalctl -u systemd-tmpfiles-clean
grep -R "myapp" /etc/tmpfiles.d /usr/lib/tmpfiles.d
```

**When to use each directory**

A quick mental model helps during the exam:

- `/tmp`
  - Short-lived scratch space
  - May be cleaned automatically
  - Often `tmpfs` or disk-backed depending on configuration
- `/var/tmp`
  - Longer-lived temporary storage
  - Survives reboot more often than `/tmp`
  - Cleaner runs less aggressively
- `/run`
  - Runtime state for the current boot
  - Usually `tmpfs`
  - Ideal for PID files, sockets, and service state
- `/dev/shm`
  - Shared memory and RAM-backed temporary data
  - Fast, volatile, and limited by memory

**Practical verification workflow**

When you need to inspect temporary storage on a system, follow a simple workflow:

1. Check permissions and ownership.
2. Check mount type and available space.
3. Confirm whether the directory is volatile or persistent.
4. Review cleanup policy if files disappear unexpectedly.

Example commands:

```bash
ls -ld /tmp /var/tmp /run /dev/shm
df -h /tmp /var/tmp /run /dev/shm
findmnt /tmp /var/tmp /run /dev/shm
```

If a service depends on a runtime directory, verify it after startup:

```bash
systemctl status myservice
ls -ld /run/myservice
ss -xl | grep myservice
```

**Troubleshooting common issues**

- If a service cannot create a PID file or socket, check whether `/run` exists and whether the service has the right `RuntimeDirectory=` or permissions.
- If temporary files vanish after reboot, check whether they were stored in `/tmp` or `/run` instead of `/var/tmp`.
- If a system runs out of memory unexpectedly, inspect large files in `tmpfs` mounts such as `/tmp`, `/run`, or `/dev/shm`.
- If users can delete each other’s files in a shared temp directory, verify the sticky bit and mode `1777`.

**Exam-focused reminders**

- Know the difference between volatile and persistent temporary locations.
- Remember that `tmpfs` uses memory, not disk, and clears on reboot.
- Expect `/tmp`, `/var/tmp`, and `/dev/shm` to be world-writable with sticky bit in standard setups.
- Use `RuntimeDirectory=` for systemd-managed runtime paths under `/run`.
- Understand that `/tmp` cleanup is typically more aggressive than `/var/tmp` cleanup on RHEL systems.
