## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Wildcards and globbing overview**

<abbr title="Command-line interface">CLI</abbr> globbing is the shell’s filename pattern matching feature. You use it to match multiple pathnames with a single pattern, which is essential for efficient administration on RHCSA tasks such as copying, moving, deleting, and inspecting groups of files.

Globbing happens in the shell before the command runs. That means the shell expands a pattern like `*.log` into a list of matching pathnames, and then the command receives those expanded names as arguments. This is different from commands that do their own matching, such as `find`.

- `*` matches any string, including an empty string.
- `?` matches exactly one character.
- `[abc]` matches one character from a set.
- `[a-z]` matches one character in a range.
- `[!abc]` or `[^abc]` matches one character not in the set, depending on shell behavior.

**Shell expansion order matters**

To predict results correctly, you need to know the order in which the shell processes text. In Bash, the important sequence for pattern-related expansions is:

1. Brace expansion
2. Tilde expansion
3. Parameter expansion, command substitution, and arithmetic expansion
4. Word splitting
5. Pathname expansion, also called globbing
6. Quote removal

This order explains why a pattern may change before globbing happens. For example, variables and command output can create a glob pattern that is expanded later.

```bash
pattern="*.txt"
echo $pattern
echo "$pattern"
```

- `echo $pattern` allows pathname expansion if the variable contains a glob pattern.
- `echo "$pattern"` prevents pathname expansion because quotes suppress globbing.
- Brace expansion is not globbing; it is a separate shell feature and is not universally available in all shells.

> Brace expansion is a Bash feature and similar behavior exists in some other shells, but you should not treat it as portable shell globbing.

**Core wildcard patterns**

Use wildcards when you want the shell to select files by name pattern instead of typing each pathname manually.

```bash
ls *.conf
cp /etc/sysconfig/network-scripts/ifcfg-* /backup/
rm report?.txt
```

- `*` is broad and can match many files, so use it carefully with destructive commands.
- `?` is useful when names differ by a single character, such as `file1` through `file9`.
- Character classes are precise and help avoid accidental matches.
- Ranges depend on locale and collation rules, so results can vary if you assume ASCII ordering.

**Hidden files and dotfiles**

Files beginning with `.` are hidden in Linux. Standard globbing does not match them with `*`, so `ls *` will not show `.bashrc` or `.config`.

To match hidden files, you often need a pattern that starts with a dot:

```bash
ls -d .*
ls -d .[!.]*
ls -d .??*
```

The pattern `.*` matches `.` and `..` as well as hidden files. That is important because `.` means the current directory and `..` means the parent directory.

- `rm -r .*` is dangerous because it can try to act on `.` and `..`.
- Use safer patterns such as `.[!.]*` and `.??*` when you want hidden files but not `.` or `..`.
- `shopt -s dotglob` changes Bash so that `*` also matches dotfiles.
- `shopt -u dotglob` restores the default behavior.

```bash
shopt -s dotglob
printf '%s\n' *
shopt -u dotglob
```

**Shell options that affect globbing**

Bash provides options that change how pathname expansion behaves. These are exam-relevant because they can change command results in subtle ways.

- `nullglob` makes unmatched patterns expand to nothing instead of staying literal.
- `failglob` makes unmatched patterns produce an error.
- `dotglob` makes `*` and `?` match leading dots in filenames.
- `extglob` enables extended pattern syntax such as `@(a|b)` and `!(pattern)`.

```bash
shopt -s nullglob
files=(*.txt)
printf '%s\n' "${files[@]}"
```

```bash
shopt -s extglob
ls @(*.txt|*.log)
```

- `nullglob` is useful in scripts when you want to avoid processing a literal unmatched pattern.
- `failglob` is safer when an unmatched pattern should be treated as an error.
- `extglob` is powerful, but you should enable it only when you need advanced matching.
- These are Bash shell options, not universal POSIX shell behavior.

**Brace expansion is separate from globbing**

Brace expansion creates text combinations before pathname expansion occurs. It does not inspect the filesystem.

```bash
echo file{1,2,3}.txt
echo {a,b}/config
```

- `file{1,2,3}.txt` becomes `file1.txt file2.txt file3.txt`.
- Brace expansion does not require matching files to exist.
- Brace expansion is not the same as `*` or `?`.
- Do not assume brace expansion works in every shell.

**Practical command examples**

Globbing is common in file management and inspection tasks.

```bash
ls /var/log/*.log
cp /etc/*.conf /tmp/config-backup/
mv report-2024-??.txt archive/
chmod 640 /var/log/httpd/*.log
```

- `ls /var/log/*.log` lists only files ending in `.log` in that directory.
- `cp /etc/*.conf /tmp/config-backup/` copies all matching configuration files.
- `chmod 640 /var/log/httpd/*.log` changes permissions on all matching files.
- If no files match, the behavior depends on shell options such as `nullglob` and `failglob`.

**Quoting and escaping patterns**

Quotes control whether the shell expands a pattern. Use them when you want to pass a literal wildcard character to a command.

```bash
echo "*.txt"
touch 'file*name'
grep '\*' somefile
```

- Double quotes still allow parameter expansion, command substitution, and arithmetic expansion, but they suppress pathname expansion.
- Single quotes preserve the literal text exactly.
- Backslash escapes a single special character.
- If you want the shell to expand a pattern, do not quote the wildcard characters.

**Globbing versus `find` patterns**

A common exam distinction is that shell globbing and `find` matching are not the same thing. The shell expands globs before `find` runs, while `find` evaluates its own patterns internally.

```bash
find /var/log -name '*.log'
find /home -iname 'report*'
```

- `-name` uses shell-style wildcard matching inside `find`.
- `-iname` is case-insensitive.
- Quote the pattern so the shell does not expand it first.
- `find` is often better when you need recursive search or more control than shell globbing provides.

> If you write `find /var/log -name *.log` without quotes, the shell may expand `*.log` before `find` sees it, which changes the command and can break the search.

**Safe deletion and modification habits**

Wildcards are powerful, but they can be risky with destructive commands. Always verify what will match before you run `rm`, `mv`, or `chmod`.

```bash
printf '%s\n' *.tmp
rm -- *.tmp
```

- Preview matches with `printf '%s\n' pattern` or `ls pattern` before deleting.
- Use `--` to stop option parsing when filenames might begin with `-`.
- Prefer narrow patterns over broad ones.
- Be especially careful in directories with many similarly named files.

**RHCSA-style practice tasks**

Use these exercises to build exam-ready pattern selection skills.

1. Create a practice directory and sample files.
   ```bash
   mkdir -p ~/glob-practice
   cd ~/glob-practice
   touch app1.log app2.log appA.log notes.txt .hidden1 .hidden2
   ```
2. List only the `.log` files.
   ```bash
   ls *.log
   ```
3. List hidden files without matching `.` or `..`.
   ```bash
   ls -d .[!.]* .??*
   ```
4. Copy all `.txt` files to a backup directory.
   ```bash
   mkdir -p ~/glob-backup
   cp *.txt ~/glob-backup/
   ```
5. Use `find` to locate files recursively by name pattern.
   ```bash
   find ~ -type f -name '*.log'
   ```
6. Test how shell options change behavior.
   ```bash
   shopt -s nullglob dotglob extglob
   printf '%s\n' *
   shopt -u nullglob dotglob extglob
   ```

**Common pitfalls**

- `*` does not match hidden files unless `dotglob` is enabled or the pattern starts with a dot.
- `.*` includes `.` and `..`, which can cause accidental recursion or deletion mistakes.
- Unquoted patterns in `find -name` are expanded by the shell too early.
- Brace expansion is not globbing and should not be used as a substitute for filename matching.
- `nullglob` and `failglob` can change whether unmatched patterns remain literal or cause errors.
- Locale and shell settings can affect range matching, so verify results when precision matters.

**Exam-focused takeaways**

Globbing is a shell-side filename selection tool, not a command feature. You should know when the shell expands patterns, how quoting changes behavior, and how shell options alter matching. For RHCSA tasks, the safest workflow is to preview matches, use the narrowest pattern that fits the requirement, and choose `find` when you need recursive or conditional selection.
