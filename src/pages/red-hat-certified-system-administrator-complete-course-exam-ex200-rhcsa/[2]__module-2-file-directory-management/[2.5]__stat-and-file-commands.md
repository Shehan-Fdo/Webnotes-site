## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

`stat` and `file` solve different problems, and RHCSA tasks often require you to choose the right one quickly. Use `stat` when you need metadata such as size, permissions, ownership, timestamps, inode, or link count. Use `file` when you need to identify what a path actually is: regular file, directory, symlink, socket, FIFO, block device, character device, or a file with a recognizable format.

A practical way to remember the difference:

- **`stat`** answers “What are this object’s attributes?”
- **`file`** answers “What kind of object is this?”

Both commands are safe for inspection and do not modify the target. They are especially useful during troubleshooting, validation, and exam verification.

**Why these commands matter in RHCSA work**

You use these commands when a path behaves unexpectedly, when a script needs to branch on file type, or when you must confirm that a configuration file, device node, or symlink is correct. In exam scenarios, they help you verify your work without guessing.

Common RHCSA-style uses include:

- confirming that a file exists and is the expected type
- checking whether a symlink points where you think it does
- verifying ownership and permissions after a configuration change
- identifying sockets, FIFOs, and device nodes
- checking timestamps to understand whether a file was recently modified

**`stat` fundamentals**

`stat` prints detailed metadata from the filesystem. It is more precise than `ls` for troubleshooting because it exposes inode-level information and multiple timestamps.

Typical output includes:

- **Size** in bytes
- **Blocks** allocated on disk
- **IO Block** size
- **File type**
- **Device** and **inode** numbers
- **Links** count
- **Access**, **Modify**, and **Change** times
- **UID** and **GID** ownership
- **Permissions** in both symbolic and octal form

Example:

```bash
stat /etc/passwd
```

Sample output shape:

```text
  File: /etc/passwd
  Size: 2684       Blocks: 8          IO Block: 4096   regular file
Device: 253,0      Inode: 131073      Links: 1
Access: (0644/-rw-r--r--)  Uid: (    0/    root)   Gid: (    0/    root)
Access: 2026-04-26 10:12:01.000000000 +0000
Modify: 2026-04-25 18:40:10.000000000 +0000
Change: 2026-04-25 18:40:10.000000000 +0000
 Birth: -
```

**Key `stat` fields**

- **Access time** (`atime`) changes when the file is read, depending on mount options and filesystem behavior.
- **Modify time** (`mtime`) changes when file contents change.
- **Change time** (`ctime`) changes when metadata changes, such as permissions, ownership, or link count.
- **Birth time** may be unavailable on many Linux filesystems and often appears as `-`.
- **Inode** identifies the file object on the filesystem, which is useful when comparing hard links.

> `ctime` is not “creation time.” It is the metadata change time.

**When to use `stat`**

Use `stat` when you need exact metadata, especially in these cases:

- verifying permissions after `chmod`
- checking ownership after `chown`
- comparing hard links by inode and link count
- confirming whether a file changed recently
- diagnosing why a script sees a file as “old” or “new”
- checking the size of a file before copying or archiving it

Example: compare two hard links

```bash
ln /tmp/report.txt /tmp/report-link.txt
stat /tmp/report.txt /tmp/report-link.txt
```

If both names point to the same inode, they are hard links to the same file data.

**Useful `stat` formatting options**

`stat` can print custom output, which is helpful in scripts and quick checks.

Common format examples:

```bash
stat -c '%n %F %s bytes' /etc/passwd
stat -c 'inode=%i links=%h perms=%a owner=%U group=%G' /etc/passwd
stat -c '%n -> %N' /bin/sh
```

Useful format specifiers:

- `%n` file name
- `%N` quoted file name and dereference info
- `%F` file type
- `%s` total size
- `%i` inode number
- `%h` hard link count
- `%a` permissions in octal
- `%A` permissions in symbolic form
- `%U` owner name
- `%G` group name
- `%x` access time
- `%y` modify time
- `%z` change time

Example for scripting:

```bash
#!/bin/bash
path="$1"

if [ -e "$path" ]; then
  stat -c '%n exists: type=%F size=%s inode=%i' "$path"
else
  echo "$path does not exist"
fi
```

**`file` fundamentals**

`file` identifies the type of a path by examining the file contents and metadata. It is especially useful when the filename extension is misleading or absent.

Examples of what `file` can identify:

- plain text
- shell scripts
- ELF binaries
- compressed archives
- image formats
- device nodes
- sockets and FIFOs
- symlinks

Example:

```bash
file /etc/passwd
file /usr/bin/ls
file /tmp/archive.tar.gz
```

Typical output:

```text
/etc/passwd: ASCII text
/usr/bin/ls: ELF 64-bit LSB pie executable, x86-64, dynamically linked, ...
/tmp/archive.tar.gz: gzip compressed data, was "archive.tar", last modified: ...
```

**How `file` behaves with symlinks**

By default, `file` reports information about the symlink itself rather than following it. If you want it to inspect the target, use `-L`.

Examples:

```bash
file /bin/sh
file -L /bin/sh
```

This distinction matters because a symlink can point to a different object type than the link itself. In exam work, you often want to know both:

- what the path name is
- what the target resolves to

**Important note**

> Do not assume a path like `/bin` or `/bin/sh` has the same layout on every system. Use a path that exists on the target machine and verify it with `ls -l` or `readlink` if needed.

**When to use `file`**

Use `file` when you need to identify the nature of a path or content, especially when:

- a file has no extension
- a file extension is misleading
- you need to confirm a script or binary type
- you are checking whether a path is a symlink, socket, FIFO, or device node
- you want to know whether a file is text or binary before opening it in an editor

Example troubleshooting workflow:

```bash
file /var/log/messages
file /dev/null
file /run/cron.pid
```

**Comparing `stat` and `file`**

| Command | Best for | Typical output focus | Strength |
|---|---|---|---|
| `stat` | metadata inspection | size, inode, permissions, timestamps | precise filesystem attributes |
| `file` | type identification | file format or special file type | quick recognition of content/type |

A simple decision rule:

- Use **`stat`** when you care about ownership, permissions, timestamps, inode, or link count.
- Use **`file`** when you care about what the path or content actually is.

**Practical RHCSA scenarios**

1. **You changed permissions and want to verify them**
   ```bash
   chmod 640 /etc/myapp.conf
   stat /etc/myapp.conf
   ```
   Check the octal mode and symbolic permissions.

2. **You suspect a path is a symlink**
   ```bash
   ls -l /usr/bin/python
   file /usr/bin/python
   file -L /usr/bin/python
   ```
   Use `ls -l` to see the link target, `file` to identify the link itself, and `file -L` to inspect the target.

3. **You need to confirm a device node**
   ```bash
   file /dev/sda
   stat /dev/sda
   ```
   `file` identifies the node type; `stat` shows ownership and permissions.

4. **You are checking whether a script is executable and what it is**
   ```bash
   stat -c '%A %U %G %n' /usr/local/bin/backup.sh
   file /usr/local/bin/backup.sh
   ```
   This helps you confirm both execution bits and file format.

5. **You need to compare two names for the same file**
   ```bash
   stat -c '%n inode=%i links=%h' /etc/hosts /tmp/hosts-copy
   ```
   If the inode matches, the names refer to the same underlying file via hard links.

**Common special file types**

`file` is useful for identifying special files that are not regular documents or binaries.

- **Regular file**: normal data file
- **Directory**: container for names
- **Symbolic link**: pointer to another path
- **FIFO/pipe**: named pipe used for process communication
- **Socket**: endpoint for interprocess communication
- **Block device**: buffered device such as disks
- **Character device**: unbuffered device such as terminals

Example:

```bash
file /dev/null
file /tmp/myfifo
file /run/systemd/private
```

If you need to create a FIFO for testing:

```bash
mkfifo /tmp/demo.pipe
file /tmp/demo.pipe
stat /tmp/demo.pipe
```

**Reading `stat` output for troubleshooting**

`stat` is especially helpful when a file seems correct in `ls -l` but still behaves unexpectedly.

Look for these clues:

- **Link count** greater than 1 suggests hard links.
- **Inode** mismatch means two paths are different files.
- **Change time** newer than modify time may indicate metadata changes only.
- **Permissions** may look correct, but ownership or ACLs may still block access.
- **Size** may be zero even when the file exists, which matters for config files and logs.

Example:

```bash
stat /etc/ssh/sshd_config
```

If the file exists but service behavior is wrong, check whether the file was edited, whether the timestamps make sense, and whether the permissions are restrictive enough for the service to read it.

**Reading `file` output for troubleshooting**

`file` can reveal when a file is not what its name suggests.

Examples:

- a file named `backup.txt` may actually be a gzip archive
- a file named `script.sh` may be a binary
- a path may be a symlink rather than the expected regular file
- a configuration file may contain binary data due to corruption

Example:

```bash
file backup.txt
```

If the output says compressed data, you should treat it as an archive rather than plain text.

**Useful command combinations**

`stat` and `file` are often paired with other commands:

```bash
ls -l /path/to/item
readlink -f /path/to/symlink
namei -l /path/to/deep/path
find /var/log -type f -name '*.log' -exec stat -c '%n %s %y' {} \;
```

How these help:

- `ls -l` gives a quick human-readable view.
- `readlink -f` resolves a symlink chain to the final target.
- `namei -l` shows each path component and permissions.
- `find ... -exec stat ...` lets you audit many files at once.

**Troubleshooting patterns**

- If a file “looks right” but access fails, use `stat` to confirm ownership and mode.
- If a file opens as gibberish, use `file` to check whether it is binary or compressed.
- If a path behaves strangely, use `file` and `stat` together to distinguish the link from the target.
- If a script depends on file age, use `stat` timestamps rather than `ls` output alone.

**Quick reference**

- `stat <path>`: show detailed metadata
- `stat -c '<format>' <path>`: custom output
- `file <path>`: identify file type
- `file -L <path>`: inspect symlink target
- `readlink <path>`: show symlink target
- `ls -l <path>`: quick listing with permissions and link info

**Mini decision flow**

1. Need permissions, ownership, timestamps, inode, or link count?
   - Use `stat`.
2. Need to know whether the path is text, binary, archive, symlink, socket, or device?
   - Use `file`.
3. Need to know where a symlink points?
   - Use `readlink` or `ls -l`.
4. Need to inspect the target of a symlink?
   - Use `file -L` or `stat` on the resolved path.

**Exam tips**

- Verify your changes with `stat` after `chmod`, `chown`, or `ln`.
- Use `file` before editing unknown files to avoid opening binaries in a text editor.
- Do not rely on filename extensions during the exam.
- Prefer exact checks over assumptions, especially for symlinks and special files.
- If a path is missing, confirm the parent directories with `namei -l` before assuming the file is wrong.

**Common command examples**

```bash
stat /etc/hosts
stat -c '%n %a %U %G %s' /etc/hosts
file /etc/hosts
file -L /usr/bin/python3
readlink -f /usr/bin/python3
```

These commands give you a fast, reliable workflow for validating file identity and metadata during RHCSA tasks.
