## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**File types in Linux**

Linux treats “everything as a file” in the sense that many resources are represented through file-like interfaces, but the filesystem still distinguishes several file types. On RHCSA systems, you should be able to identify them quickly because commands, permissions, and troubleshooting steps often depend on the type.

- **Regular files** store text, binaries, logs, and data.
- **Directories** contain filename-to-inode mappings for other files.
- **Symbolic links** point to another pathname.
- **Hard links** are additional directory entries for the same inode.
- **Character devices** and **block devices** represent hardware or virtual devices under `/dev`.
- **FIFOs** (named pipes) support one-way interprocess communication.
- **Sockets** support bidirectional communication between processes.

Use `ls -l` to inspect the leading file-type character:

```bash
-rw-r--r-- 1 user user  1200 Apr 26 10:00 report.txt
drwxr-xr-x 2 user user  4096 Apr 26 10:01 docs
lrwxrwxrwx 1 user user    11 Apr 26 10:02 latest -> report.txt
crw-rw-rw- 1 root root 1, 3 Apr 26 10:03 null
brw-rw---- 1 root disk 8, 0 Apr 26 10:04 sda
prw-r--r-- 1 user user     0 Apr 26 10:05 mypipe
srw-rw-rw- 1 root root     0 Apr 26 10:06 mysock
```

The first character is the file type indicator:

- `-` regular file
- `d` directory
- `l` symbolic link
- `c` character device
- `b` block device
- `p` FIFO
- `s` socket

`file` is often more reliable than `ls` when you need to identify content or special file behavior:

```bash
file /bin/ls
file /etc/passwd
file latest
```

**Inodes and metadata**

An <abbr title="index node">inode</abbr> stores metadata about a file, not the filename itself. The directory entry maps a name to an inode number. This distinction matters for hard links, deletion behavior, and space usage.

Typical inode metadata includes:

- file type
- permissions and mode bits
- owner and group
- size
- timestamps
- link count
- pointers to data blocks
- extended attributes and ACL-related metadata, depending on filesystem support

View inode numbers with `ls -i`:

```bash
ls -i /etc/passwd
ls -li /var/log/messages
```

Use `stat` for a detailed metadata view:

```bash
stat /etc/passwd
```

A typical `stat` output shows:

- file name
- size in bytes
- blocks allocated
- inode number
- device ID
- access mode
- UID and GID ownership
- access, modify, and change times

**Why inode knowledge matters**

- A file can have multiple names if it has multiple hard links.
- Deleting a filename removes one directory entry, not necessarily the data.
- A file’s data is freed only when the link count reaches zero and no process keeps it open.
- Renaming a file usually changes the directory entry, not the inode.

**Permissions and mode bits**

File mode combines the file type and permission bits. The standard permission model uses three classes:

- **user**: the file owner
- **group**: members of the file’s group
- **others**: everyone else

Each class can have:

- `r` read
- `w` write
- `x` execute

Interpretation depends on file type:

- For regular files, `r` allows reading contents, `w` allows modifying contents, and `x` allows execution.
- For directories, `r` allows listing names, `w` allows creating/removing entries, and `x` allows entering the directory and accessing entries by name.

Example:

```bash
drwxr-x--- 2 alice devops 4096 Apr 26 10:15 project
```

This means:

- owner `alice` can read, write, and enter the directory
- group `devops` can read and enter, but not create or delete entries
- others have no access

Use `chmod`, `chown`, and `chgrp` to manage ownership and permissions:

```bash
chmod 640 file.txt
chmod u+x script.sh
chown alice:devops report.txt
chgrp wheel report.txt
```

**Special mode bits**

Linux also supports special permission bits that affect file behavior:

- **setuid**: executable runs with the file owner’s effective UID
- **setgid**: executable runs with the file group’s effective GID
- **sticky bit**: on a directory, only the owner of a file, the directory owner, or root can delete or rename entries

Common examples:

- `setuid` on trusted administrative binaries
- `setgid` on shared project directories so new files inherit the directory group
- sticky bit on `/tmp`

View special bits in symbolic form:

```bash
ls -ld /tmp
ls -l /usr/bin/passwd
```

View them numerically with `stat`:

```bash
stat -c '%A %a %n' /usr/bin/passwd /tmp
```

Set them with `chmod`:

```bash
chmod u+s /usr/local/bin/tool
chmod g+s /shared/team
chmod +t /shared/public
```

> Be careful with special bits. Incorrect use can create privilege escalation risks or unexpected access patterns.

**Hard links and symbolic links**

Hard links and symbolic links solve different problems.

| Feature | Hard link | Symbolic link |
|---|---|---|
| Points to | Same inode | Pathname |
| Cross-filesystem | No | Yes |
| Can link directories | Usually no | Yes, but avoid manually |
| Breaks if target deleted | No, data remains until last link removed | Yes, becomes dangling |
| Inode count | Shared | Separate inode |

Create links with `ln`:

```bash
ln original.txt hardlink.txt
ln -s original.txt symlink.txt
```

Check link counts and targets:

```bash
ls -li original.txt hardlink.txt symlink.txt
readlink symlink.txt
```

Practical implications:

- Hard links are useful when you need another name for the same data without duplication.
- Symbolic links are useful for convenience, version switching, and compatibility paths.
- A symlink can point to a missing target; the link still exists, but access fails.
- A hard link cannot normally span filesystems because inodes are local to a filesystem.

**Timestamps**

Linux files commonly track three core timestamps:

- **atime**: last access time
- **mtime**: last content modification time
- **ctime**: last metadata change time

These are not the same:

- Reading a file may update `atime`, depending on mount options and filesystem behavior.
- Editing file contents updates `mtime` and `ctime`.
- Changing permissions, ownership, or link count updates `ctime` but not `mtime`.

Inspect timestamps with `stat`:

```bash
stat /etc/hosts
```

You can also format them:

```bash
stat -c 'Access: %x
Modify: %y
Change: %z
Birth: %w' /etc/hosts
```

Use timestamps carefully in troubleshooting:

- `mtime` helps determine when content changed.
- `ctime` helps determine when metadata changed.
- `atime` is less reliable for auditing user reads because it may be disabled or reduced for performance.

**Extended metadata**

Beyond basic Unix permissions, files may carry additional metadata:

- **ACLs** (Access Control Lists) for fine-grained permissions
- **SELinux** security contexts on Red Hat systems
- **extended attributes** for filesystem-specific tags

Inspect ACLs with `getfacl`:

```bash
getfacl /shared/project
```

Inspect SELinux context with `ls -Z`:

```bash
ls -Z /etc/passwd
```

Inspect extended attributes with `getfattr` if available:

```bash
getfattr -d /path/to/file
```

These layers matter because a file can appear correctly permissioned but still be inaccessible due to ACLs or SELinux policy.

**Common metadata inspection commands**

Use the right tool for the job:

- `ls -l` for quick permission and type checks
- `ls -li` for inode numbers and link counts
- `stat` for detailed metadata
- `file` for file type and content hints
- `readlink` for symlink targets
- `find` for locating files by type or metadata
- `getfacl` and `ls -Z` for access-control context

Examples:

```bash
find /var/log -type f -name '*.log'
find /dev -type c
find /tmp -type l
find /home -user alice -group devops
```

**Practical troubleshooting workflow**

When a file behaves unexpectedly, inspect it in layers:

1. Check the file type with `ls -l` or `file`.
2. Check ownership and permissions with `stat` or `ls -l`.
3. Check link count and inode with `ls -li`.
4. Check ACLs with `getfacl`.
5. Check SELinux context with `ls -Z`.
6. Check timestamps to understand recent changes.

Example workflow:

```bash
ls -li /usr/local/bin/tool
stat /usr/local/bin/tool
getfacl /usr/local/bin/tool
ls -Z /usr/local/bin/tool
```

**Real-world examples**

A symlink is often used for versioned deployments:

```bash
ln -sfn /opt/app/releases/2026-04-26 /opt/app/current
```

Here `-s` creates a symbolic link, `-f` forces replacement, and `-n` treats the destination as a normal file if it is a symlink. This pattern lets you switch application versions by updating one path.

A shared directory may use setgid so new files inherit the group:

```bash
mkdir /srv/team
chgrp devops /srv/team
chmod 2775 /srv/team
```

The leading `2` sets the setgid bit on the directory, which helps keep group ownership consistent for collaborative work.

A temporary collaboration directory may use the sticky bit:

```bash
mkdir /srv/public-drop
chmod 1777 /srv/public-drop
```

This allows many users to create files while preventing them from deleting each other’s files.

**Exam-focused reminders**

- Know the difference between a filename, inode, and file content.
- Recognize file types from `ls -l` output.
- Understand how hard links differ from symlinks.
- Interpret `r`, `w`, and `x` differently for files and directories.
- Use `stat` to answer metadata questions quickly.
- Remember that `ctime` is metadata change time, not creation time.
- Treat ACLs and SELinux as separate layers from standard mode bits.

**Quick reference**

```bash
ls -l      # permissions, type, ownership, link count
ls -li     # inode number and link count
stat file  # detailed metadata
file file  # file type/content guess
ln file a  # hard link
ln -s file a  # symbolic link
chmod 644 file
chown user:group file
getfacl file
ls -Z file
```

Understanding file types and metadata helps you diagnose access problems, preserve data correctly, and reason about how Linux stores and protects files. On RHCSA, this knowledge is foundational because many later tasks depend on accurate interpretation of permissions, links, and filesystem state.
