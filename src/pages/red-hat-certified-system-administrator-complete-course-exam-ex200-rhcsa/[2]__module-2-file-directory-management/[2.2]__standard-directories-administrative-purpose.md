## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Filesystem hierarchy overview**

Linux systems follow the <abbr title="Filesystem Hierarchy Standard">FHS</abbr>, which defines where files should live so administrators can predict locations across systems. On Red Hat Enterprise Linux (RHEL), this matters for troubleshooting, package management, backups, and service administration.

The main idea is simple: place each file type where the system expects it. That makes upgrades safer, automation easier, and recovery more reliable.

- **`/`** is the root of the entire filesystem tree.
- **`/bin`** and **`/sbin`** contain essential commands needed for boot, repair, and basic administration.
- **`/usr`** holds most userland programs, libraries, and shared data.
- **`/var`** stores data that changes during normal operation.
- **`/etc`** contains system-wide configuration files.
- **`/home`** stores user home directories.
- **`/tmp`** is for temporary files and should not hold important data.
- **`/opt`** and **`/usr/local`** are common places for software not delivered by the base OS packages.

**Major top-level directories**

The table below gives an exam-oriented summary of the most important standard directories and their administrative purpose.

| Directory | Purpose | Typical contents | Admin notes |
|---|---|---|---|
| `/` | Root of the filesystem tree | All other directories mount from here | Keep it small and stable |
| `/bin` | Essential user commands | `ls`, `cp`, `mv`, `cat` | Needed for basic system recovery |
| `/sbin` | Essential system administration commands | `ip`, `mount`, `fsck`, `shutdown` | Often used by root or via `sudo` |
| `/boot` | Boot loader files and kernel images | `vmlinuz`, `initramfs`, `grub2` files | Protect from accidental changes |
| `/dev` | Device files | Disk, terminal, and pseudo-device nodes | Managed dynamically by `udev` |
| `/etc` | Host-specific configuration | `sshd_config`, `fstab`, `passwd`, `chrony.conf` | Text-based config is common |
| `/home` | User home directories | Personal files and dotfiles | Back up separately from system files |
| `/root` | Root user's home directory | Root shell profile and admin files | Not the same as `/` |
| `/lib`, `/lib64` | Essential shared libraries and kernel modules | Runtime libraries, loader files | Required by core binaries |
| `/media` | Mount points for removable media | USB drives, optical media | Usually mounted automatically |
| `/mnt` | Temporary mount point | Manual mounts for testing or recovery | Create mount point before mounting |
| `/opt` | Add-on application software | Third-party vendor packages | Common for self-contained vendor apps |
| `/proc` | Virtual process and kernel information | Process IDs, kernel parameters | Not stored on disk |
| `/run` | Volatile runtime data | PID files, sockets, lock files | Recreated at boot |
| `/srv` | Data served by system services | Web or FTP content, if used | Less common on many systems |
| `/sys` | Kernel and device information | Hardware and driver interfaces | Virtual filesystem |
| `/tmp` | Temporary files | Short-lived scratch files | Cleanup depends on policy and configuration |
| `/usr` | Shareable, mostly read-only userland | Programs, libraries, docs, shared data | Large part of installed software lives here |
| `/usr/bin` | Most user commands | `dnf`, `vim`, `ssh`, `tar` | Common PATH location |
| `/usr/sbin` | Non-essential admin commands | `useradd`, `semanage`, `restorecon` | Often available to admins |
| `/usr/lib`, `/usr/lib64` | Libraries for `/usr` programs | Shared objects, modules | Architecture-dependent |
| `/usr/share` | Architecture-independent data | Man pages, icons, locale data, docs | Shared across architectures |
| `/usr/local` | Locally installed software and site-specific additions | Custom scripts, locally built binaries | Not managed by RPM packages |
| `/var` | Variable data | Logs, spool, caches, databases | Grows over time |
| `/var/log` | Log files | `messages`, `secure`, application logs | Monitor for troubleshooting |
| `/var/lib` | Persistent application state | Databases, package metadata, service state | Common place for service data |
| `/var/cache` | Cached data | Rebuildable caches | Safe to clear in many cases, but verify first |
| `/var/spool` | Queued data | Print queues, mail queues, cron/spool data | Often service-managed |
| `/var/tmp` | Temporary files kept longer than `/tmp` | Large scratch files | May persist across reboots depending on policy |
| `/dev/shm` | Shared memory filesystem | In-memory temporary data | Useful for IPC and fast temporary storage |

**`/etc`: system configuration**

`/etc` contains host-specific configuration that controls how the system behaves. On RHEL, many administrative tasks involve editing files here and then reloading or restarting the relevant service.

Common examples include:

- ` /etc/fstab` for static filesystem mounts
- `/etc/ssh/sshd_config` for OpenSSH server settings
- `/etc/chrony.conf` for time synchronization
- `/etc/hosts` for local name resolution
- `/etc/resolv.conf` for DNS resolver settings, often managed by NetworkManager or other tools

A practical example of mount configuration:

```fstab
UUID=3f2a1d8c-9b2e-4d3a-8f5a-1c2d3e4f5a6b  /data  xfs  defaults  0  0
```

After editing `/etc/fstab`, verify the syntax before rebooting:

```bash
sudo mount -a
findmnt --verify
```

> Be careful with `/etc/fstab`: a bad entry can delay boot or drop the system into emergency mode.

**`/usr`: installed software and shared data**

`/usr` is one of the most important directories for exam work. On modern Linux systems, it contains the majority of installed userland software, libraries, and documentation. Many commands you run daily come from `/usr/bin`, while administrative utilities may live in `/usr/sbin`.

Use `/usr/share` for architecture-independent data such as:

- man pages
- documentation
- icons
- locale files
- application templates

This distinction matters because `/usr/share` can be shared across systems with different CPU architectures, while binaries in `/usr/bin` and libraries in `/usr/lib64` are architecture-specific.

Examples:

```bash
command -v dnf
ls -l /usr/share/man/man1/
```

For exam purposes, remember that `/usr` is not where you place local custom software by default. It is primarily for software delivered by the operating system or installed in a standard, shareable layout.

**`/usr/local` versus `/opt`**

These two directories are often confused, but they serve different administrative patterns.

- **`/usr/local`** is for software installed locally on the system, often from source or from custom packaging that is not managed by RPM.
- **`/opt`** is commonly used for third-party, self-contained application bundles, especially vendor software that installs into its own subdirectory.

A useful rule of thumb:

- Choose `/usr/local` when you want locally built tools to blend into the normal system PATH and hierarchy.
- Choose `/opt` when a vendor ships a standalone application suite that should remain isolated under its own directory tree.

Examples:

```bash
/usr/local/bin/mybackup
/opt/vendorapp/bin/start.sh
```

This distinction helps keep custom software separate from OS-managed files and reduces the risk of package conflicts.

**`/var`: variable and persistent runtime data**

`/var` stores data that changes frequently during normal operation. Unlike `/usr`, which is mostly static, `/var` grows and changes as services run.

Important subdirectories include:

- **`/var/log`** for logs
- **`/var/lib`** for persistent application state
- **`/var/cache`** for rebuildable cache data
- **`/var/spool`** for queued work
- **`/var/tmp`** for temporary files that may survive reboots depending on system policy

A service’s persistent state often lives under `/var/lib/<service>`. Examples include:

- `/var/lib/pgsql` for PostgreSQL data on some systems
- `/var/lib/containers` for container storage on some setups
- `/var/lib/rpm` for RPM database metadata

This is a key administrative distinction: logs and caches are not the same as durable service state. If you back up a database service, you usually care about its data directory under `/var/lib`, not just its logs.

Example log inspection:

```bash
sudo tail -n 50 /var/log/messages
sudo journalctl -u sshd
```

**`/run`, `/tmp`, `/var/tmp`, and `/dev/shm`: temporary and runtime storage**

These locations are all temporary in different ways, and exam questions often test the differences.

- **`/run`** holds volatile runtime data created after boot, such as PID files and sockets.
- **`/tmp`** is for short-lived temporary files; cleanup behavior depends on system configuration and policies.
- **`/var/tmp`** is also temporary, but files are intended to last longer than those in `/tmp` and may persist across reboots depending on policy.
- **`/dev/shm`** is a shared-memory filesystem backed by RAM, used for fast in-memory temporary data and interprocess communication.

Examples:

```bash
ls -ld /run /tmp /var/tmp /dev/shm
```

Typical uses:

- `/run/sshd.pid` for a daemon PID file
- `/tmp` for editor scratch files or installer work files
- `/var/tmp` for large temporary exports that should survive a reboot if policy allows
- `/dev/shm` for applications that need fast shared memory

> Do not store important data in `/tmp` or `/run`. Their contents are temporary by design.

**`/dev`, `/proc`, and `/sys`: virtual and device-related filesystems**

These directories are essential for system administration, but they are not normal disk directories.

- **`/dev`** contains device nodes that represent hardware and pseudo-devices.
- **`/proc`** exposes process and kernel information as a virtual filesystem.
- **`/sys`** exposes kernel objects, devices, and driver attributes through sysfs.

Examples:

```bash
ls -l /dev/sda
cat /proc/cpuinfo
ls /sys/class/net
```

Administrative value:

- Use `/dev` when working with disks, terminals, and other devices.
- Use `/proc` to inspect process IDs, memory, and kernel parameters.
- Use `/sys` to inspect or sometimes tune device and driver settings.

These files are generated by the kernel, not stored as ordinary files on disk.

**`/boot`, `/lib`, and `/lib64`: boot and runtime essentials**

`/boot` contains the files needed to start the system, including the kernel and bootloader configuration. On RHEL, this directory is especially important during kernel updates and boot troubleshooting.

`/lib` and `/lib64` contain essential shared libraries and kernel modules required by core system binaries. On 64-bit systems, many libraries live in `/lib64`.

Examples:

```bash
ls -l /boot
ls -l /lib64
uname -r
```

Administrative caution:

- Avoid unnecessary manual changes in `/boot`.
- If `/boot` fills up, kernel installation can fail.
- Library files are critical dependencies; removing them carelessly can break the system.

**`/home` and `/root`: user data**

`/home` stores regular users’ home directories, while `/root` is the home directory for the root account. These are separate for security and administrative clarity.

Typical contents:

- shell configuration files such as `.bashrc`
- user documents and downloads
- application settings in hidden dotfiles

Examples:

```bash
ls -la /home/alice
ls -la /root
```

Administrative practice:

- Back up `/home` separately from system configuration.
- Treat `/root` as sensitive because it may contain privileged scripts, keys, and admin history.

**`/mnt`, `/media`, and mount points**

Mount points are directories where filesystems become accessible. The directory itself is just a placeholder until a filesystem is mounted there.

- **`/mnt`** is commonly used for temporary manual mounts during troubleshooting or recovery.
- **`/media`** is commonly used for removable media such as USB drives or optical discs.

A typical workflow is:

1. Create the mount point if it does not exist.
2. Identify the device or filesystem by UUID or label.
3. Mount the filesystem.
4. Verify access.
5. Unmount when finished.

Example:

```bash
sudo mkdir -p /mnt/backup
sudo mount /dev/sdb1 /mnt/backup
df -h /mnt/backup
sudo umount /mnt/backup
```

Using UUIDs is usually safer than relying on device names, because device names can change across boots or hardware changes.

```bash
blkid
sudo mount UUID=3f2a1d8c-9b2e-4d3a-8f5a-1c2d3e4f5a6b /mnt/backup
```

**`/srv`: service data**

`/srv` is intended for data served by system services, such as web or FTP content. In practice, many distributions and administrators use other layouts, but the directory remains part of the FHS and may appear in exam questions.

Examples:

- `/srv/www` for web content
- `/srv/ftp` for FTP files

Use `/srv` when you want service content separated from user home directories and from application state in `/var/lib`.

**Administrative decision points**

When you choose a directory, think about ownership, volatility, and package management.

- Use **`/etc`** for configuration that defines system behavior.
- Use **`/var/lib`** for persistent service state.
- Use **`/var/log`** for logs.
- Use **`/usr`** for OS-managed software and shared data.
- Use **`/usr/local`** for locally installed software that should integrate with the system.
- Use **`/opt`** for vendor-provided standalone application trees.
- Use **`/tmp`** or **`/var/tmp`** only for temporary data.
- Use **`/mnt`** for temporary mounts and **`/media`** for removable media.

A quick comparison helps with exam questions:

| Need | Best fit | Why |
|---|---|---|
| System configuration | `/etc` | Host-specific settings |
| Persistent app data | `/var/lib/<service>` | Survives restarts and is service-owned |
| Logs | `/var/log` | Variable diagnostic output |
| Local custom binaries | `/usr/local/bin` | Not managed by RPM |
| Vendor application bundle | `/opt/<vendor>` | Self-contained third-party software |
| Temporary scratch files | `/tmp` | Short-lived workspace |
| Temporary manual mount | `/mnt` | Admin-controlled mount point |

**Practical commands for directory inspection**

Use these commands to identify where files live and what type of filesystem you are dealing with:

```bash
pwd
ls -ld /etc /var /usr /opt /tmp /dev/shm
findmnt
df -hT
stat /etc/fstab
```

Useful lookup commands:

```bash
command -v dnf
type -a python3
rpm -ql bash | head
```

These help you distinguish between shell lookup paths, package-owned files, and filesystem layout.

**Exam-oriented takeaways**

- Know the purpose of the major top-level directories and the difference between static, variable, and temporary data.
- Remember that `/usr` is for installed software and shared data, while `/usr/local` is for local additions and `/opt` is for vendor bundles.
- Understand that `/var/lib/<service>` is where persistent service state usually belongs.
- Recognize that `/proc`, `/sys`, and parts of `/dev` are virtual or kernel-managed, not ordinary disk directories.
- Use UUIDs and mount points carefully when working with filesystems.

**Mental model for troubleshooting**

When something breaks, ask three questions:

- Is the file a configuration, a log, or persistent data?
- Is it supposed to survive reboot?
- Is it managed by the OS package manager or by a local/vendor installation?

That simple classification usually points you to the correct directory and the right recovery approach.
