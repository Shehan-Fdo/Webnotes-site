## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Mount options: what they control**

Mount options define how the kernel presents a filesystem after it is attached to a mount point. They affect security, performance, compatibility, and behavior. In RHCSA work, you use them to make a filesystem safer, more predictable, or more suitable for a workload.

A mount option can be applied:
- at mount time with `mount`
- persistently through `/etc/fstab`
- temporarily with `mount -o remount,...`

Common reasons to use mount options include:
- reducing risk by blocking executable files or device nodes
- improving compatibility with legacy applications
- controlling write behavior and access semantics
- ensuring a filesystem mounts automatically after reboot

**Core mount commands**

You usually inspect current mounts before changing anything. This helps you avoid breaking a running system and lets you verify the active options.

```bash
findmnt
mount
lsblk -f
df -hT
```

Useful inspection commands:
- `findmnt` shows mount points, sources, and options in a readable tree
- `mount` without arguments lists current mounts
- `lsblk -f` shows filesystem type, UUID, and labels
- `df -hT` shows mounted filesystems and their types

To mount a filesystem manually with options:

```bash
mount -o noexec,nodev,nosuid /dev/sdb1 /mnt/data
```

This example applies three common hardening options:
- `noexec` prevents direct execution of binaries from that filesystem
- `nodev` ignores device files on that filesystem
- `nosuid` ignores setuid and setgid bits

**Common mount options**

These are the options you are most likely to use in RHCSA-level administration.

| Option | Meaning | Typical use |
|---|---|---|
| `defaults` | Uses the standard set of mount behaviors | Common baseline in `/etc/fstab` |
| `ro` | Read-only mount | Protect data, mount media safely |
| `rw` | Read-write mount | Normal writable access |
| `noexec` | Disallow execution of binaries | Reduce risk on shared or user-writable mounts |
| `nodev` | Ignore device files | Prevent device-node abuse |
| `nosuid` | Ignore setuid/setgid bits | Reduce privilege escalation risk |
| `relatime` | Update access time only when needed | Common default on modern systems |
| `sync` | Write changes synchronously | Safer but slower for some workloads |
| `async` | Write changes asynchronously | Better performance, default in many cases |
| `auto` | Mount automatically with `mount -a` | Standard for normal filesystems |
| `noauto` | Do not mount automatically | Useful for removable or optional storage |
| `user` | Allow a non-root user to mount | Controlled desktop or lab use |
| `users` | Allow any user to mount and unmount | Less restrictive than `user` |
| `exec` | Allow execution of binaries | Default behavior on many filesystems |
| `remount` | Change options on an already mounted filesystem | Safe option updates without reboot |

> Be careful with `ro` and `noauto` on critical filesystems. A wrong entry in `/etc/fstab` can prevent a system from booting normally or from mounting required data.

**Temporary mounts versus persistent mounts**

A temporary mount exists only until reboot or unmount. A persistent mount survives reboot because it is recorded in `/etc/fstab`.

- Use a temporary mount when you are testing options or troubleshooting
- Use a persistent mount when the system must always mount the filesystem the same way
- Prefer testing with `mount` or `mount -o remount` before editing `/etc/fstab`

A practical workflow is:
1. Identify the device and filesystem with `lsblk -f`
2. Test the mount manually with the desired options
3. Verify the behavior
4. Add the entry to `/etc/fstab`
5. Validate with `mount -a`
6. Reboot or remount to confirm persistence

**The `/etc/fstab` file**

`/etc/fstab` is the filesystem table used at boot and by `mount -a`. Each line describes one filesystem and how the system should mount it.

Typical fields:
1. Device or UUID
2. Mount point
3. Filesystem type
4. Mount options
5. Dump field
6. Fsck field

Example entry:

```fstab
UUID=3f2c1d8a-7b4e-4d1a-9f1a-2c7a9f0b1c22  /data  xfs  defaults,noatime  0  0
```

Field details:
- **UUID** is preferred over device names because it is stable across reboots and device renaming
- **Mount point** must already exist, such as `/data`
- **Filesystem type** might be `xfs`, `ext4`, `vfat`, or another supported type
- **Mount options** are comma-separated with no spaces
- **Dump** is usually `0` on modern systems
- **Fsck** controls filesystem check order; for `xfs`, this is typically `0`

Common best practices:
- use `UUID=` or `LABEL=` instead of `/dev/sdX` names
- create the mount point before adding the entry
- test with `mount -a` after editing
- keep a root shell open when changing critical mounts so you can recover if needed

**Finding UUIDs and labels**

Stable identifiers reduce boot-time surprises.

```bash
blkid
lsblk -f
```

Example output pattern:
- `UUID` identifies a specific filesystem
- `LABEL` is a human-friendly name you can assign and use in `/etc/fstab`

Example `/etc/fstab` line using a label:

```fstab
LABEL=DATA  /data  xfs  defaults  0  0
```

Use labels when they improve readability, but remember that UUIDs are often more precise in larger environments.

**Testing and validating `/etc/fstab`**

After editing `/etc/fstab`, validate the file before rebooting.

```bash
mount -a
findmnt --verify
```

What these checks do:
- `mount -a` attempts to mount all filesystems marked for automatic mounting
- `findmnt --verify` checks the syntax and consistency of mount table entries

If `mount -a` fails, inspect:
- spelling of the filesystem type
- mount point existence
- option syntax
- UUID or label correctness
- whether the filesystem is already mounted differently

A safe validation pattern is:
```bash
mkdir -p /data
mount -a
findmnt /data
```

If the mount appears in `findmnt`, the entry is working.

**Remounting to change options**

You can change some mount options without unmounting by using `remount`. This is useful for live systems where downtime is undesirable.

```bash
mount -o remount,ro /data
mount -o remount,rw /data
```

Typical uses:
- switch a filesystem to read-only for maintenance
- apply a new option after testing
- adjust behavior without rebooting

Not every option can be changed with a remount. Some options require unmounting and mounting again. If a remount fails, check whether the filesystem type supports the requested change.

**Security-focused options**

Mount options are often used to reduce attack surface.

- `nosuid` blocks setuid and setgid program behavior
- `nodev` prevents device files from being interpreted as devices
- `noexec` prevents direct execution of binaries from that mount
- `ro` limits write access and can protect static content or recovery media

A common hardening example for a shared temporary area:

```fstab
UUID=8a7d2c1f-1111-2222-3333-444455556666  /shared  xfs  defaults,nodev,nosuid,noexec  0  0
```

Use these options carefully:
- `noexec` can break scripts or applications that expect to run from that path
- `nosuid` can affect legitimate administrative tools
- `nodev` is usually safe on data mounts but not on device-related filesystems

**Performance-related options**

Some options affect performance and metadata behavior.

- `noatime` disables access-time updates, which can reduce write activity
- `relatime` updates access time only when needed and is a common default
- `sync` forces immediate writes and can reduce performance
- `async` allows buffered writes and is usually faster

Example for a workload where access-time updates are unnecessary:

```fstab
UUID=3f2c1d8a-7b4e-4d1a-9f1a-2c7a9f0b1c22  /data  xfs  defaults,noatime  0  0
```

Use `noatime` when:
- applications do not depend on access times
- you want to reduce metadata writes
- the filesystem is read-heavy

Avoid changing performance options blindly. Some applications or backup tools may rely on access-time semantics.

**Mounting by UUID, label, or device path**

There are three common ways to identify a filesystem in `/etc/fstab`.

| Method | Example | Strengths | Weaknesses |
|---|---|---|---|
| Device path | `/dev/sdb1` | Simple to read | Can change across boots |
| UUID | `UUID=...` | Stable and reliable | Less human-friendly |
| Label | `LABEL=DATA` | Easy to recognize | Labels must be managed carefully |

In most RHCSA scenarios, use `UUID=` unless the task specifically asks for a label or device path.

**Troubleshooting mount problems**

When a mount fails, work from the bottom up.

1. Confirm the device exists with `lsblk`
2. Confirm the filesystem type with `blkid` or `lsblk -f`
3. Confirm the mount point exists
4. Check `/etc/fstab` syntax
5. Review `dmesg` or `journalctl -b` for kernel messages
6. Test the mount manually

Useful commands:

```bash
journalctl -b
dmesg | tail -n 20
mount -v -a
```

Common causes of failure:
- wrong filesystem type in `/etc/fstab`
- typo in UUID or label
- missing mount directory
- invalid option name
- filesystem corruption
- trying to mount a filesystem that is already mounted elsewhere

If a system fails to boot because of a bad `/etc/fstab` entry, you may need to boot into rescue mode or emergency mode and correct the file.

**Example workflow for persistent data mount**

Suppose you add a new disk and want it mounted at `/data` with safer defaults.

1. Identify the new partition:
```bash
lsblk -f
```

2. Create the mount point:
```bash
mkdir -p /data
```

3. Test a temporary mount:
```bash
mount /dev/sdb1 /data
```

4. Verify the filesystem works:
```bash
df -hT /data
findmnt /data
```

5. Get the UUID:
```bash
blkid /dev/sdb1
```

6. Add the persistent entry:
```fstab
UUID=3f2c1d8a-7b4e-4d1a-9f1a-2c7a9f0b1c22  /data  xfs  defaults,noatime  0  0
```

7. Validate the configuration:
```bash
mount -a
findmnt /data
```

This workflow reduces the chance of boot-time errors and confirms the mount behaves as expected before you rely on it.

**Special cases and practical cautions**

Some filesystems and mount points have special behavior:
- `proc`, `sysfs`, `tmpfs`, and `devtmpfs` are virtual filesystems and are usually managed by the system
- removable media often use `noauto` so they do not mount at boot
- network filesystems may require network availability before mounting
- XFS filesystems cannot be shrunk, so plan storage layout carefully

> Do not edit `/etc/fstab` casually on a production system without a recovery plan. A single bad line can delay boot or drop you into emergency mode.

**Quick reference**

- Use `mount` to attach a filesystem now
- Use `/etc/fstab` to make the mount persistent
- Use `UUID=` for stable identification
- Use `mount -o remount,...` to change some options live
- Use `mount -a` and `findmnt --verify` to validate changes
- Use `noexec`, `nodev`, and `nosuid` to harden shared mounts
- Use `noatime` when access-time updates are unnecessary

**Exam focus**

For RHCSA, you should be able to:
- create a mount point
- identify a filesystem by UUID or label
- mount a filesystem manually with options
- edit `/etc/fstab` correctly
- validate persistent mounts
- troubleshoot a failed mount entry
- remount a filesystem with new options when possible
