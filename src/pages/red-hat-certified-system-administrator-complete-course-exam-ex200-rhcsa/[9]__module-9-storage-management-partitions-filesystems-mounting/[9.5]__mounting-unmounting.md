## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Mounting fundamentals**

A mount makes a filesystem available at a directory called a mount point. Until you mount it, the data on the block device, network share, or pseudo-filesystem is not visible through the normal directory tree.

On RHEL, you typically work with:
- **Block devices** such as `/dev/sdb1`
- **UUIDs** and **labels** for stable identification
- **Pseudo-filesystems** such as `proc` and `sysfs`
- **Network filesystems** such as NFS and SMB/CIFS
- **Bind mounts** for reusing an existing directory tree at another path

A mount point must be an existing directory. If the directory contains files before the mount, those files are hidden while the filesystem is mounted there.

**Core commands**

Use `mount` to attach filesystems and `umount` to detach them. Use `findmnt` and `mountpoint` to verify the result.

```bash
mount
findmnt
mountpoint /data
umount /data
```

Useful verification commands:
- `findmnt` shows the current mount tree in a readable form.
- `mountpoint /path` tells you whether a directory is a mount point.
- `df -h` shows mounted filesystems and usage.
- `lsblk -f` shows block devices, filesystem types, UUIDs, and mount points.

**Temporary mounts**

A manual mount is temporary unless you add it to `/etc/fstab`. Temporary mounts disappear after reboot.

Typical syntax:
```bash
mount /dev/sdb1 /data
mount -t xfs /dev/sdb1 /data
mount -o noatime /dev/sdb1 /data
```

Key points:
- `mount` can infer the filesystem type in many cases, but specifying `-t` is often clearer.
- `-o` passes mount options such as `noatime`, `ro`, `rw`, `nodev`, `nosuid`, and `noexec`.
- You mount a device to a directory; `mount` does not accept `UUID=...` directly as a device argument.
- If you want to mount by UUID manually, use the device path under `/dev/disk/by-uuid/`, for example:
  ```bash
  mount /dev/disk/by-uuid/3f2a1c2d-9b7e-4f1a-8c2d-1d2e3f4a5b6c /data
  ```

> `mount -a` processes entries from `/etc/fstab`. It mounts eligible entries that are not already mounted; it does not mount arbitrary filesystems outside `fstab`.

**Persistent mounts with `/etc/fstab`**

The `/etc/fstab` file defines filesystems that should be mounted automatically at boot or when you run `mount -a`.

A common entry format is:
```fstab
<device>  <mountpoint>  <fstype>  <options>  <dump>  <pass>
```

Example using UUID:
```fstab
UUID=3f2a1c2d-9b7e-4f1a-8c2d-1d2e3f4a5b6c  /data  xfs  defaults  0  0
```

Example using a label:
```fstab
LABEL=DATA  /data  xfs  defaults,noatime  0  0
```

Example using a network filesystem:
```fstab
server:/exports/projects  /mnt/projects  nfs  defaults,_netdev  0  0
```

Example using SMB/CIFS:
```fstab
//fileserver/share  /mnt/share  cifs  credentials=/root/.smbcred,iocharset=utf8,_netdev  0  0
```

Field meanings:
- **device**: device path, UUID, label, or network source
- **mountpoint**: directory where the filesystem appears
- **fstype**: filesystem type such as `xfs`, `ext4`, `nfs`, or `cifs`
- **options**: comma-separated mount options
- **dump**: legacy backup flag, usually `0`
- **pass**: `fsck` order at boot; use `1` for root, `2` for other local filesystems, `0` to skip

Practical guidance:
- Prefer `UUID=` or `LABEL=` for local disks because device names like `/dev/sdb1` can change.
- Use `_netdev` for network filesystems so systemd treats them as network-dependent.
- Use `nofail` when a noncritical mount should not block boot.
- Use `x-systemd.automount` when you want on-demand mounting through systemd.

**Editing and validating `/etc/fstab`**

Always validate before rebooting. A bad `fstab` entry can prevent normal boot or drop you into emergency mode.

A safe workflow:
1. Create the mount point:
   ```bash
   mkdir -p /data
   ```
2. Identify the filesystem:
   ```bash
   lsblk -f
   blkid
   ```
3. Add the entry to `/etc/fstab`.
4. Test the file without rebooting:
   ```bash
   mount -a
   ```
5. Verify the mount:
   ```bash
   findmnt /data
   mountpoint /data
   df -h /data
   ```

Important validation rule:
- `mount -a` checks and mounts entries from `/etc/fstab`.
- To test only one new entry, you can temporarily comment out unrelated lines or use a controlled test environment.
- If you need to verify syntax without mounting everything, inspect the file carefully and test one entry at a time in a lab.

> If `mount -a` reports an error, fix the line before rebooting. A typo in the device, mount point, filesystem type, or options can break boot-time mounting.

**Filesystem types and mount options**

Common filesystem types in RHCSA tasks:
- `xfs` for the default RHEL local filesystem
- `ext4` for compatibility scenarios
- `nfs` for network shares
- `cifs` for SMB shares
- `tmpfs` for memory-backed temporary storage
- `proc` and `sysfs` for kernel interfaces

Common options:
- `defaults` uses the standard set of mount options
- `ro` mounts read-only
- `rw` mounts read-write
- `noatime` reduces metadata writes by not updating access times
- `nodev` prevents device files from being interpreted
- `nosuid` ignores setuid and setgid bits
- `noexec` prevents execution of binaries from that mount
- `relatime` is common on modern systems and balances performance with access-time updates

Example:
```bash
mount -o ro,noexec /dev/sdb1 /mnt/recovery
```

Use restrictive options when you want to reduce risk on shared, temporary, or untrusted mounts.

**Bind mounts**

A bind mount makes an existing directory tree appear at another path. It is not a separate copy of the data.

Example:
```bash
mount --bind /srv/projects /mnt/projects
```

You can also make a bind mount persistent with `/etc/fstab`:
```fstab
/srv/projects  /mnt/projects  none  bind  0  0
```

Common use cases:
- Exposing a directory inside a chroot or container-like environment
- Reorganizing access paths without duplicating data
- Providing alternate paths for applications or services

If you need recursive bind behavior for nested mounts, use:
```bash
mount --rbind /srv/projects /mnt/projects
```

**Network filesystems**

Network mounts are common in admin tasks and can appear in RHCSA-style scenarios.

NFS example:
```bash
mkdir -p /mnt/nfs
mount -t nfs server:/exports/data /mnt/nfs
```

SMB/CIFS example:
```bash
mkdir -p /mnt/share
mount -t cifs //fileserver/share /mnt/share -o credentials=/root/.smbcred,iocharset=utf8
```

Practical notes:
- Ensure the network is available before mounting.
- Use `_netdev` in `fstab` for network-dependent mounts.
- Store SMB credentials in a protected file rather than on the command line.
- Confirm the server export or share name is correct before troubleshooting the client.

**Pseudo-filesystems**

Some mounts are managed by the system and are not usually added to `/etc/fstab` manually on modern RHEL systems.

Examples:
```bash
mount -t proc proc /proc
mount -t sysfs sysfs /sys
mount -t tmpfs tmpfs /run/tmp
```

Use these examples to understand the concept, but do not treat them as normal persistent admin mounts for routine RHCSA work. `proc` and `sysfs` are typically mounted automatically during boot.

**Unmounting safely**

Use `umount` to detach a filesystem.

Examples:
```bash
umount /data
umount /dev/sdb1
```

Rules to remember:
- You can unmount by mount point or by device.
- The filesystem must not be busy.
- If a process is using the mount, `umount` fails with a busy error.

Troubleshooting a busy mount:
```bash
lsof +f -- /data
fuser -vm /data
```

If needed, stop the service or exit the shell using the mount, then unmount again.

> Avoid force-unmounting unless you understand the risk. Forcing can cause data loss or filesystem corruption.

**Troubleshooting boot failures from bad `fstab` entries**

A broken `/etc/fstab` entry can stop the system from reaching the normal target. In exam scenarios, you may need to recover, fix the file, and reboot.

Common recovery approaches:
- Boot into `rescue.target` or `emergency.target` from the GRUB menu by editing the kernel command line.
- Use `rd.break` for early initramfs-level recovery when the root filesystem or boot process is affected.
- Remount the root filesystem read-write if needed:
  ```bash
  mount -o remount,rw /
  ```
- Edit `/etc/fstab` with `vi` or `nano`, then test with `mount -a`.

Typical recovery flow:
1. Boot to a rescue shell.
2. Mount the root filesystem if it is not already available.
3. Edit `/etc/fstab`.
4. Comment out or correct the bad line.
5. Run `mount -a` to validate.
6. Reboot normally.

Example of a safer `fstab` line for a noncritical mount:
```fstab
UUID=3f2a1c2d-9b7e-4f1a-8c2d-1d2e3f4a5b6c  /data  xfs  defaults,nofail  0  0
```

This helps prevent a nonessential filesystem from blocking boot if the device is missing.

**Exam-oriented workflow**

When you are given a storage task, work in this order:
1. Identify the device or network source.
2. Confirm the filesystem type.
3. Create the mount point.
4. Mount it manually and verify access.
5. Add the correct `/etc/fstab` entry.
6. Test with `mount -a`.
7. Reboot only after validation.

A practical checklist:
- Use `lsblk -f` or `blkid` to find UUIDs.
- Use `findmnt` to confirm the active mount.
- Use `mountpoint` to verify the directory is mounted.
- Use `mount -a` to test persistent configuration.
- Use `umount` before changing filesystem layout or replacing a device.

**Quick reference**

| Task | Command |
|---|---|
| Show mounted filesystems | `findmnt` |
| Check whether a directory is a mount point | `mountpoint /path` |
| Mount a filesystem | `mount /dev/sdb1 /data` |
| Mount with options | `mount -o noatime /dev/sdb1 /data` |
| Unmount a filesystem | `umount /data` |
| Test `/etc/fstab` | `mount -a` |
| Show UUIDs and filesystem types | `lsblk -f` |
| Show device metadata | `blkid` |

**Common mistakes**

- Using the wrong filesystem type in `/etc/fstab`
- Forgetting to create the mount point directory
- Typing the wrong UUID or label
- Using `/dev/sdX` names in persistent configs when UUIDs are safer
- Leaving a noncritical mount without `nofail`
- Trying to mount a device that is already mounted elsewhere
- Forgetting that `mount UUID=...` is not valid syntax
- Assuming `mount -a` tests only one line instead of processing all eligible `fstab` entries

**Practical examples**

Mount a local XFS filesystem:
```bash
mkdir -p /data
mount /dev/sdb1 /data
```

Make it persistent:
```bash
blkid /dev/sdb1
vi /etc/fstab
```

Add:
```fstab
UUID=3f2a1c2d-9b7e-4f1a-8c2d-1d2e3f4a5b6c  /data  xfs  defaults  0  0
```

Validate:
```bash
mount -a
findmnt /data
```

Create a bind mount:
```bash
mkdir -p /srv/projects /mnt/projects
mount --bind /srv/projects /mnt/projects
```

Mount an NFS share:
```bash
mkdir -p /mnt/nfs
mount -t nfs server:/exports/data /mnt/nfs
```

Unmount after use:
```bash
umount /mnt/projects
umount /mnt/nfs
```

**Summary**

Mounting connects storage to the directory tree, and unmounting cleanly detaches it. For RHCSA, focus on three habits: identify devices reliably with UUIDs or labels, validate `/etc/fstab` before rebooting, and verify mounts with `findmnt` or `mountpoint`. That workflow prevents most storage-related exam failures.
