## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Filesystem choice in RHCSA tasks**

On Red Hat Enterprise Linux, you commonly work with **XFS** and **ext4**. The right choice depends on the task, the mount point, and whether you need to grow or shrink the filesystem later. For exam work, the most important habit is to verify the current device, mount state, and filesystem type before you format anything.

A practical workflow is:

1. Identify the target block device or logical volume.
2. Check whether it is already mounted and what filesystem it uses.
3. Choose the filesystem based on the task requirements.
4. Create the filesystem only after confirming the device is correct.

Useful checks:

```bash
lsblk -f
blkid
findmnt
df -Th
```

- `lsblk -f` shows filesystem type, UUID, and mount points.
- `blkid` helps confirm device identity before formatting.
- `findmnt` shows the active mount relationship.
- `df -Th` shows mounted filesystems and their types.

> Never format a device until you confirm it is the intended target. A mistaken `mkfs` command destroys existing data immediately.

**XFS vs ext4 at a glance**

| Feature | XFS | ext4 |
|---|---|---|
| Typical RHCSA use | Large filesystems, general-purpose server storage | General-purpose storage, compatibility, shrinkable volumes |
| Growth | Supported online | Supported online |
| Shrink | Not supported | Supported offline |
| Repair tool | `xfs_repair` | `e2fsck` / `fsck.ext4` |
| Creation command | `mkfs.xfs` | `mkfs.ext4` |
| Best fit when | You expect to expand storage later | You may need to reduce size later |

**Choosing a filesystem for common mount points**

RHCSA-style scenarios often imply a sensible filesystem choice rather than explicitly stating it.

- `/home`
  - Use **XFS** when you expect the volume to grow and you want a modern default-style server filesystem.
  - Use **ext4** when you need the option to shrink the filesystem later.
- `/var`
  - Use **XFS** for log-heavy or general server storage when growth is the likely change.
  - Use **ext4** if the storage layout may need later reduction during lab work.
- `/srv`
  - Use **XFS** for service data that may expand over time.
  - Use **ext4** if you anticipate resizing downward in a lab or migration scenario.

A simple decision flow:

```text
Need to shrink later?
  yes -> ext4
  no  -> XFS
Need a common RHCSA default for new server storage?
  often -> XFS
Need compatibility with shrink workflows?
  ext4
```

**Creating filesystems**

Use the correct `mkfs` command for the chosen filesystem. The device can be a partition, a logical volume, or another block device.

```bash
mkfs.xfs /dev/rhel/data
mkfs.ext4 /dev/rhel/backup
```

If you want to label the filesystem for easier identification, set the label at creation time or afterward.

```bash
mkfs.xfs -L DATA /dev/rhel/data
mkfs.ext4 -L BACKUP /dev/rhel/backup
```

After creation, verify the result:

```bash
blkid /dev/rhel/data
lsblk -f /dev/rhel/data
```

- Labels help you identify volumes in `lsblk`, `blkid`, and `/etc/fstab`.
- UUIDs are usually more reliable than device names for persistent mounts.

**Mounting and persistence**

A filesystem is useful only after you mount it. For exam tasks, you often need to create the filesystem, mount it, and make the mount persistent.

```bash
mkdir -p /data
mount /dev/rhel/data /data
```

To make the mount persistent, add an `/etc/fstab` entry:

```fstab
UUID=1234-ABCD  /data  xfs   defaults  0 0
```

Then test the entry safely:

```bash
mount -a
findmnt /data
```

- Use `mount -a` to catch syntax errors before rebooting.
- Use `findmnt` to confirm the mount is active and mapped to the expected device.
- Prefer UUID-based entries unless the task specifically requires a label or device path.

**XFS growth workflow**

XFS supports online growth, which makes it a strong choice for expanding storage without downtime. The key exam detail is that you grow the underlying block device or logical volume first, then grow the mounted filesystem.

Typical workflow for an LVM-backed XFS filesystem:

```bash
lvextend -L +10G /dev/rhel/data
xfs_growfs /data
```

Important points:

- The logical volume must already be mounted at `/data`.
- `xfs_growfs` operates on the **mounted mount point**, not on the block device path.
- The filesystem grows to use the newly available space after the LV expansion.

A more complete example:

```bash
lvextend -r -L +10G /dev/rhel/data
```

- `-r` asks LVM to resize the filesystem after extending the volume.
- This is convenient, but you should still understand the manual two-step workflow for exam troubleshooting.

If you need to confirm the result:

```bash
df -Th /data
xfs_info /data
```

- `xfs_info` requires the mount point of a mounted XFS filesystem.
- It shows geometry and allocation details for the active filesystem.

**ext4 growth workflow**

ext4 also supports online growth. The usual pattern is similar to XFS: expand the block device or logical volume first, then grow the mounted filesystem.

```bash
lvextend -L +5G /dev/rhel/backup
resize2fs /dev/rhel/backup
```

You can also use the combined LVM option:

```bash
lvextend -r -L +5G /dev/rhel/backup
```

Practical notes:

- The filesystem must already be mounted if you are growing it online.
- `resize2fs` can grow ext4 while mounted in many cases.
- Verify the result with `df -Th` and `lsblk -f`.

**ext4 shrink workflow**

ext4 is the practical choice when you may need to reduce filesystem size later. Shrinking is more delicate than growing and requires the correct order.

Safe general workflow:

1. Unmount the filesystem.
2. Check and repair it if needed.
3. Shrink the filesystem first.
4. Reduce the underlying logical volume or partition.
5. Mount it again and verify.

Example:

```bash
umount /dev/rhel/backup
e2fsck -f /dev/rhel/backup
resize2fs /dev/rhel/backup 8G
lvreduce -L 8G /dev/rhel/backup
mount /dev/rhel/backup /backup
```

Critical details:

- Shrink the **filesystem before the block device**.
- Do not reduce the LV or partition first unless the filesystem has already been safely reduced to fit.
- `e2fsck -f` is a common prerequisite before shrinking ext4.

> Shrinking in the wrong order can destroy data. For RHCSA tasks, always reduce the filesystem first, then reduce the logical volume or partition.

**Why XFS cannot shrink**

XFS is designed for high performance and online growth, but it does not support shrinking. If you need a smaller XFS filesystem, the usual solution is to back up the data, recreate the filesystem at the smaller size, and restore the data.

Typical approach:

1. Back up the data.
2. Unmount the filesystem.
3. Recreate the filesystem at the new size.
4. Restore the data.
5. Remount and verify.

This is why XFS is a strong default for expandable server storage, but not for workflows that require frequent size reduction.

**Repair and recovery**

Filesystem repair tools differ by type.

- **XFS**
  - Use `xfs_repair` for offline repair.
  - Run it on an unmounted XFS filesystem.
  - In severe log corruption cases, `xfs_repair -L` may be required, but it can discard the log and lose recent metadata updates.
- **ext4**
  - Use `e2fsck` or `fsck.ext4`.
  - Run it on an unmounted filesystem.
  - It checks and repairs ext-family metadata.

Examples:

```bash
umount /dev/rhel/data
xfs_repair /dev/rhel/data
```

```bash
umount /dev/rhel/backup
e2fsck -f /dev/rhel/backup
```

Operational guidance:

- Do not run repair tools on a mounted filesystem unless the tool explicitly supports that mode.
- If a filesystem is in use, unmount it first or boot into rescue mode if necessary.
- For XFS, `xfs_repair` is the standard tool; `fsck` is not the normal repair path.

**RHCSA troubleshooting habits**

A lot of storage mistakes come from skipping verification. Before formatting, resizing, or repairing, confirm the current state.

```bash
lsblk -f
mount | grep -E '/data|/backup'
df -Th
blkid /dev/rhel/data
```

Useful checks during troubleshooting:

- Confirm the device is the one you intended.
- Confirm the filesystem type matches the command you plan to use.
- Confirm the mount point is active before using `xfs_growfs`.
- Confirm the filesystem is unmounted before shrinking or repairing.

Common exam-style mistakes:

- Running `xfs_growfs` against a block device instead of the mounted path.
- Trying to shrink XFS instead of recreating it.
- Reducing an ext4 LV before shrinking the filesystem.
- Formatting the wrong partition because you skipped `lsblk` or `blkid`.
- Forgetting to mount the filesystem before testing `/etc/fstab`.

**Practical RHCSA scenarios**

- `/home` needs more space after user growth:
  - Choose XFS if you only need expansion.
  - Extend the LV, then run `xfs_growfs /home`.
- `/var` was over-allocated and must be reduced:
  - ext4 is the better fit if shrinkability matters.
  - Unmount, check with `e2fsck`, shrink with `resize2fs`, then reduce the LV.
- `/srv` will host application data that may grow over time:
  - XFS is a strong choice because online growth is straightforward.
- A lab task asks you to create a filesystem and make it persistent:
  - Create the filesystem with `mkfs.xfs` or `mkfs.ext4`.
  - Mount it.
  - Add the UUID to `/etc/fstab`.
  - Test with `mount -a`.

**Quick command reference**

```bash
# Inspect
lsblk -f
blkid
findmnt
df -Th

# Create
mkfs.xfs /dev/rhel/data
mkfs.ext4 /dev/rhel/backup

# Mount
mount /dev/rhel/data /data
mount /dev/rhel/backup /backup

# Grow XFS
lvextend -L +10G /dev/rhel/data
xfs_growfs /data

# Grow ext4
lvextend -L +5G /dev/rhel/backup
resize2fs /dev/rhel/backup

# Shrink ext4
umount /backup
e2fsck -f /dev/rhel/backup
resize2fs /dev/rhel/backup 8G
lvreduce -L 8G /dev/rhel/backup

# Repair
xfs_repair /dev/rhel/data
e2fsck -f /dev/rhel/backup
```

**Exam strategy**

For RHCSA, the safest approach is to think in terms of the full storage lifecycle:

- Verify the device.
- Choose the filesystem based on the task.
- Create it with the correct `mkfs` command.
- Mount it and make it persistent.
- Grow XFS or ext4 only after expanding the underlying storage.
- Shrink ext4 only after shrinking the filesystem first.
- Repair only after unmounting and confirming the filesystem type.

That workflow helps you avoid the most common storage errors and makes your commands easier to reason about under exam pressure.
