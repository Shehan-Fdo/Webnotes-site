**Persistent mounts and why they matter**

Linux can mount filesystems manually with `mount`, but those mounts disappear after reboot unless you define them persistently. The standard mechanism is `/etc/fstab` (file system table), which tells the system what to mount, where to mount it, and with which options during boot or when you run `mount -a`.

Using UUIDs (Universally Unique Identifiers) is the preferred approach for most local filesystems because device names like `/dev/sdb1` can change across boots, hardware changes, or controller reordering. A UUID stays tied to the filesystem itself, so it is more stable than a device node.

- `/etc/fstab` controls automatic and manual mount behavior.
- **UUIDs** reduce the risk of mounting the wrong disk after reboot.
- `mount -a` is a key validation command before you rely on a new entry.

**Core `/etc/fstab` format**

Each line in `/etc/fstab` describes one mount entry. The fields are whitespace-separated and follow a fixed order:

```text
<fs_spec>  <fs_file>  <fs_vfstype>  <fs_mntops>  <fs_freq>  <fs_passno>
```

- `<fs_spec>` is the device, UUID, label, or remote source.
- `<fs_file>` is the mount point, such as `/data`.
- `<fs_vfstype>` is the filesystem type, such as `xfs`, `ext4`, `vfat`, or `swap`.
- `<fs_mntops>` is a comma-separated list of mount options.
- `<fs_freq>` is usually `0` for modern systems.
- `<fs_passno>` controls filesystem check order at boot; `1` is typically for `/`, `2` for other local filesystems, and `0` disables dump/fsck ordering.

A typical entry using a UUID looks like this:

```fstab
UUID=3f2a1d7e-8b4c-4d1f-9f2a-2d5d8f7c9a11  /data  xfs  defaults  0  0
```

A swap entry may look like this:

```fstab
UUID=9c1b2a3d-4e5f-6789-abcd-ef0123456789  swap  swap  defaults  0  0
```

**Finding the correct UUID**

You usually identify the filesystem UUID with `blkid` or `lsblk`. `blkid` is the most direct tool for this task.

```bash
sudo blkid
```

Example output:

```text
/dev/sda1: UUID="A1B2-C3D4" TYPE="vfat" PARTUUID="12345678-01"
/dev/sda2: UUID="3f2a1d7e-8b4c-4d1f-9f2a-2d5d8f7c9a11" TYPE="xfs"
/dev/sdb1: UUID="9c1b2a3d-4e5f-6789-abcd-ef0123456789" TYPE="swap"
```

You can also inspect block devices in a tree view:

```bash
lsblk -f
```

This shows filesystem type, UUID, and mount points in one place, which is useful when you are matching a partition to a planned mount point.

- Use `blkid` when you need exact identifiers.
- Use `lsblk -f` when you want a quick visual map of disks, partitions, and mounts.
- Prefer UUIDs over `/dev/sdX` names for persistent local mounts.

**Creating a mount point and testing manually**

Before editing `/etc/fstab`, create the mount point and test the filesystem manually. This helps you confirm the filesystem is valid and the path is correct.

```bash
sudo mkdir -p /data
sudo mount /dev/sdb1 /data
df -hT /data
```

If the mount works, unmount it before adding the persistent entry:

```bash
sudo umount /data
```

This workflow matters because a bad `/etc/fstab` entry can prevent a system from booting normally or can drop you into emergency mode if the mount is required early in boot.

> Be careful with `/etc/fstab` edits. A syntax error, wrong UUID, or invalid mount option can cause boot problems.

**Editing `/etc/fstab` safely**

Use a text editor such as `vi` or `vim` and make one change at a time. Keep a backup copy if you are working on a production system.

```bash
sudo cp /etc/fstab /etc/fstab.bak
sudo vi /etc/fstab
```

A common entry for an XFS data volume is:

```fstab
UUID=3f2a1d7e-8b4c-4d1f-9f2a-2d5d8f7c9a11  /data  xfs  defaults  0  0
```

After saving, validate the file without rebooting:

```bash
sudo mount -a
```

If the command returns no output, the entries it processed were accepted. If there is a problem, it prints an error describing the mount failure.

- `mount -a` tests all entries that are not already mounted.
- Errors often point to a typo in the UUID, mount point, filesystem type, or options.
- Always verify with `findmnt` or `df` after a successful test.

```bash
findmnt /data
df -hT /data
```

**Mount options you should know**

Mount options in `/etc/fstab` control behavior and security. The `defaults` option is a shorthand for a standard set of options, but you often need to add more specific controls.

Common options include:

- `ro` mounts read-only.
- `rw` mounts read-write.
- `noexec` prevents execution of binaries from the mount.
- `nosuid` ignores setuid and setgid bits.
- `nodev` ignores device files on the mount.
- `noatime` reduces metadata writes by not updating access times on reads.
- `x-systemd.automount` delays mounting until first access on systemd-based systems.
- `nofail` allows boot to continue even if the mount is unavailable.

Example:

```fstab
UUID=3f2a1d7e-8b4c-4d1f-9f2a-2d5d8f7c9a11  /data  xfs  defaults,noatime,nosuid  0  0
```

Use restrictive options when the mount stores shared data, user-uploaded content, or anything that should not execute programs. Use `nofail` for optional storage that should not block boot if the device is missing.

**Why UUID-based mounting is preferred**

Device names such as `/dev/sdb1` are assigned dynamically by the kernel and can change if disks are added, removed, or detected in a different order. UUIDs are written into the filesystem metadata, so they remain stable unless you reformat the filesystem.

Example comparison:

| Method | Example | Stability | Typical use |
|---|---|---:|---|
| Device node | `/dev/sdb1` | Lower | Temporary testing |
| UUID | `UUID=...` | High | Persistent local mounts |
| Label | `LABEL=data` | Medium | Human-friendly environments |

UUIDs are especially useful when:

- You have multiple similar disks.
- You boot from removable or hot-plug storage.
- You want predictable mounts after hardware changes.
- You are configuring systems that must recover cleanly after reboot.

**Troubleshooting common `/etc/fstab` problems**

When a mount fails, work from the error message and verify each field in the entry.

1. Check the syntax in `/etc/fstab`.
2. Confirm the UUID with `blkid`.
3. Confirm the mount point exists.
4. Confirm the filesystem type matches the actual filesystem.
5. Test with `mount -a`.
6. Inspect logs if the failure is unclear.

Useful commands:

```bash
sudo blkid
lsblk -f
findmnt
mount -a
journalctl -b
```

Common failure causes:

- Wrong UUID copied from another device.
- Mount point directory does not exist.
- Filesystem type in `/etc/fstab` does not match the actual format.
- Invalid mount option, such as a typo in `defaults`.
- Attempting to mount a filesystem that is already mounted elsewhere.
- Using `xfs` options on an `ext4` filesystem or vice versa.

If the system fails to boot because of a bad entry, you may need to boot into rescue mode or emergency mode, fix `/etc/fstab`, and then run `mount -a` again before rebooting.

**Special cases: swap and nonstandard filesystems**

`/etc/fstab` is not only for regular filesystems. It also handles swap and other mount types.

Swap entry example:

```fstab
UUID=9c1b2a3d-4e5f-6789-abcd-ef0123456789  swap  swap  defaults  0  0
```

For removable media or special filesystems, you may use different options and mount behavior. In RHCSA-style administration, the important part is to match the source, type, and options correctly and to verify the result with the appropriate tools.

- Swap uses `swap` as both the mount point field and filesystem type field.
- Network filesystems use different source syntax, such as `server:/export/path`, but the same `/etc/fstab` structure still applies.
- Temporary mounts can be done with `mount` alone, but persistent mounts require `/etc/fstab`.

**Practical workflow for exam tasks**

When asked to configure persistent storage, follow a repeatable process.

1. Identify the target block device with `lsblk` or `blkid`.
2. Create the mount point with `mkdir -p`.
3. Test the filesystem manually with `mount`.
4. Record the UUID with `blkid`.
5. Add the `/etc/fstab` entry using the UUID.
6. Validate with `mount -a`.
7. Confirm with `findmnt` or `df -hT`.

Example:

```bash
lsblk -f
sudo mkdir -p /srv/projects
sudo mount /dev/sdb1 /srv/projects
sudo blkid /dev/sdb1
sudo vi /etc/fstab
sudo mount -a
findmnt /srv/projects
```

This workflow reduces mistakes because you verify the filesystem before making it persistent.

**Useful verification commands**

These commands help you confirm that the mount is active and correct:

```bash
findmnt /data
df -hT /data
mount | grep ' /data '
cat /etc/fstab
```

`findmnt` is especially useful because it shows the source, target, filesystem type, and options in a readable tree format.

**Key takeaways**

- `/etc/fstab` defines persistent mounts and swap configuration.
- UUIDs are the safest common identifier for local filesystems.
- Always test with `mount -a` before rebooting.
- Use `blkid` and `lsblk -f` to discover identifiers.
- Keep mount options intentional and minimal.
- A bad `fstab` entry can break boot, so validate carefully.
