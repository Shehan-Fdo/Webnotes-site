## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Filesystem creation overview**

`mkfs.xfs` and `mkfs.ext4` create filesystems on block devices, partitions, or logical volumes after you have prepared the storage target. On RHEL-based systems, XFS is the default and most common choice for general-purpose local storage, while ext4 remains widely supported and useful in mixed environments or when you specifically need ext4 behavior.

A filesystem format step writes metadata structures to the device. That means you should treat it as a destructive operation: existing data on the target is overwritten. Before you format anything, confirm the device name with `lsblk`, `blkid`, or `findmnt` and make sure you are targeting the correct disk or partition.

- **XFS** is optimized for large filesystems and high concurrency.
- **ext4** is a mature general-purpose filesystem with broad tooling support.
- Formatting does not create a mount point; it only prepares the storage.
- You usually format a partition or logical volume, not an entire disk, unless the disk is dedicated to one filesystem.

**Identifying the correct block device**

Before formatting, inspect the storage layout so you do not destroy the wrong device. RHCSA tasks often require you to create a partition, format it, and mount it persistently.

```bash
lsblk -f
blkid
fdisk -l
```

Typical output helps you identify:
- Device names such as `/dev/sdb1` or `/dev/mapper/vg0-lv_data`
- Existing filesystem types
- Labels and UUIDs
- Mount points already in use

If you are working with a new disk, you may first create a partition table and partition, then format the new partition. A safe partitioning example with `parted` is:

```bash
parted /dev/sdb --script mklabel gpt mkpart primary 1MiB 100%
```

This creates a GPT partition table and one partition spanning the disk. It does not assign a filesystem type; the filesystem is created later with `mkfs.xfs` or `mkfs.ext4`.

> Do not format a device until you have verified it is not already in use. Formatting the wrong disk can destroy the operating system or user data.

**mkfs.xfs basics**

`mkfs.xfs` creates an XFS filesystem. On modern RHEL systems, this is the default choice for many server workloads because it scales well and supports online growth.

Common usage:

```bash
mkfs.xfs /dev/sdb1
```

You can also format a logical volume:

```bash
mkfs.xfs /dev/vgdata/lvshare
```

Useful options include:
- `-L <label>` to assign a filesystem label
- `-f` to force creation if the target appears to contain an existing filesystem signature

Example:

```bash
mkfs.xfs -L DATA /dev/sdb1
```

XFS characteristics to remember:
- **Online grow** is supported after mounting.
- **Shrink is not supported** with standard XFS tools.
- It performs well with large files and parallel I/O.
- It is commonly used for `/home`, application data, and general storage.

Because XFS cannot be shrunk, plan capacity carefully. If you need a filesystem that can be reduced later, ext4 is the more flexible choice.

**mkfs.ext4 basics**

`mkfs.ext4` creates an ext4 filesystem. ext4 is a reliable general-purpose filesystem and is often chosen when you want broad compatibility or the option to shrink the filesystem later.

Common usage:

```bash
mkfs.ext4 /dev/sdb1
```

Label example:

```bash
mkfs.ext4 -L BACKUP /dev/sdb1
```

Useful options include:
- `-L <label>` to set a label
- `-F` to force formatting
- `-m <percent>` to reserve a percentage of blocks for root, if needed
- `-b <block-size>` for advanced tuning, though this is rarely needed in RHCSA tasks

ext4 characteristics to remember:
- **Online grow** is supported.
- **Shrink** is possible, but only offline and with the correct order of operations.
- It is a safe default for many lab and exam scenarios.
- It supports journaling, which improves recovery after unclean shutdowns.

**Safe workflow for creating and formatting storage**

A typical RHCSA workflow is:

1. Identify the target disk with `lsblk`.
2. Create a partition table and partition if needed.
3. Create the filesystem with `mkfs.xfs` or `mkfs.ext4`.
4. Verify the filesystem with `blkid` or `lsblk -f`.
5. Mount it manually.
6. Add a persistent `/etc/fstab` entry.
7. Test with `mount -a`.

Example workflow:

```bash
lsblk
parted /dev/sdb --script mklabel gpt mkpart primary 1MiB 100%
mkfs.xfs -L DATA /dev/sdb1
mkdir -p /data
mount -L DATA /data
blkid /dev/sdb1
```

This sequence is safe because the filesystem is created only after the partition exists, and the mount uses the label assigned during formatting.

**Mounting by label or UUID**

After formatting, you usually mount by label or UUID rather than by raw device name. This reduces the risk of boot-time failures if device enumeration changes.

Examples:

```bash
mount -L DATA /data
mount UUID=1234-ABCD /data
```

You can also mount using the by-label path:

```bash
mount /dev/disk/by-label/DATA /data
```

For persistent mounts in `/etc/fstab`, labels and UUIDs are both common. UUIDs are more unique, while labels are easier to read.

Example `/etc/fstab` entry:

```fstab
LABEL=DATA  /data  xfs  defaults  0 0
```

For ext4:

```fstab
UUID=1234-ABCD  /backup  ext4  defaults  0 0
```

After editing `/etc/fstab`, test the entry:

```bash
mount -a
```

`mount -a` attempts to mount all entries that are not already mounted and reports syntax or mount errors during that test run. A clean run is a good sign, but it does not guarantee every future boot will succeed under all conditions.

**XFS versus ext4**

| Feature | XFS | ext4 |
|---|---|---|
| Default on RHEL | Yes | No |
| Online grow | Yes | Yes |
| Shrink | No | Yes, offline |
| Typical use | Large data volumes, servers | General-purpose storage, compatibility |
| Label support | Yes | Yes |
| RHCSA relevance | Very high | High |

Choose XFS when:
- You want the default RHEL filesystem.
- You expect large files or high throughput.
- You do not need shrink capability.

Choose ext4 when:
- You want a familiar, broadly supported filesystem.
- You may need to shrink the filesystem later.
- You are following a lab or application requirement that specifies ext4.

**Resizing considerations**

Filesystem resizing is a common exam topic, and the order matters.

For XFS:
- You can grow the filesystem after increasing the underlying block device or partition.
- You cannot shrink XFS with standard tools.

Example growth workflow on a mounted XFS filesystem:

```bash
xfs_growfs /data
```

For ext4:
- You must unmount the filesystem before shrinking it.
- You must shrink the filesystem first, then reduce the partition or logical volume.
- Reversing that order can destroy data.

Safe ext4 shrink concept:
1. Unmount the filesystem.
2. Check and repair it if needed with `e2fsck`.
3. Shrink the filesystem with `resize2fs`.
4. Reduce the partition or logical volume.
5. Mount and verify.

Example:

```bash
umount /backup
e2fsck -f /dev/sdb1
resize2fs /dev/sdb1 10G
```

Only after the filesystem is safely reduced should you resize the partition or logical volume to match. If you reduce the partition first, the filesystem may extend beyond the new boundary and become corrupted.

**Verification commands**

After formatting or mounting, verify the result immediately.

```bash
lsblk -f
blkid /dev/sdb1
df -hT /data
mount | grep /data
```

What to check:
- The filesystem type matches what you created.
- The label or UUID is correct.
- The mount point is active.
- The available size matches expectations.

If a mount fails, check:
- The device path in `/etc/fstab`
- The filesystem type field
- The mount point directory exists
- The filesystem was created successfully
- SELinux context or permissions if the mount is accessible but applications cannot use it

**Practical RHCSA examples**

Create and mount an XFS filesystem:

```bash
parted /dev/sdb --script mklabel gpt mkpart primary 1MiB 100%
mkfs.xfs -L PROJECTS /dev/sdb1
mkdir -p /projects
mount -L PROJECTS /projects
echo 'LABEL=PROJECTS /projects xfs defaults 0 0' >> /etc/fstab
mount -a
```

Create and mount an ext4 filesystem:

```bash
parted /dev/sdc --script mklabel gpt mkpart primary 1MiB 100%
mkfs.ext4 -L ARCHIVE /dev/sdc1
mkdir -p /archive
mount -L ARCHIVE /archive
echo 'LABEL=ARCHIVE /archive ext4 defaults 0 0' >> /etc/fstab
mount -a
```

Format a logical volume directly:

```bash
mkfs.xfs -L APPDATA /dev/vgapp/lvdata
mkdir -p /appdata
mount -L APPDATA /appdata
```

**Common mistakes to avoid**

- **Formatting the wrong device** because you skipped verification.
- Using `parted` partition type tokens as if they were filesystem types.
- Mounting with `mount LABEL=DATA /data` instead of `mount -L DATA /data`.
- Assuming `mount -a` proves future boots will always succeed.
- Trying to shrink XFS with ext4-style procedures.
- Reducing a partition before shrinking an ext4 filesystem.
- Forgetting to create the mount point directory before mounting.

**Exam-focused reminders**

RHCSA tasks usually reward a clean, repeatable workflow:
- Identify the target storage.
- Create the partition or logical volume.
- Format with the correct filesystem.
- Use labels or UUIDs for stable mounting.
- Test `/etc/fstab` with `mount -a`.
- Verify with `lsblk -f` and `df -hT`.

If the task specifies XFS, use `mkfs.xfs` and remember that growth is supported but shrink is not. If the task specifies ext4, use `mkfs.ext4` and remember that shrink is possible only offline and in the correct order.
