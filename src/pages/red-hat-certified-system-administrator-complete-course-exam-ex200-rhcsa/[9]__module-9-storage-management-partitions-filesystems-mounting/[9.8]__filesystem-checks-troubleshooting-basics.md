## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Filesystem troubleshooting mindset**

Filesystem problems usually show up as mount failures, read-only remounts, missing data, boot delays, or kernel messages about I/O errors and corruption. On RHCSA, you are expected to identify the affected device, inspect logs, verify mount configuration, and choose the safest repair path.

A good workflow is:

- Confirm the symptom from the shell and logs.
- Identify the filesystem type and underlying block device.
- Check whether the filesystem is mounted, read-only, or damaged.
- Repair only after protecting data and understanding the risk.

> If a filesystem is actively mounted read-write and you suspect corruption, do not run destructive repair tools casually. Unmount it first when possible, or boot into rescue/emergency mode for root filesystem repair.

**Common sources of evidence**

Use multiple tools together because each one shows a different layer of the problem.

- `journalctl -xb` shows boot-time messages and service failures.
- `dmesg` shows recent kernel and filesystem errors quickly.
- `lsblk` shows disks, partitions, and logical volumes.
- `blkid` shows UUIDs, labels, and filesystem types.
- `findmnt` shows current mount relationships.
- `/etc/fstab` shows persistent mount configuration.

Example triage commands:

```bash
lsblk -f
blkid
findmnt
journalctl -xb | tail -50
dmesg | tail -50
```

**Understanding mount and filesystem state**

A filesystem can fail in several ways:

- It may not mount because the device name changed.
- It may mount with the wrong type or wrong options.
- It may be forced read-only after errors.
- It may be missing its mount point directory.
- It may have metadata corruption that requires repair.

Use `findmnt` to confirm what is mounted and where:

```bash
findmnt -o TARGET,SOURCE,FSTYPE,OPTIONS
```

Use `mount` to inspect current mounts:

```bash
mount | column -t
```

If a filesystem is unexpectedly read-only, confirm the mount options:

```bash
findmnt /data
```

A read-only remount often indicates the kernel detected I/O or metadata errors. In that case, you usually need to:

1. Stop writes to the filesystem.
2. Review `dmesg` and `journalctl`.
3. Unmount the filesystem if possible.
4. Run the appropriate repair tool.
5. Remount and verify access.

**`/etc/fstab` troubleshooting**

`/etc/fstab` controls persistent mounts at boot and with `mount -a`. A small syntax error can prevent a filesystem from mounting correctly.

Typical fields are:

1. Device or UUID
2. Mount point
3. Filesystem type
4. Mount options
5. Dump field
6. `fs_passno` field for `fsck` order

Example:

```fstab
UUID=7c1f2d7a-8d4c-4d2a-9a2f-1f2a3b4c5d6e  /data  xfs  defaults  0  0
UUID=2b3c4d5e-6f70-4a81-9b2c-3d4e5f607182  /home  ext4  defaults  0  2
```

Important details:

- `mount -a` attempts to mount entries from `/etc/fstab` that are not already mounted.
- `mount -a` skips entries marked with `noauto`.
- `findmnt --verify` checks mount table syntax and consistency, but it does not guarantee every mount will succeed in all runtime conditions.
- `fs_passno` influences `fsck` checking order during boot, but parallel checking and system behavior can affect the exact sequence.

Validate the file before rebooting:

```bash
findmnt --verify
mount -a
```

If `mount -a` fails, inspect the exact line in `/etc/fstab` and compare it with the real device name, UUID, filesystem type, and mount point.

**Mount point directory checks**

A mount can fail even when the device and filesystem are correct if the target directory is missing or has the wrong permissions.

Check the mount point:

```bash
ls -ld /data
```

Create it if needed:

```bash
mkdir -p /data
chmod 755 /data
```

Use the correct ownership and permissions for the intended service. For example, a shared application directory may need a specific group or tighter access controls.

Also verify that the directory is empty before mounting if you expect to see the filesystem contents there. Mounting over existing files hides the underlying directory contents until unmounted.

**Filesystem check tools and how they are selected**

`fsck` is a front end that chooses the correct filesystem checker based on filesystem type. On RHEL, it commonly invokes ext-family checkers such as `e2fsck` for ext2/ext3/ext4 filesystems, but it can also dispatch to other supported checkers when available.

Use the generic front end when you want automatic selection:

```bash
fsck /dev/sdb1
```

Use the filesystem-specific tool when you know the type and want direct control:

```bash
e2fsck /dev/sdb1
```

For ext-family filesystems, `e2fsck` is the direct repair utility. This is useful when you want explicit control over repair behavior, especially in exam-style troubleshooting.

For XFS, use `xfs_repair` instead of `fsck`:

```bash
xfs_repair /dev/mapper/vg0-data
```

> Do not run `fsck` on a mounted XFS filesystem. XFS uses `xfs_repair`, and repair is performed offline.

**Safe repair workflow**

Before repair, protect data and reduce risk.

1. Identify the filesystem type and device.
2. Confirm whether it is mounted.
3. Back up important data if the filesystem is still readable.
4. Unmount the filesystem if possible.
5. Run the correct repair tool.
6. Recheck logs and remount.

Example workflow for an ext4 filesystem:

```bash
lsblk -f
findmnt /dev/sdb1
umount /data
e2fsck /dev/sdb1
mount /data
```

If the filesystem is busy, find the processes using it:

```bash
lsof +f -- /data
fuser -vm /data
```

Then stop the relevant service or move to rescue mode if needed.

**Interpreting `fsck` and `e2fsck` output**

Repair tools report what they found and what they changed. Common messages include:

- `clean` means the filesystem metadata appears consistent.
- `UNEXPECTED INCONSISTENCY; RUN fsck MANUALLY` means automatic boot-time checking detected a problem.
- `Inode ... has illegal blocks` suggests metadata damage.
- `Free blocks count wrong` or `Free inodes count wrong` indicates metadata counters need correction.
- `Superblock invalid` suggests the primary superblock may be damaged or unreadable.

Example output:

```text
e2fsck 1.46.5 (30-Dec-2021)
Pass 1: Checking inodes, blocks, and sizes
Inode 128: illegal block(s).  Clear? yes
Pass 2: Checking directory structure
Pass 5: Checking group summary information
Free blocks count wrong for group #0 (1234, counted=1228).
Fix? yes
```

Interpretation:

- The tool found metadata inconsistencies.
- It asked for confirmation because the repair changes on-disk structures.
- Answering `yes` with `-y` automates those fixes, but that is riskier on important data.

Use `-n` for a read-only check when you want to inspect without modifying:

```bash
e2fsck -n /dev/sdb1
```

Use `-y` only when you understand the risk and want automatic repair:

```bash
e2fsck -y /dev/sdb1
```

**Superblock-related recovery for ext filesystems**

The superblock stores critical filesystem metadata. If the primary superblock is damaged, ext filesystems may fail to mount or may report that the filesystem is invalid.

A common recovery approach is to try alternate superblocks. First, list backup superblock locations:

```bash
mke2fs -n /dev/sdb1
```

This prints candidate superblock locations without creating a filesystem. Then test one with `e2fsck`:

```bash
e2fsck -b 32768 /dev/sdb1
```

If needed, specify an alternate block size or use another backup superblock from the list.

Practical caution:

- Use this only for ext-family filesystems.
- Confirm the device carefully before running repair.
- Prefer a read-only check first when possible.

**XFS troubleshooting and repair**

XFS is common on RHEL systems and uses a different repair model from ext filesystems.

Key points:

- `xfs_repair` works offline.
- If the filesystem is mounted, unmount it first.
- If the filesystem was automatically remounted read-only, treat that as a warning sign and inspect logs before attempting repair.
- `xfs_repair` may require the `-L` option if the log is corrupted, but that can discard metadata updates and should be used only when necessary.

Example:

```bash
umount /data
xfs_repair /dev/mapper/vg0-data
```

If the log is damaged:

```bash
xfs_repair -L /dev/mapper/vg0-data
```

Example output interpretation:

```text
Phase 1 - find and verify superblock...
Phase 2 - using internal log
bad primary superblock - bad magic number !!!
```

This usually means the tool cannot read the expected metadata and you should verify:

- The correct device path
- The filesystem type
- Whether the volume is actually the one you intended to repair
- Whether the underlying storage has a larger problem

**Read-only remount troubleshooting**

A filesystem may switch to read-only after the kernel detects errors. This is a protective behavior, not a fix.

Typical symptoms:

- Writes fail with `Read-only file system`
- Applications cannot create or update files
- `mount` or `findmnt` shows `ro` in the options
- `dmesg` contains I/O or journal errors

Example:

```bash
touch /data/testfile
# touch: cannot touch '/data/testfile': Read-only file system
```

Troubleshooting steps:

1. Check `dmesg` for the first error.
2. Check `journalctl -xb` for related messages.
3. Confirm the filesystem type and device.
4. Unmount the filesystem if possible.
5. Run the correct offline repair tool.
6. Remount and retest write access.

If the filesystem is root (`/`) and has been remounted read-only, you often need rescue or emergency mode because you cannot safely repair the active root filesystem in normal operation.

**Emergency mode vs rescue mode**

Use the right recovery environment based on the problem.

- **Emergency mode** gives a minimal environment with the root filesystem often mounted read-only or not fully mounted.
- **Rescue mode** provides a more complete environment and is useful when you need tools, networking, or broader system access.

For root filesystem repair, you often boot into one of these modes so you can run offline checks safely.

Typical recovery flow:

1. Reboot and interrupt the boot process if needed.
2. Enter emergency or rescue mode.
3. Identify the root device.
4. Remount read-write only if appropriate and safe.
5. Run the filesystem repair tool.
6. Reboot and verify normal boot.

**LVM context in filesystem repair**

On RHCSA systems, filesystems are often on Logical Volume Manager (LVM) volumes rather than raw partitions. Before repair, identify the logical volume behind the mount.

Useful commands:

```bash
lsblk -f
pvs
vgs
lvs
```

Example mapping:

```bash
lsblk -f
NAME            FSTYPE LABEL UUID                                 MOUNTPOINT
sda
├─sda1          xfs          1111-2222                            /boot
└─sda2          LVM2_member  ...
  ├─rhel-root   xfs          aaaa-bbbb-cccc-dddd                  /
  └─rhel-home   xfs          eeee-ffff-gggg-hhhh                  /home
```

If `/home` is on `rhel-home`, repair the logical volume device, not the top-level physical disk:

```bash
umount /home
xfs_repair /dev/rhel/home
```

This matters because the visible mount point is not the same as the underlying block device.

**Practical case study: ext filesystem corruption**

A server reports that `/data` will not mount after a power loss.

Symptoms:

```bash
mount /data
# mount: /data: wrong fs type, bad option, bad superblock on /dev/sdb1, missing codepage or helper program, or other error.
```

Triage:

```bash
lsblk -f
dmesg | tail -20
findmnt /data
```

Recovery:

```bash
umount /data 2>/dev/null
e2fsck -f /dev/sdb1
mount /data
```

If the primary superblock is damaged:

```bash
mke2fs -n /dev/sdb1
e2fsck -b 32768 /dev/sdb1
```

After repair, verify:

```bash
mount | grep /data
ls -la /data
```

**Practical case study: XFS volume remounted read-only**

A database volume at `/var/lib/app` becomes read-only during heavy writes.

Symptoms:

```bash
touch /var/lib/app/test
# touch: cannot touch '/var/lib/app/test': Read-only file system
```

Triage:

```bash
findmnt /var/lib/app
dmesg | tail -50
journalctl -xb | tail -50
```

Likely findings include XFS log or I/O errors. Recovery:

```bash
systemctl stop app.service
umount /var/lib/app
xfs_repair /dev/mapper/vgapp-lvdata
mount /var/lib/app
systemctl start app.service
```

If `xfs_repair` reports log corruption and cannot proceed, you may need:

```bash
xfs_repair -L /dev/mapper/vgapp-lvdata
```

Use `-L` only when necessary because it can discard recent metadata updates.

**Verifying the fix**

After repair, confirm the filesystem is healthy and mounted correctly.

Check mount status:

```bash
findmnt /data
```

Check available space and inode usage:

```bash
df -hT /data
df -i /data
```

Check for lingering kernel errors:

```bash
dmesg | tail -50
```

If the filesystem is still unstable, suspect underlying storage problems such as:

- Failing disk hardware
- Storage controller issues
- Cabling or SAN problems
- Repeated I/O errors from the kernel

Filesystem repair does not fix bad hardware.

**Quick command reference**

- `lsblk -f` — show block devices, filesystems, and mount points
- `blkid` — show UUIDs and filesystem types
- `findmnt` — inspect current mounts
- `findmnt --verify` — validate mount table syntax and consistency
- `mount -a` — mount eligible entries from `/etc/fstab`
- `dmesg` — review kernel messages
- `journalctl -xb` — review boot logs
- `fsck` — generic filesystem check front end
- `e2fsck` — ext-family repair tool
- `xfs_repair` — XFS repair tool
- `mke2fs -n` — print ext backup superblock locations
- `lsof` / `fuser` — identify processes using a mount

**Exam-focused habits**

- Always identify the filesystem type before choosing a repair tool.
- Always confirm the underlying device, especially on LVM systems.
- Always check logs before and after repair.
- Always prefer a safe offline repair over guessing.
- Always verify `/etc/fstab` when a mount fails at boot.
- Always consider mount point existence and permissions, not just the filesystem itself.
