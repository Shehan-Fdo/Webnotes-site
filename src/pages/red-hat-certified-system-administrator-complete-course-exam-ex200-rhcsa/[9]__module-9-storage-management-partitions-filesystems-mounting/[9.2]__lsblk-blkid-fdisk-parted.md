## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Block device discovery and partitioning overview**

RHCSA storage tasks often start with identifying the correct disk, creating or adjusting partitions, formatting them, and then verifying the result before mounting. The four tools in this topic serve different roles:

- `lsblk` shows the block-device tree and helps you map disks, partitions, and mount points.
- `blkid` reports identifiers such as filesystem UUIDs and partition UUIDs.
- `fdisk` edits partition tables interactively, especially useful for MBR and GPT workflows.
- `parted` creates and manages partitions with explicit units and alignment control.

A safe workflow is to inspect first, modify second, and verify last. That habit matters in the exam because a wrong disk selection can destroy data.

> Always confirm the target device with multiple commands before writing changes. `lsblk`, `blkid`, and `fdisk -l` often reveal different details that help you avoid mistakes.

**`lsblk`: view the block-device hierarchy**

`lsblk` lists block devices in a tree, which makes it easy to see how disks, partitions, LVM logical volumes, and mount points relate to each other. It is one of the fastest ways to confirm whether a new partition exists and whether it is mounted.

Common uses:

- Identify the device name, size, and type.
- See parent/child relationships between disks and partitions.
- Check mount points and filesystem labels.
- Produce machine-readable output for scripts.

Useful options:

- `-f` shows filesystem information such as `FSTYPE`, `UUID`, `LABEL`, and `MOUNTPOINTS`.
- `-o` selects specific columns.
- `-a` shows empty devices as well as populated ones.
- `-p` prints full device paths such as `/dev/sda`.
- `-J` outputs JSON for automation.
- `-n` suppresses headings.
- `-e` excludes devices by major number, which is useful when filtering out loop or RAM devices.

Example:

```bash
lsblk
lsblk -f
lsblk -o NAME,SIZE,TYPE,FSTYPE,UUID,MOUNTPOINTS
lsblk -p -o NAME,SIZE,MODEL,SERIAL
lsblk -J -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINTS
```

Typical columns to understand:

- `NAME` is the device or partition name.
- `SIZE` is the device size.
- `TYPE` is the device node type, such as `disk`, `part`, `rom`, `lvm`, or `loop` depending on the device and system.
- `FSTYPE` is the filesystem type if one exists.
- `UUID` is the filesystem UUID when available.
- `MOUNTPOINTS` shows where the device is mounted.

Practical interpretation:

- A disk with no children usually has no partitions yet.
- A partition with no `FSTYPE` may be unformatted.
- A partition with a filesystem but no mount point may be ready for mounting.
- A mounted filesystem should appear in both `lsblk` and `mount` output.

Example verification after partitioning and formatting:

```bash
lsblk -f /dev/sdb
```

**`blkid`: identify UUIDs, labels, and filesystem signatures**

`blkid` queries block devices for metadata stored in filesystem and partition signatures. It is especially useful when you need the UUID for `/etc/fstab` or when you want to confirm that a device has been formatted.

What it reports:

- Filesystem type, such as `ext4`, `xfs`, or `vfat`.
- Filesystem UUID.
- Filesystem label.
- Partition UUID on GPT disks when present.
- Signature information for recognized block devices.

Example:

```bash
blkid
blkid /dev/sdb1
blkid -o full
blkid -o export /dev/sdb1
```

Important distinctions:

- **Filesystem UUID** belongs to the filesystem created by `mkfs`.
- **Partition UUID** belongs to the partition entry itself, especially on GPT.
- These are not the same value, and `/etc/fstab` usually uses the filesystem UUID unless you intentionally choose another identifier.

Behavior on empty or wiped devices:

- An unformatted partition may produce no output or only partial information.
- After `wipefs`, signatures may disappear, so `blkid` may stop recognizing the device.
- If a device has multiple stale signatures, `blkid` may still detect them until they are removed.

Example of using `blkid` for `fstab`:

```bash
blkid /dev/sdb1
# /dev/sdb1: UUID="2f3d2c1a-7c4b-4d1f-9f2a-8b7f0d8c9a11" TYPE="xfs"
```

Then in `/etc/fstab`:

```fstab
UUID=2f3d2c1a-7c4b-4d1f-9f2a-8b7f0d8c9a11  /data  xfs  defaults  0 0
```

**`fdisk`: interactive partition table editor**

`fdisk` is a classic interactive tool for creating, deleting, and modifying partitions. It works with both MBR (`msdos`) and GPT partition tables, but the workflow differs slightly between them.

Use `fdisk` when you want:

- An interactive, menu-driven partitioning session.
- A quick way to inspect and edit a disk.
- To work with legacy MBR disks or modern GPT disks.
- To view and change partition types in a human-friendly interface.

Start by opening the target disk, not a partition:

```bash
fdisk /dev/sdb
```

Common `fdisk` actions:

- `p` prints the current partition table.
- `n` creates a new partition.
- `d` deletes a partition.
- `t` changes the partition type.
- `w` writes changes and exits.
- `q` quits without saving.
- `m` shows help.

GPT vs MBR behavior:

- On **MBR**, partition types are represented by type codes.
- On **GPT**, partition types are GUID-based and `fdisk` shows human-readable type names.
- The `t` command still changes the partition type, but the meaning is different depending on the partition table style.

Example session outline:

```bash
fdisk /dev/sdb
```

Inside `fdisk`:

```text
Command (m for help): p
Command (m for help): n
Partition number (1-4, default 1): 1
First sector (2048-..., default 2048): 
Last sector, +/-sectors or +/-size{K,M,G,T,P}: +5G
Command (m for help): t
Selected partition 1
Partition type or alias (type L to list all): 83
Command (m for help): w
```

Notes for exam accuracy:

- On GPT, `fdisk` does not use MBR-style type codes in the same way; it uses GPT partition type names or GUID-related aliases.
- `fdisk` edits the partition table; it does not format filesystems.
- After writing changes, the kernel may not immediately see the new layout.

**`parted`: partitioning with explicit units and alignment**

`parted` is useful when you want precise control over start and end boundaries, especially with `MiB`, `GiB`, or sectors. It is commonly used for scripted partition creation and for disks where alignment matters.

Use `parted` when you need:

- Explicit unit control.
- GPT or MBR partition table creation.
- Non-interactive scripting.
- Better handling of large disks and modern alignment practices.

Important accuracy note:

- `parted` can create, delete, resize partitions, and print partition tables.
- It does **not** resize the partition table itself.
- It is not a filesystem formatter.

Check the current layout:

```bash
parted /dev/sdb print
parted /dev/sdb unit MiB print
```

Create a new partition table if needed:

```bash
parted /dev/sdb mklabel gpt
```

Create a partition with aligned boundaries:

```bash
parted -s /dev/sdb mkpart primary xfs 1MiB 5GiB
```

Key syntax points:

- The first token after `mkpart` is the partition name or legacy label context depending on the table type.
- The filesystem type hint, such as `xfs`, helps `parted` choose alignment and metadata expectations.
- The start and end values must be valid boundaries.
- Units matter: `MiB`, `GiB`, and sectors are all accepted, but you should be consistent.

More examples:

```bash
parted -s /dev/sdb mkpart data ext4 1MiB 10GiB
parted -s /dev/sdb mkpart 1 10GiB 20GiB
parted -s /dev/sdb set 1 lvm on
```

Alignment guidance:

- Start partitions at `1MiB` when possible to align with common storage boundaries.
- Avoid arbitrary sector starts unless the task specifically requires them.
- Use `print` after creation to confirm the exact boundaries.

**Choosing between `fdisk` and `parted`**

Both tools can solve many RHCSA partitioning tasks, but they fit different situations.

| Tool | Best for | Strengths | Cautions |
|---|---|---|---|
| `fdisk` | Interactive partition edits | Easy to inspect and modify; familiar menus | GPT type handling differs from MBR; not ideal for scripted precision |
| `parted` | Precise boundary control and scripting | Explicit units, alignment, non-interactive use | Syntax is less forgiving; easy to misread start/end values |

Practical preference:

- Use `fdisk` when the task is interactive and you want a quick human-readable workflow.
- Use `parted` when the task requires exact sizes, scripted commands, or alignment-sensitive partitioning.
- Use the tool that matches the existing partition table and the exam prompt.

> Do not assume GPT is always the right choice. If the task specifies `msdos`/MBR or you are modifying an existing legacy disk, follow the requirement exactly.

**Typical RHCSA workflow: create, format, verify, mount**

A common exam task is to create a partition, format it, and prepare it for mounting. The sequence below shows the full flow.

1. Inspect the disk.
2. Create the partition.
3. Re-read the partition table if needed.
4. Format the new partition.
5. Verify identifiers.
6. Mount it and persist the mount in `/etc/fstab`.

Example using `parted`:

```bash
lsblk -f
parted /dev/sdb print
parted -s /dev/sdb mklabel gpt
parted -s /dev/sdb mkpart data xfs 1MiB 5GiB
partprobe /dev/sdb
lsblk /dev/sdb
mkfs.xfs /dev/sdb1
blkid /dev/sdb1
mkdir -p /data
mount /dev/sdb1 /data
```

Then add it to `/etc/fstab` using the UUID:

```bash
UUID=2f3d2c1a-7c4b-4d1f-9f2a-8b7f0d8c9a11  /data  xfs  defaults  0 0
```

Verify:

```bash
mount -a
lsblk -f
findmnt /data
```

**Kernel re-read and device refresh**

After partition changes, the kernel may need to re-read the partition table. Sometimes the new partition appears immediately; sometimes it does not.

Useful commands:

```bash
partprobe /dev/sdb
udevadm settle
lsblk /dev/sdb
```

When `partprobe` is often sufficient:

- The disk is not busy.
- The partition table change is simple.
- The system accepts the new layout without errors.

When you may need more:

- The device is in use.
- The kernel refuses to re-read the table.
- The new partition does not appear after `partprobe`.

In RHCSA lab conditions, if the kernel does not refresh cleanly, you may need to log out, rescan, or reboot if the task allows it. Always verify the new device node before formatting.

**Troubleshooting and safety checks**

Before writing changes, confirm the target disk with more than one command.

- Use `lsblk` to see the device tree.
- Use `blkid` to confirm existing signatures.
- Use `fdisk -l /dev/sdb` or `parted /dev/sdb print` to inspect the partition table.
- Use `wipefs -n /dev/sdb1` if you need to preview signatures without erasing them.

Common mistakes:

- Formatting the disk instead of the partition, such as using `/dev/sdb` instead of `/dev/sdb1`.
- Confusing filesystem UUIDs with partition UUIDs.
- Assuming a new partition is ready before the kernel has re-read the table.
- Using the wrong partition table style for the task.
- Misreading `fdisk` type changes on GPT as if they were MBR type codes.

Useful inspection commands:

```bash
fdisk -l /dev/sdb
parted /dev/sdb print
blkid
lsblk -f
wipefs -n /dev/sdb1
```

**Quick reference**

- `lsblk` is for structure and mount visibility.
- `blkid` is for UUIDs, labels, and signature checks.
- `fdisk` is for interactive partition table editing.
- `parted` is for precise, script-friendly partition creation and alignment.

A reliable exam habit is to inspect with `lsblk`, confirm signatures with `blkid`, edit with `fdisk` or `parted`, refresh the kernel view, and verify again before mounting.
