## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

`df`, `du`, `mount`, and `findmnt` are core Linux tools for checking storage capacity, identifying where data lives, and validating filesystem mounts. On RHCSA systems, you use them together to answer practical questions such as:

- Is the filesystem full, or is one directory consuming space?
- Is a device mounted where you expect?
- Did an `/etc/fstab` entry work after reboot-style validation?
- Which process is preventing a filesystem from being unmounted?

These commands solve different problems:

- **`df`** reports filesystem-level usage.
- **`du`** reports directory and file usage.
- **`mount`** attaches filesystems and shows current mounts.
- **`findmnt`** displays the mount tree and is especially useful for filtering and scripting.

RHEL commonly uses **XFS** as the default filesystem for local storage, so exam workflows often involve XFS formatting, mounting, and verifying persistence. You should still recognize other filesystem types such as `ext4`, `vfat`, and network filesystems when they appear in tasks.

**`df`: filesystem capacity and inode usage**

`df` shows free and used space from the filesystem’s perspective. It is the right tool when you need to know whether a mounted filesystem is nearing capacity.

Typical usage:

```bash
df
df -h
df -Th
df -i
df -h /home
```

Key options:

- **`-h`** shows human-readable sizes.
- **`-T`** includes the filesystem type.
- **`-i`** shows inode usage instead of block usage.
- **`-a`** includes pseudo and inaccessible filesystems.
- **`-x TYPE`** excludes a filesystem type.
- **`-t TYPE`** limits output to a filesystem type.

How to read the output:

- **Filesystem**: the device, pseudo filesystem, or remote source.
- **1K-blocks / Size**: total capacity in the unit shown.
- **Used**: space currently allocated.
- **Available**: free space available to non-root users, depending on filesystem reservation behavior.
- **Use%**: percentage of used blocks.
- **Mounted on**: mount point path.

Example:

```bash
df -hT
```

Sample interpretation:

```text
Filesystem     Type  Size  Used Avail Use% Mounted on
/dev/sda2      xfs    40G   18G   22G  45% /home
tmpfs          tmpfs  1.9G     0  1.9G   0% /run
```

Important RHCSA detail:

- **XFS** does not support shrinking, so if a filesystem is too small, you usually create a new one or extend the underlying storage if possible.
- `df` reports filesystem usage, not directory usage. A directory can look large while the filesystem still has plenty of free space, or vice versa if deleted-but-open files exist.

Inode troubleshooting:

- If `df -h` shows free space but writes still fail, check `df -i`.
- A filesystem can run out of **inodes** before it runs out of blocks, especially on workloads with many small files.
- Inode exhaustion often appears as “No space left on device” even when block space remains.

**`du`: directory and file usage**

`du` measures disk usage by walking directories and files. Use it when you need to find what is consuming space inside a filesystem.

Typical usage:

```bash
du
du -h
du -sh /var/log
du -h --max-depth=1 /var
du -ah /home/student
```

Key options:

- **`-h`** shows human-readable sizes.
- **`-s`** summarizes only the total for each argument.
- **`-a`** includes files as well as directories.
- **`--max-depth=N`** limits recursion depth.
- **`-x`** stays on one filesystem and avoids crossing mount points.
- **`-c`** prints a grand total.

Practical examples:

```bash
du -sh /var/log/*
du -h --max-depth=1 /var
du -xhd1 /
```

Use `du` when:

- You need to identify a large directory tree.
- `df` shows a filesystem is full, but you do not know which path is responsible.
- You want to compare usage across subdirectories.

Common exam scenario:

- `df` says `/var` is nearly full.
- `du -sh /var/*` reveals that `/var/log` or `/var/lib` is the main consumer.
- You then inspect logs, caches, or package data to decide what to clean up.

Why `du` and `df` can disagree:

- Deleted files may still be held open by a process, so `df` still counts the space but `du` cannot see the file.
- `du` does not account for filesystem metadata the same way `df` does.
- Sparse files can appear large in `du` depending on options and filesystem behavior.

Useful troubleshooting pattern:

```bash
du -xhd1 /var | sort -h
```

This gives a quick ranked view of top-level usage on the same filesystem.

**`mount`: attach filesystems and inspect mount options**

`mount` is used to attach a filesystem to the directory tree and to display current mounts. On RHCSA, you need to know both the command syntax and how to verify the result.

Basic syntax:

```bash
mount [options] <source> <target>
```

Examples:

```bash
mount /dev/sdb1 /mnt/data
mount -t xfs /dev/sdb1 /mnt/data
mount -o ro /dev/sdb1 /mnt/data
```

Common inspection forms:

```bash
mount
mount -l
mount -v
mount -t xfs
```

What these do:

- **`mount`** with no arguments prints mounted filesystems.
- **`mount -l`** includes labels where available.
- **`mount -v`** gives verbose output, which can help during troubleshooting.
- **`mount -t TYPE`** filters by filesystem type.

Mount options you should recognize:

- **`ro`**: read-only
- **`rw`**: read-write
- **`noexec`**: prevents execution of binaries from the mount
- **`nosuid`**: ignores setuid/setgid bits
- **`nodev`**: ignores device files
- **`relatime`**: common default access-time behavior
- **`defaults`**: shorthand for the standard set of options

Example `/etc/fstab` entry:

```fstab
UUID=1234-ABCD  /data  xfs  defaults,noatime  0  0
```

How to mount persistently:

1. Create the mount point if needed:
   ```bash
   mkdir -p /data
   ```
2. Identify the device UUID:
   ```bash
   blkid /dev/sdb1
   ```
3. Add the entry to `/etc/fstab`.
4. Test without rebooting:
   ```bash
   mount -a
   ```
5. Verify the target:
   ```bash
   findmnt /data
   ```

> `mount -a` only mounts entries that are not already mounted and are valid in `/etc/fstab`. It is a non-destructive validation step, but a bad entry can still fail the command and should be corrected before reboot.

Common mount failure cases:

- The target directory does not exist.
- The source device or UUID is wrong.
- The filesystem type is incorrect.
- The mount options are invalid.
- The filesystem is already mounted elsewhere.
- The mount point is busy during unmount attempts.

Bind mounts:

A **bind mount** makes an existing directory appear at another path.

```bash
mount --bind /srv/projects /mnt/projects
```

Use cases:

- You need the same data visible in two locations.
- You want to expose a directory inside a chroot or container-related workflow.
- You need a quick exam-friendly way to map an existing directory to a required path.

Persistent bind mount example in `/etc/fstab`:

```fstab
/srv/projects  /mnt/projects  none  bind  0  0
```

You can also combine bind mounts with read-only remounting:

```bash
mount --bind /srv/projects /mnt/projects
mount -o remount,bind,ro /mnt/projects
```

**`umount`: cleanly detach filesystems**

`umount` detaches a mounted filesystem. The command requires the mount point or the source device, but the mount point is usually clearer and less error-prone.

Examples:

```bash
umount /mnt/data
umount /dev/sdb1
```

Important requirements:

- The filesystem must not be actively in use.
- You need sufficient privileges, usually root.
- The mount point should not be the current working directory of a shell or process.

Common failure cases:

- **Busy mount point**: a process has an open file, current directory, or working handle on the filesystem.
- **Open files**: editors, shells, services, or loggers keep files in use.
- **Nested mounts**: child mounts must be unmounted before the parent.

Troubleshooting busy mounts:

```bash
lsof +f -- /mnt/data
fuser -vm /mnt/data
```

What to do:

1. Identify the process holding the mount open.
2. Stop the service or exit the shell using the mount.
3. Retry `umount`.
4. If needed, unmount child mounts first.

> Avoid forceful unmounting unless you understand the risk. A forced or lazy unmount can hide the problem rather than solve it, and it may leave data unsafely buffered.

**`findmnt`: structured mount inspection**

`findmnt` is one of the best tools for exam troubleshooting because it shows the mount tree in a structured way. It is especially useful when you need to confirm parent-child relationships, mount options, or a specific target.

Basic usage:

```bash
findmnt
findmnt /data
findmnt /mnt/projects
```

Useful filtering and output options:

- **`-r`**: raw output, useful for scripts and predictable parsing.
- **`-n`**: no headings.
- **`-J`**: JSON output for structured processing.
- **`--evaluate`**: resolves `LABEL=`, `UUID=`, and similar specifiers to actual devices.
- **`-o`**: choose specific columns.
- **`-t TYPE`**: filter by filesystem type.
- **`-S SOURCE`**: filter by source device or specifier.
- **`-T TARGET`**: filter by target mount point.

Examples:

```bash
findmnt -rno TARGET,SOURCE,FSTYPE,OPTIONS
findmnt -J /data
findmnt --evaluate -S UUID=1234-ABCD
findmnt -t xfs
findmnt -o TARGET,SOURCE,FSTYPE,OPTIONS /var
```

Why `findmnt` matters:

- It shows the actual mount hierarchy, not just a flat list.
- It helps you confirm whether a bind mount or nested mount is in place.
- It is easier to script than parsing `mount` output.
- It is ideal for verifying that `/etc/fstab` changes took effect.

Example of a mount tree:

```text
TARGET        SOURCE      FSTYPE OPTIONS
/             /dev/sda2   xfs    rw,relatime
├─/home       /dev/sda3   xfs    rw,relatime
├─/var        /dev/sda4   xfs    rw,relatime
└─/mnt/data   /dev/sdb1   xfs    rw,relatime
```

This view helps you see that `/mnt/data` is a separate filesystem, while a bind mount would appear as another target pointing to the same source or directory.

JSON example:

```bash
findmnt -J /data
```

This is useful when you need machine-readable output in scripts or when checking multiple systems.

**Comparing the tools**

| Command | Best for | Typical question answered |
|---|---|---|
| `df` | Filesystem capacity | “Is the filesystem full?” |
| `du` | Directory/file usage | “What is consuming space?” |
| `mount` | Mounting and quick inspection | “Is it mounted, and with what options?” |
| `findmnt` | Structured mount relationships | “What is mounted where?” |

A practical workflow often uses all four:

1. Use `df -hT` to confirm the filesystem is full.
2. Use `du -xhd1 /path` to find the largest directory.
3. Use `findmnt /path` to confirm the mount source and options.
4. Use `mount -a` after editing `/etc/fstab`.
5. Use `umount` and `lsof` or `fuser` if the filesystem must be detached.

**Real-world RHCSA scenarios**

A filesystem appears full, but `du` does not show enough usage:

- Check `df -i` for inode exhaustion.
- Check for deleted-but-open files with `lsof`.
- Confirm the filesystem type with `df -Th` or `findmnt`.

A new disk is added and must be mounted persistently:

```bash
lsblk
blkid /dev/sdb1
mkdir -p /data
echo 'UUID=1234-ABCD /data xfs defaults 0 0' >> /etc/fstab
mount -a
findmnt /data
```

A bind mount is required for a service path:

```bash
mkdir -p /srv/appdata /var/www/appdata
mount --bind /srv/appdata /var/www/appdata
findmnt /var/www/appdata
```

A mount will not unmount:

```bash
fuser -vm /mnt/data
lsof +f -- /mnt/data
umount /mnt/data
```

If a shell is sitting inside the mount point, change directories first:

```bash
cd /
umount /mnt/data
```

**Filesystem and mount defaults to remember**

- RHEL commonly uses **XFS** for local filesystems.
- XFS is mounted and formatted with standard tools such as `mkfs.xfs`, `mount`, and `findmnt`.
- `mount` often shows defaults such as `rw`, `relatime`, `seclabel`, `attr2`, `inode64`, and `logbufs=8` on XFS systems, though exact options vary by kernel and configuration.
- `/etc/fstab` should use stable identifiers such as `UUID=` or `LABEL=` rather than device names that may change.

**Quick reference commands**

```bash
df -hT
df -i
du -sh /var/log
du -xhd1 /var | sort -h
mount -l
mount -v
findmnt -rno TARGET,SOURCE,FSTYPE,OPTIONS
findmnt -J /data
mount -a
umount /mnt/data
lsof +f -- /mnt/data
fuser -vm /mnt/data
```

**Exam tips**

- Use `df` for filesystem health, not directory hunting.
- Use `du` for directory hunting, not filesystem totals.
- Use `findmnt` when you need a clean answer about what is mounted where.
- Use `mount -a` to validate `/etc/fstab` before you rely on a reboot.
- Use `lsof` or `fuser` before unmounting a busy filesystem.
- Remember that XFS is the default local filesystem on many RHCSA systems, so mount and format tasks often involve XFS-specific workflows.
