## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Partitioning fundamentals**

A partition table describes how a disk is divided into addressable regions. On Linux, you use partitions to separate operating system files, application data, swap, and boot-related content so that each area can be managed independently.

A partition is not a filesystem. You first create the partition, then create a filesystem on it, and then mount it. That separation matters on the RHCSA exam because many storage tasks require you to identify the correct layer before making changes.

- **Disk**: the physical or virtual storage device, such as `/dev/sda` or `/dev/nvme0n1`.
- **Partition table**: metadata that records partition boundaries and identifiers.
- **Partition**: a slice of the disk, such as `/dev/sda1`.
- **Filesystem**: the structure created inside a partition, such as `xfs` or `ext4`.
- **Mount point**: the directory where the filesystem becomes accessible, such as `/data`.

**Partition table types**

Modern Linux systems commonly use either MBR or GPT. You should know the differences because they affect disk size limits, partition count, and boot behavior.

| Feature | MBR | GPT |
|---|---:|---:|
| Full name | Master Boot Record | GUID Partition Table |
| Typical use | Older systems, compatibility cases | Modern systems |
| Partition limit | Usually 4 primary partitions, or 3 primary + 1 extended | Many partitions, commonly 128 by default in tools |
| Disk size support | Limited by 32-bit addressing | Supports very large disks |
| Boot metadata | Boot code in first sector | Protective MBR plus GPT headers |
| Reliability | Single primary table area | Redundant headers and partition arrays |

MBR still appears in legacy environments, but GPT is the normal choice on modern RHEL systems, especially for large disks and UEFI-based systems. For exam work, you should recognize both, but do not assume MBR is the default.

> On GPT systems, the EFI System Partition uses the `esp` flag. Do not treat `boot` and `esp` as interchangeable.

**Partition naming and device paths**

Linux names partitions differently depending on the device type.

- Traditional SATA/SCSI-style disks use names like `/dev/sda1`, `/dev/sda2`.
- NVMe devices use names like `/dev/nvme0n1p1`, `/dev/nvme0n1p2`.
- Virtual disks may appear as `/dev/vda1` or similar depending on the hypervisor.

You should always verify the actual device name before partitioning. A wrong device path can destroy data on the wrong disk.

Useful inspection commands:

```bash
lsblk
lsblk -f
blkid
fdisk -l
parted -l
```

`lsblk` is often the fastest way to see disks, partitions, filesystems, and mount points together. `blkid` is useful when you need UUIDs or filesystem types. `fdisk -l` and `parted -l` help you inspect partition tables and sizes.

**Choosing a partitioning tool**

Several tools can create and inspect partitions. The best choice depends on the task.

- **`parted`**: useful for GPT and MBR, supports scripted workflows, and handles large disks well.
- **`fdisk`**: interactive and familiar; modern versions support GPT, but it is often used for general partition editing rather than GPT-specific workflows.
- **`gdisk`**: GPT-focused tool, useful when you need explicit GPT management.
- **`lsblk` / `blkid`**: inspection tools, not partition editors.

For RHCSA-style tasks, `parted` is especially common because it supports noninteractive commands and works well in automation.

**Creating partitions with `parted`**

`parted` edits the partition table, not the filesystem. When you create a partition, you specify the start and end boundaries. The optional token such as `xfs` in `mkpart` is only a heuristic label for alignment and convenience; it does not format the partition.

Example workflow:

```bash
parted /dev/sdb --script mklabel gpt
parted /dev/sdb --script mkpart primary 1MiB 2GiB
parted /dev/sdb --script mkpart primary 2GiB 100%
```

After creating the partition, format it separately:

```bash
mkfs.xfs /dev/sdb1
mkfs.ext4 /dev/sdb2
```

Then mount it:

```bash
mkdir -p /data
mount /dev/sdb1 /data
```

If you want persistence across reboots, add the filesystem to `/etc/fstab` using the UUID.

```bash
blkid /dev/sdb1
```

Example `/etc/fstab` entry:

```fstab
UUID=1234-ABCD  /data  xfs  defaults  0  0
```

**Creating partitions with `fdisk`**

`fdisk` is interactive and useful when you want to inspect and edit a disk manually. On modern systems it supports GPT as well as MBR, but you should still understand the table type you are working with.

Typical workflow:

```bash
fdisk /dev/sdb
```

Inside `fdisk`, you usually:

- press `g` to create a new GPT table
- press `n` to create a new partition
- accept or specify start and end sectors
- press `w` to write changes

Example session outline:

```text
Command (m for help): g
Created a new GPT disklabel.

Command (m for help): n
Partition number (1-128, default 1):
First sector (2048-..., default 2048):
Last sector, +/-sectors or +/-size{K,M,G,T,P}:

Command (m for help): w
```

Use `fdisk` when you want a quick interactive workflow, but remember that it does not format the partition. You still need `mkfs` afterward.

**MBR-specific concepts**

MBR has a few concepts that matter for troubleshooting and exam questions.

- **Primary partition**: one of the main partitions recorded in the MBR table.
- **Extended partition**: a container used to work around the four-partition limit.
- **Logical partition**: a partition inside the extended partition.

This structure exists because classic MBR only supports four partition entries. If you need more than four partitions on an MBR disk, one entry becomes an extended partition and the rest become logical partitions inside it.

MBR also has a boot code area in the first sector. That is one reason GPT is preferred for modern systems: GPT is more flexible and more robust.

**GPT-specific concepts**

GPT stores partition data in a more modern format and is the standard for current Linux installations.

- **GUID**: Globally Unique Identifier, used to identify partitions and disk metadata.
- **Protective MBR**: a compatibility structure that helps older tools recognize the disk.
- **EFI System Partition (ESP)**: a small FAT-formatted partition used by UEFI firmware to load boot files.

For UEFI booting, the ESP is typically formatted as FAT32 and flagged as `esp`. The bootloader files live there, not in the root filesystem.

Example with `parted`:

```bash
parted /dev/sda --script mklabel gpt
parted /dev/sda --script mkpart ESP fat32 1MiB 513MiB
parted /dev/sda --script set 1 esp on
mkfs.vfat -F32 /dev/sda1
```

The `fat32` token in `mkpart` helps describe the intended use and alignment, but the actual filesystem is created by `mkfs.vfat`.

**Partition flags and labels**

Partition flags are metadata used by tools and firmware to interpret partitions. They are not filesystems.

Common examples:

- `esp`: marks an EFI System Partition on GPT
- `boot`: may be used in some contexts, but it is not the same as `esp`
- `lvm`: indicates a partition intended for LVM physical volume use
- `raid`: indicates a partition intended for software RAID use

Example:

```bash
parted /dev/sda --script set 1 esp on
parted /dev/sdb --script set 2 lvm on
```

Use flags only when they match the intended role of the partition. Do not assume a flag creates any bootability or filesystem by itself.

**Filesystem creation after partitioning**

Once the partition exists, create the filesystem that matches your use case.

Common commands:

```bash
mkfs.xfs /dev/sdb1
mkfs.ext4 /dev/sdb2
mkfs.vfat -F32 /dev/sda1
mkswap /dev/sdb3
```

Choose the filesystem based on the task:

- **XFS**: common default on RHEL for general-purpose Linux filesystems
- **ext4**: widely supported and common in many environments
- **vfat**: used for EFI System Partitions
- **swap**: used for swap space, not mounted like a normal filesystem

After formatting swap, enable it:

```bash
swapon /dev/sdb3
```

Persist swap in `/etc/fstab` with its UUID:

```bash
blkid /dev/sdb3
```

Example entry:

```fstab
UUID=5678-EFGH  none  swap  defaults  0  0
```

**Practical RHCSA workflow**

A typical storage task on the exam follows a predictable sequence.

1. Identify the correct disk with `lsblk`, `blkid`, or `fdisk -l`.
2. Create or modify the partition table if needed.
3. Create the partition with the correct start and end boundaries.
4. Format the partition with the required filesystem.
5. Mount it and verify access.
6. Add a persistent `/etc/fstab` entry if the mount must survive reboot.

Example:

```bash
lsblk
parted /dev/sdb --script mklabel gpt
parted /dev/sdb --script mkpart primary 1MiB 5GiB
mkfs.xfs /dev/sdb1
mkdir -p /srv/data
mount /dev/sdb1 /srv/data
blkid /dev/sdb1
```

Then add the UUID to `/etc/fstab`:

```fstab
UUID=1234-ABCD  /srv/data  xfs  defaults  0  0
```

Verify the mount:

```bash
mount | grep /srv/data
df -hT /srv/data
```

**Alignment and start/end boundaries**

Modern partitioning tools usually align partitions to MiB boundaries for performance. Starting at `1MiB` is common because it avoids overwriting early disk metadata and aligns well with many storage devices.

Good practice:

- start partitions at `1MiB` or another aligned boundary
- leave room for boot metadata when needed
- size partitions according to the role, not just by convenience

Example:

```bash
parted /dev/sdb --script mkpart primary 1MiB 1025MiB
```

This creates a roughly 1 GiB partition. The exact size depends on the tool’s interpretation of boundaries, so always verify with `lsblk` or `parted print`.

**Verification and troubleshooting**

After partitioning, confirm that the kernel sees the new layout.

```bash
partprobe
udevadm settle
lsblk
parted /dev/sdb print
```

If the new partition does not appear immediately, the kernel may need a rescan. `partprobe` asks the kernel to reread the partition table. In some cases, unplugging and replugging a virtual disk or rebooting may be necessary, but that is usually a last resort.

Common issues:

- **Wrong disk selected**: always verify the device name before writing changes.
- **Partition exists but no filesystem**: create the filesystem with `mkfs`.
- **Filesystem exists but is not mounted**: mount it manually or fix `/etc/fstab`.
- **`/etc/fstab` entry fails**: check UUID, filesystem type, and mount point spelling.
- **UEFI boot partition confusion**: use `esp` for the EFI System Partition on GPT, not a generic `boot` assumption.

**Example comparison of layers**

| Layer | Example | Purpose |
|---|---|---|
| Disk | `/dev/sda` | Physical or virtual storage device |
| Partition table | GPT | Describes partition layout |
| Partition | `/dev/sda1` | Allocated region on the disk |
| Filesystem | `xfs` | Organizes files and directories |
| Mount point | `/data` | Directory where the filesystem is accessed |

Understanding these layers helps you avoid common exam mistakes. If a task says “create a partition and mount it,” you must do both the partitioning step and the filesystem step before mounting.

**Quick reference commands**

```bash
lsblk -f
blkid
fdisk -l
parted -l
parted /dev/sdb print
parted /dev/sdb mklabel gpt
parted /dev/sdb mkpart primary 1MiB 2GiB
mkfs.xfs /dev/sdb1
mount /dev/sdb1 /mnt
```

**Key takeaways**

- A partition table defines disk layout; a partition is only a slice of the disk.
- Formatting is a separate step from partition creation.
- GPT is the modern default; MBR remains relevant for legacy compatibility.
- `parted` is a strong choice for RHCSA partitioning tasks, especially scripted ones.
- Use `lsblk`, `blkid`, and `parted print` to verify results before and after changes.
- For UEFI booting on GPT, the EFI System Partition uses the `esp` flag and a FAT filesystem.
