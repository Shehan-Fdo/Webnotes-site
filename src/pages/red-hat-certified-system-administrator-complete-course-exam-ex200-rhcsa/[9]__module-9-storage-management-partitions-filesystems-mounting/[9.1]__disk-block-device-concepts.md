## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Disk, block device, and storage model**

Linux storage troubleshooting starts with a clear mental model: a physical or virtual disk is not the same thing as a partition, filesystem, or mount point. RHCSA tasks often require you to identify which layer is broken before you change anything.

A **block device** is a device that reads and writes data in fixed-size blocks. Disks such as `/dev/sda`, NVMe namespaces such as `/dev/nvme0n1`, and virtual disks presented by a hypervisor are all block devices. Linux exposes them as special files under `/dev`.

A **partition** is a logical subdivision of a disk. Partitions are also block devices, but they represent a slice of the parent disk, such as `/dev/sda1` or `/dev/nvme0n1p1`.

A **filesystem** is the on-disk structure that stores files and directories, such as `xfs`, `ext4`, or `vfat`. A filesystem can exist on a whole disk, a partition, a logical volume, or another block device.

A **mount** connects a filesystem to the directory tree at a mount point such as `/data` or `/home`. Mounting does not create data; it makes existing data accessible.

- **Block device**: raw storage interface exposed by the kernel.
- **Partition**: a block device carved out of a disk.
- **Filesystem**: the format that organizes files on a block device.
- **Mount point**: the directory where a filesystem becomes visible.

> If you confuse these layers during an exam, you may format the wrong object or troubleshoot the wrong problem. Always identify the device type first.

**How Linux names storage devices**

Device names vary by bus and driver, so you should recognize the common patterns quickly.

- **SATA/SCSI/SAS disks** often appear as `/dev/sda`, `/dev/sdb`, and so on.
- **NVMe** devices appear as `/dev/nvme0n1`, with partitions like `/dev/nvme0n1p1`.
- **Virtual disks** in KVM or cloud environments may still appear as SCSI-style names or as NVMe devices depending on the platform.
- **Loop devices** such as `/dev/loop0` represent files mapped as block devices.
- **Multipath devices** may appear as `/dev/mapper/mpatha` or similar when multiple paths lead to the same storage.
- **Removable media** such as USB drives or optical media may appear as `/dev/sdb`, `/dev/sr0`, or another device name depending on what is already present.

The name alone does not tell you whether the device contains a partition table, a filesystem, or both. You must inspect it.

**Device nodes, `/dev`, and `/dev/mapper`**

Linux represents devices as special files in `/dev`. These files are not regular files; they are interfaces to kernel drivers.

`/dev/mapper` is especially important for RHCSA storage work because it is where **device-mapper** devices appear. Device-mapper is the kernel framework used by technologies such as LVM, multipath, and some encrypted storage setups.

- **Device-mapper device**: a virtual block device created by the kernel from one or more underlying block devices.
- **LVM logical volume**: appears as a device-mapper node under `/dev/mapper`, for example `/dev/mapper/rhel-root` or `/dev/mapper/vgdata-lvdata`.
- **Relationship**: the logical volume is a block device presented by device-mapper, and it is backed by physical volumes on real disks or partitions.

Example:

```bash
ls -l /dev/mapper
```

Typical output may include symlinks such as:

```text
lrwxrwxrwx. 1 root root 7 Apr 26 10:00 rhel-root -> ../dm-0
lrwxrwxrwx. 1 root root 7 Apr 26 10:00 rhel-swap -> ../dm-1
lrwxrwxrwx. 1 root root 7 Apr 26 10:00 vgdata-lvdata -> ../dm-2
```

Here, `/dev/mapper/vgdata-lvdata` is a friendly name for a device-mapper node, and `/dev/dm-2` is the underlying kernel device node. The logical volume is still a block device, so you can place a filesystem on it, mount it, or inspect it with block-device tools.

**Partition tables and filesystem signatures**

A disk can contain a **partition table**, a **filesystem signature**, both, or neither.

- A **partition table** describes how the disk is divided into partitions.
- A **filesystem signature** identifies the filesystem type stored on a block device.
- A disk may have a partition table but no filesystem yet.
- A partition may have a filesystem but no partition table on the parent disk if the filesystem is written directly to the whole disk.

This distinction matters in troubleshooting. For example, a disk may show up in `lsblk`, but if it has no filesystem signature, it cannot be mounted yet. Conversely, a device may have a filesystem signature but no mount entry, which means it is not currently attached to the directory tree.

Useful inspection commands:

```bash
lsblk -f
blkid
file -s /dev/sdb
fdisk -l /dev/sdb
parted -l
```

What to look for:

- `lsblk -f` shows device hierarchy, filesystem type, UUID, and mount point.
- `blkid` reports filesystem and swap signatures by device.
- `file -s` can help identify unknown signatures on a raw block device.
- `fdisk -l` shows partition tables and partition boundaries.
- `parted -l` can show partition tables and disk layout, including GPT or MBR.

Example troubleshooting scenario:

```bash
lsblk -f
```

Possible output:

```text
NAME   FSTYPE FSVER LABEL UUID                                 MOUNTPOINT
sda
├─sda1 xfs          boot  1111-2222                            /boot
└─sda2 LVM2_member        3333-4444
  ├─rhel-root xfs         5555-6666                            /
  └─rhel-swap swap        7777-8888                            [SWAP]
sdb
```

Interpretation:

- `/dev/sda` is the disk.
- `/dev/sda1` is a partition with an `xfs` filesystem.
- `/dev/sda2` is a partition used as an LVM physical volume, not a mountable filesystem.
- `/dev/mapper/rhel-root` is a logical volume with an `xfs` filesystem mounted at `/`.
- `/dev/sdb` has no filesystem signature yet, so it is likely unused or uninitialized.

**Block devices versus filesystems in exam troubleshooting**

When a mount fails, you should determine whether the problem is at the device, partition, filesystem, or mount layer.

- If the device is missing, check hardware detection, virtualization settings, or pathing.
- If the partition exists but the filesystem is absent, create the filesystem before mounting.
- If the filesystem exists but the mount fails, check the mount point, UUID, `/etc/fstab`, and filesystem type.
- If the device is present but not visible where expected, verify whether it is mounted elsewhere or whether you are looking at the wrong device name.

A practical decision flow:

1. Confirm the block device exists with `lsblk`.
2. Check whether the device has a partition table with `fdisk -l` or `parted -l`.
3. Check for a filesystem signature with `blkid` or `lsblk -f`.
4. Mount the correct filesystem to the correct directory.
5. Verify the result with `findmnt` or `mount`.

Example:

```bash
lsblk -f /dev/sdb
blkid /dev/sdb
file -s /dev/sdb
```

If `file -s` reports `data`, the device likely has no recognizable filesystem signature. If it reports `XFS filesystem data`, the filesystem exists even if it is not mounted.

**Common storage device types you may encounter**

RHCSA environments can include more than simple disks and partitions.

- **Loop devices**: files attached as block devices, often used for testing or image files.
- **Multipath devices**: a single logical disk presented through multiple physical paths for redundancy.
- **Removable media**: USB storage or optical media, which may appear and disappear depending on insertion.
- **LVM logical volumes**: virtual block devices created from one or more physical volumes.
- **Encrypted devices**: often exposed through device-mapper after unlocking, such as `/dev/mapper/luks-...`.

You do not need to memorize every possible naming scheme, but you should recognize that the same storage stack can be layered:

```text
Disk -> Partition -> LVM physical volume -> Volume group -> Logical volume -> Filesystem -> Mount point
```

Or, in some cases:

```text
Disk -> Filesystem -> Mount point
```

The second pattern is valid when a filesystem is created directly on a whole disk or on a logical volume.

**LVM and device-mapper relationships**

LVM uses device-mapper to present logical volumes as block devices. This is a common exam topic because it combines storage concepts with device identification.

- **Physical volume (PV)**: a disk or partition initialized for LVM.
- **Volume group (VG)**: a pool of storage made from one or more PVs.
- **Logical volume (LV)**: a virtual block device carved from a VG.
- **Device-mapper node**: the kernel representation of the LV under `/dev/dm-*` and `/dev/mapper/*`.

Example:

```bash
pvs
vgs
lvs
ls -l /dev/mapper
```

Typical interpretation:

- `pvs` shows which disks or partitions are LVM physical volumes.
- `vgs` shows the storage pool.
- `lvs` shows logical volumes and their sizes.
- `/dev/mapper/vgname-lvname` is the usable block device path for the logical volume.

You may also see the same LV through `/dev/<vgname>/<lvname>` if the system creates that symlink. Both paths refer to the same underlying device-mapper block device.

> Do not format the PV itself with a filesystem if it is intended for LVM. The filesystem belongs on the logical volume, not on the physical volume.

**Inspecting storage with command-line tools**

RHCSA expects you to use standard tools efficiently, and you should know how to consult help pages when needed.

- Use `man` pages for full command syntax and options.
- Use `--help` for quick usage summaries.
- Use `man lsblk`, `man blkid`, `man mount`, `man fdisk`, and `man parted` when you need exact behavior.

Examples:

```bash
man lsblk
man blkid
mount --help
fdisk --help
```

Useful commands and what they tell you:

- `lsblk`: tree view of block devices and mount points.
- `blkid`: UUIDs and filesystem signatures.
- `findmnt`: current mount relationships.
- `df -h`: mounted filesystem usage.
- `mount`: attach a filesystem.
- `umount`: detach a filesystem.
- `fdisk` or `parted`: partition table inspection and editing.
- `wipefs`: view or remove filesystem signatures carefully.

**Exam-like troubleshooting examples**

A device appears in `lsblk`, but mounting fails:

```bash
lsblk -f /dev/sdb
mount /dev/sdb /mnt
```

If `lsblk -f` shows no `FSTYPE`, the device is not formatted. Create a filesystem first:

```bash
mkfs.xfs /dev/sdb
mount /dev/sdb /mnt
```

A partition exists, but the wrong filesystem type is used in `/etc/fstab`:

```bash
blkid /dev/sdb1
cat /etc/fstab
```

If the device is `xfs` but `/etc/fstab` says `ext4`, the mount will fail. Match the filesystem type to the actual signature.

A logical volume exists, but you are looking for it on the wrong path:

```bash
lvs
ls -l /dev/mapper
lsblk
```

If the LV is `vgdata/lvapp`, the usable device may be `/dev/mapper/vgdata-lvapp` or `/dev/vgdata/lvapp`. Use the device-mapper path if needed.

A disk has leftover signatures from previous use:

```bash
wipefs -n /dev/sdb
```

This shows existing signatures without erasing them. It is useful when a disk behaves unexpectedly because old metadata remains.

**Practical identification workflow**

Use a consistent workflow when you inspect storage on the exam.

1. Identify the device with `lsblk`.
2. Determine whether it is a disk, partition, LVM LV, or other block device.
3. Check for partition tables with `fdisk -l` or `parted -l`.
4. Check for filesystem signatures with `blkid` or `file -s`.
5. Confirm the mount state with `findmnt` or `mount`.
6. Use `man` pages if a command’s output is unclear.

Example workflow:

```bash
lsblk -f
blkid
findmnt
```

This gives you a fast view of what exists, what is formatted, and what is mounted.

**Key exam takeaways**

- A **disk** is the storage device; a **partition** is a slice of it.
- A **filesystem** is the data structure that stores files.
- A **mount** makes a filesystem available in the directory tree.
- LVM logical volumes appear as **device-mapper** devices under `/dev/mapper`.
- `lsblk`, `blkid`, `fdisk`, `parted`, `findmnt`, and `wipefs` are core inspection tools.
- Always verify whether you are dealing with a raw disk, a partition, or a logical volume before formatting or mounting.
- Use `man` and `--help` to confirm command syntax during troubleshooting.
