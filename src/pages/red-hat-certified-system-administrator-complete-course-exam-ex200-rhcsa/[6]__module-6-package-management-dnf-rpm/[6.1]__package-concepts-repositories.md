## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Package management fundamentals**

On RHEL, package management centers on **RPM** (Red Hat Package Manager) files and **DNF** (Dandified YUM) as the high-level tool for installing, updating, and resolving dependencies. RPM handles individual package files and metadata, while DNF resolves dependencies from repositories and manages transactions.

A package name usually follows the pattern `name-version-release.arch`. For example, `bash-5.1.8-6.el9.x86_64` tells you:

- **name**: the software package
- **version**: upstream version
- **release**: Red Hat build iteration
- **arch**: target architecture such as `x86_64`, `noarch`, or `aarch64`

Architecture suffixes matter during troubleshooting:

- `x86_64` packages are built for 64-bit Intel/AMD systems
- `noarch` packages are architecture-independent, often scripts or documentation
- `src` packages are source RPMs and are not installed like normal binaries

RPM databases track installed packages locally, while repositories provide package metadata and content over HTTP, HTTPS, FTP, or local file paths. DNF uses repository metadata to determine what can be installed and which dependencies are required.

**Core RPM operations**

Use RPM when you need to query, verify, or install a local package file directly. DNF is usually preferred for dependency resolution, but RPM remains important for inspection and low-level package tasks.

Common query commands:

```bash
rpm -q bash
rpm -qi bash
rpm -ql bash
rpm -qc bash
rpm -qd bash
rpm -qf /bin/bash
rpm -qa | grep '^bash'
```

What these do:

- `rpm -q` queries whether a package is installed
- `rpm -qi` shows package information
- `rpm -ql` lists installed files
- `rpm -qc` lists configuration files
- `rpm -qd` lists documentation files
- `rpm -qf` identifies which package owns a file
- `rpm -qa` lists all installed packages

Install, upgrade, verify, and remove with RPM:

```bash
sudo rpm -ivh ./package.rpm
sudo rpm -Uvh ./package.rpm
sudo rpm -V bash
sudo rpm -e bash
```

Use these carefully:

- `-i` installs a package if it is not already present
- `-U` upgrades or installs a package
- `-V` verifies installed files against package metadata
- `-e` erases a package

Verification checks file size, permissions, checksum, owner, group, and modification time. It helps detect tampering or accidental changes, especially for critical system binaries and configuration files.

> `rpm -e` fails if another installed package depends on the target package. DNF is better for dependency-aware removal.

**DNF package operations**

DNF is the standard tool for package installation and lifecycle management on RHEL 8/9. It resolves dependencies, consults enabled repositories, and records transaction history.

Typical tasks:

```bash
sudo dnf install httpd
sudo dnf update
sudo dnf remove httpd
sudo dnf reinstall httpd
sudo dnf downgrade httpd
sudo dnf list installed
sudo dnf list available
sudo dnf info httpd
```

Important behaviors:

- `install` adds a package and dependencies
- `update` refreshes installed packages to newer available versions
- `remove` erases a package and unneeded dependencies
- `reinstall` replaces files from the same version
- `downgrade` moves to an older available version if the repository provides it
- `list installed` and `list available` help you compare system state with repository content

You can also install by package file path:

```bash
sudo dnf install ./example.rpm
```

This still uses dependency resolution, unlike raw `rpm -ivh`.

**Repository concepts**

A repository is a source of package metadata and RPM content. DNF reads repository definitions from `.repo` files, usually in `/etc/yum.repos.d/`. Each repo file can define one or more repositories with settings such as:

- `name`
- `baseurl`
- `enabled`
- `gpgcheck`
- `gpgkey`
- `metadata_expire`

Example repository file:

```ini
[appstream]
name=RHEL AppStream
baseurl=https://example.invalid/rhel/9/AppStream/x86_64/os/
enabled=1
gpgcheck=1
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-redhat-release
```

Key points:

- `baseurl` points to the repository root that contains `repodata/`
- `enabled=1` makes the repo active
- `gpgcheck=1` enforces package signature verification
- `gpgkey` points to the public key used to validate packages

Temporary repo control is useful for one-off tasks:

```bash
sudo dnf install --enablerepo=appstream --disablerepo='*' vim
```

This is different from permanently editing repo files. Use temporary flags when you want a single transaction to use a specific repository without changing system configuration.

**Repository listing and troubleshooting**

Use DNF to inspect repository status:

```bash
dnf repolist
dnf repolist -v
dnf repolist --enabled
dnf repolist --disabled
```

`dnf repolist -v` is especially useful because it shows repo IDs, base URLs, metadata status, and other details.

If `subscription-manager` is installed and the system is registered, it can also help you inspect entitlement-based repositories:

```bash
subscription-manager status
subscription-manager repos --list-enabled
subscription-manager repos --list-disabled
```

Use these commands only when the system is registered and the tool is available. On unregistered systems, repository access may come from local `.repo` files instead.

Common repository failure symptoms and likely causes:

| Symptom | Likely cause | Fix |
|---|---|---|
| `Cannot find a valid baseurl` | Wrong `baseurl`, DNS, proxy, or no network | Check the URL, connectivity, and repo file syntax |
| `Failed to download metadata` | Repo unreachable or metadata stale | Verify access, then run `dnf clean all` and retry |
| `GPG check FAILED` | Wrong or missing public key | Import the correct key and verify `gpgcheck` settings |
| Package not found | Repo disabled or wrong module stream | Enable the correct repo or module stream |
| Conflicting requests` | Modular package masking or dependency mismatch | Check module streams and available package versions |

A broken repo file often causes DNF to fail before package resolution begins. Check for:

- malformed INI syntax
- duplicate repo IDs
- incorrect `baseurl`
- unreachable mirror or proxy
- missing `gpgkey` path

Useful cleanup commands:

```bash
sudo dnf clean all
sudo rm -rf /var/cache/dnf/*
sudo dnf makecache
```

**Configuring repositories permanently**

On RHEL 8/9, `dnf config-manager` comes from `dnf-plugins-core`, so do not assume it is installed on a minimal system. If available, it helps enable, disable, or edit repositories.

Examples:

```bash
sudo dnf install dnf-plugins-core
sudo dnf config-manager --set-enabled appstream
sudo dnf config-manager --set-disabled appstream
sudo dnf config-manager --add-repo https://example.invalid/custom.repo
```

For RHCSA-style tasks, you should also know how to edit repo files directly with a text editor:

```bash
sudo vi /etc/yum.repos.d/custom.repo
```

Direct file editing is often the most reliable exam approach because it does not depend on extra plugins.

**GPG keys and package trust**

GPG (GNU Privacy Guard) signatures help confirm that packages come from a trusted source and have not been altered. When `gpgcheck=1`, DNF verifies package signatures against imported public keys.

Typical workflow:

1. Identify the correct key file on the installation media or repository.
2. Import the key with RPM.
3. Verify the key fingerprint if possible.
4. Retry the package operation.

Example import:

```bash
sudo rpm --import /etc/pki/rpm-gpg/RPM-GPG-KEY-redhat-release
```

To inspect imported keys, query the `gpg-pubkey` packages:

```bash
rpm -qa 'gpg-pubkey*'
rpm -qi gpg-pubkey-*
```

The exact package identifier includes a key ID and timestamp-like release values, so do not expect a normal package name.

If you need to verify a key fingerprint, compare the output against trusted documentation or the installation media metadata:

```bash
rpm -qi gpg-pubkey-<keyid>
```

> Import only keys you trust. A valid signature from the wrong key does not prove the package is from Red Hat or your intended repository.

**Offline and local repositories**

Offline repositories are common in labs and exam scenarios. You may need to install packages from mounted installation media or from a copied repository tree on disk.

A local repository must contain valid repository metadata under a `repodata/` directory. If you copy packages manually, create metadata with `createrepo_c`:

```bash
sudo dnf install createrepo_c
sudo mkdir -p /srv/repo/custom
sudo cp *.rpm /srv/repo/custom/
sudo createrepo_c /srv/repo/custom
```

Then define a local repo:

```ini
[custom]
name=Custom Local Repo
baseurl=file:///srv/repo/custom
enabled=1
gpgcheck=0
```

If you want signature checking, set `gpgcheck=1` and provide a valid `gpgkey=` path.

For installation media, the exact directory layout depends on the media version and type. Point `baseurl` at the directory that actually contains `repodata/`, not at an assumed path. A mounted ISO might look like this:

```bash
mount /dev/sr0 /mnt
find /mnt -name repodata -type d
```

Then use the discovered path in the repo file:

```ini
[baseos]
name=BaseOS from ISO
baseurl=file:///mnt/BaseOS/x86_64/os/
enabled=1
gpgcheck=1
```

If the media layout differs, adjust the path accordingly. The important rule is that DNF must reach the repository root that contains `repodata/`.

**Modules and streams**

RHEL modularity lets you choose among different versions of a software stack, called **streams**. A module can also expose **profiles**, which are predefined package sets such as `common`, `default`, or `development` depending on the module.

Inspect modules with:

```bash
dnf module list
dnf module list nodejs
dnf module info nodejs
dnf module list --enabled
dnf module list --installed
```

Module actions:

```bash
sudo dnf module enable nodejs:18
sudo dnf module install nodejs:18/default
sudo dnf module reset nodejs
sudo dnf module remove nodejs
```

Important distinctions:

- `enable` selects a stream for future package visibility
- `install` installs packages from a stream, often via a profile
- `reset` clears the selected stream state
- `remove` removes installed modular packages

A stream being enabled does not install anything by itself. It only changes which versions DNF considers valid for modular packages.

Modular packages can mask non-modular packages. In practice, this means:

- a module stream may hide a newer or older non-modular package version
- DNF may report conflicts when a package exists in both modular and non-modular forms
- resetting or disabling a module can restore visibility to non-modular packages

Example troubleshooting flow:

```bash
dnf module list php
dnf module reset php
dnf module enable php:8.1
dnf install php
```

If a package is unexpectedly unavailable, check whether a module stream is active before assuming the repository is broken.

**History, rollback, and transaction control**

DNF records transaction history, which helps you review and sometimes reverse package actions.

Useful commands:

```bash
dnf history
dnf history list
dnf history info 12
dnf history undo 12
dnf history rollback 12
```

Use these carefully:

- `history info <ID>` shows details for a specific transaction ID
- `undo <ID>` reverses a transaction when possible
- `rollback <ID>` attempts to return to the state before that transaction

Rollback is not guaranteed to restore an exact prior state. It depends on package availability, repository content, and later transactions that may interfere with reversal.

A practical workflow is:

1. Run `dnf history`.
2. Identify the transaction ID.
3. Inspect it with `dnf history info <ID>`.
4. Use `undo` or `rollback` only after confirming the packages are still available.

**Package locking and version control**

Sometimes you need to prevent a package from changing during updates. DNF supports version locking through the `versionlock` plugin, which is provided by `dnf-plugins-core` on many systems.

Example:

```bash
sudo dnf install dnf-plugins-core
sudo dnf versionlock add kernel
sudo dnf versionlock list
sudo dnf versionlock delete kernel
```

Use version locking when a specific package version must remain stable for compatibility or troubleshooting. Remember that locks can block security updates, so remove them when they are no longer needed.

**Practical RHCSA workflows**

A common exam task is to install a package from a specific repository while avoiding others:

```bash
sudo dnf install --disablerepo='*' --enablerepo=baseos vim
```

Another common task is to fix a broken repository definition:

1. Open the repo file in `/etc/yum.repos.d/`.
2. Correct the `baseurl`, `enabled`, and `gpgcheck` values.
3. Clear cached metadata.
4. Retry the install.

```bash
sudo vi /etc/yum.repos.d/broken.repo
sudo dnf clean all
sudo dnf makecache
sudo dnf install httpd
```

If a package install fails because the repo is unreachable, test the URL directly:

```bash
curl -I https://example.invalid/rhel/9/BaseOS/x86_64/os/repodata/repomd.xml
```

If the file is local, confirm the mount and path:

```bash
mount | grep /mnt
ls -l /mnt
find /mnt -name repomd.xml
```

If a package is already installed but you need to confirm its integrity:

```bash
rpm -V bash
```

If a file belongs to an unexpected package:

```bash
rpm -qf /usr/bin/ssh
```

**Quick comparison of RPM and DNF**

| Task | RPM | DNF |
|---|---|---|
| Query installed package | `rpm -q` | `dnf list installed` |
| Show package details | `rpm -qi` | `dnf info` |
| Install local file | `rpm -ivh` | `dnf install ./pkg.rpm` |
| Remove package | `rpm -e` | `dnf remove` |
| Verify files | `rpm -V` | Not primary tool |
| Resolve dependencies | Limited | Full dependency resolution |

Use RPM for inspection and low-level control. Use DNF for normal package lifecycle management.

**Exam-focused takeaways**

- Know the difference between a package file, an installed package, and a repository source.
- Use `rpm` for queries and verification, and `dnf` for dependency-aware installs and updates.
- Understand repository files in `/etc/yum.repos.d/` and how `baseurl`, `enabled`, and `gpgcheck` work.
- Remember that `dnf config-manager` may require `dnf-plugins-core`.
- Treat module streams as a visibility and version-selection mechanism, not an install by itself.
- Use `dnf history info <ID>` with a real transaction ID, and treat rollback as conditional, not guaranteed.
- For offline work, ensure the repository root contains `repodata/` or generate it with `createrepo_c`.
