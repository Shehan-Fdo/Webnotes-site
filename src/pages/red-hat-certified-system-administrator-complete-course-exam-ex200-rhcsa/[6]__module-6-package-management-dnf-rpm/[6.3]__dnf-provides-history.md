## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

`dnf` is the Red Hat package manager used on modern RHEL-based systems to install, remove, update, and inspect software packages. Two especially useful subcommands for administration are `dnf provides` and `dnf history`.

- `dnf provides` helps you identify which package owns a file, command, or capability.
- `dnf history` lets you audit and manage package transactions after they happen.
- Together, they help you troubleshoot missing commands, verify package provenance, and recover from bad updates.

**DNF provides: find the package that supplies a file or command**

Use `dnf provides` when you know *what* you need but not *which package* contains it. This is common when a command is missing, a library file is referenced in an error, or you need to determine the package that should be installed.

Typical uses include:

- Finding the package that installs a command such as `semanage` or `ss`.
- Identifying the package that owns a shared library like `libssl.so.3`.
- Discovering which package provides a configuration file or binary path.

A common pattern is to search by filename or path:

```bash
dnf provides /usr/sbin/semanage
dnf provides /usr/bin/ss
dnf provides '*/libssl.so.3'
```

You can also search by capability or package name fragments, but file-path searches are the most practical for RHCSA tasks. If the output is noisy, narrow the query with the exact path or wildcard pattern.

- **Path matching** is often the fastest way to locate the owning package.
- **Wildcards** help when you only know part of the filename.
- **Installed vs available** packages may both appear in results, so read the package name and repository carefully.

A typical result shows the package name, version, repository, and a short description. Once you identify the package, install it with `dnf install <package>`.

```bash
dnf provides /usr/sbin/semanage
dnf install policycoreutils-python-utils
```

> `dnf provides` searches repository metadata, so the package may not be installed yet. If the package is unavailable, verify that the correct repositories are enabled.

**Practical workflow for missing commands**

When a command is missing, follow a simple workflow:

1. Confirm the command is absent or not in your `PATH`.
2. Use `dnf provides` with the full path or command name.
3. Install the package that owns the command.
4. Re-run the command to verify it is available.

Example:

```bash
command -v semanage
dnf provides /usr/sbin/semanage
dnf install policycoreutils-python-utils
semanage --help
```

This approach is useful because many administrative tools are split into separate packages. The command name alone may not be enough; the full path often gives a cleaner match.

**DNF history: audit and manage package transactions**

`dnf history` shows a record of package transactions performed on the system. This is valuable for troubleshooting because it tells you what changed, when it changed, and which packages were affected.

Use it to:

- Review recent installs, updates, removals, and downgrades.
- Identify the transaction that introduced a problem.
- Undo or redo a transaction when appropriate.
- Compare package state before and after a change.

Basic history commands:

```bash
dnf history
dnf history list
dnf history info 12
```

The transaction list typically includes:

- Transaction ID
- Date and time
- Action type
- Number of packages affected
- Whether the transaction completed successfully

`dnf history info <ID>` gives detailed information about a specific transaction, including the packages involved and the action taken on each package.

**Common history actions**

`dnf history` supports several useful operations:

```bash
dnf history undo <ID>
dnf history redo <ID>
dnf history rollback <ID>
```

- **undo** reverses the selected transaction when possible.
- **redo** repeats a transaction.
- **rollback** returns the system to the state before a given transaction, if the history allows it.

Use these carefully. Package state changes can affect dependencies, services, and configuration files. Always review the transaction details before reversing it.

> `undo` and `rollback` are not magic recovery tools. If configuration files were edited manually after the transaction, DNF may not restore your exact system state.

**Reading transaction details**

`dnf history info` is the best starting point when you need to understand what happened. It can show:

- Packages installed or removed
- Packages upgraded or downgraded
- The command that triggered the transaction
- The repository used
- Whether the transaction succeeded or failed

Example:

```bash
dnf history info 12
```

If you suspect a recent update caused a service failure, inspect the latest transactions first:

```bash
dnf history
dnf history info last
```

This helps you correlate package changes with service behavior, logs, or configuration issues.

**Troubleshooting with DNF history**

A practical troubleshooting pattern is to combine `dnf history` with service and log checks.

1. Check recent package transactions.
2. Identify the package that changed.
3. Review the service status and logs.
4. Decide whether to undo, downgrade, or reinstall.

Example workflow:

```bash
dnf history
dnf history info last
systemctl status httpd
journalctl -u httpd --since "1 hour ago"
```

If a package update broke a service, you may be able to revert the transaction:

```bash
dnf history undo last
```

If the issue is caused by a missing package, use `dnf provides` to find and install the correct provider.

**Useful command patterns**

- Find the package that owns a file:
  ```bash
  dnf provides /path/to/file
  ```
- Find the package that supplies a command:
  ```bash
  dnf provides /usr/bin/command
  ```
- Review all transactions:
  ```bash
  dnf history
  ```
- Inspect one transaction:
  ```bash
  dnf history info <ID>
  ```
- Reverse the last transaction:
  ```bash
  dnf history undo last
  ```

**Best practices**

- Use the full file path with `dnf provides` for the most accurate results.
- Check whether the package is already installed before installing it again.
- Review `dnf history info` before using `undo` or `rollback`.
- Keep in mind that package transactions do not always capture manual configuration changes.
- Use history records as part of a broader troubleshooting process, not as the only source of truth.

**Exam-focused takeaways**

For RHCSA-style tasks, you should be able to:

- Identify which package provides a missing command or file.
- Install the correct package using `dnf`.
- Inspect recent package transactions with `dnf history`.
- Explain what changed in a transaction using `dnf history info`.
- Safely revert a recent package change when needed.

These commands are especially useful in timed environments because they reduce guesswork. Instead of searching manually through package names, you can ask the package manager directly which package provides the needed file and what changed on the system recently.
