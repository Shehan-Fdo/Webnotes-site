## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Package groups and modularity overview**

DNF (Dandified YUM) manages software on RHEL by installing individual packages, package groups, and modular content. Package groups help you install a related set of packages for a task, while modularity lets Red Hat ship multiple versions of the same software in parallel through module streams.

Use package groups when you want a role-based bundle, such as a workstation or server toolset. Use modules when you need a specific major version of software, such as a particular language runtime, database, or web stack component.

- **Package groups** are collections of RPM packages managed as a unit.
- **Modules** organize software into streams, profiles, and artifacts.
- **Streams** let multiple major versions coexist in repositories.
- **Profiles** are predefined package sets for a module stream.
- **Modular filtering** hides non-selected package versions when a module is active.

**DNF package group basics**

A package group is a named collection of packages defined in repository metadata. Groups are useful when you want to install a common set of tools without selecting each package manually.

Common group operations include listing, inspecting, installing, removing, and marking group state.

```bash
dnf group list
dnf group info "Development Tools"
dnf group install "Development Tools"
dnf group remove "Development Tools"
dnf group mark install "Development Tools"
dnf group mark remove "Development Tools"
```

- **Group names** may contain spaces and can vary by repository or locale.
- **Group IDs** are more stable than display names when names differ across systems.
- **Quoted names** are required when the group name contains spaces.
- **Marking** a group records intent without necessarily changing the installed package set immediately.

When a group name is ambiguous or localized, use the group ID if available. This is especially helpful in automation and exam environments where display names may differ.

```bash
dnf group list --ids
dnf group info "development-tools"
```

**Group names vs group IDs**

Group names are human-readable labels. Group IDs are machine-oriented identifiers that are less likely to change across languages or minor metadata differences.

| Aspect | Group name | Group ID |
|---|---|---|
| Readability | Human-friendly | Machine-friendly |
| Stability | Can vary by locale or repo metadata | Usually more stable |
| CLI usage | Often quoted | Often safer for scripts |
| Example | `"Development Tools"` | `development-tools` |

Use group IDs when:
- The name contains spaces or punctuation.
- You are writing scripts.
- You want to avoid locale-related mismatches.

**Group install and removal behavior**

Installing a group pulls in the packages defined by the group metadata, but the exact result depends on repository content and package availability. Removing a group reverses the group selection, but it does not guarantee removal of every package that was ever installed while the group was active.

- **Group install** installs the selected package set for the group.
- **Group remove** removes packages associated with the group selection.
- **Group mark install** records the group as installed.
- **Group mark remove** clears the installed state for the group.
- **Group info** shows mandatory, default, and optional packages.

A practical workflow is to inspect first, then install only if the group matches the task.

```bash
dnf group info "Development Tools"
dnf group install "Development Tools"
rpm -q gcc make autoconf
```

> `dnf group remove` is not a perfect cleanup tool for every package that may have been pulled in indirectly. Verify with `rpm -q` if package-level removal matters.

**Module basics and why they matter**

Modules let Red Hat publish multiple versions of a software stack in the same repository. This is common for languages and server components where major versions are not compatible with each other.

A module has these core concepts:
- **Module name**: the software family, such as a language or service.
- **Stream**: the version track, such as a major release line.
- **Profile**: a predefined package set for a stream.
- **Artifacts**: the RPM packages provided by the module stream.

Modules are useful when the default repository package version is not the one you need. Instead of replacing the system package set, you select a stream that matches your application requirements.

**Enable a stream vs install a profile**

These are related but not identical actions.

- **Enabling a stream** makes that stream available for package selection.
- **Installing a profile** installs the package set defined by that stream’s profile.
- Some workflows combine both in one command.
- Some modules require you to enable the stream first, then install the profile separately.

Typical patterns:

```bash
dnf module list nodejs
dnf module enable nodejs:18
dnf module install nodejs
```

Or, when a profile is explicitly needed and available:

```bash
dnf module install nodejs:18/default
```

Use the module metadata on the target system to confirm the exact stream and profile names. Do not assume a profile exists just because a module exists.

- **Enable** when you want the stream available but are not ready to install packages yet.
- **Install** when you want the stream’s packages and profile content on the system.
- **Profile selection** matters when the module offers multiple package sets.

**Module listing and inspection**

Before changing anything, inspect available and installed module metadata.

```bash
dnf module list
dnf module list --enabled
dnf module list --installed
dnf module info nodejs
```

- **`dnf module list`** shows available streams and their status.
- **`--enabled`** shows streams currently enabled.
- **`--installed`** shows streams with installed modular content or installed state.
- **`dnf module info`** shows stream details, profiles, and artifacts.

> `dnf module list --installed` helps you see installed module streams, but it does not by itself prove which individual RPMs are present. Use `rpm -q` or `dnf module info` for package-level verification.

**Module install workflows**

The exact command depends on the module metadata and the RHEL release, but the general workflow is consistent: inspect, enable the stream if needed, then install the desired profile or module content.

Example workflow:

```bash
dnf module list postgresql
dnf module enable postgresql:15
dnf module install postgresql
rpm -q postgresql-server
```

If the module provides a profile you need, install that profile explicitly after confirming it exists.

```bash
dnf module info postgresql
dnf module install postgresql:15/<profile-name>
```

Use the actual profile name from `dnf module info`. Avoid assuming a profile such as `server` or `common` exists on every release.

**Reset, disable, and stream switching**

`dnf module reset` and `dnf module disable` are not the same.

- **Reset** clears the module’s enabled/disabled state and returns it to the repository default behavior.
- **Disable** prevents the module from being selected in future transactions.
- **Reset** does not remove installed modular packages by itself.
- **Disable** is stronger for preventing future use of that module stream.

A common exam scenario is switching from one stream to another. The safe approach is:

1. Identify the currently enabled or installed stream.
2. Remove installed modular packages if they conflict with the new stream.
3. Reset or disable the old stream as needed.
4. Enable the new stream.
5. Install the new stream or profile.
6. Verify the result.

Example:

```bash
dnf module list --enabled
dnf module remove nodejs
dnf module reset nodejs
dnf module enable nodejs:20
dnf module install nodejs
dnf module list --enabled
rpm -q nodejs
```

> `dnf module reset` alone does not resolve package conflicts from already installed modular RPMs. You usually need to remove the installed modular content before switching streams.

**Practical stream switch example**

Suppose a system has one stream of a language runtime installed and you need a newer major version for an application.

1. Check the active stream.
2. Remove the installed modular packages tied to the old stream.
3. Reset the module state.
4. Enable the new stream.
5. Install the new stream or profile.
6. Verify both the stream and the RPMs.

```bash
dnf module list --installed
dnf module remove ruby
dnf module reset ruby
dnf module enable ruby:3.3
dnf module install ruby
rpm -q ruby
```

This sequence avoids the common mistake of resetting first and expecting the old packages to disappear automatically.

**Modular filtering and unavailable packages**

When a module stream is active, DNF filters package versions so only the selected modular content is visible. This can make packages appear missing even though they exist in the repository.

Symptoms include:
- `dnf install <package>` says no match found.
- A package version you expect does not appear in `dnf list`.
- A different stream is enabled than the one you intended.

Troubleshooting steps:

```bash
dnf module list --enabled
dnf module list <module-name>
dnf module info <module-name>
dnf repoquery --available <package-name>
```

- **Active streams** can hide non-selected versions.
- **Repository metadata** may contain the package, but modular filtering suppresses it.
- **Stream selection** often resolves the issue more cleanly than forcing a package install.

If a package is hidden by modular filtering, confirm whether a module stream owns it before assuming the repository is broken.

**Repository enablement and plugin-based commands**

Some troubleshooting tasks require enabling a repository before the needed module or group becomes available. On RHEL, repository management may use the `dnf config-manager` plugin, which is not a core DNF feature.

```bash
dnf repolist
dnf config-manager --set-enabled <repo-id>
dnf clean all
dnf makecache
```

- **`dnf config-manager`** depends on the plugin being installed and available.
- **Repository IDs** are the names you enable or disable, not package names.
- **Cache refresh** helps after repository changes.

If `config-manager` is unavailable, verify the plugin package and the system’s repository configuration before proceeding.

**Real-world RHCSA-style workflow**

A typical task may combine repository access, module selection, and verification.

Example scenario:
- You need a specific application stack.
- The package is not available in the default stream.
- The repository containing the module is disabled.
- You must enable the repository, select the correct stream, and verify the installed package.

Workflow:

```bash
dnf repolist
dnf config-manager --set-enabled appstream
dnf clean all
dnf module list <module-name>
dnf module enable <module-name>:<stream>
dnf module install <module-name>
rpm -q <package-name>
```

If the module requires a profile, confirm it first:

```bash
dnf module info <module-name>
```

Then install the exact profile name shown in the metadata.

**Verification strategy**

Verify at two levels: module state and package state.

- **Module state**: use `dnf module list --enabled` or `--installed`.
- **Package state**: use `rpm -q`, `rpm -qa`, or `dnf list installed`.
- **Metadata state**: use `dnf module info` to confirm profiles and artifacts.

Example:

```bash
dnf module list --enabled
dnf module info nodejs
rpm -q nodejs npm
```

This avoids the common mistake of assuming an enabled stream means the packages are installed, or assuming installed packages prove the intended stream is active.

**Common exam pitfalls**

- **Assuming group names are stable** across systems; use group IDs when possible.
- **Assuming a module profile exists** without checking `dnf module info`.
- **Assuming reset removes packages**; it only resets module state.
- **Assuming remove deletes everything**; verify package-level cleanup separately.
- **Ignoring modular filtering** when a package seems unavailable.
- **Using a stream/profile example from another release** without checking the target system.

**Quick reference**

- Use `dnf group list`, `dnf group info`, `dnf group install`, and `dnf group remove` for package groups.
- Use `dnf module list`, `dnf module info`, `dnf module enable`, `dnf module install`, `dnf module reset`, and `dnf module disable` for modular content.
- Use `rpm -q` to verify installed packages.
- Use `dnf config-manager` only when the plugin is available.
- Use group IDs and module metadata to avoid name mismatches.

**Key takeaway**

Package groups simplify role-based installs, while modularity controls software version selection. On RHCSA tasks, the important skill is not memorizing one command, but choosing the right workflow: inspect metadata, select the correct group or stream, install the right profile, and verify the result at both the module and package levels.
