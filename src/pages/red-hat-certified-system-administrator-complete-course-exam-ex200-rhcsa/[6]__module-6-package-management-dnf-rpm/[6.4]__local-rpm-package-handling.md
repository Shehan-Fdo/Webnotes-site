## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Overview**

Local RPM package handling is the foundation of software management on Red Hat Enterprise Linux (RHEL). You use it when you already have a package file on disk, such as a downloaded `.rpm`, and need to inspect, install, verify, or remove it without relying on a repository transaction. The package format is managed by the RPM <abbr title="RPM Package Manager">RPM</abbr> toolchain, while higher-level dependency resolution is usually handled by `dnf`.

For RHCSA-level work, you should know how to:
- inspect package metadata before installation,
- install local RPM files,
- query installed packages and package contents,
- verify package integrity and file ownership,
- remove packages cleanly,
- and troubleshoot common dependency or file-conflict issues.

**RPM package basics**

An RPM package is an archive that contains software files plus metadata. The metadata describes the package name, version, release, architecture, dependencies, file list, scripts, and checksums. On RHEL, package names often follow the pattern:

```text
name-version-release.arch.rpm
```

Examples:
- `httpd-2.4.57-1.el9.x86_64.rpm`
- `vim-enhanced-9.0.1-1.el9.x86_64.rpm`

The package filename is useful, but the internal package name may differ from the filename. Always query the package metadata when you need certainty.

**RPM vs DNF**

`rpm` is the low-level package manager. `dnf` is the higher-level tool that resolves dependencies from repositories and can also install local RPM files.

| Task | `rpm` | `dnf` |
|---|---:|---:|
| Install a local `.rpm` | Yes | Yes |
| Resolve dependencies automatically | No | Yes |
| Query installed package database | Yes | Yes |
| Verify file integrity | Yes | Limited |
| Remove installed package | Yes | Yes |

Use `rpm` when you need direct control or offline inspection. Use `dnf` when you want dependency resolution and a more complete transaction experience.

**Inspecting a local RPM before installation**

Before installing a package, inspect its metadata so you know what it contains and what it needs.

Common query options:
- `-q` queries package information.
- `-p` queries a package file instead of the installed database.
- `-i` shows package information.
- `-l` lists files in the package.
- `-R` lists dependencies.
- `-qf` identifies which package owns a file.
- `-qi` shows installed package info.
- `-ql` lists installed files.

Examples:

```bash
rpm -qip /tmp/httpd-2.4.57-1.el9.x86_64.rpm
rpm -qlp /tmp/httpd-2.4.57-1.el9.x86_64.rpm
rpm -qpR /tmp/httpd-2.4.57-1.el9.x86_64.rpm
```

These commands help you answer:
- What is the package name and version?
- Which architecture does it target?
- Which files will it install?
- Which dependencies must already exist?

If you are unsure whether a package is appropriate for the system, check the architecture first. Installing an `x86_64` package on an `aarch64` system will fail.

**Installing local RPM packages**

You can install a local RPM file with either `rpm` or `dnf`.

Using `rpm`:

```bash
sudo rpm -ivh /tmp/package.rpm
```

Using `dnf`:

```bash
sudo dnf install /tmp/package.rpm
```

Key flags for `rpm`:
- `-i` installs a package.
- `-v` increases verbosity.
- `-h` shows hash marks during installation.
- `-U` upgrades an existing package or installs it if missing.
- `--replacepkgs` reinstalls the same version.
- `--force` overrides some conflicts, but use it only when you understand the risk.

Practical guidance:
- Use `dnf install /path/to/package.rpm` when you want dependency resolution.
- Use `rpm -ivh /path/to/package.rpm` when you are working offline and dependencies are already satisfied.
- Use `rpm -Uvh /path/to/package.rpm` when you want an upgrade-style transaction.

Example:

```bash
sudo dnf install ./tree-1.8.0-10.el9.x86_64.rpm
```

The `./` prefix matters because it tells `dnf` to treat the argument as a local file rather than a package name.

> `rpm` does not resolve dependencies automatically. If required libraries are missing, installation fails unless those dependencies are already present.

**Upgrading and reinstalling packages**

`rpm -U` performs an upgrade transaction. If the package is not installed, `-U` installs it. This is often safer than `-i` when you are replacing an older version.

Examples:

```bash
sudo rpm -Uvh /tmp/httpd-2.4.57-1.el9.x86_64.rpm
sudo rpm -Uvh /tmp/httpd-2.4.58-1.el9.x86_64.rpm
```

Use reinstall behavior when you need to restore package files without changing the version:

```bash
sudo rpm -ivh --replacepkgs /tmp/vim-enhanced-9.0.1-1.el9.x86_64.rpm
```

Be careful with `--force`. It can overwrite files and bypass normal safety checks, which may leave the system in an inconsistent state.

**Querying installed packages**

Once a package is installed, you can query it from the RPM database.

Common queries:
- `rpm -q package_name` checks whether a package is installed.
- `rpm -qi package_name` shows package information.
- `rpm -ql package_name` lists installed files.
- `rpm -qf /path/to/file` identifies the owning package.
- `rpm -qa` lists all installed packages.

Examples:

```bash
rpm -q bash
rpm -qi bash
rpm -ql bash
rpm -qf /usr/bin/ls
rpm -qa | grep '^httpd'
```

Use these queries to confirm:
- the package is installed,
- the correct version is present,
- and the expected files exist on disk.

**Verifying package integrity**

Verification compares installed files against the RPM database. It checks attributes such as size, checksum, permissions, ownership, and modification time.

Basic verification:

```bash
rpm -V bash
```

If verification returns output, it indicates a difference. A blank result usually means the package files match the database state.

You can verify all installed packages, though this may produce a lot of output:

```bash
rpm -Va
```

Verification is useful when:
- you suspect a file was modified manually,
- you want to confirm package integrity after troubleshooting,
- or you need to identify unexpected changes on a system.

The output uses symbols to indicate what changed. For example:
- `S` size differs,
- `M` mode differs,
- `5` checksum differs,
- `U` user ownership differs,
- `G` group ownership differs,
- `T` modification time differs.

**Removing packages**

Remove installed packages with `rpm -e` or `dnf remove`.

Examples:

```bash
sudo rpm -e vim-enhanced
sudo dnf remove vim-enhanced
```

Removal uses the package name, not the filename. If you are unsure of the exact name, query the installed database first:

```bash
rpm -qa | grep vim
```

Important considerations:
- Removing a package may also remove functionality used by other software.
- `rpm -e` does not automatically solve dependency chains in the same way `dnf` does.
- If a package is required by another installed package, removal may fail.

**Working with package files and ownership**

You often need to answer questions such as “Which package owns this file?” or “What files came from this package?”

Examples:

```bash
rpm -qf /etc/httpd/conf/httpd.conf
rpm -ql httpd | head
rpm -qf /usr/bin/python3
```

This is especially useful when troubleshooting:
- unexpected file changes,
- missing binaries,
- or configuration files that belong to a package.

Package-owned configuration files are often marked as `%config` in the RPM metadata. These files may be preserved during upgrades if you have modified them locally.

**Common local RPM workflow**

A practical workflow for handling a downloaded package is:

1. Inspect the package metadata.
2. Check dependencies.
3. Install the package.
4. Verify the installation.
5. Confirm file ownership and service availability.
6. Remove or upgrade if needed.

Example workflow:

```bash
rpm -qip ./example.rpm
rpm -qpR ./example.rpm
sudo dnf install ./example.rpm
rpm -q example
rpm -ql example
rpm -V example
```

This sequence reduces mistakes because you validate the package before changing the system.

**Troubleshooting installation problems**

Common RPM installation issues include missing dependencies, file conflicts, and architecture mismatches.

Typical symptoms:
- `Failed dependencies` means required packages are missing.
- `file ... conflicts with file from package ...` means two packages want the same file path.
- `package ... is intended for a different architecture` means the RPM does not match the system CPU architecture.
- `package ... is already installed` means you are trying to install the same version with `-i`.

Useful checks:

```bash
uname -m
rpm -qp --qf '%{NAME} %{VERSION}-%{RELEASE} %{ARCH}\n' ./package.rpm
rpm -qR ./package.rpm
```

If a dependency is missing, prefer installing the dependency from a trusted repository rather than forcing the package in place.

**Useful RPM query formats**

`rpm` can print custom output with `--queryformat`. This is helpful when you need concise, script-friendly results.

Examples:

```bash
rpm -q --queryformat '%{NAME} %{VERSION}-%{RELEASE} %{ARCH}\n' bash
rpm -qp --queryformat '%{NAME}\n%{SUMMARY}\n' ./package.rpm
```

Common tags include:
- `%{NAME}`
- `%{VERSION}`
- `%{RELEASE}`
- `%{ARCH}`
- `%{SUMMARY}`
- `%{INSTALLTIME}`

This is useful for automation, inventory checks, and quick audits.

**Practical examples**

Install a local package and confirm it:

```bash
sudo dnf install ./htop-3.2.2-1.el9.x86_64.rpm
rpm -q htop
htop --version
```

Inspect a package before installation:

```bash
rpm -qip ./nginx-1.24.0-1.el9.x86_64.rpm
rpm -qpR ./nginx-1.24.0-1.el9.x86_64.rpm
rpm -qlp ./nginx-1.24.0-1.el9.x86_64.rpm
```

Verify a package after a configuration change:

```bash
rpm -V openssh-server
```

Find which package owns a file:

```bash
rpm -qf /usr/sbin/sshd
```

Remove a package:

```bash
sudo dnf remove htop
```

**Exam tips**

- Use `dnf` for local RPM installation when dependency resolution matters.
- Use `rpm` for direct package inspection and verification.
- Remember that package names and filenames are not always identical.
- Check architecture before installation.
- Use `rpm -qf` and `rpm -ql` frequently during troubleshooting.
- Prefer verification over guessing when a file looks altered.

**Summary**

Local RPM handling is about controlling software at the package-file level. You inspect the package first, install it with the right tool, verify the result, and remove it cleanly when needed. For RHCSA, focus on the core commands and the reasoning behind them: inspect before install, resolve dependencies carefully, and verify after changes.
