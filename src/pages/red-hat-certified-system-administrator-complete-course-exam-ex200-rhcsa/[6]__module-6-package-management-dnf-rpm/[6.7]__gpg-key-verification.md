## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**GPG key verification overview**

GPG (GNU Privacy Guard) keys are used to verify that RPM packages and repository metadata come from a trusted source and have not been altered. On Red Hat Enterprise Linux (RHEL), this is a core package trust workflow for `dnf` and `rpm`.

You typically verify two different trust layers:

- **Package signatures**: checks the RPM file itself.
- **Repository metadata signatures**: checks the repository metadata used by `dnf`.

These are related but separate controls. `gpgcheck=1` validates package signatures, while `repo_gpgcheck=1` validates repository metadata signatures.

**Why this matters in RHCSA tasks**

In EX200-style work, you often configure a custom repository, confirm the `gpgkey=` location, import the key, and then install packages safely. You should know how to inspect the key before trusting it, how to confirm what is already imported, and how to remove a bad key if needed.

**Core concepts**

- **GPG key**: a public key used to verify signatures.
- **Fingerprint**: a long, unique identifier for a key; this is what you compare against a trusted source.
- **Key ID**: a shorter identifier derived from the fingerprint; useful, but less authoritative than the full fingerprint.
- **RPM database**: local package database where imported public keys are stored as `gpg-pubkey` entries.
- **Trust prompt**: the confirmation message `dnf` shows when it needs to import a repository signing key.

**How DNF uses GPG keys**

When `dnf` accesses a repository with signature checking enabled, it may need to import the repository’s public key. If the key is not already trusted, `dnf` prompts you before import.

A typical workflow looks like this:

1. `dnf` reads the repository file in `/etc/yum.repos.d/*.repo`.
2. It checks `gpgcheck` and optionally `repo_gpgcheck`.
3. If the key is missing, `dnf` displays the key information and asks whether to import it.
4. You compare the fingerprint with a trusted source.
5. You accept only if the fingerprint matches.

> Do not accept a key just because the repository URL looks correct. Always verify the fingerprint from a trusted source first.

**Repository configuration fields**

A repository file commonly includes these settings:

```ini
[customrepo]
name=Custom Repository
baseurl=https://repo.example.com/rhel/9/x86_64/
enabled=1
gpgcheck=1
repo_gpgcheck=1
gpgkey=https://repo.example.com/RPM-GPG-KEY-customrepo
```

- `gpgcheck=1` enforces package signature verification.
- `repo_gpgcheck=1` enforces metadata signature verification.
- `gpgkey=` points to the public key file or URL used for import.

If `gpgcheck` is disabled, package signatures are not verified. If `repo_gpgcheck` is disabled, repository metadata signatures are not verified. They are independent settings.

**Inspecting a key before import**

If you have a key file locally, inspect it with GnuPG tools when the file format is compatible:

```bash
gpg --show-keys --with-fingerprint /path/to/RPM-GPG-KEY-customrepo
```

If the file is not directly parseable by `gpg`, use RPM-oriented methods or inspect the repository documentation for the published fingerprint. In practice, the safest exam approach is to compare the fingerprint shown by the repository owner with the fingerprint displayed by the import prompt or by local key inspection.

You can also check the key file contents directly:

```bash
cat /etc/pki/rpm-gpg/RPM-GPG-KEY-redhat-release
```

This is useful for confirming that the expected key file exists, but it does not by itself prove trust. The fingerprint comparison is the important step.

**What the DNF import prompt looks like**

When a key is not yet imported, `dnf` may show a prompt similar to this:

```text
Importing GPG key 0xXXXXXXXX:
 Userid     : "Repository Signing Key <repo@example.com>"
 Fingerprint: AAAA BBBB CCCC DDDD EEEE FFFF 1111 2222 3333 4444
 From       : https://repo.example.com/RPM-GPG-KEY-customrepo
Is this ok [y/N]:
```

What you should do:

- Compare the fingerprint to the vendor or internal repository documentation.
- Confirm the key ID and source URL match expectations.
- Type `y` only if the fingerprint is correct.
- Type `N` or press <kbd>Enter</kbd> if anything looks wrong.

This prompt is a security checkpoint, not a formality.

**Verifying imported keys in the RPM database**

Imported public keys are stored in the RPM database as packages named `gpg-pubkey`. You can list them with:

```bash
rpm -qa gpg-pubkey\*
```

To inspect a specific imported key:

```bash
rpm -qi gpg-pubkey-<keyid>-<version>
```

Example:

```bash
rpm -qi gpg-pubkey-fd431d51-4ae0493b
```

The output typically includes:

- **Name**: `gpg-pubkey`
- **Version**: the key ID fragment
- **Release**: a timestamp-like value
- **Summary**: public key information
- **Description**: often includes the key’s fingerprint or user ID

You can also query the package files:

```bash
rpm -ql gpg-pubkey-fd431d51-4ae0493b
```

For a more detailed view:

```bash
rpm -qi gpg-pubkey-fd431d51-4ae0493b | less
```

**Checking the fingerprint of an already imported key**

A practical way to verify an imported key is to inspect the `gpg-pubkey` entry and compare its fingerprint with the trusted source.

```bash
rpm -qi gpg-pubkey-fd431d51-4ae0493b
```

Look for the fingerprint in the description or related output. If the fingerprint is not obvious, use the repository’s published fingerprint and compare it with the key file or import prompt. On some systems, you may also export the key from the RPM database and inspect it with GnuPG:

```bash
rpm -q gpg-pubkey-fd431d51-4ae0493b --qf '%{VERSION}-%{RELEASE}\n'
rpm -qi gpg-pubkey-fd431d51-4ae0493b
```

If you need to extract the key material for inspection, use RPM query output or repository-provided key files rather than assuming every vendor key file will be directly readable by `gpg`.

**Removing a key you no longer trust**

If a key is incorrect, outdated, or no longer trusted, remove it from the RPM database.

1. Identify the key package:
   ```bash
   rpm -qa gpg-pubkey\*
   ```
2. Inspect it:
   ```bash
   rpm -qi gpg-pubkey-fd431d51-4ae0493b
   ```
3. Remove it:
   ```bash
   sudo rpm -e gpg-pubkey-fd431d51-4ae0493b
   ```

After removal, future package operations that require that key may prompt for import again, or fail if the repository is not configured correctly.

**Package verification with RPM**

`rpm` can verify signatures and digests on a local package file.

```bash
rpm -K package.rpm
```

You may also see:

```bash
rpm --checksig package.rpm
```

Typical results include:

- `package.rpm: digests signatures OK`
- `package.rpm: OK`
- `package.rpm: NOT OK`

Use this when you have a local RPM file and want to confirm it has not been tampered with. This is different from repository metadata verification and different from checking whether a key is imported.

**Common verification commands**

```bash
# List imported public keys
rpm -qa gpg-pubkey\*

# Inspect a specific imported key
rpm -qi gpg-pubkey-fd431d51-4ae0493b

# Check a local RPM package signature
rpm -K httpd-2.4.57-1.el9.x86_64.rpm

# Show repository configuration
dnf repolist -v

# Display repo files
ls -l /etc/yum.repos.d/

# View a repo file
cat /etc/yum.repos.d/custom.repo
```

**Practical RHCSA workflow**

A realistic exam task often involves setting up a repository and validating its key.

1. Create or edit the repository file in `/etc/yum.repos.d/`.
2. Set `baseurl=`, `enabled=1`, `gpgcheck=1`, and optionally `repo_gpgcheck=1`.
3. Confirm the `gpgkey=` path or URL is correct.
4. Verify the key fingerprint from the vendor or internal documentation.
5. Run `dnf install <package>`.
6. Review the import prompt carefully.
7. Accept only if the fingerprint matches.
8. Confirm the package installs successfully.

Example repository file:

```ini
[appstream-custom]
name=AppStream Custom
baseurl=file:///mnt/repo/AppStream
enabled=1
gpgcheck=1
repo_gpgcheck=0
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-custom
```

Example install flow:

```bash
sudo dnf clean all
sudo dnf makecache
sudo dnf install -y vim-enhanced
```

If the key is not imported, `dnf` prompts you before continuing. If the key is already imported and trusted, installation proceeds without prompting.

**Troubleshooting checklist**

- **Repository not reachable**: verify `baseurl`, DNS, proxy, and network access.
- **Key file missing**: confirm the `gpgkey=` path or URL exists.
- **Import prompt appears unexpectedly**: check whether the repository was rebuilt or the key changed.
- **Signature verification fails**: confirm the package came from the intended repository and that the correct key is imported.
- **Metadata verification fails**: check `repo_gpgcheck` and the repository’s metadata signing setup.
- **Wrong key imported**: remove it with `rpm -e gpg-pubkey-...` and import the correct one.

**Best practices**

- Verify the fingerprint from a trusted source before import.
- Use `gpgcheck=1` for package integrity.
- Use `repo_gpgcheck=1` when the repository supports signed metadata.
- Keep repository key files in a predictable location such as `/etc/pki/rpm-gpg/`.
- Remove obsolete keys after repository changes.
- Prefer explicit repository configuration over ad hoc trust decisions.

**Exam-focused reminders**

- GPG key verification is often part of repository setup tasks in EX200.
- You may need to edit `.repo` files, confirm `gpgkey` paths or URLs, and then install packages.
- Know the difference between package signature checks and repository metadata checks.
- Know how to inspect imported keys in the RPM database and how to remove them if needed.
- Be prepared to trust a key only after comparing fingerprints, not just after seeing a matching repository name.
