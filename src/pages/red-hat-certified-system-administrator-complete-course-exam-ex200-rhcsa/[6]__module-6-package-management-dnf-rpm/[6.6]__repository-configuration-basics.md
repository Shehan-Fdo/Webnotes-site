## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**Repository fundamentals**

A software repository is a source of RPM packages and metadata that DNF (Dandified YUM) uses to resolve dependencies, verify package availability, and install updates. On RHEL, repository access may come from Red Hat subscription content, local media, internal mirrors, or custom repositories you create yourself.

For RHCSA tasks, you need to know how to:
- identify enabled and disabled repositories
- enable or disable repositories temporarily or persistently
- create a valid `.repo` file
- verify package availability before installation
- troubleshoot metadata and repository parsing problems

A repository is not just a URL. DNF expects repository metadata, usually under a `repodata/` directory, and it uses repo IDs to distinguish sources. If the metadata is missing or the repo file is malformed, package operations fail even if the package files exist.

**Repository file structure**

Repository definitions live in `/etc/yum.repos.d/` and use INI-style syntax. Each repository section begins with a unique repo ID in square brackets.

```ini
[baseos]
name=RHEL 9 BaseOS
baseurl=file:///mnt/repo/BaseOS/
enabled=1
gpgcheck=1
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-redhat-release
```

Key points:
- **`[repoid]`** must be unique across all repo files.
- **`name`** is a human-readable label.
- **`baseurl`** points to the directory that contains `repodata/`.
- **`enabled`** controls whether DNF uses the repo by default.
- **`gpgcheck`** enables RPM signature verification.
- **`gpgkey`** points to the public key used to verify packages, but the exact key path can vary by release and system state.

> The `gpgkey` line above is an example only. On current RHEL systems, the exact key file name or location may differ, and the key may not exist at that path unless the system or media provides it.

**How DNF uses repositories**

DNF reads repository definitions, downloads metadata, and builds a package view from all enabled sources. It then resolves dependencies across repositories and modules. If multiple repositories provide the same package, DNF chooses based on repository priority, version, and modular constraints.

Useful inspection commands:
```bash
dnf repolist
dnf repolist all
dnf list available 'httpd*'
dnf info httpd
dnf search httpd
dnf provides /usr/sbin/httpd
```

Practical interpretation:
- `dnf repolist` shows enabled repositories only.
- `dnf repolist all` shows enabled and disabled repositories.
- `dnf list available 'httpd*'` checks package names that match a pattern.
- `dnf info httpd` shows package details if DNF can resolve it.
- `dnf search httpd` searches package names and summaries.
- `dnf provides /usr/sbin/httpd` helps when you know a file path but not the package name.

**Managing repositories with subscription-manager**

On registered RHEL systems, `subscription-manager` can enable or disable Red Hat-provided repositories. This is a common enterprise workflow because it changes entitlement-based access without editing repo files manually.

Examples:
```bash
subscription-manager repos --list
subscription-manager repos --enable=rhel-9-for-x86_64-baseos-rpms
subscription-manager repos --enable=rhel-9-for-x86_64-appstream-rpms
subscription-manager repos --disable=rhel-9-for-x86_64-appstream-rpms
```

Use this method when:
- the system is registered to Red Hat
- you need to control official content access
- you want a persistent repository state tied to subscription configuration

This is different from a temporary DNF override. `subscription-manager` changes repository availability at the entitlement layer, while `dnf --enablerepo` affects only the current command.

**Temporary versus persistent repository control**

You often need both styles during troubleshooting.

Temporary enablement:
```bash
dnf --enablerepo=customrepo install vim
dnf --disablerepo='*' --enablerepo=baseos install bash
```

Persistent enablement:
- edit the `.repo` file and set `enabled=1`
- or use `subscription-manager repos --enable=...` on registered systems

Practical difference:
- **Temporary overrides** affect one DNF command only.
- **Persistent changes** remain after logout, reboot, and future package operations.

Use temporary overrides when you want to test a repository without changing system state. Use persistent changes when the repository should remain available for normal administration.

**Using dnf config-manager**

`dnf config-manager` is useful for enabling, disabling, or adding repository definitions, but it depends on the `dnf-plugins-core` package. On many minimal RHEL systems, that package is not installed by default.

Check availability:
```bash
rpm -q dnf-plugins-core
dnf list installed dnf-plugins-core
```

Install it if the repository providing it is available:
```bash
dnf install -y dnf-plugins-core
```

Common workflows:
```bash
dnf config-manager --set-enabled baseos
dnf config-manager --set-disabled appstream
dnf config-manager --add-repo=file:///mnt/repo/custom.repo
```

Important exam note:
- If the plugin is missing and you cannot install it, edit the repo file directly.
- Do not assume `dnf config-manager` is always available on an exam system.

**Creating a repository file from scratch**

A realistic RHCSA task is to configure a local or internal repository by creating a repo file manually.

Example:
```ini
[local-baseos]
name=Local BaseOS Repository
baseurl=file:///mnt/repo/BaseOS/
enabled=1
gpgcheck=0
```

Procedure:
1. Create the file in `/etc/yum.repos.d/`, for example `/etc/yum.repos.d/local.repo`.
2. Ensure the section name is unique.
3. Point `baseurl` to the directory containing `repodata/`.
4. Save the file with correct permissions and no syntax errors.
5. Verify the repo is visible with `dnf repolist all`.
6. Test package access with `dnf info`, `dnf list available`, or `dnf install --assumeno`.

Validation commands:
```bash
dnf repolist all
dnf clean all
dnf makecache
dnf info bash
dnf install --assumeno bash
```

If the repository is valid, `dnf makecache` succeeds and package queries return metadata instead of errors.

**Local repository layout**

For a local repository, the directory you reference must contain repository metadata. DNF expects the directory that holds `repodata/`, not just a folder with RPM files.

Typical layout:
```text
/mnt/repo/
├── BaseOS/
│   ├── Packages/
│   └── repodata/
└── AppStream/
    ├── Packages/
    └── repodata/
```

If you are building your own repository from RPMs, generate metadata with `createrepo_c`:

```bash
dnf install -y createrepo_c
createrepo_c /srv/repo/custom
```

After adding or removing RPMs, refresh metadata:
```bash
createrepo_c --update /srv/repo/custom
```

Use `createrepo_c` when:
- you create an internal package share
- you mirror RPMs into a custom directory
- you need DNF to recognize a directory of RPMs as a repository

**AppStream and module awareness**

RHEL AppStream content can be modular. That means some software is delivered as modules or streams rather than as plain packages alone. Repository availability does not always guarantee installability if the required module stream is disabled or a different stream is active.

Useful commands:
```bash
dnf module list
dnf module list nodejs
dnf module enable nodejs:18
dnf module reset nodejs
```

Why this matters:
- A package may appear in repository metadata but still be constrained by module stream selection.
- If installation fails unexpectedly, check whether the package belongs to a module stream.
- For exam troubleshooting, repository configuration and module state are related but not identical problems.

**Common syntax and parsing pitfalls**

Small mistakes in repo files can break DNF parsing.

Watch for:
- duplicate repo IDs across files
- missing `[` or `]` around the section name
- malformed key/value lines
- accidental spaces in the repo ID
- commented-out lines that you expected to be active
- invalid `baseurl` paths
- pointing `baseurl` at the wrong directory level
- using a URL without reachable metadata
- forgetting that `enabled=0` disables the repo even if the file exists

Example of a broken file:
```ini
[custom repo]
name=Bad Repo ID
baseurl=file:///srv/repo/custom
enabled=1
```

Problems here:
- the repo ID contains a space
- the directory may not contain `repodata/`
- the file may parse, but the repo may still fail to load if metadata is missing

A safer pattern:
```ini
[custom-repo]
name=Custom Repo
baseurl=file:///srv/repo/custom/
enabled=1
gpgcheck=0
```

**Troubleshooting repository access**

When a repository does not work, isolate the failure step by step.

1. Confirm the repo file exists:
```bash
ls -l /etc/yum.repos.d/
```

2. Check repo visibility:
```bash
dnf repolist all
```

3. Clear stale metadata:
```bash
dnf clean all
rm -rf /var/cache/dnf/*
```

4. Rebuild cache:
```bash
dnf makecache
```

5. Test package discovery:
```bash
dnf search httpd
dnf provides '*/httpd.conf'
dnf list available 'httpd*'
```

6. Inspect the repo file for syntax errors:
```bash
cat /etc/yum.repos.d/local.repo
```

7. Confirm the base URL is reachable:
```bash
ls -l /mnt/repo/BaseOS/repodata/
```

If the repository is local and the path is wrong, DNF usually reports metadata download or repository not found errors. If the repo file is malformed, DNF may ignore it or emit parsing warnings.

**GPG signature verification**

`gpgcheck=1` tells DNF to verify package signatures before installation. This protects against tampered packages and is the preferred setting for trusted repositories.

Example:
```ini
[secure-repo]
name=Secure Repo
baseurl=file:///srv/repo/secure/
enabled=1
gpgcheck=1
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-example
```

Practical notes:
- If the key is missing, package installation fails until you import the correct public key.
- For exam labs, `gpgcheck=0` is sometimes used for simplicity in local practice environments, but real systems should use signature verification whenever possible.
- If you import a key manually, verify that it matches the repository owner.

**Exam-oriented workflow**

A common RHCSA scenario is to configure a repository, verify it, and install a package from it.

Example workflow:
```bash
mkdir -p /mnt/repo/custom
mount /dev/sr0 /mnt/repo/custom
cat > /etc/yum.repos.d/custom.repo <<'EOF'
[custom]
name=Custom Installation Media
baseurl=file:///mnt/repo/custom/
enabled=1
gpgcheck=0
EOF
dnf clean all
dnf repolist
dnf install --assumeno vim-enhanced
```

What this proves:
- the repo file is syntactically valid
- DNF can read metadata from the mounted media
- the package is available before you attempt a real install

If the media contains multiple repository trees, make sure you point to the correct subdirectory, such as `BaseOS/` or `AppStream/`, not the top-level mount point unless that mount point itself contains `repodata/`.

**Best practices for RHCSA tasks**

- Prefer direct repo file editing when you need a guaranteed method.
- Use `dnf repolist all` to confirm the final state.
- Use `dnf clean all` after changing repository definitions.
- Use `dnf info`, `dnf search`, and `dnf provides` to verify package visibility.
- Remember that AppStream modules can affect package availability.
- Use `subscription-manager repos` on registered systems when working with Red Hat content.
- Use `createrepo_c` for custom local repositories that contain your own RPM collection.
- Keep repo IDs simple, unique, and free of spaces.

**Quick comparison of repository methods**

| Method | Scope | Typical use | Persistence |
|---|---|---|---|
| Edit `.repo` file | System-wide | Local or internal repo setup | Persistent |
| `dnf --enablerepo/--disablerepo` | Single command | Temporary testing | Temporary |
| `dnf config-manager` | System-wide | Enable/disable or add repos | Persistent |
| `subscription-manager repos` | Registered RHEL system | Red Hat entitlement repositories | Persistent |

**Key takeaways**

Repository configuration is a core RHCSA skill because package management depends on it. You should be able to create a repo file, validate metadata, troubleshoot parsing issues, and distinguish between temporary and persistent repository changes. For exam success, always confirm the repository state with DNF before you try to install packages.
