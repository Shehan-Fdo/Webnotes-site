## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**RPM command purpose and exam context**

`rpm` is the low-level Red Hat Package Manager tool. On RHCSA, you use it when you need direct control over package files, package metadata, verification, or local installation and upgrade behavior. It does **not** resolve dependencies automatically, so you must know when a command will fail because required packages are missing.

Use `rpm` when you need to inspect a package file, verify installed files, or install a local `.rpm` package without using `dnf`. Use `dnf` when you want dependency resolution and repository handling.

- **Query mode** starts with `-q` and inspects the RPM database or a local package file.
- **Install mode** uses `-i` and installs a local RPM file.
- **Upgrade mode** uses `-U` and upgrades or installs a package file.
- **Verify mode** uses `-V` and compares installed files to package metadata.
- **Signature check** uses `-K` to inspect digest and signature status of a local RPM file.

> `rpm` works against the RPM database and package files, not against a repository index. If you need dependency resolution, use `dnf` instead.

**Core syntax patterns**

The most important exam habit is to recognize whether you are querying an installed package, querying a local RPM file, or acting on a package file for install or upgrade.

```bash
rpm [options] package-name
rpm [options] /path/to/package.rpm
```

Common patterns:

```bash
rpm -q package-name
rpm -qip /path/to/package.rpm
rpm -qpl /path/to/package.rpm
rpm -qi package-name
rpm -ql package-name
rpm -qf /usr/bin/ls
rpm -qa
rpm -q --whatprovides '*/httpd.conf'
rpm -V package-name
rpm -K /path/to/package.rpm
rpm -ivh /path/to/package.rpm
rpm -Uvh /path/to/package.rpm
```

**Querying packages with `rpm -q`**

`-q` means query. This is the base for most inspection tasks. Without `-q`, options like `-i` and `-l` do **not** mean “show info” or “list files” for installed packages.

Use `rpm -q` when you want to ask the RPM database about installed packages, or when you want to inspect a local RPM file with `-p`.

- `rpm -q package` checks whether a package is installed and prints the installed NEVRA-style package name if present.
- `rpm -qa` lists all installed packages.
- `rpm -qf /path/to/file` shows which installed package owns a file path.
- `rpm -qp /path/to/file.rpm` queries a local RPM file instead of the installed database.
- `rpm -q --whatprovides 'pattern'` asks which installed package or available package name provides a capability or file pattern, depending on context and repository metadata.

NEVRA means **Name, Epoch, Version, Release, Architecture**. You do not need to memorize the acronym for every task, but you should recognize that RPM package names often include version and architecture details.

**Common `rpm -q` suboptions**

| Command | What it does | Typical use |
|---|---|---|
| `rpm -q package` | Checks whether a package is installed | Quick presence check |
| `rpm -qa` | Lists all installed packages | Inventory or filtering |
| `rpm -qf /path` | Shows owning package for a file path | Identify package ownership |
| `rpm -qp file.rpm` | Queries a local RPM file | Inspect package before install |
| `rpm -q --whatprovides name` | Finds what provides a capability or name | Dependency or capability lookup |

Examples:

```bash
rpm -q bash
rpm -qa | grep '^kernel'
rpm -qf /usr/bin/ssh
rpm -qp /tmp/httpd-2.4.57-1.el9.x86_64.rpm
rpm -q --whatprovides 'webserver'
```

A common exam trap is confusing `rpm -qf` with a filesystem search. `rpm -qf` does not search the disk for arbitrary files; it asks the RPM database which installed package owns the path you provide. The path does not have to exist at the moment if it matches an installed file entry, but it must correspond to a file recorded in the RPM database.

**Package information with `rpm -qi` and `rpm -qip`**

`-i` is **install mode** when used by itself with `rpm -i`. It is **not** a query flag by itself. To display package information, you must combine it with query mode:

- `rpm -qi package` shows information for an installed package.
- `rpm -qip file.rpm` shows information from a local RPM file.

This distinction is important because `rpm -i bash` is not a valid way to inspect package metadata. If you want metadata, use query mode.

Typical output includes:

- package name
- version and release
- architecture
- install date for installed packages
- summary and description
- vendor and license
- build date and signature-related fields

Examples:

```bash
rpm -qi bash
rpm -qip /tmp/bash-5.2.15-5.el9.x86_64.rpm
```

Use `-qi` when you need to confirm what is installed on the system. Use `-qip` when you want to inspect a package file before installing it.

**Listing files with `rpm -ql` and `rpm -qpl`**

`-l` lists files, but only in query mode.

- `rpm -ql package` lists files installed by a package.
- `rpm -qpl file.rpm` lists files contained in a local RPM file.

This is useful when you need to know where a package will place files or where an installed package has placed them.

Examples:

```bash
rpm -ql bash
rpm -qpl /tmp/bash-5.2.15-5.el9.x86_64.rpm
```

Use `-ql` for installed packages and `-qpl` for package files. If you omit `-q`, you are no longer querying installed package contents.

**Querying file ownership with `rpm -qf`**

`rpm -qf` answers the question: “Which installed package owns this file path?”

Examples:

```bash
rpm -qf /usr/bin/ls
rpm -qf /etc/ssh/sshd_config
```

Important points:

- It queries the RPM database, not a directory listing.
- It is most useful for troubleshooting or identifying which package to reinstall.
- If the path is not owned by any installed RPM package, RPM reports that no package owns it.

A practical workflow is:

```bash
rpm -qf /usr/bin/ls
rpm -qi coreutils
```

This tells you both the owning package and the package details.

**Listing all installed packages with `rpm -qa`**

`rpm -qa` lists every installed package in the RPM database.

Examples:

```bash
rpm -qa
rpm -qa | grep '^openssl'
rpm -qa --qf '%{NAME}-%{VERSION}-%{RELEASE}.%{ARCH}\n'
```

Use it when you need to inventory the system or confirm whether a package family is present. The `--qf` option is a query format string and is useful for exam-style filtering.

**Finding providers with `rpm --whatprovides`**

`--whatprovides` is used with query mode to identify which package provides a capability, file, or name pattern. This is more common in repository-aware workflows, but it is still useful to understand as an RPM query concept.

Examples:

```bash
rpm -q --whatprovides 'perl(File::Basename)'
rpm -q --whatprovides /usr/bin/python3
```

On RHCSA, you are more likely to use `dnf provides` for repository searches, but you should still recognize the RPM query form.

**Verifying installed packages with `rpm -V`**

`rpm -V` compares installed files against the metadata stored in the RPM database. It does not reinstall anything. It tells you whether files have changed since installation.

Use it when you suspect:

- a configuration file was edited
- a binary was replaced or corrupted
- permissions or ownership changed unexpectedly
- a package was tampered with

Examples:

```bash
rpm -V bash
rpm -V httpd
rpm -Vf /usr/bin/ls
```

If a package verifies cleanly, `rpm -V` usually prints no output. Blank output means no differences were detected for the files checked.

If differences exist, RPM prints a verification string. The characters correspond to changed attributes:

| Character | Meaning |
|---|---|
| `S` | File size differs |
| `M` | File mode differs |
| `5` | MD5 digest differs |
| `D` | Device major/minor number differs |
| `L` | Symbolic link path differs |
| `U` | User ownership differs |
| `G` | Group ownership differs |
| `T` | Modification time differs |
| `P` | Capability differs |

The leading field may also indicate file state or type-related information in the output format, but for exam purposes focus on the attribute letters above.

Example interpretation:

```bash
rpm -V bash
```

Possible output:

```text
S.5....T.  c /etc/bashrc
```

This means:

- `S` size changed
- `5` digest changed
- `T` timestamp changed
- `c` indicates a configuration file

Key exam takeaway:

- Blank output means no differences.
- Any letter in the verification string means that attribute changed.
- A changed configuration file is not always a problem; it may simply reflect an intentional admin edit.

**Checking package signatures with `rpm -K`**

`rpm -K` or `rpm --checksig` checks the digest and signature status of a local RPM file.

Examples:

```bash
rpm -K /tmp/httpd-2.4.57-1.el9.x86_64.rpm
rpm --checksig /tmp/httpd-2.4.57-1.el9.x86_64.rpm
```

Typical conclusions:

- `digests OK` means the package file content matches its recorded digest.
- `pgp md5 OK` or similar signature-related output indicates the file has a valid signature record.
- If the signature cannot be verified because the public key is missing, RPM may still report the file’s digest status but not establish trust.

Important distinction:

> `rpm -K` checks integrity and signature status, but it does not by itself prove trust unless the relevant GPG public key is imported and trusted on the system.

On RHCSA, you should know that `rpm -K` is a pre-install check for a local package file. It helps you confirm that the file has not been altered and that signature metadata is present.

**Installing local packages with `rpm -ivh`**

`rpm -i` installs a local RPM package file. The `-v` flag increases verbosity, and `-h` shows hash marks during installation.

Examples:

```bash
rpm -ivh /tmp/httpd-2.4.57-1.el9.x86_64.rpm
rpm -ivh /tmp/custom-tool-1.0-1.el9.noarch.rpm
```

What the flags mean:

- `-i` install
- `-v` verbose output
- `-h` progress hash marks

Use `rpm -ivh` when you want to install a package file directly and you understand that RPM will not resolve dependencies for you.

Practical notes:

- If required dependencies are missing, installation fails.
- If the exact same package is already installed, you usually need `--replacepkgs` to reinstall it.
- If a file conflict or dependency issue exists, RPM stops and reports the problem rather than resolving it automatically.

Example with reinstall behavior:

```bash
rpm -ivh --replacepkgs /tmp/httpd-2.4.57-1.el9.x86_64.rpm
```

**Upgrading local packages with `rpm -Uvh`**

`rpm -Uvh` upgrades an installed package if an older version exists, or installs it if the package is not already present.

Examples:

```bash
rpm -Uvh /tmp/httpd-2.4.57-2.el9.x86_64.rpm
rpm -Uvh /tmp/kernel-*.rpm
```

Use `-Uvh` when you want the package to move forward to a newer version while preserving the package name and replacing older files as appropriate.

Key behavior:

- If an older version is installed, RPM upgrades it.
- If no version is installed, RPM installs it.
- Like `-i`, it does not resolve dependencies automatically.
- It is the common choice for local package updates.

A useful distinction:

- `rpm -ivh` is for a fresh install or explicit reinstall behavior.
- `rpm -Uvh` is for upgrade or install-if-missing behavior.

**Install vs upgrade behavior**

| Command | Primary behavior | Typical use |
|---|---|---|
| `rpm -ivh package.rpm` | Install a package file | First-time install |
| `rpm -Uvh package.rpm` | Upgrade if installed, otherwise install | Updating to a newer build |

Examples:

```bash
rpm -ivh /tmp/tool-1.0-1.el9.x86_64.rpm
rpm -Uvh /tmp/tool-1.1-1.el9.x86_64.rpm
```

**Dependency handling and why it matters**

RPM is a package manager, but it is not a dependency resolver in the way `dnf` is. This is one of the most important RHCSA distinctions.

- `rpm -ivh` and `rpm -Uvh` install or upgrade only the package file you provide.
- If dependencies are missing, RPM reports the problem and stops.
- `dnf install` or `dnf upgrade` is the usual choice when you want automatic dependency resolution.

Example workflow:

```bash
rpm -K /tmp/httpd-2.4.57-1.el9.x86_64.rpm
rpm -qip /tmp/httpd-2.4.57-1.el9.x86_64.rpm
rpm -Uvh /tmp/httpd-2.4.57-1.el9.x86_64.rpm
```

This sequence lets you inspect, verify, and then install or upgrade a local package file.

**Quick exam reference**

| Task | Command |
|---|---|
| Check if installed | `rpm -q package` |
| Show package info | `rpm -qi package` |
| Show info from local RPM | `rpm -qip file.rpm` |
| List installed package files | `rpm -ql package` |
| List files in local RPM | `rpm -qpl file.rpm` |
| Find owning package | `rpm -qf /path/to/file` |
| List all installed packages | `rpm -qa` |
| Verify installed package files | `rpm -V package` |
| Check local RPM signature/digest | `rpm -K file.rpm` |
| Install local RPM | `rpm -ivh file.rpm` |
| Upgrade or install local RPM | `rpm -Uvh file.rpm` |

**Practical command examples**

```bash
# Check whether Apache is installed
rpm -q httpd

# Show details for the installed package
rpm -qi httpd

# Inspect a downloaded RPM before installing
rpm -qip /tmp/httpd-2.4.57-1.el9.x86_64.rpm

# List files that belong to the installed package
rpm -ql httpd

# List files inside the RPM file
rpm -qpl /tmp/httpd-2.4.57-1.el9.x86_64.rpm

# Identify which package owns a file
rpm -qf /usr/sbin/httpd

# Verify installed files
rpm -V httpd

# Check package digest and signature status
rpm -K /tmp/httpd-2.4.57-1.el9.x86_64.rpm

# Install a local package
rpm -ivh /tmp/httpd-2.4.57-1.el9.x86_64.rpm

# Upgrade a local package
rpm -Uvh /tmp/httpd-2.4.57-2.el9.x86_64.rpm
```

**Common mistakes to avoid**

- Using `rpm -i` when you meant `rpm -qi`.
- Using `rpm -l` without `-q` and expecting installed-package file listing.
- Assuming `rpm -K` proves trust without a trusted GPG key.
- Expecting `rpm -ivh` or `rpm -Uvh` to resolve dependencies automatically.
- Forgetting that `rpm -qf` queries the RPM database for ownership, not a general filesystem search.
- Reading `rpm -V` output as a reinstall command; it only verifies.

**RHCSA study strategy**

For the exam, memorize the action behind each flag and the object it acts on:

- `-q` asks questions about packages.
- `-i` installs package files.
- `-l` lists package files.
- `-V` verifies installed files.
- `-K` checks a local RPM file’s digest and signature status.
- `-ivh` installs with verbose output and progress hashes.
- `-Uvh` upgrades with verbose output and progress hashes.

If you can quickly decide whether the task is about an installed package, a local RPM file, or a file owned by a package, you can choose the correct command under exam pressure.
