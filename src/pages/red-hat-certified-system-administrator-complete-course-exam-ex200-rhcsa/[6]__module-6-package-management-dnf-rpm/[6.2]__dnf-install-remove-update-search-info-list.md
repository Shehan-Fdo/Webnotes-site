## **Course:** Red Hat Certified System Administrator Complete Course (Exam EX200 – RHCSA)

**DNF overview**

`dnf` is the Red Hat family package manager used on modern RHEL systems to install, remove, update, and inspect software packages. It resolves dependencies from enabled repositories, tracks package metadata, and can also install local RPM files when you provide a path or URL.

For RHCSA, you should know how to:
- find the right package name,
- verify what a package contains,
- install or remove software safely,
- update packages selectively or in bulk,
- troubleshoot repository and metadata issues.

`dnf` is the successor to `yum` in RHEL 8 and later. In many environments, `yum` is still present as a compatibility wrapper, but exam tasks should be done with `dnf`.

**Core workflow**

A practical package-management workflow usually looks like this:

1. Search for the package.
2. Inspect package details.
3. Install the package.
4. Verify the installation.
5. Update or remove it when needed.

```text
dnf search httpd
dnf info httpd
dnf install httpd
dnf list installed httpd
dnf update httpd
dnf remove httpd
```

**Package naming and repository basics**

DNF works with package names, not just filenames. A package name may differ from the command or service name you expect. For example, the web server package is `httpd`, while the service is `httpd.service` and the binary may be `httpd`.

- **Package name** is what DNF installs and removes.
- **Repository metadata** tells DNF what packages are available.
- **Enabled repositories** determine what DNF can see.
- **Dependencies** are additional packages required for the software to function.

If a package is not found, the issue is often one of these:
- the repository is not enabled,
- metadata is stale,
- the package name is slightly wrong,
- the package exists in a different repository stream or module.

> `dnf remove` removes the named package. It may also remove dependencies later if they become orphaned, but it does not automatically purge every dependency in all cases.

**Install packages**

Use `dnf install` to install software from enabled repositories or from a local RPM path/URL.

```bash
sudo dnf install httpd
```

When you install a package, DNF:
- resolves dependencies,
- downloads required RPMs,
- verifies signatures if configured,
- installs the package set transactionally.

You can install multiple packages in one transaction:

```bash
sudo dnf install vim-enhanced git tree
```

For a local RPM file, provide a path or URL:

```bash
sudo dnf install ./example-package-1.2.3-1.el9.x86_64.rpm
sudo dnf install https://example.com/repo/example-package-1.2.3-1.el9.x86_64.rpm
```

A local path or URL tells DNF to treat the input as a package source rather than a repository package. The `./` prefix is common when the file is in the current directory, but the important part is that you are passing a local file path.

**Real-world install scenario**

You download an RPM to a lab directory and need to install it without adding a repository.

```bash
cd /tmp
ls -l example-package-1.2.3-1.el9.x86_64.rpm
sudo dnf install ./example-package-1.2.3-1.el9.x86_64.rpm
```

If dependencies are missing from enabled repositories, DNF reports the problem instead of completing the transaction. In an exam setting, that usually means you need to enable the correct repository or use a package source that includes the required dependencies.

**Remove packages**

Use `dnf remove` to uninstall a package.

```bash
sudo dnf remove httpd
```

This removes the package you named. DNF may also mark now-unused dependencies as removable, but it does not always delete them immediately. If you want to clean up orphaned dependencies, you can inspect them separately.

Useful follow-up commands:
```bash
dnf repoquery --unneeded
dnf autoremove
```

- `dnf autoremove` removes packages that were installed as dependencies and are no longer needed.
- Use it carefully on shared systems, because a package may appear unused even though another admin expects it to remain installed.

**Update packages**

Use `dnf update` to bring installed packages to the latest available version from enabled repositories.

```bash
sudo dnf update
```

You can update a single package:

```bash
sudo dnf update httpd
```

In DNF, `update` and `upgrade` are effectively equivalent for normal package maintenance. On RHCSA, treat them as the same behavior unless a specific lab or policy says otherwise.

Common update patterns:
- update everything before troubleshooting a system,
- update only a security-sensitive package,
- refresh metadata before checking for newer versions.

```bash
sudo dnf --refresh update
```

`--refresh` is an option for the invoked DNF command; it forces metadata refresh before the transaction starts. It is not a standalone command.

**Search packages**

Use `dnf search` when you know part of a package name, summary, or description but not the exact package name.

```bash
dnf search httpd
dnf search web server
```

Search results can include:
- package names,
- summaries,
- descriptions.

That makes `search` useful when you are unsure whether the package is called `httpd`, `apache`, or something else.

**Practical search scenario**

You need a package that provides a command, but you only know the tool name from a lab prompt.

```bash
dnf search podman
dnf search "container tools"
```

If the search result is too broad, narrow it by using a more specific keyword or by checking package info next.

**Show package information**

Use `dnf info` to inspect a package before installing it or to confirm what is already installed.

```bash
dnf info httpd
```

`dnf info` typically shows:
- name,
- epoch, version, release, architecture,
- summary,
- description,
- repository source,
- size,
- license,
- URL.

This is useful when you need to confirm:
- whether the package is installed,
- which repository provides it,
- whether the version matches your requirement.

You can also inspect an installed package by name:

```bash
dnf info bash
```

If the package is not installed, `dnf info` can still show repository metadata for available versions, provided the package exists in enabled repositories.

**List packages**

Use `dnf list` to see package availability and installation state.

```bash
dnf list httpd
```

This is the most common form when you want to check a specific package. DNF can show whether the package is installed, available, or both depending on the query and repository state.

Useful list variants:
```bash
dnf list installed
dnf list available
dnf list updates
dnf list httpd
dnf list installed httpd
```

- `dnf list installed` shows installed packages.
- `dnf list available` shows packages available from enabled repositories.
- `dnf list updates` shows packages with updates available.
- `dnf list httpd` is a quick way to compare installed and available states for one package.

**Decision flow for common tasks**

```text
Need to find a package?
  -> dnf search

Need to inspect details?
  -> dnf info

Need to see whether it is installed or available?
  -> dnf list

Need to install it?
  -> dnf install

Need to update it?
  -> dnf update

Need to remove it?
  -> dnf remove
```

**Typical exam-style scenarios**

A lab task may ask you to install a service package and verify it is present.

1. Search for the package if the name is unclear.
2. Inspect it with `dnf info`.
3. Install it with `dnf install`.
4. Confirm installation with `dnf list installed` or `dnf info`.

```bash
dnf search httpd
dnf info httpd
sudo dnf install httpd
dnf list installed httpd
```

Another common scenario is troubleshooting a package that cannot be found.

1. Check whether the repository metadata is current.
2. Refresh metadata with `--refresh`.
3. Confirm the package name.
4. Verify that the needed repository is enabled.

```bash
sudo dnf --refresh search nginx
dnf repolist
dnf list available nginx
```

If the package still does not appear, the issue is usually repository access, not the install command itself.

**Troubleshooting package operations**

Package management problems often come from metadata, naming, or repository issues rather than from DNF syntax.

- **Package not found**
  - Check spelling and package name.
  - Search by summary or description.
  - Confirm the repository is enabled.
- **No matching package to install**
  - Refresh metadata.
  - Verify the package exists in the current release.
  - Check whether a module stream or alternate repository is required.
- **Dependency resolution failure**
  - Ensure all required repositories are enabled.
  - Use a local RPM only when you also have access to its dependencies.
- **Installed package not visible**
  - Use `dnf list installed`.
  - Confirm you are querying the correct package name, not the service name.

> If you install a local RPM that depends on packages not present in enabled repositories, DNF cannot complete the transaction unless the dependencies are available from another source.

**Useful command examples**

```bash
# Find packages related to Apache
dnf search apache

# Show package details
dnf info httpd

# Install a package
sudo dnf install httpd

# Install from a local RPM
sudo dnf install ./httpd-2.4.57-1.el9.x86_64.rpm

# List installed packages
dnf list installed

# Check one installed package
dnf list installed httpd

# Show available versions
dnf list available httpd

# Update a package
sudo dnf update httpd

# Update all packages
sudo dnf update

# Remove a package
sudo dnf remove httpd
```

**Quick comparison**

| Command | Main purpose | Typical use |
|---|---|---|
| `dnf search` | Find packages by name, summary, or description | You do not know the exact package name |
| `dnf info` | Show package metadata | You want version, repo, size, or description |
| `dnf list` | Show installed/available/update state | You want to confirm package status |
| `dnf install` | Install packages | You need software on the system |
| `dnf update` | Update installed packages | You want newer versions from repos |
| `dnf remove` | Uninstall packages | You no longer need the software |

**Exam tips**

- Use the package name, not the service name, when working with DNF.
- Use `dnf search` before guessing package names.
- Use `dnf info` to confirm what you are about to install.
- Use `dnf list installed` to verify installation quickly.
- Use `dnf --refresh` with the command you are running when metadata may be stale.
- Remember that local RPM installation still depends on dependency availability.
- Treat `update` and `upgrade` as equivalent in normal DNF usage.

**Mental model**

Think of DNF as a transaction manager:
- `search` helps you discover packages,
- `info` helps you inspect them,
- `list` helps you verify state,
- `install` changes the system by adding packages,
- `update` changes the system by replacing older versions,
- `remove` changes the system by uninstalling packages.

That sequence is the foundation for most RHCSA package-management tasks.
