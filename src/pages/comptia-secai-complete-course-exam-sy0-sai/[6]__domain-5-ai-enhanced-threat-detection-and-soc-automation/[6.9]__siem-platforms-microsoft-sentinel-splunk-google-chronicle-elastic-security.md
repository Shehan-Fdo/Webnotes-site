## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**SIEM platform overview**

A <abbr title="Security Information and Event Management">SIEM</abbr> platform centralizes log collection, normalization, correlation, alerting, investigation, and reporting. In practice, you use a SIEM to turn raw telemetry into security-relevant events, then enrich and prioritize those events so analysts can respond quickly.

Across Microsoft Sentinel, Splunk, Google Chronicle, and Elastic Security, the core workflow is similar:

- Collect data from endpoints, cloud services, identity systems, network devices, and applications.
- Parse and normalize logs into a common schema.
- Correlate events with rules, analytics, and threat intelligence.
- Enrich alerts with context such as asset criticality, user identity, geolocation, and reputation.
- Investigate and respond through cases, incidents, playbooks, or automation.

The exam often tests both the shared SIEM concepts and the platform-specific tradeoffs. You should know where each product fits best, what it depends on, and how analysts move from detection to response.

**Core SIEM functions you should recognize**

- **Normalization** converts vendor-specific fields into a consistent model so queries and detections work across sources.
- **Parsing** extracts fields from raw text, JSON, syslog, or API payloads.
- **Correlation** links multiple low-signal events into a higher-confidence security event.
- **Enrichment** adds context such as threat intelligence, asset inventory, identity data, or vulnerability data.
- **Retention** controls how long data stays searchable and how storage tiers affect cost and performance.
- **Response** includes alert triage, case management, ticketing, containment actions, and orchestration.

A useful exam shortcut is to separate the SIEM lifecycle into ingestion, detection, investigation, and response. If a question asks about “making logs usable,” think parsing and normalization. If it asks about “turning events into action,” think correlation, enrichment, and automation.

**Microsoft Sentinel**

Microsoft Sentinel is a cloud-native SIEM and <abbr title="Security Orchestration, Automation, and Response">SOAR</abbr> platform built on Azure. It is tightly integrated with Microsoft 365, Azure, Entra ID, Defender, and the broader Azure ecosystem, which makes it a strong fit for organizations already standardized on Microsoft cloud services.

Sentinel is typically deployed in an Azure Log Analytics workspace. Data connectors bring in logs from Microsoft services, third-party products, and custom sources. Analytics rules generate incidents, and automation rules or playbooks can trigger response actions.

Key concepts:

- **Workspace-based architecture** means data lands in a Log Analytics workspace.
- **Data connectors** ingest logs from Azure, Microsoft 365, Defender, AWS, syslog, CEF, and APIs.
- **KQL** (<abbr title="Kusto Query Language">KQL</abbr>) is the primary query language for hunting and analytics.
- **Incidents** group related alerts for analyst workflow.
- **Playbooks** use Azure Logic Apps for orchestration and response.

A practical Sentinel onboarding flow usually looks like this:

1. Create or select an Azure subscription and Log Analytics workspace.
2. Enable the Sentinel solution on that workspace.
3. Connect data sources through built-in connectors or custom ingestion.
4. Configure analytics rules, watchlists, and automation.
5. Review incidents, tune detections, and build response playbooks.

Example KQL hunting query:

```kusto
SecurityEvent
| where EventID == 4625
| summarize FailedLogons = count() by Account, Computer, bin(TimeGenerated, 15m)
| where FailedLogons > 10
| order by FailedLogons desc
```

This query looks for repeated failed logons and groups them into a time window. In exam scenarios, the important idea is not the exact syntax but the use of KQL for hunting and threshold-based detection.

Example automation rule concept:

```text
If incident severity is High
AND entity contains a privileged account
THEN assign to Tier 2 queue
AND trigger playbook for ticket creation and user disable workflow
```

Sentinel strengths:

- Strong Microsoft ecosystem integration.
- Native cloud scalability with minimal infrastructure management.
- Good support for automation through Logic Apps.
- Flexible hunting with KQL and workbook-based visualization.

Sentinel tradeoffs:

- Best value usually appears when you already use Azure and Microsoft security products.
- Costs can rise with ingestion volume and retention.
- Some advanced use cases require careful workspace, connector, and rule design.

Retention and cost considerations:

- Log Analytics retention settings affect how long data remains searchable.
- Archive and long-term retention reduce cost but can slow investigations.
- Data ingestion pricing is often a major planning factor.

**Splunk**

Splunk is a widely used data platform and SIEM that excels at indexing machine data and searching it with <abbr title="Search Processing Language">SPL</abbr> (Search Processing Language). It is often deployed in larger environments where flexible search, custom parsing, and broad data ingestion are important.

Splunk architecture commonly includes:

- **Forwarders** that collect and send data.
- **Indexers** that parse, index, and store events.
- **Search heads** that run searches, dashboards, and correlation logic.
- Optional **deployment servers**, **cluster managers**, and **heavy forwarders** for scale and management.

A common exam point is that Splunk separates search-time and index-time processing. You can parse data during ingestion or at search time, and that choice affects performance, flexibility, and storage.

Typical deployment flow:

1. Install Universal Forwarder on endpoints or servers.
2. Configure forwarding to an indexer or intermediate forwarder.
3. Enable receiving on the indexer side.
4. Define indexes, sourcetypes, and field extractions.
5. Build searches, alerts, dashboards, and correlation searches.

Example forwarder output configuration:

```ini
# outputs.conf
[tcpout]
defaultGroup = main_indexers

[tcpout:main_indexers]
server = indexer1.example.com:9997,indexer2.example.com:9997
```

Example indexer receiving concept:

```bash
splunk enable listen 9997
```

Example SPL search:

```spl
index=wineventlog EventCode=4625
| stats count by user, host, src_ip
| where count > 10
```

Splunk strengths:

- Very flexible search and reporting.
- Strong ecosystem for custom data onboarding.
- Mature support for dashboards, correlation searches, and alerting.
- Good fit for heterogeneous environments with many log formats.

Splunk tradeoffs:

- Requires careful index and sourcetype design.
- Search performance depends heavily on data modeling and field extraction strategy.
- Licensing and storage costs can grow quickly with high ingest volumes.

Retention and tiering considerations:

- Splunk retention is usually managed by index settings and storage architecture.
- Hot, warm, cold, and frozen data tiers affect performance and cost.
- Older data may be moved to cheaper storage or archived, depending on deployment design.

Splunk investigation workflow often uses:

- Searches to identify suspicious patterns.
- Dashboards to summarize trends.
- Notable events or alerts to surface detections.
- Cases or ticketing integrations for tracking response.

**Google Chronicle / Google SecOps**

Google Chronicle, now commonly referred to as Google SecOps, is a cloud-native security operations platform focused on large-scale log analytics, threat detection, and investigation. It is designed for high-volume ingestion and long retention with strong normalization into the <abbr title="Unified Data Model">UDM</abbr> (Unified Data Model).

Chronicle is often described as emphasizing fast search over large datasets and consistent normalization. That makes UDM mapping a central concept: if your data is not normalized well, detections and investigations become less effective.

Core ideas:

- **UDM** standardizes security telemetry across sources.
- **Detections** are commonly expressed with YARA-L rules.
- **Cases** support analyst workflow and incident tracking.
- **Threat intelligence** and entity context are important parts of investigation.
- **Long retention** is a common selling point for large-scale environments.

A practical Chronicle onboarding flow usually includes:

1. Set up the Google SecOps tenant and administrative access.
2. Connect log sources through collectors, APIs, or supported integrations.
3. Map source fields into UDM.
4. Validate normalization and entity resolution.
5. Create detections, triage alerts, and investigate in the case workflow.

Example UDM-style concept:

```text
principal.user.userid = "alice"
target.hostname = "server01"
metadata.event_type = "USER_LOGIN"
security_result.action = "FAIL"
```

Example YARA-L-style detection concept:

```text
rule repeated_failed_logons {
  meta:
    description = "Detect repeated failed logons from one user"
  events:
    $e.metadata.event_type = "USER_LOGIN"
    $e.security_result.action = "FAIL"
  condition:
    #e > 10
}
```

> The exact YARA-L syntax can vary by version and rule type, so treat this as a conceptual example rather than a guaranteed copy-paste rule.

Chronicle strengths:

- Scales well for large log volumes.
- Strong normalization through UDM.
- Useful for long-term retention and fast search across large datasets.
- Good fit when you want cloud-managed security analytics with less infrastructure overhead.

Chronicle tradeoffs:

- Detection quality depends heavily on normalization quality.
- Analysts must understand UDM mappings to write effective detections.
- Platform-specific rule syntax and workflows differ from traditional SIEM query languages.

Investigation workflow often centers on:

- Searching normalized entities and events.
- Pivoting from one entity to related users, hosts, IPs, and alerts.
- Reviewing cases and detections.
- Using threat intelligence to prioritize suspicious activity.

**Elastic Security**

Elastic Security is the security application built on the Elastic Stack, especially Elasticsearch, Kibana, and Elastic Agent. It is popular for organizations that want flexible search, schema control, and a strong open ecosystem.

Elastic Security can be deployed self-managed or in Elastic Cloud. Modern deployments often use **Fleet** to manage Elastic Agents centrally. That architecture matters for exam questions because agent management, integration setup, and policy assignment are different from older standalone Beats-only designs.

Core concepts:

- **Elastic Agent** collects telemetry from endpoints and integrations.
- **Fleet** manages agent policies and integrations centrally.
- **Kibana** provides the security UI, detections, cases, and dashboards.
- **ECS** (<abbr title="Elastic Common Schema">ECS</abbr>) standardizes field names across data sources.
- **Detection rules** generate alerts, which analysts can manage in the security app.

A practical Elastic onboarding flow usually looks like this:

1. Deploy Elasticsearch and Kibana, or use Elastic Cloud.
2. Enable Fleet and create an agent policy.
3. Add integrations such as Windows, Linux, endpoint, cloud, or network sources.
4. Install Elastic Agent on hosts and enroll them in Fleet.
5. Create detection rules, tune exceptions, and review alerts in the Security app.

Example Fleet enrollment command:

```bash
sudo ./elastic-agent install \
  --url=https://fleet.example.com:8220 \
  --enrollment-token=REDACTED
```

Example KQL-style detection query in Elastic Security:

```kql
event.code: "4625" and user.name: *
```

A better practical pattern is to target a specific suspicious condition rather than a broad wildcard. For example:

```kql
event.code: "4625" and user.name: "alice" and source.ip: "10.10.10.25"
```

Example detection rule idea:

```text
If a single user generates more than 10 failed logons from one source IP in 15 minutes,
create an alert and add the host and user to the investigation timeline.
```

Elastic strengths:

- Flexible schema and powerful search.
- Strong support for custom data models and dashboards.
- Fleet simplifies modern agent management.
- Good investigation experience with timelines, cases, and entity context.

Elastic tradeoffs:

- Schema and mapping management matter a lot.
- Poor ECS alignment can make detections inconsistent.
- Self-managed deployments require more operational effort than fully managed options.

Retention considerations:

- Elasticsearch uses index lifecycle management <abbr title="Index Lifecycle Management">ILM</abbr> to move data through hot, warm, cold, and frozen phases.
- Retention policy design affects cost, performance, and search speed.
- Snapshot and restore strategies matter for long-term archive needs.

**Platform comparison**

| Platform | Primary strength | Common query/detection model | Typical deployment style | Key tradeoff |
|---|---|---|---|---|
| Microsoft Sentinel | Azure-native SIEM and SOAR | KQL, analytics rules, playbooks | Cloud-native in Azure workspace | Best fit in Microsoft ecosystem |
| Splunk | Flexible search and broad ingestion | SPL, correlation searches, dashboards | Indexers, search heads, forwarders | Cost and tuning complexity |
| Google Chronicle / SecOps | Large-scale cloud analytics and long retention | UDM, YARA-L detections | Cloud-managed with collectors/integrations | Depends on normalization quality |
| Elastic Security | Search flexibility and schema control | KQL, detection rules, timelines | Elastic Cloud or self-managed stack | Requires careful schema and policy design |

When a question asks which platform is best for a Microsoft-heavy organization, Sentinel is often the strongest answer. When it asks for broad custom log search and mature search language flexibility, Splunk is a common fit. When it emphasizes massive scale and normalized cloud analytics, Chronicle is a strong candidate. When it emphasizes open search, schema control, and integrated endpoint/security workflows, Elastic Security is a likely answer.

**Normalization, parsing, and enrichment**

These concepts appear in every SIEM and are easy exam targets.

- **Parsing** extracts fields from raw logs.
- **Normalization** maps fields into a common schema.
- **Enrichment** adds context from external or internal sources.
- **Correlation** combines events into a meaningful security story.

Examples of enrichment sources:

- Threat intelligence feeds.
- Asset inventory and CMDB data.
- Identity providers such as Entra ID or LDAP.
- Vulnerability scanners.
- GeoIP and reputation services.

Practical example:

- A failed login from a known malicious IP is more important than a failed login from a corporate VPN.
- A login to a domain controller by a privileged account is more important than the same event on a lab workstation.
- A detection with no asset context may be noisy; enrichment helps prioritize it.

**Investigation and response workflow**

Analysts usually move through a similar sequence regardless of platform:

1. Review the alert or incident.
2. Validate the source data and time window.
3. Pivot to related entities such as user, host, IP, process, or cloud resource.
4. Check enrichment and threat intelligence.
5. Determine scope and impact.
6. Contain, eradicate, and document the case.

Platform-specific response objects differ:

- Sentinel uses incidents, automation rules, and playbooks.
- Splunk often uses notable events, alerts, and integrations with ticketing or SOAR tools.
- Chronicle uses detections and cases in Google SecOps workflows.
- Elastic Security uses alerts, cases, timelines, and response actions.

A common exam trap is assuming all platforms use the same terminology. They do not. Learn the object names and the workflow around them.

**Automation and orchestration**

SOAR capabilities help reduce analyst workload by automating repetitive response steps.

Common automated actions:

- Disable a user account.
- Isolate an endpoint.
- Block an IP address.
- Open or update a ticket.
- Enrich an alert with threat intelligence.
- Notify a chat channel or on-call system.

Examples by platform:

- **Sentinel**: Logic Apps playbooks can call Microsoft and third-party APIs.
- **Splunk**: Often integrates with Splunk SOAR or external automation tools.
- **Chronicle**: Integrates with Google SecOps workflows and response tooling.
- **Elastic**: Uses cases, connectors, and response actions through the security app and integrations.

Automation should be tuned carefully. Over-automation can create outages or block legitimate users, so high-confidence conditions and approval steps are often appropriate for destructive actions.

**Retention, storage tiers, and lifecycle management**

Retention is not just a cost topic; it affects investigations and compliance.

- **Hot storage** is fast and expensive.
- **Warm storage** is cheaper but slower.
- **Cold storage** is cheaper still and often used for older data.
- **Archive/frozen** tiers reduce cost further but may require restore steps before analysis.

Platform mapping:

- Sentinel uses workspace retention and archive options.
- Splunk uses index retention and storage tiering.
- Chronicle emphasizes long retention in the cloud.
- Elastic uses ILM policies and snapshots.

If a scenario asks about compliance retention, think about both searchable retention and archival retention. If it asks about fast investigations, think about hot or recently indexed data.

**Licensing and feature availability**

Licensing can affect what you can actually use.

- Sentinel pricing is commonly tied to ingestion and retention.
- Splunk licensing is often tied to daily ingest volume or platform edition.
- Chronicle pricing and packaging depend on Google SecOps offerings and tenant capabilities.
- Elastic features vary by subscription tier and deployment model.

For exam purposes, the key idea is that feature availability is not always identical across editions. A platform may support a capability in principle, but the specific license or cloud plan may determine whether you can use it.

**Common exam scenarios**

- If the environment is heavily Microsoft-based and needs cloud-native SIEM plus automation, choose Sentinel.
- If the organization needs highly flexible search across many log types and already has Splunk expertise, choose Splunk.
- If the organization ingests very large volumes and wants normalized cloud analytics with long retention, choose Chronicle.
- If the organization wants open schema control, strong search, and integrated security workflows, choose Elastic Security.

**Quick memory cues**

- **Sentinel** = Azure, KQL, incidents, playbooks.
- **Splunk** = SPL, indexers, search heads, forwarders.
- **Chronicle** = UDM, YARA-L, long retention, cloud scale.
- **Elastic** = ECS, Fleet, Kibana, alerts and cases.

**Practical tuning tips**

- Reduce false positives by adding asset, identity, and time-based context.
- Use allowlists carefully and review them regularly.
- Prefer threshold and sequence logic over single noisy events.
- Validate detections against known-good and known-bad test data.
- Document what each rule is meant to catch and what it intentionally ignores.

**What to remember for the exam**

- SIEMs do more than alerting; they normalize, correlate, enrich, retain, and support response.
- Deployment architecture matters as much as query syntax.
- Each platform has a distinct workflow and terminology.
- Retention, licensing, and ecosystem integration often drive platform choice.
- Detection quality depends on data quality, schema consistency, and tuning.
