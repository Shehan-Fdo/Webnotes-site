**Detection engineering with AI support**

Detection engineering is the discipline of designing, validating, deploying, and improving detections that identify malicious or risky behavior in logs, telemetry, and security tools. AI can accelerate research, rule drafting, test generation, and triage support, but it does not replace analyst judgment. The goal is to produce detections that are accurate, explainable, maintainable, and aligned to business risk.

A strong detection program starts with a clear use case, not with a tool or a model. You define what behavior matters, what data is available, how the detection will be measured, and how it will be maintained over time. AI is most useful when it helps you move faster through repetitive work while you keep control over requirements, validation, and promotion.

**Detection engineering lifecycle**

A practical lifecycle usually moves through these stages:

1. Identify the threat or behavior you want to detect.
2. Define detection requirements and acceptance criteria.
3. Map the use case to available telemetry and analytics logic.
4. Build and test the detection in a lab or dev environment.
5. Tune for false positives and false negatives.
6. Promote to production with change control.
7. Monitor performance and iterate.

The lifecycle is continuous. A detection that works today may degrade when logging changes, attackers adapt, or the environment evolves.

**Detection requirements and acceptance criteria**

Detection requirements describe what the rule must accomplish. Acceptance criteria define when the detection is good enough to deploy. These artifacts matter because they prevent vague rules and make validation objective.

Common requirement elements include:

- **Threat behavior**: the action or sequence you want to catch, such as credential dumping or suspicious PowerShell use.
- **Data sources**: the logs, sensors, or APIs required, such as endpoint telemetry, identity logs, DNS logs, or cloud audit logs.
- **Coverage scope**: which assets, users, platforms, or environments the detection applies to.
- **Latency target**: how quickly the alert should arrive after the event.
- **Severity expectation**: how the alert should be prioritized.
- **Response action**: whether the alert is informational, triage-worthy, or should trigger containment.

Acceptance criteria should be testable. For example:

- The rule detects the target behavior in at least one representative lab scenario.
- The rule does not alert on known benign administrative activity.
- The rule includes required fields for triage, such as host, user, process, and timestamp.
- The rule can be explained in plain language to an analyst.
- The rule is versioned and documented before production promotion.

A useful pattern is to write requirements in a way that supports both human review and automated testing.

```text
Use case: Detect suspicious encoded PowerShell execution
Requirement: Alert when PowerShell launches with encoded or obfuscated command content
Acceptance criteria:
- Detects base64-encoded PowerShell in lab replay
- Excludes approved admin scripts
- Includes parent process, command line, user, host, and time
- False positive rate remains acceptable after one week of pilot
```

**Use-case prioritization and ATT&CK-based coverage management**

Not every possible detection deserves equal effort. Prioritize use cases based on risk, likelihood, and operational value. High-value detections usually cover behaviors that are common, high impact, and observable in your environment.

A practical prioritization model considers:

- **Business impact**: what happens if the behavior succeeds.
- **Threat relevance**: whether current adversaries use the technique.
- **Telemetry availability**: whether you can actually observe the behavior.
- **Analyst cost**: how much triage effort the alert creates.
- **Coverage gap**: whether the technique is already covered by another detection.

<abbr title="MITRE Adversarial Tactics, Techniques, and Common Knowledge">MITRE ATT&CK</abbr> mapping helps you organize coverage, but it should not become a checkbox exercise. Use ATT&CK to identify gaps, compare coverage across tactics and techniques, and track which detections are preventive, detective, or compensating controls.

Common coverage management practices include:

- Mapping each detection to one or more ATT&CK techniques.
- Tracking which techniques are covered by multiple detections versus not covered at all.
- Reviewing coverage by platform, identity, endpoint, cloud, and network telemetry.
- Prioritizing detections for techniques that are both high-risk and observable.
- Reassessing coverage after major environment or threat changes.

**Frameworks and standards for detection content**

Detection engineering often uses structured content formats and lifecycle practices to keep rules portable and maintainable.

Key concepts include:

- **Sigma**: a generic, YAML-based detection rule format that can be translated into backend-specific queries. Sigma is useful for authoring detections once and adapting them to multiple platforms.
- **ATT&CK-based coverage management**: a method for tracking which adversary behaviors are covered, where gaps exist, and which detections need tuning.
- **Use-case prioritization**: ranking detections by risk, feasibility, and operational value.
- **Detection-as-code**: storing rules, tests, and documentation in version control so changes are reviewable and repeatable.

Sigma-style content usually separates the detection logic from the backend query language. That makes it easier to review intent before translation.

```yaml
title: "Suspicious PowerShell Encoded Command"
id: 2f1c2d1a-1111-2222-3333-444444444444
status: test
logsource:
  product: windows
  category: process_creation
detection:
  selection:
    Image|endswith: '\powershell.exe'
    CommandLine|contains:
      - ' -enc '
      - ' -encodedcommand '
  condition: selection
fields:
  - Image
  - CommandLine
  - ParentImage
  - User
falsepositives:
  - Approved admin automation
level: medium
```

**Data sources and field mapping**

Detection quality depends on telemetry quality. A rule can be logically correct and still fail if the required fields are missing, normalized incorrectly, or inconsistent across sources.

Important field-mapping concerns include:

- **Normalization**: different platforms may name the same concept differently, such as `process.name`, `Image`, or `process_path`.
- **Parsing**: logs may need extraction before they are usable in detections.
- **Enrichment**: asset criticality, user role, geolocation, and threat intel can improve triage.
- **Consistency**: the same event should produce the same meaning across environments.
- **Completeness**: missing command lines, parent process data, or user context can make a detection brittle.

A common pitfall is overreliance on indicators of compromise, or **IOCs**. IOCs such as hashes, domains, and IP addresses are useful for short-term blocking, but they age quickly and often miss attacker variation. Behavior-based detections are usually more durable.

> If your detection depends on a single field that is often absent or inconsistently parsed, expect brittle logic and noisy alerts.

**False positives, false negatives, and triage impact**

Detection engineering is a tradeoff between catching more malicious activity and avoiding alert fatigue. A **false positive** is a benign event that triggers an alert. A **false negative** is malicious activity that the detection misses.

These tradeoffs affect the security operations center in different ways:

- High false positives increase triage workload and can cause analysts to ignore alerts.
- High false negatives create blind spots and reduce trust in the detection program.
- Overly broad rules may catch more activity but produce poor signal.
- Overly narrow rules may look clean but miss real attacks.

When tuning a detection, ask:

- What benign behavior looks similar to the threat?
- What attacker variations would bypass the current logic?
- Which fields are stable enough to rely on?
- Is the alert actionable, or does it require too much manual investigation?

A useful metric is alert precision, which is the proportion of alerts that are truly relevant. Another useful measure is recall, which reflects how much of the target behavior the detection catches. In practice, you often improve one at the expense of the other.

**Measuring detection effectiveness over time**

A detection should be measured after deployment, not just during development. Useful metrics include:

- **Alert volume**: how many alerts the rule generates over time.
- **True positive rate**: how often alerts represent real suspicious activity.
- **False positive rate**: how often alerts are benign.
- **Mean time to triage**: how long analysts take to review the alert.
- **Mean time to detect**: how quickly the detection identifies the behavior.
- **Coverage drift**: whether the rule still matches current telemetry and attacker behavior.
- **Suppression rate**: how often alerts are muted or deduplicated.

Trend analysis matters more than a single snapshot. A rule that is acceptable in pilot may become noisy after a software rollout, logging change, or new business process.

**Testing methods and validation workflows**

Testing is where detection engineering becomes reliable. You should validate both the logic and the data path.

Common testing methods include:

- **Atomic tests**: small, controlled actions that simulate a single technique, often used to verify whether a rule fires on a specific behavior.
- **Adversary emulation**: a broader simulation of attacker tradecraft across multiple steps.
- **Lab simulation**: reproducing the target behavior in a safe environment with representative telemetry.
- **Historical replay**: running the rule against past logs to see how it would have behaved.
- **Unit-style rule tests**: checking whether known inputs produce expected outputs.

Tools and approaches often used in practice include:

- **Atomic Red Team** for technique-level validation.
- **MITRE Caldera** for adversary emulation and campaign-style testing.
- **Prelude Operator** or similar frameworks for controlled execution of test actions.
- SIEM query testing in a sandbox or dev tenant before production deployment.

A practical validation sequence looks like this:

1. Run the test action in a lab.
2. Confirm the raw telemetry is present.
3. Confirm the parser or normalization layer preserves the needed fields.
4. Run the detection query.
5. Compare the alert to the expected outcome.
6. Tune the rule and repeat.

```bash
# Example: validate a Windows process-creation detection in a lab
Get-WinEvent -LogName Security -MaxEvents 20 |
  Where-Object { $_.Id -eq 4688 } |
  Select-Object TimeCreated, Id, Message
```

```bash
# Example: search for suspicious encoded PowerShell in a Linux SIEM CLI workflow
grep -Ei 'powershell|encodedcommand|-enc' /var/log/suricata/eve.json
```

**Backend-specific detection examples**

Different platforms express the same detection idea in different query languages. The underlying logic should remain consistent.

```kql
// Microsoft Sentinel / KQL example
SecurityEvent
| where EventID == 4688
| where Process has "powershell.exe"
| where CommandLine has_any ("-enc", "-encodedcommand")
| project TimeGenerated, Computer, Account, ParentProcessName, Process, CommandLine
```

```spl
# Splunk SPL example
index=endpoint sourcetype=WinEventLog:Security EventCode=4688
| search ProcessName="*powershell.exe*" (CommandLine="*-enc*" OR CommandLine="*-encodedcommand*")
| table _time host user ParentProcessName ProcessName CommandLine
```

```sql
-- SQL-like example for a security data lake
SELECT event_time, host, user_name, parent_process, process_name, command_line
FROM process_events
WHERE process_name LIKE '%powershell.exe%'
  AND (command_line LIKE '%-enc%' OR command_line LIKE '%-encodedcommand%');
```

```yaml
# Elastic-style detection logic example
name: Suspicious Encoded PowerShell
index: logs-windows.process*
query: process.name: "powershell.exe" and process.command_line: (*-enc* or *-encodedcommand*)
```

When you translate detections across platforms, verify that field names, case sensitivity, wildcard behavior, and tokenization rules are equivalent. A query that works in one backend may behave differently in another.

**Content lifecycle management and promotion**

Detection content should move through controlled environments just like software. A common model is dev, test, and prod.

- **Dev**: you draft and iterate on the rule.
- **Test**: you validate against representative telemetry and lab scenarios.
- **Prod**: you deploy with monitoring, ownership, and rollback plans.

Good lifecycle management includes:

- Version control for rules, tests, and documentation.
- Pull requests or peer review before promotion.
- Change logs that explain why a rule changed.
- Rollback procedures if a rule becomes noisy or breaks.
- Ownership assignment so someone is accountable for maintenance.

A repository structure might look like this:

```text
detections/
  windows/
    suspicious_powershell.yml
    suspicious_powershell.test.yml
  cloud/
    impossible_travel.yml
docs/
  suspicious_powershell.md
```

CI/CD, or **continuous integration/continuous delivery**, can automate linting, schema checks, unit tests, and deployment packaging for detections. This reduces manual errors and helps ensure that changes are tested before release.

**Documentation standards**

Documentation is part of the detection artifact, not an optional extra. Good documentation helps analysts understand intent, helps engineers maintain the rule, and helps auditors verify control coverage.

A detection document should usually include:

- Purpose and threat scenario.
- ATT&CK mapping.
- Data sources and required fields.
- Logic summary in plain language.
- Known false positives and exclusions.
- Validation method and test evidence.
- Tuning history and version notes.
- Owner and review date.

A concise documentation template can look like this:

```markdown
# Suspicious Encoded PowerShell
Purpose: Detect obfuscated PowerShell execution used in post-exploitation.
Data sources: Windows process creation logs
ATT&CK: T1059.001
Validation: Atomic test + lab replay
Known false positives: Approved admin scripts
Owner: Detection Engineering
Review cadence: Quarterly
```

**AI support in detection engineering**

AI can assist with detection engineering in several ways:

- Drafting initial rule logic from a use-case description.
- Translating a detection idea into multiple backend query languages.
- Summarizing logs or alert context for analysts.
- Suggesting test cases and edge conditions.
- Generating documentation or change notes.

The best results come from using AI as a drafting and analysis assistant, not as an autonomous rule publisher. You still need human review for correctness, safety, and operational fit.

**Prompt engineering patterns for detection tasks**

Prompt quality strongly affects output quality. Good prompts are specific, constrained, and context-rich.

Useful prompt patterns include:

- Ask for a detection rule plus assumptions and limitations.
- Provide sample log fields and ask the model to map them correctly.
- Request multiple backend translations from the same logic.
- Ask for false positive scenarios and tuning ideas.
- Ask the model to generate test cases, not just the rule.

Example prompt:

```text
Write a Sigma rule for suspicious encoded PowerShell.
Use these fields: Image, CommandLine, ParentImage, User.
Include false positives, required fields, and a short validation plan.
Then translate the logic into KQL and Splunk SPL.
```

Better prompts often include constraints such as:

- “Do not use hash-based indicators.”
- “Prefer behavior-based logic.”
- “Assume Windows process creation logs only.”
- “List any missing fields that would weaken the rule.”

**Validating AI-generated detections**

AI-generated detections should be treated as drafts until they pass validation. Common validation steps include:

- Checking that field names exist in the target data source.
- Confirming the logic matches the intended behavior.
- Testing against benign and malicious examples.
- Reviewing for brittle assumptions or overfitting.
- Verifying that the rule does not leak sensitive data or create unsafe automation.

A practical review checklist:

- Does the rule rely on a single IOC?
- Does it assume a field that your logs do not collect?
- Does it overfit to one sample command line?
- Does it miss common attacker variations?
- Does it produce an actionable alert?

AI can also help generate negative test cases, which are important for spotting false positives. For example, if a rule detects encoded PowerShell, test it against approved admin scripts, software deployment tools, and legitimate automation.

**Common pitfalls and how to avoid them**

Detection engineering fails when rules are too clever, too narrow, or too dependent on unstable data.

Common pitfalls include:

- **Overreliance on IOCs**: hashes and domains change quickly.
- **Poor field mapping**: the rule references fields that are missing or renamed.
- **Brittle logic**: exact string matching breaks when attackers add spaces, casing changes, or minor obfuscation.
- **No acceptance criteria**: the team cannot tell whether the rule is ready.
- **No testing discipline**: rules are deployed without lab validation.
- **No ownership**: nobody maintains the rule after deployment.
- **No lifecycle tracking**: content drifts from dev to prod without review.
- **No analyst feedback loop**: triage lessons never feed back into tuning.

A robust detection usually combines multiple signals, such as process behavior, parent-child relationships, command-line patterns, and context from identity or network telemetry.

**Practical tuning strategies**

Tuning is the process of improving signal quality without losing meaningful coverage. Common strategies include:

- Adding allowlists for approved automation.
- Requiring multiple conditions instead of one weak indicator.
- Narrowing scope to high-risk hosts or users.
- Using thresholding or aggregation to reduce noise.
- Enriching alerts with context so analysts can decide faster.

Example of a more resilient logic pattern:

```kql
SecurityEvent
| where EventID == 4688
| where Process has "powershell.exe"
| where CommandLine has_any ("-enc", "-encodedcommand")
| where not(CommandLine has "ApprovedAdminScript")
| summarize count() by Computer, Account, bin(TimeGenerated, 15m)
| where count_ > 0
```

This kind of logic reduces noise, but you should be careful not to suppress real attacker activity that resembles approved tools. Allowlisting should be narrow and reviewed regularly.

**Operational workflow for AI-assisted detection engineering**

A practical workflow combines human expertise, automation, and review:

1. Define the threat behavior and required telemetry.
2. Ask AI to draft candidate logic and test ideas.
3. Review the draft for field accuracy and operational fit.
4. Run atomic tests or lab simulations.
5. Compare expected and actual alert behavior.
6. Tune the rule and document the result.
7. Promote through dev, test, and prod with version control.
8. Monitor metrics and revisit the rule on a schedule.

This workflow keeps AI in a support role while preserving accountability for detection quality.

**Exam focus points**

For exam preparation, focus on the relationships between design, validation, and operations:

- Detection engineering is a lifecycle, not a one-time rule-writing task.
- Requirements and acceptance criteria make detections testable.
- ATT&CK mapping helps manage coverage, but it does not guarantee quality.
- Sigma and detection-as-code improve portability and maintainability.
- False positives and false negatives directly affect triage workload and trust.
- AI is useful for drafting, translation, and analysis, but human validation remains essential.
- Testing with atomic tests, emulation, and lab simulation is critical before production.
- Content promotion, version control, and documentation are part of the control itself.
