## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Overview**

These attacks target different parts of an AI system’s confidentiality boundary. **Model extraction** tries to steal the model itself, **model inversion** tries to reconstruct sensitive training attributes or inputs, and **membership inference** tries to determine whether a specific record was used during training.

In practice, the attacks often overlap. A public API, a weakly protected internal endpoint, or an overexposed prediction service can leak enough signal for an attacker to infer model behavior, training data, or both. Your goal is to understand what each attack reveals, how it works, and which controls reduce the risk.

**Core concepts**

- **Model extraction** copies model behavior by querying an interface and training a substitute model.
- **Model inversion** uses outputs, gradients, or confidence scores to infer sensitive features of training data or inputs.
- **Membership inference** determines whether a target sample was part of the training set.
- **Confidence scores** and rich prediction outputs increase leakage risk.
- **Overfitting** often makes membership inference easier because the model memorizes training examples.
- **API exposure** is a common attack surface for all three attack types.

**Model extraction**

Model extraction, sometimes called model stealing, aims to replicate a target model’s decision boundary or predictions. The attacker does not need the original weights if they can query the model repeatedly and observe outputs. With enough queries, they can train a surrogate model that behaves similarly.

This matters because the model may represent expensive research, proprietary logic, or a competitive advantage. Even if the attacker does not recover the exact weights, a close functional clone can still cause business loss and enable downstream abuse.

Typical extraction workflow:

1. Query the target model with many inputs.
2. Record outputs, labels, or confidence scores.
3. Use the collected data to train a substitute model.
4. Validate the substitute against the target on held-out inputs.
5. Refine queries to improve coverage of the target’s decision space.

Example of a simple black-box query loop:

```python
import requests
import random

endpoint = "https://api.example.com/predict"

for _ in range(1000):
    sample = {"text": f"test input {random.randint(1, 100000)}"}
    r = requests.post(endpoint, json=sample, timeout=5)
    print(r.json())
```

A more realistic extraction attempt uses adaptive sampling. The attacker starts with random inputs, then focuses on regions where the model is uncertain or where outputs change sharply. That helps approximate the decision boundary more efficiently than brute force.

**Why extraction works**

- Many models expose enough output detail to reconstruct behavior.
- Prediction APIs often return probabilities, rankings, or logits instead of only the final label.
- Large query volumes can reveal patterns even when individual responses seem harmless.
- Public-facing services may lack rate limits, authentication, or anomaly detection.

**Indicators of extraction attempts**

- High-volume, systematic API calls from a small set of clients.
- Repeated queries with minor input variations.
- Unusual coverage of edge cases or random inputs.
- Requests designed to probe confidence thresholds or class boundaries.

**Model inversion**

Model inversion attempts to recover sensitive information about the training data or a specific input by exploiting model outputs. The attacker may use class probabilities, gradients, embeddings, or other intermediate signals to infer attributes that were never intended to be exposed.

A classic example is reconstructing a representative face image from a face recognition model or inferring private attributes such as age, health status, or membership in a protected group. The exact result may not be a perfect copy, but even partial reconstruction can reveal sensitive information.

Model inversion is especially concerning when the model is trained on personal, medical, financial, or biometric data. If the model leaks enough signal, an attacker can infer details about individuals who never consented to disclosure.

Common inversion techniques:

- **Gradient-based reconstruction** uses gradients or model updates to optimize an input until the model output matches a target.
- **Confidence-based inversion** uses class probabilities to infer features associated with a target class.
- **Embedding inversion** maps latent vectors back into approximate input space.
- **Generative priors** improve reconstruction quality by constraining outputs to realistic samples.

Example of a simplified gradient-style reconstruction concept:

```python
# Conceptual example only
# Start with random noise and iteratively adjust it to maximize a target class score
x = initialize_noise()
for step in range(1000):
    score = model.predict_proba(x)[target_class]
    grad = compute_gradient(score, x)
    x = x + learning_rate * grad
```

This is not a production exploit by itself, but it shows the basic idea: if the model reveals enough signal, optimization can push a synthetic input toward a sensitive target.

**Why inversion works**

- Models can memorize rare or unique training examples.
- Rich outputs expose more information than a single label.
- Intermediate representations may preserve sensitive structure.
- Overparameterized models can encode details that are not obvious from the final prediction.

**What inversion can reveal**

- Faces, voices, or other biometric traits.
- Medical or demographic attributes.
- Sensitive text fragments or document characteristics.
- Hidden features correlated with a target class.

**Membership inference**

Membership inference asks a narrower question: was this specific record in the training set? The attacker does not need to reconstruct the record. They only need to determine whether the model behaves differently for members versus non-members.

This attack is often successful when the model overfits. Training examples tend to produce higher confidence, lower loss, or more stable predictions than unseen examples. An attacker can compare the target’s response to a threshold or to a shadow model trained on similar data.

A common membership inference approach uses shadow models:

1. Train one or more shadow models that mimic the target’s behavior.
2. Collect outputs for known member and non-member samples.
3. Train an attack model to distinguish the two cases.
4. Apply the attack model to the target’s outputs.

Example of a simple threshold-based membership test:

```python
def is_member(confidence, threshold=0.95):
    return confidence >= threshold

print(is_member(0.98))  # likely member
print(is_member(0.62))  # likely non-member
```

This example is simplified, but it captures the idea that unusually high confidence can indicate membership. Real attacks often use multiple features, such as loss, entropy, top-k probabilities, and prediction stability.

**Why membership inference works**

- Overfitting creates a measurable gap between training and non-training samples.
- Confidence scores leak more than final labels.
- Models trained on small or sensitive datasets are easier to attack.
- Repeated queries can reveal output consistency patterns.

**Signals that increase membership risk**

- Per-class probabilities or logits.
- Loss values returned by the service.
- Fine-grained embeddings or similarity scores.
- Personalized outputs that differ based on training exposure.

**Comparison of the three attacks**

| Attack | Primary goal | What the attacker learns | Common signal used |
|---|---|---|---|
| Model extraction | Clone model behavior | Approximate model function | Query outputs, probabilities |
| Model inversion | Recover sensitive input traits | Training data attributes or reconstructed samples | Gradients, embeddings, confidence scores |
| Membership inference | Determine training-set inclusion | Whether a record was used in training | Loss, confidence, entropy |

The attacks are related but not identical. Extraction focuses on the model as an asset, inversion focuses on the data behind the model, and membership inference focuses on a yes/no privacy question about a specific sample.

**Attack surface and real-world examples**

Public AI services often expose one or more of these surfaces:

- Prediction APIs that return probabilities or ranked classes.
- Chat or assistant endpoints that reveal too much context.
- Embedding endpoints that expose semantic vectors.
- Fine-tuning or training endpoints that leak gradients or loss.
- Debug endpoints, logs, or telemetry that include sensitive outputs.

A common enterprise risk is a model deployed behind an API gateway but still returning detailed metadata. Even if the gateway is authenticated, a compromised account or insider can still abuse the service at scale.

**Defensive controls**

The best defense is layered. No single control fully prevents extraction, inversion, or membership inference.

- **Limit output detail**
  - Return only the final label when possible.
  - Avoid exposing logits, full probability vectors, or internal scores.
  - Round or bucket confidence values if they are not required.

- **Rate limit and monitor**
  - Restrict query volume per user, token, IP, or tenant.
  - Detect repeated probing, random sampling, and boundary-search patterns.
  - Alert on unusual access to prediction endpoints.

- **Reduce overfitting**
  - Use regularization, early stopping, and proper validation.
  - Increase training data diversity where possible.
  - Avoid memorizing rare records or unique identifiers.

- **Use privacy-preserving training**
  - Apply **differential privacy** when the use case allows it.
  - Consider privacy-aware optimization and noise injection.
  - Minimize retention of raw training data and intermediate artifacts.

- **Harden the API**
  - Require authentication and authorization.
  - Use least privilege for service accounts and internal callers.
  - Separate public inference endpoints from internal admin or debug endpoints.

- **Protect model assets**
  - Store weights securely and restrict access.
  - Sign and verify model artifacts.
  - Track model lineage and deployment integrity.

- **Limit side channels**
  - Avoid verbose error messages.
  - Prevent timing differences that reveal confidence or class structure.
  - Sanitize logs and telemetry.

**Practical API hardening example**

A simple inference service should avoid returning unnecessary metadata:

```json
{
  "label": "approved"
}
```

Instead of:

```json
{
  "label": "approved",
  "confidence": 0.9974,
  "logits": [4.2, -1.1, 0.3],
  "embedding": [0.12, 0.44, -0.08]
}
```

The second response gives an attacker far more information for extraction, inversion, and membership inference.

**Example rate limiting with NGINX**

```nginx
limit_req_zone $binary_remote_addr zone=ai_api:10m rate=5r/s;

server {
    location /predict {
        limit_req zone=ai_api burst=10 nodelay;
        proxy_pass http://model_backend;
    }
}
```

This does not stop a determined attacker, but it raises the cost of large-scale probing.

**Example of safer model serving in Python**

```python
from flask import Flask, request, jsonify

app = Flask(__name__)

@app.post("/predict")
def predict():
    data = request.get_json(force=True)
    label = model.predict(data["features"])
    return jsonify({"label": label})
```

If you must expose confidence, consider coarse buckets instead of exact values:

```python
def bucket_confidence(p):
    if p >= 0.9:
        return "high"
    if p >= 0.6:
        return "medium"
    return "low"
```

**Detection and response**

You should treat these attacks as both security and privacy incidents. Detection often depends on telemetry from the API layer, model service, and identity provider.

Useful monitoring ideas:

- Track request frequency, entropy, and input similarity.
- Flag repeated near-duplicate queries.
- Compare caller behavior against normal application usage.
- Alert on access to unusual endpoints or debug parameters.
- Correlate spikes in inference traffic with model export or admin activity.

Response actions may include:

1. Disable or throttle the affected endpoint.
2. Rotate credentials and revoke suspicious tokens.
3. Review logs for query patterns and data exposure.
4. Retrain or fine-tune the model if leakage is confirmed.
5. Reduce output detail and redeploy with stronger controls.

**Exam focus**

- **Model extraction** = stealing model behavior through queries.
- **Model inversion** = inferring sensitive inputs or attributes from outputs or gradients.
- **Membership inference** = determining whether a sample was in training data.
- **Overfitting** and **rich outputs** increase privacy risk.
- **Differential privacy**, output minimization, and rate limiting are key mitigations.
- **API security** matters because most attacks begin with repeated queries.

**Quick memory aid**

- **Extraction**: copy the model.
- **Inversion**: recover the data.
- **Membership**: test inclusion.

**Key takeaways**

These attacks exploit information leakage from AI systems, not just code flaws. If your model returns too much detail, overfits its training data, or accepts unlimited probing, it becomes easier to steal, reverse engineer, or use against the people whose data trained it.

Design your AI service to reveal only what users need, protect training data as sensitive, and assume that any exposed prediction interface can be studied at scale.
