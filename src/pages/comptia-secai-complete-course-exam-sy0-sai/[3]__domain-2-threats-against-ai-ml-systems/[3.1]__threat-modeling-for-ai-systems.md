## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Threat modeling for AI systems**

Threat modeling is a structured way to identify what can go wrong in an AI system before attackers, misuse, or failures cause harm. For AI, the model must cover not only traditional software risks, but also data poisoning, prompt injection, model extraction, privacy leakage, unsafe outputs, and supply-chain issues in models, datasets, and plugins.

A useful exam distinction is this:

- **Threat modeling** asks what can be attacked, how, and where controls should go.
- **Risk assessment** asks how likely and how damaging those threats are, then prioritizes them.
- In practice, threat modeling feeds risk assessment; risk assessment turns identified threats into a ranked mitigation plan.

For AI systems, you usually model the full lifecycle:

- Data collection and labeling
- Training and fine-tuning
- Model storage and deployment
- Inference and user interaction
- Monitoring, logging, and retraining
- Third-party integrations and downstream consumers

**Core objectives**

A good AI threat model helps you:

- Identify assets worth protecting.
- Map system boundaries and trust boundaries.
- Enumerate entry points and attack paths.
- Find threats using a repeatable method.
- Rank risks and choose mitigations that fit the business and compliance context.

**AI assets to enumerate**

Start by listing the things an attacker would want, or the things whose compromise would cause harm.

- **Training data**: proprietary, personal, regulated, or sensitive data used to train or tune the model.
- **Model weights and artifacts**: the trained model, checkpoints, embeddings, and adapters.
- **Prompts and system instructions**: hidden prompts, policies, guardrails, and tool instructions.
- **Inference inputs and outputs**: user prompts, retrieved context, generated responses, and logs.
- **Feature stores and vector databases**: retrieval indexes, embeddings, and metadata.
- **Secrets and credentials**: API keys, service tokens, signing keys, and database credentials.
- **Infrastructure**: GPUs, containers, orchestration, CI/CD pipelines, and storage.
- **Governance artifacts**: policies, approval records, audit logs, and model cards.

A practical rule is to treat anything that influences model behavior or reveals sensitive information as an asset.

**System boundary and trust boundaries**

The **system boundary** defines what is inside the AI solution you are protecting. The **trust boundary** marks where trust changes, such as between a user and the application, or between the application and a third-party model API.

Common trust boundaries in AI systems include:

- User device to web app
- Web app to API gateway
- Application to model serving endpoint
- Application to retrieval system or vector store
- Internal services to external SaaS or foundation model providers
- Training environment to production environment

Trust boundaries matter because threats often appear when data crosses them. For example, a prompt from an untrusted user becomes dangerous when it reaches a tool-using agent with privileged access.

**Diagramming artifacts you should know**

Threat modeling usually starts with a **data flow diagram (DFD)**. A DFD shows how data moves through the system and where trust changes.

Typical DFD elements:

- **External entities**: users, admins, attackers, partner systems
- **Processes**: API service, prompt router, model inference service, retriever, policy engine
- **Data stores**: training corpus, vector database, logs, model registry
- **Data flows**: prompts, embeddings, responses, telemetry, training updates
- **Trust boundaries**: lines separating different trust zones

A simple AI DFD might look like this:

```text
[User] --> (Web App) --> (Prompt Orchestrator) --> (LLM API)
                    \--> (Vector DB)
                    \--> (Audit Log)
```

A more complete view includes training and governance:

```text
[Data Sources] --> (Ingestion) --> [Training Data Store] --> (Training Pipeline) --> [Model Registry]
                                                           --> (Evaluation) --> [Approval Gate]
[User] --> (UI/API) --> (Policy Engine) --> (Inference Service) --> (LLM/Model)
                                 \--> [Vector DB] <--> (Retriever)
                                 \--> [Logs/SIEM]
```

When you build the diagram, explicitly mark:

- Assets
- Entry points
- Trust boundaries
- Privileged components
- External dependencies

**Entry-point enumeration**

Entry points are the places an attacker can interact with the system. In AI systems, they are often broader than in traditional apps.

Common entry points:

- Chat UI and API endpoints
- File upload endpoints
- Batch inference jobs
- Retrieval queries
- Plugin or tool invocation interfaces
- Admin consoles
- Model update and retraining pipelines
- Webhooks and event consumers
- SDKs and client libraries

For exam purposes, remember that an entry point is not only a login page. Any interface that accepts input, data, or commands can be an entry point.

**Threat modeling methodologies**

Several methodologies are used in practice. You do not need to memorize every framework in depth, but you should know how they guide analysis.

- **STRIDE**: Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege.
- **PASTA**: Process for Attack Simulation and Threat Analysis; risk-centric and business-focused.
- **LINDDUN**: Privacy threat modeling framework; useful when personal data is involved.
- **Attack trees**: Break an attacker goal into subgoals and paths.
- **Kill chain / attack path analysis**: Useful for understanding how an attacker progresses through the system.
- **Misuse cases**: User-story style abuse scenarios.
- **OWASP-style AI threat catalogs**: Helpful for prompt injection, data leakage, and model abuse patterns.

A strong exam answer often combines methods:

- Use DFDs to map the system.
- Use STRIDE to enumerate threats.
- Use risk scoring to prioritize.
- Use privacy or governance frameworks when regulated data is involved.

**Applying STRIDE to an AI system**

STRIDE works well because it forces you to ask a different question for each threat class.

- **Spoofing**
  - Fake users, fake services, or fake model endpoints.
  - Example: an attacker impersonates a trusted retrieval service and injects malicious context.
- **Tampering**
  - Altering prompts, training data, model artifacts, or logs.
  - Example: poisoning a dataset or modifying a model checkpoint.
- **Repudiation**
  - Denying actions because logs are missing or weak.
  - Example: a user claims they never submitted a harmful prompt, but audit trails are incomplete.
- **Information disclosure**
  - Leaking training data, system prompts, secrets, or private outputs.
  - Example: prompt injection causes the model to reveal hidden instructions.
- **Denial of service**
  - Exhausting tokens, compute, memory, or rate limits.
  - Example: long prompts or repeated tool calls drive up cost and latency.
- **Elevation of privilege**
  - Gaining access to tools, data, or admin functions beyond intended scope.
  - Example: a prompt injection tricks an agent into calling a privileged internal API.

A practical STRIDE workflow for AI:

1. Draw the DFD.
2. Mark trust boundaries and entry points.
3. For each process, data store, and flow, ask the six STRIDE questions.
4. Record threats with affected assets and likely impact.
5. Select mitigations and assign owners.

**Worked STRIDE example: customer support chatbot**

Suppose you have a chatbot that answers support questions using a vector database and can create tickets through an internal tool.

System components:

- Web chat UI
- API gateway
- Prompt orchestrator
- Retrieval service
- Vector database
- LLM provider
- Ticketing tool
- Audit log

Threat enumeration:

- **Spoofing**: attacker uses stolen session tokens to access the chatbot as a valid user.
- **Tampering**: attacker uploads malicious documents to poison retrieval results.
- **Repudiation**: support agents can create tickets, but actions are not tied to strong identity or immutable logs.
- **Information disclosure**: the model reveals internal policy text or customer data from retrieved documents.
- **Denial of service**: attacker sends oversized prompts and forces repeated retrieval calls.
- **Elevation of privilege**: prompt injection convinces the agent to create tickets or query internal systems outside policy.

Mitigations:

- Strong authentication and session binding
- Input validation and file scanning
- Retrieval access controls and document provenance checks
- Prompt hardening and tool-use allowlists
- Rate limiting and token budgets
- Centralized logging with tamper-evident audit trails

**Threats specific to AI systems**

AI systems introduce threat patterns that are less common in ordinary web apps.

- **Prompt injection**
  - Malicious text manipulates the model or agent into ignoring instructions.
  - Can be direct, where the user prompt is malicious, or indirect, where malicious content is embedded in retrieved documents or web pages.
- **Data poisoning**
  - Training or retrieval data is manipulated to bias outputs or insert backdoors.
- **Model extraction**
  - Attackers query the model to reconstruct behavior or steal intellectual property.
- **Membership inference**
  - Attackers infer whether a specific record was in the training set.
- **Model inversion**
  - Attackers recover sensitive features or approximate training examples.
- **Jailbreaks**
  - Users bypass safety policies through crafted prompts.
- **Tool abuse**
  - Agents misuse APIs, file systems, browsers, or internal services.
- **Hallucination exploitation**
  - Attackers rely on fabricated outputs to trigger unsafe actions or bad decisions.
- **Supply-chain compromise**
  - Malicious models, dependencies, datasets, or plugins are introduced into the pipeline.

**Threat modeling an AI agent with tools**

Agents are especially important because they can take actions, not just generate text. That increases the attack surface.

Key questions:

- What tools can the agent call?
- What permissions does each tool have?
- Can the agent access secrets, files, or internal APIs?
- Are tool calls reviewed, constrained, or sandboxed?
- Can untrusted content influence tool selection or parameters?

Example agent risks:

- A prompt injection in a webpage causes the agent to exfiltrate a file.
- A malicious tool response causes the agent to chain into another privileged action.
- A poorly scoped API token lets the agent perform destructive operations.

Mitigations:

- Least privilege for tool credentials
- Human approval for high-impact actions
- Output filtering before tool execution
- Sandboxed execution environments
- Strict schemas for tool inputs and outputs
- Separate read-only and write-capable tools

**Privacy, governance, and compliance considerations**

AI threat models are often shaped by legal and policy requirements. These are not optional extras; they influence what counts as a threat and what mitigations are acceptable.

Common considerations:

- **Privacy**: personal data in prompts, logs, embeddings, or training sets may require minimization, retention limits, and access controls.
- **Regulatory obligations**: sector rules or regional privacy laws may require auditability, consent, purpose limitation, or data subject rights handling.
- **Policy requirements**: internal policies may prohibit certain data from being sent to external model providers.
- **Data residency**: some workloads must keep data in specific regions.
- **Retention and deletion**: logs, prompts, and training artifacts may need lifecycle controls.
- **Explainability and accountability**: high-impact decisions may require traceability and human review.
- **Vendor governance**: third-party model providers may need security review, contractual controls, and monitoring.

In practice, compliance requirements change the threat model because they define additional assets and failure modes. For example, sending regulated data to a public model API may be both a privacy risk and a policy violation.

**Risk assessment versus threat modeling**

These terms are related but not identical.

| Aspect | Threat modeling | Risk assessment |
|---|---|---|
| Primary question | What can go wrong and where? | Which threats matter most? |
| Output | Threat list, attack paths, mitigations | Ranked risks, treatment decisions |
| Focus | System design and attack surface | Likelihood, impact, business priority |
| Timing | Early and iterative | Often after threats are identified |
| AI example | Prompt injection on retrieval flow | High-risk because it can expose regulated data and trigger tool abuse |

A practical workflow is:

- Model the system.
- Identify threats.
- Estimate likelihood and impact.
- Decide whether to mitigate, transfer, accept, or avoid the risk.
- Revisit the model when the system changes.

**Risk ranking and mitigation selection**

After you identify threats, rank them using a consistent method. You can use qualitative labels such as low, medium, and high, or a numeric matrix.

Factors to consider:

- Likelihood of exploitation
- Impact on confidentiality, integrity, and availability
- Impact on safety and business operations
- Exposure of regulated or sensitive data
- Ease of detection
- Existing controls and compensating controls

Mitigation choices usually fall into these categories:

- **Prevent**: block the attack path.
- **Detect**: alert on suspicious behavior.
- **Contain**: limit blast radius.
- **Recover**: restore service or data.
- **Accept**: document and approve residual risk.

Example ranking:

- Prompt injection against a public chatbot may be high likelihood and high impact.
- Model extraction against a rate-limited internal model may be medium likelihood and medium impact.
- Membership inference against a small, sensitive training set may be lower likelihood but high impact.

**Worked end-to-end example: AI document assistant**

Scenario: An internal AI assistant summarizes HR documents and answers employee questions. It uses a vector database, a hosted LLM, and a ticketing integration.

1. **Define scope**
   - In scope: web UI, API, retrieval pipeline, vector DB, LLM API, logs, ticketing tool.
   - Out of scope: unrelated corporate systems not reachable from the assistant.

2. **Draw the DFD**
   - Employee -> UI -> orchestrator -> retriever -> vector DB -> LLM -> response
   - Orchestrator -> ticketing tool
   - Orchestrator -> audit log

3. **Mark trust boundaries**
   - Employee to internal app
   - Internal app to external LLM provider
   - Internal app to ticketing SaaS
   - Training/ingestion pipeline to production retrieval store

4. **Enumerate assets**
   - HR policies, employee records, prompt templates, API keys, logs, embeddings

5. **Identify threats**
   - Prompt injection in uploaded documents
   - Unauthorized access to HR content
   - Leakage of employee data in logs
   - Abuse of ticket creation tool
   - Poisoned documents in the retrieval corpus

6. **Rank risks**
   - Prompt injection leading to disclosure of HR data: high
   - Ticketing abuse: medium to high
   - Log leakage of personal data: high
   - Poisoned corpus from untrusted uploads: high

7. **Select mitigations**
   - Document provenance and approval workflow
   - Retrieval filtering by role and document classification
   - Redaction of sensitive fields before logging
   - Tool allowlists and human approval for ticket creation
   - Content scanning and quarantine for uploads
   - Rate limiting and anomaly detection

8. **Validate**
   - Test with malicious prompts and poisoned documents
   - Verify logs do not store unnecessary personal data
   - Confirm the assistant cannot access unauthorized documents
   - Reassess after each major change

This is the kind of end-to-end reasoning exam questions often expect: boundary, assets, threats, risk, and mitigation in one coherent flow.

**Practical controls for AI threat mitigation**

Controls should match the threat and the trust boundary.

- **Authentication and authorization**
  - Use strong identity, role-based access control, and scoped service accounts.
- **Input validation**
  - Sanitize prompts, files, URLs, and tool parameters.
- **Content filtering**
  - Detect sensitive data, malware, and policy violations.
- **Prompt hardening**
  - Separate system instructions from user content and retrieved content.
- **Tool governance**
  - Use allowlists, schema validation, and approval gates.
- **Data protection**
  - Encrypt data at rest and in transit; minimize retention.
- **Monitoring**
  - Log prompt patterns, tool calls, and anomaly signals.
- **Rate limiting**
  - Reduce brute force, extraction, and cost abuse.
- **Segmentation**
  - Isolate training, staging, and production environments.
- **Supply-chain security**
  - Verify model provenance, dependency integrity, and dataset lineage.

**Example hardening pattern for an agent**

```python
ALLOWED_TOOLS = {"search_docs", "create_ticket"}

def execute_tool(tool_name, args, user_role):
    if tool_name not in ALLOWED_TOOLS:
        raise PermissionError("Tool not allowed")

    if tool_name == "create_ticket" and user_role != "support_agent":
        raise PermissionError("Insufficient privileges")

    validate_schema(tool_name, args)
    return call_tool(tool_name, args)
```

This pattern is simple, but it captures the key idea: the model does not get unrestricted action just because it can generate a tool call.

You may use standard security tooling to support the threat model.


These commands do not replace threat modeling, but they help validate whether the implementation matches the intended design.

**Common exam cues**

Look for wording that implies one of these tasks:

- “Identify assets and trust boundaries” means start with a DFD.
- “Find ways an attacker could manipulate the model” often points to prompt injection or poisoning.
- “Rank and prioritize” points to risk assessment after threat identification.
- “Protect user data and comply with policy” points to privacy and governance controls.
- “Agent can call tools” means you should consider privilege escalation and tool abuse.

**Quick memory aid**

- **DFD**: draw the data flow.
- **Assets**: list what matters.
- **Boundaries**: mark where trust changes.
- **STRIDE**: enumerate threat classes.
- **Risk**: rank by likelihood and impact.
- **Mitigate**: prevent, detect, contain, recover.

**Key takeaways**

Threat modeling for AI systems is broader than traditional application modeling because the model itself, its data, and its tool ecosystem all create attack surface. The most effective approach is to map the system with a DFD, identify assets and trust boundaries, enumerate entry points, apply a structured method such as STRIDE, and then rank the resulting risks in light of privacy, governance, and compliance requirements.
