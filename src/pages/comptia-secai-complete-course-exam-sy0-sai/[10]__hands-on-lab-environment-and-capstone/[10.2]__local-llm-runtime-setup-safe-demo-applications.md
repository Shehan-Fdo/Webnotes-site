## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Local LLM runtime fundamentals**

A local <abbr title="Large Language Model">LLM</abbr> runtime lets you run model inference on your own workstation, server, or private cloud instead of sending prompts to a third-party API. This is useful when you need lower latency, offline operation, tighter data control, or a controlled demo environment. It also shifts responsibility to you for model trust, patching, access control, and resource management.

Common runtime choices include:
- **Ollama** for simple local model management and serving.
- **llama.cpp** for CPU-friendly or quantized model inference.
- **vLLM** for higher-throughput GPU serving.
- **Text Generation Inference (TGI)** for production-oriented model serving.
- **Docker** or **Podman** to isolate the runtime and its dependencies.

A safe demo setup starts with a clear separation between:
- The **model artifact** you download.
- The **runtime** that loads and serves it.
- The **application** that calls the runtime.
- The **network boundary** that controls who can reach it.

**Model source trust and supply-chain risk**

Model files are software artifacts and should be treated like untrusted dependencies. A model repository can be compromised, a mirror can be stale, and a package can be tampered with in transit or after publication. You should verify both the source and the artifact before using it in a demo or lab.

Key checks include:
- **Repository reputation**: prefer official model hubs, vendor-maintained repos, or well-known community sources with active maintenance.
- **Publisher identity**: confirm the account or organization that published the model.
- **Artifact integrity**: compare published hashes when available, and verify signatures if the source provides them.
- **File format safety**: prefer formats and loaders that are widely used and actively maintained.
- **Dependency provenance**: inspect runtime packages, container images, and Python wheels for unexpected changes.

Practical verification examples:
```bash
# Verify a downloaded file hash
sha256sum model.gguf

# Compare against a known-good checksum
echo "expected_sha256  model.gguf" | sha256sum -c -

# Inspect a container image digest
docker image inspect ollama/ollama:latest --format '{{.RepoDigests}}'
```

For higher assurance, pin exact versions and digests instead of floating tags:
```bash
docker pull ollama/ollama@sha256:exampledigestvalue
```

> Do not trust `latest` tags for repeatable demos. Floating tags can change without notice and may introduce regressions or malicious updates.

**Runtime installation and basic setup**

You usually install the runtime on a dedicated host or inside a container. The goal is to keep the environment reproducible and easy to reset.

Example Docker-based setup:
```bash
docker run -d \
  --name local-llm \
  -p 127.0.0.1:11434:11434 \
  -v ollama_data:/root/.ollama \
  ollama/ollama:latest
```

Why this pattern helps:
- Binding to `127.0.0.1` keeps the service local by default.
- A named volume persists models across restarts.
- Containerization reduces dependency drift on the host.

Example host-level checks:
```bash
curl http://127.0.0.1:11434/api/tags
ps aux | grep -i ollama
ss -lntp | grep 11434
```

If you use a package manager or installer, verify:
- The package signature or repository metadata.
- The version you installed.
- The service account and file permissions created by the installer.

**Model selection and quantization**

Local runtimes often use quantized models to reduce memory and compute requirements. Quantization lowers precision to make models smaller and faster, but it can reduce quality depending on the task.

Common tradeoffs:
- **Smaller quantized models**: better for laptops, edge devices, and demos.
- **Larger models**: better reasoning and generation quality, but higher RAM and GPU needs.
- **CPU inference**: simpler to deploy, slower for large models.
- **GPU inference**: faster, but requires driver compatibility and stronger resource controls.

A practical selection process:
1. Choose a model size that fits your hardware with headroom.
2. Prefer a known-good quantization level for your runtime.
3. Test latency, memory use, and output quality on representative prompts.
4. Keep a fallback model for low-resource environments.

**Safe demo application architecture**

A demo app should not expose the raw model endpoint directly to the internet. Instead, place a controlled application layer in front of the runtime.

A common secure topology is:
- User browser or internal client
- Reverse proxy with TLS and authentication
- Demo application
- Local LLM runtime on a private network or localhost

This design lets you:
- Enforce authentication and authorization.
- Filter prompts and outputs.
- Log usage without exposing the model service.
- Rate-limit requests to prevent abuse.

Example reverse proxy flow:
```text
Client -> Nginx/Traefik/Caddy -> Demo App -> Local LLM Runtime
```

**Authentication and authorization for local APIs**

Even local APIs need access control when they are reachable from other hosts, containers, or VPN users. Authentication proves identity; authorization limits what that identity can do.

Useful options include:
- **API keys** for simple service-to-service access.
- **Bearer tokens** for short-lived authenticated sessions.
- **Reverse proxy auth** using basic auth, SSO, or an identity-aware proxy.
- **mTLS** (<abbr title="mutual Transport Layer Security">mutual TLS</abbr>) for strong machine-to-machine trust.
- **Network ACLs** and firewall rules to restrict source IPs.

Example API key check in a demo app:
```python
from fastapi import FastAPI, Header, HTTPException

app = FastAPI()
API_KEY = "change-me"

@app.get("/chat")
def chat(x_api_key: str = Header(default="")):
    if x_api_key != API_KEY:
        raise HTTPException(status_code=401, detail="Unauthorized")
    return {"status": "ok"}
```

Better practice is to avoid hardcoded secrets and load them from environment variables:
```bash
export DEMO_API_KEY='replace-with-strong-secret'
```

Example reverse proxy basic auth with Nginx:
```nginx
server {
    listen 443 ssl;
    server_name demo.example.com;

    ssl_certificate     /etc/ssl/certs/fullchain.pem;
    ssl_certificate_key /etc/ssl/private/privkey.pem;

    location / {
        auth_basic "Restricted";
        auth_basic_user_file /etc/nginx/.htpasswd;

        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

For stronger remote access, place the app behind:
- A VPN.
- An identity-aware proxy.
- A zero-trust access gateway.

**Secrets management for demo apps**

Demo apps often fail because credentials are embedded in code, notebooks, or container images. Treat secrets as sensitive data and keep them out of source control.

Recommended patterns:
- Use environment variables for local development.
- Store local overrides in `.env` files that are excluded from Git.
- Use a secrets manager or vault for shared environments.
- Rotate secrets after demos, especially if they were exposed in logs or screenshots.

Example `.env` usage:
```env
DEMO_API_KEY=replace-with-strong-secret
LLM_BASE_URL=http://127.0.0.1:11434
DATABASE_URL=postgresql://demo_user:strongpass@db:5432/demo
```

Example Python loading pattern:
```python
import os
from dotenv import load_dotenv

load_dotenv()

api_key = os.getenv("DEMO_API_KEY")
base_url = os.getenv("LLM_BASE_URL")
```

Operational rules:
- Add `.env` to `.gitignore`.
- Never commit tokens, private keys, or cloud credentials.
- Prefer short-lived tokens over long-lived static secrets when possible.
- Use separate secrets for dev, test, and demo environments.

**Sandboxing and OS-level hardening**

A local runtime should be isolated from the rest of the system as much as practical. Sandboxing reduces the impact of a compromised model server, malicious plugin, or vulnerable dependency.

Useful controls include:
- **Containers** with minimal images and non-root users.
- **AppArmor** or **SELinux** to confine process behavior on Linux.
- **seccomp** to restrict dangerous system calls.
- **Windows Defender Application Control** or similar application control on Windows.
- **Filesystem restrictions** to limit write access to only required paths.
- **Read-only mounts** for model files when updates are not needed.

Example Docker hardening options:
```bash
docker run -d \
  --name local-llm \
  --read-only \
  --cap-drop ALL \
  --security-opt no-new-privileges:true \
  --pids-limit 256 \
  -p 127.0.0.1:11434:11434 \
  -v ollama_data:/root/.ollama \
  ollama/ollama:latest
```

Why these controls matter:
- Dropping capabilities reduces kernel-level attack surface.
- `no-new-privileges` blocks privilege escalation through setuid-style paths.
- `--pids-limit` helps contain fork bombs and runaway processes.
- Read-only root filesystems reduce persistence opportunities.

On Linux hosts, you can also apply AppArmor or SELinux profiles to limit file, network, and process access. Use the most restrictive profile that still allows the runtime to function.

**Resource governance and denial-of-service prevention**

A demo environment can be exhausted by large prompts, repeated requests, or too many concurrent users. Resource governance keeps the system responsive and prevents one user from starving others.

Controls to consider:
- **CPU limits** to cap compute usage.
- **GPU memory limits** where supported by the platform.
- **Container memory limits** to prevent host exhaustion.
- **Concurrency limits** on the API or queue.
- **Request size limits** for prompt and file uploads.
- **Rate limiting** per user, token, or IP address.

Example Docker resource limits:
```bash
docker run -d \
  --name local-llm \
  --cpus="4.0" \
  --memory="8g" \
  --pids-limit 256 \
  -p 127.0.0.1:11434:11434 \
  ollama/ollama:latest
```

Example Nginx rate limiting:
```nginx
http {
    limit_req_zone $binary_remote_addr zone=demo_limit:10m rate=5r/s;

    server {
        location / {
            limit_req zone=demo_limit burst=10 nodelay;
            proxy_pass http://127.0.0.1:8000;
        }
    }
}
```

For GPU-backed systems, also watch:
- VRAM exhaustion.
- Driver resets.
- Thermal throttling.
- Multi-tenant contention.

**Logging, monitoring, and prompt safety**

Logs help you troubleshoot and detect abuse, but they can also leak sensitive prompts, outputs, or secrets. Log only what you need and redact sensitive fields.

Good logging practice:
- Record request IDs, timestamps, user IDs, and latency.
- Avoid storing full prompts unless required.
- Redact secrets, tokens, and regulated data.
- Separate operational logs from content logs.

Prompt safety controls for demos:
- Block obvious secret patterns.
- Limit file uploads and external tool use.
- Disable internet access unless the demo requires it.
- Add output filters for unsafe or policy-violating content when appropriate.

Example structured logging fields:
```json
{
  "request_id": "b7f1",
  "user": "demo-user",
  "model": "llama3.1:8b",
  "latency_ms": 842,
  "status": "ok"
}
```

**Data governance and regulated data handling**

Local does not automatically mean compliant. If you process personal data, health data, financial data, or other regulated content, you still need governance controls.

Key governance questions:
- What data classes are allowed in the demo?
- How long are prompts, outputs, and logs retained?
- Who can access transcripts and telemetry?
- Where is data stored, and is it encrypted?
- Is the environment approved for regulated data at all?

Practical controls:
- Define a **retention policy** for logs, chat history, and uploaded files.
- Classify data before it enters the demo environment.
- Use synthetic or anonymized data for testing.
- Disable persistence for sensitive sessions when possible.
- Encrypt disks and backups.
- Delete temporary files after use.

> If the environment is not approved for regulated data, do not use real customer, patient, or payment information in the demo.

**Update, patch, and vulnerability management**

Local runtimes age quickly because they depend on kernels, drivers, container engines, Python packages, model-serving libraries, and model artifacts. You need a patch process for all of them.

Track and update:
- The host OS and kernel.
- GPU drivers and CUDA or ROCm components.
- Container runtime and base images.
- Python or Node.js dependencies.
- The LLM server itself.
- Model files when a safer or better-maintained version is available.

Operational practices:
- Subscribe to vendor advisories and CVE feeds.
- Rebuild images regularly instead of patching ad hoc.
- Scan containers and dependencies before deployment.
- Retest prompts after updates because behavior can change.
- Keep rollback artifacts for known-good versions.

Example vulnerability scanning workflow:
```bash
docker scout cves ollama/ollama:latest
trivy image ollama/ollama:latest
pip-audit
```

**Practical secure deployment example**

A simple secure demo stack can run the model locally, expose only the app through a reverse proxy, and require authentication.

Example `docker-compose.yml`:
```yaml
services:
  llm:
    image: ollama/ollama:latest
    ports:
      - "127.0.0.1:11434:11434"
    volumes:
      - ollama_data:/root/.ollama
    deploy:
      resources:
        limits:
          cpus: "4.0"
          memory: 8G

  app:
    build: ./app
    environment:
      - LLM_BASE_URL=http://llm:11434
      - DEMO_API_KEY=${DEMO_API_KEY}
    depends_on:
      - llm
    expose:
      - "8000"

  proxy:
    image: nginx:stable
    ports:
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
      - ./certs:/etc/ssl:ro
    depends_on:
      - app

volumes:
  ollama_data:
```

Example app request flow:
```python
import os
import requests
from fastapi import FastAPI, Header, HTTPException

app = FastAPI()
API_KEY = os.getenv("DEMO_API_KEY")
LLM_BASE_URL = os.getenv("LLM_BASE_URL", "http://127.0.0.1:11434")

@app.post("/generate")
def generate(prompt: str, x_api_key: str = Header(default="")):
    if x_api_key != API_KEY:
        raise HTTPException(status_code=401, detail="Unauthorized")

    resp = requests.post(
        f"{LLM_BASE_URL}/api/generate",
        json={"model": "llama3.1:8b", "prompt": prompt, "stream": False},
        timeout=30,
    )
    return resp.json()
```

This pattern gives you:
- A private model endpoint.
- A single authenticated entry point.
- A place to add logging, filtering, and rate limits.
- Easier rollback and environment reset.

**Operational checklist**

Before you expose a local demo, verify:
1. The model source is trusted and the artifact hash or signature is checked.
2. The runtime is pinned to a known version or digest.
3. Secrets are loaded from environment variables or a vault, not hardcoded.
4. The service is bound to localhost or protected by a firewall and reverse proxy.
5. Authentication and authorization are enabled for all remote access.
6. Container, OS, and runtime sandboxing controls are in place.
7. CPU, memory, concurrency, and request limits are configured.
8. Logs avoid sensitive content and retention is defined.
9. The environment is not used for unapproved regulated data.
10. Patch and vulnerability monitoring are active.

**Exam focus reminders**

For exam scenarios, look for the safest design that balances usability and control:
- Prefer private network exposure over public exposure.
- Prefer authenticated reverse proxies over direct model ports.
- Prefer environment variables or vaults over hardcoded secrets.
- Prefer pinned versions and verified artifacts over floating tags.
- Prefer least privilege, sandboxing, and resource limits over default permissive settings.
