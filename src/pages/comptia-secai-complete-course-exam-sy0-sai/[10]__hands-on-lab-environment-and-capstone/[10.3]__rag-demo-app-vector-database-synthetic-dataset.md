**RAG demo app overview**

A Retrieval-Augmented Generation (RAG) demo app shows how a large language model (LLM) can answer questions using external knowledge instead of relying only on its training data. In practice, the app retrieves relevant documents from a vector database, injects them into the prompt, and then asks the model to generate a grounded response.

This pattern is useful when you need:
- More current information than the base model may know.
- Answers tied to internal policies, product docs, or case files.
- Better traceability for security, compliance, and audit use cases.
- A controlled environment for testing prompt injection, data leakage, and retrieval quality.

A demo app is usually a simplified reference implementation. It helps you validate the architecture, measure retrieval quality, and test security controls before building a production system.

**Core RAG flow**

RAG combines search and generation in a pipeline:

1. You ingest source content such as PDFs, wiki pages, tickets, or logs.
2. You split content into chunks so the system can retrieve smaller, relevant passages.
3. You convert each chunk into an embedding, which is a numeric vector representation of meaning.
4. You store vectors and metadata in a vector database.
5. A user submits a query.
6. The app embeds the query and performs similarity search.
7. The app retrieves top-matching chunks and adds them to the model prompt.
8. The LLM generates an answer using the retrieved context.

This approach reduces hallucination risk, but it does not eliminate it. If retrieval is poor, the model may still answer incorrectly or confidently invent details.

**Why vector databases matter**

A vector database stores embeddings and supports fast nearest-neighbor search. Unlike a traditional keyword index, it finds semantically similar content even when the query wording differs from the source text.

Key capabilities include:
- **Similarity search** for matching meaning rather than exact terms.
- **Metadata filtering** to narrow results by source, date, tenant, classification, or document type.
- **Hybrid retrieval** when you combine vector search with keyword search.
- **Scalability** for large corpora and frequent queries.
- **Index tuning** to balance speed, recall, and memory use.

Common vector index approaches include approximate nearest neighbor methods such as HNSW (Hierarchical Navigable Small World) or IVF (Inverted File Index). Exact search is simpler but often too slow at scale.

**Typical architecture**

A practical RAG demo app often includes these components:

- **User interface**
  - Chat UI, search box, or API endpoint.
  - Displays retrieved sources and generated answer.
- **Application layer**
  - Handles authentication, authorization, logging, and prompt assembly.
  - Applies guardrails and input validation.
- **Embedding service**
  - Converts text into vectors.
  - May be local or hosted.
- **Vector database**
  - Stores embeddings, chunk text, and metadata.
- **LLM inference endpoint**
  - Generates the final response.
- **Document pipeline**
  - Ingests, cleans, chunks, and indexes content.

A simple topology looks like this:

```text
User -> Demo App -> Query Embedding -> Vector DB -> Retrieved Chunks -> LLM -> Answer
                      ^                                      |
                      |                                      v
                 Metadata filters                     Citations / sources
```

**Synthetic dataset purpose**

A synthetic dataset is artificially generated data that mimics the structure and behavior of real data without exposing sensitive records. In RAG demos, synthetic data is often used to populate the knowledge base, test retrieval, and validate security controls.

Use synthetic data when you need to:
- Avoid using production secrets, customer records, or regulated data.
- Create repeatable test cases for retrieval and prompt behavior.
- Simulate edge cases such as conflicting facts, stale documents, or malicious content.
- Build demos before real content is approved for use.

Synthetic data is especially valuable in security training because it lets you test:
- Prompt injection attempts.
- Data exfiltration scenarios.
- Role-based access control failures.
- Cross-tenant retrieval mistakes.
- Unsafe summarization of restricted content.

> Synthetic data should resemble the target domain closely enough to test the system, but it must not contain real sensitive information or realistic secrets that could be mistaken for production data.

**Dataset design considerations**

Good synthetic datasets are not just random text. They should preserve the structure that retrieval depends on.

Important design elements:
- **Document variety** such as policies, FAQs, incident reports, runbooks, and tickets.
- **Chunkable structure** with headings, paragraphs, tables, and lists.
- **Metadata fields** like `source`, `department`, `classification`, `created_at`, and `tenant_id`.
- **Known ground truth** so you can measure whether retrieval returns the correct chunk.
- **Intentional ambiguity** to test disambiguation and ranking.
- **Adversarial samples** to test prompt injection and unsafe instructions.

Example metadata schema:

```json
{
  "doc_id": "hr-policy-014",
  "title": "Remote Access Policy",
  "source": "synthetic",
  "department": "HR",
  "classification": "internal",
  "created_at": "2026-01-15",
  "tenant_id": "demo-a",
  "chunk_id": 3
}
```

**Chunking and embedding**

Chunking is the process of splitting documents into smaller pieces before embedding. The goal is to keep each chunk semantically coherent and small enough for retrieval and prompt context limits.

Common chunking strategies:
- **Fixed-size chunking** with overlap for general-purpose text.
- **Semantic chunking** based on headings or topic boundaries.
- **Structure-aware chunking** for markdown, HTML, or policy documents.
- **Sliding window chunking** when context continuity matters.

Practical guidance:
- Keep chunks large enough to preserve meaning, but small enough to retrieve precisely.
- Use overlap to avoid splitting important context across boundaries.
- Preserve headings in the chunk text so the model can infer document structure.
- Store the original source reference for citations and auditability.

Embedding models convert text into vectors. Similar meanings should map to nearby vectors in the embedding space. The quality of embeddings strongly affects retrieval quality, so model choice matters.

**Vector database operations**

A vector database usually supports the following operations:

- **Upsert**
  - Insert or update vectors and metadata.
- **Query**
  - Search for nearest vectors to a query embedding.
- **Filter**
  - Restrict results by metadata.
- **Delete**
  - Remove vectors when documents expire or are revoked.
- **Reindex**
  - Rebuild indexes after schema or model changes.

Example using a Python client pattern:

```python
from sentence_transformers import SentenceTransformer
from pinecone import Pinecone

model = SentenceTransformer("all-MiniLM-L6-v2")
pc = Pinecone(api_key="YOUR_API_KEY")
index = pc.Index("rag-demo")

text = "Remote access requires MFA and approved devices."
vector = model.encode(text).tolist()

index.upsert([
    {
        "id": "hr-policy-014-chunk-3",
        "values": vector,
        "metadata": {
            "doc_id": "hr-policy-014",
            "source": "synthetic",
            "department": "HR",
            "classification": "internal"
        }
    }
])
```

Example query with metadata filtering:

```python
query = "What are the requirements for remote access?"
qvec = model.encode(query).tolist()

results = index.query(
    vector=qvec,
    top_k=3,
    include_metadata=True,
    filter={
        "department": {"$eq": "HR"},
        "classification": {"$eq": "internal"}
    }
)
```

**Prompt assembly**

After retrieval, the app builds a prompt that includes the user question and the retrieved context. The prompt should instruct the model to answer only from the provided sources when possible.

A simple prompt template:

```text
You are a helpful assistant.
Use only the provided context to answer the question.
If the context is insufficient, say you do not know.

Context:
[1] Remote access requires MFA and approved devices.
[2] Exceptions require security approval and a documented business need.

Question:
What is required for remote access?

Answer:
```

Good prompt assembly practices:
- Separate system instructions from retrieved content.
- Label sources clearly.
- Limit the number of chunks to avoid context overload.
- Include citation identifiers so the answer can reference sources.
- Reject or sanitize retrieved text that contains suspicious instructions.

**Security risks in RAG demos**

RAG systems introduce security issues beyond standard LLM risks. The retrieval layer can become a new attack surface.

Common risks include:
- **Prompt injection** inside retrieved documents.
- **Data poisoning** through malicious or low-quality content.
- **Sensitive data exposure** from over-broad retrieval.
- **Cross-tenant leakage** when metadata filters fail.
- **Model inversion or memorization concerns** if training data is reused improperly.
- **Supply chain risk** from embedding services, plugins, or document parsers.

Mitigations:
- Validate and sanitize ingested content.
- Separate trusted instructions from untrusted retrieved text.
- Enforce strict metadata-based access control.
- Log retrieval decisions for audit and incident response.
- Use synthetic data for demos and testing.
- Red-team the system with adversarial prompts and poisoned documents.

**Access control and tenant isolation**

In multi-user or multi-tenant environments, retrieval must respect authorization. A user should only retrieve documents they are allowed to see.

Controls to apply:
- Filter by `tenant_id`, `department`, or `classification`.
- Enforce authorization before retrieval, not only before display.
- Avoid sharing a single index without strong metadata isolation.
- Use separate namespaces or indexes for high-risk tenants when needed.
- Audit access to sensitive chunks and generated outputs.

Example policy logic:
- A user in `finance` can retrieve `finance` documents marked `internal`.
- A contractor can retrieve only `public` or explicitly shared content.
- A security analyst may access `restricted` incident data under approved conditions.

**Evaluation metrics**

You should measure both retrieval quality and answer quality.

Retrieval metrics:
- **Recall@k**: whether the correct chunk appears in the top `k` results.
- **Precision@k**: how many retrieved chunks are relevant.
- **MRR** (Mean Reciprocal Rank): how early the correct result appears.
- **NDCG** (Normalized Discounted Cumulative Gain): ranking quality with graded relevance.

Generation metrics:
- Faithfulness to retrieved context.
- Citation accuracy.
- Completeness of the answer.
- Refusal behavior when context is insufficient.

Operational metrics:
- Query latency.
- Index build time.
- Embedding throughput.
- Cost per query.
- Cache hit rate.

A demo app is useful when you want to compare retrieval settings, such as chunk size, overlap, top-k, and embedding model choice.

**Practical demo workflow**

A typical lab workflow looks like this:

1. Generate a synthetic corpus with policies, FAQs, and incident examples.
2. Chunk the documents and assign metadata.
3. Embed the chunks and load them into the vector database.
4. Build a simple chat or API interface.
5. Test normal questions and adversarial prompts.
6. Review retrieval results and adjust chunking or filters.
7. Add citations, logging, and access controls.
8. Re-run tests to confirm improvements.

**Example local stack**

A lightweight local demo can use:
- `FastAPI` for the API layer.
- `sentence-transformers` for embeddings.
- `FAISS` for local vector search.
- `Docker` for reproducible deployment.
- `PostgreSQL` or `SQLite` for metadata and audit logs.

**When to use synthetic versus real data**

Use synthetic data when:
- You are prototyping architecture.
- You are testing security controls.
- You need repeatable benchmark cases.
- You cannot expose real records.

Use real data when:
- You need production relevance and final accuracy testing.
- You have approval, governance, and masking controls in place.
- You are validating retrieval against actual document distributions.

In many secure AI programs, the best practice is to start with synthetic data, then move to sanitized or approved real data only after the pipeline is stable.

**Troubleshooting common issues**

If retrieval quality is poor, check:
- Chunk size and overlap.
- Embedding model suitability.
- Metadata filters that may be too strict.
- Index freshness after document updates.
- Query wording versus document language.

If answers are hallucinated, check:
- Whether the prompt tells the model to rely on context.
- Whether retrieved chunks are relevant enough.
- Whether too many irrelevant chunks are crowding the context window.
- Whether the model is allowed to answer without evidence.

If sensitive data appears in results, check:
- Ingestion filters and redaction rules.
- Access control enforcement.
- Namespace or tenant isolation.
- Logging and alerting for unusual retrieval patterns.

**Key takeaways**

- RAG improves grounding by combining retrieval with generation.
- Vector databases enable semantic search over embeddings.
- Synthetic datasets let you test securely without exposing real data.
- Chunking, metadata, and access control strongly affect retrieval quality and safety.
- Security testing should include prompt injection, poisoning, and leakage scenarios.
- Evaluation must cover both retrieval accuracy and answer faithfulness.
