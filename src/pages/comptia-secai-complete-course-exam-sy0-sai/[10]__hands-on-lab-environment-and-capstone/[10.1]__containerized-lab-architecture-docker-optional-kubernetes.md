## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Containerized lab architecture overview**

Containerized labs let you package tools, services, and test targets into repeatable environments that start quickly and reset cleanly. For security and AI-focused training, this matters because you often need isolated sandboxes for malware analysis, model testing, API experimentation, log generation, and adversarial simulations.

A good lab design separates concerns:

- **Host layer** runs the container engine and provides CPU, memory, storage, and networking.
- **Container layer** holds the application, tool, or service you want to test.
- **Network layer** controls how containers talk to each other and to the outside world.
- **Persistence layer** preserves data that must survive container recreation.
- **Orchestration layer** becomes useful when you need scaling, scheduling, or multi-node coordination.

Containers are not virtual machines. A container shares the host kernel and isolates processes, filesystems, and networking through Linux kernel features such as namespaces and control groups. That makes containers lighter and faster to deploy, but also means host security matters a lot.

**Why use Docker in a lab**

Docker is the most common container platform for single-host and small-team lab work. It gives you a consistent way to build images, run containers, attach networks, and manage volumes.

Use Docker when you need:

- Fast environment rebuilds after destructive testing
- Reproducible toolchains for analysts and developers
- Easy packaging of services like databases, message brokers, and web apps
- Simple local testing before moving to Kubernetes
- Isolation between competing experiments

Docker is especially useful for security labs because you can spin up vulnerable targets, logging stacks, and analysis tools without polluting the base operating system.

**Core Docker concepts**

Docker uses a few key building blocks:

- **Image**: a read-only template that contains the filesystem and metadata needed to start a container
- **Container**: a running instance of an image with a writable layer
- **Dockerfile**: a text file that defines how to build an image
- **Registry**: a repository for storing and distributing images
- **Volume**: persistent storage managed by Docker
- **Network**: a virtual network that connects containers

A practical mental model is:

- You build an image from a Dockerfile.
- You run a container from the image.
- You attach storage and networking as needed.
- You destroy and recreate containers without losing the image or volume data.

**Typical lab architecture patterns**

A simple lab often uses a single Docker host with multiple containers on a user-defined bridge network. For example, you might run:

- A web application container
- A database container
- A log collector container
- An attacker simulation container
- A monitoring container

This pattern works well for local development and exam practice because it is easy to understand and troubleshoot.

A more advanced pattern uses separate networks:

- **Frontend network** for user-facing services
- **Backend network** for databases and internal APIs
- **Management network** for admin access and observability
- **Egress-controlled network** for tools that should not reach the internet

Segmentation reduces accidental exposure and helps you model real-world security boundaries.

**Docker installation and verification**

On Linux, Docker is commonly installed through the package manager or Docker’s official repository. After installation, verify the daemon and client.

```bash
docker version
docker info
systemctl status docker
```

If the daemon is not running, start it:

```bash
sudo systemctl enable --now docker
```

To avoid using `sudo` for every command, you can add your user to the `docker` group:

```bash
sudo usermod -aG docker $USER
newgrp docker
```

> Membership in the `docker` group is effectively root-equivalent on many systems. Treat it as a privileged access path.

**Building images with Dockerfile**

A `Dockerfile` defines repeatable builds. Use it when you want to pin dependencies, automate setup, and avoid manual configuration drift.

Example:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8080
CMD ["python", "app.py"]
```

Key instructions:

- **FROM** sets the base image
- **WORKDIR** sets the working directory
- **COPY** moves files into the image
- **RUN** executes build-time commands
- **EXPOSE** documents the intended port
- **CMD** defines the default startup command

Build and tag the image:

```bash
docker build -t lab-webapp:1.0 .
```

Run it:

```bash
docker run -d --name webapp -p 8080:8080 lab-webapp:1.0
```

**Container lifecycle and common commands**

You should know the basic lifecycle operations because lab work often involves rapid iteration.

```bash
docker ps
docker ps -a
docker logs webapp
docker exec -it webapp /bin/sh
docker stop webapp
docker start webapp
docker rm webapp
docker rmi lab-webapp:1.0
```

Practical notes:

- Use `docker ps` to see running containers.
- Use `docker logs` to inspect application output.
- Use `docker exec` for interactive troubleshooting.
- Use `docker stop` before `docker rm` when you want a clean shutdown.
- Use `docker rm -f` only when you need to force removal.

**Networking in Docker labs**

Docker networking determines how containers communicate. The default bridge network is convenient, but user-defined bridge networks are usually better for labs because they provide automatic DNS resolution by container name and cleaner isolation.

Create a network:

```bash
docker network create lab-net
```

Run containers on it:

```bash
docker run -d --name db --network lab-net postgres:16
docker run -d --name api --network lab-net my-api:latest
```

The API container can reach the database using the hostname `db`.

Useful network commands:

```bash
docker network ls
docker network inspect lab-net
docker network connect lab-net webapp
docker network disconnect lab-net webapp
```

Common network modes:

- **bridge**: default for isolated container networking on a single host
- **host**: container shares the host network stack
- **none**: no network access
- **macvlan**: container appears as a device on the physical network

Use `host` mode sparingly because it reduces isolation. Use `macvlan` when you need containers to look like separate hosts on the LAN, but be aware that it can complicate routing and switch configuration.

**Ports, exposure, and access control**

Port mapping publishes container services to the host.

```bash
docker run -d --name nginx -p 8080:80 nginx:latest
```

This maps host port `8080` to container port `80`.

Important distinctions:

- **EXPOSE** in a Dockerfile documents intent
- `-p` publishes a port to the host
- `-P` publishes all exposed ports to random host ports

For security labs, prefer explicit mappings so you know exactly what is reachable.

**Persistent storage with volumes and bind mounts**

Containers are disposable, so you need persistence for databases, logs, and artifacts.

Use a **volume** when you want Docker-managed persistent storage:

```bash
docker volume create pgdata
docker run -d --name postgres \
  -e POSTGRES_PASSWORD=secret \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16
```

Use a **bind mount** when you want to map a host directory into a container:

```bash
docker run -d --name analyzer \
  -v /opt/lab/reports:/reports \
  my-analyzer:latest
```

Choose volumes when:

- You want portability and simpler lifecycle management
- The data belongs to the containerized service
- You want Docker to manage the storage path

Choose bind mounts when:

- You need direct access to files on the host
- You are developing and editing files locally
- You need to integrate with existing directory structures

**Environment variables and secrets**

Environment variables are common for non-sensitive configuration.

```bash
docker run -d --name app \
  -e APP_ENV=lab \
  -e LOG_LEVEL=debug \
  my-app:latest
```

For sensitive values, avoid hardcoding secrets in images or shell history. Prefer secret management mechanisms when available. In plain Docker labs, you may use mounted files with restricted permissions, but that is still weaker than a dedicated secrets system.

> Do not bake passwords, API keys, or tokens into images unless the lab is intentionally disposable and the risk is understood.

**Docker Compose for multi-service labs**

Docker Compose is useful when your lab has multiple services that should start together. It defines the stack in a YAML file and simplifies repeatable deployment.

Example `compose.yaml`:

```yaml
services:
  web:
    image: nginx:latest
    ports:
      - "8080:80"
    networks:
      - lab-net

  api:
    build: ./api
    environment:
      APP_ENV: lab
    networks:
      - lab-net

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data
    networks:
      - lab-net

networks:
  lab-net:

volumes:
  pgdata:
```

Common commands:

```bash
docker compose up -d
docker compose ps
docker compose logs -f
docker compose down
docker compose down -v
```

Use Compose when you want:

- A reproducible multi-container stack
- Easier onboarding for teammates
- A single file to version control
- Fast teardown and rebuild during testing

**Security considerations for Docker labs**

Containers improve isolation, but they are not a complete security boundary. You still need to harden the host and the container runtime.

Key practices:

- Run containers as non-root when possible
- Drop unnecessary Linux capabilities
- Use read-only filesystems for immutable services
- Limit CPU and memory to reduce blast radius
- Keep images updated and rebuild regularly
- Scan images for known vulnerabilities
- Restrict access to the Docker socket
- Avoid `--privileged` unless you truly need it

Example of a more constrained run:

```bash
docker run -d --name app \
  --read-only \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  --memory 512m \
  --cpus 1.0 \
  my-app:latest
```

The Docker socket at `/var/run/docker.sock` is highly sensitive. If a container can access it, that container can often control the host’s Docker daemon.

**Image hygiene and supply chain awareness**

Lab images should be small, reproducible, and traceable. Use minimal base images when possible, pin versions, and rebuild on a schedule.

Good practices:

- Pin image tags instead of relying on `latest`
- Prefer slim or distroless bases when compatible
- Remove build tools from runtime images
- Use multi-stage builds to keep runtime images small
- Track image provenance and maintain a change log

Example multi-stage build:

```dockerfile
FROM golang:1.23 AS build
WORKDIR /src
COPY . .
RUN go build -o /out/labtool .

FROM gcr.io/distroless/base-debian12
COPY --from=build /out/labtool /usr/local/bin/labtool
ENTRYPOINT ["/usr/local/bin/labtool"]
```

This pattern reduces attack surface by excluding compilers and package managers from the final image.

**Logging and observability**

In containerized labs, logs are often the fastest way to understand failures. Applications should write to standard output and standard error so Docker can capture them.

Useful commands:

```bash
docker logs -f webapp
docker inspect webapp
docker stats
```

For larger labs, add:

- A centralized log collector
- Metrics collection
- Health checks
- Alerting for failed services

Health checks help you detect when a service is running but not ready.

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1
```

**When Kubernetes becomes useful**

Kubernetes is optional for many labs, but it becomes valuable when you need scheduling, scaling, self-healing, or multi-node orchestration. It is more complex than Docker Compose, so use it when the added control is worth the overhead.

Use Kubernetes when you need:

- Multiple nodes
- Automated rescheduling after failure
- Horizontal scaling
- Declarative deployment management
- Service discovery and load balancing at cluster scale

For small local labs, Docker Compose is often simpler. For production-like simulations, Kubernetes gives you a closer model of real-world platform operations.

**Kubernetes architecture basics**

Kubernetes separates control and workload responsibilities.

- **Control plane** manages the cluster state
- **Worker nodes** run application pods
- **Pod** is the smallest deployable unit and usually contains one main container
- **Deployment** manages replica sets and rollout behavior
- **Service** provides stable networking to pods
- **ConfigMap** stores non-sensitive configuration
- **Secret** stores sensitive data in encoded form, though not encrypted by default unless the cluster is configured for it

A simple mental model:

- You define desired state in YAML.
- The control plane reconciles actual state to match desired state.
- Pods are created, replaced, or scaled automatically.

**Docker and Kubernetes relationship**

Docker is often used to build images even when Kubernetes runs the workload. Kubernetes does not require Docker as its runtime anymore, but it still consumes OCI-compatible images from registries.

Practical workflow:

1. Build an image locally with Docker.
2. Push it to a registry.
3. Reference the image in a Kubernetes manifest.
4. Let the cluster pull and run it.

Example:

```bash
docker build -t registry.example.com/lab/webapp:1.0 .
docker push registry.example.com/lab/webapp:1.0
```

**Basic Kubernetes manifest example**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
        - name: webapp
          image: registry.example.com/lab/webapp:1.0
          ports:
            - containerPort: 8080
```

Expose it with a Service:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-svc
spec:
  selector:
    app: webapp
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP
```

Useful commands:

```bash
kubectl apply -f deployment.yaml
kubectl get pods
kubectl get svc
kubectl describe pod webapp-xxxx
kubectl logs deploy/webapp
kubectl exec -it pod/webapp-xxxx -- sh
kubectl delete -f deployment.yaml
```

**Kubernetes networking and service exposure**

Kubernetes networking is more abstract than Docker networking. Pods get cluster IPs, and Services provide stable access points.

Common Service types:

- **ClusterIP**: internal-only access inside the cluster
- **NodePort**: exposes a service on each node at a static port
- **LoadBalancer**: integrates with cloud or external load balancers
- **Ingress**: HTTP/HTTPS routing through an ingress controller

For lab work, `ClusterIP` is often enough unless you need external access.

**Storage in Kubernetes**

Persistent storage in Kubernetes uses PersistentVolumes and PersistentVolumeClaims. This is the cluster-native equivalent of Docker volumes, but with more abstraction.

Use persistent storage when:

- A database must survive pod replacement
- You need shared storage across restarts
- You want storage provisioning to be managed by the cluster

Example claim:

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pgdata
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
```

**Choosing Docker, Compose, or Kubernetes**

| Tool | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| Docker | Single containers and simple labs | Fast, simple, low overhead | Limited orchestration |
| Docker Compose | Multi-service local stacks | Easy startup and teardown | Single-host focus |
| Kubernetes | Scaled or production-like labs | Scheduling, resilience, declarative control | Higher complexity |

A practical rule:

- Use Docker for one-off services and quick tests.
- Use Compose for repeatable multi-service labs.
- Use Kubernetes when you need cluster behavior or want to practice platform operations.

**Troubleshooting workflow**

When a containerized lab fails, check the problem in layers:

1. Confirm the container is running.
2. Inspect logs for application errors.
3. Verify port mappings and network reachability.
4. Check volume mounts and file permissions.
5. Validate environment variables and secrets.
6. Review resource limits and health checks.

Useful commands:

```bash
docker ps -a
docker logs <container>
docker inspect <container>
docker exec -it <container> sh
docker network inspect <network>
docker volume inspect <volume>
```

For Kubernetes, use:

```bash
kubectl get all
kubectl describe pod <pod>
kubectl logs <pod>
kubectl exec -it <pod> -- sh
kubectl get events --sort-by=.metadata.creationTimestamp
```

**Practical lab example: isolated web app stack**

A common security lab stack includes:

- `web` container for the application
- `db` container for persistence
- `proxy` container for reverse proxying
- `logger` container for centralized logs

Example Compose file:

```yaml
services:
  proxy:
    image: nginx:latest
    ports:
      - "8443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    networks:
      - front

  web:
    image: my-webapp:1.0
    networks:
      - front
      - back

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data
    networks:
      - back

networks:
  front:
  back:

volumes:
  pgdata:
```

This design keeps the database off the public-facing network while still allowing the web app to reach it.

**Exam-focused takeaways**

- **Containers** package applications with dependencies for repeatable execution.
- **Docker** is the primary tool for building and running containers in local labs.
- **Volumes** preserve data across container recreation.
- **Bridge networks** are the default choice for isolated single-host labs.
- **Compose** simplifies multi-service deployments.
- **Kubernetes** adds orchestration, scaling, and self-healing when Docker alone is not enough.
- **Security** depends on host hardening, least privilege, image hygiene, and careful network exposure.
