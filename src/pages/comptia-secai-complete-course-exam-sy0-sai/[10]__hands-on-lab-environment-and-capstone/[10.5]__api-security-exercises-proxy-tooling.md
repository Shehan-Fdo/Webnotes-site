## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**API security testing workflow**

API testing is most effective when you treat the API as a set of observable requests, schemas, and trust boundaries rather than as a black box. In practice, you use a proxy to intercept traffic, identify endpoints, replay requests, and compare expected authorization behavior against actual behavior.

A good workflow starts with discovery, then moves to authentication, authorization, input handling, and abuse cases. You should also verify how the API behaves under replay, throttling, malformed input, and protocol variations such as WebSockets or gRPC.

- **Discovery** begins with documentation, app traffic, and endpoint enumeration.
- **Replay** helps you confirm whether requests are idempotent, destructive, or rate-limited.
- **Validation** checks whether the server enforces auth, schema, and content rules consistently.

**Common API discovery sources**

You often find the first useful request in browser traffic, mobile app traffic, or documentation. API exercises usually expect you to combine passive discovery with active inspection.

- **OpenAPI** and Swagger docs often expose paths, methods, parameters, and response models.
- **Postman collections** can reveal hidden test endpoints, headers, and environment variables.
- **Developer docs** may list versioned routes, sample payloads, and auth requirements.
- **Traffic capture** from a browser, mobile app, or desktop client often reveals undocumented endpoints.
- **Endpoint enumeration** can uncover alternate verbs, version prefixes, and admin-only routes.

A practical approach is to compare documentation with live traffic. If docs show `/v1/users/{id}` but the app calls `/api/users/123`, you should test both for access control and schema differences.

```bash
# Example discovery commands
curl -s https://api.example.com/openapi.json | jq '.paths'
curl -s https://api.example.com/swagger.json | jq '.components.schemas'
```

**Proxy tooling and where each module fits**

Proxy tools let you inspect, modify, and replay requests. Burp Suite is the most common exam-relevant workflow because it combines interception, history, repeater, intruder, decoder, and protocol-aware tooling in one place. Other proxies can work similarly, but you should know where each module fits in a lab.

| Tool / module | Best use | Notes |
|---|---|---|
| **Proxy** | Intercept live traffic | Use to capture requests from browsers, mobile apps, and API clients. |
| **HTTP history** | Review observed requests | Useful for finding hidden endpoints, headers, and tokens. |
| **Repeater** | Manual replay and mutation | Best for authorization checks, schema testing, and one-off payload changes. |
| **Intruder** | Controlled fuzzing and parameter testing | Useful for rate-limit checks, ID enumeration, and mass assignment probes. |
| **Decoder** | Inspect encodings | Use for Base64, URL encoding, hex, and JWT segment inspection. |
| **Comparer** | Diff responses | Helpful when comparing allowed vs denied requests. |
| **Logger / project history** | Track evidence | Useful in labs for documenting request/response changes. |

Burp Suite workflows usually follow this pattern:

1. Configure the browser or app to use the proxy.
2. Install the proxy CA certificate if you need to inspect HTTPS.
3. Capture traffic and identify the API host, paths, and auth headers.
4. Send interesting requests to Repeater for controlled edits.
5. Use Intruder only when you need systematic variation or fuzzing.

**Proxy configuration basics**

For browser-based testing, you typically point the browser at `127.0.0.1:8080` or another local listener. For command-line clients, you can set proxy environment variables or client-specific proxy options.

```bash
# curl through a local proxy
curl -x http://127.0.0.1:8080 https://api.example.com/v1/me

# httpie through a local proxy
http --proxy=http://127.0.0.1:8080 GET https://api.example.com/v1/me

# environment variables for many CLI tools
export HTTP_PROXY=http://127.0.0.1:8080
export HTTPS_PROXY=http://127.0.0.1:8080
```

For browsers, you usually configure the proxy in the browser settings or OS network settings. For Firefox, you can set the proxy in the network preferences or use a profile configured for testing. Chromium-based browsers often work well with a local intercepting proxy because they follow system proxy settings more predictably in lab environments.

> Do not assume HTTPS interception works until the proxy CA certificate is trusted by the client. Without trust, you may only see CONNECT tunnels or certificate errors.

**Mobile interception and certificate handling**

Mobile API testing often requires extra setup because apps may use certificate pinning, custom trust stores, or proxy bypass logic. You should know how to intercept traffic from both emulators and physical devices.

- **Android emulator** testing is usually easier because you can install the proxy CA certificate and route traffic through the host proxy.
- **Physical Android devices** often require manual Wi-Fi proxy settings and certificate installation.
- **iOS devices** require installing and trusting the proxy certificate in the device trust settings.
- **Certificate pinning** can block interception even when the CA is installed.

Typical mobile workflow:

1. Set the device Wi-Fi proxy to the proxy host and port.
2. Install the proxy CA certificate on the device.
3. Trust the certificate in the OS trust settings if required.
4. Launch the app and confirm traffic appears in the proxy.
5. If traffic is missing, check for pinning, hardcoded proxy bypass, or alternate transports.

For Android, you may need to install the CA certificate as a user certificate and, in some cases, use a debug build or emulator image that trusts user CAs. For iOS, you usually install the certificate profile and then enable full trust for the root certificate in settings.

**Authentication testing and token edge cases**

API exercises commonly include bearer tokens, session cookies, API keys, or OAuth 2.0 access tokens. You should test how the API handles token lifecycle events rather than only whether a token is accepted.

- **Expiration**: confirm expired tokens are rejected consistently.
- **Refresh**: verify refresh tokens cannot be reused indefinitely or replayed from another client.
- **Logout / revocation**: check whether invalidated tokens still work after logout.
- **Claim tampering**: modify JWT claims and confirm the server validates signatures and claims.
- **Audience and issuer checks**: verify the API rejects tokens issued for a different audience or issuer.

A JWT (JSON Web Token) is not just an encoded string; it is a signed token with header, payload, and signature. You can decode the segments to inspect claims, but decoding does not validate trust. The server must verify the signature and enforce claim checks.

```bash
# Inspect a JWT payload locally without trusting it
python3 - <<'PY'
import base64, json
token = "eyJhbGciOi..."
payload = token.split('.')[1] + '=='
print(json.loads(base64.urlsafe_b64decode(payload)))
PY
```

Common API key placement includes custom headers such as `X-API-Key` or an `Authorization: Bearer <token>` scheme for bearer tokens. Do not assume a custom scheme like `Authorization: ApiKey <value>` is standard.

**Authorization and access control testing**

Authorization flaws are among the most important API findings. You should compare what a low-privilege user can do versus what a higher-privilege user can do, then test whether the server enforces object-level and function-level controls.

- **Broken object-level authorization** occurs when you can access another user’s object by changing an ID.
- **Broken function-level authorization** occurs when a normal user can call admin-only endpoints.
- **Horizontal privilege escalation** occurs when you access peer data.
- **Vertical privilege escalation** occurs when you access higher-privilege functions.

Practical tests include changing numeric IDs, UUIDs, usernames, tenant IDs, and nested object references. You should also test whether the API relies on client-side hiding rather than server-side enforcement.

```http
GET /api/v1/orders/1001 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJ...

GET /api/v1/orders/1002 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJ...
```

If the second request returns another user’s order, you have an object-level access control issue. If the server returns `403` for one endpoint but not another similar endpoint, compare route patterns, verbs, and version prefixes.

**Mass assignment and parameter pollution**

Mass assignment happens when the server binds client-supplied fields directly to internal objects without filtering sensitive properties. This is common in JSON APIs and can expose fields such as `role`, `isAdmin`, `status`, or `accountBalance`.

- Add unexpected fields to create or update requests.
- Compare server behavior when fields are omitted, duplicated, or reordered.
- Test nested objects for hidden writable properties.
- Check whether read-only fields can be set by the client.

```json
{
  "name": "Alice",
  "email": "alice@example.com",
  "role": "admin",
  "isAdmin": true
}
```

Parameter pollution can change how the server parses repeated keys or conflicting values. Different frameworks may use the first value, last value, or merge values into arrays. You should test both query strings and form bodies.

```http
POST /api/v1/profile HTTP/1.1
Content-Type: application/x-www-form-urlencoded

role=user&role=admin&isAdmin=true
```

**Rate limiting, throttling, and abuse detection**

APIs often rely on rate limiting to prevent brute force, scraping, and resource exhaustion. In labs, you should verify whether limits apply per IP, per account, per token, or per endpoint.

- **Login endpoints** may need lockout or progressive delays.
- **OTP and password reset flows** should resist brute force and replay.
- **Search and export endpoints** may need stricter throttling.
- **Destructive endpoints** should be protected against repeated replay.

When testing rate limits, avoid uncontrolled flooding. Use small, deliberate bursts and observe response codes, headers, and timing differences. Look for `429 Too Many Requests`, `Retry-After`, or custom quota headers.

```bash
# Small burst to observe throttling behavior
for i in $(seq 1 8); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Authorization: Bearer $TOKEN" \
    https://api.example.com/v1/search?q=test
done
```

Rate-limit bypass attempts may include rotating tokens, changing headers, using alternate endpoints, or exploiting inconsistent limits across methods such as `GET` versus `POST`.

**Replay implications and destructive requests**

Replay testing is useful, but destructive requests can create real side effects. You should understand whether a request is idempotent before repeating it.

- **Safe requests** like `GET` usually do not change state, but they can still trigger logging or analytics.
- **Idempotent requests** like `PUT` may be safe to repeat if implemented correctly.
- **Non-idempotent requests** like `POST` can create duplicate records or duplicate charges.
- **State-changing requests** should be tested in a controlled lab or with test accounts.

If you replay a payment, password reset, or delete request, you may trigger irreversible actions. Use test data, confirm rollback options, and document the exact request before sending it again.

**CORS testing**

Cross-Origin Resource Sharing (CORS) controls whether a browser allows JavaScript from one origin to read responses from another origin. CORS is a browser enforcement mechanism, not a server-side access control replacement.

- `Access-Control-Allow-Origin` should match trusted origins explicitly when credentials are involved.
- `Access-Control-Allow-Credentials: true` cannot be combined with `Access-Control-Allow-Origin: *` in a way browsers will accept for credentialed reads.
- A reflected origin policy can be risky if the server blindly echoes attacker-controlled origins.
- Preflight requests use `OPTIONS` and may reveal allowed methods and headers.

```http
OPTIONS /api/v1/account HTTP/1.1
Origin: https://evil.example
Access-Control-Request-Method: GET
Access-Control-Request-Headers: Authorization
```

If the response reflects the attacker origin and allows credentials, the browser may permit cross-origin reads from a malicious site. The real risk is not the wildcard alone; it is the combination of credentialed access and unsafe origin reflection or overly broad trust.

**Injection in JSON, XML, and headers**

APIs often accept structured input, which means injection can appear in JSON fields, XML elements, query parameters, or headers. You should test whether the backend passes input into SQL, command execution, template rendering, or downstream services.

- **SQL injection** can occur in JSON values if the server concatenates input into queries.
- **Command injection** can occur in fields used by shell commands or system utilities.
- **XML injection** and XXE (XML External Entity) issues can appear in XML-based APIs.
- **Header injection** can occur if user input is copied into downstream headers.

Example JSON payload for testing server-side validation:

```json
{
  "search": "' OR '1'='1",
  "filter": "${7*7}",
  "callback": "test"
}
```

You should also test content-type handling. Some APIs parse JSON only when `Content-Type: application/json` is present, while others may accept multiple formats or auto-detect content. Mismatched parsing can create bypasses.

**File upload testing**

API file upload endpoints deserve the same scrutiny as web upload forms, plus API-specific checks for metadata and object access. You should test file type validation, storage behavior, and whether uploaded files are directly accessible.

- **Content validation** checks MIME type, extension, magic bytes, and server-side scanning.
- **Path traversal** in filenames can expose unsafe path handling.
- **Direct object access** can expose uploaded files if predictable URLs are used.
- **Metadata handling** may leak internal paths, hashes, or storage keys.

```http
POST /api/v1/upload HTTP/1.1
Content-Type: multipart/form-data; boundary=----demo

------demo
Content-Disposition: form-data; name="file"; filename="../../etc/passwd"
Content-Type: image/png

PNGDATA
------demo--
```

You should verify whether the server renames files, strips path separators, scans content, and stores uploads outside the web root. If the API returns a file URL, test whether that URL is guessable or accessible without authorization.

**WebSockets and gRPC in proxy workflows**

Some APIs use WebSockets for real-time updates or gRPC for high-performance service calls. These protocols require protocol-aware tooling or special handling in the proxy.

- **WebSockets** can often be inspected after the initial HTTP upgrade request.
- **gRPC** uses HTTP/2 and Protobuf, so you may need a proxy that understands HTTP/2 framing.
- **Replay** is possible, but you often need to preserve message structure and metadata.

For WebSockets, you can inspect the handshake and then observe frames in a proxy that supports WebSocket message views. For gRPC, you may need to use Burp’s HTTP/2 support, a gRPC-aware client, or a companion tool such as `grpcurl` for direct calls.

```bash
# Example gRPC inspection with grpcurl
grpcurl -plaintext api.example.com:50051 list
grpcurl -plaintext -d '{"user_id":"123"}' api.example.com:50051 com.example.UserService/GetUser
```

If the proxy cannot decode the body, you can still use it to observe the endpoint, headers, and transport behavior, then use a protocol-specific client to replay requests.

**Schema validation and contract testing**

Schema validation helps you determine whether the API enforces the expected request and response structure. OpenAPI definitions, JSON Schema, and documented examples are useful references, but the live API is the source of truth.

- Compare required fields against optional fields.
- Test type mismatches such as strings where integers are expected.
- Check whether extra fields are ignored, rejected, or stored.
- Verify whether response fields leak internal identifiers or debug data.

A mismatch between schema and implementation can reveal hidden functionality or weak validation. For example, if docs say `status` is server-controlled but the API accepts it from the client, that is a likely business logic flaw.

**Practical lab checklist**

Use a repeatable checklist so you do not miss common exam-relevant issues.

1. Capture a baseline request in the proxy.
2. Identify auth method, endpoint path, and content type.
3. Replay the request in Repeater and confirm normal behavior.
4. Change object IDs, roles, and tenant identifiers.
5. Add unexpected fields to test mass assignment.
6. Test expired, altered, and revoked tokens.
7. Probe rate limits with small controlled bursts.
8. Check CORS headers and preflight behavior.
9. Test file uploads for validation, scanning, and path handling.
10. Inspect alternate transports such as WebSockets or gRPC when present.

**Useful CLI examples for API labs**

Command-line tools are useful when you want reproducible requests outside the browser or proxy UI. They also help you compare proxy behavior with direct client behavior.

```bash
# Send a JSON request with curl
curl -i https://api.example.com/v1/profile \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"Alice","isAdmin":true}'

# Inspect response headers only
curl -I https://api.example.com/v1/account

# Follow redirects and keep verbose output
curl -vk https://api.example.com/v1/me
```

If a request behaves differently through the proxy than directly, check for proxy-added headers, TLS interception issues, or app-specific bypass logic.

**Key exam takeaways**

API security exercises are about methodical verification, not random fuzzing. You should know how to discover endpoints, intercept traffic, replay requests, and test the server’s actual enforcement of authentication, authorization, schema, and rate limits.

- **Burp Suite** is the most practical all-in-one proxy workflow to know.
- **OpenAPI** and live traffic are both valuable discovery sources.
- **JWTs** should be decoded for inspection, but trust depends on signature and claim validation.
- **Mass assignment**, **parameter pollution**, and **rate-limit abuse** are common API-specific weaknesses.
- **Mobile interception** often fails because of certificate trust or pinning, not because the API is unreachable.
- **WebSockets** and **gRPC** require protocol-aware inspection and replay strategies.
