**LLM architecture overview**

A <abbr title="large language model">LLM</abbr> is a neural network trained to predict or generate text by learning statistical patterns from large corpora. In practice, the architecture determines how the model represents tokens, processes context, and produces outputs. For exam purposes, focus on how the major design choices affect latency, memory, quality, and deployment fit.

Core architectural dimensions include:

- **Tokenization**: how text becomes model inputs.
- **Hidden size**: the width of internal representations.
- **Layer count**: how many transformation blocks the model uses.
- **Attention heads**: how the model splits representation space for parallel attention.
- **Context window**: how much prior text the model can condition on.
- **Parameter count**: the total learned weights, which strongly influences memory and compute needs.

These dimensions interact. For example, a larger context window increases memory pressure, while more layers and wider hidden size usually increase compute cost. A model can be small in parameter count but still expensive to serve if it uses long context or high-throughput decoding.

**Model families and when to use them**

LLMs are commonly grouped into three architecture families. Each family is optimized for a different task pattern.

| Architecture | Primary input/output flow | Typical training objective | Common use cases |
|---|---|---|---|
| **Encoder-only** | Text in, contextual embeddings out | Masked language modeling or similar denoising objectives | Classification, retrieval, embedding generation, semantic search |
| **Decoder-only** | Prompt in, next token out | Causal language modeling | Chatbots, code generation, completion, instruction following |
| **Encoder-decoder** | Input sequence encoded, output sequence decoded | Sequence-to-sequence training | Translation, summarization, transformation tasks |

Encoder-only models read the full input bidirectionally, so they are strong at understanding tasks. Decoder-only models generate autoregressively, meaning each new token depends on previous tokens only. Encoder-decoder models separate understanding and generation, which is useful when the output should be a transformed version of the input rather than a continuation of it.

**Training objectives and why they matter**

The training objective shapes the model’s behavior at inference time.

- **Masked language modeling** trains the model to predict missing tokens from surrounding context.
  - This supports strong bidirectional understanding.
  - It is common in encoder-only models.
- **Causal language modeling** trains the model to predict the next token from previous tokens only.
  - This matches text generation behavior.
  - It is the standard objective for decoder-only models.
- **Sequence-to-sequence objectives** train the model to map an input sequence to an output sequence.
  - This is common in encoder-decoder systems.
  - It supports tasks where the output is not just a continuation of the input.

A useful exam distinction is that the training objective usually matches the inference pattern. If the model must generate text token by token, causal language modeling is the natural fit. If the task is understanding or classification, bidirectional encoding is often more efficient.

**Tokenization fundamentals**

Tokenization converts raw text into discrete units called tokens. Most modern LLMs use subword tokenization methods such as Byte Pair Encoding (BPE), WordPiece, or Unigram models. Subword tokenization helps the model handle rare words, misspellings, and multilingual text better than word-level tokenization.

Important tokenization concepts:

- **Special tokens** mark boundaries or control behavior.
  - Examples include beginning-of-sequence, end-of-sequence, padding, and instruction markers.
- **Byte fallback** allows the tokenizer to represent any text, even unusual Unicode or unseen characters, by falling back to byte-level pieces.
- **Token count** affects cost, latency, and context usage.
  - More tokens mean more compute and memory.
  - Long prompts can crowd out useful conversation history.
- **Token boundaries are not word boundaries**.
  - A single word may split into multiple tokens.
  - Short common words may be one token, while rare or technical terms may split into several.

Practical example with a Hugging Face tokenizer:

```python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("gpt2")
text = "Zero-trust architecture reduces implicit trust."
tokens = tokenizer.tokenize(text)
ids = tokenizer.encode(text)

print(tokens)
print(ids)
print("Token count:", len(ids))
```

A practical deployment concern is that tokenization directly affects billing and throughput. If a prompt expands into many tokens, you pay more and may hit the model’s context limit sooner than expected.

> Token limits are not just a model feature; they are an operational constraint that affects prompt design, retrieval strategy, and cost control.

**Core model parameters**

Several architectural parameters define the capacity and behavior of an LLM.

- **Hidden size** is the dimensionality of the internal representation.
  - Larger hidden sizes increase expressiveness and compute cost.
  - They also increase memory use for activations and weights.
- **Layer count** is the number of stacked transformer blocks.
  - More layers can increase abstraction and capacity.
  - More layers also increase latency and training cost.
- **Attention head count** determines how many parallel attention subspaces the model uses.
  - More heads can improve representational flexibility, but only within the constraints of hidden size.
  - Increasing heads does not guarantee better quality.
- **Context window** is the maximum number of tokens the model can process at once.
  - Larger windows help with long documents and multi-turn conversations.
  - Larger windows increase memory and inference cost.
- **Parameter count** is the total number of learned weights.
  - More parameters often improve capability, but only if training data, optimization, and architecture are appropriate.

A common relationship is that hidden size must be divisible by the number of attention heads. This is because each head typically receives a slice of the hidden representation. If you increase head count without changing hidden size, each head becomes narrower, which can reduce per-head capacity.

**Transformer attention basics**

The transformer architecture uses attention to let tokens influence one another. Attention computes relationships between tokens so the model can focus on relevant context when producing an output.

The main components are:

- **Query (Q)**: what the current token is looking for.
- **Key (K)**: what each token offers as a match.
- **Value (V)**: the information carried forward if the token is attended to.

A simplified attention flow looks like this:

```text
input embeddings -> linear projections -> Q, K, V
Q compares with K -> attention weights
attention weights combine V -> contextual output
```

Self-attention means the model attends within the same sequence. This is the core mechanism in transformer blocks. It allows the model to capture long-range dependencies better than many older recurrent approaches.

**Causal masking and autoregressive generation**

Decoder-only models use **causal masking** so each token can only attend to earlier tokens, not future ones. This is what makes next-token prediction possible.

Why causal masking matters:

- It prevents information leakage during training.
- It matches the way the model is used at inference time.
- It enables autoregressive generation, where the model emits one token at a time.

A simplified generation flow is:

1. You send a prompt to the model.
2. The model processes the prompt in a prefilling pass.
3. The model predicts the next token.
4. The new token is appended to the sequence.
5. The model repeats the decode step until it stops.

This is different from processing the entire prompt one token at a time in a loop. In practice, the prompt is usually encoded first, then incremental decoding begins.

**KV cache and inference efficiency**

The **KV cache** stores past key and value tensors during generation so the model does not recompute attention over the full history at every step. This is one of the most important inference optimizations for decoder-only LLMs.

Benefits of KV caching:

- Reduces repeated computation during token-by-token decoding.
- Lowers latency for long generations.
- Improves throughput in interactive chat systems.

Tradeoffs:

- Increases memory usage, especially for long context windows.
- Can become a bottleneck when serving many concurrent sessions.
- Requires careful batching and memory management.

A practical inference example with Hugging Face Transformers:

```python
from transformers import AutoTokenizer, AutoModelForCausalLM

model_name = "gpt2"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)

prompt = "Explain zero trust in one paragraph."
inputs = tokenizer(prompt, return_tensors="pt")

output_ids = model.generate(
    **inputs,
    max_new_tokens=60,
    do_sample=False
)

print(tokenizer.decode(output_ids[0], skip_special_tokens=True))
```

In production, frameworks such as vLLM, TensorRT-LLM, and TGI often manage KV cache more efficiently than a basic Python loop.

**Attention variants and architectural extensions**

Standard multi-head attention is common, but several variants matter for deployment and exam comparisons.

- **Multi-head attention**
  - Each head attends differently.
  - Useful for capturing diverse relationships.
- **Multi-query attention (MQA)**
  - Uses one shared key/value set across heads.
  - Reduces memory and speeds up decoding.
- **Grouped-query attention (GQA)**
  - Shares key/value sets across groups of heads.
  - Balances quality and efficiency better than full MQA in many cases.
- **Cross-attention**
  - Used in encoder-decoder models.
  - Lets the decoder attend to encoder outputs.
  - Essential when generating output conditioned on an input sequence.
- **Sparse or local attention**
  - Limits attention to selected tokens or windows.
  - Helps with long-context scaling and efficiency.
- **Rotary positional embeddings (RoPE)**
  - Encode token position by rotating query and key vectors.
  - Widely used in modern decoder-only models.
  - Long-context performance often depends on scaling or extension methods, not RoPE alone.

Cross-attention is a major architectural difference between encoder-decoder systems and decoder-only systems. It is one reason encoder-decoder models are strong at transformation tasks such as translation and summarization.

**Positional information**

Attention alone does not know token order, so models need positional information. Positional encoding methods tell the model where a token appears in the sequence.

Common approaches include:

- **Absolute positional embeddings**
  - Add a learned or fixed position vector to each token.
  - Simple, but can be less flexible for long-context extrapolation.
- **Relative positional methods**
  - Encode distance between tokens.
  - Often improve generalization to varying sequence lengths.
- **RoPE**
  - Integrates position into attention calculations.
  - Common in many modern LLMs.

When studying long-context models, remember that extending context length often requires more than just changing the positional encoding. Training data, fine-tuning, and scaling methods also matter.

**Context window and practical limits**

The context window is the maximum number of tokens the model can consider at once. It includes the prompt, system instructions, retrieved documents, conversation history, and generated output in progress.

Operational implications:

- Long prompts reduce room for output.
- Large context windows increase memory use.
- Retrieval-augmented generation can reduce the need to stuff everything into the prompt.
- Token budgeting is a real design task in production systems.

Example of a simple token budget mindset:

- Reserve tokens for system instructions.
- Reserve tokens for user input.
- Reserve tokens for retrieved context.
- Reserve tokens for expected output.

If the model has an 8k token context window, a 6k-token document plus a 1.5k-token conversation history leaves little room for the answer. This is why chunking and retrieval are often necessary.

**Encoder-only, decoder-only, and encoder-decoder in practice**

Choose the architecture based on the task.

- **Encoder-only**
  - Best when you need embeddings, classification, or semantic matching.
  - Often cheaper for understanding tasks because it does not generate tokens autoregressively.
- **Decoder-only**
  - Best when you need open-ended generation, chat, code completion, or instruction following.
  - Most common for general-purpose assistant models.
- **Encoder-decoder**
  - Best when the output should be a transformed version of the input.
  - Strong for translation, summarization, and structured rewriting.

A practical comparison:

| Task | Best fit | Why |
|---|---|---|
| Sentiment classification | Encoder-only | Efficient bidirectional understanding |
| Chat assistant | Decoder-only | Natural autoregressive generation |
| Translation | Encoder-decoder | Input-output separation with cross-attention |
| Embedding search | Encoder-only | Produces compact semantic vectors |
| Summarization | Encoder-decoder or decoder-only | Depends on latency, quality, and deployment constraints |

**Deployment tradeoffs: quantization, distillation, and pruning**

Model architecture is only part of deployment. You often need compression or optimization to fit latency, memory, or cost targets.

- **Quantization**
  - Reduces numeric precision, such as from FP16 to INT8 or INT4.
  - Best when you need lower memory use and faster inference.
  - Can slightly reduce quality if pushed too far.
- **Distillation**
  - Trains a smaller student model to mimic a larger teacher model.
  - Best when you need a smaller model with better quality retention than naive compression.
  - Often useful for edge or cost-sensitive deployment.
- **Pruning**
  - Removes weights, heads, or neurons that contribute less.
  - Can reduce model size and sometimes speed.
  - Quality impact can be unpredictable without careful retraining.

Rule of thumb:

- Use **quantization** when deployment memory or cost is the main issue.
- Use **distillation** when you need a smaller model with better task fidelity.
- Use **pruning** when you can tolerate more experimentation and want structural reduction.

> Compression methods are not interchangeable. A model that is too large for memory may need quantization first, while a model that is too slow for quality-sensitive use may benefit more from distillation.

**Practical configuration example**

A simple local inference setup often combines a model, tokenizer, and generation settings.

```python
from transformers import AutoTokenizer, AutoModelForCausalLM

model_name = "meta-llama/Llama-3.1-8B-Instruct"  # example model name
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    device_map="auto",
    torch_dtype="auto"
)

messages = [
    {"role": "system", "content": "You are a concise security assistant."},
    {"role": "user", "content": "Summarize the risks of prompt injection."}
]

prompt = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=True
)

inputs = tokenizer(prompt, return_tensors="pt").to(model.device)

outputs = model.generate(
    **inputs,
    max_new_tokens=120,
    temperature=0.2,
    top_p=0.9
)

print(tokenizer.decode(outputs[0], skip_special_tokens=True))
```

This example shows several real deployment concerns:

- Chat templates standardize role formatting.
- `device_map="auto"` helps place weights across available hardware.
- `max_new_tokens` controls output length and cost.
- Sampling settings affect determinism and creativity.

**Common exam pitfalls**

- Confusing **token count** with word count.
- Assuming more heads always means better performance.
- Treating RoPE as a complete long-context solution by itself.
- Mixing up encoder-only understanding models with decoder-only generation models.
- Forgetting that encoder-decoder models use **cross-attention**.
- Assuming KV cache reduces memory overall; it usually reduces compute per step but increases memory use.
- Overlooking that context window includes both input and output budget.

**Quick comparison checklist**

- If the task is classification or embeddings, think **encoder-only**.
- If the task is generation or chat, think **decoder-only**.
- If the task is transformation from input to output, think **encoder-decoder**.
- If the model must predict the next token, think **causal language modeling**.
- If the model must understand masked text, think **masked language modeling**.
- If the model must attend to an input sequence while generating output, think **cross-attention**.
- If memory is tight, consider **quantization** before scaling hardware.
- If quality must stay high in a smaller footprint, consider **distillation**.

**Summary**

LLM architecture fundamentals center on how text becomes tokens, how attention builds context, and how model family choices affect training and inference. The most important exam-ready ideas are the differences between encoder-only, decoder-only, and encoder-decoder models; the role of training objectives; the impact of tokenization and context limits; and the deployment tradeoffs among quantization, distillation, and pruning. Understanding these relationships helps you choose the right model type and explain why one architecture fits a task better than another.
