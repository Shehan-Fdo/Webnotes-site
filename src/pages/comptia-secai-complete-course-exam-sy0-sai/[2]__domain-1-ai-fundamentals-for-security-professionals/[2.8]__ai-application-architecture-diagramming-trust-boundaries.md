**Why diagram AI application architecture**

AI systems are not just models. They usually include user interfaces, APIs, orchestration layers, retrieval systems, model endpoints, logging, identity services, and external tools. Diagramming these parts helps you identify where data enters, where it is transformed, and where it crosses a **trust boundary**.

A good architecture diagram answers three questions:

- What components exist?
- What data moves between them?
- Which components can be trusted equally, and which cannot?

In security work, the diagram is often more valuable than the model itself because it exposes hidden dependencies such as third-party APIs, prompt injection paths, and data leakage points.

**Core diagramming goals**

When you diagram an AI application, focus on security-relevant structure rather than visual polish.

- Show **data flow** from source to sink.
- Mark **trust boundaries** where control, ownership, or assurance changes.
- Identify **sensitive data** such as personal data, secrets, prompts, embeddings, and logs.
- Show **decision points** where the system calls tools, retrieves documents, or routes to different models.
- Distinguish **runtime components** from offline training or indexing pipelines.

A useful diagram is usually layered:

- **User layer**: humans, admins, operators, external clients
- **Application layer**: web app, mobile app, API gateway, orchestration service
- **AI layer**: prompt builder, model router, LLM (large language model), embedding service, guardrails
- **Data layer**: vector database, document store, object storage, feature store, secrets vault
- **Integration layer**: SaaS tools, ticketing systems, email, code repositories, payment systems
- **Control layer**: IAM (Identity and Access Management), logging, monitoring, policy engine, DLP (Data Loss Prevention)

**Common AI application components**

AI applications often combine deterministic software with probabilistic model behavior. That mix creates unique security concerns.

- **User interface**
  - Collects prompts, files, and commands.
  - Can expose the system to malicious input, including prompt injection.
- **API gateway**
  - Authenticates requests and enforces rate limits.
  - Often becomes the first trust boundary.
- **Orchestrator**
  - Builds prompts, selects tools, and manages conversation state.
  - Can accidentally pass untrusted content into privileged actions.
- **Model endpoint**
  - May be hosted internally or consumed as a managed service.
  - Can be a local model, private endpoint, or external API.
- **Retrieval system**
  - Pulls documents from a knowledge base for retrieval-augmented generation (RAG).
  - Can introduce stale, poisoned, or unauthorized content.
- **Vector database**
  - Stores embeddings used for semantic search.
  - Should be treated as sensitive if it indexes proprietary or personal data.
- **Tooling and plugins**
  - Connect the model to external systems such as calendars, code repos, or cloud services.
  - Expand the blast radius if compromised.
- **Logging and telemetry**
  - Capture prompts, outputs, traces, and errors.
  - Often become an overlooked data exposure path.

**Trust boundaries in AI systems**

A trust boundary is any point where data or control crosses from one security domain to another. In AI systems, boundaries are common and easy to miss because the application may look like a single service while actually spanning many systems.

Typical trust boundaries include:

- **User to application**
  - Untrusted user input enters the system.
- **Application to model**
  - The application sends prompts or context to a model that may be external or less trusted.
- **Application to retrieval store**
  - The app queries documents that may have different ownership or classification.
- **Application to external tools**
  - The model or orchestrator invokes third-party services with elevated privileges.
- **Internal network to cloud service**
  - Data leaves a controlled environment and enters a provider-managed environment.
- **Production to development**
  - Test data, logs, or model artifacts move between environments with different protections.

When you cross a boundary, you should ask:

- Is the data authenticated?
- Is the data validated?
- Is the data minimized?
- Is the destination allowed to receive this data?
- Is the action reversible or auditable?

**How to diagram trust boundaries**

Use boundaries to show where assumptions change. A boundary can be a box, dashed line, shaded zone, or labeled zone. The exact style matters less than consistency.

A practical approach:

1. Draw the system from the outside in.
2. Place users and external systems outside the most trusted zone.
3. Group components by control plane, data plane, and external dependencies.
4. Mark every place where data leaves one zone and enters another.
5. Label the type of trust change, such as `internet`, `internal`, `partner`, `managed service`, or `privileged admin`.

Example boundary labels:

```text
[Internet]
   |
   v
[WAF/API Gateway] --- [Auth Service]
   |
   v
[App Orchestrator] --- [Secrets Vault]
   |
   +--> [RAG Retriever] ---> [Vector DB]
   |
   +--> [LLM API Provider]
   |
   +--> [Ticketing / Email / Code Tools]
```

This kind of diagram makes it easier to see where you need authentication, authorization, encryption, logging, and content filtering.

**Data flow analysis**

Data flow analysis tracks how information moves through the system. In AI applications, this is essential because prompts can contain sensitive data and model outputs can trigger actions.

Track these data types explicitly:

- **User input**
  - Free text, files, voice transcripts, images, or structured forms.
- **System prompts**
  - Hidden instructions that shape model behavior.
- **Context data**
  - Conversation history, retrieved documents, memory, and metadata.
- **Secrets**
  - API keys, tokens, certificates, and credentials.
- **Outputs**
  - Answers, summaries, code, recommendations, and actions.
- **Telemetry**
  - Logs, traces, metrics, and audit records.

A secure diagram shows both the path and the classification of each data type. For example, a prompt may be low sensitivity, while the same prompt combined with customer records becomes high sensitivity.

**RAG and boundary placement**

RAG systems are common in enterprise AI because they let the model answer using internal documents without retraining. They also create extra trust boundaries.

Typical RAG flow:

1. User submits a question.
2. Orchestrator creates a search query.
3. Retriever queries a document store or vector database.
4. Relevant chunks are inserted into the prompt.
5. Model generates an answer.
6. Output is logged or sent to the user.

Security concerns include:

- **Prompt injection** inside retrieved documents
- **Data poisoning** in the knowledge base
- **Overexposure** of documents due to weak access control
- **Stale content** causing incorrect answers
- **Cross-tenant leakage** in shared retrieval infrastructure

A strong diagram separates:

- The user request path
- The retrieval path
- The generation path
- The logging path

That separation helps you see where to apply filtering and authorization.

**Tool use and agentic workflows**

Agentic AI systems can call tools, make decisions, and chain actions. This increases capability and risk.

Common tool-use patterns:

- Search the web
- Query a database
- Create a ticket
- Send an email
- Run a script
- Deploy code
- Access cloud resources

Each tool call is a trust boundary because the model may influence a privileged action. You should diagram:

- Which tools are available
- What permissions each tool has
- Whether the model can call tools directly or through a policy engine
- Whether human approval is required for high-impact actions

> If a model can trigger actions, treat the action path as more sensitive than the chat path.

**Example architecture diagram in text**

The following example shows a common enterprise AI assistant pattern:

```text
[User]
  |
  v
[Web App] -> [API Gateway] -> [Orchestrator] -> [Prompt Builder]
                                   |                |
                                   |                +--> [Policy / Guardrails]
                                   |
                                   +--> [RAG Retriever] -> [Vector DB] -> [Document Store]
                                   |
                                   +--> [LLM Provider API]
                                   |
                                   +--> [Tool Executor] -> [Email / Ticketing / CRM]
                                   |
                                   +--> [Logging / SIEM]
```

Security interpretation:

- The **web app** and **API gateway** form the first control point.
- The **orchestrator** is a high-value component because it can combine user input, retrieved data, and tool access.
- The **LLM provider API** is a trust boundary if it is external.
- The **tool executor** is especially sensitive because it can convert text into real-world actions.
- The **logging/SIEM** path must avoid storing secrets or regulated data unnecessarily.

**Practical diagramming conventions**

Use consistent symbols so the diagram is readable by both technical and nontechnical stakeholders.

- **Boxes** for components
- **Arrows** for data flow
- **Dashed lines** for trust boundaries
- **Color** for sensitivity levels if your tooling supports it
- **Labels** for protocols, such as `HTTPS`, `mTLS`, `REST`, `gRPC`, or `S3`
- **Icons** sparingly, only when they improve clarity

Recommended labels:

- `untrusted input`
- `validated input`
- `retrieved context`
- `system prompt`
- `tool call`
- `audit log`
- `encrypted at rest`
- `encrypted in transit`

**Threats that diagrams help reveal**

Diagramming is not just documentation. It is a threat modeling aid.

Common issues you can spot visually:

- **Prompt injection**
  - Untrusted content influences instructions or tool calls.
- **Data exfiltration**
  - Sensitive data flows to logs, external models, or third-party tools.
- **Privilege escalation**
  - A low-trust component can trigger high-trust actions.
- **Model inversion or leakage**
  - Sensitive training or retrieval data may be exposed through outputs.
- **Supply chain risk**
  - External models, plugins, or libraries introduce hidden dependencies.
- **Cross-environment contamination**
  - Test data or secrets leak into production workflows.

**Security controls to map onto the diagram**

Once the architecture is visible, map controls to the boundaries and flows.

- **Authentication**
  - Verify user and service identity.
- **Authorization**
  - Limit what each actor and service can access.
- **Input validation**
  - Sanitize prompts, files, and tool parameters.
- **Content filtering**
  - Detect unsafe or disallowed content before model use or tool execution.
- **Secrets management**
  - Store credentials in a vault, not in prompts or code.
- **Encryption**
  - Protect data in transit and at rest.
- **Logging and auditing**
  - Record actions without leaking sensitive content.
- **Rate limiting**
  - Reduce abuse, scraping, and denial-of-service risk.
- **Human approval**
  - Require review for high-impact actions such as payments or deployments.

**Example control mapping**

| Boundary or flow | Risk | Typical control |
|---|---|---|
| User to app | Malicious input | Validation, auth, rate limiting |
| App to LLM | Sensitive prompt leakage | Minimization, redaction, encryption |
| App to retrieval store | Unauthorized document access | Access control, tenant isolation |
| Model to tool | Unsafe action execution | Policy engine, approval workflow |
| App to logs | Secret exposure | Redaction, structured logging |
| Internal to external API | Data exfiltration | Egress control, allowlists, DLP |

**CLI and implementation examples**

You may not always build diagrams with code, but practical commands help you verify the architecture you drew.

Check network paths and listening services:

```bash
ss -tulpn
ip route
curl -I https://api.example.com
```

Inspect containerized AI services:

```bash
docker ps
docker inspect ai-orchestrator
docker logs ai-orchestrator --tail 50
```

Review Kubernetes resources if the AI stack runs in a cluster:

```bash
kubectl get pods -n ai
kubectl get svc -n ai
kubectl describe networkpolicy -n ai
```

Test an API gateway boundary:

```bash
curl -X POST https://gateway.example.com/chat \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"message":"Summarize the policy document"}'
```

Example of a simple policy check before tool execution:

```python
def can_execute_tool(user_role, tool_name, action):
    high_risk_tools = {"email_send", "deploy_code", "delete_record"}
    if tool_name in high_risk_tools and user_role != "admin":
        return False
    if action.startswith("transfer") and user_role not in {"finance", "admin"}:
        return False
    return True
```

Example of redacting sensitive fields before logging:

```python
import re

def redact(text):
    text = re.sub(r'(?i)api[_-]?key\s*[:=]\s*\S+', 'api_key=[REDACTED]', text)
    text = re.sub(r'(?i)password\s*[:=]\s*\S+', 'password=[REDACTED]', text)
    return text
```

**How to think about trust levels**

Not every component needs the same level of trust. A useful mental model is to classify components by control and exposure.

- **High trust**
  - Internal services with strong identity, strict access control, and audited changes.
- **Medium trust**
  - Managed services or shared internal platforms with limited control.
- **Low trust**
  - User input, external documents, public APIs, and third-party content.

Do not assume a component is trustworthy just because it is inside your network. In AI systems, internal data can still be malicious, stale, or overprivileged.

**Common exam-style distinctions**

Be ready to distinguish these concepts:

- **Architecture diagram**
  - Shows components and relationships.
- **Data flow diagram**
  - Shows how data moves between components.
- **Trust boundary**
  - Shows where security assumptions change.
- **Threat model**
  - Identifies what can go wrong and how to reduce risk.
- **Control mapping**
  - Shows which safeguards protect each boundary or flow.

If a question asks where to place a boundary, look for:

- Different owners
- Different authentication domains
- Different network zones
- Different data classifications
- Different privilege levels
- External services or SaaS integrations

**Best practices for secure AI diagrams**

- Keep the diagram current as the system changes.
- Show external dependencies explicitly.
- Mark where prompts are assembled and where outputs are consumed.
- Separate retrieval, generation, and action execution.
- Treat logs as a data destination, not just an operational tool.
- Highlight any path that can move from text to action.
- Review diagrams with both security and application teams.

**Quick review checklist**

- Does the diagram show all major AI components?
- Are trust boundaries clearly labeled?
- Are external services and APIs visible?
- Are sensitive data flows marked?
- Are tool calls separated from chat responses?
- Are logging and monitoring paths included?
- Are controls mapped to the highest-risk boundaries?

**Key takeaway**

AI application architecture diagrams help you see where trust changes, where data can leak, and where model behavior becomes operational risk. The most important skill is not drawing every box; it is identifying the boundaries where validation, authorization, redaction, and approval must happen.
