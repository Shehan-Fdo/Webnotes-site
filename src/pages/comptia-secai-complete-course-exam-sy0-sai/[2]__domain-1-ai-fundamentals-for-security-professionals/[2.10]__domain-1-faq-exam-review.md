## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Domain 1 overview**

Domain 1 typically tests whether you can explain AI security fundamentals, identify system components, and reason about risk across the AI lifecycle. Expect scenario questions that mix terminology, governance, compliance, and operational controls. The exam often rewards you for understanding *why* a control exists, not just what it is called.

At a high level, you should be able to describe:

- **AI/ML terminology** and how it differs from traditional software security.
- **System components** such as data sources, models, prompts, retrieval layers, tools, and orchestration.
- **Lifecycle stages** from data collection through deployment, monitoring, and retirement.
- **Governance and compliance** expectations for privacy, accountability, and auditability.
- **Secure development and operations** practices for AI systems.

**AI and ML fundamentals**

AI security questions often assume you know the basic vocabulary. AI is the broader field of systems that perform tasks associated with human intelligence. <abbr title="Machine Learning">ML</abbr> is a subset of AI where models learn patterns from data rather than following only explicit rules. Deep learning uses multi-layer neural networks, and generative AI produces new content such as text, images, code, or audio.

Key terms you should be able to distinguish:

- **Model**: The trained artifact that produces predictions or generated output.
- **Training data**: Data used to fit the model.
- **Inference**: Using a trained model to produce output on new input.
- **Prompt**: The input that steers a generative model.
- **Embedding**: A numeric representation of text, images, or other data used for similarity search and retrieval.
- **Token**: A unit of text processed by many language models.
- **Hallucination**: Plausible but incorrect output generated by a model.
- **Fine-tuning**: Additional training to adapt a base model to a specific task or domain.

A useful exam distinction is that traditional software usually fails because of code defects, while AI systems can fail because of data quality, model behavior, prompt manipulation, or distribution shift. That means your security controls must cover both software and data.

**AI system components and attack surface**

A modern AI application is usually more than a model endpoint. It often includes a pipeline of services and data flows that create a larger attack surface.

Common components include:

- **Data ingestion** from files, APIs, logs, sensors, or user submissions.
- **Preprocessing** for cleaning, normalization, labeling, and feature extraction.
- **Training pipeline** for model fitting, evaluation, and artifact creation.
- **Model registry** for versioning and approval of model artifacts.
- **Inference service** for runtime predictions or generation.
- **Retrieval-augmented generation** <abbr title="Retrieval-Augmented Generation">RAG</abbr> components that fetch external context.
- **Tool or function calling** that lets the model trigger APIs, scripts, or workflows.
- **Orchestration layer** that routes prompts, tools, and memory.
- **Monitoring and logging** for quality, safety, and security events.

A simple reference topology looks like this:

```text
User -> App/API -> Prompt Filter -> LLM -> Retrieval Layer -> Internal Docs
                         |             |
                         |             -> Tool/API Calls -> SaaS/DB/ITSM
                         v
                    Logging/SIEM
```

Security issues can appear at every hop:

- **Data ingestion** may introduce poisoned or sensitive data.
- **Retrieval** may expose unauthorized documents.
- **Tool calls** may execute unsafe actions if permissions are too broad.
- **Logging** may leak prompts, secrets, or personal data if not sanitized.

**AI lifecycle and security controls**

The AI lifecycle is a recurring exam theme. You should know the stages and the security concerns at each stage.

1. **Problem definition**
   - Define the business use case, risk tolerance, and success criteria.
   - Identify whether the system handles regulated, sensitive, or safety-critical data.
2. **Data collection and preparation**
   - Validate source trustworthiness, labeling quality, and consent.
   - Remove or minimize unnecessary personal data.
3. **Training and tuning**
   - Protect training data, code, and compute resources.
   - Track model versions, hyperparameters, and evaluation results.
4. **Evaluation and validation**
   - Test accuracy, robustness, bias, safety, and security behavior.
   - Include adversarial testing where appropriate.
5. **Deployment**
   - Enforce access control, secrets management, and environment separation.
   - Approve only signed or reviewed artifacts when possible.
6. **Monitoring and maintenance**
   - Watch for drift, abuse, prompt injection, and anomalous tool use.
   - Reassess risk after model updates or data changes.
7. **Retirement**
   - Archive artifacts according to retention policy.
   - Revoke credentials, remove endpoints, and preserve audit evidence.

A common exam point is that security is not a one-time gate. You need controls across the full lifecycle, especially when models are retrained or connected to new tools.

**Governance, accountability, and control mapping**

Governance questions usually focus on who is responsible, what policy applies, and how exceptions are handled. In practice, governance creates the rules that security teams, developers, and business owners must follow.

A typical policy hierarchy is:

- **Policy**: High-level mandatory direction from leadership.
- **Standard**: Specific required rules that support the policy.
- **Procedure**: Step-by-step instructions for implementation.
- **Guideline**: Recommended practices that are not always mandatory.

Important governance concepts:

- **Accountability** means a named owner is responsible for the AI system and its risks.
- **Risk acceptance** means leadership formally accepts a residual risk after review.
- **Exception handling** means a documented deviation from standard controls with approval, scope, and expiration.
- **Control mapping** means linking AI risks to existing controls, such as access control, logging, encryption, or change management.
- **Segregation of duties** reduces the chance that one person can build, approve, and deploy a risky model without oversight.

For exam scenarios, map AI controls to familiar security domains:

| AI risk area | Example control |
|---|---|
| Data leakage | Data classification, DLP, encryption, redaction |
| Unauthorized model changes | Change control, code review, signed artifacts |
| Unsafe tool execution | Least privilege, allowlists, approval workflows |
| Prompt injection | Input filtering, output constraints, retrieval scoping |
| Poor auditability | Central logging, immutable records, model versioning |

A practical governance workflow might look like this:

1. Document the use case and data classification.
2. Perform a risk assessment and identify control gaps.
3. Assign an accountable owner and approver.
4. Decide whether to mitigate, transfer, avoid, or accept the risk.
5. Record exceptions with expiration dates and compensating controls.
6. Review the system periodically or after major changes.

**Compliance, privacy, and legal themes**

Domain 1 often includes compliance-oriented scenarios. You do not need to memorize every law, but you should understand the recurring themes.

Common compliance concerns in AI contexts:

- **Privacy**: Personal data should be minimized, protected, and used only for approved purposes.
- **Consent**: Users may need notice or consent before their data is used for training or retention.
- **Data residency**: Data may need to stay in a specific country or region.
- **Retention**: Prompts, outputs, logs, and training data should follow defined retention schedules.
- **Audit requirements**: You may need evidence of access, changes, approvals, and model decisions.
- **Purpose limitation**: Data collected for one purpose should not be reused without authorization.
- **Cross-border transfer**: Moving data to another region or vendor may trigger legal review.

In AI systems, privacy risk often increases because prompts can contain secrets, customer data, or regulated content. Retrieval systems can also expose documents beyond the intended audience if authorization is not enforced at query time.

Practical compliance examples:

- A customer support chatbot should not retain full conversation logs indefinitely if the policy requires short retention.
- A healthcare assistant should avoid sending protected health information to a third-party model unless the contract, controls, and legal review allow it.
- A multinational deployment may need region-specific model hosting to satisfy residency requirements.

> If a scenario mentions regulated data, assume you must consider classification, consent, retention, residency, and auditability before deployment.

**Secure AI development practices**

Secure SDLC (Software Development Life Cycle) still applies, but AI adds extra gates around data and model behavior. You should expect questions about how to build and release AI systems safely.

Good practices include:

- **Threat modeling** for prompts, retrieval, tools, training data, and model outputs.
- **Code review** for application code, prompt templates, orchestration logic, and API integrations.
- **Testing gates** before promotion to production.
- **Validation** of training and evaluation pipelines.
- **Artifact signing** or integrity checks for model packages and dependencies.
- **Environment separation** between development, test, staging, and production.
- **Dependency management** for libraries, model weights, and container images.

A secure AI release pipeline might include:

```bash
# Example: verify container image and scan dependencies
trivy image myregistry.example.com/ai-app:1.4.2
cosign verify myregistry.example.com/ai-app:1.4.2
```

```yaml
# Example: CI gate for model promotion
stages:
  - lint
  - test
  - security_scan
  - eval
  - approve
  - deploy

eval:
  script:
    - python evaluate_model.py --dataset eval.json --threshold 0.92
```

Testing should include more than accuracy:

- **Functional tests** for expected behavior.
- **Security tests** for injection, data leakage, and privilege escalation.
- **Bias and fairness checks** where relevant.
- **Robustness tests** for malformed inputs and adversarial prompts.
- **Regression tests** to ensure updates do not break prior safety behavior.

For training and evaluation pipelines, protect the integrity of:

- Source datasets.
- Labeling workflows.
- Feature stores.
- Evaluation sets.
- Model registry entries.
- Promotion approvals.

**Access control and secrets management**

AI systems often fail because permissions are too broad. The model itself may be harmless, but the tools it can call are not. Use least privilege everywhere.

Practical access control patterns:

- Give the application only the API scopes it needs.
- Restrict model registry write access to a small release group.
- Separate read-only retrieval access from document administration.
- Use service accounts instead of shared human credentials.
- Rotate credentials and store them in a secrets manager.

Example of scoping a tool or API permission:

```json
{
  "service_account": "ai-assistant-prod",
  "permissions": [
    "docs.read:department=finance",
    "tickets.create",
    "tickets.read:own"
  ],
  "denied": [
    "payments.refund",
    "admin.*"
  ]
}
```

Secrets management should avoid hardcoding keys in prompts, notebooks, or source code.

- Store secrets in a vault or cloud secret manager.
- Inject secrets at runtime.
- Mask secrets in logs and traces.
- Rotate keys after exposure or role changes.
- Use short-lived tokens when possible.

A common exam trap is assuming prompt filtering alone is enough. It is not. You also need authorization checks, scoped credentials, and safe tool design.

**Monitoring, logging, and telemetry**

Monitoring is essential because AI behavior changes over time and can be manipulated in real time. Good telemetry helps you detect abuse, drift, and policy violations.

Useful logging sources include:

- **Prompt logs** with redaction for secrets and personal data.
- **Retrieval logs** showing which documents or chunks were fetched.
- **Model version logs** identifying the exact model and prompt template used.
- **Tool invocation logs** recording API calls, parameters, and outcomes.
- **Policy decision logs** showing allow, block, or escalate actions.
- **User identity and session logs** for attribution and investigation.
- **Training and deployment logs** for change tracking and rollback.

Monitoring signals to watch:

- Sudden spikes in token usage or request volume.
- Repeated blocked prompts or jailbreak attempts.
- Unexpected document retrieval outside normal scope.
- Unusual tool calls, especially destructive actions.
- Drift in output quality or safety scores.
- Changes in latency that may indicate abuse or failure.

A practical logging example:

```json
{
  "timestamp": "2026-04-26T14:22:31Z",
  "user": "u12345",
  "model_version": "llm-3.2.1",
  "prompt_id": "p-8891",
  "retrieval_ids": ["doc-44", "doc-81"],
  "tool_calls": [
    {"name": "create_ticket", "status": "allowed"}
  ],
  "policy_result": "allow"
}
```

Keep in mind that logs themselves can become sensitive data. Redact secrets, limit access, and define retention.

**Threats and attack patterns**

Domain 1 often asks you to recognize common AI threats. You should know both the attack and the control that reduces it.

Common threats include:

- **Prompt injection**: Malicious input tries to override system instructions or manipulate tool use.
- **Indirect prompt injection**: Malicious instructions hidden in retrieved content, web pages, or documents.
- **Data poisoning**: Corrupting training or retrieval data to influence model behavior.
- **Model inversion**: Inferring sensitive training data from model outputs or behavior.
- **Membership inference**: Determining whether a specific record was in the training set.
- **Model theft**: Copying model behavior through repeated queries or unauthorized access.
- **Supply chain compromise**: Tampered datasets, dependencies, containers, or model artifacts.
- **Jailbreaks**: Attempts to bypass safety controls and content restrictions.

Controls that help:

- Input validation and content filtering.
- Retrieval authorization and document sanitization.
- Dataset provenance checks and approval workflows.
- Rate limiting and abuse detection.
- Output filtering and policy enforcement.
- Model watermarking or usage monitoring where appropriate.
- Signed artifacts and trusted repositories.

Example of a simple prompt-handling policy:

```python
def handle_prompt(user_prompt, user_role):
    if contains_secrets(user_prompt):
        return "Blocked: sensitive data detected"
    if not is_allowed_role(user_role):
        return "Blocked: insufficient privileges"
    context = retrieve_authorized_docs(user_role, user_prompt)
    response = model.generate(user_prompt, context=context)
    return sanitize_output(response)
```

**Incident response for AI systems**

AI incident response should be adapted to the failure mode. The first step is usually to contain the issue without destroying evidence.

AI-specific incident examples:

- **Prompt injection incident**
  - Disable the affected tool or retrieval source.
  - Preserve prompts, retrieval results, and tool logs.
  - Identify whether the model exposed data or executed unsafe actions.
- **Poisoned dataset discovery**
  - Quarantine the dataset and stop retraining jobs.
  - Rebuild from trusted sources and compare hashes or lineage.
  - Review downstream model versions that used the dataset.
- **Compromised model registry event**
  - Revoke access, isolate the registry, and verify artifact integrity.
  - Roll back to a known-good signed model.
  - Check whether unauthorized deployments occurred.
- **Sensitive output leakage**
  - Remove or redact exposed content where possible.
  - Notify stakeholders according to policy and legal requirements.
  - Review logging, retention, and access controls.

A practical response sequence is:

1. Contain the system or feature.
2. Preserve evidence, including prompts, logs, artifacts, and approvals.
3. Determine scope and impact.
4. Eradicate the root cause.
5. Recover with validated artifacts.
6. Document lessons learned and update controls.

**FAQ-style exam reminders**

- **Is AI security just application security?**
  - No. It includes application security, data security, model governance, and operational controls.
- **Do I need to know the math behind models?**
  - Usually not in depth. Focus on security implications, lifecycle, and control selection.
- **What matters most in scenario questions?**
  - Data sensitivity, access scope, model provenance, logging, and whether the control matches the risk.
- **Should I trust model output?**
  - No. Treat it as untrusted until validated, especially for decisions, code, or compliance content.
- **What is the best first control for AI risk?**
  - Usually data classification, least privilege, and clear governance, because they reduce many downstream issues.

**What to remember for the exam**

- Know the AI lifecycle and the security concern at each stage.
- Distinguish model behavior issues from software defects.
- Map AI risks to governance, access control, logging, and change management.
- Recognize privacy, residency, retention, consent, and audit requirements.
- Understand secure development gates for code, data, and model artifacts.
- Be ready to choose the control that best reduces the specific risk in the scenario.
