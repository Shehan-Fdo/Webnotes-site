## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Why logging and observability matter for LLM apps**

Large language model (LLM) applications fail in ways that are often subtle: prompt injection, data leakage, tool abuse, hallucinated actions, rate-limit storms, and degraded retrieval quality. Logging and observability give you the evidence to detect these issues, understand impact, and respond quickly.

For exam scenarios, separate three ideas:

- **Logging** records discrete events for audit, debugging, and forensics.
- **Observability** helps you infer internal system state from metrics, traces, and logs.
- **Incident response** defines how you contain, eradicate, recover, and document a security event.

LLM systems need all three because the model itself is only one component. You also have application code, retrieval pipelines, vector databases, tool integrations, identity systems, and external APIs.

**Core telemetry for LLM applications**

A practical telemetry strategy captures both security and operational signals. You want enough detail to reconstruct what happened without storing unnecessary sensitive content.

Common event types include:

- **Prompt events**: user input, system prompt version, template ID, policy decisions, and redaction status.
- **Completion events**: model name, temperature, token counts, latency, refusal reason, and safety filter outcome.
- **Retrieval events**: query text hash, document IDs, embedding model version, top-k results, and relevance scores.
- **Tool events**: tool name, arguments, authorization result, execution time, and output summary.
- **Identity events**: user ID, session ID, API key ID, OAuth subject, and privilege changes.
- **Security events**: blocked prompt injection, jailbreak detection, anomalous token usage, and data exfiltration indicators.

A good log entry is structured, timestamped, and correlated across services. Use a shared request ID or trace ID so you can follow one user interaction through the API gateway, orchestration layer, model call, retrieval service, and downstream tools.

**Structured vs. unstructured logs**

LLM environments produce both human-readable text and machine-parseable events. In exam questions, prefer structured logging for security-relevant data.

- **Structured logs** use JSON or another schema so you can query fields reliably.
- **Unstructured logs** are free-form text and are harder to search, correlate, and alert on.
- **Hybrid logging** is common: structured metadata plus a redacted text summary.

A recommended JSON schema includes:

- `timestamp`
- `service`
- `environment`
- `request_id`
- `trace_id`
- `user_id`
- `session_id`
- `model`
- `prompt_hash`
- `response_hash`
- `tool_name`
- `decision`
- `risk_score`
- `latency_ms`
- `token_in`
- `token_out`
- `status`
- `error_code`

Example structured log:

```json
{
  "timestamp": "2026-04-26T14:22:31.184Z",
  "service": "llm-gateway",
  "environment": "prod",
  "request_id": "req-8f2c1a",
  "trace_id": "3f2b9d7c1a4e",
  "user_id": "u-10482",
  "session_id": "sess-77aa",
  "model": "gpt-4.1",
  "prompt_hash": "sha256:9c1d...",
  "response_hash": "sha256:2ab4...",
  "decision": "allow",
  "risk_score": 0.12,
  "latency_ms": 842,
  "token_in": 512,
  "token_out": 214,
  "status": "success"
}
```

**Log collection and forwarding patterns**

Exam questions may ask how logs move from the application to a central platform. Know the common collection patterns and when to use them.

- **Agent-based collection** installs a local process on the host or node to read files, enrich events, and forward them.
  - Examples include Fluent Bit, Filebeat, Vector, and OpenTelemetry Collector.
  - Use this when you need local buffering, parsing, or host-level context.
- **Sidecar collection** runs a companion container next to the app container.
  - This is common in Kubernetes when you want per-pod isolation and easy deployment.
  - Use this when the app should not manage log shipping directly.
- **Syslog forwarding** sends events over UDP or TCP to a central syslog receiver.
  - Syslog is common for network devices, Linux hosts, and legacy systems.
  - Prefer TCP or TLS when reliability and confidentiality matter.
- **OpenTelemetry** provides a vendor-neutral way to collect logs, metrics, and traces.
  - It is useful when you want consistent instrumentation across services.
  - The OpenTelemetry Collector can receive telemetry and export it to multiple back ends.
- **SIEM ingestion** sends normalized events into a Security Information and Event Management platform.
  - SIEMs correlate logs from apps, identity providers, cloud services, and endpoints.
  - Examples include Splunk, Microsoft Sentinel, QRadar, and Elastic Security.

A typical pipeline looks like this:

```text
LLM app -> local agent/sidecar -> log processor -> message queue or collector -> SIEM / data lake / observability platform
```

Practical forwarding example with Fluent Bit:

```ini
[INPUT]
    Name tail
    Path /var/log/llm-app/*.log
    Parser json
    Tag llm.app

[FILTER]
    Name modify
    Match llm.app
    Add environment prod

[OUTPUT]
    Name http
    Match llm.app
    Host siem.example.com
    Port 443
    URI /ingest
    Format json
    tls On
```

Syslog example:

```bash
logger -p authpriv.notice -t llm-gateway "request_id=req-8f2c1a model=gpt-4.1 decision=blocked reason=prompt_injection"
```

**Parsing, field extraction, and schema design**

Raw logs are only useful if you can query them consistently. Parsing and field extraction turn text into fields that support dashboards, alerts, and investigations.

Key design principles:

- **Normalize early** so the same concept uses the same field name across services.
- **Preserve context** such as request IDs, user IDs, and model version.
- **Avoid over-logging** sensitive prompts, secrets, or full tool outputs.
- **Version your schema** so downstream dashboards do not break when fields change.

When logs are unstructured, use parsers or grok patterns to extract fields. When logs are structured, validate them against a schema and reject malformed events.

Example parsing with a regex-style extraction:

```text
timestamp=2026-04-26T14:22:31Z service=llm-gateway user=u-10482 model=gpt-4.1 decision=blocked risk=0.91 reason=prompt_injection
```

Extracted fields:

- `service = llm-gateway`
- `user = u-10482`
- `model = gpt-4.1`
- `decision = blocked`
- `risk = 0.91`
- `reason = prompt_injection`

A useful schema strategy is to separate:

- **Identity fields**: who initiated the request.
- **Context fields**: where and when it happened.
- **Model fields**: which model and configuration were used.
- **Security fields**: policy decision, risk score, and detection result.
- **Outcome fields**: success, failure, latency, and downstream effects.

**Metrics, traces, and dashboards**

Observability is broader than logging. You use metrics for trends, traces for request flow, and logs for detail.

- **Metrics** are numeric time-series values such as latency, error rate, token usage, and blocked requests.
- **Traces** show the path of a request across services and are essential for debugging multi-step LLM workflows.
- **Logs** provide the event-level evidence behind a metric spike or trace anomaly.

Useful LLM metrics include:

- request count
- p50/p95/p99 latency
- token input/output volume
- model refusal rate
- safety filter hit rate
- retrieval hit rate
- tool invocation count
- tool failure rate
- cache hit rate
- cost per request
- prompt injection detection rate

Example Prometheus metrics:

```promql
rate(llm_requests_total[5m])
histogram_quantile(0.95, sum(rate(llm_request_latency_seconds_bucket[5m])) by (le))
rate(llm_policy_blocks_total[5m])
rate(llm_tool_failures_total[5m])
```

Example dashboard panels:

- Requests per minute by model
- p95 latency by endpoint
- Blocked prompts by reason
- Tool calls by user role
- Retrieval failures by index
- Token spend by tenant

A trace example helps you identify whether the issue is in the model, retrieval layer, or tool execution. If latency spikes, a trace may show that the model call is fast but the vector database query is slow, which changes your remediation plan.

**SLOs, SLIs, and alert thresholds**

An **SLI** (Service Level Indicator) is the measured signal. An **SLO** (Service Level Objective) is the target you want to meet. Alert thresholds should usually map to SLO risk, not just arbitrary numbers.

Examples:

- **SLI**: 99th percentile response latency.
- **SLO**: 99% of requests complete in under 2 seconds.
- **Alert threshold**: page when the error budget burn rate indicates the SLO will be violated soon.

For LLM apps, useful SLIs include:

- successful response rate
- policy block rate
- hallucination proxy rate, if you have a validated internal measure
- retrieval relevance score
- tool execution success rate
- unauthorized tool invocation attempts
- sensitive data leakage detections

Alerting guidance:

- Alert on sustained deviations, not one-off noise.
- Use separate thresholds for warning and critical conditions.
- Tie alerts to user impact and security impact.
- Suppress duplicate alerts during a known incident.

Example alert logic:

```yaml
groups:
  - name: llm-alerts
    rules:
      - alert: HighLLMLatency
        expr: histogram_quantile(0.95, sum(rate(llm_request_latency_seconds_bucket[5m])) by (le)) > 2
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "LLM p95 latency above 2s"

      - alert: PromptInjectionSpike
        expr: rate(llm_policy_blocks_total{reason="prompt_injection"}[5m]) > 5
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Prompt injection detections spiking"
```

**Security monitoring use cases**

LLM apps create new detection opportunities. You should monitor for both abuse and misconfiguration.

Common detections include:

- repeated jailbreak attempts from one account
- sudden increases in token usage
- tool calls from unusual geographies or IPs
- access to restricted retrieval collections
- prompt patterns that request secrets or system prompts
- output containing API keys, credentials, or personal data
- model version changes outside change windows

Example SIEM correlation logic:

```text
IF user_id = same AND
   prompt_injection_blocks >= 3 IN 10m AND
   token_usage > baseline * 4 AND
   tool_calls include "export_data"
THEN create critical incident
```

Example Splunk-style query:

```spl
index=llm_prod sourcetype=llm:events
| stats count as events, sum(token_out) as tokens_out, values(decision) as decisions by user_id, session_id
| where events > 20 AND tokens_out > 5000
```

Example Elastic-style query:

```text
event.dataset: llm AND security.decision: blocked AND security.reason: prompt_injection
```

**Incident response lifecycle for LLM apps**

Use a standard incident response lifecycle: preparation, detection, analysis, containment, eradication, recovery, and lessons learned. The LLM-specific part is that you often need to contain both the application and the model access path.

1. **Prepare**
   - Define roles, escalation paths, and communication channels.
   - Pre-stage logging, snapshots, and access controls.
   - Keep runbooks for prompt injection, data leakage, and tool abuse.
2. **Detect and analyze**
   - Confirm the event with logs, traces, and SIEM correlation.
   - Determine scope: affected users, models, tools, datasets, and time window.
   - Preserve evidence before making changes.
3. **Contain**
   - Disable risky tools or routes.
   - Switch to a safe fallback model or read-only mode.
   - Block malicious accounts, IPs, or sessions.
4. **Eradicate**
   - Remove malicious prompts, poisoned data, or compromised credentials.
   - Fix the vulnerable control or policy.
5. **Recover**
   - Restore service with heightened monitoring.
   - Validate that the issue is resolved and no new indicators appear.
6. **Lessons learned**
   - Update detections, playbooks, and controls.
   - Document root cause, impact, and corrective actions.

**Containment actions for credential and token compromise**

Containment must go beyond rotating a single secret. If an attacker obtains credentials or session artifacts, you need to invalidate all related access paths.

Use these actions when compromise is suspected:

- **Rotate credentials** for API keys, service accounts, database passwords, and model provider secrets.
- **Invalidate sessions** for affected users and administrators.
- **Revoke API keys** that were exposed in prompts, logs, or tool outputs.
- **Expire access tokens** and refresh tokens.
- **Disable compromised service accounts** until you verify scope.
- **Purge cached credentials** from secret stores, CI/CD variables, and runtime environments.
- **Review delegated permissions** for tool integrations and third-party connectors.

Example operational steps:

```bash
# Revoke an exposed API key
cloud-cli api-keys revoke --id key_12345

# Invalidate user sessions
iam-cli sessions revoke --user u-10482

# Rotate a service secret
vault kv put secret/llm/prod api_key="$(openssl rand -hex 32)"
```

> If you suspect prompt leakage of secrets, treat the secret as compromised even if you do not yet have proof of exfiltration.

**Evidence handling and chain of custody**

Forensic integrity matters in exam scenarios. You need to preserve evidence so it remains trustworthy for investigation, legal review, or internal audit.

Best practices:

- **Record chain of custody** for every artifact you collect.
- **Timestamp** collection, transfer, and analysis actions.
- **Hash evidence** before and after transfer using a strong algorithm such as SHA-256.
- **Limit access** to evidence to authorized responders.
- **Preserve originals** and work from copies when possible.
- **Document context** such as who collected the data, from where, and under what conditions.

Evidence may include:

- application logs
- model gateway logs
- SIEM exports
- traces
- container logs
- cloud audit logs
- prompt and response snapshots, if policy allows
- memory captures or disk images in severe cases

Example evidence record:

```text
Artifact: llm-gateway-2026-04-26.log
Collected by: A. Patel
Source: prod-llm-gateway-3
Time collected: 2026-04-26T14:40:00Z
SHA-256: 8f3c...a91d
Stored at: evidence://case-2026-0412/artifacts/llm-gateway-2026-04-26.log
Purpose: prompt injection investigation
```

**Practical detection and response examples**

A strong exam answer usually connects telemetry to action. Use the signal to decide what to do next.

Example 1: prompt injection spike

- Detection: blocked prompt injection events rise from 2 per hour to 40 per hour.
- Investigation: same user account is probing system prompts and tool instructions.
- Response:
  - block the account
  - invalidate the session
  - review recent tool calls
  - increase logging for the affected tenant
  - update the prompt filter rules

Example 2: data leakage through retrieval

- Detection: logs show retrieval queries against restricted collections from a low-privilege role.
- Investigation: access control misconfiguration on the retrieval service.
- Response:
  - disable the collection
  - revoke the affected token
  - audit all retrieval access
  - confirm whether sensitive documents were returned
  - notify stakeholders if exposure occurred

Example 3: tool abuse

- Detection: the model repeatedly invokes an export tool with unusual parameters.
- Investigation: the prompt contains instructions to exfiltrate records.
- Response:
  - disable the export tool
  - require human approval for high-risk actions
  - inspect the trace for all downstream calls
  - preserve logs and hashes for the case file

**Real-world command-line and query examples**

Use commands that help you inspect logs quickly during an incident.

```bash
# Search recent gateway logs for blocked requests
grep '"decision":"blocked"' /var/log/llm-gateway/*.log | tail -n 20

# Count requests by model
jq -r '.model' /var/log/llm-gateway/*.log | sort | uniq -c

# Find high-latency requests
jq 'select(.latency_ms > 2000) | {request_id, user_id, model, latency_ms}' /var/log/llm-gateway/*.log
```

Example SQL for an analytics store:

```sql
SELECT user_id, COUNT(*) AS blocked_events, SUM(token_out) AS total_tokens
FROM llm_events
WHERE decision = 'blocked'
  AND timestamp >= NOW() - INTERVAL '15 minutes'
GROUP BY user_id
HAVING COUNT(*) > 10
ORDER BY blocked_events DESC;
```

Example dashboard query for a time-series system:

```promql
sum by (reason) (rate(llm_policy_blocks_total[5m]))
```

**Common exam traps**

- Treating logs as a substitute for observability instead of one part of it.
- Logging full prompts or outputs without redaction or retention controls.
- Ignoring trace correlation across retrieval and tool services.
- Using arbitrary alert thresholds that do not map to user impact or SLOs.
- Rotating one secret while leaving sessions, tokens, and API keys active.
- Failing to preserve evidence with hashes and chain-of-custody records.
- Assuming a model issue is only a model issue when the root cause is often in retrieval, identity, or tool authorization.

**Quick review checklist**

- You can explain the difference between logs, metrics, and traces.
- You can describe agent, sidecar, syslog, OpenTelemetry, and SIEM ingestion patterns.
- You can design a structured log schema for LLM requests.
- You can distinguish unstructured from structured logs and explain parsing needs.
- You can relate SLIs and SLOs to alert thresholds and incident triggers.
- You can outline containment steps for sessions, tokens, API keys, and credentials.
- You can preserve evidence with chain of custody and hashing.
- You can write or interpret basic alert rules and SIEM correlation queries.
