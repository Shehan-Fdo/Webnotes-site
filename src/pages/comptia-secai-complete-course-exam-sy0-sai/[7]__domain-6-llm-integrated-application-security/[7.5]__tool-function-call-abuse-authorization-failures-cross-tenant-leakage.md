## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Overview**

Agentic AI systems often connect a model to external tools such as databases, ticketing systems, email, cloud APIs, and internal knowledge bases. That connectivity creates a new attack surface: if the model can call tools, an attacker may be able to influence those calls, expand their privileges, or move across tenant boundaries. The three closely related risks in this topic are **tool and function-call abuse**, **authorization failures**, and **cross-tenant leakage**.

These issues matter because the model is not the security boundary. The real boundary is the application’s authorization layer, tenant isolation controls, and tool execution policy. If those controls are weak, a prompt injection, malicious user input, or flawed orchestration logic can cause the system to perform actions or reveal data it should never expose.

**Core concepts**

- **Tool abuse** happens when an attacker manipulates an AI system into invoking a tool in an unsafe way.
- **Function-call abuse** is a specific form of tool abuse where the model emits a structured call with dangerous arguments, unintended parameters, or unauthorized actions.
- **Authorization failure** occurs when the system does not correctly verify whether the caller may access a resource or perform an action.
- **Cross-tenant leakage** occurs when data, prompts, embeddings, logs, or tool outputs from one tenant become visible to another tenant.

A useful mental model is that the model can *suggest* actions, but the application must *decide* whether those actions are allowed. If the application trusts the model too much, the model becomes an untrusted intermediary rather than a safe automation layer.

**How tool and function-call abuse happens**

Modern LLM (large language model) applications often expose tools through function calling, JSON schemas, or agent frameworks. The model may be allowed to call functions such as `search_customer`, `create_ticket`, `send_email`, or `get_invoice`. Attackers target the gap between natural-language intent and executable action.

Common abuse patterns include:

- **Prompt injection**
  - The attacker places instructions in user content, retrieved documents, web pages, or emails.
  - The model follows the malicious instruction and calls a tool with attacker-chosen parameters.
- **Argument manipulation**
  - The attacker causes the model to pass a different `user_id`, `tenant_id`, `role`, or `resource_id` than intended.
  - The tool executes exactly as requested, even though the request should have been rejected.
- **Overbroad tool scope**
  - A single tool can read, write, delete, and export data.
  - If the model can invoke it freely, one bad call can have severe impact.
- **Indirect tool invocation**
  - The model retrieves content that contains hidden instructions.
  - The hidden instructions influence later tool calls.
- **Chained abuse**
  - One tool call leaks data that is then used to craft a second malicious call.
  - The attacker escalates from information disclosure to unauthorized action.

A simple example is an assistant that can query customer records. If the model is asked, “Show me the latest invoice for my account,” but the attacker adds “use account 1042 instead,” the model may generate a call like this:

```json
{
  "tool": "get_invoice",
  "arguments": {
    "account_id": "1042",
    "format": "pdf"
  }
}
```

If the application does not verify that the authenticated user owns account `1042`, the tool returns another customer’s invoice.

**Authorization failures**

Authorization is the process of deciding what an authenticated identity is allowed to do. In AI systems, authorization must apply at multiple layers:

- **User-to-app authorization**
  - Is the logged-in user allowed to request this action?
- **App-to-tool authorization**
  - Is the AI agent allowed to invoke this tool at all?
- **Object-level authorization**
  - Is the user allowed to access this specific record, file, or tenant?
- **Action-level authorization**
  - Is the user allowed to read, write, delete, export, approve, or send?
- **Contextual authorization**
  - Is the action allowed from this location, device, time, or risk score?

Failures often appear as variants of **broken access control**:

- **Missing object-level checks**
  - The tool accepts an ID and returns the object without verifying ownership.
- **Insecure direct object reference (IDOR)**
  - The attacker changes an identifier and accesses another user’s data.
- **Privilege escalation**
  - The model or user can trigger admin-only actions through a weak tool interface.
- **Confused deputy**
  - The AI agent has more privilege than the user and performs actions on the user’s behalf without proper delegation checks.
- **Inconsistent policy enforcement**
  - The UI blocks an action, but the API or tool endpoint still allows it.
- **Over-trusted model output**
  - The system treats the model’s generated parameters as if they were validated user intent.

A secure design treats the model as a request generator, not an authority. The backend must independently enforce policy using server-side checks, not rely on the prompt, the UI, or the model’s “understanding.”

**Cross-tenant leakage**

In multi-tenant systems, one platform instance serves multiple customers or business units. Each tenant expects strong isolation. Leakage can occur through several channels:

- **Direct data access**
  - A query or tool call returns another tenant’s records.
- **Shared retrieval indexes**
  - A vector database or search index mixes embeddings or documents from multiple tenants.
- **Prompt or conversation reuse**
  - Session state from one tenant is accidentally reused in another session.
- **Logs and traces**
  - Debug logs, telemetry, or observability tools expose sensitive prompts or outputs across tenants.
- **Cache contamination**
  - Cached responses are keyed incorrectly and served to the wrong tenant.
- **Model memory or agent state**
  - Long-lived agent context retains tenant-specific data and leaks it into later interactions.
- **Training or fine-tuning contamination**
  - Tenant data is used in a shared training pipeline without proper segregation or consent.

Cross-tenant leakage is especially dangerous because it can be silent. The system may appear to work normally while exposing confidential data to the wrong customer. In regulated environments, this can create privacy, contractual, and compliance failures.

**Typical architecture risks**

A common agentic architecture includes:

- A user interface
- An orchestration layer or agent runtime
- One or more tools or plugins
- A retrieval layer such as search or vector database
- External APIs and SaaS integrations
- Logging, monitoring, and analytics

Each layer can fail independently:

- The UI may hide restricted actions, but the API still accepts them.
- The agent may be instructed to respect tenant boundaries, but the tool ignores them.
- The retrieval layer may filter by tenant in the query, but the cache may not.
- The logging system may store raw prompts, including secrets or PII (personally identifiable information).

A secure architecture enforces isolation at the data layer, validates authorization at the service layer, and constrains tool execution at the orchestration layer.

**Practical defensive controls**

Use layered controls rather than a single safeguard.

- **Server-side authorization**
  - Enforce access checks in the backend for every tool call.
  - Never trust the model to enforce policy.
- **Least privilege**
  - Give each tool only the permissions it needs.
  - Split read, write, delete, and admin functions.
- **Scoped credentials**
  - Use short-lived tokens and tenant-scoped service accounts.
  - Avoid shared superuser credentials.
- **Allowlists**
  - Permit only approved tools, endpoints, domains, and actions.
  - Reject unexpected function names or parameters.
- **Parameter validation**
  - Validate IDs, tenant identifiers, enums, ranges, and formats.
  - Bind sensitive identifiers to the authenticated session when possible.
- **Policy engines**
  - Use centralized authorization logic such as ABAC (attribute-based access control) or RBAC (role-based access control).
- **Tenant isolation**
  - Partition data by tenant in storage, cache, search, and logs.
  - Include tenant context in every query and cache key.
- **Output filtering**
  - Redact secrets, tokens, and sensitive fields before returning tool output.
- **Human approval**
  - Require confirmation for destructive or high-risk actions.
- **Audit logging**
  - Record who requested the action, what tool ran, what data was accessed, and whether policy allowed it.

A simple policy pattern is to map the user identity to a tenant and then derive all tool access from that identity, not from model-supplied fields.

```python
def get_invoice(request, invoice_id):
    user = request.authenticated_user
    invoice = db.invoices.find_one({"id": invoice_id, "tenant_id": user.tenant_id})

    if not invoice:
        raise PermissionError("Not found or not authorized")

    return invoice
```

This pattern prevents the model from selecting an arbitrary tenant or object. The server decides the tenant scope.

**Example of safer tool design**

Instead of exposing a broad tool like `manage_customer_record`, split it into narrow actions:

```json
[
  {
    "name": "get_customer_profile",
    "description": "Read-only customer profile lookup",
    "parameters": {
      "type": "object",
      "properties": {
        "customer_id": { "type": "string" }
      },
      "required": ["customer_id"]
    }
  },
  {
    "name": "update_customer_email",
    "description": "Update email for the authenticated customer only",
    "parameters": {
      "type": "object",
      "properties": {
        "new_email": { "type": "string" }
      },
      "required": ["new_email"]
    }
  }
]
```

This design reduces blast radius. The update tool should not accept an arbitrary `customer_id` if the action is meant only for the current user.

**Prompt injection and tool abuse workflow**

A typical attack chain looks like this:

1. The attacker places malicious instructions in content the model will read.
2. The model interprets the instructions as relevant context.
3. The model generates a tool call with attacker-influenced parameters.
4. The backend executes the call without sufficient validation.
5. Sensitive data is returned or an unauthorized action is completed.

Defenses should break the chain at multiple points:

- Sanitize and isolate untrusted content.
- Distinguish instructions from data.
- Restrict which tools the model can call in each context.
- Validate every tool call server-side.
- Require explicit confirmation for risky operations.

> If a model can read untrusted text and then act on it, treat that text as potentially adversarial input, not as instructions.

**Detection and monitoring**

You should monitor for signs of abuse and leakage:

- Repeated requests for unrelated tenant IDs
- Unusual tool call frequency or sequence changes
- Access to records outside the user’s normal scope
- Large exports, bulk reads, or repeated pagination
- Tool calls triggered by suspicious document content
- Cache hits across tenant boundaries
- Unexpected prompts containing secrets, tokens, or policy text

Useful logging fields include:

- `user_id`
- `tenant_id`
- `session_id`
- `tool_name`
- `arguments_hash`
- `decision`
- `policy_reason`
- `request_id`

Example audit log entry:

```json
{
  "timestamp": "2026-04-26T12:30:00Z",
  "user_id": "u-1842",
  "tenant_id": "tenant-a",
  "tool_name": "get_invoice",
  "resource_id": "inv-9912",
  "decision": "deny",
  "policy_reason": "cross-tenant object access"
}
```

This kind of logging helps you investigate abuse without exposing full sensitive payloads in every log line.

**Testing and validation**

Security testing for these issues should include both manual and automated checks.

- **Authorization tests**
  - Verify that users cannot access objects outside their tenant.
  - Test direct object reference changes.
- **Prompt injection tests**
  - Place malicious instructions in documents, emails, and web pages.
  - Confirm the agent ignores them as instructions.
- **Tool abuse tests**
  - Attempt to trigger destructive tools from low-privilege contexts.
  - Verify confirmation and policy enforcement.
- **Tenant isolation tests**
  - Use two tenants with distinct data and confirm no cross-read or cross-write occurs.
- **Cache and retrieval tests**
  - Ensure search results, embeddings, and cached responses are tenant-scoped.
- **Regression tests**
  - Re-run abuse cases after every tool, policy, or schema change.

A practical test matrix can help:

| Test area | What you try | Expected result |
|---|---|---|
| Object access | Change `customer_id` | Request denied or scoped to current tenant |
| Tool scope | Invoke admin tool as standard user | Tool blocked |
| Retrieval | Query another tenant’s document | No result returned |
| Cache | Repeat request across tenants | No cross-tenant response |
| Prompt injection | Hidden instruction in content | Model treats it as data |

**Implementation patterns**

Use these patterns when building or reviewing AI-enabled systems:

- **Bind identity to context**
  - Derive tenant and role from the authenticated session.
- **Separate policy from generation**
  - Let the model propose; let the backend decide.
- **Minimize tool power**
  - Keep tools narrow and composable.
- **Validate every parameter**
  - Reject unexpected values early.
- **Partition data**
  - Use tenant-aware storage, indexes, and caches.
- **Review high-risk actions**
  - Add human-in-the-loop approval for payments, deletions, and external communications.

A simple request flow might look like this:

1. User authenticates.
2. Application resolves user identity, role, and tenant.
3. Agent receives only the minimum context needed.
4. Model proposes a tool call.
5. Policy engine validates the call.
6. Tool executes only if the policy allows it.
7. Output is filtered and logged.

**Key takeaways**

- **Tool abuse** is an application security problem, not just a model behavior problem.
- **Authorization failures** usually come from missing server-side checks, overbroad permissions, or trusting model-generated parameters.
- **Cross-tenant leakage** often appears in storage, retrieval, caching, logging, or session handling.
- The safest design uses least privilege, tenant-aware enforcement, and independent policy checks.
- Treat model output as untrusted input until the backend validates it.

**Quick review checklist**

- Does every tool call enforce server-side authorization?
- Are tenant IDs derived from authenticated identity rather than model output?
- Are tools split into narrow, least-privilege actions?
- Are caches, logs, and retrieval indexes tenant-scoped?
- Are prompt injection and indirect instruction attacks tested?
- Are destructive actions gated by confirmation or human review?
