## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Poisoned RAG overview**

A poisoned <abbr title="Retrieval-Augmented Generation">RAG</abbr> knowledge base is a retrieval corpus that has been intentionally or accidentally seeded with malicious, misleading, or policy-violating content. In a RAG system, the model answers by combining user prompts with retrieved documents, so a compromised corpus can directly shape outputs, leak secrets, or trigger unsafe behavior.

The containment goal is to stop the bad content from influencing retrieval, preserve evidence, and restore a trusted index without reintroducing the same contamination. In practice, you treat the knowledge base like a production security incident: isolate, identify, validate, remediate, and monitor.

- **RAG risk** is not only prompt injection; it also includes poisoned documents, malicious metadata, and compromised embeddings.
- **Containment** means preventing further retrieval of untrusted content while you investigate.
- **Recovery** requires rebuilding trust in the corpus, not just deleting one bad file.

**How poisoning happens**

Poisoning can enter through many ingestion paths. The most common failure is assuming that internal content is automatically safe. Any source that can write to the corpus, document store, vector database, or metadata pipeline can become an attack path.

- **Direct document upload** can introduce adversarial instructions, hidden text, or fabricated policy guidance.
- **Web ingestion** can pull in hostile pages, comment sections, or SEO-spam content.
- **Metadata abuse** can manipulate ranking, filters, or source trust labels.
- **Embedding poisoning** can make malicious content cluster near legitimate queries, increasing retrieval likelihood.
- **Supply chain compromise** can alter connectors, crawlers, or preprocessing jobs before indexing.

A poisoned corpus often aims to do one of three things:
- Override system instructions by embedding “ignore previous instructions” style content.
- Exfiltrate secrets by inducing the model to reveal context, prompts, or internal data.
- Degrade trust by making the assistant answer incorrectly, inconsistently, or with policy violations.

**Threat model and impact**

RAG poisoning is dangerous because retrieval happens before generation. If the retriever surfaces a malicious chunk, the model may treat it as authoritative context. Even a strong system prompt can be weakened when the retrieved text is highly relevant to the user query.

Typical impacts include:
- **Data leakage** from retrieved confidential documents or embedded secrets.
- **Hallucination amplification** when the model is anchored to false retrieved facts.
- **Policy bypass** when malicious chunks instruct the model to ignore safeguards.
- **Operational disruption** when users lose confidence in the assistant.
- **Compliance exposure** if regulated or sensitive data is indexed without authorization.

A useful mental model is that the vector store is not just a database; it is part of the model’s decision surface. If the corpus is compromised, the model’s behavior is compromised.

**Containment priorities**

Containment is about reducing blast radius first, then preserving evidence, then restoring service safely. Do not start by bulk-deleting everything unless you have no other option. You want to know what was poisoned, how it entered, and whether the same path is still open.

1. **Freeze ingestion**
   - Disable new document uploads, crawls, sync jobs, and connector refreshes.
   - Pause scheduled reindexing and embedding jobs.
   - Stop automated content promotion from staging to production.

2. **Quarantine retrieval**
   - Remove the affected index from production routing.
   - Switch the assistant to a safe fallback mode, such as static FAQ or human escalation.
   - Restrict retrieval to a known-good allowlist if partial service is required.

3. **Preserve evidence**
   - Snapshot the vector database, document store, and ingestion logs.
   - Record document IDs, hashes, timestamps, source URLs, and connector identities.
   - Export suspicious chunks before deletion if policy allows.

4. **Validate trust boundaries**
   - Confirm which sources are authoritative.
   - Check whether the poisoning came from user-generated content, a vendor feed, or an internal system.
   - Review access controls on ingestion and indexing pipelines.

> If you cannot prove a document is trustworthy, do not let it remain retrievable in production.

**Lab containment workflow**

A practical containment lab usually mirrors a real incident response flow. You isolate the corpus, inspect the poisoned artifacts, and rebuild a clean index from trusted sources.

1. **Identify the affected scope**
   - Query the assistant with known trigger phrases and compare outputs.
   - Inspect retrieval traces to see which chunks are being returned.
   - Map the poisoned documents to source systems and ingestion jobs.

2. **Isolate the corpus**
   - Disable the production retriever or route traffic to a clean replica.
   - Block the affected collection, namespace, or tenant.
   - If the platform supports it, mark suspicious documents as non-retrievable instead of deleting immediately.

3. **Triage the content**
   - Look for prompt injection phrases, hidden instructions, or policy overrides.
   - Check for unusual formatting, base64 blobs, invisible text, or repeated keywords.
   - Review metadata for suspicious authorship, timestamps, or source domains.

4. **Rebuild from trusted inputs**
   - Re-ingest only verified sources.
   - Recompute embeddings after cleaning the corpus.
   - Recreate indexes with versioned snapshots so you can roll back if needed.

5. **Test before re-enabling**
   - Run adversarial queries against the rebuilt index.
   - Confirm the retriever no longer surfaces malicious chunks.
   - Validate that the assistant answers from approved sources only.

**Example containment architecture**

A safe design separates ingestion, quarantine, and production retrieval. Suspicious content lands in a staging area first, where it is scanned and approved before indexing.

```text
[Sources] -> [Ingestion API] -> [Quarantine Store] -> [Content Review]
                                   |                     |
                                   | approved            | rejected
                                   v                     v
                             [Embedding Job]        [Evidence Archive]
                                   |
                                   v
                           [Trusted Vector Index]
                                   |
                                   v
                           [RAG Application]
```

This pattern reduces the chance that a single bad upload immediately affects production answers. It also gives you a clean place to inspect documents before they become retrievable.

**Detection signals**

You usually detect poisoning through retrieval behavior, not just static scanning. The assistant may start citing odd sources, repeating instructions that should never appear in answers, or producing inconsistent policy behavior.

Common indicators:
- **Unexpected source dominance** where one suspicious document appears in many unrelated queries.
- **Instructional content** inside retrieved chunks, especially content aimed at the model rather than the user.
- **Metadata anomalies** such as future timestamps, mismatched owners, or strange source labels.
- **Embedding drift** where a new batch of documents changes nearest-neighbor behavior.
- **User reports** of bizarre answers, secret leakage, or policy violations.

Useful checks include:
- Compare top-k retrieval results for benign and adversarial queries.
- Inspect chunk boundaries to see whether malicious instructions were split across segments.
- Review similarity scores for outlier documents that rank too highly.
- Search for terms like `ignore previous instructions`, `system prompt`, `developer message`, or `reveal secrets`.

**CLI and operational examples**

The exact tooling varies, but the containment actions are similar across platforms. The examples below show the kind of operational commands you might use in a lab or incident response workflow.

```bash
# Freeze ingestion jobs
systemctl stop rag-ingest.service
systemctl stop rag-embed.service

# Inspect recent ingestion activity
journalctl -u rag-ingest.service --since "24 hours ago"
journalctl -u rag-embed.service --since "24 hours ago"

# Snapshot a vector database volume before changes
tar -czf vectorstore-backup-$(date +%F).tgz /var/lib/vectorstore

# Search for suspicious instruction phrases in source documents
grep -RniE "ignore previous instructions|system prompt|developer message|reveal secrets" /data/corpus
```

If you use an object store or document repository, you may also quarantine by tag or prefix:

```bash
# Example: mark suspicious objects with a quarantine prefix
aws s3 mv s3://kb-prod/docs/ s3://kb-quarantine/docs/ --recursive

# Example: disable a connector temporarily
curl -X POST https://kb.example/api/connectors/vendor-feed/disable \
  -H "Authorization: Bearer $TOKEN"
```

**Validation and cleaning**

Cleaning a poisoned corpus is not just deletion. You need to verify that the malicious content is gone, the index is rebuilt, and the assistant no longer retrieves the bad material.

- **Hash verification** confirms whether a document changed unexpectedly.
- **Allowlist validation** ensures only approved sources are indexed.
- **Chunk review** checks whether malicious instructions were hidden in a small segment.
- **Re-embedding** is required after content changes because old vectors may still point to removed text.
- **Regression testing** confirms the assistant behaves correctly after remediation.

A practical validation checklist:
- Confirm the source inventory matches the approved catalog.
- Rebuild the index from a clean snapshot or trusted export.
- Run retrieval tests for known malicious prompts.
- Compare answer quality before and after remediation.
- Verify that deleted content is not still present in caches, replicas, or backups exposed to production.

**Example safe reindexing workflow**

```bash
# 1. Export trusted documents only
python export_trusted_docs.py --source-catalog approved_sources.yml --output clean_corpus/

# 2. Rebuild embeddings
python embed_corpus.py --input clean_corpus/ --model text-embedding-3-large

# 3. Load into a fresh index
python load_vector_index.py --index-name kb_clean_v2 --input embeddings/

# 4. Run retrieval tests
python run_rag_redteam.py --index-name kb_clean_v2 --test-suite adversarial_queries.yml
```

The key idea is to create a new trusted index rather than trying to “patch” a contaminated one in place. Fresh builds reduce the risk of hidden remnants and make rollback easier.

**Hardening controls**

Containment is temporary unless you fix the ingestion path. After recovery, add controls that reduce the chance of reinfection.

- **Source allowlisting** limits ingestion to approved domains, repositories, or tenants.
- **Content scanning** detects prompt injection patterns, secrets, and policy-violating text.
- **Human review** is useful for high-risk sources or externally supplied documents.
- **Document signing** helps verify provenance and detect tampering.
- **Least privilege** restricts who can upload, approve, or publish knowledge base content.
- **Versioned indexes** let you roll back to a known-good corpus quickly.
- **Telemetry and alerts** flag unusual retrieval patterns or sudden ranking shifts.

A strong control set usually combines preventive and detective measures. No single scanner can reliably catch every poisoned document, especially when the attacker uses subtle wording or hidden formatting.

**Practical configuration example**

A simple policy-driven ingestion gate can reject suspicious content before indexing.

```yaml
ingestion_policy:
  allowed_sources:
    - docs.internal.example
    - wiki.internal.example
  reject_patterns:
    - "ignore previous instructions"
    - "system prompt"
    - "developer message"
    - "reveal secrets"
  require_review_for:
    - external_upload
    - vendor_feed
    - user_generated_content
  quarantine_on_match: true
  create_audit_event: true
```

This kind of policy is not perfect, but it creates a measurable control point. You can tune it over time as you learn which patterns are noisy and which are high confidence.

**Incident response considerations**

Treat poisoned RAG as both a content integrity issue and an AI safety issue. The response team should include security, platform engineering, data owners, and whoever owns the assistant’s behavior and policy.

- **Security** handles containment, evidence, and root cause analysis.
- **Platform engineering** restores the index and connector pipeline.
- **Data owners** approve what content is authoritative.
- **AI governance** reviews whether the assistant exposed unsafe or regulated information.

Important questions to answer during the incident:
- Which source introduced the malicious content?
- Did the content affect only retrieval, or did it also influence logs, caches, or downstream exports?
- Were any secrets, credentials, or internal prompts exposed?
- Are other tenants, environments, or indexes also affected?
- Do you need to notify stakeholders or trigger compliance review?

**Common mistakes**

- **Deleting first, investigating later** can destroy evidence and hide the entry path.
- **Trusting internal content blindly** ignores insider risk and compromised accounts.
- **Reusing the same index** after partial cleanup can leave residual contamination.
- **Skipping retrieval tests** can let the poisoned behavior return unnoticed.
- **Ignoring metadata** misses a common place for ranking and trust manipulation.

**Key takeaways**

Poisoned RAG containment is about controlling the retrieval layer as much as the model layer. You stop ingestion, isolate the index, preserve evidence, rebuild from trusted sources, and validate behavior before returning to production. The safest long-term posture is a layered ingestion pipeline with allowlists, quarantine, review, and continuous monitoring.
