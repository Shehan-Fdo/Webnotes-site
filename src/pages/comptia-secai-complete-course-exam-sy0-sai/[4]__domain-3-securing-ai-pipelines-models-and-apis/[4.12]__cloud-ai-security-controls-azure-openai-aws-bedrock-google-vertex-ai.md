## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Cloud AI security control model**

Cloud AI services expose a similar security stack across providers: identity, network, data protection, logging, governance, and runtime guardrails. The exam often tests whether you can map a control to the right layer and explain why it reduces risk.

A useful mental model is:

- **Identity** controls who can call the service.
- **Network** controls where calls can originate.
- **Data** controls what content can be stored, encrypted, or exported.
- **Policy** controls what the organization allows.
- **Runtime guardrails** control what the model can say or do.
- **Audit** controls what you can prove after the fact.

For AI workloads, the biggest risks are prompt injection, data exfiltration, unauthorized model access, unsafe tool use, and weak governance around model promotion and deployment.

**Shared security objectives across providers**

All three platforms support the same core goals, but they implement them differently:

- Limit access to the model endpoint or API.
- Prevent public exposure of inference traffic.
- Protect prompts, completions, embeddings, and training data.
- Log access and content events for investigation.
- Restrict which models, regions, and tenants can be used.
- Control tool calling, agents, and external integrations.
- Govern model lifecycle changes with approval and rollback.

> <mark>Exam tip:</mark> If a scenario mentions “the model leaked sensitive data,” think beyond the model itself. The root cause is often weak identity, poor network isolation, missing DLP, or unsafe tool permissions.

**Azure OpenAI security controls**

Azure OpenAI is primarily governed through **Azure RBAC** (Role-Based Access Control) on the Azure resource, plus Azure networking and governance services. Authorization is resource-centric: you grant users, groups, managed identities, or service principals access to the Azure OpenAI resource and its operations.

Common control layers include:

- **Azure RBAC**
  - Use built-in roles where possible.
  - Prefer least privilege for inference-only users.
  - Separate admin, developer, and application identities.
- **Private networking**
  - Use **Private Link** and private endpoints to keep traffic off the public internet.
  - Disable public network access when the design allows it.
- **Encryption**
  - Azure-managed encryption is the default.
  - Use customer-managed keys (CMK) when policy requires tenant-controlled key rotation or stronger governance.
- **Governance**
  - Use **Azure Policy** to enforce private endpoints, approved regions, tagging, and diagnostic settings.
  - Use resource locks to reduce accidental deletion or modification.
  - Use Microsoft Purview and DLP-style controls where the organization needs data discovery, classification, or exfiltration monitoring.
- **Monitoring**
  - Send diagnostic logs to Log Analytics, Event Hub, or Storage.
  - Monitor Azure Activity Log for control-plane changes.
  - Monitor resource logs for authentication failures, request patterns, and quota issues.

A typical Azure OpenAI hardening pattern looks like this:

```bash
# Example: inspect role assignments on an Azure OpenAI resource
az role assignment list \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.CognitiveServices/accounts/<resource-name> \
  --output table
```

```bash
# Example: disable public network access on a Cognitive Services resource when supported
az cognitiveservices account update \
  --name <resource-name> \
  --resource-group <rg> \
  --public-network-access Disabled
```

```bash
# Example: create a private endpoint for the resource
az network private-endpoint create \
  --name pe-aoai \
  --resource-group <rg> \
  --vnet-name <vnet> \
  --subnet <subnet> \
  --private-connection-resource-id /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.CognitiveServices/accounts/<resource-name> \
  --group-id account \
  --connection-name aoai-privatelink
```

The exact private-link group ID can vary by resource type and service surface, so verify the current Azure documentation for the specific Azure OpenAI or Cognitive Services resource you deploy.

**Azure logging and auditing**

For exam prep, remember the difference between control-plane and data-plane visibility:

- **Azure Activity Log**
  - Records management actions such as resource creation, deletion, and policy changes.
  - Useful for detecting unauthorized configuration changes.
- **Diagnostic logs**
  - Capture service-specific events when enabled.
  - Useful for request volume, failures, and operational troubleshooting.
- **Microsoft Defender for Cloud**
  - Helps identify misconfigurations and risky exposure.
- **Microsoft Purview**
  - Helps classify and govern sensitive data that may flow into prompts or outputs.

Important events to monitor include:

- New role assignments on the AI resource.
- Public network access changes.
- Private endpoint creation or deletion.
- Diagnostic setting removal.
- Unusual request spikes or repeated authentication failures.
- Large output volumes that may indicate data leakage or abuse.

**Azure governance and data protection**

Azure-specific governance controls are often tested as “defense in depth” answers:

- **Azure Policy**
  - Enforce private endpoints.
  - Require diagnostic settings.
  - Restrict allowed regions.
  - Block public network access where feasible.
- **Resource locks**
  - Prevent accidental deletion or modification of critical AI resources.
- **Purview and DLP**
  - Classify sensitive data before it reaches prompts.
  - Detect regulated data in logs, documents, or outputs.
- **Key management**
  - Use Azure Key Vault for CMK scenarios.
  - Rotate keys on a defined schedule.
  - Separate key administrators from AI operators.

A practical Azure design for sensitive workloads is:

- Users authenticate through Entra ID.
- Applications use managed identities.
- The Azure OpenAI resource is private-only.
- Logs flow to Log Analytics.
- Policy blocks public exposure.
- Purview classifies source documents before retrieval-augmented generation (RAG).

**AWS Bedrock security controls**

Amazon Bedrock uses **AWS IAM** for authorization and integrates with VPC networking, CloudTrail, CloudWatch, KMS, and guardrails. The security model is service-oriented: you control who can invoke models, which models they can use, and how the surrounding environment is isolated.

Core Bedrock controls include:

- **IAM policies**
  - Grant only the actions needed for inference, agent use, or model management.
  - Be careful with resource scoping; some Bedrock actions still require `Resource: "*"`.
- **VPC endpoints**
  - Use interface VPC endpoints for private access where supported.
  - Private connectivity support can vary by region and feature.
- **Encryption**
  - Use AWS Key Management Service (KMS) for data at rest where supported.
  - Rotate keys according to policy.
- **Logging**
  - Use CloudTrail for API activity.
  - Use CloudWatch for operational metrics and alarms.
  - Enable service-specific logging where available for prompts, outputs, or guardrail events according to policy and privacy rules.
- **Guardrails**
  - Apply content filters, topic restrictions, and sensitive-data controls.
  - Use guardrails at runtime to inspect prompts and outputs.
- **Model governance**
  - Control which foundation models are allowed.
  - Restrict model access by account, region, and IAM principal.
  - Track model version changes and approval status.

A minimal IAM pattern for Bedrock invocation might look like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "InvokeSpecificModel",
      "Effect": "Allow",
      "Action": [
        "bedrock:InvokeModel",
        "bedrock:InvokeModelWithResponseStream"
      ],
      "Resource": "*"
    }
  ]
}
```

In practice, some Bedrock permissions can be narrowed further, but not every action supports resource-level scoping. For exam questions, choose the least-privilege option that is actually supported by the service and action.

**Bedrock guardrails and runtime enforcement**

Guardrails are not just configuration objects; they are runtime controls that inspect content before or after model generation depending on the feature path. They help reduce unsafe outputs, policy violations, and sensitive-data leakage.

Use guardrails to:

- Block disallowed topics.
- Detect and redact sensitive information.
- Reduce harmful, toxic, or policy-violating responses.
- Enforce organization-specific acceptable-use rules.

Operationally, think of guardrails as sitting between the user prompt and the model, and again between the model output and the caller when the service supports output inspection. This matters because prompt injection can arrive through user input, retrieved documents, or tool outputs.

A conceptual Bedrock flow is:

1. User submits a prompt.
2. Guardrail evaluates the prompt.
3. Model generates a response.
4. Guardrail evaluates the response.
5. The application receives the approved or modified output.

**Bedrock private networking and logging**

For private connectivity, use VPC interface endpoints where the service and region support them. In exam scenarios, the key idea is that private access reduces exposure to the public internet and keeps traffic inside AWS networking boundaries.

Monitor:

- **CloudTrail**
  - `InvokeModel`, model access changes, guardrail configuration changes, agent actions, and IAM changes.
- **CloudWatch**
  - Latency, throttling, errors, and service health.
- **Application logs**
  - Prompt templates, retrieval sources, tool calls, and moderation outcomes, with sensitive content minimized or redacted.

A practical Bedrock logging strategy is to log metadata rather than raw prompts when possible:

- Request ID
- Principal ARN
- Model ID
- Guardrail decision
- Tool invocation name
- Timestamp
- Outcome status

This gives you forensic value without overexposing sensitive content.

**Bedrock agents, tools, and model lifecycle**

Bedrock agents and tool use introduce a second trust boundary: the model can trigger actions outside the model itself. That means you must secure:

- **Tool permissions**
  - Give the agent only the IAM permissions required for the downstream action.
- **Action schemas**
  - Validate inputs before execution.
- **Retrieval sources**
  - Restrict which knowledge bases or documents the agent can read.
- **Human approval**
  - Require approval for high-risk actions such as payments, deletions, or external messages.

For model lifecycle security:

- Track approved model IDs and versions.
- Use change control for prompt templates and system instructions.
- Test rollback paths before promoting a new model version.
- Separate development, staging, and production accounts or environments.

> <mark>Exam tip:</mark> If a Bedrock agent can send email, delete records, or call APIs, the real risk is not only model output quality. It is unauthorized action through over-permissioned tools.

**Google Vertex AI security controls**

Vertex AI security is built on **Google Cloud IAM**, service accounts, VPC Service Controls, Cloud Audit Logs, Cloud KMS, and organization policies. The main exam idea is that access is controlled by Google Cloud identities and project/resource boundaries, while network and perimeter controls reduce data exfiltration risk.

Key controls include:

- **IAM**
  - Grant users and service accounts only the permissions needed for model deployment, endpoint invocation, or model management.
  - Use separate service accounts for training, deployment, and inference.
- **Service accounts**
  - Deployed models and pipelines should run with dedicated identities.
  - Avoid using broad project-wide default service accounts.
- **VPC Service Controls**
  - Create a service perimeter around sensitive projects and services.
  - Reduce the risk of data exfiltration from managed AI services.
- **Private networking**
  - Use private access patterns where supported.
  - Keep inference traffic inside controlled network paths.
- **Encryption**
  - Google-managed encryption is default.
  - Use Cloud KMS for customer-managed encryption key (CMEK) requirements.
- **Logging**
  - Use Cloud Audit Logs for admin activity and data access where applicable.
  - Use model and endpoint logs for operational visibility.
- **Organization policy**
  - Restrict allowed regions, external IP usage, and risky service configurations.
  - Enforce constraints that reduce shadow AI deployments.

A practical IAM example for endpoint invocation is:

```bash
gcloud projects add-iam-policy-binding <project-id> \
  --member="serviceAccount:<sa-name>@<project-id>.iam.gserviceaccount.com" \
  --role="roles/aiplatform.user"
```

Role selection can vary by task and product surface, so verify the current role mapping for endpoint prediction, model deployment, and admin operations. For exam purposes, the important concept is that Vertex AI uses IAM roles tied to the project and service account identity.

**Vertex AI endpoint deployment and access**

Vertex AI endpoints are a common security boundary for online prediction. You typically deploy a model to an endpoint, then authorize callers to invoke that endpoint.

A deployment workflow often includes:

1. Upload or register the model.
2. Create or select an endpoint.
3. Deploy the model to the endpoint.
4. Assign IAM permissions to the calling identity.
5. Monitor traffic, latency, and errors.

A representative deployment command may require several flags depending on the CLI version and model type:

```bash
gcloud ai endpoints deploy-model <ENDPOINT_ID> \
  --region=<REGION> \
  --model=<MODEL_ID> \
  --display-name=<DEPLOYED_MODEL_NAME> \
  --traffic-split=0=<TRAFFIC_PERCENT>
```

Because `gcloud` syntax changes across versions and model types, treat this as a pattern rather than a copy-paste guarantee. The exam is more likely to test the security implications: endpoint access is controlled by IAM, and the deployed model should run under a dedicated service account.

**Vertex AI logging and auditing**

Use the following logs and signals:

- **Cloud Audit Logs**
  - Admin activity: endpoint creation, IAM changes, model registration, policy changes.
  - Data access: prediction requests where enabled and supported by the service.
- **Cloud Logging**
  - Application and platform logs for request IDs, errors, and latency.
- **Cloud Monitoring**
  - Endpoint health, throughput, error rates, and saturation.
- **Security Command Center**
  - Findings for misconfigurations, public exposure, and policy violations.

Important events to watch:

- New endpoint deployments.
- Service account changes on deployed models.
- IAM policy changes on the project or endpoint.
- VPC Service Controls perimeter changes.
- Unexpected cross-project access.
- Large or unusual prediction volumes.

**Vertex AI governance and organization controls**

Google Cloud governance is often tested through organization-level restrictions:

- **Organization policies**
  - Restrict resource locations.
  - Limit external IPs.
  - Prevent risky service account usage.
  - Constrain who can create or expose endpoints.
- **VPC Service Controls**
  - Reduce exfiltration from managed services.
  - Protect sensitive datasets and model artifacts.
- **CMEK**
  - Use Cloud KMS when regulatory or contractual requirements demand customer-managed keys.
- **Model registry governance**
  - Approve model promotion from development to production.
  - Track versions and rollback points.
  - Require review before a model becomes endpoint-ready.

**Model lifecycle security across all three providers**

Model lifecycle security is a frequent exam theme because the model itself is not static. You must secure the path from training or import to deployment and retirement.

Key controls:

- **Versioning**
  - Track model versions and prompt template versions separately.
- **Approval**
  - Require review before production promotion.
- **Rollback**
  - Keep a known-good version ready for rapid rollback.
- **Provenance**
  - Record where the model came from, who approved it, and what data it used.
- **Retirement**
  - Remove unused models, endpoints, and keys to reduce attack surface.

A secure lifecycle process usually includes:

1. Train or import the model in a nonproduction environment.
2. Scan data and artifacts for sensitive content.
3. Review model behavior, safety, and bias.
4. Approve promotion through change control.
5. Deploy with least privilege and private networking.
6. Monitor drift, abuse, and policy violations.
7. Roll back or retire when risk increases.

**Agent and tool security by provider**

AI agents can call tools, APIs, databases, and external services. That expands the attack surface beyond inference.

- **Azure OpenAI**
  - Secure the application layer that orchestrates tool use.
  - Use managed identities for downstream Azure services.
  - Restrict access to storage, search, and function apps used by the agent.
- **AWS Bedrock**
  - Secure agent execution roles and action groups.
  - Validate tool inputs and outputs.
  - Restrict access to Lambda, API Gateway, databases, and knowledge bases.
- **Vertex AI**
  - Use dedicated service accounts for extensions, pipelines, and deployed models.
  - Restrict access to Cloud Functions, Cloud Run, BigQuery, and storage sources.
  - Apply organization policies to reduce uncontrolled integrations.

The exam often frames this as prompt injection plus over-permissioned tools. The remediation is least privilege, input validation, allowlists, and human approval for sensitive actions.

**Encryption and key management**

Encryption is a baseline control, but exam questions often ask when to use provider-managed keys versus customer-managed keys.

- **Provider-managed encryption**
  - Suitable for many standard workloads.
  - Reduces operational overhead.
  - Often enough when compliance requirements are moderate.
- **Customer-managed keys**
  - Use when policy requires customer control over key rotation, revocation, or separation of duties.
  - Useful for regulated workloads and strict governance.
- **Key rotation**
  - Rotate on schedule and after incidents.
  - Ensure applications can tolerate key updates.
- **Access separation**
  - Separate key administrators from AI operators.

Provider mapping:

- Azure: **Key Vault** and CMK.
- AWS: **KMS** and customer-managed keys.
- Google Cloud: **Cloud KMS** and CMEK.

**Scenario comparison for exam prep**

A common exam pattern is to describe the same business need across all three clouds. Use the control that best matches the platform.

| Scenario | Azure OpenAI | AWS Bedrock | Google Vertex AI |
|---|---|---|---|
| Keep inference traffic off the public internet | Private Link, disable public access | VPC interface endpoints where supported | Private access patterns and perimeter controls |
| Restrict who can call the model | Azure RBAC | IAM policy | Google Cloud IAM |
| Detect unsafe or disallowed content | Azure-side app controls plus policy and DLP | Bedrock guardrails | Application controls plus logging and policy |
| Audit admin and access activity | Activity Log and diagnostic logs | CloudTrail and CloudWatch | Cloud Audit Logs and Cloud Logging |
| Control exfiltration from sensitive data | Policy, private endpoints, Purview, DLP, RBAC | IAM, guardrails, VPC endpoints, logging | VPC Service Controls, IAM, org policy, CMEK |
| Secure agent/tool actions | Managed identities and least privilege | Agent execution roles and action groups | Service accounts and restricted integrations |

**How to choose the right control**

Use this decision logic in scenario questions:

- If the issue is “who can use the service,” choose **IAM/RBAC**.
- If the issue is “traffic must not traverse the internet,” choose **private networking**.
- If the issue is “sensitive content must not be generated or returned,” choose **guardrails** or content policy controls.
- If the issue is “prove what happened,” choose **audit logging**.
- If the issue is “prevent accidental exposure or misconfiguration,” choose **policy enforcement** and **resource governance**.
- If the issue is “the model can take actions,” secure the **tooling and agent permissions**, not just the model endpoint.

**Common misconfigurations and remediation**

- Public endpoint left enabled
  - Remediate by disabling public access and using private connectivity.
- Overbroad IAM or RBAC
  - Remediate by narrowing roles, separating duties, and using dedicated service accounts.
- Missing audit logs
  - Remediate by enabling diagnostic logs, CloudTrail, or Cloud Audit Logs.
- Unrestricted tool access
  - Remediate by allowlisting actions and requiring approval for high-risk operations.
- No model version control
  - Remediate by introducing approval gates, version tags, and rollback plans.
- Sensitive data in prompts or logs
  - Remediate by applying DLP, redaction, and log minimization.
- No perimeter or network isolation
  - Remediate by using private endpoints, VPC endpoints, or VPC Service Controls where supported.

**Quick memory anchors**

- **Azure OpenAI**: Azure RBAC, Private Link, Policy, Purview, Log Analytics.
- **AWS Bedrock**: IAM, VPC endpoints, CloudTrail, CloudWatch, guardrails, KMS.
- **Vertex AI**: IAM, service accounts, VPC Service Controls, Audit Logs, Cloud KMS, org policy.

**Practical exam framing**

When you see a question about cloud AI security, ask:

- Is the risk identity, network, data, or runtime behavior?
- Is the control provider-specific or generic?
- Does the scenario involve inference only, or also agents and tools?
- Is the goal prevention, detection, or recovery?
- Is the workload regulated enough to require CMK/CMEK and stricter governance?

If you answer those five questions, you can usually eliminate distractors and choose the best control.
