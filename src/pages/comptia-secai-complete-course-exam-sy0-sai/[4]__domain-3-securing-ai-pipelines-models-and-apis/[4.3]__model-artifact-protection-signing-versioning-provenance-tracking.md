## **Course:** CompTIA SecAI+ Complete Course (Exam SY0-SAI+)

**Overview**

Model artifacts are the files and metadata that make an AI system usable, reproducible, and deployable. They include trained weights, tokenizer files, configuration, preprocessing logic, evaluation reports, and sometimes container images or dependency manifests. Protecting these artifacts matters because a compromised model can produce unsafe outputs, leak data, or behave differently from what was tested.

A secure model supply chain focuses on four related controls:

- **Protection** keeps artifacts confidential and intact.
- **Signing** proves who produced the artifact and whether it changed.
- **Versioning** tracks which artifact is current and how it differs from earlier builds.
- **Provenance tracking** records where the artifact came from, how it was built, and what inputs influenced it.

These controls help you answer practical questions such as:

- Is this the exact model that passed validation?
- Was the artifact altered after release?
- Which dataset and code produced this version?
- Can you roll back to a known-good release quickly?

**Model artifacts and what you protect**

In real deployments, a model is rarely a single file. You usually protect a bundle of related assets so the runtime behaves consistently.

- **Weights or checkpoints** store learned parameters.
- **Tokenizer files** define how text is split into tokens.
- **Model config** captures architecture and inference settings.
- **Preprocessing code** transforms input before inference.
- **Postprocessing code** formats or filters output.
- **Dependency manifests** pin library versions used at build time.
- **Evaluation artifacts** document accuracy, bias, safety, and robustness results.
- **Container images** package the runtime environment and model files together.

If any one of these changes unexpectedly, the deployed system may no longer match the tested system. That is why artifact protection is broader than file encryption alone.

**Threats to model artifacts**

Model artifacts face the same supply-chain risks as software, plus AI-specific risks.

- **Tampering** changes weights, prompts, or preprocessing logic to alter behavior.
- **Model substitution** replaces a trusted model with a malicious or lower-quality one.
- **Rollback attacks** force deployment of an older vulnerable version.
- **Poisoned artifacts** embed backdoors or unsafe behavior during training or packaging.
- **Unauthorized disclosure** exposes proprietary weights, prompts, or training logic.
- **Dependency drift** changes runtime libraries and causes silent behavior changes.
- **Metadata manipulation** hides the true source, training data, or evaluation status.

> If you cannot verify artifact integrity and origin, you cannot reliably trust the model’s behavior in production.

**Artifact protection controls**

Protection starts with access control and secure storage, then extends to integrity and lifecycle management.

- Store artifacts in a **trusted repository** with authentication and authorization.
- Restrict write access to build systems and release pipelines only.
- Encrypt artifacts at rest and in transit.
- Use immutable or append-only storage for released versions when possible.
- Separate development, staging, and production repositories.
- Scan artifacts and associated containers for malware, secrets, and known vulnerabilities.
- Log all access, uploads, downloads, and promotion events.

Common storage patterns include:

- **Object storage** for large checkpoints and datasets.
- **Artifact repositories** for versioned releases and metadata.
- **Model registries** for lifecycle states such as `staging`, `approved`, and `production`.
- **Container registries** for packaged inference services.

A simple repository layout might look like this:

```text
models/
  fraud-detector/
    1.0.0/
      model.safetensors
      tokenizer.json
      config.json
      sha256sums.txt
      provenance.json
      signature.sig
    1.1.0/
      model.safetensors
      tokenizer.json
      config.json
      sha256sums.txt
      provenance.json
      signature.sig
```

**Integrity verification with hashes**

A cryptographic hash detects accidental or malicious changes to a file. You compute the hash at build time and verify it before deployment or loading.

Common algorithms include:

- **SHA-256** for general integrity checks
- **SHA-384** or **SHA-512** when policy requires stronger digests
- Avoid weak hashes such as MD5 or SHA-1 for security decisions

Example:

```bash
sha256sum model.safetensors tokenizer.json config.json > sha256sums.txt
sha256sum -c sha256sums.txt
```

Hashes are useful, but they do not prove who created the artifact. Anyone can recompute a hash after modifying a file. For origin assurance, you need signing.

**Signing model artifacts**

Signing adds authenticity to integrity. A signer uses a private key to create a signature over the artifact or its digest. Verifiers use the corresponding public key to confirm the artifact came from the expected source and was not altered.

Typical signing approaches:

- **Detached signatures** store the signature separately from the artifact.
- **Embedded signatures** place the signature inside a package or container metadata.
- **Manifest signing** signs a list of files and hashes rather than each file individually.
- **Certificate-based signing** uses a public key infrastructure (PKI) chain to identify the signer.

Example with GPG-style signing of a digest file:

```bash
gpg --armor --detach-sign sha256sums.txt
gpg --verify sha256sums.txt.asc sha256sums.txt
```

Example with OpenSSL for a simple RSA signature workflow:

```bash
openssl dgst -sha256 -sign private.key -out model.sig model.safetensors
openssl dgst -sha256 -verify public.key -signature model.sig model.safetensors
```

In practice, many organizations sign:

- the model package
- the container image
- the provenance attestation
- the release manifest

That layered approach makes it harder for an attacker to swap one component without detection.

**What signatures do and do not prove**

A signature proves that a trusted key signed the artifact. It does not automatically prove the artifact is safe, unbiased, or correct.

- A valid signature means the artifact is authentic and intact relative to the signed content.
- A valid signature does not mean the model was trained on approved data.
- A valid signature does not mean the model passed all safety tests.
- A valid signature does not prevent a compromised signer from approving a bad artifact.

This is why signing must be combined with review, policy checks, and provenance validation.

**Versioning and release control**

Versioning lets you identify exactly which artifact is deployed and how it relates to previous releases. It supports rollback, auditability, and reproducibility.

Use versioning for:

- model weights
- preprocessing pipelines
- prompt templates
- evaluation baselines
- container images
- dependency lockfiles

A practical versioning scheme often combines:

- **Semantic versioning** for human-readable releases, such as `1.4.2`
- **Build metadata** for CI/CD (continuous integration/continuous delivery) traceability
- **Immutable digests** for exact artifact identity, such as a SHA-256 hash or container digest

Example:

```text
fraud-detector:1.4.2
fraud-detector@sha256:9f2c...a81d
```

Use semantic versions when you need to communicate compatibility:

- `MAJOR` changes may break interfaces or behavior.
- `MINOR` changes add features or improve performance without expected breaking changes.
- `PATCH` changes fix defects or update metadata.

Use immutable digests when you need exact reproducibility. Tags can be moved; digests cannot.

> Do not rely on mutable tags alone for production deployment. Always pin to an immutable digest or equivalent content address.

**Model registry lifecycle states**

A model registry helps you control promotion through the release pipeline. Typical states include:

- **Draft** for work in progress
- **Staging** for validation and integration testing
- **Approved** for security and governance review
- **Production** for active deployment
- **Archived** for retired versions kept for audit or rollback

A promotion workflow might look like this:

1. Train and package the model in a controlled build environment.
2. Generate hashes and provenance metadata.
3. Sign the artifact and manifest.
4. Run validation, bias, and safety checks.
5. Promote the artifact from staging to approved.
6. Deploy only the approved, signed version to production.
7. Archive older versions with retention controls.

**Provenance tracking**

Provenance is the record of an artifact’s origin and transformation history. It answers questions such as:

- Which code commit produced this model?
- Which dataset versions were used?
- Which training environment and dependencies were present?
- Who approved the release?
- Which tests ran before deployment?

Provenance is especially important for AI because model behavior depends on more than source code. Data, preprocessing, hyperparameters, and environment all affect the final artifact.

A strong provenance record usually includes:

- source repository and commit hash
- build pipeline identifier
- training job ID
- dataset identifiers and checksums
- dependency versions
- hyperparameters
- evaluation results
- signer identity
- timestamp and environment details

Example provenance document:

```json
{
  "model_name": "fraud-detector",
  "version": "1.4.2",
  "source_commit": "a1b2c3d4",
  "training_data": [
    {
      "dataset": "transactions-2026q1",
      "sha256": "7c9f..."
    }
  ],
  "framework": "PyTorch 2.6.0",
  "build_system": "ci-runner-17",
  "signer": "ml-release-key-01",
  "evaluation": {
    "accuracy": 0.982,
    "f1": 0.941
  }
}
```

Provenance can be stored as JSON, signed metadata, or an attestation attached to the artifact.

**Attestations and supply-chain metadata**

An attestation is a signed statement about how an artifact was built or verified. It can include build steps, test results, policy checks, and environment details.

Useful attestation content includes:

- build timestamp
- builder identity
- source repository
- dependency lockfile hash
- test suite results
- vulnerability scan results
- approval status

Attestations help you enforce policy such as:

- only artifacts built by the CI pipeline may be deployed
- only artifacts with passing safety tests may enter production
- only artifacts with approved provenance may be consumed by downstream systems

This is common in modern software supply-chain frameworks and applies directly to AI model delivery.

**Secure build and release workflow**

A secure workflow reduces the chance that a malicious or accidental change reaches production.

1. Pull source code and data from trusted repositories.
2. Build the model in an isolated, reproducible environment.
3. Record exact dependency versions and environment details.
4. Generate hashes for all release files.
5. Create provenance metadata and attestations.
6. Sign the manifest or artifact bundle with a protected private key.
7. Store the release in an immutable or controlled repository.
8. Verify signatures and hashes before promotion and deployment.
9. Monitor runtime behavior after release for drift or anomalies.

Example CI/CD-style verification:

```bash
sha256sum -c sha256sums.txt
gpg --verify release-manifest.sig release-manifest.json
cosign verify-blob --key cosign.pub model.safetensors
```

**Container images and model packaging**

Many AI systems deploy models inside containers. In that case, you must protect both the model files and the image that carries them.

Key controls:

- Pin base images by digest.
- Scan images for vulnerabilities and secrets.
- Sign images after build.
- Store image provenance alongside model provenance.
- Rebuild images when dependencies change.

Example Dockerfile pattern:

```dockerfile
FROM python:3.12-slim@sha256:exampledigest
WORKDIR /app
COPY model.safetensors tokenizer.json config.json /app/model/
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY serve.py .
CMD ["python", "serve.py"]
```

Example image signing and verification with `cosign`:

```bash
cosign sign --key cosign.key registry.example.com/ml/fraud-detector:1.4.2
cosign verify --key cosign.pub registry.example.com/ml/fraud-detector:1.4.2
```

**Access control and key management**

Signing is only as strong as the protection around the private key.

- Store signing keys in a hardware security module (HSM) or managed key service when possible.
- Limit key access to release automation and a small set of authorized operators.
- Rotate keys on a defined schedule and after suspected compromise.
- Separate signing keys by environment or product line.
- Use short-lived credentials for build jobs.
- Audit every signing operation.

If a signing key is compromised, attackers can produce artifacts that appear trusted. In that case, you must revoke trust, rotate keys, and reissue signed releases.

**Verification points in the pipeline**

You should verify artifacts at multiple stages, not just once.

- **Build time** to confirm the package matches the expected source.
- **Promotion time** to ensure the artifact entering the registry is unchanged.
- **Deployment time** to block tampered or unsigned releases.
- **Runtime startup** to validate the loaded artifact before serving traffic.
- **Periodic audit** to detect repository drift or unauthorized replacement.

This layered verification reduces the chance that a single missed check leads to compromise.

**Practical deployment example**

A common pattern is to deploy only a signed, hashed, and provenance-attested model from a registry.

```bash
# Verify the release bundle
sha256sum -c sha256sums.txt
gpg --verify release-manifest.sig release-manifest.json

# Pull the exact image digest
docker pull registry.example.com/ml/fraud-detector@sha256:9f2c...a81d

# Start the service
docker run --rm -p 8080:8080 registry.example.com/ml/fraud-detector@sha256:9f2c...a81d
```

In a Kubernetes environment, you can enforce image signature policy with admission controls and require that only approved digests are admitted to the cluster.

**Monitoring and drift detection**

Even a protected artifact can become risky if the runtime environment changes. Monitor for:

- unexpected model output shifts
- changes in input distribution
- dependency updates that alter inference behavior
- unauthorized file changes in mounted volumes
- registry drift between declared and deployed versions

Useful operational signals include:

- model version in logs
- artifact digest at startup
- signature verification status
- provenance ID in telemetry
- alerting on unapproved promotions

**Common mistakes**

- Using mutable tags without digest pinning
- Signing after deployment instead of before release
- Storing signing keys on the same host as the build job
- Treating a valid signature as proof of model quality
- Failing to version preprocessing code and tokenizer files
- Allowing manual uploads to production registries
- Not retaining provenance records for audit and rollback

**Exam focus summary**

For exam purposes, remember the relationship between the controls:

- **Hashing** detects change.
- **Signing** proves origin and integrity.
- **Versioning** identifies the exact release.
- **Provenance** explains how the artifact was produced.
- **Attestation** provides signed evidence about the build or validation process.

A secure AI release process uses all of them together. That combination supports trust, traceability, rollback, and compliance across the model lifecycle.
