## **Course:** CompTIA Security+ (Exam SY0-701)

**Shared responsibility model overview**

The shared responsibility model defines how security duties split between the cloud service provider (CSP) and the customer. In practice, the exact boundary changes with the service model you use.

Cloud providers secure the underlying cloud infrastructure, while you secure what you place, configure, and control in that environment. This division helps reduce operational burden, but it does not remove your responsibility for security.

- **CSP** responsibilities typically include the physical environment and core platform layers.
- **Customer** responsibilities typically include data, applications, configurations, and access controls.
- Responsibility changes based on the service model and deployment context.

**How responsibility shifts by service model**

The service model determines how much management you keep versus how much the provider handles. As you move from infrastructure-heavy services to fully managed services, your operational responsibility usually decreases.

- **IaaS**: You manage virtual machines, operating systems, and applications.
- **PaaS**: You deploy applications without managing virtual machines or operating systems.
- **SaaS**: The provider handles more of the stack, so your focus shifts toward data use, access, and configuration.
- **On-premises**: You retain the broadest responsibility because you operate the environment yourself.

<details>
<summary>Quick comparison</summary>

| Model | Provider handles more of | You handle more of |
|---|---|---|
| IaaS | Physical infrastructure, host layers | VMs, OS, applications |
| PaaS | Platform services | Application deployment and data-related controls |
| SaaS | Application delivery and underlying service | User access, data use, and configuration |

</details>

**What you still own in the cloud**

Even when the provider manages the infrastructure, you remain responsible for protecting what you deploy and how you use it. Misunderstanding this boundary creates gaps, especially around configuration and access.

- **Guest OS** management remains with you in IaaS, including updates and security patches.
- **Application software** remains your responsibility in IaaS and often in PaaS.
- **Security groups and firewall settings** are customer-managed in provider environments that expose them.
- **Data protection** stays with you across cloud service types.
- **Access control** remains a customer duty, including least-privilege decisions.

> <mark>Do not assume the cloud provider automatically secures your data, applications, or permissions.</mark>

**Practical security implications**

The shared responsibility model is not just a contract concept; it affects daily security work. You need to know which controls belong to you before you deploy or migrate a workload.

- Document ownership for each cloud service you use.
- Review configuration settings before putting services into production.
- Validate access controls instead of assuming defaults are secure.
- Monitor for misconfigurations and configuration drift.
- Match security controls to the service model you selected.

**Exam focus**

For Security+ purposes, focus on the idea that cloud security is shared, but not evenly. The provider secures the cloud, and you secure what you put in the cloud.

- **IaaS** gives you the most control and the most responsibility.
- **PaaS** reduces infrastructure management but still requires application and data oversight.
- **SaaS** shifts the most responsibility to the provider, but you still manage users, data, and settings.
- The exact split depends on the provider, service type, and deployment model.

**Common mistake to avoid**

A frequent assumption gap is believing that cloud services automatically enforce every security control for you. That is risky because responsibility boundaries vary and can leave exposed gaps if you do not verify them.

- Check who manages patching before you deploy.
- Check who controls network rules before you expose a service.
- Check who owns encryption, access, and configuration before you store sensitive data.
