## **Course:** CompTIA Security+ (Exam SY0-701)

**Serverless security overview**

Serverless computing shifts more operational responsibility to the cloud provider, but it does not remove security work. You still need to think about how functions run, how they communicate, and how data is protected across the application lifecycle.

A useful way to organize serverless security is by layers. Recent research groups protections into runtime, network, function, orchestration, and data concerns. That model helps you map threats to the right control area instead of treating serverless as a single security problem.

- **Runtime** security focuses on the execution environment for functions.
- **Network** security covers traffic between services and external endpoints.
- **Function** security addresses the code and permissions tied to each function.
- **Orchestration** security covers how services are coordinated.
- **Data** security protects information across storage and lifecycle stages.

Serverless security still has gaps that matter in practice. Two recurring concerns are limited function-level observability and weak data lifecycle protection. Those gaps make monitoring and data handling especially important when you design controls.

**Serverless security priorities**

You should treat serverless security as a combination of visibility, access control, and data protection. The main goal is to reduce exposure without assuming the provider handles every risk.

- Monitor function activity so you can detect unusual behavior.
- Restrict permissions to the minimum needed for each function.
- Protect data throughout its lifecycle, not only at rest.
- Review how functions interact with other services and APIs.
- Validate security controls across the full serverless workflow.

**Virtualization security overview**

Virtualization improves efficiency and scalability, and it can strengthen isolation between virtual machines (VMs) and physical hardware. It also supports availability and can work with security tools that fit virtual environments. At the same time, virtualization creates its own attack surface and needs active management.

One major risk is **VM sprawl**, which happens when virtual machines grow without control. Unused or forgotten VMs can become weak points, especially if they still contain sensitive information. Periodic review and decommissioning help reduce that risk.

- **VM sprawl** increases the chance that stale systems remain exposed.
- Compromised VMs may retain sensitive data if you do not retire them properly.
- Virtualization security risks resemble other infrastructure risks, so you still need standard controls.

**Common virtualization risks**

Virtualization is widely adopted, which makes it attractive to attackers. The security challenge is not that virtualization is inherently unsafe, but that scale and complexity can hide problems.

- Uncontrolled VM growth can leave systems unmanaged.
- Compromised virtual machines can spread risk across the environment.
- Sensitive information inside old VMs can persist longer than expected.
- Security teams need policies for review, cleanup, and lifecycle management.

**Practical control focus**

For both serverless and virtualization, the best approach is to manage lifecycle, visibility, and access. The technologies differ, but the security pattern is similar: know what is running, limit what it can do, and remove what you no longer need.

1. Identify the functions, VMs, and services in scope.
2. Apply least privilege to identities, permissions, and service interactions.
3. Monitor activity for abnormal execution or unexpected changes.
4. Review unused resources and decommission them on a schedule.
5. Protect data across storage, transfer, and retirement stages.

**Exam focus reminders**

Security+ questions in this area usually test whether you can match the technology to the risk and the control. Expect scenario-based prompts about visibility, lifecycle management, and isolation.

- Serverless questions often center on function behavior, orchestration, and data protection.
- Virtualization questions often center on VM sprawl, isolation, and cleanup.
- The safest answer usually emphasizes monitoring, least privilege, and lifecycle control.

> If a scenario mentions hidden or forgotten resources, think about sprawl and decommissioning.
> If a scenario mentions limited visibility in serverless, think about observability and logging.
