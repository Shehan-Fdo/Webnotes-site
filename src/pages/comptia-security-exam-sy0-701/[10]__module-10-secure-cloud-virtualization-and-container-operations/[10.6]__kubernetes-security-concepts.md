## **Course:** CompTIA Security+ (Exam SY0-701)

**Kubernetes security overview**

Kubernetes security centers on controlling access to the Kubernetes API and cluster resources. A key theme is **least privilege**: you grant only the permissions a user or service needs, which reduces the chance of misuse or breach.

Kubernetes environments are dynamic, so access controls must be reviewed and adjusted over time. Misconfigured permissions can create serious exposure, so you should treat authorization as an active security control rather than a one-time setup.

- **RBAC** is the main access-control model highlighted in the evidence.
- Security practices should evolve as cluster roles and workloads change.
- Continuous monitoring helps you catch permission drift and risky access patterns.

**Role-based access control**

Role-based access control, or **RBAC**, regulates access based on roles and permissions. In Kubernetes, RBAC uses the `rbac.authorization.k8s.io` API group to make authorization decisions through the Kubernetes API.

You can enable RBAC at the API server level. The evidence shows two supported approaches:

1. Use the `--authorization-config` flag with a file that includes the RBAC authorizer.
2. Use the `--authorization-mode` flag and include `RBAC` in the comma-separated list.

```bash
kube-apiserver --authorization-mode=...,RBAC --other-options --more-options
```

> Caution: RBAC objects are designed to impose access restrictions. When you change them, you can unintentionally block access if you do not verify the impact first.

**Core RBAC objects**

The RBAC API defines four Kubernetes object types. These objects are managed like other Kubernetes resources, including with `kubectl`.

| Object | Purpose |
|---|---|
| `Role` | Defines permissions within a namespace |
| `ClusterRole` | Defines permissions at the cluster scope |
| `RoleBinding` | Grants a `Role` to a subject |
| `ClusterRoleBinding` | Grants a `ClusterRole` to a subject |

These objects work together to map permissions to users, groups, or service accounts. The practical security goal is to keep permissions narrow and review them regularly.

**Security best practices**

RBAC is only effective when you manage it carefully. The evidence emphasizes that security incidents can result from misconfigured permissions, so you should validate access rules and keep them aligned with current needs.

- Review why each permission exists before you grant it.
- Automate policy enforcement where possible.
- Monitor access controls continuously for changes and anomalies.
- Prefer the smallest permission set that still supports the workload.

**Exam focus points**

For Security+ purposes, focus on the security outcome of Kubernetes controls rather than deep administration detail. You should be able to explain how RBAC supports access control and why least privilege matters in containerized environments.

- Kubernetes security often means protecting the API and the resources it manages.
- RBAC is a foundational control for authorization.
- Good security practice includes ongoing review, not just initial configuration.
