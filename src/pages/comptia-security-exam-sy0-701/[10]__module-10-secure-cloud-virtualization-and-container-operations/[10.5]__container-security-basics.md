## **Course:** CompTIA Security+ (Exam SY0-701)

**Container security overview**

Container security focuses on reducing risk across the container lifecycle. A container image packages an application with its libraries, framework components, and dependencies, so the image itself becomes a major trust boundary. You want to know that the image contains only the software you expect and that it does not introduce malware or known vulnerabilities.

Container security also depends on the surrounding ecosystem:

- **Images** should come from trusted sources and be kept clean of unexpected components.
- **Registries** should be controlled so you can manage who publishes and pulls images.
- **Runtime settings** should limit what a container can do if it is compromised.
- **Host security** still matters because containers share the host kernel.

> Keep both the host and container platform updated. A vulnerable host kernel can expose containers to escape or privilege-escalation risk.

**Image and registry security**

Images are often stored in repositories that belong to either a public registry or a private registry. A private registry gives you more control over image distribution and reduces reliance on public sources. This is especially useful when you need to manage approved images and limit exposure to untrusted content.

A secure image strategy starts with provenance and integrity:

- Use images from sources you expect.
- Prefer base images and libraries that are maintained and free from critical vulnerabilities.
- Review what is included in the image, not just the application code you wrote.
- Treat image and artifact integrity as part of the CI/CD (continuous integration and continuous delivery) pipeline.

**Practical hardening priorities**

Container security best practices commonly include limiting privileges, implementing access controls, securing the runtime, and segregating container networks. These controls reduce the blast radius if one container is compromised.

Focus on the following priorities:

- **Least privilege**: give containers only the permissions they need.
- **Access control**: restrict who can deploy, modify, or pull images.
- **Runtime protection**: reduce the impact of misconfiguration or exploitation.
- **Network separation**: isolate workloads that do not need direct communication.

**Host and daemon protection**

Containers do not replace host security. Because containers share the host kernel, weaknesses in the host can affect every container running on it. Keeping the host and container engine current is a practical baseline defense against known escape vulnerabilities.

One especially important rule is to avoid exposing the Docker daemon socket. If an attacker can reach the daemon socket, they may gain powerful control over the container environment. Protecting that interface helps preserve administrative boundaries.

**Security checklist**

1. Use a private registry when you need tighter control over image distribution.
2. Verify that images contain only expected libraries, base images, and custom components.
3. Limit container privileges to the minimum required for the workload.
4. Apply access controls to image publishing, deployment, and runtime administration.
5. Keep the host kernel and container platform updated.
6. Avoid exposing the Docker daemon socket.
7. Segment container networks when workloads do not need broad connectivity.

**Exam focus**

For Security+ purposes, remember the relationship between image security, registry trust, runtime restrictions, and host patching. The key idea is that container security is not only about the container itself; it is also about the image supply chain, the platform configuration, and the underlying host.
