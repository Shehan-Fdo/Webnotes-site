## **Course:** CompTIA Security+ (Exam SY0-701)

**Data security design overview**

Data security design focuses on protecting information based on how sensitive it is and where it exists in its lifecycle. A practical design starts with classification, then applies controls that match the data’s exposure risk.

Security+ expects you to recognize that data can be protected differently when it is stored, transmitted, or actively being processed. The goal is not one control for everything, but the right control for the right state.

- **Data classification** helps you prioritize controls by sensitivity and business impact.
- Sensitive data can create reputational, financial, and competitive risk if exposed.
- Design decisions should account for compliance requirements and operational use.

**Classifying data**

Classification is the foundation for data protection. You organize data by sensitivity and importance so you can apply appropriate safeguards instead of treating all information the same.

A useful classification approach considers both the type of data and how much harm would result from exposure. This helps you decide where to invest stronger controls and where lighter controls are acceptable.

- Classify data by sensitivity, not just by file type.
- Use labels or categories to signal handling requirements.
- Revisit classification as data changes or moves between systems.

**Protecting data at rest, in motion, and in use**

Data protection changes based on the data state. Encryption is a core control for data at rest and data in transit, while data in use often needs additional safeguards because it is actively being processed.

Data at rest is stored data, such as files or database records. Data in motion is moving across a network. Data in use is being accessed or processed by an application or user.

- **At rest:** use encryption and isolation to reduce exposure.
- **In motion:** use strong, authenticated protocols.
- **In use:** limit exposure through access controls and careful system design.

> Encryption protects data, but it does not remove sensitivity. If an attacker gets the key, the protection fails.

**Encryption, masking, and tokenization**

These controls solve different problems, so you should not treat them as interchangeable. Encryption is reversible with a key and is commonly used for transit and storage protection. Masking hides visible values, which is useful for user interfaces and test environments. Tokenization replaces sensitive values with placeholders and stores the original data separately.

Use the control that matches the business need:

- **Encryption** protects the underlying value and is reversible with the correct key.
- **Masking** reduces what users can see, but the original data may still exist in the system.
- **Tokenization** removes the sensitive value from the exposed context and maps it to a separate vault.

**Access control and information flow**

Data security design also depends on controlling who can access data and where data can move. Least privilege reduces unnecessary exposure, and contextual access control helps you make decisions based on user, device, or situation.

Information flow control prevents unauthorized movement of data between systems or trust zones. This matters when you want to stop sensitive data from reaching places where it should not exist, such as lower-trust environments.

- Grant only the access needed for the task.
- Restrict data flows between environments when the destination is not appropriate.
- Minimize exposure in non-production systems.

**Operational safeguards**

Good design includes more than encryption and access control. You also need controls that reduce misuse, support recovery, and limit long-term exposure.

Retention and deletion matter because old data becomes unnecessary risk. Audit trails help you detect misuse, and redundancy with integrity checks supports resilience.

- Limit data retention to what is required.
- Enforce secure deletion when data is no longer needed.
- Use audit logs and behavioral signals to detect misuse.
- Design for redundancy and verifiable integrity where availability matters.

**Implementation priorities**

A practical Security+ approach is to build data protection into the system from the start. That means you classify first, then choose controls based on sensitivity, state, and business use.

1. Identify the data and classify it by sensitivity.
2. Decide where the data exists: at rest, in motion, or in use.
3. Apply the right protection method for each state.
4. Restrict access and control information flow.
5. Add retention, logging, and deletion controls.

**Exam focus**

For exam questions, look for the control that best matches the data state and the business scenario. If the question mentions stored records, think at-rest protection. If it mentions network transfer, think in-transit protection. If it mentions users viewing partial values or test data, think masking or tokenization.

Keep in mind that classification drives everything else. Once you know how sensitive the data is, the rest of the design choices become easier to justify.
