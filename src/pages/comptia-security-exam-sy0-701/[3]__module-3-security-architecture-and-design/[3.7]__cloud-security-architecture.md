## **Course:** CompTIA Security+ (Exam SY0-701)

**Shared responsibility model**

The shared responsibility model defines how security duties split between the cloud service provider (CSP) and the customer. The CSP secures the cloud infrastructure, while you secure what you place in and configure on top of it.

This boundary is not fixed across all cloud services. It changes with the service model, so you need to document ownership for each environment instead of assuming one rule applies everywhere.

- **CSP** responsibilities typically include the underlying infrastructure.
- **Customer** responsibilities typically include data, applications, configurations, and access controls.
- Responsibility shifts across `IaaS`, `PaaS`, `SaaS`, `FaaS`, and serverless services.
- Assumption gaps create risk when teams expect the provider to enforce controls such as least privilege or API protection automatically.

**What the provider secures**

Cloud providers manage the security of the cloud itself. That includes the physical and core platform layers that customers do not directly administer. AWS describes this as covering components from the host operating system and virtualization layer down to the physical security of the facilities.

In practical terms, the provider reduces your operational burden, but it does not remove your need to secure workloads and identities.

- Data center and facility security stay with the provider.
- Network hardware and core services stay with the provider.
- Host-level platform management stays with the provider.
- Provider-managed patching applies only to the layers in scope for that service.

**What you secure**

You remain responsible for the security of your cloud usage. That includes the data you store, the applications you deploy, and the configurations you choose.

Misconfigurations account for most cloud breaches, so configuration control matters as much as service selection. Continuous monitoring and automated validation help you detect exposure and configuration drift before it becomes an incident.

- Protect data with appropriate access controls.
- Secure applications you deploy into the cloud.
- Review configurations continuously.
- Validate settings automatically where possible.
- Treat identity and access management as a core control area.

**Service model differences**

The shared responsibility boundary changes as abstraction increases. In lower-level services, you manage more of the stack. In higher-level services, the provider manages more of it.

| Service type | Customer responsibility increases for | Provider responsibility increases for |
|---|---|---|
| `IaaS` | Guest operating system, applications, and configuration | Physical infrastructure and virtualization layers |
| `PaaS` | Application code, data, and access settings | Managed platform components |
| `SaaS` | Data, user access, and tenant configuration | Application and platform operation |
| `FaaS` / serverless | Function code, data, and permissions | Execution environment and underlying service |

Use the service model to decide who owns patching, hardening, and access enforcement. Do not assume the same control set applies to every cloud offering.

**Modern cloud security concerns**

Cloud security changes as services evolve. Containers, functions, low-code, and no-code platforms expand the number of places where responsibility can shift or become unclear.

That makes clear ownership documentation important. If teams do not define who secures each layer, gaps appear in patching, access control, logging, and validation.

- New service paradigms can blur old service-model assumptions.
- Ownership should be explicit for each cloud service.
- Security reviews should include integration points and regulatory requirements.
- Control expectations should match the actual service contract.

**Exam focus**

For Security+, focus on the relationship between service model and responsibility boundary. The key idea is simple: the provider secures the cloud infrastructure, and you secure your data, applications, configurations, and access.

A good test answer usually identifies:
- Which party owns the layer in question.
- Whether the service is `IaaS`, `PaaS`, `SaaS`, or serverless.
- Whether the issue is a provider control or a customer misconfiguration.

> Misconfiguration is a common cloud risk, so always check whether the problem is caused by a customer setting rather than a provider failure.

**Quick recall**

- Provider: infrastructure, facilities, host platform.
- Customer: data, apps, configs, access.
- More abstraction means more provider responsibility.
- More control means more customer responsibility.
