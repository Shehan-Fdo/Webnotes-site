## **Course:** CompTIA Security+ (Exam SY0-701)

**Secure system design and hardening**

Secure system design focuses on building systems that resist tampering, preserve integrity, and support trusted operations from boot through runtime. Hardening reduces attack surface by enabling only the controls you need and by validating platform state before the system is trusted.

In practice, this topic centers on the relationship between firmware, boot validation, and hardware-backed security features. The evidence highlights <abbr title="Trusted Platform Module">TPM</abbr> 2.0, Secure Boot, and the broader idea of a hardware root of trust.

- **TPM 2.0** is a hardware foundation for storing key material, measurements, and cryptographic operations.
- **Secure Boot** uses the UEFI (Unified Extensible Firmware Interface) trust model to validate the boot path.
- These controls work together as a **chain of trust**, not as isolated settings.
- Platform security can include automatic unlocking of encrypted volumes when the platform state is trusted.
- Mature environments may use remote attestation before admitting a node into a cluster or segment.

**Trusted boot and platform integrity**

Trusted boot starts before the operating system loads. UEFI defines the mechanics of Secure Boot and the trusted key model, while TPM 2.0 supports measurement and key handling. Together, they help verify that the system has not been altered in ways that would undermine trust.

A practical way to think about this is that Secure Boot checks what is allowed to start, and TPM-backed measurements help record what actually started. That distinction matters when you need to prove platform state or protect secrets tied to a specific machine.

- Secure Boot protects the boot path by validating trusted components.
- TPM 2.0 supports platform measurements and secure key storage.
- The combination supports a broader **hardware root of trust** model.
- Platform state validation can support encrypted storage workflows.
- Remote attestation extends trust decisions beyond the local machine.

> Use conservative language on the exam: these features improve trust and integrity, but they do not make a system invulnerable.

**Hardening goals and design outcomes**

Hardening is about reducing unnecessary exposure while preserving required functionality. For SY0-701, the key idea is that security controls should support confidentiality, integrity, and availability without relying on software alone.

The evidence also ties TPM 2.0 to common security features such as Windows Hello for identity protection and BitLocker for data protection. That makes TPM a useful example of how hardware support enables higher-level security functions.

- **Identity protection** can use TPM-backed features such as Windows Hello.
- **Data protection** can use TPM-backed features such as BitLocker.
- Hardware support helps protect secrets even when software is compromised.
- Security design should account for boot-path protection and key handling.
- Platform resiliency improves when trust decisions are based on measured state.

**Implementation considerations**

When you evaluate a system design, look for where trust begins and how it is maintained. If the platform cannot establish a trusted boot path, later controls may inherit a weak foundation.

Use the following sequence as a mental model for secure startup:

1. The firmware initializes the platform and applies the UEFI trust model.
2. Secure Boot validates allowed boot components.
3. TPM 2.0 stores or measures security-relevant state.
4. The system uses that trusted state for key handling or encrypted volume access.
5. Higher-level trust decisions can rely on the measured platform state.

**Exam focus points**

- Know that **TPM 2.0** is a hardware security component, not just a software setting.
- Know that **Secure Boot** is part of the boot-path trust chain.
- Know that both features support a broader platform security model.
- Know that trusted boot can support encrypted storage and attestation use cases.
- Know that hardening aims to reduce risk by validating and constraining system behavior.

**Quick comparison**

| Concept | Primary role | Exam takeaway |
|---|---|---|
| Secure Boot | Validates boot components | Protects the boot path |
| TPM 2.0 | Stores measurements and key material | Provides hardware-backed trust support |
| Chain of trust | Connects boot and platform validation | Treat as a linked security model |
| Remote attestation | Verifies node state before admission | Used in mature environments |

**Practical memory aid**

Think of the sequence as: <mark>validate boot</mark>, <mark>measure state</mark>, <mark>protect keys</mark>, then <mark>trust the platform</mark>.
