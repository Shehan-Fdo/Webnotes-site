## **Course:** CompTIA Security+ (Exam SY0-701)

**Secure design mindset**

Secure design means you build systems to handle data responsibly and reduce exposure to attacks from the start. The goal is not just functional software, but software that protects confidentiality, integrity, and availability while supporting trust.

Security+ emphasizes practical architecture and design skills, so you should think in terms of how a system behaves under normal use, error conditions, and attack conditions. A secure design is easier to defend, easier to reason about, and less likely to contain avoidable flaws.

- **Secure by Design** focuses on building security into the system from the beginning.
- **Privacy by Design** extends that idea to how you handle user data and privacy expectations.
- **Defense in Depth** uses multiple layers of protection so one failure does not expose the whole system.

**Core principles**

There is no such thing as perfect security, so design choices usually balance prevention and risk management. Prevention aims to reduce vulnerabilities before they are exploited. Risk management accepts that resources are limited and focuses on reducing expected harm.

A practical design also considers accountability. If you can attribute malicious activity to a person or system, you improve deterrence and response options.

- **Prevention** tries to eliminate vulnerabilities before attacks succeed.
- **Risk management** weighs the cost of controls against the harm they reduce.
- **Deterrence through accountability** depends on being able to attribute attacks.

**Saltzer and Schroeder principles**

Several classic principles help you design systems that fail safely and stay understandable. These principles are especially useful when you need to justify why a control exists or why a default should be restrictive.

- **Economy of mechanism** means keep the design simple.
  - Simpler systems are easier to understand.
  - Simpler systems are less likely to contain flaws.
- **Fail-safe default** means deny access unless you explicitly allow it.
  - This is the basis for default-deny and whitelisting approaches.
  - A firewall policy should block traffic unless a rule allows it.
- **Complete mediation** means every access must be checked.
  - You should not allow a path that bypasses access control.

**How these ideas show up in practice**

Secure design principles shape how you configure access, services, and controls. When you choose defaults, you should prefer denial over allowance unless there is a clear business need. When you simplify a design, you reduce the number of places where mistakes can hide.

These ideas also support secure architecture decisions in modern environments, including cloud, on-premises, and software patterns such as microservices or serverless computing. The exact implementation varies, but the design goal stays the same: reduce exposure and keep control points consistent.

- Use **default-deny** rules for network and object access.
- Keep access control checks in one consistent path.
- Reduce unnecessary complexity in system design.
- Layer controls so one weakness does not become a full compromise.

**Exam focus**

For Security+ SY0-701, connect each principle to its practical effect:

- **Secure by Design**: security is built in, not added later.
- **Privacy by Design**: data handling respects privacy from the start.
- **Defense in Depth**: multiple controls protect the same asset.
- **Economy of mechanism**: simpler design lowers error risk.
- **Fail-safe default**: deny by default, allow by exception.
- **Complete mediation**: every access is checked.

> If a question describes a system that blocks access unless explicitly permitted, think of fail-safe default or whitelisting.

**Quick comparison**

| Principle | Main idea | Practical outcome |
|---|---|---|
| Economy of mechanism | Keep it simple | Fewer flaws and easier review |
| Fail-safe default | Deny by default | Safer default behavior |
| Complete mediation | Check every access | No bypass of controls |
| Defense in depth | Layer protections | One failure does not end security |

**Study approach**

When you review a scenario, ask what the design is trying to prevent and what happens if a control fails. If the question emphasizes simplicity, default denial, or layered protection, it is testing secure design thinking rather than a single product or tool.

1. Identify the asset being protected.
2. Determine whether the design uses prevention, risk reduction, or accountability.
3. Check whether access is denied by default.
4. Confirm that every access path is mediated.
5. Look for layered controls that limit blast radius.
