## **Course:** CompTIA Security+ (Exam SY0-701)

**Cryptographic technology overview**

Representative cryptographic technologies are the practical tools and protocols you use to protect data in transit and at rest. The main exam focus is on how these technologies support confidentiality, integrity, and secure key handling.

- **TLS** (Transport Layer Security) protects network traffic between a client and a server.
- **Encryption at rest** protects stored data on disks, images, and databases.
- **KMS** (Key Management Service) generates, stores, uses, and destroys cryptographic key material.
- **HSM** (Hardware Security Module) provides dedicated hardware for stronger key isolation and control.

Encryption is not just about the algorithm itself. Secure implementation depends on protocol version, cipher suite selection, certificate handling, and where the keys live.

**TLS and secure transport**

TLS is the standard protocol for securing data in transit. It encrypts communication so unauthorized parties cannot easily read or alter traffic. It also improves on older SSL approaches by using stronger encryption and better certificate validation.

TLS 1.3 is the modern baseline to know for exam prep. Compared with TLS 1.2, it removes legacy symmetric algorithms, uses only AEAD (Authenticated Encryption with Associated Data) algorithms, and encrypts handshake messages after `ServerHello`. It also replaces older key derivation approaches with HKDF (HMAC-based Extract-and-Expand Key Derivation Function).

- TLS 1.2 is still widely supported.
- TLS 1.3 improves security and performance.
- TLS 1.3 removes static RSA and Diffie-Hellman cipher suites.
- TLS 1.3 supports forward secrecy through public-key-based key exchange.
- TLS 1.3 can use 0-RTT (zero round-trip time) for some application data, but that comes with tradeoffs.

> When a service requires modern transport security, prefer TLS 1.2 or later, and use TLS 1.3 where supported.

**Cloud key management services**

A KMS is a common cloud service for managing cryptographic keys. It typically handles key generation, storage, use, and destruction. In practice, you use a KMS to protect things like database passwords, virtual machine disk images, and other sensitive cloud assets.

A well-designed KMS supports secure storage of encryption keys and can also support encryption and digital signature algorithms. It becomes a foundational part of cloud security because effective encryption depends on secure key management.

- Use a KMS when you need centralized key lifecycle control.
- Use provider KMS features when they meet your security and compliance needs.
- Treat key management as part of the encryption design, not an afterthought.
- Verify that the KMS supports the algorithms and operational controls you need.

**HSM versus KMS**

HSMs and KMSs both support cryptographic protection, but they serve different operational goals. A KMS is usually easier to consume as a managed service, while an HSM gives you stronger hardware-backed isolation and more direct control over key usage.

- **KMS** fits common cloud encryption and signing workflows.
- **HSM** fits higher-assurance use cases where dedicated hardware isolation matters.
- HSMs are useful when you need separation from provider-managed control planes.
- KMSs are useful when you want managed key operations with less operational overhead.

A practical way to think about the difference is that KMS focuses on managed key lifecycle services, while HSM focuses on protected key storage and cryptographic operations inside dedicated hardware.

**Algorithm and cipher suite selection**

Modern cryptographic systems depend on approved algorithms and secure cipher suites. The evidence emphasizes that current TLS deployments should avoid legacy options and use suites that provide perfect forward secrecy (PFS), such as DHE (Diffie-Hellman Ephemeral) or ECDHE (Elliptic Curve Diffie-Hellman Ephemeral).

- Prefer algorithms that are current and supported by the platform.
- Avoid legacy cipher suites when a modern alternative exists.
- Use PFS-capable suites for stronger session protection.
- Confirm that clients support the required TLS version and cipher suites.

AWS KMS, for example, requires TLS 1.2 and recommends TLS 1.3 in all regions. It also requires clients to support TLS 1.2 or later and cipher suites with PFS such as DHE or ECDHE. That pattern reflects a broader security principle: secure transport and secure key exchange are both part of cryptographic hygiene.

**Encryption in transit and at rest**

Cloud data commonly exists in two states: in transit and at rest. In transit, TLS creates the secure wrapper around data moving between systems. At rest, encryption protects data stored on disks, images, and other persistent media.

- Use TLS for data moving across networks.
- Use encryption at rest for stored data.
- Use cloud KMS features to manage the keys behind at-rest encryption.
- Match the control to the data state you are protecting.

Encryption at rest is especially important for cloud storage, database content, and virtual machine images. The key point is that encryption only works well when the keys are protected and the service uses approved algorithms correctly.

**Exam focus reminders**

For Security+, focus on recognizing the role of each technology rather than memorizing vendor-specific details.

- TLS secures communication channels.
- KMS manages cryptographic keys.
- HSM provides dedicated hardware protection for keys.
- TLS 1.3 is more secure and streamlined than TLS 1.2.
- PFS reduces the impact of key compromise on past sessions.

If a question asks what to use for secure cloud encryption, think in layers: transport security, key management, and storage protection all work together.
