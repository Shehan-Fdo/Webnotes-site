## **Course:** CompTIA Security+ (Exam SY0-701)

**Federation and identity access basics**

Federation and single sign-on (SSO) both reduce the burden of managing many passwords, but they solve slightly different problems. SSO lets a user access multiple applications with one set of credentials. Federation extends trust across organizational boundaries so a user can use a verified identity from one domain to access trusted services in another.

In practice, these approaches help reduce password sprawl and improve user experience. They also support centralized identity management, which is easier to administer than storing separate local accounts on every system.

- **SSO** gives one login for multiple applications.
- **Federation** lets trusted identity providers and service providers share authentication trust.
- **Identity provider** is the system that verifies the user.
- **Service provider** is the application or service that accepts the identity assertion.

**Why organizations use them**

Identity sprawl creates operational and security problems. Users often need access to many systems, and local account storage becomes difficult to manage at scale. Centralized authentication protocols help devices and applications check credentials against a central server instead of maintaining separate records everywhere.

Common benefits include:

- Fewer passwords for users to remember
- Easier access to internal and external applications
- Centralized control over authentication
- Better support for partner, customer, and employee access scenarios

**Common identity protocols**

Several protocols appear frequently in enterprise authentication and federation designs. They are not always direct competitors; many solve different parts of the identity problem.

| Protocol | Typical role | Practical note |
|---|---|---|
| **SAML** | Federation and SSO | Common for enterprise web application sign-in |
| **OAuth 2.0** | Authorization delegation | Used to grant limited access without sharing the user’s password |
| **OpenID Connect (OIDC)** | Authentication layer on top of OAuth 2.0 | Used when an application needs identity information |
| **LDAP** | Directory access | Used to locate and query users, groups, and resources |
| **RADIUS** | Network authentication | Common for centralized access to network resources |
| **Kerberos** | Network authentication | Used in many Windows and enterprise environments |
| **TACACS+** | Administrative network access | Often used for device administration |

**SAML and federation**

Security Assertion Markup Language (SAML) is a common federation protocol for exchanging authentication information between an identity provider and a service provider. It is widely used in enterprise environments where users sign in once and then access multiple web applications.

SAML is useful when you want:

- Centralized login for business applications
- Trust between separate organizations or systems
- A way to pass authentication assertions instead of reusing passwords

> SAML is about exchanging trusted assertions, not sharing the user’s password.

**OAuth 2.0 and OpenID Connect**

OAuth 2.0 is primarily an authorization framework. It lets an application obtain limited access to a resource on behalf of a user. It is not the same thing as authentication by itself.

OpenID Connect adds an identity layer to OAuth 2.0. Use it when an application needs to know who the user is, not just what the user can access.

Key distinctions:

- **OAuth 2.0** focuses on delegated access.
- **OpenID Connect** adds user identity information.
- **SAML** is often used for enterprise SSO.
- **OAuth 2.0** is common in modern web and API access flows.

**Directory and network authentication protocols**

LDAP, RADIUS, Kerberos, and TACACS+ are common in enterprise identity systems, especially where centralized authentication is needed for network resources and administration.

- **LDAP** stands for Lightweight Directory Access Protocol.
  - It is used to locate users, groups, devices, and other resources in a directory.
  - It helped form the basis for directory services such as Active Directory.
- **RADIUS** is used for centralized authentication and authorization of network access.
- **Kerberos** is a widely used protocol for verifying identity in enterprise networks.
- **TACACS+** is often used for administrative access to network devices.

These protocols support different layers of access control, so you choose them based on the resource you are protecting.

**How to think about SSO vs. federation**

SSO and federation are related, but they are not identical.

- **SSO** is about reducing repeated logins within a trusted environment.
- **Federation** is about trusting identities across domains or organizations.
- A single sign-in experience can use federation, but not every SSO deployment is federated.

A practical way to remember it:

- If users access multiple apps under one organization, think **SSO**.
- If users access a trusted external system using an identity from another organization, think **federation**.

**Enterprise authentication considerations**

Enterprise systems often support multiple authentication backends, including LDAP, RADIUS, SAML, and generic OIDC. In some platforms, users authenticated through enterprise backends are treated differently from local users.

Important operational points:

- Enterprise users may be created only after a successful first login from the remote backend.
- A local user with the same name can block enterprise user creation.
- Enterprise user passwords may need to remain empty when the account is backend-enabled.
- If enterprise backends are disabled, an enterprise user may be converted to a normal local user by setting a password.

**Study approach for Security+**

For exam purposes, focus on the purpose of each protocol rather than memorizing every implementation detail. You should be able to identify which protocol fits a scenario.

1. Determine whether the question is about login reuse, cross-domain trust, or directory access.
2. Match the scenario to the protocol:
   - SSO for one login across multiple apps
   - Federation for trust across organizations
   - LDAP for directory queries
   - RADIUS or TACACS+ for network or device access
   - Kerberos for enterprise authentication
3. Watch for wording that separates authentication from authorization.
4. Look for clues about web apps, APIs, or enterprise directories.

**Quick recall**

- **SSO** reduces repeated logins.
- **Federation** shares trust between identity domains.
- **SAML** is a common federation and SSO protocol.
- **OAuth 2.0** is for delegated authorization.
- **OpenID Connect** adds authentication to OAuth 2.0.
- **LDAP** queries directory information.
- **RADIUS**, **Kerberos**, and **TACACS+** support centralized network authentication.
