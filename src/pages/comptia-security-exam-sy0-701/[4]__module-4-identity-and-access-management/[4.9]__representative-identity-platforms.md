## **Course:** CompTIA Security+ (Exam SY0-701)

**Representative identity platforms**

Identity and access management (IAM) platforms help you control who can access resources, when access is granted, and how authentication happens. In modern environments, these platforms often support cloud, hybrid, and distributed workforces, so identity becomes a core security control rather than a back-office function.

Common platform capabilities include:
- **Authentication** for verifying user identity.
- **Single sign-on (SSO)** for reducing repeated logins across apps.
- **Federation** for trusting identities across systems or organizations.
- **Multi-factor authentication (MFA)** for adding a second verification step.
- **Lifecycle management** for provisioning and deprovisioning accounts.

**Microsoft identity platform and Entra ID**

Microsoft identity platform is a cloud identity service that lets users and customers sign in with Microsoft identities or social accounts. It also authorizes access to Microsoft APIs such as Microsoft Graph and supports both single-tenant line-of-business applications and multitenant software-as-a-service applications.

Microsoft Entra ID is commonly used in enterprise identity workflows and can participate in external authentication scenarios. In practice, this means you may see it used as part of a broader identity architecture that connects applications, users, and policy-driven access.

Key points to remember:
- **OAuth 2.0** and **OpenID Connect** are standard protocols in the Microsoft identity platform.
- The platform supports application registration and identity-aware access to APIs.
- Entra ID can be part of multi-factor and external authentication designs.

**Ping Identity and external authentication**

Ping Identity products support identity workflows that extend beyond a single directory. One example is using PingOne Advanced Identity Cloud as an external authentication method for Microsoft Entra ID through OpenID Connect (OIDC).

This type of integration shows how identity platforms can work together:
- A user authenticates through one platform.
- Another platform provides a second-factor or external authentication step.
- A custom journey can enforce the desired MFA flow.

Use this model when you need flexible authentication orchestration across systems rather than a single isolated login source.

**Cisco ISE and directory integration**

Cisco Identity Services Engine (ISE) is used in enterprise access control environments and can integrate with Microsoft Active Directory, Entra ID, and Intune. This reflects a common security pattern: the identity platform supplies user and device context, while the access-control system uses that context to make policy decisions.

Practical takeaways:
- **Active Directory (AD)** remains relevant in traditional enterprise identity setups.
- **Entra ID** extends identity into cloud-based access scenarios.
- **Intune** adds device management context that can support access policy decisions.

**Google Cloud Identity Platform**

Google Cloud Identity Platform provides authentication as a service for web and mobile applications. It is designed to be customizable and to scale on Google Cloud infrastructure.

Notable capabilities include:
- **User sign-up and sign-in** for application access.
- **MFA** to help protect accounts from takeover.
- **Enterprise-grade support** and service-level agreement (SLA) options for demanding workloads.
- **Global scale** for applications that need broad reach and reliability.

This platform is a good example of customer identity and access management (CIAM), where the focus is on application users rather than only internal employees.

**How to compare representative platforms**

When you evaluate identity platforms, focus on what problem each one solves. Some platforms emphasize enterprise workforce identity, while others emphasize customer identity, external authentication, or application sign-in services.

| Platform | Typical emphasis | Notable idea |
|---|---|---|
| Microsoft identity platform / Entra ID | Workforce and application identity | Supports Microsoft identities, social accounts, and API authorization |
| Ping Identity | Federation and external authentication | Can act as an external authentication method with OIDC |
| Cisco ISE | Access control integration | Uses identity and device context for policy decisions |
| Google Cloud Identity Platform | Customer identity for apps | Authentication as a service with MFA and scale |

**Exam focus**

For Security+ purposes, focus on the security function of each platform rather than vendor branding. You should be able to recognize:
- **IAM** as the umbrella discipline for identity control.
- **SSO** as a usability and security improvement.
- **Federation** as trust across identity domains.
- **MFA** as a control that strengthens authentication.
- **Directory integration** as a way to connect identity sources to access policy.

> Identity platforms are often combined. A real environment may use one system for directory services, another for MFA, and another for device or access policy enforcement.

**Quick review**

1. Identify the platform’s main role: workforce identity, customer identity, federation, or access control.
2. Check whether it supports SSO, MFA, or OIDC.
3. Determine whether it integrates with directories, APIs, or device-management systems.
4. Match the platform to the access problem you are solving.
