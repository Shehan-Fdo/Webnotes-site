## **Course:** CompTIA Security+ (Exam SY0-701)

**Incident response overview**

Incident response is a structured process organizations use to identify, contain, and recover from security incidents. A clear lifecycle reduces damage, restores normal operations faster, and helps prevent repeat attacks.

The NIST (National Institute of Standards and Technology) model is the most common reference point in the evidence. It organizes response into four phases, while some other models expand the workflow into more steps. For exam purposes, focus on the core sequence and the purpose of each phase.

- **Structured process** means you follow a repeatable workflow instead of improvising during an incident.
- **Containment** limits the spread or impact of the incident.
- **Recovery** returns systems and services to normal operation.
- **Lessons learned** improves future response planning.

**NIST incident response lifecycle**

The NIST lifecycle is commonly described in four phases:

1. **Preparation and prevention**
   - Build the incident response plan before an event occurs.
   - Define roles, procedures, and communication paths.
   - Prepare tools and processes that support detection and recovery.

2. **Detection and analysis**
   - Identify whether an event is a real security incident.
   - Analyze alerts, logs, and other evidence to understand scope and impact.
   - Confirm what happened before taking broader action.

3. **Containment, eradication, and recovery**
   - Contain the incident to reduce damage.
   - Eradicate the threat by removing malicious presence or cause.
   - Recover affected systems and restore operations.

4. **Post-incident activity**
   - Review what happened and what worked.
   - Capture lessons learned.
   - Update plans and controls to strengthen future response.

> The exact wording of the phases can vary across sources, but the practical flow stays the same: prepare, detect, contain, remove, recover, and improve.

**Why the lifecycle matters**

A well-defined incident response lifecycle gives the organization a repeatable way to handle cyber incidents. It supports faster decision-making during pressure, helps reduce operational disruption, and creates a path for continuous improvement after the event.

- It helps you respond in a consistent way.
- It reduces confusion during high-stress incidents.
- It supports faster restoration of business activity.
- It improves future defenses through review and refinement.

**Comparing common models**

Some sources describe incident response in four phases, while others break the work into more detailed steps such as identification, containment, eradication, recovery, lessons learned, and continuous enhancement. The extra detail does not change the core idea; it adds granularity around the same response flow.

| Model style | Typical emphasis |
|---|---|
| NIST 4-phase model | Broad lifecycle: preparation, detection and analysis, containment/eradication/recovery, post-incident activity |
| Expanded step model | More granular handling: identification, containment, eradication, recovery, lessons learned, continuous improvement |

Use the model language that appears in the question, but map it back to the same operational goals.

**Practical response priorities**

During an incident, your priorities follow the lifecycle order. You first confirm the event, then limit damage, remove the threat, and restore service. Afterward, you review the response so the organization can respond better next time.

- **Detection** comes before containment because you need enough confidence that an incident exists.
- **Eradication** follows containment because you remove the cause after limiting spread.
- **Recovery** should restore normal operations only after the threat is addressed.
- **Post-incident review** turns the event into a learning opportunity.

**Exam focus**

For Security+ exam prep, remember the lifecycle as a sequence of response actions rather than isolated tasks. If a question asks what happens after containment, the next practical steps are eradication and recovery. If it asks what happens after the incident ends, the answer is post-incident activity or lessons learned.

- Know the four NIST phases.
- Recognize alternate wording for the same phases.
- Associate each phase with its main goal.
- Treat post-incident activity as a feedback loop, not an optional extra.
