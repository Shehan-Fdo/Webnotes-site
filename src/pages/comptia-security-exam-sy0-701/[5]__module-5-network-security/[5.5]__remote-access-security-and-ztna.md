## **Course:** CompTIA Security+ (Exam SY0-701)

**Remote access security overview**

Remote access security focuses on letting users reach internal resources without assuming the connection is trustworthy. In modern environments, hybrid and remote work increase exposure on remote access pathways, so security controls must verify identity, device health, and context before granting access.

- **Identity-driven** access is central to modern remote access design.
- **Least privilege** reduces exposure by limiting what a user can reach.
- **Continuous verification** helps maintain trust throughout a session, not just at login.
- Remote access controls should support users without exposing internal applications broadly to the internet.

**Zero Trust Network Access (ZTNA)**

Zero Trust Network Access, or **ZTNA**, is a category of technologies that provides secure remote access to applications and services based on defined access control policies. It aligns with the Zero Trust model by avoiding implicit trust and granting access only when policy conditions are met.

A practical way to think about ZTNA is that it gives users access to specific applications rather than placing them on the network. This reduces the chance of unauthorized access and limits lateral movement if a session is compromised.

- ZTNA uses **granular policies** to decide access.
- ZTNA commonly considers user identity, device health, and context.
- ZTNA supports secure connectivity to private apps without exposing those apps directly.
- ZTNA is often positioned as a modern alternative to legacy VPN-style broad access.

> ZTNA does not automatically trust a user after sign-in; it keeps evaluating access based on policy and context.

**ZTNA compared with VPN-style access**

Traditional virtual private networks, or **VPNs**, often provide broader network access once a user connects. The evidence consistently contrasts this with ZTNA’s application-level access model.

| Aspect | ZTNA | VPN-style access |
|---|---|---|
| Access scope | Specific applications and services | Broader network access |
| Trust model | No implicit trust | More implicit trust after connection |
| Exposure | Does not place users on the network | Can place users on the network |
| Policy style | Granular, least-privileged | Less granular by default |

Use ZTNA when you want to reduce exposure and keep access tightly scoped to what the user needs. This is especially useful for distributed workforces and third-party access scenarios.

**SASE and how it relates to ZTNA**

Secure access service edge, or **SASE**, is a cloud-native architecture that combines networking and security services into a unified framework. The evidence identifies technologies such as SD-WAN, SWG, CASB, firewall as a service, and ZTNA as part of that framework.

ZTNA is one component that can fit inside a broader SASE design. In other words, SASE is the larger architecture, while ZTNA is one of the access-control capabilities that helps deliver secure remote access.

- **SD-WAN** optimizes traffic routing across multiple connections.
- **SWG** or secure web gateway helps protect users from online threats.
- **CASB** or cloud access security broker supports cloud app security controls.
- **Firewall as a service** extends firewall capabilities through the cloud.
- **ZTNA** provides secure access to applications based on policy.

**Why these models matter for hybrid work**

Hybrid work changes where users connect from and how organizations protect data, apps, and corporate information assets. The evidence points to identity-driven security as a response to this shift, because it helps protect resources without sacrificing performance.

- Remote users need access that is secure and efficient.
- Security teams need to protect devices, data, apps, and information assets.
- Compliance and risk reduction improve when access is individualized.
- Network performance remains important, so access controls should not create unnecessary friction.

**Practical exam takeaways**

For Security+ purposes, focus on the relationship between access model and trust model. ZTNA is about limiting access to what is explicitly allowed, while SASE is the broader cloud-delivered framework that can include ZTNA.

1. Identify whether the question is asking about broad network access or application-level access.
2. Match **ZTNA** to least privilege, granular policy, and no implicit trust.
3. Match **SASE** to the broader cloud-native architecture that combines networking and security services.
4. Use the context of hybrid work to explain why identity-driven access matters.

**Key terms to remember**

- **Zero Trust**: trust is not assumed by default.
- **ZTNA**: secure remote access to specific applications and services.
- **SASE**: unified cloud-native networking and security architecture.
- **Identity-driven security**: access decisions based on who the user is and related context.
