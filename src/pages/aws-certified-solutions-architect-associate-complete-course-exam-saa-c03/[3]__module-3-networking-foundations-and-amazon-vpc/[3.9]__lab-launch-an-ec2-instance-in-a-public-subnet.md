## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab overview**

A public subnet is a subnet whose route table sends internet-bound traffic to an <abbr title="Internet Gateway">IGW</abbr>. In practice, you use it when an instance must be reachable from the internet, such as a bastion host, a web server, or a test system that needs outbound package updates and inbound access for validation.

The key idea is simple: the subnet must have a route to an Internet Gateway, and the instance must have a public IPv4 address or Elastic IP. Security groups and network ACLs still control what traffic is allowed.

- **Public subnet**: a subnet associated with a route table that has a `0.0.0.0/0` route to an IGW.
- **Public IPv4 address**: an address that allows the instance to be reached from the internet when routing and security allow it.
- **Security group**: stateful firewall attached to the instance.
- **Network ACL**: stateless subnet-level filter.

**Core VPC components**

Before launching the instance, confirm the network path. A public subnet is not defined by its name; it is defined by its route table.

| Component | Purpose | Notes |
|---|---|---|
| VPC | Isolated virtual network | Contains subnets, route tables, gateways, and instances |
| Subnet | IP range inside the VPC | Public if its route table points `0.0.0.0/0` to an IGW |
| Internet Gateway | Enables internet connectivity for resources in a VPC | Attach it to the VPC before using it in routes |
| Route table | Controls traffic forwarding | Associate the public subnet with the correct table |
| Security group | Instance-level allow rules | Must allow inbound traffic you want to test |
| Network ACL | Subnet-level allow/deny rules | Must not block required traffic |

A common exam trap is assuming that assigning a public IP alone makes a subnet public. It does not. Without a route to an Internet Gateway, the instance cannot reach the internet, even if it has a public address.

**Launch prerequisites**

You typically need the following before launch:

- A VPC with at least one subnet.
- An Internet Gateway attached to the VPC.
- A route table with `0.0.0.0/0` pointing to the IGW.
- The route table associated with the target subnet.
- A security group that allows the required inbound traffic, such as SSH on port `22` from your IP.
- A key pair if you plan to connect with SSH.
- A subnet setting that can auto-assign public IPv4 addresses, or a launch method that explicitly assigns one.

> The Internet Gateway is a horizontally scaled, redundant VPC component that provides a path between instances in a VPC and the internet. It is not a NAT device and should not be described as performing address translation in the way a NAT Gateway does.

**Subnet configuration**

A subnet becomes public when its route table includes a default route to the IGW.

```text
Destination      Target
0.0.0.0/0        igw-xxxxxxxx
10.0.0.0/16      local
```

If you are creating the subnet in the console, you may see an option to enable auto-assign public IPv4 addresses. That setting helps new instances receive a public IPv4 address at launch, but it does not replace the route table requirement.

- **Auto-assign public IPv4**: convenient for instances that should be internet-reachable by default.
- **Manual public IP assignment**: possible through a network interface configuration or by associating an Elastic IP after launch.
- **Elastic IP**: a static public IPv4 address that you can remap between instances.

**Launching the instance in the console**

A typical launch flow in the AWS Management Console is:

1. Open the EC2 console and choose **Launch instance**.
2. Select an Amazon Machine Image (AMI), such as Amazon Linux.
3. Choose an instance type that fits the lab, such as `t3.micro`.
4. Select the VPC and the public subnet.
5. Ensure public IPv4 assignment is enabled if the subnet does not auto-assign it.
6. Choose or create a security group that allows SSH from your IP.
7. Select or create a key pair.
8. Launch the instance and wait for the status checks to pass.

The instance needs both network reachability and security-group permission. If you can see the instance but cannot connect, check the route table first, then the security group, then the network ACL.

**Launching with the AWS CLI**

For CLI-based labs, you usually launch into a subnet that already has the correct route table association. Public IPv4 assignment is controlled by the subnet setting or by specifying a network interface with `AssociatePublicIpAddress`.

```bash
aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --subnet-id subnet-0123456789abcdef0 \
  --security-group-ids sg-0123456789abcdef0 \
  --key-name my-key-pair \
  --count 1
```

If you need to explicitly request a public IPv4 address at launch, use a network interface specification:

```bash
aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --network-interfaces '[
    {
      "DeviceIndex": 0,
      "SubnetId": "subnet-0123456789abcdef0",
      "Groups": ["sg-0123456789abcdef0"],
      "AssociatePublicIpAddress": true
    }
  ]' \
  --key-name my-key-pair
```

Useful verification commands:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].{State:State.Name,PublicIp:PublicIpAddress,SubnetId:SubnetId,PrivateIp:PrivateIpAddress}'
```

```bash
aws ec2 describe-route-tables \
  --filters Name=association.subnet-id,Values=subnet-0123456789abcdef0 \
  --query 'RouteTables[0].Routes'
```

**Connecting to the instance**

For Linux instances, SSH is the usual method.

```bash
ssh -i my-key-pair.pem ec2-user@<public-ip>
```

Common default usernames vary by AMI:

- Amazon Linux: `ec2-user`
- Ubuntu: `ubuntu`
- Debian: `admin` or `debian`, depending on image
- RHEL: `ec2-user`
- CentOS: `centos`

If SSH fails, do not assume the instance lacks internet access. SSH failure can come from:

- Wrong username.
- Incorrect key file permissions.
- Security group missing inbound port `22`.
- NACL blocking ephemeral return traffic.
- Wrong public IP.
- Instance not yet in `running` state.

**Verifying internet access from inside the instance**

Use package updates or HTTP-based checks rather than relying only on ICMP. `ping` can be blocked by security groups, NACLs, or remote hosts, so it is not a definitive test.

```bash
curl -I https://aws.amazon.com
```

```bash
sudo yum update -y
```

```bash
sudo dnf update -y
```

```bash
sudo apt-get update
```

You can also inspect the instance metadata service to confirm local networking details.

```bash
curl http://169.254.169.254/latest/meta-data/public-ipv4
curl http://169.254.169.254/latest/meta-data/local-ipv4
```

If the instance has a public IP and can reach external HTTPS endpoints, the subnet routing and security rules are usually correct.

**Security group design**

Security groups are stateful, so return traffic is automatically allowed. For a lab instance, a minimal inbound rule set might look like this:

```text
Type        Protocol  Port Range  Source
SSH         TCP       22          your-public-ip/32
HTTP        TCP       80          0.0.0.0/0
HTTPS       TCP       443         0.0.0.0/0
```

- **Stateful** means return traffic is allowed automatically.
- Restrict SSH to your own IP whenever possible.
- Open HTTP or HTTPS only if the instance is serving web traffic.

For a web server test, you can install and start a simple server:

```bash
sudo yum install -y httpd
sudo systemctl enable --now httpd
echo "hello from public subnet" | sudo tee /var/www/html/index.html
```

Then test from your local machine:

```bash
curl http://<public-ip>
```

**Network ACL considerations**

Network ACLs are stateless, so you must allow both directions explicitly. If you customize the NACL, ensure it allows:

- Inbound SSH or web ports as needed.
- Outbound ephemeral ports for return traffic.
- Inbound ephemeral ports for responses if you use strict rules.

A restrictive NACL can cause confusing symptoms where the security group looks correct but connections still fail.

**Troubleshooting checklist**

Use a layered approach when something does not work:

1. Confirm the instance is `running` and has passed status checks.
2. Confirm the subnet route table has `0.0.0.0/0` to the Internet Gateway.
3. Confirm the Internet Gateway is attached to the VPC.
4. Confirm the instance has a public IPv4 address or Elastic IP.
5. Confirm the security group allows the required inbound traffic.
6. Confirm the NACL allows the traffic in both directions.
7. Confirm the OS firewall is not blocking the service.
8. Confirm the service is actually listening on the expected port.

Useful commands on the instance:

```bash
ip addr
ip route
ss -tulpn
sudo systemctl status sshd
sudo systemctl status httpd
```

Useful AWS CLI checks:

```bash
aws ec2 describe-instances --instance-ids i-0123456789abcdef0
aws ec2 describe-security-groups --group-ids sg-0123456789abcdef0
aws ec2 describe-network-acls --filters Name=association.subnet-id,Values=subnet-0123456789abcdef0
```

**Exam-focused takeaways**

- A public subnet is defined by its route table, not by its name.
- An Internet Gateway must be attached to the VPC before it can be used in routes.
- A public IPv4 address alone does not guarantee internet access.
- Security groups are stateful; NACLs are stateless.
- For CLI launches, use subnet auto-assign public IPv4 or a network interface with `AssociatePublicIpAddress`.
- Use HTTP or package update tests to validate internet access more reliably than `ping`.

**Common lab pattern**

A simple public-subnet lab often follows this topology:

```text
Internet
   |
[Internet Gateway]
   |
[VPC Route Table: 0.0.0.0/0 -> IGW]
   |
[Public Subnet]
   |
[EC2 Instance with Public IPv4]
```

This pattern is useful for learning basic connectivity, hosting a temporary web server, or creating a bastion host for private subnet access.
