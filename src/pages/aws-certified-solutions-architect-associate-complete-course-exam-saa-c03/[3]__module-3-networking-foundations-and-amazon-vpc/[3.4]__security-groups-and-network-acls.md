## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Overview**

Amazon Virtual Private Cloud (<abbr title="Virtual Private Cloud">VPC</abbr>) uses **security groups** and **network ACLs** (<abbr title="Network Access Control Lists">NACLs</abbr>) to control traffic at different layers. For the exam, the key distinction is simple: security groups are **stateful** and attached to resources through elastic network interfaces (<abbr title="Elastic Network Interface">ENIs</abbr>), while network ACLs are **stateless** and apply at the subnet boundary.

Security groups are usually your primary control for instance-level access. Network ACLs are a broader subnet-level guardrail that can explicitly allow or deny traffic. In real architectures, you often use both: security groups for fine-grained application access and NACLs for coarse subnet filtering or compliance requirements.

**Security Groups**

A security group acts as a virtual firewall for an ENI. Because it is associated with the ENI, the same security group can protect multiple resources if those resources use ENIs that reference it. This is a common exam detail: the security group is not attached directly to the EC2 instance in a conceptual sense; it is enforced on the network interface attached to the instance.

Security groups are **stateful**, which means return traffic is automatically allowed if the initial request is allowed. You do not need a separate inbound rule for the response path.

- **Default behavior**
  - The default security group allows inbound traffic only from itself.
  - The default security group allows all outbound traffic.
  - This matters when you launch multiple instances into the same default security group and expect them to communicate with each other.
- **Rule behavior**
  - Security groups support **allow rules only**; there are no explicit deny rules.
  - Rules are evaluated as a set, and if any rule matches, traffic is allowed.
  - You can reference another security group as a source or destination, which is useful for tier-to-tier application design.
- **Common use cases**
  - Allow SSH from a trusted admin IP range.
  - Allow HTTP/HTTPS from the internet to a public web tier.
  - Allow app tier access only from the web tier security group.
  - Allow database access only from the app tier security group.

A typical web tier security group might look like this:

```text
Inbound:
- TCP 80 from 0.0.0.0/0
- TCP 443 from 0.0.0.0/0
- TCP 22 from 203.0.113.10/32

Outbound:
- All traffic to 0.0.0.0/0
```

For a private application tier, you often restrict inbound to the web tier security group rather than to an IP range:

```text
Inbound:
- TCP 8080 from sg-web-tier
- TCP 443 from sg-alb

Outbound:
- TCP 3306 to sg-rds
- All traffic to 0.0.0.0/0 if the instance needs updates or external APIs
```

**Network ACLs**

A network ACL is a subnet-level firewall. It is **stateless**, so you must explicitly allow both directions of a flow if you want the traffic to work. If inbound traffic is allowed, the return traffic must also be allowed by outbound rules, and vice versa.

NACLs support both **allow** and **deny** rules. Rules are processed in order from the lowest numbered rule to the highest numbered rule, and the first match wins. This makes rule numbering important.

- **Key properties**
  - Associated with a subnet, not an individual instance.
  - Stateless, so request and response paths are evaluated separately.
  - Supports explicit deny rules.
  - Uses rule numbers to determine evaluation order.
- **Default NACL behavior**
  - The default NACL allows all inbound traffic.
  - The default NACL allows all outbound traffic.
  - Custom NACLs start with a deny-all posture until you add rules.
- **Typical use cases**
  - Add a subnet-wide deny for a known malicious CIDR block.
  - Enforce a coarse compliance boundary around a subnet.
  - Provide an additional layer of defense for public subnets.

A NACL rule set for a public web subnet might look like this:

```text
Inbound:
100 ALLOW TCP 80 from 0.0.0.0/0
110 ALLOW TCP 443 from 0.0.0.0/0
120 ALLOW TCP 1024-65535 from 0.0.0.0/0
* DENY ALL

Outbound:
100 ALLOW TCP 80 to 0.0.0.0/0
110 ALLOW TCP 443 to 0.0.0.0/0
120 ALLOW TCP 1024-65535 to 0.0.0.0/0
* DENY ALL
```

The ephemeral port range is especially important for exam troubleshooting. AWS exam questions commonly expect you to remember **1024-65535** as the typical ephemeral range for return traffic in NACL scenarios. Because NACLs are stateless, you may need inbound and outbound rules for these ports depending on the direction of the connection and which side initiates it.

> If a connection works at the security group layer but fails at the subnet layer, check the NACL first. Stateless filtering often breaks return traffic, especially when ephemeral ports are missing.

**Security Groups vs Network ACLs**

| Feature | Security Group | Network ACL |
|---|---|---|
| Scope | ENI / resource level | Subnet level |
| State | Stateful | Stateless |
| Rules | Allow only | Allow and deny |
| Evaluation | All rules are evaluated together | Ordered by rule number |
| Default behavior | Inbound only from itself, outbound all allowed | Allow all inbound and outbound |
| Typical use | Instance or service access control | Subnet guardrail and explicit deny |

A useful exam shortcut is to remember that security groups answer the question, “Should this resource accept the connection?” while NACLs answer, “Should this subnet permit this packet in this direction?”

**Traffic Flow Examples**

A common architecture is a three-tier application with a public web subnet, a private app subnet, and a private database subnet.

- **Web tier**
  - Security group allows inbound 80/443 from the internet.
  - Outbound allows traffic to the app tier security group.
- **App tier**
  - Security group allows inbound from the web tier security group.
  - Outbound allows traffic to the database security group on the database port.
- **Database tier**
  - Security group allows inbound only from the app tier security group.
  - Outbound is usually minimal, depending on patching, backups, or replication needs.

A practical NACL design for the same flow is more coarse. For example, if the web subnet must accept HTTPS from the internet and return responses, the NACL needs both the service port and the ephemeral return ports.

```text
Web subnet NACL
Inbound:
100 ALLOW TCP 443 from 0.0.0.0/0
110 ALLOW TCP 1024-65535 from 0.0.0.0/0

Outbound:
100 ALLOW TCP 1024-65535 to 0.0.0.0/0
110 ALLOW TCP 443 to 0.0.0.0/0
```

For an app tier talking to Amazon RDS, the security group usually controls the database port, while the NACL must still permit the subnet traffic in both directions.

```text
App subnet NACL
Inbound:
100 ALLOW TCP 8080 from web-subnet-cidr
110 ALLOW TCP 1024-65535 from db-subnet-cidr

Outbound:
100 ALLOW TCP 3306 to db-subnet-cidr
110 ALLOW TCP 1024-65535 to web-subnet-cidr
```

This example is intentionally broad to show the concept: with NACLs, you must think about the full packet path, not just the application port.

**Troubleshooting and Diagnostics**

When traffic is blocked, isolate the layer that is failing. Security groups and NACLs are common exam distractors because both can block traffic, but they fail differently.

- **Security group troubleshooting**
  - Check whether the source is allowed by CIDR or by security group reference.
  - Confirm the correct port and protocol.
  - Verify the resource is using the expected ENI and security group.
- **NACL troubleshooting**
  - Check rule order and rule numbers.
  - Confirm both inbound and outbound rules exist for the flow.
  - Verify ephemeral ports are allowed for return traffic.
  - Look for an explicit deny that matches before an allow.
- **Helpful tools**
  - **VPC Flow Logs** show accepted and rejected traffic at the ENI, subnet, or VPC level depending on configuration.
  - Flow logs are useful when you need to confirm whether traffic is being dropped by a security group or NACL.
  - Reachability Analyzer can help trace network paths and identify where connectivity breaks.

A simple flow log query pattern can help you spot rejects:

```text
fields @timestamp, srcAddr, dstAddr, srcPort, dstPort, action, pktSrcaddr, pktDstaddr
| filter action = "REJECT"
| sort @timestamp desc
| limit 20
```

If you see `REJECT` entries for the expected port, compare the source and destination subnet rules, then check whether the return path is missing ephemeral port access.

**Practical CLI Examples**

You can inspect security groups and NACLs with the AWS CLI (<abbr title="Command Line Interface">CLI</abbr>).

```bash
aws ec2 describe-security-groups --group-ids sg-0123456789abcdef0
aws ec2 describe-network-acls --filters Name=vpc-id,Values=vpc-0123456789abcdef0
aws ec2 describe-network-interfaces --network-interface-ids eni-0123456789abcdef0
```

To create a security group rule that allows HTTPS from the internet:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 443 \
  --cidr 0.0.0.0/0
```

To add a NACL rule allowing HTTPS inbound to a subnet:

```bash
aws ec2 create-network-acl-entry \
  --network-acl-id acl-0123456789abcdef0 \
  --ingress \
  --rule-number 100 \
  --protocol tcp \
  --rule-action allow \
  --cidr-block 0.0.0.0/0 \
  --port-range From=443,To=443
```

To allow ephemeral return traffic in a NACL:

```bash
aws ec2 create-network-acl-entry \
  --network-acl-id acl-0123456789abcdef0 \
  --egress \
  --rule-number 110 \
  --protocol tcp \
  --rule-action allow \
  --cidr-block 0.0.0.0/0 \
  --port-range From=1024,To=65535
```

**Exam Tips**

- **Security groups**
  - Think stateful.
  - Remember inbound can reference another security group.
  - Remember the default SG allows inbound only from itself and all outbound.
- **NACLs**
  - Think stateless.
  - Remember ordered rules and explicit deny.
  - Remember ephemeral ports, commonly `1024-65535`, for return traffic.
- **Architecture**
  - Use security groups for fine-grained access control.
  - Use NACLs for subnet-level filtering or explicit deny requirements.
- **Troubleshooting**
  - If a connection fails intermittently or only in one direction, suspect NACLs.
  - If a resource should not be reachable at all, check both SG and NACL, then confirm route tables and service endpoints if relevant.

**Quick Mental Model**

- **Security group** = resource firewall, stateful, allow-only, attached to ENIs.
- **Network ACL** = subnet firewall, stateless, allow and deny, ordered rules.
- **Default SG** = inbound only from itself, outbound all allowed.
- **Default NACL** = allow all.
- **Custom NACL** = deny all until you add rules.
