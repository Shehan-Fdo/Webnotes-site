## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab objective and routing model**

This lab focuses on verifying how Amazon Virtual Private Cloud (Amazon VPC) routing works from both the AWS Management Console and the AWS Command Line Interface (AWS CLI). You confirm which route table a subnet uses, inspect route entries, and validate that traffic follows the most specific matching route.

In AWS, routing is controlled by route tables associated with subnets. A subnet can use only one route table at a time, but a route table can be associated with multiple subnets. When you verify routing, you are checking both the association and the destination-to-target mapping.

- **Route table association** determines which routes apply to a subnet.
- **Longest prefix match** determines which route entry wins when multiple routes match a destination.
- **Implicit local route** enables communication inside the VPC CIDR block.
- **Target type** determines where traffic goes, such as an Internet Gateway (IGW), NAT Gateway, Transit Gateway, VPC peering connection, or virtual private gateway.

**Core concepts to verify**

A route table contains destination CIDR blocks and targets. The destination can be a specific subnet, a VPC CIDR, a peered VPC CIDR, or `0.0.0.0/0` for default IPv4 internet-bound traffic. For IPv6, the default route is `::/0`.

<dl>
<dt><strong>Destination</strong></dt>
<dd>The CIDR block or prefix that the route matches.</dd>
<dt><strong>Target</strong></dt>
<dd>The AWS resource that receives matching traffic.</dd>
<dt><strong>Main route table</strong></dt>
<dd>The default route table automatically associated with subnets that do not have an explicit association.</dd>
<dt><strong>Local route</strong></dt>
<dd>The automatically created route that allows traffic within the VPC CIDR range.</dd>
</dl>

A common exam scenario is to determine why a subnet cannot reach the internet, another VPC, or an on-premises network. In many cases, the issue is not the instance security group or network ACL; it is the route table association or a missing route entry.

**Verify routing in the AWS Management Console**

Use the console when you want a visual check of subnet associations and route targets. The console is useful for quickly confirming whether the correct route table is attached to the subnet and whether the expected route exists.

1. Open the **VPC** console.
2. Select **Route tables**.
3. Identify the route table associated with the subnet.
4. Review the **Routes** tab for destination and target values.
5. Review the **Subnet associations** tab to confirm which subnets use the table.
6. If needed, compare the route table with the subnet’s CIDR and the instance’s placement.

A practical console check often looks like this:

- Confirm the subnet is associated with the intended route table.
- Confirm `0.0.0.0/0` points to an IGW for public subnets.
- Confirm `0.0.0.0/0` points to a NAT Gateway for private subnets that need outbound internet access.
- Confirm the local route exists for the VPC CIDR.
- Confirm any peering or Transit Gateway routes point to the correct attachment.

**Verify routing with the AWS CLI**

The CLI is the fastest way to inspect route tables in a repeatable lab or exam environment. You can query route tables, associations, and routes directly.

```bash
aws ec2 describe-route-tables
```

To narrow the output to a specific VPC:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

To inspect a specific route table:

```bash
aws ec2 describe-route-tables \
  --route-table-ids rtb-0123456789abcdef0
```

To view only the routes in a compact format:

```bash
aws ec2 describe-route-tables \
  --route-table-ids rtb-0123456789abcdef0 \
  --query 'RouteTables[0].Routes'
```

To view subnet associations:

```bash
aws ec2 describe-route-tables \
  --route-table-ids rtb-0123456789abcdef0 \
  --query 'RouteTables[0].Associations'
```

To find the route table associated with a subnet:

```bash
aws ec2 describe-route-tables \
  --filters Name=association.subnet-id,Values=subnet-0123456789abcdef0
```

To identify the main route table for a VPC:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'RouteTables[?Associations[?Main==`true`]]'
```

**Useful CLI fields to inspect**

When you read the output, focus on these fields:

- `RouteTableId` for the route table identifier.
- `VpcId` for the VPC scope.
- `Associations` for subnet bindings and main table status.
- `Routes` for destination and target pairs.
- `State` for route status, which is usually `active`.
- `Origin` for whether the route is `CreateRouteTable`, `CreateRoute`, or `EnableVgwRoutePropagation`.

**Typical route targets and when to use them**

| Target | Use case | Notes |
|---|---|---|
| `igw-...` | Public subnet internet access | Requires an attached Internet Gateway |
| `nat-...` | Private subnet outbound internet access | Supports outbound-only IPv4 traffic |
| `vgw-...` | On-premises connectivity via Site-to-Site VPN | Often used with route propagation |
| `tgw-...` | Hub-and-spoke or multi-VPC routing | Centralized routing through Transit Gateway |
| `pcx-...` | VPC peering | No transitive routing through the peer |
| `eni-...` | Appliance or network virtual appliance routing | Used for custom inspection paths |
| `local` | Intra-VPC traffic | Automatically present and not removable |

**Example route table output interpretation**

A public subnet route table often includes a default route to an Internet Gateway:

```json
{
  "RouteTableId": "rtb-0123456789abcdef0",
  "VpcId": "vpc-0123456789abcdef0",
  "Routes": [
    {
      "DestinationCidrBlock": "10.0.0.0/16",
      "GatewayId": "local",
      "State": "active",
      "Origin": "CreateRouteTable"
    },
    {
      "DestinationCidrBlock": "0.0.0.0/0",
      "GatewayId": "igw-0abc123def4567890",
      "State": "active",
      "Origin": "CreateRoute"
    }
  ]
}
```

This means:

- Traffic inside `10.0.0.0/16` stays inside the VPC.
- All other IPv4 traffic goes to the Internet Gateway.
- Instances in the subnet still need a public IPv4 address or Elastic IP to be reachable from the internet.

A private subnet route table often looks like this:

```json
{
  "RouteTableId": "rtb-0fedcba9876543210",
  "Routes": [
    {
      "DestinationCidrBlock": "10.0.0.0/16",
      "GatewayId": "local",
      "State": "active"
    },
    {
      "DestinationCidrBlock": "0.0.0.0/0",
      "NatGatewayId": "nat-0123abc456def7890",
      "State": "active"
    }
  ]
}
```

This means:

- The subnet can reach internal VPC resources.
- Outbound internet traffic is translated through the NAT Gateway.
- Inbound internet connections are not allowed through the NAT path.

**How AWS chooses a route**

AWS uses the most specific matching route. If multiple routes match a destination, the route with the longest prefix wins.

- `10.0.0.0/16` matches a broad internal range.
- `10.0.1.0/24` is more specific and overrides the broader route for that subnet range.
- `0.0.0.0/0` is the least specific IPv4 default route.

This matters when you add more specific routes for inspection, peering, or service insertion. If a more specific route exists, it takes precedence over the default route.

> Be careful when testing connectivity: a route table can be correct and the instance can still fail to connect because of security groups, network ACLs, missing public IPs, or disabled source/destination checks on appliances.

**Practical verification workflow**

Use a repeatable sequence to isolate routing issues.

1. Identify the subnet where the instance runs.
2. Find the route table associated with that subnet.
3. Confirm the route table contains the expected destination and target.
4. Confirm the target resource exists and is attached or available.
5. Confirm the route is active.
6. Confirm the instance has the correct addressing for the path you expect.
7. Test connectivity from the instance or from a reachable host.

A simple connectivity test from an EC2 instance might use:

```bash
curl -I https://example.com
ping -c 4 8.8.8.8
traceroute 8.8.8.8
```

Use these carefully:

- `curl` is useful for HTTP and HTTPS reachability.
- `ping` may fail even when routing is correct because ICMP can be blocked.
- `traceroute` can help reveal the next hop, but AWS-managed paths may not show every hop.

**Common lab checks and what they mean**

- If a public subnet cannot reach the internet, verify:
  - The subnet route table has `0.0.0.0/0` to an IGW.
  - The instance has a public IPv4 address or Elastic IP.
  - The IGW is attached to the VPC.
- If a private subnet cannot reach the internet, verify:
  - The subnet route table has `0.0.0.0/0` to a NAT Gateway.
  - The NAT Gateway is in a public subnet.
  - The public subnet has a route to the IGW.
- If traffic to another VPC fails, verify:
  - Both VPCs have non-overlapping CIDR blocks.
  - The route table has a route to the peering connection or Transit Gateway.
  - The peer route table also has a return route.
- If on-premises connectivity fails, verify:
  - The route table has a route to the VPN or Direct Connect path.
  - Route propagation is enabled if you rely on propagated routes.
  - The on-premises side has return routes.

**Route propagation and static routes**

Some AWS networking services can propagate routes into a route table. This is common with virtual private gateways and Transit Gateway attachments, depending on configuration. Propagated routes reduce manual updates, but you still need to verify that propagation is enabled and that the resulting routes are present.

Static routes are manually created and are useful when you need explicit control. In exam scenarios, static routes often override broader propagated paths if they are more specific.

**Example: checking propagated routes**

```bash
aws ec2 describe-route-tables \
  --route-table-ids rtb-0123456789abcdef0 \
  --query 'RouteTables[0].Routes[?Origin==`EnableVgwRoutePropagation`]'
```

This helps you distinguish between routes you created and routes learned through propagation.

**Troubleshooting patterns**

If a route is missing, the issue is usually one of these:

- The route table is not associated with the subnet.
- The target resource does not exist or is not attached.
- The route was created in the wrong VPC.
- The destination CIDR overlaps with another more specific route.
- The path requires a return route that is missing on the other side.

If the route exists but traffic still fails, check the layers around routing:

- Security groups for stateful instance-level filtering.
- Network ACLs for stateless subnet-level filtering.
- Instance OS firewall rules.
- Public IP assignment for internet-facing workloads.
- Source/destination check for routed appliances.

**Example: public subnet verification checklist**

```text
Subnet: 10.0.1.0/24
Route table:
- 10.0.0.0/16 -> local
- 0.0.0.0/0 -> igw-0abc123def4567890

Instance:
- Public IPv4: assigned
- Security group: inbound SSH allowed from your IP
- NACL: inbound and outbound allowed for required ports
```

This configuration supports inbound SSH from the internet if the instance also has a public address and the security group allows it.

**Example: private subnet verification checklist**

```text
Subnet: 10.0.2.0/24
Route table:
- 10.0.0.0/16 -> local
- 0.0.0.0/0 -> nat-0123abc456def7890

Instance:
- No public IPv4
- Security group: outbound allowed
- NAT Gateway: in a public subnet with IGW access
```

This configuration supports outbound internet access for patching, package downloads, and API calls without exposing the instance directly.

**Exam-relevant takeaways**

- **Subnet association** is the first thing to verify when routing seems wrong.
- **Longest prefix match** determines the effective route.
- **Local routes** are always present for the VPC CIDR.
- **Public subnets** need an IGW route and a public address on the instance.
- **Private subnets** need a NAT Gateway for outbound internet access.
- **Return routes** are required for peering, VPN, and Transit Gateway scenarios.
- **Routing alone** does not guarantee connectivity; security groups and NACLs still apply.

**Quick CLI reference**

```bash
# List route tables in a VPC
aws ec2 describe-route-tables --filters Name=vpc-id,Values=vpc-0123456789abcdef0

# Find the route table for a subnet
aws ec2 describe-route-tables --filters Name=association.subnet-id,Values=subnet-0123456789abcdef0

# Show routes only
aws ec2 describe-route-tables --route-table-ids rtb-0123456789abcdef0 --query 'RouteTables[0].Routes'

# Show subnet associations only
aws ec2 describe-route-tables --route-table-ids rtb-0123456789abcdef0 --query 'RouteTables[0].Associations'
```

**Mental model for the lab**

Think of routing as a two-part check:

- First, determine which route table the subnet uses.
- Second, determine which route entry matches the destination.

If both are correct, move on to the next layer of troubleshooting. If either is wrong, fix the association or the route target before testing again.
