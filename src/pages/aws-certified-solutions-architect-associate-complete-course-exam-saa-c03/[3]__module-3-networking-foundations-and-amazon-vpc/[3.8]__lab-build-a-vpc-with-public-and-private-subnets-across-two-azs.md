## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab overview**

This lab builds a foundational Amazon Virtual Private Cloud (VPC) architecture with public and private subnets spread across two Availability Zones (AZs). The goal is to create a network layout that supports internet-facing workloads in public subnets while keeping application or data-tier resources isolated in private subnets.

A well-designed two-AZ VPC improves availability and fault tolerance. If one AZ becomes unavailable, you can still place resources in the other AZ. For the SAA-C03 exam, you should understand not only how to create the components, but also why each routing and security choice matters.

- **VPC**: Your isolated network boundary in AWS.
- **Public subnet**: A subnet with a route to an Internet Gateway (IGW).
- **Private subnet**: A subnet without direct internet routing.
- **Availability Zone**: A physically separate location within an AWS Region.

**Target architecture**

A common lab topology looks like this:

- One VPC with a CIDR block such as `10.0.0.0/16`
- Two public subnets, one in each AZ
- Two private subnets, one in each AZ
- One Internet Gateway attached to the VPC
- One public route table associated with both public subnets
- One private route table associated with both private subnets
- Optional NAT Gateway in a public subnet for outbound internet access from private subnets

```text
VPC 10.0.0.0/16
├── Public Subnet A 10.0.1.0/24  (AZ-a) -> IGW
├── Public Subnet B 10.0.2.0/24  (AZ-b) -> IGW
├── Private Subnet A 10.0.11.0/24 (AZ-a) -> NAT Gateway -> IGW
└── Private Subnet B 10.0.12.0/24 (AZ-b) -> NAT Gateway -> IGW
```

**Core design principles**

The main purpose of this layout is separation of concerns. Public subnets host resources that must be reachable from the internet, such as load balancers or bastion hosts. Private subnets host application servers, databases, or internal services that should not accept direct inbound traffic from the internet.

- **Routing** determines whether traffic can leave a subnet.
- **Security groups** control instance-level traffic.
- **Network ACLs** provide optional subnet-level stateless filtering.
- **High availability** comes from distributing subnets across AZs.

> A subnet is considered public only when its route table includes a route to an Internet Gateway. Merely naming it “public” does not make it public.

**Step-by-step build process**

Use the AWS Management Console, AWS CLI, or Infrastructure as Code. The console is common for labs, but the CLI helps you verify the underlying resources and is useful for exam practice.

1. Create the VPC.
   - Choose a CIDR block large enough for future growth, such as `10.0.0.0/16`.
   - Enable DNS hostnames and DNS resolution so instances can use AWS-provided DNS names.

2. Create two public subnets.
   - Place one subnet in each AZ.
   - Use non-overlapping CIDR blocks, such as `10.0.1.0/24` and `10.0.2.0/24`.

3. Create two private subnets.
   - Place one subnet in each AZ.
   - Use separate CIDR blocks, such as `10.0.11.0/24` and `10.0.12.0/24`.

4. Create and attach an Internet Gateway.
   - Attach the IGW to the VPC.
   - Without this attachment, public subnet internet routing will not work.

5. Create route tables.
   - Associate the public route table with both public subnets.
   - Associate the private route table with both private subnets.

6. Add routes.
   - Public route table: add `0.0.0.0/0` to the IGW.
   - Private route table: add `0.0.0.0/0` to a NAT Gateway if private instances need outbound internet access.

7. Launch test instances.
   - Place a test instance in a public subnet and assign a public IPv4 address.
   - Place another test instance in a private subnet without a public IPv4 address.

8. Validate connectivity.
   - Confirm the public instance is reachable according to its security group rules.
   - Confirm the private instance is not directly reachable from the internet.
   - If NAT is configured, confirm the private instance can reach outbound endpoints such as package repositories.

**AWS CLI example**

The CLI is useful for understanding the exact resources involved. The following commands show the basic flow.

```bash
# Create the VPC
aws ec2 create-vpc \
  --cidr-block 10.0.0.0/16

# Enable DNS support and hostnames
aws ec2 modify-vpc-attribute \
  --vpc-id vpc-xxxxxxxx \
  --enable-dns-support
aws ec2 modify-vpc-attribute \
  --vpc-id vpc-xxxxxxxx \
  --enable-dns-hostnames

# Create subnets
aws ec2 create-subnet \
  --vpc-id vpc-xxxxxxxx \
  --cidr-block 10.0.1.0/24 \
  --availability-zone us-east-1a

aws ec2 create-subnet \
  --vpc-id vpc-xxxxxxxx \
  --cidr-block 10.0.2.0/24 \
  --availability-zone us-east-1b

aws ec2 create-subnet \
  --vpc-id vpc-xxxxxxxx \
  --cidr-block 10.0.11.0/24 \
  --availability-zone us-east-1a

aws ec2 create-subnet \
  --vpc-id vpc-xxxxxxxx \
  --cidr-block 10.0.12.0/24 \
  --availability-zone us-east-1b
```

**Internet Gateway and routing**

The Internet Gateway is the AWS-managed component that enables communication between resources in your VPC and the internet. It is horizontally scaled and highly available by design, but it must be attached to the VPC and referenced in a route table.

A public subnet typically has these characteristics:

- A route table entry for `0.0.0.0/0` pointing to the IGW
- Instances with public IPv4 addresses or Elastic IPs
- Security group rules that allow the required inbound traffic

A private subnet typically has these characteristics:

- No direct route to the IGW
- No public IPv4 address on instances
- Optional outbound-only access through a NAT Gateway

```bash
# Create and attach an Internet Gateway
aws ec2 create-internet-gateway
aws ec2 attach-internet-gateway \
  --internet-gateway-id igw-xxxxxxxx \
  --vpc-id vpc-xxxxxxxx

# Create a public route table and add the default route
aws ec2 create-route-table --vpc-id vpc-xxxxxxxx
aws ec2 create-route \
  --route-table-id rtb-xxxxxxxx \
  --destination-cidr-block 0.0.0.0/0 \
  --gateway-id igw-xxxxxxxx
```

**NAT Gateway for private subnets**

A NAT Gateway allows instances in private subnets to initiate outbound connections to the internet while preventing unsolicited inbound connections from the internet. This is common for patching, package downloads, and external API calls.

Important points:

- Place the NAT Gateway in a public subnet.
- Allocate an Elastic IP for the NAT Gateway.
- Update the private route table to send `0.0.0.0/0` to the NAT Gateway.
- For high availability, deploy one NAT Gateway per AZ and route each private subnet to the NAT Gateway in the same AZ.

> A NAT Gateway is for outbound internet access only. It does not make a private subnet publicly reachable.

```bash
# Allocate an Elastic IP for the NAT Gateway
aws ec2 allocate-address --domain vpc

# Create the NAT Gateway in a public subnet
aws ec2 create-nat-gateway \
  --subnet-id subnet-public-a \
  --allocation-id eipalloc-xxxxxxxx

# Add a default route in the private route table
aws ec2 create-route \
  --route-table-id rtb-private-xxxxxxxx \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-xxxxxxxx
```

**Security groups and network ACLs**

Routing alone does not control access. Security groups and network ACLs add the traffic controls that make the design secure.

- **Security groups**
  - Stateful
  - Attached to ENIs (Elastic Network Interfaces)
  - Commonly used as the primary access control
  - Return traffic is automatically allowed

- **Network ACLs**
  - Stateless
  - Applied at the subnet level
  - Require explicit inbound and outbound rules
  - Useful for coarse-grained subnet filtering

For a typical lab:

- Public instance security group:
  - Allow SSH (`22`) from your IP only, or use Session Manager instead of SSH
  - Allow HTTP (`80`) and HTTPS (`443`) if hosting a web app

- Private instance security group:
  - Allow application ports only from the public tier or load balancer security group
  - Do not allow direct inbound from `0.0.0.0/0`

```text
Example security group rules

Public SG:
- Inbound SSH 22 from 203.0.113.10/32
- Inbound HTTP 80 from 0.0.0.0/0
- Inbound HTTPS 443 from 0.0.0.0/0

Private SG:
- Inbound app port 8080 from Public SG
- No public inbound rules
```

**Verification and troubleshooting**

After deployment, verify each layer separately. This helps you isolate whether a problem is caused by routing, addressing, or security rules.

- Confirm the subnet associations on each route table.
- Confirm the public route table has `0.0.0.0/0 -> igw-...`.
- Confirm the private route table has either no default route or a default route to `nat-...`.
- Confirm the public instance has a public IPv4 address if you expect direct internet access.
- Confirm the security group allows the traffic you are testing.
- Confirm the instance OS firewall is not blocking the port.

Useful CLI checks:

```bash
aws ec2 describe-route-tables \
  --route-table-ids rtb-xxxxxxxx

aws ec2 describe-subnets \
  --subnet-ids subnet-xxxxxxxx subnet-yyyyyyyy

aws ec2 describe-instances \
  --instance-ids i-xxxxxxxx
```

If a public instance is unreachable, check these common causes:

- The subnet is not associated with the correct route table.
- The route table does not point `0.0.0.0/0` to the IGW.
- The instance has no public IPv4 address.
- The security group does not allow the inbound port.
- The network ACL blocks the traffic.

If a private instance cannot reach the internet through NAT, check these common causes:

- The NAT Gateway is in a private subnet instead of a public subnet.
- The NAT Gateway does not have an Elastic IP.
- The private route table does not point `0.0.0.0/0` to the NAT Gateway.
- The public subnet route table does not point `0.0.0.0/0` to the IGW.
- The NAT Gateway AZ placement or route design is inconsistent.

**Exam-relevant concepts**

This lab maps directly to several SAA-C03 networking ideas:

- **Subnet design**: Separate public and private tiers for security and clarity.
- **Availability design**: Use multiple AZs to reduce single-AZ dependency.
- **Routing**: Public access requires an IGW route; private access can use NAT.
- **Security layering**: Use security groups for instance access and route tables for path selection.
- **Cost awareness**: NAT Gateways add cost, so use them only when private subnets need outbound internet access.

**Common architecture patterns**

This VPC layout is a building block for many AWS architectures:

- Public subnets for Application Load Balancers
- Private subnets for EC2 application servers
- Private subnets for Amazon RDS databases
- Separate route tables for different tiers
- Multi-AZ deployments for resilience

A practical example is a three-tier web application:

- Web tier in public subnets behind an internet-facing load balancer
- Application tier in private subnets
- Database tier in isolated private subnets with no internet route

**Practical validation example**

After launching instances, you can test connectivity from a public instance and a private instance.

```bash
# On a public instance
curl https://checkip.amazonaws.com

# On a private instance with NAT access
sudo dnf update -y
curl https://example.com
```

If you use SSH for testing, remember that SSH access should be tightly restricted.

```bash
ssh -i my-key.pem ec2-user@<public-ip>
```

For better operational practice, consider AWS Systems Manager Session Manager instead of opening SSH to the internet. That reduces exposure and avoids the need for inbound port 22 in many cases.

**Key takeaways**

The essential lesson of this lab is that subnet behavior is determined by route tables, not by subnet names. Public subnets need an IGW route and public addressing to be reachable from the internet. Private subnets stay isolated and can optionally use a NAT Gateway for outbound-only connectivity. Spreading both subnet types across two AZs gives you a resilient baseline architecture for many AWS workloads.
