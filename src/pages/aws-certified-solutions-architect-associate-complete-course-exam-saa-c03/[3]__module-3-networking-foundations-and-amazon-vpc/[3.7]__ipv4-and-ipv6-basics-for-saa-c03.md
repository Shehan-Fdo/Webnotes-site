## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**IPv4 fundamentals**

IPv4 uses 32-bit addresses written as four decimal octets, such as `192.168.1.10`. Each octet ranges from `0` to `255`, so the total address space is about 4.3 billion addresses. In exam questions, you usually care less about memorizing the full range and more about recognizing whether a block is private, public, network-only, or reserved for special use.

A subnet in IPv4 has two important boundary addresses:

- **Network address**: the first address in the block, where host bits are all `0`
- **Broadcast address**: the last address in the block, where host bits are all `1`

Usable host addresses are the addresses between those two boundaries. For a subnet with prefix length `/n`, the number of total addresses is `2^(32-n)`, and the usable host count is usually `2^(32-n) - 2` because the network and broadcast addresses are reserved.

Quick examples:

- `/24` gives `256` total addresses and `254` usable hosts
- `/25` gives `128` total addresses and `126` usable hosts
- `/26` gives `64` total addresses and `62` usable hosts
- `/28` gives `16` total addresses and `14` usable hosts
- `/30` gives `4` total addresses and `2` usable hosts

In AWS, the first four IP addresses and the last IP address in each subnet are reserved, so the usable count is lower than the simple IPv4 formula suggests. For example, a `/24` subnet in AWS has `251` usable addresses, not `254`.

**CIDR and subnetting**

Classless Inter-Domain Routing (<abbr title="Classless Inter-Domain Routing">CIDR</abbr>) replaces class-based addressing with prefix notation like `10.0.0.0/16`. The prefix length tells you how many bits identify the network, and the remaining bits identify hosts.

To interpret a CIDR block quickly in exam questions:

- A smaller prefix number means a larger network
- A larger prefix number means a smaller network
- `10.0.0.0/16` contains `10.0.0.0` through `10.0.255.255`
- `10.0.1.0/24` is one subnet inside that `/16`
- `10.0.1.128/25` is half of a `/24`

A practical way to think about subnetting is to separate the address into:

- **Network bits**: fixed for the subnet
- **Host bits**: vary within the subnet

For example, `192.168.10.0/26` means the first 26 bits are network bits and the last 6 bits are host bits. That gives `64` total addresses. The block spans `192.168.10.0` to `192.168.10.63`, with `192.168.10.0` as the network address and `192.168.10.63` as the broadcast address.

A few exam-friendly patterns:

- `10.0.0.0/8` is very large and commonly used for private networks
- `172.16.0.0/12` is private, but only the range `172.16.0.0` to `172.31.255.255`
- `192.168.0.0/16` is private and common in home and lab networks
- `169.254.0.0/16` is link-local in IPv4 and often indicates fallback or no DHCP response

> In AWS, you cannot choose arbitrary IPv4 ranges for a VPC. The VPC CIDR must be within RFC 1918 private ranges or other AWS-supported non-overlapping ranges, and it must not overlap with connected networks if you need routing between them.

**IPv4 in AWS VPC design**

A VPC (<abbr title="Virtual Private Cloud">VPC</abbr>) is your isolated network boundary in AWS. You assign one or more IPv4 CIDR blocks to the VPC, then carve those blocks into subnets across Availability Zones.

Typical design considerations:

- Use a large enough VPC CIDR to support future growth
- Split subnets by tier, such as public, private, and database
- Keep subnet sizes aligned with expected instance density
- Avoid overlap with on-premises networks if you plan to connect through VPN or Direct Connect

Example VPC layout:

- VPC: `10.0.0.0/16`
- Public subnet A: `10.0.0.0/24`
- Public subnet B: `10.0.1.0/24`
- Private subnet A: `10.0.10.0/24`
- Private subnet B: `10.0.11.0/24`

This layout is easy to reason about in exam questions because each subnet is a clean `/24`, and the address space leaves room for expansion.

**IPv6 fundamentals**

IPv6 uses 128-bit addresses written in hexadecimal, separated by colons, such as `2001:db8:1234:5678::1`. The address space is enormous, so IPv6 does not rely on broadcast. Instead, it uses multicast and neighbor discovery mechanisms.

IPv6 notation rules matter in exam questions:

- Leading zeros in each hextet can be omitted
- One contiguous run of all-zero hextets can be compressed once using `::`
- You cannot use `::` more than once in the same address
- `2001:0db8:0000:0000:0000:0000:0000:0001` can be written as `2001:db8::1`
- `2001:db8:0:1:0:0:0:1` can be written as `2001:db8:0:1::1`

Common IPv6 scopes:

- **Global unicast**: publicly routable addresses, typically `2000::/3`
- **Unique local**: private-like internal addresses, typically `fc00::/7` and commonly `fd00::/8` in practice
- **Link-local**: local-link communication only, typically `fe80::/10`

Link-local addresses are especially important because many IPv6 systems use them automatically for neighbor discovery and local routing behavior. You often see them on interfaces even when no global address is configured.

IPv6 also has no broadcast address. That difference is useful in exam questions because it changes how you think about subnet behavior and traffic patterns.

**IPv6 in AWS**

AWS commonly assigns IPv6 at the VPC level and then allocates IPv6 ranges to subnets and network interfaces. In practice, you enable IPv6 on the VPC, associate an IPv6 CIDR block, then assign IPv6 addresses to subnets and instances or ENIs (<abbr title="Elastic Network Interface">ENI</abbr>s).

Important AWS IPv6 points:

- IPv6 CIDR blocks for a VPC are typically allocated by AWS
- Subnets receive IPv6 ranges from the VPC allocation
- Instances and ENIs can receive IPv6 addresses from the subnet range
- Security groups and route tables must allow IPv6 traffic explicitly
- There is no IPv6 broadcast concept
- Dual-stack means the workload supports both IPv4 and IPv6

A common exam trap is assuming IPv6 replaces IPv4 everywhere. In AWS, many services and client paths still require IPv4, so dual-stack is often the safest design.

**End-to-end IPv6 enablement example in AWS**

A typical workflow for enabling IPv6 on an existing VPC looks like this:

1. Associate an IPv6 CIDR block with the VPC.
2. Assign an IPv6 CIDR to each subnet.
3. Update the route table with an IPv6 default route.
4. Update security groups and network ACLs if needed.
5. Launch or modify instances to receive IPv6 addresses.
6. Verify DNS, routing, and application reachability.

Console flow:

1. Open the VPC console.
2. Select the VPC.
3. Choose `Actions` > `Edit CIDRs` or `Add IPv6 CIDR`.
4. Associate an Amazon-provided IPv6 block.
5. Edit each subnet and assign an IPv6 subnet CIDR.
6. Enable auto-assign IPv6 if you want new instances to receive addresses automatically.
7. Update the route table with `::/0` to the appropriate egress target.
8. Confirm security group inbound and outbound rules allow the required IPv6 ports.

CLI example:

```bash
# Associate an IPv6 CIDR block with the VPC
aws ec2 associate-vpc-cidr-block \
  --vpc-id vpc-0123456789abcdef0 \
  --amazon-provided-ipv6-cidr-block

# View the assigned IPv6 block
aws ec2 describe-vpcs \
  --vpc-ids vpc-0123456789abcdef0 \
  --query 'Vpcs[0].Ipv6CidrBlockAssociationSet'

# Assign an IPv6 CIDR to a subnet
aws ec2 associate-subnet-cidr-block \
  --subnet-id subnet-0123456789abcdef0 \
  --ipv6-cidr-block 2600:1f18:abcd:1200::/64

# Enable auto-assign IPv6 on the subnet
aws ec2 modify-subnet-attribute \
  --subnet-id subnet-0123456789abcdef0 \
  --assign-ipv6-address-on-creation

# Add an IPv6 default route
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-ipv6-cidr-block ::/0 \
  --gateway-id igw-0123456789abcdef0
```

Security group example for IPv6 web access:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --ip-permissions '[
    {
      "IpProtocol":"tcp",
      "FromPort":443,
      "ToPort":443,
      "Ipv6Ranges":[{"CidrIpv6":"::/0"}]
    }
  ]'
```

A matching security group rule in the console would allow TCP `443` from `::/0` for public HTTPS access. For private workloads, restrict the source to a narrower IPv6 prefix or to another security group where possible.

**Dual-stack, IPv4-only, and IPv6-only design choices**

Dual-stack is the most common AWS exam answer when you need broad compatibility. It lets clients use either protocol and reduces migration risk.

Choose **dual-stack** when:

- You need compatibility with older clients or services
- You are migrating gradually from IPv4 to IPv6
- You want public reachability without breaking existing integrations
- You expect mixed client networks

Choose **IPv6-only** when:

- The workload and dependencies support IPv6 end to end
- You want to reduce IPv4 address pressure
- You are designing for modern client networks and controlled dependencies
- You can use translation or proxy services where IPv4-only dependencies remain

Choose **IPv4-only** when:

- A required service, appliance, or client path does not support IPv6
- A third-party integration is IPv4-only
- You need the simplest compatibility path for a legacy environment

AWS exam scenarios often test tradeoffs like these:

- A public application with modern clients and legacy backend dependencies usually favors dual-stack
- A private service that only talks to IPv4-only databases or third-party APIs may need IPv4 support even if the front end is IPv6-capable
- A new internal service in a controlled environment may be a candidate for IPv6-only if all dependencies support it

**DNS and dual-stack endpoints**

DNS (<abbr title="Domain Name System">DNS</abbr>) is central to IPv4 and IPv6 adoption because clients usually discover addresses through records rather than hardcoding IPs.

Relevant record types:

- `A` record maps a name to an IPv4 address
- `AAAA` record maps a name to an IPv6 address
- `CNAME` points one name to another name
- Route 53 alias records can point to AWS resources without using a fixed IP

In AWS, Route 53 alias records are commonly used with load balancers and other AWS resources. For dual-stack services, the load balancer often exposes both IPv4 and IPv6 connectivity, and DNS returns the appropriate address family based on client capability.

Practical exam pattern:

- A public application behind an Application Load Balancer may use Route 53 alias records
- The load balancer can present dual-stack endpoints
- Clients that support IPv6 can connect over `AAAA`
- Clients that only support IPv4 can connect over `A`

This is why DNS design matters: it lets you support both address families without changing application code in many cases.

**AWS routing and security implications**

IPv6 routing in AWS uses the same conceptual model as IPv4, but the route destinations and security rules must match the protocol family.

Key points:

- IPv4 default route is `0.0.0.0/0`
- IPv6 default route is `::/0`
- Security groups are stateful for both IPv4 and IPv6
- Network ACLs are stateless and need matching inbound and outbound rules
- Internet Gateway supports IPv4 and IPv6 internet access for public subnets
- Egress-only Internet Gateway is used for outbound-only IPv6 internet access from private subnets

A private subnet that needs outbound IPv6 access but no inbound internet exposure is a classic use case for an egress-only Internet Gateway. That is a common exam distinction from the standard Internet Gateway used for public inbound and outbound access.

**Practical subnet interpretation examples**

When you see a CIDR in an exam question, translate it into size and purpose quickly:

- `10.0.0.0/16`
  - Large VPC range
  - Good for multiple tiers and future growth
- `10.0.1.0/24`
  - Standard subnet size
  - Good for a single tier in one Availability Zone
- `10.0.1.128/25`
  - Half of a `/24`
  - Useful when you need smaller segmentation
- `10.0.1.64/26`
  - Smaller subnet with `62` usable IPv4 hosts in general, fewer in AWS due to reserved addresses
- `2600:1f18:abcd:1200::/64`
  - Typical IPv6 subnet size in AWS
  - IPv6 subnets are commonly `/64`

For IPv6, `/64` is the standard subnet size in most practical deployments because many IPv6 mechanisms assume it. In AWS, subnets commonly receive `/64` IPv6 ranges.

**Common exam traps**

- Confusing network address with usable host address
- Forgetting that IPv4 has broadcast but IPv6 does not
- Assuming `::/0` is the IPv6 equivalent of `0.0.0.0/0` for default routing, which it is
- Forgetting to update security groups for IPv6, not just IPv4
- Assuming Route 53 alias records automatically make a service dual-stack
- Assuming all AWS services support IPv6 equally
- Ignoring client compatibility when choosing IPv6-only designs

**CLI verification commands**

Use these commands to inspect addressing and routing on an EC2 instance:

```bash
ip addr show
ip -6 addr show
ip route show
ip -6 route show
curl -4 https://example.com
curl -6 https://example.com
```

On Linux, `curl -4` forces IPv4 and `curl -6` forces IPv6. This is useful when you want to verify whether a service is truly dual-stack.

You can also inspect DNS resolution:

```bash
dig A example.com
dig AAAA example.com
```

If `AAAA` returns an address but `curl -6` fails, the problem is often routing, security groups, or the application path rather than DNS.

**Mental model for SAA-C03 questions**

When you read a networking question, ask yourself:

- Is the workload public or private?
- Does it need IPv4, IPv6, or both?
- Is the subnet size sufficient for the expected number of resources?
- Does the design require inbound internet access or outbound-only access?
- Are DNS and load balancing exposing the right address family?
- Do all dependencies support the chosen protocol?

A strong exam answer usually balances compatibility, scalability, and operational simplicity. In many cases, dual-stack is the safest choice, while IPv6-only is a more specialized optimization when the ecosystem fully supports it.

**Quick reference**

- **IPv4**: 32-bit, dotted decimal, broadcast exists
- **IPv6**: 128-bit, hexadecimal, no broadcast
- **CIDR**: prefix notation that defines network size
- **AWS VPC**: network boundary where you assign CIDR blocks
- **Subnet**: smaller segment inside a VPC
- **Route 53**: DNS service that can publish `A`, `AAAA`, and alias records
- **Dual-stack**: supports IPv4 and IPv6
- **Egress-only Internet Gateway**: outbound-only IPv6 internet access for private subnets
