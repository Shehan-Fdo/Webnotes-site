## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab overview**

This lab focuses on a common Amazon Virtual Private Cloud (Amazon VPC) pattern: placing compute in a **private subnet** while still allowing it to reach the internet for software updates, package downloads, and external APIs. The standard solution is a **NAT Gateway** in a public subnet, with private subnet route tables sending default outbound traffic to that gateway.

The key design goal is one-way internet access:
- Instances in the private subnet can initiate outbound connections.
- External hosts cannot initiate inbound connections to those instances.
- The private subnet remains isolated from direct public exposure.

This pattern appears frequently in AWS Certified Solutions Architect scenarios because it balances security and operational access.

**Core architecture**

A typical deployment includes:
- A **VPC** with at least two Availability Zones for resilience.
- One or more **public subnets** with a route to an Internet Gateway.
- One or more **private subnets** with no direct route to the Internet Gateway.
- A **NAT Gateway** deployed in a public subnet.
- An **Elastic IP address** attached to the NAT Gateway.
- A private subnet route table with a default route (`0.0.0.0/0`) pointing to the NAT Gateway.

A simple traffic flow looks like this:

```text
Private EC2 instance
   -> Private subnet route table
   -> NAT Gateway in public subnet
   -> Internet Gateway
   -> Internet
```

**Why use a NAT Gateway**

Use a NAT Gateway when instances in private subnets need outbound internet access without being reachable from the internet.

Common use cases:
- Operating system patching and package installation.
- Pulling container images from public registries.
- Accessing third-party APIs.
- Downloading application dependencies during bootstrap.

Important characteristics:
- **Managed service**: AWS handles availability and scaling within an Availability Zone.
- **Outbound only**: It supports return traffic for connections initiated from private instances.
- **Public subnet placement**: The NAT Gateway must live in a public subnet with a route to an Internet Gateway.
- **Elastic IP requirement**: The NAT Gateway uses an Elastic IP for internet-bound traffic.

> A NAT Gateway is not the same as a NAT instance. A NAT instance is an EC2-based solution you manage yourself, while a NAT Gateway is the recommended managed option for most modern architectures.

**Subnet and route table design**

The subnet layout determines whether a resource is public or private.

- A **public subnet** has a route table entry for `0.0.0.0/0` that targets an Internet Gateway.
- A **private subnet** does not have a direct route to an Internet Gateway.
- A private subnet can still reach the internet if its route table sends `0.0.0.0/0` to a NAT Gateway.

Example route tables:

| Route table | Destination | Target |
|---|---:|---|
| Public route table | `10.0.0.0/16` | local |
| Public route table | `0.0.0.0/0` | Internet Gateway |
| Private route table | `10.0.0.0/16` | local |
| Private route table | `0.0.0.0/0` | NAT Gateway |

This separation is important because the instance itself does not become public just because it can reach the internet. Public exposure depends on the subnet route, the instance’s public IP assignment, and security group and network ACL rules.

**Security groups and network ACLs**

Security groups and network ACLs both affect connectivity, but they operate differently.

- **Security groups** are stateful.
  - If you allow outbound traffic, return traffic is automatically allowed.
  - For private instances, outbound rules often allow all traffic or specific destinations as needed.
- **Network ACLs** are stateless.
  - You must allow both outbound and inbound ephemeral ports if you use restrictive ACLs.
  - Misconfigured ACLs are a common cause of failed outbound connectivity.

For a basic lab setup:
- Keep the instance security group simple.
- Allow outbound traffic to the internet.
- Ensure the subnet network ACL does not block ephemeral return traffic.

**Launching a workload in the private subnet**

A private workload is usually an EC2 instance launched without a public IPv4 address.

Typical launch choices:
- Place the instance in the **private subnet**.
- Disable auto-assign public IP.
- Attach a security group that allows only required inbound traffic from trusted sources, such as a bastion host or load balancer.
- Use Systems Manager Session Manager if you want administrative access without a bastion host.

Example AWS CLI launch pattern:

```bash
aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --subnet-id subnet-0abc1234def567890 \
  --security-group-ids sg-0123456789abcdef0 \
  --associate-public-ip-address false \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=private-workload}]'
```

If you use a launch template or Auto Scaling Group, ensure the network settings place instances in the private subnet and do not assign public IPs.

**Creating the NAT Gateway**

The NAT Gateway must be created in a public subnet.

Typical steps:
1. Allocate an **Elastic IP**.
2. Create the NAT Gateway in a public subnet.
3. Wait for the NAT Gateway state to become available.
4. Update the private subnet route table to send default traffic to the NAT Gateway.

AWS CLI example:

```bash
aws ec2 allocate-address --domain vpc
```

```bash
aws ec2 create-nat-gateway \
  --subnet-id subnet-0public1234567890 \
  --allocation-id eipalloc-0123456789abcdef0 \
  --tag-specifications 'ResourceType=natgateway,Tags=[{Key=Name,Value=private-egress-nat}]'
```

After creation, verify the NAT Gateway state:

```bash
aws ec2 describe-nat-gateways \
  --filter Name=subnet-id,Values=subnet-0public1234567890
```

**Routing private subnet traffic through NAT**

The private subnet route table needs a default route to the NAT Gateway.

Example AWS CLI route update:

```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-0123456789abcdef0
```

If the route already exists, you may need to replace it:

```bash
aws ec2 replace-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-0123456789abcdef0
```

A correct private route table usually contains:
- The local VPC route.
- A default route to the NAT Gateway.

It should not contain:
- A direct default route to an Internet Gateway.
- A route that assigns a public path to the private subnet.

**Verifying outbound access**

Once the route is in place, test from the private instance.

Common checks:
- Confirm the instance has no public IPv4 address.
- Confirm the route table points `0.0.0.0/0` to the NAT Gateway.
- Confirm the NAT Gateway is in a public subnet with an Internet Gateway route.
- Confirm security groups and network ACLs allow the traffic.

Example Linux checks:

```bash
ip route
curl -I https://example.com
curl https://checkip.amazonaws.com
```

If the NAT Gateway is working, `curl https://checkip.amazonaws.com` returns the Elastic IP associated with the NAT Gateway, not the private instance’s address.

Package manager examples:

```bash
sudo yum update -y
```

```bash
sudo apt-get update
```

These commands are useful because they prove outbound HTTPS and repository access.

**Troubleshooting checklist**

If the private instance cannot reach the internet, check the following in order:

1. The instance is actually in the private subnet.
2. The private subnet route table has `0.0.0.0/0` pointing to the NAT Gateway.
3. The NAT Gateway is in a public subnet.
4. The public subnet route table has `0.0.0.0/0` pointing to the Internet Gateway.
5. The NAT Gateway has an Elastic IP and is in the `available` state.
6. Security groups allow outbound traffic.
7. Network ACLs allow outbound traffic and ephemeral return ports.
8. DNS resolution is enabled in the VPC if you are testing by hostname.

Useful VPC DNS settings:
- `enableDnsSupport` should be enabled.
- `enableDnsHostnames` should be enabled when you need public DNS hostnames for instances and related services.

Example VPC attribute checks:

```bash
aws ec2 describe-vpc-attribute \
  --vpc-id vpc-0123456789abcdef0 \
  --attribute enableDnsSupport
```

```bash
aws ec2 describe-vpc-attribute \
  --vpc-id vpc-0123456789abcdef0 \
  --attribute enableDnsHostnames
```

**Common exam points**

For SAA-C03, remember these distinctions:

- **Public subnet** means the route table has a path to an Internet Gateway.
- **Private subnet** means no direct inbound internet path.
- **NAT Gateway** provides outbound internet access for private IPv4 instances.
- **Elastic IP** is required for the NAT Gateway.
- **Security groups** are stateful; network ACLs are stateless.
- **NAT Gateway** is the preferred managed solution over a NAT instance in most designs.

A common exam trap is assuming that a private subnet cannot access the internet at all. In reality, it can access the internet through a NAT Gateway while still remaining non-public.

**Practical deployment example**

A minimal production-style layout often looks like this:
- VPC: `10.0.0.0/16`
- Public subnet AZ-a: `10.0.1.0/24`
- Private subnet AZ-a: `10.0.11.0/24`
- Internet Gateway attached to the VPC
- NAT Gateway in the public subnet
- Private route table default route to the NAT Gateway
- EC2 instance in the private subnet with no public IP

Example topology:

```text
VPC 10.0.0.0/16
├─ Public subnet 10.0.1.0/24
│  ├─ NAT Gateway
│  └─ Route: 0.0.0.0/0 -> Internet Gateway
└─ Private subnet 10.0.11.0/24
   ├─ EC2 workload
   └─ Route: 0.0.0.0/0 -> NAT Gateway
```

This design is simple, secure, and easy to reason about during troubleshooting.

**Operational considerations**

NAT Gateways are convenient, but they are not free. You pay for:
- The NAT Gateway itself.
- Data processed through the gateway.
- Cross-AZ data transfer if traffic from a private subnet in one AZ uses a NAT Gateway in another AZ.

For higher availability and lower cross-AZ charges:
- Deploy a NAT Gateway in each AZ.
- Route each private subnet to the NAT Gateway in the same AZ.

This is a common best practice in multi-AZ architectures.

**When to choose alternatives**

Consider alternatives when the use case is different:
- Use **VPC endpoints** for AWS service access without internet egress.
- Use **AWS Systems Manager Session Manager** instead of bastion hosts for administration.
- Use a **NAT instance** only when you need custom routing or legacy behavior and accept the management overhead.

VPC endpoints are especially useful when private instances need access to services like Amazon S3 or Amazon DynamoDB without traversing the public internet.

**Summary**

The lab demonstrates a foundational AWS networking pattern:
- Keep workloads private.
- Provide controlled outbound access through a NAT Gateway.
- Use route tables to separate public and private traffic paths.
- Validate connectivity with real commands and route inspection.

If you understand how the private subnet, public subnet, Internet Gateway, and NAT Gateway work together, you can solve many SAA-C03 networking questions and design secure, practical VPC architectures.
