## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**VPC endpoint fundamentals**

VPC endpoints let you privately connect resources in a VPC to supported AWS services without traversing the public internet. They are a common exam topic because they reduce exposure, simplify routing, and can improve security posture for workloads that must stay on the AWS network.

The two endpoint types you need to distinguish are:

- **Gateway endpoints** for Amazon S3 and DynamoDB.
- **Interface endpoints** for many other AWS services, powered by AWS PrivateLink.

A useful mental model is:

- Gateway endpoints are a **route table feature**.
- Interface endpoints are a **network interface feature**.

**Gateway endpoints**

Gateway endpoints are the simplest option and are supported only for Amazon S3 and DynamoDB. When you create one, AWS adds a route to your route table that points traffic for the service to the endpoint. The route target is the endpoint itself, and the destination is an AWS-managed prefix list for the service.

This is a common exam detail:

- The route table contains a destination like an AWS-managed prefix list for S3 or DynamoDB.
- The target is the **gateway endpoint**.
- Traffic stays on the AWS backbone and does not need an internet gateway, NAT gateway, or VPN for access to the service.

Because gateway endpoints are route-based, they do not create elastic network interfaces (ENIs) in your subnets and do not require security groups.

Typical use cases:

- Private EC2 instances reading objects from S3.
- Private application tiers querying DynamoDB without internet access.
- Reducing NAT gateway data processing costs for S3 and DynamoDB traffic.

Example route table entry conceptually looks like this:

```text
Destination: pl-xxxxxxxx  # AWS-managed prefix list for S3 or DynamoDB
Target: vpce-xxxxxxxx     # Gateway endpoint ID
```

A practical S3 example with the AWS Command Line Interface (CLI):

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.s3 \
  --vpc-endpoint-type Gateway \
  --route-table-ids rtb-0123456789abcdef0
```

For DynamoDB, the pattern is the same:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.dynamodb \
  --vpc-endpoint-type Gateway \
  --route-table-ids rtb-0123456789abcdef0
```

Gateway endpoint policies are resource-based policies attached to the endpoint. They are commonly used to restrict which buckets or tables can be accessed through the endpoint.

Example S3 endpoint policy:

```json
{
  "Statement": [
    {
      "Sid": "AllowOnlyFinanceBucket",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": [
        "arn:aws:s3:::finance-data",
        "arn:aws:s3:::finance-data/*"
      ]
    }
  ]
}
```

Example DynamoDB endpoint policy:

```json
{
  "Statement": [
    {
      "Sid": "AllowOnlySpecificTable",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:PutItem",
        "dynamodb:Query"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders"
    }
  ]
}
```

**Interface endpoints**

Interface endpoints use AWS PrivateLink and create one or more ENIs in your subnets. They are used for many AWS services, such as CloudWatch, Systems Manager, KMS, ECR, Secrets Manager, STS, and others, depending on Region and service support.

Key exam distinction:

- **Interface endpoints require security groups and ENIs.**
- **Gateway endpoints do not require security groups or ENIs.**

Interface endpoints are the right choice when the service is not S3 or DynamoDB, or when you need private access to a service that supports PrivateLink.

Important characteristics:

- You choose one or more subnets for the endpoint ENIs.
- You attach security groups to control inbound traffic to the endpoint.
- The endpoint can be created with **private DNS enabled or disabled**.
- The VPC must have **DNS resolution** and **DNS hostnames** enabled for private DNS to work properly.

Private DNS matters because it lets standard service names resolve to the private IP addresses of the endpoint ENIs. Without private DNS, you often need to use the endpoint-specific DNS name explicitly.

Example CLI creation:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.logs \
  --vpc-endpoint-type Interface \
  --subnet-ids subnet-aaa111 subnet-bbb222 \
  --security-group-ids sg-0123456789abcdef0 \
  --private-dns-enabled
```

If you need to disable private DNS for a specific endpoint, you can do so at creation time or modify it later, depending on the service and configuration.

Interface endpoint policies are also resource-based policies, but in practice they are used to control which actions and resources are allowed through that endpoint for the supported service. The policy is evaluated in addition to IAM permissions and the service’s own authorization model.

A practical example for an interface endpoint to AWS Systems Manager might restrict access to specific SSM actions:

```json
{
  "Statement": [
    {
      "Sid": "AllowSSMReadOnly",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "ssm:GetParameter",
        "ssm:GetParameters",
        "ssm:GetParametersByPath"
      ],
      "Resource": "*"
    }
  ]
}
```

**Gateway vs interface endpoints**

Use this compare-and-contrast view for exam questions:

| Feature | Gateway endpoint | Interface endpoint |
|---|---|---|
| AWS services | S3, DynamoDB | Many AWS services via PrivateLink |
| Network construct | Route table entry | ENI in subnet |
| Security groups | Not used | Required |
| DNS | Not required for endpoint operation | Often required for private DNS behavior |
| Route target | Endpoint | ENI IPs via DNS |
| Prefix lists | Uses AWS-managed prefix lists | Not used as the primary mechanism |
| Cost model | No hourly endpoint charge, but data transfer considerations still apply | Hourly endpoint charge plus data processing charges |
| Best for | Private S3/DynamoDB access | Private access to most other supported services |

A direct exam shortcut:

- If the question mentions **route tables** and **prefix lists**, think **gateway endpoint**.
- If the question mentions **security groups**, **private DNS**, or **ENIs**, think **interface endpoint**.

**DNS behavior and private DNS**

Private DNS is one of the most tested interface endpoint details. When enabled, the standard public service hostname resolves to the private IP address of the endpoint ENI inside your VPC.

This is useful because your applications can keep using normal service names such as:

- `ssm.us-east-1.amazonaws.com`
- `logs.us-east-1.amazonaws.com`
- `secretsmanager.us-east-1.amazonaws.com`

For private DNS to work, the VPC must have:

- **DNS resolution enabled**
- **DNS hostnames enabled**

If either setting is disabled, private DNS resolution may fail or behave unexpectedly.

Example VPC attribute check:

```bash
aws ec2 describe-vpc-attribute \
  --vpc-id vpc-0123456789abcdef0 \
  --attribute enableDnsSupport

aws ec2 describe-vpc-attribute \
  --vpc-id vpc-0123456789abcdef0 \
  --attribute enableDnsHostnames
```

If private DNS is disabled, you can still use the endpoint-specific DNS name returned by AWS, but your application must target it explicitly.

**Regional and zonal placement**

Interface endpoint ENIs are created in the subnets you choose, which means they are tied to specific Availability Zones (AZs). In practice, this creates two important considerations:

- You can place endpoint ENIs in one or more AZs for resilience and locality.
- If your workload in one AZ uses an endpoint ENI in another AZ, cross-AZ data transfer charges may apply.

This matters for both cost and latency. For multi-AZ architectures, place interface endpoint ENIs in the same AZs as the workloads that use them whenever possible.

Gateway endpoints do not have this AZ placement concern because they are not ENI-based.

**Service coverage caveats**

Not every AWS service supports interface endpoints in every Region, and not every service supports private DNS in every case. For exam purposes, avoid assuming universal support.

Practical guidance:

- Verify that the service supports PrivateLink in the Region you are using.
- Check whether private DNS is supported for that service.
- Use the service-specific endpoint name if private DNS is unavailable.

This is especially important in hybrid or multi-Region designs where service availability can differ.

**Security and policy evaluation**

Endpoint policies are not a replacement for IAM. They are an additional control point.

How to think about the layers:

- **IAM policy** answers: “Is this principal allowed to call the API?”
- **Endpoint policy** answers: “Is this request allowed through this endpoint?”
- **Service policy or resource policy** answers: “Is the target resource allowing access?”

For gateway endpoints, endpoint policies are especially common for S3 and DynamoDB because they let you constrain access to specific buckets or tables.

For interface endpoints, endpoint policies are useful when you want to limit which API actions can traverse the endpoint, but the exact behavior depends on the service.

**Troubleshooting connectivity**

When a private service call fails, check the network path systematically.

1. Confirm the endpoint type matches the service.
2. Confirm the route table association for gateway endpoints.
3. Confirm the subnet selection for interface endpoints.
4. Confirm security groups allow the required traffic.
5. Confirm network ACLs (NACLs) do not block the traffic.
6. Confirm DNS is enabled if you rely on private DNS.
7. Confirm the service and Region support the endpoint type you are using.

Common failure patterns:

- **Wrong subnet or AZ selection**: the endpoint ENI exists, but the workload is in a different AZ and experiences cross-AZ traffic or cannot reach the expected local endpoint path.
- **NACL restrictions**: stateless NACL rules block ephemeral ports or return traffic.
- **Security group mismatch**: the endpoint SG does not allow inbound traffic from the workload SG on the required port, usually TCP 443.
- **Private DNS disabled**: the application resolves the public service endpoint instead of the private one.
- **Missing route table association**: gateway endpoint traffic never matches the route.
- **Unsupported service/Region**: the endpoint cannot be created or does not behave as expected.

A simple connectivity test from an EC2 instance:

```bash
curl -I https://s3.us-east-1.amazonaws.com
nslookup ssm.us-east-1.amazonaws.com
aws sts get-caller-identity
```

If you are validating an interface endpoint, also inspect DNS resolution and the endpoint ENI IPs:

```bash
dig ssm.us-east-1.amazonaws.com
ip route
```

**Practical architecture examples**

A private application tier that reads from S3 and writes logs to CloudWatch often uses both endpoint types:

- Gateway endpoint for S3 access.
- Interface endpoint for CloudWatch Logs.
- Interface endpoint for Systems Manager if you want private management access.

Example topology:

- Private subnets host EC2 instances.
- Route tables include an S3 gateway endpoint.
- Interface endpoints are deployed in the same AZs as the instances.
- Security groups allow HTTPS from the application instances to the endpoint ENIs.

This design avoids NAT gateway dependency for supported service traffic and keeps service calls inside the AWS network.

**Exam decision guide**

Use these cues to choose the right answer quickly:

- Choose a **gateway endpoint** when the question mentions S3 or DynamoDB and route table entries.
- Choose an **interface endpoint** when the question mentions PrivateLink, ENIs, security groups, or private DNS.
- Choose **private DNS enabled** when you want applications to use standard AWS service hostnames.
- Choose multiple subnets/AZs for interface endpoints when availability and locality matter.
- Remember that gateway endpoints use AWS-managed prefix lists and the route target is the endpoint.

**Quick reference**

- **Gateway endpoint**
  - S3 and DynamoDB only
  - Route table integration
  - No ENIs
  - No security groups
  - Uses AWS-managed prefix lists

- **Interface endpoint**
  - Many AWS services
  - PrivateLink-based
  - ENIs in subnets
  - Security groups required
  - Private DNS optional but commonly used
  - VPC DNS resolution and hostnames must be enabled for private DNS
