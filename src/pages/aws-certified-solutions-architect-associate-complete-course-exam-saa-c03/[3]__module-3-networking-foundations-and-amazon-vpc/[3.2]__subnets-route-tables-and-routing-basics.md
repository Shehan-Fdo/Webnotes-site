## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Subnet routing fundamentals**

A subnet route table is the control plane object that decides where traffic from a subnet goes. In AWS, routing is evaluated at the subnet level, not the instance level, so the route table attached to a subnet determines the next hop for packets leaving that subnet.

A route table contains destination prefixes and targets. AWS uses the most specific matching route, so a more specific CIDR or prefix list entry overrides a broader default route. This is the core idea behind internet access, private egress, VPC endpoints, and hybrid connectivity.

- **Subnet association** determines which route table a subnet uses.
- **Longest prefix match** decides which route wins when multiple routes match.
- **Stateful return paths** matter for peering, VPN, Transit Gateway, and Direct Connect.
- **Security groups and network ACLs** still apply after routing chooses the path.

**The local route and VPC-internal traffic**

Every route table automatically includes a **local** route for the VPC CIDR block. You cannot remove it. This route enables communication between resources in the same VPC, including across subnets, as long as security controls allow it.

For IPv4, the local route points to the VPC IPv4 CIDR, such as `10.0.0.0/16 local`. For IPv6, the local route points to the VPC IPv6 CIDR block, not a subnet-specific prefix. The route is automatically present in every route table associated with the VPC.

> The local route is not optional. If it is missing from a route table view, the table is not behaving as a normal VPC route table or the view is incomplete.

Practical implications:

- Instances in different subnets can talk to each other through the local route.
- You do not need an internet gateway for east-west traffic inside the VPC.
- Security groups and network ACLs can still block traffic even when routing is correct.
- The local route does not send traffic to the internet or to another VPC.

**Route table association rules and limits**

A subnet can be associated with only one route table at a time. If you change the association, new packets from that subnet use the new table immediately. Existing flows may not survive cleanly if return paths, stateful devices, or session tracking depend on the old path.

Important constraints to remember for exam questions:

- A subnet has exactly one active route table association.
- A VPC can have multiple route tables.
- A route table can be associated with multiple subnets.
- The **main route table** is the default table for any subnet that does not have an explicit association.

Operationally, this means a route table change can have broad impact:

- Reassociating a subnet can instantly change internet access or private connectivity.
- Editing the main route table affects all unassociated subnets.
- Accidentally moving a subnet to the wrong table can isolate workloads or expose them publicly.

Example association workflow with the AWS CLI:

```bash
aws ec2 associate-route-table \
  --route-table-id rtb-0123456789abcdef0 \
  --subnet-id subnet-0abc1234def567890
```

To inspect current associations:

```bash
aws ec2 describe-route-tables \
  --route-table-ids rtb-0123456789abcdef0
```

**Public subnets and internet access**

An internet gateway (**IGW**) provides a path between a VPC and the public internet for IPv4 and IPv6 traffic, but routing alone is not enough for IPv4 internet access. The instance or resource also needs a public IPv4 address or an Elastic IP, and security controls must allow the traffic.

A typical public subnet has:

- A route to `0.0.0.0/0` targeting the IGW.
- Optionally a route to `::/0` targeting the IGW for IPv6.
- Instances with public IPv4 addresses or Elastic IPs when IPv4 internet access is required.

Example public subnet route table:

```text
Destination        Target
10.0.0.0/16        local
0.0.0.0/0          igw-0a1b2c3d4e5f67890
::/0               igw-0a1b2c3d4e5f67890
```

Key exam point:

- An IGW route by itself does **not** make a private IPv4 instance reachable from the internet.
- For inbound IPv4 internet access, the instance needs a public IPv4 address or Elastic IP.
- Security groups and network ACLs must also permit the traffic.

For IPv6, addresses are globally unique, but that does not mean they are automatically reachable from the internet. You still need correct routing, security group rules, and network ACL rules. AWS also supports an **egress-only internet gateway** for IPv6 outbound-initiated traffic from private subnets.

**Private subnets, NAT, and egress design**

A private subnet typically has no direct route to an IGW. For outbound IPv4 internet access, you usually use a **NAT gateway** in a public subnet. The private subnet routes `0.0.0.0/0` to the NAT gateway, and the NAT gateway routes to the IGW.

Example private subnet route table:

```text
Destination        Target
10.0.0.0/16        local
0.0.0.0/0          nat-0a1b2c3d4e5f67890
```

For IPv6 outbound-only access, you can use an **egress-only internet gateway**:

```text
Destination        Target
2001:db8:1234::/56 local
::/0               eigw-0a1b2c3d4e5f67890
```

Use this carefully in exam scenarios:

- An egress-only internet gateway supports outbound-initiated IPv6 connections and return traffic.
- It does not provide inbound internet reachability.
- It is not a translation device like NAT; it is a routing control for IPv6 egress.

**NAT gateway vs NAT instance**

A NAT gateway and a NAT instance both allow private IPv4 subnets to reach the internet without exposing private instances directly. The exam often asks you to choose between them.

| Feature | NAT Gateway | NAT Instance |
|---|---|---|
| Managed service | Yes | No |
| Scaling | Automatic | Manual |
| Availability | High within an AZ design | Depends on your setup |
| Maintenance | AWS-managed | You manage patching and hardening |
| Source/destination checks | Not applicable | Must disable for NAT use |
| Typical recommendation | Preferred | Legacy or specialized use |

Practical guidance:

- Choose a **NAT gateway** for most production workloads.
- Choose a **NAT instance** only when you need custom routing, packet inspection, or legacy behavior and accept the operational overhead.
- Place the NAT gateway in a public subnet with a route to the IGW.
- Route private subnet default traffic to the NAT gateway.

Example NAT gateway creation flow:

```bash
aws ec2 create-nat-gateway \
  --subnet-id subnet-0public1234567890 \
  --allocation-id eipalloc-0123456789abcdef0
```

**VPC endpoints and route table behavior**

VPC endpoints let you access AWS services without traversing the public internet. Route table behavior depends on the endpoint type.

**Gateway endpoints**:
- Supported for services such as Amazon S3 and DynamoDB.
- They add route table entries that target a prefix list for the service.
- You associate the endpoint with one or more route tables.
- They do not act like a generic next hop for arbitrary CIDRs.

Example gateway endpoint route entry:

```text
Destination        Target
pl-68a54001        vpce-0abc1234def567890
```

**Interface endpoints**:
- Use AWS PrivateLink.
- Create elastic network interfaces in your subnets.
- Do not add route table entries in the same way gateway endpoints do.
- Traffic reaches the endpoint through private IPs and DNS resolution.

Exam distinction:

- If the question mentions S3 or DynamoDB and route table entries, think **gateway endpoint**.
- If the question mentions private access to many AWS services or third-party services over PrivateLink, think **interface endpoint**.
- Do not assume every endpoint modifies the route table.

**Hybrid connectivity and return-path symmetry**

Routing for peering, Transit Gateway (**TGW**), VPN, and Direct Connect often fails because the forward path exists but the return path does not. AWS routing is not enough by itself; the remote side must also know how to send traffic back.

Common scenarios:

- **VPC peering** requires routes in both VPCs for the relevant CIDRs.
- **Transit Gateway** requires route table associations and propagations on the TGW side, plus VPC route table entries pointing to the TGW attachment.
- **Site-to-Site VPN** requires routes on the VPC side and on the on-premises side.
- **Direct Connect** requires correct routing through a virtual private gateway or TGW, plus propagation and on-premises return routes.

A useful troubleshooting rule:

- If traffic goes out but never returns, check the return path first.
- If a route exists in one direction only, the connection may fail even though the subnet route table looks correct.

Example VPC route table for TGW:

```text
Destination        Target
10.0.0.0/16        local
10.20.0.0/16       tgw-0a1b2c3d4e5f67890
192.168.0.0/16     tgw-0a1b2c3d4e5f67890
```

**Route propagation and learned routes**

AWS can propagate routes into a route table in some connectivity designs, but this depends on the attachment type and whether propagation is enabled.

Use this mental model:

- **Static routes** are manually added.
- **Propagated routes** are learned from supported attachments such as VPN or Direct Connect through a virtual private gateway, or from Transit Gateway route propagation.
- Propagation is not automatic in every setup.
- You must enable the relevant propagation or association for the route to appear.

This matters in exam questions because a route may be missing even though the connection exists. The fix may be to enable propagation rather than to add a static route.

**Route selection and longest prefix match**

AWS chooses the most specific matching route. This is the same basic longest prefix match behavior used in IP routing generally.

Example:

```text
Destination        Target
10.0.0.0/16        local
10.0.1.0/24        tgw-0a1b2c3d4e5f67890
0.0.0.0/0          nat-0a1b2c3d4e5f67890
```

Traffic to `10.0.1.25` uses the `/24` route, not the broader `/16` local route, if the destination is outside the local VPC CIDR and the route is valid in context. More specific routes always win when they match.

Practical use cases:

- Send one subnet to a firewall appliance while the rest uses a default route.
- Override a broad default route with a more specific service path.
- Direct a partner CIDR to a VPN or TGW while keeping internet traffic on NAT.

**Exam-style decision guide**

Use the following patterns to choose the right routing design:

- **Need public inbound access to an EC2 instance**
  - Put the instance in a public subnet.
  - Attach an IGW to the VPC.
  - Assign a public IPv4 address or Elastic IP.
  - Allow traffic in security groups and network ACLs.

- **Need private subnet outbound internet for IPv4**
  - Use a NAT gateway in a public subnet.
  - Route `0.0.0.0/0` from the private subnet to the NAT gateway.
  - Ensure the NAT gateway has a route to the IGW.

- **Need private subnet outbound-only IPv6**
  - Use an egress-only internet gateway.
  - Route `::/0` to the egress-only IGW.
  - Keep security rules restrictive.

- **Need private access to S3 without internet**
  - Use a gateway endpoint for S3.
  - Associate it with the relevant route tables.
  - Confirm the route table contains the service prefix list entry.

- **Need private access to another VPC**
  - Use VPC peering or TGW.
  - Add routes on both sides.
  - Verify CIDR overlap does not exist for peering.

- **Need on-premises connectivity**
  - Use VPN, Direct Connect, or TGW.
  - Ensure propagation and return routes exist on both sides.

**Route table editing pitfalls**

Route table mistakes are a common cause of outages because the effects are immediate and subnet-scoped.

Watch for these pitfalls:

- Changing the **main route table** can affect every subnet without an explicit association.
- Reassociating a subnet can silently move workloads to a different path.
- Deleting a route can break egress, endpoint access, or hybrid connectivity.
- Adding a broad route can unintentionally override a more specific design if the destination overlaps in an unexpected way.
- Assuming a route table change alone fixes connectivity can hide missing security group, NACL, or return-path issues.

A safe change process:

1. Identify the subnet and its current route table.
2. Confirm the target route table has the required local, default, endpoint, and hybrid routes.
3. Verify security groups and network ACLs.
4. Apply the association change during a maintenance window if the workload is sensitive.
5. Test both forward and return traffic.

**CLI examples for inspection and troubleshooting**

List route tables and associations:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

Add a default IPv4 route to a NAT gateway:

```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-0a1b2c3d4e5f67890
```

Add an IPv6 default route to an egress-only internet gateway:

```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-ipv6-cidr-block ::/0 \
  --egress-only-internet-gateway-id eigw-0a1b2c3d4e5f67890
```

Add a route to a Transit Gateway:

```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 10.20.0.0/16 \
  --transit-gateway-id tgw-0a1b2c3d4e5f67890
```

**Quick mental model for the exam**

Think in layers:

- **Route table** decides the next hop.
- **Subnet association** decides which table applies.
- **Local route** keeps VPC-internal traffic working.
- **IGW, NAT, endpoints, TGW, VPN, and Direct Connect** are the possible targets.
- **Security groups and network ACLs** still decide whether the packet is allowed.
- **Return path symmetry** determines whether the connection actually succeeds.

If a scenario asks why traffic fails, check these in order:

1. Correct route table association?
2. Correct route target?
3. Correct source address or public IP requirement?
4. Security group and network ACL rules?
5. Return route on the other side?

**Common exam traps**

- Assuming an IGW alone makes a private instance internet-accessible.
- Treating gateway endpoints and interface endpoints as the same thing.
- Forgetting that the local route cannot be removed.
- Ignoring return routes in peering, VPN, TGW, or Direct Connect designs.
- Assuming route propagation is automatic in every hybrid setup.
- Confusing NAT gateway behavior with IPv6 egress-only routing.
