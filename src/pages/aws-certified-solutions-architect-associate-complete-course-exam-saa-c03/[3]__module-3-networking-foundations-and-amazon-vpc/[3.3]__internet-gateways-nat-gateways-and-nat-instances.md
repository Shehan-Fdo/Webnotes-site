## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**VPC internet access model**

An Amazon VPC (Virtual Private Cloud) uses route tables plus gateway components to control how traffic enters and leaves subnets. The exam often tests whether you understand that a subnet being “public” is not enough by itself: the subnet must have a route to an Internet Gateway (IGW), and the instance must also have a public IPv4 address or Elastic IP (EIP) for direct internet reachability.

A subnet is considered public when its route table contains a default route to an IGW, such as `0.0.0.0/0 -> igw-...`. That route makes the subnet internet-routable, but it does not automatically make every instance reachable from the internet. For inbound traffic to work, the instance needs a public IPv4 address or EIP, and security groups and network ACLs (NACLs) must allow the traffic. Without the public address, the IGW has no destination to map the packet to, so the instance remains unreachable even though the subnet is public.

- **IGW** provides a horizontally scaled, highly available path between a VPC and the internet.
- **Public subnet** means the route table points to an IGW.
- **Public IP/EIP** is required for direct inbound and outbound internet communication for an instance in that subnet.
- **Security groups** and **NACLs** still control whether traffic is allowed.

> A public subnet alone is not enough for inbound internet access. You need the route to the IGW and a public IPv4 address on the instance or load balancer.

**Internet Gateway fundamentals**

An Internet Gateway is a VPC component that enables communication between instances in your VPC and the internet. It is attached to a single VPC, but it is managed as a highly available AWS service component rather than something you size or patch.

Use an IGW when you need:
- Public-facing web servers
- Bastion hosts
- Public load balancers
- Outbound internet access from resources with public IPs

Important characteristics:
- **Stateful at the network edge** in the sense that return traffic is handled by the VPC routing and address mapping model, not by a separate appliance you manage.
- Supports IPv4 internet access for resources with public IPv4 addresses.
- Supports IPv6 internet access when the subnet route table includes an IPv6 route to the IGW and the instance has an IPv6 address.
- Does not require an instance to be in a public subnet if another route path exists, but in practice public subnets are the standard design for direct internet exposure.

A common route table for a public subnet looks like this:

```text
Destination        Target
10.0.0.0/16        local
0.0.0.0/0          igw-0123456789abcdef0
::/0               igw-0123456789abcdef0
```

**Public subnet design and reachability**

A public subnet is defined by its route table, not by a special subnet flag. The route to the IGW is what makes the subnet public. However, the instance still needs a public address to be reachable from the internet.

The relationship works like this:
- The subnet route table sends internet-bound traffic to the IGW.
- The IGW performs address translation for instances with public IPv4 addresses or EIPs.
- The public IP/EIP is the internet-facing identity of the instance.
- Security groups and NACLs determine whether the traffic is allowed.

Practical implications:
- If you launch an EC2 instance in a public subnet but disable auto-assign public IPv4 and do not attach an EIP, the instance cannot be reached directly from the internet.
- If you attach an EIP, the IGW can map the public address to the instance’s private address.
- If the route to the IGW is missing, a public IP alone does not help because the subnet has no internet path.

**NAT Gateway overview**

A NAT Gateway allows instances in a private subnet to initiate outbound connections to the internet or other AWS services that are reachable over the internet, while preventing unsolicited inbound connections from the internet.

Use a NAT Gateway when private instances need:
- OS package updates
- Software downloads
- External API calls
- Access to public endpoints without exposing private IPs

Key exam points:
- **NAT Gateway is managed by AWS** and you do not patch or administer the underlying host.
- It must be created in a **public subnet** because it needs a route to the IGW for internet egress.
- Private subnets route `0.0.0.0/0` to the NAT Gateway, not directly to the IGW.
- NAT Gateways are **AZ-scoped**. A NAT Gateway lives in one Availability Zone and is not shared across AZs for high availability without introducing cross-AZ routing tradeoffs.
- NAT Gateways support **IPv4 only**. For private IPv6 egress, use an **Egress-Only Internet Gateway** instead of NAT.

A typical private subnet route table for IPv4 egress looks like this:

```text
Destination        Target
10.0.0.0/16        local
0.0.0.0/0          nat-0abc123def4567890
```

**Why a NAT Gateway must be in a public subnet**

A NAT Gateway itself needs internet access to forward traffic. AWS places it in a public subnet so the subnet route table can send its outbound traffic to the IGW. The private subnet does not point to the IGW directly; it points to the NAT Gateway, which then uses the public subnet’s route to reach the internet.

This creates a two-hop design:
1. Private instance sends traffic to the NAT Gateway.
2. NAT Gateway sends translated traffic out through the IGW.

That is why a private subnet can reach the internet without being directly internet-routable.

**NAT Gateway limitations**

NAT Gateways are often tested through what they cannot do.

- **No inbound connections** from the internet to private instances through the NAT Gateway.
- **No security group association** on the NAT Gateway itself.
- **No port forwarding** or custom listener configuration.
- **IPv4 only**; it does not translate IPv6 traffic.
- **No source/destination filtering** beyond routing and the associated network controls on the instances and subnets.

Because of these limits, NAT Gateways are not a replacement for a firewall, reverse proxy, or load balancer.

**NAT Gateway high availability and AZ design**

NAT Gateways are AZ-scoped, so the usual best practice is to deploy one NAT Gateway per AZ and route each private subnet to the NAT Gateway in the same AZ.

Why this matters:
- If you use a single NAT Gateway for multiple AZs, traffic from other AZs may cross AZ boundaries.
- Cross-AZ routing can add latency and data transfer charges.
- If the NAT Gateway’s AZ fails, private subnets depending on that NAT Gateway lose outbound internet access unless you have a failover design.

A resilient pattern is:
- One public subnet per AZ
- One NAT Gateway per AZ
- One private route table per AZ pointing to the local NAT Gateway

Example topology:

```text
VPC 10.0.0.0/16
├─ AZ-a
│  ├─ Public subnet 10.0.1.0/24 -> IGW
│  ├─ NAT Gateway A in public subnet
│  └─ Private subnet 10.0.11.0/24 -> NAT Gateway A
└─ AZ-b
   ├─ Public subnet 10.0.2.0/24 -> IGW
   ├─ NAT Gateway B in public subnet
   └─ Private subnet 10.0.12.0/24 -> NAT Gateway B
```

**NAT Gateway and IPv6**

A common exam trap is assuming NAT Gateway handles both IPv4 and IPv6. It does not.

For private IPv6 instances:
- Use an **Egress-Only Internet Gateway** for outbound-only IPv6 internet access.
- Route `::/0` from the private subnet to the Egress-Only Internet Gateway.
- This preserves the “no inbound from the internet” behavior for IPv6.

Comparison:
- **NAT Gateway**: IPv4 outbound translation for private subnets.
- **Egress-Only Internet Gateway**: IPv6 outbound-only internet access.

Example IPv6 route table entry:

```text
Destination        Target
::/0               eigw-0123456789abcdef0
```

**NAT Instance overview**

A NAT Instance is an EC2 instance configured to perform network address translation for private subnet traffic. It is the older, more manual pattern and is still relevant for exam questions because it exposes operational tradeoffs.

Use a NAT Instance when you need:
- Custom routing or firewall behavior
- Legacy architectures
- Lower-cost small-scale environments where you accept operational overhead

Important characteristics:
- You manage the EC2 instance, operating system, and patching.
- You must disable source/destination checks on the instance.
- You must configure the instance to act as a router/firewall appliance.
- You must size the instance appropriately for expected throughput and connections.
- You must place it in a public subnet with a route to the IGW.

**NAT Instance operational requirements**

A NAT Instance is not just a normal EC2 instance with a route table entry. You must configure it as a forwarding appliance.

Typical setup steps:
1. Launch an EC2 instance in a public subnet.
2. Choose an AMI that supports your needs, often a Linux AMI with packet forwarding capabilities.
3. Disable source/destination checks on the instance.
4. Enable IP forwarding in the operating system.
5. Configure iptables or nftables rules for NAT masquerading.
6. Update private subnet route tables to point `0.0.0.0/0` to the NAT Instance.

Example Linux configuration:

```bash
# Enable IPv4 forwarding
sudo sysctl -w net.ipv4.ip_forward=1

# Persist the setting
echo 'net.ipv4.ip_forward = 1' | sudo tee -a /etc/sysctl.conf

# Example iptables masquerade rule
sudo iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
```

AWS CLI example to disable source/destination checks:

```bash
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --source-dest-check "{\"Value\": false}"
```

**NAT Instance sizing and AMI considerations**

Because a NAT Instance is an EC2 instance, its performance depends on the chosen instance type, network bandwidth, and CPU capacity. This is a common exam distinction from NAT Gateway, which is managed and scales automatically.

Consider:
- **Instance type** limits throughput and concurrent connections.
- **Burstable instances** may be insufficient for sustained NAT traffic.
- **AMI choice** should support your OS-level forwarding and firewall configuration.
- **EBS performance** can matter if the instance also logs heavily or runs additional services, though NAT forwarding itself is primarily network-bound.

Operationally, you should treat a NAT Instance as a specialized router/firewall appliance, not as a general-purpose server.

**NAT Instance failure modes and recovery**

NAT Instances and NAT Gateways fail differently, and the exam may ask which design is easier to recover.

NAT Instance:
- If the instance stops, terminates, or becomes unhealthy, private subnet outbound access fails.
- You can replace it, but you must restore the configuration and update routes if the target changes.
- You can use Auto Scaling and health checks to improve availability, but that adds complexity.
- Recovery is your responsibility.

NAT Gateway:
- AWS manages the underlying infrastructure.
- It is highly available within its AZ and does not require OS patching or instance recovery.
- If the AZ fails, the NAT Gateway in that AZ is unavailable, so multi-AZ designs still matter.
- Recovery is simpler because you do not rebuild an appliance.

Practical comparison:
- **NAT Gateway** is preferred for production because it is managed and more resilient within its AZ.
- **NAT Instance** is useful when you need custom packet handling or lower-cost, low-throughput environments.

**End-to-end VPC design example**

A complete design often includes both public and private subnets, an IGW, and NAT for private egress.

Example VPC:
- VPC CIDR: `10.0.0.0/16`
- Public subnet A: `10.0.1.0/24`
- Public subnet B: `10.0.2.0/24`
- Private subnet A: `10.0.11.0/24`
- Private subnet B: `10.0.12.0/24`

Route tables:

Public route table A:
```text
Destination        Target
10.0.0.0/16        local
0.0.0.0/0          igw-0123456789abcdef0
::/0               igw-0123456789abcdef0
```

Public route table B:
```text
Destination        Target
10.0.0.0/16        local
0.0.0.0/0          igw-0123456789abcdef0
::/0               igw-0123456789abcdef0
```

Private route table A:
```text
Destination        Target
10.0.0.0/16        local
0.0.0.0/0          nat-aaa111aaa111aaa11
::/0               eigw-0123456789abcdef0
```

Private route table B:
```text
Destination        Target
10.0.0.0/16        local
0.0.0.0/0          nat-bbb222bbb222bbb22
::/0               eigw-0123456789abcdef0
```

This design gives you:
- Direct internet access for public resources through the IGW
- Outbound-only IPv4 internet access for private resources through NAT Gateways
- Outbound-only IPv6 internet access through an Egress-Only Internet Gateway
- Better fault isolation across AZs

**Exam scenario patterns**

Watch for these common question cues:
- “Private instances need to download patches from the internet” → NAT Gateway or NAT Instance.
- “Public web server in a subnet” → public subnet plus public IP/EIP plus IGW route.
- “Outbound-only IPv6 from private subnet” → Egress-Only Internet Gateway.
- “Need custom firewall rules or port forwarding” → NAT Instance, not NAT Gateway.
- “High availability with minimal management” → NAT Gateway.
- “Single point of failure concern across AZs” → deploy NAT per AZ.

**Quick comparison**

| Feature | Internet Gateway | NAT Gateway | NAT Instance |
|---|---|---|---|
| Internet access type | Public inbound/outbound | Private outbound IPv4 | Private outbound IPv4 |
| Managed by AWS | Yes | Yes | No |
| Requires public subnet | IGW attached to VPC, not subnet-specific | Yes | Yes |
| Supports inbound from internet | Yes, with public IP/EIP and routing | No | No, unless you build custom forwarding, which is not the NAT use case |
| Security group on component | No | No | Yes, because it is an EC2 instance |
| Port forwarding | No | No | Possible with custom config |
| IPv6 support | Yes | No | Possible only with custom instance configuration, but not the standard exam answer |
| AZ-scoped | No | Yes | Depends on deployment |
| Operational overhead | Low | Low | High |

**Troubleshooting checklist**

If a private instance cannot reach the internet:
- Confirm the subnet route table points `0.0.0.0/0` to the NAT Gateway or NAT Instance.
- Confirm the NAT device is in a public subnet with a route to the IGW.
- Confirm the NAT Gateway or NAT Instance has a public IP path to the internet.
- Confirm security groups and NACLs allow outbound traffic and return traffic.
- Confirm source/destination checks are disabled on a NAT Instance.
- Confirm IPv6 traffic is not being sent to a NAT Gateway.

If a public instance is not reachable:
- Confirm the subnet route table has `0.0.0.0/0 -> igw-...`.
- Confirm the instance has a public IPv4 address or EIP.
- Confirm security groups allow the inbound port.
- Confirm NACLs allow the traffic.
- Confirm the route table is associated with the correct subnet.

**Key takeaways**

- A **public subnet** is defined by a route to an IGW, but that alone does not make instances reachable from the internet.
- A public instance also needs a **public IPv4 address or EIP** for inbound connectivity.
- A **NAT Gateway** provides outbound-only IPv4 internet access for private subnets and is the preferred managed solution.
- NAT Gateways are **AZ-scoped**, so multi-AZ designs usually deploy one per AZ.
- NAT Gateways do **not** support IPv6; use an **Egress-Only Internet Gateway** for private IPv6 egress.
- A **NAT Instance** is flexible but requires OS-level configuration, source/destination check changes, sizing, and ongoing maintenance.
- For exam questions, choose the simplest service that matches the required direction of traffic, protocol family, and operational burden.
