## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Connectivity design goals**

AWS network connectivity choices usually come down to three questions: do you need full network reachability, do you need hub-and-spoke routing, or do you only need private access to a service? VPC peering, Transit Gateway (TGW), and PrivateLink solve different problems, and exam questions often hinge on choosing the least complex option that still meets the requirement.

- **VPC peering** gives direct private IP connectivity between two VPCs.
- **Transit Gateway** centralizes routing across many VPCs and networks.
- **PrivateLink** exposes a service privately without full network-level connectivity.

A good mental model is:

- Use **peering** for simple, low-latency, point-to-point connectivity.
- Use **TGW** for scalable multi-VPC or hybrid routing.
- Use **PrivateLink** when consumers only need to reach a specific service, not the whole VPC.

**VPC peering fundamentals**

VPC peering is a one-to-one networking connection between two VPCs. Traffic stays on the AWS backbone and uses private IP addresses, but peering is not a transit mechanism. That means the peering link only connects the two VPCs directly; it does not let one VPC use the other as a router.

- **Non-transitive** routing is the key exam concept.
- You must add routes in both VPC route tables.
- Security groups and network ACLs still control traffic.
- Peering can be created between VPCs in the same Region or different Regions.

Typical use cases:

- Shared services between two VPCs.
- Application and database VPCs with a simple relationship.
- Migration scenarios where you temporarily connect old and new environments.

Example route table entries for peering:

```bash
aws ec2 create-route \
  --route-table-id rtb-11111111111111111 \
  --destination-cidr-block 10.20.0.0/16 \
  --vpc-peering-connection-id pcx-0123456789abcdef0

aws ec2 create-route \
  --route-table-id rtb-22222222222222222 \
  --destination-cidr-block 10.10.0.0/16 \
  --vpc-peering-connection-id pcx-0123456789abcdef0
```

**VPC peering limitations**

Peering is simple, but the limitations are what make it a frequent exam trap. It does not support transitive routing, so you cannot chain VPCs together and expect traffic to pass through one peer to another. It also does not act as a gateway for VPN, Direct Connect, or internet traffic.

> VPC peering does not support gateway transit, edge-to-edge routing, or routing through NAT gateways, internet gateways, or virtual private gateways in the peer VPC.

Important limitations:

- **No transitive routing** between multiple VPCs.
- **No overlapping CIDRs** in any peering scenario.
- **No gateway transit** through the peer VPC.
- **No edge-to-edge routing** through NAT Gateway, Internet Gateway, or Virtual Private Gateway in the peer VPC.
- **No centralized routing** or hub-and-spoke design.
- **No inter-Region DNS forwarding behavior**; DNS resolution is limited to the specific peering DNS settings and does not create a general resolver path.

For exam accuracy, remember:

- Same-Region peering can support security group referencing in supported configurations.
- Inter-Region peering does **not** support security group referencing.
- Cross-Region peering still requires route entries in both VPCs, just like same-Region peering.

**DNS behavior with VPC peering**

DNS support in peering is often misunderstood. Enabling DNS resolution over a peering connection helps instances resolve the peer VPC’s private DNS hostnames when the relevant DNS settings are enabled. It does not create centralized DNS forwarding or allow arbitrary private DNS resolution across all VPCs.

- Enable DNS resolution only when you need private hostname resolution across the peering link.
- Use Route 53 Resolver endpoints if you need centralized DNS forwarding.
- Do not assume peering DNS settings replace a resolver architecture.

Example of the relevant peering options:

```bash
aws ec2 modify-vpc-peering-connection-options \
  --vpc-peering-connection-id pcx-0123456789abcdef0 \
  --requester-peering-connection-options AllowDnsResolutionFromRemoteVpc=true \
  --accepter-peering-connection-options AllowDnsResolutionFromRemoteVpc=true
```

**Transit Gateway overview**

Transit Gateway is a regional network transit hub that connects multiple VPCs, VPNs, and Direct Connect environments through a central routing layer. It is the preferred design when you need many-to-many connectivity without building a mesh of peering connections.

- **Hub-and-spoke** is the core TGW pattern.
- TGW simplifies route management at scale.
- TGW supports segmentation using multiple route tables.
- TGW is regional, but you can peer TGWs across Regions.

Common attachments and integrations:

- VPC attachments
- VPN attachments
- AWS Direct Connect integration through a Direct Connect gateway association
- Transit Gateway peering attachments
- Connect attachments for SD-WAN or third-party appliance integration

> A Direct Connect gateway association is not a TGW attachment type; it is an integration used with TGW and Direct Connect.

**TGW route tables and propagation**

TGW route tables control which attachments can communicate. This is one of the most important design topics for the exam because the default behavior can be confusing.

By default:

- A new attachment can be associated with the default TGW route table.
- Routes from an attachment can be propagated to the default TGW route table if propagation is enabled.
- If you leave defaults on, you often get broad connectivity sooner than intended.

If you disable default association or propagation:

- You must explicitly associate each attachment with the correct TGW route table.
- You must explicitly enable propagation from attachments that should advertise routes.
- This gives you tighter segmentation and more predictable traffic flow.

Practical implications:

- Use separate TGW route tables for prod, dev, shared services, or inspection domains.
- Disable default association/propagation when you need strict control.
- Verify both TGW route table entries and VPC route table entries during troubleshooting.

Example TGW attachment creation:

```bash
aws ec2 create-transit-gateway-vpc-attachment \
  --transit-gateway-id tgw-0123456789abcdef0 \
  --vpc-id vpc-0123456789abcdef0 \
  --subnet-ids subnet-aaa111 subnet-bbb222
```

The subnets you choose must be in different Availability Zones so TGW can place attachment ENIs across AZs for resilience.

Example route propagation and association workflow:

```bash
aws ec2 associate-transit-gateway-route-table \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --transit-gateway-attachment-id tgw-attach-0123456789abcdef0

aws ec2 enable-transit-gateway-route-table-propagation \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0 \
  --transit-gateway-attachment-id tgw-attach-0123456789abcdef0
```

**TGW design considerations**

TGW is more scalable than peering, but it introduces routing design choices you must understand.

- **Overlapping CIDRs** can be attached, but TGW cannot route between overlapping networks.
- **Security group referencing** across VPCs is not a TGW feature the way it can be in some same-Region peering scenarios.
- **CIDR-based rules** are often required when using TGW.
- **Asymmetric routing** can break stateful appliances and firewalls if you do not design carefully.

For centralized inspection, use appliance mode or a routing design that preserves symmetry. This matters when traffic must pass through a firewall, IDS/IPS, or network virtual appliance.

- **Appliance mode** helps keep traffic for a flow symmetric through the same attachment path.
- Use it when inserting inspection appliances between spoke VPCs and shared services.
- Validate return-path routing in both directions.

Example centralized inspection pattern:

- Spoke VPCs send traffic to TGW.
- TGW routes traffic to an inspection VPC.
- Inspection VPC forwards approved traffic back to TGW.
- TGW then routes to the destination VPC or on-premises network.

**TGW and hybrid connectivity**

TGW is often the best answer when AWS must connect to on-premises networks at scale.

- Attach VPN connections directly to TGW.
- Use Direct Connect gateway association for private connectivity from on-premises.
- Centralize routing for multiple VPCs and multiple on-premises networks.
- Use route tables to segment environments and control blast radius.

Example conceptual flow:

```text
On-premises -> Direct Connect -> Direct Connect Gateway -> Transit Gateway -> VPCs
On-premises -> Site-to-Site VPN -> Transit Gateway -> VPCs
```

**TGW peering and inter-Region routing**

TGW peering connects two Transit Gateways in different Regions or accounts. It is useful when you need inter-Region network connectivity with centralized routing on each side.

Key points:

- TGW peering is not transitive.
- You must add routes in both TGW route tables.
- You must also add routes in the associated VPC route tables.
- TGW peering does not automatically propagate routes across Regions.

Example conceptual route setup:

```text
VPC-A route table -> local TGW-A
TGW-A route table -> peering attachment to TGW-B
TGW-B route table -> VPC-B attachment
VPC-B route table -> local TGW-B
```

Practical example:

```bash
aws ec2 create-transit-gateway-peering-attachment \
  --transit-gateway-id tgw-aaaa1111 \
  --peer-transit-gateway-id tgw-bbbb2222 \
  --peer-account-id 123456789012 \
  --peer-region us-west-2
```

After acceptance, you still need:

- TGW route table entries on both sides.
- VPC route table entries pointing to the local TGW attachment.
- Security rules that allow the traffic.

**PrivateLink as an alternative**

PrivateLink is often the best choice when a consumer only needs access to a specific service, not the provider’s entire network. It avoids broad network exposure and reduces routing complexity.

Use PrivateLink when:

- You want to expose a service privately to other VPCs or accounts.
- You do not want full VPC-to-VPC connectivity.
- You want to reduce the need for route table changes.
- You want to avoid transitive routing concerns entirely.

Do not use PrivateLink when:

- You need general network access between VPCs.
- You need bidirectional routing for many subnets and services.
- You need to reach multiple internal hosts across the provider VPC.

**Comparison: peering vs TGW vs PrivateLink**

| Feature | VPC Peering | Transit Gateway | PrivateLink |
|---|---|---|---|
| Connectivity model | Point-to-point | Hub-and-spoke | Service-to-consumer |
| Transitive routing | No | Yes, through TGW routing | No |
| Scale | Low to moderate | High | High for service access |
| CIDR overlap | Not supported | Can attach, but cannot route between overlaps | Not relevant |
| Route table complexity | Low per link, high at scale | Centralized | Minimal |
| Best for | Simple VPC pairs | Many VPCs, hybrid networks | Private service exposure |

**Exam decision guide**

When you read an SAA-C03 question, map the requirement to the connectivity pattern:

1. If the question says two VPCs need direct private communication and the design is simple, choose VPC peering.
2. If the question says many VPCs, shared services, hybrid connectivity, or centralized routing, choose Transit Gateway.
3. If the question says a service must be privately consumed without exposing the whole VPC, choose PrivateLink.
4. If the question mentions inspection, segmentation, or multiple environments, think about TGW route tables and appliance mode.
5. If the question mentions overlapping CIDRs, avoid peering and be cautious with TGW routing expectations.

**Practical troubleshooting checklist**

When connectivity fails, verify the layers in order:

1. Confirm the attachment or peering connection is accepted and active.
2. Check route tables on both sides.
3. Confirm security groups and network ACLs allow the traffic.
4. Verify DNS settings if the issue is hostname resolution rather than IP reachability.
5. For TGW, confirm route table association and propagation.
6. For TGW inspection paths, confirm symmetric routing and appliance behavior.

Useful CLI checks:

```bash
aws ec2 describe-vpc-peering-connections
aws ec2 describe-route-tables
aws ec2 describe-transit-gateways
aws ec2 describe-transit-gateway-attachments
aws ec2 describe-transit-gateway-route-tables
aws ec2 search-transit-gateway-routes \
  --transit-gateway-route-table-id tgw-rtb-0123456789abcdef0
```

**Common exam traps**

- Assuming VPC peering can route through a third VPC.
- Assuming TGW automatically shares routes without route table association or propagation.
- Forgetting that peering requires routes in both VPCs.
- Confusing TGW peering with VPC peering.
- Choosing peering when the question really needs centralized routing or many VPCs.
- Choosing TGW when the requirement is only private access to one service.
- Assuming security group referencing works the same way in all connectivity models.

**Quick memory aid**

- **Peering**: simple, direct, non-transitive.
- **TGW**: scalable, centralized, route-table driven.
- **PrivateLink**: private service access, not full network access.

**Summary**

Use VPC peering for the simplest direct VPC-to-VPC connection, but remember its strict limitations around transitivity, overlapping CIDRs, and gateway transit. Use Transit Gateway when you need scalable hub-and-spoke routing, hybrid connectivity, segmentation, or centralized inspection. Use PrivateLink when the goal is private access to a specific service rather than broad network connectivity. For exam success, focus on route tables, transitivity, DNS behavior, and whether the design needs full network reachability or only service exposure.
