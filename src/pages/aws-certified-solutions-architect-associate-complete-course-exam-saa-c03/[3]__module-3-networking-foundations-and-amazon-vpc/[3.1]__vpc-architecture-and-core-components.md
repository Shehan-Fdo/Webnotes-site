## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon VPC fundamentals**

An <abbr title="Virtual Private Cloud">VPC</abbr> is your logically isolated network in AWS. You define IP ranges, subnets, routing, security boundaries, and connectivity to other networks. For the exam, focus on how VPC components work together rather than memorizing isolated definitions.

A VPC is regional, while subnets are AZ-scoped. That distinction drives high availability design: you spread subnets across multiple Availability Zones (AZs) and place resources where resilience matters.

- **CIDR** blocks define the IP space for the VPC and subnets.
- **Route tables** control traffic direction, not security.
- **Security groups** are stateful instance-level firewalls.
- **Network ACLs** are stateless subnet-level filters.

**Core VPC objects and defaults**

A VPC contains several foundational objects that show up in exam scenarios. The default behavior of these objects matters because AWS creates them automatically in a default VPC.

- **Main route table**: every subnet is associated with exactly one route table, and if you do not explicitly associate a subnet with another table, it uses the main route table.
- **Multiple subnet associations**: one route table can be associated with many subnets.
- **Default security group**: allows all outbound traffic and allows inbound traffic only from resources associated with the same security group.
- **Default network ACL**: allows all inbound and outbound traffic until you replace it with a custom NACL.
- **Default VPC**: includes a default subnet in each AZ, an internet gateway, default route table, default security group, and default network ACL.

> A subnet is not “public” just because its route table points to an Internet Gateway. For internet reachability, the instance also needs a public IPv4 address or an IPv6 address with appropriate routing.

**CIDR planning and subnet sizing**

CIDR planning affects scalability, routing, and future expansion. Choose a VPC range that leaves room for growth and for additional network segments such as shared services, private application tiers, and endpoints.

Each subnet reserves 5 IP addresses for AWS use:
- The first 4 addresses in the subnet range.
- The last address in the subnet range.

This reservation reduces usable capacity, which matters in small subnets and for services that consume many IPs, such as container platforms or interface endpoints.

Example:
- A `/28` subnet has 16 total addresses.
- AWS reserves 5.
- You get 11 usable addresses.

Practical design guidance:
- Use larger subnets than you think you need for private application tiers.
- Avoid overly small subnets for NAT gateways, interface endpoints, and autoscaling workloads.
- Reserve separate CIDR ranges for future expansion and network segmentation.

**Subnets, public access, and private access**

A subnet becomes effectively public when:
- Its route table has a route to an Internet Gateway.
- Instances in that subnet have public IPv4 addresses, or IPv6 addresses with Internet Gateway routing.

A subnet is private when it does not provide direct internet reachability for instances. Private subnets often route outbound internet traffic through a NAT gateway or use VPC endpoints for AWS service access.

Common patterns:
- **Public subnet**: load balancers, bastion hosts, NAT gateways.
- **Private application subnet**: app servers, containers, internal services.
- **Private data subnet**: databases, caches, storage proxies.

**Internet Gateway, NAT Gateway, and Elastic IP**

An <abbr title="Internet Gateway">IGW</abbr> enables internet connectivity for resources in a VPC that have public addresses and appropriate routes. It supports IPv4 and IPv6 traffic, but the resource still needs a public address or public-facing service path.

An <abbr title="Network Address Translation">NAT</abbr> gateway lets instances in private subnets initiate outbound IPv4 connections to the internet or AWS public endpoints while preventing unsolicited inbound internet traffic.

- **NAT gateway** lives in a public subnet.
- It uses an **Elastic IP** for a stable public IPv4 address.
- Private subnet route tables point `0.0.0.0/0` to the NAT gateway.
- For IPv6, use an egress-only internet gateway instead of NAT.

An <abbr title="Elastic IP">EIP</abbr> is a static public IPv4 address allocated to your account. Use it when you need a stable public endpoint, such as for a NAT gateway or a legacy public-facing instance.

Example private subnet route:
```text
Destination      Target
10.0.0.0/16      local
0.0.0.0/0        nat-0abc123def4567890
```

**Route tables and route propagation**

Route tables determine where traffic goes. They do not decide whether traffic is allowed; security groups and NACLs do that.

Important route table concepts:
- Every subnet has exactly one associated route table.
- A route table can be associated with multiple subnets.
- The `local` route is automatically created for the VPC CIDR.
- Static routes are manually added.
- Propagated routes can appear from VPN or Direct Connect attachments when enabled.

Typical routes:
- `local` for intra-VPC traffic.
- `0.0.0.0/0` to IGW for public subnets.
- `0.0.0.0/0` to NAT gateway for private subnets.
- Specific CIDRs to peering connections, Transit Gateway, VPN, or Direct Connect.

**Security groups and network ACLs**

Security groups are stateful. If you allow inbound traffic, the return traffic is automatically allowed. You usually attach them to ENIs (Elastic Network Interfaces), EC2 instances, load balancers, RDS, and other supported resources.

Network ACLs are stateless. You must explicitly allow both inbound and outbound traffic. They apply at the subnet boundary and evaluate rules in order.

Comparison:
| Feature | Security Group | Network ACL |
|---|---|---|
| Scope | Resource/ENI | Subnet |
| State | Stateful | Stateless |
| Rule order | All rules evaluated together | Ordered by number |
| Default behavior | Deny inbound, allow outbound | Allow all |
| Use case | Fine-grained instance control | Broad subnet guardrails |

Practical guidance:
- Use security groups for primary access control.
- Use NACLs for coarse subnet-level restrictions, compliance guardrails, or explicit deny requirements.
- Do not rely on flow logs alone to identify the exact rule that blocked traffic.

**VPC endpoints and private service access**

VPC endpoints let you access AWS services without traversing the public internet. This improves security and can reduce NAT gateway usage.

There are two main types:
- **Gateway endpoints** for Amazon S3 and DynamoDB.
- **Interface endpoints** powered by AWS PrivateLink for many other AWS services.

Gateway endpoint characteristics:
- Added as route table entries.
- Commonly used for S3 and DynamoDB access from private subnets.
- Can be controlled with endpoint policies.

Interface endpoint characteristics:
- Creates elastic network interfaces in your subnets.
- Uses security groups.
- Supports private DNS when enabled.
- Often requires DNS support and DNS hostnames enabled in the VPC.

Endpoint policy notes:
- Endpoint policies restrict which actions and resources are allowed through the endpoint.
- They are not a replacement for IAM policies.
- Use them to reduce blast radius for service access from a VPC.

Example interface endpoint creation:
```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Interface \
  --service-name com.amazonaws.us-east-1.ssm \
  --subnet-ids subnet-aaa111 subnet-bbb222 \
  --security-group-ids sg-0123456789abcdef0 \
  --private-dns-enabled
```

Private DNS caveats:
- VPC DNS support must be enabled.
- The service must support private DNS for the endpoint.
- Conflicting custom DNS settings can break name resolution if you do not plan resolver rules carefully.

**DNS, DHCP options, and Route 53 private hosted zones**

VPC DNS behavior is controlled by VPC attributes and DHCP options.

Key points:
- `enableDnsSupport` must be on for DNS resolution in the VPC.
- `enableDnsHostnames` must be on for instances to receive DNS hostnames where applicable.
- The default DHCP option for Amazon DNS is `AmazonProvidedDNS`.

If you replace the DHCP options set with custom DNS servers:
- Instances use those servers for name resolution.
- You may lose direct resolution of Route 53 private hosted zones unless you configure Route 53 Resolver forwarding or conditional forwarding correctly.

A Route 53 private hosted zone can be associated with one or more VPCs. This is a common pattern for internal service discovery.

Example association concept:
- Private hosted zone: `corp.internal`
- Associated VPCs: shared-services VPC, app VPC, analytics VPC
- Result: instances in associated VPCs can resolve internal names like `db.corp.internal`

**VPC peering**

VPC peering connects two VPCs privately using AWS networking. It is simple and low-latency, but it has important limits.

Core exam facts:
- **Non-transitive**: if VPC A peers with VPC B, and VPC B peers with VPC C, A cannot reach C through B.
- **No overlapping CIDRs**: the VPC CIDR ranges must not overlap.
- **Manual routing**: you must add routes in each VPC route table for the peer CIDR.
- **No edge-to-edge routing**: peering does not provide transitive access to VPN, Direct Connect, or other peering connections.

Example peering route update:
```text
VPC A route table
Destination      Target
10.1.0.0/16      local
10.2.0.0/16      pcx-0abc123def4567890

VPC B route table
Destination      Target
10.2.0.0/16      local
10.1.0.0/16      pcx-0abc123def4567890
```

When to use peering:
- Two VPCs need direct communication.
- You want a simple point-to-point connection.
- You do not need transitive routing or centralized inspection.

When not to use peering:
- Many VPCs need shared connectivity.
- You need hub-and-spoke routing.
- You need centralized inspection, segmentation, or route control.

**Transit Gateway**

<abbr title="Transit Gateway">TGW</abbr> is the hub for connecting many VPCs and on-premises networks. It is the preferred architecture when you need scalable hub-and-spoke connectivity.

Why use TGW:
- Centralized routing between many VPCs.
- Easier management than many peering connections.
- Supports VPC attachments, VPN attachments, and Direct Connect integration.
- Enables route tables on the TGW itself for segmentation and controlled propagation.

Core TGW concepts:
- **Attachment**: connects a VPC, VPN, or Direct Connect gateway to the TGW.
- **TGW route table**: controls which attachments can reach which destinations.
- **Propagation**: automatically learns routes from attachments.
- **Association**: determines which TGW route table an attachment uses for lookup.

Example hub-and-spoke idea:
- Shared services VPC attached to TGW.
- App VPCs attached to TGW.
- On-prem network attached via VPN or Direct Connect.
- TGW route tables control which spokes can talk to shared services and whether spokes can talk to each other.

Practical routing example:
- App VPC route table sends `10.50.0.0/16` to TGW.
- TGW route table sends `10.50.0.0/16` to the shared-services attachment.
- On-prem routes propagate into TGW from VPN or Direct Connect attachment.

**Site-to-Site VPN and Direct Connect**

AWS Site-to-Site VPN provides encrypted connectivity over the internet between your network and AWS. It commonly terminates on a Virtual Private Gateway or Transit Gateway.

Direct Connect provides a dedicated private network connection from your premises to AWS. It is used for more consistent throughput, lower jitter, and private connectivity requirements.

Integration patterns:
- **VPN to VPC**: attach a Virtual Private Gateway to the VPC and route on-prem CIDRs to it.
- **VPN to TGW**: attach the VPN to Transit Gateway for hub-and-spoke designs.
- **Direct Connect to VPC**: often uses a Direct Connect gateway and then routes into one or more VPCs through VGW or TGW.
- **Direct Connect to TGW**: common for centralized enterprise networking.

Route considerations:
- Static routes are simple but require manual maintenance.
- Route propagation reduces operational overhead when supported.
- You must ensure return routes exist on both sides.

Example hybrid routing concept:
```text
VPC route table
Destination      Target
10.0.0.0/16      local
192.168.0.0/16   tgw-0abc123def4567890

On-prem router
10.20.0.0/16     via Direct Connect
10.30.0.0/16     via VPN
```

**VPC sharing**

VPC sharing lets multiple AWS accounts use subnets in a centrally owned VPC. The VPC owner manages the shared network resources, while participant accounts launch resources into shared subnets.

Important distinctions:
- The VPC owner controls the VPC, route tables, IGW, NAT gateways, and shared subnet configuration.
- Participant accounts can create and manage their own resources in the shared subnets, including security groups for those resources.
- Shared subnets still follow the owner’s routing and network design.

Use cases:
- Central networking team owns the VPC.
- Application teams deploy workloads into shared private subnets.
- Security and governance remain centralized while account autonomy improves.

**Flow logs and observability**

VPC Flow Logs capture metadata about IP traffic going to and from network interfaces, subnets, or the VPC. They are useful for troubleshooting and auditing, but they do not show packet payloads.

What flow logs can tell you:
- Source and destination addresses.
- Source and destination ports.
- Protocol.
- Accepted or rejected status.
- Bytes and packets observed.

What they cannot tell you:
- The exact security group or NACL rule that matched.
- The packet contents.
- Definitive root cause in every case.

Interpretation guidance:
- `REJECT` indicates the traffic was denied somewhere in the path.
- `ACCEPT` means the traffic was accepted for logging purposes, but it does not prove the application succeeded.
- Use flow logs alongside security group rules, NACLs, route tables, and application logs.

Example flow log creation:
```bash
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-0123456789abcdef0 \
  --traffic-type ALL \
  --log-destination-type cloud-watch-logs \
  --log-group-name /aws/vpc/flowlogs \
  --deliver-logs-permission-arn arn:aws:iam::123456789012:role/FlowLogsRole
```

**Common exam scenarios**

1. You need private instances to download patches from the internet.
   - Place instances in a private subnet.
   - Route `0.0.0.0/0` to a NAT gateway in a public subnet.
   - Ensure the NAT gateway has an Elastic IP and the public subnet routes to an IGW.

2. You need internal AWS service access without internet exposure.
   - Use a gateway endpoint for S3 or DynamoDB.
   - Use an interface endpoint for services such as Systems Manager or Secrets Manager.
   - Enable private DNS when appropriate.

3. You need two VPCs to communicate directly.
   - Use VPC peering if the design is simple and non-transitive.
   - Add routes in both VPCs.
   - Ensure CIDRs do not overlap.

4. You need many VPCs and on-premises connectivity.
   - Use Transit Gateway.
   - Attach VPCs, VPN, and/or Direct Connect.
   - Control propagation and association with TGW route tables.

5. You need a stable public IP for outbound internet access from private subnets.
   - Use a NAT gateway with an Elastic IP.
   - Do not assign public IPs to private instances.

**CLI and configuration examples**

Create a route to a NAT gateway:
```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-0123456789abcdef0
```

Create a route to a peering connection:
```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 10.2.0.0/16 \
  --vpc-peering-connection-id pcx-0abc123def4567890
```

Create a gateway endpoint for S3:
```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Gateway \
  --service-name com.amazonaws.us-east-1.s3 \
  --route-table-ids rtb-0123456789abcdef0
```

**Troubleshooting checklist**

- Confirm the subnet route table association.
- Confirm the route target is correct.
- Confirm the instance has the right public or private addressing.
- Confirm security group inbound and outbound rules.
- Confirm NACL rules in both directions.
- Confirm DNS support and hostnames are enabled when using private DNS.
- Confirm endpoint policies, IAM permissions, and service-specific requirements.
- Confirm peering CIDRs do not overlap and routes exist on both sides.
- Confirm TGW route table association and propagation are configured correctly.

**Exam focus summary**

VPC questions often test whether you can choose the right connectivity model and identify the correct layer of control.

- Use **security groups** for stateful instance-level access control.
- Use **NACLs** for stateless subnet-level filtering.
- Use **NAT gateways** for private subnet outbound IPv4 internet access.
- Use **IGWs** for public internet connectivity with public addresses.
- Use **VPC endpoints** to keep AWS service traffic off the public internet.
- Use **VPC peering** for simple non-transitive VPC-to-VPC connectivity.
- Use **Transit Gateway** for scalable hub-and-spoke networking.
- Use **VPN** and **Direct Connect** for hybrid connectivity.
- Use **Route 53 private hosted zones** for internal DNS across associated VPCs.
