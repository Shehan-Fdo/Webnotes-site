## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**IAM execution permissions overview**

IAM execution permissions control what an AWS service can do on your behalf after it assumes a role. In this lab, the core idea is to separate two concerns:

- **Who can pass the role** to a service, controlled by the caller’s permissions such as `iam:PassRole`.
- **Which service can assume the role**, controlled by the role’s trust policy.
- **What the role can do once assumed**, controlled by identity-based policies attached to the role.

This distinction matters on the exam and in real deployments. A role can be perfectly permissioned and still fail if the trust policy is wrong, if the caller lacks `iam:PassRole`, or if downstream resources require additional permissions such as KMS key access.

**Core IAM concepts**

A role is an AWS identity with permissions that can be assumed temporarily. For execution scenarios, AWS services commonly assume roles to access other services on your behalf.

- **Trust policy** defines the principals allowed to assume the role.
- **Permissions policy** defines what actions the role can perform after assumption.
- **Pass role permission** allows a user, group, or role to delegate a role to an AWS service.
- **Session credentials** are temporary credentials issued when the service assumes the role.

A useful mental model is:

1. You create a role with the right permissions.
2. You allow the target AWS service to assume that role.
3. You allow the human or automation caller to pass that role to the service.

**Trust policy versus permissions policy**

The trust policy answers: “Who can assume this role?” The permissions policy answers: “What can this role do?”

Example trust policy for an EC2 instance role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example permissions policy for read-only access to an S3 bucket:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket"
      ],
      "Resource": "arn:aws:s3:::example-bucket"
    },
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

Important details:

- `s3:ListBucket` applies to the bucket ARN.
- `s3:GetObject` applies to object ARNs with `/*`.
- If objects use **SSE-KMS** encryption, the role also needs permission to use the KMS key, such as `kms:Decrypt`.
- If the caller is outside the role and must launch or attach it, the caller also needs `iam:PassRole`.

> A resource-based policy alone is not always sufficient. Access can still be blocked by the caller’s identity policy, permission boundaries, service control policies (SCPs), or KMS key policy requirements.

**`iam:PassRole` and why it matters**

`iam:PassRole` is one of the most tested IAM concepts for service execution. It does not grant the role’s permissions to the caller. Instead, it allows the caller to hand a role to an AWS service.

Example policy that allows a user to pass only one role to EC2:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "iam:PassRole",
      "Resource": "arn:aws:iam::123456789012:role/EC2S3ReadRole",
      "Condition": {
        "StringEquals": {
          "iam:PassedToService": "ec2.amazonaws.com"
        }
      }
    }
  ]
}
```

Key points:

- `iam:PassRole` is evaluated on the caller.
- `iam:PassedToService` narrows the permission to a specific AWS service.
- The role’s trust policy must still allow that service principal to assume the role.
- The role name and instance profile name are not always the same, so do not assume they are interchangeable.

**EC2 execution role pattern**

For EC2, you usually attach an IAM role to an instance through an instance profile. The instance profile is the container that EC2 uses to deliver the role to the instance.

Typical flow:

1. Create a role with an EC2 trust policy.
2. Attach permissions such as S3 read access.
3. Create or use an instance profile that contains the role.
4. Launch the instance with the instance profile.
5. Use the instance metadata service to retrieve temporary credentials.

CLI example:

```bash
aws iam create-role \
  --role-name EC2S3ReadRole \
  --assume-role-policy-document file://ec2-trust.json

aws iam attach-role-policy \
  --role-name EC2S3ReadRole \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

Launch example using an instance profile name:

```bash
aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --iam-instance-profile Name=EC2S3ReadInstanceProfile \
  --subnet-id subnet-0123456789abcdef0
```

Practical cautions:

- The `--iam-instance-profile` value must reference an existing **instance profile**, not just a role name.
- If you create the role with the console, AWS may create the instance profile for you.
- If the instance cannot access S3, check the role policy, bucket policy, KMS permissions, and network path such as VPC endpoints.

**Lambda execution role pattern**

Lambda functions use an execution role to call AWS services on your behalf. The role must trust the Lambda service principal.

Example trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

A common baseline permissions policy includes CloudWatch Logs access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    }
  ]
}
```

Important nuance:

- These logs permissions are often necessary, but they are not universally sufficient.
- Your function may also need permissions for S3, DynamoDB, SNS, SQS, KMS, Secrets Manager, or other downstream services.
- If the function is triggered by another AWS service, that trigger may require its own permissions and resource policy configuration.

Example of adding S3 read access to a Lambda execution role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

**S3 access and policy layering**

S3 authorization often involves multiple layers. A bucket policy can grant access, but the request may still fail if another layer denies it.

Common layers:

- **Identity-based policy** on the caller or role.
- **Bucket policy** on the S3 bucket.
- **KMS key policy** if objects are encrypted with SSE-KMS.
- **SCPs** in AWS Organizations.
- **Permission boundaries** on the principal.

Example bucket policy allowing a role to read objects:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowRoleRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/EC2S3ReadRole"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

Use this pattern when you want the bucket itself to grant access to a specific principal. Still verify:

- The principal can reach the bucket through its own permissions model.
- The object is not blocked by explicit denies.
- The KMS key allows decrypt if encryption is enabled.

**KMS permissions in execution workflows**

AWS Key Management Service (KMS) is a frequent hidden dependency. If a service role reads encrypted data, it may need KMS permissions in addition to service permissions.

Common actions:

- `kms:Decrypt` for reading encrypted data.
- `kms:Encrypt` for writing encrypted data.
- `kms:GenerateDataKey` for envelope encryption workflows.
- `kms:DescribeKey` for validation and metadata access.

Example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt",
        "kms:DescribeKey"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ab-cdef-1234567890ab"
    }
  ]
}
```

If access fails even though the IAM policy looks correct, check:

- The KMS key policy.
- Grants on the key.
- Whether the service role is allowed to use the key.
- Whether the request is coming from the expected AWS Region.

**Testing and simulation**

You can use IAM policy simulation to validate identity-based permissions before deploying. This is useful for checking whether a role or user can perform a specific action.

Example:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/EC2S3ReadRole \
  --action-names s3:GetObject s3:ListBucket \
  --resource-arns arn:aws:s3:::example-bucket arn:aws:s3:::example-bucket/test.txt
```

Use simulation carefully:

- It helps validate identity-based policy logic.
- It does not fully model every real-world authorization layer.
- It may not capture all effects of SCPs, permission boundaries, session policies, or some resource-based policy interactions unless you explicitly include them in your analysis.

For real troubleshooting, combine simulation with:

- CloudTrail event history.
- Service-specific error messages.
- IAM Access Analyzer.
- The AWS CLI `--debug` flag when appropriate.

**Troubleshooting workflow**

When execution permissions fail, isolate the failure by checking the layers in order.

1. Confirm the service is using the intended role.
2. Verify the trust policy includes the correct service principal.
3. Verify the caller has `iam:PassRole` if the role is being attached or passed.
4. Check the role’s permissions policy for the required action.
5. Check resource policies such as S3 bucket policies or KMS key policies.
6. Check for explicit denies from SCPs or permission boundaries.
7. Review CloudTrail for `AccessDenied` events and the exact denied action.

Common error patterns:

- `AccessDenied` on `sts:AssumeRole` usually points to a trust policy issue.
- `AccessDenied` when launching or configuring a service often points to missing `iam:PassRole`.
- `AccessDenied` on data access often points to missing service permissions or resource policy restrictions.
- `KMS.AccessDeniedException` usually means the role lacks key permissions or the key policy blocks access.

**Practical lab checklist**

Use this checklist when configuring execution permissions for a lab or exam scenario:

- Create the role with the correct trust policy for the target service.
- Attach the minimum permissions needed for the workload.
- Add KMS permissions if encrypted resources are involved.
- Ensure the caller can pass the role with `iam:PassRole`.
- Use the correct instance profile for EC2.
- Confirm the Lambda execution role includes logging and any downstream service permissions.
- Test with the actual service, not only with policy simulation.

**Example end-to-end pattern**

A common architecture is an EC2 instance that reads from S3:

- EC2 assumes an instance role through an instance profile.
- The role trusts `ec2.amazonaws.com`.
- The role has `s3:GetObject` and `s3:ListBucket`.
- The user launching the instance has `iam:PassRole` for that role.
- If the bucket uses SSE-KMS, the role also has `kms:Decrypt`.

Minimal CLI sequence:

```bash
aws iam create-role \
  --role-name EC2S3ReadRole \
  --assume-role-policy-document file://ec2-trust.json

aws iam put-role-policy \
  --role-name EC2S3ReadRole \
  --policy-name S3ReadPolicy \
  --policy-document file://s3-read.json

aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --iam-instance-profile Name=EC2S3ReadInstanceProfile
```

**Exam focus**

For SAA-C03, focus on these distinctions:

- **Trust policy** controls who can assume the role.
- **Permissions policy** controls what the role can do.
- **PassRole** controls who can delegate the role to a service.
- **Instance profile** is required for EC2 role attachment.
- **Resource policies** can grant access, but they do not always override other restrictions.
- **KMS** often explains “mystery” access failures.
