## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lambda fundamentals**

AWS Lambda is a serverless compute service that runs your code in response to events and automatically manages the underlying infrastructure. You use it when you want to execute short-lived, event-driven logic without provisioning or patching servers.

Common Lambda use cases include:

- API backends behind Amazon API Gateway
- File processing from Amazon S3 events
- Stream processing from Amazon Kinesis or DynamoDB Streams
- Scheduled jobs using Amazon EventBridge
- Automation tasks such as cleanup, notifications, and orchestration

Lambda is a strong fit when your workload is stateless, bursty, and can tolerate the execution model of short invocations. It is not a good fit for long-running processes, persistent local state, or workloads that need full OS control.

**Core concepts**

A Lambda function has a few important parts:

- **Function code**: The logic you write in a supported runtime such as Python, Node.js, Java, or .NET.
- **Handler**: The entry point Lambda calls when the function runs.
- **Execution role**: An AWS Identity and Access Management (IAM) role that grants the function permission to access other AWS services.
- **Trigger**: The event source that invokes the function.
- **Environment variables**: Key-value pairs used for configuration without hardcoding values in code.
- **Timeout and memory**: Runtime settings that affect performance, cost, and execution limits.

A Lambda invocation is typically stateless. If you need to preserve data between invocations, store it in an external service such as Amazon S3, Amazon DynamoDB, Amazon RDS, or Amazon ElastiCache.

**Creating a Lambda function**

You can create a function from the AWS Management Console, AWS Command Line Interface (CLI), AWS CloudFormation, AWS Serverless Application Model (SAM), or AWS CDK. In labs, the console is often the fastest way to understand the moving parts, while infrastructure as code is better for repeatable deployments.

A typical console workflow is:

1. Open the Lambda console and choose to create a function.
2. Select a runtime such as Python 3.x or Node.js.
3. Create or choose an execution role.
4. Add your code or upload a deployment package.
5. Configure basic settings such as memory, timeout, and environment variables.
6. Add a trigger if the function should respond to an event source.

A minimal Python example looks like this:

```python
def lambda_handler(event, context):
    return {
        "statusCode": 200,
        "body": "Hello from Lambda"
    }
```

A minimal Node.js example looks like this:

```javascript
exports.handler = async (event) => {
  return {
    statusCode: 200,
    body: "Hello from Lambda"
  };
};
```

The handler name must match the runtime configuration. For example, if your file is `index.py` and your function is `lambda_handler`, the handler setting is typically `index.lambda_handler`.

**Execution role and permissions**

Lambda uses an IAM execution role to call AWS APIs on your behalf. This role should follow the principle of least privilege, meaning it should only include the permissions the function actually needs.

A basic trust policy for Lambda allows the service to assume the role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

A common managed policy for logging is `AWSLambdaBasicExecutionRole`, which allows the function to write logs to Amazon CloudWatch Logs. If your function needs to read from S3, write to DynamoDB, or publish to SNS, add only those specific permissions.

Example inline policy for reading one S3 bucket:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::my-input-bucket/*"
    }
  ]
}
```

> If your function cannot write logs, first check the execution role. Missing CloudWatch Logs permissions are a common cause of silent-looking failures during testing.

**Invocation models**

Lambda supports different invocation patterns:

- **Synchronous invocation**: The caller waits for the response, such as API Gateway invoking a function.
- **Asynchronous invocation**: The event is queued and processed later, such as S3 or EventBridge events.
- **Poll-based invocation**: Lambda polls a source like Amazon SQS or DynamoDB Streams and processes batches.

These models matter because they affect error handling, retries, and response behavior. For example, synchronous invocations return errors directly to the caller, while asynchronous invocations may retry automatically depending on the source and configuration.

**Event payloads and context**

The `event` object contains the input data from the trigger. The `context` object contains runtime metadata such as request ID, remaining time, and function name.

Example Python handler that inspects both:

```python
import json

def lambda_handler(event, context):
    print("Request ID:", context.aws_request_id)
    print("Event:", json.dumps(event))
    return {
        "statusCode": 200,
        "body": json.dumps({
            "message": "Processed successfully"
        })
    }
```

You use the event payload to pass data into the function. For example, an API Gateway request includes HTTP details, while an S3 event includes bucket and object information.

**Testing and debugging**

You should test Lambda functions with representative sample events before wiring them to production triggers. The console test feature is useful for quick validation, but command-line and automated tests are better for repeatability.

Useful CLI commands include:

```bash
aws lambda list-functions
aws lambda get-function --function-name my-function
aws lambda invoke \
  --function-name my-function \
  --payload '{"name":"test"}' \
  response.json
cat response.json
```

For logs, use CloudWatch Logs:

```bash
aws logs describe-log-groups
aws logs tail /aws/lambda/my-function --follow
```

When debugging, check these areas first:

- **Handler mismatch**: The configured handler does not match the file or function name.
- **Permission errors**: The execution role lacks access to required services.
- **Timeouts**: The function takes longer than the configured timeout.
- **Memory pressure**: The function runs out of memory or becomes slow due to insufficient allocation.
- **Malformed event data**: The input structure does not match what your code expects.

**Configuration settings**

Lambda configuration affects both performance and cost. Memory allocation also influences CPU allocation, so increasing memory can improve execution speed for CPU-bound workloads.

Important settings include:

- **Timeout**: Maximum runtime for one invocation.
- **Memory size**: Controls available memory and indirectly CPU.
- **Ephemeral storage**: Temporary disk space in `/tmp` for intermediate files.
- **Environment variables**: Useful for stage names, table names, and feature flags.
- **Reserved concurrency**: Limits how many concurrent executions a function can use.
- **Provisioned concurrency**: Keeps pre-initialized execution environments ready to reduce cold starts.

A practical example of environment variables:

```bash
aws lambda update-function-configuration \
  --function-name my-function \
  --environment Variables={STAGE=dev,LOG_LEVEL=info}
```

Use environment variables for non-secret configuration. For secrets, prefer AWS Secrets Manager or AWS Systems Manager Parameter Store.

**Networking with VPC**

By default, Lambda functions can access public AWS endpoints and the internet through AWS-managed networking. If the function needs access to private resources in a Virtual Private Cloud (VPC), you attach it to subnets and security groups.

Use a VPC when the function must reach:

- Amazon RDS in private subnets
- Internal services behind private IPs
- Resources that are not publicly accessible

Be aware that VPC attachment can add complexity and may affect cold start time. If the function only needs AWS service APIs, avoid unnecessary VPC configuration unless required.

**Common lab workflow**

A typical lab for creating a Lambda function follows this pattern:

1. Create or select an IAM execution role with logging permissions.
2. Create the Lambda function with a runtime and handler.
3. Paste or upload the code.
4. Configure environment variables and timeout.
5. Add a trigger if needed.
6. Test the function with a sample event.
7. Review CloudWatch Logs for output and errors.
8. Adjust permissions or configuration based on the result.

This workflow helps you verify the full path from code to invocation to logging.

**Example: S3-triggered Lambda**

A common architecture is an S3 bucket that invokes Lambda when a new object is created. The function can validate uploads, resize images, or write metadata to DynamoDB.

Topology:

- User uploads file to S3
- S3 event notification invokes Lambda
- Lambda processes the object
- Lambda writes logs to CloudWatch Logs
- Lambda optionally stores results in DynamoDB or another bucket

Example Python code for an S3 event:

```python
import json

def lambda_handler(event, context):
    record = event["Records"][0]
    bucket = record["s3"]["bucket"]["name"]
    key = record["s3"]["object"]["key"]

    print(f"Bucket: {bucket}, Key: {key}")

    return {
        "statusCode": 200,
        "body": json.dumps({
            "bucket": bucket,
            "key": key
        })
    }
```

For S3 notifications, ensure the bucket permission and Lambda resource-based policy allow invocation. In practice, AWS manages much of this when you configure the trigger in the console, but you should still understand the permission boundary.

**Example: CLI deployment pattern**

A simple deployment flow from the CLI often looks like this:

```bash
zip function.zip index.py

aws lambda create-function \
  --function-name my-function \
  --runtime python3.12 \
  --role arn:aws:iam::123456789012:role/lambda-execution-role \
  --handler index.lambda_handler \
  --zip-file fileb://function.zip
```

After creation, you can update code without recreating the function:

```bash
zip function.zip index.py

aws lambda update-function-code \
  --function-name my-function \
  --zip-file fileb://function.zip
```

This pattern is useful when you want to separate code changes from configuration changes.

**Operational considerations**

Lambda is easy to start with, but production design still needs attention to reliability and observability.

Key operational points:

- **Cold starts**: The first invocation after idle time may take longer because Lambda initializes a new execution environment.
- **Retries**: Asynchronous sources may retry automatically, so design idempotent handlers.
- **Idempotency**: Your function should safely handle duplicate events.
- **Logging**: Use structured logs when possible to simplify troubleshooting.
- **Monitoring**: Track errors, duration, throttles, and concurrent executions in Amazon CloudWatch.
- **Dead-letter handling**: For some asynchronous workflows, configure failure destinations or dead-letter queues where appropriate.

Idempotency is especially important for event-driven systems. If the same event is delivered more than once, your function should not create duplicate records or repeat side effects unintentionally.

**Security best practices**

Lambda security starts with IAM and extends to code and configuration.

Follow these practices:

- Grant only the permissions the function needs.
- Store secrets outside code.
- Validate and sanitize input events.
- Use encryption for sensitive data at rest and in transit.
- Restrict network access when the function is attached to a VPC.
- Review resource-based policies if other AWS services invoke the function.

If the function is exposed through API Gateway, also consider authentication and authorization at the API layer rather than relying only on the function code.

**Exam focus**

For SAA-C03, you should be able to identify when Lambda is the right solution and how it integrates with other AWS services. Pay attention to these exam themes:

- Event-driven architecture
- Serverless compute selection
- IAM permissions for service integration
- Logging and monitoring with CloudWatch
- Triggers from S3, EventBridge, SQS, and API Gateway
- Configuration choices that affect performance and cost
- VPC access for private resources

A good exam answer usually favors Lambda when the requirement is to run code without managing servers, scale automatically, and respond to events with minimal operational overhead.

**Quick reference**

- Use Lambda for short-lived, stateless, event-driven tasks.
- Attach an execution role with least-privilege permissions.
- Check CloudWatch Logs first when debugging.
- Tune memory and timeout based on workload behavior.
- Use environment variables for configuration, not secrets.
- Add VPC only when private network access is required.
- Design for retries and duplicate events.
