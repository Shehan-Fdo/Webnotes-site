## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab overview**

This lab focuses on deploying an <abbr title="Function as a Service">FaaS</abbr> workload to AWS Lambda using the AWS Command Line Interface (<abbr title="Command Line Interface">CLI</abbr>) and then invoking it from the terminal. The core workflow is simple: create an execution role, package code, create the function, test it, and inspect logs and errors.

Lambda is useful when you want event-driven compute without managing servers. In practice, the CLI is valuable for repeatable deployments, automation, and exam scenarios where you need to reason about IAM permissions, runtime configuration, and invocation behavior.

- **AWS Lambda** runs code in response to events and scales automatically.
- **AWS CLI** lets you create, update, and invoke functions from a shell.
- **IAM role** permissions determine what the function can access at runtime.
- **CloudWatch Logs** capture function output and errors for troubleshooting.

**Prerequisites and setup**

Before you deploy, make sure the AWS CLI is configured with credentials and a default region. The CLI uses your profile settings to determine which account and region receive the Lambda resources.

```bash
aws configure
aws sts get-caller-identity
aws configure list
```

Typical setup checks include:

- Confirm the active AWS account with `aws sts get-caller-identity`.
- Confirm the region with `aws configure list` or `aws configure get region`.
- Ensure your IAM user or role has permissions for Lambda, IAM, and CloudWatch Logs.
- Use a region that supports the runtime you plan to deploy.

A minimal local project structure often looks like this:

```text
lambda-lab/
├── index.js
├── function.zip
└── trust-policy.json
```

If you use Python, the structure changes, but the deployment pattern stays the same: create a handler file, package it, and upload the ZIP archive.

**Lambda execution role**

Lambda needs an execution role so the service can assume permissions on behalf of your function. The role trust policy must allow the Lambda service principal to assume the role.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Create the role and attach the basic logging policy so the function can write to CloudWatch Logs.

```bash
aws iam create-role \
  --role-name lambda-execution-role \
  --assume-role-policy-document file://trust-policy.json

aws iam attach-role-policy \
  --role-name lambda-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole
```

Key points:

- **Trust policy** controls who can assume the role.
- **Permissions policy** controls what the function can do after it starts.
- `AWSLambdaBasicExecutionRole` is the standard managed policy for writing logs.
- If your function accesses S3, DynamoDB, or other services, you must add those permissions separately.

> If the role is missing logging permissions, the function may still run, but you will not see useful output in CloudWatch Logs.

**Write the function code**

The handler name must match the runtime’s expected entry point. For Node.js, the file and exported function determine the handler string. For Python, the module and function name determine the handler.

Example Node.js function:

```javascript
exports.handler = async (event) => {
  console.log("Received event:", JSON.stringify(event));
  return {
    statusCode: 200,
    body: JSON.stringify({
      message: "Hello from Lambda",
      input: event
    })
  };
};
```

Example Python function:

```python
import json

def lambda_handler(event, context):
    print("Received event:", json.dumps(event))
    return {
        "statusCode": 200,
        "body": json.dumps({
            "message": "Hello from Lambda",
            "input": event
        })
    }
```

Practical notes:

- **Handler** names must match the runtime configuration exactly.
- `console.log()` and `print()` are the fastest way to verify execution.
- Return a structured object if you want to simulate API Gateway-style responses.
- Keep the first version simple so you can isolate packaging or permission issues.

**Package and upload the code**

For ZIP-based deployment, package the handler file at the root of the archive. This is a common source of errors: if the file is nested inside another folder, Lambda may not find the handler.

```bash
zip function.zip index.js
```

For Python:

```bash
zip function.zip lambda_function.py
```

If you have dependencies, install them into a local directory first and zip the contents of that directory.

```bash
mkdir -p package
pip install requests -t package
cp lambda_function.py package/
cd package
zip -r ../function.zip .
```

Important packaging rules:

- The handler file must be at the archive root unless your handler path reflects a subdirectory.
- Native dependencies must match the Lambda runtime and architecture.
- Keep the deployment package as small as practical to reduce upload time and cold-start overhead.

**Create the Lambda function**

Use `aws lambda create-function` to deploy the ZIP archive. You must provide the function name, runtime, handler, role ARN, and code location.

```bash
aws lambda create-function \
  --function-name hello-cli-lambda \
  --runtime nodejs20.x \
  --handler index.handler \
  --role arn:aws:iam::123456789012:role/lambda-execution-role \
  --zip-file fileb://function.zip
```

For Python:

```bash
aws lambda create-function \
  --function-name hello-cli-lambda \
  --runtime python3.12 \
  --handler lambda_function.lambda_handler \
  --role arn:aws:iam::123456789012:role/lambda-execution-role \
  --zip-file fileb://function.zip
```

Useful options to know:

- `--timeout` sets the maximum execution time in seconds.
- `--memory-size` controls memory and indirectly affects CPU allocation.
- `--environment` injects environment variables.
- `--architectures` lets you choose `x86_64` or `arm64` if supported by your code.

Example with configuration settings:

```bash
aws lambda create-function \
  --function-name hello-cli-lambda \
  --runtime python3.12 \
  --handler lambda_function.lambda_handler \
  --role arn:aws:iam::123456789012:role/lambda-execution-role \
  --zip-file fileb://function.zip \
  --timeout 10 \
  --memory-size 256 \
  --environment Variables="{STAGE=dev,APP_NAME=notes}"
```

**Invoke the function**

You can invoke Lambda synchronously from the CLI and save the response to a file. This is the most common test pattern for a lab.

```bash
aws lambda invoke \
  --function-name hello-cli-lambda \
  --payload '{"name":"Orbit"}' \
  response.json
```

If you want to inspect the response immediately:

```bash
cat response.json
```

For better readability, use `jq` if it is installed:

```bash
cat response.json | jq
```

Invocation modes matter:

- **RequestResponse** is synchronous and returns the function result.
- **Event** is asynchronous and returns immediately after queuing the event.
- **DryRun** validates permissions and parameters without executing the function.

Example asynchronous invocation:

```bash
aws lambda invoke \
  --function-name hello-cli-lambda \
  --invocation-type Event \
  --payload '{"name":"Orbit"}' \
  response.json
```

Example dry run:

```bash
aws lambda invoke \
  --function-name hello-cli-lambda \
  --invocation-type DryRun \
  --payload '{}' \
  response.json
```

**Inspect function configuration**

After deployment, verify the function settings to confirm the runtime, handler, role, and timeout.

```bash
aws lambda get-function-configuration \
  --function-name hello-cli-lambda
```

You can also retrieve the full function metadata:

```bash
aws lambda get-function \
  --function-name hello-cli-lambda
```

Useful fields to check:

- **Runtime** matches the code you uploaded.
- **Handler** matches the file and exported function.
- **Role** is the correct execution role ARN.
- **LastModified** confirms the latest deployment.
- **Environment** contains the expected variables.

**View logs in CloudWatch**

Lambda writes logs to CloudWatch Logs when the execution role includes the basic logging policy. The log group name follows the pattern `/aws/lambda/<function-name>`.

```bash
aws logs describe-log-groups \
  --log-group-name-prefix /aws/lambda/hello-cli-lambda
```

To inspect recent log streams:

```bash
aws logs describe-log-streams \
  --log-group-name /aws/lambda/hello-cli-lambda \
  --order-by LastEventTime \
  --descending
```

To fetch log events from a stream:

```bash
aws logs get-log-events \
  --log-group-name /aws/lambda/hello-cli-lambda \
  --log-stream-name <log-stream-name>
```

What to look for:

- `START`, `END`, and `REPORT` lines indicate a successful invocation lifecycle.
- `Task timed out` indicates the function exceeded the configured timeout.
- `AccessDenied` often means the execution role lacks permissions.
- `Runtime.ImportModuleError` usually means the handler path or package layout is wrong.

**Update the function**

When code changes, you usually update the ZIP archive and push a new version of the code. You can also update configuration separately.

```bash
aws lambda update-function-code \
  --function-name hello-cli-lambda \
  --zip-file fileb://function.zip
```

Update configuration example:

```bash
aws lambda update-function-configuration \
  --function-name hello-cli-lambda \
  --timeout 15 \
  --memory-size 512 \
  --environment Variables="{STAGE=prod,APP_NAME=notes}"
```

Practical guidance:

- Use `update-function-code` for code-only changes.
- Use `update-function-configuration` for runtime, handler, memory, timeout, and environment changes.
- Re-invoke after each update to confirm the new behavior.

**Common troubleshooting patterns**

Most Lambda CLI lab issues fall into a few categories. Start with the simplest checks: packaging, handler name, role permissions, and region.

- **Handler mismatch**
  - The handler string does not match the file or function name.
  - Example: `index.handler` requires `index.js` exporting `handler`.
- **Bad ZIP structure**
  - The code file is nested inside a folder instead of being at the archive root.
- **Missing permissions**
  - The execution role cannot write logs or access downstream services.
- **Wrong region**
  - You created the function in one region and invoked it in another.
- **Timeout too low**
  - The function needs more time than the configured limit.
- **Memory too low**
  - The function may run slowly or fail under heavier workloads.

A quick validation sequence helps isolate the issue:

```bash
aws lambda get-function-configuration --function-name hello-cli-lambda
aws lambda invoke --function-name hello-cli-lambda --payload '{}' response.json
aws logs describe-log-streams --log-group-name /aws/lambda/hello-cli-lambda --order-by LastEventTime --descending
```

> If the function deploys successfully but invocation fails, check the CloudWatch log stream first. The error message there is usually more specific than the CLI response.

**Exam-relevant concepts**

This lab reinforces several SAA-C03 ideas that appear frequently in scenario questions.

- **Least privilege** applies to the execution role and any downstream service access.
- **Event-driven architecture** often uses Lambda as the compute layer.
- **Managed service integration** reduces operational overhead compared with EC2-based solutions.
- **Observability** depends on CloudWatch Logs and the correct IAM permissions.
- **Deployment automation** is easier when you understand CLI-based workflows.

**Practical command sequence**

A compact end-to-end workflow looks like this:

```bash
aws configure
aws iam create-role \
  --role-name lambda-execution-role \
  --assume-role-policy-document file://trust-policy.json

aws iam attach-role-policy \
  --role-name lambda-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

zip function.zip index.js

aws lambda create-function \
  --function-name hello-cli-lambda \
  --runtime nodejs20.x \
  --handler index.handler \
  --role arn:aws:iam::123456789012:role/lambda-execution-role \
  --zip-file fileb://function.zip

aws lambda invoke \
  --function-name hello-cli-lambda \
  --payload '{"test":"value"}' \
  response.json

cat response.json
```

**Key takeaways**

- Lambda deployment with the CLI is a repeatable sequence: configure, package, create role, create function, invoke, and inspect logs.
- The execution role is essential for both runtime permissions and CloudWatch logging.
- Handler names and ZIP structure are the most common deployment mistakes.
- Synchronous invocation is best for immediate validation; asynchronous invocation is best for event processing.
- CloudWatch Logs are the primary source of runtime troubleshooting information.
