## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lambda overview**

AWS Lambda is a serverless compute service that runs code in response to events without you managing servers. You package a function, choose a runtime, and Lambda handles provisioning, scaling, patching, and high availability across Availability Zones. For SAA-C03, the key idea is that Lambda is best for short-lived, event-driven, stateless workloads.

Common use cases include:

- API backends
- File processing
- Stream and queue consumers
- Scheduled jobs
- Automation and glue logic between AWS services

Lambda is not a fit for every workload. Long-running processes, workloads that need local disk persistence, or applications that require full OS control are usually better suited to EC2, containers, or other services.

**Execution model and runtime behavior**

A Lambda function executes inside an isolated runtime environment. The first invocation may incur a **cold start**, where Lambda initializes the execution environment and loads your code. Subsequent invocations may reuse the same environment, which can reduce latency.

Important runtime characteristics:

- **Statelessness** is the default design assumption.
- Temporary files can be stored in `/tmp`, but they are ephemeral.
- The execution environment may be reused, so cached objects can persist between invocations.
- You should not rely on in-memory state for correctness.

A typical handler in Python looks like this:

```python
import json

def lambda_handler(event, context):
    print("Received event:", json.dumps(event))
    return {
        "statusCode": 200,
        "headers": {"Content-Type": "application/json"},
        "body": json.dumps({"message": "Hello from Lambda"})
    }
```

For Node.js, the handler pattern is similar:

```javascript
export const handler = async (event) => {
  console.log("Received event:", JSON.stringify(event));
  return {
    statusCode: 200,
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ message: "Hello from Lambda" })
  };
};
```

**Invocation types and event flow**

Lambda supports multiple invocation patterns, and exam questions often hinge on the differences.

- **Synchronous invocation**
  - The caller waits for the response.
  - Examples: API Gateway, Application Load Balancer, direct SDK calls.
  - Errors are returned to the caller immediately.
  - Retries are usually handled by the caller or integration layer, not by Lambda itself.

- **Asynchronous invocation**
  - The event is queued by Lambda and processed later.
  - Examples: Amazon S3 notifications, Amazon EventBridge, SNS.
  - Lambda retries automatically on failure.
  - If processing still fails, you can route the event to a dead-letter queue or destination.

- **Poll-based invocation**
  - Lambda polls a source and processes batches.
  - Examples: Amazon SQS and Amazon Kinesis Data Streams.
  - Lambda manages the polling and scaling of consumers.
  - Retry behavior is tied to the source and batch handling semantics.

A useful exam shortcut:

- If the service pushes an event and the caller does not wait, think **asynchronous**.
- If Lambda reads from a queue or stream, think **poll-based**.
- If the caller expects an immediate response, think **synchronous**.

**Pricing and cost model**

Lambda pricing is based on two main dimensions:

- **Request count**
- **Duration**, measured in GB-seconds

The duration charge depends on the amount of memory you allocate and how long the function runs. More memory gives the function more CPU and network throughput, but it also increases cost per millisecond. This means memory tuning is both a performance and cost decision.

Key pricing concepts:

- You pay for each request after the free tier.
- You pay for compute time from the moment your code starts until it finishes, rounded according to Lambda billing rules.
- Higher memory allocation can reduce runtime, but the total cost may go up or down depending on the workload.
- Provisioned Concurrency is a separate feature and can add cost when you need predictable low latency.

A simplified mental model:

```text
Cost ≈ requests + (allocated memory × execution duration)
```

Free tier considerations:

- Lambda includes a monthly free tier for requests and compute.
- This is useful for low-volume workloads, prototypes, and intermittent automation.
- For exam scenarios, remember that “serverless” does not mean “free”; heavy traffic or long runtimes can still be expensive.

**Memory, CPU, and performance tuning**

Memory allocation is one of the most important Lambda tuning knobs. As memory increases, Lambda also allocates more CPU power and often more network throughput. This can reduce execution time for CPU-bound or network-heavy functions.

Use higher memory when:

- The function is CPU-bound, such as image processing or compression.
- The function makes many outbound network calls.
- You need lower latency and can trade memory for speed.

Use lower memory when:

- The function is simple and short-lived.
- Cost matters more than latency.
- The workload is mostly waiting on external services and does not benefit much from extra CPU.

A practical tuning workflow:

1. Start with a reasonable memory size.
2. Measure duration and cost.
3. Increase memory if runtime drops enough to offset the higher per-ms price.
4. Re-test after code changes or dependency updates.

**Event payload and response size limits**

Lambda has payload size limits that matter in architecture design.

- **Synchronous invocation request payload**: up to 6 MB
- **Synchronous invocation response payload**: up to 6 MB
- **Asynchronous invocation event payload**: up to 256 KB
- **Poll-based sources**: batch size and source-specific limits apply, but individual event records still have service constraints

These limits are common exam traps. If a payload is too large, you usually move the data to Amazon S3 and pass a pointer, such as an object key or pre-signed URL, instead of the full content.

Example pattern for large uploads:

- Client uploads file to S3.
- S3 event triggers Lambda.
- Lambda reads the object from S3.
- Lambda writes results to DynamoDB, S3, or another downstream service.

**Timeouts, concurrency, and scaling**

Lambda functions have a configurable timeout. If the function does not finish before the timeout, Lambda stops it. This is important when calling slow downstream systems or processing large batches.

Concurrency concepts:

- **Concurrent executions** are the number of functions running at the same time.
- Lambda scales automatically with demand.
- **Reserved concurrency** can cap a function’s maximum concurrency and also guarantee capacity.
- **Provisioned concurrency** keeps environments initialized to reduce cold starts.

Use reserved concurrency when:

- You need to protect a downstream database from overload.
- You want to isolate one function’s usage from others.

Use provisioned concurrency when:

- You need predictable low latency.
- You have latency-sensitive APIs.

**Retries, failure handling, and event age**

Retry behavior depends on the invocation type.

- **Synchronous**
  - The caller receives the error.
  - Lambda does not automatically retry the request in the same way as async processing.
  - The client, API Gateway, or application logic typically handles retries.

- **Asynchronous**
  - Lambda retries failed events automatically.
  - By default, Lambda makes two retry attempts after the initial attempt.
  - You can configure the maximum event age and retry attempts.
  - If the event expires before successful processing, it is discarded or routed according to your failure configuration.

- **Poll-based**
  - Lambda retries based on the source and batch behavior.
  - For SQS, failed messages return to the queue after the visibility timeout.
  - For Kinesis and DynamoDB Streams, a failed batch can block progress until the batch is handled or expires according to the event source mapping settings.

Event age matters for asynchronous invocations:

- Lambda can drop events that are too old if they exceed the configured maximum age.
- This prevents stale work from being processed long after the original event is relevant.
- In exam questions, this is often the right answer when you need bounded retry windows.

**Dead-letter queues and destinations**

Lambda supports two common failure-routing patterns: **dead-letter queues (DLQs)** and **destinations**.

DLQ details:

- A DLQ is typically an Amazon SQS queue or Amazon SNS topic.
- It captures failed asynchronous invocations after retries are exhausted.
- Use a DLQ when you mainly need to preserve failed events for later inspection or reprocessing.
- DLQs are a simpler, older pattern and are still relevant in exam questions.

Destinations are more flexible:

- Supported targets include Amazon SQS, Amazon SNS, Lambda, and EventBridge.
- You can configure **on-success** and **on-failure** destinations.
- Destinations can include richer invocation metadata than a DLQ.
- Use destinations when you need workflow routing, auditing, or downstream automation.

Exam-oriented distinction:

- Use a **DLQ** when you want a simple failure sink.
- Use **destinations** when you want explicit success/failure routing and richer event handling.

Example failure-routing decision:

- A batch image processor that only needs failed jobs stored for manual review can use an SQS DLQ.
- A workflow that must notify another service on success and send failures to an EventBridge bus is a better fit for destinations.

**Supported destination targets and routing**

Lambda destinations support these target types:

- Amazon SQS
- Amazon SNS
- Lambda
- Amazon EventBridge

Routing behavior:

- **On-success** sends the invocation result after successful completion.
- **On-failure** sends the failure context after retries are exhausted or the invocation fails.

This is useful when you want to decouple the function from follow-up processing. For example, a successful order validation function can send a message to SQS for downstream fulfillment, while failures can go to SNS for alerting.

**Function URLs and direct HTTP access**

Lambda function URLs provide a simple HTTPS endpoint directly on a Lambda function. They are useful when you want direct HTTP access without configuring API Gateway.

Typical use cases:

- Lightweight webhooks
- Simple internal tools
- Quick prototypes
- Low-overhead HTTP integrations

Important considerations:

- Function URLs are simpler than API Gateway.
- They are not a full API management solution.
- You still need to control access with IAM or a resource-based policy, depending on the auth mode.

Example URL pattern:

```text
https://abcde12345.lambda-url.us-east-1.on.aws/
```

**API Gateway to Lambda integration**

API Gateway is a common front door for Lambda-backed APIs. It handles routing, throttling, authentication, and request transformation, while Lambda handles business logic.

A typical architecture:

```text
Client -> API Gateway -> Lambda -> DynamoDB
```

Example setup flow:

1. Create a Lambda function.
2. Create an API Gateway HTTP API or REST API.
3. Add a route such as `GET /items`.
4. Integrate the route with the Lambda function.
5. Deploy the API.
6. Test the endpoint with `curl`.

Example request:

```bash
curl -X GET https://api-id.execute-api.us-east-1.amazonaws.com/items
```

Example Lambda proxy event shape:

```json
{
  "version": "2.0",
  "routeKey": "GET /items",
  "rawPath": "/items",
  "rawQueryString": "limit=10",
  "headers": {
    "host": "api-id.execute-api.us-east-1.amazonaws.com"
  },
  "requestContext": {
    "http": {
      "method": "GET",
      "path": "/items"
    }
  },
  "isBase64Encoded": false
}
```

Example proxy response:

```json
{
  "statusCode": 200,
  "headers": {
    "Content-Type": "application/json"
  },
  "body": "{\"items\":[\"a\",\"b\"]}"
}
```

For exam questions, remember that API Gateway is often the better choice when you need:

- Authentication and authorization
- Throttling and usage plans
- Request/response transformation
- API lifecycle management

**Lambda permissions and cross-account invocation**

Lambda uses IAM (Identity and Access Management) for execution permissions, and it also supports resource-based policies for who can invoke the function.

Two permission directions matter:

- **Execution role**
  - The role Lambda assumes to access AWS services on your behalf.
  - Example: read from S3, write to DynamoDB, publish to SNS.

- **Resource-based policy**
  - Grants other principals permission to invoke the function.
  - Common for API Gateway, EventBridge, S3, SNS, and cross-account access.

Cross-account invocation is a frequent exam topic:

- Account A can grant Account B permission to invoke a Lambda function.
- This is done with a resource-based policy on the function.
- The caller still needs the right IAM permissions in its own account.

Example permission command:

```bash
aws lambda add-permission \
  --function-name my-function \
  --statement-id allow-cross-account-invoke \
  --action lambda:InvokeFunction \
  --principal 123456789012 \
  --source-account 123456789012
```

**Environment variables and encryption**

Lambda environment variables are useful for configuration, but sensitive values should be protected. Lambda can encrypt environment variables at rest using AWS Key Management Service (KMS).

Key points:

- Lambda encrypts environment variables using an AWS-managed key by default.
- You can use a customer-managed KMS key for tighter control.
- The function’s execution role must have permission to use the key.
- Environment variables are not a substitute for a secrets manager when you need rotation or advanced secret handling.

Example environment variable usage:

```bash
aws lambda update-function-configuration \
  --function-name my-function \
  --environment Variables="{STAGE=prod,LOG_LEVEL=info}"
```

When to use each option:

- Use environment variables for non-secret configuration or simple secrets with encryption.
- Use AWS Secrets Manager or Systems Manager Parameter Store for managed secret retrieval and rotation.

**Networking and VPC access**

By default, Lambda functions run in an AWS-managed network environment and can reach public AWS endpoints and the internet if the service path allows it. When you attach a Lambda function to a VPC, it gains access to private resources such as Amazon RDS in private subnets, but networking becomes more complex.

VPC-related exam points:

- Lambda creates elastic network interfaces (ENIs) in your subnets.
- The function can reach private resources like RDS, ElastiCache, or internal services.
- If the function in a private subnet needs internet access, you usually need a NAT gateway or NAT instance.
- Security groups and subnet routing must allow the traffic.

Common trap:

- Putting Lambda in a VPC does not automatically give it internet access.
- If the function needs to call public APIs from private subnets, you need a NAT gateway or VPC endpoints where appropriate.

Example topology:

```text
Lambda in private subnet -> RDS in private subnet
Lambda in private subnet -> NAT Gateway -> Internet
Lambda in VPC -> Interface VPC Endpoint -> AWS service
```

Use VPC endpoints when:

- You want private access to AWS services without traversing the public internet.
- You want to reduce NAT gateway usage and cost.

**Integration patterns and event sources**

Lambda integrates with many AWS services. The integration style often determines retry behavior and scaling.

Common event sources:

- Amazon S3 object events
- Amazon SNS notifications
- Amazon EventBridge rules
- Amazon SQS queues
- Amazon Kinesis Data Streams
- DynamoDB Streams
- API Gateway and Application Load Balancer

Practical examples:

- Resize images after upload to S3.
- Process messages from SQS in batches.
- React to database change streams from DynamoDB.
- Run scheduled cleanup jobs with EventBridge.

A simple S3-triggered workflow:

```text
S3 PutObject -> Lambda -> Thumbnail in S3
```

A queue-based workflow:

```text
Producer -> SQS -> Lambda -> Database
```

**Operational best practices**

Good Lambda design improves reliability and cost efficiency.

- Keep functions small and focused.
- Make handlers idempotent so retries do not create duplicate side effects.
- Use structured logging for easier troubleshooting.
- Set appropriate timeouts and memory sizes.
- Avoid large deployment packages when possible.
- Use layers for shared dependencies if they reduce duplication.
- Monitor with CloudWatch metrics and logs.

Useful CloudWatch metrics include:

- `Invocations`
- `Errors`
- `Duration`
- `Throttles`
- `ConcurrentExecutions`
- `IteratorAge` for stream-based sources

**CLI examples for common tasks**

Create a function:

```bash
aws lambda create-function \
  --function-name hello-lambda \
  --runtime python3.12 \
  --handler app.lambda_handler \
  --role arn:aws:iam::123456789012:role/lambda-exec-role \
  --zip-file fileb://function.zip
```

Invoke synchronously:

```bash
aws lambda invoke \
  --function-name hello-lambda \
  --payload '{"name":"Ada"}' \
  response.json
```

Update memory and timeout:

```bash
aws lambda update-function-configuration \
  --function-name hello-lambda \
  --memory-size 512 \
  --timeout 15
```

Add a function URL:

```bash
aws lambda create-function-url-config \
  --function-name hello-lambda \
  --auth-type AWS_IAM
```

**Exam traps and quick distinctions**

- **Lambda vs EC2**
  - Lambda is event-driven and managed.
  - EC2 gives you full server control.

- **Lambda vs containers**
  - Lambda is best for short, stateless tasks.
  - Containers are better for long-running or custom runtime needs.

- **API Gateway vs Function URL**
  - API Gateway adds API management features.
  - Function URL is simpler and lighter.

- **DLQ vs destination**
  - DLQ stores failed events.
  - Destination routes success or failure outcomes with richer context.

- **VPC attachment**
  - Needed for private resources like RDS in private subnets.
  - Requires careful subnet, route, and NAT planning for internet access.

- **Payload limits**
  - Large payloads should go through S3, not directly through Lambda event bodies.

**Practical architecture example**

A common SAA-C03 scenario is an image upload pipeline:

1. A user uploads an image to S3.
2. S3 sends an event notification to Lambda.
3. Lambda validates the file and creates a thumbnail.
4. Lambda stores the thumbnail in another S3 bucket.
5. If processing fails, the event is routed to an SQS DLQ.
6. CloudWatch captures logs and metrics for monitoring.

This design works well because it is:

- Event-driven
- Scalable
- Cost-efficient
- Easy to decouple and monitor

**When Lambda is the right answer**

Choose Lambda when you need:

- Event-driven compute
- Automatic scaling
- Minimal operational overhead
- Short-lived processing
- Tight integration with AWS services

Avoid Lambda when you need:

- Long-running jobs beyond practical timeout limits
- Persistent local state
- Heavy OS-level customization
- Consistent low-latency execution without cold starts unless provisioned concurrency is acceptable
