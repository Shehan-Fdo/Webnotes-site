## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lambda concurrency fundamentals**

AWS Lambda concurrency is the number of in-flight invocations being processed at the same time in a Region. It is one of the most important scaling and protection controls for serverless architectures because it determines how much parallel work Lambda can execute and how quickly requests are accepted or delayed.

A single invocation consumes one unit of concurrency for the duration of its execution. If a function runs for 2 seconds and receives 100 requests per second, the concurrency demand is roughly 200 concurrent executions. This relationship is often summarized as:

```text
Concurrency ≈ Requests per second × Average duration in seconds
```

That simple formula is useful in exam questions because it helps you reason about whether a workload needs more concurrency, lower duration, buffering, or throttling.

- **Concurrency** is regional, not global.
- **Duration** directly affects how much concurrency you need.
- **Burst traffic** can exceed available concurrency even when average traffic looks safe.
- **Reserved concurrency** and **provisioned concurrency** solve different problems.

**Account-level concurrency quota**

Lambda has an account-level concurrency quota per Region. That regional limit is shared across all functions in the account in that Region. If one function consumes a large amount of concurrency, it reduces what remains available for other functions.

Reserved concurrency changes how that shared pool is divided:

- **Reserved concurrency** sets aside a fixed amount of concurrency for a specific function.
- That reserved amount is removed from the account’s unreserved pool.
- A function with reserved concurrency cannot use more than its reservation, even if unused capacity exists elsewhere in the account.
- Other functions cannot consume the reserved portion.

The remaining capacity is called **unreserved concurrency**. In practice, it is calculated as:

```text
Unreserved concurrency = Account regional concurrency limit - Sum of reserved concurrency allocations
```

AWS also keeps a minimum amount of unreserved concurrency available for the account unless you request an adjustment through support. This prevents a single function from consuming the entire regional quota and starving other functions. For exam purposes, remember that the unreserved pool is not always the full remainder you might expect if AWS has applied an internal reservation for the account.

> If you set reserved concurrency too aggressively, you can unintentionally starve the unreserved pool and throttle unrelated functions.

**Reserved concurrency behavior**

Reserved concurrency is both a limit and a guarantee.

- It guarantees capacity for a function when the account is busy.
- It caps the function so it cannot exceed the configured amount.
- It is the primary tool for isolating noisy neighbors inside the same account.

Use reserved concurrency when you need to:

- Protect downstream systems from a single function’s traffic spikes.
- Guarantee that a critical function always has capacity.
- Prevent one workload from consuming the entire regional quota.

A practical example:

```bash
aws lambda put-function-concurrency \
  --function-name order-processor \
  --reserved-concurrent-executions 50
```

This configuration ensures `order-processor` can run at most 50 concurrent executions and that 50 units are carved out of the account pool for that function.

**Provisioned concurrency and its relationship to reserved concurrency**

Provisioned concurrency pre-initializes execution environments so invocations can start with reduced cold-start latency. It is commonly used for latency-sensitive APIs and interactive workloads.

Important exam point:

- A function or alias using provisioned concurrency must also have enough reserved concurrency available.
- Reserved concurrency must be at least as large as the provisioned concurrency configured for that function or alias.
- If reserved concurrency is lower than provisioned concurrency, the configuration is invalid or cannot operate as intended because the function lacks enough concurrency capacity to hold the pre-warmed environments.

Provisioned concurrency is about performance predictability, not just capacity. Reserved concurrency is about capacity isolation. They are often used together for critical low-latency endpoints.

Example:

```bash
aws lambda put-provisioned-concurrency-config \
  --function-name checkout-api \
  --qualifier live \
  --provisioned-concurrent-executions 20
```

If `checkout-api` needs 20 provisioned environments, you must ensure the function or alias has at least 20 reserved concurrency available.

**Throttling and invocation type**

Lambda handles throttling differently depending on how the function is invoked.

**Synchronous invocations**

Synchronous callers wait for a response from Lambda, such as:

- API Gateway
- Application Load Balancer
- direct SDK calls
- Lambda function URLs

If concurrency is exhausted, synchronous invocations are throttled immediately. The caller receives an error response, typically `429 Too Many Requests` or a Lambda throttling error. There is no built-in queue inside Lambda for synchronous requests.

This matters for exam questions because the user experience is immediate failure unless the caller retries or another buffering layer exists.

**Asynchronous invocations**

Asynchronous sources include services such as EventBridge, SNS, and direct async Lambda invocation. Lambda accepts the event and places it into an internal queue for later processing.

If Lambda cannot process the event immediately:

- the event is buffered internally
- Lambda retries delivery automatically
- the caller does not wait for the function result

This is very different from synchronous throttling. The request is accepted first, then processed later. If retries are exhausted, the event can be sent to a destination or dead-letter queue depending on configuration.

**Poll-based event sources**

Poll-based sources include:

- Amazon SQS
- Amazon Kinesis Data Streams
- Amazon DynamoDB Streams

Lambda polls these sources and invokes the function with batches of records. If concurrency is constrained, the records remain in the source queue or stream until Lambda can process them. The source itself acts as the buffer.

This is a common exam distinction:

- Synchronous: immediate throttle to caller
- Asynchronous: internal queue and retry
- Poll-based: source backlog accumulates until processed

**Event source mapping scaling behavior**

Event source mappings control how Lambda reads from poll-based sources and how much concurrency it uses.

**Amazon SQS**

SQS is a queue, so it naturally buffers work. Lambda scales by increasing the number of concurrent pollers as backlog grows, up to the function’s concurrency limits and the event source mapping settings.

Key points:

- Larger backlog generally leads to more Lambda concurrency.
- Batch size affects how many messages each invocation processes.
- A larger batch can improve throughput but may increase per-invocation duration.
- If you enable **partial batch response**, Lambda can return only failed messages for retry instead of failing the entire batch.
- Newer **maximum concurrency** settings on the event source mapping let you cap how many concurrent batches Lambda processes from that queue.

That maximum concurrency setting is useful when you want to protect a downstream database or API even though the queue may contain a large backlog. It gives you a queue-level throttle without needing to reduce the function’s global reserved concurrency too aggressively.

Example configuration pattern:

```bash
aws lambda create-event-source-mapping \
  --function-name process-orders \
  --event-source-arn arn:aws:sqs:us-east-1:123456789012:orders-queue \
  --batch-size 10 \
  --maximum-batching-window-in-seconds 5 \
  --scaling-config '{"MaximumConcurrency":20}'
```

Practical interpretation:

- `batch-size 10` means each invocation can receive up to 10 messages.
- `MaximumConcurrency: 20` caps the queue mapping at 20 concurrent batches.
- With partial batch response, failed messages can be retried without reprocessing the whole batch.

**Kinesis Data Streams and DynamoDB Streams**

Stream sources are constrained by shard processing order.

- A shard must be processed in order.
- Lambda cannot arbitrarily parallelize records within the same shard without breaking ordering guarantees.
- Concurrency is therefore tied to shard count and shard-level processing constraints.

Exam implications:

- More shards can increase parallelism.
- A hot shard can become a bottleneck.
- If ordering matters, you trade throughput for sequence integrity.
- For DynamoDB Streams, the same shard-ordering constraints apply because the stream is partitioned and ordered per shard.

**SQS scaling versus stream scaling**

| Source type | Buffering model | Concurrency driver | Ordering constraint |
|---|---|---|---|
| SQS | Queue backlog | Backlog, batch size, mapping concurrency | No strict ordering unless FIFO semantics are used |
| Kinesis Data Streams | Stream shards | Number of shards and shard processing rate | Ordered per shard |
| DynamoDB Streams | Stream shards | Number of shards and shard processing rate | Ordered per shard |

**Dead-letter queues and destinations**

For asynchronous Lambda invocations, **dead-letter queues** and **destinations** solve different problems.

**Dead-letter queue (DLQ)**

A DLQ receives the original event after Lambda has exhausted retries for an asynchronous invocation. It is typically an SQS queue or SNS topic.

Use a DLQ when you want:

- the failed original event preserved for later inspection
- a simple failure sink for async invocations
- a retry-exhausted event archive

**Destinations**

Destinations are more flexible and can receive a richer invocation record after success or failure. They can target services such as SQS, SNS, EventBridge, or Lambda.

Use destinations when you want:

- success and failure routing
- metadata about the invocation outcome
- downstream automation based on result status

Important distinction:

- A DLQ receives the original event payload.
- A destination receives a structured invocation record that includes context about the result, and for failure cases can include the original event as part of the record.

For exam questions, choose DLQ when the requirement is “store the failed event.” Choose destinations when the requirement is “route success or failure outcomes with metadata.”

**Practical configuration examples**

Reserved concurrency:

```bash
aws lambda put-function-concurrency \
  --function-name image-resizer \
  --reserved-concurrent-executions 10
```

Provisioned concurrency on an alias:

```bash
aws lambda put-provisioned-concurrency-config \
  --function-name image-resizer \
  --qualifier prod \
  --provisioned-concurrent-executions 5
```

SQS event source mapping with batch and concurrency controls:

```bash
aws lambda create-event-source-mapping \
  --function-name process-payments \
  --event-source-arn arn:aws:sqs:us-east-1:123456789012:payments-queue \
  --batch-size 10 \
  --scaling-config '{"MaximumConcurrency":15}'
```

Asynchronous invocation with a destination:

```json
{
  "FunctionName": "report-generator",
  "InvocationType": "Event",
  "Payload": "{\"reportId\":\"r-123\"}"
}
```

**Exam scenario patterns**

When you see a question about protecting a backend from spikes, think about where to absorb the load.

- Use **API Gateway throttling** when you need to limit request rate at the edge before Lambda is invoked.
- Use **reserved concurrency** when you need to cap a function and protect downstream dependencies.
- Use **SQS buffering** when you want to absorb bursts and process work asynchronously.
- Use **provisioned concurrency** when latency and cold starts matter for a critical endpoint.
- Use **event source mapping maximum concurrency** when you need to limit how aggressively Lambda drains a queue.

Example scenario:

A checkout API receives unpredictable bursts during promotions. The business wants low latency, but the payment database can only handle 30 concurrent writes.

A strong design is:

1. Put API Gateway in front of Lambda and configure throttling to limit request rate.
2. Use reserved concurrency on the payment Lambda to cap concurrent executions at 30.
3. If the workload can tolerate asynchronous processing, place requests onto SQS and let Lambda drain the queue with a controlled maximum concurrency.
4. If the API must remain interactive and low-latency, add provisioned concurrency to reduce cold starts.

This is the kind of tradeoff the exam expects you to recognize: edge throttling controls request rate, reserved concurrency controls function execution, SQS absorbs bursts, and provisioned concurrency improves responsiveness.

**Operational considerations**

Concurrency tuning is not only about performance; it is also about failure isolation and cost control.

- Higher concurrency can increase pressure on databases, caches, and third-party APIs.
- Lower concurrency can increase queue depth and end-to-end latency.
- Reserved concurrency can unintentionally throttle a function if set too low.
- Provisioned concurrency adds cost because environments are kept ready.
- SQS plus Lambda is often the safest pattern for bursty workloads because the queue decouples producers from consumers.

Monitor these metrics when troubleshooting:

- `ConcurrentExecutions`
- `UnreservedConcurrentExecutions`
- `Throttles`
- `Duration`
- `IteratorAge` for stream sources
- SQS queue depth and age of oldest message

A rising `Throttles` metric with a flat `ConcurrentExecutions` line often indicates a concurrency cap. A growing queue depth with stable Lambda concurrency often indicates the mapping is intentionally capped or the function is too slow.

**Quick decision guide**

- Use **reserved concurrency** to isolate and cap a function.
- Use **provisioned concurrency** to reduce cold starts.
- Use **SQS** to buffer bursts and decouple producers from consumers.
- Use **maximum concurrency on SQS mappings** to control drain rate.
- Use **DLQ** for failed async events you want to preserve.
- Use **destinations** for richer success/failure routing.
- Use **API Gateway throttling** to stop traffic before it reaches Lambda.

**Key exam takeaways**

- Lambda concurrency is regional and shared across functions.
- Reserved concurrency carves out capacity and reduces the unreserved pool.
- AWS maintains some unreserved concurrency for the account unless adjusted through support.
- Synchronous invocations throttle immediately when concurrency is exhausted.
- Asynchronous invocations are queued and retried.
- Poll-based sources buffer in the source and are limited by mapping and shard behavior.
- SQS supports batch size, partial batch response, and maximum concurrency controls.
- Kinesis and DynamoDB Streams are constrained by shard ordering.
- DLQs store failed original events; destinations route invocation outcomes with metadata.
- Provisioned concurrency requires enough reserved concurrency to support it.
