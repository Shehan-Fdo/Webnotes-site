## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**API Gateway in the AWS architecture**

Amazon API Gateway is a managed service for creating, publishing, securing, monitoring, and versioning APIs. In architecture questions, it often sits in front of Lambda, ECS, EC2, or on-premises services to provide a stable API layer while the backend evolves independently.

Use API Gateway when you need:
- A front door for serverless or microservice backends.
- Request throttling, authentication, and authorization at the API edge.
- A managed way to expose RESTful or HTTP-based endpoints without running API servers yourself.
- A consistent public or private API surface with observability and deployment stages.

API Gateway is not the same as a load balancer. A load balancer distributes traffic to targets, while API Gateway understands API methods, resources, request/response mapping, authorizers, usage plans, and API-specific controls.

**Core API types and when to choose them**

AWS commonly tests the difference between **REST APIs** and **HTTP APIs**.

| Capability | REST API | HTTP API |
|---|---:|---:|
| Primary use case | Full-featured API management | Lower-cost, lower-latency APIs |
| Request validation | Yes | Limited compared with REST API |
| API keys and usage plans | Yes | No |
| Caching | Yes | No |
| Custom authorizers | Yes | Yes, with simpler support model |
| JWT authorizers | Not native in the same way as HTTP API | Yes |
| Private endpoints | Yes | Yes |
| Feature depth | Highest | Simpler and cheaper |

Choose **REST API** when you need advanced API management features such as API keys, usage plans, request validation, caching, or detailed mapping behavior. Choose **HTTP API** when you want a simpler, cheaper API for Lambda or HTTP backends and do not need the full REST API feature set.

> For exam questions, do not assume “API Gateway” always means REST API. The feature set matters.

**API Gateway resource model for REST APIs**

REST APIs use a hierarchy that is frequently tested:
- **API**: The top-level container.
- **Resources**: URL paths such as `/orders` or `/users/{id}`.
- **Methods**: HTTP verbs such as `GET`, `POST`, `PUT`, or `DELETE` attached to a resource.
- **Integrations**: The backend target for a method.
- **Method responses**: What the client can receive from the method.
- **Integration responses**: How backend responses are transformed before returning to the client.

A deployable REST API usually requires all of the following:
- A REST API.
- At least one resource.
- At least one method on that resource.
- An integration for the method.
- A deployment to a stage.
- A Lambda permission if the backend is Lambda.

This distinction matters because a resource without a method does nothing, and a method without an integration cannot reach a backend.

**Common integration types**

API Gateway can integrate with several backend styles:
- **Lambda proxy integration**: API Gateway forwards the request to Lambda with minimal transformation.
- **Lambda non-proxy integration**: You define mapping templates and can transform requests and responses.
- **HTTP integration**: API Gateway forwards to an HTTP endpoint such as an internal service or ALB.
- **AWS service integration**: API Gateway calls another AWS service directly.
- **Mock integration**: Useful for testing or static responses.

For architecture design:
- Use **Lambda proxy** for simple serverless APIs.
- Use **HTTP integration** when fronting an existing service.
- Use **non-proxy** when you need strict request/response shaping or legacy payload transformation.

**End-to-end example: REST API to Lambda**

A minimal deployable CloudFormation example needs the API, resource, method, integration, deployment, stage, and Lambda permission.

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: API Gateway REST API integrated with Lambda

Resources:
  OrdersFunction:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: orders-handler
      Runtime: nodejs20.x
      Handler: index.handler
      Role: arn:aws:iam::123456789012:role/lambda-execution-role
      Code:
        ZipFile: |
          exports.handler = async (event) => {
            return {
              statusCode: 200,
              headers: { "Content-Type": "application/json" },
              body: JSON.stringify({ message: "ok" })
            };
          };

  OrdersApi:
    Type: AWS::ApiGateway::RestApi
    Properties:
      Name: orders-api

  OrdersResource:
    Type: AWS::ApiGateway::Resource
    Properties:
      RestApiId: !Ref OrdersApi
      ParentId: !GetAtt OrdersApi.RootResourceId
      PathPart: orders

  GetOrdersMethod:
    Type: AWS::ApiGateway::Method
    Properties:
      RestApiId: !Ref OrdersApi
      ResourceId: !Ref OrdersResource
      HttpMethod: GET
      AuthorizationType: NONE
      Integration:
        Type: AWS_PROXY
        IntegrationHttpMethod: POST
        Uri: !Sub >
          arn:aws:apigateway:${AWS::Region}:lambda:path/2015-03-31/functions/${OrdersFunction.Arn}/invocations

  OrdersLambdaPermission:
    Type: AWS::Lambda::Permission
    Properties:
      FunctionName: !Ref OrdersFunction
      Action: lambda:InvokeFunction
      Principal: apigateway.amazonaws.com
      SourceArn: !Sub arn:aws:execute-api:${AWS::Region}:${AWS::AccountId}:${OrdersApi}/*/GET/orders

  ApiDeployment:
    Type: AWS::ApiGateway::Deployment
    DependsOn: GetOrdersMethod
    Properties:
      RestApiId: !Ref OrdersApi
      StageName: prod
```

Important implementation notes:
- `ParentId` must be the actual root resource ID, not the literal string `root`.
- `AWS::ApiGateway::Deployment` creates a snapshot of the API configuration.
- Stage configuration is separate from the deployment snapshot, even if a template creates both together.
- Lambda permission is required so API Gateway can invoke the function.

**CLI workflow for REST APIs**

A typical REST API build sequence is:
1. Create the REST API.
2. Discover the root resource ID.
3. Create a child resource.
4. Attach a method.
5. Create an integration.
6. Deploy to a stage.
7. Add permissions and test.

Example commands:

```bash
aws apigateway create-rest-api --name orders-api

aws apigateway get-resources --rest-api-id abc123

aws apigateway create-resource \
  --rest-api-id abc123 \
  --parent-id a1b2c3 \
  --path-part orders

aws apigateway put-method \
  --rest-api-id abc123 \
  --resource-id d4e5f6 \
  --http-method GET \
  --authorization-type NONE

aws apigateway put-integration \
  --rest-api-id abc123 \
  --resource-id d4e5f6 \
  --http-method GET \
  --type AWS_PROXY \
  --integration-http-method POST \
  --uri arn:aws:apigateway:us-east-1:lambda:path/2015-03-31/functions/arn:aws:lambda:us-east-1:123456789012:function:orders-handler/invocations

aws apigateway create-deployment \
  --rest-api-id abc123 \
  --stage-name prod
```

Use the actual root resource ID from `get-resources`; do not use the literal string `root` for `--parent-id`.

**HTTP API architecture patterns**

HTTP APIs are often the better choice when you want:
- Lower cost.
- Lower latency.
- Simpler routing.
- Native JWT authorization for modern identity providers.
- Straightforward Lambda or HTTP backend exposure.

HTTP APIs are commonly used for:
- Serverless applications.
- Lightweight microservices.
- Simple public APIs.
- Internal APIs where advanced REST API features are unnecessary.

They are less suitable when you need:
- API keys and usage plans.
- Detailed request validation.
- Advanced transformation behavior.
- API Gateway caching.

**Security and authorization**

API Gateway supports multiple security patterns:
- **IAM authorization**: Clients sign requests with AWS Signature Version 4.
- **Lambda authorizers**: Custom authorization logic in Lambda.
- **JWT authorizers**: Common with HTTP APIs and OpenID Connect or OAuth 2.0 providers.
- **Cognito authorizers**: Useful for user authentication in REST APIs.
- **Resource policies**: Restrict who can invoke the API based on account, source IP, VPC endpoint, or other conditions.

For exam scenarios:
- Use **IAM** for AWS-to-AWS access or internal AWS clients.
- Use **JWT** for modern web/mobile apps with an external identity provider.
- Use **Lambda authorizers** when you need custom token validation or fine-grained logic.
- Use **resource policies** to restrict access to private APIs or specific networks.

**Private APIs versus private integrations**

These are different concepts and are often confused.

- **Private API**: The API itself is only reachable through an **interface VPC endpoint** (`execute-api`).
- **Private integration**: API Gateway connects to a private backend, often through a **VPC link**.

A private API usually also needs a **resource policy** to restrict access to the VPC endpoint or specific principals. Without the policy, the design is incomplete for access control.

A private integration does not make the API private to callers. It only makes the backend connection private.

**VPC link and private backends**

Use a **VPC link** when API Gateway must reach private resources inside a VPC, such as:
- An internal Application Load Balancer (ALB).
- A Network Load Balancer (NLB).
- Private HTTP services.

Typical pattern:
- Client calls API Gateway.
- API Gateway uses a VPC link to reach an internal load balancer.
- The load balancer forwards traffic to private targets in subnets.

This is different from a private API, where the client itself reaches API Gateway through an interface VPC endpoint.

**Custom domains and base path mapping**

Custom domains help you present a stable API URL such as `api.example.com` instead of the default execute-api hostname.

Use custom domains when you need:
- A branded endpoint.
- Versioned APIs under different base paths.
- A stable URL across deployments or stages.

Base path mapping lets you route parts of a custom domain to different APIs or stages.

Example:
- `api.example.com/v1` → REST API stage `prod`
- `api.example.com/v2` → a newer API or stage

This is useful for versioning and gradual migration.

**Throttling, quotas, and caching**

API Gateway can protect backends and shape traffic:
- **Throttling** limits request rate and burst capacity.
- **Usage plans** and **API keys** help meter and control consumer access for REST APIs.
- **Caching** reduces backend load and improves latency for repeated requests.

Use these when:
- You need to protect a fragile backend.
- You want to enforce per-client consumption limits.
- You have read-heavy traffic with repeated responses.

Do not confuse API Gateway throttling with backend autoscaling. Throttling protects the API layer; autoscaling protects the compute layer.

**Request and response transformation**

REST APIs can transform payloads between client and backend using mapping templates and parameter mappings.

Use transformation when:
- The backend expects a different schema than the client sends.
- You need to normalize legacy payloads.
- You want to hide backend implementation details.

Lambda proxy integration reduces the need for mapping because the Lambda function receives the full request context.

**Observability and auditing**

API Gateway provides operational visibility through:
- **CloudWatch Logs** for execution and access logs.
- **CloudWatch Metrics** for latency, 4XX/5XX errors, count, and throttles.
- **X-Ray** for tracing request paths when enabled.
- **CloudTrail** for control-plane auditing such as API creation, updates, and deletions.

Use CloudTrail when you need to answer “who changed the API configuration?” rather than “what happened to a request?”

**Deployment and stage behavior**

A REST API deployment is a snapshot of the API configuration at a point in time. Stages point to deployments and can have stage variables, logging settings, throttling settings, and canary settings.

Key implications:
- Updating a method or integration does not affect live traffic until you create a new deployment.
- A stage can be reused across deployments.
- Stage variables can parameterize backend values in some designs.

This is a common exam trap: deployment and stage are related, but they are not the same thing.

**When to choose API Gateway versus alternatives**

Choose API Gateway when you need:
- API-specific features such as auth, throttling, usage plans, or request validation.
- A managed API front door for Lambda or HTTP backends.
- A serverless-friendly API layer.

Consider other services when:
- You only need Layer 4 or Layer 7 load balancing for web traffic, where ALB may be better.
- You need direct private connectivity to a service inside a VPC, where VPC link plus API Gateway or an internal ALB may fit.
- You need event-driven asynchronous processing, where SQS or EventBridge may be more appropriate.

**Practical architecture examples**

- **Serverless CRUD API**
  - API Gateway REST API or HTTP API.
  - Lambda for business logic.
  - DynamoDB for persistence.
  - Cognito or JWT authorizer for user auth.

- **Internal enterprise API**
  - Private API with interface VPC endpoint.
  - Resource policy restricting access to specific VPC endpoints or accounts.
  - Lambda or private HTTP backend.

- **Legacy service modernization**
  - API Gateway REST API.
  - HTTP integration to an internal service behind an ALB.
  - Custom domain and base path mapping for versioning.

- **Partner-facing API**
  - REST API with usage plans and API keys.
  - Throttling and quotas.
  - CloudWatch alarms for abuse or unexpected traffic.

**Exam-focused distinctions to remember**

- **Private API** means the client reaches API Gateway through an interface VPC endpoint.
- **Private integration** means API Gateway reaches a private backend through a VPC link.
- **REST API** has API keys, usage plans, caching, and richer request/response mapping.
- **HTTP API** is simpler, cheaper, and often preferred for modern Lambda or HTTP backends.
- **Deployment** is a snapshot; **stage** is the named environment pointing to a deployment.
- **CloudTrail** records API Gateway control-plane changes.
- **Lambda permission** is required for API Gateway to invoke a Lambda function.

**Common pitfalls**

> Do not treat API Gateway as a generic proxy without considering methods, integrations, stages, and permissions. For REST APIs, those pieces must fit together for the API to work.

- Forgetting Lambda permission for API Gateway invocation.
- Using `root` literally instead of the actual root resource ID.
- Confusing private APIs with private integrations.
- Assuming a deployment automatically updates live traffic without creating a new snapshot.
- Choosing REST API when HTTP API is sufficient and simpler.
- Expecting API keys and usage plans on HTTP APIs.
