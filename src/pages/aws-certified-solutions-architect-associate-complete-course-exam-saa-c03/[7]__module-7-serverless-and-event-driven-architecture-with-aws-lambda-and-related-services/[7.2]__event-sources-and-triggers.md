## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Event-driven architecture overview**

AWS event sources and triggers determine how work starts, how messages move, and where failures are handled. For the exam, you should distinguish between **push-based**, **poll-based**, and **scheduled** patterns, then choose the service that best matches ordering, durability, fan-out, filtering, and retry requirements.

A useful mental model is:

- **Push**: a service sends an event to a target when something happens.
- **Poll**: a consumer reads from a stream or queue on a schedule or via an event source mapping.
- **Async**: a service accepts work, stores it, and retries delivery later if needed.
- **Schedule**: a timer starts work at a specific time or interval.

**Core trigger categories**

- **Push-based sources** deliver events directly to a target.
  - Common examples: S3 event notifications, EventBridge rules, SNS subscriptions, API Gateway, and Application Load Balancer (ALB) target groups.
- **Poll-based sources** are read by consumers.
  - Common examples: SQS, Kinesis Data Streams, DynamoDB Streams, Amazon MQ, and Amazon MSK.
- **Scheduled sources** start work on a time basis.
  - Common examples: EventBridge Scheduler and EventBridge rules with cron or rate expressions.
- **Asynchronous invocation** is a delivery mode, not a source.
  - Common examples: Lambda async invocation from S3, SNS, EventBridge, or direct service integrations.

**Lambda as a target**

AWS Lambda is a common target for both push and poll integrations. The exam often tests whether you know how the upstream service behaves before Lambda runs.

- **Push to Lambda** works well when the source can invoke Lambda directly.
  - Examples: S3 notifications, SNS subscriptions, EventBridge rules, API Gateway, and ALB.
- **Poll to Lambda** uses an event source mapping.
  - Examples: SQS, Kinesis Data Streams, DynamoDB Streams, Amazon MQ, and Amazon MSK.
- **Async Lambda invocation** is useful when the caller should not wait for completion.
  - Lambda stores the event and retries on failure according to its async retry policy.

> If you need a durable buffer between producers and consumers, use **SNS to SQS** or **SQS directly**, not just a direct Lambda subscription. SNS-to-Lambda is still valid, but SQS gives you per-consumer buffering and backpressure control.

**Amazon S3 event notifications**

S3 can emit notifications when objects are created or removed. This is a classic exam topic because it is simple, cheap, and highly targeted.

- **Object-created events** include uploads, copies, multipart completions, and restores.
- **Object-removed events** include deletes and lifecycle expirations.
- **Prefix and suffix filters** let you route only matching objects.
  - Example: trigger only for `images/` uploads or only for `.csv` files.
- **Typical use cases**:
  - Image thumbnail generation
  - ETL (extract, transform, load) preprocessing
  - Virus scanning
  - Data lake ingestion

Example S3 notification pattern:

```json
{
  "LambdaFunctionConfigurations": [
    {
      "Id": "ProcessCsvUploads",
      "LambdaFunctionArn": "arn:aws:lambda:us-east-1:123456789012:function:process-csv",
      "Events": ["s3:ObjectCreated:*"],
      "Filter": {
        "Key": {
          "FilterRules": [
            { "Name": "prefix", "Value": "incoming/" },
            { "Name": "suffix", "Value": ".csv" }
          ]
        }
      }
    }
  ]
}
```

Practical CLI example:

```bash
aws s3api put-bucket-notification-configuration \
  --bucket my-data-bucket \
  --notification-configuration file://notification.json
```

**Amazon SNS for fan-out**

Amazon Simple Notification Service (SNS) is a pub/sub service used for fan-out to multiple subscribers. It is a strong fit when one event should reach many consumers.

- **Fan-out** means one publisher sends to one topic, and the topic delivers to many subscribers.
- **Subscribers** can include Lambda, SQS, HTTP/S endpoints, email, SMS, and mobile push.
- **Message filtering** can route only matching messages to specific subscribers.
- **Delivery semantics** are at-least-once, so consumers should be idempotent.

SNS is often used when:

- Multiple systems need the same event.
- You want loose coupling between producer and consumers.
- You want to combine SNS with SQS for buffering and independent scaling.

Example topology:

```text
Producer -> SNS topic -> Lambda subscriber
                     -> SQS subscriber -> worker fleet
                     -> HTTPS endpoint
```

CLI example:

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:orders-topic \
  --message '{"orderId":"123","status":"created"}'
```

**SNS to Lambda versus SNS to SQS**

Both are valid, but they solve different problems.

| Pattern | Best for | Notes |
|---|---|---|
| SNS -> Lambda | Direct processing | Fast, simple, serverless push delivery |
| SNS -> SQS -> Lambda/EC2/ECS | Buffering and backpressure | Each consumer can process independently |
| SNS -> HTTP/S | External webhook delivery | Requires endpoint availability and retries |

Use SNS-to-Lambda when you want immediate processing and can tolerate retry-based delivery. Use SNS-to-SQS when you need a durable queue per consumer or want to decouple consumer speed from publisher speed.

**Amazon SQS for decoupling**

Amazon Simple Queue Service (SQS) is a managed message queue that decouples producers from consumers. It is a common exam answer when you need buffering, retry control, or asynchronous processing.

- **Standard queues** provide high throughput and at-least-once delivery.
- **FIFO queues** provide ordering and exactly-once processing semantics within the queue’s deduplication model.
- **Long polling** reduces empty receives and cost by waiting for messages before returning.
- **Visibility timeout** hides a message while a consumer processes it.
- **Redrive policy** moves repeatedly failing messages to a dead-letter queue (DLQ).

SQS is a strong choice when:

- You need to absorb traffic spikes.
- Consumers scale independently from producers.
- You want retry isolation and failure buffering.
- You need to protect downstream systems from overload.

Example consumer flow:

```text
Producer -> SQS queue -> Lambda event source mapping -> downstream service
```

CLI example for sending a message:

```bash
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders-queue \
  --message-body '{"orderId":"123","action":"ship"}'
```

**SQS long polling and visibility timeout**

Long polling waits for messages instead of returning immediately when the queue is empty.

- Set `ReceiveMessageWaitTimeSeconds` to reduce empty receives.
- Use long polling to lower cost and improve efficiency.
- Tune the **visibility timeout** to exceed the expected processing time.
- If processing may take longer than the visibility timeout, extend it or increase the timeout.

Example queue attributes:

```bash
aws sqs set-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders-queue \
  --attributes ReceiveMessageWaitTimeSeconds=20,VisibilityTimeout=60
```

**SQS FIFO queues**

FIFO queues preserve order within a message group and deduplicate messages within the deduplication window.

- Use the same **message group ID** for messages that must remain ordered relative to each other.
- Different message groups can process in parallel.
- A single message group serializes processing for that group.
- Use **content-based deduplication** or explicit `MessageDeduplicationId` to avoid duplicates.

Example send request:

```bash
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders.fifo \
  --message-body '{"orderId":"123","step":"payment"}' \
  --message-group-id order-123 \
  --message-deduplication-id payment-123
```

**Dead-letter queues and redrive policies**

A DLQ is a queue that receives messages that cannot be processed successfully after repeated attempts.

- **SQS DLQ** is configured with a redrive policy and `maxReceiveCount`.
- A message is moved to the DLQ after it is received too many times without successful deletion.
- Use DLQs to isolate poison-pill messages and preserve queue health.
- For FIFO queues, the DLQ should also be FIFO if you need to preserve ordering characteristics.

Example redrive policy:

```json
{
  "deadLetterTargetArn": "arn:aws:sqs:us-east-1:123456789012:orders-dlq",
  "maxReceiveCount": 5
}
```

**Lambda asynchronous invocation**

Lambda async invocation is used when the caller should not wait for the function to finish. The event is queued by Lambda, then retried if the function fails.

- Lambda retries async events automatically.
- If retries are exhausted, you can send the event to a **failure destination** or configure a DLQ for the function.
- Async invocation is common with S3, SNS, EventBridge, and direct `Invoke` calls using `InvocationType=Event`.
- Use async when you want fire-and-forget behavior with managed retries.

Example:

```bash
aws lambda invoke \
  --function-name process-order \
  --invocation-type Event \
  --payload '{"orderId":"123"}' \
  response.json
```

Failure handling options:

- **On success/failure destinations** can send the result or failed event to SQS, SNS, Lambda, or EventBridge.
- **DLQ** captures failed async events for later inspection.
- Use destinations when you want richer context about the invocation outcome.

**EventBridge rules, buses, and Scheduler**

Amazon EventBridge is the event routing service for AWS service events, custom events, and SaaS partner events.

- **Event buses** receive events.
  - Default bus: AWS service events and custom events often land here.
  - Custom bus: your applications publish domain events here.
  - Partner bus: SaaS partners send events here.
- **Rules** match events and route them to targets.
  - Targets can include Lambda, SQS, SNS, Step Functions, Kinesis, ECS, API destinations, and more.
- **EventBridge Scheduler** creates time-based schedules.
  - Use it for one-time or recurring jobs without needing a rule tied to an event bus.

EventBridge is a strong choice when:

- You need content-based routing.
- You want to decouple producers and consumers.
- You need multiple targets from one event.
- You need schema-aware event-driven integration.

Example rule pattern:

```json
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["stopped"]
  }
}
```

CLI example for a rule:

```bash
aws events put-rule \
  --name ec2-stopped-rule \
  --event-pattern file://pattern.json
```

Example target attachment:

```bash
aws events put-targets \
  --rule ec2-stopped-rule \
  --targets "Id"="1","Arn"="arn:aws:lambda:us-east-1:123456789012:function:notify-team"
```

**EventBridge Scheduler example**

Scheduler uses the `scheduler` API, not `events put-rule`.

```bash
aws scheduler create-schedule \
  --name nightly-report \
  --schedule-expression "cron(0 2 * * ? *)" \
  --flexible-time-window Mode=OFF \
  --target "Arn=arn:aws:lambda:us-east-1:123456789012:function:generate-report,RoleArn=arn:aws:iam::123456789012:role/SchedulerInvokeRole"
```

Use Scheduler when you need:

- One-off delayed execution
- Recurring jobs
- Time-based orchestration without an event bus rule

**EventBridge versus SNS**

These services overlap, but the exam expects you to know the difference.

- **SNS** is primarily pub/sub fan-out.
- **EventBridge** is event routing with filtering on structured event fields.
- **EventBridge** is better for AWS service events, SaaS partner events, and custom application events with schema and routing needs.
- **SNS** is better for simple broadcast notifications and mobile/email/webhook style delivery.

**EventBridge Pipes**

EventBridge Pipes connects a supported source to a supported target with optional filtering, enrichment, and transformation. It reduces the need for custom glue code.

Supported sources commonly include:

- SQS
- Kinesis Data Streams
- DynamoDB Streams
- Amazon MSK
- Amazon MQ

Supported targets commonly include:

- Lambda
- SNS
- SQS
- EventBridge event buses
- Step Functions
- API destinations

Use Pipes when you want:

- Simple source-to-target integration
- Built-in filtering
- Optional enrichment without writing a consumer
- Less operational overhead than a custom poller

Example conceptual flow:

```text
SQS queue -> EventBridge Pipe -> Lambda
```

Example CLI skeleton:

```bash
aws pipes create-pipe \
  --name orders-pipe \
  --source arn:aws:sqs:us-east-1:123456789012:orders-queue \
  --target arn:aws:lambda:us-east-1:123456789012:function:process-order \
  --role-arn arn:aws:iam::123456789012:role/PipesInvokeRole
```

**API Gateway and ALB as triggers**

API Gateway and Application Load Balancer both commonly trigger Lambda, but they solve different problems.

- **API Gateway** is preferred for API management.
  - Authentication and authorization
  - Throttling and quotas
  - Usage plans and API keys
  - Request/response transformation
  - REST and HTTP APIs
- **ALB** is preferred for path-based or host-based routing to Lambda or containers.
  - Simple routing
  - Lower-latency HTTP entry point
  - Web applications and microservices behind one load balancer

Choose API Gateway when you need API governance. Choose ALB when you need flexible routing and do not need full API management features.

Example Lambda proxy event from API Gateway:

```json
{
  "httpMethod": "GET",
  "path": "/orders/123",
  "headers": {
    "host": "api.example.com"
  }
}
```

**Kinesis Data Streams**

Kinesis Data Streams is a streaming service for ordered records within a shard. Lambda can consume it through an event source mapping.

- Records are written to shards.
- Consumers read records in order within a shard.
- Lambda checkpoints progress as it processes batches.
- If a batch fails, the shard can be blocked until the failure is resolved or the record is retried successfully.
- Use Kinesis when you need near-real-time streaming, replay, or multiple consumers.

Operational points that matter on the exam:

- **Shard-level concurrency** limits parallel processing per shard.
- **Checkpointing** tracks the last successfully processed record.
- **Batch failure** can delay later records in the same shard.
- **Enhanced fan-out** gives dedicated throughput to consumers when needed.

Example stream ARN format:

```text
arn:aws:kinesis:us-east-1:123456789012:stream/orders-stream
```

Example Lambda event source mapping:

```bash
aws lambda create-event-source-mapping \
  --function-name process-stream \
  --event-source-arn arn:aws:kinesis:us-east-1:123456789012:stream/orders-stream \
  --starting-position LATEST
```

**DynamoDB Streams**

DynamoDB Streams captures item-level changes in a DynamoDB table. Lambda commonly consumes the stream for reactive workflows.

- Stream records include inserts, updates, and deletes.
- Ordering is preserved for changes to the same item.
- Use streams for denormalization, audit logging, cache invalidation, and event-driven workflows.
- Streams are not a general-purpose message bus; they are tied to table changes.

Exam-relevant behavior:

- Lambda processes stream records in batches.
- Failures can delay progress for records in the affected shard.
- Use idempotent consumers because retries can occur.
- Choose the stream view type based on how much item data you need:
  - `KEYS_ONLY`
  - `NEW_IMAGE`
  - `OLD_IMAGE`
  - `NEW_AND_OLD_IMAGES`

Example event handling logic:

```python
def handler(event, context):
    for record in event["Records"]:
        event_name = record["eventName"]
        keys = record["dynamodb"]["Keys"]
        if event_name == "MODIFY":
            new_image = record["dynamodb"].get("NewImage")
            # update downstream projection
```

**Amazon MQ and Amazon MSK**

Amazon MQ and Amazon Managed Streaming for Apache Kafka (MSK) are used when you need compatibility with existing messaging or streaming ecosystems.

- **Amazon MQ** supports brokers such as ActiveMQ and RabbitMQ.
  - Useful for lift-and-shift messaging workloads.
  - Lambda consumes via event source mapping and polls the broker.
- **Amazon MSK** is managed Kafka.
  - Useful for high-throughput streaming and Kafka-native applications.
  - Lambda consumes from Kafka topics via event source mapping and polls for records.

Exam distinctions:

- These are **poll-based** integrations with Lambda.
- They are not direct push triggers.
- They fit when you need existing protocol compatibility or Kafka semantics.
- Consumer lag, offsets, and batching matter more than simple webhook-style triggers.

**Event source mapping behavior**

Event source mappings connect poll-based sources to Lambda.

- Lambda polls the source on your behalf.
- Lambda batches records before invoking the function.
- You can tune batch size and batching window.
- Failures can cause retries and delayed progress.
- For queues and streams, design the function to be idempotent.

Common tuning considerations:

- Increase batch size for throughput.
- Reduce batch size for lower latency.
- Use partial batch response where supported to avoid reprocessing successful records.
- Align visibility timeout with Lambda timeout and expected processing time.

**Choosing the right source or trigger**

Use the following decision logic:

1. If you need a simple HTTP API, choose **API Gateway** or **ALB**.
2. If you need fan-out to many consumers, choose **SNS**.
3. If you need buffering and decoupling, choose **SQS**.
4. If you need ordered streaming or replay, choose **Kinesis Data Streams**.
5. If you need table-change events, choose **DynamoDB Streams**.
6. If you need AWS service event routing, choose **EventBridge**.
7. If you need time-based execution, choose **EventBridge Scheduler**.
8. If you need broker or Kafka compatibility, choose **Amazon MQ** or **Amazon MSK**.
9. If you need object-change triggers, choose **S3 event notifications**.

**Common exam scenarios**

- **Image upload processing**
  - Use S3 event notifications to Lambda.
  - Add prefix/suffix filters for specific file types.
- **Order processing with buffering**
  - Use API Gateway or SNS to SQS, then Lambda.
  - Add a DLQ for poison messages.
- **Audit trail from database changes**
  - Use DynamoDB Streams to Lambda.
  - Write changes to OpenSearch, S3, or another table.
- **Real-time analytics**
  - Use Kinesis Data Streams with Lambda or Kinesis consumers.
- **Cross-service orchestration**
  - Use EventBridge rules to route events to multiple targets.
- **Scheduled cleanup**
  - Use EventBridge Scheduler to invoke Lambda on a cron schedule.

**Failure handling patterns**

Different services fail differently, so the recovery pattern matters.

- **SQS**: message becomes visible again after the visibility timeout; DLQ after `maxReceiveCount`.
- **SNS**: retries delivery to subscribers; use subscription DLQs where supported.
- **Lambda async**: retries automatically; use destinations or a DLQ for failed events.
- **Kinesis/DynamoDB Streams**: failed batches can block progress until handled.
- **EventBridge**: retries target delivery; use a DLQ for failed target invocations when needed.
- **S3 notifications**: delivery is best-effort to the configured target; design for retries and idempotency.

**Practical design guidance**

- Use **idempotency** for every event consumer.
  - Duplicate delivery is normal in distributed systems.
- Use **DLQs** for failed messages or events you need to inspect later.
- Use **filters** early to reduce unnecessary invocations.
- Use **decoupling** when producer and consumer lifecycles differ.
- Use **ordering** only when the business process truly requires it, because ordering reduces parallelism.
- Use **least privilege IAM** so the source can invoke only the intended target.

**Quick comparison**

| Service | Trigger style | Best use | Key exam clue |
|---|---|---|---|
| S3 notifications | Push | Object-created/removed workflows | Prefix/suffix filters |
| SNS | Push/pub-sub | Fan-out notifications | Many subscribers |
| SQS | Poll/queue | Buffering and decoupling | Visibility timeout, DLQ |
| Lambda async | Async | Fire-and-forget compute | Retries and destinations |
| EventBridge | Push/routing | Event bus and filtering | Rules, buses, partner events |
| EventBridge Scheduler | Time-based | Cron or one-time schedules | Uses `scheduler` API |
| Kinesis Data Streams | Poll/stream | Ordered streaming | Shards and checkpointing |
| DynamoDB Streams | Poll/stream | Table-change reactions | Item-level change events |
| Amazon MQ | Poll/broker | Legacy messaging | Broker compatibility |
| Amazon MSK | Poll/Kafka | Kafka-native streaming | Topic offsets and consumer groups |
| API Gateway | Push/HTTP | API front door | Auth, throttling, usage plans |
| ALB | Push/HTTP | Routing to Lambda or containers | Path/host-based routing |

**Exam traps to avoid**

- Do not confuse **EventBridge Scheduler** with an EventBridge rule.
- Do not assume SNS is only for email or SMS; it commonly triggers Lambda and SQS.
- Do not use SQS when you need immediate fan-out to many independent consumers without a topic.
- Do not use Kinesis when you only need simple queue semantics.
- Do not use DynamoDB Streams as a general event bus.
- Do not forget that ordering usually reduces throughput.
- Do not ignore DLQs when the question asks how to handle repeated failures.

**Minimal architecture examples**

S3 to Lambda:

```text
S3 bucket -> Lambda -> DynamoDB
```

SNS fan-out with buffering:

```text
Producer -> SNS topic -> SQS queue A -> Lambda A
                      -> SQS queue B -> Lambda B
```

EventBridge routing:

```text
AWS service event -> EventBridge bus -> Lambda
                                   -> Step Functions
                                   -> SQS
```

Kinesis stream processing:

```text
Producers -> Kinesis Data Streams -> Lambda consumer -> S3/OpenSearch
```

**Key takeaways**

- Choose the trigger based on delivery model, not just the target service.
- Use push for direct invocation, poll for durable consumption, and async for fire-and-forget workflows.
- Use SQS for buffering, SNS for fan-out, EventBridge for routing, and Kinesis/DynamoDB Streams for ordered stream processing.
- Match failure handling to the service: retries, visibility timeout, checkpointing, or DLQs.
