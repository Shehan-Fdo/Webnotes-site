## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Event-driven architecture fundamentals**

Event-driven systems react to state changes instead of relying on tight request/response coupling. Producers emit events when something happens, and consumers process those events asynchronously. This improves scalability, resilience, and team independence because services do not need to know who consumes the event.

Common AWS building blocks include:

- **Amazon EventBridge** for routing application and SaaS events with content-based rules.
- **Amazon Simple Queue Service (SQS)** for durable buffering and work queues.
- **Amazon Simple Notification Service (SNS)** for fanout and push-based pub/sub delivery.
- **AWS Lambda** for serverless event processing.
- **AWS Step Functions** for orchestrating multi-step workflows.
- **DynamoDB Streams** for reacting to item-level changes in tables.

A good exam answer usually depends on the delivery semantics you need:

- Use **queues** when you want one consumer to process each message.
- Use **topics** when you want one event to reach many subscribers.
- Use **event buses** when you want content-based routing across many producers and consumers.
- Use **workflows** when the process has branching, retries, waits, or human/long-running steps.

**Why decoupling matters**

Decoupling reduces blast radius. If a downstream service is slow or unavailable, the producer can continue operating when you place a buffer between them. This is especially important for bursty workloads, cross-team integrations, and systems that must tolerate partial failure.

Decoupling also helps you evolve systems independently:

- Producers publish events without knowing consumer implementation details.
- Consumers can be added, removed, or replaced with minimal producer changes.
- You can scale each component separately based on demand.

**Amazon SQS as a decoupling buffer**

SQS is a managed message queue that stores messages until a consumer processes them. It is the standard choice when you need reliable asynchronous handoff between components.

Key SQS concepts:

- **Standard queues** provide high throughput and at-least-once delivery.
- **FIFO queues** preserve order within a message group and support exactly-once processing semantics through deduplication features.
- Messages are retained until consumed or until the retention period expires.
- Consumers poll the queue; SQS does not push messages to consumers.

Typical use cases:

- Buffering web requests during traffic spikes.
- Decoupling microservices.
- Offloading long-running work from synchronous APIs.
- Smoothing downstream capacity constraints.

Important operational details:

- **Visibility timeout** hides a message from other consumers after receipt.
- If processing takes longer than the visibility timeout, you must extend it with `ChangeMessageVisibility`.
- If the consumer fails to delete the message before the timeout expires, the message becomes visible again and may be processed twice.
- Configure a **dead-letter queue (DLQ)** for messages that repeatedly fail processing.

Example CLI workflow:

```bash
aws sqs create-queue --queue-name orders-work-queue

aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders-work-queue \
  --message-body '{"orderId":"A1001","action":"reserve-inventory"}'

aws sqs receive-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders-work-queue \
  --max-number-of-messages 10 \
  --wait-time-seconds 20
```

A long-running worker should extend visibility before the timeout expires:

```bash
aws sqs change-message-visibility \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/orders-work-queue \
  --receipt-handle AQEB123... \
  --visibility-timeout 300
```

Batching considerations:

- SQS supports sending and receiving messages in batches to reduce API calls.
- Lambda event source mappings can process multiple SQS messages per invocation.
- Larger batch sizes improve throughput but increase the amount of work lost or retried together if the function fails.
- For workloads with mixed processing times, smaller batches often reduce tail latency and retry amplification.

Partial batch failure handling is important for Lambda + SQS integrations:

- If one message in a batch fails, the entire batch may be retried unless you use partial batch response support.
- With partial batch response, you return only the failed message identifiers so successful messages are not reprocessed.
- This is especially useful when a single poison message should not block unrelated messages in the same batch.

Example Lambda handler pattern for partial batch response:

```python
def lambda_handler(event, context):
    failures = []
    for record in event["Records"]:
        try:
            process(record)
        except Exception:
            failures.append({"itemIdentifier": record["messageId"]})
    return {"batchItemFailures": failures}
```

Security and encryption basics:

- **SSE-SQS** encrypts messages at rest with an AWS-managed key.
- **SSE-KMS** uses AWS Key Management Service for customer-managed control and auditability.
- Queue policies control who can send, receive, and manage messages.
- Use IAM permissions plus resource-based queue policies when cross-account access is required.

Example queue policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSNSPublish",
      "Effect": "Allow",
      "Principal": { "Service": "sns.amazonaws.com" },
      "Action": "sqs:SendMessage",
      "Resource": "arn:aws:sqs:us-east-1:123456789012:orders-queue",
      "Condition": {
        "ArnEquals": {
          "aws:SourceArn": "arn:aws:sns:us-east-1:123456789012:orders-topic"
        }
      }
    }
  ]
}
```

**Amazon SNS for fanout**

SNS is a pub/sub service that pushes messages to multiple subscribers. It is the right choice when one event must reach many destinations quickly.

Subscription protocol differences matter on the exam:

- **Lambda** subscriptions invoke functions directly and are common for event fanout.
- **SQS** subscriptions deliver messages to queues for durable buffering and independent consumption.
- **HTTP/HTTPS** subscriptions push to web endpoints and require endpoint availability and proper retry handling.
- **Email** and other protocols are useful for notifications but are not typical for application integration.

Delivery behavior:

- SNS retries delivery for transient failures.
- For HTTP/HTTPS endpoints, SNS uses retry logic with backoff for failed deliveries.
- If a subscription endpoint remains unavailable, delivery can eventually fail depending on protocol and configuration.
- For SQS subscriptions, SNS delivers to the queue, and SQS then handles durable storage and consumer polling.

SNS fanout pattern example:

```text
Producer -> SNS topic -> [SQS queue for billing]
                         -> [SQS queue for shipping]
                         -> [Lambda for notifications]
```

This pattern is useful when multiple downstream systems need the same event but should process it independently.

**Amazon EventBridge for content-based routing**

EventBridge is the preferred service when you need event routing based on event content rather than simple broadcast. It supports application events, AWS service events, and SaaS partner events.

EventBridge rules match events using JSON patterns. Common exam-tested matching fields include:

- `source`
- `detail-type`
- `detail`
- nested fields inside `detail`

A rule can match on exact values, arrays of values, prefixes, numeric comparisons, and nested object fields.

Example event pattern:

```json
{
  "source": ["app.orders"],
  "detail-type": ["OrderCreated"],
  "detail": {
    "region": ["us-east-1"],
    "amount": [{ "numeric": [">=", 1000] }],
    "customer": {
      "tier": ["gold"]
    }
  }
}
```

This pattern matches events where:

- the event source is `app.orders`
- the event type is `OrderCreated`
- the order amount is at least 1000
- the customer tier is `gold`

Example rule creation:

```bash
aws events put-rule \
  --name high-value-orders \
  --event-pattern file://pattern.json \
  --state ENABLED
```

Example target attachment to an SQS queue:

```bash
aws events put-targets \
  --rule high-value-orders \
  --targets "Id"="1","Arn"="arn:aws:sqs:us-east-1:123456789012:orders-review-queue"
```

EventBridge delivery and failure handling:

- EventBridge retries target delivery on transient failures.
- If a target continues to fail, EventBridge can send the event to a **dead-letter queue** for later inspection.
- Use a DLQ when you need to preserve failed events instead of dropping them silently.
- For critical workflows, combine retries, DLQs, and idempotent consumers.

A practical EventBridge target configuration often includes:

- a rule with a narrow event pattern
- one or more targets
- an IAM role that allows EventBridge to invoke the target
- a DLQ for failed deliveries
- optional retry policy settings

Example target configuration concept:

```json
{
  "Id": "review-queue",
  "Arn": "arn:aws:sqs:us-east-1:123456789012:orders-review-queue",
  "DeadLetterConfig": {
    "Arn": "arn:aws:sqs:us-east-1:123456789012:eventbridge-dlq"
  },
  "RetryPolicy": {
    "MaximumRetryAttempts": 185,
    "MaximumEventAgeInSeconds": 86400
  }
}
```

Use EventBridge when you need:

- content-based filtering
- multiple targets with different routing logic
- integration with AWS services and SaaS providers
- a central event bus for loosely coupled systems

**EventBridge Pipes versus direct integrations**

EventBridge Pipes is useful when you want a simpler, point-to-point integration between a source and a target with optional filtering and enrichment. It reduces the need for custom glue code.

Use Pipes when:

- you want to connect a source such as SQS, Kinesis, or DynamoDB Streams to a target
- you need lightweight filtering or enrichment
- you do not need a full event bus with many rules and many producers

Use direct service integrations or EventBridge rules when:

- multiple consumers need the same event
- you need broad routing across many event sources
- you want centralized event governance and fanout

A practical rule of thumb:

- **Pipes** = streamlined transport with optional transform
- **EventBridge bus** = routing hub for event-driven architectures

**AWS Step Functions for workflow orchestration**

Step Functions coordinates multiple steps in a reliable state machine. It is the right choice when a process needs branching, retries, waits, parallel execution, or human-in-the-loop steps.

Core state types commonly tested:

- **Task** runs work, often by invoking Lambda or an AWS service integration.
- **Choice** branches based on input data.
- **Wait** pauses execution for a fixed time or until a timestamp.
- **Parallel** runs branches concurrently.
- **Map** iterates over a collection of items.
- **Succeed** ends successfully.
- **Fail** ends with an error.

Retries and catches are central to exam scenarios:

- Use `Retry` for transient errors such as throttling or temporary service issues.
- Use `Catch` to handle failures and route execution to fallback logic.
- Retries should be limited and targeted; do not retry permanent validation errors indefinitely.

Example state machine fragment:

```json
{
  "StartAt": "ValidateOrder",
  "States": {
    "ValidateOrder": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:validate-order",
      "Retry": [
        {
          "ErrorEquals": ["Lambda.ServiceException", "Lambda.TooManyRequestsException"],
          "IntervalSeconds": 2,
          "MaxAttempts": 3,
          "BackoffRate": 2.0
        }
      ],
      "Catch": [
        {
          "ErrorEquals": ["States.ALL"],
          "Next": "SendToDLQ"
        }
      ],
      "Next": "ChargePayment"
    },
    "ChargePayment": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:charge-payment",
      "Next": "Done"
    },
    "SendToDLQ": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:send-to-dlq",
      "End": true
    },
    "Done": {
      "Type": "Succeed"
    }
  }
}
```

Step Functions service integration patterns:

- **Request-response** waits for the service call to complete.
- **Run a job** starts a long-running job and waits for completion.
- **Callback** waits for an external worker to send a task token back.

These patterns are useful when you need orchestration without writing custom polling logic.

**DynamoDB Streams for change data capture**

DynamoDB Streams capture item-level changes in a table and are commonly used for event-driven processing and change data capture (CDC).

Important stream details:

- Stream records are ordered per partition key, not globally across the table.
- Streams have a retention window of about 24 hours.
- You choose a **stream view type** based on how much item data you need:
  - `KEYS_ONLY`
  - `NEW_IMAGE`
  - `OLD_IMAGE`
  - `NEW_AND_OLD_IMAGES`

Use cases:

- Triggering Lambda when a table item changes.
- Replicating data to another system.
- Building audit or analytics pipelines.
- Maintaining derived views or search indexes.

Lambda event source mapping behavior:

- Lambda polls the stream and processes records in batches.
- If processing fails, the batch may be retried.
- Batch size and parallelization affect throughput and retry behavior.
- For ordered processing requirements, keep concurrency and partitioning in mind.

Example stream-enabled table concept:

```json
{
  "StreamSpecification": {
    "StreamEnabled": true,
    "StreamViewType": "NEW_AND_OLD_IMAGES"
  }
}
```

**Choosing between SNS, SQS, EventBridge, and Step Functions**

Use the service that matches the integration problem, not the one with the most features.

| Service | Best for | Delivery model | Key exam clue |
|---|---|---|---|
| SNS | Fanout notifications | Push to subscribers | One event to many consumers |
| SQS | Buffering and work queues | Pull by consumers | Decouple producer and consumer speed |
| EventBridge | Content-based routing | Push to targets via rules | Match on event fields and route selectively |
| Step Functions | Workflow orchestration | Managed state machine | Branching, retries, waits, and long-running flows |

A common architecture combines them:

- Application emits an event to EventBridge.
- EventBridge routes high-value orders to an SQS queue.
- Lambda consumes the queue and performs processing.
- Step Functions orchestrates exception handling or multi-step fulfillment.
- SNS sends notifications to users or downstream teams.

**Practical architecture patterns**

**1. Fanout with independent consumers**

Use SNS when multiple systems need the same notification and each can process independently.

```text
Order Service -> SNS Topic -> SQS Billing Queue -> Billing Worker
                           -> SQS Shipping Queue -> Shipping Worker
                           -> Lambda Notification -> Email/SMS
```

This pattern avoids direct coupling between the producer and each consumer.

**2. Buffer and absorb spikes**

Use SQS between a web tier and a worker tier when traffic is bursty.

```text
API Gateway / ALB -> Producer -> SQS -> Auto-scaled Lambda or EC2 workers
```

This protects downstream systems from overload and lets you scale consumers separately.

**3. Content-based routing**

Use EventBridge when only some events should go to a target.

```text
App Events -> EventBridge
  - source = app.orders
  - detail-type = OrderCreated
  - detail.amount >= 1000 -> Review Queue
  - detail.customer.tier = gold -> Priority Workflow
```

This avoids custom filtering logic in application code.

**4. Orchestrated fulfillment**

Use Step Functions when the process has multiple dependent steps.

```text
Validate -> Reserve Inventory -> Charge Payment -> Ship -> Notify
```

Add retries for transient failures and catches for fallback paths.

**Operational and exam-relevant nuances**

- Prefer idempotent consumers because at-least-once delivery can produce duplicates.
- Use DLQs to isolate poison messages and failed deliveries.
- Tune batch size carefully for throughput versus retry blast radius.
- Extend SQS visibility timeout for long-running tasks.
- Use EventBridge patterns to filter by nested JSON fields, not just top-level fields.
- Choose SNS for fanout, SQS for buffering, EventBridge for routing, and Step Functions for orchestration.
- Use DynamoDB Streams when you need item-change events from a table, not a general-purpose event bus.

**Quick decision guide**

1. If you need one producer to feed many consumers, start with SNS or EventBridge.
2. If consumers must process independently and at their own pace, add SQS.
3. If the process has multiple steps, branching, or waits, use Step Functions.
4. If you need to react to DynamoDB item changes, use DynamoDB Streams with Lambda or Pipes.
5. If you need simple source-to-target transport with filtering, consider EventBridge Pipes.

**Common pitfalls**

- Assuming SNS stores messages for later consumption like SQS does.
- Forgetting that SQS consumers must delete messages after successful processing.
- Ignoring visibility timeout when processing can exceed the default.
- Using EventBridge when you really need a durable work queue.
- Using SQS when you need content-based routing to multiple targets.
- Forgetting that DynamoDB Streams are time-limited and ordered only within a partition key.
- Designing Lambda + SQS without handling partial batch failures.

**CLI examples for event-driven setup**

Create an SNS topic and subscribe an SQS queue:

```bash
aws sns create-topic --name order-events

aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:billing-queue
```

Create an EventBridge rule from a JSON pattern:

```json
{
  "source": ["app.checkout"],
  "detail-type": ["PaymentAuthorized"],
  "detail": {
    "currency": ["USD"],
    "amount": [{ "numeric": [">", 500] }]
  }
}
```

```bash
aws events put-rule \
  --name payment-review-rule \
  --event-pattern file://payment-pattern.json
```

Create a Step Functions state machine from a definition file:

```bash
aws stepfunctions create-state-machine \
  --name order-fulfillment \
  --definition file://state-machine.json \
  --role-arn arn:aws:iam::123456789012:role/StepFunctionsExecutionRole
```

**Summary**

Event-driven architecture on AWS is about choosing the right decoupling primitive for the job. SQS buffers work, SNS fans out notifications, EventBridge routes events by content, Step Functions orchestrates multi-step processes, and DynamoDB Streams react to table changes. For SAA-C03, focus on delivery semantics, retries, DLQs, batching, visibility timeout, and the differences between push, pull, routing, and orchestration.
