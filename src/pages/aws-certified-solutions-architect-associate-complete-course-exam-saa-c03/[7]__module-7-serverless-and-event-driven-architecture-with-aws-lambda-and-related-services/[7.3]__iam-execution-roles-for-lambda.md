## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lambda execution roles: the core idea**

An <abbr title="Identity and Access Management">IAM</abbr> execution role is the role that AWS Lambda assumes on behalf of your function at runtime. The function code does not “become” the role; instead, the Lambda service uses the role to call other AWS APIs such as Amazon S3, Amazon DynamoDB, Amazon KMS, or Amazon CloudWatch Logs.

For exam purposes, separate three concepts clearly:

- **Trust policy**: who can assume the role.
- **Permissions policy**: what the role can do after it is assumed.
- **Resource-based policy**: who can invoke the Lambda function or access the function as a resource.

The most important trust policy principal is `lambda.amazonaws.com`. That service principal tells AWS that the Lambda service, not your application code, is allowed to assume the role.

**Trust policy vs permissions policy**

The trust policy and permissions policy are separate documents with different jobs. This distinction is a common exam trap.

- **Trust policy**
  - Attached to the IAM role.
  - Defines the trusted principal.
  - For Lambda, the principal is typically `lambda.amazonaws.com`.
- **Permissions policy**
  - Attached to the same IAM role.
  - Defines allowed actions such as `logs:CreateLogGroup`, `s3:GetObject`, or `kms:Decrypt`.
- **Why it matters**
  - A role can trust Lambda but still have no useful permissions.
  - A role can have broad permissions but still be unusable if Lambda is not trusted to assume it.

A minimal trust policy for a Lambda execution role looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

**Default role behavior in the console**

When you create a function in the Lambda console, AWS can auto-create an execution role for you in some workflows. This is a common exam detail.

- If you choose to create a new function with basic permissions, Lambda can create a role with the basic logging policy attached.
- That basic role usually includes permissions for CloudWatch Logs so the function can write logs on first run.
- If you later add VPC access or access to other AWS services, you must extend the role with additional permissions.

This auto-created role is convenient for quick starts, but in production you usually manage the role explicitly so you can apply least privilege and review permissions carefully.

**Basic logging permissions and CloudWatch Logs**

Lambda writes function output to <abbr title="Amazon CloudWatch Logs">CloudWatch Logs</abbr>. The basic execution role is required for first-time creation of the log group and log stream.

Typical logging permissions include:

- `logs:CreateLogGroup`
- `logs:CreateLogStream`
- `logs:PutLogEvents`

A common managed policy for this is `AWSLambdaBasicExecutionRole`.

Example inline policy for logging:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    }
  ]
}
```

If the role lacks these permissions, the function may still run, but you will not get the logs you expect. On the exam, missing logs often points to an execution role problem rather than a code problem.

**VPC access for Lambda**

If your Lambda function needs to access resources inside a <abbr title="Virtual Private Cloud">VPC</abbr>, the execution role needs additional permissions so Lambda can manage network interfaces.

A common AWS managed policy for this is:

- `AWSLambdaVPCAccessExecutionRole`

This policy allows Lambda to create and manage the elastic network interfaces it needs to connect to private subnets.

Key points:

- The function code does not directly create network interfaces.
- The Lambda service uses the execution role to attach to the VPC.
- VPC access is separate from application-level permissions such as reading from S3 or decrypting with KMS.

**What Lambda needs the execution role for**

Use the execution role when the Lambda function itself needs to call AWS APIs.

Common examples:

- Read objects from S3.
- Write items to DynamoDB.
- Publish messages to SNS.
- Decrypt data with KMS.
- Write logs to CloudWatch Logs.
- Access resources in a VPC.

Example S3 read permissions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
      ]
    }
  ]
}
```

Example KMS decrypt permissions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt",
        "kms:Encrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ef-abcd-1234567890ab"
    }
  ]
}
```

**Full example: Lambda execution role policy document**

A realistic Lambda execution role often combines logging, S3 access, KMS access, and possibly DynamoDB access. This example shows a full permissions policy attached to the role.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "CloudWatchLogs",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    },
    {
      "Sid": "ReadInputFromS3",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-input-bucket",
        "arn:aws:s3:::example-input-bucket/*"
      ]
    },
    {
      "Sid": "WriteResultsToDynamoDB",
      "Effect": "Allow",
      "Action": [
        "dynamodb:PutItem",
        "dynamodb:UpdateItem",
        "dynamodb:GetItem"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/ExampleTable"
    },
    {
      "Sid": "DecryptSecrets",
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ef-abcd-1234567890ab"
    }
  ]
}
```

This is the kind of policy you design when the function needs to interact with multiple services. Keep the permissions narrow and scoped to the exact resources whenever possible.

**Resource-based policies vs execution-role permissions**

This distinction is especially important for Lambda integrations.

- **Execution-role permissions**
  - Used when Lambda code calls AWS services.
  - Example: your function reads from S3 or writes to SNS.
- **Resource-based policies**
  - Used when another AWS service or account needs permission to invoke the Lambda function.
  - Example: S3 event notifications invoke Lambda.
  - Example: SNS topic subscription invokes Lambda.
  - Example: another AWS account invokes your function cross-account.

In other words:

- If the function is the caller, use the execution role.
- If the function is the target of an invocation, use a resource-based policy on the Lambda function.

This is a frequent exam distinction.

**Invocation permissions from AWS services and other accounts**

Some AWS services invoke Lambda directly. In those cases, the Lambda function needs a resource-based policy that allows the service principal or external account to invoke it.

Common scenarios:

- **S3 event notifications**
  - S3 needs permission to invoke the function.
  - The function’s resource policy grants `lambda:InvokeFunction` to the S3 service principal with source restrictions.
- **SNS subscriptions**
  - SNS invokes Lambda using a resource-based policy on the function.
- **Cross-account invocation**
  - Another AWS account can invoke the function only if the function policy allows it.

Example Lambda resource-based policy for S3 invocation:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Invoke",
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": "lambda:InvokeFunction",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:ProcessUploads",
      "Condition": {
        "ArnLike": {
          "AWS:SourceArn": "arn:aws:s3:::example-input-bucket"
        }
      }
    }
  ]
}
```

Example cross-account invocation policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountInvoke",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::210987654321:root"
      },
      "Action": "lambda:InvokeFunction",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:ProcessUploads"
    }
  ]
}
```

**Common exam scenarios**

- A Lambda function cannot write logs.
  - Check the execution role for CloudWatch Logs permissions.
- A Lambda function in a private subnet cannot reach the network.
  - Check for VPC configuration and `AWSLambdaVPCAccessExecutionRole`.
- A Lambda function cannot read from S3.
  - Add S3 permissions to the execution role.
- S3 cannot trigger Lambda.
  - Add or fix the Lambda resource-based policy, not the execution role.
- Another account cannot invoke the function.
  - Add a resource-based policy on the Lambda function.

**CLI examples**

You can create and attach an execution role with the AWS CLI. In practice, you often create the role first, then attach policies, then reference the role when creating the function.

Create a trust policy file:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Create the role:

```bash
aws iam create-role \
  --role-name lambda-execution-role \
  --assume-role-policy-document file://trust-policy.json
```

Attach the basic logging policy:

```bash
aws iam attach-role-policy \
  --role-name lambda-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole
```

Attach VPC access if needed:

```bash
aws iam attach-role-policy \
  --role-name lambda-execution-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaVPCAccessExecutionRole
```

Create the function with the role:

```bash
aws lambda create-function \
  --function-name ProcessUploads \
  --runtime python3.12 \
  --handler app.handler \
  --zip-file fileb://function.zip \
  --role arn:aws:iam::123456789012:role/lambda-execution-role
```

**Troubleshooting checklist**

When Lambda fails, use a quick permission check:

1. Confirm the trust policy includes `lambda.amazonaws.com`.
2. Confirm the execution role has the needed service permissions.
3. Confirm CloudWatch Logs permissions exist for first-time logging.
4. Confirm VPC access permissions if the function is in a VPC.
5. Confirm the Lambda resource policy if another service or account invokes the function.
6. Confirm KMS permissions if encrypted data or environment variables are involved.

**Exam memory aids**

- **Lambda assumes the role**: the service does, not your code.
- **Trust policy is separate**: it answers “who can assume?”
- **Permissions policy is separate**: it answers “what can the role do?”
- **Execution role is for outbound calls**: Lambda code calling AWS APIs.
- **Resource policy is for inbound invokes**: other services or accounts calling Lambda.
- **Basic logging is essential**: no logs often means missing execution-role permissions.
- **VPC access has its own managed policy**: `AWSLambdaVPCAccessExecutionRole`.

**Practical design guidance**

Use the smallest set of permissions that still lets the function operate. Start with logging, then add only the service permissions the function truly needs. If the function is triggered by S3, SNS, or another account, remember that invocation permission belongs on the Lambda function resource policy, not on the execution role.

For production systems, keep the role narrowly scoped, review it alongside the function’s event sources, and test both runtime access and invocation permissions separately.
