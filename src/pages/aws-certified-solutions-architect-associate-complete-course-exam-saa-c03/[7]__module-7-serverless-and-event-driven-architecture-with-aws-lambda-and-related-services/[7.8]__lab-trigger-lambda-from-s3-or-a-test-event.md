## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and architecture**

This lab shows two common ways to invoke an AWS Lambda function: by uploading an object to Amazon S3 and by sending a manual test event from the Lambda console. The core pattern is event-driven serverless processing: S3 emits an event, Lambda receives it, and your function processes the payload.

A typical setup includes:

- An **S3 bucket** configured to send object-created notifications.
- A **Lambda function** with code that reads the event payload.
- An **IAM execution role** that allows Lambda to write logs to Amazon CloudWatch Logs.
- A **resource-based policy** on the Lambda function that allows S3 to invoke it.

This lab is useful because it reinforces how AWS services integrate without servers, polling, or custom schedulers.

**Event-driven invocation model**

Lambda does not “listen” to S3 directly. Instead, S3 publishes an event notification when a supported action occurs, such as `s3:ObjectCreated:*`. AWS then invokes the function on your behalf.

Key points:

- **Asynchronous invocation** is the normal pattern for S3 notifications.
- S3 event payloads include bucket name, object key, size, and event time.
- The function should be idempotent because retries can happen.
- You usually test the function first with a manual event, then validate the S3 trigger.

A simple mental model is:

```text
User uploads file -> S3 event notification -> Lambda invocation -> CloudWatch Logs
```

**Prerequisites and permissions**

Before you configure the trigger, make sure the function can run and log output.

Minimum requirements:

- A Lambda execution role with permissions for CloudWatch Logs.
- An S3 bucket in the same Region as the Lambda function for the simplest lab setup.
- A function code path that can parse the S3 event structure.
- Permission for S3 to invoke the Lambda function.

A common execution role policy for logging is:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "LambdaBasicLogging",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    }
  ]
}
```

If you create the function in the console, AWS can attach the basic execution role for you. If you use the CLI, you must create or attach the role yourself.

**Lambda test event workflow**

A test event is the fastest way to verify that your code works before wiring in S3. You use a sample event that matches the expected payload shape.

Typical workflow:

1. Open the Lambda function in the console.
2. Choose **Test**.
3. Create a new test event using an S3 event template or a custom JSON payload.
4. Run the test and inspect the response and logs.

Example test event payload:

```json
{
  "Records": [
    {
      "eventVersion": "2.1",
      "eventSource": "aws:s3",
      "awsRegion": "us-east-1",
      "eventTime": "2026-04-26T12:00:00.000Z",
      "eventName": "ObjectCreated:Put",
      "s3": {
        "bucket": {
          "name": "my-lab-bucket"
        },
        "object": {
          "key": "uploads/example.txt",
          "size": 128
        }
      }
    }
  ]
}
```

A simple Python handler for this event looks like:

```python
import json

def lambda_handler(event, context):
    print("Received event:", json.dumps(event))
    
    record = event["Records"][0]
    bucket = record["s3"]["bucket"]["name"]
    key = record["s3"]["object"]["key"]
    
    return {
        "statusCode": 200,
        "body": f"Processed {key} from {bucket}"
    }
```

Use test events when you want to:

- Validate parsing logic.
- Confirm logging and return values.
- Debug without waiting for an S3 upload.
- Reproduce failures with a controlled payload.

**Configuring S3 as an event source**

To trigger Lambda from S3, you configure bucket event notifications for object creation. The most common event type in labs is `s3:ObjectCreated:*`, which covers `Put`, `Post`, `Copy`, and multipart upload completion.

Important configuration details:

- The bucket and function should be in the same Region for straightforward setup.
- S3 event notifications can target Lambda, SNS, or SQS.
- You can filter by prefix and suffix to limit which objects trigger the function.
- S3 must have permission to invoke the function.

Example notification filter logic:

- Prefix: `uploads/`
- Suffix: `.txt`

This means only objects like `uploads/report.txt` trigger the function.

**Granting S3 permission to invoke Lambda**

When you add an S3 trigger in the console, AWS usually adds the required resource-based permission automatically. If you configure it manually, you need a statement that allows the S3 service principal to invoke the function.

Example CLI command:

```bash
aws lambda add-permission \
  --function-name my-s3-handler \
  --statement-id s3-invoke-permission \
  --action lambda:InvokeFunction \
  --principal s3.amazonaws.com \
  --source-arn arn:aws:s3:::my-lab-bucket \
  --source-account 123456789012
```

This permission is important because Lambda uses resource-based policies for service invocations. Without it, S3 cannot call the function even if the bucket notification exists.

**Example S3 notification configuration**

If you configure notifications with the AWS CLI or infrastructure as code, the structure resembles this:

```json
{
  "LambdaFunctionConfigurations": [
    {
      "Id": "TriggerOnTxtUploads",
      "LambdaFunctionArn": "arn:aws:lambda:us-east-1:123456789012:function:my-s3-handler",
      "Events": ["s3:ObjectCreated:*"],
      "Filter": {
        "Key": {
          "FilterRules": [
            {
              "Name": "prefix",
              "Value": "uploads/"
            },
            {
              "Name": "suffix",
              "Value": ".txt"
            }
          ]
        }
      }
    }
  ]
}
```

You can apply a notification configuration with:

```bash
aws s3api put-bucket-notification-configuration \
  --bucket my-lab-bucket \
  --notification-configuration file://notification.json
```

**Testing the S3 trigger**

After the notification is in place, upload a matching object to the bucket.

Example:

```bash
aws s3 cp ./example.txt s3://my-lab-bucket/uploads/example.txt
```

Then verify:

- The Lambda function was invoked.
- CloudWatch Logs contain the event payload.
- The function returned success.
- The object key in the event matches the uploaded file.

If the function does not run, check:

- The bucket notification configuration.
- The Lambda resource policy.
- The Region of the bucket and function.
- The object key filter rules.
- CloudWatch Logs for runtime errors.

**CloudWatch Logs and troubleshooting**

CloudWatch Logs is the primary place to confirm what happened during invocation. Lambda automatically sends `stdout` and `stderr` output to the log stream if the execution role has the correct permissions.

Useful debugging pattern:

```python
import json

def lambda_handler(event, context):
    print(json.dumps(event, indent=2))
    return {"ok": True}
```

Common issues include:

- **Missing permissions**: Lambda cannot write logs or S3 cannot invoke the function.
- **Wrong event shape**: Your code expects a test event but receives an S3 event, or vice versa.
- **Region mismatch**: The trigger is not configured where you expect.
- **Filter mismatch**: Prefix or suffix rules prevent invocation.
- **Handler errors**: The function throws an exception before returning.

A practical troubleshooting sequence is:

1. Confirm the function works with a manual test event.
2. Confirm the S3 notification exists on the bucket.
3. Confirm Lambda permission for `s3.amazonaws.com`.
4. Upload a file that matches the filter.
5. Inspect CloudWatch Logs for the invocation details.

**Real-world considerations**

In production, you often add more than simple logging. For example, the function may resize images, validate uploads, extract metadata, or move files to another prefix after processing.

Design considerations:

- Keep the function small and focused on one event type.
- Make processing idempotent so duplicate events do not cause bad side effects.
- Use object key prefixes to separate input and output paths.
- Consider dead-letter handling or retries for downstream failures.
- Use least privilege for both the Lambda execution role and the S3 trigger permission.

A common pattern is to upload files to `uploads/`, process them in Lambda, and write results to `processed/` or another bucket.

**Exam relevance**

For SAA-C03, you should understand:

- How S3 event notifications invoke Lambda.
- The difference between a manual Lambda test event and a real S3 event.
- Why Lambda needs an execution role and why S3 needs invoke permission.
- How prefix and suffix filters control event delivery.
- How CloudWatch Logs help you verify and troubleshoot the integration.

If you see a question about automatically processing uploaded objects, the likely answer is an S3 event notification triggering Lambda, not a scheduled job or a client-side poller.

**Quick reference**

| Topic | Key idea |
|---|---|
| Lambda test event | Manually invokes the function with sample JSON |
| S3 trigger | Invokes Lambda when an object event occurs |
| Execution role | Lets Lambda write logs and access AWS resources |
| Resource policy | Lets S3 invoke the Lambda function |
| Prefix/suffix filter | Limits which objects trigger the function |

**Minimal end-to-end example**

1. Create a Lambda function with a basic execution role.
2. Add code that prints the incoming event.
3. Add an S3 trigger for `s3:ObjectCreated:*`.
4. Optionally filter on `uploads/` and `.txt`.
5. Upload a matching object to the bucket.
6. Check CloudWatch Logs to confirm the invocation.

Example handler in Node.js:

```javascript
exports.handler = async (event) => {
  console.log(JSON.stringify(event, null, 2));

  const record = event.Records[0];
  const bucket = record.s3.bucket.name;
  const key = record.s3.object.key;

  return {
    statusCode: 200,
    body: `Processed ${key} from ${bucket}`
  };
};
```

This lab reinforces the AWS serverless event model: services emit events, Lambda processes them, and permissions control who can invoke what.
