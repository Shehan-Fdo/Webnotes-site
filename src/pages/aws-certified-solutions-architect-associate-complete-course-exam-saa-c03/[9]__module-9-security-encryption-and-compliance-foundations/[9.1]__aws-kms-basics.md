## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**AWS KMS overview**

AWS Key Management Service (<abbr title="Key Management Service">KMS</abbr>) is a managed service for creating and controlling cryptographic keys used to protect data at rest and to support signing and verification workflows. You use KMS when you want centralized key control, auditability through AWS CloudTrail, and integration with many AWS services.

KMS is not a general-purpose secrets store. It manages keys, not plaintext secrets. In practice, you often use KMS with services such as Amazon S3, Amazon EBS, Amazon RDS, Amazon EFS, AWS Lambda, and Amazon DynamoDB for server-side encryption.

- **Customer managed keys** give you the most control over policy, rotation, and lifecycle.
- **AWS managed keys** are created and managed by AWS for a specific service and account.
- **AWS owned keys** are fully managed by AWS and are not visible in your account.
- **Symmetric keys** are the default choice for encrypting and decrypting data.
- **Asymmetric keys** support public/private key operations such as encrypt/decrypt or sign/verify, depending on key spec and usage.

**Core concepts**

A KMS key is a logical resource that is regional. Standard KMS keys exist in one AWS Region, and multi-Region keys are still regional resources with replicas in other Regions. You do not get one global key that automatically spans all Regions.

KMS uses envelope encryption for most data protection workflows. The service encrypts a data key under a KMS key, and your application or AWS service uses the data key to encrypt the actual data. This design reduces latency and limits direct KMS calls for large payloads.

- **KMS key**: The top-level cryptographic resource in KMS.
- **Data key**: A plaintext key used locally to encrypt data, plus an encrypted copy stored alongside the ciphertext.
- **Envelope encryption**: Encrypt data with a data key, then protect that data key with KMS.
- **Key material**: The cryptographic material behind a KMS key.
- **Alias**: A friendly name that points to a KMS key in the same account and Region.

> KMS aliases are convenient for humans and automation, but they are not universal identifiers. For cross-account or cross-Region references, use the key ARN or key ID that matches the target resource.

**Key types and when to use them**

KMS supports several key categories. The right choice depends on how much control you need and whether an AWS service can use the key directly.

| Type | What you control | Typical use |
|---|---|---|
| Customer managed key | Policy, grants, rotation, deletion, aliases | Fine-grained application and compliance control |
| AWS managed key | Service-managed policy and lifecycle | Default encryption for an AWS service |
| AWS owned key | No direct control | Transparent encryption where you do not need key visibility |

Customer managed keys are the most common exam focus because they let you reason about permissions, rotation, and cross-account access. AWS managed keys are service-specific and usually appear as the default encryption option for a service. AWS owned keys are invisible to you and are not suitable when you need audit or policy control.

**Symmetric and asymmetric keys**

Symmetric keys use the same key material for encryption and decryption. They are the standard choice for most AWS service integrations and application data encryption.

Asymmetric keys use a public/private key pair. You can distribute the public key and keep the private key protected in KMS. These keys are useful for digital signatures, verification, and some specialized encryption workflows. For code-signing workflows, AWS Code Signing is the typical AWS-native service; KMS asymmetric keys can support signing and verification, but they are not the same thing as the Code Signing service.

- **Symmetric key**: Best for general encryption at rest and envelope encryption.
- **Asymmetric key**: Best for signing, verification, and niche cryptographic use cases.
- **HMAC key**: Used for message authentication code operations where supported.

**Encryption context**

Encryption context is a set of nonsecret key-value pairs that KMS binds to the encrypted data. It is not secret, but it becomes part of the cryptographic context. If you encrypt with an encryption context, you must provide the same context to decrypt.

This is useful when you want additional integrity checks or want to bind ciphertext to a specific application, tenant, or resource identifier.

```bash
aws kms encrypt \
  --key-id alias/app-prod \
  --plaintext fileb://message.txt \
  --encryption-context App=Payments,Env=Prod \
  --output text \
  --query CiphertextBlob
```

```bash
aws kms decrypt \
  --ciphertext-blob fileb://ciphertext.bin \
  --encryption-context App=Payments,Env=Prod \
  --output text \
  --query Plaintext
```

- **Use encryption context** when you need application-level binding.
- **Do not treat it as a secret**; it is visible in logs and metadata.
- **Match the context exactly** during decrypt if one was used during encrypt.

**Access control model**

KMS authorization is based on a combination of IAM (Identity and Access Management) policies, key policies, grants, and service integrations. The exact evaluation path depends on the account relationship and the service involved.

For same-account access, a key policy can delegate permissions to IAM by allowing the account root principal and then letting IAM policies decide which principals can use the key. For cross-account access, the key policy must explicitly allow the external principal or account, and the caller also needs IAM permission in its own account.

- **Key policy** is the primary resource policy for a KMS key.
- **IAM policy** can grant permissions when the key policy allows delegation.
- **Grant** provides a lightweight, often temporary permission path.
- **Service integration** may use grants or service-linked permissions behind the scenes.

> A caller does not automatically get access just because an IAM policy allows `kms:Decrypt`. The key policy must also permit the use, either directly or through delegation.

**Common permissions**

KMS permissions are action-based. The most common actions include encrypting, decrypting, generating data keys, describing keys, and managing key lifecycle operations.

- `kms:Encrypt`
- `kms:Decrypt`
- `kms:GenerateDataKey`
- `kms:GenerateDataKeyWithoutPlaintext`
- `kms:DescribeKey`
- `kms:CreateKey`
- `kms:EnableKeyRotation`
- `kms:ScheduleKeyDeletion`
- `kms:CreateGrant`

A practical pattern is to allow applications only the minimum set they need. For example, an application that encrypts data usually needs `kms:Encrypt` and `kms:GenerateDataKey`, but not `kms:Decrypt` unless it must read the data back.

**Practical CLI examples**

You can create a customer managed symmetric key with the AWS CLI. `--origin AWS_KMS` is the default, so you usually omit it unless you want to be explicit.

```bash
aws kms create-key \
  --description "App production key" \
  --key-usage ENCRYPT_DECRYPT \
  --key-spec SYMMETRIC_DEFAULT
```

You can create an alias to make the key easier to reference in scripts.

```bash
aws kms create-alias \
  --alias-name alias/app-prod \
  --target-key-id 1234abcd-12ab-34cd-56ef-1234567890ab
```

You can inspect a key with `describe-key`. Use a key ID, key ARN, or a same-account same-Region alias that exists in the account.

```bash
aws kms describe-key --key-id 1234abcd-12ab-34cd-56ef-1234567890ab
```

To encrypt a file, use `encrypt` with `fileb://` for binary-safe input.

```bash
aws kms encrypt \
  --key-id arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab \
  --plaintext fileb://secret.txt \
  --output text \
  --query CiphertextBlob > ciphertext.b64
```

To decrypt, pass the ciphertext blob back to KMS.

```bash
aws kms decrypt \
  --ciphertext-blob fileb://ciphertext.bin
```

To generate a data key for envelope encryption:

```bash
aws kms generate-data-key \
  --key-id alias/app-prod \
  --key-spec AES_256
```

**Envelope encryption workflow**

Envelope encryption is the standard pattern for protecting larger data sets efficiently. KMS protects a data key, and your application uses that data key locally.

1. You call `GenerateDataKey` or `GenerateDataKeyWithoutPlaintext`.
2. KMS returns a plaintext data key and an encrypted copy of that data key.
3. You use the plaintext data key to encrypt your data locally.
4. You store the encrypted data key with the ciphertext.
5. When decrypting, you send the encrypted data key to KMS to recover the plaintext data key.

This pattern reduces the number of KMS API calls and keeps bulk encryption outside the service.

**Rotation and lifecycle**

Key rotation helps reduce the risk of long-term key exposure. For customer managed symmetric keys, automatic rotation is available and commonly enabled for compliance. Rotation creates new cryptographic material for the same logical key, so existing ciphertext remains decryptable.

KMS keys also have lifecycle states such as enabled, disabled, pending deletion, and unavailable. Disabling a key immediately prevents new cryptographic operations, which can break workloads that depend on it.

- **Enable rotation** for long-lived customer managed keys.
- **Disable a key** to stop use without deleting it.
- **Schedule deletion** only when you are sure no workloads need the key.
- **Retain old ciphertext** because it still depends on the original key history.

**Multi-Region keys**

Multi-Region keys help with disaster recovery and regional failover. Each replica is still a regional KMS key, but replicas share the same key material and key ID structure for easier replication and failover planning.

Use multi-Region keys when you need the ability to decrypt data in another Region after a failover event. They are not a substitute for application design; you still need to replicate data and coordinate service configuration.

- **Primary key** exists in one Region.
- **Replica key** exists in another Region.
- **Same logical relationship** helps with cross-Region recovery.
- **Separate regional resources** still require regional permissions and configuration.

**Grants**

Grants are a fast way to delegate limited KMS permissions, often to AWS services or temporary workloads. They are useful when you do not want to modify the key policy every time a service needs access.

A grant can allow a principal to use a key for a specific purpose, such as decrypting data for an attached EBS volume or using a key on behalf of an AWS service.

- **Use grants** for temporary or service-driven access.
- **Use key policies** for durable authorization structure.
- **Use IAM** to scope who can create or manage grants.

**AWS service integrations**

Many AWS services integrate with KMS for server-side encryption. The service typically handles the KMS calls on your behalf, while you control access through IAM and key policy.

Common examples include:

- **Amazon S3** with SSE-KMS for object encryption.
- **Amazon EBS** for encrypted volumes and snapshots.
- **Amazon RDS** for encrypted databases and snapshots.
- **Amazon EFS** for file system encryption.
- **AWS Lambda** for environment variable encryption.
- **Amazon DynamoDB** for table encryption.
- **Amazon SNS** and **Amazon SQS** for encrypted messaging.

When a service integrates with KMS, the service may create or use an AWS managed key, or you may choose a customer managed key for tighter control.

**Operational and security best practices**

Use KMS as part of a broader security design, not as a standalone control. Pair it with least privilege, logging, and data classification.

- **Prefer customer managed keys** when you need auditability or custom access control.
- **Use aliases** for application configuration, but keep the underlying key ID in mind for troubleshooting.
- **Log and monitor** KMS API activity with CloudTrail.
- **Restrict key administration** separately from key usage.
- **Test key disablement and recovery** in nonproduction environments.
- **Use encryption context** to bind ciphertext to application metadata where appropriate.

**Exam focus**

For exam questions, focus on the relationship between key policy, IAM, and grants. Many scenarios are about choosing the right key type or understanding why access fails even when IAM looks correct.

- **KMS keys are regional**; multi-Region keys are replicated regional resources.
- **Aliases are not cross-account identifiers** and are scoped to account and Region.
- **Encryption context must match** when decrypting ciphertext created with it.
- **AWS managed keys** are service-specific and simpler, but less flexible than customer managed keys.
- **Envelope encryption** is the standard pattern for large data encryption.
- **Asymmetric KMS keys** can support signing and verification, but AWS Code Signing is the dedicated service for code-signing workflows.
