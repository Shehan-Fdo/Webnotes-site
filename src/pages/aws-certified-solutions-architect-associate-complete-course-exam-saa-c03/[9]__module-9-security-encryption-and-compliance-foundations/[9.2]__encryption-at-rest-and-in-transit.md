## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Encryption fundamentals**

Encryption protects data in two main states:

- **At rest**: data stored on disk, in object storage, in databases, in snapshots, or in backups.
- **In transit**: data moving between clients, services, Regions, VPCs, or on-premises networks.

For exam scenarios, first identify the data path, then decide whether you need storage encryption, transport encryption, or both. Many AWS services do both by default, but the mechanism and configuration differ by service.

A useful mental model is:

- **At rest**: storage layer protection, usually with AWS Key Management Service (AWS KMS), service-managed keys, or service-specific encryption.
- **In transit**: transport-layer protection, usually Transport Layer Security (TLS), HTTPS, or VPN/IPsec.

**AWS Key Management Service basics**

AWS KMS is the central managed key service for many AWS encryption workflows. It stores and manages cryptographic keys used by AWS services and applications.

Key types you should know:

- **AWS managed keys**: created and managed by AWS for a service; you do not control the policy in the same way as customer managed keys.
- **Customer managed keys**: created by you; you control key policy, grants, rotation, and lifecycle.
- **AWS owned keys**: fully managed by AWS and not visible to you.
- **CloudHSM-backed keys**: keys stored in a dedicated HSM cluster that you control; CloudHSM is a separate service and is not the same as KMS.

KMS uses a **key hierarchy** and commonly performs **envelope encryption**:

- KMS generates or protects a **data key**.
- Your application or AWS service uses the data key to encrypt the actual data.
- The data key is then encrypted by a KMS key and stored alongside the ciphertext.
- This reduces the need to call KMS for every byte of data and improves performance.

Example envelope encryption flow:

```text
1. Request data key from KMS
2. KMS returns:
   - plaintext data key
   - encrypted data key
3. Encrypt data locally with plaintext data key
4. Discard plaintext data key from memory as soon as possible
5. Store encrypted data + encrypted data key
6. Later, decrypt encrypted data key through KMS
```

Practical CLI example:

```bash
aws kms create-key --description "App data key"
aws kms enable-key-rotation --key-id alias/app-data
aws kms encrypt \
  --key-id alias/app-data \
  --plaintext fileb://secret.txt \
  --output text \
  --query CiphertextBlob
```

**KMS rotation, grants, and access control**

Customer managed KMS keys support **automatic rotation** for symmetric keys. Rotation changes the backing key material while preserving the logical key identity, which helps with compliance and operational hygiene.

Important points:

- Rotation does not re-encrypt all existing data immediately.
- Existing ciphertext remains decryptable because KMS retains older key material needed for decryption.
- You still need to design for key lifecycle, access control, and auditability.

KMS access is controlled by a combination of:

- **Key policy**: the primary authorization document for the key.
- **IAM policy**: identity-based permissions for the caller.
- **Grants**: temporary, scoped permissions often used by AWS services.

Use **grants** when a service needs short-lived permission to use a key on your behalf. Many AWS services create grants automatically so they can encrypt and decrypt data without you manually updating policies for every operation.

Example grant-related command:

```bash
aws kms create-grant \
  --key-id alias/app-data \
  --grantee-principal arn:aws:iam::111122223333:role/AppRole \
  --operations Encrypt Decrypt GenerateDataKey
```

For exam questions, remember:

- A key policy can allow another account or role to use the key, but the overall access pattern must still be valid for the service and the caller.
- Cross-account encryption scenarios often require both the key policy and the caller’s IAM permissions to align.
- Some AWS services use grants behind the scenes rather than direct long-term permissions.

**Services that encrypt at rest**

Many AWS services encrypt data at rest by default or offer encryption as a standard option. Know the common patterns:

- **Amazon S3**: supports server-side encryption with S3 managed keys (SSE-S3), KMS keys (SSE-KMS), or customer-provided keys in some scenarios.
- **Amazon EBS**: volumes and snapshots can be encrypted; encryption is often enabled by default in modern accounts and Regions.
- **Amazon RDS**: supports encryption for database instances and snapshots.
- **Amazon Aurora**: supports encryption for clusters and snapshots.
- **Amazon DynamoDB**: supports encryption at rest with AWS owned or AWS managed keys by default, and customer managed keys when configured.
- **AWS Lambda**: environment variables are encrypted at rest; you can use a customer managed KMS key for additional control.
- **AWS Secrets Manager**: encrypts secrets at rest with KMS.
- **AWS Systems Manager Parameter Store**: SecureString parameters are encrypted with KMS.
- **Amazon EFS**: supports encryption at rest with KMS.
- **Amazon Redshift**: supports encryption at rest.
- **Amazon OpenSearch Service**: supports encryption at rest.
- **Amazon SQS and SNS**: support server-side encryption with KMS.

When a question asks whether a service encrypts by default, be careful:

- Some services encrypt by default for new resources.
- Some require you to enable encryption at creation time.
- Some cannot be encrypted after creation without migration.

**S3 encryption and HTTPS enforcement**

For S3, you often need both storage encryption and transport encryption.

Common controls:

- **SSE-S3**: S3 manages the keys.
- **SSE-KMS**: KMS manages the keys and gives you auditability and tighter control.
- **Bucket policy**: enforce HTTPS and optionally require a specific encryption method.

To enforce HTTPS-only access, deny requests that are not using secure transport:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyInsecureTransport",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    }
  ]
}
```

To require SSE-KMS on uploads, you usually also need to require the encryption header and, if needed, the specific KMS key:

```json
{
  "Sid": "DenyUnencryptedObjectUploads",
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::my-bucket/*",
  "Condition": {
    "StringNotEquals": {
      "s3:x-amz-server-side-encryption": "aws:kms"
    }
  }
}
```

If you need to force a specific key, add a condition on `s3:x-amz-server-side-encryption-aws-kms-key-id`. This matters because requiring only `aws:kms` does not guarantee the intended key is used.

Practical upload example:

```bash
aws s3 cp file.txt s3://my-bucket/ \
  --sse aws:kms \
  --sse-kms-key-id alias/my-key
```

**RDS and Aurora encryption**

Amazon RDS and Amazon Aurora support encryption at rest, but the migration path depends on the engine and resource type.

Key exam points:

- You cannot simply toggle encryption on an existing unencrypted DB instance or Aurora cluster in place.
- For many RDS engines, the common approach is to create an encrypted copy from a snapshot, then restore a new encrypted database.
- For Aurora, encryption is applied at the cluster level, and migration workflows differ from standard RDS instances.
- Cross-Region and cross-account snapshot copy workflows often require careful KMS handling.

Typical RDS workflow:

1. Take a snapshot of the unencrypted DB.
2. Copy the snapshot and encrypt it with a KMS key.
3. Restore a new encrypted DB instance from the encrypted snapshot.
4. Cut over applications to the new endpoint.

Example snapshot copy:

```bash
aws rds copy-db-snapshot \
  --source-db-snapshot-identifier arn:aws:rds:us-east-1:111122223333:snapshot:my-snapshot \
  --target-db-snapshot-identifier my-encrypted-snapshot \
  --kms-key-id alias/rds-key
```

Important nuances:

- For cross-Region copy, the destination Region needs a usable KMS key.
- For cross-account copy, you may need to share the snapshot and allow the destination account to use or re-encrypt with its own key.
- Aurora read replica and global database workflows have engine/version-specific constraints, so do not assume every migration uses the same snapshot pattern.

**EBS, EFS, and storage-layer encryption**

Amazon EBS encryption is transparent to the instance. When you encrypt an EBS volume:

- Data at rest is encrypted.
- Snapshots are encrypted.
- Data moving between the volume and the instance is encrypted within the AWS infrastructure.

Useful commands:

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --size 100 \
  --encrypted \
  --kms-key-id alias/ebs-key
```

Amazon EFS supports encryption at rest and in transit, but in-transit encryption requires the **EFS mount helper** or the **Amazon EFS CSI driver** with TLS. Plain NFS mounts do not provide TLS.

Example mount:

```bash
sudo mount -t efs -o tls fs-12345678:/ /mnt/efs
```

For exam questions, distinguish:

- **At rest**: EFS encryption with KMS.
- **In transit**: TLS via mount helper or CSI driver.

**DynamoDB, Lambda, Secrets Manager, and Parameter Store**

These services often appear in architecture questions because they simplify encryption management.

- **DynamoDB**: encrypts tables at rest; use customer managed keys when you need tighter control or audit requirements.
- **Lambda**: environment variables are encrypted at rest; use a customer managed KMS key if you need explicit control over the encryption key.
- **Secrets Manager**: stores secrets encrypted with KMS and is designed for secret rotation and retrieval.
- **SSM Parameter Store**: use `SecureString` for encrypted parameters; choose KMS keys based on access and compliance needs.

Example SecureString parameter:

```bash
aws ssm put-parameter \
  --name "/app/db/password" \
  --type "SecureString" \
  --value "SuperSecret123!" \
  --key-id alias/ssm-key
```

**Encryption in transit with TLS**

TLS (Transport Layer Security) protects data while it moves over networks. In AWS, many managed services terminate TLS for you, but you still need to choose the right endpoint type, certificate location, and protocol settings.

Common patterns:

- **HTTPS** for web traffic.
- **TLS listeners** on load balancers.
- **VPN/IPsec** for encrypted network tunnels.
- **PrivateLink / VPC endpoints** to keep traffic on the AWS network path while still using service-specific security controls.

TLS version and cipher suite considerations:

- AWS services generally manage TLS negotiation for you.
- In exam scenarios, if a minimum TLS version is required, look for service settings that let you enforce it.
- For example, CloudFront, ALB, and API Gateway custom domains can be configured to require modern TLS policies.

**ACM and certificate management**

AWS Certificate Manager (ACM) issues, imports, renews, and associates public and private certificates for supported AWS services.

Use ACM when you need certificates for:

- **Application Load Balancer (ALB)**
- **Network Load Balancer (NLB)** with TLS listeners
- **Amazon CloudFront**
- **API Gateway custom domains**
- Some other integrated AWS services

Key ACM facts:

- ACM can issue public certificates for supported domains.
- ACM automatically renews eligible public certificates when validation remains in place.
- You can import third-party certificates if needed.
- Certificate placement depends on the service and endpoint type.

Examples:

- **CloudFront**: certificate must be in `us-east-1`.
- **API Gateway edge-optimized custom domain**: certificate must be in `us-east-1`.
- **API Gateway regional custom domain**: certificate must be in the same Region as the API.
- **ALB/NLB**: certificate must be in the same Region as the load balancer.

**CloudFront, ALB, NLB, and API Gateway**

These services terminate TLS differently, so exam questions often hinge on endpoint type.

CloudFront:

- Terminates TLS at the edge.
- Uses ACM certificates in `us-east-1`.
- Can enforce HTTPS by redirecting HTTP to HTTPS.
- Supports modern TLS policies for viewer connections.

Example CloudFront behavior:

- Viewer connects with HTTPS.
- CloudFront presents the ACM certificate.
- CloudFront can use HTTPS to the origin as well.

ALB:

- Supports HTTPS listeners with ACM certificates.
- Can redirect HTTP to HTTPS.
- Can forward traffic to targets over HTTP or HTTPS depending on your design.

Example ALB listener rule concept:

```text
HTTP :80  -> redirect to HTTPS :443
HTTPS:443 -> forward to target group
```

NLB:

- Supports **TCP** and **TLS** listeners.
- Can terminate TLS at the load balancer with a TLS listener.
- Can also pass TCP traffic through to targets if you want the target to handle TLS termination.
- Do not confuse TCP pass-through with TLS pass-through; they are not the same thing.

API Gateway:

- Terminates TLS for client connections.
- Supports custom domains with ACM certificates.
- Edge-optimized and regional custom domains use different certificate placement rules.
- Private APIs are commonly accessed through VPC endpoints and have different network exposure patterns.

Example API Gateway custom domain association:

```bash
aws apigateway create-domain-name \
  --domain-name api.example.com \
  --certificate-arn arn:aws:acm:us-east-1:111122223333:certificate/abcd-1234
```

**PrivateLink and VPC endpoints**

AWS PrivateLink and VPC endpoints are common in-transit security patterns because they keep traffic on the AWS backbone rather than traversing the public internet.

Use them when you want:

- Private access to AWS services from a VPC.
- Reduced exposure to public endpoints.
- Simpler network controls for internal applications.

Endpoint types:

- **Gateway endpoints**: commonly for S3 and DynamoDB.
- **Interface endpoints**: powered by AWS PrivateLink for many services.

Important distinction:

- VPC endpoints improve network privacy and routing.
- They do not replace encryption by themselves.
- You still use TLS or service encryption where required.

**Direct Connect and VPN**

AWS Direct Connect provides a private network connection to AWS, but it does not inherently encrypt traffic.

Options for encryption:

- Use **Site-to-Site VPN** for IPsec-encrypted tunnels.
- Use **TLS** at the application layer.
- Combine Direct Connect with VPN if you need both private routing and encryption.

Exam-safe phrasing:

- Direct Connect improves network path control and bandwidth.
- It does not automatically satisfy encryption requirements.
- Add encryption explicitly when the scenario requires confidentiality.

**Comparing common encryption choices**

| Need | Typical AWS choice | Notes |
|---|---|---|
| Encrypt S3 objects at rest | SSE-S3 or SSE-KMS | Use SSE-KMS when audit/control matters |
| Enforce HTTPS to S3 | Bucket policy with `aws:SecureTransport` | Often paired with SSE-KMS |
| Encrypt EBS volumes | EBS encryption with KMS | Snapshots inherit encryption |
| Encrypt RDS data | RDS encryption with KMS | Existing unencrypted DBs require migration |
| Encrypt DynamoDB tables | DynamoDB encryption at rest | Often enabled by default |
| Encrypt secrets | Secrets Manager or SSM SecureString | Both use KMS |
| Encrypt web traffic | TLS/HTTPS | Use ACM for certificates |
| Keep service traffic private | VPC endpoints / PrivateLink | Still use encryption as needed |

**Practical exam patterns**

When you see a scenario, map it to the right control:

- If the requirement is “no one can read stored data without authorization,” choose **encryption at rest** with KMS or a service-managed equivalent.
- If the requirement is “traffic must not be readable on the wire,” choose **TLS**, **HTTPS**, **VPN**, or **IPsec**.
- If the requirement is “access must stay off the public internet,” consider **PrivateLink**, **VPC endpoints**, or private connectivity.
- If the requirement is “auditable key usage and customer control,” prefer **customer managed KMS keys**.
- If the requirement is “simple managed encryption with minimal operations,” prefer the service’s default encryption option.

**Common pitfalls**

- Do not assume encryption at rest also encrypts traffic in transit.
- Do not assume a private network path removes the need for TLS.
- Do not assume all AWS services use the same certificate placement rules.
- Do not assume you can enable encryption in place for every existing database or storage resource.
- Do not assume a KMS key policy alone is enough for every cross-account scenario.
- Do not confuse service-managed encryption defaults with customer managed key control.

**Quick reference commands**

```bash
# Check an S3 bucket policy
aws s3api get-bucket-policy --bucket my-bucket

# Upload with SSE-KMS
aws s3 cp file.txt s3://my-bucket/ --sse aws:kms --sse-kms-key-id alias/my-key

# Create a KMS key
aws kms create-key --description "App key"

# Enable KMS rotation
aws kms enable-key-rotation --key-id alias/my-key

# Create an encrypted EBS volume
aws ec2 create-volume --availability-zone us-east-1a --size 20 --encrypted --kms-key-id alias/ebs-key

# Put a SecureString parameter
aws ssm put-parameter --name "/app/token" --type SecureString --value "value" --key-id alias/ssm-key
```

**Summary**

Encryption at rest protects stored data, while encryption in transit protects data moving across networks. In AWS, KMS is the central control plane for many encryption workflows, but each service has its own defaults, certificate rules, and migration constraints. For exam success, identify the service, the data state, and whether the requirement is confidentiality, compliance, private routing, or key control.
