## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**AWS Config overview**

AWS Config records the configuration state of supported AWS resources and lets you inspect how those resources change over time. It is useful when you need an audit trail, a point-in-time view, or evidence for compliance checks.

At a high level, AWS Config does three things:

- Tracks configuration changes for supported resources.
- Stores configuration history and snapshots in the delivery channel S3 bucket, while also keeping queryable configuration data in AWS Config.
- Evaluates resources against rules to produce compliance results.

A useful mental model is:

- **Configuration history** answers: “What changed, and when?”
- **Compliance status** answers: “Does this resource meet the rule?”
- **Rule evaluation** answers: “Why is it compliant or noncompliant?”

**Core concepts**

AWS Config uses several terms that are easy to mix up:

- **Configuration item**: A point-in-time record of a resource’s configuration.
- **Configuration history**: The sequence of configuration items for a resource.
- **Configuration snapshot**: A collection of configuration items for resources at a specific time.
- **Delivery channel**: The mechanism that delivers snapshots and history files to an S3 bucket.
- **Config rule**: A policy that evaluates whether a resource configuration is compliant.

A resource can be recorded by AWS Config and still have no compliance result for a specific rule if that rule does not apply to it. Compliance is evaluated per rule, not globally per resource.

**Inspecting configuration history**

When you inspect configuration history, you are usually looking for drift, change timing, or the exact state of a resource before and after an event. This is especially helpful for troubleshooting outages, security incidents, and accidental changes.

Common use cases include:

- Verifying when a security group rule changed.
- Checking whether an IAM role, bucket policy, or EC2 instance setting was modified.
- Comparing the current state with a previous configuration item.
- Establishing a timeline for incident response.

The AWS Management Console is the easiest way to browse history, but the AWS CLI is useful for repeatable investigations.

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::SecurityGroup \
  --resource-id sg-0123456789abcdef0 \
  --limit 10
```

This command returns configuration history entries for the specified resource. In practice, the response can be paginated, so use `--limit` carefully and add pagination handling when you need a full timeline.

Useful options to know:

- `--resource-type` identifies the AWS resource type.
- `--resource-id` identifies the specific resource.
- `--later-time` and `--earlier-time` narrow the time window.
- `--limit` controls how many items are returned per page.

For a more targeted investigation, you can query a time range:

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::Instance \
  --resource-id i-0123456789abcdef0 \
  --earlier-time 2026-04-01T00:00:00Z \
  --later-time 2026-04-26T00:00:00Z \
  --limit 5
```

> The exact output shape and pagination behavior depend on the API response size. For large histories, use the CLI paginator or filter the results with `--query`.

A practical workflow is:

1. Identify the resource type and ID from the console or another AWS service.
2. Pull the configuration history for the relevant time window.
3. Compare successive configuration items to isolate the change.
4. Correlate the change time with CloudTrail events if you need the actor or API call.

**Snapshots and delivery**

AWS Config can deliver configuration snapshots and history files to an S3 bucket through the delivery channel. This is useful for archival, offline analysis, and integration with other tooling.

Important points:

- The S3 bucket is the delivery destination, not the only place where AWS Config data exists.
- AWS Config also maintains queryable configuration data and compliance state in the service.
- Snapshots are point-in-time collections, while history is a sequence of changes.

A typical delivery channel setup includes:

- An S3 bucket for configuration history and snapshots.
- An SNS topic for notifications, if configured.
- A recorder that defines which resource types are tracked.

Example CLI to describe the recorder and delivery channel:

```bash
aws configservice describe-configuration-recorders
aws configservice describe-delivery-channels
```

If you need to start recording, the recorder must be enabled and associated with a delivery channel:

```bash
aws configservice start-configuration-recorder \
  --configuration-recorder-name default
```

**Compliance settings and rules**

AWS Config rules evaluate resource configurations against desired conditions. Rules can be managed by AWS or custom-built with Lambda.

Two broad rule types matter for exam scenarios:

- **Managed rules**: Prebuilt rules provided by AWS.
- **Custom rules**: Rules you define with AWS Lambda or other supported mechanisms.

A common managed rule example is the AWS Config rule name `restricted-ssh`, which uses the managed rule source identifier `INCOMING_SSH_DISABLED`. The rule checks whether inbound SSH access is disabled, typically by inspecting security groups.

Example CLI for creating the rule:

```bash
aws configservice put-config-rule --config-rule '{
  "ConfigRuleName": "restricted-ssh",
  "Description": "Checks that security groups do not allow inbound SSH from 0.0.0.0/0",
  "Scope": {
    "ComplianceResourceTypes": ["AWS::EC2::SecurityGroup"]
  },
  "Source": {
    "Owner": "AWS",
    "SourceIdentifier": "INCOMING_SSH_DISABLED"
  }
}'
```

Key distinctions:

- **ConfigRuleName** is the name you assign in AWS Config.
- **SourceIdentifier** is the AWS-managed rule identifier.
- The rule name and source identifier are not the same thing.

For custom rules, the source is usually `Owner: "CUSTOM_LAMBDA"` and includes a Lambda function ARN.

```bash
aws configservice put-config-rule --config-rule '{
  "ConfigRuleName": "s3-bucket-versioning-enabled",
  "Scope": {
    "ComplianceResourceTypes": ["AWS::S3::Bucket"]
  },
  "Source": {
    "Owner": "CUSTOM_LAMBDA",
    "SourceIdentifier": "arn:aws:lambda:us-east-1:123456789012:function:check-s3-versioning"
  }
}'
```

**Reading compliance results**

Compliance results tell you whether a resource is compliant with a specific rule. They do not replace the underlying configuration history; they summarize rule evaluation.

Useful APIs and console views include:

- `describe-compliance-by-config-rule`
- `describe-compliance-by-resource`
- `describe-compliance-by-resource-type`

These are related but not interchangeable:

- `describe-compliance-by-config-rule` shows compliance counts and status for a rule.
- `describe-compliance-by-resource` shows compliance across rules for a resource.
- `describe-compliance-by-resource-type` groups compliance by resource type.

Example:

```bash
aws configservice describe-compliance-by-resource \
  --resource-type AWS::EC2::SecurityGroup \
  --resource-id sg-0123456789abcdef0
```

This is useful when you want to see which rules affect a specific resource. It does not provide the full evaluation details for every rule, so use rule-specific APIs when you need deeper diagnostics.

To inspect a rule’s evaluation results, use:

```bash
aws configservice get-compliance-details-by-config-rule \
  --config-rule-name restricted-ssh
```

You can also query the latest evaluation status for a rule:

```bash
aws configservice describe-config-rule-evaluation-status \
  --config-rule-names restricted-ssh
```

**How history and compliance work together**

The strongest troubleshooting pattern is to combine configuration history with compliance results.

Use history when you need to answer:

- What changed?
- When did it change?
- What was the previous state?

Use compliance when you need to answer:

- Which rule failed?
- Is the resource currently compliant?
- Which resources are affected by a rule?

A practical incident workflow looks like this:

1. Find the noncompliant resource in AWS Config.
2. Identify the rule that failed.
3. Inspect the resource’s configuration history around the failure time.
4. Correlate the change with CloudTrail to find the API caller.
5. Remediate the configuration and confirm the rule returns to compliant.

**Example investigation scenario**

Suppose a security group becomes noncompliant because SSH is opened to the internet.

You might:

- Check the rule result for `restricted-ssh`.
- Inspect the security group history to find when port 22 was opened.
- Review CloudTrail for the `AuthorizeSecurityGroupIngress` event.
- Remove the offending rule and verify compliance again.

Example CLI for the security group history:

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::SecurityGroup \
  --resource-id sg-0123456789abcdef0 \
  --limit 3
```

Example CLI for the compliance view:

```bash
aws configservice describe-compliance-by-config-rule \
  --config-rule-names restricted-ssh
```

**Common exam distinctions**

AWS Config questions often test subtle differences:

- **History vs. compliance**
  - History shows configuration changes.
  - Compliance shows rule evaluation results.

- **Snapshot vs. history**
  - Snapshot is a point-in-time collection.
  - History is a sequence of changes.

- **Managed vs. custom rules**
  - Managed rules use AWS-provided source identifiers.
  - Custom rules use Lambda or another supported custom source.

- **Resource-level vs. rule-level views**
  - Resource-level views help you inspect one resource across rules.
  - Rule-level views help you inspect one rule across resources.

- **S3 delivery vs. service data**
  - S3 stores delivered snapshots and history files.
  - AWS Config also retains queryable records and compliance state in the service.

**Practical CLI patterns**

When you work with AWS Config from the CLI, these commands are especially useful:

```bash
aws configservice describe-configuration-recorders
aws configservice describe-delivery-channels
aws configservice describe-config-rules
aws configservice describe-compliance-by-config-rule --config-rule-names restricted-ssh
aws configservice describe-compliance-by-resource --resource-type AWS::EC2::SecurityGroup --resource-id sg-0123456789abcdef0
```

For focused output, combine with `--query`:

```bash
aws configservice describe-compliance-by-config-rule \
  --config-rule-names restricted-ssh \
  --query 'ComplianceByConfigRules[0].Compliance.ComplianceType'
```

You can also use `jq` to inspect JSON responses:

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::Instance \
  --resource-id i-0123456789abcdef0 \
  | jq '.configurationItems[] | {captureTime, configurationItemStatus, resourceType, resourceId}'
```

**Troubleshooting tips**

If you do not see expected data, check the following:

- AWS Config recording is enabled for the account and region.
- The resource type is supported and included in recording scope.
- The delivery channel exists and points to the correct S3 bucket.
- The rule applies to the resource type you are inspecting.
- The resource has had enough time to be recorded and evaluated.

If compliance appears missing, remember that:

- A resource may be recorded but not evaluated by a given rule.
- A rule may be created after the resource already existed.
- Some rules evaluate only specific resource types or properties.

**Key takeaways**

AWS Config is the primary service for tracking resource configuration changes and evaluating compliance over time. For exam scenarios, focus on the difference between configuration history, snapshots, and compliance results, and remember that rule names and managed rule source identifiers are separate values.

When you need to investigate a change, use history. When you need to assess policy adherence, use compliance. When you need both, combine AWS Config with CloudTrail for a complete audit trail.
