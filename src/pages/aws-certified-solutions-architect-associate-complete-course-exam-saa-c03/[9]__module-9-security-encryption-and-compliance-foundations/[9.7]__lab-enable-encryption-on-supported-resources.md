## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and exam context**

This lab focuses on enabling encryption for common AWS resources and understanding the difference between **encryption at rest** and **encryption in transit**. For exam prep, you need to know not only how to turn encryption on, but also which services use AWS-owned keys by default, which support AWS managed keys, and which can use customer managed keys in AWS Key Management Service (**KMS**).

A strong SAA-C03 answer usually depends on service-specific behavior:

- Some services encrypt by default and do not expose a “turn on encryption” workflow in the same way as others.
- Some services let you choose a KMS key only at creation time.
- Some services support only certain engine, version, or node-type combinations.
- Some encryption controls are enforced by policy, while others are enforced by service configuration.

**Core KMS concepts**

AWS KMS is the central service for managing encryption keys used by many AWS services. The most important exam concepts are:

- **AWS owned keys** are fully managed by AWS and are not visible in your account.
- **AWS managed keys** are created and managed by AWS for a service in your account.
- **Customer managed keys** give you the most control, including key policy, rotation, aliasing, and cross-account sharing patterns.
- **Data keys** are commonly used by services for envelope encryption, where KMS protects a key that encrypts the actual data.

A practical KMS workflow often includes:

1. Create or select a KMS key.
2. Grant the service permission to use the key.
3. Enable encryption in the target service.
4. Verify the resource shows encrypted status and that the key ARN or alias is attached where applicable.

**KMS key management essentials**

Use aliases to make operational changes easier. An alias lets you point applications and automation at a stable name such as `alias/app-prod-data` instead of a raw key ID.

```bash
aws kms create-key \
  --description "Key for application data encryption" \
  --key-usage ENCRYPT_DECRYPT \
  --origin AWS_KMS

aws kms create-alias \
  --alias-name alias/app-prod-data \
  --target-key-id <key-id>
```

Key rotation is important for long-lived data protection:

- **Automatic rotation** is available for customer managed symmetric KMS keys.
- Rotation changes the backing cryptographic material, not the alias or key ID used by your applications.
- For exam scenarios, rotation is a security best practice, but it does not usually require re-encrypting every object manually.

Cross-account usage requires a careful key policy and, often, grants:

- The key policy must allow the external principal or the service role.
- The consuming account must be authorized to use the key.
- The resource policy on the target service may also need to allow access.

> Be careful with cross-account encryption questions: access to the encrypted resource and permission to use the KMS key are separate problems.

**Service-by-service at-rest encryption**

**Amazon S3**

Amazon S3 supports server-side encryption with several options:

- **SSE-S3** uses AWS-managed keys.
- **SSE-KMS** uses KMS keys, including customer managed keys.
- **SSE-C** uses customer-provided keys, which is less common in exam scenarios.

S3 bucket default encryption applies to new objects uploaded after the setting is enabled. It does not automatically re-encrypt existing objects.

Console path:

1. Open the S3 bucket.
2. Go to the **Properties** tab.
3. Find **Default encryption**.
4. Choose **Server-side encryption with Amazon S3 managed keys (SSE-S3)** or **AWS Key Management Service key (SSE-KMS)**.
5. Save the configuration.

CLI example:

```bash
aws s3api put-bucket-encryption \
  --bucket my-secure-bucket \
  --server-side-encryption-configuration '{
    "Rules": [
      {
        "ApplyServerSideEncryptionByDefault": {
          "SSEAlgorithm": "aws:kms",
          "KMSMasterKeyID": "alias/app-prod-data"
        }
      }
    ]
  }'
```

Verification:

```bash
aws s3api get-bucket-encryption --bucket my-secure-bucket
aws s3api head-object --bucket my-secure-bucket --key path/to/object.txt
```

Practical notes:

- Bucket default encryption is a default, not a hard enforcement control.
- To enforce SSE-KMS, use a bucket policy that denies uploads unless the request uses the expected encryption headers and, when needed, the expected KMS key.
- Multipart uploads and alternate upload paths can bypass simplistic policy checks if you do not validate carefully.

Example enforcement pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnencryptedUploads",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-secure-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    }
  ]
}
```

This is a starting point only. In real designs, you often also restrict the KMS key ID and account for multipart upload behavior and other write paths.

**Amazon EBS**

Amazon Elastic Block Store (**EBS**) supports encryption at rest, and encryption by default can be enabled at the account and Region level. When encryption by default is on, new volumes and snapshots created from them are encrypted automatically unless a workflow explicitly changes the source or destination behavior.

Console path:

1. Open the **EC2** console.
2. Go to **Elastic Block Store** > **Encryption**.
3. Enable **EBS encryption by default**.
4. Optionally set the default KMS key for EBS encryption.

CLI example:

```bash
aws ec2 enable-ebs-encryption-by-default
aws ec2 modify-ebs-default-kms-key-id --kms-key-id alias/ebs-prod
aws ec2 get-ebs-encryption-by-default
```

For a new encrypted volume:

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --size 20 \
  --volume-type gp3 \
  --encrypted \
  --kms-key-id alias/ebs-prod
```

Snapshot copy and re-encryption:

- Copying a snapshot to another Region can apply encryption in the destination Region.
- The destination Region must have a usable KMS key.
- If the source snapshot is unencrypted, you can copy it and encrypt the copy.
- If the source snapshot is encrypted, you can re-encrypt the copy with a different key if permissions allow it.

Example:

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --encrypted \
  --kms-key-id alias/ebs-prod \
  --destination-region us-west-2
```

In practice, you usually run the copy in the destination Region context and ensure the key exists there.

**Amazon RDS and Aurora**

Amazon Relational Database Service (**RDS**) supports encryption at rest for supported engines. For many engines, encryption is chosen at creation time and cannot be turned on later for an existing unencrypted database instance. The common migration path is snapshot copy and restore into an encrypted instance.

Console path for a new encrypted DB instance:

1. Open the **RDS** console.
2. Choose **Create database**.
3. Select a supported engine.
4. Under **Encryption**, enable encryption and choose a KMS key.
5. Complete the networking and instance settings.

CLI example for supported engines:

```bash
aws rds create-db-instance \
  --db-instance-identifier prod-db \
  --db-instance-class db.t3.medium \
  --engine mysql \
  --allocated-storage 100 \
  --master-username admin \
  --master-user-password 'ChangeMe123!' \
  --storage-encrypted \
  --kms-key-id alias/rds-prod
```

For Aurora, encryption is applied at the cluster level and the real workflow usually includes subnet group, security group, and parameter context. A simplified example is:

```bash
aws rds create-db-cluster \
  --db-cluster-identifier prod-aurora \
  --engine aurora-mysql \
  --master-username admin \
  --master-user-password 'ChangeMe123!' \
  --storage-encrypted \
  --kms-key-id alias/rds-prod
```

Operationally, you also need the appropriate subnet group and network access controls.

Restoring an encrypted database from a snapshot:

```bash
aws rds copy-db-snapshot \
  --source-db-snapshot-identifier arn:aws:rds:us-east-1:123456789012:snapshot:prod-snap \
  --target-db-snapshot-identifier prod-snap-encrypted \
  --kms-key-id alias/rds-prod

aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier prod-db-restored \
  --db-snapshot-identifier prod-snap-encrypted
```

Exam focus:

- You generally do not “toggle” encryption on an existing RDS instance in place.
- Snapshot copy and restore is the standard migration pattern.
- Aurora and RDS have different operational shapes, but both rely on KMS for at-rest encryption.

**Amazon DynamoDB**

Amazon DynamoDB is encrypted at rest by default. This is an important exam nuance: you do not usually enable encryption as a separate table-creation step the way you might for some other services.

Key points:

- Encryption at rest is always on.
- DynamoDB uses AWS-owned or AWS-managed encryption under the hood.
- You do not normally manage table encryption through a table creation parameter in the same way as EBS or RDS.
- For exam purposes, focus on the fact that the service is encrypted by default and that the main design question is usually access control, not whether encryption is enabled.

CLI example for table creation, without encryption flags:

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=OrderId,AttributeType=S \
  --key-schema AttributeName=OrderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

Verification:

```bash
aws dynamodb describe-table --table-name Orders
```

Look for the table status and general metadata, but do not expect a typical “turn on encryption” workflow in the create-table output.

**Amazon EFS**

Amazon Elastic File System (**EFS**) supports encryption at rest and in transit. Encryption at rest is selected when you create the file system, and you can choose a KMS key if you want customer managed control.

Console path:

1. Open the **EFS** console.
2. Choose **Create file system**.
3. Expand **Customize**.
4. Enable **Encryption**.
5. Select the KMS key if needed.
6. Complete mount target and network settings.

CLI example:

```bash
aws efs create-file-system \
  --creation-token my-efs-01 \
  --encrypted \
  --kms-key-id alias/efs-prod
```

Verification:

```bash
aws efs describe-file-systems --file-system-id fs-1234567890abcdef0
```

In-transit encryption for EFS:

- Use TLS when mounting the file system.
- The `amazon-efs-utils` mount helper can mount with TLS.
- This protects traffic between the client and EFS mount target.

Example mount command:

```bash
sudo mount -t efs -o tls fs-1234567890abcdef0:/ /mnt/efs
```

**Amazon ElastiCache**

Amazon ElastiCache supports encryption at rest and in transit for supported engine, version, and node-type combinations. Do not assume every Redis or Valkey configuration supports encryption.

Important constraints:

- Encryption support depends on the engine and version.
- Not all node types or cluster modes support every security feature.
- For exam questions, always check whether the scenario mentions a supported replication group or cluster configuration.

Console path:

1. Open the **ElastiCache** console.
2. Choose **Create**.
3. Select the engine and a supported configuration.
4. Enable **Encryption at rest** if available.
5. Enable **Encryption in transit** if available.
6. Choose the KMS key if the workflow exposes it.

CLI example pattern for a supported replication group:

```bash
aws elasticache create-replication-group \
  --replication-group-id prod-cache \
  --replication-group-description "Encrypted cache" \
  --engine redis \
  --cache-node-type cache.r6g.large \
  --num-node-groups 1 \
  --replicas-per-node-group 1 \
  --transit-encryption-enabled \
  --at-rest-encryption-enabled \
  --auth-token 'UseAStrongSecret123!' \
  --kms-key-id alias/cache-prod
```

Use this as a pattern only. Real-world creation requires the correct engine/version and supported topology.

Verification:

```bash
aws elasticache describe-replication-groups --replication-group-id prod-cache
```

Look for encryption-related fields and confirm the cluster is in a supported state.

**Amazon FSx**

Amazon FSx supports encryption at rest, but the exact CLI shape depends on the FSx type. This is a common exam trap: there is no single universal `create-file-system` command that works identically for all FSx variants.

FSx variants include:

- **FSx for Windows File Server**
- **FSx for Lustre**
- **FSx for NetApp ONTAP**
- **FSx for OpenZFS**

General guidance:

- Encryption at rest is enabled during file system creation.
- You can often choose a customer managed KMS key.
- Network and storage parameters vary by FSx type.

Example pattern for a Windows file system:

```bash
aws fsx create-file-system \
  --file-system-type WINDOWS \
  --storage-capacity 300 \
  --subnet-ids subnet-0123456789abcdef0 \
  --windows-configuration ActiveDirectoryId=d-1234567890,ThroughputCapacity=32,PreferredSubnetId=subnet-0123456789abcdef0,DeploymentType=SINGLE_AZ_2 \
  --kms-key-id alias/fsx-prod
```

Example pattern for Lustre:

```bash
aws fsx create-file-system \
  --file-system-type LUSTRE \
  --storage-capacity 1200 \
  --subnet-ids subnet-0123456789abcdef0 \
  --lustre-configuration DeploymentType=PERSISTENT_2,PerUnitStorageThroughput=125 \
  --kms-key-id alias/fsx-prod
```

Treat these as service-specific patterns, not universal commands.

**Amazon Redshift**

Amazon Redshift supports encryption at rest, and cluster encryption is typically selected at creation time. In-transit encryption is supported through TLS, but client behavior and cluster settings matter.

Console path:

1. Open the **Redshift** console.
2. Choose **Create cluster**.
3. Enable encryption.
4. Select a KMS key if you want customer managed encryption.
5. Complete the network and parameter settings.

CLI example:

```bash
aws redshift create-cluster \
  --cluster-identifier analytics-prod \
  --node-type ra3.xlplus \
  --master-username adminuser \
  --master-user-password 'ChangeMe123!' \
  --encrypted \
  --kms-key-id alias/redshift-prod
```

Verification:

```bash
aws redshift describe-clusters --cluster-identifier analytics-prod
```

For in-transit encryption, use a TLS-capable JDBC or ODBC connection string and verify the client is configured to require SSL/TLS.

Example JDBC pattern:

```text
jdbc:redshift://analytics-prod.xxxxxx.us-east-1.redshift.amazonaws.com:5439/dev?ssl=true&sslmode=verify-full
```

**Messaging services: SQS and SNS**

Amazon Simple Queue Service (**SQS**) and Amazon Simple Notification Service (**SNS**) support encryption at rest with KMS. These services are often tested in scenarios where you need to protect message payloads without changing application logic much.

SQS example:

```bash
aws sqs create-queue \
  --queue-name secure-queue \
  --attributes KmsMasterKeyId=alias/messaging-prod
```

SNS example:

```bash
aws sns create-topic \
  --name secure-topic \
  --attributes KmsMasterKeyId=alias/messaging-prod
```

Practical notes:

- Encryption at rest protects stored messages and topic data.
- You still need IAM permissions for producers and consumers.
- If a service publishes to SNS or reads from SQS, the KMS key policy must allow the service to use the key.

**Encryption in transit**

Encryption in transit protects data while it moves between clients, AWS services, and endpoints. The most common mechanism is **TLS** (Transport Layer Security), often exposed as HTTPS, SSL/TLS, or a service-specific secure connection option.

Service-specific examples:

- **S3**: Use `https://` endpoints and block plain HTTP where possible.
- **RDS/Aurora**: Use SSL/TLS in the database client and verify the server certificate.
- **EFS**: Mount with TLS using `amazon-efs-utils`.
- **ElastiCache**: Use in-transit encryption and a client that supports TLS.
- **Redshift**: Use SSL/TLS in the JDBC or ODBC connection string.

A better verification pattern than “just curl the endpoint” is to test the actual protocol requirement:

```bash
curl -I https://my-bucket.s3.amazonaws.com/
```

This confirms HTTPS connectivity, but it does not prove object-level encryption or bucket policy enforcement. For exam prep, distinguish between transport security and storage encryption.

For RDS, a client-side check is more meaningful:

```bash
mysql -h prod-db.xxxxxx.us-east-1.rds.amazonaws.com -u admin -p --ssl-mode=REQUIRED
```

For EFS, the mount helper is the verification:

```bash
sudo mount -t efs -o tls fs-1234567890abcdef0:/ /mnt/efs
```

**Default encryption behavior and common traps**

Understanding defaults is often more important than memorizing commands.

- **EBS** can be configured for encryption by default at the account and Region level.
- **S3** bucket default encryption applies to new objects only.
- **DynamoDB** is encrypted at rest by default.
- **RDS/Aurora** usually require encryption at creation time for new encrypted resources.
- **EFS** encryption is chosen at file system creation.
- **Redshift** encryption is typically chosen at cluster creation.
- **ElastiCache** and **FSx** depend on supported configurations and creation-time settings.

Common exam traps:

- Assuming you can enable encryption later on every service.
- Assuming a bucket default encryption setting retroactively changes existing S3 objects.
- Assuming a service supports customer managed keys when it may only support AWS-owned or AWS-managed keys in a given workflow.
- Assuming a single CLI command shape works for all FSx variants.
- Confusing HTTPS connectivity with proof of at-rest encryption.

**Migration and re-encryption strategies**

When a resource is already live, you often need a migration path rather than an in-place toggle.

For EBS:

1. Create an encrypted snapshot copy.
2. Create a new encrypted volume from the snapshot.
3. Attach the new volume to the instance.
4. Validate the application before decommissioning the old volume.

For RDS/Aurora:

1. Take a snapshot of the source database.
2. Copy the snapshot and encrypt the copy if needed.
3. Restore a new encrypted instance or cluster from the encrypted snapshot.
4. Cut over the application endpoint.

For S3:

1. Enable bucket default encryption for future objects.
2. Re-upload or copy existing objects to apply encryption to old data.
3. Use S3 Batch Operations or scripted copy workflows for large datasets.

Example object copy with SSE-KMS:

```bash
aws s3 cp s3://my-bucket/legacy-data.csv s3://my-bucket/legacy-data.csv \
  --sse aws:kms \
  --sse-kms-key-id alias/app-prod-data
```

For large-scale re-encryption, S3 Batch Operations is often the operationally realistic answer.

**Verification checklist**

Use service-specific checks rather than generic assumptions.

- **S3**
  - Confirm bucket default encryption in the console or with `get-bucket-encryption`.
  - Confirm object metadata with `head-object`.
- **EBS**
  - Confirm encryption by default with `get-ebs-encryption-by-default`.
  - Confirm a volume shows `Encrypted: true`.
- **RDS/Aurora**
  - Confirm the instance or cluster shows encryption enabled and the KMS key ARN.
  - Confirm restored resources inherit the intended encryption state.
- **DynamoDB**
  - Confirm the table exists and remember encryption is on by default.
- **EFS**
  - Confirm the file system shows encryption enabled and the selected KMS key.
- **ElastiCache**
  - Confirm the replication group shows at-rest and in-transit encryption where supported.
- **FSx**
  - Confirm the file system type and encryption settings match the chosen variant.
- **Redshift**
  - Confirm the cluster is encrypted and the client uses TLS.

**Exam-ready decision guide**

Use this mental model when you see an encryption question:

- If the service supports encryption by default, ask whether the question is about **default behavior** or **enforcement**.
- If the resource already exists unencrypted, ask whether the service supports **in-place encryption changes** or requires **migration**.
- If the question mentions a KMS key, ask whether the service supports **customer managed keys** or only AWS-managed/AWS-owned keys in that workflow.
- If the question mentions cross-account access, separate **resource permissions** from **KMS permissions**.
- If the question mentions transport security, verify whether the client must use **TLS**, **HTTPS**, or a service-specific secure mount/connection option.

**Quick reference**

| Service | At-rest encryption | In-transit encryption | Customer managed KMS key support | Common exam note |
|---|---|---|---|---|
| S3 | Yes | Yes | Yes | Default encryption applies to new objects only |
| EBS | Yes | N/A for block storage path | Yes | Encryption by default can be enabled per Region |
| RDS | Yes | Yes | Yes | Often chosen at creation; migration uses snapshot copy/restore |
| Aurora | Yes | Yes | Yes | Cluster-level encryption and network context matter |
| DynamoDB | Yes, by default | Yes for API traffic | Limited to service behavior; not a manual table toggle | Do not treat encryption as a create-table option |
| EFS | Yes | Yes | Yes | TLS mount is the usual in-transit control |
| ElastiCache | Yes, for supported configs | Yes, for supported configs | Yes, for supported configs | Engine/version/node-type constraints apply |
| FSx | Yes | Depends on FSx type | Yes | No universal CLI shape across all variants |
| Redshift | Yes | Yes | Yes | TLS depends on client and cluster settings |
| SQS/SNS | Yes | API traffic uses TLS | Yes | KMS key policy must allow service use |

**Troubleshooting tips**

- If encryption settings do not appear, check whether the service or engine version supports them.
- If a KMS key is rejected, check the key policy, alias, Region, and account.
- If a restore fails, confirm the snapshot is in the correct Region and the destination key is usable there.
- If a client cannot connect after enabling transport encryption, verify the client library supports TLS and the correct certificate or SSL mode.
- If a policy does not block unencrypted uploads, review all write paths, not just `PutObject`.

**Hands-on lab flow**

1. Create a KMS key and alias for the lab.
2. Enable S3 default encryption on a test bucket.
3. Enable EBS encryption by default in the Region.
4. Create an encrypted EBS volume and verify it.
5. Create an encrypted RDS instance or restore one from an encrypted snapshot.
6. Confirm DynamoDB table creation works with the understanding that encryption is already on.
7. Create an encrypted EFS file system and mount it with TLS.
8. Create a supported ElastiCache replication group with encryption enabled.
9. Create an encrypted FSx file system using the correct FSx-specific workflow.
10. Create an encrypted Redshift cluster and connect with TLS.
11. Test SQS or SNS with a KMS key if your lab includes messaging.
12. Record which services require creation-time decisions versus migration workflows.

**Key takeaways**

- Encryption at rest and encryption in transit are separate controls.
- KMS is the central service for customer managed encryption in AWS.
- Some services encrypt by default, while others require creation-time configuration.
- Existing unencrypted resources often need snapshot, copy, restore, or re-upload workflows.
- Exam questions often test the difference between default encryption, enforcement, and client-side TLS.
