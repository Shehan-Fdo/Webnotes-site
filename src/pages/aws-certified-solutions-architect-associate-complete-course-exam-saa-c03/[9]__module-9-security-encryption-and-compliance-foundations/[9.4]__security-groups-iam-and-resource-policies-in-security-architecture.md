## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Security architecture layers**

AWS authorization is layered. For exam questions, you usually need to identify which layer controls network reachability, which layer controls API authorization, and which layer grants cross-account access to a specific resource.

- **Security groups** control instance-level network traffic.
- **Network ACLs (NACLs)** control subnet-level traffic.
- **IAM** controls who can call AWS APIs and what they can do.
- **Resource policies** control who can access a specific resource directly.
- **Trust policies** control who can assume a role.
- **Permissions boundaries**, **service control policies (SCPs)**, **session policies**, and **endpoint policies** can further restrict access.

A common exam trap is assuming one policy type replaces another. In practice, AWS evaluates multiple policy layers together, and the request succeeds only if every required layer allows it.

**Security groups**

A security group is a stateful virtual firewall attached to an Elastic Network Interface (ENI). It allows inbound and outbound traffic based on rules, and return traffic is automatically allowed.

- **Stateful behavior** means you do not need an explicit return rule for response packets.
- Rules are allow-only; there are no deny rules.
- You can reference another security group as a source or destination in many same-VPC and peered-VPC designs, but architecture constraints apply.
- Security groups are not a universal cross-network authorization tool; in some designs involving VPC peering, Transit Gateway, or overlapping routing domains, you may need different patterns such as CIDR-based rules, load balancers, or private connectivity design changes.

A practical example for an application tier:

```text
Web SG:
- Inbound: TCP 80 from 0.0.0.0/0
- Inbound: TCP 443 from 0.0.0.0/0
- Outbound: TCP 8080 to App SG

App SG:
- Inbound: TCP 8080 from Web SG
- Outbound: TCP 3306 to DB SG

DB SG:
- Inbound: TCP 3306 from App SG
- Outbound: none required for return traffic because SGs are stateful
```

Use security groups when you want instance or ENI-level filtering and simple allow-based rules. They are the primary answer for “how do I permit traffic to this EC2 instance?” questions.

**Network ACLs and ephemeral ports**

A NACL is stateless and applies at the subnet boundary. Because it is stateless, you must allow both directions explicitly.

- NACLs evaluate rules in ascending rule number order.
- The first matching rule wins.
- Lower rule numbers take precedence.
- You can create explicit deny rules, which is useful for subnet-wide blocking.
- Return traffic requires ephemeral port handling.

A concrete example:

```text
Rule 100: ALLOW TCP 80 from 0.0.0.0/0
Rule 110: ALLOW TCP 443 from 0.0.0.0/0
Rule 120: ALLOW TCP 1024-65535 to 0.0.0.0/0
Rule 130: DENY ALL 0.0.0.0/0
```

Why the ephemeral range matters:

- A client connects to a web server on port 443.
- The server responds from port 443 to the client’s ephemeral source port, often in the range `1024-65535`.
- If the outbound or inbound return path is blocked by the NACL, the connection fails even if the security group allows it.

For exam scenarios, remember:

- Security groups are stateful and usually simpler.
- NACLs are stateless and better for coarse subnet-level control or explicit deny requirements.
- If a question mentions “block a malicious CIDR at the subnet edge,” think NACL.
- If it mentions “allow only this app tier to reach the database,” think security group.

**IAM identity policies, resource policies, and trust policies**

IAM is the authorization system for AWS API actions. The most common exam distinction is whether you should use an identity-based policy, a resource-based policy, or a trust policy.

- **Identity-based policies** attach to users, groups, or roles and define what the identity can do.
- **Resource-based policies** attach to the resource and define who can access that resource.
- **Trust policies** attach to roles and define who can assume the role.

Use identity-based policies when:
- You want to grant permissions to a principal inside your account.
- You want centralized permission management for many resources.
- You need to control what a user or role can do across multiple services.

Use resource-based policies when:
- You want to grant direct access to a specific resource.
- You need cross-account access without requiring the caller to assume a role first.
- The service supports resource policies, such as S3 buckets, KMS keys, SNS topics, SQS queues, Lambda functions, and Secrets Manager secrets.

Use trust policies when:
- You want to allow a principal to assume an IAM role.
- You are setting up cross-account access through role assumption.
- You are configuring federation, AWS service access, or temporary credentials.

A useful exam shortcut:

- Identity policy answers “what can this principal do?”
- Resource policy answers “who can use this resource?”
- Trust policy answers “who can become this role?”

**IAM policy evaluation order and edge cases**

AWS authorization is not a simple single-policy check. The effective decision comes from multiple layers.

A practical evaluation model is:

1. **Implicit deny** applies by default.
2. **Explicit deny** in any applicable policy wins.
3. AWS checks identity-based policies and resource-based policies for allows.
4. AWS then applies higher-level restrictions such as permissions boundaries, SCPs, session policies, and endpoint policies where relevant.
5. The request succeeds only if the action is allowed by all required layers and not denied by any layer.

Important interactions:

- **Identity-based policy** can allow an action, but a **permissions boundary** can cap the maximum permissions of that principal.
- **SCPs** in AWS Organizations can block an action even if the identity policy allows it.
- **Session policies** can further restrict temporary credentials from `AssumeRole` or federation.
- **Endpoint policies** can restrict access through a VPC endpoint, even if IAM and resource policies allow the request.
- **Resource policies** can grant access directly, but they do not bypass explicit denies from SCPs or permissions boundaries where those apply.

A common edge case is S3 access through a VPC endpoint:

- IAM policy allows `s3:GetObject`.
- Bucket policy allows the principal.
- VPC endpoint policy allows the bucket.
- If any one of these denies or omits the required permission, the request fails.

Another edge case is KMS:

- The caller needs IAM permission to use the KMS API action.
- The KMS key policy must also allow the principal or an enabling path such as account-level delegation.
- Both layers matter in the same request flow.

**Resource policies in common exam scenarios**

Resource policies are often the best answer when the question asks for direct, resource-specific access control.

Common examples:

- **S3 bucket policies** for cross-account access, public access control, or VPC endpoint restrictions.
- **S3 access points** for application-specific access patterns and simplified policy management on shared buckets.
- **KMS key policies** for cryptographic permissions.
- **SNS topic policies** for allowing publishers or subscribers from other accounts.
- **SQS queue policies** for cross-account send/receive access.
- **Lambda resource policies** for invocation permissions from other services or accounts.

S3 access points are especially useful when multiple applications need different access patterns to the same bucket. Instead of one large bucket policy, you can create access points with scoped policies.

Example bucket policy pattern for cross-account read access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AnalyticsRole"
      },
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::example-bucket/reports/*"
    }
  ]
}
```

Use resource policies when the resource owner wants to control access directly, especially across accounts or across service boundaries.

**S3 private access with IAM, bucket policy, and endpoint policy**

For private S3 access from a VPC, three policy layers often combine:

- **IAM policy** on the role or user
- **Bucket policy** on the S3 bucket
- **VPC endpoint policy** on the gateway endpoint or interface endpoint

Example flow:

1. An EC2 instance uses an IAM role with `s3:GetObject`.
2. The bucket policy allows that role to read `arn:aws:s3:::example-bucket/private/*`.
3. The VPC endpoint policy allows access to `example-bucket`.
4. The request goes through the endpoint without using the public internet.

If the endpoint policy denies the bucket, the request fails even if IAM and bucket policy allow it.

Example endpoint policy:

```json
{
  "Statement": [
    {
      "Sid": "AllowOnlySpecificBucket",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

Exam takeaway:

- IAM policy answers whether the principal can call S3.
- Bucket policy answers whether the bucket accepts that principal.
- Endpoint policy answers whether traffic through the VPC endpoint is allowed to reach that bucket.

**KMS permission flow**

AWS Key Management Service (KMS) is a frequent exam topic because it requires both IAM and key policy alignment.

A typical request flow:

- The principal calls `kms:Encrypt`, `kms:Decrypt`, or `kms:GenerateDataKey`.
- IAM must allow the action.
- The KMS key policy must also allow the principal, or allow the account to delegate through IAM.
- If either layer blocks the request, the operation fails.

Example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowUseOfKeyForAppRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AppRole"
      },
      "Action": [
        "kms:Decrypt",
        "kms:Encrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "*"
    }
  ]
}
```

Practical interpretation:

- IAM policy alone is not enough if the key policy does not permit use.
- Key policy alone is not enough if the caller’s IAM permissions do not allow the action.
- This dual-control model is often tested for encrypted S3 objects, EBS volumes, and application secrets.

**When to choose each policy type**

| Policy type | Attach to | Best use case | Common exam clue |
|---|---|---|---|
| Identity-based policy | User, group, role | Grant permissions to a principal | “What can this role do?” |
| Resource-based policy | Resource | Direct access to a specific resource, often cross-account | “Allow another account to access this bucket/topic/queue” |
| Trust policy | Role | Allow assumption of a role | “Who can assume this role?” |
| Permissions boundary | User or role | Cap maximum permissions | “Prevent privilege escalation” |
| SCP | AWS account or OU | Restrict accounts in Organizations | “Organization-wide guardrail” |
| Session policy | Temporary session | Restrict assumed-role session | “Limit this session further” |
| Endpoint policy | VPC endpoint | Restrict private service access | “Only allow S3 through this endpoint” |

**End-to-end exam-style scenarios**

Scenario 1: EC2 instance reads from a private S3 bucket through a gateway endpoint

- EC2 instance has an IAM role with `s3:GetObject`.
- Bucket policy allows that role on `arn:aws:s3:::app-data/*`.
- Gateway endpoint policy allows `app-data`.
- Security group allows outbound HTTPS if needed for other services, but S3 gateway endpoint traffic is routed privately.
- Result: access succeeds only if all three policy layers align.

Scenario 2: Cross-account Lambda invocation

- Account A owns the Lambda function.
- Account B needs to invoke it.
- A resource-based policy on the Lambda function allows the principal from Account B.
- The caller in Account B also needs IAM permission to invoke the function.
- If the function uses a role, the trust policy is separate and controls who can assume that role, not who can invoke the function.

Scenario 3: Encrypted S3 object access

- A user has `s3:GetObject` in IAM.
- The bucket policy allows the user.
- The object is encrypted with a KMS key.
- The user also needs `kms:Decrypt` in IAM and permission in the key policy.
- If KMS is missing, S3 returns an access failure even though the bucket policy looks correct.

Scenario 4: Restricting a subnet with NACLs

- Web servers sit in a public subnet.
- Rule 100 allows inbound TCP 80.
- Rule 110 allows inbound TCP 443.
- Rule 120 allows outbound ephemeral ports `1024-65535`.
- Rule 130 denies all else.
- Return traffic works because the ephemeral range is explicitly allowed.

**Practical CLI examples**

Inspect a security group:

```bash
aws ec2 describe-security-groups \
  --group-ids sg-0123456789abcdef0
```

Add an inbound rule to a security group:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 443 \
  --cidr 0.0.0.0/0
```

View an S3 bucket policy:

```bash
aws s3api get-bucket-policy \
  --bucket example-bucket
```

Simulate IAM authorization:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::111122223333:role/AppRole \
  --action-names s3:GetObject kms:Decrypt \
  --resource-arns arn:aws:s3:::example-bucket/private/report.csv
```

Check who can assume a role:

```bash
aws iam get-role \
  --role-name CrossAccountReadRole
```

**Troubleshooting patterns**

If access fails, isolate the layer:

- Check network reachability first.
- Check security groups and NACLs for packet flow.
- Check IAM permissions for API authorization.
- Check resource policies for direct resource access.
- Check KMS key policy if encryption is involved.
- Check SCPs, permissions boundaries, session policies, and endpoint policies for hidden restrictions.

A useful mental model is:

- Network controls decide whether packets can reach the service.
- IAM and resource policies decide whether the API call is authorized.
- KMS decides whether encrypted data can be used.
- Higher-level guardrails can still override local allows.

**High-value exam reminders**

- Security groups are stateful and allow-only.
- NACLs are stateless and rule-number ordered.
- IAM identity policies grant permissions to principals.
- Resource policies grant access to resources.
- Trust policies allow role assumption.
- KMS often requires both IAM and key policy permission.
- S3 access points can simplify cross-account and application-specific access.
- VPC endpoint policies can block private S3 access even when IAM and bucket policy allow it.
- SCPs and permissions boundaries can silently cap permissions.
- For cross-account access, ask whether the question wants direct resource access or role assumption.

**Quick decision guide**

- Use a **security group** to control instance traffic.
- Use a **NACL** to enforce subnet-level allow/deny rules.
- Use an **identity policy** to grant a user or role permissions.
- Use a **resource policy** to let another principal access a specific resource.
- Use a **trust policy** to let a principal assume a role.
- Use a **permissions boundary** or **SCP** to limit maximum permissions.
- Use an **endpoint policy** to restrict private service access through VPC endpoints.
