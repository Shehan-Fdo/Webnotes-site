## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Secrets management overview**

Secrets management is the practice of storing, distributing, rotating, and auditing sensitive values such as database passwords, API keys, OAuth tokens, and certificates. In AWS, the two most common services are **AWS Secrets Manager** and **AWS Systems Manager Parameter Store**. For exam scenarios, you usually choose based on rotation needs, access patterns, size limits, cost, and operational overhead.

A good design keeps secrets out of source code, AMIs, user data, and plain-text environment files. You should also separate secret storage from secret consumption so applications can retrieve values at runtime or through managed injection mechanisms.

- **Secrets** are values that must remain confidential and often need rotation.
- **Credentials** are a common secret type, especially for RDS, Aurora, and third-party APIs.
- **Encryption at rest** is typically handled with AWS Key Management Service (AWS KMS).
- **Least privilege** matters for both secret read access and KMS decrypt permissions.

**AWS Secrets Manager**

Secrets Manager is purpose-built for secrets that need lifecycle management. It stores encrypted secret values, supports automatic rotation, and integrates with many AWS services and databases. It is usually the best exam answer when the question mentions password rotation, database credentials, or managed secret lifecycle.

Secrets Manager stores each secret as a resource with versions and staging labels. The most important labels are:

- `AWSCURRENT` for the active version used by applications
- `AWSPENDING` for the version being prepared during rotation
- `AWSPREVIOUS` for the prior version after a successful rotation

A secret can contain a single string, binary data, or structured JSON. JSON is common for database credentials because one secret can hold multiple fields such as username, password, host, and port.

```bash
aws secretsmanager create-secret \
  --name prod/app/db \
  --description "Application database credentials" \
  --secret-string '{"username":"appuser","password":"REDACTED","host":"db.example.com","port":3306}'
```

**Rotation model and prerequisites**

Automatic rotation is one of the biggest differentiators for Secrets Manager. Rotation usually uses an AWS Lambda function that knows how to update the target system and then update the secret value in Secrets Manager.

Typical rotation flow:

1. Secrets Manager creates a new version and labels it `AWSPENDING`.
2. The rotation Lambda updates the database or external system with the new credential.
3. The Lambda tests the new credential.
4. Secrets Manager promotes the version to `AWSCURRENT`.
5. The previous version becomes `AWSPREVIOUS`.

Rotation prerequisites commonly include:

- A supported secret type or a custom Lambda rotation implementation
- A Lambda function with network access to the target resource if it is in a VPC
- IAM permissions for the Lambda to read and update the secret
- Permissions for the Lambda to update the downstream system, such as an RDS database user
- A rotation schedule that matches application tolerance for credential change

> If rotation fails, the secret may remain on the previous `AWSCURRENT` version, and the new version can stay in `AWSPENDING` until the issue is fixed. This is why application-side caching and retry behavior matter.

Example rotation configuration:

```bash
aws secretsmanager rotate-secret \
  --secret-id prod/app/db \
  --rotation-lambda-arn arn:aws:lambda:us-east-1:123456789012:function:RotateDbSecret \
  --rotation-rules AutomaticallyAfterDays=30
```

**Supported rotation patterns**

Secrets Manager commonly supports managed rotation for services such as Amazon RDS and Amazon Redshift, and it also supports custom rotation for other systems through Lambda. In exam questions, if the target is a database and the requirement is automatic password rotation, Secrets Manager is usually the correct choice.

Common rotation patterns:

- **Single-user rotation**: the same database user password changes in place.
- **Alternating users rotation**: two users alternate so one remains active while the other is updated.
- **Custom rotation**: Lambda implements the steps for third-party APIs or nonstandard systems.

**Access control and resource-based policies**

Secrets Manager supports both IAM policies and **resource-based policies** on the secret itself. This is important for cross-account sharing and for tightly controlling who can read a secret.

Use a resource-based policy when you want to allow another AWS account, role, or service principal to access the secret without attaching permissions only in the caller account. This is a common exam scenario for shared services or centralized security accounts.

Example resource policy allowing a role in another account to read a secret:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::222222222222:role/AppReadSecretsRole"
      },
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "*"
    }
  ]
}
```

For cross-account access, you usually need all of the following:

- A resource policy on the secret allowing the external principal
- An IAM policy on the caller role allowing `secretsmanager:GetSecretValue`
- KMS key permissions allowing decrypt if a customer-managed key is used
- Network access if the consumer is in a private subnet and uses interface endpoints

**KMS key choices**

Secrets Manager encrypts secrets with AWS KMS. You can use either an AWS-managed key or a customer-managed key, and the choice affects control, auditing, and cross-account use.

- **AWS-managed KMS key**
  - Easier to use
  - Lower operational overhead
  - Good for standard internal use cases
  - Less control over key policy customization

- **Customer-managed KMS key**
  - Better for fine-grained access control
  - Needed when you want explicit key policy management
  - Often preferred for cross-account sharing, compliance, or separation of duties
  - Lets you control rotation, grants, and key administrators

For exam scenarios, choose a customer-managed key when the question emphasizes compliance, auditability, or cross-account access with explicit control. Choose the AWS-managed key when simplicity is the priority and no special key policy requirements are mentioned.

**Secrets Manager vs Parameter Store**

Both services can store configuration data, but they are not interchangeable in every scenario. The exam often tests whether you can identify the right service based on operational requirements.

| Capability | Secrets Manager | Parameter Store |
|---|---|---|
| Primary use | Secrets and credentials | Configuration and some secrets |
| Rotation | Built-in automatic rotation | No native secret rotation workflow |
| Versioning | Secret versions with staging labels | Parameter versions, but simpler lifecycle |
| Size limit | Larger than Parameter Store standard parameters | Smaller for standard parameters; advanced parameters support more |
| Throughput/latency | Designed for secret retrieval use cases | Often used for configuration lookups; can be cheaper for high-volume simple reads |
| Cost | Higher per secret and API usage | Lower cost, especially for standard parameters |
| Cross-account sharing | Supported with resource policies | Supported, but patterns differ and are less feature-rich for secrets |
| Best fit | Rotating credentials, database passwords, API keys | App config, feature flags, non-rotating values |

Practical exam guidance:

- Use **Secrets Manager** when rotation, secret lifecycle, or database credential management is required.
- Use **Parameter Store** when you need low-cost configuration storage or simple values that do not require rotation.
- Use **SecureString** in Parameter Store when you need encryption but not the full secrets lifecycle.

**Retrieval patterns in applications**

Applications should retrieve secrets at runtime rather than baking them into images or code. The retrieval method depends on the compute platform and how often the secret changes.

Common retrieval patterns:

- **Lambda**
  - Read the secret during initialization or on first use.
  - Cache the value in memory for the lifetime of the execution environment.
  - Refresh on a timer or when a request fails due to authentication errors.

- **EC2**
  - Use an instance profile with permission to call `GetSecretValue`.
  - Retrieve the secret from the application or a startup script.
  - Prefer short-lived in-memory caching over writing secrets to disk.

- **ECS**
  - Inject secrets into container environment variables or files using task definition secret references.
  - Use IAM task roles for access.
  - Avoid hardcoding values in the image or task definition.

- **EKS**
  - Use the Secrets Store CSI driver or external secrets controllers to mount or sync secrets.
  - Use IAM Roles for Service Accounts (IRSA) so pods can access Secrets Manager without node-wide credentials.

Example ECS task definition secret injection:

```json
{
  "containerDefinitions": [
    {
      "name": "app",
      "image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/app:latest",
      "secrets": [
        {
          "name": "DB_PASSWORD",
          "valueFrom": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/app/db-AbCdEf"
        }
      ]
    }
  ]
}
```

Example Lambda retrieval in Python:

```python
import json
import boto3

client = boto3.client("secretsmanager")

def get_db_secret():
    response = client.get_secret_value(SecretId="prod/app/db")
    return json.loads(response["SecretString"])

def handler(event, context):
    secret = get_db_secret()
    return {"user": secret["username"]}
```

Example EC2 retrieval with AWS CLI:

```bash
aws secretsmanager get-secret-value \
  --secret-id prod/app/db \
  --query SecretString \
  --output text
```

**Caching and refresh strategies**

Caching is critical because secret rotation can break applications that assume credentials never change. You want to reduce API calls, but you also need a refresh strategy that handles rotation gracefully.

Recommended patterns:

- Cache the secret in memory with a short TTL.
- Refresh proactively before the rotation window if you know the schedule.
- Retry authentication failures by reloading the secret once before failing the request.
- Avoid long-lived disk caches unless you can secure and invalidate them reliably.

A common application pattern is:

1. Load the secret on startup.
2. Keep it in memory.
3. On authentication failure, fetch the latest version.
4. If the new value works, replace the cached copy.

This pattern helps during rotation because the application can survive the brief overlap between old and new credentials.

**Replication and disaster recovery**

Secrets Manager supports multi-Region replication for secrets. This is useful for disaster recovery, regional failover, and active/passive architectures. Replication helps you keep a copy of the secret in another Region so workloads can continue if the primary Region is unavailable.

Important considerations:

- Replicated secrets are separate regional resources.
- You should plan how applications discover the correct regional endpoint.
- Rotation and replication behavior must be tested together.
- If your application fails over to another Region, it should read the local replica rather than reaching back to the primary Region.

A practical DR pattern is:

- Store the primary secret in the active Region.
- Replicate it to the standby Region.
- Deploy the application with Region-aware secret lookup.
- Test failover to confirm the replica is current and accessible.

**Auditing and monitoring**

Secrets Manager integrates with AWS CloudTrail for API auditing. You should expect to see events such as:

- `GetSecretValue`
- `DescribeSecret`
- `PutSecretValue`
- `UpdateSecret`
- `RotateSecret`
- `DeleteSecret`

CloudTrail helps you answer who accessed a secret and when. For operational monitoring, use Amazon CloudWatch alarms and logs to detect rotation failures, Lambda errors, or unusual access patterns.

Useful monitoring ideas:

- Alarm on rotation Lambda errors or throttles
- Alarm when `RotationFailed` or similar operational signals appear
- Track secret access spikes that may indicate abuse or misconfiguration
- Review CloudTrail for unexpected `GetSecretValue` calls from unfamiliar roles

Example CloudWatch alarm concept for a rotation Lambda:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name SecretsRotationLambdaErrors \
  --metric-name Errors \
  --namespace AWS/Lambda \
  --statistic Sum \
  --period 300 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --dimensions Name=FunctionName,Value=RotateDbSecret
```

**Security best practices**

Secrets management is not just about storage. It is about limiting exposure across the full lifecycle.

- Store secrets in a dedicated service, not in code repositories.
- Use IAM roles instead of long-term access keys where possible.
- Grant read access only to the workloads that need the secret.
- Use KMS key policies and grants carefully for cross-account scenarios.
- Rotate secrets on a schedule that matches risk and operational tolerance.
- Prefer runtime retrieval and in-memory caching over static embedding.
- Separate production and nonproduction secrets to reduce blast radius.

**Common exam scenarios**

When you see a question, map the requirement to the service and feature set:

- If the requirement is automatic database password rotation, choose **Secrets Manager**.
- If the requirement is low-cost storage for app configuration, choose **Parameter Store**.
- If another account must read the secret, think about **resource-based policies** and KMS permissions.
- If the workload is in ECS or EKS, think about managed secret injection or pod/task identity.
- If the application must survive rotation, think about **caching and refresh** logic.
- If the architecture spans Regions, think about **replication** and failover readiness.

**Quick reference**

- **Secrets Manager**: best for secrets, rotation, and lifecycle management
- **Parameter Store**: best for configuration and lower-cost storage
- **AWS-managed KMS key**: simplest encryption option
- **Customer-managed KMS key**: best for control, compliance, and advanced access patterns
- **Resource-based policy**: required for many cross-account secret sharing scenarios
- **Lambda rotation function**: performs the update/test/promotion workflow
- **CloudTrail**: audit secret access and management actions
- **CloudWatch**: monitor rotation failures and access anomalies
