## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Overview**

AWS CloudTrail and AWS Config are complementary governance services. CloudTrail answers the question, “Who did what, when, and from where?” AWS Config answers, “What changed, what is the current configuration, and is it compliant?” For exam scenarios, the key distinction is that CloudTrail records API activity, while AWS Config records resource configuration state over time.

Use CloudTrail for audit, investigation, and security monitoring. Use AWS Config for configuration history, compliance evaluation, and drift detection. In practice, you often use both together: CloudTrail identifies the actor and request, while AWS Config shows the resulting resource state.

**CloudTrail fundamentals**

CloudTrail captures AWS API activity across your account and can aggregate events from multiple Regions into a single trail. It is the primary service for governance and forensic analysis of AWS actions.

- **Management events** record control-plane actions such as creating, modifying, or deleting resources.
- **Data events** record high-volume resource activity such as S3 object-level actions or Lambda function invocations.
- **Event history** in the CloudTrail console provides a default 90-day view of recent management events without requiring a trail.
- **Trails** deliver long-term event logs to Amazon S3, and can optionally publish to Amazon CloudWatch Logs for alerting and analysis.
- **CloudTrail Lake** is separate from trails and provides a managed data lake for querying and retention.

CloudTrail logs include details such as the caller identity, source IP address, event time, event name, request parameters, and response elements. This makes it useful for incident response and compliance evidence.

**CloudTrail event types and when to use them**

Management events are enabled by default for many AWS services and are the most common exam focus. Data events are not enabled by default because they can generate large volumes and cost more.

| Event type | Typical examples | Default behavior | Why it matters |
|---|---|---:|---|
| Management events | `CreateBucket`, `RunInstances`, `AuthorizeSecurityGroupIngress` | Enabled for trails by default | Tracks control-plane changes |
| Data events | `GetObject`, `PutObject`, Lambda invoke | Disabled unless selected | Tracks resource-level access |
| Insights events | Unusual API activity patterns | Optional | Helps detect anomalies |

A common exam trap is confusing configuration state with API activity. CloudTrail does not store the current configuration of a resource; it stores the request and response metadata for the API call.

**CloudTrail trail design**

A trail defines where CloudTrail delivers logs and how broadly it collects events. For most production environments, you create at least one organization-wide or multi-Region trail.

- **Multi-Region trails** capture events from all enabled Regions in the account.
- **Organization trails** capture events across all accounts in AWS Organizations.
- **S3 delivery** is the primary storage path for trail logs.
- **CloudWatch Logs integration** is optional and requires an IAM role that allows CloudTrail to publish log events.
- **Log file validation** helps detect tampering by using digest files and integrity checks.

A practical trail setup often looks like this:

```bash
aws cloudtrail create-trail \
  --name org-audit-trail \
  --s3-bucket-name my-cloudtrail-logs-bucket \
  --is-multi-region-trail

aws cloudtrail start-logging \
  --name org-audit-trail
```

If you want CloudTrail to publish to CloudWatch Logs, you must also configure the log group and IAM role association. That path is useful when you want metric filters and alarms for suspicious activity.

**CloudTrail and CloudWatch Logs for alerting**

CloudTrail itself is not an alerting service. To alert on suspicious actions, you typically send trail logs to CloudWatch Logs and create metric filters and alarms.

A root login detection pattern is usually based on the `ConsoleLogin` event and `userIdentity.type = "Root"`. Exact filter syntax must match the CloudTrail JSON structure and CloudWatch Logs filter rules.

```json
{
  "eventName": "ConsoleLogin",
  "userIdentity": {
    "type": "Root"
  }
}
```

A practical workflow is:

1. Deliver CloudTrail logs to CloudWatch Logs.
2. Create a metric filter for the suspicious event pattern.
3. Create a CloudWatch alarm on the metric.
4. Notify an SNS topic or trigger an automated response.

> CloudTrail logs are not a substitute for AWS Config. CloudTrail tells you that a change happened; AWS Config tells you what the resource looks like after the change.

**CloudTrail CLI examples**

You can inspect trails and recent events with the AWS CLI.

```bash
aws cloudtrail describe-trails
aws cloudtrail get-trail-status --name org-audit-trail
aws cloudtrail lookup-events --lookup-attributes AttributeKey=EventName,AttributeValue=DeleteBucket
```

For object-level activity in S3, you must explicitly enable data events for the relevant bucket or prefix. This is a common exam point because data events are not automatically captured.

**AWS Config fundamentals**

AWS Config records the configuration of supported AWS resources and evaluates those configurations against desired settings. It is the service you use for compliance, drift detection, and configuration history.

- **Configuration recorder** captures resource configuration changes.
- **Delivery channel** delivers configuration snapshots and history to an S3 bucket and can optionally notify via SNS.
- **Configuration history** shows the timeline of changes for a resource.
- **Configuration snapshots** are point-in-time captures of the current state of recorded resources.
- **Config rules** evaluate resources against compliance logic.
- **Remediations** can automatically fix noncompliant resources, often using Systems Manager Automation.

AWS Config is not an API audit log. It does not record the full request parameters of an API call. Instead, it records the resulting configuration state and how that state changes over time.

**AWS Config setup and recording flow**

AWS Config requires both a recorder and a delivery channel. The recorder must be started after the delivery channel exists, otherwise recording cannot function properly.

A typical setup flow is:

1. Create an S3 bucket for AWS Config delivery.
2. Create an optional SNS topic if you want notifications.
3. Create the configuration recorder.
4. Create the delivery channel.
5. Start the configuration recorder.

Example CLI flow:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder '{
    "name": "default",
    "roleARN": "arn:aws:iam::123456789012:role/AWSConfigRole",
    "recordingGroup": {
      "allSupported": true,
      "includeGlobalResourceTypes": true
    }
  }'

aws configservice put-delivery-channel \
  --delivery-channel '{
    "name": "default",
    "s3BucketName": "my-config-bucket",
    "snsTopicARN": "arn:aws:sns:us-east-1:123456789012:config-notifications"
  }'

aws configservice start-configuration-recorder \
  --configuration-recorder-name default
```

The recorder role needs permission to describe resources and deliver configuration data. The S3 bucket policy must also allow AWS Config to write objects.

**Configuration history versus snapshots**

These two terms are easy to mix up, but they serve different purposes.

- **Configuration history** is a chronological record of changes for a specific resource.
- **Configuration snapshots** are periodic or on-demand point-in-time captures of all recorded resources.

Use history when you need to answer, “What changed on this resource and when?” Use snapshots when you need a broader inventory-style view of the environment at a specific time.

**AWS Config rules and compliance**

Config rules evaluate whether resources comply with your policies. Rules can be managed by AWS or custom-built.

- **Managed rules** are prebuilt by AWS, such as checks for encryption or public access.
- **Custom rules** use AWS Lambda to evaluate resource configurations.
- **Periodic rules** run on a schedule.
- **Configuration change rules** run when a relevant resource changes.

A custom rule does not simply return a JSON object with a compliance status. The Lambda function receives an evaluation event and must call the AWS Config API, typically `PutEvaluations`, to report the result.

Example Lambda pseudocode for a custom rule:

```python
import json
import boto3

config = boto3.client("config")

def lambda_handler(event, context):
    invoking_event = json.loads(event["invokingEvent"])
    configuration_item = invoking_event.get("configurationItem", {})
    resource_id = configuration_item.get("resourceId")

    compliant = configuration_item.get("configuration", {}).get("encrypted", False)

    config.put_evaluations(
        Evaluations=[
            {
                "ComplianceResourceType": configuration_item.get("resourceType"),
                "ComplianceResourceId": resource_id,
                "ComplianceType": "COMPLIANT" if compliant else "NON_COMPLIANT",
                "OrderingTimestamp": configuration_item.get("configurationItemCaptureTime")
            }
        ],
        ResultToken=event["resultToken"]
    )

    return {"statusCode": 200}
```

A practical remediation pattern is to attach a Systems Manager Automation document to a noncompliant rule. For example, if an S3 bucket is found to be public, the remediation can remove the public ACL or block public access settings.

**AWS Config remediation example**

A common governance workflow is:

1. AWS Config detects a noncompliant resource.
2. The rule marks the resource `NON_COMPLIANT`.
3. A remediation action invokes Systems Manager Automation.
4. The automation document applies the fix.
5. AWS Config reevaluates the resource.

This is useful for guardrails because it reduces manual response time and helps enforce baseline security.

**AWS Config aggregators**

Aggregators provide centralized visibility across multiple accounts and Regions. They are especially useful in AWS Organizations environments where you need a single compliance view.

- **Cross-account visibility** helps security and governance teams review all accounts from one place.
- **Cross-region visibility** helps you compare compliance across Regions.
- **Aggregated compliance** is useful for dashboards and reporting.

For exam questions, remember that aggregators help with reporting and visibility, not with recording itself.

**Practical AWS Config CLI examples**

You can inspect compliance and resource history with the CLI.

```bash
aws configservice describe-compliance-by-config-rule
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::SecurityGroup \
  --resource-id sg-0123456789abcdef0
aws configservice describe-configuration-recorders
aws configservice describe-delivery-channels
```

If you need a compliance summary, use the appropriate describe or aggregate APIs depending on whether you are querying a single account or an aggregator. In practice, exam questions often care more about the concept than the exact command.

**CloudTrail versus AWS Config**

These services are often tested together because they solve adjacent but different problems.

| Capability | CloudTrail | AWS Config |
|---|---|---|
| Records API activity | Yes | No |
| Records resource configuration state | No | Yes |
| Shows who made a change | Yes | Indirectly, via change context |
| Shows current and historical configuration | No | Yes |
| Supports compliance rules | Limited | Yes |
| Supports forensic investigation | Yes | Limited |
| Default retention in console | 90 days event history | Depends on delivery and retention settings |

A useful memory aid is:

- CloudTrail = activity log
- AWS Config = configuration timeline

> CloudTrail does not tell you the full before-and-after resource state. AWS Config does not tell you the exact API request payload. Use both when you need a complete audit story.

**Common exam scenarios**

These services appear in questions about security, governance, and troubleshooting.

- You need to know who deleted an IAM policy or changed a security group rule.
  - Use **CloudTrail**.
- You need to know whether an S3 bucket was ever public and when it changed.
  - Use **AWS Config**.
- You need to alert on root account sign-in.
  - Use **CloudTrail** with CloudWatch Logs metric filters and alarms.
- You need to enforce encryption on EBS volumes or S3 buckets.
  - Use **AWS Config** managed rules and remediation.
- You need a centralized compliance dashboard across accounts.
  - Use **AWS Config aggregators**.
- You need long-term audit retention beyond the console’s 90-day event history.
  - Use a **trail** with S3 or CloudTrail Lake.

**Troubleshooting guidance**

When CloudTrail or AWS Config is not working as expected, check the service-specific prerequisites first.

- **CloudTrail not logging**
  - Confirm the trail is started.
  - Confirm the S3 bucket policy allows delivery.
  - Confirm the trail includes the correct Regions.
  - Confirm data events are explicitly enabled when needed.
- **CloudTrail to CloudWatch Logs not working**
  - Confirm the IAM role trust policy allows CloudTrail to assume the role.
  - Confirm the log group exists and permissions are correct.
- **AWS Config not recording**
  - Confirm the configuration recorder exists and is started.
  - Confirm the delivery channel exists before starting the recorder.
  - Confirm the recorder role has the required permissions.
  - Confirm the S3 bucket policy allows AWS Config to write snapshots and history.
- **AWS Config rule not evaluating**
  - Confirm the rule is attached to the correct resource types.
  - Confirm the Lambda function returns evaluations with `PutEvaluations`.
  - Confirm the result token and event format are handled correctly.

**Quick comparison checklist**

- Use **CloudTrail** for API auditing, investigation, and alerting.
- Use **AWS Config** for configuration drift, compliance, and resource history.
- Remember that CloudTrail has a 90-day console event history by default.
- Remember that AWS Config needs a recorder, delivery channel, and started recorder.
- Remember that data events in CloudTrail require explicit selection.
- Remember that AWS Config custom rules report results through the AWS Config API.

**Exam-ready summary**

If a question asks about an action, identity, or API call, think CloudTrail. If it asks about resource state, compliance, or drift, think AWS Config. If it asks for both, combine them: CloudTrail for the event and AWS Config for the resulting configuration.
