## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Encryption strategy overview**

AWS encryption questions usually test three ideas: where data is protected, who controls the keys, and whether encryption is automatic or must be enabled. For the exam, separate **encryption at rest** from **encryption in transit** and from **client-side encryption**.

- **At rest** means data is encrypted while stored on disk or in a managed service.
- **In transit** means data is encrypted while moving between client and service, usually with TLS (Transport Layer Security).
- **Client-side encryption** means you encrypt before sending data to AWS, so AWS stores ciphertext.

A useful mental model is:

- S3 often focuses on object encryption and access control.
- EBS focuses on block volume encryption and snapshot behavior.
- RDS focuses on database storage, backups, and replica/copy constraints.
- EFS focuses on file system encryption and mount-time TLS.

**Encryption options you should distinguish**

| Option | Where encryption happens | Key control | Common exam use |
|---|---|---|---|
| **SSE-S3** | Server-side in S3 | AWS-managed keys | Simple default protection for objects |
| **SSE-KMS** | Server-side in S3 | AWS Key Management Service (KMS) key | Compliance, auditability, key rotation, access control |
| **Client-side encryption** | Before upload | You control encryption process and keys | Strong control, sensitive workloads, app-managed crypto |
| **EBS encryption** | At the volume and snapshot layer | KMS key | Encrypt root/data volumes and snapshots |
| **RDS encryption** | Storage, backups, snapshots, replicas where supported | KMS key | Database compliance and backup protection |
| **EFS encryption** | File system storage; TLS for transit | KMS key for at-rest | Shared file systems with managed encryption |

**Amazon S3 encryption**

S3 supports multiple encryption patterns, and exam questions often ask you to choose between simplicity, compliance, and key control. The main server-side options are **SSE-S3** and **SSE-KMS**. With server-side encryption, S3 encrypts the object after upload and decrypts it on download for authorized users.

- **SSE-S3** uses AWS-managed keys and is the simplest server-side option.
- **SSE-KMS** uses KMS keys and gives you audit logs, key policies, and tighter access control.
- **Client-side encryption** is the right choice when your application must encrypt data before it reaches S3.

S3 bucket default encryption helps ensure new objects are encrypted automatically, but it does not retroactively re-encrypt existing objects. It also does not replace access control, because encryption does not stop unauthorized reads if the principal already has permission to access the object and key.

> Bucket default encryption applies to new uploads, including multipart uploads and copy operations that create new objects, but it does not change objects already stored in the bucket.

**S3 encryption behavior and exam nuances**

- **Multipart uploads** inherit the encryption settings applied to the upload request or bucket default encryption.
- **Copy operations** can create a new encrypted object, but the destination encryption depends on the copy request and bucket defaults.
- **Existing objects** remain as they were until you explicitly rewrite or copy them.
- **Re-encryption workflows** usually involve copying the object onto itself or to a new key with the desired encryption settings.

**S3 bucket policy enforcement example**

If you want to require SSE-KMS for uploads, you usually enforce both the encryption algorithm and the KMS key usage. A common pattern is to deny uploads unless the request uses `aws:kms` and specifies the approved KMS key.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyIncorrectEncryptionHeader",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-secure-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    },
    {
      "Sid": "DenyIncorrectKmsKey",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-secure-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption-aws-kms-key-id": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555"
        }
      }
    }
  ]
}
```

This pattern works because the first statement blocks uploads that do not request KMS encryption, and the second statement blocks uploads that use the wrong key. In practice, you may also combine this with a TLS-only policy so uploads must use HTTPS.

```json
{
  "Sid": "DenyInsecureTransport",
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": [
    "arn:aws:s3:::my-secure-bucket",
    "arn:aws:s3:::my-secure-bucket/*"
  ],
  "Condition": {
    "Bool": {
      "aws:SecureTransport": "false"
    }
  }
}
```

**S3 practical CLI examples**

```bash
# Upload with SSE-S3
aws s3 cp report.csv s3://my-bucket/ --sse AES256

# Upload with SSE-KMS using a specific key
aws s3 cp report.csv s3://my-bucket/ \
  --sse aws:kms \
  --sse-kms-key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555

# Set default bucket encryption to SSE-KMS
aws s3api put-bucket-encryption \
  --bucket my-bucket \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "aws:kms",
        "KMSMasterKeyID": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555"
      }
    }]
  }'
```

**S3 client-side encryption**

Client-side encryption is useful when the application must control the encryption process before data leaves the host. This is different from SSE because AWS never sees plaintext.

- Use it when regulatory or architectural requirements demand app-level control.
- Use it when you need to keep encryption logic outside the storage service.
- Expect more application complexity, including key handling and decryption logic.

Common approaches include:

- AWS Encryption SDK
- Application-managed libraries
- Envelope encryption with a data key protected by a master key

**When to choose S3 encryption options**

- Choose **SSE-S3** when you want simple, automatic encryption with minimal operational overhead.
- Choose **SSE-KMS** when you need audit trails, key policies, or tighter compliance controls.
- Choose **client-side encryption** when the application must own encryption before upload.
- Enforce **HTTPS/TLS** when the question mentions secure transport or public endpoints.

**Amazon EBS encryption**

Amazon Elastic Block Store (EBS) encryption protects block volumes and the snapshots created from them. Encryption is handled with KMS, and the data is encrypted at rest on the volume, in snapshots, and in the data moving between the volume and the instance.

- The default AWS managed key for EBS is `alias/aws/ebs`.
- You can also use a customer managed KMS key for tighter control.
- Encryption is transparent to applications and the operating system.

When you create an encrypted volume from an encrypted snapshot, the new volume remains encrypted. If you copy an unencrypted snapshot, you can choose to encrypt the copy during the copy operation. This is a common migration path.

**EBS snapshot and sharing behavior**

- **Encrypted snapshots** can be shared only with the right KMS permissions and key access.
- **Cross-account use** of encrypted snapshots requires careful KMS key policy and grants.
- **Unencrypted snapshots** can be copied and encrypted during the copy.
- **Encrypted snapshots** cannot be directly shared in the same way as unencrypted snapshots without KMS coordination.

A typical exam scenario is: you have an unencrypted volume in one account, and you need an encrypted copy in another account. The usual approach is to create a snapshot, copy it with encryption enabled, and then restore a new encrypted volume from the copied snapshot.

**EBS copy and encrypt example**

```bash
# Create a snapshot from an existing volume
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-migration snapshot"

# Copy the snapshot and encrypt it with a KMS key
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --encrypted \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555 \
  --description "Encrypted migration copy"
```

The copy command returns a new snapshot ID. Use that returned ID when you create the new volume.

```bash
# Restore a volume from the copied encrypted snapshot
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0fedcba9876543210 \
  --volume-type gp3
```

**EBS encryption defaults and key management**

You can enable encryption by default for new EBS volumes in an account or Region. This is useful when you want to prevent accidental creation of unencrypted volumes.

- Default encryption applies to new volumes created after the setting is enabled.
- It does not automatically encrypt existing unencrypted volumes.
- It is often paired with IAM controls or AWS Config rules to detect drift.

**EBS practical considerations**

- Use encryption for root and data volumes when the workload stores sensitive data.
- Use customer managed keys when you need explicit key rotation control, separation of duties, or compliance evidence.
- Remember that encryption does not replace instance hardening, patching, or least privilege.

**Amazon RDS encryption**

Amazon Relational Database Service (RDS) encryption protects database storage, automated backups, read replicas where supported, and snapshots. RDS uses KMS keys for encryption at rest. The exact behavior depends on engine, deployment model, and replica type.

- You must enable encryption when you create the DB instance or cluster.
- You cannot convert an existing unencrypted RDS instance into an encrypted one in place.
- The common migration path is snapshot copy and restore into a new encrypted instance.

For exam purposes, remember that encrypted database deployments often require a **customer managed KMS key** when the organization needs cross-account sharing, explicit key policy control, or compliance separation.

**RDS replica and snapshot behavior**

- Encrypted snapshots can be copied and restored, subject to KMS permissions.
- Cross-Region snapshot copy is a common way to move encrypted databases.
- Read replica support for encrypted sources depends on the engine and deployment type.
- Some engines and versions have restrictions on encrypted cross-Region replicas, so read the scenario carefully.

A safe exam rule is: if the question asks for encrypted migration, backup, or regional move, think **snapshot copy + restore** first. If it asks for low-latency replication, verify whether the engine and replica type support encrypted replicas in that topology.

**RDS SSL/TLS enforcement example**

RDS encryption at rest is separate from SSL/TLS in transit. If the question asks for secure database connections, you need client-side SSL/TLS settings and often security group restrictions.

```sql
-- Example: require SSL for PostgreSQL connections
ALTER SYSTEM SET rds.force_ssl = 1;
SELECT pg_reload_conf();
```

For MySQL-compatible engines, the exact parameter name and enforcement method depend on the engine and version. In practice, you often combine:

- A parameter group setting that requires SSL
- Security groups that allow only trusted sources
- Application connection strings that request TLS

```bash
# Example client connection using TLS verification
psql "host=mydb.abc123.us-east-1.rds.amazonaws.com port=5432 dbname=app user=admin sslmode=require"
```

**RDS practical migration pattern**

1. Take a snapshot of the source DB instance.
2. Copy the snapshot and encrypt it if needed.
3. Restore a new DB instance from the encrypted snapshot.
4. Update the application endpoint to point to the new instance.
5. Validate parameter groups, option groups, and security groups.

**RDS exam triggers**

- Choose **snapshot copy and restore** when moving from unencrypted to encrypted.
- Choose **customer managed KMS keys** when the scenario mentions compliance, audit, or cross-account control.
- Choose **SSL/TLS** when the question mentions secure database connections, not storage encryption.
- Be careful with replica wording; not every engine or topology supports every encrypted replica pattern.

**Amazon EFS encryption**

Amazon Elastic File System (EFS) supports encryption at rest and encryption in transit, but these are separate features. Encryption at rest is configured when the file system is created. In general, you cannot switch an existing unencrypted file system to encrypted in place, so you create a new encrypted file system and migrate data.

- EFS at rest uses KMS keys.
- EFS in transit uses TLS.
- Plain NFS mounts do not automatically provide TLS.

For in-transit encryption, you typically use the EFS mount helper or a compatible client configuration. This matters for EC2 instances, containers, and other Linux clients that mount EFS.

**EFS mount example with TLS**

```bash
# Mount EFS with TLS using the mount helper
sudo mount -t efs -o tls fs-1234567890abcdef0:/ /mnt/efs
```

For Amazon ECS or EKS workloads, you usually enable TLS in the mount configuration or storage driver settings rather than relying on the application itself.

**EFS practical considerations**

- Use EFS when multiple instances need shared POSIX file storage.
- Use encryption at rest for sensitive shared files.
- Use TLS for in-transit protection, especially across VPC paths or shared workloads.
- If the question mentions “encrypted EFS,” check whether it means storage encryption, mount encryption, or both.

**Service-by-service selection matrix**

| Service | Choose this when | Avoid this when | Exam clue |
|---|---|---|---|
| **S3** | Object storage, simple default encryption, compliance with KMS | You need app-side ciphertext before upload | “Bucket policy,” “object upload,” “HTTPS,” “default encryption” |
| **EBS** | EC2 block storage, OS disks, database volumes on EC2 | You need shared file access across instances | “Volume,” “snapshot,” “root disk,” “encrypt by default” |
| **RDS** | Managed relational database encryption and backup protection | You need direct control over the database host filesystem | “Snapshot restore,” “read replica,” “KMS key,” “SSL” |
| **EFS** | Shared file system for multiple Linux instances or containers | You need block storage or database storage | “Mount helper,” “TLS,” “shared POSIX,” “NFS” |

**Common exam scenarios and anti-patterns**

- If a question says “encrypt data before AWS sees it,” think **client-side encryption**.
- If it says “encrypt S3 objects automatically,” think **bucket default encryption** or **SSE-KMS**.
- If it says “encrypt an existing EBS volume,” think **snapshot, copy if needed, and restore**.
- If it says “move an unencrypted RDS database to encrypted,” think **snapshot copy and restore**.
- If it says “secure EFS traffic,” think **TLS on the mount**, not just encryption at rest.

**Troubleshooting checklist**

- Verify the KMS key policy allows the service and the caller.
- Confirm the IAM principal has permission to use the key.
- Check whether the service supports the requested encrypted replica or copy pattern.
- Confirm whether the issue is at rest or in transit.
- For S3, verify bucket policy, object ACLs, and HTTPS enforcement.
- For EBS and RDS, verify snapshot ownership and KMS permissions.
- For EFS, verify the mount helper, TLS option, and network path.

**Quick memory cues**

- **S3**: objects, bucket policy, SSE-S3, SSE-KMS, client-side encryption.
- **EBS**: volumes, snapshots, `alias/aws/ebs`, copy and encrypt.
- **RDS**: database storage, snapshot restore, replica constraints, SSL/TLS.
- **EFS**: shared file system, encryption at rest at creation, TLS on mount.
