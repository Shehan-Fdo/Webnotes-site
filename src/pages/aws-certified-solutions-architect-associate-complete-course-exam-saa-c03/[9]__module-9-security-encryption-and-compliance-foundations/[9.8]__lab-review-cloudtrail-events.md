## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**CloudTrail overview**

<abbr title="AWS CloudTrail">CloudTrail</abbr> records AWS API activity so you can answer who did what, when, from where, and with which credentials. For exam purposes, separate the three main uses:

- **Event history** for recent management events in a Region.
- **Trails** for delivering events to Amazon S3, optionally with CloudWatch Logs integration.
- **CloudTrail Lake** for querying event data with SQL-like statements.

CloudTrail is most useful when you need an audit trail for control-plane actions such as `CreateBucket`, `RunInstances`, `PutRolePolicy`, or `DeleteSecurityGroup`. It can also capture data-plane activity such as `GetObject` or `Invoke` when you explicitly configure data event logging.

> `lookup-events` queries CloudTrail event history for management events only. It does not search S3 trail logs or CloudTrail Lake.

**What you review in the lab**

In a typical review workflow, you create or trigger an AWS action, then inspect the resulting event in CloudTrail. The goal is to connect the console action or CLI command to the recorded API call and understand the event structure.

- **Event name** identifies the API action, such as `StartInstances` or `PutBucketPolicy`.
- **Event source** shows the service namespace, such as `ec2.amazonaws.com` or `s3.amazonaws.com`.
- **User identity** shows who or what made the call, including IAM user, assumed role, federated user, or AWS service.
- **Source IP address** helps you trace where the request originated.
- **Read-only flag** tells you whether the API call was read-only, not whether it is management or data plane.
- **Resources** lists the affected resource ARNs when CloudTrail records them.

A useful mental model is that CloudTrail answers the question: “What API call happened?” It does not replace service-specific logs such as VPC Flow Logs, ALB access logs, or application logs.

**Event history and timing**

CloudTrail event history is available in the console for recent management events, but it is not instantaneous. In practice, expect a short delay, often up to about 15 minutes after the API call. That delay matters during troubleshooting because a missing event may simply be delayed rather than absent.

Event history is Region-specific. If you perform an action in `us-east-1`, you must inspect the same Region’s event history unless you are using a trail or centralized logging strategy that aggregates elsewhere.

**Common event fields**

CloudTrail events are JSON documents. The exact shape varies by service and event type, but the following fields are common and worth recognizing:

```json
{
  "eventVersion": "1.08",
  "userIdentity": {
    "type": "IAMUser",
    "principalId": "AIDAEXAMPLE",
    "arn": "arn:aws:iam::123456789012:user/Alice",
    "accountId": "123456789012",
    "userName": "Alice"
  },
  "eventTime": "2026-04-26T12:34:56Z",
  "eventSource": "ec2.amazonaws.com",
  "eventName": "StartInstances",
  "awsRegion": "us-east-1",
  "sourceIPAddress": "203.0.113.10",
  "userAgent": "aws-cli/2.15.0",
  "requestParameters": {
    "instancesSet": {
      "items": [
        { "instanceId": "i-0123456789abcdef0" }
      ]
    }
  },
  "responseElements": {
    "instancesSet": {
      "items": [
        {
          "instanceId": "i-0123456789abcdef0",
          "currentState": { "name": "pending" }
        }
      ]
    }
  },
  "readOnly": false,
  "eventType": "AwsApiCall",
  "recipientAccountId": "123456789012"
}
```

Key fields to interpret:

- **`eventTime`** is the timestamp of the API call.
- **`eventSource`** identifies the AWS service endpoint.
- **`eventName`** is the operation name.
- **`userIdentity`** explains the caller context.
- **`requestParameters`** shows what was requested.
- **`responseElements`** shows what the service returned, when present.
- **`readOnly`** indicates whether the API call is read-only.
- **`eventType`** often appears as `AwsApiCall` for normal service API activity.

**Management events vs data events**

CloudTrail distinguishes between event categories:

- **Management events**
  - Record control-plane operations that create, modify, or delete resources.
  - Include actions like `CreateUser`, `AttachRolePolicy`, `RunInstances`, and `PutBucketPolicy`.
  - Are available in event history for recent activity.
- **Data events**
  - Record high-volume resource operations on or within a resource.
  - Include S3 object-level actions such as `GetObject` and `PutObject`.
  - Include Lambda function invocation events when configured.
  - Are not shown in the basic event history view in the same way as management events.

A common exam trap is to assume `readOnly: true` means “management event” or `readOnly: false` means “data event.” That is not reliable. `readOnly` only tells you whether the API call is read-only.

**Reviewing events with the AWS CLI**

The AWS CLI is often the fastest way to inspect recent management events.

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=PutRolePolicy \
  --max-results 10
```

Useful lookup attributes include:

- `EventName`
- `EventSource`
- `Username`
- `ResourceName`
- `AccessKeyId`

You can also filter by time window:

```bash
aws cloudtrail lookup-events \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T23:59:59Z \
  --max-results 50
```

For readable output, combine the command with `jq`:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=Username,AttributeValue=Alice \
  --max-results 5 | jq '.Events[] | {EventTime, EventName, Username, Resources}'
```

Practical interpretation tips:

- Use `EventName` when you know the action you are hunting.
- Use `Username` when you know the human or role involved.
- Use `ResourceName` when you know the target resource identifier.
- Use `AccessKeyId` when you are tracing a specific credential.

**Console workflow**

A typical console review process is:

1. Trigger an action, such as creating a security group rule or starting an EC2 instance.
2. Wait for CloudTrail event history to update.
3. Open the CloudTrail console and inspect the matching event.
4. Compare the event fields with the action you performed.
5. Expand the event JSON to inspect request and response details.

When you review the event, focus on the relationship between the API call and the resource change. For example, a console click to “Start instance” usually maps to an EC2 `StartInstances` API call, not a generic “console action.”

**Interpreting identity and authentication**

CloudTrail is especially valuable for understanding caller identity.

- **IAM user** events show the user ARN directly.
- **Assumed role** events show the role session and often include the session name.
- **AWS service** events may show a service principal when one service acts on behalf of another.
- **Federated access** events may show the identity provider context.

This matters when you troubleshoot unexpected changes. If an action came from an assumed role, the role may be correct while the session origin is the real clue. If an AWS service performed the action, the event may reflect service automation rather than a human operator.

**S3 data event configuration**

To review S3 object-level activity, you must enable data events on a trail or in CloudTrail Lake. For S3, the resource ARN must match the documented object-level pattern. A bucket-level data event selector is not the same as an object-level selector.

Example trail configuration for S3 object data events:

```bash
aws cloudtrail put-event-selectors \
  --trail-name my-trail \
  --event-selectors '[
    {
      "ReadWriteType": "All",
      "IncludeManagementEvents": true,
      "DataResources": [
        {
          "Type": "AWS::S3::Object",
          "Values": [
            "arn:aws:s3:::my-example-bucket/"
          ]
        }
      ]
    }
  ]'
```

Important details:

- **`AWS::S3::Object`** is the data resource type for object-level logging.
- The bucket ARN for object data events uses the trailing slash form in the selector value.
- **`ReadWriteType`** controls whether you log `ReadOnly`, `WriteOnly`, or `All` data events.
- `IncludeManagementEvents` applies to management events and is separate from data event logging.

> Be careful not to assume any bucket ARN string is acceptable. CloudTrail expects the exact resource pattern for the data resource type you configure.

For Lambda data events, the pattern is different and depends on the service’s supported data resource type. Always verify the documented ARN format before enabling logging.

**Example event patterns**

A management event for IAM policy changes often looks like this:

```json
{
  "eventSource": "iam.amazonaws.com",
  "eventName": "PutRolePolicy",
  "readOnly": false,
  "userIdentity": {
    "type": "AssumedRole"
  }
}
```

An S3 object read event, when data events are enabled, may look like this:

```json
{
  "eventSource": "s3.amazonaws.com",
  "eventName": "GetObject",
  "readOnly": true,
  "eventCategory": "Data",
  "resources": [
    {
      "type": "AWS::S3::Object",
      "ARN": "arn:aws:s3:::my-example-bucket/report.csv"
    }
  ]
}
```

Use these patterns as clues, not as rigid rules. Different services emit different fields, and some events omit fields such as `responseElements` or `requestParameters` when the service does not return them.

**Troubleshooting with CloudTrail**

CloudTrail helps you answer several common troubleshooting questions:

- **Who changed this resource?**
  - Check `userIdentity`, `eventName`, and `eventTime`.
- **What exact API call was made?**
  - Check `eventSource` and `eventName`.
- **Was the call successful?**
  - Inspect `errorCode` and `errorMessage` if present.
- **What resource was targeted?**
  - Check `requestParameters` and `resources`.
- **Did an automation tool make the change?**
  - Check `userAgent`, `userIdentity`, and session context.

If a resource changed unexpectedly, search for the relevant `eventName` around the time of the change, then work backward from the caller identity. This is often faster than searching by resource first.

**Exam-relevant distinctions**

CloudTrail is frequently tested alongside other AWS observability services.

| Service | Primary purpose | Typical question it answers |
|---|---|---|
| CloudTrail | API auditing | Who made the AWS API call? |
| CloudWatch Logs | Log storage and analysis | What did the application or system log say? |
| CloudWatch Metrics | Numeric monitoring | Did a threshold get breached? |
| AWS Config | Configuration history and compliance | What did the resource configuration look like over time? |

Use CloudTrail when the question is about API activity and identity. Use AWS Config when the question is about resource configuration drift. Use CloudWatch when the question is about operational metrics or application logs.

**Practical lab checklist**

1. Trigger a known AWS action, such as `StartInstances`, `CreateBucket`, or `PutRolePolicy`.
2. Wait for CloudTrail event history to update in the same Region.
3. Search by `EventName`, `Username`, or `ResourceName`.
4. Open the event and identify `eventSource`, `eventName`, `userIdentity`, and `sourceIPAddress`.
5. Confirm whether the event is management or data plane based on the event category and configuration.
6. If needed, use the CLI to narrow the search and export the event JSON for deeper inspection.

**Common mistakes**

- Assuming event history is immediate.
- Searching the wrong Region.
- Treating `lookup-events` as a search across trail-delivered S3 logs.
- Assuming `readOnly` tells you whether an event is management or data.
- Using an incorrect S3 data resource ARN pattern.
- Expecting object-level S3 activity to appear without enabling data event logging.

**Quick reference**

<dl>
<dt><strong>Event history</strong></dt>
<dd>Recent management events visible in the CloudTrail console for a Region.</dd>
<dt><strong>Trail</strong></dt>
<dd>A configuration that delivers CloudTrail events to S3 and optionally CloudWatch Logs.</dd>
<dt><strong>Data event</strong></dt>
<dd>High-volume resource activity such as S3 object access or Lambda invocation, enabled explicitly.</dd>
<dt><strong>Management event</strong></dt>
<dd>Control-plane API activity that creates, changes, or deletes AWS resources.</dd>
</dl>

**Takeaway**

CloudTrail is your audit lens for AWS API activity. In a lab, your job is to connect the action you performed to the event CloudTrail recorded, then interpret the caller identity, target resource, and request details. For SAA-C03, focus on the distinction between management and data events, the Region-specific nature of event history, and the correct use of CLI tools for recent event lookup.
