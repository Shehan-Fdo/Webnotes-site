## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Right-Sizing and Elasticity**

Right-sizing means matching AWS resources to actual workload demand so you avoid paying for unused capacity while still meeting performance and availability goals. Elasticity means the system can automatically add or remove resources as demand changes. For the SAA-C03 exam, you should distinguish these ideas clearly: right-sizing is a design and optimization practice, while elasticity is a runtime behavior enabled by scaling mechanisms.

In practice, you use both together. You right-size the baseline capacity, then add elasticity so the workload can absorb spikes without overprovisioning for peak traffic all the time.

- **Right-sizing** reduces waste by selecting the smallest practical instance, storage, or throughput level that still meets requirements.
- **Elasticity** lets AWS resources expand and contract with demand.
- **Scalability** is the broader ability to handle growth; elasticity is the automatic, dynamic part of scalability.
- **Overprovisioning** improves headroom but increases cost and can hide design inefficiencies.

**Core concepts**

A good architecture starts with understanding workload patterns:

- Steady-state workloads often benefit from reserved capacity, Savings Plans, or smaller fixed fleets.
- Spiky workloads need automatic scaling and decoupling so they can absorb bursts.
- Unpredictable workloads often need serverless or queue-based designs to smooth demand.
- Latency-sensitive systems may need warm capacity even if they also scale out.

AWS gives you multiple levers for right-sizing:

- Compute size and family
- Storage type, size, and IOPS
- Database instance class and storage configuration
- Network architecture and data transfer patterns
- Caching and content delivery
- Purchase model and commitment strategy

A common exam trap is assuming the cheapest instance is always the best answer. In reality, the best choice is the one that meets performance, availability, and operational requirements at the lowest sustainable cost.

**How to identify right-sizing opportunities**

You usually start with metrics and usage history. Look for consistently low CPU, memory, network, or disk utilization, but interpret them carefully. Low CPU does not always mean a workload is oversized if memory pressure, burst credits, or I/O wait are the real bottlenecks.

Useful signals include:

- Average and peak utilization over time
- P95 or P99 latency
- Memory consumption and swap activity
- Disk queue depth and IOPS usage
- Network throughput and packet drops
- Application-level saturation indicators
- Cost allocation by service, tag, or account

AWS services and tools commonly used for this analysis:

- **Amazon CloudWatch** for metrics, alarms, dashboards, and anomaly detection
- **AWS Compute Optimizer** for instance, Auto Scaling group, EBS, Lambda, and ECS recommendations
- **AWS Cost Explorer** for spend trends and usage patterns
- **AWS Trusted Advisor** for underutilized resource checks
- **AWS Cost and Usage Report (CUR)** for detailed billing analysis

A practical workflow is:

1. Measure actual utilization and performance.
2. Identify resources with sustained headroom.
3. Validate whether the bottleneck is compute, memory, storage, or network.
4. Resize or change architecture.
5. Recheck metrics after the change.

> Do not right-size based on a short observation window. A workload that looks idle during business hours may be heavily used at night, during batch jobs, or during seasonal peaks.

**Elasticity patterns**

Elasticity is usually implemented with one or more of these patterns:

- **Horizontal scaling** adds more instances, containers, or tasks.
- **Vertical scaling** increases the size of a single instance or database node.
- **Scheduled scaling** changes capacity at known times.
- **Dynamic scaling** reacts to metrics in near real time.
- **Predictive scaling** uses historical patterns to pre-scale before demand arrives.

Horizontal scaling is the most common AWS exam answer for web and application tiers because it improves availability and avoids single-node limits. Vertical scaling is simpler but often requires downtime or a restart and eventually hits a ceiling.

**Auto Scaling for EC2**

Amazon EC2 Auto Scaling is the standard service for elastic compute fleets. It maintains desired capacity and can launch or terminate instances based on policies.

Key components:

- **Launch template** defines AMI, instance type, security groups, user data, and other launch settings.
- **Auto Scaling group (ASG)** manages the fleet and scaling behavior.
- **Scaling policy** determines when and how to scale.
- **Health checks** replace unhealthy instances automatically.

Common policy types:

- **Target tracking** keeps a metric near a target value, such as average CPU at 50%.
- **Step scaling** adds or removes capacity in steps based on alarm thresholds.
- **Simple scaling** is older and less flexible.
- **Scheduled scaling** adjusts capacity at specific times.

Example target tracking policy concept:

```json
{
  "AutoScalingGroupName": "web-asg",
  "PolicyName": "cpu-target-50",
  "PolicyType": "TargetTrackingScaling",
  "TargetTrackingConfiguration": {
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ASGAverageCPUUtilization"
    },
    "TargetValue": 50.0
  }
}
```

Useful CLI examples:

```bash
aws autoscaling describe-auto-scaling-groups --auto-scaling-group-names web-asg
aws autoscaling put-scaling-policy \
  --auto-scaling-group-name web-asg \
  --policy-name cpu-target-50 \
  --policy-type TargetTrackingScaling \
  --target-tracking-configuration file://policy.json
```

For exam purposes, remember:

- Use an ASG behind an Application Load Balancer for stateless web tiers.
- Use health checks to replace failed instances automatically.
- Use multiple Availability Zones for resilience and scaling.
- Use lifecycle hooks if you need to finish bootstrap or drain connections before termination.

**Elastic Load Balancing and stateless design**

Elasticity works best when the application tier is stateless. A load balancer distributes requests across multiple targets, allowing you to add or remove instances without user impact.

Common patterns:

- Store sessions in a shared cache or database instead of instance memory.
- Use Amazon S3 or EFS for shared files when appropriate.
- Keep application servers disposable and immutable.
- Use sticky sessions only when necessary, because they reduce flexibility.

Load balancers that often appear in scaling scenarios:

- **Application Load Balancer (ALB)** for HTTP/HTTPS and path-based routing
- **Network Load Balancer (NLB)** for ultra-low latency and TCP/UDP workloads
- **Gateway Load Balancer (GWLB)** for third-party network appliances

A stateless web tier behind an ALB is a classic elasticity design because new instances can join quickly and old ones can be removed without losing user state.

**Serverless elasticity**

Serverless services are often the easiest answer when you want automatic elasticity without managing servers.

Examples:

- **AWS Lambda** scales per request and is well suited for event-driven workloads.
- **Amazon API Gateway** handles API front ends and integrates with Lambda.
- **Amazon SQS** buffers bursts so downstream consumers can scale independently.
- **Amazon EventBridge** routes events to decoupled targets.

Why serverless helps:

- No instance fleet to manage
- Automatic scaling to match demand
- Pay-per-use pricing
- Good fit for spiky or intermittent traffic

Limitations to remember:

- Cold starts can affect latency-sensitive workloads.
- Concurrency limits may need adjustment.
- Long-running or stateful workloads may not fit well.
- Some integrations require careful timeout and retry design.

Example Lambda concurrency configuration:

```bash
aws lambda put-provisioned-concurrency-config \
  --function-name process-orders \
  --qualifier live \
  --provisioned-concurrent-executions 20
```

Provisioned concurrency is useful when you need to reduce cold starts for predictable traffic, but it adds cost and should be used selectively.

**Containers and elasticity**

Amazon ECS and Amazon EKS can also scale elastically.

Typical approaches:

- Scale tasks or pods horizontally based on CPU, memory, or custom metrics.
- Use Cluster Auto Scaling or managed node groups to add worker capacity.
- Use Fargate for serverless container execution when you want to avoid managing hosts.

Practical considerations:

- Application code should tolerate instance or pod replacement.
- Use readiness and liveness checks.
- Keep container images small to reduce startup time.
- Externalize state to managed services.

For Kubernetes, the **Horizontal Pod Autoscaler (HPA)** scales pods, while cluster autoscaling adds nodes when the cluster lacks capacity. This distinction matters on the exam.

**Databases and elasticity**

Databases are often the hardest part of elasticity because stateful systems do not scale as easily as stateless ones.

Common AWS options:

- **Amazon RDS** supports vertical scaling and read replicas for read-heavy workloads.
- **Amazon Aurora** supports read scaling with replicas and can use Aurora Serverless for variable demand.
- **Amazon DynamoDB** offers on-demand capacity or auto scaling for predictable key-value and document workloads.
- **Amazon ElastiCache** can reduce database load by caching hot data.

Right-sizing database resources means looking at:

- CPU and memory
- Storage growth and IOPS
- Connection counts
- Query latency
- Replica lag
- Buffer cache hit ratio

Important patterns:

- Use read replicas to offload read traffic.
- Use caching to reduce repeated reads.
- Use partitioning or sharding when a single node becomes a bottleneck.
- Use Aurora Serverless when demand is variable and you want automatic capacity adjustment.

DynamoDB capacity modes:

| Mode | Best for | Notes |
|---|---|---|
| On-demand | Unpredictable traffic | Simplest elasticity model |
| Provisioned with auto scaling | Steady but variable traffic | More control over cost |
| Provisioned without auto scaling | Very predictable traffic | Requires manual tuning |

**Storage right-sizing**

Storage is often overlooked because it seems cheap, but it can still create waste or bottlenecks.

Consider:

- **Amazon EBS** volume type, size, and IOPS
- **Amazon S3** storage class and lifecycle policies
- **Amazon EFS** throughput mode and performance mode
- **Amazon FSx** deployment type and capacity

Examples:

- Use gp3 instead of older gp2 when you want independent control of size, IOPS, and throughput.
- Use S3 lifecycle policies to move infrequently accessed objects to cheaper classes.
- Use EFS when multiple instances need shared POSIX file storage.
- Use provisioned IOPS only when workload latency requires it.

CLI example for EBS volume modification:

```bash
aws ec2 modify-volume \
  --volume-id vol-0123456789abcdef0 \
  --volume-type gp3 \
  --iops 6000 \
  --throughput 250
```

Storage right-sizing is not only about cost. Too little IOPS or throughput can cause application latency, queue buildup, and failed scaling because the compute layer waits on storage.

**Cost optimization and commitment choices**

Right-sizing and elasticity directly affect cost. You should choose the purchase model that matches the workload shape.

General guidance:

- Use **On-Demand** for unpredictable or short-lived workloads.
- Use **Reserved Instances** or **Savings Plans** for steady baseline usage.
- Use **Spot Instances** for fault-tolerant, interruptible workloads.
- Use **Auto Scaling** to keep baseline capacity low and burst only when needed.

Spot Instances are especially useful for batch, CI/CD, rendering, and distributed processing. They are not ideal for critical single-instance workloads unless you design for interruption.

A common architecture is:

- Baseline capacity on On-Demand or Reserved capacity
- Burst capacity on Spot
- Queue-based buffering to absorb interruptions

**Design patterns that improve elasticity**

Several architecture patterns make scaling easier:

- **Decouple with queues** so producers and consumers scale independently.
- **Use asynchronous processing** for non-interactive work.
- **Cache frequently accessed data** to reduce backend load.
- **Partition workloads** by tenant, shard, or message type.
- **Use immutable infrastructure** so new instances are interchangeable.
- **Externalize state** to managed services.

Example queue-based worker pattern:

```text
Users -> ALB -> Web tier -> SQS queue -> Worker Auto Scaling group -> RDS/DynamoDB
```

This pattern helps because the queue absorbs spikes while workers scale based on queue depth or age of oldest message.

**Metrics and alarms to watch**

Good scaling decisions depend on the right metrics.

Common compute metrics:

- CPUUtilization
- MemoryUtilization
- NetworkIn / NetworkOut
- DiskReadOps / DiskWriteOps
- DiskQueueDepth

Common application metrics:

- Request count
- Error rate
- Latency
- Concurrent sessions
- Queue depth
- Message age

Scaling alarms should reflect user experience, not just infrastructure health. For example, CPU at 80% may be acceptable if latency stays low, while CPU at 40% may still be a problem if memory is exhausted.

**Practical examples**

Example: web application with bursty traffic

- Put the app tier behind an ALB.
- Run the app on EC2 Auto Scaling across multiple AZs.
- Store sessions in ElastiCache or DynamoDB.
- Use target tracking on average CPU or request count per target.
- Use CloudWatch alarms for latency and 5xx errors.

Example: nightly batch processing

- Use SQS to queue jobs.
- Run workers on Spot Instances with an On-Demand baseline.
- Scale workers based on queue depth.
- Store results in S3 or RDS.
- Use scheduled scaling if the batch window is predictable.

Example: unpredictable API workload

- Use API Gateway and Lambda.
- Use DynamoDB on-demand for the data layer.
- Add provisioned concurrency only if latency requires it.
- Use CloudWatch alarms for throttles and errors.

**Exam decision guide**

When you see a question about right-sizing or elasticity, ask:

- Is the workload steady, spiky, or unpredictable?
- Is the bottleneck compute, memory, storage, or database connections?
- Can the application be made stateless?
- Do you need automatic scaling or just a smaller baseline?
- Is the workload fault tolerant enough for Spot?
- Would serverless remove the need to manage capacity?

Typical answers:

- Use Auto Scaling for stateless EC2 fleets.
- Use Lambda for event-driven elasticity.
- Use DynamoDB on-demand for variable NoSQL traffic.
- Use read replicas or caching for read-heavy databases.
- Use SQS to buffer bursts and decouple components.
- Use Savings Plans or Reserved Instances for steady baseline usage.

> If a question emphasizes “minimize operational overhead,” prefer managed or serverless options over self-managed scaling logic.

**Common pitfalls**

- Confusing vertical scaling with elasticity.
- Scaling only the web tier while the database remains the bottleneck.
- Ignoring session state when adding or removing instances.
- Using fixed-size fleets for bursty traffic.
- Choosing Spot Instances for workloads that cannot tolerate interruption.
- Right-sizing based on average utilization alone and missing peak demand.
- Forgetting that storage and network can be the limiting factors, not CPU.

**Quick reference**

- **Right-sizing**: match resources to actual demand.
- **Elasticity**: automatically adjust capacity as demand changes.
- **Horizontal scaling**: add more nodes.
- **Vertical scaling**: make a node bigger.
- **Auto Scaling group**: manages EC2 fleet elasticity.
- **Target tracking**: simplest common scaling policy.
- **Serverless**: automatic scaling with minimal infrastructure management.
- **Decoupling**: use queues/events to absorb bursts.
- **Stateless design**: essential for easy scale-out.
