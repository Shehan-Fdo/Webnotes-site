## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and exam framing**

This lab trains you to inspect a reference architecture and identify where AWS spend is likely to concentrate. In the exam, you are rarely asked for exact prices; you are asked to recognize the dominant cost drivers, compare design alternatives, and choose the most cost-effective option that still meets performance, availability, and operational requirements.

A good cost review starts with the workload shape:

- **Steady production traffic** usually favors predictable pricing and right-sized managed services.
- **Dev/test** often benefits from aggressive scheduling, smaller instances, and lower retention.
- **Batch** workloads often prefer Spot Instances, Auto Scaling, and ephemeral storage.
- **Fault-tolerant** systems may justify higher baseline cost for Multi-AZ, redundancy, and data replication.
- **Compliance-heavy** workloads may add logging, encryption, retention, and cross-account controls that increase cost.

**How to analyze a reference design**

Use a repeatable workflow so you do not miss hidden charges:

1. Identify every component in the diagram.
2. Classify each component as compute, storage, network, database, security, or observability.
3. Estimate recurring charges first, then one-time or variable charges.
4. Rank the likely hotspots by cost impact.
5. Validate assumptions with AWS pricing tools.
6. Propose optimizations that preserve the workload’s requirements.

A practical review is not just “what is expensive,” but “what is expensive relative to traffic, durability, latency, and resilience needs.”

**Common AWS pricing tools for validation**

Use AWS pricing and billing tools to confirm your assumptions instead of guessing.

- **AWS Pricing Calculator** helps you estimate future cost before deployment.
  - Use it to compare instance families, storage classes, load balancers, and data transfer patterns.
  - It is especially useful when you need to compare two architectures side by side.
- **Cost Explorer** helps you analyze actual spend after deployment.
  - Use it to group by service, linked account, tag, region, or usage type.
  - It is useful for spotting trends such as rising NAT Gateway or data transfer charges.
- **Cost and Usage Report (CUR)** provides the most detailed billing data.
  - Use it when you need line-item analysis, chargeback, or deeper filtering by usage type.
  - It is the best source for validating whether a hotspot comes from storage, requests, transfer, or provisioned capacity.
- **AWS Trusted Advisor cost recommendations** highlight underused or idle resources.
  - Use it to find idle EC2 instances, underutilized EBS volumes, or unattached Elastic IP addresses.
  - It is a fast way to identify obvious waste in an account.

> In exam scenarios, you usually do not need to name the exact tool unless the question asks how to validate or investigate spend. If it does, choose the tool that matches the level of detail required.

**Reference design example**

Consider this sample architecture:

```text
Users
  |
  v
CloudFront
  |
  v
Application Load Balancer (ALB)
  |
  +--> Auto Scaling Group of EC2 app servers in 2 AZs
  |
  +--> Amazon RDS Multi-AZ
  |
  +--> Amazon S3 for static assets and logs
  |
  +--> NAT Gateway for private subnet outbound access
  |
  +--> CloudWatch Logs and metrics
```

This design is common, but several components can dominate cost depending on traffic patterns and data volume.

**Step-by-step hotspot analysis**

1. **Identify the always-on services**
   - ALB charges per hour and per load balancer capacity unit.
   - NAT Gateway charges per hour and per GB processed.
   - RDS Multi-AZ charges for the primary and standby capacity.
   - EC2 instances run continuously unless you scale them down.
   - CloudWatch Logs can accumulate ingestion and retention charges.

2. **Estimate recurring charges**
   - EC2 cost depends on instance family, size, purchase option, and utilization.
   - RDS cost depends on engine, instance class, storage, IOPS, and Multi-AZ.
   - NAT Gateway cost grows with both uptime and outbound traffic.
   - S3 cost depends on storage class, request volume, lifecycle policy, and retrieval.
   - Data transfer can become significant when traffic crosses AZs, regions, or the internet.

3. **Rank likely hotspots**
   - For a small app with moderate traffic, the biggest costs are often EC2, RDS, and NAT Gateway.
   - For a content-heavy app, CloudFront and S3 request patterns may matter more than compute.
   - For chatty microservices, cross-AZ data transfer and load balancer traffic can become major drivers.
   - For logging-heavy compliance workloads, CloudWatch Logs and S3 retention can be surprisingly large.

4. **Validate with pricing tools**
   - Use the Pricing Calculator to compare `t3.medium` vs `m6i.large`, or ALB vs NLB.
   - Use Cost Explorer to see whether spend is concentrated in `EC2-Other`, `NatGateway`, `DataTransfer`, or `RDS`.
   - Use CUR to inspect usage types such as `NatGateway-Bytes`, `TimedStorage-ByteHrs`, or `DataTransfer-Out-Bytes`.
   - Use Trusted Advisor to identify idle resources that should be terminated or rightsized.

5. **Propose optimizations**
   - Replace always-on capacity with Auto Scaling where traffic is variable.
   - Move static content to S3 and CloudFront.
   - Reduce NAT Gateway traffic by using VPC endpoints.
   - Choose the right EC2 purchase option for the workload.
   - Tune storage class, retention, and backup policies.

**EC2 cost hotspots and purchase options**

EC2 is often the first place to look, but the cheapest instance is not always the best choice. Cost depends on instance family, size, region, operating system, and purchase model.

- **On-Demand Instances**
  - Best for unpredictable or short-lived workloads.
  - Highest flexibility, no commitment.
  - Good for dev/test, spiky traffic, and early-stage systems.
- **Reserved Instances**
  - Best for steady-state usage with predictable baseline demand.
  - Lower effective hourly cost in exchange for commitment.
  - Useful when you know you will run a specific instance family or size for a long period.
- **Savings Plans**
  - Best for flexible steady usage across instance families or even services, depending on the plan type.
  - Often easier to manage than strict instance reservations.
  - Good when you expect consistent compute spend but want some flexibility.
- **Spot Instances**
  - Best for fault-tolerant, interruptible, or batch workloads.
  - Lowest cost, but instances can be reclaimed.
  - Not suitable for stateful production tiers that cannot tolerate interruption.

Other EC2 cost levers:

- Right-size instance types based on CPU, memory, and network usage.
- Use Auto Scaling to match capacity to demand.
- Stop non-production instances outside business hours.
- Prefer Graviton-based instances when the application supports ARM architecture and you want better price-performance.
- Avoid overprovisioning for peak traffic if the workload can scale horizontally.

**Example CLI checks for EC2 cost review**

```bash
aws ec2 describe-instances \
  --query 'Reservations[].Instances[].[InstanceId,InstanceType,State.Name,Placement.AvailabilityZone]' \
  --output table

aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistics Average \
  --start-time 2026-04-01T00:00:00Z \
  --end-time 2026-04-26T00:00:00Z \
  --period 86400
```

Low CPU alone does not prove an instance is wasteful, but it is a strong signal to investigate memory, network, and disk metrics before rightsizing.

**Storage hotspots: EBS, snapshots, and S3**

Storage is often underestimated because individual charges look small, but they accumulate across volumes, snapshots, and retention.

- **EBS volumes**
  - Charged for provisioned capacity, not just used data.
  - General Purpose SSD volumes are common, but overprovisioned volumes waste money.
  - Provisioned IOPS volumes add cost for performance guarantees.
- **EBS snapshots**
  - Stored incrementally, but long retention and many snapshots increase cost.
  - Orphaned snapshots from old AMIs or deleted instances can linger.
  - Snapshot lifecycle policies help control growth.
- **S3 storage**
  - Cost depends on storage class, object size, request rate, and retrieval.
  - Frequent access favors Standard; infrequent access may favor Standard-IA, One Zone-IA, Glacier Instant Retrieval, or Glacier Flexible Retrieval depending on recovery needs.
  - Lifecycle policies reduce cost by moving older data to cheaper classes.
- **S3 request costs**
  - High request volume can matter for small-object or API-heavy workloads.
  - Logging pipelines and analytics jobs can generate many PUT, GET, and LIST requests.

Practical storage optimizations:

- Delete unattached EBS volumes and unused snapshots.
- Use lifecycle rules for S3 logs and backups.
- Compress logs before storing them.
- Choose the right durability and retrieval model for archives.
- Avoid storing transient application data on expensive persistent volumes when instance store or ephemeral storage is acceptable.

**Example S3 lifecycle policy**

```json
{
  "Rules": [
    {
      "ID": "MoveLogsToIA",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER"
        }
      ],
      "Expiration": {
        "Days": 365
      }
    }
  ]
}
```

**Network cost hotspots: data transfer, NAT, and load balancing**

Network charges are a frequent exam trap because they are easy to overlook in diagrams.

- **Data transfer out to the internet**
  - Usually charged and often significant for public-facing apps.
  - CloudFront can reduce origin egress and improve latency.
- **Cross-AZ data transfer**
  - Can occur when components in different Availability Zones communicate.
  - Common in multi-AZ architectures with chatty application tiers.
- **Cross-region data transfer**
  - Usually more expensive than same-region transfer.
  - Appears in disaster recovery, replication, and global applications.
- **NAT Gateway**
  - Charges for uptime and processed bytes.
  - Often a hidden hotspot in private subnet designs that send all outbound traffic through NAT.
- **Load balancers**
  - **ALB** is usually best for HTTP/HTTPS and content-based routing.
  - **NLB** is better for very high throughput, low latency, or non-HTTP protocols.
  - Cost depends on hours and traffic processed, so the cheaper choice depends on protocol and traffic shape.
- **CloudFront**
  - Can reduce origin load and internet egress from the origin.
  - Often cost-effective for global content delivery and cacheable assets.

When to choose each:

- Use **CloudFront** for static assets, global users, and cacheable content.
- Use **ALB** for web applications, path-based routing, and HTTP features.
- Use **NLB** for TCP/UDP workloads, static IP needs, or extreme throughput.
- Use **VPC endpoints** to reduce NAT Gateway traffic for AWS service access.
- Use **private connectivity** patterns carefully when compliance or latency requirements justify them.

**Example: reducing NAT Gateway spend with VPC endpoints**

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.s3 \
  --vpc-endpoint-type Gateway \
  --route-table-ids rtb-0123456789abcdef0
```

Gateway endpoints for S3 and DynamoDB can remove a large amount of NAT traffic when private subnets only need AWS service access.

**Database cost hotspots**

Databases are often one of the largest recurring costs because they combine compute, storage, backups, and high availability.

- **Amazon RDS**
  - Cost depends on instance class, engine, storage, IOPS, backup retention, and Multi-AZ.
  - Multi-AZ improves availability but roughly increases database-related cost because you run standby capacity.
  - Read replicas add cost but can offload read traffic.
- **Amazon Aurora**
  - Can be cost-effective for managed relational workloads, but storage and I/O patterns still matter.
  - Aurora Serverless may help variable workloads, though it is not always the cheapest choice for steady high utilization.
- **Amazon DynamoDB**
  - On-demand mode suits unpredictable traffic.
  - Provisioned capacity with Auto Scaling suits predictable workloads.
  - Read/write patterns, item size, and indexes affect cost.
- **Amazon ElastiCache**
  - Useful when caching reduces expensive database reads.
  - Adds cost, but can lower total system cost by reducing database load.

Exam-relevant database tradeoffs:

- Use Multi-AZ for production systems that require high availability.
- Use read replicas when read scaling is the main issue.
- Use smaller instances or storage optimization for dev/test databases.
- Use serverless or on-demand models for spiky or unpredictable access.
- Avoid overbuilding database redundancy for workloads that do not require it.

**Observability and logging costs**

Monitoring is necessary, but logging everything forever is expensive.

- **CloudWatch Logs**
  - Ingestion and retention both contribute to cost.
  - Verbose application logs, debug logs, and long retention periods increase spend.
- **CloudWatch metrics**
  - Custom metrics can add cost if you publish too many dimensions or high-frequency data.
- **X-Ray**
  - Tracing can help with debugging distributed systems, but sampling should be tuned.
- **S3-based log archives**
  - Often cheaper for long-term retention than keeping everything in CloudWatch Logs.

Practical controls:

- Set retention periods for log groups.
- Reduce debug logging in production.
- Use structured logs and sampling.
- Archive older logs to S3 and transition them to cheaper storage classes.

**Example CloudWatch Logs retention**

```bash
aws logs put-retention-policy \
  --log-group-name /aws/lambda/my-function \
  --retention-in-days 14
```

**Workload-specific optimization choices**

The right optimization depends on the workload type.

- **Production**
  - Prioritize reliability, predictable performance, and controlled scaling.
  - Use Savings Plans or Reserved Instances for steady baseline compute.
  - Keep Multi-AZ and backups where availability requirements justify them.
- **Dev/test**
  - Stop resources when not in use.
  - Use smaller instances and shorter retention.
  - Prefer On-Demand with scheduling over long commitments.
- **Batch**
  - Use Spot Instances where interruption is acceptable.
  - Use Auto Scaling and ephemeral storage.
  - Stage data efficiently to reduce transfer and storage overhead.
- **Fault-tolerant**
  - Accept higher cost for redundancy, but avoid unnecessary duplication.
  - Use managed services that reduce operational overhead.
  - Optimize cross-AZ traffic if the application is chatty.
- **Compliance-heavy**
  - Expect higher logging, encryption, and retention costs.
  - Use lifecycle policies and archive tiers to control long-term storage.
  - Keep only the required retention period for each data class.

**Worked end-to-end hotspot outcome**

For the sample architecture, a realistic cost review might conclude:

- **Highest recurring costs**
  - RDS Multi-AZ database
  - NAT Gateway processing
  - EC2 application servers
  - Cross-AZ data transfer if the app tier is chatty
- **Secondary costs**
  - ALB traffic and hours
  - CloudWatch Logs ingestion and retention
  - EBS volumes and snapshots
  - S3 storage and requests
- **Likely optimizations**
  - Add S3 and DynamoDB VPC endpoints to reduce NAT usage.
  - Move static assets behind CloudFront.
  - Right-size EC2 instances and use Auto Scaling.
  - Evaluate whether RDS Multi-AZ is required for all environments.
  - Set log retention and archive older logs to S3.
  - Review snapshot policies and delete stale backups.

If the workload is a production web app, keep Multi-AZ and focus on reducing waste around it. If it is dev/test, the same design should be aggressively scheduled, downsized, and simplified. If it is batch-oriented, Spot and ephemeral storage become much more attractive.

**How to use AWS pricing tools in the lab**

A practical lab workflow looks like this:

1. Open the reference design and list all billable components.
2. Enter the expected usage into the AWS Pricing Calculator.
3. Compare at least two alternatives for the biggest hotspots.
4. Check Cost Explorer for actual spend patterns if the environment already exists.
5. Use CUR to confirm the exact usage types behind the largest charges.
6. Review Trusted Advisor for idle or underused resources.
7. Document the top three hotspots and the best optimization for each.

**CLI and console habits that help in cost analysis**

- Use tags such as `Environment=dev`, `Environment=prod`, and `Owner=team-name` so you can group spend.
- Filter by region because pricing varies by region.
- Compare monthly and daily trends to catch spikes early.
- Look for resources that are running but not serving traffic.
- Check whether backups, snapshots, and logs are growing faster than expected.

**Quick exam cues**

- If the question mentions private subnets with heavy outbound AWS service access, think **VPC endpoints** before NAT Gateway.
- If it mentions global static content, think **CloudFront + S3**.
- If it mentions steady compute usage, think **Savings Plans** or **Reserved Instances**.
- If it mentions interruptible batch jobs, think **Spot Instances**.
- If it mentions long-term logs or backups, think **S3 lifecycle policies** and retention control.
- If it mentions multi-AZ chatty traffic, think about **cross-AZ data transfer**.

**Key takeaway**

Cost hotspot analysis is about identifying where the architecture spends money continuously, where variable traffic amplifies spend, and which optimizations preserve the workload’s requirements. The best answer in the exam is usually not the cheapest possible design; it is the cheapest design that still satisfies availability, performance, security, and operational needs.
