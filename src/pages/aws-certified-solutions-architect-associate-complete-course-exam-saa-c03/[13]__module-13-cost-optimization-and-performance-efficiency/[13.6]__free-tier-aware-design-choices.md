## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Free Tier fundamentals**

AWS Free Tier is useful for labs, prototypes, and exam practice, but you need to design with its limits in mind. The exam often tests whether you can choose a service or architecture that stays low-cost while still meeting a requirement.

Free Tier is not a single universal allowance. It varies by:

- **Account eligibility**: many benefits apply only to new AWS accounts and may require specific signup conditions.
- **Service**: each service has its own free allowance and metering model.
- **Region**: pricing and allowances can differ by region.
- **Usage type**: charges may be based on instance-hours, requests, storage, data transfer, log ingestion, or retention.

A safe mental model is: Free Tier helps you start cheaply, but it does not guarantee zero cost if you leave resources running or choose the wrong architecture.

<dl>
<dt>Instance-hour</dt>
<dd>The amount of time a compute instance runs, usually billed per second or hour depending on the service.</dd>
<dt>Request-based billing</dt>
<dd>Charges based on the number of API calls, invocations, or operations performed.</dd>
<dt>Data transfer</dt>
<dd>Charges for data moving between services, across Availability Zones, or out to the internet.</dd>
</dl>

> <mark>Always verify current Free Tier terms in the AWS console or pricing pages before labs or exams that involve cost-sensitive design.</mark>

**What commonly matters on the exam**

The exam usually focuses on whether you understand the main cost drivers, not exact dollar amounts. The most important usage dimensions are:

- **Compute time**: EC2 instance-hours, Lambda invocations and duration, container runtime.
- **Storage**: EBS volumes, snapshots, S3 storage, database storage.
- **Requests**: S3 requests, API Gateway requests, CloudWatch API usage, DynamoDB read/write capacity.
- **Network transfer**: internet egress, cross-AZ traffic, NAT Gateway traffic, public IPv4 usage.
- **Observability**: CloudWatch logs, metrics, alarms, retention.

When you see a design question, ask yourself which component is likely to accumulate cost fastest. Often the answer is not the obvious compute service, but the supporting network or monitoring layer.

**Free Tier-aware design principles**

Design for the smallest practical footprint first, then add only what the use case needs. For labs and proofs of concept, you usually want to minimize always-on resources and avoid managed services that charge per request or per GB in ways that are easy to overlook.

- Prefer **serverless** or managed services when they reduce idle cost.
- Stop or delete resources when you are done with a lab.
- Use the smallest viable instance size and storage allocation.
- Avoid architectures that require always-on network appliances unless the scenario demands them.
- Keep logging and monitoring intentionally limited during experiments.

A good exam-safe pattern is to choose the simplest architecture that meets the requirement and has the fewest hidden cost drivers.

**Compute choices: EC2, Lambda, and containers**

For low-cost labs, AWS Lambda is often cheaper than EC2 when the workload is intermittent or event-driven because you do not pay for idle time. EC2 can still be appropriate when you need full OS control, persistent processes, or software that does not fit Lambda constraints.

- **EC2** is cost-sensitive because you pay for running instances, attached storage, and sometimes public IPv4 usage.
- **Lambda** is cost-sensitive because you pay per invocation and execution duration, so very chatty or long-running workloads can still cost money.
- **Containers** can be efficient, but always-on container services may introduce hidden costs through compute, load balancing, or orchestration.

For a static website or simple API-backed demo, a serverless approach is often the better Free Tier-aware choice than a small EC2 instance that sits idle most of the day.

**Example: static site design comparison**

| Requirement | Better low-cost choice | Why |
|---|---|---|
| Static website only | S3 static website hosting or S3 + CloudFront | No server to keep running |
| Static site with occasional dynamic calls | S3 + Lambda + API Gateway | Low idle cost, but watch request charges |
| Full custom OS or legacy app | EC2 | Needed when OS-level control matters |

S3 static hosting is usually the simplest low-cost option for static content. If you add CloudFront, remember that CloudFront has its own pricing model and may have Free Tier allowances for some usage patterns, but it is not free in all cases. Use it when you need caching, global delivery, or HTTPS at the edge, not just because it sounds modern.

> Do not assume a service is “Free Tier friendly” just because it is serverless or managed. API Gateway, CloudFront, and CloudWatch can all generate charges if you use them heavily or configure them carelessly.

**Storage choices: S3, EBS, snapshots, and databases**

Storage is a common source of accidental cost because it persists after you stop compute resources.

- **S3** is usually inexpensive for small objects, but request volume and data transfer still matter.
- **EBS** volumes continue billing while attached or unattached until deleted.
- **Snapshots** accumulate over time and are easy to forget.
- **Databases** often charge for storage, provisioned capacity, backups, and I/O.

For labs, delete what you no longer need. Stopping an EC2 instance does not delete its EBS volume, and deleting the instance does not always remove snapshots or other attached resources.

Practical cleanup checklist:

- Delete unused **EBS volumes**.
- Remove old **snapshots**.
- Empty and delete temporary **S3 buckets** if they are no longer needed.
- Delete test **RDS** databases and automated backups if the lab is finished.
- Review **DynamoDB** tables for provisioned capacity or on-demand usage that is no longer required.

**Networking cost traps**

Networking is one of the easiest places to overspend in a lab because the charges are not always obvious.

- **NAT Gateway** is a common cost trap because it charges for uptime and data processing.
- **Public IPv4 addresses** can incur charges whether they are attached or detached, depending on current AWS pricing rules.
- **Cross-AZ traffic** can add cost when resources communicate across Availability Zones.
- **Data transfer out** to the internet is often billable after small allowances.

For exam scenarios, if you need private subnets only for security but the workload is simple, consider whether you can avoid a NAT Gateway by using:

- **VPC endpoints** for AWS service access.
- **Public subnets** for a temporary lab when security requirements are relaxed.
- **Bastion-less access patterns** such as Systems Manager Session Manager when appropriate.

**Example: replacing NAT Gateway in a lab**

A common lab pattern is private EC2 instances needing access to S3 or Systems Manager. Instead of placing a NAT Gateway in the design, you can use VPC endpoints.

```text
Private subnet EC2
   |
   +-- Interface endpoint -> Systems Manager, EC2 Messages, SSMMessages
   |
   +-- Gateway endpoint -> S3
```

This reduces cost and avoids the always-on NAT Gateway charge. It is especially useful in exam questions where the goal is secure access with minimal operational overhead.

**Observability and CloudWatch**

CloudWatch is useful, but it is not automatically cheap. Basic metrics may be low-cost or included in allowances, but logs, alarms, custom metrics, high-resolution metrics, and long retention can add up quickly.

Use CloudWatch intentionally:

- Set short **log retention** for labs.
- Avoid shipping verbose application logs unless you need them.
- Delete test alarms after validation.
- Use only the metrics you need for the scenario.

A practical log-retention example:

```bash
aws logs put-retention-policy \
  --log-group-name /aws/lambda/demo-function \
  --retention-in-days 7
```

This is a good habit for temporary environments because it limits storage growth and keeps old logs from accumulating indefinitely.

**Example: EC2 cleanup workflow**

If you use EC2 for a lab, the cost-safe workflow is to stop, verify, and delete.

1. Stop the instance when you are done testing.
2. Confirm whether the attached EBS volume is still present.
3. Delete the instance if you no longer need it.
4. Remove unused EBS volumes and snapshots.
5. Check for public IPv4 charges and other attached resources.

Useful CLI commands:

```bash
aws ec2 stop-instances --instance-ids i-0123456789abcdef0
aws ec2 describe-volumes --filters Name=status,Values=available
aws ec2 delete-volume --volume-id vol-0123456789abcdef0
aws ec2 describe-snapshots --owner-ids self
aws ec2 delete-snapshot --snapshot-id snap-0123456789abcdef0
```

If you are creating a bucket for a lab, remember that region behavior matters. In many regions, you should explicitly specify the region when creating resources so you do not accidentally place them somewhere unexpected.

```bash
aws s3api create-bucket \
  --bucket my-lab-bucket-123456 \
  --region us-west-2 \
  --create-bucket-configuration LocationConstraint=us-west-2
```

For `us-east-1`, the bucket creation behavior differs, so do not blindly reuse the same command everywhere.

**API Gateway and serverless cost awareness**

API Gateway is often paired with Lambda in low-ops architectures, but it is not a “free by default” service. It can still be a good choice when you need a managed HTTP front end, authentication integration, throttling, or request routing.

Use it when:

- You need a managed API endpoint.
- You want to avoid running a web server on EC2.
- You need to connect a static front end to Lambda or other back-end services.

Be cautious when:

- The workload is high-traffic and request-heavy.
- You only need a simple internal demo and could use a lighter alternative.
- You are building a lab where request charges could accumulate unexpectedly.

A common exam-safe architecture is:

- **S3** for static content.
- **Lambda** for small dynamic tasks.
- **API Gateway** only if an HTTP API is actually required.
- **DynamoDB** for simple key-value persistence.

That design is valid, but it is not automatically the cheapest if the API layer sees lots of traffic.

**Free Tier-aware vs production-aware design**

Free Tier-aware design is ideal for labs, demos, and proofs of concept. Production-aware design prioritizes durability, observability, scalability, and security even if it costs more.

- In a **lab**, you may accept public subnets, minimal logging, and temporary resources.
- In **production**, you usually need multi-AZ design, backups, monitoring, and tighter network controls.
- In an **exam question**, choose the design that best matches the stated requirement, not the cheapest design by default.

This distinction matters because the exam often asks for the “most cost-effective” solution that still satisfies the business need. That does not always mean the absolute lowest-cost architecture. It means the lowest-cost architecture that still meets availability, security, and operational requirements.

**Scenario-based exam guidance**

If the requirement is a static brochure site with minimal traffic, choose S3 static hosting rather than EC2.

If the requirement is a small event-driven backend, choose Lambda and a managed data store rather than a continuously running server.

If the requirement is a private lab environment, use VPC endpoints and avoid NAT Gateway where possible.

If the requirement is a short-lived test environment, stop and delete resources aggressively, and limit logs and snapshots.

If the requirement is a production workload, do not optimize only for Free Tier. Use the architecture that meets resilience and security needs first.

**Common overage prevention habits**

These habits help you stay within Free Tier-like budgets and avoid surprise bills:

- Turn off or terminate idle compute.
- Delete unused storage and backups.
- Set CloudWatch log retention to a short period.
- Avoid NAT Gateway unless required.
- Watch public IPv4 usage.
- Prefer request-efficient designs for APIs and data access.
- Review region-specific pricing before deploying.

A simple lab cleanup command set can save money:

```bash
aws ec2 describe-instances --filters Name=instance-state-name,Values=running
aws ec2 describe-addresses
aws s3 ls
aws logs describe-log-groups
```

Use the output to identify resources that should be stopped, deleted, or retained.

**Key takeaways**

- Free Tier is limited by account eligibility, service, region, and usage type.
- The biggest cost drivers are often compute time, storage, requests, data transfer, and logs.
- EC2, NAT Gateway, CloudWatch, API Gateway, and public IPv4 usage are common cost traps.
- S3, Lambda, VPC endpoints, and short-lived resources are often better choices for labs.
- For exams, choose the simplest architecture that satisfies the requirement and avoids hidden charges.
