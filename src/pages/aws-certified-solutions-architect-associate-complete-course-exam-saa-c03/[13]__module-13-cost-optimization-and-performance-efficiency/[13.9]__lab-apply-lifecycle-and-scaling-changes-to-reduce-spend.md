## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and cost-optimization mindset**

This lab focuses on reducing spend by changing how data and compute resources scale over time. In AWS, the biggest savings usually come from matching resource cost to actual usage: move infrequently accessed data to cheaper storage classes, remove expired objects automatically, and scale compute capacity only when demand requires it.

The key idea is to use managed automation instead of manual cleanup. You want policies that enforce retention, lifecycle transitions, and elastic scaling so that the environment stays right-sized as traffic and data age change.

- **Lifecycle management** reduces storage cost by transitioning or expiring objects automatically.
- **Scaling changes** reduce compute cost by adding capacity only when needed and removing it when demand falls.
- **Monitoring** validates whether the changes actually lower spend without hurting performance.

**Storage lifecycle optimization**

Amazon S3 lifecycle policies let you automate object transitions and deletions based on object age, current version status, or incomplete multipart uploads. This is one of the most common ways to reduce storage spend in a lab or production environment.

Typical use cases include:
- Moving objects from `S3 Standard` to `S3 Standard-IA` after a short period.
- Moving older objects to `S3 Glacier Instant Retrieval`, `S3 Glacier Flexible Retrieval`, or `S3 Glacier Deep Archive` when access becomes rare.
- Expiring temporary files, logs, or test artifacts after a fixed number of days.
- Cleaning up incomplete multipart uploads to avoid paying for abandoned parts.

A lifecycle configuration is attached to a bucket and evaluated by S3 automatically. You do not need to run a cron job or custom cleanup script.

```json
{
  "Rules": [
    {
      "ID": "Archive-and-expire-logs",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER_IR"
        }
      ],
      "Expiration": {
        "Days": 365
      },
      "AbortIncompleteMultipartUpload": {
        "DaysAfterInitiation": 7
      }
    }
  ]
}
```

- **Transition** changes the storage class while keeping the object.
- **Expiration** deletes the object when it is no longer needed.
- **AbortIncompleteMultipartUpload** stops paying for unfinished uploads.

> Lifecycle transitions are not instantaneous. S3 evaluates them asynchronously, so plan for a delay between the rule becoming eligible and the object actually moving.

**When to choose each storage class**

Choosing the right storage class matters because the cheapest class is not always the best fit. You balance retrieval speed, retrieval cost, minimum storage duration, and access frequency.

| Storage class | Best for | Cost pattern | Notes |
|---|---|---|---|
| `S3 Standard` | Frequently accessed data | Higher storage cost, low access friction | Default choice for active workloads |
| `S3 Standard-IA` | Infrequently accessed data that still needs fast retrieval | Lower storage cost, retrieval charges apply | Has minimum storage duration |
| `S3 One Zone-IA` | Re-creatable infrequent data in one AZ | Lower cost than Standard-IA | Less resilient because data is stored in one Availability Zone |
| `S3 Glacier Instant Retrieval` | Archive data needing millisecond access | Very low storage cost, retrieval cost applies | Good for long-lived archives with occasional reads |
| `S3 Glacier Flexible Retrieval` | Archive data with occasional retrieval | Lower storage cost, slower retrieval options | Useful when retrieval can wait |
| `S3 Glacier Deep Archive` | Long-term retention with rare access | Lowest storage cost, slowest retrieval | Best for compliance or archival retention |

Use `One Zone-IA` only when you can tolerate the loss of a single Availability Zone and can regenerate the data if needed. For durable business data, `Standard-IA` or a Glacier class is usually safer.

**Lifecycle policy design patterns**

Lifecycle rules work best when they reflect data purpose rather than a generic age threshold. For example, application logs, build artifacts, and temporary exports often have predictable retention windows.

Common patterns:
- **Logs**
  - Keep recent logs in `S3 Standard` for quick troubleshooting.
  - Transition older logs to an archive class.
  - Expire logs after the compliance retention period.
- **Backups**
  - Store recent backups in a faster class.
  - Transition older backups to cheaper archival storage.
  - Keep only the number of restore points you actually need.
- **Temporary data**
  - Expire test output, exports, and scratch files quickly.
  - Abort incomplete uploads to avoid hidden waste.

A practical S3 lifecycle rule often combines all three actions:
1. Transition after a short active period.
2. Transition again to a colder archive class.
3. Expire the object after retention ends.

**AWS CLI examples for S3 lifecycle**

You can manage lifecycle rules with the AWS Command Line Interface (CLI). This is useful in labs because it makes the configuration explicit and repeatable.

```bash
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-cost-optimized-bucket \
  --lifecycle-configuration file://lifecycle.json
```

To verify the configuration:

```bash
aws s3api get-bucket-lifecycle-configuration \
  --bucket my-cost-optimized-bucket
```

To remove lifecycle rules:

```bash
aws s3api delete-bucket-lifecycle \
  --bucket my-cost-optimized-bucket
```

If you are using versioned buckets, remember that lifecycle behavior changes:
- Current object versions can transition and expire.
- Noncurrent versions can also be transitioned or expired with separate rules.
- Delete markers may remain unless you explicitly manage them.

**Versioning and noncurrent object cleanup**

Versioning can increase storage spend if old versions accumulate. This is especially common in buckets used for application uploads, static website assets, or frequent overwrites.

Lifecycle rules can target noncurrent versions so that old data does not remain forever.

Example pattern:
- Keep current versions in a standard class.
- Transition noncurrent versions after 30 days.
- Expire noncurrent versions after 90 days.
- Remove expired delete markers when appropriate.

This is useful when you need rollback capability for a limited time but do not want to pay indefinitely for historical versions.

**Compute scaling to reduce spend**

Compute cost often drops when you replace fixed capacity with elastic scaling. The right scaling method depends on the service and the workload pattern.

For EC2-based workloads, common options include:
- **Auto Scaling groups (ASGs)** for horizontal scaling.
- **Target tracking scaling** to maintain a metric such as average CPU utilization.
- **Step scaling** for larger changes based on thresholds.
- **Scheduled scaling** for predictable traffic patterns.
- **Instance type right-sizing** to reduce overprovisioning.

For example, if traffic spikes during business hours and falls overnight, you can use scheduled scaling to increase desired capacity before the peak and reduce it afterward.

```bash
aws autoscaling update-auto-scaling-group \
  --auto-scaling-group-name web-asg \
  --min-size 1 \
  --max-size 6 \
  --desired-capacity 2
```

A target tracking policy can keep utilization near a chosen level:

```json
{
  "TargetValue": 50.0,
  "PredefinedMetricSpecification": {
    "PredefinedMetricType": "ASGAverageCPUUtilization"
  },
  "DisableScaleIn": false
}
```

- **Scale out** adds instances when demand rises.
- **Scale in** removes instances when demand falls.
- **Cooldowns** help prevent rapid oscillation.
- **Health checks** ensure unhealthy instances are replaced automatically.

**Why scaling saves money**

Scaling reduces waste by aligning capacity with demand:
- You avoid paying for peak capacity all day when peaks are short.
- You can use smaller baseline capacity and burst only when needed.
- You can replace manual overprovisioning with policy-driven elasticity.

This is especially effective when the workload is stateless or can externalize state to managed services such as Amazon RDS, Amazon DynamoDB, or Amazon ElastiCache.

**Launch templates and right-sizing**

An Auto Scaling group is only as efficient as the instance configuration behind it. Launch templates help you standardize the instance type, AMI, security groups, and user data so that scaling uses the intended cost profile.

Practical right-sizing tips:
- Start with the smallest instance that meets baseline performance.
- Use CloudWatch metrics to identify CPU, memory, and network bottlenecks.
- Prefer newer generation instances when they offer better price-performance.
- Remove unused EBS volumes and reduce oversized root volumes where possible.

Example launch template snippet:

```json
{
  "LaunchTemplateName": "web-template",
  "VersionDescription": "cost-optimized",
  "LaunchTemplateData": {
    "InstanceType": "t3.small",
    "ImageId": "ami-0123456789abcdef0",
    "Monitoring": {
      "Enabled": true
    }
  }
}
```

**Scaling signals and CloudWatch**

Amazon CloudWatch provides the metrics that drive scaling decisions. If the metric does not reflect real load, scaling can become expensive or ineffective.

Useful signals include:
- `CPUUtilization` for general compute pressure.
- `NetworkIn` and `NetworkOut` for traffic-heavy workloads.
- `RequestCountPerTarget` for Application Load Balancer target groups.
- Custom application metrics for queue depth, latency, or active sessions.

For queue-based workloads, scaling on queue depth is often better than scaling on CPU because it reflects actual backlog.

Example alarm for high CPU:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name high-cpu-web-asg \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 300 \
  --threshold 70 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=AutoScalingGroupName,Value=web-asg \
  --evaluation-periods 2
```

**Scheduled scaling versus dynamic scaling**

Use the right scaling mode for the traffic pattern.

- **Scheduled scaling**
  - Best for predictable daily or weekly patterns.
  - Reduces cost by pre-scaling before known peaks.
  - Works well for office-hour traffic or batch windows.
- **Dynamic scaling**
  - Best for unpredictable or bursty demand.
  - Responds to real-time metrics.
  - Prevents underprovisioning during unexpected spikes.

In many environments, you combine both:
- Scheduled scaling sets a sensible baseline.
- Dynamic scaling handles unexpected variation.

**Practical lab workflow**

A typical cost-reduction lab follows a simple sequence:
1. Identify the expensive resource or pattern.
2. Decide whether the waste is in storage, compute, or both.
3. Apply lifecycle rules to data that no longer needs premium storage.
4. Apply scaling policies to compute that is overprovisioned.
5. Verify the effect with metrics and billing visibility.

For S3, you usually confirm:
- Objects transition to the intended class.
- Expired objects disappear after the retention period.
- Incomplete multipart uploads are cleaned up.

For Auto Scaling, you usually confirm:
- Desired capacity changes as expected.
- Instances launch and terminate correctly.
- Metrics remain within acceptable thresholds.

**Troubleshooting and validation**

If spend does not drop as expected, check the following:
- The lifecycle rule is enabled and scoped to the correct prefix or tags.
- The object age has actually reached the transition threshold.
- The storage class supports the intended access pattern.
- Versioned objects are not accumulating unexpectedly.
- The Auto Scaling policy is attached to the correct group.
- CloudWatch metrics are accurate and not delayed.
- The workload is not pinned by a hidden dependency that prevents scale-in.

A common mistake is to reduce compute too aggressively and then create performance issues that increase retry traffic or operational overhead. Cost optimization should preserve service quality.

> Do not optimize only for the lowest monthly bill. If a change increases latency, failure rate, or recovery time, the total cost of ownership can rise.

**Exam focus**

For SAA-C03, you should recognize when AWS-managed automation is the best answer:
- Use S3 lifecycle policies for automatic storage class transitions and expiration.
- Use Auto Scaling for elastic compute capacity.
- Use CloudWatch metrics and alarms to drive dynamic scaling.
- Use scheduled scaling for predictable demand.
- Use versioning-aware lifecycle rules when old object versions are contributing to cost.

A strong solution usually combines:
- A cheaper storage class for older data.
- Automatic deletion of data that no longer has business value.
- Elastic compute that scales with demand instead of staying fixed at peak.

**Quick reference**

- **S3 lifecycle**: automate transitions, expirations, and multipart cleanup.
- **Storage classes**: choose based on access frequency, retrieval speed, and durability needs.
- **Auto Scaling**: reduce compute waste by matching capacity to demand.
- **CloudWatch**: provide the metrics that trigger scaling.
- **Right-sizing**: remove excess capacity before adding more automation.
