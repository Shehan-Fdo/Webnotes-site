## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Cost estimation workflow**

Estimating AWS cost is a design skill, not just a billing task. You translate an architecture into measurable inputs such as instance hours, storage GB-month, request counts, data transfer, and managed service usage. The goal is to compare design options early, before deployment, and to identify the main cost drivers.

A practical estimate usually starts with a simple architecture diagram and a set of assumptions:

- Region
- Availability requirements
- Traffic profile
- Storage growth
- Backup retention
- Data transfer patterns
- Instance family and size
- Managed service tiers

For exam scenarios, you should focus on the biggest cost contributors first. Small items like a few extra API calls rarely matter compared with compute, NAT Gateway usage, or cross-AZ data transfer.

**Core AWS pricing tools**

AWS provides several tools that help you estimate and validate cost. Use them together rather than relying on one source.

- **AWS Pricing Calculator** is the main planning tool for estimating monthly cost before deployment.
- **AWS Cost Explorer** helps you analyze actual spend after resources exist.
- **AWS Budgets** lets you set thresholds and alerts for expected or actual spend.
- **AWS Cost and Usage Report (CUR)** provides detailed line-item billing data for deeper analysis.
- **AWS Trusted Advisor** can highlight underused or idle resources that increase cost.

A common exam pattern is to ask which tool you should use for a pre-deployment estimate. The answer is usually the **AWS Pricing Calculator**.

**How to estimate a sample architecture**

When you estimate a sample architecture, break the design into service components and estimate each one separately. Then combine the results.

1. Identify all services in the architecture.
2. Determine the usage model for each service.
3. Apply the correct pricing dimensions.
4. Add supporting services such as logging, monitoring, and networking.
5. Include data transfer and backup costs.
6. Review assumptions and add a buffer for uncertainty.

A simple web application architecture might include:

- Amazon EC2 for application servers
- Amazon RDS for the database
- Amazon S3 for static assets and backups
- Elastic Load Balancing for traffic distribution
- Amazon CloudWatch for logs and metrics
- Amazon Route 53 for DNS
- NAT Gateway for private subnet internet access

Each of these services has different billing dimensions, so you must estimate them separately.

**Compute cost drivers**

Compute is often the easiest place to start because the pricing model is familiar.

- **Amazon EC2** is usually billed by instance type, operating system, tenancy, and running time.
- **Amazon EBS** is billed by provisioned storage, provisioned IOPS, and snapshot storage.
- **AWS Lambda** is billed by number of requests and compute duration.
- **Amazon ECS** and **Amazon EKS** may include both infrastructure cost and control plane or cluster-related charges, depending on the deployment model.

For EC2-based estimates, you should account for:

- Number of instances
- Instance size
- Hours per month
- Purchase option such as On-Demand, Reserved Instances, or Savings Plans
- Operating system licensing differences
- Auto Scaling behavior

Example EC2 estimate inputs:

```text
2 x t3.medium Linux instances
730 hours per month
On-Demand pricing
```

If the architecture uses Auto Scaling, estimate average and peak capacity separately. A design that runs two instances most of the time but scales to six during business hours should not be priced as if it always runs six.

**Storage and database cost drivers**

Storage and database services often have hidden cost components beyond the base resource.

- **Amazon S3** is billed by storage class, storage amount, requests, lifecycle transitions, and data transfer.
- **Amazon EBS** adds cost for volume type, size, and snapshots.
- **Amazon RDS** includes instance hours, storage, I/O, backup retention, and Multi-AZ replication overhead.
- **Amazon Aurora** may have separate charges for instance capacity, storage, I/O, and replicas.
- **Amazon DynamoDB** can be billed by provisioned capacity or on-demand request units, plus storage and optional features.

For RDS, remember that Multi-AZ is not just a failover feature; it usually increases cost because AWS maintains synchronous standby infrastructure. That tradeoff is often justified for availability, but it should be reflected in the estimate.

For S3, the storage class matters:

- **S3 Standard** is best for frequent access.
- **S3 Intelligent-Tiering** can reduce cost when access patterns are unknown or change over time.
- **S3 Glacier** classes are for archival data with lower storage cost and higher retrieval tradeoffs.

**Networking cost drivers**

Networking is a common source of surprise charges. In many architectures, the network line items are small individually but significant in aggregate.

- **Elastic Load Balancing** is billed by load balancer hours and capacity units or processed bytes, depending on the type.
- **NAT Gateway** is billed by hourly usage and data processed.
- **Data transfer out to the internet** is usually charged.
- **Cross-AZ data transfer** can add cost when resources communicate across Availability Zones.
- **VPC endpoints** can reduce NAT Gateway usage, but they may have their own hourly and data processing charges.

A frequent design mistake is placing private subnets behind a NAT Gateway for all outbound traffic without considering the cost of large data transfers. If your workload downloads updates, pulls container images, or sends large amounts of telemetry, NAT Gateway charges can become material.

Example network cost considerations:

- Private EC2 instances in two AZs
- NAT Gateway in each AZ for resilience
- Application traffic through an Application Load Balancer
- S3 access through a gateway endpoint to avoid NAT charges

**Example architecture estimate**

Consider a sample three-tier application:

- 2 application servers on EC2
- 1 database on RDS Multi-AZ
- 100 GB of S3 storage
- 1 Application Load Balancer
- 1 NAT Gateway
- CloudWatch logs and metrics
- Moderate internet traffic

A rough estimate process looks like this:

- EC2: instance hours for two servers
- RDS: primary plus standby cost for Multi-AZ
- S3: storage plus request charges
- ALB: hourly cost plus load balancer traffic
- NAT Gateway: hourly cost plus processed data
- CloudWatch: log ingestion and retention
- Data transfer: internet egress and any cross-AZ traffic

This approach helps you identify whether the database, network, or compute layer dominates the bill. In many real designs, the NAT Gateway and database are more expensive than the application servers themselves.

**Pricing Calculator usage pattern**

The AWS Pricing Calculator is useful when you need a structured estimate with assumptions documented. You typically add each service, configure usage, and review the monthly total.

Common inputs include:

- Region
- Usage hours
- Storage amount
- Request volume
- Data transfer volume
- Purchase model
- Backup retention
- Number of environments

A practical workflow:

1. Open the AWS Pricing Calculator.
2. Add each AWS service used by the architecture.
3. Enter realistic monthly usage assumptions.
4. Compare On-Demand and commitment-based pricing.
5. Review the total and export or share the estimate.

Example CLI-style thinking for assumptions:

```text
EC2: 2 instances x 730 hours
RDS: 1 primary + 1 standby
S3: 100 GB standard storage
ALB: 1 load balancer, moderate traffic
NAT: 1 gateway, 200 GB processed data
```

The calculator is not a billing guarantee. It is a planning tool, so your estimate is only as good as your assumptions.

**Cost optimization considerations**

Cost estimation and cost optimization are closely related. When you estimate, you should also ask whether the design is cost-efficient for the workload.

- Use **Savings Plans** or Reserved Instances for steady-state compute.
- Use **Auto Scaling** to match capacity to demand.
- Use **S3 lifecycle policies** to move older data to cheaper storage classes.
- Use **VPC endpoints** to reduce NAT Gateway traffic where appropriate.
- Use **CloudWatch log retention** policies to avoid indefinite log growth.
- Use **right-sizing** to avoid overprovisioned instances.

A good estimate often includes both the current design and a lower-cost alternative. For example, you might compare:

| Component | Higher-cost option | Lower-cost option |
|---|---|---|
| Compute | On-Demand EC2 | Savings Plans for steady usage |
| Storage | S3 Standard | S3 Intelligent-Tiering or Glacier |
| Database | RDS Multi-AZ | Single-AZ for noncritical dev/test |
| Networking | NAT Gateway for all egress | VPC endpoints for AWS service access |

**Common exam and lab pitfalls**

Be careful with assumptions that can distort the estimate.

- Ignoring data transfer charges
- Forgetting NAT Gateway hourly and per-GB costs
- Pricing only the primary database and omitting standby resources
- Treating Multi-AZ as free failover
- Ignoring backup storage and snapshot retention
- Assuming peak capacity all month instead of average usage
- Forgetting logs, metrics, and monitoring overhead
- Using the wrong Region, which changes pricing

> If the question asks for the most accurate pre-deployment estimate, use the AWS Pricing Calculator and include all major service dimensions, especially compute, storage, and data transfer.

**Practical command examples for validation**

After deployment, you can validate assumptions with AWS CLI commands and service consoles. These commands do not estimate cost directly, but they help you inspect the resources that drive cost.

```bash
aws ec2 describe-instances --filters "Name=instance-state-name,Values=running"
aws ec2 describe-volumes
aws rds describe-db-instances
aws s3 ls
aws elbv2 describe-load-balancers
aws ec2 describe-nat-gateways
```

You can also inspect usage patterns in CloudWatch:

```bash
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-1234567890abcdef0 \
  --start-time 2026-04-01T00:00:00Z \
  --end-time 2026-04-26T00:00:00Z \
  --period 3600 \
  --statistics Average
```

These checks help you compare actual utilization against the assumptions used in the estimate.

**When to use each billing-related service**

Use the right AWS billing tool for the job:

- **Pricing Calculator** for planning and design-time estimates
- **Cost Explorer** for historical spend analysis
- **Budgets** for alerts and guardrails
- **CUR** for detailed billing analytics and chargeback
- **Trusted Advisor** for identifying waste and idle resources

A simple memory aid:

- Estimate before deployment with the calculator.
- Analyze after deployment with Cost Explorer and CUR.
- Control spend with Budgets.
- Reduce waste with Trusted Advisor.

**Key takeaways**

Cost estimation is about decomposing an architecture into billable units and applying realistic usage assumptions. The biggest mistakes come from ignoring networking, backup, and standby resources. For exam scenarios, choose the AWS Pricing Calculator when you need a pre-deployment estimate, and remember that service-specific pricing dimensions matter more than the headline service name.
