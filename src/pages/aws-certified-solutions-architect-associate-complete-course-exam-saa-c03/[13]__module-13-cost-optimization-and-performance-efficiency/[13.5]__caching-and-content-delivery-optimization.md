## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Caching and content delivery goals**

Amazon CloudFront is AWS’s content delivery network (CDN). You use it to reduce latency, offload origin traffic, and improve availability for static and dynamic content. For exam scenarios, focus on how CloudFront decides what to cache, how it routes requests to origins, and how you tune delivery for performance, cost, and security.

CloudFront sits in front of origins such as Amazon S3, Application Load Balancer (ALB), API Gateway, or custom HTTP servers. It caches content at edge locations and can also use regional edge caches and <mark>Origin Shield</mark> to reduce repeated origin fetches.

- **Latency reduction** comes from serving content closer to users.
- **Origin offload** reduces load on S3, EC2, ALB, or on-premises servers.
- **Availability improvement** comes from distributed caching and failover patterns.
- **Security controls** include HTTPS, signed URLs, signed cookies, field-level encryption, and origin access controls.

**CloudFront request flow and cache hierarchy**

A viewer request first reaches the nearest CloudFront edge location. CloudFront checks whether the object is in cache and whether it is still fresh. If the object is missing or stale, CloudFront forwards the request to the origin according to the matching cache behavior and origin request policy.

A useful mental model is:

1. Viewer request arrives at the edge.
2. CloudFront matches a cache behavior by path pattern.
3. CloudFront checks cache key components.
4. CloudFront serves from cache or fetches from origin.
5. CloudFront stores the response according to TTL rules and cache policy.

CloudFront can also use a regional edge cache between the edge location and the origin. This improves hit ratio for content requested across multiple edge locations in the same region. <mark>Origin Shield</mark> adds another centralized caching layer to further reduce origin requests, which is especially useful for popular objects, bursty traffic, and expensive origins.

**Cache behaviors, default behavior, and path precedence**

Cache behaviors are one of the most exam-relevant CloudFront topics. Each behavior defines how CloudFront handles a request for a path pattern, including which origin to use, what to cache, and which headers, cookies, and query strings affect the cache key.

CloudFront evaluates behaviors by path pattern, and the most specific matching pattern wins. The default cache behavior is the fallback when no ordered behavior matches.

- **Ordered cache behaviors** are evaluated by path pattern specificity.
- **Default cache behavior** applies when no other pattern matches.
- **Path pattern precedence** matters when multiple patterns could match the same request.

Example precedence:

- `/images/*.jpg` matches `https://example.com/images/cat.jpg`
- `/images/*` matches the same request, but is less specific
- `Default (*)` applies only if no ordered behavior matches

A practical configuration often looks like this:

| Path pattern | Typical use | Notes |
|---|---|---|
| `/static/*` | Versioned assets | Long TTL, minimal forwarding |
| `/api/*` | Dynamic API responses | Short TTL or no cache |
| `/private/*` | Protected content | Signed URLs/cookies, strict controls |
| Default `*` | Catch-all | Safe fallback behavior |

> If two behaviors could match, choose the more specific path pattern. Exam questions often test whether you know that the default behavior is not the “first” behavior; it is the fallback.

**Cache key design and cache policies**

CloudFront caching is driven by the cache key. The cache key determines whether two requests are considered the same object for caching purposes. Modern CloudFront uses cache policies to control this behavior.

A cache policy can include or exclude:

- **Headers**
- **Cookies**
- **Query strings**
- **Compression support**
- **TTL settings**

The goal is to include only request attributes that truly change the response. Overly broad cache keys reduce hit ratio; overly narrow keys can serve the wrong content.

Common patterns:

- Static assets:
  - Exclude all query strings, cookies, and headers except those required for versioning.
  - Use long TTLs.
- Personalized content:
  - Include only the minimum cookie or header values needed.
  - Consider bypassing cache if content is highly user-specific.
- API responses:
  - Include query strings that change the response, such as `page`, `sort`, or `lang`.
  - Avoid forwarding unnecessary headers.

Example cache policy ideas:

```json
{
  "Name": "StaticAssetsPolicy",
  "DefaultTTL": 86400,
  "MaxTTL": 31536000,
  "MinTTL": 0,
  "ParametersInCacheKeyAndForwardedToOrigin": {
    "HeadersConfig": {
      "HeaderBehavior": "none"
    },
    "CookiesConfig": {
      "CookieBehavior": "none"
    },
    "QueryStringsConfig": {
      "QueryStringBehavior": "none"
    }
  },
  "EnableAcceptEncodingGzip": true,
  "EnableAcceptEncodingBrotli": true
}
```

For dynamic content, you might include only a few query strings:

```json
{
  "Name": "ApiPolicy",
  "DefaultTTL": 30,
  "MaxTTL": 300,
  "MinTTL": 0,
  "ParametersInCacheKeyAndForwardedToOrigin": {
    "HeadersConfig": {
      "HeaderBehavior": "whitelist",
      "Headers": {
        "Quantity": 1,
        "Items": ["Accept-Language"]
      }
    },
    "CookiesConfig": {
      "CookieBehavior": "none"
    },
    "QueryStringsConfig": {
      "QueryStringBehavior": "whitelist",
      "QueryStrings": {
        "Quantity": 2,
        "Items": ["page", "sort"]
      }
    }
  }
}
```

**Origin request policies and when to forward data**

An origin request policy controls what CloudFront forwards to the origin. This is separate from the cache key. A value can be forwarded to the origin without being part of the cache key, which is useful when the origin needs context that should not fragment the cache.

Use origin request policies to forward:

- Authentication-related headers when the origin needs them
- Selected cookies for app logic
- Query strings required by the origin
- Device or language headers when the origin generates variants

Use them carefully. Forwarding too much data increases origin load and reduces cache efficiency.

Typical exam distinction:

- **Cache policy** affects what CloudFront stores and how it identifies cache entries.
- **Origin request policy** affects what CloudFront sends upstream.

Example scenario:

- You want CloudFront to cache `/products/123` once.
- The origin needs `Authorization` for logging or authorization checks.
- If you include `Authorization` in the cache key, every user may get a separate cache entry.
- If you forward `Authorization` only in the origin request policy, CloudFront can still cache the response if the content is truly public and the origin does not vary the body by user.

**TTL precedence and browser cache control**

CloudFront TTL behavior is a common exam trap. There are multiple layers of caching:

- Browser cache
- CloudFront edge cache
- Regional edge cache
- Origin cache, if present

CloudFront decides its own freshness using origin headers and cache policy TTL settings. Browser caching is separate and controlled by response headers such as `Cache-Control`, `Expires`, and `ETag`.

Key precedence concepts:

- Origin `Cache-Control: max-age` can influence CloudFront caching.
- Cache policy `MinTTL`, `DefaultTTL`, and `MaxTTL` can override or constrain origin behavior.
- `MinTTL` can force CloudFront to cache even if the origin says not to cache, which is risky for personalized content.
- `MaxTTL` caps how long CloudFront keeps an object fresh.
- Browser cache directives do not automatically equal CloudFront TTLs.

Practical interpretation:

- If the origin sends `Cache-Control: max-age=60`, CloudFront may cache for 60 seconds unless the cache policy changes that result.
- If the cache policy `MinTTL` is higher than the origin’s no-cache directive, CloudFront can still cache longer than the origin intended.
- If the origin sends `Cache-Control: no-cache`, CloudFront may still cache if the policy allows it, but it will revalidate based on freshness rules.

Example response headers:

```http
Cache-Control: public, max-age=3600
ETag: "a1b2c3d4"
```

This means browsers can cache for an hour, and CloudFront can also cache based on its policy. If you need frequent updates, lower the TTL or use invalidations.

**Compression and content optimization**

CloudFront can serve compressed objects to reduce transfer size and improve performance. The main compression methods are:

- **Gzip**
- **Brotli**

CloudFront can compress text-based content such as:

- HTML
- CSS
- JavaScript
- JSON
- SVG

Compression is most useful for text assets and APIs. It is not typically useful for already compressed formats such as JPEG, PNG, MP4, or ZIP.

Important points:

- CloudFront can compress on the fly or serve precompressed content depending on configuration and object type.
- Compression improves delivery speed and reduces bandwidth.
- Compression works best when the cache key varies by `Accept-Encoding` so CloudFront can serve the correct compressed variant.

Example behavior:

- Browser sends `Accept-Encoding: br, gzip`
- CloudFront returns Brotli if supported and cached
- Otherwise CloudFront returns Gzip or uncompressed content

A practical origin setup for static assets often includes versioned filenames and compression:

```bash
aws s3 cp app.js s3://my-bucket/assets/app.js \
  --content-type application/javascript \
  --cache-control "public,max-age=31536000,immutable"
```

For best results, combine:

- Long-lived cache headers for versioned assets
- Compression for text content
- File fingerprinting such as `app.8f3a1c.js`

**Origin Shield and origin load reduction**

Origin Shield is an additional caching layer that sits between CloudFront edge locations and the origin. It helps consolidate cache misses so that multiple edge locations do not all hit the origin for the same object.

Use Origin Shield when:

- Your origin is expensive or rate-limited
- Traffic is global and bursty
- You want to improve cache hit ratio
- You need another layer of protection against origin overload

Why it matters:

- It reduces duplicate origin fetches
- It can improve cache efficiency across regions
- It is especially helpful for popular content with uneven geographic demand

Exam framing often compares it to a “centralized regional cache” that reduces origin load.

**Private content delivery and signed access**

CloudFront supports private content delivery using signed URLs and signed cookies. These are used when you want to restrict access to content without exposing the origin directly.

Use signed URLs when:

- You need access to a single object
- You want a link that expires
- You are distributing one-off downloads or media files

Use signed cookies when:

- You need access to multiple protected objects
- You want to authorize a session for a path prefix
- You are serving a protected site with many assets

Signed access includes an expiration time, and the signature is validated by CloudFront before the request reaches the origin.

Key exam points:

- <mark>Expiration</mark> limits how long the URL or cookie is valid.
- Short expirations reduce replay risk.
- Key rotation is important for operational security.
- CloudFront supports **trusted key groups** for modern key management.
- Legacy **trusted signers** are older and tied to AWS account root or IAM users; key groups are the preferred approach in most current designs.

Practical guidance:

- Use key groups for new implementations.
- Rotate public/private key pairs periodically.
- Keep expiration short for sensitive downloads.
- Use signed cookies for session-based access to many objects.

Example signed URL concept:

```text
https://d111111abcdef8.cloudfront.net/private/report.pdf?Expires=1710000000&Signature=...&Key-Pair-Id=K1234567890
```

**Field-level encryption and sensitive data handling**

Field-level encryption lets CloudFront encrypt specific fields in a request before forwarding them to the origin. This is useful when you want to protect sensitive data such as:

- Credit card numbers
- Personal identifiers
- Health-related fields
- Other regulated form inputs

This feature is different from TLS. TLS protects data in transit between the viewer and CloudFront, and between CloudFront and the origin if configured. Field-level encryption protects selected fields end-to-end through CloudFront so that only the origin can decrypt them.

Use it when:

- You need to minimize exposure of sensitive form fields
- The origin must receive plaintext for processing
- You want to reduce the number of systems that can see the raw data

**Lambda@Edge and CloudFront Functions**

CloudFront can run lightweight code at the edge to modify requests and responses.

Use **CloudFront Functions** for:

- Very low-latency viewer request/response logic
- Simple header manipulation
- URL rewrites
- Redirects
- Basic authentication or routing decisions

Use **Lambda@Edge** for:

- More complex request/response processing
- Origin request/response manipulation
- Dynamic personalization
- Advanced header or body-related logic, within service limits

Typical exam distinctions:

- CloudFront Functions are faster and cheaper for simple edge logic.
- Lambda@Edge is more flexible for complex transformations.
- Both can help with cache optimization by normalizing requests before cache lookup.

Example use cases:

- Remove tracking query parameters before cache key evaluation
- Redirect `http` to `https`
- Rewrite `/` to `/index.html`
- Add security headers on responses
- Route mobile users to a different origin path

Example CloudFront Function snippet:

```javascript
function handler(event) {
  var request = event.request;
  request.uri = request.uri.replace(/\/+$/, '');
  if (request.uri === '') {
    request.uri = '/index.html';
  }
  return request;
}
```

**Practical cache and origin policy examples**

A common static website setup uses:

- S3 origin
- Default behavior for all content
- Separate behavior for `/api/*`
- Long TTL for versioned assets
- No cookies or query strings for static files
- Compression enabled

Example topology:

```text
Viewer -> CloudFront -> S3 bucket
                    -> ALB for /api/*
```

Example behavior design:

- `/assets/*`
  - Cache policy: long TTL
  - Origin request policy: none
  - Compression: enabled
- `/api/*`
  - Cache policy: short TTL or disabled
  - Origin request policy: forward `Authorization`, selected query strings
  - Compression: enabled for JSON
- Default `*`
  - Fallback behavior
  - Conservative forwarding

Example AWS CLI to create a distribution is verbose, but the important exam idea is that behaviors are attached to origins and policies:

```bash
aws cloudfront create-distribution \
  --distribution-config file://distribution.json
```

A simplified `distribution.json` concept:

```json
{
  "Origins": {
    "Quantity": 1,
    "Items": [
      {
        "Id": "s3-origin",
        "DomainName": "my-bucket.s3.amazonaws.com",
        "S3OriginConfig": {
          "OriginAccessControlId": "E1234567890"
        }
      }
    ]
  },
  "DefaultCacheBehavior": {
    "TargetOriginId": "s3-origin",
    "ViewerProtocolPolicy": "redirect-to-https",
    "CachePolicyId": "658327ea-f89d-4fab-a63d-7e88639e58f6"
  }
}
```

**Invalidations and cache refresh strategies**

CloudFront invalidations remove objects from edge caches before TTL expiry. Use them when you need immediate propagation of changed content.

Use invalidations for:

- Incorrectly deployed assets
- Emergency content changes
- Non-versioned files such as `index.html`

Avoid invalidations for:

- Frequently changing versioned assets
- Large-scale routine updates

Better pattern:

- Version static assets in filenames
- Keep long TTLs on immutable files
- Invalidate only the small set of entry-point files

Example:

```bash
aws cloudfront create-invalidation \
  --distribution-id E1234567890 \
  --paths "/index.html" "/app.js"
```

**Performance and design tradeoffs**

CloudFront optimization is usually about balancing cache hit ratio, freshness, and security.

- Increase hit ratio by reducing unnecessary cache key variation.
- Reduce origin load by using Origin Shield and long TTLs for immutable content.
- Improve freshness by shortening TTLs or using invalidations.
- Protect sensitive content with signed URLs, signed cookies, and field-level encryption.
- Normalize requests with CloudFront Functions or Lambda@Edge to avoid cache fragmentation.

A strong exam answer usually identifies the best combination of:

- Path-based cache behaviors
- Correct cache policy and origin request policy
- Appropriate TTLs
- Compression
- Secure access controls
- Edge logic only when needed

**Common exam scenarios**

- A static website needs global low-latency delivery:
  - Use CloudFront with S3 origin, long TTLs, compression, and versioned assets.
- An API needs selective caching:
  - Cache only safe responses, forward required query strings and headers, and keep TTL short.
- A private download portal needs expiring access:
  - Use signed URLs or signed cookies with trusted key groups.
- A sensitive form must protect specific fields:
  - Use field-level encryption.
- Origin is overloaded during traffic spikes:
  - Add Origin Shield, tighten cache keys, and increase TTLs where safe.
- Requests need URL normalization before caching:
  - Use CloudFront Functions or Lambda@Edge.

**Quick exam checklist**

- **Cache behavior precedence**: ordered behaviors beat the default behavior when path patterns match.
- **Cache key**: include only values that change the response.
- **Origin request policy**: forward only what the origin needs.
- **TTL precedence**: origin headers, cache policy settings, and browser cache are related but not identical.
- **Compression**: enable Brotli/Gzip for text assets.
- **Origin Shield**: use it to reduce origin load and improve hit ratio.
- **Private content**: prefer signed URLs/cookies with trusted key groups.
- **Field-level encryption**: protect sensitive request fields.
- **Edge logic**: use CloudFront Functions for simple rewrites and Lambda@Edge for more complex processing.
