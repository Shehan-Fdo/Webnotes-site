## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Overview**

Amazon RDS offers two commonly tested patterns for relational database resilience and scale: **Multi-AZ** and **Read Replicas**. They solve different problems, even though both can involve additional database instances. Multi-AZ focuses on **high availability (HA)** and automatic failover. Read Replicas focus on **read scaling** and, in some cases, disaster recovery support.

A good exam answer usually depends on the primary goal:

- Use **Multi-AZ** when you need automatic failover and minimal downtime.
- Use **Read Replicas** when you need to offload read traffic or distribute read-heavy workloads.
- Use both when you need availability plus read scale.

**Core distinction**

Multi-AZ creates a synchronous standby in another Availability Zone (AZ). The standby is not used for reads. Read Replicas create asynchronous copies that can serve read traffic, but they are not a substitute for HA because replication lag can occur.

| Feature | Multi-AZ | Read Replica |
|---|---|---|
| Primary purpose | High availability | Read scaling |
| Replication type | Synchronous | Asynchronous |
| Read traffic on secondary | No | Yes |
| Automatic failover | Yes | No |
| Typical use case | Production resilience | Read-heavy applications |
| Cross-Region support | Some engines support it | Yes, commonly used for DR and global reads |

**Multi-AZ architecture**

Multi-AZ deployments keep a primary DB instance in one AZ and a standby in another AZ. AWS continuously replicates data to the standby using synchronous replication, so the standby stays closely aligned with the primary.

This design matters because it protects you from:

- AZ failure
- Storage failure on the primary instance
- Planned maintenance with reduced downtime

The standby is passive. You do not connect applications to it for normal reads. AWS handles failover by promoting the standby and updating the database endpoint.

> Multi-AZ is about <mark>availability</mark>, not read performance.

**How failover works**

When the primary instance becomes unavailable, AWS automatically fails over to the standby. The database endpoint remains the same from the application’s perspective, but the underlying target changes.

Typical failover behavior:

- AWS detects the failure.
- The standby is promoted to primary.
- DNS and endpoint mappings are updated.
- The application reconnects after a short interruption.

Failover is usually faster and simpler than rebuilding a database manually, but you should still design your application to retry connections.

**When to choose Multi-AZ**

Use Multi-AZ when you need:

- Automatic failover for production databases
- Better resilience against AZ outages
- Lower recovery time objective (RTO) than manual recovery
- Managed maintenance with reduced service interruption

Common examples:

- Customer-facing transactional systems
- Payment processing databases
- Internal systems that cannot tolerate long outages

**What Multi-AZ does not do**

Multi-AZ does not:

- Improve read throughput
- Provide a secondary endpoint for read traffic
- Replace backups
- Eliminate the need for application retries

It is a resilience feature, not a scaling feature.

**Read Replicas architecture**

Read Replicas are separate database instances that receive changes from the source database asynchronously. Because replication is asynchronous, the replica may lag behind the primary by a small amount of time.

This makes Read Replicas useful for:

- Offloading read queries from the primary
- Supporting reporting and analytics workloads
- Serving geographically closer reads in another Region
- Creating a promoted copy for disaster recovery scenarios

A common topology is:

```text
Application writes -> Primary DB
Application reads  -> Read Replica(s)
```

You can also use multiple replicas:

```text
                 -> Read Replica 1
Primary DB  -----> Read Replica 2
                 -> Read Replica 3
```

**When to choose Read Replicas**

Use Read Replicas when you need:

- More read capacity
- Lower latency for read-only users or services
- Separation of reporting queries from transactional writes
- A warm copy of data for DR or migration

Typical examples:

- E-commerce product catalog reads
- Analytics dashboards
- Mobile apps with heavy read traffic
- Cross-Region read access for global users

**Replication lag**

Because Read Replicas use asynchronous replication, they can lag behind the primary. This is the main tradeoff.

Implications of lag:

- A user may not immediately see a just-written record on a replica.
- Reporting queries may run on slightly stale data.
- You should not use replicas for strongly consistent read-after-write workflows.

If your application requires immediate consistency after writes, read from the primary or redesign the workflow.

**Promotion and failover differences**

Read Replicas can be promoted to standalone DB instances. This is useful for migration or recovery, but it is not the same as automatic failover.

Important differences:

- Promotion is typically manual or operationally controlled.
- The promoted replica becomes a new primary.
- Existing replication relationships may change or stop.
- Applications may need endpoint updates if they were pointed at the replica.

By contrast, Multi-AZ failover is automatic and intended to preserve the same database endpoint.

**Multi-AZ vs Read Replica in exam terms**

If the question says:

- “Need automatic failover” → choose **Multi-AZ**
- “Need to scale reads” → choose **Read Replica**
- “Need disaster recovery and read scaling” → consider **both**
- “Need a standby that is not used for reads” → choose **Multi-AZ**
- “Need a copy in another Region for reporting” → choose **Read Replica**

**Practical AWS CLI examples**

Create a Multi-AZ RDS instance:

```bash
aws rds create-db-instance \
  --db-instance-identifier prod-db \
  --db-instance-class db.m6g.large \
  --engine mysql \
  --master-username admin \
  --master-user-password 'ChangeMe123!' \
  --allocated-storage 100 \
  --multi-az
```

Create a Read Replica:

```bash
aws rds create-db-instance-read-replica \
  --db-instance-identifier prod-db-replica-1 \
  --source-db-instance-identifier prod-db
```

Describe replication and failover-related details:

```bash
aws rds describe-db-instances \
  --db-instance-identifier prod-db
```

Look for fields such as:

- `MultiAZ`
- `ReadReplicaSourceDBInstanceIdentifier`
- `DBInstanceStatus`
- `AvailabilityZone`

**Operational considerations**

Multi-AZ and Read Replicas affect operations differently:

- **Multi-AZ**
  - Better for maintenance windows
  - Better for AZ resilience
  - Usually higher cost than single-AZ because you pay for standby capacity
- **Read Replicas**
  - Better for read-heavy workloads
  - Can reduce load on the primary
  - Require monitoring for lag and replica health

You should monitor:

- CPU and memory on the primary
- Read latency on replicas
- Replication lag
- Connection counts
- Failover events

**Common misconceptions**

- A Read Replica is not a high-availability standby.
- Multi-AZ is not a read-scaling solution.
- A standby in Multi-AZ is not normally queryable.
- Read Replicas do not guarantee zero data loss because replication is asynchronous.
- Multi-AZ improves availability, but you still need backups for point-in-time recovery.

**Practical design patterns**

For a typical production web application:

- Use **Multi-AZ** for the primary database.
- Add **Read Replicas** if read traffic grows.
- Keep writes on the primary.
- Route reporting jobs to replicas.
- Use application retries and connection pooling.

Example topology:

```text
Users
  |
Application tier
  |---- writes ----> Primary RDS (Multi-AZ)
  |---- reads  -----> Read Replica 1
  |---- reports ----> Read Replica 2
```

This pattern gives you:

- Availability from Multi-AZ
- Read scaling from replicas
- Separation of transactional and analytical workloads

**Backup and recovery relationship**

Neither feature replaces backups.

- **Multi-AZ** helps with availability and failover.
- **Read Replicas** help with scaling and optional recovery workflows.
- **Automated backups** and snapshots support point-in-time recovery and long-term retention.

If you need to restore to a specific time, use backups or snapshots, not Multi-AZ failover.

**Quick decision guide**

1. Ask whether the main problem is downtime or read load.
2. If downtime is the problem, choose Multi-AZ.
3. If read load is the problem, choose Read Replicas.
4. If both are problems, use Multi-AZ plus Read Replicas.
5. If the workload needs immediate consistency after writes, avoid reading from replicas for that path.

**Exam memory aid**

- **Multi-AZ = failover**
- **Read Replica = scale reads**
- **Synchronous = Multi-AZ**
- **Asynchronous = Read Replica**

**Summary**

Multi-AZ and Read Replicas are often confused because both add database copies, but they solve different architectural problems. Multi-AZ is the right answer for resilience and automatic failover. Read Replicas are the right answer for read scaling and read distribution. In exam scenarios, identify whether the requirement is availability, performance, or both, then choose the feature that matches the primary goal.
