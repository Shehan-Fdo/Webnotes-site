## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**What database selection questions test**

SAA-C03 database questions usually ask you to choose the service that best fits a workload, not the service with the most features. You should identify the access pattern, consistency needs, latency target, operational burden, and migration constraints before selecting a database engine.

The exam often mixes similar options:

- Managed relational workloads point to **Amazon RDS** or **Amazon Aurora**.
- Key-value, document, session, and high-scale low-latency workloads point to **Amazon DynamoDB**.
- Cache-heavy or session-offload patterns point to **Amazon ElastiCache** or **Amazon MemoryDB for Redis**.
- Search-heavy workloads point to **Amazon OpenSearch Service**.
- Analytics and lake-style queries often point to **Amazon S3** with **AWS Glue** and **Amazon Athena** instead of a database.

A good rule is to choose the simplest service that satisfies the workload and operational requirements.

**Relational databases: RDS vs Aurora**

Use **Amazon RDS** when you need a managed relational database and the workload fits a standard engine such as MySQL, PostgreSQL, MariaDB, Oracle, or SQL Server. RDS handles much of the undifferentiated heavy lifting, including automated backups, patching, and managed replication features, but you still configure monitoring, alarms, retention, parameter groups, and failover behavior.

Use **Amazon Aurora** when the question emphasizes cloud-native relational performance, high availability, fast failover, or compatibility with MySQL or PostgreSQL. Aurora is not automatically the best answer for every relational workload; standard RDS can be the better fit when you need a specific engine, licensing model, or simpler migration path.

Aurora comes in two compatibility flavors:

- **Aurora MySQL-compatible** for workloads that expect MySQL behavior and tooling.
- **Aurora PostgreSQL-compatible** for workloads that expect PostgreSQL features and extensions.

Choose based on application compatibility, SQL dialect, extensions, and migration effort. If the app already depends on PostgreSQL-specific features, Aurora PostgreSQL is usually the better fit than Aurora MySQL.

**RDS operational patterns**

RDS is often the right answer when the exam mentions:

- Managed relational storage with familiar engines.
- Lift-and-shift migrations from on-premises databases.
- Standard SQL queries and joins.
- Moderate scaling with read replicas.
- Multi-AZ availability for automatic failover.

RDS read replicas are asynchronous. They help with read scaling and can reduce load on the primary, but they are not a replacement for Multi-AZ failover. A read replica may lag behind the primary, so do not use it when the question requires zero data loss or immediate failover.

> Read replicas are for scaling reads and offloading traffic; Multi-AZ is for high availability and failover.

A simple RDS deployment pattern looks like this:

```text
Primary DB instance
  ├── Multi-AZ standby in another AZ
  ├── Read replica in same Region
  └── Optional cross-Region read replica for disaster recovery
```

Example CLI for creating an RDS read replica:

```bash
aws rds create-db-instance-read-replica \
  --db-instance-identifier appdb-replica-1 \
  --source-db-instance-identifier appdb-primary \
  --db-instance-class db.m6g.large \
  --availability-zone us-east-1b
```

Example CLI for enabling Multi-AZ on an existing instance:

```bash
aws rds modify-db-instance \
  --db-instance-identifier appdb-primary \
  --multi-az \
  --apply-immediately
```

**Aurora architecture and when it wins**

Aurora separates compute from a distributed storage layer and is designed for high availability and fast recovery. It is often a strong answer when the question highlights:

- Faster failover than standard single-instance relational setups.
- High read scaling with Aurora Replicas.
- Storage that grows automatically.
- Better durability and availability characteristics than a basic self-managed database.

Aurora replicas are used for read scaling and failover within a Region. For cross-Region disaster recovery or global low-latency access, Aurora has its own cross-Region replication and global database options. Do not confuse these with standard RDS read replicas.

Aurora Serverless is best interpreted carefully:

- **Aurora Serverless v2** is the current exam-relevant serverless relational option for variable or intermittent workloads.
- **Aurora Serverless v1** had more limitations and is largely superseded.

Use Aurora Serverless v2 when the workload is spiky, unpredictable, or has long idle periods and you want automatic capacity scaling without managing instance sizing as tightly.

**DynamoDB: when the workload is not relational**

Use **Amazon DynamoDB** when the question emphasizes:

- Single-digit millisecond latency at scale.
- Key-value or document access patterns.
- Massive throughput with predictable access by partition key.
- Serverless operations with minimal administration.
- Session storage, user profiles, shopping carts, IoT metadata, or event-driven applications.

DynamoDB is not a relational database. You model data around access patterns, not normalized joins. If the question requires complex joins, ad hoc SQL, or relational constraints, DynamoDB is usually the wrong choice.

A common table design pattern is:

- **Partition key** for the primary access path.
- **Sort key** for grouping and range queries.
- **Secondary indexes** for alternate query patterns.

Example table design:

```text
Table: Orders
PK: customerId
SK: orderDate#orderId
Attributes: status, totalAmount, shippingAddress
```

This design lets you query all orders for a customer and sort them by date. If you need to query by status or fulfillment state, add a secondary index.

Example AWS CLI table creation with a composite key and a GSI:

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions \
      AttributeName=customerId,AttributeType=S \
      AttributeName=orderDate,AttributeType=S \
      AttributeName=status,AttributeType=S \
  --key-schema \
      AttributeName=customerId,KeyType=HASH \
      AttributeName=orderDate,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST \
  --global-secondary-indexes '[
    {
      "IndexName": "StatusIndex",
      "KeySchema": [
        {"AttributeName": "status", "KeyType": "HASH"},
        {"AttributeName": "orderDate", "KeyType": "RANGE"}
      ],
      "Projection": {"ProjectionType": "ALL"}
    }
  ]'
```

DynamoDB features that matter for exam scenarios:

- **TTL** (Time to Live) automatically expires items after a timestamp you store in an attribute.
- **Streams** capture item-level changes for event-driven processing, auditing, or replication workflows.
- **Transactions** provide atomic writes across multiple items or tables when you need all-or-nothing behavior.

Example TTL configuration pattern:

```bash
aws dynamodb update-time-to-live \
  --table-name Sessions \
  --time-to-live-specification "Enabled=true,AttributeName=expiresAt"
```

Example item with TTL:

```json
{
  "sessionId": "abc123",
  "userId": "u-42",
  "expiresAt": 1767225600
}
```

Use DynamoDB TTL for temporary sessions, carts, or ephemeral state. Use Streams when downstream systems need to react to inserts, updates, or deletes. Use transactions when the question requires consistency across multiple writes without partial updates.

DynamoDB consistency details:

- Default reads are eventually consistent.
- Strongly consistent reads are supported on base tables and local secondary indexes.
- Strongly consistent reads are not supported on global secondary indexes, global tables replicas, or streams.

That distinction matters when the question asks for the freshest possible read after a write. If the workload needs immediate read-after-write consistency, make sure the access pattern supports it.

**DynamoDB indexes and access patterns**

Use a **GSI** (Global Secondary Index) when you need to query by a different partition key than the base table. Use an **LSI** (Local Secondary Index) when you need an alternate sort key but want to keep the same partition key.

Example LSI concept:

```text
Base table PK: customerId
Base table SK: orderDate
LSI SK: status
```

Choose carefully:

- **GSI** for flexible alternate query patterns and scale-out reads.
- **LSI** when you need alternate sort ordering within the same partition key and can define it at table creation time.

**Global tables and multi-Region design**

Use **DynamoDB global tables** when the question requires active-active multi-Region access with low-latency local reads and writes. Replication is asynchronous and eventually consistent across Regions.

Do not describe global tables as synchronous cross-Region consistency. They improve availability and regional proximity, but they do not guarantee immediate consistency between Regions.

Use global tables when you need:

- Multi-Region disaster recovery.
- Regional failover with minimal application changes.
- Local reads and writes in multiple Regions.

**Caching and ultra-low-latency choices**

If the workload is primarily caching, session offload, leaderboard storage, or extremely fast in-memory reads, consider **Amazon ElastiCache** or **Amazon MemoryDB for Redis** before choosing a database.

Use **ElastiCache** when you want a managed cache to reduce database load and improve read latency. Common engines are Redis and Memcached.

Use **MemoryDB for Redis** when you need Redis-compatible in-memory performance with durability and a primary database-like role for certain low-latency workloads. It is more durable than a pure cache and can be appropriate when the exam emphasizes persistence plus Redis semantics.

Typical cache pattern:

```text
Application -> Cache -> Database
             -> miss -> RDS or DynamoDB
```

Choose caching when the question says:

- The database is overloaded by repeated reads.
- The same data is requested frequently.
- Session state should be removed from the primary database.
- Millisecond or sub-millisecond response time is the main goal.

**Search workloads**

Use **Amazon OpenSearch Service** when the question is about:

- Full-text search.
- Log analytics and search.
- Faceted search.
- Near real-time indexing and query of documents.

OpenSearch is not a general-purpose transactional database. If the workload is search-centric, it is usually a better answer than forcing the data into RDS or DynamoDB.

Example search-oriented architecture:

```text
Application logs -> Kinesis / Firehose -> OpenSearch
Users -> search queries -> OpenSearch dashboards or API
```

**Analytics and data lake patterns**

Do not use a database service when the workload is primarily analytical over large files or historical datasets and the question points to a data lake pattern. In those cases, use:

- **Amazon S3** for durable object storage.
- **AWS Glue** for cataloging and ETL (Extract, Transform, Load).
- **Amazon Athena** for serverless SQL queries over data in S3.
- **Amazon Redshift** when the question requires a dedicated data warehouse.

Choose S3 plus Athena/Glue when the exam emphasizes:

- Ad hoc SQL over files in S3.
- Low operational overhead.
- Pay-per-query analytics.
- Schema-on-read.

Choose Redshift when the question emphasizes:

- Large-scale data warehousing.
- Complex analytical queries.
- BI dashboards over structured warehouse data.
- Performance that exceeds what Athena is intended to provide for repeated warehouse-style workloads.

**Migration and backup considerations**

Engine selection questions often include migration constraints. Pay attention to compatibility and operational tooling.

Use **snapshots** when you need point-in-time backups or to restore an RDS or Aurora database to a new instance or cluster. Snapshots are also common for cloning environments.

Use **AWS Database Migration Service (DMS)** when you need to migrate data with minimal downtime, especially between different engines or when ongoing replication is required during cutover.

Use native engine tools when the question mentions:

- Logical export/import.
- Dump and restore.
- Engine-specific backup utilities.

Migration examples:

- MySQL to Aurora MySQL-compatible: often easier because of engine compatibility.
- PostgreSQL to Aurora PostgreSQL-compatible: often easier when extensions and SQL features align.
- Oracle or SQL Server to another engine: may require more planning, schema conversion, and DMS.

Be careful with engine compatibility:

- Aurora MySQL is not the same as Aurora PostgreSQL.
- Not every extension or feature is portable.
- Licensing and version support can affect the best choice.

**How to eliminate wrong answers quickly**

Use these exam shortcuts:

- If the question says **joins, foreign keys, normalized schema**, think relational: RDS or Aurora.
- If it says **massive scale, key-value, single-digit millisecond latency**, think DynamoDB.
- If it says **cache, session store, leaderboard, read acceleration**, think ElastiCache or MemoryDB.
- If it says **full-text search**, think OpenSearch.
- If it says **analytics over files in S3**, think Athena and Glue.
- If it says **warehouse**, think Redshift.
- If it says **multi-AZ failover**, think RDS Multi-AZ or Aurora, not read replicas.
- If it says **cross-Region active-active**, think DynamoDB global tables or Aurora global database depending on the workload.

**Practical decision matrix**

| Workload need | Best fit | Why |
|---|---|---|
| OLTP with SQL and joins | RDS | Familiar relational engines and managed operations |
| Managed relational with high availability and fast failover | Aurora | Cloud-native architecture and strong HA characteristics |
| Key-value at scale | DynamoDB | Low latency and elastic throughput |
| Temporary sessions or cache | ElastiCache | Offloads reads and reduces database pressure |
| Durable Redis-style low-latency store | MemoryDB for Redis | Redis semantics with durability |
| Search and log analytics | OpenSearch | Built for indexing and search |
| Data lake SQL over S3 | Athena + Glue | Serverless analytics without a database |
| Data warehouse | Redshift | Optimized for analytical queries |

**Common exam traps**

- Choosing a database when the workload is really a cache.
- Choosing Aurora when standard RDS is sufficient.
- Choosing DynamoDB for relational joins or ad hoc SQL.
- Assuming read replicas provide automatic failover.
- Assuming global tables provide synchronous cross-Region consistency.
- Using a database for analytics when S3 plus Athena is the better fit.
- Ignoring engine compatibility during migration.

**Example scenario mapping**

A web app stores user sessions and needs fast lookups with automatic expiration. Use DynamoDB with TTL or ElastiCache depending on whether durability matters.

A SaaS application needs relational transactions, SQL joins, and Multi-AZ failover. Use RDS or Aurora, with Aurora favored when the question emphasizes higher availability and faster failover.

A product catalog needs full-text search and filtering by keywords. Use OpenSearch.

A reporting team wants to query CSV files in S3 with SQL. Use Athena and Glue, not a database engine.

A gaming leaderboard needs extremely fast reads and writes with sorted access. Use DynamoDB with a carefully designed partition and sort key, or Redis-based storage if the question emphasizes in-memory ranking.

**Key takeaways**

Database engine selection on SAA-C03 is mostly about matching the workload to the right managed service. Start with the access pattern, then check consistency, latency, scaling, and operational needs. RDS and Aurora solve relational problems, DynamoDB solves high-scale key-value and document problems, ElastiCache and MemoryDB solve caching and ultra-low-latency problems, OpenSearch solves search problems, and S3 plus Athena/Glue solves many analytics problems without a traditional database.
