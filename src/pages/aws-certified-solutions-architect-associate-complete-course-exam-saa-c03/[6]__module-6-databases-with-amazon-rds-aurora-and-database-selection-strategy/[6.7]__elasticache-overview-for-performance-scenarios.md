## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**ElastiCache in performance-focused architectures**

Amazon ElastiCache is a managed in-memory data store service used to reduce latency and offload read pressure from primary data sources. In exam scenarios, you usually choose it when an application needs faster response times, high request rates, or temporary data that does not need durable relational storage.

ElastiCache is commonly used for:

- **Caching** frequently read data such as product catalogs, user profiles, and computed results.
- **Session storage** for web applications that need shared session state across multiple application servers.
- **Leaderboards and counters** where low-latency updates matter more than long-term durability.
- **Queue-like or stream-like patterns** when the workload fits Redis data structures.

The two main engines are:

- **Redis** for advanced data structures, persistence options, pub/sub, Lua scripting, and richer feature sets.
- **Memcached** for simple, horizontally scalable caching with a lighter operational model.

**When to choose ElastiCache**

Use ElastiCache when the bottleneck is read latency or database load, not when you need a system of record. It is especially useful when:

- The same data is read repeatedly by many users.
- The database is expensive to query for every request.
- You need sub-millisecond to millisecond response times for hot data.
- You can tolerate cache misses and stale data for short periods.

A typical architecture places ElastiCache between the application tier and the database tier:

```text
Users -> ALB -> Application -> ElastiCache -> Amazon RDS / Aurora / DynamoDB
```

The application checks the cache first. If the item is present, it returns immediately. If not, it queries the database, stores the result in cache, and then returns the response.

**Core caching patterns**

The most common pattern is **cache-aside**:

1. The application looks up data in ElastiCache.
2. If the key exists, the application uses the cached value.
3. If the key is missing, the application queries the database.
4. The application writes the result back to the cache with a TTL (time to live).
5. Future requests use the cached value until it expires.

Example cache-aside flow:

```python
value = cache.get("user:123")
if value is None:
    value = db.query("SELECT * FROM users WHERE id = 123")
    cache.set("user:123", value, ttl=300)
return value
```

This pattern works well because it keeps the application in control of cache population and avoids caching unnecessary data. It also lets you tune TTLs based on freshness requirements.

Other common patterns include:

- **Read-through** behavior, where the cache layer fetches missing data on behalf of the application. This is more abstracted, but the application still needs a strategy for consistency.
- **Write-through** behavior, where writes go to cache and backing store together. This improves read speed but adds write overhead.
- **Write-behind** behavior, where cache writes are flushed to the database asynchronously. This can improve throughput but increases complexity and risk.

For SAA-C03, cache-aside is the most important pattern to recognize.

**Redis vs Memcached**

Choose the engine based on the workload and feature needs.

| Feature | Redis | Memcached |
|---|---|---|
| Data structures | Strings, hashes, lists, sets, sorted sets, streams | Simple key-value |
| Persistence | Supported | Not supported |
| Replication | Supported | Not supported in the same way |
| Multi-AZ support | Yes, with replication groups | Limited compared to Redis |
| Advanced features | Pub/Sub, Lua, transactions, TTL, clustering | Simpler caching model |
| Best fit | Session store, leaderboards, richer app patterns | Basic caching at scale |

Use **Redis** when you need:

- Session persistence across app instances.
- Sorted sets for ranking or scoring.
- Atomic operations for counters and coordination.
- Pub/Sub for lightweight messaging.
- Optional persistence for recovery.

Use **Memcached** when you need:

- A straightforward distributed cache.
- Simple key-value lookups.
- Easy horizontal scaling for ephemeral data.
- Minimal feature overhead.

**High availability and scaling**

ElastiCache supports scaling and availability patterns that matter in production and in exam questions.

For Redis:

- **Replication groups** provide a primary node and one or more read replicas.
- **Automatic failover** promotes a replica if the primary fails.
- **Multi-AZ** improves resilience by placing nodes across Availability Zones.
- **Cluster mode enabled** partitions data across shards for horizontal scaling.

For Memcached:

- Nodes are added to increase capacity and throughput.
- The client typically handles partitioning across nodes.
- There is no equivalent to Redis replication-based failover in the same sense.

Important design implications:

- A cache is not a durable source of truth.
- If a node fails, cached data can be lost and rebuilt from the database.
- Your application should tolerate cache misses gracefully.
- Use TTLs to prevent stale data from living forever.

> Do not store critical business data in ElastiCache unless the design explicitly accounts for Redis persistence and recovery. Even then, the cache should not replace the primary datastore for most architectures.

**Eviction, TTL, and memory management**

Because ElastiCache is memory-based, capacity planning matters. When memory fills up, the service evicts data according to the configured policy.

Key concepts:

- **TTL** limits how long a key remains valid.
- **Eviction policy** determines what happens when memory is full.
- **Hot keys** can create uneven load if many requests target the same item.

Common Redis eviction policies include:

- `volatile-lru` — evict keys with TTL using least recently used behavior.
- `allkeys-lru` — evict any key using least recently used behavior.
- `volatile-ttl` — evict keys with the shortest remaining TTL.
- `noeviction` — reject writes when memory is full.

Example Redis configuration concepts:

```conf
maxmemory 4gb
maxmemory-policy allkeys-lru
```

Practical guidance:

- Use TTLs for data that can be recomputed.
- Size the cache for the working set, not the entire database.
- Monitor hit rate, memory usage, and evictions.
- Avoid caching large objects if smaller derived values are sufficient.

**Session management use case**

A common exam scenario is a stateless web tier behind an Application Load Balancer (ALB) where user sessions must be shared across multiple instances. ElastiCache is a strong fit because it centralizes session state and avoids sticky sessions as a hard dependency.

Example session flow:

- User logs in through the application.
- The app stores session data in Redis with a TTL.
- Any application instance can validate the session on later requests.
- Expired sessions are removed automatically.

Example Redis session write:

```bash
redis-cli -h mycache.xxxxxx.use1.cache.amazonaws.com -p 6379
SET session:abc123 '{"userId":42,"role":"admin"}' EX 1800
GET session:abc123
```

Why this helps:

- Application servers remain stateless.
- Horizontal scaling becomes easier.
- Session expiration is automatic.
- Failover is simpler than with local in-memory sessions.

**Performance tuning and operational considerations**

ElastiCache improves performance only when the application uses it correctly. Poor key design or excessive cache churn can reduce the benefit.

Best practices:

- Use stable, predictable cache keys.
- Keep values compact.
- Cache expensive computations and repeated reads.
- Set TTLs based on business freshness requirements.
- Avoid caching highly volatile data unless staleness is acceptable.
- Use connection pooling in the application to reduce overhead.

Operational checks:

- Monitor cache hit ratio.
- Watch CPU, memory, and network throughput.
- Track evictions and replication lag.
- Review slow commands in Redis if latency increases.
- Test failover behavior before production cutover.

Example application-side cache key strategy:

```text
user:42
product:sku-1001
search:shoes:page:2
report:sales:2026-04-26
```

Good keys are:

- Consistent.
- Human-readable.
- Namespaced by entity or feature.
- Easy to invalidate selectively.

**Common exam scenarios**

ElastiCache is often the best answer when the question mentions one or more of these symptoms:

- A relational database is overloaded by repeated reads.
- The application needs faster response times for frequently accessed data.
- Session state must be shared across multiple app servers.
- A leaderboard or ranking system needs sorted, low-latency updates.
- Temporary data can be regenerated if lost.

It is usually not the best answer when:

- You need durable long-term storage.
- You need complex analytical queries.
- You need a full-text search engine.
- You need cross-region active-active data consistency without additional design.

**Redis data structures and practical uses**

Redis is more than a simple cache. Its data structures support patterns that show up in architecture questions.

- **Strings** for basic key-value caching and counters.
- **Hashes** for storing object-like records.
- **Lists** for queues and ordered processing.
- **Sets** for unique membership checks.
- **Sorted sets** for leaderboards and ranking.
- **Streams** for event-like processing and ordered records.

Examples:

```bash
INCR pageviews:home
HSET user:42 name "Ava" tier "gold"
ZADD leaderboard 1200 "user:42"
ZRANGE leaderboard 0 9 WITHSCORES
```

These structures are useful when you need fast in-memory operations that would be expensive in a relational database.

**Deployment and access**

ElastiCache is deployed inside a VPC (Virtual Private Cloud). Your application must be able to reach the cache endpoint over the network, usually from EC2, ECS, EKS, or Lambda in the same VPC or connected network.

Practical considerations:

- Place the cache in private subnets.
- Restrict access with security groups.
- Allow only application servers to connect to the cache port.
- Use TLS if supported and required by your security posture.
- Use authentication features where available for Redis.

Example security group intent:

- Application security group can connect to Redis port `6379`.
- No public internet access is allowed.
- Administrative access is limited to private network paths.

**Troubleshooting mindset**

If performance does not improve after adding ElastiCache, check the following:

- The cache hit rate may be too low.
- TTLs may be too short.
- The application may be serializing too much data.
- The working set may exceed memory.
- The cache key strategy may be causing fragmentation.
- The database may still be the bottleneck for uncached requests.

A cache is most effective when the hot set is small enough to fit in memory and the access pattern is repetitive.

**Quick decision guide**

- Use **Redis** for richer features, session state, counters, and leaderboards.
- Use **Memcached** for simple distributed caching.
- Use **cache-aside** when the application should control cache population.
- Use **TTL** to manage freshness and memory pressure.
- Use **Multi-AZ** and replication for higher availability with Redis.
- Treat the cache as a performance layer, not the source of truth.
