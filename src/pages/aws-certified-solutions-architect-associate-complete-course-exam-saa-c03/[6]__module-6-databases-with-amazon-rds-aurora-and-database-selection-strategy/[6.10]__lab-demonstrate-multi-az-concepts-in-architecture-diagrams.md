## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Multi-AZ architecture fundamentals**

A Multi-AZ design places resources across two or more <abbr title="Availability Zone">AZ</abbr>s within the same AWS Region to reduce the impact of an AZ failure. In diagrams, you show this by separating compute, data, and networking components into distinct AZ boundaries and by drawing failover paths that keep the application available if one AZ becomes impaired.

The key exam idea is that Multi-AZ improves **availability** and **fault tolerance**, not geographic disaster recovery. For disaster recovery across Regions, you would use Multi-Region patterns instead.

- An **AZ** is a physically separate data center location within a Region.
- A **Region** contains multiple AZs and is the usual boundary for Multi-AZ high availability.
- Multi-AZ designs typically use redundant load balancing, compute, and data layers.
- You should show traffic continuing through a healthy AZ when another AZ fails.

**How to represent Multi-AZ in architecture diagrams**

Use clear visual separation so the reader can immediately see redundancy. A good diagram labels each AZ, places duplicate components in each zone, and shows shared entry points such as Route 53, an Application Load Balancer, or a Network Load Balancer.

A practical diagram often includes:
- A public entry layer with Route 53 and a load balancer spanning multiple AZs.
- Application instances or containers in at least two private subnets, one per AZ.
- A database layer using a Multi-AZ managed service or synchronous replication.
- NAT gateways, if needed, deployed per AZ to avoid cross-AZ dependency for outbound internet access.

A simple topology might look like this:

```text
Users
  |
Route 53
  |
Application Load Balancer
  |-----------------------------|
  |                             |
AZ-a                         AZ-b
Public Subnet               Public Subnet
Private Subnet              Private Subnet
EC2 / ECS / EKS              EC2 / ECS / EKS
  |                             |
  |----------- Shared DB --------|
          Multi-AZ database
```

**Load balancing across AZs**

An Application Load Balancer (ALB) or Network Load Balancer (NLB) can distribute traffic across targets in multiple AZs. In diagrams, show the load balancer as the single entry point and the targets as duplicated across zones.

Important behaviors to remember:
- The load balancer health checks targets and stops sending traffic to unhealthy instances.
- If one AZ fails, the load balancer routes traffic to healthy targets in the remaining AZ.
- Cross-zone load balancing may be enabled depending on the service and configuration, but the main exam point is that the load balancer spans AZs.

Example ALB creation with multiple subnets:

```bash
aws elbv2 create-load-balancer \
  --name app-alb \
  --type application \
  --subnets subnet-aaa111 subnet-bbb222 \
  --security-groups sg-1234567890abcdef0
```

You can verify target health with:

```bash
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/app-tg/abcdef1234567890
```

**Compute layer patterns**

For the compute tier, Multi-AZ usually means you run identical application capacity in at least two AZs. The exact service changes the diagram, but the resilience pattern stays the same.

Common examples:
- **Amazon EC2** in an Auto Scaling group across multiple subnets.
- **Amazon ECS** services spread across AZs.
- **Amazon EKS** worker nodes in multiple AZs.
- **AWS Lambda** is already regional and abstracted from AZ placement, but it can still integrate with Multi-AZ backends.

For EC2-based designs, show:
- An Auto Scaling group spanning subnets in different AZs.
- A launch template or AMI used to create identical instances.
- Stateless application servers so any instance can serve any request.

Example Auto Scaling group spanning two AZs:

```bash
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name app-asg \
  --launch-template LaunchTemplateName=app-lt,Version=1 \
  --min-size 2 \
  --max-size 6 \
  --desired-capacity 2 \
  --vpc-zone-identifier "subnet-aaa111,subnet-bbb222"
```

**Database layer patterns**

The database tier is where Multi-AZ matters most on the exam. Many AWS managed databases support Multi-AZ deployments that provide synchronous replication to a standby in another AZ and automatic failover.

Typical services and patterns:
- **Amazon RDS Multi-AZ** uses a primary and a standby in different AZs.
- **Amazon Aurora** stores data across multiple AZs and can fail over quickly to another instance.
- **Amazon DynamoDB** is regional and automatically replicated across multiple AZs within the Region.
- **Amazon ElastiCache** can use Multi-AZ with automatic failover for supported engines and configurations.

For RDS, the diagram should show:
- One writer instance in AZ-a.
- One standby instance in AZ-b.
- Application traffic connecting to the RDS endpoint, not directly to the standby.

Example RDS Multi-AZ creation:

```bash
aws rds create-db-instance \
  --db-instance-identifier app-db \
  --db-instance-class db.t3.medium \
  --engine mysql \
  --allocated-storage 20 \
  --master-username admin \
  --master-user-password 'ChangeMe123!' \
  --multi-az
```

Key points:
- The standby is not used for read traffic in standard RDS Multi-AZ.
- Failover changes the primary endpoint target automatically.
- You should not draw the application connecting to both primary and standby directly unless the service explicitly supports that pattern.

**Networking considerations**

A strong Multi-AZ diagram includes subnet placement and routing. Each AZ should usually have its own public and private subnets, especially when you want to avoid a single-AZ dependency.

Best practices to show:
- One public subnet per AZ for load balancers or bastion access if used.
- One private subnet per AZ for application servers.
- One private database subnet group spanning multiple AZs.
- Separate NAT gateways per AZ for resilient outbound internet access from private subnets.

Why this matters:
- If a NAT gateway exists only in one AZ, instances in another AZ may depend on cross-AZ routing for outbound traffic.
- If a route table or subnet is misconfigured, the diagram should make the dependency visible.

Example route table concept:

```text
Private Subnet AZ-a -> NAT Gateway AZ-a -> Internet Gateway
Private Subnet AZ-b -> NAT Gateway AZ-b -> Internet Gateway
```

**What to label in the diagram**

Labels make the diagram useful for exam review. Include the following where relevant:
- AZ names such as `us-east-1a` and `us-east-1b`.
- Subnet types such as public and private.
- Service names such as ALB, RDS Multi-AZ, Auto Scaling group, or Aurora.
- Traffic direction arrows showing request flow and failover flow.
- Health check or standby indicators where they clarify behavior.

A concise annotation set might be:
- `Route 53` for DNS entry.
- `ALB` spanning both AZs.
- `ASG` with instances in each AZ.
- `RDS Multi-AZ` with primary and standby.
- `Failover` arrow from failed AZ to healthy AZ.

**How to explain failover in words**

When one AZ becomes unavailable, the application should continue operating from the remaining AZ. The exact failover behavior depends on the service:
- Load balancers stop routing to unhealthy targets.
- Auto Scaling replaces failed instances in healthy AZs.
- RDS promotes the standby to primary.
- Aurora performs fast failover to another instance in the cluster.

In notes and diagrams, distinguish between:
- **Active-active** patterns, where both AZs serve traffic.
- **Active-passive** patterns, where one AZ is standby for a specific layer.
- **Synchronous replication**, where data is written to more than one location before success is acknowledged.

**Common exam traps**

Be careful not to confuse Multi-AZ with Multi-Region. They solve different problems.

- Multi-AZ is for high availability within a Region.
- Multi-Region is for disaster recovery, latency optimization, or global resilience.
- A single instance in one AZ is not Multi-AZ, even if backups exist.
- A read replica is not automatically a Multi-AZ standby; it is usually for read scaling or cross-Region replication, depending on the service.
- An application can be highly available even if one layer is active-passive, as long as the overall design survives AZ loss.

> If a diagram shows only one subnet, one instance, or one database node, it is not demonstrating Multi-AZ resilience.

**Practical diagramming workflow**

1. Draw the Region boundary first.
2. Split the Region into at least two AZ columns.
3. Place identical network layers in each AZ.
4. Add the shared entry point at the top.
5. Add the compute tier in both AZs.
6. Add the data tier with the correct Multi-AZ pattern.
7. Draw failover arrows and label the unhealthy path.
8. Verify that no single component outside the shared entry point creates a hidden single point of failure.

**Example architecture summary**

A typical SAA-C03 Multi-AZ web application looks like this:
- Users resolve DNS through Route 53.
- Traffic enters an ALB spanning two AZs.
- The ALB forwards requests to EC2 instances in an Auto Scaling group across private subnets in both AZs.
- The application stores data in an RDS Multi-AZ database.
- Each AZ has its own subnet layout and, if needed, its own NAT gateway.

This design keeps the application available if one AZ fails because the remaining AZ can continue serving traffic and the database can fail over automatically.

**CLI and verification examples**

You can inspect subnet placement and confirm that resources span AZs:

```bash
aws ec2 describe-subnets \
  --subnet-ids subnet-aaa111 subnet-bbb222 \
  --query 'Subnets[*].{SubnetId:SubnetId,AZ:AvailabilityZone,CIDR:CidrBlock}'
```

You can check an Auto Scaling group’s subnet coverage:

```bash
aws autoscaling describe-auto-scaling-groups \
  --auto-scaling-group-names app-asg \
  --query 'AutoScalingGroups[*].VPCZoneIdentifier'
```

You can inspect an RDS instance for Multi-AZ status:

```bash
aws rds describe-db-instances \
  --db-instance-identifier app-db \
  --query 'DBInstances[*].{DBInstanceIdentifier:DBInstanceIdentifier,MultiAZ:MultiAZ,AZ:AvailabilityZone,SecondaryAZ:SecondaryAvailabilityZone}'
```

**Key takeaways**

- Multi-AZ means redundancy across Availability Zones in the same Region.
- Diagrams should show duplicated resources, shared entry points, and failover behavior.
- Load balancers, Auto Scaling groups, and managed databases are the core building blocks.
- Multi-AZ improves availability and fault tolerance, but it is not the same as Multi-Region disaster recovery.
- The best diagrams make the failure path obvious and show how traffic continues after an AZ outage.
