## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon DynamoDB at a glance**

Amazon DynamoDB is a fully managed, serverless <abbr title="NoSQL">NoSQL</abbr> database service designed for low-latency, high-scale workloads. You use it when you need predictable performance, automatic scaling, and minimal operational overhead. For SAA-C03, focus on how DynamoDB differs from relational databases, how you model data around access patterns, and how you choose the right indexing and consistency options.

DynamoDB is not a general-purpose relational engine. You do not design it by normalizing tables first; you design it by asking how your application reads and writes data. That shift in thinking is central to using DynamoDB well.

**Core characteristics**

- **Managed service**: AWS handles patching, replication, hardware, and most operational tasks.
- **Serverless**: You do not provision or manage servers.
- **Single-digit millisecond latency**: Typical design goal for reads and writes at scale.
- **Flexible schema**: Items in the same table can have different attributes.
- **High availability**: Data is automatically replicated across multiple Availability Zones in a Region.
- **Elastic scaling**: Capacity can scale with demand, especially with on-demand mode.

**When to choose DynamoDB**

Use DynamoDB when you need:

- Very high request rates with low latency.
- Simple key-based access patterns.
- Event-driven or session-based workloads.
- Shopping carts, user profiles, gaming state, IoT telemetry, metadata stores, and leaderboards.
- Global applications that benefit from multi-Region replication.

Avoid DynamoDB when you need:

- Complex joins across many entities.
- Ad hoc SQL-style querying.
- Strong relational constraints and foreign keys.
- Heavy transactional reporting or analytics over large datasets.

**Data model fundamentals**

DynamoDB stores data in **tables**, **items**, and **attributes**.

- A **table** is a collection of items.
- An **item** is similar to a row, but each item can have different attributes.
- An **attribute** is a field/value pair, similar to a column value.

The most important design decision is the **primary key**. DynamoDB supports two primary key types:

- **Partition key only**: Also called a simple primary key.
- **Partition key + sort key**: Also called a composite primary key.

The partition key determines how DynamoDB distributes data across partitions. The sort key lets you store multiple related items under the same partition key and query them efficiently by range, prefix, or exact match.

**Primary key design**

A good key design spreads traffic evenly and supports your most common queries.

- Use a partition key with high cardinality to avoid hot partitions.
- Choose values that distribute requests evenly.
- Use the sort key to model one-to-many relationships and ordered data.
- Design around access patterns before you create the table.

Common examples:

- `UserId` as partition key for user-centric data.
- `OrderId` as partition key for order records.
- `CustomerId` as partition key and `OrderDate` as sort key for customer order history.
- `DeviceId` as partition key and timestamp as sort key for telemetry.

A composite key enables queries like:

- All orders for one customer.
- All events for one device in a time range.
- The latest record for a given entity when you sort descending or use a time-based pattern.

**Example table design**

```text
Table: Orders
Partition key: CustomerId
Sort key: OrderDateTime

Item examples:
- CustomerId = C123, OrderDateTime = 2026-04-01T10:15:00Z, OrderId = O1001, Status = Shipped
- CustomerId = C123, OrderDateTime = 2026-04-15T08:30:00Z, OrderId = O1044, Status = Pending
- CustomerId = C456, OrderDateTime = 2026-04-20T14:05:00Z, OrderId = O2001, Status = Delivered
```

This design lets you query all orders for `C123` and filter by date range efficiently.

**Access patterns and single-table thinking**

DynamoDB works best when you know your access patterns in advance. An access pattern is a specific way your application reads or writes data.

Typical questions to ask:

- What do users need to look up most often?
- Which queries must be fast?
- Which items are read together?
- Which data should be grouped under the same partition key?

In many DynamoDB designs, you store multiple entity types in one table. This is often called **single-table design**. The goal is to reduce joins and make related data queryable with key lookups.

Example pattern:

- Partition key: `PK`
- Sort key: `SK`
- `PK = USER#123`
- `SK = PROFILE`, `ORDER#2026-04-01`, `ORDER#2026-04-15`

This lets you store a user profile and related orders in the same table while querying them by key.

**Read and write operations**

DynamoDB supports both item-level and query-based access.

- **GetItem**: Retrieves one item by primary key. Fastest and most direct read path.
- **PutItem**: Creates or replaces an item.
- **UpdateItem**: Updates one or more attributes.
- **DeleteItem**: Removes an item.
- **Query**: Reads items by partition key and optional sort key conditions.
- **Scan**: Reads every item in a table or index.

Use `Query` whenever possible. Use `Scan` only when you have no better option, because it reads the entire dataset and can become expensive and slow.

Example AWS CLI query:

```bash
aws dynamodb query \
  --table-name Orders \
  --key-condition-expression "CustomerId = :cid AND OrderDateTime BETWEEN :start AND :end" \
  --expression-attribute-values '{
    ":cid": {"S": "C123"},
    ":start": {"S": "2026-04-01T00:00:00Z"},
    ":end": {"S": "2026-04-30T23:59:59Z"}
  }'
```

Example item write:

```bash
aws dynamodb put-item \
  --table-name Orders \
  --item '{
    "CustomerId": {"S": "C123"},
    "OrderDateTime": {"S": "2026-04-26T12:00:00Z"},
    "OrderId": {"S": "O3001"},
    "Status": {"S": "Pending"}
  }'
```

**Indexes**

Indexes let you query data using alternate keys.

- **Global Secondary Index (GSI)**: Uses a partition key and optional sort key different from the base table. You can query it independently of the table’s primary key.
- **Local Secondary Index (LSI)**: Uses the same partition key as the base table but a different sort key. You must define it when creating the table.

Use a GSI when you need a new access pattern that does not match the base table key. Use an LSI when you need alternate sort order within the same partition key and want to keep the same partition grouping.

Important differences:

- **GSI** can be created after table creation.
- **LSI** must be created with the table.
- **GSI** has its own throughput behavior and can be eventually consistent only.
- **LSI** supports strongly consistent reads because it shares the table partition key.

Example GSI use case:

- Base table key: `OrderId`
- GSI key: `CustomerId` + `OrderDateTime`
- Purpose: Find all orders for a customer.

Example CLI table creation with a GSI:

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions \
    AttributeName=OrderId,AttributeType=S \
    AttributeName=CustomerId,AttributeType=S \
    AttributeName=OrderDateTime,AttributeType=S \
  --key-schema \
    AttributeName=OrderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --global-secondary-indexes '[
    {
      "IndexName": "CustomerOrdersIndex",
      "KeySchema": [
        {"AttributeName": "CustomerId", "KeyType": "HASH"},
        {"AttributeName": "OrderDateTime", "KeyType": "RANGE"}
      ],
      "Projection": {"ProjectionType": "ALL"}
    }
  ]'
```

**Capacity modes**

DynamoDB offers two main capacity modes:

- **On-demand mode**: AWS automatically handles capacity. Good for unpredictable traffic or new applications.
- **Provisioned mode**: You specify read and write capacity. Good when traffic is predictable and you want cost control.

In provisioned mode, you can enable **auto scaling** to adjust capacity based on demand. For exam purposes, remember:

- On-demand is simpler and reduces capacity planning.
- Provisioned can be cheaper for steady workloads.
- Sudden traffic spikes can be easier to absorb with on-demand mode.

**Throughput and partition behavior**

DynamoDB partitions data behind the scenes. Performance depends on how evenly traffic is distributed.

- A **hot partition** occurs when too many requests target the same partition key.
- A poor key design can create uneven load and throttling.
- A high-cardinality partition key helps spread traffic.
- Large items and frequent reads of the same key can also create pressure.

To reduce hot partitions:

- Add more entropy to the partition key.
- Use write sharding for very high write rates.
- Avoid monotonically increasing keys as the only partition key when they concentrate traffic.
- Distribute time-series writes carefully.

Example of write sharding:

```text
Partition key: DeviceId#ShardId
Examples:
- Device42#01
- Device42#02
- Device42#03
```

This spreads writes across multiple partitions while preserving logical grouping.

**Consistency model**

DynamoDB supports two read consistency options:

- **Eventually consistent reads**: Default for most reads. Lower latency and higher availability, but a recent write may not be visible immediately.
- **Strongly consistent reads**: Returns the latest committed data from the base table in the same Region.

Use strongly consistent reads when correctness matters more than latency, such as after a write that must be read back immediately. Be aware that strongly consistent reads are not available for GSIs.

**Transactions**

DynamoDB supports ACID-style transactions for multiple items across one or more tables.

- Use transactions when you need all-or-nothing behavior.
- They help maintain consistency for workflows like money movement, inventory reservation, or coordinated updates.
- They add complexity and cost, so use them only when needed.

Example transaction write:

```bash
aws dynamodb transact-write-items \
  --transact-items '[
    {
      "Put": {
        "TableName": "Inventory",
        "Item": {
          "ProductId": {"S": "P100"},
          "Available": {"N": "9"}
        }
      }
    },
    {
      "Update": {
        "TableName": "Orders",
        "Key": {
          "OrderId": {"S": "O3001"}
        },
        "UpdateExpression": "SET #s = :new",
        "ExpressionAttributeNames": {"#s": "Status"},
        "ExpressionAttributeValues": {
          ":new": {"S": "Confirmed"}
        }
      }
    }
  ]'
```

**Conditional writes**

Conditional expressions let you write data only if a condition is true.

- Prevent overwriting existing items accidentally.
- Enforce optimistic locking.
- Ensure inventory counts do not go below zero.

Example:

```bash
aws dynamodb update-item \
  --table-name Inventory \
  --key '{"ProductId": {"S": "P100"}}' \
  --update-expression "SET Available = Available - :one" \
  --condition-expression "Available >= :one" \
  --expression-attribute-values '{":one": {"N": "1"}}'
```

This pattern is useful when multiple clients may update the same item concurrently.

**TTL, streams, and backups**

DynamoDB includes several operational features that often appear in architecture questions.

- **Time to Live (TTL)**: Automatically expires items after a specified timestamp. Useful for sessions, temporary tokens, and ephemeral data.
- **DynamoDB Streams**: Captures item-level changes for event-driven processing, replication, or auditing.
- **Backups**: Supports on-demand backups and point-in-time recovery (PITR) for restoring data to a previous state.

Use TTL to reduce storage for data that should not live forever. Use Streams when another service must react to changes, such as AWS Lambda processing updates.

Example TTL attribute:

```json
{
  "SessionId": {"S": "abc123"},
  "UserId": {"S": "U100"},
  "ExpiresAt": {"N": "1770000000"}
}
```

**Global tables**

**Global Tables** replicate DynamoDB tables across multiple AWS Regions.

- They support multi-Region, active-active architectures.
- They improve latency for globally distributed users.
- They provide regional resilience and disaster recovery options.
- Replication is asynchronous, so design for conflict handling if multiple Regions write the same item.

Use global tables when you need low-latency local reads and writes in more than one Region. They are a strong fit for globally distributed applications, but they add replication complexity and cost.

**Security and access control**

DynamoDB integrates with AWS Identity and Access Management (IAM).

- Use IAM policies to control table and index access.
- Prefer least privilege.
- Restrict actions like `dynamodb:Scan` when not needed.
- Use encryption at rest with AWS Key Management Service (AWS KMS) when required by policy or compliance.

Example IAM policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:Query",
        "dynamodb:PutItem",
        "dynamodb:UpdateItem"
      ],
      "Resource": [
        "arn:aws:dynamodb:us-east-1:123456789012:table/Orders",
        "arn:aws:dynamodb:us-east-1:123456789012:table/Orders/index/CustomerOrdersIndex"
      ]
    }
  ]
}
```

**Monitoring and troubleshooting**

Use Amazon CloudWatch metrics and logs to understand performance and throttling.

Watch for:

- **ThrottledRequests**: Requests rejected due to insufficient capacity or hot partitions.
- **ConsumedReadCapacityUnits** and **ConsumedWriteCapacityUnits**: Actual usage.
- **SuccessfulRequestLatency**: End-to-end latency.
- **SystemErrors** and **UserErrors**: Service or application issues.

Troubleshooting checklist:

- Check whether the partition key is too popular.
- Review whether `Scan` is causing unnecessary load.
- Confirm whether capacity mode matches traffic patterns.
- Inspect index design for missing access patterns.
- Verify item size and request size limits.

**Limits and practical considerations**

DynamoDB has practical limits that influence design.

- Item size is limited, so do not store large blobs directly unless necessary.
- Query is efficient only when you know the partition key.
- Secondary indexes add write overhead because updates must propagate to indexes.
- Large items increase storage and read costs.
- Many-to-many relationships often require careful modeling or multiple tables.

For large binary objects, store the object in Amazon S3 and keep metadata in DynamoDB.

**Common exam patterns**

Expect questions that ask you to choose the best database or the best DynamoDB feature for a scenario.

Look for these clues:

- Need for low-latency key-value access at scale → DynamoDB.
- Need for flexible schema and serverless operations → DynamoDB.
- Need to query by an alternate attribute → GSI.
- Need to keep the same partition key but change sort order → LSI.
- Need temporary data expiration → TTL.
- Need event-driven processing of item changes → Streams.
- Need multi-Region active-active replication → Global Tables.
- Need predictable capacity with steady traffic → Provisioned mode.
- Need simple scaling for unpredictable traffic → On-demand mode.

**Quick comparison with relational databases**

| Capability | DynamoDB | Relational DB |
|---|---|---|
| Schema | Flexible | Structured |
| Joins | Not native | Native |
| Scaling | Horizontal, managed | Often more manual |
| Latency | Very low at scale | Depends on design |
| Query style | Key-based access patterns | SQL and ad hoc queries |
| Best fit | High-scale operational workloads | Complex relational data |

**Practical design workflow**

1. Identify your top read and write access patterns.
2. Choose a partition key that spreads traffic evenly.
3. Add a sort key if you need grouping or range queries.
4. Add GSIs only for real access patterns.
5. Use on-demand mode if traffic is unpredictable.
6. Add TTL, Streams, or Global Tables only when the use case requires them.
7. Test for hot partitions and throttling before production.

**Example architecture scenario**

A mobile app stores user sessions, preferences, and recent activity.

- Use DynamoDB for session state and profile metadata.
- Use `UserId` as the partition key.
- Use `SessionId` or timestamp as the sort key.
- Enable TTL for session expiration.
- Use Streams to trigger downstream processing when preferences change.
- Use a GSI if you need to look up users by email address.

This design keeps reads fast, avoids server management, and supports elastic traffic spikes.

**Key takeaways**

- DynamoDB is a managed NoSQL database optimized for scale and low latency.
- Model data around access patterns, not normalization.
- Choose partition keys carefully to avoid hot partitions.
- Use `Query` instead of `Scan` whenever possible.
- Add GSIs and LSIs only for specific query needs.
- Pick on-demand or provisioned capacity based on traffic predictability.
- Use TTL, Streams, transactions, and global tables when the workload requires them.
