## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon RDS overview**

Amazon Relational Database Service (Amazon RDS) is a managed database service that handles provisioning, patching, backups, and many operational tasks for you. In a lab, you typically launch a single database instance, connect it to a VPC (Virtual Private Cloud), and verify that your application or client can reach it securely.

For the exam, focus on the tradeoffs: RDS reduces administrative overhead, but you still control important choices such as engine, instance class, storage, networking, backups, and security groups.

- **Managed service** means AWS handles routine database administration tasks.
- **Engine choice** affects features, compatibility, and licensing.
- **Availability** depends on whether you choose a single-AZ or Multi-AZ deployment.
- **Security** is enforced through VPC security groups, subnet placement, and authentication.

**Core launch decisions**

When you create an RDS instance, you make several decisions that shape cost, performance, and resilience. These are the most common lab settings and the ones most likely to matter in scenario questions.

- **Database engine**: Common options include MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, and Aurora-compatible engines.
- **Deployment model**: Single-AZ is simpler and cheaper; Multi-AZ provides synchronous standby replication for high availability.
- **Instance class**: Determines CPU and memory capacity.
- **Storage type**: General Purpose SSD is common for labs; Provisioned IOPS is used for higher and more predictable performance.
- **Backup retention**: Automated backups support point-in-time recovery within the retention window.
- **Public access**: Usually disabled unless you explicitly need direct internet access for a lab.
- **Master credentials**: You define the initial database administrator username and password.

A practical rule is to start with the smallest instance class that meets the lab requirement, then scale only if performance or connection issues appear.

**Typical launch workflow**

A standard RDS launch follows a predictable sequence. The exact console labels may vary slightly by engine and AWS console updates, but the workflow remains the same.

1. Open the Amazon RDS console and choose to create a database.
2. Select the database engine that matches the lab requirement.
3. Choose a template such as production, dev/test, or free tier if available.
4. Configure the DB instance identifier, master username, and password.
5. Select the instance class and storage settings.
6. Choose the VPC, subnet group, and security group rules.
7. Decide whether the instance is publicly accessible.
8. Configure backup retention, monitoring, and maintenance preferences.
9. Review the settings and create the database.
10. Wait for the status to become available before testing connectivity.

> If the instance is not in `available` status, connection attempts usually fail. Wait for provisioning to complete before troubleshooting the client.

**Networking and connectivity**

RDS is deployed inside your VPC, so network access is controlled by security groups and subnet placement. In most labs, you connect from an EC2 instance in the same VPC or from a bastion host rather than exposing the database to the public internet.

- **Security group inbound rules** should allow the database port from the trusted source only.
- **Subnet groups** define which subnets RDS can use; for Multi-AZ, use subnets in at least two Availability Zones.
- **Publicly accessible** should be `No` unless the lab explicitly requires external access.
- **Route tables** do not directly open RDS access; security groups and network reachability matter more for client connectivity.

Common default ports:

| Engine | Default port |
|---|---:|
| MySQL / Aurora MySQL | 3306 |
| PostgreSQL / Aurora PostgreSQL | 5432 |
| MariaDB | 3306 |
| Oracle | 1521 |
| SQL Server | 1433 |

A secure inbound rule usually looks like this:

```text
Type: MySQL/Aurora
Protocol: TCP
Port range: 3306
Source: sg-0123456789abcdef0
```

Using a security group as the source is preferred over a broad CIDR block because it limits access to instances that already have the trusted security group attached.

**Example CLI launch pattern**

You can also create an RDS instance with the AWS CLI (Command Line Interface). This is useful for automation, repeatability, and exam familiarity.

```bash
aws rds create-db-instance \
  --db-instance-identifier lab-mysql-01 \
  --db-instance-class db.t3.micro \
  --engine mysql \
  --master-username adminuser \
  --master-user-password 'ChangeMe123!' \
  --allocated-storage 20 \
  --vpc-security-group-ids sg-0123456789abcdef0 \
  --db-subnet-group-name lab-subnet-group \
  --backup-retention-period 7 \
  --no-publicly-accessible \
  --storage-type gp2
```

Useful verification commands:

```bash
aws rds describe-db-instances \
  --db-instance-identifier lab-mysql-01

aws rds wait db-instance-available \
  --db-instance-identifier lab-mysql-01
```

If you need the endpoint for connection testing, query it directly:

```bash
aws rds describe-db-instances \
  --db-instance-identifier lab-mysql-01 \
  --query 'DBInstances[0].Endpoint'
```

**Connecting to the database**

After the instance is available, connect from an allowed client. In labs, that is often an EC2 instance with the database client installed.

For MySQL:

```bash
mysql -h lab-mysql-01.xxxxxx.us-east-1.rds.amazonaws.com \
  -P 3306 \
  -u adminuser \
  -p
```

For PostgreSQL:

```bash
psql "host=lab-pg-01.xxxxxx.us-east-1.rds.amazonaws.com port=5432 dbname=postgres user=adminuser sslmode=require"
```

For SQL Server, you typically use a GUI client or `sqlcmd`:

```bash
sqlcmd -S lab-sql-01.xxxxxx.us-east-1.rds.amazonaws.com,1433 -U adminuser -P 'ChangeMe123!'
```

If the connection fails, check these in order:

- **Security group rules** on the RDS instance
- **Security group rules** on the client instance
- **Public accessibility** setting
- **Subnet routing** and whether the client can reach the VPC
- **Correct endpoint, port, and engine-specific client**
- **Database status** and whether the instance is still provisioning

**Security and authentication**

RDS supports standard database authentication, and some engines also support IAM (Identity and Access Management) database authentication. For most labs, username and password authentication is enough, but you should understand the security model.

- **Master user** is the initial administrative account created at launch.
- **Least privilege** means you should create application users with only the permissions they need.
- **Encryption at rest** uses AWS Key Management Service (AWS KMS) keys.
- **Encryption in transit** is typically enabled with TLS (Transport Layer Security).
- **Secrets Manager** can store and rotate database credentials in more advanced setups.

A practical PostgreSQL example for creating an application user:

```sql
CREATE USER app_user WITH PASSWORD 'StrongPassword123!';
GRANT CONNECT ON DATABASE appdb TO app_user;
GRANT USAGE ON SCHEMA public TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;
```

For MySQL:

```sql
CREATE USER 'app_user'@'%' IDENTIFIED BY 'StrongPassword123!';
GRANT SELECT, INSERT, UPDATE, DELETE ON appdb.* TO 'app_user'@'%';
FLUSH PRIVILEGES;
```

**Backups, snapshots, and recovery**

RDS provides automated backups and manual snapshots. These are different tools with different purposes.

- **Automated backups** support point-in-time recovery within the retention period.
- **Manual snapshots** are user-managed and retained until you delete them.
- **Restoring from a snapshot** creates a new DB instance rather than overwriting the original.
- **Retention period** should be long enough to cover your recovery needs and compliance requirements.

A common exam distinction is that automated backups are for recovery to a point in time, while snapshots are for long-term retention or cloning environments.

**High availability and failover**

Multi-AZ deployments improve availability by synchronously replicating data to a standby in another Availability Zone. If the primary fails, RDS performs failover to the standby.

- **Single-AZ** is suitable for development, testing, and cost-sensitive workloads.
- **Multi-AZ** is suitable for production workloads that need higher availability.
- **Failover** changes the endpoint target, but the database endpoint name stays the same.
- **Read replicas** are for read scaling and disaster recovery use cases, not automatic failover in the same way as Multi-AZ.

> Multi-AZ is about availability, not read scaling. If you need more read throughput, consider read replicas instead.

**Monitoring and troubleshooting**

RDS integrates with CloudWatch metrics and logs. In a lab, you may not need deep observability, but basic monitoring helps you confirm the instance is healthy.

Common metrics and signals:

- **CPUUtilization** indicates compute pressure.
- **FreeStorageSpace** helps you detect storage exhaustion.
- **DatabaseConnections** shows connection load.
- **ReadIOPS / WriteIOPS** help you understand storage activity.
- **FreeableMemory** can indicate memory pressure.

Useful CLI checks:

```bash
aws rds describe-db-instances \
  --db-instance-identifier lab-mysql-01 \
  --query 'DBInstances[0].{Status:DBInstanceStatus,Endpoint:Endpoint.Address,Port:Endpoint.Port,AZ:AvailabilityZone,MultiAZ:MultiAZ}'
```

If you need to inspect security groups:

```bash
aws ec2 describe-security-groups \
  --group-ids sg-0123456789abcdef0
```

**Common lab mistakes**

These are the issues that most often block a successful launch or connection.

- **Wrong security group source**: The database allows the wrong CIDR or no inbound rule at all.
- **Public access confusion**: The instance is private, but you try to connect from your laptop.
- **Incorrect port**: The client uses the wrong engine port.
- **Wrong subnet group**: The DB subnet group does not include the required subnets or Availability Zones.
- **Password mismatch**: The master password is mistyped or copied with extra spaces.
- **Instance still creating**: The endpoint exists, but the database is not yet ready.
- **Engine/client mismatch**: You use a MySQL client against a PostgreSQL instance.

**Exam-focused takeaways**

For SAA-C03, you should be able to choose the right RDS configuration for a scenario and explain the operational impact.

- Use **RDS** when you want a managed relational database with reduced administrative overhead.
- Use **Multi-AZ** when availability matters more than cost.
- Use **read replicas** when you need read scaling or asynchronous replication.
- Use **security groups** to control network access to the database.
- Use **automated backups** for point-in-time recovery.
- Use **snapshots** for manual backups, migration, and cloning.
- Use **KMS encryption** when data at rest must be protected.
- Use **private subnets** and no public access for the most secure default posture.

A simple mental model helps: RDS gives you the database service, but you still design the network path, access policy, backup strategy, and availability model.

**Quick reference checklist**

Before you finish the lab, confirm the following:

- The DB instance is in `available` status.
- The engine and port match your client.
- The security group allows inbound traffic from the correct source.
- The instance is in the intended VPC and subnet group.
- Public access is set appropriately for the lab.
- Backups and Multi-AZ settings match the scenario.
- You can connect successfully from an approved client.
