## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and operational context**

Amazon Relational Database Service (<abbr title="Relational Database Service">RDS</abbr>) backups and maintenance settings are core availability controls. In practice, you use them to reduce data-loss risk, control patch timing, and plan safe recovery paths during incidents.

This lab focuses on two related but distinct behaviors:

- **Backups** protect data and support point-in-time recovery (<abbr title="Point-in-Time Recovery">PITR</abbr>) within the retention period.
- **Maintenance windows** control when AWS applies routine operational changes such as engine patches and some host-level updates.

For exam purposes, always separate these concepts:

- Backups answer: “How far back can I restore?”
- Maintenance windows answer: “When might AWS apply changes that could cause a brief interruption?”

**RDS backup fundamentals**

RDS automated backups capture the database state plus transaction logs so you can restore to a specific time within the retention window. The retention period is configured per DB instance and is typically from `0` to `35` days, depending on engine and configuration. A value of `0` disables automated backups for standard RDS instances.

Important operational details:

- **Automated backups** are not the same as manual snapshots.
- **Manual snapshots** persist until you delete them.
- **Automated backups** expire after the retention period.
- **PITR** is available only while automated backups are retained.
- Backup storage is generally free up to the size of the provisioned database storage for many engines, but exact billing behavior can vary by engine and region.

> For exam questions, do not assume “backup retention” means the same thing for every database service. Aurora and standard RDS behave differently.

**Standard RDS versus Aurora backup behavior**

Standard RDS DB instances and Aurora clusters use different backup models.

- **Standard RDS**
  - Automated backups are configured on the DB instance.
  - Backups are tied to the instance’s retention setting.
  - Restores create a new DB instance from a snapshot or from a point in time.

- **Amazon Aurora**
  - Automated backups are configured at the **cluster** level.
  - Aurora continuously backs up cluster volume data and transaction activity.
  - Aurora restore operations create a new cluster, not an in-place overwrite.
  - Aurora does not use the same backup-window model as standard RDS in the same way; the cluster-level retention setting is the key control.

Practical implication:

- If a question mentions a **DB instance backup retention period**, think standard RDS.
- If it mentions a **cluster-level retention period** or Aurora restore, think Aurora cluster behavior.

**Backup retention settings and defaults**

When you configure automated backups, keep these rules in mind:

- Valid retention values are usually `0` to `35` days.
- Many RDS engines default to a short retention period if you do not specify one, but the exact default can vary by engine, creation method, and console workflow.
- Some engines and deployment patterns have constraints on backup retention or snapshot behavior.
- Automated backups are commonly enabled when you set retention greater than `0`, but the effective behavior also depends on how the DB instance is created and whether deletion protection or final snapshot settings are used.

Use longer retention when:

- You need a wider recovery window for accidental deletes or bad deployments.
- Compliance requires a longer restore history.
- You want more flexibility for forensic recovery.

Use shorter retention when:

- Storage cost is a concern.
- The database is noncritical and can be rebuilt quickly.
- You rely primarily on application-level backups or external replication.

**Maintenance window fundamentals**

The maintenance window is the preferred time range for AWS to apply routine maintenance. This can include:

- Engine version patches
- OS or host maintenance
- Minor infrastructure updates
- Some parameter-related changes that require a reboot

The maintenance window does not guarantee that every change happens only then, but it is the main scheduling control for disruptive routine maintenance.

Key points:

- The window is specified as a weekly time range.
- AWS uses UTC for RDS maintenance windows.
- The format is `ddd:hh24:mi-ddd:hh24:mi`, where `ddd` is a three-letter day abbreviation.
- Example: `sun:04:00-sun:04:30`

> Use a maintenance window that aligns with your lowest-traffic period, but remember that emergency fixes may still occur outside the window if AWS must address a critical issue.

**What applies immediately versus what is deferred**

Some RDS changes take effect immediately, while others are queued for the next maintenance window.

Typical patterns:

- **Immediate changes**
  - Some storage scaling operations
  - Certain parameter changes if the parameter is dynamic
  - Modifications where you explicitly choose immediate application

- **Deferred changes**
  - Engine patching
  - Reboots required by static parameter changes
  - Some instance class or configuration changes when you do not apply them immediately

In the console, many settings expose an **Apply immediately** option. If you clear it, AWS often defers the change until the next maintenance window.

Practical exam takeaway:

- If a change requires a reboot and you do not want downtime now, defer it.
- If you need the change to take effect immediately and can tolerate interruption, apply it now.

**Console workflow for configuring backups and maintenance**

Use the AWS Management Console when you want a quick, visual change to an existing DB instance.

1. Open the **Amazon RDS** console.
2. Choose **Databases**.
3. Select the DB instance you want to modify.
4. Choose **Modify**.
5. Review the **Backup** section:
   - Set **Backup retention period**.
   - Confirm whether automated backups are enabled for the instance.
6. Review the **Maintenance** section:
   - Set the **Preferred maintenance window**.
   - Decide whether to apply changes immediately or during the next window.
7. Review any reboot or downtime warnings.
8. Choose **Continue** and then **Modify DB instance**.

Useful field names you may see:

- **Backup retention period**
- **Preferred backup window**
- **Preferred maintenance window**
- **Apply immediately**
- **Deletion protection**
- **Create final snapshot**

The console is best when you need to validate settings interactively. Infrastructure as code is better when you want repeatability, version control, and environment parity.

**AWS CLI examples**

Use the AWS CLI when you need repeatable changes or want to script operational tasks.

Set backup retention and maintenance window on an existing DB instance:

```bash
aws rds modify-db-instance \
  --db-instance-identifier mydb \
  --backup-retention-period 7 \
  --preferred-backup-window "03:00-04:00" \
  --preferred-maintenance-window "sun:04:00-sun:04:30" \
  --apply-immediately
```

Describe the current settings:

```bash
aws rds describe-db-instances \
  --db-instance-identifier mydb \
  --query "DBInstances[0].{BackupRetentionPeriod:BackupRetentionPeriod,PreferredBackupWindow:PreferredBackupWindow,PreferredMaintenanceWindow:PreferredMaintenanceWindow,PendingModifiedValues:PendingModifiedValues}"
```

For Aurora, inspect the cluster-level backup retention:

```bash
aws rds describe-db-clusters \
  --db-cluster-identifier my-aurora-cluster \
  --query "DBClusters[0].{BackupRetentionPeriod:BackupRetentionPeriod,PreferredBackupWindow:PreferredBackupWindow,PreferredMaintenanceWindow:PreferredMaintenanceWindow}"
```

Restore a standard RDS instance from a snapshot:

```bash
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier restored-db \
  --db-snapshot-identifier mydb-snapshot-2026-04-26
```

Important restore behavior:

- Restoring from a snapshot creates a **new DB instance**.
- It does **not** overwrite the source instance.
- You usually need to update application endpoints after validation.

Restore to a point in time:

```bash
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier mydb \
  --target-db-instance-identifier mydb-restore-test \
  --restore-time "2026-04-26T10:15:00Z"
```

**Infrastructure as code examples**

Use infrastructure as code when you want consistent database settings across environments.

Terraform example for a standard RDS instance:

```hcl
resource "aws_db_instance" "app" {
  identifier              = "app-db"
  engine                  = "mysql"
  instance_class          = "db.t3.micro"
  allocated_storage       = 20
  username                = "adminuser"
  password                = var.db_password
  backup_retention_period = 7
  backup_window           = "03:00-04:00"
  maintenance_window      = "sun:04:00-sun:04:30"
  skip_final_snapshot     = false
  deletion_protection     = true
}
```

CloudFormation example for a deployable RDS instance:

```yaml
Resources:
  AppDB:
    Type: AWS::RDS::DBInstance
    Properties:
      DBInstanceIdentifier: app-db
      AllocatedStorage: 20
      DBInstanceClass: db.t3.micro
      Engine: mysql
      MasterUsername: adminuser
      MasterUserPassword: '{{resolve:secretsmanager:app-db-secret:SecretString:password}}'
      BackupRetentionPeriod: 7
      PreferredBackupWindow: 03:00-04:00
      PreferredMaintenanceWindow: sun:04:00-sun:04:30
      DeletionProtection: true
      PubliclyAccessible: false
```

Why use IaC here:

- You can standardize backup and maintenance policy across dev, test, and prod.
- You can review changes in code before deployment.
- You reduce drift between environments.
- You can recreate environments with the same operational settings.

**Restore workflow and validation**

A restore is only useful if you verify the new database before cutover. In production, treat restore as a controlled validation process.

1. Identify the recovery method:
   - Use **snapshot restore** for a known-good backup point.
   - Use **PITR** when you need the database just before a bad change.
2. Restore to a **new DB instance** or **new cluster**.
3. Confirm the restored resource uses the correct:
   - DB subnet group
   - Parameter group
   - Option group, if applicable
   - Security groups
   - KMS key for encrypted restores
4. Connect to the restored database and validate:
   - Schema presence
   - Row counts or sample queries
   - Application login and permissions
   - Read/write behavior
5. Compare endpoints and update the application only after validation.
6. Keep the original database intact until you confirm the restore is usable.

Example validation commands:

```bash
mysql -h restored-db.xxxxxx.us-east-1.rds.amazonaws.com -u adminuser -p
```

```sql
SHOW DATABASES;
SELECT COUNT(*) FROM orders;
```

Operational checks to perform before cutover:

- Confirm the restored instance is in `available` status.
- Confirm the endpoint resolves and accepts connections.
- Confirm parameter changes did not alter expected behavior.
- Confirm encryption and certificate settings match your client requirements.

**Encrypted restore considerations**

If the source DB instance is encrypted, the restore process must preserve encryption.

- The target restore uses the same AWS Key Management Service (<abbr title="Key Management Service">KMS</abbr>) key or a permitted key configuration.
- You must have permission to use the key.
- Cross-region or cross-account restore workflows may require additional key sharing or re-encryption steps.
- If the KMS key is unavailable or disabled, restore can fail.

Practical troubleshooting:

- If restore fails with an encryption-related error, check KMS permissions first.
- If the restored instance cannot be reached, verify the security group and subnet group.
- If the application breaks after restore, confirm the endpoint and parameter group settings.

**Maintenance actions and pending changes**

RDS exposes pending maintenance information so you can see what AWS plans to apply.

Use the CLI to review pending maintenance actions:

```bash
aws rds describe-pending-maintenance-actions
```

You can also inspect a specific resource:

```bash
aws rds describe-pending-maintenance-actions \
  --resource-identifier arn:aws:rds:us-east-1:123456789012:db:mydb
```

Typical maintenance actions may include:

- OS patching
- Database engine patching
- Hardware maintenance
- Certificate rotation-related updates

When you see pending maintenance:

- Review the impact and urgency.
- Decide whether to apply during the next window or immediately.
- Coordinate with application owners if a reboot is likely.

**Real-world decision scenario**

A DBA notices a bad schema migration caused application errors at 10:20 UTC.

Recommended response:

- If the migration is recent and automated backups are enabled, use **PITR** to a time just before the change.
- If the issue is isolated to a known snapshot point, use **snapshot restore**.
- If the database also has pending maintenance, defer it until after recovery unless the maintenance is urgent.
- Validate the restored database in a staging-like cutover path before switching production traffic.

This is the exam-relevant decision pattern:

- **PITR** for accidental changes with a known time boundary.
- **Snapshot restore** for a known backup artifact.
- **Maintenance deferral** when you want to avoid compounding downtime during recovery.

**Common pitfalls**

- Confusing backup windows with maintenance windows.
- Assuming restore overwrites the original database.
- Forgetting that Aurora backup behavior is cluster-based.
- Ignoring the need to update application endpoints after restore.
- Overlooking KMS permissions for encrypted databases.
- Setting a maintenance window without considering UTC.
- Assuming every change waits for the maintenance window; some changes can apply immediately.

**Exam focus summary**

- **Automated backups** enable PITR within the retention period.
- **Manual snapshots** are long-lived and must be deleted manually.
- **Standard RDS** uses instance-level backup settings.
- **Aurora** uses cluster-level backup behavior and restore semantics.
- **Maintenance windows** schedule routine updates, usually in UTC.
- **Restore operations** create new resources, so validation and endpoint updates are required.
- **IaC** tools like Terraform and CloudFormation are ideal for repeatable backup and maintenance configuration.

**Quick reference**

| Topic | Standard RDS | Aurora |
|---|---|---|
| Backup scope | DB instance | DB cluster |
| Restore target | New DB instance | New DB cluster |
| PITR | Yes, within retention | Yes, cluster-level continuous backups |
| Maintenance window | Preferred weekly window | Preferred weekly window at cluster level |
| Common exam trap | Thinking restore overwrites source | Treating Aurora like standard RDS backups |
