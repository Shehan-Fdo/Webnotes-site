## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Backup, recovery, and failover fundamentals**

Backup and failover are related but not identical. A **backup** preserves data so you can restore it later, while **failover** shifts traffic or workloads to a healthy resource when the primary fails. In AWS exam scenarios, you usually choose the least complex design that meets the required recovery objectives:

- **RPO** (Recovery Point Objective): how much data loss is acceptable.
- **RTO** (Recovery Time Objective): how long recovery can take.
- **Durability**: how likely data is to survive failure.
- **Availability**: how quickly the service stays usable.

A good architecture often combines multiple layers:

- **Local recovery** for fast restore from snapshots or AMIs.
- **Cross-AZ resilience** for infrastructure failure.
- **Cross-Region DR** for regional outages.
- **Application-level replication** for low RPO and faster failover.

**Backup consistency and recovery types**

Backups are not all the same. The exam may ask whether a backup is crash-consistent or application-consistent.

- **Crash-consistent** backups capture storage as if the instance lost power.
  - This is common for EBS snapshots taken without quiescing the file system.
  - It is usually sufficient for many workloads, but databases may need recovery on startup.
- **Application-consistent** backups capture data in a transactionally clean state.
  - You typically quiesce the application or use database-native backup tooling.
  - This is preferred for databases and critical transactional systems.

For example, before taking a file-system or database backup on Linux, you might flush buffers and pause writes:

```bash
sync
fsfreeze -f /data
aws ec2 create-snapshot --volume-id vol-0123456789abcdef0 --description "Pre-freeze backup"
fsfreeze -u /data
```

> `fsfreeze` is not appropriate for every workload. Use it only when the file system and application support it, and keep the freeze window short.

**Amazon EBS snapshots**

An **Amazon Elastic Block Store (EBS)** snapshot is a point-in-time backup of an EBS volume stored in Amazon S3-managed infrastructure. Snapshots are incremental after the first full backup, which reduces storage cost over time.

Key properties:

- A snapshot is tied to a **volume**, not permanently to an Availability Zone.
- You can create a new volume from a snapshot in any Availability Zone in the same Region where the snapshot exists.
- You can copy snapshots across Regions for disaster recovery.
- You can share snapshots with other AWS accounts if permissions allow it.
- Snapshots are commonly used for:
  - restoring a failed volume,
  - cloning environments,
  - creating AMIs,
  - long-term retention.

Typical workflow:

1. Create a snapshot of the source volume.
2. Copy the snapshot to another Region if you need cross-Region recovery.
3. Create a new volume from the snapshot in the target Availability Zone.
4. Attach the new volume to an instance.

Example CLI:

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Prod data volume backup"
```

Copying to another Region requires running the command against the destination Region, not by passing a destination-region parameter:

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --region us-west-2 \
  --description "Cross-Region copy for DR"
```

Create a new volume from the copied snapshot:

```bash
aws ec2 create-volume \
  --snapshot-id snap-0fedcba9876543210 \
  --availability-zone us-west-2a \
  --volume-type gp3
```

Important EBS exam concepts:

- **Incremental snapshots** reduce storage cost, but restore time can still depend on how much data must be retrieved.
- **Snapshot archive tier** is a lower-cost storage tier for long-term retention.
  - It is cheaper than standard snapshot storage.
  - Restore takes longer, so it is not ideal for frequent recovery.
- **Fast Snapshot Restore (FSR)** pre-warms snapshots in selected Availability Zones.
  - It reduces the latency of creating volumes from snapshots.
  - It is useful when you need rapid, predictable restores for critical workloads.
  - It increases cost, so use it selectively.

A practical decision rule:

- Use standard snapshots for routine backups and moderate recovery speed.
- Use archive tier for infrequently restored, long-retention backups.
- Use FSR when restore latency matters more than cost.

**Amazon Machine Images for instance recovery**

An **Amazon Machine Image (AMI)** is a template for launching EC2 instances. It includes:

- the root volume snapshot,
- launch permissions,
- block device mapping,
- optional instance configuration metadata.

AMIs are different from EBS snapshots:

- A **snapshot** backs up a single EBS volume.
- An **AMI** is used to recreate an EC2 instance configuration and its boot volume.
- AMIs are better for **instance recovery** and immutable server replacement.
- Snapshots are better for **data volume backup** and volume-level recovery.

Use an AMI when you want to restore the whole server quickly, including OS configuration and installed software. Use snapshots when you only need the data disk or when the instance is stateless and configuration is managed elsewhere.

Example AMI creation:

```bash
aws ec2 create-image \
  --instance-id i-0123456789abcdef0 \
  --name "webserver-prod-2026-04-26" \
  --no-reboot
```

`--no-reboot` can reduce disruption, but it may produce a less consistent image. If the workload is stateful, prefer a controlled shutdown or application quiescing.

**Amazon RDS backups and recovery**

Amazon Relational Database Service (RDS) supports automated backups, manual snapshots, and point-in-time recovery. The right choice depends on whether you need short-term restore, long-term retention, or cross-Region disaster recovery.

Core concepts:

- **Automated backups**
  - Enable point-in-time recovery within the retention window.
  - Include transaction logs for fine-grained restore points.
  - Are commonly used for operational recovery.
- **Manual snapshots**
  - Are user-initiated and retained until deleted.
  - Are useful for long-term retention or before major changes.
- **Point-in-time recovery**
  - Restores the database to a specific second within the backup window.
  - Is useful after accidental deletes or bad deployments.

RDS Multi-AZ and read replicas are not the same:

- **Multi-AZ**
  - Designed for high availability.
  - Uses synchronous replication for supported deployment models.
  - Provides automatic failover to a standby or another node depending on the engine and deployment type.
  - Is not primarily for read scaling.
- **Read replicas**
  - Use asynchronous replication.
  - Are primarily for read scaling and can also support disaster recovery patterns.
  - May be promoted manually during failover.
  - Have replication lag, so they are not ideal when zero or near-zero data loss is required.

For exam purposes, remember:

- Use **Multi-AZ** for automatic failover and high availability.
- Use **read replicas** for read scaling and cross-Region DR patterns.
- Do not assume read replicas provide automatic failover in the same way as Multi-AZ.

Cross-Region database recovery patterns:

- Copy automated backups or manual snapshots to another Region for restore.
- Use cross-Region read replicas where supported for lower RPO.
- Promote a replica during disaster recovery if the primary Region is unavailable.

Example snapshot copy:

```bash
aws rds copy-db-snapshot \
  --source-db-snapshot-identifier arn:aws:rds:us-east-1:123456789012:snapshot:prod-snap-2026-04-26 \
  --target-db-snapshot-identifier prod-snap-2026-04-26-copy \
  --source-region us-east-1 \
  --region us-west-2
```

**Amazon S3 replication and recovery**

Amazon S3 is highly durable, but replication is still useful for disaster recovery, compliance, and low-latency access in another Region.

Replication options:

- **Same-Region Replication (SRR)** for local redundancy or compliance workflows.
- **Cross-Region Replication (CRR)** for disaster recovery and geographic separation.
- **Replication Time Control (RTC)** for predictable replication timing and monitoring.

Important replication details:

- Replication requires versioning on both source and destination buckets.
- Existing objects are not replicated automatically; replication applies to new objects unless you use batch operations.
- Delete markers and delete behavior are configurable and should not be assumed to replicate in every setup.
- Replication is asynchronous, so there is always some lag.

Useful exam distinctions:

- S3 replication is not a backup substitute by itself if you need protection from accidental deletion, ransomware, or bad writes without proper versioning and retention controls.
- Versioning, object lock, and lifecycle policies often complement replication.
- RTC and replication metrics help when you need visibility into replication delay.

Example replication configuration snippet:

```json
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "Status": "Enabled",
      "Priority": 1,
      "DeleteMarkerReplication": {
        "Status": "Disabled"
      },
      "Filter": {
        "Prefix": "critical/"
      },
      "Destination": {
        "Bucket": "arn:aws:s3:::dr-bucket",
        "StorageClass": "STANDARD_IA"
      }
    }
  ]
}
```

**AWS Backup**

AWS Backup is a centralized backup service for multiple AWS resources. It helps you define backup policies once and apply them across supported services.

Commonly supported resource types include:

- Amazon EBS volumes
- Amazon RDS databases
- Amazon Aurora clusters
- Amazon EFS file systems
- Amazon DynamoDB tables
- AWS Storage Gateway volumes
- Amazon FSx file systems
- Some VMware workloads through AWS Backup for VMware integration

Use AWS Backup when you want centralized governance, policy-based scheduling, and cross-account or cross-Region copy management.

Important AWS Backup features:

- **Backup plans**
  - Define schedules, retention, and lifecycle rules.
  - Can include copy actions to another Region or account.
- **Backup vaults**
  - Store recovery points.
  - Can be protected with access policies.
  - Can be locked with vault lock for immutability controls.
- **Backup vault lock**
  - Helps enforce WORM-like retention behavior.
  - Prevents shortening retention after lock is enforced, depending on configuration.
  - Is useful for compliance and ransomware resilience.
- **Backup vault access policies**
  - Control who can write to or restore from a vault.
  - Are important for cross-account backup governance.
- **Copy rules**
  - Automatically copy backups to another vault, account, or Region.
  - Are a common DR pattern.

Example backup plan fragment:

```json
{
  "BackupPlanName": "daily-prod-backups",
  "Rules": [
    {
      "RuleName": "daily",
      "TargetBackupVaultName": "prod-vault",
      "ScheduleExpression": "cron(0 5 ? * * *)",
      "StartWindowMinutes": 60,
      "CompletionWindowMinutes": 180,
      "Lifecycle": {
        "MoveToColdStorageAfterDays": 30,
        "DeleteAfterDays": 365
      },
      "CopyActions": [
        {
          "DestinationBackupVaultArn": "arn:aws:backup:us-west-2:123456789012:backup-vault:dr-vault",
          "Lifecycle": {
            "DeleteAfterDays": 365
          }
        }
      ]
    }
  ]
}
```

Operationally, AWS Backup is strongest when you need:

- centralized policy enforcement,
- consistent retention across services,
- cross-account backup isolation,
- auditability for compliance.

**Route 53 failover and health checks**

Amazon Route 53 can route traffic away from unhealthy endpoints using health checks and failover routing policies. This is a common DNS-based disaster recovery pattern.

Key concepts:

- **Health checks** monitor endpoint health or CloudWatch alarms.
- **Failover routing** sends traffic to a primary record when healthy and a secondary record when the primary fails.
- **Alias records** can point to AWS resources such as load balancers, CloudFront distributions, and S3 website endpoints.
- **Evaluate target health** can integrate with AWS resource health for alias targets where supported.

Important limitations and behaviors:

- DNS failover is not instant because clients and resolvers cache responses according to TTL.
- Lower TTL values can reduce failover delay, but they do not eliminate caching.
- Health checks are only as good as what they can observe; they may not detect every application-layer failure.
- If the endpoint is behind a load balancer, health checks should target a meaningful path, not just the root page if that page can succeed while the app is broken.

Example failover record set:

```json
{
  "Comment": "Primary and secondary failover records",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "primary",
        "Failover": "PRIMARY",
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "alb-primary-123.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "secondary",
        "Failover": "SECONDARY",
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "alb-secondary-456.us-west-2.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

**Failover design patterns**

Choose the failover pattern based on how much downtime and data loss you can tolerate.

- **Backup and restore**
  - Lowest cost.
  - Highest RTO.
  - Suitable for non-critical workloads.
- **Pilot light**
  - Core services are running in the DR Region.
  - Faster than backup and restore.
  - Good when you need moderate RTO with controlled cost.
- **Warm standby**
  - A scaled-down version of the full environment is always running.
  - Faster recovery than pilot light.
  - Higher cost.
- **Multi-site active-active**
  - Traffic is served from multiple Regions or sites simultaneously.
  - Lowest RTO and often lowest perceived downtime.
  - Highest complexity and cost.

A simple decision guide:

- Use **backup and restore** when cost matters most.
- Use **pilot light** when you need a small always-on footprint.
- Use **warm standby** when you need faster recovery with moderate cost.
- Use **active-active** when availability requirements justify the complexity.

**Practical failover runbook**

A runbook helps you recover consistently during an incident. In exam scenarios, the best answer often includes automation, validation, and rollback planning.

1. Detect the failure using CloudWatch alarms, health checks, or application monitoring.
2. Stop writes to the primary system if possible to prevent split-brain or data divergence.
3. Promote or restore the latest consistent data copy.
4. Bring up compute in the recovery Region or Availability Zone.
5. Update DNS, load balancer targets, or application configuration.
6. Validate application health, data integrity, and user access.
7. Re-enable replication or backup jobs after the system stabilizes.
8. Document the incident and reconcile any lost or delayed data.

Example recovery sequence for an EC2-based application with EBS data:

```bash
# 1. Identify latest snapshot
aws ec2 describe-snapshots \
  --owner-ids self \
  --filters Name=volume-id,Values=vol-0123456789abcdef0 \
  --query 'Snapshots | sort_by(@, &StartTime)[-1].SnapshotId'

# 2. Copy snapshot to DR Region if needed
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --region us-west-2 \
  --description "DR restore copy"

# 3. Create volume in target AZ
aws ec2 create-volume \
  --snapshot-id snap-0fedcba9876543210 \
  --availability-zone us-west-2a \
  --volume-type gp3

# 4. Attach to replacement instance
aws ec2 attach-volume \
  --volume-id vol-0abc1234def567890 \
  --instance-id i-0abc1234def567890 \
  --device /dev/sdf
```

**Common exam traps**

- **Snapshot vs AMI**
  - A snapshot restores a volume.
  - An AMI restores an instance template and boot volume.
- **Multi-AZ vs read replica**
  - Multi-AZ is for failover.
  - Read replicas are for reads and asynchronous DR patterns.
- **Replication vs backup**
  - Replication copies changes quickly.
  - Backup preserves recoverability from corruption or deletion.
- **DNS failover**
  - Route 53 failover depends on health checks and caching.
  - It is not instantaneous.
- **Cross-Region restore**
  - You usually copy the backup or snapshot to the target Region first, then restore there.
- **Archive tier**
  - Cheaper storage does not mean faster recovery.
- **Fast Snapshot Restore**
  - Faster restore does not mean lower cost.

**How to choose the right recovery mechanism**

Use the following mental model:

- If you need to restore a single disk, use an **EBS snapshot**.
- If you need to recreate an EC2 server, use an **AMI**.
- If you need database point-in-time recovery, use **RDS automated backups**.
- If you need low-latency read scaling or DR replication, use **read replicas**.
- If you need automatic database failover, use **Multi-AZ**.
- If you need centralized backup governance, use **AWS Backup**.
- If you need traffic steering during outage, use **Route 53 failover**.
- If you need geographic redundancy for objects, use **S3 replication**.

**Quick comparison**

| Mechanism | Best for | Recovery speed | Data loss profile | Notes |
|---|---|---:|---|---|
| EBS snapshot | Volume backup | Medium | Depends on snapshot age | Can be copied cross-Region |
| AMI | EC2 instance recovery | Medium | Depends on image freshness | Includes boot volume and launch metadata |
| RDS Multi-AZ | High availability | Fast | Low | Synchronous for supported models |
| RDS read replica | Read scaling, DR | Medium | Possible lag | Asynchronous replication |
| S3 replication | Object DR | Medium | Possible lag | Use versioning and retention controls |
| AWS Backup | Centralized governance | Varies | Depends on schedule | Supports copy rules and vault controls |
| Route 53 failover | Traffic steering | Fast at DNS level | Depends on backend state | TTL and caching affect timing |

**Exam-ready summary**

- Use snapshots for point-in-time storage recovery.
- Use AMIs for whole-instance recovery.
- Use Multi-AZ for automatic failover and high availability.
- Use read replicas for asynchronous replication and read scaling.
- Use AWS Backup for centralized policy-based backup management.
- Use Route 53 health checks and failover routing for DNS-based traffic recovery.
- Use S3 replication for cross-Region object redundancy, but pair it with versioning and retention controls.
- Use archive tiers and lifecycle policies to reduce cost when restore speed is less important.
- Use Fast Snapshot Restore when restore latency is a critical requirement.
