## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and operational context**

This lab focuses on using the **AWS Command Line Interface (AWS CLI)** to inspect the health and backup state of an Amazon Relational Database Service (**Amazon RDS**) DB instance. In real operations, you use these checks to confirm whether a database is available, identify recent state changes, and verify that snapshots exist before maintenance, migration, or recovery work.

The main value of CLI-based inspection is speed and repeatability. You can query the same information from scripts, automation pipelines, or a terminal session without opening the console. That makes it useful for troubleshooting, incident response, and exam scenarios where you need to reason about instance state and backup readiness.

**Core RDS concepts to know**

Amazon RDS manages database infrastructure tasks such as provisioning, patching, backups, and snapshot creation. A DB instance can be in several states, and the state determines whether it can accept connections or administrative actions.

- **DB instance status** tells you whether the instance is `available`, `modifying`, `backing-up`, `storage-full`, `incompatible-network`, or another operational state.
- **DB snapshots** are point-in-time backups that you can use for restore operations or cloning.
- **Automated backups** are managed by RDS and support point-in-time recovery within the retention window.
- **Manual snapshots** are user-created and persist until you delete them.

A practical inspection workflow usually checks the instance first, then reviews snapshots, and finally looks at recent events if something seems wrong.

**Useful AWS CLI setup**

Before querying RDS, make sure your CLI is configured with the correct credentials, region, and output format. The region matters because RDS resources are regional.

```bash
aws configure
```

Typical configuration values include:

```text
AWS Access Key ID [None]: AKIA...
AWS Secret Access Key [None]: ...
Default region name [None]: us-east-1
Default output format [None]: json
```

You can also override the region per command:

```bash
aws rds describe-db-instances --region us-east-1
```

If you work across multiple accounts or roles, confirm the identity first:

```bash
aws sts get-caller-identity
```

This returns the account, user, or role currently in use, which helps avoid inspecting the wrong environment.

**Inspecting DB instance status**

The primary command for instance inspection is `describe-db-instances`. It returns metadata such as the DB instance identifier, engine, class, endpoint, status, storage allocation, and backup settings.

```bash
aws rds describe-db-instances \
  --db-instance-identifier mydbinstance \
  --query 'DBInstances[0].{Identifier:DBInstanceIdentifier,Status:DBInstanceStatus,Engine:Engine,Class:DBInstanceClass,Endpoint:Endpoint.Address,MultiAZ:MultiAZ,BackupRetention:BackupRetentionPeriod}' \
  --output table
```

This command is useful because it narrows the output to the fields you care about during a lab or troubleshooting session.

Common fields to inspect:

- **DBInstanceStatus** indicates whether the instance is ready or busy.
- **Engine** identifies the database engine, such as MySQL, PostgreSQL, MariaDB, Oracle, or SQL Server.
- **DBInstanceClass** shows the compute size.
- **MultiAZ** indicates whether the instance has a standby in another Availability Zone.
- **BackupRetentionPeriod** shows how long automated backups are retained.
- **Endpoint.Address** gives the connection endpoint when the instance is available.

If you want to inspect all instances in the account and region:

```bash
aws rds describe-db-instances --output table
```

For scripting, JSON output is often easier to parse:

```bash
aws rds describe-db-instances \
  --query 'DBInstances[].{Identifier:DBInstanceIdentifier,Status:DBInstanceStatus,Engine:Engine}' \
  --output json
```

**Interpreting instance states**

The status value tells you what kind of action is in progress or whether the instance is healthy.

- `available` means the DB instance is ready for connections.
- `modifying` means a configuration change is in progress.
- `backing-up` means RDS is creating a backup or snapshot-related operation is underway.
- `storage-full` means the allocated storage is exhausted and the instance may stop accepting writes.
- `rebooting` means the instance is restarting.
- `failed` or `incompatible-*` states usually require investigation and may block normal use.

> If the instance is not `available`, do not assume the endpoint is usable. Always confirm the state before testing application connectivity.

**Checking recent DB events**

When the status alone is not enough, inspect RDS events. Events often explain why an instance changed state, rebooted, failed a maintenance action, or completed a backup.

```bash
aws rds describe-events \
  --source-type db-instance \
  --source-identifier mydbinstance \
  --duration 1440 \
  --output table
```

Key points:

- **source-type** should be `db-instance` for instance-level events.
- **source-identifier** is the DB instance identifier.
- **duration** is measured in minutes and controls how far back you query.

Events are especially useful when the instance is stuck in a transitional state or when a snapshot operation did not behave as expected.

**Inspecting snapshots**

Use `describe-db-snapshots` to list snapshots for a DB instance. This helps you confirm whether backups exist before making changes or restoring data.

```bash
aws rds describe-db-snapshots \
  --db-instance-identifier mydbinstance \
  --output table
```

You can filter by snapshot type:

```bash
aws rds describe-db-snapshots \
  --db-instance-identifier mydbinstance \
  --snapshot-type manual \
  --output table
```

Snapshot types commonly include:

- **manual** snapshots created by you.
- **automated** snapshots created by RDS.
- **shared** snapshots shared from another account.
- **public** snapshots, which are uncommon and should be used carefully.

Useful fields to inspect:

- **DBSnapshotIdentifier** identifies the snapshot.
- **SnapshotCreateTime** shows when it was taken.
- **Status** indicates whether it is `available`, `creating`, or another state.
- **Engine** confirms the database engine.
- **AllocatedStorage** shows the storage size at snapshot time.

A targeted query can make snapshot review easier:

```bash
aws rds describe-db-snapshots \
  --db-instance-identifier mydbinstance \
  --query 'DBSnapshots[].{Snapshot:DBSnapshotIdentifier,Created:SnapshotCreateTime,Status:Status,Type:SnapshotType}' \
  --output table
```

**Automated backups and point-in-time recovery**

Automated backups are not the same as manual snapshots. They support **point-in-time recovery (PITR)** within the retention period, which means you can restore the database to a specific time before a failure or bad change.

To inspect backup retention on the instance:

```bash
aws rds describe-db-instances \
  --db-instance-identifier mydbinstance \
  --query 'DBInstances[0].{Identifier:DBInstanceIdentifier,BackupRetention:BackupRetentionPeriod,PreferredBackupWindow:PreferredBackupWindow}' \
  --output table
```

What to verify:

- **BackupRetentionPeriod** is greater than `0` if you want automated backups.
- **PreferredBackupWindow** shows when backups usually occur.
- **MultiAZ** can improve availability, but it is separate from backup retention.

If retention is `0`, automated backups are disabled and PITR is not available.

**Restoration-related inspection**

Before restoring from a snapshot, confirm that the snapshot is `available` and that it matches the engine and major version requirements of the target restore operation.

```bash
aws rds describe-db-snapshots \
  --db-snapshot-identifier mydbsnapshot \
  --query 'DBSnapshots[0].{Snapshot:DBSnapshotIdentifier,Status:Status,Engine:Engine,Version:EngineVersion,Created:SnapshotCreateTime}' \
  --output table
```

Important considerations:

- The snapshot must be in an `available` state.
- The target restore instance must support the engine and version.
- Some restore operations may require parameter group or subnet group adjustments.
- Cross-region or cross-account workflows may require snapshot copy or sharing first.

**Practical CLI patterns**

Use `--query` to reduce noise and `--output table` for quick human review. Use `--output json` when you need to feed results into scripts.

```bash
aws rds describe-db-instances \
  --query 'DBInstances[].{ID:DBInstanceIdentifier,Status:DBInstanceStatus,AZ:AvailabilityZone,Storage:AllocatedStorage}' \
  --output table
```

```bash
aws rds describe-db-snapshots \
  --query 'DBSnapshots[].{ID:DBSnapshotIdentifier,Instance:DBInstanceIdentifier,Status:Status,Time:SnapshotCreateTime}' \
  --output json
```

You can also use JMESPath filtering to focus on one state:

```bash
aws rds describe-db-instances \
  --query 'DBInstances[?DBInstanceStatus==`available`].[DBInstanceIdentifier,Endpoint.Address]' \
  --output table
```

This is helpful when you want to list only healthy instances.

**Troubleshooting checklist**

When a DB instance or snapshot does not look right, work through the checks in order.

1. Confirm the AWS identity and region with `aws sts get-caller-identity` and the configured region.
2. Query the DB instance with `describe-db-instances`.
3. Check the instance status and backup retention.
4. Review recent events with `describe-events`.
5. List snapshots with `describe-db-snapshots`.
6. Verify whether the snapshot is `available` before attempting restore.
7. If the instance is not healthy, inspect whether storage, maintenance, or network settings are involved.

Common issues to watch for:

- Wrong region, which makes the instance appear missing.
- Wrong identifier, which returns no results.
- Snapshot still `creating`, which means restore is not yet possible.
- Backup retention set to `0`, which disables automated backups.
- Instance in `modifying` state after a configuration change.

**Example operational workflow**

A simple real-world workflow might look like this:

```bash
aws sts get-caller-identity
aws rds describe-db-instances \
  --db-instance-identifier prod-db \
  --query 'DBInstances[0].{Status:DBInstanceStatus,Endpoint:Endpoint.Address,BackupRetention:BackupRetentionPeriod}' \
  --output table
aws rds describe-events \
  --source-type db-instance \
  --source-identifier prod-db \
  --duration 60 \
  --output table
aws rds describe-db-snapshots \
  --db-instance-identifier prod-db \
  --query 'DBSnapshots[].{Snapshot:DBSnapshotIdentifier,Status:Status,Created:SnapshotCreateTime}' \
  --output table
```

This sequence gives you a fast operational snapshot of the database: identity, health, recent activity, and backup inventory.

**Exam and lab takeaways**

For exam purposes, remember the distinction between instance health and backup state.

- **describe-db-instances** is your primary command for instance status and configuration.
- **describe-db-snapshots** is your primary command for snapshot inventory.
- **describe-events** helps explain state changes and operational history.
- **Automated backups** support PITR, while **manual snapshots** are explicit backups you manage.
- Region and identifier accuracy are essential when using the AWS CLI.

If you can quickly determine whether a DB instance is `available`, whether backups are enabled, and whether a valid snapshot exists, you can answer most lab-style RDS inspection questions with confidence.
