## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**CloudTrail purpose and exam framing**

AWS CloudTrail records AWS API activity so you can answer who did what, when, from where, and with what result. For the exam, treat it as the primary service for **API auditing**, security investigation, and governance evidence across AWS accounts.

CloudTrail is not the same as a metrics or application logging service. It captures control-plane activity and selected data-plane activity, then stores events in S3, streams them to CloudWatch Logs if configured, and can query them in CloudTrail Lake.

- **Management events** record control-plane actions such as creating an EC2 instance, changing an IAM policy, or updating an S3 bucket policy.
- **Data events** record high-volume resource activity such as S3 object-level access or Lambda function invocation.
- **Insights events** help detect unusual API activity patterns, such as spikes in `Write` calls.

> CloudTrail is enabled by default for recent management event history in the console, but that does not replace creating a trail for durable S3 archival, retention, and centralized governance.

**CloudTrail event anatomy**

CloudTrail events are JSON records. On the exam, you should recognize the common fields and what they tell you during incident response or audit review.

- `eventTime` shows when the API call occurred.
- `eventSource` identifies the AWS service, such as `ec2.amazonaws.com`, `iam.amazonaws.com`, or `s3.amazonaws.com`.
- `eventName` identifies the action, such as `RunInstances`, `CreateUser`, or `PutObject`.
- `userIdentity` identifies the actor, such as an IAM user, assumed role, federated user, AWS service, or root user.
- `sourceIPAddress` shows the origin of the request, which may be an IP address, AWS service endpoint, or internal AWS value.
- `requestParameters` contains the request payload details, such as the bucket name, instance type, or policy document.
- `responseElements` contains the API response details when available.
- `errorCode` and `errorMessage` show failures, such as `AccessDenied` or `Client.UnauthorizedOperation`.

A simplified event looks like this:

```json
{
  "eventTime": "2026-04-26T12:34:56Z",
  "eventSource": "ec2.amazonaws.com",
  "eventName": "RunInstances",
  "awsRegion": "us-east-1",
  "sourceIPAddress": "203.0.113.10",
  "userIdentity": {
    "type": "AssumedRole",
    "arn": "arn:aws:sts::123456789012:assumed-role/AdminRole/Alice"
  },
  "requestParameters": {
    "instanceType": "t3.micro",
    "minCount": 1,
    "maxCount": 1
  },
  "responseElements": {
    "instancesSet": {
      "items": [
        {
          "instanceId": "i-0123456789abcdef0"
        }
      ]
    }
  },
  "errorCode": null
}
```

**Management events, read-only vs write-only**

Management events are split into **read-only** and **write-only** categories. This distinction matters for cost control and for reducing noise.

- **Read-only management events** include actions like `DescribeInstances`, `ListBuckets`, and `GetCallerIdentity`.
- **Write-only management events** include actions like `RunInstances`, `CreateBucket`, `PutRolePolicy`, and `DeleteTrail`.

For many workloads, you keep write events for security and change tracking, while excluding read-only events if you want lower volume. Read-only events can still be useful for investigations, but they often generate much more traffic.

**Event selectors and cost control**

Event selectors define which events a trail records. Use them to scope logging and control cost.

- **Management events** can be included as all events or write-only events.
- **Data events** are selected separately and usually only for specific resources.
- **Advanced event selectors** let you filter by fields such as event source, event name, resource type, and read/write type.

A common exam scenario is choosing between broad visibility and lower cost. If you only need audit evidence for changes, select write-only management events. If you need object access or function invocation tracing, add targeted data event selectors.

Example CLI for a trail that logs write-only management events:

```bash
aws cloudtrail put-event-selectors \
  --trail-name prod-audit-trail \
  --event-selectors '[
    {
      "ReadWriteType": "WriteOnly",
      "IncludeManagementEvents": true,
      "DataResources": []
    }
  ]'
```

Example advanced selector for S3 object-level logging on a specific bucket:

```bash
aws cloudtrail put-event-selectors \
  --trail-name prod-audit-trail \
  --advanced-event-selectors '[
    {
      "Name": "S3ObjectWrites",
      "FieldSelectors": [
        { "Field": "eventCategory", "Equals": ["Data"] },
        { "Field": "resources.type", "Equals": ["AWS::S3::Object"] },
        { "Field": "resources.ARN", "StartsWith": ["arn:aws:s3:::my-audit-bucket/"] },
        { "Field": "readOnly", "Equals": ["false"] }
      ]
    }
  ]'
```

**Data events and when to enable them**

Data events are optional and can become expensive because they are high volume. Enable them only for resources that matter to your audit or security use case.

- **S3 object-level logging** records actions such as `GetObject`, `PutObject`, and `DeleteObject`.
- **Lambda invocation logging** records function invoke activity.
- Other supported data event sources may include some AWS services that expose object or resource-level access patterns.

Use data events when you need to answer questions like:
- Who downloaded a sensitive object from S3?
- Which principal invoked a Lambda function?
- Was a specific object overwritten or deleted?

Example CLI for enabling S3 data events on one bucket:

```bash
aws cloudtrail put-event-selectors \
  --trail-name prod-audit-trail \
  --event-selectors '[
    {
      "ReadWriteType": "All",
      "IncludeManagementEvents": true,
      "DataResources": [
        {
          "Type": "AWS::S3::Object",
          "Values": ["arn:aws:s3:::my-sensitive-bucket/"]
        }
      ]
    }
  ]'
```

Example advanced selector for Lambda invocation events:

```bash
aws cloudtrail put-event-selectors \
  --trail-name prod-audit-trail \
  --advanced-event-selectors '[
    {
      "Name": "LambdaInvocations",
      "FieldSelectors": [
        { "Field": "eventCategory", "Equals": ["Data"] },
        { "Field": "resources.type", "Equals": ["AWS::Lambda::Function"] },
        { "Field": "readOnly", "Equals": ["false"] }
      ]
    }
  ]'
```

**Trail types: single-region, multi-region, and organization trails**

A **single-region trail** records events in one AWS Region. It is useful when your compliance scope is regional or when you only need local visibility.

A **multi-region trail** records events from all Regions in the account. For the exam, this is usually the recommended default because attackers and administrators can act in any Region, and many services are global or used cross-region.

- Multi-region trails reduce the risk of missing activity in a Region you forgot to monitor.
- They simplify centralized auditing and incident response.
- They are especially useful when workloads span multiple Regions or when you want one consistent audit trail.

Global services are a common nuance:
- IAM, STS, and CloudFront are global services.
- CloudTrail still records their events, and those events are associated with the home Region behavior of the service or the trail’s delivery model.
- For exam questions, remember that a multi-region trail helps ensure you do not miss these service events when activity is not tied to a single application Region.

An **organization trail** is created in AWS Organizations and logs activity across accounts in the organization. Use it when you need centralized governance and a single audit destination for multiple accounts.

Example CLI for creating a multi-region trail:

```bash
aws cloudtrail create-trail \
  --name org-audit-trail \
  --s3-bucket-name my-cloudtrail-logs-bucket \
  --is-multi-region-trail
```

Example CLI for creating an organization trail:

```bash
aws cloudtrail create-trail \
  --name org-audit-trail \
  --s3-bucket-name my-cloudtrail-logs-bucket \
  --is-organization-trail
```

**Log storage, encryption, and access control**

CloudTrail delivers logs to Amazon S3. The S3 bucket should be locked down so only the right security and audit roles can read the logs.

- Use **SSE-S3** for simple server-side encryption managed by Amazon S3.
- Use **SSE-KMS** when you need tighter key control, key rotation, and stronger compliance alignment.
- Restrict bucket access with bucket policies and least-privilege IAM permissions.
- Consider S3 Object Lock if you need write-once-read-many style retention controls, where supported by your design.

A typical bucket policy allows the CloudTrail service to write logs and denies public access. Example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSCloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs-bucket"
    },
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs-bucket/AWSLogs/123456789012/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control"
        }
      }
    }
  ]
}
```

**Digest files and log validation**

CloudTrail can generate digest files that support log integrity validation. This is important when you need to prove logs were not altered after delivery.

The workflow is:
1. CloudTrail writes log files to S3.
2. CloudTrail writes digest files that contain hashes and references to the log files.
3. You validate the chain of files to confirm integrity.

Example validation command:

```bash
aws cloudtrail validate-logs \
  --trail-arn arn:aws:cloudtrail:us-east-1:123456789012:trail/prod-audit-trail \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T23:59:59Z
```

Use validation when you need tamper-evidence for compliance or incident response. If the exam asks how to verify that CloudTrail logs were not modified, think digest files and validation.

**CloudTrail Lake and when to use it**

CloudTrail Lake is a managed data lake for CloudTrail events. It is useful when you want SQL-based analysis over event history without building your own ingestion pipeline.

Use CloudTrail Lake when you need:
- Ad hoc querying across large event sets
- Longer-term analysis and investigation
- SQL-style searches for security operations or audit teams

Use standard trails when you need:
- S3 archival
- Integration with existing log retention controls
- Simple delivery to a bucket for compliance or downstream processing

A practical example query:

```sql
SELECT eventTime, eventName, userIdentity.arn, sourceIPAddress
FROM $EDS_ID
WHERE eventSource = 'iam.amazonaws.com'
  AND eventName = 'CreateUser'
ORDER BY eventTime DESC
LIMIT 20;
```

**CloudTrail vs CloudWatch vs AWS Config vs CloudTrail Lake**

These services are often tested together, so choose based on the question.

| Service | Best for | What it records | Typical exam clue |
|---|---|---|---|
| CloudTrail | API auditing and governance | AWS API calls and selected data events | “Who changed this?” |
| CloudWatch | Metrics, logs, alarms, and operational monitoring | Application logs, metrics, alarms | “Alert me when CPU is high” |
| AWS Config | Resource configuration history and compliance | Configuration snapshots and change history | “Was this resource compliant?” |
| CloudTrail Lake | Querying and analyzing CloudTrail events | CloudTrail event data in a queryable store | “Search API activity across time” |

A useful mental model:
- Use **CloudTrail** to know who made the API call.
- Use **CloudWatch** to know how the system is performing.
- Use **AWS Config** to know how the resource configuration changed over time.
- Use **CloudTrail Lake** to query the audit trail at scale.

**Alerting patterns and integrations**

CloudTrail itself is not primarily an alerting service, but it can feed alerting workflows.

- Send CloudTrail logs to **CloudWatch Logs** and create metric filters and alarms.
- Use **EventBridge** rules for near-real-time reactions to specific API events.
- Use **SNS** to notify security teams when a suspicious event occurs.

Example EventBridge pattern for root activity:

```json
{
  "source": ["aws.cloudtrail"],
  "detail-type": ["AWS API Call via CloudTrail"],
  "detail": {
    "userIdentity": {
      "type": ["Root"]
    }
  }
}
```

Example SNS-based workflow:
- EventBridge matches `DeleteTrail` or `StopLogging`.
- The rule targets an SNS topic.
- Security receives an email or downstream automation trigger.

**Practical investigation workflow**

When you investigate an incident, start with the event fields and work outward.

1. Identify the suspicious `eventName` and `eventSource`.
2. Check `eventTime` to build a timeline.
3. Inspect `userIdentity` to determine the principal type and ARN.
4. Review `sourceIPAddress` to understand origin.
5. Examine `requestParameters` for the exact resource or payload.
6. Check `responseElements` for created resource IDs or returned values.
7. Look at `errorCode` and `errorMessage` to see whether the action succeeded.

Example investigation query in CloudTrail Lake or Athena-style analysis:

```sql
SELECT eventTime, eventName, userIdentity.arn, sourceIPAddress, errorCode
FROM cloudtrail_logs
WHERE eventSource = 'ec2.amazonaws.com'
  AND eventName IN ('RunInstances', 'TerminateInstances')
  AND eventTime >= TIMESTAMP '2026-04-26 00:00:00'
ORDER BY eventTime DESC;
```

**Common exam scenarios**

- You need to know who deleted an S3 bucket policy: use CloudTrail management events.
- You need to track object downloads from a sensitive bucket: enable S3 data events.
- You need organization-wide audit visibility: use an organization multi-region trail.
- You need to detect unusual API spikes: use CloudTrail Insights.
- You need compliance evidence that logs were not tampered with: use digest validation.
- You need to query historical API activity quickly: use CloudTrail Lake.
- You need operational alarms for application failures: use CloudWatch, not CloudTrail.
- You need resource drift and configuration compliance: use AWS Config.

**Key configuration reminders**

- Enable trails for durable retention; console event history alone is not enough for long-term audit needs.
- Prefer multi-region trails unless you have a strong reason to scope to one Region.
- Log write-only management events when you want lower noise and lower cost.
- Add data events only for the specific S3 buckets or Lambda functions that matter.
- Protect the S3 log bucket with least privilege and encryption.
- Validate logs when integrity matters.

**Quick CLI reference**

Create and start a trail:

```bash
aws cloudtrail create-trail \
  --name prod-audit-trail \
  --s3-bucket-name my-cloudtrail-logs-bucket \
  --is-multi-region-trail

aws cloudtrail start-logging --name prod-audit-trail
```

Describe trails:

```bash
aws cloudtrail describe-trails
```

Lookup recent events:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=CreateUser
```
