## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**CloudWatch alarm fundamentals**

Amazon CloudWatch alarms evaluate a metric over time and change state when the metric crosses a threshold. In this lab, you typically create alarms for either an Amazon Elastic Compute Cloud (<abbr title="Elastic Compute Cloud">EC2</abbr>) instance or an Application Load Balancer (<abbr title="Application Load Balancer">ALB</abbr>) target group or load balancer metric. The goal is to detect operational issues early and route notifications through Amazon Simple Notification Service (<abbr title="Simple Notification Service">SNS</abbr>).

A CloudWatch alarm has three states:

- **OK**: the metric is within the expected range.
- **ALARM**: the threshold is breached.
- **INSUFFICIENT_DATA**: CloudWatch does not have enough datapoints to evaluate the alarm.

For exam purposes, focus on the relationship between the metric, the statistic, the period, and the evaluation periods. A poorly chosen period can delay detection, while an overly sensitive threshold can create noisy alerts.

**EC2 metrics you commonly alarm on**

EC2 publishes basic host metrics automatically at 5-minute intervals, and detailed monitoring can increase resolution to 1 minute. You usually alarm on metrics that indicate availability or saturation rather than raw instance identity.

Common EC2 alarm candidates include:

- **CPUUtilization**: useful for sustained compute pressure.
- **StatusCheckFailed**: indicates instance or system reachability issues.
- **StatusCheckFailed_Instance**: points to guest OS or instance-level problems.
- **StatusCheckFailed_System**: points to underlying AWS infrastructure issues.
- **NetworkIn** and **NetworkOut**: useful for traffic trend monitoring, but less commonly used as primary health alarms.

A practical EC2 alarm often uses `CPUUtilization > 80` for several consecutive periods, or `StatusCheckFailed >= 1` for immediate failure detection.

**ALB metrics you commonly alarm on**

Application Load Balancer metrics help you detect user-facing issues before they become outages. ALB metrics are published in CloudWatch and are especially useful when you want to monitor request quality, backend health, or traffic anomalies.

Common ALB alarm candidates include:

- **HTTPCode_ELB_5XX_Count**: errors generated by the load balancer itself.
- **HTTPCode_Target_5XX_Count**: errors returned by targets.
- **TargetResponseTime**: backend latency.
- **HealthyHostCount**: number of healthy targets in a target group.
- **UnHealthyHostCount**: number of unhealthy targets in a target group.
- **RequestCount**: traffic volume, often used with anomaly detection or baseline comparisons.

For ALB, alarms are often more meaningful when scoped to a specific load balancer or target group dimension rather than the entire service.

**Alarm design choices**

CloudWatch alarms are not just threshold checks; they are operational policies. You choose the metric namespace, dimensions, statistic, period, threshold, and action targets.

Key design decisions:

- **Statistic**
  - Use `Average` for CPU or latency trends.
  - Use `Sum` for count-based metrics such as error counts.
  - Use `Minimum` or `Maximum` when you care about the worst observed value in a period.
- **Period**
  - Short periods detect issues faster but can be noisy.
  - Longer periods smooth spikes but delay response.
- **Evaluation periods**
  - Multiple periods reduce false positives.
  - A single period is appropriate for hard failure signals like status checks.
- **Datapoints to alarm**
  - You can require a subset of datapoints to breach the threshold, which helps tolerate brief spikes.
- **Treat missing data**
  - `missing`, `breaching`, `notBreaching`, or `ignore` can change behavior when telemetry stops.

A common pattern is `3 out of 3` periods breaching for CPU alarms and `1 out of 1` for status check alarms.

**Creating an alarm in the console**

You usually create the alarm from the CloudWatch console by selecting a metric, defining the threshold, and attaching an action.

Typical workflow:

1. Open **CloudWatch** and go to **Alarms**.
2. Choose **Create alarm**.
3. Select a metric namespace such as **EC2** or **ApplicationELB**.
4. Choose the specific metric and dimensions.
5. Configure the statistic, period, and threshold.
6. Select or create an SNS topic for notifications.
7. Name the alarm and create it.

A practical EC2 example might use the following settings:

- Metric: `CPUUtilization`
- Statistic: `Average`
- Period: `5 minutes`
- Threshold type: static
- Condition: `Greater than 80`
- Evaluation periods: `3`
- Datapoints to alarm: `3`

A practical ALB example might use:

- Metric: `HTTPCode_Target_5XX_Count`
- Statistic: `Sum`
- Period: `1 minute`
- Threshold: `>= 1`
- Evaluation periods: `1`
- Datapoints to alarm: `1`

**CLI examples for alarm creation**

You can create alarms with the AWS Command Line Interface (<abbr title="Command Line Interface">CLI</abbr>) using `put-metric-alarm`. This is useful for automation, repeatability, and infrastructure as code workflows.

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name ec2-high-cpu-prod \
  --alarm-description "Alarm when EC2 CPU exceeds 80 percent for 15 minutes" \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistic Average \
  --period 300 \
  --evaluation-periods 3 \
  --datapoints-to-alarm 3 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --treat-missing-data missing \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-alerts
```

For an ALB target group error alarm:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name alb-target-5xx-prod \
  --alarm-description "Alarm when target group returns 5XX errors" \
  --namespace AWS/ApplicationELB \
  --metric-name HTTPCode_Target_5XX_Count \
  --dimensions Name=LoadBalancer,Value=app/my-alb/50dc6c495c0c9188 \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 1 \
  --datapoints-to-alarm 1 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --treat-missing-data notBreaching \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-alerts
```

> Use the exact dimension names required by the metric. For ALB metrics, the dimension may be `LoadBalancer` or `TargetGroup` depending on the metric you choose.

**SNS notification setup**

CloudWatch alarms commonly notify an SNS topic, which can fan out to email, SMS, Lambda, or other subscribers. This makes alarms actionable instead of merely visible.

A typical SNS setup includes:

- Create an SNS topic such as `ops-alerts`.
- Subscribe an email address or automation endpoint.
- Confirm the subscription before relying on it.
- Attach the topic ARN to the alarm action.

Example SNS topic creation:

```bash
aws sns create-topic --name ops-alerts
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:ops-alerts \
  --protocol email \
  --notification-endpoint admin@example.com
```

Alarm actions can also be configured for state transitions:

- **Alarm actions**: triggered when the alarm enters `ALARM`.
- **OK actions**: triggered when the alarm returns to `OK`.
- **Insufficient data actions**: triggered when the alarm enters `INSUFFICIENT_DATA`.

**Testing and validation**

After creating the alarm, validate that it behaves as expected. In a lab environment, you can generate load, stop an instance, or temporarily adjust thresholds to force a state change.

Practical validation approaches:

- Increase CPU load on an EC2 instance with a stress tool.
- Stop or impair the instance to trigger status checks.
- Generate backend errors behind an ALB.
- Temporarily lower the threshold to confirm notification delivery.

Example CPU stress command on Linux:

```bash
sudo yum install -y stress-ng || sudo apt-get update && sudo apt-get install -y stress-ng
stress-ng --cpu 2 --cpu-load 90 --timeout 10m
```

If you do not want to install a tool, you can use a simple loop to create load:

```bash
while true; do :; done
```

For ALB testing, you can simulate target failures by stopping the application process or deregistering a target from the target group. Then watch the relevant metric and alarm state in CloudWatch.

**Troubleshooting alarm behavior**

Alarm issues usually come from metric selection, dimensions, or timing rather than the alarm service itself.

Common problems:

- **Wrong dimension**
  - The alarm never receives datapoints because the metric is scoped to a different resource.
- **Incorrect statistic**
  - Using `Average` on a count metric can hide spikes.
- **Period mismatch**
  - The alarm evaluates too slowly or too frequently for the metric resolution.
- **Missing data handling**
  - A quiet metric may remain in `INSUFFICIENT_DATA` if missing data is not treated appropriately.
- **Notification misconfiguration**
  - The SNS topic exists, but the subscription is unconfirmed or the IAM permissions are incomplete.

A useful debugging workflow is:

1. Open the metric graph and confirm datapoints exist.
2. Verify the resource dimensions match the actual EC2 instance or ALB.
3. Check the alarm history for state transitions.
4. Confirm the SNS topic subscription is active.
5. Review IAM permissions if automation or cross-account delivery is involved.

**EC2 versus ALB alarm strategy**

EC2 alarms focus on host health and resource saturation, while ALB alarms focus on request quality and backend behavior. In a well-designed architecture, you often use both.

| Aspect | EC2 alarm focus | ALB alarm focus |
|---|---|---|
| Primary signal | Instance health and CPU | Request errors and latency |
| Common metric | `CPUUtilization`, `StatusCheckFailed` | `HTTPCode_Target_5XX_Count`, `TargetResponseTime` |
| Best use | Detect host issues | Detect user-facing service degradation |
| Typical threshold style | Static threshold | Static or anomaly-based threshold |
| Scope | Single instance | Load balancer or target group |

Use EC2 alarms when you need to know whether the compute node is healthy. Use ALB alarms when you need to know whether the service is delivering successful responses to clients.

**Exam-relevant takeaways**

CloudWatch alarms are a core monitoring and response mechanism in AWS. For the exam, remember these points:

- **CloudWatch alarms** evaluate metrics against thresholds and can trigger SNS notifications.
- **EC2 status check alarms** are strong indicators of instance or infrastructure failure.
- **ALB metrics** are useful for detecting backend errors, latency, and unhealthy targets.
- **Statistic and period selection** directly affect sensitivity and noise.
- **Missing data behavior** matters when metrics stop publishing.
- **SNS integration** is the standard way to notify operators or trigger automation.

A strong architecture uses alarms as part of an operational feedback loop: detect, notify, investigate, and remediate.
