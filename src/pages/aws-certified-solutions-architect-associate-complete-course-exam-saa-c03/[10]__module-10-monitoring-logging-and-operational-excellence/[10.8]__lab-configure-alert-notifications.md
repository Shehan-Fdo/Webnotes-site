## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Alert notification architecture**

AWS alert notifications usually combine <abbr title="Amazon CloudWatch">CloudWatch</abbr> alarms with <abbr title="Amazon Simple Notification Service">Amazon SNS</abbr> topics. CloudWatch evaluates metrics, triggers an alarm when a threshold is crossed, and SNS delivers the message to one or more subscribers such as email, SMS, or downstream automation.

This pattern is common in production because it separates detection from delivery:

- CloudWatch handles metric evaluation and alarm state changes.
- SNS handles fan-out to multiple endpoints.
- You can add or remove subscribers without changing the alarm itself.

A typical flow looks like this:

```text
CloudWatch metric -> CloudWatch alarm -> SNS topic -> email/SMS/Lambda/HTTP endpoint
```

**Core concepts**

<dl>
<dt><strong>Metric</strong></dt>
<dd>A time-series value published by AWS services, custom applications, or agents.</dd>
<dt><strong>Alarm</strong></dt>
<dd>A CloudWatch rule that watches a metric and changes state when conditions are met.</dd>
<dt><strong>Topic</strong></dt>
<dd>An SNS resource that receives messages and forwards them to subscribers.</dd>
<dt><strong>Subscription</strong></dt>
<dd>A delivery target attached to a topic, such as an email address or Lambda function.</dd>
</dl>

CloudWatch alarms are stateful. They move between `OK`, `ALARM`, and `INSUFFICIENT_DATA`. For alerting, the important part is that the alarm can publish a notification whenever it enters `ALARM`, and optionally when it returns to `OK`.

**When to use this pattern**

Use CloudWatch plus SNS when you need:

- Threshold-based alerting for CPU, memory, latency, errors, or queue depth.
- Human notification by email or SMS.
- Automated response through Lambda or Systems Manager.
- One alarm to notify multiple teams or tools.

Use it less often when the event source already has built-in event routing, or when you need complex event filtering better suited to EventBridge.

**Step-by-step lab flow**

1. Create an SNS topic for alerts.
2. Subscribe an email address to the topic.
3. Confirm the subscription from the email inbox.
4. Create a CloudWatch alarm on a relevant metric.
5. Attach the SNS topic as the alarm action.
6. Generate a condition that crosses the threshold.
7. Verify the notification arrives and the alarm state changes.

**Creating an SNS topic**

You can create a topic in the console or with the CLI. A topic name should be descriptive and environment-specific.

```bash
aws sns create-topic \
  --name prod-high-cpu-alerts
```

The command returns a Topic ARN (Amazon Resource Name), which you use in subscriptions and alarm actions.

Example output:

```json
{
  "TopicArn": "arn:aws:sns:us-east-1:123456789012:prod-high-cpu-alerts"
}
```

**Adding an email subscription**

Email is the simplest way to validate the notification path. The subscriber must confirm the subscription before SNS delivers messages.

```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:prod-high-cpu-alerts \
  --protocol email \
  --notification-endpoint admin@example.com
```

Important behavior:

- The recipient receives a confirmation email.
- The subscription remains pending until confirmed.
- Unconfirmed subscriptions do not receive alarm notifications.

> If you do not confirm the subscription, the alarm can still enter `ALARM`, but no email will be delivered.

**Creating a CloudWatch alarm**

A CloudWatch alarm watches a metric and evaluates it against a threshold. For example, you can alert when EC2 CPU utilization stays above 80% for 5 minutes.

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name high-cpu-prod \
  --alarm-description "Alarm when CPU exceeds 80 percent" \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistic Average \
  --period 300 \
  --evaluation-periods 1 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --treat-missing-data missing \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:prod-high-cpu-alerts
```

Key settings to understand:

- **Namespace** identifies the metric family, such as `AWS/EC2`.
- **Metric name** identifies the specific metric, such as `CPUUtilization`.
- **Dimensions** scope the metric to a resource, such as an instance ID.
- **Statistic** defines how the metric is aggregated, such as `Average` or `Maximum`.
- **Period** is the evaluation window in seconds.
- **Evaluation periods** is how many consecutive periods must breach the threshold.
- **Treat missing data** controls how the alarm behaves when data is absent.

**Alarm action behavior**

Alarm actions are the targets that run when the alarm changes state. For notifications, the action is usually an SNS topic ARN.

Common patterns:

- `--alarm-actions`: sends a notification when the alarm enters `ALARM`.
- `--ok-actions`: sends a notification when the alarm returns to `OK`.
- `--insufficient-data-actions`: sends a notification when the alarm enters `INSUFFICIENT_DATA`.

For operational alerting, `ALARM` and `OK` are the most useful. `INSUFFICIENT_DATA` can be noisy unless you specifically want to track telemetry gaps.

**Practical example: EC2 CPU alert**

A common lab scenario is to alert on EC2 CPU spikes. The alarm watches the instance metric and notifies the operations team.

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name ec2-cpu-80-percent \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistic Average \
  --period 300 \
  --evaluation-periods 2 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:prod-high-cpu-alerts \
  --ok-actions arn:aws:sns:us-east-1:123456789012:prod-high-cpu-alerts
```

This configuration reduces false positives by requiring two consecutive 5-minute periods above the threshold.

**Testing the alert path**

To validate the setup, you need both the alarm and the delivery channel to work.

1. Confirm the SNS email subscription.
2. Force the metric above the threshold, if possible.
3. Wait for the alarm evaluation period to complete.
4. Check the alarm state in CloudWatch.
5. Verify the email notification content.

If you cannot easily generate real load, you can still validate the configuration by checking the alarm definition and subscription status.

Useful CLI checks:

```bash
aws sns list-subscriptions-by-topic \
  --topic-arn arn:aws:sns:us-east-1:123456789012:prod-high-cpu-alerts
```

```bash
aws cloudwatch describe-alarms \
  --alarm-names ec2-cpu-80-percent
```

**Notification message content**

SNS alarm notifications include useful operational details such as:

- Alarm name
- New state
- Reason for the state change
- Metric and threshold information
- Timestamp
- Region and account context

This makes the email actionable without requiring immediate console access. In practice, teams often include the alarm name in the subject line and use the message body for triage.

**Common subscription endpoints**

SNS supports multiple protocols, which makes it useful for both human and machine workflows.

| Protocol | Typical use | Notes |
|---|---|---|
| `email` | Human notification | Requires confirmation |
| `sms` | Mobile alerts | Availability varies by region and account settings |
| `lambda` | Automated remediation | SNS invokes the function |
| `http` / `https` | Webhook delivery | Endpoint must accept SNS message format |
| `sqs` | Queue-based processing | Useful for decoupling consumers |

For exam scenarios, remember that SNS is a fan-out service. One topic can notify many subscribers.

**IAM and permissions**

You need permission to create topics, subscriptions, and alarms. In a lab or exam-style environment, the relevant permissions often include:

- `sns:CreateTopic`
- `sns:Subscribe`
- `sns:ConfirmSubscription`
- `cloudwatch:PutMetricAlarm`
- `cloudwatch:DescribeAlarms`

If you automate alerting with Lambda or other services, you also need the correct resource-based or identity-based permissions for those targets.

**Troubleshooting checklist**

If notifications do not arrive, check the following:

- The SNS subscription is confirmed.
- The alarm is actually in `ALARM`.
- The alarm action points to the correct topic ARN.
- The topic and alarm are in the expected region.
- The email address is not filtered by spam rules.
- The metric has enough data to evaluate.
- The threshold and comparison operator are correct.

A few common mistakes cause most failures:

- Using the wrong region for the topic or alarm.
- Forgetting to confirm the email subscription.
- Choosing a metric that does not publish at the expected frequency.
- Setting the evaluation period too short for the metric cadence.

**Infrastructure as code example**

In real environments, you usually define alerts in CloudFormation or Terraform rather than clicking through the console.

CloudFormation example:

```yaml
Resources:
  AlertTopic:
    Type: AWS::SNS::Topic
    Properties:
      TopicName: prod-high-cpu-alerts

  CpuAlarm:
    Type: AWS::CloudWatch::Alarm
    Properties:
      AlarmName: ec2-cpu-80-percent
      Namespace: AWS/EC2
      MetricName: CPUUtilization
      Dimensions:
        - Name: InstanceId
          Value: i-0123456789abcdef0
      Statistic: Average
      Period: 300
      EvaluationPeriods: 2
      Threshold: 80
      ComparisonOperator: GreaterThanThreshold
      AlarmActions:
        - !Ref AlertTopic
```

This approach improves repeatability and makes alerting part of your deployment pipeline.

**Operational best practices**

- Use descriptive alarm names that include the service and threshold.
- Route critical alerts to a dedicated topic.
- Separate production and non-production notifications.
- Tune thresholds to reduce alert fatigue.
- Pair alarms with runbooks so responders know what to do next.
- Use `OK` notifications only when they add value to incident resolution.

**Exam focus**

For SAA-C03, the key idea is that CloudWatch detects the condition and SNS distributes the notification. If a question asks how to send an alarm email, the usual answer is:

- Create an SNS topic.
- Subscribe the email address.
- Confirm the subscription.
- Attach the topic to a CloudWatch alarm action.

Also remember:

- CloudWatch alarms are metric-based.
- SNS is a publish/subscribe service.
- One topic can have multiple subscribers.
- Email subscriptions require confirmation.

**Quick reference**

- Use CloudWatch alarms for threshold monitoring.
- Use SNS for notification fan-out.
- Use email for simple human validation.
- Use Lambda or SQS when you need automation or decoupling.
- Verify region, subscription status, and alarm state when troubleshooting.
