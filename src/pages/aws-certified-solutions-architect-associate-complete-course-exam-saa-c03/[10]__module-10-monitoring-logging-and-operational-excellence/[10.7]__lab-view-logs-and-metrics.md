## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and observability model**

This lab focuses on the core troubleshooting workflow in AWS: start with a symptom in a metric, inspect the related logs for context, and then confirm the API activity in CloudTrail. The practical skill is not just “finding data,” but narrowing a problem from service-level health to a specific request, function invocation, or configuration change.

In AWS, the three most useful observability layers are:

- **CloudWatch metrics** for time-series signals such as latency, errors, throttles, and utilization.
- **CloudWatch Logs** for application output, service logs, and structured event details.
- **CloudTrail** for the audit trail of AWS API calls, including who changed what and when.

A strong troubleshooting flow is:

1. Detect the anomaly in a metric.
2. Identify the affected resource using dimensions or resource identifiers.
3. Inspect the related log group and log stream.
4. Correlate the timestamp with a CloudTrail event if the issue may be caused by a configuration or control-plane change.
5. Confirm the fix by watching the metric return to normal.

**CloudWatch metrics: what to inspect first**

CloudWatch metrics are stored in namespaces, and each metric can be filtered by dimensions. A namespace groups metrics for a service or application, while dimensions add context such as `FunctionName`, `LoadBalancer`, `TargetGroup`, `InstanceId`, or `AutoScalingGroupName`.

Common examples:

- `AWS/Lambda`
  - `Invocations`
  - `Errors`
  - `Throttles`
  - `Duration`
  - `ConcurrentExecutions`
- `AWS/ApplicationELB`
  - `RequestCount`
  - `TargetResponseTime`
  - `HTTPCode_Target_5XX_Count`
  - `HTTPCode_ELB_5XX_Count`
- `AWS/EC2`
  - `CPUUtilization`
  - `NetworkIn`
  - `NetworkOut`
- `AWS/RDS`
  - `CPUUtilization`
  - `DatabaseConnections`
  - `FreeStorageSpace`

When you troubleshoot, always check:

- The **namespace** to ensure you are looking at the correct service.
- The **dimension values** to isolate the exact resource.
- The **time range** to match the incident window.
- The **statistic** used, such as `Average`, `Sum`, `Minimum`, `Maximum`, or a percentile like `p95`.

A metric spike is only useful if you know what it represents. For example, a high `Average` latency can hide short bursts, while `Maximum` can reveal a single bad request. For request-based services, `Sum` is often the most meaningful statistic for counts such as errors or invocations.

**Alarm states and what they mean in a troubleshooting scenario**

CloudWatch alarms help you detect when a metric crosses a threshold. In a lab or exam scenario, you should interpret the alarm state before assuming the service is broken.

- **OK**
  - The metric is within the configured threshold.
  - The issue may have already resolved or may be intermittent.
- **ALARM**
  - The metric breached the threshold for the configured evaluation periods.
  - This is the state you expect during an active incident.
- **INSUFFICIENT_DATA**
  - CloudWatch does not have enough datapoints to evaluate the alarm.
  - This can happen when the metric is new, the resource is idle, or the alarm is misconfigured.

Useful alarm troubleshooting checks:

- Confirm the alarm is attached to the correct metric and dimension.
- Verify the period and evaluation periods are appropriate for the workload.
- Check whether missing data is treated as `missing`, `breaching`, `notBreaching`, or `ignore`.
- Compare the alarm threshold to the normal baseline for the workload.

> If an alarm is in `INSUFFICIENT_DATA`, do not assume the service is healthy. It may simply mean the metric is not being published.

**CloudWatch Logs: how to navigate the console**

CloudWatch Logs stores log data in **log groups**, and each log group contains one or more **log streams**. A log group usually maps to a service, application, or environment, while a log stream is typically a single source such as one Lambda execution environment, one EC2 instance, or one container task.

In the CloudWatch console, you typically:

- Open **CloudWatch**.
- Choose **Logs** or **Log groups**.
- Select the relevant log group.
- Open the newest or most relevant log stream.
- Use the search box or Logs Insights to filter by timestamp, request ID, error text, or status code.

Common navigation patterns:

- For **Lambda**, inspect `/aws/lambda/<function-name>`.
- For **API Gateway**, inspect the configured execution or access logs if enabled.
- For **EC2**, inspect application logs collected by the CloudWatch agent.
- For **ALB**, remember that access logs are usually delivered to Amazon S3, not CloudWatch Logs.

Useful log-reading habits:

- Match the log timestamp to the metric spike.
- Search for request identifiers, correlation IDs, or error keywords.
- Look for stack traces, timeout messages, throttling messages, or permission errors.
- Compare a failing request with a successful request from the same time window.

**CloudWatch Logs Insights: fast log analysis**

CloudWatch Logs Insights is the preferred tool when you need to query large log sets quickly. It supports a query language for filtering, parsing, sorting, and aggregating log events.

A basic workflow is:

1. Select the log group or log groups.
2. Set the time range to the incident window.
3. Run a query to isolate errors or high-latency requests.
4. Narrow the results by request ID, status code, or message pattern.
5. Use the output to identify the exact failing event.

Example query patterns:

```sql
fields @timestamp, @message
| sort @timestamp desc
| limit 20
```

```sql
fields @timestamp, @message
| filter @message like /ERROR|Exception|Task timed out/
| sort @timestamp desc
| limit 50
```

```sql
fields @timestamp, requestId, status, latency
| filter status >= 500
| sort latency desc
| limit 20
```

```sql
stats count(*) as errors by bin(5m)
| sort bin(5m) desc
```

If your logs are structured JSON, you can query fields directly. This is much more effective than searching raw text because you can aggregate by status, function name, or request path.

**CloudTrail: confirm the control-plane change**

CloudTrail records AWS API activity. It is the best place to verify whether a configuration change, permission change, or deployment action caused the issue. CloudTrail is not a performance monitoring tool; it is an audit and investigation tool.

Use CloudTrail when you need to answer questions such as:

- Who changed the security group?
- When was the Lambda function updated?
- Which IAM role was attached or modified?
- Did someone change the target group, listener rule, or scaling policy?
- Was a resource deleted, replaced, or reconfigured?

In the CloudTrail console, you usually:

- Open **CloudTrail**.
- Go to **Event history** for recent management events.
- Filter by **Event name**, **Resource name**, **Username**, or **Event time**.
- Open the event to inspect the request parameters and response elements.

Important CloudTrail concepts:

- **Management events** capture control-plane actions such as `CreateFunction`, `UpdateFunctionCode`, `PutMetricAlarm`, `AuthorizeSecurityGroupIngress`, and `ModifyTargetGroup`.
- **Data events** capture high-volume resource activity for selected services such as S3 object-level actions or Lambda function invocations, if enabled.
- **Event history** retains recent management events by default, but long-term retention requires a trail and storage destination such as S3.

**End-to-end lab pattern: metric spike to log entry to CloudTrail event**

A realistic troubleshooting example is a Lambda-backed application that starts returning errors.

Suppose you notice:

- `AWS/Lambda` `Errors` increases.
- `Duration` rises close to the timeout.
- `Invocations` stays steady, so traffic is still arriving.

Your investigation path should be:

1. Open **CloudWatch Metrics** and select the Lambda function namespace.
2. Add `Errors`, `Duration`, and `Throttles` to the graph.
3. Set the statistic to `Sum` for `Errors` and `Throttles`, and inspect `p95` or `Maximum` for `Duration` if available.
4. Confirm the spike aligns with the incident time.
5. Open the function’s log group, usually `/aws/lambda/<function-name>`.
6. Inspect the newest log stream or query with Logs Insights for `ERROR`, `Task timed out`, or `AccessDenied`.
7. Note the request ID from the failing invocation.
8. Open **CloudTrail Event history** and filter around the same timestamp.
9. Search for `UpdateFunctionConfiguration`, `UpdateFunctionCode`, `PutFunctionConcurrency`, or IAM-related changes.
10. Correlate the change time with the first failed log entry.

What you should observe:

- The metric spike shows the problem started at a specific time.
- The log entry reveals the immediate runtime symptom, such as a timeout, missing environment variable, or downstream permission failure.
- The CloudTrail event reveals the change that likely introduced the issue, such as a code deployment, memory change, timeout change, or role update.

This is the key exam skill: metrics tell you *that* something is wrong, logs tell you *what* failed, and CloudTrail tells you *who changed the environment*.

**Concrete example: Lambda timeout investigation**

A common Lambda failure pattern is a sudden increase in `Duration` followed by `Errors`.

- Metric symptom:
  - `Duration` approaches the configured timeout.
  - `Errors` increases.
- Log symptom:
  - `Task timed out after 3.00 seconds`
  - Or a downstream call hangs before the timeout.
- CloudTrail clue:
  - `UpdateFunctionConfiguration` changed the timeout from `10` seconds to `3` seconds.
  - Or `UpdateFunctionCode` introduced a slower code path.

Example log line:

```text
2026-04-26T12:10:15.123Z 9f1c2d3e-aaaa-bbbb-cccc-1234567890ab ERROR Task timed out after 3.00 seconds
```

Example CloudTrail event fields to inspect:

- `eventName`
- `eventTime`
- `userIdentity`
- `requestParameters`
- `responseElements`
- `sourceIPAddress`

**Concrete example: ALB request errors and target health**

For an Application Load Balancer, a spike in `HTTPCode_Target_5XX_Count` or `TargetResponseTime` often points to the backend targets rather than the load balancer itself.

Troubleshooting flow:

- Check `AWS/ApplicationELB` metrics for the load balancer and target group.
- Compare `HTTPCode_ELB_5XX_Count` with `HTTPCode_Target_5XX_Count`.
  - ELB 5xx suggests the load balancer or listener path.
  - Target 5xx suggests the application behind the load balancer.
- Inspect application logs on the target instances, containers, or Lambda targets.
- Check CloudTrail for changes to:
  - target group health check settings
  - listener rules
  - security groups
  - auto scaling policies

If the target group health checks fail, you may see:

- Healthy host count drops.
- Requests are routed only to healthy targets.
- Logs show startup failures, dependency errors, or port mismatches.

**CloudWatch agent: prerequisites and where data appears**

The CloudWatch agent is used to collect custom metrics, memory and disk metrics, and log files from EC2 instances and on-premises servers. It is not automatic; you must prepare permissions and installation correctly.

Prerequisites:

- An IAM role attached to the EC2 instance or server with permissions to publish metrics and logs.
- The CloudWatch agent installed on the host.
- A configuration file that defines which metrics and logs to collect.
- Network access to CloudWatch endpoints, either through the internet or a VPC endpoint.

Typical IAM permissions include:

- `cloudwatch:PutMetricData`
- `logs:CreateLogGroup`
- `logs:CreateLogStream`
- `logs:PutLogEvents`
- `ssm:GetParameter` if you store the agent config in Systems Manager Parameter Store

Example agent configuration snippet:

```json
{
  "metrics": {
    "namespace": "MyApp/Host",
    "metrics_collected": {
      "mem": {
        "measurement": [
          "mem_used_percent"
        ]
      },
      "disk": {
        "measurement": [
          "used_percent"
        ],
        "resources": [
          "*"
        ]
      }
    }
  },
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/messages",
            "log_group_name": "/ec2/system/messages",
            "log_stream_name": "{instance_id}"
          }
        ]
      }
    }
  }
}
```

Where the data appears:

- Custom metrics appear under the namespace you configured, such as `MyApp/Host`.
- Logs appear in the log group names you defined, such as `/ec2/system/messages`.
- Each instance can create its own log stream, often named with the instance ID.

Common setup considerations:

- If the instance role is missing, the agent cannot publish data.
- If the log group does not exist, the agent may create it only if permissions allow.
- If the config file is wrong, metrics may appear but logs may not, or vice versa.
- If the agent is running but no data appears, check the agent status and the IAM role first.

Useful commands on Linux:

```bash
sudo systemctl status amazon-cloudwatch-agent
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a fetch-config \
  -m ec2 \
  -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json \
  -s
```

**Practical CLI examples for investigation**

You can use the AWS CLI to confirm what you see in the console.

List Lambda log groups:

```bash
aws logs describe-log-groups \
  --log-group-name-prefix "/aws/lambda/"
```

Tail a log stream:

```bash
aws logs tail /aws/lambda/my-function --follow
```

Query CloudWatch Logs Insights from the CLI:

```bash
aws logs start-query \
  --log-group-name "/aws/lambda/my-function" \
  --start-time 1714123200 \
  --end-time 1714126800 \
  --query-string 'fields @timestamp, @message | filter @message like /ERROR|Exception|Task timed out/ | sort @timestamp desc | limit 20'
```

Inspect recent CloudTrail events:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=UpdateFunctionConfiguration \
  --max-results 10
```

These commands are useful when you want to validate the console view or automate incident review.

**Step-by-step lab procedure**

1. Open the **CloudWatch** console and go to **Metrics**.
2. Select the relevant namespace, such as `AWS/Lambda` or `AWS/ApplicationELB`.
3. Add the key metrics for the resource:
   - Lambda: `Errors`, `Duration`, `Throttles`
   - ALB: `HTTPCode_Target_5XX_Count`, `TargetResponseTime`, `RequestCount`
4. Set the time range to the incident window and identify the spike.
5. Open the related **log group** in CloudWatch Logs.
6. Choose the newest relevant **log stream** or run a Logs Insights query.
7. Find the exact error message and note the timestamp and request ID.
8. Open **CloudTrail Event history** and filter by the same time window.
9. Search for recent API changes affecting the resource, such as code updates, configuration updates, IAM changes, or scaling changes.
10. Compare the CloudTrail event time with the first failing log entry.
11. Confirm the root cause by matching the metric symptom, the log error, and the control-plane change.
12. Recheck the metric after the fix to ensure the alarm returns to `OK`.

**How to interpret common troubleshooting outcomes**

- A metric spike with no log errors often means:
  - the issue is outside the application logs
  - the wrong log group is being inspected
  - the service is failing before the application starts
- Log errors with no metric spike often mean:
  - the issue is isolated to a small subset of requests
  - the metric is too coarse or the wrong dimension is selected
- CloudTrail changes with no immediate metric impact often mean:
  - the change is latent
  - the workload has not yet exercised the changed path
  - the issue will appear only under specific traffic patterns

**Exam-relevant takeaways**

- **CloudWatch metrics** show health trends, but you must choose the correct namespace, dimension, and statistic.
- **CloudWatch Logs** provide the runtime detail needed to identify the failing request or exception.
- **CloudTrail** proves whether a recent AWS API action caused or contributed to the issue.
- **Logs Insights** is the fastest way to search and aggregate large log sets during an incident.
- **CloudWatch agent** requires IAM permissions, installation, and correct configuration before data appears.
- **Alarm states** matter: `ALARM` indicates threshold breach, `OK` indicates normal operation, and `INSUFFICIENT_DATA` often indicates missing or insufficient metrics rather than a healthy system.

**Quick reference: what to check first**

- If latency rises:
  - inspect `Duration` or `TargetResponseTime`
  - check logs for timeouts or downstream dependency failures
- If errors rise:
  - inspect `Errors` or `HTTPCode_Target_5XX_Count`
  - search logs for exceptions and permission failures
- If a resource was changed:
  - inspect CloudTrail for `Update*`, `Put*`, `Modify*`, or `Delete*` events
- If custom host data is missing:
  - verify the CloudWatch agent, instance role, and config file
