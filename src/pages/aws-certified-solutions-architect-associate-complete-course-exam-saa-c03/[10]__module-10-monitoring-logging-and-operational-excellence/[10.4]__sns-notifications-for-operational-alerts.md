**Amazon SNS for operational alerting**

Amazon Simple Notification Service (SNS) is a managed pub/sub messaging service that you use to fan out operational alerts to multiple consumers. In practice, you publish an event once and let SNS deliver it to email, SMS, HTTP/S endpoints, AWS Lambda, Amazon SQS, or mobile push endpoints. This makes SNS a strong fit for alarms, incident notifications, deployment events, and workflow triggers.

For exam scenarios, focus on the difference between *publishing* and *subscribing*. Your application, monitoring system, or AWS service publishes a message to a topic. Subscribers receive the message independently, which avoids tight coupling and lets you add or remove alert destinations without changing the producer.

- **Topic** is the logical channel that receives published messages.
- **Publisher** is the component that sends alerts to the topic.
- **Subscriber** is the endpoint that receives notifications.
- **Fan-out** means one published message can reach many destinations.

A common operational pattern is CloudWatch Alarm -> SNS Topic -> email, Lambda, and incident automation. You can also connect EventBridge rules to SNS when you want event-driven alerting from AWS service events.

**Core alerting architecture**

SNS works best when you need near-real-time notification delivery and simple distribution. It is not a queue and does not guarantee durable message storage for later replay. If you need buffering, ordering, or consumer decoupling with retention, pair SNS with SQS. If you need workflow orchestration, use Step Functions or Lambda downstream.

```text
CloudWatch Alarm
      |
      v
   SNS Topic
   /   |    \
  v    v     v
Email Lambda  SQS
```

Operational alerts usually follow this flow:

1. A metric breaches a threshold or an event occurs.
2. CloudWatch, EventBridge, or an application publishes to SNS.
3. SNS delivers the message to each subscribed endpoint.
4. Downstream systems notify humans or trigger remediation.

**When to use SNS**

Use SNS when you need broad distribution of the same alert to multiple targets. Typical examples include:

- Pager or email notifications for critical alarms.
- SMS alerts for high-priority incidents.
- Lambda-based automation for ticket creation or remediation.
- SQS fan-out for multiple independent processing pipelines.
- HTTP/S webhooks for external incident management tools.

Use a different service when the requirement changes:

- **SQS** for queueing, buffering, and worker-based processing.
- **EventBridge** for event routing across AWS services and SaaS integrations.
- **CloudWatch Alarms** for metric evaluation and threshold detection.
- **SES** for richer email composition and outbound email workflows.

**Topic types and delivery behavior**

SNS topics are regional. You create the topic in a specific AWS Region, and publishers and subscribers interact with that regional endpoint. This matters for latency, compliance, and service integration design.

SNS supports standard topics and FIFO topics in some contexts, but for operational alerting the usual exam focus is on standard topics. Standard topics prioritize throughput and fan-out. If ordering or exactly-once processing is required, that is usually a queueing or workflow concern rather than a notification concern.

- **Standard topic** is the default choice for alerts and broad distribution.
- **FIFO topic** is for ordered, deduplicated messaging where supported.
- **Regional scope** means you design around the Region where the topic exists.

> Be careful not to treat SNS as a message archive. If a subscriber is unavailable and delivery fails, SNS is not a substitute for durable storage or replay.

**Subscriptions and endpoints**

SNS supports multiple subscription protocols. The endpoint type determines how the message is delivered and what confirmation steps are required.

| Protocol | Typical use | Notes |
|---|---|---|
| `email` | Human-readable alerts | Requires subscription confirmation |
| `sms` | Mobile incident alerts | Subject to regional and carrier limits |
| `http` / `https` | Webhooks and integrations | HTTPS is preferred for security |
| `lambda` | Automated remediation | SNS invokes the function directly |
| `sqs` | Fan-out to queues | Useful for decoupled processing |
| `application` | Mobile push notifications | Used for mobile apps |

For email and HTTP/S subscriptions, the endpoint owner usually confirms the subscription before delivery begins. Lambda and SQS subscriptions are commonly used for automation and downstream processing without human confirmation in the same way.

Example subscription workflow with the AWS CLI:

```bash
aws sns create-topic --name ops-alerts

aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:ops-alerts \
  --protocol email \
  --notification-endpoint ops-team@example.com
```

After creating an email subscription, the recipient must confirm the subscription from the email link before messages are delivered.

**Publishing alerts**

You publish messages to a topic using the AWS CLI, SDKs, or AWS service integrations. Messages can include a subject and a body. For operational alerts, keep the payload concise and actionable: what happened, where it happened, severity, and what to do next.

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:ops-alerts \
  --subject "High CPU on web-1" \
  --message "Instance i-0123456789abcdef0 exceeded 90% CPU for 5 minutes."
```

A good alert message includes:

- **Resource identifier** so responders know what failed.
- **Condition** that triggered the alert.
- **Severity** to prioritize response.
- **Suggested action** or link to runbook.

If you publish structured JSON, you can support multiple protocols with protocol-specific formatting. This is useful when the same alert must be readable by humans and machine-consumable by automation.

```json
{
  "default": "High CPU on web-1",
  "email": "High CPU on web-1\nInstance i-0123456789abcdef0 exceeded 90% CPU for 5 minutes.",
  "lambda": {
    "instanceId": "i-0123456789abcdef0",
    "metric": "CPUUtilization",
    "threshold": 90,
    "durationMinutes": 5
  }
}
```

**CloudWatch alarms with SNS**

The most common operational alert pattern is CloudWatch Alarm integrated with SNS. CloudWatch evaluates a metric, and when the alarm state changes, it sends a notification to the topic.

Typical examples:

- EC2 instance CPU utilization above threshold.
- ALB request count or 5xx error rate spikes.
- RDS CPU, storage, or connection anomalies.
- Lambda errors or throttles.
- Custom application metrics from CloudWatch Embedded Metric Format or `PutMetricData`.

Example alarm creation:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name HighCPU-web-1 \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 300 \
  --evaluation-periods 2 \
  --threshold 90 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-alerts
```

Operationally, you often want separate notifications for state transitions:

- **ALARM** when the threshold is breached.
- **OK** when the condition clears.
- **INSUFFICIENT_DATA** when CloudWatch cannot evaluate the metric reliably.

This helps responders understand whether the issue is ongoing or resolved.

**Message filtering**

SNS message filtering lets you route only relevant alerts to each subscriber. This reduces noise and lets different teams receive different severities or categories from the same topic.

Common filter dimensions include:

- **Severity** such as `critical`, `warning`, or `info`.
- **Environment** such as `prod`, `staging`, or `dev`.
- **Service** such as `payments`, `auth`, or `search`.
- **Region** for localized operations.

Example filter policy on a subscription:

```json
{
  "severity": ["critical", "high"],
  "environment": ["prod"]
}
```

This is useful when one topic carries all operational events, but only a subset should page the on-call team. Other subscribers can receive lower-severity events for dashboards or ticketing.

**Security and access control**

SNS uses AWS Identity and Access Management (IAM) for authorization. You control who can create topics, publish messages, subscribe endpoints, and manage subscriptions with IAM policies and topic policies.

Key security practices:

- **Least privilege** for publishers and administrators.
- **Topic policies** to allow cross-account publishing when needed.
- **HTTPS** for webhook delivery to protect data in transit.
- **KMS** encryption for sensitive message payloads at rest when required.
- **Subscription confirmation** to prevent accidental or malicious endpoint registration.

Example topic policy allowing a specific account to publish:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountPublish",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:123456789012:ops-alerts"
    }
  ]
}
```

If you need to protect alert content, enable server-side encryption with AWS Key Management Service (KMS). This is especially relevant when alerts may contain identifiers, internal hostnames, or incident details.

**Reliability and delivery semantics**

SNS is designed for high availability and scalable delivery, but delivery semantics depend on the protocol and subscriber behavior. For operational alerting, understand these practical points:

- Delivery is generally *at-least-once*, so duplicate notifications can occur.
- Subscribers should be idempotent when automation matters.
- HTTP/S endpoints should return success quickly and handle retries safely.
- Lambda subscribers should avoid side effects that break on duplicate invocation.
- SQS subscribers can absorb bursts and provide retry handling downstream.

If a downstream system must not miss alerts, consider combining SNS with SQS and a worker process. SNS handles fan-out, while SQS provides buffering and retry control.

**Dead-letter handling and retries**

For HTTP/S and Lambda subscriptions, delivery failures can be retried according to SNS behavior and endpoint response. For more robust failure handling, use dead-letter queues where supported in the subscription or downstream service design.

Practical guidance:

- Use retries for transient failures.
- Use dead-letter queues for messages that repeatedly fail processing.
- Monitor delivery metrics and subscription health.
- Alert on failed deliveries if the notification path itself is critical.

**Operational design patterns**

SNS is often part of a larger alerting pipeline rather than the entire solution.

- **Human notification pattern**
  - CloudWatch Alarm -> SNS -> email/SMS
  - Best for paging and awareness

- **Human plus automation pattern**
  - CloudWatch Alarm -> SNS -> email + Lambda
  - Best when you want both visibility and automatic remediation

- **Fan-out processing pattern**
  - Event source -> SNS -> multiple SQS queues
  - Best when several teams or systems need the same event independently

- **Incident management pattern**
  - EventBridge or CloudWatch -> SNS -> HTTPS webhook
  - Best for external incident tools and chat integrations

Example Lambda subscriber for automated remediation:

```python
import json

def lambda_handler(event, context):
    for record in event["Records"]:
        msg = json.loads(record["Sns"]["Message"])
        instance_id = msg.get("instanceId")
        metric = msg.get("metric")
        threshold = msg.get("threshold")

        print(f"Alert for {instance_id}: {metric} exceeded {threshold}")
```

**Troubleshooting alert delivery**

When alerts do not arrive, check the path end to end rather than only the topic.

1. Confirm the publisher is sending to the correct topic ARN.
2. Verify the subscription exists and is confirmed.
3. Check topic and subscription policies for permission issues.
4. Validate the endpoint is reachable and accepting messages.
5. Review CloudWatch metrics and logs for delivery failures.
6. Confirm message filtering is not excluding the event.

Common mistakes include:

- Publishing to the wrong Region.
- Forgetting email subscription confirmation.
- Using a filter policy that blocks the message.
- Assuming SNS stores messages for later replay.
- Sending alerts without a clear subject or severity.

**Exam focus summary**

For SAA-C03, remember these high-value distinctions:

- **SNS** is for pub/sub notification and fan-out.
- **CloudWatch Alarms** detect metric conditions and often trigger SNS.
- **SQS** buffers work; SNS notifies subscribers.
- **EventBridge** routes events; SNS notifies endpoints.
- **Lambda** can subscribe directly for automation.
- **Message filtering** reduces alert noise.
- **IAM and topic policies** control who can publish and subscribe.

A concise operational alert design usually looks like this:

```text
Metric/Event -> Alarm/Rule -> SNS Topic -> Humans + Automation
```

When you see a question about sending one alert to many destinations, decoupling the producer from consumers, or notifying both people and systems, SNS is usually the correct service.
