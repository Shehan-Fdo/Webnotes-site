## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Incident response lifecycle**

Incident response on AWS follows the same broad lifecycle as any security program: prepare, detect, analyze, contain, eradicate, recover, and improve. For exam purposes, the key distinction is how AWS services map to each phase.

- **Preventive controls** reduce the chance of compromise.
  - Examples: IAM least privilege, security groups, network ACLs (NACLs), encryption, MFA.
- **Detective controls** identify suspicious activity.
  - Examples: CloudTrail, CloudWatch alarms, GuardDuty, Security Hub, VPC Flow Logs, AWS WAF logs.
- **Responsive controls** limit impact and restore service.
  - Examples: isolating instances, revoking credentials, scaling down workloads, restoring from backup, incident workflows.

A strong architecture uses all three. Detection without response automation creates delay; response without good telemetry creates guesswork.

**Core AWS detection sources**

AWS incident detection depends on collecting the right telemetry. You usually combine control-plane logs, data-plane logs, and application logs.

- **AWS CloudTrail** records API activity.
  - Use it to answer who called what, from where, and when.
  - It is essential for forensic timelines and unauthorized API investigation.
- **Amazon CloudWatch Logs** stores application and system logs.
  - Use metric filters to turn log patterns into alarms.
- **Amazon VPC Flow Logs** capture IP traffic metadata for network interfaces, subnets, or VPCs.
  - Use them to investigate unusual connections, exfiltration patterns, or denied traffic.
- **AWS WAF logs** show web request details and rule matches.
  - Use them to analyze web attacks, bot activity, and blocked requests.
- **AWS Config** tracks resource configuration changes over time.
  - Use it to detect drift, insecure changes, and noncompliant resources.
- **Amazon GuardDuty** analyzes CloudTrail, VPC Flow Logs, DNS logs, and other signals to generate findings.
  - Use it for managed threat detection rather than building your own correlation logic.
- **AWS Security Hub** aggregates findings from multiple services and third-party tools.
  - Use it to centralize posture and prioritize remediation.

**CloudTrail for investigation**

CloudTrail is one of the most important services for incident response because it provides an audit trail of AWS API calls. For large investigations, CloudTrail Lake is especially useful because it lets you query large event sets with SQL-like syntax.

- Use **CloudTrail Lake** when you need to search across many events quickly.
- Use organization trails to centralize logs across accounts.
- Store logs in a dedicated logging account for stronger separation of duties.
- Protect log integrity with S3 versioning, MFA delete where appropriate, and restrictive bucket policies.

Example CloudTrail Lake query:

```sql
SELECT eventTime, eventName, userIdentity.arn, sourceIPAddress, awsRegion
FROM $EDS_ID
WHERE eventSource = 'ec2.amazonaws.com'
  AND eventName IN ('AuthorizeSecurityGroupIngress', 'RunInstances', 'TerminateInstances')
ORDER BY eventTime DESC
LIMIT 50;
```

CloudTrail is best for control-plane questions such as:
- Who created an IAM access key?
- Who changed a security group?
- Who disabled logging?
- Which role assumed access from an unusual source?

**CloudWatch alarms and log-based detection**

CloudWatch is often the fastest path from raw logs to actionable alerting. You can create metric filters from log patterns and then alarm on those metrics.

Example metric filter for failed SSH attempts in a Linux auth log:

```bash
aws logs put-metric-filter \
  --log-group-name /aws/ec2/auth \
  --filter-name FailedSSHAttempts \
  --filter-pattern '"Failed password"' \
  --metric-transformations metricName=FailedSSHAttempts,metricNamespace=Security,metricValue=1
```

Example alarm:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name FailedSSHAttemptsHigh \
  --metric-name FailedSSHAttempts \
  --namespace Security \
  --statistic Sum \
  --period 300 \
  --evaluation-periods 1 \
  --threshold 10 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:SecurityAlerts
```

Use log-based alarms when:
- You need to detect application-specific abuse.
- You want to alert on known bad patterns.
- You need a simple, low-latency detection mechanism.

**GuardDuty and Security Hub**

GuardDuty is a managed threat detection service. It looks for suspicious behavior such as credential exfiltration, reconnaissance, crypto mining, unusual API calls, and compromised instances.

- GuardDuty findings are prioritized and categorized by severity.
- It is useful for detection, not remediation.
- It can integrate with EventBridge for automated response.

Security Hub is the central security posture and findings aggregation service.

- It normalizes findings from GuardDuty, Inspector, Macie, IAM Access Analyzer, and partner tools.
- It supports security standards such as:
  - **CIS AWS Foundations Benchmark**
  - **AWS Foundational Security Best Practices**
- It helps you track compliance and prioritize remediation across accounts.

A common exam pattern is:
- GuardDuty detects a threat.
- Security Hub aggregates the finding.
- EventBridge routes the finding to an automated response.

**Event routing and decoupling**

Event routing matters because incident response systems should be loosely coupled. AWS offers several common patterns.

| Service | Best for | Characteristics |
|---|---|---|
| **Amazon SNS** | Fan-out notifications | Pushes messages to email, Lambda, HTTP, SQS, and more |
| **Amazon SQS** | Buffering and decoupling | Durable queue, consumer pulls messages, good for retry handling |
| **Amazon EventBridge** | Event-driven routing | Matches events by pattern and routes to targets with minimal glue code |

Use them together:
- Use **EventBridge** to match a GuardDuty or CloudTrail event.
- Use **SNS** to notify humans.
- Use **SQS** to buffer work for a remediation worker.
- Use **Lambda** or Step Functions to automate containment.

Example EventBridge rule pattern for a GuardDuty finding:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [{ "numeric": [">=", 7] }]
  }
}
```

**Automated response patterns**

Automated response should be scoped and reversible. Start with low-risk actions and add stronger containment only when confidence is high.

Common response actions:
- Disable or rotate compromised IAM credentials.
- Detach an IAM role from a compromised instance profile when appropriate.
- Quarantine an EC2 instance by changing its security group.
- Restrict traffic with NACLs for broader subnet-level isolation.
- Stop or scale down an ECS service or EKS deployment if the workload is compromised.
- Snapshot disks or preserve logs before making destructive changes.
- Open an incident ticket and notify responders.

Example Lambda containment action for an EC2 instance:

```python
import boto3

ec2 = boto3.client("ec2")

def lambda_handler(event, context):
    instance_id = event["detail"]["resource"]["instanceDetails"]["instanceId"]
    quarantine_sg = "sg-0123456789abcdef0"

    ec2.modify_instance_attribute(
        InstanceId=instance_id,
        Groups=[quarantine_sg]
    )

    return {"status": "quarantined", "instance_id": instance_id}
```

> A new security group is not a complete quarantine by itself. By default, a security group allows no inbound traffic but allows all outbound traffic. For true isolation, you usually also restrict egress, use a dedicated quarantine group, and consider NACLs or host-level controls.

**Containment options for compromised workloads**

Containment depends on the workload type and blast radius.

- **EC2**
  - Move the instance to a quarantine security group.
  - Restrict egress with security groups or NACLs.
  - Detach or replace the IAM instance profile if credentials may be abused.
  - Preserve evidence with snapshots and log exports.
- **ECS**
  - Scale the service down or replace the task definition.
  - Update the service to use a restricted security group.
  - Rotate task role credentials if compromise is suspected.
- **EKS**
  - Scale deployments down or cordon and drain nodes.
  - Restrict network paths with security groups for pods, NACLs, or network policies where applicable.
  - Rotate service account credentials and review cluster access.
- **Serverless**
  - Disable event sources or remove permissions from the execution role.
  - Update function configuration to stop further invocation if needed.

Containment should preserve evidence whenever possible. If you terminate too early, you may lose the data needed for root cause analysis.

**Incident coordination services**

AWS provides services that help coordinate response rather than just detect threats.

- **AWS Systems Manager Incident Manager**
  - Helps you manage incidents, responders, timelines, and runbooks.
  - Useful for operational coordination during outages or security events.
- **AWS Systems Manager OpsCenter**
  - Centralizes operational issues as OpsItems.
  - Useful for tracking remediation work and linking operational context.

These services are not replacements for security monitoring. They help organize the human and automated response once an incident is detected.

**Multi-account incident response**

In real AWS environments, incident response is usually multi-account. A common pattern uses AWS Organizations with centralized security and logging accounts.

Recommended structure:
- **Management account** for organization administration only.
- **Security account** for GuardDuty delegated administration, Security Hub aggregation, and response tooling.
- **Log archive account** for immutable or tightly controlled log storage.
- **Workload accounts** for applications and data.

Benefits:
- Centralized visibility across accounts.
- Reduced risk of tampering with logs.
- Easier delegated administration.
- Cleaner separation of duties.

Practical patterns:
- Enable organization-wide CloudTrail.
- Aggregate GuardDuty and Security Hub findings centrally.
- Forward CloudWatch Logs or application logs to a central account when needed.
- Use cross-account IAM roles for responders with least privilege.
- Automate response with EventBridge rules in the security account.

**Recovery and backup strategy**

Recovery is more than restoring a single resource. You need a tested plan that proves you can return to a known-good state.

Key recovery concepts:
- **AWS Backup** centralizes backup policy management across supported services.
- **Backup vault lock** helps make backups harder to delete or alter.
- **Cross-region recovery** protects against regional disruption.
- **Cross-account recovery** protects against account compromise.
- **Restore testing** verifies that backups are usable, not just present.

Recovery workflow:
1. Confirm the incident is contained.
2. Identify the last known-good state.
3. Restore from a trusted backup or rebuild from immutable infrastructure.
4. Validate application integrity and data consistency.
5. Reintroduce traffic gradually.
6. Monitor for recurrence.

> Backups are only useful if you test restores. A backup strategy without restore validation is an assumption, not a control.

**AWS Config and change detection**

AWS Config records resource configuration history and evaluates resources against rules. It is useful for detecting drift and identifying when a secure baseline changes.

Typical uses:
- Detect public S3 buckets.
- Detect security groups with overly broad ingress.
- Track changes to IAM policies, encryption settings, or network resources.
- Trigger remediation when a resource becomes noncompliant.

Example AWS Config setup flow:
1. Create a configuration recorder.
2. Create a delivery channel.
3. Start the recorder.
4. Add managed or custom rules.
5. Trigger remediation or alerting from noncompliance.

Example CLI sequence:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder name=default,roleARN=arn:aws:iam::123456789012:role/AWSConfigRole

aws configservice put-delivery-channel \
  --delivery-channel name=default,s3BucketName=my-config-bucket

aws configservice start-configuration-recorder \
  --configuration-recorder-name default
```

**Security Hub standards and response posture**

Security Hub is often used as the security operations dashboard for AWS environments.

- Enable standards to measure baseline posture.
- Use findings to prioritize remediation.
- Route findings to EventBridge for automation.
- Combine with Config rules for continuous compliance.

Common exam-relevant standards:
- CIS AWS Foundations Benchmark
- AWS Foundational Security Best Practices

Security Hub is especially useful when you need:
- A single pane of glass for findings.
- Cross-account aggregation.
- Automated triage and routing.

**Alerting and notification design**

Alerting should match the urgency and audience of the event.

- **SNS** is best when you need immediate push notifications.
- **SQS** is best when you need reliable processing and backpressure.
- **EventBridge** is best when you need event pattern matching and routing.

A practical design:
- GuardDuty finding arrives in EventBridge.
- High-severity findings go to SNS for on-call notification.
- Medium-severity findings go to SQS for queued investigation.
- Lambda enriches the event with account, resource, and tag data.
- Incident Manager creates an incident and assigns responders.

**Forensics and evidence preservation**

Good incident response preserves evidence before making changes.

- Export logs to a protected account or bucket.
- Snapshot EBS volumes before remediation if disk evidence matters.
- Record timestamps, resource IDs, and actions taken.
- Keep a chain of custody for sensitive investigations.
- Avoid overwriting logs or terminating resources before capture.

Useful evidence sources:
- CloudTrail for API activity.
- VPC Flow Logs for network metadata.
- WAF logs for web attacks.
- CloudWatch Logs for application behavior.
- GuardDuty findings for threat context.
- AWS Config history for configuration drift.

**Practical incident workflow**

A typical AWS incident workflow looks like this:

1. Detect the event through GuardDuty, CloudWatch, Config, or CloudTrail.
2. Triage the finding and confirm scope.
3. Preserve evidence with logs, snapshots, and timestamps.
4. Contain the blast radius using security groups, NACLs, IAM changes, or service scaling controls.
5. Eradicate the cause by removing malicious artifacts or fixing misconfigurations.
6. Recover from trusted backups or rebuilt infrastructure.
7. Review the incident and improve controls.

**Exam-focused distinctions**

- **Preventive** controls stop or reduce incidents before they happen.
- **Detective** controls identify suspicious activity after it occurs.
- **Responsive** controls limit damage and restore service after detection.

- **CloudTrail** is for API auditing and investigation.
- **CloudWatch** is for metrics, alarms, and log-based detection.
- **GuardDuty** is for managed threat detection.
- **Security Hub** is for aggregation and posture management.
- **Config** is for configuration history and compliance.
- **EventBridge** is for event routing and automation.
- **SNS** notifies, **SQS** buffers, and **EventBridge** routes.

**Common pitfalls**

- Assuming a security group alone creates full quarantine.
- Forgetting that CloudTrail Lake is useful for large-scale investigation queries.
- Treating GuardDuty as a remediation service instead of a detection service.
- Using SNS when you need durable queueing and retry control.
- Using SQS when you need direct fan-out notifications.
- Forgetting to start the AWS Config recorder after creating it.
- Relying on backups without restore testing.
- Keeping logs only in the workload account instead of centralizing them.

**Quick reference commands**

Create a CloudWatch alarm from a metric:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name ExampleAlarm \
  --metric-name 5XXError \
  --namespace AWS/ApplicationELB \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 1 \
  --threshold 5 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:OpsAlerts
```

Query GuardDuty findings:

```bash
aws guardduty list-findings \
  --detector-id d-1234567890abcdef \
  --finding-criteria Criterion={severity={Gte=4.0}}
```

Retrieve finding details:

```bash
aws guardduty get-findings \
  --detector-id d-1234567890abcdef \
  --finding-ids finding-id-1 finding-id-2
```

**Summary**

Incident detection and response on AWS is a layered discipline. You combine telemetry, managed detections, alert routing, and automated containment to reduce dwell time and limit impact. For the exam, focus on which service does what, how services integrate, and how to design for centralized, multi-account response with tested recovery.
