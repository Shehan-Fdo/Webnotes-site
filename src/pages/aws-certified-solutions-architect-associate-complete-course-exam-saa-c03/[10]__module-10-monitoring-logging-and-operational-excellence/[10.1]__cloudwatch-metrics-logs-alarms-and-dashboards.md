## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon CloudWatch overview**

Amazon CloudWatch is AWS’s monitoring and observability service for collecting **metrics**, centralizing **logs**, creating **alarms**, and building **dashboards**. For SAA-C03, focus on what each feature is best at and where the boundaries are.

CloudWatch is not a single “everything monitoring” tool. It is a set of related capabilities:

- **Metrics** track numeric time-series data such as CPU utilization, request count, or error rate.
- **Logs** store event records and application output for troubleshooting and analysis.
- **Alarms** evaluate metrics or composite conditions and trigger actions.
- **Dashboards** visualize selected metrics and text widgets in one place.

A useful exam mindset is to choose the right CloudWatch feature for the job:
- Use **metrics** for trends, thresholds, and alerting.
- Use **logs** for detailed investigation and search.
- Use **alarms** for automated response.
- Use **dashboards** for operational visibility.

**CloudWatch metrics**

Metrics are time-series data points stored in namespaces and dimensions. AWS services publish many standard metrics automatically, and you can also publish **custom metrics** for application-specific signals.

Key metric concepts:
- **Namespace** groups related metrics, such as `AWS/EC2` or `AWS/Lambda`.
- **Dimension** is a name/value pair that identifies a metric subset, such as `InstanceId=i-1234567890abcdef0`.
- **Statistic** is the aggregation over a period, such as `Average`, `Sum`, `Minimum`, `Maximum`, or `p99` where supported.
- **Period** is the time interval used to aggregate data points.
- **Resolution** can be standard or high-resolution for more frequent publishing and faster alarm evaluation.

Common exam-relevant points:
- AWS service metrics are usually available without extra setup.
- **Custom metrics** require you to publish data explicitly.
- Custom metrics incur charges, so avoid high-cardinality designs with too many unique dimension combinations.
- CloudWatch stores metric data for a limited retention window at full resolution, then aggregates older data into coarser intervals.

Typical custom metric publishing with the AWS CLI:

```bash
aws cloudwatch put-metric-data \
  --namespace "MyApp/Payments" \
  --metric-data '[
    {
      "MetricName": "CheckoutLatency",
      "Dimensions": [
        {"Name": "Service", "Value": "api"},
        {"Name": "Environment", "Value": "prod"}
      ],
      "Value": 248,
      "Unit": "Milliseconds"
    }
  ]'
```

Practical guidance:
- Use a stable namespace convention such as `Company/Application/Component`.
- Keep dimensions meaningful and limited.
- Publish metrics for business signals as well as infrastructure signals when they help operations.

**CloudWatch Agent and host-level monitoring**

The CloudWatch agent collects operating system and application telemetry from EC2 instances and on-premises servers. This is important when the built-in AWS service metrics are not enough.

Use the agent when you need:
- Memory usage, disk usage, swap, or process-level metrics.
- System logs from Linux or Windows hosts.
- Custom application logs shipped to CloudWatch Logs.
- A unified way to monitor hybrid environments.

Typical installation and configuration flow:
1. Install the agent on the instance or server.
2. Attach an IAM role or credentials with permission to publish metrics and logs.
3. Create the agent configuration file.
4. Start the agent and verify data in CloudWatch.

Example configuration snippet:

```json
{
  "metrics": {
    "metrics_collected": {
      "cpu": {
        "measurement": [
          "cpu_usage_idle",
          "cpu_usage_user",
          "cpu_usage_system"
        ],
        "metrics_collection_interval": 60
      },
      "mem": {
        "measurement": [
          "mem_used_percent"
        ],
        "metrics_collection_interval": 60
      },
      "disk": {
        "measurement": [
          "used_percent"
        ],
        "resources": [
          "*"
        ],
        "metrics_collection_interval": 60
      }
    }
  },
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/messages",
            "log_group_name": "/ec2/system/messages",
            "log_stream_name": "{instance_id}"
          }
        ]
      }
    }
  }
}
```

**CloudWatch Logs**

CloudWatch Logs stores log events in **log groups** and **log streams**. A log group is the container for retention, permissions, and metric filters; a log stream is the sequence of events from a specific source.

Common ingestion sources include:
- **Lambda** function logs, which are delivered automatically to CloudWatch Logs.
- **API Gateway** execution/access logs, when enabled.
- **VPC Flow Logs**, which can be delivered to CloudWatch Logs or S3.
- **CloudTrail** management events, which are typically delivered to S3 and can also be sent to CloudWatch Logs for near-real-time alerting.
- **EC2** and on-premises systems through the CloudWatch agent.
- Many other AWS services through explicit logging configuration or integration.

Important distinctions:
- Not every AWS service sends logs to CloudWatch Logs by default.
- Some services require you to enable logging or choose a destination.
- CloudWatch Logs is for operational log storage and analysis, not long-term archival by itself.

Retention and archival:
- Set **log retention** on log groups to control how long logs remain in CloudWatch Logs.
- Use **export to S3** for asynchronous, one-time export of existing log data when you need cheaper long-term storage or compliance retention.
- S3 lifecycle policies can then transition data to Glacier classes if needed.

Example retention command:

```bash
aws logs put-retention-policy \
  --log-group-name "/aws/lambda/my-function" \
  --retention-in-days 14
```

Example export task concept:
- You create an export task for a time range.
- CloudWatch Logs writes the selected log data to S3 asynchronously.
- This is not continuous streaming.

**CloudWatch Logs Insights**

Logs Insights is the query engine for CloudWatch Logs. Use it when you need fast search, filtering, aggregation, and troubleshooting across large log sets.

Core query capabilities:
- `fields` to select or derive fields.
- `filter` to narrow results.
- `parse` to extract structured data from unstructured text.
- `stats` to aggregate values.
- `sort` to order results.
- `limit` to cap output.

Example query:

```sql
fields @timestamp, @message, @logStream
| filter @message like /ERROR|Exception/
| parse @message "* latency=*ms *" as request, latency, rest
| stats count(*) as errorCount, avg(latency) as avgLatency by bin(5m)
| sort errorCount desc
| limit 20
```

Use cases:
- Find error spikes after a deployment.
- Group logs by request path, status code, or user ID.
- Measure approximate latency from application log lines.
- Investigate patterns without exporting logs elsewhere.

Exam tips:
- Logs Insights is better for interactive analysis than raw log browsing.
- Metric filters are better when you want to turn log patterns into alarms.
- Dashboards are not a replacement for Logs Insights when you need log search.

**Metric filters**

Metric filters turn matching log patterns into CloudWatch metrics. This is a common pattern for alerting on application errors, security events, or business events that only appear in logs.

Use metric filters when:
- A condition is visible in logs but not exposed as a native metric.
- You want an alarm on a log-derived signal.
- You need a lightweight count of specific events.

Example metric filter pattern:
- Match `ERROR` lines in a log group.
- Increment a custom metric in a namespace such as `MyApp/Logs`.

Conceptual flow:
1. Application writes a log line.
2. CloudWatch Logs matches the pattern.
3. The filter publishes a metric data point.
4. A CloudWatch alarm watches that metric.

Example AWS CLI pattern creation:

```bash
aws logs put-metric-filter \
  --log-group-name "/aws/lambda/my-function" \
  --filter-name "ErrorCount" \
  --filter-pattern '"ERROR"' \
  --metric-transformations \
    metricName=ErrorCount,metricNamespace=MyApp/Logs,metricValue=1
```

**CloudWatch alarms**

Alarms watch metrics and change state when thresholds are crossed. They are central to automated monitoring and incident response.

Alarm states:
- **OK** means the metric is within the defined threshold.
- **ALARM** means the threshold condition is met.
- **INSUFFICIENT_DATA** means CloudWatch does not have enough data to evaluate the alarm.

Alarm evaluation details matter on the exam:
- The **period** controls the granularity of evaluation.
- The **evaluation periods** setting determines how many consecutive periods are considered.
- The **datapoints to alarm** setting can require only some of those periods to breach the threshold.
- Missing data can be treated in different ways, such as `breaching`, `notBreaching`, `ignore`, or `missing`.

Example of why this matters:
- If you alarm on a metric that publishes intermittently, treating missing data as breaching can create false positives.
- If you alarm on a critical heartbeat metric, treating missing data as notBreaching can hide outages.
- Choose missing-data behavior based on the meaning of the signal.

Example CLI alarm:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "HighCPU-EC2" \
  --alarm-description "Alarm when EC2 CPU exceeds 80%" \
  --namespace "AWS/EC2" \
  --metric-name "CPUUtilization" \
  --dimensions Name=InstanceId,Value=i-1234567890abcdef0 \
  --statistic Average \
  --period 300 \
  --evaluation-periods 2 \
  --datapoints-to-alarm 2 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --treat-missing-data missing
```

Alarm actions:
- Send notifications to **SNS** topics.
- Trigger **Auto Scaling** policies.
- Perform certain **EC2 instance actions** for supported instance status check alarms, such as stop, terminate, reboot, or recover.
- Invoke some other supported integrations depending on alarm type and configuration.

Important limitations:
- Not every action is available for every alarm type.
- **Composite alarms** do not directly execute the same actions as metric alarms; they combine alarm states from other alarms.
- EC2 actions are limited to specific supported scenarios, not arbitrary metric alarms on any metric.

**Composite alarms**

Composite alarms combine multiple alarms using boolean logic. They help reduce noise and express higher-level conditions.

Use composite alarms when:
- You want to alert only when multiple signals indicate a real incident.
- You want to suppress noisy single-metric alarms.
- You need a more meaningful operational condition than one threshold alone.

Example logic:
- Alarm only when `HighCPU` AND `High5xxErrors` are both in ALARM state.
- Alarm when `PrimaryServiceDown` OR `DatabaseDown` is active.

Example conceptual expression:

```text
ALARM(HighCPU) AND ALARM(High5xxErrors)
```

Design note:
- Composite alarms are a design choice, not a universal best practice.
- Sometimes a single metric alarm is the right signal, especially for capacity or saturation issues.
- Use them to reduce noise, not to hide important alerts.

**Dashboards**

CloudWatch dashboards provide a visual view of metrics, alarms, and text widgets. They are useful for operations, executive summaries, and incident rooms.

Dashboard strengths:
- Display multiple metrics in one place.
- Show alarm status alongside graphs.
- Add text widgets for runbooks, links, or incident notes.
- Support cross-account and cross-Region visibility when you configure permissions and sharing correctly.

Dashboard limitations and cautions:
- Dashboards are primarily for metrics and text; they are not a general log analytics tool.
- Cross-account and cross-Region views require setup, and not every widget or metric source behaves identically.
- There are service quotas, including limits on dashboard count and widget count per dashboard.
- Large or overly dense dashboards can become hard to read and expensive to maintain.

Example dashboard JSON snippet:

```json
{
  "widgets": [
    {
      "type": "metric",
      "x": 0,
      "y": 0,
      "width": 12,
      "height": 6,
      "properties": {
        "metrics": [
          [ "AWS/EC2", "CPUUtilization", "InstanceId", "i-1234567890abcdef0" ]
        ],
        "period": 300,
        "stat": "Average",
        "region": "us-east-1",
        "title": "EC2 CPU"
      }
    },
    {
      "type": "text",
      "x": 12,
      "y": 0,
      "width": 12,
      "height": 6,
      "properties": {
        "markdown": "## Runbook\n- Check ASG scaling events\n- Review application logs\n- Validate downstream dependencies"
      }
    }
  ]
}
```

Create or update a dashboard with the CLI:

```bash
aws cloudwatch put-dashboard \
  --dashboard-name "Prod-Operations" \
  --dashboard-body file://dashboard.json
```

**Cross-account and cross-Region monitoring**

Cross-account and cross-Region observability is useful in multi-account AWS organizations and distributed architectures. The setup is more involved than a single-account dashboard.

Typical prerequisites:
- Appropriate IAM permissions and trust relationships.
- Metric sharing or access configuration for the source account.
- Correct Region selection for the metrics being displayed.
- Awareness of widget and metric source limitations.

Practical exam angle:
- If a question asks for centralized visibility across accounts, think about permissions, sharing, and the fact that dashboards are a presentation layer, not a data replication mechanism.
- If a question asks for centralized log analysis, CloudWatch Logs Insights or a centralized logging architecture may be more appropriate than a dashboard.

**Common monitoring patterns**

A few patterns appear often in architecture questions:

- **EC2 host monitoring**
  - Use built-in EC2 metrics for CPU and network.
  - Add the CloudWatch agent for memory, disk, and OS logs.
  - Alarm on sustained CPU or disk pressure.

- **Application error monitoring**
  - Send application logs to CloudWatch Logs.
  - Create a metric filter for `ERROR` or specific exception patterns.
  - Alarm on the resulting custom metric.

- **Serverless monitoring**
  - Use Lambda metrics such as invocations, errors, duration, and throttles.
  - Review Lambda logs in CloudWatch Logs.
  - Use Logs Insights to troubleshoot failures.

- **Security and audit visibility**
  - Send CloudTrail to S3 for durable audit storage.
  - Optionally send CloudTrail to CloudWatch Logs for near-real-time alerting.
  - Use metric filters or alarms for suspicious patterns.

- **Network visibility**
  - Use VPC Flow Logs for accepted/rejected traffic analysis.
  - Store them in CloudWatch Logs or S3 depending on the use case.
  - Use Logs Insights or downstream analytics for investigation.

**CLI and operational examples**

List log groups:

```bash
aws logs describe-log-groups
```

Tail a log stream:

```bash
aws logs get-log-events \
  --log-group-name "/aws/lambda/my-function" \
  --log-stream-name "2026/04/26/[$LATEST]abcdef1234567890"
```

List alarms:

```bash
aws cloudwatch describe-alarms
```

Get metric data:

```bash
aws cloudwatch get-metric-statistics \
  --namespace "AWS/EC2" \
  --metric-name "CPUUtilization" \
  --dimensions Name=InstanceId,Value=i-1234567890abcdef0 \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T01:00:00Z \
  --period 300 \
  --statistics Average
```

**Exam-focused distinctions**

- **Metrics vs logs**
  - Metrics are numeric and time-based.
  - Logs are detailed event records.
  - Use metrics for alerting and logs for diagnosis.

- **Logs Insights vs metric filters**
  - Logs Insights queries logs interactively.
  - Metric filters convert log patterns into metrics for alarms.

- **Metric alarms vs composite alarms**
  - Metric alarms evaluate one metric or expression.
  - Composite alarms evaluate alarm states from other alarms.

- **Dashboards vs alarms**
  - Dashboards visualize.
  - Alarms notify or automate.

- **Retention vs export**
  - Retention controls how long data stays in CloudWatch Logs.
  - Export to S3 is a separate asynchronous task for existing log data.

- **Native metrics vs custom metrics**
  - Native metrics come from AWS services.
  - Custom metrics come from your application or agent and cost extra.

**Practical troubleshooting approach**

When a CloudWatch-based question appears, identify the signal first:
- Is the problem a threshold on a numeric value?
- Is the problem hidden in logs?
- Do you need an automated action?
- Do you need a visual summary for operators?

Then choose the feature:
1. Use a metric if the signal already exists.
2. Use a metric filter if the signal is only in logs.
3. Use an alarm if you need notification or automation.
4. Use a dashboard if you need a shared operational view.
5. Use Logs Insights if you need to investigate the root cause.

**Key takeaways**

CloudWatch is the AWS monitoring backbone, but each component has a distinct role. Metrics provide time-series signals, logs provide detail, alarms provide automation, and dashboards provide visibility. For SAA-C03, pay close attention to alarm evaluation behavior, log ingestion paths, metric filters, CloudWatch agent use cases, and the difference between visualization and analysis.
