## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**AWS Config compliance overview**

AWS Config records resource configurations, evaluates them against rules, and helps you answer two exam-relevant questions: *what changed* and *is it compliant*. It is a governance and audit service, not a prevention control. You use it to detect drift, prove compliance, and trigger remediation after a resource changes.

At a high level, AWS Config does three things:

- Records configuration state changes for supported resources.
- Evaluates resources against managed or custom rules.
- Delivers compliance history, snapshots, and aggregated views for audits.

**Core concepts**

AWS Config continuously tracks configuration items for supported resource types in a Region. A configuration item is a point-in-time record of a resource’s settings, relationships, and metadata. This is what lets you compare current state to prior state during troubleshooting or audit reviews.

Key terms:

- **Configuration recorder**: Captures configuration changes for selected resource types.
- **Delivery channel**: Sends configuration snapshots and history to an S3 bucket, and can publish notifications to SNS.
- **Configuration item**: The recorded state of a resource at a specific time.
- **Rule**: Evaluates whether a resource or configuration is compliant.
- **Remediation**: Automated or manual action to fix noncompliance.

**Recording scope and resource coverage**

You control what AWS Config records. This matters for cost, signal quality, and exam scenarios involving noisy accounts.

Common recording choices:

- Record all supported resource types.
- Record only specific resource types.
- Exclude specific resource types from an otherwise broad recording scope.

Practical considerations:

- **Supported resource types** vary by service and Region.
- **Global resource types** such as IAM resources are recorded in a single Region per account, not in every Region.
- If you do not record a resource type, AWS Config cannot evaluate it with rules that depend on recorded configuration data.
- Recording more resource types increases visibility but also increases cost and data volume.

Example recorder configuration with the AWS CLI:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder name=default,roleARN=arn:aws:iam::123456789012:role/AWSConfigRole,recordingGroup="{allSupported=true,includeGlobalResourceTypes=true}"
```

To record only selected types:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder name=default,roleARN=arn:aws:iam::123456789012:role/AWSConfigRole,recordingGroup="{allSupported=false,resourceTypes=[\"AWS::EC2::Instance\",\"AWS::S3::Bucket\"]}"
```

**Snapshots vs. configuration history**

AWS Config produces two different operational views that often appear in exam questions:

- **Configuration history**: A chronological record of changes for a resource over time.
- **Configuration snapshot**: A point-in-time capture of the current configuration state for all recorded resources in a Region.

Use history when you need to investigate a specific change. Use snapshots when you need a full inventory-style view for audit or recovery analysis.

Operationally:

- History is change-oriented.
- Snapshots are inventory-oriented.
- Both are delivered to S3 through the delivery channel.
- Snapshots are useful for periodic compliance evidence and bulk review.

**Managed rules and custom rules**

AWS Config rules come in two main forms:

- **Managed rules**: Prebuilt by AWS for common checks.
- **Custom rules**: Your own logic, usually implemented with AWS Lambda.

Managed rules are best when the compliance requirement maps cleanly to a standard control, such as:

- EBS volumes should be encrypted.
- S3 buckets should not allow public read access.
- Security groups should not allow unrestricted SSH or RDP.

Custom rules are best when you need organization-specific logic, such as:

- Enforcing a tag policy with exceptions.
- Checking for a combination of settings across multiple resources.
- Validating a naming convention or business-specific threshold.

A rule evaluates a resource and returns a compliance state such as compliant, noncompliant, or not applicable.

**Custom rule evaluation flow**

For Lambda-backed custom rules, AWS Config invokes your function with an event that includes the invoking event type and resource details. Your function must then return evaluations in the AWS Config format, typically by calling `PutEvaluations`.

A simplified event shape looks like this:

```json
{
  "invokingEvent": "{\"messageType\":\"ConfigurationItemChangeNotification\",\"configurationItem\":{...}}",
  "ruleParameters": "{\"requiredTagKey\":\"Owner\"}",
  "resultToken": "EXAMPLE_TOKEN",
  "eventLeftScope": false
}
```

A correct Lambda-backed rule flow is:

1. AWS Config sends the event to Lambda.
2. Lambda parses the configuration item and rule parameters.
3. Lambda determines compliance.
4. Lambda calls `PutEvaluations` with the result token.

Example Python Lambda handler:

```python
import json
import boto3

config = boto3.client("config")

def lambda_handler(event, context):
    invoking_event = json.loads(event["invokingEvent"])
    config_item = invoking_event.get("configurationItem", {})
    resource_id = config_item.get("resourceId")

    # Example logic: require a tag named Owner
    tags = config_item.get("tags", {})
    is_compliant = "Owner" in tags

    evaluation = {
        "ComplianceResourceType": config_item.get("resourceType"),
        "ComplianceResourceId": resource_id,
        "ComplianceType": "COMPLIANT" if is_compliant else "NON_COMPLIANT",
        "OrderingTimestamp": config_item.get("configurationItemCaptureTime")
    }

    config.put_evaluations(
        Evaluations=[evaluation],
        ResultToken=event["resultToken"]
    )

    return {"statusCode": 200}
```

Important points:

- You do not return a plain compliance dictionary as the final AWS Config response.
- You use `PutEvaluations` for the compliance result.
- The `resultToken` is required for the evaluation call.
- The Lambda function should be idempotent and fast.

**Managed rule example**

A managed rule can be enabled to check a common control without writing code. For example, a rule can evaluate whether S3 buckets have public read access blocked.

Typical workflow:

1. Enable the rule in the target account and Region.
2. AWS Config evaluates recorded resources.
3. Noncompliant resources appear in the console and API.
4. Optional remediation runs automatically.

**Compliance reporting and advanced queries**

AWS Config provides multiple ways to search and report on compliance data. For exam prep, the most important distinction is between rule-level summaries and resource-level queries.

Useful capabilities:

- **Compliance details by rule**: Shows resources evaluated by a specific rule.
- **Advanced queries**: SQL-like queries across recorded configuration data.
- **Resource timeline**: Helps you inspect the history of a single resource.
- **Compliance summary**: Gives a high-level view of rule status.

Advanced queries are especially useful for audits and troubleshooting because they let you search across resource types, tags, and compliance state.

Example advanced query:

```sql
SELECT
  resourceId,
  resourceType,
  configuration.complianceType,
  awsRegion,
  accountId
WHERE
  configuration.complianceType = 'NON_COMPLIANT'
```

Another example for tagged resources:

```sql
SELECT
  resourceId,
  resourceType,
  tags.Owner
WHERE
  tags.Owner IS NULL
```

Use advanced queries when you need to:

- Find all noncompliant resources across many rules.
- Investigate drift by resource type or tag.
- Build audit evidence without manually opening each rule.

**Aggregators and organization aggregators**

For centralized compliance reporting, AWS Config supports aggregators. These are important in multi-account and multi-Region architectures.

Two common patterns:

- **Configuration aggregator**: Collects data from multiple accounts and Regions into one place.
- **Organization aggregator**: Aggregates data across an AWS Organizations hierarchy.

Why this matters:

- You can view compliance centrally instead of logging into each account.
- You can support audit teams with one reporting account.
- You can compare compliance across business units, accounts, and Regions.

Organization-level setup typically requires:

- AWS Organizations enabled.
- A delegated administrator account for AWS Config aggregation.
- Cross-account permissions that allow the aggregator to read source account data.

Practical exam note:

- The delegated administrator is the account that manages the organization-wide AWS Config view.
- Source accounts must allow aggregation access.
- Aggregation is read-oriented; it does not replace local recording in each account and Region.

**Conformance packs**

Conformance packs bundle multiple AWS Config rules and optional remediation actions into a single deployable package. They are useful when you want to apply a standard control set repeatedly.

Common use cases:

- CIS-style baseline controls.
- Internal security baselines.
- Environment-specific compliance packs for dev, test, and prod.

A conformance pack is usually defined in YAML or JSON and deployed through AWS Config. At scale, organizations often distribute packs across accounts and Regions using AWS Organizations and automation patterns such as StackSets or organization-wide deployment workflows.

Example conformance pack snippet:

```yaml
Resources:
  S3BucketPublicReadProhibited:
    Type: AWS::Config::ConfigRule
    Properties:
      ConfigRuleName: s3-bucket-public-read-prohibited
      Source:
        Owner: AWS
        SourceIdentifier: S3_BUCKET_PUBLIC_READ_PROHIBITED
```

Practical deployment guidance:

- Use organization deployment when you need consistent controls across many accounts.
- Use Region-specific deployment when controls depend on local services or data residency.
- Pair packs with remediation for controls that can be safely auto-fixed.

**Remediation with SSM Automation**

AWS Config remediation commonly uses AWS Systems Manager (SSM) Automation documents. This is the preferred pattern when the fix is deterministic and safe to automate.

Typical remediation flow:

1. AWS Config detects a noncompliant resource.
2. A remediation configuration maps the rule to an SSM Automation document.
3. The automation document runs with an IAM role.
4. The resource is corrected or a ticket/manual step is triggered.

Example scenario:

- Rule: S3 bucket public read access prohibited
- Remediation: Apply a bucket policy or block public access settings

A practical SSM Automation document might call AWS APIs to update the resource. The remediation role needs permission to perform the fix, and the automation document must be trusted by AWS Config and SSM.

When to use automatic remediation:

- The fix is low risk and repeatable.
- The desired state is unambiguous.
- You want fast drift correction.

When to prefer manual remediation:

- The fix could cause downtime.
- Human approval is required.
- The resource change depends on business context.

> AWS Config remediation is configured through AWS Config and SSM Automation, not by writing a simple YAML block that directly represents the remediation action itself.

**Permissions and service roles**

AWS Config uses an IAM service role to record and deliver configuration data. That role is different from the permissions you use to manage AWS Config as an administrator.

Service role responsibilities:

- Read resource configuration metadata.
- Deliver snapshots and history to S3.
- Publish notifications to SNS if configured.
- Invoke remediation or related automation when permitted.

Management permissions are separate:

- `config:PutConfigurationRecorder`
- `config:PutConfigRule`
- `config:PutConformancePack`
- `config:Describe*`
- `config:Get*`

These are for the caller managing AWS Config, not the service role that records data.

Example service role policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetBucketAcl",
        "s3:GetBucketLocation"
      ],
      "Resource": [
        "arn:aws:s3:::my-config-bucket",
        "arn:aws:s3:::my-config-bucket/AWSLogs/*"
      ]
    }
  ]
}
```

**AWS Organizations integration**

AWS Config is often used in multi-account environments, so organization integration is a common exam theme.

Important behaviors:

- You can deploy rules and conformance packs across accounts in an organization.
- You can designate a delegated administrator for centralized management.
- Aggregation can span multiple accounts and Regions for compliance visibility.
- Cross-account access must be explicitly allowed.

Use this model when you need:

- Central governance with local enforcement.
- A single audit account.
- Standardized controls across business units.

**Pricing considerations**

AWS Config pricing can influence architecture choices and exam tradeoffs.

Cost drivers usually include:

- Number of configuration items recorded.
- Number of rules evaluated.
- Number of conformance packs and evaluations.
- Number of advanced queries and related usage patterns, depending on current pricing model.

Practical implications:

- Recording every resource type in every Region can become expensive.
- Narrowing recording scope reduces cost but may reduce visibility.
- Managed rules are often cheaper operationally than building and maintaining custom logic, but they still incur evaluation costs.

**Common exam scenarios**

Use AWS Config when the question is about:

- Detecting configuration drift.
- Proving compliance over time.
- Auditing resource changes.
- Enforcing standards after deployment.
- Centralizing compliance across accounts and Regions.

Do not confuse AWS Config with:

- **AWS CloudTrail**: records API activity, not resource configuration state.
- **AWS CloudWatch**: monitors metrics and logs, not compliance state.
- **AWS Systems Manager**: can remediate and manage instances, but does not replace Config’s compliance history.

**Practical CLI examples**

List rules:

```bash
aws configservice describe-config-rules
```

Check compliance by rule:

```bash
aws configservice get-compliance-details-by-config-rule \
  --config-rule-name s3-bucket-public-read-prohibited
```

Then filter noncompliant resources client-side, for example with `jq`:

```bash
aws configservice get-compliance-details-by-config-rule \
  --config-rule-name s3-bucket-public-read-prohibited \
  | jq '.EvaluationResults[] | select(.ComplianceType=="NON_COMPLIANT")'
```

Get compliance by resource:

```bash
aws configservice get-compliance-details-by-resource \
  --resource-type AWS::S3::Bucket \
  --resource-id my-bucket-name
```

Get resource configuration history:

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::Instance \
  --resource-id i-0123456789abcdef0
```

**Troubleshooting patterns**

If a rule is not evaluating as expected, check:

- The resource type is recorded.
- The recorder is running in the correct Region.
- The rule applies to the resource type.
- The service role has the required read and delivery permissions.
- The resource is not a global resource being looked for in the wrong Region.

If remediation does not run:

- Confirm the remediation configuration exists.
- Confirm the SSM Automation document is valid.
- Confirm the automation role has permission to make the change.
- Confirm the rule is actually producing noncompliant evaluations.

**High-value exam takeaways**

- AWS Config records configuration state and evaluates compliance; it does not block changes.
- Global resource types like IAM are recorded in one Region per account.
- Advanced queries help you search compliance and configuration data across resources.
- Aggregators and organization aggregators centralize multi-account, multi-Region reporting.
- Conformance packs standardize rule deployment at scale.
- Lambda-backed custom rules must call `PutEvaluations`.
- Remediation is commonly done with SSM Automation.
- AWS Config service-role permissions are for recording and delivery, not for managing Config itself.
