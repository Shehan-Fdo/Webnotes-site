## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Sustainability in AWS architecture**

Sustainable architecture means designing workloads to use only the resources you need, when you need them. In AWS, this usually aligns with cost efficiency, operational efficiency, and right-sizing, but the goal is broader: reduce waste across compute, storage, networking, and managed services while preserving reliability and performance.

For the SAA-C03 exam, sustainability is not usually a standalone feature question. Instead, it appears as a design consideration when you choose between architectures, instance types, scaling models, storage classes, or data transfer patterns. The best answer often minimizes idle capacity, avoids overprovisioning, and uses managed services that abstract undifferentiated heavy lifting.

**Core sustainability principles**

AWS sustainability guidance is commonly summarized by a few practical design habits:

- **Right-size resources** to match actual demand instead of provisioning for peak all the time.
- **Use elasticity** so capacity scales up and down automatically.
- **Choose managed services** when they reduce operational overhead and improve utilization.
- **Reduce data movement** because unnecessary transfers increase network usage and can add latency and cost.
- **Store data efficiently** by selecting the right storage class, lifecycle policy, and retention period.
- **Turn off unused resources** such as idle instances, unattached volumes, and stale snapshots.
- **Prefer serverless or event-driven patterns** when they fit the workload, because they can eliminate always-on infrastructure.

**How sustainability maps to exam decisions**

When you compare two solutions, the more sustainable option is often the one that:

- Uses **auto scaling** instead of fixed overprovisioning.
- Uses **Spot Instances** for fault-tolerant, interruptible workloads.
- Uses **serverless** services such as AWS Lambda, Amazon SQS, or Amazon EventBridge for bursty or intermittent traffic.
- Uses **Amazon S3 lifecycle policies** to transition infrequently accessed data to cheaper storage classes.
- Uses **Amazon RDS** or Amazon Aurora managed features instead of self-managed database servers.
- Uses **content delivery** and caching to reduce repeated origin requests.
- Uses **multi-AZ** or **managed replication** only where the availability requirement justifies the extra resource footprint.

> Sustainability does not mean “use the smallest possible system.” It means use the *right* amount of infrastructure for the workload and avoid waste.

**Right-sizing compute**

Compute is often the biggest source of waste because teams provision for peak load and leave capacity idle. Right-sizing means selecting the smallest instance type, container task size, or Lambda memory setting that still meets performance and reliability targets.

Common approaches include:

- Reviewing CPU, memory, disk, and network utilization over time.
- Matching instance families to workload characteristics.
- Using burstable instances for low-to-moderate baseline workloads with occasional spikes.
- Using auto scaling groups to add or remove EC2 instances based on demand.
- Using Lambda for event-driven workloads so you pay and consume only during execution.

Example EC2 Auto Scaling policy:

```bash
aws autoscaling put-scaling-policy \
  --auto-scaling-group-name web-asg \
  --policy-name cpu-target-tracking \
  --policy-type TargetTrackingScaling \
  --target-tracking-configuration '{
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ASGAverageCPUUtilization"
    },
    "TargetValue": 50.0
  }'
```

Practical implications:

- A fixed fleet of large instances is usually less sustainable than a smaller fleet that scales.
- Overprovisioned memory is wasted even if CPU looks healthy.
- If a workload is spiky and stateless, Lambda or containers behind an Application Load Balancer can be more efficient than always-on EC2.

**Storage efficiency**

Storage sustainability is about keeping data in the most appropriate tier for its access pattern and deleting data you no longer need. The exam often tests lifecycle and archival choices.

Key patterns:

- Use **S3 Standard** for frequently accessed data.
- Transition to **S3 Standard-IA** or **S3 One Zone-IA** for infrequent access when appropriate.
- Archive to **S3 Glacier Instant Retrieval**, **S3 Glacier Flexible Retrieval**, or **S3 Glacier Deep Archive** for long-term retention.
- Use **lifecycle policies** to automate transitions and expiration.
- Enable **versioning** only when you need it, because it can increase stored object count.
- Remove unattached **EBS volumes** and old snapshots that no longer serve recovery objectives.
- Use the right database storage and retention settings rather than keeping oversized volumes or excessive logs.

Example S3 lifecycle configuration:

```json
{
  "Rules": [
    {
      "ID": "ArchiveLogs",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER"
        }
      ],
      "Expiration": {
        "Days": 365
      }
    }
  ]
}
```

Use cases:

- Application logs can move to colder storage after the active troubleshooting window.
- Compliance archives can go directly to deep archive if retrieval is rare and slow access is acceptable.
- Temporary build artifacts should expire automatically.

**Network and data transfer efficiency**

Network sustainability improves when you reduce unnecessary data movement. This also tends to improve latency and lower cost.

Design choices that help:

- Keep compute and data in the same **Region** and, when possible, the same **Availability Zone** if the architecture allows it.
- Use **VPC endpoints** for private access to AWS services instead of routing traffic through the public internet or NAT gateways when appropriate.
- Cache content with **Amazon CloudFront** to reduce repeated origin fetches.
- Compress payloads and reduce response sizes.
- Avoid chatty service-to-service communication.
- Use event-driven integration instead of constant polling where possible.

Example: create an S3 gateway endpoint to keep S3 traffic private and reduce NAT usage:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-1234567890abcdef0 \
  --service-name com.amazonaws.us-east-1.s3 \
  --vpc-endpoint-type Gateway \
  --route-table-ids rtb-11111111 rtb-22222222
```

Why this matters:

- NAT gateways add cost and can become a persistent always-on component.
- Private endpoints reduce exposure and often simplify traffic paths.
- Caching reduces repeated compute and storage reads at the origin.

**Managed services and shared responsibility**

Managed services can improve sustainability because AWS handles much of the underlying infrastructure, patching, scaling, and maintenance. That usually means fewer customer-managed servers and better utilization at the platform level.

Examples:

- **Amazon RDS** reduces the need to run and maintain database hosts.
- **Amazon Aurora** can provide high performance with managed storage and replication.
- **AWS Lambda** removes server management entirely for event-driven code.
- **Amazon DynamoDB** abstracts capacity management and can scale without provisioning servers.
- **Amazon SQS** decouples producers and consumers so each side can scale independently.

This does not mean managed services are always the right answer. Choose them when they fit the workload’s access pattern, latency needs, and operational model.

**Scaling patterns that support sustainability**

Elasticity is one of the strongest sustainability tools in AWS. You want capacity to follow demand rather than remain fixed.

Common patterns:

- **Horizontal scaling** with Auto Scaling groups for stateless EC2 workloads.
- **Vertical scaling** only when a workload cannot be distributed and needs a larger instance class.
- **Event-driven scaling** with Lambda, SQS, and EventBridge.
- **Queue buffering** to smooth bursts and prevent overprovisioning.
- **Scheduled scaling** for predictable business hours.

Example scheduled scaling:

```bash
aws autoscaling put-scheduled-update-group-action \
  --auto-scaling-group-name app-asg \
  --scheduled-action-name scale-out-business-hours \
  --recurrence "0 8 * * MON-FRI" \
  --min-size 4 \
  --desired-capacity 6 \
  --max-size 10
```

Use scheduled scaling when demand is predictable, such as office hours, batch windows, or nightly processing.

**Choosing the right EC2 purchasing model**

Instance purchasing choices can support sustainability by matching cost and utilization to workload tolerance.

- **On-Demand Instances** are best for unpredictable or steady critical workloads.
- **Reserved Instances** or **Savings Plans** improve commitment efficiency for steady usage, though they are primarily cost tools rather than direct sustainability features.
- **Spot Instances** are highly efficient for fault-tolerant, interruptible workloads such as batch processing, CI/CD, rendering, and analytics.
- **Dedicated Hosts** are usually not a sustainability optimization because they reserve physical capacity for a single customer.

Example Spot usage pattern:

- Use Spot for worker nodes in a batch processing fleet.
- Keep a small On-Demand baseline for critical control-plane or queue-draining tasks.
- Design the application to handle interruption gracefully.

**Data lifecycle and retention**

A sustainable architecture keeps data only as long as it is needed. Retention policies reduce storage growth and simplify operations.

Consider:

- Log retention periods that match troubleshooting and compliance needs.
- Automated deletion of temporary files, exports, and test data.
- Database retention and backup policies that avoid indefinite growth.
- Object expiration for generated reports and media derivatives.

Example CloudWatch Logs retention:

```bash
aws logs put-retention-policy \
  --log-group-name /aws/lambda/orders-api \
  --retention-in-days 30
```

This is useful when:

- You need recent logs for debugging but not years of history.
- Compliance does not require long-term retention in the log service itself.
- You want to avoid accumulating unnecessary storage.

**Architecture patterns to prefer**

Some patterns are often more sustainable because they reduce idle infrastructure:

- **Serverless APIs** with API Gateway and Lambda for intermittent traffic.
- **Event-driven workflows** with EventBridge, SQS, and Lambda.
- **Container services** with autoscaling for microservices that need more control than Lambda.
- **Caching layers** to reduce repeated backend work.
- **Batch processing** that runs on demand rather than continuously.

A simple event-driven topology:

```text
Client -> API Gateway -> Lambda -> SQS -> Worker Lambda
                           |
                           +-> DynamoDB
```

Why this is efficient:

- API Gateway and Lambda scale with demand.
- SQS buffers spikes so workers do not need to be overprovisioned.
- DynamoDB removes database server management for key-value access patterns.

**Operational practices that reduce waste**

Sustainability is also an operations discipline. You should continuously identify and remove unused resources.

Useful practices:

- Tag resources by owner, environment, and expiration date.
- Review idle EC2 instances, load balancers, EBS volumes, and Elastic IPs.
- Delete old AMIs and snapshots that are no longer needed.
- Use AWS Cost Explorer and Trusted Advisor to find underutilized resources.
- Automate cleanup in nonproduction environments.
- Use Infrastructure as Code so you can recreate environments instead of keeping them running.

Example cleanup commands:

```bash
aws ec2 describe-volumes --filters Name=status,Values=available
aws ec2 describe-addresses
aws ec2 describe-snapshots --owner-ids self
```

If you find unattached resources, verify they are not needed before deletion.

**Exam-focused comparisons**

| Scenario | More sustainable choice | Why |
|---|---|---|
| Spiky web traffic | Auto Scaling or Lambda | Matches capacity to demand |
| Batch jobs with interruption tolerance | Spot Instances | Uses spare capacity efficiently |
| Long-term infrequently accessed files | S3 Glacier class | Reduces storage footprint |
| Repeated static content delivery | CloudFront | Reduces origin load |
| Bursty asynchronous processing | SQS + Lambda | Decouples and scales on demand |
| Idle development environments | Stop or schedule shutdown | Eliminates unnecessary runtime |

**Common traps**

- Choosing a large always-on fleet when a smaller autoscaled fleet works.
- Keeping data in hot storage when access is rare.
- Using polling loops instead of event-driven messaging.
- Leaving test environments running after hours.
- Retaining logs, snapshots, and backups far beyond business need.
- Using self-managed servers when a managed service fits the workload.

**How to answer sustainability questions on the exam**

When you see a question about sustainability, ask yourself:

1. Can you reduce always-on capacity?
2. Can you use a managed or serverless service?
3. Can you move data to a colder storage tier?
4. Can you reduce network transfer or repeated work?
5. Can you automate scaling, retention, or cleanup?

The best answer usually improves one or more of these without violating availability, security, or performance requirements.

**Quick reference**

- **Compute:** right-size, autoscale, use serverless where appropriate.
- **Storage:** lifecycle policies, archival tiers, delete unused data.
- **Network:** cache, endpoint, compress, reduce chatter.
- **Operations:** tag, monitor, clean up, automate.
- **Design:** prefer managed, event-driven, and elastic architectures.

**Summary**

Sustainable AWS design is about eliminating waste while meeting workload requirements. For SAA-C03, the most sustainable architecture is usually the one that scales with demand, stores data in the right tier, minimizes unnecessary data transfer, and avoids idle infrastructure. If two solutions both work, prefer the one that uses fewer resources over time without compromising the stated requirements.
