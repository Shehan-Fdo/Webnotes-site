## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Security pillar mindset**

AWS security design is usually about reducing blast radius, enforcing least privilege, protecting data at rest and in transit, and making detection and response fast. For SAA-C03, you should be able to choose the right control for the right layer: identity, network, application, storage, logging, and automation.

A strong answer often combines multiple controls:
- **Preventive** controls stop bad access before it happens.
- **Detective** controls reveal suspicious activity.
- **Corrective** controls automate remediation after a finding.
- **Deterrent** controls reduce the chance of misuse through visibility and governance.

**IAM fundamentals: authentication vs authorization**

<abbr title="Identity and Access Management">IAM</abbr> separates who you are from what you can do.

- **Authentication** proves identity.
  - IAM users authenticate with passwords and optional MFA.
  - Roles are assumed and issue temporary credentials.
  - Federation lets external identities sign in through an identity provider.
- **Authorization** determines allowed actions.
  - Policies grant or deny permissions.
  - Evaluation considers explicit denies first.
  - Permissions can come from identity-based policies, resource-based policies, permissions boundaries, session policies, SCPs, and service-specific controls.

Common identity types:
- **IAM users** are long-term identities for humans or workloads that cannot use federation.
- **IAM groups** simplify permission management for multiple users.
- **IAM roles** are assumed identities with temporary credentials.
- **AWS STS** (<abbr title="Security Token Service">STS</abbr>) issues temporary credentials for role assumption, federation, and cross-account access.

When to use role assumption:
- Cross-account access to shared services.
- EC2, Lambda, ECS, or EKS workloads that need temporary permissions.
- Human access through federation or AWS IAM Identity Center.
- Short-lived elevated access instead of long-lived access keys.

Example CLI role assumption:
```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/ReadOnlyAuditRole \
  --role-session-name audit-session
```

Temporary credentials are preferred because they:
- Expire automatically.
- Reduce the risk of leaked long-term keys.
- Support session tagging and traceability.

**IAM policy evaluation and common exam traps**

Policy evaluation is a frequent exam topic because the answer depends on how multiple policy types interact.

- **Explicit deny wins** over any allow.
- **Identity-based policy** must allow the action for the principal.
- **Resource-based policy** must allow the principal when the resource supports it.
- **Service control policies** (<abbr title="Service Control Policies">SCPs</abbr>) set the maximum permissions in an AWS Organization.
- **Permissions boundaries** limit what an IAM principal can ever receive, even if another policy allows more.
- **Session policies** further restrict a role session.

Practical interpretation:
- For an IAM user or role in the same account, access is allowed only when the relevant identity policy and resource policy, if applicable, both permit it and no explicit deny applies.
- For cross-account access, a resource-based policy or role trust policy is usually required in addition to the caller’s permissions.

Example identity policy:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListAndReadS3",
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket",
        "s3:GetObject"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

**AWS Organizations security basics**

AWS Organizations helps centralize governance across multiple accounts.

- Use **SCPs** to prevent risky actions, such as disabling CloudTrail or deleting security logs.
- Apply guardrails at the organizational unit level when possible.
- Use a dedicated security account for logging and detective controls.
- Protect the management account with extra care because it can administer the organization.

Root account best practices:
- Enable **MFA** on the root user immediately.
- Do not create access keys for the root user.
- Use the root user only for tasks that require it, such as changing account settings or closing the account.
- Store root credentials securely and monitor root activity.

**KMS: key types, access model, and encryption patterns**

<abbr title="Key Management Service">KMS</abbr> is central to AWS encryption design. The exam often tests which key type to use and how access is granted.

Key types:
- **AWS owned keys**
  - Fully managed by AWS.
  - You do not view or manage the key policy.
  - Common for default encryption in some services.
- **AWS managed keys**
  - Created and managed by AWS for a specific service in your account.
  - The service manages most of the lifecycle.
  - You can usually see usage in CloudTrail, but control is limited.
- **Customer managed keys**
  - You create, configure, rotate, disable, and delete them.
  - You control the key policy and optional grants.
  - Best when you need fine-grained control, cross-account use, or compliance requirements.

Key algorithms:
- **Symmetric keys** use the same key for encrypt and decrypt.
  - Most AWS service integrations use symmetric KMS keys.
  - Best for envelope encryption and service-side encryption.
- **Asymmetric keys** use a public/private key pair.
  - Used for signing or specialized encryption workflows.
  - Less common in SAA-C03 scenarios.

Access model:
- KMS authorization is not just “IAM or key policy alone.”
- For IAM principals in the same account, access generally requires both the IAM policy and the key policy to allow the action, unless a grant or service integration changes the pattern.
- Some AWS services can use a customer managed key when the key policy directly allows the service principal or the caller, so do not assume IAM permissions are always required in every service path.
- **Grants** are useful for temporary, constrained access, especially for AWS services that need to use a key on your behalf.

Use cases for grants:
- Allowing an AWS service to use a key without broad IAM permissions.
- Short-lived access for a specific application or workflow.
- Delegating limited KMS operations such as encrypt, decrypt, or data key generation.

Envelope encryption:
- KMS encrypts a **data key**.
- Your application uses the plaintext data key to encrypt the data locally.
- The encrypted data key is stored with the ciphertext.
- This reduces KMS API calls and scales better for large data volumes.

Envelope encryption flow:
1. Call `GenerateDataKey` or `GenerateDataKeyWithoutPlaintext`.
2. Use the plaintext data key to encrypt data locally.
3. Store the encrypted data key with the encrypted object or record.
4. Decrypt by sending the encrypted data key to KMS.

Example CLI:
```bash
aws kms generate-data-key \
  --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555 \
  --key-spec AES_256
```

Rotation and lifecycle:
- Customer managed symmetric keys can use automatic rotation.
- Rotation changes the backing key material while preserving the logical key ID.
- Asymmetric keys do not use the same automatic rotation pattern as symmetric keys.
- Disabling a key immediately affects dependent workloads, so use it carefully for incident response.

Multi-Region keys:
- Multi-Region KMS keys help with disaster recovery and regional failover.
- You replicate the primary key to another Region as a related replica key.
- Applications in the failover Region can use the replica without redesigning the encryption model.
- Plan for key policy consistency and application failover together.

> If you encrypt data with a Region-specific key and fail over to another Region without a replica, decryption and re-encryption workflows can break.

**S3 security: public access, ownership, ACLs, and encryption**

Amazon S3 security is a common exam area because multiple controls overlap.

**Block Public Access**
- Account-level and bucket-level settings can block public ACLs and public bucket policies.
- Use it as a safety net when you do not intend public access.
- It can override attempts to make a bucket public, even if a policy allows it.

Important interactions:
- Block Public Access can prevent public exposure even if a bucket policy or ACL tries to allow it.
- It does not replace good policy design; it reduces accidental exposure.
- Use it broadly for private data unless public hosting is intentional.

**Object Ownership and ACLs**
- ACLs are legacy-style access controls and are usually avoided in new designs.
- **Bucket policies** are preferred for most access control.
- **Object Ownership** with `Bucket owner enforced` disables ACLs and makes the bucket owner own all objects.
- This is especially useful for cross-account uploads because it removes ownership confusion.

When ACLs still matter:
- Some older integrations still rely on them.
- Cross-account object ownership scenarios may appear in legacy architectures.
- In modern designs, prefer bucket policies and Object Ownership.

**Encryption at rest**
- Use **SSE-S3** for simple managed encryption.
- Use **SSE-KMS** when you need auditability, key control, or cross-service governance.
- Use **SSE-C** only when the customer provides the key on each request; this is uncommon in exam scenarios.

Example bucket policy to require TLS and deny unencrypted transport:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyInsecureTransport",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ],
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    }
  ]
}
```

A common encryption-enforcement pattern is to deny uploads unless the request uses the expected server-side encryption header. In practice, you should remember that multipart uploads and different encryption modes can require additional conditions, so exam questions may simplify the pattern.

Example policy to require SSE-KMS on `PutObject`:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnencryptedUploads",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::example-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    }
  ]
}
```

**S3 Object Lock**
- Enable Object Lock when you create the bucket.
- Versioning is automatically enabled for that bucket.
- You cannot turn on Object Lock later for an existing bucket.

Modes:
- **Governance mode**
  - Most users cannot overwrite or delete protected versions.
  - Privileged users with special permissions can bypass retention.
- **Compliance mode**
  - No one can overwrite or delete protected versions until retention expires.
  - Use this for strict regulatory retention.

Additional controls:
- **Retention period** defines how long a version is protected.
- **Legal hold** blocks deletion until the hold is removed, regardless of retention date.

Use cases:
- Audit logs.
- Financial records.
- Regulatory archives.
- Ransomware-resistant backups.

**S3 access patterns and cross-account sharing**

For cross-account access, prefer:
- Bucket policies for direct access.
- IAM roles for temporary access.
- Pre-signed URLs for time-limited object access.

Pre-signed URL example:
```bash
aws s3 presign s3://example-bucket/reports/q1.pdf --expires-in 3600
```

Use pre-signed URLs when:
- You need temporary access without exposing credentials.
- You want to share a single object, not the whole bucket.
- You need a simple download or upload workflow.

**Network security: security groups, NACLs, and private connectivity**

Security groups and network ACLs (NACLs) are both tested, but they behave differently.

Security groups:
- Act as stateful virtual firewalls at the instance or ENI level.
- Allow rules only; no explicit deny rules.
- Return traffic is automatically allowed.
- Best for most workload-level filtering.

NACLs:
- Act as stateless subnet-level filters.
- Support allow and deny rules.
- Require matching inbound and outbound rules for return traffic.
- Useful for coarse subnet guardrails or explicit deny requirements.

Practical guidance:
- Use security groups for primary workload protection.
- Use NACLs for subnet-level restrictions, such as blocking known bad CIDRs or adding an extra boundary.
- Do not rely on NACLs as your only application security layer.

Example security group rule concept:
- Allow inbound TCP 443 from the application load balancer security group.
- Allow outbound to the database security group on TCP 3306.
- Deny everything else by default through the implicit deny model.

Example AWS CLI to add a security group rule:
```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 443 \
  --source-group sg-0fedcba9876543210
```

NACL considerations:
- Rules are evaluated in order from lowest number to highest.
- The first matching rule wins.
- You must allow both directions for stateful-less traffic.
- Ephemeral port ranges vary by client and operating system, so be careful with blanket assumptions in exam questions.

Private connectivity:
- Use VPC endpoints to keep traffic on the AWS network.
- Use interface endpoints for many AWS services.
- Use gateway endpoints for S3 and DynamoDB.

Endpoint policy notes:
- Endpoint policies scope what actions and resources are allowed through the endpoint.
- They are not the same as resource-based policies and do not use a `Principal` element in the same way.
- For S3 gateway endpoints, focus on allowed actions and resources.

**Encryption in transit**

Encryption at rest is not enough. Many architectures need TLS between tiers.

Use TLS when:
- A client talks to an ALB or NLB listener.
- A web tier talks to an application tier.
- An application tier talks to a database or cache that supports TLS.
- Services exchange sensitive data across VPCs or accounts.

Common patterns:
- Terminate TLS at the load balancer and re-encrypt to the backend.
- Use end-to-end TLS when compliance or threat modeling requires it.
- Use ACM certificates for public certificates and private certificates.

Example ALB listener configuration concept:
- HTTPS listener on port 443.
- ACM certificate attached.
- Redirect HTTP 80 to HTTPS 443.

**AWS Certificate Manager and Private CA**

<abbr title="AWS Certificate Manager">ACM</abbr> simplifies certificate lifecycle management.

- Use ACM for public certificates on supported AWS services.
- Use **ACM Private CA** when you need private certificates for internal services, mTLS, or enterprise PKI.
- Private CA is useful for internal APIs, service-to-service authentication, and private trust chains.

When to choose Private CA:
- Internal microservices need certificates issued by your organization.
- You need certificate issuance automation for private workloads.
- You want centralized control over certificate policy and trust.

**Logging, detection, and auditability**

Security design is incomplete without visibility.

**CloudTrail**
- Records AWS API activity.
- Helps answer who did what, when, and from where.
- Essential for audit and incident response.

Important CloudTrail options:
- **Management events** capture control plane actions such as creating buckets or changing security groups.
- **Data events** capture object-level or function-level activity, such as S3 object access or Lambda invocation, and can increase cost.
- **Multi-Region trails** capture events across Regions.
- **Log file validation** helps detect tampering.
- **Organization trails** collect logs across all accounts in an AWS Organization.

Example CLI to create a trail:
```bash
aws cloudtrail create-trail \
  --name org-security-trail \
  --s3-bucket-name my-cloudtrail-logs-bucket \
  --is-multi-region-trail
```

Use CloudTrail data events when:
- You need object-level S3 access visibility.
- You need Lambda invocation auditing.
- You are investigating sensitive data access.

**VPC Flow Logs**
- Capture IP traffic metadata for network interfaces, subnets, or VPCs.
- Useful for troubleshooting and security analysis.
- Often feeds detection workflows and can support GuardDuty investigations.

What Flow Logs help you see:
- Accepted and rejected traffic.
- Source and destination addresses.
- Port and protocol information.
- Traffic patterns that suggest scanning or lateral movement.

**GuardDuty**
- Detects suspicious activity using signals from CloudTrail, VPC Flow Logs, DNS logs, and other sources.
- Good for threat detection, not prevention.
- Common findings include unusual API calls, port probing, and compromised credentials.

**AWS Config**
- Tracks resource configuration changes over time.
- Evaluates resources against desired rules.
- Helps answer whether a resource is compliant.

Remediation patterns:
- Use AWS Config rules to detect noncompliance.
- Trigger remediation through **EventBridge** or **Systems Manager Automation**.
- Automate fixes such as re-enabling encryption, restoring security group settings, or reverting public access.

Example remediation flow:
1. AWS Config detects a noncompliant S3 bucket.
2. EventBridge receives the compliance change event.
3. Systems Manager Automation runs a document to correct the setting.
4. The resource is re-evaluated.

**IAM Access Analyzer**
- Identifies resources that are shared externally or cross-account.
- Helps find unintended public or cross-account access.
- Useful for S3 buckets, IAM roles, KMS keys, Lambda permissions, and more.

Use it when:
- You want to validate resource policies before deployment.
- You need continuous visibility into external access.
- You are reviewing least-privilege posture across accounts.

**Secrets management**

AWS Secrets Manager stores and rotates secrets such as database credentials, API keys, and tokens.

Why use it:
- Centralized secret storage.
- Automatic rotation.
- Integration with RDS, Lambda, and custom rotation workflows.

Common rotation patterns:
- **Single-user rotation** changes the password for one database user.
- **Alternating users rotation** uses two users and switches between them to reduce downtime.
- **Custom Lambda rotation** handles non-database secrets or special workflows.

Example retrieval:
```bash
aws secretsmanager get-secret-value \
  --secret-id prod/db-credentials
```

Operational notes:
- Rotation requires the application to tolerate credential changes.
- Cache secrets carefully and refresh them before expiry.
- Use IAM permissions to restrict who can read or rotate secrets.

**WAF and Shield**

<abbr title="Web Application Firewall">WAF</abbr> protects web applications at Layer 7.

Use WAF for:
- SQL injection and cross-site scripting protections.
- IP reputation filtering.
- Geo-based restrictions.
- Rate limiting.
- Custom request inspection.

Useful WAF rule types:
- **Managed rule groups** for common threats.
- **Rate-based rules** to limit abusive clients.
- **Custom rules** for application-specific patterns.

Example WAF concepts:
- Block requests with suspicious query strings.
- Count requests first to validate impact.
- Rate-limit a login endpoint to reduce brute force attempts.

<abbr title="AWS Shield">Shield</abbr> protects against distributed denial-of-service attacks.

- **Shield Standard** is automatic and available by default.
- **Shield Advanced** adds enhanced detection, response support, and cost protection for supported resources.
- Choose Shield Advanced when you need stronger DDoS response and operational support for critical internet-facing workloads.

**Incident response and quarantine patterns**

A common exam scenario is isolating a compromised resource quickly.

Typical response actions:
- Remove or tighten security group rules.
- Detach the instance from load balancers.
- Snapshot EBS volumes for forensics.
- Preserve logs in a separate account or bucket.
- Rotate credentials and revoke sessions.

For EC2 isolation, the safest approach is usually to move the instance into a quarantine security group or restrict its network path rather than trying to “fix” the workload in place.

Example CLI to replace security groups on an attached network interface:
```bash
aws ec2 modify-network-interface-attribute \
  --network-interface-id eni-0123456789abcdef0 \
  --groups sg-quarantine12345678
```

Why this matters:
- It is operationally clearer than assuming a generic instance attribute change will always work.
- It avoids confusion about which interface is actually attached to the instance.
- It is a realistic containment step for a running workload.

**Practical security design patterns**

Common SAA-C03 patterns:
- Put private application servers in private subnets.
- Expose only load balancers in public subnets.
- Use security groups for tier-to-tier access.
- Store secrets in Secrets Manager, not in code.
- Encrypt S3, EBS, RDS, and backups.
- Use CloudTrail, Config, and GuardDuty together for visibility.
- Use IAM roles instead of access keys for AWS workloads.
- Use SCPs and permission boundaries to prevent privilege escalation.

Example three-tier security flow:
- Internet users connect to an ALB over HTTPS.
- The ALB forwards to app servers in private subnets.
- App servers access RDS over TLS using a security group rule.
- Secrets come from Secrets Manager.
- Logs go to a centralized S3 bucket with Object Lock and restricted access.

**High-value exam distinctions**

- **Security group vs NACL**
  - Security group is stateful and instance-scoped.
  - NACL is stateless and subnet-scoped.
- **IAM role vs IAM user**
  - Role is temporary and preferred for workloads.
  - User is long-term and better for human identities only when federation is not available.
- **KMS key types**
  - AWS owned: no customer control.
  - AWS managed: service-managed in your account.
  - Customer managed: full control and best for advanced requirements.
- **CloudTrail vs Config**
  - CloudTrail records API activity.
  - Config records configuration state and drift.
- **WAF vs Shield**
  - WAF filters HTTP(S) requests.
  - Shield mitigates DDoS.
- **Bucket policy vs ACL**
  - Bucket policy is preferred.
  - ACLs are legacy and often disabled by Object Ownership.
- **Governance vs compliance mode**
  - Governance can be bypassed by privileged users.
  - Compliance cannot be bypassed until retention expires.

```bash

```

**How to answer security questions on the exam**

When you see a scenario, ask:
- What is the asset being protected?
- Is the problem identity, network, data, logging, or response?
- Do you need prevention, detection, or remediation?
- Is the access human, application, cross-account, or service-to-service?
- Is the requirement public, private, or regulated retention?

Then choose the simplest control that satisfies the requirement:
- Use IAM roles for AWS workloads.
- Use KMS for encryption control and auditability.
- Use S3 Block Public Access for accidental exposure prevention.
- Use CloudTrail and Config for visibility and compliance.
- Use WAF for web attacks and Shield for DDoS.
- Use Object Lock for immutable retention.
