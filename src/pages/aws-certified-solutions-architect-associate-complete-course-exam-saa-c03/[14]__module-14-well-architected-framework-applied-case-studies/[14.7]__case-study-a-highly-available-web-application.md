## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**High availability design goals**

A highly available web application is designed to keep serving users when individual components fail. In AWS, that usually means removing single points of failure, spreading resources across multiple Availability Zones (AZs), and using managed services that can recover or scale automatically.

For exam purposes, focus on the difference between:
- **High availability**: survive component or AZ failure with minimal downtime.
- **Scalability**: handle growth in traffic or data volume.
- **Fault tolerance**: continue operating even when parts fail, often with more redundancy than a basic HA design.

A common reference architecture uses:
- Amazon Route 53 for DNS and traffic steering
- Application Load Balancer (ALB) across multiple AZs
- Auto Scaling groups for EC2-based application tiers
- Amazon RDS Multi-AZ or Amazon Aurora for the database tier
- Amazon CloudFront and Amazon S3 for static content
- ElastiCache for read-heavy caching
- Amazon CloudWatch for monitoring and alarms

**Core architecture pattern**

The standard HA web application pattern separates the application into tiers and distributes each tier across AZs. You typically place the load balancer in public subnets, application instances in private subnets, and the database in isolated private subnets.

A simple topology looks like this:

```text
Users
  |
Route 53
  |
CloudFront (optional for static/global edge delivery)
  |
Application Load Balancer
  |
Auto Scaling Group of EC2 instances
  |
RDS Multi-AZ / Aurora
```

Key design principles:
- **No single instance dependency**: one EC2 failure should not take down the app.
- **No single AZ dependency**: one AZ outage should not stop service.
- **Managed failover**: prefer AWS-managed recovery where possible.
- **Stateless app tier**: store sessions and uploads outside the instance when possible.

**Route 53 for DNS and routing**

Amazon Route 53 is a highly available DNS service used to route users to your application endpoints. It supports multiple routing policies, and exam questions often test when to use each one.

Common routing policies:
- **Simple routing**: one record, one endpoint.
- **Weighted routing**: split traffic by percentage for blue/green or gradual migration.
- **Latency-based routing**: route users to the AWS Region that is expected to provide the lowest latency based on AWS latency measurements.
- **Failover routing**: active/passive setup with health checks.
- **Geolocation routing**: route based on user location.
- **Geoproximity routing**: route based on geographic distance, often with bias adjustments.
- **Multivalue answer routing**: return multiple healthy records for simple client-side selection.

Practical use cases:
- Use **weighted routing** to shift 10% of traffic to a new version.
- Use **failover routing** for disaster recovery between primary and secondary Regions.
- Use **latency-based routing** when you have the same app deployed in multiple Regions and want users sent to the best-performing Region.

Example Route 53 failover record intent:
- Primary record points to the main ALB.
- Secondary record points to a standby ALB in another Region.
- Health checks determine when Route 53 should stop returning the primary endpoint.

> Route 53 health checks evaluate endpoint health from AWS health checkers, not from every end user request.

**Application Load Balancer**

An Application Load Balancer is the usual choice for HTTP and HTTPS web applications. It operates at Layer 7 and supports host-based and path-based routing, which is useful for microservices or multi-tenant apps.

Why ALB fits this case:
- Terminates TLS at the edge of the application tier.
- Distributes traffic across targets in multiple AZs.
- Supports health checks and automatic removal of unhealthy targets.
- Integrates with Auto Scaling groups.
- Supports routing rules such as `/api/*` to one target group and `/images/*` to another.

Useful ALB features:
- **Host-based routing**: `app.example.com` and `api.example.com` can go to different target groups.
- **Path-based routing**: `/admin` and `/shop` can route separately.
- **Sticky sessions**: available, but avoid relying on them for HA if you can externalize session state.
- **Cross-zone load balancing**: helps distribute traffic across healthy targets in all enabled AZs.

Example target group health check configuration:
```bash
aws elbv2 create-target-group \
  --name web-tg \
  --protocol HTTP \
  --port 80 \
  --vpc-id vpc-1234567890abcdef0 \
  --health-check-protocol HTTP \
  --health-check-path /health \
  --health-check-interval-seconds 30 \
  --health-check-timeout-seconds 5 \
  --healthy-threshold-count 5 \
  --unhealthy-threshold-count 2
```

Practical notes:
- Keep the health check endpoint lightweight and independent of the database when possible.
- If the app depends on the database, a database outage can cause all targets to fail health checks, which may be correct or may create unnecessary cascading failure. Design the health check intentionally.

**Auto Scaling for the application tier**

Auto Scaling groups keep the application tier available by replacing failed instances and adjusting capacity to demand. In a highly available design, you usually run at least two instances across two AZs.

Why Auto Scaling matters:
- Replaces unhealthy instances automatically.
- Spreads instances across AZs.
- Scales out during traffic spikes.
- Scales in when demand drops to reduce cost.

Typical configuration choices:
- **Minimum capacity**: at least 2 for HA.
- **Desired capacity**: enough to handle normal load with headroom.
- **Maximum capacity**: upper bound for burst traffic.
- **Health checks**: EC2 status checks plus ELB health checks.

Example launch template user data for a web server:
```bash
#!/bin/bash
set -euxo pipefail

dnf update -y
dnf install -y httpd
systemctl enable httpd
cat >/var/www/html/index.html <<'EOF'
<h1>Highly Available Web App</h1>
EOF
systemctl start httpd
```

If you need compatibility across Amazon Linux variants, verify the package manager for the AMI you selected rather than assuming one command works everywhere.

Operational tips:
- Bake as much configuration as possible into the launch template or AMI.
- Avoid manual instance setup after launch.
- Store app configuration in user data, SSM Parameter Store, or Secrets Manager rather than on the instance filesystem.

**Stateless application design**

A highly available web app is easier to operate when the application tier is stateless. Stateless means any instance can serve any request without relying on local disk state.

Common state to externalize:
- User sessions
- Uploaded files
- Cache data
- Application secrets
- Temporary job state

Typical AWS services for external state:
- **Amazon S3** for uploads, static assets, and backups
- **Amazon ElastiCache** for session caching or hot data
- **Amazon DynamoDB** for session state or metadata when a key-value model fits
- **Amazon EFS** when shared file storage is required across instances

Session handling patterns:
- Store sessions in ElastiCache or DynamoDB instead of instance memory.
- Use ALB stickiness only when you must, and understand it reduces flexibility during failover.
- Prefer token-based authentication when appropriate.

**Amazon S3 and CloudFront for static content**

Amazon S3 is the standard durable object storage service for static assets, backups, and user uploads. For a web application, S3 often stores images, CSS, JavaScript bundles, and downloadable files.

CloudFront is a content delivery network (CDN) that caches and serves content from edge locations closer to users. It is especially useful for static content and cacheable dynamic content.

What CloudFront does well:
- Reduces latency for globally distributed users.
- Offloads repeated requests from the origin.
- Supports TLS, custom domains, and origin failover.
- Works with S3, ALB, and custom origins.

What CloudFront does not do by default:
- It does not automatically cache most dynamic responses unless you configure cache behavior and the origin response is cacheable.
- It is not a substitute for application-level scaling.
- It does not make a stateful backend stateless.

Example CloudFront origin pattern:
- S3 origin for `/static/*`
- ALB origin for `/api/*`
- Separate cache behaviors for different paths

Example cache policy idea:
- Cache static assets for a long time with versioned filenames.
- Use short TTLs or no caching for personalized API responses.

Practical example of versioned asset naming:
```text
app.8f3c2a1.js
styles.2026-04-26.css
logo.v2.png
```

This lets you cache aggressively without serving stale content after deployments.

**Database high availability**

The database tier is often the most important part of the design. For SAA-C03, know the difference between Multi-AZ, read replicas, and Aurora.

**Amazon RDS Multi-AZ**
- Provides synchronous replication to a standby in another AZ.
- Supports automatic failover.
- Improves availability, not read scaling.
- The standby is not normally used for reads.

Use Multi-AZ when:
- You need automatic failover for a relational database.
- You want minimal application changes.
- You care more about availability than read scaling.

**Amazon RDS Read Replicas**
- Provide asynchronous replication.
- Improve read throughput.
- Can be promoted to standalone databases.
- Are not the same as Multi-AZ failover replicas.

Use read replicas when:
- Your workload is read-heavy.
- You need to offload reporting or analytics reads.
- You can tolerate replication lag.

**Amazon Aurora**
- Is a managed relational database engine compatible with MySQL and PostgreSQL.
- Uses distributed storage across multiple AZs.
- Provides fast failover and high availability.
- Supports Aurora Replicas for read scaling.

Aurora is often a strong choice when the exam asks for:
- High availability plus read scaling
- Fast failover
- Managed relational database with better resilience characteristics than standard single-AZ deployments

Comparison summary:
| Feature | RDS Multi-AZ | RDS Read Replica | Aurora |
|---|---|---|---|
| Primary purpose | HA/failover | Read scaling | HA + performance |
| Replication | Synchronous | Asynchronous | Distributed storage |
| Read traffic | Not primary use | Yes | Yes |
| Failover | Automatic | Manual promotion | Fast automatic failover |

**ElastiCache for performance**

Amazon ElastiCache is an in-memory data store service used to reduce database load and improve response times. The two common engines are Redis and Memcached.

Use ElastiCache when:
- You have repeated reads for the same data.
- You want to cache expensive database queries.
- You need a session store or ephemeral state store.
- You want to reduce latency for hot data.

Redis vs Memcached:
- **Redis**: supports richer data structures, persistence options, replication, and advanced features.
- **Memcached**: simpler, multi-threaded, and good for basic caching.

Important exam nuance:
- ElastiCache is a cache, not your primary durable system of record.
- If you use Redis persistence features, treat them as durability aids for the cache layer, not as a replacement for a proper database design.

Example cache-aside pattern:
1. Check ElastiCache for the key.
2. If present, return the cached value.
3. If absent, query the database.
4. Store the result in cache with a TTL.
5. Return the result to the user.

Example Redis CLI check:
```bash
redis-cli -h mycache.xxxxxx.use1.cache.amazonaws.com -p 6379
PING
GET user:1234
```

**Monitoring, alarms, and health**

Amazon CloudWatch collects metrics, logs, and events for AWS resources and custom applications. In a highly available design, monitoring is part of the architecture, not an afterthought.

What to monitor:
- ALB `HTTPCode_Target_5XX_Count`
- ALB `TargetResponseTime`
- EC2 CPU, status checks, and memory via agent
- RDS CPU, connections, storage, and failover events
- Auto Scaling activity
- Application logs and error rates
- Cache hit rate and eviction count for ElastiCache

Useful alarm patterns:
- Alarm on elevated 5xx errors from the load balancer.
- Alarm on unhealthy host count dropping below expected capacity.
- Alarm on RDS failover or storage exhaustion.
- Alarm on latency spikes that indicate origin or database pressure.

Example CloudWatch alarm for ALB 5xx errors:
```bash
aws cloudwatch put-metric-alarm \
  --alarm-name alb-target-5xx-high \
  --namespace AWS/ApplicationELB \
  --metric-name HTTPCode_Target_5XX_Count \
  --dimensions Name=LoadBalancer,Value=app/my-alb/1234567890abcdef \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 10 \
  --comparison-operator GreaterThanThreshold
```

Be careful with metric dimensions:
- Use the exact load balancer or target group dimension names emitted by CloudWatch.
- Match the metric to the question’s intent.
- For target health, consider the target group health metrics and the operational meaning of the threshold rather than using a generic example blindly.

**Security and network layout**

A secure HA web app usually uses a three-tier VPC layout:
- **Public subnets**: ALB, NAT Gateway
- **Private subnets**: application instances
- **Isolated private subnets**: database instances

Security best practices:
- Place the ALB in public subnets only.
- Place EC2 application servers in private subnets.
- Restrict database access to the application security group.
- Use security groups as the primary stateful firewall.
- Use network ACLs only when you need subnet-level stateless filtering.

Example security group flow:
- Internet → ALB security group on 443
- ALB security group → app security group on app port
- App security group → database security group on 3306 or 5432

Example inbound rule intent:
```text
ALB SG:
- Allow HTTPS from 0.0.0.0/0

App SG:
- Allow HTTP from ALB SG only

DB SG:
- Allow MySQL/Aurora from App SG only
```

**Deployment and failover strategies**

A highly available web app often needs safe deployment patterns so updates do not create downtime.

Common strategies:
- **Rolling deployment**: replace instances gradually.
- **Blue/green deployment**: deploy a parallel environment and switch traffic.
- **Canary deployment**: send a small percentage of traffic to the new version first.

When to use each:
- Use rolling updates for simple EC2 fleets.
- Use blue/green when you want a clean rollback path.
- Use canary when you want to validate production behavior with limited exposure.

Example blue/green flow:
1. Deploy the new version to a green environment.
2. Run health checks and smoke tests.
3. Shift Route 53 weighted traffic or ALB target group traffic.
4. Monitor errors and latency.
5. Roll back quickly if metrics degrade.

**Disaster recovery considerations**

High availability within one Region is not the same as disaster recovery across Regions. Exam questions may ask you to distinguish between them.

Common DR patterns:
- **Backup and restore**: lowest cost, highest recovery time.
- **Pilot light**: minimal core services running in secondary Region.
- **Warm standby**: scaled-down but functional secondary environment.
- **Multi-site active/active**: both Regions serve traffic.

For a highly available web app:
- Use Multi-AZ for intra-Region resilience.
- Use Route 53 failover or latency-based routing for multi-Region resilience.
- Replicate data appropriately, such as S3 Cross-Region Replication or database replication where supported.

Important tradeoff:
- Multi-Region active/active improves resilience but increases complexity, cost, and data consistency challenges.

**Common exam traps**

Watch for these distinctions:
- **ALB vs NLB**: ALB is for HTTP/HTTPS and Layer 7 routing; Network Load Balancer is for Layer 4 and ultra-low latency TCP/UDP use cases.
- **Multi-AZ vs read replica**: Multi-AZ is for failover; read replicas are for read scaling.
- **CloudFront vs S3**: S3 stores objects; CloudFront caches and delivers them globally.
- **ElastiCache vs database**: cache data is ephemeral; do not treat it as the system of record.
- **Route 53 latency routing**: it uses AWS latency measurements, not a live per-request client probe.
- **Health checks**: a failing health check should reflect real service health, not just instance reachability.

**Practical reference architecture**

A strong exam-ready architecture for a highly available web app looks like this:

- Route 53 routes users to CloudFront or directly to ALB.
- CloudFront serves static assets from S3 and forwards dynamic requests to ALB.
- ALB spans at least two AZs.
- Auto Scaling group runs EC2 instances in private subnets across multiple AZs.
- Application instances use ElastiCache for session or query caching.
- Database runs on RDS Multi-AZ or Aurora.
- CloudWatch alarms monitor errors, latency, and health.
- Security groups restrict traffic between tiers.

Example request flow:
```text
Client
  -> Route 53
  -> CloudFront
  -> ALB
  -> App instances in private subnets
  -> ElastiCache / RDS / S3
```

**How to reason about scenario questions**

When the exam describes a web app with downtime risk, ask:
- Is the failure domain an instance, AZ, or Region?
- Is the bottleneck compute, storage, or DNS?
- Does the app need read scaling, failover, or both?
- Is the data tier stateful and how is it replicated?
- Can static content be offloaded to S3 and CloudFront?

A good answer usually:
- Removes single points of failure
- Uses managed services
- Keeps the app tier stateless
- Separates availability from scalability
- Chooses the simplest service that meets the requirement

**Quick command examples**

Check an ALB target group:
```bash
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/web-tg/abcdef1234567890
```

Inspect Auto Scaling group health:
```bash
aws autoscaling describe-auto-scaling-groups \
  --auto-scaling-group-names web-asg
```

Query CloudWatch metrics:
```bash
aws cloudwatch get-metric-statistics \
  --namespace AWS/ApplicationELB \
```

**Final takeaways**

A highly available web application on AWS is usually built by combining DNS routing, load balancing, stateless compute, resilient storage, and strong monitoring. The most important exam skill is matching the service to the failure mode: use Route 53 for traffic steering, ALB for HTTP routing, Auto Scaling for instance replacement, Multi-AZ or Aurora for database resilience, and CloudFront/S3 for global static delivery.
