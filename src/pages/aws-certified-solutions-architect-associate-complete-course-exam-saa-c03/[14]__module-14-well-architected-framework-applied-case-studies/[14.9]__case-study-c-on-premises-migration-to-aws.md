## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Migration strategy overview**

An on-premises migration to AWS usually starts with discovery, assessment, landing zone design, connectivity, and then workload-by-workload migration. For exam scenarios, choose the service that matches the workload type, downtime tolerance, and data change rate.

A practical migration plan often includes:

- **Assess** applications, servers, databases, and dependencies.
- **Design** the target AWS environment with accounts, networking, security, and logging.
- **Replicate** data and systems with minimal disruption.
- **Cut over** during a controlled window.
- **Validate** performance, security, and business functionality.
- **Retire** or keep the source as rollback insurance until stabilization.

**Discovery and assessment**

Before you migrate, you need a clear inventory of what exists and how systems depend on each other. This reduces surprises during cutover and helps you choose between rehost, replatform, refactor, or retire.

Common assessment inputs include:

- Server count, OS versions, CPU, memory, storage, and network usage.
- Application dependencies, ports, and authentication flows.
- Database size, write rate, and replication feasibility.
- Compliance constraints, data residency, and encryption requirements.
- Peak traffic patterns and maintenance windows.

Useful AWS and partner tools in this phase include:

- **AWS Application Discovery Service** for collecting server and dependency data.
- **AWS Migration Evaluator** for cost and sizing analysis.
- **AWS Migration Hub** for tracking migration progress across tools.
- **AWS Server Migration Connector** or **VM Import/Export** for legacy scenarios where they fit the environment.

> Use assessment data to size the target environment conservatively. Under-sizing the landing zone creates avoidable performance issues during the first wave of migrations.

**Landing zone and target foundation**

A landing zone is the secure, governed AWS foundation where migrated workloads land. In exam questions, this usually means multi-account structure, centralized logging, guardrails, and network segmentation.

A strong landing zone typically includes:

- **AWS Control Tower** for account vending and guardrails.
- Separate accounts for shared services, security, logging, and workloads.
- Centralized CloudTrail, AWS Config, and log archive storage.
- IAM Identity Center or federated access for workforce authentication.
- Standardized VPC patterns and naming conventions.

A common account layout is:

- **Management account** for governance.
- **Log archive account** for immutable logs.
- **Security account** for detection and response tooling.
- **Shared services account** for DNS, directory services, or CI/CD.
- **Application accounts** for migrated workloads.

When you need repeatable governance, Control Tower is usually the best starting point. If the scenario emphasizes manual control, you can still build the same structure with Organizations, SCPs (service control policies), and Infrastructure as Code.

**Hybrid connectivity design**

Migration traffic often needs private connectivity between on-premises and AWS for replication, testing, and phased cutover. The right choice depends on bandwidth, latency, and resilience requirements.

Typical options:

- **Site-to-Site VPN** for fast setup and lower-cost connectivity.
- **AWS Direct Connect** for consistent throughput and lower jitter.
- **Direct Connect Gateway** when you need to reach multiple VPCs across Regions from a single Direct Connect attachment.
- **Transit Gateway** when you need hub-and-spoke routing across many VPCs and on-premises networks.
- **VPN failover** as backup for Direct Connect in case the dedicated circuit fails.

A common resilient pattern is:

1. Build Direct Connect for primary private connectivity.
2. Add Site-to-Site VPN as backup.
3. Attach VPCs through Transit Gateway if multiple networks must share the same hybrid path.
4. Use route priorities and health checks to fail over cleanly.

Example hybrid routing pattern:

```text
On-premises network
   |\
   | \__ Site-to-Site VPN (backup)
   |
   \____ Direct Connect -> Direct Connect Gateway -> Transit Gateway -> VPCs
```

For exam prep, remember:

- Use **Direct Connect** for predictable, high-volume replication.
- Use **VPN** when speed of deployment matters more than throughput.
- Use **Transit Gateway** to simplify many-to-many routing.
- Use **Direct Connect Gateway** when one connection must reach multiple VPCs or Regions.

**Server migration patterns**

For virtual machines and physical servers, AWS Application Migration Service (MGN) is the primary lift-and-shift tool. It continuously replicates source servers into AWS and launches test or cutover instances when ready.

MGN is a good fit when:

- You want minimal changes to the source application.
- You need low downtime.
- You are migrating many servers with similar patterns.
- You want a repeatable wave-based cutover process.

A typical MGN workflow is:

1. Install the MGN agent on the source server.
2. Configure replication settings and staging area networking.
3. Let continuous block-level replication run.
4. Launch a test instance in AWS.
5. Validate application behavior.
6. Schedule cutover and launch the production instance.

Example MGN setup commands on a Linux source server often look like this:

```bash
curl -O https://aws-application-migration-service-agent.s3.amazonaws.com/latest/linux/aws-replication-installer-init
chmod +x aws-replication-installer-init
sudo ./aws-replication-installer-init \
  --region us-east-1 \
  --aws-access-key-id AKIAEXAMPLE \
  --aws-secret-access-key EXAMPLESECRET \
  --no-prompt
```

In practice, you usually prefer IAM roles or temporary credentials rather than long-lived keys. The important exam concept is that MGN uses continuous replication and supports test launches before cutover.

**Database migration patterns**

Database migration is often the most sensitive part of an on-premises move because it requires consistency, low downtime, and careful cutover planning. AWS Database Migration Service (DMS) is the standard service for moving databases with minimal interruption.

DMS supports:

- **Full load** for initial copy.
- **Change data capture (CDC)** for ongoing replication of inserts, updates, and deletes.
- **Full load plus CDC** for most production migrations.

This pattern is common:

1. Start a full load from source to target.
2. Keep CDC running to capture ongoing changes.
3. Validate schema, indexes, and application compatibility.
4. Put the source database into a read-only or maintenance window.
5. Let CDC catch up to zero lag.
6. Switch the application to the AWS target database.

Example DMS task settings for a PostgreSQL-to-Aurora migration:

```json
{
  "TargetMetadata": {
    "TargetSchema": "",
    "SupportLobs": true,
    "FullLobMode": false,
    "LimitedSizeLobMode": true,
    "LobMaxSize": 32
  },
  "FullLoadSettings": {
    "TargetTablePrepMode": "DROP_AND_CREATE"
  },
  "Logging": {
    "EnableLogging": true
  }
}
```

A practical DMS CLI flow is:

```bash
aws dms create-replication-instance \
  --replication-instance-identifier prod-migration-ri \
  --replication-instance-class dms.t3.medium \
  --allocated-storage 100 \
  --vpc-security-group-ids sg-0123456789abcdef0 \
  --replication-subnet-group-identifier dms-subnets

aws dms create-endpoint \
  --endpoint-identifier source-pg \
  --endpoint-type source \
  --engine-name postgres \
  --username appuser \
  --password 'ExamplePassword1!' \
  --server-name onprem-db.example.local \
  --port 5432 \
  --database-name appdb

aws dms create-endpoint \
  --endpoint-identifier target-aurora \
  --endpoint-type target \
  --engine-name aurora-postgresql \
  --username admin \
  --password 'ExamplePassword1!' \
  --server-name aurora-cluster.cluster-abc123.us-east-1.rds.amazonaws.com \
  --port 5432 \
  --database-name appdb

aws dms create-replication-task \
  --replication-task-identifier appdb-full-load-cdc \
  --source-endpoint-arn arn:aws:dms:us-east-1:123456789012:endpoint:SRC \
  --target-endpoint-arn arn:aws:dms:us-east-1:123456789012:endpoint:TGT \
  --replication-instance-arn arn:aws:dms:us-east-1:123456789012:rep:RI \
  --migration-type full-load-and-cdc \
  --table-mappings file://table-mappings.json \
  --replication-task-settings file://task-settings.json
```

Important DMS planning points:

- Use **CDC** when you need near-zero downtime.
- Use a read-only window for the final sync if the application cannot tolerate write divergence.
- Validate character sets, LOBs (large objects), triggers, and stored procedures separately.
- Rehearse cutover timing so you know how long the final lag catch-up takes.

**Data migration for files and objects**

For file shares, home directories, media libraries, and bulk object movement, AWS DataSync is usually the best fit. It automates transfer, verification, scheduling, and incremental sync.

DataSync is useful when you need:

- NFS or SMB file transfer.
- Incremental synchronization.
- Integrity verification.
- Scheduled recurring transfers.
- High-throughput movement without custom scripts.

A typical DataSync pattern for on-premises file systems is:

1. Deploy a DataSync agent in your on-premises environment.
2. Create source and destination locations.
3. Configure a task with filters, bandwidth limits, and verification.
4. Run an initial full transfer.
5. Schedule incremental syncs until cutover.
6. Freeze writes, run a final sync, and switch clients.

Example DataSync task configuration:

```json
{
  "Name": "home-directories-migration",
  "SourceLocationArn": "arn:aws:datasync:us-east-1:123456789012:location/loc-11111111111111111",
  "DestinationLocationArn": "arn:aws:datasync:us-east-1:123456789012:location/loc-22222222222222222",
  "Options": {
    "VerifyMode": "POINT_IN_TIME_CONSISTENT",
    "OverwriteMode": "ALWAYS",
    "Atime": "BEST_EFFORT",
    "Mtime": "PRESERVE",
    "TransferMode": "CHANGED"
  }
}
```

A practical CLI example:

```bash
aws datasync create-task \
  --name home-directories-migration \
  --source-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-11111111111111111 \
  --destination-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-22222222222222222 \
  --options VerifyMode=POINT_IN_TIME_CONSISTENT,OverwriteMode=ALWAYS,TransferMode=CHANGED
```

For exam scenarios, remember:

- Use **DataSync** for file and object movement.
- Use **S3 Transfer Acceleration** only when internet-based S3 uploads need edge optimization.
- Use **Snowball** when network transfer is too slow or impractical for very large datasets.

**Large-scale offline transfer**

When bandwidth is insufficient or the data volume is huge, AWS Snowball can be the right answer. It is especially relevant for initial bulk loads before ongoing replication takes over.

Use Snowball when:

- The dataset is too large for timely network transfer.
- The link is unreliable or expensive.
- You need to seed a migration quickly.

A common hybrid pattern is:

- Ship initial data with Snowball.
- Start DMS CDC or DataSync incremental sync afterward.
- Cut over once the delta is small enough to transfer online.

**Application cutover and DNS strategy**

Cutover is where many migrations fail if the plan is vague. You need a controlled sequence for stopping writes, syncing final changes, switching traffic, and validating the new environment.

A safe cutover sequence often looks like this:

1. Lower DNS TTL well before the cutover window.
2. Freeze writes on the source system.
3. Run the final replication or sync.
4. Validate data consistency and application health.
5. Update DNS or load balancer targets.
6. Monitor logs, metrics, and user transactions.
7. Keep the source environment intact for rollback.

For DNS-based traffic shifts, Route 53 can help you move traffic gradually or quickly. If you need weighted or failover routing, include the routing policy fields in the record set.

Example weighted record change:

```json
{
  "Comment": "Shift 10 percent of traffic to AWS",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "aws-primary",
        "Weight": 10,
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack.alb-1234567890.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

If you want a simple all-at-once switch, a standard `UPSERT` to the new target is enough. If you want gradual rollout or rollback control, use weighted or failover routing explicitly.

**Rollback planning**

Rollback is not an afterthought. You need a plan that preserves the source environment long enough to recover if the AWS cutover exposes a problem.

A practical rollback plan includes:

- Keep the source database and servers powered on but isolated from user writes.
- Preserve the original DNS record values and TTL settings.
- Snapshot AWS targets before cutover if possible.
- Define a hard decision point for rollback versus forward fix.
- Document who can authorize rollback and how long the window lasts.

A simple rollback procedure is:

1. Detect a critical issue after cutover.
2. Stop new writes to the AWS target.
3. Revert DNS to the source endpoint.
4. Re-enable source application services.
5. Reconcile any data written during the failed cutover window.
6. Postpone the next migration wave until root cause analysis is complete.

> If the source system accepts writes after cutover, rollback becomes much harder because you must reconcile divergent data. Keep the source controlled until the new path is stable.

**Hybrid database and application patterns**

Some migrations use a phased approach where the application moves before the database, or vice versa. The right order depends on coupling and latency sensitivity.

Common patterns:

- **Database first** when the app can tolerate temporary connectivity changes.
- **Application first** when the database remains on-premises during a transition period.
- **Parallel run** when you need to compare outputs before final cutover.

For stateful applications, consider:

- Session storage in ElastiCache or DynamoDB instead of local memory.
- Shared files moved to Amazon EFS or S3.
- Secrets moved to AWS Secrets Manager or Systems Manager Parameter Store.
- Background jobs and queues moved to Amazon SQS or Amazon MQ if needed.

**Security and governance during migration**

Migration increases risk because you temporarily operate two environments. Security controls must cover both source and target.

Key controls include:

- Encrypt data in transit with TLS.
- Encrypt data at rest with AWS KMS-managed keys.
- Restrict replication and migration roles to least privilege.
- Log all administrative actions in CloudTrail.
- Use security groups and NACLs to limit migration traffic.
- Scan migrated images and containers before production use.

If the scenario mentions compliance, prioritize:

- Centralized logging.
- Immutable audit trails.
- Separation of duties.
- Controlled access to migration tooling.

**Common exam traps**

Watch for these distinctions:

- **MGN** is for server replication and lift-and-shift.
- **DMS** is for database migration and CDC.
- **DataSync** is for file and object transfer.
- **Snowball** is for offline bulk transfer.
- **Control Tower** is for landing zone governance.
- **Transit Gateway** simplifies hybrid routing across many networks.
- **Direct Connect Gateway** extends private connectivity across VPCs and Regions.

Also remember legacy or alternative options that may appear in scenario wording:

- **VM Import/Export** for importing existing VM images.
- **Server Migration Connector** in older or niche migration discussions.
- **Migration Evaluator** for cost and sizing analysis.

**End-to-end example migration plan**

A realistic migration for an on-premises three-tier application might look like this:

1. Build the landing zone with Control Tower, logging, and security accounts.
2. Establish Direct Connect with VPN backup.
3. Use Application Discovery Service and Migration Evaluator to inventory and size.
4. Replicate application servers with MGN.
5. Migrate the database with DMS using full load plus CDC.
6. Move shared file data with DataSync.
7. Test in a non-production VPC and validate dependencies.
8. Lower DNS TTL and rehearse cutover.
9. Freeze writes, complete final sync, and switch traffic.
10. Monitor, then decommission the source after the rollback window closes.

This sequence is a strong exam answer because it combines governance, connectivity, replication, and cutover discipline rather than treating migration as a single tool choice.

**Practical validation checklist**

Before you cut over, verify:

- Application health checks pass.
- Database replication lag is near zero.
- Security groups allow only required traffic.
- DNS points to the correct target.
- Logs and metrics are visible in the target account.
- Backups and snapshots are enabled.
- Rollback steps are documented and tested.

**Summary**

On-premises migration to AWS is usually a multi-service effort, not a single product decision. Use the right tool for each workload type, build a governed landing zone, design resilient hybrid connectivity, and plan cutover and rollback as first-class tasks.

The exam typically rewards answers that show you understand:

- Discovery before migration.
- Landing zone before workload movement.
- Continuous replication before cutover.
- Controlled DNS and write freezes during final switch.
- Rollback readiness until the new environment is stable.
