## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Disaster recovery design goals**

Disaster recovery (DR) is about restoring service after a disruptive event while balancing recovery time objective (<abbr title="Recovery Time Objective">RTO</abbr>), recovery point objective (<abbr title="Recovery Point Objective">RPO</abbr>), cost, and operational complexity. In AWS, the right design depends on how quickly you must recover, how much data loss is acceptable, and whether you need regional isolation or just zone-level resilience.

A strong exam answer usually starts by identifying the failure domain:

- **Single instance failure**: use Auto Scaling, health checks, or managed services.
- **Availability Zone (AZ) failure**: use Multi-AZ architectures.
- **Region failure**: use cross-Region replication, backups, and failover automation.
- **Account compromise or blast-radius reduction**: isolate DR assets in a separate AWS account.

For SAA-C03, you should choose the simplest architecture that meets the stated RTO and RPO. Overengineering is a common trap.

**DR strategy patterns**

AWS commonly frames DR in four patterns. Know the tradeoffs because exam questions often ask you to choose the cheapest design that still meets recovery targets.

| Pattern | RTO | RPO | Cost | Typical use |
|---|---:|---:|---:|---|
| Backup and restore | High | High | Low | Non-critical workloads, archival recovery |
| Pilot light | Medium | Low to medium | Low to medium | Core services kept running in DR Region |
| Warm standby | Low | Low | Medium to high | Faster recovery with scaled-down duplicate stack |
| Multi-site active/active | Very low | Very low | High | Mission-critical, global applications |

- **Backup and restore** means you restore data and infrastructure after a disaster.
- **Pilot light** keeps only the essential components running, such as databases and minimal app services.
- **Warm standby** keeps a scaled-down but functional environment ready to scale up.
- **Active/active** serves traffic from multiple Regions simultaneously and needs careful data consistency design.

> For exam decisions, prefer the least complex pattern that satisfies the RTO and RPO in the prompt.

**Multi-Region architecture and blast-radius reduction**

A resilient DR design often uses a separate DR Region and, for stronger isolation, a separate DR account in AWS Organizations. This reduces the blast radius if the primary account is compromised, misconfigured, or locked out.

Common isolation patterns include:

- **Separate DR account** in AWS Organizations
  - Store replicated backups and snapshots in a different account.
  - Use cross-account IAM roles for controlled restore access.
  - Apply service control policies (<abbr title="Service Control Policy">SCP</abbr>s) to restrict destructive actions.
- **Separate DR Region**
  - Replicate data to a second Region for regional outage recovery.
  - Keep infrastructure templates ready in the DR Region.
- **Separate DR VPC and network controls**
  - Use distinct CIDR ranges to avoid overlap during failover.
  - Pre-create security groups, route tables, and endpoints.

A separate DR account is especially useful when you need to protect against accidental deletion, ransomware, or compromised credentials in the primary account. It also helps with governance because the DR environment can have stricter permissions and limited human access.

**Backups, snapshots, and cross-Region copy**

Backups are the foundation of many DR designs. In AWS, you usually combine AWS Backup, service-native snapshots, and object replication.

Typical options:

- **AWS Backup**
  - Centralizes backup policies across supported services.
  - Supports cross-Region and cross-account copy.
  - Helps standardize retention and lifecycle rules.
- **Amazon EBS snapshots**
  - Incremental and stored in Amazon S3-managed infrastructure.
  - Can be copied to another Region and another account.
- **Amazon RDS snapshots**
  - Manual snapshots can be copied cross-Region.
  - Automated backups support point-in-time recovery within the same Region.
- **Amazon S3 replication**
  - Cross-Region Replication (<abbr title="Cross-Region Replication">CRR</abbr>) copies objects asynchronously.
  - Versioning must be enabled on source and destination buckets.

When backups are encrypted, you must plan for key access in the destination. Cross-Region or cross-account restore often requires:

- A destination <abbr title="Key Management Service">KMS</abbr> key in the target Region or account.
- Permission for the backup service or restore role to use the destination key.
- Re-encryption during copy or restore if the original key is not available in the DR environment.

For example, if you copy an encrypted EBS snapshot to another account, the destination account needs a KMS key policy that allows the copy and restore workflow.

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --destination-region us-west-2 \
  --encrypted \
  --kms-key-id arn:aws:kms:us-west-2:111122223333:key/abcd-1234
```

For S3 replication with encryption, you should ensure the destination bucket can encrypt replicated objects with a destination KMS key.

```json
{
  "Rules": [
    {
      "Status": "Enabled",
      "Priority": 1,
      "DeleteMarkerReplication": { "Status": "Disabled" },
      "Filter": { "Prefix": "" },
      "Destination": {
        "Bucket": "arn:aws:s3:::dr-bucket",
        "EncryptionConfiguration": {
          "ReplicaKmsKeyID": "arn:aws:kms:us-west-2:111122223333:key/abcd-1234"
        }
      }
    }
  ]
}
```

**Application-layer dependencies**

A DR design fails if the application tier is restored but its dependencies are not. In exam scenarios, always check the full dependency chain.

Key dependencies to evaluate:

- **Queues**
  - Amazon SQS queues may need cross-Region replication logic or replay from producers.
  - Message ordering and deduplication matter if you use FIFO queues.
  - Decide whether messages can be lost, replayed, or drained during failover.
- **Caches**
  - Amazon ElastiCache is usually treated as disposable unless cache warm state is critical.
  - Do not rely on cache contents as the system of record.
  - Rebuild caches after failover from the database or origin service.
- **Session state**
  - Store sessions in a shared, durable store such as DynamoDB, ElastiCache with persistence, or a database.
  - Avoid sticky sessions if you need Region-level failover.
- **External integrations**
  - Third-party APIs, payment gateways, identity providers, and webhooks may not be multi-Region aware.
  - Validate whether failover requires DNS changes, endpoint updates, or vendor coordination.
- **Secrets and configuration**
  - Replicate secrets securely using AWS Secrets Manager or Parameter Store where appropriate.
  - Ensure the DR Region has the same configuration values, feature flags, and certificates.

A practical DR checklist for application dependencies:

1. Identify every stateful component.
2. Classify each component as durable, rebuildable, or external.
3. Decide whether it is replicated, restored, or recreated during failover.
4. Test the application in the DR Region with real traffic or synthetic traffic.

**Route 53 failover and DNS timing**

Amazon Route 53 is a common control plane for DR traffic shifting. It can route traffic based on health checks, latency, weighted policies, or failover policies.

A basic failover setup uses a primary and secondary record with health checks.

```txt
Primary record: app.example.com -> ALB in us-east-1
Secondary record: app.example.com -> ALB in us-west-2
Health check: HTTPS /health on primary endpoint
Routing policy: Failover
```

Health check configuration considerations:

- **Protocol**: use HTTP or HTTPS for application-level checks when possible.
- **Path**: choose a lightweight endpoint such as `/health` that verifies critical dependencies only if needed.
- **Thresholds**: tune failure threshold and request interval to balance sensitivity and false positives.
- **TTL**: use a lower TTL, such as 30 to 60 seconds, when you need faster DNS convergence.
- **Tradeoff**: very low TTLs can increase DNS query volume and do not guarantee instant failover because client resolvers may cache responses.

A practical timing model:

- Health check detects failure.
- Route 53 marks the primary unhealthy.
- DNS resolvers expire cached records based on TTL.
- Clients reconnect to the secondary endpoint.

> DNS failover is not instantaneous. For strict RTOs, combine DNS with application retry logic and pre-warmed infrastructure.

**Compute and infrastructure recovery**

For EC2-based workloads, you usually restore infrastructure from templates rather than manually rebuilding servers. Infrastructure as Code (<abbr title="Infrastructure as Code">IaC</abbr>) is the preferred approach.

Useful tools and patterns:

- **AWS CloudFormation**
  - Recreates VPCs, subnets, security groups, load balancers, and Auto Scaling groups.
  - Supports stack sets for multi-account and multi-Region deployment.
- **AWS Systems Manager**
  - Automates runbooks for failover and failback.
  - Can execute commands, automation documents, and patching workflows.
- **Amazon Machine Images (AMIs)**
  - Store golden images for fast instance launch.
  - Copy AMIs to the DR Region before a disaster.
- **Launch templates**
  - Standardize instance configuration and bootstrap scripts.

Example Systems Manager Automation runbook pattern:

```yaml
schemaVersion: '0.3'
description: "Fail over application stack to DR Region"
assumeRole: arn:aws:iam::111122223333:role/SSMAutomationRole
mainSteps:
  - name: StopPrimaryWrites
    action: aws:executeAwsApi
    inputs:
      Service: rds
      Api: StopDBInstance
      DBInstanceIdentifier: primary-db
  - name: PromoteReplica
    action: aws:executeAwsApi
    inputs:
      Service: rds
      Api: PromoteReadReplica
      DBInstanceIdentifier: dr-replica
```

Automation matters because manual recovery is slow and error-prone. For exam questions, if the prompt mentions repeatable failover, choose Systems Manager, Lambda, Step Functions, or CloudFormation rather than manual console actions.

**Database DR choices**

Database design is often the deciding factor in DR architecture.

**Amazon RDS Multi-AZ**

- Provides synchronous replication to a standby in the same Region.
- Failover is automatic and usually fast.
- It is designed for high availability, not regional disaster recovery.
- It does not protect against Region-wide outages.

**Amazon RDS read replicas**

- Replication is asynchronous.
- You can promote a read replica to become a standalone primary.
- Promotion takes time and may involve replication lag, so RPO is not zero.
- Cross-Region read replicas are useful for DR, but they are not the same as Multi-AZ.

Comparison for exam decisions:

| Feature | Multi-AZ | Read Replica |
|---|---|---|
| Purpose | High availability | Read scaling and DR |
| Replication | Synchronous | Asynchronous |
| Failover | Automatic | Manual promotion |
| Region scope | Same Region | Same or cross-Region |
| RPO | Very low | Depends on lag |
| RTO | Low | Higher than Multi-AZ |

Use Multi-AZ when you need automatic recovery from AZ failure. Use cross-Region read replicas when you need a DR target in another Region and can tolerate promotion time and some replication lag.

For Amazon Aurora, cross-Region replication options are stronger than standard RDS in many cases, but you still need to understand whether the question is asking for HA or DR. Aurora Global Database can provide very low-latency cross-Region replication and fast regional failover, but it is still a specialized design choice.

**Failover and failback planning**

Failover is only half the story. You also need a failback plan for when the primary Region is restored. This is a common exam gap and a real operational risk.

A good failback process includes:

1. **Stabilize the DR Region**
   - Confirm the application is healthy and serving traffic.
   - Verify data consistency and backlog drain.
2. **Rebuild or repair the primary Region**
   - Restore infrastructure from IaC.
   - Recreate databases, queues, caches, and endpoints.
3. **Resynchronize data**
   - Replicate changes made during the DR period back to the primary Region.
   - Handle write conflicts if both Regions accepted writes.
4. **Test the primary Region**
   - Validate application health, integrations, and monitoring.
5. **Shift traffic back**
   - Update Route 53, load balancer targets, or global routing controls.
6. **Decommission temporary DR-only changes**
   - Remove emergency permissions, temporary scaling, and one-off scripts.

Important failback considerations:

- If the DR Region accepted writes, you need a controlled data reconciliation strategy.
- If the database is single-writer, you may need to reverse replication or restore from a fresh snapshot.
- If you use active/passive routing, failback can be simpler than in active/active designs.
- Always test failback separately from failover because the reverse path often exposes hidden dependencies.

**Queues, caches, and state during failover**

Stateful components often determine whether the DR design actually works.

- **Queues**
  - Pause producers during cutover if you need to avoid duplicate processing.
  - Drain consumers carefully to prevent message loss.
  - Use idempotent consumers so retries do not corrupt data.
- **Caches**
  - Treat cache loss as acceptable unless the workload explicitly requires warm cache continuity.
  - Rebuild cache entries after failover from the source of truth.
- **Sessions**
  - Prefer stateless application servers with tokens or centralized session storage.
  - If you must use server-side sessions, ensure the session store is replicated or globally accessible.
- **Search indexes**
  - Rebuild from primary data if the index is not the system of record.
  - Replicate if search freshness is part of the business requirement.

**Security, IAM, and KMS in DR**

Security controls must work in both Regions and, ideally, both accounts.

Key points:

- **IAM roles**
  - Pre-create roles in the DR account or Region.
  - Allow automation roles to perform restore and failover actions.
- **KMS keys**
  - Plan for destination keys in the DR Region.
  - Update key policies so backup copy and restore roles can use them.
  - Consider whether you need multi-Region KMS keys for simpler key management.
- **Secrets**
  - Replicate secrets securely and rotate them according to policy.
- **Network controls**
  - Mirror security groups, NACLs, and endpoint policies.
  - Ensure the DR environment can reach required AWS services and external dependencies.

> Encrypted backups are only useful if the DR environment can decrypt or re-encrypt them during restore.

**Practical DR runbook example**

A DR runbook should be explicit, testable, and automatable. A simple pattern for a web application might look like this:

1. Detect primary Region failure using CloudWatch alarms and Route 53 health checks.
2. Freeze writes if possible.
3. Promote the cross-Region database replica or restore the latest snapshot.
4. Deploy application stack in the DR Region using CloudFormation.
5. Recreate or attach queues, caches, and secrets.
6. Update Route 53 failover records or global routing.
7. Validate end-to-end user flows.
8. Monitor error rates, latency, and backlog.

Example CLI actions you might automate:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890 \
  --change-batch file://failover-change.json
```

```json
{
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "secondary",
        "Failover": "SECONDARY",
        "AliasTarget": {
          "HostedZoneId": "Z2P70J7EXAMPLE",
          "DNSName": "dualstack.dr-alb-123456.us-west-2.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

**Exam decision cues**

Use these cues to choose the best answer quickly:

- **Need automatic recovery within the same Region**: choose Multi-AZ.
- **Need regional disaster recovery with low cost**: choose backup and restore or pilot light.
- **Need faster regional recovery**: choose warm standby or cross-Region replica.
- **Need minimal data loss**: choose synchronous or near-synchronous replication where available.
- **Need blast-radius reduction**: use a separate DR account and cross-account copies.
- **Need repeatable recovery**: use IaC and Systems Manager automation.
- **Need DNS-based traffic shifting**: use Route 53 failover with health checks.

**Common pitfalls**

- Assuming Multi-AZ protects against Region failure.
- Forgetting to replicate KMS keys or plan for re-encryption.
- Restoring the database but not the queues, sessions, or secrets.
- Using a very low DNS TTL and expecting instant failover.
- Ignoring failback and only designing the failover path.
- Choosing active/active when the prompt only needs a simpler passive DR design.
- Overlooking cross-account isolation when the scenario mentions ransomware or compromised credentials.

**Summary**

A good DR design aligns recovery targets with the simplest workable AWS pattern. For SAA-C03, focus on the relationship between RTO, RPO, cost, and operational complexity. Then verify the full stack: data, compute, DNS, security, automation, and application dependencies. The best answer is usually the one that restores service reliably, minimizes unnecessary complexity, and includes a realistic failover and failback plan.
