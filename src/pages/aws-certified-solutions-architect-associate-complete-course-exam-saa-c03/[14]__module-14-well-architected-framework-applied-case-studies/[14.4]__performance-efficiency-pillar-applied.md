## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Performance Efficiency Pillar overview**

The **Performance Efficiency** pillar is about using AWS resources and architecture patterns to deliver the required performance at the lowest practical cost and operational effort. In exam scenarios, you usually improve performance by choosing the right service, scaling model, data layout, and network path rather than by simply “adding more instances.”

A strong answer often combines:
- **Right-sized compute** for the workload pattern
- **Managed services** that scale automatically
- **Caching** to reduce repeated work
- **Storage and database tuning** for the access pattern
- **Global delivery** choices that reduce latency for users

The exam frequently tests whether you can distinguish between:
- **Static vs dynamic content**
- **Read-heavy vs write-heavy databases**
- **Temporary vs durable storage**
- **Burst traffic vs steady traffic**
- **Regional vs global latency requirements**

**Core design principles**

Use the AWS Well-Architected mindset:
- **Democratize advanced technologies** by using managed services instead of building custom scaling logic.
- **Go global in minutes** with edge services and multi-Region patterns when latency or resilience requires it.
- **Use serverless architectures** where the workload is spiky or unpredictable.
- **Experiment more often** by measuring bottlenecks and iterating with metrics.

A practical exam approach is to ask:
1. What is the bottleneck: compute, storage, database, network, or application code?
2. Is the workload predictable or variable?
3. Is the data access pattern sequential, random, read-heavy, or write-heavy?
4. Is the user base local, multi-Region, or global?
5. Does the workload need low latency, high throughput, or both?

**Compute performance and scaling**

Choose compute based on how the application consumes CPU, memory, network, and I/O.

- **EC2** is best when you need full OS control, custom tuning, or specialized software.
- **Auto Scaling groups** help maintain performance under variable load by adding or removing instances.
- **Elastic Load Balancing (ELB)** distributes traffic across healthy targets and prevents hot spots.
- **Graviton-based instances** often provide better price/performance for many Linux workloads, but verify application compatibility.
- **Placement groups** can improve network performance for tightly coupled workloads.

Common placement group patterns:
- **Cluster placement group** for low-latency, high-throughput communication between instances in the same AZ.
- **Spread placement group** for critical instances that need isolation.
- **Partition placement group** for large distributed systems that need fault isolation across partitions.

For exam scenarios:
- Use **Auto Scaling** when traffic changes over time.
- Use **ELB** when you need horizontal scaling and health checks.
- Use **placement groups** when network latency between instances matters more than broad distribution.

Example Auto Scaling launch template snippet:
```bash
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name web-asg \
  --launch-template LaunchTemplateName=web-lt,Version='$Latest' \
  --min-size 2 \
  --max-size 10 \
  --desired-capacity 4 \
  --vpc-zone-identifier "subnet-aaa,subnet-bbb"
```

**Lambda, ECS, and EKS scaling patterns**

Serverless and container platforms are common exam topics because they reduce operational overhead while scaling efficiently.

- **AWS Lambda** scales automatically with incoming events, but concurrency and downstream limits still matter.
- **Amazon ECS** services can scale on CPU, memory, or custom CloudWatch metrics.
- **Amazon EKS** gives Kubernetes-native scaling, but you must manage node scaling and pod scheduling behavior.

Lambda event source scaling considerations:
- **SQS** event sources scale based on queue depth and polling behavior; batch size affects throughput and latency.
- **Kinesis Data Streams** scaling is constrained by shard count and per-shard throughput.
- **Amazon MSK** and other stream sources also require attention to consumer lag and partitioning.

Practical guidance:
- Use **Lambda** for bursty, event-driven workloads with short execution time.
- Use **ECS Fargate** when you want containerized workloads without managing servers.
- Use **EKS** when you need Kubernetes features, custom controllers, or portability.

Example ECS service autoscaling policy:
```bash
aws application-autoscaling register-scalable-target \
  --service-namespace ecs \
  --resource-id service/default/web \
  --scalable-dimension ecs:service:DesiredCount \
  --min-capacity 2 \
  --max-capacity 20
```

**Storage performance: EBS, instance store, and EFS**

Storage choice is often the difference between a correct and incorrect answer.

**Amazon EBS**
- **gp2** uses baseline IOPS tied to volume size and can burst; it is older and less flexible.
- **gp3** decouples performance from size, letting you provision IOPS and throughput independently.
- **io1** and **io2** are for provisioned IOPS workloads; **io2** offers higher durability and better performance consistency.
- **st1** is for throughput-oriented, sequential workloads.
- **sc1** is for infrequent access and low-cost archival-like block storage.

Exam tradeoffs:
- Choose **gp3** for general-purpose workloads that need predictable performance and cost control.
- Choose **io1/io2** for latency-sensitive databases or transactional systems needing high, consistent IOPS.
- Choose **instance store** for ultra-low-latency temporary storage, caches, buffers, or scratch space that can be lost on stop/terminate.

Example EBS tuning commands:
```bash
aws ec2 modify-volume \
  --volume-id vol-0123456789abcdef0 \
  --volume-type gp3 \
  --iops 6000 \
  --throughput 250
```

**Amazon EFS**
- **EFS Standard** is for multi-AZ durability and shared file access.
- **EFS One Zone** stores data in a single AZ and is cheaper, but less resilient.
- **Bursting Throughput** is suitable when file system size naturally drives throughput and traffic is spiky.
- **Provisioned Throughput** is appropriate when you need sustained throughput independent of stored data size.
- **Elastic Throughput** is useful when throughput demand varies and you want AWS-managed scaling without manual provisioning.

Use EFS when:
- Multiple EC2 instances or containers need shared POSIX file storage.
- You need elastic scaling without managing file servers.
- You want a managed file system for content repositories, home directories, or shared application assets.

**Amazon S3 performance and request optimization**

S3 is highly scalable, but performance still depends on request patterns and object design.

Important concepts:
- S3 automatically scales for high request rates, but you should distribute requests across prefixes when designing very high-throughput workloads.
- Use multiple prefixes when a workload generates extreme request concentration.
- **Multipart upload** improves performance for large objects and allows parallel upload of parts.
- Multipart upload is especially useful for objects larger than 100 MB and is required for very large uploads to improve reliability and throughput.
- **S3 Transfer Acceleration** uses CloudFront edge locations to speed uploads over long distances, especially for geographically distributed clients.

Practical guidance:
- Use **multipart upload** for large media files, backups, and data ingestion pipelines.
- Use **Transfer Acceleration** when clients are far from the bucket Region and upload latency is a problem.
- Use **S3 prefix design** to avoid hot partitions in very high request-rate workloads.

Example multipart upload with AWS CLI:
```bash
aws s3 cp large-video.mp4 s3://my-bucket/videos/ --storage-class STANDARD
```

Example Transfer Acceleration configuration:
```bash
aws s3api put-bucket-accelerate-configuration \
  --bucket my-bucket \
  --accelerate-configuration Status=Enabled
```

**Database performance: RDS, Aurora, DynamoDB**

Database questions often test whether you can improve performance without over-engineering.

**Amazon RDS and Aurora**
- Use **read replicas** to offload read traffic from the primary database.
- Use **Multi-AZ** for availability, not read scaling.
- Use **connection pooling** to reduce connection overhead and improve throughput.
- Use **query optimization** and indexing to reduce full scans and expensive joins.

Aurora-specific features:
- **Aurora Serverless v2** scales capacity in fine-grained increments and is useful for variable or unpredictable workloads.
- **Aurora Global Database** supports low-latency global reads and fast cross-Region disaster recovery.
- **Storage autoscaling** automatically expands storage as data grows.
- **Reader endpoints** load balance read traffic across Aurora replicas; under failover, the writer changes and the reader endpoint continues to direct reads to available replicas.

Exam scenarios:
- Use **Aurora Serverless v2** for spiky workloads that need fast scaling without managing instance sizes manually.
- Use **Aurora Global Database** when users are global and read latency matters across Regions.
- Use **reader endpoints** when you want read scaling and simple application configuration.
- Use **Multi-AZ** when you need failover protection, not read performance.

Example RDS read replica creation:
```bash
aws rds create-db-instance-read-replica \
  --db-instance-identifier mydb-replica \
  --source-db-instance-identifier mydb-primary
```

**Amazon DynamoDB**
- **On-demand capacity** is best for unpredictable traffic and rapid growth.
- **Provisioned capacity** is best when traffic is predictable and you want cost control.
- **Auto scaling** adjusts provisioned capacity based on utilization targets.
- **Write capacity units (WCU)** and **read capacity units (RCU)** determine throughput in provisioned mode.
- **Global secondary indexes (GSIs)** improve query flexibility but consume additional write capacity and storage.
- **Local secondary indexes (LSIs)** share the table partition key and are created at table creation time.

Capacity concepts:
- One **WCU** supports one write per second for an item up to 1 KB.
- One **RCU** supports one strongly consistent read per second for an item up to 4 KB, or two eventually consistent reads per second.
- Larger items consume more capacity units.
- Hot partitions reduce performance, so good partition key design matters.

When to choose each:
- Use **on-demand** for spiky, unknown, or rapidly changing workloads.
- Use **provisioned with auto scaling** for steady workloads with known patterns.
- Use **GSIs** when you need alternate query patterns, but account for write amplification.
- Use **DAX** if you need microsecond read latency for repeated reads and can tolerate cache complexity.

Example DynamoDB table creation with on-demand billing:
```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=OrderId,AttributeType=S \
  --key-schema AttributeName=OrderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

**Caching and cache invalidation**

Caching is one of the most common performance optimizations in AWS architectures.

**Amazon ElastiCache**
- **Redis** supports richer data structures, persistence options, pub/sub, and advanced features such as sorted sets and Lua scripting.
- **Memcached** is simpler, multi-threaded, and often used for straightforward key-value caching.
- Use **Redis** when you need replication, failover, data structures, or more advanced use cases.
- Use **Memcached** when you need a simple distributed cache and can tolerate node loss without replication semantics.

Cluster and replication concepts:
- **Replication groups** in Redis provide a primary node with read replicas for high availability and read scaling.
- **Cluster mode enabled** Redis partitions data across shards for horizontal scaling.
- **Cluster mode disabled** Redis keeps a single shard with replicas, which is simpler but less scalable.
- Memcached scales by adding nodes, but it does not provide Redis-style replication groups.

Exam guidance:
- Choose **Redis** for session storage, leaderboards, queues, rate limiting, and workloads needing persistence or failover.
- Choose **Memcached** for simple object caching where node replacement is acceptable.
- Choose **cluster mode enabled** when the dataset or throughput exceeds a single shard.
- Choose **replication groups** when you need HA and read scaling with Redis.

Cache invalidation patterns:
- **Cache-aside** is the most common pattern: the application checks cache first, then loads from the database on a miss.
- **Write-through** updates cache and database together for stronger consistency.
- **Write-behind** can improve write throughput but increases complexity and risk.
- Invalidation is hard; use TTLs (time to live) and versioned keys to reduce stale data.

Example Redis connection:
```bash
redis-cli -h my-redis.xxxxxx.0001.use1.cache.amazonaws.com -p 6379
```

**Networking and global performance**

Network design affects both latency and user experience.

**CloudFront**
- Best for **static content** and cacheable dynamic content.
- Reduces origin load and improves global latency by serving from edge locations.
- Supports TLS termination, compression, and origin shielding patterns.

**AWS Global Accelerator**
- Best for **TCP and UDP** workloads that need fast, stable global routing.
- Improves performance for non-HTTP protocols and dynamic applications that are not cache-friendly.
- Uses the AWS global network to route users to the nearest healthy endpoint.

**Route 53 latency-based routing**
- Best when you want DNS to direct users to the lowest-latency Region.
- Useful for multi-Region active-active or active-passive architectures.
- DNS changes are not instant, so it is less suitable than Global Accelerator for rapid failover behavior.

Common exam distinctions:
- Use **CloudFront** for static websites, media delivery, and cacheable APIs.
- Use **Global Accelerator** for gaming, VoIP, TCP/UDP applications, and fast failover.
- Use **Route 53 latency routing** when you need simple global DNS-based steering across Regions.

**Database and application tuning techniques**

Performance is not only about AWS services; application design matters too.

- Add the right **indexes** to support frequent query patterns.
- Avoid over-indexing because it slows writes and increases storage.
- Use **query plans** to identify full table scans and inefficient joins.
- Use **connection pooling** to reduce database connection churn.
- Cache expensive results when data freshness requirements allow it.
- Prefer pagination and selective columns instead of large unbounded result sets.

Practical examples:
- Replace `SELECT *` with only the columns you need.
- Use composite indexes for common multi-column filters.
- Move read-heavy reporting to replicas or analytics stores.
- Use asynchronous processing for non-critical work.

Example PostgreSQL connection pooling concept:
```ini
# pgbouncer.ini
[databases]
appdb = host=mydb.cluster-xxxx.us-east-1.rds.amazonaws.com port=5432 dbname=appdb

[pgbouncer]
pool_mode = transaction
max_client_conn = 1000
default_pool_size = 50
```

**Observability for performance troubleshooting**

You improve performance faster when you measure the right layer.

- **CloudWatch metrics** show resource-level signals such as CPU, memory, disk, network, latency, and queue depth.
- **CloudWatch dashboards** help you correlate metrics across services in one view.
- **CloudWatch alarms** alert you when thresholds indicate saturation or degradation.
- **AWS X-Ray** helps trace requests across distributed services and identify latency hotspots.
- **VPC Flow Logs** help diagnose network path issues, rejected traffic, and unexpected connectivity patterns.

What to look for:
- High CPU with low network and disk usage often indicates compute-bound workloads.
- High disk queue depth or latency suggests storage bottlenecks.
- Rising queue depth with stable compute suggests downstream saturation.
- High p95 or p99 latency with normal averages suggests tail latency or contention.

Example CloudWatch alarm creation:
```bash
aws cloudwatch put-metric-alarm \
  --alarm-name HighCPU-Web \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 300 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=AutoScalingGroupName,Value=web-asg \
  --evaluation-periods 2
```

**Exam scenario patterns**

Use these quick mappings when choosing an answer:
- **Need low-latency shared file storage** → EFS
- **Need temporary ultra-fast local storage** → instance store
- **Need predictable block performance** → gp3
- **Need very high consistent IOPS** → io2
- **Need global static content delivery** → CloudFront
- **Need global TCP/UDP acceleration** → Global Accelerator
- **Need DNS-based regional steering** → Route 53 latency routing
- **Need spiky database scaling** → Aurora Serverless v2 or DynamoDB on-demand
- **Need read scaling for relational DB** → read replicas or Aurora reader endpoints
- **Need simple cache** → Memcached
- **Need HA cache with advanced features** → Redis replication group

**Common pitfalls**

> Do not choose a service only because it is “faster” in general; choose it because it matches the access pattern, consistency needs, and scaling model.

- Using **Multi-AZ** as a read-scaling solution for RDS is incorrect.
- Using **CloudFront** for non-cacheable, low-latency TCP/UDP workloads is usually incorrect.
- Using **DynamoDB GSIs** without accounting for extra write capacity can create bottlenecks.
- Using **EBS gp2** when you need predictable performance may lead to burst-credit surprises.
- Using **EFS One Zone** for critical multi-AZ workloads can reduce resilience.
- Using **Memcached** when you need replication or persistence can create availability gaps.
- Using **on-demand DynamoDB** for stable, high-volume workloads may cost more than provisioned capacity with auto scaling.

**Practical decision checklist**

1. Identify the bottleneck and the access pattern.
2. Prefer managed scaling before custom scaling logic.
3. Match storage type to latency, durability, and throughput needs.
4. Use caching to reduce repeated reads and expensive computation.
5. Use the closest edge or Region when latency is the main problem.
6. Validate with metrics, traces, and load tests before changing architecture again.

**Quick reference examples**

- Static website with global users: S3 + CloudFront
- API with bursty traffic: API Gateway or ALB + Lambda or ECS autoscaling
- Shared application files: EFS Standard
- Temporary scratch space for batch jobs: instance store
- Transactional database with high IOPS: RDS on io2 or Aurora
- Variable NoSQL traffic: DynamoDB on-demand
- Global low-latency TCP app: Global Accelerator
- Read-heavy relational app: Aurora with reader endpoint and replicas
