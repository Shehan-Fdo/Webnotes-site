## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Route 53 overview**

Amazon Route 53 is AWS’s managed DNS service. You use it to register domains, host public or private hosted zones, and route traffic to endpoints using routing policies that fit availability, latency, geography, or traffic distribution goals.

Route 53 is a frequent exam topic because it connects DNS design with high availability. The key idea is that DNS answers are not just names-to-IPs; they are part of your failover and traffic management strategy.

- **Hosted zone**: a container for DNS records.
- **Record set**: a DNS entry such as `A`, `AAAA`, `CNAME`, `MX`, or `TXT`.
- **Alias record**: a Route 53-specific record that points to AWS resources and can be used at the zone apex.

**Core routing policy concepts**

A single record set uses one routing policy. You do not combine standard routing policies on the same record set. If you need multiple behaviors, you design multiple record sets or use Route 53 Traffic Flow policies.

Common standard routing policies include:

- **Simple routing**: one record or a small set of identical answers.
- **Weighted routing**: split traffic across records by configured weights.
- **Latency-based routing**: send users to the region with the lowest latency.
- **Failover routing**: active-passive routing with primary and secondary records.
- **Geolocation routing**: route based on user location.
- **Multivalue answer routing**: return multiple healthy records for simple client-side selection.

Traffic Flow adds policy-based routing features such as geoproximity and IP-based routing, but those are configured through traffic policy documents rather than standard hosted-zone routing policies.

**Simple routing**

Simple routing is the default mental model: one DNS name maps to one endpoint or a small set of identical endpoints. Route 53 returns the configured record without traffic splitting logic.

Use simple routing when:

- You have a single endpoint.
- You want all clients to receive the same answer.
- You are not doing active traffic management.

Example:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [
      {
        "Action": "UPSERT",
        "ResourceRecordSet": {
          "Name": "app.example.com",
          "Type": "A",
          "TTL": 300,
          "ResourceRecords": [
            { "Value": "203.0.113.10" }
          ]
        }
      }
    ]
  }'
```

**Weighted routing**

Weighted routing lets you assign a relative weight to each record set with the same name and type. Route 53 uses the weights as a probabilistic selection hint, but DNS caching and recursive resolver behavior mean you should not expect exact short-term proportions.

Use weighted routing for:

- Blue/green deployments.
- Canary releases.
- Gradual traffic shifting.
- A/B testing at the DNS layer.

Important behavior:

- Route 53 returns answers according to the configured weights over time, not as a strict per-request guarantee.
- If health checks are associated, Route 53 prefers healthy records.
- If all weighted records are unhealthy, Route 53 may still return records rather than returning nothing, depending on the record set and health-check context.

Example weighted record set:

```json
{
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "api.example.com",
        "Type": "A",
        "SetIdentifier": "blue",
        "Weight": 80,
        "TTL": 60,
        "ResourceRecords": [
          { "Value": "198.51.100.10" }
        ]
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "api.example.com",
        "Type": "A",
        "SetIdentifier": "green",
        "Weight": 20,
        "TTL": 60,
        "ResourceRecords": [
          { "Value": "198.51.100.20" }
        ]
      }
    }
  ]
}
```

**Latency-based routing**

Latency-based routing sends users to the AWS Region that Route 53 determines will provide the lowest latency. This is useful for globally distributed applications where you want to optimize user experience without manually managing geography.

Use latency-based routing when:

- You have regional deployments of the same application.
- You want lower response times for users around the world.
- You can serve the same content from multiple Regions.

Exam points:

- You create separate record sets for each Region.
- Route 53 chooses the Region with the best measured latency from the resolver’s perspective.
- Latency-based routing is a routing policy type, not a Traffic Flow-only feature.

**Geolocation routing**

Geolocation routing sends traffic based on the user’s location, such as continent, country, or in some cases subdivision. This is useful when content, legal requirements, language, or pricing differ by geography.

Route 53 selects the most specific match available:

- Subdivision match, if configured.
- Country match.
- Continent match.
- Default record, if no location-specific record matches.

The default record is the fallback for unmatched queries. In exam terms, you create a geolocation record without a specific country, continent, or subdivision target to catch all remaining traffic.

Use geolocation routing when:

- You need region-specific content.
- You must comply with data residency or licensing rules.
- You want to direct users to local websites or support portals.

Example scenario:

- Users in the United States go to `us.example.com`.
- Users in Europe go to `eu.example.com`.
- All other users go to a default global site.

**Failover routing**

Failover routing supports active-passive architectures. You define a primary record and a secondary record. Route 53 returns the primary record when it is healthy and the secondary record when the primary is unhealthy.

Use failover routing when:

- You want a standby site or region.
- You need a clear primary/backup design.
- You want DNS-level disaster recovery.

Practical behavior:

- Health checks are typically associated with the primary record, and often with the secondary as well.
- Route 53 uses health status to decide which record to answer.
- If both records are unhealthy, behavior depends on the exact configuration and record type; do not assume a guaranteed empty response in every case.

Example topology:

- Primary: `app-us-east-1.example.com`
- Secondary: `app-us-west-2.example.com`

**Multivalue answer routing**

Multivalue answer routing returns multiple healthy records in response to a single DNS query. It is not a load balancer replacement. It is best understood as a DNS-level way to provide several possible endpoints, with client-side selection or retry behavior.

Use multivalue answer routing when:

- You want simple resilience with multiple endpoints.
- You want clients to receive more than one answer.
- You need a lightweight alternative to a single-record design.

Important details:

- Route 53 returns up to eight healthy records per query.
- It does not guarantee that all healthy records are returned.
- Health checks influence which records are eligible.
- It is not intended to replace Elastic Load Balancing or application-aware load balancing.

**Routing policy comparison**

| Policy | Typical use | Answers returned | Health check behavior |
|---|---|---:|---|
| Simple | One endpoint | One answer | Optional, limited use |
| Weighted | Traffic splitting | One selected answer per query | Prefers healthy records when configured |
| Latency-based | Lowest-latency Region | One answer | Can be combined with health checks |
| Failover | Active-passive DR | Primary or secondary | Health determines active target |
| Geolocation | User location targeting | One answer | Can be health-aware |
| Multivalue answer | Multiple healthy endpoints | Up to 8 answers | Returns healthy records only when possible |

**Alias records and AWS targets**

Alias records are a major exam topic because they let you point DNS names to AWS-managed resources and avoid some limitations of standard DNS records.

Common alias targets include:

- Elastic Load Balancing load balancers.
- CloudFront distributions.
- S3 static website endpoints.
- API Gateway custom domains.
- VPC interface endpoints in some scenarios.
- Other Route 53 records in the same hosted zone.

Key benefits:

- You can use alias records at the zone apex, such as `example.com`.
- Route 53 does not charge for alias queries to AWS resources in the same way as standard DNS queries.
- You can often enable `EvaluateTargetHealth` for supported AWS targets.

Example alias record for an Application Load Balancer:

```json
{
  "Action": "UPSERT",
  "ResourceRecordSet": {
    "Name": "app.example.com",
    "Type": "A",
    "AliasTarget": {
      "HostedZoneId": "Z35SXDOTRQ7X7K",
      "DNSName": "dualstack.my-alb-123456.us-east-1.elb.amazonaws.com",
      "EvaluateTargetHealth": true
    }
  }
}
```

Practical exam guidance:

- Use alias records for AWS resources when possible.
- Use `EvaluateTargetHealth` only where Route 53 supports it for that target.
- CloudFront and ALB are common alias targets in architecture questions.
- S3 website endpoints use the website endpoint, not the S3 REST endpoint, when you need DNS aliasing for a static website.

**Health checks**

Route 53 health checks monitor endpoint availability and can influence DNS answers. They are useful for failover, weighted routing, and multivalue answer routing.

Health check types and behaviors:

- **Endpoint health checks**: probe a specific endpoint over HTTP, HTTPS, or TCP.
- **Calculated health checks**: combine the status of multiple health checks using logic such as `AND`/`OR`.
- **CloudWatch alarm health checks**: use a CloudWatch alarm as the health signal.

This matters for exam scenarios because Route 53 health checks are not limited to public web servers. You can also use calculated health checks and alarm-based health checks to represent composite or indirect health.

Common configuration options:

- **Protocol**: `HTTP`, `HTTPS`, or `TCP`.
- **Port**: the target port to probe.
- **Path**: for HTTP/HTTPS checks.
- **String matching**: verify that the response body contains expected text.
- **Request interval**: how often Route 53 checks the endpoint.
- **Failure threshold**: how many consecutive failures trigger unhealthy status.
- **Inverted health check**: flips healthy/unhealthy logic.
- **Disabled**: temporarily stops evaluation without deleting the check.

Example health check creation:

```bash
aws route53 create-health-check \
  --caller-reference "$(date +%s)" \
  --health-check-config '{
    "IPAddress": "198.51.100.10",
    "Port": 443,
    "Type": "HTTPS",
    "ResourcePath": "/health",
    "FullyQualifiedDomainName": "app.example.com",
    "RequestInterval": 30,
    "FailureThreshold": 3,
    "MeasureLatency": true
  }'
```

Use care with endpoint identity fields:

- `IPAddress` targets a specific IP.
- `FullyQualifiedDomainName` is used for the hostname Route 53 sends in the request.
- For HTTPS, the hostname can matter for TLS and virtual hosting behavior.

A practical HTTP health check example:

```bash
aws route53 create-health-check \
  --caller-reference "$(date +%s)" \
  --health-check-config '{
    "FullyQualifiedDomainName": "app.example.com",
    "Port": 80,
    "Type": "HTTP",
    "ResourcePath": "/status",
    "SearchString": "OK",
    "RequestInterval": 30,
    "FailureThreshold": 3
  }'
```

**Health check design patterns**

Use health checks to support DNS failover, but understand their limits.

- Route 53 health checkers are external probes, so they are best suited to public endpoints or endpoints reachable through a design that exposes them appropriately.
- For private-only systems, you often use calculated health checks, CloudWatch alarms, or a public health endpoint that reflects private service state.
- Health checks are not a replacement for application-level monitoring.

Common patterns:

- **Direct endpoint check**: probe the application URL.
- **Synthetic health endpoint**: expose `/health` or `/status`.
- **Composite health**: combine multiple checks into one calculated result.
- **Alarm-driven health**: route based on CloudWatch metrics or alarms.

**Alias health behavior with AWS targets**

Route 53 alias records can inherit health from supported AWS resources when `EvaluateTargetHealth` is enabled.

Typical exam-relevant cases:

- **ALB/NLB**: alias records commonly support `EvaluateTargetHealth`.
- **CloudFront**: alias records are common, but health evaluation behavior is limited by the service’s own availability model.
- **S3 website endpoints**: aliasing is used for static websites; health evaluation is not the same as for load balancers.
- **Other Route 53 records**: alias-to-alias chaining can be used within the same hosted zone.

Important distinction:

- `EvaluateTargetHealth` is not universally available for every alias target.
- You should know whether the target is AWS-managed and whether Route 53 can evaluate its health directly.

**Apex and routing policy interactions**

Apex records are the root of a zone, such as `example.com`. Standard DNS cannot use `CNAME` at the apex, so Route 53 alias records are commonly used there.

Exam scenarios often combine apex routing with routing policies:

- Apex alias to an ALB for a web application.
- Apex weighted alias records for blue/green deployments.
- Apex failover alias records for disaster recovery.
- Apex latency-based alias records for multi-Region apps.

Example apex failover design:

- `example.com` primary alias points to ALB in `us-east-1`.
- `example.com` secondary alias points to ALB in `us-west-2`.
- Health checks determine which region serves traffic.

**Traffic Flow distinction**

Traffic Flow is a Route 53 feature for building advanced traffic policies. It is not the same as standard hosted-zone routing policies.

Use Traffic Flow when you need:

- Geoproximity routing.
- IP-based routing.
- Complex policy combinations.
- Visual policy management.

Exam distinction:

- Standard hosted-zone routing policies are the common SAA-C03 focus.
- Traffic Flow features are configured through traffic policy records and policy documents.
- Do not confuse IP-based routing with a standard record-set routing policy.

**CLI examples**

Create a weighted record set:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch file://weighted-records.json
```

Create a failover record set:

```json
{
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "primary",
        "Failover": "PRIMARY",
        "TTL": 60,
        "ResourceRecords": [
          { "Value": "198.51.100.10" }
        ],
        "HealthCheckId": "abcd1234-ef56-7890-abcd-1234567890ab"

    {
      "Action": "UPSERT",

        "Name": "app.example.com",
        "Type": "A",

        "Failover": "SECONDARY",
        "TTL": 60,
```

**Exam scenarios to recognize**

- You need active-passive disaster recovery with DNS failover.
- You want to shift 10% of traffic to a new version.
- You need users in Europe to see EU-specific content.
- You need the same hostname at the zone apex pointing to an ALB.
- You need multiple healthy endpoints returned for client-side selection.
- You need the lowest-latency Region for global users.

**Troubleshooting and operational tips**

- Lower TTLs when you need faster DNS change propagation, but remember recursive resolvers may still cache answers.
- Use health checks with realistic endpoints, not just the home page.
- Verify that the record type and routing policy are compatible.
- Check whether the target supports aliasing and health evaluation.
- Remember that DNS-based routing is influenced by client resolvers, caches, and network paths.

**Quick memory cues**

- **Weighted**: split traffic.
- **Latency**: closest Region by latency.
- **Geolocation**: user location.
- **Failover**: primary then secondary.
- **Multivalue**: several healthy answers.
- **Alias**: AWS target or apex-friendly record.
