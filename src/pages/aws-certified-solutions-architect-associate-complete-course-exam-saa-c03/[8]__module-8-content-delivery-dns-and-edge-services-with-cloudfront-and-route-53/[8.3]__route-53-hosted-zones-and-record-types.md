## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Route 53 overview**

Amazon Route 53 is AWS’s managed DNS (Domain Name System) service. You use it to register domains, create hosted zones, and answer DNS queries for public internet traffic or private internal networks.

A hosted zone is the container for DNS records for a domain. In practice, you create records such as `A`, `AAAA`, `CNAME`, `MX`, `TXT`, and `NS` inside a hosted zone, then Route 53 answers queries based on those records and any routing policy you configure.

- **Public hosted zone**: answers DNS queries from the public internet.
- **Private hosted zone**: answers DNS queries only inside associated Amazon VPCs.
- **Record set**: a DNS record entry in a hosted zone, identified primarily by `Name` and `Type`, plus routing-policy-specific fields when applicable.

**Hosted zone types and when to use them**

Use a public hosted zone when you want internet users to resolve your domain, such as `www.example.com` or `api.example.com`. Use a private hosted zone when you want internal names like `db.internal.example.com` to resolve only from your AWS network.

A private hosted zone is associated with one or more VPCs. Queries resolve only when they originate from an associated VPC, or from a network path that ultimately uses Route 53 Resolver in a VPC that has access to that private zone.

- **Public zone**:
  - Internet-facing applications
  - Public website endpoints
  - External mail and verification records
- **Private zone**:
  - Internal service discovery
  - Database endpoints
  - Shared names for multi-account or multi-VPC environments

> A VPC can be associated with only one private hosted zone for a given domain name at a time. Overlapping public and private zones can coexist, but resolution depends on where the query originates.

**Hosted zone delegation and association**

Public DNS delegation uses `NS` records in the parent zone. If you delegate `dev.example.com` to a separate hosted zone, you publish the child zone’s name servers in the parent public zone.

Private hosted zones do not use public DNS delegation in the same way. Instead, you associate the private hosted zone with VPCs.

1. Create the child hosted zone.
2. Copy the child zone’s `NS` values if you need public delegation.
3. Add `NS` records in the parent public zone for delegation.
4. Associate private zones directly with the required VPCs.

```bash
aws route53 create-hosted-zone \
  --name dev.example.com \
  --caller-reference "$(date +%s)" \
  --hosted-zone-config Comment="Public child zone",PrivateZone=false
```

```bash
aws route53 associate-vpc-with-hosted-zone \
  --hosted-zone-id Z1234567890ABC \
  --vpc VPCRegion=us-east-1,VPCId=vpc-0123456789abcdef0
```

**Record set identity and creation rules**

Route 53 record sets are not identified by a single universal key. The practical identity is the combination of:

- `Name`
- `Type`
- routing-policy-specific fields when needed, such as `SetIdentifier`
- additional policy fields for some routing types

This matters because multiple records can share the same name and type only when the routing policy allows it and the required distinguishing fields are present.

Common constraints:

- `CNAME` cannot coexist with other record types at the same exact name.
- Apex records at the zone root cannot use `CNAME`.
- Some routing policies require health checks or additional attributes.
- Some same-name combinations are valid, especially with alias records and delegated subdomains.

**Record types you should know**

Route 53 supports standard DNS record types and several AWS-specific behaviors. For the exam, focus on how each type behaves and where it is allowed.

- **A**: maps a name to an IPv4 address.
- **AAAA**: maps a name to an IPv6 address.
- **CNAME**: maps one DNS name to another DNS name.
- **MX**: mail exchange record for email routing.
- **TXT**: arbitrary text, often used for verification and SPF.
- **NS**: name server delegation.
- **SOA**: start of authority, automatically created for hosted zones.
- **PTR**: pointer record, used for reverse DNS.
- **SRV**: service locator record.
- **CAA**: certificate authority authorization.
- **SPF**: historically used as a separate record type in some systems, but in Route 53 you typically publish SPF data in `TXT` records because SPF as a standalone DNS record type is deprecated in practice.

**A and AAAA records**

Use `A` for IPv4 and `AAAA` for IPv6. These are the most common records for application endpoints.

For simple routing, a single record can return one or more values. Route 53 may return multiple IP addresses, but you should not assume strict round-robin behavior or client-side load balancing consistency.

```json
{
  "Name": "app.example.com",
  "Type": "A",
  "TTL": 60,
  "ResourceRecords": [
    { "Value": "203.0.113.10" },
    { "Value": "203.0.113.11" }
  ]
}
```

Use multiple values when you want a simple DNS answer set and your application can tolerate client-side selection or failover logic outside DNS.

**CNAME versus alias**

A `CNAME` record points one DNS name to another DNS name. It is the correct choice for most non-AWS targets when you need name-to-name mapping.

An alias record is a Route 53 feature that points to supported AWS resources, and in some cases to another record in the same hosted zone. Alias is not a general replacement for `CNAME`.

Key differences:

- **CNAME**
  - Works for arbitrary DNS names
  - Cannot exist at the zone apex
  - Adds an extra DNS lookup
  - Has a user-configurable TTL
- **Alias**
  - Works for supported AWS targets such as Application Load Balancer, Network Load Balancer, CloudFront distribution, API Gateway custom domain, S3 static website endpoint, and some Route 53 records
  - Can be used at the zone apex
  - Does not have a user-configurable TTL
  - Route 53 returns a TTL based on the target/resource behavior

> For S3, alias records apply to the **static website endpoint**, not the standard S3 REST endpoint.

Example alias record for an ALB:

```json
{
  "Name": "app.example.com",
  "Type": "A",
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "dualstack.my-alb-123456.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

**Apex/root record behavior**

The zone apex is the root of the domain, such as `example.com`. You cannot create a `CNAME` at the apex because DNS rules require the apex to host other records like `NS` and `SOA`.

Common apex-safe record types:

- `A`
- `AAAA`
- `MX`
- `TXT`
- `NS`
- `SOA`
- `CAA`
- alias `A` or `AAAA`

Typical exam trap:

- `www.example.com` can be a `CNAME` to a non-AWS name.
- `example.com` should usually be an alias `A` or `AAAA` if you want to point to an AWS resource.
- `example.com` can also host `MX` and `TXT` records for mail and verification.

**MX, TXT, and PTR records**

`MX` records direct email delivery to mail servers. They include a priority value, where lower numbers indicate higher preference.

```txt
example.com.  10 mail1.example.com.
example.com.  20 mail2.example.com.
```

`TXT` records store arbitrary text. Common uses include:

- domain ownership verification
- SPF data
- DKIM-related values
- DMARC policy records

Example SPF-in-TXT:

```txt
example.com.  "v=spf1 include:_spf.google.com -all"
```

`PTR` records support reverse DNS lookup, mapping an IP address back to a hostname. They are usually managed in reverse lookup zones and are less common in day-to-day application design, but they can appear in exam questions about email reputation or reverse resolution.

**Routing policies**

Route 53 routing policies control how DNS answers are selected. The policy affects how you create the record set and what extra fields are required.

- **Simple routing**
  - Returns one or more values for a record
  - Good for a single endpoint or basic multi-value answers
  - Does not guarantee strict round-robin behavior
- **Weighted routing**
  - Splits traffic by percentage
  - Useful for blue/green deployments and canary releases
- **Latency-based routing**
  - Returns the record with the lowest latency from the user’s region
  - Useful for global applications
- **Failover routing**
  - Returns primary or secondary based on health check status
  - Useful for disaster recovery
- **Geolocation routing**
  - Routes based on user location
  - Useful for compliance, localization, or regional content control
- **Geoproximity routing**
  - Available with traffic flow and advanced use cases
  - Routes based on geographic distance and bias
- **Multi-value answer routing**
  - Returns multiple healthy records
  - Useful for simple client-side resilience

**Weighted routing example**

Weighted routing uses multiple records with the same name and type, each with a different `SetIdentifier` and weight.

```json
{
  "Name": "api.example.com",
  "Type": "A",
  "SetIdentifier": "blue",
  "Weight": 80,
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "blue-alb-123.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

```json
{
  "Name": "api.example.com",
  "Type": "A",
  "SetIdentifier": "green",
  "Weight": 20,
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "green-alb-456.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

Use weighted routing when you want to gradually shift traffic or test a new version with a small percentage of users.

**Latency-based routing example**

Latency-based routing requires separate records for each region, each with its own `SetIdentifier` and region value.

```json
{
  "Name": "app.example.com",
  "Type": "A",
  "SetIdentifier": "us-east-1",
  "Region": "us-east-1",
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "app-use1-alb-123.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

```json
{
  "Name": "app.example.com",
  "Type": "A",
  "SetIdentifier": "eu-west-1",
  "Region": "eu-west-1",
  "AliasTarget": {
    "HostedZoneId": "Z32O12XQLNTSW2",
    "DNSName": "app-euw1-alb-456.eu-west-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

Use latency routing when you have active endpoints in multiple regions and want users directed to the region with the best network performance.

**Failover routing and health checks**

Failover routing uses health checks to decide whether Route 53 should return the primary or secondary record. Health checks can monitor endpoint availability, HTTP/HTTPS status, TCP connectivity, or calculated health from other checks.

Important points:

- **Primary** record answers when healthy.
- **Secondary** record answers when the primary fails.
- `EvaluateTargetHealth` can be used with some alias targets, especially load balancers, to incorporate target health.
- Health checks are not the same as application-level autoscaling; they only influence DNS answer selection.

Example failover records:

```json
{
  "Name": "shop.example.com",
  "Type": "A",
  "SetIdentifier": "primary",
  "Failover": "PRIMARY",
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "primary-alb-123.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

```json
{
  "Name": "shop.example.com",
  "Type": "A",
  "SetIdentifier": "secondary",
  "Failover": "SECONDARY",
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "secondary-alb-456.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

**Geolocation routing**

Geolocation routing returns answers based on the user’s country, continent, or subdivision. It is useful when content, legal requirements, or language support differ by location.

Geolocation records require:

- `SetIdentifier`
- one of `ContinentCode`, `CountryCode`, or `SubdivisionCode`
- a default record if you want a fallback for unmatched locations

Example:

```json
{
  "Name": "content.example.com",
  "Type": "A",
  "SetIdentifier": "us",
  "CountryCode": "US",
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "us-content-alb-123.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

```json
{
  "Name": "content.example.com",
  "Type": "A",
  "SetIdentifier": "eu",
  "ContinentCode": "EU",
  "AliasTarget": {
    "HostedZoneId": "Z32O12XQLNTSW2",
    "DNSName": "eu-content-alb-456.eu-west-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

Use geolocation when you need deterministic location-based responses rather than latency-based optimization.

**Multi-value answer routing**

Multi-value answer routing returns multiple healthy records for the same name. It is often used with simple endpoints where you want DNS to provide a small pool of healthy targets.

- It is not a full load balancer.
- It can improve resilience by returning multiple healthy values.
- It works best when clients can retry or choose among returned addresses.

Example:

```json
{
  "Name": "db-read.example.com",
  "Type": "A",
  "SetIdentifier": "node-1",
  "MultiValueAnswer": true,
  "ResourceRecords": [{ "Value": "10.0.1.10" }]
}
```

```json
{
  "Name": "db-read.example.com",
  "Type": "A",
  "SetIdentifier": "node-2",
  "MultiValueAnswer": true,
  "ResourceRecords": [{ "Value": "10.0.2.10" }]
}
```

**Private hosted zone details and exam traps**

Private hosted zones are tied to VPC association. They are not globally visible, and they do not behave like public DNS delegation.

Key operational details:

- A private hosted zone can be associated with multiple VPCs.
- A VPC can be associated with only one private hosted zone for a given domain name at a time.
- If a public and private hosted zone share the same domain name, queries from associated VPCs resolve to the private zone, while public internet queries resolve to the public zone.
- Cross-region association is allowed, but you must specify the correct VPC region when associating the VPC.

Example private zone creation:

```bash
aws route53 create-hosted-zone \
  --name internal.example.com \
  --caller-reference "$(date +%s)" \
  --hosted-zone-config Comment="Private zone for internal services",PrivateZone=true \
  --vpc VPCRegion=us-east-1,VPCId=vpc-0123456789abcdef0
```

**Health checks and alias targets**

Health checks are central to failover and can also influence other routing policies. For alias records, `EvaluateTargetHealth` tells Route 53 to consider the target’s health when deciding whether to answer with that alias.

Use it carefully:

- With load balancers, it is often useful.
- With CloudFront, health evaluation is not the same as checking origin health directly.
- With some AWS targets, Route 53 manages the target behavior and health integration differently than with standard records.

Practical rule:

- Use explicit health checks when you need Route 53 to probe an endpoint.
- Use `EvaluateTargetHealth` when the alias target supports health evaluation and you want Route 53 to respect target health.

**Common exam traps**

- `CNAME` at the apex is invalid.
- Alias is not a generic DNS feature for arbitrary external names.
- Alias records do not have a user-set TTL.
- `SPF` is usually stored in `TXT`, not as a standalone record in modern designs.
- Weighted, latency, failover, and geolocation records often need `SetIdentifier`, but geolocation also needs location fields.
- Public delegation uses `NS` records in the parent zone; private hosted zones use VPC association instead.
- A private hosted zone is not reachable from the internet.
- S3 alias support applies to the website endpoint, not the REST endpoint.
- Route 53 simple routing does not guarantee strict round-robin distribution.

**CLI examples for record management**

Create a simple `A` record:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "www.example.com",
        "Type": "A",
        "TTL": 300,
        "ResourceRecords": [
          { "Value": "198.51.100.10" }
        ]
      }
    }]
  }'
```

Create an alias `A` record to an ALB:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "my-alb-123.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }]
  }'
```

Create a `TXT` record for verification:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "_acme-challenge.example.com",
        "Type": "TXT",
        "TTL": 300,
        "ResourceRecords": [
          { "Value": "\"verification-token-123\"" }
        ]
      }
    }]
  }'
```

**Practical design patterns**

For a public web app, use an apex alias `A` record to an ALB and a `www` `CNAME` or alias depending on the target. For internal service discovery, use a private hosted zone with `A` or `CNAME` records that resolve only inside the VPCs that need them.

For global applications:

- Use latency-based routing for active-active regional deployments.
- Use failover routing for active-passive disaster recovery.
- Use weighted routing for controlled traffic shifting.
- Use geolocation routing when location-specific content or compliance matters.

**Quick comparison**

| Topic | Best choice | Notes |
|---|---|---|
| Apex to ALB | Alias `A`/`AAAA` | No user TTL |
| Non-AWS hostname | `CNAME` | Not at apex |
| Email routing | `MX` + `TXT` | SPF usually in `TXT` |
| Reverse DNS | `PTR` | Reverse lookup zones |
| Internal service name | Private hosted zone | VPC association required |
| Traffic shifting | Weighted | Use `SetIdentifier` |
| Regional performance | Latency-based | One record per region |
| DR failover | Failover + health checks | Primary/secondary |

**Study focus for SAA-C03**

For the exam, you should be able to choose the correct record type and hosted zone type for a scenario. The most common decisions are:

- public versus private hosted zone
- `CNAME` versus alias
- apex versus non-apex naming
- routing policy selection
- health check usage
- delegation versus VPC association

If you can explain why a record is allowed, where it resolves, and how Route 53 selects the answer, you are ready for most Route 53 questions.
