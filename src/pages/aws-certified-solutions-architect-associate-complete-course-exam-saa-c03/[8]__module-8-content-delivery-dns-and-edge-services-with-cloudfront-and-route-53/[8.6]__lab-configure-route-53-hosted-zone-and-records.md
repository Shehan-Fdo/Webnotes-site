## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Route 53 fundamentals**

Amazon Route 53 is AWS’s managed DNS (Domain Name System) service. You use it to register domains, create hosted zones, and publish record sets that route traffic to AWS resources or external endpoints.

A hosted zone is the container for DNS records for a domain. In practice, you create a hosted zone for a domain such as `example.com`, then add records like `A`, `AAAA`, `CNAME`, `MX`, `TXT`, and Route 53-specific alias records.

- **Public hosted zone**: answers DNS queries from the internet.
- **Private hosted zone**: answers DNS queries only inside one or more associated VPCs (Virtual Private Clouds).
- **Record set**: a DNS entry inside a hosted zone, such as `www.example.com -> 203.0.113.10`.
- **TTL**: Time To Live, the cache duration for DNS responses.

Route 53 is often tested in the exam because it combines DNS, routing policy, health checks, and AWS integration. You should know when to use standard DNS records versus Route 53 alias records, and how routing policies affect failover and load distribution.

**Hosted zones and delegation**

When you create a hosted zone, Route 53 automatically creates the authoritative `NS` (name server) and `SOA` (start of authority) records for that zone. These are required for DNS delegation and zone management.

If you are moving a domain to Route 53, you typically:

1. Create a hosted zone for the domain.
2. Copy the four assigned name servers from the hosted zone.
3. Update the domain registrar to delegate the domain to those name servers.
4. Wait for DNS propagation.

Example AWS CLI command to create a public hosted zone:

```bash
aws route53 create-hosted-zone \
  --name example.com \
  --caller-reference "$(date +%s)" \
  --hosted-zone-config Comment="Public zone for example.com",PrivateZone=false
```

The response includes the hosted zone metadata and a change info object. The `NS` and `SOA` records are created automatically in the zone, but they are not returned as separate record sets in the create response.

For a private hosted zone, you must associate it with one or more VPCs:

```bash
aws route53 create-hosted-zone \
  --name internal.example.com \
  --caller-reference "$(date +%s)" \
  --hosted-zone-config Comment="Private zone",PrivateZone=true \
  --vpc VPCRegion=us-east-1,VPCId=vpc-0123456789abcdef0
```

Private hosted zones are useful when you want internal DNS names such as `db.internal.example.com` to resolve only inside your AWS network.

**Record types you should know**

Route 53 supports standard DNS record types and several AWS-specific behaviors.

- **A record**: maps a name to an IPv4 address.
- **AAAA record**: maps a name to an IPv6 address.
- **CNAME record**: maps a name to another DNS name.
- **MX record**: mail exchange routing.
- **TXT record**: text values, often for verification and SPF/DMARC-related use cases.
- **NS record**: delegates a subdomain or identifies authoritative name servers.
- **SOA record**: zone authority metadata.

Important exam distinction:

- A `CNAME` cannot be used at the zone apex, such as `example.com`.
- Route 53 alias records can be used at the zone apex, but they are Route 53-specific and are not a general DNS replacement for `CNAME`.
- Alias records point to supported AWS resources or to certain Route 53 record targets, and Route 53 resolves them internally.

**Alias records versus standard DNS records**

Alias records are one of the most tested Route 53 features. They let you map a DNS name to an AWS resource without using a traditional `CNAME`.

Use alias records when you want:

- Apex support for `example.com`.
- No extra DNS lookup for the target in many cases.
- Integration with AWS resource health evaluation.
- Cleaner management for AWS endpoints.

Common alias targets include:

- Application Load Balancer (ALB)
- Network Load Balancer (NLB)
- CloudFront distribution
- API Gateway custom domain
- S3 static website endpoint
- Another Route 53 record in the same hosted zone

> Alias records are Route 53-specific. Do not treat them as a universal DNS record type or as a drop-in replacement for `CNAME` in every situation.

A practical comparison:

| Feature | CNAME | Route 53 alias |
|---|---|---|
| Can use at zone apex | No | Yes |
| Stores a DNS name directly | Yes | No, Route 53 resolves the target internally |
| Works for AWS resources | Not directly | Yes, for supported targets |
| Works for non-apex names | Yes | Yes |
| Route 53-specific | No | Yes |

Example alias record for an ALB in the console or API typically uses the load balancer DNS name and the AWS-provided hosted zone ID for that target type. The hosted zone ID is service-specific, so do not assume one value works for every AWS resource.

Example CLI change batch for an alias `A` record:

```json
{
  "Comment": "Alias app.example.com to ALB",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "AliasTarget": {
          "HostedZoneId": "ZXXXXXXXXXXXXX",
          "DNSName": "dualstack.my-alb-123456.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

**S3 static website endpoints**

Route 53 alias records can point to an S3 static website endpoint, but not to the S3 REST endpoint. The bucket must be configured for static website hosting, and the endpoint is regional.

Use this pattern when you want a friendly domain like `www.example.com` to serve a static site from S3.

Typical setup:

- Enable static website hosting on the bucket.
- Configure the bucket policy for public read access if the site is public.
- Create an alias record in Route 53 that points to the S3 website endpoint.
- Use the website endpoint, not the REST endpoint.

Example website endpoint pattern:

```text
http://example-bucket.s3-website-us-east-1.amazonaws.com
```

If you need HTTPS for a custom domain, remember that S3 static website endpoints do not provide HTTPS directly through the website endpoint itself. In practice, you often place CloudFront in front of S3 for HTTPS and caching.

**Routing policies**

Route 53 routing policies control how DNS answers are selected. The exam often asks you to choose the right policy for availability, latency, or traffic distribution.

- **Simple routing**: returns one record set for a name.
- **Weighted routing**: splits traffic by configured weights.
- **Latency-based routing**: returns the record with the lowest latency to the user.
- **Failover routing**: returns a primary record unless it is unhealthy, then returns the secondary.
- **Geolocation routing**: routes based on user location.
- **Geoproximity routing**: routes based on geographic proximity, with bias control.
- **Multivalue answer routing**: returns multiple healthy records for a name.

Weighted routing is useful for blue/green deployments, canary releases, and gradual traffic shifts. Health checks are optional for weighted records, but if you want Route 53 to stop returning an unhealthy endpoint, you should attach health evaluation.

Failover routing depends on health evaluation to work as intended. Route 53 uses health checks or alias target health to determine whether the primary is healthy. If the primary is unhealthy, Route 53 returns the secondary. If you do not configure health evaluation appropriately, failover behavior may not match your expectation.

Example weighted record idea:

- `app.example.com` -> ALB blue environment, weight `90`
- `app.example.com` -> ALB green environment, weight `10`

This lets you shift a small percentage of traffic to the new version before a full cutover.

**Health checks**

Route 53 health checks monitor endpoints and can influence routing decisions. They are commonly used with failover routing and multivalue answer routing, and they can also be used with weighted records when you want unhealthy targets removed from answers.

Health checks can monitor:

- HTTP or HTTPS endpoints
- TCP endpoints
- CloudWatch alarms
- Calculated health checks based on other health checks

A health check is not the same as an application health endpoint, but it often probes one. For example, you might expose `/health` on an ALB-backed service and have Route 53 check that path.

Example health check concept:

```json
{
  "CallerReference": "healthcheck-001",
  "HealthCheckConfig": {
    "IPAddress": "203.0.113.10",
    "Port": 443,
    "Type": "HTTPS",
    "ResourcePath": "/health",
    "FullyQualifiedDomainName": "app.example.com",
    "RequestInterval": 30,
    "FailureThreshold": 3
  }
}
```

Use health checks when you need DNS-level failover or when you want Route 53 to stop returning unhealthy endpoints. Do not assume weighted routing requires health checks; it does not.

**Failover design patterns**

Failover routing is a common high-availability pattern. You create a primary record and a secondary record, then attach health evaluation to the primary, secondary, or both depending on the design.

Typical pattern:

- Primary: ALB in `us-east-1`
- Secondary: ALB in `us-west-2`
- Health check: monitors the primary application endpoint

When the primary becomes unhealthy, Route 53 returns the secondary record. This is useful for disaster recovery and regional resilience.

Practical considerations:

- Make sure the health check reflects real service availability.
- Ensure the secondary environment is ready to serve traffic.
- Test DNS failover behavior before relying on it in production.
- Remember that DNS caching can delay client switchover even after Route 53 changes the answer.

**Private hosted zone use cases**

Private hosted zones are ideal for internal service discovery and split-horizon DNS patterns.

Examples:

- `api.internal.example.com` resolves to a private ALB.
- `db.internal.example.com` resolves to a private IP address.
- `service.dev.example.com` resolves differently inside a VPC than outside.

You can associate a private hosted zone with multiple VPCs, including VPCs in different accounts when permissions allow it. This is useful for shared services and multi-account architectures.

**CLI workflow for record creation**

A common workflow is to create a hosted zone, then add records using `change-resource-record-sets`.

Example `A` record for a static IP:

```json
{
  "Comment": "Create A record for web.example.com",
  "Changes": [
    {
      "Action": "CREATE",
      "ResourceRecordSet": {
        "Name": "web.example.com",
        "Type": "A",
        "TTL": 300,
        "ResourceRecords": [
          { "Value": "198.51.100.25" }
        ]
      }
    }
  ]
}
```

Apply it with:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch file://change-batch.json
```

Example `CNAME` record:

```json
{
  "Comment": "Create CNAME for www.example.com",
  "Changes": [
    {
      "Action": "CREATE",
      "ResourceRecordSet": {
        "Name": "www.example.com",
        "Type": "CNAME",
        "TTL": 300,
        "ResourceRecords": [
          { "Value": "app.example.net" }
        ]
      }
    }
  ]
}
```

Remember that `CNAME` is valid for subdomains like `www.example.com`, but not for the apex `example.com`.

**Troubleshooting and exam traps**

Route 53 questions often hide small but important details.

- **Apex limitation**: `CNAME` cannot be used at the root of a zone.
- **Alias target support**: alias records work only for supported AWS targets and certain Route 53 record targets.
- **S3 website vs REST endpoint**: alias to the website endpoint, not the REST endpoint, for static websites.
- **Health evaluation**: failover depends on health checks or alias target health.
- **TTL and caching**: DNS changes are not instant for clients that cache responses.
- **Hosted zone type**: private zones are only resolvable inside associated VPCs.

A useful mental model is:

- Use standard DNS records when you need generic DNS behavior.
- Use alias records when you need AWS integration or apex support.
- Use routing policies when you need intelligent answer selection.
- Use health checks when you need Route 53 to react to endpoint failure.

**Practical lab checklist**

When you configure Route 53 in a lab, verify each layer in order.

1. Create the hosted zone.
2. Confirm the zone is public or private as intended.
3. Copy the assigned `NS` records if you are delegating a public domain.
4. Add the required record sets.
5. Test resolution with `dig` or `nslookup`.
6. Validate failover or alias behavior from a client perspective.

Example DNS test commands:

```bash
dig example.com NS
dig www.example.com A
nslookup app.example.com
```

If you are testing a private hosted zone, run the lookup from an EC2 instance or other resource inside the associated VPC.

**Key exam takeaways**

- Route 53 hosted zones store DNS records for a domain.
- Public zones are internet-facing; private zones are VPC-scoped.
- `NS` and `SOA` records are created automatically for each hosted zone.
- `CNAME` cannot be used at the apex.
- Alias records are Route 53-specific and support selected AWS targets.
- Alias records can be used at the apex and are often preferred for AWS endpoints.
- Failover routing requires health evaluation to behave correctly.
- Weighted routing is useful for traffic shifting and does not inherently require health checks.
- S3 static website aliasing uses the website endpoint, not the REST endpoint.
