## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**CloudFront caching and invalidation overview**

Amazon CloudFront is a <abbr title="Content Delivery Network">CDN</abbr> that caches content at edge locations to reduce latency and offload origin traffic. For exam purposes, the key idea is that CloudFront serves cached objects until they expire or are explicitly invalidated.

CloudFront cache behavior is driven by:
- **TTL** values from the origin response headers or cache policy
- **Path-based cache keys** and query string/header/cookie forwarding rules
- **Invalidations** when you need to remove cached objects before natural expiration

A common exam scenario is a static website or API response that changes before the cache expires. You must choose between:
- **Invalidation** for immediate removal of old content
- **Versioned filenames** for predictable deployments and lower operational cost

**How CloudFront invalidation works**

An invalidation tells CloudFront to remove cached copies of matching objects from edge locations. It does not delete the object from the origin. The next request after invalidation fetches a fresh copy from the origin and repopulates the cache.

Typical behavior:
- You create an invalidation for one or more paths such as `/index.html` or `/*`
- CloudFront marks the invalidation as **InProgress**
- Edge locations purge matching cached objects asynchronously
- The invalidation becomes **Completed** when propagation finishes

You can verify completion in:
- The **CloudFront console** under the distribution’s invalidations list
- The **AWS CLI** using `get-invalidation`

Example CLI workflow:

```bash
aws cloudfront create-invalidation \
  --distribution-id E123ABC456DEF \
  --paths "/index.html" "/assets/app.js"

aws cloudfront get-invalidation \
  --distribution-id E123ABC456DEF \
  --id I1234567890ABC
```

The response includes the invalidation status. In exam questions, if the status is still `InProgress`, CloudFront may still serve the old object from some edge locations until propagation completes.

**When to use invalidation versus versioned filenames**

Use invalidation when you need to replace an object at the same URL and cannot change the path. Use versioned filenames when you control the deployment process and want fewer invalidations.

A practical deployment example:
- You deploy `app.js` and `index.html`
- `app.js` is fingerprinted as `app.8f3a1c.js`
- `index.html` references the new fingerprinted file
- You invalidate only `/index.html`
- Users get the new HTML immediately, and the new asset path naturally bypasses the old cache entry

This pattern is usually better than invalidating every asset because:
- It reduces invalidation volume
- It avoids waiting for broad cache purge propagation
- It lowers the chance of serving mixed old/new assets

> If you update a file in place, CloudFront may continue serving the old version until the TTL expires unless you invalidate the path.

**Invalidation pricing and free-tier allowance**

CloudFront invalidation has a small free allowance that is commonly tested in exam prep. The important exam takeaway is:
- The first set of invalidation paths each month is free up to the service’s free-tier allowance
- Additional invalidation paths are billed per path

A frequently tested detail is that invalidating `/*` counts as a single path, but it can be operationally expensive in terms of cache efficiency because it flushes everything. Use it only when necessary.

Practical implications:
- Prefer targeted invalidations like `/index.html`
- Use `/*` only for emergency rollbacks or major site-wide changes
- For frequent releases, versioned filenames are usually cheaper and cleaner

**Propagation timeline example**

Imagine this sequence:
1. At 10:00, you upload a new `index.html` to S3.
2. At 10:01, CloudFront still serves the cached old page.
3. At 10:02, you create an invalidation for `/index.html`.
4. At 10:03, the invalidation shows `InProgress`.
5. At 10:05, some edge locations have purged the object, others are still finishing.
6. At 10:07, the invalidation is `Completed`, and new requests fetch the updated file.

For exam questions, the key point is that invalidation is asynchronous. It is not an instant global switch, even though it is usually fast.

**CloudFront invalidation API and console behavior**

You can create invalidations from:
- The **CloudFront console**
- The **AWS CLI**
- The **CloudFront API**

Console behavior:
- You enter one or more object paths
- CloudFront shows the invalidation request in a list
- Status changes from `InProgress` to `Completed`
- You can refresh the list or open the invalidation details to confirm completion

CLI behavior:
- `create-invalidation` returns an invalidation ID
- `get-invalidation` checks the current status
- `list-invalidations` shows recent requests

Example:

```bash
aws cloudfront list-invalidations --distribution-id E123ABC456DEF
```

**Viewer-side TLS and SSL/TLS security policies**

CloudFront supports HTTPS for viewer connections, meaning the browser or client connects to CloudFront over TLS. The certificate presented to the viewer must be in <abbr title="AWS Certificate Manager">ACM</abbr> in the `us-east-1` region for CloudFront distributions.

Important exam concepts:
- CloudFront terminates TLS at the edge for viewer connections
- You choose a **security policy** that controls supported TLS versions and ciphers
- Modern policies support TLS 1.2, and newer policies may support TLS 1.3 for viewers
- Older policies may allow legacy TLS versions, but they are less secure and less desirable

A practical rule:
- Use the newest security policy your clients can support
- Prefer TLS 1.2 or TLS 1.3 for modern applications
- Avoid legacy policies unless you must support old clients

Example viewer connection settings:
- Viewer protocol policy: `Redirect HTTP to HTTPS`
- Security policy: a modern TLS policy such as one that supports TLS 1.2 and TLS 1.3
- Certificate: ACM certificate in `us-east-1`

**SNI versus dedicated IP custom SSL certificates**

CloudFront supports two main ways to associate a custom certificate with a distribution:
- **SNI** (<abbr title="Server Name Indication">SNI</abbr>) custom SSL
- **Dedicated IP** custom SSL

SNI is the standard choice:
- Multiple distributions can share the same IP addresses
- The client sends the hostname during the TLS handshake
- CloudFront selects the correct certificate based on the requested domain name

Dedicated IP is the legacy option:
- CloudFront allocates dedicated IP addresses for the distribution
- It can help with very old clients that do not support SNI
- It is more expensive and less commonly used

Exam-style takeaway:
- Choose SNI unless the question explicitly mentions legacy clients without SNI support
- Dedicated IP is mainly for compatibility with older TLS clients

**Viewer TLS troubleshooting scenarios**

A misconfigured viewer certificate usually causes browser or client handshake failures.

Common failure patterns:
- The certificate is not in `us-east-1`
- The certificate does not match the alternate domain name, such as `www.example.com`
- The security policy is too strict for an old client
- The distribution is configured for HTTPS-only but the client attempts HTTP

Example symptom:
- Users see a certificate mismatch error when accessing `https://www.example.com`
- Root cause: the certificate covers `example.com` but not `www.example.com`

Another scenario:
- A legacy client cannot connect after you switch to a TLS 1.2-only policy
- Root cause: the client only supports older protocols
- Exam answer: adjust the security policy only if backward compatibility is required

**Origin-side TLS**

CloudFront can also connect to the origin over HTTPS. This is separate from viewer-side TLS.

A simple mental model:

```text
Viewer --HTTPS--> CloudFront edge --HTTPS--> Origin
         cert at edge                 cert at origin
```

Origin-side TLS matters when you want encryption in transit between CloudFront and the origin. CloudFront acts as the TLS client when it connects to the origin.

Common origin types:
- **Amazon S3** origin with HTTPS access
- **Application Load Balancer** or **Classic Load Balancer** origin over HTTPS
- Custom HTTPS origin on EC2 or on-premises

For an S3 origin:
- CloudFront can use the S3 REST endpoint over HTTPS
- The origin certificate is managed by AWS for the S3 endpoint
- If you use the S3 website endpoint, note that it is HTTP-only and does not support HTTPS in the same way as the REST endpoint

For an ALB origin:
- You configure the ALB listener on port 443 with an ACM certificate
- CloudFront connects to the ALB using HTTPS
- CloudFront validates the origin certificate against the origin domain name and trust chain

Example origin configuration:
- Origin domain name: `my-alb-123456.us-east-1.elb.amazonaws.com`
- Origin protocol policy: `HTTPS only`
- Origin SSL protocols: modern TLS versions only, if supported by the origin
- Origin certificate: ACM certificate attached to the ALB listener

**Origin certificate validation in practice**

CloudFront validates the origin certificate when it connects over HTTPS. If the certificate is invalid, expired, self-signed, or does not match the origin hostname, the connection can fail.

Practical failure examples:
- You point CloudFront to `api.example.com`, but the origin certificate only covers `internal.example.local`
- CloudFront cannot validate the certificate chain because the origin uses a self-signed cert without proper trust
- The origin only supports outdated TLS versions, and CloudFront cannot negotiate a secure connection

Exam-style outcome:
- If the origin TLS setup is wrong, CloudFront returns errors such as 502/504-style origin failures rather than viewer certificate errors
- Viewer-side and origin-side TLS are separate troubleshooting domains

**CloudFront SSL/TLS deployment patterns**

Use the right certificate placement for the right side of the connection:
- Viewer-side certificate: attached to the CloudFront distribution
- Origin-side certificate: attached to the origin service, such as ALB or custom HTTPS server

A practical deployment checklist:
1. Request an ACM certificate in `us-east-1` for the CloudFront alternate domain names.
2. Attach the certificate to the CloudFront distribution.
3. Set the viewer protocol policy to redirect or require HTTPS.
4. Configure the origin to support HTTPS if you need end-to-end encryption.
5. Verify the origin certificate matches the origin hostname.
6. Test with `curl` and browser developer tools.

Example `curl` checks:

```bash
curl -I http://www.example.com
curl -I https://www.example.com
curl -Iv https://www.example.com
```

If you need to inspect the origin directly:

```bash
curl -Iv https://my-alb-123456.us-east-1.elb.amazonaws.com
```

**Practical deployment example: invalidation plus TLS**

Suppose you deploy a React app behind CloudFront with an S3 origin and a custom domain.

Recommended pattern:
- Upload hashed assets like `main.91c2d.js`
- Keep `index.html` at a stable path
- Configure CloudFront to redirect HTTP to HTTPS
- Use an ACM certificate in `us-east-1`
- Invalidate only `/index.html` after each release

Why this works:
- The HTML file updates quickly
- Asset files are cache-friendly because their names change when content changes
- TLS is enforced for users
- Origin access remains secure through HTTPS or origin access controls, depending on the design

**Exam-focused comparison**

| Topic | Best choice | Why |
|---|---|---|
| Frequent content updates | Versioned filenames | Avoids repeated invalidations |
| Same-path content replacement | Invalidation | Forces CloudFront to fetch fresh content |
| Viewer HTTPS | ACM cert in `us-east-1` | Required for CloudFront custom domains |
| Modern client support | SNI | Standard, scalable, lower cost |
| Legacy client compatibility | Dedicated IP | Supports clients without SNI |
| Secure origin connection | HTTPS origin | Encrypts traffic between CloudFront and origin |

**Common exam traps**

- Confusing viewer-side TLS with origin-side TLS
- Assuming invalidation is instant everywhere
- Forgetting that CloudFront certificates must be in `us-east-1`
- Choosing dedicated IP without a legacy-client requirement
- Using the S3 website endpoint when HTTPS to origin is required
- Invalidating every asset when versioned filenames would be better

**Quick mental model**

- **Invalidation** removes cached objects from edge locations
- **TTL** controls how long CloudFront keeps content without invalidation
- **Viewer TLS** protects the client-to-CloudFront connection
- **Origin TLS** protects the CloudFront-to-origin connection
- **SNI** is the default certificate selection method
- **Dedicated IP** is the legacy fallback

**Troubleshooting checklist**

When a CloudFront HTTPS setup fails, check:
- The certificate is in `us-east-1`
- The certificate name matches the alternate domain name
- The viewer security policy supports the client’s TLS version
- The origin certificate matches the origin hostname
- The origin supports HTTPS on the configured port
- The invalidation status is `Completed` if you expect fresh content

**Key exam takeaways**

- CloudFront invalidation is useful when you must purge cached objects before TTL expiry.
- Invalidation requests are asynchronous and show `InProgress` until propagation completes.
- CloudFront invalidation has a free allowance, and extra invalidation paths are billed.
- Versioned filenames are usually better than broad invalidations for regular deployments.
- CloudFront viewer certificates must be in `us-east-1`.
- Use SNI by default; use dedicated IP only for legacy clients without SNI support.
- Choose a modern TLS security policy, typically TLS 1.2 or TLS 1.3 capable, unless compatibility requires otherwise.
- Origin TLS is separate from viewer TLS and must be configured and validated independently.
