## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab overview**

This lab focuses on using the AWS Command Line Interface (AWS CLI) to inspect how a custom domain is wired through Amazon Route 53 and Amazon CloudFront. In practice, you use this workflow when a site is not resolving correctly, a certificate does not match the domain, or a distribution is pointing at the wrong origin.

The exam-relevant skill is not memorizing one command. It is learning how to trace a request end to end:

- DNS name in Route 53
- CloudFront distribution behind the alias
- Distribution settings such as origin, cache behavior, and certificate
- Supporting resources such as cache policies, origin request policies, and S3 access controls

**What you are trying to verify**

A typical troubleshooting path answers these questions:

- Which hosted zone contains the record for the domain?
- Is the record an alias, CNAME, or simple A record?
- Which CloudFront distribution does the record point to?
- Which origin serves the content?
- Is the attached ACM certificate in the correct Region for CloudFront?
- Are cache and origin request policies configured as expected?

> CloudFront custom domains use an ACM certificate in `us-east-1` for the distribution’s viewer certificate. That requirement applies to the certificate attached to CloudFront, not to every certificate you inspect in AWS.

**Core CLI setup**

Before you inspect resources, confirm that your CLI session can read the relevant services. You usually need permissions for Route 53, CloudFront, ACM, and possibly S3.

```bash
aws sts get-caller-identity
aws configure list
aws configure get region
```

Useful environment checks:

```bash
export AWS_PAGER=""
aws --version
```

If you work across multiple accounts or profiles, always specify the profile explicitly:

```bash
aws route53 list-hosted-zones --profile prod
aws cloudfront list-distributions --profile prod
```

**Route 53 hosted zone inspection**

Route 53 hosted zones are visible based on AWS account access and IAM permissions, not your network location. A private hosted zone is still visible in the console or CLI if your identity can list it; the difference is that its DNS answers are only resolvable from associated VPCs.

Start by finding the hosted zone for the domain. If multiple zones exist, filter carefully by name and type.

```bash
aws route53 list-hosted-zones-by-name \
  --dns-name example.com \
  --query 'HostedZones[?Name==`example.com.` || Name==`example.com`].[Id,Name,Config.PrivateZone]' \
  --output table
```

If you want to inspect all zones and identify public versus private:

```bash
aws route53 list-hosted-zones \
  --query 'HostedZones[*].[Id,Name,Config.PrivateZone,ResourceRecordSetCount]' \
  --output table
```

Key points:

- **Public hosted zone** answers DNS queries on the internet.
- **Private hosted zone** answers DNS queries only from associated VPCs.
- Multiple hosted zones can share similar names, so verify the zone ID before reading records.
- The zone ID is returned in the form `/hostedzone/Z1234567890ABC`.

**Inspecting record sets**

Once you have the hosted zone ID, list the record sets and locate the domain or subdomain you care about.

```bash
aws route53 list-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --query 'ResourceRecordSets[?Name==`www.example.com.` || Name==`example.com.`]' \
  --output json
```

Understand the record type before interpreting the target:

- **A / AAAA alias** records point to AWS resources such as CloudFront, ALB, or S3 website endpoints.
- **CNAME** records point to a DNS name in `ResourceRecords[].Value`.
- **MX**, **TXT**, and other records use their own value structures.

A common mistake is to query `AliasTarget` for a CNAME record. CNAME records do not use `AliasTarget`; they use `ResourceRecords[].Value`.

Example output patterns:

```json
[
  {
    "Name": "www.example.com.",
    "Type": "A",
    "AliasTarget": {
      "HostedZoneId": "Z2FDTNDATAQYW2",
      "DNSName": "d111111abcdef8.cloudfront.net.",
      "EvaluateTargetHealth": false
    }
  },
  {
    "Name": "api.example.com.",
    "Type": "CNAME",
    "TTL": 300,
    "ResourceRecords": [
      {
        "Value": "d222222abcdef8.cloudfront.net"
      }
    ]
  }
]
```

A practical query for alias records:

```bash
aws route53 list-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --query 'ResourceRecordSets[?AliasTarget!=null].[Name,Type,AliasTarget.DNSName,AliasTarget.HostedZoneId]' \
  --output table
```

A practical query for CNAME records:

```bash
aws route53 list-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --query 'ResourceRecordSets[?Type==`CNAME`].[Name,ResourceRecords[0].Value,TTL]' \
  --output table
```

**Tracing a domain to CloudFront**

For CloudFront alias records, the DNS name usually looks like `d123456abcdef8.cloudfront.net`. That value is the distribution domain name, not the distribution ID. You can use it to find the matching distribution.

```bash
aws cloudfront list-distributions \
  --query 'DistributionList.Items[*].[Id,DomainName,Aliases.Items[0]]' \
  --output table
```

A more targeted lookup by domain name:

```bash
aws cloudfront list-distributions \
  --query 'DistributionList.Items[?DomainName==`d111111abcdef8.cloudfront.net`].[Id,DomainName,Status,Enabled]' \
  --output table
```

If you only know the alias, inspect the aliases list:

```bash
aws cloudfront list-distributions \
  --query 'DistributionList.Items[?Aliases.Items && contains(Aliases.Items, `www.example.com`)].[Id,DomainName,Aliases.Items]' \
  --output table
```

> The fixed hosted zone ID `Z2FDTNDATAQYW2` is the CloudFront hosted zone ID used when Route 53 creates an alias to a CloudFront distribution. It is not a general CloudFront DNS value for every record type.

**Inspecting the distribution configuration**

After you identify the distribution ID, retrieve the full configuration.

```bash
aws cloudfront get-distribution-config \
  --id E1ABCDEF23456 \
  --output json
```

This returns the distribution configuration plus an `ETag`. The `ETag` matters when you update the distribution later, because CloudFront requires it for optimistic locking.

Useful fields to inspect:

- **Aliases**: custom domain names attached to the distribution
- **Origins**: where CloudFront fetches content
- **DefaultCacheBehavior**: default request handling
- **CacheBehaviors**: path-based behavior overrides
- **ViewerCertificate**: ACM certificate or alternate certificate settings
- **DefaultRootObject**: object served for `/`
- **CustomErrorResponses**: custom error mapping
- **Restrictions**: geo restriction settings
- **Logging**: access log configuration
- **PriceClass**: edge location coverage
- **Enabled**: whether the distribution is active

A focused query can make the output easier to read:

```bash
aws cloudfront get-distribution-config \
  --id E1ABCDEF23456 \
  --query 'DistributionConfig.{Aliases:Aliases.Items,Origins:Origins.Items,DefaultRootObject:DefaultRootObject,ViewerCertificate:ViewerCertificate,DefaultCacheBehavior:DefaultCacheBehavior,CustomErrorResponses:CustomErrorResponses.Items}' \
  --output json
```

**Certificate verification**

For a CloudFront distribution with a custom domain, verify the attached ACM certificate and its Region.

```bash
aws cloudfront get-distribution-config \
  --id E1ABCDEF23456 \
  --query 'DistributionConfig.ViewerCertificate' \
  --output json
```

Example output:

```json
{
  "ACMCertificateArn": "arn:aws:acm:us-east-1:123456789012:certificate/abcd1234-5678-90ab-cdef-1234567890ab",
  "SSLSupportMethod": "sni-only",
  "MinimumProtocolVersion": "TLSv1.2_2021",
  "Certificate": "arn:aws:acm:us-east-1:123456789012:certificate/abcd1234-5678-90ab-cdef-1234567890ab",
  "CertificateSource": "acm"
}
```

Important checks:

- The certificate ARN for CloudFront should be in `us-east-1`.
- The certificate must cover the exact hostname or a matching wildcard.
- The distribution must use a supported SSL/TLS policy.
- If the certificate is missing or in the wrong Region, the custom domain can fail even if DNS is correct.

To inspect the certificate itself:

```bash
aws acm describe-certificate \
  --certificate-arn arn:aws:acm:us-east-1:123456789012:certificate/abcd1234-5678-90ab-cdef-1234567890ab \
  --region us-east-1 \
  --output json
```

Look for:

- `DomainName`
- `SubjectAlternativeNames`
- `Status`
- `NotAfter`
- `InUseBy`

A quick validation query:

```bash
aws acm describe-certificate \
  --certificate-arn arn:aws:acm:us-east-1:123456789012:certificate/abcd1234-5678-90ab-cdef-1234567890ab \
  --region us-east-1 \
  --query 'Certificate.{DomainName:DomainName,SANs:SubjectAlternativeNames,Status:Status,NotAfter:NotAfter}' \
  --output table
```

**Inspecting origins**

The origin tells you where CloudFront fetches content. This is often the fastest way to determine whether the distribution fronts an S3 bucket, an Application Load Balancer, or a custom origin.

```bash
aws cloudfront get-distribution-config \
  --id E1ABCDEF23456 \
  --query 'DistributionConfig.Origins.Items[*].[Id,DomainName,OriginPath,CustomHeaders.Quantity,S3OriginConfig.OriginAccessIdentity,OriginAccessControlId]' \
  --output table
```

Common origin patterns:

- **S3 origin**: `bucket-name.s3.amazonaws.com` or regional S3 endpoint
- **Custom origin**: ALB, EC2, on-premises, or another HTTP endpoint
- **Origin Access Identity (OAI)**: legacy S3 access model
- **Origin Access Control (OAC)**: newer S3 access model with SigV4 signing

If the origin is S3, inspect whether the bucket is private and whether CloudFront is allowed to read it.

```bash
aws s3api get-bucket-policy --bucket my-private-bucket
aws s3api get-public-access-block --bucket my-private-bucket
```

For OAI, the origin config may include an identity string such as `origin-access-identity/cloudfront/E127EXAMPLE51Z`. For OAC, inspect the `OriginAccessControlId` and then describe the OAC resource:

```bash
aws cloudfront get-origin-access-control \
  --id E2ABCDEFGHIJKL \
  --output json
```

Example OAC fields to review:

- `Name`
- `Description`
- `SigningProtocol`
- `SigningBehavior`
- `OriginAccessControlOriginType`

**Inspecting cache and origin request policies**

CloudFront often returns only policy IDs in the distribution config. You then inspect the policy resources themselves to understand header, cookie, and query string forwarding behavior.

```bash
aws cloudfront get-distribution-config \
  --id E1ABCDEF23456 \
  --query 'DistributionConfig.DefaultCacheBehavior.{CachePolicyId:CachePolicyId,OriginRequestPolicyId:OriginRequestPolicyId,ResponseHeadersPolicyId:ResponseHeadersPolicyId}' \
  --output table
```

Then inspect each policy by ID.

Cache policy:

```bash
aws cloudfront get-cache-policy \
  --id 658327ea-f89d-4fab-a63d-7e88639e58f6 \
  --output json
```

Origin request policy:

```bash
aws cloudfront get-origin-request-policy \
  --id 88a5eaf4-2fd4-4709-b370-b4c650ea3fcf \
  --output json
```

Response headers policy:

```bash
aws cloudfront get-response-headers-policy \
  --id 67f7725c-6f97-4210-82d7-5512b31e9d03 \
  --output json
```

What to look for:

- **Cache policy**: which headers, cookies, and query strings affect cache keys
- **Origin request policy**: which viewer values are forwarded to the origin
- **Response headers policy**: security headers and CORS behavior returned to viewers

**Default root object and custom error responses**

These settings often explain why a site works at one path but fails at `/`.

```bash
aws cloudfront get-distribution-config \
  --id E1ABCDEF23456 \
  --query 'DistributionConfig.{DefaultRootObject:DefaultRootObject,CustomErrorResponses:CustomErrorResponses.Items}' \
  --output json
```

Examples:

- `DefaultRootObject: index.html` means requests to `/` map to `/index.html`
- A custom error response can map `403` or `404` to a friendly page
- A misconfigured default root object can cause a blank homepage or `403 Access Denied`

**End-to-end troubleshooting example: broken custom domain**

Suppose `www.example.com` returns a certificate error or a 404. You can trace it like this:

1. Find the hosted zone for `example.com`.
2. Locate the `www.example.com` record.
3. Confirm whether it is an alias to a CloudFront distribution or a CNAME to a CloudFront domain name.
4. Identify the distribution ID from the CloudFront domain name.
5. Inspect the distribution’s `ViewerCertificate`, `Aliases`, and `DefaultRootObject`.
6. Inspect the origin and confirm the bucket or backend is reachable.
7. Inspect cache and origin request policies if the origin depends on forwarded headers or query strings.

Example workflow:

```bash
aws route53 list-hosted-zones-by-name \
  --dns-name example.com \
  --query 'HostedZones[0].[Id,Name]' \
  --output table
```

```bash
aws route53 list-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --query 'ResourceRecordSets[?Name==`www.example.com.`].[Name,Type,AliasTarget.DNSName,ResourceRecords[0].Value]' \
  --output table
```

```bash
aws cloudfront list-distributions \
  --query 'DistributionList.Items[?DomainName==`d111111abcdef8.cloudfront.net`].[Id,DomainName]' \
  --output table
```

```bash
aws cloudfront get-distribution-config \
  --id E1ABCDEF23456 \
  --query 'DistributionConfig.{Aliases:Aliases.Items,ViewerCertificate:ViewerCertificate,DefaultRootObject:DefaultRootObject,Origins:Origins.Items}' \
  --output json
```

If the certificate ARN is not in `us-east-1`, that is a strong clue for a CloudFront custom domain failure.

**End-to-end troubleshooting example: private S3 origin with access denied**

A second common scenario is a CloudFront distribution that returns `403 Forbidden` even though the S3 bucket contains the object.

Checklist:

- Confirm the origin is an S3 bucket.
- Check whether the bucket is private.
- Determine whether the distribution uses OAI or OAC.
- Verify the bucket policy allows the CloudFront principal or OAI/OAC.
- Confirm the default root object exists in the bucket.

Useful commands:

```bash
aws cloudfront get-distribution-config \
  --id E1ABCDEF23456 \
  --query 'DistributionConfig.Origins.Items[*].[Id,DomainName,S3OriginConfig.OriginAccessIdentity,OriginAccessControlId]' \
  --output table
```

```bash
aws s3api head-object \
  --bucket my-private-bucket \
  --key index.html
```

```bash
aws s3api get-bucket-policy \
  --bucket my-private-bucket \
  --query 'Policy' \
  --output text
```

If the bucket policy is missing the correct CloudFront access statement, the distribution can be healthy while the origin still denies requests.

**Practical query patterns**

Use `--query` to reduce noise and focus on the fields that matter.

Hosted zones:

```bash
aws route53 list-hosted-zones \
  --query 'HostedZones[?Config.PrivateZone==`false`].[Name,Id]' \
  --output table
```

Alias records:

```bash
aws route53 list-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --query 'ResourceRecordSets[?AliasTarget!=null].[Name,Type,AliasTarget.DNSName]' \
  --output table
```

CloudFront distribution summary:

```bash
aws cloudfront list-distributions \
  --query 'DistributionList.Items[*].[Id,DomainName,Status,Enabled,Aliases.Items[0]]' \
  --output table
```

Policy IDs:

```bash
aws cloudfront get-distribution-config \
  --id E1ABCDEF23456 \
  --query 'DistributionConfig.DefaultCacheBehavior.[CachePolicyId,OriginRequestPolicyId,ResponseHeadersPolicyId]' \
  --output table
```

**Common exam takeaways**

- Use Route 53 to identify the DNS record and its target.
- Use CloudFront to inspect the distribution behind the target domain.
- Use ACM in `us-east-1` for the certificate attached to a CloudFront custom domain.
- Use the distribution config to verify aliases, origins, root object, and error handling.
- Use policy lookup commands when CloudFront returns only IDs.
- Distinguish public hosted zones from private hosted zones by DNS reachability, not by whether you can list them in AWS.

**Mental model for the lab**

Think of the path as a chain of evidence:

- DNS record tells you where traffic goes.
- CloudFront distribution tells you how traffic is handled.
- Viewer certificate tells you whether the custom domain can terminate TLS.
- Origin settings tell you where content comes from.
- Cache and request policies tell you what CloudFront forwards and caches.

If you can trace each link with the AWS CLI, you can diagnose most exam-style DNS and CloudFront troubleshooting scenarios quickly and confidently.
