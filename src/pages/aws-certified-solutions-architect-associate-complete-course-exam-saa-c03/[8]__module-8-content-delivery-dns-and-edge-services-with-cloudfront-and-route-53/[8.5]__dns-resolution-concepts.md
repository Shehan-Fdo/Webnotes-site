## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**DNS resolution fundamentals**

DNS (<abbr title="Domain Name System">Domain Name System</abbr>) translates human-readable names into IP addresses and other records. In AWS, you need to understand both the public DNS system on the internet and the private DNS behavior inside a VPC (<abbr title="Virtual Private Cloud">Virtual Private Cloud</abbr>).

A DNS lookup usually follows this pattern:

- Your client checks local caches first.
- The client asks a recursive resolver if it does not already know the answer.
- The resolver queries authoritative name servers for the zone that owns the record.
- The final answer is cached according to the record’s TTL (<abbr title="Time To Live">Time To Live</abbr>).

In AWS exam scenarios, the key question is often not “what is DNS?” but “which resolver answers this name from this network location?”

**How resolution works in a VPC**

Every VPC has an AmazonProvidedDNS resolver available at the VPC base address plus 2 within the VPC CIDR block. For example, in a `10.0.0.0/16` VPC, the resolver is at `10.0.0.2`. In a different CIDR block, the address changes accordingly.

This resolver is a recursive resolver for instances in the VPC. It can answer:

- Public DNS names by resolving them through normal recursive lookup and caching results.
- Private DNS names for AWS resources and private hosted zones that are associated with the VPC.
- Some AWS service endpoints that rely on DNS-based access patterns.

Important distinctions:

- **DNS support** must be enabled for the AmazonProvidedDNS resolver to work.
- **DNS hostnames** mainly control whether instances in the VPC receive DNS hostnames, especially public DNS hostnames for instances with public IPs.
- These settings are related but not identical, so treat them separately in troubleshooting and exam questions.

> If DNS support is disabled, instances cannot use the VPC-provided resolver. If DNS hostnames are disabled, name resolution may still work, but instance hostname assignment and some workflows change.

**Public DNS, private DNS, and hosted zones**

Route 53 is AWS’s DNS service. It manages hosted zones, which are containers for DNS records.

- A **public hosted zone** is resolvable from the internet.
- A **private hosted zone** is resolvable only from associated VPCs.
- The same domain name can exist in both public and private hosted zones, which is a common split-horizon DNS pattern.

Split-horizon DNS is useful when:

- Internal clients should resolve a private IP address.
- External clients should resolve a public endpoint.
- You want the same name to work differently depending on where the query originates.

Example:

- `app.example.com` in the public hosted zone points to a public load balancer.
- `app.example.com` in the private hosted zone points to an internal load balancer.
- Instances inside the associated VPC resolve the private answer.
- Clients on the internet resolve the public answer.

This works only if the private hosted zone is correctly associated with the VPC. External clients do not query the private hosted zone at all.

**Record types you should know**

Common Route 53 record types appear frequently in architecture questions.

- **A record** maps a name to an IPv4 address.
- **AAAA record** maps a name to an IPv6 address.
- **CNAME** maps one name to another name.
- **Alias record** is an AWS-specific Route 53 feature that points a DNS name to AWS resources such as Application Load Balancers, CloudFront distributions, S3 website endpoints, and some other supported targets.

Practical differences:

- Use a **CNAME** for name-to-name mapping, but not at the zone apex in standard DNS.
- Use an **Alias** at the zone apex when you need to point the root domain to an AWS resource.
- Alias records are resolved by Route 53 and can return the target’s address without an extra client-side lookup.

Example Route 53 alias record in JSON-like form:

```json
{
  "Name": "app.example.com",
  "Type": "A",
  "AliasTarget": {
    "DNSName": "internal-my-alb-1234567890.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": false
  }
}
```

The exact target details depend on the AWS resource and region. In practice, you obtain the correct DNS name and hosted zone information from the AWS console, CLI, or API for that specific load balancer or service.

**Recursive, authoritative, and caching behavior**

A recursive resolver answers on behalf of the client. It is not authoritative for the zone; it finds the authoritative answer and caches it.

Key behaviors:

- **Caching** reduces latency and DNS query volume.
- **TTL** controls how long a cached answer can be reused.
- **Negative caching** can also occur for failed lookups, depending on DNS behavior and record settings.
- **Forwarding** may be used in hybrid DNS designs when one resolver delegates queries to another.

Why this matters:

- A recently changed record may continue to resolve to the old value until caches expire.
- Troubleshooting DNS often requires checking both the authoritative record and the resolver cache path.

**AWS service name resolution patterns**

Many AWS services rely on DNS names rather than fixed IPs.

Common examples:

- Application Load Balancers and Network Load Balancers expose DNS names that map to changing IPs.
- Amazon RDS endpoints are DNS names that can change during failover.
- Amazon S3 website endpoints and CloudFront distributions are accessed by DNS names.
- Interface VPC endpoints use private DNS names when enabled, so service names resolve to private IPs inside the VPC.

This is why you should prefer DNS names over hardcoded IP addresses in AWS architectures.

**Hybrid DNS with on-premises networks**

In hybrid environments, you often need DNS resolution across AWS and on-premises systems.

Typical patterns include:

- On-premises clients resolving private AWS names through conditional forwarding to Route 53 Resolver inbound endpoints.
- AWS workloads resolving on-premises names through Route 53 Resolver outbound endpoints and rules.
- Centralized DNS forwarding through corporate DNS servers.

Route 53 Resolver is the AWS service that supports these hybrid patterns.

Core components:

- **Inbound endpoint**: lets on-premises DNS servers forward queries into AWS.
- **Outbound endpoint**: lets AWS forward selected queries to on-premises DNS servers.
- **Resolver rules**: define which domain names are forwarded and to where.

Example forwarding rule concept:

- Queries for `corp.example.com` are forwarded to on-premises DNS servers.
- Queries for `aws.example.com` are resolved inside AWS.
- Queries for public internet names use normal recursive resolution.

Example `dig` test from an EC2 instance:

```bash
dig app.example.com
dig @10.0.0.2 app.example.com
dig corp.example.com
```

Use these commands to compare the default resolver path with a direct query to the VPC resolver when appropriate.

**Troubleshooting DNS in AWS**

When DNS does not behave as expected, isolate the layer that is failing.

1. Confirm the record exists in the correct hosted zone.
2. Confirm the hosted zone is public or private as intended.
3. For private hosted zones, confirm the VPC association is correct.
4. Confirm the VPC has DNS support enabled.
5. Confirm the client is using a resolver path that can reach the intended zone.
6. Check for stale caches and TTL-related delays.
7. Verify security groups, NACLs (<abbr title="Network Access Control Lists">Network Access Control Lists</abbr>), and firewall rules if DNS traffic is blocked in a hybrid design.

Useful commands:

```bash
nslookup app.example.com
dig app.example.com
dig +trace app.example.com
resolvectl status
```

Notes on Linux troubleshooting:

- `/etc/resolv.conf` may point to a local stub resolver such as `127.0.0.53` on systems using `systemd-resolved`.
- That file does not always show the upstream DNS server directly.
- Use `resolvectl status`, `nmcli dev show`, or your OS-specific tooling to inspect the actual resolver path.

**Private hosted zones and VPC association**

Private hosted zones are only visible to associated VPCs. This is a common exam trap.

What to remember:

- A private hosted zone can be associated with one or more VPCs.
- Cross-account association is possible with the proper authorization workflow.
- If the VPC is not associated, instances in that VPC cannot resolve the private records.
- Overlapping public and private zones with the same name are resolved according to where the query originates.

Example topology:

- VPC A hosts `internal.example.com` private records.
- VPC B is not associated and cannot resolve them.
- On-premises DNS can resolve them only if you forward queries through Route 53 Resolver and the VPC association exists.

**DNS and load balancers**

Load balancers are a frequent DNS use case.

- **ALB** (<abbr title="Application Load Balancer">Application Load Balancer</abbr>) and **NLB** (<abbr title="Network Load Balancer">Network Load Balancer</abbr>) are addressed by DNS names.
- Their underlying IPs can change, so you should not hardcode them.
- Route 53 alias records are the preferred way to map your domain to these services.

Practical example:

```text
example.com -> Route 53 alias -> ALB DNS name -> targets in multiple AZs
```

Why alias records are useful:

- They work at the zone apex.
- They integrate with AWS resource health evaluation in some cases.
- They avoid an extra CNAME indirection.

**Exam-focused distinctions**

These differences are easy to mix up:

| Concept | What it does | Common exam clue |
|---|---|---|
| DNS support | Enables the VPC-provided resolver | “Instances cannot resolve names in the VPC” |
| DNS hostnames | Controls DNS hostname assignment for instances | “Public DNS names for EC2 instances” |
| Public hosted zone | Resolvable from the internet | “External users must resolve the name” |
| Private hosted zone | Resolvable only from associated VPCs | “Internal-only name resolution” |
| Alias record | Route 53 pointer to AWS resources | “Root domain to ALB/CloudFront/S3” |
| Resolver rule | Forwards queries for a domain | “Hybrid DNS forwarding” |

**Practical architecture examples**

A common architecture uses split DNS for an application:

- Internet users resolve `app.example.com` to a public ALB.
- Internal users in the VPC resolve the same name to an internal ALB.
- The application tier uses private IPs and security groups.
- Route 53 private hosted zone is associated with the application VPC.

Another common architecture uses hybrid forwarding:

- On-premises DNS forwards `aws.corp.example.com` to Route 53 Resolver inbound endpoints.
- AWS workloads forward `corp.example.com` to on-premises DNS through outbound endpoints.
- Both sides resolve each other’s names without exposing private records publicly.

**Quick command examples**

Use these commands to inspect DNS behavior:

```bash
# Query the default resolver path
dig app.example.com

# Query a specific resolver, such as the VPC resolver when appropriate
dig @10.0.0.2 app.example.com

# Show the full resolution chain
dig +trace app.example.com

# Check local resolver configuration on systemd-based Linux
resolvectl status
```

If you need to test a private hosted zone from inside a VPC, run the query from an EC2 instance in an associated VPC or from a host that forwards to the VPC resolver through a supported path.

**Key takeaways**

- DNS in AWS is a combination of Route 53, VPC resolver behavior, and hybrid forwarding tools.
- The AmazonProvidedDNS resolver is available at the VPC base address plus 2, but the exact IP depends on the VPC CIDR.
- Private hosted zones are only visible to associated VPCs.
- Alias records are the standard way to map domains to many AWS resources.
- Route 53 Resolver inbound and outbound endpoints solve hybrid DNS use cases.
- Troubleshooting requires checking the record, the zone type, the VPC association, the resolver path, and cache effects.
