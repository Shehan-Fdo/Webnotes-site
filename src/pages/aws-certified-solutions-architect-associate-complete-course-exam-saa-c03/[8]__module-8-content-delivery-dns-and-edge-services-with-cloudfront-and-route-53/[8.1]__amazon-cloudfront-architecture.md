## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon CloudFront overview**

Amazon CloudFront is AWS’s <abbr title="Content Delivery Network">CDN</abbr> for delivering content with low latency and high transfer speeds. You place CloudFront in front of one or more origins, and CloudFront serves content from edge locations close to viewers when possible. It is commonly used for static websites, media delivery, APIs, and private content distribution.

CloudFront sits at the edge of the AWS network and reduces load on origins by caching responses. It also adds security and traffic management features such as TLS termination, signed URLs, signed cookies, origin access controls, and edge compute.

- **Viewer** requests content from the nearest edge location.
- **Origin** is the backend source, such as Amazon S3, an Application Load Balancer, or an HTTP server.
- **Cache behavior** controls how requests are matched and how CloudFront forwards them.

**Core request flow**

A typical request follows this path:

1. The viewer requests `https://d123example.cloudfront.net/index.html`.
2. CloudFront checks the edge cache.
3. If the object is cached and fresh, CloudFront returns it immediately.
4. If not cached, CloudFront forwards the request to the origin.
5. CloudFront caches the response according to the cache policy and TTL settings.

This model improves latency and reduces origin traffic, but only if you tune cache keys, headers, cookies, and query strings appropriately.

**Distribution architecture**

CloudFront uses a distribution as the top-level configuration object. A distribution maps viewer requests to origins and behaviors.

- **Origins** define where CloudFront fetches content.
- **Behaviors** define how CloudFront handles path patterns such as `/images/*` or `/api/*`.
- **Default behavior** applies when no path pattern matches.
- **Alternate domain names** let you use your own DNS name, such as `cdn.example.com`.
- **TLS certificate** from AWS Certificate Manager is required for custom domains.

CloudFront no longer uses the old RTMP distribution model for new deployments. In modern architectures, you work with standard CloudFront distributions and configure delivery behavior through origins, cache behaviors, and policies.

**Origins and origin types**

CloudFront can front several origin types. The origin choice affects caching, security, and request forwarding.

- **Amazon S3 origin**
  - Best for static assets, downloads, and private content.
  - Use Origin Access Control to keep the bucket private.
  - Prefer bucket policies over public access.
- **Application Load Balancer origin**
  - Best for dynamic web applications and APIs.
  - Supports HTTP and HTTPS origins.
  - Useful when you need path-based routing at the application layer.
- **Custom HTTP origin**
  - Use for on-premises servers or third-party endpoints.
  - CloudFront still provides caching and edge security, but origin availability is your responsibility.

A common exam pattern is to choose S3 for static content and ALB for dynamic content behind the same distribution using separate behaviors.

**Cache behaviors and path patterns**

Cache behaviors let you apply different rules to different URL paths.

- `/static/*` can use long TTLs and aggressive caching.
- `/api/*` can forward headers, cookies, and query strings with little or no caching.
- `/private/*` can require signed URLs or signed cookies.

CloudFront evaluates behaviors in order of specificity. More specific path patterns take precedence over the default behavior.

**Caching fundamentals**

CloudFront caching is driven by the cache key and freshness settings.

- **TTL** stands for time to live.
- **Minimum TTL** sets the lower bound for freshness.
- **Default TTL** applies when the origin does not provide cache headers.
- **Maximum TTL** caps how long CloudFront can keep an object.

CloudFront can cache based on:
- Query strings
- Headers
- Cookies
- Compression variants such as `gzip` and `br`

Use the smallest cache key that still preserves correctness. Forwarding too many headers or cookies reduces cache hit ratio and increases origin load.

> Cache invalidation is not a substitute for good TTL design. Use invalidations for urgent changes, not as your normal deployment mechanism.

**Cache policies and origin request policies**

Modern CloudFront uses policies to control caching and forwarding.

- **Cache policy**
  - Determines what becomes part of the cache key.
  - Controls TTL behavior.
  - Example use: cache static assets by path only.
- **Origin request policy**
  - Determines what CloudFront forwards to the origin.
  - Example use: forward `Authorization` to an API origin without caching on it.
- **Response headers policy**
  - Adds or overrides response headers at the edge.
  - Example use: set security headers such as `Strict-Transport-Security`.

This separation is important because you often want to forward a value to the origin without using it in the cache key.

**Viewer protocol policy**

Viewer protocol policy controls how CloudFront handles HTTP and HTTPS requests from viewers.

- **Allow all**
  - Accepts both HTTP and HTTPS.
  - Use only when encryption is not required.
- **Redirect HTTP to HTTPS**
  - Sends an HTTP 301 or 302 redirect to HTTPS.
  - Common choice for public websites.
- **HTTPS only**
  - Rejects HTTP requests.
  - Use when you want strict transport security and no redirect behavior.

For exam questions, choose redirect-to-HTTPS for user-friendly web traffic and HTTPS-only when the application must refuse plaintext requests.

**Supported methods and caching behavior**

CloudFront supports standard HTTP methods, but caching behavior differs by method.

- **GET** and **HEAD**
  - Commonly cached.
  - Used for retrieving objects and metadata.
- **OPTIONS**
  - Often used for CORS preflight requests.
  - Can be cached if configured appropriately.
- **PUT**, **POST**, **PATCH**, **DELETE**
  - Usually forwarded to the origin.
  - Not typically cached because they change state.

For APIs, you often forward all required methods but cache only safe responses. If you cache dynamic API responses incorrectly, users can receive stale or incorrect data.

**Invalidations and deployment strategy**

CloudFront invalidation removes objects from edge caches before TTL expiration.

Use invalidation when:
- You need to replace a file immediately.
- You deployed a critical fix to a cached object.
- You cannot wait for TTL expiry.

Prefer versioned file names when:
- You deploy static assets frequently.
- You want to avoid invalidation cost and operational overhead.
- You can publish `app.4f3c2.js` instead of overwriting `app.js`.

Practical invalidation example:

```bash
aws cloudfront create-invalidation \
  --distribution-id E123EXAMPLE \
  --paths "/index.html" "/assets/*"
```

Important operational points:
- Invalidations are not free at scale.
- Wildcards are limited; `/*` is the main wildcard pattern and applies to the end of a path pattern.
- Invalidating `/*` can be expensive and should be reserved for broad emergency refreshes.
- Invalidations do not instantly purge every edge location in a deterministic way; propagation takes time.

**Origin Shield**

Origin Shield adds an extra centralized caching layer between CloudFront edge locations and your origin. It is useful when many edge locations request the same objects and you want to reduce origin load further.

Use Origin Shield when:
- You have a high-volume global audience.
- Your origin is sensitive to burst traffic.
- You want to improve cache hit ratio across multiple edge locations.

Origin Shield can reduce the number of origin fetches by consolidating misses through a designated shield region. This is especially useful for large media files, popular downloads, and traffic spikes after cache expiry.

**Security at the edge**

CloudFront provides several security controls that commonly appear in exam scenarios.

- **AWS WAF**
  - Filters malicious requests at the edge.
  - Useful for rate limiting and web exploit protection.
- **AWS Shield**
  - Helps protect against DDoS attacks.
  - Standard protection is included; Advanced adds more controls.
- **Signed URLs**
  - Grant access to a specific object for a limited time.
  - Good for one-off downloads.
- **Signed cookies**
  - Grant access to multiple private objects.
  - Better for authenticated sessions over many files.
- **Origin Access Control**
  - Lets CloudFront access private S3 buckets securely.
  - Preferred over public bucket access.
- **Field-level encryption**
  - Encrypts sensitive form fields at the edge before forwarding to the origin.
  - Useful for data such as account numbers or personal identifiers.

Field-level encryption is a niche but important feature for scenarios where you must protect specific fields while still allowing the rest of the request to pass through.

**Private content delivery pattern**

A common private content design uses CloudFront in front of a private S3 bucket.

1. Store objects in a private S3 bucket.
2. Create a CloudFront distribution with the S3 origin.
3. Use Origin Access Control so only CloudFront can read the bucket.
4. Require signed URLs or signed cookies for viewer access.
5. Optionally add AWS WAF for request filtering.

Example S3 bucket policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontReadOnly",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-private-bucket/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/E123EXAMPLE"
        }
      }
    }
  ]
}
```

This pattern is preferred when you need secure global delivery without exposing the bucket publicly.

**Lambda@Edge vs CloudFront Functions**

Both services run code at the edge, but they solve different problems.

| Feature | CloudFront Functions | Lambda@Edge |
|---|---|---|
| Trigger points | Viewer request, viewer response | Viewer request, viewer response, origin request, origin response |
| Runtime | Lightweight JavaScript | Lambda runtime model |
| Latency | Very low | Higher than CloudFront Functions |
| Typical use | URL rewrites, header manipulation, simple auth logic | Complex request/response processing, origin customization |
| Deployment scope | Global at CloudFront edge | Replicated to edge locations from a Lambda version |

Use CloudFront Functions for simple, fast viewer-side logic. Use Lambda@Edge when you need origin-time logic, more complex processing, or broader event coverage.

Example CloudFront Function snippet:

```javascript
function handler(event) {
  var request = event.request;
  if (request.uri.endsWith('/')) {
    request.uri += 'index.html';
  }
  return request;
}
```

Example Lambda@Edge use case:
- Add or modify origin headers before CloudFront contacts the origin.
- Implement advanced A/B routing.
- Customize origin selection based on request attributes.

**Origin failover and high availability**

CloudFront can improve availability by using multiple origins and origin groups. If the primary origin fails, CloudFront can fail over to a secondary origin based on configured HTTP status codes.

A practical failover design:
- Primary origin: Application Load Balancer in `us-east-1`
- Secondary origin: S3 bucket with a maintenance page
- Origin group: Fail over on `500`, `502`, `503`, and `504`

This pattern is useful when you want a static fallback page or disaster recovery behavior at the edge.

Example topology:
- Viewer requests `www.example.com`
- CloudFront routes `/app/*` to the ALB
- If the ALB returns a failover status, CloudFront serves content from the secondary origin

**Real-world configuration example**

A common architecture for a modern web app looks like this:

- `cdn.example.com` points to CloudFront
- `/static/*` routes to S3 with long caching
- `/api/*` routes to ALB with minimal caching
- Viewer protocol policy redirects HTTP to HTTPS
- Signed cookies protect premium content
- AWS WAF blocks abusive traffic
- Origin Shield reduces origin load during spikes

Example behavior intent:
- Static assets: cache for days or weeks
- HTML documents: short TTL or invalidation on deploy
- API responses: forward required headers and methods, cache selectively

**CLI examples**

Create a distribution is usually done through the console or infrastructure as code, but the AWS CLI is useful for inspection and invalidation.

List distributions:

```bash
aws cloudfront list-distributions
```

Get distribution config:

```bash
aws cloudfront get-distribution-config --id E123EXAMPLE
```

Create an invalidation:

```bash
aws cloudfront create-invalidation \
  --distribution-id E123EXAMPLE \
  --paths "/index.html" "/app.js"
```

Check invalidation status:

```bash
aws cloudfront list-invalidations --distribution-id E123EXAMPLE
```

**Exam-focused distinctions**

- **CloudFront vs S3 static website hosting**
  - CloudFront adds caching, HTTPS, custom domains, and edge security.
  - S3 website endpoints are simpler but less capable.
- **CloudFront vs Application Load Balancer**
  - CloudFront is global edge delivery.
  - ALB is regional traffic distribution.
- **CloudFront Functions vs Lambda@Edge**
  - Functions are best for lightweight viewer-side logic.
  - Lambda@Edge is better for more complex edge processing.
- **Invalidation vs TTL**
  - TTL is the normal expiration mechanism.
  - Invalidation is for urgent refreshes.
- **Signed URLs vs signed cookies**
  - Signed URLs are object-specific.
  - Signed cookies are better for multiple protected objects in one session.

**Common pitfalls**

- Forwarding too many headers or cookies and destroying cache efficiency.
- Using `/*` invalidations as a routine deployment strategy.
- Choosing CloudFront Functions when you need origin-request logic.
- Leaving an S3 origin public when Origin Access Control is the better design.
- Caching API responses without considering authorization or user-specific data.
- Assuming HTTP is acceptable when the viewer protocol policy should enforce HTTPS.

**When to choose CloudFront**

Use CloudFront when you need:
- Global low-latency delivery
- Static asset acceleration
- Secure private content distribution
- Edge-based request filtering
- TLS termination and custom domains
- Better origin offload and resilience

CloudFront is often the right answer for exam scenarios involving global users, static websites, media delivery, or secure content access at scale.
