## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Overview**

Pointing a domain to either Amazon CloudFront or an Application Load Balancer (ALB) is a common production task. You usually combine DNS, TLS certificates, and an AWS resource that can answer for the hostname. The main difference is that CloudFront is a global content delivery network, while an ALB is a regional Layer 7 load balancer.

Use CloudFront when you want edge caching, global latency reduction, and a single entry point for static or dynamic content. Use an ALB when you want direct regional routing to targets such as EC2 instances, containers, or IP addresses.

- **DNS** maps your domain name to the AWS endpoint.
- **TLS** certificates provide HTTPS for the custom hostname.
- **Alias records** in Route 53 simplify pointing apex domains to AWS resources.

**Core DNS and routing concepts**

A custom domain such as `www.example.com` must resolve to the correct AWS endpoint. For AWS-managed endpoints, Route 53 alias records are preferred because they can point to CloudFront distributions and ALBs without requiring a fixed IP address.

CloudFront distributions and ALBs both expose DNS names that you can target:

- CloudFront distribution domain names look like `d123example.cloudfront.net`.
- ALB DNS names look like `my-alb-123456.us-east-1.elb.amazonaws.com`.

You typically do not use a standard `A` record with a hardcoded IP address because these services do not have stable public IPs. Instead, you use:

- **Alias A/AAAA records** in Route 53 for the zone apex, such as `example.com`.
- **CNAME records** for subdomains, such as `www.example.com`, when alias is not required.
- **AAAA records** if you want IPv6 support and the target supports it.

> A zone apex like `example.com` cannot use a CNAME record in standard DNS. In Route 53, an alias record solves this limitation for AWS targets.

**CloudFront domain setup**

CloudFront sits in front of your origin and serves content from edge locations. When you point a domain to CloudFront, you usually configure three things:

1. Add the custom domain name as an alternate domain name, also called a CNAME, on the distribution.
2. Attach an ACM (AWS Certificate Manager) certificate that covers the hostname.
3. Create a DNS record that routes the domain to the CloudFront distribution.

A typical setup for `www.example.com` looks like this:

```text
Client -> Route 53 -> CloudFront distribution -> Origin (S3, ALB, or custom server)
```

For HTTPS, the certificate must be in `us-east-1` for CloudFront because CloudFront is a global service and uses certificates from that region. The certificate must include the exact hostname or a matching wildcard, such as `www.example.com` or `*.example.com`.

Example Route 53 alias record for CloudFront:

```text
Name: www.example.com
Type: A
Alias: Yes
Alias target: d123example.cloudfront.net
```

If you also want IPv6, create an `AAAA` alias record to the same distribution.

**CloudFront configuration details**

CloudFront requires the alternate domain name to match the hostname in the request. If the DNS record points to CloudFront but the distribution does not list the hostname, CloudFront returns a certificate or host mismatch error.

Common CloudFront requirements:

- Add the hostname under **Alternate domain name (CNAME)**.
- Use an ACM certificate in `us-east-1`.
- Ensure the origin is reachable and correctly configured.
- Set the viewer protocol policy to redirect HTTP to HTTPS when appropriate.

Example origin and behavior considerations:

- If the origin is an S3 bucket, use the S3 REST endpoint for private content with Origin Access Control or Origin Access Identity, depending on the design.
- If the origin is an ALB, CloudFront can front dynamic content and still provide caching for cacheable responses.
- If the origin is a custom web server, make sure it accepts the `Host` header and serves the expected content.

A practical CLI example for creating a CloudFront distribution is usually done through infrastructure as code or the console, but you can inspect the distribution with the AWS CLI:

```bash
aws cloudfront list-distributions
aws cloudfront get-distribution --id E123EXAMPLE
```

**ALB domain setup**

An ALB is regional and commonly fronts web applications, APIs, and microservices. To point a domain to an ALB, you create a DNS record that targets the ALB DNS name and attach an HTTPS listener with a certificate for the hostname.

Typical flow:

1. Create or identify the ALB.
2. Add an HTTPS listener on port `443`.
3. Attach an ACM certificate for the domain.
4. Create a Route 53 alias record to the ALB.

Example topology:

```text
Client -> Route 53 -> ALB -> Target Group -> EC2 / ECS / IP targets
```

For an ALB, the certificate can be in the same region as the ALB. The certificate must cover the hostname you use, such as `app.example.com` or `example.com`.

Example Route 53 alias record for an ALB:

```text
Name: app.example.com
Type: A
Alias: Yes
Alias target: my-alb-123456.us-east-1.elb.amazonaws.com
```

You can also create an `AAAA` alias record if IPv6 is enabled and supported in your architecture.

**CloudFront vs ALB**

Choose based on the traffic pattern and the role of the service.

| Aspect | CloudFront | ALB |
|---|---|---|
| Scope | Global edge network | Regional |
| Primary use | CDN, caching, edge delivery | Load balancing to targets |
| TLS certificate region | `us-east-1` | Same region as ALB |
| DNS target | CloudFront distribution domain | ALB DNS name |
| Best for | Static assets, accelerated delivery, edge caching | Web apps, APIs, internal routing patterns |
| Hostname support | Alternate domain name required | Listener certificate and DNS record required |

CloudFront is often the better choice when you want to reduce latency worldwide or protect the origin behind a CDN. ALB is the better choice when you need direct application routing, path-based routing, host-based routing, or integration with target groups.

**Route 53 record types and when to use them**

Route 53 alias records are the most common choice for AWS endpoints because they behave like `A` or `AAAA` records but can point to AWS resources.

- Use an **alias A record** for `example.com` pointing to CloudFront or ALB.
- Use an **alias AAAA record** when you want IPv6 support.
- Use a **CNAME** for subdomains only when alias is not needed.
- Avoid hardcoded IP addresses for CloudFront and ALB because the endpoints are managed and can change.

Example DNS patterns:

```text
example.com      -> ALB alias A
www.example.com  -> CloudFront alias A
api.example.com   -> ALB alias A
static.example.com -> CloudFront alias A
```

If you need both `example.com` and `www.example.com` to work, you often configure one as the primary and redirect the other at the application or CloudFront layer.

**Certificate planning**

Certificate placement is a frequent source of errors.

For CloudFront:

- Request or import the certificate in `us-east-1`.
- Include the exact hostname or wildcard.
- Attach the certificate to the distribution.

For ALB:

- Request or import the certificate in the same region as the ALB.
- Attach it to the HTTPS listener.
- Ensure the listener rule forwards traffic to the correct target group.

Example ACM certificate names:

- `example.com`
- `www.example.com`
- `*.example.com`

A wildcard certificate for `*.example.com` covers `app.example.com` and `www.example.com`, but not the apex `example.com`.

**Practical setup examples**

Example 1: Point `www.example.com` to CloudFront

1. Create a CloudFront distribution with an origin.
2. Add `www.example.com` as an alternate domain name.
3. Request an ACM certificate in `us-east-1` for `www.example.com`.
4. Attach the certificate to the distribution.
5. Create a Route 53 alias `A` record for `www.example.com` to the distribution.

Example 2: Point `app.example.com` to an ALB

1. Create an ALB in the target region.
2. Create an HTTPS listener on port `443`.
3. Attach an ACM certificate for `app.example.com`.
4. Create a Route 53 alias `A` record for `app.example.com` to the ALB.
5. Verify the target group health checks are passing.

Example Route 53 record JSON-style representation:

```json
{
  "Name": "app.example.com",
  "Type": "A",
  "AliasTarget": {
    "DNSName": "my-alb-123456.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": false,
    "HostedZoneId": "Z35SXDOTRQ7X7K"
  }
}
```

**Verification and troubleshooting**

After you configure DNS, verify resolution and HTTPS behavior.

Useful commands:

```bash
dig +short www.example.com
dig +short app.example.com
curl -I https://www.example.com
curl -I https://app.example.com
```

If you use Route 53 alias records, `dig` should return the AWS-managed target or its resolved address chain. `curl` should confirm that HTTPS works and that the certificate matches the hostname.

Common issues:

- **Certificate mismatch**: The hostname is not included in the certificate SANs (Subject Alternative Names).
- **Wrong region for CloudFront certificate**: The certificate is not in `us-east-1`.
- **Missing alternate domain name**: CloudFront does not recognize the custom hostname.
- **Health check failure**: The ALB target group has unhealthy targets.
- **DNS propagation delay**: The record exists but resolvers have not refreshed yet.
- **Using CNAME at the apex**: Standard DNS does not allow it; use an alias record instead.

Example troubleshooting checklist:

1. Confirm the DNS record points to the correct AWS endpoint.
2. Confirm the certificate covers the hostname.
3. Confirm the service is configured to accept the hostname.
4. Confirm backend targets are healthy.
5. Confirm security groups and network ACLs allow traffic.

**Security and operational best practices**

Use HTTPS everywhere and redirect HTTP to HTTPS when possible. Keep certificates renewed automatically through ACM when AWS manages them. Prefer Route 53 alias records for AWS resources because they simplify management and avoid stale IP addresses.

Operationally, remember that CloudFront and ALB solve different problems:

- CloudFront improves global delivery and can reduce origin load.
- ALB distributes traffic across targets in one region.
- You can combine them by placing CloudFront in front of an ALB when you want both edge caching and regional application routing.

**Exam focus**

For the exam, focus on these distinctions:

- CloudFront uses a global distribution and requires an ACM certificate in `us-east-1`.
- ALB is regional and uses a certificate in the same region.
- Route 53 alias records are the standard way to point apex domains to AWS resources.
- CloudFront needs alternate domain names configured on the distribution.
- ALB needs an HTTPS listener and healthy target groups.

If a question asks how to expose a website at `example.com` with low latency worldwide, CloudFront is usually the best answer. If it asks how to route traffic to application instances in one region with host-based or path-based rules, ALB is usually the best answer.
