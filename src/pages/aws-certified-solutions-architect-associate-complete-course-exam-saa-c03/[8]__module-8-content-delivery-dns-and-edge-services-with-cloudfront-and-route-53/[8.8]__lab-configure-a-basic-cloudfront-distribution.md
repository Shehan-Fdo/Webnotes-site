## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**CloudFront fundamentals for the lab**

Amazon CloudFront is a <abbr title="Content Delivery Network">CDN</abbr> that caches content at edge locations to reduce latency and offload origins. In this lab, you typically connect CloudFront to an Amazon S3 bucket, then verify that the distribution serves content through the CloudFront domain rather than directly from S3.

For exam prep, focus on the relationship between:
- the **origin** that stores content,
- the **distribution** that exposes content globally,
- the **cache behavior** that controls how requests are handled,
- and the **security controls** that keep a private S3 bucket private.

A basic CloudFront lab is usually about proving three things:
- CloudFront can fetch content from the origin.
- The origin is not publicly exposed when using private S3 access.
- Cache and invalidation behavior match what you expect during testing.

**Typical architecture**

A common setup looks like this:

```text
Viewer -> CloudFront distribution -> S3 bucket origin
```

If you use a private S3 origin, CloudFront should access the bucket through an <abbr title="Origin Access Control">OAC</abbr> or, in older designs, an Origin Access Identity (OAI). For current best practice, prefer OAC.

**Creating the S3 origin**

Start with an S3 bucket that contains a simple test object such as `index.html` or `lab.txt`. Upload a file that makes it easy to confirm whether CloudFront is serving the latest version.

Example upload:

```bash
aws s3 cp index.html s3://my-lab-cloudfront-bucket/index.html
aws s3 ls s3://my-lab-cloudfront-bucket/
```

If you want a simple website-style test object:

```html
<!doctype html>
<html>
  <body>
    <h1>CloudFront Lab</h1>
    <p>Version 1</p>
  </body>
</html>
```

Keep the object name stable during testing so you can observe cache hits, stale content, and invalidation behavior.

**Private S3 origin and access control**

For a secure lab, the S3 bucket should not be public. You usually do this by enabling S3 Block Public Access and then granting CloudFront permission through a bucket policy tied to the distribution.

Key points:
- **Block Public Access** should remain enabled at the bucket level unless the lab explicitly requires public access.
- The bucket policy should allow the CloudFront distribution to read objects.
- The policy should reference the distribution ARN or the CloudFront service principal, depending on the access model you use.
- OAC signs requests to S3 using SigV4, which is the preferred modern approach.

A representative bucket policy for OAC looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontServicePrincipalReadOnly",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-lab-cloudfront-bucket/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/E123ABC456DEF"
        }
      }
    }
  ]
}
```

Practical verification steps:
- Confirm the bucket is not publicly readable.
- Confirm the policy allows only the intended distribution.
- Test the S3 object URL directly; it should fail if the bucket is private.
- Test the CloudFront URL; it should succeed if OAC and the policy are correct.

> If the S3 object is publicly accessible, the lab may still “work,” but you lose the security lesson. For exam scenarios, assume the intended design is private-by-default.

**Step-by-step: create and attach an Origin Access Control**

Use the CloudFront console to create OAC and attach it to the origin.

1. Open the CloudFront console and choose your distribution or create a new one.
2. In the origin settings, select the S3 bucket origin.
3. Choose to create a new **Origin Access Control**.
4. Set the signing behavior to sign requests.
5. Set the signing protocol to SigV4.
6. Save the origin settings.
7. Update the S3 bucket policy to allow the CloudFront distribution ARN.
8. Confirm S3 Block Public Access remains enabled.
9. Deploy the distribution and wait for the status to become `Deployed`.

When you attach OAC, CloudFront uses signed requests to access S3. That means the bucket can stay private while CloudFront still reads objects.

**CloudFront distribution settings**

A basic distribution usually needs:
- an origin pointing to S3,
- a default cache behavior,
- a viewer protocol policy,
- and optional alternate domain and certificate settings if you use a custom domain.

Important cache behavior fields:
- **Viewer protocol policy**: controls whether HTTP is redirected to HTTPS, allowed, or blocked.
- **Allowed HTTP methods**: for static content, `GET`, `HEAD`, and sometimes `OPTIONS` are enough.
- **Cache policy**: defines what headers, cookies, and query strings affect caching.
- **Origin request policy**: controls what CloudFront forwards to the origin.
- **TTL settings**: minimum, default, and maximum TTL determine how long objects stay cached.

A practical default cache behavior for a static S3 lab often uses:
- `GET`, `HEAD`
- redirect HTTP to HTTPS
- a managed cache policy such as `CachingOptimized`
- no unnecessary query string or cookie forwarding

**TTL behavior and why it matters in the lab**

TTL means <abbr title="Time To Live">Time To Live</abbr>. CloudFront uses TTL values to decide how long an object remains in cache before it checks the origin again.

The three TTL values matter:
- **Minimum TTL**: the shortest time CloudFront will cache an object.
- **Default TTL**: the fallback cache duration when the origin does not specify caching headers.
- **Maximum TTL**: the upper limit CloudFront will honor.

Why this matters during testing:
- If TTL is long, you may keep seeing old content after updating S3.
- If TTL is short, changes appear faster but origin requests increase.
- If the origin sends cache headers, CloudFront may use them within the configured limits.

For lab validation, you often want one of these approaches:
- wait for TTL expiration,
- or create an invalidation for the changed object.

Example cache-control header for a test object:

```bash
aws s3 cp index.html s3://my-lab-cloudfront-bucket/index.html \
  --cache-control "max-age=60"
```

This can help you observe cache refresh behavior in a controlled way.

**Invalidations**

An invalidation tells CloudFront to remove cached copies of specific objects so the next request fetches fresh content from the origin.

Use invalidations when:
- you need immediate propagation of a changed file,
- you are testing a new version and do not want to wait for TTL expiration,
- you changed a file with the same name, such as `index.html`.

Wait for TTL expiration when:
- the change is not urgent,
- you want to observe normal cache lifecycle behavior,
- you are minimizing invalidation usage.

Example invalidation command:

```bash
aws cloudfront create-invalidation \
  --distribution-id E123ABC456DEF \
  --paths "/index.html"
```

Practical notes:
- Invalidation paths are path-based, not bucket-based.
- `/*` invalidates everything and is useful for small labs, but it is broader than necessary.
- Invalidations are not instantaneous; they usually complete quickly, but you should still check status.

**Verifying the distribution**

After deployment, test the CloudFront domain name, not the S3 URL.

Useful checks:
- Open the CloudFront domain in a browser.
- Use `curl -I` to inspect headers.
- Refresh the page after updating the origin to observe cache behavior.

Example:

```bash
curl -I https://d123example.cloudfront.net/index.html
```

Look for headers such as:
- `x-cache`
- `age`
- `via`
- `x-amz-cf-pop`
- `x-amz-cf-id`

Interpretation:
- `x-cache: Hit from cloudfront` means the object came from cache.
- `x-cache: Miss from cloudfront` means CloudFront fetched from the origin.
- `age` shows how long the object has been cached.
- `via` confirms the request passed through CloudFront.

**Common lab errors and how to read them**

CloudFront error codes often point directly to the misconfiguration.

- **403 Forbidden**
  - Usually means access is denied.
  - Common causes:
    - S3 bucket policy does not allow the distribution.
    - OAC is missing or not attached correctly.
    - The object exists but CloudFront cannot read it.
    - You requested a private S3 URL directly.
  - In a private-origin lab, 403 is often a permissions problem, not a missing file.

- **404 Not Found**
  - Usually means the object path is wrong or the file does not exist.
  - Common causes:
    - Wrong key name in S3.
    - Missing `index.html`.
    - Incorrect default root object.
  - If CloudFront returns 404, verify the object key in S3 first.

- **502 Bad Gateway**
  - Usually means CloudFront could not get a valid response from the origin.
  - Common causes:
    - Origin hostname is wrong.
    - S3 origin configuration is malformed.
    - Custom origin TLS or certificate problems.
    - ALB origin is unhealthy or misconfigured.

- **504 Gateway Timeout**
  - Usually means the origin did not respond in time.
  - Common causes:
    - Slow custom origin.
    - Network or listener issues.
    - Security group or routing problems for an ALB origin.

A quick troubleshooting rule:
- **403** = permissions or access control
- **404** = object missing or wrong path
- **502/504** = origin connectivity or origin response problem

**Custom origin with an ALB**

Some labs use an Application Load Balancer (ALB) instead of S3. In that case, CloudFront acts as a CDN in front of a custom origin.

For an ALB origin, make sure:
- the ALB has a listener on `80` or `443`,
- the target group is healthy,
- the security group allows traffic from CloudFront or from the expected network path,
- the origin protocol policy matches the listener setup.

If you use HTTPS to the ALB:
- the ALB needs a valid certificate from AWS Certificate Manager (ACM),
- the certificate must match the ALB DNS name or your custom domain,
- CloudFront origin protocol policy should be set to `HTTPS only` or `Match viewer` depending on the design.

If you use HTTP to the ALB:
- the origin protocol policy can be `HTTP only`,
- but this is less secure and usually not preferred for production.

Practical guidance:
- Use HTTPS between CloudFront and ALB when possible.
- Use HTTP only for simple labs where TLS is not the focus.
- Confirm the ALB responds directly before adding CloudFront.

**Console workflow for a basic distribution**

1. Create or choose an S3 bucket with a test object.
2. Keep the bucket private and enable Block Public Access.
3. Create a CloudFront distribution.
4. Select the S3 bucket as the origin.
5. Create and attach an OAC.
6. Update the bucket policy to allow the distribution.
7. Configure the default cache behavior.
8. Set the default root object if you are serving a website-style file.
9. Deploy the distribution.
10. Test the CloudFront domain with a browser or `curl`.
11. Update the origin object and observe cache behavior.
12. Create an invalidation if you need immediate refresh.

**CLI example for a basic distribution structure**

The AWS CLI can create distributions, but the request is verbose. For exam prep, it helps to understand the shape of the configuration even if you use the console in the lab.

Example skeleton:

```json
{
  "CallerReference": "lab-cloudfront-001",
  "Comment": "Basic CloudFront distribution for S3 origin",
  "Enabled": true,
  "Origins": {
    "Quantity": 1,
    "Items": [
      {
        "Id": "S3-my-lab-cloudfront-bucket",
        "DomainName": "my-lab-cloudfront-bucket.s3.us-east-1.amazonaws.com",
        "S3OriginConfig": {
          "OriginAccessIdentity": ""
        },
        "OriginAccessControlId": "E2ABCDEF123456"
      }
    ]
  },
  "DefaultCacheBehavior": {
    "TargetOriginId": "S3-my-lab-cloudfront-bucket",
    "ViewerProtocolPolicy": "redirect-to-https",
    "AllowedMethods": {
      "Quantity": 2,
      "Items": ["GET", "HEAD"],
      "CachedMethods": {
        "Quantity": 2,
        "Items": ["GET", "HEAD"]
      }
    },
    "Compress": true,
    "CachePolicyId": "658327ea-f89d-4fab-a63d-7e88639e58f6"
  },
  "PriceClass": "PriceClass_100",
  "HttpVersion": "http2",
  "IsIPV6Enabled": true
}
```

This structure highlights the fields you should recognize:
- **Origins** define where content comes from.
- **DefaultCacheBehavior** defines how requests are handled.
- **ViewerProtocolPolicy** controls HTTP-to-HTTPS behavior.
- **CachePolicyId** can point to a managed policy.
- **PriceClass** affects edge location coverage and cost.

**Practical testing sequence**

1. Upload a file to S3.
2. Open the CloudFront URL and confirm the content appears.
3. Run `curl -I` and note the cache headers.
4. Change the file in S3.
5. Refresh the CloudFront URL immediately.
6. If you still see old content, check TTL or create an invalidation.
7. Confirm the new version appears after invalidation completes.

**What to remember for the exam**

CloudFront is often the right answer when you need:
- lower latency for global users,
- caching for static content,
- HTTPS delivery at the edge,
- private S3 origins with OAC,
- or a front door for custom origins like ALB.

For SAA-C03, be ready to distinguish:
- **CloudFront vs S3 static website hosting**
- **OAC vs public bucket access**
- **TTL vs invalidation**
- **S3 origin vs ALB custom origin**
- **403 vs 404 vs 502/504**

A strong answer usually includes both performance and security:
- cache content close to users,
- keep the origin private when possible,
- and use the right origin type for the workload.
