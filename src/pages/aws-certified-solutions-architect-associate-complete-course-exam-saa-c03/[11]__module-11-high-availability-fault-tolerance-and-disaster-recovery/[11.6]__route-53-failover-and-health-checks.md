## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Route 53 failover routing overview**

Amazon Route 53 is a DNS service, so its failover behavior is based on DNS answers, not packet-level traffic steering. You use failover routing when you want Route 53 to return a primary record under normal conditions and a secondary record when the primary is unhealthy.

Failover routing and health checks are related but separate topics:

- **Failover routing** decides which DNS record Route 53 returns.
- **Health checks** determine whether an endpoint, resource, or alarm is considered healthy.
- You can use health checks with failover routing, but health checks also work independently for monitoring and automation.

A common exam pattern is a user-facing application with a primary Region, a standby Region, and a requirement to redirect DNS only when the primary is unavailable.

**Core failover record behavior**

Failover routing uses two records with the same name and type:

- A **primary** record
- A **secondary** record

Route 53 returns the primary record when it is healthy. If the primary is unhealthy, Route 53 returns the secondary record.

Typical use cases include:

- Active/passive multi-Region web applications
- Disaster recovery for a public website
- Failover from AWS to on-premises
- Failover from an application load balancer to a static maintenance page

Important constraints:

- Failover routing is a DNS decision, so clients may cache the previous answer until TTL expires.
- DNS failover is not instant for every client because recursive resolvers cache responses.
- You should keep TTL low enough to support acceptable failover time, but not so low that it creates unnecessary DNS query volume.

**Health checks: what they monitor**

Route 53 health checks evaluate whether a target is reachable and behaving as expected. They can monitor:

- An **endpoint** by IP address or domain name
- A **CloudWatch alarm**
- A **calculated health check** based on other health checks

For endpoint checks, Route 53 sends probes from multiple AWS health checker locations around the world. You can scope checks to specific regions or use global checking behavior depending on the configuration.

Health checks are useful when you need:

- DNS failover based on application availability
- Monitoring of a public endpoint
- Alarm-driven failover for private or internal systems via CloudWatch

> Route 53 health checks do not directly probe private-only endpoints on a private network. If the target is not publicly reachable, use a publicly reachable proxy, a CloudWatch alarm-based health check, or a calculated health check pattern.

**Health check types and when to use them**

| Type | What it checks | Best use case |
|---|---|---|
| Endpoint | HTTP, HTTPS, TCP, or string match against a public target | Public web apps, load balancers, websites |
| CloudWatch alarm | Alarm state | Private services, custom application health, internal dependencies |
| Calculated | Combines child health checks | Composite availability logic, multi-component systems |

Endpoint checks are the most common for SAA-C03 exam scenarios. CloudWatch alarm checks are important when the service is not directly reachable from the internet.

**Endpoint health check configuration**

A Route 53 endpoint health check is usually defined with one target type:

- `IPAddress`
- or `FullyQualifiedDomainName`

You normally choose one target style, not both, because the check is meant to probe a single resolved endpoint.

Common options include:

- `Port`
- `Type` such as `HTTP`, `HTTPS`, or `TCP`
- `ResourcePath` for HTTP/S checks
- `SearchString` for content verification
- `FailureThreshold`
- `RequestInterval`
- `MeasureLatency`
- `Inverted`
- `Disabled`
- `Regions`

Example CLI for a public HTTP health check:

```bash
aws route53 create-health-check \
  --caller-reference "webapp-primary-20260426" \
  --health-check-config '{
    "IPAddress":"203.0.113.10",
    "Port":80,
    "Type":"HTTP",
    "ResourcePath":"/health",
    "FailureThreshold":3,
    "RequestInterval":30
  }'
```

Example for a domain-based HTTPS check:

```bash
aws route53 create-health-check \
  --caller-reference "api-primary-20260426" \
  --health-check-config '{
    "FullyQualifiedDomainName":"api.example.com",
    "Port":443,
    "Type":"HTTPS",
    "ResourcePath":"/status",
    "FailureThreshold":3,
    "RequestInterval":30
  }'
```

Practical notes:

- Use `HTTP` or `HTTPS` when you want Route 53 to verify application behavior, not just TCP reachability.
- Use `TCP` when you only need port-level availability.
- Use `SearchString` when the endpoint returns a predictable body and you want to confirm the correct application response.
- Use `MeasureLatency` when you want latency data in addition to health status.

**Health determination and failure behavior**

Route 53 health checks are evaluated by multiple health checkers in different locations. The service uses the configured failure threshold and its internal health evaluation logic to decide whether the endpoint is healthy.

Key behaviors:

- A check is not marked unhealthy after a single transient failure if the failure threshold is higher than 1.
- `FailureThreshold` helps reduce false positives from brief network issues.
- `RequestInterval` controls how often Route 53 probes the endpoint.
- `Inverted` flips the meaning of healthy and unhealthy, which is useful in special routing patterns.

Do not think of this as a simple generic quorum model. The practical exam takeaway is that Route 53 uses distributed health checking and configurable thresholds to avoid reacting to one-off failures.

**Health check status, metrics, and alarms**

Route 53 health checks integrate with monitoring and alerting:

- Each health check has a status page in the Route 53 console.
- Health check status is published as a CloudWatch metric.
- You can create CloudWatch alarms on health check metrics.
- CloudWatch alarm-based health checks let Route 53 treat an alarm state as unhealthy.

This is useful when you want:

- Operations visibility into endpoint health
- Alerting before or during failover
- A health signal for private resources that cannot be probed directly

Example CloudWatch alarm pattern:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "PrimaryAppHealthAlarm" \
  --metric-name "HealthCheckStatus" \
  --namespace "AWS/Route53" \
  --statistic Minimum \
  --period 60 \
  --evaluation-periods 3 \
  --threshold 1 \
  --comparison-operator LessThanThreshold \
  --dimensions Name=HealthCheckId,Value=abcd1234-ef56-7890-abcd-1234567890ab
```

**Calculated health checks**

Calculated health checks combine the status of child health checks. They are useful when one application depends on multiple components.

Common patterns:

- Mark the service healthy only if all critical components are healthy.
- Mark the service healthy if at least one of several endpoints is healthy.
- Combine regional checks into a higher-level availability signal.

Example use cases:

- A web app depends on an API and a database proxy
- A multi-AZ service needs both front-end and back-end checks
- A failover decision should reflect multiple health signals

Calculated checks are especially useful when you need custom logic that Route 53 endpoint probing alone cannot express.

**Failover routing record setup**

Failover routing requires two records with the same name and type, but different failover roles.

Typical setup:

- Primary record: points to the active endpoint
- Secondary record: points to the standby endpoint
- Health check: attached to the primary record, and sometimes also to the secondary if you want independent validation

Example DNS topology:

- `www.example.com` primary → ALB in `us-east-1`
- `www.example.com` secondary → ALB in `us-west-2`

If the primary health check fails, Route 53 returns the secondary answer.

Example Route 53 record set JSON conceptually:

```json
{
  "Name": "www.example.com",
  "Type": "A",
  "SetIdentifier": "primary-us-east-1",
  "Failover": "PRIMARY",
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "dualstack.primary-alb-123456.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  },
  "HealthCheckId": "abcd1234-ef56-7890-abcd-1234567890ab"
}
```

```json
{
  "Name": "www.example.com",
  "Type": "A",
  "SetIdentifier": "secondary-us-west-2",
  "Failover": "SECONDARY",
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "dualstack.secondary-alb-987654.us-west-2.elb.amazonaws.com",
    "EvaluateTargetHealth": false
  }
}
```

**Alias records and Evaluate Target Health**

Route 53 alias records can point to AWS resources without charging for DNS queries to the alias target. For failover scenarios, alias records are often used with load balancers, CloudFront, API Gateway, and some other AWS targets.

`EvaluateTargetHealth` tells Route 53 to consider the health of the alias target itself when deciding whether to answer with that record.

Important exam points:

- `EvaluateTargetHealth` is supported for certain AWS resources, not all targets.
- It is commonly used with Application Load Balancers, Network Load Balancers, and Classic Load Balancers.
- It is also supported for some AWS-managed endpoints such as CloudFront and API Gateway in specific alias scenarios.
- It is not a universal health signal for every DNS target.

For S3, the distinction matters:

- Route 53 alias records support **S3 static website hosting endpoints**
- They do not alias to a general S3 bucket endpoint for website failover in the same way
- This is a common exam trap

**Failover routing versus health checks**

These are separate concepts and the exam may test them independently.

- **Failover routing**
  - A DNS routing policy
  - Returns primary or secondary answers
  - Used for active/passive designs

- **Health checks**
  - A monitoring and health evaluation feature
  - Can drive failover routing
  - Can also be used for alarms, dashboards, and calculated checks

You can create a health check without using failover routing. For example, you might monitor a public endpoint and alert on failures without changing DNS.

**Private hosted zone limitations**

Route 53 health checks generally require a publicly reachable endpoint when checking an endpoint directly.

Implications:

- A private hosted zone does not make a private endpoint health-checkable from Route 53 public health checkers.
- If the service is internal-only, use a CloudWatch alarm-based health check or a calculated health check.
- If you need DNS failover for a private service, you usually need a different design, such as internal load balancer health signals combined with automation.

This is a common architecture question: Route 53 can host private DNS names, but its public health checkers cannot directly probe private RFC 1918-only addresses.

**Common exam scenarios**

1. **Failover to a standby ALB**
   - Primary ALB serves the main Region.
   - Secondary ALB serves the disaster recovery Region.
   - Route 53 failover routing returns the secondary ALB when the primary health check fails.

2. **Failover to on-premises**
   - Primary application runs in AWS.
   - Secondary endpoint is an on-premises web server or reverse proxy.
   - Route 53 health checks monitor the public endpoint of the on-premises service.

3. **Static website failover**
   - Primary site is hosted on an S3 static website endpoint.
   - Secondary site is another static website or maintenance page.
   - Use alias records where supported for the static website endpoint.

4. **Private application with custom health**
   - The app is not publicly reachable.
   - A CloudWatch alarm reflects application health.
   - Route 53 uses the alarm state to decide whether to fail over.

5. **Multi-component service**
   - Front end, API, and dependency checks all matter.
   - A calculated health check combines child checks.
   - DNS failover occurs only when the overall service is unhealthy.

**Console workflow for an end-to-end setup**

1. Create or identify the primary and secondary endpoints.
2. Create a Route 53 health check for the primary endpoint.
3. Verify the health check status in the Route 53 console.
4. Create a failover record set for the primary name and type.
5. Attach the health check to the primary record.
6. Create the secondary failover record with the same name and type.
7. Test by simulating failure on the primary endpoint.
8. Confirm that DNS answers switch after TTL expiration.

**Practical console considerations**

- In the Route 53 console, health checks appear under the health checks section with status and configuration details.
- Record sets must match on name and type for failover routing.
- Alias records are often preferred for AWS resources because they avoid an extra DNS lookup and can integrate with AWS target health.
- You should test failover from the client side, not only by looking at the console status.

**Troubleshooting and design tradeoffs**

- If failover seems slow, check TTL values and client DNS caching.
- If a health check never turns healthy, verify that the endpoint is publicly reachable and that security groups, NACLs, and firewalls allow the probe.
- If the endpoint is private, do not expect Route 53 public health checkers to reach it directly.
- If the secondary record is never returned, confirm that the primary record actually has a failing health check and that the record set is configured as failover, not simple routing.
- If you use alias records, confirm whether the target type supports `EvaluateTargetHealth`.

> Route 53 failover is DNS-based resilience, not instant traffic switching. Design for DNS caching, public reachability of health checks, and the limitations of alias target support.

**Exam-focused summary**

- Use **failover routing** for active/passive DNS-based disaster recovery.
- Use **health checks** to evaluate endpoint availability, CloudWatch alarms, or calculated conditions.
- Remember that health checks can exist without failover routing.
- Prefer alias records for supported AWS targets, especially load balancers.
- Know the S3 static website endpoint distinction.
- Treat private endpoints carefully: direct Route 53 endpoint health checks are for public reachability, not private-only targets.
- Expect questions about TTL, DNS caching, and the difference between monitoring and routing.
