## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Disaster recovery design goals**

A disaster recovery (DR) architecture aims to restore service after a failure with acceptable recovery time and data loss. In AWS, you usually balance four factors:

- **RTO**: Recovery Time Objective, the maximum acceptable downtime.
- **RPO**: Recovery Point Objective, the maximum acceptable data loss.
- **Cost**: More redundancy and replication usually increase spend.
- **Complexity**: Faster recovery often requires more automation and operational maturity.

For a sample web application, you typically separate the stack into layers:

- **DNS and traffic management** with Amazon Route 53.
- **Application tier** on Amazon EC2, Auto Scaling, or containers.
- **Data tier** on Amazon RDS, Amazon Aurora, Amazon DynamoDB, or S3.
- **State and artifacts** such as uploads, logs, and AMIs.
- **Automation** with CloudFormation, Systems Manager, Lambda, or scripts.

A good DR design is not just “copy everything to another Region.” It defines what must be restored, how quickly, and what can be rebuilt from code or automation.

**Common DR strategies**

AWS exam questions often map to one of these patterns:

| Strategy | Typical use | RTO | RPO | Notes |
|---|---|---:|---:|---|
| Backup and restore | Lowest cost, non-critical workloads | High | High | Restore from snapshots, backups, or exports |
| Pilot light | Core services running in secondary Region | Medium | Low to medium | Keep minimal data and infrastructure ready |
| Warm standby | Scaled-down full stack in secondary Region | Low | Low | Faster failover, higher cost |
| Multi-site active/active | Highest availability | Very low | Very low | Complex, expensive, and data-layer dependent |

For a sample web app, pilot light or warm standby is often the best exam answer when you need a realistic balance of cost and recovery speed.

**Route 53 traffic management**

Amazon Route 53 is the DNS service you use to direct users to healthy endpoints. It supports several routing policies that matter for DR:

- **Failover routing** sends traffic to a primary endpoint and switches to a secondary endpoint when the primary is unhealthy.
- **Latency-based routing** sends users to the Region with the lowest network latency.
- **Weighted routing** splits traffic across endpoints by percentage.
- **Geolocation routing** routes based on user location.
- **Multivalue answer routing** returns multiple healthy records for simple client-side selection.

For DR, failover routing is the clearest choice when you want automatic primary-to-secondary switching. Latency-based routing is useful for active/active designs, but it does not provide the same explicit failover semantics as failover routing.

> Route 53 health checks evaluate the endpoint you configure. For alias records, health behavior depends on the target type and whether AWS can evaluate target health; it is not the same as attaching a separate health check to a non-alias record.

A practical DNS setup often looks like this:

```text
example.com
├── Primary: ALB in us-east-1
└── Secondary: ALB in us-west-2
```

A failover record set can be created with the AWS CLI:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [
      {
        "Action": "UPSERT",
        "ResourceRecordSet": {
          "Name": "app.example.com",
          "Type": "A",
          "SetIdentifier": "primary-us-east-1",
          "Failover": "PRIMARY",
          "AliasTarget": {
            "HostedZoneId": "Z35SXDOTRQ7X7K",
            "DNSName": "dualstack.my-alb-123456.us-east-1.elb.amazonaws.com",
            "EvaluateTargetHealth": true
          }
        }
      },
      {
        "Action": "UPSERT",
        "ResourceRecordSet": {
          "Name": "app.example.com",
          "Type": "A",
          "SetIdentifier": "secondary-us-west-2",
          "Failover": "SECONDARY",
          "AliasTarget": {
            "HostedZoneId": "Z1H1FL5HABSF5",
            "DNSName": "dualstack.my-alb-789012.us-west-2.elb.amazonaws.com",
            "EvaluateTargetHealth": true
          }
        }
      }
    ]
  }'
```

**Application tier design**

The application layer should be easy to recreate in another Region. The best practice is to treat instances as disposable and keep configuration in code.

- Use **Auto Scaling groups** so you can recreate capacity quickly.
- Use **Launch Templates** instead of manually configured instances.
- Store application code in a deployment pipeline or artifact repository.
- Keep configuration in user data, environment variables, or parameter stores.
- Avoid relying on local instance storage for critical state.

For DR, the application tier often uses one of these patterns:

- **Rebuild from AMI and user data** in the secondary Region.
- **Pre-provisioned instances** in a warm standby.
- **Immutable deployments** using a pipeline that can deploy to both Regions.

A simple launch template example:

```json
{
  "LaunchTemplateName": "webapp-lt",
  "LaunchTemplateData": {
    "ImageId": "ami-0123456789abcdef0",
    "InstanceType": "t3.micro",
    "IamInstanceProfile": {
      "Name": "webapp-instance-profile"
    },
    "SecurityGroupIds": ["sg-0123456789abcdef0"],
    "UserData": "IyEvYmluL2Jhc2gKc3lzdGVtY3RsIGVuYWJsZSAtLW5vdyBodHRwZAo="
  }
}
```

That `UserData` value is base64-encoded, which is the format EC2 expects in launch templates. In practice, you usually generate it from a script or CloudFormation `Fn::Base64`.

A safer bootstrap script should be idempotent and avoid assumptions about package managers:

```bash
#!/bin/bash
set -euo pipefail

if command -v dnf >/dev/null 2>&1; then
  dnf install -y nginx
elif command -v yum >/dev/null 2>&1; then
  yum install -y nginx
elif command -v apt-get >/dev/null 2>&1; then
  apt-get update
  apt-get install -y nginx
fi

systemctl enable nginx
systemctl start nginx
```

**CloudFormation pattern for repeatable recovery**

Infrastructure as code helps you recreate the stack in a second Region with minimal manual work. A common pattern is to parameterize Region-specific values and deploy the same template twice.

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Web app DR stack

Parameters:
  AmiId:
    Type: String
  InstanceType:
    Type: String
    Default: t3.micro

Resources:
  WebServerRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              Service: ec2.amazonaws.com
            Action: sts:AssumeRole

  WebServerProfile:
    Type: AWS::IAM::InstanceProfile
    Properties:
      Roles:
        - !Ref WebServerRole

  WebLaunchTemplate:
    Type: AWS::EC2::LaunchTemplate
    Properties:
      LaunchTemplateData:
        ImageId: !Ref AmiId
        InstanceType: !Ref InstanceType
        IamInstanceProfile:
          Name: !Ref WebServerProfile
        UserData:
          Fn::Base64: !Sub |
            #!/bin/bash
            set -euo pipefail
            echo "Hello from ${AWS::Region}" > /var/www/html/index.html
```

This pattern is useful because you can deploy the same template in the secondary Region and only change the AMI or parameters as needed.

**Database recovery options**

The database tier usually determines your RPO. Different services offer different DR capabilities:

- **Amazon RDS automated backups** support point-in-time restore within the backup retention window.
- **Manual snapshots** are useful for long-term retention and cross-Region copy.
- **Amazon Aurora Global Database** supports low-latency replication across Regions for fast disaster recovery.
- **Read replicas** can reduce recovery time, but they are not always a full DR solution by themselves.
- **Amazon DynamoDB global tables** provide multi-Region active/active replication for NoSQL workloads.

For a relational web app, common DR choices include:

- **Single-Region RDS with cross-Region snapshot copy** for backup/restore.
- **Cross-Region read replica** for faster promotion.
- **Aurora Global Database** when low RPO and fast failover matter.

A cross-Region snapshot copy example:

```bash
aws rds copy-db-snapshot \
  --source-region us-east-1 \
  --source-db-snapshot-identifier arn:aws:rds:us-east-1:123456789012:snapshot:prod-db-snap-20260426 \
  --target-db-snapshot-identifier prod-db-snap-copy-us-west-2 \
  --kms-key-id arn:aws:kms:us-west-2:123456789012:key/11111111-2222-3333-4444-555555555555
```

Important considerations:

- The source snapshot must be accessible from the source Region.
- If the snapshot is encrypted, the destination Region needs a valid KMS key.
- Cross-Region copy is slower than replication, so it fits backup/restore or pilot light patterns better than active/active.

For Aurora, you may use a global database or a cross-Region snapshot restore depending on the target RTO and RPO.

**Storage and static content**

Static assets and user uploads often live outside the application server. That makes them easier to protect and restore.

- Use **Amazon S3** for static website assets, uploads, and backups.
- Enable **versioning** to protect against accidental deletion or overwrite.
- Use **cross-Region replication** when you need a second copy in another Region.
- Use **S3 lifecycle policies** to move older backups to cheaper storage classes.
- Consider **S3 Object Lock** for compliance or ransomware resistance.

A simple replication setup requires:

- Versioning enabled on source and destination buckets.
- An IAM role for replication.
- A replication rule that targets the secondary bucket.

Example replication configuration:

```json
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "Status": "Enabled",
      "Priority": 1,
      "DeleteMarkerReplication": { "Status": "Disabled" },
      "Filter": { "Prefix": "" },
      "Destination": {
        "Bucket": "arn:aws:s3:::myapp-dr-bucket",
        "StorageClass": "STANDARD"
      }
    }
  ]
}
```

**Networking and load balancing**

A DR design usually includes a load balancer in each Region.

- Use an **Application Load Balancer** for HTTP/HTTPS traffic.
- Place the load balancer in public subnets.
- Keep application instances in private subnets.
- Use security groups to allow only required traffic.
- Mirror the VPC design in the secondary Region.

A common topology is:

```text
Users
  -> Route 53
    -> ALB in primary Region
      -> EC2 Auto Scaling group
        -> RDS primary
        -> S3 for static assets/backups

Failover path:
Users
  -> Route 53
    -> ALB in secondary Region
      -> EC2 Auto Scaling group
        -> restored or replicated database
        -> replicated S3 objects
```

If you use latency-based routing, both Regions should be able to serve traffic. If you use failover routing, the secondary stack can remain scaled down until needed.

**Backup and restore workflow**

Backup and restore is the simplest DR model and often the cheapest. It works well when downtime is acceptable and the application can tolerate a longer recovery window.

Typical steps:

1. Back up the database with automated backups or snapshots.
2. Copy snapshots to the secondary Region if needed.
3. Replicate or back up S3 data.
4. Store infrastructure templates in version control.
5. Recreate the application stack in the secondary Region.
6. Restore the database and update DNS.

A restore command for RDS might look like this:

```bash
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier myapp-dr-db \
  --db-snapshot-identifier prod-db-snap-copy-us-west-2 \
  --db-instance-class db.t3.micro \
  --vpc-security-group-ids sg-0123456789abcdef0
```

This approach is slower, but it is often enough for non-critical workloads or exam scenarios that emphasize cost savings.

**Pilot light workflow**

Pilot light keeps the most critical components running in the secondary Region. Usually, that means:

- Database replication or frequent snapshot copies.
- Minimal application infrastructure.
- DNS and automation ready to scale up.

This pattern reduces recovery time compared with backup and restore because the core data layer is already present. You still need to scale out the application tier and possibly promote the database.

Use pilot light when:

- You need better RTO than backup and restore.
- You want to control cost.
- You can tolerate some manual or automated scaling during failover.

**Warm standby workflow**

Warm standby is a more complete version of pilot light. The secondary Region runs a smaller but functional copy of the application.

- Keep the secondary ALB active.
- Run a reduced Auto Scaling group.
- Maintain replicated or continuously synchronized data.
- Test failover regularly.

Warm standby is a strong choice when the application must recover quickly but full active/active is too expensive or complex.

**Failover runbook**

A DR plan should include a tested runbook. In a real incident, you want a repeatable sequence.

1. Confirm the primary Region is unavailable or degraded.
2. Stop writes to the primary database if possible.
3. Promote or restore the secondary database.
4. Scale up the application tier in the secondary Region.
5. Verify health checks on the secondary load balancer.
6. Update Route 53 or allow failover routing to shift traffic.
7. Validate application functionality and data integrity.
8. Communicate status to stakeholders.

A simple Systems Manager or Lambda automation can help reduce manual steps, but you should still understand the underlying sequence for the exam.

**Testing and validation**

DR plans fail when they are never tested. Regular validation should include:

- Restoring a database snapshot.
- Recreating the stack from templates.
- Verifying DNS failover behavior.
- Checking that static assets are available in the secondary Region.
- Confirming IAM permissions and KMS keys work in the target Region.

Useful CLI checks:

```bash
aws route53 test-dns-answer \
  --hosted-zone-id Z1234567890ABC \
  --record-name app.example.com \
  --record-type A
```

```bash
aws rds describe-db-snapshots \
  --db-snapshot-identifier prod-db-snap-copy-us-west-2
```

```bash
aws s3 ls s3://myapp-dr-bucket/
```

Testing should include both planned failover and unplanned failure scenarios. Planned failover is safer and helps you measure actual RTO.

**Security and operational considerations**

A DR architecture must preserve security controls in both Regions.

- Replicate **IAM** roles and policies as part of infrastructure as code.
- Recreate **security groups** and network ACLs consistently.
- Ensure **KMS** keys exist in each Region if you encrypt snapshots or objects.
- Store secrets in **AWS Secrets Manager** or **Systems Manager Parameter Store** and replicate them appropriately.
- Verify that logging and monitoring continue in the secondary Region.

You should also consider:

- **CloudWatch alarms** for health and failover triggers.
- **AWS Backup** for centralized backup policies.
- **AWS Config** for drift detection.
- **CloudTrail** for auditability during failover events.

**Exam-focused decision cues**

When you see a DR question, identify the required recovery objective first.

- If the question emphasizes **lowest cost**, choose backup and restore.
- If it emphasizes **faster recovery with moderate cost**, choose pilot light or warm standby.
- If it emphasizes **minimal downtime and automatic traffic shifting**, choose Route 53 failover with replicated or promoted backend services.
- If it emphasizes **multi-Region low-latency reads/writes**, consider active/active patterns such as latency-based routing or DynamoDB global tables.

Also watch for service-specific clues:

- **RDS snapshots** are for restore-based recovery.
- **Read replicas** can reduce recovery time but may require promotion.
- **Aurora Global Database** is designed for cross-Region disaster recovery.
- **S3 versioning and replication** protect static content and backups.
- **Route 53 failover** is the DNS layer for switching traffic.

**Practical sample architecture**

A balanced DR design for a sample web application might look like this:

- Primary Region hosts the live application behind an ALB.
- Database uses RDS with automated backups and cross-Region snapshot copy.
- Static assets are stored in S3 with versioning and replication.
- Secondary Region has the same VPC layout, ALB, launch template, and Auto Scaling group.
- Route 53 failover routing sends traffic to the secondary ALB if the primary becomes unhealthy.
- CloudFormation or Terraform recreates the stack consistently in both Regions.

This design gives you a realistic mix of cost control and recovery capability. It is also a strong exam pattern because it combines DNS failover, infrastructure as code, and data protection without assuming perfect active/active synchronization.

**Key takeaways**

- **RTO** and **RPO** drive every DR decision.
- Route 53 failover is the clearest DNS-based failover mechanism.
- Application servers should be rebuildable from code, not treated as pets.
- Database strategy usually determines how much data loss you can tolerate.
- S3 versioning and replication protect static content and backups.
- Testing is essential; a DR plan that is not tested is not reliable.
