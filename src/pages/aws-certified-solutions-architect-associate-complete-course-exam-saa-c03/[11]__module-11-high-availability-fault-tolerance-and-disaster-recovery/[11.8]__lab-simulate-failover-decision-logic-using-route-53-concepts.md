## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Route 53 failover routing overview**

Amazon Route 53 failover routing is a DNS-based high availability pattern that returns a **primary** record when it is healthy and a **secondary** record when the primary is unhealthy. You use it when you want active-passive behavior at the DNS layer, not load balancing across multiple endpoints.

Failover routing is evaluated per record set. Route 53 decides whether a record is eligible based on health checks, alias target health evaluation, and record configuration. This makes it useful for disaster recovery, regional failover, and simple service continuity designs.

- **Primary record** serves traffic during normal operation.
- **Secondary record** is returned only when the primary is considered unhealthy.
- **Health checks** determine endpoint status when you attach them to non-alias records or use them in calculated checks.
- **Alias records** can optionally use **Evaluate target health** when the target type supports it.

**Failover routing prerequisites and constraints**

Failover routing has a few exam-relevant rules that are easy to miss. You must create exactly one PRIMARY and one SECONDARY record for the same DNS name and record type when you want a classic failover pair. If you create more than one primary or more than one secondary for the same name/type combination, the design no longer matches the standard failover model.

Route 53 failover is not the same as application-level failover. DNS answers are cached by resolvers, so failover is not instantaneous. You should always consider TTL (time to live) and client caching when designing recovery behavior.

- **Same name and type** are required for the primary-secondary pair.
- **Exactly one PRIMARY** and **exactly one SECONDARY** are the standard failover setup.
- **TTL** affects how quickly clients observe the change.
- **DNS caching** can delay failover even after Route 53 changes its answer.

> Route 53 failover routing is a DNS decision mechanism, not a session-aware traffic manager. Existing client sessions may continue until they fail independently.

**Health checks and eligibility**

Route 53 health checks can probe endpoints using supported protocols such as HTTP, HTTPS, and TCP, and they can also use calculated health checks to combine the status of multiple checks. For HTTPS checks, Route 53 can validate the endpoint over TLS, and some configurations support Server Name Indication (SNI) behavior when the endpoint requires it.

A record becomes eligible only when Route 53 considers it healthy or when the record is configured to ignore health evaluation. For failover routing, the primary is preferred if healthy. If the primary is unhealthy, Route 53 evaluates the secondary. If both are unhealthy, Route 53 can return no answer for that name.

- **HTTP/HTTPS/TCP** checks validate endpoint reachability and response behavior.
- **Calculated checks** aggregate multiple health checks into one logical result.
- **Healthy primary** wins over secondary.
- **Unhealthy primary** causes Route 53 to consider the secondary.
- **Both unhealthy** can produce an empty DNS response for the failover pair.

**Alias records and Evaluate target health**

Alias records point to AWS resources such as Application Load Balancers, Network Load Balancers, CloudFront distributions, API Gateway, and some other AWS targets. For alias records, you can enable **Evaluate target health** only when the target service supports Route 53 health evaluation for that alias type.

This setting is service-specific. Do not assume every alias target behaves the same way. Some AWS resources support alias health evaluation, while others do not, and the behavior can differ by service and record type.

- **Alias record** removes the need for a separate CNAME at the zone apex.
- **Evaluate target health** lets Route 53 use the AWS resource’s health state when supported.
- **Service support varies** by alias target type.
- **Not all alias targets** can participate in health-based failover the same way.

**How Route 53 decides which record to return**

The decision logic is straightforward when you model it as a priority check:

1. Route 53 looks for the PRIMARY record for the queried name and type.
2. If the primary is healthy, Route 53 returns the primary answer.
3. If the primary is unhealthy, Route 53 checks whether the SECONDARY record is eligible.
4. If the secondary is healthy, Route 53 returns the secondary answer.
5. If neither record is eligible, Route 53 returns no answer for that failover pair.

This is different from weighted or latency routing, where multiple records may be returned or selected based on policy rather than a strict primary-secondary hierarchy.

- **Simple routing** returns a single record with no health-based decision tree unless you attach health checks.
- **Weighted routing** distributes responses by weight.
- **Latency routing** chooses the lowest-latency region.
- **Geolocation routing** answers based on user location.
- **Multivalue answer** can return multiple healthy records.

**End-to-end lab model**

A practical lab usually includes a hosted zone, a primary endpoint, a secondary endpoint, and a health signal for the primary. You can model this with two web servers, two load balancers, or one live endpoint and one standby endpoint.

A common architecture is:

- `example.com` as the hosted zone name.
- `www.example.com` as the failover record name.
- Primary endpoint in one Region.
- Secondary endpoint in another Region.
- Health check attached to the primary, or alias health evaluation enabled where supported.

**Example topology**

```text
Client
  |
  v
Route 53 hosted zone
  |
  +-- PRIMARY: www.example.com -> ALB in us-east-1
  |      health check / evaluate target health
  |
  +-- SECONDARY: www.example.com -> ALB in us-west-2
         healthy standby
```

**Console workflow for a failover lab**

Use the AWS Management Console when you want to visualize the record states and health transitions.

1. Create or open a hosted zone for your domain.
2. Create the primary record with routing policy set to **Failover** and role set to **Primary**.
3. Attach a health check to the primary record, or enable alias target health if the target supports it.
4. Create the secondary record with the same name and type, routing policy set to **Failover**, and role set to **Secondary**.
5. Verify that the secondary has no health check unless your design requires one for monitoring.
6. Test by making the primary unhealthy and confirming that Route 53 returns the secondary answer.

**CLI example: hosted zone and failover records**

The following example shows the general shape of a failover setup. Replace IDs, names, and hosted zone values with your own resources.

```bash
aws route53 create-hosted-zone \
  --name example.com \
  --caller-reference "$(date +%s)"
```

Create a health check for a single endpoint. Use either `IPAddress` or `FullyQualifiedDomainName` depending on how you want Route 53 to reach the target.

```bash
aws route53 create-health-check \
  --caller-reference "primary-web-1" \
  --health-check-config '{
    "IPAddress":"203.0.113.10",
    "Port":80,
    "Type":"HTTP",
    "ResourcePath":"/health",
    "RequestInterval":30,
    "FailureThreshold":3
  }'
```

Create the primary failover record:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Comment":"Primary failover record",
    "Changes":[
      {
        "Action":"UPSERT",
        "ResourceRecordSet":{
          "Name":"www.example.com",
          "Type":"A",
          "SetIdentifier":"primary-us-east-1",
          "Failover":"PRIMARY",
          "TTL":60,
          "HealthCheckId":"abcd1234-ef56-7890-ab12-cd34ef567890",
          "ResourceRecords":[
            {"Value":"203.0.113.10"}
          ]
        }
      }
    ]
  }'
```

Create the secondary failover record:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Comment":"Secondary failover record",
    "Changes":[
      {
        "Action":"UPSERT",
        "ResourceRecordSet":{
          "Name":"www.example.com",
          "Type":"A",
          "SetIdentifier":"secondary-us-west-2",
          "Failover":"SECONDARY",
          "TTL":60,
          "ResourceRecords":[
            {"Value":"198.51.100.20"}
          ]
        }
      }
    ]
  }'
```

**Alias failover example**

If you use alias records, the target hosted zone ID must match the actual AWS resource and Region. Do not reuse a hosted zone ID from another Region unless it is correct for that specific alias target.

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes":[
      {
        "Action":"UPSERT",
        "ResourceRecordSet":{
          "Name":"app.example.com",
          "Type":"A",
          "SetIdentifier":"primary-alb",
          "Failover":"PRIMARY",
          "AliasTarget":{
            "HostedZoneId":"Z35SXDOTRQ7X7K",
            "DNSName":"dualstack.primary-alb-123456789.us-east-1.elb.amazonaws.com",
            "EvaluateTargetHealth":true
          }
        }
      },
      {
        "Action":"UPSERT",
        "ResourceRecordSet":{
          "Name":"app.example.com",
          "Type":"A",
          "SetIdentifier":"secondary-alb",
          "Failover":"SECONDARY",
          "AliasTarget":{
            "HostedZoneId":"Z2P70J7EXAMPLE",
            "DNSName":"dualstack.secondary-alb-987654321.us-west-2.elb.amazonaws.com",
            "EvaluateTargetHealth":true
          }
        }
      }
    ]
  }'
```

**Testing failover behavior**

You should test both the health signal and the DNS answer. A good lab includes a before-and-after comparison so you can see the decision change.

1. Query the record while the primary is healthy.
2. Confirm that the answer resolves to the primary endpoint.
3. Mark the primary unhealthy by stopping the service, blocking the health check path, or disabling the target.
4. Wait for the health check to fail based on its interval and threshold.
5. Query the record again and confirm that Route 53 returns the secondary endpoint.
6. Restore the primary and verify that Route 53 switches back after the health check recovers.

Example DNS queries:

```bash
dig www.example.com +short
nslookup www.example.com
```

Expected behavior:

- Healthy primary: DNS answer points to the primary endpoint.
- Unhealthy primary, healthy secondary: DNS answer points to the secondary endpoint.
- Both unhealthy: DNS may return no answer for the failover pair.

**Interpreting health states**

Route 53 health checks are not binary in the way application logs are. A check can be healthy, unhealthy, or in transition depending on timing and thresholds. When you troubleshoot, always account for the delay between the actual outage and the health check result.

- **FailureThreshold** controls how many consecutive failures are needed.
- **RequestInterval** controls how often Route 53 probes.
- **TTL** controls how long resolvers cache the answer.
- **Transition time** is the combined effect of probe timing and DNS caching.

**Troubleshooting checklist**

If failover does not behave as expected, verify the following:

- The primary and secondary records have the same name and type.
- The failover roles are exactly one PRIMARY and one SECONDARY.
- The health check is attached to the correct record.
- The endpoint actually fails the health check path, not just the application homepage.
- The alias target supports **Evaluate target health** if you rely on it.
- The TTL is low enough for your recovery objective.
- The resolver cache has expired before you retest.

**Common exam distinctions**

Failover routing is often confused with other Route 53 policies. The exam may ask you to choose the best policy for a scenario, so you should recognize the differences quickly.

| Routing policy | Best use case | Key behavior |
|---|---|---|
| Simple | One endpoint or basic DNS answer | Returns a single record; can use health checks |
| Weighted | A/B testing or gradual traffic shifting | Returns answers based on configured weights |
| Latency | Lowest-latency regional endpoint | Chooses the region with the best latency |
| Geolocation | Country/region-based routing | Answers based on user location |
| Multivalue answer | Multiple healthy endpoints | Returns up to eight healthy records |
| Failover | Active-passive high availability | Returns primary unless unhealthy |

**Practical design guidance**

Use failover routing when you want a standby endpoint that should not receive traffic during normal operation. Use weighted routing when you want both endpoints to receive traffic. Use latency routing when the goal is performance rather than standby recovery.

For exam scenarios, remember that Route 53 failover is a DNS-level control plane feature. It does not move data, preserve sessions, or automatically synchronize application state. Your application and data layer must already be designed for recovery.

- **Active-passive** fits failover routing.
- **Active-active** usually fits weighted, latency, or multivalue patterns.
- **State replication** is outside Route 53 and must be handled separately.
- **Low TTL** helps, but it does not eliminate caching delays.

Failover routing in Route 53 gives you a clean primary-secondary DNS pattern for high availability. The key exam points are the record pair requirements, the health-based decision logic, the effect of TTL and caching, and the service-specific nature of alias target health evaluation. If you can model the health state transitions and predict the DNS answer, you understand the core of this lab.
