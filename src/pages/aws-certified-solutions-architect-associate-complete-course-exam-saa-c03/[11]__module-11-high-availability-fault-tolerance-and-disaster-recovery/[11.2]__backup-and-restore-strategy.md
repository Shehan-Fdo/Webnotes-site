## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Backup and restore strategy overview**

A strong backup strategy protects against accidental deletion, corruption, ransomware, and regional failure. In AWS, you usually combine multiple layers:

- **Point-in-time recovery** for databases and logs
- **Snapshots** for block storage and managed services
- **Replication** for faster recovery in another location
- **Archive and retention controls** for compliance
- **Restore testing** to prove that recovery actually works

For exam purposes, separate **availability** from **backup**. Multi-AZ, Auto Scaling, and replication improve uptime, but they do not replace backups. A backup lets you recover from bad writes, accidental deletes, and logical corruption.

**Core recovery concepts**

A useful way to think about recovery is by objective:

- **RPO**: Recovery Point Objective, the maximum acceptable data loss measured in time
- **RTO**: Recovery Time Objective, the maximum acceptable downtime
- **Retention**: how long you keep recovery points
- **Immutability**: whether backups can be changed or deleted before expiry

Lower RPO usually means more frequent backups or continuous log capture. Lower RTO usually means faster restore paths, warm standby systems, or replicated data.

**Amazon RDS and Aurora backups**

Amazon RDS supports **automated backups** and **manual snapshots**. These are not the same.

- **Automated backups**
  - Enable point-in-time recovery (PITR) within the retention window
  - Include the database volume and transaction logs
  - Are retained for a configurable period, typically **1 to 35 days**
  - Default retention is often **7 days** for many engines, but you should verify service defaults in the console or documentation for the specific engine
  - Are deleted when you delete the DB instance unless you choose to retain them or create a final snapshot

- **Manual snapshots**
  - Are created by you
  - Are retained until you delete them
  - Are useful for long-term retention, migration, and pre-change checkpoints
  - Do not expire automatically

For exam questions, remember:

- Automated backups support restore to a specific time **only within the retention window**
- Manual snapshots restore to the snapshot point in time, not to arbitrary timestamps
- Deleting an RDS instance can remove automated backups unless you preserve them with the deletion options

A typical restore workflow looks like this:

1. Create a manual snapshot before a risky change.
2. If needed, restore from the snapshot or use PITR from automated backups.
3. Launch a new DB instance from the recovery point.
4. Update the application endpoint to point to the restored instance.

Example AWS CLI commands:

```bash
aws rds create-db-snapshot \
  --db-instance-identifier prod-db \
  --db-snapshot-identifier prod-db-predeploy-20260426
```

```bash
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier prod-db-restore \
  --db-snapshot-identifier prod-db-predeploy-20260426
```

For PITR, you restore to a timestamp:

```bash
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier prod-db \
  --target-db-instance-identifier prod-db-pitr-restore \
  --restore-time 2026-04-26T12:30:00Z
```

**Aurora-specific notes**

Aurora uses cluster-level backup and restore behavior.

- Automated backups and snapshots are taken at the cluster level
- Restores typically create a **new cluster**
- You then create or attach instances to that restored cluster
- Aurora also supports fast cloning in some scenarios, but cloning is not a backup substitute

A common exam trap is confusing **Aurora Multi-AZ** with backup. Multi-AZ improves availability and failover, while backups protect against logical mistakes and data loss.

**Amazon RDS Multi-AZ versus backups**

Multi-AZ is an **availability** feature, not a backup feature.

- **Multi-AZ**
  - Synchronously replicates to a standby in another AZ
  - Supports automatic failover
  - Helps with infrastructure failure
  - Does not protect against accidental deletes or bad application writes

- **Backups**
  - Protect against logical corruption and deletion
  - Allow restore to earlier points in time
  - Are required for recovery from user error

Use Multi-AZ for uptime and backups for recovery.

**Amazon EC2 backup and restore**

For EC2, the main backup mechanism is **EBS snapshots** and **AMI** creation.

- **EBS snapshots**
  - Back up individual EBS volumes
  - Are incremental after the first snapshot
  - Are stored in Amazon S3-managed infrastructure
  - Can be copied across Regions and accounts
  - Restore a volume, not the whole instance

- **Amazon Machine Image (AMI)**
  - Captures the instance configuration plus one or more EBS snapshots
  - Is the standard way to recover a full EC2 instance
  - Lets you launch a replacement instance with the same OS, software, and attached volumes

For full instance recovery, create an AMI. For volume-level recovery, create snapshots.

Example workflow:

```bash
aws ec2 create-image \
  --instance-id i-0123456789abcdef0 \
  --name "webserver-prepatch-20260426" \
  --no-reboot
```

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Data volume snapshot before migration"
```

> `--no-reboot` reduces disruption, but the backup may be less application-consistent than a rebooted or quiesced capture.

**Application-consistent backups**

A crash-consistent backup captures storage as if power failed at that moment. An application-consistent backup flushes in-memory data and transaction logs so the restored system is clean.

Use application-consistent backups for:

- Databases
- SAP HANA
- Transaction-heavy workloads
- File systems with write caching

Common techniques include:

- Quiescing the application
- Flushing buffers and logs
- Using VSS (Volume Shadow Copy Service) on Windows
- Coordinating with database-native backup tools

Example Linux pre-freeze and post-thaw pattern:

```bash
#!/bin/bash
systemctl stop app-service
sync
# take snapshot or AMI
systemctl start app-service
```

For databases, prefer native backup integration when available because it preserves consistency better than raw volume snapshots alone.

**EBS snapshot restore behavior**

Restoring from an EBS snapshot creates a new volume. Performance is important here.

- Data is lazily loaded from snapshot storage on first access
- Initial reads can be slower until blocks are warmed
- Performance-sensitive workloads may need pre-warming or initialization
- You can reduce cold-start impact by reading the volume sequentially after restore

Example warm-up pattern:

```bash
sudo dd if=/dev/xvdf of=/dev/null bs=1M status=progress
```

Use this carefully on production systems because it generates read traffic. The key exam idea is that restored EBS volumes may not perform at full speed immediately.

**Amazon S3 backup and protection features**

S3 is often used as a durable backup target, but you need to understand the difference between features.

- **Versioning**
  - Keeps multiple versions of objects
  - Protects against overwrite and delete mistakes
  - Is a prerequisite for several protection features

- **Replication**
  - Copies objects to another bucket, often in another Region or account
  - Replicates new object versions after the rule is enabled
  - Can replicate delete markers if configured
  - Does not “repair” corruption; it copies object versions according to the replication rule
  - Is not a backup by itself unless combined with retention and access controls

- **MFA Delete**
  - Requires multi-factor authentication for version deletion and versioning suspension
  - Adds protection against accidental or malicious deletion
  - Is a niche but exam-relevant control

- **Object Lock**
  - Enforces write-once-read-many (WORM) retention
  - Supports compliance mode and governance mode
  - Prevents deletion or overwrite for a retention period or legal hold
  - Is useful for immutable backups and ransomware resistance

A practical backup pattern for S3 is:

1. Enable versioning.
2. Enable replication to a separate account or Region.
3. Use Object Lock for immutable retention if compliance requires it.
4. Restrict delete permissions and consider MFA Delete where appropriate.

Example replication configuration concept:

```json
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "Status": "Enabled",
      "Priority": 1,
      "DeleteMarkerReplication": { "Status": "Enabled" },
      "Filter": { "Prefix": "" },
      "Destination": {
        "Bucket": "arn:aws:s3:::dr-bucket",
        "StorageClass": "STANDARD_IA"
      }
    }
  ]
}
```

**Amazon EFS backup and restore**

Amazon EFS is a managed file system, and the common backup path is **AWS Backup**.

- Use AWS Backup to create backups of EFS file systems
- Restore creates a new file system from the recovery point
- EFS replication is a separate feature for disaster recovery, not a substitute for backup retention
- Backups are useful for accidental deletion, corruption, and recovery testing

For exam purposes, do not assume EFS has a native snapshot feature like EBS. The backup story is primarily AWS Backup plus optional replication for DR.

**Amazon FSx backup and restore**

FSx services support backups, but restore behavior depends on the FSx type.

- **FSx for Windows File Server**
  - Supports backups and restore to a new file system
  - Useful for file shares and Windows workloads

- **FSx for Lustre**
  - Supports backups for persistent file systems
  - Restores create a new file system from the backup

- **FSx for NetApp ONTAP**
  - Supports backups and snapshots through ONTAP capabilities
  - Often used for enterprise storage workflows

- **FSx for OpenZFS**
  - Supports snapshots and backups
  - Restore behavior creates new resources from recovery points

The exam usually tests the idea that FSx backups are service-specific and that restores often create a **new file system**, not an in-place rollback.

**Amazon Redshift backups**

Amazon Redshift supports automated and manual snapshots.

- Automated snapshots are retained for a configurable period
- Manual snapshots are retained until deleted
- Restores create a new cluster
- Cross-Region snapshot copy is available for disaster recovery

Redshift is a data warehouse, so backup strategy often focuses on cluster recovery rather than point-in-time application consistency. For large clusters, snapshot copy and retention planning matter more than frequent full restores.

**Amazon DocumentDB and Amazon Neptune**

These managed databases also support snapshot-based recovery.

- **Amazon DocumentDB**
  - Supports automated backups and manual snapshots
  - Restore creates a new cluster
  - PITR is available within the backup retention window

- **Amazon Neptune**
  - Supports automated backups and manual snapshots
  - Restore creates a new cluster
  - Useful for graph database recovery after accidental changes

The exam trap is assuming all managed databases restore in place. In many AWS services, restore means creating a new resource and then switching the application over.

**Amazon ElastiCache backup behavior**

ElastiCache backup support depends on the engine.

- **Redis**
  - Supports snapshots and backup/restore workflows
  - Useful for persistence and recovery
  - Can be restored into a new replication group or cluster configuration

- **Memcached**
  - Does not provide the same snapshot-based persistence model
  - Is generally treated as an in-memory cache without durable backup semantics

This distinction matters on the exam. If you need recoverable state, Redis is the relevant engine, not Memcached.

**AWS Backup**

AWS Backup is a centralized backup service for multiple AWS resources.

It helps you:

- Define **backup plans**
- Assign resources by **tags** or resource selection
- Set backup schedules and retention
- Use backup windows to control when jobs run
- Copy backups across Regions and accounts
- Restore supported resources from a central console or API
- Test restores and audit compliance

A backup plan typically includes:

- A schedule
- A retention rule
- A backup vault
- Optional copy actions
- Resource selection by tags or explicit ARNs

Example backup plan JSON:

```json
{
  "BackupPlanName": "daily-backup-plan",
  "Rules": [
    {
      "RuleName": "daily-0300-utc",
      "TargetBackupVaultName": "primary-vault",
      "ScheduleExpression": "cron(0 3 ? * * *)",
      "StartWindowMinutes": 60,
      "CompletionWindowMinutes": 180,
      "Lifecycle": {
        "DeleteAfterDays": 35
      },
      "CopyActions": [
        {
          "DestinationBackupVaultArn": "arn:aws:backup:us-west-2:123456789012:vault:dr-vault",
          "Lifecycle": {
            "DeleteAfterDays": 90
          }
        }
      ]
    }
  ]
}
```

Important AWS Backup concepts:

- **Backup vault**
  - Logical container for recovery points
  - Can have access policies
  - Can be encrypted with a KMS key

- **Backup window**
  - The time window when backup jobs should start and complete
  - Useful to avoid peak business hours

- **Tag-based selection**
  - Lets you automatically protect resources with matching tags
  - Scales better than manual selection in large environments

- **Restore testing**
  - Helps validate that backups are usable
  - Important for compliance and operational readiness

- **AWS Backup Audit Manager**
  - Provides audit and compliance reporting for backup policies
  - Useful for proving that resources are protected according to controls

**Cross-account and cross-Region backup**

For disaster recovery, you often copy backups to another account and Region.

Why this matters:

- A separate account reduces blast radius
- A separate Region protects against regional outages
- A separate vault can enforce different access policies and retention

Mechanics to remember:

- The destination vault must allow the source account to copy backups
- IAM permissions must allow backup copy and restore actions
- KMS permissions must allow encryption and decryption as needed
- For encrypted backups, cross-account restore or copy usually requires either shared key access or re-encryption with a destination key

A common pattern is:

1. Back up in the source account.
2. Copy the recovery point to a vault in the DR account.
3. Re-encrypt with a KMS key owned by the DR account.
4. Restore from the DR vault during an incident.

**AWS Backup resource access and permissions**

Backup and restore are controlled by a mix of IAM, vault policies, and KMS permissions.

- **IAM**
  - Grants users and roles permission to create, copy, and restore backups
  - Controls who can start restore jobs

- **Backup vault access policy**
  - Controls which principals can access a vault
  - Useful for cross-account backup sharing

- **KMS key policy**
  - Controls encryption and decryption of backup data
  - Must allow the backup service and destination principals when copying or restoring encrypted backups

If restore fails, check these three layers first.

**Restore testing and operational readiness**

A backup is only useful if you can restore it.

Good practice includes:

- Periodic restore tests
- Documented runbooks
- Validation of application dependencies
- DNS and endpoint cutover plans
- Data integrity checks after restore

A restore test should answer:

- Can you restore the resource?
- Does the application start?
- Is the data consistent?
- Can users reach the restored service?
- Does the restore meet RTO and RPO targets?

**Backup strategy patterns**

Common patterns include:

- **Local snapshots plus cross-Region copy**
  - Fast recovery in the same Region
  - Disaster recovery in another Region

- **Immutable backup vault**
  - Protects against deletion and ransomware
  - Often paired with Object Lock or vault lock controls

- **Tiered retention**
  - Short-term frequent backups
  - Medium-term retained snapshots
  - Long-term archived copies

- **Application-native plus infrastructure backup**
  - Database-native dumps or logs for consistency
  - EBS or AWS Backup for infrastructure recovery

**Practical decision guide**

Use the following rule of thumb:

- Choose **RDS automated backups** when you need PITR for a managed database
- Choose **manual snapshots** when you need a long-lived checkpoint
- Choose **AMI** when you need full EC2 instance recovery
- Choose **EBS snapshots** when you need volume recovery
- Choose **AWS Backup** when you want centralized policy management
- Choose **S3 versioning + Object Lock** when you need immutable object protection
- Choose **replication** when you need a secondary copy for DR, not just backup retention

**Common exam traps**

- **Multi-AZ is not backup**
  - It improves availability, not recovery from bad data

- **Replication is not the same as backup**
  - It copies data changes, including mistakes, according to the replication model

- **Snapshots are not always in-place restores**
  - Many services restore to a new resource

- **Automated backups expire**
  - They only support PITR within the retention window

- **Manual snapshots persist**
  - They remain until you delete them

- **Memcached is not a durable cache**
  - Do not assume Redis-style backup behavior

- **EBS restore performance may be cold**
  - Plan for initialization after restore

**Example architecture for a resilient backup design**

A practical SAA-C03 design might look like this:

- EC2 application servers
  - Back up with AMIs and EBS snapshots
  - Store snapshots in a separate account copy for DR

- RDS database
  - Enable automated backups with a 7- to 35-day retention window
  - Create manual snapshots before major changes
  - Use Multi-AZ for availability, not backup

- S3 data lake
  - Enable versioning
  - Use replication to a DR account
  - Apply Object Lock for immutable retention if required

- Shared file storage
  - Use AWS Backup for EFS and FSx
  - Test restores regularly

- Central governance
  - Use AWS Backup plans with tag-based selection
  - Enforce retention and copy rules
  - Review compliance with AWS Backup Audit Manager

**Quick reference**

- **PITR**: point-in-time recovery within a retention window
- **AMI**: full EC2 instance recovery image
- **Snapshot**: point-in-time copy of storage or service state
- **Replication**: secondary copy for DR, not a substitute for backup
- **Object Lock**: immutable object retention
- **AWS Backup**: centralized backup orchestration and policy management
