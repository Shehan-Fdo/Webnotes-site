## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Disaster recovery patterns in AWS**

AWS disaster recovery (DR) patterns describe how much of your workload you keep running in a secondary Region and how quickly you can recover after a failure. The exam often tests the tradeoff between cost and recovery speed.

The main patterns are:

- **Backup and restore**: lowest cost, slowest recovery.
- **Pilot light**: minimal core services run in the recovery Region.
- **Warm standby**: a scaled-down but fully functional copy runs in the recovery Region.
- **Multi-site active-active**: multiple Regions serve traffic at the same time.

A useful way to think about them is:

- More pre-provisioned capacity means faster recovery.
- More active Regions means higher cost and more operational complexity.
- Stateful components usually drive the design more than stateless web tiers.

**Recovery objectives and exam framing**

Two terms appear frequently in DR questions:

- **RTO** (Recovery Time Objective): how long you can afford to be down.
- **RPO** (Recovery Point Objective): how much data loss you can tolerate.

Lower RTO and RPO usually require more replication, more automation, and more always-on infrastructure.

| Pattern | Typical cost | Typical RTO | Typical RPO | Core idea |
|---|---:|---:|---:|---|
| Backup and restore | Lowest | Highest | Highest | Restore from backups after failure |
| Pilot light | Low | Moderate | Low to moderate | Keep only critical core services running |
| Warm standby | Medium | Low | Low | Keep a scaled-down full stack running |
| Multi-site active-active | Highest | Lowest | Lowest | Serve traffic from multiple Regions |

**Pilot light architecture**

Pilot light means you keep the essential data layer or minimal services running in the secondary Region, while the rest of the application is stopped or not fully scaled. The goal is to preserve the most important state and reduce recovery time without paying for a full duplicate environment.

Typical pilot light characteristics:

- **Core data services** are already running in the recovery Region.
- **Application capacity** is minimal or absent until failover.
- **Infrastructure templates** and automation are ready to scale the environment quickly.
- **DNS or traffic management** can shift users to the recovery Region during a disaster.

Common AWS service mappings:

- **Amazon RDS cross-Region read replica** for relational data when you need a warm copy of the database.
- **Amazon Aurora Global Database** for low-latency cross-Region replication and fast promotion of a secondary Region.
- **Amazon DynamoDB global tables** when the workload is designed for multi-Region data replication and active-active access.
- **Amazon S3 Cross-Region Replication (CRR)** for object data and backups.
- **AWS CloudFormation** or **AWS CDK** to recreate application infrastructure quickly.
- **Amazon Route 53** failover routing to direct traffic to the recovery Region.

A pilot light setup often looks like this:

- Primary Region:
  - Full application stack
  - Live traffic
  - Primary database
- Secondary Region:
  - Database replica or replicated data store
  - Minimal supporting services
  - Infrastructure definitions ready for rapid scale-out

> Pilot light is not a full idle copy of the application. If the secondary Region is already running the full workload at reduced capacity, that is closer to warm standby.

**Pilot light failover flow**

A pilot light recovery usually follows a controlled sequence:

1. Detect the primary Region failure using health checks, alarms, or operational monitoring.
2. Promote or activate the data layer in the secondary Region if needed.
3. Deploy or scale the application tier in the secondary Region.
4. Update DNS, load balancer targets, or traffic policies to send users to the recovery Region.
5. Validate application health, data consistency, and downstream integrations.
6. Keep the system in recovery mode until the primary Region is restored.

Practical example with Aurora Global Database:

```text
Primary Region: us-east-1
- Aurora writer cluster
- EC2 Auto Scaling group
- ALB
- Route 53 alias record

Secondary Region: us-west-2
- Aurora secondary cluster
- Minimal EC2 capacity or no app tier
- CloudFormation stack ready
- Route 53 failover record prepared
```

If the primary Region fails, you promote the secondary Aurora cluster and scale the application tier in the secondary Region.

**Warm standby architecture**

Warm standby keeps a scaled-down but fully functional copy of the workload running in the secondary Region. Unlike pilot light, the secondary Region already has the application stack deployed and can serve traffic quickly.

Warm standby characteristics:

- **Full stack exists** in the secondary Region.
- **Reduced capacity** lowers cost compared with active-active.
- **Fast failover** because the environment is already live.
- **Operational parity** is important so the standby behaves like production.

Typical AWS service mappings:

- **Amazon EC2 Auto Scaling** with a smaller desired capacity in the secondary Region.
- **Elastic Load Balancing** configured in both Regions.
- **Amazon RDS cross-Region read replica** or **Aurora Global Database** for database replication.
- **Amazon DynamoDB global tables** if the application is designed for multi-Region writes.
- **Amazon ElastiCache** can be used for local caching, but do not treat it as a durable cross-Region session store.
- **Route 53 failover routing** to move traffic from primary to standby.

Warm standby is often appropriate when:

- You need a low RTO but cannot justify active-active cost.
- The application is stateful and needs a ready environment.
- You want a tested, production-like secondary stack.

A common warm standby layout is:

- Primary Region:
  - 100% traffic
  - Full capacity
  - Primary database writer
- Secondary Region:
  - 10% to 50% of normal compute capacity, depending on workload
  - Same application version deployed
  - Replicated data store
  - Ready to scale up quickly

The exact percentage is not fixed. The key exam idea is that the secondary Region is fully functional, not that it runs at a specific ratio.

**Warm standby failover flow**

1. Detect the primary Region outage.
2. Stop writes to the failed Region if possible.
3. Promote the secondary database or switch the writer endpoint.
4. Increase compute capacity in the secondary Region.
5. Shift traffic using Route 53, Global Accelerator, or load balancer-based routing.
6. Monitor latency, errors, and data consistency after cutover.

Example scaling action:

```bash
aws autoscaling update-auto-scaling-group \
  --auto-scaling-group-name app-asg-us-west-2 \
  --desired-capacity 6 \
  --min-size 4 \
  --max-size 12
```

This kind of command is typical during failover when the standby Region must absorb production traffic.

**Multi-site active-active architecture**

Multi-site active-active means multiple Regions serve traffic at the same time. This pattern gives the best availability and the fastest recovery because there is no need to bring a dormant site online.

Key properties:

- **Traffic is distributed** across Regions continuously.
- **Both Regions are production-ready**.
- **Failover is often automatic** because traffic already flows to multiple sites.
- **Data design is the hardest part** because writes may occur in more than one Region.

This pattern is best when:

- You need very low RTO and RPO.
- The application can tolerate distributed-system complexity.
- You can design for stateless compute and replicated data.

Common AWS service mappings:

- **Amazon Route 53 latency-based routing** to send users to the nearest healthy Region.
- **AWS Global Accelerator** for fast anycast-based routing and health-based failover.
- **Amazon DynamoDB global tables** for multi-Region active-active writes.
- **Amazon Aurora Global Database** for fast cross-Region replication, though only one Region is the writer at a time in the standard model.
- **Amazon S3 CRR** for replicated object storage.
- **Amazon CloudFront** for global edge caching, often combined with regional origins.

Important distinction:

- **True multi-Region active-active writes** are not supported by every AWS database.
- Many managed databases support one writer Region and one or more read replicas.
- DynamoDB global tables are the clearest AWS-native example of multi-Region active-active writes.
- Aurora Global Database improves read locality and disaster recovery, but it is not the same as simultaneous multi-Region writers.

**Data and state handling**

Stateful components are the main challenge in DR design.

Best practices:

- Keep application servers **stateless** whenever possible.
- Store sessions in a durable shared system or use token-based sessions.
- Replicate databases across Regions according to the DR pattern.
- Avoid relying on local instance storage for critical state.
- Design idempotent writes so retries do not create duplicate side effects.

Session handling options:

- **Stateless tokens** such as signed JWTs can reduce dependency on shared session storage.
- **DynamoDB** can store session state durably when you need centralized persistence.
- **ElastiCache** is useful for low-latency caching, but it is not a durable DR session backbone by itself.

> Do not assume cache replication equals disaster recovery. A cache can improve performance, but it is not a substitute for a durable replicated data store.

**DynamoDB global tables**

DynamoDB global tables are a strong fit for multi-site active-active designs because they replicate data across Regions and allow local reads and writes.

What to remember:

- They support multi-Region replication.
- They are designed for active-active access patterns.
- Conflict resolution is handled by DynamoDB replication semantics, commonly described as last writer wins based on timestamps and replication metadata.
- They are not a guarantee of application-level conflict correctness.

Practical implications:

- You should design writes to be idempotent.
- You should avoid assuming version attributes provide global serialization across Regions.
- You should test how concurrent updates behave under replication lag.

Example table creation is usually done by creating a standard table first and then enabling replication through the console, SDK, or `update-table` workflow rather than a single legacy-style CLI command.

Example of enabling a replica with the AWS CLI:

```bash
aws dynamodb update-table \
  --table-name Orders \
  --replica-updates '[{"Create":{"RegionName":"us-west-2"}}]'
```

This is the modern style of operation you should recognize for exam purposes.

**Aurora and RDS in DR patterns**

Aurora and RDS are common exam services for DR questions, but they behave differently.

Aurora options:

- **Aurora Global Database**:
  - Best for fast cross-Region replication.
  - Useful for pilot light or warm standby.
  - Secondary Region can be promoted quickly.
- **Aurora cross-Region read replicas**:
  - Useful for read scaling and DR.
  - Promotion is possible, but failover is not as seamless as active-active.

RDS options:

- **Cross-Region read replicas**:
  - Good for DR and read scaling.
  - Promotion is manual or operationally controlled.
- **Multi-AZ**:
  - Protects against Availability Zone failure, not Region failure.
  - Do not confuse Multi-AZ with multi-Region DR.

Exam tip:

- If the question says “Region failure,” think cross-Region replication or multi-Region design.
- If the question says “AZ failure,” think Multi-AZ.

**Route 53 routing policies for DR**

Route 53 is often the traffic control layer in DR architectures.

Use cases:

- **Failover routing**:
  - Best for pilot light and warm standby.
  - Primary endpoint serves traffic until health checks fail.
  - Secondary endpoint takes over when the primary is unhealthy.
- **Weighted routing**:
  - Useful for gradual traffic shifting, canary releases, and controlled failback.
  - Also useful in active-active when you want to split traffic intentionally.
- **Latency-based routing**:
  - Best for active-active global applications.
  - Sends users to the Region with the lowest latency.
- **Geolocation routing**:
  - Useful when you need to route by user location or regulatory boundary.
- **Geoproximity routing**:
  - Useful when you want to bias traffic toward a Region based on geography, often with traffic flow policies.

A practical DR pattern mapping is:

- **Pilot light**: failover routing
- **Warm standby**: failover routing, sometimes weighted during cutover
- **Active-active**: latency-based routing or Global Accelerator

Example weighted routing change for failback should update both records, not just one:

```json
{
  "Comment": "Shift traffic back toward primary Region",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "primary-us-east-1",
        "Weight": 80,
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack.alb-primary-123.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "secondary-us-west-2",
        "Weight": 20,
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack.alb-secondary-456.us-west-2.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}
```

**Choosing the right pattern**

Use the pattern that matches your business requirements, not the cheapest or most advanced one.

| Requirement | Best fit | Why |
|---|---|---|
| Lowest cost, acceptable downtime | Backup and restore | Minimal always-on infrastructure |
| Moderate cost, faster recovery | Pilot light | Core data is already available |
| Low downtime, production-like standby | Warm standby | Full stack is already running |
| Near-zero downtime, global users | Active-active | Traffic already spans Regions |

A simple decision rule:

- Choose **pilot light** when you can tolerate some recovery time and want to keep costs low.
- Choose **warm standby** when you need faster recovery and a ready-to-scale environment.
- Choose **active-active** when availability is critical and your application can handle distributed writes and traffic management.

**Operational best practices**

- Keep compute layers stateless.
- Use infrastructure as code so you can recreate environments consistently.
- Test failover regularly; DR that is never tested is not reliable.
- Automate DNS, scaling, and database promotion steps.
- Monitor replication lag and health checks continuously.
- Document runbooks for failover and failback.
- Validate dependencies such as IAM roles, KMS keys, secrets, and VPC connectivity in both Regions.

Practical checklist:

- Ensure AMIs, container images, or deployment artifacts are available in both Regions.
- Replicate secrets with AWS Secrets Manager or AWS Systems Manager Parameter Store where appropriate.
- Confirm KMS key strategy supports cross-Region recovery.
- Verify security groups, route tables, and VPC endpoints exist in the recovery Region.
- Test application startup time under scaled conditions.

**Common exam traps**

- **Multi-AZ is not multi-Region**.
- **Pilot light is not a full idle copy** of the application.
- **Warm standby is not defined by a fixed percentage** of capacity.
- **DynamoDB global tables are not a magic conflict-free system** for arbitrary business logic.
- **ElastiCache is not a durable DR session store**.
- **Aurora Global Database is not the same as simultaneous multi-writer across Regions**.

**Quick architecture references**

Pilot light example:

- Route 53 failover
- Aurora Global Database or RDS cross-Region replica
- Minimal EC2 or container capacity in secondary Region
- CloudFormation to scale out on demand

Warm standby example:

- Route 53 failover or weighted routing
- Full application stack in both Regions
- Reduced EC2 Auto Scaling desired capacity in secondary Region
- Replicated database and tested runbooks

Active-active example:

- Route 53 latency-based routing or Global Accelerator
- Stateless app tier in multiple Regions
- DynamoDB global tables or another multi-Region data strategy
- Strong observability and idempotent application design

**Summary**

Pilot light, warm standby, and multi-site active-active are all about balancing cost, recovery speed, and operational complexity. Pilot light keeps only the essential core running, warm standby keeps a full but smaller copy ready, and active-active serves traffic from multiple Regions at once. For the exam, focus on the data layer, traffic routing, and whether the secondary Region is merely prepared, fully running at reduced capacity, or actively serving users.
