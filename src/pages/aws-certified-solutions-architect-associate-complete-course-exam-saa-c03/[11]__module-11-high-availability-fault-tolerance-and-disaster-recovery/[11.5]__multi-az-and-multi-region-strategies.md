## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**High availability and disaster recovery goals**

Multi-AZ and Multi-Region designs solve different problems. **Multi-AZ** protects against an Availability Zone failure inside a Region, while **Multi-Region** protects against a full-Region outage and can also reduce latency for global users. In exam scenarios, first identify the failure domain, then choose the smallest architecture that meets the recovery objective.

- **Availability** focuses on keeping a service running during component failure.
- **Durability** focuses on not losing stored data.
- **RTO** (Recovery Time Objective) is how long recovery can take.
- **RPO** (Recovery Point Objective) is how much data loss is acceptable.

A common exam trap is assuming every replicated system is active-active. Some services replicate synchronously, some asynchronously, and some require manual promotion. The right answer depends on whether you need near-zero data loss, fast failover, global read access, or cost efficiency.

**Multi-AZ fundamentals**

Multi-AZ deployments place resources across separate Availability Zones in the same Region. An Availability Zone is a physically separate data center group with independent power, networking, and cooling. Use Multi-AZ when you need regional resilience without the complexity and cost of cross-Region architecture.

- **EC2** instances can be distributed across multiple AZs behind an Elastic Load Balancer.
- **Auto Scaling groups** can span multiple AZs to replace unhealthy instances automatically.
- **Amazon RDS Multi-AZ** is for high availability, not read scaling.
- **Amazon EFS** is regional and stores data across multiple AZs by design.
- **Amazon S3** is regional and stores objects redundantly across multiple AZs.

For application tiers, Multi-AZ usually means stateless compute in multiple AZs plus a data layer that survives AZ loss. If the app tier is not stateless, failover becomes slower and more error-prone.

**Amazon RDS Multi-AZ behavior**

Amazon RDS Multi-AZ is a managed high-availability deployment, but the exact implementation depends on the engine and deployment model. For many engines, RDS maintains a standby in a different AZ and uses synchronous replication for storage or data changes so that failover can happen with minimal data loss. Newer Multi-AZ DB cluster deployments use a different architecture with multiple database instances and low-latency replication designed for faster failover.

- **Primary use case** is automatic failover, not read scaling.
- **Standby** is not normally used for reads in standard Multi-AZ deployments.
- **Failover** is automatic when the primary instance or AZ becomes unavailable.
- **Endpoint** remains the same after failover, which reduces application changes.

Use Multi-AZ RDS when you need managed HA for OLTP workloads and do not want to manage replication yourself. Do not confuse it with read replicas, which are primarily for read scaling and disaster recovery.

```bash
aws rds create-db-instance \
  --db-instance-identifier app-db \
  --db-instance-class db.m6g.large \
  --engine mysql \
  --allocated-storage 100 \
  --multi-az \
  --master-username admin \
  --master-user-password 'ChangeMe123!'
```

> Standard RDS Multi-AZ is for failover resilience. It is not the same as a read replica, and it is not a substitute for cross-Region disaster recovery.

**RDS read replicas and disaster recovery**

RDS read replicas use asynchronous replication. They help offload reads and can be promoted to a standalone database, including in another Region. However, promotion is a manual recovery action, and replication lag means RPO is not zero.

- **Read scaling** is the primary reason to use replicas.
- **Cross-Region replicas** improve disaster recovery options.
- **Promotion** converts the replica into a writable primary.
- **Lag** means recent writes may not be present at the time of failure.

A practical DR pattern is to keep a cross-Region read replica ready, then promote it during a Region outage. This is useful when you can tolerate some data loss and a manual recovery step.

```bash
aws rds create-db-instance-read-replica \
  --db-instance-identifier app-db-dr \
  --source-db-instance-identifier app-db \
  --region us-west-2
```

**Amazon Aurora high availability**

Aurora separates compute from storage. The storage layer is distributed across multiple AZs and is designed for high durability, while failover occurs by promoting another DB instance in the Aurora cluster. This distinction matters: Aurora does not fail over to storage; it fails over to another database instance that can attach to the shared cluster volume.

- **Cluster volume** is replicated across multiple AZs.
- **Writer instance** handles writes.
- **Reader instances** can serve reads and can be promoted.
- **Failover** is typically faster than standard RDS because the storage is already shared and distributed.

Aurora is a strong choice when you want managed relational HA with fast recovery and optional read scaling. Aurora Global Database extends this model across Regions for low-latency global reads and disaster recovery.

```bash
aws rds create-db-cluster \
  --db-cluster-identifier aurora-prod \
  --engine aurora-mysql \
  --master-username admin \
  --master-user-password 'ChangeMe123!' \
  --vpc-security-group-ids sg-0123456789abcdef0
```

**Multi-Region strategies**

Multi-Region architectures protect against Region-wide outages and support global user distribution. They are more complex than Multi-AZ because you must design for replication, routing, failover, and data consistency across Regions.

- **Active-passive** keeps one Region primary and another standby.
- **Active-active** serves traffic from multiple Regions at the same time.
- **Pilot light** keeps core data replicated and minimal services running.
- **Warm standby** keeps a scaled-down but functional environment ready.
- **Backup and restore** is the lowest-cost DR pattern but has the highest RTO.

Choose the pattern based on business requirements:
- If you need the lowest cost, use backup and restore.
- If you need faster recovery, use pilot light or warm standby.
- If you need global low latency and high resilience, use active-active.

**Route 53 routing and failover**

Amazon Route 53 is the DNS service used to direct users to healthy endpoints. It supports several routing policies, and the right one depends on whether you want latency optimization, health-based failover, weighted distribution, or location-based routing.

- **Simple routing** points a name to one resource.
- **Weighted routing** splits traffic by percentage.
- **Latency routing** sends users to the Region with the lowest latency.
- **Failover routing** sends traffic to a primary endpoint and a secondary endpoint when health checks fail.
- **Geolocation routing** routes based on the user’s geographic location.
- **Geoproximity routing** is used with Route 53 traffic flow policies and lets you bias traffic toward or away from resources based on geography.

For exam purposes, remember that geoproximity is not just a standalone checkbox in the same way as weighted or failover routing. It is part of Route 53 traffic flow and is used when you need more control over geographic traffic distribution.

```text
Users -> Route 53
       -> Primary Region ALB
       -> Secondary Region ALB
       -> Health checks decide failover
```

A common pattern is active-passive with Route 53 failover records:
- Primary record points to the main Region.
- Secondary record points to the DR Region.
- Health checks determine when to switch.

**Global load balancing patterns**

When you need active-active multi-Region traffic, combine Route 53 with regional load balancers and replicated data stores. This improves latency and resilience, but it requires careful handling of session state and data consistency.

- **Stateless services** are easiest to run active-active.
- **Session state** should be externalized to ElastiCache, DynamoDB, or another shared store when possible.
- **Sticky sessions** can complicate failover and should be avoided when you can design statelessly.
- **Data writes** need a conflict strategy if multiple Regions can write simultaneously.

A practical topology is:
- Route 53 latency routing to two Regional Application Load Balancers.
- Application tier deployed in both Regions.
- Data layer replicated with a service that supports multi-Region access or controlled failover.

**DynamoDB global tables**

DynamoDB Global Tables provide multi-Region, multi-active replication for DynamoDB tables. They are useful when you need low-latency global reads and writes with managed replication. Replication is asynchronous, so you should expect eventual consistency across Regions.

- **Global Tables** support active-active access patterns.
- **Replication** is managed by DynamoDB.
- **Conflict resolution** uses last writer wins based on timestamps.
- **Eventual consistency** means writes may not appear immediately in every Region.

Create a table and add replicas in supported Regions. Modern workflows use table creation or table updates to add replicas rather than older deprecated patterns.

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=OrderId,AttributeType=S \
  --key-schema AttributeName=OrderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --replica-updates '[{"Create":{"RegionName":"us-west-2"}},{"Create":{"RegionName":"eu-west-1"}}]'
```

If the table already exists, add a replica with `update-table`:

```bash
aws dynamodb update-table \
  --table-name Orders \
  --replica-updates '[{"Create":{"RegionName":"us-west-2"}}]'
```

Use Global Tables when your application can tolerate eventual consistency and you need regional write locality. Do not use them when strict cross-Region transactional consistency is required.

**S3 and cross-Region data protection**

Amazon S3 is inherently multi-AZ within a Region, but cross-Region protection is your responsibility. Use replication when you need a second copy in another Region.

- **Same-Region Replication** copies objects within a Region.
- **Cross-Region Replication** copies objects to another Region.
- **Versioning** must be enabled for replication.
- **Replication Time Control** can provide predictable replication timing for eligible workloads.

S3 replication is useful for DR, compliance, and data locality. It does not automatically replicate existing objects unless configured appropriately, and delete markers and permissions require careful planning.

```bash
aws s3api put-bucket-versioning \
  --bucket source-bucket \
  --versioning-configuration Status=Enabled
```

**Elastic Load Balancing and Auto Scaling across AZs**

Elastic Load Balancing and Auto Scaling are core building blocks for Multi-AZ application tiers. They distribute traffic and capacity across multiple AZs so that the loss of one AZ does not take the service down.

- **Application Load Balancer** is best for HTTP and HTTPS routing.
- **Network Load Balancer** is best for ultra-low latency and TCP/UDP traffic.
- **Auto Scaling groups** should span at least two AZs for resilience.
- **Health checks** remove unhealthy targets automatically.

A good baseline architecture is:
- ALB in public subnets across two or more AZs.
- EC2 instances in private subnets across the same AZs.
- RDS Multi-AZ or Aurora for the database tier.

**Choosing the right pattern**

Use the simplest architecture that meets the requirement. Exam questions often hide the answer in the recovery objective or the scope of the failure.

| Requirement | Best fit | Why |
|---|---|---|
| Survive one AZ failure | Multi-AZ | Fast failover within a Region |
| Reduce read latency globally | Multi-Region active-active | Users hit the nearest Region |
| Minimize cost for DR | Backup and restore | Lowest steady-state cost |
| Faster DR with limited cost | Pilot light | Core data and minimal services ready |
| Managed relational HA | RDS Multi-AZ or Aurora | Automatic failover and managed replication |
| Global low-latency NoSQL writes | DynamoDB Global Tables | Multi-Region active-active access |

**Exam distinctions to remember**

- **Multi-AZ** is about regional high availability.
- **Multi-Region** is about regional disaster recovery and global performance.
- **RDS Multi-AZ** is not for read scaling.
- **Read replicas** are asynchronous and can be promoted manually.
- **Aurora failover** promotes another instance in the cluster, not the storage layer.
- **Route 53 failover** depends on health checks.
- **Geoproximity** belongs to Route 53 traffic flow, not a simple standalone routing policy.
- **DynamoDB Global Tables** are modern multi-Region replication, not a manual copy-and-switch process.

**Practical design examples**

A typical web application with strong availability requirements might use:
- Route 53 latency routing
- ALB in two Regions
- Auto Scaling groups in multiple AZs per Region
- Aurora Global Database or DynamoDB Global Tables
- S3 cross-Region replication for static assets and backups

A cost-conscious DR design might use:
- Primary Region with Multi-AZ compute and database
- Cross-Region backups in S3
- Warm standby or pilot light in the secondary Region
- Route 53 failover records for DNS switching

**Operational considerations**

Multi-Region systems require more than replication. You also need deployment automation, observability, and tested failover procedures.

- **Infrastructure as Code** helps recreate environments consistently.
- **Runbooks** define the failover and failback process.
- **Monitoring** should include health checks, replication lag, and error rates.
- **Testing** should validate both planned and unplanned failover.

> A DR plan that has never been tested is usually incomplete. Validate DNS switching, data promotion, and application startup before you rely on it in production.

**Quick reference commands**

```bash
# Check RDS instance status
aws rds describe-db-instances --db-instance-identifier app-db

# Check Route 53 health checks
aws route53 list-health-checks

# Inspect DynamoDB table replicas
aws dynamodb describe-table --table-name Orders

# Verify S3 replication configuration
aws s3api get-bucket-replication --bucket source-bucket
```

**Summary**

Multi-AZ gives you resilience against an AZ failure with minimal operational overhead. Multi-Region extends resilience to Region-wide outages and global performance needs, but it adds complexity in replication, routing, and consistency. For the exam, always map the requirement to the failure domain, then choose the simplest architecture that satisfies the required RTO and RPO.
