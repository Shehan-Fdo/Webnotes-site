## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Backup and restore strategy overview**

A good AWS backup design matches the data type to the recovery method. For the SAA-C03 exam, you should know the native recovery features for each service and when they are the right tool:

- **Amazon S3**: versioning, replication, and Object Lock are the core native protection features.
- **Amazon RDS**: automated backups and snapshots support point-in-time recovery (PITR) and manual restore workflows.
- **Amazon EC2**: EBS snapshots back up attached volumes; AMIs capture instance launch configuration plus EBS-backed root volume state.
- **Application data**: consistency matters more than raw snapshotting, especially when data spans multiple volumes or services.

A practical recovery plan usually separates:
- **Operational recovery** for accidental deletes or bad writes.
- **Disaster recovery** for AZ, Region, or account loss.
- **Compliance retention** for immutable or long-lived records.

**Decision path for common AWS recovery choices**

```text
Need to recover a single deleted object?
  -> S3 versioning or delete marker removal

Need to recover a database to a specific time?
  -> RDS automated backups / PITR

Need to recover an EC2 volume or instance?
  -> EBS snapshot -> new volume -> attach, or create AMI

Need immutable retention?
  -> S3 Object Lock, plus versioning
```

**Amazon S3 backup and restore workflows**

S3 does not use traditional backups in the same way as block storage or databases. The exam expects you to understand that the primary native protection mechanisms are **versioning**, **replication**, and **Object Lock**.

- **Versioning** keeps multiple versions of an object in the same bucket.
- **Replication** copies objects to another bucket, often in another Region or account.
- **Object Lock** prevents deletion or overwrite for a retention period or legal hold.
- **Lifecycle rules** can transition older versions to cheaper storage or expire noncurrent versions.

Versioning is the first feature to enable when you want protection from accidental overwrite or delete. If a user deletes an object in a versioned bucket, S3 adds a delete marker rather than removing the data immediately. To restore the current object, you can either remove the delete marker or copy a previous version back to the same key.

```bash
aws s3api put-bucket-versioning \
  --bucket my-app-bucket \
  --versioning-configuration Status=Enabled
```

To inspect versions and delete markers:

```bash
aws s3api list-object-versions \
  --bucket my-app-bucket \
  --prefix reports/annual.pdf
```

To restore a previous version by copying it over the current key:

```bash
aws s3api copy-object \
  --bucket my-app-bucket \
  --copy-source my-app-bucket/reports/annual.pdf?versionId=3HL4kqtJlcpXroDTDmjVBH40Nrjfkd \
  --key reports/annual.pdf
```

This creates a new current version with the restored content. It does not erase the older versions; it simply makes the copied object the latest version.

If the current object is hidden by a delete marker, you can delete the delete marker to reveal the latest prior version:

```bash
aws s3api delete-object \
  --bucket my-app-bucket \
  --key reports/annual.pdf \
  --version-id 2Lgk4exampleDeleteMarker
```

**S3 versioning, replication, and Object Lock comparison**

| Feature | Best use case | Recovery behavior | Exam note |
|---|---|---|---|
| Versioning | Accidental delete or overwrite | Restore prior version or remove delete marker | Core protection feature for S3 |
| Replication | Cross-Region or cross-account resilience | Recover from replica bucket if source is unavailable | Requires versioning on source bucket |
| Object Lock | Compliance retention and immutability | Prevents deletion or overwrite during retention | Use for WORM-style requirements |

Replication is not a backup substitute by itself if the same bad write is replicated immediately. It is best for resilience, isolation, and disaster recovery. If you need protection from malicious deletion, combine replication with versioning and, when required, Object Lock.

A simple S3 recovery lab flow looks like this:

1. Enable versioning on the source bucket.
2. Upload a test object.
3. Overwrite the object with bad content.
4. Restore the previous version by copying it back.
5. Delete the object and observe the delete marker.
6. Remove the delete marker to make the prior version current again.

**Amazon RDS backup and restore workflows**

Amazon RDS supports automated backups, manual snapshots, and point-in-time recovery. For exam purposes, the key distinction is:

- **Automated backups** support PITR within the retention window.
- **Manual snapshots** are user-managed restore points that persist until deleted.
- **Restores** create a new DB instance rather than in-place recovery.

RDS automated backups capture transaction logs and backups of the database storage. This lets you restore to a specific second within the retention period, which is useful after accidental deletes, bad deployments, or data corruption.

To enable automated backups, set a retention period greater than zero. In practice, you also want Multi-AZ for high availability, but Multi-AZ is not a backup strategy; it is a failover strategy.

```bash
aws rds modify-db-instance \
  --db-instance-identifier mydb \
  --backup-retention-period 7 \
  --apply-immediately
```

To create a manual snapshot:

```bash
aws rds create-db-snapshot \
  --db-snapshot-identifier mydb-manual-2026-04-26 \
  --db-instance-identifier mydb
```

To restore from a snapshot:

```bash
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier mydb-restored \
  --db-snapshot-identifier mydb-manual-2026-04-26
```

To restore to a point in time:

```bash
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier mydb \
  --target-db-instance-identifier mydb-pitr \
  --restore-time 2026-04-26T12:30:00Z
```

RDS restore always creates a new DB instance. After restore, you usually update the application connection string, DNS record, or secret reference to point to the new instance.

**RDS snapshot vs PITR comparison**

| Method | Granularity | Retention | Best use case |
|---|---|---|---|
| Manual snapshot | Specific backup point | Until deleted | Long-lived restore point before change windows |
| Automated backup / PITR | Any second within retention window | Configured retention period | Recover from recent corruption or accidental changes |

A practical RDS lab scenario:

1. Create a manual snapshot before a schema migration.
2. Run the migration and confirm the app breaks.
3. Restore a new DB instance from the snapshot.
4. Point the application to the restored instance.
5. Verify the old data and schema are back.
6. If the issue happened after the snapshot, use PITR instead of the snapshot to recover a more recent state.

**Amazon EC2 and EBS backup workflows**

For EC2, the backup target is usually the **Elastic Block Store (EBS)** volume, not the instance itself. EBS snapshots are incremental backups of volume data stored in Amazon S3-managed infrastructure. You can use them to create new volumes in the same Region.

Important constraints:
- A snapshot is Region-scoped.
- A restored volume must be created in an Availability Zone (AZ).
- The new volume must be in the same AZ as the EC2 instance before you can attach it.
- If you need a full instance recovery, create an AMI or launch a new instance from an AMI.

Create a snapshot:

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-deployment backup"
```

Create a new volume from a snapshot in a specific AZ:

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0123456789abcdef0 \
  --volume-type gp3
```

Attach the restored volume to an instance in the same AZ:

```bash
aws ec2 attach-volume \
  --volume-id vol-0abc1234def567890 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdf
```

If you need to preserve the full launch configuration, use an AMI:

```bash
aws ec2 create-image \
  --instance-id i-0123456789abcdef0 \
  --name "webserver-ami-2026-04-26" \
  --no-reboot
```

An AMI is useful when you want to recreate the instance with its OS, installed software, and attached EBS-backed root volume state. It is the better choice for repeatable server recovery than a single volume snapshot.

> Freezing a filesystem with `fsfreeze` only works on a mounted filesystem path, and it does not by itself make multi-volume application data consistent. For databases or apps spanning multiple volumes, you still need application quiescing or coordinated snapshots.

Example of a mounted filesystem freeze workflow:

```bash
sudo fsfreeze -f /data
aws ec2 create-snapshot --volume-id vol-0123456789abcdef0 --description "Frozen filesystem snapshot"
sudo fsfreeze -u /data
```

Use this only when `/data` is actually a mounted filesystem. For database workloads, prefer native database backup features or coordinated snapshot procedures.

**EC2 restore workflow lab**

1. Identify the source EBS volume and its AZ.
2. Create a snapshot of the volume.
3. Create a new volume from the snapshot in the same AZ as the target instance.
4. Stop the instance if you need to replace the root volume safely.
5. Detach the old volume if required.
6. Attach the restored volume.
7. Mount the filesystem and verify the data.

For a root volume recovery, the safer pattern is often:
1. Create an AMI or snapshot.
2. Launch a replacement instance from the AMI.
3. Reassociate Elastic IPs, security groups, and IAM roles as needed.
4. Validate application health before decommissioning the old instance.

**End-to-end recovery scenarios**

These scenarios help you map the service to the right recovery method.

- **S3 accidental delete**
  - Enable versioning.
  - Delete the object.
  - Restore by removing the delete marker or copying a previous version.
  - Expected result: the object becomes current again without changing bucket structure.

- **S3 bad overwrite**
  - Upload a bad file to the same key.
  - Copy the earlier version over the key.
  - Expected result: a new current version contains the correct content.

- **RDS bad deployment**
  - Take a manual snapshot before deployment.
  - Deploy schema changes that break the app.
  - Restore a new DB instance from the snapshot.
  - Point the app to the restored instance.
  - Expected result: the database returns to the snapshot state.

- **RDS accidental delete**
  - Use automated backups if the deletion happened within the retention window.
  - Restore to a point in time just before the event.
  - Expected result: the database is recovered to the chosen timestamp.

- **EC2 data volume corruption**
  - Snapshot the affected EBS volume if possible.
  - Create a new volume from the snapshot in the same AZ.
  - Attach it to the instance.
  - Expected result: the filesystem data returns to the snapshot state.

- **EC2 instance failure**
  - Use an AMI or launch template to create a replacement instance.
  - Reattach data volumes or restore them from snapshots.
  - Expected result: the server is rebuilt with the same software baseline.

**Operational best practices**

- **Test restores regularly** because a backup is only useful if recovery works.
- **Automate backup creation** with AWS Backup, snapshots, lifecycle policies, or scripts where appropriate.
- **Tag backup resources** so you can identify retention, owner, and environment.
- **Encrypt backups** with AWS Key Management Service (AWS KMS) when data sensitivity requires it.
- **Separate backup copies** across accounts or Regions for stronger disaster recovery.
- **Document RTO and RPO**:
  - **Recovery Time Objective (RTO)** is how fast you must recover.
  - **Recovery Point Objective (RPO)** is how much data loss you can tolerate.

**When to use each AWS feature**

- Use **S3 versioning** for accidental delete and overwrite protection.
- Use **S3 replication** for cross-Region or cross-account resilience.
- Use **S3 Object Lock** for immutable retention requirements.
- Use **RDS automated backups** when you need PITR.
- Use **RDS manual snapshots** before risky changes or for long-term restore points.
- Use **EBS snapshots** for volume-level recovery.
- Use **AMIs** for full EC2 instance rebuilds.

**Common exam traps**

- **Multi-AZ is not backup**: it improves availability, not historical recovery.
- **Replication is not version history**: it copies current objects, including bad writes.
- **EBS snapshots are not in-place restores**: you create a new volume from the snapshot.
- **RDS restore creates a new instance**: you do not overwrite the existing DB instance.
- **S3 Object Lock is for immutability**: it is not a general-purpose restore mechanism.

**Quick reference checklist**

- S3:
  - Enable versioning.
  - Use replication for DR.
  - Use Object Lock for compliance.
- RDS:
  - Set automated backup retention.
  - Take manual snapshots before changes.
  - Restore to a new instance.
- EC2:
  - Snapshot EBS volumes.
  - Restore volumes in the correct AZ.
  - Use AMIs for full instance recovery.

**Practical lab validation**

After each restore, verify the expected outcome rather than assuming success:

- For S3, confirm the object version ID and content hash.
- For RDS, confirm the schema version, row counts, and application connectivity.
- For EC2, confirm the filesystem mounts, service startup, and application logs.

A simple validation sequence for a restored Linux instance might look like this:

```bash
df -h
lsblk
sudo mount -a
systemctl status nginx
curl -I http://localhost
```

For a restored database, validate with a read-only query:

```sql
SELECT COUNT(*) FROM orders;
SELECT version();
```

The exam often tests whether you choose the right recovery mechanism for the failure mode. Think in terms of object, database, or volume recovery, then match the AWS feature to the required recovery point and recovery time.
