## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**High availability fundamentals**

High availability (HA) means designing systems to continue operating when components fail. In AWS, HA usually comes from removing single points of failure, spreading workloads across Availability Zones (AZs), and using automated recovery instead of manual intervention.

A strong HA design usually combines:
- **Redundancy** across instances, AZs, and sometimes Regions.
- **Automation** for health checks, replacement, and failover.
- **Statelessness** so any healthy node can serve traffic.
- **Decoupling** so one failing component does not cascade into others.

The exam often tests whether you choose the simplest design that meets the availability target. Multi-AZ is the default first step for most workloads; multi-Region is reserved for stricter recovery objectives or regional resilience requirements.

**Stateless vs stateful design**

Stateless applications are easier to scale and recover because any instance can handle any request. Stateful systems require careful handling of sessions, caches, queues, and data stores.

To improve HA, externalize mutable state:
- Store sessions in ElastiCache, DynamoDB, or a database instead of instance memory.
- Put uploads and shared files in Amazon S3 or Amazon EFS rather than local disks.
- Use managed databases with replication and failover instead of self-managed single-node databases.
- Keep application instances disposable so Auto Scaling can replace them without data loss.

> If your application depends on local instance storage for critical state, it is not truly highly available.

**Elastic Load Balancing for HA**

Elastic Load Balancing (ELB) distributes traffic across healthy targets and removes failed targets from rotation. The load balancer type matters.

**Application Load Balancer vs Network Load Balancer**

Use **Application Load Balancer (ALB)** when you need Layer 7 features:
- HTTP/HTTPS routing
- Host-based or path-based routing
- WebSocket support
- Content-aware routing to multiple services
- Integration with AWS WAF and authentication features

Use **Network Load Balancer (NLB)** when you need Layer 4 performance or static network behavior:
- TCP, UDP, and TLS traffic
- Very high throughput and low latency
- Static IP addresses or Elastic IPs
- Preserving client source IP
- Non-HTTP protocols and strict network appliances

For HA scenarios:
- Choose **ALB** for web applications, microservices, and content-based routing.
- Choose **NLB** for latency-sensitive services, TCP/UDP workloads, and systems that require static IPs.
- Use **NLB** when clients or partners must allowlist fixed IPs.
- Use **ALB** when you need intelligent routing across multiple target groups.

**Cross-zone load balancing behavior**

Cross-zone load balancing spreads traffic across targets in all enabled AZs rather than only within the same AZ as the incoming request.

- **ALB**: cross-zone load balancing is enabled by default.
- **NLB**: cross-zone load balancing is available, but behavior and cost considerations differ from ALB.
- Cross-zone balancing helps avoid uneven traffic distribution when one AZ has more requests than another.
- Without cross-zone balancing, an AZ with fewer healthy targets can become overloaded even if capacity exists elsewhere.

For exam questions, remember:
- Cross-zone balancing improves resilience to uneven traffic.
- It does not replace multi-AZ deployment.
- You still need healthy targets in multiple AZs for true HA.

**Static IPs and NLB use cases**

NLB supports static IP addresses, including Elastic IPs in some configurations. This is useful when:
- External firewalls require allowlisted IPs.
- Partners connect over fixed network rules.
- You need a stable entry point for TCP/UDP services.

Common examples:
- Game servers using UDP
- Financial or trading systems using TCP
- Legacy clients that cannot resolve DNS frequently
- Private connectivity patterns where fixed endpoints simplify routing

**Health checks for load balancers**

Health checks determine whether a target should receive traffic. Good health check design is critical because a weak check can keep broken instances in service, while an overly strict check can cause unnecessary failover.

Design guidance:
- Check an application endpoint that reflects real readiness, not just process liveness.
- Prefer a dedicated path such as `/health` or `/ready`.
- Make the endpoint verify dependencies only if those dependencies are required for serving traffic.
- Use sensible thresholds so transient blips do not trigger failover.

Typical settings to consider:
- **Interval**: how often the check runs.
- **Healthy threshold**: how many consecutive successes are needed.
- **Unhealthy threshold**: how many consecutive failures are needed.
- **Timeout**: how long to wait for a response.

Practical example:
```text
Health check path: /health
Interval: 30 seconds
Timeout: 5 seconds
Healthy threshold: 3
Unhealthy threshold: 2
```

This setup avoids flapping while still detecting failures quickly enough for most web workloads.

**Auto Scaling for availability**

Auto Scaling Groups (ASGs) improve HA by replacing unhealthy instances and maintaining desired capacity.

Key behaviors:
- Instances failing EC2 or ELB health checks can be terminated and replaced automatically.
- Desired capacity is maintained across AZs when configured for multi-AZ.
- Scaling policies add or remove capacity based on demand.
- Cooldowns or instance warmup settings prevent rapid oscillation after scaling events.

Important exam concepts:
- **Health check replacement**: unhealthy instances are removed and replaced.
- **Cooldowns**: prevent repeated scaling actions from reacting to the same metric spike.
- **Termination policies**: control which instance is removed first during scale-in.
- **Instance refresh**: rolls out a new launch template or configuration gradually.
- **Rolling updates**: replace instances in batches to preserve capacity during deployments.

Example launch template and ASG flow:
```bash
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name web-asg \
  --launch-template LaunchTemplateName=web-lt,Version='$Latest' \
  --min-size 2 \
  --max-size 6 \
  --desired-capacity 2 \
  --vpc-zone-identifier "subnet-aaa,subnet-bbb"
```

A safe deployment pattern is:
1. Create a new launch template version.
2. Start an instance refresh.
3. Replace instances in small batches.
4. Verify health before continuing.

**Multi-AZ application design**

Multi-AZ is the most common HA pattern in AWS. You place compute, load balancers, and data services across at least two AZs so an AZ failure does not take down the application.

Typical pattern:
- ALB spans multiple AZs.
- EC2 instances run in an Auto Scaling Group across multiple AZs.
- Session state is externalized.
- Database uses Multi-AZ or a managed HA service.
- Shared files use EFS or S3.

Benefits:
- Survives an AZ outage.
- Supports maintenance with minimal downtime.
- Usually meets common SAA-C03 HA requirements without the complexity of multi-Region design.

**Amazon RDS Multi-AZ and read replicas**

Amazon RDS Multi-AZ is designed for availability, not read scaling. It synchronously replicates data to a standby in another AZ and automatically fails over when the primary fails.

Important distinctions:
- **Multi-AZ**: synchronous replication, automatic failover, primary/standby architecture.
- **Read replica**: asynchronous replication, used for read scaling and some DR patterns, not automatic failover by default.

Exam-relevant details:
- Multi-AZ failover typically completes in minutes, but exact timing varies.
- Applications should reconnect after failover because the database endpoint remains the same while the underlying instance changes.
- Multi-AZ improves write availability, but it does not increase read throughput.
- Read replicas can lag behind the source, so they are not ideal for strict consistency requirements.

Common RDS HA pattern:
- Use Multi-AZ for the primary production database.
- Add read replicas for read-heavy workloads or cross-Region DR.
- Use application retry logic and connection pooling to handle failover.

**Aurora HA architecture**

Amazon Aurora uses a distributed storage architecture designed for high availability and fast recovery.

Core concepts:
- A cluster has one **writer** and zero or more **readers**.
- The **cluster endpoint** points to the writer.
- The **reader endpoint** load balances read traffic across available readers.
- Aurora storage is replicated across multiple AZs automatically.

Failover behavior:
- If the writer fails, Aurora promotes a reader to writer.
- Applications reconnect using the cluster endpoint.
- Failover is usually faster than traditional RDS Multi-AZ, though exact timing depends on workload and topology.

Useful patterns:
- Use the writer endpoint for all writes.
- Use the reader endpoint for read scaling.
- Add multiple readers for both availability and read throughput.
- Place readers in different AZs to improve resilience.

Aurora is often a strong choice when you need:
- Fast failover
- Read scaling
- Managed HA with minimal operational overhead

**Amazon EBS availability limits**

Amazon EBS volumes are AZ-scoped. That means a volume exists in one AZ and cannot be attached to an instance in another AZ.

HA implications:
- EBS is not a multi-AZ shared storage solution.
- If an instance or AZ fails, the volume is not automatically available elsewhere.
- Use snapshots, AMIs, or replication strategies for recovery.

Recovery patterns:
- Create EBS snapshots to S3-backed durable storage.
- Use AMIs to capture bootable system state.
- Restore volumes into another AZ during recovery.
- For shared file access across AZs, prefer EFS instead of EBS.

**Amazon EFS for shared HA storage**

Amazon Elastic File System (EFS) is a regional, multi-AZ file system designed for shared access across instances and AZs.

Why it helps HA:
- It is accessible from multiple AZs in the same Region.
- You create mount targets in each AZ where your EC2 instances run.
- If one AZ fails, instances in another AZ can continue mounting the same file system.

Operational notes:
- Use one mount target per AZ for low-latency access.
- Ensure security groups allow NFS traffic on port 2049.
- EFS is a good fit for shared application assets, uploads, and content that must survive instance replacement.

Example mount command:
```bash
sudo mount -t efs fs-12345678:/ /mnt/shared
```

**Decoupling with SQS, SNS, and EventBridge**

Decoupling reduces direct dependencies and improves availability. If one service is temporarily unavailable, messages or events can buffer work until consumers recover.

**Amazon SQS**
- Buffers work between producers and consumers.
- Prevents request spikes from overwhelming downstream services.
- Supports retry and dead-letter queue patterns.

**Amazon SNS**
- Fan-out messaging to multiple subscribers.
- Useful when one event must trigger several independent consumers.
- Helps avoid tight coupling between services.

**Amazon EventBridge**
- Routes events based on rules.
- Useful for event-driven architectures and cross-service integration.
- Reduces the need for synchronous service-to-service calls.

Practical HA pattern:
- Web tier writes jobs to SQS.
- Worker tier processes jobs asynchronously.
- SNS or EventBridge distributes notifications or domain events.
- If a worker AZ fails, messages remain available for another worker group.

**Route 53 routing policies for HA**

Amazon Route 53 provides DNS-based routing that supports both failover and traffic distribution across endpoints.

Common routing policies:
- **Failover routing**: active-passive design with health checks.
- **Latency-based routing**: sends users to the Region with the lowest latency.
- **Weighted routing**: splits traffic by percentage for blue/green, canary, or gradual migration.
- **Geolocation routing**: routes based on user location, useful for compliance or regional content.
- **Multivalue answer routing**: returns multiple healthy records, useful for simple client-side resilience.

How to choose:
- Use **failover** for primary/secondary architectures.
- Use **latency-based** for active-active multi-Region applications.
- Use **weighted** for controlled traffic shifting.
- Use **geolocation** when business rules depend on country or continent.
- Use **multivalue answer** when you want DNS to return several healthy endpoints without a full load balancer.

**Route 53 health checks**

Route 53 health checks determine whether DNS should return a record. They are especially important in failover and multivalue routing.

Design guidance:
- Check a meaningful endpoint, not just the root page.
- Use a path that validates application readiness, such as `/health` or `/status`.
- Set thresholds to avoid false positives from brief network issues.
- Consider checking a dependency only if the application cannot function without it.

Behavior to remember:
- Route 53 health checks are DNS-level signals, not application transactions.
- DNS caching means failover is not instantaneous for every client.
- Lower TTL values can reduce failover delay, but they increase DNS query frequency.

Example failover pattern:
```text
Primary record: app.example.com -> ALB in us-east-1
Secondary record: app.example.com -> ALB in us-west-2
Health check: /health on primary
TTL: 60 seconds
```

**CloudFront for edge availability**

Amazon CloudFront improves availability by caching content at edge locations and reducing dependence on a single origin.

HA benefits:
- Serves cached content even if the origin is slow or temporarily unavailable.
- Reduces latency for global users.
- Can fail over between origins in some architectures.

Common patterns:
- CloudFront in front of ALB for web applications.
- CloudFront in front of S3 for static websites.
- Origin failover between primary and secondary origins for resilience.

CloudFront is especially useful when:
- You need global performance and edge caching.
- Your origin should not receive all traffic directly.
- You want an additional layer of resilience for static or cacheable content.

**Active-active vs active-passive**

These are common exam patterns for multi-Region HA.

**Active-active**
- Multiple Regions serve traffic at the same time.
- Route 53 latency-based or weighted routing distributes users.
- Data replication must support concurrent reads and writes or carefully managed write ownership.
- Best for very high availability and global performance.

**Active-passive**
- One Region serves traffic while another stays on standby.
- Route 53 failover routing shifts traffic when the primary fails.
- Simpler data consistency model.
- Often easier and cheaper than active-active.

Selection guidance:
- Choose active-active when downtime tolerance is very low and the application can support distributed operation.
- Choose active-passive when you want simpler operations and lower cost.
- Choose active-active only if your data layer and application logic can handle it.

**Disaster recovery strategy selection**

AWS exam questions often ask you to choose the right recovery model based on RTO and RPO.

**Backup and restore**
- Lowest cost.
- Highest recovery time.
- Suitable for non-critical workloads or archival recovery.

**Pilot light**
- Core components are always running.
- Scale up during disaster.
- Good balance of cost and recovery speed.

**Warm standby**
- A smaller but functional environment is always running.
- Faster recovery than pilot light.
- Higher cost than pilot light.

**Active-active**
- Multiple environments serve traffic simultaneously.
- Fastest recovery and highest resilience.
- Highest complexity and cost.

Decision hints:
- If cost matters most and downtime is acceptable, choose backup and restore.
- If you need moderate recovery speed, choose pilot light or warm standby.
- If you need minimal downtime and global resilience, choose active-active.

**Practical multi-AZ web architecture**

A common HA web stack looks like this:

```text
Route 53
  -> ALB across 2+ AZs
      -> Auto Scaling Group across 2+ AZs
          -> App instances
      -> RDS Multi-AZ or Aurora cluster
      -> EFS for shared files
      -> SQS for async jobs
```

Why this works:
- Route 53 provides DNS-level routing and failover.
- ALB removes unhealthy instances from service.
- Auto Scaling replaces failed compute.
- RDS Multi-AZ or Aurora protects the database layer.
- EFS keeps shared files available across AZs.
- SQS buffers background work during spikes or partial outages.

**Practical multi-Region failover example**

A simple active-passive design might use:
- Primary Region: ALB, ASG, database, EFS or S3
- Secondary Region: warm standby stack
- Route 53 failover routing with health checks
- Cross-Region replication for data and backups

Example flow:
1. Primary Region serves traffic normally.
2. Route 53 health check fails on the primary endpoint.
3. DNS shifts new requests to the secondary Region.
4. Users reconnect to the standby stack.
5. Operators investigate and restore the primary.

**CLI examples for HA operations**

Check Auto Scaling group health:
```bash
aws autoscaling describe-auto-scaling-groups \
  --auto-scaling-group-names web-asg
```

Start an instance refresh:
```bash
aws autoscaling start-instance-refresh \
  --auto-scaling-group-name web-asg
```

Describe RDS DB instances:
```bash
aws rds describe-db-instances \
  --db-instance-identifier prod-db
```

Create a Route 53 health check:
```bash
aws route53 create-health-check \
  --caller-reference "prod-app-001" \
  --health-check-config '{
    "IPAddress":"203.0.113.10",
    "Port":443,
    "Type":"HTTPS",
    "ResourcePath":"/health",
    "FullyQualifiedDomainName":"app.example.com",
    "RequestInterval":30,
    "FailureThreshold":3
  }'
```

**Exam tips**

- Choose **multi-AZ** first when the question asks for high availability within a Region.
- Choose **multi-Region** when the question explicitly requires regional resilience or very low RTO.
- Choose **ALB** for HTTP/HTTPS and content-based routing.
- Choose **NLB** for TCP/UDP, static IPs, and low-latency network traffic.
- Choose **Aurora** when fast failover and read scaling matter.
- Choose **RDS Multi-AZ** when you need managed synchronous failover for a traditional relational database.
- Choose **EFS** for shared file storage across AZs.
- Choose **SQS/SNS/EventBridge** to decouple services and improve resilience.
- Choose **Route 53 latency-based routing** for active-active global designs.
- Choose **Route 53 failover routing** for active-passive designs.

**Common pitfalls**

- Treating read replicas as automatic failover targets.
- Using EBS as shared multi-AZ storage.
- Keeping session state on EC2 instance storage.
- Using a health check that only verifies the web server process, not the application.
- Assuming DNS failover is immediate for all clients.
- Overengineering with multi-Region when multi-AZ is sufficient.

**Quick comparison**

| Pattern | Best for | Key tradeoff |
|---|---|---|
| Multi-AZ | Regional HA | More cost than single-AZ |
| Multi-Region active-passive | DR with simpler ops | Slower failover |
| Multi-Region active-active | Global resilience | Higher complexity |
| NLB | TCP/UDP, static IPs | Less content-aware routing |
| ALB | HTTP/HTTPS apps | No static IPs by default |
| Aurora | Fast DB failover | Engine-specific behavior |
| RDS Multi-AZ | Managed relational HA | No read scaling |

**Summary**

High availability in AWS is about removing single points of failure and automating recovery. For SAA-C03, the most important patterns are multi-AZ compute, managed database failover, DNS-based routing, decoupled messaging, and choosing the right load balancer for the traffic type.

When in doubt, design for:
- Multiple AZs
- Stateless application tiers
- Managed failover
- Externalized state
- Automated health checks and replacement
