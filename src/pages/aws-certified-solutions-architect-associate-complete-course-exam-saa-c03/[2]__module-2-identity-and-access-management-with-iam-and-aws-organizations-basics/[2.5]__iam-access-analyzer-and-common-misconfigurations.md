## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**IAM Access Analyzer overview**

IAM Access Analyzer helps you identify when AWS resources are exposed beyond what you intended. It is built around the <abbr title="AWS Identity and Access Management">IAM</abbr> policy evaluation model and is especially useful for spotting unintended public, cross-account, or cross-organization access.

At a high level, you use it for three distinct tasks:

- **External access analysis** to find resources shared outside your account or organization.
- **Unused access analysis** to find IAM identities or credentials that are not being used.
- **Policy validation** to catch syntax and semantic issues before you deploy a policy.

These are related but not identical features. External and unused access are analyzer findings. Policy validation is a separate check that evaluates a policy document for mistakes and risky patterns.

**Why it matters for SAA-C03**

In exam scenarios, Access Analyzer often appears when you need to determine:

- whether an S3 bucket, KMS key, SQS queue, SNS topic, Secrets Manager secret, or IAM role trust policy is too open;
- whether a policy grants access to the wrong principal;
- whether a permission is never used and can be removed;
- whether a resource policy is valid for the intended AWS service integration.

**Core concepts**

<dl>
<dt><strong>Analyzer</strong></dt>
<dd>A regional or organization-level configuration that evaluates supported resources and policies for findings.</dd>
<dt><strong>Finding</strong></dt>
<dd>A result that indicates external access, unused access, or a policy issue.</dd>
<dt><strong>External access</strong></dt>
<dd>Access granted to principals outside your account or organization, depending on analyzer scope.</dd>
<dt><strong>Unused access</strong></dt>
<dd>Permissions, roles, or credentials that have not been used within the analyzer’s lookback window.</dd>
</dl>

**Analyzer scope and what it can see**

Access Analyzer behavior depends on where you create it:

- **Account analyzer** evaluates resources in one AWS account.
- **Organization analyzer** evaluates resources across accounts in an AWS Organizations organization, which is useful for centralized security teams.
- **Region scope** matters because many AWS resource policies are regional, and findings are generated per Region.

A practical mental model is:

- If you want to know “Who outside my boundary can access this resource?”, use **external access analysis**.
- If you want to know “Which identities or credentials are not being used?”, use **unused access analysis**.
- If you want to know “Is this policy valid and safe to deploy?”, use **policy validation**.

**Quick decision flow**

```text
Need to check a policy before deployment?
  -> Use policy validation

Need to find resources shared outside your account/org?
  -> Use external access analysis

Need to remove stale permissions or credentials?
  -> Use unused access analysis
```

**External access analysis**

External access analysis inspects supported resource-based policies and trust policies to determine whether a resource is accessible from outside your intended boundary.

Common examples include:

- **S3 bucket policies** that allow `Principal: "*"` or a broad AWS account list.
- **KMS key policies** that allow external principals to use the key.
- **SNS topic policies** that allow publishing from unexpected principals.
- **SQS queue policies** that allow cross-account send or receive access.
- **Secrets Manager resource policies** that expose secrets to other accounts.
- **IAM role trust policies** that allow assumption by external accounts or services.

A finding does not always mean “public.” It may mean:

- access from another AWS account;
- access from another organization;
- access from a service principal that you did not intend;
- access through a wildcard principal combined with permissive conditions.

**Common external access patterns**

- `Principal: "*"` without a restrictive condition is usually a red flag.
- Cross-account access is valid when intentional, but you should scope it tightly.
- Service principals are valid for AWS service integrations, but the exact principal and condition keys must match the service.

> Be careful with service integrations: the correct principal, `SourceArn`, `SourceAccount`, and other condition keys vary by service. A pattern that works for one service may be invalid or overly broad for another.

**Example: S3 bucket policy with unintended public access**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "TooBroadReadAccess",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

This policy allows anyone on the internet to read objects if the bucket is otherwise reachable. Access Analyzer can flag this as external access.

A safer pattern is to scope access to a specific principal or to use conditions such as VPC endpoint restrictions, AWS Organizations conditions, or explicit account IDs where appropriate.

**Example: KMS key policy with service-restricted use**

When you allow a service to use a KMS key, the request typically comes through that service, not directly from the IAM principal. A condition such as `kms:ViaService` restricts usage to requests made via a specific AWS service.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowUseViaEBSOnly",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AppRole"
      },
      "Action": [
        "kms:Decrypt",
        "kms:Encrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "kms:ViaService": "ec2.us-east-1.amazonaws.com"
        }
      }
    }
  ]
}
```

This means the role can use the key only when the request is made through the specified service path. Direct KMS API calls from the role do not satisfy the condition.

**Unused access analysis**

Unused access analysis helps you identify stale IAM access that may be safe to remove. This is useful for least privilege and for reducing blast radius.

Typical findings include:

- **Unused IAM roles** that have not been assumed during the lookback period.
- **Unused access keys** for IAM users.
- **Unused passwords or console credentials** for IAM users, depending on the finding type and configuration.

This feature is about usage, not about whether a permission is technically present in a policy. It answers questions like:

- Has this role been used recently?
- Has this access key been used?
- Is this credential still active but dormant?

It does not generally mean “this permission statement was never exercised” in the same way a policy simulator might evaluate actions. Keep that distinction clear.

**Why unused access matters**

- Old access keys are a common compromise vector.
- Dormant roles can hide forgotten trust relationships.
- Removing unused credentials reduces operational and security risk.

**Example remediation workflow for unused access**

1. Review the finding and confirm the identity is truly inactive.
2. Check CloudTrail for recent activity if you need additional confidence.
3. Disable or delete the unused access key, or remove the unused role if it is no longer needed.
4. Notify application owners before removing credentials used by automation.
5. Re-run analysis to confirm the finding is resolved.

**Policy validation**

Policy validation checks whether a policy is syntactically and semantically correct and whether it contains risky constructs. It is especially useful before deployment through CloudFormation, Terraform, or direct CLI/API updates.

Validation can help catch:

- malformed JSON;
- invalid actions or resource ARNs;
- unsupported condition keys;
- overly permissive statements;
- missing resource-type context for resource policies.

For resource policies, validation often needs the specific resource type to interpret the policy correctly. For example, a resource policy for an S3 bucket is not the same as one for an SNS topic or KMS key.

**AWS CLI examples**

Validate an identity policy:

```bash
aws accessanalyzer validate-policy \
  --policy-document file://policy.json \
  --policy-type IDENTITY_POLICY
```

Validate a resource policy with a resource type:

```bash
aws accessanalyzer validate-policy \
  --policy-document file://bucket-policy.json \
  --policy-type RESOURCE_POLICY \
  --validate-policy-resource-type AWS::S3::Bucket
```

Validate a trust policy:

```bash
aws accessanalyzer validate-policy \
  --policy-document file://trust-policy.json \
  --policy-type TRUST_POLICY
```

The exact resource type value depends on the AWS resource you are validating. Use the service-specific type expected by the CLI and API.

**Common misconfigurations Access Analyzer helps uncover**

**1. Wildcard principals**

A wildcard principal is often the fastest way to create unintended exposure.

Examples:

- `Principal: "*"` in a bucket policy.
- `Principal: { "AWS": "*" }` in a resource policy.
- Trust policies that allow broad assumption without conditions.

Use wildcard principals only when the resource is intentionally public or when the service integration requires it and you have strong conditions in place.

**2. Missing or weak conditions**

Conditions are often what make a policy safe.

Useful condition keys include:

- `aws:PrincipalArn` to scope to a specific principal pattern.
- `aws:PrincipalOrgID` to limit access to your AWS Organization.
- `aws:SourceArn` to bind a service invocation to a specific resource.
- `aws:SourceAccount` to reduce confused deputy risk.
- `kms:ViaService` to force KMS usage through a specific AWS service.

If you omit these when they are needed, Access Analyzer may flag the policy or you may create a real exposure.

**3. Overly broad trust policies**

IAM role trust policies determine who can assume a role. A common mistake is trusting too many principals or trusting an entire account without conditions.

Example of a broad trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::444455556666:root"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

This allows any principal in that account with permission to assume the role, which may be broader than intended. A tighter policy usually names a specific role or uses conditions.

**4. Public or cross-account data stores**

Commonly affected services include:

- **Amazon S3** buckets and objects;
- **Amazon SQS** queues;
- **Amazon SNS** topics;
- **AWS KMS** keys;
- **AWS Secrets Manager** secrets;
- **Amazon ECR** repositories.

For ECR, repository policies can allow cross-account pull or push. That is valid for shared images, but it should be intentional and scoped to the right account or role.

Example ECR repository policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountPull",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": [
        "ecr:BatchGetImage",
        "ecr:GetDownloadUrlForLayer"
      ],
      "Resource": "arn:aws:ecr:us-east-1:999988887777:repository/app-repo"
    }
  ]
}
```

This is acceptable if you truly want cross-account image pulls, but it should be reviewed carefully.

**5. Secrets Manager resource policy mistakes**

Secrets Manager supports resource-based policies for cross-account access. A common mistake is granting broad read access to a secret without limiting the principal or the source.

Safer patterns include:

- granting access to a specific role in another account;
- using `aws:PrincipalOrgID` when appropriate;
- combining resource policy permissions with KMS key policy permissions if the secret uses a customer managed key.

**6. SNS and CloudWatch alarm publishing**

CloudWatch alarms can publish to SNS topics, but the service principal and conditions must match the integration.

Example pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudWatchAlarmsToPublish",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudwatch.amazonaws.com"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:123456789012:alarm-topic",
      "Condition": {
        "ArnLike": {
          "aws:SourceArn": "arn:aws:cloudwatch:us-east-1:123456789012:alarm:*"
        }
      }
    }
  ]
}
```

This is a common integration pattern, but you should verify the exact service requirements for the AWS service you are integrating. Not every service uses the same principal or source condition.

**How to investigate a finding**

1. Open the finding and identify the resource, principal, and access path.
2. Determine whether the access is intentional.
3. Check whether the policy is resource-based, identity-based, or a trust policy.
4. Review the conditions and confirm they are restrictive enough.
5. Decide whether to tighten, replace, or remove the statement.
6. Re-test with policy validation and, if needed, IAM Access Analyzer again.

**Practical remediation examples**

**S3 bucket exposed to another account**

- Replace `Principal: "*"` with a specific AWS account or role.
- Add `aws:PrincipalOrgID` if the access should stay within your organization.
- Consider using bucket policies plus S3 Block Public Access for defense in depth.

**KMS key shared too broadly**

- Restrict the principal to the exact role or service.
- Add `kms:ViaService` when the key should only be used through a service.
- Ensure the key policy and IAM permissions both allow the intended use.

**IAM role trust policy too open**

- Replace account-root trust with a specific role ARN.
- Add `ExternalId` for third-party access when appropriate.
- Use `aws:PrincipalOrgID` for organization-wide trust patterns if that matches your design.

**Unused IAM user access key**

- Disable the key first if you need a rollback window.
- Confirm no application or script depends on it.
- Delete the key after validation.
- Rotate any downstream secrets if compromise is suspected.

**CLI commands you should know**

List analyzers:

```bash
aws accessanalyzer list-analyzers
```

List findings for an analyzer:

```bash
aws accessanalyzer list-findings \
  --analyzer-arn arn:aws:access-analyzer:us-east-1:123456789012:analyzer/example-analyzer
```

Get a specific finding:

```bash
aws accessanalyzer get-finding \
  --analyzer-arn arn:aws:access-analyzer:us-east-1:123456789012:analyzer/example-analyzer \
  --id finding-id-123
```

Archive a finding after you have reviewed and accepted it:

```bash
aws accessanalyzer update-findings \
  --analyzer-arn arn:aws:access-analyzer:us-east-1:123456789012:analyzer/example-analyzer \
  --ids finding-id-123 \
  --status ARCHIVED
```

Create an analyzer:

```bash
aws accessanalyzer create-analyzer \
  --analyzer-name org-analyzer \
  --type ORGANIZATION
```

**Exam-oriented distinctions**

| Feature | What it checks | Typical output | When to use |
|---|---|---|---|
| External access analysis | Resource exposure outside your boundary | Finding | Review shared resources |
| Unused access analysis | Dormant roles or credentials | Finding | Reduce stale access |
| Policy validation | Policy correctness and risky patterns | Validation result | Before deployment |

**Real-world scenario 1: cross-account S3 sharing**

A data team wants to share a bucket with a partner account. They add a bucket policy that allows the partner role to read objects. Access Analyzer flags the bucket.

What you should check:

- Is the principal the exact partner role ARN?
- Is the resource ARN scoped to the bucket or prefix you intended?
- Are there conditions that limit access further?
- Is public access blocked elsewhere?

If the sharing is intentional, the finding may be expected and can be documented or archived after review. If not, tighten the policy.

**Real-world scenario 2: stale automation credentials**

An application team rotates to IAM roles for compute, but an old IAM user access key remains active. Access Analyzer reports unused access.

What you should do:

- Confirm the key is not used by a legacy script.
- Replace the key with an IAM role or temporary credentials.
- Disable and then delete the key.
- Review CloudTrail for any suspicious historical use.

This is a common least-privilege cleanup task and a strong exam pattern.

**Real-world scenario 3: service integration with KMS**

An application stores encrypted data in an AWS service that uses a customer managed KMS key. The key policy allows the application role directly, but the service actually calls KMS on the application’s behalf.

What you should verify:

- Does the service require `kms:ViaService`?
- Does the key policy allow the service path?
- Does the IAM role also have the necessary permissions?
- Are you confusing direct KMS access with service-mediated access?

This distinction is important because KMS authorization often involves both the key policy and the calling context.

**Best practices**

- Use **least privilege** in both identity and resource policies.
- Prefer specific principals over account roots and wildcards.
- Add conditions that bind access to the intended service, account, or organization.
- Review findings regularly, not only after incidents.
- Validate policies before deployment to catch mistakes early.
- Treat unused access findings as cleanup opportunities, but confirm business impact first.
- Use organization-level analyzers when you manage multiple accounts centrally.

**Common exam traps**

- Confusing policy validation with analyzer findings.
- Assuming `Principal: "*"` is acceptable if the action seems harmless.
- Forgetting that service integrations often require `SourceArn` or `SourceAccount`.
- Treating unused access as the same thing as unused permissions.
- Assuming a KMS permission works for direct API calls when the policy is actually restricted by `kms:ViaService`.
- Thinking all AWS services use the same resource-policy pattern.

**Summary**

IAM Access Analyzer is a security and governance tool that helps you detect unintended access, stale credentials, and policy mistakes. For SAA-C03, focus on the difference between external access analysis, unused access analysis, and policy validation. In practice, the most common fixes are to narrow principals, add conditions, remove stale credentials, and verify service-specific integration requirements before you trust a policy.
