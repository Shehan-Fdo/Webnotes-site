## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**IAM identity fundamentals**

AWS Identity and Access Management (<abbr title="Identity and Access Management">IAM</abbr>) controls who can access AWS resources and what they can do. In this lab, you focus on two core identity types:

- **Users** represent individual people or workloads that need long-term credentials.
- **Groups** are collections of users that share the same permissions.
- **Policies** define permissions in JSON and attach to users, groups, or roles.
- **Least privilege** means granting only the permissions required for the task.

IAM is a global service, not tied to a specific AWS Region. That matters because user and group configuration applies account-wide. For exam scenarios, prefer groups over attaching the same policy to many users individually, because groups simplify administration and reduce drift.

**When to use users and groups**

Use IAM users when you need a named identity with long-term access keys or console access. Use groups when multiple users need the same baseline permissions, such as a team of developers or a read-only operations group. For applications running on AWS, prefer IAM roles instead of users whenever possible.

- **IAM user**: one identity for one person or workload.
- **IAM group**: permission container for many users.
- **IAM policy**: permission document in JSON.
- **IAM role**: temporary credentials for AWS services, federated users, or cross-account access.

> Avoid using the root user for daily administration. Use it only for account-level tasks that require root privileges, such as changing the account email or closing the account.

**Creating users in the AWS Management Console**

You typically create users from the IAM console and decide whether they need console access, programmatic access, or both. Console access uses a password; programmatic access uses access keys for the AWS CLI, SDKs, or API calls.

1. Open the IAM console.
2. Choose **Users** and then **Create user**.
3. Enter a user name.
4. Select access type based on the use case:
   - **AWS Management Console access** for browser sign-in.
   - **Access key** for CLI or SDK access.
5. Set password options if console access is enabled.
6. Add the user to one or more groups, or attach policies directly if needed.
7. Review and create the user.
8. Save credentials securely if access keys are generated.

A practical example is creating a developer user who can sign in to the console and use the CLI:

```bash
aws iam create-user --user-name dev1
aws iam create-login-profile \
  --user-name dev1 \
  --password 'TempPassword123!' \
  --password-reset-required

aws iam create-access-key --user-name dev1
```

The `create-login-profile` command enables console sign-in for an IAM user. The access key command returns an access key ID and secret access key; the secret is shown only once, so you must store it securely.

**Creating and managing groups**

Groups let you manage permissions at scale. Instead of attaching the same policy to every user, you attach it once to the group and add users as needed. This is the standard pattern for teams with shared access requirements.

Common examples include:

- **Developers** with access to development resources.
- **Support** with read-only access to logs and monitoring.
- **Billing** with access to billing-related console pages.
- **Admins** with elevated permissions, though you should still scope carefully.

You can create a group in the console or with the CLI:

```bash
aws iam create-group --group-name Developers
aws iam add-user-to-group --user-name dev1 --group-name Developers
aws iam add-user-to-group --user-name dev2 --group-name Developers
```

Groups do not have credentials of their own. They only aggregate permissions for their members. If a user belongs to multiple groups, AWS combines the allowed permissions from all applicable policies, subject to explicit denies and any permission boundaries or service control policies in effect.

**Attaching permissions to groups**

The most common lab pattern is to attach a managed policy to a group. Managed policies are reusable and easier to maintain than inline policies. For example, you might attach `AmazonS3ReadOnlyAccess` to a read-only team group.

```bash
aws iam attach-group-policy \
  --group-name Developers \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

You can also create a custom customer-managed policy for tighter control. This is useful when AWS managed policies are broader than your requirements.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListAndReadSpecificBucket",
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket",
        "s3:GetObject"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

This policy allows listing the bucket and reading objects only in `example-bucket`. It does not grant write access or access to other buckets.

**Console sign-in and credential management**

If you enable console access, IAM creates a password-based login profile for the user. You should require a password reset on first sign-in when creating temporary credentials for another person. For CLI access, create access keys and configure them locally.

```bash
aws configure
```

This command stores credentials in `~/.aws/credentials` and default settings in `~/.aws/config`. On a shared system, protect these files carefully because they contain sensitive material.

Useful credential hygiene practices include:

- Rotate access keys regularly.
- Delete unused users and keys.
- Prefer temporary credentials when possible.
- Use MFA for privileged users.
- Avoid embedding credentials in scripts or source code.

**Verifying access with the CLI**

After assigning a user to a group, test the effective permissions. A common verification step is to use the CLI as that user and attempt an allowed and a denied action.

```bash
aws sts get-caller-identity
aws s3 ls s3://example-bucket
aws s3 cp test.txt s3://example-bucket/
```

`sts get-caller-identity` confirms which identity is active. If the user has only read permissions, the `ls` command may succeed while the `cp` upload fails with an access denied error. This is a useful way to validate least-privilege design.

**Permission evaluation basics**

AWS evaluates permissions by combining identity-based policies, resource-based policies, and any higher-level controls. For this lab, the key idea is that group membership contributes identity-based permissions to the user.

- **Allow** statements grant access.
- **Explicit deny** overrides any allow.
- **Implicit deny** applies when no allow exists.
- **Policy inheritance** from groups applies to all group members.

A user can belong to multiple groups, and permissions are additive unless a deny blocks them. This is why group design should be intentional and easy to audit.

**Practical lab workflow**

A typical lab flow looks like this:

1. Create a group for a job function.
2. Attach a managed policy or custom policy to the group.
3. Create one or more IAM users.
4. Add the users to the group.
5. Test access with the console or CLI.
6. Adjust the policy if the permissions are too broad or too narrow.

This workflow mirrors real administration tasks and is a common exam pattern. If a scenario asks how to grant the same permissions to many users, the best answer is usually a group with an attached policy.

**Troubleshooting common issues**

If a user cannot access a resource, check the following:

- The user is in the expected group.
- The group has the correct policy attached.
- The policy resource ARN matches the target resource.
- There is no explicit deny in another policy.
- The user is not missing required MFA conditions.
- The credentials are valid and not expired.

If console sign-in fails, verify:

- The login profile exists.
- The password is correct.
- The account alias or sign-in URL is correct.
- The user is not required to reset a password and has not completed the reset flow.

If CLI access fails, verify:

- The access key is active.
- The AWS CLI is using the intended profile.
- The region and endpoint are correct for the service.
- The local credentials file contains the expected values.

**Exam-relevant takeaways**

IAM users and groups are foundational for account access management, but they are not the preferred solution for workloads. For the exam, remember these patterns:

- Use **groups** to manage permissions for multiple users.
- Use **managed policies** for reusable permission sets.
- Use **custom policies** for tighter scoping.
- Use **roles** for AWS services and temporary access.
- Use **MFA** for sensitive or administrative access.
- Use **least privilege** to reduce risk.

A simple mental model helps: users authenticate, groups organize permissions, and policies authorize actions. If you keep those roles distinct, you can reason through most IAM questions quickly and accurately.
