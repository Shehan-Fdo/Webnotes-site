## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Policy evaluation overview**

AWS authorization is a decision process that determines whether a request is allowed to perform an action on a resource. You usually design for the <mark>least privilege</mark> principle: grant only the permissions required for a task, and nothing more. In practice, this means you combine identity-based policies, resource-based policies, permissions boundaries, service control policies (SCPs), and sometimes session policies to shape the final effective permissions.

The key exam idea is that AWS evaluates permissions by looking for any applicable <abbr title="explicit deny">Deny</abbr> first, then checking whether an <abbr title="explicit allow">Allow</abbr> exists. If no allow matches, the request is denied by default.

- **Default deny** means access is blocked unless a policy explicitly allows it.
- **Explicit deny** always wins over any allow.
- **Least privilege** reduces blast radius if credentials are compromised.
- **Policy scope** matters because different policy types apply at different layers.

**Core evaluation logic**

AWS authorization is not a single policy check. It is the result of multiple policy sources that can overlap. A request succeeds only when all applicable guardrails permit it and at least one applicable allow grants it.

A useful mental model is:

1. Start with an implicit deny.
2. Evaluate all applicable policies.
3. If any policy returns an explicit deny, stop and deny the request.
4. If no deny exists, look for a matching allow.
5. If no allow exists, deny the request.

This logic applies broadly across IAM (Identity and Access Management), but the exact policy types involved depend on the resource and the AWS service.

- **Identity-based policies** attach to users, groups, or roles and grant permissions to identities.
- **Resource-based policies** attach to resources such as S3 buckets, KMS keys, SNS topics, or SQS queues.
- **Permissions boundaries** limit the maximum permissions an identity can receive.
- **SCPs** set the maximum permissions available in an AWS Organizations account or OU.
- **Session policies** further restrict temporary credentials from STS (AWS Security Token Service).

> An explicit deny in any applicable policy overrides all allows, even if the allow comes from another policy type.

**Least privilege design**

Least privilege is not just a security slogan; it is a practical design method. You begin with the smallest set of actions, resources, and conditions needed for a workload, then expand only when a real requirement appears. This is especially important in AWS because many services support fine-grained resource ARNs and condition keys.

Good least-privilege policies are specific in three dimensions:

- **Action**: allow only the exact API operations needed, such as `s3:GetObject` instead of `s3:*`.
- **Resource**: scope permissions to specific ARNs instead of `*` whenever possible.
- **Condition**: add context constraints such as source IP, VPC endpoint, MFA, or tags.

A common pattern is to start with a broad policy during development, observe actual usage, and then tighten it. AWS tools such as IAM Access Analyzer and CloudTrail help you identify unused or overbroad permissions.

**Example: overly broad policy**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:*",
      "Resource": "*"
    }
  ]
}
```

This works, but it violates least privilege because it grants every S3 action on every bucket and object.

**Example: tighter policy for read-only access to one bucket**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket"
      ],
      "Resource": "arn:aws:s3:::my-app-bucket"
    },
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::my-app-bucket/*"
    }
  ]
}
```

This separates bucket-level and object-level permissions, which is a common AWS pattern.

**How AWS evaluates multiple policy types**

Different policy types combine in different ways, but the exam usually tests the following principles:

- **Identity-based + resource-based**: both can grant access, and either may be sufficient depending on the service and principal type.
- **Permissions boundary**: acts as a ceiling; the identity cannot exceed it even if another policy allows more.
- **SCP**: acts as an organization-level ceiling; it does not grant permissions by itself.
- **Session policy**: reduces permissions for a specific STS session.
- **Explicit deny**: overrides everything else.

A practical way to think about it is that AWS first checks whether the request is allowed by the identity and resource context, then checks whether any higher-level guardrail blocks it.

**Identity-based policies**

Identity-based policies are the most common IAM policies. They are attached to IAM users, groups, or roles and define what those identities can do.

Use them when:

- You want to grant permissions to a human user or application role.
- You want centralized permission management.
- You need reusable access across multiple resources.

Example IAM policy for an EC2 instance role that can read from one S3 bucket:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadAppData",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::app-data-bucket",
        "arn:aws:s3:::app-data-bucket/*"
      ]
    }
  ]
}
```

**Resource-based policies**

Resource-based policies are attached directly to the resource. They are common when the resource itself needs to trust a principal from the same or another account.

Use them when:

- Cross-account access is needed.
- The service supports resource policies.
- You want the resource owner to control access directly.

Examples include:

- S3 bucket policies
- KMS key policies
- SNS topic policies
- SQS queue policies
- Lambda function resource policies

A bucket policy can allow a specific role from another account to read objects:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AnalyticsRole"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-shared-bucket/*"
    }
  ]
}
```

**Explicit deny patterns**

Explicit deny is one of the most important exam concepts because it is absolute. You use it to create guardrails that should never be bypassed.

Common uses:

- Block access from outside a corporate network.
- Prevent deletion of critical resources.
- Deny actions unless MFA is present.
- Restrict access to specific regions.

Example: deny S3 object deletion for everyone except a break-glass role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyDeleteObject",
      "Effect": "Deny",
      "Action": "s3:DeleteObject",
      "Resource": "arn:aws:s3:::critical-bucket/*"
    }
  ]
}
```

Even if another policy allows `s3:DeleteObject`, this deny still blocks it.

**Conditions and context keys**

Conditions make policies more precise by adding context. They are essential for least privilege because they let you say not only *what* can happen, but *when*, *from where*, and *under what circumstances*.

Common condition keys include:

- `aws:SourceIp` for IP-based restrictions
- `aws:MultiFactorAuthPresent` for MFA enforcement
- `aws:RequestedRegion` for region restrictions
- `aws:PrincipalTag` for tag-based access control
- `aws:SourceVpce` for VPC endpoint restrictions

Example: require MFA for sensitive actions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "RequireMFA",
      "Effect": "Deny",
      "Action": [
        "iam:DeleteUser",
        "iam:DeleteRole",
        "iam:PutRolePolicy"
      ],
      "Resource": "*",
      "Condition": {
        "BoolIfExists": {
          "aws:MultiFactorAuthPresent": "false"
        }
      }
    }
  ]
}
```

This pattern is powerful because it denies dangerous actions unless the session proves MFA was used.

**Permissions boundaries**

A permissions boundary is an advanced IAM feature that limits the maximum permissions an identity can have. It does not grant permissions on its own. Instead, it defines the outer edge of what identity-based policies can allow.

Use permissions boundaries when:

- You delegate IAM administration to teams but want to prevent privilege escalation.
- You create roles or users programmatically and need a safety cap.
- You want to let developers manage their own roles within approved limits.

Think of it as a ceiling:

- Identity policy says what the identity wants.
- Permissions boundary says what the identity is allowed to ever get.

Example conceptually:

- Identity policy allows `ec2:*`
- Permissions boundary allows only `ec2:Describe*`
- Effective permission is only `ec2:Describe*`

**Service control policies**

SCPs are organization-level guardrails in AWS Organizations. They apply to accounts, organizational units, and the root of the organization. SCPs do not grant permissions; they only define the maximum permissions available in member accounts.

Use SCPs when:

- You need to enforce compliance across many accounts.
- You want to block risky services or regions globally.
- You need centralized governance without editing every IAM policy.

Important exam points:

- SCPs affect IAM users and roles in member accounts.
- The management account is treated differently and is not governed the same way as member accounts for all actions.
- An SCP must allow an action for it to be possible, but an IAM policy still must also allow it.

Example SCP that denies use of a region:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

This is a common governance pattern for multi-account environments.

**Session policies and temporary credentials**

When you assume a role with STS, you receive temporary credentials. A session policy can further restrict those credentials. This is useful when you want to narrow access for a specific task without changing the underlying role.

Use session policies when:

- You need temporary, task-specific restrictions.
- You want to reduce permissions for a federated session.
- You want to pass a narrower scope to automation.

Example AWS CLI assume-role call with a session policy:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/DeployRole \
  --role-session-name deploy-session \
  --policy '{
    "Version":"2012-10-17",
    "Statement":[
      {
        "Effect":"Allow",
        "Action":["s3:GetObject"],
        "Resource":"arn:aws:s3:::release-artifacts/*"
      }
    ]
  }'
```

The session policy does not expand permissions; it only narrows them.

**Practical evaluation examples**

A request is allowed only when all relevant checks pass. Consider these scenarios:

- A user policy allows `ec2:StartInstances`, but an SCP denies all EC2 actions in the account.
  - Result: denied.
- A role policy allows `s3:GetObject`, but the bucket policy denies access from outside a VPC endpoint.
  - Result: denied outside the endpoint.
- A user policy allows `iam:CreateRole`, but the permissions boundary excludes IAM write actions.
  - Result: denied.
- A policy allows `dynamodb:PutItem`, but no statement matches the table ARN.
  - Result: denied by default.

**CLI tools for troubleshooting**

You often validate policy behavior with AWS CLI and service-specific logs. The IAM policy simulator is especially useful for understanding why a request is denied.

Useful commands:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:user/Alice \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::my-app-bucket/report.csv
```

You can also inspect the caller identity to confirm which principal is making the request:

```bash
aws sts get-caller-identity
```

For S3 access issues, test with a direct request:

```bash
aws s3api get-object \
  --bucket my-app-bucket \
  --key report.csv \
  /tmp/report.csv
```

If the request fails, check:

- The identity policy
- The bucket policy
- Any SCPs
- Any permissions boundary
- Whether the resource ARN is correct
- Whether a condition key is blocking access

**Common least-privilege patterns**

These patterns appear frequently in real architectures and exam questions:

- **Separate read and write roles** for applications.
- **Use one role per workload** instead of sharing a broad admin role.
- **Scope S3 permissions** to bucket and prefix level.
- **Use condition keys** for network, MFA, and region restrictions.
- **Prefer roles over long-term access keys** for workloads.
- **Use managed policies for reusable baseline access** and inline policies for tightly scoped exceptions.
- **Review permissions regularly** with Access Analyzer and CloudTrail.

**Typical exam traps**

The exam often tests subtle differences between policy types and evaluation order.

Watch for these traps:

- Thinking an allow in one policy can override a deny in another policy.
- Assuming SCPs grant permissions instead of limiting them.
- Confusing resource-based policies with identity-based policies.
- Forgetting that `*` in `Action` or `Resource` is broad and often violates least privilege.
- Assuming a role can exceed its permissions boundary.
- Ignoring condition keys that silently block access.

**Design guidance**

When you design access control in AWS, start from the workload requirement and work backward:

1. Identify the exact API actions needed.
2. Identify the exact resources involved.
3. Add conditions for environment, network, or MFA requirements.
4. Apply guardrails with SCPs or permissions boundaries where appropriate.
5. Test with the IAM policy simulator and real service calls.
6. Remove any permissions that are not actively required.

This approach gives you predictable authorization and reduces the risk of accidental overexposure.

**Quick comparison**

| Concept | What it does | Common use |
|---|---|---|
| Identity-based policy | Grants permissions to a user, group, or role | App and human access |
| Resource-based policy | Grants access to a resource | Cross-account or service trust |
| Permissions boundary | Sets the maximum permissions for an identity | Delegated IAM administration |
| SCP | Sets the maximum permissions in an account or OU | Organization-wide guardrails |
| Session policy | Narrows temporary credentials | Task-specific access |

**Key takeaways**

- AWS authorization starts with implicit deny.
- Explicit deny overrides every allow.
- Least privilege means narrowing actions, resources, and conditions.
- Identity policies grant permissions; resource policies trust principals.
- Permissions boundaries and SCPs limit maximum permissions.
- Session policies narrow temporary credentials.
- Troubleshooting requires checking every applicable policy layer, not just one.
