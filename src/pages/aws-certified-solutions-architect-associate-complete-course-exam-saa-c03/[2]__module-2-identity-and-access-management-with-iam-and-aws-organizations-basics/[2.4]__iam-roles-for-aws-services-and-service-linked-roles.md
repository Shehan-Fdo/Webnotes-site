## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**IAM roles for AWS services: the core idea**

An IAM <abbr title="Identity and Access Management">IAM</abbr> role is an identity with permissions that can be assumed by a trusted principal. For AWS services, roles let a service perform actions on your behalf without embedding long-term credentials in code or configuration.

The exam usually tests whether you understand:
- **Who can assume the role**: controlled by the trust policy.
- **What the role can do**: controlled by the permissions policy.
- **Who manages the role**: customer-managed for service roles, AWS-managed for service-linked roles.

A role is not a user. It has no permanent password or access keys by default. Instead, a principal assumes it and receives temporary credentials from AWS Security Token Service (<abbr title="Security Token Service">STS</abbr>).

**Role policy model: trust policy vs permissions policy**

A role has two distinct policy types, and confusing them is a common exam trap.

- **Trust policy**
  - Defines the principals allowed to assume the role.
  - Uses the `sts:AssumeRole` action.
  - Answers: “Who can use this role?”
- **Permissions policy**
  - Defines what the role can do after it is assumed.
  - Answers: “What actions are allowed?”

A simple service role trust policy for EC2 might look like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

A matching permissions policy might allow the instance to read from S3:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
      ]
    }
  ]
}
```

**IAM roles for AWS services**

A service role is a role that an AWS service assumes to perform operations in your account. You create and manage the role, choose the trust relationship, and attach the permissions the service needs.

Common use cases include:
- EC2 instances accessing S3, DynamoDB, or Secrets Manager.
- Lambda functions reading from S3 or writing to CloudWatch Logs.
- ECS tasks pulling images and accessing application data.
- RDS features integrating with other AWS services.
- EKS workloads using IAM for service accounts in Kubernetes-based access patterns.

Service roles are useful when you want:
- Fine-grained control over permissions.
- Reuse of the same role across multiple resources of the same service.
- Clear separation between application identity and human IAM users.

**How service roles work in practice**

You usually attach the role to a resource or configure the service to assume it. The service then requests temporary credentials from STS and uses those credentials to call AWS APIs.

Typical examples:
- An EC2 instance profile provides a role to the instance metadata service.
- A Lambda execution role is assumed by the Lambda service when the function runs.
- An ECS task role is assumed by the task, not by the EC2 host, when the task needs AWS API access.

Example CLI flow for creating a customer-managed service role:

```bash
aws iam create-role \
  --role-name MyAppEc2Role \
  --assume-role-policy-document file://trust-policy.json

aws iam attach-role-policy \
  --role-name MyAppEc2Role \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

If you need a custom policy instead of a managed policy, create and attach it explicitly:

```bash
aws iam put-role-policy \
  --role-name MyAppEc2Role \
  --policy-name MyAppS3Access \
  --policy-document file://permissions-policy.json
```

**Service-linked roles**

A service-linked role is a distinct IAM role type that is linked to a specific AWS service. AWS creates and manages it for that service, and the role is intended to let the service perform actions in your account with the permissions it needs.

Key exam points:
- **AWS manages the trust policy** for the service-linked role.
- The role is tied to a single service and is not meant for cross-service reuse.
- The role usually has a service-linked role path and a name with the `AWSServiceRoleFor*` prefix.
- AWS attaches a service-specific permissions policy, often named in the `AWSServiceRolePolicy` style.
- You generally do not customize the trust relationship like you would for a customer-managed service role.

Service-linked roles are common in services such as:
- Auto Scaling
- Elastic Load Balancing features
- Amazon RDS
- Amazon ECS
- Amazon EKS
- AWS Lambda in some service integrations
- Other AWS services that need tightly scoped, service-owned access

A service-linked role is not the same as a generic service role:
- A **service role** is customer-managed and reusable where appropriate.
- A **service-linked role** is AWS-managed, service-scoped, and created for a specific integration.

**How to identify a service-linked role**

In IAM, service-linked roles are typically recognizable by:
- The `AWSServiceRoleFor*` naming pattern.
- A description indicating the linked AWS service.
- A service-linked role path rather than a normal customer role path.
- AWS-managed trust policy and service-managed permissions attachment.

Example of a service-linked role name you might see:
- `AWSServiceRoleForAutoScaling`

Do not assume every service has the same role name pattern beyond the `AWSServiceRoleFor*` convention. The exact name depends on the service and feature.

**Creation and lifecycle behavior**

Service-linked roles are usually created automatically the first time a service needs them. In some cases, you can create them explicitly with the dedicated IAM service-linked role workflow or the CLI.

CLI example:

```bash
aws iam create-service-linked-role \
  --aws-service-name autoscaling.amazonaws.com
```

Important behavior:
- The service must support service-linked roles.
- The caller must have permission to create the role.
- Some accounts or organizational controls can block creation.
- If the service cannot create the role, the service operation may fail with an access or configuration error.

Common failure causes:
- Missing IAM permission for the user or automation role.
- Service control policies (<abbr title="Service Control Policies">SCPs</abbr>) blocking IAM actions.
- Region-specific service limitations.
- The service has not yet been enabled or configured correctly.

**Permissions boundaries and service-linked roles**

Permissions boundaries are not a normal customization mechanism for service-linked roles. In exam terms, treat service-linked roles as AWS-managed identities whose permissions are controlled by AWS for the linked service, not by customer-applied boundaries.

For customer-managed roles, permissions boundaries can limit maximum effective permissions. For service-linked roles, the key idea is that AWS owns the role’s service relationship and the attached service policy.

**Customer-managed service role example**

A realistic customer-managed service role has both a trust policy and a permissions policy that you define.

Trust policy for an EC2 instance role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Permissions policy allowing access to a specific S3 bucket and CloudWatch Logs:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::app-data-bucket/*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/ec2/app:*"
    }
  ]
}
```

This pattern is common when you want the service to access only the exact resources it needs.

**Service-linked role example behavior**

For a service-linked role, you typically do not handcraft the trust policy for the service. AWS creates the role with the correct trust relationship and attaches the service-managed permissions policy.

You may inspect the role in IAM, but the service owns the linkage. That is the main distinction the exam wants you to recognize.

**When to choose a service role vs a service-linked role**

Use a **service role** when:
- You need explicit control over permissions.
- You want to scope access to a specific application or resource.
- The service supports standard role assumption and you want to manage the role yourself.

Use a **service-linked role** when:
- AWS provides a tightly integrated, service-owned role for a feature.
- The service expects to manage the role lifecycle.
- You want the simplest supported setup for that service integration.

**Exam traps and distinctions**

- A role’s **trust policy** is not the same as its permissions policy.
- A service-linked role is not a general-purpose role you reuse across services.
- A service-linked role is not usually created through the generic “create role” flow; it uses the service-linked role creation path or automatic creation by the service.
- A service-linked role is not the same as an instance profile, though EC2 uses instance profiles to deliver roles to instances.
- A service role can be customer-managed and highly customized; a service-linked role is AWS-managed and service-scoped.

**Practical troubleshooting approach**

When a service cannot access a resource, check the following in order:
1. Confirm the service is using the intended role.
2. Verify the trust policy allows the correct service principal.
3. Verify the permissions policy allows the required API actions on the correct resources.
4. Check whether the service expects a service-linked role instead of a custom role.
5. Review IAM, SCPs, and resource-based policies that may block access.

A useful CLI inspection pattern is:

```bash
aws iam get-role --role-name MyAppEc2Role
aws iam list-attached-role-policies --role-name MyAppEc2Role
aws iam list-role-policies --role-name MyAppEc2Role
```

For service-linked roles, inspect the role metadata and attached policy rather than assuming you can fully customize it like a normal customer-managed role.

> If a service-linked role is missing, do not jump straight to policy simulation. First confirm the service can create the role and that the caller has permission to do so.

**Real-world scenarios to remember**

- An EC2 application needs S3 access: use an instance role with a trust policy for `ec2.amazonaws.com`.
- A Lambda function writes logs and reads DynamoDB: use a Lambda execution role.
- Auto Scaling needs to manage instances: AWS may use a service-linked role for the Auto Scaling service.
- ECS or EKS needs service-owned access for orchestration features: service-linked roles are common in these integrations.

**Quick comparison**

| Aspect | Service role | Service-linked role |
|---|---|---|
| Managed by | Customer | AWS |
| Trust policy | Customer-defined | AWS-managed |
| Permissions policy | Customer-defined | AWS-managed service policy |
| Scope | Can be reused where appropriate | Tied to one service |
| Creation | Standard IAM role creation | Dedicated service-linked role flow or auto-created |
| Typical use | Custom service access | AWS service integration |

**Key takeaways**

- **Trust policy** controls who can assume the role.
- **Permissions policy** controls what the role can do.
- Service roles are customer-managed and flexible.
- Service-linked roles are AWS-managed and service-specific.
- For SAA-C03, focus on when AWS services assume roles, how temporary credentials are issued, and why service-linked roles reduce operational overhead while limiting customization.
