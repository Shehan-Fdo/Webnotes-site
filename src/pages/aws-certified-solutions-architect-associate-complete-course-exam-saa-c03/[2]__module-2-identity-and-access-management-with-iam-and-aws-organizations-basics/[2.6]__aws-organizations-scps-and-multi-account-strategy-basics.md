## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**AWS Organizations overview**

AWS Organizations lets you centrally manage multiple AWS accounts under one organization. It is the foundation for multi-account governance, not just billing. You use it to group accounts, apply policies, and standardize account creation across teams and environments.

A common exam trap is assuming that consolidated billing automatically means centralized access control. It does not. Billing aggregation and governance are related but separate capabilities.

- **Consolidated billing** groups charges across member accounts.
- **Governance** uses policies such as service control policies (SCPs) to set guardrails.
- **Account structure** helps you isolate workloads, reduce blast radius, and delegate ownership.

By default, a new member account joins the organization with the organization’s default state, but SCP enforcement is only effective when you enable all features in AWS Organizations. If you stay in consolidated billing mode only, you do not get full SCP-based governance.

**Organization structure and inheritance**

AWS Organizations uses a hierarchy of root, organizational units (OUs), and accounts. The root is the top-level container for the organization, and policies attached there affect everything beneath it.

Think of the hierarchy as an inheritance tree:

- **Root**
  - Applies to all OUs and accounts in the organization.
  - SCPs attached here are inherited by every descendant OU and account.
- **OU**
  - Groups accounts by function, environment, or compliance boundary.
  - SCPs attached here apply to all child OUs and accounts below it.
- **Account**
  - Receives the intersection of all applicable SCPs from root, parent OUs, and the account itself.

Multiple SCPs combine by intersection, not by override. That means the effective permissions are the common allowed set across all attached policies. If one SCP denies an action, the action is denied even if another SCP allows it.

> **Important:** SCPs do not grant permissions. They only define the maximum permissions available to IAM identities in member accounts.

**Root account, management account, and root OU behavior**

The AWS Organizations **management account** is the account that creates and administers the organization. It is not governed by SCPs in the same way as member accounts. This is a frequent exam distinction.

The organization **root** is not the same thing as the management account root user. The root in Organizations is the top-level container for all OUs and accounts.

Key behaviors to remember:

- SCPs attached to the **root** apply to all descendant OUs and member accounts.
- SCPs do **not** restrict the management account in the same way they restrict member accounts.
- The management account should be kept for billing and organization administration, not for workloads.
- The root user of any account is highly privileged, but SCPs are not a substitute for securing root credentials.

**SCP evaluation and IAM interaction**

AWS authorization is evaluated across multiple layers. For exam purposes, remember that SCPs act as a guardrail before IAM permissions can be used.

A practical mental model:

1. The request must be allowed by the SCPs that apply to the account.
2. The request must also be allowed by IAM identity policies, permission boundaries, and any session policies.
3. Resource-based policies can grant access directly to a resource, but they still cannot bypass an SCP deny in the member account.

This means SCPs set the outer boundary. IAM then decides what an identity can do inside that boundary.

- **IAM identity policies** grant permissions to users, groups, and roles.
- **Permission boundaries** limit the maximum permissions an IAM principal can receive.
- **Resource-based policies** grant access to a resource such as an S3 bucket or KMS key.
- **SCPs** limit what is possible in the account, regardless of IAM grants.

A useful exam nuance is that SCPs do not affect service-linked roles in the same way they affect normal IAM principals. Some AWS-managed operations also continue to function even when you try to restrict them with SCPs, depending on the service and action. In other words, SCPs are powerful, but they are not a universal kill switch.

**Common SCP design patterns**

SCPs are often used in two main styles: deny-list and allow-list.

- **Deny-list SCPs**
  - Start with broad access and explicitly deny risky actions.
  - Useful for guardrails such as blocking disabling CloudTrail or preventing region usage.
  - Easier to adopt incrementally.
- **Allow-list SCPs**
  - Explicitly allow only approved services or actions.
  - Common in tightly controlled OUs such as security or regulated workloads.
  - Riskier if you remove `FullAWSAccess` too early, because you can accidentally block legitimate access.

The default AWS-managed `FullAWSAccess` SCP is commonly attached at the root so that accounts start with broad permissions boundaries. If you remove it without replacing it carefully, you can unintentionally lock down accounts.

> **Warning:** An allow-list SCP strategy requires careful testing. If you detach `FullAWSAccess` from a parent OU or root and do not explicitly allow required actions, workloads may fail in surprising ways.

**Practical SCP examples**

A deny-only SCP can block dangerous actions while preserving general access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyLeavingOrganization",
      "Effect": "Deny",
      "Action": "organizations:LeaveOrganization",
      "Resource": "*"
    },
    {
      "Sid": "DenyDisablingCloudTrail",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:DeleteTrail",
        "cloudtrail:StopLogging"
      ],
      "Resource": "*"
    }
  ]
}
```

A region restriction SCP can help enforce data residency or operational boundaries:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "NotAction": [
        "iam:*",
        "organizations:*",
        "route53:*",
        "cloudfront:*",
        "support:*"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

An allow-list SCP for a tightly controlled OU might look like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOnlyApprovedServices",
      "Effect": "Allow",
      "Action": [
        "ec2:*",
        "s3:*",
        "cloudwatch:*",
        "logs:*",
        "kms:*"
      ],
      "Resource": "*"
    }
  ]
}
```

This style is powerful but easy to break if a workload needs an unlisted service.

**Troubleshooting SCP vs IAM denies**

When an action fails, you need to determine whether the denial comes from SCPs, IAM, or another layer.

A practical troubleshooting flow:

1. Check the error message and the AWS service event details.
2. Review the effective SCPs attached to the account’s root and parent OUs.
3. Confirm the IAM identity policy allows the action.
4. Check permission boundaries and session policies if the principal is a role.
5. Inspect resource-based policies for explicit denies or missing allows.
6. Use AWS CloudTrail to identify the denied API call and the principal involved.

Helpful CLI commands:

```bash
aws organizations list-policies --filter SERVICE_CONTROL_POLICY
aws organizations list-policies-for-target --target-id ou-abcd-12345678 --filter SERVICE_CONTROL_POLICY
aws organizations describe-effective-policy --policy-type SERVICE_CONTROL_POLICY --target-id 111122223333
aws iam simulate-principal-policy --policy-source-arn arn:aws:iam::111122223333:role/AppRole --action-names s3:PutObject
```

If IAM simulation says “allowed” but the action still fails, suspect SCPs or a resource policy. If the action is denied before IAM evaluation can succeed, SCPs are often the cause.

**Multi-account strategy for architecture and security**

A multi-account strategy reduces blast radius, improves separation of duties, and makes governance easier. For SAA-C03, you should know why organizations split workloads across accounts instead of putting everything in one account.

A practical least-privilege layout often includes:

- **Management account**
  - Used only for Organizations administration and billing.
- **Security account**
  - Centralizes security tooling, detective controls, and audit access.
- **Log archive account**
  - Stores immutable or tightly controlled logs such as CloudTrail and AWS Config.
- **Shared services account**
  - Hosts common services like directory integration, CI/CD tooling, or shared networking components.
- **Workload accounts**
  - Separate `dev`, `test`, and `prod` accounts for each application or business unit.

This layout helps you isolate production from non-production and keep security controls centralized.

A simple example topology:

```text
Management Account
├── Security OU
│   ├── Security Account
│   └── Log Archive Account
├── Infrastructure OU
│   ├── Shared Services Account
│   └── Network Account
└── Workloads OU
    ├── Dev Account
    ├── Test Account
    └── Prod Account
```

Why this matters:

- **Dev/test/prod separation** reduces accidental impact from experimentation.
- **Security and logging accounts** centralize sensitive controls and evidence.
- **Shared services** avoid duplicating common infrastructure in every account.
- **Prod isolation** limits the blast radius of mistakes and supports stricter SCPs.

**Cross-account access patterns**

Multi-account designs often require controlled access between accounts. The two most common patterns are IAM role assumption and resource-based policies.

- **IAM role assumption**
  - A principal in Account A assumes a role in Account B using AWS Security Token Service (STS).
  - Best for human access, automation, and temporary credentials.
  - Example: a deployment pipeline in a tooling account assumes a role in a production account.
- **Resource-based policy**
  - The resource itself grants access to a principal in another account.
  - Common for S3 buckets, KMS keys, SQS queues, SNS topics, and Lambda permissions.
  - Example: an S3 bucket in a logging account allows CloudTrail from member accounts to write logs.

Example cross-account role trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::222233334444:role/DeploymentRole"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example S3 bucket policy for cross-account access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowLogDeliveryFromOrg",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::central-log-bucket/AWSLogs/*",
      "Condition": {
        "StringEquals": {
          "aws:SourceArn": "arn:aws:cloudtrail:us-east-1:111122223333:trail/OrgTrail"
        }
      }
    }
  ]
}
```

These patterns are often combined. For example, a CI/CD role may assume a deployment role in the target account, while the target account’s S3 bucket or KMS key uses a resource policy to permit the operation.

**Control Tower relationship to Organizations**

AWS Control Tower builds on AWS Organizations. It automates the setup of a landing zone, account vending, and guardrails. It is useful when you want a faster, opinionated multi-account baseline.

Important distinctions:

- **Control Tower is optional**
  - You can use AWS Organizations without Control Tower.
- **Control Tower depends on Organizations**
  - It uses OUs, accounts, and policies underneath.
- **Landing zone automation**
  - It creates a standardized foundation for governance.
- **Guardrails**
  - It applies preventive and detective controls, often backed by SCPs and AWS Config rules.
- **Account factory**
  - It helps provision new accounts in a standardized way.

For exam questions, remember that Control Tower simplifies setup, but Organizations and SCPs are the underlying governance primitives.

**Default account and governance behavior**

New member accounts typically join with the organization’s default policy posture, but effective governance depends on how you structure the root and OUs.

- If `FullAWSAccess` is attached at the root, new accounts inherit broad access unless further restricted.
- If you attach restrictive SCPs at a parent OU, every child account inherits those restrictions.
- If you move an account between OUs, its effective SCP set changes immediately based on the new path.

This makes OU placement a governance decision, not just an organizational one.

**Practical exam scenario**

Suppose a company has one AWS organization and wants to protect production while allowing developers freedom in non-production.

A strong design is:

- Put `dev` and `test` accounts in a non-prod OU.
- Put `prod` accounts in a prod OU with stricter SCPs.
- Put logging and security accounts in dedicated OUs with tightly controlled access.
- Use IAM roles for cross-account deployment.
- Use resource-based policies for shared buckets, queues, and KMS keys.
- Keep the management account out of application workloads.

You might apply an SCP to the prod OU that denies:
- `ec2:TerminateInstances` except through approved automation
- `iam:CreateUser`
- `organizations:LeaveOrganization`
- `cloudtrail:StopLogging`

This gives you guardrails without needing to micromanage every IAM policy in every account.

**How to think about billing vs governance**

Consolidated billing helps you pay for multiple accounts from one payer account. Governance helps you control what those accounts can do.

- Billing answers: “Who pays?”
- Governance answers: “What is allowed?”
- Access control answers: “Who can do it?”

These are related but not interchangeable. A company can have consolidated billing without strong SCP governance, and it can have strong governance only after enabling all features in Organizations.

**Exam tips**

- **Root** SCPs affect all descendant OUs and accounts.
- **Multiple SCPs** are intersected; they do not override each other.
- **Management account** is not governed like member accounts.
- **SCPs** do not grant permissions; they only limit them.
- **FullAWSAccess** is a common baseline, and removing it can break access.
- **Control Tower** automates governance but is not required for Organizations.
- **Cross-account access** usually uses STS roles or resource-based policies.
- **Multi-account design** improves isolation, compliance, and blast-radius control.

**Quick reference**

| Concept | What it does | Exam takeaway |

**Minimal CLI workflow**

A typical admin workflow might include creating OUs, attaching SCPs, and verifying effective policies:

```bash
aws organizations create-organizational-unit \
  --parent-id r-exampleroot \
  --name Prod

aws organizations attach-policy \
  --policy-id p-abc123example \
  --target-id ou-abcd-12345678

aws organizations list-children \
  --parent-id ou-abcd-12345678 \
  --child-type ACCOUNT
```

Use these commands to confirm where an account sits in the hierarchy before you troubleshoot permissions.

**Key takeaways**

- Use AWS Organizations to centralize account governance.
- Use SCPs to define guardrails, not to grant access.
- Design multi-account environments around isolation, logging, security, and shared services.
- Use IAM roles and resource-based policies for cross-account access.
- Treat Control Tower as an accelerator, not a requirement.
- Always verify whether a denial comes from SCPs, IAM, permission boundaries, or a resource policy.
