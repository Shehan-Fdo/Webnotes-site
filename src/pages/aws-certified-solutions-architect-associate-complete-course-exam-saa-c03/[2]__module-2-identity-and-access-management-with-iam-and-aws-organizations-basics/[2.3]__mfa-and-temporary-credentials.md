## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**MFA and temporary credentials overview**

Multi-Factor Authentication (MFA) adds a second verification factor to sign-in, usually a time-based one-time password (TOTP) from a virtual device or a hardware token. In AWS, MFA is most important for protecting privileged access, especially the root user and administrative IAM users. Temporary credentials reduce long-lived secret exposure by issuing short-lived access keys that expire automatically.

AWS uses temporary credentials in several places:

- **AWS Security Token Service (STS)** issues short-lived credentials for users, roles, federated identities, and services.
- **IAM roles** provide permissions without storing long-term access keys on the entity that assumes the role.
- **MFA-protected sessions** require a valid MFA code before AWS returns session credentials.

The core exam idea is simple: use long-term credentials only when necessary, and prefer temporary credentials for human access, automation, and cross-account access.

**Why temporary credentials matter**

Long-lived access keys are risky because they can be copied, leaked, or forgotten. Temporary credentials reduce blast radius because they expire after a defined duration. They are also easier to rotate operationally because the session ends automatically.

Use temporary credentials when you need:

- **Short-lived access** for CLI, SDK, or console sessions.
- **Cross-account access** through role assumption.
- **Federated access** from an identity provider such as SAML 2.0 or OIDC.
- **Service-to-service access** where AWS services assume roles on your behalf.

MFA strengthens authentication, but it does not replace authorization. MFA proves the user has a second factor; IAM policies still determine what the session can do.

**MFA fundamentals**

MFA requires two or more factors from different categories:

- **Something you know**: password or passphrase.
- **Something you have**: authenticator app, hardware token, or security key.
- **Something you are**: biometrics, depending on the identity system.

In AWS IAM, MFA commonly uses:

- **Virtual MFA devices** such as authenticator apps.
- **Hardware MFA devices** for stronger physical possession.
- **FIDO2/WebAuthn security keys** in supported identity workflows, though exam questions often focus on IAM virtual/hardware MFA.

MFA is especially important for:

- Root user sign-in.
- IAM users with console access.
- Sensitive operations such as changing billing, deleting resources, or modifying IAM policies.

> The root user should be locked down with MFA and used only for tasks that require root-level permissions.

**IAM users, root user, and roles**

Understanding the identity type helps you choose the right credential model.

- **Root user**
  - Has full account access.
  - Should not have access keys.
  - Should have MFA enabled immediately.
- **IAM user**
  - Represents a person or application within one account.
  - Can have long-term access keys, but best practice is to avoid them for humans.
  - Can use MFA for console sign-in and for API operations that require MFA.
- **IAM role**
  - Has no permanent credentials.
  - Is assumed by a trusted principal and returns temporary credentials.
  - Is the preferred mechanism for workloads, cross-account access, and delegated administration.

A common exam pattern is choosing a role over an IAM user when you need temporary access without distributing static secrets.

**AWS STS and temporary credentials**

AWS Security Token Service (STS) is the service that issues temporary credentials. These credentials include:

- `AccessKeyId`
- `SecretAccessKey`
- `SessionToken`
- Expiration time

The `SessionToken` is required when using temporary credentials. Without it, the credentials are incomplete.

Common STS operations include:

- `AssumeRole`
- `AssumeRoleWithMFA`
- `GetSessionToken`
- `GetFederationToken`
- `AssumeRoleWithSAML`
- `AssumeRoleWithWebIdentity`

Typical use cases:

- **`GetSessionToken`**: create temporary credentials for an IAM user, often with MFA.
- **`AssumeRole`**: switch into a role in the same or another account.
- **`AssumeRoleWithMFA`**: require MFA before role assumption.
- **`AssumeRoleWithSAML`**: federated enterprise sign-in.
- **`AssumeRoleWithWebIdentity`**: identity federation from OIDC providers such as Kubernetes or mobile/web identity flows.

**MFA with STS**

MFA can be enforced in IAM policies by requiring the `aws:MultiFactorAuthPresent` condition key. This is a common way to ensure that sensitive actions only succeed when the session was authenticated with MFA.

Example policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSensitiveActionsOnlyWithMFA",
      "Effect": "Allow",
      "Action": [
        "iam:DeleteUser",
        "iam:CreateAccessKey",
        "iam:UpdateLoginProfile"
      ],
      "Resource": "*",
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

This pattern is useful for:

- Protecting IAM changes.
- Restricting administrative actions.
- Requiring step-up authentication for high-risk operations.

You can also use MFA in trust policies to control role assumption. For example, a role trust policy can require MFA before allowing `sts:AssumeRole`.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:user/Alice"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

**CLI workflow for MFA-protected sessions**

A common CLI pattern is:

1. Use your long-term IAM user credentials only to request a temporary session.
2. Provide the MFA code.
3. Export the returned session credentials.
4. Use the temporary session for AWS API calls.

Example with `get-session-token`:

```bash
aws sts get-session-token \
  --serial-number arn:aws:iam::111122223333:mfa/alice \
  --token-code 123456 \
  --duration-seconds 3600
```

The response includes temporary credentials:

```json
{
  "Credentials": {
    "AccessKeyId": "ASIA....",
    "SecretAccessKey": "....",
    "SessionToken": "....",
    "Expiration": "2026-04-26T18:00:00Z"
  }
}
```

Export them into your shell:

```bash
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
```

Then verify identity:

```bash
aws sts get-caller-identity
```

Practical notes:

- The session expires automatically.
- The maximum duration depends on the API and role settings.
- `SessionToken` is mandatory for temporary credentials.
- If you omit the token, requests fail with authentication errors.

**Role assumption and cross-account access**

Roles are the preferred way to grant access across accounts. Instead of creating IAM users in every account, you create a role in the target account and allow a trusted principal to assume it.

Typical cross-account flow:

1. A user or workload in Account A authenticates.
2. The principal calls `sts:AssumeRole` for a role in Account B.
3. STS returns temporary credentials.
4. The session uses the permissions attached to the role in Account B.

Example assume-role command:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/ReadOnlyAuditRole \
  --role-session-name audit-session
```

If the role requires MFA, include the MFA parameters:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/ReadOnlyAuditRole \
  --role-session-name audit-session \
  --serial-number arn:aws:iam::111122223333:mfa/alice \
  --token-code 123456
```

Use cases for roles:

- **Cross-account administration**
- **Temporary elevated access**
- **EC2 instance permissions**
- **Lambda execution permissions**
- **ECS task permissions**
- **Federated workforce access**

**Temporary credentials in AWS SDKs and applications**

Applications should avoid embedding static access keys. Instead, they should obtain credentials from the environment, instance metadata, task roles, or an identity provider.

Common credential sources:

- **EC2 instance profile** via the Instance Metadata Service (IMDS)
- **ECS task role**
- **Lambda execution role**
- **AWS CLI profile with role assumption**
- **Federated identity provider**

Example AWS CLI profile configuration:

```ini
[profile base]
region = us-east-1
output = json

[profile audit]
role_arn = arn:aws:iam::444455556666:role/ReadOnlyAuditRole
source_profile = base
mfa_serial = arn:aws:iam::111122223333:mfa/alice
```

Then use:

```bash
aws s3 ls --profile audit
```

This pattern is useful because the CLI can automatically assume the role and request MFA when needed.

**Console sign-in with MFA**

For console access, MFA adds a second step after password entry. The user enters:

- Username
- Password
- MFA code

This is especially important for privileged IAM users and the root user. If you use AWS IAM Identity Center, MFA is often enforced by the identity source rather than directly by IAM user policies, but the security principle is the same: require a second factor for sensitive access.

**Best practices**

- Enable MFA on the root user immediately.
- Avoid using root for daily operations.
- Prefer IAM roles over IAM users for workloads and cross-account access.
- Avoid long-term access keys for human users.
- Use temporary credentials for CLI and SDK access whenever possible.
- Require MFA for sensitive IAM actions.
- Rotate and remove unused access keys.
- Use least privilege on roles and users.
- Set short session durations where practical.
- Monitor STS and IAM activity with CloudTrail.

**Common exam scenarios**

- If the question asks for the most secure way to let a user access AWS temporarily, choose **STS temporary credentials** or **role assumption**.
- If the question asks how to protect privileged console access, choose **MFA**.
- If the question asks how to avoid distributing credentials to an application, choose an **IAM role** attached to the compute service.
- If the question asks for cross-account access without creating IAM users in the target account, choose **assume role**.
- If the question asks for step-up authentication before deleting IAM users or changing policies, use an IAM policy condition with `aws:MultiFactorAuthPresent`.

**Troubleshooting and operational considerations**

Temporary credentials fail for a few common reasons:

- The session expired.
- The MFA code was incorrect or already used.
- The role trust policy does not allow the principal.
- The IAM policy lacks permission for `sts:AssumeRole` or `sts:GetSessionToken`.
- The session token was not exported or included in the request.
- The requested duration exceeds the role or API limit.

Useful checks:

```bash
aws sts get-caller-identity
aws configure list
aws iam list-mfa-devices --user-name alice
```

If you are debugging role assumption, inspect:

- Trust policy on the role
- Permission policy on the caller
- MFA condition keys
- Session duration settings
- CloudTrail events for `AssumeRole` and `GetSessionToken`

**Practical architecture patterns**

A secure enterprise pattern often looks like this:

- Users authenticate through a central identity provider.
- The identity provider federates into AWS.
- AWS issues temporary role credentials.
- MFA is enforced at the identity provider or during role assumption.
- Workloads use instance roles or task roles instead of embedded keys.

Example topology:

- Developer signs in to corporate IdP.
- IdP asserts identity to AWS using SAML.
- AWS STS returns a role session.
- Developer uses the session to access only approved resources.

This design reduces credential sprawl and improves auditability.

**Key takeaways**

- **MFA** strengthens authentication but does not grant permissions by itself.
- **Temporary credentials** are short-lived and safer than static access keys.
- **STS** is the AWS service that issues temporary credentials.
- **Roles** are the preferred way to grant temporary access.
- **`SessionToken`** is required when using temporary credentials.
- **`aws:MultiFactorAuthPresent`** helps enforce MFA for sensitive actions.
- For the exam, prefer roles, STS, and MFA over long-lived credentials whenever possible.
