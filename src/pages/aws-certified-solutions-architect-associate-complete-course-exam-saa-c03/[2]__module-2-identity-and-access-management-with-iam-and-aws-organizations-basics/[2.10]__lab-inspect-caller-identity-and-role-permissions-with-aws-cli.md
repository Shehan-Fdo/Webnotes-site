## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and mental model**

This lab helps you answer three questions quickly:

- Who am I in AWS right now?
- What permissions does this role or user likely have?
- Why did a request succeed or fail?

The AWS CLI is useful because it exposes the same authorization layers you see in real systems: identity-based policies, resource-based policies, permission boundaries, service control policies (SCPs), and temporary credentials from AWS Security Token Service (STS). In practice, you use the CLI to confirm the active principal, inspect attached policies, and test whether an action is allowed before you automate or troubleshoot.

**Core identity checks with STS**

The most direct way to inspect the current caller is `sts get-caller-identity`. It returns the account, the ARN, and the unique user ID for the credentials currently in use.

```bash
aws sts get-caller-identity
```

Typical output:

```json
{
  "UserId": "AROAEXAMPLEID:session-name",
  "Account": "123456789012",
  "Arn": "arn:aws:sts::123456789012:assumed-role/MyRole/session-name"
}
```

Use this command when you need to confirm whether the CLI is using:

- An IAM user access key
- An assumed role session
- Federated credentials
- A different profile than expected

A few important details matter for troubleshooting:

- **Caller identity** is about the credentials currently active in the CLI, not the profile name you intended to use.
- `get-caller-identity` is often allowed even when many other permissions are missing, but you should not assume it is universally available in every environment.
- The ARN format tells you the principal type:
  - `arn:aws:iam::...:user/...` indicates an IAM user
  - `arn:aws:sts::...:assumed-role/.../...` indicates an assumed role session

If you want to see which profile the CLI is using, inspect your configuration files rather than relying on a nonexistent profile-listing command:

```bash
cat ~/.aws/config
cat ~/.aws/credentials
```

You can also check the active profile with an environment variable:

```bash
echo $AWS_PROFILE
echo $AWS_DEFAULT_PROFILE
```

**CLI configuration and profile selection**

AWS CLI behavior depends on the selected profile, region, and credential source. A profile can point to static keys, an assumed role, SSO, or another credential provider.

Common configuration files:

- `~/.aws/config` for regions, output format, and role/profile settings
- `~/.aws/credentials` for access keys in older or simpler setups

Example configuration:

```ini
[default]
region = us-east-1
output = json

[profile dev]
region = us-west-2
role_arn = arn:aws:iam::123456789012:role/DeveloperRole
source_profile = default
```

Useful verification commands:

```bash
aws configure list
aws configure get region --profile dev
aws configure get output --profile dev
```

`aws configure list` shows where the CLI is sourcing credentials and settings from. It is especially useful when environment variables override a profile file.

To run a command with a specific profile:

```bash
aws sts get-caller-identity --profile dev
```

If the result is unexpected, check for these common causes:

- Environment variables override the profile:
  - `AWS_ACCESS_KEY_ID`
  - `AWS_SECRET_ACCESS_KEY`
  - `AWS_SESSION_TOKEN`
  - `AWS_PROFILE`
- The profile assumes a role, so the identity becomes an STS assumed-role ARN
- The region is wrong, which can affect service calls even if identity is correct

**Inspecting IAM role details**

If you know the role name and have permission to read IAM metadata, you can inspect the role itself.

```bash
aws iam get-role --role-name MyRole
```

This returns the role ARN, trust policy, creation date, and path. The trust policy is especially important because it controls who can assume the role.

Example trust policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:user/Alice"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Use `get-role` when you need to confirm:

- The role exists in the same account you are querying
- The trust relationship allows the expected principal
- The role ARN matches the one used by your application or profile

Related read-only commands:

```bash
aws iam list-attached-role-policies --role-name MyRole
aws iam list-role-policies --role-name MyRole
aws iam get-role-policy --role-name MyRole --policy-name InlinePolicyName
```

These commands require IAM permissions to read role metadata and only work for roles in the account you are querying. They are useful for troubleshooting, but they do not tell you the full effective permissions by themselves.

What each command shows:

- `list-attached-role-policies` lists managed policies attached to the role
- `list-role-policies` lists inline policy names on the role
- `get-role-policy` retrieves the JSON document for a specific inline policy

Example attached policy output pattern:

```json
{
  "AttachedPolicies": [
    {
      "PolicyName": "AmazonS3ReadOnlyAccess",
      "PolicyArn": "arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess"
    }
  ]
}
```

**Understanding effective permissions**

A role’s effective permissions come from more than one place. You should think in layers:

- **Identity-based policies** attached to the user or role
- **Resource-based policies** on the target resource
- **Permission boundaries** that cap what the identity can do
- **SCPs** in AWS Organizations that restrict accounts or OUs
- **Session policies** passed during role assumption
- **Service-specific conditions** such as VPC endpoints, MFA, or source IP

A role can have an attached policy that allows an action, yet the action still fails because another layer denies it. This is why “the policy looks correct” is not enough.

A practical example:

- The role policy allows `s3:GetObject`
- The bucket policy denies access unless the request comes from a specific VPC endpoint
- The request fails even though the identity policy allows it

**Testing authorization with dry-run and simulation**

Some AWS APIs support `--dry-run`. This is useful because it checks whether the request is syntactically valid and whether the caller is authorized, without making the actual change.

Example with Amazon EC2:

```bash
aws ec2 terminate-instances --instance-ids i-0123456789abcdef0 --dry-run
```

Interpretation:

- `DryRunOperation` usually means the caller is authorized and the request format is valid
- `UnauthorizedOperation` usually means the caller is not allowed
- Other errors can still occur if the request is malformed, the resource ID is invalid, or the service has additional constraints

Use dry-run when you want a low-risk authorization check before automation or destructive actions.

For broader policy analysis, AWS IAM policy simulation can help:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/MyRole \
  --action-names s3:GetObject ec2:DescribeInstances
```

This is helpful for checking whether a principal is allowed to perform specific actions, but it has limits:

- It does not perfectly reproduce every real-world authorization path
- It may not fully account for SCPs unless you provide the relevant context
- It does not model every resource-based policy scenario in the same way the live service does
- It is best used as a diagnostic tool, not as the final authority

If you need to test a resource-scoped decision, include resource ARNs and context keys where appropriate:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/MyRole \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::my-bucket/example.txt
```

**Assuming a role and verifying the session**

`sts assume-role` returns temporary credentials for a role session. You can export those credentials into your shell or store them in a profile, but the call succeeds only if:

- The source principal is allowed by the role trust policy
- The caller has permission to call `sts:AssumeRole` when required
- The requested session duration is within the role’s maximum session duration and any applicable limits

Example:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/MyRole \
  --role-session-name lab-session
```

Typical response:

```json
{
  "Credentials": {
    "AccessKeyId": "ASIA...",
    "SecretAccessKey": "...",
    "SessionToken": "...",
    "Expiration": "2026-04-26T18:30:00Z"
  },
  "AssumedRoleUser": {
    "AssumedRoleId": "AROAEXAMPLEID:lab-session",
    "Arn": "arn:aws:sts::123456789012:assumed-role/MyRole/lab-session"
  }
}
```

To use the temporary credentials in your shell:

```bash
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
aws sts get-caller-identity
```

A safer pattern is to create a dedicated profile for the role:

```ini
[profile lab-role]
role_arn = arn:aws:iam::123456789012:role/MyRole
source_profile = default
region = us-east-1
```

Then test it:

```bash
aws sts get-caller-identity --profile lab-role
```

**Troubleshooting common failures**

When a command fails, identify whether the problem is identity, permission, trust, or region.

- **AccessDenied** usually means the caller lacks permission or a higher-level policy denies the action
- **InvalidClientTokenId** often means the credentials are wrong, expired, or not being loaded
- **ExpiredToken** means temporary credentials have expired
- **AccessDenied when assuming a role** often points to the trust policy, not just the identity policy
- **SignatureDoesNotMatch** can indicate clock skew, wrong secret key, or mismatched region/service signing

A practical troubleshooting sequence:

1. Confirm the active identity with `aws sts get-caller-identity`.
2. Confirm the selected profile and region with `aws configure list`.
3. Inspect the role trust policy with `aws iam get-role --role-name ...` if you have permission.
4. Check attached and inline policies for the role.
5. Test the exact action with `--dry-run` or a minimal read-only API call.
6. Use policy simulation with the correct resource ARN and context where possible.

> If the role is in another account, you may not be able to inspect it with IAM read commands unless you have cross-account permissions. In that case, rely on the trust policy, the role ARN, and the behavior of the assumed session.

**Practical lab workflow**

A simple workflow for this lab is:

1. Set or select the intended profile.
2. Verify the current caller identity.
3. Confirm whether the identity is a user or an assumed role.
4. Inspect the role’s trust policy and attached policies if permitted.
5. Test a harmless API call or `--dry-run`.
6. Compare the observed result with the expected authorization path.

Example session:

```bash
aws configure list
aws sts get-caller-identity
aws iam get-role --role-name MyRole
aws iam list-attached-role-policies --role-name MyRole
aws ec2 describe-instances --max-items 5
```

If you are using a role-based profile, the identity should change from an IAM user ARN to an STS assumed-role ARN. That change confirms the role assumption path is working.

**Exam-relevant takeaways**

- **STS** is the fastest way to confirm who the CLI is acting as.
- **IAM role trust policies** control who can assume a role.
- **Attached and inline policies** describe what the role can do, but they are not the only authorization layer.
- **Dry-run** is useful for checking authorization without making changes.
- **Policy simulation** helps diagnose permissions, but it does not perfectly mirror every live authorization path.
- **Temporary credentials** expire, so session duration matters in labs and automation.
