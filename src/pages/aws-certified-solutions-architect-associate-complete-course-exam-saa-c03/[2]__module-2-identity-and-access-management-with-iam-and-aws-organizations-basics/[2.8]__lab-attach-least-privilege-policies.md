## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Least-privilege policy design**

Least privilege means you grant only the permissions required to complete a task, and nothing more. In AWS Identity and Access Management (IAM), that usually means starting with a broad policy, observing the exact API calls a workload makes, and then narrowing actions, resources, and conditions until the policy matches the real use case.

This lab topic is about attaching those scoped permissions to the right identity, then validating that the identity can do the intended work and nothing else. The practical goal is not just “make it work,” but “make it work with the smallest safe permission set.”

- **IAM identity-based policies** attach to users, groups, or roles and define what the identity can do.
- **Resource-based policies** attach to the resource itself, such as an S3 bucket policy or KMS key policy.
- **Explicit deny** always overrides allow, so one deny statement can block access even when another policy allows it.
- **Permissions boundaries** and **service control policies (SCPs)** can further restrict effective permissions beyond the attached identity policy.

**How AWS evaluates access**

AWS authorization is the result of multiple policy layers, not just one attached policy. When you troubleshoot access, you should think in terms of the full evaluation path:

- Identity-based policies on the user, group, or role
- Resource-based policies on the target resource
- Permissions boundaries on the principal
- Session policies for temporary credentials
- SCPs in AWS Organizations
- Service-specific rules, such as bucket ownership or KMS key policy requirements

A common mistake is to test only the identity policy and assume the result reflects real access. In practice, a role may still be blocked by a permissions boundary or an SCP, or it may need a resource policy to allow cross-account access.

> **Important:** `aws iam simulate-principal-policy` is useful for checking identity-based policy decisions, but it does not fully reproduce every effective authorization layer in AWS. Use it as one tool, not as the final authority.

**Typical lab workflow**

A least-privilege lab usually follows a repeatable pattern:

1. Identify the task the principal must perform.
2. List the AWS API actions required for that task.
3. Scope each action to the narrowest valid resource ARN.
4. Add conditions where they improve safety, such as source IP, MFA, or tags.
5. Attach the policy to the user, group, or role.
6. Test the workflow and inspect denied actions.
7. Refine the policy until only required access remains.

This workflow is especially useful when you are building policies for application roles, automation users, or delegated admin access.

**Policy structure**

An IAM policy is JSON with one or more statements. Each statement typically includes `Effect`, `Action`, `Resource`, and optionally `Condition`.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadOnlyAccessToReportsBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-reports",
        "arn:aws:s3:::example-reports/*"
      ]
    }
  ]
}
```

Key points:

- `Action` names the API operations you allow.
- `Resource` should be as specific as the service permits.
- Some actions support resource-level permissions; others require `*`.
- `Sid` is optional but helps you document intent.
- `Version` is usually `2012-10-17` for modern IAM policies.

**S3 least-privilege patterns**

Amazon S3 is a common lab target because it clearly shows the difference between bucket-level and object-level permissions.

For read-only access to a bucket, you often need both:

- `s3:ListBucket` on the bucket ARN
- `s3:GetObject` on the object ARN pattern

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListReportsBucket",
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::example-reports",
      "Condition": {
        "StringLike": {
          "s3:prefix": [
            "reports/"
          ]
        }
      }
    },
    {
      "Sid": "ReadReportsObjects",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-reports/reports/*"
    }
  ]
}
```

Important S3 details:

- `s3:prefix` filters the results of `ListBucket`; it is not an object ARN wildcard.
- Use a prefix like `reports/`, not `reports/*`, when you want to constrain listing.
- `s3:GetObject` applies to object ARNs, so `arn:aws:s3:::bucket-name/path/*` is appropriate there.
- If you need uploads, add `s3:PutObject` on the object ARN.
- If versioning matters, you may also need `s3:GetObjectVersion` or `s3:DeleteObjectVersion`.

A practical CLI test might look like this:

```bash
aws s3 ls s3://example-reports/reports/
aws s3 cp ./local-file.txt s3://example-reports/reports/local-file.txt
aws s3 cp s3://example-reports/reports/local-file.txt .
```

If listing works but object access fails, the policy may be missing object-level permissions. If object access works but listing fails, the bucket-level `ListBucket` permission is likely missing or too narrowly conditioned.

**CloudWatch Logs permissions**

CloudWatch Logs often requires a mix of resource-scoped and wildcard permissions. A common example is creating a log group and then writing log streams and events.

For `logs:CreateLogGroup`, you generally must use `Resource: "*"`, because the log group does not exist yet and cannot be referenced by ARN at creation time.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "CreateLogGroup",
      "Effect": "Allow",
      "Action": "logs:CreateLogGroup",
      "Resource": "*"
    },
    {
      "Sid": "WriteLogEvents",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/lambda/my-function:*"
    }
  ]
}
```

Why this matters:

- Some actions support resource-level permissions only after the resource exists.
- `logs:CreateLogStream` and `logs:PutLogEvents` can usually be scoped to a specific log group or stream pattern.
- If you are granting an application permission to write logs, avoid broad `logs:*` access unless the use case truly requires it.

A common operational pattern is to let the service create the log group once, then tighten the policy to the exact log group ARN for ongoing writes.

**Using conditions to tighten access**

Conditions are one of the most effective ways to reduce risk without breaking functionality.

Common condition keys include:

- `aws:SourceIp` to restrict access by client IP
- `aws:MultiFactorAuthPresent` to require MFA
- `aws:PrincipalTag/*` and `aws:ResourceTag/*` for tag-based access control
- `s3:prefix` for bucket listing constraints
- `kms:ViaService` for service-mediated KMS usage

Example with MFA:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSensitiveActionsWithMFA",
      "Effect": "Allow",
      "Action": [
        "iam:ChangePassword",
        "ec2:StopInstances"
      ],
      "Resource": "*",
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

Use conditions when they match the business requirement. They are especially valuable for human access, but they can also support workload isolation through tags and service-specific keys.

**Attaching policies to the right principal**

You can attach permissions to:

- A **user** for direct human access
- A **group** for shared human permissions
- A **role** for applications, EC2 instances, Lambda functions, or cross-account access

In modern AWS designs, roles are usually preferred for workloads because they avoid long-term credentials.

Common attachment commands:

```bash
aws iam attach-user-policy \
  --user-name alice \
  --policy-arn arn:aws:iam::123456789012:policy/ReadReportsPolicy

aws iam attach-group-policy \
  --group-name Analysts \
  --policy-arn arn:aws:iam::123456789012:policy/ReadReportsPolicy

aws iam attach-role-policy \
  --role-name AppRole \
  --policy-arn arn:aws:iam::123456789012:policy/WriteLogsPolicy
```

If you need a custom policy, create it first:

```bash
aws iam create-policy \
  --policy-name ReadReportsPolicy \
  --policy-document file://read-reports-policy.json
```

Then attach it to the identity that actually performs the task.

**Testing and troubleshooting access**

Testing is where least privilege becomes real. You should verify both success paths and denied paths.

Useful commands:

```bash
aws sts get-caller-identity
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:user/alice \
  --action-names s3:ListBucket s3:GetObject \
  --resource-arns arn:aws:s3:::example-reports arn:aws:s3:::example-reports/reports/file1.txt
```

Use simulation to understand identity policy behavior, then validate with real service calls. If the real call fails but simulation says allowed, check for:

- Resource-based policy restrictions
- Permissions boundaries
- SCPs
- Session policies
- Region-specific service behavior
- Missing dependent permissions

For example, a Lambda function that writes to CloudWatch Logs may need permissions for log group creation, stream creation, and event publishing. If one action is missing, the function may partially work and still fail during initialization.

**Practical policy refinement strategy**

A safe way to tighten permissions is to start with the minimum set of actions needed for the workflow, then reduce resource scope and add conditions.

1. Start with the exact service actions required.
2. Replace `*` resources with specific ARNs where supported.
3. Add conditions for prefix, tags, MFA, or source IP.
4. Test the real workflow.
5. Remove any action that is not actually used.
6. Re-test after every change.

This approach is especially useful when you are building policies from CloudTrail logs or from application error messages. CloudTrail shows the API calls that were attempted, which helps you identify missing permissions without guessing.

**Common exam and lab pitfalls**

- **Overbroad resources**: granting `Resource: "*"` when the service supports ARN scoping.
- **Wrong ARN format**: using a bucket ARN where an object ARN is required, or vice versa.
- **Incorrect S3 prefix usage**: treating `s3:prefix` like an object wildcard instead of a list filter.
- **Missing dependent actions**: allowing a write action but forgetting the create or describe action needed first.
- **Assuming simulation is complete**: policy simulation does not replace end-to-end testing.
- **Ignoring higher-level controls**: permissions boundaries and SCPs can still block access.
- **Using direct user credentials for workloads**: roles are usually safer and easier to rotate.

**Real-world examples**

A developer role that writes application logs might need:

- `logs:CreateLogGroup` on `*`
- `logs:CreateLogStream` on the target log group ARN
- `logs:PutLogEvents` on the target log group ARN

A data analyst role that reads a specific S3 prefix might need:

- `s3:ListBucket` on the bucket with `s3:prefix` set to `reports/`
- `s3:GetObject` on `arn:aws:s3:::bucket-name/reports/*`

A deployment role that manages EC2 instances might need a small set of EC2 actions, but often also needs `iam:PassRole` if it launches resources that assume another role. That permission should be tightly scoped to the exact role ARN, not all roles.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PassOnlyTheAppRole",
      "Effect": "Allow",
      "Action": "iam:PassRole",
      "Resource": "arn:aws:iam::123456789012:role/AppInstanceRole"
    }
  ]
}
```

**Why this lab matters for SAA-C03**

The exam expects you to recognize how AWS permissions are granted, constrained, and evaluated. Least privilege is a recurring design principle across S3, IAM, Lambda, CloudWatch Logs, EC2, and cross-account access. If you can read a policy and explain what it allows, what it blocks, and what additional control layers may apply, you are well prepared for scenario-based questions.

The strongest habit is to think in terms of the exact API call, the exact resource, and the exact condition. That mindset leads to safer architectures and better exam answers.
