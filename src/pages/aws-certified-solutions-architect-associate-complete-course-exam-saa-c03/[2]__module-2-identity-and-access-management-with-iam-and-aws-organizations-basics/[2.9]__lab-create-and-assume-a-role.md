## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Overview**

An IAM role is an AWS identity with permissions that can be assumed by trusted principals instead of being permanently attached to a single user. In this lab, you focus on creating a role, defining who can assume it, and then using AWS Security Token Service (AWS STS) to obtain temporary credentials.

This pattern is central to secure AWS architecture because it avoids long-term access keys, supports least privilege, and enables cross-account access, service access, and federated access.

**Core concepts**

- **IAM role**: An identity with permissions and a trust policy.
- **Trust policy**: A resource-based policy that defines who can assume the role.
- **Permissions policy**: An identity-based policy that defines what the role can do after assumption.
- **AWS STS**: The service that issues temporary security credentials.
- **Temporary credentials**: Short-lived access key, secret key, and session token returned when a role is assumed.

A role is not “logged into” the way a user is. Instead, you call `sts:AssumeRole` and receive temporary credentials that represent the role session.

**When to use a role**

Use roles when you need:

- **Cross-account access** between AWS accounts without sharing long-term credentials.
- **AWS service access** such as EC2 instances accessing S3 through an instance profile.
- **Federated access** for workforce identities from an external identity provider.
- **Temporary elevated access** for administrative tasks or break-glass workflows.

**Role anatomy**

An IAM role has two distinct policy layers:

- **Trust policy**
  - Controls *who* can assume the role.
  - Common principals include an IAM user, IAM role, AWS service, or federated identity provider.
- **Permissions policy**
  - Controls *what* the role can do after it is assumed.
  - Can be inline or managed policies.

This separation is important. A principal must be trusted first, and then the role session must still be authorized for the target actions.

**Example trust policy**

The following trust policy allows a specific IAM user to assume the role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSpecificUserToAssumeRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:user/Alice"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

A service trust policy uses a service principal instead:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowEC2ToAssumeRole",
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

**Creating a role in the console**

A typical workflow is:

1. Open the IAM console and choose **Roles**.
2. Select **Create role**.
3. Choose the trusted entity type:
   - AWS account
   - AWS service
   - Web identity
   - SAML 2.0 federation
   - Custom trust policy
4. Attach permissions policies that define the role’s capabilities.
5. Name the role and review the trust relationship.

The console often creates both the trust policy and the permissions policy attachment in one flow, but conceptually they remain separate.

**Creating a role with the AWS CLI**

You can create a role by supplying a trust policy document:

```bash
aws iam create-role \
  --role-name DemoAssumeRole \
  --assume-role-policy-document file://trust-policy.json
```

Then attach a permissions policy:

```bash
aws iam attach-role-policy \
  --role-name DemoAssumeRole \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

If you need a custom permissions policy, create and attach it:

```bash
aws iam put-role-policy \
  --role-name DemoAssumeRole \
  --policy-name S3ListBucketOnly \
  --policy-document file://permissions-policy.json
```

**Assuming a role with AWS STS**

The `assume-role` command returns temporary credentials and a session token:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/DemoAssumeRole \
  --role-session-name demo-session
```

Typical response fields include:

- `AccessKeyId`
- `SecretAccessKey`
- `SessionToken`
- `Expiration`

You export these values into environment variables to use the role session from the CLI:

```bash
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
```

After exporting, verify the identity:

```bash
aws sts get-caller-identity
```

This should show the assumed role session ARN rather than the original IAM user.

**Practical example: cross-account access**

A common architecture uses two accounts:

- **Account A**: contains the IAM user or role that initiates access.
- **Account B**: contains the target role and resources.

The flow is:

1. Create a role in Account B.
2. Trust the principal from Account A in the role’s trust policy.
3. Attach permissions in Account B that allow access to the required resources.
4. From Account A, call `sts:AssumeRole` against the role ARN in Account B.
5. Use the temporary credentials to access the target resources.

This is preferred over creating duplicate IAM users in each account.

**Example permissions policy**

This policy allows read-only access to a specific S3 bucket:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListBucket",
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::example-bucket"
    },
    {
      "Sid": "ReadObjects",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

Attach this policy to the role if the assumed identity should only read from that bucket.

**Role session duration**

Role sessions are temporary and expire automatically. The maximum duration depends on the role configuration and the type of assumption. Shorter sessions reduce exposure if credentials are leaked.

Use shorter sessions when:

- The task is sensitive.
- The role is highly privileged.
- You want to reduce the blast radius of temporary credentials.

**Common lab verification commands**

Use these commands to confirm the role works as expected:

```bash
aws iam get-role --role-name DemoAssumeRole
aws iam list-attached-role-policies --role-name DemoAssumeRole
aws sts get-caller-identity
aws s3 ls s3://example-bucket
```

If the role is configured correctly, `get-caller-identity` should reflect the assumed role, and resource access should match the attached permissions.

**Troubleshooting**

> If `AssumeRole` fails, check both sides of the authorization path: the caller must be allowed by the trust policy, and the role session must be allowed by the permissions policy.

Common issues include:

- **Trust policy mismatch**
  - The principal ARN is wrong.
  - The trust policy uses the wrong service principal.
- **Missing caller permission**
  - The source identity lacks `sts:AssumeRole` permission on the target role ARN.
- **Incorrect role ARN**
  - The account ID, role name, or path is wrong.
- **Session duration or MFA requirements**
  - The role may require MFA or a shorter session than requested.
- **Permission boundary or SCP restrictions**
  - AWS Organizations service control policies can block the action even when IAM allows it.

**CLI pattern for source-side permission**

The caller also needs permission to invoke `sts:AssumeRole`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::123456789012:role/DemoAssumeRole"
    }
  ]
}
```

Without this, the trust policy alone is not enough.

**Service-linked and instance roles**

Roles are also used by AWS services. For example, an EC2 instance profile attaches a role to an instance so applications can access AWS APIs without embedded credentials.

Typical EC2 role flow:

1. Create a role with `ec2.amazonaws.com` as the trusted service.
2. Attach the required permissions policy.
3. Attach the role to an instance profile.
4. Launch or modify the EC2 instance to use that profile.
5. Retrieve credentials automatically from the instance metadata service.

This is a standard exam pattern for secure application access.

**Security best practices**

- **Use least privilege** for both trust and permissions policies.
- **Prefer roles over IAM users** for workloads and cross-account access.
- **Avoid long-lived access keys** when temporary credentials are sufficient.
- **Restrict trust to specific principals** instead of using broad account-wide trust unless required.
- **Use MFA** for sensitive human assumption flows when appropriate.
- **Monitor role usage** with AWS CloudTrail to see who assumed what and when.

**Exam focus**

For SAA-C03, be ready to identify:

- When to use an IAM role instead of an IAM user.
- The difference between a trust policy and a permissions policy.
- How `sts:AssumeRole` works.
- Why temporary credentials are safer than permanent access keys.
- How cross-account access is implemented with roles.
- How AWS services assume roles through service principals.

**Quick mental model**

- Trust policy answers: *Who can assume this role?*
- Permissions policy answers: *What can this assumed role do?*
- STS answers: *Give me temporary credentials for this session.*

**Minimal end-to-end example**

1. Create a trust policy that allows `arn:aws:iam::111122223333:user/Alice`.
2. Create the role with `aws iam create-role`.
3. Attach `AmazonS3ReadOnlyAccess`.
4. Grant Alice permission to call `sts:AssumeRole` on the role ARN.
5. Run `aws sts assume-role`.
6. Export the returned credentials.
7. Confirm access with `aws s3 ls`.

This is the core workflow behind the lab.
