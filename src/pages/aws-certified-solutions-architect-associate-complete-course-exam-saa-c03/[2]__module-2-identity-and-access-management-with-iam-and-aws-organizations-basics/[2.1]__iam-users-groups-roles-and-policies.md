## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**IAM fundamentals**

AWS Identity and Access Management (<abbr title="Identity and Access Management">IAM</abbr>) controls who can authenticate to AWS and what they can do after authentication. In exam scenarios, you usually decide between long-term identities such as users and groups, and temporary identities such as roles.

The core idea is simple:

- **Authentication** answers who you are.
- **Authorization** answers what you can do.

IAM is a global service, so identities are not tied to a single Region. Permissions are evaluated whenever a request is made, and the final decision depends on multiple policy layers, not just one attached policy.

**IAM users**

An IAM user represents a long-term identity for a person or application that needs direct AWS access. Users can have console passwords, access keys, or both, but you should avoid using users for workloads when roles are possible.

Use IAM users when:

- A human needs a named identity with individualized permissions.
- You need a long-lived identity for a legacy integration that cannot assume a role.
- You want to separate administrative access from application access.

Important user characteristics:

- **Credentials** can include a password for the console and access keys for the CLI or SDKs.
- **Access keys** are long-term secrets and should be rotated regularly.
- **MFA** (multi-factor authentication) should be enabled for privileged users.
- **Least privilege** means you grant only the actions required for the task.

> The root user is the account owner identity and should be locked down aggressively: enable MFA, do not create access keys, and use it only for tasks that require root-level account actions such as changing support or billing settings.

A practical CLI example for creating a user and attaching a managed policy:

```bash
aws iam create-user --user-name app-deployer
aws iam attach-user-policy \
  --user-name app-deployer \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

**Root user versus IAM users**

The root user has unrestricted access to the AWS account. IAM users are scoped identities that you create and manage. For exam questions, the root user is the answer when the task requires account-level ownership, but it is almost never the right identity for day-to-day operations.

Key differences:

- **Root user**
  - Created automatically when the account is opened.
  - Uses the email address associated with the account.
  - Should not have access keys.
  - Should have MFA enabled immediately.
- **IAM user**
  - Created by an administrator.
  - Can be assigned granular permissions.
  - Can be disabled, deleted, or rotated without affecting account ownership.

**IAM groups**

Groups are collections of IAM users. A group is not an identity and cannot be authenticated directly. Instead, it is a permission management tool that simplifies administration.

Use groups to:

- Apply the same permissions to multiple users.
- Reduce policy duplication.
- Separate job functions such as developers, auditors, and administrators.

Groups are especially useful when permissions change frequently. You update the group policy once, and all members inherit the change.

Example group workflow:

```bash
aws iam create-group --group-name Developers
aws iam attach-group-policy \
  --group-name Developers \
  --policy-arn arn:aws:iam::aws:policy/AWSCodeCommitPowerUser
aws iam add-user-to-group --group-name Developers --user-name alice
```

Group best practices:

- Keep groups aligned to job functions, not individuals.
- Avoid putting users into too many groups, which makes troubleshooting harder.
- Use groups for coarse-grained access and individual policies only for exceptions.

**IAM roles**

A role is an identity with permissions that can be assumed temporarily. Roles do not have long-term credentials. Instead, a trusted principal assumes the role and receives temporary security credentials from AWS Security Token Service (<abbr title="AWS Security Token Service">STS</abbr>).

Roles are the preferred choice for:

- EC2 instances that need to call AWS APIs.
- Lambda functions that access other services.
- Cross-account access.
- Federated access from an external identity provider.
- Temporary elevated access for administrators.

Role anatomy:

- **Trust policy** defines who can assume the role.
- **Permissions policy** defines what the role can do after it is assumed.
- **Session** represents the temporary credentials issued by STS.

A common EC2 instance profile example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-app-bucket/*"
    }
  ]
}
```

For EC2, you attach the role to an instance profile, and the instance retrieves credentials automatically from the metadata service. This avoids storing access keys on the instance.

**Policies overview**

Policies are JSON documents that define permissions. IAM uses several policy types, and exam questions often hinge on which type applies and how they combine.

Common policy types:

- **Identity-based policies** attach to users, groups, or roles and grant permissions to the identity.
- **Resource-based policies** attach to resources such as S3 buckets, KMS keys, SNS topics, or SQS queues and grant access to principals.
- **Permissions boundaries** set the maximum permissions an IAM identity can receive.
- **Service control policies** (<abbr title="Service Control Policies">SCPs</abbr>) set the maximum permissions in AWS Organizations accounts.
- **Session policies** further restrict permissions during role assumption.
- **Inline policies** are embedded directly in a single user, group, or role.
- **Managed policies** are reusable standalone policies that can be attached to multiple identities.

Inline versus managed policies:

- **Inline policies**
  - Live inside one identity.
  - Are tightly coupled to that identity.
  - Are harder to reuse and audit at scale.
- **Managed policies**
  - Are separate objects.
  - Can be AWS managed or customer managed.
  - Are easier to reuse, version, and standardize.

Use managed policies for most production access control. Use inline policies only when you need a one-off permission set that should not be reused.

**Policy evaluation order and edge cases**

AWS evaluates permissions by combining all applicable policy layers. The final result is not “most permissive wins”; explicit denies always win.

A practical evaluation model:

1. Start with an implicit deny.
2. Check for any explicit deny in any applicable policy.
3. Evaluate identity-based allows.
4. Evaluate resource-based allows, if the resource supports them.
5. Apply permissions boundaries, if present.
6. Apply SCPs, if the account is in AWS Organizations.
7. Apply session policies, if the principal assumed a role with session restrictions.

The important nuance is that different policy types constrain different parts of the request path:

- **Identity-based policies** grant permissions to the caller.
- **Resource-based policies** can grant access directly to a principal, including cross-account access.
- **Permissions boundaries** cap what an IAM user or role can ever do, even if another policy allows more.
- **SCPs** cap the maximum permissions available in the account, including to administrators.
- **Session policies** narrow the permissions of a specific assumed-role session.

Edge cases to remember:

- A resource-based policy can allow access even when the caller has no identity-based allow, as long as no boundary or SCP blocks it.
- A permissions boundary does not grant permissions; it only limits them.
- An SCP does not grant permissions; it only limits what identities in the account can do.
- A session policy can reduce permissions below the role’s base permissions.
- An explicit deny in any applicable policy overrides all allows.

A useful mental model is: **allow must survive every layer, and deny only needs to appear once**.

**Policy structure**

IAM policies use a standard JSON structure. The most common elements are `Version`, `Statement`, `Effect`, `Action`, `Resource`, and `Condition`.

Example identity-based policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadOnlyAccessToOneBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

Key fields:

- **Effect** is `Allow` or `Deny`.
- **Action** lists API operations such as `s3:GetObject`.
- **Resource** specifies the ARN or ARNs affected.
- **Condition** adds contextual logic such as source IP, MFA, or tags.

**Common condition keys and policy variables**

Conditions are a major exam topic because they let you write context-aware access rules. They often appear in scenarios involving MFA, network restrictions, or tag-based access control.

Common condition keys include:

- `aws:MultiFactorAuthPresent` for MFA enforcement.
- `aws:SourceIp` for IP-based restrictions.
- `aws:PrincipalArn` to match the calling principal.
- `aws:PrincipalTag/<tag-key>` for principal tags.
- `aws:ResourceTag/<tag-key>` for resource tags.
- `aws:RequestedRegion` to restrict Region usage.
- `aws:username` for user-specific logic.
- `s3:prefix` and `s3:delimiter` for S3 list controls.
- `kms:ViaService` for KMS usage through a specific AWS service.

Policy variables let you build reusable policies with dynamic values. They are commonly used with usernames or tags.

Example using a policy variable:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::company-home/${aws:username}/*"
    }
  ]
}
```

This pattern lets each user write only to a personal prefix that matches their username.

Tag-based access control example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "ec2:StartInstances",
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:ResourceTag/Environment": "Dev"
        }
      }
    }
  ]
}
```

Use conditions when you need:

- Time-based or MFA-based restrictions.
- Network-based restrictions.
- Environment separation such as `Dev`, `Test`, and `Prod`.
- Attribute-based access control (<abbr title="Attribute-Based Access Control">ABAC</abbr>) using tags.

**Managed policies, inline policies, and permissions boundaries**

Managed policies are the standard reusable unit for permissions. AWS managed policies are maintained by AWS, while customer managed policies are created in your account.

Permissions boundaries are often confused with managed policies, but they serve a different purpose. A boundary is a maximum permission set for an identity. It is especially useful in delegated administration models where you let teams create IAM users or roles but prevent privilege escalation.

Example boundary concept:

- A developer can create a role.
- The role can only ever get permissions allowed by the boundary.
- Even if the developer attaches a broader policy, the boundary still caps the result.

This is a common exam pattern for preventing accidental or malicious over-permissioning.

**Cross-account access and role assumption flow**

Cross-account access is usually implemented with roles, not users. The trusted account creates a role in the target account, and the caller assumes that role using STS.

A full flow looks like this:

1. In the target account, create a role with a trust policy that allows the source account or a specific principal to assume it.
2. Attach a permissions policy to the role that grants the required access in the target account.
3. In the source account, allow the caller to invoke `sts:AssumeRole` on that role.
4. The caller uses STS to obtain temporary credentials.
5. The caller uses the temporary credentials to access resources in the target account.

Example trust policy in the target account:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "example-external-id"
        }
      }
    }
  ]
}
```

Example permissions policy attached to the role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:Query"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:444455556666:table/Orders"
    }
  ]
}
```

Example assume-role command:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/CrossAccountReadRole \
  --role-session-name alice-session \
  --external-id example-external-id
```

After you receive the temporary credentials, export them and use them for subsequent API calls:

```bash
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
aws dynamodb query --table-name Orders --key-condition-expression "OrderId = :id" \
  --expression-attribute-values '{":id":{"S":"123"}}'
```

For federated access, the identity provider authenticates the user first, then AWS maps that identity to a role. The user never receives long-term AWS credentials.

**Federated access and SSO-style access**

Federation is the pattern where an external identity source, such as an enterprise directory or SAML 2.0 provider, authenticates the user and AWS grants temporary access through a role.

Typical flow:

- The user signs in to the corporate identity provider.
- The identity provider asserts the user identity to AWS.
- AWS issues temporary role credentials.
- The user accesses the console or CLI without an IAM user password.

This is preferred for workforce access because it centralizes identity management and reduces credential sprawl.

**IAM policy simulator and validation**

The IAM policy simulator helps you test whether a principal can perform an action on a resource before you deploy or troubleshoot access. It is useful when you need to distinguish between an identity policy problem, a resource policy problem, or an SCP/boundary restriction.

Use it when:

- A user gets `AccessDenied` and you need to isolate the cause.
- You want to verify a new policy before production rollout.
- You need to test conditions such as MFA or tags.

Practical validation tools:

- **IAM Policy Simulator** in the console.
- `aws iam simulate-principal-policy` for CLI-based testing.
- CloudTrail for auditing actual authorization failures and API calls.

Example CLI simulation:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::111122223333:user/alice \
  --action-names s3:GetObject s3:PutObject \
  --resource-arns arn:aws:s3:::example-bucket/report.csv
```

If the result is unexpected, check these layers in order:

- Identity policy attached to the user, group, or role.
- Resource-based policy on the target resource.
- Permissions boundary on the principal.
- SCP on the account or OU.
- Session policy on the assumed role session.
- Explicit deny statements anywhere in the path.

**Practical troubleshooting patterns**

When access fails, the fastest path is to identify the policy layer that blocks the request.

Common causes:

- The action is missing from the identity policy.
- The resource ARN is too narrow or incorrect.
- A condition key does not match the request context.
- A permissions boundary blocks the allow.
- An SCP denies the action at the organization level.
- The role session is restricted by a session policy.
- The resource policy trusts the wrong principal.

Useful debugging approach:

1. Confirm the exact API action.
2. Confirm the exact resource ARN.
3. Check for explicit denies.
4. Check conditions such as MFA, source IP, or tags.
5. Validate with the policy simulator.
6. Review CloudTrail for the denied event.

**Exam-relevant design guidance**

Choose the identity type based on the access pattern:

- Use **IAM users** for named human identities only when federation is not available.
- Use **groups** to simplify permission management for users with similar jobs.
- Use **roles** for AWS services, cross-account access, and temporary access.
- Use **resource-based policies** when the resource itself should grant access, especially for cross-account sharing.
- Use **permissions boundaries** when you delegate IAM administration but need guardrails.
- Use **SCPs** when you need organization-wide guardrails across accounts.

Common exam traps:

- Using an IAM user with access keys for an EC2 application instead of an instance role.
- Assuming a managed policy can override an SCP or permissions boundary.
- Forgetting that explicit deny wins over allow.
- Confusing a group with an identity.
- Using the root user for routine administration.
- Ignoring session policies when a role is assumed with additional restrictions.

**Quick comparison**

| Concept | What it is | Best use |
|---|---|---|
| IAM user | Long-term named identity | Human access when federation is unavailable |
| IAM group | Collection of users | Shared permissions by job function |
| IAM role | Temporary assumable identity | AWS services, cross-account, federation |
| Identity-based policy | Attached to identity | Grant permissions to users, groups, roles |
| Resource-based policy | Attached to resource | Direct access to a resource, often cross-account |
| Permissions boundary | Maximum permissions for an identity | Delegated IAM administration |
| SCP | Maximum permissions for an account | Organization-wide guardrails |
| Session policy | Temporary restriction on assumed role | Narrow access for a specific session |

**High-value commands**

Inspect attached policies:

```bash
aws iam list-attached-user-policies --user-name alice
aws iam list-attached-group-policies --group-name Developers
aws iam list-attached-role-policies --role-name AppRole
```

View inline policies:

```bash
aws iam list-user-policies --user-name alice
aws iam list-role-policies --role-name AppRole
```

Get a policy document:

```bash
aws iam get-policy-version \
  --policy-arn arn:aws:iam::123456789012:policy/DeveloperAccess \
  --version-id v1
```

Simulate a permission:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names ec2:DescribeInstances \
  --resource-arns "*"
```

**Key takeaways**

IAM access control is layered, and the final authorization decision depends on all applicable policy types. Users and groups are best for long-term human administration, while roles are the preferred mechanism for temporary access and AWS service permissions. Managed policies scale better than inline policies, permissions boundaries and SCPs cap privilege, and conditions plus policy variables let you write precise, context-aware rules.
