## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Placement groups overview**

Placement groups are an **Amazon EC2** feature that influences how instances are placed on underlying hardware. You use them when you need either lower network latency, higher throughput, or better fault isolation for a set of EC2 instances. They are not a general-purpose feature for other AWS services such as RDS or EBS.

A placement group is created in a **single VPC** and cannot span multiple VPCs. This is an important exam distinction because the placement decision applies to EC2 instances launched into that VPC only. If you need similar behavior in another VPC, you create a separate placement group there.

- **EC2-only** feature
- Created **per VPC**
- Used for performance or isolation goals
- Not used for **RDS**, **EBS**, or other non-EC2 services

**Placement group strategies**

AWS supports three placement group strategies. Each one solves a different architecture problem, so the exam often asks you to choose the correct strategy based on workload behavior.

| Strategy | Primary goal | Key exam point | Typical use case |
|---|---|---|---|
| Cluster | Lowest latency, highest packet-per-second performance | Instances are placed close together in a single AZ | HPC, tightly coupled distributed systems |
| Spread | Reduce correlated failure risk | Limited number of running instances per AZ per placement group | Critical small fleet, quorum nodes |
| Partition | Isolate large distributed workloads | Each partition can contain multiple instances; partitions are spread across racks | Big data, distributed databases |

**Cluster placement groups**

Cluster placement groups pack instances close together inside a single Availability Zone (AZ). This improves network performance between instances and is the best choice when your application depends on very low latency or high bandwidth between nodes.

Use cluster placement groups when the workload is tightly coupled and network-sensitive, such as:
- High-performance computing (HPC)
- Financial modeling
- Distributed analytics that require fast east-west traffic
- Latency-sensitive application tiers

Important constraints:
- All instances must be in the **same AZ**
- You get the best network performance when instances are launched into the same cluster placement group
- This strategy improves performance, not fault tolerance

> A cluster placement group is about **proximity**, not resilience. If the AZ has an issue, all instances in that group can still be affected.

**Spread placement groups**

Spread placement groups place each instance on distinct underlying hardware to reduce the chance that a single hardware failure impacts multiple critical instances. This is useful for small numbers of important instances that should not fail together.

The key exam constraint is that there is a limit of **7 running instances per Availability Zone per spread placement group**. This limit matters when you design highly available control-plane or quorum-based systems.

Use spread placement groups for:
- Critical instances that must be isolated from each other
- Small fleets where each node should fail independently
- Quorum nodes, bastion hosts, or standalone critical services

Practical implications:
- Instances are intentionally separated
- You trade off performance density for fault isolation
- The limit applies per AZ within the placement group

**Partition placement groups**

Partition placement groups are designed for large distributed systems that need both scale and fault isolation. AWS divides instances into partitions, and each partition is placed on separate racks or failure domains.

Each partition can contain **multiple instances**, which is a key distinction from spread placement groups. The maximum number of partitions is an exam-relevant limit that is commonly **7** for EC2 placement groups, so you should know that number for SAA-C03-style questions.

Use partition placement groups for:
- Large distributed databases
- Hadoop, Cassandra, Kafka, and similar systems
- Workloads where you want failure isolation across partitions, not necessarily across every single instance

Why partition groups matter:
- A failure in one partition affects only the instances in that partition
- You can scale to larger fleets than spread placement groups
- You still gain isolation, but with more flexibility than spread

**Direct placement group example**

You often need to create the placement group first, then reference it when launching instances.

```bash
aws ec2 create-placement-group \
  --group-name app-cluster-pg \
  --strategy cluster

aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type c6i.large \
  --count 2 \
  --subnet-id subnet-0abc1234def567890 \
  --placement GroupName=app-cluster-pg
```

For a spread placement group, the creation command changes only the strategy:

```bash
aws ec2 create-placement-group \
  --group-name critical-spread-pg \
  --strategy spread
```

For a partition placement group, you specify the partition strategy:

```bash
aws ec2 create-placement-group \
  --group-name data-partition-pg \
  --strategy partition
```

**When to choose each placement group**

Think in terms of the workload’s dominant requirement:

- **Cluster** when you need the best network performance
- **Spread** when you need the strongest instance-level fault isolation for a small number of instances
- **Partition** when you need isolation at scale for distributed systems

A useful exam shortcut:
- If the question says “lowest latency between nodes,” choose **cluster**
- If it says “keep critical instances on separate hardware,” choose **spread**
- If it says “large distributed system with isolated failure domains,” choose **partition**

**Launch templates overview**

Launch templates define the configuration used to launch EC2 instances consistently. They are the preferred way to standardize instance settings for Auto Scaling Groups (ASGs), EC2 Fleet, Spot Fleet, and manual launches.

A launch template can include:
- AMI (Amazon Machine Image)
- Instance type
- Key pair
- Security groups
- Subnet or network settings
- IAM instance profile
- User data
- Block device mappings
- Tag specifications
- Metadata options

Launch templates are especially important because they support **versioning**. You can create multiple versions of the same template as your configuration evolves, then choose:
- a specific version
- the **default version**
- the **$Latest** version

This distinction is exam-relevant because different services can reference different versions depending on how you configure them.

**Launch template versions**

Versioning lets you safely change instance configuration without overwriting the original template. This is useful when you want to test a new AMI, update user data, or change security settings while preserving rollback options.

Common version usage patterns:
- **Default version**: stable baseline used when no version is explicitly specified
- **$Latest**: newest created version, useful for rapid iteration but riskier in production
- **Specific version**: best for controlled deployments and predictable rollouts

Example launch template creation:

```bash
aws ec2 create-launch-template \
  --launch-template-name web-app-template \
  --version-description "Initial version" \
  --launch-template-data '{
    "ImageId":"ami-0123456789abcdef0",
    "InstanceType":"t3.micro",
    "KeyName":"my-keypair",
    "SecurityGroupIds":["sg-0123abcd4567efgh8"],
    "IamInstanceProfile":{"Name":"EC2AppRole"},
    "MetadataOptions":{
      "HttpTokens":"required",
      "HttpEndpoint":"enabled"
    },
    "UserData":"IyEvYmluL2Jhc2gKZWNobyAnSGVsbG8nID4gL3Zhci93d3cvaHRtbC9pbmRleC5odG1s"
  }'
```

Example of creating a new version:

```bash
aws ec2 create-launch-template-version \
  --launch-template-name web-app-template \
  --source-version 1 \
  --version-description "Update AMI and user data" \
  --launch-template-data '{
    "ImageId":"ami-0fedcba9876543210",
    "InstanceType":"t3.micro"
  }'
```

You can set the default version explicitly:

```bash
aws ec2 modify-launch-template \
  --launch-template-name web-app-template \
  --default-version 2
```

**Launch templates in Auto Scaling Groups and Fleets**

Launch templates are commonly used with ASGs and EC2 Fleet because they make scaling behavior repeatable. The version you reference determines which configuration new instances receive.

Typical patterns:
- Use a **specific version** for controlled production rollouts
- Use **$Latest** for development or rapid iteration
- Update the **default version** when you want a stable baseline to move forward

Example ASG configuration with a launch template version:

```json
{
  "LaunchTemplate": {
    "LaunchTemplateName": "web-app-template",
    "Version": "$Latest"
  },
  "MinSize": 2,
  "MaxSize": 6,
  "DesiredCapacity": 3
}
```

Or with a specific version:

```json
{
  "LaunchTemplate": {
    "LaunchTemplateName": "web-app-template",
    "Version": "2"
  }
}
```

This matters because if you update the template but the ASG still points to version `1`, new instances continue using version `1` until you change the ASG configuration.

**Instance metadata options**

Launch templates commonly include **instance metadata options**, which control access to the Instance Metadata Service (IMDS). The most exam-relevant setting is enforcing **IMDSv2** by requiring session tokens.

Why this matters:
- IMDS provides instance identity and temporary credentials-related metadata
- IMDSv2 improves security by requiring a token-based session
- Requiring tokens helps reduce SSRF (Server-Side Request Forgery) risk

Example metadata options in a launch template:

```json
{
  "MetadataOptions": {
    "HttpTokens": "required",
    "HttpEndpoint": "enabled",
    "HttpPutResponseHopLimit": 2
  }
}
```

Key settings:
- `HttpTokens: required` enforces IMDSv2
- `HttpEndpoint: enabled` allows metadata access
- `HttpPutResponseHopLimit` controls how far metadata responses can travel

> If a question asks how to enforce IMDSv2 for new instances at launch time, the launch template is a strong answer.

**Practical launch template workflow**

A common workflow is to create a template, create a new version when settings change, and then point your ASG or Fleet to the desired version.

1. Create the initial launch template.
2. Create a new version when you change AMI, user data, or metadata settings.
3. Set the default version if you want a stable baseline.
4. Reference `$Latest`, the default version, or a specific version in your scaling service.
5. Roll out changes by updating the ASG or Fleet configuration.

**Example: ASG using a launch template**

```bash
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name web-asg \
  --min-size 2 \
  --max-size 4 \
  --desired-capacity 2 \
  --vpc-zone-identifier "subnet-aaa,subnet-bbb" \
  --launch-template LaunchTemplateName=web-app-template,Version=2
```

This is a common exam pattern: the ASG does not use the template name alone; it uses a versioned reference.

**Placement groups vs. launch templates**

These two features solve different problems, and the exam may try to mix them together.

| Feature | Primary purpose | Scope | Common exam clue |
|---|---|---|---|
| Placement group | Control instance placement for performance or isolation | EC2 instances in a VPC | “low latency,” “separate hardware,” “failure domain” |
| Launch template | Standardize instance configuration | EC2 launch settings | “versioned config,” “IMDSv2,” “ASG/Fleet launch settings” |

Important distinction:
- Placement groups affect **where** instances land
- Launch templates affect **how** instances are configured

**Common exam traps**

- Placement groups are **not** for RDS, EBS, or other non-EC2 services
- A placement group is created **per VPC** and cannot span VPCs
- Spread placement groups have a **7 running instances per AZ per group** limit
- Partition placement groups allow **multiple instances per partition**
- Partition groups have a maximum partition count that is commonly **7**
- Launch templates support **versions**, and services may use the default, `$Latest`, or a specific version
- IMDSv2 enforcement belongs in the launch template, not in the placement group

**Quick decision guide**

- Need best network performance between EC2 instances? Use **cluster placement groups**
- Need to keep a few critical instances apart? Use **spread placement groups**
- Need scalable isolation for a distributed system? Use **partition placement groups**
- Need repeatable EC2 configuration with version control? Use **launch templates**
- Need to enforce IMDSv2 or standardize ASG launches? Use **launch templates**

**Exam-ready summary**

Placement groups are an EC2-specific way to influence instance placement inside a VPC. Cluster groups optimize performance, spread groups maximize isolation for a small number of instances, and partition groups isolate large distributed workloads across partitions. Launch templates standardize EC2 configuration and support versioning, making them the preferred choice for ASGs, Fleets, and controlled instance launches.
