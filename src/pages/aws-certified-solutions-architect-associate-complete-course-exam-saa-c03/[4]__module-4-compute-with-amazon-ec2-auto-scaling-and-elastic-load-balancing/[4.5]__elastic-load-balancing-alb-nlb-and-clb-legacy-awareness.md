## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Elastic Load Balancing overview**

Elastic Load Balancing (ELB) distributes incoming traffic across multiple targets to improve availability, scale, and fault tolerance. In AWS exam questions, the key is not just knowing that load balancers exist, but choosing the right family for the workload.

- **ALB** = Application Load Balancer for HTTP/HTTPS application traffic.
- **NLB** = Network Load Balancer for ultra-low latency, high throughput, and static IP use cases.
- **CLB** = Classic Load Balancer, a legacy option with fewer features and weaker exam relevance unless the question is explicitly old or migration-related.

ELB is usually deployed across multiple Availability Zones (AZs) for resilience. For internet-facing architectures, the load balancer is the public entry point; for internal architectures, it stays private and fronts services inside the VPC.

**Core comparison: when to choose each load balancer**

Use the load balancer type that matches the protocol and routing needs of the application.

| Feature | ALB | NLB | CLB |
|---|---|---|---|
| Primary layer | Layer 7 | Layer 4 | Layer 4 and limited Layer 7 legacy behavior |
| Best for | Web apps, APIs, microservices | TCP/UDP/TLS traffic, static IP, extreme performance | Legacy workloads |
| Content-based routing | Yes: host, path, header, query string, source IP condition | No | Very limited, legacy |
| Target types | Instance, IP, Lambda | Instance, IP, ALB | Instance, EC2-Classic legacy |
| TLS termination | Yes | Yes, or pass-through with TLS listener | Yes for HTTPS/SSL listeners |
| WebSocket support | Yes | No native Layer 7 awareness | Limited/legacy |
| gRPC support | Yes | No native Layer 7 awareness | No |
| Security groups | Yes | Not in the same way as ALB/CLB | Yes |
| Cross-zone load balancing | Always on | Supported, configurable behavior depends on setup | Supported, behavior depends on type |
| Modern recommendation | Preferred for HTTP/HTTPS | Preferred for TCP/UDP/TLS | Avoid unless required |

**Application Load Balancer details**

ALB operates at Layer 7, so it understands HTTP request content. This makes it the right choice when routing decisions depend on the request itself.

- **Host-based routing** sends `api.example.com` and `www.example.com` to different target groups.
- **Path-based routing** sends `/images/*` to one service and `/api/*` to another.
- **Header-based routing** can inspect HTTP headers for canary or tenant routing.
- **Query-string routing** can direct traffic based on URL parameters.
- **Source IP condition** can be used in listener rules, but it is not the same as content-based routing on host/path/header/query string; treat it as a separate rule condition.

ALB is the only ELB family in this topic that supports advanced content-based routing. NLB cannot inspect HTTP content, and CLB does not provide the same modern rule model.

**ALB target groups and target types**

ALB target groups can register different target types depending on the architecture.

- **Instance targets** register EC2 instances by instance ID.
- **IP targets** register private IP addresses, which is useful for containers, on-premises connectivity, or non-EC2 endpoints inside a VPC.
- **Lambda targets** are supported only by ALB, which is a common exam trap.
- **NLB and CLB do not support Lambda targets.**

Practical implications:

- Use **instance** targets when your service runs on EC2 and you want simple registration.
- Use **IP** targets when you need to route to ECS tasks with awsvpc networking, appliances, or hybrid endpoints.
- Use **Lambda** targets when you want synchronous HTTP-style invocation through ALB without managing servers.

Example target group creation:

```bash
aws elbv2 create-target-group \
  --name app-tg \
  --protocol HTTP \
  --port 80 \
  --vpc-id vpc-0123456789abcdef0 \
  --target-type instance
```

Lambda target group example:

```bash
aws elbv2 create-target-group \
  --name lambda-tg \
  --target-type lambda \
  --vpc-id vpc-0123456789abcdef0
```

**ALB listeners and rules**

A listener checks for connection requests on a protocol and port, then applies rules to decide where to send traffic.

Common listener patterns:

- `HTTP:80` redirecting to `HTTPS:443`
- `HTTPS:443` with ACM (AWS Certificate Manager) certificates
- Multiple rules on one listener for different hostnames or paths

Example listener creation:

```bash
aws elbv2 create-listener \
  --load-balancer-arn arn:aws:elasticloadbalancing:region:acct:loadbalancer/app/my-alb/1234567890abcdef \
  --protocol HTTP \
  --port 80 \
  --default-actions Type=redirect,RedirectConfig='{Protocol=HTTPS,Port=443,StatusCode=HTTP_301}'
```

Example rule for path-based routing:

```bash
aws elbv2 create-rule \
  --listener-arn arn:aws:elasticloadbalancing:region:acct:listener/app/my-alb/1234567890abcdef/abcdef1234567890 \
  --priority 10 \
  --conditions Field=path-pattern,Values='/api/*' \
  --actions Type=forward,TargetGroupArn=arn:aws:elasticloadbalancing:region:acct:targetgroup/api-tg/abcdef1234567890
```

ALB is a strong fit for microservices because you can route requests to different services without exposing each service directly.

**ALB TLS, WebSocket, and gRPC behavior**

ALB can terminate TLS at the load balancer, which simplifies certificate management and offloads encryption from targets.

- **TLS termination** means the client connects with HTTPS, and ALB decrypts traffic before forwarding to targets.
- You can also re-encrypt traffic from ALB to targets if your backend requires HTTPS.
- ALB supports **WebSocket** connections, which is useful for real-time web applications.
- ALB supports **gRPC** over HTTP/2, which matters for modern service-to-service and API designs.

Use ALB when you need application-aware features and modern HTTP behavior. If the question emphasizes HTTP routing, URL paths, hostnames, or gRPC/WebSocket support, ALB is usually the correct answer.

**ALB health checks and idle timeout**

ALB target group health checks are application-aware and typically use HTTP or HTTPS.

- Health checks can verify a specific path such as `/health`.
- You can tune matcher codes, interval, timeout, healthy threshold, and unhealthy threshold.
- ALB has an **idle timeout** setting that affects how long an idle connection stays open before the load balancer closes it.

Example target group health check configuration:

```bash
aws elbv2 modify-target-group \
  --target-group-arn arn:aws:elasticloadbalancing:region:acct:targetgroup/app-tg/abcdef1234567890 \
  --health-check-protocol HTTP \
  --health-check-path /health \
  --health-check-interval-seconds 30 \
  --health-check-timeout-seconds 5 \
  --healthy-threshold-count 5 \
  --unhealthy-threshold-count 2
```

Idle timeout example:

```bash
aws elbv2 modify-load-balancer-attributes \
  --load-balancer-arn arn:aws:elasticloadbalancing:region:acct:loadbalancer/app/my-alb/1234567890abcdef \
  --attributes Key=idle_timeout.timeout_seconds,Value=120
```

Use a longer idle timeout for long-lived HTTP connections, but keep it aligned with application behavior to avoid unexpected disconnects.

**Network Load Balancer details**

NLB operates at Layer 4, so it forwards connections based on IP protocol, port, and TCP/UDP/TLS transport behavior rather than HTTP content.

- **TCP** is used for generic TCP traffic.
- **UDP** is supported for UDP workloads.
- **TLS** listener support allows encrypted transport handling at Layer 4.
- NLB is ideal for very high throughput, low latency, and static IP requirements.

NLB is commonly chosen when:

- You need **static IP addresses** or Elastic IPs.
- You need to preserve the client source IP to the target.
- You need to handle non-HTTP protocols.
- You need extreme performance with minimal load balancer processing.

NLB does not do host-based or path-based routing because it does not inspect HTTP request content. If a question asks for routing by URL path or hostname, NLB is not the answer.

**NLB TLS handling: termination vs pass-through**

NLB can either terminate TLS at the load balancer or pass encrypted traffic through to the target, depending on the listener and target configuration.

- **TLS termination** at NLB decrypts traffic at the load balancer and forwards plaintext or re-encrypted traffic to targets.
- **TLS pass-through** preserves encryption end to end, which is useful when the backend must handle its own certificates or when you want the load balancer to remain protocol-agnostic.

Choose TLS termination when you want centralized certificate management and simpler backend services. Choose pass-through when the backend must inspect or terminate its own TLS session.

**NLB target groups and health checks**

NLB target groups commonly use instance or IP targets.

- **Instance targets** are for EC2 instances.
- **IP targets** are for IP-based endpoints.
- NLB does **not** support Lambda targets.

Health checks for NLB target groups can use:

- **TCP**
- **HTTP**
- **HTTPS**

NLB health checks are not a TLS health check feature in the same sense as a listener protocol. Be careful not to confuse listener protocols with health check protocols.

Example NLB target group:

```bash
aws elbv2 create-target-group \
  --name nlb-tg \
  --protocol TCP \
  --port 443 \
  --vpc-id vpc-0123456789abcdef0 \
  --target-type instance
```

Example TCP health check:

```bash
aws elbv2 modify-target-group \
  --target-group-arn arn:aws:elasticloadbalancing:region:acct:targetgroup/nlb-tg/abcdef1234567890 \
  --health-check-protocol TCP \
  --health-check-port traffic-port
```

**NLB architecture and security behavior**

NLB is often used when you need network-level exposure with minimal interference.

- NLB can preserve the client source IP to the target.
- NLB is commonly associated with static IPs and Elastic IPs.
- NLB historically does not use security groups in the same way ALB and CLB do, so network access control is often enforced with target security groups, NACLs, and application controls.
- Because NLB is Layer 4, it cannot make routing decisions based on HTTP headers, paths, or query strings.

For exam scenarios, if you need a public endpoint with fixed IPs for allowlisting, NLB is often the best fit.

**Classic Load Balancer legacy awareness**

Classic Load Balancer is the older ELB generation and appears in legacy architectures or migration questions.

- CLB is the original Elastic Load Balancer offering.
- It supports older-style listener configurations and simpler routing.
- It lacks the modern rule engine of ALB.
- It is not the preferred choice for new designs.

CLB behavior depends on the load balancer type:

- **HTTP/HTTPS CLB** uses HTTP/HTTPS listeners and application-style health checks.
- **TCP CLB** uses TCP listeners and TCP-oriented health checks.

This is why CLB can be confusing in exam questions: it mixes legacy application and network behavior without the clean separation found in ALB and NLB.

**CLB listener and health check behavior**

CLB supports a smaller, older feature set than ALB.

- It can front HTTP, HTTPS, TCP, and SSL-style traffic depending on the load balancer type.
- It does not provide ALB-style host/path/header/query-string routing.
- It does not support Lambda targets.
- It is generally a migration or compatibility topic rather than a first-choice architecture.

Example CLB creation with the older CLI:

```bash
aws elb create-load-balancer \
  --load-balancer-name legacy-clb \
  --listeners Protocol=HTTP,LoadBalancerPort=80,InstanceProtocol=HTTP,InstancePort=80 \
  --availability-zones us-east-1a us-east-1b
```

Example CLB health check:

```bash
aws elb configure-health-check \
  --load-balancer-name legacy-clb \
  --health-check Target=HTTP:80/health,Interval=30,Timeout=5,UnhealthyThreshold=2,HealthyThreshold=5
```

For TCP CLB, the health check target is TCP-based rather than HTTP-based. Do not assume one universal health check model across all CLB types.

**Security groups and network controls**

Security group behavior is an important exam comparison point.

- **ALB** uses security groups and is commonly placed in public subnets for internet-facing workloads.
- **CLB** also uses security groups.
- **NLB** does not behave like ALB/CLB with respect to security groups, so you often rely more on target security groups, NACLs, and subnet design.

Practical implication:

- If the question emphasizes security group-based filtering at the load balancer layer, ALB or CLB is more likely than NLB.
- If the question emphasizes static IPs and network-level forwarding, NLB is more likely.

**Internet-facing vs internal load balancers**

The `scheme` determines whether the load balancer is public or private.

- **Internet-facing** load balancers have public DNS names and sit in public subnets.
- **Internal** load balancers are private to the VPC and are used for internal services, shared services, and multi-tier architectures.

Use an internet-facing ALB for public web apps and APIs. Use an internal ALB for service-to-service routing inside a VPC. Use an internal NLB when you need private, high-performance TCP/UDP/TLS forwarding.

Example ALB creation with scheme:

```bash
aws elbv2 create-load-balancer \
  --name app-alb \
  --type application \
  --scheme internet-facing \
  --subnets subnet-aaa subnet-bbb
```

Example internal NLB:

```bash
aws elbv2 create-load-balancer \
  --name internal-nlb \
  --type network \
  --scheme internal \
  --subnets subnet-ccc subnet-ddd
```

**Cross-zone load balancing**

Cross-zone load balancing distributes traffic across targets in multiple AZs.

- **ALB** always uses cross-zone load balancing; you do not toggle it on or off.
- **NLB** supports cross-zone behavior, but it is not the same always-on model as ALB.
- **CLB** has legacy behavior that depends on the load balancer type and configuration.

For exam purposes, remember that ALB automatically balances across enabled AZs without a user-configurable cross-zone switch.

**Access logs, metrics, and monitoring**

ELB integrates with CloudWatch for metrics and with S3 for access logs, but the exact logging and observability story differs by load balancer family.

- **ALB** provides detailed access logs and rich HTTP metrics.
- **NLB** provides network-oriented metrics and flow visibility, with less application context.
- **CLB** has older logging and monitoring capabilities.

Useful CloudWatch metrics include:

- `RequestCount`
- `TargetResponseTime`
- `HTTPCode_Target_5XX_Count`
- `HealthyHostCount`
- `UnHealthyHostCount`
- `ProcessedBytes`

Example CloudWatch alarm idea:

- Alert when `UnHealthyHostCount` is greater than `0` for 2 consecutive periods.
- Alert when `HTTPCode_Target_5XX_Count` spikes after a deployment.

Example access log destination pattern:

```text
s3://my-lb-logs/AWSLogs/<account-id>/elasticloadbalancing/<region>/
```

Use logs to troubleshoot 4xx/5xx errors, target registration issues, and latency spikes.

**Practical architecture patterns**

Common exam-ready patterns help you map requirements to the right ELB family.

- **Public web app with multiple services**
  - Use an internet-facing ALB.
  - Route by host and path.
  - Terminate TLS with ACM.
  - Forward to separate target groups.

- **Private internal API tier**
  - Use an internal ALB.
  - Keep traffic inside the VPC.
  - Use path-based routing for microservices.

- **High-performance TCP service**
  - Use an NLB.
  - Preserve source IP.
  - Use static IPs or Elastic IPs for allowlisting.

- **Legacy migration**
  - Use CLB only if the workload depends on older behavior or the question explicitly references it.
  - Prefer migration to ALB or NLB when possible.

**Exam traps and decision cues**

Watch for wording that points to one ELB family over another.

- If the question says **hostnames, paths, headers, query strings, or gRPC**, choose ALB.
- If the question says **static IP, low latency, TCP/UDP, or source IP preservation**, choose NLB.
- If the question says **legacy application** or **old ELB**, consider CLB.
- If the question says **Lambda target**, choose ALB.
- If the question says **WebSocket**, choose ALB.
- If the question says **TLS pass-through** or **network-level forwarding**, choose NLB.
- If the question says **security groups at the load balancer**, ALB or CLB is more likely than NLB.

> Be careful with legacy wording: some older materials describe CLB as if it were a general-purpose load balancer. For SAA-C03, ALB and NLB are the modern defaults, and CLB is mainly a compatibility topic.

**Quick reference summary**

- **ALB**: best for HTTP/HTTPS, content-based routing, Lambda targets, WebSocket, and gRPC.
- **NLB**: best for TCP/UDP/TLS, static IPs, source IP preservation, and very high performance.
- **CLB**: legacy only, with older listener and health check behavior.

**Hands-on CLI workflow**

A simple deployment flow usually looks like this:

1. Create the target group.
2. Register targets.
3. Create the load balancer.
4. Create listeners.
5. Add rules if using ALB.
6. Verify health checks and logs.

Example target registration:

```bash
aws elbv2 register-targets \
  --target-group-arn arn:aws:elasticloadbalancing:region:acct:targetgroup/app-tg/abcdef1234567890 \
  --targets Id=i-0123456789abcdef0 Id=i-0fedcba9876543210
```

Example listener verification:

```bash
aws elbv2 describe-listeners \
  --load-balancer-arn arn:aws:elasticloadbalancing:region:acct:loadbalancer/app/my-alb/1234567890abcdef
```

Example target health check inspection:

```bash
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:region:acct:targetgroup/app-tg/abcdef1234567890
```

**Final exam mindset**

When you see ELB in a question, identify three things first:

- The **protocol**: HTTP/HTTPS vs TCP/UDP/TLS.
- The **routing need**: content-aware vs network-level forwarding.
- The **operational constraint**: static IPs, Lambda, legacy compatibility, or internal-only access.

That sequence usually leads you to the correct answer faster than memorizing feature lists alone.
