## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon EBS fundamentals**

Amazon Elastic Block Store (EBS) provides persistent block storage for Amazon EC2 instances. You use it when you need low-latency storage that behaves like a disk, not like an object store. EBS volumes live in a single Availability Zone (AZ), so you must attach them to instances in the same AZ.

EBS is a common exam topic because it sits at the intersection of storage durability, instance lifecycle, and recovery design. The key idea is simple: the instance can fail or be replaced, but the EBS volume can remain and be reattached.

- **Block storage** is best for file systems, databases, boot volumes, and applications that expect a mounted disk.
- **Persistence** means the data remains after instance stop/start and after instance termination if the volume is configured to keep it.
- **AZ scope** means you cannot attach one EBS volume across AZs without creating a new volume from a snapshot or using another replication pattern.

**Root volumes and data volumes**

An EC2 instance can have a root volume and one or more data volumes. The root volume contains the operating system and boot files. Data volumes store application data, logs, databases, or shared files.

The distinction matters because termination behavior is different. Root volumes are often configured to delete on termination, while additional data volumes are commonly retained unless you explicitly delete them. In exam scenarios, always check the block device mapping and the delete-on-termination setting.

- **Root volume**
  - Set at launch through the instance’s block device mapping.
  - Contains the OS and boot loader.
  - Often deleted when the instance is terminated, depending on the mapping.
- **Data volume**
  - Attached after launch or at launch as an additional disk.
  - Used for application data and persistence.
  - Usually survives instance termination if configured to do so.

> A common trap: instance store is not the same as EBS. Instance store is ephemeral and should not be treated as a durable general-purpose root volume strategy for EBS-based designs.

**Volume types and performance behavior**

EBS volume type selection is often about balancing cost, latency, and throughput. For SAA-C03, you should know the practical differences between general-purpose SSD, provisioned IOPS SSD, and throughput-oriented options.

- **gp3**
  - Lets you configure size, IOPS, and throughput independently.
  - Good default choice for many workloads.
  - Useful when you need predictable performance without overprovisioning capacity.
- **gp2**
  - Performance scales with volume size.
  - Larger volumes generally provide more IOPS.
  - Can be cost-inefficient if you only need performance, not capacity.
- **io1 / io2**
  - Designed for high-performance, low-latency workloads.
  - Provisioned IOPS is the main tuning lever.
  - io2 is typically preferred when you need stronger durability characteristics.
- **st1 / sc1**
  - HDD-based options for throughput-oriented or infrequently accessed workloads.
  - Not ideal for boot volumes or latency-sensitive databases.

A useful exam heuristic is to choose gp3 when you want a modern default and io1/io2 when the question emphasizes sustained high IOPS, mission-critical databases, or strict performance predictability.

**Attachment limits and Multi-Attach**

Most EBS volumes attach to one EC2 instance at a time. That is the normal operating model and the one you should assume unless the question explicitly says otherwise.

Multi-Attach is a special capability for specific io1 and io2 volumes in supported configurations. It allows a single volume to be attached to multiple instances simultaneously, but it is not a general-purpose feature for all volume types.

- **Default behavior**
  - One EBS volume to one instance at a time.
  - You detach before attaching elsewhere.
- **Multi-Attach**
  - Supported only for specific io1/io2 use cases.
  - Requires applications that can coordinate concurrent block access.
  - Not a substitute for shared file storage unless the application is designed for it.

> Multi-Attach does not magically make a block volume behave like a shared file system. If multiple instances write to the same block device without coordination, you can corrupt data.

**Snapshots and restore behavior**

EBS snapshots are point-in-time backups of EBS volumes. They are incremental after the first snapshot, which means only changed blocks are stored for subsequent snapshots. For exam purposes, the important operational detail is that restoring from a snapshot creates a new EBS volume.

That new volume can then be attached to an instance in the same AZ as the volume. The first read from blocks that have not yet been fully initialized may be slower because restored data is lazily loaded. In practice, this means you may see performance variation immediately after restore until the volume is warmed up.

- **Snapshot characteristics**
  - Point-in-time backup of a volume.
  - Incremental storage after the initial snapshot.
  - Used for backup, migration, and recovery.
- **Restore behavior**
  - Creates a new volume, not an in-place replacement.
  - The new volume can be attached to an instance.
  - First reads may incur extra latency until blocks are initialized.

A common recovery pattern is to create a new volume from a snapshot, attach it to a replacement instance, and then mount it as the original data disk.

**Volume resizing and modification**

EBS volumes can be modified without replacing the instance. This is a common operational task and a frequent exam concept. You can increase volume size, and for some volume types you can also change performance characteristics such as IOPS and throughput.

After the EBS volume is expanded, you still need to grow the file system inside the operating system. The storage layer and the file system layer are separate steps.

1. Modify the EBS volume in AWS.
2. Wait for the modification to complete or reach an appropriate state.
3. Extend the partition if one exists.
4. Grow the file system.
5. Verify the new capacity.

Example Linux workflow:

```bash
# Check the attached disk and file system
lsblk
df -h

# Example for an ext4 file system after partition expansion
sudo growpart /dev/nvme0n1 1
sudo resize2fs /dev/nvme0n1p1

# Example for XFS
sudo xfs_growfs /
```

Example AWS CLI volume modification:

```bash
aws ec2 modify-volume \
  --volume-id vol-0123456789abcdef0 \
  --size 200 \
  --volume-type gp3 \
  --iops 6000 \
  --throughput 250
```

- **Modify volume** changes the EBS resource in AWS.
- **Resize file system** changes the OS view of the disk.
- **Partition growth** may be required before file system expansion.

**Instance lifecycle and EBS persistence**

The instance lifecycle affects EBS in predictable ways. Stop/start is not the same as terminate. Stop/start preserves the instance’s EBS volumes and usually changes the underlying host, while terminate ends the instance and may delete volumes depending on the mapping.

- **Stop**
  - Instance state is preserved on EBS.
  - Root and data volumes remain attached to the instance configuration.
  - Useful for maintenance and cost control.
- **Start**
  - Instance resumes from EBS-backed state.
  - Public IP may change unless you use an Elastic IP.
- **Terminate**
  - Instance is deleted.
  - Volumes marked for deletion are removed.
  - Volumes not marked for deletion remain and can be reattached.

This is why EBS is often used for persistent workloads that need recovery from instance failure. The compute layer is disposable; the storage layer is durable if you design it that way.

**Delete-on-termination behavior**

Delete-on-termination is a block device mapping setting that controls whether a volume is removed when the instance terminates. This setting is especially important for root volumes and for any data volume you want to preserve.

- **True**
  - Volume is deleted when the instance terminates.
  - Common for temporary or boot-only storage.
- **False**
  - Volume remains after termination.
  - Useful for data retention and recovery.

For exam scenarios, remember that the setting is tied to the specific block device mapping used at launch or modified later. Do not assume every volume behaves the same way.

Example CLI to inspect block device mappings:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].BlockDeviceMappings'
```

If you need to preserve a data volume, verify the mapping before termination. If you need to recover data after an instance failure, detach the retained volume and attach it to a replacement instance in the same AZ.

**Recovery patterns you should know**

EBS is often tested through practical recovery scenarios rather than isolated definitions. The most common patterns are data volume recovery and root volume recovery.

- **Recover a failed application server**
  - Stop the broken instance if possible.
  - Detach the data volume.
  - Attach it to a healthy replacement instance in the same AZ.
  - Mount the volume and validate the data.
- **Recover a root volume from snapshot**
  - Create a new volume from the snapshot.
  - Attach it to a replacement instance or use it as a replacement root volume in a launch workflow.
  - Boot the instance and verify the OS state.

Example data volume recovery on Linux:

```bash
# On the source or replacement instance, identify the disk
lsblk

# Create a mount point
sudo mkdir -p /data

# Mount the volume using the correct device or, preferably, UUID-based mapping
sudo mount /dev/nvme1n1p1 /data

# Verify
df -h /data
```

> On Nitro-based instances, NVMe device names can change across reboots or reattachments. For durable mounts, use UUIDs in `/etc/fstab` instead of relying on a fixed `/dev/nvme*` path.

Example `/etc/fstab` entry using UUID:

```bash
sudo blkid
# UUID=1234-ABCD  /data  xfs  defaults,nofail  0  2
```

**Root vs data volume behavior in practice**

Root and data volumes behave differently during launch and termination, and that difference is often the key to answering exam questions correctly.

- **Root volume**
  - Defined in the launch template or launch configuration block device mapping.
  - Used to boot the operating system.
  - Commonly recreated from an AMI or snapshot when replacing an instance.
- **Data volume**
  - Added for application data, logs, or databases.
  - Can be detached and reattached independently.
  - Often retained after termination if you want to preserve data.

A practical design pattern is to keep the OS on the root volume and store mutable application data on separate EBS volumes. That makes replacement easier because you can rebuild the instance and reattach the data disk.

**CLI examples for common EBS operations**

Inspect volumes:

```bash
aws ec2 describe-volumes \
  --filters Name=status,Values=available,in-use
```

Create a snapshot:

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-maintenance backup"
```

Create a new volume from a snapshot:

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0123456789abcdef0 \
  --volume-type gp3
```

Attach a volume:

```bash
aws ec2 attach-volume \
  --volume-id vol-0123456789abcdef0 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdf
```

Detach a volume:

```bash
aws ec2 detach-volume \
  --volume-id vol-0123456789abcdef0
```

These commands are useful for understanding the lifecycle, but in real operations you should always confirm the AZ, attachment state, and device mapping before making changes.

**Exam-oriented comparisons**

| Topic | Key point | Why it matters |
|---|---|---|
| EBS vs instance store | EBS is persistent; instance store is ephemeral | Determines whether data survives stop/terminate |
| Stop vs terminate | Stop preserves EBS; terminate may delete it | Common recovery and cost questions |
| Snapshot vs volume | Snapshot is backup; volume is attachable storage | Restore creates a new volume |
| gp2 vs gp3 | gp2 scales with size; gp3 configures size, IOPS, throughput separately | Helps choose the right default |
| Single attach vs Multi-Attach | Most volumes attach to one instance; Multi-Attach is limited | Prevents incorrect shared-storage assumptions |

**Practical exam scenarios**

- **You need to preserve logs after instance termination**
  - Put logs on a separate EBS data volume.
  - Set delete-on-termination to false.
- **You need to recover a database server after host failure**
  - Restore the data volume from snapshot or reattach the retained volume.
  - Launch a replacement instance in the same AZ.
- **You need to increase storage without downtime planning**
  - Modify the EBS volume size.
  - Expand the file system after the change.
- **You need higher performance for a database**
  - Consider io1/io2 for provisioned IOPS.
  - Use gp3 if you need a cost-effective balance with independent tuning.
- **You need to move a volume to another AZ**
  - Create a snapshot.
  - Restore a new volume in the target AZ.
  - Attach the new volume there.

**Key takeaways**

EBS is durable block storage for EC2, and the exam focuses on how it behaves across instance lifecycle events. The most important ideas are persistence, AZ scope, snapshot-based recovery, and the difference between root and data volumes.

- **Persistence** survives stop/start and can survive termination if configured correctly.
- **Snapshots** are the backup and migration mechanism, and restore creates a new volume.
- **Resize** requires both AWS-side modification and file system expansion.
- **Recovery** often means detaching and reattaching a data volume or restoring a root volume from snapshot.
- **Performance** choices often come down to gp3, gp2, or io1/io2 depending on workload needs.
