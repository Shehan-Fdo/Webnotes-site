## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**EC2 user data overview**

Amazon Elastic Compute Cloud (<abbr title="Elastic Compute Cloud">EC2</abbr>) user data is a first-boot bootstrap mechanism that runs when an instance starts for the first time. You use it to automate tasks such as package installation, service startup, configuration file creation, and simple application deployment.

User data is especially useful when you want repeatable provisioning without baking every change into a custom Amazon Machine Image (<abbr title="Amazon Machine Image">AMI</abbr>). It helps you keep launch-time configuration separate from the base image, which improves flexibility and reduces image sprawl.

- **Bootstrap** tasks run automatically during instance initialization.
- **Cloud-init** is commonly responsible for processing user data on Linux AMIs.
- **Idempotency** matters because scripts may run more than once depending on the OS and configuration.
- **Launch templates** are a common place to store reusable user data.

**What user data can do**

User data is best for lightweight, deterministic setup tasks that should happen at launch. Typical examples include installing a web server, writing a configuration file, starting a service, or pulling application code from a repository.

```bash
#!/bin/bash
yum update -y
yum install -y httpd
systemctl enable httpd
systemctl start httpd
echo "<h1>Hello from EC2 user data</h1>" > /var/www/html/index.html
```

This example installs Apache HTTP Server, enables it to start on boot, and writes a simple web page. On Amazon Linux, `yum` or `dnf` may be available depending on the AMI version.

- **Package managers** differ by distribution, so match the script to the AMI.
- **Service enablement** ensures the application survives reboots.
- **Static content** is useful for quick validation during labs and demos.
- **Logging** helps you verify whether the bootstrap completed successfully.

**How user data is delivered**

You provide user data at instance launch through the EC2 console, AWS Command Line Interface (<abbr title="AWS Command Line Interface">AWS CLI</abbr>), or infrastructure as code such as AWS CloudFormation or Terraform. The data is passed to the instance metadata/bootstrap system and executed during initialization.

```bash
aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --key-name my-key-pair \
  --security-group-ids sg-0123456789abcdef0 \
  --subnet-id subnet-0123456789abcdef0 \
  --user-data file://user-data.sh
```

- **Console** entry is convenient for labs and one-off launches.
- **CLI** is useful for repeatable automation and scripting.
- **Infrastructure as code** is preferred for version control and review.
- **Base64 encoding** is handled by many tools automatically, but some APIs require it explicitly.

**Linux user data format**

For Linux instances, user data is usually a shell script beginning with a shebang such as `#!/bin/bash`. You can also provide cloud-init directives when the AMI supports them. Keep scripts short, predictable, and safe to rerun.

```bash
#!/bin/bash
set -euxo pipefail

dnf update -y || yum update -y
dnf install -y nginx || yum install -y nginx
systemctl enable nginx
systemctl start nginx

cat >/usr/share/nginx/html/index.html <<'EOF'
<html>
  <body>
    <h1>EC2 user data worked</h1>
  </body>
</html>
EOF
```

- **`set -euxo pipefail`** improves visibility and fails fast on errors.
- **Fallback commands** help when AMIs differ across distributions.
- **Here documents** are useful for writing multi-line files.
- **Permissions** matter if your script writes to protected paths.

**Windows user data format**

For Windows instances, user data is typically a PowerShell script or a script wrapped for EC2Launch/EC2Config depending on the AMI generation. Windows user data often installs roles, configures IIS, or runs startup automation.

```powershell
<powershell>
Install-WindowsFeature -Name Web-Server
Set-Content -Path "C:\inetpub\wwwroot\index.html" -Value "<h1>Hello from Windows EC2 user data</h1>"
Start-Service W3SVC
</powershell>
```

- **PowerShell** is the standard choice for modern Windows AMIs.
- **IIS** is a common lab target because it is easy to validate in a browser.
- **Service names** may differ from display names, so verify the exact service.
- **Encoding** and line endings can affect script execution if you paste content manually.

**Verification and troubleshooting**

After launch, you verify the result by checking the application endpoint, instance logs, or bootstrap logs. If the instance has a public IP and the security group allows inbound traffic, you can test the web server from a browser or with `curl`.

```bash
curl http://<public-ip>
```

If the page does not load, inspect the instance logs and the user data execution logs. On many Linux distributions, cloud-init logs are the first place to look.

```bash
sudo tail -n 100 /var/log/cloud-init-output.log
sudo tail -n 100 /var/log/cloud-init.log
```

- **Security groups** must allow the relevant inbound port, such as `80` for HTTP.
- **Public IP addressing** is required for direct internet access unless you use a load balancer or bastion path.
- **Cloud-init logs** show script output and errors on supported Linux AMIs.
- **Instance status checks** confirm whether the VM itself is healthy, but not whether your app started correctly.

> User data is generally intended for first boot. If you edit user data after launch, the change does not automatically rerun on an already initialized instance.

**Common lab workflow**

A typical lab follows a simple sequence: choose an AMI, define networking and security, paste user data, launch the instance, and validate the result. The goal is to prove that the instance can self-configure without manual SSH or RDP intervention.

1. Select an AMI such as Amazon Linux or a Windows Server image.
2. Choose an instance type, such as `t3.micro`, for a low-cost lab.
3. Attach a security group that allows the required inbound traffic.
4. Paste a user data script that installs and starts the target service.
5. Launch the instance and wait for initialization to complete.
6. Test the service endpoint and inspect logs if needed.

**Practical design considerations**

User data works best when you keep the bootstrap small and deterministic. If the setup becomes complex, move toward configuration management tools, container orchestration, or a custom AMI built from a known-good baseline.

- **Repeatability** is the main benefit of user data.
- **Secrets** should not be hardcoded in scripts.
- **Dependencies** should be explicit so the launch does not rely on hidden state.
- **Long-running tasks** can delay boot and make troubleshooting harder.
- **External downloads** introduce availability and versioning risk.

**Security and operational notes**

Treat user data as configuration, not a secure secret store. Depending on how you launch instances, user data may be visible to users or roles with sufficient permissions to describe the instance. Avoid embedding passwords, API keys, or private certificates unless you have a secure retrieval mechanism.

A safer pattern is to fetch secrets at runtime from AWS Secrets Manager or AWS Systems Manager Parameter Store using an instance profile with least-privilege permissions.

```bash
#!/bin/bash
SECRET_VALUE=$(aws ssm get-parameter \
  --name /app/config/db-password \
  --with-decryption \
  --query 'Parameter.Value' \
  --output text)
```

- **IAM role** access is preferable to static credentials.
- **Least privilege** reduces blast radius if the instance is compromised.
- **Parameter Store** and **Secrets Manager** are better suited for sensitive values.
- **Auditability** improves when configuration is managed centrally.

**Exam focus**

For the exam, remember the core purpose of user data: automate instance initialization at launch. It is a simple, low-friction way to configure EC2 instances without logging in manually.

- **Use user data** for first-boot setup and lightweight automation.
- **Use launch templates** when you want reusable instance configuration.
- **Use cloud-init** terminology for Linux bootstrap behavior.
- **Use PowerShell** for Windows bootstrap tasks.
- **Use logs** to troubleshoot failed initialization.

**Minimal example for a web server lab**

```bash
#!/bin/bash
yum update -y
yum install -y httpd
systemctl enable httpd
systemctl start httpd
echo "EC2 user data success" > /var/www/html/index.html
```

This is enough for a basic lab because it proves the full path: launch, bootstrap, service start, and browser validation. If the page appears, your user data executed successfully and the instance is configured as intended.
