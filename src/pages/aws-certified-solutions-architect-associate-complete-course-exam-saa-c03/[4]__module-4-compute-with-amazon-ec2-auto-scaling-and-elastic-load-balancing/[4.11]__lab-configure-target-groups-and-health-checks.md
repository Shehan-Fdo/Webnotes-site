## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Target groups and health checks overview**

A target group is the routing destination for a load balancer. You register targets such as EC2 instances, IP addresses, Lambda functions, or an Application Load Balancer (ALB) behind a Network Load Balancer (NLB) in supported designs. The load balancer uses health checks to decide which targets can receive traffic.

For exam purposes, separate the concepts by load balancer type:

- **ALB** supports content-aware routing at the listener level and commonly uses HTTP or HTTPS target groups.
- **NLB** is Layer 4 and forwards traffic without host/path/header/query-based routing rules.
- **Lambda target groups** invoke functions directly and do not use instance-style port/path health checks.

A target group is not just a list of targets. It also defines protocol, port, health check behavior, deregistration delay, and optional stickiness settings. Those settings affect failover speed, traffic distribution, and how quickly unhealthy targets stop receiving requests.

**Target group types and when to use them**

Use the target group type that matches the backend you want to register.

| Target group type | Typical use | Key notes |
|---|---|---|
| `instance` | EC2 instances in a VPC | Registers by instance ID and target port |
| `ip` | IP-based backends, containers, on-prem via VPN/Direct Connect | Registers private IPs in supported networks |
| `lambda` | Serverless invocation | No port-based registration; health checks are not configured like instance/IP targets |
| `alb` | NLB forwarding to an ALB | Used in specific supported architectures only |

For ALB and NLB, the target group protocol and health check behavior depend on the target type and listener protocol. Do not assume every target group supports the same health check fields in the same way.

**ALB target groups**

ALB target groups commonly use `HTTP` or `HTTPS` health checks. The load balancer sends requests to the configured health check path and expects a success code matcher. The default success code matcher for HTTP/HTTPS health checks is `200`.

Typical ALB target group settings include:

- **Protocol**: `HTTP` or `HTTPS`
- **Port**: application port such as `80`, `443`, `8080`, or `5000`
- **Health check path**: for example `/health`, `/status`, or `/`
- **Success codes**: `200` by default for HTTP/HTTPS
- **Healthy threshold**: number of consecutive successes required
- **Unhealthy threshold**: number of consecutive failures required
- **Interval**: time between checks
- **Timeout**: how long the load balancer waits for a response

A practical example for a web app:

```bash
aws elbv2 create-target-group \
  --name app-tg \
  --protocol HTTP \
  --port 80 \
  --vpc-id vpc-0123456789abcdef0 \
  --health-check-protocol HTTP \
  --health-check-path /health \
  --health-check-interval-seconds 30 \
  --health-check-timeout-seconds 5 \
  --healthy-threshold-count 5 \
  --unhealthy-threshold-count 2 \
  --matcher HttpCode=200
```

Use a dedicated health endpoint that returns a fast, deterministic response. Avoid using a heavy application page that depends on databases or third-party services unless you intentionally want the target marked unhealthy when those dependencies fail.

**NLB target groups**

NLB target groups are Layer 4 oriented. They are commonly used with `TCP`, `TLS`, `UDP`, or `TCP_UDP` protocols. Health checks for NLB are protocol-specific and are not the same as ALB path-based checks.

Important distinctions:

- **TCP health checks** verify that the port accepts connections.
- **HTTP/HTTPS health checks** can be used in some NLB configurations, but they still follow HTTP semantics.
- **UDP** traffic has no connection handshake, so health check behavior is more limited and depends on the configured protocol support.
- **Listener rules** on NLB do not provide ALB-style host/path/header/query routing. NLB listeners typically forward based on port and protocol.

Example NLB target group creation for TCP:

```bash
aws elbv2 create-target-group \
  --name nlb-tcp-tg \
  --protocol TCP \
  --port 443 \
  --target-type instance \
  --vpc-id vpc-0123456789abcdef0 \
  --health-check-protocol TCP \
  --health-check-port traffic-port \
  --healthy-threshold-count 3 \
  --unhealthy-threshold-count 3 \
  --health-check-interval-seconds 10
```

For TCP health checks, the load balancer checks whether the target port is reachable. This is useful when the application protocol is not HTTP, such as custom TCP services, TLS pass-through, or some database front ends.

**Lambda target groups**

Lambda target groups are operationally different from instance and IP target groups.

- You register a Lambda function instead of a network endpoint.
- The load balancer invokes the function with an event payload.
- There is no target port, path, or instance-style listener binding.
- Health check configuration is not used in the same way as for network targets.

In practice, Lambda target groups are best when you want ALB to invoke serverless code directly. You should understand that the function must return a valid response in the expected format for the load balancer integration. If the function errors or times out, the request fails and the target is treated as unhealthy for routing purposes.

Example Lambda target group creation:

```bash
aws elbv2 create-target-group \
  --name lambda-tg \
  --target-type lambda \
  --vpc-id vpc-0123456789abcdef0
```

Then register the function:

```bash
aws elbv2 register-targets \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/lambda-tg/abc123def456 \
  --targets Id=arn:aws:lambda:us-east-1:123456789012:function:my-function
```

> Lambda target groups are not configured like instance or IP target groups. Do not try to apply path, port, or protocol health-check assumptions from ALB web backends.

**Creating and attaching a target group in the console**

A common lab flow is to create the target group first, then attach it to a listener rule or default action.

1. Open the **EC2** console and go to **Target Groups**.
2. Choose **Create target group**.
3. Select the target type:
   - `Instances` for EC2
   - `IP addresses` for IP-based backends
   - `Lambda function` for serverless
4. Choose the protocol and port that match the backend.
5. Configure health checks:
   - Set the protocol
   - Set the path for HTTP/HTTPS
   - Adjust interval, timeout, and thresholds if needed
6. Register targets:
   - Select instances or enter IPs
   - For Lambda, select the function
7. Create or update the load balancer listener:
   - Add a forward action to the target group
   - Or create a rule that forwards matching requests to it

For ALB, you usually attach the target group to a listener rule based on host or path conditions. For NLB, you typically attach the target group directly to the listener because content-based routing is not the primary model.

**AWS CLI workflow for EC2 targets**

A typical EC2 workflow includes creating the target group, registering instances, and then associating the group with a listener.

```bash
aws elbv2 create-target-group \
  --name web-tg \
  --protocol HTTP \
  --port 80 \
  --target-type instance \
  --vpc-id vpc-0123456789abcdef0 \
  --health-check-path /health \
  --matcher HttpCode=200
```

Register targets:

```bash
aws elbv2 register-targets \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/web-tg/abc123def456 \
  --targets Id=i-0123456789abcdef0 Id=i-0fedcba9876543210
```

Attach to an ALB listener:

```bash
aws elbv2 create-listener \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/app/my-alb/abc123 \
  --protocol HTTP \
  --port 80 \
  --default-actions Type=forward,TargetGroupArn=arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/web-tg/abc123def456
```

**AWS CLI workflow for NLB targets**

For NLB, the listener usually forwards traffic on a port to the target group. A simple TCP example looks like this:

```bash
aws elbv2 create-target-group \
  --name api-nlb-tg \
  --protocol TCP \
  --port 443 \
  --target-type instance \
  --vpc-id vpc-0123456789abcdef0 \
  --health-check-protocol TCP \
  --health-check-port traffic-port
```

Register targets:

```bash
aws elbv2 register-targets \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/api-nlb-tg/abc123def456 \
  --targets Id=i-0123456789abcdef0
```

Create the listener:

```bash
aws elbv2 create-listener \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/net/my-nlb/abc123 \
  --protocol TCP \
  --port 443 \
  --default-actions Type=forward,TargetGroupArn=arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/api-nlb-tg/abc123def456
```

If you need TLS termination at the NLB, use a `TLS` listener and a compatible target group protocol. If you need end-to-end encryption, ensure the backend service also speaks the expected protocol.

**Health check tuning**

Health checks are a tradeoff between fast failure detection and avoiding false positives. Short intervals and low thresholds detect failures quickly, but they can mark targets unhealthy during brief application pauses or deployment restarts.

Common defaults and practical guidance:

- **Interval**: often `30` seconds for ALB; shorter values increase sensitivity
- **Timeout**: usually less than the interval; keep it below the interval
- **Healthy threshold**: often `5` for ALB; lower values recover faster
- **Unhealthy threshold**: often `2` for ALB; lower values fail over faster
- **Matcher**: `200` for standard HTTP health endpoints

Change these values when:

- Your application starts slowly and needs more time before it can answer health checks.
- You want faster failover for critical services.
- Your backend performs expensive startup work and would otherwise flap between healthy and unhealthy.
- You deploy frequently and need to avoid marking targets unhealthy during rolling updates.

Example of editing an existing target group health check:

```bash
aws elbv2 modify-target-group \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/web-tg/abc123def456 \
  --health-check-path /ready \
  --health-check-interval-seconds 15 \
  --health-check-timeout-seconds 5 \
  --healthy-threshold-count 3 \
  --unhealthy-threshold-count 2 \
  --matcher HttpCode=200
```

After changing health checks, you may need to wait for the next probe cycle before the target state updates. If the target remains unhealthy, verify the application endpoint directly from inside the VPC.

**Stickiness and session behavior**

Stickiness keeps a client routed to the same target for a period of time. This can help with session affinity for stateful applications, but it is not always appropriate.

Key points:

- **ALB** supports application-based stickiness for some target group types.
- **NLB** supports source IP stickiness in supported configurations.
- **Lambda** target groups do not use the same stickiness model as instance/IP targets.
- Stickiness can hide load imbalance during testing, so disable it unless the application needs affinity.

Use stickiness only when the application cannot externalize session state to a shared store such as ElastiCache, DynamoDB, or a database.

**Common troubleshooting causes**

When a target shows as unhealthy, check the full path from the load balancer to the application.

- **Wrong target port**: the target group points to a port where nothing is listening.
- **Incorrect health check path**: the endpoint returns `404`, `500`, or redirects unexpectedly.
- **Security group rules**: the instance security group does not allow traffic from the load balancer.
- **Network ACLs**: subnet-level rules block the health check traffic.
- **Wrong VPC or subnet placement**: the target is not reachable from the load balancer.
- **Application not bound to the interface**: the service listens only on `localhost` instead of the instance IP.
- **Instance not registered**: the target group has no registered targets or the wrong target ID.
- **TLS mismatch**: the health check protocol does not match the backend listener.
- **Dependency failure**: the health endpoint depends on a database or cache that is unavailable.
- **Startup delay**: the application is still booting when the first health checks run.

A useful validation sequence is:

1. Confirm the target is registered and in the correct target group.
2. Test the health endpoint from the instance itself.
3. Test connectivity from another instance in the same VPC.
4. Check security groups, NACLs, and route tables.
5. Review application logs for health check requests and failures.

**Console-based edit and re-register workflow**

In a lab, you may need to fix a target group after initial creation.

1. Open the target group in the console.
2. Choose **Health checks** and edit the path, protocol, or thresholds.
3. Save the changes.
4. Go to **Targets** and deregister the unhealthy target if needed.
5. Re-register the target after correcting the backend configuration.
6. Wait for the health status to change to healthy.

This workflow is useful when you correct a backend port, deploy a new health endpoint, or update a security group rule.

**Exam-focused distinctions**

Use these distinctions to avoid common exam traps:

- **ALB** is best for HTTP/HTTPS applications that need host-based or path-based routing.
- **NLB** is best for low-latency, high-throughput, static IP, or non-HTTP workloads.
- **Lambda target groups** are for direct function invocation behind ALB.
- **Health checks** determine routing eligibility, not application correctness in a full business sense.
- **Target group protocol** must match the backend behavior, especially for TLS and TCP services.
- **NLB forwarding** is not the same as ALB content-based routing.

**Quick reference**

- Use `instance` targets for EC2 workloads in a VPC.
- Use `ip` targets for IP-based backends, including some container and hybrid patterns.
- Use `lambda` targets for serverless invocation behind ALB.
- Use `alb` targets only when an NLB must forward to an ALB in a supported architecture.
- Use HTTP health checks with a dedicated endpoint for web apps.
- Use TCP health checks when you only need port reachability.

**Practical lab checklist**

Before you finish the lab, confirm the following:

- The target group type matches the backend.
- The protocol and port match the application listener.
- The health check path returns the expected success code.
- Security groups allow traffic from the load balancer.
- The target is in the correct VPC and subnet context.
- The listener forwards to the correct target group.
- The target status changes to healthy after the configured thresholds are met.
