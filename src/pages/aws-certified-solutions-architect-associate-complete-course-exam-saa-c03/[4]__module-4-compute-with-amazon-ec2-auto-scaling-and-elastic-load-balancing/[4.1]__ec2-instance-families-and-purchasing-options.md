## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**EC2 instance family overview**

Amazon EC2 <abbr title="Elastic Compute Cloud">EC2</abbr> instance families group instance types by workload profile. For the exam, you usually choose based on CPU, memory, storage, networking, architecture, and pricing model rather than by raw instance size alone.

A full instance type name has a predictable pattern:

- `family` identifies the workload class, such as `m`, `c`, `r`, `t`, `i`, `g`, or `p`.
- `generation` is the number after the family, such as `m7`, `c7`, or `r7`.
- `size` is the suffix, such as `large`, `xlarge`, or `2xlarge`.
- Optional suffixes can indicate capabilities, such as `g` for Graviton-based ARM architecture, `a` for AMD, `n` for higher networking, `d` for local NVMe storage in some families, or `i` for Intel in some families.

Examples:

- `m7g.large` = general purpose, seventh generation, Graviton-based, small size.
- `c7i.2xlarge` = compute optimized, seventh generation, Intel-based, larger size.
- `r7a.xlarge` = memory optimized, seventh generation, AMD-based.
- `i4i.large` = storage optimized with local NVMe SSD emphasis.

> Be careful with suffixes: their meaning is family-specific. Do not assume every suffix means the same thing across all instance families.

**How to read the family string in practice**

Use the family to infer the workload, then confirm architecture compatibility and storage needs.

- **Architecture matters**: ARM-based instances require ARM-compatible AMIs and software.
- **Size matters**: larger sizes increase vCPU, memory, and network throughput.
- **Generation matters**: newer generations usually improve price/performance, but older generations may still appear in legacy environments or exam distractors.
- **Suffix matters**: a suffix can change the processor vendor, network profile, or storage behavior.

A quick mental model:

```text
family -> workload fit
generation -> how current the platform is
suffix -> special capability or processor variant
size -> capacity
```

**Common EC2 instance families**

The exam often tests whether you can match a workload to the right family.

| Family | Primary use | Typical exam clue |
|---|---|---|
| `t` | Burstable general purpose | Low baseline usage with occasional spikes |
| `m` | General purpose | Balanced compute, memory, and networking |
| `c` | Compute optimized | Batch processing, web servers, CPU-heavy apps |
| `r` | Memory optimized | In-memory databases, caching, analytics |
| `x` | High memory | Large in-memory workloads, SAP-like scenarios |
| `z` | High CPU and memory balance | High single-thread performance, licensing-sensitive workloads |
| `p` | Accelerated computing | Machine learning training, GPU workloads |
| `g` | Graphics / GPU acceleration | Graphics rendering, video encoding, ML inference |
| `inf` | Inferentia-based ML inference | Cost-efficient inference at scale |
| `trn` | Trainium-based ML training | Specialized training workloads |
| `i` | Storage optimized | High IOPS, low-latency local NVMe storage |
| `d` | Dense storage in some families | Large local storage capacity, family-dependent |
| `h` | High disk throughput in some families | Storage-heavy workloads, family-dependent |
| `u` | High memory / SAP HANA class in some families | Very large memory footprints, specialized enterprise workloads |

> The `d`, `h`, and `u` letters are not universal categories by themselves. Their meaning depends on the family and generation, so use them as clues only after checking the specific instance type.

**Burstable instances: `t` family**

`T` instances are designed for workloads that spend most of their time at low to moderate CPU usage but occasionally need bursts.

Use them when:

- Your application is idle most of the time.
- You run dev/test, small web apps, or low-traffic services.
- You want lower cost than always-on compute-heavy families.

Key concepts:

- **CPU credits** accumulate when CPU usage is below baseline.
- Credits are spent when the instance bursts above baseline.
- `T` instances are a poor fit for sustained CPU-heavy workloads because they can run out of credits or become expensive in unlimited mode.

Practical example:

- A small API server with occasional traffic spikes fits `t3` or `t4g`.
- A continuously busy video transcoding job does not.

**General purpose: `m` family**

`M` instances balance compute, memory, and networking. They are the default exam answer when no single resource dominates.

Use them for:

- Application servers
- Small-to-medium databases
- Enterprise workloads with mixed resource usage
- General web tiers

Why they matter:

- They are often the safest “middle ground” choice.
- They reduce the risk of over-optimizing for one resource and under-provisioning another.

**Compute optimized: `c` family**

`C` instances favor vCPU performance over memory.

Use them for:

- High-performance web servers
- Batch processing
- Scientific modeling
- Ad serving
- CPU-bound microservices

Exam clue:

- If the workload is “compute intensive” or “CPU bound,” `c` is usually the best first choice.

**Memory optimized: `r`, `x`, `z`, and related families**

Memory optimized instances provide more RAM per vCPU.

Use them for:

- In-memory databases
- Real-time analytics
- Caching layers
- Large JVM applications
- SAP HANA and similar enterprise systems

Family distinctions:

- `r` is the common memory-optimized choice.
- `x` is for very large memory footprints.
- `z` can appear in scenarios needing a balance of high CPU and memory with strong single-thread performance.
- `u` may appear in specialized high-memory enterprise scenarios, especially SAP-related questions.

**Storage optimized: `i`, `d`, `h`, and related families**

Storage optimized instances are designed for workloads that need high IOPS, low latency, or large local storage.

Use them for:

- NoSQL databases
- Distributed file systems
- Data warehousing components
- Log processing
- High-performance local scratch space

Important nuance:

- `i` families commonly emphasize local NVMe SSD performance and high random I/O.
- `d` and `h` can indicate storage-heavy designs in some families, but their exact meaning varies.
- Do not memorize them as universal labels; instead, read the full instance type and family documentation in context.

**Accelerated computing: `p`, `g`, `inf`, and `trn`**

These families add hardware acceleration for specialized workloads.

Use them when:

- You need GPU acceleration for graphics or ML.
- You need inference acceleration at scale.
- You need specialized training hardware.

Typical exam mappings:

- `p` for GPU-heavy machine learning training and HPC-style acceleration.
- `g` for graphics, visualization, and inference workloads that benefit from GPUs.
- `inf` for cost-efficient ML inference.
- `trn` for ML training on AWS-designed accelerators.

**Architecture and AMI compatibility**

Instance architecture affects what software you can run.

- **x86_64** instances run standard Intel/AMD-compatible AMIs.
- **ARM64** instances require ARM-compatible AMIs and binaries.
- Managed services and modern container images often support both, but legacy software may not.

Practical check:

```bash
aws ec2 describe-instance-types \
  --instance-types m7g.large c7i.large r7a.large \
  --query 'InstanceTypes[].{Type:InstanceType,VCpu:VCpuInfo.DefaultVCpus,MemoryMiB:MemoryInfo.SizeInMiB,Arch:ProcessorInfo.SupportedArchitectures}' \
  --output table
```

This kind of query helps you compare architecture and sizing before you launch.

**Purchasing options: the exam decision tree**

EC2 purchasing options are often tested as a cost-versus-flexibility-versus-capacity question.

A simple decision flow:

```text
Need guaranteed capacity?
  -> Capacity Reservation or zonal Reserved Instance

Need lowest price for flexible workloads?
  -> Spot Instances

Need predictable steady-state discount?
  -> Savings Plans or Reserved Instances

Need dedicated physical server for licensing/compliance?
  -> Dedicated Host

Need isolated tenancy without host control?
  -> Dedicated Instance

Need on-demand flexibility?
  -> On-Demand
```

**On-Demand Instances**

On-Demand is the default pay-as-you-go model.

Use it when:

- You need short-term or unpredictable workloads.
- You are prototyping or testing.
- You want no long-term commitment.

Pros:

- No upfront commitment
- Immediate launch
- Best flexibility

Cons:

- Highest per-hour cost compared with commitment-based options

**Reserved Instances: discount model with capacity nuance**

Reserved Instances <abbr title="Reserved Instances">RIs</abbr> provide a billing discount for steady-state usage. They are not the same as a pure capacity guarantee in every case.

Key exam distinctions:

- **Standard RIs** offer the deepest discount for steady-state workloads.
- **Convertible RIs** trade some discount for flexibility to change attributes later.
- **Regional RIs** primarily provide billing discounts and can apply across Availability Zones in a Region.
- **Zonal RIs** can provide both a billing discount and capacity reservation in a specific Availability Zone.
- **Payment options** include:
  - All Upfront
  - Partial Upfront
  - No Upfront

When to choose RIs:

- You have predictable, steady usage for one or three years.
- You want a discount for a known baseline.
- You may need zonal capacity for a specific AZ.

When not to choose RIs:

- Your workload is highly variable.
- You expect major architecture changes.
- You need the ability to scale down frequently.

Practical scenario:

- A production database that runs 24/7 for years may fit a Standard RI.
- A workload that may move from `m6i` to `m7i` or to ARM later may fit a Convertible RI.

**Savings Plans: billing discount, not capacity reservation**

Savings Plans are a commitment to spend a certain amount per hour for a term, in exchange for lower prices.

Key points:

- They are a **billing discount model**.
- They do **not** reserve capacity.
- They are often simpler than RIs for broad usage patterns.

Types:

- **Compute Savings Plans**
  - Most flexible
  - Apply across EC2, AWS Fargate, and AWS Lambda
  - Can change instance family, size, OS, tenancy, and Region
- **EC2 Instance Savings Plans**
  - Less flexible than Compute Savings Plans
  - Offer deeper discounts for a specific instance family in a Region

When to choose Savings Plans:

- You want predictable savings but expect instance changes.
- You run mixed compute services.
- You want less management overhead than RIs.

Exam contrast:

- Choose **Savings Plans** for discount flexibility.
- Choose **Capacity Reservation** when you need guaranteed EC2 capacity.
- Choose **RIs** when you want discounting and, in zonal form, possible capacity reservation.

**Capacity Reservations**

Capacity Reservations guarantee that EC2 capacity is available in a specific Availability Zone.

Use them when:

- You must launch instances in a specific AZ.
- You need capacity for a planned event, migration, or disaster recovery test.
- You want to ensure launch success during constrained capacity periods.

Important distinctions:

- Capacity Reservations are about **availability of capacity**, not discounting.
- They can be created on-demand and later targeted by instances.
- They are useful when “launch must succeed” is the key requirement.

Example CLI:

```bash
aws ec2 create-capacity-reservation \
  --instance-type m7i.large \
  --availability-zone us-east-1a \
  --instance-platform Linux/UNIX \
  --instance-count 2
```

You can then launch instances into that reservation or target it with placement and reservation settings.

**Spot Instances**

Spot Instances let you use spare EC2 capacity at a steep discount.

Use them for:

- Fault-tolerant batch jobs
- Stateless containers
- Big data processing
- CI/CD runners
- Distributed workloads that can retry

Core behavior:

- AWS can interrupt Spot Instances when capacity is needed elsewhere.
- You receive a short interruption notice before termination, hibernation, or stop, depending on support and configuration.
- Spot is ideal when interruption is acceptable and cost matters more than continuity.

Important exam details:

- Spot is not suitable for critical single-instance stateful workloads unless you design for interruption.
- Spot interruption handling depends on the instance type, platform, and supported features.
- Spot hibernation is not universal; it is only available for specific supported configurations.
- Spot Blocks are no longer available, so do not rely on them in exam answers.

Practical pattern with Auto Scaling:

- Use a mixed instances policy with a base On-Demand capacity and additional Spot capacity.
- Diversify across instance types and Availability Zones to reduce interruption risk.

Example Auto Scaling mixed instances policy snippet:

```json
{
  "LaunchTemplate": {
    "LaunchTemplateSpecification": {
      "LaunchTemplateId": "lt-0123456789abcdef0",
      "Version": "$Latest"
    },
    "Overrides": [
      { "InstanceType": "m7i.large" },
      { "InstanceType": "m7g.large" },
      { "InstanceType": "m6i.large" }
    ]
  },
  "InstancesDistribution": {
    "OnDemandBaseCapacity": 2,
    "OnDemandPercentageAboveBaseCapacity": 30,
    "SpotAllocationStrategy": "capacity-optimized"
  }
}
```

Why this matters:

- `capacity-optimized` helps AWS choose pools with more available Spot capacity.
- A small On-Demand base protects minimum service availability.
- Multiple instance types improve resilience.

Standalone Spot versus Auto Scaling Spot:

- Standalone Spot instances require your application to handle interruption directly.
- Auto Scaling can replace interrupted instances automatically, which is usually the safer exam answer for scalable services.

**Dedicated Instances and Dedicated Hosts**

These options address tenancy and compliance concerns.

**Dedicated Instances**

- Run on hardware dedicated to a single customer.
- You do not control the underlying host.
- Useful when you need isolation but not host-level control.

**Dedicated Hosts**

- Give you visibility and control over the physical server.
- Useful for software licensing that is tied to sockets, cores, or physical servers.
- Often the best choice for BYOL <abbr title="Bring Your Own License">BYOL</abbr> scenarios with strict vendor licensing rules.

When to choose which:

- Choose **Dedicated Instances** for isolation without host management.
- Choose **Dedicated Hosts** when licensing or compliance requires host-level control and placement visibility.

Exam clue:

- If the question mentions Oracle, Microsoft, or other vendor licensing tied to physical cores or sockets, consider Dedicated Hosts first.
- If the question only says “dedicated hardware” or “single-tenant isolation,” Dedicated Instances may be enough.

**Tenancy settings in launch templates**

You can set tenancy at launch time or in a launch template when the workload requires it.

Example launch template data:

```json
{
  "ImageId": "ami-0123456789abcdef0",
  "InstanceType": "m7i.large",
  "Placement": {
    "Tenancy": "dedicated"
  },
  "NetworkInterfaces": [
    {
      "DeviceIndex": 0,
      "AssociatePublicIpAddress": true,
      "SubnetId": "subnet-0123456789abcdef0",
      "Groups": ["sg-0123456789abcdef0"]
    }
  ]
}
```

Use this when:

- You need consistent tenancy across launches.
- You want Auto Scaling groups to inherit the correct placement behavior.

**Practical selection scenarios**

1. A startup runs a small API with occasional traffic spikes.
   - Choose `t` or `m` depending on baseline load.
   - Use On-Demand if traffic is uncertain.
   - Consider Savings Plans once usage becomes predictable.

2. A nightly ETL job processes large datasets and can retry on failure.
   - Choose `c` or `i` depending on whether CPU or local I/O dominates.
   - Use Spot Instances for cost savings.
   - Use Auto Scaling or job orchestration to handle interruption.

3. A production database must run continuously and needs guaranteed capacity in one AZ.
   - Choose `r` or `x` for memory needs.
   - Use a zonal Capacity Reservation if launch certainty matters.
   - Consider a Standard RI or Savings Plan for cost reduction.

4. A vendor-licensed enterprise app requires physical server control.
   - Choose Dedicated Hosts.
   - Confirm licensing terms before deployment.

5. A machine learning inference service needs GPU acceleration but not training hardware.
   - Choose `g` or `inf` depending on the model and cost profile.
   - Use Savings Plans if usage is steady enough.

**CLI examples for inspection and planning**

Describe instance types:

```bash
aws ec2 describe-instance-types \
  --instance-types c7i.large m7g.large r7a.large i4i.large \
  --query 'InstanceTypes[].{Type:InstanceType,VCpu:VCpuInfo.DefaultVCpus,MemoryMiB:MemoryInfo.SizeInMiB,Arch:ProcessorInfo.SupportedArchitectures,Network:NetworkInfo.NetworkPerformance}' \
  --output table
```

List offerings in a Region:

```bash
aws ec2 describe-instance-type-offerings \
  --location-type region \
  --filters Name=location,Values=us-east-1 \
  --query 'InstanceTypeOfferings[].InstanceType' \
  --output text
```

Check Spot price history:

```bash
aws ec2 describe-spot-price-history \
  --instance-types m7i.large \
  --product-descriptions "Linux/UNIX" \
  --availability-zone us-east-1a \
  --start-time 2026-04-25T00:00:00Z \
  --output table
```

These commands help you validate availability, compare families, and estimate cost before deployment.

**Exam traps and how to avoid them**

- **Savings Plans vs RIs**
  - Savings Plans are flexible billing discounts.
  - RIs can be more specific and may provide zonal capacity reservation when zonal.

- **Spot vs Capacity Reservation**
  - Spot is cheap but interruptible.
  - Capacity Reservation is for guaranteed capacity, not discounting.

- **Dedicated Instances vs Dedicated Hosts**
  - Dedicated Instances isolate hardware.
  - Dedicated Hosts expose the physical host for licensing control.

- **Instance family suffixes**
  - Do not overgeneralize suffix meanings.
  - Always interpret suffixes in the context of the specific family.

- **Architecture compatibility**
  - ARM instances need ARM-compatible AMIs and software.
  - x86 instances are not interchangeable with ARM without validation.

**Quick comparison table**

| Option | Main benefit | Main limitation | Best use case |
|---|---|---|---|
| On-Demand | Maximum flexibility | Highest cost | Unpredictable workloads |
| Reserved Instances | Strong discount | Commitment required | Steady-state EC2 usage |
| Savings Plans | Flexible discount | No capacity reservation | Broad, predictable compute spend |
| Capacity Reservation | Guaranteed capacity | No discount by itself | AZ-specific launch certainty |
| Spot Instances | Lowest cost | Interruptible | Fault-tolerant batch and stateless workloads |
| Dedicated Instances | Single-tenant isolation | Less control than hosts | Isolation without host management |
| Dedicated Hosts | Host-level control | Higher cost and management | BYOL and compliance-sensitive workloads |

**How to answer SAA-C03 questions**

When you see an EC2 scenario, ask these questions in order:

- Is the workload CPU, memory, storage, or GPU heavy?
- Is the workload steady, bursty, or interruptible?
- Does the question ask for cost savings or guaranteed capacity?
- Is licensing or tenancy a requirement?
- Does the workload require a specific architecture?

Then map the answer:

- Workload profile -> instance family
- Cost model -> purchasing option
- Capacity requirement -> reservation or Spot
- Licensing requirement -> tenancy choice

**Memory aid**

- `t` = bursty
- `m` = balanced
- `c` = compute
- `r` = RAM
- `i` = I/O
- `g` / `p` / `inf` / `trn` = acceleration
- On-Demand = flexibility
- RI / Savings Plans = discounts
- Capacity Reservation = capacity
- Spot = cheapest, interruptible
- Dedicated Host = licensing control
