## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and mental model**

This lab focuses on tracing how an EC2 instance was created and how it relates to a launch template and an Auto Scaling group (ASG). In practice, you use the AWS Command Line Interface (AWS CLI) to answer three questions:

- What is the instance running now?
- Which launch template version or configuration created it?
- Which Auto Scaling group manages replacement and scaling behavior?

The exam often tests your ability to connect these resources from different angles. An instance may be launched directly, by an ASG, or from a launch template used by an ASG. Your job is to identify the source of truth and understand which settings are inherited at launch versus which are controlled later by the ASG.

**Core resource relationships**

An EC2 instance can be launched from a launch template, but the template itself is only a blueprint. The ASG decides when to launch or terminate instances, while the launch template defines the instance configuration used at launch time.

- **EC2 instance**: The running compute resource you inspect with `describe-instances`.
- **Launch template**: The reusable launch blueprint that can define AMI, instance type, network settings, IAM role, user data, and storage.
- **Auto Scaling group**: The controller that maintains desired capacity and replaces unhealthy instances.

A useful way to think about the relationship is:

```text
Launch template -> Auto Scaling group -> EC2 instances
```

The ASG may reference a launch template version explicitly, or it may reference `$Default` or `$Latest`. That distinction matters because future launches can change when the template default changes, even if the ASG itself is not edited.

**Describe an EC2 instance**

Start by identifying the instance and its launch metadata. The most useful fields are the instance ID, state, image ID, instance type, subnet, security groups, and the `LaunchTime`. If the instance was created from a launch template, the instance metadata can also show the template ID or name and the version used.

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].{
    InstanceId: InstanceId,
    State: State.Name,
    ImageId: ImageId,
    InstanceType: InstanceType,
    SubnetId: SubnetId,
    VpcId: VpcId,
    LaunchTime: LaunchTime,
    LaunchTemplate: LaunchTemplate,
    SecurityGroups: SecurityGroups[*].GroupId
  }' \
  --output table
```

Key points:

- **LaunchTemplate** on the instance shows the template reference used at launch, when present.
- **SecurityGroups** on the instance are the effective groups attached to that instance.
- **VpcId** and **SubnetId** help you confirm the network placement.
- **ImageId** tells you which Amazon Machine Image (AMI) the instance used.

If you want a compact JSON view for scripting, use `--output json` and a narrower query:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].{InstanceId:InstanceId,LaunchTemplate:LaunchTemplate,State:State.Name}'
```

> If you query across regions or profiles, you may get an empty result even when the instance exists. Always confirm `--region` and `--profile` before assuming the resource is missing.

**Launch template inspection**

A launch template stores one or more versions. Each version can change fields such as AMI, instance type, block device mappings, user data, security groups, and IAM instance profile. The template is not the running instance; it is the launch specification.

List templates:

```bash
aws ec2 describe-launch-templates \
  --query 'LaunchTemplates[*].{Name:LaunchTemplateName,Id:LaunchTemplateId,DefaultVersion:DefaultVersionNumber,LatestVersion:LatestVersionNumber}' \
  --output table
```

Inspect a specific template:

```bash
aws ec2 describe-launch-template-versions \
  --launch-template-name my-launch-template \
  --versions 1 \
  --query 'LaunchTemplateVersions[0].LaunchTemplateData'
```

You can also inspect multiple versions by repeating the parameter:

```bash
aws ec2 describe-launch-template-versions \
  --launch-template-name my-launch-template \
  --versions 1 \
  --versions 2 \
  --versions 3 \
  --query 'LaunchTemplateVersions[*].{Version:VersionNumber,ImageId:LaunchTemplateData.ImageId,InstanceType:LaunchTemplateData.InstanceType}'
```

Important launch template fields:

- **ImageId**: The AMI used when instances launch from this version.
- **InstanceType**: The default instance size, unless overridden by the ASG or launch request.
- **IamInstanceProfile**: The IAM role attached to the instance.
- **UserData**: Startup script, usually base64-encoded in the API response.
- **NetworkInterfaces**: Network configuration, including security groups in some template shapes.
- **SecurityGroupIds** or **SecurityGroupNames**: The exact field depends on how the template is defined and the API shape.

A practical example of user data retrieval:

```bash
aws ec2 describe-launch-template-versions \
  --launch-template-name my-launch-template \
  --versions 1 \
  --query 'LaunchTemplateVersions[0].LaunchTemplateData.UserData' \
  --output text
```

**Auto Scaling group inspection**

An ASG can reference either a launch template or a launch configuration. In exam scenarios, you should check which one is populated rather than assuming one field is always absent from raw output.

```bash
aws autoscaling describe-auto-scaling-groups \
  --auto-scaling-group-names my-asg \
  --query 'AutoScalingGroups[0].{
    AutoScalingGroupName:AutoScalingGroupName,
    MinSize:MinSize,
    MaxSize:MaxSize,
    DesiredCapacity:DesiredCapacity,
    LaunchTemplate:LaunchTemplate,
    LaunchConfigurationName:LaunchConfigurationName,
    VPCZoneIdentifier:VPCZoneIdentifier
  }' \
  --output table
```

What to look for:

- **LaunchTemplate**: Indicates the ASG uses a launch template.
- **LaunchConfigurationName**: Indicates the ASG uses a launch configuration instead.
- **VPCZoneIdentifier**: Lists the subnets used by the ASG.
- **DesiredCapacity**: The current target number of instances.
- **MinSize** and **MaxSize**: The scaling boundaries.

If the ASG uses a launch template, inspect the version reference carefully:

- `Version: "$Default"` means the ASG follows the template’s default version.
- `Version: "$Latest"` means the ASG follows the latest template version.
- `Version: "3"` means the ASG is pinned to version 3.

That distinction is critical. If the ASG is pinned to a fixed version, future launches keep using that version until you update the ASG. If the ASG uses `$Default` or `$Latest`, changing the template’s default or creating a newer version can affect future launches without changing the ASG definition.

**End-to-end example**

Suppose you have:

- Instance ID: `i-0123456789abcdef0`
- ASG name: `web-asg`
- Launch template: `web-lt`
- Launch template version: `3`

A realistic workflow is:

1. Inspect the instance.
2. Confirm the launch template reference on the instance.
3. Inspect the ASG to see whether it uses the same template and version.
4. Compare the template version with the instance launch time and current ASG settings.

Example instance output shape:

```json
{
  "InstanceId": "i-0123456789abcdef0",
  "State": "running",
  "ImageId": "ami-0abc1234def567890",
  "InstanceType": "t3.micro",
  "LaunchTemplate": {
    "LaunchTemplateId": "lt-0a1b2c3d4e5f67890",
    "LaunchTemplateName": "web-lt",
    "Version": "3"
  }
}
```

Example ASG output shape:

```json
{
  "AutoScalingGroupName": "web-asg",
  "DesiredCapacity": 2,
  "MinSize": 2,
  "MaxSize": 4,
  "LaunchTemplate": {
    "LaunchTemplateId": "lt-0a1b2c3d4e5f67890",
    "LaunchTemplateName": "web-lt",
    "Version": "$Default"
  }
}
```

Interpretation:

- The instance was launched from version `3`.
- The ASG currently follows `$Default`.
- If the default version changes to `4`, new instances launched by the ASG may use version `4`, while the existing instance remains on version `3`.

**Common troubleshooting scenarios**

A few realistic problems come up often when you inspect instances and scaling groups.

- **Instance not matching expected AMI**
  - Check the launch template version used by the instance.
  - Compare it with the ASG’s current template version reference.
  - Verify whether the ASG uses `$Default`, `$Latest`, or a fixed version.

- **Security group confusion**
  - Confirm whether the instance has the expected effective security groups.
  - Check the launch template for network interface settings or security group references.
  - Remember that the template may define security groups differently depending on whether it uses network interfaces or direct instance-level settings.

- **ASG launches the wrong size or count**
  - Inspect `DesiredCapacity`, `MinSize`, and `MaxSize`.
  - Check scaling policies and recent scaling activities.
  - Confirm whether the ASG is healthy enough to maintain desired capacity.

- **Instances appear outside the ASG**
  - An instance launched manually is not automatically part of the ASG.
  - If you want it managed by the ASG, it must be explicitly attached or launched by the ASG workflow.
  - Do not assume every instance in the same subnet belongs to the group.

**Scaling activity and event history**

When an ASG behaves unexpectedly, scaling activities help you see what happened and why. Use them to confirm launches, terminations, health checks, and replacement events.

```bash
aws autoscaling describe-scaling-activities \
  --auto-scaling-group-name my-asg \
  --max-items 10 \
  --query 'Activities[*].{Time:StartTime,Status:StatusCode,Description:Description,Cause:Cause}' \
  --output table
```

Notes on this command:

- **`--auto-scaling-group-name`** scopes the activity history to one ASG.
- **`--max-items`** is a paginator control and may not equal the service-side maximum returned in one call.
- **Cause** often explains whether the event came from a policy, a scheduled action, a health check, or a manual change.

Example activity descriptions you might see:

- `Launching a new EC2 instance: i-...`
- `Terminating EC2 instance: i-...`
- `At 2026-04-26T12:00:00Z a user request updated desired capacity`
- `Instance became unhealthy in ELB`

**Practical CLI pitfalls**

Small CLI mistakes can make a correct resource look missing.

- **Region mismatch**
  - Use `--region us-east-1` or set `AWS_REGION`.
  - A template or ASG in one region does not appear in another.

- **Profile mismatch**
  - Use `--profile dev` when your credentials are split across accounts.
  - Confirm the account with `aws sts get-caller-identity`.

- **Query mistakes**
  - A bad JMESPath query can return `null` or an empty list even when the resource exists.
  - Test the raw output first, then refine the query.

- **Pagination confusion**
  - `--max-items` controls CLI pagination behavior, not necessarily the service’s internal page size.
  - For a single resource, prefer direct identifiers and a focused query.

- **Shell quoting**
  - Quote JMESPath expressions in single quotes on Unix-like shells.
  - On Windows PowerShell, quoting rules differ and may require adjustment.

A quick identity check before troubleshooting:

```bash
aws sts get-caller-identity
aws configure list
```

**Exam-safe interpretation rules**

Use these rules when reading outputs under time pressure:

- **Instance output** tells you what is running now.
- **Launch template output** tells you what the blueprint contains.
- **ASG output** tells you what will be used for future launches.
- **Version pinning** determines whether future launches stay stable or follow template changes.
- **Existing instances** do not automatically change when the template changes; they keep their current configuration unless replaced.

**Hands-on comparison checklist**

Use this checklist to compare the three layers quickly:

1. Confirm the instance ID and state with `describe-instances`.
2. Check the instance’s `LaunchTemplate` reference and version.
3. Inspect the launch template version data with `describe-launch-template-versions`.
4. Inspect the ASG with `describe-auto-scaling-groups`.
5. Verify whether the ASG uses a fixed version, `$Default`, or `$Latest`.
6. Review scaling activities if the instance count or replacement behavior is unexpected.

**Quick reference commands**

```bash
aws ec2 describe-instances --instance-ids i-0123456789abcdef0
aws ec2 describe-launch-templates
aws ec2 describe-launch-template-versions --launch-template-name my-launch-template --versions 1
aws autoscaling describe-auto-scaling-groups --auto-scaling-group-names my-asg
aws autoscaling describe-scaling-activities --auto-scaling-group-name my-asg
aws sts get-caller-identity
```

**Key takeaways**

- **Launch templates** define how instances should be launched.
- **Auto Scaling groups** decide when instances are launched or terminated.
- **EC2 instance metadata** helps you trace the exact launch source.
- **Version selection** is the most important detail when diagnosing drift between expected and actual instance configuration.
- **CLI accuracy** matters: region, profile, quoting, and query syntax can all affect what you see.
