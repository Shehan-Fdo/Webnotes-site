## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Health checks: what they do and why they matter**

Health checks are the mechanism AWS services use to decide whether a target, instance, or endpoint should receive traffic. In practice, they are the first line of defense against sending requests to unhealthy resources and the trigger for automated recovery actions.

A health check usually answers one question: “Should this resource still be considered available?” The answer can come from:
- **Network reachability** such as TCP connectivity.
- **Application response** such as HTTP status codes.
- **Service-specific signals** such as instance status checks or target group health.

In exam scenarios, always separate two ideas:
- **Detection**: identifying that something is unhealthy.
- **Replacement behavior**: deciding what AWS does after the failure is detected.

**Core AWS services that use health checks**

Several AWS services evaluate health, but they do not all react the same way.

- **Elastic Load Balancing (ELB)** routes traffic away from unhealthy targets.
- **Auto Scaling** can terminate and replace unhealthy EC2 instances.
- **Route 53** can stop returning unhealthy endpoints in DNS answers.
- **EC2 status checks** detect infrastructure or instance-level problems.
- **Amazon ECS / EKS** and other orchestrators may restart or reschedule workloads, but the exact behavior depends on the service and deployment model.

A useful mental model is:
- Load balancers protect request routing.
- Auto Scaling protects fleet size and instance replacement.
- DNS health checks protect endpoint selection.
- Instance status checks protect underlying compute health.

**Elastic Load Balancing health checks**

ELB health checks determine whether a target should stay in rotation. If a target fails health checks, the load balancer stops sending it new requests.

For Application Load Balancer (ALB) and Network Load Balancer (NLB) target groups, health checks are configured at the target group level. The load balancer probes each target on a schedule and evaluates the response.

Typical configuration elements include:
- **Protocol**: HTTP, HTTPS, TCP, TLS, or gRPC depending on load balancer type and target group support.
- **Port**: the port used for the health check.
- **Path**: for HTTP/HTTPS checks, the request path such as `/health`.
- **Matcher**: acceptable response codes, often `200-399` for HTTP checks.
- **Interval**: how often checks run.
- **Timeout**: how long the check waits for a response.
- **Healthy threshold**: number of consecutive successes needed to mark healthy.
- **Unhealthy threshold**: number of consecutive failures needed to mark unhealthy.

A practical ALB example:

```bash
aws elbv2 create-target-group \
  --name app-tg \
  --protocol HTTP \
  --port 80 \
  --vpc-id vpc-1234567890abcdef0 \
  --health-check-protocol HTTP \
  --health-check-path /health \
  --health-check-interval-seconds 30 \
  --health-check-timeout-seconds 5 \
  --healthy-threshold-count 5 \
  --unhealthy-threshold-count 2 \
  --matcher HttpCode=200-399
```

A simple health endpoint in an application might look like this:

```python
from flask import Flask, jsonify

app = Flask(__name__)

@app.route("/health")
def health():
    return jsonify(status="ok"), 200
```

Important behavior:
- An unhealthy target is removed from rotation, but it is not automatically terminated by the load balancer.
- Existing connections may continue briefly depending on the load balancer type and connection handling.
- Health checks should test the app path that reflects real readiness, not just whether the process is running.

> A target can be “alive” at the OS level and still be unhealthy from the load balancer’s perspective if the application returns errors or times out.

**Auto Scaling health checks and replacement behavior**

Auto Scaling uses health checks to decide whether to keep or replace EC2 instances in a group. This is the main AWS feature that performs automatic instance replacement.

Auto Scaling groups can use:
- **EC2 status checks**: detect instance or system impairment.
- **Elastic Load Balancing health checks**: detect whether the instance is healthy behind a load balancer.
- **Custom health checks**: you can mark an instance unhealthy using the API.

When an instance is marked unhealthy, Auto Scaling typically:
1. Terminates the unhealthy instance.
2. Launches a replacement instance to maintain desired capacity.
3. Waits for the new instance to pass health checks before fully serving traffic.

Example: mark an instance unhealthy manually.

```bash
aws autoscaling set-instance-health \
  --instance-id i-0123456789abcdef0 \
  --health-status Unhealthy \
  --should-respect-grace-period
```

Key concepts:
- **Grace period**: a startup window during which Auto Scaling ignores health check failures so bootstrapping can complete.
- **Desired capacity**: the target number of instances Auto Scaling tries to maintain.
- **Health check type**: determines what signals Auto Scaling uses to judge health.

A common configuration pattern is:
- Use a load balancer health check for application readiness.
- Use a grace period long enough for software installation, configuration, and warm-up.
- Use lifecycle hooks if you need custom initialization before traffic is allowed.

Example Auto Scaling group creation with ELB health checks:

```bash
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name web-asg \
  --launch-template LaunchTemplateName=web-lt,Version=1 \
  --min-size 2 \
  --max-size 6 \
  --desired-capacity 2 \
  --vpc-zone-identifier subnet-aaa,subnet-bbb \
  --health-check-type ELB \
  --health-check-grace-period 300 \
  --target-group-arns arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/app-tg/abcdef1234567890
```

Replacement behavior details:
- Auto Scaling replaces instances that fail health checks, but it does not repair application code.
- If the root cause is systemic, replacement may create a loop of failing instances.
- Use launch templates, user data, and configuration management to ensure replacements come up correctly.

**EC2 status checks**

EC2 status checks are AWS-managed diagnostics that detect problems at two levels:
- **System status checks**: issues with AWS infrastructure, such as host hardware or networking.
- **Instance status checks**: issues inside the instance, such as failed boot, impaired networking, or OS-level problems.

These checks are useful because they distinguish between:
- Problems AWS can fix by moving or recovering the instance.
- Problems inside the guest OS that require your intervention.

Operationally:
- A failed system status check often indicates an AWS-side infrastructure issue.
- A failed instance status check often indicates a software, OS, or configuration issue.

You can monitor them in CloudWatch and create alarms to trigger automation.

Example CloudWatch alarm for instance status check failure:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name ec2-instance-status-check-failed \
  --metric-name StatusCheckFailed_Instance \
  --namespace AWS/EC2 \
  --statistic Maximum \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0
```

Useful recovery actions:
- Reboot the instance if the OS is hung.
- Stop and start the instance if you need a new host placement.
- Replace the instance if the application or OS is irrecoverable.

**Route 53 health checks and DNS failover**

Route 53 health checks evaluate endpoints and can be used with DNS failover routing policies. If an endpoint fails health checks, Route 53 can stop returning it in DNS responses and instead return a healthy failover target.

This is especially useful when:
- You need multi-region failover.
- You want DNS-level traffic steering without a load balancer in front.
- You need to check endpoints outside AWS or across multiple AWS regions.

Route 53 health checks can monitor:
- **HTTP/HTTPS endpoints**
- **TCP endpoints**
- **CloudWatch alarms** as a proxy for application health

A common pattern is:
- Primary record points to the main region.
- Secondary record points to the failover region.
- Health check determines whether the primary should be returned.

Important DNS behavior:
- DNS caching means failover is not instantaneous for all clients.
- TTL (time to live) affects how quickly clients pick up the new answer.
- Route 53 health checks do not “move” traffic; they only influence DNS responses.

**Replacement behavior by service**

Different AWS services respond differently when health fails.

| Service | Health check purpose | Replacement behavior |
|---|---|---|
| ELB | Detect unhealthy targets | Stops routing new requests to the target |
| Auto Scaling | Detect unhealthy instances | Terminates and replaces instances |
| EC2 | Detect system or instance impairment | No automatic replacement by itself |
| Route 53 | Detect unhealthy endpoints | Removes unhealthy answers from DNS responses |
| ECS service | Detect task failure | Replaces tasks to maintain desired count |
| EKS | Detect pod/node issues | Kubernetes controllers reschedule or replace workloads |

The exam often tests whether you know which service actually performs replacement:
- **ELB** does not replace EC2 instances.
- **Auto Scaling** does replace EC2 instances.
- **Route 53** does not repair endpoints; it only changes DNS answers.

**Designing effective health checks**

A good health check should be fast, meaningful, and low-risk.

Best practices:
- Check a dependency only if that dependency is required for serving traffic.
- Keep the endpoint lightweight and deterministic.
- Return a clear success code such as `200 OK`.
- Avoid expensive database queries unless database availability is part of readiness.
- Separate liveness from readiness when possible.

A practical distinction:
- **Liveness**: “Is the process running?”
- **Readiness**: “Can this instance safely receive traffic?”

Example of a more realistic readiness check in an app:
- Verify the app can access required config.
- Verify critical downstream dependencies are reachable.
- Return failure if the instance should be removed from rotation.

Example Nginx reverse proxy health endpoint:

```nginx
location = /health {
    access_log off;
    add_header Content-Type text/plain;
    return 200 "ok\n";
}
```

**Troubleshooting unhealthy targets**

When a target stays unhealthy, work through the problem systematically.

1. Confirm the target is registered in the correct target group.
2. Verify security groups and network ACLs allow health check traffic.
3. Check the health check path, protocol, port, and matcher.
4. Confirm the application returns the expected status code.
5. Review instance logs, load balancer logs, and CloudWatch metrics.
6. Validate startup time and increase the grace period if needed.

Common causes:
- Wrong health check path, such as `/status` instead of `/health`.
- Application returns `301` or `302` when the matcher expects only `200`.
- Security group blocks the load balancer from reaching the instance.
- The app takes longer to boot than the grace period allows.
- The endpoint depends on a database that is temporarily unavailable.

**Practical exam patterns**

Look for wording that hints at the correct service behavior.

- If the question says “remove from rotation,” think **load balancer**.
- If it says “terminate and launch a replacement instance,” think **Auto Scaling**.
- If it says “DNS should stop returning the bad endpoint,” think **Route 53**.
- If it says “AWS infrastructure or OS issue on EC2,” think **status checks**.
- If it says “keep desired capacity constant,” think **Auto Scaling replacement**.

A common architecture:
- ALB performs application health checks.
- Auto Scaling replaces failed EC2 instances.
- Route 53 provides regional failover if the entire stack is unavailable.

**Example end-to-end flow**

In a web tier behind an ALB and Auto Scaling group:
- The ALB checks `/health` on each instance.
- One instance starts returning `500`.
- The ALB stops sending new requests to that instance.
- Auto Scaling sees the instance as unhealthy through ELB health checks.
- Auto Scaling terminates the instance and launches a replacement.
- The replacement boots, passes health checks, and re-enters rotation.

This layered approach is powerful because it separates traffic management from fleet recovery.

**Key takeaways**

- **Health checks detect failure; replacement behavior is service-specific.**
- **ELB removes bad targets from rotation but does not replace instances.**
- **Auto Scaling replaces unhealthy EC2 instances to maintain capacity.**
- **Route 53 changes DNS answers; it does not repair endpoints.**
- **EC2 status checks help identify infrastructure versus guest OS problems.**
- **Grace periods and realistic readiness checks prevent premature replacement.**
