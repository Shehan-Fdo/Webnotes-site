## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab goal and architecture**

This lab builds a common AWS web tier pattern: an <abbr title="Application Load Balancer">ALB</abbr> in front of an <abbr title="Auto Scaling Group">ASG</abbr> so traffic is distributed across multiple instances and unhealthy instances can be replaced automatically. The design improves availability, supports horizontal scaling, and separates client-facing traffic from instance management.

A typical secure layout is:

- Internet-facing ALB in public subnets
- EC2 instances in private subnets
- Target group attached to the ASG
- Security groups that allow the ALB to reach the instances on the application port

> In most production designs, you place application instances in private subnets behind an internet-facing ALB. Public subnets are usually reserved for the load balancer and other internet-facing entry points.

**Core components**

- **Application Load Balancer**: Layer 7 load balancer for HTTP and HTTPS traffic.
- **Target group**: Logical destination for requests; health checks determine which targets are considered healthy.
- **Auto Scaling Group**: Maintains the desired number of EC2 instances and replaces failed instances.
- **Launch template**: Defines the instance configuration used by the ASG.
- **Security groups**: Control traffic between the ALB, instances, and the internet.

**How the traffic flow works**

The ALB receives client requests and forwards them to healthy targets in the target group. The ASG launches instances from the launch template and registers them with the target group. When an instance fails health checks or becomes impaired, the ASG can terminate and replace it, depending on the health check type and scaling configuration.

A simplified request path looks like this:

```text
Client -> Internet Gateway -> ALB -> Target Group -> EC2 Instance
```

**Why use an ALB with an ASG**

- **Availability**: Multiple instances reduce the impact of a single failure.
- **Elasticity**: The ASG adds or removes instances based on demand.
- **Health awareness**: The ALB checks application health, not just instance reachability.
- **Separation of concerns**: The ALB handles routing; the ASG handles fleet management.

**Launch template essentials**

A launch template is the preferred way to define instance settings for an ASG. It can include:

- AMI ID
- Instance type
- Key pair
- Security groups
- User data
- IAM instance profile
- Block device mappings
- Network interface settings

Example launch template creation:

```bash
aws ec2 create-launch-template \
  --launch-template-name web-lt \
  --version-description v1 \
  --launch-template-data '{
    "ImageId":"ami-0123456789abcdef0",
    "InstanceType":"t3.micro",
    "SecurityGroupIds":["sg-0123456789abcdef0"],
    "UserData":"IyEvYmluL2Jhc2gKeXVtIGluc3RhbGwgLXkgaHR0cAplY2hvICJIZWxsbyBmcm9tIEFTRyIgPiAvdmFyL3d3dy9odG1sL2luZGV4Lmh0bWwKc3lzdGVtY3RsIGVuYWJsZSAtLWh0dHBkCnN5c3RlbWN0bCByZXN0YXJ0IGh0dHBkCg=="
  }'
```

The `UserData` value is base64-encoded. In practice, you often generate it from a script file or use the console editor. Make sure the launch template exists in the same AWS Region and account where you create the ASG.

**User data and web server bootstrap**

User data runs on first boot and is commonly used to install and start a web server. The exact commands depend on the AMI family.

For Amazon Linux 2 or other RHEL-based images:

```bash
#!/bin/bash
yum update -y
yum install -y httpd
echo "Hello from $(hostname -f)" > /var/www/html/index.html
systemctl enable httpd
systemctl start httpd
```

For Ubuntu-based images, package and service names differ:

```bash
#!/bin/bash
apt-get update -y
apt-get install -y apache2
echo "Hello from $(hostname -f)" > /var/www/html/index.html
systemctl enable apache2
systemctl start apache2
```

Use user data to make instances self-configuring so the ASG can replace them without manual intervention.

**Target group configuration**

The target group defines where the ALB sends traffic and how it checks health.

Common settings:

- **Target type**: `instance` for EC2 instances
- **Protocol**: HTTP for a basic web lab
- **Port**: Usually `80`
- **Health check path**: Often `/`
- **Health check matcher**: `200` or `200-399` depending on the app

Example CLI creation:

```bash
aws elbv2 create-target-group \
  --name web-tg \
  --protocol HTTP \
  --port 80 \
  --vpc-id vpc-0123456789abcdef0 \
  --target-type instance \
  --health-check-protocol HTTP \
  --health-check-path / \
  --health-check-interval-seconds 30 \
  --health-check-timeout-seconds 5 \
  --healthy-threshold-count 2 \
  --unhealthy-threshold-count 2
```

Health checks are important because the ALB uses them to decide which targets should receive traffic. During startup, registration, or grace periods, a target may not be immediately eligible for traffic, so do not assume instant routing.

**Application Load Balancer setup**

An ALB is usually created in at least two public subnets across different Availability Zones for high availability.

Key points:

- **Scheme**: `internet-facing` for public access
- **Listeners**: HTTP on `80`, optionally HTTPS on `443`
- **Default action**: Forward to the target group
- **Subnets**: Use multiple AZs for resilience
- **Security group**: Allow inbound `80` or `443` from the internet

Example listener creation:

```bash
aws elbv2 create-load-balancer \
  --name web-alb \
  --subnets subnet-aaa111 subnet-bbb222 \
  --security-groups sg-alb1234567890 \
  --scheme internet-facing \
  --type application
```

Then create a listener that forwards to the target group:

```bash
aws elbv2 create-listener \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/app/web-alb/50dc6c495c0c9188 \
  --protocol HTTP \
  --port 80 \
  --default-actions Type=forward,TargetGroupArn=arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/web-tg/73e2d6bc24d8a067
```

**Auto Scaling Group configuration**

The ASG uses the launch template and target group to manage the instance fleet.

Important settings:

- **Desired capacity**: Number of instances you want running
- **Minimum size**: Lowest allowed instance count
- **Maximum size**: Highest allowed instance count
- **VPC subnets**: Usually private subnets for application instances
- **Target group attachment**: Registers instances automatically
- **Health check type**: `EC2`, `ELB`, or both depending on design

Example CLI creation:

```bash
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name web-asg \
  --launch-template LaunchTemplateName=web-lt,Version='$Default' \
  --min-size 2 \
  --max-size 4 \
  --desired-capacity 2 \
  --vpc-zone-identifier "subnet-ccc333,subnet-ddd444" \
  --target-group-arns arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/web-tg/73e2d6bc24d8a067 \
  --health-check-type ELB \
  --health-check-grace-period 300
```

Use `Version='$Default'` when you want the ASG to follow the launch template’s default version. You can also pin a numeric version if you need strict change control.

**Health checks and replacement behavior**

The ASG can use different health check sources:

- **EC2 health checks**: Detect underlying instance status issues.
- **ELB health checks**: Use target group health to determine whether the application is healthy.
- **Combined behavior**: In practice, you choose the health check type that matches your recovery goal.

For a web tier behind an ALB, `ELB` health checks are often preferred because they reflect application availability, not just VM status. If the app fails but the instance still runs, the ALB can mark it unhealthy and the ASG can replace it.

**Security group design**

Security groups should be tightly scoped.

Recommended pattern:

- **ALB security group**
  - Inbound: `80` and/or `443` from `0.0.0.0/0`
  - Outbound: application port to instance security group
- **Instance security group**
  - Inbound: application port from ALB security group only
  - Outbound: as required for updates, package downloads, and dependencies

Example instance security group rule concept:

```text
Type: HTTP
Port: 80
Source: sg-alb1234567890
```

This prevents direct internet access to the instances while still allowing the ALB to reach them.

**Step-by-step lab flow**

1. Create or choose a VPC with at least two Availability Zones.
2. Create public subnets for the ALB and private subnets for the ASG instances.
3. Create security groups for the ALB and the instances.
4. Create a launch template with the AMI, instance type, security group, and user data.
5. Create a target group with HTTP health checks.
6. Create an internet-facing ALB in the public subnets.
7. Create a listener that forwards traffic to the target group.
8. Create an ASG in the private subnets and attach the target group.
9. Verify that instances register as healthy.
10. Test the ALB DNS name in a browser or with `curl`.
11. Terminate an instance or fail the health check to confirm replacement behavior.

**Verification commands**

Check target health:

```bash
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/web-tg/73e2d6bc24d8a067
```

Check ASG state:

```bash
aws autoscaling describe-auto-scaling-groups \
  --auto-scaling-group-names web-asg
```

Check ALB DNS name:

```bash
aws elbv2 describe-load-balancers \
  --names web-alb
```

You can also test from a browser:

```text
http://web-alb-1234567890.us-east-1.elb.amazonaws.com
```

**Troubleshooting checklist**

- **No response from ALB**
  - Confirm the listener exists and forwards to the correct target group.
  - Confirm the ALB security group allows inbound traffic.
- **Targets unhealthy**
  - Confirm the web server is running on the expected port.
  - Confirm the health check path returns a success code.
  - Confirm the instance security group allows traffic from the ALB security group.
- **Instances not launching**
  - Confirm the launch template exists in the same Region.
  - Confirm the ASG subnets are valid and have available IP addresses.
  - Confirm the IAM role and AMI are correct.
- **Traffic not reaching instances**
  - Confirm the target group type is `instance`.
  - Confirm the ASG is attached to the target group.
  - Confirm the application listens on the port expected by the target group.

**Exam-relevant takeaways**

- An ALB is the standard choice for HTTP and HTTPS web traffic.
- An ASG provides automatic scaling and instance replacement.
- A target group connects the ALB to the ASG-managed instances.
- Launch templates are the modern way to define ASG instance settings.
- Health checks drive both routing decisions and replacement behavior.
- Private subnets for instances and public subnets for the ALB are the common secure pattern.

**Common misconceptions to avoid**

- The ALB does not make instances healthy; it only routes to targets that pass health checks.
- The ASG does not replace instances solely because traffic is low; scaling policies and health checks are separate concerns.
- User data is not universal across all AMIs; package managers and service names vary by operating system.
- An ASG can use EC2 health checks, ELB health checks, or both depending on configuration; do not assume only one model exists.

**Mental model for the lab**

Think of the ALB as the front door and the ASG as the self-healing server pool behind it. The target group is the directory that tells the ALB which instances are eligible to receive requests. If you understand that separation, you can troubleshoot most deployment issues quickly.
