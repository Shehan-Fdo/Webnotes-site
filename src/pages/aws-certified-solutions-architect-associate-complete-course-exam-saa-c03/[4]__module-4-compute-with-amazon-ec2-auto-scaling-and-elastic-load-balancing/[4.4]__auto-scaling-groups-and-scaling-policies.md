## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Auto Scaling overview**

Amazon EC2 Auto Scaling helps you keep the right number of instances running to match demand, improve availability, and reduce cost. In exam scenarios, the key idea is that Auto Scaling reacts to signals such as CPU, request count, queue depth, or custom metrics, then adjusts capacity within the limits you define.

An **Auto Scaling group** (ASG) manages a fleet of EC2 instances across one or more Availability Zones. You define:
- **Minimum capacity**: the lowest number of instances the group must keep running.
- **Desired capacity**: the target number of instances the group tries to maintain.
- **Maximum capacity**: the upper limit the group cannot exceed.

If desired capacity is outside the min/max range, AWS adjusts it to stay within bounds. In practice:
- If demand rises, the ASG launches more instances up to `max`.
- If demand falls, the ASG terminates instances down to `min`.
- If an instance fails health checks, the ASG replaces it automatically.

**Core ASG configuration**

A well-designed ASG starts with a launch template or, less commonly, a launch configuration. Launch templates are the modern choice because they support newer EC2 features such as instance metadata options, multiple versions, and broader configuration flexibility.

Typical launch template settings include:
- AMI ID
- instance type or mixed instances policy
- security groups
- key pair
- IAM instance profile
- user data
- block device mappings
- network interface settings
- metadata options such as IMDSv2 enforcement

A simple launch template example:

```bash
aws ec2 create-launch-template \
  --launch-template-name web-asg-template \
  --version-description "v1" \
  --launch-template-data '{
    "ImageId":"ami-0123456789abcdef0",
    "InstanceType":"t3.micro",
    "IamInstanceProfile":{"Name":"EC2WebRole"},
    "SecurityGroupIds":["sg-0123456789abcdef0"],
    "UserData":"IyEvYmluL2Jhc2gKZHVtbXkgdXNlciBkYXRhCg=="
  }'
```

A basic ASG creation example:

```bash
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name web-asg \
  --launch-template LaunchTemplateName=web-asg-template,Version=1 \
  --min-size 2 \
  --max-size 6 \
  --desired-capacity 2 \
  --vpc-zone-identifier "subnet-aaa,subnet-bbb" \
  --health-check-type EC2 \
  --health-check-grace-period 300
```

**Health checks and replacement behavior**

Auto Scaling can use EC2 status checks, Elastic Load Balancing health checks, or both. This matters because an instance can be “running” at the EC2 layer but still be unhealthy from the application’s point of view.

- **EC2 health checks** detect underlying instance or system issues.
- **ELB health checks** detect whether the application is responding correctly behind a load balancer.
- **Health check grace period** gives new instances time to boot, install software, and register with the load balancer before health evaluation starts.

Use a longer grace period when:
- the application has a slow startup
- bootstrap scripts install packages at launch
- the instance must warm caches or download artifacts

> If the grace period is too short, the ASG may terminate healthy-but-still-initializing instances before they finish booting.

A common exam distinction:
- **Grace period** delays health evaluation for new instances.
- **Warm-up** delays how new capacity is counted for scaling decisions.
- They solve different problems and are not interchangeable.

**Scaling policy types**

Auto Scaling supports several policy types. The exam often tests when to use each one and how they behave.

**Target tracking scaling**

Target tracking is the simplest and most common policy type. You define a target value, and AWS adjusts capacity to keep the metric near that target.

Examples:
- average CPU utilization at 50%
- ALB request count per target at a chosen threshold
- custom metric average at a target value

Target tracking is useful when you want AWS to manage the scaling logic for you. It automatically creates and manages the CloudWatch alarms behind the scenes.

Example target tracking policy for CPU:

```bash
aws autoscaling put-scaling-policy \
  --auto-scaling-group-name web-asg \
  --policy-name cpu-target-tracking \
  --policy-type TargetTrackingScaling \
  --target-tracking-configuration '{
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ASGAverageCPUUtilization"
    },
    "TargetValue": 50.0,
    "DisableScaleIn": false
  }'
```

Common target tracking metrics:
- `ASGAverageCPUUtilization`
- `ALBRequestCountPerTarget`
- custom CloudWatch metrics

Why it is useful:
- simpler than step scaling
- good for steady, proportional workloads
- AWS handles alarm creation and adjustment logic

**Step scaling**

Step scaling changes capacity by different amounts depending on how far the metric breaches a threshold. It uses CloudWatch alarms that you create and attach to a scaling policy.

Use step scaling when:
- you want different scale-out sizes for different load levels
- you need explicit control over thresholds and increments
- you want to react to custom metrics with tailored behavior

A step scaling setup has two parts:
1. A CloudWatch alarm on a metric.
2. A scaling policy attached as the alarm action.

Example scaling policy:

```bash
aws autoscaling put-scaling-policy \
  --auto-scaling-group-name web-asg \
  --policy-name cpu-step-scale-out \
  --policy-type StepScaling \
  --adjustment-type ChangeInCapacity \
  --step-adjustments '[{"MetricIntervalLowerBound":0,"ScalingAdjustment":1},{"MetricIntervalLowerBound":10,"ScalingAdjustment":2},{"MetricIntervalLowerBound":20,"ScalingAdjustment":3}]'
```

Example alarm that triggers the policy:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name web-asg-high-cpu \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 70 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=AutoScalingGroupName,Value=web-asg \
  --alarm-actions arn:aws:autoscaling:us-east-1:123456789012:scalingPolicy:abcd1234-5678-90ab-cdef-123456789012:autoScalingGroupName/web-asg:policyName/cpu-step-scale-out
```

Important behavior:
- CloudWatch evaluates the alarm.
- The alarm action invokes the scaling policy.
- The policy determines how many instances to add or remove.

**Simple scaling**

Simple scaling uses a single adjustment when an alarm breaches a threshold. It is older and less flexible than step scaling, but you may still see it in exam questions or legacy environments.

Behavior to remember:
- one alarm threshold maps to one adjustment
- scaling activity can wait for cooldown before another action
- it is less granular than step scaling

**Scheduled scaling**

Scheduled scaling changes capacity at specific times. Use it for predictable traffic patterns such as business hours, batch windows, or known event schedules.

Example scheduled action:

```bash
aws autoscaling put-scheduled-update-group-action \
  --auto-scaling-group-name web-asg \
  --scheduled-action-name business-hours-scale-up \
  --recurrence "cron(30 7 ? * MON-FRI *)" \
  --min-size 4 \
  --max-size 10 \
  --desired-capacity 6
```

Use scheduled scaling when:
- traffic is predictable
- you want to pre-scale before demand arrives
- you need to reduce cost outside business hours

AWS scheduled actions use cron expressions in the `cron(...)` format, not standard Linux cron syntax.

**Predictive scaling**

Predictive scaling uses historical load patterns and machine learning to forecast future demand and scale ahead of time. It is useful when traffic is recurring and predictable, such as daily peaks.

Typical use cases:
- e-commerce traffic with daily cycles
- reporting jobs with known spikes
- workloads where reactive scaling is too slow

Predictive scaling works best when:
- you have enough historical data
- the workload has repeating patterns
- you want to reduce latency during ramp-up periods

**Warm-up, cooldown, and scaling behavior**

Scaling policies need time to take effect because new instances do not become useful immediately.

- **Instance warm-up** is the time AWS waits before counting a new instance’s capacity in scaling calculations.
- **Cooldown** is a pause between scaling activities to prevent rapid repeated actions.
- **Warm-up** is especially important for target tracking and step scaling.
- **Cooldown** still matters in simple scaling and in scenarios where you want to avoid thrashing.

Practical distinction:
- Use warm-up to avoid overreacting to capacity that is not yet serving traffic.
- Use cooldown to avoid repeated scale actions from the same metric spike.

Example scenario:
- A new instance takes 4 minutes to boot and register with the load balancer.
- Set health check grace period to 300 seconds.
- Set instance warm-up to a similar value so the ASG does not assume the instance is fully contributing too early.

**Load balancer integration**

ASGs commonly sit behind an Application Load Balancer (ALB) or Network Load Balancer (NLB). The load balancer distributes traffic while the ASG maintains capacity.

Common exam patterns:
- Use **ALB request count per target** for web workloads.
- Use **CPU utilization** for general-purpose compute.
- Use **SQS queue depth** for worker fleets.
- Use **custom CloudWatch metrics** for business-specific signals.

Example architecture:
- ALB receives HTTP requests.
- Target group registers ASG instances.
- Target tracking policy scales on request count per target.
- Health checks use the target group path, such as `/health`.

Example target group health check idea:
- path: `/health`
- matcher: `200-399`
- interval: 30 seconds
- healthy threshold: 5
- unhealthy threshold: 2

**Common scaling metrics and when to use them**

| Metric | Best for | Notes |
|---|---|---|
| CPUUtilization | General compute workloads | Easy to understand, but may lag real demand |
| ALBRequestCountPerTarget | Web apps behind ALB | Often better than CPU for request-driven apps |
| SQS queue depth | Worker fleets | Scale based on backlog and processing rate |
| Custom CloudWatch metric | Business-specific signals | Useful when standard metrics do not reflect demand |

For SQS-based workers, you often scale on:
- `ApproximateNumberOfMessagesVisible`
- messages per instance
- age of oldest message

This helps you match worker count to backlog and processing latency.

**ASG lifecycle states and instance lifecycle**

Auto Scaling instance states are broader than just “pending” and “in service.” You may see these states in exam questions:
- `Pending`
- `Pending:Wait`
- `Pending:Proceed`
- `InService`
- `Standby`
- `Terminating`
- `Terminating:Wait`
- `Terminating:Proceed`
- `Terminated`
- `Detaching`
- `EnteringStandby`
- `Quarantined`

Why they matter:
- `Standby` lets you remove an instance from service temporarily without terminating it.
- `Detaching` removes an instance from the ASG while optionally preserving it.
- `Quarantined` can appear when AWS isolates an instance due to internal issues.

**Lifecycle hooks**

Lifecycle hooks let you pause instance launch or termination so you can run custom actions. This is useful for:
- bootstrapping software
- registering with external systems
- draining connections before termination
- sending notifications to automation systems

Launch hook flow:
1. Instance enters `Pending:Wait`.
2. Your automation runs.
3. You call `CompleteLifecycleAction`.
4. Instance moves to `InService`.

Termination hook flow:
1. Instance enters `Terminating:Wait`.
2. You drain traffic or archive data.
3. You complete the lifecycle action.
4. Instance terminates.

Example completion command:

```bash
aws autoscaling complete-lifecycle-action \
  --lifecycle-hook-name web-launch-hook \
  --auto-scaling-group-name web-asg \
  --lifecycle-action-result CONTINUE \
  --instance-id i-0123456789abcdef0 \
  --lifecycle-action-token "token-from-notification"
```

The token is important when the hook is triggered by a notification workflow.

**Warm pools**

Warm pools keep pre-initialized instances ready for faster scale-out. This reduces launch latency because instances can be preloaded before they enter service.

Warm pool instances can be:
- **Stopped**
- **Hibernated**
- **Running**

When the ASG needs capacity, instances move from the warm pool through launch-related states before reaching `InService`. In practice, they may pass through `Pending` and `Pending:Wait` depending on lifecycle hooks and configuration.

Use warm pools when:
- startup time is long
- you need fast scale-out
- you want to reduce cold-start impact without keeping all instances in service

**Instance refresh and rolling updates**

When you change the launch template, AMI, or user data, existing instances do not update automatically. Use **instance refresh** to replace instances gradually with new ones.

Why it matters:
- safer than manually terminating instances
- supports controlled rollout
- helps you avoid full downtime

Typical refresh behavior:
- launch new instances using the updated template
- wait for health checks to pass
- terminate old instances in batches

Example:

```bash
aws autoscaling start-instance-refresh \
  --auto-scaling-group-name web-asg
```

Use instance refresh when:
- you patch the AMI
- you change bootstrap scripts
- you need a rolling deployment across the ASG

**Termination policies**

When the ASG must remove instances, termination policies decide which instance to terminate first. The default behavior generally tries to preserve availability and balance across Availability Zones before considering other factors.

Common considerations:
- balance capacity across AZs
- remove instances that are oldest or least useful for the current configuration
- avoid disrupting the most recently launched capacity unless needed

Exam takeaway:
- Do not assume termination is based on billing-hour alignment.
- Focus on availability, AZ balance, and policy-driven selection.

**Suspending and resuming processes**

You can suspend ASG processes during maintenance, deployments, or troubleshooting. This prevents the group from reacting to temporary conditions while you make changes.

Useful suspended processes may include:
- `Launch`
- `Terminate`
- `AlarmNotification`
- `AZRebalance`
- `HealthCheck`
- `ReplaceUnhealthy`
- `ScheduledActions`

Example suspension:

```bash
aws autoscaling suspend-processes \
  --auto-scaling-group-name web-asg \
  --scaling-processes AlarmNotification Launch Terminate
```

Resume later:

```bash
aws autoscaling resume-processes \
  --auto-scaling-group-name web-asg
```

Use suspension carefully:
- suspend only what you need
- remember to resume processes after maintenance
- avoid leaving health checks or replacement disabled longer than necessary

**Mixed instances policy**

A mixed instances policy lets you use multiple instance types in one ASG, often with Spot and On-Demand capacity together. This improves flexibility and can reduce cost.

Key parts:
- **Launch template**: defines the base configuration
- **Overrides**: specify acceptable instance types
- **InstancesDistribution**: controls On-Demand vs Spot mix and allocation strategy

Example structure:

```json
{
  "LaunchTemplate": {
    "LaunchTemplateSpecification": {
      "LaunchTemplateName": "web-asg-template",
      "Version": "1"
    },
    "Overrides": [
      { "InstanceType": "t3.small" },
      { "InstanceType": "t3.medium" },
      { "InstanceType": "m6i.large" }
    ]
  },
  "InstancesDistribution": {
    "OnDemandBaseCapacity": 1,
    "OnDemandPercentageAboveBaseCapacity": 50,
    "SpotAllocationStrategy": "capacity-optimized"
  }
}
```

Use mixed instances when:
- you want flexibility across instance families
- you want to improve Spot resilience
- you want to optimize cost without locking into one type

**Practical exam scenarios**

**Web application behind ALB**
- Use an ASG across at least two AZs.
- Set health check type to ELB or both EC2 and ELB.
- Use target tracking on `ALBRequestCountPerTarget` or CPU.
- Set a health check grace period long enough for app startup.
- Use instance refresh for AMI updates.

**Queue worker fleet**
- Use an ASG with no load balancer if workers pull from SQS.
- Scale on queue depth or age of oldest message.
- Use step scaling if you want different scale-out sizes for different backlog levels.
- Consider warm pools if worker startup is slow.

**Batch or business-hours workload**
- Use scheduled scaling to pre-scale before the work window.
- Combine with target tracking for unexpected spikes.
- Keep minimum capacity low outside the active window.

**Blue/green or rolling deployment**
- Use instance refresh or a new ASG behind a load balancer.
- Use lifecycle hooks to run pre-termination cleanup.
- Suspend scale-in if you need to stabilize capacity during deployment.

**ECS and capacity providers**

Auto Scaling is also used as the compute layer for Amazon ECS when you run ECS tasks on EC2 instances. In that pattern:
- ECS schedules containers onto EC2 instances in an ASG.
- Capacity providers help ECS manage cluster capacity.
- The ASG scales based on ECS demand signals or custom metrics.

This is useful when:
- you need container orchestration with EC2 control
- you want ECS to request more underlying instances automatically
- you want to align cluster capacity with task placement needs

**Troubleshooting checklist**

- Verify the ASG subnets span multiple AZs.
- Confirm the launch template AMI, security groups, and IAM role are correct.
- Check whether the health check grace period is too short.
- Confirm the load balancer target group health check path returns success.
- Review CloudWatch alarm thresholds and evaluation periods.
- Ensure scaling policies are attached correctly.
- Check whether scaling processes were suspended during maintenance.
- Verify instance warm-up is not causing premature scale-in or delayed scale-out.

**High-value exam reminders**

- **Desired capacity** is the target; min and max are boundaries.
- **Health check grace period** protects new instances from premature replacement.
- **Warm-up** affects scaling calculations; **cooldown** limits repeated actions.
- **Target tracking** is usually the simplest policy for steady workloads.
- **Step scaling** is best when you need different adjustments for different breach sizes.
- **Scheduled scaling** is for predictable time-based demand.
- **Instance refresh** is the preferred way to roll out launch template changes.
- **Lifecycle hooks** let you run custom actions before instances enter or leave service.
- **Mixed instances policies** require `InstancesDistribution` when using the API/CLI.
