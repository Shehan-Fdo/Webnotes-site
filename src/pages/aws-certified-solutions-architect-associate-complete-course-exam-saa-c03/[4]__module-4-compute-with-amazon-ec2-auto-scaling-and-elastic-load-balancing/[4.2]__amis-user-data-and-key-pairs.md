## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**AMIs, user data, and key pairs: the launch-time building blocks**

An <abbr title="Amazon Machine Image">AMI</abbr> is the template you use to create an <abbr title="Elastic Compute Cloud">EC2</abbr> instance. User data is the bootstrap script or configuration you pass at launch. Key pairs provide the SSH or Windows password-decryption trust anchor for accessing the instance after it starts.

These three features solve different problems:

- **AMI** defines the operating system, packages, and baseline configuration.
- **User data** automates first-boot setup.
- **Key pair** enables secure administrative access.

A common exam pattern is to ask which feature you use for a specific goal. If you need a reusable server image, choose an AMI. If you need to run commands on first boot, choose user data. If you need to log in securely, use a key pair.

**Amazon Machine Images (AMIs)**

An AMI is a launch template for EC2 at the operating system and disk-image level. It includes the information needed to start an instance, such as:

- A root volume snapshot or image data
- Launch permissions
- Block device mapping
- Architecture and virtualization details

AMIs are region-scoped. If you want to use an AMI in another Region, you typically copy it to that Region first. This matters for disaster recovery, multi-Region deployments, and reducing launch latency near users.

You usually create AMIs in one of these ways:

- From an existing EC2 instance
- From snapshots and block device mappings
- By copying an existing AMI

When you create an AMI from an instance, AWS captures the instance’s root volume and any attached EBS-backed volumes you include in the image process. This is useful when you have a known-good server state that you want to reproduce.

**When to use an AMI**

Use an AMI when you need:

- Fast, repeatable provisioning
- Immutable infrastructure patterns
- A preconfigured OS image with software already installed
- Consistent scaling in an Auto Scaling Group

AMIs are especially useful when you want to avoid manual configuration drift. Instead of logging into each server and installing software by hand, you bake the software into the image and launch identical instances.

**Practical AMI workflow**

1. Launch a base EC2 instance.
2. Install and configure the required software.
3. Test the instance until it is in a known-good state.
4. Create an AMI from that instance.
5. Launch new instances from the AMI when you need the same setup.

Example AWS CLI commands:

```bash
aws ec2 create-image \
  --instance-id i-0123456789abcdef0 \
  --name "webserver-golden-image" \
  --description "Golden image for web tier" \
  --no-reboot
```

You can then launch from that AMI:

```bash
aws ec2 run-instances \
  --image-id ami-0abc1234def567890 \
  --instance-type t3.micro \
  --key-name my-key-pair
```

> If you use `--no-reboot`, the image may capture a filesystem state that is not fully consistent. Use it only when you understand the tradeoff between speed and data consistency.

**User data**

User data is data you provide to an EC2 instance at launch time. It is commonly used for bootstrapping, such as:

- Installing packages
- Starting services
- Pulling application code
- Writing configuration files
- Joining a cluster or registering with a service

On Linux, user data is often a shell script. On Windows, it is commonly PowerShell or cloud-init-compatible content depending on the AMI and tooling.

User data runs only during the initial boot sequence by default. If you stop and start an instance, the user data script does not automatically run again. This is a frequent exam detail.

**Why user data matters**

User data lets you separate the base image from environment-specific setup. For example:

- The AMI contains the OS and common dependencies.
- User data configures the instance for a specific role, such as `web`, `worker`, or `cache`.

This approach improves reusability. You can launch the same AMI into different environments and let user data customize behavior at boot.

**Linux user data example**

```bash
#!/bin/bash
yum update -y
yum install -y httpd
systemctl enable httpd
systemctl start httpd
echo "<h1>Hello from user data</h1>" > /var/www/html/index.html
```

This script updates packages, installs Apache HTTP Server, enables the service at boot, starts it immediately, and writes a simple web page.

**Cloud-init style user data example**

```yaml
#cloud-config
package_update: true
packages:
  - nginx
runcmd:
  - systemctl enable nginx
  - systemctl start nginx
```

Cloud-init is a common initialization framework on Linux distributions. It interprets metadata and user data to perform first-boot configuration.

**Windows user data example**

```powershell
<powershell>
Install-WindowsFeature Web-Server
Set-Content -Path "C:\inetpub\wwwroot\index.html" -Value "<h1>Hello from Windows user data</h1>"
</powershell>
```

For Windows instances, user data often helps automate IIS setup, application installation, or domain-join tasks.

**User data best practices**

- Keep scripts idempotent when possible so reruns do not break the system.
- Log output to a file for troubleshooting.
- Avoid placing secrets directly in user data.
- Use IAM roles, AWS Systems Manager Parameter Store, or AWS Secrets Manager for sensitive values.
- Test scripts locally or in a staging environment before production use.

A simple pattern for logging on Linux is:

```bash
#!/bin/bash
exec > /var/log/user-data.log 2>&1
set -x

yum update -y
yum install -y nginx
systemctl enable nginx
systemctl start nginx
```

This redirects output and command tracing to a log file, which helps you debug launch failures.

**Key pairs**

A key pair consists of a public key and a private key. In EC2, AWS stores the public key material, and you keep the private key file. Key pairs are used for secure access to instances.

For Linux instances, the key pair is typically used for SSH access. For Windows instances, AWS uses the key pair to decrypt the administrator password after launch.

**How key pairs work**

- You create or import a key pair in AWS.
- AWS injects the public key into the instance at launch.
- You use the private key to authenticate or decrypt access credentials.

Typical Linux SSH access:

```bash
ssh -i my-key-pair.pem ec2-user@ec2-203-0-113-25.compute-1.amazonaws.com
```

Typical Windows password retrieval uses the console or CLI after the instance is ready. The private key is required to decrypt the generated password.

**Key pair management**

- Store the private key securely.
- Restrict file permissions on your local machine.
- Rotate key pairs when access changes.
- Use separate key pairs for different teams or environments when appropriate.

On Linux, set restrictive permissions before SSH:

```bash
chmod 400 my-key-pair.pem
```

If permissions are too open, SSH may refuse to use the key.

**AMI vs user data vs key pair**

| Feature | Primary purpose | When you use it | Common exam clue |
|---|---|---|---|
| AMI | Reusable machine image | You need a prebuilt server template | “Launch identical servers” |
| User data | First-boot automation | You need to install/configure software at startup | “Run a script when the instance boots” |
| Key pair | Secure access | You need SSH or Windows password decryption | “Connect to the instance” |

The distinction is important:

- **AMI** is about what the instance is.
- **User data** is about what the instance does on first boot.
- **Key pair** is about how you access the instance.

**Common architecture patterns**

A typical web tier pattern uses all three:

1. Build a hardened base AMI with the OS and common agents.
2. Pass user data to install the application version or environment-specific config.
3. Attach a key pair for emergency access, though day-to-day access may be limited.

Example topology:

- Public subnet: bastion host or load balancer
- Private subnet: EC2 application instances
- AMI: standardized app server image
- User data: bootstraps app registration
- Key pair: administrative SSH access if needed

In Auto Scaling, AMIs are especially important because new instances must launch quickly and consistently. User data can then personalize each instance at boot, such as registering with a target group or fetching non-secret configuration.

**Troubleshooting launch and access issues**

If an instance does not behave as expected, check these areas:

- **AMI integrity**
  - Confirm the AMI contains the expected packages and OS version.
  - Verify the AMI was copied to the correct Region.
- **User data execution**
  - Check `/var/log/cloud-init-output.log` on many Linux distributions.
  - Check your custom log file if you redirected output.
  - Confirm the script is valid for the AMI’s OS family.
- **Key pair access**
  - Confirm you selected the correct key pair at launch.
  - Verify local file permissions on the private key.
  - Ensure the security group allows SSH on port `22` or RDP on port `3389` as appropriate.

Example Linux troubleshooting commands:

```bash
sudo cat /var/log/cloud-init-output.log
sudo journalctl -u cloud-init
sudo systemctl status nginx
```

If the instance is reachable but the application is not, the issue is often in user data or service startup rather than the AMI itself.

**Security considerations**

Key pairs and user data can both create security risk if handled poorly.

- Do not embed long-lived secrets in user data.
- Treat the private key as sensitive material.
- Prefer IAM-based access patterns where possible.
- Use least privilege for instance roles and security groups.
- Bake only non-sensitive defaults into AMIs.

A secure pattern is to store the application binary in the AMI, then use user data to retrieve configuration from a secure AWS service at boot. That keeps the image reusable without hardcoding secrets.

**Exam focus**

For SAA-C03, remember these high-value distinctions:

- Use an **AMI** to create repeatable EC2 instances.
- Use **user data** to automate first boot.
- Use a **key pair** to access the instance securely.
- AMIs are Region-specific unless copied.
- User data usually runs only on first launch.
- Key pairs are required for SSH access on Linux and password decryption on Windows.

A quick memory aid:

- **Image** = AMI
- **Init** = user data
- **Identity** = key pair

**Practical launch example**

The following example combines all three concepts in one launch:

```bash
aws ec2 run-instances \
  --image-id ami-0abc1234def567890 \
  --instance-type t3.small \
  --key-name my-key-pair \
  --user-data file://user-data.sh \
  --security-group-ids sg-0123456789abcdef0 \
  --subnet-id subnet-0123456789abcdef0
```

Where `user-data.sh` contains:

```bash
#!/bin/bash
yum update -y
yum install -y httpd
systemctl enable httpd
systemctl start httpd
echo "Bootstrapped by user data" > /var/www/html/index.html
```

This pattern is common in real deployments because it separates image management, startup automation, and access control.
