## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon EBS fundamentals**

Amazon Elastic Block Store (EBS) provides persistent block storage for Amazon EC2 instances. You use it when you need low-latency storage for a single instance, such as a database, file system, or application data volume. Unlike instance store, EBS data persists independently of the instance lifecycle as long as the volume remains available.

Key properties to remember:

- **AZ-scoped** storage: an EBS volume exists in one Availability Zone (AZ) and can only be attached to instances in the same AZ.
- **Persistent** data: stopping or terminating an instance does not delete the volume unless you explicitly configure deletion on termination for the root volume.
- **Single-writer by default**: most EBS volumes attach to one instance at a time.
- **Snapshot-backed** recovery: snapshots are stored in Amazon S3-managed infrastructure and are the standard way to back up EBS data.

A practical lab usually covers creating a volume, attaching it to an EC2 instance, formatting and mounting it, then creating and restoring snapshots.

**Attaching an EBS volume to an EC2 instance**

Before attaching, verify that the volume and instance are in the same AZ. If they are not, you must create a new volume from a snapshot in the target AZ or copy the snapshot and restore it there.

Typical workflow:

1. Create or select an EBS volume.
2. Attach it to the target EC2 instance.
3. Log in to the instance and identify the new block device.
4. Create a file system if the volume is new.
5. Mount the volume to a directory.
6. Optionally configure automatic mounting at boot.

Example AWS CLI commands:

```bash
aws ec2 attach-volume \
  --volume-id vol-0123456789abcdef0 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdf
```

On Nitro-based instances, Linux device names may appear as NVMe devices such as `/dev/nvme1n1` even if you specify `/dev/sdf`. Confirm the actual device with `lsblk`.

Useful instance-side commands:

```bash
lsblk
sudo file -s /dev/nvme1n1
sudo mkfs -t xfs /dev/nvme1n1
sudo mkdir -p /data
sudo mount /dev/nvme1n1 /data
df -h
```

If the volume is brand new, `file -s` often shows `data`, which indicates no file system exists yet. After formatting, mount it to a directory that matches your application purpose, such as `/data`, `/app`, or `/var/lib/mysql`.

> Do not format a volume that already contains important data unless you intend to erase it.

**Mounting and persistence**

Mounting makes the file system available to the operating system. The mount point is just a directory, but the underlying block device becomes accessible through that path.

To make the mount persistent across reboots, add an entry to `/etc/fstab`. Use the volume UUID rather than the device name because Linux device names can change.

Example:

```bash
sudo blkid
```

Sample `/etc/fstab` entry:

```fstab
UUID=1234abcd-56ef-7890-abcd-1234567890ab /data xfs defaults,nofail 0 2
```

Important considerations:

- **UUID-based mounting** is more reliable than `/dev/nvme...` paths.
- `nofail` helps the instance boot even if the volume is unavailable.
- Use the correct file system type, such as `xfs` or `ext4`.
- Test the configuration with `sudo mount -a` before rebooting.

If you need to unmount safely:

```bash
sudo umount /data
```

Always stop applications and ensure no process is using the mount before unmounting.

**Snapshot creation and purpose**

An EBS snapshot is a point-in-time backup of an EBS volume. Snapshots are incremental after the first full snapshot, which means only changed blocks are stored in subsequent snapshots. This reduces storage cost and speeds up ongoing backups.

Use snapshots when you need:

- **Backup and restore** for disaster recovery.
- **Volume cloning** to create a new volume with the same data.
- **Cross-AZ recovery** by restoring a volume into another AZ.
- **Cross-Region copy** for geographic resilience.
- **Pre-change protection** before patching or schema updates.

Create a snapshot with the AWS CLI:

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-maintenance backup"
```

You can also tag snapshots for lifecycle management:

```bash
aws ec2 create-tags \
  --resources snap-0123456789abcdef0 \
  --tags Key=Name,Value=pre-maintenance-backup Key=Environment,Value=dev
```

For consistent application backups, quiesce the application or flush buffers before snapshotting. For databases, use the database-native backup mechanism when possible, or coordinate with application freeze steps. EBS snapshots are crash-consistent by default, which is often sufficient for many workloads but not always application-consistent.

**Restoring from a snapshot**

To restore data, create a new volume from the snapshot and attach it to an instance. The restored volume must be created in the same AZ as the target instance.

Example:

```bash
aws ec2 create-volume \
  --snapshot-id snap-0123456789abcdef0 \
  --availability-zone us-east-1a \
  --volume-type gp3
```

Then attach it:

```bash
aws ec2 attach-volume \
  --volume-id vol-0fedcba9876543210 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdg
```

After attachment, mount it on the instance. If the snapshot contains a file system, you usually do not need to format it again. Instead, identify the file system and mount it:

```bash
lsblk -f
sudo mkdir -p /restore
sudo mount /dev/nvme2n1 /restore
```

If you are restoring a root volume, the process is different: you typically create a new volume from the snapshot and replace the root volume by launching a new instance or by using the restored volume as the root device in a recovery workflow.

**Snapshot copy, sharing, and encryption**

Snapshots can be copied to another Region. This is useful for disaster recovery and compliance requirements that need regional separation.

Example:

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --description "DR copy"
```

You can share unencrypted snapshots with other AWS accounts by modifying snapshot permissions. Encrypted snapshots are more restrictive and require KMS (Key Management Service) permissions and compatible key sharing.

Important points:

- **Encrypted snapshots** require access to the AWS KMS key used for encryption.
- You cannot directly share snapshots encrypted with the default AWS-managed key in the same way as customer-managed keys.
- To move encrypted data across accounts or Regions, you often copy the snapshot and re-encrypt it with a customer-managed KMS key.

**EBS volume types and when to use them**

The lab may use a general-purpose volume, but you should know the common choices:

| Volume type | Best for | Notes |
|---|---|---|
| gp3 | General-purpose workloads | Baseline performance is decoupled from size; common default choice |
| gp2 | General-purpose legacy workloads | Performance scales with size; older default |
| io1/io2 | High IOPS workloads | Suitable for databases and latency-sensitive systems |
| st1 | Throughput-oriented HDD | Good for large sequential workloads |
| sc1 | Cold HDD | Lowest cost, infrequent access |

For most labs and many production workloads, `gp3` is the preferred starting point because it offers predictable performance and flexible tuning.

**Linux device naming and troubleshooting**

On Linux, the device name you specify during attachment is not always the name you see inside the OS. Nitro instances commonly expose EBS volumes as NVMe devices.

Troubleshooting checklist:

- Use `lsblk` to identify attached disks and partitions.
- Use `sudo file -s /dev/<device>` to check whether a file system exists.
- Use `sudo dmesg | tail` if the device does not appear.
- Confirm the volume is in the same AZ as the instance.
- Confirm the volume is in the `available` state before attachment.
- Ensure the instance role or user has permission to manage EC2 volumes and snapshots.

Example inspection commands:

```bash
lsblk -o NAME,SIZE,FSTYPE,MOUNTPOINT,UUID
sudo dmesg | tail -n 20
sudo blkid
```

If the volume is attached but not visible, wait a few seconds and rescan. In some cases, rebooting is unnecessary; a device rescan or reattachment is enough.

**Snapshot lifecycle and automation**

Manual snapshots are useful for labs and ad hoc backups, but production environments usually automate snapshot creation and retention. AWS Backup or Data Lifecycle Manager (DLM) can schedule snapshots and manage retention policies.

Why automation matters:

- Reduces human error.
- Enforces backup frequency and retention.
- Supports compliance and recovery objectives.
- Simplifies multi-volume backup strategies.

A typical retention policy might keep hourly snapshots for 24 hours, daily snapshots for 7 days, and monthly snapshots for 12 months. The exact policy depends on your recovery point objective (RPO) and retention requirements.

**Practical lab sequence**

A realistic lab flow looks like this:

1. Launch or select an EC2 instance in a specific AZ.
2. Create a new EBS volume in the same AZ.
3. Attach the volume to the instance.
4. SSH into the instance and identify the device.
5. Create a file system if the volume is blank.
6. Mount the volume and verify access.
7. Write test data to the mount point.
8. Create a snapshot of the volume.
9. Detach or delete the volume if needed.
10. Restore a new volume from the snapshot and mount it again.

Example test write:

```bash
echo "EBS snapshot test" | sudo tee /data/testfile.txt
cat /data/testfile.txt
```

This confirms that the mounted volume is writable and that data survives snapshot-based recovery.

**Common exam and lab takeaways**

- **EBS is AZ-specific**, so attachment requires matching AZs.
- **Snapshots are incremental** and are the standard backup mechanism for EBS.
- **Mounting is an OS task**, while attachment is an AWS control plane task.
- **Use UUIDs in `/etc/fstab`** for reliable persistence.
- **`gp3` is the common default** for general-purpose workloads.
- **Snapshots can be copied across Regions** for disaster recovery.
- **Encrypted snapshot workflows** depend on KMS permissions and key strategy.

**Quick reference commands**

```bash
# List disks
lsblk

# Check file system type
sudo file -s /dev/nvme1n1

# Format a new volume
sudo mkfs -t xfs /dev/nvme1n1

# Create mount point and mount
sudo mkdir -p /data
sudo mount /dev/nvme1n1 /data

# Show UUID for persistent mount
sudo blkid

# Add snapshot
aws ec2 create-snapshot --volume-id vol-0123456789abcdef0 --description "Backup"

# Restore volume from snapshot
aws ec2 create-volume --snapshot-id snap-0123456789abcdef0 --availability-zone us-east-1a --volume-type gp3
```
