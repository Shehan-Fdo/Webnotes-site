## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**What mixed-domain scenario sets test**

Mixed-domain scenario questions combine multiple AWS services and constraints in one architecture decision. You are rarely choosing a single “best” service in isolation; instead, you balance availability, durability, performance, cost, security, and operational overhead across domains such as networking, storage, databases, identity, and monitoring.

These questions usually reward you for recognizing the dominant requirement first:

- **Availability** drives Multi-AZ, failover, and health checks.
- **Durability** drives S3 versioning, replication, and backup strategy.
- **Latency** drives Route 53 routing, CloudFront, and regional placement.
- **Connectivity** drives Transit Gateway, VPC peering, VPN, and Direct Connect.
- **Access control** drives IAM, resource policies, and KMS key design.
- **Operations** drives CloudWatch, AWS Config, and Systems Manager.

A good exam answer often uses the simplest architecture that satisfies the requirement, not the most feature-rich one.

**How to approach scenario questions**

Read the prompt in layers:

1. Identify the business goal.
2. Identify the hard constraints.
3. Identify the failure domain.
4. Match the AWS service to the domain.
5. Eliminate answers that solve only part of the problem.

Common traps include:

- Choosing a service that works technically but does not meet the resilience requirement.
- Using cross-region features when the question only needs Multi-AZ.
- Overengineering with unnecessary replication or custom failover.
- Confusing shared storage, block storage, and object storage use cases.
- Ignoring IAM or resource-based policy requirements for cross-service access.

**Networking and hybrid connectivity**

Networking questions often combine routing, segmentation, and connectivity. The key decision is whether you need many VPCs, transitive routing, or simple one-to-one connectivity.

**VPC peering**

Use VPC peering when you need direct private connectivity between two VPCs and the design is simple.

- It is non-transitive.
- It does not support edge-to-edge routing through a transit VPC.
- It works well for a small number of VPCs.
- It becomes difficult to manage at scale because each VPC pair needs a separate peering connection.

Example use case:

- A shared services VPC needs access to one application VPC.
- You do not need centralized routing across many accounts or VPCs.

**Transit Gateway**

Use AWS Transit Gateway when you need hub-and-spoke routing across many VPCs, accounts, or hybrid connections.

- It supports transitive routing.
- It simplifies multi-VPC and multi-account connectivity.
- It is a strong choice for centralized inspection, shared services, and hybrid network hubs.
- It reduces route table complexity compared with many peering links.

Example topology:

```text
On-premises
   |
VPN / Direct Connect
   |
Transit Gateway
 /    |     \
VPC-A VPC-B  VPC-C
```

A practical route table pattern:

```text
VPC-A route table:
10.20.0.0/16 -> Transit Gateway

Transit Gateway route table:
10.10.0.0/16 -> VPC-A attachment
10.20.0.0/16 -> VPC-B attachment
10.30.0.0/16 -> VPC-C attachment
0.0.0.0/0    -> inspection or egress VPC
```

**When to prefer Transit Gateway over peering**

- You have more than a few VPCs.
- You need transitive routing.
- You need centralized inspection or egress.
- You need multi-account network segmentation.
- You need to connect on-premises networks to multiple VPCs.

**Load balancing and traffic distribution**

Load balancer questions often mix availability, TLS termination, static addressing, and target type selection.

**Application Load Balancer**

Use an Application Load Balancer (ALB) for Layer 7 HTTP/HTTPS traffic, path-based routing, host-based routing, and modern web applications.

- It supports content-based routing.
- It integrates well with ECS, EKS, and Auto Scaling groups.
- It is the usual choice for web apps and APIs.
- For a real multi-AZ ALB, you should specify at least two subnets in different Availability Zones.

Example CLI:

```bash
aws elbv2 create-load-balancer \
  --name app-alb \
  --type application \
  --scheme internet-facing \
  --subnets subnet-aaa111 subnet-bbb222 \
  --security-groups sg-0123456789abcdef0
```

Example listener and target group flow:

```bash
aws elbv2 create-target-group \
  --name app-tg \
  --protocol HTTP \
  --port 80 \
  --vpc-id vpc-0123456789abcdef0 \
  --target-type instance
```

**Network Load Balancer**

Use a Network Load Balancer (NLB) for Layer 4 TCP/UDP traffic, very high performance, and static IP requirements.

- It supports static IPs by assigning Elastic IPs to the NLB in each enabled subnet.
- It is suitable for non-HTTP protocols and low-latency workloads.
- It preserves source IP by default in many scenarios.
- It is often used for TLS pass-through or TCP-based services.

**Gateway Load Balancer**

Use a Gateway Load Balancer when you need transparent insertion of third-party network appliances.

- It is useful for firewalls, IDS/IPS, and inspection appliances.
- It helps centralize traffic inspection without redesigning application routing.

**Load balancer selection summary**

| Requirement | Best fit | Why |
|---|---|---|
| HTTP/HTTPS routing | ALB | Layer 7 routing and integration |
| Static IPs | NLB | Elastic IP support per subnet |
| TCP/UDP performance | NLB | Layer 4 efficiency |
| Appliance insertion | GWLB | Transparent inspection path |

**Storage choices in mixed scenarios**

Storage questions often hinge on whether data must be shared, durable, low-latency, or portable across instances.

**EBS vs EFS vs S3**

- **EBS** is block storage for a single EC2 instance attachment in most common designs.
- **EFS** is shared file storage for multiple EC2 instances and Linux workloads.
- **S3** is object storage for durable, scalable, decoupled data access.

**Use EBS when**

- You need low-latency block storage.
- The application expects a filesystem on a single instance.
- You need boot volumes or database-like block access patterns.

**Use EFS when**

- Multiple EC2 instances need concurrent shared file access.
- You need a managed NFS file system.
- You want elastic capacity without managing storage servers.

Example EFS mount on Linux:

```bash
sudo yum install -y amazon-efs-utils
sudo mkdir -p /mnt/shared
sudo mount -t efs fs-12345678:/ /mnt/shared
```

Example `/etc/fstab` entry:

```fstab
fs-12345678:/ /mnt/shared efs defaults,_netdev 0 0
```

**Use S3 when**

- You need durable object storage.
- You want decoupled application storage.
- You need lifecycle management, replication, or archival tiers.

**S3 durability, availability, and DR features**

S3 is designed for very high durability and regional availability. In scenario questions, the important part is not just “store files,” but how you protect, replicate, and control access to them.

Key features:

- **Versioning** protects against accidental overwrite and delete.
- **Lifecycle policies** move objects to cheaper storage classes or expire them.
- **Cross-Region Replication (CRR)** copies objects to another AWS Region for disaster recovery.
- **Same-Region Replication (SRR)** copies objects within the same Region for compliance or operational separation.
- **Replication Time Control** can help when predictable replication timing matters, though it is not always required.
- **Object Lock** supports write-once-read-many retention for compliance use cases.

Example versioning and lifecycle configuration:

```bash
aws s3api put-bucket-versioning \
  --bucket my-app-logs \
  --versioning-configuration Status=Enabled
```

```json
{
  "Rules": [
    {
      "ID": "ArchiveLogs",
      "Status": "Enabled",
      "Filter": { "Prefix": "logs/" },
      "Transitions": [
        { "Days": 30, "StorageClass": "STANDARD_IA" },
        { "Days": 90, "StorageClass": "GLACIER" }
      ],
      "Expiration": { "Days": 365 }
    }
  ]
}
```

Example CRR configuration concept:

```json
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "Status": "Enabled",
      "Priority": 1,
      "DeleteMarkerReplication": { "Status": "Enabled" },
      "Filter": { "Prefix": "" },
      "Destination": {
        "Bucket": "arn:aws:s3:::my-dr-bucket",
        "StorageClass": "STANDARD"
      }
    }
  ]
}
```

**CloudFront access to S3**

For private S3 origins behind CloudFront, use **Origin Access Control (OAC)** as the modern recommended approach.

- OAC is the current preferred mechanism.
- Origin Access Identity (OAI) is legacy.
- Use bucket policies to allow only the CloudFront distribution to read objects.

Example bucket policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontRead",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-private-bucket/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/E123ABC456"
        }
      }
    }
  ]
}
```

**Database high availability and replication**

Database questions often test whether you need synchronous failover, read scaling, or cross-region disaster recovery.

**RDS Multi-AZ**

Use Amazon RDS Multi-AZ for high availability and automatic failover.

- It provides a standby in another Availability Zone.
- It is designed for failover, not read scaling.
- It is the right answer when the prompt emphasizes availability and minimal recovery time.

**RDS read replicas**

Use read replicas for read scaling and, in some engines, disaster recovery support.

- They are asynchronous.
- They reduce read load on the primary.
- They can be promoted in some failure scenarios, but they are not the same as Multi-AZ standby.

**Aurora**

Amazon Aurora adds stronger replication and failover capabilities than standard RDS in many scenarios.

- Aurora clusters separate storage from compute.
- Aurora replicas can fail over quickly.
- Aurora Global Database supports cross-region replication for disaster recovery and low-latency global reads.

**How to choose**

- Use **Multi-AZ** for automatic failover.
- Use **read replicas** for read-heavy workloads.
- Use **Aurora Global Database** for cross-region DR and global applications.
- Use **cross-region read replicas** when the engine supports them and the scenario calls for regional separation.

Example RDS Multi-AZ creation:

```bash
aws rds create-db-instance \
  --db-instance-identifier appdb \
  --db-instance-class db.t3.medium \
  --engine mysql \
  --allocated-storage 100 \
  --multi-az \
  --master-username admin \
  --master-user-password 'ChangeMe123!'
```

Example Aurora cluster concept:

```bash
aws rds create-db-cluster \
  --db-cluster-identifier aurora-prod \
  --engine aurora-mysql \
  --master-username admin \
  --master-user-password 'ChangeMe123!'
```

**Route 53 routing policies**

Route 53 questions often combine DNS with failover, latency, or traffic distribution. You should know the main routing policies and when each one fits.

| Routing policy | Best use case | Key behavior |
|---|---|---|
| Simple | One endpoint or basic aliasing | Returns one record set |
| Weighted | Blue/green, canary, traffic split | Sends traffic by weight |
| Latency-based | Global users, lowest-latency region | Routes to the lowest-latency healthy endpoint |
| Failover | Primary/secondary DR | Uses health checks to switch endpoints |
| Geolocation | Country/region-specific content | Routes by user location |
| Multivalue answer | Simple health-aware distribution | Returns multiple healthy records |

**Simple routing**

Use simple routing when you have one endpoint or a straightforward alias target.

**Weighted routing**

Use weighted routing when you want to split traffic between versions or regions.

Example:

- 90% to current production
- 10% to new version for validation

**Latency-based routing**

Use latency-based routing when users are global and you want the lowest-latency AWS Region.

**Failover routing**

Use failover routing when you need active-passive behavior with health checks.

Example failover pattern:

```text
Primary record -> us-east-1 ALB
Secondary record -> us-west-2 ALB
Health check -> primary endpoint
```

**Geolocation routing**

Use geolocation routing when content or compliance depends on user location.

**Multivalue answer routing**

Use multivalue answer routing when you want Route 53 to return multiple healthy records, often for simple client-side resilience.

**Security, IAM, and cross-service access**

Security questions in mixed scenarios often depend on how AWS evaluates permissions across identity-based and resource-based policies.

**IAM policy evaluation basics**

AWS evaluates permissions by combining several policy types:

- **Identity-based policies** attached to users, groups, or roles.
- **Resource-based policies** attached to resources such as S3 buckets, KMS keys, SNS topics, and SQS queues.
- **Permission boundaries** that limit maximum permissions.
- **Service control policies (SCPs)** in AWS Organizations that set account-level guardrails.

A request must be allowed by the relevant policies and not explicitly denied.

**Resource-based policies**

Use resource-based policies when another account, service, or principal needs direct access to a resource.

Common examples:

- S3 bucket policies
- KMS key policies
- SNS topic policies
- SQS queue policies
- Lambda permissions

Example S3 bucket policy for cross-account read:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::111122223333:role/AnalyticsRole" },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-data-bucket/*"
    }
  ]
}
```

**Cross-service access patterns**

Many AWS services assume roles or use service principals.

- EC2 instances use instance profiles.
- Lambda functions use execution roles.
- ECS tasks use task roles.
- CloudWatch Logs can deliver to S3 or Kinesis with service permissions.
- CloudFront reads from S3 using OAC and bucket policy conditions.

**KMS considerations**

When encryption is involved, remember that the caller needs both IAM permission and KMS key permission.

- A bucket policy may allow S3 access.
- The KMS key policy must also allow decrypt or encrypt operations.
- Cross-account access often fails because the KMS key policy is incomplete.

**Observability, governance, and operations**

Operational questions often combine monitoring, compliance, and automation.

**CloudWatch**

Use Amazon CloudWatch for metrics, logs, and alarms.

- Metrics drive alarms and dashboards.
- Logs capture application and system output.
- Alarms can trigger Auto Scaling, SNS, or automation.

For EC2 CPU alarms, dimension by `InstanceId` when monitoring a single instance.

Example:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name high-cpu-instance \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 300 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --evaluation-periods 2 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-alerts
```

For Auto Scaling group scaling, use a scaling policy or a metric that is actually published for the group, rather than mixing an EC2 instance metric with an ASG dimension.

**AWS Config**

Use AWS Config to track resource configuration history and compliance.

- It helps answer “what changed?” and “is this resource compliant?”
- It is useful for governance, audit, and drift detection.
- It pairs well with Config rules and remediation.

**Systems Manager**

Use AWS Systems Manager for fleet operations and automation.

- Run Command executes commands on managed instances.
- Session Manager provides shell access without opening inbound SSH.
- Patch Manager helps with patch compliance.
- Automation documents standardize operational tasks.

Example Session Manager benefit:

- You can manage EC2 instances without bastion hosts.
- You reduce inbound network exposure.

**Mixed-domain decision patterns**

These patterns show how multiple domains combine in exam questions.

**Web application with global users**

A common design is:

- Route 53 latency-based routing
- CloudFront for caching and edge delivery
- ALB in each Region
- RDS Multi-AZ in each Region or Aurora Global Database for cross-region needs
- S3 with versioning and CRR for static assets or backups

Why this works:

- DNS sends users to the nearest healthy Region.
- CloudFront reduces latency and origin load.
- Multi-AZ protects regional availability.
- Replication supports DR.

**Shared services across multiple accounts**

A common design is:

- Transit Gateway as the network hub
- Centralized inspection VPC
- Shared services VPC for DNS, AD, or tooling
- IAM roles and resource policies for cross-account access
- AWS Config and Systems Manager for governance

Why this works:

- It scales better than peering.
- It centralizes control.
- It supports multi-account growth.

**EC2 application with shared files and database**

A common design is:

- ALB for web traffic
- EC2 Auto Scaling for compute
- EFS for shared uploads or application state
- RDS Multi-AZ for transactional data
- CloudWatch alarms and Systems Manager for operations

Why this works:

- ALB distributes traffic.
- EFS gives shared file access across instances.
- RDS Multi-AZ provides failover.
- CloudWatch and Systems Manager improve operability.

**Disaster recovery scenario**

A common DR design is:

- S3 versioning plus CRR for object data
- RDS cross-region replica or Aurora Global Database
- Route 53 failover routing
- Infrastructure as code for rapid rebuild
- AWS Backup or snapshots for additional recovery points

Why this works:

- It separates data protection from compute recovery.
- It supports regional failover.
- It reduces recovery time compared with manual rebuilds.

**Practical topology examples**

**Example 1: Multi-VPC enterprise network**

```text
On-prem -> Direct Connect -> Transit Gateway -> Shared Services VPC
                                         \-> App VPC 1
                                         \-> App VPC 2
                                         \-> Security Inspection VPC
```

Use this when:

- You need centralized routing.
- You have multiple accounts.
- You want consistent egress and inspection.

**Example 2: Active-passive web DR**

```text
Users -> Route 53 failover
   Primary -> ALB in us-east-1 -> RDS Multi-AZ
   Secondary -> ALB in us-west-2 -> warm standby app tier
```

Use this when:

- You need regional failover.
- You can tolerate passive standby costs.
- You want a simpler DR model than active-active.

**Example 3: Shared file processing**

```text
EC2 Auto Scaling Group -> EFS shared mount
                         -> S3 for input/output
                         -> CloudWatch for logs and alarms
```

Use this when:

- Multiple instances must read and write the same files.
- You need durable object storage for artifacts.
- You want elastic compute without local state.

**Exam tips for mixed-domain questions**

- Start with the requirement that is hardest to satisfy.
- Prefer managed services when they meet the requirement.
- Distinguish failover from scaling.
- Distinguish shared storage from object storage.
- Use Route 53 routing policy names precisely.
- Remember that IAM and resource policies often decide whether a design actually works.
- Check whether the question is asking for regional resilience or AZ resilience.
- Look for words like “lowest latency,” “multi-account,” “shared file system,” “static IP,” “automatic failover,” and “cross-region DR.”

**Quick reference**

- **ALB** for HTTP/HTTPS and content-based routing.
- **NLB** for static IPs and Layer 4 traffic.
- **Transit Gateway** for scalable hub-and-spoke networking.
- **VPC peering** for simple non-transitive connectivity.
- **EBS** for block storage.
- **EFS** for shared file storage.
- **S3** for object storage, versioning, and replication.
- **RDS Multi-AZ** for failover.
- **Read replicas** for read scaling.
- **Aurora Global Database** for cross-region database DR.
- **Route 53** for DNS routing and failover.
- **CloudWatch, Config, and Systems Manager** for operations and governance.
