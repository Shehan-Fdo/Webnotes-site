## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**How to approach distractor-heavy questions**

SAA-C03 questions often hide the correct answer behind a service that is technically possible but operationally wrong for the requirement. Your job is to identify the *best fit* for the workload, not just a valid AWS service.

Common exam traps usually involve:
- Choosing a service that solves the problem but adds unnecessary management overhead.
- Picking a storage or database service that does not match access patterns.
- Confusing identity, encryption, and secret storage responsibilities.
- Missing the difference between global routing, regional routing, and application-level routing.
- Overlooking high availability, durability, and recovery time objectives.

A reliable method is to read the question in this order:
1. Identify the workload type: web app, batch job, analytics, file sharing, or event-driven.
2. Identify the constraint: cost, latency, scale, compliance, durability, or minimal operations.
3. Eliminate services that are “close” but not ideal.
4. Choose the simplest service that satisfies all stated requirements.

**High-frequency exam decision patterns**

The exam repeatedly tests whether you can distinguish between services that appear similar on the surface.

| Scenario | Best fit | Common distractor | Why the distractor fails |
|---|---|---|---|
| Shared file system for many EC2 instances | `EFS` | `EBS` | `EBS` is attached to one instance at a time in normal use |
| Durable object storage for static content | `S3` | `EFS` | `EFS` is a file system, not object storage |
| Low-latency block storage for a single instance | `EBS` | `S3` | `S3` is not mounted as a block device |
| Windows shared file system with SMB | `FSx for Windows File Server` | `EFS` | `EFS` uses NFS, not SMB |
| Linux shared file system with POSIX semantics | `EFS` | `FSx for Windows File Server` | Wrong protocol and platform fit |
| Serverless event-driven compute | `Lambda` | `EC2` | `EC2` requires instance management |
| Container orchestration with full control | `ECS` or `EKS` | `Elastic Beanstalk` | Beanstalk abstracts too much for container-native control |
| Simple web app deployment with managed platform | `Elastic Beanstalk` | `ECS` | Beanstalk is easier when you do not need orchestration detail |
| Relational transactions and joins | `RDS` or `Aurora` | `DynamoDB` | `DynamoDB` is not relational |
| Massive key-value scale with predictable access | `DynamoDB` | `RDS` | `RDS` scales differently and is not ideal for extreme key-value workloads |

**Storage distractors: EBS vs EFS vs S3 vs FSx**

Storage questions are among the most common exam traps because each service is “storage,” but each solves a different problem.

**EBS**
- Use `EBS` for block storage attached to a single EC2 instance.
- Choose it when you need low-latency disk-like storage for a database, boot volume, or application data on one server.
- It supports snapshots to `S3` for backup and recovery.
- It is not the right answer for shared access across many instances.

**EFS**
- Use `EFS` when multiple Linux instances need concurrent shared file access.
- It is elastic and managed, so it fits web tiers, content repositories, and shared application assets.
- It uses NFS and supports POSIX-style file system semantics.
- It is usually the right answer when the question says “shared file system” and “multiple EC2 instances.”

**S3**
- Use `S3` for object storage, static assets, backups, logs, data lakes, and durable content distribution.
- It is not mounted like a traditional file system in the exam context.
- If the question mentions lifecycle policies, versioning, cross-region replication, or static website hosting, `S3` is often the intended answer.
- If the workload needs file locking or shared POSIX semantics, `S3` is usually not the best fit.

**FSx**
- Use `FSx for Windows File Server` for Windows workloads that need SMB and Active Directory integration.
- Use `FSx for Lustre` for high-performance computing, media processing, or workloads that need very high throughput and low latency.
- `FSx` is the distractor when the question says “file system” but also includes Windows, SMB, or specialized performance requirements.

> If the question says “shared file system for Linux instances,” think `EFS`. If it says “shared file system for Windows servers,” think `FSx for Windows File Server`. If it says “durable object storage,” think `S3`. If it says “single-instance disk,” think `EBS`.

**Practical storage decision examples**

A media team needs a shared repository for Linux render nodes that all mount the same directory at once. The best answer is usually `EFS`, not `S3`, because the workload needs a file system, not object storage.

A database server needs fast attached storage for transaction logs. The best answer is usually `EBS`, not `EFS`, because the database expects block storage attached to one host.

A company wants to store millions of images and serve them through a CDN. The best answer is usually `S3` with `CloudFront`, not `EFS`, because the content is object-based and globally distributed.

A Windows-based application requires shared file access and Active Directory permissions. The best answer is usually `FSx for Windows File Server`, not `EFS`, because SMB and Windows integration matter.

**Compute distractors: EC2 Auto Scaling vs Elastic Beanstalk vs Lambda vs ECS/EKS**

Compute questions often test how much operational control you need.

**EC2 Auto Scaling**
- Use it when you need to scale EC2 instances based on demand.
- It is a scaling mechanism, not a full application platform.
- Pair it with an Application Load Balancer when you need horizontal scaling for web apps.
- It is the right answer when the question emphasizes instance fleets, health checks, and replacing unhealthy instances.

**Elastic Beanstalk**
- Use it when you want a managed deployment platform for web applications with minimal infrastructure management.
- It handles provisioning, load balancing, scaling, and deployment orchestration.
- It is often the best fit for “deploy my app quickly” scenarios.
- It is a distractor when the question requires deep container orchestration or custom scheduling.

**Lambda**
- Use `Lambda` for event-driven, short-lived, stateless compute.
- It is ideal for file processing, API backends, automation, and asynchronous workflows.
- It is not the best fit for long-running processes, persistent connections, or workloads that need full OS control.
- If the question mentions “run code in response to an event” and “no servers,” `Lambda` is usually the answer.

**ECS and EKS**
- Use `ECS` for container orchestration when you want AWS-native control and simpler operations.
- Use `EKS` when you need Kubernetes compatibility or an existing Kubernetes ecosystem.
- These are the right answers when the question explicitly says containers, task definitions, pods, services, or orchestration.
- They are distractors when the requirement is just “deploy a web app” and no container-specific need exists.

**Decision cues**
- Choose `Lambda` for event-driven, stateless, bursty workloads.
- Choose `EC2 Auto Scaling` for traditional servers that need horizontal scaling.
- Choose `Elastic Beanstalk` for managed application deployment with minimal ops.
- Choose `ECS` or `EKS` for container-native architectures.

**Example architecture choices**
- A photo upload pipeline that resizes images after upload: `S3` event → `Lambda` → `S3`.
- A customer portal that needs rolling deployments and automatic scaling without managing servers: `Elastic Beanstalk`.
- A microservices platform with service discovery and container scheduling: `ECS` or `EKS`.
- A legacy Java app that must run on EC2 but scale with traffic: `EC2 Auto Scaling` + `ALB`.

**Database distractors: DynamoDB vs RDS vs Aurora**

Database questions often hinge on whether the data is relational or non-relational, and whether the access pattern is predictable.

**RDS**
- Use `RDS` for managed relational databases such as MySQL, PostgreSQL, MariaDB, Oracle, or SQL Server.
- It fits transactions, joins, foreign keys, and structured schemas.
- It is the safer answer when the question mentions SQL queries or relational integrity.
- It is not ideal for extremely high-scale key-value access patterns.

**Aurora**
- Use `Aurora` when the question wants a high-performance managed relational database with AWS-specific scaling and availability features.
- It is often the best relational choice when the workload needs better performance or read scaling than standard RDS.
- It still remains relational, so it is not a substitute for NoSQL.

**DynamoDB**
- Use `DynamoDB` for serverless NoSQL key-value or document workloads.
- It is ideal for massive scale, low-latency access, and predictable access patterns.
- It is the right answer when the question mentions single-digit millisecond latency, flexible schema, or high request volume.
- It is not the best fit for complex joins or ad hoc relational queries.

**On-demand vs provisioned capacity**
- Use `on-demand` when traffic is unpredictable or spiky and you want minimal capacity planning.
- Use `provisioned` when traffic is steady and you want cost control with predictable throughput.
- If the question says “unknown traffic pattern,” “bursty workload,” or “no capacity planning,” `on-demand` is often the safer answer.
- If the question says “steady traffic” and “cost optimization,” `provisioned` may be better.

**DynamoDB practical examples**
- A gaming leaderboard with frequent reads and writes: `DynamoDB`.
- A shopping cart service with low-latency lookups by user ID: `DynamoDB`.
- A payroll system with complex joins and reporting: `RDS` or `Aurora`.
- A content management system with relational relationships: `RDS` or `Aurora`.

```bash
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=OrderId,AttributeType=S \
  --key-schema AttributeName=OrderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

This example uses `PAY_PER_REQUEST`, which maps to on-demand billing. It is useful when traffic is unpredictable and you want to avoid provisioning read and write capacity.

```json
{
  "TableName": "Sessions",
  "BillingMode": "PROVISIONED",
  "ProvisionedThroughput": {
    "ReadCapacityUnits": 20,
    "WriteCapacityUnits": 10
  }
}
```

This configuration is a reminder that provisioned mode is a deliberate capacity choice. The exam may ask you to optimize cost for stable workloads, which is where provisioned capacity can be appropriate.

**Route 53 and DNS traps**

DNS questions often hide the answer in routing behavior, alias support, or health-based failover.

**Alias vs CNAME**
- Use a `CNAME` for mapping one DNS name to another DNS name, typically for non-apex records.
- Use a Route 53 `alias` record when pointing to AWS resources such as `ALB`, `CloudFront`, `S3` website endpoints, or another Route 53 record.
- `Alias` records can be used at the zone apex, while `CNAME` cannot be used at the root domain.
- In exam questions, if the target is an AWS resource and the domain is the root, `alias` is usually the correct choice.

**Health checks**
- Route 53 health checks let DNS respond based on endpoint availability.
- They are often used with failover routing, latency routing, or weighted routing.
- If the question says “route traffic away from unhealthy endpoints,” think health checks plus failover policy.

**Routing policies**
- Use `simple` routing for one resource.
- Use `weighted` routing for gradual traffic shifting or blue/green testing.
- Use `latency` routing to send users to the lowest-latency region.
- Use `failover` routing for active-passive disaster recovery.
- Use `geolocation` routing when traffic must be based on user location.
- Use `multivalue answer` when you want multiple healthy records returned without a full load balancer.

**Common DNS traps**
- `CloudFront` distributions use `alias` records in Route 53.
- `S3` static website endpoints can be targets for `alias` records, but the exact endpoint type matters.
- `ALB` and `NLB` are common `alias` targets.
- If the question mentions apex domain routing to AWS infrastructure, `alias` is usually the intended answer.

```text
example.com -> Route 53 alias -> ALB -> EC2 Auto Scaling group
```

This is a common web architecture. The alias record avoids the `CNAME` apex limitation and keeps DNS aligned with AWS-managed endpoints.

**Security distractors: KMS vs Secrets Manager vs SSM Parameter Store**

Security questions often test whether you are storing a secret, encrypting data, or controlling access.

**KMS**
- Use `KMS` for encryption key management.
- It protects data at rest by managing encryption keys used by AWS services and your applications.
- It is not a secret store.
- If the question asks about encrypting `S3`, `EBS`, `RDS`, or `EFS`, `KMS` is often involved.

**Secrets Manager**
- Use `Secrets Manager` for rotating and storing secrets such as database passwords, API keys, and credentials.
- It is the best fit when the question mentions automatic rotation.
- It is a distractor if the problem is only parameter storage without rotation.

**SSM Parameter Store**
- Use `Systems Manager Parameter Store` for configuration values, plain text parameters, and secure strings.
- It is often simpler and cheaper than Secrets Manager for non-rotating configuration.
- It is a good fit for application settings, feature flags, and environment variables.
- If the question explicitly says “rotate secrets automatically,” `Secrets Manager` is usually better.

**IAM policy vs resource policy vs SCP vs permissions boundary**
- Use an `IAM identity policy` to grant permissions to a user, group, or role.
- Use a `resource policy` to grant access directly to a resource such as `S3`, `KMS`, or `SNS`.
- Use an `SCP` (Service Control Policy) in AWS Organizations to set the maximum permissions for accounts or OUs.
- Use a `permissions boundary` to limit the maximum permissions an IAM principal can receive.

**How to distinguish them**
- If the question is about one user or role getting access, think identity policy.
- If the question is about a bucket, key, or topic allowing cross-account access, think resource policy.
- If the question is about organization-wide guardrails, think SCP.
- If the question is about limiting what a delegated admin can grant, think permissions boundary.

> An SCP does not grant permissions by itself. It only sets the ceiling. A permissions boundary also does not grant permissions by itself. It limits the maximum permissions of the principal.

**Networking distractors: VPC peering vs Transit Gateway vs VPN vs Direct Connect**

Networking questions often ask how to connect networks, and the wrong answer usually has the right connectivity but the wrong scale or latency.

**VPC peering**
- Use `VPC peering` for simple one-to-one private connectivity between two VPCs.
- It does not support transitive routing.
- It is a good fit for small numbers of VPCs.
- It becomes hard to manage when many VPCs need to communicate.

**Transit Gateway**
- Use `Transit Gateway` for hub-and-spoke connectivity across many VPCs and on-premises networks.
- It simplifies routing at scale.
- It is the right answer when the question mentions many VPCs, centralized connectivity, or transitive routing.
- It is usually the better choice than peering in multi-account enterprise networks.

**VPN**
- Use `Site-to-Site VPN` for encrypted connectivity over the public internet.
- It is faster to set up than dedicated circuits.
- It is a common answer when the question mentions secure connectivity without dedicated physical links.
- It is not the best fit for the lowest latency or highest throughput requirements.

**Direct Connect**
- Use `Direct Connect` for dedicated private network connectivity between on-premises and AWS.
- It is the right answer when the question emphasizes consistent latency, high bandwidth, or private connectivity.
- It is not an internet VPN replacement if the requirement is simply “encrypted over the internet.”

**Connectivity decision cues**
- Choose `VPC peering` for a small, simple mesh.
- Choose `Transit Gateway` for many VPCs and centralized routing.
- Choose `VPN` for quick encrypted connectivity over the internet.
- Choose `Direct Connect` for dedicated, predictable enterprise connectivity.

**Load balancing and application delivery traps**

Load balancer questions often test session persistence, protocol support, and target type.

**ALB**
- Use `Application Load Balancer` for HTTP and HTTPS traffic.
- It supports path-based and host-based routing.
- It can route to EC2 instances, containers, and Lambda functions.
- It is the right answer for modern web apps and microservices.

**NLB**
- Use `Network Load Balancer` for ultra-high performance, low latency, and TCP/UDP traffic.
- It preserves source IP and handles millions of requests per second.
- It is often the right answer for non-HTTP protocols or when static IPs matter.

**Sticky sessions**
- Use sticky sessions when a user must stay on the same backend instance for session state.
- This is a common distractor because it can solve session affinity, but it is often a sign that the app should be redesigned to be stateless.
- If the question mentions session persistence and HTTP, `ALB` with sticky sessions may be appropriate.
- If the question requires stateful session handling at scale, consider moving session state to `ElastiCache`, `DynamoDB`, or another shared store.

**Target groups**
- `ALB` routes traffic to target groups.
- Target groups can contain EC2 instances, IP addresses, containers, or Lambda functions.
- If the question mentions blue/green deployments or canary releases, target groups and weighted routing patterns often appear together.

```text
Client -> ALB
       -> Target Group A (v1)
       -> Target Group B (v2)
```

This pattern supports controlled traffic shifting. It is useful when the exam asks how to reduce deployment risk.

**Troubleshooting scenario: choosing the right delivery layer**

A company runs a web app with path-based routing:
- `/api` should go to one service.
- `/images` should go to another service.
- The app uses containers and needs automatic scaling.

The best fit is usually `ALB` in front of `ECS` services with separate target groups. `NLB` is not the best answer because path-based routing is an application-layer feature.

**Disaster recovery and resilience distractors**

DR questions often test whether you need backup, pilot light, warm standby, or multi-site active-active.

**Backup and restore**
- Lowest cost.
- Highest recovery time.
- Best when downtime is acceptable and data loss tolerance is moderate.

**Pilot light**
- Core services are running in a minimal form.
- Faster recovery than backup and restore.
- Good when you need a small always-on footprint.

**Warm standby**
- A scaled-down but functional environment is always running.
- Faster failover than pilot light.
- Higher cost than pilot light.

**Multi-site active-active**
- Multiple sites serve traffic simultaneously.
- Best availability and lowest recovery time.
- Highest complexity and cost.

**Common DR trap**
- If the question says “lowest cost disaster recovery,” do not overbuild with active-active.
- If it says “minimal downtime,” do not choose backup and restore.
- If it says “fast recovery with limited cost,” pilot light or warm standby is often the right compromise.

**Practical DR example**
- Primary region hosts `ALB`, `Auto Scaling`, `RDS`, and `S3`.
- Secondary region keeps a minimal stack ready.
- `Route 53 failover` sends traffic to the secondary region if health checks fail.
- `S3 Cross-Region Replication` keeps object data available in the backup region.

**Common wording traps and how to decode them**

AWS exam questions often use phrases that point to a service family rather than a specific service name.

- “No servers to manage” often points to `Lambda`, `S3`, `DynamoDB`, or managed services.
- “Shared file system” usually points to `EFS` or `FSx`.
- “Object storage” points to `S3`.
- “Block storage” points to `EBS`.
- “Low-latency key-value store” points to `DynamoDB`.
- “Managed relational database” points to `RDS` or `Aurora`.
- “Many VPCs” points to `Transit Gateway`.
- “Dedicated private link” points to `Direct Connect`.
- “Encrypted over the internet” points to `VPN`.
- “Path-based routing” points to `ALB`.
- “Static website” points to `S3` and often `CloudFront`.

**Practical elimination strategy**

When two answers both seem correct, ask which one matches the operational model.

1. Does the workload need a file system, object storage, or block storage?
2. Does it need relational queries or key-value access?
3. Does it need server management or serverless execution?
4. Does it need one VPC connection or many?
5. Does it need application-layer routing or network-layer routing?
6. Does it need a secret store, encryption keys, or both?

This approach helps you eliminate distractors that are technically valid but architecturally wrong.

**Mini reference: service boundary reminders**

- `S3` stores objects; it does not behave like a mounted shared file system.
- `EBS` attaches to one instance; it is not the shared storage answer.
- `EFS` is shared Linux file storage; it is not the Windows SMB answer.
- `DynamoDB` is NoSQL; it is not the relational join-heavy answer.
- `Lambda` is event-driven; it is not the long-running server answer.
- `ALB` is application-aware; `NLB` is transport-aware.
- `KMS` manages keys; `Secrets Manager` stores secrets; `Parameter Store` stores configuration and secure strings.
- `VPC peering` is simple and non-transitive; `Transit Gateway` is scalable and transitive.
- `IAM policies` grant permissions to identities; `resource policies` grant access to resources; `SCPs` restrict accounts; `permissions boundaries` cap principals.

**Quick scenario drills**

A startup wants to host a static marketing site with global caching and low operational overhead.
- Best fit: `S3` + `CloudFront`
- Distractor: `EC2` behind a load balancer

A SaaS company needs a shared file system for Linux app servers.
- Best fit: `EFS`
- Distractor: `EBS`

A finance app needs ACID transactions and complex joins.
- Best fit: `Aurora` or `RDS`
- Distractor: `DynamoDB`

A mobile backend receives unpredictable traffic spikes.
- Best fit: `Lambda` or `DynamoDB on-demand`
- Distractor: overprovisioned EC2 fleet

A company needs to connect 20 VPCs and two on-premises sites.
- Best fit: `Transit Gateway`
- Distractor: full mesh `VPC peering`

A web app needs path-based routing to multiple container services.
- Best fit: `ALB`
- Distractor: `NLB`

A team needs to store database credentials and rotate them automatically.
- Best fit: `Secrets Manager`
- Distractor: `SSM Parameter Store`

**Exam mindset for tricky questions**

The exam often rewards the answer that is:
- Most aligned with the stated access pattern.
- Least operationally complex.
- Most scalable for the described growth.
- Most specific to the protocol or storage type mentioned.
- Most consistent with AWS-managed best practices.

If you see a service that is “close enough,” verify whether it matches:
- Data model
- Access pattern
- Scale profile
- Network topology
- Management overhead
- Security boundary

That final check is often what separates the correct answer from a plausible distractor.
