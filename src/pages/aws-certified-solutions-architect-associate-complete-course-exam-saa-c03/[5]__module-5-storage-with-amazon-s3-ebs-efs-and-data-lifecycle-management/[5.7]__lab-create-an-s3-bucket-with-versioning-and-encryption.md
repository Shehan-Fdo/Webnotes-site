## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab overview**

This lab focuses on creating an <abbr title="Amazon Simple Storage Service">Amazon S3</abbr> bucket with two foundational protections: **versioning** and **encryption**. Versioning helps you recover from accidental deletes and overwrites, while encryption protects data at rest. These are common baseline controls for production S3 workloads and frequent exam topics for SAA-C03.

In practice, you use S3 buckets for static assets, backups, logs, data lakes, and application uploads. The lab is useful because it combines storage durability with operational safety and security controls.

- **Versioning** preserves multiple object revisions in the same bucket.
- **Encryption** protects stored objects without changing how applications read and write them.
- **Bucket-level settings** are usually the first controls you apply after bucket creation.

**S3 bucket fundamentals**

An S3 bucket is a globally unique container for objects. Objects are stored as key-value pairs, where the key is the full path-like name of the object. S3 is highly durable and scales automatically, so you do not manage servers or disks.

Important bucket characteristics:

- **Global namespace** means bucket names must be unique across all AWS accounts and Regions.
- **Regional placement** determines where the bucket metadata and data reside.
- **Object ownership** and access policies control who can read or write data.
- **Default encryption** applies automatically to new objects unless overridden by a client request.

A typical secure setup starts with a bucket, then layers on versioning, encryption, public access blocking, and least-privilege access.

**Create the bucket**

You can create a bucket from the AWS Management Console or the AWS Command Line Interface (CLI). The console is useful for learning the workflow, while the CLI is better for repeatable automation.

Example CLI creation:

```bash
aws s3api create-bucket \
  --bucket my-secure-lab-bucket-123456 \
  --region us-east-1
```

For Regions other than `us-east-1`, you usually need a location constraint:

```bash
aws s3api create-bucket \
  --bucket my-secure-lab-bucket-123456 \
  --region us-west-2 \
  --create-bucket-configuration LocationConstraint=us-west-2
```

Key points:

- **Bucket names** should be unique, lowercase, and DNS-compatible.
- **Region choice** affects latency, compliance, and data residency.
- **Tags** help with cost allocation and governance.
- **Block Public Access** should remain enabled unless you intentionally host public content.

A practical naming pattern is to include the project, environment, and a random suffix:

```text
myapp-dev-logs-7f3a2c
```

**Enable versioning**

Versioning keeps multiple versions of an object in the same bucket. When enabled, S3 assigns a unique version ID to each object write. If you overwrite or delete an object, the previous versions remain available unless you permanently remove them.

Enable versioning with the CLI:

```bash
aws s3api put-bucket-versioning \
  --bucket my-secure-lab-bucket-123456 \
  --versioning-configuration Status=Enabled
```

Verify the setting:

```bash
aws s3api get-bucket-versioning \
  --bucket my-secure-lab-bucket-123456
```

Expected output:

```json
{
  "Status": "Enabled"
}
```

Operational behavior to remember:

- **Overwrite protection** is logical, not physical; the latest version becomes current.
- **Delete markers** appear when you delete an object in a versioned bucket.
- **Recovery** is possible by restoring a prior version or removing a delete marker.
- **Storage cost** can increase because old versions continue to consume space.

Common use cases:

- Protecting application uploads from accidental replacement
- Preserving audit logs and evidence files
- Supporting rollback for configuration artifacts
- Retaining backup history in a simple object store

> Versioning does not replace backups by itself. It improves recoverability, but you still need lifecycle rules, retention planning, and access controls.

**Enable encryption at rest**

S3 supports server-side encryption, which means AWS encrypts data before storing it and decrypts it when authorized clients read it. The most common options are:

- **SSE-S3**: S3-managed keys, simple default encryption
- **SSE-KMS**: AWS Key Management Service managed keys, more control and auditability
- **SSE-C**: Customer-provided keys, less common and operationally heavier

For most labs and many production workloads, default bucket encryption with SSE-S3 is the simplest starting point. If you need tighter key control, audit trails, or integration with key policies, use SSE-KMS.

Enable default encryption with SSE-S3:

```bash
aws s3api put-bucket-encryption \
  --bucket my-secure-lab-bucket-123456 \
  --server-side-encryption-configuration '{
    "Rules": [
      {
        "ApplyServerSideEncryptionByDefault": {
          "SSEAlgorithm": "AES256"
        }
      }
    ]
  }'
```

Enable default encryption with SSE-KMS:

```bash
aws s3api put-bucket-encryption \
  --bucket my-secure-lab-bucket-123456 \
  --server-side-encryption-configuration '{
    "Rules": [
      {
        "ApplyServerSideEncryptionByDefault": {
          "SSEAlgorithm": "aws:kms",
          "KMSMasterKeyID": "arn:aws:kms:us-west-2:123456789012:key/11111111-2222-3333-4444-555555555555"
        }
      }
    ]
  }'
```

Verify encryption:

```bash
aws s3api get-bucket-encryption \
  --bucket my-secure-lab-bucket-123456
```

Why encryption matters:

- **Compliance** requirements often mandate encryption at rest.
- **Defense in depth** reduces risk if storage media or snapshots are exposed.
- **Default encryption** removes reliance on every client remembering to encrypt uploads.
- **KMS integration** can provide audit logs and access control through key policies.

**SSE-S3 vs SSE-KMS**

Use this comparison when deciding which default encryption mode to apply.

| Feature | SSE-S3 | SSE-KMS |
|---|---|---|
| Key management | Managed by S3 | Managed in KMS |
| Operational complexity | Low | Moderate |
| Auditability | Basic | Stronger via KMS logs |
| Access control | Bucket/IAM policies | Bucket/IAM plus KMS key policy |
| Cost | No KMS request charges | KMS request charges may apply |

Practical guidance:

- Choose **SSE-S3** when you want simple encryption with minimal overhead.
- Choose **SSE-KMS** when you need granular control, key rotation policies, or audit requirements.
- Use **customer-managed KMS keys** when security teams need ownership over key lifecycle and permissions.

**Console workflow**

If you use the AWS Management Console, the workflow is straightforward:

1. Open the S3 console and choose **Create bucket**.
2. Enter a globally unique bucket name.
3. Select the AWS Region closest to your workload or required by policy.
4. Keep **Block all public access** enabled unless public access is intentional.
5. Create the bucket.
6. Open the bucket and enable **Versioning**.
7. Open **Properties** and configure **Default encryption**.
8. Confirm the bucket settings are saved.

The console is useful for visual verification, but the CLI is better for repeatable labs and exam-style automation.

**Verify object behavior**

After enabling versioning and encryption, upload a file and inspect its metadata.

Upload with the CLI:

```bash
aws s3 cp report.txt s3://my-secure-lab-bucket-123456/
```

List object versions:

```bash
aws s3api list-object-versions \
  --bucket my-secure-lab-bucket-123456
```

If versioning is enabled, you should see version IDs for the object. If you upload the same file again, S3 creates a new version rather than replacing the old one.

To confirm encryption on an object, inspect the object metadata:

```bash
aws s3api head-object \
  --bucket my-secure-lab-bucket-123456 \
  --key report.txt
```

Look for fields such as:

- `ServerSideEncryption`
- `SSEKMSKeyId` when using KMS
- `VersionId` when versioning is enabled

**Delete and restore workflow**

Versioning changes how deletion works. A delete request usually adds a delete marker instead of permanently removing the object.

Example:

```bash
aws s3 rm s3://my-secure-lab-bucket-123456/report.txt
```

If you list versions afterward, you may see:

- A **delete marker**
- One or more prior object versions

To restore an object, remove the delete marker or copy a previous version over the current key. This is one of the most important recovery patterns in S3.

**Lifecycle and cost control**

Versioning can increase storage usage because old versions remain stored. Lifecycle rules help you manage cost and retention.

Example lifecycle policy:

```json
{
  "Rules": [
    {
      "ID": "ExpireOldVersions",
      "Status": "Enabled",
      "Filter": {
        "Prefix": ""
      },
      "NoncurrentVersionExpiration": {
        "NoncurrentDays": 30
      }
    }
  ]
}
```

Apply it with the CLI:

```bash
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-secure-lab-bucket-123456 \
  --lifecycle-configuration file://lifecycle.json
```

Use lifecycle rules when:

- You need to keep recent versions but remove stale history
- You want to transition older data to cheaper storage classes
- You need automated retention without manual cleanup

**Security best practices**

A secure S3 bucket usually includes more than versioning and encryption.

- **Block Public Access** should stay enabled for private data.
- **Least privilege** IAM policies should restrict who can list, read, write, or delete.
- **Bucket policies** can enforce encryption headers or deny unencrypted uploads.
- **KMS key policies** must allow the right principals if you use SSE-KMS.
- **Access logging** or CloudTrail data events can help with auditing, depending on requirements.

Example bucket policy that denies uploads without encryption:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnencryptedObjectUploads",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-secure-lab-bucket-123456/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "AES256"
        }
      }
    }
  ]
}
```

If you use SSE-KMS, change the condition to require `aws:kms` and ensure the KMS key policy allows the uploader.

**Troubleshooting**

Common issues in this lab usually come from permissions, Region mismatches, or encryption settings.

- **Bucket already exists**: the name is globally unique, so choose a different name.
- **AccessDenied**: your IAM identity may lack `s3:CreateBucket`, `s3:PutBucketVersioning`, or `s3:PutEncryptionConfiguration`.
- **KMS permission errors**: the caller may lack `kms:Encrypt`, `kms:Decrypt`, or key policy access.
- **Wrong Region**: bucket creation and CLI commands must target the correct Region.
- **Versioning not visible immediately**: confirm the bucket settings and re-run `get-bucket-versioning`.

Useful verification commands:

```bash
aws s3api get-bucket-location --bucket my-secure-lab-bucket-123456
aws s3api get-bucket-versioning --bucket my-secure-lab-bucket-123456
aws s3api get-bucket-encryption --bucket my-secure-lab-bucket-123456
aws s3api get-public-access-block --bucket my-secure-lab-bucket-123456
```

**Exam focus**

For SAA-C03, know the following relationships:

- **Versioning** helps with recovery from accidental deletion and overwrite.
- **Default encryption** protects data at rest automatically for new objects.
- **SSE-S3** is the simplest managed encryption option.
- **SSE-KMS** adds key management and audit capabilities.
- **Lifecycle policies** control cost growth from noncurrent versions.
- **Block Public Access** is a standard safety control for private buckets.

A strong exam answer often combines these controls. For example, a secure application upload bucket may use versioning, SSE-KMS, a restrictive bucket policy, and lifecycle expiration for old versions.

**Quick reference**

- Create bucket: `aws s3api create-bucket`
- Enable versioning: `aws s3api put-bucket-versioning`
- Enable encryption: `aws s3api put-bucket-encryption`
- Verify settings: `get-bucket-versioning`, `get-bucket-encryption`
- Inspect object metadata: `aws s3api head-object`
- List versions: `aws s3api list-object-versions`

**Summary**

This lab establishes a secure S3 baseline. Versioning gives you recoverability, encryption protects stored data, and lifecycle rules keep storage costs under control. In real architectures, you usually combine these with access restrictions, logging, and public access blocking to build a production-ready bucket.
