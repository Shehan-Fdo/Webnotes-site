## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**EBS snapshot fundamentals**

Amazon Elastic Block Store (Amazon EBS) provides persistent block storage for Amazon EC2 instances. A **snapshot** is a point-in-time backup of an EBS volume stored in Amazon S3-managed infrastructure by AWS. You use snapshots to protect data, move volumes across Availability Zones, and restore a volume after deletion or corruption.

Snapshots are incremental after the first full backup. That means AWS stores only changed blocks after the initial snapshot, which reduces storage cost and speeds up subsequent backups. You still restore a full volume from any snapshot because AWS reconstructs the data behind the scenes.

- **Point-in-time** backups capture the volume state at the moment the snapshot starts.
- **Incremental** storage means only changed blocks are billed after the first snapshot.
- **Region-scoped** snapshots can be copied to another Region for disaster recovery.
- **Encrypted** volumes produce encrypted snapshots; encryption status is preserved on restore.

A snapshot is not a running volume. It is a backup artifact that you use to create a new EBS volume when needed.

**When to use snapshots**

Use snapshots when you need durable backup and recovery for EC2-attached data volumes. Common scenarios include:

- Recovering from accidental file deletion or application corruption.
- Replacing a failed volume without rebuilding data manually.
- Creating a clone of a production dataset for testing.
- Migrating data between Availability Zones or Regions.
- Capturing a pre-change backup before patching or schema updates.

Snapshots are especially useful when the workload is stateful and you need a recovery point objective (RPO) measured in minutes or hours rather than days.

**Lab workflow overview**

The lab typically follows this pattern:

1. Create or identify an EBS volume attached to an EC2 instance.
2. Write test data to the mounted filesystem.
3. Create a snapshot of the volume.
4. Modify or delete the data to simulate loss.
5. Create a new volume from the snapshot.
6. Attach the restored volume to an instance.
7. Mount the filesystem and verify the recovered data.

This workflow demonstrates that snapshots back up the block device, not just individual files. The restored volume contains the filesystem state from the snapshot time.

**Creating a snapshot from the AWS Management Console**

In the console, you usually create a snapshot from the EBS volume page.

- Open the **EC2** console.
- Select **Volumes** under **Elastic Block Store**.
- Choose the target volume.
- Select **Actions** > **Create snapshot**.
- Add a descriptive name and optional tags.
- Confirm creation.

Useful tagging practice:

- Tag snapshots with `Name`, `Environment`, `Application`, and `BackupType`.
- Use tags to support lifecycle automation and cost allocation.
- Keep naming consistent so you can identify restore points quickly.

A good snapshot description should explain why the backup exists, such as `pre-patch backup`, `before schema migration`, or `daily backup`.

**Creating a snapshot with the AWS CLI**

The AWS Command Line Interface (AWS CLI) is useful for repeatable lab work and automation.

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Lab snapshot before data deletion" \
  --tag-specifications 'ResourceType=snapshot,Tags=[{Key=Name,Value=lab-snapshot},{Key=Environment,Value=dev}]'
```

To verify the snapshot state:

```bash
aws ec2 describe-snapshots \
  --snapshot-ids snap-0123456789abcdef0
```

Important snapshot states:

- `pending` means AWS is still creating the snapshot.
- `completed` means the snapshot is ready to use.
- `error` indicates a failure and usually requires a retry or investigation.

You should wait for the snapshot to complete before deleting the source volume or making destructive changes.

**Preparing test data on the source volume**

Before snapshotting, write a file that you can later use to confirm recovery.

```bash
sudo mkdir -p /mnt/data
echo "snapshot test $(date)" | sudo tee /mnt/data/restore-check.txt
sync
```

The `sync` command helps flush buffered writes to disk so the snapshot captures the latest filesystem state more reliably. For application data, you should also stop or quiesce the application when consistency matters.

> A snapshot taken while a database is actively writing may be crash-consistent rather than application-consistent unless you coordinate with the application or use a backup tool that quiesces it.

**Consistency considerations**

EBS snapshots are generally **crash-consistent** by default. That means the volume is captured as if the instance lost power at that moment. For many filesystems, this is enough to recover cleanly, but it may not be sufficient for databases or transactional systems.

- **Crash-consistent** snapshots are suitable for many general-purpose workloads.
- **Application-consistent** snapshots require coordination with the application, such as pausing writes or using database-native backup tools.
- **Filesystem consistency** improves when you flush writes and unmount cleanly before snapshotting.

For Linux filesystems, unmounting the volume before snapshotting is the safest way to ensure consistency when the workload allows downtime.

**Restoring a volume from a snapshot**

To restore, you create a new EBS volume from the snapshot. The restored volume can be attached to an EC2 instance in the same Availability Zone as the instance that will use it.

Console workflow:

- Open **Snapshots** in the EC2 console.
- Select the snapshot.
- Choose **Actions** > **Create volume from snapshot**.
- Select the target **Availability Zone**.
- Choose volume type, size, and encryption settings if needed.
- Create the volume.

CLI example:

```bash
aws ec2 create-volume \
  --snapshot-id snap-0123456789abcdef0 \
  --availability-zone us-east-1a \
  --volume-type gp3
```

If you need a larger volume than the original, you can specify a larger size at creation time. You cannot create a smaller volume than the snapshot’s source data requires.

**Attaching the restored volume**

After the new volume is created, attach it to an instance.

```bash
aws ec2 attach-volume \
  --volume-id vol-0fedcba9876543210 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdf
```

On Nitro-based instances, Linux device names may appear as NVMe devices such as `/dev/nvme1n1` rather than the requested `/dev/sdf`. Always confirm the actual device mapping on the instance.

Check attached devices:

```bash
lsblk
sudo file -s /dev/nvme1n1
```

If the filesystem is present, mount it:

```bash
sudo mkdir -p /mnt/restored
sudo mount /dev/nvme1n1 /mnt/restored
cat /mnt/restored/restore-check.txt
```

**Filesystem and device verification**

After attaching a restored volume, verify the data before using it in production.

- Use `lsblk` to identify block devices and mount points.
- Use `blkid` to inspect filesystem UUIDs and types.
- Use `file -s` to confirm whether a device contains a filesystem.
- Use `df -h` to confirm mounted capacity and usage.

Example:

```bash
lsblk -f
blkid
df -h
```

If the restored volume contains a filesystem with the same UUID as another attached volume, avoid mounting both simultaneously if your operating system or application depends on unique identifiers. In some cases, you may need to update `/etc/fstab` or mount by device path carefully.

**Common restore issues**

Several issues can appear during restore labs:

- **Wrong Availability Zone**: EBS volumes can only be attached to instances in the same AZ.
- **Volume still in use**: Detach the source volume before reusing the device name or replacing it.
- **Filesystem not mounted**: Create a mount point and mount the correct device.
- **Permission issues**: Use `sudo` when accessing root-owned mount points or files.
- **Stale device mapping**: On Nitro instances, use `lsblk` and `nvme list` to identify the correct device.

If the restored volume is encrypted, the instance role and KMS (Key Management Service) permissions must allow use of the encryption key. Missing permissions can prevent attachment or access.

**Encryption and KMS**

EBS encryption protects data at rest. When you create a snapshot from an encrypted volume, the snapshot remains encrypted. When you restore a volume from that snapshot, the new volume is also encrypted.

Key points:

- Encryption is transparent to most applications.
- AWS manages encryption and decryption on the host side.
- The AWS KMS key must be available and permitted for the account and principal performing the restore.
- Copying encrypted snapshots across Regions may require re-encryption with a different KMS key.

Example encrypted volume creation from a snapshot:

```bash
aws ec2 create-volume \
  --snapshot-id snap-0123456789abcdef0 \
  --availability-zone us-east-1a \
  --encrypted \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ef-abcd-1234567890ab
```

**Snapshot copy and disaster recovery**

Snapshots are Region-specific, so cross-Region recovery requires copying the snapshot.

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --description "DR copy of lab snapshot"
```

Use snapshot copy when you need:

- Regional disaster recovery.
- Data locality in another Region.
- A backup copy isolated from the source Region.

For multi-account strategies, you can share snapshots or copy them into another account, depending on encryption and permissions. For encrypted snapshots, sharing and copying require careful KMS configuration.

**Automation patterns**

You can automate snapshot creation with scripts, cron, or AWS-native services.

Example shell script:

```bash
#!/usr/bin/env bash
set -euo pipefail

VOLUME_ID="vol-0123456789abcdef0"
DESC="nightly backup $(date +%F)"

SNAPSHOT_ID=$(aws ec2 create-snapshot \
  --volume-id "$VOLUME_ID" \
  --description "$DESC" \
  --query 'SnapshotId' \
  --output text)

aws ec2 create-tags \
  --resources "$SNAPSHOT_ID" \
  --tags Key=Name,Value=nightly-backup Key=Retention,Value=7d

echo "Created snapshot: $SNAPSHOT_ID"
```

For production, consider:

- AWS Backup for centralized backup policies.
- Lifecycle policies for snapshot retention.
- Event-driven automation for pre- and post-change backups.

**Retention and cleanup**

Snapshots incur storage charges for the blocks they retain. Old snapshots can accumulate and increase cost.

Best practices:

- Keep only the recovery points you need.
- Use tags to identify retention class.
- Automate deletion of expired snapshots.
- Validate restore procedures regularly so backups are not just stored but usable.

Deleting a snapshot does not affect existing volumes created from it. However, if a volume depends on a snapshot for future recovery, deleting that snapshot removes that recovery point.

**Exam-relevant takeaways**

For SAA-C03, focus on these concepts:

- EBS snapshots are durable backups of EBS volumes.
- Snapshots are incremental after the first full snapshot.
- You restore by creating a new volume from a snapshot.
- The restored volume must be in the same Availability Zone as the target instance.
- Encryption and KMS permissions matter for encrypted snapshots.
- Snapshots are useful for backup, recovery, cloning, and migration.

**Quick reference lab commands**

```bash
# Create snapshot
aws ec2 create-snapshot --volume-id vol-0123456789abcdef0 --description "lab backup"

# Check snapshot status
aws ec2 describe-snapshots --snapshot-ids snap-0123456789abcdef0

# Create volume from snapshot
aws ec2 create-volume --snapshot-id snap-0123456789abcdef0 --availability-zone us-east-1a

# Attach volume
aws ec2 attach-volume --volume-id vol-0fedcba9876543210 --instance-id i-0123456789abcdef0 --device /dev/sdf

# Inspect devices
lsblk
sudo file -s /dev/nvme1n1

# Mount restored filesystem
sudo mount /dev/nvme1n1 /mnt/restored
```

**Practical mental model**

Think of an EBS snapshot as a backup checkpoint for a block device. You do not “open” the snapshot directly; instead, you create a new volume from it, attach that volume to an instance, and mount the filesystem. This distinction is important in labs and on the exam because AWS often tests whether you understand the difference between backup storage, block storage, and instance attachment constraints.
