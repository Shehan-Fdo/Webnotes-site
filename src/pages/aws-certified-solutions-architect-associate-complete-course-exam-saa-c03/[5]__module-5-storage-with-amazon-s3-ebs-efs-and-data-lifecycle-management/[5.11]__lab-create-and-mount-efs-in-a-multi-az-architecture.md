## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon EFS in a multi-AZ architecture**

Amazon Elastic File System (<abbr title="Elastic File System">EFS</abbr>) is a managed, elastic, NFS-based file system for Linux workloads. In a multi-AZ design, you mount the same file system from instances in different Availability Zones so each instance sees the same shared data. This is a common pattern for web tiers, content repositories, shared application assets, and clustered workloads that need concurrent access.

EFS is regional, not zonal. AWS creates mount targets in subnets, and each mount target provides a local network path for instances in that Availability Zone. That design reduces cross-AZ traffic and improves availability if one AZ has issues.

- **Shared storage** lets multiple EC2 instances read and write the same files.
- **Regional durability** means the file system is replicated across multiple AZs by AWS.
- **Mount targets** are required in each AZ where you want low-latency access.
- **Security groups** control NFS access on port `2049`.

**Core concepts and when to use EFS**

Use EFS when you need a POSIX-style file system with concurrent access from many Linux instances. It is a better fit than Amazon EBS when storage must be shared, and a better fit than Amazon S3 when applications expect a mounted file system rather than object storage.

<dl>
<dt><strong>POSIX</strong></dt>
<dd>Portable Operating System Interface; a standard for file-system behavior such as permissions, directories, and file locking semantics.</dd>
<dt><strong>NFS</strong></dt>
<dd>Network File System; the protocol EFS uses for mounting storage over the network.</dd>
</dl>

Typical use cases include:

- Shared web content for Auto Scaling groups
- User home directories
- Centralized application uploads
- Shared build artifacts or media assets
- Container workloads that need a common file system

Avoid EFS when:

- You need block storage attached to a single instance; use EBS instead.
- You need object storage, lifecycle policies, or event-driven object access; use S3 instead.
- You need very low-latency local disk semantics for a single host; use instance store or EBS.

**Architecture pattern for a multi-AZ lab**

A standard lab topology includes:

- A VPC with at least two private subnets in different AZs
- Two EC2 instances, one in each subnet
- One EFS file system
- One mount target in each subnet or AZ
- Security groups allowing NFS from the EC2 instances to EFS

The instances mount the same file system path, such as `/mnt/efs`, and any file created on one instance appears on the other after synchronization through the shared file system.

```text
VPC
├── AZ-a
│   ├── Private Subnet A
│   ├── EC2 Instance A
│   └── EFS Mount Target A
└── AZ-b
    ├── Private Subnet B
    ├── EC2 Instance B
    └── EFS Mount Target B
```

**Create the EFS file system**

In the AWS Management Console, you typically:

1. Open the EFS console and create a file system.
2. Choose the VPC that contains your EC2 instances.
3. Select the Availability Zones and subnets where you want mount targets.
4. Attach a security group that allows inbound NFS from your application instances.
5. Accept or configure encryption, performance mode, throughput mode, and lifecycle settings.

For most lab scenarios, the default settings are sufficient unless the exercise specifies otherwise.

Important configuration choices:

- **Performance mode**
  - General Purpose is the default and suits most workloads.
  - Max I/O is for highly parallel workloads, but it adds latency.
- **Throughput mode**
  - Bursting Throughput is common for variable workloads.
  - Provisioned Throughput is useful when you need predictable performance.
- **Encryption**
  - Enable encryption at rest for security best practice.
  - EFS also supports encryption in transit when mounted with TLS.
- **Lifecycle management**
  - Can transition infrequently accessed files to lower-cost storage classes.
  - Useful for cost optimization, but not always needed in a lab.

**Security groups and network access**

EFS uses NFS over TCP port `2049`. The file system’s security group must allow inbound traffic from the EC2 instances’ security group or from the instance subnet CIDR if you are using broader rules.

A common and safer pattern is security-group referencing:

```text
EFS-SG
Inbound:
- TCP 2049 from EC2-SG
Outbound:
- All traffic allowed
```

On the EC2 side, the instance security group usually allows outbound traffic to the EFS security group. In most default configurations, outbound is already open.

> If the security group does not allow NFS on port `2049`, the mount will fail even if DNS and routing are correct.

**Mounting EFS on Linux instances**

Before mounting, install the EFS mount helper if needed. On Amazon Linux 2 or Amazon Linux 2023, the package is typically available through the package manager.

```bash
sudo yum install -y amazon-efs-utils
# or on newer distributions
sudo dnf install -y amazon-efs-utils
```

Create a mount point and mount the file system using the DNS name or the mount helper.

```bash
sudo mkdir -p /mnt/efs
sudo mount -t efs fs-1234567890abcdef0:/ /mnt/efs
```

If you want TLS encryption in transit, use the `tls` option:

```bash
sudo mount -t efs -o tls fs-1234567890abcdef0:/ /mnt/efs
```

You can also mount directly with NFSv4.1 if you prefer the native client:

```bash
sudo mount -t nfs4 -o nfsvers=4.1 fs-1234567890abcdef0.efs.us-east-1.amazonaws.com:/ /mnt/efs
```

Practical verification commands:

```bash
df -hT | grep efs
mount | grep /mnt/efs
touch /mnt/efs/test-file-from-instance-a
ls -l /mnt/efs
```

**Make the mount persistent**

To remount after reboot, add an entry to `/etc/fstab`. The EFS mount helper is usually the easiest option because it handles DNS and TLS more cleanly.

```fstab
fs-1234567890abcdef0:/ /mnt/efs efs _netdev,tls 0 0
```

Useful options:

- `_netdev` delays mounting until the network is available
- `tls` enables encryption in transit
- `nofail` prevents boot failure if the mount is temporarily unavailable

Example with a safer boot behavior:

```fstab
fs-1234567890abcdef0:/ /mnt/efs efs _netdev,tls,nofail 0 0
```

After editing `fstab`, test it carefully:

```bash
sudo umount /mnt/efs
sudo mount -a
```

**Validate multi-AZ sharing**

A simple validation workflow is to write a file from one instance and read it from another.

1. On Instance A, create a file:
   ```bash
   echo "hello from az-a" | sudo tee /mnt/efs/shared.txt
   ```
2. On Instance B, confirm the file exists:
   ```bash
   cat /mnt/efs/shared.txt
   ```
3. Update the file from Instance B:
   ```bash
   echo "updated from az-b" | sudo tee /mnt/efs/shared.txt
   ```
4. Recheck from Instance A:
   ```bash
   cat /mnt/efs/shared.txt
   ```

This confirms that both instances are using the same regional file system rather than separate local disks.

**Troubleshooting common issues**

Mount failures usually come from networking, permissions, or DNS.

- **Security group mismatch**
  - Ensure inbound TCP `2049` is allowed to the EFS mount target.
  - Confirm the EC2 instance can reach the mount target in its AZ.
- **Missing mount target**
  - If there is no mount target in the instance’s AZ, the instance may use a cross-AZ path or fail depending on routing and DNS behavior.
  - Best practice is one mount target per AZ used by clients.
- **DNS resolution problems**
  - EFS mount helpers rely on DNS.
  - Verify the VPC has DNS resolution and DNS hostnames enabled.
- **IAM or access point restrictions**
  - If using EFS access points, ensure the client mounts the correct access point and the permissions match.
- **File permissions**
  - Linux file ownership and mode bits still apply.
  - Use `chown`, `chmod`, or access points to control access.

Useful diagnostics:

```bash
nslookup fs-1234567890abcdef0.efs.us-east-1.amazonaws.com
showmount -e fs-1234567890abcdef0.efs.us-east-1.amazonaws.com
sudo mount -v -t efs fs-1234567890abcdef0:/ /mnt/efs
dmesg | tail
```

**EFS access points**

Access points provide application-specific entry points into a file system. They are useful when different applications or tenants need isolated directories or enforced POSIX identities.

Benefits:

- Simplify application mounts
- Enforce a root directory
- Apply a specific user and group identity
- Reduce accidental access to unrelated paths

Example mount with an access point:

```bash
sudo mount -t efs -o tls,accesspoint=fsap-0123456789abcdef0 fs-1234567890abcdef0:/ /mnt/efs
```

Use access points when you want to standardize permissions for a workload, especially in containerized environments.

**Exam-relevant distinctions**

EFS is often tested against EBS and S3.

| Service | Storage type | Shared across EC2 instances | Typical use |
|---|---|---:|---|
| EBS | Block | No, attached to one instance at a time | Boot volumes, databases, single-host storage |
| EFS | File | Yes | Shared Linux file system, multi-AZ access |
| S3 | Object | Yes, but not mounted as a native file system | Static content, backups, data lakes |

Key exam cues:

- If the requirement says “shared file system across multiple AZs,” think **EFS**.
- If it says “single instance block storage,” think **EBS**.
- If it says “store objects, scale massively, and access via API,” think **S3**.

**Operational best practices**

- Place mount targets in every AZ where clients run.
- Use security groups instead of broad CIDR rules when possible.
- Enable encryption at rest and in transit for sensitive data.
- Use access points for application-specific directories and permissions.
- Monitor throughput and burst credit usage for performance-sensitive workloads.
- Keep application paths consistent, such as `/mnt/efs`, across instances.

A minimal automation example for Linux bootstrap:

```bash
#!/bin/bash
set -e

yum install -y amazon-efs-utils
mkdir -p /mnt/efs
echo "fs-1234567890abcdef0:/ /mnt/efs efs _netdev,tls,nofail 0 0" >> /etc/fstab
mount -a
```

This pattern is useful in user data scripts, launch templates, or configuration management tools so every instance mounts the same shared storage automatically.

**Summary of what the lab demonstrates**

This lab shows how to build a highly available shared storage layer with Amazon EFS in a multi-AZ architecture. You create one regional file system, add mount targets in multiple AZs, allow NFS traffic through security groups, and mount the same file system on multiple EC2 instances. The result is a resilient shared directory that supports concurrent access and survives instance replacement or AZ-level placement changes.
