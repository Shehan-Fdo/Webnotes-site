## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Static website hosting on Amazon S3**

Amazon Simple Storage Service (Amazon S3) can serve static content directly, which makes it a common choice for HTML, CSS, JavaScript, images, and other assets that do not require server-side processing. In a static website setup, S3 acts as the origin, and users access the site through the S3 website endpoint rather than the REST API endpoint.

This pattern is useful when you want low operational overhead, high durability, and simple scaling. It is also a common exam scenario because it combines storage, permissions, routing, and public access controls.

- **Static content** includes files such as `index.html`, `error.html`, CSS, JavaScript, and images.
- **Website endpoint** is different from the S3 REST endpoint; the website endpoint supports browser-friendly behavior such as index and error documents.
- **Public access** must be intentionally configured if the site is meant to be reachable from the internet.

**Core S3 website hosting concepts**

S3 static website hosting requires a bucket, website configuration, and objects uploaded with the correct names and permissions. The bucket name should usually match the domain name if you plan to use a custom DNS record, especially for root-domain hosting.

A typical setup includes:
- An S3 bucket containing website files.
- Bucket website hosting enabled.
- Public read access for the objects, or a policy that allows anonymous `s3:GetObject`.
- An `index.html` document and optionally an `error.html` document.

> S3 website endpoints are HTTP-only. If you need HTTPS for a custom domain, place CloudFront in front of the bucket and use an ACM (AWS Certificate Manager) certificate there.

**Bucket naming and object layout**

For a simple lab, you usually upload a small site structure like this:

```text
index.html
error.html
css/
  styles.css
images/
  logo.png
```

If you are using a custom domain, the bucket name often matches the domain exactly, such as `www.example.com`. This is especially important when you want to point a DNS record directly at the S3 website endpoint.

- **Root object** is the default file returned when a user opens the site.
- **Error document** is shown when the requested page is missing or access fails in a website-hosting context.
- **Object keys** should reflect the paths referenced by your HTML files.

**Typical lab workflow**

A static website hosting lab usually follows a predictable sequence. You create the bucket, upload content, enable website hosting, and then test the endpoint in a browser.

1. Create an S3 bucket.
2. Upload website files.
3. Enable static website hosting.
4. Configure public access so anonymous users can read the content.
5. Open the website endpoint and verify the page loads.
6. Test the error page by requesting a missing object.

**Example AWS CLI workflow**

You can complete most of the setup with the AWS CLI (Command Line Interface). The exact bucket name must be globally unique.

```bash
aws s3 mb s3://my-static-site-example-12345
aws s3 cp ./site/ s3://my-static-site-example-12345/ --recursive
aws s3 website s3://my-static-site-example-12345/ \
  --index-document index.html \
  --error-document error.html
```

To make the site publicly readable, you typically need a bucket policy similar to the following:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadForWebsiteContent",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-static-site-example-12345/*"
    }
  ]
}
```

If Block Public Access is enabled at the bucket or account level, the policy alone may not be enough. You must review the public access settings before the website becomes reachable.

**Website hosting configuration**

In the S3 console, the website hosting settings define the documents that S3 returns for the root path and for errors. The index document is required for a basic site.

Common settings:
- **Index document**: usually `index.html`
- **Error document**: usually `error.html`
- **Redirect rules**: optional, used for path-based redirects or legacy URL handling

A minimal website configuration looks like this:

```json
{
  "IndexDocument": {
    "Suffix": "index.html"
  },
  "ErrorDocument": {
    "Key": "error.html"
  }
}
```

The website endpoint is generated by S3 and is region-specific. It usually looks like:

```text
http://my-static-site-example-12345.s3-website-us-east-1.amazonaws.com
```

**Public access and bucket policy**

Static website hosting requires anonymous access to the objects if the site is public. The most common approach is a bucket policy that grants `s3:GetObject` to everyone for the bucket contents.

Important considerations:
- **Block Public Access** can override public bucket policies.
- **Bucket policy** is the preferred way to grant public read access for website content.
- **ACLs** (Access Control Lists) are generally discouraged for new designs; bucket policies are clearer and easier to manage.

If you use the AWS Management Console, you may need to disable the relevant Block Public Access settings for the bucket. In exam scenarios, always check whether public access is being denied by a bucket-level or account-level control.

> Do not confuse public read access for website content with public write access. The site should allow anonymous reads only.

**Testing the site**

After configuration, validate the site in a browser or with `curl`. This helps confirm both the website endpoint and the error document behavior.

```bash
curl -I http://my-static-site-example-12345.s3-website-us-east-1.amazonaws.com
curl http://my-static-site-example-12345.s3-website-us-east-1.amazonaws.com/missing-page.html
```

What to verify:
- The root URL returns the `index.html` content.
- A missing path returns the configured error page.
- The response is accessible without AWS credentials.
- The endpoint is the website endpoint, not the REST endpoint.

**Common troubleshooting issues**

Static website hosting problems usually come from permissions, endpoint confusion, or incorrect object names.

- **AccessDenied** often means the bucket policy does not allow public `s3:GetObject`, or Block Public Access is still enabled.
- **404 Not Found** often means the object key is wrong, the file was not uploaded, or the index document name does not match the configured value.
- **Wrong endpoint** often means you are using the S3 REST endpoint instead of the website endpoint.
- **HTTPS failure** is expected if you try to use the S3 website endpoint directly over HTTPS, because the website endpoint supports HTTP only.

**When to use CloudFront**

If you need HTTPS, custom headers, caching, or a custom domain with TLS, place Amazon CloudFront in front of the S3 bucket. CloudFront can serve the site over HTTPS and use an ACM certificate for the alternate domain name.

A common architecture is:
- Users access `https://www.example.com`
- DNS points to CloudFront
- CloudFront fetches content from the S3 bucket
- The S3 bucket remains private, with access controlled through an origin access mechanism

This is often the preferred production pattern because it improves security and performance compared with exposing the S3 website endpoint directly.

**Exam-relevant distinctions**

SAA-C03 questions often test whether you can choose the right service and configuration for a static site.

| Requirement | Best fit | Notes |
|---|---|---|
| Simple static website | S3 website hosting | Lowest complexity |
| Public HTTP website | S3 website endpoint | Direct browser access |
| HTTPS with custom domain | CloudFront + S3 | S3 website endpoint alone does not provide HTTPS |
| Private origin with public site | CloudFront + private S3 bucket | Better security posture |
| Dynamic server-side logic | Not S3 static hosting | Use a compute service instead |

**Practical example: simple HTML site**

A minimal `index.html` file might look like this:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8" />
    <title>Static Site</title>
  </head>
  <body>
    <h1>Welcome to my S3 static website</h1>
  </body>
</html>
```

A simple `error.html` file can provide a friendlier message:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8" />
    <title>Page Not Found</title>
  </head>
  <body>
    <h1>Sorry, that page does not exist.</h1>
  </body>
</html>
```

**Operational best practices**

Even in a lab, it helps to follow production-minded habits.

- Use versioned object names for assets when you expect frequent updates.
- Keep the bucket policy as narrow as possible while still allowing anonymous reads.
- Prefer CloudFront for internet-facing production sites that need HTTPS.
- Test both the root path and a missing path after every change.
- Keep the bucket name aligned with the intended DNS name when using direct website hosting.

**Quick mental model**

Think of S3 static website hosting as three layers:
- **Storage**: the bucket and objects
- **Website behavior**: index and error documents
- **Access**: public read permissions and public access settings

If any one of these layers is misconfigured, the site will not behave as expected.

**Key takeaways**

- S3 can host static websites directly.
- The website endpoint is not the same as the REST endpoint.
- Public read access is required for a public site.
- Block Public Access can prevent the site from working.
- S3 website endpoints are HTTP-only; use CloudFront for HTTPS.
- Correct object names and index/error document settings are essential.
