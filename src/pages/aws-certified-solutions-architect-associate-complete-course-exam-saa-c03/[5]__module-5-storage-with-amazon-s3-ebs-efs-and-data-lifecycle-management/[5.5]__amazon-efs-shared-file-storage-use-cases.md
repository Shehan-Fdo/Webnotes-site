## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon EFS overview**

Amazon Elastic File System (EFS) is a managed, elastic, shared file system for Linux workloads in AWS. You use it when multiple instances or containers need concurrent access to the same files with standard file-system semantics. EFS grows and shrinks automatically, so you do not pre-provision capacity the way you do with block storage.

EFS is a strong fit when you need:
- Shared content across many EC2 instances
- A POSIX-compliant file system for Linux applications
- Persistent storage that survives instance replacement
- Simple scaling without manual volume management

EFS is not the same as object storage or block storage:
- **EBS** is attached to one instance at a time for low-latency block access.
- **S3** is object storage, not a mounted file system.
- **EFS** is a network file system that many clients can mount simultaneously.

**Core use cases**

Use EFS when your application expects a shared directory tree and multiple compute nodes must read and write the same data.

Common exam-friendly scenarios include:
- Web server fleets sharing application assets
- Content management systems with shared uploads
- Home directories for Linux users
- Container workloads that need shared persistent storage
- Analytics or batch jobs that coordinate through a shared file system

A practical topology often looks like this:

```text
EC2 Auto Scaling Group
   |      |      |
   |      |      +--> Mount EFS in multiple AZs
   |      +----------> Mount EFS in multiple AZs
   +------------------> Mount EFS in multiple AZs
```

Because EFS is regional, you can mount it from instances in multiple Availability Zones within the same Region. That makes it useful for highly available architectures where compute nodes come and go.

**Deployment modes and file system types**

EFS offers two deployment options that matter in exam questions:

- **Regional file systems**
  - Store data redundantly across multiple Availability Zones.
  - Best for production workloads that need high availability and durability.
  - Support mounting from multiple AZs in the same Region.
- **One Zone file systems**
  - Store data in a single Availability Zone.
  - Lower cost than Regional.
  - Useful for dev/test, secondary copies, or workloads that can tolerate AZ-level risk.

Choose Regional when availability matters more than cost. Choose One Zone when the workload is less critical or already protected elsewhere.

> If a question emphasizes multi-AZ resilience, do not choose One Zone.

**Performance modes and throughput**

EFS has performance and throughput choices that affect how it behaves under load.

- **General Purpose performance mode**
  - Lowest latency.
  - Best for most workloads.
  - Typical choice for web serving, home directories, and general application storage.
- **Max I/O performance mode**
  - Designed for highly parallel workloads with many clients.
  - Higher latency than General Purpose.
  - Useful when many instances or containers access the file system concurrently at scale.

Throughput options control how much data EFS can deliver.

- **Bursting Throughput**
  - Throughput scales with the size of the file system.
  - Good default for many workloads.
  - Works well when usage is variable and storage size grows over time.
- **Provisioned Throughput**
  - You set throughput independently of stored data size.
  - Useful when you need high throughput but do not store enough data to earn it through bursting.
  - Common for workloads with sustained read/write demand.

A simple decision rule:
- Small or moderate shared file systems: use General Purpose + Bursting.
- Very high concurrency: consider Max I/O.
- Throughput-heavy workloads with small stored data: consider Provisioned Throughput.

**Access points and application-specific access**

EFS access points simplify how applications reach a shared file system. An access point is an application-specific entry point that can:
- Enforce a root directory for the application
- Apply a specific POSIX user and group identity
- Reduce the need to manage permissions inside the application itself

This is especially useful when multiple applications share one EFS file system but must remain isolated from each other.

Example use case:
- `/app1` should only see its own directory
- `/app2` should only see its own directory
- Each app should run as a different Linux user

An access point can map each app to a different root path and identity, which reduces permission mistakes and simplifies deployment.

Example mount command:

```bash
sudo mount -t efs -o tls,accesspoint=fsap-0123456789abcdef0 fs-12345678:/ /mnt/efs
```

In practice, you often combine access points with IAM and POSIX permissions so the application gets only the access it needs.

**Security model and permissions**

EFS security is layered. Exam questions often test whether you understand the difference between network access, IAM authorization, and file-system permissions.

- **Security groups**
  - Control network traffic to the mount target.
  - EFS mount targets use NFS over TCP port 2049.
- **IAM client permissions**
  - Control whether a principal can mount or write to the file system.
  - Useful when you want identity-based access control.
- **File system policies**
  - Resource-based policies attached to the EFS file system.
  - Can allow or deny actions for specific principals, accounts, or conditions.
  - Useful for cross-account access or centralized control.
- **POSIX permissions**
  - Standard Linux file ownership and mode bits.
  - Still matter after the mount succeeds.

A common pattern is:
1. Security group allows NFS traffic.
2. IAM or file system policy authorizes the mount.
3. POSIX permissions control file-level access.

Example file system policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowMountFromAppRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/AppServerRole"
      },
      "Action": [
        "elasticfilesystem:ClientMount",
        "elasticfilesystem:ClientWrite"
      ],
      "Resource": "arn:aws:elasticfilesystem:us-east-1:123456789012:file-system/fs-12345678"
    }
  ]
}
```

Use encrypted EFS when data protection is required. EFS supports encryption at rest and in transit. For in-transit encryption, mount with TLS.

```bash
sudo mount -t efs -o tls fs-12345678:/ /mnt/efs
```

**Linux clients and Windows distinctions**

EFS is typically mounted by Linux clients and is the standard AWS choice for shared POSIX file storage. It is not the right default for Windows workloads.

Use EFS when:
- Your workload runs on Linux
- You need NFS-style shared storage
- You need POSIX semantics

Do not choose EFS when:
- You need native Windows file sharing
- You need SMB (Server Message Block) semantics
- Your workload expects Windows ACLs and Windows-integrated file sharing

For Windows shared file storage, the exam usually points you toward **Amazon FSx for Windows File Server**. That service provides SMB access and integrates with Windows features such as Active Directory.

**EFS versus FSx options**

AWS offers several FSx services, and the right choice depends on the workload.

| Service | Best for | Protocol / behavior | Exam clue |
|---|---|---|---|
| Amazon EFS | Shared Linux file storage | NFS, POSIX, elastic | Multiple Linux instances need shared mounted storage |
| Amazon FSx for Windows File Server | Windows shared file storage | SMB, Windows ACLs | Windows apps, Active Directory, shared drives |
| Amazon FSx for Lustre | High-performance compute workloads | Lustre, HPC-oriented | Machine learning, HPC, scratch or fast parallel access |
| Amazon FSx for NetApp ONTAP | Enterprise NAS features | NFS/SMB, snapshots, replication | Need advanced data management and hybrid NAS features |

A useful exam shortcut:
- Linux shared mount: **EFS**
- Windows shared drive: **FSx for Windows File Server**
- HPC or ML scratch storage: **FSx for Lustre**
- Enterprise NAS with rich storage features: **FSx for NetApp ONTAP**

**EFS versus EBS versus S3**

You often need to choose among EFS, EBS, and S3 in architecture questions.

| Service | Storage type | Access pattern | Typical use |
|---|---|---|---|
| EBS | Block | Attached to one instance at a time | Boot volumes, databases, low-latency single-instance storage |
| EFS | File | Shared by many instances | Shared application data, Linux home directories |
| S3 | Object | HTTP/API access | Static assets, backups, data lakes, archival content |

Decision guidance:
- Choose **EBS** when one instance needs fast block storage.
- Choose **EFS** when many instances need the same mounted file system.
- Choose **S3** when you need object storage, durability, and simple retrieval rather than a mounted file system.

Example exam scenario:
- A company runs a fleet of Linux web servers behind an Application Load Balancer.
- Each server must read and write uploaded files.
- Instances scale in and out automatically.
- The storage must remain available when instances terminate.

Best answer: **Amazon EFS**.

Another scenario:
- A single database server needs low-latency attached storage.

Best answer: **Amazon EBS**.

Another scenario:
- A static website stores images and documents for download.

Best answer: **Amazon S3**.

**Practical mount and configuration examples**

Install the EFS mount helper on Amazon Linux or compatible distributions, then mount the file system.

```bash
sudo yum install -y amazon-efs-utils
sudo mkdir -p /mnt/efs
sudo mount -t efs -o tls fs-12345678:/ /mnt/efs
```

For `/etc/fstab`, use a persistent mount entry:

```fstab
fs-12345678:/ /mnt/efs efs _netdev,tls 0 0
```

If you use an access point:

```fstab
fs-12345678:/ /mnt/efs efs _netdev,tls,accesspoint=fsap-0123456789abcdef0 0 0
```

A container example with shared storage might mount EFS into multiple tasks or pods through the platform’s storage integration. The key idea is the same: multiple compute nodes can share one persistent file system.

**Exam decision patterns**

When you read a scenario, look for these clues:

- **Multiple Linux instances need shared files**
  - Choose EFS.
- **Windows file sharing or SMB**
  - Choose FSx for Windows File Server.
- **Single-instance low-latency disk**
  - Choose EBS.
- **Static content or backup archive**
  - Choose S3.
- **Need application-specific directories and simplified permissions**
  - Use EFS access points.
- **Need multi-AZ resilience**
  - Prefer Regional EFS.
- **Need lower cost and can tolerate single-AZ risk**
  - Consider One Zone EFS.
- **Need many concurrent clients**
  - Consider Max I/O.
- **Need throughput independent of file system size**
  - Consider Provisioned Throughput.

**Operational considerations**

EFS is managed, so you do not manage servers or storage nodes. AWS handles scaling, availability, and maintenance of the file system service. That reduces operational overhead compared with self-managed NFS servers on EC2.

Still, you should think about:
- Mount target placement in each AZ where clients run
- Security group rules for NFS
- IAM and file system policy alignment
- POSIX ownership and permissions
- Performance mode and throughput selection

A well-designed EFS deployment usually places mount targets in the same AZs as the compute fleet to reduce cross-AZ traffic and improve resilience.

**Quick reference**

- **EFS**: shared Linux file storage, NFS, elastic, multi-AZ with Regional mode
- **One Zone EFS**: lower-cost single-AZ option
- **Access points**: simplify app-specific access and root directory control
- **General Purpose**: default low-latency mode
- **Max I/O**: high-concurrency mode
- **Bursting**: throughput scales with stored data
- **Provisioned**: fixed throughput independent of size
- **FSx for Windows File Server**: Windows SMB shared storage
- **EBS**: single-instance block storage
- **S3**: object storage, not mounted

**Key takeaways**

EFS is the right answer when the problem is shared, elastic, Linux-based file storage across multiple instances. The most common exam traps are confusing it with EBS, S3, or FSx for Windows File Server. Pay close attention to the operating system, access pattern, availability needs, and whether the workload needs a mounted file system or object storage.
