## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Durability, availability, and backup: the core distinction**

AWS exam questions often mix these ideas, so you need to separate them clearly:

- **Durability** means data is unlikely to be lost once stored.
- **Availability** means a service stays reachable and usable.
- **Backup** means you can restore data after deletion, corruption, or a bad change.

A system can be highly available without being a backup solution. For example, an RDS Multi-AZ deployment improves failover behavior, but it does not replace backups for accidental deletes or logical corruption. Likewise, replication can improve recovery speed, but it can also replicate mistakes.

**Key recovery terms**

- **RPO** (Recovery Point Objective): how much data loss you can tolerate, measured in time.
- **RTO** (Recovery Time Objective): how long recovery can take before the business is impacted.
- **PITR** (Point-in-Time Recovery): restore to a specific time within a retention window.
- **WORM** (Write Once, Read Many): immutable retention model used to resist deletion or tampering.

A good exam answer usually matches the recovery objective to the mechanism:
- Very low RPO and low RTO often point to synchronous or near-synchronous replication.
- Moderate RPO with restore capability often points to snapshots or backups.
- Compliance and ransomware resistance often point to immutability controls such as object lock or vault lock.

**Backup versus replication versus high availability**

These are related but not interchangeable.

- **Backup**
  - Creates a recoverable copy for later restore.
  - Protects against accidental deletion, corruption, and ransomware.
  - Usually restores to a new resource or a previous state.

- **Replication**
  - Copies data to another location or resource.
  - Improves resilience and can reduce recovery time.
  - Can replicate bad writes, deletes, or corruption if not designed carefully.

- **High availability**
  - Keeps the service running through component failure.
  - Uses redundancy, failover, and health checks.
  - Does not automatically protect against logical data loss.

A common exam trap is assuming that Multi-AZ, read replicas, or cross-Region replication are backups. They are not backups by themselves.

**AWS storage durability basics**

AWS storage services are designed for high durability, but the recovery model differs by service.

- **Amazon S3**
  - Designed for very high durability.
  - Stores objects redundantly across multiple facilities within a Region.
  - Supports versioning, replication, and Object Lock for protection against deletion and overwrite.
- **Amazon EBS**
  - Provides block storage for EC2 instances.
  - Volumes are replicated within an Availability Zone for durability.
  - Snapshots are the main backup mechanism.
- **Amazon EFS**
  - Managed file system for Linux workloads.
  - Highly available within a Region.
  - Can be backed up with AWS Backup or service-native mechanisms.
- **Amazon FSx**
  - Managed file systems with service-specific backup and restore behavior.
  - Backup support depends on the FSx family, but backups are a standard recovery tool.
- **Amazon RDS and Aurora**
  - Managed databases with automated backups, snapshots, and Multi-AZ options.
  - Recovery options depend on engine and deployment model.
- **Amazon DynamoDB**
  - Supports on-demand backups, point-in-time recovery, and global tables for replication.

**Amazon S3 protection patterns**

S3 is often the best answer for durable object storage, but you still need protection against accidental or malicious deletion.

- **Versioning**
  - Keeps multiple versions of an object.
  - Helps recover from overwrite and delete operations.
  - Is required on both source and destination buckets for replication.

- **S3 Object Lock**
  - Enforces retention using WORM controls.
  - Helps protect backups and compliance data from deletion or modification.
  - Supports governance mode and compliance mode.

- **Replication**
  - Copies new objects from a source bucket to a destination bucket.
  - Can replicate delete markers if configured.
  - Does not automatically copy all historical objects unless you backfill or replicate existing objects with the appropriate setup.
  - Requires correct IAM permissions, versioning, and replication configuration.

- **Cross-Region Replication**
  - Helps with disaster recovery and regional isolation.
  - Is asynchronous, so it is not zero-RPO.
  - Is useful when you need a secondary copy in another Region.

Practical example for versioning and replication:

```bash
aws s3api put-bucket-versioning \
  --bucket source-bucket \
  --versioning-configuration Status=Enabled

aws s3api put-bucket-versioning \
  --bucket destination-bucket \
  --versioning-configuration Status=Enabled
```

> Replication is not a backup strategy by itself if the source bucket is compromised and the same bad change is replicated to the destination.

**S3 Object Lock and ransomware resistance**

Object Lock is a key exam concept when you need immutable retention.

- Use it for backup archives, audit logs, and regulated datasets.
- Use **governance mode** when administrators may need controlled override.
- Use **compliance mode** when retention must be enforced even against privileged users.
- Combine it with versioning and restricted delete permissions.

A common pattern is:
- Application writes to S3.
- Lifecycle or backup process copies data to a locked bucket.
- Retention prevents deletion for a defined period.

**Amazon EBS snapshots and restore behavior**

EBS snapshots are incremental backups of EBS volumes. They are stored in AWS-managed snapshot storage and are not customer-visible S3 objects.

- **Incremental nature**
  - The first snapshot captures all used blocks.
  - Later snapshots store only changed blocks.
  - You can delete older snapshots after newer ones exist, but restore chains still depend on retained data.

- **Restore**
  - You restore by creating a new EBS volume from a snapshot.
  - You can then attach the volume to an EC2 instance.
  - Restore time depends on volume size and how much data must be initialized.

- **Encryption**
  - Encrypted volumes produce encrypted snapshots.
  - Restoring an encrypted snapshot keeps encryption in place.
  - You need the right AWS Key Management Service (AWS KMS) permissions to copy or restore encrypted snapshots across accounts or Regions.

- **Cross-Region copy**
  - Useful for disaster recovery.
  - Often paired with backup automation or AWS Backup.
  - Requires careful key management if the snapshot is encrypted.

Example restore workflow:

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0123456789abcdef0 \
  --volume-type gp3
```

**EBS Multi-Attach and shared storage**

EBS Multi-Attach lets multiple EC2 instances attach to the same volume at the same time, but it is not a backup feature.

- Supported only for specific volume types and Regions.
- Requires application-level coordination or a clustered file system.
- Useful for clustered workloads that need shared block access.
- Does not protect you from data corruption if multiple writers are not coordinated.

Use it when the workload is designed for shared block storage, not as a general durability strategy.

**Amazon RDS backup and recovery**

RDS offers automated backups, manual snapshots, and Multi-AZ deployments. These solve different problems.

- **Automated backups**
  - Enable point-in-time recovery within the retention window.
  - Use transaction logs plus backups to restore to a specific time.
  - Are the right choice when you need recovery from accidental changes or corruption within the retention period.

- **Manual snapshots**
  - Are user-initiated and retained until you delete them.
  - Are useful for long-term retention or before major changes.
  - Do not provide point-in-time recovery by themselves.

- **Multi-AZ**
  - Improves availability and failover.
  - Replicates synchronously to a standby in another Availability Zone.
  - Is not a substitute for backups because it can still replicate logical errors.

- **Read replicas**
  - Help with read scaling.
  - Can support disaster recovery patterns in some designs.
  - Are not a backup mechanism.

Practical distinction:
- Use **Multi-AZ** when you need automatic failover.
- Use **automated backups** when you need PITR.
- Use **manual snapshots** when you need a retained restore point before a change.

Example snapshot command:

```bash
aws rds create-db-snapshot \
  --db-instance-identifier mydb \
  --db-snapshot-identifier mydb-pre-upgrade
```

> A Multi-AZ database can fail over quickly, but it does not protect you from deleting the wrong table or corrupting data in both copies.

**Amazon Aurora recovery patterns**

Aurora has its own storage and recovery model, which is different from standard RDS engines.

- **Automated backups**
  - Support point-in-time restore within the retention window.
  - Use continuous backup of the cluster volume and logs.

- **Snapshots**
  - Can be manual or automated.
  - Useful for long-term retention and cloning.

- **Aurora replicas**
  - Improve availability and read scaling.
  - Can help with fast failover.
  - Are not backups.

- **Aurora Global Database**
  - Provides asynchronous cross-Region replication.
  - Is mainly for disaster recovery and global read access.
  - Is not a backup mechanism and does not replace snapshots or PITR.

Aurora is often the right answer when the question emphasizes fast recovery, managed replication, and database-specific failover.

**Amazon DynamoDB backup and recovery**

DynamoDB supports both backup and replication-oriented features.

- **On-demand backups**
  - Capture a table backup that you can restore later.
  - Are useful for long-term retention or before risky changes.
  - Are not the same as continuous point-in-time recovery.

- **Point-in-time recovery**
  - Lets you restore to any second in the last 35 days.
  - Protects against accidental writes or deletes.
  - Is the best answer when you need fine-grained rollback.

- **Global tables**
  - Replicate data across Regions.
  - Improve availability and low-latency access.
  - Are not a backup substitute because they can replicate bad writes.

Example backup command:

```bash
aws dynamodb create-backup \
  --table-name Orders \
  --backup-name Orders-pre-migration
```

**Amazon EFS and FSx backup concepts**

File systems have service-specific recovery patterns, and exam questions may test whether you know that backups are not identical across services.

- **Amazon EFS**
  - Supports AWS Backup integration.
  - Can also use file-level copy tools, but those are application-managed.
  - Is often used when you need shared POSIX file storage across instances.

- **Amazon FSx**
  - Supports backups depending on the FSx family.
  - Restore behavior is service-specific.
  - Is commonly used for Windows file shares, Lustre, NetApp ONTAP, or OpenZFS workloads.

For both services:
- Backups are separate from availability.
- Replication or multi-AZ features do not replace restore testing.
- You should verify application consistency if the workload writes actively during backup.

**AWS Backup as a centralized strategy**

AWS Backup is useful when you want centralized policy management across supported AWS services.

- **Benefits**
  - Centralizes backup policies.
  - Supports cross-account backup strategies.
  - Can help standardize retention and lifecycle rules.
  - Integrates with backup vaults and restore workflows.

- **When to use it**
  - You manage multiple services and want one policy plane.
  - You need consistent retention across accounts.
  - You want to reduce custom scripts for backup scheduling.

- **When native service tools may be better**
  - The service is not supported by AWS Backup.
  - You need service-specific features or restore behavior.
  - You need a specialized workflow that AWS Backup does not expose.

- **Vault Lock**
  - Adds immutability controls to backup vaults.
  - Helps enforce retention and compliance.
  - Is important for ransomware-resistant backup design.

Example AWS Backup plan concept:

```json
{
  "BackupPlanName": "daily-backups",
  "Rules": [
    {
      "RuleName": "daily",
      "TargetBackupVaultName": "primary-vault",
      "ScheduleExpression": "cron(0 5 * * ? *)",
      "Lifecycle": {
        "MoveToColdStorageAfterDays": 30,
        "DeleteAfterDays": 365
      }
    }
  ]
}
```

Use this as a conceptual pattern, not as a guaranteed complete API payload for every context.

**Cross-account and cross-Region backup strategy**

A strong recovery design usually separates the source account from the backup location.

- Store backups in a different account to reduce blast radius.
- Use a different Region for regional disaster recovery.
- Restrict delete permissions on backup vaults and S3 buckets.
- Test restore permissions separately from backup permissions.

This matters because an attacker or bad automation in the source account should not be able to destroy every recovery copy.

**Backup validation and restore testing**

Backups are only useful if they restore correctly.

- Test restores on a schedule.
- Verify application consistency, not just file presence.
- Document runbooks for who restores, where to restore, and how to validate.
- Confirm encryption keys, IAM permissions, and network access before an incident.
- Measure actual restore time against your RTO.

A practical restore test checklist:
1. Restore to an isolated environment.
2. Validate data integrity and application startup.
3. Confirm permissions, secrets, and dependencies.
4. Record the observed RTO and issues.
5. Update the runbook after each test.

**Choosing the right recovery pattern**

Use the following guidance when answering exam questions:

| Requirement | Best fit | Why |
|---|---|---|
| Protect against accidental deletion | Versioning, snapshots, PITR, Object Lock | Lets you roll back or prevent deletion |
| Very low downtime | Multi-AZ, failover, active-active design | Keeps service running during failure |
| Very low data loss | Synchronous replication, clustered design | Minimizes lost writes |
| Long-term retention | Manual snapshots, backup vault retention, Object Lock | Preserves restore points |
| Regional disaster recovery | Cross-Region replication, cross-Region backups | Keeps a copy outside the Region |
| Ransomware resistance | Immutable backups, Object Lock, Vault Lock | Prevents tampering with recovery data |

**Common exam traps**

- Treating **replication** as a backup.
- Confusing **Multi-AZ** with a backup strategy.
- Assuming **read replicas** provide automatic backup or PITR.
- Forgetting that **versioning** is required for S3 replication.
- Ignoring **immutability** when the scenario mentions ransomware or compliance.
- Choosing snapshots when the question explicitly asks for **point-in-time recovery**.
- Using a backup solution when the question really asks for **availability** or failover.

**Practical architecture examples**

A few common patterns help you map requirements to services:

- **Web app with user uploads**
  - Store objects in S3.
  - Enable versioning.
  - Use Object Lock for critical archives.
  - Replicate to another Region for DR.

- **Transactional database**
  - Use RDS Multi-AZ for availability.
  - Enable automated backups for PITR.
  - Take manual snapshots before schema changes.
  - Copy snapshots cross-Region if needed.

- **Shared file storage**
  - Use EFS or FSx.
  - Enable AWS Backup or service-native backups.
  - Test restore into a separate environment.

- **Analytics or batch workload**
  - Use S3 as the durable landing zone.
  - Protect important datasets with versioning and replication.
  - Use lifecycle policies for cost control, but do not confuse lifecycle with backup.

**CLI and operational examples**

You may see or use commands like these during operations:

```bash
# Enable S3 versioning
aws s3api put-bucket-versioning \
  --bucket my-bucket \
  --versioning-configuration Status=Enabled

# Create an RDS snapshot
aws rds create-db-snapshot \
  --db-instance-identifier prod-db \
  --db-snapshot-identifier prod-db-before-change

# Create a DynamoDB backup
aws dynamodb create-backup \
  --table-name Customers \
  --backup-name Customers-2026-04-26

# Create an EBS volume from a snapshot
aws ec2 create-volume \
  --snapshot-id snap-0123456789abcdef0 \
  --availability-zone us-east-1a \
  --volume-type gp3
```

**Final exam mindset**

When you read a scenario, ask three questions:

- Is the problem about **availability**, **durability**, or **recovery**?
- Is the concern **accidental change**, **regional failure**, or **ransomware**?
- Does the requirement need **failover**, **restore**, or **immutable retention**?

If you answer those correctly, the service choice usually becomes clear.
