## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon S3 overview**

Amazon Simple Storage Service (Amazon S3) is an object storage service designed for massive scale, high durability, and broad integration with AWS services. You store data as **objects** inside **buckets**, and you access those objects through HTTP-based APIs, the AWS CLI, SDKs, or console workflows.

S3 is commonly used for backups, data lakes, static website hosting, application assets, log storage, and shared data distribution. For exam scenarios, focus on choosing S3 when you need virtually unlimited object storage, strong durability, lifecycle automation, and flexible access control.

- **Object storage** means you manage objects, not block devices or file system mounts.
- **Bucket** names are globally unique within a partition.
- **Key** is the full object path-like name, such as `images/logo.png`.
- **Metadata** can be user-defined or system-defined and is stored with the object.
- **Versioning** helps protect against accidental overwrite and delete operations.

**Core S3 data model**

An S3 object consists of the object data, a key, metadata, and an optional version ID. Buckets are regional resources, but S3 is designed for global access through AWS endpoints and integrations.

A common exam trap is confusing S3 with EBS or EFS. Use S3 when you need object-level access and high durability rather than low-latency block storage or shared POSIX file semantics.

- **Bucket**: container for objects.
- **Object**: data plus metadata.
- **Key**: unique name within a bucket.
- **Version ID**: identifier assigned when versioning is enabled.
- **Prefix**: logical grouping in a key namespace, such as `logs/2026/04/`.

**Consistency model**

S3 provides strong read-after-write consistency for new object PUTs and DELETEs in all regions. If you upload a new object and immediately read it, you get the latest version. This is important for exam questions that compare S3 with older eventual-consistency behavior.

For overwrite PUTs and DELETEs, S3 behaves with eventual consistency semantics. That means a subsequent read may briefly return the previous version or reflect the delete after a short propagation delay. In practice, modern S3 is strongly consistent for most operations, but exam scenarios may still test the distinction between new object reads and overwrite/delete propagation behavior.

- **Read-after-write** consistency applies to new objects.
- **Eventual consistency** can affect overwrite and delete visibility.
- Design workflows to tolerate brief propagation delays when replacing objects.
- Use versioning when you need recovery from accidental changes.

> For exam answers, do not assume immediate global visibility for overwrite/delete side effects in every workflow.

**Bucket naming, regions, and limits**

Bucket names must be globally unique and follow DNS-compatible naming rules. Because bucket names appear in virtual-hosted-style URLs, avoid uppercase letters, underscores, and ambiguous patterns.

Important limits and behaviors often tested:

- **Single-object limit** is 5 TB.
- **Multipart upload** is required for large objects and is the preferred method for objects above 100 MB or when you want resilient parallel uploads.
- **Bucket versioning suspension** does not remove existing versions; it stops creating new versions and may create a null version behavior for future writes.
- **Buckets are regional**, so choose the Region close to your workload or compliance boundary.
- **Prefix performance** is no longer constrained by the old “random prefix” guidance, but key design still matters for organization and lifecycle rules.

**Request pricing and cost concepts**

S3 cost is not just storage. Exam scenarios often hinge on request volume and data movement.

- **PUT, COPY, POST, LIST, and lifecycle transition requests** incur charges.
- **GET and other retrieval requests** also incur charges.
- **Data transfer out** to the internet costs money.
- **Cross-Region replication** adds replication request and inter-Region data transfer costs.
- **Storage class retrieval fees** can dominate costs for archival data.
- **Lifecycle transitions** can create request charges, so frequent transitions may be inefficient.

Cost-aware design examples:

- Use **CloudFront** in front of S3 to reduce origin GET traffic and improve global delivery.
- Use **S3 Select** or compression when you want to reduce data scanned or transferred.
- Use **lifecycle policies** to move infrequently accessed data to cheaper classes, but avoid overly aggressive transitions for objects with short retention.
- Use **multipart upload** for large files to reduce retry waste on failed transfers.

**Access control fundamentals**

S3 access is controlled through a combination of IAM (Identity and Access Management), bucket policies, access point policies, ACLs, and object ownership settings. In modern architectures, prefer IAM policies, bucket policies, and access points over ACLs.

- **IAM policies** control what identities can do.
- **Bucket policies** attach resource-based permissions to a bucket.
- **Access points** provide application-specific network and permission entry points.
- **ACLs** are legacy and usually avoided unless a specific compatibility requirement exists.
- **Object Ownership** can disable ACLs and enforce bucket-owner control.

Use bucket policies when you want centralized control over a bucket. Use access points when multiple teams, applications, or VPCs need distinct access patterns to the same bucket.

Example bucket policy allowing read access from a specific IAM role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowAppRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AppRole"
      },
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::my-data-bucket/*"
    }
  ]
}
```

**Access Points and Multi-Region Access Points**

S3 Access Points simplify access management for shared buckets by giving each application or team its own alias, policy, and optional VPC restriction. This is useful when a single bucket serves many consumers with different permissions.

- **Access Point**: a named network and policy endpoint for a bucket.
- **VPC-only access point**: restricts access to traffic from a VPC endpoint.
- **Policy isolation**: lets you manage permissions per use case instead of one large bucket policy.

Multi-Region Access Points (MRAP) provide a global endpoint that routes requests to the closest healthy AWS Region replica. They are useful for multi-Region applications that need lower latency and regional resilience.

- **MRAP** is designed for active-active or globally distributed access patterns.
- It works with replication to keep data available across Regions.
- It is a strong fit for global read-heavy applications and disaster recovery designs.

**Object Lambda**

S3 Object Lambda lets you add custom code to transform data as it is retrieved from S3. You attach a Lambda function to an access point, and S3 invokes that function on GET requests.

Use Object Lambda when you need to:

- redact sensitive fields from objects on read,
- convert formats dynamically,
- personalize content without storing multiple object copies,
- enforce custom response shaping for different consumers.

Example use case: a single CSV file in S3 is transformed into filtered JSON for one application while another application reads the raw object through a different access path.

**Static website hosting**

S3 can host static websites for HTML, CSS, JavaScript, and image assets. This is a common exam pattern when the requirement is public website delivery without server-side code.

Typical architecture:

- S3 bucket stores website files.
- CloudFront provides caching, HTTPS, and global edge delivery.
- Route 53 points a custom domain to CloudFront.
- An origin access control or equivalent private origin pattern keeps the bucket private when CloudFront is used.

Example static website workflow:

1. Upload `index.html` and `error.html` to the bucket.
2. Enable static website hosting or use S3 as a private origin behind CloudFront.
3. Configure Route 53 alias records to the CloudFront distribution.
4. Use HTTPS at CloudFront rather than exposing the S3 website endpoint directly for secure production delivery.

Example CLI upload:

```bash
aws s3 sync ./site s3://my-static-site-bucket --delete
```

**Storage classes and retrieval behavior**

S3 storage classes let you balance cost, availability, durability, and retrieval time. Exam questions often ask you to choose the right class based on access frequency and recovery expectations.

- **S3 Standard**: general-purpose, low latency, high availability.
- **S3 Intelligent-Tiering**: automatically moves objects between access tiers when access patterns are unknown or change over time.
- **S3 Standard-IA**: for infrequently accessed data that still needs rapid retrieval.
- **S3 One Zone-IA**: lower cost, stored in one AZ, suitable only when you can tolerate AZ loss risk.
- **S3 Glacier Instant Retrieval**: archival data with millisecond retrieval.
- **S3 Glacier Flexible Retrieval**: archival data with retrieval options ranging from minutes to hours.
- **S3 Glacier Deep Archive**: lowest-cost archival storage with the longest retrieval times.

Retrieval expectations matter:

- **Glacier Flexible Retrieval** supports expedited, standard, and bulk retrieval options, with restore times varying by option and load.
- **Glacier Deep Archive** is for long-term retention and typically requires hours for restore.
- Objects in archival classes are not immediately available in the same way as Standard or Standard-IA.

A practical rule:

- Use **Standard** for active workloads.
- Use **Intelligent-Tiering** when access patterns are unpredictable.
- Use **Glacier Flexible Retrieval** when you need archival storage but may occasionally need faster restores.
- Use **Deep Archive** when retrieval is rare and delay is acceptable.

**Restore workflow for archival objects**

When you need an object from Glacier classes, you initiate a restore request. The object becomes temporarily available for a specified duration after restoration.

Typical restore flow:

1. Request a restore for the object.
2. Wait for the restore to complete.
3. Read the restored temporary copy during the restore window.
4. Re-restore if the temporary copy expires and you still need access.

Example CLI restore:

```bash
aws s3api restore-object \
  --bucket my-archive-bucket \
  --key reports/2020-audit.pdf \
  --restore-request '{"Days":7,"GlacierJobParameters":{"Tier":"Standard"}}'
```

**Versioning, delete markers, and suspension**

Versioning protects against accidental data loss by preserving prior object versions. When versioning is enabled, each write creates a new version ID.

Important behaviors:

- A delete on a versioned bucket usually adds a **delete marker** rather than removing data immediately.
- You can recover a deleted object by removing the delete marker or restoring a previous version.
- **Suspending versioning** stops new version creation, but existing versions remain.
- A bucket can accumulate many versions, so lifecycle rules are important for cost control.

Exam nuance:

- A delete marker can make the object appear deleted in normal reads while older versions still exist.
- Replication and lifecycle rules may treat delete markers differently from object versions.

**Multipart upload**

Multipart upload improves reliability and performance for large objects by splitting them into parts and uploading them independently. It is the standard approach for large files and supports parallelism and retry efficiency.

Key points:

- Parts are numbered from `1` to `10,000`.
- Each part must be at least 5 MB except the last part.
- The final object can be up to 5 TB.
- You can upload parts in parallel.
- If a multipart upload is not completed, you should abort it to avoid orphaned storage and cost.

Example multipart workflow:

```bash
aws s3api create-multipart-upload --bucket my-bucket --key videos/big-file.mp4
aws s3api upload-part --bucket my-bucket --key videos/big-file.mp4 --part-number 1 --body part1.bin
aws s3api upload-part --bucket my-bucket --key videos/big-file.mp4 --part-number 2 --body part2.bin
aws s3api complete-multipart-upload --bucket my-bucket --key videos/big-file.mp4 --multipart-upload file://parts.json
```

Operationally, use lifecycle rules to abort incomplete multipart uploads after a set number of days.

**Replication**

S3 replication copies objects between buckets for compliance, latency reduction, and disaster recovery. The main patterns are Cross-Region Replication (CRR) and Same-Region Replication (SRR).

Important replication details:

- The source bucket must have versioning enabled.
- The destination bucket usually also needs versioning enabled.
- Replication is asynchronous.
- Existing objects are not automatically replicated unless you use batch replication or a similar backfill process.
- You need appropriate IAM roles and permissions for replication to work.

Edge cases often tested:

- **Delete marker replication** can be enabled, but delete behavior is not always the same as object replication.
- **Replica modification sync** can allow changes made to replicas to be synchronized back depending on configuration and use case.
- **Ownership and policy requirements** matter: the destination bucket policy and object ownership settings must allow the replication role to write objects.
- Replication does not replace backup strategy; it is for availability and locality, not full historical protection by itself.

Use CRR when you need cross-Region resilience or data locality. Use SRR when you need copies within the same Region for compliance, analytics, or account separation.

**Presigned URLs and presigned POSTs**

Presigned URLs provide temporary, signed access to S3 objects without requiring the recipient to have AWS credentials. They are ideal for controlled downloads or uploads from untrusted clients.

- **Presigned GET URL**: temporary read access to a specific object.
- **Presigned PUT URL**: temporary upload access to a specific object.
- **Presigned POST**: browser-friendly upload mechanism with form fields and policy conditions.

Use presigned POST when you want browser uploads with constraints such as file size limits, key prefixes, or content-type restrictions.

Example presigned URL generation:

```bash
aws s3 presign s3://my-bucket/uploads/report.pdf --expires-in 3600
```

Example browser upload policy pattern:

```json
{
  "expiration": "2026-04-27T00:00:00Z",
  "conditions": [
    ["starts-with", "$key", "uploads/"],
    ["content-length-range", 1, 10485760]
  ]
}
```

**Event notifications**

S3 event notifications can trigger Lambda, SQS (Simple Queue Service), or SNS (Simple Notification Service) when objects are created or deleted. They are useful for lightweight event-driven workflows such as thumbnail generation, indexing, or ingestion pipelines.

Important limitations:

- Notifications are **not guaranteed to be ordered**.
- Notifications can be **delivered more than once**.
- They are **not a replacement for durable event streaming** or a high-reliability message bus.
- Design consumers to be idempotent.

Use S3 notifications for triggers, not as the sole source of truth for mission-critical ordered processing.

**Inventory, analytics, and reporting**

For large buckets, operational visibility matters. S3 provides reporting features that help you understand object usage, storage patterns, and optimization opportunities.

- **S3 Inventory** produces scheduled reports of objects and metadata, useful for audits, replication checks, and lifecycle analysis.
- **S3 Storage Lens** provides organization-wide visibility into storage usage, activity trends, and cost optimization signals.
- **S3 Analytics** helps identify access patterns and can support lifecycle transition decisions for infrequently accessed data.

These features are especially useful when you manage millions of objects and need reporting without scanning the bucket manually.

**Encryption and security**

S3 supports encryption at rest and in transit. Use HTTPS for data in transit and server-side encryption for data at rest.

Common encryption options:

- **SSE-S3**: S3-managed keys.
- **SSE-KMS**: AWS Key Management Service (KMS) keys with auditability and tighter control.
- **SSE-C**: customer-provided keys, used less often and with more operational burden.

Practical guidance:

- Use **SSE-KMS** when you need key control, audit trails, or compliance requirements.
- Use **bucket policies** to enforce encryption and deny unencrypted uploads.
- Combine encryption with access logging and least privilege.

Example policy condition to require encryption:

```json
{
  "Sid": "DenyUnencryptedUploads",
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::my-secure-bucket/*",
  "Condition": {
    "StringNotEquals": {
      "s3:x-amz-server-side-encryption": "aws:kms"
    }
  }
}
```

**Lifecycle management**

Lifecycle rules automate transitions and expiration. They are essential for cost control and retention management.

Common uses:

- Transition logs to cheaper storage after a set number of days.
- Expire temporary uploads automatically.
- Abort incomplete multipart uploads.
- Remove noncurrent versions in versioned buckets.

Example lifecycle configuration:

```json
{
  "Rules": [
    {
      "ID": "MoveLogsToGlacier",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "GLACIER"
        }
      ],
      "Expiration": {
        "Days": 365
      },
      "AbortIncompleteMultipartUpload": {
        "DaysAfterInitiation": 7
      }
    }
  ]
}
```

**Common exam patterns**

- Choose **S3 + CloudFront + Route 53** for a public static website with global performance.
- Choose **S3 + presigned URLs** for temporary client uploads without exposing credentials.
- Choose **S3 versioning + MFA delete** when accidental deletion protection is important.
- Choose **S3 replication** for cross-Region resilience, but remember it is asynchronous.
- Choose **Glacier Flexible Retrieval** when restore time can be minutes to hours.
- Choose **Intelligent-Tiering** when access patterns are unknown.
- Choose **Access Points** when multiple applications need controlled access to the same bucket.
- Choose **Object Lambda** when you need to transform objects on retrieval.
- Choose **S3 Inventory or Storage Lens** when you need reporting at scale.

**CLI examples**

List buckets:

```bash
aws s3 ls
```

Upload a file:

```bash
aws s3 cp ./report.pdf s3://my-bucket/reports/report.pdf
```

Sync a directory:

```bash
aws s3 sync ./public s3://my-static-site-bucket
```

Enable versioning:

```bash
aws s3api put-bucket-versioning \
  --bucket my-bucket \
  --versioning-configuration Status=Enabled
```

Check object metadata:

```bash
aws s3api head-object --bucket my-bucket --key reports/report.pdf
```

**Exam-focused takeaways**

S3 is the default answer for durable object storage, but the correct design depends on access pattern, cost, and retrieval expectations. Pay close attention to consistency, storage class behavior, request pricing, and access control mechanisms.

- **New object reads** are strongly consistent.
- **Overwrite and delete visibility** can still be a subtle exam distinction.
- **Request charges** and **data transfer** can change the best architecture.
- **Archival classes** trade cost for restore time.
- **Replication** improves resilience but does not replace backups.
- **Notifications** are useful triggers, not a guaranteed event backbone.
- **Access Points, MRAP, and Object Lambda** are modern features that appear in advanced scenarios.
