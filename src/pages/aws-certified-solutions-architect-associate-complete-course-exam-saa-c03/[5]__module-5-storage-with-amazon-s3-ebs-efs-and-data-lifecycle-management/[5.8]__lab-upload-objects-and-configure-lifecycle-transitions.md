## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab overview**

This lab focuses on two core Amazon S3 capabilities: uploading objects and using lifecycle rules to move data into lower-cost storage classes over time. In real projects, you use these features to reduce storage spend without changing application code, especially for logs, backups, media archives, and compliance-oriented retention patterns.

The exam angle is important: you need to know how S3 object storage works, how lifecycle rules behave in versioned and non-versioned buckets, and how storage class transition rules affect cost and retrieval time.

- **S3** stores objects, not files in a traditional filesystem sense.
- **Lifecycle rules** automate transitions and expirations based on object age, prefixes, or tags.
- **Versioning** changes how expiration behaves and how old object versions are retained.
- **Storage classes** trade off cost, durability, and retrieval latency.

**What you are trying to prove in the lab**

A good end-to-end lab outcome is not just “the rule exists,” but that you can verify the object moved or is scheduled to move according to the policy. In practice, you should be able to show:

- An object uploaded to a bucket.
- A lifecycle configuration attached to that bucket.
- A rule that transitions objects after a defined number of days.
- Evidence that the rule is active and scoped correctly.
- A clear understanding of what happens to current and noncurrent versions.

**Uploading objects to S3**

You can upload objects through the AWS Management Console, AWS CLI, SDKs, or automation tools. For exam prep, the CLI is especially useful because it makes the underlying API behavior visible.

Common upload commands:

```bash
aws s3 cp ./report.pdf s3://my-lab-bucket/reports/report.pdf
aws s3api put-object --bucket my-lab-bucket --key logs/app.log --body ./app.log
aws s3 sync ./site-assets s3://my-lab-bucket/assets/
```

Use the right command for the job:

- `aws s3 cp` is convenient for single files or small sets of files.
- `aws s3 sync` is useful when you want to mirror a directory tree.
- `aws s3api put-object` gives you direct API-level control.

If you create a bucket with the CLI, remember the regional rule:

```bash
aws s3api create-bucket \
  --bucket my-lab-bucket \
  --region us-east-1
```

For regions other than `us-east-1`, you must include the location constraint:

```bash
aws s3api create-bucket \
  --bucket my-lab-bucket \
  --region us-west-2 \
  --create-bucket-configuration LocationConstraint=us-west-2
```

> If you omit `LocationConstraint` outside `us-east-1`, bucket creation fails.

**Practical upload verification**

After upload, verify the object exists and inspect metadata:

```bash
aws s3api head-object \
  --bucket my-lab-bucket \
  --key reports/report.pdf
```

Useful fields to check:

- `ContentLength`
- `LastModified`
- `ETag`
- `StorageClass`
- `VersionId` if versioning is enabled

A simple console workflow also works:

1. Open the S3 bucket.
2. Choose **Upload**.
3. Add one or more files.
4. Confirm the object appears in the bucket listing.
5. Open the object details to inspect metadata and storage class.

**Lifecycle transitions: why they matter**

Lifecycle transitions let you move objects from a more expensive storage class to a cheaper one as they age. This is ideal when data is accessed frequently at first and then becomes cold.

Typical examples:

- Application logs that are hot for a few days, then rarely read.
- Monthly reports that are needed briefly, then archived.
- Backup exports that must be retained but not frequently retrieved.

Lifecycle rules are evaluated by S3 automatically. You define the rule once, and S3 applies it based on object age and rule filters.

**Common storage classes in lifecycle planning**

| Storage class | Typical use | Retrieval behavior | Exam-relevant note |
|---|---|---|---|
| `STANDARD` | Frequent access | Immediate | Default for general-purpose data |
| `STANDARD_IA` | Infrequent access | Immediate | Lower storage cost, retrieval fee applies |
| `ONEZONE_IA` | Infrequent access in one AZ | Immediate | Less resilient than multi-AZ classes |
| `GLACIER_IR` | Archive with millisecond retrieval | Immediate retrieval | Minimum storage duration is 90 days |
| `GLACIER` | Archive with flexible retrieval | Minutes to hours | Minimum storage duration is 90 days |
| `DEEP_ARCHIVE` | Long-term archive | Hours | Minimum storage duration is 180 days |

The minimum storage duration matters because lifecycle transitions can still incur early deletion or minimum-duration charges if you move or delete data too soon. For exam questions, always think about the full cost model, not just the destination class.

**Lifecycle rule structure**

A lifecycle configuration usually includes:

- A rule ID.
- A filter, such as prefix or object tags.
- A transition action.
- An expiration action, if needed.
- A status of `Enabled` or `Disabled`.

Example lifecycle configuration:

```json
{
  "Rules": [
    {
      "ID": "MoveLogsToIAThenArchive",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER_IR"
        }
      ]
    }
  ]
}
```

This pattern is useful when you want a staged cost reduction:

- Keep recent logs in `STANDARD`.
- Move them to `STANDARD_IA` after the active window.
- Move them to archive storage later.

**Applying lifecycle rules with the CLI**

You can attach a lifecycle policy with `put-bucket-lifecycle-configuration`:

```bash
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-lab-bucket \
  --lifecycle-configuration file://lifecycle.json
```

Then confirm it is attached:

```bash
aws s3api get-bucket-lifecycle-configuration \
  --bucket my-lab-bucket
```

A practical lab file might look like this:

```json
{
  "Rules": [
    {
      "ID": "TransitionReports",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "reports/"
      },
      "Transitions": [
        {
          "Days": 60,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 180,
          "StorageClass": "GLACIER"
        }
      ]
    }
  ]
}
```

This is a realistic pattern for documents that are read soon after creation and then retained for compliance or audit.

**Versioning and lifecycle behavior**

Versioning changes how expiration and retention work. Once versioning is enabled, S3 can keep multiple versions of the same key.

Enable versioning:

```bash
aws s3api put-bucket-versioning \
  --bucket my-lab-bucket \
  --versioning-configuration Status=Enabled
```

Check versioning status:

```bash
aws s3api get-bucket-versioning \
  --bucket my-lab-bucket
```

Important behavior:

- Expiring the current version in a versioned bucket adds a **delete marker**.
- The underlying object version is not immediately removed from storage.
- The old version remains billable until a noncurrent version expiration rule removes it.
- Noncurrent version transitions and expirations are separate lifecycle actions.

This distinction is a common exam trap. If you only expire the current version, you may not actually reduce storage cost right away because older versions can still exist and accrue charges.

**Versioned bucket lifecycle example**

A safer lifecycle policy for versioned data often includes both transition and cleanup rules:

```json
{
  "Rules": [
    {
      "ID": "VersionedReportsPolicy",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "reports/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        }
      ],
      "NoncurrentVersionTransitions": [
        {
          "NoncurrentDays": 90,
          "StorageClass": "GLACIER"
        }
      ],
      "NoncurrentVersionExpiration": {
        "NoncurrentDays": 365
      },
      "Expiration": {
        "Days": 730
      }
    }
  ]
}
```

Why this matters:

- Current versions stay available for active use.
- Noncurrent versions move to cheaper storage later.
- Old versions are eventually removed to control cost.
- The policy supports both recovery and retention goals.

> Be careful with aggressive transition timing. Moving objects too early can trigger minimum-storage-duration charges in archive classes and may not produce the savings you expect.

**A realistic migration scenario**

Imagine a team storing daily application logs in S3. The logs are heavily accessed for the first month during troubleshooting, then rarely used. A cost-aware design might look like this:

- Days 0–30: `STANDARD`
- Days 30–90: `STANDARD_IA`
- Days 90+: `GLACIER_IR` or `GLACIER`
- After retention period: expire the object or noncurrent versions

This pattern works because it matches access frequency to storage cost:

- Hot data stays immediately accessible.
- Warm data moves to a cheaper class with retrieval fees.
- Cold data moves to archive storage for long-term retention.

**Before-and-after validation**

A strong lab result includes evidence that the policy is active and the object is in the expected state.

Before applying lifecycle:

```bash
aws s3api head-object \
  --bucket my-lab-bucket \
  --key reports/report.pdf
```

Example output fields:

```json
{
  "ContentLength": 48291,
  "LastModified": "2026-04-26T10:15:00+00:00",
  "ETag": "\"9b2cf535f27731c974343645a3985328\"",
  "StorageClass": "STANDARD"
}
```

After the lifecycle rule has had time to act, you may see the storage class change:

```json
{
  "ContentLength": 48291,
  "LastModified": "2026-04-26T10:15:00+00:00",
  "ETag": "\"9b2cf535f27731c974343645a3985328\"",
  "StorageClass": "STANDARD_IA"
}
```

For versioned objects, validate both current and noncurrent versions:

```bash
aws s3api list-object-versions \
  --bucket my-lab-bucket \
  --prefix reports/report.pdf
```

Look for:

- Current version
- Noncurrent versions
- Delete markers
- Version IDs
- Transitioned storage classes

**Troubleshooting lifecycle transitions**

If a rule does not appear to work, check the following:

- The rule is `Enabled`.
- The filter matches the object key or tags.
- The object age is old enough for the transition.
- The destination storage class is valid for the transition path.
- The bucket is in the expected region.
- Versioning-specific rules are not being confused with current-version rules.

Common issues:

- Objects uploaded after the rule was created still need to age before transition.
- A prefix filter like `logs/` does not match `log/`.
- A tag-based filter requires the object to have the correct tags.
- Archive transitions may not be immediate because S3 applies lifecycle asynchronously.

**When to use lifecycle transitions versus expiration**

Use transitions when you still need the object but want to lower storage cost.

Use expiration when the object is no longer needed after a retention period.

A practical decision guide:

- Use **transition** for compliance archives, backups, and historical logs.
- Use **expiration** for temporary files, build artifacts, and stale exports.
- Use both when data has a useful life followed by a retention window.

**Exam-focused takeaways**

- S3 lifecycle rules automate cost optimization.
- `STANDARD_IA` and `ONEZONE_IA` are for infrequent access with immediate retrieval.
- `GLACIER_IR`, `GLACIER`, and `DEEP_ARCHIVE` are archive-oriented classes with different retrieval times and minimum storage durations.
- In a versioned bucket, expiring the current version creates a delete marker, but old versions can still remain billable.
- Noncurrent version expiration is essential for real cost cleanup.
- For `aws s3api create-bucket`, use `LocationConstraint` outside `us-east-1`.

**Quick lab checklist**

1. Create or select an S3 bucket.
2. Upload one or more test objects.
3. Enable versioning if the lab requires version-aware behavior.
4. Create a lifecycle policy with a realistic prefix or tag filter.
5. Attach the policy to the bucket.
6. Verify the configuration with `get-bucket-lifecycle-configuration`.
7. Validate object state with `head-object` or `list-object-versions`.
8. Confirm the expected storage class or version behavior after the rule age threshold.

**Why this lab matters in architecture design**

Lifecycle transitions are one of the simplest ways to reduce S3 cost without changing application behavior. In architecture questions, they often appear alongside backup retention, log retention, and data lake tiering. The best answer usually balances:

- Access frequency
- Retrieval latency
- Storage cost
- Retention requirements
- Versioning behavior

If you can explain those tradeoffs clearly, you can usually identify the correct S3 design choice on the exam.
