**Amazon S3 replication overview**

Amazon S3 replication copies objects from a **source bucket** to a **destination bucket** automatically after upload. You configure replication on the source bucket only, and S3 uses an IAM role to perform the copy. Replication is commonly used for compliance, disaster recovery, latency reduction, and cross-account data sharing.

Replication is asynchronous. It does not make the destination bucket a synchronous mirror, so you should expect a delay between source writes and replica availability. For exam scenarios, always separate the ideas of *durable storage* and *replication lag*.

- **Source bucket only**: you define the replication configuration on the source bucket.
- **Versioning required**: both source and destination buckets must have versioning enabled before replication is configured.
- **Asynchronous**: replicas appear after S3 processes the replication task.
- **Cross-Region** and **cross-account** replication are both supported.

**Replication rules, scope, and filtering**

Replication rules let you control which objects are copied and where they go. A rule can filter by **prefix**, **tag**, or both, which is useful when only part of a bucket needs replication. If multiple rules could match the same object, S3 uses rule priority to decide which rule applies.

Replication is not a full bucket clone in every case. You should think in terms of object-level selection and rule-based routing.

- **Prefix filtering**: replicate only objects under a path-like prefix such as `logs/`.
- **Tag filtering**: replicate only objects with a specific tag, such as `env=prod`.
- **Priority**: when rules overlap, the higher-priority rule wins.
- **Destination routing**: different rules can send objects to different destination buckets or storage classes.

A practical example is replicating only compliance logs to a separate account while leaving application uploads in the source Region.

**What S3 replication copies by default**

Replication behavior depends on the configuration and object type. In general, S3 can replicate object data and selected metadata, but you should not assume every attribute is copied automatically.

- **Object data**: the object content is replicated.
- **Object tags**: can be replicated if enabled.
- **Object metadata**: user-defined metadata is typically replicated with the object.
- **Delete markers**: can be replicated if you enable delete marker replication.
- **ACLs**: can be replicated if configured, though modern designs often prefer bucket owner controls over ACL reliance.
- **Replica ownership**: in cross-account replication, the destination account typically owns the replica unless you explicitly use ownership override behavior.

> Do not assume replication preserves every source-side access control pattern. Bucket policies and ownership controls matter more than ACLs in modern S3 designs.

**Replication permissions and ownership controls**

S3 replication requires permissions in two places: the IAM role used by S3 and the destination bucket policy. The source bucket must allow S3 to read objects for replication, and the destination bucket must allow the replication role to write replicas.

A typical replication role needs permissions such as:

- `s3:GetReplicationConfiguration`
- `s3:ListBucket`
- `s3:GetObjectVersion`
- `s3:GetObjectVersionAcl`
- `s3:GetObjectVersionTagging`
- `s3:ReplicateObject`
- `s3:ReplicateDelete`
- `s3:ReplicateTags`

` s3:ObjectOwnerOverrideToBucketOwner` is only needed when you intentionally use bucket owner override behavior. Do not treat it as universally required.

A minimal trust policy for the replication role allows S3 to assume the role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

A representative permissions policy looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "SourceReadPermissions",
      "Effect": "Allow",
      "Action": [
        "s3:GetReplicationConfiguration",
        "s3:ListBucket"
      ],
      "Resource": "arn:aws:s3:::source-bucket"
    },
    {
      "Sid": "SourceObjectReadPermissions",
      "Effect": "Allow",
      "Action": [
        "s3:GetObjectVersion",
        "s3:GetObjectVersionAcl",
        "s3:GetObjectVersionTagging"
      ],
      "Resource": "arn:aws:s3:::source-bucket/*"
    },
    {
      "Sid": "DestinationWritePermissions",
      "Effect": "Allow",
      "Action": [
        "s3:ReplicateObject",
        "s3:ReplicateDelete",
        "s3:ReplicateTags"
      ],
      "Resource": "arn:aws:s3:::destination-bucket/*"
    }
  ]
}
```

For cross-account replication, the destination bucket policy must allow the replication role from the source account to write to the destination bucket. Ownership controls on the destination bucket are often the cleanest way to ensure the destination account owns replicas.

**Cross-account replication pattern**

Cross-account replication is a common exam scenario. The source account owns the source bucket and the replication role, while the destination account owns the destination bucket. The destination bucket policy must trust the source account role, and the source bucket must allow the role to read versions and tags.

A practical setup usually includes:

1. Enable versioning on both buckets.
2. Create a replication IAM role in the source account.
3. Attach a permissions policy that allows source reads and destination writes.
4. Add a destination bucket policy that allows the role to replicate objects.
5. Configure the replication rule on the source bucket.
6. Optionally enable ownership override if the destination account must own all replicas.

Example destination bucket policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReplicationFromSourceRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/S3ReplicationRole"
      },
      "Action": [
        "s3:ReplicateObject",
        "s3:ReplicateDelete",
        "s3:ReplicateTags",
        "s3:ObjectOwnerOverrideToBucketOwner"
      ],
      "Resource": "arn:aws:s3:::destination-bucket/*"
    }
  ]
}
```

Use `s3:ObjectOwnerOverrideToBucketOwner` only when you need the destination bucket owner to become the object owner through override behavior. Otherwise, omit it.

**Replication status, metrics, and troubleshooting**

S3 exposes replication state on objects, which helps you determine whether an object is pending, completed, or failed. This is important for troubleshooting and for exam questions about delayed copies.

Common object replication status values include:

- **PENDING**: S3 has accepted the replication task but has not completed it yet.
- **COMPLETED**: the object has been replicated successfully.
- **FAILED**: replication did not succeed.
- **REPLICA**: the object is itself a replica in the destination bucket.

Replication Time Control (RTC) is an optional feature for workloads that need predictable replication. RTC provides monitoring and an SLA-oriented model for replication timing, which is useful when business requirements specify bounded replication delay.

Troubleshooting checklist:

- Verify versioning is enabled on both buckets.
- Confirm the replication rule matches the object key and tags.
- Check the IAM role trust policy and permissions.
- Confirm the destination bucket policy allows the role.
- Review object replication status and CloudWatch metrics if RTC is enabled.
- Check whether delete markers or tags are expected to replicate and whether the rule includes them.

**What replication does not do**

Replication is not a backup substitute by itself unless you design it that way. It also does not automatically solve every data protection problem.

- **No retroactive copy**: existing objects are not replicated unless you explicitly use batch replication or reprocess them.
- **No bidirectional sync**: replication is one-way from source to destination.
- **No application-level consistency**: your application still needs to handle eventual arrival of replicas.
- **No automatic conflict resolution**: if you write to both buckets independently, replication does not merge changes.

**S3 event notifications overview**

S3 event notifications let a bucket send messages when specific events occur, such as object creation or deletion. You can send notifications to **Amazon SNS**, **Amazon SQS**, or **AWS Lambda**. These notifications are asynchronous and delivered at least once, so consumers must be idempotent.

S3 event notifications are useful for decoupled workflows such as thumbnail generation, indexing, ETL triggers, and audit pipelines.

- **SNS**: fan-out to multiple subscribers.
- **SQS**: buffer events for downstream consumers.
- **Lambda**: invoke code directly for lightweight processing.
- **At least once**: duplicate deliveries can happen.
- **Asynchronous**: the upload is not blocked by notification delivery.

> Design consumers to tolerate duplicates and out-of-order arrival.

**Event types and rule behavior**

S3 notification rules match event types and object key filters. Common event types include:

- `s3:ObjectCreated:*`
- `s3:ObjectRemoved:*`
- `s3:ObjectRestore:*`

You can also use more specific create events such as `s3:ObjectCreated:Put`, `s3:ObjectCreated:Post`, `s3:ObjectCreated:Copy`, and `s3:ObjectCreated:CompleteMultipartUpload`.

Filtering is typically based on:

- **Prefix**: for example, `images/`
- **Suffix**: for example, `.jpg`

Be careful with overlapping rules. If multiple rules match the same event, S3 can send multiple notifications. This is useful when you intentionally want fan-out, but it can surprise you if you expect only one trigger.

S3 notification configuration is attached to the bucket and can include multiple destination types, but each rule targets a specific destination. If you need multiple destinations for the same event, you usually configure separate rules or use SNS as a fan-out layer.

**Destination policies and regional constraints**

For SNS, SQS, and Lambda targets, the destination resource policy must allow S3 to invoke or publish to the target. In practice, the destination is typically in the same Region as the S3 bucket for native event notifications.

A common SQS queue policy example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3SendMessage",
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": "sqs:SendMessage",
      "Resource": "arn:aws:sqs:us-east-1:111122223333:my-queue",
      "Condition": {
        "StringEquals": {
          "aws:SourceArn": "arn:aws:s3:::my-bucket"
        }
      }
    }
  ]
}
```

A common SNS topic policy example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Publish",
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:111122223333:my-topic",
      "Condition": {
        "StringEquals": {
          "aws:SourceArn": "arn:aws:s3:::my-bucket"
        }
      }
    }
  ]
}
```

A Lambda permission example:

```bash
aws lambda add-permission \
  --function-name process-s3-event \
  --statement-id s3invoke \
  --action lambda:InvokeFunction \
  --principal s3.amazonaws.com \
  --source-arn arn:aws:s3:::my-bucket
```

**Event notifications versus EventBridge**

S3 can also integrate with **Amazon EventBridge**, which is often better when you need richer routing, filtering, or centralized event handling. EventBridge is useful for event buses, archive/replay patterns, and multi-target routing without tightly coupling the bucket to each destination.

Use native S3 event notifications when you want:

- Simple bucket-triggered workflows
- Direct delivery to SNS, SQS, or Lambda
- Low configuration overhead

Use EventBridge when you want:

- Advanced filtering and routing
- Multiple downstream consumers
- Centralized event governance
- Integration with broader event-driven architectures

A practical distinction is that native notifications are bucket-scoped and destination-specific, while EventBridge gives you a more general event bus model.

**Practical CLI example: configure replication**

The AWS CLI can create or update replication configuration on the source bucket. The configuration references the IAM role and the destination bucket ARN.

```json
{
  "Role": "arn:aws:iam::111122223333:role/S3ReplicationRole",
  "Rules": [
    {
      "ID": "ReplicateLogs",
      "Status": "Enabled",
      "Priority": 1,
      "Filter": {
        "Prefix": "logs/"
      },
      "Destination": {
        "Bucket": "arn:aws:s3:::destination-bucket",
        "StorageClass": "STANDARD_IA"
      },
      "DeleteMarkerReplication": {
        "Status": "Enabled"
      }
    }
  ]
}
```

Apply it with:

```bash
aws s3api put-bucket-replication \
  --bucket source-bucket \
  --replication-configuration file://replication.json
```

Verify the configuration:

```bash
aws s3api get-bucket-replication --bucket source-bucket
```

**Practical CLI example: test event notifications**

You can configure a bucket notification to send object-created events to an SQS queue.

```json
{
  "QueueConfigurations": [
    {
      "Id": "ImageUploadEvents",
      "QueueArn": "arn:aws:sqs:us-east-1:111122223333:image-events",
      "Events": [
        "s3:ObjectCreated:*"
      ],
      "Filter": {
        "Key": {
          "FilterRules": [
            {
              "Name": "prefix",
              "Value": "images/"
            },
            {
              "Name": "suffix",
              "Value": ".jpg"
            }
          ]
        }
      }
    }
  ]
}
```

Apply it with:

```bash
aws s3api put-bucket-notification-configuration \
  --bucket my-bucket \
  --notification-configuration file://notification.json
```

Test by uploading an object:

```bash
aws s3 cp ./photo.jpg s3://my-bucket/images/photo.jpg
```

**CloudFormation example for replication**

CloudFormation is a common exam-friendly way to express replication infrastructure. The source bucket owns the replication configuration, and the role plus destination policy must be in place.

```yaml
Resources:
  ReplicationRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              Service: s3.amazonaws.com
            Action: sts:AssumeRole
      Policies:
        - PolicyName: S3ReplicationPolicy
          PolicyDocument:
            Version: '2012-10-17'
            Statement:
              - Effect: Allow
                Action:
                  - s3:GetReplicationConfiguration
                  - s3:ListBucket
                Resource: arn:aws:s3:::source-bucket
              - Effect: Allow
                Action:
                  - s3:GetObjectVersion
                  - s3:GetObjectVersionAcl
                  - s3:GetObjectVersionTagging
                Resource: arn:aws:s3:::source-bucket/*
              - Effect: Allow
                Action:
                  - s3:ReplicateObject
                  - s3:ReplicateDelete
                  - s3:ReplicateTags
                Resource: arn:aws:s3:::destination-bucket/*

  SourceBucket:
    Type: AWS::S3::Bucket
    Properties:
      VersioningConfiguration:
        Status: Enabled
      ReplicationConfiguration:
        Role: !GetAtt ReplicationRole.Arn
        Rules:
          - Id: ReplicateLogs
            Status: Enabled
            Priority: 1
            Filter:
              Prefix: logs/
            Destination:
              Bucket: arn:aws:s3:::destination-bucket
```

**Exam-oriented comparison**

| Capability | S3 Replication | S3 Event Notifications |
|---|---|---|
| Primary purpose | Copy objects to another bucket | Trigger downstream actions on object events |
| Direction | One-way object copy | Event-driven message delivery |
| Configuration location | Source bucket only | Source bucket only |
| Delivery model | Asynchronous replication | Asynchronous, at least once |
| Common targets | Destination S3 bucket | SNS, SQS, Lambda, EventBridge |
| Filtering | Prefix, tag, priority | Prefix, suffix, event type |
| Duplicate handling | Replica state and retries | Consumers must be idempotent |
| Typical use cases | DR, compliance, cross-account sharing | Workflow automation, processing pipelines |

**Common pitfalls**

- **Versioning**: replication requires versioning on both buckets before setup.
- **Rule overlap**: overlapping filters can replicate or notify more than you expect.
- **Idempotency**: event consumers must handle duplicate notifications safely.
- **Ownership**: do not assume the source uploader owns the replica in the destination bucket.
- **Policies**: destination resource policies must explicitly allow S3 access.
- **Region**: native notification targets are typically in the same Region as the bucket.
- **Scope**: replication is configured on the source bucket only.

**When to choose each feature**

Use replication when you need a second copy of the object in another bucket, often in another Region or account. Use event notifications when you need to react to object activity rather than store another copy.

A simple decision rule:

- Choose **replication** for data durability, locality, or governance.
- Choose **event notifications** for automation and decoupled processing.
- Choose **EventBridge** when you need broader routing and event management.

**Quick recall points**

- Replication copies objects; notifications trigger actions.
- Replication needs versioning on both buckets.
- Replication is configured on the source bucket only.
- Event notifications are at least once and can duplicate.
- SQS is a common buffering target, but consumers still need idempotency.
- EventBridge is the better choice for advanced routing.
