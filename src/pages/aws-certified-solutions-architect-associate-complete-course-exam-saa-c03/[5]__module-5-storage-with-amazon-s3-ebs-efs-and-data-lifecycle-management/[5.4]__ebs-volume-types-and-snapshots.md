## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Amazon EBS overview**

Amazon Elastic Block Store (EBS) provides persistent block storage for Amazon EC2 instances. You use it for operating systems, databases, file systems, and workloads that need low-latency storage attached to a single instance. EBS volumes live in one Availability Zone (AZ) and are designed for high durability within that AZ.

EBS is not the same as instance store:
- **EBS** persists independently of the instance lifecycle unless you delete it.
- **Instance store** is ephemeral and disappears when the instance stops or terminates.

A key architectural constraint is that an EBS volume is normally attached to only one EC2 instance at a time. The main exception is **Multi-Attach** for certain io1/io2 volumes, which allows multiple Nitro-based instances in the same AZ to attach the same volume for clustered applications.

**EBS volume types**

Choose the volume type based on workload pattern, latency sensitivity, and cost. The exam often tests whether you can match the right volume to the right use case.

| Volume type | Backing | Performance model | Typical use case |
|---|---|---|---|
| `gp3` | SSD | Baseline performance is decoupled from size; you can provision IOPS and throughput independently | General purpose workloads, boot volumes, most apps |
| `gp2` | SSD | Performance scales with size and uses burst credits | Legacy general purpose workloads |
| `io2` | SSD | High, consistent IOPS with durability focus | Mission-critical databases, latency-sensitive systems |
| `io1` | SSD | Provisioned IOPS, older generation than io2 | Legacy high-performance workloads |
| `st1` | HDD | Throughput-optimized, sequential access | Big data, log processing, data warehouses |
| `sc1` | HDD | Cold, lowest-cost HDD option for infrequent access | Cold data, infrequently accessed files |
| `standard` | Magnetic | Legacy magnetic volume type | Older workloads; generally avoid for new designs |

**General purpose SSD: gp3 vs gp2**

`gp3` is usually the best default choice because it gives predictable performance without forcing you to buy more storage just to get more IOPS. `gp2` ties performance to volume size and burst credits, which can be a problem for sustained workloads.

- **gp3**
  - Baseline performance is available regardless of volume size.
  - You can provision extra IOPS and throughput separately.
  - It is usually the preferred choice for boot volumes and general workloads.
- **gp2**
  - IOPS scale with size, so small volumes have lower baseline performance.
  - Burst credits can temporarily boost performance.
  - Sustained high I/O can drain credits and cause underperformance.

Practical implication:
- Use `gp3` when you want stable performance and cost control.
- Use `gp2` only when you are maintaining older designs or have a specific reason to keep it.

**Performance characteristics and exam-relevant limits**

EBS performance depends on the volume type, size, and instance attachment limits. The exact limits can vary by AWS updates and instance family, so focus on the model rather than memorizing every number.

- **gp3**
  - Baseline: 3,000 IOPS and 125 MiB/s throughput.
  - You can provision additional IOPS and throughput independently.
  - Good for workloads that need predictable performance without oversizing storage.
- **gp2**
  - Baseline performance scales with volume size.
  - Small volumes can be limited by low baseline IOPS.
  - Burst credits allow temporary performance above baseline.
- **io1 / io2**
  - Provisioned IOPS volumes for consistent low-latency workloads.
  - Best when you need sustained random I/O performance.
  - `io2` is the newer, more durable option and is preferred for critical systems.
- **st1**
  - Optimized for large sequential I/O.
  - Better throughput than IOPS-focused SSD types for streaming workloads.
  - Not suitable for random I/O-heavy databases.
- **sc1**
  - Lowest-cost HDD option for infrequently accessed data.
  - Designed for cold workloads, not performance-sensitive systems.
  - Use when cost matters more than latency.

> Do not assume the cheapest volume is always the best exam answer. Match the workload pattern first, then optimize cost.

**Boot volumes and data volumes**

EBS is commonly used as both a root volume and a data volume, but the behavior differs in important ways.

- **Root volume**
  - Contains the operating system and boot files.
  - If the instance is terminated, the root volume is often deleted by default depending on the `DeleteOnTermination` setting.
  - You should verify the root volume’s termination behavior when designing for recovery.
- **Data volume**
  - Stores application data, logs, databases, or shared content.
  - Usually preserved independently of instance lifecycle if you configure it that way.
  - You can detach and reattach it to another instance in the same AZ.

Example of checking block devices and termination behavior:
```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].BlockDeviceMappings'
```

If you need a root volume to survive termination, set `DeleteOnTermination` to `false` when launching or modifying the instance mapping.

**Attachment and filesystem considerations**

When you attach an EBS volume, the device name you request is not always the device name the operating system shows. This matters especially on Nitro-based instances, where EBS volumes typically appear as NVMe devices.

- On older virtualization types, you may see names like `/dev/xvdf`.
- On Nitro instances, the OS often exposes volumes as `/dev/nvme*n1`.
- You should discover the actual device using `lsblk`, `nvme list`, or filesystem labels/UUIDs rather than hardcoding a device path.

Example workflow:
```bash
aws ec2 attach-volume \
  --volume-id vol-0123456789abcdef0 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdf
```

Then on the instance:
```bash
lsblk
sudo file -s /dev/nvme1n1
sudo mkfs -t xfs /dev/nvme1n1
sudo mkdir -p /data
sudo mount /dev/nvme1n1 /data
```

A safer production pattern is to mount by UUID:
```bash
blkid
sudo mount UUID=1234-ABCD /data
```

**Multi-Attach**

Multi-Attach lets you attach a single `io1` or `io2` volume to multiple instances in the same AZ. This is useful for clustered applications that can coordinate access at the application layer.

Use cases:
- Shared clustered databases
- High-availability applications that manage their own locking
- Specialized workloads that need shared block access

Important limitations:
- Only supported for certain `io1` and `io2` volumes.
- Instances must be Nitro-based and in the same AZ.
- Your application must handle concurrent writes safely.
- It is not a general-purpose shared filesystem replacement.

**Snapshot fundamentals**

An EBS snapshot is a point-in-time backup of a volume. Snapshots are stored in AWS-managed, S3-backed storage, but they are not customer-visible S3 objects and you do not manage them like files in an S3 bucket.

Key properties:
- **Region-scoped**: snapshots belong to a Region.
- **Incremental**: after the first snapshot, only changed blocks are stored.
- **Durable**: AWS manages the underlying storage.
- **Copyable**: you can copy snapshots to another Region.
- **Shareable**: you can share snapshots within or across accounts, subject to encryption rules.

Snapshot use cases:
- Backup and restore
- Volume migration
- Cross-Region disaster recovery
- Creating golden images or cloned environments

**Snapshot creation and consistency**

A snapshot captures the state of the volume at a point in time, but application consistency is your responsibility.

- **Crash-consistent snapshot**
  - Captures the volume as if power was lost.
  - Suitable for many workloads, especially when the application can recover.
- **Application-consistent snapshot**
  - You quiesce the filesystem or application before taking the snapshot.
  - Preferred for databases and transactional systems.

Practical quiescing example:
```bash
sudo fsfreeze -f /data
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "App-consistent snapshot for /data"
sudo fsfreeze -u /data
```

For databases, you often use pre- and post-snapshot scripts:
1. Put the application into a consistent state.
2. Flush buffers or pause writes.
3. Create the snapshot.
4. Resume normal operations.

**Snapshot restore behavior**

When you create a new volume from a snapshot, the volume is immediately available, but the first read of blocks that have not yet been loaded can experience additional latency. This is often described as lazy loading or initial restore performance impact.

Implications:
- The first access to data may be slower than steady-state performance.
- Performance-sensitive restores may need warming before production use.
- You should test restore time and read patterns for critical systems.

For workloads that need fast, predictable restore performance, use **Fast Snapshot Restore (FSR)** where appropriate. FSR pre-warms snapshots in selected AZs so restored volumes can deliver full performance immediately.

**Snapshot copy and cross-Region design**

Snapshots are Region-scoped, so if you want a snapshot in another Region, you must copy it. You cannot directly use a snapshot from one Region in another Region without copying it first.

Example copy command:
```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --description "Copy for us-west-2 DR"
```

To create the destination snapshot in another Region, run the command with the target Region configured in your CLI profile or with `--region us-west-2`.

Encryption considerations:
- If the source snapshot is encrypted, the copy remains encrypted.
- You can re-encrypt the copied snapshot with a different AWS Key Management Service (AWS KMS) key in the destination Region.
- Cross-account sharing of encrypted snapshots requires careful KMS permissions and usually a copy-and-re-encrypt workflow in the target account.

Example copy with re-encryption:
```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --encrypted \
  --kms-key-id arn:aws:kms:us-west-2:123456789012:key/abcd1234-ef56-7890-abcd-1234567890ab \
  --description "Encrypted copy in destination Region"
```

**Sharing snapshots**

You can share unencrypted snapshots with other AWS accounts. For encrypted snapshots, sharing is more constrained because the recipient must also be able to use the KMS key or receive a copied, re-encrypted version.

Common patterns:
- Share an unencrypted snapshot for a controlled migration.
- Copy an encrypted snapshot into the target account and re-encrypt it with a target-owned KMS key.
- Use AWS Organizations or controlled IAM policies to manage access.

> Be careful with encryption and sharing. A snapshot that is encrypted with a customer-managed KMS key is not automatically usable by another account unless the key policy and sharing workflow are set up correctly.

**Snapshot automation**

You should not rely on manual snapshots for production backup hygiene. AWS provides managed automation options:

- **Amazon Data Lifecycle Manager (DLM)**
  - Automates EBS snapshot creation and retention.
  - Good for scheduled backups of tagged volumes.
  - Useful when you want policy-based lifecycle management.
- **AWS Backup**
  - Centralized backup service for EBS and other AWS services.
  - Supports backup plans, retention, and cross-Region/cross-account strategies.
  - Better when you want a unified backup governance model.

Example DLM-style tagging strategy:
```text
Backup = Daily
Environment = Prod
Application = Orders
```

Then apply a lifecycle policy to all volumes with matching tags.

**Restore and migration workflow**

A common exam scenario is migrating an application server or database volume to a new instance.

1. Stop writes or quiesce the application if consistency matters.
2. Create a snapshot of the source volume.
3. Copy the snapshot if you need another Region.
4. Create a new volume from the snapshot.
5. Attach the new volume to the target instance.
6. Mount it using the correct device discovery method.
7. Validate data and application startup.

Example restore command:
```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0123456789abcdef0 \
  --volume-type gp3
```

Then attach it:
```bash
aws ec2 attach-volume \
  --volume-id vol-0123456789abcdef0 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdf
```

**Common exam distinctions**

- **EBS vs instance store**
  - EBS persists; instance store does not.
- **Single-attach vs Multi-Attach**
  - Most EBS volumes attach to one instance only.
  - Multi-Attach is limited to specific `io1`/`io2` use cases.
- **gp2 vs gp3**
  - `gp2` depends on size and burst credits.
  - `gp3` gives more predictable baseline performance.
- **Snapshot vs AMI**
  - A snapshot backs up one volume.
  - An Amazon Machine Image (AMI) can include one or more EBS snapshots plus launch metadata.
- **Region scope**
  - Snapshots are Region-specific until copied.
- **Restore performance**
  - First reads after restore may be slower unless you use FSR or warm the volume.

**Practical design guidance**

Use these patterns when designing for the exam and for real systems:
- Choose `gp3` for most general-purpose workloads.
- Choose `io2` when you need sustained, low-latency, high-IOPS storage.
- Choose `st1` for throughput-heavy sequential workloads.
- Choose `sc1` for cold data that is rarely accessed.
- Use snapshots for backups, cloning, and migration.
- Automate snapshots with DLM or AWS Backup.
- Quiesce applications before snapshots when consistency matters.
- Copy snapshots across Regions for disaster recovery.
- Verify root volume termination behavior with `DeleteOnTermination`.
- Discover actual device names on Nitro instances instead of assuming `/dev/nvme1n1`.

**Quick reference**

- **Persistent block storage**: EBS
- **Default general-purpose choice**: `gp3`
- **High-performance SSD**: `io2`
- **Throughput-optimized HDD**: `st1`
- **Cold HDD**: `sc1`
- **Backup mechanism**: snapshots
- **Cross-Region move**: copy the snapshot
- **Automated backup policy**: DLM or AWS Backup
- **Shared block access**: Multi-Attach on supported `io1`/`io2`
