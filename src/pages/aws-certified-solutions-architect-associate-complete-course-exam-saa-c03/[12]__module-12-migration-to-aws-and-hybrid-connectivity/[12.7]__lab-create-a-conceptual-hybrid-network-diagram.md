## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Hybrid network design goals**

A hybrid network connects an on-premises environment to AWS so you can extend private connectivity, support migration, and keep sensitive traffic off the public internet where possible. For exam purposes, focus on the relationship between connectivity, routing, and segmentation rather than on a single “best” topology.

A conceptual diagram should show:
- The **on-premises network** with users, applications, and a customer edge device.
- The AWS side with one or more **VPCs**.
- The connectivity layer, such as **Site-to-Site VPN**, **AWS Direct Connect**, or both.
- The routing control point, often a **Virtual Private Gateway (VGW)** or **Transit Gateway (TGW)**.
- Optional shared services such as inspection, DNS, or centralized egress.

**Core hybrid connectivity building blocks**

A **Customer Gateway (CGW)** is the AWS resource that represents your on-premises VPN device. It stores the public IP address and routing information for the customer-side endpoint. The actual on-premises device is still outside AWS.

A **Virtual Private Gateway (VGW)** is the AWS-side VPN endpoint attached to a single VPC. It is commonly used for simpler hybrid designs where one VPC needs private connectivity to on-premises.

A **Transit Gateway (TGW)** is a regional hub that connects multiple VPCs and on-premises networks through attachments and route tables. It is the preferred pattern when you need centralized routing, segmentation, or many VPCs.

A **Direct Connect Gateway (DXGW)** does not connect directly to VPCs. It connects a Direct Connect private virtual interface to a **VGW** or **TGW**, which then provides access to VPCs. This distinction matters on the exam because the DXGW is a routing abstraction, not the VPC attachment itself.

**Common topology patterns**

For a simple single-VPC hybrid design, you often use:
- On-premises network
- Customer gateway
- Site-to-Site VPN
- Virtual private gateway
- One VPC

For a scalable multi-VPC design, you often use:
- On-premises network
- Customer gateway
- Site-to-Site VPN and/or Direct Connect
- Transit gateway
- Multiple VPC attachments
- Shared services VPC or inspection VPC

For a resilient enterprise design, you may combine:
- **Direct Connect** for predictable private connectivity
- **Site-to-Site VPN** as backup
- **Transit Gateway** for centralized routing
- Multiple Availability Zones for fault tolerance

**How to think about the diagram**

A good conceptual diagram answers four questions:
- Where does traffic enter AWS?
- Which AWS resource terminates or routes the traffic?
- Which VPCs can communicate, and under what rules?
- How do routes propagate between on-premises and AWS?

The diagram should separate:
- **Connectivity**: physical or logical transport such as VPN tunnels or Direct Connect links
- **Routing**: which prefixes are advertised and where they are learned
- **Segmentation**: which networks are allowed to talk to each other
- **Security**: encryption, security groups, network ACLs, and inspection appliances

**Site-to-Site VPN concepts**

AWS Site-to-Site VPN uses IPsec (Internet Protocol Security) tunnels over the internet. It is encrypted and suitable for quick setup, backup connectivity, or smaller environments.

Key points:
- A VPN connection typically uses two tunnels for redundancy.
- The customer side is represented by a CGW.
- The AWS side is a VGW or TGW attachment, depending on the design.
- Routing can be static or dynamic with BGP (Border Gateway Protocol).

Use VPN when:
- You need fast deployment.
- You want encrypted connectivity without dedicated circuits.
- You need a backup path for Direct Connect.

Example AWS CLI workflow for a VPN-based design:

```bash
aws ec2 create-customer-gateway \
  --type ipsec.1 \
  --public-ip 203.0.113.10 \
  --bgp-asn 65010

aws ec2 create-vpn-gateway \
  --type ipsec.1

aws ec2 attach-vpn-gateway \
  --vpn-gateway-id vgw-0123456789abcdef0 \
  --vpc-id vpc-0123456789abcdef0
```

> The exact CLI flow varies by design. In modern architectures, you may attach a TGW instead of a VGW, and you may create a VPN connection to the TGW rather than to a VPC-attached VGW.

**Direct Connect concepts**

AWS Direct Connect provides a dedicated private network connection between your environment and AWS. It is useful for consistent throughput, lower jitter, and private connectivity at scale.

Important distinctions:
- Direct Connect is a connectivity service, not a VPN.
- It does not inherently encrypt traffic.
- Encryption is typically achieved by:
  - Running a VPN overlay over Direct Connect
  - Using MACsec (Media Access Control Security) on supported links
  - Encrypting at higher layers, such as TLS (Transport Layer Security)

A Direct Connect architecture usually includes:
- A physical connection to an AWS Direct Connect location
- A private virtual interface for private traffic
- A DXGW when you need to reach multiple VPCs or multiple Regions through VGW or TGW

Use Direct Connect when:
- You need stable bandwidth and predictable latency.
- You have large data transfer requirements.
- You want private connectivity for hybrid workloads.

**Transit Gateway routing model**

Transit Gateway centralizes routing between many attachments. It is especially useful when you have multiple VPCs, multiple accounts, or multiple on-premises connections.

Think of TGW in terms of:
- **Attachments**: VPCs, VPNs, Direct Connect associations through DXGW, and other supported network attachments
- **Route tables**: policy and reachability control
- **Propagation**: routes learned from an attachment are inserted into a TGW route table
- **Association**: an attachment uses a specific TGW route table for its traffic decisions

A common pattern is:
- Application VPC attachments propagate their CIDRs into a shared TGW route table.
- On-premises routes propagate into the TGW route table used by application VPCs.
- Separate route tables isolate shared services, inspection, or development environments.

This is not the same as “routes belonging to a VPC.” The route table belongs to the TGW, and attachments are associated with it.

Example conceptual TGW route policy:
- Application VPCs can reach on-premises networks.
- On-premises can reach only approved application CIDRs.
- Shared services VPC can inspect or provide DNS to multiple spokes.
- Development VPCs are isolated from production unless explicitly allowed.

**Practical diagram layout**

A clear hybrid diagram usually places:
- On-premises on the left
- AWS on the right
- Connectivity in the middle
- Shared services or inspection above or below the main traffic path

A simple ASCII sketch:

```text
[On-Prem DC]
   |  \
   |   \  Site-to-Site VPN (IPsec)
   |    \
   |   [Customer Gateway]
   |         |
   |         +----------------------+
   |                                |
   |                         [Virtual Private Gateway]
   |                                |
   |                             [VPC]
   |
   +---- Direct Connect ---- [DX Location] ---- [Direct Connect Gateway] ---- [Transit Gateway] ---- [VPC A]
                                                                                                   \-- [VPC B]
```

For a TGW-centric design, show:
- One TGW in the center
- Multiple VPC attachments
- VPN attachment and/or DXGW path
- Route tables controlling which attachments can communicate

**Security and encryption considerations**

Hybrid diagrams should show where encryption exists and where it does not.

Use these rules:
- **VPN** provides encryption in transit over the internet.
- **Direct Connect** is private but not automatically encrypted.
- **MACsec** can encrypt supported Direct Connect links at Layer 2.
- **TLS** can protect application traffic regardless of the network path.

Do not assume that private connectivity alone satisfies encryption requirements. If the workload requires encryption, show the mechanism explicitly.

**Routing and failover behavior**

Hybrid designs often use multiple paths for resilience:
- Primary path: Direct Connect
- Secondary path: Site-to-Site VPN
- Optional tertiary path: another Direct Connect location or another VPN endpoint

With BGP, you can prefer Direct Connect by using route attributes and keep VPN as failover. The exact behavior depends on route advertisement, prefix specificity, and local preference on your on-premises routers.

Practical considerations:
- Advertise only the prefixes you need.
- Avoid overlapping CIDRs between on-premises and VPCs.
- Keep route summaries consistent to reduce complexity.
- Validate asymmetric routing risks when using multiple paths.

**Example AWS CLI snippets**

Create a customer gateway:

```bash
aws ec2 create-customer-gateway \
  --type ipsec.1 \
  --public-ip 198.51.100.25 \
  --bgp-asn 65000
```

Create a VPN connection to a VGW:

```bash
aws ec2 create-vpn-connection \
  --type ipsec.1 \
  --customer-gateway-id cgw-0123456789abcdef0 \
  --vpn-gateway-id vgw-0123456789abcdef0
```

Create a Transit Gateway:

```bash
aws ec2 create-transit-gateway \
  --description "Hybrid core transit hub"
```

Attach a VPC to a Transit Gateway:

```bash
aws ec2 create-transit-gateway-vpc-attachment \
  --transit-gateway-id tgw-0123456789abcdef0 \
  --vpc-id vpc-0123456789abcdef0 \
  --subnet-ids subnet-aaa subnet-bbb
```

These commands are useful for understanding the resource relationships, even if the lab itself is conceptual.

**What to label in your final diagram**

Include labels that make the architecture unambiguous:
- On-premises CIDR blocks
- VPC CIDR blocks
- CGW public IP
- VPN tunnels
- Direct Connect link
- DXGW, VGW, or TGW
- Route tables or propagation boundaries
- Shared services or inspection VPC if present
- Encryption method, if relevant

**Exam-focused takeaways**

- A **CGW** represents the on-premises VPN device in AWS.
- A **VGW** is the AWS-side VPN endpoint for a single VPC design.
- A **TGW** is the hub for multi-VPC and centralized routing designs.
- A **DXGW** connects Direct Connect to **VGW** or **TGW**, not directly to VPCs.
- **VPN** encrypts traffic; **Direct Connect** does not encrypt by default.
- TGW route tables control reachability through **association** and **propagation**.
- Use Direct Connect for predictable private connectivity and VPN for encryption or backup.

**Quick design checklist**

Before you finalize a conceptual hybrid diagram, verify:
- The AWS endpoint is correct for the chosen design.
- The on-premises device is represented by a CGW.
- Route flow is clear from on-premises to AWS and back.
- Encryption is shown where required.
- Failover paths are visible and labeled.
- TGW route tables and attachments are not confused with VPC route tables.

**Common mistakes to avoid**

- Drawing a DXGW as if it directly attaches to a VPC.
- Treating a CGW as the AWS-side endpoint.
- Assuming Direct Connect is encrypted without an overlay or MACsec.
- Mixing up TGW route table propagation with VPC route tables.
- Showing only one VPN tunnel when the design expects redundancy.

**Suggested study approach**

1. Draw a simple single-VPC VPN diagram first.
2. Add Direct Connect as a second path.
3. Replace the VGW with a TGW when you need multiple VPCs.
4. Add route propagation and segmentation boundaries.
5. Annotate encryption, failover, and routing decisions.

This progression helps you understand why each AWS networking component exists and how the exam expects you to choose between them.
