## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Migration tool selection mindset**

AWS migration questions usually test *fit*, not just feature recall. You need to match the tool to the workload type, data size, downtime tolerance, and whether you are moving servers, databases, files, or entire data centers.

A useful mental model is:

```text
Discover -> Assess -> Transfer/Replicate -> Cut over -> Validate -> Optimize
```

- **Discovery** identifies what you have.
- **Assessment** estimates effort, cost, and compatibility.
- **Transfer** moves bulk data.
- **Replication** keeps data in sync for cutover.
- **Cutover** switches traffic or workloads.
- **Validation** confirms integrity and performance.

**Core AWS migration services**

AWS provides several migration services, and each solves a different problem. In exam questions, the best answer is often the one that minimizes downtime while matching the source system type.

- **AWS Application Discovery Service** inventories on-premises servers and dependencies.
- **AWS Migration Hub** centralizes tracking across migration tools.
- **AWS Application Migration Service (MGN)** lifts and shifts servers into AWS with continuous block-level replication.
- **AWS Database Migration Service (DMS)** migrates databases and supports ongoing replication.
- **AWS DataSync** moves file-based data between supported storage systems.
- **AWS Snow Family** handles offline data transfer when network bandwidth is insufficient.
- **AWS Transfer Family** provides managed file transfer endpoints for SFTP, FTPS, and FTP into AWS storage.

**Discovery and assessment**

Discovery tools help you understand the current environment before you migrate. This is especially important for large estates where application dependencies are not obvious.

**AWS Application Discovery Service**

Application Discovery Service helps you collect information about on-premises servers, software, and relationships between systems. It supports two broad approaches:

- **Agent-based discovery**
  - Installs an agent on servers.
  - Can collect richer host-level detail, including process-level information in supported modes.
  - Is useful when you need deeper dependency mapping.
- **Agentless discovery**
  - Uses VMware vCenter integration for virtualized environments.
  - Is easier to deploy when you cannot install agents.
  - Typically provides less granular process detail than agent-based collection.

Use discovery data to answer questions like:

- Which servers are active?
- Which applications depend on which hosts?
- Which systems are candidates for rehost, replatform, or retire?
- What is the likely migration wave order?

**AWS Migration Hub**

Migration Hub gives you a single place to track migration progress across multiple AWS tools and partner solutions. It does not perform the migration itself; it coordinates visibility.

- Use it when you want a consolidated view of migration status.
- It helps teams avoid fragmented tracking across spreadsheets and separate tool consoles.
- It is useful in multi-wave programs where many servers, databases, and file transfers move in parallel.

**AWS Migration Evaluator**

Migration Evaluator helps estimate the cost and sizing implications of moving workloads to AWS. It is often used early in planning to support business case development.

- It can help compare current infrastructure cost to AWS consumption models.
- It is useful for right-sizing and financial planning.
- It is not a migration execution tool.

**Server migration: AWS Application Migration Service**

AWS Application Migration Service, often called **MGN**, is the primary AWS service for lift-and-shift server migration. It continuously replicates source servers into AWS and lets you launch test and cutover instances from the replicated data.

Use MGN when:

- You want to migrate physical servers, virtual machines, or cloud servers into AWS.
- You need low-downtime cutover.
- You want to avoid manual reinstallation and reconfiguration of each server.

Key characteristics:

- **Block-level replication** keeps source and target in sync.
- **Test launches** let you validate the migrated server before production cutover.
- **Cutover launches** switch to the AWS-hosted server when ready.
- It is best for rehosting, not for redesigning the application.

A typical MGN workflow looks like this:

1. Install the MGN replication agent on the source server.
2. Configure replication settings and staging resources in AWS.
3. Wait for initial sync and ongoing replication.
4. Launch a test instance in AWS.
5. Validate application behavior, networking, and dependencies.
6. Perform cutover during the approved maintenance window.

> MGN migrates servers, not application architecture. If the workload needs refactoring, use a modernization approach instead of a pure lift-and-shift.

**Database migration: AWS Database Migration Service**

AWS DMS is designed for database migration and replication. It is commonly used when you need to move data with minimal downtime, but it does not replace a full application migration plan.

Use DMS when:

- You are migrating between database engines or versions.
- You need ongoing replication during a transition period.
- You want to keep a source database running while the target is being populated.

DMS is especially useful for:

- Homogeneous migrations, such as Oracle to Oracle or MySQL to MySQL.
- Heterogeneous migrations, such as Oracle to PostgreSQL, with schema conversion support from AWS Schema Conversion Tool (SCT).
- Continuous replication for cutover planning.

Important exam nuance:

- DMS focuses on data movement and replication.
- It does **not** automatically migrate all database objects, stored procedures, application code, or dependencies.
- You often pair DMS with SCT for schema conversion and validation.

A common migration pattern is:

```text
Source DB -> DMS replication instance -> Target DB
```

Example conceptual DMS task flow:

```bash
aws dms create-replication-instance \
  --replication-instance-identifier my-repl-instance \
  --replication-instance-class dms.t3.medium \
  --allocated-storage 100

aws dms create-endpoint \
  --endpoint-identifier source-db \
  --endpoint-type source \
  --engine-name postgres \
  --username appuser \
  --password 'ExamplePassword123!' \
  --server-name source.example.internal \
  --port 5432 \
  --database-name appdb

aws dms create-endpoint \
  --endpoint-identifier target-db \
  --endpoint-type target \
  --engine-name aurora-postgresql \
  --username appuser \
  --password 'ExamplePassword123!' \
  --server-name target.cluster-abcdefghijkl.us-east-1.rds.amazonaws.com \
  --port 5432 \
  --database-name appdb
```

- **Replication instance** runs the DMS tasks.
- **Source endpoint** defines where data comes from.
- **Target endpoint** defines where data lands.
- **Task settings** control full load, CDC, or both.

> CLI examples are conceptual and must be adapted to the engine, network path, authentication method, and AWS Region you actually use.

**File and object transfer: AWS DataSync**

AWS DataSync automates online transfer of file-based data between supported storage services and on-premises locations. It is optimized for moving large amounts of file data over the network with scheduling, verification, and incremental transfer support.

Use DataSync when:

- You need to move files from NFS, SMB, or object storage sources to AWS storage.
- You want scheduled or recurring transfers.
- You need a managed service instead of custom scripts or rsync jobs.

Common targets include:

- Amazon S3
- Amazon EFS
- Amazon FSx for Windows File Server
- Amazon FSx for Lustre

Common sources include:

- On-premises NFS or SMB file systems
- Amazon S3
- Other supported AWS storage locations

Practical exam points:

- DataSync is primarily for **file-based** movement.
- It does not behave like a general-purpose application migration tool.
- It can preserve many file attributes, but you should not assume every metadata or permission model transfers identically across all source and destination types.
- For S3 locations, you specify a bucket and optional prefix semantics, not a local filesystem path.

Example conceptual DataSync setup:

```bash
aws datasync create-location-nfs \
  --server-hostname nfs.example.internal \
  --subdirectory /exports/data \
  --on-prem-config AgentArns=arn:aws:datasync:us-east-1:123456789012:agent/agent-0123456789abcdef0

aws datasync create-location-s3 \
  --s3-bucket-arn arn:aws:s3:::my-migration-bucket \
  --subdirectory incoming

aws datasync create-task \
  --source-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-11111111111111111 \
  --destination-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-22222222222222222
```

- Use `--subdirectory` to map a path or prefix within the source or destination location.
- Use a task to define what gets copied and how often.
- Use DataSync when you need a managed, repeatable transfer rather than a one-time manual copy.

**Offline transfer: AWS Snow Family**

The AWS Snow Family is for moving large amounts of data when network transfer is too slow, too expensive, or unreliable. It is the right answer when bandwidth constraints make online migration impractical.

**Snowcone**

- Small, portable device for edge computing and data transfer.
- Useful for remote or disconnected environments.
- Best for smaller edge workloads or limited data movement.

**Snowball Edge**

- Rugged device for larger offline transfers.
- Supports storage and optional compute capabilities.
- Useful when you need to move substantial data from sites with poor connectivity.

**Snowmobile**

- A truck-sized data transfer solution for extremely large migrations.
- Intended for petabyte-scale and beyond.
- Used when the data volume is so large that shipping devices is more practical than network transfer.

Selection guidance:

- Choose **Snowcone** for compact edge use cases and small-scale offline transfer.
- Choose **Snowball Edge** for substantial offline transfer and optional edge compute.
- Choose **Snowmobile** for very large, data-center-scale migrations where network transfer is not realistic.

**Managed file transfer: AWS Transfer Family**

AWS Transfer Family provides managed endpoints for secure file transfer into and out of AWS storage services.

Protocols supported:

- **SFTP** (SSH File Transfer Protocol)
- **FTPS** (FTP over TLS)
- **FTP** in supported scenarios

Use it when:

- Partners or applications already use SFTP/FTPS/FTP.
- You want to land files directly in Amazon S3 or Amazon EFS.
- You need a managed service instead of running your own file transfer server.

Typical architecture:

```text
External client -> AWS Transfer Family endpoint -> Amazon S3 or Amazon EFS
```

This is often the best answer when the question mentions:

- External trading partners
- Legacy file transfer workflows
- Secure managed inbound file ingestion

**How the tools fit together**

A real migration program often uses several AWS services together rather than one service alone.

**Example 1: Server migration with database cutover**

- Use **Application Discovery Service** to inventory servers and dependencies.
- Use **Migration Hub** to track progress.
- Use **MGN** to replicate and launch application servers in AWS.
- Use **DMS** to replicate the database with minimal downtime.
- Cut over the application after validation.

This pattern is common when the application tier can be rehosted, but the database needs continuous sync until the final switch.

**Example 2: File share migration**

- Use **DataSync** to move file shares from on-premises NFS to Amazon EFS.
- Use **Transfer Family** if external partners must continue uploading files via SFTP.
- Use **Migration Hub** if the file migration is part of a broader program.

**Example 3: Disconnected data center migration**

- Use **Snowball Edge** or **Snowmobile** for bulk offline transfer.
- Use **DMS** or **DataSync** afterward for incremental synchronization if needed.
- Use **MGN** only for server workloads that need full machine replication.

**Exam-style comparison**

| Tool | Best for | Not ideal for |
|---|---|---|
| Application Discovery Service | Inventorying servers and dependencies | Moving data or workloads |
| Migration Hub | Tracking migration progress | Performing migration tasks itself |
| Migration Evaluator | Cost and sizing analysis | Execution or replication |
| MGN | Rehosting servers with low downtime | Refactoring applications |
| DMS | Database replication and migration | Full application migration |
| DataSync | File-based transfers | Arbitrary server migration |
| Snow Family | Offline bulk transfer | Small routine transfers over good networks |
| Transfer Family | Managed SFTP/FTPS/FTP file ingestion | General server or database migration |

**Practical decision rules**

Use these rules to answer exam questions quickly:

- If the question says **server lift-and-shift**, think **MGN**.
- If it says **database migration with ongoing replication**, think **DMS**.
- If it says **file share transfer**, think **DataSync**.
- If it says **SFTP to S3**, think **Transfer Family**.
- If it says **no reliable network** or **massive data volume**, think **Snow Family**.
- If it says **inventory dependencies before migration**, think **Application Discovery Service**.
- If it says **track many migration tasks in one place**, think **Migration Hub**.

**Common pitfalls**

- Do not use **DMS** as a generic answer for moving an entire application stack.
- Do not use **MGN** when the question is really about database replication only.
- Do not use **DataSync** for server image migration.
- Do not assume **Snowball Edge** is always the answer for large data; scale and network constraints matter.
- Do not confuse **Migration Hub** with a migration execution service.
- Do not assume discovery data is equally detailed in all modes; agent-based discovery is richer than agentless discovery.

**End-to-end migration lifecycle example**

A typical enterprise migration might look like this:

1. You run **Application Discovery Service** to map servers and dependencies.
2. You use **Migration Evaluator** to estimate AWS cost and right-size targets.
3. You register the project in **Migration Hub** for centralized tracking.
4. You use **MGN** to replicate application servers into AWS.
5. You use **DMS** to keep the database synchronized until cutover.
6. You use **DataSync** to move shared file data to Amazon EFS.
7. You validate the migrated environment and then cut over during a maintenance window.
8. You decommission the source environment after business sign-off.

This sequence is a strong exam pattern because it combines discovery, planning, transfer, and cutover in a realistic way.

**Quick reference for migration scenarios**

- **Rehost Windows/Linux servers to EC2**: MGN
- **Move Oracle to Amazon Aurora PostgreSQL with minimal downtime**: DMS plus SCT
- **Move a large NFS share to Amazon EFS**: DataSync
- **Accept partner files over SFTP into S3**: Transfer Family
- **Move petabytes from an isolated data center**: Snowmobile or Snowball Edge depending on scale
- **Map application dependencies before migration**: Application Discovery Service
- **Track many migration waves**: Migration Hub

**CLI and configuration reminders**

When you work with migration services, the exact CLI syntax depends on the service and resource type. In practice, you should verify:

- Region and account IDs
- IAM permissions
- VPC and subnet placement
- Security group rules
- Endpoint connectivity
- Encryption requirements
- Source and target engine compatibility

Example of a simple migration-oriented security group rule set:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 5432 \
  --source-group sg-0fedcba9876543210
```

This kind of rule is often necessary for DMS or application cutover testing when the replication instance or target database must reach the source.

**Final exam takeaway**

Choose the service based on the migration problem:

- **Discover** with Application Discovery Service.
- **Track** with Migration Hub.
- **Rehost servers** with MGN.
- **Replicate databases** with DMS.
- **Move files** with DataSync.
- **Transfer offline at scale** with Snow Family.
- **Handle managed SFTP/FTPS/FTP** with Transfer Family.

If you remember the workload type and downtime requirement, you can usually eliminate the wrong answers quickly.
