## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Lab purpose and troubleshooting mindset**

This lab focuses on using the AWS Command Line Interface (AWS CLI) to inspect the networking resources that determine whether traffic can flow in and out of an Amazon Virtual Private Cloud (VPC). In practice, you use these commands to answer questions like:

- Which route table is actually associated with this subnet?
- Is traffic going to an Internet Gateway, NAT Gateway, Transit Gateway, or VPC peering connection?
- Are security groups or network ACLs blocking traffic?
- Does DNS resolution work in this VPC?
- Are flow logs showing accepted or rejected packets?

A good troubleshooting workflow starts with the path of the packet and works outward:

1. Confirm the instance or workload has the expected IP addressing.
2. Check subnet association and route tables.
3. Verify security groups and network ACLs.
4. Inspect gateways, peering, and propagation.
5. Review DNS settings, DHCP options, and flow logs.
6. Validate load balancer and target group reachability if the workload is behind an ELB.

**Core AWS CLI setup**

Before you inspect resources, make sure your CLI is configured for the correct account and Region.

```bash
aws configure
aws sts get-caller-identity
aws configure list
```

Useful habits:

- Set the correct **Region** explicitly when troubleshooting.
- Use `--profile` if you manage multiple accounts.
- Add `--output table` or `--output json` depending on whether you want readability or machine parsing.
- Use `--query` with JMESPath to reduce noisy output.

Example:

```bash
aws ec2 describe-vpcs --region us-east-1 --output table
```

**VPC and subnet discovery**

Start by identifying the VPC and subnet that contain the workload. You often need the VPC ID before you can inspect routes, DNS settings, or DHCP options.

```bash
aws ec2 describe-vpcs \
  --filters Name=tag:Name,Values=prod-vpc \
  --query 'Vpcs[*].{VpcId:VpcId,Cidr:CidrBlock,IsDefault:IsDefault}' \
  --output table
```

```bash
aws ec2 describe-subnets \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'Subnets[*].{SubnetId:SubnetId,Cidr:CidrBlock,Az:AvailabilityZone,MapPublicIpOnLaunch:MapPublicIpOnLaunch}' \
  --output table
```

What to look for:

- **Public subnet** behavior usually depends on a route to an Internet Gateway and whether instances receive a public IPv4 address.
- **Private subnet** behavior usually depends on a route to a NAT Gateway for outbound internet access.
- `MapPublicIpOnLaunch` can help explain why new instances do or do not receive public IPv4 addresses automatically.

**Route tables and route evaluation**

Route tables are one of the most important resources to inspect. A subnet uses either an explicitly associated route table or the VPC main route table if no explicit association exists.

> Be careful: `describe-route-tables --filters Name=association.subnet-id,...` only returns route tables explicitly associated with that subnet. If the subnet uses the main route table, this filter may not show the effective route table you need to troubleshoot.

Inspect all route tables in a VPC:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'RouteTables[*].{
    RouteTableId:RouteTableId,
    Main:Associations[?Main==`true`]|[0].Main,
    Associations:Associations[*].SubnetId,
    Routes:Routes[*].{Destination:CidrBlock,Ipv6Destination:Ipv6CidrBlock,Target:GatewayId}
  }' \
  --output json
```

A more practical route summary:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'RouteTables[*].{
    RouteTableId:RouteTableId,
    Associations:Associations[*].SubnetId,
    Routes:Routes[*].{
      DestinationCidrBlock:CidrBlock,
      DestinationIpv6CidrBlock:Ipv6CidrBlock,
      GatewayId:GatewayId,
      NatGatewayId:NatGatewayId,
      TransitGatewayId:TransitGatewayId,
      VpcPeeringConnectionId:VpcPeeringConnectionId,
      NetworkInterfaceId:NetworkInterfaceId,
      EgressOnlyInternetGatewayId:EgressOnlyInternetGatewayId,
      State:State,
      Origin:Origin
    }
  }' \
  --output json
```

Important route concepts:

- **Static routes** are created manually and usually have `Origin` set to `CreateRoute`.
- **Propagated routes** can appear from VPN or Transit Gateway propagation and usually have `Origin` values such as `EnableVgwRoutePropagation` or similar service-specific origins.
- A route can be `active` or `blackhole`.
- A `blackhole` route often means the target resource was deleted or is unavailable.

Safer JMESPath pattern for route targets:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'RouteTables[*].Routes[*].{
    Destination:CidrBlock,
    Target:GatewayId || NatGatewayId || TransitGatewayId || VpcPeeringConnectionId || NetworkInterfaceId || EgressOnlyInternetGatewayId,
    State:State,
    Origin:Origin
  }' \
  --output json
```

This avoids the common mistake of trying to `join()` mutually exclusive route target fields that may be null.

**Internet Gateway, NAT Gateway, and egress paths**

Inspect the gateways directly instead of relying only on route table references. This helps confirm whether the target exists and is attached.

Internet Gateway:

```bash
aws ec2 describe-internet-gateways \
  --filters Name=attachment.vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'InternetGateways[*].{InternetGatewayId:InternetGatewayId,Attachments:Attachments[*].State,Tags:Tags}' \
  --output table
```

NAT Gateway:

```bash
aws ec2 describe-nat-gateways \
  --filter Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'NatGateways[*].{NatGatewayId:NatGatewayId,SubnetId:SubnetId,State:State,PublicIp:NatGatewayAddresses[0].PublicIp,ElasticIp:NatGatewayAddresses[0].AllocationId}' \
  --output table
```

What matters:

- A public subnet typically needs a route like `0.0.0.0/0 -> igw-...`.
- A private subnet typically needs a route like `0.0.0.0/0 -> nat-...` for outbound IPv4 internet access.
- NAT Gateways require a public subnet and an Elastic IP.
- If the NAT Gateway is `deleted`, `pending`, or in the wrong subnet, private instances may lose internet access.

**Elastic IP and public IP checks**

Internet reachability often fails because the instance does not actually have a public IPv4 address, or the Elastic IP is not associated as expected.

Inspect instance addressing:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[*].Instances[*].{
    InstanceId:InstanceId,
    SubnetId:SubnetId,
    PrivateIp:PrivateIpAddress,
    PublicIp:PublicIpAddress,
    VpcId:VpcId,
    State:State.Name
  }' \
  --output table
```

Inspect Elastic IP associations:

```bash
aws ec2 describe-addresses \
  --query 'Addresses[*].{
    AllocationId:AllocationId,
    PublicIp:PublicIp,
    InstanceId:InstanceId,
    NetworkInterfaceId:NetworkInterfaceId,
    AssociationId:AssociationId
  }' \
  --output table
```

Troubleshooting clues:

- No `PublicIpAddress` on an instance in a public subnet can indicate `MapPublicIpOnLaunch` is disabled or the instance was launched without a public IP.
- An Elastic IP attached to the wrong network interface can make the instance unreachable.
- A public subnet without a route to an Internet Gateway still cannot reach the internet.

**Security groups and stateful filtering**

Security groups are stateful virtual firewalls attached to ENIs (Elastic Network Interfaces). If inbound traffic is allowed, return traffic is automatically allowed.

Inspect security groups:

```bash
aws ec2 describe-security-groups \
  --group-ids sg-0123456789abcdef0 \
  --query 'SecurityGroups[*].{
    GroupId:GroupId,
    GroupName:GroupName,
    VpcId:VpcId,
    Ingress:IpPermissions,
    Egress:IpPermissionsEgress
  }' \
  --output json
```

Common checks:

- Confirm the source CIDR, source security group, or prefix list matches the expected client.
- Confirm the destination port is open, such as `80`, `443`, or `22`.
- Confirm outbound rules allow the response path if you use restrictive egress rules.
- Remember that security groups are **stateful**, so you usually do not need an explicit return rule.

Example ingress rule inspection:

```bash
aws ec2 describe-security-groups \
  --group-ids sg-0123456789abcdef0 \
  --query 'SecurityGroups[0].IpPermissions[*].{
    Protocol:IpProtocol,
    FromPort:FromPort,
    ToPort:ToPort,
    Sources:IpRanges[*].CidrIp
  }' \
  --output table
```

**Network ACLs and stateless filtering**

Network ACLs (NACLs) are stateless subnet-level filters. You must allow both directions explicitly.

Inspect NACLs:

```bash
aws ec2 describe-network-acls \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'NetworkAcls[*].{
    NetworkAclId:NetworkAclId,
    IsDefault:IsDefault,
    Associations:Associations[*].SubnetId,
    Entries:Entries[*].{RuleNumber:RuleNumber,Egress:Egress,Protocol:Protocol,RuleAction:RuleAction,CidrBlock:CidrBlock,Ipv6CidrBlock:Ipv6CidrBlock,PortRange:PortRange}
  }' \
  --output json
```

Key points:

- The default NACL initially allows all inbound and outbound traffic, but once you modify it, it no longer behaves as an automatic allow-all.
- Custom NACLs are evaluated in rule-number order.
- The first matching rule wins.
- Because NACLs are stateless, you must allow ephemeral return ports as well as the application port.

Example of a restrictive NACL pattern:

```text
Inbound:
- Allow TCP 80 from 0.0.0.0/0
- Allow TCP 443 from 0.0.0.0/0
- Allow ephemeral ports 1024-65535 from 0.0.0.0/0 for return traffic

Outbound:
- Allow TCP 80 to 0.0.0.0/0
- Allow TCP 443 to 0.0.0.0/0
- Allow ephemeral ports 1024-65535 to 0.0.0.0/0
```

**DNS support, hostnames, and DHCP options**

VPC DNS behavior depends on VPC settings and DHCP options. Two common VPC attributes are `enableDnsSupport` and `enableDnsHostnames`.

Inspect VPC DNS attributes:

```bash
aws ec2 describe-vpc-attribute \
  --vpc-id vpc-0123456789abcdef0 \
  --attribute enableDnsSupport
```

```bash
aws ec2 describe-vpc-attribute \
  --vpc-id vpc-0123456789abcdef0 \
  --attribute enableDnsHostnames
```

Interpretation:

- `enableDnsSupport` enables use of the Amazon-provided DNS resolver in the VPC.
- `enableDnsHostnames` affects whether the VPC can assign DNS hostnames to resources in supported cases.
- Public DNS hostnames for instances depend on multiple conditions, including VPC settings and whether the instance has a public IPv4 address or Elastic IP.

Inspect DHCP options sets:

```bash
aws ec2 describe-vpcs \
  --vpc-ids vpc-0123456789abcdef0 \
  --query 'Vpcs[0].DhcpOptionsId' \
  --output text
```

```bash
aws ec2 describe-dhcp-options \
  --dhcp-options-ids dopt-0123456789abcdef0 \
  --query 'DhcpOptions[*].DhcpConfigurations[*].{Key:Key,Values:Values[*].Value}' \
  --output json
```

Why DHCP options matter:

- They can define the domain name and DNS servers used by instances.
- A custom DHCP options set can break name resolution if it points to unreachable or incorrect DNS servers.
- DNS issues are often misdiagnosed as application failures, so always verify VPC DNS settings early.

**IPv6 inspection**

IPv6 introduces different routing and egress patterns. A subnet may have an IPv6 CIDR block, and routes may point to an egress-only Internet Gateway for outbound-only IPv6 internet access.

Inspect IPv6 subnet association:

```bash
aws ec2 describe-subnets \
  --subnet-ids subnet-0123456789abcdef0 \
  --query 'Subnets[0].Ipv6CidrBlockAssociationSet'
```

Inspect IPv6 routes:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'RouteTables[*].Routes[*].{
    DestinationIpv6:Ipv6CidrBlock,
    GatewayId:GatewayId,
    EgressOnlyInternetGatewayId:EgressOnlyInternetGatewayId,
    State:State
  }' \
  --output table
```

Useful IPv6 checks:

- Confirm the subnet has an IPv6 CIDR block.
- Confirm the route table includes `::/0` to the correct target.
- Use an **egress-only Internet Gateway** for outbound-only IPv6 internet access.
- Security groups and NACLs must also allow IPv6 traffic explicitly using IPv6 CIDRs.

Example security group rule for IPv6:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --ip-permissions '[
    {
      "IpProtocol":"tcp",
      "FromPort":443,
      "ToPort":443,
      "IpRanges":[],
      "Ipv6Ranges":[{"CidrIpv6":"2001:db8::/32"}]
    }
  ]'
```

**Transit Gateway, VPC peering, and propagated routes**

When traffic crosses VPC boundaries or on-premises networks, route propagation becomes important.

Inspect Transit Gateway attachments:

```bash
aws ec2 describe-transit-gateway-attachments \
  --filters Name=resource-id,Values=vpc-0123456789abcdef0 \
  --query 'TransitGatewayAttachments[*].{AttachmentId:TransitGatewayAttachmentId,ResourceId:ResourceId,State:State,ResourceType:ResourceType}' \
  --output table
```

Inspect VPC peering connections:

```bash
aws ec2 describe-vpc-peering-connections \
  --filters Name=requester-vpc-info.vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'VpcPeeringConnections[*].{PeeringId:VpcPeeringConnectionId,Status:Status.Code,RequesterVpc:RequesterVpcInfo.VpcId,AccepterVpc:AccepterVpcInfo.VpcId}' \
  --output table
```

Route propagation clues:

- Propagated routes often appear automatically after VPN or Transit Gateway configuration.
- Static routes are manually added and are easier to reason about during troubleshooting.
- If a route should exist but does not, check whether propagation is enabled on the route table or attachment.

Inspect route propagation settings:

```bash
aws ec2 describe-route-tables \
  --route-table-ids rtb-0123456789abcdef0 \
  --query 'RouteTables[0].PropagatingVgws'
```

For Transit Gateway environments, also inspect the route table on the Transit Gateway side if you have access, because the VPC route table alone does not tell the full story.

**VPC flow logs**

Flow logs help you determine whether traffic is being accepted or rejected at the VPC, subnet, or ENI level. They do not show packet payloads, but they are very useful for confirming whether traffic reaches the network layer.

Inspect flow logs:

```bash
aws ec2 describe-flow-logs \
  --filter Name=resource-id,Values=vpc-0123456789abcdef0 \
  --query 'FlowLogs[*].{FlowLogId:FlowLogId,ResourceId:ResourceId,ResourceType:ResourceType,TrafficType:TrafficType,LogDestinationType:LogDestinationType,DeliverLogsStatus:DeliverLogsStatus}' \
  --output table
```

What to look for:

- `ACCEPT` or `REJECT` records can confirm whether traffic is blocked before the application layer.
- If you see `REJECT`, compare the source, destination, port, and protocol with security group and NACL rules.
- If you see no records, confirm the flow log is enabled for the correct resource and that the destination is configured correctly.

Example CloudWatch Logs query idea:

```sql
fields @timestamp, srcAddr, dstAddr, srcPort, dstPort, protocol, action
| filter action = "REJECT"
| sort @timestamp desc
| limit 20
```

**Load balancers and target group reachability**

If the workload sits behind an Elastic Load Balancing (ELB) service, inspect the load balancer and target group before blaming the instance.

Inspect load balancers:

```bash
aws elbv2 describe-load-balancers \
  --query 'LoadBalancers[*].{Name:LoadBalancerName,Type:Type,Scheme:Scheme,VpcId:VpcId,State:State.Code,SecurityGroups:SecurityGroups}' \
  --output table
```

Inspect target groups:

```bash
aws elbv2 describe-target-groups \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/app/example/50dc6c495c0c9188 \
  --query 'TargetGroups[*].{TargetGroupArn:TargetGroupArn,Protocol:Protocol,Port:Port,TargetType:TargetType,VpcId:VpcId}' \
  --output table
```

Inspect target health:

```bash
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/example/6d0ecf831eec9f09 \
  --output table
```

Why this matters:

- A target can be unhealthy because the instance security group does not allow traffic from the load balancer security group.
- The target group port may not match the application listener port.
- Health checks can fail if the route table, NACL, or host firewall blocks the health check path.

**End-to-end troubleshooting walkthrough**

Use a structured approach when an instance cannot reach the internet or cannot be reached from the internet.

1. Identify the instance and its subnet.
   ```bash
   aws ec2 describe-instances \
     --instance-ids i-0123456789abcdef0 \
     --query 'Reservations[0].Instances[0].{SubnetId:SubnetId,VpcId:VpcId,PrivateIp:PrivateIpAddress,PublicIp:PublicIpAddress,SecurityGroups:SecurityGroups[*].GroupId}' \
     --output json
   ```

2. Find the effective route table for the subnet.
   ```bash
   aws ec2 describe-route-tables \
     --filters Name=association.subnet-id,Values=subnet-0123456789abcdef0 \
     --output table
   ```
   If nothing returns, inspect the VPC main route table.

3. Verify the default route target.
   - Public subnet: `0.0.0.0/0 -> igw-...`
   - Private subnet: `0.0.0.0/0 -> nat-...`
   - IPv6 subnet: `::/0 -> eigw-...` for outbound-only internet access

4. Confirm the gateway exists and is attached.
   ```bash
   aws ec2 describe-internet-gateways --filters Name=attachment.vpc-id,Values=vpc-0123456789abcdef0
   aws ec2 describe-nat-gateways --filter Name=vpc-id,Values=vpc-0123456789abcdef0
   ```

5. Check security groups and NACLs.
   ```bash
   aws ec2 describe-security-groups --group-ids sg-0123456789abcdef0
   aws ec2 describe-network-acls --filters Name=association.subnet-id,Values=subnet-0123456789abcdef0
   ```

6. Verify DNS and DHCP settings if name resolution fails.
   ```bash
   aws ec2 describe-vpc-attribute --vpc-id vpc-0123456789abcdef0 --attribute enableDnsSupport
   aws ec2 describe-vpc-attribute --vpc-id vpc-0123456789abcdef0 --attribute enableDnsHostnames
   aws ec2 describe-vpcs --vpc-ids vpc-0123456789abcdef0 --query 'Vpcs[0].DhcpOptionsId'
   ```

7. Review flow logs for `REJECT` entries.
   ```bash
   aws ec2 describe-flow-logs --filter Name=resource-id,Values=vpc-0123456789abcdef0
   ```

8. If a load balancer is involved, verify target health.
   ```bash
   aws elbv2 describe-target-health --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/example/6d0ecf831eec9f09
   ```

**Practical scenario**

Suppose a private EC2 instance cannot reach `https://example.com`:

- The instance has only a private IPv4 address.
- The subnet route table has no `0.0.0.0/0` route to a NAT Gateway.
- The NAT Gateway exists, but it is in a different subnet and the route table points to a deleted NAT Gateway.
- Flow logs show `REJECT` for outbound connections.

Likely fix:

- Restore the route to a healthy NAT Gateway.
- Confirm the NAT Gateway is in a public subnet with a route to an Internet Gateway.
- Ensure the security group allows outbound HTTPS if egress is restricted.
- Ensure the NACL allows outbound and return ephemeral ports.

**Exam focus**

For SAA-C03, you should be able to reason about networking by reading AWS CLI output and connecting it to traffic flow.

- **Route tables** determine where packets go.
- **Security groups** control instance-level stateful access.
- **NACLs** control subnet-level stateless access.
- **DNS settings** affect name resolution and hostname behavior.
- **Flow logs** help confirm whether traffic is accepted or rejected.
- **Gateways and attachments** must exist and be in the correct state.
- **Public IP and Elastic IP** assignment often explains internet reachability issues.
- **IPv6** uses different route targets and should be checked separately.
