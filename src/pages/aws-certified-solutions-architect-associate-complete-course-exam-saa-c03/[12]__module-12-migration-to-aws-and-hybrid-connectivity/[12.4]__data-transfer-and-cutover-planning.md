## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Data transfer and cutover planning overview**

Cutover planning is the controlled transition from a source environment to a target AWS environment. In exam scenarios, you choose the transfer method, reduce risk before go-live, validate the target, and define rollback if something fails.

A good plan answers four questions:

- How do you move the data?
- How do you minimize downtime?
- How do you prove the target is ready?
- How do you recover if cutover fails?

For SAA-C03, expect tradeoffs between speed, consistency, cost, and operational complexity. The best answer is often the one that reduces downtime while preserving a safe rollback path.

**Core migration patterns**

Different workloads need different transfer approaches. You usually choose based on data size, network bandwidth, change rate, and acceptable downtime.

- **Online transfer** keeps the source running while data is copied and changes are replicated.
- **Offline transfer** pauses writes or takes the system down during the move.
- **Hybrid cutover** stages data first, then performs a short final sync and switch.
- **Blue/green deployment** runs old and new environments in parallel and shifts traffic gradually or all at once.

Common AWS tools and where they fit:

- **AWS DataSync** moves files between supported storage systems such as NFS, SMB, Amazon S3, Amazon EFS, and Amazon FSx.
- **AWS Database Migration Service (AWS DMS)** migrates databases with minimal downtime using full load plus change data capture (CDC).
- **AWS Schema Conversion Tool (AWS SCT)** helps convert schemas and code for heterogeneous database migrations.
- **AWS Snow Family** supports large-scale offline data transfer when network transfer is impractical.
- **Amazon S3 replication** copies objects between buckets, often for cross-Region resilience or migration staging.
- **AWS Backup**, EBS snapshots, AMIs, and database snapshots provide rollback and recovery points.

**Choosing the right transfer method**

Use the transfer method that matches the workload and the cutover window.

| Scenario | Better fit | Why |
|---|---|---|
| Large file shares with ongoing changes | DataSync | Incremental transfer and scheduling reduce manual effort |
| Relational database migration with low downtime | DMS + SCT | Full load plus CDC supports near-zero-downtime cutover |
| Very large datasets with limited bandwidth | Snow Family | Offline transfer avoids long network copy times |
| Static website or object data migration | S3 replication or batch copy | Simple object movement with low operational overhead |
| EC2 server migration | AMIs, snapshots, or AWS Application Migration Service | Preserves server state and reduces rebuild effort |

> Use caution with “verification” claims. DataSync validates transfer integrity, but it is not a universal application-consistency checker for every workload.

**AWS DataSync planning**

DataSync is useful for file and object transfers when you need automation, scheduling, and incremental syncs. It works well for migrations, ongoing replication, and pre-cutover staging.

Typical use cases:

- Migrate on-premises NFS or SMB file shares to Amazon EFS, FSx, or S3.
- Pre-stage data before a maintenance window.
- Keep a target in sync until final cutover.

Practical considerations:

- Deploy a DataSync agent when transferring from on-premises storage.
- Ensure network connectivity, firewall rules, and bandwidth are ready before the task runs.
- Run an initial bulk transfer early, then schedule incremental syncs closer to cutover.
- Validate file counts, sizes, and application-level access after transfer.

Example CLI flow for DataSync is usually:

1. Create the source location.
2. Create the destination location.
3. Create the task using the returned location ARNs.
4. Start the task execution.
5. Monitor completion and re-run incremental syncs as needed.

Example pattern:

```bash
aws datasync create-location-nfs \
  --server-hostname nfs.example.internal \
  --subdirectory /exports/data \
  --on-prem-config AgentArns=arn:aws:datasync:us-east-1:123456789012:agent/agent-0123456789abcdef0

aws datasync create-location-s3 \
  --s3-bucket-arn arn:aws:s3:::my-target-bucket \
  --s3-config BucketAccessRoleArn=arn:aws:iam::123456789012:role/DataSyncS3Access

aws datasync create-task \
  --source-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-11111111111111111 \
  --destination-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-22222222222222222 \
  --name "nfs-to-s3-cutover"

aws datasync start-task-execution \
  --task-arn arn:aws:datasync:us-east-1:123456789012:task/task-33333333333333333
```

The important exam idea is not the exact ARN value, but the sequence: create locations first, then create the task with those location ARNs.

**AWS DMS and database cutover**

AWS DMS is a common answer for database migrations when you need minimal downtime. It performs an initial full load and can continue with CDC to capture ongoing source changes.

Key cutover concepts:

- **Replication instance** runs the migration task and needs enough CPU, memory, and network throughput.
- **Source and target endpoints** must be reachable and correctly configured.
- **Task settings** control LOB handling, logging, and table mappings.
- **CDC lag** must be near zero before you switch traffic.
- **Task stop/start behavior** matters during cutover and rollback planning.

Practical cutover flow:

1. Start the DMS task with full load and CDC.
2. Let the target catch up until CDC lag is minimal.
3. Quiesce writes on the source application.
4. Wait for the CDC queue to drain completely.
5. Stop the task only after you confirm replication is caught up.
6. Point the application to the target database.
7. Validate application behavior and data consistency.

Important exam points:

- You often stop application writes before final cutover to avoid split-brain updates.
- You should confirm that the last replicated transaction has been applied before switching.
- If the cutover fails, you may restart the source writes and re-run CDC if the source remains authoritative.
- For heterogeneous migrations, use AWS SCT first to convert schema objects, stored procedures, and SQL dialect differences.

Example DMS endpoint creation pattern:

```bash
aws dms create-endpoint \
  --endpoint-identifier source-postgres \
  --endpoint-type source \
  --engine-name postgres \
  --username appuser \
  --password 'SecretPass123!' \
  --server-name source-db.example.internal \
  --port 5432 \
  --database-name appdb

aws dms create-endpoint \
  --endpoint-identifier target-aurora \
  --endpoint-type target \
  --engine-name aurora-postgresql \
  --username appuser \
  --password 'SecretPass123!' \
  --server-name target-cluster.cluster-abcdefghijkl.us-east-1.rds.amazonaws.com \
  --port 5432 \
  --database-name appdb
```

**Schema conversion planning with AWS SCT**

AWS SCT is important when the source and target database engines differ. It helps you identify incompatible objects before migration day.

Use SCT to:

- Assess conversion effort.
- Convert schema objects and code where possible.
- Identify manual remediation for unsupported features.
- Plan testing for stored procedures, triggers, views, and SQL syntax differences.

Exam-relevant guidance:

- Homogeneous migrations, such as PostgreSQL to PostgreSQL, usually need less conversion work.
- Heterogeneous migrations, such as Oracle to Aurora PostgreSQL, often require schema redesign and code changes.
- Schema conversion should happen before data transfer so you do not discover incompatibilities during cutover.

**Snow Family and offline transfer**

AWS Snow Family is used when network transfer is too slow, too expensive, or not feasible for very large datasets. It is especially useful for bulk data migration to Amazon S3 and for some edge and disaster recovery workflows.

Use Snow Family when:

- You need to move many terabytes or petabytes.
- The network link is unreliable or too slow.
- You want to seed a cloud migration before incremental online sync.

Common exam framing:

- Snowball is often used to import data into Amazon S3.
- Snowball can also support export workflows from S3.
- Some Snow workflows support EBS snapshot transfer, but S3 is the primary and most common destination in migration scenarios.

> Do not assume Snow Family is the best answer for small or frequently changing datasets. For those, online replication or DataSync is usually better.

**Backup and rollback planning**

Rollback is not optional. Before cutover, create a recovery point that matches the workload.

Useful rollback mechanisms:

- **Amazon RDS snapshots** for managed relational databases.
- **EBS snapshots** for block storage and EC2-backed workloads.
- **AMIs** for EC2 instance rollback or redeployment.
- **AWS Backup** for centralized backup policy management across supported services.
- **S3 versioning** and replication for object rollback.
- **Database native backups** when you need engine-specific restore options.

Practical rollback patterns:

- Restore an RDS snapshot to a new instance if the target database has issues.
- Revert an EC2 application tier by launching from a known-good AMI.
- Roll back an EBS-backed server by restoring the latest snapshot and reattaching volumes.
- Re-enable the source system if DNS or load balancer cutover fails and the source is still intact.

A strong cutover plan includes:

- A pre-cutover backup taken close to the change window.
- A clear decision point for rollback.
- A tested restore procedure, not just a backup file.

**DNS planning with Route 53**

DNS cutover is common, but DNS is not instant. Client resolvers cache records according to TTL (time to live), and some clients or intermediate resolvers may ignore short TTLs longer than expected.

Best practices:

- Lower TTL well in advance of cutover, often 24 to 48 hours before the change.
- Use a low TTL only after the old higher TTL has expired from caches.
- Expect some clients to continue using the old endpoint until their cached record expires.
- Plan for long-lived connections that may remain open even after DNS changes.

Route 53 routing policies relevant to cutover:

- **Weighted routing** shifts a percentage of DNS responses to different endpoints.
- **Failover routing** sends traffic to a primary endpoint and fails over to a secondary endpoint when health checks fail.
- **Latency-based routing** chooses the lowest-latency endpoint for the client.
- **Geolocation routing** directs users based on location.

Weighted routing is useful for gradual traffic shifting, but it does not guarantee session continuity or connection draining. It is DNS-based, so application sessions may still break if the backend changes.

Example Route 53 record set pattern:

```json
{
  "Name": "app.example.com",
  "Type": "A",
  "SetIdentifier": "blue",
  "Weight": 90,
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "blue-alb-123456.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

A safer cutover approach is to combine low TTL, health checks, and a rollback-ready source environment.

**Load balancer cutover**

For many web applications, you cut over by switching target groups behind an Application Load Balancer (ALB) or Network Load Balancer (NLB).

Important concepts:

- **Target group switching** lets you move traffic from old instances to new ones.
- **Health checks** ensure only healthy targets receive traffic.
- **Deregistration delay** gives in-flight requests time to finish before a target is removed.
- **Connection draining** is the practical effect of waiting for active connections to complete.

Typical ALB cutover pattern:

1. Register the new targets in a separate target group.
2. Confirm health checks pass.
3. Shift listener rules or default actions to the new target group.
4. Keep the old target group available for rollback.
5. Wait for active sessions to drain before terminating old resources.

Example AWS CLI update for a listener default action:

```bash
aws elbv2 modify-listener \
  --listener-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:listener/app/my-alb/50dc6c495c0c9188/6f7e8a9b0c1d2e3f \
  --default-actions Type=forward,TargetGroupArn=arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/new-tg/73e2d6bc24d8a067
```

For NLB, the same general idea applies, but remember that NLB is layer 4 and does not provide the same application-layer routing features as ALB.

**Health checks and failover**

Health checks are critical when you want automated failover or safe traffic shifting. They help Route 53, load balancers, and orchestration systems decide whether a target is ready.

Use health checks to:

- Detect failed endpoints.
- Prevent traffic from going to unhealthy targets.
- Support Route 53 failover routing.
- Validate new targets before increasing traffic.

Route 53 failover is often a better exam answer than weighted routing when the goal is primary/secondary resilience rather than gradual release.

**Validation after cutover**

Validation should prove that the target works for users, not just that the data copied successfully.

Good validation checks include:

- Application login and session creation.
- Read and write operations.
- Database query results and row counts.
- File upload and retrieval.
- Background job execution.
- Monitoring and log review.
- Error rate, latency, and throughput comparison.

Service-specific examples:

- **RDS/Aurora**: compare row counts, run application smoke tests, confirm parameter group and security group settings.
- **S3**: verify object counts, prefixes, permissions, lifecycle rules, and application access.
- **EFS**: confirm mount targets, POSIX permissions, and file visibility from application hosts.
- **EC2-based workloads**: verify instance health, attached volumes, IAM roles, user data, and service startup.
- **DMS migrations**: compare source and target table counts, check CDC lag, and validate recent transactions.

A practical validation checklist:

1. Confirm the application reaches the new endpoint.
2. Verify authentication and authorization.
3. Test critical user journeys.
4. Compare key business records.
5. Review logs and metrics for anomalies.
6. Keep the source available until validation passes.

**Rollback strategies by service**

Rollback should be specific, not generic.

- **DNS rollback**: restore the old Route 53 record or increase weight back to the source.
- **Load balancer rollback**: switch the listener back to the old target group.
- **Database rollback**: restore from snapshot, promote the old replica, or reverse replication if the source remains authoritative.
- **EC2 rollback**: launch from the previous AMI or restore EBS snapshots.
- **S3 rollback**: use versioning, object restore, or replication failback.
- **DMS rollback**: stop the target application, resume source writes, and re-establish CDC if the source remains intact.

For database cutovers, a common safe pattern is:

- Keep the source database read-only or paused rather than deleting it.
- Take a final snapshot before switching.
- Retain the old replication setup until the new system is stable.

**Multi-account and multi-Region considerations**

Cutover planning becomes more complex when the source and target are in different accounts or Regions.

Key concerns:

- **Cross-Region latency** can affect replication lag and application performance.
- **Data transfer costs** may increase with inter-Region movement.
- **IAM permissions** must allow the migration service to access both sides.
- **KMS keys** may need cross-account or cross-Region planning for encrypted data.
- **Regional failover** should be tested before production cutover.

Common patterns:

- Seed data in the target Region, then use replication or CDC for final sync.
- Use Route 53 failover or weighted routing to shift users between Regions.
- Keep backups in a separate Region for disaster recovery.
- Validate that dependencies such as secrets, certificates, and parameter stores exist in the target Region.

**Practical cutover runbook**

A simple runbook helps reduce mistakes during the maintenance window.

1. Freeze nonessential changes.
2. Take a final backup or snapshot.
3. Run the last incremental sync or CDC catch-up.
4. Verify health checks and application readiness.
5. Switch traffic using DNS, load balancer rules, or database endpoints.
6. Validate critical user flows.
7. Monitor logs, metrics, and alarms.
8. Keep rollback available until the system is stable.

**Common exam traps**

- Assuming DNS cutover is immediate for all users.
- Treating weighted routing as a true session-preserving migration method.
- Using Snow Family for small or frequently changing datasets.
- Forgetting schema conversion work in heterogeneous database migrations.
- Ignoring CDC lag before database cutover.
- Deleting the source too early and losing rollback options.
- Relying on transfer completion alone instead of application validation.
- Overlooking load balancer health checks and deregistration delay.

**Quick decision guide**

- Use **DMS** for database migrations with minimal downtime.
- Use **SCT** when database engines differ.
- Use **DataSync** for file and object transfers with incremental sync.
- Use **Snow Family** for very large offline transfers.
- Use **Route 53 failover** for primary/secondary resilience.
- Use **weighted routing** for gradual traffic shifting, not guaranteed session continuity.
- Use **snapshots, AMIs, and AWS Backup** for rollback readiness.

**Key takeaways**

Cutover planning is about controlled change, not just copying data. The best migration plan stages data early, validates the target thoroughly, shifts traffic carefully, and preserves a rollback path until the new environment is proven stable.

For SAA-C03, focus on matching the tool to the workload, understanding DNS and load balancer behavior, and planning for rollback before you need it.
