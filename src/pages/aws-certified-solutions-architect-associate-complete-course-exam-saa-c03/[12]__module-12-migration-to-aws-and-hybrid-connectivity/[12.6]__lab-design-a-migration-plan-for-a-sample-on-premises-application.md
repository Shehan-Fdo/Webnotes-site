## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Migration planning overview**

A migration plan turns an existing on-premises application into a sequence of low-risk steps for AWS. The goal is not just to “move servers,” but to preserve functionality, reduce downtime, and choose the right migration pattern for each component.

In practice, you start by understanding the application’s dependencies, traffic patterns, data stores, and operational constraints. Then you map each part to an AWS target that fits its needs: rehost, replatform, refactor, retire, retain, or replace. For exam scenarios, the best answer usually balances speed, risk, cost, and operational effort.

- **Discovery** identifies what the application uses and what depends on it.
- **Dependency mapping** shows which systems must move together.
- **Migration strategy** defines how much change you make during the move.
- **Cutover plan** defines how you switch users from old to new.
- **Rollback plan** protects you if validation fails.

**Common migration approaches**

AWS migration planning often uses the “6 Rs” model. You choose the approach based on business urgency, technical debt, and how much modernization you can afford during the migration.

| Approach | Meaning | When to use |
|---|---|---|
| Rehost | Lift and shift with minimal changes | Fast migration, limited time, legacy app |
| Replatform | Small optimizations without redesign | You want quick wins like managed databases |
| Refactor | Redesign for cloud-native architecture | You need scalability, resilience, or agility |
| Retire | Decommission unused systems | The app or component is no longer needed |
| Retain | Keep on-premises for now | Compliance, latency, or dependency blockers |
| Replace | Move to SaaS or managed product | A commercial service meets the requirement |

A sample on-premises application often includes a web tier, application tier, and database tier. The migration plan should treat each tier separately if their constraints differ. For example, you might rehost the web servers, replatform the database to Amazon RDS, and later refactor the application layer to use Amazon SQS or Amazon DynamoDB if the design calls for it.

**Assessment and discovery**

Before you design the target architecture, you need a clear inventory of the source environment. This is where you identify what exists, what is critical, and what can be changed safely.

- **Application inventory** includes servers, services, ports, OS versions, and installed software.
- **Data inventory** includes databases, file shares, object stores, and backup schedules.
- **Network inventory** includes subnets, firewall rules, DNS records, and VPN or Direct Connect links.
- **Operational inventory** includes monitoring, patching, identity, and incident response processes.

Useful discovery questions include:

- What is the application’s peak and average traffic?
- Which components are stateful?
- What are the recovery time objective (RTO) and recovery point objective (RPO)?
- Which integrations are synchronous and which are asynchronous?
- Are there hard dependencies on local hardware, licenses, or IP allowlists?

A practical way to document dependencies is to build a simple matrix.

| Component | Depends on | Dependency type | Migration note |
|---|---|---|---|
| Web server | Load balancer, app server | Network/runtime | Can usually be rehosted first |
| App server | Database, cache, identity provider | Data/service | Often needs coordinated cutover |
| Database | Backup system, reporting jobs | Stateful | Requires replication and validation |
| Batch job | File share, scheduler | File/data | May need Amazon S3 or Amazon EFS |

**Target architecture design**

A good migration plan defines the target state before the move begins. In AWS, the target architecture should improve availability, security, and manageability where possible without introducing unnecessary risk.

Typical building blocks include:

- **Amazon VPC** for network isolation and segmentation.
- **Application Load Balancer** for HTTP/HTTPS traffic distribution.
- **Amazon EC2** for rehosted application servers.
- **Amazon RDS** or **Amazon Aurora** for managed relational databases.
- **Amazon S3** for static assets, backups, and data exchange.
- **Amazon EFS** when shared POSIX file storage is required.
- **Amazon CloudWatch** for logs, metrics, and alarms.
- **AWS IAM** for access control and least privilege.

A common pattern is a multi-AZ deployment for production workloads. Multi-AZ improves availability by placing resources across separate Availability Zones. For databases, managed Multi-AZ options reduce failover complexity and are often preferred over self-managed replication when the exam asks for high availability with less operational overhead.

Example target topology:

```text
Users
  |
Route 53
  |
Application Load Balancer
  |
+---------------------------+
|        Auto Scaling       |
|  EC2 App Servers (AZ-a/b) |
+---------------------------+
  |
Amazon RDS Multi-AZ
```

If the application depends on shared files, you may need to preserve that behavior during migration.

```text
EC2 App Servers ---> Amazon EFS
EC2 App Servers ---> Amazon RDS
EC2 App Servers ---> Amazon S3 (static content / exports)
```

**Migration tooling and data transfer**

The right AWS service depends on what you are moving.

- **AWS Application Migration Service (MGN)** is used for lift-and-shift server migrations with continuous block-level replication.
- **AWS Database Migration Service (DMS)** is used to migrate databases with minimal downtime, often with ongoing replication.
- **AWS DataSync** is used to move file data to Amazon S3, Amazon EFS, or Amazon FSx.
- **AWS Snowball** is useful for very large offline transfers when network bandwidth is a bottleneck.
- **AWS Migration Hub** helps track migration progress across multiple workloads.

For a sample application, you might combine tools:

- Use MGN for the web and application servers.
- Use DMS for the relational database.
- Use DataSync for shared file storage.
- Use Route 53 for DNS cutover after validation.

Example DMS task flow:

```text
On-prem database -> DMS replication instance -> Amazon RDS target
```

A basic DMS migration often follows this pattern:

```bash
aws dms create-replication-instance \
  --replication-instance-identifier app-migration-ri \
  --replication-instance-class dms.t3.medium \
  --allocated-storage 100

aws dms create-endpoint \
  --endpoint-identifier source-db \
  --endpoint-type source \
  --engine-name mysql \
  --username admin \
  --password 'REDACTED' \
  --server-name onprem-db.example.local \
  --port 3306
```

**Network connectivity design**

Migration projects usually need secure connectivity between on-premises and AWS during replication and testing. The choice depends on bandwidth, latency, and duration.

- **Site-to-Site VPN** is quick to deploy and suitable for smaller or temporary migrations.
- **AWS Direct Connect** provides more consistent throughput and lower latency for large or long-running migrations.
- **Transit Gateway** can simplify routing when multiple VPCs or networks are involved.

For exam questions, choose VPN when speed of setup matters and throughput needs are moderate. Choose Direct Connect when the workload is large, steady, or latency-sensitive. In many real migrations, you use both: Direct Connect for primary connectivity and VPN as backup.

Important network planning items:

- Ensure DNS resolution works across environments.
- Open only required ports between source and target.
- Validate routing for replication traffic and application testing.
- Plan for overlapping CIDR blocks before creating VPC connectivity.

> If the on-premises network and AWS VPC use overlapping CIDR ranges, routing becomes difficult. You may need to redesign addressing or use a different connectivity approach.

**Identity, security, and access**

A migration plan should preserve security controls while adapting them to AWS.

- **IAM roles** replace long-lived credentials where possible.
- **Security groups** act as stateful instance-level firewalls.
- **Network ACLs** provide subnet-level stateless filtering.
- **AWS KMS** encrypts data at rest for supported services.
- **AWS Secrets Manager** or **SSM Parameter Store** stores application secrets more safely than config files.

For a migrated application, you should avoid hardcoding credentials in scripts or AMIs. Instead, inject secrets at runtime and use instance profiles or task roles.

Example secure configuration pattern:

```bash
export DB_HOST=mydb.cluster-abc123.us-east-1.rds.amazonaws.com
export DB_USER=appuser
export DB_PASSWORD="$(aws secretsmanager get-secret-value \
  --secret-id prod/app/db \
  --query SecretString \
  --output text)"
```

Security planning also includes:

- Reviewing compliance requirements before moving regulated data.
- Encrypting backups and replication traffic where supported.
- Restricting administrative access with MFA and least privilege.
- Logging access and configuration changes for auditability.

**Cutover strategy**

Cutover is the point where users begin using the AWS-hosted application. The best cutover strategy depends on how much downtime the business can tolerate.

Common patterns:

- **Big bang cutover** moves everything at once.
  - Use when the app is small or downtime is acceptable.
  - Risk is higher because rollback is more disruptive.
- **Phased cutover** moves components in stages.
  - Use when tiers can be separated safely.
  - Reduces risk and makes validation easier.
- **Blue/green deployment** keeps two environments and switches traffic.
  - Use when you want fast rollback and controlled release.
  - Works well for stateless tiers and modernized apps.

A practical phased migration might look like this:

1. Replicate the database to AWS and validate data consistency.
2. Deploy the application tier in AWS and point it to the replicated database.
3. Test internal users and integration endpoints.
4. Update DNS to send production traffic to the AWS load balancer.
5. Monitor errors, latency, and database load.
6. Decommission the old environment only after a stable period.

Example Route 53 cutover concept:

```text
app.example.com
  -> old on-prem load balancer
  -> new AWS Application Load Balancer
```

If you need a low-risk switch, lower the DNS TTL before cutover so clients refresh records faster.

```dns
app.example.com.  60  IN  A  203.0.113.10
```

**Validation and testing**

Testing is not optional in a migration plan. You need to prove that the application behaves correctly in AWS before final cutover.

- **Functional testing** verifies that user workflows still work.
- **Performance testing** checks response times and throughput.
- **Data validation** confirms row counts, checksums, and business totals.
- **Failover testing** confirms the system survives an AZ or instance failure.
- **Security testing** checks access controls, encryption, and logging.

Useful validation commands and checks:

```bash
curl -I https://app.example.com
nslookup app.example.com
aws elbv2 describe-target-health --target-group-arn arn:aws:elasticloadbalancing:...
aws rds describe-db-instances --db-instance-identifier prod-db
```

For database validation, compare source and target at a business level, not just by raw size. A table count match is helpful, but it does not guarantee that all transactions replicated correctly. For critical systems, use application-level reconciliation.

**Rollback planning**

A rollback plan is the safety net for failed cutover. It should be written before migration day and tested if possible.

- Keep the source environment intact until the new environment is stable.
- Preserve DNS or load balancer settings so you can revert quickly.
- Snapshot or back up the target before major changes.
- Define the exact conditions that trigger rollback.

Rollback is easier when you avoid irreversible changes during the first cutover. For example, if you switch writes to the AWS database, make sure you understand how to reverse replication or re-seed the source if needed.

**Operational readiness**

After migration, the application must be supportable in AWS. Operational readiness is often overlooked, but it is essential for a successful migration.

- Set up **CloudWatch alarms** for CPU, memory, latency, and error rates.
- Centralize logs in CloudWatch Logs or S3.
- Automate patching with Systems Manager where appropriate.
- Document runbooks for restart, failover, and incident response.
- Confirm backup retention and restore procedures.

Example CloudWatch alarm creation:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name high-cpu-app-server \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 300 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --evaluation-periods 2
```

**Exam-focused decision points**

When you see migration questions, look for the constraint that matters most.

- **Fastest move with minimal change** usually means rehost.
- **Need managed database benefits** usually means replatform to RDS or Aurora.
- **Need major scalability or resilience gains** usually means refactor.
- **Temporary coexistence with on-premises systems** usually means hybrid connectivity.
- **Large data transfer with limited bandwidth** may point to Snowball.
- **Minimal downtime database migration** often points to DMS.

Also pay attention to hidden requirements:

- If the app needs shared file storage, EFS may be required.
- If the app is stateless, Auto Scaling and blue/green become easier.
- If the database is the bottleneck, moving only the web tier will not solve the problem.
- If compliance requires data to stay on-premises, retain that component until constraints change.

**Practical migration checklist**

Use a simple checklist to keep the plan organized.

1. Inventory the application, data, and dependencies.
2. Classify each component using the 6 Rs.
3. Choose the target AWS services and network design.
4. Set up connectivity and security controls.
5. Replicate data and deploy the target environment.
6. Test functionality, performance, and failover.
7. Plan DNS or load balancer cutover.
8. Define rollback conditions and recovery steps.
9. Monitor closely after go-live.
10. Decommission the source only after validation and approval.

**Key takeaways**

A strong migration plan reduces risk by separating discovery, design, testing, and cutover. For AWS SAA-C03 scenarios, the best answer usually reflects the least disruptive path that still meets availability, security, and operational goals. Choose the migration pattern that fits each component, not just the whole application.
