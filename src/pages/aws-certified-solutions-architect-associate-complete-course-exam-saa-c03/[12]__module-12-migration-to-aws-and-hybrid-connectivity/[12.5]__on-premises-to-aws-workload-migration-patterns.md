## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Migration strategy overview**

On-premises to AWS migration is usually a portfolio exercise, not a single server move. You assess applications, dependencies, data gravity, downtime tolerance, compliance constraints, and target architecture before choosing a migration pattern. In exam scenarios, the best answer often depends on whether you need speed, minimal change, lower risk, or a long-term modernization path.

Common migration outcomes include:

- **Rehost**: lift and shift to EC2 with minimal changes.
- **Replatform**: make small changes, such as moving a database to Amazon RDS.
- **Refactor**: redesign for cloud-native services such as containers or serverless.
- **Retire**: decommission unused systems.
- **Retain**: keep some workloads on premises for now.
- **Relocate**: move VMware-based workloads to AWS without changing the guest OS or app much.

A practical migration plan usually starts with discovery, then landing zone preparation, then pilot migrations, then wave-based execution, and finally cutover and optimization.

**Assessment and discovery**

Before you migrate, you need a clear inventory of what exists and how systems depend on each other. This reduces surprises during cutover and helps you sequence shared services correctly.

Key AWS services and concepts for assessment:

- **AWS Application Discovery Service**: collects inventory, utilization, and dependency data from on-premises servers.
- **AWS Migration Hub**: central place to track migration progress across tools, applications, and waves.
- **AWS Server Migration Service (SMS)**: legacy server migration service that may still appear in exam contexts, but AWS Migration Hub and AWS Application Migration Service are the modern direction.
- **AWS Application Migration Service (AWS MGN)**: preferred service for lift-and-shift server migration to AWS.

Discovery should answer:

- Which applications are internet-facing, internal, batch, or stateful?
- Which systems depend on shared services such as **Active Directory (AD)**, **Lightweight Directory Access Protocol (LDAP)**, **Simple Mail Transfer Protocol (SMTP)** relays, file shares, DNS, or NTP?
- Which workloads have strict recovery point objective (**RPO**) and recovery time objective (**RTO**) requirements?
- Which databases can tolerate replication lag and which need near-zero downtime?
- Which systems can be retired instead of migrated?

A dependency map matters because shared services often need to move first or be made available in AWS before application waves begin.

**Landing zone prerequisites**

A landing zone is the baseline AWS environment where migrated workloads land. For exam purposes, think of it as the foundation that makes migration safe, repeatable, and governable.

Typical landing zone requirements:

- **Multi-account structure** for separation of production, non-production, shared services, logging, and security.
- **AWS Organizations** for centralized account governance and service control policies.
- **IAM boundaries** and least-privilege roles for migration teams and automation.
- **CloudTrail** enabled in all accounts for audit logging.
- **AWS Config** enabled for configuration history and compliance checks.
- **Baseline networking** with VPCs, subnets, route tables, security groups, network ACLs, and connectivity to on-premises.
- **Centralized logging and monitoring** with CloudWatch, CloudTrail, and optionally S3-based log archives.
- **Security controls** such as KMS keys, guardrails, and inspection points where required.

A common exam trap is assuming you can migrate first and design governance later. In practice, you need at least a minimal landing zone before production cutover.

**Migration patterns and when to use them**

Different workloads justify different migration patterns. The right choice depends on business urgency, technical debt, and how much change the organization can absorb.

| Pattern | Best when | Typical AWS outcome |
|---|---|---|
| Rehost | You need speed and low change | EC2, EBS, Auto Scaling, ALB |
| Replatform | You want some managed services without a full redesign | RDS, ElastiCache, managed storage |
| Refactor | You want cloud-native scale or agility | Containers, Lambda, event-driven services |
| Relocate | You want to move VMware workloads with minimal app change | VMware Cloud on AWS or similar lift-and-shift approach |
| Retire | The workload is no longer needed | Decommission and archive data |
| Retain | The workload must stay on premises for now | Hybrid architecture |

For exam questions, choose **rehost** when the goal is to move quickly with minimal code change. Choose **replatform** when a small service substitution reduces operational burden. Choose **refactor** when the question explicitly asks for modernization, elasticity, or microservices.

> If the workload is tightly coupled to legacy hardware, unsupported OS features, or local appliances, a pure lift-and-shift answer may be less appropriate than a phased hybrid approach.

**Server migration options**

AWS provides multiple ways to move servers, and the exam may reference older and newer services.

- **AWS Application Migration Service (AWS MGN)**:
  - Preferred for lift-and-shift server migration.
  - Replicates source servers into AWS staging resources.
  - Supports cutover with minimal downtime.
  - Often uses a lightweight agent or installer on supported source systems, but the exact setup varies by source type and deployment method.
- **AWS Server Migration Service (SMS)**:
  - Older, exam-relevant legacy service.
  - Historically used for virtual machine migration.
  - Know it as a predecessor concept, not the primary modern choice.
- **VMware Cloud on AWS**:
  - Useful when you want to move VMware-based workloads with minimal re-architecture.
  - Can be a bridge strategy before deeper modernization.

A good migration wave often starts with low-risk, low-dependency systems to validate tooling, networking, and operational runbooks.

**Database migration patterns**

Database migrations are often the hardest part because they combine schema, data, application compatibility, and downtime constraints.

Common AWS services:

- **AWS Database Migration Service (AWS DMS)**:
  - Migrates data between databases.
  - Supports homogeneous and heterogeneous migrations.
  - Can use **change data capture (CDC)** to keep source and target in sync during cutover.
  - Does not replace full schema design, stored procedure conversion, or application testing.
- **AWS Schema Conversion Tool (AWS SCT)**:
  - Helps convert schema and code when moving between engines.
  - Useful for heterogeneous migrations such as Oracle to Aurora PostgreSQL.
- **Amazon RDS** and **Amazon Aurora**:
  - Common targets for managed database replatforming.

Important exam nuance:

- DMS is primarily for **data replication and migration**, not a complete database application migration by itself.
- Even same-engine migrations may still require schema creation, permissions, parameter tuning, and validation outside DMS.

Practical cutover strategies:

1. **Full backup and restore** for simple, low-change migrations with acceptable downtime.
2. **CDC-based migration** with DMS for reduced downtime.
3. **Blue/green style cutover** where the new database is warmed up and then traffic is switched.
4. **Read replica promotion** when supported by the engine and architecture.
5. **Dual-write or temporary synchronization** only when the application design can safely support it.

Example DMS task settings often emphasize ongoing replication:

```json
{
  "MigrationType": "full-load-and-cdc",
  "TargetMetadata": {
    "TargetSchema": "app"
  },
  "FullLoadSettings": {
    "TargetTablePrepMode": "DROP_AND_CREATE"
  }
}
```

For a low-downtime migration, you typically:

- Load the initial data.
- Keep CDC running.
- Validate row counts and application behavior.
- Freeze writes briefly.
- Apply final changes.
- Switch the application endpoint.

**Application dependency handling**

Application migration is not just about moving compute. Shared services can break a wave if they are not sequenced correctly.

Common dependencies to plan for:

- **Active Directory / LDAP**:
  - Identity and authentication often need to be available in AWS before application cutover.
  - You may extend AD into AWS or use AWS Managed Microsoft AD depending on the use case.
- **DNS**:
  - Name resolution must work across on-premises and AWS during coexistence.
- **SMTP relays**:
  - Applications that send email may need a new relay or updated allow lists.
- **File shares**:
  - Shared storage may need migration to Amazon EFS, Amazon FSx, or S3 depending on access pattern.
- **Licensing servers**:
  - Some commercial software depends on on-premises license managers.
- **Monitoring and logging**:
  - Agents, log forwarders, and alerting should be ready before production traffic moves.

A common sequencing approach is:

1. Migrate shared identity and network services.
2. Establish connectivity and DNS resolution.
3. Move low-risk application tiers.
4. Move stateful back-end systems.
5. Cut over user-facing services last.

**Route 53 cutover patterns**

Amazon Route 53 is often used to shift traffic from on-premises endpoints to AWS endpoints during migration.

Useful routing patterns:

- **Weighted routing**:
  - Send a percentage of traffic to AWS and the rest to on-premises.
  - Good for gradual cutover and canary testing.
- **Failover routing**:
  - Keep on-premises as primary and AWS as secondary, or vice versa.
  - Good for disaster recovery or staged migration.
- **Latency-based routing**:
  - Route users to the lowest-latency endpoint when you have multiple regional deployments.
- **Simple alias or CNAME changes**:
  - Useful for straightforward endpoint swaps, but less flexible than weighted or failover routing.

Practical considerations:

- Use **TTL (time to live)** values that are low enough to support cutover, but not so low that they create unnecessary DNS churn.
- Prefer **alias records** when pointing to AWS resources that support them, such as ALB or CloudFront.
- Use **CNAME** only where appropriate; remember it cannot be used at the zone apex.
- Test DNS propagation before the final cutover window.

Example weighted record idea:

```text
app.example.com
- 90% -> on-premises load balancer
- 10% -> AWS Application Load Balancer
```

This pattern lets you validate performance and error rates before full migration.

**Connectivity patterns: VPN, Direct Connect, and failover**

Hybrid migration usually needs reliable connectivity between on-premises and AWS.

- **AWS Site-to-Site VPN**:
  - Fast to deploy.
  - Uses the public internet.
  - Good for initial connectivity, smaller environments, or backup paths.
- **AWS Direct Connect**:
  - Dedicated private network connection.
  - More consistent latency and bandwidth.
  - Often preferred for large data transfers and steady hybrid traffic.

A common design is:

- Use **Direct Connect** as the primary path.
- Use **VPN** as backup or failover.
- Use dynamic routing with **BGP (Border Gateway Protocol)** where appropriate.

This is a common exam pattern because it balances performance and resilience. Direct Connect improves predictability, while VPN provides a fallback if the dedicated link fails.

**Direct Connect Gateway and Transit Gateway**

These services are often confused, so the topology matters.

- **Direct Connect Gateway (DXGW)**:
  - Extends a Direct Connect connection to multiple virtual private gateways and, in supported designs, to Transit Gateways.
  - It does not by itself create transitive routing to arbitrary VPCs.
  - Cross-Region use has specific constraints and is not a universal “connect everything” feature.
- **Transit Gateway (TGW)**:
  - Central hub for connecting multiple VPCs and on-premises networks.
  - Simplifies hub-and-spoke routing.
- **DXGW to TGW association**:
  - Used when you want Direct Connect traffic to reach networks attached to a Transit Gateway.

Correct mental model:

- Direct Connect gives you the private circuit.
- DXGW helps extend that circuit to AWS network constructs.
- TGW centralizes routing among VPCs and hybrid attachments.

> Do not think of this as “attaching a Direct Connect Gateway to a Transit Gateway” in a generic sense. The supported association model and routing design matter.

**Data transfer options**

AWS offers multiple ways to move large datasets, and the right choice depends on bandwidth, time, and whether you need edge compute.

- **AWS DataSync**:
  - Automates online data transfer between on-premises storage and AWS storage services.
  - Common for NFS, SMB, and object storage transfers.
  - Useful when you need scheduled, incremental, or recurring transfers.
- **AWS Snow Family**:
  - Physical devices for offline transfer when network bandwidth is insufficient.
  - Includes different device classes for different scale and edge needs.
- **AWS Snowcone**:
  - Small, portable edge device.
  - Good for constrained environments, rugged field use, or small-scale edge compute and data collection.
- **AWS Snowball Edge**:
  - Larger device with more storage and compute.
  - Good for bulk offline transfer and edge processing.
- **AWS Snowmobile**:
  - Extremely large-scale data transfer using a shipping container.
  - Used for massive migrations, not typical exam day-to-day scenarios.

When to choose each:

- Use **DataSync** when you have network connectivity and want automated online transfer.
- Use **Snowcone** when you need portability, small footprint, or edge compute in constrained environments.
- Use **Snowball Edge** when you need bulk offline transfer or more substantial edge processing.
- Use **Snowmobile** when the data volume is so large that network transfer is impractical.

Example DataSync agent deployment pattern:

```bash
aws datasync create-agent \
  --activation-key ABCD-1234-EFGH-5678 \
  --agent-name onprem-fileserver-agent
```

Example NFS location creation:

```bash
aws datasync create-location-nfs \
  --server-hostname nfs01.example.local \
  --subdirectory /exports/data \
  --on-prem-config AgentArns=arn:aws:datasync:us-east-1:123456789012:agent/agent-0123456789abcdef0
```

Example task creation:

```bash
aws datasync create-task \
  --source-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-0123456789abcdef0 \
  --destination-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-0fedcba9876543210 \
  --name migrate-files
```

**Cutover planning and validation**

A migration is not complete until you validate the application in AWS and confirm rollback options.

Good cutover practice includes:

- Freeze changes on the source system.
- Run final synchronization or replication catch-up.
- Validate application health, authentication, and data consistency.
- Switch DNS or load balancer traffic.
- Monitor logs, metrics, and user experience.
- Keep a rollback plan for a defined window.

Validation checks often include:

- Application login and authorization.
- Database row counts and sample queries.
- File access and permissions.
- Batch job execution.
- External integrations such as email, APIs, and payment gateways.

A rollback plan should be explicit. If the new environment fails validation, you need to know whether to revert DNS, re-enable source writes, or restore from backup.

**Migration waves and sequencing**

Large migrations are usually executed in waves. A wave groups applications with similar dependencies, risk, or business priority.

A practical wave structure:

1. **Wave 0**: landing zone, connectivity, identity, logging, and pilot workloads.
2. **Wave 1**: low-risk internal apps and non-production systems.
3. **Wave 2**: business-critical but well-understood applications.
4. **Wave 3**: complex, stateful, or highly integrated systems.
5. **Wave 4**: remaining edge cases, legacy systems, and retirement candidates.

Sequence shared services early:

- AD/LDAP
- DNS
- SMTP
- file services
- monitoring and backup

This reduces the chance that later waves fail because a dependency was overlooked.

**Migration Hub and governance**

AWS Migration Hub helps coordinate the migration program across tools and teams. It is especially useful when you have multiple applications, multiple migration methods, and multiple waves.

Use it to:

- Track application status across discovery and migration tools.
- Coordinate ownership and progress.
- Provide a central view for program management.

In exam terms, Migration Hub is the coordination layer, not the migration engine itself.

**Common exam decision points**

When you see a migration question, ask:

- Is the goal to move fast with minimal change?
- Is downtime tightly constrained?
- Is the database the main challenge?
- Is the workload better modernized than copied?
- Are there shared services that must move first?
- Is the data volume too large for network transfer?
- Is hybrid connectivity temporary or long term?

Typical answers:

- **Lift and shift to EC2** when speed matters and the app can run as-is.
- **RDS/Aurora replatforming** when you want managed database operations.
- **Containers or serverless** when the question emphasizes modernization, elasticity, or reduced server management.
- **Direct Connect with VPN backup** when you need reliable hybrid connectivity.
- **DataSync** for online file transfer.
- **Snowball Edge** for large offline transfer.
- **Snowcone** for portable edge or small-scale transfer.
- **Route 53 weighted routing** for gradual traffic shifting.
- **DMS with CDC** for low-downtime database migration.

**Practical migration example**

A common enterprise pattern looks like this:

1. Build a landing zone with separate accounts for shared services, logging, security, and workloads.
2. Establish connectivity using Direct Connect, with VPN as backup.
3. Use Application Discovery Service to inventory servers and dependencies.
4. Register applications in Migration Hub and group them into waves.
5. Migrate identity, DNS, and file services first.
6. Rehost low-risk servers with AWS MGN.
7. Move file data with DataSync.
8. Migrate databases with DMS and CDC, using SCT if schema conversion is needed.
9. Cut over traffic with Route 53 weighted or failover routing.
10. Validate, monitor, and optimize after cutover.

**Quick comparison reference**

| Need | Best fit | Notes |
|---|---|---|
| Fast server lift-and-shift | AWS MGN | Preferred modern server migration service |
| Inventory and dependency mapping | Application Discovery Service | Helps plan waves and dependencies |
| Central migration tracking | Migration Hub | Coordinates tools and progress |
| Online file transfer | DataSync | Good for recurring or incremental sync |
| Large offline transfer | Snowball Edge | Bulk data movement |
| Small portable edge use | Snowcone | Compact and rugged |
| Database data replication | DMS | Use CDC for low downtime |
| Schema conversion | SCT | Needed for heterogeneous migrations |
| Gradual traffic shift | Route 53 weighted routing | Useful for canary cutover |
| Hybrid private connectivity | Direct Connect | Often paired with VPN backup |

**Key takeaways**

- Start with discovery, dependency mapping, and a landing zone.
- Use Migration Hub to coordinate the program and Application Discovery Service to understand the environment.
- Choose the migration pattern based on business goals: rehost, replatform, refactor, retire, retain, or relocate.
- Use AWS MGN for server lift-and-shift, DMS for database data movement, and DataSync or Snow Family for file and bulk transfer.
- Plan shared services, DNS, identity, and connectivity before application cutover.
- Use Route 53 and hybrid networking patterns to shift traffic safely and support rollback.
