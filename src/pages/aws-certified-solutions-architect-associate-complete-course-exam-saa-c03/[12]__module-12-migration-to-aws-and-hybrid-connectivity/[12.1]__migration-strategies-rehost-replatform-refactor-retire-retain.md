## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Migration strategy overview**

The 5Rs are a practical framework for deciding how to move an application to AWS. You use them to balance speed, cost, operational effort, and the amount of change you want to introduce. In exam scenarios, the right answer usually depends on whether the goal is to move quickly, reduce management overhead, modernize the application, or avoid change entirely.

The five strategies are:

- **Rehost**: move as-is, often called “lift and shift”
- **Replatform**: make a small optimization during migration
- **Refactor**: redesign parts of the application for cloud-native benefits
- **Retire**: decommission the application or workload
- **Retain**: keep the application where it is for now

A useful mental model is:

- If you need speed and low risk, start with **rehost**
- If you want a small improvement without major redesign, choose **replatform**
- If you need long-term agility and cloud-native architecture, choose **refactor**
- If the system is no longer needed, **retire** it
- If migration is not justified yet, **retain** it

**Rehost: lift and shift**

Rehost moves an application to AWS with minimal or no changes. This is often the fastest migration path and is common when you need to exit a data center quickly, reduce hardware costs, or meet a deadline.

Typical use cases:

- Legacy applications that are stable but not worth redesigning
- Large portfolios where you want to migrate first and optimize later
- Workloads with limited time, budget, or engineering capacity

Practical AWS examples:

- Move a VM from on-premises to an **Amazon EC2** instance
- Copy application files and configuration to EC2 and keep the same runtime
- Use **AWS Application Migration Service** to replicate servers and cut over with minimal downtime

Example migration flow:

```bash
# Example: verify an EC2 instance after cutover
aws ec2 describe-instances \
  --filters "Name=tag:Name,Values=legacy-app-prod" \
  --query "Reservations[].Instances[].{ID:InstanceId,State:State.Name,IP:PublicIpAddress}"
```

Why you choose it:

- Fastest path to AWS
- Lowest application change effort
- Easier to estimate and execute at scale

Trade-offs:

- You may carry over technical debt
- You do not automatically gain cloud-native benefits
- Operational patterns may remain similar to on-premises

**Replatform: lift, tinker, and shift**

Replatform keeps the core application architecture intact but introduces a few targeted improvements during migration. The goal is to gain some cloud benefits without a full redesign.

Common examples:

- Move a database from self-managed MySQL on EC2 to **Amazon RDS**
- Replace a local file server with **Amazon S3**
- Use **Amazon ElastiCache** to offload repeated reads
- Move from self-managed web servers to managed scaling on EC2 with an **Application Load Balancer**

Why you choose it:

- You want better availability, scalability, or manageability
- You want to reduce operational burden without rewriting the app
- You want a moderate modernization step with limited risk

Example configuration idea:

```yaml
# Example: application points to managed database endpoint
DB_HOST: mydb.abc123xyz.us-east-1.rds.amazonaws.com
DB_PORT: 3306
DB_NAME: appdb
```

Trade-offs:

- More effort than rehost
- Less flexibility than refactor
- Some application changes may still be required, especially for connection handling, storage paths, or session state

**Refactor: re-architect for cloud-native design**

Refactor changes the application significantly so it can take advantage of AWS managed services, elasticity, and distributed design patterns. This is the most transformative option and usually the most expensive and time-consuming.

Common refactor patterns:

- Break a monolith into microservices
- Move from synchronous processing to event-driven workflows
- Replace server-based components with managed services
- Use **AWS Lambda**, **Amazon API Gateway**, **Amazon SQS**, **Amazon SNS**, or **Step Functions** for decoupled design

Why you choose it:

- You need faster feature delivery and better scalability
- You want to reduce operational overhead over the long term
- The existing architecture limits reliability, agility, or cost efficiency

Example event-driven pattern:

```text
Client -> API Gateway -> Lambda -> SQS -> Worker Lambda -> DynamoDB
```

Practical benefits:

- Better horizontal scaling
- Easier fault isolation
- Smaller deployment units
- More opportunities to pay only for what you use

Trade-offs:

- Highest engineering effort
- Requires testing, redesign, and often data model changes
- Can introduce distributed systems complexity such as retries, idempotency, and eventual consistency

**Retire: decommission what you do not need**

Retire means you stop using the application or workload and remove it from service. This is often the best answer when the system is redundant, obsolete, or replaced by another platform.

Examples:

- A legacy reporting tool replaced by a modern SaaS product
- An internal application no longer used by the business
- Duplicate systems created during a merger or acquisition

Why you choose it:

- Reduces cost immediately
- Eliminates maintenance and security exposure
- Simplifies the portfolio

Operational considerations:

- Confirm business ownership before shutdown
- Archive data if retention rules apply
- Validate dependencies so you do not break downstream systems

A safe retirement checklist often includes:

1. Identify users, integrations, and data retention requirements
2. Export or archive required data
3. Disable inbound traffic and scheduled jobs
4. Monitor for unexpected dependency failures
5. Decommission infrastructure and remove access

**Retain: keep it where it is for now**

Retain means you leave the application in its current environment, usually temporarily. This is a valid strategy when migration is not practical or not worth the risk at the moment.

Common reasons to retain:

- The application has a hard dependency on local hardware or low-latency on-premises systems
- Licensing, compliance, or vendor constraints make migration difficult
- The workload is too risky to move during the current phase
- The application is already scheduled for retirement later

Examples:

- A mainframe workload that is not ready for migration
- A system tightly coupled to on-premises identity or manufacturing equipment
- A workload with a short remaining life that does not justify migration cost

Why you choose it:

- Avoids unnecessary risk
- Buys time for planning, dependency mapping, or budget approval
- Lets you migrate other workloads first

Trade-offs:

- You continue paying for the current environment
- You do not gain AWS benefits for that workload yet
- Retaining too much can delay modernization indefinitely

**How to choose the right strategy**

The exam often tests your ability to match business goals to the least disruptive valid option. Use the following decision logic:

- Choose **rehost** when speed matters most and the app can move with minimal change
- Choose **replatform** when you want a small cloud optimization without a rewrite
- Choose **refactor** when the business wants long-term cloud-native benefits and accepts redesign effort
- Choose **retire** when the workload is no longer needed
- Choose **retain** when migration is not feasible or not justified right now

A simple comparison:

| Strategy | Change level | Speed | Cost to migrate | Long-term benefit | Typical use |
|---|---:|---:|---:|---:|---|
| Rehost | Low | High | Low | Low to moderate | Fast migration |
| Replatform | Low to medium | High to medium | Moderate | Moderate | Small optimization |
| Refactor | High | Low | High | High | Cloud-native redesign |
| Retire | None | High | Low | Immediate savings | Remove unused systems |
| Retain | None | N/A | None | None now | Defer migration |

**AWS services commonly associated with migration**

Several AWS services appear frequently in migration scenarios:

- **AWS Application Migration Service** for server replication and cutover
- **AWS Database Migration Service (DMS)** for database migration with minimal downtime
- **AWS Schema Conversion Tool (SCT)** for schema and code conversion during database modernization
- **Amazon EC2** for rehosted workloads
- **Amazon RDS** for replatformed databases
- **Amazon S3** for object storage and static content
- **AWS Lambda** and managed messaging services for refactored architectures

Example database migration pattern:

```text
On-prem database -> AWS DMS replication instance -> Amazon RDS target
```

Example CLI check for a migration target:

```bash
aws rds describe-db-instances \
  --db-instance-identifier app-prod-db \
  --query "DBInstances[0].{Status:DBInstanceStatus,Endpoint:Endpoint.Address,Engine:Engine}"
```

**Exam tips and common traps**

- If the question says “move quickly with minimal changes,” think **rehost**
- If it says “move to AWS and use a managed database,” think **replatform**
- If it says “redesign for microservices or serverless,” think **refactor**
- If it says “application is no longer needed,” think **retire**
- If it says “cannot migrate now due to constraints,” think **retain**

Watch for wording that implies:

- **Lift and shift** = rehost
- **Lift, tinker, and shift** = replatform
- **Re-architect** = refactor
- **Decommission** = retire
- **Keep as-is** = retain

> If the question asks for the least disruptive option, do not overcomplicate the answer with modernization unless the prompt explicitly requires it.

**Practical migration planning considerations**

Before choosing a strategy, assess the workload across technical and business dimensions:

- Application dependencies
- Data gravity and database size
- Downtime tolerance
- Security and compliance requirements
- Licensing constraints
- Team skills and operational maturity
- Budget and timeline

A common phased approach is:

1. **Assess** the portfolio and classify workloads by complexity
2. **Mobilize** the landing zone, networking, identity, and security controls
3. **Migrate** with the appropriate 5R strategy
4. **Modernize** selected workloads after the initial move

This phased model is useful because many organizations do not refactor everything at once. They often rehost first, then replatform or refactor the highest-value applications later.

**Quick memory aid**

- **Rehost**: move it
- **Replatform**: improve it a little
- **Refactor**: redesign it
- **Retire**: remove it
- **Retain**: leave it
