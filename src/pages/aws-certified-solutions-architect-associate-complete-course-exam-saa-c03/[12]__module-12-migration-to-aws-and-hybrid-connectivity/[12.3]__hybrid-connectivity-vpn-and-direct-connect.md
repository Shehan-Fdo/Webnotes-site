## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Hybrid connectivity overview**

Hybrid connectivity links your on-premises network to AWS so workloads can span both environments. In exam scenarios, you usually choose between **AWS Site-to-Site VPN**, **AWS Direct Connect (DX)**, or a combination of both.

- Use **VPN** when you need fast setup, encryption, and internet-based connectivity.
- Use **Direct Connect** when you need predictable latency, higher throughput, and private network transport.
- Use both when you want DX as the primary path and VPN as encrypted backup or failover.

A common architecture is:

```text
On-prem router
   |\
   | \__ Internet --> AWS Site-to-Site VPN --> VGW or TGW --> VPCs
   |
   \____ Cross connect at DX location --> Direct Connect --> DXGW --> VGW/TGW --> VPCs
```

**AWS Site-to-Site VPN fundamentals**

AWS Site-to-Site VPN creates an encrypted IPsec tunnel between your customer gateway device and AWS. It is the standard answer when the question emphasizes encryption, quick deployment, or a low-setup hybrid link.

Core components:

- **Customer gateway (CGW)**: your on-premises VPN device or software appliance.
- **Virtual private gateway (VGW)**: the AWS-side VPN endpoint attached to a VPC.
- **Transit gateway (TGW)**: a hub that can terminate VPNs and route to multiple VPCs.
- **Tunnel endpoints**: AWS provides two tunnels per VPN connection for high availability.
- **BGP (Border Gateway Protocol)**: used for dynamic route exchange when you want automatic failover.

A VPN connection is not just “a tunnel.” It is a managed pair of tunnels with routing and endpoint configuration. For exam questions, remember that AWS gives you two tunnels so you can design for resilience.

**VPN routing: static vs dynamic**

AWS Site-to-Site VPN supports both static and dynamic routing, and the choice affects failover behavior.

- **Static routing**
  - You manually define routes.
  - Use it when the customer gateway does not support BGP.
  - It is simpler, but failover is less flexible.
- **Dynamic routing with BGP**
  - Routes are exchanged automatically.
  - Preferred for production and failover scenarios.
  - If one tunnel fails, BGP can withdraw routes and shift traffic to the other tunnel.

Practical exam guidance:

- Choose **BGP** when the question mentions automatic route failover, multiple prefixes, or enterprise routing.
- Choose **static routes** when the device is limited or the scenario explicitly says no BGP support.
- If you use a **VGW**, route propagation can automatically inject learned routes into the VPC route table.
- If you use a **TGW**, route tables and propagation control which attachments can reach each other.

Example route behavior with BGP:

```text
On-premises prefix 10.10.0.0/16
  -> advertised over Tunnel 1 and Tunnel 2
  -> AWS prefers the active path based on BGP attributes
  -> if Tunnel 1 fails, Tunnel 2 remains available
```

**VPN with Virtual Private Gateway**

A **virtual private gateway** is the AWS VPN endpoint attached to a VPC. It must be in the same Region as the VPC.

Typical use case:

- One VPC needs encrypted connectivity to on-premises.
- You want a straightforward site-to-site design.
- You do not need a hub for many VPCs.

Important points:

- A VGW attaches to one VPC at a time.
- It supports route propagation into the VPC route table.
- It is a common exam answer for “connect one VPC to on-premises with VPN.”

Example AWS CLI flow:

```bash
# Create a virtual private gateway
aws ec2 create-vpn-gateway --type ipsec.1

# Attach it to a VPC in the same Region
aws ec2 attach-vpn-gateway \
  --vpn-gateway-id vgw-0123456789abcdef0 \
  --vpc-id vpc-0123456789abcdef0
```

> A VPN connection also requires a **customer gateway** and a **VPN connection** resource. The tunnel endpoints are created by AWS, but you still configure your on-premises device with the provided parameters.

**VPN with Transit Gateway**

A **transit gateway** is the preferred hub when you need many VPCs, multiple accounts, or centralized routing. It can terminate VPN connections and route traffic to attached VPCs and other network attachments.

Why use VPN with TGW:

- You want one hub for multiple VPCs.
- You want centralized inspection or segmentation.
- You need scalable hybrid routing across a larger environment.

Exam-relevant behavior:

- TGW route tables control which attachments can communicate.
- Route propagation can simplify hybrid routing.
- You can design **hub-and-spoke** connectivity with on-premises at the hub.

Example topology:

```text
On-premises
   |
   |-- Site-to-Site VPN
   |
Transit Gateway
  |      |      |
 VPC-A  VPC-B  VPC-C
```

**Direct Connect fundamentals**

**AWS Direct Connect** provides a dedicated network connection from your premises to AWS through a DX location. It is not instant like VPN; it is provisioned through a partner or colocation workflow.

Key provisioning concepts:

- **DX location**: the physical facility where AWS has presence.
- **Cross connect**: the physical fiber connection from your router or colocation cage to AWS.
- **LOA-CFA (Letter of Authorization and Connecting Facility Assignment)**: the document that authorizes the cross connect.
- **Partner or colocation workflow**: you usually order DX through a network provider or colocated facility, then complete the physical setup.

Use Direct Connect when you need:

- More consistent latency and throughput than the public internet.
- Private connectivity for large data transfers.
- A stable hybrid link for enterprise workloads.

Important exam distinction:

- DX is about **private transport**, not encryption by default.
- If the question requires encryption, the usual answer is **VPN** or **VPN over Direct Connect**.

**Direct Connect connection types**

Direct Connect supports different virtual interface types, and the exam often tests which one fits the target traffic.

- **Private virtual interface (private VIF)**
  - Connects to private AWS resources in a VPC.
  - Typically used with VGW or DXGW.
- **Transit virtual interface (transit VIF)**
  - Connects to a Transit Gateway through a Direct Connect Gateway.
  - Useful for multi-VPC hub designs.
- **Public virtual interface (public VIF)**
  - Reaches supported AWS public services over DX.
  - Requires public IP prefixes and BGP advertisement.

Public VIF considerations:

- You advertise public prefixes that you own or are authorized to use.
- It is not for private VPC CIDRs.
- It is used for AWS public endpoints, not general private VPC routing.

**Direct Connect Gateway**

A **Direct Connect Gateway (DXGW)** lets you connect a Direct Connect connection to one or more gateways in a broader design. It is especially useful when you need to reach multiple VPCs through a hub.

What to remember:

- DXGW is commonly used with **private VIFs** and **transit VIFs**.
- It can help connect DX to a **VGW** or **TGW**.
- It is often used for multi-Region designs.
- Association rules and routing constraints matter; do not assume unrestricted cross-account or cross-Region reachability without the correct association model.

Exam-safe way to think about it:

- Use **VGW** for a simple single-VPC private DX design.
- Use **TGW + DXGW** for centralized multi-VPC hybrid routing.
- Use **DXGW** when you need to extend DX beyond a single gateway attachment model.

**Direct Connect routing and BGP**

Direct Connect uses **BGP** to exchange routes between your network and AWS. You advertise prefixes from on-premises, and AWS advertises reachable prefixes back to you.

Practical routing points:

- BGP is required for route exchange on DX virtual interfaces.
- You must advertise only the prefixes you intend to reach.
- Route preference can be influenced by your on-premises routing policy and AWS-side design.

Example conceptual route preference:

```text
Primary path: Direct Connect
Backup path: Site-to-Site VPN

On-prem router:
- Prefer DX routes with lower local preference
- Keep VPN routes as backup
- If DX fails, BGP withdraws DX routes and VPN becomes active
```

**Direct Connect CLI example**

A private VIF requires a structured JSON object for `--new-private-virtual-interface`. The exact fields depend on the interface and gateway target.

```bash
aws directconnect create-private-virtual-interface \
  --connection-id dxcon-abc12345 \
  --new-private-virtual-interface '{
    "virtualInterfaceName": "private-vif",
    "vlan": 101,
    "asn": 65000,
    "authKey": "optional-shared-secret",
    "amazonAddress": "169.254.100.2/30",
    "customerAddress": "169.254.100.1/30",
    "addressFamily": "ipv4",
    "virtualGatewayId": "vgw-0123456789abcdef0"
  }'
```

Notes:

- The JSON structure is the important part.
- Use the correct target, such as a VGW or DXGW association model.
- The exact address fields and options depend on the design and AWS requirements.

**Public VIF example**

A public VIF is used for AWS public services, not private VPC CIDRs.

```bash
aws directconnect create-public-virtual-interface \
  --connection-id dxcon-abc12345 \
  --new-public-virtual-interface '{
    "virtualInterfaceName": "public-vif",
    "vlan": 102,
    "asn": 65000,
    "addressFamily": "ipv4",
    "amazonAddress": "169.254.200.2/30",
    "customerAddress": "169.254.200.1/30",
    "routeFilterPrefixes": [
      { "cidr": "203.0.113.0/24" }
    ]
  }'
```

**Resiliency patterns**

High availability in hybrid networking usually comes from combining multiple layers of redundancy.

- **VPN dual tunnels**
  - AWS provides two tunnels per VPN connection.
  - Use both tunnels with BGP for automatic failover.
- **Dual Direct Connect connections**
  - Use separate physical connections for better resilience.
  - Prefer diverse devices and, ideally, diverse locations.
- **LAG (Link Aggregation Group)**
  - Increases bandwidth and can provide link-level redundancy.
  - It does not protect against a full location failure by itself.
- **DX plus VPN backup**
  - Common enterprise pattern.
  - DX is primary; VPN is encrypted backup over the internet.
- **Active/passive**
  - One path carries traffic, the other waits.
  - Simpler and common for backup designs.
- **Active/active**
  - Both paths carry traffic.
  - Requires careful routing policy and consistent prefix advertisement.

> A single Direct Connect connection is not highly available by itself. For true resilience, use diverse paths, separate devices, or separate locations.

**VPN over Direct Connect**

You can run an IPsec VPN tunnel over a Direct Connect path when you need encryption but still want the dedicated transport characteristics of DX. This is a customer-managed IPsec overlay, not a separate AWS-managed hybrid product.

When to choose it:

- You need encryption in transit.
- You want to avoid sending the encrypted tunnel over the public internet.
- You want DX as the underlying transport.

When not to overstate it:

- It is not a replacement for all DX designs.
- It adds encryption overhead.
- It still depends on the underlying DX path.

**End-to-end hybrid architecture example**

A common exam scenario is an enterprise with one on-premises data center and multiple VPCs.

Design:

- On-premises router connects to:
  - a **Direct Connect** circuit for primary private connectivity
  - a **Site-to-Site VPN** for backup and encryption
- DX terminates at a **DX location**
- DX connects to a **DXGW**
- DXGW associates with a **TGW**
- TGW connects to multiple VPCs

Text diagram:

```text
On-prem router
   |\
   | \__ Internet --> VPN --> TGW --> VPC-A / VPC-B
   |
   \____ DX location --> DX --> DXGW --> TGW --> VPC-A / VPC-B
```

How traffic behaves:

- Normal state: on-prem prefers DX routes.
- Failure state: if DX fails, BGP withdraws those routes.
- Backup state: VPN routes remain available and take over.
- Multi-VPC routing: TGW propagates routes to the correct attachments.

This pattern is common because it balances performance, resilience, and encryption options.

**Exam decision guide**

Use the following mental model when choosing a service:

- Choose **Site-to-Site VPN** when:
  - you need encryption
  - you need quick setup
  - you can tolerate internet-based performance
- Choose **Direct Connect** when:
  - you need predictable network performance
  - you transfer large volumes of data
  - you want private connectivity
- Choose **Direct Connect plus VPN** when:
  - you want DX as primary and VPN as backup
  - you need encryption over a dedicated path
- Choose **Transit Gateway** when:
  - you connect multiple VPCs
  - you need centralized hybrid routing
  - you want hub-and-spoke connectivity
- Choose **Virtual Private Gateway** when:
  - you have a simpler single-VPC VPN or private DX design

**Troubleshooting and operational checks**

Hybrid connectivity issues often come from routing, security, or provisioning mismatches rather than the tunnel or circuit itself.

Check these first:

- **Security groups** and **network ACLs** allow the traffic.
- VPC route tables contain the correct propagated or static routes.
- On-premises BGP sessions are established.
- The correct tunnel or VIF is in an available state.
- The target gateway association is correct.

Useful commands and checks:

```bash
# View VPN connections
aws ec2 describe-vpn-connections

# View virtual private gateways
aws ec2 describe-vpn-gateways

# View transit gateway attachments
aws ec2 describe-transit-gateway-attachments

# View Direct Connect virtual interfaces
aws directconnect describe-virtual-interfaces
```

Operational symptoms:

- If traffic reaches AWS but not the instance, check security groups, NACLs, and instance OS firewall.
- If routes are missing, check BGP advertisements and route propagation.
- If failover does not happen, verify that both tunnels or both DX paths are actually configured and preferred correctly.

**Common exam traps**

- Confusing **VPN encryption** with **Direct Connect private transport**.
- Assuming **DX** is instantly provisioned like a cloud resource.
- Forgetting that **VPN has two tunnels** by default for high availability.
- Using **VGW** when the question clearly needs a hub for multiple VPCs.
- Assuming **LAG** alone solves all redundancy needs.
- Treating **public VIF** as a private VPC connection.
- Ignoring **TGW route tables** in multi-VPC hybrid designs.
- Overstating DXGW reachability without the correct gateway association model.

**Quick comparison**

| Feature | Site-to-Site VPN | Direct Connect |
|---|---|---|
| Transport | Internet-based | Dedicated private circuit |
| Encryption | Yes, IPsec | No by default |
| Setup speed | Fast | Slower, partner/colocation workflow |
| Performance | Variable | More consistent |
| High availability | Two tunnels per connection | Use diverse connections/locations |
| Best for | Secure quick hybrid access | Predictable throughput and latency |

**Practical design patterns**

- **Small environment**
  - One VPC
  - VGW
  - Site-to-Site VPN with BGP if possible
- **Enterprise hub**
  - TGW
  - Multiple VPCs
  - VPN and/or DX into the hub
- **Performance-sensitive hybrid**
  - DX primary
  - VPN backup
  - BGP-based failover
- **Encrypted dedicated transport**
  - DX with VPN overlay
  - Use when encryption is required and DX is available

**Key takeaways**

- **VPN** is the default answer for encrypted hybrid connectivity.
- **Direct Connect** is the default answer for predictable private connectivity.
- **BGP** is central to dynamic routing and failover.
- **TGW** is the scalable hub for multiple VPCs and hybrid routing.
- **DXGW** extends Direct Connect into broader hub designs, but association and routing rules matter.
- Resiliency comes from **multiple tunnels, multiple paths, and diverse locations**, not from a single link.
