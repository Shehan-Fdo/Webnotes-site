## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**AWS global infrastructure overview**

AWS organizes infrastructure into geographic and network layers so you can balance latency, fault tolerance, and compliance. The main building blocks are **Regions**, **Availability Zones (AZs)**, **Local Zones**, **Wavelength Zones**, and **Edge Locations**.

A good architecture choice starts with the question: do you need:
- lower latency to users,
- high availability across isolated failure domains,
- data residency or regulatory control,
- or edge delivery for static and dynamic content?

Each location type solves a different problem. For the exam, you should know not only what each one is, but also when to use it.

**Regions**

A **Region** is a physical geographic area, such as `us-east-1` or `eu-west-1`, that contains multiple Availability Zones. Regions are isolated from one another for fault tolerance and compliance.

Use a Region when you need:
- to place resources near users or data sources,
- to meet legal or regulatory requirements,
- to design for disaster recovery across geography,
- to choose from services that may vary by Region.

Important Region characteristics:
- **Isolation**: a Region is independent from other Regions.
- **Service availability**: not every AWS service or feature is available in every Region.
- **Data residency**: you can keep data within a specific geographic boundary.
- **Latency**: choosing a nearby Region can reduce round-trip time.

Common operational examples:
- Deploy a production stack in `us-east-1` and a disaster recovery stack in `us-west-2`.
- Use a Region close to your primary user base for application servers and databases.
- Replicate backups to another Region for recovery.

You can inspect available Regions with the AWS CLI:

```bash
aws ec2 describe-regions --all-regions
```

You can also target a specific Region in commands:

```bash
aws s3 ls --region us-east-1
aws ec2 describe-instances --region eu-west-1
```

**Availability Zones**

An **Availability Zone** is one or more discrete data centers within a Region, connected by high-bandwidth, low-latency networking. AZs are designed to be physically separated from each other so that a failure in one AZ does not automatically affect the others.

Use multiple AZs when you need:
- high availability,
- fault tolerance,
- automatic failover,
- load balancing across isolated infrastructure.

Key AZ concepts:
- **Independent power, cooling, and networking**: AZs are engineered to reduce correlated failures.
- **Low-latency connectivity**: AZs in the same Region are close enough for synchronous replication in many designs.
- **AZ IDs vs AZ names**: the lettered AZ name can differ between accounts, while the underlying AZ ID is consistent. This matters when sharing architecture diagrams or coordinating across accounts.

Typical multi-AZ patterns:
- An Application Load Balancer in front of EC2 instances in at least two AZs.
- An Amazon RDS Multi-AZ deployment for automatic failover.
- An Auto Scaling group spanning multiple AZs.

Example topology:
- Public subnets in two AZs for load balancers.
- Private subnets in two AZs for application servers.
- Database in a Multi-AZ configuration.

A simple VPC layout often looks like this:

```text
Region: us-east-1
  AZ-a
    Public subnet -> ALB
    Private subnet -> App servers
  AZ-b
    Public subnet -> ALB
    Private subnet -> App servers
  Database layer -> Multi-AZ deployment
```

For the exam, remember:
- A single AZ is a single failure domain.
- Multi-AZ improves availability, not geographic disaster recovery.
- Cross-AZ traffic can add cost, so place components thoughtfully.

**Local Zones**

A **Local Zone** extends an AWS Region closer to a large population or industry center. It places select AWS services near users who need very low latency, while still connecting back to the parent Region for control plane operations and broader service access.

Use Local Zones when you need:
- single-digit millisecond latency to end users,
- workloads close to a metro area,
- media, gaming, VDI (Virtual Desktop Infrastructure), or interactive applications,
- local processing without moving the entire application to a separate Region.

Local Zones are useful when:
- the main Region is too far away for latency-sensitive workloads,
- you want to keep the application architecture tied to a parent Region,
- you need a subset of services near a city rather than a full Region.

Practical examples:
- Game streaming or interactive gaming in a metro area.
- Video rendering or media processing near content creators.
- Virtual desktops for users in a specific city.

Important limitations:
- Only a subset of AWS services is available.
- Not all Region features are present.
- You still manage resources through the parent Region.

**Wavelength Zones**

A **Wavelength Zone** brings AWS services to the edge of a telecommunications carrier network. This is designed for applications that need ultra-low latency to 5G devices and mobile users.

Use Wavelength when you need:
- very low latency for mobile and 5G applications,
- local processing for telecom or carrier-integrated workloads,
- edge compute close to end devices,
- reduced network hops between mobile devices and application logic.

Common use cases:
- Augmented reality and virtual reality for mobile users.
- Connected vehicles and real-time telemetry.
- Smart factories with carrier-connected devices.
- Gaming or media experiences for 5G users.

How it works:
- Your application runs in a Wavelength Zone.
- Traffic enters through the carrier network.
- The workload can still integrate with the parent AWS Region for management and broader services.

Key point:
- Wavelength is not a general replacement for a Region or AZ.
- It is specialized for carrier-network edge use cases.

**Edge Locations**

An **Edge Location** is part of AWS’s global content delivery and edge network. Edge Locations are primarily used by Amazon CloudFront and related edge services to cache content close to users and reduce latency.

Use Edge Locations when you need:
- content delivery for static assets,
- caching of dynamic or semi-dynamic content,
- TLS termination and request acceleration at the edge,
- lower latency for globally distributed users.

Typical services associated with Edge Locations:
- Amazon CloudFront
- AWS Global Accelerator uses the AWS global network and edge presence to improve routing
- Route 53 latency-based and health-aware routing complements edge delivery, though it is not itself an edge cache

What Edge Locations are good for:
- serving images, CSS, JavaScript, and video segments,
- reducing load on origin servers,
- improving user experience for global audiences.

What they are not:
- full application data centers,
- a substitute for Regions or AZs,
- the place where you normally run your primary database.

Example CloudFront origin setup:

```text
User -> Edge Location -> CloudFront -> Origin in Region
```

A simple CloudFront distribution example:

```json
{
  "Origins": [
    {
      "Id": "app-origin",
      "DomainName": "myapp-alb-123456.us-east-1.elb.amazonaws.com",
      "CustomOriginConfig": {
        "OriginProtocolPolicy": "https-only",
        "HTTPSPort": 443
      }
    }
  ],
  "DefaultCacheBehavior": {
    "TargetOriginId": "app-origin",
    "ViewerProtocolPolicy": "redirect-to-https",
    "AllowedMethods": ["GET", "HEAD", "OPTIONS"]
  }
}
```

**How to choose the right location type**

Use the simplest location that satisfies your requirements, then add complexity only when needed.

| Location type | Primary purpose | Best for | Not ideal for |
|---|---|---|---|
| Region | Geographic isolation and service availability | General workloads, compliance, DR | Ultra-low latency to a city or carrier network |
| Availability Zone | Fault isolation within a Region | High availability, multi-AZ apps | Geographic disaster recovery |
| Local Zone | Metro-area low latency | Gaming, VDI, media, interactive apps | Full-service regional deployments |
| Wavelength Zone | Carrier-network edge | 5G and mobile edge apps | General-purpose enterprise workloads |
| Edge Location | Content delivery and caching | Static content, acceleration, global users | Primary application hosting |

A practical decision flow:
1. If you need global content caching, start with **Edge Locations** and CloudFront.
2. If you need high availability, deploy across multiple **AZs**.
3. If you need geographic separation or compliance, use multiple **Regions**.
4. If you need metro-level low latency, consider a **Local Zone**.
5. If you need carrier-network edge latency, consider a **Wavelength Zone**.

**High availability and disaster recovery patterns**

High availability and disaster recovery are related but not the same.

- **High availability** usually means surviving an AZ failure inside one Region.
- **Disaster recovery** usually means surviving a Region-wide outage or major event.

Common patterns:
- **Multi-AZ active/passive**: one AZ serves traffic, another stands by.
- **Multi-AZ active/active**: traffic is spread across AZs.
- **Multi-Region pilot light**: minimal core services in a secondary Region.
- **Multi-Region warm standby**: scaled-down but ready secondary environment.
- **Multi-Region active/active**: traffic served from multiple Regions.

Example:
- Use Route 53 health checks to fail over from `us-east-1` to `us-west-2`.
- Use Amazon RDS Multi-AZ for database failover within a Region.
- Use cross-Region replication for backups or object storage.

> A Multi-AZ design protects against AZ failure, but it does not automatically protect you from a Region outage.

**Networking and latency considerations**

Latency depends on physical distance, network hops, and service placement. AWS gives you several ways to reduce it:
- place compute in the nearest Region,
- use Multi-AZ for local resilience,
- use Local Zones for metro proximity,
- use Wavelength for mobile edge proximity,
- use Edge Locations for cached content delivery.

Practical guidance:
- Keep chatty application tiers close together.
- Avoid unnecessary cross-Region synchronous calls.
- Minimize cross-AZ traffic when it is not needed.
- Cache aggressively at the edge for read-heavy content.

Example of a latency-sensitive architecture:
- Static assets on CloudFront.
- Application servers in two AZs.
- Database in Multi-AZ.
- Optional Local Zone for users in a dense metro area.

**CLI and operational examples**

List AZs in a Region:

```bash
aws ec2 describe-availability-zones --region us-east-1
```

Check your default Region configuration:

```bash
aws configure list
```

Set a Region for a single command:

```bash
AWS_REGION=eu-west-1 aws s3 ls
```

Launch an EC2 instance in a specific subnet tied to an AZ:

```bash
aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --subnet-id subnet-0abc1234def567890 \
  --security-group-ids sg-0123456789abcdef0 \
  --count 1
```

Create a CloudFront distribution with an origin in a Region, then let users connect to the nearest Edge Location through the distribution. This is the standard pattern for global web delivery.

**Exam tips**

- **Region** means geographic area with multiple AZs.
- **AZ** means isolated data center group inside a Region.
- **Local Zone** means low-latency extension near a metro area.
- **Wavelength Zone** means edge compute in a telecom carrier network.
- **Edge Location** means caching and delivery point for CloudFront and related services.

Watch for wording clues:
- “single-digit millisecond latency to a city” often points to **Local Zones**.
- “5G/mobile edge” often points to **Wavelength Zones**.
- “cache content close to users worldwide” points to **Edge Locations**.
- “survive an AZ failure” points to **Multi-AZ**.
- “survive a Region failure” points to **Multi-Region**.

**Common misconceptions**

- A Region is not a single data center; it contains multiple AZs.
- An AZ is not the same as a data center, though it may consist of one or more data centers.
- Edge Locations do not replace Regions for application hosting.
- Local Zones and Wavelength Zones are specialized extensions, not general-purpose Regions.
- Multi-AZ is for availability, not for geographic disaster recovery.
