## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Shared Responsibility Model overview**

The **Shared Responsibility Model** defines how security and compliance duties are split between AWS and you. AWS secures the cloud infrastructure that runs AWS services, while you secure what you place in the cloud, including data, identities, configurations, and workloads.

This model is a frequent exam topic because many AWS security questions hinge on whether a control is managed by AWS or by the customer. The key idea is simple: AWS is responsible for the security *of* the cloud, and you are responsible for security *in* the cloud.

- **AWS responsibility** covers the physical and foundational layers.
- **Customer responsibility** covers access, data, and configuration choices.
- The exact split changes by service type, especially between **IaaS** and **managed services**.

**Core security boundary**

AWS operates and protects the global infrastructure that supports AWS services. This includes facilities, hardware, networking, and the managed service platform itself. You do not patch AWS data centers or replace failed disks; AWS handles those tasks.

You remain responsible for how you use the service. That includes deciding who can access resources, whether data is encrypted, how security groups are configured, and whether operating systems and applications are patched when you manage them.

**What AWS typically manages**

AWS takes responsibility for the foundational layers that customers cannot directly access. This usually includes:

- **Physical security** of data centers
- **Hardware maintenance** such as servers, storage devices, and networking equipment
- **Infrastructure software** that supports AWS services
- **Availability and resilience** of the AWS-managed control plane
- **Patching** of managed service components that AWS operates

For example, if an Amazon S3 storage node fails, AWS handles the replacement and service continuity. If an AWS-managed database engine patch is required for a fully managed service, AWS may apply it according to the service model and maintenance window.

**What you typically manage**

Your responsibilities depend on the service and the architecture you build. Common customer responsibilities include:

- **Identity and access management (IAM)** users, roles, policies, and permission boundaries
- **Data classification and encryption** at rest and in transit
- **Network configuration** such as security groups, network ACLs, route tables, and public exposure
- **Operating system patching** for Amazon EC2 instances you administer
- **Application code** and runtime security
- **Logging and monitoring** configuration, including CloudTrail, CloudWatch, and VPC Flow Logs
- **Backup and recovery** strategy for your data and workloads
- **Compliance settings** you choose to enable or enforce

A practical example is an EC2-based web server. AWS provides the host hardware, hypervisor, and physical security. You install the operating system, patch it, configure the web server, restrict inbound traffic, and protect application secrets.

**Service model differences**

The shared responsibility boundary changes with the service model. The more AWS manages, the less you manage.

| Service model | AWS manages | You manage |
|---|---|---|
| **Infrastructure as a Service (IaaS)** | Physical infrastructure, virtualization layer, core networking | Guest OS, patches, applications, data, IAM, network rules |
| **Platform as a Service (PaaS)** | Infrastructure, OS, runtime, managed platform components | Application code, data, IAM, configuration |
| **Software as a Service (SaaS)** | Most or all of the application stack | User access, data governance, configuration, usage policies |

For exam purposes, remember that **Amazon EC2** is closer to IaaS, while services like **Amazon RDS** and **AWS Lambda** shift more responsibility to AWS. You still own your data, access controls, and configuration choices even when AWS manages the platform.

**EC2 example**

With Amazon EC2, you are responsible for the guest operating system and everything above it. AWS does not patch your Linux or Windows instance unless you use a managed automation tool or service that you configure.

Typical EC2 customer tasks include:

- Installing and updating packages
- Managing SSH or RDP access
- Configuring host firewalls
- Hardening the OS
- Deploying application code
- Encrypting attached volumes where required

Example commands for a Linux EC2 instance:

```bash
sudo yum update -y
sudo systemctl status nginx
sudo ufw allow 443/tcp
aws s3 cp s3://my-bucket/app.tar.gz .
```

**RDS example**

With Amazon RDS, AWS manages the database engine infrastructure, storage subsystem, backups in many configurations, and patching of the managed service components. You still manage:

- Database users and privileges
- Schema design
- Query performance tuning
- Data encryption choices
- Backup retention settings
- Network access to the DB instance

A common exam trap is assuming AWS manages everything in RDS. It does not. You still control who can connect, what data is stored, and how the database is used.

Example configuration pattern:

```bash
aws rds modify-db-instance \
  --db-instance-identifier prod-db \
  --backup-retention-period 7 \
  --apply-immediately
```

**Identity and access management**

**IAM** is one of the most important customer responsibilities. AWS provides the IAM service, but you decide how to use it. That means you must design least-privilege access, rotate credentials, and avoid overly broad permissions.

Key practices include:

- Use roles instead of long-term access keys where possible
- Apply least privilege to users and services
- Enable multi-factor authentication (**MFA**) for privileged accounts
- Review policies regularly
- Use temporary credentials for workloads

Example IAM policy snippet:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlyS3Access",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

**Data protection responsibilities**

You are responsible for the data you store in AWS. That includes deciding whether data should be encrypted, retained, backed up, or deleted. AWS provides encryption features and key management services, but you must enable and configure them.

Important data responsibilities:

- Classify sensitive data before storing it
- Encrypt data at rest using services such as AWS Key Management Service (**KMS**)
- Encrypt data in transit with TLS
- Define retention and deletion policies
- Test restore procedures regularly
- Protect secrets with services such as AWS Secrets Manager or Systems Manager Parameter Store

Example encrypted S3 upload:

```bash
aws s3 cp report.csv s3://secure-bucket/ \
  --sse aws:kms \
  --sse-kms-key-id alias/my-key
```

**Network and perimeter controls**

AWS provides the networking backbone, but you configure how your resources are exposed. This is a major customer responsibility in VPC-based architectures.

You typically manage:

- **Security groups** for instance-level stateful filtering
- **Network ACLs** for subnet-level stateless filtering
- **Route tables** for traffic paths
- **Internet Gateway** and **NAT Gateway** placement
- **Load balancer** listener and target group settings
- **Public IP exposure** and DNS records

A secure design often uses private subnets for application and database tiers, with only the load balancer in a public subnet.

```text
Internet
   |
[Route 53]
   |
[Application Load Balancer]
   |
[Private Subnet: EC2 App Tier]
   |
[Private Subnet: RDS]
```

**Logging, monitoring, and auditability**

AWS provides the tools, but you must enable and interpret them. Security visibility is part of your responsibility.

Common controls include:

- **AWS CloudTrail** for API activity auditing
- **Amazon CloudWatch** for metrics, logs, and alarms
- **VPC Flow Logs** for network traffic visibility
- **AWS Config** for configuration drift and compliance checks
- **GuardDuty** for threat detection

A practical baseline is to enable CloudTrail in all regions, centralize logs in a dedicated account, and protect log buckets from deletion.

Example CloudTrail creation pattern:

```bash
aws cloudtrail create-trail \
  --name org-trail \
  --s3-bucket-name my-cloudtrail-logs \
  --is-multi-region-trail
```

**Compliance and governance**

AWS provides compliance programs and attestations for its infrastructure, but your compliance posture depends on how you configure services and handle data. AWS compliance does not automatically make your application compliant.

You are responsible for:

- Choosing compliant architectures
- Enforcing encryption and access controls
- Maintaining evidence and audit trails
- Restricting data residency where required
- Applying organizational guardrails with AWS Organizations, Service Control Policies (**SCPs**), and AWS Config

This distinction matters in regulated environments. For example, AWS may support a compliant platform, but if you expose an S3 bucket publicly or store unencrypted sensitive data, the customer side of the model has failed.

**Security responsibilities by layer**

A useful way to remember the model is by stack layer:

- **Facilities layer**: AWS
- **Hardware layer**: AWS
- **Virtualization layer**: AWS
- **Operating system layer**: depends on service
  - AWS for managed services
  - You for EC2
- **Application layer**: you
- **Data layer**: you
- **Identity layer**: you
- **Configuration layer**: you

This layer-based view helps you answer exam questions quickly. If the question mentions a failed disk, data center access, or host replacement, it is usually AWS. If it mentions a security group rule, IAM policy, or database password, it is usually the customer.

**Practical exam cues**

Look for wording that signals responsibility:

- **AWS-managed** terms:
  - data center
  - hardware failure
  - hypervisor
  - managed patching
  - service availability
- **Customer-managed** terms:
  - IAM policy
  - encryption key usage
  - security group
  - OS patching
  - application code
  - data retention

If a question asks who is responsible for securing an EC2 instance’s operating system, the answer is you. If it asks who secures the physical servers running Amazon S3, the answer is AWS.

**Common misconceptions**

- **“AWS handles all security.”** False. AWS secures the platform, but you still configure access and protect data.
- **“Using a managed service removes all responsibility.”** False. Managed services reduce operational burden, but you still manage identity, data, and configuration.
- **“Compliance is automatic in AWS.”** False. AWS provides compliant services and controls, but your implementation determines compliance outcomes.
- **“Encryption is always enabled by default.”** Not universally. You must verify service-specific defaults and enable encryption where needed.

**Quick reference**

- **AWS responsibility**: physical security, infrastructure, managed service platform
- **Customer responsibility**: IAM, data, network rules, application security, OS patching on EC2
- **Shared controls**: logging, monitoring, compliance, encryption configuration, incident response processes
- **Best exam habit**: identify the service type first, then map the responsibility boundary

**Hands-on verification examples**

You can inspect and validate your own responsibilities with AWS CLI and console checks.

```bash
aws ec2 describe-security-groups --group-ids sg-0123456789abcdef0
aws iam get-user
aws s3api get-bucket-encryption --bucket secure-bucket
aws cloudtrail describe-trails
```

These commands help you confirm whether access controls, encryption, and audit logging are configured as intended.

**Summary**

The Shared Responsibility Model is the foundation of AWS security thinking. AWS protects the cloud infrastructure, while you protect your workloads, identities, and data inside that infrastructure. The exact boundary depends on the service, but the exam consistently tests whether you can identify who owns each control.

When in doubt, ask three questions:
- What service is being used?
- Is the control part of the underlying platform or the customer workload?
- Does the issue involve infrastructure, or does it involve access, data, or configuration?
