## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**Account safety baseline overview**

A new AWS account should start with controls that reduce the chance of accidental exposure, unauthorized access, and uncontrolled spend. The baseline is not one service; it is a set of guardrails across identity, logging, detection, network exposure, and billing.

The practical goal is to make the account safer before workloads grow. You want:
- **Strong identity controls** for human and machine access
- **Centralized logging** for auditability and incident response
- **Guardrails** that prevent risky changes
- **Exposure checks** for public resources and overly permissive policies
- **Cost controls** so mistakes are visible quickly

**Day-one account setup**

Start with the root user and then move day-to-day access away from it. The root user has unrestricted power, so you should protect it aggressively and avoid using it for routine administration.

Recommended first actions:
- Enable **MFA** (multi-factor authentication) on the root user immediately
- Store root credentials securely and use them only for tasks that require root
- Create an administrative role or group for normal operations
- Prefer **AWS IAM Identity Center** for workforce access instead of long-lived IAM users
- Set a strong account password policy if you still use IAM users

A practical password policy baseline for IAM users often includes:
- Minimum length of 14 characters or more
- Password expiration if your organization requires it
- Prevention of password reuse
- Requirement for uppercase, lowercase, numbers, and symbols

If you still need IAM users for legacy reasons, keep them tightly scoped:
- Use **least privilege**
- Avoid attaching broad managed policies like `AdministratorAccess` unless absolutely necessary
- Rotate access keys regularly
- Remove unused keys and users

**Identity and access model**

For human access, AWS IAM Identity Center is the preferred starting point in modern multi-account environments. It centralizes sign-in, reduces password sprawl, and makes it easier to assign permission sets across accounts.

A common baseline pattern is:
- Use **IAM Identity Center** for workforce authentication
- Map users to groups such as `Admins`, `Developers`, and `ReadOnly`
- Assign permission sets to AWS accounts or organizational units
- Use IAM roles for applications and automation instead of IAM users

For machine access:
- Prefer **IAM roles** attached to compute services such as EC2, Lambda, ECS, or EKS
- Avoid embedding access keys in code or AMIs
- Use short-lived credentials through role assumption

Example IAM policy fragment for a narrowly scoped S3 read-only role:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlySpecificBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

**Organizations and guardrails**

If you manage more than one account, place them under **AWS Organizations**. This gives you centralized billing, account structure, and policy-based guardrails.

Key baseline steps:
- Create a management account for organization administration
- Add member accounts for workloads, security, and logging
- Use **organizational units** (OUs) to group accounts by function or environment
- Apply **service control policies** (SCPs) to restrict what accounts can do, even if IAM permissions allow it

SCPs are guardrails, not permissions grants. They set the maximum available permissions in an account or OU.

Common SCP baseline ideas:
- Deny disabling CloudTrail, AWS Config, or GuardDuty
- Deny leaving the organization
- Deny changes to centralized logging resources
- Restrict regions if your organization uses only a subset of AWS Regions

Example SCP concept that denies disabling CloudTrail:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyCloudTrailStop",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail",
        "cloudtrail:UpdateTrail"
      ],
      "Resource": "*"
    }
  ]
}
```

> SCPs can block actions only in accounts that are part of AWS Organizations. They do not replace IAM; they constrain it.

**Centralized logging baseline**

Logging is one of the most important safety controls because it gives you evidence after a change, incident, or exposure event. A strong baseline usually includes an **organization trail** in AWS CloudTrail, plus AWS Config for configuration history.

For CloudTrail:
- Create an **organization trail** so all accounts are covered
- Deliver logs to a dedicated **log archive account**
- Store logs in an S3 bucket with tight bucket policies
- Enable **log file validation** to detect tampering
- Encrypt logs with **AWS KMS** if your compliance requirements call for customer-managed encryption keys
- Send management events at minimum; add data events where needed for sensitive S3 or Lambda activity

CloudTrail records API activity, including console sign-ins and service actions. For root console sign-in monitoring, look for `ConsoleLogin` events where `userIdentity.type` is `Root` and `responseElements.ConsoleLogin` is `Success`.

Example CloudTrail event pattern to inspect:
```json
{
  "eventName": "ConsoleLogin",
  "userIdentity": {
    "type": "Root"
  },
  "responseElements": {
    "ConsoleLogin": "Success"
  }
}
```

A practical organization trail setup often looks like this:
- **Management account** creates the trail
- Trail applies to **all accounts in the organization**
- Logs are written to a centralized S3 bucket in a **log archive account**
- Bucket policy allows only CloudTrail delivery and security administrators
- KMS key policy allows CloudTrail to encrypt logs

Example AWS CLI to create a trail:
```bash
aws cloudtrail create-trail \
  --name org-trail \
  --s3-bucket-name my-org-cloudtrail-logs \
  --is-organization-trail
```

Enable log file validation:
```bash
aws cloudtrail update-trail \
  --name org-trail \
  --enable-log-file-validation
```

**AWS Config baseline**

AWS Config records configuration changes and maintains a history of resource configurations over time. It is useful for compliance checks, drift detection, and forensic review.

Important components:
- **Configuration recorder**: captures resource configuration changes
- **Delivery channel**: sends configuration snapshots and history to an S3 bucket and optionally SNS
- **Aggregator**: centralizes Config data from multiple accounts and Regions
- **Conformance packs**: bundle managed and custom Config rules into a deployable compliance package

AWS Config does not provide a simple before-and-after diff view in the way a change-management tool might. Instead, it stores configuration snapshots and history that you can compare over time.

Typical baseline use cases:
- Detect public S3 buckets
- Detect security groups with open ingress
- Detect unencrypted resources
- Track whether logging or encryption settings changed

Example AWS CLI to enable a configuration recorder:
```bash
aws configservice put-configuration-recorder \
  --configuration-recorder '{
    "name": "default",
    "roleARN": "arn:aws:iam::123456789012:role/AWSConfigRole",
    "recordingGroup": {
      "allSupported": true,
      "includeGlobalResourceTypes": true
    }
  }'
```

Example delivery channel:
```bash
aws configservice put-delivery-channel \
  --delivery-channel '{
    "name": "default",
    "s3BucketName": "my-config-bucket",
    "snsTopicARN": "arn:aws:sns:us-east-1:123456789012:config-alerts"
  }'
```

A conformance pack is useful when you want a repeatable compliance baseline across many accounts. For example, you can package rules for:
- S3 public access
- EBS encryption
- IAM password policy
- Security group ingress restrictions

**Detection and exposure review**

A baseline should include services that help you find risky configurations and suspicious activity.

Useful services:
- **AWS Security Hub** for centralized security findings and standards checks
- **Amazon GuardDuty** for threat detection using logs and anomaly signals
- **AWS IAM Access Analyzer** for identifying unintended external access to resources

How they fit together:
- GuardDuty detects suspicious behavior such as unusual API calls or compromised credentials
- Security Hub aggregates findings and helps you prioritize them
- IAM Access Analyzer reviews resource policies for external access paths

IAM Access Analyzer is especially useful for:
- S3 bucket policies
- KMS key policies
- IAM roles trusted by external principals
- Other resource-based policies that may expose access outside your account

Example CLI to create an analyzer:
```bash
aws accessanalyzer create-analyzer \
  --analyzer-name account-analyzer \
  --type ACCOUNT
```

**Public exposure guardrails**

Public exposure often happens through S3, security groups, or permissive resource policies. The baseline should reduce the chance of accidental public access and make it easy to detect when it happens.

For S3:
- Enable **S3 Block Public Access** at the account level
- Also enable it at the bucket level for defense in depth
- Use bucket policies carefully and test them before rollout
- Prefer presigned URLs or CloudFront for controlled external access

The four S3 Block Public Access settings behave differently:
- `BlockPublicAcls` blocks new public ACLs and public ACL updates
- `IgnorePublicAcls` ignores public ACLs that already exist
- `BlockPublicPolicy` blocks public bucket policies
- `RestrictPublicBuckets` limits access to buckets that already have public policies

For security groups:
- Avoid `0.0.0.0/0` unless the service truly must be public
- Restrict SSH (`22`) and RDP (`3389`) to trusted source IPs or use Session Manager instead
- Review inbound rules for broad CIDR ranges
- Remove unused rules and stale security groups

Example security group rule with the AWS CLI:
```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 443 \
  --cidr 203.0.113.0/24
```

For S3 public access, a safer pattern is:
- Keep the bucket private
- Use CloudFront with an origin access control or origin access identity
- Use presigned URLs for temporary access
- Use IAM policies for authenticated access

**Cost controls and billing alarms**

A safety baseline should also protect against accidental spend. Cost controls are not just finance concerns; they are operational safeguards.

Recommended controls:
- Create **AWS Budgets** with alerts for actual and forecasted spend
- Send budget notifications to **Amazon SNS**
- Use billing alarms or notifications for early warning
- Tag resources for cost allocation and cleanup
- Set service quotas where appropriate to limit accidental scale-out

Example budget notification flow:
1. Create an AWS Budget for monthly spend.
2. Attach an SNS topic for alerts.
3. Subscribe email or automation endpoints to the topic.
4. Trigger notifications at thresholds such as 50%, 80%, and 100%.

Example AWS CLI for a simple budget definition:
```bash
aws budgets create-budget \
  --account-id 123456789012 \
  --budget '{
    "BudgetName": "monthly-baseline-budget",
    "BudgetLimit": {
      "Amount": "100",
      "Unit": "USD"
    },
    "TimeUnit": "MONTHLY",
    "BudgetType": "COST"
  }'
```

Example SNS topic creation:
```bash
aws sns create-topic --name budget-alerts
```

**Practical baseline checklist**

Use this checklist when you create a new account or review an existing one:

1. Secure the root user with MFA and remove routine use.
2. Set up IAM Identity Center for workforce access.
3. Create least-privilege roles and avoid long-lived access keys.
4. Organize accounts in AWS Organizations.
5. Apply SCPs to prevent disabling core security services.
6. Enable an organization CloudTrail trail with log file validation.
7. Centralize logs in a dedicated archive account.
8. Enable AWS Config with a recorder, delivery channel, and aggregator.
9. Add conformance packs for baseline compliance.
10. Turn on GuardDuty, Security Hub, and IAM Access Analyzer.
11. Enable S3 Block Public Access at account and bucket levels.
12. Review security groups for public ingress.
13. Create AWS Budgets and SNS notifications.
14. Define password policy and access key rotation practices.

**Example multi-account topology**

A common safety baseline architecture uses separate accounts for separation of duties:

```text
Management Account
├── Security Account
│   ├── Security Hub
│   ├── GuardDuty
│   └── IAM Access Analyzer
├── Log Archive Account
│   ├── CloudTrail S3 bucket
│   └── AWS Config history bucket
└── Workload Accounts
    ├── Dev
    ├── Test
    └── Prod
```

This layout helps you:
- Keep logs isolated from workload administrators
- Centralize security findings
- Apply different guardrails to different environments
- Reduce blast radius if one account is compromised

**Operational examples**

A few practical checks help validate the baseline after setup:

- Confirm CloudTrail is logging:
```bash
aws cloudtrail get-trail-status --name org-trail
```

- Confirm Config is recording:
```bash
aws configservice describe-configuration-recorders
```

- Check for public S3 exposure:
```bash
aws s3api get-public-access-block --bucket my-bucket
```

- Review GuardDuty detectors:
```bash
aws guardduty list-detectors
```

- Inspect IAM Access Analyzer findings:
```bash
aws accessanalyzer list-analyzers
```

**Why this baseline matters**

The baseline reduces the most common early-account risks:
- Accidental public exposure
- Undetected administrative changes
- Uncontrolled root or IAM user usage
- Missing audit trails
- Surprise cost spikes

It also creates a foundation for later architecture decisions. Once the account is instrumented and guarded, you can deploy workloads with more confidence and less manual review.

**Exam focus**

For SAA-C03, focus on how the services complement each other:
- **CloudTrail** answers who did what and when
- **AWS Config** answers how resource configuration changed over time
- **GuardDuty** detects suspicious behavior
- **Security Hub** centralizes findings
- **IAM Access Analyzer** identifies unintended access
- **SCPs** enforce organization-wide guardrails
- **Budgets** help control spend

A strong answer usually combines multiple controls rather than relying on one service alone.
