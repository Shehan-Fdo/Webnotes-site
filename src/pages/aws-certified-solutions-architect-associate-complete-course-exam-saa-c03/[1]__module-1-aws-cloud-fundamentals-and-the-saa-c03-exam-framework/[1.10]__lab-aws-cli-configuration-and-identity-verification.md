## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**AWS CLI setup overview**

The AWS Command Line Interface (CLI) lets you manage AWS services from a terminal using signed API requests. In this lab, you configure local credentials, choose a default Region and output format, and verify the identity behind those credentials before you run any operational commands.

A clean CLI setup matters because many AWS issues are really identity or Region issues:
- You may be using the wrong <abbr title="Identity and Access Management">IAM</abbr> principal.
- You may be targeting the wrong Region.
- You may have stale or missing credentials in your local environment.
- You may be assuming a role without realizing which session is active.

**What the AWS CLI stores**

The CLI typically uses two local files under `~/.aws/`:
- `credentials` for access keys and session tokens.
- `config` for default Region, output format, and profile settings.

A common layout looks like this:

```ini
# ~/.aws/credentials
[default]
aws_access_key_id = AKIAEXAMPLE
aws_secret_access_key = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY

[dev]
aws_access_key_id = AKIAEXAMPLEDEV
aws_secret_access_key = devSecretExampleKey
```

```ini
# ~/.aws/config
[default]
region = us-east-1
output = json

[profile dev]
region = us-west-2
output = table
```

Use profiles when you need to separate environments, accounts, or roles. This reduces accidental changes in the wrong account and makes automation easier to reason about.

**Installing and checking the CLI**

Before you configure anything, confirm that the CLI is installed and available in your shell path.

```bash
aws --version
which aws
```

On Windows PowerShell, you can use:

```powershell
aws --version
Get-Command aws
```

If the command is missing, install AWS CLI v2 using the platform-specific installer. AWS CLI v2 is the standard choice for modern labs because it includes current features and improved authentication support.

**Initial configuration with `aws configure`**

The fastest way to create a default profile is with `aws configure`. It prompts you for the access key, secret key, default Region, and output format.

```bash
aws configure
```

You will be prompted for:
- AWS Access Key ID
- AWS Secret Access Key
- Default region name
- Default output format

Example values:
- Region: `us-east-1`
- Output: `json`

This command writes values into `~/.aws/credentials` and `~/.aws/config`. It is useful for quick setup, but in real environments you often prefer more controlled methods such as environment variables, named profiles, or IAM Identity Center.

> Do not hardcode long-term access keys into scripts or shared repositories. Use profiles, temporary credentials, or role-based access instead.

**Named profiles for multiple identities**

Profiles let you switch between accounts, roles, or permission sets without overwriting your default identity. You can create or edit them manually, or use `aws configure --profile <name>`.

```bash
aws configure --profile dev
aws configure --profile prod
```

Then use a profile explicitly:

```bash
aws s3 ls --profile dev
aws ec2 describe-regions --profile prod
```

You can also set a profile for the current shell session:

```bash
export AWS_PROFILE=dev
aws sts get-caller-identity
```

On PowerShell:

```powershell
$env:AWS_PROFILE = "dev"
aws sts get-caller-identity
```

Use `AWS_PROFILE` when you want every CLI command in the session to target the same identity without repeating `--profile`.

**Identity verification with STS**

The most important verification step in this lab is checking who the CLI thinks you are. Use the Security Token Service (STS) command `get-caller-identity`.

```bash
aws sts get-caller-identity
```

Typical output:

```json
{
  "UserId": "AIDAEXAMPLE123456789",
  "Account": "123456789012",
  "Arn": "arn:aws:iam::123456789012:user/lab-user"
}
```

This response tells you:
- `UserId`: the unique identifier for the principal.
- `Account`: the AWS account ID being used.
- `Arn`: the Amazon Resource Name for the current identity.

This command is especially valuable because it works even when the principal has very limited permissions. It is the fastest way to confirm whether you are using:
- an IAM user,
- an assumed role,
- or a federated session.

**Why identity verification comes first**

Before you troubleshoot an AWS service, verify identity. Many errors are caused by the wrong credentials rather than the service itself.

Use `get-caller-identity` to confirm:
- the active profile is correct,
- the account ID matches the target environment,
- the role session is the one you intended,
- the CLI is not falling back to unexpected environment variables.

A practical workflow is:
1. Set or select the profile.
2. Run `aws sts get-caller-identity`.
3. Confirm the account and ARN.
4. Run the service command only after identity is correct.

**Common credential sources and precedence**

The CLI can obtain credentials from multiple places. In practice, the order of precedence matters because one source can override another.

Common sources include:
- Command-line `--profile`
- `AWS_PROFILE`
- Environment variables such as `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, and `AWS_SESSION_TOKEN`
- Shared credentials file
- IAM role credentials from an attached compute environment, such as an EC2 instance profile

Environment variables often override shared files, which can surprise you during labs. If a command behaves unexpectedly, inspect the environment first.

```bash
env | grep '^AWS_'
```

If you see unexpected values, unset them before continuing:

```bash
unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN AWS_PROFILE
```

**Region configuration and why it matters**

The AWS CLI needs a default Region for many commands. Some APIs are global, but many are Region-specific. If you omit the Region, the CLI may fail or target the wrong location.

Check the active Region:

```bash
aws configure get region
```

Override the Region for a single command:

```bash
aws ec2 describe-instances --region us-west-2
```

Set a profile-specific Region in `~/.aws/config` when you work in multiple Regions. This is safer than repeatedly typing `--region` and helps prevent accidental changes in the wrong place.

**Output formats**

The CLI supports multiple output formats:
- `json`
- `table`
- `text`

`json` is best for automation and detailed inspection. `table` is useful for quick human review. `text` is compact and script-friendly, but less readable for complex structures.

Examples:

```bash
aws ec2 describe-regions --output table
aws sts get-caller-identity --output json
```

Use `json` as the default unless you have a specific reason to choose another format.

**Practical verification commands**

After configuration, run a small set of checks to confirm the environment is ready.

```bash
aws configure list
aws sts get-caller-identity
aws ec2 describe-regions --output table
```

`aws configure list` shows where the CLI is sourcing values from. This is useful for debugging because it reveals whether a value comes from:
- environment variables,
- shared credentials files,
- config files,
- or an explicit command-line option.

Example output fields to inspect:
- access key source
- secret key source
- region source
- profile name

**Troubleshooting identity and configuration issues**

If the CLI fails, isolate the problem systematically.

1. Confirm the binary is installed.
   ```bash
   aws --version
   ```

2. Check for active environment variables.
   ```bash
   env | grep '^AWS_'
   ```

3. Verify the current profile and Region.
   ```bash
   aws configure list
   ```

4. Confirm the active identity.
   ```bash
   aws sts get-caller-identity
   ```

5. Test a simple read-only API call.
   ```bash
   aws s3 ls
   ```

Common issues include:
- `Unable to locate credentials`: no usable credentials are available.
- `InvalidClientTokenId`: the access key or session token is invalid.
- `SignatureDoesNotMatch`: the secret key, Region, or system clock may be wrong.
- `AccessDenied`: the identity is valid but lacks permission.
- `ExpiredToken`: temporary credentials or role session expired.

**Temporary credentials and session tokens**

If you use assumed roles, federated access, or temporary lab credentials, you may need a session token in addition to the access key and secret key.

Example environment variables:

```bash
export AWS_ACCESS_KEY_ID=ASIAEXAMPLE
export AWS_SECRET_ACCESS_KEY=exampleSecretKey
export AWS_SESSION_TOKEN=exampleSessionToken
```

Temporary credentials expire, so identity verification is not a one-time task. Re-run `aws sts get-caller-identity` whenever you refresh credentials or switch roles.

**Using the CLI in real workflows**

Once identity is verified, you can safely use the CLI for read-only discovery and later for changes. A few common examples:

```bash
aws s3 ls
aws ec2 describe-vpcs
aws iam list-users
aws cloudformation list-stacks
```

These commands help you confirm that:
- the account is correct,
- the Region is correct,
- and the permissions match your expectations.

For scripting, combine identity checks with guardrails:

```bash
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
echo "Current account: $ACCOUNT_ID"
```

You can also compare against an expected account ID before running destructive actions:

```bash
EXPECTED_ACCOUNT="123456789012"
CURRENT_ACCOUNT=$(aws sts get-caller-identity --query Account --output text)

if [ "$CURRENT_ACCOUNT" != "$EXPECTED_ACCOUNT" ]; then
  echo "Wrong account: $CURRENT_ACCOUNT"
  exit 1
fi
```

**Best practices for lab and exam readiness**

Use these habits to avoid mistakes:
- Verify identity before every new lab task.
- Prefer named profiles for separate environments.
- Keep Region settings explicit and consistent.
- Use `json` output when you need structured data.
- Treat `aws sts get-caller-identity` as your first diagnostic command.
- Remove stale environment variables when results look inconsistent.

**Quick reference**

| Task | Command |
|---|---|
| Check CLI version | `aws --version` |
| Configure default profile | `aws configure` |
| Configure named profile | `aws configure --profile dev` |
| Show active config | `aws configure list` |
| Verify identity | `aws sts get-caller-identity` |
| List Regions | `aws ec2 describe-regions` |
| Check environment variables | `env | grep '^AWS_'` |

**Key takeaway**

AWS CLI configuration is not just about entering keys. It is about establishing a predictable identity, Region, and output format so every command runs against the intended account and environment. If you can reliably answer “who am I, where am I, and which profile is active?” you can avoid most CLI-related mistakes.
