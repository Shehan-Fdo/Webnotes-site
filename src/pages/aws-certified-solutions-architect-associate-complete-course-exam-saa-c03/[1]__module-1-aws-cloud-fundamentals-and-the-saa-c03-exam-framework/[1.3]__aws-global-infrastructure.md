## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**AWS global infrastructure overview**

AWS global infrastructure is the physical and logical foundation that lets you place workloads close to users, isolate failures, and design for availability, latency, and compliance. For SAA-C03, you need to know how AWS organizes infrastructure into **Regions**, **Availability Zones (AZs)**, **edge locations**, and global services, and how those choices affect architecture decisions.

A common exam pattern is to ask whether a solution should stay within one Region, span multiple AZs, or span multiple Regions. The correct answer usually depends on whether you need high availability, disaster recovery, data residency, or low-latency global delivery.

- **Region**: a geographic area containing multiple AZs.
- **AZ**: one or more discrete data centers with independent power, networking, and cooling.
- **Edge location**: a site used for content delivery and edge services, not for running your own EC2 instances.
- **Global service**: a service whose control plane is not tied to a single Region in the same way as regional services.

**Regions, Availability Zones, and account-specific AZ mapping**

An AWS Region is a separate geographic area, such as `us-east-1` or `eu-west-1`. Each Region contains multiple AZs, and AWS designs those AZs to be physically separated enough to reduce correlated failures while still being close enough for low-latency replication.

AZ names like `us-east-1a` or `us-east-1b` are **account-specific labels**. The lettered name you see in one AWS account may map to a different physical AZ than the same letter in another account. This matters on the exam because a question may show two accounts using different AZ letters and still refer to the same physical AZ.

- Use the **AZ ID** concept when you need to reason about the actual physical zone across accounts.
- Do not assume `us-east-1a` in one account is the same physical location as `us-east-1a` in another account.
- This explains why cross-account diagrams can appear inconsistent even when they are technically correct.

> If an exam question compares AZ letters across accounts, treat the letters as labels, not universal physical identifiers.

**Region vs AZ vs multi-Region design**

Use **Multi-AZ** when you want high availability within a single Region. This is the default choice for many production workloads because it protects against an AZ failure without adding the complexity, latency, and data consistency challenges of cross-Region design.

Use **Multi-Region** when you need disaster recovery across a broader failure domain, regulatory separation, or active global presence. Multi-Region is appropriate when a Region-wide outage, geopolitical risk, or data sovereignty requirement would make single-Region resilience insufficient.

- **Multi-AZ is sufficient** when:
  - You need high availability for a regional application.
  - You can tolerate a Region-level dependency.
  - You want simpler failover and lower operational overhead.
- **Multi-Region is required** when:
  - You need disaster recovery beyond a single Region.
  - You need active-active global traffic distribution.
  - You must survive a full Region outage or isolate workloads by geography.

A practical exam heuristic is:

1. Start with Multi-AZ for availability.
2. Add Multi-Region only when the scenario explicitly requires disaster recovery, global latency optimization, or regulatory separation.
3. Prefer managed replication and DNS-based failover over custom replication logic unless the question demands otherwise.

**Global, regional, and zonal service scope**

AWS services are often described by scope. Knowing whether a service is global, regional, or zonal helps you predict failure impact and where to configure it.

| Scope | Meaning | Examples |
|---|---|---|
| Global | One control plane spans AWS globally | IAM, Route 53, CloudFront, AWS Organizations |
| Regional | Resources live in a specific Region | S3 buckets, EC2, RDS, DynamoDB tables, Lambda |
| Zonal | Resources are tied to a specific AZ | EC2 placement in an AZ, EBS volumes, subnet placement |

Some services are commonly misunderstood:

- **S3** has a global namespace for bucket names, but buckets are created in a specific Region.
- **IAM** is global in exam terms; identities and policies are not Region-scoped.
- **Route 53** is global and provides DNS and traffic routing.
- **CloudFront** is global and uses the edge network to cache and deliver content.
- **EC2** is regional, but instances launch into a specific AZ.
- **EBS** is zonal because volumes are attached to instances in one AZ.
- **RDS** is regional, but a single-AZ deployment is zonal in effect; Multi-AZ changes the availability model.
- **DynamoDB** is regional by default, but **Global Tables** extend it across Regions.

**S3 global namespace and regional buckets**

Amazon S3 is a frequent exam topic because it combines global naming with regional storage. Bucket names must be globally unique across all AWS accounts, but the bucket itself exists in one Region.

This distinction matters for latency, compliance, and replication:

- Choose the bucket Region close to your users or compute when latency matters.
- Use **S3 Cross-Region Replication (CRR)** when you need copies in another Region for disaster recovery, compliance, or global distribution.
- Use **Same-Region Replication (SRR)** when you want redundancy or data processing within the same Region.

Example bucket creation:

```bash
aws s3api create-bucket \
  --bucket my-company-logs-123456 \
  --region us-east-1
```

Example CRR configuration conceptually requires:

- A source bucket with versioning enabled.
- A destination bucket in another Region.
- An IAM role that allows replication.
- A replication rule that targets the destination bucket.

A simplified replication rule looks like this:

```json
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "Status": "Enabled",
      "Priority": 1,
      "DeleteMarkerReplication": { "Status": "Enabled" },
      "Filter": { "Prefix": "" },
      "Destination": {
        "Bucket": "arn:aws:s3:::my-company-logs-dr"
      }
    }
  ]
}
```

**CloudFront, edge locations, and edge network flow**

Amazon CloudFront is AWS’s content delivery network (CDN). It uses the **edge network** to bring content closer to users and reduce latency. In exam terms, CloudFront is not just “a cache”; it is a globally distributed delivery layer that can serve cached content from edge locations and forward cache misses to an origin.

Key terminology:

- **Edge location**: a point of presence where CloudFront can cache and serve content.
- **Regional edge cache**: a larger cache layer between edge locations and the origin that improves cache hit rates for less frequently requested content.
- **Origin**: the backend source, such as an S3 bucket, ALB, or custom HTTP server.

Typical request flow:

1. A user requests content through CloudFront.
2. DNS resolves the CloudFront distribution to a nearby edge location.
3. The edge location serves the object if it is cached.
4. If the object is not cached, CloudFront checks the regional edge cache.
5. If still missing, CloudFront fetches the object from the origin.
6. CloudFront caches the response according to cache policy and TTL settings.

This matters on the exam because CloudFront reduces latency and origin load, while also supporting TLS termination, signed URLs/cookies, and integration with AWS WAF.

Example distribution origin concept:

```json
{
  "Origins": [
    {
      "Id": "s3-origin",
      "DomainName": "my-site-bucket.s3.us-east-1.amazonaws.com",
      "S3OriginConfig": {}
    }
  ],
  "DefaultCacheBehavior": {
    "TargetOriginId": "s3-origin",
    "ViewerProtocolPolicy": "redirect-to-https"
  }
}
```

Use CloudFront when you need:

- Global content delivery.
- Lower latency for static and dynamic content.
- Edge caching to reduce origin traffic.
- Protection and acceleration in front of an S3 bucket, ALB, or custom origin.

**Route 53 routing policies and exam patterns**

Amazon Route 53 is a global DNS service. It is often the right answer when the question asks for traffic steering, health checks, or failover between endpoints.

Common routing policies include:

- **Simple routing**: one record set, basic DNS response.
- **Weighted routing**: split traffic by percentage.
- **Latency-based routing**: send users to the Region with the lowest latency.
- **Geolocation routing**: route based on user location.
- **Failover routing**: active-passive behavior using health checks.
- **Multi-value answer**: return multiple healthy records.
- **Geoproximity routing**: bias traffic based on geography, often with traffic flow and routing controls.

Weighted routing example:

```bash
aws route53 change-resource-record-sets --hosted-zone-id Z1234567890ABC --change-batch '{
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "blue",
        "Weight": 80,
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack-blue-alb-123.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    },
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "SetIdentifier": "green",
        "Weight": 20,
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "dualstack-green-alb-456.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }
  ]
}'
```

Latency-based routing example:

```json
{
  "Name": "api.example.com",
  "Type": "A",
  "SetIdentifier": "us-east-1",
  "Region": "us-east-1",
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "dualstack-api-use1-alb-123.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

Geolocation routing example:

```json
{
  "Name": "www.example.com",
  "Type": "A",
  "SetIdentifier": "europe",
  "GeoLocation": {
    "ContinentCode": "EU"
  },
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "dualstack-eu-alb-123.eu-west-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

Failover routing example:

```json
{
  "Name": "app.example.com",
  "Type": "A",
  "SetIdentifier": "primary",
  "Failover": "PRIMARY",
  "HealthCheckId": "abcd1234",
  "AliasTarget": {
    "HostedZoneId": "Z35SXDOTRQ7X7K",
    "DNSName": "dualstack-primary-alb-123.us-east-1.elb.amazonaws.com",
    "EvaluateTargetHealth": true
  }
}
```

Exam guidance:

- Use **latency-based routing** when the goal is to send users to the closest healthy Region.
- Use **geolocation routing** when legal, language, or country-specific behavior matters.
- Use **failover routing** when you want active-passive disaster recovery.
- Use **weighted routing** for gradual cutovers, A/B testing, or blue/green deployments.

**Global services and global governance**

Some AWS services are global by design and often appear in architecture questions because they coordinate access, identity, or multi-account governance.

- **IAM** controls authentication and authorization across AWS.
- **Route 53** provides global DNS.
- **CloudFront** delivers content globally.
- **AWS Organizations** helps you manage multiple accounts centrally.
- **AWS Control Tower** helps establish a governed multi-account landing zone.

Organizations and Control Tower are not always the direct answer to a networking question, but they matter when the scenario involves multiple accounts, centralized logging, guardrails, or separating production and non-production environments.

A common real-world pattern is:

- One account for shared networking.
- One account for security and logging.
- Separate accounts for application environments.
- Centralized governance through Organizations and SCPs (Service Control Policies).

**DynamoDB Global Tables and global data design**

Amazon DynamoDB is regional by default, but **Global Tables** replicate data across multiple Regions. This is useful when you need low-latency reads and writes in multiple geographies or when you need multi-Region resilience for a key-value or document workload.

Use Global Tables when:

- You need active-active multi-Region access.
- You want automatic replication across Regions.
- You can accept the operational model and conflict behavior of multi-Region replication.

Be careful with assumptions:

- Global Tables are not the same as a single regional table with backups.
- They are designed for multi-Region applications, not just backup storage.
- They fit well when the application can tolerate eventual consistency across Regions.

**Availability, disaster recovery, and resilience choices**

The exam often tests whether you understand the difference between **availability** and **disaster recovery (DR)**.

- **Availability** means the system keeps working during component failures.
- **Disaster recovery** means the system can be restored or failed over after a larger outage.

Multi-AZ is usually the right answer for availability within a Region. Multi-Region is usually the right answer for DR or global user distribution.

Common patterns:

- **Single Region, Multi-AZ**:
  - Best for most production workloads.
  - Handles AZ failure.
  - Lower complexity and cost than multi-Region.
- **Multi-Region active-passive**:
  - One Region serves traffic; another is standby.
  - Good for DR with simpler operations than active-active.
  - Often paired with Route 53 failover.
- **Multi-Region active-active**:
  - Multiple Regions serve traffic simultaneously.
  - Best for global latency and resilience.
  - Requires careful data replication and conflict handling.

> If the question only asks for high availability, Multi-AZ is usually enough. If it asks for surviving a Region outage, you need Multi-Region.

**Common exam service scope summary**

| Service | Scope | Exam takeaway |
|---|---|---|
| IAM | Global | Identity and access are not Region-specific |
| Route 53 | Global | DNS and routing decisions happen globally |
| CloudFront | Global | Uses edge locations and regional edge caches |
| S3 | Global namespace, regional buckets | Bucket names are unique worldwide, data lives in a Region |
| EC2 | Regional, launched in an AZ | Instances are placed in a specific AZ |
| EBS | Zonal | Volume is tied to one AZ |
| RDS | Regional | Multi-AZ improves availability |

**Practical architecture examples**

A static website with global delivery often uses:

- S3 for origin storage.
- CloudFront for caching and global delivery.
- Route 53 for DNS.
- ACM (AWS Certificate Manager) for TLS certificates, often in `us-east-1` for CloudFront.

A resilient web application often uses:

- ALB in multiple AZs.
- Auto Scaling across AZs.
- RDS Multi-AZ or Aurora Multi-AZ.
- Route 53 failover or latency-based routing for multi-Region designs.
- S3 CRR for object replication if the app depends on shared assets.

Example high-level topology:

```text
Users
  -> Route 53
  -> CloudFront
  -> ALB in us-east-1
  -> EC2 Auto Scaling across 2+ AZs
  -> RDS Multi-AZ
  -> S3 for static assets and logs
```

For a multi-Region DR design:

```text
Primary Region: us-east-1
  - ALB
  - EC2 / ECS / Lambda
  - RDS primary
  - S3 source bucket

Secondary Region: us-west-2
  - Warm standby or pilot light
  - Replica database or restored snapshot
  - S3 replicated bucket
  - Route 53 failover record
```

List Regions:

```bash
aws ec2 describe-regions --all-regions
```

List AZs in a Region:

```bash
aws ec2 describe-availability-zones --region us-east-1
```

Create a CloudFront distribution from an S3 origin is usually done through the console or infrastructure as code, but the key exam idea is that CloudFront sits in front of the origin and caches at the edge.

**How to answer SAA-C03 questions faster**

When you see a global infrastructure scenario, ask:

1. Is the problem about latency, availability, or disaster recovery?
2. Is the service global, regional, or zonal?
3. Does the design need one Region, multiple AZs, or multiple Regions?
4. Is DNS routing, caching, or replication the right mechanism?
5. Does the question mention account boundaries, which may affect AZ naming?

A few reliable exam shortcuts:

- Choose **CloudFront** for global content delivery and edge caching.
- Choose **Route 53 latency routing** for directing users to the nearest healthy Region.
- Choose **Route 53 failover** for active-passive DR.
- Choose **S3 CRR** for cross-Region object replication.
- Choose **DynamoDB Global Tables** for multi-Region active-active data access.
- Choose **Multi-AZ** for high availability within a Region.
- Choose **Multi-Region** for disaster recovery or global user distribution.

**Key takeaways**

AWS global infrastructure is about placing the right service in the right scope. The exam expects you to distinguish between global control planes, regional resources, and zonal dependencies, then choose the simplest architecture that meets the requirement.

- **AZ letters are account-specific labels**, so do not over-interpret them.
- **CloudFront** uses edge locations and regional edge caches to reduce latency.
- **Route 53** steers traffic with DNS policies such as latency, geolocation, weighted, and failover.
- **S3** is globally named but regionally stored.
- **Multi-AZ** solves availability; **Multi-Region** solves broader DR and global access needs.
- **Global Tables** and **S3 CRR** are common building blocks for multi-Region designs.
