## **Course:** AWS Certified Solutions Architect – Associate Complete Course (Exam SAA-C03)

**AWS account structure fundamentals**

An AWS account is the primary isolation boundary for billing, identity, and many governance controls. In practice, you use multiple accounts to separate workloads, reduce blast radius, and apply different security and cost controls by environment or business unit.

A common design is a multi-account landing zone with dedicated accounts for:
- **Management**: owns AWS Organizations and global governance.
- **Security**: centralizes logging, detective controls, and security tooling.
- **Log archive**: stores immutable or long-retention logs.
- **Shared services**: hosts common networking, directory, or platform services.
- **Workload accounts**: separate dev, test, and production workloads.

This structure helps you apply least privilege, isolate failures, and simplify chargeback or showback.

**AWS Organizations overview**

AWS Organizations lets you centrally manage multiple AWS accounts. It is the core service for account hierarchy, policy enforcement, and consolidated billing.

Key capabilities include:
- **Consolidated billing**: one payer account receives charges for all linked accounts.
- **Organizational hierarchy**: organize accounts into organizational units (OUs) for policy targeting.
- **Service access enablement**: allow integrated AWS services to act across the organization.
- **Delegated administrator accounts**: assign specific admin duties to member accounts without using the management account for day-to-day operations.

The management account is the top-level account for the organization. You should use it sparingly because it has special control-plane authority.

**Consolidated billing and cost management**

Consolidated billing aggregates usage across accounts under one payer. This simplifies payment and can improve pricing benefits such as volume discounts and some Savings Plans or Reserved Instance sharing behavior, depending on the service and purchase model.

Important billing concepts:
- **Cost allocation tags**: tag resources or accounts so you can group and analyze spend.
- **Cost Explorer**: visualize historical spend, trends, and forecasts.
- **AWS Budgets**: set cost, usage, or reservation alerts and thresholds.
- **Chargeback/showback**: attribute spend to teams, products, or environments.

Practical guidance:
- Use separate accounts for production and non-production to avoid mixed cost reporting.
- Apply consistent account tags such as `Environment`, `Owner`, and `CostCenter`.
- Use budgets at both the organization and account level for early warning.

Example budget creation with the AWS CLI:
```bash
aws budgets create-budget \
  --account-id 123456789012 \
  --budget '{
    "BudgetName":"prod-monthly-budget",
    "BudgetLimit":{"Amount":"5000","Unit":"USD"},
    "TimeUnit":"MONTHLY",
    "BudgetType":"COST"
  }'
```

**AWS Organizations hierarchy and policy types**

Organizations uses a tree structure:
- Root
- OUs
- Accounts

Policies attached at higher levels affect all child OUs and accounts. This is useful for broad guardrails while still allowing exceptions lower in the tree.

Common policy types include:
- **Service control policies (SCPs)**: define the maximum permissions available to accounts and OUs.
- **Tag policies**: standardize tag keys and allowed values.
- **Backup policies**: centrally define backup plans and retention behavior for accounts.
- **AI services opt-out policies** and other governance policies may also appear depending on account features and exam scope.

SCPs do not grant permissions. They only limit what IAM identities in member accounts can do.

> An SCP must allow an action somewhere in the hierarchy, and IAM must also allow it. An explicit deny in an SCP overrides any allow in IAM.

**SCP evaluation behavior**

SCPs are evaluated as a permissions boundary for the account. The effective permissions are the intersection of:
- The SCPs attached to the root
- The SCPs attached to the OU path
- The SCPs attached to the account
- The IAM permissions in the account

Important exam-relevant behaviors:
- **Explicit deny wins** at every level.
- If an action is not allowed by SCPs, IAM cannot re-enable it.
- Root-level SCPs apply to all accounts unless overridden by more specific SCPs that still remain within the allowed set.
- Moving an account between OUs changes the SCP set immediately, which is useful for quarantine or environment separation.

Example SCP that denies disabling CloudTrail and Config:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenySecurityServiceTampering",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail",
        "config:StopConfigurationRecorder",
        "config:DeleteConfigurationRecorder"
      ],
      "Resource": "*"
    }
  ]
}
```

**Root user security best practices**

The account root user has unrestricted access to the account. You should treat it as a break-glass identity, not a daily admin account.

Best practices:
- Enable **MFA** on the root user immediately.
- Do not create access keys for the root user.
- Use the root user only for tasks that require it, such as changing account settings, closing the account, or certain billing and support operations.
- Store root credentials securely and limit knowledge of them.
- Prefer IAM roles, IAM users with MFA, or federated access for daily administration.

Additional controls:
- Review root activity through CloudTrail.
- Use strong password policies and secure recovery contacts.
- Lock down who can access the email address associated with the root account.

> If you find root access keys in an account, treat that as a security issue and remove them.

**Identity and access patterns across accounts**

For cross-account access, the most common pattern is IAM role assumption using AWS Security Token Service (STS). You create a role in the target account and allow a trusted principal from another account to assume it.

This is the preferred pattern when:
- A human or workload needs temporary access to another account.
- You want centralized identity with short-lived credentials.
- You need auditable, least-privilege access.

Example trust policy in the target account:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/DevOpsRole"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example permission policy attached to the role:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::shared-artifacts-bucket/*"
    }
  ]
}
```

You can assume the role with the CLI:
```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/CrossAccountReadOnly \
  --role-session-name audit-session
```

Resource-based policies are another cross-account option. Use them when the resource itself supports direct access control, such as S3 buckets, KMS keys, SQS queues, SNS topics, or Lambda functions.

**When to use IAM roles, resource policies, or AWS RAM**

Choose the access model based on what you are sharing:

| Use case | Best fit | Why |
|---|---|---|
| Human or workload needs temporary admin or app access in another account | IAM role assumption | Centralized, temporary, auditable |
| Another account needs direct access to a specific resource | Resource-based policy | Simple for supported services |
| Share subnets, Transit Gateway, or other supported resources across accounts | AWS RAM | Designed for resource sharing at scale |

AWS Resource Access Manager (AWS RAM) is especially useful for shared networking and centralized services. It lets you share resources such as:
- Subnets
- Transit Gateway attachments or route tables in supported scenarios
- License Manager configurations
- Other supported shared resources

Use AWS RAM when the goal is to share a resource itself, not to grant a principal broad permissions in another account.

**Security baseline for a multi-account environment**

A strong baseline usually starts with centralized logging and detective controls.

Recommended setup:
- Create a **security account** for central monitoring and response.
- Create a **log archive account** for long-term log storage.
- Enable an **organization trail** in AWS CloudTrail so all accounts send management events to a central S3 bucket.
- Use **AWS Config aggregator** to collect configuration snapshots and compliance data across accounts and Regions.
- Centralize CloudWatch Logs, Security Hub, GuardDuty, and related findings where appropriate.

Example organization trail concept:
```bash
aws cloudtrail create-trail \
  --name org-trail \
  --s3-bucket-name my-org-cloudtrail-logs \
  --is-organization-trail
```

Example AWS Config aggregator creation:
```bash
aws configservice put-configuration-aggregator \
  --configuration-aggregator-name org-aggregator \
  --organization-aggregation-source RoleArn=arn:aws:iam::123456789012:role/AWSConfigRole,AllAwsRegions=true
```

Practical logging pattern:
- Workload accounts send logs to central destinations.
- Security teams read from the security account.
- The log archive account stores immutable or tightly controlled log data.
- SCPs prevent disabling logging services in member accounts.

**AWS Control Tower and landing zones**

AWS Control Tower helps you set up and govern a multi-account environment using a prescriptive landing zone. It automates many baseline tasks that would otherwise be manual.

Typical Control Tower flow:
1. Set up the landing zone in the management account.
2. Create core accounts such as log archive and audit/security.
3. Register OUs for governed account placement.
4. Use Account Factory to provision new accounts with baseline guardrails.
5. Apply preventive and detective controls across the organization.

Account Factory is useful when you want standardized account creation with preapproved networking, logging, and governance settings. It reduces drift and helps enforce consistent account structure.

**Governance policies beyond SCPs**

SCPs are only one part of governance. Tag and backup policies are often tested in exam-adjacent scenarios because they solve common enterprise problems.

Tag policies:
- Standardize tag keys and values across accounts.
- Help with cost allocation, automation, and inventory.
- Do not automatically enforce tags on resources in the same way an SCP blocks actions; they define standards and visibility.

Backup policies:
- Centralize backup plan definitions across accounts.
- Help ensure retention and recovery objectives are consistent.
- Are useful when you need organization-wide backup governance for workloads with shared compliance requirements.

Use these policies when you need consistency across many accounts without manually configuring each one.

**Account security and operational guardrails**

A secure account baseline usually includes:
- MFA for privileged identities
- Least privilege IAM roles
- No root access keys
- CloudTrail enabled everywhere
- AWS Config enabled everywhere
- Centralized log retention
- GuardDuty and Security Hub in the security account
- SCPs that prevent disabling security services
- Standard tags for ownership and cost tracking

You should also separate duties:
- Security engineers manage detective controls.
- Platform teams manage account vending and baseline networking.
- Application teams manage workload resources in their own accounts.

This separation reduces accidental privilege escalation and makes audits easier.

**Practical account structure example**

A simple enterprise layout might look like this:
- Root
  - Security OU
    - Audit account
    - Log archive account
  - Infrastructure OU
    - Shared networking account
    - Shared services account
  - Workloads OU
    - Dev account
    - Test account
    - Prod account

In this model:
- The root and OU-level SCPs define global guardrails.
- The security account receives organization-wide findings.
- The log archive account stores CloudTrail and Config data.
- Workload accounts run applications with limited permissions.

**Exam-focused distinctions**

Remember these high-value differences:
- **Organizations** manages accounts and policies; it is not a networking service.
- **SCPs** limit permissions; they do not grant them.
- **IAM roles** are for principal access; **resource policies** are for resource access.
- **AWS RAM** shares resources; it is not a substitute for cross-account authentication.
- **Control Tower** automates landing zone setup and account vending.
- **Consolidated billing** simplifies cost management but does not merge account identities or resources.

**Quick decision guide**

Use this mental model:
- Need to control what accounts can do? Use **SCPs**.
- Need to standardize tags or backups? Use **tag policies** or **backup policies**.
- Need a user or service to access another account? Use **IAM role assumption**.
- Need direct access to a supported resource? Use a **resource-based policy**.
- Need to share infrastructure resources across accounts? Use **AWS RAM**.
- Need a governed multi-account foundation? Use **AWS Control Tower**.

**Common pitfalls**

- Assuming the management account should host workloads; it usually should not.
- Using the root user for daily administration.
- Thinking SCPs grant access instead of limiting it.
- Forgetting that IAM allow is still required even when SCPs permit an action.
- Centralizing logs without protecting the log archive account from tampering.
- Using one account for all environments and losing isolation and cost clarity.
- Confusing AWS RAM sharing with permission delegation.

**CLI and configuration examples**

List accounts in an organization:
```bash
aws organizations list-accounts
```

List OUs:
```bash
aws organizations list-organizational-units-for-parent \
  --parent-id r-examplerootid
```

Attach an SCP to an OU:
```bash
aws organizations attach-policy \
  --policy-id p-abc12345 \
  --target-id ou-abcd-11111111
```

Create a role trust relationship for cross-account access:
```json
{
```

**Summary**

AWS account structure is about isolation, governance, and cost control. AWS Organizations provides the hierarchy and policy framework, while SCPs, tag policies, and backup policies shape what accounts can do and how they are managed. For security, use root sparingly, centralize logs and configuration data, and prefer cross-account role assumption or resource sharing patterns that fit the use case.
