## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Reconnaissance tool selection**

These tools support different phases of the reconnaissance workflow. In CEH-style questions, the key is to choose the tool that matches the goal: passive intelligence gathering, relationship mapping, internet exposure discovery, or active validation.

- **Nmap** is an active network discovery and port-scanning tool.
- **Maltego** is a link-analysis and relationship-mapping platform.
- **theHarvester** collects public information from search engines, DNS, and other sources.
- **Shodan** indexes internet-exposed services and helps you query what is already visible on the public internet.

A useful exam rule is to separate **passive recon** from **active recon**:

- Passive recon uses public sources and does not directly probe the target.
- Active recon sends traffic to the target and can be detected by logs, IDS, or firewalls.
- Shodan is passive from your perspective, but Shodan itself scans and indexes exposed services on the internet.
- Nmap is the tool you run when you need live validation of hosts, ports, services, or firewall behavior.

**Nmap fundamentals**

Nmap is the most testable of the four tools because it combines host discovery, port scanning, service detection, OS fingerprinting, and scripting. It is often used to move from a broad target list to a smaller set of validated services.

Common scan types include:

- **TCP SYN scan** (`-sS`) sends a SYN and interprets the response without completing the full handshake.
- **TCP connect scan** (`-sT`) uses the operating system’s full TCP connection logic.
- **UDP scan** (`-sU`) checks UDP services, which are often slower and less deterministic than TCP.
- **ACK scan** (`-sA`) helps map firewall rules and identify `filtered` versus `unfiltered` ports.
- **FIN/NULL/Xmas scans** (`-sF`, `-sN`, `-sX`) are stealth-oriented TCP flag combinations that can behave differently depending on the target stack and filtering.
- **Window scan** (`-sW`) is a variation that can sometimes infer open versus closed state from TCP window size behavior.
- **Maimon scan** (`-sM`) uses FIN/ACK behavior and is less common, but it is worth recognizing for exam questions.

> ACK, FIN, NULL, Xmas, Window, and Maimon scans are often tested as “what do they tell you?” questions. In general, they are more about filtering behavior and stack responses than simple open/closed identification.

**Core Nmap commands**

Use these commands as baseline patterns:

```bash
nmap 192.168.1.10
nmap -sS -Pn -p 22,80,443 192.168.1.10
nmap -sV -O -Pn 192.168.1.10
nmap -A -T4 192.168.1.10
```

What they do:

- `nmap 192.168.1.10` runs a default scan against the target.
- `-sS` performs a SYN scan.
- `-Pn` skips host discovery and treats the host as up.
- `-p` limits the scan to selected ports.
- `-sV` attempts service and version detection.
- `-O` attempts OS detection.
- `-A` enables aggressive detection features, including version detection, OS detection, default scripts, and traceroute.
- `-T4` increases timing aggressiveness for faster scans on reasonably stable networks.

Nmap timing templates are commonly referenced as `-T0` through `-T5`. For exam prep, remember that `-T3` is the default timing template value used by Nmap’s timing presets, while actual scan behavior still depends on host discovery, retries, and network conditions.

**Host discovery and port selection**

Before full enumeration, you often need to identify live hosts and reduce scan scope.

```bash
nmap -sn 192.168.1.0/24
nmap -Pn -p 1-1000 192.168.1.50
nmap --top-ports 100 192.168.1.0/24
```

- `-sn` performs host discovery only and skips port scanning.
- `--top-ports` scans the most common ports instead of a full range.
- `-Pn` is useful when ICMP is blocked or host discovery is unreliable.
- `-p 1-1000` is a practical way to focus on common service ports first.

For UDP, scans are slower and often benefit from narrower scope:

```bash
nmap -sU --top-ports 20 192.168.1.10
nmap -sU -Pn -p 53,67,68,69,123,161 192.168.1.10
```

- `-sU` is useful for DNS, DHCP, TFTP, NTP, SNMP, and other UDP services.
- `--top-ports` helps when you want a quick UDP triage instead of a full sweep.
- `--defeat-rst-ratelimit` is sometimes relevant in edge cases where TCP response handling is distorted by rate limiting, though it is not a default choice.

**Service, version, and OS detection**

Service detection and OS fingerprinting are central to CEH-style interpretation.

```bash
nmap -sV --version-light 192.168.1.10
nmap -sV --version-all 192.168.1.10
nmap -O --osscan-guess 192.168.1.10
```

- `-sV` probes open ports to identify services and versions.
- `--version-light` reduces probe intensity for faster results.
- `--version-all` uses more probes and can improve accuracy.
- `-O` attempts to infer the operating system from TCP/IP stack behavior.
- `--osscan-guess` makes a best-effort guess when confidence is low.

Interpret results carefully:

- Version detection is probabilistic and can be affected by banners, proxies, load balancers, and service hardening.
- OS detection is strongest when Nmap has enough open and closed ports to compare responses.
- A single banner string is not proof of a specific product version.

**Nmap scripting engine**

The Nmap Scripting Engine (NSE) extends scanning with Lua-based scripts for discovery, enumeration, and some vulnerability checks. In exam scenarios, NSE is often the right answer when you need targeted service-specific enumeration.

Common script patterns:

```bash
nmap --script http-title -p 80,443 192.168.1.10
nmap --script http-enum -p 80,443 192.168.1.10
nmap --script smb-enum-shares -p 445 192.168.1.10
nmap --script ftp-anon -p 21 192.168.1.10
nmap --script ssl-enum-ciphers -p 443 192.168.1.10
nmap --script snmp-info -p 161 192.168.1.10
nmap --script vuln 192.168.1.10
```

Useful script categories:

- **default** runs scripts considered safe and broadly useful.
- **discovery** gathers information about services and hosts.
- **auth** checks authentication-related behavior.
- **vuln** runs vulnerability-oriented checks.
- **safe** avoids intrusive behavior.
- **intrusive** may be noisy or disruptive.

Practical guidance:

- Use `http-title` and `http-enum` for web enumeration.
- Use `smb-enum-shares` for SMB share discovery.
- Use `ftp-anon` to check for anonymous FTP access.
- Use `ssl-enum-ciphers` to assess TLS cipher support and protocol exposure.
- Use `snmp-info` when SNMP is suspected on UDP 161.
- Use `vuln` when you want broad vulnerability-oriented checks, but treat results as leads, not proof.

**Nmap output and reporting**

Nmap output matters in real engagements because you often need to preserve evidence and feed results into other tools.

```bash
nmap -oN scan.txt 192.168.1.10
nmap -oX scan.xml 192.168.1.10
nmap -oG scan.gnmap 192.168.1.10
nmap -oA recon/host1 192.168.1.10
```

Output formats:

- `-oN` writes normal human-readable output.
- `-oX` writes XML for parsing and reporting pipelines.
- `-oG` writes grepable output for quick text processing.
- `-oA` writes all major formats with one base filename.

A common workflow is to organize scans by target or phase:

```bash
mkdir -p recon/192.168.1.10
nmap -sV -O -oA recon/192.168.1.10/full 192.168.1.10
```

This helps when you later:

- parse XML into reporting tools,
- compare scan deltas over time,
- import results into spreadsheets, ticketing systems, or custom scripts.

**Nmap practical workflow**

A simple lab workflow is:

1. Discover live hosts with `-sn`.
2. Run a focused port scan on likely targets.
3. Add `-sV` and `-O` for service and OS identification.
4. Use NSE scripts for service-specific enumeration.
5. Save output with `-oA` for later analysis.

Example:

```bash
nmap -sn 192.168.56.0/24
nmap -sS -Pn --top-ports 100 -oA recon/top100 192.168.56.0/24
nmap -sV -O -Pn -p 22,80,139,445,3389 -oA recon/deep 192.168.56.10
nmap --script http-enum,ssl-enum-ciphers -p 80,443 192.168.56.10
```

**Maltego fundamentals**

Maltego is a link-analysis platform used to visualize relationships among people, domains, IP addresses, email addresses, organizations, DNS records, and other entities. It is especially useful when you need to turn scattered intelligence into a graph.

Core concepts:

- **Entities** are the objects on the graph, such as domains, IPs, emails, or social profiles.
- **Transforms** are actions that query a source and expand the graph.
- **Graphs** show relationships and help you spot pivots, clusters, and hidden infrastructure.
- **Machines** are automated transform workflows that run a sequence of steps.

Maltego is often used for:

- mapping an organization’s public footprint,
- pivoting from a domain to subdomains, emails, and infrastructure,
- identifying shared assets across multiple entities,
- visualizing relationships that are hard to see in a flat list.

**Maltego editions and transforms**

Maltego is available in different editions, and licensing affects what transforms and data sources you can use. In practice, some transforms are local, some are remote, and some require API keys or paid access.

Important practical points:

- Local transforms run on your machine and do not require a remote service.
- Remote transforms query external services and may require credentials or API keys.
- Built-in transform packs vary by edition and installation.
- Some transforms are limited by rate limits, licensing, or third-party API terms.

Common transform categories include:

- DNS and infrastructure lookups.
- Email and username pivots.
- Social media and web presence pivots.
- Metadata and document-related enrichment.
- Organization and relationship expansion.

**Maltego workflow example**

A typical workflow starts with a domain and expands outward:

1. Add a domain entity such as `example.com`.
2. Run DNS-related transforms to find subdomains and name servers.
3. Pivot to IP addresses and hosting infrastructure.
4. Add email addresses or employee names if discovered.
5. Use additional transforms to map related accounts or assets.

Example conceptual flow:

```text
Domain -> DNS names -> IP addresses -> Netblocks -> Related domains
Domain -> Email addresses -> Person entities -> Social profiles
```

This is useful when you need to answer questions like:

- What public assets belong to this organization?
- Which subdomains are exposed?
- Are multiple brands or services hosted on the same infrastructure?
- Which people or accounts are publicly associated with the target?

**Maltego practical notes**

- Use Maltego when you need visual correlation, not just raw enumeration.
- Use it to pivot from one clue to another, especially during OSINT (Open Source Intelligence) investigations.
- Treat transform results as leads that need validation.
- Be careful with false positives from shared hosting, CDN (Content Delivery Network) infrastructure, and recycled data.

**theHarvester fundamentals**

theHarvester is a reconnaissance tool focused on gathering public information such as emails, subdomains, hosts, names, and sometimes metadata from search engines and other sources. It is commonly used early in an engagement to build a target inventory.

Typical uses:

- discover email addresses associated with a domain,
- identify subdomains and hostnames,
- collect names and public identifiers,
- gather intelligence from multiple public sources.

A basic command looks like this:

```bash
theHarvester -d example.com -b all
```

However, backend availability and syntax vary by version. Some sources may be deprecated, renamed, or require API keys. For exam prep, remember that you should verify supported backends in the installed version rather than assuming every backend is universally available.

Common output formats include:

- HTML reports,
- XML output,
- JSON output,
- plain text summaries.

Example patterns:

```bash
theHarvester -d example.com -b bing -f results.html
theHarvester -d example.com -b duckduckgo -f results.xml
theHarvester -d example.com -b google -l 200
```

Use these examples cautiously: backend names and support differ across releases, and some sources may require authentication or may no longer be present in a given build.

Practical guidance:

- Use theHarvester for quick public footprint collection.
- Use it before active scanning to reduce blind probing.
- Cross-check results with DNS tools, Nmap, and Shodan.
- Expect duplicates and stale records; normalize and validate findings.

**theHarvester workflow example**

1. Start with a domain and a limited source set.
2. Export results in a machine-readable format.
3. Review emails, hosts, and subdomains.
4. Validate promising hosts with Nmap or DNS queries.

Example:

```bash
theHarvester -d example.com -b all -f recon/example-com
nmap -sn 203.0.113.0/24
```

If theHarvester returns subdomains, you can validate them with DNS resolution or targeted scans.

**Shodan fundamentals**

Shodan is a search engine for internet-exposed devices and services. It indexes banners and metadata from exposed systems, which makes it valuable for external attack surface discovery and exposure assessment.

Important distinction:

- Shodan is not a scanner you run against a target on demand.
- Shodan does perform its own scanning and indexing of exposed services on the internet.
- You query Shodan’s indexed data to find what is already visible publicly.

Core CLI commands:

```bash
shodan init YOUR_API_KEY
shodan info
shodan search apache
shodan host 8.8.8.8
```

Useful commands:

- `shodan count <query>` summarizes how many results match a query.
- `shodan download <filename> <query>` exports results to a file.
- `shodan domain <domain>` lists data Shodan has associated with a domain.
- `shodan scan submit <ip>` submits hosts for scanning, subject to account and policy limits.
- `shodan parse <file>` helps process downloaded data.
- `shodan convert` can transform between output formats in some workflows.

Example:

```bash
shodan count "org:Example Corp port:443"
shodan download example-443 "org:Example Corp port:443"
shodan domain example.com
```

**Shodan query basics**

Shodan queries combine keywords and filters. Results depend on how services were indexed, so version matching is not always exact string equality.

Common filters and patterns:

- `port:` targets a specific port.
- `org:` filters by organization.
- `net:` targets an IP range or subnet.
- `country:` filters by country code.
- `city:` filters by city.
- `hostname:` filters by hostname.
- `product:` filters by detected product name.
- `version:` filters by version metadata when available.
- `before:` and `after:` filter by scan date.
- `has_screenshot:` finds services with captured screenshots.
- `ssl.cert.subject.cn:` filters by certificate common name.
- `ssl.version:` filters by TLS version.
- `ssl.ja3s:` filters by TLS server fingerprint.
- `ssl.cert.*` filters are useful for certificate-based pivots.

Example queries:

```bash
shodan search "apache port:80 country:US"
shodan search "net:203.0.113.0/24 port:22"
shodan search "hostname:example.com has_screenshot:true"
shodan search "ssl.cert.subject.cn:example.com"
```

Use `net:` when you want subnet-focused searches, and use certificate filters when you want to pivot on TLS metadata.

> Do not assume Shodan version matching is exact in the same way as a local package manager. Banner normalization, proxying, and service wrappers can affect what appears in results.

**Shodan facets and summarization**

Facets help you summarize results by common attributes such as port, country, or organization. This is useful when you want a quick overview before drilling into individual hosts.

Example:

```bash
shodan search --facets port,org "apache"
```

Use facets to answer questions like:

- Which ports appear most often?
- Which organizations or countries dominate the result set?
- Which services are most exposed?

This is especially helpful when you need to prioritize follow-up validation.

**Shodan practical workflow**

A practical workflow is:

1. Search for a product, organization, or subnet.
2. Use `count` or facets to summarize exposure.
3. Export results with `download`.
4. Pivot to `host` or `domain` for deeper inspection.
5. Validate critical findings with Nmap.

Example:

```bash
shodan count "net:198.51.100.0/24"
shodan search --facets port "net:198.51.100.0/24"
shodan download subnet-198-51-100 "net:198.51.100.0/24"
shodan host 198.51.100.10
```

**Choosing the right tool**

Use the tool that matches the question being asked.

- Use **theHarvester** when you need public emails, subdomains, and names from open sources.
- Use **Maltego** when you need to visualize relationships and pivot across entities.
- Use **Shodan** when you need to find exposed services already indexed on the internet.
- Use **Nmap** when you need live validation, service detection, or port-state confirmation.

Exam-style guidance:

- If the question asks for a passive OSINT tool, think of theHarvester, Maltego, or Shodan.
- If the question asks for a live port scan or OS fingerprinting, choose Nmap.
- If the question asks for relationship mapping or pivoting, choose Maltego.
- If the question asks for internet-wide exposure search, choose Shodan.
- If the question asks for email and subdomain harvesting from public sources, choose theHarvester.

**Comparison of typical use cases**

| Tool | Best for | Output style | Typical limitation |
|---|---|---|---|
| Nmap | Live host, port, service, and OS validation | Scan results and scripts | Active and detectable |
| Maltego | Relationship mapping and pivots | Graphs and transforms | Depends on transforms and licensing |
| theHarvester | Public emails, hosts, and subdomains | Lists and reports | Source availability varies by version |
| Shodan | Internet-exposed service discovery | Query results and metadata | Data is only what Shodan has indexed |

**Lab-style decision examples**

- You need to confirm whether port 445 is actually open on a host you already suspect is exposed: use **Nmap**.
- You need to find public email addresses and subdomains for a company before scanning: use **theHarvester**.
- You need to visualize how a domain, IP range, and employee email addresses relate: use **Maltego**.
- You need to find all internet-facing SSH services for a subnet or organization already indexed online: use **Shodan**.

**Practical validation workflow**

A strong recon workflow often combines all four tools:

1. Use theHarvester to collect public names, emails, and subdomains.
2. Use Maltego to pivot and visualize relationships.
3. Use Shodan to see what is already exposed on the internet.
4. Use Nmap to validate live services and enumerate them directly.

Example combined workflow:

```bash
theHarvester -d example.com -b all -f recon/example
shodan search "org:Example Corp port:443"
nmap -sV -Pn -p 80,443,445 example.com
```

This sequence reduces noise, improves targeting, and gives you both passive intelligence and active confirmation.

**Common exam pitfalls**

- Do not treat passive recon results as proof of a live service.
- Do not assume every theHarvester backend exists in every version.
- Do not assume Shodan results are a complete inventory of a target.
- Do not assume Nmap banner strings are always exact version matches.
- Do not confuse ACK scan results with open/closed state; they usually indicate `filtered` or `unfiltered`.
- Do not assume `--spoof-mac` hides your IP address; it only changes the apparent MAC address on local-layer segments.

```bash
# theHarvester

# Maltego

```

**Key takeaways**

- **Nmap** is the active validation and enumeration tool.
- **Maltego** is the relationship-mapping and pivoting tool.
- **theHarvester** is the public-source harvesting tool.
- **Shodan** is the internet exposure search tool.
- CEH questions often test whether you can choose the right tool for passive recon, active scanning, or graph-based analysis.
