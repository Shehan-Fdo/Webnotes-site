**Domain and DNS fundamentals**

A domain name is the human-readable label used to reach internet resources, while the Domain Name System (<abbr title="Domain Name System">DNS</abbr>) translates names into IP addresses and other service data. In exam terms, you should understand both the naming hierarchy and how queries move through resolvers, authoritative servers, and caches.

- **FQDN**: A fully qualified domain name includes the complete path in the DNS hierarchy, such as `www.example.com.`.
- **TLD**: The top-level domain is the rightmost label, such as `.com`, `.org`, or `.net`.
- **Second-level domain**: The label directly left of the TLD, such as `example` in `example.com`.
- **Subdomain**: Any label left of the registered domain, such as `dev.example.com` or `mail.example.com`.
- **Zone**: The portion of the namespace managed by a DNS authority, which may cover a domain and its delegated children.

A practical way to think about DNS is as a distributed database with caching. Recursive resolvers answer on behalf of clients, iterative lookups follow referrals from root to TLD to authoritative servers, and authoritative servers provide the final records for a zone. Caches improve performance, but they also create security and troubleshooting considerations when records change.

**DNS query behavior and record types**

DNS queries are not all the same. In CEH-style questions, you should distinguish between recursive, iterative, and authoritative lookups, and know why specific record types matter during reconnaissance.

- **Recursive query**: The resolver does the work and returns a final answer to the client.
- **Iterative query**: Each server returns the best referral it has, and the client or resolver continues the search.
- **Authoritative answer**: The response comes from the server that hosts the zone data.
- **NXDOMAIN**: The queried name does not exist.
- **NODATA**: The name exists, but the requested record type does not.

Use `dig` to inspect query behavior and request specific record types:

```bash
dig example.com
dig example.com A
dig example.com AAAA
dig example.com MX
dig example.com NS
dig example.com TXT
dig example.com SOA
dig example.com CAA
dig example.com SRV
```

You can also query a specific resolver or authoritative server:

```bash
dig @1.1.1.1 example.com A
dig @8.8.8.8 example.com MX
dig @ns1.example.net example.com SOA
```

For trace-style resolution, `dig +trace` shows the referral chain from root servers to the authoritative answer:

```bash
dig +trace example.com
```

`nslookup` is still common in labs and interviews, although `dig` is usually preferred for detail:

```bash
nslookup -type=mx example.com
nslookup -type=txt example.com
nslookup 93.184.216.34
```

**Common DNS records and why they matter**

Different record types reveal different parts of an organization’s infrastructure. During reconnaissance, you use them to map mail flow, hosting providers, validation mechanisms, and service endpoints.

| Record | Purpose | Recon value |
|---|---|---|
| `A` | Maps a name to an IPv4 address | Identifies hosts and web endpoints |
| `AAAA` | Maps a name to an IPv6 address | Reveals dual-stack exposure |
| `CNAME` | Alias to another name | Shows CDN, SaaS, or delegated service use |
| `MX` | Mail exchanger | Reveals mail infrastructure and vendors |
| `NS` | Authoritative name servers | Identifies DNS hosting and delegation |
| `SOA` | Start of Authority | Shows primary server, admin contact, and zone timers |
| `TXT` | Arbitrary text | Often contains SPF, DKIM hints, verification tokens |
| `SRV` | Service location | Exposes service host and port pairs |
| `PTR` | Reverse mapping from IP to name | Useful for host discovery and asset naming |
| `CAA` | Certificate Authority Authorization | Shows which CAs may issue certificates |
| `NAPTR` | Naming authority pointer | Appears in some telephony and service discovery setups |

The `SOA` record is especially important. It contains the primary name server, the responsible party, a serial number, and timing fields. Those timing fields are `refresh`, `retry`, `expire`, and `minimum TTL`; they govern how secondary servers refresh zone data and how negative caching behaves, not zone transfer behavior in general.

Example `SOA` output:

```bash
dig example.com SOA +noall +answer
```

Example interpretation:

```text
example.com.  3600  IN  SOA  ns1.example.com. hostmaster.example.com. 2026042601 7200 900 1209600 300
```

- `ns1.example.com.` is the primary server.
- `hostmaster.example.com.` is the administrative contact.
- `2026042601` is the serial number.
- `7200` is the refresh interval.
- `900` is the retry interval.
- `1209600` is the expire interval.
- `300` is the minimum TTL used for negative caching in modern practice.

**WHOIS and RDAP reconnaissance**

WHOIS and RDAP (<abbr title="Registration Data Access Protocol">RDAP</abbr>) help you identify domain ownership, registration details, and infrastructure clues. WHOIS output is often partially redacted because of privacy services and registrar policies, so you should treat it as a lead source rather than a complete source of truth.

Key fields to look for:

- **Registrar**: The company managing the registration.
- **Registry**: The authoritative registry for the TLD.
- **Status codes**: Values such as `clientTransferProhibited` or `serverHold` can indicate lock or suspension states.
- **Name servers**: Often reveal DNS hosting providers.
- **Creation and expiry dates**: Useful for timeline analysis.
- **Registrant contact**: May be redacted or replaced by privacy proxy data.
- **Admin and tech contacts**: May expose organizational or outsourced support details.

Examples:

```bash
whois example.com
whois 93.184.216.34
```

RDAP is the modern, structured alternative to WHOIS and is often easier to parse programmatically:

```bash
curl -s https://rdap.org/domain/example.com | jq .
```

When privacy redaction is enabled, you may only see registrar details, status codes, and name servers. That is still useful because it can reveal hosting relationships, transfer locks, and whether the domain is managed through a privacy proxy.

**Reverse DNS and PTR-based host discovery**

Reverse DNS maps IP addresses back to hostnames using `PTR` records under the `in-addr.arpa` namespace for IPv4 and `ip6.arpa` for IPv6. This is useful for identifying naming conventions, infrastructure roles, and sometimes hidden assets.

Examples:

```bash
dig -x 93.184.216.34
dig -x 8.8.8.8
```

You can also query the reverse zone directly:

```bash
dig 34.216.184.93.in-addr.arpa PTR
```

Reverse DNS is not guaranteed to exist, and many organizations leave it incomplete. Even so, PTR data can help you:

- Identify mail servers and VPN endpoints.
- Infer internal naming patterns such as `web01`, `db02`, or `vpn-gw`.
- Correlate IP ranges with business units or cloud regions.
- Validate whether a discovered IP belongs to a known asset family.

**Subdomain enumeration workflow**

Subdomain enumeration is the process of discovering hostnames beneath a registered domain. It is a core recon skill because subdomains often expose staging systems, admin panels, APIs, forgotten services, and cloud assets.

A practical workflow combines passive and active methods:

1. Collect passive data from search engines, certificate transparency logs, public datasets, and third-party APIs.
2. Resolve discovered names to confirm which hosts are live.
3. Expand with brute force, permutations, and wordlists.
4. Validate findings with HTTP probing, DNS resolution, and response fingerprinting.
5. Prioritize interesting targets such as login portals, APIs, and non-production environments.

**Passive enumeration sources**

Passive enumeration does not directly query the target’s authoritative DNS for every guess. Instead, it uses public data sources that already observed the names.

Common sources and tools:

- **crt.sh**: Certificate Transparency log search; useful for names present in TLS certificates.
- **subfinder**: Aggregates passive sources quickly and is widely used in recon workflows.
- **amass**: Supports passive and active discovery, graphing, and permutation workflows.
- **dnsrecon**: Combines DNS enumeration, zone checks, and brute force options.
- **fierce**: Focuses on DNS reconnaissance and can help identify misconfigurations.
- **assetfinder**: Lightweight passive subdomain discovery.
- **theHarvester**: Useful for broader OSINT, including emails and hosts.

Example `crt.sh` usage:

```bash
curl -s "https://crt.sh/?q=%25.example.com&output=json" | jq -r '.[].name_value' | sort -u
```

Interpretation tips:

- Multiple names may appear in a single certificate entry, separated by newlines.
- Wildcards like `*.example.com` indicate broad certificate coverage, not necessarily a live host.
- Certificate data can reveal internal naming patterns, such as `dev`, `staging`, `api`, or region-specific hostnames.

Example `subfinder` workflow:

```bash
subfinder -d example.com -silent -all
```

Example `amass` passive enumeration:

```bash
amass enum -passive -d example.com
```

Example `dnsrecon` passive and targeted checks:

```bash
dnsrecon -d example.com
dnsrecon -d example.com -t std
```

**Active enumeration and brute force**

Active enumeration queries DNS directly with a wordlist. This is useful when passive sources miss internal or newly created names.

Common approaches:

- **Wordlist brute force**: Try common labels such as `dev`, `test`, `vpn`, `mail`, `api`, and `admin`.
- **Permutation generation**: Combine known names with prefixes and suffixes such as `staging-api`, `api-dev`, or `web01`.
- **Recursive discovery**: Feed newly found subdomains back into the process to discover deeper levels.

Example with `dnsx`:

```bash
dnsx -d example.com -w subdomains.txt -a -aaaa -cname -resp
```

Notes for exam prep:

- `dnsx` flags can vary by version, so focus on the concept: resolve names, collect record types, and inspect responses.
- Use a resolver list or default resolvers according to your installed version and lab environment.
- Combine resolution with filtering to reduce false positives.

Example with `ffuf` for subdomain brute force against a DNS wildcard-aware target is not the same as HTTP fuzzing. For DNS-based discovery, use a DNS-aware tool or a dedicated vhost workflow. For HTTP virtual host enumeration, calibrate carefully because wildcard responses can mislead results.

A safer vhost workflow is:

```bash
ffuf -u https://target.example/ -H "Host: FUZZ.example.com" -w subdomains.txt -ac
```

Use `-ac` for auto-calibration and then refine with explicit status code or size filters after you observe the baseline response. Do not assume `-fs 0` is meaningful unless you have verified the target’s response behavior.

**Permutation and wordlist strategy**

Wordlists are only as good as your target context. The best results come from combining generic lists with organization-specific clues.

Useful wordlist sources and ideas:

- Common DNS wordlists from SecLists.
- Names derived from technology stacks, such as `jira`, `git`, `jenkins`, `grafana`, or `kibana`.
- Environment labels such as `dev`, `qa`, `uat`, `stage`, and `prod`.
- Geographic or cloud labels such as `us-east`, `eu-west`, or `aws`.
- Role-based labels such as `vpn`, `mail`, `sso`, `auth`, and `portal`.

Permutation tools and workflows:

- **amass** can generate permutations from discovered names.
- **gotator** and similar tools can expand naming patterns.
- Manual permutations are still valuable when you see repeated naming conventions.

A simple workflow:

```bash
subfinder -d example.com -silent | tee subs.txt
amass enum -passive -d example.com | tee -a subs.txt
sort -u subs.txt > subs-unique.txt
dnsx -d example.com -w subs-unique.txt -resp
```

**Zone transfers and misconfigurations**

A zone transfer (<abbr title="Authoritative zone transfer">AXFR</abbr>) copies DNS zone data from a primary server to a secondary server. It is legitimate for replication, but it becomes a security issue when misconfigured to allow unauthorized clients.

Prerequisites for a successful transfer:

- The authoritative server must permit the request.
- The client must be allowed by ACLs or transfer policy.
- The zone must be configured to answer AXFR for that requester.
- The server must actually host the zone data.

Correct `dig` syntax:

```bash
dig @ns1.authoritative-example.net example.com AXFR
```

You can also target an IP address:

```bash
dig @203.0.113.53 example.com AXFR
```

How to verify success:

- You receive a full list of zone records, not a single answer.
- The output includes `SOA` at the start and end of the transfer.
- You see multiple hostnames, subdomains, and record types from the zone.

If the transfer is denied, you may see `Transfer failed`, `REFUSED`, or a timeout. In a lab, a successful AXFR often reveals the entire internal naming structure in one request, which is why it is a high-value misconfiguration.

**Security implications of DNS**

DNS is a frequent target because it is both foundational and widely trusted. Attackers use it for reconnaissance, redirection, and persistence, while defenders use it for validation, policy enforcement, and detection.

Important security concepts:

- **DNS spoofing**: Forging DNS responses to redirect users to malicious destinations.
- **Cache poisoning**: Inserting false records into a resolver cache so future clients receive the wrong answer.
- **DNS tunneling**: Encoding data in DNS queries and responses to bypass controls.
- **Typosquatting and lookalike domains**: Registering similar names to capture traffic or credentials.
- **Subdomain takeover**: Claiming a dangling DNS record that points to an unclaimed cloud resource.

DNSSEC (<abbr title="Domain Name System Security Extensions">DNSSEC</abbr>) adds cryptographic signatures to DNS data so resolvers can validate authenticity and integrity. It does not encrypt DNS traffic, but it helps prevent forged answers and some poisoning attacks.

`CAA`, `SPF`, `DKIM`, and `DMARC` records are also security-relevant:

- **CAA** restricts which certificate authorities may issue certificates for a domain.
- **SPF** (<abbr title="Sender Policy Framework">SPF</abbr>) lists authorized mail senders.
- **DKIM** (<abbr title="DomainKeys Identified Mail">DKIM</abbr>) uses signed mail headers and public keys in DNS.
- **DMARC** (<abbr title="Domain-based Message Authentication, Reporting, and Conformance">DMARC</abbr>) defines policy and reporting for email authentication.

Example records:

```bash
dig example.com TXT
dig selector1._domainkey.example.com TXT
dig _dmarc.example.com TXT
dig example.com CAA
```

These records are useful for reconnaissance because they can reveal mail providers, key selectors, and policy maturity. They also help defenders understand whether a domain is hardened against spoofing and certificate abuse.

**Modern resolver privacy and transport**

Traditional DNS queries are usually sent in cleartext over UDP or TCP port 53. Modern environments may also use DNS over TLS (<abbr title="DNS over TLS">DoT</abbr>) or DNS over HTTPS (<abbr title="DNS over HTTPS">DoH</abbr>) to improve privacy and reduce on-path inspection.

Practical implications:

- **DoT** encrypts DNS between client and resolver using TLS.
- **DoH** carries DNS inside HTTPS, which can blend with normal web traffic.
- **Privacy tradeoff**: Encryption protects against local interception, but it can also reduce visibility for network defenders.
- **Exam relevance**: Know that encrypted DNS changes monitoring and filtering strategies, but it does not change the DNS record model itself.

**Quick exam-oriented comparison**

| Topic | What to remember |
|---|---|
| Recursive vs iterative | Recursive returns a final answer; iterative returns referrals |
| WHOIS vs RDAP | WHOIS is legacy text output; RDAP is structured and modern |
| AXFR | Legitimate replication feature that becomes risky when misconfigured |
| PTR | Reverse DNS can reveal hostnames from IPs |
| Passive enumeration | Uses public sources like CT logs and OSINT |
| Active enumeration | Uses direct DNS queries and wordlists |
| DNSSEC | Validates integrity, not confidentiality |
| CAA | Limits which CAs can issue certificates |

**Practical workflow for CEH-style recon**

1. Start with WHOIS or RDAP to identify registrar, name servers, and registration clues.
2. Query `NS`, `SOA`, `MX`, `TXT`, and `CAA` records to map infrastructure and policy.
3. Pull passive subdomains from CT logs and aggregators.
4. Resolve discovered names and remove duplicates.
5. Brute force likely labels with a DNS-aware tool.
6. Check reverse DNS for IP ranges that appear related to the target.
7. Probe live hosts with HTTP tooling to identify exposed services and technologies.
8. Look for misconfigurations such as open AXFR, dangling CNAMEs, or weak email security records.

**Common pitfalls**

- Treating `ANY` as a guaranteed enumeration method; many servers minimize or refuse broad responses.
- Assuming WHOIS data is complete; privacy services often redact useful fields.
- Confusing a certificate wildcard with a live subdomain.
- Using only one source for subdomain discovery and missing assets hidden in another dataset.
- Ignoring reverse DNS, which can expose naming conventions and asset roles.
- Forgetting that `SOA` timing fields are about refresh and caching behavior, not a general zone-transfer timer.

**Minimal lab examples**

```bash
# Inspect authoritative servers
dig example.com NS +short

# Check mail and text policy
dig example.com MX +short
dig example.com TXT +short

# Look for certificate issuance policy
dig example.com CAA +short

# Enumerate reverse DNS
dig -x 203.0.113.10 +short

# Pull passive subdomains from CT logs
curl -s "https://crt.sh/?q=%25.example.com&output=json" | jq -r '.[].name_value' | sort -u

# Resolve a discovered list
dnsx -d example.com -w subs.txt -resp
```

**Key takeaways**

DNS is both an infrastructure service and a reconnaissance surface. For CEH v13, you should be able to explain the hierarchy, query the right record types, interpret WHOIS/RDAP output, enumerate subdomains with passive and active methods, and recognize security issues such as open zone transfers, cache poisoning risk, and weak email or certificate policy records.
