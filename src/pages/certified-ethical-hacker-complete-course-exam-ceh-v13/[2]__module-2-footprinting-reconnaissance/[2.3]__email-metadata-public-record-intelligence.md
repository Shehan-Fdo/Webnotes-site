## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

Email, metadata, and public record intelligence are core reconnaissance sources in ethical hacking and threat analysis. They help you identify people, infrastructure, naming patterns, document origins, and organizational relationships without directly touching protected systems. In practice, these sources often reveal more than intended because users, applications, and public registries leak context through headers, file properties, DNS records, and legal filings.

This topic sits in the **OSINT** (Open-Source Intelligence) phase of reconnaissance. You use it to build a target profile, validate assumptions, and support later activities such as phishing simulation, asset discovery, or social engineering assessments. The same techniques also help defenders find exposure before an attacker does.

**Email Intelligence**

Email intelligence focuses on what messages reveal beyond their visible content. A single message can expose sender infrastructure, mail routing, authentication results, and sometimes internal naming conventions. You usually analyze both the message body and the full header set.

Common email artifacts include:

- **From**, **To**, **Cc**, and **Bcc** fields, which may reveal relationships or hidden recipients.
- **Reply-To**, which can differ from the visible sender and redirect responses.
- **Message-ID**, which often contains domain or system identifiers.
- **Received** headers, which show the path a message took through mail servers.
- **Authentication-Results**, which may show SPF, DKIM, and DMARC outcomes.
- **X-** headers, which are vendor- or organization-specific and can leak internal details.

A practical workflow is to inspect headers first, then correlate them with DNS and public records.

```bash
# View full headers in common mail clients or exported message files
grep -iE '^(from|to|cc|bcc|reply-to|message-id|received|authentication-results|x-)' email.eml

# Extract all Received lines for routing analysis
grep -i '^received:' email.eml

# Check whether a sender domain publishes SPF, DKIM, and DMARC records
dig txt example.com
dig txt _dmarc.example.com
dig txt selector1._domainkey.example.com
```

**Why email headers matter**

Headers help you determine whether a message is legitimate, spoofed, relayed, or forwarded. They also help you identify the originating infrastructure and the mail transfer agents (MTAs) involved. If a message claims to come from a company but the routing path or authentication results do not align, that discrepancy is a strong indicator of spoofing or misconfiguration.

Key points to evaluate:

- **SPF** (Sender Policy Framework) checks whether the sending IP is authorized for the domain.
- **DKIM** (DomainKeys Identified Mail) verifies a cryptographic signature attached to the message.
- **DMARC** (Domain-based Message Authentication, Reporting, and Conformance) tells receivers how to handle failures and whether alignment is required.
- **MTA** (Mail Transfer Agent) names in `Received` headers can reveal hosting providers or internal relays.
- Time stamps can expose time zones, delays, or unusual forwarding behavior.

> Be careful when interpreting `Received` headers: the topmost entries are usually the most recent hops, and forged headers can appear in the body of a message. Trust the chain only when it is consistent with the mail system’s behavior.

**Email address intelligence**

Email addresses often encode naming conventions, roles, and organizational structure. Even when the mailbox is not active, the address pattern can help you infer valid usernames for later testing.

Typical patterns include:

- `first.last@company.tld`
- `flast@company.tld`
- `firstl@company.tld`
- `firstname.lastname@company.tld`
- role accounts such as `support@`, `sales@`, or `security@`

Useful checks include:

- Searching public sources for the same address or username.
- Comparing address formats across employees to infer a naming convention.
- Looking for role-based mailboxes that may be monitored by multiple people.
- Testing whether a domain accepts mail for a recipient without confirming delivery.

A common validation technique is SMTP enumeration, but modern systems often limit or log it heavily. Use it only in authorized assessments and expect anti-abuse controls.

```bash
# Basic DNS checks for mail infrastructure
dig mx example.com
dig a mail.example.com
dig txt example.com

# Test SMTP banner and capabilities
nc -nv mail.example.com 25
```

**Metadata intelligence**

Metadata is data about data. It can reveal authorship, software versions, file paths, timestamps, geolocation, device identifiers, and revision history. In security assessments, metadata often provides the bridge between a public document and the internal environment that created it.

Common metadata sources:

- Office documents such as `.docx`, `.xlsx`, and `.pptx`
- PDFs and image files
- Email files such as `.eml` and `.msg`
- Web pages and downloadable assets
- Source code repositories and exported archives

Important metadata fields include:

- **Author** and **last modified by**
- **Creation** and **modification** timestamps
- **Application** or software version
- **Company** and **manager** fields
- **Document path** or template names
- **GPS** coordinates in images, when present
- **EXIF** (Exchangeable Image File Format) data in photos

A document may expose internal usernames, workstation names, or directory structures. For example, a file created on a Windows workstation can reveal a local path such as `C:\Users\jdoe\Documents\...`, which may help you infer a username format.

```bash
# Inspect metadata on Linux
exiftool report.pdf
exiftool image.jpg
exiftool presentation.pptx

# Extract strings from a file for quick triage
strings suspicious.docx | head -n 50

# Identify file type before deeper analysis
file suspicious.pdf
```

**Practical metadata workflow**

1. Identify the file type with `file`.
2. Extract metadata with `exiftool` or a similar tool.
3. Review author names, timestamps, software versions, and embedded paths.
4. Correlate usernames, hostnames, and domains with other OSINT findings.
5. Remove or sanitize metadata before publishing or sharing files externally.

**Why metadata is valuable**

Metadata can confirm whether a document is authentic, recently edited, or repurposed from another environment. It can also expose operational mistakes, such as leaving internal comments, tracked changes, or hidden slides in a presentation. In an assessment, metadata often helps you pivot from a single artifact to a broader target map.

**Public record intelligence**

Public record intelligence uses legally accessible records to identify people, organizations, assets, and relationships. These records are often overlooked because they are not “technical” in the traditional sense, but they can reveal ownership, registration dates, contact details, and infrastructure clues.

Common public record sources include:

- **WHOIS** registration data for domains
- **DNS** records and passive DNS history
- **Certificate Transparency** logs
- Business registrations and corporate filings
- Court records and procurement documents
- Trademark and patent databases
- Property records and tax records, where publicly available
- Social media profiles and professional directories

WHOIS data can reveal registrant organization names, contact emails, and registration timelines, although privacy services often mask personal details.

```bash
# Domain registration and DNS intelligence
whois example.com
dig ns example.com
dig soa example.com
dig any example.com

# Check certificate transparency via browser or API tools
# Example: search for subdomains in CT logs using a public service
```

**DNS and certificate intelligence**

DNS records often expose more than the primary website. You may find mail servers, VPN endpoints, staging hosts, and third-party services. Certificate transparency logs can reveal subdomains that were issued certificates even if they are not linked from the main site.

Useful record types:

- **A** and **AAAA** records for host addresses
- **MX** records for mail exchangers
- **NS** records for authoritative name servers
- **TXT** records for SPF, DMARC, verification tokens, and service ownership
- **CNAME** records for aliases and hosted services
- **SRV** records for service discovery in some environments

A simple DNS review can uncover naming patterns such as `dev`, `test`, `mail`, `vpn`, `portal`, or region-specific hostnames.

```bash
dig a www.example.com
dig aaaa www.example.com
dig mx example.com
dig txt example.com
dig cname portal.example.com
```

**How attackers use public records**

Attackers use public records to map an organization’s external footprint, identify likely technology stacks, and find weakly protected assets. They may combine domain registration data, certificate logs, and social profiles to infer employee names and email formats. They may also use public filings to identify subsidiaries, vendors, or acquisition history.

Defenders use the same data to:

- Find forgotten subdomains and exposed services.
- Detect unauthorized certificates or shadow IT.
- Verify that public contact information is accurate.
- Identify impersonation risks from lookalike domains.
- Support incident response with historical ownership and infrastructure context.

**Correlation and pivoting**

The real value comes from correlation. A single email address, author name, or domain rarely tells the full story, but multiple weak signals can converge into a reliable profile.

Example pivot chain:

- A PDF metadata field shows `Author: j.smith`.
- The document path reveals `C:\Users\jsmith\Desktop\`.
- WHOIS or public profiles show a `jsmith@example.com` pattern.
- DNS records reveal `mail.example.com` and `vpn.example.com`.
- Certificate logs expose `dev.example.com` and `portal.example.com`.

This kind of pivoting helps you build a target map without direct intrusion.

**Tools and commands**

Common tools for this topic include:

- `exiftool` for metadata extraction
- `strings` for quick text discovery
- `file` for file identification
- `whois` for registration data
- `dig` and `nslookup` for DNS queries
- `grep`, `awk`, and `sed` for parsing headers and logs
- Mail client header viewers for message inspection

```bash
# Parse a message file for key header fields
awk 'BEGIN{IGNORECASE=1}
     /^from:|^to:|^cc:|^reply-to:|^message-id:|^received:|^authentication-results:|^x-/' email.eml

# Pull EXIF GPS data if present
exiftool -gps:all image.jpg

# Query a domain's mail policy
dig txt _dmarc.example.com
```

**Defensive best practices**

To reduce exposure, organizations should treat metadata and public records as part of their attack surface.

Recommended controls:

- Strip metadata from documents before external sharing.
- Use secure document templates that avoid internal paths and usernames.
- Configure email authentication with SPF, DKIM, and DMARC.
- Monitor for lookalike domains and unauthorized certificates.
- Limit sensitive details in public filings and contact pages where possible.
- Train users to avoid oversharing in attachments and profile fields.

**Common mistakes and pitfalls**

- Assuming visible sender information is trustworthy without checking headers.
- Treating metadata as proof of current ownership rather than historical context.
- Ignoring role accounts and shared mailboxes during enumeration.
- Overlooking DNS and certificate logs because they are publicly accessible.
- Forgetting that privacy services can hide WHOIS details while other records still expose infrastructure.

**Exam focus**

For CEH v13, expect questions that test your ability to:

- Identify what email headers reveal during reconnaissance.
- Recognize the purpose of SPF, DKIM, and DMARC.
- Explain how metadata can expose usernames, software, and file paths.
- Use public records to discover domains, subdomains, and organizational details.
- Correlate multiple OSINT sources into a broader intelligence picture.

**Summary**

Email, metadata, and public record intelligence are low-cost, high-value reconnaissance sources. They often expose the structure behind a target’s communications and public footprint, especially when combined. In practice, you use them to validate identities, discover infrastructure, and identify opportunities for deeper assessment while staying within authorized boundaries.
