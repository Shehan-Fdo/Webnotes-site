**Session fundamentals**

A web application is stateless by default, so it needs a way to remember who you are between requests. A **session** is the server-side concept that ties multiple requests to one authenticated user or browser context. The browser usually carries a session identifier, while the server stores the actual state.

In practice, sessions support login persistence, shopping carts, multi-step forms, and authorization decisions. The security goal is to keep the session bound to the right user while making it hard for an attacker to steal, predict, or reuse it.

- **State** is the information that must persist across requests.
- **Session ID** is the opaque value used to reference server-side session data.
- **Authentication** proves identity; **authorization** decides what the identity can do.
- Session handling should minimize exposure of secrets in URLs, logs, and client-side storage.

**Session lifecycle**

A typical session lifecycle starts when the user authenticates and the server creates a new session identifier. The application then sends that identifier to the browser, usually in a cookie. On later requests, the browser returns the cookie automatically, and the server looks up the associated session data.

A secure implementation rotates the session identifier after privilege changes, especially after login. This reduces the risk of **session fixation**, where an attacker forces a victim to use a known session ID and then reuses it after authentication.

1. User visits the application and receives an anonymous session.
2. User authenticates with credentials or another factor.
3. Server creates or upgrades the session and issues a fresh session ID.
4. Browser sends the session ID on each request.
5. Server validates the session, checks expiry, and applies access control.
6. Session ends by logout, timeout, revocation, or browser closure depending on policy.

> Never trust a session ID that appears in a URL. URL-based session tracking leaks through browser history, referrers, logs, and shared links.

**Cookies as session carriers**

A **cookie** is a small name-value pair stored by the browser and sent to matching domains and paths. Cookies are the most common mechanism for carrying session identifiers in web applications. They can also store preferences, tracking identifiers, and feature flags, but sensitive authentication data should not be placed directly in readable client-side values.

Important cookie attributes control scope and security:

- **`Secure`** sends the cookie only over HTTPS.
- **`HttpOnly`** blocks JavaScript access and helps reduce theft through cross-site scripting.
- **`SameSite`** limits cross-site sending and helps mitigate cross-site request forgery.
- **`Domain`** controls which hosts receive the cookie.
- **`Path`** limits where the cookie is sent within a site.
- **`Expires`** and **`Max-Age`** define persistence and lifetime.

A common secure session cookie looks like this:

```http
Set-Cookie: sessionid=8f3a9c2d7b1e4a6f; Path=/; Secure; HttpOnly; SameSite=Lax; Max-Age=1800
```

This example uses a short-lived identifier, restricts transport to HTTPS, prevents JavaScript access, and reduces cross-site request exposure.

**Cookie behavior and scope**

Cookies are matched by domain, path, and security flags. If the scope is too broad, unrelated subdomains may receive the cookie. If the scope is too narrow, the application may lose access to the session on some routes.

- Use the narrowest practical **domain** scope.
- Use a consistent **path** for the application.
- Prefer **host-only** cookies when possible.
- Avoid storing secrets or long-lived credentials in plain cookies.
- Treat cookies as bearer artifacts: whoever has the cookie can often act as the user until expiry or revocation.

A browser may store multiple cookies with the same name but different paths. This can create confusion if the application does not parse them consistently. Attackers sometimes exploit weak cookie handling to override or shadow values.

**Session identifiers**

A session identifier should be unpredictable, unique, and long enough to resist guessing. Good session IDs are generated with a cryptographically secure random number generator. They should not contain meaningful user data, timestamps, or sequential values.

Security properties to expect:

- **Entropy** is high enough to resist brute force.
- **Uniqueness** prevents collisions across users.
- **Opacity** means the value reveals nothing about the user or account.
- **Rotation** occurs after authentication and privilege elevation.
- **Expiration** limits the window for reuse.

Weak session IDs often show patterns such as incremental numbers, base64-encoded usernames, or predictable timestamps. These are dangerous because attackers can enumerate or infer valid sessions.

**Token-based authentication**

A **token** is a portable credential used to prove identity or authorization. In modern web systems, tokens often replace or complement classic server-side sessions. Common examples include API tokens, bearer tokens, and JSON Web Tokens (**JWTs**, pronounced “jots”).

Tokens are useful when:

- You need stateless authentication across distributed services.
- You expose APIs to mobile apps or single-page applications.
- You want to separate access tokens from refresh tokens.
- You integrate with identity providers and single sign-on.

A bearer token works like a session cookie in one important way: possession is enough. If an attacker steals it, they can often use it until it expires or is revoked.

Example API request with a bearer token:

```bash
curl -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
  https://api.example.com/v1/profile
```

**JWT structure and use**

A JWT is a compact token format with three parts separated by dots:

- **Header**: algorithm and token type
- **Payload**: claims such as subject, issuer, audience, and expiry
- **Signature**: integrity protection for the header and payload

Example structure:

```text
header.payload.signature
```

JWTs are often signed, not encrypted. That means the payload is usually readable by anyone who has the token, so you should not place secrets inside it. Use JWT claims for identity and authorization context, not for confidential data.

Common claims include:

- **`sub`**: subject, usually the user identifier
- **`iss`**: issuer
- **`aud`**: audience
- **`exp`**: expiration time
- **`iat`**: issued-at time
- **`nbf`**: not-before time
- **`jti`**: unique token identifier

A minimal JWT payload might look like this:

```json
{
  "sub": "user123",
  "iss": "auth.example.com",
  "aud": "api.example.com",
  "iat": 1714147200,
  "exp": 1714150800,
  "scope": "read:profile"
}
```

**Token lifecycle**

Token lifecycle management is critical because tokens are often valid outside the browser session that created them. A secure lifecycle usually includes issuance, storage, use, renewal, and revocation.

1. The client authenticates with credentials or an upstream identity provider.
2. The authorization server issues an access token, and sometimes a refresh token.
3. The client stores the token in a protected location appropriate to the platform.
4. The client presents the token to access protected resources.
5. The server validates signature, issuer, audience, expiry, and revocation status.
6. The client renews the token when needed or re-authenticates after expiry.
7. The server revokes or invalidates tokens when risk or logout occurs.

Access tokens should be short-lived. Refresh tokens should be more protected and rotated when possible. If a refresh token is stolen, it can extend an attacker’s access far beyond the original login window.

**Access token vs refresh token**

| Property | Access token | Refresh token |
|---|---|---|
| Purpose | Access APIs | Obtain new access tokens |
| Lifetime | Short | Longer |
| Exposure risk | Higher, used frequently | Higher impact if stolen |
| Storage | Memory or secure client storage | Strongly protected storage |
| Revocation need | Often expires quickly | Should support rotation and revocation |

Use short-lived access tokens to reduce replay risk. Use refresh token rotation to detect reuse and limit the damage from theft.

**Storage and transport choices**

Where you store a token matters as much as how you issue it. Browser storage options have different tradeoffs:

- **HttpOnly cookie** is safer against JavaScript theft but still vulnerable to CSRF if not paired with `SameSite` or anti-CSRF controls.
- **`localStorage`** is easy to use but exposed to JavaScript, so XSS can steal it.
- **`sessionStorage`** is limited to a tab/session context but still exposed to JavaScript.
- **In-memory storage** reduces persistence but may be lost on refresh.

For browser-based apps, many teams prefer HttpOnly cookies for session identifiers and separate anti-CSRF defenses. For APIs and native clients, bearer tokens are common, but you should protect them with platform-specific secure storage.

**Common attacks against sessions and tokens**

- **Session hijacking**: stealing a valid session cookie or token and replaying it.
- **Session fixation**: forcing a victim to authenticate into a known session.
- **Cross-site scripting (XSS)**: injecting script to steal tokens from JavaScript-accessible storage.
- **Cross-site request forgery (CSRF)**: tricking a browser into sending authenticated requests.
- **Replay attacks**: reusing captured tokens or cookies before they expire.
- **Token leakage**: exposure through logs, referrers, browser history, or insecure transport.

Mitigations should match the attack:

- Use **HTTPS** everywhere to protect tokens in transit.
- Set **`Secure`** and **`HttpOnly`** on session cookies.
- Use **`SameSite`** and anti-CSRF tokens for state-changing requests.
- Rotate session IDs after login and privilege changes.
- Keep token lifetimes short and support revocation.
- Validate issuer, audience, signature, and expiry on every request.
- Avoid putting tokens in URLs or client-side logs.

**Practical cookie inspection**

You can inspect cookies with browser developer tools or command-line tools. For example, `curl` can show response headers that set cookies:

```bash
curl -I https://example.com/login
```

If you need to test cookie handling in a lab, you can send a cookie manually:

```bash
curl -b "sessionid=8f3a9c2d7b1e4a6f" https://example.com/account
```

You can also observe how a server sets security flags:

```bash
curl -s -D - https://example.com/ -o /dev/null | grep -i set-cookie
```

These checks help you verify whether the application uses secure cookie attributes and whether session identifiers rotate as expected.

**Practical server-side patterns**

A secure application typically combines server-side session tracking with defensive cookie settings. In many frameworks, the session store may live in memory, Redis, a database, or a dedicated cache.

Example conceptual configuration:

```yaml
session:
  cookie_name: sessionid
  secure: true
  http_only: true
  same_site: Lax
  idle_timeout: 30m
  absolute_timeout: 8h
  rotate_on_login: true
  rotate_on_privilege_change: true
```

If you use a distributed environment, store session state in a shared backend so any application node can validate the session. This avoids sticky-session dependence and improves resilience.

**Best practices for exam and field use**

- Prefer **opaque session IDs** over meaningful identifiers.
- Regenerate the session after authentication and privilege escalation.
- Use **short-lived access tokens** and rotate refresh tokens.
- Store secrets in the least exposed location available to the platform.
- Enforce HTTPS and reject insecure downgrade paths.
- Validate token claims and session expiry on every request.
- Log session events carefully without exposing full credentials.
- Revoke sessions on logout, password reset, or suspicious activity.

**Quick comparison**

| Mechanism | Server state | Client carries secret | Typical use |
|---|---:|---:|---|
| Session cookie | Yes | Yes | Traditional web apps |
| Bearer token | Often no | Yes | APIs, SPAs, mobile apps |
| JWT | Often no | Yes | Distributed auth, SSO integrations |

**Key takeaways**

Sessions provide continuity, cookies commonly transport session IDs, and tokens extend the same idea into API-friendly authentication models. The main security challenge is protecting the bearer artifact from theft, fixation, replay, and leakage. Strong lifecycle controls, secure cookie flags, short expirations, and careful storage choices are the core defenses.
