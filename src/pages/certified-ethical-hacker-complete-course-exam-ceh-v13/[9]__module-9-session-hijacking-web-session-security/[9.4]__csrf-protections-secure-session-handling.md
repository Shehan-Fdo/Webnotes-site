**CSRF fundamentals**

<abbr title="Cross-Site Request Forgery">CSRF</abbr> is an attack where a victim’s authenticated browser is tricked into sending a request the user did not intend. The browser automatically includes ambient credentials such as cookies, so the server may treat the request as legitimate if it lacks anti-forgery checks.

CSRF matters most for **state-changing** actions:
- Changing a password
- Adding a payment method
- Transferring funds
- Changing email or MFA settings
- Creating API keys or admin users

CSRF is different from cross-site scripting (<abbr title="Cross-Site Scripting">XSS</abbr>). XSS runs attacker-controlled script in the victim’s origin; CSRF abuses the browser’s trust in existing authentication. XSS can often defeat CSRF defenses by reading tokens or issuing same-origin requests, so you should treat XSS prevention as part of CSRF defense.

**How CSRF works**

A typical CSRF flow uses a victim who is already logged in:
1. The victim authenticates to `bank.example`.
2. The browser stores a session cookie.
3. The victim visits an attacker-controlled page.
4. The page submits a forged request to `bank.example`.
5. The browser attaches the cookie automatically.
6. If the server does not verify intent, the action succeeds.

A simple forged request may look like this:

```html
<form action="https://bank.example/transfer" method="POST">
  <input type="hidden" name="to" value="attacker-account">
  <input type="hidden" name="amount" value="1000">
</form>
<script>
  document.forms[0].submit();
</script>
```

Modern browsers and frameworks reduce risk, but you should not rely on browser behavior alone. Defense must be enforced server-side.

**Primary CSRF defenses**

The most reliable defense is a **synchronizer token**. The server generates a random, unpredictable token and binds it to the user session. The application places the token in forms or request headers and validates it on submission.

Example HTML form:

```html
<form method="POST" action="/profile/email">
  <input type="hidden" name="csrf_token" value="b7f1c2d9e8a44f3d9f0a">
  <input type="email" name="email">
  <button type="submit">Update</button>
</form>
```

Example server-side validation logic:

```python
def update_email(request, session):
    token = request.form.get("csrf_token")
    if not token or token != session.csrf_token:
        return 403
    # continue with update
```

Use these defenses together:
- **Synchronizer tokens** for forms and state-changing requests
- **Double-submit cookies** when a server-side session store is not practical
- **SameSite cookies** to reduce cross-site cookie sending
- **Origin/Referer validation** as an additional check
- **Re-authentication** for sensitive actions
- **CAPTCHA or step-up verification** for high-risk workflows

**Synchronizer token pattern**

This pattern is the strongest general-purpose approach. The token:
- Is unique per session or per request
- Is unpredictable and high entropy
- Is validated server-side
- Is not accepted from attacker-controlled origins

Implementation guidance:
- Store the token in the server session or a signed server-side state object
- Embed it in HTML forms or return it in a response header for JavaScript clients
- Reject missing, malformed, expired, or reused tokens if your design is per-request
- Rotate tokens after login and privilege changes

For single-page applications, send the token in a custom header:

```javascript
fetch("/api/profile", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "X-CSRF-Token": window.csrfToken
  },
  body: JSON.stringify({ displayName: "Alice" })
});
```

**Double-submit cookie pattern**

In the double-submit pattern, the server sets a CSRF cookie and expects the same value in a request parameter or header. The server compares both values.

Example:
- Cookie: `csrf_token=9f3a...`
- Header: `X-CSRF-Token: 9f3a...`

This works best when:
- The token is random and unguessable
- The cookie is scoped correctly
- The application verifies equality on every state-changing request

Limitations:
- If an attacker can set or overwrite cookies for the target domain, the pattern weakens
- It is less robust than a server-side synchronizer token if implemented carelessly

**SameSite cookie attribute**

`SameSite` tells browsers when to send cookies in cross-site contexts. It is an important mitigation, but not a complete replacement for CSRF tokens.

Common values:
- `Strict` — cookie is sent only in same-site contexts
- `Lax` — cookie is sent on top-level navigations using safe methods, which helps usability
- `None` — cookie is sent in all contexts, but must also use `Secure`

Practical guidance:
- Use `SameSite=Lax` for many session cookies as a baseline
- Use `SameSite=Strict` for highly sensitive workflows if usability allows
- Use `SameSite=None; Secure` only when cross-site usage is required, such as embedded apps or federated flows

Example `Set-Cookie` header:

```http
Set-Cookie: sessionid=abc123; Path=/; HttpOnly; Secure; SameSite=Lax
```

> Do not treat `SameSite` as a standalone CSRF defense. Some flows, browser behaviors, and legacy clients can still create exposure, so keep token validation in place.

**Origin and Referer checks**

You can validate the `Origin` header for state-changing requests. If `Origin` is absent, some applications fall back to `Referer`, though this is less reliable because privacy tools or intermediaries may strip it.

Recommended approach:
- Prefer `Origin` for POST, PUT, PATCH, DELETE, and other unsafe methods
- Compare against an allowlist of trusted origins
- Reject requests with mismatched or missing values when your application can safely do so

Example Nginx-style logic is usually implemented in the application layer, but the concept is simple:
- Accept `https://app.example`
- Reject `https://evil.example`
- Reject `null` unless explicitly required by your design

**Session handling basics**

A session is the server-side or token-based mechanism that preserves authentication state across requests. Secure session handling protects the session identifier from theft, fixation, replay, and misuse.

Key goals:
- Make session identifiers unpredictable
- Keep them confidential in transit and at rest where applicable
- Limit their lifetime
- Bind them to the right security context
- Invalidate them when risk changes

Common session storage models:
- **Server-side session ID** stored in a cookie
- **Signed token** such as a JSON Web Token (<abbr title="JSON Web Token">JWT</abbr>) used carefully
- **Hybrid** designs with short-lived access tokens and refresh tokens

For CEH-level understanding, the main concern is that the session credential is effectively a bearer secret. Whoever possesses it can often act as the user.

**Session ID generation**

Session identifiers must be:
- Random and high entropy
- Unique across users and sessions
- Generated by a cryptographically secure random number generator
- Long enough to resist guessing

Avoid:
- Predictable IDs based on usernames, timestamps, or counters
- Reusing session IDs after login
- Encoding sensitive data directly into the session identifier

Example secure cookie issuance:

```http
Set-Cookie: sessionid=7f4d2c1a9b8e4f0d9c3a1b2e; Path=/; HttpOnly; Secure; SameSite=Lax
```

**Session fixation**

Session fixation occurs when an attacker forces or predicts a session identifier and then gets the victim to authenticate using that same identifier. After login, the attacker reuses the fixed session.

Defenses:
- Regenerate the session ID immediately after authentication
- Regenerate again after privilege elevation
- Invalidate the old session ID
- Refuse session IDs supplied in URLs or other unsafe channels

Example workflow:
1. Anonymous session is created.
2. User logs in.
3. Server issues a new session ID.
4. Old anonymous session is destroyed or detached.
5. New authenticated session is used going forward.

**Cookie security attributes**

Use cookie flags to reduce exposure:
- **HttpOnly** prevents JavaScript from reading the cookie, reducing theft via XSS
- **Secure** ensures the cookie is sent only over HTTPS
- **SameSite** reduces cross-site request sending
- **Path** and **Domain** should be as narrow as possible

Example:

```http
Set-Cookie: __Host-sessionid=abc123; Path=/; Secure; HttpOnly; SameSite=Lax
```

The `__Host-` prefix is useful because it requires:
- `Secure`
- `Path=/`
- No `Domain` attribute

This makes cookie scope stricter and reduces some cookie injection risks.

**Session timeout and rotation**

Use both idle and absolute timeouts:
- **Idle timeout** ends a session after inactivity
- **Absolute timeout** ends a session after a fixed maximum age

Rotate session identifiers:
- After login
- After password change
- After MFA enrollment or reset
- After privilege changes
- After suspicious activity

This reduces the value of stolen credentials and limits long-lived compromise.

**Logout and invalidation**

Logout should do more than delete a browser cookie. The server should:
- Invalidate the session record
- Remove refresh tokens if used
- Revoke related tokens where possible
- Clear cookies with matching scope

Example cookie clearing:

```http
Set-Cookie: sessionid=; Path=/; Max-Age=0; HttpOnly; Secure; SameSite=Lax
```

If you only clear the client cookie and keep the server session alive, an attacker who already stole the session can continue using it.

**Practical web server and framework examples**

In Express.js, you typically combine secure cookie settings with CSRF middleware:

```javascript
const session = require("express-session");
const csurf = require("csurf");

app.use(session({
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: {
    httpOnly: true,
    secure: true,
    sameSite: "lax"
  }
}));

app.use(csurf());

app.get("/form", (req, res) => {
  res.send(`
    <form method="POST" action="/transfer">
      <input type="hidden" name="_csrf" value="${req.csrfToken()}">
      <input name="amount">
      <button>Send</button>
    </form>
  `);
});
```

In Django, CSRF protection is built in for forms when middleware is enabled:

```python
MIDDLEWARE = [
    "django.middleware.csrf.CsrfViewMiddleware",
]
```

Template usage:

```html
<form method="post">
  {% csrf_token %}
  <button type="submit">Save</button>
</form>
```

In Spring Security, CSRF protection is commonly enabled by default for browser-based apps. For APIs, you may need to tune it carefully depending on authentication style.

**API-specific considerations**

APIs are not automatically immune to CSRF. Risk depends on how the API authenticates:
- Cookie-based authentication is CSRF-prone
- Bearer tokens in `Authorization` headers are less exposed to classic CSRF because browsers do not attach them automatically
- Mixed browser/API apps often need CSRF protection on cookie-authenticated endpoints

Guidance:
- Protect browser-facing endpoints with CSRF tokens
- Use `Authorization: Bearer <token>` for APIs when appropriate
- Do not store bearer tokens in places easily exposed to XSS, such as insecure local storage, without understanding the tradeoffs
- Consider `SameSite` and CORS separately; they solve different problems

**CORS is not CSRF protection**

<abbr title="Cross-Origin Resource Sharing">CORS</abbr> controls whether browser JavaScript can read cross-origin responses. It does not stop a browser from sending a cross-origin request.

That means:
- A request can still reach your server even if the response is blocked by CORS
- CSRF can succeed without the attacker reading the response
- You still need CSRF defenses for state-changing endpoints

**Testing and verification**

You can verify CSRF defenses by attempting a forged request from a different origin and checking whether the server rejects it.

Useful checks:
- Submit a form without a token
- Reuse an expired token
- Change the token value
- Send a request with a mismatched `Origin`
- Remove the session cookie and confirm the request fails
- Confirm sensitive actions require re-authentication

Example `curl` test:

```bash
curl -i -X POST https://app.example/profile/email \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data 'email=test@example.com'
```

If the endpoint is protected, you should see a `403 Forbidden` or a similar rejection.

For header-based token validation:

```bash
curl -i -X POST https://app.example/api/profile \
  -H 'Content-Type: application/json' \
  -H 'X-CSRF-Token: invalid' \
  --cookie 'sessionid=abc123' \
  --data '{"displayName":"Alice"}'
```

**Common mistakes**

- Relying only on `SameSite`
- Disabling CSRF protection for convenience and forgetting to re-enable it
- Accepting tokens from query strings, where they may leak via logs or referrers
- Reusing the same token forever
- Failing to rotate sessions after login
- Storing session IDs in URLs
- Not invalidating sessions on logout
- Assuming CORS blocks CSRF
- Forgetting that XSS can bypass many CSRF defenses

**Defensive checklist**

- Use HTTPS everywhere
- Set session cookies with `HttpOnly`, `Secure`, and an appropriate `SameSite`
- Regenerate session IDs after authentication and privilege changes
- Validate CSRF tokens on every unsafe request
- Check `Origin` or `Referer` as a secondary signal
- Require re-authentication for high-risk actions
- Keep sessions short-lived and revoke them on logout
- Prevent XSS to protect CSRF tokens and session cookies
- Log and alert on repeated CSRF failures or suspicious session reuse

**Quick reference**

| Control | Primary purpose | Notes |
|---|---|---|
| CSRF token | Verify user intent | Best general defense for browser forms |
| `SameSite` cookie | Reduce cross-site cookie sending | Helpful, but not sufficient alone |
| `HttpOnly` | Protect cookie from JavaScript access | Helps against cookie theft via XSS |
| `Secure` | Encrypt cookie in transit | Required for `SameSite=None` |
| Session rotation | Reduce fixation and replay risk | Rotate after login and privilege changes |
| Origin check | Validate request source | Useful secondary control |
| Re-authentication | Protect sensitive actions | Good for account recovery and payments |

**Exam focus**

For CEH-style questions, remember these distinctions:
- CSRF abuses authenticated browser state
- XSS often enables CSRF bypasses
- Session fixation is about reusing a known session ID
- `HttpOnly` protects cookies from script access, not from CSRF
- `SameSite` helps, but token validation remains the core defense
- Logout must invalidate server-side state, not just clear the browser cookie
