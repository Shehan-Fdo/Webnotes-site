## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Burp Suite overview**

Burp Suite is an intercepting proxy and web application testing platform used to observe, modify, replay, and automate HTTP(S) traffic. In CEH labs, you use it to understand how a browser and server exchange requests, how authentication works, and where input reaches the application.

Burp sits between your browser and the target application:

- Browser sends traffic to Burp proxy.
- Burp forwards traffic to the server.
- Responses return through Burp for inspection or modification.

Common use cases include:

- Intercepting login requests and session cookies.
- Replaying requests with modified parameters.
- Testing authorization by changing IDs, roles, or tokens.
- Reviewing hidden endpoints, API calls, and JavaScript-driven requests.
- Capturing traffic from mobile or alternate browsers when proxying is configured correctly.

**Burp Suite editions and Scanner role**

Burp Suite is available in different editions, and the feature set matters in exam labs:

- **Community Edition**
  - Includes core manual testing tools such as Proxy, Repeater, and Decoder.
  - Does not include the full automated **Burp Scanner** capability.
  - Is suitable for learning interception, request manipulation, and basic analysis.
- **Professional Edition**
  - Includes the full **Burp Scanner** for automated vulnerability discovery.
  - Adds workflow features that speed up testing, such as better automation and advanced analysis support.
  - Is the edition typically associated with active scanning in professional workflows.

Burp Scanner helps identify common issues such as:

- Reflected and stored input handling problems.
- Missing security headers or weak cookie flags.
- Some injection and access control patterns.
- Misconfigurations that are visible from HTTP behavior.

Limitations to remember:

- Scanner findings are not proof of exploitability by themselves.
- False positives and false negatives occur.
- Authentication, anti-CSRF controls, rate limiting, and application logic can reduce scan coverage.
- Some issues require manual validation in Repeater, browser testing, or source review.
- Scanner cannot reliably assess business logic flaws, chained attacks, or client-side-only weaknesses without context.

> Burp Scanner is useful for prioritization, but you still validate findings manually before drawing conclusions.

**Core Burp tabs and workflow**

Modern Burp workflows usually involve several main tabs and tools. You should know what each one does and when to use it.

- **Dashboard**
  - Central view for tasks, alerts, and recent activity.
  - Useful for seeing scan status, live issues, and project state.
  - Helps you track what Burp is doing without switching tools constantly.
- **Target**
  - Shows the site map and scope.
  - Lets you define what is in-scope and organize discovered paths.
  - Useful for mapping application structure and focusing on the target only.
- **Proxy**
  - Intercepts and logs browser traffic.
  - Lets you pause, inspect, edit, and forward requests and responses.
  - The most important tab for initial manual testing.
- **HTTP history**
  - Records requests passing through the proxy.
  - Useful for finding hidden API calls, redirects, and parameter changes.
- **Logger / traffic review**
  - Burp’s logging and traffic review views help you inspect request/response flow over time.
  - Similar in purpose to Logger++-style workflows: filter, search, and correlate traffic.
  - Helpful when you need to trace a sequence such as login, token refresh, and privileged action.
- **Repeater**
  - Sends a request to a controlled editor for repeated manual testing.
  - Best for changing one variable at a time and observing server behavior.
- **Intruder**
  - Automates repeated requests with payload variations.
  - Useful for fuzzing parameters, testing usernames, or checking input handling.
- **Decoder**
  - Encodes and decodes data such as URL encoding, Base64, and hex.
  - Useful when tokens or parameters are transformed.
- **Comparer**
  - Compares responses or request bodies.
  - Helpful for spotting subtle differences in authorization or error handling.
- **Sequencer**
  - Analyzes token randomness.
  - Useful when evaluating session identifiers or CSRF token quality.
- **Extender / BApp Store**
  - Adds extensions for extra functionality.
  - Common workflow additions include logging helpers, content discovery aids, and quality-of-life tools.

**Setting up Burp interception in a browser**

A reliable browser setup matters more than many beginners expect. Use a dedicated testing profile so you do not mix personal cookies, extensions, or cached sessions with lab traffic.

Typical lab setup steps:

1. Start Burp and confirm the proxy listener is active, usually on `127.0.0.1:8080`.
2. Create or use a dedicated browser profile for testing.
3. Configure the browser proxy to send HTTP and HTTPS traffic to Burp.
4. Install Burp’s CA certificate in the browser or operating system trust store for HTTPS interception.
5. Verify interception by visiting a test site and checking that requests appear in Burp Proxy.

Example proxy configuration concepts:

```text
HTTP proxy: 127.0.0.1
Port: 8080
HTTPS proxy: 127.0.0.1
Port: 8080
```

If you use a proxy helper such as FoxyProxy, you can switch Burp on and off quickly without changing system settings repeatedly. This is especially useful when you need to compare proxied and direct traffic.

Practical browser considerations:

- Use a separate profile to avoid extension noise and cookie contamination.
- Disable or isolate extensions that alter requests.
- Add proxy bypass rules only when needed.
- Exclude `localhost`, `127.0.0.1`, and internal lab hosts if proxying them causes loops or breaks local services.
- Remember that some browsers or apps may ignore system proxy settings unless explicitly configured.

**HTTPS interception and certificate issues**

Burp intercepts HTTPS by acting as a trusted man-in-the-middle in your lab environment. The browser must trust Burp’s CA certificate, or you will see certificate warnings and failed connections.

Common troubleshooting points:

- Import the Burp CA certificate into the correct trust store.
- Ensure the browser profile you are using has the certificate installed.
- Check whether the target uses certificate pinning.
- Confirm that the proxy listener is bound to the correct interface and port.
- Verify that the browser is not bypassing the proxy for the target domain.

Certificate pinning awareness is important for mobile apps and some hardened web clients. Pinning means the client expects a specific certificate or public key and may reject Burp’s certificate even if the CA is trusted. In those cases, you may need a different testing approach, such as a test build, instrumentation, or a lab-approved bypass method.

> If a site works over HTTP but fails over HTTPS through Burp, suspect certificate trust, pinning, or proxy bypass rules first.

**Proxy interception workflow**

Burp Proxy is where you inspect and modify traffic in real time. A typical workflow is:

1. Turn interception on when you want to stop a request.
2. Browse the application and capture a request.
3. Inspect headers, cookies, parameters, and body content.
4. Edit the request if needed.
5. Forward it and observe the response.
6. Send interesting traffic to Repeater for deeper testing.

Useful things to inspect in each request:

- Method, path, and query string.
- Host and scheme.
- Authentication headers such as `Authorization`.
- Cookies and session identifiers.
- CSRF tokens and anti-automation tokens.
- Content type and body format.
- Redirects and response headers.

Example intercepted request:

```http
POST /account/update HTTP/1.1
Host: example.test
Cookie: session=abc123; role=user
Content-Type: application/x-www-form-urlencoded
Content-Length: 41

email=test@example.com&csrf=9f2a1c
```

From here, you can test:

- Cookie tampering by changing `role=user` to another value.
- Token replay by resending the same request after a delay.
- CSRF validation by removing or altering `csrf`.
- Parameter tampering by changing `email` or adding hidden fields.

**Repeater for manual validation**

Repeater is the main tool for controlled request testing. It lets you resend a request as many times as needed while changing one element at a time.

Use Repeater when you want to:

- Confirm whether a parameter is validated.
- Compare responses before and after a change.
- Test access control by swapping identifiers.
- Replay a token or session-related request.
- Observe timing differences or error messages.

Example workflow:

```http
GET /api/orders/1001 HTTP/1.1
Host: example.test
Cookie: session=abc123
```

Try variations such as:

```http
GET /api/orders/1002 HTTP/1.1
Host: example.test
Cookie: session=abc123
```

Look for:

- Different response sizes.
- Different status codes.
- Error messages that reveal authorization logic.
- Data leakage from adjacent records.

**Intruder and controlled automation**

Intruder is useful when you need to test many payloads or parameter values. It is not a replacement for understanding the application; it is a force multiplier for focused testing.

Common uses:

- Username or identifier enumeration.
- Input fuzzing.
- Parameter value testing.
- Session or token pattern checks.
- Simple access control probing across many IDs.

Be careful with rate limits and account lockout controls. In real environments, aggressive Intruder use can trigger defenses or distort results.

**Decoder, Comparer, and Sequencer**

These tools support analysis when data is encoded or when differences are subtle.

- **Decoder**
  - Decode URL-encoded, Base64, hex, and other common formats.
  - Useful for tokens, cookies, and obfuscated parameters.
- **Comparer**
  - Compare two responses to identify small but meaningful differences.
  - Useful when one request returns a nearly identical page with a hidden authorization clue.
- **Sequencer**
  - Evaluate randomness in session IDs or tokens.
  - Helps you judge whether identifiers appear predictable.

Example decoding workflow:

```text
Original: dXNlcj1hZG1pbg==
Base64 decode: user=admin
URL decode: role%3Duser -> role=user
```

**Session, cookie, and authentication testing**

Authentication and session handling are central to CEH-style web testing. You should test how the application creates, stores, and validates identity state.

Key checks include:

- **Session fixation**
  - See whether the application accepts a pre-set session ID before login.
  - If the session ID remains unchanged after authentication, fixation may be possible.
- **Cookie tampering**
  - Modify cookie values to see whether the server trusts client-side state.
  - Test role flags, feature flags, and user identifiers.
- **Token replay**
  - Reuse a captured token or authenticated request.
  - Determine whether the token expires, rotates, or binds to context.
- **Session invalidation**
  - Log out and test whether the old session still works.
  - Check whether password changes or privilege changes invalidate active sessions.
- **Privilege switching**
  - Compare low-privilege and high-privilege requests.
  - Look for endpoints that rely on hidden UI controls instead of server-side authorization.

Practical validation example:

1. Log in as a normal user.
2. Capture the authenticated request in Burp.
3. Change a cookie or identifier that appears role-related.
4. Replay the request in Repeater.
5. Compare the response with the original.

If the server accepts the modified state, the application may trust client-controlled data too much.

**CSRF, tokens, and hidden fields**

Cross-Site Request Forgery (CSRF) testing often overlaps with session analysis. You want to know whether the application binds a request to the correct user and whether tokens are actually enforced.

Test for:

- Missing CSRF tokens.
- Static or reusable tokens.
- Tokens not tied to session or action.
- Hidden fields that can be modified without server-side checks.
- Requests that succeed when the token is removed.

Example form submission:

```http
POST /profile/change-email HTTP/1.1
Host: example.test
Cookie: session=abc123
Content-Type: application/x-www-form-urlencoded

email=new@example.com&csrf=token123
```

Validation ideas:

- Remove `csrf` and resend.
- Reuse the same token in a second request.
- Submit the request from a different browser session.
- Compare behavior across GET and POST variants if the application exposes both.

**Browser Developer Tools for web testing**

Browser Developer Tools are essential for client-side inspection and complement Burp rather than replacing it. They help you understand what the browser sees before and after requests are sent.

The most useful panels are:

- **Network**
  - Shows requests, responses, timing, and resource loading.
  - Lets you filter by XHR/fetch, document, JS, CSS, image, and websocket traffic.
  - Supports preserving logs across navigation so you do not lose earlier requests.
  - Lets you inspect request and response headers, cookies, payloads, and response bodies.
  - Often allows request replay or “resend” style actions depending on browser.
- **Elements**
  - Shows the live DOM (Document Object Model).
  - Useful for inspecting hidden fields, injected markup, and client-side changes.
- **Console**
  - Shows JavaScript errors, logs, and manual script execution.
  - Useful for testing functions, reading variables, and checking runtime behavior.
- **Sources**
  - Shows JavaScript files, source maps, and breakpoints.
  - Useful for stepping through client-side logic and understanding security checks.
- **Application / Storage**
  - Shows cookies, local storage, session storage, IndexedDB, and cache.
  - Useful for checking where tokens or state are stored.
- **Security**
  - Shows certificate and origin-related information in some browsers.

**Using the Network panel effectively**

The Network panel is one of the most practical tools for web testing. It helps you see what the browser actually sends, including API calls that may not be obvious in the page source.

Useful techniques:

- Enable **Preserve log** before navigating through login or multi-step flows.
- Use filters such as `Fetch/XHR` to isolate API calls.
- Search for keywords like `token`, `auth`, `admin`, or endpoint names.
- Inspect request headers for `Authorization`, `Origin`, `Referer`, and cookies.
- Inspect response headers for redirects, cache behavior, and security headers.
- Right-click a request and replay or copy it as cURL, fetch, or raw text when available.

Example of copying a request to cURL:

```bash
curl 'https://example.test/api/profile' \
  -H 'Cookie: session=abc123' \
  -H 'Accept: application/json'
```

This is useful because you can move from browser observation to command-line reproduction quickly.

**DevTools for JavaScript and client-side analysis**

Client-side code often reveals hidden endpoints, validation logic, and security assumptions. DevTools helps you inspect and test that logic directly.

Use the **Sources** panel to:

- Open JavaScript bundles and readable source files.
- Search across all loaded scripts for endpoint names, API paths, or secrets.
- Set breakpoints on functions that handle login, form submission, or token generation.
- Step through code to see how values are transformed before sending.
- Inspect source maps when available to recover more readable code.

Use the **Console** to:

- Read DOM values with JavaScript.
- Modify page state for testing.
- Call application functions directly.
- Check whether client-side validation can be bypassed.

Example console actions:

```javascript
document.querySelector('#role').value = 'admin';
localStorage.getItem('token');
fetch('/api/me').then(r => r.text()).then(console.log);
```

Security-testing uses include:

- Checking whether sensitive logic exists only in JavaScript.
- Verifying whether hidden fields are merely UI controls.
- Testing whether client-side checks can be bypassed by direct request submission.
- Looking for source maps that expose internal file names or comments.

**Storage, cookies, and state review**

The Application or Storage panel helps you understand where the browser keeps state.

Review:

- **Cookies**
  - Session identifiers.
  - Persistent login tokens.
  - Flags such as `Secure`, `HttpOnly`, `SameSite`, and `Domain`.
- **Local storage**
  - Often used for app preferences or tokens.
  - Accessible to JavaScript, so it is sensitive to XSS (Cross-Site Scripting).
- **Session storage**
  - Scoped to the browser tab or session.
  - Useful for temporary state and workflow data.
- **IndexedDB**
  - Sometimes used by modern single-page applications for offline or cached data.

Security observations:

- `HttpOnly` reduces JavaScript access to cookies.
- `Secure` limits cookies to HTTPS.
- `SameSite` affects cross-site request behavior and CSRF exposure.
- Overly broad `Domain` scope can expose cookies to subdomains.

**WebSockets and modern app traffic**

Many modern applications use WebSockets for live updates, chat, notifications, or collaborative features. Burp and DevTools can both help you observe this traffic.

What to look for:

- WebSocket handshake requests.
- Message frames and event names.
- Authentication tokens sent during connection setup.
- Client-side messages that may be replayed or modified.

Testing ideas:

- Observe whether the WebSocket connection reuses session cookies.
- Check whether messages are validated server-side.
- Compare behavior when you alter message content or sequence.
- Confirm whether sensitive actions are protected by authorization, not just UI controls.

**Practical lab procedure: set up Burp interception**

1. Open Burp and confirm the proxy listener is running on `127.0.0.1:8080`.
2. Create a dedicated browser profile for testing.
3. Configure the browser or FoxyProxy to use Burp as the HTTP and HTTPS proxy.
4. Import the Burp CA certificate into the browser trust store.
5. Visit a test site and confirm requests appear in Burp Proxy history.
6. Turn interception on and capture a login or form submission.
7. Send the request to Repeater and test one change at a time.
8. Use DevTools Network to compare what the browser sends with what Burp captures.
9. If HTTPS fails, check certificate trust, proxy bypass rules, and pinning behavior.
10. If local services break, add `localhost` and `127.0.0.1` to proxy bypass exclusions.

**Practical lab procedure: inspect a login flow**

1. Open DevTools and enable Network logging with Preserve log.
2. Log in to the application.
3. Identify the login request, redirect chain, and any token exchange.
4. Send the login request to Burp Repeater.
5. Modify the username, password, or token and observe the response.
6. Check whether the session cookie changes after authentication.
7. Log out and test whether the old session still works.
8. Review cookies and storage for any persistent authentication artifacts.

**Practical lab procedure: test authorization**

1. Capture a request to a user-specific resource.
2. Note the object identifier, such as `/api/orders/1001`.
3. Change the identifier to another plausible value.
4. Replay the request in Repeater.
5. Compare status codes, response bodies, and timing.
6. Confirm whether the server enforces authorization on the backend.

**Common troubleshooting checklist**

- Burp shows no traffic:
  - Proxy is not configured in the browser.
  - Wrong listener port or interface.
  - Browser profile is bypassing the proxy.
- HTTPS errors:
  - Burp CA certificate is missing or not trusted.
  - Certificate pinning is blocking interception.
  - Proxy bypass rules are interfering.
- Requests loop or fail locally:
  - `localhost` or `127.0.0.1` is not excluded properly.
  - A local app is not expecting a proxy.
- Missing requests in Burp:
  - Traffic is going through another browser profile or extension.
  - The app uses a different client, such as a native app or WebSocket channel.
- DevTools and Burp disagree:
  - DevTools shows browser-side behavior.
  - Burp shows proxy-visible traffic.
  - Compare both to understand redirects, caching, and client-side rewrites.

**Exam focus and practical takeaways**

For CEH v13, you should be able to explain not just what Burp Suite is, but how you use it in a real workflow. That includes proxy setup, request interception, manual validation, and the difference between automated scanning and manual testing.

You should also be comfortable with browser DevTools because many findings start there:

- Network panel for request tracing and replay.
- Sources and Console for client-side logic.
- Storage panels for cookies and tokens.
- Filtering and preserving logs for multi-step workflows.

The strongest workflow is to combine both tools:

- Use DevTools to understand what the browser is doing.
- Use Burp to modify and replay traffic.
- Use Repeater for precise validation.
- Use Scanner only as a helper, not as proof.
