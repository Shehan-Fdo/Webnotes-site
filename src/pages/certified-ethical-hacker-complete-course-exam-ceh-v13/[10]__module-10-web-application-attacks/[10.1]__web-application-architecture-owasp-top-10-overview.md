## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Web application architecture fundamentals**

A web application is usually a chain of components that process a browser request, apply business logic, and return a response. In CEH-style questions, you often identify the role of each layer from a diagram, a log snippet, or a traffic flow description.

Common request flow:

- Browser or mobile client sends an **HTTP** or **HTTPS** request.
- A **reverse proxy** or **load balancer** may terminate TLS and distribute traffic.
- A **web server** serves static content or forwards dynamic requests.
- An **application server** runs business logic, authentication, and API handlers.
- A **database server** stores persistent data, often on a separate network segment.

A typical 3-tier architecture looks like this:

```text
Client
  |
  v
[Load Balancer / Reverse Proxy]
  |
  v
[Web Tier]  --->  [App Tier]  --->  [Database Tier]
```

This separation matters because it limits exposure. If the web tier is compromised, the attacker should not automatically reach the database tier unless network controls, credentials, or application flaws allow it.

**Common deployment models**

- **Single-tier**: web, app, and database run on one host.
  - Easy to deploy.
  - Weak isolation and poor scalability.
- **Two-tier**: client talks directly to a server that also handles data access.
  - Common in small internal apps.
  - Fewer moving parts, but less separation.
- **Three-tier**: presentation, logic, and data are separated.
  - Most common exam model.
  - Easier to scale and secure.
- **N-tier / distributed**: multiple app servers, caches, queues, API gateways, and microservices.
  - Common in modern cloud environments.
  - More complex trust boundaries and attack paths.

**Core components you should recognize**

- **Web server**: handles HTTP requests, static files, TLS, and routing.
  - Examples: Apache, Nginx, Microsoft IIS.
- **Application server**: executes code and business logic.
  - Examples: Tomcat, JBoss/WildFly, Node.js, .NET runtime.
- **Database server**: stores records and enforces data integrity.
  - Examples: MySQL, PostgreSQL, Microsoft SQL Server, Oracle.
- **Reverse proxy**: hides internal servers, performs TLS offload, caching, or routing.
- **Load balancer**: distributes requests across multiple back-end servers.
- **Web Application Firewall (WAF)**: filters malicious HTTP patterns before they reach the app.

**Why architecture matters in attacks**

- A misconfigured reverse proxy can expose internal headers, admin paths, or origin servers.
- A load balancer can create session persistence issues if cookies are not handled correctly.
- A WAF may block obvious payloads but still miss encoded, logic-based, or multi-step attacks.
- Database separation reduces blast radius, but SQL injection can still reach the database through the application.

**HTTP request and response elements**

CEH questions often expect you to identify what part of the request is being manipulated.

- **Method**: action requested from the server.
  - `GET` retrieves data.
  - `POST` submits data.
  - `PUT` updates or replaces data.
  - `PATCH` partially updates data.
  - `DELETE` removes data.
  - `OPTIONS` reveals supported methods.
- **Path**: resource location, such as `/admin/users`.
- **Query string**: parameters after `?`, such as `?id=10`.
- **Headers**: metadata such as `Host`, `Cookie`, `Authorization`, `Referer`, `Origin`, `User-Agent`.
- **Body**: form data, JSON, XML, or file content.
- **Status code**: server result.
  - `200` OK
  - `301`/`302` redirect
  - `400` bad request
  - `401` unauthenticated
  - `403` authenticated but forbidden
  - `404` not found
  - `500` server error
  - `502`/`503` upstream or service unavailable

Example request:

```http
POST /login HTTP/1.1
Host: app.example.com
Content-Type: application/x-www-form-urlencoded
Cookie: session=abc123

username=alice&password=Secret123!
```

**Cookies, sessions, and tokens**

These terms are often confused in exam questions.

- **Cookie**: data stored by the browser and sent automatically with requests.
- **Session**: server-side state associated with a user after authentication.
- **Session ID**: token that links the browser to the server-side session.
- **Token**: a credential used for authentication or authorization, often in APIs.

Important distinctions:

- Authentication answers: “Who are you?”
- Authorization answers: “What are you allowed to do?”
- A user can be authenticated but still not authorized to access a resource.

Common session properties:

- `HttpOnly` reduces JavaScript access to cookies.
- `Secure` sends cookies only over HTTPS.
- `SameSite` helps reduce CSRF risk.
- Session IDs should be random, long, and rotated after login.

Example cookie:

```http
Set-Cookie: sessionid=9f3a2c1d8e7b4a6f; Secure; HttpOnly; SameSite=Lax
```

**Session fixation, hijacking, and privilege escalation**

- **Session fixation**: the attacker forces or predicts a session ID, then the victim logs in using that same ID.
  - Defense: regenerate the session ID after authentication.
- **Session hijacking**: the attacker steals a valid session token and reuses it.
  - Defense: HTTPS, secure cookies, short lifetimes, anomaly detection.
- **Privilege escalation**: the attacker gains higher permissions than intended.
  - Vertical escalation moves from user to admin.
  - Horizontal escalation accesses another user’s data at the same privilege level.

**Authentication vs authorization**

- **Authentication** verifies identity using passwords, MFA, certificates, or tokens.
- **Authorization** checks access rights after identity is known.
- A broken access control issue often occurs when the app authenticates correctly but fails to enforce authorization on every request.

Example:

- User A logs in successfully.
- User A changes `/profile?id=1002` to `/profile?id=1003`.
- If the app returns another user’s profile, the flaw is authorization failure, not authentication failure.

**Web application testing methodology**

In CEH scenarios, you usually identify issues by combining reconnaissance, proxy inspection, and manual validation.

1. **Map the application**
   - Identify pages, forms, APIs, parameters, and hidden endpoints.
   - Use crawling, directory discovery, and manual browsing.
2. **Observe requests**
   - Intercept traffic with a proxy such as Burp Suite or OWASP ZAP.
   - Review methods, headers, cookies, and response codes.
3. **Test input handling**
   - Try boundary values, special characters, encoding, and injection markers.
   - Compare server responses for errors, redirects, or timing differences.
4. **Test access control**
   - Modify IDs, roles, paths, and HTTP methods.
   - Replay requests as another user or without a token.
5. **Validate impact**
   - Confirm whether the issue exposes data, changes state, or bypasses controls.
   - Avoid assuming a vulnerability from a single error message.

Useful CLI examples:

```bash
curl -i https://app.example.com/account
curl -i -X OPTIONS https://app.example.com/api/users
curl -i -H "Cookie: session=abc123" https://app.example.com/admin
```

**OWASP Top 10 2021 overview**

The OWASP Top 10 is a risk awareness list, not a complete vulnerability catalog. CEH questions often ask you to match a weakness to the correct category or recognize a likely test technique.

2021 categories:

- **A01: Broken Access Control**
- **A02: Cryptographic Failures**
- **A03: Injection**
- **A04: Insecure Design**
- **A05: Security Misconfiguration**
- **A06: Vulnerable and Outdated Components**
- **A07: Identification and Authentication Failures**
- **A08: Software and Data Integrity Failures**
- **A09: Security Logging and Monitoring Failures**
- **A10: Server-Side Request Forgery (SSRF)**

Older OWASP Top 10 versions used different labels. For exam purposes, know that:
- **Injection** remains a major category, but older lists separated SQL injection more explicitly.
- **Broken Authentication** in older versions is now part of **Identification and Authentication Failures**.
- **Sensitive Data Exposure** is now covered mainly by **Cryptographic Failures**.
- **XML External Entities (XXE)** was a separate older category and is now usually treated as an injection or parser-related issue depending on context.

**Category-to-technique mapping for CEH recognition**

| OWASP 2021 category | Typical testing or attack technique |
|---|---|
| A01 Broken Access Control | IDOR testing, forced browsing, parameter tampering, method tampering, role changes |
| A02 Cryptographic Failures | Weak TLS, missing encryption, exposed secrets, predictable tokens, poor key handling |
| A03 Injection | SQL injection, command injection, LDAP injection, template injection, OS command payloads |
| A04 Insecure Design | Abuse of business logic, missing rate limits, weak workflow controls, insecure reset flows |
| A05 Security Misconfiguration | Default credentials, verbose errors, exposed admin panels, directory listing, bad CORS |
| A06 Vulnerable and Outdated Components | Known CVEs, outdated libraries, dependency scanning, version fingerprinting |
| A07 Identification and Authentication Failures | Brute force, credential stuffing, session fixation, weak MFA handling, password reset abuse |
| A08 Software and Data Integrity Failures | Supply-chain tampering, insecure deserialization, unsigned updates, CI/CD trust issues |
| A09 Security Logging and Monitoring Failures | Missing alerts, log tampering, no audit trail, delayed detection |
| A10 SSRF | Internal URL fetch abuse, metadata service access, redirect chaining, URL parser tricks |

**Broken access control and IDOR**

Broken access control is one of the most common web risks in exam questions. A frequent form is **IDOR**: Insecure Direct Object Reference.

Example:

```http
GET /api/invoices/1001 HTTP/1.1
Host: app.example.com
Authorization: Bearer eyJ...
```

If changing `1001` to `1002` returns another user’s invoice without checking ownership, the app has an IDOR flaw.

Common signs:

- Sequential object IDs in URLs or JSON bodies.
- Hidden fields that control role, price, or account number.
- Direct access to admin or internal endpoints.
- HTTP method changes that bypass UI restrictions.

Testing ideas:

- Change numeric IDs.
- Swap UUIDs if predictable patterns exist.
- Remove the session token and observe the response.
- Replay a request as a lower-privileged user.
- Try `GET`, `POST`, `PUT`, and `DELETE` on the same endpoint.

**Injection examples**

Injection occurs when untrusted input becomes part of a command or query.

SQL injection example:

```sql
' OR '1'='1
```

Possible test payloads:

```text
admin'--
' UNION SELECT null, version()--
1' AND SLEEP(5)--
```

Command injection example:

```text
127.0.0.1; id
127.0.0.1 && whoami
```

Path traversal example:

```text
../../../../etc/passwd
..%2f..%2f..%2f..%2fetc%2fpasswd
```

File upload bypass ideas:

- Change extension from `.php` to `.php.jpg`.
- Use double extensions such as `shell.php.png`.
- Alter MIME type in the request.
- Bypass client-side checks by intercepting the upload in a proxy.
- Test whether the server validates content, not just filename.

Example upload request:

```http
POST /upload HTTP/1.1
Host: app.example.com
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary

------WebKitFormBoundary
Content-Disposition: form-data; name="file"; filename="image.php.jpg"
Content-Type: image/jpeg

<?php system($_GET['cmd']); ?>
------WebKitFormBoundary--
```

**SSRF patterns**

**SSRF** means the server makes a request to a destination chosen by the attacker. This is especially important when the server can reach internal services that the attacker cannot.

Common payload patterns:

```text
http://127.0.0.1/
http://localhost:8080/
http://169.254.169.254/latest/meta-data/
http://10.0.0.5/admin
```

What to look for:

- URL fetch features such as image import, webhook testing, PDF generation, or link preview.
- Differences in response time, error messages, or DNS lookups.
- Access to cloud metadata endpoints or internal admin panels.

Defenses include allowlists, URL parsing hardening, egress filtering, and blocking access to link-local metadata services.

**Insecure deserialization**

Insecure deserialization happens when an application trusts serialized data from an untrusted source.

Why it matters:

- It can lead to authentication bypass, privilege escalation, or remote code execution.
- It often appears in frameworks that store objects in cookies, hidden fields, or API payloads.

Exam clues:

- Binary or encoded blobs in cookies.
- Unexpected object names in request bodies.
- Errors mentioning deserialization, serialization, or object graphs.

Defenses:

- Avoid deserializing untrusted data.
- Use signed and integrity-protected formats.
- Prefer simple data structures such as JSON with strict schema validation.
- Enforce allowlists for allowed types.

**Authentication bypass and common weaknesses**

Authentication bypass means the attacker gains access without valid credentials or without completing the intended login flow.

Common causes:

- Default or weak credentials.
- Logic flaws in password reset or MFA flows.
- Missing checks on alternate endpoints.
- Trusting client-side role values.
- Exposed debug or staging interfaces.

Examples:

- A login form accepts `admin` with a blank password because of a logic bug.
- A password reset token is predictable or reusable.
- An API endpoint accepts a bearer token but never validates its signature.

**Security misconfiguration**

Misconfiguration is often easier to exploit than a coding flaw.

Examples:

- Default admin credentials left unchanged.
- Directory listing enabled.
- Debug mode enabled in production.
- Verbose stack traces reveal file paths and secrets.
- CORS allows `*` with credentials in an unsafe way.
- Unnecessary HTTP methods remain enabled.

Useful checks:

```bash
curl -i -X OPTIONS https://app.example.com/
curl -i https://app.example.com/debug
curl -i https://app.example.com/.git/HEAD
```

**Vulnerable and outdated components**

This category covers third-party libraries, frameworks, and server software with known weaknesses.

What to recognize:

- Old CMS versions.
- Outdated JavaScript libraries.
- Unpatched application servers.
- Reused components with published CVEs.

Testing approach:

- Fingerprint versions from headers, error pages, and asset names.
- Compare versions against known advisories.
- Check dependency manifests when available.

**Cryptographic failures**

These issues involve weak or missing protection for data at rest or in transit.

Examples:

- HTTP instead of HTTPS for login pages.
- Weak TLS configuration.
- Hard-coded secrets in source code.
- Sensitive data stored without encryption.
- Poor key rotation or key storage.

Exam clue:

- If the data is exposed because encryption is missing or misused, think **Cryptographic Failures** rather than injection.

**Logging and monitoring failures**

If an attack succeeds but no one notices, the organization has a detection problem.

Indicators:

- No audit trail for admin actions.
- Failed logins are not recorded.
- Logs can be altered by attackers.
- Alerts do not trigger on suspicious behavior.

Why it matters:

- It increases dwell time.
- It makes incident response and forensics harder.
- It often appears as a secondary weakness after another exploit.

**Server-Side Request Forgery in practice**

SSRF is often tested through features that fetch remote content.

Example vulnerable logic:

```python
import requests
from flask import request

url = request.args.get("url")
r = requests.get(url)
return r.text
```

If the application does not restrict destinations, an attacker may use it to reach internal hosts.

Operational defenses:

- Allowlist approved domains.
- Resolve and verify IP addresses after DNS lookup.
- Block private, loopback, and link-local ranges.
- Disable redirects or revalidate each redirect target.
- Restrict outbound traffic from application servers.

**WAF, reverse proxy, and what they can and cannot do**

A **WAF** inspects HTTP traffic and blocks patterns that look malicious. A **reverse proxy** sits in front of servers and forwards requests. They are related but not the same.

Typical Nginx reverse proxy example:

```nginx
server {
    listen 443 ssl;
    server_name app.example.com;

    location / {
        proxy_pass http://app_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

What these controls can help with:

- Blocking obvious payloads.
- Hiding internal server details.
- Centralizing TLS termination.
- Rate limiting and request filtering.
- Logging and traffic normalization.

What they cannot guarantee:

- They do not fix broken authorization logic.
- They do not stop all SQL injection or SSRF variants.
- They do not replace secure coding.
- They can be bypassed by encoding, alternate routes, or business-logic abuse.
- They may miss attacks that look legitimate at the HTTP layer.

Example WAF rule concept:

```text
If request URI contains "/admin" and source IP is not in allowlist, block.
If request body contains common SQLi markers, inspect or challenge.
If request rate exceeds threshold, throttle or deny.
```

**How to identify issues during proxy testing**

When you intercept traffic, focus on differences between what the browser shows and what the server actually receives.

- Compare hidden form fields with visible UI values.
- Modify IDs, roles, and prices in requests.
- Remove headers such as `Origin`, `Referer`, or `X-Requested-With` and observe behavior.
- Replay requests with a different session.
- Change content types from `application/json` to `text/plain` or `multipart/form-data` when testing parser confusion.
- Look for inconsistent responses that suggest access control or input validation flaws.

Example Burp-style workflow:

1. Capture the request.
2. Send it to Repeater.
3. Change one parameter at a time.
4. Compare status code, length, and timing.
5. Confirm whether the server enforces the control server-side.

**Common exam identification clues**

- A user changes an object ID and sees another user’s data: **Broken Access Control / IDOR**
- A login succeeds without valid credentials due to logic flaws: **Identification and Authentication Failures**
- A server fetches attacker-supplied URLs: **SSRF**
- A payload changes SQL query behavior: **Injection**
- A stack trace reveals internal paths and versions: **Security Misconfiguration**
- A vulnerable library version is named in a banner: **Vulnerable and Outdated Components**
- Sensitive data is exposed because it is not encrypted: **Cryptographic Failures**
- An attacker uploads a script disguised as an image: **Security Misconfiguration** or **Injection**, depending on the control failure
- A workflow can be abused to skip payment or approval: **Insecure Design**
- Logs do not record suspicious admin actions: **Security Logging and Monitoring Failures**

**Practical command-line validation**

Use simple tools to confirm behavior before deeper testing.

```bash
# Inspect headers and cookies
curl -i https://app.example.com/

# Test allowed methods
curl -i -X OPTIONS https://app.example.com/api/orders

# Send a custom header
curl -i -H "X-Forwarded-For: 127.0.0.1" https://app.example.com/

# Test a parameter for injection behavior
curl -i "https://app.example.com/search?q=test%27"
```

**Defensive controls and their exam relevance**

- **Input validation**: reject unexpected characters, formats, and lengths.
- **Output encoding**: prevent browser-side interpretation of dangerous content.
- **Parameterized queries**: stop SQL injection by separating code from data.
- **Least privilege**: limit what each account and service can access.
- **MFA**: reduces password-only compromise, but does not fix authorization flaws.
- **Secure session management**: rotate IDs, expire sessions, and protect cookies.
- **Network segmentation**: limits lateral movement between tiers.
- **Logging and alerting**: improves detection and response.
- **Patch management**: reduces exposure to known vulnerabilities.

> A WAF, proxy, or MFA layer can reduce risk, but none of them replaces secure application logic or server-side authorization checks.

**Quick exam memory map**

- **Architecture**: client, proxy, web tier, app tier, database tier.
- **Identity**: authentication proves who you are; authorization proves what you can do.
- **Session**: server-side state; cookie often carries the session ID.
- **Top 10**: know the 2021 names and the likely test technique for each.
- **Testing**: intercept, modify, replay, compare, and validate impact.
