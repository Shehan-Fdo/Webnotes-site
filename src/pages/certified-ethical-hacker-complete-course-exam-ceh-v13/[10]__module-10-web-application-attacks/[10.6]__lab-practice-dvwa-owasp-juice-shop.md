## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

DVWA (Damn Vulnerable Web Application) and OWASP Juice Shop are intentionally vulnerable web applications used to practice reconnaissance, testing methodology, and vulnerability validation in a safe lab. They help you build repeatable habits for identifying common web flaws such as injection, authentication weaknesses, access control issues, and client-side trust problems.

DVWA is useful when you want a more traditional, form-driven lab with clear vulnerability categories and adjustable difficulty. Juice Shop is more modern and realistic, with a single-page application (SPA) style interface, API-driven behavior, and many challenges that reflect current web application attack surfaces.

**Why these labs matter for CEH practice**

These applications let you practice the workflow, not just the exploit. In a CEH-style lab, you should focus on:

- **Discovery** of the application surface with browser tools and network enumeration.
- **Validation** of weak controls in a controlled environment.
- **Documentation** of evidence, impact, and remediation.
- **Repeatability** so you can re-test after changing settings or applying fixes.

A good lab session should teach you how to move from “I found something interesting” to “I can prove the issue, explain the risk, and suggest a fix.”

**Typical lab topology**

A simple practice setup often includes your attacker machine, the vulnerable target, and optionally a proxy for inspection.

```text
[Attacker VM] ---> [DVWA / Juice Shop Container or VM] ---> [Optional Proxy: Burp Suite]
      |
      +--> Browser, curl, ffuf, nmap, nikto, gobuster
```

If you use Docker, keep the target isolated on a private network or local host binding. Avoid exposing vulnerable services to untrusted networks.

**Setting up DVWA**

DVWA is commonly deployed with Docker or a local LAMP stack. The exact image name and tags can vary by environment, so treat any image reference as version-dependent rather than universal. If you use a container image, verify the current repository and tag before relying on it in a lab.

A typical Docker workflow looks like this:

```bash
docker pull vulnerables/web-dvwa
docker run --rm -it -p 8080:80 vulnerables/web-dvwa
```

After startup, you usually complete a web-based setup step and configure the database connection if the image or deployment requires it. In many DVWA deployments, the application expects a database backend and an initialization step before the vulnerable modules work correctly.

Common post-deployment checks:

- Confirm the web server responds on the mapped port.
- Open the setup page and initialize the database if required.
- Log in with the default lab credentials only if your local build uses them.
- Set the security level before testing.

> The exact setup flow depends on the DVWA package or image you use. Do not assume every build is ready immediately after container start.

**Setting up OWASP Juice Shop**

Juice Shop is usually deployed as a container or Node.js application. It often starts with a more modern default experience than DVWA, but you should still verify that the app has fully initialized before testing. Some deployments may need extra time for startup, environment variables, or persistent storage.

A common Docker run command is:

```bash
docker pull bkimminich/juice-shop
docker run --rm -it -p 3000:3000 bkimminich/juice-shop
```

Useful checks after startup:

- Open the application in a browser and confirm the UI loads.
- Inspect the browser console and network tab for startup errors.
- Verify the API responds if the frontend appears partially loaded.
- Check whether any challenge or scoring features are enabled in your build.

Juice Shop uses bundled and minified assets, so you usually analyze the application through browser dev tools, API traffic, and source maps or bundle references rather than expecting a single stable `main.js` file at the site root.

**Reconnaissance workflow**

Start with passive observation before you test anything. In both labs, you want to identify routes, parameters, cookies, headers, and client-side behavior.

Useful commands:

```bash
nmap -sV -Pn 127.0.0.1
curl -i http://127.0.0.1:8080/
curl -i http://127.0.0.1:3000/
```

What to look for:

- **Headers** such as `Server`, `X-Powered-By`, `Set-Cookie`, and cache behavior.
- **Cookies** that may indicate session handling or security mode.
- **Forms** and input fields that accept user-controlled data.
- **API endpoints** exposed by the frontend.
- **Redirects** that reveal login or setup flows.

Browser developer tools are essential. Use the Network tab to inspect requests, parameters, and response codes. Use the Application or Storage tabs to review cookies, local storage, and session storage.

**DVWA testing focus**

DVWA is especially useful for practicing classic web vulnerabilities in a controlled way. The modules and exact behavior can vary by version and security level, but the lab commonly includes:

- **SQL injection**
- **Cross-site scripting (XSS)**
- **Command injection**
- **File inclusion**
- **File upload**
- **CSRF (Cross-Site Request Forgery)**
- **Brute force and authentication weaknesses**

DVWA’s security level matters. Lower settings often reduce input validation, escaping, or token checks, which makes the vulnerability easier to demonstrate. Higher settings may add filtering, encoding, or token-based protections.

Practical approach:

1. Set the security level to a known value.
2. Test one module at a time.
3. Record the request and response before changing payloads.
4. Compare behavior across security levels.

Example of a version-dependent reflected XSS test in DVWA:

```bash
curl -i "http://127.0.0.1:8080/vulnerabilities/xss_r/?name=test"
```

This kind of request is useful as a lab illustration, but the exact endpoint behavior, required cookies, and parameter handling depend on the DVWA version and security settings. In a browser, you may need an authenticated session and the correct module path before the payload is reflected.

For SQL injection practice, you often inspect how the application handles numeric or string input in a query parameter. A safe lab example is to compare normal and malformed input and observe differences in error messages, row counts, or response timing.

```bash
curl -i "http://127.0.0.1:8080/vulnerabilities/sqli/?id=1"
curl -i "http://127.0.0.1:8080/vulnerabilities/sqli/?id=1%27"
```

Use these tests to learn how the application reacts, not to assume a specific exploit works in every build.

**Juice Shop testing focus**

Juice Shop is designed around modern web application weaknesses and often requires more attention to client-side logic, API calls, and workflow abuse. It is a good lab for practicing:

- **Broken access control**
- **Authentication and session flaws**
- **Injection-style issues**
- **Sensitive data exposure**
- **Security misconfiguration**
- **Client-side trust problems**
- **API abuse and parameter tampering**

Because Juice Shop is often a single-page application, you should inspect the browser’s network traffic carefully. Many actions are driven by API calls rather than traditional page reloads.

Useful browser and CLI techniques:

```bash
curl -i http://127.0.0.1:3000/
curl -s http://127.0.0.1:3000/robots.txt
```

You can also enumerate routes and assets, but be careful with assumptions. Some paths return SPA fallback pages with HTTP 200, so status-code filtering alone may not identify valid routes reliably.

A more practical discovery approach is:

- Review the frontend bundle references in the HTML.
- Inspect API calls in the Network tab.
- Search for challenge hints in the UI.
- Use proxy history to map endpoints and parameters.

**Proxy-based workflow**

Burp Suite or a similar intercepting proxy helps you understand request structure and replay traffic safely.

A typical workflow is:

1. Configure the browser proxy to `127.0.0.1:8080` or your chosen listener.
2. Install the proxy CA certificate if you need HTTPS interception.
3. Capture a login or search request.
4. Send the request to a repeater tool.
5. Modify one parameter at a time and compare responses.

Example proxy environment variables for CLI tools:

```bash
export http_proxy=http://127.0.0.1:8080
export https_proxy=http://127.0.0.1:8080
```

This is useful when you want `curl` or other tools to follow the same inspection path as your browser.

**Common validation patterns**

When you test a finding, validate it in a way that is easy to reproduce and explain.

- **Input reflection**: confirm whether user input appears in the response.
- **Error behavior**: compare normal input with malformed input.
- **State change**: verify whether a request changes data without proper authorization.
- **Access control**: test whether a lower-privileged user can reach a restricted function.
- **Token handling**: check whether anti-CSRF or session tokens are present and enforced.
- **Encoding**: determine whether output is escaped in HTML, JavaScript, or attributes.

A simple validation note template:

- Request used
- Expected behavior
- Actual behavior
- Security impact
- Suggested remediation

**Example command-line toolkit**

These tools are commonly useful in both labs:

```bash
nmap -sV -Pn 127.0.0.1
curl -i http://127.0.0.1:8080/
curl -i http://127.0.0.1:3000/
ffuf -u http://127.0.0.1:3000/FUZZ -w /usr/share/seclists/Discovery/Web-Content/common.txt
```

Use `ffuf` carefully on Juice Shop. Because many SPA routes may return a generic application page, you should validate findings by inspecting response content, not just status codes. For DVWA, directory and parameter fuzzing can still help you discover module paths and hidden files.

**Security levels, sessions, and state**

DVWA often uses a security setting that changes how vulnerable each module behaves. Lower levels may disable protections, while higher levels add filtering or validation. Always note the level you used when documenting results.

Session handling matters in both labs:

- Log in before testing authenticated modules.
- Preserve cookies when replaying requests.
- Watch for session expiration during long test runs.
- Re-check the application after restarting containers.

If a test suddenly stops working, verify whether the session expired, the container restarted, or the security level changed.

**Reporting findings**

A strong lab report should include enough detail for another tester to reproduce the issue.

Include:

- **Target** and version or build notes
- **Endpoint** or feature tested
- **Payload** or request pattern used
- **Evidence** such as screenshots, request/response pairs, or logs
- **Impact** in plain language
- **Remediation** such as input validation, output encoding, authentication checks, or least privilege

Example remediation guidance:

- Use parameterized queries for database access.
- Encode output based on context.
- Enforce server-side authorization checks.
- Add CSRF tokens and verify them server-side.
- Validate file uploads by type, content, and storage location.
- Avoid trusting client-side state for security decisions.

**Troubleshooting**

If DVWA does not behave as expected:

- Confirm the database is initialized.
- Check that the correct security level is selected.
- Verify the login session is active.
- Review container logs for startup or database errors.

If Juice Shop does not load correctly:

- Wait for full startup before testing.
- Check browser console errors.
- Confirm the correct port mapping.
- Verify that any required environment variables or persistent storage settings are present.

Useful container checks:

```bash
docker ps
docker logs <container_id>
docker exec -it <container_id> sh
```

**What to remember for exam practice**

These labs are not just about finding vulnerabilities. They train you to think like a tester:

- Start with enumeration.
- Observe how the app behaves normally.
- Change one variable at a time.
- Confirm the effect with evidence.
- Document the risk and the fix.

DVWA is best for learning classic vulnerability mechanics. Juice Shop is best for practicing modern web testing workflows, especially API-driven and client-heavy applications. Together, they give you a strong foundation for CEH-style web application assessment practice.
