## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

These tools support a common web assessment workflow: intercept traffic, enumerate content, validate server behavior, and test for injection flaws. In CEH-style labs, you often combine them rather than use them in isolation. Burp Suite helps you observe and modify requests, Gobuster and Dirb help you discover hidden paths, Nikto helps you identify common web server issues, and SQLmap helps you validate and exploit SQL injection when you already have a likely vulnerable request.

A practical mindset matters more than memorizing tool names. You should know what each tool is best at, what it cannot do well, and how to hand findings from one tool to the next.

**Burp Suite Fundamentals**

Burp Suite is an intercepting proxy and web testing platform. It sits between your browser and the target application so you can inspect, modify, replay, and automate HTTP(S) requests. The core value is visibility: you see exactly what the browser sends and what the server returns.

Burp Suite Community Edition is useful for learning and manual testing, but it has important limits. It does not include the full Scanner, and Intruder is heavily rate-limited compared with Professional. For exam prep, know the difference between manual validation features and automated testing features.

Common Burp components include:

- **Proxy** for intercepting and forwarding traffic.
- **Target** for site map, scope, and content mapping.
- **Repeater** for manual request editing and replay.
- **Intruder** for controlled attack automation such as fuzzing and parameter testing.
- **Decoder** for encoding and decoding data.
- **Comparer** for diffing responses.
- **Extender** for adding BApp extensions, which are third-party or built-in extensions from the Burp extension ecosystem.

**Typical Burp workflow**

1. Configure your browser to use Burp as the proxy, usually `127.0.0.1:8080`.
2. Install Burp’s CA certificate in the browser if you need to inspect HTTPS traffic.
3. Add the target host to scope so you can focus on relevant traffic.
4. Intercept a request, send it to Repeater, and test how the application responds to controlled changes.
5. Use Intruder for repeated payload testing when manual replay becomes slow.
6. Save useful requests for later handoff to other tools such as SQLmap.

```text
Browser -> Burp Proxy -> Target Application
```

**Common configuration tasks**

- Set the browser proxy to `127.0.0.1:8080`.
- Import Burp’s CA certificate into the browser trust store for HTTPS interception.
- Use scope rules to reduce noise and avoid testing unrelated hosts.
- Turn interception on only when you need to capture a request; otherwise, leave it off to avoid blocking normal browsing.
- Use session handling rules or macros when the application requires a fresh token or cookie.

A simple proxy configuration example in a browser or system proxy setting looks like this:

```text
HTTP Proxy: 127.0.0.1
Port: 8080
HTTPS Proxy: 127.0.0.1
Port: 8080
```

**Repeater use cases**

Repeater is the most important Burp tool for manual validation. You use it when you want to change one parameter at a time and observe the response.

- Test whether a parameter is reflected, filtered, or sanitized.
- Compare responses when you change cookies, headers, or form fields.
- Reproduce a bug reliably before attempting automation.
- Validate whether a suspected injection point behaves consistently.

Example request editing flow:

```http
GET /product?id=10 HTTP/1.1
Host: target.local
Cookie: session=abc123
User-Agent: Mozilla/5.0
```

You can change `id=10` to `id=11`, `id=10'`, or `id=10 OR 1=1` in a lab to observe differences in status code, response length, or error messages. The goal is not to guess blindly; it is to identify a repeatable signal.

**Intruder concepts**

Intruder is used for automated, repeated request manipulation. In CEH labs, it is commonly used for fuzzing parameters, testing login behavior, and enumerating values. Community Edition is limited in speed, so it is best for small-scale testing.

Attack types to know:

- **Sniper**: one payload position at a time; useful for single-parameter fuzzing.
- **Battering ram**: the same payload is inserted into all positions.
- **Pitchfork**: multiple payload lists advance in parallel.
- **Cluster bomb**: all combinations of payload lists are tested; useful but expensive.

Useful Intruder settings:

- Define payload positions carefully to avoid noisy results.
- Use grep match and grep extract to highlight interesting responses.
- Filter by status code, length, or keywords to spot anomalies.
- Throttle requests when testing against fragile lab targets.

Example payload position concept:

```http
POST /login HTTP/1.1
Host: target.local
Content-Type: application/x-www-form-urlencoded

username=§admin§&password=§test§
```

**Scope, intercept rules, and cookies**

Scope keeps your work focused. Add only the target domain or IP range you are authorized to test. This reduces accidental traffic to third-party services and makes Burp’s site map easier to read.

- Use **Target** scope to include the intended host.
- Use intercept rules to capture only relevant requests.
- Use session handling to keep authenticated traffic valid.
- Update cookies when the application rotates session identifiers.
- Watch for CSRF tokens, which may need to be refreshed before each replay.

A practical authenticated testing pattern is:

1. Log in through the browser.
2. Capture the authenticated request in Burp.
3. Send it to Repeater.
4. Replace the session cookie or token if the application expires it.
5. Replay the request and confirm the response still reflects the authenticated state.

**Upstream proxy chaining**

Sometimes Burp must sit behind another proxy, such as a corporate proxy or a VPN gateway. In that case, configure an upstream proxy so Burp forwards traffic correctly.

```text
Burp -> Upstream Proxy -> Internet/Target
```

Use this when:

- Your lab requires outbound traffic through a specific proxy.
- You need to chain Burp with another interception or logging layer.
- You are testing from a restricted network environment.

**Burp limitations and troubleshooting**

- If HTTPS interception fails, verify the CA certificate is installed and trusted.
- If requests do not appear, confirm the browser proxy settings and that Burp is listening on the expected interface.
- If the application uses a strict login flow, check for CSRF tokens, anti-automation controls, or expired cookies.
- If responses look incomplete, inspect whether the browser or application is using redirects, chunked responses, or application-specific behavior rather than assuming a protocol problem.
- Community Edition is suitable for manual validation, but it is not a replacement for Professional when you need full automation and scanning.

**SQLmap Fundamentals**

SQLmap automates detection and exploitation of SQL injection vulnerabilities. It is most effective after you have a likely vulnerable request from Burp or from manual testing. SQLmap can test parameters, identify the backend DBMS (Database Management System), enumerate data, and in some cases perform file operations or command execution depending on privileges and DBMS support.

Use SQLmap when you already suspect a parameter is injectable. It is not a substitute for understanding the request context.

**Common input sources**

SQLmap can work from a raw request file, a URL, or a captured request from Burp.

```bash
sqlmap -u "http://target.local/item.php?id=1" --batch
```

For authenticated testing, save the request from Burp to a file and point SQLmap at it:

```bash
sqlmap -r request.txt --batch
```

This is often the best CEH workflow because it preserves cookies, headers, and POST bodies exactly as the application sent them.

**Important SQLmap options**

- `-u` targets a URL with parameters.
- `-r` reads a raw HTTP request file.
- `--cookie` supplies session cookies when needed.
- `--data` sends POST data.
- `--batch` accepts default answers for non-interactive use.
- `--level` and `--risk` increase test depth and aggressiveness.
- `--technique` limits tests to specific injection techniques.
- `--dbs` enumerates databases after confirmation.
- `--tables` and `--columns` enumerate schema objects.
- `--dump` extracts data.
- `--os-shell` attempts an operating system shell when conditions allow.
- `--file-read` and `--file-write` attempt file access on supported DBMS targets.
- `--output-dir` stores results for later review.
- `-v` increases verbosity for troubleshooting.

Example with a saved request:

```bash
sqlmap -r request.txt --batch --dbs --level=3 --risk=2 -v 2
```

**Typical SQLmap workflow**

1. Capture a request in Burp that includes the suspected parameter.
2. Save the request to a file.
3. Run SQLmap with `-r` and `--batch`.
4. Confirm the injection point and DBMS fingerprint.
5. Enumerate databases, tables, and columns only after validation.
6. Dump only the data you are authorized to test in the lab.
7. If supported and appropriate, test file read/write or OS shell capabilities.

**DBMS-specific post-exploitation concepts**

SQLmap may support different actions depending on the backend DBMS and privileges:

- **File read** can expose configuration files or web root content.
- **File write** can sometimes place a web shell or helper file if the DBMS account has sufficient permissions.
- **OS shell** may be possible when the DBMS can execute commands or write to executable locations.
- **Privilege checks** matter because low-privilege database accounts often block these actions.

Be conservative: these features are conditional, not guaranteed.

**Tamper scripts and evasion**

Tamper scripts modify payloads to bypass filters or WAFs (Web Application Firewalls). They are DBMS- and payload-specific, so you should choose them based on the target behavior rather than assuming one script works everywhere.

Examples of tamper usage:

```bash
sqlmap -r request.txt --tamper=space2comment,between --batch
```

Use tamper scripts when:

- A WAF blocks obvious payloads.
- The application normalizes or filters certain characters.
- You need to test alternate encodings or syntax variants.

Do not treat tamper scripts as universal fixes. If a payload fails, first confirm the injection point, then adjust technique, risk, and tamper selection.

**SQLmap troubleshooting**

- If SQLmap reports no injection, verify the request is valid and authenticated.
- If the application uses anti-CSRF tokens, refresh the request or capture a new one.
- If the site rate-limits requests, lower aggressiveness and retry.
- If the parameter is in JSON, use a raw request file so the body format is preserved.
- If the target uses a WAF, try a different technique or a carefully chosen tamper script.

**Nikto Fundamentals**

Nikto is a web server scanner that checks for common misconfigurations, dangerous files, outdated software, and known issues. It is broad and noisy by design, so it is best used as a quick reconnaissance and validation tool rather than a precision exploit tool.

Nikto is useful for:

- Identifying default files and directories.
- Checking for outdated server components.
- Finding risky HTTP methods or headers.
- Spotting common misconfigurations.
- Testing virtual hosts and host header behavior in some scenarios.

Basic usage:

```bash
nikto -h http://target.local
```

If you need a specific port or HTTPS target, specify it explicitly when necessary:

```bash
nikto -h target.local -p 443
```

Depending on version and target behavior, HTTPS may also be inferred from the URL or port, so do not assume `-ssl` is always required.

**Useful Nikto options and concepts**

- `-h` sets the host or URL.
- `-p` sets the port.
- `-Tuning` narrows scan categories.
- `-o` writes output to a file.
- `-Format` selects report format.
- `-useragent` changes the client identity.
- `-id` supplies HTTP authentication credentials.
- `-H` adds custom headers.
- `-vhost` tests a virtual host name.
- `-ssl` may be used in some versions or scenarios to force TLS behavior.

Example with authentication and a custom host header:

```bash
nikto -h 10.10.10.20 -p 443 -H "Host: app.internal" -id "admin:Password123" -o nikto.txt -Format txt
```

**Interpreting Nikto output**

Nikto findings usually fall into categories such as:

- **Informational**: useful context, server banners, or headers.
- **Potentially interesting**: files or paths worth manual review.
- **High-value issues**: dangerous methods, exposed admin panels, or known vulnerable components.

Treat Nikto output as leads, not proof. Many findings require manual confirmation in Burp or a browser.

**Nikto limitations and troubleshooting**

- It can generate false positives, especially on custom applications.
- It may miss issues hidden behind authentication or dynamic routing.
- It is sensitive to virtual host configuration, so test the correct hostname when needed.
- It is not a replacement for manual review of application logic.

**Gobuster Fundamentals**

Gobuster is a fast brute-force enumeration tool for discovering hidden content. It is commonly used for directories, files, DNS subdomains, and virtual hosts. It is faster and more flexible than older tools in many lab environments.

Main modes:

- **dir** for directories and files on web servers.
- **dns** for subdomain enumeration.
- **vhost** for virtual host discovery.
- **fuzz** for generic fuzzing of URL components or headers.

Directory mode example:

```bash
gobuster dir -u http://target.local -w /usr/share/wordlists/dirb/common.txt
```

**Directory mode details**

In `dir` mode, Gobuster requests candidate paths and checks responses for signs of existence.

Important options:

- `-u` target URL.
- `-w` wordlist.
- `-x` file extensions to append, such as `php,txt,bak`.
- `-r` follow redirects.
- `-k` skip TLS certificate verification.
- `-t` set thread count.
- `-s` specify status codes to show.
- `-b` specify status codes to hide.
- `-f` append a trailing slash to each request.
- `-e` show full URLs.
- `-n` do not show response length.

Example with extensions:

```bash
gobuster dir -u http://target.local -w /usr/share/wordlists/raft-small-words.txt -x php,txt,bak -t 30
```

Use extensions when you suspect backup files, scripts, or alternate content types. This is especially useful when the application exposes `index.php`, `admin.php`, or `config.txt`-style resources.

**Recursion and nested discovery**

Gobuster can recurse into discovered directories when configured to do so. Recursion is not the same as following redirects. Redirect following uses `-r`; recursion requires the recursion-related options for the version you are using.

A practical approach is:

1. Find a directory such as `/admin/`.
2. Confirm it is real by checking the response manually.
3. Re-run Gobuster against the discovered path if you want deeper enumeration.
4. Use a focused wordlist to reduce noise.

This avoids confusing redirect behavior with actual nested content discovery.

**DNS and vhost modes**

Use `dns` mode when you want to enumerate subdomains:

```bash
gobuster dns -d target.local -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt
```

Use `vhost` mode when the server routes content based on the `Host` header:

```bash
gobuster vhost -u http://target.local -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt
```

These modes are useful when a single IP hosts multiple applications or when DNS names are partially hidden.

**Status code handling and false positives**

Gobuster output is only useful if you interpret status codes correctly.

- `200` often indicates a valid resource.
- `301` and `302` may indicate redirects, which can still be useful.
- `403` may mean the path exists but access is denied.
- `404` usually means not found, but custom error pages can blur this.
- `401` may indicate authentication is required.

To reduce false positives:

- Compare response lengths.
- Check whether the server returns a custom 404 page with a consistent body size.
- Manually open suspicious paths in a browser or Burp Repeater.
- Use a known invalid path to learn the baseline response.

**Gobuster troubleshooting**

- If results are noisy, tune status code filters and response length handling.
- If TLS errors occur, use `-k` only when appropriate in a lab.
- If redirects hide the real content, test both with and without `-r`.
- If the target uses a virtual host, test the correct hostname or use `vhost` mode.

**Dirb Fundamentals**

Dirb is a classic content discovery tool that brute-forces directories and files using wordlists. It is still seen in legacy labs and older training environments, so CEH candidates should recognize its output and behavior.

Basic usage:

```bash
dirb http://target.local /usr/share/wordlists/dirb/common.txt
```

Dirb is often straightforward for quick checks, especially when you want a simple recursive-style scan with minimal setup. Its behavior depends on the target responses and the options you choose, so do not assume every scan behaves the same way.

**Dirb options and behavior**

- `-X` appends extensions such as `.php` or `.bak`.
- `-r` disables recursion in many Dirb versions.
- `-o` writes output to a file.
- `-S` suppresses status codes in output.
- `-w` enables warnings.
- `-z` sets a delay between requests.

Example with extensions:

```bash
dirb http://target.local /usr/share/wordlists/dirb/common.txt -X .php,.txt,.bak -o dirb.txt
```

Use `-X` when you want to test common alternate file types. This is especially helpful for backup files and script endpoints.

**Interpreting Dirb output**

Dirb typically reports discovered paths with status codes and response sizes. Use those clues to decide what to validate next.

- A `200` response is usually worth manual review.
- A `301` or `302` may indicate a moved resource or canonical path.
- A `403` can still be valuable because it confirms the resource exists.
- Repeated response sizes may indicate a custom error page rather than a real hit.

**Dirb vs Gobuster**

| Feature | Gobuster | Dirb |
|---|---|---|
| Speed | Usually faster | Usually slower |
| Modes | dir, dns, vhost, fuzz | Primarily web content discovery |
| Extensions | `-x` | `-X` |
| Recursion control | Version-dependent recursion options | Often recursive behavior depending on options and target |
| Output | Flexible and concise | Simple and familiar |
| Best use | Modern labs and fast enumeration | Legacy labs and straightforward brute force |

Use Gobuster when you want speed, multiple modes, and cleaner control. Use Dirb when you are in a legacy environment, need a simple brute-force check, or want to match older lab instructions.

**Practical CEH-style workflows**

**Workflow 1: Discover, validate, then test injection**

1. Run Gobuster or Dirb to find hidden paths such as `/admin/`, `/login.php`, or `/search.php`.
2. Open the discovered page in Burp and capture the request.
3. Send the request to Repeater and test parameter behavior.
4. If a parameter looks injectable, hand the saved request to SQLmap.
5. Use SQLmap to confirm the injection and enumerate only what is needed.

Example handoff:

```bash
sqlmap -r login-request.txt --batch --dbs
```

This workflow is common because enumeration tools reveal attack surface, Burp validates behavior, and SQLmap automates exploitation only after you have a likely target.

**Workflow 2: Server hardening review**

1. Run Nikto against the host to identify common issues.
2. Use Burp to inspect headers, cookies, and authentication flow.
3. Use Gobuster or Dirb to find hidden admin or backup files.
4. Validate suspicious findings manually before reporting them.

This sequence helps you separate true positives from generic scanner noise.

**Workflow 3: Authenticated application testing**

1. Log in through the browser with Burp intercept enabled.
2. Capture the authenticated request and confirm the session cookie.
3. Use Repeater to test one parameter at a time.
4. Use Intruder for small-scale fuzzing if needed.
5. Save the request and use SQLmap only if the parameter behavior suggests injection.

**Common limitations and exam traps**

- Burp Community is not the same as Burp Professional; do not assume full automation or full Intruder speed.
- SQLmap needs a valid request context; it is not magic against every URL.
- Nikto is broad and noisy; it finds leads, not final proof.
- Gobuster and Dirb can produce false positives if you do not account for custom error pages and redirects.
- Redirect following is not recursion in Gobuster.
- Extension handling differs between tools and versions, so use the correct option syntax.
- Tool order depends on the scenario; there is no universal fixed sequence.

**Quick comparison**

| Tool | Primary purpose | Best for | Key limitation |
|---|---|---|---|
| Burp Suite | Interception and manual web testing | Request inspection, replay, and validation | Community Edition limits automation |
| SQLmap | SQL injection automation | Confirming and exploiting injectable parameters | Needs a likely vulnerable request |
| Nikto | Web server checks | Fast reconnaissance and common misconfigurations | Noisy and prone to false positives |
| Gobuster | Fast brute-force enumeration | Directories, DNS, vhosts, and fuzzing | Requires good wordlists and tuning |
| Dirb | Classic content discovery | Legacy labs and simple brute force | Less flexible than Gobuster |

**Study priorities**

- Know how to capture and replay requests in Burp.
- Know when to use Repeater versus Intruder.
- Know how to save a raw request for SQLmap.
- Know the difference between directory brute force, DNS enumeration, and virtual host testing.
- Know how to interpret status codes, redirects, and custom error pages.
- Know that scanner output must be validated manually.

**Practical command set**

```bash
# Burp proxy target in browser
127.0.0.1:8080

# Nikto quick scan
nikto -h http://target.local

# Gobuster directory scan with extensions
gobuster dir -u http://target.local -w /usr/share/seclists/Discovery/Web-Content/common.txt -x php,txt,bak

# Gobuster DNS scan
gobuster dns -d target.local -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt

# Dirb scan with extensions
dirb http://target.local /usr/share/wordlists/dirb/common.txt -X .php,.txt,.bak

# SQLmap from a saved Burp request
sqlmap -r request.txt --batch --dbs
```

**Final exam mindset**

Use Burp to understand the application, use Nikto and Gobuster or Dirb to expand the attack surface, and use SQLmap only when the request behavior supports injection testing. The strongest answers in CEH scenarios explain why a tool is appropriate, what evidence it produces, and how you would validate the result before reporting it.
