## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

File inclusion, command injection, and directory traversal are closely related web application flaws that often appear together during assessment. They all involve untrusted input influencing how the server resolves files, paths, or operating-system commands. In CEH-style scenarios, you should be able to recognize the difference between a file read, a file include, and actual command execution, then choose the right validation and defense strategy.

These issues matter because they can expose source code, configuration files, secrets, logs, and internal system details. In the worst case, they can lead to remote code execution (RCE), which means the attacker can run commands on the server.

**Core concepts**

- **Directory traversal** manipulates path resolution to escape an intended directory and access files elsewhere on the filesystem.
- **File inclusion** makes the application load a file as code or content, often through parameters that control templates, language files, or plugin paths.
- **Command injection** occurs when user input reaches a shell or command interpreter without safe escaping or allowlisting.
- **Local File Inclusion (LFI)** reads local files from the server.
- **Remote File Inclusion (RFI)** attempts to include content from a remote source, though modern PHP settings and frameworks often reduce this risk.

**How these flaws differ**

| Issue | Primary effect | Typical impact | Common signal |
|---|---|---|---|
| Directory traversal | File path escape | Read arbitrary files | `../` patterns, path normalization issues |
| File inclusion | Load attacker-controlled or chosen file | Source disclosure, code execution in some cases | Template or language file parameters |
| Command injection | Shell execution | Full system compromise | Output from system commands, timing, or side effects |

**Why CEH emphasizes them**

These flaws are classic exam topics because they test your understanding of input handling, server-side normalization, and defense-in-depth. They also illustrate how small parsing mistakes can become major security failures when the application trusts path fragments or command arguments.

**Directory traversal fundamentals**

Traversal attacks work when an application concatenates user input into a file path without properly constraining the final resolved location. The attacker uses parent directory references to move outside the intended directory and reach sensitive files.

A vulnerable pattern often looks like this:

```php
<?php
$file = $_GET['page'];
include("pages/" . $file);
?>
```

If the application does not validate `page`, an attacker may try to escape `pages/` and access files outside that directory.

**Common traversal targets**

- Configuration files such as `/etc/passwd`, `/etc/shadow`, `web.config`, `.env`, or application config files.
- Source code and templates that reveal logic, credentials, or internal endpoints.
- Log files that may contain tokens, session identifiers, or injected content.
- Backup files, archives, and editor artifacts such as `.bak`, `.old`, `~`, or `.swp`.

> On Windows, traversal often targets paths like `C:\Windows\win.ini`, `C:\boot.ini` on legacy systems, or application-specific configuration files. Path separators and normalization behavior differ from Unix-like systems.

**Traversal bypass techniques**

Attackers often bypass naive filters that only block literal `../`. You should know the broader classes of bypasses because exam questions frequently describe them indirectly.

- **URL encoding**: `..%2f`, `%2e%2e/`, or mixed-encoding variants.
- **Double decoding**: the application or proxy decodes input more than once, turning a harmless-looking string into traversal after the second pass.
- **Path segment tricks**: repeated separators, `./`, `..//`, `.../`, or redundant segments that normalize to a parent directory escape.
- **Overlong paths**: very long sequences of traversal segments can defeat simplistic regex filters or truncation logic.
- **Alternate encodings**: UTF-8, Unicode normalization edge cases, or platform-specific separator handling.
- **Null byte injection**: in older stacks, `%00` could terminate a string early and bypass appended suffixes such as `.php` or `.txt`. Modern runtimes usually block this, but it remains a classic CEH concept.
- **Absolute path confusion**: supplying a path that begins with `/` or a drive letter may override the intended base path if validation is weak.
- **Separator variation**: using backslashes on Windows or mixed separators where the application normalizes inconsistently.

**Normalization and decoding order**

The order of operations matters. A filter that checks input before decoding can miss dangerous content that appears only after URL decoding. Likewise, a reverse proxy, web server, framework, and application may each normalize paths differently.

Key edge cases:

- **Single decode vs. double decode** can change a benign string into a traversal payload.
- **Server-side normalization** may collapse `.` and `..` segments after the application’s validation step.
- **Proxy rewriting** can alter the path before it reaches the app, so the tested string is not always the string the application sees.
- **Filesystem resolution** may differ from URL path parsing, especially on Windows and in containerized environments.

**Testing methodology for traversal and inclusion**

A systematic workflow helps you distinguish a real flaw from a harmless parameter.

1. Identify parameters that look path-related, such as `file`, `page`, `template`, `lang`, `doc`, `path`, `include`, or `view`.
2. Replace the value with a harmless known file name and observe whether the response changes.
3. Try a traversal sequence and compare the response body, status code, length, and error messages.
4. Increase or decrease traversal depth until you confirm the application’s base directory.
5. Test encoding variants to see whether filtering occurs before or after decoding.
6. Compare behavior for missing files, valid files, and files that should not be web-accessible.
7. Determine whether the application is reading a file, including it, or simply reflecting the supplied string.

Useful indicators:

- A **file read** often returns raw file contents or a download response.
- A **file include** may execute server-side code, alter page structure, or trigger parser errors.
- A **reflection-only** issue echoes input without accessing the filesystem.

**Practical enumeration examples**

Use low-risk probes first and observe differences carefully.

```bash
curl -i "https://target.example/view?file=about.html"
curl -i "https://target.example/view?file=../../../../etc/passwd"
curl -i "https://target.example/view?file=..%2f..%2f..%2fetc%2fpasswd"
curl -i "https://target.example/view?file=....//....//etc/passwd"
```

If the application returns different content lengths, error messages, or response times, you may have confirmed path handling behavior.

**File inclusion fundamentals**

File inclusion flaws occur when the application loads a file based on user input. This is common in template systems, language packs, plugin loaders, and legacy PHP applications.

Typical patterns include:

- `include`, `require`, `include_once`, and `require_once` in PHP.
- Template engines that accept a template name or partial path.
- Dynamic import or module loading in other languages when input is not constrained.
- Custom “download,” “preview,” or “render” endpoints that map a parameter to a file.

A vulnerable PHP example:

```php
<?php
$page = $_GET['page'];
include("pages/" . $page . ".php");
?>
```

If validation is weak, an attacker may attempt traversal to reach unintended files. In older PHP environments, null byte injection could sometimes bypass the appended `.php` suffix, though modern versions generally prevent this.

**LFI versus RFI**

- **LFI** uses files already present on the server.
- **RFI** attempts to include remote content from a URL or remote resource.
- RFI is less common in modern deployments because PHP settings such as `allow_url_include` are usually disabled, and many frameworks disallow remote includes by design.

Even when RFI is not possible, LFI can still be dangerous because it may expose source code or enable code execution through log poisoning, session file inclusion, or upload abuse.

**Wrapper-based inclusion variants**

Stream wrappers are important in PHP-oriented CEH material because they expand the attack surface beyond plain file paths.

Common wrappers to know:

- `php://filter` for reading source code through encoding or transformation filters.
- `php://input` for including raw request body content in some contexts.
- `data://` for embedding inline data as a stream.
- `zip://` for accessing files inside archives.
- `phar://` for interacting with PHP archives, sometimes relevant in deserialization-adjacent scenarios.
- `expect://` on some systems for command execution through an external process wrapper, though availability is environment-specific.

Examples:

```text
php://filter/convert.base64-encode/resource=index.php
php://input
data://text/plain,<?php echo "test"; ?>
zip://archive.zip#payload.php
```

Use these concepts carefully in notes and exams: wrapper availability depends on configuration, PHP version, and server hardening.

**Why `php://filter` matters**

`php://filter` is often used to read source code when direct execution would otherwise hide it. By applying a transformation such as Base64 encoding, you can retrieve the file contents in a safer-to-display form.

Example:

```bash
curl "https://target.example/view?file=php://filter/convert.base64-encode/resource=index.php"
```

If the response contains Base64 text, you can decode it locally:

```bash
echo 'PD9waHAgZWNobyAiSGVsbG8iOyA/Pg==' | base64 -d
```

**Command injection fundamentals**

Command injection happens when application input reaches a shell command without proper escaping or allowlisting. The danger is not the command itself, but the shell interpretation of metacharacters.

A vulnerable pattern might look like this:

```php
<?php
$host = $_GET['host'];
system("ping -c 1 " . $host);
?>
```

If the input is not sanitized, an attacker may append shell metacharacters to execute additional commands.

**Common shell metacharacters**

- `;` to separate commands.
- `&&` to chain commands conditionally.
- `||` to run a fallback command.
- `|` to pipe output into another command.
- `` `command` `` and `$(command)` for command substitution.
- `&` to background a process in some shells.
- Newlines can also act as separators in some contexts.

**Command injection testing workflow**

1. Identify parameters that influence system utilities, diagnostics, file operations, or network checks.
2. Start with benign separators such as `; id` or `&& whoami` in a controlled lab.
3. Observe whether the response includes command output, timing changes, or error messages.
4. Try blind techniques if output is suppressed, such as time delays or out-of-band callbacks in authorized environments.
5. Determine whether the application uses a shell directly or passes arguments safely to an API.

**Blind command injection indicators**

- Delayed responses after a sleep command.
- DNS or HTTP callbacks to a controlled listener.
- Changes in server-side logs or side effects without visible output.
- Different behavior when using separators versus plain text.

**Safer command execution patterns**

Prefer APIs that do not invoke a shell. For example, in Python, use `subprocess.run()` with a list of arguments and `shell=False`.

```python
import subprocess

host = "example.com"
subprocess.run(["ping", "-c", "1", host], check=True)
```

This reduces shell interpretation risk, though you still need input validation because the command arguments themselves may be dangerous.

**Practical defense for command injection**

- Use allowlists for expected values such as hostnames, filenames, or numeric IDs.
- Avoid shell invocation when a direct library call exists.
- Escape arguments only as a secondary control, not the primary defense.
- Run services with least privilege so command execution has limited impact.
- Log and alert on suspicious metacharacters and abnormal command patterns.

**Enumeration and validation workflow**

A disciplined workflow helps you separate traversal, inclusion, and command injection during assessment.

1. Map the application surface and identify parameters that affect files, templates, or system utilities.
2. Test with harmless values to establish baseline behavior.
3. Introduce one change at a time, such as a traversal segment, wrapper prefix, or shell separator.
4. Compare response content, headers, timing, and error messages.
5. Confirm whether the issue is read-only, include-based, or execution-capable.
6. Document the exact normalization and decoding behavior you observed.

**How to tell the flaw type**

- If the response shows file contents, you likely have **file disclosure** or LFI.
- If the response executes code or changes page logic, you may have **file inclusion** with code execution.
- If the response runs system commands or shows command output, you have **command injection**.
- If the response only changes the path or error message, you may have a **path handling** weakness without direct exploitation.

**Common bypass and evasion patterns**

Attackers often combine multiple weak points to defeat filters.

- **Overlong paths** can bypass simplistic length checks or regexes.
- **Segment normalization tricks** can defeat filters that only block exact `../`.
- **Mixed encodings** can bypass filters that decode only once.
- **Suffix bypasses** can defeat appended extensions when null byte handling is weak in legacy systems.
- **Wrapper prefixes** can bypass assumptions that only local filenames are accepted.
- **Case variation** can matter on case-insensitive filesystems or poorly written filters.
- **Whitespace and control characters** may be trimmed inconsistently by proxies, frameworks, or shell parsers.

**Real-world exploitation chains**

These flaws often become more dangerous when chained with other weaknesses.

- LFI plus log poisoning can lead to code execution if attacker-controlled content is written into a log and later included.
- LFI plus file upload can lead to execution if an uploaded file is included as code.
- Traversal plus backup exposure can reveal credentials that enable broader compromise.
- Command injection plus weak privilege separation can expose internal services, secrets, or persistence opportunities.

**Defensive controls for file inclusion and traversal**

Strong defense starts with design, not filtering alone.

- Use **allowlists** for file names, templates, and languages.
- Map user choices to internal identifiers instead of raw paths.
- Resolve the final path and verify it stays inside an approved base directory.
- Reject absolute paths, traversal segments, and unexpected wrappers.
- Store sensitive files outside the web root.
- Disable remote includes and unnecessary stream wrappers where possible.
- Keep framework and runtime versions current to reduce legacy bypass risk.

**Example: safe path resolution in Python**

```python
from pathlib import Path

BASE_DIR = Path("/var/www/app/content").resolve()

def safe_read(user_file: str) -> str:
    candidate = (BASE_DIR / user_file).resolve()
    if BASE_DIR not in candidate.parents and candidate != BASE_DIR:
        raise ValueError("Invalid path")
    return candidate.read_text(encoding="utf-8")
```

This pattern resolves the final path and checks that it remains under the approved directory.

**Example: safe file selection in PHP**

```php
<?php
$allowed = [
    'home' => 'home.php',
    'about' => 'about.php',
    'help' => 'help.php'
];

$key = $_GET['page'] ?? 'home';
if (!array_key_exists($key, $allowed)) {
    http_response_code(400);
    exit('Invalid page');
}

include __DIR__ . '/pages/' . $allowed[$key];
?>
```

This approach uses an allowlist instead of trusting user-supplied paths.

**Apache hardening examples**

Apache should not expose sensitive directories or allow arbitrary script execution in upload or data directories.

```apache
# Restrict access to sensitive directories
<Directory "/var/www/html/private">
    Require all denied
</Directory>

# Disable directory listing
Options -Indexes

# Limit script execution in uploads
<Directory "/var/www/html/uploads">
    php_admin_flag engine off
    Options -ExecCGI
</Directory>
```

Additional Apache guidance:

- Keep application data outside the document root when possible.
- Use `Require all denied` for directories that should never be web-accessible.
- Avoid permissive `AllowOverride` settings unless needed.
- Review rewrite rules so they do not accidentally expose internal paths.

**Nginx hardening examples**

Nginx should block direct access to sensitive files and prevent accidental execution in writable directories.

```nginx
server {
    location ~ /\.(?!well-known) {
        deny all;
    }

    location ~* \.(env|ini|log|bak|old|swp)$ {
        deny all;
    }

    location /uploads/ {
        autoindex off;
        try_files $uri =404;
    }

    location ~ \.php$ {
        include fastcgi_params;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
        fastcgi_pass unix:/run/php/php-fpm.sock;
    }
}
```

Additional Nginx guidance:

- Use `try_files` to constrain file resolution.
- Deny access to dotfiles and backup artifacts.
- Keep upload directories non-executable.
- Ensure `SCRIPT_FILENAME` is set correctly so path confusion does not expose unintended files.

**IIS hardening examples**

IIS environments often rely on request filtering and handler restrictions.

```xml
<configuration>
  <system.webServer>
    <security>
      <requestFiltering>
        <denyUrlSequences>
          <add sequence="../" />
          <add sequence="..\" />
        </denyUrlSequences>
        <hiddenSegments>
          <add segment=".git" />
          <add segment="App_Data" />
        </hiddenSegments>
      </requestFiltering>
    </security>
    <handlers>
      <remove name="PHP_via_FastCGI" />
    </handlers>
  </system.webServer>
</configuration>
```

Additional IIS guidance:

- Use request filtering to block dangerous sequences and hidden segments.
- Restrict script mappings to only the directories that need them.
- Store sensitive application data outside public web roots.
- Review URL Rewrite rules for unintended path normalization effects.

**Logging and detection**

Detection is easier when you log both the raw request and the normalized path or command target.

- Log suspicious traversal strings such as `../`, `%2e%2e`, and repeated separators.
- Alert on wrapper prefixes like `php://`, `data://`, `zip://`, and `phar://`.
- Watch for shell metacharacters in parameters that should be simple identifiers.
- Correlate repeated 404s, 500s, and unusual response lengths with the same client.

**Exam-oriented takeaways**

- Traversal is about escaping a directory boundary.
- Inclusion is about controlling what file gets loaded.
- Command injection is about reaching a shell or command interpreter.
- Null byte injection is a classic legacy bypass concept, even if modern systems block it.
- Wrapper abuse is a major clue in PHP-based inclusion questions.
- Decoding order and normalization are often the difference between blocked input and exploitable input.
- Strong allowlists and path resolution checks are more reliable than blacklist filters.

**Quick reference**

- Use `../` and encoding variants to think about traversal.
- Use `php://filter`, `php://input`, `data://`, `zip://`, and `phar://` to think about inclusion wrappers.
- Use `;`, `&&`, `|`, `$(...)`, and backticks to think about command injection.
- Use allowlists, canonicalization, least privilege, and non-web-root storage to defend.
