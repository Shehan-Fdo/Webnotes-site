## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**SQL injection overview**

SQL injection (SQLi) happens when an application builds database queries from untrusted input without proper parameterization or validation. The core issue is not the presence or absence of quotes; the issue is that attacker-controlled data changes the structure or logic of the SQL statement.

SQLi often appears in:
- Login forms
- Search fields
- URL parameters
- JSON APIs
- Cookies, headers, and hidden form fields
- Second-order flows where stored input is reused later in a query

The impact can include:
- Authentication bypass
- Data disclosure
- Data modification or deletion
- Privilege escalation
- Remote command execution in some DBMS-specific scenarios
- Lateral movement through exposed credentials or linked systems

**How SQL injection works**

A vulnerable application may concatenate input into a query like this:

```sql
SELECT id, username, role
FROM users
WHERE username = 'alice' AND password = 'secret';
```

If user input is inserted directly, an attacker can alter the query logic. For example, a login form that builds SQL unsafely may be bypassed with a payload that changes the `WHERE` clause to always evaluate true.

A safer design uses parameterized queries so the database treats input as data, not executable SQL.

```python
# Safe pattern: parameterized query
cursor.execute(
    "SELECT id, username, role FROM users WHERE username = %s AND password_hash = %s",
    (username, password_hash)
)
```

> Do not compare plaintext passwords in SQL. Hash passwords with a strong algorithm and verify them in application logic or with a secure authentication flow.

**Common SQLi categories**

- **In-band SQLi**
  - The attacker uses the same channel to inject and retrieve results.
  - Common forms include error-based and UNION-based injection.
- **Inferential SQLi**
  - The application does not return query results directly.
  - The attacker infers data from behavior changes, such as true/false responses or timing differences.
- **Out-of-band SQLi**
  - The database sends data through a different channel, such as DNS or HTTP callbacks.
  - This depends on DBMS features, network reachability, and privileges.

**Authentication bypass patterns**

Authentication bypass usually targets queries that check credentials or session state. A classic pattern is forcing a condition to evaluate true, but the exact payload depends on the DBMS, query structure, and input context.

Examples of unsafe patterns:
- String concatenation in login queries
- Dynamic `ORDER BY`, `LIMIT`, or column name insertion
- Direct use of user input in stored procedures or dynamic SQL
- Trusting client-side values for role or tenant selection

Typical bypass goals:
- Skip password checks
- Force a known user row to be returned
- Manipulate session creation logic
- Abuse weak account recovery or password reset queries

**Payload variation by context**

SQLi payloads are not universal. You choose them based on:
- Quote context: inside string, numeric context, or identifier context
- Query type: `SELECT`, `INSERT`, `UPDATE`, `DELETE`
- DBMS syntax: MySQL, PostgreSQL, SQL Server, Oracle, SQLite
- Application behavior: verbose errors, blind responses, or no output
- WAF (Web Application Firewall) filtering and normalization

Examples of common contexts:
- String context: `' OR '1'='1`
- Numeric context: `1 OR 1=1`
- Comment termination: `-- `, `/* ... */`, or DBMS-specific comment forms
- Identifier injection: dynamic table or column names require different techniques than value injection

> Comment syntax is DBMS-specific. `--` commonly requires a trailing space or control character in many systems, `#` is MySQL-specific, and block comments may be filtered or normalized differently.

**UNION-based SQL injection**

`UNION` combines the results of two `SELECT` statements. For exploitation, the injected query must match the original query’s column count and compatible data types.

Workflow:
1. Determine the number of columns.
2. Identify which columns are reflected in the response.
3. Replace selected columns with data-bearing expressions.
4. Extract database metadata or application data.

Example discovery steps:

```sql
' ORDER BY 1--
' ORDER BY 2--
' ORDER BY 3--
```

If `ORDER BY 3` fails but `ORDER BY 2` succeeds, the original query likely returns two columns.

A generic UNION test may look like:

```sql
' UNION SELECT NULL, NULL--
```

Once the column count is known, you can place visible markers in reflected columns:

```sql
' UNION SELECT 'test', NULL--
```

If the application reflects the first column, you can then attempt metadata extraction. The exact payload syntax varies by DBMS.

**DBMS-specific UNION examples**

These examples are illustrative and not interchangeable across platforms.

- **MySQL**
  - Metadata functions often include `database()`, `user()`, and `version()`.
  - Example:
    ```sql
    ' UNION SELECT database(), user(), version()--
    ```
- **PostgreSQL**
  - Common functions include `current_database()`, `current_user`, and `version()`.
  - Example:
    ```sql
    ' UNION SELECT current_database(), current_user, version()--
    ```
- **SQL Server**
  - Common functions include `db_name()`, `system_user`, and `@@version`.
  - Example:
    ```sql
    ' UNION SELECT db_name(), system_user, @@version--
    ```
- **Oracle**
  - Metadata access often uses `v$version`, `dual`, and session views, but privileges matter.
  - Example:
    ```sql
    ' UNION SELECT banner, NULL FROM v$version--
    ```
  - Access to `v$version` is not guaranteed; it depends on permissions and query context.
- **SQLite**
  - Common metadata is more limited; `sqlite_version()` is often useful.
  - Example:
    ```sql
    ' UNION SELECT sqlite_version(), NULL--
    ```

**Error-based SQL injection**

Error-based SQLi forces the database to reveal information through error messages. This is useful when the application displays database errors or when verbose logging leaks details.

Examples of useful error conditions:
- Type conversion failures
- Invalid function arguments
- Division by zero
- XML or JSON parsing errors in DBMS-specific functions
- Duplicate key violations that expose concatenated data

Error messages can reveal:
- Table names
- Column names
- Query structure
- DBMS version
- Internal file paths
- Stack traces or driver details

**Blind SQL injection**

Blind SQLi is used when the application does not show query results or errors. You infer data by observing differences in:
- Page content
- HTTP status codes
- Redirects
- Response length
- Response time

Two common forms are:
- **Boolean-based blind SQLi**
  - You send a condition that evaluates true or false and compare responses.
- **Time-based blind SQLi**
  - You trigger a delay when a condition is true and measure the response time.

Boolean-based example:

```sql
' AND 1=1--
' AND 1=2--
```

If the responses differ, the parameter may be injectable.

Time-based examples vary by DBMS:

- **MySQL**
  ```sql
  ' AND IF(1=1, SLEEP(5), 0)--
  ```
- **PostgreSQL**
  ```sql
  ' AND CASE WHEN 1=1 THEN pg_sleep(5) ELSE pg_sleep(0) END--
  ```
- **SQL Server**
  ```sql
  '; IF 1=1 WAITFOR DELAY '0:0:5'--
  ```
- **Oracle**
  ```sql
  ' AND CASE WHEN 1=1 THEN DBMS_LOCK.SLEEP(5) ELSE 0 END--
  ```

> Time-based payloads may fail if the application blocks stacked statements, the DBMS function is unavailable, or the WAF strips the delay function.

**Stacked queries and multiple statements**

Stacked queries let an attacker append a second SQL statement after the original one. This is powerful but often restricted.

Examples:
- MySQL may allow multiple statements depending on the driver and connection settings.
- SQL Server often supports statement stacking in many contexts.
- PostgreSQL and Oracle usually require specific conditions or different exploitation paths.
- SQLite behavior depends on the API and driver configuration.

Why stacked queries matter:
- They can enable `INSERT`, `UPDATE`, or `DELETE` operations after a `SELECT`
- They may support privilege changes or configuration abuse
- They can be used for file access or command execution in DBMS-specific cases

Why they fail:
- The application driver disables multiple statements
- The database account lacks privileges
- The input is sanitized or parameterized
- The query context does not permit statement termination

**Second-order SQL injection**

Second-order SQLi occurs when malicious input is stored safely at first, then later reused in a vulnerable query. This makes detection harder because the injection point and exploitation point are separated in time or workflow.

Common examples:
- A profile field is stored in the database and later used in an admin report query
- A username is saved during registration and later concatenated into a maintenance script
- A support ticket field is reused in a dynamic search or export function

Detection workflow:
1. Submit a harmless marker string, such as `test'payload`.
2. Find where the application stores that value.
3. Trigger the feature that reuses the stored value in a query.
4. Observe whether the later query breaks, changes behavior, or leaks data.
5. Trace logs, exports, admin views, and scheduled jobs for the reuse path.

Second-order SQLi often appears in:
- Reporting dashboards
- Batch jobs
- Admin tools
- Search indexing
- Data export routines

**Database fingerprinting**

Fingerprinting helps you choose payloads that match the target DBMS. You can infer the platform from error messages, function behavior, metadata tables, and timing functions.

Common indicators:
- **MySQL**
  - `version()`, `database()`, `@@version`
  - Error messages may mention `MySQL` or `MariaDB`
- **PostgreSQL**
  - `version()`, `current_database()`, `current_user`
  - Errors may mention `PG::`, `psql`, or schema-related details
- **SQL Server**
  - `@@version`, `db_name()`, `system_user`
  - Errors may mention `Microsoft SQL Server`
- **Oracle**
  - `v$version`, `dual`, `user`, `sys_context`
  - Errors may mention `ORA-` codes
- **SQLite**
  - `sqlite_version()`
  - Errors may be sparse, and the engine is often embedded

Useful fingerprinting checks:
- Response to `ORDER BY` column counts
- Behavior of string concatenation operators
- Comment syntax acceptance
- Error code format
- Time-delay function support
- Case sensitivity and type coercion behavior

**Practical blind extraction workflow**

A blind extraction lab usually follows a repeatable process.

1. Confirm injection with a boolean test.
   ```sql
   ' AND 1=1--
   ' AND 1=2--
   ```
2. Determine the number of columns if you suspect `UNION` is possible.
   ```sql
   ' ORDER BY 1--
   ' ORDER BY 2--
   ' ORDER BY 3--
   ```
3. Identify the DBMS using version or error behavior.
4. Choose a boolean or time-based payload that matches the DBMS.
5. Extract one character at a time using substring logic.

Example concept for character extraction:
- Ask whether the first character of the database name is `a`
- If true, the page changes or the delay occurs
- Repeat across the alphabet and digits

A generic pattern looks like this:

```sql
' AND SUBSTRING(database(),1,1)='a'--
```

For SQL Server, the equivalent function may be `SUBSTRING()` with `db_name()`. For Oracle, you may use `SUBSTR()` and `user` or a metadata query. For PostgreSQL, `SUBSTRING()` also works, but the surrounding syntax differs.

**Out-of-band SQL injection**

Out-of-band SQLi is useful when the application does not return useful errors or timing differences are unreliable. The database initiates a network request to an attacker-controlled host.

Typical channels:
- DNS lookup
- HTTP request
- SMB or UNC path resolution in some environments

Requirements:
- The DBMS must support the feature
- The database host must have outbound network access
- The account must have sufficient privileges
- Network monitoring must not block the callback

Practical verification:
- Set up a listener or controlled domain
- Trigger a payload that causes the database to resolve or contact that domain
- Confirm the callback in DNS logs, web logs, or packet capture

**Real-world CLI tools**

`sqlmap` is a common automated testing tool for SQLi discovery and exploitation. It can identify injection points, fingerprint DBMSs, enumerate databases, and sometimes dump data.

Basic usage:

```bash
sqlmap -u "https://target.example/item?id=1" --batch
```

Useful options:
- `--dbs` to enumerate databases
- `--tables -D <db>` to list tables
- `--columns -D <db> -T <table>` to list columns
- `--dump -D <db> -T <table>` to extract rows
- `--risk` and `--level` to broaden test coverage
- `--technique` to constrain test types when needed
- `--tamper` to apply payload transformations against filters or WAFs

Example:

```bash
sqlmap -u "https://target.example/item?id=1" --batch --dbs
sqlmap -u "https://target.example/item?id=1" --batch -D appdb --tables
sqlmap -u "https://target.example/item?id=1" --batch -D appdb -T users --columns
sqlmap -u "https://target.example/item?id=1" --batch -D appdb -T users --dump
```

> `sqlmap` technique flags are version-dependent. Use the tool’s help output for the exact accepted values in your installed version rather than assuming a fixed canonical set.

**WAFs and defensive limitations**

A WAF can reduce exposure, but it is not a complete fix. Attackers may bypass naive filters with encoding, alternate syntax, comments, case changes, or DBMS-specific quirks.

Common WAF weaknesses:
- Pattern matching that misses encoded payloads
- Blocking only known signatures
- Failing to normalize input before inspection
- Not understanding application context
- Allowing dangerous dynamic SQL behind authenticated endpoints

Defensive controls that matter more:
- Parameterized queries
- Stored procedures that avoid dynamic SQL
- Input validation with allowlists
- Output encoding where relevant
- Least-privilege database accounts
- Segmentation between app and database tiers
- Monitoring and alerting on anomalous query patterns

**Least privilege and database hardening**

A secure database account should have only the permissions required by the application.

Examples:
- Read-only accounts for reporting services
- Separate accounts for read and write operations
- No `DROP`, `ALTER`, or administrative privileges for web apps
- No direct access to system catalogs unless required
- Restricted access to file import/export features

Hardening examples:
- Disable multiple statements if the driver supports it
- Remove unused stored procedures and dangerous functions
- Restrict outbound network access from the database server
- Rotate credentials and avoid hardcoded secrets
- Log failed queries and unusual query shapes

**Database activity monitoring**

Database activity monitoring helps detect exploitation attempts and post-exploitation behavior.

Monitor for:
- Repeated syntax errors
- Unusual `UNION` usage
- Excessive `ORDER BY` probing
- Time-delay function calls
- Metadata enumeration
- Large exports or dumps
- Access to system tables and version views
- Queries from unexpected application accounts

Useful telemetry sources:
- Database audit logs
- Web server logs
- Application logs
- WAF logs
- Network flow logs
- SIEM correlation rules

**Practical defensive examples**

Parameterized queries in Python:

```python
cursor.execute(
    "SELECT id, email, role FROM users WHERE email = %s",
    (email,)
)
```

Parameterized queries in Node.js with PostgreSQL:

```javascript
const text = 'SELECT id, email, role FROM users WHERE email = $1';
const values = [email];
const result = await client.query(text, values);
```

Parameterized queries in Java with JDBC:

```java
PreparedStatement ps = conn.prepareStatement(
    "SELECT id, email, role FROM users WHERE email = ?"
);
ps.setString(1, email);
ResultSet rs = ps.executeQuery();
```

These patterns prevent SQL structure from being altered by user input. They do not replace authorization checks, password hashing, or secure session management.

**Exam-focused takeaways**

- SQLi is a query-structure flaw, not just a quoting mistake.
- Payloads must match the input context and DBMS syntax.
- `UNION`, boolean-based, time-based, error-based, stacked, second-order, and out-of-band techniques solve different testing problems.
- Fingerprinting the DBMS improves payload selection and reduces false starts.
- Stacked queries are powerful but often blocked by drivers, settings, or privileges.
- Second-order SQLi requires tracing stored input into later queries.
- WAFs help, but parameterization and least privilege are the real fixes.
- Database monitoring can reveal probing, enumeration, and extraction attempts.

**Quick reference: DBMS quirks**

| DBMS | Common version/metadata functions | Delay function | Comment notes |
|---|---|---|---|
| MySQL | `version()`, `database()`, `user()` | `SLEEP()` | `#` is MySQL-specific; `-- ` often needs trailing space |
| PostgreSQL | `version()`, `current_database()`, `current_user` | `pg_sleep()` | Uses `--` and `/* */`; syntax is stricter in some contexts |
| SQL Server | `@@version`, `db_name()`, `system_user` | `WAITFOR DELAY` | Statement stacking is often useful |
| Oracle | `v$version`, `user`, `sys_context` | `DBMS_LOCK.SLEEP()` | Access to catalog views depends on privileges |
| SQLite | `sqlite_version()` | Limited native delay support | Often embedded; fewer metadata views |

**Lab mindset**

When you test a target, move from low-risk to high-confidence checks:
- Confirm the parameter is injectable
- Identify the DBMS
- Determine whether results are visible, inferential, or hidden
- Choose the least noisy payload that proves impact
- Document the exact request, response, and timing evidence

This approach helps you avoid false positives and makes your findings easier to reproduce and defend.
