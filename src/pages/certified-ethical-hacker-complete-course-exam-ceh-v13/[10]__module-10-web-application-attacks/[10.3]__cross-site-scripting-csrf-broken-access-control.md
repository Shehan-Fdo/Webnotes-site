## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

Cross-Site Scripting (XSS), Cross-Site Request Forgery (CSRF), and broken access control are core web application security topics because they exploit trust boundaries in different ways. XSS abuses the browser’s trust in a site, CSRF abuses the site’s trust in the user’s browser, and broken access control abuses the application’s failure to enforce authorization.

These issues often appear together in real assessments. For example, an XSS flaw can steal a session token or trigger privileged actions, while weak access control can expose APIs even when the front end looks secure. For exam prep, focus on how each flaw works, how to test it, and which defenses actually stop it.

**Cross-Site Scripting (XSS)**

XSS occurs when an application includes untrusted input in a page without proper output encoding or sanitization, allowing attacker-controlled script to run in a victim’s browser. The impact ranges from session theft and credential capture to account takeover, CSRF token theft, and malicious actions performed in the user’s context.

XSS is usually grouped into three main types:

- **Stored XSS**: The payload is saved on the server, such as in comments, profiles, tickets, or chat messages, and executes when another user views the content.
- **Reflected XSS**: The payload is immediately reflected in the response, often through search results, error pages, or URL parameters.
- **DOM-based XSS**: The vulnerability exists in client-side JavaScript that reads attacker-controlled data from the DOM, URL, fragment, or browser storage and writes it into a dangerous sink.

A useful sub-variant is **blind XSS**, where the payload executes in a different context than the one used to submit it. This is common in admin panels, support systems, and log viewers. You may inject a payload into a ticket field and only see execution later when an administrator opens the record.

**Common XSS payload patterns**

Use these as testing examples, not as production code:

```html
<script>alert(1)</script>
```

```html
<img src=x onerror=alert(1)>
```

```html
<svg onload=alert(1)>
```

For DOM XSS, the issue often comes from unsafe sinks such as `innerHTML`, `document.write()`, `insertAdjacentHTML()`, or jQuery methods that interpret HTML. Safer alternatives include `textContent`, `setAttribute()` with validation, and framework-specific escaping.

```javascript
// Unsafe
document.getElementById("msg").innerHTML = location.hash;

// Safer
document.getElementById("msg").textContent = location.hash;
```

**Why XSS works**

XSS succeeds when the application fails to separate data from code. Typical causes include:

- Missing output encoding for HTML, attribute, JavaScript, or URL contexts
- Unsafe templating or string concatenation
- Weak sanitization that removes only obvious tags but not event handlers or protocol handlers
- Client-side rendering that inserts untrusted data into dangerous DOM sinks
- Incomplete Content Security Policy (CSP) that still allows inline script or unsafe sources

**XSS filter evasion and encoding pitfalls**

Attackers often bypass weak filters by changing the context, encoding style, or payload shape. The key exam idea is that filtering is not the same as contextual encoding.

Common pitfalls include:

- **Context mismatch**: HTML encoding does not protect JavaScript string contexts, and URL encoding does not protect HTML attributes by itself.
- **Double decoding**: Input may be decoded more than once by the application or framework.
- **Partial sanitization**: Removing `<script>` tags does not stop event handlers like `onerror`, `onload`, or `onclick`.
- **Protocol abuse**: `javascript:` or `data:` URLs may execute if inserted into links without validation.
- **Broken attribute quoting**: Unescaped quotes can break out of attributes and inject new ones.
- **Unicode and entity tricks**: Encoded characters may bypass naive blacklist filters.

Examples of safer output handling:

```html
<!-- HTML context -->
<div>{{ user_input }}</div>

<!-- Attribute context -->
<input value="{{ escaped_user_input }}">
```

If you are testing manually, try variations that target different contexts:

- Break out of HTML text nodes with tags or entities
- Break out of quoted attributes with quote characters
- Test event-handler injection in HTML elements that allow attributes
- Test JavaScript string contexts with quote and escape characters
- Test DOM sinks by changing the URL fragment, query string, or hash

**XSS defenses**

The strongest defenses are contextual output encoding, safe DOM APIs, and a restrictive CSP. Sanitization is useful only when you must allow limited HTML, and it must be done with a trusted library.

- **Output encoding**: Encode for the exact context where data is inserted.
- **Input validation**: Restrict expected formats, but do not rely on validation alone.
- **Safe DOM methods**: Prefer `textContent`, `createElement()`, and `appendChild()` over HTML parsing sinks.
- **CSP**: Reduce impact by limiting script sources and blocking inline script where possible.
- **HttpOnly cookies**: Prevent JavaScript from reading session cookies, which reduces cookie theft but does not stop XSS itself.
- **SameSite cookies**: Help reduce CSRF risk and can also limit some cross-site abuse patterns.

A practical CSP example:

```http
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'self'; frame-ancestors 'none'
```

> A CSP is a defense-in-depth control, not a substitute for proper encoding and safe coding practices.

**Testing XSS in practice**

When you assess a target, map the input flow from source to sink:

1. Identify where user-controlled data enters the application.
2. Determine whether the data is reflected, stored, or processed in the browser.
3. Check the output context: HTML body, attribute, script block, URL, or DOM sink.
4. Test with harmless markers first, then with context-specific payloads.
5. Confirm whether CSP, sanitization, or framework escaping blocks execution.

Useful tools and commands:

```bash
# Intercept and modify requests during testing
burpsuite

# Quick local server for callback testing
python3 -m http.server 8000
```

**CSRF (Cross-Site Request Forgery)**

CSRF is an attack where a victim’s browser sends an authenticated request to a target site without the victim intending it. The attack works because browsers automatically attach credentials such as cookies, client certificates, or integrated authentication headers in some scenarios.

The classic CSRF target is a state-changing action such as:

- Changing an email address
- Resetting a password
- Adding a new payment method
- Transferring funds
- Changing MFA or recovery settings

A typical attack uses a hidden form, image request, or auto-submitting page:

```html
<form action="https://target.example/account/email" method="POST">
  <input type="hidden" name="email" value="attacker@example.com">
</form>
<script>
  document.forms[0].submit();
</script>
```

**When CSRF works**

CSRF is most effective when all of the following are true:

- Authentication relies on cookies or other automatically sent browser credentials
- The action changes server state
- The application does not verify a CSRF token
- The server does not validate `Origin` or `Referer`
- Cookies are not protected with effective `SameSite` settings

**When CSRF does not work**

Modern applications often resist CSRF when they use one or more of these controls:

- **Non-cookie-based auth**: If the app uses `Authorization: Bearer ...` headers, the browser does not add them automatically to cross-site requests.
- **Custom headers**: Requests that require a custom header such as `X-CSRF-Token` or `X-Requested-With` are harder to forge from a plain HTML form.
- **SameSite cookies**: `SameSite=Lax` or `SameSite=Strict` can block cookies on many cross-site requests.
- **Origin/Referer enforcement**: The server can reject requests whose origin does not match the expected site.
- **Re-authentication or step-up checks**: Sensitive actions may require password entry, MFA, or a fresh token.

This distinction matters for exam questions: CSRF is not “any forged request.” It depends on browser behavior and server-side trust in ambient credentials.

**CSRF defenses**

The standard defenses are:

- **Anti-CSRF tokens**: Unique, unpredictable tokens tied to the user session and validated server-side.
- **SameSite cookies**: Use `Lax` or `Strict` where compatible.
- **Origin/Referer validation**: Check that requests come from the expected site.
- **Custom request headers**: Require headers that browsers do not add automatically in cross-site form submissions.
- **Re-authentication**: Require confirmation for high-risk actions.
- **Avoid GET for state changes**: GET should be safe and idempotent.

Example cookie settings:

```http
Set-Cookie: session=abc123; Secure; HttpOnly; SameSite=Lax
```

Example server-side token pattern:

```html
<form method="POST" action="/profile/update">
  <input type="hidden" name="csrf_token" value="RANDOM_SERVER_GENERATED_VALUE">
  <input type="email" name="email">
  <button type="submit">Save</button>
</form>
```

**CSRF testing workflow**

1. Log in as a test user.
2. Capture a legitimate state-changing request.
3. Remove or alter the CSRF token.
4. Replay the request from a different origin.
5. Check whether the server validates `Origin`, `Referer`, and token freshness.
6. Verify whether `SameSite` blocks the cookie in cross-site conditions.

A quick test with `curl` can help validate server-side token checks:

```bash
curl -i -X POST https://target.example/profile/update \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data 'email=test@example.com'
```

If the request succeeds without a valid token or origin check, the application may be vulnerable.

**Broken Access Control**

Broken access control means the application fails to enforce who can access a resource or perform an action. This is one of the most common and highest-impact web vulnerabilities because it can expose data, allow unauthorized changes, or grant administrative capabilities.

Access control failures usually fall into two broad categories:

- **Vertical privilege escalation**: A lower-privileged user gains access to higher-privileged functions, such as a normal user reaching admin-only endpoints.
- **Horizontal privilege escalation**: A user accesses another user’s data or actions at the same privilege level, such as viewing another customer’s order or profile.

These terms are frequently tested because they describe different authorization failures:

- Vertical escalation changes privilege level.
- Horizontal escalation stays at the same privilege level but crosses user boundaries.

**Common broken access control patterns**

- Direct object references without authorization checks
- Hidden UI elements that are not enforced server-side
- Predictable identifiers in URLs, JSON bodies, or API paths
- Missing role checks on admin endpoints
- Insecure file access or path traversal combined with weak authorization
- Mass assignment that lets users set privileged fields
- Function-level authorization bypass in APIs

A common example is an endpoint that trusts a user-supplied ID:

```http
GET /api/orders/1001
```

If changing the ID to `1002` returns another user’s order without authorization, that is a horizontal access control flaw.

**API-specific access control issues**

Modern applications often expose APIs that are more vulnerable than the web UI. Two important terms are:

- **BOLA**: Broken Object Level Authorization
- **BFLA**: Broken Function Level Authorization

BOLA occurs when an API lets you access objects you do not own, often by changing an object identifier such as `user_id`, `account_id`, or `order_id`. BFLA occurs when an API exposes functions that should be restricted, such as admin actions, export endpoints, or user management operations.

Examples:

```http
GET /api/v1/users/12345
```

```http
POST /api/v1/admin/export
```

If the server does not verify ownership or role membership, the API may leak data or permit unauthorized actions even if the front end hides the controls.

**Broken access control testing**

When testing, compare what the application shows in the UI with what the server actually enforces.

1. Identify roles such as guest, user, manager, and admin.
2. Capture requests for normal and privileged actions.
3. Modify object IDs, usernames, account numbers, or path segments.
4. Replay requests with a lower-privileged account.
5. Remove client-side indicators such as hidden fields or disabled buttons.
6. Check whether the server still authorizes the action.

Useful checks include:

- Changing `user_id` values in JSON bodies
- Editing path parameters like `/users/1` to `/users/2`
- Reusing admin requests with a standard account
- Testing direct access to undocumented endpoints
- Trying HTTP methods such as `PUT`, `PATCH`, `DELETE`, and `POST` on endpoints that may not enforce method-level authorization

Example with `curl`:

```bash
curl -i https://target.example/api/users/1002 \
  -H 'Cookie: session=USER_SESSION'
```

If the response returns another user’s data, authorization is likely broken.

**Defensive controls for access control**

- Enforce authorization on the server for every request
- Check ownership and role membership before returning data
- Use deny-by-default policies
- Validate object references against the authenticated user
- Avoid relying on hidden fields, client-side logic, or front-end route guards
- Log and alert on repeated authorization failures
- Use centralized authorization middleware or policy engines

A simple policy mindset helps:

- Authenticate the user first.
- Identify the requested object or action.
- Verify the user is allowed to access that object or action.
- Return a generic denial if the check fails.

**How these issues relate in real attacks**

These vulnerabilities often chain together:

- XSS can steal a CSRF token or trigger privileged actions in the victim’s browser.
- CSRF can change account settings if the application relies on cookies and lacks token validation.
- Broken access control can expose sensitive APIs even when the UI appears secure.
- XSS plus broken access control can let an attacker discover hidden endpoints and then abuse them directly.

A practical assessment mindset is to ask:

- Can I inject script into any trusted page?
- Can I force a browser to submit a state-changing request?
- Can I access another user’s object by changing an identifier?
- Can I reach admin or internal API functions without the right role?

**Quick comparison**

| Topic | Primary trust abused | Typical impact | Key defense |
|---|---|---|---|
| XSS | Browser trusts site content | Script execution, session abuse, data theft | Contextual encoding, safe DOM APIs, CSP |
| CSRF | Site trusts browser credentials | Unauthorized state change | CSRF tokens, SameSite, Origin checks |
| Broken access control | Server trusts user identity or object IDs | Unauthorized data access or actions | Server-side authorization, ownership checks |

**Practical lab mindset**

In a lab or assessment, use a repeatable workflow and keep notes on each request and response. Focus on whether the server enforces security or merely the interface.

- Inspect requests in a proxy and compare roles.
- Test direct object access with different IDs.
- Check whether tokens are required and validated.
- Observe whether cookies are sent cross-site.
- Confirm whether client-side restrictions are only cosmetic.

**Exam focus points**

- XSS is about injecting script into a trusted page or DOM context.
- Stored, reflected, DOM-based, and blind XSS are all important variants.
- CSRF depends on automatic credential sending, especially cookies.
- CSRF is weaker or ineffective with non-cookie auth, custom headers, SameSite cookies, and origin validation.
- Broken access control includes both horizontal and vertical privilege escalation.
- API authorization failures often appear as BOLA and BFLA.
- Server-side authorization is mandatory; UI restrictions are not security controls.
