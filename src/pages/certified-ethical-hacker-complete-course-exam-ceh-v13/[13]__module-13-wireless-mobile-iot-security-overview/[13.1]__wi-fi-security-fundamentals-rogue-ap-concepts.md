**Wi-Fi security fundamentals**

Wi-Fi security protects wireless networks from unauthorized access, eavesdropping, and impersonation. Because radio signals extend beyond physical walls, wireless attacks often begin outside the target perimeter and can bypass assumptions that apply to wired networks.

At a high level, you secure Wi-Fi by combining:
- **Strong authentication** to verify users and devices
- **Encryption** to protect traffic confidentiality and integrity
- **Access control** to limit who can join the network
- **Monitoring** to detect unauthorized radios, clients, and configuration drift

The most important wireless security lesson is that the access point (AP) is only one part of the trust chain. Client behavior, authentication method, management plane exposure, and RF coverage all affect risk.

**Wireless network architecture**

A typical enterprise wireless deployment includes:
- **Wireless clients** such as laptops, phones, and IoT devices
- **Access points** that bridge wireless stations to the wired LAN
- **Wireless LAN controller (WLC)** or cloud-managed platform for centralized policy
- **Authentication services** such as RADIUS, LDAP, or directory-backed identity
- **Switching and routing infrastructure** that carries client traffic after association

Common wireless terms:
- **SSID**: Service Set Identifier, the network name broadcast or hidden by an AP
- **BSSID**: Basic Service Set Identifier, usually the AP radio MAC address
- **ESS**: Extended Service Set, multiple APs sharing one logical WLAN
- **STA**: Station, a wireless client device
- **RF**: Radio frequency, the spectrum used for wireless communication

A client does not connect to “Wi-Fi” in the abstract; it associates to a specific BSSID that advertises an SSID and security profile.

**802.11 security building blocks**

Wireless security depends on both link-layer protection and authentication.

- **Open networks**
  - No encryption at the Wi-Fi layer
  - Traffic is visible to anyone within radio range unless protected by higher-layer encryption such as HTTPS or VPN
  - Useful only for guest or captive portal scenarios with compensating controls

- **WEP**
  - Wired Equivalent Privacy, an obsolete and broken protocol
  - Uses RC4 with weak initialization vector handling
  - Should never be used

- **WPA**
  - Wi-Fi Protected Access, an interim improvement over WEP
  - Commonly associated with TKIP, which is also deprecated
  - Considered legacy and insecure for modern deployments

- **WPA2**
  - Uses AES-based CCMP for strong confidentiality and integrity
  - Supports personal and enterprise modes
  - Still widely deployed, but must be configured correctly

- **WPA3**
  - Modern standard with stronger handshake protections
  - Uses SAE (Simultaneous Authentication of Equals) in personal mode
  - Improves resistance to offline password guessing compared with WPA2-Personal

**Personal vs enterprise authentication**

Wireless security usually falls into two deployment models:

- **WPA2/WPA3-Personal**
  - Uses a shared passphrase
  - Simpler to deploy
  - Harder to manage at scale because all users share the same secret
  - Risk increases when the passphrase leaks or is reused

- **WPA2/WPA3-Enterprise**
  - Uses 802.1X with a RADIUS backend
  - Provides per-user authentication and often per-session keys
  - Better for auditing, revocation, and policy enforcement
  - Supports stronger identity controls such as certificates and MFA integration in some environments

Enterprise Wi-Fi is preferred when you need:
- Individual accountability
- Centralized access revocation
- Dynamic VLAN assignment or role-based access
- Better resistance to shared-secret compromise

**802.1X and EAP**

<abbr title="Extensible Authentication Protocol">EAP</abbr> is a framework used inside 802.1X authentication. The AP acts as a pass-through, while the RADIUS server validates the user or device.

Common EAP methods include:
- **EAP-TLS**
  - Uses client and server certificates
  - Strongest common enterprise option
  - Requires certificate lifecycle management

- **PEAP**
  - Protects an inner authentication exchange inside a TLS tunnel
  - Common in enterprise environments
  - Often paired with username/password credentials

- **EAP-TTLS**
  - Similar to PEAP in concept
  - Encapsulates legacy authentication methods inside TLS

The security of 802.1X depends heavily on server certificate validation. If clients do not validate the RADIUS server certificate, they may connect to a fake AP or evil twin and expose credentials.

**Wi-Fi encryption and key management**

Encryption protects data after association, but the key exchange matters as much as the cipher.

- **CCMP**
  - Counter Mode with Cipher Block Chaining Message Authentication Code Protocol
  - AES-based and standard for WPA2
  - Provides confidentiality and integrity

- **GCMP**
  - Galois/Counter Mode Protocol
  - Used in newer WPA3 deployments
  - Designed for stronger performance and modern security

- **4-way handshake**
  - Establishes session keys between client and AP
  - Can be targeted for capture and offline analysis in weak configurations
  - Strong passwords and WPA3 reduce practical attack value

- **PMK**
  - Pairwise Master Key
  - Root key material used in deriving session keys

- **PTK**
  - Pairwise Transient Key
  - Derived for a specific client-AP session

- **GTK**
  - Group Temporal Key
  - Used for broadcast and multicast traffic

A secure wireless design uses strong authentication plus modern encryption. Encryption alone does not stop rogue AP impersonation if the client accepts the wrong network identity.

**Wireless threats and attack surface**

Wireless attacks often exploit trust, convenience, and weak user awareness.

- **Evil twin**
  - A rogue AP that imitates a legitimate SSID
  - Often uses stronger signal power to attract clients
  - Can capture credentials or force clients onto a malicious network

- **Rogue AP**
  - Any unauthorized AP connected to or near the network
  - May be malicious, misconfigured, or installed by an employee
  - Creates policy, security, and compliance risk

- **Ad hoc network abuse**
  - Peer-to-peer wireless connections can bypass central controls
  - Can create hidden pathways into a trusted environment

- **Deauthentication abuse**
  - Attackers may disrupt client connections to encourage reconnection to a rogue AP
  - Useful in evil twin scenarios

- **Misconfigured AP**
  - Weak passwords, default credentials, outdated firmware, or open management interfaces
  - Can be exploited without sophisticated radio attacks

- **Rogue client hotspot**
  - A laptop or phone sharing its connection as a hotspot
  - Can bridge corporate and untrusted networks

**Rogue AP categories**

Rogue APs are often classified by intent and connectivity.

- **Unauthorized but benign**
  - Installed by an employee for convenience
  - Still dangerous because it bypasses policy and monitoring

- **Malicious rogue**
  - Deliberately deployed to intercept traffic or harvest credentials
  - Often paired with phishing or captive portal tricks

- **External rogue**
  - Nearby AP not owned by the organization
  - May interfere with channels or impersonate corporate SSIDs

- **Internal rogue**
  - Connected to the corporate wired network
  - Highest risk because it can provide direct access to internal resources

An internal rogue AP is usually the most serious finding because it combines wireless exposure with wired trust.

**Rogue AP detection concepts**

Detection focuses on identifying unauthorized radios, suspicious SSIDs, and abnormal client behavior.

Indicators include:
- **Duplicate SSID** with different BSSID or security settings
- **Unexpected channel or power levels**
- **Unknown MAC addresses** advertising corporate names
- **Clients associating to unapproved APs**
- **Management frames** indicating deauth or spoofing activity
- **WIDS/WIPS alerts** from wireless intrusion detection/prevention systems

Useful monitoring sources:
- Wireless controller logs
- Spectrum analysis tools
- Packet captures from monitor-mode adapters
- DHCP and authentication logs
- NAC and endpoint telemetry

A wireless intrusion detection system (WIDS) observes suspicious activity, while a wireless intrusion prevention system (WIPS) can actively block or contain it. In practice, organizations often combine both capabilities in controller-managed platforms.

**Practical detection workflow**

1. Inventory authorized APs, SSIDs, channels, and BSSIDs.
2. Compare live RF observations against the approved inventory.
3. Investigate duplicate SSIDs, unknown BSSIDs, and unusual signal strength.
4. Correlate wireless events with RADIUS, DHCP, and switch port data.
5. Locate the physical source of the rogue device.
6. Remove or quarantine the device and document the incident.

This workflow matters because wireless incidents are often ambiguous at first. A suspicious SSID may be a neighbor’s AP, a misconfigured extender, or a deliberate attack. Correlation reduces false positives.

**Example reconnaissance commands**

On Linux, you can inspect wireless interfaces and nearby networks with standard tools:

```bash
ip link show
iw dev
nmcli dev wifi list
```

To capture wireless traffic, you typically need monitor mode and a compatible adapter:

```bash
sudo ip link set wlan0 down
sudo iw dev wlan0 set type monitor
sudo ip link set wlan0 up
sudo airodump-ng wlan0
```

To inspect a specific AP and its security settings:

```bash
sudo iwlist wlan0 scan | less
```

For packet analysis, capture frames and review them in Wireshark:

```bash
sudo tcpdump -i wlan0 -w wifi-capture.pcap
```

These commands help you identify SSIDs, BSSIDs, channels, and encryption types, but they do not by themselves prove malicious intent.

**Example evil twin scenario**

A user sees a corporate SSID named `CorpWiFi` and connects automatically. An attacker nearby broadcasts the same SSID with a stronger signal and a captive portal that asks for credentials.

What makes this dangerous:
- The SSID looks familiar
- The client may prefer the stronger signal
- The user may not verify the certificate or BSSID
- Credentials entered into the portal can be harvested

Defenses include:
- Certificate validation for enterprise Wi-Fi
- Disabling auto-join for untrusted networks
- User training on SSID verification
- WIDS alerts for duplicate SSIDs
- Protected management frames where supported

**Protected management frames**

Management frames control association, disassociation, and roaming. Historically, many of these frames were unauthenticated, which made spoofing easier.

- **802.11w**
  - Protects management frames
  - Helps reduce deauthentication and disassociation abuse
  - Commonly integrated into WPA2/WPA3 deployments as protected management frames

Protected management frames do not eliminate all wireless attacks, but they raise the cost of disruption and make some spoofing techniques less effective.

**WPA3 improvements**

WPA3 improves on WPA2 in several important ways:
- Uses **SAE** in personal mode instead of a traditional PSK-only handshake
- Reduces exposure to offline dictionary attacks
- Encourages stronger cryptographic defaults
- Supports better protection for open networks through enhanced open modes in some implementations

WPA3 is not a magic shield. Weak passwords, poor client validation, and insecure management still create risk.

**Common misconfigurations**

Wireless security failures often come from configuration mistakes rather than advanced exploits.

- Reusing the same passphrase across many sites
- Leaving default SSIDs and admin credentials in place
- Allowing weak encryption modes for legacy compatibility
- Disabling certificate validation on enterprise clients
- Exposing AP management interfaces to untrusted networks
- Failing to segment guest and internal traffic
- Not updating AP firmware
- Ignoring rogue AP alerts because of alert fatigue

A secure wireless program treats firmware, identity, and monitoring as ongoing tasks, not one-time setup steps.

**Defensive best practices**

- Use **WPA3-Enterprise** or WPA2-Enterprise where supported
- Prefer **EAP-TLS** for high-assurance environments
- Enforce certificate validation on all managed clients
- Disable legacy protocols such as WEP and WPA/TKIP
- Segment guest, IoT, and corporate networks
- Enable protected management frames where possible
- Maintain an approved AP inventory with BSSID and location data
- Monitor for rogue APs, evil twins, and unauthorized hotspots
- Patch AP firmware and controller software regularly
- Use strong, unique credentials for management access
- Restrict AP management to dedicated admin networks or VPN access

**Troubleshooting and validation**

When validating wireless security, check both policy and behavior.

- Confirm the SSID broadcasts the expected security mode
- Verify the AP advertises the correct cipher suite
- Test that clients reject invalid server certificates
- Confirm RADIUS authentication logs match user activity
- Check that guest traffic cannot reach internal subnets
- Validate that rogue AP alerts trigger response procedures

A simple validation checklist can prevent many incidents:

```text
[ ] SSID matches approved inventory
[ ] BSSID matches known AP
[ ] Encryption is WPA2-AES or WPA3
[ ] Enterprise clients validate server certificate
[ ] Management frames are protected where supported
[ ] Guest VLAN is isolated
[ ] Rogue detection is enabled
```

**Exam focus points**

For CEH v13, focus on the relationships between concepts:
- **Rogue AP** means unauthorized access point, not necessarily malicious
- **Evil twin** means impersonation of a legitimate SSID
- **WIDS/WIPS** detect and may block suspicious wireless activity
- **WPA2/WPA3-Enterprise** use 802.1X and RADIUS
- **WPA3** improves resistance to offline password attacks
- **Certificate validation** is critical in enterprise Wi-Fi
- **Protected management frames** reduce deauth-style disruption

Understanding these distinctions helps you answer scenario-based questions where the same wireless symptom can have multiple causes.

**Quick comparison**

| Topic | Key idea | Security impact |
|---|---|---|
| Open Wi-Fi | No link-layer encryption | High exposure unless higher-layer protections exist |
| WPA2-Personal | Shared passphrase | Simple but weak if the password is reused or guessed |
| WPA2-Enterprise | 802.1X with RADIUS | Stronger identity and revocation control |
| WPA3-Personal | SAE handshake | Better resistance to offline guessing |
| Rogue AP | Unauthorized AP | Policy violation and possible compromise |
| Evil twin | Fake AP impersonating a real SSID | Credential theft and traffic interception |
| WIDS/WIPS | Detection and prevention | Helps identify and contain wireless threats |

**Operational example**

A secure enterprise deployment might look like this:

```text
SSID: CorpWiFi
Security: WPA3-Enterprise
Authentication: 802.1X / RADIUS
EAP method: EAP-TLS
Guest SSID: GuestWiFi
Guest access: Internet only
Management: Admin VLAN + VPN
Rogue detection: Enabled on controller
PMF: Required where supported
```

This design separates identity, access, and monitoring so that a compromise in one area does not automatically expose the entire environment.

**Key takeaways**

Wireless security is about trust, identity, and radio visibility. You protect the network by using strong authentication, modern encryption, certificate validation, segmentation, and continuous rogue AP monitoring. The most dangerous wireless threats often succeed by looking legitimate, so verification and inventory discipline matter as much as cryptography.
