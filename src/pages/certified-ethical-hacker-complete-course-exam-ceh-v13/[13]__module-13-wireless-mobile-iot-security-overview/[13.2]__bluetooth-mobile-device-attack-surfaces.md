## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

Bluetooth, mobile, and embedded devices expand the attack surface beyond traditional endpoints. In CEH-style assessments, these targets are often evaluated for weak pairing, insecure app behavior, exposed management interfaces, outdated firmware, and poor trust controls. The key idea is simple: if a device communicates, stores secrets, or accepts updates, it can usually be attacked through its protocol, its software, or its physical interfaces.

These notes focus on how attackers chain weaknesses across radio, app, OS, and hardware layers. You should understand not only the individual vulnerabilities, but also the assessment workflow: discover the target, identify the trust boundary, test authentication and encryption, inspect update paths, and validate whether the device can be coerced into exposing data or accepting malicious control.

**Bluetooth Fundamentals and Security Model**

Bluetooth is a short-range wireless protocol family used for peripherals, audio, wearables, medical devices, and IoT controllers. CEH questions often distinguish between **Bluetooth Classic** and **Bluetooth Low Energy (BLE)**, because their pairing flows, attack surfaces, and security assumptions differ.

- **Bluetooth Classic** is optimized for continuous connections such as audio and file transfer.
- **BLE** is optimized for low power and uses advertising, scanning, and GATT (Generic Attribute Profile) services.
- **Secure Simple Pairing (SSP)** is used in Classic to improve pairing security over legacy PIN-based methods.
- **LE Secure Connections** improves BLE pairing by using stronger cryptographic key exchange than older LE pairing modes.

A practical exam point is that older Bluetooth versions and legacy pairing modes are much easier to attack. Weak or legacy pairing can expose devices to passive capture, downgrade behavior, or key recovery attacks.

**Bluetooth Versions, Pairing, and Security Differences**

Bluetooth security depends heavily on the version and pairing mode in use. Older implementations often rely on weaker key exchange or predictable pairing behavior, while newer implementations improve confidentiality and integrity but still fail when vendors misconfigure services or expose insecure characteristics.

| Area | Classic Bluetooth | BLE |
|---|---|---|
| Primary use | Audio, peripherals, file transfer | Sensors, wearables, low-power devices |
| Discovery model | Inquiry and paging | Advertising and scanning |
| Common profile layer | RFCOMM, A2DP, HFP, OBEX | GATT services and characteristics |
| Pairing security | Legacy PIN or SSP | Legacy pairing or LE Secure Connections |
| Typical weakness | Weak PINs, insecure services, downgrade | GATT abuse, weak permissions, replay-like logic flaws |

Important security concepts:

- **Legacy pairing** is weaker because it may rely on short PINs or less robust key exchange.
- **SSP** reduces exposure to passive eavesdropping, but it is still vulnerable to user-interface attacks such as Just Works acceptance.
- **LE Secure Connections** improves BLE key exchange, but it does not protect against insecure application logic, exposed characteristics, or poor authorization.
- **Just Works** pairing is convenient but provides no meaningful MITM (man-in-the-middle) protection, so it is risky for sensitive devices.

> <mark>Security improves with newer Bluetooth modes, but application design still determines whether the device is actually safe.</mark>

**Bluetooth Attack Surface and Common Techniques**

Bluetooth attacks usually target discovery, pairing, service exposure, or protocol implementation flaws. In CEH-style scenarios, you should think in terms of what the attacker can do before, during, and after pairing.

- **Sniffing and traffic analysis** target weak or legacy pairing and poorly protected services.
- **Bluejacking** sends unsolicited messages to nearby devices; it is usually nuisance-level but demonstrates proximity abuse.
- **Bluesnarfing** attempts unauthorized access to data such as contacts or files through vulnerable services.
- **Bluebugging** aims to gain control over device functions such as calls or messages.
- **Device impersonation** exploits trust in known device names, MAC addresses, or service UUIDs.
- **Downgrade attacks** try to force weaker pairing or older protocol behavior.

More exam-oriented Bluetooth vulnerabilities include:

- **BlueBorne**: a set of vulnerabilities affecting Bluetooth stacks that can enable remote code execution or information disclosure without pairing in some cases.
- **KNOB** (Key Negotiation of Bluetooth): attacks the key negotiation process in Classic Bluetooth and can reduce encryption key strength if the implementation is vulnerable.
- **BIAS** (Bluetooth Impersonation AttackS): targets authentication weaknesses in Bluetooth Classic, allowing impersonation of a previously paired device in certain conditions.
- **BLE GATT spoofing**: a rogue device imitates a legitimate GATT server or service layout to trick a client application.
- **BLE replay or logic replay**: reuses captured commands or characteristic writes when the application fails to enforce freshness, sequence checks, or session binding.

A realistic attack chain might look like this:

1. The attacker scans for nearby BLE advertisements.
2. They identify a device exposing a writable characteristic without strong authorization.
3. They connect using a rogue client or spoofed service.
4. They replay a previously observed command or write a crafted value.
5. The device accepts the action because it trusts the connection too much.

**Bluetooth Tools and Assessment Workflow**

Common tools help you enumerate devices, inspect services, and test protocol behavior. In assessments, you usually start passive, then move to active testing.

- `bluetoothctl` is useful for Linux-based discovery, pairing, trust, and connection testing.
- `btmgmt` helps inspect and manage the Bluetooth controller.
- `hcitool` and `hciconfig` are legacy utilities still seen in labs and older documentation.
- `gatttool` is deprecated but still appears in older workflows for BLE service interaction.
- `nmap` with Bluetooth scripts can help enumerate some services and device details.
- `Wireshark` can analyze Bluetooth traffic when you have capture access.
- `bettercap` and similar frameworks may be used for wireless assessment workflows.
- Specialized research tools may be used for BLE fuzzing, GATT enumeration, or protocol testing.

A practical workflow often looks like this:

```bash
bluetoothctl
power on
scan on
devices
pair AA:BB:CC:DD:EE:FF
trust AA:BB:CC:DD:EE:FF
connect AA:BB:CC:DD:EE:FF
info AA:BB:CC:DD:EE:FF
```

For BLE service inspection, you may enumerate GATT characteristics and look for weak permissions:

```bash
# Example with a BLE client utility
# Read a characteristic
gatttool -b AA:BB:CC:DD:EE:FF --char-read --handle=0x0025

# Write a value to test authorization
gatttool -b AA:BB:CC:DD:EE:FF --char-write-req --handle=0x0028 --value=0100
```

When you assess Bluetooth, ask these questions:

- Is the device discoverable when it should not be?
- Does it accept pairing without user verification?
- Are services exposed before authentication?
- Are GATT characteristics readable or writable without authorization?
- Does the device enforce bonding, whitelisting, or device identity checks?
- Can the device be impersonated by a rogue peripheral or client?

**Bluetooth Defensive Controls**

Good Bluetooth security is a mix of protocol choice, policy, and device hardening. The strongest control is not “turn Bluetooth off” in all cases, but rather to reduce exposure and enforce trust boundaries.

- Prefer **LE Secure Connections** or modern secure pairing modes where supported.
- Disable discoverability when not needed.
- Restrict pairing to approved devices or managed enrollment windows.
- Use whitelisting or bonding controls for trusted peripherals.
- Avoid Just Works for sensitive workflows.
- Keep Bluetooth stack firmware and OS patches current.
- Monitor for rogue advertisements, unknown pairings, and repeated connection attempts.

A policy example for managed endpoints:

```ini
# Conceptual Bluetooth policy
[Bluetooth]
Discoverable=false
PairingMode=SecureOnly
AllowUnknownDevices=false
AutoTrust=false
RequireUserApproval=true
```

**Mobile Attack Surface Overview**

Mobile devices combine radio interfaces, apps, sensors, cloud sync, and local secrets. The attack surface includes the operating system, app sandbox, permissions, storage, network stack, and device management layer. CEH questions often focus on how attackers abuse user trust, weak app controls, and insecure enterprise enrollment.

Major mobile attack categories include:

- **Malware** such as trojans, spyware, ransomware, banking malware, and stalkerware.
- **Smishing**: phishing via SMS or messaging apps.
- **Vishing**: voice phishing that tricks users into revealing credentials or approving actions.
- **Malicious app sideloading**: installing apps outside official stores.
- **App repackaging**: modifying a legitimate app to inject malicious code or ads.
- **Mobile OS exploitation**: targeting kernel, baseband, browser, or media parsing flaws.
- **Credential theft** through fake login pages, overlay attacks, or accessibility abuse.
- **Data exfiltration** from insecure storage, backups, or cloud sync.

A common real-world chain is:

1. The attacker sends a smishing message with a fake delivery or security alert.
2. The victim installs a sideloaded app or opens a malicious link.
3. The app requests excessive permissions or abuses accessibility services.
4. The malware steals tokens, intercepts OTPs (one-time passwords), or overlays a banking login.
5. The attacker uses the stolen session to access enterprise or financial accounts.

**Mobile Malware Types and Delivery Methods**

Mobile malware is often designed to blend in with legitimate apps or abuse trust relationships already present on the device.

- **Trojanized apps** look legitimate but contain malicious payloads.
- **Spyware** captures messages, calls, location, screenshots, or microphone data.
- **Ransomware** locks the device or encrypts user data.
- **Banking trojans** target financial apps, often using overlays and accessibility abuse.
- **Adware** may seem low risk but can be a delivery vehicle for more serious payloads.
- **Stalkerware** is often installed by someone with physical access and may hide from the user.

Delivery methods commonly include:

- Third-party app stores with weak vetting.
- Sideloaded APKs (Android Package Kits).
- Malicious links in SMS, email, QR codes, or social media.
- Repackaged apps distributed through unofficial channels.
- Compromised developer accounts or supply-chain abuse.

**Android and iOS Security Concepts**

Android and iOS differ in openness, app distribution, and management, but both rely on sandboxing, signing, and permission controls.

- **Android** allows broader sideloading and more varied device ecosystems, which increases fragmentation and patch inconsistency.
- **iOS** is more tightly controlled, but it still faces phishing, enterprise certificate abuse, jailbreak-related risk, and malicious configuration profiles.
- **App signing** establishes trust that the app package came from a known developer identity.
- **Sandboxing** limits app access to other apps and system resources.
- **Permissions** control access to camera, microphone, contacts, location, Bluetooth, and storage.

Important exam concepts:

- **App repackaging** is easier when users install from unofficial sources or when signature validation is weak in downstream ecosystems.
- **Jailbreak/root detection bypass** allows malware or testers to hide elevated access and evade app defenses.
- **Device attestation** helps a server verify that a device is genuine, unmodified, and running expected software.
- **Secure storage** uses Android Keystore or iOS Keychain, but secrets can still be exposed if apps misuse them or if the device is compromised.
- **Overlay attacks** can trick users into entering credentials into fake UI elements.

A practical Android permission example:

```xml
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.example.secureapp">

    <uses-permission android:name="android.permission.CAMERA" />
    <uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />

    <application
        android:allowBackup="false"
        android:usesCleartextTraffic="false">
    </application>
</manifest>
```

This configuration reduces some risk, but it does not guarantee security. You still need runtime checks, secure network transport, and proper secret handling.

**Mobile Forensics and Defensive Controls**

Mobile defense is not only about blocking malware; it is also about preserving trust in enterprise data and detecting compromise early. In CEH-style environments, you should understand how MDM (Mobile Device Management) and MAM (Mobile Application Management) differ.

- **MDM** manages the whole device, including policy, compliance, and remote wipe.
- **MAM** focuses on protecting corporate apps and data, often without full device control.
- **Enrollment abuse** occurs when an attacker or unauthorized user enrolls a device into management to gain policy exceptions or access.
- **Compliance checks** verify OS version, encryption, passcode strength, and jailbreak/root status.
- **Remote wipe** can remove corporate data from lost or compromised devices.

Defensive controls to know:

- Enforce strong screen locks and biometric fallback policies.
- Require full-disk encryption.
- Restrict sideloading and unknown sources.
- Block rooted or jailbroken devices from accessing sensitive apps.
- Use certificate-based authentication where possible.
- Store secrets in Keychain or Keystore, not in plain files or shared preferences.
- Use app attestation and device integrity checks before granting access.

A simple MDM-style policy example:

```yaml
device_compliance:
  require_encryption: true
  minimum_os_version: "16.0"
  allow_rooted_or_jailbroken: false
  allow_unknown_sources: false
  require_screen_lock: true
  corporate_data_wipe_on_noncompliance: true
```

**Mobile Security Testing Focus Areas**

When testing mobile apps or devices, you should look for both technical and behavioral weaknesses. The goal is to identify where trust is assumed but not verified.

- Check whether the app stores tokens, API keys, or session cookies in insecure locations.
- Inspect whether network traffic uses TLS (Transport Layer Security) correctly and validates certificates.
- Test whether the app accepts debug builds, test endpoints, or hardcoded credentials.
- Verify whether the app detects rooted/jailbroken environments and whether that detection can be bypassed.
- Review whether the app exposes exported activities, intents, or deep links that can be abused.
- Examine whether backups, logs, screenshots, or clipboard data leak sensitive information.

Useful assessment tools and approaches include:

- `adb` for Android device interaction, log collection, and app inspection.
- `frida` for dynamic instrumentation and runtime analysis.
- `objection` for mobile app testing workflows.
- `apktool` for decompiling Android packages.
- `jadx` for Java/Kotlin decompilation review.
- Proxy tools such as Burp Suite for intercepting and modifying app traffic.

Example Android inspection commands:

```bash
adb devices
adb shell pm list packages | grep example
adb shell dumpsys package com.example.secureapp
adb logcat
```

**Device Attack Surface and Embedded Systems**

Embedded and IoT devices often fail because they assume physical access is trusted. In reality, attackers may have local access, network access, or both. These devices frequently expose debug ports, weak update mechanisms, and hardcoded credentials.

Common device categories include:

- Routers and access points
- Cameras and DVRs
- Smart locks and access control systems
- Printers and scanners
- Industrial controllers
- Wearables and medical devices
- Consumer IoT hubs and sensors

Typical weaknesses include:

- Default or hardcoded credentials.
- Insecure web admin panels.
- Exposed debug interfaces such as UART, JTAG, SPI, and USB.
- Weak or unsigned firmware updates.
- Insecure boot chains.
- Poor certificate validation.
- Hardcoded API endpoints or cloud tokens.

**Firmware Extraction, Reverse Engineering, and Boot Security**

Firmware analysis is a major part of device assessment because it reveals hidden services, credentials, and update logic. Attackers often extract firmware from vendor downloads, update packages, or physical flash chips.

Common firmware analysis steps:

1. Obtain the firmware image from the vendor or device.
2. Identify file system type and architecture.
3. Extract the contents and search for credentials, keys, and endpoints.
4. Review startup scripts, web interfaces, and update routines.
5. Compare the firmware version against known vulnerabilities.

Example commands:

```bash
file firmware.bin
binwalk -e firmware.bin
strings -a extracted-rootfs/* | grep -iE "password|token|api|admin"
```

Security concepts to know:

- **Secure boot** verifies each boot stage before execution.
- **Insecure boot** allows modified bootloaders or kernels to run.
- **Chain of trust** ensures each stage validates the next stage.
- **OTA** (over-the-air) updates should be signed and integrity-protected.
- **Unsigned updates** can allow malicious firmware installation if the device accepts them.

> If a device accepts firmware without signature verification, the update channel becomes a direct code execution path.

**Hardware Interfaces and Physical Attack Paths**

Physical interfaces are often the fastest route to compromise when the attacker can touch the device. Even when the network is hardened, debug and maintenance ports may remain open.

- **UART** provides serial console access and may expose boot logs or shells.
- **JTAG** supports low-level debugging and can allow memory inspection or code execution on some devices.
- **SPI** and **I2C** can expose flash or peripheral communication.
- **USB** may provide debug, recovery, or mass storage modes.
- **SD card** or removable media can be used for boot or data extraction on some devices.

Assessment considerations:

- Look for unprotected console access.
- Check whether bootloader menus allow password resets or firmware flashing.
- Determine whether debug pins are populated or merely hidden on the board.
- Verify whether the device disables debug interfaces in production builds.

**CEH Attack Methodology Mapping**

CEH-style questions often ask you to classify attacks by technique or phase. For these targets, the following mappings are useful:

- **Reconnaissance**: scan Bluetooth advertisements, identify mobile app package names, enumerate device models and firmware versions.
- **Scanning and enumeration**: inspect GATT services, app permissions, exposed ports, and management interfaces.
- **Gaining access**: exploit weak pairing, sideload malicious apps, abuse default credentials, or use a vulnerable update path.
- **Maintaining access**: persist through malicious profiles, startup services, background apps, or modified firmware.
- **Covering tracks**: hide malware icons, suppress logs, disable alerts, or use root/jailbreak evasion.

Common named attack categories you should recognize:

- **Bluejacking**
- **Bluesnarfing**
- **Bluebugging**
- **BlueBorne**
- **KNOB**
- **BIAS**
- **Smishing**
- **Vishing**
- **Sideloading**
- **Repackaging**
- **Overlay attacks**
- **Firmware tampering**
- **Bootloader abuse**

**Practical Scenario: Bluetooth to Mobile to Device Chain**

A realistic assessment may combine all three surfaces. For example, a smart lock uses BLE for provisioning, a mobile app for control, and an embedded controller for actuation.

- The attacker first enumerates BLE advertisements and discovers a provisioning service.
- They test whether the GATT characteristic accepts writes without proper authorization.
- They inspect the mobile app and find hardcoded API endpoints and weak certificate pinning.
- They analyze the firmware update package and discover it is unsigned.
- They then determine that the device exposes UART pads on the board, allowing recovery-mode access.

This kind of chain matters because the weakest layer often determines the outcome. A secure mobile app cannot fully protect an insecure BLE service, and a secure BLE service cannot compensate for unsigned firmware.

**Quick Comparison of Common Controls**

| Control | Protects Against | Limitation |
|---|---|---|
| LE Secure Connections | Weak BLE pairing | Does not fix bad app authorization |
| App signing | Tampered app packages | Does not stop phishing or overlays |
| MDM compliance | Unmanaged or risky devices | Can be bypassed on compromised devices |
| Secure boot | Modified firmware | Depends on trusted root keys |
| Certificate pinning | MITM on mobile traffic | Can be bypassed in rooted/jailbroken testing environments |
| Whitelisting | Rogue Bluetooth devices | Does not stop trusted-device impersonation if trust is abused |

**Exam Tips**

- Distinguish **Bluetooth Classic** from **BLE** by use case and security model.
- Remember that **pairing security** is not the same as **application authorization**.
- Treat **Just Works** as weak for sensitive environments.
- Associate **BlueBorne**, **KNOB**, and **BIAS** with Bluetooth attack knowledge expected in exam prep.
- Link mobile threats to delivery methods: smishing, vishing, sideloading, and repackaging.
- Remember that mobile defense includes **MDM**, **MAM**, attestation, and secure storage.
- For devices, focus on **firmware**, **boot trust**, **OTA updates**, and **hardware interfaces**.
