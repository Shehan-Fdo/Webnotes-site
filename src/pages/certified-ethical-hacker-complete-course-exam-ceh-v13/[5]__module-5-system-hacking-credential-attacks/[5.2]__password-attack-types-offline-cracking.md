## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Password attack overview**

Password attacks aim to obtain authentication secrets or bypass authentication controls. In CEH scenarios, the key distinction is whether the attacker must interact with a live service or can work from captured credential material.

- **Online attacks** target a login service directly and are constrained by lockouts, rate limits, MFA, and monitoring.
- **Offline cracking** targets captured hashes, tickets, or encrypted credential material without repeatedly querying the authentication service.
- **Credential theft** often starts the attack chain, but the cracking phase may happen later and elsewhere.

A useful exam mindset is to ask: “Can the attacker test guesses against a local copy of the secret, or must they ask the target system each time?”

**Common password attack types**

Password attacks are not all the same. Some try many guesses, some reuse leaked credentials, and some abuse protocol behavior or user psychology.

- **Brute force** tries every possible combination within a defined keyspace.
- **Dictionary attacks** try words from a list, often with common variations.
- **Hybrid attacks** combine a wordlist with appended or prepended characters.
- **Mask attacks** use a known pattern, such as a fixed prefix plus unknown digits.
- **Credential stuffing** reuses username/password pairs leaked from other breaches.
- **Password spraying** tries a small set of common passwords across many accounts to avoid lockouts.
- **Phishing and social engineering** trick users into revealing passwords or approving access.
- **Keylogging and malware** capture credentials as they are typed or stored.
- **Pass-the-hash** reuses a captured hash for authentication in protocols that accept it, rather than cracking it.

> **Important:** Pass-the-hash is not offline cracking. It is an authentication abuse technique that uses the hash directly, so do not confuse it with password recovery.

**Online versus offline attacks**

Online attacks are limited by the target’s defenses. Offline attacks are limited mainly by the attacker’s hardware, the hash algorithm, and the quality of the password.

| Aspect | Online attack | Offline cracking |
|---|---|---|
| Target interaction | Repeated requests to a live service | Local testing against captured material |
| Rate limits | Common and effective | Not applicable |
| Lockouts | Often triggered | Not applicable |
| MFA impact | Strongly reduces success | Does not protect the hash file itself |
| Detection | Easier to log and alert | Harder to detect once data is stolen |
| Speed | Slow to moderate | Potentially very fast for weak hashes, slower for strong KDFs |

MFA (multi-factor authentication) matters in both worlds, but differently. It can block many online login attempts even if a password is known, yet it does not stop an attacker from cracking a stolen hash file. It still reduces the value of a cracked password if the account requires a second factor for access.

**Where attackers get hashes or credential material**

CEH questions often focus on how attackers obtain material for offline cracking. The source matters because it hints at the likely hash type and the best next step.

- **Windows local credential stores**
  - `SAM` (Security Account Manager) contains local account password hashes.
  - `SYSTEM` is often needed with `SAM` to decrypt the stored hashes.
- **Domain controller databases**
  - `NTDS.dit` stores Active Directory credential data on domain controllers.
  - Attackers often pair it with the `SYSTEM` hive and sometimes `SECURITY` for decryption context.
- **Memory scraping**
  - `LSASS` (Local Security Authority Subsystem Service) memory can expose plaintext credentials, hashes, Kerberos material, or tickets if protections are weak or bypassed.
- **Network captures**
  - Weak or legacy protocols may expose challenge-response material, not always reusable hashes.
  - Examples include NTLM challenge-response exchanges or Kerberos ticket material.
- **Application databases and config files**
  - Web apps may store password hashes in databases or configuration files.
  - Backup files, exports, and logs can also leak credential material.
- **Unix/Linux systems**
  - `/etc/shadow` stores password hashes, while `/etc/passwd` historically held account metadata.
- **Cloud and SaaS environments**
  - Misconfigured backups, exposed secrets stores, or leaked tokens can provide authentication material, though not always traditional password hashes.

> **Critical:** A captured network exchange is not always a reusable password hash. In many cases it is challenge-response data or ticket material that must be attacked with protocol-specific tools.

**Hash types and cracking implications**

Different hash families behave very differently under offline attack. The exam often expects you to recognize which ones are weak, which are salted, and which are intentionally slow.

| Hash / format | Typical use | Offline cracking notes |
|---|---|---|
| LM | Legacy Windows | Very weak, splits passwords, often cracked quickly if present |
| NTLM | Windows authentication | Fast to test, common target for offline cracking and pass-the-hash scenarios |
| NTLMv1 / NTLMv2 | Legacy challenge-response | Usually attacked from captured responses, not treated like a simple stored hash |
| Kerberos-related material | Active Directory auth | Tickets and responses may be attacked with specialized methods; not all are “hashes” in the simple sense |
| MD5 / SHA-1 / SHA-256 | General-purpose hashes | Fast hashes are easier to brute force if unsalted or poorly protected |
| bcrypt / scrypt / Argon2 | Password hashing | Designed to be slow and memory-hard, making offline cracking much more expensive |
| Unix `crypt` formats | Linux/Unix password storage | Strength depends on the algorithm variant and parameters |

NTLM is a common CEH focus because it is widely encountered in Windows environments and is relatively fast to test offline compared with modern password hashing schemes. By contrast, bcrypt, scrypt, and Argon2 are designed to slow guessing and increase cost per attempt.

**Why salts matter**

A salt is a unique random value stored alongside a password hash before hashing. It does not hide the password by itself, but it changes the resulting hash so that identical passwords do not produce identical stored values.

- **Salts prevent reuse of precomputed tables** across many users.
- **Unique salts** mean each password must be attacked separately, even if two users chose the same password.
- **Salts do not stop cracking**; they make cracking less efficient by removing shared precomputation.
- **Rainbow tables** become far less useful when salts are unique and sufficiently large.

A common exam trap is to overstate salts as a complete defense. They are not. They mainly defeat large-scale precomputation and make mass cracking less efficient.

**Precomputation and rainbow tables**

Rainbow tables are precomputed hash-to-password lookup structures. They were more useful when systems used fast, unsalted hashes.

- They work best against **unsalted** or weakly salted hashes.
- They are much less effective against modern password storage that uses unique salts.
- They are largely obsolete in practice for well-designed password storage because:
  - unique salts break table reuse,
  - slow KDFs (key derivation functions) increase compute cost,
  - memory-hard algorithms reduce GPU efficiency.

For CEH, remember the principle: precomputation trades storage for speed, but salts and modern password hashing remove most of that advantage.

**Offline cracking workflow**

A practical offline cracking workflow usually follows a repeatable sequence. The goal is to identify the hash, choose the right attack mode, and tune the attack based on likely password patterns.

1. **Identify the hash type**
   - Use tools such as `hashid`, `hash-identifier`, or `John the Ripper`’s format detection.
   - Confirm whether you are dealing with NTLM, bcrypt, SHA-512 crypt, or another format.

2. **Prepare the hash file**
   - Put one hash per line when the tool expects that format.
   - Remove usernames or metadata only if the cracking tool requires it.

3. **Select a wordlist**
   - Start with common lists such as `rockyou.txt` or organization-specific wordlists.
   - Add contextual lists from OSINT (open-source intelligence), such as company names, product names, seasons, years, and local events.

4. **Apply rules or mutations**
   - Use mangling rules to transform base words into likely passwords.
   - Common mutations include capitalization, suffix digits, leetspeak, and symbol appends.

5. **Try hybrid or mask attacks**
   - Use a known pattern when you have clues, such as `CompanyName` plus a year and symbol.
   - Use masks when password policy or user behavior suggests a structure.

6. **Review cracked results**
   - Validate recovered passwords against the target format and account context.
   - Check whether the password is current, reused, or only useful for historical analysis.

7. **Escalate if needed**
   - Move from dictionary to hybrid to mask to brute force only when the evidence supports it.
   - For strong hashes, prioritize smarter candidate generation over blind brute force.

**Practical tools and commands**

The following examples show common offline cracking workflows. Exact syntax can vary by tool version, so verify the help output on your system.

```bash
# Identify likely hash type
hashid hashes.txt
hash-identifier

# John the Ripper: auto-detect and crack
john --wordlist=/usr/share/wordlists/rockyou.txt hashes.txt
john --show hashes.txt

# Hashcat: NTLM example (-m 1000)
hashcat -m 1000 -a 0 hashes.txt /usr/share/wordlists/rockyou.txt

# Hashcat: mask attack for a known pattern
hashcat -m 1000 -a 3 hashes.txt 'Summer20?d?d!'

# Hashcat: apply rules to a wordlist
hashcat -m 1000 -a 0 hashes.txt /usr/share/wordlists/rockyou.txt -r rules/best64.rule
```

In hashcat, `-a 0` is straight dictionary mode, while `-a 3` is mask mode. The mask string describes the candidate pattern, and tokens such as `?d` represent digits. In the example `Summer20?d?d!`, `Summer20` and `!` are literal characters, while the two `?d` tokens stand for two unknown digits.

**Wordlists, rules, and candidate generation**

Good cracking is usually about candidate quality, not just raw speed.

- **Wordlists**
  - Use general lists for broad coverage.
  - Use targeted lists for the organization, region, or user population.
- **Rules**
  - Transform base words into realistic guesses.
  - Common rule effects include adding years, replacing letters with symbols, and changing case.
- **Combinator attacks**
  - Combine two wordlists, such as a company name plus a season or a pet name plus a year.
- **Custom generation**
  - Build lists from usernames, email prefixes, project names, and public documents.
  - Use OSINT to derive likely password themes.

Example with `crunch` to generate a bounded candidate set and save it to a file:

```bash
# Generate 10-character lowercase/digit candidates and save to a file
crunch 10 10 abcdefghijklmnopqrstuvwxyz0123456789 -o candidates.txt
```

Use `crunch` carefully. Unbounded generation can create enormous outputs and waste time and storage. In practice, you usually constrain length, charset, and output size.

**Hash extraction scenarios**

CEH exam questions often describe a compromise path rather than a direct “hash file found” event. You should recognize the likely extraction source.

- **Compromised Windows workstation**
  - Attacker dumps local hashes from `SAM`.
  - If admin rights are obtained, `LSASS` memory may reveal additional secrets.
- **Domain controller compromise**
  - Attacker targets `NTDS.dit` to obtain domain credential material.
  - This is high impact because it can expose many accounts at once.
- **Phishing followed by privilege escalation**
  - Initial access may come from a user password.
  - Later, memory scraping or token theft may expose more valuable material.
- **Legacy server exposure**
  - Weak NTLM or LM remnants may still exist on older systems.
  - These are often easier to crack than modern password hashes.
- **Web application breach**
  - Database dumps may contain salted password hashes.
  - The cracking difficulty depends on the hash algorithm and salt design.

**Pass-the-hash versus offline cracking**

These are related but distinct concepts.

- **Offline cracking**
  - You recover the password by testing guesses against a captured hash or response.
  - Success gives you the actual password, which may work across services if reused.
- **Pass-the-hash**
  - You use the hash itself as an authentication secret in protocols that accept it.
  - You may gain access without ever learning the plaintext password.

This distinction matters because a cracked password can unlock many systems if users reuse it, while pass-the-hash is often limited to specific Windows authentication contexts and compatible services.

**Example attack decision tree**

When you see a CEH scenario, use the evidence to choose the attack.

- If the target is a live login page with rate limits, think **online attack**.
- If you have `NTDS.dit`, `SAM`, or a database dump, think **offline cracking**.
- If you have an NTLM challenge-response capture, think **protocol-specific cracking** or **pass-the-hash** depending on the objective.
- If the password policy is known, think **mask attack** or **hybrid attack**.
- If the organization uses MFA, remember it reduces account takeover value but does not protect stolen hash material.

**Defensive controls**

Defenses should make offline cracking expensive and reduce the value of any recovered password.

- **Use modern password hashing**
  - Prefer Argon2, bcrypt, or scrypt with strong parameters.
  - Tune work factors so verification is acceptable but guessing is costly.
- **Salt every password uniquely**
  - Store a unique salt per password.
  - Do not reuse salts across accounts.
- **Consider a pepper**
  - A pepper is a secret value stored separately from the password database.
  - It can help if the database is stolen alone, but it creates operational risk because losing the pepper can break verification.
- **Enforce strong password policy**
  - Encourage long passphrases over short complex strings.
  - Block common and breached passwords.
- **Screen against breached-password lists**
  - Reject passwords known from public breach corpora.
  - This reduces the chance of easy offline cracking success.
- **Protect credential stores**
  - Restrict access to `SAM`, `NTDS.dit`, backups, and secrets stores.
  - Harden endpoints to reduce `LSASS` dumping and credential theft.
- **Reduce password reuse**
  - Use MFA, password managers, and user education.
  - MFA does not secure the hash file, but it reduces the usefulness of a cracked password for account access.
- **Monitor for dumping behavior**
  - Alert on suspicious access to credential stores, LSASS access, and domain controller database extraction.

**Pepper tradeoffs**

A pepper can improve resilience, but it is not a universal fix.

- It helps when the password database is stolen without the application secret.
- It complicates rotation and recovery if the pepper is exposed.
- It should be stored separately, ideally in a hardware security module (HSM) or secret manager.
- It does not replace salting or a strong password hashing algorithm.

**CEH-style exam cues**

Look for wording that hints at the correct attack class.

- “Captured `NTDS.dit`” usually points to **offline cracking** of Active Directory material.
- “Too many failed logins would trigger lockout” usually points away from online brute force.
- “The attacker uses the hash directly to authenticate” suggests **pass-the-hash**.
- “The attacker has a password dump from a breach” suggests **dictionary, hybrid, or mask attacks**.
- “The password policy is `SeasonYear!` style” suggests a **mask attack** or **targeted hybrid attack**.
- “The hash is bcrypt/Argon2” suggests **slower cracking** and a need for better candidate selection.

**Quick comparison of attack choices**

| Situation | Best first approach |
|---|---|
| Known company naming pattern | Hybrid or combinator attack |
| Password policy with fixed structure | Mask attack |
| Large breach dump with unknown patterns | Dictionary plus rules |
| Strong salted KDF hashes | Targeted candidates, not blind brute force |
| Live login with lockout | Password spraying or social engineering, not brute force |
| Captured NTLM material | Protocol-specific offline cracking or pass-the-hash analysis |

**Practical lab mindset**

When you practice, focus on the full workflow rather than a single command.

1. Identify the source of the credential material.
2. Determine whether it is a stored hash, challenge-response, or ticket.
3. Match the hash type to the correct tool and mode.
4. Start with the most likely candidates from wordlists and OSINT.
5. Add rules, masks, or combinators based on password policy clues.
6. Check results carefully and avoid assuming every recovered password is still valid.
7. Document the impact, including reuse risk and lateral movement potential.

**Key takeaways**

- Offline cracking works on captured credential material without repeatedly querying the target.
- The hash type matters more than raw speed alone.
- Salts defeat precomputation, not cracking itself.
- Modern password hashing algorithms slow guessing and raise attacker cost.
- Pass-the-hash is different from offline cracking.
- The best cracking strategy combines identification, targeted candidate generation, and realistic attack selection.
