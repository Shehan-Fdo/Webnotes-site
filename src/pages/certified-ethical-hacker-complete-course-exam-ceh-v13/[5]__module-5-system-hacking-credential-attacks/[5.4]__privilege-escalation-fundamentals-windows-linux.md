## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

Privilege escalation is the process of moving from a lower-privileged context to a higher-privileged one on the same host. In CEH-style assessments, you usually look for two broad classes of opportunities:

- **Misconfiguration-based escalation**: weak permissions, unsafe services, writable scripts, exposed credentials, or dangerous policy settings.
- **Exploit-based escalation**: kernel or service vulnerabilities that allow local code execution with elevated rights.

The practical workflow is always the same: enumerate, identify a path, validate the impact, and then confirm the privilege boundary you crossed. On Windows, that often means moving from a standard user to an administrator context or from an administrator context to `NT AUTHORITY\SYSTEM`. On Linux, it often means moving from a regular user to `root`.

**Core mindset**

Privilege escalation is rarely a single trick. It is usually a chain:

1. Find a weak control point.
2. Confirm you can influence execution, configuration, or credentials.
3. Trigger code execution in a higher-privileged context.
4. Verify the resulting token, UID, or group membership.

A good exam answer explains both the *mechanism* and the *condition that makes it possible*.

**Windows privilege model**

Windows access control is built around access tokens, privileges, integrity levels, and discretionary access control lists (DACLs). A user process inherits a token that defines what the process can do. Higher integrity does not automatically mean full administrative power, and administrative users may still run with filtered tokens because of User Account Control (UAC).

Key terms:

- **Access token**: the security context attached to a process or thread.
- **Privilege**: a special right such as `SeDebugPrivilege` or `SeImpersonatePrivilege`.
- **Integrity level**: a trust label such as Low, Medium, High, or System.
- **DACL**: a list that controls who can access an object.

Useful checks:

```powershell
whoami
whoami /groups
whoami /priv
whoami /all
```

You should also inspect local administrators, services, scheduled tasks, and installed software:

```powershell
net localgroup administrators
sc query type= service state= all
schtasks /query /fo LIST /v
wmic product get name,version
```

**Windows escalation categories**

Windows privilege escalation usually falls into one of these buckets:

- **Service misconfiguration**
- **Scheduled task abuse**
- **Unquoted service path issues**
- **Weak file or registry permissions**
- **Credential exposure**
- **UAC and auto-elevation abuse**
- **Token impersonation**
- **Kernel or driver exploitation**

**Service misconfiguration**

A service runs with a fixed account, often `LocalSystem`, `LocalService`, or a domain account. If you can modify the service binary path, its arguments, or the executable it launches, you may be able to execute code as that service account.

Common checks:

```powershell
sc qc VulnService
wmic service get name,displayname,pathname,startmode
accesschk.exe -uwcqv "Authenticated Users" *
```

Look for:

- Writable service binaries
- Writable service directories
- Weak service DACLs
- Services that run as `LocalSystem`
- Restartable services that you can trigger after replacement

If the service binary is writable, you can often replace it with a payload and restart the service. If only the directory is writable, you may be able to drop a malicious DLL or replace a helper executable depending on how the service loads components.

**Unquoted service paths**

An unquoted service path becomes dangerous when the executable path contains spaces and is not wrapped in quotes. Windows parses the command line by tokenizing on spaces, so it may attempt to resolve the first token as the executable path. For example, a path like:

```text
C:\Program Files\Vendor App\service.exe
```

if unquoted, can create opportunities for execution of `C:\Program.exe` or other earlier space-delimited prefixes *if those files exist and the service configuration causes that resolution to be used*. The exact behavior depends on the service command line and how Windows resolves the executable versus arguments, so do not oversimplify it as “Windows tries every folder.”

What to check:

```powershell
wmic service get name,displayname,pathname | findstr /i /v "\""
```

If the service runs as a privileged account and you can write to one of the candidate locations, you may be able to plant a malicious executable with the expected name.

**Scheduled task abuse**

Scheduled tasks can run with elevated privileges, especially when configured to run as `SYSTEM` or a privileged user. Weak permissions on the task definition, its action path, or the target executable can create escalation opportunities.

Check tasks and their security descriptors:

```powershell
schtasks /query /fo LIST /v
Get-ScheduledTask | Select-Object TaskName,TaskPath,State
```

Look for:

- Writable task XML or task folder permissions
- Actions that point to writable executables or scripts
- Tasks that run as `SYSTEM`
- Triggers you can wait for or manually invoke

A common abuse pattern is replacing the executable referenced by the task action. Another is editing a writable task definition to point to your payload. If the task is protected but the target binary is writable, the task still becomes a launch vector.

**UAC, auto-elevation, and trusted binary abuse**

UAC is a consent and filtering mechanism, not a full boundary in every case. An already-administrative user may have a filtered token at medium integrity and a linked elevated token at high integrity. Some techniques abuse trusted binaries, COM objects, or registry settings to trigger elevated execution without a normal prompt.

Be precise:

- UAC bypasses usually require an account that is already in the local administrators group.
- They do not turn a standard user into an administrator.
- They may still produce elevated code execution inside an admin context.

Common abuse patterns include:

- **COM hijacking**: redirecting a COM class registration to a malicious handler when a trusted binary activates it.
- **Registry hijacking**: modifying per-user registry keys that trusted auto-elevated binaries consult.
- **Trusted binary abuse**: leveraging signed Microsoft binaries or installers that execute child processes or scripts in a privileged context.

Examples of registry-based checks:

```powershell
reg query HKCU\Software\Classes\ms-settings\Shell\Open\command
reg query HKCU\Software\Classes\CLSID
```

> UAC bypass techniques are highly version- and patch-dependent. Treat them as context-specific opportunities, not universal admin-to-SYSTEM methods.

**Token abuse and impersonation**

Windows tokens can sometimes be abused when a process holds privileges such as `SeImpersonatePrivilege` or `SeAssignPrimaryTokenPrivilege`. These privileges do not guarantee escalation by themselves; you still need a suitable impersonation opportunity, such as a service, named pipe, RPC, or COM interaction that lets you capture or impersonate a higher-privileged token.

Important nuance:

- `SeImpersonatePrivilege` is often useful when a privileged service authenticates to your process or pipe.
- `SeAssignPrimaryTokenPrivilege` matters when a process can assign a primary token to a new process.
- The exploit path depends on local conditions and available services.

Useful checks:

```powershell
whoami /priv
```

Common exam-relevant examples include service-based token impersonation chains and named-pipe style abuse. The exact exploit depends on the environment, so focus on recognizing the privilege and the prerequisite interaction.

**Credential-based escalation on Windows**

Sometimes escalation is not about code execution first; it is about finding credentials that unlock a higher-privileged session.

Common sources:

- Saved credentials in Credential Manager
- `runas /savecred` usage
- Unattended installation files
- Configuration files with plaintext passwords
- PowerShell history
- Registry-stored secrets
- LSASS memory, if you already have sufficient rights to dump it

Useful locations and checks:

```powershell
cmdkey /list
dir C:\Users\*\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\
dir C:\Windows\Panther\
dir C:\Windows\System32\Sysprep\
```

Examples of credential reuse:

```powershell
runas /user:Administrator /savecred cmd.exe
```

That command is only useful if credentials were previously saved for that target. It is not a universal bypass.

**Windows registry and COM persistence-style escalation paths**

Some escalation paths rely on hijacking how Windows resolves objects or launches helper components.

Look for:

- Writable registry keys under `HKCU\Software\Classes`
- Per-user COM registrations that override machine-wide behavior
- Auto-elevated binaries that consult user-controlled registry values
- DLL search order issues in privileged processes

A practical example is a per-user COM hijack where a trusted process loads a class registration from the current user hive instead of the machine hive. Another is a registry key that points a trusted binary to a command or handler you control.

**Windows scheduled and service-based validation**

When you suspect a service or task path, validate it carefully before attempting exploitation:

1. Confirm the run account.
2. Confirm whether the binary or script path is writable.
3. Confirm whether the service or task can be restarted or triggered.
4. Confirm the resulting privilege level after execution.

Useful commands:

```powershell
icacls "C:\Program Files\Vendor App"
icacls "C:\Path\To\Target.exe"
sc stop VulnService
sc start VulnService
schtasks /run /tn "\Vendor\UpdateTask"
```

**Linux privilege model**

Linux privilege escalation usually means moving from a regular user to `root`, but the path may also involve group membership, capabilities, or service accounts. Linux authorization depends on file permissions, ownership, capabilities, sudo policy, and service configuration.

Useful baseline checks:

```bash
id
whoami
groups
sudo -l
uname -a
cat /etc/os-release
```

You should always identify:

- Kernel version
- Distribution and patch level
- Sudo rules
- Writable scripts, timers, and services
- SUID/SGID binaries
- File capabilities
- Cron jobs and systemd units

**Sudo misconfigurations**

`sudo` is one of the most important Linux escalation surfaces. The `sudoers` file can allow command execution as root with or without a password.

Common risky patterns:

- `NOPASSWD` rules
- Wildcard arguments that can be abused
- `SETENV` or preserved environment variables
- Overly broad command allowances
- Unsafe `secure_path` assumptions
- Commands that can spawn shells, edit files, or run interpreters

Check your rights:

```bash
sudo -l
```

Examples of dangerous patterns:

```text
user ALL=(ALL) NOPASSWD: /usr/bin/vim
user ALL=(root) NOPASSWD: /usr/bin/less
user ALL=(root) NOPASSWD: /usr/bin/find
user ALL=(root) SETENV: /usr/bin/python3
```

Why these matter:

- Editors and pagers can often spawn shells or write files.
- `find`, `awk`, `perl`, `python`, and similar tools may allow command execution depending on flags and version.
- `SETENV` can let you influence library loading or command behavior if the rule permits it.

Be careful with version dependence. A command that is exploitable in one environment may be restricted or patched in another.

**SUID and SGID binaries**

SUID (Set User ID) and SGID (Set Group ID) binaries run with the file owner’s or group’s privileges. A root-owned SUID binary is a classic escalation target.

Find them:

```bash
find / -perm -4000 -type f 2>/dev/null
find / -perm -2000 -type f 2>/dev/null
```

Look for:

- Unusual custom binaries
- Interpreters or helper tools with SUID
- Binaries that call external programs
- Binaries with unsafe file handling

A SUID binary is not automatically exploitable. You need a behavior that lets you read, write, execute, or spawn in a privileged context. For example, a SUID program that invokes a shell, loads plugins, or writes to predictable files may be abused.

**Linux file capabilities**

File capabilities grant specific privileges to executables without making them fully SUID. Common capabilities include `cap_setuid`, `cap_setgid`, `cap_dac_read_search`, and `cap_net_raw`.

Check them with:

```bash
getcap -r / 2>/dev/null
```

Important nuance:

- `cap_setuid` does not automatically mean you can get root.
- The binary must actually use the capability in a way that changes UID or otherwise enables privileged behavior.
- Some interpreters or helper binaries become dangerous only if they can invoke `setuid(0)` or launch a shell after changing identity.

Example:

```bash
/usr/bin/python3 cap_setuid+ep
```

If the binary or interpreter can call `os.setuid(0)` and then spawn a shell, escalation may be possible. If it cannot, the capability may be less useful than it first appears.

**Linux PATH hijacking and writable scripts**

PATH hijacking works when a privileged script or service calls a command without an absolute path, and you can control the search path or place a malicious binary earlier in the lookup order.

Typical conditions:

- A root-owned script uses `tar`, `cp`, `sh`, `service`, or another command without a full path.
- You can write to a directory in the PATH used by that script.
- The script preserves or inherits your manipulated environment.

Example check:

```bash
echo $PATH
strings /path/to/script | grep -E '(^|[^/])(cp|tar|sh|python|perl)([^/]|$)'
```

A simple abuse pattern is to create a fake command in a writable directory and ensure the privileged script resolves it first. This is often paired with `sudo` rules that preserve environment variables or with poorly written maintenance scripts.

**Cron jobs, systemd services, and timers**

Linux scheduled execution is a major escalation category. Cron jobs and systemd timers/services often run as root and may reference writable scripts or directories.

Cron checks:

```bash
cat /etc/crontab
ls -la /etc/cron.*
crontab -l
```

Systemd checks:

```bash
systemctl list-timers --all
systemctl list-units --type=service
systemctl cat some-service.service
```

Look for:

- Writable cron scripts
- Root-owned jobs that execute files in writable directories
- Systemd unit files with writable `ExecStart`
- Drop-in overrides in writable locations
- Timer units that trigger privileged services

Example of a vulnerable systemd unit:

```ini
[Unit]
Description=Backup Job

[Service]
Type=oneshot
ExecStart=/opt/backup/backup.sh
```

If `/opt/backup/backup.sh` is writable by your user or group, you can replace it with a payload and wait for the service or timer to run. If the unit file itself is writable, you may be able to change `ExecStart` directly.

Useful validation:

```bash
ls -l /etc/systemd/system/
ls -l /opt/backup/backup.sh
systemctl status backup.service
journalctl -u backup.service --no-pager
```

**Linux writable service and init script abuse**

Older systems may still use SysV init scripts, and many environments still have service wrappers or custom startup scripts. If a root-run script is writable, it is a direct escalation path.

Check:

```bash
ls -l /etc/init.d/
ls -l /etc/systemd/system/
find /etc/systemd -type f -writable 2>/dev/null
```

Common abuse patterns:

- Writable init script executed at boot or by service management
- Writable helper script called by a root service
- Writable environment file sourced by a service
- Writable drop-in override that changes the service command

**Linux kernel and local exploit fundamentals**

Kernel exploits are a major escalation category when misconfigurations are absent or insufficient. For CEH prep, you should know when to consider them, even if you do not exploit them in detail.

Start with version and patch identification:

```bash
uname -r
cat /proc/version
hostnamectl
```

Then compare the kernel and distribution against known local privilege escalation classes:

- Dirty Pipe-style write primitives
- OverlayFS and namespace-related issues
- `polkit`-adjacent local escalation bugs
- `sudo` or `pkexec`-related vulnerabilities
- Driver or module flaws
- Container escape paths that lead to host privilege gain

Use kernel exploits only when:

- The host is patched poorly or not at all.
- Misconfiguration paths are absent.
- You have a reliable exploit matched to the exact kernel and distro build.

> Kernel exploits are high-risk and version-sensitive. For exam purposes, recognize the category and the validation steps before assuming exploitability.

**Linux credential exposure**

Credential hunting is often faster than kernel exploitation. Look for secrets in:

- Shell history files
- Application config files
- Backup archives
- Deployment scripts
- SSH keys
- Environment files
- Database connection strings

Examples:

```bash
find /home -name ".bash_history" -o -name ".zsh_history" 2>/dev/null
find / -name "*.bak" -o -name "*.old" -o -name "*.save" 2>/dev/null
grep -RniE "password|passwd|secret|token|key" /etc /opt /var/www 2>/dev/null
```

If you find reusable credentials, test them carefully against `sudo`, SSH, service accounts, or local admin-equivalent groups.

**Practical enumeration workflow**

A disciplined workflow helps you avoid missing the obvious.

1. Identify your current context.
   - `whoami`, `id`, `whoami /priv`, `whoami /groups`
2. Enumerate local misconfigurations.
   - Services, tasks, cron, sudoers, SUID, capabilities, writable paths
3. Hunt for credentials.
   - History files, configs, saved creds, registry, unattended files
4. Check version-sensitive exploit classes.
   - Kernel version, service versions, patch level
5. Validate the path.
   - Confirm write access, trigger conditions, and resulting privilege

A compact Linux triage command set:

```bash
id
sudo -l
find / -perm -4000 -type f 2>/dev/null
getcap -r / 2>/dev/null
systemctl list-timers --all
cat /etc/crontab
uname -r
```

A compact Windows triage command set:

```powershell
whoami /all
whoami /priv
net localgroup administrators
schtasks /query /fo LIST /v
wmic service get name,displayname,pathname,startmode
cmdkey /list
```

**Common exam distinctions**

- **Standard user to admin**: usually requires a local misconfiguration, credential reuse, or an exploit.
- **Admin to SYSTEM on Windows**: often involves service abuse, token impersonation, scheduled tasks, or a trusted auto-elevation path.
- **User to root on Linux**: often involves sudo misconfigurations, SUID/capabilities, writable scripts, cron/systemd abuse, or a kernel exploit.
- **Misconfiguration vs exploit**: misconfiguration is usually more stable and preferred; exploit paths are more version-dependent.

**What to remember for CEH**

- Always start with enumeration before attempting exploitation.
- Treat UAC as a filtering mechanism, not a universal privilege boundary.
- Verify whether a Windows service path, scheduled task, or registry key is actually writable before assuming escalation.
- On Linux, check `sudo -l`, SUID, capabilities, cron, and systemd before moving to kernel exploits.
- Be precise about version dependence: many classic techniques work only in specific configurations or older builds.
