## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

Persistence, lateral movement, and credential abuse are core post-compromise concepts. After initial access, an attacker tries to stay in the environment, expand reach, and use trusted identities to avoid detection. For CEH-style questions, you should distinguish the goal of each phase:

- **Persistence** keeps access across reboots, logouts, or remediation gaps.
- **Lateral movement** expands from one host to another inside the network.
- **Credential abuse** uses stolen, reused, or forged credentials to impersonate legitimate users or services.

These techniques often overlap in real intrusions. A single campaign may use a phishing foothold, dump credentials, create a scheduled task for persistence, then move laterally with remote administration tools.

**Persistence Concepts**

Persistence is any technique that helps an attacker regain or maintain access after the initial foothold would otherwise be lost. It is not limited to malware; legitimate system features can be abused for long-term access.

Common persistence mechanisms include:

- **Startup execution**
  - Windows: Run keys, Startup folders, services, scheduled tasks, WMI event subscriptions.
  - Linux: `cron`, `systemd` services, shell profile files such as `~/.bashrc`, `~/.profile`, `~/.zshrc`.
  - macOS: LaunchAgents, LaunchDaemons, login items, `cron` on older systems, shell profile files such as `~/.zshrc` and `~/.bash_profile`.
- **Account persistence**
  - Creating new local or domain accounts.
  - Adding users to privileged groups.
  - Enabling dormant accounts or resetting passwords.
- **Remote access persistence**
  - Backdoors, reverse shells, web shells, SSH authorized keys, remote management agents.
- **Application persistence**
  - Modifying application startup scripts, plugins, or service wrappers.
  - Abusing scheduled jobs in enterprise software.
- **Web persistence**
  - Web shell deployment on a compromised server.
  - Session or browser-based persistence through stolen cookies, tokens, or malicious browser extensions.

A useful exam distinction is that persistence is about surviving time and cleanup, not necessarily moving to other systems. An attacker may persist on a single host for weeks without lateral movement.

**Windows Persistence Examples**

Windows offers many built-in persistence points because it is heavily service-driven and registry-based.

- **Registry Run keys**
  - `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`
  - `HKLM\Software\Microsoft\Windows\CurrentVersion\Run`
- **Scheduled tasks**
  - Run at logon, startup, or on a timer.
- **Services**
  - A malicious service can start automatically at boot.
- **WMI event subscriptions**
  - Useful for stealthy trigger-based execution.
- **Startup folder**
  - User or all-users startup locations.

Example commands:

```powershell
# Create a scheduled task that runs at logon
schtasks /create /tn "Updater" /tr "C:\Users\Public\updater.exe" /sc onlogon /ru SYSTEM

# Add a Run key for current user
reg add "HKCU\Software\Microsoft\Windows\CurrentVersion\Run" /v OneDriveSync /t REG_SZ /d "C:\Users\Public\sync.exe" /f

# Create a service
sc create WinUpdateSvc binPath= "C:\Windows\Temp\svc.exe" start= auto
```

For defenders, these artifacts are often visible in autoruns tools, event logs, and endpoint detection and response (EDR) telemetry. For exam purposes, remember that services and scheduled tasks are common because they survive reboots and can run with elevated privileges.

**Linux and Unix-Like Persistence**

On Linux, persistence often relies on startup scripts, service managers, and user shell initialization files.

Common examples:

- **`cron` jobs**
  - System-wide: `/etc/crontab`, `/etc/cron.d/`
  - User-specific: `crontab -e`
- **`systemd` services**
  - User or system services can auto-start at boot or login.
- **Shell profiles**
  - `~/.bashrc`, `~/.bash_profile`, `~/.profile`, `~/.zshrc`
- **SSH keys**
  - Adding a key to `~/.ssh/authorized_keys`
- **Init scripts**
  - Older systems may still use `/etc/init.d/`

Example workflow:

```bash
# Add a cron job for the current user
crontab -e

# Example entry: run every 5 minutes
*/5 * * * * /usr/local/bin/backup-helper.sh >/dev/null 2>&1

# Create a systemd service
sudo tee /etc/systemd/system/update-check.service >/dev/null <<'EOF'
[Unit]
Description=Update Check

[Service]
ExecStart=/usr/local/bin/update-check.sh
Restart=always

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl enable update-check.service
sudo systemctl start update-check.service
```

On Linux, persistence is often easier to hide in plain sight because administrators expect cron jobs and services. That makes naming, file location, and execution context important indicators.

**macOS Persistence**

macOS persistence is commonly associated with LaunchAgents and LaunchDaemons, but it also includes login items, shell profiles, and application support mechanisms.

Common mechanisms:

- **LaunchAgents**
  - Run in the user context after login.
  - Common path: `~/Library/LaunchAgents/`
- **LaunchDaemons**
  - Run at boot as root or system context.
  - Common path: `/Library/LaunchDaemons/`
- **Login items**
  - User-approved startup items in the GUI.
- **Shell profiles**
  - `~/.zshrc`, `~/.zprofile`, `~/.bash_profile`
- **Application support abuse**
  - Malicious components placed in app-specific support directories.
- **Browser extensions or profiles**
  - Can survive user sessions if installed persistently.

Example LaunchAgent plist:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
  <dict>
    <key>Label</key>
    <string>com.example.updater</string>
    <key>ProgramArguments</key>
    <array>
      <string>/Users/shared/updater.sh</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
  </dict>
</plist>
```

Load it with:

```bash
launchctl load ~/Library/LaunchAgents/com.example.updater.plist
```

For CEH-style understanding, macOS persistence is important because it differs from Windows registry-based methods and Linux cron/systemd patterns, but the objective is the same: automatic execution after reboot or login.

**Web Shell and Browser/Session Persistence**

Web shell persistence is distinct from general web application compromise. A **web shell** is a server-side script that gives remote command execution through HTTP or HTTPS. Attackers often upload it to a writable web directory or abuse file upload flaws.

Typical web shell examples include:

- `shell.php` on PHP servers
- `cmd.aspx` on ASP.NET servers
- `shell.jsp` on Java-based applications

A simple PHP web shell pattern:

```php
<?php
if (isset($_GET['cmd'])) {
    system($_GET['cmd']);
}
?>
```

Browser and session persistence are different from server-side web shells:

- **Session cookie theft**
  - Reusing a valid session cookie can bypass password prompts until the session expires or is revoked.
- **Token theft**
  - OAuth access tokens, refresh tokens, and API tokens can remain valid even after password changes.
- **Browser profile abuse**
  - Malicious extensions, saved credentials, or synced browser profiles can preserve access.
- **Session fixation or hijacking**
  - The attacker leverages an existing authenticated session rather than reauthenticating.

These methods matter in cloud and SaaS environments because access is often governed by tokens rather than only passwords. A stolen browser session may provide direct access to email, collaboration tools, or admin consoles without triggering a password reset.

**Credential Abuse Concepts**

Credential abuse means using legitimate credentials, tokens, or secrets in ways the owner did not intend. It is one of the most common post-compromise techniques because it blends into normal authentication traffic.

Common credential abuse patterns include:

- **Password reuse**
  - Reusing credentials from one service on another.
- **Credential stuffing**
  - Trying known username/password pairs at scale.
- **Pass-the-hash**
  - Using NTLM hash material without knowing the plaintext password.
- **Pass-the-ticket**
  - Reusing Kerberos tickets in Windows environments.
- **Token theft**
  - Stealing bearer tokens, refresh tokens, or session cookies.
- **API key abuse**
  - Reusing exposed keys for cloud services, CI/CD systems, or third-party APIs.
- **OAuth abuse**
  - Using stolen OAuth consent grants, refresh tokens, or malicious app registrations.
- **Service account abuse**
  - Exploiting overprivileged machine, app, or automation accounts.

In cloud and SaaS environments, attackers often target:

- **API keys**
  - Hardcoded in source code, CI logs, or environment variables.
- **OAuth tokens**
  - Long-lived refresh tokens can outlast password changes.
- **Session cookies**
  - Browser session theft can bypass MFA if the session is already trusted.
- **Federated identities**
  - Compromising SSO-linked accounts can open multiple services at once.

A practical exam takeaway: credentials are not only passwords. Any reusable authentication artifact can be abused.

**Common Credential-Dumping Tools and Workflows**

Credential dumping is the extraction of secrets from memory, disk, or authentication stores. You should recognize common tools and the general workflow, even if exact commands vary by environment and privilege level.

Common Windows tools include:

- **Mimikatz**
  - Extracts credentials, tickets, and hashes from memory and security packages.
- **ProcDump**
  - Can dump process memory for offline analysis.
- **LSASS-focused tools**
  - Target the Local Security Authority Subsystem Service process.
- **SecretsDump**
  - Commonly used to extract NTLM hashes, Kerberos keys, and cached secrets from remote or offline sources.
- **LaZagne**
  - Targets stored credentials in browsers and applications.

Typical lab-style workflow:

1. Gain local administrator or SYSTEM-level access.
2. Dump memory or authentication material from `lsass.exe` or equivalent stores.
3. Extract hashes, tickets, or plaintext credentials.
4. Reuse the material for remote authentication or privilege escalation.
5. Pivot to additional hosts or services.

Example recognition commands:

```powershell
# Enumerate logged-on sessions and local groups
whoami /all
net user
net localgroup administrators

# Dump LSASS with a built-in tool for analysis in a lab
procdump -ma lsass.exe lsass.dmp
```

> Dumping credentials from protected processes can trigger EDR alerts, require elevated privileges, and be blocked by protections such as Credential Guard or protected process light.

**Lateral Movement Concepts**

Lateral movement is the act of moving from one compromised system to another inside the target environment. It is different from persistence because the goal is expansion, not survival.

Common lateral movement methods include:

- **Remote service execution**
  - Windows: PsExec, WMI, WinRM, SMB-based execution.
  - Linux: SSH with stolen keys or passwords.
- **Remote desktop access**
  - RDP, VNC, remote support tools.
- **Admin shares and file copy**
  - Copy payloads to `C$`, `ADMIN$`, or shared directories.
- **Remote scheduled tasks**
  - Create tasks on another host using administrative credentials.
- **Remote management protocols**
  - PowerShell Remoting, SSH, WinRM, RPC, SNMP in some environments.

Example Windows lateral movement commands:

```powershell
# Remote command execution via WinRM
Enter-PSSession -ComputerName WS-02 -Credential domain\adminuser

# Copy a file to an admin share
copy .\tool.exe \\WS-02\C$\Windows\Temp\tool.exe

# Create a remote service with PsExec-style behavior
psexec \\WS-02 -u domain\adminuser -p 'Password123!' cmd.exe
```

Example Linux lateral movement:

```bash
# SSH using a stolen key
ssh -i ~/.ssh/id_rsa user@10.10.10.25

# Copy a file and execute remotely
scp payload.sh user@10.10.10.25:/tmp/
ssh user@10.10.10.25 'bash /tmp/payload.sh'
```

Lateral movement usually requires valid credentials, trust relationships, or a reachable management plane. It is often noisy because it touches multiple hosts and generates authentication events.

**Pivoting vs Tunneling vs Lateral Movement**

These terms are often confused on exams, so keep them separate.

| Concept | Meaning | Typical Goal | Example |
|---|---|---|---|
| Pivoting | Using a compromised host as a stepping stone to reach another network | Reach otherwise inaccessible systems | SSH to Host A, then connect from Host A to Host B |
| Tunneling / Port forwarding | Encapsulating traffic through a channel to bypass network restrictions | Transport traffic through a controlled path | `ssh -L 8080:10.0.0.5:80 user@jumpbox` |
| Lateral movement | Moving from one internal host to another after compromise | Expand control across the environment | Use stolen admin creds to access another workstation |

A simple mental model:

- **Pivoting** is about path.
- **Tunneling** is about transport.
- **Lateral movement** is about compromise expansion.

Example SSH local port forward:

```bash
ssh -L 8080:10.0.0.5:80 user@jumpbox
```

This creates a tunnel from your machine to an internal web server through `jumpbox`. That is not, by itself, lateral movement unless you also compromise the internal host.

**Credential Abuse in Windows Environments**

Windows environments are especially susceptible to credential reuse because of domain trust, Kerberos, NTLM, and administrative delegation.

Key concepts:

- **NTLM hashes**
  - Can sometimes be reused directly in pass-the-hash attacks.
- **Kerberos tickets**
  - Can be stolen and reused in pass-the-ticket attacks.
- **LSASS memory**
  - Often contains credentials, tickets, and cached secrets.
- **Domain admin reuse**
  - A single privileged account used across many systems increases blast radius.
- **Local admin reuse**
  - Shared local administrator passwords enable lateral movement.

Useful commands for enumeration:

```powershell
whoami /groups
net user /domain
net group "Domain Admins" /domain
klist
```

If you see the same local administrator password across many endpoints, that is a strong indicator of lateral movement risk. Tools such as Microsoft LAPS or Windows LAPS reduce this risk by randomizing local admin passwords per machine.

**Credential Abuse in Cloud and SaaS**

Cloud and SaaS environments shift the target from passwords alone to tokens, keys, and federated identities.

Common abuse patterns:

- **API key reuse**
  - Keys exposed in repositories, build logs, or configuration files are reused against cloud APIs.
- **OAuth token abuse**
  - Stolen refresh tokens can mint new access tokens without the user re-entering credentials.
- **Session cookie theft**
  - Browser sessions for email, CRM, or admin portals can be replayed.
- **Service principal abuse**
  - Overprivileged app identities in cloud platforms can be used for automation or data access.
- **CI/CD secret theft**
  - Pipeline secrets can expose cloud credentials, signing keys, or deployment tokens.

Practical examples of where to look:

- `.env` files
- Git history and pull requests
- Build artifacts and logs
- Browser password stores
- Cloud metadata services
- Secret managers with weak access controls

For exam prep, remember that cloud compromise often starts with a secret in code or a token in a browser, not a traditional password spray.

**Attack Chain Example**

A realistic intrusion chain may look like this:

1. The attacker gains initial access through phishing or a vulnerable web app.
2. They establish persistence with a scheduled task or LaunchAgent.
3. They dump credentials from memory or browser stores.
4. They abuse a reused admin password or stolen token.
5. They move laterally to file servers, domain controllers, or cloud consoles.
6. They maintain access with additional backdoors, service accounts, or token refresh abuse.

This chain shows why the three topics are connected. Persistence keeps the foothold alive, credential abuse opens trusted doors, and lateral movement spreads the compromise.

**Detection and Defensive Indicators**

You should recognize common signs of each technique.

- **Persistence indicators**
  - New autoruns, services, cron jobs, LaunchAgents, or startup items.
  - Unexpected changes to shell profiles or login scripts.
  - New web shells in application directories.
- **Lateral movement indicators**
  - Unusual remote service creation.
  - Authentication from atypical hosts or at odd times.
  - SMB, WinRM, RDP, SSH, or WMI activity across many endpoints.
- **Credential abuse indicators**
  - Multiple failed logins followed by success.
  - Logins from impossible travel locations.
  - Token reuse from new devices or user agents.
  - Access to cloud APIs from unfamiliar IPs.

Useful defensive commands:

```powershell
# Windows: inspect scheduled tasks
schtasks /query /fo LIST /v

# Windows: inspect services
sc query type= service state= all

# Linux: inspect cron
crontab -l
sudo ls -la /etc/cron.d/

# macOS: inspect LaunchAgents
ls -la ~/Library/LaunchAgents/
launchctl list
```

**Exam Tips**

- **Persistence** means staying in place over time.
- **Lateral movement** means moving to other systems.
- **Credential abuse** means using trusted authentication material.
- **Pivoting** and **tunneling** are transport concepts, not always compromise expansion.
- **Web shells** are server-side persistence on web hosts.
- **Session cookies, OAuth tokens, and API keys** are credentials too.
- **macOS** persistence commonly uses LaunchAgents and LaunchDaemons.
- **Linux** persistence commonly uses cron, systemd, and SSH keys.
- **Windows** persistence commonly uses services, scheduled tasks, and registry Run keys.
