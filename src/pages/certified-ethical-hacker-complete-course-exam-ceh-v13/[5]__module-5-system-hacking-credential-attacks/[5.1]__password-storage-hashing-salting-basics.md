**Password storage fundamentals**

Passwords should never be stored in plaintext. A secure system stores a one-way representation of the password so that the original secret is not directly recoverable from the database. The goal is to make offline cracking expensive even if an attacker steals the credential store.

- **Plaintext storage** is a critical failure because any database leak immediately exposes user passwords.
- **Hashing** converts a password into a fixed-length digest using a one-way function.
- **Verification** compares a newly computed hash of the candidate password against the stored hash, rather than decrypting anything.
- **Password hashing** is different from general-purpose hashing because it is intentionally slow and resistant to brute force.

A password storage design should assume the database will eventually be exposed. That mindset drives the use of slow, salted, memory-hard algorithms and careful operational controls.

**Hashing versus encryption**

Hashing and encryption solve different problems. Encryption is reversible with a key; hashing is designed to be irreversible in practice. For password storage, reversibility is a liability, so encryption is usually the wrong primitive.

- **Encryption** is appropriate when you must recover the original data later.
- **Hashing** is appropriate when you only need to verify a secret.
- **Key management** becomes a separate risk if you encrypt passwords, because the decryption key becomes a high-value target.
- **One-way design** reduces the impact of a database breach.

Common password hashing algorithms include `bcrypt`, `scrypt`, `Argon2`, and, in some legacy systems, `PBKDF2` (Password-Based Key Derivation Function 2). Modern guidance generally favors `Argon2id` where available because it balances GPU resistance and side-channel considerations better than many older schemes.

**Why fast hashes are unsafe for passwords**

General-purpose hashes such as `MD5`, `SHA-1`, and `SHA-256` are fast by design. That speed is useful for integrity checks, but it is harmful for password storage because attackers can test huge numbers of guesses per second.

- **Fast hashes** make brute-force and dictionary attacks cheaper.
- **GPU cracking** benefits from algorithms that are computationally light.
- **Rainbow tables** become less effective when salts are used, but fast hashes still remain vulnerable to large-scale guessing.
- **Password hashing functions** deliberately slow down each guess to raise attacker cost.

A good rule is: use cryptographic hashes for integrity, and use password hashing functions for credentials.

**Salting basics**

A salt is a unique value added to each password before hashing. Its main purpose is to ensure that identical passwords do not produce identical stored hashes and to defeat precomputed lookup tables.

- **Uniqueness** is the key property of a salt.
- **Randomness** helps prevent collisions and precomputation.
- **Secrecy** is not required; salts are typically stored alongside the hash.
- **Per-password salts** ensure two users with the same password still have different stored records.

> A salt does not need to be secret. It needs to be unique enough and generated with sufficient randomness to avoid predictable reuse and collisions.

A conceptual, non-standard illustration looks like this:

```text
stored_record = algorithm(password, salt)
```

This is only a conceptual model. Real password hashing schemes define their own exact input handling, internal mixing, and output format. Do not assume a generic `salt + password` concatenation is how every algorithm works.

**What salting protects against**

Salting primarily blocks mass reuse of precomputed attacks.

- **Rainbow tables** become impractical because each salt requires a separate precomputation.
- **Duplicate passwords** no longer reveal themselves by matching hashes across accounts.
- **Bulk cracking efficiency** drops because attackers must attack each hash individually.

Salting does not stop guessing attacks against a single account. If a password is weak, an attacker can still brute-force it offline after stealing the hash database.

**Common password hashing algorithms**

Different algorithms trade off speed, memory use, and implementation maturity. In practice, choose a modern password hashing function supported by your platform.

| Algorithm | Main property | Notes |
|---|---|---|
| `Argon2id` | Memory-hard and modern | Strong default choice when available |
| `bcrypt` | Adaptive cost factor | Widely supported; limited password length behavior in some implementations |
| `scrypt` | Memory-hard | Designed to make GPU/ASIC attacks more expensive |
| `PBKDF2` | Iteration-based | Still common in standards and enterprise systems; less resistant to specialized hardware than memory-hard schemes |
| `MD5` / `SHA-1` / `SHA-256` | Fast general-purpose hashes | Not suitable for password storage by themselves |

**Argon2id**

`Argon2id` is often preferred for new deployments because it combines data-independent and data-dependent memory access patterns. That makes it a strong general-purpose choice for password hashing.

- **Memory hardness** increases attacker cost on GPUs and ASICs.
- **Time cost** controls how long each verification takes.
- **Parallelism** can be tuned for the server environment.
- **Parameter tuning** matters more than the algorithm name alone.

Example using a common CLI utility:

```bash
# Example: generate an Argon2id hash with a tool that supports it
argon2 somesalt -id -t 3 -m 16 -p 1
```

The exact command syntax depends on the tool installed on your system. The important idea is that the hash output should include the algorithm parameters so the verifier can reproduce the check later.

**bcrypt**

`bcrypt` remains widely used and is still acceptable in many environments. It uses a cost factor that increases the work required to compute each hash.

- **Cost factor** controls the number of rounds.
- **Adaptive tuning** lets you increase cost as hardware improves.
- **Password length limits** may exist in some implementations, so you should understand your library’s behavior.
- **Salt handling** is built into the format.

Example in Python:

```python
import bcrypt

password = b"CorrectHorseBatteryStaple!"
salt = bcrypt.gensalt(rounds=12)
hashed = bcrypt.hashpw(password, salt)

print(hashed.decode())
```

Verification:

```python
import bcrypt

candidate = b"CorrectHorseBatteryStaple!"
stored_hash = b"$2b$12$uQm8mQxw0xvQ0m0Qxw0m0eQm8mQxw0xvQ0m0Qxw0m0eQm8mQxw0xv"

if bcrypt.checkpw(candidate, stored_hash):
    print("Password valid")
else:
    print("Invalid password")
```

**PBKDF2**

`PBKDF2` is a key derivation function that applies a pseudorandom function repeatedly. It is widely standardized and still appears in many enterprise systems, especially where compatibility matters.

- **Iteration count** increases the cost of guessing.
- **HMAC** is commonly used as the underlying pseudorandom function.
- **Compatibility** is a major reason it remains in use.
- **Memory hardness** is limited compared with `Argon2` and `scrypt`.

Example in Python:

```python
import hashlib
import os
import binascii

password = b"CorrectHorseBatteryStaple!"
salt = os.urandom(16)
iterations = 200_000

dk = hashlib.pbkdf2_hmac("sha256", password, salt, iterations, dklen=32)

print("salt:", binascii.hexlify(salt).decode())
print("hash:", binascii.hexlify(dk).decode())
```

Verification uses the same salt and iteration count:

```python
import hashlib
import hmac

candidate = b"CorrectHorseBatteryStaple!"
salt = bytes.fromhex("9f2c1d3a4b5e6f708192a3b4c5d6e7f8")
iterations = 200_000
stored = bytes.fromhex("...stored_derived_key...")

derived = hashlib.pbkdf2_hmac("sha256", candidate, salt, iterations, dklen=len(stored))

if hmac.compare_digest(derived, stored):
    print("Password valid")
```

**Why constant-time comparison matters**

When comparing a computed hash to a stored hash, use a constant-time comparison function where possible. This reduces the risk of timing attacks that infer information from early-exit comparisons.

- **Timing attacks** exploit differences in comparison duration.
- **Constant-time compare** reduces information leakage.
- **Language helpers** often exist for this purpose.
- **Do not use** ordinary string comparison for security-sensitive checks if a constant-time alternative is available.

Examples:

```python
import hmac

hmac.compare_digest(a, b)
```

```go
import "crypto/subtle"

subtle.ConstantTimeCompare(a, b)
```

**Storage format and metadata**

A secure password record usually stores more than just the hash. It also stores the algorithm identifier and parameters needed for verification.

- **Algorithm name** identifies the hashing scheme.
- **Parameters** include cost, iterations, memory, and parallelism.
- **Salt** is stored with the record.
- **Versioning** helps you migrate to stronger settings later.

A practical record might look like this conceptually:

```text
$argon2id$v=19$m=65536,t=3,p=1$base64(salt)$base64(hash)
```

This format is useful because the verifier can parse the algorithm and parameters directly from the stored string. That makes upgrades and migrations easier.

**Password verification flow**

A secure login system follows a repeatable verification process.

1. You receive the candidate password from the user.
2. You retrieve the stored record for that account.
3. You parse the algorithm, salt, and parameters.
4. You recompute the derived value using the same settings.
5. You compare the computed value to the stored value using a constant-time method.
6. You return success or failure without revealing which part was wrong.

This flow avoids exposing whether the username exists, whether the password was close, or whether the stored hash format changed.

**Practical CLI examples**

You can inspect and generate hashes with common tools, but be careful to use the right utility for the algorithm.

```bash
# Generate a random salt
openssl rand -hex 16
```

```bash
# Derive a PBKDF2 key with OpenSSL's pkeyutl or kdf support, depending on version
# Syntax varies by OpenSSL release; check `openssl kdf -help`
openssl kdf -keylen 32 -kdfopt digest:SHA256 -kdfopt pass:CorrectHorseBatteryStaple! -kdfopt salt:$(openssl rand -hex 16) -kdfopt iter:200000 PBKDF2
```

> OpenSSL command syntax varies across versions. Verify the exact `kdf` options on the target system before using it in production or lab notes.

For Linux password file contexts, tools such as `mkpasswd` or language libraries are often clearer than trying to force a generic OpenSSL command into a password-hashing workflow.

**Attack perspective**

From an attacker’s point of view, a stolen password database enables offline attacks. That means the attacker can guess passwords without interacting with the application, which removes rate limits and account lockouts from the equation.

- **Dictionary attacks** test common passwords and wordlist variants.
- **Brute-force attacks** enumerate all possible combinations.
- **Credential stuffing** reuses passwords leaked from other services.
- **Hybrid attacks** combine dictionary words with rules, suffixes, and substitutions.

Salts slow these attacks by preventing reuse across accounts, but they do not stop them. The real defense is a strong password hashing algorithm plus strong user passwords and, where possible, multi-factor authentication.

**Operational best practices**

A secure implementation is not just about the algorithm. It also depends on how you deploy and maintain it.

- **Use a modern algorithm** such as `Argon2id` when supported.
- **Tune cost parameters** so verification is slow enough for attackers but acceptable for users.
- **Store unique salts** per password.
- **Use constant-time comparison** for verification.
- **Support rehashing** when parameters become outdated.
- **Protect the database** with least privilege and encryption at rest.
- **Add MFA** (multi-factor authentication) to reduce the impact of password compromise.

A common migration pattern is to rehash on successful login:

```python
if verify(candidate, stored_hash):
    if needs_upgrade(stored_hash):
        new_hash = hash_password(candidate, stronger_params)
        save_new_hash(user_id, new_hash)
```

This lets you gradually move users to stronger settings without forcing an immediate password reset.

**What not to do**

Avoid common mistakes that weaken password storage.

- **Do not store plaintext passwords.**
- **Do not use fast hashes alone** such as `MD5` or `SHA-256`.
- **Do not reuse salts** across users.
- **Do not compare hashes with normal string equality** if a constant-time option exists.
- **Do not invent your own password hashing scheme.**
- **Do not rely on encryption as a substitute** for password hashing.

**Quick mental model**

Think of password storage as a layered defense:

- **Hashing** makes the password non-reversible in practice.
- **Salting** makes each password record unique.
- **Slow parameters** make guessing expensive.
- **Constant-time verification** reduces side-channel leakage.
- **MFA** limits the damage if the password is still compromised.

If you remember only one thing, remember this: a secure password system is designed for the assumption that the database will be stolen, and it still resists offline cracking.
