## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

Mimikatz, Impacket, and BloodHound are core tools in Windows and Active Directory (AD) attack workflows. You use them to extract credentials, abuse authentication protocols, move laterally, and map privilege relationships. In exam scenarios, the key is not memorizing commands in isolation; it is understanding what each tool does, what privileges it needs, and how the output informs the next step.

These tools often appear together in a realistic chain:

- You gain a foothold on one Windows host.
- You collect credentials or reusable authentication material.
- You use network protocols and remote execution to pivot.
- You analyze AD relationships to find the shortest path to higher privilege.

> Use these tools only in authorized lab or assessment environments.

**Active Directory attack workflow**

A practical CEH workflow usually follows this pattern:

1. Enumerate the domain and identify likely targets.
2. Harvest credentials or authentication artifacts from memory, tickets, or configuration.
3. Validate access with remote protocol tools.
4. Use graph analysis to identify privilege escalation paths.
5. Execute lateral movement or privilege escalation where authorized.

A simple mental model helps:

- **Mimikatz** focuses on local credential material and Windows security internals.
- **Impacket** focuses on network protocol abuse and remote administration over SMB, Kerberos, LDAP, and related services.
- **BloodHound** focuses on AD relationship mapping and path discovery.

**Mimikatz fundamentals**

Mimikatz is a Windows post-exploitation tool used to interact with authentication secrets, security packages, and credential-related memory structures. It is commonly associated with credential dumping, ticket extraction, and domain trust abuse.

The tool is modular. For exam prep, group the major modules by what they target:

- **sekurlsa**: LSASS (Local Security Authority Subsystem Service) memory and logon sessions.
- **lsadump**: Local Security Authority secrets, domain secrets, and replication-related techniques.
- **kerberos**: Kerberos tickets and ticket operations.
- **crypto**: Certificates, keys, and cryptographic material.
- **dpapi**: Data Protection API secrets and protected user data.

**Common Mimikatz modules and use cases**

- **`sekurlsa`**
  - Targets logon sessions and authentication material in LSASS memory.
  - Useful for recovering plaintext credentials when legacy settings or weak protections are present.
  - Often yields hashes, tickets, or other reusable material on modern systems rather than plaintext.

- **`lsadump`**
  - Targets LSA secrets, cached secrets, and domain replication abuse.
  - Commonly used for techniques such as DCSync when the account has replication privileges.
  - Can expose NTLM hashes, Kerberos keys, and other domain secrets.

- **`kerberos`**
  - Works with Kerberos tickets such as TGTs (Ticket Granting Tickets) and TGSs (Ticket Granting Service tickets).
  - Useful for pass-the-ticket style workflows and ticket inspection.

- **`crypto`**
  - Helps inspect certificates, private keys, and related cryptographic artifacts.
  - Relevant when certificate-based authentication or stored keys are in scope.

- **`dpapi`**
  - Targets secrets protected by Windows DPAPI.
  - Useful for browser credentials, saved secrets, and application data protected by user or machine keys.

**Core Mimikatz commands**

Typical interactive usage starts with elevated privileges:

```text
privilege::debug
sekurlsa::logonpasswords
lsadump::sam
lsadump::secrets
lsadump::cache
kerberos::list
```

What these do:

- `privilege::debug` enables the debug privilege needed for many memory and secret access operations.
- `sekurlsa::logonpasswords` enumerates logon sessions and associated credential material.
- `lsadump::sam` extracts local account password hashes from the Security Account Manager.
- `lsadump::secrets` reads LSA secrets stored on the system.
- `lsadump::cache` inspects cached domain logons.
- `kerberos::list` shows Kerberos tickets in the current logon session.

**DCSync with Mimikatz**

DCSync is a network-based replication request against a domain controller. You do not need to run code on the DC itself, but you do need directory replication privileges such as `Replicating Directory Changes`, `Replicating Directory Changes All`, and often `Replicating Directory Changes In Filtered Set`.

Example:

```text
lsadump::dcsync /domain:corp.local /user:corp\Administrator
```

Use DCSync when:

- You have replication rights.
- You want domain password hashes or Kerberos keys without touching LSASS on the DC.
- You need a stealthier alternative to direct memory dumping on a domain controller.

> `lsadump::dcsync` is not a local-only technique. It requests replication data over the network from a domain controller.

**Pass-the-hash, pass-the-ticket, and Mimikatz**

Mimikatz supports credential reuse in different forms:

- **Pass-the-hash (PtH)** uses an NTLM hash instead of a plaintext password.
- **Pass-the-ticket (PtT)** injects a Kerberos ticket into a session.
- **Over-Pass-the-Hash (OPTH)** uses an NTLM hash to request Kerberos tickets, often through tools or workflows that support it.

Practical examples:

```text
sekurlsa::pth /user:alice /domain:corp.local /ntlm:8846f7eaee8fb117ad06bdd830b7586c /run:cmd.exe
kerberos::ptt ticket.kirbi
```

Use these when:

- You have a reusable hash but not the password.
- You have a stolen ticket and want to impersonate the session.
- You need to avoid plaintext password use in the tool chain.

**DPAPI and certificate material**

DPAPI protects secrets using user or machine keys. Mimikatz can help recover master keys and decrypt protected data when you have the right context.

Common exam-relevant examples include:

- Browser saved credentials.
- Wi-Fi profiles.
- RDP saved credentials.
- Application secrets protected by the user profile.

Certificate-related abuse can matter when:

- A private key is exportable or recoverable.
- A certificate maps to a privileged identity.
- A certificate template or enrollment path is misconfigured.

**Impacket fundamentals**

Impacket is a Python collection of network protocol implementations and attack utilities. It is especially useful for SMB, Kerberos, LDAP, MS-RPC, and remote execution workflows. In CEH-style questions, Impacket often appears as the bridge between credential material and actual remote action.

Common strengths:

- Works well from Linux or Windows with Python.
- Supports NTLM and Kerberos-based authentication in many tools.
- Provides reusable utilities for enumeration, dumping, relay, and execution.

**Authentication concepts in Impacket**

Many Impacket tools support multiple authentication styles:

- **Password authentication**: username and password.
- **Pass-the-hash**: NTLM hash in place of a password.
- **Kerberos authentication**: ticket cache or Kerberos-based auth.
- **No-pass / anonymous-style checks**: where the protocol and target allow it.

Typical syntax patterns vary by tool version, but these examples are representative:

```bash
python3 smbclient.py corp.local/alice:'Password123!'@10.10.10.25
python3 smbclient.py -hashes :8846f7eaee8fb117ad06bdd830b7586c corp.local/alice@10.10.10.25
python3 smbclient.py -k corp.local/alice@10.10.10.25
```

Notes:

- `-hashes` is commonly used for pass-the-hash.
- `-k` usually means Kerberos authentication using a ticket cache or Kerberos setup.
- Exact argument placement can vary by tool version, so verify syntax with `-h`.

**Kerberos cache setup**

For Kerberos-based Impacket use, you often need a valid ticket cache:

```bash
export KRB5CCNAME=/tmp/alice.ccache
klist
python3 psexec.py -k -no-pass corp.local/alice@10.10.10.25
```

Use Kerberos when:

- You already have a valid TGT or service ticket.
- NTLM is blocked or monitored.
- You want to avoid sending reusable NTLM material.

**Common Impacket tools**

- **`smbclient.py`**
  - SMB file share access and enumeration.
  - Useful for checking share permissions and validating credentials.

- **`psexec.py`**
  - Remote command execution through SMB and the Service Control Manager.
  - Typically requires administrative access, write access to `ADMIN$`, and the ability to create and start a service remotely.
  - Can fail if SMB signing, firewall rules, or host hardening blocks the path.

- **`wmiexec.py`**
  - Remote execution through Windows Management Instrumentation (WMI).
  - Often quieter than service-based execution, but still depends on permissions and WMI access.

- **`atexec.py`**
  - Schedules commands via the Task Scheduler service.
  - Useful when service creation is blocked but scheduled task execution is allowed.

- **`secretsdump.py`**
  - Dumps local SAM, LSA secrets, cached credentials, and domain replication data depending on privileges and target.
  - Can use remote registry, SMB, and replication methods.

- **`GetUserSPNs.py`**
  - Requests TGS tickets for accounts with Service Principal Names (SPNs).
  - Used for Kerberoasting, where you crack the service ticket hash offline.

- **`GetNPUsers.py`**
  - Enumerates users and, with the right options, requests AS-REP data for accounts without Kerberos preauthentication.
  - Used for AS-REP roasting.

- **`addcomputer.py`**
  - Adds a computer account to the domain when the account has quota or delegated rights.
  - Useful for abuse of machine account creation in some AD environments.

- **`lookupsid.py`**
  - Enumerates SIDs and can help map users and groups.

- **`rpcdump.py`**
  - Enumerates RPC endpoints and services.

**Impacket command examples**

Remote share access:

```bash
python3 smbclient.py corp.local/alice:'Password123!'@10.10.10.25
```

Remote execution with PsExec-style technique:

```bash
python3 psexec.py corp.local/alice:'Password123!'@10.10.10.25
```

Kerberoasting:

```bash
python3 GetUserSPNs.py corp.local/alice:'Password123!' -dc-ip 10.10.10.10 -request
```

AS-REP roasting:

```bash
python3 GetNPUsers.py corp.local/ -dc-ip 10.10.10.10 -usersfile users.txt -request -no-pass
```

Domain dump with replication rights:

```bash
python3 secretsdump.py corp.local/Administrator@10.10.10.10
```

If you only want domain replication data, use the appropriate flags and ensure the account truly has replication privileges:

```bash
python3 secretsdump.py corp.local/Administrator@10.10.10.10 -just-dc
```

**Impacket failure conditions**

Common reasons tools fail:

- Wrong domain format or username syntax.
- Missing SMB access or blocked port 445.
- SMB signing or LDAP signing requirements.
- Kerberos time skew or missing ticket cache.
- Insufficient privileges for remote execution or replication.
- Host firewall blocking service creation, WMI, or RPC.

**BloodHound fundamentals**

BloodHound is an AD relationship analysis platform that helps you discover attack paths through graph data. It visualizes how users, groups, computers, sessions, ACLs (Access Control Lists), and delegation relationships connect.

Classic BloodHound uses Neo4j as the graph database backend. BloodHound CE (Community Edition) uses a newer architecture, so do not assume every deployment uses Neo4j in the same way.

BloodHound helps you answer questions like:

- Which users can administer which computers?
- Which groups have dangerous rights over other objects?
- Which sessions exist on high-value systems?
- What is the shortest path from a low-privilege user to Domain Admin?

**SharpHound collection**

SharpHound is the data collector used with BloodHound. It gathers AD objects and relationships for later analysis.

Common collection concepts:

- **All**: broad collection of many relationship types.
- **Session**: logged-on user sessions on computers.
- **LocalAdmin**: local administrator relationships.
- **Trusts**: domain and forest trust relationships.
- **ACL**: object permissions and control rights.
- **Group**: group membership relationships.
- **ObjectProps**: object properties useful for analysis.
- **LoggedOn**: active or recent logon data, depending on method and permissions.

Because SharpHound syntax changes across versions, treat collection names as version-dependent. The important exam concept is what each collection mode contributes.

Example usage pattern:

```powershell
.\SharpHound.exe -c All
.\SharpHound.exe -c Session,LocalAdmin,Trusts
.\SharpHound.exe -c ACL,Group,ObjectProps
```

**When to choose each collection type**

- **Session**
  - Use when you want to find where privileged users are logged in.
  - Valuable for lateral movement planning.

- **LocalAdmin**
  - Use when you want to identify machines where a user or group has local admin rights.
  - Useful for privilege escalation and pivoting.

- **Trusts**
  - Use when you need to understand cross-domain or cross-forest relationships.
  - Important in multi-domain environments.

- **ACL**
  - Use when you want to find dangerous permissions such as `GenericAll`, `GenericWrite`, `WriteDACL`, `WriteOwner`, or `AddMember`.
  - Critical for identifying object takeover paths.

- **Group**
  - Use when you need group nesting and membership relationships.
  - Helps reveal inherited privilege.

- **ObjectProps**
  - Use when you need object attributes such as SPNs, delegation settings, or password-related flags.

**BloodHound analysis concepts**

BloodHound is most useful when you understand edge meanings. Common edges include:

- **MemberOf**: user or group membership.
- **AdminTo**: a principal has administrative rights on a computer.
- **HasSession**: a user session exists on a computer.
- **CanRDP**: Remote Desktop access is allowed.
- **CanPSRemote**: PowerShell Remoting is allowed.
- **GenericAll**: full control over an object.
- **GenericWrite**: write access to many object attributes.
- **WriteDACL**: modify the object’s discretionary access control list.
- **WriteOwner**: take ownership of the object.
- **AddMember**: add users to a group.
- **ForceChangePassword**: reset a user’s password without knowing the old one.
- **GetChanges / GetChangesAll**: replication-related rights that can enable DCSync.

**Interpreting dangerous ACLs**

ACL abuse is one of the most important BloodHound topics.

- **GenericAll**
  - Usually means full control over the object.
  - On a user object, this can allow password reset, attribute modification, and other control actions depending on the object type.
  - On a group object, it can allow membership changes and broader control.

- **GenericWrite**
  - Allows modification of many attributes.
  - May enable abuse such as adding SPNs, changing script paths, or altering delegation-related properties.

- **WriteDACL**
  - Lets you change permissions on the object.
  - Often a stepping stone to full control.

- **WriteOwner**
  - Lets you take ownership, then modify permissions.

- **AddMember**
  - Lets you add a principal to a group, which can be enough to inherit privilege.

**Common BloodHound query patterns**

BloodHound queries usually focus on shortest paths and dangerous relationships.

Examples of useful questions:

- Which users can reach Domain Admin through group nesting?
- Which computers have sessions from privileged users?
- Which principals have `GenericAll` over a high-value group?
- Which users can reset passwords for privileged accounts?
- Which computers are local admin targets for a compromised user?

Practical interpretation:

- A `HasSession` edge on a server with a Domain Admin session may indicate a lateral movement opportunity.
- A `CanRDP` or `CanPSRemote` edge can be enough for remote access if you already have valid credentials.
- A `GenericAll` edge on a group often means you can add yourself to that group and inherit its rights.

**End-to-end workflow example**

A realistic CEH-style chain might look like this:

1. You compromise a workstation and obtain local admin.
2. You run Mimikatz to inspect logon sessions and extract reusable material.
3. You use Impacket to validate access to SMB and remote execution services.
4. You collect BloodHound data to identify where privileged users log in.
5. You find a server where a privileged account has an active session.
6. You use the appropriate remote technique, if authorized, to pivot or escalate.
7. You dump domain secrets only when you have the required replication rights.

Example sequence:

```text
privilege::debug
sekurlsa::logonpasswords
```

```bash
python3 smbclient.py corp.local/alice@10.10.10.25
python3 psexec.py corp.local/alice@10.10.10.25
python3 GetUserSPNs.py corp.local/alice@10.10.10.10 -request
```

```powershell
.\SharpHound.exe -c Session,LocalAdmin,Trusts,ACL
```

This chain shows the relationship between local credential access, network execution, and graph-based path discovery.

**Practical scenario walkthroughs**

**Scenario 1: Local admin to domain insight**

- You gain local admin on a workstation.
- You run Mimikatz to inspect sessions and cached material.
- You identify a domain user with a reusable ticket or hash.
- You use Impacket to test SMB access to nearby systems.
- You run SharpHound to see whether that user has admin rights elsewhere.

Why this matters:

- Local compromise often becomes domain compromise through reused credentials or privileged sessions.
- BloodHound helps you avoid blind lateral movement.

**Scenario 2: Kerberoasting path**

- You enumerate SPNs with `GetUserSPNs.py`.
- You request TGS tickets for service accounts.
- You crack the ticket hash offline.
- You reuse the recovered password or hash for remote access.

Why this matters:

- Service accounts often have weak passwords.
- The attack targets the service ticket hash, not the SPN itself.

**Scenario 3: AS-REP roasting path**

- You enumerate users without preauthentication.
- You request AS-REP data with `GetNPUsers.py`.
- You crack the returned material offline.
- You authenticate as the recovered user if the password is weak.

Why this matters:

- Preauthentication-disabled accounts can leak crackable data without first proving knowledge of the password.

**Scenario 4: DCSync path**

- You identify replication rights in BloodHound.
- You confirm the account has `GetChanges` and `GetChangesAll`.
- You run `lsadump::dcsync` or `secretsdump.py` with the proper privileges.
- You extract domain hashes and Kerberos keys.

Why this matters:

- Replication rights are equivalent to high-impact domain compromise.
- This is one of the most important privilege paths to recognize.

**Defensive and detection considerations**

Defenders often detect these tools through a mix of endpoint, network, and directory signals.

**Mimikatz indicators**

- LSASS access attempts from unusual processes.
- Debug privilege enablement.
- Suspicious handle access to `lsass.exe`.
- Memory dump artifacts or crash telemetry.
- Event logs related to privilege use and process creation.

**Impacket indicators**

- SMB authentication from unusual hosts.
- Service creation events from remote execution tools.
- WMI or scheduled task activity from non-admin workstations.
- Kerberos ticket requests that do not match normal user behavior.
- LDAP or RPC enumeration bursts.

**BloodHound indicators**

- SharpHound collection traffic to many domain hosts.
- Large numbers of LDAP queries and SMB session checks.
- Unusual enumeration of group membership, sessions, and ACLs.
- Graph collection from a workstation that normally does not perform admin inventory.

**Useful Windows logging sources**

- **Security log**
  - Logon events, privilege use, account changes, and group membership changes.
- **Sysmon**
  - Process creation, network connections, and LSASS access patterns.
- **Directory Services logs**
  - Replication and directory changes.
- **PowerShell logs**
  - Script execution and suspicious module loading.

**Hardening and prevention**

- Restrict local administrator reuse.
- Use unique local admin passwords such as LAPS or Windows LAPS.
- Protect LSASS with RunAsPPL where appropriate.
- Limit debug privilege and administrative logon exposure.
- Enforce SMB signing on targets where relay resistance matters.
- Enforce LDAP signing and channel binding for LDAP relay resistance.
- Reduce unnecessary SPNs and review service account password strength.
- Monitor and restrict replication privileges.
- Segment administrative access and use tiered admin models.
- Review ACLs for dangerous rights such as `GenericAll`, `WriteDACL`, and `AddMember`.

**Quick comparison**

| Tool | Primary purpose | Typical output | Common exam use |
|---|---|---|---|
| Mimikatz | Credential and secret access | Hashes, tickets, secrets, DPAPI material | Dump credentials, abuse tickets, DCSync |
| Impacket | Network protocol abuse and remote execution | Remote shell, dumps, tickets, enumeration | Lateral movement, roasting, dumping |
| BloodHound | AD graph analysis | Attack paths and privilege relationships | Find escalation paths and session targets |

**Exam reminders**

- Know the difference between **hashes**, **tickets**, and **plaintext credentials**.
- Know when a tool needs local admin, domain admin, or replication rights.
- Know that BloodHound shows relationships, not exploitation by itself.
- Know that Impacket tools often depend on SMB, RPC, Kerberos, or LDAP being reachable.
- Know that Mimikatz is powerful but often blocked by modern protections and EDR.

**What to remember for CEH**

- Use **Mimikatz** when the question is about local secrets, tickets, or LSASS-related credential access.
- Use **Impacket** when the question is about protocol-based enumeration, roasting, remote execution, or dumping over the network.
- Use **BloodHound** when the question is about finding the shortest path to privilege in AD.
- Combine them in a workflow: collect, analyze, validate, pivot, and escalate.
