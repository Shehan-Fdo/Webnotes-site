## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Blueprint mindset and exam strategy**

CEH v13 questions often test whether you can recognize an attack from a short scenario, choose the most likely tool or protocol, and distinguish similar concepts under time pressure. You should study by domain, but also by pattern: recon leads to enumeration, enumeration leads to exploitation, exploitation leads to privilege escalation, and post-exploitation leads to persistence, lateral movement, and evidence handling.

A useful exam habit is to map every scenario to four questions:

- What asset is targeted?
- What protocol, service, or control is involved?
- What is the attacker trying to gain?
- What is the most likely next step or best defense?

**High-frequency domain map**

CEH v13 commonly emphasizes the following areas:

- **Reconnaissance and footprinting**: passive vs active collection, OSINT, DNS, WHOIS, social engineering, and metadata.
- **Scanning and enumeration**: host discovery, port scanning, service fingerprinting, banner grabbing, and vulnerability identification.
- **Network attacks**: sniffing, spoofing, session hijacking, man-in-the-middle (MITM), DNS poisoning, ARP poisoning, and denial of service.
- **Web application attacks**: authentication abuse, session flaws, injection, file handling, and XML-related issues.
- **Wireless attacks**: WEP/WPA differences, rogue access points, evil twins, Bluetooth, and NFC basics.
- **Cryptography and PKI**: symmetric vs asymmetric algorithms, TLS/SSL, certificates, and trust validation.
- **Cloud and container security**: service models, shared responsibility, misconfiguration, and escape risks.
- **Malware and threats**: persistence, command and control (C2), fileless behavior, and infection vectors.
- **Incident response and forensics**: volatile evidence, logs, memory, and chain of custody.
- **Vulnerability assessment and reporting**: CVSS, false positives, risk prioritization, and remediation.

**Reconnaissance and footprinting**

Reconnaissance is the information-gathering phase. In CEH questions, the key distinction is usually passive versus active.

- **Passive recon** uses public or indirect sources and avoids direct contact with the target.
  - Examples: search engines, social media, public DNS records, job postings, leaked documents, GitHub, and WHOIS.
- **Active recon** interacts with the target and is more likely to be detected.
  - Examples: ping sweeps, port scans, banner grabbing, and DNS zone transfer attempts.

Common tools and commands:

```bash
whois example.com
nslookup example.com
dig example.com any
theHarvester -d example.com -b all
recon-ng
```

Practical clues:

- A question mentioning public records, employee names, or exposed metadata usually points to passive recon.
- A question mentioning probes, scans, or service responses usually points to active recon.
- Metadata in documents can reveal usernames, software versions, printer names, and internal paths.

**Scanning, enumeration, and service clues**

Scanning finds live hosts and open ports; enumeration extracts useful details from those services. CEH often asks you to identify the service from a port, banner, or protocol behavior.

Common service and protocol clues:

| Service | Port | High-frequency clue |
|---|---:|---|
| SMB | 445 | Windows file sharing, shares, NTLM, lateral movement |
| FTP | 21 | Cleartext credentials unless protected, anonymous login |
| SSH | 22 | Secure remote shell, key-based auth, brute-force targets |
| RDP | 3389 | Remote desktop, credential attacks, exposed Windows hosts |
| SNMP | 161/162 | Device inventory, community strings, network management |
| LDAP | 389/636 | Directory queries, users/groups, authentication |
| DNS | 53 | Name resolution, zone transfer, poisoning, tunneling |
| SMTP | 25/587/465 | Mail relay, user enumeration, spoofing clues |
| HTTP/HTTPS | 80/443 | Web apps, cookies, auth, injection, TLS |
| NTP | 123 | Time sync, amplification in DoS scenarios |

Useful commands:

```bash
nmap -Pn -sS -sV -O -p 21,22,53,80,135,139,161,389,443,445,3389 192.168.1.0/24
nmap --script smb-enum-shares,smb-enum-users -p 445 192.168.1.10
nmap --script snmp-info -p 161 192.168.1.20
nmap --script ldap-rootdse -p 389 192.168.1.30
```

Enumeration examples:

```bash
ftp 192.168.1.50
ssh user@192.168.1.60
smbclient -L //192.168.1.10 -U user
snmpwalk -v2c -c public 192.168.1.20
ldapsearch -x -H ldap://192.168.1.30 -b "dc=example,dc=com"
curl -I https://example.com
```

**Network attacks: sniffing, spoofing, hijacking, and poisoning**

These topics are heavily tested because they connect protocol weaknesses to attacker goals.

- **Sniffing** is capturing traffic on a network.
  - On switched networks, sniffing often requires a position such as a compromised host, SPAN port access, ARP spoofing, or a rogue AP.
- **Spoofing** is forging identity or source information.
  - Common forms: IP spoofing, MAC spoofing, ARP spoofing, DNS spoofing, email spoofing.
- **Session hijacking** is taking over an authenticated session.
  - It may involve stealing cookies, predicting session tokens, or intercepting traffic.
- **MITM** places the attacker between two parties to intercept, modify, or relay traffic.
  - Often enabled by ARP poisoning, rogue APs, DNS poisoning, or SSL stripping.
- **ARP poisoning** corrupts the ARP cache so traffic is sent to the attacker’s MAC address.
- **DNS poisoning** corrupts name resolution so a victim reaches a malicious host.

Example ARP spoofing workflow:

```bash
# Enable forwarding so traffic can pass through the attacker
echo 1 > /proc/sys/net/ipv4/ip_forward

# Poison victim and gateway
arpspoof -i eth0 -t 192.168.1.100 192.168.1.1
arpspoof -i eth0 -t 192.168.1.1 192.168.1.100

# Capture traffic
tcpdump -i eth0 -w mitm.pcap
```

Detection and defense clues:

- Static ARP entries reduce poisoning risk on critical systems.
- DHCP snooping and Dynamic ARP Inspection help on managed switches.
- DNSSEC helps validate DNS data, though deployment is uneven.
- HTTPS with valid certificates reduces MITM success, but users can still be tricked into ignoring warnings.

> If a scenario says the attacker changes traffic in transit, steals cookies, or redirects a victim to a fake site, think MITM, session hijacking, or DNS/ARP poisoning depending on the mechanism described.

**Denial of service and distributed denial of service**

CEH questions often distinguish bandwidth exhaustion, protocol abuse, and application-layer exhaustion.

Common attack types:

- **SYN flood**: sends many TCP SYN packets and leaves half-open connections.
- **Smurf attack**: uses ICMP broadcast amplification with spoofed source addresses.
- **Ping of Death**: sends oversized or malformed ICMP packets.
- **Teardrop**: uses overlapping fragmented packets to confuse reassembly.
- **Application-layer DoS**: targets expensive requests such as login, search, or API calls.
- **Amplification attacks**: abuse services like DNS or NTP to reflect large responses to a victim.

Defensive ideas:

- Rate limiting and connection throttling
- SYN cookies
- WAF and API gateway controls
- Anycast and DDoS scrubbing services
- Proper fragmentation handling and patching

**Web application attacks and OWASP-style distinctions**

Web questions often hinge on whether the issue is authentication, authorization, session management, or injection.

**Authentication attacks**

- **Password spraying** uses a few common passwords across many accounts.
- **Credential stuffing** reuses leaked username/password pairs.
- **Brute force** tries many passwords against one account.
- **Default credentials** are common on appliances, admin panels, and IoT devices.
- **MFA fatigue** or push bombing abuses repeated prompts to trick a user.

**Authorization and access control**

- **Broken access control** means a user can access data or actions they should not.
- **IDOR** (Insecure Direct Object Reference) is a common pattern where changing an object ID exposes another user’s data.
- **Privilege escalation** in web apps often comes from role manipulation, hidden endpoints, or insecure admin functions.

**Session management**

- Session IDs should be unpredictable, protected in transit, and invalidated on logout.
- Cookies should use `HttpOnly`, `Secure`, and appropriate `SameSite` settings.
- Session fixation occurs when an attacker forces or reuses a known session identifier.
- Session hijacking occurs when the attacker steals a valid session token.

Cookie example:

```http
Set-Cookie: sessionid=abc123; Secure; HttpOnly; SameSite=Lax
```

**Injection and file handling**

- **SQL injection** targets database queries.
- **Command injection** executes operating system commands through unsafe input handling.
- **LDAP injection** targets directory queries.
- **XPath injection** targets XML query logic.
- **File inclusion** can be local or remote depending on the application and configuration.
- **Path traversal** uses sequences like `../` to escape intended directories.

Command injection example:

```bash
curl "http://target.local/ping?host=127.0.0.1;id"
```

**XML and XXE**

- **XXE** (XML External Entity) attacks abuse XML parsers that allow external entity resolution.
- Impacts can include file disclosure, SSRF (Server-Side Request Forgery), and internal network probing.
- Defenses include disabling external entities, using secure parser settings, and validating input.

Example vulnerable XML pattern:

```xml
<?xml version="1.0"?>
<!DOCTYPE data [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<data>&xxe;</data>
```

**Cross-site scripting and CSRF**

- **XSS** (Cross-Site Scripting) injects script into a page viewed by other users.
  - Stored XSS persists in the application.
  - Reflected XSS returns payloads immediately in a response.
  - DOM-based XSS occurs in client-side JavaScript.
- **CSRF** (Cross-Site Request Forgery) tricks a logged-in user into submitting an unwanted action.
- SameSite cookies, anti-CSRF tokens, and origin checks reduce CSRF risk.

**Wireless attacks and wireless exam clues**

Wireless scenarios often ask you to identify the security mode, attack path, or rogue infrastructure.

**WEP vs WPA/WPA2/WPA3**

- **WEP** is weak because of IV reuse and RC4-related design flaws; it is considered obsolete.
- **WPA** improved security but still used TKIP in older deployments.
- **WPA2** commonly uses AES-CCMP and is much stronger than WEP/WPA.
- **WPA3** improves handshake security and resists offline guessing better than WPA2-Personal.

**Common wireless attacks**

- **Evil twin**: a rogue AP mimics a legitimate SSID to lure clients.
- **Rogue AP**: unauthorized access point connected to or near the network.
- **Deauthentication attack**: forces clients off a network to capture handshakes or redirect them.
- **Handshake capture**: used to attempt offline password cracking against WPA/WPA2-Personal.
- **WPS attack**: targets weak Wi-Fi Protected Setup PIN behavior.

Typical tool flow:

```bash
airmon-ng start wlan0
airodump-ng wlan0mon
aireplay-ng --deauth 10 -a AA:BB:CC:DD:EE:FF wlan0mon
aircrack-ng capture.cap -w wordlist.txt
```

Capture indicators:

- `WPA handshake` in `airodump-ng` output indicates a usable capture for offline cracking.
- A strong signal from an unexpected SSID with the same name as a known network suggests an evil twin.
- Multiple deauth frames often indicate an active disruption or capture attempt.

**Bluetooth and NFC basics**

- **Bluetooth** is short-range wireless used for peripherals, audio, and device pairing.
  - Risks include weak pairing, unauthorized discovery, and proximity-based attacks.
- **NFC** (Near Field Communication) works at very short range and is used for payments, access cards, and tap interactions.
  - Risks include relay attacks and unauthorized reads in poorly protected implementations.

**Cryptography and PKI**

CEH questions often test whether you can distinguish confidentiality, integrity, authentication, and non-repudiation.

**Symmetric vs asymmetric cryptography**

- **Symmetric encryption** uses one shared key for encryption and decryption.
  - Examples: AES, 3DES, ChaCha20.
  - Fast and suitable for bulk data.
- **Asymmetric encryption** uses a public/private key pair.
  - Examples: RSA, ECC.
  - Slower, but useful for key exchange, signatures, and identity.

**Hashing and signatures**

- Hashes provide integrity, not confidentiality.
- Digital signatures provide integrity and authentication, and support non-repudiation when properly managed.
- Common hash algorithms include SHA-256 and SHA-3; MD5 and SHA-1 are considered weak for collision resistance.

**TLS/SSL and certificate validation**

- **TLS** (Transport Layer Security) protects data in transit; SSL is the older predecessor term often used loosely.
- A certificate chain typically includes:
  - End-entity certificate
  - Intermediate CA certificate
  - Root CA certificate
- Validation checks include:
  - Trusted chain to a root CA
  - Matching hostname
  - Valid dates
  - Revocation status when checked
  - Appropriate key usage and extended key usage

MITM with invalid certs is a classic exam scenario:

- The browser warns because the certificate does not match the site or is signed by an untrusted CA.
- If the user ignores the warning, the attacker may still intercept traffic.
- Proper certificate pinning and user awareness reduce this risk.

Key exchange examples:

- **Diffie-Hellman** allows two parties to derive a shared secret over an insecure channel.
- **ECDHE** (Elliptic Curve Diffie-Hellman Ephemeral) supports forward secrecy in modern TLS.

**Cloud and container security**

Cloud questions usually focus on service models, shared responsibility, and misconfiguration.

**Service models**

| Model | Provider manages | Customer manages |
|---|---|---|
| IaaS | Physical hardware, virtualization, core infrastructure | OS, apps, data, access controls |
| PaaS | Infrastructure and platform runtime | App code, data, identity, configuration |
| SaaS | Full application stack | Data, users, access, configuration choices |

**Shared responsibility model**

- The provider secures the cloud platform.
- You secure what you deploy, configure, and store.
- Misunderstanding this model leads to exposed storage buckets, overly permissive IAM (Identity and Access Management), and public admin interfaces.

**Common cloud attack surfaces**

- Public object storage
- Overprivileged IAM roles
- Exposed management APIs
- Metadata service abuse
- Weak secrets handling in CI/CD pipelines
- Insecure security groups and firewall rules

**Container risks**

- **Container escape** occurs when a process breaks out of the container boundary to the host.
- Misconfigurations include:
  - Running as root
  - Privileged containers
  - Mounted Docker socket
  - Excessive Linux capabilities
  - Secrets baked into images

Example hardening ideas:

```bash
docker run --read-only --cap-drop ALL --security-opt no-new-privileges nginx
```

Kubernetes clues:

- Exposed dashboards, weak RBAC (Role-Based Access Control), and overly broad service accounts are common issues.
- Secrets should not be stored in plain text in manifests or images.

**Malware, persistence, and command and control**

Malware questions often ask how infection happens, how it persists, and how it communicates.

**Common malware behaviors**

- **Persistence** keeps malware active after reboot.
  - Examples: registry run keys, scheduled tasks, services, startup folders, cron jobs, launch agents.
- **Command and control** lets an attacker issue instructions to infected hosts.
  - Channels may use HTTP/S, DNS, social media, or custom protocols.
- **Fileless malware** lives largely in memory and uses legitimate tools or scripts.
- **Macro malware** uses document macros to trigger payload execution.
- **Ransomware** encrypts data and demands payment, often after exfiltration.

Common infection vectors:

- Phishing attachments
- Malicious links
- Drive-by downloads
- Exploited public-facing services
- USB/removable media
- Supply chain compromise

Indicators of compromise:

- Unexpected outbound connections
- New autoruns or scheduled tasks
- Suspicious PowerShell or WMI (Windows Management Instrumentation) activity
- Disabled security tools
- Unusual child processes from Office applications

Example persistence checks on Windows:

```powershell
Get-ScheduledTask | Where-Object {$_.TaskName -like "*update*"}
Get-ItemProperty "HKCU:\Software\Microsoft\Windows\CurrentVersion\Run"
```

**Incident response and forensics**

CEH often tests the order of evidence collection and the difference between volatile and non-volatile data.

**Evidence acquisition order**

1. Capture the most volatile data first.
2. Preserve memory and active network state.
3. Collect logs and disk artifacts.
4. Document everything with timestamps and chain of custody.

**Volatile data**

- RAM contents
- Running processes
- Network connections
- Logged-in users
- Open files
- ARP cache and routing table

**Non-volatile data**

- Disk images
- Log files
- Registry hives
- Configuration files
- Backups

Useful commands:

```bash
ps aux
netstat -ano
ss -tulpn
lsof -i
ip addr
arp -a
```

Memory forensics basics:

- Memory can reveal injected code, decrypted strings, active malware, and network artifacts.
- Tools such as Volatility are commonly associated with memory analysis.

Log sources often referenced in scenarios:

- Windows Event Logs
- Syslog
- Web server logs
- Firewall logs
- DNS logs
- Proxy logs
- Authentication logs
- EDR (Endpoint Detection and Response) telemetry

**Vulnerability assessment, CVSS, and reporting**

CEH questions often distinguish a vulnerability from a threat and from risk.

- **Vulnerability**: a weakness that can be exploited.
- **Threat**: anything that can exploit a vulnerability or cause harm.
- **Risk**: the likelihood and impact of a threat exploiting a vulnerability.

**CVSS** (Common Vulnerability Scoring System) is used to rate severity.

- Higher scores generally indicate more urgent remediation.
- Severity should be combined with asset criticality and exposure, not treated as the only factor.

False positives and false negatives matter:

- **False positive**: scanner reports a vulnerability that is not actually present.
- **False negative**: scanner misses a real vulnerability.
- Validation with manual testing reduces both errors.

Prioritization logic:

- Internet-facing systems usually come first.
- Critical assets and exploitable remote code execution issues outrank low-impact findings.
- Compensating controls can reduce urgency, but they do not eliminate the underlying weakness.

Example reporting structure:

- Executive summary
- Scope and methodology
- Findings with evidence
- Risk rating and business impact
- Remediation steps
- Retest plan

**Practical hardening examples**

CEH questions often ask what defense best addresses a scenario. Use the control that matches the weakness.

```bash
# Linux: basic service exposure review
ss -tulpn
ufw status verbose
sudo nft list ruleset
```

```powershell
# Windows: check firewall and services
Get-NetFirewallProfile
Get-Service | Where-Object {$_.Status -eq "Running"}
```

```bash
# Web server: reduce information leakage
curl -I https://example.com
```

```nginx
server_tokens off;
add_header X-Frame-Options DENY;
add_header X-Content-Type-Options nosniff;
```

**Common exam distinctions to memorize**

- **Sniffing** captures traffic; **spoofing** forges identity.
- **MITM** intercepts and may alter traffic; **session hijacking** takes over an authenticated session.
- **Passive recon** observes; **active recon** interacts.
- **Scanning** finds services; **enumeration** extracts details.
- **Vulnerability** is a weakness; **risk** is the business impact of exploitation.
- **WEP** is weak and obsolete; **WPA2/WPA3** are stronger, with WPA3 improving resistance to offline attacks.
- **Symmetric** crypto is fast and uses one key; **asymmetric** crypto uses key pairs and supports exchange/signatures.
- **Stored XSS** persists in the app; **reflected XSS** returns in the response; **DOM XSS** happens in client-side code.
- **IaaS** gives you the most control and the most responsibility; **SaaS** gives you the least control and the least operational burden.

**Quick scenario practice**

- A user receives a fake login page after connecting to public Wi-Fi: think evil twin, MITM, or DNS poisoning.
- A scanner shows port 445 open on a Windows host: think SMB enumeration, shares, and NTLM exposure.
- A web app accepts `;id` in a parameter and executes it: think command injection.
- A document launches PowerShell and creates a scheduled task: think macro malware with persistence.
- A certificate warning appears during login to a banking site: think invalid cert, possible MITM, or misconfigured PKI.
- A cloud storage bucket is public: think misconfiguration under the shared responsibility model.
- A DNS server responds with incorrect records after poisoning: think DNS spoofing or cache poisoning.
- A flood of half-open TCP sessions overwhelms a server: think SYN flood.

**Exam-day approach**

When you see a scenario, identify the layer first:

- Network layer issues usually involve spoofing, poisoning, sniffing, or DoS.
- Application layer issues usually involve auth, session, injection, or file handling.
- Wireless issues usually involve rogue APs, weak encryption, or handshake capture.
- Crypto issues usually involve trust, certificates, key exchange, or invalid validation.
- Cloud issues usually involve misconfiguration, IAM, or shared responsibility.
- Forensics issues usually involve evidence order, logs, memory, and chain of custody.

If two answers seem plausible, choose the one that best matches the mechanism described in the question rather than the broad category.
