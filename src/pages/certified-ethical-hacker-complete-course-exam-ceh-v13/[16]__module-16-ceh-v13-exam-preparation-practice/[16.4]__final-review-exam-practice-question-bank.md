## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Final Review Strategy**

Use this section as a last-pass consolidation of the CEH v13 blueprint. The exam rewards broad recognition of attack paths, defensive controls, and tool purpose more than deep exploit coding. You should be able to identify the right technique from a scenario, explain why it works, and choose the best mitigation.

Focus on:
- Reconnaissance and enumeration workflows
- Scanning and vulnerability assessment
- System, web, wireless, cloud, and mobile attack concepts
- Cryptography and authentication basics
- Incident response, forensics, and reporting
- Common tools and what they are used for

A good final review is not just memorization. You should connect each attack to its prerequisites, artifacts, and defenses.

**Reconnaissance and Footprinting**

Reconnaissance is the information-gathering phase that happens before exploitation. In CEH-style questions, the key is often to distinguish passive collection from active probing.

- **Passive recon** uses public sources and does not directly touch the target.
  - Examples include WHOIS lookups, search engine queries, social media, job postings, and public code repositories.
- **Active recon** interacts with target systems.
  - Examples include DNS queries, port scans, banner grabbing, and web directory enumeration.
- **OSINT** means Open Source Intelligence.
  - You use it to build a target profile from public data.
- **Footprinting** often includes:
  - Domain registration data
  - Subdomains
  - IP ranges
  - Email formats
  - Technology stack clues

Practical examples:

```bash
whois example.com
nslookup example.com
dig example.com MX
dig example.com NS
```

For DNS enumeration, prefer record-specific queries over broad assumptions. Many DNS servers minimize or refuse `ANY` responses, so `ANY` is not a reliable enumeration method.

```bash
dig www.example.com A
dig example.com MX
dig example.com TXT
dig example.com NS
```

If zone transfers are misconfigured, you may be able to retrieve the full zone from an authoritative server:

```bash
dig axfr example.com @ns1.example.com
```

> Zone transfers are usually blocked. Treat AXFR as a misconfiguration check, not a normal enumeration path.

**Scanning and Vulnerability Assessment**

Scanning identifies live hosts, open ports, services, and potential weaknesses. Vulnerability assessment goes one step further by mapping findings to known issues and prioritizing remediation.

- **Port scanning** identifies exposed services.
- **Service detection** helps you infer versions and likely vulnerabilities.
- **OS fingerprinting** estimates the target operating system.
- **Vulnerability scanners** such as Nessus and OpenVAS automate checks against known issues.
- **False positives** are findings that appear vulnerable but are not.
- **False negatives** are missed vulnerabilities.

Common workflow:
1. Discover hosts.
2. Scan ports and services.
3. Run authenticated or unauthenticated vulnerability checks.
4. Validate high-risk findings manually.
5. Prioritize remediation by exposure, exploitability, and business impact.

Example commands:

```bash
nmap -sn 192.168.1.0/24
nmap -sS -sV -O 192.168.1.10
nmap -Pn -p 1-1000 --script vuln 192.168.1.10
```

Nessus-style remediation thinking:
- Patch internet-facing systems first.
- Fix credentialed findings before unauthenticated guesses.
- Reduce attack surface by disabling unused services.
- Re-scan after remediation to confirm closure.

**Password Attacks and Authentication**

Password attacks are common in CEH scenarios because weak credentials remain one of the easiest entry points.

- **Online attacks** target a live login service.
  - They are limited by lockouts, rate limiting, MFA, and logging.
- **Offline attacks** target captured hashes or encrypted material.
  - They are limited mainly by hash strength and password complexity.
- **Dictionary attacks** try known words and variants.
- **Brute force attacks** try all combinations within a defined space.
- **Hybrid attacks** combine dictionary words with rules or suffixes.
- **Rainbow tables** are precomputed hash tables.
  - They are less effective when passwords are salted.
- **Salting** adds unique random data before hashing.
  - It prevents identical passwords from producing identical hashes.

Example hash checking:

```bash
hashcat -m 1000 hashes.txt rockyou.txt
john --wordlist=rockyou.txt hashes.txt
```

Important exam distinctions:
- **LM hashes** are weak and split into halves.
- **NTLM** is stronger than LM but still vulnerable to offline cracking if captured.
- **Account lockout policies** reduce online guessing.
- **MFA** reduces the value of stolen passwords, but not always the value of session tokens or legacy protocols.

**Windows and Active Directory Concepts**

Active Directory (AD) is a major CEH topic because many enterprise attacks pivot through domain identity and trust relationships.

- **Domain Controller (DC)** authenticates and authorizes domain users.
- **Kerberos** is the primary AD authentication protocol.
- **TGT** means Ticket Granting Ticket.
- **TGS** means Ticket Granting Service ticket.
- **SPN** means Service Principal Name.

Common attack concepts:
- **Kerberoasting**
  - Request service tickets for accounts with SPNs and crack the offline ticket material if the service account uses a weak password.
- **AS-REP roasting**
  - Targets accounts configured without Kerberos preauthentication.
- **Pass-the-Hash**
  - Uses an NTLM hash directly instead of the plaintext password.
- **Pass-the-Ticket**
  - Reuses Kerberos tickets.
- **Golden Ticket**
  - Forged TGT created with the KRBTGT account secret.
- **Silver Ticket**
  - Forged service ticket for a specific service.

Enumeration examples:

```powershell
Get-ADUser -Filter * | Select-Object Name,SamAccountName
Get-NetTCPConnection
net user /domain
nltest /dclist:example.local
```

Useful concepts:
- **Domain trusts** can extend compromise across forests or domains.
- **Privilege escalation** often follows weak service account passwords, excessive group membership, or misconfigured delegation.
- **Lateral movement** often uses remote management protocols such as SMB, WinRM, RDP, and WMI.

**Linux and Unix Fundamentals**

Linux questions often test service exposure, file permissions, and privilege escalation basics.

- **SUID** and **SGID** bits can allow privilege escalation if misused.
- **Cron jobs** can be abused if writable by low-privilege users.
- **SSH keys** are a common persistence and access mechanism.
- **World-writable files** and directories are risky.
- **Capabilities** can grant fine-grained privileges.

Useful commands:

```bash
id
sudo -l
find / -perm -4000 -type f 2>/dev/null
ps aux
ss -tulpn
```

Common review points:
- Check for weak service permissions.
- Inspect scheduled tasks and startup scripts.
- Review `/etc/passwd`, `/etc/shadow`, and SSH configuration.
- Use `ss` or `netstat` to identify listening services.

**Web Application Attacks**

Web questions often focus on input handling, session management, and trust boundaries.

- **SQL injection (SQLi)** manipulates database queries.
- **Cross-Site Scripting (XSS)** injects script into a victim’s browser context.
- **Cross-Site Request Forgery (CSRF)** abuses a logged-in user’s browser state.
- **Command injection** executes OS commands through unsanitized input.
- **File inclusion** can load unintended local or remote files.
- **Directory traversal** accesses files outside the intended web root.
- **Authentication flaws** include weak password reset logic, credential stuffing, and broken MFA flows.
- **Session management flaws** include predictable session IDs, missing expiration, and insecure cookie flags.
- **Clickjacking** tricks users into clicking hidden UI elements.
- **XML External Entity (XXE)** attacks abuse XML parsers that resolve external entities.
- **Server-Side Request Forgery (SSRF)** makes the server send requests to internal or protected resources.
- **API security issues** often involve broken object-level authorization, weak tokens, and excessive data exposure.

Example payload ideas:
- SQLi: `' OR '1'='1' --`
- XSS: `<script>alert(1)</script>`
- Path traversal: `../../../../etc/passwd`

Defensive controls:
- Parameterized queries
- Output encoding
- CSRF tokens
- Secure cookie flags: `HttpOnly`, `Secure`, `SameSite`
- Strong session expiration
- Input validation and allowlists
- XML parser hardening
- API authorization checks on every object request

**Wireless Attacks and Defenses**

Wireless questions often compare WPA, WPA2, and WPA3, and ask how to secure or attack each.

- **WEP** is obsolete and broken.
- **WPA/WPA2-Personal** use a pre-shared key (PSK).
- **WPA2-Enterprise** uses 802.1X with a RADIUS server and per-user authentication.
- **WPA3** improves resistance to offline guessing with SAE, Simultaneous Authentication of Equals.

The classic 4-way handshake is most useful for offline cracking of WPA/WPA2-Personal PSK networks. It does not give you a shared enterprise PSK to crack in the same way.

Common wireless concepts:
- **Evil twin**: rogue access point that imitates a legitimate SSID.
- **Rogue AP detection**: identify unauthorized APs on the network.
- **Deauthentication attacks**: force clients to reconnect.
- **WPS** means Wi-Fi Protected Setup.
- **Reaver**-style attacks target weak WPS PIN implementations.
- **MAC filtering** is not strong security by itself.
- **Hidden SSIDs** do not provide real protection.

Example commands:

```bash
airmon-ng check kill
airmon-ng start wlan0
airodump-ng wlan0mon
aireplay-ng --deauth 10 -a <AP_MAC> -c <CLIENT_MAC> wlan0mon
```

Defensive controls:
- Use WPA3 or WPA2-AES with strong passphrases.
- Disable WPS.
- Segment guest and corporate networks.
- Monitor for rogue APs and evil twins.
- Use 802.1X for enterprise authentication.

**Cryptography and PKI**

Cryptography questions usually test what each algorithm is used for and how certificates support trust.

- **Symmetric encryption** uses one shared key.
  - Examples: AES, 3DES, ChaCha20.
- **Asymmetric encryption** uses a public/private key pair.
  - Examples: RSA, ECC.
- **Hashing** is one-way and used for integrity.
  - Examples: SHA-256, SHA-3, bcrypt, scrypt, Argon2.
- **Digital signatures** provide integrity, authenticity, and nonrepudiation.
- **PKI** means Public Key Infrastructure.

Common mode examples:
- **CBC**: Cipher Block Chaining
- **GCM**: Galois/Counter Mode
- **ECB**: Electronic Codebook, generally insecure for repeated patterns

TLS basics:
- The client and server negotiate cipher suites.
- The server presents a certificate.
- The client validates the certificate chain and hostname.
- Key exchange establishes shared session keys.

Practical checks:

```bash
openssl s_client -connect example.com:443 -servername example.com
openssl x509 -in cert.pem -text -noout
```

Important concepts:
- **Certificate pinning** restricts trust to expected certificates or public keys.
- **HSTS** means HTTP Strict Transport Security.
  - It helps prevent downgrade attacks and SSL stripping.
- **SSL stripping** is less effective against modern browsers and HSTS-protected sites.

**Cloud Security**

Cloud questions often test the shared responsibility model and common misconfigurations.

- The **shared responsibility model** means the provider secures the cloud platform, while you secure your data, identities, configurations, and workloads.
- **IAM** means Identity and Access Management.
- **Least privilege** is critical for cloud roles and policies.
- **Security groups** and network ACLs control traffic exposure.
- **Object storage** misconfigurations can expose sensitive data.
- **Metadata service abuse** usually involves **SSRF** to the instance metadata service, especially when protections are weak or misconfigured.

Example AWS-style policy idea:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": ["arn:aws:s3:::example-bucket/public/*"]
    }
  ]
}
```

Cloud review points:
- Log identity and API activity.
- Restrict metadata access where possible.
- Rotate credentials and use short-lived tokens.
- Review public storage, exposed keys, and overly permissive roles.

**Incident Response and Forensics**

Incident response questions often ask what to do first, what to preserve, and how to avoid contaminating evidence.

- **Preserve evidence** before making changes when possible.
- **Chain of custody** documents who handled evidence and when.
- **Volatile data** disappears quickly.
  - Memory, active connections, running processes, and logged-in sessions are high priority.
- **Timeline analysis** reconstructs events from logs and artifacts.
- **Memory forensics** can reveal injected code, credentials, and active malware.

Typical order of operations:
1. Identify the incident.
2. Contain the threat.
3. Preserve evidence.
4. Eradicate the cause.
5. Recover systems.
6. Review lessons learned.

Useful commands:

```bash
Get-Process
Get-NetTCPConnection
tasklist /v
netstat -ano
ps aux
lsof -i
```

Common artifacts:
- Windows Event Logs
- Prefetch files
- Registry hives
- Scheduled tasks
- Bash history
- Syslog
- Auth logs
- Web server logs

> Do not rely on a single artifact. Correlate logs, timestamps, and host activity to build a defensible timeline.

**Malware Analysis and Evasion**

Malware-related questions often focus on behavior, persistence, and how defenders detect it.

- **Virus** attaches to a host file.
- **Worm** self-replicates across systems.
- **Trojan** disguises itself as legitimate software.
- **Ransomware** encrypts data for extortion.
- **Spyware** collects information covertly.
- **Rootkit** hides malicious activity.

Analysis approaches:
- **Static analysis** inspects files without executing them.
- **Dynamic analysis** observes behavior in a controlled environment.
- **Sandboxing** helps isolate suspicious samples.
- **Indicators of compromise (IOCs)** include hashes, domains, IPs, file paths, and registry keys.

Evasion and obfuscation concepts:
- **Packing** compresses or encrypts malware to hide code.
- **Polymorphism** changes the code appearance while preserving behavior.
- **Metamorphism** rewrites the malware body more extensively.
- **Anti-debugging** and **anti-VM** checks hinder analysis.
- **Living off the land** uses legitimate tools like PowerShell or `rundll32.exe`.

**Mobile Security**

Mobile questions usually test app permissions, insecure storage, and platform-specific risks.

- **Jailbreaking** and **rooting** weaken platform protections.
- **Insecure local storage** can expose tokens, keys, or personal data.
- **Insecure communication** can leak data over weak TLS or cleartext.
- **Reverse engineering** can reveal hardcoded secrets and API endpoints.
- **Mobile malware** may abuse accessibility services, overlays, or SMS permissions.

Defensive ideas:
- Use secure storage APIs.
- Enforce certificate validation.
- Minimize permissions.
- Obfuscate sensitive client-side logic.
- Protect APIs with server-side authorization.

**IoT and OT/ICS Security**

IoT and operational technology questions often involve weak defaults and legacy protocols.

- **IoT** devices often ship with default credentials and weak update mechanisms.
- **OT/ICS** means Operational Technology / Industrial Control Systems.
- **SCADA** means Supervisory Control and Data Acquisition.
- Legacy systems may prioritize availability over confidentiality.
- Segmentation is critical because many devices cannot be patched quickly.

Common risks:
- Default passwords
- Exposed management interfaces
- Insecure firmware updates
- Weak or missing encryption
- Flat networks that allow lateral movement

Defensive controls:
- Change default credentials immediately.
- Segment OT from IT.
- Restrict remote access.
- Monitor firmware integrity.
- Use allowlists for management traffic.

**Tools You Should Recognize**

CEH questions often ask what a tool does rather than how to use it deeply.

- `nmap` for discovery, port scanning, and service enumeration
- `Nessus` and `OpenVAS` for vulnerability assessment
- `Wireshark` for packet analysis
- `tcpdump` for command-line packet capture
- `Burp Suite` for web testing
- `Metasploit` for exploitation and payload handling
- `Hydra` for online password attacks
- `hashcat` and `John the Ripper` for offline cracking
- `Aircrack-ng` suite for wireless assessment
- `sqlmap` for SQL injection testing
- `Nikto` for web server checks
- `Gobuster` or `ffuf` for content discovery
- `enum4linux-ng` for SMB enumeration in modern environments

Examples:

```bash
gobuster dir -u https://example.com -w /usr/share/wordlists/dirb/common.txt
ffuf -u https://example.com/FUZZ -w words.txt
sqlmap -u "https://example.com/item?id=1" --dbs
```

**High-Yield Defensive Controls**

Many exam questions ask for the best mitigation rather than the attack name.

- Use **least privilege** everywhere.
- Patch exposed systems quickly.
- Enforce MFA for remote and privileged access.
- Segment networks and restrict east-west movement.
- Log authentication, privilege changes, and admin actions.
- Validate input and encode output in web apps.
- Disable unused services and close unnecessary ports.
- Use secure defaults for cloud, wireless, and mobile deployments.
- Train users to recognize phishing and social engineering.

**Scenario-Based Review**

Use these patterns to reason through exam questions.

- If you see a captured Wi-Fi handshake and a weak passphrase, think offline PSK cracking.
- If you see a domain service account with an SPN and weak password, think Kerberoasting.
- If you see a server making internal HTTP requests on behalf of user input, think SSRF.
- If you see a browser executing injected script, think XSS.
- If you see a login page accepting repeated guesses, think rate limiting and lockout weaknesses.
- If you see a cloud instance exposing credentials through a local endpoint, think metadata service abuse via SSRF.
- If you see a rogue AP impersonating a corporate SSID, think evil twin.
- If you see a web app reading arbitrary files, think path traversal or file inclusion.

**Final Exam Tactics**

- Read the question for the asset, not just the attack name.
- Eliminate answers that are technically true but do not fit the scenario.
- Watch for wording around passive versus active, online versus offline, and prevention versus detection.
- Prefer the most specific correct answer.
- If two answers seem plausible, choose the one that best matches the control or workflow described.

**Last-Minute Memory Hooks**

- **Passive vs active**: public data versus direct probing
- **Online vs offline**: live login limits versus captured material
- **Kerberoasting vs AS-REP roasting**: SPN tickets versus no preauth
- **XSS vs CSRF**: script execution versus forced action
- **WPA2-Personal vs Enterprise**: PSK cracking versus per-user 802.1X
- **SSRF vs XSS**: server-side request versus browser-side script
- **HSTS vs SSL stripping**: enforced HTTPS blocks downgrade tricks
- **Golden vs Silver Ticket**: domain-wide versus service-specific forgery

**Exam Mindset**

You do not need to know every tool option. You do need to know:
- What the attack is
- What prerequisite it needs
- What artifact it leaves
- What control stops it
- What tool or command best fits the scenario

That pattern is enough to eliminate distractors and choose the best answer under time pressure.
