## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**What scenario-based reasoning means**

Scenario-based questions test whether you can choose the right action, not just define a tool. You read the target context, identify the objective, and map that objective to the safest, most effective workflow. In CEH-style questions, the correct answer often depends on protocol, environment, access level, and whether you need passive observation, active validation, exploitation, or post-exploitation verification.

A good mental model is:

- **Goal first**: determine what the task is asking for.
- **Constraints second**: note network access, permissions, stealth requirements, and time.
- **Tool last**: choose the tool that fits the task and the constraints.

**How to reason through a scenario**

Use a repeatable decision process when you see a question:

1. Identify the asset type: host, web app, DNS, wireless, cloud, identity, or internal network.
2. Identify the phase: reconnaissance, enumeration, exploitation, privilege escalation, lateral movement, persistence, or reporting.
3. Identify the access path: direct, VPN, proxy, pivot, or isolated segment.
4. Identify the output needed: proof of vulnerability, credentials, logs, packet capture, or remediation evidence.
5. Choose the least disruptive tool that answers the question.

This approach prevents common exam mistakes such as using an exploitation tool when the question only asks for enumeration, or using an active scanner when passive validation is safer.

**Reconnaissance and enumeration tool mapping**

Reconnaissance tools answer “what exists?” Enumeration tools answer “what details can I confirm?” The distinction matters because many CEH questions are about selecting the right tool for discovery without triggering unnecessary alerts.

- **`nmap`**: port scanning, service detection, OS fingerprinting, NSE (Nmap Scripting Engine) checks, and basic vulnerability validation.
- **`masscan`**: very fast large-range port discovery when speed matters more than detail.
- **`netcat` / `nc`**: banner grabbing, simple TCP/UDP checks, and ad hoc connectivity testing.
- **`telnet`**: legacy protocol testing and manual banner checks for text-based services.
- **`curl`** and **`wget`**: HTTP(S) request testing, header inspection, and file retrieval.
- **`whatweb`**: quick web technology fingerprinting.
- **`wappalyzer`**: browser-based web stack identification.
- **`nikto`**: web server misconfiguration and known issue checks.
- **`gobuster`** and **`ffuf`**: content and directory brute forcing.
- **`dirsearch`**: web path discovery with flexible wordlists and extensions.
- **`amass`** and **`subfinder`**: subdomain enumeration and asset discovery.
- **`dnsrecon`** and **`fierce`**: DNS-focused enumeration, including zone transfer checks and subdomain discovery.
- **`dig`** and **`nslookup`**: direct DNS queries, record lookups, and zone transfer validation.
- **`arp-scan`** and **`netdiscover`**: local network host discovery on Layer 2 segments.
- **`enum4linux-ng`**: SMB and Windows domain enumeration.
- **`rpcclient`**: SMB/RPC enumeration such as users, groups, shares, and domain info.
- **`snmpwalk`**: SNMP enumeration when community strings are known or guessed.

For DNS-focused scenarios, choose the tool based on the question:

- Use `dig` when you need a precise record query or to test a zone transfer.
- Use `nslookup` when the question emphasizes quick interactive DNS checks.
- Use `dnsrecon` when you need a broader DNS enumeration workflow.
- Use `fierce` when the scenario asks for subdomain discovery and zone transfer testing.

Example DNS checks:

```bash
dig example.com A
dig example.com MX
dig @10.10.10.53 example.com AXFR
nslookup -type=txt example.com
dnsrecon -d example.com
fierce --domain example.com
```

> A zone transfer only succeeds if the DNS server is misconfigured to allow it. In exam scenarios, that detail often determines whether the correct answer is a DNS enumeration tool or a generic scanner.

**Web application testing and validation**

Web scenarios usually ask you to identify injection points, hidden content, authentication weaknesses, or misconfigurations. Match the tool to the task rather than defaulting to a scanner.

- **`burpsuite`**: intercepting proxy, request tampering, session analysis, and manual testing.
- **Burp Suite Scanner**: active scanning in Burp Suite Professional or Enterprise, not Community.
- **`sqlmap`**: automated SQL injection detection and exploitation validation.
- **`wpscan`**: WordPress enumeration, plugin/theme checks, and known issue discovery.
- **`nikto`**: quick server-side checks for common web misconfigurations.
- **`feroxbuster`**: fast recursive content discovery.
- **`gobuster`**: directory and DNS brute forcing.
- **`ffuf`**: fuzzing parameters, paths, headers, and virtual hosts.

A practical workflow is:

1. Intercept the request in Burp Suite.
2. Identify parameters, cookies, and headers that influence behavior.
3. Replay and modify the request.
4. Use `sqlmap` only when the scenario suggests injection validation or extraction.
5. Use a content discovery tool when the question is about hidden paths or admin panels.

Example Burp-style request testing:

```http
GET /product?id=10 HTTP/1.1
Host: target.local
Cookie: session=abc123
```

Example `sqlmap` validation:

```bash
sqlmap -u "http://target.local/product?id=10" --batch --dbs
```

**Exploitation-specific tool mapping**

When a scenario moves from discovery to exploitation, choose tools that validate a known weakness or deliver a controlled payload. Do not use exploitation tooling when the question only asks for identification.

- **`metasploit`**: exploit validation, payload delivery, auxiliary scanning, and post-exploitation modules.
- **`searchsploit`**: offline exploit lookup against known product versions.
- **`exploit-db`** references: useful when the scenario asks how to find a public exploit for a versioned target.
- **`msfconsole`**: interactive exploitation and module chaining.
- **`msfvenom`**: payload generation for controlled testing.
- **`nc`** and **`socat`**: simple listeners, reverse shell handling, and port forwarding.
- **`python3 -m http.server`**: quick file hosting for payload transfer in lab environments.

Example exploit validation workflow:

```bash
searchsploit apache 2.4.49
msfconsole
```

Inside Metasploit, you typically:

```text
search cve-2021-41773
use exploit/multi/http/apache_path_traversal
set RHOSTS 10.10.10.20
set TARGETURI /
run
```

Use `searchsploit` when the question asks you to identify a public exploit before attempting validation. Use Metasploit when the scenario asks for a framework that can test, deliver, or manage an exploit chain.

**Password attacks, hash identification, and cracking**

A common exam trap is choosing a cracking tool before identifying the hash type. First determine the format, then select the correct attack mode and tool.

- **Hash identification**: inspect the hash length, prefix, and context.
- **NTLM**: common Windows hash format; often appears as 32 hexadecimal characters.
- **bcrypt**: usually starts with `$2a$`, `$2b$`, or `$2y$`.
- **SHA-1**: 40 hexadecimal characters.
- **SHA-256**: 64 hexadecimal characters.
- **MD5**: 32 hexadecimal characters, but context matters because NTLM also uses 32 hex characters.
- **Kerberos-related hashes**: may require specific handling depending on the capture type.

Choose the cracking tool and mode based on the hash:

- **`hashcat`**: GPU-accelerated cracking; best when you know the hash mode.
- **`john`**: flexible cracking and hash format detection.
- **`hashid`** and **`haiti`**: hash identification helpers.
- **`hash-identifier`**: older but still seen in training labs.

Example workflow:

```bash
hashid hashes.txt
hashcat -m 1000 hashes.txt rockyou.txt
john --format=NT hashes.txt --wordlist=rockyou.txt
```

Practical mapping:

- Use `hashid` or `haiti` first when the hash type is unclear.
- Use `hashcat` when the hash mode is known and GPU speed matters.
- Use `john` when you need flexible format support or quick validation.
- Use the correct mode number in `hashcat`; the wrong mode wastes time and produces misleading failures.

**Password policy auditing and spraying strategy**

Scenario questions often test whether you can attack credentials without causing lockouts. The right answer depends on policy, rate limits, and account sensitivity.

- **Password spraying**: try one password against many accounts to avoid lockouts.
- **Credential stuffing**: reuse known username/password pairs from breaches.
- **Brute force**: try many passwords against one account; high lockout risk.
- **Policy auditing**: assess complexity, history, lockout thresholds, and MFA (multi-factor authentication) enforcement.

Useful tools include:

- **`crackmapexec`**: SMB/WinRM/LDAP validation and spraying workflows.
- **`kerbrute`**: Kerberos username enumeration and password spraying in AD (Active Directory) contexts.
- **`hydra`**: multi-protocol login testing.
- **`medusa`**: parallel login attempts across many services.
- **`ncrack`**: network authentication auditing.

Example safe spraying pattern:

```bash
kerbrute passwordspray -d corp.local users.txt Spring2026!
```

Or with Hydra for a single service in a lab:

```bash
hydra -L users.txt -p Spring2026! ssh://10.10.10.20
```

Use password spraying when the scenario mentions account lockout avoidance, many users, or a domain environment. Use brute force only when the question explicitly allows it or when lockout risk is not a concern.

**Windows and Linux privilege escalation mapping**

Privilege escalation questions usually ask you to find misconfigurations, weak permissions, or credential artifacts. Match the tool to the operating system and the type of evidence needed.

Windows-focused tools:

- **`winPEAS`**: automated local privilege escalation checks.
- **`Seatbelt`**: Windows host enumeration and security posture checks.
- **`Mimikatz`**: credential extraction, token inspection, and Kerberos ticket abuse in lab contexts.
- **Sysinternals tools**: process, service, autorun, and persistence inspection.
- **`whoami /priv`**: privilege review.
- **`net user`** and **`net localgroup`**: account and group enumeration.
- **`sc`** and **`wmic`**: service and system inspection.

Linux-focused tools:

- **`linPEAS`**: automated local privilege escalation checks.
- **`sudo -l`**: enumerate allowed sudo commands.
- **`id`** and **`groups`**: confirm group membership.
- **`find`**: locate SUID/SGID files, writable paths, and sensitive artifacts.
- **`ps`**, **`crontab -l`**, and **`systemctl`**: inspect running processes and scheduled tasks.

Example Linux enumeration:

```bash
id
sudo -l
find / -perm -4000 -type f 2>/dev/null
```

Example Windows enumeration:

```cmd
whoami /priv
whoami /groups
net user
net localgroup administrators
```

Use `winPEAS` or `linPEAS` when the scenario asks for a broad automated check. Use manual commands when the question asks for a specific misconfiguration or when you need to verify a finding.

**Pivoting, tunneling, reverse shells, and segmented networks**

When a target sits behind a jump host, firewall, or isolated subnet, the correct tool is often one that creates a path rather than one that attacks directly.

- **Pivoting**: using a compromised host to reach another network.
- **Tunneling**: encapsulating traffic through another protocol or channel.
- **Port forwarding**: exposing a remote service locally or vice versa.
- **Reverse shell**: target initiates the connection back to the attacker.
- **Bind shell**: target listens for an incoming connection.

Common tools:

- **`ssh -L`**, **`ssh -R`**, and **`ssh -D`**: local, remote, and dynamic forwarding.
- **`chisel`**: lightweight TCP/UDP tunneling and SOCKS proxying.
- **`socat`**: flexible relay and port forwarding.
- **`proxychains`**: route tools through a SOCKS proxy.
- **`meterpreter`**: post-exploitation session with routing and pivot support.
- **`ligolo-ng`**: modern tunneling and pivoting in lab environments.
- **`netcat`**: simple reverse shell and listener handling.

Example SOCKS proxy via SSH:

```bash
ssh -D 1080 user@jumpbox
proxychains nmap -sT -Pn 10.20.30.0/24
```

Example port forwarding:

```bash
ssh -L 8080:10.20.30.15:80 user@jumpbox
```

Use pivoting tools when the scenario mentions internal-only services, segmented VLANs, or a host that can reach the target but your workstation cannot.

**VPN, proxy, and anonymity routing tools**

Some scenarios are about access control, traffic routing, or hiding the source of testing traffic. Choose the tool that matches the network path requirement.

- **VPN (Virtual Private Network)**: extends your network presence into a remote environment.
- **Proxy**: relays traffic through an intermediate host or service.
- **SOCKS proxy**: supports many TCP-based tools through a generic relay.
- **HTTP proxy**: useful for web traffic and browser-based testing.
- **Tor**: anonymity network for routing traffic through multiple relays, though it is not suitable for all testing tasks.

Tools and use cases:

- **`openvpn`**: connect to a lab or client VPN.
- **`wireguard`**: lightweight VPN connectivity.
- **`proxychains`**: force CLI tools through a proxy.
- **Burp Suite upstream proxy**: route web traffic through another proxy or VPN-aware path.
- **`torsocks`**: route supported applications through Tor.

Example proxy routing:

```bash
proxychains curl http://10.10.10.20
```

Use these tools when the scenario says you must test from a specific network location, preserve source separation, or reach an internal segment through a controlled path.

**DNS, SMB, SNMP, and directory services scenarios**

Many CEH questions are protocol-specific. The right tool depends on the service being enumerated.

DNS:

- Use `dig` for record queries, authoritative server checks, and AXFR testing.
- Use `dnsrecon` for broader DNS enumeration.
- Use `fierce` for subdomain discovery and transfer checks.

SMB and Windows domain services:

- Use `enum4linux-ng` for shares, users, groups, and policy discovery.
- Use `rpcclient` for targeted RPC enumeration.
- Use `smbclient` for share access and file listing.
- Use `crackmapexec` for validation across many hosts.

Example SMB enumeration:

```bash
enum4linux-ng -A 10.10.10.20
rpcclient -U "" 10.10.10.20 -c enumdomusers
smbclient -L //10.10.10.20 -N
```

SNMP:

- Use `snmpwalk` to enumerate system details, interfaces, and sometimes process or routing data.
- Use `snmpget` for specific OIDs (Object Identifiers).

Example SNMP checks:

```bash
snmpwalk -v2c -c public 10.10.10.20
snmpget -v2c -c public 10.10.10.20 sysDescr.0
```

**Log analysis and incident triage**

Scenario questions may ask which logs to inspect after suspicious activity. Choose the source that best matches the event type.

Common log sources:

- **Windows Event Logs**: authentication, service creation, PowerShell, scheduled tasks, and logon events.
- **Sysmon**: process creation, network connections, image loads, and persistence indicators.
- **Linux auth logs**: SSH logins, sudo usage, and authentication failures.
- **Web server logs**: request paths, user agents, status codes, and injection attempts.
- **Firewall logs**: blocked connections, unusual ports, and source/destination patterns.
- **DNS logs**: suspicious lookups, tunneling indicators, and domain generation patterns.
- **Proxy logs**: outbound web activity and command-and-control indicators.

Useful tools:

- **`grep`**, **`awk`**, and **`sed`**: quick text triage.
- **`journalctl`**: systemd log review on Linux.
- **`Event Viewer`**: Windows log inspection.
- **SIEM (Security Information and Event Management)** platforms: correlation and alerting.
- **EDR (Endpoint Detection and Response)** consoles: endpoint telemetry and containment.

Example Linux triage:

```bash
grep -i "failed password" /var/log/auth.log
journalctl -u ssh --since "1 hour ago"
```

Example Windows triage focus:

- Look for repeated failed logons.
- Check for new services or scheduled tasks.
- Review PowerShell execution and suspicious parent-child process chains.
- Correlate timestamps across host, network, and application logs.

**Wireless and mobile scenario mapping**

Wireless questions usually ask about discovery, authentication, or capture rather than exploitation first.

- **`airmon-ng`**: enable monitor mode.
- **`airodump-ng`**: capture wireless traffic and identify access points and clients.
- **`aireplay-ng`**: frame injection and deauthentication in lab contexts.
- **`aircrack-ng`**: key recovery from captured handshakes.
- **`kismet`**: passive wireless discovery and monitoring.
- **`bettercap`**: wireless and network MITM (man-in-the-middle) workflows in lab settings.

Use passive tools first when the scenario emphasizes stealth or discovery. Use active injection tools only when the question explicitly requires them.

**Cloud, container, and orchestration scenarios**

Cloud and container questions often focus on misconfiguration, exposed metadata, or weak access control.

- **`awscli`**, **`az`**, and **`gcloud`**: cloud control-plane enumeration when credentials are available.
- **`kubectl`**: Kubernetes cluster interaction and resource inspection.
- **`kube-bench`**: Kubernetes benchmark and hardening checks.
- **`kube-hunter`**: Kubernetes attack surface discovery.
- **`trivy`**: container image and filesystem vulnerability scanning.
- **`docker`**: container inspection, image review, and runtime checks.

Example Kubernetes validation:

```bash
kubectl get pods -A
kubectl get nodes
kube-bench
```

Use `kube-bench` as a standalone command or in a container/job workflow; do not assume a `run` subcommand. Use `kubectl` when the scenario asks for cluster state or permissions. Use `trivy` when the question is about image vulnerabilities or exposed secrets.

**Social engineering and phishing workflows**

Social engineering questions often test whether you understand the difference between payload generation, delivery, and credential capture.

- **SET (Social-Engineer Toolkit)**: phishing workflows, credential capture, payload generation, and social engineering simulations.
- **`msfvenom`**: payload creation for controlled lab testing.
- **`gophish`**: phishing campaign management and tracking in authorized environments.

SET is not just “training-oriented”; it is a penetration-testing framework that can automate phishing and credential-capture workflows. In exam scenarios, choose it when the task is to build or simulate a social engineering attack chain.

**Practical tool-to-task mapping table**

| Task | Best-fit tools | Why |
|---|---|---|
| Find open ports quickly | `masscan`, `nmap` | Speed first, then detail |
| Enumerate DNS records | `dig`, `nslookup`, `dnsrecon` | Direct record and zone checks |
| Discover hidden web content | `gobuster`, `ffuf`, `feroxbuster` | Path and parameter discovery |
| Validate SQL injection | `sqlmap`, Burp Suite | Automated and manual verification |
| Find public exploit for a version | `searchsploit` | Fast offline exploit lookup |
| Deliver a controlled payload | `metasploit`, `msfvenom` | Exploit and payload workflow |
| Crack a known hash | `hashcat`, `john` | Mode-based cracking |
| Avoid lockouts while testing creds | `kerbrute`, `crackmapexec` | Spraying workflows |
| Enumerate Windows host weaknesses | `winPEAS`, `Seatbelt` | Local privilege escalation checks |
| Enumerate Linux host weaknesses | `linPEAS`, `sudo -l`, `find` | Sudo and file permission review |
| Reach an internal subnet | `ssh -D`, `chisel`, `proxychains` | Pivoting and routing |
| Inspect logs after an alert | `grep`, `journalctl`, Event Viewer, SIEM | Triage and correlation |

**Common exam traps and how to avoid them**

- Do not choose an active scanner when the question asks for passive discovery.
- Do not choose an exploit framework when the question only asks for validation or enumeration.
- Do not crack hashes before identifying the hash type and selecting the correct mode.
- Do not use brute force when the scenario implies lockout-safe spraying.
- Do not assume Community edition Burp has active scanning.
- Do not use a tool without checking whether the environment is Windows, Linux, web, DNS, wireless, or cloud.
- Do not ignore network constraints such as VPN access, proxies, or pivot requirements.

**Quick decision checklist**

Before answering a scenario question, ask:

- What is the target service or platform?
- What phase of testing am I in?
- Do I need passive, active, or exploit-based validation?
- Is the environment local, remote, segmented, or proxied?
- Is the task about discovery, access, escalation, persistence, or evidence?

If you can answer those five questions, the correct tool usually becomes obvious.
