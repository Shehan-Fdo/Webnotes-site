## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Cloud logging, monitoring, and posture management**

Cloud environments generate high-volume telemetry across identities, APIs, workloads, networks, and managed services. For CEH v13, you need to understand how this telemetry supports detection, investigation, compliance, and hardening. The exam focus is not just “what logs exist,” but how you use them to spot suspicious behavior, correlate events, and reduce risk through posture management.

A useful mental model is:

- **Logging** records what happened.
- **Monitoring** watches for patterns, thresholds, and anomalies.
- **Posture management** measures and improves the security state of cloud assets against baselines and policy.

In practice, these functions overlap. A suspicious API call may trigger a metric alert, appear in a SIEM (Security Information and Event Management) correlation rule, and also reveal a posture issue such as public storage exposure or overly permissive IAM (Identity and Access Management) policy.

**Cloud shared responsibility for logging and monitoring**

Cloud providers and customers split responsibility differently depending on the service model.

- In **IaaS** (Infrastructure as a Service), you usually manage guest OS logs, application logs, agent deployment, and many security controls.
- In **PaaS** (Platform as a Service), the provider manages more of the platform, but you still need to configure diagnostic logs, access logs, and alerting.
- In **SaaS** (Software as a Service), you often rely on the vendor’s audit logs, admin logs, and API logs, while you control identity, access, and retention settings where available.

This matters because missing logs are often a configuration problem, not a provider failure. Exam questions may ask which party is responsible for enabling or retaining a specific log source.

**Core cloud-native monitoring services**

Cloud-native monitoring platforms centralize metrics, logs, traces, and alerts. Know the major services and their roles.

- **AWS CloudWatch**
  - Collects metrics, logs, and events from AWS resources and applications.
  - Supports metric alarms, log insights queries, dashboards, and anomaly detection.
  - Commonly used for EC2, Lambda, RDS, API Gateway, and custom application telemetry.
- **AWS CloudTrail**
  - Records AWS API activity and account actions.
  - Essential for auditing who did what, when, from where, and with which parameters.
  - Often feeds CloudWatch Logs, SIEM tools, and detective controls.
- **Azure Monitor**
  - Central monitoring service for metrics, logs, and alerts across Azure resources.
  - Integrates with activity logs, platform logs, and application telemetry.
- **Log Analytics**
  - Query engine and workspace in Azure Monitor for collecting and analyzing log data.
  - Uses KQL (Kusto Query Language) for searching and correlation.
- **Google Cloud Operations Suite**
  - Formerly Stackdriver; includes Cloud Logging, Cloud Monitoring, Error Reporting, Trace, and Profiler.
  - Cloud Audit Logs are critical for administrative and data access visibility.
- **Cloud threat detection services**
  - **AWS GuardDuty** detects suspicious activity using threat intelligence, anomaly detection, and behavior analysis.
  - **Microsoft Defender for Cloud** provides security posture management and threat protection across Azure and multicloud environments.
  - **Google Security Command Center** centralizes security findings, asset inventory, and risk visibility.

These services are often the first layer before logs are exported to a SIEM or SOAR platform.

**Important log types and fields**

Exam questions often focus on which fields help you investigate an incident. You should recognize both the log source and the key attributes.

- **Authentication logs**
  - Success and failure events
  - Username or principal
  - Source IP address
  - Device or user agent
  - MFA (Multi-Factor Authentication) result
  - Timestamp and geolocation if available
- **Authorization and access logs**
  - Resource ID
  - Action or operation name
  - Permission decision
  - Role or policy used
  - Session ID or token identifier
- **API and control plane logs**
  - Request ID
  - Caller identity
  - Service name
  - Parameters and response code
  - Event outcome
- **Network logs**
  - Source and destination IP
  - Port and protocol
  - Flow direction
  - Bytes transferred
  - Allow/deny decision
- **Workload and application logs**
  - Process name
  - Container or pod ID
  - Error codes
  - Transaction ID
  - Correlation ID
- **Storage and data access logs**
  - Bucket or container name
  - Object path
  - Read/write/delete action
  - Accessing principal
  - Data access outcome

A strong investigation usually depends on joining several of these fields across systems. For example, a failed login followed by a successful login from the same source IP and a new API key creation is more suspicious than any single event alone.

**Log normalization, parsing, and enrichment**

Raw logs from different clouds and services rarely share the same format. Before correlation, you usually normalize them into a common schema.

- **Parsing** extracts fields from JSON, CSV, syslog, or vendor-specific formats.
- **Normalization** maps different field names to a standard model, such as `src_ip`, `user`, `action`, or `event_outcome`.
- **Enrichment** adds context such as asset criticality, geolocation, threat intelligence, identity risk, or business unit.
- **Correlation** links events across identity, network, endpoint, and workload sources.

This is a core SIEM function. Without normalization, a rule looking for “failed authentication from unusual country” may miss equivalent events because one source uses `clientIP`, another uses `source.address`, and another uses `src`.

Example normalization mapping:

```json
{
  "raw_event": {
    "userPrincipalName": "alice@example.com",
    "clientIP": "203.0.113.10",
    "operationName": "CreateAccessKey",
    "result": "Success"
  },
  "normalized_event": {
    "user": "alice@example.com",
    "src_ip": "203.0.113.10",
    "action": "CreateAccessKey",
    "event_outcome": "success"
  }
}
```

Enrichment makes alerts more actionable. A login from a foreign IP is more important if the account belongs to an administrator, the asset is internet-facing, or the event occurs outside business hours.

**Monitoring workflows and alerting**

Monitoring is most effective when you combine thresholds, baselines, and behavior-based detection.

- **Metric alerts**
  - Trigger when a numeric value crosses a threshold.
  - Useful for CPU spikes, error rates, failed logins, or denied requests.
- **Log-based alerts**
  - Trigger when a specific event pattern appears in logs.
  - Useful for privilege escalation, new key creation, disabled logging, or public bucket changes.
- **Anomaly detection**
  - Detects deviations from normal behavior.
  - Useful for unusual API volume, impossible travel, or rare administrative actions.

Practical examples:

```bash
# AWS CloudWatch: create a metric alarm for high 5xx errors
aws cloudwatch put-metric-alarm \
  --alarm-name High5xxErrors \
  --metric-name 5XXError \
  --namespace AWS/ApiGateway \
  --statistic Sum \
  --period 300 \
  --evaluation-periods 2 \
  --threshold 50 \
  --comparison-operator GreaterThanThreshold
```

```bash
# Azure Monitor: create a log alert using KQL
az monitor scheduled-query create \
  --name FailedLoginsAlert \
  --resource-group rg-security \
  --scopes /subscriptions/<sub-id>/resourceGroups/rg-security \
  --condition "SigninLogs | where ResultType != 0 | summarize count() by bin(TimeGenerated, 5m) | where count_ > 20" \
  --description "Alert on repeated failed sign-ins"
```

```bash
# Google Cloud: create a log-based metric from audit logs
gcloud logging metrics create suspicious_admin_action \
  --description="Counts sensitive admin actions" \
  --log-filter='resource.type="gce_instance" AND protoPayload.methodName:"setIamPolicy"'
```

Alert tuning matters. Too many low-value alerts create fatigue, while overly broad thresholds miss real attacks. Use severity levels, suppression windows, and context-aware conditions.

**SIEM and SOAR in cloud log analysis**

A SIEM ingests logs from cloud and on-prem sources, normalizes them, correlates events, and generates alerts. A SOAR (Security Orchestration, Automation, and Response) platform automates response actions such as ticket creation, account disablement, or IP blocking.

A typical workflow is:

1. Ingest cloud audit, identity, network, and workload logs into the SIEM.
2. Normalize fields and enrich with asset and threat intelligence data.
3. Correlate events into a single incident.
4. Triage the alert based on severity, confidence, and business impact.
5. Trigger SOAR playbooks for containment and notification.
6. Preserve evidence and document remediation.

Common correlation examples:

- Multiple failed logins followed by a success from a new location.
- Creation of a new access key after a privilege escalation event.
- Public storage exposure followed by large data egress.
- Disabled logging or tampered audit settings before suspicious activity.

A good SIEM rule does not rely on one event alone. It combines identity, network, and workload signals to reduce false positives.

**Log analysis workflow**

When you investigate a cloud incident, use a repeatable workflow.

1. **Identify the trigger**
   - Determine whether the alert came from a metric, log rule, anomaly, or user report.
2. **Scope the event**
   - Find the affected account, resource, region, subscription, project, or tenant.
3. **Correlate related logs**
   - Check identity, API, network flow, and workload logs around the same time window.
4. **Validate the outcome**
   - Confirm whether the action succeeded, failed, or partially completed.
5. **Assess impact**
   - Determine whether data was accessed, configuration changed, or persistence established.
6. **Contain and remediate**
   - Revoke credentials, isolate workloads, roll back changes, and patch the control gap.
7. **Preserve evidence**
   - Export logs, snapshot relevant resources, and maintain chain of custody where required.

Example investigation pattern:

- A user reports unusual activity.
- You find a successful login after several failures from a new IP.
- CloudTrail or equivalent audit logs show `CreateAccessKey`.
- Network logs show outbound traffic to an unfamiliar destination.
- Posture findings reveal the account had excessive permissions.
- You disable the key, reset credentials, review IAM policy, and open an incident ticket.

**Cloud security posture management**

CSPM (Cloud Security Posture Management) continuously evaluates cloud configurations against security baselines and policy rules. It helps you find misconfigurations before attackers exploit them.

Key CSPM capabilities:

- Detects public storage, open security groups, and overly permissive IAM roles.
- Compares resources against benchmark frameworks and compliance standards.
- Prioritizes findings using risk scoring and asset criticality.
- Tracks drift from approved configurations over time.

Common baseline and benchmark references include:

- CIS (Center for Internet Security) Benchmarks
- NIST (National Institute of Standards and Technology) guidance
- ISO 27001 control mapping
- PCI DSS (Payment Card Industry Data Security Standard) for payment environments
- HIPAA (Health Insurance Portability and Accountability Act) for healthcare data
- SOC 2 control expectations in service organizations

Posture tools often map findings to controls such as encryption, logging, identity hardening, and network segmentation. For exam purposes, remember that a posture finding is not just a technical issue; it is also a compliance and governance signal.

**Risk scoring, scorecards, and prioritization**

Modern posture platforms often present a scorecard or risk score to help teams prioritize remediation.

- **Security scorecards**
  - Summarize overall cloud hygiene in a single score or grade.
  - Useful for executive reporting and trend tracking.
- **Risk scoring**
  - Weighs severity, exploitability, exposure, and asset value.
  - Helps you focus on internet-facing, high-impact misconfigurations first.
- **Prioritization engines**
  - Rank findings by likely attack path, blast radius, and business context.
  - Reduce noise by surfacing the issues most likely to matter operationally.

A low-severity issue on a critical production account may deserve faster action than a high-severity issue on an isolated test environment. That is the practical value of prioritization.

**Compliance, retention, and tiered storage**

Retention is both a security and legal requirement. Different log types often need different retention periods based on regulation, investigation needs, and storage cost.

- **Authentication and admin logs**
  - Often retained longer because they are central to investigations and audits.
- **Network flow logs**
  - May be kept for shorter operational windows unless required for forensic analysis.
- **Application logs**
  - Retention depends on business need, incident response, and privacy rules.
- **Audit logs**
  - Usually retained longer due to compliance and non-repudiation needs.

Retention tiers commonly work like this:

- **Hot storage**
  - Fast access for active investigations and dashboards.
  - Higher cost.
- **Warm storage**
  - Slower but still searchable for routine analysis.
- **Cold storage**
  - Low-cost archival for long-term retention and legal hold.

Examples of retention drivers:

- PCI DSS environments often require strong auditability and longer retention for security events.
- Financial services may retain logs to satisfy regulatory review and fraud investigations.
- Privacy laws may require minimizing retention of personal data unless there is a lawful basis.

> Be careful not to confuse retention with backup. Retention keeps logs available for analysis and compliance; backup protects against loss.

**Log integrity and tamper resistance**

Security logs are only useful if you can trust them. Attackers often try to disable logging, delete records, or alter timestamps.

Integrity controls include:

- **Hash chaining**
  - Each record includes a hash of the previous record, making tampering detectable.
- **Checksum validation**
  - Verifies that archived logs have not changed since export.
- **Immutable storage**
  - Uses write-once or object-lock features to prevent deletion or modification.
- **Periodic audit**
  - Confirms retention settings, immutability policies, and access controls remain intact.

Practical checks:

```bash
# Example: verify a downloaded archive checksum
sha256sum -c cloud-audit-logs.sha256
```

```bash
# Example: inspect an S3 Object Lock configuration
aws s3api get-object-lock-configuration --bucket security-archive-bucket
```

If logs are stored in a SIEM or archive, ensure only a limited set of administrators can modify retention, delete indexes, or disable ingestion.

You should be able to map common services to their security roles.

- **AWS**
  - CloudTrail for API auditing.
  - CloudWatch for metrics, logs, and alarms.
  - GuardDuty for threat detection.
  - Security Hub for posture aggregation and findings.
- **Azure**
  - Activity Log for subscription-level events.
  - Azure Monitor and Log Analytics for collection and query.
  - Microsoft Defender for Cloud for posture and threat protection.
  - Microsoft Sentinel as a SIEM/SOAR platform.
- **Google Cloud**
  - Cloud Audit Logs for administrative and data access events.

  - Security Command Center for risk and findings.

A common exam pattern is to ask which service provides audit trails versus which service provides alerting or posture scoring. Audit logs answer “what happened,” while posture tools answer “what is misconfigured or risky.”

**Practical detection and response example**

A realistic end-to-end workflow helps connect the concepts.

1. A CloudWatch alarm detects a spike in `AccessDenied` events.
2. CloudTrail shows repeated failed `AssumeRole` attempts from an unfamiliar IP.
3. Azure Monitor or Cloud Logging equivalents would show the same pattern in their respective environments through audit and sign-in logs.
4. The SIEM correlates the failures with a successful login from the same source and a new API key creation.
5. A posture platform flags the account as overprivileged and notes that MFA was not enforced.
6. SOAR disables the key, revokes sessions, and opens an incident ticket.
7. The team reviews retention, confirms logs are immutable, and updates the baseline to require MFA and least privilege.

This sequence shows how monitoring, logging, and posture management work together. Detection alone is not enough; you need investigation context and remediation controls.

**Common misconfigurations and attack indicators**

- Logging disabled on critical accounts or subscriptions.
- Audit logs not exported to a central archive.
- Public storage buckets or containers.
- Overly broad IAM roles and service principals.
- Missing alerts for privilege escalation or key creation.
- No anomaly detection on admin activity.
- Retention too short for investigations.
- No immutability or checksum validation for archived logs.
- No correlation between identity and network telemetry.

Attack indicators often include:

- Repeated authentication failures followed by success.
- Unusual user agent strings or impossible travel.
- New access keys, tokens, or service accounts.
- Changes to logging configuration.
- Sudden spikes in data egress.
- Policy changes that weaken security controls.

**Quick reference**

| Area | What to look for | Why it matters |
|---|---|---|
| Identity logs | Success/failure, MFA, source IP, user agent | Detect account abuse and brute force |
| API logs | Request ID, action, outcome, caller | Trace administrative activity |
| Network logs | Src/dst IP, port, protocol, bytes | Spot exfiltration and lateral movement |
| Posture findings | Public exposure, weak IAM, missing encryption | Reduce attack surface |
| SIEM correlation | Multi-source event linkage | Improve confidence and reduce false positives |
| SOAR response | Automated containment | Speed up incident handling |

**Exam tips**

- Know the difference between **CloudTrail**, **CloudWatch**, **Azure Monitor**, **Log Analytics**, and **Cloud Logging**.
- Remember that **audit logs** are essential for attribution and forensics.
- Expect questions on **metric alerts**, **log-based alerts**, and **anomaly detection**.
- Be able to explain **normalization**, **enrichment**, and **correlation** in a SIEM.
- Understand how **CSPM** uses baselines, benchmarks, and risk scoring to prioritize fixes.
- Recognize that retention, immutability, and integrity checks are part of trustworthy logging.
