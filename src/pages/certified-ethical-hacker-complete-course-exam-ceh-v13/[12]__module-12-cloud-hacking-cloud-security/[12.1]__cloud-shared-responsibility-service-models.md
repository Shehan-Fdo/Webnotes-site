## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Cloud shared responsibility model**

The **shared responsibility model** defines which security, compliance, and operational tasks belong to the cloud provider and which remain with you, the customer. The exact split depends on the service model and the provider, but the core idea stays the same: the provider secures the cloud, while you secure what you put in the cloud and how you configure it.

This model matters in CEH because many cloud incidents are not caused by a provider breach. They come from misconfiguration, weak identity controls, exposed storage, poor logging, or unclear ownership of data protection tasks.

- **Provider responsibility** usually covers the physical data center, hardware, networking, hypervisor, and core managed service platform.
- **Customer responsibility** usually covers identities, access policies, data classification, tenant settings, application configuration, and workload hardening.
- **Shared responsibility** means both sides contribute to security outcomes, but at different layers.

**Why the model matters**

You use the model to answer exam questions about who secures what, but also to design controls correctly. If you assume the provider handles everything, you may leave storage public, skip logging, or fail to encrypt sensitive data. If you assume the customer controls everything, you may waste effort trying to patch layers you do not own.

> <mark>Always verify the exact boundary for the specific provider and service.</mark> AWS, Azure, and Google Cloud Platform (GCP) do not draw every line in the same place, and managed services shift more responsibility to the provider than raw infrastructure.

**Core responsibility areas**

A practical way to think about the model is by control layer:

- **Physical security**: provider owns facilities, guards, power, cooling, and hardware replacement.
- **Network infrastructure**: provider owns backbone, routers, switches, and core fabric; you own virtual network design and security groups or firewall rules.
- **Compute platform**: provider owns the host and hypervisor; you own guest operating systems in IaaS.
- **Identity and access management (IAM)**: you usually own users, roles, MFA, conditional access, and least privilege; the provider may supply the IAM service or integrate with your external identity provider.
- **Data**: you own data classification, retention, encryption choices, key management decisions, and access governance.
- **Applications**: you own application code and secure configuration in IaaS and often much of PaaS; in SaaS you usually control tenant-level settings rather than the application internals.
- **Monitoring and logging**: responsibility is layered. In IaaS, you manage guest OS and application logs while the provider manages infrastructure logs. In SaaS, the provider often owns most logging capability, and you consume the logs or audit events that are exposed.
- **Compliance and legal obligations**: the provider may supply attestations and audit reports, but you remain responsible for meeting your own regulatory, contractual, and data governance obligations.

**Service model comparison**

The service model determines how much of the stack you manage. The more managed the service, the less you control at the infrastructure layer, but the more important your configuration and governance become.

| Model | Provider typically manages | Customer typically manages | Security focus |
|---|---|---|---|
| **IaaS** | Physical hosts, storage fabric, networking, hypervisor | Guest OS, patches, middleware, apps, data, IAM, network rules | Harden VMs, patch OS, restrict ports, monitor logs |
| **PaaS** | Infrastructure, OS, runtime, middleware, scaling platform | App code, data, identities, tenant settings, secrets, access policies | Secure code, protect data, configure identity and network exposure |
| **SaaS** | Application, platform, infrastructure, availability, most logging features | User access, tenant settings, data governance, sharing controls, retention, conditional access via IdP | Control access, configure tenants, review audit logs, govern data |
| **Serverless / FaaS** | Runtime, scaling, underlying infrastructure | Function code, triggers, permissions, secrets, event sources, data | Least privilege, event validation, secret handling, observability |
| **CaaS / managed containers** | Container orchestration platform or much of the control plane, depending on service | Container image, app code, secrets, pod/task config, IAM, network policy | Image hygiene, runtime permissions, supply chain security |

**IaaS responsibility boundaries**

Infrastructure as a Service (IaaS) gives you the most control and the most operational burden. You manage the guest operating system, installed software, patching, and application security. The provider manages the physical and virtualization layers.

Typical IaaS tasks you own:

- Patch the guest OS and installed packages.
- Configure host firewalls and security groups.
- Harden SSH, RDP, and remote administration.
- Install and maintain endpoint protection where appropriate.
- Encrypt data at rest and in transit.
- Collect and forward logs to a central SIEM (Security Information and Event Management) platform.

Example AWS security group rule:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --ip-permissions IpProtocol=tcp,FromPort=443,ToPort=443,IpRanges='[{CidrIp=203.0.113.0/24,Description="HTTPS from office"}]'
```

Example Azure Network Security Group rule:

```bash
az network nsg rule create \
  --resource-group rg-prod \
  --nsg-name nsg-web \
  --name Allow-HTTPS \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 203.0.113.0/24 \
  --destination-port-ranges 443
```

In IaaS, logging is not fully shared in a vague sense. The provider logs control-plane and infrastructure events, while you must enable guest logging, application logging, and log forwarding. If you do not configure these, you may have no usable evidence after an incident.

**PaaS responsibility boundaries**

Platform as a Service (PaaS) removes much of the OS and runtime management. The provider patches the platform, manages scaling, and maintains the runtime. You focus on application logic, data, identity integration, and secure configuration.

PaaS is useful when you want to reduce patching overhead and accelerate development, but it increases the importance of secure deployment settings. Common mistakes include:

- Exposing a database or app service to the public internet without need.
- Storing secrets in code instead of a secret manager.
- Assuming the platform encrypts everything in the way your policy requires.
- Ignoring platform-specific identity and access settings.

Example Azure App Service configuration concept:

```bash
az webapp config appsettings set \
  --resource-group rg-app \
  --name myapp-prod \
  --settings \
    "WEBSITE_RUN_FROM_PACKAGE=1" \
    "APPINSIGHTS_INSTRUMENTATIONKEY=<key>" \
    "DB_CONNECTION_STRING=@Microsoft.KeyVault(SecretUri=https://kv-prod.vault.azure.net/secrets/db-conn/)"
```

Example Google Cloud Run deployment with a managed service account:

```bash
gcloud run deploy my-service \
  --image gcr.io/my-project/my-image:latest \
  --region us-central1 \
  --service-account my-service-sa@my-project.iam.gserviceaccount.com \
  --allow-unauthenticated=false
```

In PaaS, the provider usually owns the runtime patching, but you still own secure code, dependency management, and access to the service. If the platform offers built-in identity features, you configure them; if it integrates with an external identity provider, you still own the trust relationship and policy design.

**SaaS responsibility boundaries**

Software as a Service (SaaS) shifts the most responsibility to the provider. The provider runs the application, platform, and infrastructure. You usually manage tenant-level settings, user access, data governance, and integrations.

SaaS does not mean you have no security work. You still need to:

- Configure users, groups, roles, and MFA.
- Set sharing and collaboration controls.
- Restrict external sharing and guest access.
- Review audit logs and alerting.
- Define retention, legal hold, and data export requirements.
- Integrate with your identity provider for SSO (Single Sign-On) using SAML or OIDC (OpenID Connect), when supported.

Practical SaaS tenant examples:

- In Microsoft 365, you may disable anonymous sharing links, require MFA, and restrict external collaboration.
- In Salesforce, you may configure profile permissions, session policies, and login IP ranges.
- In Google Workspace, you may control Drive sharing, OAuth app access, and admin audit logging.

Example SaaS-style tenant control concept:

```text
Tenant settings:
- External sharing: Disabled
- MFA required: Yes
- Guest access: Restricted to approved domains
- Audit log retention: 365 days
- Data export: Admin approval required
```

> In SaaS, the provider often controls the application itself, but you still control the tenant configuration surface exposed to you. Do not assume you can patch or harden the application internals.

**Serverless and managed container models**

Serverless and managed container services sit between PaaS and SaaS in terms of responsibility. They reduce infrastructure management, but they do not remove application security responsibility.

For **Function as a Service (FaaS)**, you typically manage:

- Function code.
- Event triggers.
- IAM permissions.
- Secrets and environment variables.
- Input validation and output handling.
- Logging and tracing configuration.

Example AWS Lambda permission and environment configuration:

```bash
aws lambda update-function-configuration \
  --function-name process-orders \
  --environment "Variables={LOG_LEVEL=info,STAGE=prod}"
```

For **managed containers** or **Container as a Service (CaaS)**, you often manage:

- Container image contents.
- Base image selection and patching.
- Orchestration manifests.
- Secrets injection.
- Network policy.
- Service account permissions.

Example Kubernetes deployment snippet:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  template:
    spec:
      serviceAccountName: api-sa
      containers:
        - name: api
          image: registry.example.com/api:1.2.3
          ports:
            - containerPort: 8080
```

The provider may manage the control plane, node infrastructure, or both, depending on the service. Your responsibility is to understand which layer is abstracted and which layer remains under your control.

**Compliance, governance, and legal responsibility**

Cloud providers often publish certifications, audit reports, and compliance attestations such as ISO 27001, SOC 2, or PCI DSS-related documentation. These are useful, but they do not transfer your obligations.

You remain responsible for:

- Classifying data correctly.
- Choosing lawful processing and retention rules.
- Meeting sector-specific regulations.
- Maintaining evidence for audits.
- Configuring controls that support policy enforcement.
- Demonstrating that your own use of the cloud matches your obligations.

This is a common exam trap: provider compliance does not equal customer compliance. A provider may be certified, but you can still violate policy by misconfiguring access, storing regulated data in the wrong region, or failing to retain logs.

**Disaster recovery and business continuity**

Disaster recovery (DR) and business continuity are shared, but the split is often misunderstood. The provider may offer resilient infrastructure, multi-region services, backups, and failover features. You decide the recovery objectives and implement the application strategy.

Key terms:

- **RTO** (Recovery Time Objective): how long you can tolerate downtime.
- **RPO** (Recovery Point Objective): how much data loss you can tolerate.

Typical ownership:

- You define RTO and RPO based on business needs.
- The provider supplies the platform capabilities to meet them.
- You configure backups, replication, failover, and restore testing.
- You validate that the application can actually recover within the target.

Example backup strategy for a cloud workload:

- Daily database snapshots.
- Cross-region object replication for critical files.
- Infrastructure as code to rebuild environments.
- Periodic restore tests.
- Documented runbooks for failover and rollback.

If you do not test recovery, you do not know whether the design works. A backup that cannot be restored is not a recovery plan.

**Provider differences and service-specific nuance**

The shared responsibility model is not identical across AWS, Azure, and GCP. The broad pattern is similar, but the exact control surface differs by service.

Examples of nuance:

- **AWS** often exposes granular control through IAM, security groups, AWS Config, CloudTrail, and service-specific public access blocks.
- **Azure** often emphasizes Azure RBAC, Azure Policy, Defender for Cloud, resource locks, and tenant-level controls in Microsoft services.
- **GCP** often uses IAM, Organization Policy, Cloud Audit Logs, VPC firewall rules, and service-specific IAM roles.

The same category can shift responsibility depending on the service:

- A managed database service reduces patching responsibility compared with a self-managed VM database.
- A managed Kubernetes service shifts control-plane management to the provider, but you still secure workloads, namespaces, RBAC, and network policy.
- A SaaS collaboration suite gives you tenant controls, but not OS-level access.

**Classic service models versus modern managed services**

Modern cloud services blur the old IaaS/PaaS/SaaS boundaries. Exam questions may describe a service without naming the model, so you should identify the responsibility boundary from the control surface.

| Service type | What you manage more heavily | What the provider abstracts more heavily |
|---|---|---|
| **Classic IaaS VM** | OS, patches, agents, apps, local firewall | Hardware, hypervisor, physical network |
| **Managed database** | Schema, users, queries, backups policy, access | OS, database engine patching, replication plumbing |
| **Managed Kubernetes** | Pods, images, secrets, RBAC, network policy | Control plane, often node lifecycle or much of it |
| **Serverless function** | Code, triggers, permissions, observability | Runtime, scaling, host OS |
| **SaaS app** | Tenant settings, users, data governance | Application, platform, infrastructure |

This comparison helps you answer questions about whether a control belongs to the customer or provider. If you can log into the OS, patch it, or install software, you are likely in an IaaS-style responsibility zone. If you only configure tenant settings or deploy code into a managed runtime, the provider owns more of the stack.

**Security controls you should expect to own**

Even when the provider offers strong security, you still need customer-side controls.

- **Identity governance**: enforce MFA, least privilege, role reviews, and privileged access workflows.
- **Configuration auditing**: use CSPM (Cloud Security Posture Management) tools to detect risky settings.
- **Data protection**: classify data, encrypt sensitive content, and manage keys appropriately.
- **Logging and detection**: centralize logs, enable alerts, and retain evidence.
- **Network exposure control**: restrict public endpoints, segment workloads, and review ingress rules.
- **Backup and recovery**: define and test restore procedures.
- **Change management**: track configuration drift and approve risky changes.

Example CSPM-style workflow:

1. Scan cloud accounts or subscriptions for public storage, open security groups, and missing logging.
2. Prioritize findings by exposure and data sensitivity.
3. Remediate misconfigurations with policy-as-code or manual changes.
4. Re-scan to confirm the risk is closed.
5. Record evidence for audit and compliance review.

Example policy-as-code concept for blocking public storage:

```json
{
  "effect": "deny",
  "condition": {
    "field": "publicAccess",
    "equals": true
  }
}
```

**Practical exam cues**

When you read a scenario, ask these questions:

- Who controls the OS?
- Who patches the runtime?
- Who configures identity and access?
- Who owns the data?
- Who can change tenant settings?
- Who can see logs?
- Who defines recovery objectives?
- Who proves compliance?

If the scenario mentions a provider-managed service, assume the provider handles more of the lower layers. If it mentions tenant settings, application code, or access policies, assume the customer still owns those controls.

**Common mistakes to avoid**

- Assuming SaaS means the provider handles all security.
- Assuming compliance attestations remove customer obligations.
- Confusing infrastructure logging with application logging.
- Treating all cloud services as if they have the same responsibility split.
- Forgetting that disaster recovery requires both provider capability and customer planning.
- Ignoring configuration auditing and CSPM in cloud security operations.

**Quick memory aid**

- **IaaS**: you manage the most.
- **PaaS**: you manage the app and data.
- **SaaS**: you manage the tenant and users.
- **Serverless**: you manage code and permissions.
- **Managed services**: the provider manages more of the stack, but you still own governance.
