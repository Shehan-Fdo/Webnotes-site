## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Container security fundamentals**

Containers package an application with its runtime dependencies, but they do not create a full virtual machine boundary. The shared host kernel is the key security distinction: if an attacker escapes a container, they may reach the host or other workloads. For CEH-level understanding, focus on the attack surface created by images, registries, runtimes, orchestration control planes, and misconfigurations.

A secure container posture starts with minimizing what enters the environment and limiting what a running container can do. You reduce risk by using trusted base images, removing unnecessary packages, running as a non-root user, and applying Linux controls such as namespaces, cgroups, seccomp, AppArmor, and SELinux where available.

- **Image** security begins before deployment.
- **Runtime** security limits what a compromised container can access.
- **Orchestration** security protects scheduling, networking, and control-plane access.

**Container architecture and isolation model**

Containers rely on kernel features rather than hardware virtualization. That makes them lightweight and fast, but also means kernel weaknesses can become escape paths. A container typically uses:

- **Namespaces** to isolate process IDs, mounts, networking, users, and IPC.
- **Control groups (cgroups)** to constrain CPU, memory, and I/O usage.
- **Capabilities** to split root privileges into smaller units.
- **Mandatory access control** systems such as AppArmor or SELinux to restrict actions even if a process is compromised.

The security implication is simple: if you weaken the container boundary with privileged mode, broad capabilities, host mounts, or a shared Docker socket, you increase the chance of breakout or lateral movement.

**Common container attack vectors**

Attackers often target the weakest control in the chain rather than the container itself. Common vectors include poisoned images, exposed management interfaces, and privilege escalation through misconfiguration.

- **Kernel escape**: a vulnerable host kernel or container runtime can allow breakout from the container to the host.
- **Misused capabilities**: granting `CAP_SYS_ADMIN` or similar broad privileges can enable mount, namespace, or device abuse.
- **Privileged containers**: `--privileged` disables many isolation benefits and can expose host devices and kernel interfaces.
- **Docker socket abuse**: mounting `/var/run/docker.sock` into a container often gives effective root-equivalent control over the host Docker daemon.
- **HostPath mounts**: mounting sensitive host paths can expose credentials, binaries, or configuration files.
- **Insecure environment variables and secrets**: secrets in env vars, logs, or image layers are easy to leak.
- **Supply-chain compromise**: malicious base images, compromised dependencies, or poisoned build pipelines can insert backdoors before deployment.

> A container compromise is often a stepping stone. The real objective is frequently host takeover, credential theft, or access to the orchestration plane.

**Container escape techniques and breakout paths**

Container escape means breaking out of the isolated container context into the host or another trust zone. In exam scenarios, you should recognize the main breakout patterns rather than memorize a single exploit.

- **Kernel vulnerabilities** can allow privilege escalation from container to host when the host kernel is outdated or unpatched.
- **Overly permissive capabilities** can let an attacker manipulate mounts, network settings, or kernel interfaces.
- **Writable host mounts** can be used to plant SSH keys, modify startup scripts, or tamper with system files.
- **Docker socket access** can be abused to start a new container with host mounts, elevated privileges, or access to the host filesystem.
- **Container runtime misconfiguration** can expose debugging or administrative interfaces that bypass normal isolation.
- **Setuid binaries and weak file permissions** inside the image can be leveraged if the container runs as root or with elevated rights.

Practical example of Docker socket abuse:

```bash
docker run -it --rm \
  -v /var/run/docker.sock:/var/run/docker.sock \
  alpine sh
```

From inside that container, an attacker can often control the host Docker daemon by launching a new privileged container:

```bash
docker -H unix:///var/run/docker.sock run --rm -it \
  --privileged \
  -v /:/host \
  alpine chroot /host sh
```

This is not a guaranteed exploit in every environment, but it is a classic high-risk misconfiguration and a common CEH exam concept.

**Container runtime and daemon security**

The container runtime is the component that creates and manages containers. In Docker environments, the daemon (`dockerd`) is especially sensitive because it often runs as root and controls the lifecycle of all containers. If an attacker reaches the daemon API, they may gain broad control over the host.

Key risks include:

- **Exposed Docker daemon** on TCP without strong authentication and TLS.
- **Docker socket exposure** to containers or untrusted users.
- **Weak daemon configuration** that allows insecure registries, broad privileges, or remote API access.
- **Unrestricted containerd or CRI-O access** through local sockets or misconfigured permissions.
- **Debug endpoints** or management ports bound to all interfaces.

Hardening ideas:

- Keep the daemon socket local and protected by file permissions.
- Use TLS client authentication for any remote Docker API access.
- Restrict who can access `docker`, `containerd`, or CRI sockets.
- Run the host with timely kernel and runtime patching.
- Disable unnecessary experimental or debug features in production.

Useful verification commands:

```bash
ps -ef | grep -E 'dockerd|containerd|crio'
ss -lntp | grep -E '2375|2376|dockerd|containerd|crio'
ls -l /var/run/docker.sock
docker info
```

If you see Docker listening on `2375` without TLS, treat it as a critical exposure. Port `2376` is commonly used for TLS-protected Docker API access, but you still need certificate validation and access control.

**Image security and registry trust**

Container security starts with the image supply chain. Pulling an untrusted image is equivalent to running unreviewed code from the internet. A secure workflow verifies where the image came from, what it contains, and whether it has changed unexpectedly.

Important registry and image controls:

- **Registry authentication** prevents anonymous pushes and reduces unauthorized pulls.
- **Image provenance** helps you confirm who built the image and from which source.
- **Tag immutability** prevents a tag like `latest` or `1.0` from silently changing to a different image.
- **Digest pinning** ensures you deploy a specific image hash rather than a mutable tag.
- **Vulnerability scanning** identifies known CVEs in packages and libraries.
- **Signature verification** helps detect tampering or unauthorized publishing.

Risks of untrusted images:

- Hidden backdoors or cryptominers.
- Outdated packages with known exploits.
- Embedded secrets or malicious startup scripts.
- Dependency confusion or compromised upstream packages.
- Typosquatting, where a malicious image name resembles a legitimate one.

Example of safer image pinning:

```bash
docker pull nginx@sha256:3f5d...
docker run --rm nginx@sha256:3f5d...
```

Example registry login:

```bash
docker login registry.example.com
docker pull registry.example.com/team/app:1.4.2
```

For stronger trust, organizations often use image signing and policy enforcement. The exact tooling varies, but the security goal is consistent: only deploy images that are authenticated, scanned, and approved.

**Supply-chain threats and CI/CD security**

Container environments are only as trustworthy as the pipeline that builds them. If an attacker compromises source control, build agents, dependency repositories, or artifact storage, they can inject malicious code before the image ever reaches production.

Common supply-chain threats:

- **Poisoned base images** with hidden malware or backdoors.
- **Dependency compromise** through malicious packages or dependency confusion.
- **CI/CD pipeline compromise** that alters build scripts, injects secrets, or publishes tampered images.
- **Artifact registry tampering** that replaces approved images with malicious ones.
- **Secret leakage** from build logs, environment variables, or cached layers.

Practical controls:

- Use minimal, trusted base images.
- Pin package versions where possible.
- Scan dependencies and images in the pipeline.
- Separate build, test, and release permissions.
- Store secrets in a dedicated secret manager, not in source code.
- Sign artifacts and verify signatures before deployment.

Example Dockerfile hardening pattern:

```dockerfile
FROM alpine:3.20

RUN addgroup -S app && adduser -S app -G app
WORKDIR /app

COPY --chown=app:app . /app
USER app

ENTRYPOINT ["./app"]
```

This reduces risk by avoiding root execution and limiting writable paths.

**Kubernetes security basics**

Kubernetes introduces a control plane, worker nodes, and a large API surface. The security model depends on strong authentication, authorization, admission control, and network segmentation. A weak cluster can expose workloads, secrets, and infrastructure at scale.

Core components to know:

- **API server**: the front door for all cluster operations.
- **etcd**: the key-value store that holds cluster state and secrets.
- **Scheduler**: assigns pods to nodes.
- **Controller manager**: reconciles desired and actual state.
- **kubelet**: the node agent that manages pods on each worker.
- **kube-proxy**: handles service networking rules on nodes.

Key risks:

- **API server exposure** to the internet or untrusted networks.
- **etcd exposure** without encryption, authentication, or network restriction.
- **kubelet exposure** with anonymous or unauthenticated access.
- **Weak node authorization** that lets workloads or users access node-level resources.
- **Overly permissive RBAC** that grants cluster-admin or broad namespace access.
- **Insecure dashboards or add-ons** exposed without authentication.

Verification commands:

```bash
kubectl cluster-info
kubectl get nodes -o wide
kubectl get pods -A
kubectl get svc -A
kubectl get secrets -A
kubectl auth can-i create pods --as=alice
kubectl auth can-i get secrets -n dev --as=alice
```

You can also inspect cluster configuration and node security posture:

```bash
kubectl describe node <node-name>
kubectl get apiservices
kubectl get events -A --sort-by=.metadata.creationTimestamp
```

**Control-plane hardening**

The control plane is the highest-value target in a Kubernetes environment. If an attacker gains API server access, they may create privileged pods, read secrets, or modify workloads. If they reach etcd, they may extract cluster state and credentials.

Hardening priorities:

- Restrict API server access with network controls and strong authentication.
- Use TLS for all control-plane communication.
- Protect etcd with encryption at rest, access control, and network isolation.
- Limit access to kubelet ports and secure kubelet authentication and authorization.
- Keep control-plane components patched and monitored.
- Disable anonymous access and insecure ports where supported.

etcd protection matters because it often stores secrets and cluster metadata. If etcd is exposed or unencrypted, an attacker may retrieve sensitive data directly.

Example checks:

```bash
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  endpoint health
```

**Authentication, authorization, and RBAC**

Kubernetes uses authentication to identify the caller and authorization to decide what the caller can do. The most common authorization model is RBAC, or Role-Based Access Control.

Important concepts:

- **Authentication** can use client certificates, bearer tokens, service account tokens, or external identity providers.
- **Authorization** determines whether the authenticated identity can perform an action.
- **RBAC** uses Roles, ClusterRoles, RoleBindings, and ClusterRoleBindings.
- **Least privilege** means granting only the permissions required for the task.

Common mistakes:

- Binding users or service accounts to `cluster-admin`.
- Allowing wildcard verbs or resources.
- Reusing service accounts across unrelated workloads.
- Leaving default service account tokens mounted into pods unnecessarily.

Example RBAC inspection:

```bash
kubectl get clusterrolebindings
kubectl describe clusterrolebinding cluster-admin
kubectl get role,rolebinding -n dev
kubectl auth can-i list pods -n dev --as system:serviceaccount:dev:app-sa
```

Example of a minimal Role:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]
```

**Pod Security Standards and Pod Security Admission**

Pod Security Standards define baseline security expectations for pods. Pod Security Admission (PSA) is the built-in admission controller that enforces these standards at the namespace level. This replaced the older PodSecurityPolicy (PSP) approach, which is deprecated and removed in modern Kubernetes versions.

The three PSA levels are:

- **Privileged**: broadest permissions, intended for trusted system workloads.
- **Baseline**: blocks known privilege escalations while allowing common container patterns.
- **Restricted**: enforces the strongest practical hardening, including non-root execution and tighter controls.

Why this matters:

- It prevents accidental deployment of privileged or host-accessing pods.
- It gives you a policy layer before workloads run.
- It is a common exam topic when discussing Kubernetes hardening.

Example namespace labels for PSA:

```bash
kubectl label namespace dev pod-security.kubernetes.io/enforce=restricted
kubectl label namespace dev pod-security.kubernetes.io/audit=restricted
kubectl label namespace dev pod-security.kubernetes.io/warn=restricted
```

Legacy PSP context:

- PSP controlled pod security through cluster-wide policies.
- It was difficult to manage and is no longer the recommended mechanism.
- If you see PSP in older environments or exam questions, recognize it as the predecessor to PSA.

**Pod hardening and workload restrictions**

A secure pod should run with the smallest possible privilege set. The goal is to make compromise harder and limit blast radius if compromise occurs.

Recommended controls:

- Run as a non-root user.
- Drop unnecessary Linux capabilities.
- Set the root filesystem to read-only when possible.
- Avoid privileged mode.
- Avoid host networking, host PID, and host IPC unless required.
- Avoid mounting the Docker socket or sensitive host paths.
- Use seccomp profiles to restrict system calls.
- Use AppArmor or SELinux profiles where supported.

Example hardened pod spec:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: hardened-app
spec:
  securityContext:
    runAsNonRoot: true
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: nginx:1.27
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]
```

**Node security and kubelet basics**

Worker nodes are part of the trust boundary. If a node is compromised, every pod on that node is at risk. The kubelet is the node agent that communicates with the API server and manages pod lifecycle.

Important node security points:

- **kubelet authentication** should not allow anonymous access.
- **kubelet authorization** should be enabled and restrictive.
- **Read-only ports** should be disabled where possible.
- **Node access** should be limited to administrators and automation systems.
- **SSH access** to nodes should be tightly controlled and logged.

Useful checks:

```bash
kubectl get nodes
kubectl describe node <node-name>
curl -k https://<node-ip>:10250/pods
```

If the kubelet API is exposed without proper authentication, an attacker may enumerate pods, inspect workloads, or in some cases interact with node-managed resources. Treat kubelet exposure as a serious finding.

**Network security, ingress, and service exposure**

Containerized applications are often exposed through Services, Ingress controllers, and service meshes. These layers can create security gaps if misconfigured.

Common risks:

- **Exposed dashboards** or admin UIs without authentication.
- **Misconfigured ingress** that routes traffic to internal services.
- **Weak TLS** or missing TLS on public endpoints.
- **Overly permissive network policies** that allow unrestricted pod-to-pod traffic.
- **Insecure service mesh defaults** that leave east-west traffic unencrypted or unauthenticated.

Practical checks:

```bash
kubectl get ingress -A
kubectl describe ingress -n prod web
kubectl get networkpolicy -A
kubectl get svc -A
```

If you use a service mesh, verify mutual TLS, sidecar injection, and policy enforcement. A mesh is not automatically secure; it can also expand complexity and misconfiguration risk.

**Secrets management**

Secrets are often the easiest target in a container environment. Kubernetes Secrets are base64-encoded, not encrypted by default. That means you should not treat them as protected unless encryption at rest and access controls are in place.

Best practices:

- Use external secret managers when possible.
- Enable encryption at rest for Kubernetes Secrets.
- Restrict secret access with RBAC.
- Avoid placing secrets in images, environment variables, or logs.
- Mount secrets only into workloads that need them.

Example secret inspection:

```bash
kubectl get secret -n dev
kubectl describe secret app-secret -n dev
```

Remember that anyone with permission to read secrets can often pivot to databases, cloud APIs, or internal services.

**Practical hardening workflow**

1. Pull only approved images from authenticated registries.
2. Scan images and dependencies before deployment.
3. Enforce non-root, read-only, and capability-drop settings.
4. Apply namespace-level Pod Security Admission labels.
5. Restrict RBAC to least privilege.
6. Protect the API server, etcd, and kubelet endpoints.
7. Validate network exposure with `kubectl get svc`, `kubectl get ingress`, and `kubectl get networkpolicy`.
8. Monitor runtime behavior and audit logs for unusual container or cluster activity.

**Quick verification checklist**

Use these commands to assess a cluster or container host quickly:

```bash
docker info
docker ps
docker inspect <container>
kubectl get nodes -o wide
kubectl get pods -A
kubectl get secrets -A
kubectl auth can-i '*' '*' --as <user>
kubectl get networkpolicy -A
kubectl get ingress -A
```

Look for these red flags:

- Docker daemon exposed on a network interface.
- Containers running as root with privileged mode enabled.
- Mounted Docker socket or sensitive host paths.
- Publicly reachable API server, dashboard, or kubelet.
- Missing RBAC boundaries or broad cluster-admin bindings.
- Untrusted images pulled from unknown registries.
- No Pod Security Admission enforcement in sensitive namespaces.

**Exam focus summary**

For CEH v13, remember the relationship between container weaknesses and orchestration weaknesses. Containers are not inherently secure just because they are isolated. The most important exam themes are:

- **Escape paths** through kernel flaws, capabilities, privileged mode, and Docker socket abuse.
- **Registry trust** through authentication, provenance, immutability, and image verification.
- **Runtime hardening** through daemon protection, socket control, and minimal privileges.
- **Kubernetes control-plane security** through API server, etcd, kubelet, and RBAC protection.
- **Pod security enforcement** through Pod Security Admission and hardened pod specs.
- **Supply-chain defense** through secure CI/CD, dependency control, and signed artifacts.
- **Exposure reduction** through network policies, ingress review, and dashboard/service hardening.
