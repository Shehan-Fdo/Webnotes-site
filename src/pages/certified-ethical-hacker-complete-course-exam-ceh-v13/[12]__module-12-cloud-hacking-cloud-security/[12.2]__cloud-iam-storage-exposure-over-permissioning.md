## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Cloud IAM and storage exposure overview**

Cloud environments concentrate risk in two places: identity and data. **Identity and Access Management (IAM)** controls who can do what, while storage services hold the data attackers want most. When IAM is too broad or storage is exposed, a single mistake can reveal secrets, customer data, backups, or credentials.

The main exam idea is simple: you often do not need to “hack the cloud” in a dramatic way. You exploit weak trust boundaries, excessive permissions, public access settings, leaked keys, and misconfigured storage policies.

- **IAM** defines users, groups, roles, policies, and trust relationships.
- **Storage exposure** usually means public access, weak sharing links, or overly broad read permissions.
- **Over-permissioning** creates lateral movement and privilege escalation paths.
- Misconfiguration is often more common than a software vulnerability.

**Core cloud identity concepts**

Cloud providers use different terminology, but the security model is similar. You authenticate as a principal, receive authorization through policies, and access resources based on explicit allow/deny rules and inherited trust.

<dl>
<dt>Principal</dt>
<dd>An identity that can act in the cloud, such as a user, role, service account, or federated identity.</dd>
<dt>Policy</dt>
<dd>A document that grants or denies permissions on actions and resources.</dd>
<dt>Role</dt>
<dd>A temporary or assumable identity used by users, workloads, or services.</dd>
<dt>Trust policy</dt>
<dd>A rule that defines who can assume a role or use a service identity.</dd>
</dl>

Cloud IAM is not just about human users. Workloads, CI/CD pipelines, containers, serverless functions, and virtual machines also use identities. Attackers often target these non-human identities because they are less monitored and frequently over-privileged.

**How over-permissioning happens**

Over-permissioning usually comes from convenience, not malice. Teams grant broad access to avoid breaking applications, speed up deployment, or simplify troubleshooting. Over time, these permissions accumulate and become difficult to audit.

Common causes include:

- Using wildcard permissions such as `*` for actions or resources.
- Assigning admin-like roles to service accounts or automation jobs.
- Reusing the same identity across multiple applications.
- Granting read access to entire storage accounts or buckets when only one path is needed.
- Allowing role assumption without strict trust conditions.
- Leaving temporary exceptions in place after incidents or migrations.

A useful mental model is least privilege: each identity should have only the permissions required for its current task, for the shortest practical time.

**Cloud storage exposure patterns**

Storage exposure is one of the most common cloud security failures. The exact service differs by provider, but the patterns are similar: object storage, file shares, and backups become reachable by unintended users.

Typical exposure scenarios include:

- Public bucket or container access.
- Anonymous read permissions on objects.
- Shared access links that never expire.
- Misconfigured ACLs (Access Control Lists) or bucket policies.
- Overly broad IAM permissions that allow listing, reading, or writing data.
- Sensitive data stored in backups, logs, or snapshots.

Object storage often separates permissions into bucket/container-level and object-level actions. That distinction matters because an attacker may be able to read an object if they know its name, even if they cannot list the container contents. Conversely, list permissions can reveal filenames, project names, and sensitive structure even without object read access.

**AWS S3 exposure and IAM**

Amazon S3 is a common target because it is widely used for backups, static assets, logs, and data exports. Access can be controlled through IAM policies, bucket policies, ACLs, and account-level public access settings.

Important S3 concepts:

- **Bucket policy** controls access at the bucket and object level for the bucket it is attached to.
- **IAM policy** controls what a principal can do across resources.
- **ACLs** are legacy-style access controls and are generally discouraged for new designs.
- **Block Public Access** can override or prevent public exposure even when a policy appears to allow it.

A bucket can look public in policy terms but still be inaccessible because Block Public Access is enabled at the account or bucket level. For exam questions, always check the full access path: policy, ACL, and public access blocking.

Example S3 policy that grants read access to a specific prefix:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlyReports",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::example-bucket/reports/*"
    }
  ]
}
```

This policy allows object reads under `reports/`, but it does not automatically allow listing the bucket. If you need listing, that is a separate permission such as `s3:ListBucket` on the bucket ARN.

Useful AWS CLI checks:

```bash
aws s3api get-bucket-policy --bucket example-bucket
aws s3api get-bucket-acl --bucket example-bucket
aws s3api get-public-access-block --bucket example-bucket
aws s3api list-objects-v2 --bucket example-bucket --max-items 10
```

Practical interpretation:

- `get-bucket-policy` shows whether the bucket policy grants access.
- `get-bucket-acl` helps identify legacy public grants.
- `get-public-access-block` shows whether public exposure is blocked.
- `list-objects-v2` only works if your caller already has list permission.

> Do not assume a bucket is publicly enumerable just because it contains public objects. Listing and reading are separate permissions.

**AWS IAM over-permissioning examples**

A common mistake is attaching broad managed policies to users or roles. For example, `AdministratorAccess` or wildcard permissions can turn a low-risk identity into a full compromise path.

Examples of risky permissions:

- `s3:*` on `*`
- `iam:*` on `*`
- `sts:AssumeRole` without tight trust conditions
- `kms:*` on all keys
- `ec2:*` on all resources

A safer pattern is to scope both action and resource:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::app-uploads/*"
    }
  ]
}
```

This still may be too broad for production, but it is much better than granting access to every bucket in the account.

**Azure storage exposure and SAS**

Azure Blob Storage commonly uses Azure role-based access control (RBAC), shared access signatures (SAS), and access keys. A SAS token is a time-bound URL parameter that grants limited access to a storage resource.

Key Azure terms:

- **SAS**: Shared Access Signature.
- **RBAC**: Role-Based Access Control.
- **Container**: A grouping of blobs, similar to a bucket.
- **Blob**: An object stored in Azure Blob Storage.

A SAS token can grant read access to a blob or container, and list access is meaningful at the container level. If a URL points to a single blob, `r` is the relevant permission; `l` matters when the token applies to a container and the attacker wants to enumerate blobs.

Example SAS-style URL:

```text
https://examplestorage.blob.core.windows.net/reports/q4.xlsx?sv=2023-11-03&sp=r&se=2026-05-01T00:00:00Z&sig=...
```

Security implications:

- A long expiry increases exposure.
- A broad `sp` value increases what the token can do.
- A container-level SAS can reveal filenames and object structure.
- Shared URLs are often copied into chat, tickets, or code repositories.

Azure RBAC example for storage access:

```json
{
  "properties": {
    "roleDefinitionId": "/subscriptions/00000000-0000-0000-0000-000000000000/providers/Microsoft.Authorization/roleDefinitions/ba92f5b4-2d11-453d-a403-e96b0029c9fe",
    "principalId": "11111111-1111-1111-1111-111111111111",
    "scope": "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-demo/providers/Microsoft.Storage/storageAccounts/demoacct"
  }
}
```

That example represents a storage data role assignment at the storage account scope. In practice, scope matters: a role at the account level is broader than one at a container or blob scope.

**GCP IAM and storage exposure**

Google Cloud Platform (GCP) uses IAM policies attached to projects, folders, organizations, and resources. Cloud Storage buckets can be exposed through IAM bindings, legacy ACLs, or public access settings.

Important GCP concepts:

- **Service account**: A non-human identity used by applications and workloads.
- **IAM binding**: A mapping between a role and a member.
- **Uniform bucket-level access**: A setting that centralizes bucket permissions in IAM.
- **Public access prevention**: A control that blocks public grants.

Example bucket IAM binding:

```json
{
  "bindings": [
    {
      "role": "roles/storage.objectViewer",
      "members": [
        "allUsers"
      ]
    }
  ]
}
```

This grants public read access to objects in the bucket. If public access prevention is enabled, the effective exposure may be blocked even if the policy appears permissive.

Useful GCP CLI checks:

```bash
gcloud storage buckets describe gs://example-bucket
gcloud storage buckets get-iam-policy gs://example-bucket
gcloud storage ls gs://example-bucket
gcloud projects get-iam-policy example-project
```

Interpretation:

- `describe` shows bucket settings such as public access prevention and uniform access.
- `get-iam-policy` shows who can access the bucket.
- `ls` lists objects only if your identity has permission.
- Project-level IAM can indirectly grant access to storage resources.

**Service accounts and workload identity risk**

Service accounts are often more powerful than human users because they are used by automation and workloads. If an attacker compromises a workload that has a privileged service account, the attacker may inherit that workload’s permissions.

Common abuse paths include:

- Reading service account keys from disk, environment variables, or source control.
- Querying instance metadata services from a compromised VM or container.
- Abusing overly broad workload identities.
- Using a compromised CI/CD runner to access cloud APIs.

Example of a risky pattern in a VM environment:

```bash
curl -H "Metadata-Flavor: Google" \
  http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token
```

This kind of request is only possible from within the workload environment and depends on platform protections. The key lesson is that metadata services can expose temporary credentials if the attacker gains code execution on the instance.

**Privilege escalation through IAM**

Privilege escalation in cloud often means moving from a low-privilege identity to a higher-privilege one by abusing permissions, trust relationships, or attached roles.

Common escalation patterns:

- Assuming a role that trusts your current identity.
- Modifying a policy attached to a role you can edit.
- Attaching a more privileged policy to a user, group, or role.
- Creating or downloading service account keys if allowed.
- Changing the permissions boundary or trust policy if you have write access.
- Using a workload identity to reach metadata-based credentials.

A role-assumption example in AWS:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/ReadOnlyReports \
  --role-session-name audit-session
```

This only works if the role trust policy allows your principal to assume it. The presence of `sts:AssumeRole` in your identity policy is not enough by itself.

**Detection and assessment workflow**

When assessing cloud exposure, you should verify both configuration and effective access. A policy that looks permissive may still be blocked by a higher-level control, while a restrictive policy may be bypassed through another identity path.

A practical assessment flow:

1. Identify the storage resource and its scope.
2. Review IAM policies, bucket policies, ACLs, and public access controls.
3. Check whether anonymous or cross-account access is possible.
4. Test read, list, and write actions separately.
5. Review attached identities, service accounts, and trust relationships.
6. Look for secrets, backups, logs, and exports in exposed storage.
7. Validate whether temporary credentials or SAS tokens are still active.

Example AWS enumeration commands:

```bash
aws sts get-caller-identity
aws s3api get-bucket-policy-status --bucket example-bucket
aws s3api get-bucket-acl --bucket example-bucket
aws s3api get-public-access-block --bucket example-bucket
```

Example Azure checks:

```bash
az storage account show --name examplestorage --resource-group rg-demo
az storage container show-permission --account-name examplestorage --name reports
```

Example GCP checks:

```bash
gcloud storage buckets get-iam-policy gs://example-bucket
gcloud storage buckets describe gs://example-bucket
```

**Indicators of exposure**

Look for signs that storage or IAM is too open:

- Public URLs that do not require authentication.
- Buckets or containers with `allUsers`, `AuthenticatedUsers`, or equivalent public principals.
- Long-lived SAS tokens in code, tickets, or logs.
- Service accounts with broad project-wide roles.
- Unused identities that still have access.
- Cross-account trust that is wider than necessary.
- Backups stored in the same account with weak controls.

**Defensive controls**

Good cloud security reduces both the chance and impact of exposure. The strongest controls combine identity hardening, storage restrictions, and monitoring.

Recommended controls:

- Enforce least privilege on all identities.
- Use short-lived credentials instead of long-lived keys.
- Enable public access prevention or Block Public Access where available.
- Prefer bucket-level IAM over object ACL sprawl.
- Rotate and scope SAS tokens tightly.
- Separate production, staging, and backup accounts or projects.
- Log access to storage and IAM changes.
- Review trust policies and role assumptions regularly.
- Use conditional access, MFA, and device or network restrictions where supported.

A simple hardening checklist for storage:

- Disable public access unless explicitly required.
- Remove legacy ACL grants.
- Restrict list permissions when only object reads are needed.
- Limit write permissions to trusted automation.
- Encrypt sensitive data at rest and in transit.
- Monitor for unusual download volume or access patterns.

**Real-world attack chain example**

A common compromise chain looks like this:

- An application uses a service account with read access to a storage bucket.
- The service account key is exposed in a repository or container image.
- The attacker authenticates to the cloud API using that key.
- The attacker enumerates storage and finds backups or secrets.
- The attacker uses the secrets to access a database or another cloud service.
- The attacker escalates by finding a role or policy that allows broader access.

This chain shows why cloud incidents often start with one weak identity and end with full environment compromise.

**Exam focus points**

For CEH-style questions, focus on the relationship between identity and exposure:

- **IAM** controls authorization, not just login.
- **Storage exposure** can occur through public access, shared links, or weak policies.
- **Over-permissioning** often enables lateral movement and privilege escalation.
- Effective access depends on all layers, including public access blocks and trust policies.
- Listing and reading are different permissions.
- Service accounts and workload identities are high-value targets.

**Quick comparison of common storage exposure signals**

| Platform | Common exposure signal | What to verify |
|---|---|---|
| AWS | Public bucket policy or ACL | Block Public Access, bucket policy, object ACLs |
| Azure | SAS token or public container | Expiry, scope, permissions, RBAC |
| GCP | Public IAM binding or legacy ACL | Public access prevention, bucket IAM, uniform access |

**Practical remediation example**

If you find a public storage bucket, fix it in layers rather than only removing one policy statement.

1. Remove public principals from the bucket or container policy.
2. Disable or tighten ACL-based access.
3. Enable public access prevention or Block Public Access.
4. Review attached IAM roles and service accounts.
5. Rotate any secrets that may have been exposed.
6. Check logs for unauthorized reads or downloads.

Example AWS remediation commands:

```bash
aws s3api put-public-access-block \
  --bucket example-bucket \
  --public-access-block-configuration \
  BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

This does not replace policy review, but it helps prevent accidental public exposure.

**Key takeaways**

Cloud security failures often come from small misconfigurations with large blast radius. If you understand how IAM, storage permissions, and trust relationships interact, you can spot exposure quickly and explain why a resource is reachable.

- Check the effective permission path, not just the visible policy.
- Separate list, read, and write permissions in your analysis.
- Treat service accounts and SAS tokens as sensitive credentials.
- Assume public access controls may override permissive policies.
- Look for over-permissioning as the root cause of many cloud incidents.
