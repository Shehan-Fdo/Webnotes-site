## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview of the CIA triad**

The <abbr title="Confidentiality, Integrity, Availability">CIA</abbr> triad is the core model for information security. It helps you reason about what must be protected, what can be trusted, and what must remain accessible. In practice, security controls usually support more than one goal at once, but each control should be chosen with a primary objective in mind.

- **Confidentiality** prevents unauthorized disclosure of information.
- **Integrity** prevents unauthorized or undetected modification of information.
- **Availability** ensures authorized users can access systems and data when needed.

These three goals often compete. Stronger confidentiality controls can reduce availability, and high availability designs can increase attack surface. Good security work balances all three based on business risk.

**Confidentiality**

Confidentiality means only authorized people, processes, and systems can view data. It applies to data at rest, in transit, and in use. In ethical hacking and defense, confidentiality failures often show up as data leaks, credential theft, weak access control, misconfigured cloud storage, or intercepted traffic.

Common confidentiality controls include:

- **Authentication** to verify identity before access is granted.
- **Authorization** to limit what an authenticated identity can do.
- **Encryption** to protect data from disclosure if intercepted or stolen.
- **Access control lists** and role-based access control to enforce least privilege.
- **Data classification** to apply stronger controls to sensitive information.
- **Masking and tokenization** to reduce exposure in non-production systems.

Practical examples:

- Use `chmod 600 /etc/shadow` to restrict local file access on Linux.
- Use TLS to protect web traffic:
  ```bash
  curl --tlsv1.3 https://example.com
  ```
- Encrypt disks with tools such as LUKS on Linux or BitLocker on Windows.
- Restrict database access with separate accounts for read-only and administrative tasks.

A useful rule is that confidentiality is not just about secrecy. It is also about minimizing unnecessary exposure. For example, a backup file stored in a public bucket is a confidentiality failure even if the production database is well protected.

**Integrity**

Integrity means data remains accurate, complete, and unaltered except by authorized actions. It also means changes can be detected and traced. Integrity matters for logs, financial records, software packages, configuration files, and security events. If integrity is broken, you may still have access to data, but you cannot trust it.

Integrity controls include:

- **Hashes** to detect changes in files or messages.
- **Digital signatures** to verify origin and detect tampering.
- **Checksums** for accidental corruption detection.
- **Version control** to track changes and support rollback.
- **File integrity monitoring** to alert on unauthorized modifications.
- **Input validation** to prevent malicious data from altering application behavior.

Examples of integrity verification:

```bash
sha256sum file.iso
sha256sum -c file.iso.sha256
```

```bash
gpg --verify package.tar.gz.sig package.tar.gz
```

Integrity is especially important for software distribution. If an attacker can alter an installer, update package, or script, they can turn a trusted delivery path into a compromise path. This is why package signing, secure update channels, and trusted repositories matter.

Integrity also applies to logs. If an attacker can delete or edit logs, incident response becomes much harder. To reduce this risk, send logs to a remote system, use append-only storage where possible, and protect log access with strict permissions.

**Availability**

Availability means systems, services, and data are accessible when authorized users need them. It is not enough for data to be confidential and correct if users cannot reach it during normal operations or an incident. Availability failures can result from hardware faults, software bugs, misconfiguration, overload, natural disasters, or denial-of-service attacks.

Availability controls include:

- **Redundancy** for servers, storage, power, and network links.
- **Failover** to alternate systems when a primary system fails.
- **Load balancing** to distribute traffic across multiple nodes.
- **Backups** and tested restore procedures.
- **Monitoring and alerting** to detect outages early.
- **Rate limiting** and DDoS mitigation to absorb malicious traffic.
- **Patch management** to reduce crash and exploit risk.

Practical examples:

- Use `systemctl status nginx` to confirm a service is running.
- Use `journalctl -u nginx --since "1 hour ago"` to inspect recent failures.
- Use `ping`, `traceroute`, `ss`, and `curl` to isolate connectivity and service issues.
- Keep offline or immutable backups so ransomware cannot easily destroy recovery options.

Availability is often measured with uptime targets and service-level objectives. A system with strong confidentiality and integrity but no recovery plan is still fragile. Backups are only useful if you can restore them quickly and reliably.

**How the three goals interact**

The CIA triad is a balancing act, not three separate silos. A control that improves one area may weaken another.

- **Encryption** improves confidentiality, but key management mistakes can reduce availability.
- **Strict access controls** improve confidentiality and integrity, but overly restrictive policies can block legitimate work.
- **Redundancy** improves availability, but duplicated data increases the number of places that must be protected.
- **Logging** improves integrity and incident response, but logs can expose sensitive data if not protected.

A practical security design starts with business impact:

- Identify the most sensitive data and systems.
- Determine which failures would hurt the organization most.
- Apply controls that reduce risk without creating unacceptable operational friction.

**Threats that target each goal**

Different attack types map naturally to different CIA impacts.

- **Confidentiality threats**
  - Phishing and credential theft
  - Sniffing and man-in-the-middle attacks
  - Misconfigured cloud storage
  - Insider data theft
  - Malware that exfiltrates files

- **Integrity threats**
  - Tampering with files or transactions
  - SQL injection that alters records
  - Supply chain compromise
  - Log manipulation
  - Unauthorized configuration changes

- **Availability threats**
  - Denial-of-service attacks
  - Ransomware
  - Hardware failure
  - Power loss
  - Resource exhaustion from bugs or abuse

Understanding the likely threat helps you choose the right control. For example, packet capture on an open Wi-Fi network is primarily a confidentiality issue, while a corrupted software update is primarily an integrity issue.

**Controls by layer**

Security works best when controls are layered across the stack.

- **Physical layer**
  - Locks, badges, cameras, and environmental controls
  - UPS systems and generators for power continuity
  - Secure server rooms and media disposal

- **Network layer**
  - Firewalls, segmentation, VPNs, and IDS/IPS
  - TLS for encrypted transport
  - DDoS protection and traffic shaping

- **Host layer**
  - Patch management, endpoint protection, and hardening
  - Least privilege and secure configuration baselines
  - File integrity monitoring and audit logging

- **Application layer**
  - Strong authentication and session management
  - Input validation and output encoding
  - Secure coding and dependency management

- **Data layer**
  - Encryption at rest and in transit
  - Backups, replication, and retention policies
  - Classification, masking, and access reviews

**Practical command examples**

Use commands to verify the state of controls and troubleshoot failures.

```bash
# Check listening services
ss -tulpn

# Inspect firewall rules on Linux
sudo iptables -L -n -v
sudo nft list ruleset

# Check disk usage that may affect availability
df -h
du -sh /var/log/*

# Verify file permissions
ls -l /etc/passwd /etc/shadow

# Review authentication and service logs
journalctl -xe
tail -f /var/log/auth.log
```

On Windows, common checks include:

```powershell
Get-Service
Get-EventLog -LogName System -Newest 20
Get-FileHash .\installer.exe -Algorithm SHA256
```

These commands help you confirm whether a service is running, whether a file has changed, and whether a system is showing signs of failure or tampering.

**Security design patterns**

Some patterns appear repeatedly in CIA-focused designs.

- **Least privilege** limits damage if an account is compromised.
- **Defense in depth** ensures one failed control does not expose everything.
- **Separation of duties** reduces insider abuse and fraud.
- **Need to know** limits access to sensitive information.
- **Secure defaults** reduce accidental exposure.
- **Backup and recovery** preserve availability after incidents.

A simple example is a web application architecture:

- Public users access a load balancer.
- The load balancer forwards traffic to application servers.
- Application servers access a database on a private network.
- Sensitive secrets are stored in a vault, not in source code.
- Logs are shipped to a separate logging system.

This design improves confidentiality by isolating the database, integrity by reducing direct modification paths, and availability by allowing failover and scaling.

**Common mistakes**

- Treating encryption as a complete security solution.
- Ignoring backups until after an outage or ransomware event.
- Storing secrets in plain text files or source repositories.
- Allowing excessive permissions because they are convenient.
- Failing to verify hashes or signatures before installing software.
- Keeping logs only on the local host where an attacker can delete them.
- Designing for uptime without testing restore procedures.

> A control that is not tested is only a hope, not a guarantee.

**Exam-focused distinctions**

For CEH-style questions, distinguish the goals by the type of harm:

- **Confidentiality** answers: “Who can see it?”
- **Integrity** answers: “Can I trust it?”
- **Availability** answers: “Can I use it now?”

If a scenario involves stolen records, think confidentiality. If it involves altered logs, tampered transactions, or corrupted files, think integrity. If it involves outages, overload, or blocked access, think availability.

**Key takeaways**

- The CIA triad is the foundation of information security decision-making.
- Controls often support more than one goal, but each has trade-offs.
- Encryption, access control, and masking support confidentiality.
- Hashing, signatures, monitoring, and validation support integrity.
- Redundancy, backups, failover, and monitoring support availability.
- Strong security requires balancing all three based on risk and business needs.
