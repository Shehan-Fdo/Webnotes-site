**Hashing fundamentals**

Hashing converts input data of arbitrary length into a fixed-length digest. In security work, you use hashes to verify integrity, store password verifiers, and detect tampering. A good cryptographic hash is designed to be one-way, meaning you cannot feasibly recover the original input from the digest alone.

Common properties you should know:

- **Deterministic**: the same input always produces the same output.
- **Fixed-length output**: digest size does not depend on input size.
- **Preimage resistance**: given a hash, it is computationally hard to find an input that matches it.
- **Second-preimage resistance**: given one input, it is hard to find a different input with the same hash.
- **Collision resistance**: it is hard to find any two different inputs with the same hash.

Hashing is not encryption. Encryption is reversible with a key; hashing is intended to be irreversible. That distinction matters when you choose a control for passwords, file integrity, or digital signatures.

**Common hash algorithms and practical use**

Older algorithms such as MD5 and SHA-1 are considered broken for collision resistance and should not be used for security-sensitive integrity or password storage. SHA-2 and SHA-3 families remain widely used for integrity and signatures, while password storage should use a dedicated password hashing or key derivation function rather than a general-purpose hash.

Typical examples:

- **MD5**: fast, legacy, collision-prone; useful only for non-security checksums in limited contexts.
- **SHA-1**: deprecated for security use due to practical collision attacks.
- **SHA-256 / SHA-512**: common in integrity checks, certificates, and HMAC constructions.
- **SHA-3**: modern alternative with a different internal design.
- **BLAKE2 / BLAKE3**: fast modern hashes; adoption varies by platform and use case.

Example integrity check on Linux:

```bash
sha256sum ubuntu.iso
sha256sum -c SHA256SUMS
```

Example in Python:

```python
import hashlib

data = b"example"
digest = hashlib.sha256(data).hexdigest()
print(digest)
```

**Why salting matters**

A **salt** is random data added to a password before hashing. Its main purpose is to ensure that identical passwords do not produce identical stored hashes. This defeats precomputed attacks such as rainbow tables and makes large-scale cracking more expensive.

Without salts:

- Two users with the same password have the same hash.
- Attackers can reuse precomputed tables across many targets.
- Common passwords become easy to identify at scale.

With unique salts:

- Each password hash is unique even if the password is reused.
- Rainbow tables become ineffective because the attacker must compute per-salt.
- Offline cracking still remains possible, but it becomes more expensive.

A salt does not need to be secret. It must be unique and unpredictable enough to prevent precomputation. Store the salt alongside the hash in the database.

Example conceptual storage:

```text
username: alice
salt: 9f3c2a1b7d4e8c11
hash: 5d2f...a91c
```

**Salt versus pepper**

A **pepper** is a secret value added to passwords before hashing, but unlike a salt it is not stored with the password record. You typically keep the pepper in application configuration, a secrets manager, or an HSM (Hardware Security Module).

Use cases:

- **Salt**: unique per password, stored with the hash.
- **Pepper**: shared secret, stored separately, adds a second barrier if the database is stolen.

A pepper helps if an attacker dumps the password database but not the application secrets. It does not replace salting.

> A pepper is only useful if you can protect it separately from the password database. If the attacker gets both, the benefit is lost.

**Password hashing versus general hashing**

For passwords, you should use slow, adaptive algorithms designed to resist brute force. General-purpose hashes are too fast and make offline cracking easier.

Preferred password hashing and derivation functions:

- **bcrypt**: adaptive cost factor, widely supported, includes salt.
- **scrypt**: memory-hard, designed to increase GPU/ASIC cracking cost.
- **Argon2**: modern winner of the Password Hashing Competition; strong choice for new systems.
- **PBKDF2**: widely standardized and still acceptable when configured with high iteration counts.

Why speed matters:

- Fast hashes help attackers test billions of guesses per second.
- Slow, memory-hard functions reduce guess rate and increase hardware cost.
- The goal is not to make hashing impossible, but to make offline attacks economically unattractive.

Example `bcrypt` in Python:

```python
import bcrypt

password = b"CorrectHorseBatteryStaple!"
salt = bcrypt.gensalt(rounds=12)
hashed = bcrypt.hashpw(password, salt)

print(hashed)
```

Example `PBKDF2` in Python:

```python
import hashlib
import os

password = b"CorrectHorseBatteryStaple!"
salt = os.urandom(16)

dk = hashlib.pbkdf2_hmac(
    "sha256",
    password,
    salt,
    310000
)

print(salt.hex())
print(dk.hex())
```

**Key derivation concepts**

A **key derivation function** (KDF) transforms a password or shared secret into a cryptographic key suitable for encryption, MACs, or other algorithms. KDFs are used when the original secret is not already a strong binary key.

Two broad categories matter:

- **Password-based KDFs**: derive keys from human-memorable passwords.
- **General KDFs**: derive multiple keys from a master secret or shared secret.

Common password-based KDFs:

- **PBKDF2**: uses repeated hashing and a salt; configurable iteration count.
- **bcrypt**: also used for password storage; not ideal for arbitrary key expansion.
- **scrypt**: adds memory hardness.
- **Argon2**: modern password hashing and key derivation option.

Common general KDFs:

- **HKDF** (HMAC-based Key Derivation Function): expands a strong input keying material into one or more keys.
- **SP 800-108 KDFs**: used in some enterprise and standards-based systems.

Use a KDF when:

- You need an encryption key from a password.
- You need separate keys for encryption and authentication.
- You need to derive session keys from a master secret.

**PBKDF2 example for file encryption**

```python
from cryptography.hazmat.primitives.kdf.pbkdf2 import PBKDF2HMAC
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.backends import default_backend
import os

password = b"my passphrase"
salt = os.urandom(16)

kdf = PBKDF2HMAC(
    algorithm=hashes.SHA256(),
    length=32,
    salt=salt,
    iterations=310000,
    backend=default_backend()
)

key = kdf.derive(password)
print(key.hex())
```

**HKDF and key separation**

HKDF is useful when you already have a strong secret, such as a Diffie-Hellman shared secret or a master key, and you want to derive multiple independent keys. It uses a two-step process:

1. **Extract**: normalize input entropy into a pseudorandom key.
2. **Expand**: generate one or more output keys for specific purposes.

This is important for key separation. You should not reuse one key for encryption, integrity, and session control if a protocol can avoid it.

Example conceptual use:

```python
from cryptography.hazmat.primitives.kdf.hkdf import HKDF
from cryptography.hazmat.primitives import hashes

hkdf = HKDF(
    algorithm=hashes.SHA256(),
    length=32,
    salt=None,
    info=b"file-encryption",
)

key = hkdf.derive(shared_secret)
```

**Attack perspective**

As a CEH candidate, you should understand how attackers abuse weak hashing choices.

Common attack paths:

- **Offline password cracking**: attacker steals a hash database and tests guesses locally.
- **Rainbow table attacks**: precomputed hashes for common passwords, effective mainly against unsalted fast hashes.
- **Collision attacks**: attacker crafts two different inputs with the same hash, relevant for signatures and integrity systems.
- **Dictionary attacks**: attacker tries common passwords and variations.
- **GPU acceleration**: attackers use parallel hardware to test many guesses quickly.

Defensive implications:

- Use unique salts for every password.
- Use slow, memory-hard password hashing.
- Enforce strong password policy and MFA (Multi-Factor Authentication).
- Protect password databases and application secrets separately.
- Monitor for credential stuffing and reuse.

**Practical command-line examples**

Check a file hash on Linux:

```bash
sha256sum important.zip
```

Generate a random salt:

```bash
openssl rand -hex 16
```

Derive a key with OpenSSL PBKDF2:

```bash
openssl enc -aes-256-cbc -pbkdf2 -iter 200000 -salt -in secret.txt -out secret.txt.enc
```

Inspect a certificate fingerprint:

```bash
openssl x509 -in cert.pem -noout -fingerprint -sha256
```

These commands are useful in labs and incident response because they show how hashes support integrity verification, certificate validation, and password-based encryption workflows.

**Implementation guidance**

When you design or review a system, choose the primitive based on the goal:

- **Integrity check**: use SHA-256 or SHA-3.
- **Password storage**: use Argon2, bcrypt, scrypt, or PBKDF2 with strong parameters.
- **Key derivation from password**: use PBKDF2, scrypt, or Argon2.
- **Key expansion from strong secret**: use HKDF.
- **Message authentication**: use HMAC, not raw hash concatenation.

A secure password storage flow typically looks like this:

1. Generate a unique random salt for each password.
2. Normalize and hash the password with a slow password hashing function.
3. Store the algorithm, parameters, salt, and hash together.
4. Verify by recomputing with the same parameters during login.
5. Rehash when parameters become outdated.

Example storage format:

```text
$argon2id$v=19$m=65536,t=3,p=4$base64salt$base64hash
```

That format is useful because it preserves the algorithm and cost settings needed for future verification and migration.

**Common mistakes**

- Using MD5 or SHA-1 for password storage.
- Reusing the same salt for every user.
- Confusing salt with pepper.
- Using a fast hash where a password hash is required.
- Truncating hashes too aggressively and reducing security margin.
- Reusing one derived key for multiple cryptographic purposes.
- Assuming hashing alone protects against online attacks.

**Quick comparison**

| Primitive | Main purpose | Salted? | Reversible? | Typical use |
|---|---|---:|---:|---|
| MD5/SHA-1/SHA-256 | Integrity hashing | Optional | No | File checks, fingerprints |
| bcrypt/scrypt/Argon2 | Password hashing | Yes | No | Password storage |
| PBKDF2 | Password-based key derivation | Yes | No | Password-derived keys |
| HKDF | Key expansion | Optional | No | Deriving multiple keys |
| HMAC | Message authentication | Uses key | No | API signing, integrity with secret |

**Exam focus points**

For CEH v13, be ready to distinguish the following:

- **Hashing** versus encryption.
- **Salt** versus pepper.
- **Password hashing** versus general-purpose hashing.
- **Collision resistance** versus preimage resistance.
- **PBKDF2** versus HKDF.
- **Fast hashes** versus memory-hard password functions.

A strong answer usually explains not just what the primitive is, but why it is chosen in a specific threat model. For example, a salted Argon2 hash protects against database theft and large-scale offline cracking better than SHA-256 alone, because the latter is too fast and not designed for password storage.

**Minimal secure examples**

Password verification flow in pseudocode:

```python
stored = get_user_record(username)
candidate = password_input.encode()

if verify_password(candidate, stored.hash, stored.salt, stored.params):
    allow_login()
else:
    deny_login()
```

File integrity verification flow:

```bash
sha256sum -c checksums.txt
```

Key derivation for encryption:

```python
key = PBKDF2(password, salt, iterations=310000, length=32)
```

These patterns are the core of secure use: hash for integrity, salt for uniqueness, and KDFs for turning weak secrets into usable cryptographic material.
