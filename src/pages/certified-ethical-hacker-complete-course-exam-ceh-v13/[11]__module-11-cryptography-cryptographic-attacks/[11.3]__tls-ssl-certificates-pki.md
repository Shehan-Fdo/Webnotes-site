## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**TLS, SSL, and the security problem they solve**

Transport Layer Security (TLS) is the modern protocol used to protect data in transit. Secure Sockets Layer (SSL) is the older predecessor; in practice, people still say “SSL” when they mean TLS. The goal is to provide:

- **Confidentiality** so eavesdroppers cannot read traffic
- **Integrity** so attackers cannot alter traffic undetected
- **Authentication** so clients can verify the server, and sometimes the server can verify the client

TLS sits between the application and transport layers. It protects protocols such as HTTPS, SMTPS, IMAPS, and LDAPS without changing the application’s core logic.

**How TLS works at a high level**

TLS uses a handshake to negotiate parameters and establish shared session keys, then uses the record layer to protect application data. The handshake is not a single step; messages are exchanged before the peer is fully authenticated and before encrypted application traffic begins.

Typical TLS 1.2 handshake flow:

1. Client sends `ClientHello` with supported versions, cipher suites, and random data.
2. Server replies with `ServerHello`, selects parameters, and sends its `Certificate`.
3. Depending on the key exchange, the server may also send `ServerKeyExchange`.
4. Server signals the end of its hello messages.
5. Client validates the certificate chain and may send `ClientKeyExchange`.
6. Both sides derive session keys.
7. `ChangeCipherSpec` and `Finished` messages confirm that encryption is active and the handshake transcript is intact.

TLS 1.3 simplifies and shortens the handshake:

1. Client sends `ClientHello` with supported versions and key shares.
2. Server responds with `ServerHello` and immediately starts encrypted handshake messages.
3. Certificate and authentication data follow under handshake protection.
4. Both sides exchange `Finished` messages.
5. Application data begins sooner than in TLS 1.2.

> The certificate is presented during the handshake, but trust is established only after the client validates the chain, hostname, validity period, and other policy checks.

**TLS record layer vs handshake layer**

Understanding the two main TLS sub-protocols helps with troubleshooting and exam questions.

- **Handshake layer**
  - Negotiates protocol version and cipher suite
  - Authenticates peers
  - Establishes shared secrets
  - Exchanges certificates and key exchange parameters
- **Record layer**
  - Fragments data into records
  - Adds integrity protection
  - Encrypts application data after keys are established

The record layer is what protects the actual payload after the handshake completes. The handshake layer is what makes that protection possible.

**SSL/TLS versions and why older protocols are avoided**

Older SSL versions are insecure and deprecated. TLS has evolved to remove weak algorithms and improve forward secrecy.

| Version | Status | Notes |
|---|---:|---|
| SSL 2.0 | Broken | Do not use |
| SSL 3.0 | Broken | Vulnerable to POODLE-style attacks |
| TLS 1.0 | Deprecated | Weak by modern standards |
| TLS 1.1 | Deprecated | Weak by modern standards |
| TLS 1.2 | Widely deployed | Still common and secure when configured well |
| TLS 1.3 | Preferred | Faster handshake, stronger defaults, simpler design |

TLS 1.3 removes many legacy options, including static RSA key exchange and several older cipher modes. That reduction in complexity is a security improvement.

**Cipher suites, key exchange, and forward secrecy**

A TLS cipher suite defines the algorithms used for key exchange, authentication, encryption, and integrity. In modern TLS, the important distinction is not just the cipher suite name but the key exchange method.

- **RSA key exchange**
  - The client encrypts a premaster secret with the server’s RSA public key
  - The server uses its private key to recover it
  - This is deprecated because it does not provide forward secrecy
- **(EC)DHE key exchange**
  - Ephemeral Diffie-Hellman or Elliptic Curve Diffie-Hellman Ephemeral
  - Both sides contribute ephemeral keys
  - Provides **forward secrecy**, meaning past sessions remain protected even if the server private key is later compromised

Forward secrecy matters because captured traffic cannot be decrypted later just by stealing the server certificate private key.

Common modern preference is ECDHE-based suites with authenticated encryption such as AES-GCM or ChaCha20-Poly1305.

Example OpenSSL inspection:

```bash
openssl s_client -connect example.com:443 -tls1_2
openssl s_client -connect example.com:443 -tls1_3
```

Useful output fields include:

- Negotiated protocol version
- Selected cipher suite
- Certificate chain
- Verification result
- Session ticket or resumption details

**Certificates and what they prove**

A digital certificate binds an identity to a public key. In web PKI, the certificate is usually an X.509 certificate.

A certificate typically contains:

- Subject name, such as a domain or organization
- Subject Alternative Name (SAN), which is the modern place for DNS names and IPs
- Issuer name
- Validity period
- Public key
- Signature algorithm
- Certificate extensions
- Serial number
- Key usage and extended key usage

The public key in the certificate is used to verify signatures in the certificate chain and, in TLS, to verify the server’s handshake signature or authenticate the server’s key exchange. It does not by itself “create trust”; trust comes from the chain to a trusted root and policy validation.

**X.509 certificate fields and extensions**

Important fields and extensions include:

- **Subject**
  - Identifies the entity the certificate is issued to
  - For web certificates, the CN (Common Name) is legacy; SAN is what matters for hostname matching
- **Issuer**
  - The certificate authority that signed the certificate
- **Validity**
  - `Not Before` and `Not After` dates
  - Expired certificates fail validation
- **SAN**
  - Supports multiple DNS names, IP addresses, and sometimes other identifiers
  - Required for modern browser hostname validation
- **Key Usage**
  - Restricts allowed cryptographic use
  - Examples: `Digital Signature`, `Key Encipherment`, `Key Cert Sign`
- **Extended Key Usage (EKU)**
  - Further narrows intended use
  - Examples: `Server Authentication`, `Client Authentication`
- **Basic Constraints**
  - Indicates whether a certificate is a CA certificate
- **Authority Key Identifier / Subject Key Identifier**
  - Helps chain building and issuer matching

Example inspection:

```bash
openssl x509 -in server.crt -text -noout
```

**Certificate chain and trust model**

Most systems do not trust a server certificate directly. They trust a chain:

- **Leaf certificate**
  - The server certificate presented to clients
- **Intermediate certificate**
  - Signed by a higher CA and used to issue leaf certificates
- **Root certificate**
  - Self-signed and stored in the client trust store

The client builds a chain from the leaf to a trusted root. If the chain cannot be built, validation fails.

Common chain-building issues:

- Missing intermediate certificate on the server
- Wrong certificate order in the served chain
- Cross-signed intermediates causing different path choices on different clients
- Trust store differences across browsers, operating systems, and embedded devices
- Expired or revoked intermediate certificates

Practical verification examples:

```bash
openssl verify -CAfile root-ca.pem -untrusted intermediate-ca.pem server.crt
openssl s_client -connect example.com:443 -showcerts
```

> `openssl verify` is useful, but it only works correctly when the CA files and chain order match the actual trust path you want to test.

**Certificate authorities, registration authorities, and PKI**

Public Key Infrastructure (PKI) is the system of policies, roles, hardware, software, and procedures used to create and manage digital certificates.

Core PKI roles:

- **Certificate Authority (CA)**
  - Issues and signs certificates
  - Can be a root CA or intermediate CA
- **Registration Authority (RA)**
  - Verifies identity before issuance
  - Often handles enrollment workflows while the CA performs signing
- **Repository**
  - Publishes certificates, CRLs, and policy documents
- **Trust store**
  - Contains trusted root certificates on a client or device

PKI is not only about cryptography. It also includes governance:

- Certificate policies
- Certification practice statements
- Identity proofing
- Approval workflows
- Audit logging
- Key protection and HSM use
- Lifecycle management

**Certificate lifecycle management**

Certificates have a lifecycle that must be managed carefully.

- **Enrollment**
  - Generate a key pair and submit a CSR (Certificate Signing Request)
- **Issuance**
  - CA signs the certificate after validation
- **Deployment**
  - Install the certificate and private key on the service
- **Renewal**
  - Replace the certificate before expiration
- **Suspension**
  - Temporarily disable use in some PKI systems
- **Revocation**
  - Mark the certificate as no longer trusted before expiration
- **Expiration**
  - Certificate naturally becomes invalid after its validity period

Operational best practice is to automate renewal and monitor expiry dates.

Example expiry check:

```bash
openssl x509 -in server.crt -noout -enddate
```

**Certificate signing requests and key generation**

A CSR contains the public key and identity information that a CA uses to issue a certificate. The private key stays with the requester.

Generate a private key and CSR:

```bash
openssl genpkey -algorithm RSA -out server.key -pkeyopt rsa_keygen_bits:2048
openssl req -new -key server.key -out server.csr -subj "/C=US/ST=CA/L=SanJose/O=ExampleCorp/CN=www.example.com"
```

For modern deployments, ECDSA keys are also common:

```bash
openssl ecparam -name prime256v1 -genkey -noout -out server-ecdsa.key
openssl req -new -key server-ecdsa.key -out server-ecdsa.csr -subj "/CN=www.example.com"
```

**Wildcard certificates and SAN certificates**

A wildcard certificate covers one label beneath a domain, such as `*.example.com`.

- `*.example.com` matches `app.example.com`
- It does not match `a.b.example.com`
- It does not match the apex `example.com`

SAN certificates can cover multiple names in one certificate and are the preferred choice for many environments.

Use cases:

- Multiple subdomains on one service
- Load-balanced environments
- Microservices with shared TLS termination
- Internal services with several DNS names

Example SAN CSR configuration:

```ini
[ req ]
distinguished_name = dn
req_extensions = req_ext
prompt = no

[ dn ]
CN = www.example.com

[ req_ext ]
subjectAltName = @alt_names

[ alt_names ]
DNS.1 = www.example.com
DNS.2 = api.example.com
DNS.3 = admin.example.com
```

Generate the CSR with extensions:

```bash
openssl req -new -key server.key -out server.csr -config san.cnf
```

**Hostname validation and common browser errors**

Clients validate that the certificate matches the requested hostname. Modern clients use SAN first.

Common errors include:

- `NET::ERR_CERT_COMMON_NAME_INVALID`
  - Hostname does not match the certificate
- `NET::ERR_CERT_DATE_INVALID`
  - Certificate is expired or system clock is wrong
- `NET::ERR_CERT_AUTHORITY_INVALID`
  - Chain does not lead to a trusted root
- `SEC_ERROR_UNKNOWN_ISSUER`
  - Browser does not trust the issuer
- `certificate has expired`
  - The `Not After` date has passed

Troubleshooting checklist:

1. Confirm the hostname in the URL matches the SAN entries.
2. Check certificate validity dates.
3. Verify the full chain is served.
4. Compare the server certificate fingerprint with the expected value.
5. Check the client system clock and trust store.

**Manual certificate fingerprint verification**

Fingerprints help you compare a certificate against a known-good value.

```bash
openssl x509 -in server.crt -noout -fingerprint -sha256
openssl x509 -in server.crt -noout -fingerprint -sha1
```

Use SHA-256 fingerprints for modern verification. SHA-1 is legacy and should not be used for trust decisions unless a specific tool or legacy workflow requires it.

**Certificate pinning**

Certificate pinning means a client accepts only a specific certificate, public key, or CA chain for a service.

Types of pinning:

- **Leaf pinning**
  - Pin the exact server certificate
  - Strong but brittle because renewal breaks clients unless updated
- **Public key pinning**
  - Pin the public key or SPKI (Subject Public Key Info)
  - More flexible across certificate renewals
- **CA pinning**
  - Trust only a specific issuing CA for a service

Security implications:

- Reduces risk from rogue or misissued certificates
- Can break connectivity if the pinned material changes unexpectedly
- Requires careful rotation planning
- Is often used in mobile apps, internal tooling, and high-security environments

Pinning is not a replacement for PKI; it is an additional trust constraint.

**Revocation: CRL and OCSP**

Revocation is the process of invalidating a certificate before expiration.

Common mechanisms:

- **CRL** (Certificate Revocation List)
  - A signed list of revoked certificate serial numbers
  - Clients download and check the list
- **OCSP** (Online Certificate Status Protocol)
  - Clients query a responder for the certificate status
- **OCSP stapling**
  - The server includes a recent OCSP response during the handshake

Important operational note:

- Many clients and browsers do not enforce revocation uniformly
- Some environments soft-fail when revocation status cannot be checked
- High-security systems may require stricter revocation policy

**mTLS: mutual TLS and client certificates**

Mutual TLS (mTLS) authenticates both server and client using certificates. It is common in service-to-service communication, internal APIs, and zero-trust architectures.

mTLS flow:

1. Client connects to the server.
2. Server presents its certificate.
3. Server requests a client certificate.
4. Client sends its certificate and proves possession of the private key.
5. Both sides validate trust chains and policy.
6. The connection is allowed only if client authentication succeeds.

Example Nginx configuration for mTLS:

```nginx
server {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate     /etc/nginx/tls/server.crt;
    ssl_certificate_key /etc/nginx/tls/server.key;
    ssl_client_certificate /etc/nginx/tls/ca.crt;
    ssl_verify_client on;

    location / {
        proxy_pass http://backend;
    }
}
```

Example Apache configuration:

```apache
<VirtualHost *:443>
    ServerName api.example.com

    SSLEngine on
    SSLCertificateFile /etc/httpd/tls/server.crt
    SSLCertificateKeyFile /etc/httpd/tls/server.key
    SSLCACertificateFile /etc/httpd/tls/ca.crt

    SSLVerifyClient require
    SSLVerifyDepth 2
</VirtualHost>
```

**Server configuration examples**

Nginx example with modern TLS settings:

```nginx
server {
    listen 443 ssl http2;
    server_name www.example.com;

    ssl_certificate     /etc/ssl/certs/fullchain.pem;
    ssl_certificate_key /etc/ssl/private/server.key;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;

    # TLS 1.2 cipher preference; TLS 1.3 ciphers are controlled separately by OpenSSL/Nginx build
    ssl_ciphers HIGH:!aNULL:!MD5;

    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    add_header Strict-Transport-Security "max-age=31536000" always;
}
```

Apache example with a full chain file:

```apache
<VirtualHost *:443>
    ServerName www.example.com

    SSLEngine on
    SSLCertificateFile /etc/httpd/tls/fullchain.pem
    SSLCertificateKeyFile /etc/httpd/tls/server.key

    SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1
    SSLCipherSuite HIGH:!aNULL:!MD5
</VirtualHost>
```

Modern Apache deployments usually place the leaf and intermediate chain together in `SSLCertificateFile`, rather than relying on deprecated chain directives.

**Practical OpenSSL troubleshooting**

OpenSSL is one of the most useful tools for diagnosing TLS problems.

Check a remote service:

```bash
openssl s_client -connect example.com:443 -servername example.com
```

Check the full chain:

```bash
openssl s_client -connect example.com:443 -servername example.com -showcerts
```

Inspect a certificate:

```bash
openssl x509 -in server.crt -text -noout
```

Verify a certificate against a CA:

```bash
openssl verify -CAfile root-ca.pem -untrusted intermediate-ca.pem server.crt
```

Common troubleshooting patterns:

- If the chain is incomplete, the server may not be sending the intermediate certificate.
- If the hostname is wrong, the SAN list may not include the requested name.
- If the certificate is valid but the browser still complains, the client trust store may be outdated.
- If the service works in one browser but not another, trust store differences or revocation behavior may be the cause.

**Cross-signed intermediates and trust store differences**

Chain building is not always straightforward. A certificate may have multiple possible paths to a trusted root.

Why this matters:

- Some clients trust one root but not another
- Some intermediates are cross-signed by different roots
- Older devices may not support newer root certificates
- Embedded systems often have stale trust stores

Practical impact:

- A chain that works in one browser may fail in another
- Mobile apps may need bundled trust anchors
- Servers should present the correct intermediate chain for broad compatibility

**Certificate policies and governance**

PKI includes policy, not just technology.

Important governance concepts:

- **Certificate Policy (CP)**
  - Defines what a certificate may be used for and how it is issued
- **Certification Practice Statement (CPS)**
  - Describes how a CA implements the policy
- **Identity proofing**
  - Confirms the requester is entitled to the certificate
- **Key management**
  - Protects private keys with file permissions, HSMs, or secure enclaves
- **Auditability**
  - Tracks issuance, renewal, revocation, and access events

These controls matter because a technically valid certificate can still be operationally unsafe if issuance and key handling are weak.

**Private keys, storage, and protection**

The private key must remain secret. If it is exposed, an attacker can impersonate the service until the certificate is revoked or replaced.

Protect private keys by:

- Restricting file permissions
- Using hardware security modules (HSMs) where appropriate
- Avoiding key reuse across unrelated services
- Encrypting private keys at rest when operationally feasible
- Rotating keys during renewal when policy requires it

Example permission hardening:

```bash
chown root:root /etc/ssl/private/server.key
chmod 600 /etc/ssl/private/server.key
```

**Real-world deployment patterns**

Common TLS deployment topologies include:

- **Direct termination**
  - The application server terminates TLS itself
- **Reverse proxy termination**
  - Nginx, Apache, or a load balancer terminates TLS and forwards traffic internally
- **TLS passthrough**
  - The proxy forwards encrypted traffic to a backend that terminates TLS
- **End-to-end encryption**
  - TLS is used at multiple hops, such as client-to-proxy and proxy-to-backend

Choose the model based on operational control, performance, inspection requirements, and trust boundaries.

**Exam-focused distinctions to remember**

- **TLS** is the protocol you should think about today; **SSL** is legacy terminology.
- The **certificate** binds identity to a public key; it does not encrypt bulk traffic by itself.
- The **private key** proves possession and supports authentication; it is not used to decrypt modern session traffic in typical ECDHE-based TLS.
- **Forward secrecy** comes from ephemeral key exchange, not from the certificate alone.
- **SAN** is the modern hostname-matching field.
- **Revocation** exists, but enforcement varies by client and environment.
- **mTLS** authenticates both sides and is common in internal service security.
- **Pinning** increases assurance but can create operational fragility.

**Quick reference workflow for validating a server certificate**

1. Connect to the service with `openssl s_client`.
2. Confirm the negotiated protocol and cipher.
3. Inspect the certificate subject and SAN entries.
4. Verify the chain to a trusted root.
5. Check expiry, hostname, and key usage.
6. Compare the fingerprint if you have a known-good value.
7. Review client trust store and revocation behavior if validation still fails.

**Common command set**

```bash
# Inspect remote TLS details
openssl s_client -connect example.com:443 -servername example.com -showcerts

# Inspect local certificate
openssl x509 -in server.crt -text -noout

# Verify chain
openssl verify -CAfile root-ca.pem -untrusted intermediate-ca.pem server.crt

# Show SHA-256 fingerprint
openssl x509 -in server.crt -noout -fingerprint -sha256

# Check expiry
openssl x509 -in server.crt -noout -enddate
```

**Summary**

TLS provides secure transport by combining authentication, key exchange, and encrypted records. PKI supplies the trust framework that makes certificate-based authentication possible. For exam and operational readiness, focus on the handshake sequence, certificate chain validation, SAN-based hostname matching, forward secrecy, revocation concepts, and practical troubleshooting with OpenSSL and server configuration.
