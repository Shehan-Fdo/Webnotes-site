**Weak cryptography: what it means**

Weak cryptography is not only about “old algorithms.” It also includes poor key handling, bad protocol choices, unsafe defaults, and implementations that expose secrets through configuration mistakes. In CEH-style analysis, you assess both the algorithm and the deployment.

A secure algorithm can still fail in practice if the system uses short keys, weak random number generation, reused keys, outdated protocol versions, or insecure storage of secrets. For example, AES<sup><a href="#fn1">1</a></sup> can be strong, but a weak mode, poor IV handling, or exposed keys can make the overall design vulnerable.

- **Weak algorithms** include MD5, SHA-1, DES, RC4, and legacy SSL/TLS versions.
- **Weak deployment** includes hardcoded keys, predictable salts, reused nonces, and exposed private keys.
- **Weak policy** includes no rotation, no revocation process, and no baseline for approved ciphers.

**Common attack types against weak cryptography**

CEH questions often ask you to identify the attack method from the symptom. Weak cryptography is commonly attacked through brute force, dictionary attacks, rainbow tables, and cryptanalysis.

- **Brute force** tries every possible key or password until one works.
- **Dictionary attacks** try likely passwords, phrases, or known patterns first.
- **Rainbow tables** use precomputed hash lookups to crack unsalted or weakly salted password hashes.
- **Cryptanalysis** studies the structure of the algorithm or protocol to find weaknesses without exhaustive guessing.

A fast hash such as MD5 or SHA-1 is poor for password storage because it is optimized for speed, which helps attackers test guesses quickly. Password storage should use slow, adaptive password hashing functions such as bcrypt, scrypt, or Argon2.

> If a password hash is unsalted, identical passwords produce identical hashes and become much easier to crack at scale.

**Hashing weaknesses and password storage**

Hashing is for integrity and one-way verification, not for reversible secrecy. In practice, the biggest mistakes are using fast hashes for passwords, omitting salts, and storing hashes without a work factor.

- **Unsalted hashes** allow precomputed attacks and reveal password reuse.
- **Fast hashes** such as MD5, SHA-1, and SHA-256 are not suitable for password storage by themselves.
- **Adaptive hashes** such as bcrypt, scrypt, and Argon2 slow down offline cracking.
- **Salts** make identical passwords hash differently and defeat rainbow tables.
- **Pepper** is a secret value stored separately from the database; it adds another layer if the database is stolen.

Example of a better password hashing approach in application design:

```python
from argon2 import PasswordHasher

ph = PasswordHasher(time_cost=3, memory_cost=65536, parallelism=2)
hash_value = ph.hash("CorrectHorseBatteryStaple!")
print(hash_value)

# Verification
ph.verify(hash_value, "CorrectHorseBatteryStaple!")
```

For exam purposes, remember the distinction:

- **Hashing** protects integrity and password verification.
- **Encryption** protects confidentiality and is reversible with a key.
- **Encoding** changes representation and is not security.

**Symmetric and asymmetric key management**

Weak cryptography often comes from poor key management rather than the cipher itself. Key length, generation quality, storage, rotation, and revocation all matter.

- **Key length** should match the threat model and current standards.
  - AES-128 is generally acceptable in many environments.
  - AES-256 is often chosen for higher assurance or policy requirements.
  - RSA-2048 is a common minimum baseline in many environments, while stronger deployments may use RSA-3072 or elliptic curve cryptography.
- **Key generation** should use a cryptographically secure random number generator.
- **Key storage** should use hardware security modules (HSMs), key management services, or protected secret stores when possible.
- **Key rotation** reduces exposure if a key is leaked or long-lived.
- **Revocation** is required when a certificate or key is compromised.
- **Separation of duties** helps prevent one administrator from controlling both key material and all systems that use it.

Practical key handling examples:

```bash
# Generate a strong random key for testing
openssl rand -hex 32

# Inspect a certificate chain
openssl s_client -connect example.com:443 -servername example.com

# Check certificate details
openssl x509 -in server.crt -text -noout
```

> A strong key algorithm does not help if the private key is world-readable, committed to a repository, or reused across many systems.

**Protocol and cipher weaknesses**

Weak cryptography also appears in protocol negotiation. Older protocols may allow downgrade attacks, weak cipher suites, or insecure renegotiation behavior.

- **SSLv2 and SSLv3** are obsolete and should be disabled.
- **TLS 1.0 and TLS 1.1** are legacy and commonly disallowed by modern baselines.
- **TLS 1.2** remains widely used and can be secure with strong cipher suites.
- **TLS 1.3** simplifies cipher selection and removes many legacy weaknesses.

For web servers, validate both protocol support and the actual cipher policy. Do not assume that listing a cipher in a config file means it applies to every TLS version.

Example checks:

```bash
# Enumerate TLS support and ciphers
nmap --script ssl-enum-ciphers -p 443 example.com

# Test for weak protocols
openssl s_client -connect example.com:443 -tls1
openssl s_client -connect example.com:443 -tls1_1
openssl s_client -connect example.com:443 -tls1_2
openssl s_client -connect example.com:443 -tls1_3
```

For Nginx, modern hardening usually separates TLS 1.2 cipher policy from TLS 1.3 behavior:

```nginx
server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;

    # TLS 1.2 and earlier cipher selection
    ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:
                 ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305';

    ssl_certificate     /etc/ssl/certs/example.crt;
    ssl_certificate_key /etc/ssl/private/example.key;
}
```

For Apache `mod_ssl`, `SSLCipherSuite` primarily affects pre-TLS 1.3 cipher selection, so you should verify TLS 1.3 behavior separately according to the server and OpenSSL version in use.

**Misconfiguration analysis: what to look for**

Misconfiguration analysis is the process of finding insecure defaults, exposed services, excessive permissions, and policy violations. In CEH scenarios, the goal is to identify whether the issue is informational, medium risk, or high risk based on exposure and exploitability.

- **Informational** findings may include banner disclosure or non-sensitive version leakage.
- **Medium risk** findings often include weak hardening, unnecessary services, or permissive settings with limited exposure.
- **High risk** findings usually include public exposure of sensitive data, default credentials, weak authentication, or remote code execution paths.

A useful rule: if the issue directly exposes data, credentials, or administrative control, treat it as high risk.

**Systematic web server hardening checks**

When assessing a web server, check more than TLS. Review headers, directory listing, default pages, file permissions, and exposed management interfaces.

1. Confirm the service and version.
2. Check TLS protocols and cipher support.
3. Review security headers.
4. Look for directory listing and default content.
5. Test for exposed admin panels or sample applications.
6. Validate file permissions and ownership on the host.

Useful commands:

```bash
# Identify service versions
nmap -sV -p 80,443 example.com

# Fetch response headers
curl -I https://example.com

# Check for directory listing
curl -s https://example.com/uploads/ | head

# Look for default pages
curl -s https://example.com/ | grep -Ei 'welcome|default|test|sample'

# Check for common security headers
curl -sI https://example.com | grep -Ei 'content-security-policy|strict-transport-security|x-frame-options|x-content-type-options|referrer-policy'
```

Common web hardening indicators:

- **HSTS** (`Strict-Transport-Security`) helps enforce HTTPS.
- **CSP** (`Content-Security-Policy`) reduces script injection impact.
- **X-Frame-Options** or `frame-ancestors` helps prevent clickjacking.
- **X-Content-Type-Options: nosniff`** reduces MIME sniffing issues.
- **Directory listing** should usually be disabled unless intentionally required.
- **Default pages** and sample apps should be removed in production.

**Service-specific misconfiguration patterns**

CEH-style questions often test whether you can recognize insecure defaults across common services.

- **DNS**
  - Open recursion can be abused for amplification.
  - Zone transfers should be restricted to authorized secondary servers.
  - Version disclosure and misconfigured authoritative records can aid reconnaissance.
- **Email**
  - Open relay allows abuse for spam and phishing.
  - Weak SPF, DKIM, and DMARC settings reduce spoofing resistance.
  - Insecure SMTP submission or exposed admin consoles increase risk.
- **FTP**
  - Plain FTP sends credentials in cleartext.
  - Anonymous write access is dangerous.
  - Passive port ranges should be controlled and monitored.
- **LDAP**
  - Anonymous bind may expose directory data.
  - Unencrypted LDAP on port 389 can leak credentials unless StartTLS or LDAPS is enforced.
  - Overly broad directory permissions expose user and group information.
- **VPN**
  - Weak pre-shared keys, outdated protocols, and split-tunnel mistakes can expose internal resources.
  - Poor MFA (multi-factor authentication) enforcement weakens remote access security.
  - Overly permissive routes can create lateral movement paths.

**Database and authentication misconfigurations**

Database exposure is a common exam scenario because it combines weak crypto, weak authentication, and network misconfiguration.

- **Default credentials** on database consoles are a critical issue.
- **Publicly exposed ports** such as 3306, 5432, 1433, or 27017 are high risk if not tightly controlled.
- **Weak password hashing** in application databases makes credential theft more damaging.
- **Cleartext connections** to databases can expose credentials and data in transit.

PostgreSQL hardening example:

```conf
# pg_hba.conf
# Prefer SCRAM over MD5 for modern deployments
host    all     all     10.10.0.0/16    scram-sha-256
host    all     all     127.0.0.1/32    scram-sha-256
```

MySQL and MariaDB checks:

```sql
SHOW VARIABLES LIKE 'require_secure_transport';
SHOW VARIABLES LIKE 'default_authentication_plugin';
```

For modern deployments, prefer secure transport, strong authentication plugins, and least-privilege database roles.

**Cloud misconfiguration analysis**

Cloud issues often come from exposed storage, overly broad IAM (Identity and Access Management), and permissive network rules. The same pattern appears across AWS, Azure, and GCP.

- **AWS**
  - Public S3 buckets can expose files, backups, or logs.
  - Security groups with `0.0.0.0/0` on admin ports increase attack surface.
  - Overly broad IAM roles allow privilege escalation or data access.
- **Azure**
  - Public Blob containers can expose data.
  - Network security groups with broad inbound rules can expose services.
  - Excessive role assignments in Azure RBAC (Role-Based Access Control) can grant more access than intended.
- **GCP**
  - Public Cloud Storage buckets can expose objects.
  - Firewall rules that allow broad ingress to management ports are risky.
  - Over-permissive IAM bindings can expose projects, secrets, or service accounts.

Example cloud exposure checks:

```bash
# AWS: list public bucket policy indicators
aws s3api get-bucket-policy-status --bucket example-bucket

# Azure: inspect storage container access
az storage container show --name example --account-name mystorageaccount

# GCP: inspect bucket IAM
gsutil iam get gs://example-bucket
```

Common cloud findings:

- **Public storage** containing backups, logs, or source code.
- **Overly broad IAM** such as wildcard permissions.
- **Open management ports** like SSH, RDP, or database ports exposed to the internet.
- **Missing logging** or disabled audit trails that reduce detection capability.

**Configuration baselines and compliance**

Hardening is easier when you compare systems against a known baseline. In practice, organizations use CIS Benchmarks, vendor hardening guides, internal security standards, and regulatory requirements.

- **CIS Benchmarks** provide prescriptive hardening recommendations for many platforms.
- **Vendor baselines** often reflect supported secure defaults.
- **Internal policy** may be stricter than public guidance due to business risk.
- **Compliance requirements** can mandate encryption, logging, access control, and retention controls.

A baseline helps you answer two exam questions:

- Is the configuration insecure by common industry standards?
- Is the issue a policy violation even if the system still functions?

Example baseline checks:

```bash
# Linux permissions and ownership
stat /etc/ssh/sshd_config
ls -l /etc/ssl/private/

# Search for world-readable secrets
find / -type f \( -name "*.key" -o -name "*.pem" -o -name "*.env" \) -perm -o+r 2>/dev/null

# Check listening services
ss -tulpn
```

**Practical validation workflow**

A good assessment workflow reduces false positives and helps you prioritize findings.

1. Identify exposed services and versions.
2. Enumerate supported protocols, ciphers, and authentication methods.
3. Check for default credentials, anonymous access, and weak permissions.
4. Review storage, logs, and backups for sensitive data exposure.
5. Compare the configuration against a baseline or policy.
6. Rate the finding by exposure, exploitability, and impact.

Example workflow for a web host:

```bash
nmap -sV -p 21,22,25,53,80,110,143,389,443,445,3389 target.example
curl -I https://target.example
nmap --script http-title,http-headers -p 80,443 target.example
nmap --script ssl-enum-ciphers -p 443 target.example
```

**How to judge severity in exam scenarios**

Severity depends on what an attacker can do with the weakness.

- **High risk**
  - Publicly accessible sensitive data
  - Weak or default credentials
  - Cleartext credential transmission
  - Exposed private keys or secrets
  - Open admin interfaces to the internet
- **Medium risk**
  - Weak cipher suites with limited exposure
  - Missing security headers
  - Directory listing on non-sensitive content
  - Overly broad but not directly exploitable permissions
- **Informational**
  - Version disclosure without a known exploitable weakness
  - Non-sensitive banner leakage
  - Minor deviations from a preferred baseline

When in doubt, ask whether the issue enables confidentiality loss, integrity loss, or unauthorized control. If yes, it is usually more than informational.

**Quick reference: secure direction**

- Use modern protocols and disable legacy ones.
- Prefer adaptive password hashing for credentials.
- Generate keys with strong randomness and protect them at rest.
- Rotate and revoke keys and certificates on a defined schedule.
- Restrict network exposure with least privilege.
- Validate web, DNS, email, FTP, LDAP, and VPN configurations.
- Compare systems to CIS or organizational baselines.
- Verify findings with commands, not assumptions.

**Exam cues to remember**

- **Weak crypto** often means outdated algorithms, weak keys, or poor storage.
- **Misconfiguration** often means insecure defaults, excessive exposure, or missing controls.
- **Attack method** clues include brute force, dictionary, rainbow table, and cryptanalysis.
- **Cloud clues** often involve public storage and permissive IAM or firewall rules.
- **Severity** depends on exposure and impact, not just the presence of a weak setting.
