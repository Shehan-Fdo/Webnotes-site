**Cryptography fundamentals**

Cryptography protects data by transforming readable information, called **plaintext**, into unreadable **ciphertext** using an algorithm and a key. The reverse process is **decryption**. In security work, you use cryptography to support confidentiality, integrity, authentication, and non-repudiation.

Two major families dominate practical cryptography:

- **Symmetric cryptography** uses the same secret key for encryption and decryption.
- **Asymmetric cryptography** uses a public key and a private key pair.

The choice between them depends on speed, scalability, trust model, and the security goal. In real systems, you usually combine both: asymmetric cryptography establishes trust or exchanges keys, and symmetric cryptography protects bulk data.

**Symmetric cryptography**

Symmetric cryptography is fast and efficient, which makes it ideal for encrypting large amounts of data. Both sender and receiver must already share the same secret key, so the main challenge is secure key distribution.

Common symmetric algorithms include:

- **AES** (Advanced Encryption Standard), the modern standard for most general-purpose encryption.
- **DES** (Data Encryption Standard), now obsolete because its 56-bit key is too small.
- **3DES** (Triple DES), a legacy replacement for DES that is slower and increasingly deprecated.
- **Blowfish** and **Twofish**, older but still relevant in some systems.
- **ChaCha20**, a modern stream cipher often used in high-performance software encryption.

Symmetric encryption is commonly used for:

- Full-disk encryption
- File and folder encryption
- VPN payload encryption
- Database encryption
- Secure messaging payloads after session setup

A key operational advantage is speed. Symmetric ciphers are computationally cheaper than asymmetric ciphers, so they scale well for large files and high-throughput traffic.

**Modes of operation**

Block ciphers such as AES encrypt fixed-size blocks, so they need a **mode of operation** to handle data longer than one block. The mode affects security and behavior.

- **ECB** (Electronic Codebook) encrypts each block independently.
  - It is weak because identical plaintext blocks produce identical ciphertext blocks.
  - Avoid it for almost all real-world use.
- **CBC** (Cipher Block Chaining) chains each block with the previous ciphertext block.
  - It hides patterns better than ECB.
  - It requires an unpredictable initialization vector, or **IV**.
- **CFB** (Cipher Feedback) and **OFB** (Output Feedback) turn block ciphers into stream-like modes.
  - They are useful in some legacy environments.
- **CTR** (Counter mode) encrypts a counter value and XORs it with plaintext.
  - It is parallelizable and efficient.
- **GCM** (Galois/Counter Mode) provides both encryption and authentication.
  - It is widely preferred because it offers **authenticated encryption with associated data** (AEAD).

> Never reuse a nonce or IV where the algorithm or mode requires uniqueness. Reuse can break confidentiality and, in some cases, integrity.

**Practical symmetric examples**

OpenSSL can generate a symmetric key and encrypt a file:

```bash
openssl rand -hex 32
openssl enc -aes-256-cbc -salt -in report.txt -out report.txt.enc
```

To decrypt, you must use the same algorithm, password, and parameters:

```bash
openssl enc -d -aes-256-cbc -in report.txt.enc -out report.txt
```

A more modern example uses AES-GCM in application code or libraries because it provides encryption plus integrity protection. In practice, you should prefer authenticated encryption rather than encryption alone.

**Asymmetric cryptography**

Asymmetric cryptography uses two mathematically related keys:

- A **public key**, which you can share openly.
- A **private key**, which you must keep secret.

Data encrypted with one key can be decrypted only with the other key, depending on the operation. This property enables secure key exchange, digital signatures, and identity verification without a pre-shared secret.

Common asymmetric algorithms include:

- **RSA** (Rivest-Shamir-Adleman), widely used for encryption and signatures.
- **ECC** (Elliptic Curve Cryptography), which provides strong security with smaller keys.
- **DSA** (Digital Signature Algorithm), used primarily for signatures.
- **Diffie-Hellman** and **Elliptic Curve Diffie-Hellman** (ECDH), used for key agreement rather than direct encryption.

Asymmetric cryptography is slower than symmetric cryptography because the math is more expensive. For that reason, it is rarely used to encrypt large files directly. Instead, it protects a small symmetric session key or signs data.

**Primary uses of asymmetric cryptography**

- **Key exchange**: securely establish a shared symmetric key over an untrusted network.
- **Digital signatures**: prove origin and detect tampering.
- **Certificate-based trust**: bind a public key to an identity using a certificate authority, or **CA**.
- **Non-repudiation**: make it difficult for a signer to deny having signed data.

A digital signature is not the same as encryption. Signing provides authenticity and integrity; encryption provides confidentiality. Many systems use both.

**Public key infrastructure**

A **public key infrastructure** (**PKI**) manages certificates, keys, and trust relationships. It is the foundation for TLS, secure email, code signing, and many enterprise authentication systems.

Core PKI components include:

- **Certificate Authority (CA)**: issues and signs certificates.
- **Registration Authority (RA)**: verifies identity before certificate issuance in some deployments.
- **Digital certificate**: binds a public key to an identity.
- **Certificate revocation**: invalidates certificates before expiration.
- **Certificate Revocation List (CRL)** and **Online Certificate Status Protocol (OCSP)**: methods for checking revocation status.

A certificate typically contains:

- Subject identity
- Public key
- Issuer identity
- Validity period
- Serial number
- Signature from the issuing CA
- Key usage and extended key usage constraints

Example of inspecting a certificate with OpenSSL:

```bash
openssl x509 -in server.crt -text -noout
```

**How asymmetric encryption and signatures work**

For encryption, the sender uses the recipient’s public key so only the recipient’s private key can decrypt the data. For signatures, the sender uses their private key to sign data, and anyone with the public key can verify it.

Typical workflow:

1. You hash the message.
2. You sign the hash with the private key.
3. The verifier hashes the received message again.
4. The verifier checks the signature with the public key.

This design is efficient because the signature operation works on a hash rather than the full message. It also means the security of the signature depends on both the signature algorithm and the hash function.

**Practical asymmetric examples**

Generate an RSA private key and public key:

```bash
openssl genpkey -algorithm RSA -out private.key -pkeyopt rsa_keygen_bits:3072
openssl pkey -in private.key -pubout -out public.key
```

Create and verify a digital signature:

```bash
openssl dgst -sha256 -sign private.key -out file.sig document.txt
openssl dgst -sha256 -verify public.key -signature file.sig document.txt
```

Encrypt a small file with a public key:

```bash
openssl pkeyutl -encrypt -pubin -inkey public.key -in secret.txt -out secret.txt.enc
openssl pkeyutl -decrypt -inkey private.key -in secret.txt.enc -out secret.txt
```

In real deployments, direct public-key encryption is usually limited to small data because of size and performance constraints.

**Hybrid encryption**

Most modern secure systems use **hybrid encryption**. The process combines the strengths of both cryptographic families:

- Asymmetric cryptography authenticates parties or exchanges a session key.
- Symmetric cryptography encrypts the actual data stream.

This is how TLS works in practice:

- The server presents a certificate.
- The client validates the certificate chain.
- The client and server negotiate a shared session key.
- Bulk traffic uses symmetric encryption such as AES-GCM or ChaCha20-Poly1305.

Hybrid design is the standard because it solves the key distribution problem without sacrificing performance.

**Comparison of symmetric and asymmetric cryptography**

| Aspect | Symmetric | Asymmetric |
|---|---|---|
| Keys | One shared secret key | Public/private key pair |
| Speed | Fast | Slower |
| Scalability | Harder key distribution | Easier distribution |
| Main use | Bulk encryption | Key exchange, signatures, identity |
| Security dependency | Secret key protection | Private key protection and trust in public key |
| Typical examples | AES, ChaCha20, 3DES | RSA, ECC, DSA, DH, ECDH |

**Key management concerns**

Cryptography fails most often because of poor key management rather than weak algorithms.

Important practices include:

- Store private keys securely, ideally in hardware such as an **HSM** (Hardware Security Module) or TPM (Trusted Platform Module).
- Rotate keys according to policy and risk.
- Use strong randomness for key generation.
- Protect backups of keys with the same rigor as live keys.
- Separate duties so no single person controls all trust material.
- Revoke compromised certificates immediately.

For symmetric systems, the hardest problem is distributing the shared secret safely. For asymmetric systems, the hardest problem is validating that a public key really belongs to the claimed identity.

**Common attack and failure patterns**

- **Weak keys**: short or predictable keys are brute-forced more easily.
- **ECB misuse**: leaks patterns in structured data.
- **IV or nonce reuse**: can reveal plaintext relationships or break authentication.
- **Padding oracle attacks**: exploit error handling in some CBC implementations.
- **Man-in-the-middle attacks**: occur when certificate validation fails or is bypassed.
- **Private key compromise**: breaks all trust tied to that key.
- **Poor randomness**: undermines key generation and nonce selection.

> Do not assume encryption alone means security. If integrity and authenticity are missing, attackers may still modify or replay data.

**Real-world protocol examples**

- **TLS/HTTPS** uses asymmetric cryptography for authentication and key exchange, then symmetric cryptography for session traffic.
- **SSH** uses public key authentication and symmetric session encryption.
- **PGP/GPG** uses public keys for trust and symmetric session keys for message encryption.
- **VPNs** often use asymmetric methods during handshake and symmetric ciphers for tunnel traffic.
- **Code signing** uses asymmetric signatures to verify software integrity and publisher identity.

**Exam-focused distinctions**

- **Confidentiality** is usually provided by encryption.
- **Integrity** is often provided by hashes and authenticated encryption.
- **Authentication** is provided by certificates, signatures, or shared secrets.
- **Non-repudiation** is associated with digital signatures and private-key control.
- **Symmetric encryption** is best for speed and large data.
- **Asymmetric cryptography** is best for trust establishment and signatures.

A useful memory aid is:

- **Symmetric = same key, fast, secret sharing problem**
- **Asymmetric = key pair, slower, solves trust and distribution**

**Quick command examples for study**

Inspect a private key:

```bash
openssl pkey -in private.key -text -noout
```

Generate a self-signed certificate for lab use:

```bash
openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem -days 365 -nodes
```

Check certificate details and validity:

```bash
openssl x509 -in cert.pem -noout -subject -issuer -dates
```

These commands are useful in labs because they show how keys, certificates, and signatures fit together in practice.

**Summary**

Symmetric cryptography is the workhorse for fast data protection, while asymmetric cryptography provides identity, trust, and secure key exchange. Modern security systems combine both to get the performance of symmetric encryption and the trust properties of public-key cryptography. For CEH-style questions, focus on the differences in key usage, speed, typical applications, and the role of PKI in establishing trust.
