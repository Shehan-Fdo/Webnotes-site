## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**CEH v13 exam orientation**

CEH v13 focuses on the mindset, terminology, and workflow of an ethical hacker. You are expected to recognize what phase of an assessment a scenario describes, which tool or artifact fits that phase, and which action is appropriate in an authorized engagement.

The exam is commonly delivered as a **4-hour** test with **125 multiple-choice questions**. EC-Council uses **scaled scoring**, so do not memorize a simple percentage as the passing threshold. Treat the exam as a recognition and reasoning test rather than a hands-on lab exam.

- Expect scenario-based questions that ask what to do next, what phase a clue belongs to, or which artifact supports a finding.
- Expect terminology questions that distinguish reconnaissance, scanning, exploitation, post-exploitation, and reporting.
- Expect distractors that mix attacker behavior with authorized testing behavior.

**What to memorize vs. what to interpret**

A useful way to study CEH v13 is to separate facts you should memorize from concepts you should interpret in context.

- Memorize:
  - Core phase names in the ethical hacking methodology.
  - Common tool categories and what they are used for.
  - Authorization artifacts such as scope, rules of engagement, and permission letters.
  - High-level port, protocol, and service associations.
- Interpret:
  - Which phase a scenario belongs to.
  - Whether a tool output is enough to support a conclusion.
  - Whether an action is allowed under the stated engagement rules.
  - Which remediation best matches the observed weakness.

**CEH v13 exam blueprint orientation**

The official CEH v13 blueprint is organized around broad domains that cover reconnaissance, scanning, enumeration, vulnerability analysis, system hacking, malware, sniffing, social engineering, web, wireless, cloud, cryptography, and more. The exact domain names and weights can vary by version and delivery, so use the current EC-Council blueprint as the final reference.

A practical study approach is to prioritize the areas that appear most often in scenario questions:

| Study priority | Typical focus | Why it matters |
|---|---|---|
| High | Reconnaissance, scanning, enumeration, system hacking, web attacks | These domains drive many “what phase is this?” and “what tool fits?” questions |
| High | Vulnerability analysis and reporting | CEH often tests interpretation of findings and next-step decisions |
| Medium | Wireless, sniffing, social engineering | Commonly tested through tool recognition and attack flow questions |
| Medium | Cryptography and cloud | Usually conceptual, with terminology and use-case questions |
| Lower but still important | Malware, IoT, OT, mobile, and newer attack surfaces | Often appear as definition or scenario questions rather than deep command work |

> Do not assume the blueprint is static. If your course or exam provider publishes an updated domain breakdown, use that as the source of truth.

**Ethical hacking methodology**

CEH commonly frames the assessment lifecycle as a sequence of phases that mirrors attacker behavior but is used in an authorized, defensive context. The classic flow is:

1. Reconnaissance
2. Scanning
3. Gaining Access
4. Maintaining Access
5. Covering Tracks
6. Reporting

Enumeration is often treated as part of reconnaissance or scanning rather than a separate lifecycle phase. On the exam, that distinction matters because a question may describe service discovery, user enumeration, or banner grabbing and expect you to place it inside the broader discovery phase.

```text
Reconnaissance -> Scanning/Enumeration -> Gaining Access -> Maintaining Access -> Covering Tracks -> Reporting
```

**Phase-by-phase exam meaning**

**Reconnaissance**

Reconnaissance is the information-gathering stage before active intrusion attempts. In CEH questions, this phase often includes passive intelligence collection and low-noise discovery.

- Passive recon examples:
  - WHOIS lookups
  - DNS record review
  - Public website analysis
  - Search engine queries
  - Social media and employee profiling
- Active recon examples:
  - Direct host probing
  - DNS zone transfer attempts
  - Service banner checks
  - Targeted ping sweeps

Common exam clues:

- You see public data sources, metadata, or OSINT (Open Source Intelligence).
- The question emphasizes “without touching the target directly.”
- The answer involves identifying assets, people, domains, or technologies.

Practical examples:

```bash
whois example.com
dig example.com any
nslookup -type=mx example.com
theHarvester -d example.com -b all
```

**Scanning and enumeration**

Scanning identifies live hosts, open ports, services, and sometimes operating system fingerprints. Enumeration goes deeper by extracting usernames, shares, SNMP data, directory listings, or application details.

- Scanning focuses on:
  - Host discovery
  - Port discovery
  - Service/version detection
  - OS fingerprinting
- Enumeration focuses on:
  - SMB shares
  - LDAP directory data
  - SNMP values
  - SMTP user validation
  - Web directory and parameter discovery

Common exam clues:

- The question mentions open ports, service versions, or network ranges.
- The question asks for a tool that identifies hosts or services.
- The question describes banner grabbing, share listing, or user discovery.

Practical examples:

```bash
nmap -sn 10.10.10.0/24
nmap -sS -sV -O 10.10.10.25
nmap --script smb-enum-shares -p 445 10.10.10.25
snmpwalk -v2c -c public 10.10.10.25
```

Use `nmap` carefully in exam interpretation. Script categories can help identify likely issues, but they are not a substitute for a full vulnerability assessment platform.

**Gaining access**

This phase covers exploitation after you identify a weakness. CEH questions often ask you to match a vulnerability with the exploit class or the post-compromise result.

- Typical themes:
  - Password attacks
  - Web application injection
  - Buffer overflow concepts
  - Misconfiguration exploitation
  - Privilege escalation
- Common outputs:
  - Shell access
  - Unauthorized file access
  - Session hijacking
  - Command execution

Practical examples:

```bash
hydra -l admin -P passwords.txt ssh://10.10.10.25
sqlmap -u "http://10.10.10.25/item.php?id=1" --batch
msfconsole
```

On the exam, distinguish between:
- **Discovery** tools that find weaknesses
- **Exploitation** tools that attempt to use them
- **Post-exploitation** actions that happen after access is obtained

**Maintaining access**

Maintaining access refers to persistence after initial compromise. In CEH exam language, this is usually about understanding how an attacker could preserve access, not about performing those actions in a real authorized assessment unless explicitly permitted.

- Common concepts:
  - Backdoors
  - Persistence mechanisms
  - Scheduled tasks
  - Service modification
  - Remote access tooling
- Typical question pattern:
  - “How could an attacker regain access after reboot?”
  - “Which technique provides persistence?”

> In authorized testing, you should follow the engagement scope and avoid actions that exceed permission. CEH may discuss persistence as an attacker concept, but your answer should still reflect the rules of the scenario.

**Covering tracks**

Covering tracks is about reducing evidence of activity, such as log awareness, artifact cleanup, timestamp manipulation concepts, and detection evasion concepts. CEH may ask you to recognize the idea, but you should not treat it as a normal or recommended step in a legitimate assessment.

- Common concepts:
  - Log tampering
  - Clearing event records
  - Deleting temporary files
  - Hiding tools
  - Timestamp changes
- Exam focus:
  - Recognize the term
  - Identify the defensive implication
  - Distinguish it from legitimate cleanup after testing

A safer exam interpretation is that authorized testers document actions and preserve evidence rather than “cover tracks.”

**Reporting**

Reporting is the final and often underappreciated phase. CEH questions may ask what should be included after testing or which artifact best communicates risk to stakeholders.

- A strong report usually includes:
  - Scope and authorization summary
  - Methodology used
  - Findings with evidence
  - Risk ratings
  - Business impact
  - Remediation guidance
  - Retest recommendations
- Good reporting answers:
  - What was found?
  - Why does it matter?
  - How can it be fixed?
  - How can it be verified?

Example report structure:

```text
1. Executive summary
2. Scope and rules of engagement
3. Methodology
4. Findings and evidence
5. Risk analysis
6. Remediation recommendations
7. Retest results
```

**Pre-engagement and authorization**

CEH v13 expects you to understand that an ethical hack begins with permission. Questions often test whether you can identify the correct authorization artifact or engagement boundary.

- Common authorization artifacts:
  - Signed scope statement
  - Rules of engagement
  - Statement of work
  - Permission letter
  - Non-disclosure agreement (NDA)
- Scope details may include:
  - Target IP ranges
  - Allowed hours
  - Allowed techniques
  - Excluded systems
  - Contact escalation path

Practical example of a rules-of-engagement excerpt:

```text
Scope: 10.10.10.0/24 and app.example.com
Allowed: External scanning, web testing, credential attacks against test accounts
Not allowed: Denial-of-service, social engineering, physical access
Window: 22:00-02:00 UTC
Escalation: security@example.com, +1-555-0100
```

Common exam traps:

- A tool is technically useful, but the scenario forbids its use.
- A technique is valid in general, but not within the stated scope.
- The correct answer is to stop and ask for approval rather than continue.

**Engagement types**

CEH questions may describe different assessment styles. You should recognize the level of prior knowledge and access the tester has.

- **Black box**:
  - Little or no internal knowledge
  - Simulates an external attacker
  - Heavy emphasis on recon and discovery
- **White box**:
  - Full or near-full knowledge
  - May include source code, architecture, or credentials
  - Faster path to deeper validation
- **Gray box**:
  - Partial knowledge
  - Common in realistic assessments
  - Mixes recon with targeted validation

A quick comparison:

| Type | Knowledge | Typical focus | Exam clue |
|---|---|---|---|
| Black box | Minimal | Recon, scanning, external attack path | “No prior information” |
| Gray box | Partial | Targeted testing, privilege boundaries | “Limited credentials” |
| White box | Full | Deep validation, code review, architecture | “Source code provided” |

**Common CEH v13 question patterns**

CEH questions often test recognition more than memorization of command syntax. You should practice mapping clues to phases and tools.

- Phase identification:
  - “Which phase is banner grabbing?”
  - “Which phase is user enumeration?”
- Tool recognition:
  - “Which tool is best for packet capture?”
  - “Which tool identifies web directories?”
- Authorization logic:
  - “What should you do if the target is outside scope?”
- Remediation logic:
  - “What control best reduces the risk?”
- Evidence interpretation:
  - “Which log or artifact supports the finding?”

**Scenario-to-phase mapping**

Use the following mental model to classify scenarios quickly.

| Scenario clue | Likely phase | Why |
|---|---|---|
| Searching public employee data and DNS records | Reconnaissance | No direct target interaction or minimal interaction |
| Running a port scan and service version detection | Scanning | Identifies live hosts and exposed services |
| Listing SMB shares or SNMP values | Enumeration | Extracts deeper service data |
| Using a password spray against a login portal | Gaining access | Attempts to obtain unauthorized access |
| Creating a scheduled task for persistence | Maintaining access | Preserves access after compromise |
| Clearing logs or altering timestamps | Covering tracks | Reduces evidence of activity |
| Writing findings and remediation steps | Reporting | Communicates results and fixes |

**Practical command examples by phase**

**Reconnaissance**

```bash
curl -I https://example.com
dig +short example.com
whois example.com
```

**Scanning**

```bash
nmap -Pn -sS -sV -O 10.10.10.25
masscan 10.10.10.0/24 -p80,443 --rate 1000
```

**Enumeration**

```bash
enum4linux-ng -A 10.10.10.25
ldapsearch -x -H ldap://10.10.10.25 -b "dc=example,dc=com"
snmpwalk -v2c -c public 10.10.10.25
```

**Web discovery**

```bash
gobuster dir -u https://example.com -w /usr/share/wordlists/dirb/common.txt
ffuf -u https://example.com/FUZZ -w /usr/share/wordlists/dirb/common.txt
```

**Wireless discovery**

```bash
iw dev
ip link show
airmon-ng
airodump-ng wlan0mon
```

Wireless tooling and interface handling can vary by version and driver support, so focus on the concept of monitor mode and capture rather than memorizing one exact command sequence.

**Reporting and remediation examples**

A CEH answer is often strongest when it connects the weakness to a control.

- If the issue is weak passwords:
  - Enforce MFA (Multi-Factor Authentication)
  - Increase password policy strength
  - Add lockout and rate limiting
- If the issue is exposed services:
  - Restrict access with firewall rules
  - Remove unused services
  - Segment the network
- If the issue is vulnerable web input:
  - Validate input server-side
  - Use parameterized queries
  - Add WAF (Web Application Firewall) rules as a compensating control

Example remediation note:

```text
Finding: Anonymous SMB share exposes internal documents.
Impact: Confidential data exposure and reconnaissance support.
Remediation: Disable anonymous access, restrict share permissions, and audit share contents.
Verification: Confirm access is denied without authenticated credentials.
```

**Typical distractors and how to reject them**

CEH exam items often include answers that are technically related but not best for the scenario.

- A full vulnerability scanner is offered when the question asks for a port scanner.
- A persistence technique is offered when the question asks for initial access.
- A log-clearing action is offered when the question asks for legitimate post-test cleanup.
- A tool that can do many things is offered when the question asks for the most appropriate single-purpose tool.

When in doubt, choose the answer that best matches:
1. The stated phase
2. The stated authorization
3. The least assumption-heavy interpretation

**End-to-end scenario examples**

**Scenario 1: External web assessment**

You are given a public domain, a test window, and permission to assess the web application. You start with DNS lookups, identify the web server, enumerate directories, and then test a parameter for injection.

- Reconnaissance: `dig`, `whois`, website review
- Scanning: service discovery and version checks
- Enumeration: directory brute force, parameter discovery
- Gaining access: injection testing if authorized
- Reporting: document the vulnerable parameter and remediation

**Scenario 2: Internal network validation**

You receive a gray-box scope with one domain account and a subnet. You identify live hosts, enumerate SMB shares, and discover a misconfigured share with sensitive files.

- Reconnaissance: internal asset discovery
- Scanning: `nmap` host and port discovery
- Enumeration: SMB share listing
- Gaining access: only if the scenario authorizes further testing
- Reporting: explain exposure, access path, and least-privilege fix

**Scenario 3: Post-compromise question**

A question says an attacker has obtained a shell and wants to survive a reboot. The correct interpretation is persistence, not scanning or enumeration.

- The clue is about maintaining access.
- The likely answer is a persistence mechanism.
- The wrong answers are discovery tools or reporting actions.

**Study strategy for CEH v13**

A strong approach is to study in layers.

1. Learn the phase names and what each phase means.
2. Learn the common tools and outputs associated with each phase.
3. Practice scenario mapping with short prompts.
4. Review authorization and scope language carefully.
5. Practice remediation thinking, not just attack recognition.

**Final exam readiness checklist**

- You can identify the phase from a one-sentence scenario.
- You can explain why a tool fits a task.
- You can distinguish enumeration from scanning.
- You can recognize authorization artifacts and scope limits.
- You can choose a remediation that matches the weakness.
- You can read a question for what it asks, not what it implies.

**Quick memory anchors**

- **Reconnaissance**: learn about the target.
- **Scanning**: find live hosts and services.
- **Enumeration**: extract deeper service details.
- **Gaining access**: exploit a weakness.
- **Maintaining access**: preserve foothold.
- **Covering tracks**: understand evidence reduction concepts.
- **Reporting**: communicate findings and fixes.
