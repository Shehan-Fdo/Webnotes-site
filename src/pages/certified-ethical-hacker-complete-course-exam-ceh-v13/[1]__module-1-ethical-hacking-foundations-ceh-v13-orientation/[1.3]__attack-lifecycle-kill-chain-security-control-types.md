## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Attack lifecycle overview**

Attack lifecycle models help you understand how intrusions unfold, where defenders can interrupt them, and which evidence each stage leaves behind. In CEH-style questions, the goal is not just to name a stage, but to connect it to attacker behavior, likely artifacts, and the right control type.

Two models appear often:

- **Cyber Kill Chain**: a mostly linear model that describes the sequence of an intrusion from reconnaissance to actions on objectives.
- **MITRE ATT&CK**: a knowledge base of adversary tactics and techniques that is not strictly linear and is used heavily for detection engineering, threat hunting, and control mapping.

The practical exam value is this: the kill chain helps you reason about *where* to stop an attack, while ATT&CK helps you reason about *how* an attacker behaves at each step and *what* to detect.

**Cyber Kill Chain**

The Cyber Kill Chain is a staged model originally used to describe how an attacker progresses from initial research to impact. It is useful for defensive planning because each stage suggests different prevention and detection opportunities.

Typical stages:

1. **Reconnaissance**
   - The attacker gathers information about targets, staff, exposed services, domains, and technologies.
   - Common artifacts include DNS lookups, WHOIS queries, social media profiling, and passive internet scans.
   - Defenses include external attack surface management, rate limiting, security awareness, and monitoring for unusual enumeration.

2. **Weaponization**
   - The attacker prepares a payload, exploit, or malicious document.
   - Examples include a macro-enabled document, a trojanized installer, or a crafted exploit for a known vulnerability.
   - Defenses include patching, sandboxing, attachment detonation, and application allowlisting.

3. **Delivery**
   - The payload reaches the victim through email, web, USB, remote service abuse, or supply chain compromise.
   - Common artifacts include phishing emails, malicious URLs, and suspicious file hashes.
   - Defenses include secure email gateways, web filtering, DNS filtering, and user training.

4. **Exploitation**
   - The payload triggers a vulnerability or user action to execute code.
   - Artifacts include exploit traces, crash logs, abnormal process creation, and script execution.
   - Defenses include endpoint detection and response (EDR), exploit mitigation, patch management, and browser hardening.

5. **Installation**
   - The attacker installs malware, persistence mechanisms, or backdoors.
   - Artifacts include new services, scheduled tasks, registry run keys, cron jobs, and dropped binaries.
   - Defenses include host intrusion prevention, application control, file integrity monitoring, and EDR.

6. **Command and Control**
   - The compromised host communicates with attacker infrastructure.
   - Artifacts include beaconing, unusual outbound connections, DNS tunneling, and encrypted traffic to rare destinations.
   - Defenses include network intrusion detection system (NIDS), proxy inspection, egress filtering, and threat intelligence.

7. **Actions on Objectives**
   - The attacker performs the final goal: data theft, sabotage, fraud, ransomware deployment, or lateral expansion.
   - Artifacts include file access spikes, archive creation, privilege escalation, and mass encryption.
   - Defenses include data loss prevention (DLP), backups, segmentation, least privilege, and incident response.

**Why the kill chain matters in defense**

The kill chain is especially useful for showing that defense is layered. If you cannot stop reconnaissance, you may still stop delivery. If delivery fails, exploitation never happens. If installation succeeds, you may still detect command and control before exfiltration or impact.

> The kill chain is linear by design, so it is best for explaining progression and choke points. It is less flexible for modern attacks that branch, repeat, or blend multiple techniques across the same phase.

**MITRE ATT&CK tactics and techniques**

MITRE ATT&CK is a matrix of adversary tactics and techniques based on observed behavior. Unlike the kill chain, ATT&CK is not a single path. Attackers can skip, repeat, or combine tactics depending on the environment and objective.

For CEH, you should know the major enterprise tactics and what they mean operationally.

**Reconnaissance**

The attacker collects information about the target before direct interaction.

- **Active scanning**: probing ports, services, and web applications.
- **Gather victim identity information**: employee names, email formats, org charts, and public documents.
- **Search open websites/domains**: exposed metadata, code repositories, and cloud assets.
- **Phishing for information**: pretexting to elicit details from users.

Defensive focus:
- External scanning detection
- Brand monitoring
- Security awareness
- Exposure management

**Resource Development**

The attacker builds or acquires resources needed for the operation.

- **Acquire infrastructure**: domains, VPS instances, bulletproof hosting, or cloud accounts.
- **Obtain capabilities**: malware, phishing kits, exploit code, or stolen credentials.
- **Stage capabilities**: prepare payloads, lures, and command infrastructure.

Defensive focus:
- Threat intelligence
- Domain monitoring
- Sinkholing
- Takedown coordination

**Initial Access**

The attacker gains entry to the environment.

- **Phishing**
- **Exploit public-facing application**
- **Valid accounts**
- **Drive-by compromise**
- **External remote services**
- **Supply chain compromise**

Defensive focus:
- MFA (multi-factor authentication)
- Secure email gateway
- WAF (Web Application Firewall)
- VPN hardening
- Patch management

**Execution**

The attacker runs code on a system.

- **Command and scripting interpreter**: PowerShell, Bash, Python, or Windows Command Prompt.
- **User execution**: malicious attachment or link clicked by the user.
- **Scheduled task/cron**: execution through automation.
- **System services**: service abuse or service creation.

Defensive focus:
- Application allowlisting
- Script logging
- EDR
- Macro restrictions
- Behavioral analytics

**Persistence**

The attacker maintains access across reboots or credential changes.

- **Registry run keys / startup folder**
- **Scheduled tasks**
- **Services**
- **Web shells**
- **Account manipulation**
- **Valid accounts**

Defensive focus:
- Baseline monitoring
- Privileged account review
- File integrity monitoring
- EDR and host hardening

**Privilege Escalation**

The attacker gains higher permissions.

- **Exploitation for privilege escalation**
- **Abuse elevation control mechanism**
- **Token impersonation**
- **Setuid / sudo abuse**
- **Kernel exploit**
- **Weak service permissions**

Defensive focus:
- Least privilege
- Patch management
- Privileged access management (PAM)
- Credential hygiene
- Hardening of local admin rights

**Defense Evasion**

The attacker avoids detection or blocks security tools.

- **Obfuscated files or information**
- **Disable or modify tools**
- **Masquerading**
- **Clear logs**
- **Signed binary proxy execution**
- **Indicator removal on host**

Defensive focus:
- Tamper protection
- Centralized logging
- EDR with self-protection
- Script block logging
- File reputation and allowlisting

**Credential Access**

The attacker steals authentication material.

- **OS credential dumping**
- **LSASS memory access**
- **Password spraying**
- **Brute force**
- **Keylogging**
- **Credential from password stores**
- **Kerberoasting**
- **Phishing for credentials**

Defensive focus:
- MFA
- Credential Guard or equivalent protections
- Strong password policy
- Account lockout tuning
- LAPS / local admin password management
- Detection of abnormal authentication patterns

**Discovery**

The attacker maps the environment.

- **System information discovery**
- **Network service discovery**
- **Account discovery**
- **Domain trust discovery**
- **Remote system discovery**
- **Permission groups discovery**
- **Cloud service discovery**

Defensive focus:
- Alert on enumeration bursts
- Segment networks
- Restrict admin tools
- Monitor directory queries and remote admin activity

**Lateral Movement**

The attacker moves to other systems.

- **Remote services**: RDP, SMB, WMI, SSH, WinRM.
- **Pass the hash**
- **Pass the ticket**
- **Remote desktop protocol**
- **Internal spearphishing**
- **Remote file copy**

Defensive focus:
- Network segmentation
- Tiered admin model
- MFA for remote access
- Restrict lateral protocols
- Monitor east-west traffic

**Collection**

The attacker gathers data of interest.

- **Data from local system**
- **Email collection**
- **Clipboard data**
- **Screen capture**
- **Archive collected data**
- **Input capture**

Defensive focus:
- DLP
- Endpoint monitoring
- Sensitive file access alerts
- Rights management
- Data classification

**Command and Control**

The attacker communicates with compromised assets.

- **Web protocol channels**
- **DNS tunneling**
- **Encrypted channels**
- **Proxy**
- **Remote access software**
- **Fallback channels**

Defensive focus:
- Proxy logs
- DNS analytics
- Egress filtering
- TLS inspection where appropriate
- Beaconing detection

**Exfiltration**

The attacker removes data from the environment.

- **Exfiltration over C2 channel**
- **Exfiltration to cloud storage**
- **Exfiltration to removable media**
- **Automated exfiltration**
- **Archive and compress before transfer**

Defensive focus:
- DLP
- CASB (Cloud Access Security Broker)
- Egress controls
- Alert on large outbound transfers
- Encryption monitoring and anomaly detection

**Impact**

The attacker disrupts operations or destroys data.

- **Data encrypted for impact**
- **Data destruction**
- **Service stop**
- **Defacement**
- **Account lockout**
- **Firmware corruption**
- **Resource hijacking**

Defensive focus:
- Backups
- Recovery testing
- Segmentation
- Immutable storage
- Incident response playbooks

**Kill chain vs ATT&CK**

These models are often compared in exam questions, so you should know the difference in defensive use cases and limitations.

| Aspect | Cyber Kill Chain | MITRE ATT&CK |
|---|---|---|
| Structure | Linear sequence | Matrix of tactics and techniques |
| Best use | High-level intrusion progression | Detection engineering and threat hunting |
| Flexibility | Lower | Higher |
| Defensive value | Shows where to break the attack | Shows how attackers behave and what telemetry to collect |
| Limitation | Can oversimplify modern attacks | Can feel broad without a scenario or telemetry context |

The kill chain is useful when you want to explain *stage-based prevention*. ATT&CK is useful when you want to map *observed behavior* to detections, hunts, and mitigations. In real incidents, attackers often revisit tactics, use multiple techniques in parallel, or skip stages entirely. That is why ATT&CK is not strictly linear.

**Security control types**

Security controls are often classified in more than one way. CEH questions may ask you to identify the control by purpose, by implementation, or by whether it prevents, detects, or corrects.

**By function**

- **Preventive controls** stop an event before it happens.
  - Examples: MFA, firewalls, WAF, patching, least privilege, application allowlisting.
- **Detective controls** identify an event after or during occurrence.
  - Examples: IDS (Intrusion Detection System), SIEM (Security Information and Event Management), log review, file integrity monitoring.
- **Corrective controls** reduce impact or restore systems after an event.
  - Examples: backups, incident response, reimaging, rollback, disaster recovery.
- **Deterrent controls** discourage an attacker.
  - Examples: warning banners, visible cameras, legal notices, strong security posture messaging.
- **Compensating controls** provide an alternative when the primary control is not feasible.
  - Example: if you cannot deploy MFA on a legacy app, you may isolate it behind a VPN, jump host, or conditional access policy.
- **Compensatory controls** is sometimes used loosely in study material to mean a substitute or offsetting control; in exam contexts, the safer term is usually **compensating control**.

**By implementation scope**

- **Managerial controls**
  - Focus on governance, risk, and policy.
  - Examples: security policy, risk assessment, vendor review, acceptable use policy, incident response plan.
- **Operational controls**
  - Focus on people and processes.
  - Examples: security awareness training, background checks, change management, tabletop exercises, help desk verification procedures.
- **Technical controls**
  - Focus on technology enforcement.
  - Examples: firewalls, EDR, encryption, access control lists, SIEM rules, DLP, WAF.
- **Physical controls**
  - Focus on tangible barriers and environmental protection.
  - Examples: locks, guards, badges, mantraps, CCTV, fences, fire suppression.

**By control nature**

- **Administrative controls**
  - Often overlaps with managerial and operational controls.
  - Examples: policies, standards, procedures, training, audits.
- **Logical controls**
  - Technical enforcement through software or systems.
  - Examples: ACLs, MFA, password policies, session timeouts.
- **Physical controls**
  - Hardware or environmental safeguards.
  - Examples: locks, safes, biometrics, security lighting.

A useful exam shortcut is to ask: does the control change behavior, enforce access, or physically block access? That usually reveals the category.

**CIA triad mapping**

Security controls support the CIA triad in different ways.

- **Confidentiality**
  - Prevent unauthorized disclosure.
  - Controls: encryption, access control, MFA, DLP, data classification, segmentation.
  - Example: a WAF and least privilege reduce exposure of sensitive web data.
- **Integrity**
  - Prevent unauthorized modification.
  - Controls: hashing, digital signatures, file integrity monitoring, change control, code signing, database constraints.
  - Example: application allowlisting helps prevent unauthorized binaries from altering system state.
- **Availability**
  - Keep services accessible and resilient.
  - Controls: redundancy, load balancing, backups, DDoS protection, failover, UPS (Uninterruptible Power Supply).
  - Example: immutable backups and failover clusters reduce ransomware impact.

Many controls support more than one CIA goal. For example, MFA improves confidentiality by reducing unauthorized access, and backups improve availability by enabling recovery.

**Control mapping to attack stages**

A strong exam answer often links the stage to the best control and the likely artifact.

| Stage | Likely artifact | Best detection/prevention | Example telemetry |
|---|---|---|---|
| Reconnaissance | Port scans, DNS queries, web scraping | Rate limiting, external monitoring, WAF | Firewall logs, DNS logs, web server logs |
| Delivery | Phishing email, malicious URL | Secure email gateway, URL filtering | Mail gateway logs, URL reputation hits |
| Exploitation | Exploit payload, crash, script launch | Patch management, EDR, WAF | Process creation, web app logs, IDS alerts |
| Installation | New service, registry key, dropped binary | EDR, application control | Sysmon, file creation, service creation |
| C2 | Beaconing, DNS tunneling | Egress filtering, proxy, NDR | Proxy logs, DNS analytics, NetFlow |
| Lateral movement | RDP/SMB/WMI use | Segmentation, PAM, MFA | Windows event logs, authentication logs |
| Collection | Archive creation, file access burst | DLP, FIM, access monitoring | File access logs, archive events |
| Exfiltration | Large outbound transfer | DLP, CASB, egress controls | Proxy logs, cloud logs, NetFlow |
| Impact | Encryption, deletion, service stop | Backups, IR, segmentation | EDR alerts, backup logs, service logs |

**Practical control examples**

Real-world controls are easier to remember when you see how they look in configuration.

**WAF rule example**

A WAF can block common SQL injection patterns and suspicious request bodies.

```apache
# ModSecurity-style example
SecRule REQUEST_URI "@contains /login" "id:1001,phase:2,deny,status:403,msg:'Login abuse monitoring'"
SecRule ARGS "@rx (?i)(union\s+select|or\s+1=1|sleep\()" "id:1002,phase:2,deny,status:403,msg:'Possible SQL injection'"
```

This does not replace secure coding, but it reduces exposure and buys time while you patch the application.

**IDS/IPS rule example**

An IDS (Intrusion Detection System) alerts; an IPS (Intrusion Prevention System) can block inline traffic.

```snort
alert tcp any any -> $HOME_NET 445 (
  msg:"SMB lateral movement attempt";
  flow:to_server,established;
  content:"|FF|SMB";
  sid:1000001;
  rev:1;
)
```

Use this kind of rule to detect suspicious SMB activity, especially when paired with authentication logs and host telemetry.

**SIEM correlation logic example**

A SIEM can correlate weak signals into a stronger incident.

```yaml
rule_name: Possible Password Spraying
logic:
  - count_failed_logins_by_source_ip >= 20 within 10m
  - distinct_usernames >= 10 within 10m
  - no_successful_logins within 10m
response:
  - create_high_severity_alert
  - enrich_with_geoip_and_user_agent
  - notify_soc
```

This is useful because a single failed login is noise, but repeated failures across many accounts from one source often indicate password spraying.

**Email gateway filtering example**

A secure email gateway can reduce phishing delivery.

```text
If sender_domain is new OR SPF fails OR DKIM fails OR DMARC fails
AND message contains attachment type in { .iso, .img, .js, .vbs, .docm }
THEN quarantine and detonate in sandbox
```

This is a practical preventive control for the delivery stage, especially when combined with user reporting and mailbox protection.

**Endpoint hardening example**

```powershell
# Example: enable PowerShell logging and reduce script abuse
Set-ItemProperty -Path "HKLM:\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ScriptBlockLogging" `
  -Name "EnableScriptBlockLogging" -Value 1

Set-ItemProperty -Path "HKLM:\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ModuleLogging" `
  -Name "EnableModuleLogging" -Value 1
```

Logging does not stop execution by itself, but it improves detection and forensic visibility.

**Real-world incident walkthroughs**

**Phishing to ransomware**

A finance user receives a convincing invoice email. The attachment is a macro-enabled document that launches PowerShell, downloads a loader, and installs persistence through a scheduled task. The attacker then uses stolen credentials to access a file server, enumerates shares, and deploys ransomware across the domain.

What happened by model:

- **Kill chain**
  - Delivery: phishing email
  - Exploitation: macro execution
  - Installation: loader and scheduled task
  - C2: outbound beaconing
  - Actions on objectives: encryption and disruption
- **ATT&CK**
  - Initial Access: phishing
  - Execution: PowerShell
  - Persistence: scheduled task
  - Credential Access: credential dumping or token theft
  - Discovery: share enumeration
  - Lateral Movement: remote service use
  - Impact: data encrypted for impact

Best controls:

- Secure email gateway
- Macro restrictions
- EDR with script logging
- MFA
- Segmentation
- Offline or immutable backups

**Web app compromise to data exfiltration**

An attacker finds a vulnerable public web application, exploits it, drops a web shell, and uses it to enumerate the backend database. They compress records and exfiltrate them over HTTPS to cloud storage.

What happened by model:

- **Kill chain**
  - Reconnaissance: app fingerprinting
  - Delivery/Exploitation: exploit public-facing application
  - Installation: web shell
  - C2: HTTPS beaconing
  - Actions on objectives: data theft
- **ATT&CK**
  - Initial Access: exploit public-facing application
  - Persistence: web shell
  - Discovery: system and network discovery
  - Collection: database export
  - Exfiltration: cloud storage transfer

Best controls:

- Patch management
- WAF
- Least privilege on database accounts
- File integrity monitoring
- Egress filtering
- DLP and cloud monitoring

**How to think in exam scenarios**

When a question describes an attacker action, identify three things:

1. **Where in the lifecycle it occurs**
   - Is it entry, execution, persistence, movement, theft, or impact?
2. **What evidence it leaves**
   - Logs, processes, network flows, file changes, authentication events.
3. **Which control is most effective**
   - Preventive if you can stop it early, detective if you need visibility, corrective if recovery is the priority.

A common trap is confusing detection with prevention. For example, a SIEM correlates events, but it does not block them. A WAF can block malicious requests, but it may also log them for analysis. An IDS alerts, while an IPS can drop traffic.

**Quick comparison of common control examples**

| Control | Type | Primary role | Typical CEH use |
|---|---|---|---|
| Firewall | Technical, preventive | Filter network traffic | Block unauthorized ports and services |
| IDS | Technical, detective | Alert on suspicious traffic | Detect scans, exploits, and C2 |
| IPS | Technical, preventive/detective | Block malicious traffic inline | Stop known exploit patterns |
| SIEM | Technical, detective | Correlate logs | Identify multi-stage attacks |
| WAF | Technical, preventive | Protect web apps | Block SQL injection and XSS patterns |
| DLP | Technical, preventive/detective | Stop data leakage | Prevent exfiltration of sensitive data |
| MFA | Technical, preventive | Strengthen authentication | Reduce credential abuse |
| Backup | Corrective | Restore data and systems | Recover from ransomware |
| Policy | Managerial/administrative | Set rules and expectations | Define acceptable use and access requirements |
| Security training | Operational/administrative | Reduce human error | Improve phishing resistance |

**Exam-oriented distinctions**

- **Preventive vs detective**
  - Preventive stops or reduces the chance of compromise.
  - Detective identifies compromise or suspicious behavior.
- **Compensating vs compensatory**
  - In practice, exam questions usually mean **compensating control**: an alternative control used when the primary one is not possible.
- **Administrative vs managerial**
  - Many study resources use these terms loosely.
  - For exam safety, remember that managerial controls set direction and risk posture, while administrative controls are the policies and procedures that implement that direction.
- **Technical vs physical**
  - Technical controls are enforced by systems.
  - Physical controls are enforced by the environment.

**Key takeaways**

The kill chain gives you a linear story of intrusion progression, while ATT&CK gives you a richer behavior map for real-world adversaries. Security controls should be understood by function, implementation scope, and their relationship to the CIA triad. In CEH scenarios, the best answer usually connects the attacker action, the artifact it creates, and the control that either prevents, detects, or recovers from it.
