**Evidence collection fundamentals**

Evidence collection is the process of gathering artifacts that support a security finding, confirm a compromise, or preserve facts for later analysis. In ethical hacking, you collect enough evidence to prove the issue without causing unnecessary disruption or altering the target more than needed.

Good evidence is **relevant**, **repeatable**, and **defensible**. It should answer three questions:

- What happened?
- How do you know?
- Why does it matter?

In practice, you often collect a mix of screenshots, logs, packet captures, command output, timestamps, hashes, and configuration excerpts. The goal is to make the finding verifiable by another analyst.

**Types of evidence**

Different evidence types support different parts of an assessment:

- **Volatile evidence** changes quickly and may disappear after reboot or process termination.
  - Examples: running processes, active network connections, logged-in users, memory contents.
- **Non-volatile evidence** persists across reboots.
  - Examples: files, registry keys, configuration files, scheduled tasks, installed packages.
- **Direct evidence** comes from the affected system or service.
  - Examples: server logs, application errors, local command output.
- **Indirect evidence** comes from supporting systems.
  - Examples: SIEM alerts, firewall logs, DNS logs, proxy logs.
- **Corroborating evidence** strengthens a claim by showing the same event from multiple sources.
  - Example: a web server log, WAF alert, and packet capture all show the same suspicious request.

**Evidence handling and chain of custody**

When evidence may be used in a formal report or legal context, you need a clear chain of custody. That means documenting who collected the evidence, when it was collected, where it came from, how it was stored, and who accessed it.

A defensible workflow usually includes:

1. Identify the artifact or system.
2. Record the date, time, timezone, and collector identity.
3. Capture the evidence with minimal alteration.
4. Compute hashes for files when possible.
5. Store evidence securely with access control.
6. Log every transfer or review.

A common file integrity check uses hashes:

```bash
sha256sum suspicious.bin
sha256sum -c hashes.txt
```

For disk or memory acquisition, you typically preserve originals and work from copies. If you must interact with a live system, note that your actions can change timestamps, logs, or process state.

> If you cannot preserve the original state, document the limitation clearly. A well-documented partial capture is better than an undocumented one.

**Practical collection examples**

On Linux, you may gather process, network, and log evidence with standard tools:

```bash
ps auxf
ss -tulpn
lsof -i
journalctl -u nginx --since "2026-04-26 10:00:00"
```

On Windows, common evidence sources include Event Viewer, PowerShell output, and registry exports:

```powershell
Get-Process | Sort-Object CPU -Descending | Select-Object -First 10
Get-NetTCPConnection | Sort-Object State,LocalPort
wevtutil qe Security /f:text /c:20
reg export HKLM\Software\Microsoft\Windows\CurrentVersion\Run runkeys.reg
```

For network evidence, packet captures and flow logs are especially useful:

```bash
tcpdump -i eth0 -w capture.pcap host 10.10.10.25
tshark -r capture.pcap -Y "http.request"
```

**Severity assessment**

Severity describes how serious a finding is from a technical risk perspective. It is not the same as business impact, although the two are related. A vulnerability can be technically severe but have limited business impact if it affects a non-critical system with strong compensating controls.

A practical severity assessment considers:

- **Exploitability**: how easy it is to attack.
- **Impact**: what the attacker can do after exploitation.
- **Exposure**: whether the asset is internet-facing or internal.
- **Privileges required**: whether the attacker needs credentials or local access.
- **User interaction**: whether a victim must click, open, or execute something.
- **Scope**: whether the issue affects one component or multiple systems.
- **Detectability and control strength**: whether monitoring, segmentation, or hardening reduces risk.

Many organizations map findings to a scale such as low, medium, high, or critical. Some also use the <abbr title="Common Vulnerability Scoring System">CVSS</abbr> as a baseline. CVSS is useful for consistency, but it should not replace contextual judgment.

**Using CVSS as a guide**

CVSS helps standardize severity by scoring exploitability and impact. It is best used as a starting point, then adjusted for environment-specific context.

Typical considerations include:

- Internet exposure increases practical severity.
- A vulnerability on a domain controller is usually more serious than the same issue on a lab workstation.
- A flaw with a public exploit and no authentication requirement is more urgent than one requiring local admin access.
- Strong compensating controls can reduce operational risk even if the technical score remains high.

Example CVSS-style reasoning:

- A remote code execution issue on a public web server with no authentication is often **critical**.
- A local privilege escalation on a hardened admin-only host may be **high** or **medium** depending on access paths.
- An information disclosure issue on an internal test system may be **low** if the data is non-sensitive.

**Business impact**

Business impact explains what the organization loses if the issue is exploited. This is the language decision-makers understand. It translates technical findings into operational, financial, legal, and reputational consequences.

Common impact categories include:

- **Confidentiality**: exposure of sensitive data, intellectual property, credentials, or customer records.
- **Integrity**: unauthorized modification of data, transactions, logs, or configurations.
- **Availability**: downtime, service degradation, or loss of productivity.
- **Compliance**: violations of regulatory or contractual obligations.
- **Financial loss**: fraud, incident response costs, recovery expenses, or lost revenue.
- **Reputation**: customer trust erosion, media exposure, and brand damage.

A strong report connects the technical issue to a concrete business process. For example, a SQL injection vulnerability is not just a code flaw; it may expose customer records, disrupt order processing, and trigger breach notification obligations.

**Severity versus business impact**

Severity and business impact overlap, but they answer different questions.

| Aspect | Severity | Business impact |
|---|---|---|
| Focus | Technical risk | Organizational consequence |
| Audience | Security team, engineers | Management, stakeholders |
| Basis | Exploitability and technical effect | Data, process, legal, and financial loss |
| Output | Low/medium/high/critical | Operational and strategic consequences |

Use severity to prioritize remediation work. Use business impact to justify urgency, budget, and executive attention.

**How to write a finding that connects both**

A useful finding statement usually includes:

- The vulnerability or weakness.
- The affected asset or service.
- The attack path or proof.
- The likely consequence.
- The severity rating.
- The business impact.

Example structure:

- **Finding**: Unauthenticated directory listing is enabled on the public file server.
- **Evidence**: `curl` output shows browsable backup files and screenshots confirm access.
- **Severity**: High, because the issue is internet-facing and exposes sensitive files.
- **Business impact**: Attackers could retrieve internal documents, customer data, or credentials, leading to data breach and compliance exposure.

**Example report language**

Use concise, decision-oriented wording:

```text
The public-facing application accepts unsanitized input in the search parameter, allowing SQL injection. Testing confirmed unauthorized access to customer records. Because the service is internet-facing and contains regulated data, the issue is rated Critical. Successful exploitation could result in data breach, regulatory penalties, and loss of customer trust.
```

**Evidence collection for common scenarios**

Different attack types require different proof.

- **Web vulnerabilities**
  - Capture request and response pairs.
  - Save screenshots of the browser and raw HTTP traffic.
  - Record the exact payload used and the server response.
- **Credential issues**
  - Preserve proof of weak policy, exposed secrets, or reused credentials.
  - Avoid storing live secrets in plain text unless required and approved.
- **Privilege escalation**
  - Show the before-and-after privilege state.
  - Capture `whoami`, group membership, and the command that triggered escalation.
- **Lateral movement**
  - Correlate source host, destination host, authentication logs, and remote execution artifacts.
- **Data exposure**
  - Record file paths, access controls, and sample metadata.
  - Minimize exposure of sensitive content in the report.

**Useful commands for evidence gathering**

Linux examples:

```bash
whoami
id
hostnamectl
uname -a
ip a
route -n
cat /etc/passwd | head
find /var/log -type f | xargs ls -lt
```

Windows examples:

```powershell
whoami /all
systeminfo
ipconfig /all
net user
net localgroup administrators
Get-ChildItem C:\Windows\Temp
```

Network and service checks:

```bash
nmap -sV -O 10.10.10.25
curl -i http://10.10.10.25/login
openssl s_client -connect 10.10.10.25:443 -servername example.com
```

**Evidence quality checklist**

Before finalizing a finding, verify that your evidence is:

- **Accurate**: it reflects the actual state of the target.
- **Complete**: it supports the claim without major gaps.
- **Time-stamped**: it includes when the event occurred or was observed.
- **Attributable**: it identifies the host, user, or process involved.
- **Reproducible**: another tester can follow the same steps.
- **Minimal**: it avoids unnecessary sensitive data exposure.

**Common mistakes**

Avoid these errors when collecting evidence:

- Taking screenshots without context or timestamps.
- Reporting a severity score without explaining the environment.
- Overstating impact without proof.
- Modifying the target so much that the original state is lost.
- Collecting secrets or personal data that are not needed for the report.
- Failing to distinguish observed facts from assumptions.

**Practical severity mapping example**

A simple internal rubric can help standardize triage:

| Rating | Typical meaning | Example |
|---|---|---|
| Low | Limited impact, difficult to exploit, or heavily constrained | Banner disclosure on an internal host |
| Medium | Requires some conditions or has moderate impact | Authenticated XSS in a low-value app |
| High | Significant impact or easy exploitation with meaningful access | File upload leading to code execution on a public app |
| Critical | Full compromise, major data exposure, or severe business disruption | Remote code execution on a production internet-facing system |

Use the rubric consistently, but adjust for asset value and exposure. A medium technical issue on a mission-critical system may deserve higher operational priority than a high issue on a test environment.

**Reporting recommendations**

A strong report separates facts from interpretation:

- Facts describe what you observed.
- Interpretation explains why it matters.
- Recommendation explains what to do next.

Example:

- **Fact**: The web server exposes `/backup/` and allows directory browsing.
- **Interpretation**: Sensitive files are accessible without authentication.
- **Recommendation**: Disable directory listing, remove exposed backups, and restrict access to authorized users only.

When possible, include remediation guidance that is specific and actionable:

- Patch vulnerable software.
- Remove exposed secrets.
- Enforce least privilege.
- Segment critical systems.
- Add logging and alerting for suspicious activity.
- Validate fixes with retesting.

**Operational context and prioritization**

Not every severe technical issue is the top priority. Prioritize based on:

- Asset criticality.
- Internet exposure.
- Data sensitivity.
- Ease of exploitation.
- Presence of active exploitation.
- Availability of compensating controls.
- Regulatory or contractual deadlines.

A vulnerability on a payroll system with customer and employee data usually outranks the same issue on a disposable test VM. Context turns a raw finding into a meaningful risk decision.

**Quick reference**

- Collect evidence with minimal change to the target.
- Preserve timestamps, hashes, and source context.
- Use multiple sources to corroborate findings.
- Rate severity by exploitability and technical effect.
- Translate technical risk into business consequences.
- Write findings so both engineers and managers can act on them.
