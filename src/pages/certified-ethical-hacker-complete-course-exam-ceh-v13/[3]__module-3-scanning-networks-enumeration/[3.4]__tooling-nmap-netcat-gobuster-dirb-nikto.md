## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

These tools cover a common CEH workflow: discover hosts with **Nmap**, validate services with **Netcat**, enumerate web content with **Gobuster** or **Dirb**, and assess web server exposure with **Nikto**. In practice, you rarely use them in isolation. You scan first, confirm what is actually listening, then enumerate the application surface.

A useful mental model is:

- **Nmap** tells you what is reachable and what may be running.
- **Netcat** lets you manually test a port, banner, or protocol exchange.
- **Gobuster** and **Dirb** brute-force hidden paths, files, and DNS names.
- **Nikto** checks for common web server misconfigurations and known issues.

**Nmap fundamentals**

Nmap is a network discovery and port scanning tool. It supports host discovery, port scanning, service detection, OS fingerprinting, scripting, and output formatting. For CEH, you should recognize both the purpose of each scan type and the meaning of common flags.

Nmap reports several important **port states**:

- **open**: an application is actively accepting connections.
- **closed**: the host is reachable, but nothing is listening on that port.
- **filtered**: a firewall, ACL, or packet filter prevents Nmap from determining the state.
- **unfiltered**: the port is reachable, but Nmap cannot tell whether it is open or closed, often seen with ACK scans.
- **open|filtered**: Nmap cannot distinguish between open and filtered, common with UDP scans.
- **closed|filtered**: Nmap cannot distinguish between closed and filtered in some scan results.

Common scan modes and when to use them:

- **`-sS`**: TCP SYN scan. Fast and common; it sends SYN packets and interprets responses without completing a full handshake.
- **`-sT`**: TCP connect scan. Uses the OS connect call and is more likely to be logged by the target.
- **`-sU`**: UDP scan. Slower and less reliable because many UDP services do not respond.
- **`-sA`**: TCP ACK scan. Useful for firewall rule mapping and identifying filtered vs unfiltered ports.
- **`-sW`**: TCP window scan. Similar to ACK scanning; can sometimes infer open ports from TCP window size behavior.
- **`-sM`**: TCP Maimon scan. A niche scan that sends FIN/ACK probes; useful in limited cases and often blocked.
- **`-sn`**: Host discovery only. It skips port scanning and is used to identify live hosts.
- **`-Pn`**: Treat hosts as online and skip host discovery. Use this when ping probes are blocked or when you already know the host is up.

Host discovery is not limited to ICMP echo. Depending on privileges, target behavior, and network conditions, Nmap may use a mix of probes such as ICMP echo, TCP SYN to common ports, TCP ACK probes, and other discovery methods. That is why a host can appear “down” unless you use `-Pn`.

Typical discovery and scan examples:

```bash
nmap -sn 192.168.1.0/24
nmap -Pn 192.168.1.10
nmap -sS -p 1-1000 192.168.1.10
nmap -sU -p 53,67,68,123 192.168.1.10
```

**Port selection and scan speed**

Port selection matters because scanning all 65,535 ports takes longer and produces more noise.

- **`-p`**: specify ports or ranges, such as `-p 22,80,443` or `-p 1-1000`.
- **`-F`**: fast scan of the most common ports.
- **`--top-ports`**: scan the top N most common ports, such as `--top-ports 100`.
- **`-T0` to `-T5`**: timing templates from paranoid to insane. Higher values are faster and noisier.

Examples:

```bash
nmap -F 192.168.1.10
nmap --top-ports 100 -T4 192.168.1.10
nmap -p 1-65535 -T3 192.168.1.10
```

**Service and version detection**

Service detection helps you move from “port open” to “what is actually running.”

- **`-sV`**: probe open ports to identify service and version.
- **`--version-intensity`**: adjust how aggressively version probes are used.
- **`--version-all`**: use all version probes.

Example:

```bash
nmap -sS -sV -p 22,80,443 192.168.1.10
```

**OS detection and traceroute**

- **`-O`**: attempt OS detection using TCP/IP fingerprinting.
- **`-A`**: aggressive scan. It enables OS detection, version detection, default NSE scripts, and traceroute. It does not run every NSE script.
- **`--traceroute`**: show the route to the target.

Example:

```bash
nmap -A 192.168.1.10
```

Use `-A` when you want a broad first-pass assessment and can tolerate extra noise. For stealthier or more controlled work, combine individual options instead.

**NSE scripts**

The **Nmap Scripting Engine (NSE)** extends Nmap with Lua scripts for discovery, enumeration, and limited exploitation checks. Scripts are grouped into categories.

Common categories:

- **default**: scripts run by `-sC`; generally useful and relatively safe.
- **safe**: intended to be non-disruptive, but still verify context before use.
- **discovery**: gathers additional information about the target.
- **version**: enhances service version detection.
- **auth**: checks authentication-related behavior.
- **brute**: performs brute-force attempts.
- **vuln**: checks for known vulnerabilities; results may be noisy or disruptive depending on the script.
- **exploit**: attempts exploitation.
- **malware**: looks for malware-related indicators.
- **broadcast**: discovers hosts via broadcast traffic.

Important distinction:

- **`-sC`** runs the **default** script category.
- **`--script safe`** runs scripts tagged as safe.
- **`--script vuln`** runs vulnerability checks, but that does not mean every script is harmless.

Useful script examples:

```bash
nmap -sC 192.168.1.10
nmap --script http-title -p 80,443 192.168.1.10
nmap --script http-enum -p 80,443 192.168.1.10
nmap --script smb-os-discovery -p 445 192.168.1.10
nmap --script ssl-cert,ssl-enum-ciphers -p 443 192.168.1.10
```

Practical script use cases:

- **`http-title`**: quickly identifies a web page title.
- **`http-enum`**: enumerates common web paths and directories.
- **`smb-os-discovery`**: extracts SMB host information.
- **`ssl-cert`**: inspects certificate details.
- **`ssl-enum-ciphers`**: lists supported TLS ciphers.

You can combine scripts with ports and targets:

```bash
nmap -p 80,443 --script http-title,http-enum 192.168.1.10
```

**Script database and updates**

When you add or update NSE scripts locally, refresh Nmap’s script database so it can recognize script names and categories.

```bash
nmap --script-updatedb
```

This updates the local script database used for script lookup, not a general index of script behavior.

**Output and verbosity**

CEH questions often ask about output formats and verbosity flags.

- **`-v`**: increase verbosity.
- **`-d`**: increase debugging output.
- **`-oN`**: normal output to a file.
- **`-oX`**: XML output.
- **`-oG`**: grepable output, legacy but still seen in older workflows.
- **`-oA`**: output in all major formats with one basename.

Examples:

```bash
nmap -sS -sV -oN scan.txt 192.168.1.10
nmap -sS -sV -oA scan_results 192.168.1.10
nmap -v -d 192.168.1.10
```

**Common Nmap workflow**

1. Discover live hosts with `-sn` or `-Pn` if discovery is blocked.
2. Scan likely ports with `-F` or `--top-ports`.
3. Expand to targeted ranges with `-p` when you find a promising service.
4. Add `-sV` and `-sC` to identify services and gather basic enumeration.
5. Use focused NSE scripts for the service you found.
6. Save output with `-oA` so you can review and report later.

Example workflow:

```bash
nmap -sn 192.168.1.0/24
nmap -sS -sV -sC -p 22,80,443 192.168.1.10 -oA target1
nmap --script http-title,http-enum -p 80,443 192.168.1.10
```

**Netcat fundamentals**

Netcat is a lightweight TCP/UDP utility used for banner grabbing, simple listeners, file transfer patterns, and quick connectivity checks. Different implementations behave differently, so syntax is not perfectly portable. Common variants include traditional `nc`, OpenBSD `nc`, and `ncat` from the Nmap suite.

Core uses:

- Test whether a port is reachable.
- Manually speak a simple protocol.
- Set up a listener for reverse shells or ad hoc transfers in lab environments.
- Scan a small range of ports in some implementations.
- Send or receive UDP traffic.

Common flags vary by version, but these patterns are widely recognized:

- **`-l`**: listen mode.
- **`-v`**: verbose output.
- **`-n`**: numeric-only addresses, no DNS lookup.
- **`-p`**: local port in some implementations.
- **`-u`**: UDP mode.
- **`-w`**: timeout.
- **`-z`**: zero-I/O mode, useful for port scanning or checking if a port is open.

Examples that are commonly seen:

```bash
nc -nv 192.168.1.10 80
nc -nvu 192.168.1.10 53
nc -zv 192.168.1.10 1-100
```

Because syntax differs across versions, listener commands may vary. On some systems, you may use:

```bash
nc -lv 4444
```

On others, you may need a different form such as `nc -l -p 4444`. Always verify the local implementation with `nc -h` or `man nc`.

**Banner grabbing and protocol validation**

Netcat is useful when you want to see what a service says before you automate against it. For HTTP, you must send a valid request; for many other protocols, the service may speak first or expect a specific handshake.

HTTP example:

```bash
printf 'HEAD / HTTP/1.1\r\nHost: 192.168.1.10\r\nConnection: close\r\n\r\n' | nc -nv 192.168.1.10 80
```

SMTP example:

```bash
nc -nv 192.168.1.10 25
```

UDP example:

```bash
printf 'test' | nc -nvu 192.168.1.10 161
```

**Netcat as a quick port checker**

With `-z`, Netcat can probe ports without sending application data.

```bash
nc -zv 192.168.1.10 22 80 443
nc -zv 192.168.1.10 1-1000
```

This is not a replacement for Nmap, but it is useful for quick validation when you already know the target and want a fast check.

**Ncat differences**

**Ncat** is the Netcat implementation that ships with Nmap. It often supports more features than traditional `nc`, including SSL/TLS, proxying, and better scripting support.

Examples:

```bash
ncat --ssl 192.168.1.10 443
ncat -lvnp 4444
ncat -u 192.168.1.10 53
```

If an exam question asks about a feature like SSL support, proxy support, or richer scripting, the answer may point to Ncat rather than a minimal `nc`.

**Gobuster fundamentals**

Gobuster is a fast brute-force tool for discovering hidden directories, files, DNS subdomains, and virtual hosts. It is commonly used after you identify a web server with Nmap or Nikto.

Main modes:

- **dir**: directory and file brute forcing.
- **dns**: subdomain enumeration.
- **vhost**: virtual host enumeration.
- **fuzz**: generic fuzzing in some versions.

A typical directory scan uses a wordlist and a base URL:

```bash
gobuster dir -u http://192.168.1.10 -w /usr/share/wordlists/dirb/common.txt
```

Useful flags commonly seen in modern Gobuster versions:

- **`-u`**: target URL.
- **`-w`**: wordlist.
- **`-x`**: file extensions to append, such as `php,txt,bak`.
- **`-t`**: number of threads.
- **`-q`**: quiet mode.
- **`-k`**: skip TLS certificate verification.
- **`-b`**: status codes to exclude.
- **`-s`**: status codes to include.
- **`-l`**: show response length.
- **`-r`**: in some versions, follow redirects or control redirect behavior; verify your installed version before relying on it.

Example directory enumeration:

```bash
gobuster dir -u http://192.168.1.10 -w /usr/share/wordlists/dirb/common.txt -x php,txt,bak -t 50 -q
```

**Response filtering**

Web brute forcing often returns false positives because servers respond with custom 404 pages or wildcard behavior. Filtering by status code or length helps reduce noise.

Examples:

```bash
gobuster dir -u http://192.168.1.10 -w words.txt -b 404
gobuster dir -u http://192.168.1.10 -w words.txt -l
```

If the server returns the same length for every nonexistent path, response-length filtering is especially useful.

**DNS subdomain enumeration**

In DNS mode, Gobuster brute-forces subdomains against a domain or DNS server.

```bash
gobuster dns -d example.com -w subdomains.txt
```

Useful options:

- **`-d`**: domain.
- **`-r`**: custom DNS resolver in some versions.
- **`-i`**: show IP addresses.
- **`-q`**: quiet mode.

Example:

```bash
gobuster dns -d example.com -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt -i
```

Use DNS mode when you want to discover names like `dev.example.com`, `admin.example.com`, or `vpn.example.com`.

**Virtual host enumeration**

Vhost enumeration tests whether different `Host` headers map to different sites on the same IP. Gobuster’s vhost mode is designed for this workflow.

```bash
gobuster vhost -u http://192.168.1.10 -w vhosts.txt
```

In vhost mode, Gobuster fuzzes the `Host` header internally. You usually do not manually set a `Host:` header with `-H` for the core vhost workflow. If the target requires a specific base host or DNS resolution, you may need to adjust your local `/etc/hosts` or use the correct base URL and resolver behavior for your version.

**Gobuster workflow**

1. Identify a web server with Nmap or Netcat.
2. Run `gobuster dir` against the base URL.
3. Add extensions with `-x` when the application likely uses dynamic files.
4. Use `-b` or `-l` to reduce false positives.
5. Switch to `dns` or `vhost` mode when you suspect hidden names rather than hidden paths.

Example:

```bash
gobuster dir -u http://192.168.1.10 -w /usr/share/wordlists/dirb/common.txt -x php,txt
gobuster dns -d example.com -w subdomains.txt
gobuster vhost -u http://192.168.1.10 -w vhosts.txt
```

**Dirb fundamentals**

Dirb is an older web content brute-forcer that recursively searches for directories and files using wordlists. It is still common in training labs and exam prep because its output is easy to recognize.

Basic usage:

```bash
dirb http://192.168.1.10 /usr/share/wordlists/dirb/common.txt
```

Dirb is less flexible and less consistent across documentation than Gobuster, so be careful with option syntax. In exam contexts, focus on the concept: it brute-forces web paths and can recurse into discovered directories.

Commonly recognized behaviors:

- It uses a wordlist to test paths.
- It can recurse into discovered directories.
- It can append extensions in some builds and examples.
- It often appears in older lab walkthroughs and legacy notes.

Example with extensions, noting that exact syntax can vary by version:

```bash
dirb http://192.168.1.10 /usr/share/wordlists/dirb/common.txt -X .php,.txt,.bak
```

Because Dirb syntax is less uniform than Gobuster, verify the installed help output before relying on a specific flag set.

**Interpreting Dirb output**

Dirb output often includes:

- The tested URL.
- The HTTP status code.
- The response size.
- Whether a directory or file appears valid.

Watch for:

- **301/302 redirects** that may indicate a directory exists but redirects elsewhere.
- **200 responses** on nonexistent paths, which can indicate wildcard behavior.
- **403 responses** that may still confirm a real directory exists.
- **Recursion** into discovered directories, which can expand coverage quickly but also increase noise.

**Nikto fundamentals**

Nikto is a web server scanner that checks for common vulnerabilities, dangerous files, outdated software, and misconfigurations. It is noisy by design and is best used in authorized lab or assessment contexts.

Basic scan:

```bash
nikto -h http://192.168.1.10
```

Useful options:

- **`-h`**: target host or URL.
- **`-p`**: target port.
- **`-ssl`**: use SSL/TLS.
- **`-Tuning`**: limit checks to selected categories.
- **`-o`**: output file.
- **`-Format`**: output format such as `txt`, `csv`, `xml`, or `html`.
- **`-C all`**: check all cookies.
- **`-Display`**: control what is shown during the scan.

Examples:

```bash
nikto -h http://192.168.1.10 -p 8080
nikto -h 192.168.1.10 -p 443 -ssl
nikto -h http://192.168.1.10 -o nikto.txt -Format txt
nikto -h https://192.168.1.10 -o nikto.html -Format html
```

**Tuning categories**

`-Tuning` narrows the scan to specific classes of checks. The exact category letters are version-dependent in how they are documented, but the key exam idea is that tuning reduces scope and noise.

Example pattern:

```bash
nikto -h http://192.168.1.10 -Tuning x
nikto -h http://192.168.1.10 -Tuning 1,2,3
```

Use tuning when you want to focus on a subset of checks instead of running the full default set. Do not assume a space-separated value like `-Tuning x 6`; that is not valid syntax.

**Legacy references and output**

Nikto may still reference **OSVDB** in older output or historical documentation. OSVDB is defunct, so treat those references as legacy identifiers rather than an active vulnerability source.

Nikto can also identify risky HTTP methods and headers. For example, if `TRACE` is enabled, that can be relevant to **Cross-Site Tracing (XST)** in some environments.

**Practical web enumeration workflow**

A realistic workflow often looks like this:

1. Use Nmap to find open web ports.
2. Use Netcat or Nmap scripts to confirm the service and banner.
3. Use Gobuster or Dirb to discover hidden content.
4. Use Nikto to check for common server issues and misconfigurations.
5. Re-run targeted Nmap NSE scripts against the interesting service.

Example end-to-end sequence:

```bash
nmap -sS -sV -p 80,443,8080 192.168.1.10
printf 'HEAD / HTTP/1.1\r\nHost: 192.168.1.10\r\nConnection: close\r\n\r\n' | nc -nv 192.168.1.10 80
gobuster dir -u http://192.168.1.10 -w /usr/share/wordlists/dirb/common.txt -x php,txt,bak -t 50
nikto -h http://192.168.1.10 -p 80
nmap --script http-title,http-enum -p 80 192.168.1.10
```

**Common flags cheat sheet**

| Tool | Common flags | What they help you do |
|---|---|---|
| Nmap | `-sn`, `-Pn`, `-p`, `-F`, `-sS`, `-sU`, `-sV`, `-O`, `-A`, `-sC`, `--script`, `-oA`, `-v`, `-d` | Discover hosts, scan ports, identify services, run scripts, and save results |
| Netcat | `-l`, `-v`, `-n`, `-u`, `-z`, `-w` | Listen, validate connectivity, use UDP, or probe ports quickly |
| Gobuster | `dir`, `dns`, `vhost`, `-u`, `-w`, `-x`, `-t`, `-b`, `-s`, `-l`, `-q` | Brute-force paths, subdomains, and virtual hosts |
| Dirb | target URL, wordlist, recursion, extension appending | Discover hidden web content in older lab workflows |
| Nikto | `-h`, `-p`, `-ssl`, `-Tuning`, `-o`, `-Format`, `-C all` | Scan web servers for common issues and generate reports |

**Exam-oriented distinctions**

- **Nmap** is the primary reconnaissance and port-scanning tool.
- **Netcat** is the manual validation and ad hoc TCP/UDP utility.
- **Gobuster** is usually the faster, more modern brute-forcer for web paths and DNS names.
- **Dirb** is a legacy web content brute-forcer that still appears in training material.
- **Nikto** is a noisy web server checker focused on common misconfigurations and known issues.

**Quick recall examples**

- Find live hosts:
  ```bash
  nmap -sn 192.168.1.0/24
  ```
- Skip discovery and scan anyway:
  ```bash
  nmap -Pn -sS -p 80,443 192.168.1.10
  ```
- Check a banner:
  ```bash
  nc -nv 192.168.1.10 80
  ```
- Brute-force web paths:
  ```bash
  gobuster dir -u http://192.168.1.10 -w words.txt -x php,txt
  ```
- Legacy web brute force:
  ```bash
  dirb http://192.168.1.10 /usr/share/wordlists/dirb/common.txt
  ```
- Run a web server check:
  ```bash
  nikto -h http://192.168.1.10
  ```

**Key takeaways**

These tools complement each other. Nmap gives you the map, Netcat confirms what is really there, Gobuster and Dirb uncover hidden web surface, and Nikto checks for common web weaknesses. For CEH, focus on recognizing the right tool for the task, the meaning of the most common flags, and the order in which you would use them during a real assessment.
