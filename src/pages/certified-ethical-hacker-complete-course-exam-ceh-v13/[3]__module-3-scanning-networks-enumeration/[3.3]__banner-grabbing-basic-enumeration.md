## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

Banner grabbing and basic enumeration are early reconnaissance activities used to identify live services, software versions, and exposed information before deeper testing. A **banner** is the text or protocol response a service returns when you connect to it. It may reveal the application name, version, operating system hints, supported features, or authentication methods.

The goal is not just to “collect strings.” You use these responses to build a target profile, prioritize likely vulnerabilities, and choose the right next-step tools. In practice, banner grabbing works best when combined with port scanning, service detection, and protocol-specific enumeration.

**Why it matters**

- **Version intelligence** helps you map services to known weaknesses.
- **Protocol behavior** can reveal misconfigurations, weak defaults, or anonymous access.
- **Service exposure** often differs across ports, virtual hosts, and authentication states.
- **Attack surface reduction** becomes easier when you know what is actually running.

**Core concepts**

- **Banner grabbing** is the act of eliciting identifying information from a service.
- **Enumeration** is the systematic extraction of names, shares, users, records, or configuration details from a target.
- **Passive reconnaissance** uses public or indirect sources; **active enumeration** sends probes to the target.
- **Service fingerprinting** compares responses to known patterns to infer software and version.

**Typical workflow**

1. Discover open ports with a scanner such as `nmap`.
2. Identify services and versions with probes or scripts.
3. Manually verify uncertain results with protocol-aware tools.
4. Record findings and correlate them with potential weaknesses.

**Common tools**

- `nmap` for service detection and NSE (Nmap Scripting Engine) scripts
- `netcat` or `nc` for manual TCP interaction
- `telnet` for simple plaintext service testing
- `curl` for HTTP and HTTPS headers
- `openssl s_client` for TLS-enabled services
- `hping3` for packet crafting and response observation
- Specialized utilities such as `whatweb`, `nikto`, `smtp-user-enum`, `snmpwalk`, `ldapsearch`, `enum4linux-ng`, and `rpcclient`

**Banner grabbing techniques**

A banner may appear immediately after connection, or only after you send a protocol-specific request. Some services expose useful headers without authentication, while others require a valid handshake first. You should try both passive connection attempts and minimal valid requests.

- **Direct connection banners** are common on SMTP, FTP, SSH, POP3, IMAP, and some database services.
- **HTTP headers** often reveal server software, reverse proxies, frameworks, and cookies.
- **TLS certificates** can expose hostnames, organization names, and service roles.
- **Protocol negotiation** may reveal supported versions, ciphers, or extensions.

**Manual banner grabbing examples**

Use `nc` for simple plaintext services:

```bash
nc -nv 192.0.2.10 21
nc -nv 192.0.2.10 25
nc -nv 192.0.2.10 110
```

If the service waits for input, send a minimal request:

```bash
printf "HEAD / HTTP/1.1\r\nHost: example.com\r\nConnection: close\r\n\r\n" | nc -nv 192.0.2.10 80
```

For TLS services, use `openssl s_client` to inspect the certificate and negotiated parameters:

```bash
openssl s_client -connect 192.0.2.10:443 -servername example.com
openssl s_client -connect 192.0.2.10:25 -starttls smtp
openssl s_client -connect 192.0.2.10:143 -starttls imap
```

> When a service uses TLS, plain `nc` often shows nothing useful. Use `openssl s_client` or a protocol-aware client so you can complete the handshake and inspect the response.

**Nmap service detection**

`nmap` is often the fastest way to collect banners at scale. Service detection uses probes and fingerprints to infer the application and version.

```bash
nmap -sV -Pn -p 21,22,25,53,80,110,139,143,389,443,445,3389 192.0.2.10
```

Useful options:

- `-sV` enables version detection.
- `-Pn` skips host discovery when ICMP is filtered.
- `--version-all` increases probe intensity.
- `--script banner` attempts simple banner collection.
- `--script ssl-cert,ssl-enum-ciphers` helps with TLS services.

Example:

```bash
nmap -sV --script banner,ssl-cert,ssl-enum-ciphers -p 443,465,587,993,995 192.0.2.10
```

**HTTP and HTTPS enumeration**

HTTP services often expose the most useful metadata. You should inspect response headers, cookies, redirects, virtual hosts, and page content. A single IP may host multiple sites, so host-header-based discovery is important.

```bash
curl -I http://192.0.2.10/
curl -k -I https://192.0.2.10/
```

Look for:

- `Server` and `X-Powered-By` headers
- Redirect chains and canonical hostnames
- Cookie names and security flags
- Framework-specific paths, error pages, and default content
- TLS certificate subject and SAN (Subject Alternative Name) entries

Virtual host enumeration is often necessary when the IP address alone returns a generic page. You can test candidate hostnames by setting the `Host` header:

```bash
curl -I http://192.0.2.10/ -H 'Host: intranet.example.com'
curl -k -I https://192.0.2.10/ -H 'Host: app.example.com'
```

You can also brute-force hostnames with wordlists and compare response codes, lengths, or titles. Tools such as `ffuf`, `gobuster`, or `wfuzz` are commonly used for this purpose.

```bash
ffuf -u http://192.0.2.10/ -H 'Host: FUZZ.example.com' -w subdomains.txt -fs 0
```

**DNS enumeration**

DNS enumeration identifies records, naming patterns, and sometimes internal naming conventions. Do not rely on `ANY` queries as a primary method; modern servers often minimize or refuse them. Instead, query specific record types and use brute force where appropriate.

Common record types:

- `A` and `AAAA` for address mapping
- `MX` for mail exchangers
- `NS` for name servers
- `TXT` for policy and verification records
- `SOA` for zone authority details
- `CNAME` for aliases
- `PTR` for reverse DNS

Examples:

```bash
dig @192.0.2.10 example.com A
dig @192.0.2.10 example.com MX
dig @192.0.2.10 example.com NS
dig @192.0.2.10 example.com TXT
```

Reverse DNS enumeration checks whether IP ranges map back to useful hostnames:

```bash
dig -x 192.0.2.10
```

For subdomain discovery, use brute force against a wordlist and resolve candidates:

```bash
dnsrecon -d example.com -D subdomains.txt -t brt
amass enum -brute -w subdomains.txt -d example.com
```

If you have zone transfer access, you can retrieve the full zone:

```bash
dig axfr @192.0.2.10 example.com
```

> Zone transfers are rare on well-configured servers. Treat a successful `AXFR` as a significant misconfiguration.

**SMTP enumeration**

SMTP often reveals the mail server banner and supported extensions. You can also test for user enumeration if the server is misconfigured, but many modern systems limit or block this.

```bash
nc -nv 192.0.2.10 25
```

Typical SMTP commands:

```text
EHLO test.local
VRFY alice
EXPN staff
MAIL FROM:<test@example.com>
RCPT TO:<user@example.com>
QUIT
```

Useful checks:

- Banner and `EHLO` response
- `STARTTLS` support
- `VRFY` and `EXPN` behavior
- Relay restrictions
- Authentication mechanisms advertised in `EHLO`

For TLS-enabled SMTP:

```bash
openssl s_client -connect 192.0.2.10:25 -starttls smtp
```

**FTP enumeration**

FTP often exposes a clear banner and may allow anonymous access if misconfigured.

```bash
nc -nv 192.0.2.10 21
ftp 192.0.2.10
```

Check for:

- Server product and version
- Anonymous login acceptance
- Writable directories
- Passive mode behavior
- TLS support via `AUTH TLS`

Example:

```bash
printf "USER anonymous\r\nPASS anonymous@\r\n" | nc -nv 192.0.2.10 21
```

**SSH enumeration**

SSH banners often reveal the server implementation and version, which can help with fingerprinting.

```bash
nc -nv 192.0.2.10 22
ssh -vvv user@192.0.2.10
```

Look for:

- OpenSSH or vendor-specific builds
- Supported key exchange and cipher suites
- Host key types
- Authentication methods

SSH banners are usually short, but they are still useful for version correlation and policy assessment.

**SMB and Windows enumeration**

SMB (Server Message Block) and related Windows services can expose hostnames, domain membership, shares, and sometimes user or group information if anonymous access or weak permissions exist. Enumeration is often limited on hardened systems, so expect partial results.

Common tools:

```bash
nmap -p 139,445 --script smb-os-discovery,smb-enum-shares,smb-enum-users 192.0.2.10
enum4linux-ng -A 192.0.2.10
smbclient -L //192.0.2.10 -N
rpcclient -U "" 192.0.2.10
```

Practical notes:

- Anonymous or null-session access is increasingly restricted.
- Share enumeration is often more reliable than user enumeration.
- `smb-os-discovery` may reveal OS hints, but results can be inaccurate or intentionally masked.
- Authenticated enumeration is usually much richer than unauthenticated probing.

Example `rpcclient` commands:

```text
enumdomusers
enumdomgroups
querydominfo
netshareenumall
```

**LDAP and Active Directory enumeration**

LDAP (Lightweight Directory Access Protocol) is central to Active Directory (AD) environments. Enumeration may be anonymous, authenticated, or blocked depending on directory policy. Authenticated binds usually expose much more data than anonymous queries.

Anonymous RootDSE query:

```bash
ldapsearch -x -H ldap://192.0.2.10 -b "" -s base namingContexts
```

This retrieves the directory naming contexts from the RootDSE (Root Directory Service Entry). From there, you can query the base DN.

Example base DN query:

```bash
ldapsearch -x -H ldap://192.0.2.10 -b "DC=example,DC=com" "(objectClass=person)" cn mail
```

AD-oriented filters often need refinement:

- Use `(objectClass=user)` and exclude computer accounts when you want human users.
- Use `(objectClass=group)` for groups.
- Use `(objectClass=computer)` for computer objects.

Example:

```bash
ldapsearch -x -H ldap://192.0.2.10 -b "DC=example,DC=com" "(&(objectClass=user)(!(objectClass=computer)))" sAMAccountName memberOf
```

Useful tools:

```bash
ldapdomaindump -u 'EXAMPLE\alice' -p 'Password123!' ldap://192.0.2.10
```

`ldapdomaindump` helps convert LDAP data into browsable HTML and JSON outputs for later analysis.

**SNMP enumeration**

SNMP (Simple Network Management Protocol) can reveal hostnames, interfaces, routing details, processes, and device configuration data if community strings are weak or default. SNMPv1 and SNMPv2c use community strings; SNMPv3 adds authentication and encryption.

Common community strings include `public` and `private`, but you should treat them as examples, not assumptions.

```bash
snmpwalk -v2c -c public 192.0.2.10
snmpwalk -v2c -c private 192.0.2.10
snmpget -v2c -c public 192.0.2.10 sysDescr.0
```

Useful OIDs (Object Identifiers):

- `1.3.6.1.2.1.1.1.0` for `sysDescr`
- `1.3.6.1.2.1.1.5.0` for `sysName`
- `1.3.6.1.2.1.2.2` for interface tables
- `1.3.6.1.2.1.25` for host resources

SNMP agents may ignore invalid community strings without explicit error messages, and some devices rate-limit or log repeated requests. That behavior is normal and does not guarantee the target is unreachable.

**Banner grabbing over SSL/TLS**

Many services use TLS either directly or through STARTTLS. You need to distinguish between:

- **Implicit TLS**, where encryption starts immediately on connect, such as HTTPS on 443
- **Explicit TLS**, where the client upgrades the connection after a plaintext handshake, such as SMTP with `STARTTLS`

Examples:

```bash
openssl s_client -connect 192.0.2.10:443
openssl s_client -connect 192.0.2.10:587 -starttls smtp
openssl s_client -connect 192.0.2.10:993
```

You can inspect:

- Certificate subject and issuer
- SAN entries and hostnames
- Expiration dates
- Supported protocol versions
- Cipher suite selection

This is especially useful when the application banner is hidden but the certificate still exposes organizational or service information.

**Specialized banner-grabbing and enumeration tools**

Beyond `nmap`, `nc`, `curl`, and `openssl`, several tools help with targeted enumeration:

- `whatweb` identifies web technologies and frameworks
- `nikto` checks for common web misconfigurations and default files
- `hping3` crafts packets to observe firewall and service behavior
- `socat` can proxy or relay connections for testing
- `amass`, `subfinder`, and `dnsrecon` support DNS discovery
- `smtp-user-enum` tests SMTP-based user enumeration where supported
- `onesixtyone` can brute-force SNMP community strings
- `crackmapexec` can assist with SMB and AD enumeration in authorized assessments

Example `whatweb` and `nikto` usage:

```bash
whatweb http://192.0.2.10/
nikto -h https://192.0.2.10/
```

Example `hping3` probe:

```bash
hping3 -S -p 80 -c 1 192.0.2.10
```

**Interpreting results**

A banner is a clue, not proof. You should validate findings with multiple signals before concluding the exact product or version.

- **Cross-check** the banner against headers, certificates, and behavior.
- **Compare** results from different ports and protocols.
- **Watch for deception** such as generic banners, reverse proxies, or custom wrappers.
- **Record confidence levels** when version detection is uncertain.

Common pitfalls:

- A reverse proxy may hide the backend application.
- A load balancer may return a generic service banner.
- A service may intentionally suppress version strings.
- A banner may be spoofed or outdated.
- A successful connection does not imply useful enumeration access.

**Practical workflow example**

1. Scan the target for open ports.
2. Run service detection and banner collection.
3. Manually verify key services with protocol-aware clients.
4. Enumerate DNS, HTTP, SMB, LDAP, and SNMP where exposed.
5. Correlate findings into a target profile.

Example:

```bash
nmap -sV -Pn -p- 192.0.2.10
nmap -sV --script banner,ssl-cert,smb-os-discovery,smb-enum-shares -p 21,22,25,53,80,110,139,143,389,443,445,3389 192.0.2.10
curl -I http://192.0.2.10/
dig @192.0.2.10 example.com MX
snmpwalk -v2c -c public 192.0.2.10
```

**Exam-focused takeaways**

- **Banner grabbing** identifies services and versions early in reconnaissance.
- **Enumeration** extracts structured information from exposed services.
- **HTTP**, **DNS**, **SMB**, **LDAP**, **SNMP**, **SMTP**, **FTP**, and **SSH** are common targets.
- **Authenticated enumeration** usually returns more data than anonymous probing.
- **TLS** and **STARTTLS** require protocol-aware clients for meaningful results.
- **Virtual host discovery** and **reverse DNS** are important parts of modern enumeration.
- **Tool output** should be validated, not trusted blindly.

**Quick reference**

| Protocol | Common clue | Useful tools | Notes |
|---|---|---|---|
| HTTP/HTTPS | Headers, cookies, titles, certs | `curl`, `whatweb`, `nikto`, `ffuf` | Check virtual hosts and TLS |
| DNS | Records, zone data, reverse names | `dig`, `dnsrecon`, `amass` | Avoid relying on `ANY` |
| SMB | Shares, domain info, OS hints | `nmap`, `smbclient`, `enum4linux-ng`, `rpcclient` | User enumeration may be blocked |
| LDAP/AD | Base DN, users, groups, attributes | `ldapsearch`, `ldapdomaindump` | Authenticated binds are richer |
| SNMP | System info, interfaces, routes | `snmpwalk`, `snmpget`, `onesixtyone` | Community strings matter |
| SMTP | Banner, extensions, relay behavior | `nc`, `openssl s_client`, `smtp-user-enum` | `STARTTLS` is common |
| FTP | Banner, anonymous access | `nc`, `ftp` | Check for writable dirs |
| SSH | Version, ciphers, auth methods | `nc`, `ssh -vvv`, `nmap` | Banner is short but useful |

**Troubleshooting tips**

- If a service appears silent, try the correct protocol handshake instead of a raw TCP connect.
- If `nmap -sV` is inconclusive, increase version intensity or use manual verification.
- If HTTP returns a generic page, test alternate hostnames with the `Host` header.
- If LDAP anonymous queries fail, try authenticated enumeration where authorized.
- If SNMP requests time out, verify the community string, version, and ACLs.
- If a TLS service fails with plain tools, use `openssl s_client` or a client that supports `STARTTLS`.
