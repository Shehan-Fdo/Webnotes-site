**Host discovery overview**

Host discovery is the process of determining which systems are alive on a network before you spend time scanning ports or enumerating services. In a real assessment, this step reduces noise, saves time, and helps you focus on reachable targets. It also helps you avoid unnecessary full-port scans against dead hosts.

A typical workflow is:

1. Identify the target range.
2. Probe for live hosts using one or more discovery methods.
3. Confirm responsive systems with follow-up checks.
4. Scan open ports on the confirmed live hosts.

Host discovery is not always straightforward. Firewalls, packet filters, rate limits, and host-based protections can block common probes. For that reason, you often combine multiple techniques instead of relying on a single packet type.

**Why discovery matters**

Discovery is the first practical filter in reconnaissance. If you skip it, you may waste time scanning inactive addresses or miss hosts that suppress certain probe types.

- **Efficiency**: You reduce the number of unnecessary port scans.
- **Accuracy**: You confirm which systems actually respond on the network.
- **Stealth**: You can choose lower-noise methods when needed.
- **Scope control**: You keep testing aligned with the authorized target set.

In many environments, a host may ignore ICMP (Internet Control Message Protocol) echo requests but still respond to TCP connection attempts or ARP (Address Resolution Protocol) requests on the local subnet. That is why “no ping reply” does not mean “host is down.”

**Common host discovery methods**

Host discovery uses different packet types depending on the network position and the level of filtering.

- **ICMP echo request/reply**: Classic “ping” behavior.
- **ICMP timestamp and address mask probes**: Less common, but still conceptually part of ICMP-based discovery.
- **TCP SYN or ACK probes**: Useful when ICMP is filtered.
- **UDP probes**: Less reliable for discovery, but sometimes useful.
- **ARP requests**: Very effective on the local Layer 2 network.
- **Neighbor discovery**: Relevant in IPv6 environments.

On a local Ethernet segment, ARP is often the most reliable discovery mechanism because it operates below IP routing. If a host is on the same broadcast domain, it usually must answer ARP to communicate at all.

**ICMP-based discovery**

ICMP is widely used for reachability testing, but many networks block or rate-limit it. A host may still be alive even if it does not answer echo requests.

Typical ICMP discovery behavior includes:

- **Echo request**: The most familiar ping probe.
- **Echo reply**: Returned if the host allows it.
- **Time exceeded**: Can reveal routing behavior and TTL (Time To Live) expiration.
- **Destination unreachable**: Can indicate filtering, host unreachability, or port unreachability depending on the context.

Example:

```bash
ping -c 4 192.0.2.10
```

If ping fails, you should not immediately conclude the host is offline. Instead, try alternate probes such as TCP SYN discovery or ARP on the local network.

> ICMP blocking is common in hardened environments, so discovery should be multi-method rather than ping-only.

**ARP discovery on local networks**

ARP resolves IPv4 addresses to MAC (Media Access Control) addresses on the local subnet. Because it is required for local communication, it is often the most dependable way to find live hosts on the same Layer 2 segment.

Practical examples:

```bash
arp -a
ip neigh show
```

Nmap can use ARP discovery automatically on local Ethernet networks:

```bash
nmap -sn 192.168.1.0/24
```

On a local subnet, this often produces accurate results even when ICMP is blocked, because Nmap can use ARP requests instead of relying only on ping.

**TCP-based discovery**

TCP discovery uses connection-oriented behavior to infer whether a host is alive. This is useful when ICMP is filtered.

Common approaches:

- **TCP SYN probe**: Sends a SYN and watches for a SYN/ACK or RST response.
- **TCP ACK probe**: Can help infer whether a host is reachable through filtering behavior.
- **TCP connect probe**: Completes the handshake when permitted, but is noisier.

A SYN response does not always mean the port is open in a full scan context, but for host discovery it often confirms the host is reachable.

Example with Nmap:

```bash
nmap -sn -PS22,80,443 203.0.113.0/24
```

This sends TCP SYN probes to ports 22, 80, and 443. If the target responds, Nmap treats the host as up.

**UDP-based discovery**

UDP is connectionless, so discovery is less deterministic. A response may come back from a closed port as an ICMP port unreachable message, but many UDP services do not respond at all unless the probe matches an expected application format.

Use UDP discovery carefully:

- It can help in some environments.
- It is often slower and less reliable than ICMP or TCP discovery.
- It may generate ambiguous results.

Example:

```bash
nmap -sn -PU53,123 203.0.113.0/24
```

This sends UDP probes to common UDP ports such as DNS (53) and NTP (123).

**IPv6 discovery concepts**

IPv6 changes discovery because ARP is replaced by Neighbor Discovery Protocol (NDP), which uses ICMPv6. On local IPv6 networks, neighbor solicitation and advertisement are central to finding active hosts.

Key points:

- **NDP** replaces ARP in IPv6.
- **ICMPv6** is essential to IPv6 operation, so blocking it can break connectivity more severely than in IPv4.
- Discovery often relies on multicast and neighbor cache behavior.

Example:

```bash
ip -6 neigh show
ping6 -c 3 fe80::1
```

**Nmap host discovery options**

Nmap is the most common tool for host discovery and port scanning in CEH-style workflows. The `-sn` option performs host discovery only and skips port scanning.

Useful options:

- `-sn`: Ping scan; discover live hosts only.
- `-Pn`: Treat hosts as up; skip discovery and scan ports directly.
- `-PE`: ICMP echo request.
- `-PP`: ICMP timestamp request.
- `-PM`: ICMP address mask request.
- `-PS`: TCP SYN discovery.
- `-PA`: TCP ACK discovery.
- `-PU`: UDP discovery.
- `-PR`: ARP discovery on local networks.

Examples:

```bash
nmap -sn 192.168.1.0/24
nmap -sn -PE 203.0.113.0/24
nmap -sn -PS80,443 203.0.113.0/24
nmap -sn -PA80,443 203.0.113.0/24
nmap -sn -PR 192.168.1.0/24
```

`-Pn` is useful when discovery is blocked, but it increases scan time because Nmap assumes every target is alive.

```bash
nmap -Pn -sS -p 1-1000 203.0.113.10
```

**Port scanning fundamentals**

Port scanning identifies which transport-layer ports are open, closed, or filtered. This is the bridge between host discovery and service enumeration.

The main transport protocols are:

- **TCP**: Connection-oriented, widely used for web, SSH, mail, and many enterprise services.
- **UDP**: Connectionless, used by DNS, DHCP, SNMP, NTP, and others.

A port scan helps you answer:

- Which services are exposed?
- Which ports are filtered by a firewall?
- Which services may need version detection or deeper enumeration?

**TCP port states**

TCP scans commonly report several states:

- **Open**: An application is listening and accepting connections.
- **Closed**: The host is reachable, but no application is listening on that port.
- **Filtered**: A firewall or filter prevents the probe from reaching the port or returning a clear response.
- **Unfiltered**: The port is reachable, but the scan type cannot determine whether it is open.
- **Open|filtered**: Common in UDP scans when no response is received.
- **Closed|filtered**: Less common, but possible when the scan cannot distinguish the state.

Understanding these states matters because “filtered” often indicates security controls, while “closed” indicates the host is reachable but the service is absent.

**TCP scan types**

Different TCP scan types trade off speed, visibility, and privilege requirements.

- **SYN scan (`-sS`)**:
  - Sends SYN and analyzes the response.
  - Often called half-open scanning.
  - Usually faster and less noisy than a full connect scan.
  - Commonly requires elevated privileges.

- **Connect scan (`-sT`)**:
  - Uses the operating system’s TCP connect call.
  - Completes the full handshake.
  - Works without raw packet privileges.
  - More visible in logs because it creates full connections.

- **ACK scan (`-sA`)**:
  - Helps map firewall rules and filtering behavior.
  - Does not determine open versus closed directly.
  - Useful for understanding packet filtering.

- **FIN, NULL, and Xmas scans (`-sF`, `-sN`, `-sX`)**:
  - Send unusual flag combinations.
  - May behave differently on some systems and filters.
  - Can reveal filtering behavior, but results are not universally reliable.

Example:

```bash
nmap -sS -p 22,80,443 192.0.2.10
nmap -sT -p 1-1000 192.0.2.10
nmap -sA -p 80,443 192.0.2.10
```

**UDP scanning concepts**

UDP scanning is harder because UDP does not establish a session. A lack of response may mean the port is open, filtered, or simply silent.

Typical UDP scan behavior:

- If the port is closed, the host may return ICMP port unreachable.
- If the port is open, the service may respond only to a valid application payload.
- If the port is filtered, no response may be returned.

Example:

```bash
nmap -sU -p 53,67,68,123,161 192.0.2.10
```

Common UDP targets include:

- DNS on `53`
- DHCP on `67` and `68`
- NTP on `123`
- SNMP on `161`

UDP scans are often slower, so you may need to tune timing or limit the port list.

**Service and version detection**

After identifying open ports, you usually move to service detection to learn what is actually running.

Nmap version detection:

```bash
nmap -sV -p 22,80,443 192.0.2.10
```

This helps you distinguish between a port that is merely open and a service that can be fingerprinted. Version detection is valuable because the same port can host different software, and the software version often determines the next testing steps.

**OS detection and fingerprinting**

Operating system detection uses TCP/IP stack behavior, response patterns, and protocol quirks to infer the target OS (Operating System).

Example:

```bash
nmap -O 192.0.2.10
nmap -A 192.0.2.10
```

`-A` enables aggressive detection features, including OS detection, version detection, script scanning, and traceroute. It is useful in lab environments, but it is noisier than targeted scans.

**Practical scan workflow**

A disciplined workflow helps you avoid unnecessary noise and keeps results easier to interpret.

1. Start with discovery on the target subnet.
2. Confirm live hosts with a low-noise scan.
3. Scan a limited set of common ports.
4. Expand to full port ranges if needed.
5. Run version detection on confirmed open ports.
6. Add OS detection and scripts only when appropriate.

Example workflow:

```bash
nmap -sn 192.168.56.0/24
nmap -sS -p 22,80,443 192.168.56.10
nmap -sV -p 22,80,443 192.168.56.10
nmap -O 192.168.56.10
```

**Timing, rate, and reliability**

Scan timing affects both speed and detectability. Faster scans can complete quickly, but they may drop packets or trigger defenses. Slower scans are more reliable in lossy networks and less likely to overwhelm fragile devices.

Useful considerations:

- **Packet loss** can cause false negatives.
- **Rate limiting** can delay or suppress responses.
- **Retransmissions** improve reliability but increase time.
- **Parallelism** speeds up large scans but can increase noise.

Nmap timing templates:

```bash
nmap -T2 -sS 192.0.2.0/24
nmap -T4 -sS 192.0.2.0/24
```

Use conservative timing when scanning unstable links or sensitive environments.

**Firewall and filtering behavior**

Firewalls influence both discovery and port scan results. A firewall may:

- Drop packets silently.
- Reject packets with ICMP errors.
- Allow only specific source addresses.
- Permit some probes but not others.

This is why different scan types can produce different results for the same host. For example, a host may ignore ICMP echo but respond to TCP SYN on port 443, or it may allow ARP on the local subnet while blocking routed traffic.

> A filtered result often reflects network policy, not host absence.

**Practical examples of interpreting results**

If you see no response to ping but a SYN probe succeeds, the host is alive and likely filtering ICMP.

If a TCP port returns RST, the host is reachable and the port is closed.

If a UDP probe gets no response, you cannot assume the port is open; you need context, payload knowledge, or follow-up checks.

Example interpretation:

```text
Host discovery: no ICMP reply
TCP SYN to 443: SYN/ACK received
Conclusion: host is up; ICMP is filtered
```

**Command examples for common scenarios**

Local subnet discovery:

```bash
nmap -sn 192.168.1.0/24
```

Discovery when ICMP is blocked:

```bash
nmap -sn -PS22,80,443 203.0.113.0/24
```

Full TCP scan on a confirmed host:

```bash
nmap -sS -p- 192.0.2.10
```

UDP scan of common services:

```bash
nmap -sU --top-ports 20 192.0.2.10
```

Version detection on discovered services:

```bash
nmap -sV -p 22,80,443 192.0.2.10
```

**Useful supporting commands**

Outside Nmap, basic network tools help validate what you see.

```bash
ip addr show
ip route
arp -a
ip neigh show
ping -c 1 192.168.1.10
traceroute 192.0.2.10
```

These commands help you confirm your local interface, routing path, neighbor cache, and basic reachability before deeper scanning.

**Best practices**

- **Start narrow**: Scan a small set of ports first.
- **Use multiple probes**: Do not rely on one discovery method.
- **Confirm local-layer behavior**: Use ARP or neighbor discovery on local networks.
- **Interpret states carefully**: Open, closed, and filtered mean different things.
- **Tune for the environment**: Adjust timing and probe types based on filtering and loss.
- **Document assumptions**: Record which methods worked and which were blocked.

**Common mistakes**

- Treating “no ping reply” as proof that a host is down.
- Running full scans before confirming live targets.
- Ignoring the difference between TCP and UDP behavior.
- Assuming UDP silence means openness.
- Using aggressive scan settings without considering noise or rate limits.
- Forgetting that local subnet discovery may use ARP instead of ICMP.

**Quick reference**

| Concept | Meaning | Typical use |
|---|---|---|
| ICMP | Reachability and error messaging | Basic host discovery |
| ARP | IPv4 address-to-MAC resolution | Local subnet discovery |
| NDP | IPv6 neighbor discovery | IPv6 host discovery |
| SYN scan | Half-open TCP scan | Fast port discovery |
| Connect scan | Full TCP connection scan | Non-privileged scanning |
| UDP scan | Connectionless port scan | DNS, NTP, SNMP checks |
| `-sn` | Discovery only | Find live hosts |
| `-Pn` | Skip discovery | Scan hosts that block probes |

**Exam focus**

For CEH v13, focus on the relationship between discovery and scanning:

- Host discovery identifies live systems before port scanning.
- ICMP is common but often filtered.
- ARP is highly effective on local networks.
- TCP SYN probes are useful when ICMP is blocked.
- UDP scanning is less reliable because silence is ambiguous.
- Nmap options such as `-sn`, `-Pn`, `-PS`, `-PA`, `-PU`, and `-PR` are central to practical workflows.

Understanding these concepts helps you choose the right probe, interpret scan output correctly, and avoid false assumptions during reconnaissance.
