## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

Enumeration is the phase where you actively identify what a service exposes, how it behaves, and what useful metadata it leaks. In practice, it sits between discovery and exploitation: you already know a host is reachable, and now you want service names, versions, users, shares, policies, banners, and protocol-specific clues that help you choose the next step.

For exam purposes, remember that enumeration often overlaps with version detection and scripted probing. The useful distinction is not “scan vs. enumerate” as a hard rule, but whether your action is aimed at collecting actionable service details rather than just confirming open ports.

- **Goal:** extract usable information, not just confirm connectivity.
- **Value:** reduces guesswork and helps you prioritize weak services, default settings, and exposed identities.
- **Risk:** some probes are noisy, state-changing, or logged; choose the least intrusive method that answers the question.

**Common Workflow**

A practical workflow starts broad, then narrows by service:

1. Identify open ports and service names with `nmap`.
2. Run targeted version detection and safe NSE scripts.
3. Query the service with native clients where appropriate.
4. Correlate banners, responses, and authentication behavior.
5. Record usernames, domains, shares, policy hints, and version strings.

```bash
nmap -sV -Pn -p 21,22,23,25,53,80,110,111,135,139,143,389,445,464,512,513,514,636,873,993,995,1433,1521,2049,3306,3389,5432,5900,5985,5986,6379,8080 192.168.1.10
```

- `-sV` performs service/version detection.
- `-Pn` avoids host discovery when ICMP is filtered.
- A focused port list is often better than a full sweep when you already know the likely service set.

**HTTP and HTTPS Enumeration**

Web services often reveal the most information because they expose headers, directories, technologies, and application behavior. Start with headers and content, then move to technology fingerprinting and content discovery.

```bash
curl -I http://192.168.1.10
whatweb http://192.168.1.10
nikto -h http://192.168.1.10
```

- `curl -I` checks response headers, redirects, cookies, and server hints.
- `whatweb` fingerprints frameworks, CMSs, and middleware.
- `nikto` looks for common misconfigurations, dangerous files, and outdated components.

Useful checks include:

- Server and framework headers such as `Server`, `X-Powered-By`, and cookie attributes.
- Directory listing and backup files such as `.bak`, `.old`, `.zip`, or source maps.
- Authentication behavior, including default pages, login error messages, and lockout policy clues.
- Virtual host behavior, where the same IP serves different content by hostname.

```bash
ffuf -u http://192.168.1.10/FUZZ -w /usr/share/wordlists/dirb/common.txt
```

- Content discovery helps you find hidden admin panels, APIs, and documentation.
- If a site uses HTTPS, also inspect certificate subject names and SANs (Subject Alternative Names) for hostnames.

> Be careful not to treat every web fingerprint as authoritative. Reverse proxies, CDNs, and custom headers can mislead you, so confirm findings with multiple requests.

**FTP Enumeration**

FTP (File Transfer Protocol) commonly exposes banners, anonymous access, and writable directories. Even when login is required, the server may reveal version and policy details before authentication.

```bash
ftp 192.168.1.10
nmap -sV --script ftp-anon,ftp-syst,ftp-banner -p 21 192.168.1.10
```

- `ftp-anon` checks whether anonymous login is allowed.
- `ftp-syst` can reveal the server type and system details.
- `ftp-banner` captures banner text that may include product and version clues.

What to look for:

- Anonymous read access to public files.
- Anonymous write access, which is a serious misconfiguration.
- Chroot behavior and directory structure.
- TLS support via FTPS and whether cleartext login is still accepted.

A quick manual test:

```bash
ftp 192.168.1.10
Name: anonymous
Password: test@example.com
ls
pwd
```

If anonymous access works, check whether the server exposes backups, configuration files, or upload directories. Writable FTP directories can become a foothold if they are served by another service.

**SSH Enumeration**

SSH (Secure Shell) usually reveals less than other services, but banners and authentication behavior still matter. Version strings can indicate whether the host is vulnerable to known issues or whether older algorithms are enabled.

```bash
nc -nv 192.168.1.10 22
ssh -vvv user@192.168.1.10
nmap -sV --script ssh2-enum-algos,ssh-hostkey -p 22 192.168.1.10
```

- Banner grabbing can reveal the SSH implementation and version.
- `ssh2-enum-algos` lists supported key exchange, ciphers, and MACs.
- `ssh-hostkey` helps you identify host key types and fingerprints.

Why it matters:

- Weak or legacy algorithms can indicate poor hardening.
- Host key changes may reveal reinstallation, cloning, or man-in-the-middle risk.
- Authentication banners can expose policy text, but they should not be treated as proof of vulnerability.

**Telnet Enumeration**

Telnet is unencrypted, so it often leaks banners, prompts, and sometimes device-specific login behavior. It is common on legacy network devices and embedded systems.

```bash
nc -nv 192.168.1.10 23
telnet 192.168.1.10
nmap -sV --script telnet-encryption,telnet-ntlm-info -p 23 192.168.1.10
```

- Banner text may identify routers, switches, or older Unix-like systems.
- `telnet-encryption` checks whether the service supports encryption negotiation.
- `telnet-ntlm-info` is useful only when Telnet is fronting a Windows authentication path and the script is available in your Nmap build.

Use Telnet findings to infer:

- Device family and firmware generation.
- Whether the service is a management interface rather than a general shell.
- Whether credentials are likely transmitted in cleartext.

**SMTP Enumeration**

SMTP (Simple Mail Transfer Protocol) can reveal valid users, relay policy, and mail server identity. It is one of the most useful services for early reconnaissance because it often responds differently to valid and invalid recipients.

```bash
nc -nv 192.168.1.10 25
nmap -sV --script smtp-commands,smtp-enum-users -p 25,465,587 192.168.1.10
```

- `smtp-commands` shows supported SMTP extensions such as `STARTTLS`, `AUTH`, and `PIPELINING`.
- `smtp-enum-users` may help identify valid accounts, depending on server behavior and script support.

Manual checks:

```text
HELO test.local
MAIL FROM:<test@test.local>
RCPT TO:<admin@target.local>
RCPT TO:<nosuchuser@target.local>
```

Interpretation:

- Different responses to `RCPT TO` can indicate valid users.
- `VRFY` and `EXPN` may be disabled, but if enabled they can expose usernames and mailing lists.
- `STARTTLS` support matters because it changes whether credentials and content are protected in transit.

**POP3 and POP3S Enumeration**

POP3 (Post Office Protocol v3) on ports `110` and `995` often exposes mailbox access behavior, banner strings, and authentication methods. It is less verbose than SMTP, but still useful for identifying mail infrastructure and user validation behavior.

```bash
nc -nv 192.168.1.10 110
openssl s_client -connect 192.168.1.10:995 -crlf
nmap -sV --script pop3-capabilities -p 110,995 192.168.1.10
```

- POP3 banners may reveal the mail server product and version.
- `CAPA` shows supported capabilities such as `SASL`, `STLS`, and `UIDL`.
- POP3S on `995` indicates implicit TLS, which changes how you test and observe the service.

Example session:

```text
USER test
PASS test
STAT
LIST
UIDL
```

Why it matters:

- `STAT` and `LIST` confirm mailbox access and message count.
- `UIDL` can help you understand whether message identifiers are stable.
- Authentication behavior may reveal whether the server accepts common usernames or enforces lockouts.

**IMAP and IMAPS Enumeration**

IMAP (Internet Message Access Protocol) on ports `143` and `993` is often richer than POP3 because it supports folders, flags, and server-side mailbox management. It can reveal more about account structure and server capabilities.

```bash
nc -nv 192.168.1.10 143
openssl s_client -connect 192.168.1.10:993 -crlf
nmap -sV --script imap-capabilities -p 143,993 192.168.1.10
```

- `CAPABILITY` is the key first command for IMAP enumeration.
- IMAPS on `993` uses TLS from the start.
- Some servers expose folder names or namespace behavior before authentication.

Example:

```text
a1 CAPABILITY
a2 LOGIN user pass
a3 LIST "" "*"
a4 SELECT INBOX
```

Interpretation:

- `CAPABILITY` can reveal `STARTTLS`, `AUTH=PLAIN`, `IDLE`, or vendor-specific extensions.
- `LIST` may expose mailbox naming conventions.
- Folder names can hint at business processes, shared mailboxes, or archived data.

**DNS Enumeration**

DNS (Domain Name System) is a high-value target because it maps names to hosts and can expose internal structure. Enumeration focuses on records, zone transfer attempts, and subdomain discovery.

```bash
dig @192.168.1.10 example.local any
dig @192.168.1.10 example.local axfr
nslookup
```

- `AXFR` attempts a zone transfer; if allowed, it can dump many records at once.
- `ANY` queries are not always honored, but they can still reveal useful responses.
- `nslookup` is useful for quick interactive checks.

Common record types to query:

- `A` and `AAAA` for host addresses.
- `MX` for mail exchangers.
- `NS` for authoritative name servers.
- `TXT` for SPF, verification tokens, and service metadata.
- `SRV` for service location records, especially in enterprise environments.

```bash
dig @192.168.1.10 _ldap._tcp.example.local SRV
dig @192.168.1.10 _kerberos._tcp.example.local SRV
```

These SRV records are especially useful because they can reveal LDAP and Kerberos endpoints in Active Directory environments.

**NFS and RPC Enumeration**

NFS (Network File System) and RPC (Remote Procedure Call) services often expose shared storage and service mappings. Enumeration here is about identifying exported paths, mount permissions, and the RPC services listening behind `rpcbind`.

```bash
rpcinfo -p 192.168.1.10
showmount -e 192.168.1.10
nmap -sV --script nfs-showmount,rpcinfo -p 111,2049 192.168.1.10
```

- `rpcinfo -p` lists registered RPC programs and dynamic ports.
- `showmount -e` lists exported NFS shares if the server allows it.
- `rpcbind` on port `111` maps RPC program numbers to ports.

Why this matters:

- RPC services may run on dynamic ports, so port `111` is often the starting point.
- NFS exports can expose sensitive data if permissions are weak.
- Misconfigured exports may allow read access, root squashing issues, or writable mounts.

If you can mount an export, verify permissions carefully:

```bash
sudo mount -t nfs 192.168.1.10:/share /mnt/test
ls -la /mnt/test
```

> Do not assume an export is safe just because it mounts. File ownership, `root_squash`, and write permissions determine the real risk.

**SMB and NetBIOS Enumeration**

SMB (Server Message Block) and NetBIOS remain core CEH topics because they often expose hostnames, shares, users, groups, and domain information. Modern SMB enumeration usually uses `smbclient`, `rpcclient`, `nmblookup`, and Nmap scripts.

```bash
nmblookup -A 192.168.1.10
smbclient -L //192.168.1.10 -N
nmap -sV --script smb-os-discovery,smb-enum-shares,smb-enum-users,smb-enum-domains -p 139,445 192.168.1.10
```

- `nmblookup -A` can reveal NetBIOS names and workgroup/domain hints.
- `smbclient -L` lists shares when guest or anonymous access is allowed.
- SMB scripts can identify OS hints, shares, domains, and sometimes users.

Interactive `rpcclient` use is common when you have a valid session or a permissive guest context:

```bash
rpcclient -U "" 192.168.1.10
```

Inside the prompt, you can try:

```text
srvinfo
enumdomusers
enumdomgroups
querydominfo
netshareenumall
```

Interpretation:

- `srvinfo` can reveal server and domain role details.
- `enumdomusers` and `enumdomgroups` may work only when the server permits the query.
- `netshareenumall` helps you identify hidden shares and access patterns.

Practical notes:

- Null sessions are less common on modern systems, but guest or anonymous access may still expose limited information.
- SMB signing, access controls, and patched systems reduce what you can enumerate.
- If you see `IPC$`, that does not automatically mean useful access; it only indicates the inter-process communication share is present.

**LDAP and Active Directory Enumeration**

LDAP (Lightweight Directory Access Protocol) is one of the most important enumeration targets in enterprise networks because it exposes directory objects, group membership, and domain structure. In Active Directory, LDAP often reveals far more than a simple banner.

```bash
ldapsearch -x -H ldap://192.168.1.10 -s base namingcontexts
ldapsearch -x -H ldap://192.168.1.10 -b "DC=example,DC=local" "(objectClass=user)" sAMAccountName memberOf
```

- `-x` uses simple authentication, which is useful for anonymous bind testing.
- `namingcontexts` helps you discover the base DN (Distinguished Name).
- User and group queries can reveal account names, group membership, and organizational structure.

A typical anonymous bind test:

```bash
ldapwhoami -x -H ldap://192.168.1.10
ldapsearch -x -H ldap://192.168.1.10 -b "" -s base "(objectClass=*)" defaultNamingContext
```

If anonymous bind is allowed, you may be able to query:

- Domain naming context.
- Users and groups.
- Computer objects and service accounts.
- Password policy attributes, depending on permissions.

Useful AD-oriented queries:

```bash
ldapsearch -x -H ldap://192.168.1.10 -b "DC=example,DC=local" "(objectClass=group)" cn
ldapsearch -x -H ldap://192.168.1.10 -b "DC=example,DC=local" "(objectClass=computer)" dNSHostName
```

Why it matters:

- LDAP often gives you the domain structure needed for later attacks.
- User and group names help you build targeted password-spraying lists.
- Service account names can hint at applications and privileges.

**Kerberos Enumeration**

Kerberos is central to Windows authentication and is especially valuable in domain environments. Enumeration focuses on realm discovery, user validation, and ticket-related behavior rather than password guessing alone.

```bash
nmap -sV -p 88,464 --script krb5-enum-users 192.168.1.10
kerbrute userenum -d example.local --dc 192.168.1.10 users.txt
```

- Port `88` is Kerberos authentication.
- Port `464` is used for password change and related Kerberos services.
- `krb5-enum-users` and `kerbrute` can help identify valid usernames when the environment permits it.

What to look for:

- Realm names and domain controller hostnames.
- Distinct responses for valid versus invalid users.
- Pre-authentication requirements, which affect whether certain user enumeration techniques work.

Example interpretation:

- If a username exists, the KDC (Key Distribution Center) may return a different error than for a nonexistent user.
- Time synchronization matters because Kerberos is sensitive to clock skew.
- A discovered realm often becomes the anchor for later LDAP, SMB, and password-spraying work.

**RDP Enumeration**

RDP (Remote Desktop Protocol) can reveal whether Network Level Authentication (NLA) is required, what security layers are supported, and whether the host is likely a Windows system. It is not usually a passive enumeration service, so keep the focus on handshake behavior and metadata.

```bash
nmap -sV -p 3389 --script rdp-enum-encryption,rdp-ntlm-info 192.168.1.10
```

- `rdp-enum-encryption` reports supported security modes.
- `rdp-ntlm-info` can expose domain and host information in some environments.

Why it matters:

- NLA requirement changes your attack path because authentication happens earlier.
- Domain and hostname clues help you map the target into an enterprise environment.
- RDP exposure on the internet is a strong indicator to check for weak policy, MFA gaps, or legacy configurations.

**SNMP Enumeration**

SNMP (Simple Network Management Protocol) is often overlooked, but it can reveal device names, interfaces, routing details, and software versions. SNMPv1 and SNMPv2c rely on community strings, while SNMPv3 adds authentication and encryption options.

```bash
snmpwalk -v2c -c public 192.168.1.10
snmpwalk -v3 -l authPriv -u snmpuser -a SHA -A 'AuthPass123' -x AES -X 'PrivPass123' 192.168.1.10
```

- `snmpwalk` walks the Management Information Base (MIB) tree.
- SNMPv1/v2c use community strings such as `public` and `private`.
- SNMPv3 uses a username plus security level, authentication, and privacy settings.

Common OIDs (Object Identifiers) to inspect:

```bash
snmpwalk -v2c -c public 192.168.1.10 1.3.6.1.2.1.1
snmpwalk -v2c -c public 192.168.1.10 1.3.6.1.2.1.2
```

- `1.3.6.1.2.1.1` is system information.
- `1.3.6.1.2.1.2` is interface information.

Why it matters:

- Device names and descriptions can reveal roles, locations, and owners.
- Interface tables can expose IP addressing and network topology.
- SNMPv3 is more secure, but weak credentials still create exposure.

**Database Service Enumeration**

Database services often reveal version, authentication mode, and sometimes user or schema metadata. The exact tooling depends on the database engine, but the goal is the same: identify version, access model, and exposed objects.

**MySQL and MariaDB**

```bash
mysql -h 192.168.1.10 -u root -p
nmap -sV --script mysql-info -p 3306 192.168.1.10
```

- `mysql-info` can reveal server version and protocol details.
- If credentials are weak or reused, you may be able to list databases and users.

Inside the client:

```sql
SHOW DATABASES;
SELECT user, host FROM mysql.user;
```

**PostgreSQL**

```bash
psql -h 192.168.1.10 -U postgres -d postgres
nmap -sV --script pgsql-info -p 5432 192.168.1.10
```

- PostgreSQL often exposes version and authentication method details.
- If you gain access, inspect roles, schemas, and extensions.

**Microsoft SQL Server**

```bash
nmap -sV --script ms-sql-info -p 1433 192.168.1.10
sqlcmd -S 192.168.1.10 -U sa -P 'Password123'
```

- `ms-sql-info` can reveal version and instance details.
- Named instances may also use SQL Browser on UDP `1434`.

**Oracle Database**

```bash
nmap -sV -p 1521 192.168.1.10
tnsping ORCL
```

- Oracle listeners often expose service names and listener behavior.
- `tnsping` checks listener reachability and service resolution when Oracle client tools are installed.

Why database enumeration matters:

- Version strings help you assess patch level and known exposure.
- Authentication behavior reveals whether default accounts or weak policies are in place.
- Schema and role names can expose application structure and privilege boundaries.

**DHCP Enumeration**

DHCP (Dynamic Host Configuration Protocol) is usually not enumerated with a single “safe” client command in the same way as banner-based services. Instead, you observe DHCP behavior, inspect relay information, and identify server responses on the network.

Useful checks include:

```bash
sudo nmap --script broadcast-dhcp-discover
sudo tcpdump -ni eth0 port 67 or port 68
```

- `broadcast-dhcp-discover` sends a discovery broadcast and reports the responding server.
- `tcpdump` lets you observe DHCP offers, options, and relay behavior without changing the network more than necessary.

What to look for:

- DHCP server IP address and lease scope.
- Option 3 (default gateway), option 6 (DNS servers), and option 15 (domain name).
- Option 42 (NTP servers) and vendor-specific options.
- Rogue DHCP behavior if multiple servers respond unexpectedly.

> A lease request is an active network action, so treat DHCP probing as operationally sensitive. Use it only when you are authorized and understand the impact on the segment.

**NTP Enumeration**

NTP (Network Time Protocol) can reveal time sources, stratum, and synchronization state. Modern enumeration should use current query tools rather than legacy-only methods.

```bash
ntpq -p 192.168.1.10
ntpq -c rv 192.168.1.10
nmap -sU -p 123 --script ntp-info 192.168.1.10
```

- `ntpq -p` shows peers and synchronization status when the server allows queries.
- `ntpq -c rv` can return runtime variables such as stratum and reference ID.
- `ntp-info` is a safer modern script choice for basic NTP metadata.

Why it matters:

- Time drift can break Kerberos and other time-sensitive authentication systems.
- Stratum and reference data can reveal whether the host is a client, server, or relay.
- NTP exposure may indicate broader network management services.

**VoIP and SIP Enumeration**

SIP (Session Initiation Protocol) is common in voice environments and can expose extensions, server identity, and supported methods.

```bash
nmap -sU -sV -p 5060,5061 --script sip-methods,sip-enum-users 192.168.1.10
```

- `sip-methods` identifies allowed SIP verbs.
- `sip-enum-users` may help identify extensions if the server responds differently to valid and invalid users.

Useful clues:

- `REGISTER`, `INVITE`, `OPTIONS`, and `SUBSCRIBE` support.
- PBX (Private Branch Exchange) vendor fingerprints.
- Extension numbering patterns that help you build a user list.

**Exam Tips**

- **Prioritize** services that leak identity information first: DNS, LDAP, SMB, SMTP, SNMP, and Kerberos.
- **Use native clients** when you need protocol-specific behavior that scanners may not show.
- **Correlate** multiple weak signals instead of trusting one banner.
- **Document** usernames, domains, hostnames, shares, and version strings as you find them.
- **Respect scope** because some enumeration actions can trigger alerts or change state.

**Common Mistakes**

- Treating a banner as proof of a specific product version without confirmation.
- Assuming anonymous access means full access.
- Using outdated or deprecated commands as if they are current best practice.
- Ignoring TLS, NLA, or authentication requirements that change what you can observe.
- Overlooking mail, directory, and management services because they are not “classic” web targets.

**Practical Mini-Scenario**

You discover a host with ports `53`, `88`, `135`, `139`, `389`, `445`, and `3389` open. That pattern strongly suggests a Windows domain controller or a server close to one. Your next steps are to query DNS for SRV records, test LDAP anonymous bind, enumerate SMB shares and domain hints, and check Kerberos realm behavior. If LDAP reveals the base DN and SMB reveals the domain name, you can build a much more accurate picture of the environment before attempting any authentication-based testing.

```bash
dig @192.168.1.10 _ldap._tcp.example.local SRV
ldapsearch -x -H ldap://192.168.1.10 -s base namingcontexts
smbclient -L //192.168.1.10 -N
nmap -sV --script smb-os-discovery,rdp-ntlm-info,krb5-enum-users -p 88,135,139,389,445,3389 192.168.1.10
```

This kind of layered enumeration is what turns a port list into an attack surface map.
