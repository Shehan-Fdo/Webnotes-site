## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Overview**

Nmap uses multiple detection modes to identify what is running on a host and how it is configured. **Service detection** identifies the application behind an open port, **version detection** refines that identification, and **OS detection** estimates the target operating system from network behavior. In CEH-style workflows, you use these features together to turn a simple port list into actionable reconnaissance.

These features are not equally reliable in every environment. Firewalls, packet filters, proxies, load balancers, and service hardening can hide details or distort fingerprints. You should understand both the commands and the reasons results may be incomplete, ambiguous, or wrong.

**Core concepts**

- **Service detection** maps a port to a likely service name, such as `ssh`, `http`, or `smtp`.
- **Version detection** tries to identify the exact application and release, such as `OpenSSH 8.9p1` or `Apache httpd 2.4.x`.
- **OS detection** compares network fingerprints against Nmap’s OS database to estimate the target platform.
- **Aggressive scan** mode bundles several features together, but it also increases noise and visibility.

**Nmap service and version detection**

Service and version detection are usually performed with `-sV`. Nmap probes open ports with a set of application-layer tests and matches the responses against its service fingerprint database. This is more precise than relying only on port numbers, because many services can run on nonstandard ports.

```bash
nmap -sV 192.168.1.10
```

A typical output includes a `PORT` column, a `STATE` column, and a `SERVICE` column. The `PORT` field shows the port number and protocol, such as `22/tcp` or `53/udp`. The `SERVICE` field is Nmap’s best guess based on the port and response behavior, not a guarantee.

Example output:

```text
PORT     STATE SERVICE VERSION
22/tcp   open  ssh     OpenSSH 8.2p1 Ubuntu 4ubuntu0.5
80/tcp   open  http    Apache httpd 2.4.52
443/tcp  open  ssl/http nginx 1.22.1
```

Interpretation points:

- **PORT** identifies the transport and port number.
- **STATE** shows whether Nmap considers the port open, closed, filtered, or another state.
- **SERVICE** is the protocol or application family Nmap believes is present.
- **VERSION** is the refined product and version string when Nmap can determine it.

Nmap often uses a combination of banner grabbing and active probes. A banner may reveal a product string directly, but many services suppress banners or return misleading ones. Version detection is therefore probabilistic, not absolute.

**How version detection works**

Nmap sends probes that are tailored to common services and compares the responses to a large fingerprint database. It may identify:

- product name
- version number
- operating system family of the service
- protocol details
- device type in some cases

The `--version-intensity` option controls how aggressively Nmap probes for versions. Higher values increase accuracy but also increase scan time and network noise.

```bash
nmap -sV --version-intensity 7 192.168.1.10
```

Useful related options:

- `--version-light` reduces probe intensity for faster, quieter scans.
- `--version-all` uses the maximum probe set.
- `--version-trace` shows how Nmap arrived at a version match, which is useful for troubleshooting.

Example with a custom port list:

```bash
nmap -sV -p 21,22,25,80,443 192.168.1.10
```

This is useful when you already know which services are likely exposed and want to focus version probing on those ports.

**Reading version detection output**

Nmap may show one of several outcomes:

- an exact product and version match
- a generic service name with no version
- a guessed version with a confidence note
- a fingerprint mismatch that requires manual review

Example of a partial match:

```text
8080/tcp open  http    Jetty 9.4.x
```

This means Nmap believes the service is Jetty, but it may not know the exact patch level. Treat the result as a strong clue, not proof.

If Nmap cannot match a service exactly, it may show a fingerprint summary or a note that the service is unrecognized. That usually means the service is uncommon, customized, filtered, or intentionally disguised.

**Practical banner and fingerprint checks**

You can supplement `-sV` with manual checks when you need confirmation.

```bash
nc -nv 192.168.1.10 25
```

```bash
curl -I http://192.168.1.10
```

```bash
openssl s_client -connect 192.168.1.10:443
```

These commands help you inspect banners, headers, and TLS certificates directly. They are especially useful when Nmap returns a generic service label or a weak version guess.

> A banner is not always trustworthy. Some services hide their identity, and some intentionally fake banners to mislead scanners.

**OS detection fundamentals**

OS detection uses network fingerprinting to infer the target operating system. Nmap sends a series of crafted probes and compares the responses to known fingerprints in its database. The result is usually a best match with a confidence score.

```bash
nmap -O 192.168.1.10
```

Typical output may include:

```text
OS details: Linux 5.4 - 5.15
Running: Linux 5.X
OS CPE: cpe:/o:linux:linux_kernel:5
```

Interpretation points:

- **Running** is Nmap’s broader family guess.
- **OS details** is a more specific match when confidence is higher.
- **CPE** stands for Common Platform Enumeration and is a standardized identifier.
- **Accuracy** or confidence values indicate how strong the match is.

Example of an uncertain result:

```text
Aggressive OS guesses: Linux 4.15 - 5.10 (92%), Linux 5.4 (89%), Android 10 (88%)
No exact OS matches for host
```

This means Nmap found several close fingerprints but no exact database match. You should treat the top guess as a likely candidate, not a confirmed answer.

**What Nmap probes for OS detection**

Nmap’s OS fingerprinting is based on differences in how stacks respond to crafted packets. It may examine:

- TCP initial sequence number behavior
- TCP options ordering and values
- TCP window size
- ECN (Explicit Congestion Notification) behavior
- response to unusual flag combinations
- ICMP behavior
- UDP response patterns
- IP ID generation patterns

Nmap’s OS tests are often described as `T1` through `T7` plus additional probes such as `PU` for UDP and `IE` for ICMP-related behavior. These tests help distinguish between operating systems, device types, and network stacks.

You do not need to memorize every test label for CEH, but you should understand that OS detection is fingerprint-based and depends on how the target stack reacts to crafted packets.

**Requirements and privileges**

OS detection and some version-detection behavior often require raw packet access. In practice, that usually means running Nmap with administrative privileges.

```bash
sudo nmap -O 192.168.1.10
```

On Windows, you typically need to run the terminal as Administrator. Without sufficient privileges, Nmap may fall back to less accurate methods or fail to perform certain probes.

**Why OS detection can fail**

OS detection is sensitive to network conditions. Common causes of poor results include:

- packet filtering by firewalls
- stateful inspection devices
- NAT and port forwarding
- load balancers
- host-based firewalls
- spoofed or filtered responses
- insufficient open and closed ports for fingerprinting
- nonstandard or hardened TCP/IP stacks

Nmap often needs a mix of open and closed TCP ports to build a useful fingerprint. If it cannot find enough useful responses, it may report limited confidence, no exact match, or too many possible matches.

Important practical point:

> `--osscan-limit` does not force OS detection. It tells Nmap to skip OS detection unless it already has enough useful port states to make the scan worthwhile.

Example:

```bash
nmap -O --osscan-limit 192.168.1.10
```

This is useful when you want to reduce wasted OS probes on hosts that do not expose enough information.

If you want Nmap to be more willing to guess, you can use:

```bash
nmap -O --osscan-guess 192.168.1.10
```

This can produce a result even when the fingerprint is not ideal, but the output should be treated as a guess.

**Aggressive scan mode**

`-A` enables a bundle of features rather than a single test. It typically includes:

- OS detection with `-O`
- version detection with `-sV`
- default NSE (Nmap Scripting Engine) scripts with `-sC`
- traceroute

```bash
nmap -A 192.168.1.10
```

This is convenient for quick reconnaissance because it combines several useful checks in one command. However, it is also louder and more likely to trigger logging or defenses.

Use `-A` when:

- you want a broad first-pass assessment
- you are in a lab or authorized assessment
- you can tolerate more network noise

Use separate flags when:

- you want finer control over scan behavior
- you want to reduce noise
- you only need one type of detection

**Comparing `-sV`, `-O`, and `-A`**

| Option | Purpose | Strengths | Tradeoffs |
|---|---|---|---|
| `-sV` | Service/version detection | Identifies applications and versions | Can be noisy and may misidentify hidden services |
| `-O` | OS detection | Estimates target operating system | Needs useful fingerprints and can be blocked |
| `-A` | Aggressive scan bundle | Fast way to combine multiple checks | More visible and less selective |

A practical workflow is to start with a port scan, then add version detection, and finally run OS detection if the host exposes enough information.

```bash
nmap -sS -Pn -p- 192.168.1.10
nmap -sV -p 22,80,443 192.168.1.10
nmap -O 192.168.1.10
```

**Interpreting common output fields**

Nmap output is more useful when you understand the meaning of each field.

- **STATE**
  - `open` means an application appears to accept connections.
  - `closed` means the port is reachable but no service is listening.
  - `filtered` means a firewall or filter is blocking the probe or response.
- **SERVICE**
  - This is a protocol or application guess.
  - It may be based on port number, probe response, or both.
- **VERSION**
  - This is a refined identification string.
  - It may include product, version, and platform hints.
- **OS details**
  - This is the best OS match Nmap can infer from fingerprints.
  - It may be broad or specific depending on confidence.

**Service identification beyond the port number**

Port numbers are only a starting point. A port can run a nonstandard service, and the same service can appear on many ports. For example:

- SSH may run on `22/tcp`, `2222/tcp`, or another custom port.
- HTTP may run on `80/tcp`, `8080/tcp`, or `8443/tcp`.
- SMTP may run on `25/tcp`, `587/tcp`, or `2525/tcp`.

That is why version detection matters. It helps you identify the actual application even when the port number is misleading.

Example:

```bash
nmap -sV -p 2222,8080,8443 192.168.1.10
```

This is especially useful in hardened environments where administrators move services away from default ports.

**Practical examples**

1. Scan a host for service versions:
   ```bash
   nmap -sV 192.168.1.10
   ```

2. Check OS fingerprinting with privilege:
   ```bash
   sudo nmap -O 192.168.1.10
   ```

3. Combine service and OS detection:
   ```bash
   sudo nmap -sV -O 192.168.1.10
   ```

4. Run a broad reconnaissance scan:
   ```bash
   sudo nmap -A 192.168.1.10
   ```

5. Focus on likely service ports:
   ```bash
   nmap -sV -p 21,22,25,53,80,110,143,443 192.168.1.10
   ```

**Troubleshooting and reliability tips**

- Use `-Pn` when host discovery is blocked, but remember that it may increase scan time.
- Use `-vv` for more verbose output when you need to see scan progress and reasoning.
- Use `--reason` to see why Nmap marked a port as open, closed, or filtered.
- Use `--packet-trace` only when debugging, because it is very noisy.
- Compare Nmap results with manual checks before making a final conclusion.

Example:

```bash
sudo nmap -sV -O --reason -vv 192.168.1.10
```

If a firewall interferes, you may see filtered ports, incomplete version strings, or no exact OS match. In those cases, the scan result still has value because it tells you something about the target’s exposure and defensive posture.

**Exam-focused takeaways**

- `-sV` is for identifying services and versions.
- `-O` is for OS fingerprinting.
- `-A` combines multiple reconnaissance features, including `-sC`.
- OS detection is fingerprint-based and can be blocked or distorted.
- Version detection is helpful but not perfect, especially on hardened or custom services.
- Privileged execution improves the quality of many Nmap probes.
- Output interpretation matters as much as the command itself.

**Quick reference**

```bash
nmap -sV <target>                 # Detect service versions
sudo nmap -O <target>             # Detect OS using fingerprints
sudo nmap -A <target>             # Aggressive scan: -O, -sV, -sC, traceroute
nmap -sV --version-light <target> # Faster, less thorough version detection
sudo nmap -O --osscan-limit <target> # Limit OS detection to hosts with useful port states
sudo nmap -O --osscan-guess <target> # Allow Nmap to guess more aggressively
```

**Common mistakes to avoid**

- Assuming the `SERVICE` column is always correct.
- Treating a guessed version as a confirmed version.
- Expecting OS detection to work equally well behind firewalls.
- Using `-A` without understanding that it is a bundle of multiple features.
- Forgetting that administrative privileges improve raw-packet-based detection.
- Overlooking the difference between `filtered` and `closed` ports.

**Summary**

Service, version, and OS detection are core reconnaissance techniques in Nmap. They help you move from simple port enumeration to a more complete picture of the target. For CEH, focus on the command options, the meaning of the output fields, the limitations of fingerprinting, and the practical tradeoffs between speed, accuracy, and stealth.
