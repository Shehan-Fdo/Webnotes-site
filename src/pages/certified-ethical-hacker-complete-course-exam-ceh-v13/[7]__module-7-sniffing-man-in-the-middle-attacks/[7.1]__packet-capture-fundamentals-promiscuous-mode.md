**Packet capture fundamentals**

Packet capture is the process of observing and recording network traffic as frames or packets traverse an interface. You use it to troubleshoot connectivity, validate protocol behavior, detect anomalies, and analyze attacks. In ethical hacking, packet capture helps you understand what data is visible on the wire, how hosts communicate, and where controls such as switching, encryption, and segmentation limit visibility.

At a practical level, packet capture tools sit at the boundary between the network stack and the interface driver. They can capture traffic before the operating system fully processes it, which is why they are useful for low-level analysis. Common tools include `tcpdump`, Wireshark, `tshark`, and `dumpcap`.

**What a packet capture contains**

A capture file usually stores link-layer frames and metadata such as timestamps, interface information, and captured length. Depending on the capture point, you may see:

- **Ethernet headers** with source and destination MAC addresses
- **IP headers** with source and destination IP addresses
- **Transport headers** such as TCP, UDP, or ICMP
- **Application payloads** when traffic is not encrypted
- **Timing data** that helps identify retransmissions, latency, and session behavior

A capture is only as useful as the vantage point where you collect it. On a switched network, a host normally sees only its own traffic plus broadcast, multicast, and traffic explicitly mirrored or forwarded to it.

**Why packet capture matters in security work**

Packet capture supports both defensive and offensive analysis. You can use it to confirm whether a service is reachable, whether DNS queries resolve as expected, whether TLS handshakes succeed, and whether suspicious traffic patterns exist. It also helps you verify whether a host is leaking credentials or metadata in cleartext.

Typical use cases include:

- Troubleshooting intermittent application failures
- Identifying unauthorized protocols on a segment
- Confirming whether a firewall or ACL is blocking traffic
- Inspecting DNS, DHCP, ARP, and routing behavior
- Detecting scanning, beaconing, or lateral movement indicators

**Promiscuous mode**

Promiscuous mode is a network interface setting that tells the NIC (Network Interface Card) to pass all received frames up to the operating system, not just frames addressed to that interface. This is essential for packet sniffing on a shared medium or when the interface is attached to a mirrored port.

Without promiscuous mode, the NIC typically filters frames based on its own MAC address, broadcast, and multicast membership. With promiscuous mode enabled, the host can inspect traffic that would otherwise be discarded by hardware filtering.

**Promiscuous mode vs. normal mode**

| Mode | Behavior | Typical use |
|---|---|---|
| Normal | Receives frames addressed to the host, plus broadcast/multicast as allowed | Everyday networking |
| Promiscuous | Receives all frames visible on the interface | Packet analysis, sniffing, troubleshooting |

Promiscuous mode does not magically bypass switching. On a modern switched Ethernet network, you still need a valid capture point such as a SPAN/mirror port, network tap, or a host positioned on the same collision domain in legacy environments.

**How switching affects visibility**

Switches forward unicast frames only to the port where the destination MAC address resides. That means a sniffer on a random access port usually cannot see other hosts’ unicast traffic. You can increase visibility by using:

- **SPAN** port mirroring on a switch
- **Network taps** for passive inline observation
- **Hub-based** legacy segments, where all traffic is shared
- **Host-based capture** on the endpoint itself
- **Virtual switch mirroring** in cloud or hypervisor environments

> Promiscuous mode is necessary for sniffing, but it is not sufficient on its own if the network device never forwards the traffic to your interface.

**Common packet capture tools**

`tcpdump` is the standard command-line capture tool on Unix-like systems. It is lightweight, scriptable, and ideal for remote servers. Wireshark provides a graphical interface and deep protocol decoding. `tshark` offers Wireshark-style decoding from the command line. `dumpcap` is often used as a capture engine behind Wireshark.

Examples:

```bash
# Capture traffic on eth0 and print packets to the terminal
tcpdump -i eth0

# Capture only 100 packets and save to a file
tcpdump -i eth0 -c 100 -w capture.pcap

# Read a capture file
tcpdump -r capture.pcap
```

```bash
# Capture DNS traffic only
tcpdump -i eth0 port 53

# Capture traffic to or from a specific host
tcpdump -i eth0 host 192.0.2.10

# Capture HTTP and HTTPS traffic
tcpdump -i eth0 port 80 or port 443
```

**Capture filters and display filters**

Capture filters reduce what is written to disk or processed during capture. They are applied before packets are stored. In `tcpdump`, capture filters use Berkeley Packet Filter (BPF) syntax. Wireshark also supports capture filters, but it additionally provides display filters, which are applied after capture.

Use capture filters when you want to limit file size or focus on a specific conversation. Use display filters when you want to keep a broad capture and analyze it later.

Examples:

```bash
# BPF capture filter: only TCP traffic to port 22
tcpdump -i eth0 tcp port 22
```

In Wireshark, a display filter might look like:

```text
ip.addr == 192.0.2.10 && tcp.port == 22
```

**Practical capture workflow**

1. Identify the correct interface with `ip link` or `ifconfig`.
2. Decide whether you need promiscuous mode and whether the traffic is actually visible on that interface.
3. Apply a capture filter to reduce noise.
4. Save the capture to a `.pcap` or `.pcapng` file.
5. Analyze the file in Wireshark or with command-line tools.
6. Correlate packet timing and protocol behavior with logs and alerts.

Example interface discovery:

```bash
ip link show
ip addr show
```

Example capture with file output:

```bash
sudo tcpdump -i eth0 -nn -s 0 -w /tmp/session.pcap
```

- `-nn` prevents name resolution, which keeps output faster and avoids DNS noise.
- `-s 0` requests a full packet capture instead of a truncated snapshot.
- `-w` writes raw packets to a file for later analysis.

**Promiscuous mode in practice**

Most capture tools enable promiscuous mode by default when possible. You can often disable it if you only want traffic addressed to the host. In Wireshark, the capture options typically include a promiscuous mode checkbox. In `tcpdump`, promiscuous mode is usually on unless you specify `-p`.

Examples:

```bash
# Disable promiscuous mode
tcpdump -i eth0 -p
```

On Linux, you can inspect interface flags and capture behavior with:

```bash
ip link show eth0
```

If you need to verify whether a NIC is in promiscuous mode, you may see the `PROMISC` flag in interface output or use packet capture tool status indicators. Exact behavior can vary by driver and virtualization platform.

**Important limitations**

Packet capture has technical and operational limits:

- **Encryption** hides payload content even when you capture the packets.
- **Switched networks** restrict what you can see without mirroring or tapping.
- **Offloading features** such as checksum offload or segmentation offload can make captures look unusual.
- **High-speed links** may drop packets if the capture host cannot keep up.
- **Privileges** are required on most systems to capture traffic.

> A capture showing no packets does not always mean no traffic exists; it may mean the interface, filter, permissions, or mirroring setup is wrong.

**Offloading and capture artifacts**

Modern NICs and kernels use offloading to improve performance. These features can affect how packets appear in a capture:

- **Checksum offload** may show incorrect checksums in the capture because the NIC computes them later.
- **TCP segmentation offload (TSO)** can make large segments appear differently than they are on the wire.
- **Generic receive offload (GRO)** can coalesce packets before capture in some environments.

When analyzing suspicious traffic, consider disabling offloads temporarily on a lab system if you need wire-accurate visibility.

Example:

```bash
sudo ethtool -K eth0 gro off gso off tso off
```

**Linux capture permissions**

On Linux, packet capture usually requires root or capabilities such as `CAP_NET_RAW` and `CAP_NET_ADMIN`. A common approach is to grant capabilities to the capture binary rather than running everything as root.

Example:

```bash
sudo setcap cap_net_raw,cap_net_admin=eip $(which tcpdump)
```

This allows `tcpdump` to capture without full root privileges, depending on system policy.

**Useful command-line examples**

```bash
# Show live traffic with resolved names disabled
sudo tcpdump -i eth0 -nn

# Capture only ARP traffic
sudo tcpdump -i eth0 arp

# Capture ICMP traffic
sudo tcpdump -i eth0 icmp

# Capture traffic to a subnet
sudo tcpdump -i eth0 net 192.0.2.0/24

# Capture and rotate files with a size limit
sudo tcpdump -i eth0 -w /var/log/captures/net.pcap -C 100
```

**Interpreting common protocols**

Packet capture is especially useful for understanding foundational protocols:

- **ARP (Address Resolution Protocol)** maps IP addresses to MAC addresses on a local network.
- **DNS (Domain Name System)** resolves names to IP addresses and can reveal query patterns.
- **DHCP (Dynamic Host Configuration Protocol)** assigns addresses and network settings.
- **TCP (Transmission Control Protocol)** shows handshakes, retransmissions, resets, and session teardown.
- **ICMP (Internet Control Message Protocol)** helps diagnose reachability and path issues.

A basic TCP handshake appears as `SYN`, `SYN/ACK`, and `ACK`. If you see repeated `SYN` packets without a response, the destination may be unreachable, filtered, or overloaded.

**Security and ethical considerations**

Packet capture can expose credentials, session tokens, and sensitive business data. You should only capture traffic on systems and networks where you have explicit authorization. In professional environments, captures may be subject to policy, logging, retention, and privacy requirements.

Best practices include:

- Limit capture scope to the minimum necessary traffic
- Protect `.pcap` files as sensitive evidence
- Document the capture point, time, and filter used
- Avoid capturing unrelated user data when possible
- Use encryption-aware analysis when payloads are protected

**Exam-focused takeaways**

For CEH v13, focus on the relationship between capture visibility and network topology. Promiscuous mode lets the NIC accept more frames, but switching, mirroring, and encryption determine what you can actually observe. Know the difference between capture filters and display filters, and be able to identify common tools and commands used for packet analysis.

**Quick reference**

- **Packet capture** records network traffic for analysis.
- **Promiscuous mode** allows a NIC to accept all visible frames.
- **Switches** limit unicast visibility unless you use mirroring or taps.
- **`tcpdump`** is the core CLI capture tool on many systems.
- **Wireshark** is the primary GUI analysis tool.
- **Filters** reduce noise and help target specific traffic.
- **Encryption** protects payloads even when packets are captured.
