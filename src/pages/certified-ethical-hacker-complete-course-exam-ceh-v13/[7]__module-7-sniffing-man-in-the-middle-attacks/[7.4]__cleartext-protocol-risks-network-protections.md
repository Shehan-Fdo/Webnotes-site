## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Cleartext protocol fundamentals**

Cleartext protocols transmit data in a form that can be read by anyone who can observe the traffic path. That includes usernames, passwords, session tokens, file contents, and management commands. In CEH contexts, the key risk is not only disclosure but also the ease of **replay**, **session hijacking**, and **man-in-the-middle** interception.

Common attack conditions include:

- Shared networks where an attacker can sniff traffic.
- Compromised switches, routers, or wireless access points.
- Misconfigured services that allow legacy authentication.
- Flat networks with no segmentation or egress filtering.

A useful exam distinction is that a protocol can be “insecure by default” yet still support encryption as an option. For example, LDAP on port `389` can be upgraded with StartTLS, and SMTP on port `587` is typically used for message submission with STARTTLS. The risk comes from how the service is configured and whether encryption is required, not just from the port number alone.

**Why cleartext protocols are dangerous**

Cleartext traffic is easy to capture and analyze with packet tools. An attacker does not need to break encryption if none is present. In practice, the attacker often combines passive capture with active network manipulation.

Typical attack workflow:

1. Gain a position on the same broadcast domain, wireless network, or routed path.
2. Put the interface into a capture-friendly state, such as promiscuous mode on wired networks or monitor mode on wireless.
3. Use traffic capture tools to inspect packets and extract credentials or sensitive data.
4. If needed, perform **ARP poisoning** on a LAN or rogue access point attacks on Wi-Fi to redirect traffic through the attacker.

Common tools and techniques include:

- `tcpdump` for command-line packet capture.
- Wireshark for protocol inspection and credential hunting.
- `ettercap` or `bettercap` for MITM-style interception on local networks.
- Wireless monitor mode for capturing unencrypted or weakly protected Wi-Fi traffic.

Example capture commands:

```bash
# Capture traffic on a Linux interface
sudo tcpdump -i eth0 -nn -s0 -w capture.pcap

# Filter for common cleartext services
sudo tcpdump -i eth0 -nn port 21 or port 23 or port 80 or port 110 or port 143 or port 389 or port 445 or port 161
```

> Packet capture is most effective when the attacker can observe the traffic path. Switched networks reduce casual sniffing, but they do not eliminate risk if the attacker can poison ARP, compromise a switch, mirror a port, or join the same wireless segment.

**High-risk protocols and secure alternatives**

Memorize the insecure protocol and its preferred secure replacement. Some protocols have multiple secure variants, so the exact choice depends on the use case.

| Insecure protocol | Common port(s) | Risk profile | Preferred secure alternative |
|---|---:|---|---|
| FTP | `21` control, `20` data | Credentials and file contents can be exposed | SFTP over SSH or FTPS |
| Telnet | `23` | Entire session is readable, including logins | SSH |
| HTTP | `80` | Web content and cookies may be exposed | HTTPS |
| POP3 | `110` | Mail retrieval credentials and messages may be exposed | POP3S or IMAPS |
| IMAP | `143` | Mail access can be observed | IMAPS |
| SMTP | `25`, `587` | Submission and relay can expose credentials if not protected | SMTP with STARTTLS or SMTPS where appropriate |
| LDAP | `389` | Directory queries and binds may be exposed unless StartTLS is used | LDAPS or LDAP with StartTLS |
| SNMP | `161/udp`, traps `162/udp` | Weak community strings and cleartext management data in v1/v2c | SNMPv3 |
| SMB/CIFS | `445/tcp` | Legacy SMB can expose shares, auth material, and metadata; signing/encryption may be absent | SMB signing, SMB encryption, or VPN/segmentation |

**FTP, FTPS, and SFTP**

FTP is a classic exam example because it separates control and data channels and historically sends credentials in cleartext. Passive and active modes change how data connections are established, but they do not make FTP secure.

- **FTP** uses `21/tcp` for control and `20/tcp` in active mode for data.
- **FTPS** is FTP over TLS. It can be explicit or implicit, depending on how the server is configured.
- **SFTP** is not FTP over TLS; it is the SSH File Transfer Protocol and runs over SSH, usually on `22/tcp`.

Practical examples:

```bash
# Plain FTP login is visible to anyone capturing traffic
ftp 192.0.2.10

# Secure file transfer over SSH
sftp user@192.0.2.10

# FTPS with curl
curl --ftp-ssl -u user:password ftp://192.0.2.10/
```

**Telnet versus SSH**

Telnet is one of the clearest examples of a cleartext remote administration protocol. It transmits everything in readable form, including usernames, passwords, and shell commands.

- **Telnet** uses `23/tcp`.
- **SSH** uses `22/tcp` and provides encrypted remote access, tunneling, and file transfer capabilities.

Example:

```bash
# Insecure remote shell
telnet 192.0.2.20

# Secure remote shell
ssh admin@192.0.2.20
```

**HTTP versus HTTPS**

HTTP traffic can reveal URLs, form data, cookies, and API tokens. Even if the page content seems harmless, session cookies can be enough for account takeover.

- **HTTP** uses `80/tcp`.
- **HTTPS** uses `443/tcp` and protects traffic with TLS (Transport Layer Security).

Important exam point:

- HTTPS protects data in transit, but it does not fix weak application logic, stolen certificates, or compromised endpoints.
- Mixed content, weak TLS versions, and bad certificate validation can still create risk.

**Email protocols: POP3, IMAP, and SMTP**

Email protocols are frequent exam targets because they often carry credentials and message content.

- **POP3** on `110/tcp` is commonly used for mail retrieval.
- **IMAP** on `143/tcp` is used for mailbox synchronization.
- **SMTP** on `25/tcp` is used for server-to-server relay, while `587/tcp` is the standard message submission port for clients and commonly uses STARTTLS.

Secure variants:

- **POP3S** on `995/tcp`
- **IMAPS** on `993/tcp`
- **SMTP with STARTTLS** on `587/tcp`
- **SMTPS** on `465/tcp` in some environments

Example client checks:

```bash
# Test STARTTLS on SMTP submission
openssl s_client -starttls smtp -connect mail.example.com:587

# Test IMAPS
openssl s_client -connect mail.example.com:993
```

**LDAP and directory services**

LDAP is often used for authentication, directory lookups, and application integration. It is especially important in enterprise environments because directory traffic can reveal usernames, group membership, and bind credentials.

- **LDAP** commonly uses `389/tcp` and can be upgraded with StartTLS.
- **LDAPS** uses `636/tcp` and wraps LDAP in TLS from the start.

Do not assume port `389` is always cleartext. The safer exam framing is:

- `389/tcp` may be cleartext if StartTLS is not used.
- `389/tcp` can be protected if StartTLS is negotiated successfully.
- `636/tcp` is the traditional LDAPS port.

Example query:

```bash
ldapsearch -H ldap://ldap.example.com:389 -ZZ -x -D "cn=admin,dc=example,dc=com" -W
```

The `-ZZ` option requests StartTLS and fails if TLS cannot be established, which is useful when you want to avoid silent downgrade to cleartext.

**SNMP weaknesses**

SNMP (Simple Network Management Protocol) is a common source of weak authentication and information leakage. SNMPv1 and SNMPv2c use community strings instead of strong user-based authentication, and those community strings are often left at defaults such as `public` and `private`.

- **SNMPv1/v2c** commonly use `161/udp` for queries and `162/udp` for traps.
- Default or weak community strings are frequently abused for reconnaissance.
- SNMP can reveal device names, interfaces, routing data, software versions, and sometimes configuration details.

Example enumeration:

```bash
# Query with a community string
snmpwalk -v2c -c public 192.0.2.30 system

# Check for common defaults
snmpwalk -v2c -c private 192.0.2.30
```

SNMPv3 is preferred because it supports authentication and privacy. In exam terms, remember that v3 is the secure version, while v1/v2c are commonly weak because of shared community strings.

**SMB/CIFS exposure**

SMB/CIFS (Server Message Block/Common Internet File System) is often overlooked in cleartext discussions, but it is important for CEH. SMB is used for file sharing, printer access, and Windows network services. Modern SMB usually runs on `445/tcp`, while older NetBIOS-related services may also involve `137/udp`, `138/udp`, and `139/tcp`.

Risk scenarios include:

- Anonymous or guest shares exposing sensitive files.
- Weak or reused credentials captured during authentication attempts.
- Legacy SMB configurations without signing or encryption.
- Lateral movement after share enumeration.

Important nuance:

- SMB authentication is not the same as “all data is cleartext.”
- However, weak SMB configurations can still expose metadata, enable relay attacks, or allow interception when signing/encryption is absent.

Secure controls:

- Enable SMB signing where required.
- Use SMB encryption when supported.
- Disable guest access and anonymous enumeration.
- Restrict SMB to trusted subnets only.

**Network-layer protections**

You reduce cleartext risk most effectively by combining protocol hardening with network controls. Do not rely on application settings alone.

Common protections:

- **Firewall rules** to block legacy ports at the perimeter and between VLANs.
- **ACLs** (Access Control Lists) on routers and switches to restrict who can reach management services.
- **Segmentation** to isolate user, server, and management networks.
- **VPNs** for remote access to internal services.
- **Port security** and 802.1X to reduce unauthorized access on wired networks.
- **Wireless encryption** such as WPA2/WPA3 to prevent casual sniffing on Wi-Fi.

Example firewall rules on Linux with `ufw`:

```bash
# Deny legacy cleartext services from untrusted networks
sudo ufw deny 21/tcp
sudo ufw deny 23/tcp
sudo ufw deny 80/tcp
sudo ufw deny 110/tcp
sudo ufw deny 143/tcp
sudo ufw deny 389/tcp
sudo ufw deny 445/tcp
sudo ufw deny 161/udp

# Allow secure alternatives only
sudo ufw allow 22/tcp
sudo ufw allow 443/tcp
sudo ufw allow 587/tcp
sudo ufw allow 636/tcp
```

Example `nftables` policy:

```bash
nft add rule inet filter input tcp dport { 21, 23, 80, 110, 143, 389, 445 } drop
nft add rule inet filter input udp dport { 161, 162 } drop
```

Example Cisco-style ACL concept:

```text
ip access-list extended BLOCK_CLEARTEXT
 deny tcp any any eq 21
 deny tcp any any eq 23
 deny tcp any any eq 80
 deny tcp any any eq 110
 deny tcp any any eq 143
 deny tcp any any eq 389
 deny tcp any any eq 445
 deny udp any any eq 161
 permit ip any any
```

**Hardening services correctly**

When you secure a service, prefer settings that actually enforce encryption rather than merely offering it.

Postfix example:

```conf
# /etc/postfix/main.cf
smtpd_tls_security_level = encrypt
smtpd_tls_auth_only = yes
smtpd_tls_cert_file = /etc/ssl/certs/mail.pem
smtpd_tls_key_file = /etc/ssl/private/mail.key
```

- `encrypt` requires TLS for the session.
- `smtpd_tls_auth_only = yes` prevents authentication before TLS.
- Use `may` only when opportunistic encryption is acceptable.

OpenSSH example:

```conf
# /etc/ssh/sshd_config
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
```

This does not make SSH “more encrypted” in a cryptographic sense, but it reduces credential exposure and brute-force risk by removing password logins.

LDAP StartTLS example:

```bash
ldapsearch -H ldap://directory.example.com:389 -ZZ -x -b "dc=example,dc=com"
```

**Detection and verification**

You should verify whether a service is actually protected rather than assuming it is. Port numbers alone do not prove encryption.

Useful checks:

- `nmap` service detection and script output.
- `openssl s_client` for TLS handshakes.
- `ss -tulpen` or `netstat` to identify listening services.
- Packet capture to confirm whether credentials are visible.

Examples:

```bash
# Identify services and versions
nmap -sV -Pn 192.0.2.0/24

# Check for TLS on a mail submission port
openssl s_client -connect mail.example.com:587 -starttls smtp

# Show listening sockets
ss -tulpen
```

If you can read usernames, passwords, cookies, or commands in a capture, the protocol or configuration is not adequately protected.

**Exam memory cues**

- **FTP**: file transfer, cleartext by default, use SFTP or FTPS.
- **Telnet**: remote shell, cleartext, use SSH.
- **HTTP**: web traffic, cleartext, use HTTPS.
- **POP3/IMAP/SMTP**: email protocols, use TLS-enabled variants.
- **LDAP**: directory access, use StartTLS or LDAPS.
- **SNMPv1/v2c**: weak community strings, use SNMPv3.
- **SMB/CIFS**: file sharing, restrict exposure and enable signing/encryption.

A fast comparison for memorization:

| Goal | Bad choice | Better choice |
|---|---|---|
| Remote admin | Telnet | SSH |
| File transfer | FTP | SFTP or FTPS |
| Web access | HTTP | HTTPS |
| Mail retrieval | POP3/IMAP | POP3S/IMAPS |
| Mail submission | SMTP without TLS | SMTP with STARTTLS |
| Directory access | LDAP without TLS | StartTLS or LDAPS |
| Network management | SNMPv1/v2c | SNMPv3 |
| Windows file sharing | Exposed SMB | Restricted SMB with signing/encryption |

**Practical defensive checklist**

1. Inventory services and map them to ports.
2. Identify any protocol that transmits credentials or data without encryption.
3. Replace legacy services with secure alternatives where possible.
4. Enforce TLS or SSH rather than merely offering it.
5. Block unnecessary ports at firewalls and ACLs.
6. Segment management and server networks.
7. Verify with packet capture that sensitive data is no longer visible.

**Key takeaway**

Cleartext protocol risk is about exposure, not just protocol names. If an attacker can observe the traffic path, they can often steal credentials, replay sessions, or harvest sensitive data. The strongest defense is layered: use secure protocols, require encryption, and block legacy services at the network edge and between internal segments.
