## **Course:** Certified Ethical Hacker Complete Course (Exam CEH v13)

**Wireshark and Ettercap in CEH workflows**

Wireshark is a packet capture and protocol analysis tool. Ettercap is a man-in-the-middle (MITM) framework commonly used in labs to demonstrate traffic interception, ARP poisoning, and traffic redirection on switched networks. In CEH-style questions, the key is to understand what each tool does, what it does not do, and how they complement each other during reconnaissance and validation.

- **Wireshark** is for observing, decoding, and investigating traffic.
- **Ettercap** is for positioning yourself in the traffic path in a controlled lab.
- Together, they help you prove whether traffic is plaintext, encrypted, redirected, or altered.

> Use these tools only in authorized environments. MITM activity can disrupt connectivity if you do not configure forwarding and targets correctly.

**Wireshark fundamentals**

Wireshark captures packets from a network interface and decodes them into protocol layers. It is useful for identifying services, troubleshooting connectivity, and extracting evidence from unencrypted traffic. The interface is usually organized into three panes:

- **Packet list pane** shows one row per frame with summary fields such as source, destination, protocol, length, and info.
- **Packet details pane** expands protocol layers and fields for the selected packet.
- **Packet bytes pane** shows the raw frame in hexadecimal and ASCII, which is useful for confirming exact payload content.

The bytes pane matters in exam scenarios because it lets you verify whether a string is actually present in the payload, whether a header field is encoded as expected, or whether a protocol dissector is interpreting the frame correctly.

**Capture filters vs display filters**

Wireshark uses two different filter types, and CEH questions often test the distinction.

- **Capture filters** use Berkeley Packet Filter (BPF) syntax and limit what gets recorded.
- **Display filters** are applied after capture and only change what you see.

Capture filters are useful when you want to reduce file size or avoid collecting irrelevant traffic. Display filters are better when you want to keep all evidence and analyze it later.

Common capture filter examples:

```bash
src host 192.168.1.10
dst port 80
net 192.168.1.0/24
tcp and port 443
```

Common display filter examples:

```text
ip.addr == 192.168.1.10
tcp.port == 80
http
dns
tls
```

Practical distinctions:

- **Capture filters** are less expressive but more efficient at capture time.
- **Display filters** are richer and can match protocol fields, flags, and values after decoding.
- **BPF** syntax is common in capture filters, while Wireshark display filters use protocol-aware field names.

**Useful Wireshark analysis features**

Wireshark includes several features that are frequently useful in labs and exam questions.

- **Colorization** highlights traffic patterns visually, such as TCP retransmissions, DNS, or suspicious resets.
- **Packet marking** lets you flag important frames for later review.
- **Endpoints statistics** show which IPs, MAC addresses, or ports are most active.
- **Conversations statistics** summarize bidirectional flows between hosts.
- **Protocol hierarchy** shows how much traffic belongs to each protocol.
- **Follow Stream** reconstructs a single TCP or UDP conversation for easier reading.
- **Expert Information** surfaces warnings, retransmissions, malformed packets, and protocol anomalies.

A practical workflow is to start with broad statistics, then narrow to a host or protocol, and finally inspect the stream or packet bytes for evidence.

**TCP stream analysis and evidence extraction**

For unencrypted traffic, Wireshark can reveal credentials, session tokens, commands, and file content if they traverse the network in plaintext. This is one of the most important exam concepts: if the application does not encrypt the session, packet capture may expose sensitive data.

Typical workflow:

1. Capture traffic on the correct interface.
2. Filter to the target host or service.
3. Open the relevant TCP stream.
4. Inspect the reconstructed conversation for usernames, passwords, cookies, or commands.
5. Confirm the evidence in the packet bytes pane when needed.

Example display filters:

```text
ip.addr == 192.168.1.50 && tcp.port == 80
http.request
tcp.stream eq 3
```

Use **Follow TCP Stream** when you want to read the conversation in order. It is especially useful for HTTP, Telnet, FTP control traffic, and other plaintext protocols. It is less useful for encrypted TLS traffic because the payload is not readable without decryption keys and appropriate setup.

> Wireshark can reassemble some protocol data and fragmented IP packets, but it does not universally reconstruct arbitrary higher-level conversations from all fragmented content.

**HTTP vs HTTPS in troubleshooting**

A common CEH-style task is to distinguish plaintext web traffic from encrypted web traffic.

- **HTTP** usually appears on port 80 and can expose request lines, headers, cookies, and form data in readable form.
- **HTTPS** usually appears on port 443 and uses TLS (Transport Layer Security), so payload content is encrypted after the handshake.

What to look for:

- **HTTP**: `GET`, `POST`, `Host`, `User-Agent`, cookies, form fields, and readable URLs.
- **HTTPS**: TLS handshake messages such as Client Hello, Server Hello, certificates, and encrypted application data.

Example filters:

```text
http
tcp.port == 80
tls
tcp.port == 443
```

If you are validating whether a login is protected, compare the capture before and after encryption. In plaintext HTTP, credentials may appear directly in the request body or headers. In HTTPS, you should see the handshake and encrypted records instead of readable credentials.

**Wireshark statistics and investigation workflow**

Statistics views help you move from raw packets to meaningful patterns.

- **Endpoints** identifies which hosts are communicating most.
- **Conversations** shows pairs of hosts and the volume of traffic between them.
- **Protocol hierarchy** helps you identify dominant protocols in a capture.
- **IO graphs** can show bursts, spikes, or periodic behavior.
- **Flow graphs** can help visualize request/response sequences.

A practical investigation sequence is:

1. Open the capture.
2. Check **Endpoints** to identify active hosts.
3. Check **Conversations** to find the busiest flows.
4. Apply a display filter to isolate the suspected protocol.
5. Use **Follow Stream** or the packet bytes pane to extract evidence.

**Ettercap fundamentals**

Ettercap is a MITM tool used to intercept traffic between hosts on a LAN. It is commonly associated with ARP poisoning in switched Ethernet environments. In lab settings, it can help you demonstrate how a host can be tricked into sending traffic through the attacker when the attacker poisons ARP caches and forwards packets correctly.

Core concepts:

- **ARP poisoning** manipulates ARP cache mappings so a victim associates a peer IP with the attacker’s MAC address.
- **MITM** means the attacker sits between two endpoints and relays traffic.
- **IP forwarding** is required so traffic continues to pass through the attacker instead of being dropped.
- **Targets** are the hosts or groups Ettercap is configured to intercept.

> `arp:remote` is used when poisoning traffic across routed paths or different subnets by influencing the gateway path, not simply for generic “remote hosts on the LAN.”

**Ettercap operating modes**

Ettercap supports different operational approaches, and CEH questions may ask you to distinguish them.

- **Unified sniffing** captures traffic on a single interface in a switched LAN where the attacker is already on the same segment.
- **Bridged mode** uses two interfaces and forwards traffic between them, which can be useful when you need the attacker to act like a transparent bridge.
- **ARP poisoning mode** is the common MITM approach on local Ethernet segments.
- **Remote ARP poisoning** can be used in routed scenarios where the attacker influences traffic toward a gateway path.

Practical implications:

- **Unified sniffing** is simpler when you only need one interface and the network path can be poisoned.
- **Bridge mode** is useful when you want the attacker to transparently pass traffic between two interfaces.
- **IP forwarding** must be enabled in most MITM setups to preserve connectivity.

Enable forwarding on Linux:

```bash
sudo sysctl -w net.ipv4.ip_forward=1
```

Persist it if needed:

```bash
echo 'net.ipv4.ip_forward=1' | sudo tee -a /etc/sysctl.conf
sudo sysctl -p
```

**ARP poisoning mechanics**

ARP resolves IP addresses to MAC addresses on a local network. In ARP poisoning, the attacker sends forged ARP replies so each victim updates its cache with an incorrect mapping. The result is that traffic intended for one host is sent to the attacker’s MAC address first.

Typical lab topology:

- Victim A: `192.168.1.10`
- Victim B or gateway: `192.168.1.1`
- Attacker: `192.168.1.50`

The attacker poisons both sides so:

- Victim A maps `192.168.1.1` to the attacker MAC.
- The gateway maps `192.168.1.10` to the attacker MAC.

This creates a relay path through the attacker, assuming forwarding is enabled.

**Ettercap CLI examples**

Ettercap can be used from the command line for repeatable lab work. The exact syntax varies by version, but the general workflow is consistent: choose an interface, define targets, enable poisoning, and optionally load plugins.

Example of starting text mode with an interface:

```bash
sudo ettercap -T -i eth0
```

That command starts Ettercap in text mode on `eth0`, but by itself it does not complete a useful MITM attack. You still need target selection and a poisoning method.

A more practical example:

```bash
sudo ettercap -T -i eth0 -M arp:remote /192.168.1.10/ /192.168.1.1/
```

This tells Ettercap to use ARP poisoning in a routed/remote context between the specified targets. In many labs, you will instead poison a victim and the gateway on the same subnet.

Another common pattern is to specify targets and then start sniffing:

```bash
sudo ettercap -T -i eth0 -M arp /192.168.1.10/ /192.168.1.1/
```

Useful supporting commands:

```bash
ip addr show
ip route
arp -a
```

These help you confirm interface names, routing, and ARP cache state before and after the attack.

**Ettercap plugins and practical use cases**

Ettercap plugins extend functionality, but they are not a universal traffic manipulation engine. Their capabilities depend on the plugin, protocol, and whether the traffic is encrypted.

Common lab-oriented use cases include:

- **DNS spoofing** to redirect name resolution for selected domains.
- **Credential capture** from plaintext protocols when traffic passes through the attacker.
- **Session observation** for unencrypted web or legacy application traffic.
- **Protocol-specific manipulation** where a plugin supports the target protocol and the traffic is not protected by encryption.

Limitations:

- **Encrypted traffic** such as modern HTTPS cannot usually be meaningfully altered without breaking the session.
- **Plugin support** is protocol-specific and version-dependent.
- **Traffic manipulation** is not arbitrary; it is constrained by the plugin and the protocol state.

Example of loading a plugin in a lab context:

```bash
sudo ettercap -T -i eth0 -P dns_spoof -M arp /192.168.1.10/ /192.168.1.1/
```

Use this only where you are authorized to test. DNS spoofing can disrupt name resolution and may be blocked by DNS security controls, browser protections, or encrypted DNS mechanisms.

**Wireshark and Ettercap together in a lab**

A common workflow is to use Ettercap to position yourself in the path and Wireshark to verify what is actually flowing through the link.

Step-by-step lab example:

1. Confirm the interface and IP addressing on the attacker machine.
2. Enable IP forwarding so traffic can pass through the attacker.
3. Start Ettercap with the correct interface and targets.
4. Open Wireshark on the same interface or a mirrored capture point.
5. Verify that traffic from the victim now traverses the attacker.
6. Inspect packets for ARP changes, TCP sessions, and plaintext data if present.

What to look for in Wireshark during a successful MITM:

- **ARP replies** showing the attacker MAC associated with the victim or gateway IP.
- **Repeated ARP traffic** if poisoning is active.
- **TCP sessions** between victim and external hosts passing through the attacker interface.
- **HTTP requests** or other plaintext protocols if the target uses them.
- **DNS queries** and responses if name resolution is being intercepted.

Useful filters during verification:

```text
arp
ip.addr == 192.168.1.10
ip.addr == 192.168.1.1
http
dns
tcp.stream eq 0
```

If the attack is working, you should see the victim’s traffic continue normally while the attacker observes the packets. If connectivity breaks, check forwarding, target selection, interface choice, and whether the network uses protections such as dynamic ARP inspection.

**Installation and platform considerations**

Lab success often depends on permissions and adapter support.

- **Elevated privileges** are usually required for packet capture and raw network operations.
- **Interface permissions** may block capture if your user account is not allowed to access the adapter.
- **Driver support** matters for wireless captures, especially if you need monitor mode.
- **Switched networks** may require poisoning or a mirrored port because you cannot see all traffic by default.
- **Virtual machines** may need bridged networking or promiscuous mode enabled on the virtual adapter.

Common Linux checks:

```bash
sudo wireshark
sudo tshark -D
sudo tcpdump -i eth0
```

On wireless adapters, monitor mode support depends on the chipset and driver. Not every adapter can capture 802.11 frames properly, and not every environment allows monitor mode.

**Practical troubleshooting workflow**

When a lab does not behave as expected, isolate the problem systematically.

1. Verify the interface name and IP configuration.
2. Confirm that the attacker can reach both targets.
3. Check that IP forwarding is enabled.
4. Confirm ARP cache changes on the victim and gateway.
5. Use Wireshark to verify that packets are traversing the attacker.
6. Check whether encryption prevents useful payload inspection.
7. Review whether the network has protections such as static ARP entries, port security, or dynamic ARP inspection.

Common symptoms:

- **No traffic visible**: wrong interface, capture permissions, or incorrect target selection.
- **Traffic visible but no connectivity**: forwarding disabled or poisoning incomplete.
- **Traffic visible but unreadable**: protocol is encrypted, such as HTTPS or SSH.
- **Poisoning fails**: network defenses or incompatible topology.

**CEH exam focus points**

- **Wireshark** is for capture, filtering, decoding, and evidence extraction.
- **Ettercap** is for MITM positioning, especially ARP poisoning in lab scenarios.
- **Capture filters** reduce what is recorded; **display filters** reduce what is shown.
- **Follow Stream** is useful for reading plaintext conversations.
- **Endpoints** and **Conversations** help identify active hosts and flows.
- **ARP poisoning** changes cache mappings, not actual IP ownership.
- **IP forwarding** is required to keep traffic flowing through the attacker.
- **Encrypted traffic** limits what you can read or modify.
- **Plugins** are limited and protocol-dependent.

**Quick reference commands**

```bash
# Check interfaces
ip addr show

# Check routing
ip route

# Enable forwarding
sudo sysctl -w net.ipv4.ip_forward=1

# Start Wireshark capture from CLI
sudo tshark -i eth0

# Filter for HTTP traffic in Wireshark
http

# Right-click packet -> Follow -> TCP Stream
```
