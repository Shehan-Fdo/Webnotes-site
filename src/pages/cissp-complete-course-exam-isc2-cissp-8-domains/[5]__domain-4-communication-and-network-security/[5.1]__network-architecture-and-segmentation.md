## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Network architecture and segmentation overview**

Network architecture defines how systems, users, and services connect, while segmentation limits how far traffic can move once it enters the environment. In CISSP terms, segmentation is a core control for reducing attack surface, containing breaches, and enforcing trust boundaries.

A well-designed architecture separates assets by sensitivity, function, and exposure. You use physical controls when you need hard separation, and logical controls when you need flexibility, scale, or lower cost. In practice, most enterprises combine both.

- **Physical segmentation** uses separate switches, cabling, firewalls, or even separate facilities.
- **Logical segmentation** uses VLANs, subnets, ACLs, VRFs, security groups, or policy-based routing.
- **Segmentation boundary** is the point where traffic must be inspected, filtered, or explicitly allowed.

**Why segmentation matters**

Segmentation supports confidentiality, integrity, and availability by limiting lateral movement and reducing the blast radius of compromise. If one workstation is infected, a flat network lets the attacker scan and pivot freely. If the environment is segmented, the attacker must cross policy enforcement points that can log, block, or alert.

Segmentation also helps with compliance and operational separation. For example, payment systems, development systems, guest Wi-Fi, and OT (operational technology) networks usually require different trust levels and different access rules.

- Reduce lateral movement after compromise.
- Enforce least privilege between network zones.
- Support regulatory separation of sensitive workloads.
- Improve performance and troubleshooting by grouping similar traffic.

**Logical vs physical segmentation**

Physical segmentation is preferred when you need strong isolation, regulatory separation, or protection against misconfiguration in shared infrastructure. Examples include separate firewalls for a high-security enclave, isolated management networks, or air-gapped systems.

Logical segmentation is preferred when you need agility, cost efficiency, and easier scaling. Examples include VLANs for departments, VRFs (Virtual Routing and Forwarding) for overlapping address spaces, and cloud security groups for workload-level policy.

| Approach | Strengths | Limitations | Typical use |
|---|---|---|---|
| Physical | Strong isolation, simpler trust model | Higher cost, less flexible | High-security zones, OT, sensitive enclaves |
| Logical | Flexible, scalable, easier to automate | Misconfiguration risk, shared infrastructure | Enterprise LANs, cloud, multi-tenant environments |

> A logical design is only as strong as the devices that enforce it. If traffic can bypass the control plane, segmentation is weakened.

**IP addressing, subnetting, and CIDR**

Subnetting and CIDR (Classless Inter-Domain Routing) are foundational because IP addressing plans define segmentation boundaries. A subnet is not just an address range; it is a policy domain that often maps to a VLAN, security zone, or routing boundary.

You use CIDR notation to allocate address space efficiently and to summarize routes. For example, `10.10.0.0/16` can be divided into smaller subnets such as `10.10.10.0/24` for users and `10.10.20.0/24` for servers. Smaller subnets make it easier to apply different controls to different groups.

- **Subnet mask** determines which bits identify the network and which identify the host.
- **CIDR prefix** expresses the same idea in compact form, such as `/24` or `/20`.
- **Address planning** should align with trust zones, not just organizational charts.
- **Route summarization** reduces routing table size and can simplify policy design.

Example subnet plan:

```text
10.20.0.0/16
├─ 10.20.10.0/24  Users
├─ 10.20.20.0/24  Servers
├─ 10.20.30.0/24  Voice
├─ 10.20.40.0/24  Guest Wi-Fi
└─ 10.20.50.0/24  Management
```

This layout makes it easier to apply different ACLs, firewall rules, and monitoring to each zone. It also helps you avoid mixing high-risk and low-risk systems in the same broadcast domain.

**VLANs, subnets, and broadcast domains**

A VLAN (Virtual LAN) creates a logical Layer 2 segment. A subnet is a Layer 3 addressing construct. In many designs, one VLAN maps to one subnet, but they are not the same thing.

- **VLANs** isolate broadcast traffic at Layer 2.
- **Subnets** isolate IP addressing and routing at Layer 3.
- **Broadcast domains** should be kept small to reduce noise and limit exposure.

A common enterprise pattern is one VLAN per security zone. For example, user devices in VLAN 10 use `10.10.10.0/24`, servers in VLAN 20 use `10.10.20.0/24`, and guest devices in VLAN 40 use `10.10.40.0/24`.

Example Cisco-style VLAN configuration:

```bash
vlan 10
 name USERS
vlan 20
 name SERVERS
vlan 40
 name GUEST
```

Example switch port assignment:

```bash
interface GigabitEthernet1/0/10
 switchport mode access
 switchport access vlan 10
```

**Routing controls and segmentation enforcement**

Routing determines whether traffic can move between subnets. If you allow inter-VLAN routing, you are explicitly permitting communication between otherwise separate Layer 2 segments. That can be useful, but it also creates a path that must be controlled.

- **Inter-VLAN routing** allows traffic between VLANs through a router or Layer 3 switch.
- **Route filtering** blocks or limits which routes are advertised or accepted.
- **ACLs** (Access Control Lists) enforce which sources, destinations, and ports can cross boundaries.
- **L3 switches** can enforce segmentation at wire speed, but they can also bypass weak designs if too much trust is placed in routing alone.

A Layer 3 switch often performs routing between VLANs internally. That is efficient, but it means segmentation depends on policy at the switch, not just on VLAN separation. If you do not apply ACLs or firewall policy, hosts in different VLANs may still communicate freely.

Example inter-VLAN routing on a router-on-a-stick design:

```bash
interface GigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 10.10.10.1 255.255.255.0

interface GigabitEthernet0/0.20
 encapsulation dot1Q 20
 ip address 10.10.20.1 255.255.255.0
```

Example ACL to restrict user access to servers:

```bash
ip access-list extended USERS_TO_SERVERS
 permit tcp 10.10.10.0 0.0.0.255 10.10.20.0 0.0.0.255 eq 443
 permit tcp 10.10.10.0 0.0.0.255 10.10.20.0 0.0.0.255 eq 22
 deny ip any 10.10.20.0 0.0.0.255
 permit ip any any
```

Route filtering is important in larger networks and multi-site environments. You use it to prevent unwanted route propagation, reduce accidental reachability, and keep sensitive networks from being advertised broadly.

**DMZ design patterns**

A DMZ (demilitarized zone) is a buffer network that hosts public-facing services while separating them from internal systems. The goal is to expose only the minimum necessary services to untrusted networks.

Common DMZ variants include:

- **Screened subnet**: A subnet protected by one or more firewalls, usually placed between the Internet and internal network.
- **Dual-homed host**: A host with two network interfaces, one facing the Internet and one facing the internal network; it acts as a controlled gateway.
- **Back-to-back firewall**: Two firewalls with a DMZ between them, often used to create layered inspection and separate administrative domains.

Placement matters. In a screened subnet, the DMZ sits between an external firewall and an internal firewall, or behind a single firewall with separate interfaces and strict rules. In a back-to-back design, the first firewall filters inbound Internet traffic into the DMZ, and the second firewall restricts DMZ-to-internal access.

Typical DMZ services include:

- Web reverse proxies
- Mail relays
- DNS authoritative servers
- VPN concentrators
- Bastion hosts

> Do not place databases, domain controllers, or file shares directly in the DMZ unless the design explicitly requires it and the risk is accepted.

Example traffic flow in a back-to-back firewall design:

```text
Internet -> Firewall A -> DMZ -> Firewall B -> Internal Network
```

A secure DMZ rule set usually allows only specific inbound ports to public services and tightly limits return traffic and DMZ-to-internal connections.

**Zero trust architecture**

Zero trust architecture assumes no implicit trust based on network location. Traditional perimeter-based designs trust internal traffic more than external traffic. Zero trust removes that assumption and requires continuous verification of identity, device posture, context, and policy.

Key ideas:

- **Never trust, always verify** is the operating principle.
- Access decisions are based on identity, device health, risk, and least privilege.
- Network location alone does not grant trust.
- Microsegmentation often supports zero trust by limiting east-west movement.

Zero trust differs from classic perimeter security in that the internal network is not treated as safe by default. A user on the corporate LAN may still need MFA (multifactor authentication), device compliance checks, and application-specific authorization.

Practical examples include:

- Requiring SSO (single sign-on) plus MFA before accessing a SaaS app.
- Allowing a managed laptop to reach only a specific application, not the whole subnet.
- Using software-defined policy to isolate workloads even when they share the same physical host.

**Microsegmentation and east-west traffic**

Microsegmentation applies fine-grained policy between workloads, often at the VM, container, or application level. It is especially useful for limiting east-west traffic, which is traffic moving laterally inside the data center or cloud environment.

- **North-south traffic** enters or leaves the environment.
- **East-west traffic** moves between internal systems.

Microsegmentation is often implemented with host firewalls, distributed firewalls, hypervisor controls, or cloud-native policy engines. It is more granular than VLAN-based segmentation and is useful when many workloads share the same subnet.

Example host firewall concept:

```bash
ufw default deny incoming
ufw default allow outgoing
ufw allow from 10.10.20.0/24 to any port 443 proto tcp
ufw enable
```

**Cloud segmentation controls**

Cloud environments use different segmentation primitives than traditional networks, but the goal is the same: restrict reachability and enforce policy boundaries.

Common controls include:

- **Security groups**: Stateful instance- or workload-level allow rules.
- **Network ACLs (NACLs)**: Stateless subnet-level rules that filter traffic in and out of subnets.
- **Route tables**: Determine where traffic goes, including to NAT gateways, firewalls, or transit hubs.
- **Transit gateway / hub-and-spoke**: Centralized routing and policy enforcement between VPCs or VNets.
- **Private endpoints**: Keep service access on private IP space instead of the public Internet.

Security groups are typically stateful, so return traffic is allowed automatically for permitted sessions. NACLs are stateless, so you must allow both directions explicitly. That distinction matters when designing layered controls.

Example AWS-style security group concept:

```text
Inbound:
- TCP 443 from 10.0.0.0/8
Outbound:
- TCP 443 to 10.20.0.0/16
```

Example NACL concept:

```text
100 allow tcp 443 from 10.0.0.0/8
110 allow tcp 1024-65535 to 10.0.0.0/8
120 deny all
```

In a hub-and-spoke design, the hub often contains shared services such as inspection firewalls, DNS, logging, and egress control. The transit gateway or equivalent routing fabric can enforce which spokes may talk to each other, but policy must be explicit. Without route controls and inspection, spokes may become overly connected.

**Wireless segmentation and guest isolation**

Wireless networks need segmentation because radio coverage extends beyond physical walls. Guest access should never share the same trust level as corporate devices.

Common controls include:

- Separate SSIDs for corporate, guest, and IoT devices.
- VLAN assignment by SSID or user identity.
- Captive portals for guest onboarding.
- **Client isolation** or **AP isolation** to prevent wireless clients from talking directly to each other.
- Firewall rules that restrict guest traffic to Internet-only access.

Client isolation is especially useful in guest networks, conference rooms, and public hotspots. It blocks peer-to-peer traffic between clients on the same access point, reducing the risk of local attacks and unauthorized sharing.

Example wireless design:

```text
SSID: CorpWiFi  -> VLAN 10 -> internal access with 802.1X
SSID: GuestWiFi -> VLAN 40 -> Internet only + client isolation
SSID: IoT      -> VLAN 50 -> limited access to broker/services
```

**Containers and Kubernetes segmentation**

Modern workload environments need segmentation at the cluster, namespace, pod, and service level. Containers often share hosts, so traditional subnet boundaries alone are not enough.

In Kubernetes, segmentation commonly uses:

- **Namespaces** to separate logical groups of workloads.
- **NetworkPolicies** to control pod-to-pod traffic.
- **Ingress controllers** to manage north-south access.
- **Service meshes** for mutual TLS (mTLS) and service identity.
- **Node isolation** for sensitive workloads.

A namespace is not a security boundary by itself, but it helps organize policy. NetworkPolicies are more important because they define which pods can talk to which other pods.

Example Kubernetes NetworkPolicy:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-to-api
  namespace: app
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: web
      ports:
        - protocol: TCP
          port: 8080
```

This policy allows only pods labeled `app=web` to reach API pods on port `8080`. Without a default-deny posture, Kubernetes networking can be much flatter than it appears.

**Segmentation by trust zone**

A practical architecture groups systems into zones based on trust and function. Common zones include:

- **Internet edge** for public exposure.
- **DMZ** for externally reachable services.
- **User zone** for endpoints and workstations.
- **Server zone** for internal applications and databases.
- **Management zone** for admin tools, jump hosts, and monitoring.
- **Restricted zone** for highly sensitive systems.

Each zone should have a clear purpose, explicit ingress and egress rules, and monitoring. The more sensitive the zone, the fewer paths should exist into it.

**Access control mechanisms**

Segmentation is enforced by multiple control types working together:

- **Firewalls** inspect and filter traffic based on policy.
- **ACLs** provide packet filtering on routers and switches.
- **Proxy servers** mediate application access.
- **NAC** (Network Access Control) checks device posture before granting access.
- **Jump hosts** centralize administrative access.
- **VPNs** extend secure access, but should still land users in the correct zone.

A VPN does not eliminate segmentation needs. It only creates a secure tunnel. Once traffic enters the internal network, the same zone controls should still apply.

**Design principles for CISSP**

When you evaluate a segmentation design, ask whether it supports least privilege, containment, and clear trust boundaries. Good designs are simple enough to operate and audit, but strict enough to prevent unnecessary access.

- Place public services in a DMZ, not on the internal LAN.
- Use subnetting and VLANs to align addressing with trust zones.
- Enforce inter-zone traffic with firewalls or ACLs, not just routing.
- Use route filtering to prevent accidental reachability.
- Apply cloud-native controls such as security groups and NACLs together.
- Use client isolation for guest wireless where peer-to-peer access is not needed.
- Use microsegmentation for workloads that need fine-grained east-west control.
- Prefer zero trust assumptions for modern hybrid and remote access models.

**Common exam distinctions**

- **Segmentation vs isolation**: Segmentation divides a network into controlled zones; isolation prevents communication entirely or nearly entirely.
- **VLAN vs subnet**: VLAN is Layer 2; subnet is Layer 3.
- **Firewall vs router**: A router forwards traffic between networks; a firewall enforces policy on that traffic.
- **Security group vs NACL**: Security groups are stateful and workload-focused; NACLs are stateless and subnet-focused.
- **DMZ vs internal zone**: DMZ hosts exposed services; internal zones host trusted business systems.

**Practical design example**

A mid-size enterprise might use this layout:

```text
Internet
  |
Perimeter Firewall
  |
DMZ (reverse proxy, mail relay, VPN)
  |
Internal Firewall / L3 policy boundary
  |
+-------------------------------+
| User VLANs | Server VLANs     |
| Guest VLAN | Mgmt VLAN        |
+-------------------------------+
```

Policy examples:

- Guest VLAN can reach only DNS, DHCP, and Internet.
- User VLAN can reach approved application ports on server VLANs.
- Management VLAN can reach admin ports on infrastructure devices.
- DMZ can reach internal systems only on narrowly defined backend ports.
- Logging and monitoring systems can receive events from all zones.

**Troubleshooting and validation**

You validate segmentation by testing both allowed and denied paths. If everything works, the design may still be too permissive.

Useful checks include:

```bash
ip route
ip addr show
traceroute 10.10.20.15
ping 10.10.20.15
curl -I https://app.internal.example
```

On Linux, you can inspect firewall rules with:

```bash
sudo nft list ruleset
sudo iptables -S
sudo ufw status verbose
```

On Cisco devices, you may inspect VLANs and interfaces with:

```bash
show vlan brief
show ip interface brief
show access-lists
show ip route
```

Validation should confirm:

- Unapproved traffic is blocked.
- Approved traffic uses the intended path.
- Logs show denied attempts.
- Route tables do not expose sensitive networks broadly.
- Cloud policies match the intended trust model.

**Key takeaways**

Segmentation is not just a network design preference; it is a security control that limits exposure and supports least privilege. The strongest designs combine subnetting, VLANs, routing controls, firewalls, cloud policy, and workload-level enforcement. For CISSP, focus on how each control creates or protects a boundary, and how modern designs like zero trust and microsegmentation move beyond the traditional perimeter model.
