## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Network monitoring goals and security value**

Network monitoring gives you visibility into how systems communicate, how traffic changes over time, and where security controls are failing or being bypassed. In CISSP terms, it supports detection, response, forensics, capacity planning, and policy enforcement.

You use monitoring to answer practical questions:

- What is normal for this network?
- What changed, when, and by whom?
- Is the traffic pattern consistent with business use or with attack activity?
- Can you preserve evidence well enough for investigation or legal review?

Monitoring is strongest when you combine multiple views:

- **Flow data** for who talked to whom, when, and how much.
- **Packet data** for payload and protocol detail.
- **Logs** for identity, authentication, authorization, and control-plane events.
- **Telemetry** from cloud, endpoint, and application layers.

**Core monitoring architecture**

A mature monitoring program usually includes sensors, collectors, analysis platforms, and retention controls. The exact mix depends on network size, segmentation, cloud use, and regulatory requirements.

Common components include:

- **NMS** (Network Management System) platforms for device health, availability, and performance.
- **NetOps** suites for operational visibility, automation, and change correlation.
- **SIEM** (Security Information and Event Management) platforms for log correlation and alerting.
- **NDR** (Network Detection and Response) tools for behavioral detection on network telemetry.
- **Packet brokers** to aggregate, filter, and distribute traffic copies to multiple tools.
- **TAPs** and **SPAN/mirror ports** to copy traffic for analysis.

A typical flow looks like this:

1. Network devices generate telemetry and logs.
2. TAPs, SPAN ports, or exporters provide traffic visibility.
3. Collectors normalize and enrich the data.
4. Analytics platforms correlate events across sources.
5. Analysts investigate alerts and preserve evidence.

**Monitoring platforms and where they fit**

Different tools answer different questions. You rarely rely on only one platform.

| Tool / platform | Primary use | Strengths | Limitations |
|---|---|---|---|
| NMS | Availability and performance | Uptime, interface errors, device health | Limited security context |
| NetFlow/IPFIX collector | Traffic flow analysis | Low overhead, scalable, trend analysis | No payload visibility |
| SIEM | Log correlation and alerting | Cross-source correlation, compliance reporting | Depends on log quality and tuning |
| NDR / network analytics | Behavioral detection | Anomaly detection, east-west visibility | Can generate false positives |
| IDS/IPS console | Signature and policy alerts | Known threat detection, inline blocking | Needs tuning and maintenance |
| Packet analysis tools | Deep inspection | Full protocol and payload detail | Storage-heavy, privacy-sensitive |

Common enterprise platforms include vendor NMS and NetOps suites, SIEMs, and cloud-native monitoring services. The exam usually cares more about the function than the brand, but you should recognize that these tools often overlap.

**Traffic collection methods**

You need to understand how traffic is copied and what each method is best for.

- **TAP** (Test Access Point)
  - Hardware device that copies traffic passively.
  - Usually preferred for high-fidelity packet capture.
  - Does not introduce the same risk of dropped packets as overloaded SPAN ports.
- **SPAN** (Switched Port Analyzer), also called mirror port
  - Switch feature that copies traffic from one or more ports/VLANs to a monitoring port.
  - Easier to deploy than a TAP.
  - Can drop packets under load or miss traffic if the switch is congested.
- **Packet broker**
  - Aggregates traffic from multiple TAPs/SPANs.
  - Filters, deduplicates, load-balances, and forwards traffic to tools.
  - Useful when several tools need the same traffic feed.

Practical guidance:

- Use a **TAP** when you need reliable packet capture for forensics or high-throughput links.
- Use **SPAN** when you need quick, low-cost visibility and can tolerate some loss.
- Use a **packet broker** when you need to distribute traffic to IDS, packet capture, and performance tools without overloading a single sensor.

> A SPAN port is convenient, but it is not a perfect copy under all conditions. For evidence-grade capture, prefer a TAP or a brokered architecture when possible.

**Flow data and exporter configuration**

Flow records summarize conversations without storing full packets. They are ideal for trend analysis, lateral movement detection, and identifying unusual destinations.

Common flow technologies:

- **NetFlow** from Cisco environments
- **sFlow** for sampled traffic visibility
- **IPFIX** (Internet Protocol Flow Information Export), a standards-based flow format

A basic NetFlow/IPFIX exporter conceptually sends flow records from a router or switch to a collector.

Example Cisco-style configuration:

```bash
conf t
ip flow-export destination 10.10.10.50 2055
ip flow-export version 9
interface GigabitEthernet0/1
 ip flow ingress
 ip flow egress
end
write memory
```

Example IPFIX-style exporter on a generic platform:

```bash
flow exporter SEC-COLLECTOR
 destination 10.10.10.50
 transport udp 4739
 export-protocol ipfix
 template data timeout 60

flow monitor CORP-MONITOR
 exporter SEC-COLLECTOR
 cache timeout active 60
 record netflow-original

interface TenGigabitEthernet1/0/1
 ip flow monitor CORP-MONITOR input
 ip flow monitor CORP-MONITOR output
```

What to look for in flow data:

- Unusual destination countries or ASNs
- Large outbound transfers at odd hours
- Repeated short connections to many hosts
- Long-lived low-volume sessions that may indicate beaconing
- East-west traffic spikes inside a segment

**Baseline establishment and normal traffic profiling**

A baseline is a documented model of normal behavior. It is the reference point that lets you detect anomalies without treating every spike as an incident.

A strong baseline includes:

- Average and peak bandwidth by link, host, and application
- Typical source and destination pairs
- Common ports and protocols
- Normal authentication and DNS patterns
- Business-hour and off-hour behavior
- Seasonal changes such as month-end processing, holidays, or quarterly reporting

You should build baselines over enough time to capture both **diurnal** patterns and **seasonal** patterns.

- **Diurnal** patterns are daily cycles, such as office-hour peaks and overnight quiet periods.
- **Seasonal** patterns are longer cycles, such as weekly backups, end-of-month batch jobs, or annual enrollment periods.

Baseline methodology:

1. Collect telemetry over a representative period.
2. Separate normal business traffic from test, maintenance, and incident traffic.
3. Group by time of day, day of week, and business cycle.
4. Identify recurring peaks and known exceptions.
5. Document thresholds and expected variance.
6. Revisit the baseline after major changes, mergers, migrations, or new applications.

Why baselines matter:

- They reduce false positives.
- They help you spot low-and-slow attacks.
- They support capacity planning and change validation.
- They provide context for incident timelines.

A useful baseline is not a single number. It is a range with context.

**Anomaly detection and traffic analysis artifacts**

Traffic analysis often focuses on patterns that do not match the baseline. The exam expects you to recognize common indicators.

Examples include:

- **DNS tunneling**
  - Large volumes of DNS queries with encoded data in subdomains or TXT records.
  - Often used for command and control or data exfiltration.
- **Port scanning**
  - Many connection attempts across sequential or common ports.
  - May appear as SYN probes, connection resets, or short-lived failures.
- **Beaconing**
  - Repeated periodic connections to the same host or domain.
  - Often shows consistent timing, packet size, and destination.
- **Protocol anomalies**
  - Malformed headers, unexpected flags, or nonstandard port usage.
  - Example: HTTP traffic on a port that normally carries another service.
- **Data exfiltration**
  - Large outbound transfers, unusual compression, or encrypted traffic to rare destinations.
- **Lateral movement**
  - New east-west connections between hosts that rarely communicate.

Useful indicators in practice:

- Regular intervals with little jitter may suggest beaconing.
- Many failed connections to many ports may suggest scanning.
- DNS requests with long, random-looking labels may suggest tunneling.
- A service speaking on an unexpected port may indicate masquerading or misconfiguration.

**Packet capture workflow and tools**

Packet capture gives you the most detail, but it is expensive in storage and analysis time. Use it when flow and logs are not enough.

Common tools:

- **tcpdump** for command-line capture and quick triage
- **Wireshark** for interactive packet analysis
- **tshark** for command-line parsing and automation

Example capture with `tcpdump`:

```bash
sudo tcpdump -i eth0 -nn -s 0 -w incident-2026-04-26.pcap host 10.20.30.40 and port 443
```

Example readback and filtering:

```bash
tcpdump -nn -r incident-2026-04-26.pcap
```

Example `tshark` extraction:

```bash
tshark -r incident-2026-04-26.pcap -Y "dns" -T fields -e frame.time -e ip.src -e dns.qry.name
```

Example Wireshark display filters:

- `dns`
- `tcp.flags.syn == 1 and tcp.flags.ack == 0`
- `http.request`
- `tls.handshake.type == 1`

Investigation workflow:

1. Start with flow data to identify the suspicious host, time window, and destination.
2. Pivot to packet capture for protocol and payload detail.
3. Pivot to logs for identity, authentication, and authorization context.
4. Correlate all three views to build a timeline.

That flow-to-packet-to-log pivot is a common exam concept because each layer answers a different question.

**IDS and IPS operational differences**

An **IDS** (Intrusion Detection System) observes traffic and alerts on suspicious activity. An **IPS** (Intrusion Prevention System) sits inline and can block or modify traffic.

Operational differences:

- **IDS**
  - Passive placement.
  - Lower risk to availability.
  - Good for detection and tuning.
- **IPS**
  - Inline placement.
  - Can drop malicious traffic in real time.
  - Introduces availability risk if misconfigured or overloaded.

Deployment tradeoffs:

- Inline IPS can stop attacks faster, but a failure can disrupt legitimate traffic.
- Passive IDS is safer for availability, but it cannot directly block traffic.
- Some IPS devices support **fail-open**, where traffic continues if the device fails.
- Others support **fail-closed**, where traffic stops if the device fails.

Tradeoff summary:

| Mode | Availability impact | Security impact | Typical use |
|---|---|---|---|
| Fail-open | Higher availability | Lower protection during failure | Production links where uptime is critical |
| Fail-closed | Lower availability | Higher containment during failure | High-security environments where blocking is preferred |

Signature tuning workflow:

1. Deploy with conservative rules.
2. Review alerts for false positives and noisy signatures.
3. Whitelist known benign applications, hosts, or update servers.
4. Adjust thresholds, suppressions, and protocol decoders.
5. Re-test after application changes or firmware updates.
6. Keep a change record for auditability.

Practical tuning matters because an IPS that blocks legitimate business traffic becomes a self-inflicted denial of service.

**Logs, SIEM correlation, and dashboarding**

Logs provide identity and control-plane context that packets cannot always show. A SIEM correlates logs from firewalls, servers, endpoints, identity systems, and cloud services.

Useful log sources include:

- Firewall allow/deny logs
- VPN authentication logs
- DNS server logs
- Proxy logs
- Authentication logs from directory services
- Endpoint detection and response telemetry
- Cloud control-plane logs

Example SIEM-style query logic:

```sql
SELECT src_ip, dst_ip, COUNT(*) AS attempts
FROM firewall_logs
WHERE action = 'deny'
  AND event_time >= NOW() - INTERVAL '15 minutes'
GROUP BY src_ip, dst_ip
HAVING COUNT(*) > 50
ORDER BY attempts DESC;
```

Example dashboard panels:

- Top denied source IPs
- Top outbound destinations by bytes
- DNS query volume by host
- New external destinations by subnet
- Authentication failures by user and source

A good dashboard shows trends, not just alerts. It should help you answer whether the event is isolated, recurring, or part of a broader campaign.

**Cloud monitoring and cloud-native logs**

Cloud environments require different visibility sources because much of the infrastructure is abstracted away.

Important cloud-native telemetry includes:

- **VPC Flow Logs** in cloud virtual networks for traffic metadata
- **NSG flow logs** for network security group traffic visibility in Azure-like environments
- **CloudTrail** for AWS control-plane activity
- **Azure Activity Logs** for subscription-level management actions
- Kubernetes audit logs for API server actions
- Container runtime and node logs for workload behavior

Key distinctions:

- Flow logs show network metadata such as source, destination, port, and action.
- Control-plane logs show who changed what, when, and from where.
- Kubernetes audit logs show API requests, authentication, and authorization decisions.
- Application logs show workload behavior but may not reveal network path details.

Cloud monitoring examples:

- A sudden spike in denied VPC flows may indicate scanning or misconfiguration.
- A CloudTrail record for security group changes may explain a new exposure.
- Kubernetes audit logs can show a user creating a privileged pod or modifying a service account.

In cloud and container environments, you often need to correlate:

- Identity events
- Network flow logs
- Resource configuration changes
- Workload logs

**Monitoring segmented and zero-trust environments**

Segmentation changes where you place sensors and what you can see. In zero-trust and microsegmented networks, east-west traffic becomes as important as north-south traffic.

Visibility strategies:

- Place sensors at key choke points between segments.
- Use host-based telemetry where network copies are limited.
- Collect flow logs from virtual switches, firewalls, and cloud networks.
- Mirror traffic from critical enclaves or service meshes when possible.

Microsegmentation implications:

- Traffic may be blocked before it reaches a central sensor.
- East-west movement can be hidden inside a segment if you only monitor the perimeter.
- You may need distributed sensors or host agents to see lateral movement.

Practical approach:

- Monitor ingress and egress at segment boundaries.
- Add internal sensors for high-value zones.
- Use identity-aware logs to connect traffic to users and workloads.
- Validate that segmentation policies do not blind your detection stack.

**Time synchronization and timeline integrity**

Accurate time is essential for correlation. If systems drift, you cannot reliably reconstruct the sequence of events.

Use **NTP** (Network Time Protocol) or a trusted time service across servers, network devices, security tools, and cloud workloads.

Why time sync matters:

- SIEM correlation depends on aligned timestamps.
- Packet captures, firewall logs, and authentication logs must line up.
- Incident timelines become unreliable if clocks differ by minutes or hours.
- Legal or disciplinary investigations may fail if evidence timing is inconsistent.

Good practice:

- Synchronize all critical systems to approved time sources.
- Monitor for drift and failed synchronization.
- Record time zone handling and UTC usage in procedures.
- Verify time settings after restores, snapshots, and migrations.

> Even small clock drift can break event correlation when you are reconstructing a fast-moving intrusion.

**Retention, storage, and chain of custody**

Captured packets and logs can become evidence. You need controls that preserve integrity and limit access.

Retention considerations:

- Define how long you keep raw packets, flow records, and logs.
- Balance storage cost against investigative and compliance needs.
- Keep high-fidelity packet captures for shorter periods unless policy requires longer retention.
- Retain summarized flow data longer when it supports trend analysis.

Evidence preservation steps:

1. Restrict access to authorized personnel.
2. Record who collected the data, when, and from where.
3. Hash files after capture and before transfer when possible.
4. Store originals in write-protected or access-controlled repositories.
5. Maintain chain-of-custody documentation for transfers and analysis.
6. Preserve metadata such as timestamps, sensor location, and capture filters.

Practical controls:

- Use immutable or append-only storage for critical logs.
- Encrypt data at rest and in transit.
- Separate analyst access from administrative access.
- Log all access to evidence repositories.

A packet capture without context is less useful than a smaller capture with clear provenance, timestamps, and scope.

**Practical command examples**

Linux interface and socket inspection:

```bash
ip addr show
ss -tulpn
sudo ethtool eth0
```

Quick traffic summary:

```bash
sudo tcpdump -i eth0 -nn -c 50
```

DNS-focused capture:

```bash
sudo tcpdump -i eth0 -nn port 53 -w dns-trace.pcap
```

Flow review on a collector:

```bash
nfdump -r flows.nfdump -o long
```

Example `snmpd.conf` snippet for SNMPv3 monitoring:

```conf
createUser monitorUser SHA "AuthPassw0rd!" AES "PrivPassw0rd!"
rouser monitorUser authPriv
syslocation "Data Center 1"
syscontact noc@example.com
```

Example SNMPv3 query:

```bash
snmpwalk -v3 -l authPriv -u monitorUser -a SHA -A 'AuthPassw0rd!' -x AES -X 'PrivPassw0rd!' 10.10.10.20 sysUpTime
```

SNMPv3 is preferred over older versions because it supports authentication and privacy. SNMPv1 and SNMPv2c rely on community strings and are weaker for security.

**Common exam scenarios and how to reason about them**

- A host shows periodic outbound connections every 60 seconds.
  - Think beaconing, scheduled task, or monitoring agent.
  - Check destination reputation, packet size consistency, and process logs.
- A subnet suddenly generates many DNS queries with long random names.
  - Think DNS tunneling or malware lookup behavior.
  - Review DNS logs, endpoint telemetry, and packet payloads.
- A firewall shows many denied connections to sequential ports.
  - Think port scanning.
  - Correlate with IDS alerts and source host behavior.
- A cloud workload becomes reachable after a security group change.
  - Check CloudTrail or Azure Activity Logs for the change event.
  - Validate whether the change was approved.
- East-west traffic spikes inside a microsegmented zone.
  - Investigate lateral movement, misrouted service traffic, or backup jobs.
  - Use internal sensors and host logs because perimeter tools may miss it.

**Exam-oriented distinctions to remember**

- **Monitoring** is continuous visibility; **analysis** is interpretation of the data.
- **Flow data** is summarized; **packet data** is detailed.
- **IDS** detects; **IPS** can block.
- **TAPs** are passive and reliable; **SPAN** is easier but less dependable under load.
- **Baselines** must reflect daily and seasonal patterns.
- **Cloud logs** often split into flow, control-plane, and workload layers.
- **Time sync** is mandatory for trustworthy correlation.
- **Segmentation** improves security but can reduce visibility unless you place sensors carefully.

**Quick review checklist**

- Define what normal traffic looks like before you hunt for anomalies.
- Use the right collection method for the job.
- Correlate flow, packet, and log data.
- Tune IDS/IPS to reduce false positives.
- Preserve evidence with access control, hashing, and chain of custody.
- Monitor cloud, container, and east-west traffic, not just perimeter traffic.
- Keep all systems synchronized to trusted time sources.
