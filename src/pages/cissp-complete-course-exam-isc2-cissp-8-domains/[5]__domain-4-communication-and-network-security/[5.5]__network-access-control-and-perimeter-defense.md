## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Network Access Control (NAC) overview**

NAC controls *who* and *what* can connect to the network, then applies policy based on identity, device posture, location, and risk. In CISSP terms, NAC is both an access control mechanism and a policy enforcement framework. It is commonly used to reduce lateral movement, block unmanaged devices, and quarantine noncompliant endpoints before they reach sensitive resources.

NAC solutions usually combine:
- **Authentication** to verify the user or device.
- **Authorization** to decide what access is allowed.
- **Posture assessment** to check security state.
- **Enforcement** to apply the decision at the network edge or through a controller.

A practical NAC design often integrates with directory services, RADIUS, switches, wireless controllers, VPN concentrators, and endpoint management tools.

**NAC deployment modes and enforcement models**

NAC can be deployed in different ways depending on how much control you need and where you can enforce policy.

- **Inline NAC** sits in the traffic path and can directly allow, block, or redirect traffic.
  - It gives strong control but can become a bottleneck or single point of failure if poorly designed.
  - It is common in architectures where the NAC appliance or controller actively mediates access.
- **Out-of-band NAC** evaluates identity and posture separately from the data path.
  - It often uses switch, wireless, or VPN integration to push policy decisions.
  - It scales better and is less disruptive, but enforcement depends on the underlying network devices.

Enforcement can also be described by where the policy is applied:
- **Wired switchport enforcement** uses port-based controls on access switches.
- **Wireless enforcement** uses SSIDs, VLANs, ACLs, or role-based policies on the wireless LAN controller.
- **VPN enforcement** applies policy at remote-access termination points.
- **Segmentation enforcement** uses VLANs, ACLs, security groups, or software-defined networking controls to isolate devices.

Common enforcement actions include:
- Permit full access.
- Place the endpoint in a restricted VLAN.
- Apply a downloadable ACL.
- Redirect to a remediation portal.
- Quarantine the device until it becomes compliant.

> If you see “NAC” on an exam question, look for the enforcement point and the trust decision: identity, posture, and location all matter.

**802.1X operational details**

<abbr title="Institute of Electrical and Electronics Engineers">IEEE</abbr> 802.1X is the standard port-based network access control framework used on wired and wireless networks. It uses three roles:
- **Supplicant**: the endpoint requesting access.
- **Authenticator**: the switch or wireless access point enforcing access.
- **Authentication server**: usually a RADIUS server.

The protocol exchange often begins with **EAPOL** (<abbr title="Extensible Authentication Protocol over LAN">EAP over LAN</abbr>) frames between the supplicant and authenticator. The authenticator relays EAP messages to the authentication server using RADIUS.

Typical flow:
1. The port starts in an unauthorized state.
2. The supplicant sends EAPOL-Start or responds to an EAP request.
3. The authenticator forwards credentials to the RADIUS server.
4. The server validates the user or device.
5. The authenticator opens the port or applies a restricted policy.

Common EAP methods include:
- **EAP-TLS**: certificate-based and strongly preferred when you can manage certificates.
- **PEAP**: tunnels inner authentication, often username/password.
- **EAP-TTLS**: similar tunnel-based approach, less common in some environments.
- **EAP-FAST**: used in some legacy deployments.

RADIUS attributes are important because they drive policy beyond simple allow/deny:
- **Tunnel-Type**, **Tunnel-Medium-Type**, and **Tunnel-Private-Group-ID** can assign a VLAN.
- **Filter-Id** can reference an ACL or policy profile.
- **Session-Timeout** can limit how long access remains valid.
- **Idle-Timeout** can remove inactive sessions.
- **Class** can carry session identifiers for accounting or policy correlation.
- **Vendor-Specific Attributes (VSAs)** can pass role, posture, or device-class information to network devices.

A simplified RADIUS policy example:

```text
if user == "contractor" then
  assign VLAN 120
  apply ACL "contractor-restricted"
  session-timeout 28800
endif
```

A vendor-neutral switchport example:

```text
interface GigabitEthernet1/0/10
 description User Access Port
 authentication port-control auto
 dot1x pae authenticator
 mab
 spanning-tree portfast
```

**802.1X failure and guest access flows**

802.1X deployments must handle devices that cannot authenticate cleanly.

Common fallback paths:
- **Guest access** for visitors or unmanaged devices.
- **Remediation VLAN** for endpoints that fail posture checks.
- **Critical auth VLAN** or local fallback when the RADIUS server is unavailable.
- **Open mode with monitoring** in limited cases during rollout.

A typical failure flow:
1. The endpoint fails certificate validation or user authentication.
2. The switch or controller places the device into a restricted network.
3. The user is redirected to a captive portal or remediation page.
4. The device downloads updates or installs required software.
5. The NAC system re-evaluates and moves the device to the production network.

Be careful with guest access design:
- Guest networks should be isolated from internal resources.
- Guest portals should not become a bypass for corporate access.
- Remediation networks should allow only update and management traffic needed to fix compliance issues.

**MAC Authentication Bypass limitations**

**MAC Authentication Bypass (MAB)** authenticates a device using its Media Access Control address when 802.1X is not possible. It is often used for legacy printers, IoT devices, phones, or embedded systems that cannot run a supplicant.

MAB is weak because:
- MAC addresses are easy to spoof.
- It provides device identification, not strong device identity.
- It does not prove user presence.
- It is vulnerable to cloning and replay-style abuse in poorly designed environments.

Use MAB only when necessary, and treat it as a compensating control:
- Restrict MAB devices to narrow VLANs or ACLs.
- Combine MAB with device profiling and monitoring.
- Prefer certificate-based or agent-based methods when possible.

**Posture assessment: agent-based vs agentless**

NAC posture assessment checks whether the endpoint meets security requirements before granting access.

- **Agent-based posture assessment**
  - Uses software installed on the endpoint.
  - Can inspect patch level, antivirus status, disk encryption, firewall state, and running processes.
  - Usually provides deeper visibility and better remediation.
  - Requires software deployment and ongoing maintenance.

- **Agentless posture assessment**
  - Uses network checks, browser-based checks, or device fingerprinting.
  - Easier to deploy for guests or unmanaged devices.
  - Provides less assurance because it cannot inspect the endpoint as deeply.
  - Often suitable for limited access or low-risk scenarios.

A strong exam answer usually links posture depth to trust level:
- High-trust corporate devices: agent-based plus certificates.
- Low-trust or unmanaged devices: agentless plus restricted access.
- Legacy devices: MAB plus segmentation and monitoring.

**NAC in wired, wireless, VPN, and switchport environments**

NAC enforcement differs by access medium.

- **Wired**
  - The switchport is the enforcement point.
  - 802.1X, MAB, VLAN assignment, and downloadable ACLs are common.
  - Port security may complement NAC but does not replace it.
- **Wireless**
  - The access point or wireless controller enforces policy.
  - Users may be placed into role-based access profiles or VLANs.
  - Guest portals and captive portals are common for onboarding.
- **VPN**
  - The VPN concentrator checks user identity and often endpoint posture.
  - Noncompliant devices may be denied tunnel access or placed into a limited-access group.
  - Split tunneling decisions can affect exposure and should be policy-driven.
- **Switchport**
  - The access switch controls whether the port is authorized.
  - Voice and data VLANs may be separated.
  - Unauthorized devices can be blocked, quarantined, or redirected.

**Perimeter defense fundamentals**

Perimeter defense is the layered set of controls that separate trusted internal networks from untrusted external networks. In modern environments, the “perimeter” is not just the internet edge; it also includes cloud boundaries, remote access, partner links, and internal segmentation points.

A useful progression of firewall types is:
- **Packet-filtering firewall**
  - Evaluates headers such as source/destination IP, port, and protocol.
  - Fast and simple, but limited context.
- **Stateful firewall**
  - Tracks connection state and allows packets based on session context.
  - Better for real traffic patterns and return traffic handling.
- **Application-layer firewall**
  - Inspects application content and protocol behavior.
  - Can enforce policy at HTTP, DNS, SMTP, and other application layers.

Related gateway types:
- **Proxy firewall**
  - Terminates client connections and creates new connections to the destination.
  - Provides strong inspection and hides internal clients.
  - Often used for web proxying and content control.
- **Circuit-level gateway**
  - Validates sessions at the transport/session layer, such as SOCKS-style mediation.
  - Less content-aware than application proxies.
  - Useful when you want session control without full application inspection.

**Firewall architecture and DMZ design**

Firewall placement matters as much as firewall type.

Common architectures:
- **Screened host**
  - A firewall protects a single bastion host or critical server.
  - The host is hardened and exposed only through controlled paths.
- **Dual-homed host**
  - A host with two network interfaces sits between networks and forwards traffic under strict control.
  - Historically important, but less common today because dedicated firewalls are preferred.
- **Screened subnet / DMZ**
  - A demilitarized zone separates public-facing services from the internal network.
  - Internet traffic reaches the DMZ, while internal access is tightly restricted.

DMZ variations:
- **Single-firewall DMZ**
  - One firewall creates inside, outside, and DMZ zones.
  - Simpler and cheaper.
- **Dual-firewall DMZ**
  - One firewall faces the internet and another protects the internal network.
  - Adds defense in depth and clearer separation.
- **Multi-tier DMZ**
  - Separate zones for web, application, and database tiers.
  - Useful when you want to limit east-west movement between tiers.

A practical zone-based policy example:

```text
zone trust
zone untrust
zone dmz

policy trust-to-dmz allow tcp 443
policy untrust-to-dmz allow tcp 443
policy untrust-to-trust deny any
policy dmz-to-trust allow tcp 1433 from app-server to db-server
```

**NAT, PAT, and perimeter exposure**

<abbr title="Network Address Translation">NAT</abbr> translates IP addresses between networks. <abbr title="Port Address Translation">PAT</abbr> multiplexes many internal hosts behind one public IP using different source ports.

Why NAT/PAT matters at the perimeter:
- It hides internal addressing from external observers.
- It reduces direct inbound exposure by requiring explicit port forwarding or static mappings.
- It is not a security control by itself, but it can reduce attack surface and complicate reconnaissance.

Important exam distinction:
- NAT does **not** replace firewall policy.
- A host behind NAT can still be exposed if you publish a service with port forwarding.
- Outbound NAT is common; inbound NAT should be tightly controlled and documented.

**IDS and IPS deployment concepts**

<abbr title="Intrusion Detection System">IDS</abbr> and <abbr title="Intrusion Prevention System">IPS</abbr> monitor traffic for malicious or suspicious activity.

- **IDS**
  - Detects and alerts.
  - Usually out-of-band or passive.
  - Lower risk of blocking legitimate traffic.
- **IPS**
  - Detects and blocks.
  - Usually inline.
  - Higher control, but more operational risk.

Detection methods:
- **Signature-based detection**
  - Matches known patterns or indicators.
  - Good for known threats.
  - Weak against novel attacks.
- **Anomaly-based detection**
  - Flags deviations from baseline behavior.
  - Better for unknown threats.
  - Can generate more false positives.

Operational tradeoffs:
- **False positive**: benign traffic flagged as malicious.
- **False negative**: malicious traffic missed by the system.

Inline behavior matters:
- **Fail-open**
  - Traffic continues if the device fails.
  - Preserves availability.
  - Reduces security during failure.
- **Fail-closed**
  - Traffic stops if the device fails.
  - Preserves security.
  - Can create outages if the device is unstable.

A common exam answer is that IPS favors security and may fail closed, while IDS favors visibility and availability.

**Web and application perimeter controls**

Application-layer controls inspect traffic more deeply than traditional firewalls.

Common controls:
- **Web Application Firewall (WAF)**
  - Protects web apps from attacks such as SQL injection and cross-site scripting.
  - Understands HTTP semantics and application patterns.
- **Reverse proxy**
  - Sits in front of servers and hides origin systems.
  - Can terminate TLS, balance load, and enforce access policy.
- **API gateway**
  - Controls authentication, rate limiting, and routing for APIs.
  - Often used in microservices environments.

A simple WAF-style rule example:

```text
if request.uri contains "/admin" and source_ip not in allowlist then
  block
endif

if request.body matches /union\s+select/i then
  log and block
endif
```

These tools are not interchangeable:
- A WAF protects web application logic.
- A reverse proxy protects and abstracts the server edge.
- An API gateway governs API consumption and service-to-service access.

**Cloud perimeter and modern boundary controls**

Cloud environments shift perimeter defense from physical appliances to policy-driven services.

Key cloud controls:
- **Security groups** and network ACLs for instance-level and subnet-level filtering.
- **Cloud firewalls** for centralized policy enforcement.
- **Web application firewalls** for internet-facing applications.
- **Identity and access management** for control-plane access.
- **Logging and flow analysis** for visibility across distributed services.

Modern perimeter concepts:
- **Zero Trust** assumes no implicit trust based on network location.
- **Software-Defined Perimeter (SDP)** hides services until identity and policy are verified.
- **Secure Access Service Edge (SASE)** combines networking and security services in a cloud-delivered model.
- **Security Service Edge (SSE)** focuses on security functions such as secure web gateway, CASB, and zero trust network access.

Adjacent controls often tested with perimeter topics:
- **DLP** (<abbr title="Data Loss Prevention">Data Loss Prevention</abbr>) prevents sensitive data exfiltration.
- **CASB** (<abbr title="Cloud Access Security Broker">Cloud Access Security Broker</abbr>) governs cloud app usage and policy enforcement.
- **ZTNA** (<abbr title="Zero Trust Network Access">Zero Trust Network Access</abbr>) provides application-level access without broad network trust.

These controls complement, but do not replace, firewalls and segmentation.

**Practical perimeter design patterns**

A secure perimeter design usually combines multiple layers rather than relying on one device.

Common patterns:
- Internet edge firewall with default-deny inbound policy.
- DMZ for public services.
- Internal segmentation firewalls between user, server, and management networks.
- IDS/IPS for detection and prevention.
- WAF for web applications.
- VPN or ZTNA for remote access.
- NAC for endpoint admission control.

A sample layered topology:

```text
Internet
  |
[Edge Firewall] ---- [IDS Sensor]
  |
[DMZ: WAF / Reverse Proxy / Public Web]
  |
[Internal Firewall]
  |
[App Network]
  |
[DB Network]
```

A remote-access policy example:

```text
if device_compliant and user_mfa_passed then
  allow access to app-group
else if user_authenticated and device_unknown then
  allow only remediation portal
else
  deny
endif
```

**Operational tuning and common pitfalls**

Perimeter tools fail when policy is too broad, too narrow, or poorly monitored.

Watch for:
- Overreliance on NAT or firewall placement as “security.”
- Allowing broad any-any rules for troubleshooting and never removing them.
- Using MAB for too many devices without compensating controls.
- Deploying IPS inline without testing for false positives.
- Treating guest access as a temporary exception that becomes permanent.
- Assuming cloud security groups alone provide full perimeter protection.

Good practice includes:
- Logging authentication and authorization decisions.
- Reviewing denied and quarantined endpoints.
- Testing failover behavior for inline devices.
- Documenting exceptions and expiration dates.
- Aligning NAC posture checks with endpoint management and patching.

**Exam focus summary**

For CISSP-style questions, distinguish these ideas:
- **NAC** decides whether a device or user can join and what access they get.
- **802.1X** is the standard mechanism for port-based authentication.
- **MAB** is a weak fallback for legacy or IoT devices.
- **Firewall types** progress from packet filtering to stateful to application-layer inspection.
- **DMZs** isolate public services from internal assets.
- **IDS** detects; **IPS** blocks.
- **NAT/PAT** hides addresses but does not equal security.
- **Zero Trust**, **SDP**, **SASE**, and **SSE** represent modern perimeter thinking beyond the traditional edge.
