**Firewall rule fundamentals**

Firewalls enforce policy by matching traffic against rules and then taking an action such as allow, deny, reject, log, or rate-limit. For CISSP, you need to understand not just the syntax of common examples, but also the security intent behind them: least privilege, explicit deny, and careful placement of logging.

A rule is usually evaluated against:
- **Direction**: inbound, outbound, or transit.
- **Protocol**: TCP, UDP, ICMP, or others.
- **Endpoints**: source and destination IPs, subnets, interfaces, zones, or security groups.
- **Ports and flags**: service ports, TCP flags, or ICMP types/codes.
- **State**: whether the packet belongs to an established connection.

Common policy models include:
- **Default deny**: block everything unless explicitly allowed.
- **Default allow**: permit everything unless explicitly blocked.
- **Implicit deny**: unmatched traffic is denied at the end of the rule set.

For exam purposes, remember that rule order matters on many platforms. Some firewalls evaluate top-down and stop at the first match, while others use object-based policy, zones, or compiled rule tables.

**Cisco ACL examples**

Cisco <abbr title="Access Control List">ACL</abbr>s are a classic exam topic because they show how network devices filter traffic at Layer 3 and Layer 4. Standard ACLs match only source IP addresses, while extended ACLs can match protocol, source, destination, and ports.

A standard ACL example:
```cisco
access-list 10 permit 192.168.10.0 0.0.0.255
access-list 10 deny any
```

An extended ACL example that permits HTTPS to a specific server:
```cisco
access-list 101 permit tcp any host 172.16.10.20 eq 443
access-list 101 deny ip any any
```

A more specific example for a subnet:
```cisco
access-list 101 permit tcp 10.10.0.0 0.0.255.255 host 172.16.10.20 eq 443
```

Key points:
- **Wildcard masks** are inverse masks, not subnet masks.
- `host` means a single IP address.
- `any` means any source or destination, depending on position.
- Port qualifiers such as `eq 443` apply to the destination in typical extended ACL syntax.
- A final deny is often implicit, but adding an explicit deny can improve readability and logging.

A common logging pattern:
```cisco
access-list 101 permit tcp any host 172.16.10.20 eq 443 log
```

Use logging carefully because excessive logging can create noise and performance overhead. On production devices, rate-limit or sample logs when possible.

**Host firewall examples**

A host-based firewall runs on the endpoint and filters traffic at the host’s network stack. It is useful when you want protection that follows the system regardless of network location, such as laptops, servers, and virtual machines.

Typical host firewall goals:
- Restrict inbound management access.
- Limit outbound connections from sensitive servers.
- Allow only required services on specific interfaces.
- Log suspicious or unexpected traffic.

A simple Linux `iptables` baseline:
```bash
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -m conntrack --ctstate NEW -j ACCEPT
```

Why these rules matter:
- **Loopback** traffic is required for local services and inter-process communication.
- **Established,RELATED** traffic allows return packets for sessions already permitted.
- SSH is allowed only as a new inbound connection on port 22.

A more complete baseline often includes outbound control if the host is sensitive:
```bash
iptables -P OUTPUT DROP
iptables -A OUTPUT -o lo -j ACCEPT
iptables -A OUTPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A OUTPUT -p tcp -d 203.0.113.10 --dport 443 -m conntrack --ctstate NEW -j ACCEPT
```

For Windows Defender Firewall, a similar policy might be expressed as:
```powershell
New-NetFirewallRule -DisplayName "Allow SSH Inbound" -Direction Inbound -Protocol TCP -LocalPort 22 -Action Allow
```

Exam takeaway:
- A host firewall protects the endpoint itself.
- It is not the same as a perimeter firewall or a network ACL.
- It usually sees traffic after it reaches the host network stack, not as a routing device would.

**Network ACLs, security groups, and cloud controls**

Cloud and virtualized environments often use multiple layers of filtering. The exact behavior varies by provider, so avoid assuming one universal model.

Common patterns:
- **Network ACLs (NACLs)** are subnet-level controls in some clouds and are often stateless, meaning return traffic may need separate rules.
- **Security groups** are commonly stateful and attached to an instance, virtual NIC, or equivalent resource.
- **Route tables** decide where traffic goes; they do not usually enforce packet-level security policy.

A practical cloud-style example:
- Allow inbound HTTPS to a web tier.
- Allow web tier to reach an application tier on a private port.
- Deny direct database access from the internet.
- Permit database access only from the application tier.

Example policy logic:
- Internet → Web: allow TCP 443
- Web → App: allow TCP 8443
- App → DB: allow TCP 5432
- Internet → DB: deny

Because cloud defaults differ, verify:
- Whether the control is stateful or stateless.
- Whether rules are allow-only or support explicit deny.
- Whether rule order matters.
- Whether there is an implicit deny at the end.

**iptables and nftables concepts**

`iptables` remains a common reference point for firewall behavior, even where `nftables` is the modern backend. CISSP candidates should understand the policy model more than the exact syntax.

Useful `iptables` examples:
```bash
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
iptables -A INPUT -p icmp -j ACCEPT
iptables -A INPUT -j DROP
```

A logging rule before a drop:
```bash
iptables -I INPUT 1 -p tcp --dport 22 -j LOG --log-prefix "SSH DROP "
iptables -I INPUT 2 -p tcp --dport 22 -j DROP
```

Important notes:
- `LOG` does not stop rule processing.
- Put a matching `DROP` or `REJECT` after a log rule if you want enforcement.
- Logging should usually be rate-limited to avoid log flooding.

A rate-limited example:
```bash
iptables -A INPUT -p tcp --dport 22 -m limit --limit 5/min --limit-burst 10 -j LOG --log-prefix "SSH ATTEMPT "
```

`nftables` uses a different syntax but the same policy ideas:
```bash
nft add rule inet filter input tcp dport 22 accept
nft add rule inet filter input counter drop
```

**Packet capture basics**

Packet capture tools help you validate whether a firewall rule is working as intended. The two most common contexts are:
- **Capture filters**: applied before packets are captured, usually with libpcap/BPF syntax.
- **Display filters**: applied after capture, commonly in Wireshark.

Capture filters reduce noise and disk usage. Display filters help you inspect captured traffic in detail.

A safe capture filter for SSH and HTTPS to or from one host:
```bash
host 192.0.2.10 and (tcp port 22 or tcp port 443)
```

A capture filter for traffic on a specific interface and subnet:
```bash
net 192.0.2.0/24 and tcp port 443
```

A capture filter for ICMP:
```bash
icmp
```

Be careful with syntax differences across tools and versions. When in doubt, test the filter on a small capture first.

**Wireshark display filters**

Wireshark display filters are powerful because they let you inspect protocol fields after capture. They are not the same as capture filters.

Common display filters:
```wireshark
ip.addr == 192.0.2.10
tcp.port == 443
tcp.flags.syn == 1 && tcp.flags.ack == 0
```

To isolate traffic between two hosts:
```wireshark
ip.addr == 192.0.2.10 && ip.addr == 198.51.100.25
```

To view HTTPS-related TCP sessions:
```wireshark
tcp.port == 443
```

To inspect SYN packets during connection setup:
```wireshark
tcp.flags.syn == 1 && tcp.flags.ack == 0
```

To find retransmissions:
```wireshark
tcp.analysis.retransmission
```

To focus on DNS:
```wireshark
dns
```

To inspect HTTP:
```wireshark
http
```

For ICMP troubleshooting:
```wireshark
icmp
```

For IPv6 ICMP:
```wireshark
icmpv6
```

**ICMP troubleshooting filters**

ICMP is often used to test reachability and diagnose filtering behavior. In Wireshark, ICMP type and code values help you identify the reason for a message.

Examples:
```wireshark
icmp.type == 8
```
This matches IPv4 echo request packets.

```wireshark
icmp.type == 0
```
This matches IPv4 echo reply packets.

```wireshark
icmp.type == 3
```
This matches IPv4 destination unreachable messages.

```wireshark
icmp.type == 3 && icmp.code == 1
```
This matches one specific IPv4 destination unreachable subtype: host unreachable.

Use caution:
- ICMP meanings differ between IPv4 and IPv6.
- `icmp` and `icmpv6` are different protocol families.
- A firewall may block some ICMP types while allowing others.

For troubleshooting, ICMP can reveal:
- Host reachability problems.
- Fragmentation issues.
- Port unreachable responses.
- Administrative filtering.

**Practical validation workflow**

When you test a firewall rule, validate both the policy and the packet path. A rule may be correct but still not match because of routing, NAT, interface selection, or rule order.

A simple workflow:
1. Identify the expected source, destination, protocol, and port.
2. Confirm the rule exists on the correct device or host.
3. Check whether the rule is above a broader deny.
4. Generate test traffic.
5. Capture packets on the relevant interface.
6. Compare the observed packet fields with the rule conditions.
7. Review logs for deny or accept events.

Example test commands:
```bash
curl -vk https://192.0.2.10/
nc -vz 192.0.2.10 443
ping 192.0.2.10
traceroute 192.0.2.10
```

If SSH is blocked, test with:
```bash
nc -vz 192.0.2.10 22
```

If you need to confirm packet flow on Linux:
```bash
tcpdump -ni eth0 host 192.0.2.10 and tcp port 443
```

**Common exam traps**

Firewall questions often test whether you understand the difference between policy intent and implementation detail.

Watch for these traps:
- **Stateful vs stateless**: do not assume every firewall tracks sessions.
- **Inbound vs outbound**: a rule that allows one direction may not allow the return path on a stateless control.
- **Rule order**: a broad deny above a specific allow can block legitimate traffic.
- **Port direction**: in extended ACLs, the port usually applies to the destination side.
- **Capture vs display filters**: they use different syntax and serve different purposes.
- **ICMP type/code**: meanings vary by protocol version.
- **Logging**: logging is not enforcement; it is visibility.

**Quick comparison**

| Control type | Typical scope | Stateful | Common use |
|---|---|---:|---|
| Host firewall | Single endpoint | Often yes | Protect a server or workstation |
| Network firewall | Network boundary or segment | Often yes | Enforce perimeter or zone policy |
| NACL | Subnet or network segment | Often no | Coarse cloud subnet filtering |
| Security group | Instance, VM, or NIC | Often yes | Cloud workload access control |

**Study summary**

Firewall rules define what traffic is allowed, denied, logged, or inspected. For CISSP, focus on the relationship between rule order, statefulness, direction, and scope. Use packet captures and Wireshark filters to verify what actually traverses the network, and remember that capture filters and display filters solve different problems.

If you can explain why a packet matches or misses a rule, you understand the topic well enough for exam scenarios and real-world troubleshooting.
