## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**DNS security fundamentals**

The <abbr title="Domain Name System">DNS</abbr> translates names to IP addresses and other service data. Because almost every networked application depends on DNS, attackers often target it for redirection, reconnaissance, cache poisoning, and denial of service.

- **Availability** matters because DNS outages can make services appear offline even when servers are healthy.
- **Integrity** matters because poisoned or forged answers can redirect users to malicious infrastructure.
- **Confidentiality** matters because DNS queries can reveal internal hosts, applications, and user behavior.

Common record types include:

- `A` and `AAAA` for IPv4 and IPv6 address mapping
- `CNAME` for aliases
- `MX` for mail routing
- `NS` for authoritative name servers
- `TXT` for policy and verification data
- `PTR` for reverse lookups
- `SOA` for zone authority and timing
- `CAA` for certificate issuance control

`SOA` and `CAA` are security-relevant in exam contexts:

- **SOA** defines zone metadata such as serial number, refresh, retry, expire, and minimum TTL values. Poorly managed SOA values can delay propagation or complicate recovery after changes.
- **CAA** lets a domain owner restrict which certificate authorities may issue certificates for the domain. This reduces the risk of unauthorized issuance.

**DNS attack patterns and defenses**

DNS is frequently abused for reconnaissance and traffic manipulation. You should recognize both the attack and the control that limits it.

- **Cache poisoning** attempts to insert false records into a resolver cache.
- **Spoofing** forges DNS responses to redirect clients.
- **Tunneling** hides data inside DNS queries and responses.
- **Amplification** uses open resolvers and large responses to magnify DDoS traffic.
- **Zone transfer abuse** exposes internal hostnames if `AXFR` is not restricted.
- **Enumeration** uses brute-force or dictionary techniques to discover subdomains.

Practical defenses include:

- Restrict zone transfers to authorized secondary servers only.
- Disable recursion on authoritative-only servers.
- Use response rate limiting where supported.
- Randomize source ports and query IDs in resolvers.
- Keep authoritative and recursive roles separate when possible.
- Monitor for unusual query volume, long labels, and high-entropy subdomains that may indicate tunneling.

A simple BIND-style hardening example for an internal recursive resolver:

```conf
options {
    recursion yes;
    allow-recursion { 10.0.0.0/8; 192.168.0.0/16; };
    allow-query-cache { 10.0.0.0/8; 192.168.0.0/16; };
    dnssec-validation auto;
    minimal-responses yes;
    version "not disclosed";
};

zone "example.internal" {
    type master;
    file "/etc/bind/db.example.internal";
    allow-transfer { 10.0.0.53; };
};
```

Operational checks:

```bash
dig @10.0.0.53 example.com A
dig +dnssec @10.0.0.53 example.com
dig axfr example.internal @10.0.0.53
```

> `AXFR` should fail unless the requester is an approved secondary server or administrator.

**DNSSEC and resolver validation**

<abbr title="Domain Name System Security Extensions">DNSSEC</abbr> adds digital signatures to DNS data so resolvers can validate authenticity and integrity. It does not encrypt DNS traffic, but it helps prevent forged answers when validation is enabled.

Key concepts:

- **Zone signing** signs records in a zone with a private key.
- **Chain of trust** links the zone to a trusted parent through `DS` records.
- **Validation** occurs on the recursive resolver, not on the client in most deployments.
- **Trust anchors** are the starting points used by validating resolvers.

DNSSEC is only effective when validation is actually performed. Unsigned zones or non-validating resolvers do not gain protection from forged data.

Typical validation workflow:

1. The authoritative server publishes signed records.
2. The parent zone publishes `DS` records.
3. The recursive resolver checks signatures against its trust anchor.
4. The client receives a validated answer or a failure.

Example `unbound` resolver hardening:

```conf
server:
    interface: 127.0.0.1
    access-control: 127.0.0.0/8 allow
    access-control: 10.0.0.0/8 allow
    hide-identity: yes
    hide-version: yes
    qname-minimisation: yes
    harden-glue: yes
    harden-dnssec-stripped: yes
    auto-trust-anchor-file: "/var/lib/unbound/root.key"
```

Useful validation commands:

```bash
dig +dnssec +multi example.com
delv example.com
unbound-control status
```

**Split-horizon DNS and leakage risks**

Split-horizon DNS serves different answers to internal and external clients. It is useful for internal-only hostnames, private IP addressing, and service segmentation.

Common use cases:

- Internal users resolve `app.example.com` to a private address.
- External users resolve the same name to a public load balancer.
- Internal zones expose hostnames that should never appear in public DNS.

Risks and controls:

- **Leakage** occurs when internal names are accidentally published in public zones, logs, or certificates.
- **Access control** should limit who can query internal views.
- **Consistency** must be managed carefully to avoid split-brain behavior.
- **Monitoring** should detect unauthorized zone transfers and misconfigured forwarders.

**Email transport, submission, and retrieval**

Email security is broader than SMTP alone. You should distinguish between message transfer, user submission, and mailbox retrieval.

- **SMTP** (`25/tcp`) is primarily used for server-to-server relay.
- **Submission** (`587/tcp`) is used by mail clients to send outbound mail through authenticated submission.
- **SMTPS** (`465/tcp`) is commonly used for implicit TLS submission in some environments.
- **IMAP** (`143/tcp`) and **POP3** (`110/tcp`) are legacy cleartext retrieval protocols.
- **IMAPS** (`993/tcp`) and **POP3S** (`995/tcp`) protect retrieval with TLS.

Best practice is to require TLS for submission and retrieval:

- Use `587` with STARTTLS for client submission.
- Use `993` for IMAP and `995` for POP3 when those protocols are still needed.
- Disable cleartext authentication on untrusted networks.
- Prefer modern clients and MFA where supported.

A Postfix submission service should be explicitly hardened rather than enabled with a minimal listener only. A typical approach is to require TLS and authenticated submission:

```cf
# /etc/postfix/master.cf
submission inet n       -       y       -       -       smtpd
  -o syslog_name=postfix/submission
  -o smtpd_tls_security_level=encrypt
  -o smtpd_tls_auth_only=yes
  -o smtpd_sasl_auth_enable=yes
  -o smtpd_relay_restrictions=permit_sasl_authenticated,reject
  -o smtpd_recipient_restrictions=permit_sasl_authenticated,reject
```

And in `main.cf`:

```cf
smtpd_tls_cert_file=/etc/ssl/certs/mail.example.com.pem
smtpd_tls_key_file=/etc/ssl/private/mail.example.com.key
smtpd_tls_security_level=may
smtpd_sasl_auth_enable=yes
smtpd_relay_restrictions=permit_mynetworks,permit_sasl_authenticated,reject_unauth_destination
```

> Exact Postfix policy placement varies by version. The key exam idea is that submission should require authentication and TLS, and relay should be denied to unauthenticated users.

**Email threats: spoofing, phishing, spear phishing, and BEC**

Email remains a primary social engineering channel because it is trusted, ubiquitous, and easy to automate.

- **Spoofing** forges sender identity or domain information.
- **Phishing** uses deceptive messages to steal credentials or deliver malware.
- **Spear phishing** targets a specific person or role with tailored content.
- **Business email compromise** (<abbr title="Business Email Compromise">BEC</abbr>) impersonates executives, vendors, or finance staff to trigger fraudulent payments or data disclosure.

Defensive controls include:

- User awareness training and reporting workflows
- Mail gateway filtering and sandboxing
- MFA for email and identity providers
- Domain authentication with SPF, DKIM, and DMARC
- Out-of-band verification for payment or account changes
- Banner warnings for external senders
- Attachment and URL detonation in a secure environment

**SPF, DKIM, and DMARC**

These three controls work together to reduce spoofing and improve domain reputation.

- **SPF** (<abbr title="Sender Policy Framework">SPF</abbr>) tells receivers which hosts may send mail for a domain.
- **DKIM** (<abbr title="DomainKeys Identified Mail">DKIM</abbr>) signs messages so receivers can verify integrity and domain association.
- **DMARC** (<abbr title="Domain-based Message Authentication, Reporting, and Conformance">DMARC</abbr>) tells receivers how to handle messages that fail SPF and/or DKIM alignment.

SPF record structure example:

```txt
example.com. IN TXT "v=spf1 ip4:203.0.113.10 include:_spf.google.com -all"
```

Notes:

- `ip4:` and `ip6:` authorize sending IP ranges.
- `include:` delegates to another SPF policy.
- `-all` is a hard fail; `~all` is a soft fail.
- Keep SPF lookups within the DNS query limit to avoid `permerror`.

DKIM example with a selector:

```txt
selector1._domainkey.example.com. IN TXT "v=DKIM1; k=rsa; p=MIIBIjANBgkq..."
```

Practical guidance:

- Use a **selector** to support key rotation without breaking all mail at once.
- Publish separate selectors for different mail systems if needed.
- Rotate keys periodically and remove old selectors after the transition window.
- Sign headers that matter for anti-tampering and routing decisions.

DMARC example:

```txt
_dmarc.example.com. IN TXT "v=DMARC1; p=quarantine; rua=mailto:dmarc-reports@example.com; adkim=s; aspf=s"
```

Rollout considerations:

- Start with `p=none` to collect reports and understand legitimate mail flows.
- Move to `quarantine` after fixing alignment issues.
- Use `reject` only after testing forwarding, mailing lists, and third-party senders.
- Remember that DMARC is a receiver policy; receivers decide whether and how to enforce it.

**Mail gateway filtering and attachment controls**

Mail gateways reduce risk by inspecting content before delivery.

Common controls:

- Spam and phishing scoring
- Attachment type blocking
- URL rewriting and time-of-click analysis
- Malware scanning and sandboxing
- Header anomaly detection
- Impersonation detection for executive and vendor names

Attachment handling should be conservative:

- Block executable content where possible.
- Detonate archives and documents in a sandbox.
- Strip active content from office documents when business requirements allow.
- Quarantine messages that fail policy rather than auto-delivering them.

**Web security fundamentals**

Web applications are exposed to both browser-side and server-side threats. Secure design requires controls at the application, transport, and session layers.

Core goals:

- Protect credentials and session tokens.
- Validate all input on the server.
- Encode output for the correct context.
- Reduce attack surface with least privilege and secure defaults.
- Log security-relevant events without exposing secrets.

**Common server-side vulnerabilities**

These are high-value exam topics because they often lead to full application compromise.

- **SQL injection** occurs when untrusted input changes database queries.
- **Command injection** occurs when input reaches a shell or system command.
- **Path traversal** allows access to files outside the intended directory.
- **Deserialization** flaws can lead to code execution or privilege abuse.
- **Server-side request forgery** (<abbr title="Server-Side Request Forgery">SSRF</abbr>) makes the server fetch attacker-chosen resources.
- **Insecure direct object references** expose objects by predictable identifiers.
- **Broken access control** allows unauthorized actions or data access.

Defensive patterns:

- Use parameterized queries or ORM bindings for database access.
- Avoid shell invocation; use safe APIs and argument arrays.
- Canonicalize and validate file paths before access.
- Deserialize only trusted data, or avoid unsafe serializers entirely.
- Enforce authorization on every sensitive object and action.
- Deny by default and explicitly allow known-safe operations.

Example of safe database access in Python:

```python
import sqlite3

def get_user(email):
    conn = sqlite3.connect("app.db")
    cur = conn.cursor()
    cur.execute("SELECT id, email FROM users WHERE email = ?", (email,))
    return cur.fetchone()
```

Unsafe pattern to avoid:

```python
query = "SELECT * FROM users WHERE email = '" + email + "'"
```

Example of command injection risk:

```python
import os
os.system("ping -c 1 " + host)
```

Safer approach:

```python
import subprocess
subprocess.run(["ping", "-c", "1", host], check=True)
```

**File upload security**

File upload features are a common entry point for malware, web shells, and storage abuse. Validation must happen on the server and should not trust client-provided metadata.

Good controls:

- Restrict allowed file types by business need.
- Validate file signatures, not just extensions.
- Store uploads outside the web root when possible.
- Rename files to random identifiers.
- Enforce size limits and rate limits.
- Scan uploads with anti-malware tools.
- Remove active content from documents if feasible.

A safer validation example checks both extension and magic bytes:

```python
from pathlib import Path

ALLOWED_EXTENSIONS = {".png", ".jpg", ".jpeg"}
PNG_MAGIC = b"\x89PNG\r\n\x1a\n"

def is_allowed_upload(filename, fileobj):
    ext = Path(filename).suffix.lower()
    if ext not in ALLOWED_EXTENSIONS:
        return False

    head = fileobj.read(8)
    fileobj.seek(0)

    if ext == ".png":
        return head == PNG_MAGIC

    return True
```

> Do not trust the browser-supplied `Content-Type` header as proof of file type. It is easy to spoof.

**Session management**

Session security protects authenticated users from hijacking and fixation attacks.

Best practices:

- Generate a new session ID after login and privilege changes.
- Use `HttpOnly` to reduce script access to cookies.
- Use `Secure` so cookies are sent only over TLS.
- Use `SameSite=Lax` or `SameSite=Strict` where compatible.
- Set idle and absolute timeouts.
- Invalidate sessions on logout and password reset.
- Bind sessions carefully to risk signals without breaking legitimate mobility.

Example cookie:

```http
Set-Cookie: sessionid=abc123; Secure; HttpOnly; SameSite=Lax; Path=/; Max-Age=1800
```

Session fixation defense:

1. The user visits the site anonymously.
2. The application assigns a temporary session.
3. After authentication, the application regenerates the session ID.
4. The old session ID is invalidated.

**Cross-site request forgery and browser-side controls**

<abbr title="Cross-Site Request Forgery">CSRF</abbr> tricks a logged-in browser into sending an unwanted request. The browser automatically includes cookies, which is why state-changing requests need protection.

Primary defenses:

- Use anti-CSRF tokens tied to the user session.
- Validate `Origin` and `Referer` headers for sensitive requests.
- Set cookies with `SameSite` where possible.
- Require reauthentication or step-up authentication for high-risk actions.

Double-submit cookie pattern example:

```http
Set-Cookie: csrf_token=RANDOM123; SameSite=Lax
```

```html
<form method="post" action="/transfer">
  <input type="hidden" name="csrf_token" value="RANDOM123">
  <input type="text" name="amount">
  <button type="submit">Send</button>
</form>
```

Server-side check:

```python
def validate_csrf(request):
    return request.form["csrf_token"] == request.cookies["csrf_token"]
```

This pattern is useful, but server-side token storage or signed tokens are often stronger in complex applications. Origin and referer validation are useful secondary checks, not the only control.

**Browser protections: XSS, CSP, and clickjacking**

<abbr title="Cross-Site Scripting">XSS</abbr> occurs when attacker-controlled script runs in a victim’s browser. It can steal tokens, perform actions, or alter page content.

Defenses:

- Encode output for HTML, attribute, JavaScript, and URL contexts.
- Sanitize rich text with a trusted allowlist.
- Use a strong Content Security Policy (<abbr title="Content Security Policy">CSP</abbr>).
- Avoid inline scripts where possible.
- Mark cookies `HttpOnly` to reduce token theft impact.

Modern CSP example:

```http
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-abc123'; object-src 'none'; base-uri 'self'; frame-ancestors 'none'; report-to csp-endpoint
```

Notes:

- `frame-ancestors 'none'` helps prevent clickjacking.
- `report-to` is the preferred modern reporting mechanism; `report-uri` is legacy but still seen in older deployments.
- Nonces allow specific inline scripts without opening the policy broadly.

Clickjacking defenses:

- `X-Frame-Options: DENY` or `SAMEORIGIN`
- `Content-Security-Policy: frame-ancestors 'none'`
- UI design that requires explicit user confirmation for sensitive actions

**TLS and PKI for web and email**

<abbr title="Transport Layer Security">TLS</abbr> protects data in transit against eavesdropping and tampering. It is essential for HTTPS, SMTP submission, IMAPS, POP3S, and many API connections.

PKI (<abbr title="Public Key Infrastructure">PKI</abbr>) concepts to know:

- **Certificate Authority** (<abbr title="Certificate Authority">CA</abbr>) issues certificates.
- **Trust store** contains root certificates trusted by an operating system or application.
- **Intermediate certificate** chains the leaf certificate to a trusted root.
- **Leaf certificate** identifies the server or service.
- **Certificate Transparency** logs public certificate issuance to improve detection of mis-issuance.

Modern TLS deployment considerations:

- Include `subjectAltName` entries for all valid hostnames and IPs.
- Use strong key sizes and modern signature algorithms.
- Prefer TLS 1.2 or TLS 1.3.
- Disable weak ciphers and obsolete protocol versions.
- Renew certificates before expiration and monitor for chain issues.

A simple OpenSSL lab example for a self-signed certificate is useful for testing, but it is not a full production PKI workflow:

```bash
openssl req -x509 -newkey rsa:2048 -sha256 -days 365 -nodes \
  -keyout server.key -out server.crt \
  -subj "/CN=mail.example.com"
```

For real web services, include SANs with an OpenSSL config or CSR workflow:

```ini
# san.cnf
[req]
distinguished_name = dn
req_extensions = req_ext
prompt = no

[dn]
CN = www.example.com

[req_ext]
subjectAltName = @alt_names

[alt_names]
DNS.1 = www.example.com
DNS.2 = example.com
```

**Practical web hardening checklist**

Use layered controls rather than relying on a single feature.

- Enforce HTTPS with HSTS (<abbr title="HTTP Strict Transport Security">HSTS</abbr>) where appropriate.
- Redirect HTTP to HTTPS and remove mixed content.
- Validate input server-side and encode output contextually.
- Protect sessions with secure cookie flags and regeneration.
- Apply least privilege to application accounts and database users.
- Log authentication failures, privilege changes, and suspicious requests.
- Rate-limit login, password reset, and API endpoints.
- Keep frameworks, libraries, and reverse proxies patched.

Example Nginx hardening snippet:

```nginx
server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers off;

    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "no-referrer-when-downgrade" always;
    add_header Content-Security-Policy "default-src 'self'; frame-ancestors 'none'" always;
}
```

**Operational monitoring and incident response**

Security controls are only useful if you can detect abuse and respond quickly.

Monitor for:

- DNS query spikes, tunneling patterns, and unexpected `AXFR` attempts
- Mail authentication failures, spoofed domains, and unusual forwarding rules
- Web login anomalies, repeated 4xx/5xx errors, and suspicious parameter values
- Certificate expiration, chain failures, and unexpected issuance
- Changes to SPF, DKIM, DMARC, `CAA`, and DNSSEC-related records

Incident response priorities:

1. Contain the affected service or account.
2. Preserve logs and relevant DNS/mail/web artifacts.
3. Revoke or rotate compromised credentials and keys.
4. Correct the root cause and validate the fix.
5. Communicate with users, partners, and providers as needed.

**Quick comparison of key controls**

| Area | Primary goal | Common controls |
|---|---|---|
| DNS | Integrity and availability | DNSSEC, restricted recursion, zone transfer control, split-horizon governance |
| Email | Anti-spoofing and safe delivery | SPF, DKIM, DMARC, TLS submission, filtering, MFA |
| Web | Safe application behavior | Input validation, output encoding, secure sessions, CSP, TLS, least privilege |

**Exam focus reminders**

- DNSSEC protects integrity, not confidentiality.
- DMARC influences receiver handling; it does not guarantee rejection.
- SPF checks sending authorization, not message content.
- DKIM signs message content and selected headers, but it does not stop all spoofing by itself.
- Submission and retrieval should use TLS, and cleartext protocols should be avoided where possible.
- Web security is about both browser-side and server-side controls.
- Session management, CSRF protection, and output encoding are core controls, not optional extras.
