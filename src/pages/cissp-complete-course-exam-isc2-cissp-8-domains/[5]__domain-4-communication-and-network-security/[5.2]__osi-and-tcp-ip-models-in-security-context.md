## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Why models matter in security**

The OSI (Open Systems Interconnection) and TCP/IP models give you a shared language for describing where traffic flows, where controls apply, and where attacks occur. CISSP questions often test whether you can map a protocol, device, or control to the correct layer and then reason about the security impact.

The key exam skill is not memorizing a diagram in isolation. You need to understand how encapsulation, addressing, segmentation, and inspection change as data moves down and up the stack.

**OSI model overview**

The OSI model has seven layers:

1. Physical
2. Data Link
3. Network
4. Transport
5. Session
6. Presentation
7. Application

A practical way to use the model is to ask:
- What identifier is used at this layer?
- What device or control commonly operates here?
- What kind of attack or failure is easiest to associate with this layer?

**TCP/IP model overview**

The TCP/IP model is usually described as four layers:

- Network Access, also called Link or Network Interface
- Internet
- Transport
- Application

CISSP may use OSI and TCP/IP terms interchangeably, especially for the lower layers. For example, “network access” in TCP/IP often corresponds to OSI Layers 1 and 2, while “internet” corresponds to OSI Layer 3.

**Layer-by-layer security summary**

| OSI Layer | TCP/IP Layer | Main function | Common protocols / examples | Typical devices / controls | Security focus |
|---|---|---|---|---|---|
| 7 Application | Application | User-facing network services | HTTP, HTTPS, DNS, SMTP, FTP, SSH, SNMP, LDAP, RDP | WAF, proxy, secure gateways, application firewall | Authentication, input validation, content filtering, protocol abuse |
| 6 Presentation | Application | Data format, encoding, encryption, compression | TLS/SSL concepts, MIME, ASCII, UTF-8, JPEG, JSON | TLS termination, DLP, content inspection | Encryption, data representation, certificate handling |
| 5 Session | Application | Session setup, maintenance, teardown | RPC, NetBIOS session services, SIP session control | Session-aware gateways, SBCs | Session hijacking, timeout, reauthentication |
| 4 Transport | Transport | End-to-end delivery, ports, reliability | TCP, UDP, SCTP | L4 firewall, load balancer, NAT, stateful inspection | Port filtering, state tracking, SYN flood defense |
| 3 Network | Internet | Logical addressing and routing | IP, ICMP, IPsec, routing protocols | Router, L3 firewall, ACLs, VPN gateway | Subnet isolation, route control, packet filtering |
| 2 Data Link | Network Access | Frames, MAC addressing, local delivery | Ethernet, 802.1Q, ARP, STP, PPP, Wi-Fi MAC | Switch, bridge, wireless AP, port security, 802.1X, DHCP snooping | VLAN segmentation, MAC filtering, spoofing defense |
| 1 Physical | Network Access | Bits on the medium | Copper, fiber, radio, connectors, signaling | Cables, hubs, repeaters, media converters | Physical access, tamper resistance, EMI, link availability |

> ARP is often taught as a “Layer 2.5” mnemonic because it bridges IP and MAC resolution, but that is not a real OSI layer. Treat it as a convenience for remembering behavior, not as a formal classification.

**Layer 1: Physical**

Layer 1 concerns the transmission of raw bits over a medium. It includes cabling, signaling, voltage, light pulses, radio frequency, connectors, and physical topology.

Security at this layer is mostly about preventing disruption and unauthorized access to the medium.

- Protect wiring closets, patch panels, and fiber runs.
- Use locked racks, surveillance, and environmental controls.
- Reduce electromagnetic interference and signal leakage where relevant.
- Monitor for cable cuts, rogue devices, and physical taps.

Common Layer 1 issues include:
- Broken cables
- Loose connectors
- Power loss
- Interference
- Attenuation
- Hardware failure

Common Layer 1 devices include:
- Repeaters
- Hubs
- Media converters
- Physical-layer transceivers

**Layer 1 troubleshooting and domains**

Layer 1 problems often affect the entire physical segment.

- A **collision domain** is the area where frames can collide on a shared medium. In modern switched Ethernet, collisions are largely eliminated by full-duplex links, but the concept still appears in exam questions.
- A **broadcast domain** is not a Layer 1 concept; it is primarily a Layer 2 concept. Keep that distinction clear.
- A **failure domain** is the scope of impact when a component fails. Redundancy, segmentation, and diverse paths reduce failure domain size.

**Layer 2: Data Link**

Layer 2 handles framing, MAC (Media Access Control) addressing, local delivery, and error detection on the local network segment. Switches operate here, forwarding frames based on MAC addresses.

Security controls at Layer 2 are common in enterprise networks because this layer is where local spoofing, rogue access, and VLAN abuse often occur.

Key Layer 2 concepts:
- Frames use MAC addresses.
- Switches learn MAC tables dynamically.
- VLANs (Virtual LANs) segment broadcast domains.
- Trunk links carry multiple VLANs using 802.1Q tagging.
- STP (Spanning Tree Protocol) prevents loops in redundant switched topologies.

Common Layer 2 security controls:
- **Port security**: limits which MAC addresses can use a switch port.
- **802.1X**: port-based network access control using authentication before network access is granted.
- **DHCP snooping**: blocks rogue DHCP servers and builds trusted bindings.
- **Dynamic ARP Inspection**: validates ARP traffic against trusted bindings.
- **IP Source Guard**: restricts IP traffic on a port based on learned bindings.
- **VLAN segmentation**: separates traffic into distinct broadcast domains.

Practical switch examples:

```cisco
interface GigabitEthernet1/0/10
 switchport mode access
 switchport access vlan 20
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
 spanning-tree portfast
 spanning-tree bpduguard enable
```

```cisco
ip dhcp snooping
ip dhcp snooping vlan 10,20
interface GigabitEthernet1/0/1
 ip dhcp snooping trust
```

```cisco
interface GigabitEthernet1/0/10
 ip verify source
```

Important Layer 2 attacks and issues:
- **MAC spoofing**: attacker changes a MAC address to impersonate a host.
- **ARP spoofing / ARP poisoning**: attacker sends false ARP replies to redirect traffic.
- **VLAN hopping**: attacker abuses trunking or switch behavior to reach another VLAN.
- **STP manipulation**: attacker attempts to become root bridge or disrupt topology.
- **Rogue DHCP**: attacker provides malicious gateway or DNS settings.

**Layer 2 vs Layer 3 security devices**

A switch is primarily a Layer 2 device, while a router is primarily a Layer 3 device. CISSP questions often ask you to distinguish them by what they inspect and what they separate.

- A **Layer 2 switch** forwards frames within a broadcast domain and can segment traffic with VLANs.
- A **Layer 3 router** forwards packets between networks and separates broadcast domains.
- A **Layer 2 firewall** or transparent firewall can inspect frames or bridge traffic without changing the IP topology.
- A **Layer 3 firewall** routes between networks and filters based on IP addresses and other packet attributes.
- A **bridge firewall** sits inline at Layer 2 and is often used when you want security controls without redesigning IP addressing.

Transparent or bridge firewalls are useful when:
- You need to insert security without changing subnet design.
- You want to inspect traffic between two segments while preserving existing routing.
- You need a stealth deployment or minimal topology change.

**Layer 3: Network**

Layer 3 provides logical addressing and routing between networks. IP (Internet Protocol) is the core protocol here. Routers make forwarding decisions based on destination IP addresses and routing tables.

Security at Layer 3 focuses on controlling which networks can talk to each other and how packets move across boundaries.

Common Layer 3 controls:
- **ACLs (Access Control Lists)** on routers and firewalls
- **Routing policy** and route filtering
- **Subnet segmentation**
- **IPsec** for confidentiality, integrity, and authentication at the IP layer
- **NAT** as an address translation mechanism that can also reduce direct exposure

A simple router ACL example:

```cisco
ip access-list extended BLOCK_FINANCE_TO_GUEST
 deny ip 10.10.20.0 0.0.0.255 10.10.99.0 0.0.0.255
 permit ip any any
```

This is a Layer 3/4-style control when it matches ports or protocols beyond pure IP addresses. In exam terms, be precise about whether the rule filters only IP networks or also transport-layer fields.

Common Layer 3 attacks and issues:
- **IP spoofing**: attacker forges source IP addresses.
- **Route injection**: attacker advertises false routes.
- **ICMP abuse**: reconnaissance, redirect abuse, or flooding.
- **Subnet scanning**: attacker maps reachable hosts across routed boundaries.
- **Man-in-the-middle**: often enabled by routing manipulation, ARP poisoning, or rogue gateways.

**Layer 4: Transport**

Layer 4 provides end-to-end communication between processes using port numbers. TCP (Transmission Control Protocol) and UDP (User Datagram Protocol) are the most important protocols here.

TCP is connection-oriented and reliable. UDP is connectionless and lightweight. CISSP often tests when each is used and what security implications follow.

Common mappings:
- **HTTPS** uses TCP 443 and typically includes TLS for encryption.
- **SMTP** uses TCP 25, 587, or 465 depending on the service and security mode.
- **DNS** commonly uses UDP 53, but also uses TCP 53 for zone transfers and large responses.
- **SSH** uses TCP 22.
- **RDP** uses TCP 3389.

Transport-layer security controls:
- Stateful firewalls
- Port filtering
- Load balancers
- SYN flood protection
- Session tracking
- Rate limiting

Example of a stateful firewall rule conceptually:
- Allow inbound TCP 443 to a web server.
- Track the connection state.
- Deny unsolicited inbound traffic to other ports.

Common Layer 4 attacks:
- **SYN flood**: attacker exhausts TCP connection state.
- **UDP flood**: attacker overwhelms a service with datagrams.
- **Port scanning**: attacker enumerates open services.
- **Session exhaustion**: attacker consumes connection tables or resources.

Practical Linux inspection commands:

```bash
ss -tulpen
ip addr show
ip route show
```

For packet capture and transport troubleshooting:

```bash
tcpdump -i eth0 tcp port 443
tcpdump -i eth0 udp port 53
```

**Layer 5: Session**

Layer 5 manages session establishment, maintenance, and teardown. In real networks, session functions are often blended into application protocols or handled by libraries and middleware rather than a distinct OSI layer implementation.

Examples often associated with session behavior:
- RPC (Remote Procedure Call)
- NetBIOS session services
- SIP (Session Initiation Protocol) in voice and video systems
- Application login sessions and token lifetimes

Security concerns at this layer include:
- Session hijacking
- Session fixation
- Replay of session tokens
- Weak timeout handling
- Inadequate reauthentication after privilege changes

Practical controls:
- Short idle timeouts for sensitive systems
- Reauthentication for high-risk actions
- Secure cookies and token binding where supported
- Mutual authentication for sensitive session establishment

**Layer 6: Presentation**

Layer 6 handles data representation, translation, compression, and encryption. This layer is often discussed in CISSP as the place where data is transformed into a format the application can use.

Important caution:
- TLS (Transport Layer Security) is often mapped to the Presentation layer in teaching diagrams because it provides encryption and data protection.
- In practice, TLS sits between application and transport and does not cleanly belong to a single OSI layer.
- Do not overstate the OSI mapping; use it as a study aid.

Common Presentation-layer concerns:
- Character encoding such as ASCII and UTF-8
- Data serialization such as JSON or XML
- Compression
- Encryption and decryption
- Certificate handling and trust validation

Security relevance:
- Data format mismatches can create parsing vulnerabilities.
- Weak certificate validation can enable interception.
- Compression and encryption choices can affect traffic analysis and side-channel exposure.

**Layer 7: Application**

Layer 7 is where user-facing network services operate. This is the layer most people think about first because it includes web, email, name resolution, directory services, and remote access applications.

Common protocols:
- HTTP and HTTPS
- DNS
- SMTP, IMAP, POP3
- FTP and SFTP
- SSH
- SNMP
- LDAP
- RDP
- SIP

Security controls at Layer 7:
- Web application firewalls
- Reverse proxies
- Authentication and authorization
- Input validation
- Content filtering
- API gateways
- Secure coding and protocol hardening

Common Layer 7 attacks:
- Phishing
- Credential stuffing
- SQL injection
- Cross-site scripting
- Command injection
- DNS poisoning
- Email spoofing
- Application-layer denial of service

Protocol edge cases CISSP may test:
- **HTTPS** is HTTP over TCP with TLS protection.
- **SMTP** runs over TCP, not UDP.
- **DNS** usually uses UDP for queries but can use TCP for larger responses and zone transfers.
- **TLS** protects application traffic but is not itself an application protocol.
- **SSH** is an application-layer remote administration protocol that uses TCP.

**Attack-to-layer mapping**

Use the layer model to reason about where an attack starts and what control can stop it.

- **Phishing**: primarily Layer 7, because it targets users and application trust.
- **Man-in-the-middle**: often Layer 2, 3, or 7 depending on the method.
  - Layer 2 example: ARP spoofing
  - Layer 3 example: rogue routing or gateway manipulation
  - Layer 7 example: proxying or certificate abuse
- **DNS poisoning**: Layer 7, with transport and network implications.
- **ARP spoofing**: Layer 2.
- **SYN flood**: Layer 4.
- **VLAN hopping**: Layer 2.
- **Port scanning**: Layer 4, sometimes Layer 7 if application probing is involved.
- **Rogue DHCP**: Layer 2.
- **IP spoofing**: Layer 3.
- **TLS downgrade or certificate abuse**: Layer 6/7 boundary, depending on the scenario.

**Broadcast, collision, and failure domains**

These concepts are common in design and troubleshooting questions.

- A **broadcast domain** is the set of devices that receive Layer 2 broadcast frames.
- VLANs reduce broadcast domain size.
- Routers separate broadcast domains.
- A **collision domain** is where simultaneous transmissions can collide on a shared medium.
- Switch ports typically create separate collision domains.
- A **failure domain** is the scope of impact from a fault or outage.

Why this matters:
- Smaller broadcast domains reduce unnecessary traffic and limit some attack surfaces.
- Smaller failure domains improve resilience.
- Segmentation can also improve monitoring and policy enforcement.

**Encapsulation and decapsulation**

Encapsulation is the process of adding headers and trailers as data moves down the stack. Decapsulation removes them as data moves up the stack.

A simple flow looks like this:

```text
Application data
  -> TCP segment
  -> IP packet
  -> Ethernet frame
  -> bits on the wire
```

Security implications:
- Lower-layer headers can be hidden or altered by tunnels.
- Middleboxes may only inspect what they can see.
- Encryption can protect payloads but also reduce visibility for defenders.

**Tunneling and VPNs**

Tunneling wraps one protocol inside another. This can be used for privacy, connectivity, or bypassing network restrictions.

Common examples:
- GRE (Generic Routing Encapsulation)
- IPsec tunnel mode
- SSL/TLS VPNs
- Site-to-site VPNs
- SSH tunnels

Why tunneling matters in security:
- It can protect confidentiality over untrusted networks.
- It can obscure inner traffic from lower-layer inspection.
- It can complicate logging, IDS (Intrusion Detection System) visibility, and policy enforcement.

Example IPsec concept:
- In tunnel mode, the original IP packet is encapsulated inside a new packet.
- The outer packet is routed across the network.
- The inner packet remains protected from intermediate devices.

**Practical Linux and network troubleshooting**

Useful commands for mapping traffic and layers:

```bash
ip link show
ip addr show
ip neigh
ss -tulpen
ping -c 4 8.8.8.8
traceroute 8.8.8.8
tcpdump -i eth0 -nn
```

What each helps you see:
- `ip link show` reveals interfaces and link state.
- `ip addr show` shows IP addressing.
- `ip neigh` shows neighbor cache entries, including ARP/ND state.
- `ss -tulpen` shows listening sockets and transport ports.
- `ping` tests reachability and ICMP handling.
- `traceroute` helps identify routing hops.
- `tcpdump` captures packets for layer-by-layer analysis.

On Linux, prefer `ip neigh` over legacy `arp -a` when checking neighbor resolution.

**Exam-style distinctions to remember**

- A switch forwards frames using MAC addresses; a router forwards packets using IP addresses.
- Port security is a Layer 2 switch feature, not a Layer 1 control.
- A firewall can operate at multiple layers depending on design.
- A transparent firewall bridges traffic; a routed firewall changes the IP path.
- ARP is not a formal OSI layer; it is a Layer 2 protocol used to map IP to MAC on local networks.
- TLS is commonly associated with OSI Layer 6 in study diagrams, but in practice it spans application and transport concerns.
- DNS, SMTP, and HTTPS are application-layer services even though they rely on transport-layer ports.

**Quick review checklist**

- Know the seven OSI layers in order.
- Know the four TCP/IP layers and how they map to OSI.
- Distinguish Layer 2 switching from Layer 3 routing.
- Recognize common security controls at each layer.
- Map attacks to the most likely layer.
- Understand encapsulation, tunneling, and where inspection becomes harder.
- Remember broadcast domains, collision domains, and failure domains.
- Use precise protocol mappings for DNS, SMTP, HTTPS, and TLS.

**High-yield summary**

The OSI model is a diagnostic and conceptual tool; the TCP/IP model is the practical Internet stack. CISSP expects you to use both to reason about controls, attacks, and troubleshooting. When you can identify the layer, you can usually identify the right device, the right defense, and the right failure mode.
