## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Scope and exam focus**

Secure routing, switching, and wireless design is about reducing attack surface while preserving availability and operational control. For CISSP, you need to understand how Layer 2 and Layer 3 controls interact with authentication, segmentation, management-plane security, and wireless encryption/authentication choices.

This topic is not just about memorizing protocols. You should be able to choose controls for a campus, branch, data center, or guest network and explain the tradeoffs between security, performance, and manageability.

**Secure switching fundamentals**

Switches forward frames at Layer 2 using MAC addresses, but security problems often arise from how they learn, segment, and trust traffic. A secure design limits broadcast domains, prevents unauthorized trunking, and reduces the impact of spoofing or lateral movement.

- **Port security** limits which MAC addresses can use a switch port.
  - Common actions include shutdown, restrict, or protect when violations occur.
  - It helps against casual rogue device attachment, but it is not a substitute for authentication.
- **802.1X** provides port-based network access control.
  - The switch acts as the authenticator.
  - The endpoint is the supplicant.
  - The authentication server is usually RADIUS.
- **MAC Authentication Bypass (MAB)** is a fallback for devices that cannot do 802.1X, such as some printers or IoT devices.
  - Use it carefully because MAC addresses are easy to spoof.
- **Storm control** limits broadcast, multicast, or unknown unicast traffic.
  - It helps contain loops and denial-of-service conditions on access ports.
- **BPDU Guard** protects edge ports from receiving Bridge Protocol Data Units (BPDUs).
  - It is commonly paired with PortFast on access ports.
- **Root Guard** prevents an access or distribution port from becoming the Spanning Tree root path.
- **Loop Guard** helps prevent alternate ports from incorrectly transitioning to forwarding if BPDUs stop arriving.

A secure access-layer design usually assumes hostile endpoints. You harden the edge, authenticate users and devices, and keep Layer 2 trust as small as possible.

**VLANs, trunks, and segmentation controls**

Virtual LANs (VLANs) separate broadcast domains and support logical segmentation. They are a core control for isolating users, servers, voice, guest, and IoT traffic.

- **Access ports** carry one VLAN and should be used for end devices.
- **Trunk ports** carry multiple VLANs and should be explicitly configured only where needed.
- **Native VLAN** is the untagged VLAN on 802.1Q trunks.
  - Mismatched native VLANs can create leakage or confusion.
  - Best practice is to set the native VLAN to an unused VLAN and tag native traffic where supported.
- **Dynamic Trunking Protocol (DTP)** can negotiate trunking on some platforms.
  - Disable DTP on access ports and hard-set trunk behavior on uplinks.
- **Allowed VLAN lists** restrict which VLANs may traverse a trunk.
  - This reduces blast radius if a trunk is compromised or misconfigured.
- **VLAN hopping** is a classic Layer 2 attack.
  - Switch spoofing abuses trunk negotiation.
  - Double-tagging can exploit native VLAN handling in some topologies.
  - Hardening trunks and avoiding VLAN 1 for user traffic reduce risk.

Example hardening on a Cisco-like switch:

```bash
interface GigabitEthernet1/0/10
 description User access port
 switchport mode access
 switchport access vlan 20
 switchport nonegotiate
 spanning-tree portfast
 spanning-tree bpduguard enable
 switchport port-security
 switchport port-security maximum 2
 switchport port-security violation restrict
 switchport port-security mac-address sticky
```

Example trunk hardening:

```bash
interface GigabitEthernet1/0/48
 description Uplink to distribution switch
 switchport mode trunk
 switchport trunk native vlan 999
 switchport trunk allowed vlan 10,20,30,40
 switchport nonegotiate
 spanning-tree guard root
```

Practical design guidance:

- Use a dedicated unused native VLAN such as `999`.
- Prune unused VLANs from trunks.
- Avoid carrying management traffic on the same VLAN as user endpoints.
- Do not rely on VLANs alone for strong security; combine them with ACLs, 802.1X, and firewall policy.

**Private VLANs and protected ports**

Private VLANs (PVLANs) provide finer-grained Layer 2 isolation inside the same IP subnet or VLAN structure. They are useful when many hosts must reach a gateway or shared service but should not talk directly to each other.

- **Primary VLAN** is the parent VLAN.
- **Secondary VLANs** are isolated or community VLANs.
  - **Isolated** ports cannot talk to other isolated ports.
  - **Community** ports can talk within the same community but not outside it.
- **Promiscuous ports** can communicate with all secondary VLANs and are often used for gateways, firewalls, or shared services.

Protected ports are a simpler access-layer control that prevents ports in the same VLAN from communicating with each other through the switch. They are useful for guest rooms, labs, or shared workspaces where east-west traffic should be blocked at Layer 2.

Use these controls when you want segmentation without creating many separate VLANs or when you need to reduce lateral movement inside a flat subnet.

**Switch management-plane security**

The management plane is often the easiest path to full device compromise, so secure it separately from user traffic.

- Use **SSH** instead of Telnet for CLI access.
- Use **HTTPS** instead of HTTP for web management.
- Use **SNMPv3** instead of SNMPv1 or SNMPv2c.
  - SNMPv3 supports authentication and privacy.
  - Avoid community strings in production.
- Use **AAA** (Authentication, Authorization, and Accounting) with RADIUS or TACACS+.
  - RADIUS is common for network access and centralized authentication.
  - TACACS+ is often preferred for device administration because it separates authentication, authorization, and accounting more cleanly.
- Restrict management access with ACLs or a dedicated management VRF/VLAN.
- Disable unused services such as Telnet, HTTP, CDP/LLDP where not needed, and legacy discovery or remote management features.
- Use strong local fallback credentials and protect them with role-based access control.

Example management hardening:

```bash
ip ssh version 2
no ip http server
ip http secure-server
snmp-server group NMS v3 priv
snmp-server user secops NMS v3 auth sha AuthPass123 priv aes 128 PrivPass123
aaa new-model
aaa authentication login default group tacacs+ local
line vty 0 4
 transport input ssh
 access-class MGMT-ONLY in
```

Operationally, you should separate management access from production data paths whenever possible. A dedicated out-of-band management network is ideal for critical infrastructure.

**Routing security fundamentals**

Routing protocols exchange reachability information, so attackers who can inject or alter routing updates can redirect, blackhole, or intercept traffic. Secure routing focuses on authentication, adjacency control, route filtering, and limiting who can participate.

- **Static routes** are simple and predictable.
  - They are useful for small networks, default routes, and tightly controlled paths.
  - They do not scale well in dynamic environments.
- **Dynamic routing protocols** adapt to topology changes.
  - They require stronger trust controls because neighbors can influence forwarding decisions.
- **Route filtering** limits which prefixes are accepted or advertised.
  - Prefix lists, route maps, distribute lists, and policy controls help prevent route leaks.
- **Authentication** protects routing adjacencies from unauthorized peers.
  - It does not replace filtering or topology design.

A secure design often combines protocol authentication with strict neighbor definitions, passive interfaces where appropriate, and route policy at boundaries.

**OSPF security**

Open Shortest Path First (OSPF) is a link-state routing protocol commonly used in enterprise networks. It forms adjacencies with neighbors and floods link-state information within an area.

Security considerations:

- Use authentication on OSPF adjacencies where supported.
- Limit OSPF to trusted interfaces.
- Make user-facing interfaces passive when they should not form neighbors.
- Filter redistributed routes carefully to avoid accidental route injection.
- Protect area boundaries and summarize routes where appropriate.

Example OSPF hardening:

```bash
router ospf 10
 router-id 10.10.10.10
 passive-interface default
 no passive-interface GigabitEthernet1/0/1
 area 0 authentication message-digest
 network 10.10.0.0 0.0.255.255 area 0
```

OSPF authentication has historically used simple or MD5-based mechanisms, but modern environments increasingly prefer stronger operational controls and platform-supported integrity features where available. Authentication helps prevent casual spoofing, but you still need ACLs, interface filtering, and careful redistribution policy.

**EIGRP and IS-IS security**

Enhanced Interior Gateway Routing Protocol (EIGRP) and Intermediate System to Intermediate System (IS-IS) are less common in some enterprise environments than OSPF, but CISSP candidates should understand their security posture.

- **EIGRP** is an advanced distance-vector protocol.
  - It supports neighbor authentication on many platforms.
  - It is sensitive to unauthorized neighbors because adjacency formation can influence route selection quickly.
  - Use passive interfaces, authentication, and route filtering.
- **IS-IS** is a link-state protocol often used in service provider and large-scale networks.
  - It runs directly over Layer 2 rather than TCP or UDP.
  - Authentication is important because adjacency formation can occur on shared media.
  - Use adjacency control, authentication, and strict interface scoping.

Comparison guidance:

- OSPF and IS-IS are both link-state, so route database integrity matters.
- EIGRP is more sensitive to neighbor trust because it exchanges routing information directly with peers and can converge quickly.
- In all cases, authentication alone is not enough; you also need topology design and policy controls.

**BGP security and route integrity**

Border Gateway Protocol (BGP) is the interdomain routing protocol used between autonomous systems. It is highly policy-driven and a common target for route hijacking and route leaks.

Key controls:

- Use neighbor authentication where supported.
- Restrict peers to known IP addresses.
- Apply prefix filters and maximum-prefix limits.
- Use route maps or policy statements to control import and export.
- Monitor for unexpected AS paths, more-specific prefixes, and default route anomalies.
- Consider TTL security and session protection features where available.

Example BGP policy controls:

```bash
router bgp 65001
 neighbor 203.0.113.2 remote-as 65002
 neighbor 203.0.113.2 password 7 030752180500
 neighbor 203.0.113.2 maximum-prefix 1000 90 restart 5
 !
 ip prefix-list OUTBOUND seq 10 permit 10.10.0.0/16 le 24
 route-map EXPORT permit 10
  match ip address prefix-list OUTBOUND
```

Operationally, BGP security is about reducing trust and validating intent. You should assume peers can misadvertise routes accidentally or maliciously. Filtering and monitoring are essential because authentication only proves the peer is the expected neighbor, not that the routes are correct.

**Route authentication, MD5, and TCP-AO**

Route authentication mechanisms protect the control plane from spoofed or tampered routing sessions.

- **MD5-based authentication** has been widely used for routing protocols.
  - It is still encountered in legacy deployments.
  - It is operationally simple but increasingly considered weak or legacy in many environments.
- **TCP Authentication Option (TCP-AO)** is a newer mechanism for TCP-based sessions such as BGP.
  - It improves key management and supports stronger operational models than older TCP MD5 approaches.
  - Adoption depends on platform support and interoperability.
- Authentication does not encrypt routing updates.
  - It protects integrity and peer trust, not confidentiality.
  - An attacker on-path may still observe metadata unless additional protections exist.

Use stronger mechanisms where supported, but remember that routing security is layered: authentication, filtering, adjacency control, and monitoring all matter.

**Wireless fundamentals and spectrum**

Wireless networks introduce shared-medium risks because radio signals extend beyond physical walls and can be intercepted or interfered with. Security depends on encryption, authentication, RF planning, and careful segmentation.

Important bands and characteristics:

- **2.4 GHz**
  - Longer range and better wall penetration.
  - More interference from Bluetooth, microwaves, and neighboring Wi-Fi.
  - Fewer non-overlapping channels.
- **5 GHz**
  - More channels and typically less congestion.
  - Shorter range than 2.4 GHz.
- **6 GHz**
  - Used by Wi-Fi 6E and newer deployments.
  - Offers more spectrum and less legacy interference.
  - Requires compatible clients and infrastructure.

Wireless security is not only about encryption. You also need to control who can join, where they land on the network, and how far the signal reaches outside the facility.

**Wi-Fi standards and security features**

The IEEE 802.11 family defines wireless LAN behavior, while Wi-Fi Alliance security branding describes practical security modes.

Key security-relevant generations and features:

- **WEP** (Wired Equivalent Privacy)
  - Broken and obsolete.
  - Never use it.
- **WPA** (Wi-Fi Protected Access)
  - Transitional improvement over WEP.
  - Often associated with TKIP, which is also obsolete.
- **WPA2**
  - Uses AES-CCMP in secure deployments.
  - Still common and acceptable when configured correctly.
- **WPA3-Personal**
  - Uses Simultaneous Authentication of Equals (SAE) instead of PSK-only handshake behavior.
  - Better resists offline dictionary attacks than WPA2-Personal.
- **WPA3-Enterprise**
  - Uses enterprise authentication with 802.1X and RADIUS.
  - Can support stronger cipher suites and higher security modes depending on implementation.
- **Protected Management Frames (PMF)**, also known as **802.11w**
  - Protects management frames from spoofing and deauthentication attacks.
  - Required in WPA3 and strongly recommended in modern WPA2 enterprise deployments.

Practical differences:

- **WPA2-AES** is secure when you use strong credentials and proper configuration, but WPA2-Personal with weak passwords remains vulnerable to offline guessing.
- **WPA3-Personal** improves password-based security by using SAE, which makes captured handshakes less useful for offline cracking.
- **WPA3-Enterprise** is the preferred choice for managed corporate environments because it combines centralized identity with stronger wireless protections.
- Open networks provide no confidentiality and should be treated as hostile unless additional application-layer protections exist.

> Avoid legacy modes such as WEP, WPA-TKIP, and mixed compatibility settings unless you have a documented business requirement and compensating controls.

**Wireless authentication and enterprise design**

Enterprise wireless should use centralized identity and policy enforcement.

- **802.1X** is the standard framework for port-based access control on wired and wireless networks.
- **EAP** (Extensible Authentication Protocol) is the authentication framework carried through 802.1X.
- Common EAP methods include:
  - **EAP-TLS** for certificate-based mutual authentication.
  - **PEAP** for tunneled authentication with username/password inside a protected tunnel.
  - **EAP-TTLS** in some environments for similar tunneled designs.

Best practice:

- Prefer **EAP-TLS** when you can manage certificates at scale.
- Use a RADIUS backend for policy, accounting, and dynamic authorization.
- Assign users and devices to different VLANs or roles based on identity, posture, or device type.
- Use dynamic VLAN assignment or downloadable ACLs where supported.

Example wireless enterprise flow:

1. The client associates to the SSID.
2. The AP forwards 802.1X authentication to RADIUS.
3. The user or device authenticates with EAP.
4. The network assigns a role, VLAN, or ACL based on policy.
5. The client receives only the access it needs.

**Guest access, captive portals, and IoT isolation**

Guest and IoT networks should be isolated by design because they are less trusted than managed corporate endpoints.

- **Guest access**
  - Place guests in a separate VLAN or VRF.
  - Use firewall rules to allow only internet access.
  - Block access to internal subnets, management interfaces, and east-west guest traffic where possible.
  - Captive portals can provide acceptable-use notice, voucher access, or self-registration.
- **IoT networks**
  - Use separate SSIDs, VLANs, or microsegmentation.
  - Restrict outbound destinations to required services only.
  - Prefer device certificates or pre-shared keys with rotation if 802.1X is not possible.
- **Captive portals**
  - Useful for guest onboarding and policy acknowledgment.
  - They are not strong authentication by themselves.
  - Do not treat portal acceptance as a security boundary.

Example guest isolation policy:

```bash
interface Vlan200
 description Guest network
 ip address 192.0.2.1 255.255.255.0
 ip access-group GUEST-IN in
!
ip access-list extended GUEST-IN
 deny ip 192.0.2.0 0.0.0.255 10.0.0.0 0.255.255.255
 deny ip 192.0.2.0 0.0.0.255 172.16.0.0 0.15.255.255
 deny ip 192.0.2.0 0.0.0.255 192.168.0.0 0.0.255.255
 permit ip 192.0.2.0 0.0.0.255 any
```

A secure guest design assumes the guest device may be compromised. Your goal is containment, not trust.

**Wireless weaknesses and legacy risks**

Wireless attacks often exploit weak credentials, poor segmentation, or user behavior rather than cryptography alone.

- **Open networks** expose traffic to passive interception and active manipulation.
- **WPA2-Personal with weak passwords** is vulnerable to offline password guessing after handshake capture.
- **WPS** (Wi-Fi Protected Setup) can weaken security if enabled, especially PIN-based methods.
- **Rogue access points** can create unauthorized entry points.
- **Evil twin** attacks impersonate legitimate SSIDs to capture credentials or redirect users.
- **Deauthentication attacks** are mitigated by PMF/802.11w.
- **Rogue clients** can bridge trusted and untrusted networks if endpoint controls are weak.

Mitigations include strong authentication, PMF, wireless intrusion detection or prevention, certificate-based authentication, and user awareness.

**Wireless site surveys and validation**

A wireless deployment should be validated before and after rollout. Site surveys help you place APs, tune power, and detect coverage gaps or interference.

- **Predictive survey**
  - Uses floor plans and modeling software before installation.
  - Helps estimate AP count and placement.
  - Useful for planning and budgeting.
- **Passive survey**
  - Measures existing RF conditions without associating to the network.
  - Useful for detecting interference, rogue APs, and signal quality.
- **Active survey**
  - Connects to the WLAN and measures real performance.
  - Useful for validating throughput, roaming, and authentication behavior.

Use surveys to confirm that signal does not leak excessively outside secure areas and that coverage supports intended roaming without overprovisioning power.

**Wireless hardening checklist**

- Use WPA3-Enterprise where possible.
- Use WPA3-Personal only when enterprise authentication is not feasible.
- Require PMF where supported.
- Disable WEP, WPA-TKIP, and WPS.
- Separate guest, corporate, and IoT SSIDs.
- Map SSIDs to distinct VLANs or roles.
- Use strong RADIUS policies and certificate-based authentication for managed devices.
- Limit AP management access and keep firmware current.
- Monitor for rogue APs and evil twins.
- Tune transmit power and channel plans to reduce leakage and interference.

**Scenario-based selection guidance**

Choose controls based on the environment rather than defaulting to one technology everywhere.

- **Small office with a few managed laptops**
  - Use WPA3-Personal only if centralized identity is not practical.
  - Prefer strong passphrases and PMF.
  - Segment guest access separately.
- **Enterprise campus**
  - Use WPA3-Enterprise with 802.1X and RADIUS.
  - Use dynamic VLANs or ACLs for role-based access.
  - Harden trunks, management access, and switch ports.
- **Hospital or manufacturing floor**
  - Use 802.1X where possible, but allow MAB for legacy devices with strict segmentation.
  - Place IoT devices in isolated VLANs or PVLANs.
  - Restrict east-west traffic and monitor for anomalies.
- **Branch office with limited IT staff**
  - Use centralized templates, secure management protocols, and simple but enforceable routing policy.
  - Keep trunking minimal and document allowed VLANs.
  - Use site surveys to avoid over-coverage and signal bleed.

**Quick comparison table**

| Area | Preferred control | Why it matters |
|---|---|---|
| Wireless personal access | WPA3-Personal | Better resistance to offline guessing than WPA2-Personal |
| Wireless enterprise access | WPA3-Enterprise + 802.1X | Centralized identity and stronger policy control |
| Management access | SSH, HTTPS, SNMPv3 | Protects the control plane from insecure admin traffic |
| Switch edge ports | 802.1X, port security, BPDU Guard | Reduces rogue device and loop risk |
| Trunks | Explicit allowed VLANs, no DTP, hardened native VLAN | Prevents VLAN hopping and accidental exposure |
| Routing peers | Authentication, filtering, passive interfaces | Limits unauthorized route injection |
| Guest networks | Separate VLAN/VRF + firewall | Contains untrusted devices and traffic |

**Key exam takeaways**

- Secure routing is about trust, filtering, and adjacency control, not just protocol selection.
- Secure switching requires hardening access ports, trunks, and the management plane.
- VLANs provide segmentation, but they are not a complete security boundary by themselves.
- Private VLANs and protected ports reduce lateral movement in shared Layer 2 environments.
- WPA3 improves wireless security, especially with SAE and PMF.
- WPA3-Enterprise is the strongest common enterprise wireless choice when supported.
- Guest and IoT networks should be isolated and treated as untrusted.
- Site surveys help you validate both coverage and security assumptions.
