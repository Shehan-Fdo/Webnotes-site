## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Cloud networking security fundamentals**

Cloud networking extends traditional network security into provider-managed environments where you control policy, routing, identity, and exposure, but not all underlying infrastructure. The core exam idea is to separate **data plane** traffic from **control plane** access and to understand which protections apply at each layer.

- **Data plane** traffic is the application and workload traffic that moves between instances, services, subnets, regions, and on-premises networks.
- **Control plane** access is how you manage cloud resources through APIs, consoles, and automation tools.
- Cloud security design must protect both planes differently:
  - Data plane controls include security groups, network ACLs, firewalls, WAFs, routing, and encryption.
  - Control plane controls include IAM, MFA, conditional access, API restrictions, logging, and administrative network restrictions.
- Cloud networking is usually built around shared responsibility:
  - The provider secures the physical infrastructure and core services.
  - You secure configuration, identity, segmentation, encryption choices, and exposure.

**Cloud-native network boundaries and segmentation**

Cloud environments use multiple layers of network control, and the exam often tests where each one sits and what it can and cannot do. A strong design uses layered controls rather than relying on a single perimeter.

- **Security groups** are stateful, instance- or interface-level filters used to allow traffic to workloads.
- **Network ACLs** are stateless subnet-level filters that can provide coarse allow/deny control.
- **Provider network firewalls** sit at a higher network layer and can inspect and enforce policy across VPCs/VNets, subnets, or transit paths.
- **Host firewalls** run on the workload itself and protect the operating system regardless of network location.
- **Web application firewalls (WAFs)** protect HTTP/S applications from application-layer attacks such as injection, cross-site scripting, and malicious bots.
- These controls complement each other:
  - Security groups and ACLs handle basic reachability.
  - Provider firewalls add centralized inspection and policy.
  - Host firewalls enforce local least privilege.
  - WAFs protect the application layer, not generic network traffic.

A practical pattern is to place a WAF in front of public web apps, a provider firewall or transit inspection layer between trust zones, and host firewalls on every server. This reduces reliance on any single control and improves defense in depth.

**Cloud routing and path control**

Routing in cloud environments is more dynamic than in traditional networks. You must understand how route tables, propagation, and route precedence affect traffic flow, especially in hybrid designs.

- Most clouds associate a **route table per subnet** or equivalent routing domain.
- Route tables determine the next hop for traffic leaving a subnet.
- Route propagation can automatically inject routes from gateways, VPNs, or transit hubs into route tables.
- Route propagation should be controlled carefully to avoid unintended reachability between environments.
- More specific routes usually override less specific routes, so a `/24` route can take precedence over a broader `/16` route.
- Static routes often override propagated routes in many designs, but exact precedence depends on the provider and route type.
- Default routes such as `0.0.0.0/0` or `::/0` are common for internet egress and must be tightly controlled.

A common exam scenario is asymmetric routing. This happens when traffic enters through one path and returns through another, which can break stateful firewalls, load balancers, or inspection appliances.

- Symptoms of asymmetric routing:
  - Connections establish intermittently or fail after SYN/SYN-ACK.
  - Firewall logs show only one direction of the flow.
  - Return traffic bypasses the inspection device.
- Common causes:
  - Multiple route tables with inconsistent next hops.
  - Overlapping routes from VPN, transit gateway, or peering.
  - Load balancers or NAT devices placed on only one side of the path.
- Troubleshooting approach:
  1. Trace the forward path from source to destination.
  2. Trace the return path separately.
  3. Compare route tables, propagated routes, and appliance insertion points.
  4. Verify that both directions traverse the same stateful control or that the control is designed for asymmetric flows.

**Hybrid connectivity patterns**

Hybrid connectivity links cloud networks to on-premises environments through private or semi-private paths. The main exam distinction is between internet-based encrypted tunnels and dedicated private circuits.

- **Site-to-site VPN** uses encrypted tunnels over the public internet.
  - It is fast to deploy and useful for smaller or temporary links.
  - It depends on internet path quality and provider endpoints.
- **Dedicated circuits** such as AWS Direct Connect, Azure ExpressRoute, and Google Cloud Interconnect provide private connectivity through a carrier or colocation partner.
  - They improve predictability, bandwidth, and latency.
  - They do not automatically encrypt traffic unless you add encryption at another layer.
- **Transit hubs** such as AWS Transit Gateway, Azure Virtual WAN, and Google Cloud Network Connectivity Center simplify hub-and-spoke routing across many networks.
- **Peering** connects networks directly but usually does not provide transitive routing, so it is not a full replacement for a transit hub.

A useful mental model is:
- VPN = encrypted over internet
- Dedicated circuit = private path, not necessarily encrypted
- Transit hub = scalable route distribution and segmentation
- Peering = point-to-point connectivity with limited routing scope

**Dedicated circuit security controls**

Dedicated links reduce exposure to the public internet, but they do not eliminate trust and security concerns. You still need encryption, authentication, and provider risk management.

- Traffic on a private circuit may still traverse shared provider infrastructure.
- You should evaluate the provider’s physical security, carrier trust model, and cross-connect handling.
- Encryption options over private links include:
  - IPsec tunnels over the dedicated circuit.
  - Application-layer TLS for sensitive services.
  - Provider-specific MACsec where supported for link-layer encryption.
- **MACsec** (Media Access Control Security) encrypts Ethernet frames and is useful when supported on the physical handoff or interconnect.
- Dedicated links are often paired with redundant paths and diverse locations to reduce single points of failure.
- You should treat private connectivity as a transport optimization, not as a substitute for encryption or identity controls.

> Do not assume a private circuit is inherently secure enough for regulated or sensitive data. Private transport reduces exposure, but confidentiality still depends on encryption and access policy.

**Zero trust networking in cloud and hybrid environments**

Zero trust means you do not trust network location by default. Access decisions depend on identity, device posture, context, and continuous verification rather than on being “inside” a trusted subnet.

- **Identity-aware access** evaluates who the user or workload is before granting access.
- **Continuous verification** rechecks trust signals during the session, not only at login.
- **Context-aware policy** can include device compliance, geolocation, risk score, time of day, and application sensitivity.
- **Least privilege** applies to both human users and service-to-service communication.
- Network location becomes a weak signal rather than a primary trust boundary.

In practice, zero trust often uses:
- SSO (single sign-on) with MFA for administrators.
- Device compliance checks for managed endpoints.
- Short-lived credentials and workload identity for services.
- Microsegmentation and service-to-service authorization.
- Private access brokers or identity-aware proxies for internal applications.

A common pattern is to expose an internal app through an identity-aware proxy rather than a broad VPN. The user authenticates, the proxy checks policy, and only then does it forward traffic to the app.

**Cloud firewall and inspection placement**

Cloud firewall placement matters because different controls inspect different layers and sit at different points in the path.

- **Security groups** protect workloads at the interface level and are ideal for least-privilege allow rules.
- **Network ACLs** provide subnet guardrails and can block unwanted ranges before traffic reaches workloads.
- **Provider network firewalls** can centralize inspection between subnets, VPCs/VNets, or internet edges.
- **Host firewalls** protect the operating system and are useful for local service restrictions and defense against lateral movement.
- **WAFs** sit in front of web applications and inspect HTTP/S requests for malicious patterns.
- **Proxy-based egress controls** can inspect outbound web traffic, enforce URL filtering, and log user activity.
- **NAT gateways** provide source address translation for outbound internet access but do not inspect content.
- **Firewall egress patterns** are used when you need policy enforcement, logging, or TLS inspection rather than simple address translation.

Use the right tool for the job:
- Use NAT when you need private subnets to reach the internet without exposing internal addresses.
- Use a proxy when you need application-aware outbound control.
- Use a firewall when you need stateful policy, segmentation, or inspection.
- Use a WAF when you need to protect web applications from HTTP/S attacks.

**Internal load balancing and service discovery**

Cloud-native service discovery helps internal applications find each other without hard-coded IP addresses. This is especially important in autoscaling, container, and multi-zone designs.

- **Internal load balancers** expose services only inside the private network.
- **Private DNS** maps service names to internal endpoints and can integrate with service discovery systems.
- **Service endpoints** and private access features keep traffic on the provider backbone instead of traversing the public internet.
- Internal load balancing is commonly used for:
  - Microservices.
  - Shared internal APIs.
  - Database proxies.
  - Management services.
- Service discovery can be:
  - DNS-based.
  - Registry-based.
  - Mesh-based with sidecars or control planes.

A practical design is to publish `app.internal.example.com` in private DNS and point it to an internal load balancer. Clients resolve the name inside the VPC/VNet and reach the service without public exposure.

Example AWS-style internal load balancer pattern:
```bash
aws elbv2 create-load-balancer \
  --name internal-app-lb \
  --scheme internal \
  --subnets subnet-aaa subnet-bbb \
  --security-groups sg-12345678
```

Example private DNS record concept:
```dns
app.internal.example.com. 60 IN A 10.20.30.40
```

**Provider examples for hybrid connectivity**

Different providers use different names, but the design principles are similar: segment networks, control route exchange, and inspect traffic at transit points.

- **AWS**
  - Use VPCs, route tables, security groups, and network ACLs for segmentation.
  - Use Site-to-Site VPN for encrypted internet tunnels.
  - Use Direct Connect for private dedicated connectivity.
  - Use Transit Gateway for hub-and-spoke routing across many VPCs and on-premises links.
  - Use route tables on attachments to control which networks can talk to each other.
  - Use AWS Network Firewall or third-party appliances for centralized inspection.
- **Azure**
  - Use VNets, subnets, NSGs (Network Security Groups), and UDRs (User Defined Routes).
  - Use VPN Gateway for encrypted tunnels.
  - Use ExpressRoute for private connectivity.
  - Use Virtual WAN for hub routing and branch connectivity.
  - Use Azure Firewall for centralized policy and logging.
  - Use Private Link for private access to platform services.
- **Google Cloud**
  - Use VPCs, firewall rules, and subnet routes.
  - Use Cloud VPN for encrypted tunnels.
  - Use Cloud Interconnect for private connectivity.
  - Use Cloud Router for dynamic route exchange with BGP.
  - Use Network Connectivity Center for hub-and-spoke connectivity.
  - Use Cloud Armor for edge and application protection.

Example Azure route control concept:
```powershell
az network route-table create -g rg-net -n rt-spoke1
az network route-table route create -g rg-net --route-table-name rt-spoke1 -n default-to-fw `
  --address-prefix 0.0.0.0/0 --next-hop-type VirtualAppliance --next-hop-ip-address 10.0.1.4
```

Example Google Cloud hybrid route exchange concept:
```bash
gcloud compute routers create cr-hub \
  --network=prod-vpc \
  --region=us-central1

gcloud compute routers add-bgp-peer cr-hub \
  --region=us-central1 \
  --peer-name=onprem1 \
  --peer-asn=65010 \
  --interface=if-1 \
  --peer-ip-address=169.254.10.2
```

**Route propagation, overrides, and segmentation strategy**

Route propagation can simplify operations, but it can also create accidental transitive access. You should explicitly decide which routes are learned, which are advertised, and which are blocked.

- Use propagation when you want dynamic reachability from VPNs or transit hubs.
- Disable or restrict propagation when you need strict segmentation between environments.
- Prefer explicit route tables for sensitive subnets such as management, PCI, or production control planes.
- Use route summarization carefully to avoid overexposing networks.
- Watch for overlapping CIDRs in mergers, acquisitions, or multi-cloud designs.
- Use separate route domains or transit segmentation when overlapping address space cannot be avoided.

A common design is:
- Public subnets route `0.0.0.0/0` to an internet gateway or equivalent.
- Private application subnets route internet-bound traffic to NAT or proxy services.
- Sensitive subnets route only to internal services, inspection appliances, and approved on-premises prefixes.
- Management subnets have no direct internet route and only controlled access through bastions, identity-aware proxies, or private admin paths.

**IPv6 considerations**

IPv6 changes address planning, routing, and security assumptions. It is increasingly relevant in cloud and hybrid designs because many environments are dual-stack by default or by preference.

- IPv6 uses `::/0` as the default route.
- Security groups and firewall rules must explicitly allow IPv6 ranges such as `2001:db8::/32` in examples.
- NAT is not a general IPv6 requirement in the same way it is for IPv4, so you often rely more on routing and firewall policy.
- Dual-stack designs must secure both IPv4 and IPv6 paths.
- Some teams forget to mirror IPv4 controls in IPv6, creating an unintended bypass.
- Private DNS and service discovery should support AAAA records when IPv6 clients are expected.

Example dual-stack security rule concept:
```json
{
  "IpPermissions": [
    {
      "IpProtocol": "tcp",
      "FromPort": 443,
      "ToPort": 443,
      "Ipv6Ranges": [
        { "CidrIpv6": "2001:db8:1234::/48" }
      ]
    }
  ]
}
```

**Monitoring, telemetry, and anomaly detection**

Cloud network visibility is essential because you cannot inspect every packet by default. Use layered telemetry to detect misconfigurations, lateral movement, and exfiltration.

- **Flow logs** show metadata about accepted and rejected traffic.
- **Packet mirroring** copies traffic to an inspection or capture appliance for deeper analysis.
- **Traffic analytics** and managed detections can identify unusual east-west movement or unexpected destinations.
- **Alerting** should focus on:
  - New outbound destinations.
  - Spikes in denied traffic.
  - Unexpected cross-zone or cross-account communication.
  - Large transfers from sensitive subnets.
- **DNS logs** help detect command-and-control or data exfiltration via domain lookups.
- **Load balancer logs** help correlate client behavior with backend access.
- **Firewall logs** help confirm whether policy is blocking or allowing traffic as intended.

A practical monitoring stack often includes:
- Flow logs for broad visibility.
- Packet capture or mirroring for incident response.
- SIEM (Security Information and Event Management) correlation for alerting.
- Baselines for east-west traffic patterns.

Example AWS-style flow log enablement:
```bash
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-12345678 \
  --traffic-type ALL \
  --log-destination-type cloud-watch-logs \
  --log-group-name /aws/vpc/flowlogs
```

**Management plane and API access control**

Cloud management interfaces are high-value targets because compromise of the control plane can expose the entire environment. You should restrict administrative access as tightly as production traffic.

- Use MFA for all privileged accounts.
- Restrict API access by identity, role, and conditional policy.
- Limit console access to approved networks or identity-aware access paths where possible.
- Use bastions, privileged access workstations, or zero trust access brokers for administration.
- Log all administrative actions and alert on privilege escalation.
- Separate duties between network admins, security admins, and application teams.

Important distinction:
- A workload may be unreachable from the internet but still be vulnerable if its management API is exposed.
- Conversely, a public application may be safely exposed if the control plane is tightly locked down and the app is protected by WAF, IAM, and segmentation.

**Practical design patterns**

A secure cloud/hybrid network usually combines several patterns rather than relying on one architecture.

- **Hub-and-spoke**
  - Centralizes inspection and shared services.
  - Simplifies route control and logging.
  - Works well with transit gateways, virtual WANs, or connectivity hubs.
- **Private service access**
  - Uses private endpoints and private DNS for platform services.
  - Reduces public exposure.
  - Helps keep traffic on provider networks.
- **Identity-aware internal access**
  - Replaces broad VPN access with app-specific authorization.
  - Improves least privilege and auditability.
- **Egress control**
  - Uses NAT for basic translation.
  - Uses proxies or firewalls for policy enforcement and inspection.
  - Logs outbound destinations for anomaly detection.
- **Segregated management plane**
  - Keeps admin access separate from user and application traffic.
  - Uses dedicated subnets, roles, and access paths.

**Troubleshooting checklist**

When cloud or hybrid connectivity fails, isolate the problem by layer.

1. Verify identity and authorization.
   - Confirm the user, role, or workload has permission to connect.
2. Verify name resolution.
   - Check private DNS, split-horizon DNS, and service discovery records.
3. Verify routing.
   - Inspect route tables, propagated routes, and next hops.
4. Verify security policy.
   - Check security groups, ACLs, firewall rules, and WAF policy.
5. Verify translation and inspection.
   - Check NAT, proxy, load balancer, and firewall state.
6. Verify path symmetry.
   - Ensure return traffic follows a compatible route.
7. Verify logs and telemetry.
   - Review flow logs, firewall logs, and packet captures.

Useful commands for local validation:
```bash
ip route
traceroute 10.10.20.15
curl -vk https://internal-api.example.com
dig app.internal.example.com
```

**Exam takeaways**

- Cloud networking security is layered: identity, routing, segmentation, inspection, encryption, and monitoring all matter.
- Security groups, ACLs, provider firewalls, host firewalls, and WAFs solve different problems.
- Dedicated circuits improve performance and predictability, but you still need encryption and trust validation.
- Route propagation and route precedence can create hidden exposure or asymmetric routing.
- Zero trust focuses on identity, device context, and continuous verification rather than network location.
- IPv6 must be secured explicitly and should not be treated as an afterthought.
- Control plane protection is as important as data plane protection.
