## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Virtualization security fundamentals**

Virtualization introduces a software layer that abstracts compute, storage, and networking from physical hardware. That abstraction improves utilization and agility, but it also creates new trust boundaries and failure modes. For CISSP purposes, focus on how isolation can fail, how management planes become high-value targets, and how operational shortcuts create systemic risk.

A **hypervisor** is the core control layer that hosts and manages virtual machines (VMs). The two major types matter for risk analysis:

- **Type 1 hypervisor** runs directly on hardware.
  - Examples include enterprise bare-metal platforms.
  - It usually has a smaller attack surface than a hosted model.
  - It is generally preferred for stronger isolation and lower dependency on a general-purpose host OS.
- **Type 2 hypervisor** runs on top of a host operating system.
  - Examples include desktop virtualization products.
  - It inherits risk from the host OS, drivers, and user applications.
  - It is typically higher risk in production because compromise of the host can expose all guest VMs.

A useful exam distinction is that Type 1 is not automatically “secure,” but it usually reduces exposure by removing the general-purpose host OS from the critical path. Type 2 is more convenient for labs and endpoints, but the host becomes a larger attack surface and a single point of compromise.

**Virtualization risk categories**

Virtualization concentrates many workloads on shared infrastructure, so a single weakness can affect multiple systems. The main risks include isolation failure, management plane compromise, sprawl, and lifecycle mistakes.

- **VM escape** occurs when code inside a guest breaks out to the hypervisor or host.
- **Inter-VM attacks** target shared resources, such as memory, storage, or virtual networking.
- **Management plane compromise** gives an attacker control over provisioning, snapshots, templates, and access policies.
- **Resource contention** can create performance degradation and availability issues across tenants or workloads.
- **Misconfiguration** often matters more than the underlying platform.

Common operational risks include:

- **VM sprawl**
  - Too many unmanaged VMs accumulate over time.
  - Owners, patch status, and business purpose become unclear.
  - Untracked systems increase attack surface and licensing, backup, and monitoring gaps.
- **Snapshot and clone risk**
  - Snapshots can preserve sensitive data, secrets, and old vulnerabilities.
  - Clones may inherit weak configurations or stale credentials.
  - Forgotten test copies often remain online after the original purpose ends.
- **Orphaned images and templates**
  - Golden images can drift from baseline security standards.
  - Old templates may contain default passwords, outdated packages, or embedded keys.
  - Reusing compromised images spreads risk quickly.

A practical hardening approach is to treat images and templates as controlled artifacts, not convenience files. You should patch, scan, sign, and retire them with the same discipline you apply to production systems.

**Hypervisor and host hardening**

The hypervisor and its management interfaces are high-value targets because they control many guests at once. Protect them with strong administrative controls and minimal exposure.

- Restrict administrative access to dedicated management networks.
- Use multifactor authentication for console and API access.
- Patch the hypervisor, host firmware, and management tools promptly.
- Disable unused virtual devices and services.
- Separate management, storage, and tenant traffic where possible.
- Log administrative actions and alert on unusual provisioning or snapshot activity.

Example hardening checks on a Linux virtualization host:

```bash
virsh list --all
virsh dominfo app01
virsh snapshot-list app01
ip addr show
ss -tulpn
```

Example of a restrictive firewall posture for a management interface:

```bash
iptables -A INPUT -p tcp -s 10.10.10.0/24 --dport 443 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j DROP
```

**Container architecture risks**

Containers share the host kernel, so they are lighter than VMs but usually provide weaker isolation. That shared-kernel model is efficient, yet it means a kernel flaw, runtime weakness, or privilege mistake can affect many containers at once.

Important container concepts:

- **Image**: a packaged filesystem and metadata used to start a container.
- **Registry**: a repository that stores and distributes images.
- **Runtime**: the software that starts and manages containers.
- **Orchestration**: the control layer that schedules, scales, and heals containers, such as Kubernetes.

Container-specific attack examples include:

- **Insecure Docker socket exposure**
  - Exposing `/var/run/docker.sock` to a container or network gives powerful control over the host daemon.
  - An attacker can often start privileged containers, mount host paths, or inspect other workloads.
- **Shared volume abuse**
  - Multiple containers sharing a volume can leak secrets or tamper with application data.
  - A compromised container may read credentials written by another service.
- **Container breakout via misconfigured capabilities**
  - Granting excessive Linux capabilities, such as `CAP_SYS_ADMIN`, can enable host-level actions.
  - `--privileged` containers dramatically increase breakout risk.
- **Host mount misuse**
  - Mounting sensitive host directories into a container can expose the filesystem, logs, or credentials.
  - `hostPath` in Kubernetes can create similar exposure if used carelessly.

A simple example of dangerous Docker usage:

```bash
docker run -it --rm \
  -v /:/host \
  --privileged \
  alpine sh
```

That pattern is risky because it gives the container broad access to the host filesystem and kernel interfaces. In practice, even one such container can undermine the isolation model.

**Container image integrity and supply chain**

Container security depends heavily on image provenance and runtime restrictions. You should verify what you deploy and limit what the container can do after startup.

Key controls include:

- **Digest pinning**
  - Use immutable image digests instead of mutable tags when possible.
  - Tags like `latest` can change without warning.
- **Trusted base images**
  - Prefer approved base images from known sources.
  - Minimize packages and remove build tools from runtime images.
- **SBOMs**
  - A **software bill of materials** lists components and dependencies.
  - SBOMs help you assess exposure when a library vulnerability is disclosed.
- **Image signing and verification**
  - Signed images support provenance checks before deployment.
  - Verification reduces the chance of pulling tampered artifacts.
- **Registry access control**
  - Restrict who can push, delete, or overwrite images.
  - Scan images for known vulnerabilities and secrets before release.

Example of digest pinning:

```bash
docker pull nginx@sha256:3f1d2c0f0f0a0b1c2d3e4f567890abcdef1234567890abcdef1234567890abcd
```

Example Kubernetes image reference:

```yaml
containers:
  - name: web
    image: registry.example.com/app/web@sha256:3f1d2c0f0f0a0b1c2d3e4f567890abcdef1234567890abcdef1234567890abcd
```

**Container runtime and orchestration risks**

Orchestration platforms add powerful automation, but they also expand the attack surface. The control plane, API server, admission policies, and secrets store become critical assets.

Common risks include:

- **Overprivileged pods**
  - Running as root inside the container increases impact if the app is compromised.
  - `privileged: true` can expose host devices and kernel interfaces.
- **Overexposed control plane endpoints**
  - Publicly reachable Kubernetes API servers or management dashboards are high-risk.
  - Weak authentication or broad network access can lead to cluster takeover.
- **API abuse**
  - Attackers may use legitimate APIs to create workloads, exfiltrate secrets, or modify policies.
  - Rate limiting and strong authentication matter.
- **Insecure management consoles**
  - Web consoles and CI/CD integrations often have broad privileges.
  - Compromise of these tools can cascade into the entire environment.

Example of a risky Kubernetes pod spec:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: risky-app
spec:
  containers:
    - name: app
      image: app:1.0
      securityContext:
        privileged: true
      volumeMounts:
        - name: host-root
          mountPath: /host
  volumes:
    - name: host-root
      hostPath:
        path: /
```

Safer patterns include:

- Run containers as non-root.
- Drop unnecessary capabilities.
- Use read-only root filesystems where possible.
- Avoid `hostNetwork`, `hostPID`, and `hostPath` unless there is a strong business need.
- Apply network policies and admission controls.
- Separate build, test, and production clusters.

**Cloud architecture risk fundamentals**

Cloud computing changes the trust model because infrastructure is shared, abstracted, and managed by a provider. The main security challenge is not just “the cloud,” but the combination of multi-tenancy, remote administration, and rapid change.

Cloud service models affect risk:

- **Infrastructure as a Service (IaaS)**
  - You manage operating systems, applications, and much of the configuration.
  - The provider manages hardware, virtualization, and core facilities.
  - Misconfigured security groups, storage, and guest OSs are common risks.
- **Platform as a Service (PaaS)**
  - The provider manages more of the runtime and platform stack.
  - You still manage application code, identity, data, and configuration.
  - API abuse and insecure app logic become more prominent.
- **Software as a Service (SaaS)**
  - The provider manages most of the stack.
  - You mainly manage identity, access, data governance, and configuration.
  - Misuse of sharing settings and weak identity controls are common issues.

Deployment architecture changes the attack surface and trust boundaries:

- **Public cloud**
  - Highest exposure to internet-facing services and shared infrastructure.
  - Strong reliance on provider controls and customer configuration.
  - Broadest external attack surface.
- **Private cloud**
  - Usually reduces external exposure by limiting access to a dedicated environment.
  - Still vulnerable to internal threats, misconfiguration, and weak segmentation.
  - Often shifts risk from provider trust to internal operational maturity.
- **Hybrid cloud**
  - Combines on-premises and cloud environments.
  - Increases complexity in identity, routing, logging, and data movement.
  - Misaligned controls between environments can create gaps.
- **Community cloud**
  - Shared by organizations with similar requirements or missions.
  - Trust is broader than private cloud but narrower than public cloud.
  - Governance and shared responsibility agreements are critical.

**Multi-tenancy and shared infrastructure risks**

Multi-tenancy means multiple customers or workloads share physical or logical resources. This model improves efficiency but introduces isolation and fairness concerns.

Key risks include:

- **Tenant isolation failure**
  - One tenant may access another tenant’s data or compute resources.
  - Failures can occur in hypervisors, storage layers, identity systems, or management APIs.
- **Noisy neighbor effects**
  - One workload consumes disproportionate CPU, memory, I/O, or network resources.
  - Performance degradation can become an availability issue.
- **Provider-side shared infrastructure concerns**
  - Shared storage arrays, network fabrics, and management systems can create correlated failure modes.
  - A provider incident may affect many customers at once.

From an exam perspective, remember that multi-tenancy is not only a confidentiality issue. It also affects availability and integrity when resource contention, control plane failures, or shared service outages occur.

**Cloud misconfigurations and common attack paths**

Many cloud incidents come from simple configuration errors rather than advanced exploits. The most common issues are overly broad access, exposed management interfaces, and weak identity controls.

Examples of common misconfigurations:

- **Overly permissive S3 bucket policies**
  - Public read or write access can expose sensitive data.
  - Cross-account permissions may be broader than intended.
- **Security group `0.0.0.0/0` exposure**
  - Opening SSH, RDP, databases, or admin ports to the internet increases brute-force and exploitation risk.
  - Restrict by source IP, VPN, or private network where possible.
- **Kubernetes `hostPath` or privileged pod misuse**
  - These settings can expose host filesystems or kernel capabilities.
  - They are often unnecessary for normal application workloads.
- **Overexposed control plane endpoints**
  - Public API endpoints without strong authentication or network restrictions are dangerous.
  - Management consoles should be tightly controlled.
- **Weak IAM policies**
  - Overly broad identity and access management permissions can allow privilege escalation or data exfiltration.
- **Unencrypted storage or backups**
  - Data at rest may be exposed if encryption is disabled or keys are mishandled.

Example of a risky AWS-style security group rule:

```json
{
  "IpProtocol": "tcp",
  "FromPort": 22,
  "ToPort": 22,
  "CidrIp": "0.0.0.0/0"
}
```

Example of a safer pattern:

```json
{
  "IpProtocol": "tcp",
  "FromPort": 22,
  "ToPort": 22,
  "CidrIp": "10.0.0.0/24"
}
```

**Cloud encryption and key management**

Encryption is only as strong as the key management model behind it. In cloud environments, you must understand who controls the keys, who can access them, and what happens if the provider or customer account is compromised.

Key distinctions:

- **Provider-managed keys**
  - The cloud provider creates and manages the keys.
  - This reduces operational burden but increases reliance on provider custody.
  - It may simplify service integration, but it can limit customer control.
- **Customer-managed keys**
  - The customer controls key policy, rotation, and access.
  - This improves governance and separation of duties.
  - It also increases operational responsibility and the risk of misconfiguration.
- **Customer-held keys**
  - The customer retains the strongest custody model.
  - Availability and recovery become more complex.
  - Loss of the key can mean permanent data loss.

Important exam point: encryption does not eliminate provider access to metadata, control plane logs, or service-side processing in all cases. You should distinguish between data confidentiality, key custody, and administrative visibility.

**Logging, monitoring, and incident response in cloud**

Cloud logging and incident response follow the shared responsibility model. You can usually access your own application, identity, and configuration logs, but provider-side telemetry may be limited or abstracted.

What the customer can often access:

- Application logs
- OS logs for IaaS workloads
- Cloud audit logs for account activity
- Configuration history and policy changes
- Container and orchestration logs, depending on platform access

What the customer may not fully access:

- Provider internal host logs
- Physical hardware telemetry
- Some hypervisor-level evidence
- Full provider-side forensic detail during an incident

This matters during investigations because evidence may be incomplete from the customer side. You should plan for:

- Centralized log collection outside the target account or subscription.
- Time synchronization across systems.
- Immutable storage for audit logs.
- Defined provider escalation procedures.
- Evidence preservation before remediation changes the environment.

A practical AWS-style audit command example:

```bash
aws cloudtrail lookup-events --lookup-attributes AttributeKey=EventName,AttributeValue=ConsoleLogin
```

A practical Kubernetes log check:

```bash
kubectl get pods -A
kubectl logs deployment/web -n prod
kubectl describe pod web-7d9f8c7f5b-abcde -n prod
```

**Shared responsibility model**

The shared responsibility model explains who secures what. The exact boundary depends on the service model, but the principle stays the same: the provider secures the cloud, and the customer secures what they put in it.

Typical customer responsibilities include:

- Identity and access management
- Data classification and encryption choices
- Network segmentation and firewall rules
- Guest OS and application patching in IaaS
- Container image hygiene and runtime policies
- Logging, alerting, and incident response planning

Typical provider responsibilities include:

- Physical security
- Core infrastructure
- Hypervisor and managed service platform security
- Availability of the cloud service itself

A concise way to remember the exam angle is that cloud does not remove responsibility; it redistributes it. Many breaches happen because teams assume the provider covers controls that are actually customer-managed.

**Practical defensive controls**

Use layered controls to reduce the impact of virtualization, container, and cloud failures.

- **Segmentation**
  - Separate management, production, and tenant networks.
  - Use microsegmentation where feasible.
- **Least privilege**
  - Restrict admin roles, API permissions, and service accounts.
  - Remove unused capabilities and permissions.
- **Secure baselines**
  - Standardize hardened images and templates.
  - Rebuild from known-good images instead of patching drifted systems indefinitely.
- **Continuous scanning**
  - Scan VMs, containers, and images for vulnerabilities and misconfigurations.
  - Monitor cloud posture continuously.
- **Immutable infrastructure**
  - Replace rather than modify where possible.
  - This reduces configuration drift and hidden state.
- **Secrets management**
  - Store secrets in dedicated vaults, not in images, environment files, or shared volumes.
- **Backup and recovery**
  - Protect snapshots, backups, and templates with access controls and retention policies.

Example of a safer container run command:

```bash
docker run --rm \
  --read-only \
  --cap-drop ALL \
  --security-opt no-new-privileges \
  -p 8080:8080 \
  myapp:1.0
```

**Exam-oriented distinctions**

Use these contrasts to answer CISSP-style questions accurately:

- **Virtualization vs containers**
  - Virtualization isolates at the hardware abstraction layer through a hypervisor.
  - Containers isolate at the OS level and share the host kernel.
- **Type 1 vs Type 2 hypervisors**
  - Type 1 generally offers lower risk and better isolation for production.
  - Type 2 inherits host OS risk and is more exposed to endpoint compromise.
- **Public vs private cloud**
  - Public cloud has broader exposure and stronger dependence on provider controls.
  - Private cloud reduces external exposure but still requires strong internal governance.
- **IaaS vs PaaS vs SaaS**
  - As abstraction increases, customer control decreases and provider responsibility increases.
  - Customer focus shifts from infrastructure hardening to identity, data, and configuration.
- **Encryption vs key custody**
  - Encryption protects data only if keys are protected and access is controlled.
  - Key ownership determines who can decrypt, rotate, and recover data.
- **Logging vs forensic completeness**
  - Customer logs are useful but may not include provider internals.
  - Plan for partial visibility and preserve evidence early.

**Quick review checklist**

- Verify whether the environment uses Type 1 or Type 2 hypervisors.
- Inventory VMs, snapshots, clones, and templates.
- Remove public exposure from admin ports and control plane endpoints.
- Audit cloud IAM policies, security groups, and storage permissions.
- Pin container images by digest and verify trusted sources.
- Avoid privileged containers, dangerous capabilities, and unnecessary host mounts.
- Define who owns logs, keys, backups, and incident response actions.
- Test recovery from cloud and container failures, not just from hardware outages.
