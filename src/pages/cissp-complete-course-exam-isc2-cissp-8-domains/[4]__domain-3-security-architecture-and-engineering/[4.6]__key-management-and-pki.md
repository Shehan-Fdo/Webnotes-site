## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Key management and PKI overview**

Public Key Infrastructure (<abbr title="Public Key Infrastructure">PKI</abbr>) is the framework that binds identities to public keys through certificates, policies, and trusted authorities. Key management is the broader discipline of generating, protecting, distributing, using, rotating, archiving, and destroying cryptographic keys.

For CISSP, focus on the relationship between:
- **Keys**: the secret material used by cryptographic algorithms.
- **Certificates**: signed statements that bind a subject to a public key.
- **Trust**: how systems decide which certificates and keys to accept.
- **Lifecycle**: how keys and certificates are created, used, renewed, revoked, and retired.

PKI is not just “certificates.” It includes operational controls, trust stores, enrollment protocols, revocation services, policy documents, and administrative processes.

**Core cryptographic concepts**

A key is only as strong as its algorithm, length, generation quality, storage protection, and lifecycle handling. CISSP questions often test whether you can distinguish symmetric and asymmetric key use, and when each is appropriate.

- **Symmetric cryptography** uses one shared secret for encryption and decryption.
  - It is fast and suitable for bulk data protection.
  - The main challenge is secure key distribution.
- **Asymmetric cryptography** uses a public/private key pair.
  - It supports digital signatures, key exchange, and identity binding.
  - It is slower than symmetric cryptography and usually protects small data items or symmetric session keys.
- **Hashing** is one-way and supports integrity checks, certificate signatures, and message digests.
- **Digital signatures** provide integrity, authenticity, and nonrepudiation in many practical contexts.

Common algorithm families and exam-relevant tradeoffs:
- **RSA** is widely supported and historically common for certificates and signatures.
  - Security depends on key size and implementation quality.
  - Typical modern sizes are 2048 bits or larger; 3072 bits is often used for longer-term security.
- **Elliptic Curve Cryptography (<abbr title="Elliptic Curve Cryptography">ECC</abbr>)** provides comparable security with smaller keys and better performance in many environments.
  - It is common in modern TLS deployments and constrained devices.
- **Diffie-Hellman (<abbr title="Diffie-Hellman">DH</abbr>)** and **Elliptic Curve Diffie-Hellman (<abbr title="Elliptic Curve Diffie-Hellman">ECDH</abbr>)** support key agreement.
  - Ephemeral variants, such as DHE and ECDHE, support forward secrecy by generating fresh session keys.
- **Digital Signature Algorithm (<abbr title="Digital Signature Algorithm">DSA</abbr>)** is less common in modern deployments and is often a legacy consideration.

> In exam terms, the property is **forward secrecy**; the mechanism is an ephemeral key exchange such as DHE or ECDHE. Do not confuse the property with the protocol.

**Key management lifecycle**

Key management is a lifecycle problem, not a one-time setup task. You must protect keys from creation through destruction.

Typical lifecycle stages:
1. **Generation**
   - Use a strong random number generator or hardware security module (<abbr title="Hardware Security Module">HSM</abbr>) when possible.
   - Generate keys in a controlled environment.
2. **Distribution**
   - Deliver keys through secure channels or protected enrollment workflows.
   - Avoid exposing private keys during transfer.
3. **Storage**
   - Protect private keys with file permissions, encryption, HSMs, or secure enclaves.
   - Store backups securely and separately from production systems.
4. **Use**
   - Limit key access to authorized services and operators.
   - Apply least privilege and separation of duties.
5. **Rotation and renewal**
   - Replace keys before expiration or compromise.
   - Use shorter lifetimes for higher-risk environments.
6. **Archival**
   - Retain keys only when business, legal, or forensic requirements demand it.
   - Archive securely and document access controls.
7. **Destruction**
   - Wipe or zeroize keys when no longer needed.
   - Destroy backups and media according to retention policy.

Practical lifecycle controls:
- Use **key escrow** only when policy and legal requirements justify it.
- Use **split knowledge** and **dual control** for high-value CA or HSM operations.
- Maintain documented **backup and restore** procedures for CA private keys and recovery keys.
- Ensure secure **key destruction** for retired media, HSM partitions, and archived key material.

**Private key protection and operational controls**

Private keys are the most sensitive assets in PKI. If an attacker obtains a private key, they can impersonate the subject, decrypt protected data in some scenarios, or sign malicious content.

Protect private keys with:
- **HSMs** for high-assurance key storage and cryptographic operations.
- **Encrypted key files** protected by strong passphrases.
- **Access control lists** and service isolation.
- **MFA** for administrative access to CA systems and key management consoles.
- **Audit logging** for key access, issuance, renewal, and revocation actions.

Operational guidance:
- Prefer non-exportable keys for high-value identities.
- Separate CA signing keys from end-entity keys.
- Keep offline root CA keys offline except during controlled signing events.
- Back up CA keys securely, but do not weaken protection by making backups broadly accessible.

**PKI building blocks**

A typical PKI includes:
- **Root CA**: the trust anchor at the top of the hierarchy.
- **Intermediate CA**: signs end-entity certificates and reduces exposure of the root.
- **End-entity certificate**: issued to a user, server, device, or service.
- **Registration Authority (<abbr title="Registration Authority">RA</abbr>)**: validates identity before issuance.
- **Repository**: publishes certificates, CRLs, and policy documents.
- **Revocation services**: provide status information for invalidated certificates.

Hierarchy design matters:
- A root CA should be highly protected and often offline.
- Intermediate CAs handle routine issuance and can be segmented by purpose, geography, or business unit.
- End-entity certificates should have shorter lifetimes than CA certificates.

**Certificate anatomy and validation**

A certificate is a signed data structure that binds a subject identity to a public key. In X.509 environments, certificates commonly include:
- Subject name
- Issuer name
- Public key
- Serial number
- Validity period
- Signature algorithm
- Extensions such as Subject Alternative Name (<abbr title="Subject Alternative Name">SAN</abbr>), Key Usage, and Extended Key Usage

Validation checks usually include:
- The certificate chains to a trusted root.
- The certificate is within its validity period.
- The signature is valid.
- The certificate is not revoked, if revocation checking is enforced.
- The hostname or identity matches the certificate’s SAN or subject, depending on the application.

Hostname validation details:
- Modern clients primarily use the **SAN** extension for DNS names and IP addresses.
- The common name (<abbr title="Common Name">CN</abbr>) is legacy and should not be relied on when SAN is present.
- Wildcard certificates such as `*.example.com` match a single label only.
  - `api.example.com` matches.
  - `a.b.example.com` does not match.
- Wildcards do not match the apex domain unless explicitly included.
  - `example.com` is not covered by `*.example.com`.

Certificate types and common use cases:
- **Server certificates** for TLS on web and API endpoints.
- **Client certificates** for mutual TLS (<abbr title="mutual Transport Layer Security">mTLS</abbr>) and device authentication.
- **Code signing certificates** for software integrity and publisher identity.
- **Email certificates** for S/MIME encryption and signing.
- **Device or machine certificates** for IoT, VPN, Wi-Fi, and service identity.

**Certificate request types and PKI artifacts**

CISSP questions may refer to the request and issuance artifacts used in PKI workflows.

Common artifacts:
- **CSR**: Certificate Signing Request.
  - Usually encoded as **PKCS#10**.
  - Contains the public key, subject information, and a signature created with the corresponding private key.
  - Does not contain the private key itself.
- **Certificate chain**:
  - The ordered path from end-entity certificate through one or more intermediates to a trusted root.
- **Intermediate chain**:
  - The intermediate certificates needed to build trust from the leaf certificate to the root.
- **CRL**: Certificate Revocation List.
  - A signed list of revoked certificate serial numbers.
- **Delta CRL**:
  - A smaller CRL containing only changes since the last base CRL.
- **OCSP**: Online Certificate Status Protocol.
  - A real-time status query protocol for certificate validity.
- **OCSP stapling**:
  - The server includes a recent OCSP response during the TLS handshake to reduce client lookups.
- **Certificate policy and CPS**:
  - Policy documents that define issuance rules, identity proofing, and operational controls.

Practical distinction:
- A **CSR** requests issuance.
- A **certificate chain** proves trust.
- A **CRL** or **OCSP** reports revocation status.

**Issuance, enrollment, and automation**

Enterprise PKI rarely relies on manual certificate creation for long. Automation reduces outages and human error.

Common enrollment and automation protocols:
- **SCEP** (<abbr title="Simple Certificate Enrollment Protocol">SCEP</abbr>)
  - Older but still common in network and device enrollment.
  - Often used by MDM and network equipment.
- **EST** (<abbr title="Enrollment over Secure Transport">EST</abbr>)
  - More modern than SCEP and designed for secure certificate enrollment.
- **ACME** (<abbr title="Automatic Certificate Management Environment">ACME</abbr>)
  - Widely used for automated issuance and renewal, especially for web certificates.

Typical issuance workflow:
1. Generate a key pair on the target system or in an HSM.
2. Create a CSR with the subject and SAN values.
3. Submit the CSR through an enrollment protocol or CA portal.
4. Validate identity through RA or automated policy checks.
5. Issue the certificate and publish the chain.
6. Install the certificate and intermediate chain on the service.
7. Monitor expiration and renew before service impact.

Example CSR creation:
```bash
openssl req -new -newkey rsa:3072 -keyout server.key -out server.csr -sha256
```

Example with SAN in an OpenSSL config:
```ini
[ req ]
default_bits       = 3072
prompt             = no
default_md         = sha256
distinguished_name = dn
req_extensions     = req_ext

[ dn ]
C  = US
ST = NY
L  = New York
O  = Example Corp
CN = app.example.com

[ req_ext ]
subjectAltName = @alt_names

[ alt_names ]
DNS.1 = app.example.com
DNS.2 = api.example.com
```

Generate the CSR with:
```bash
openssl req -new -key server.key -out server.csr -config openssl.cnf
```

> For production, avoid `-nodes` unless you explicitly accept an unencrypted private key for a controlled lab or automation scenario with compensating protections.

**Certificate issuance, signing, and revocation operations**

Operational PKI includes more than inspection. You should understand how a CA signs and revokes certificates.

Example CA workflow with OpenSSL-style tooling:
```bash
# Create a CA private key
openssl genpkey -algorithm RSA -out ca.key -aes-256-cbc

# Create a self-signed root certificate
openssl req -x509 -new -key ca.key -sha256 -days 3650 -out ca.crt

# Sign a CSR with the CA
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key \
  -CAcreateserial -out server.crt -days 825 -sha256
```

Revocation concepts:
- **Revocation** invalidates a certificate before its expiration date.
- Reasons include key compromise, subject change, CA compromise, superseded certificates, or policy violations.
- Revocation is not the same as expiration; expiration is natural end-of-life, revocation is an early invalidation.

Example revocation workflow in a CA-managed environment:
1. Mark the certificate as revoked in the CA database.
2. Publish the updated CRL or OCSP status.
3. Notify relying parties if the certificate is high impact.
4. Replace the certificate and rotate any affected keys.

OpenSSL verification examples:
```bash
# Verify a leaf certificate against a trusted CA bundle
openssl verify -CAfile trusted-ca-bundle.pem server.crt

# Verify with an intermediate supplied separately
openssl verify -CAfile root-ca.pem -untrusted intermediate.pem server.crt
```

Inspect a certificate:
```bash
openssl x509 -in server.crt -noout -text
```

Check a TLS endpoint and request hostname verification:
```bash
openssl s_client -connect app.example.com:443 -servername app.example.com -verify_hostname app.example.com
```

**Revocation checking behavior**

Revocation checking is a common exam topic because implementations differ.

Important distinctions:
- **Hard-fail** means the client rejects the certificate if revocation status cannot be determined.
- **Soft-fail** means the client allows the certificate if revocation status is unavailable, which improves availability but weakens security.
- Many real-world clients use soft-fail behavior in some conditions because OCSP responders may be unreachable.

Revocation mechanisms:
- **CRL**
  - Simple and widely supported.
  - Can become large and stale.
  - Delta CRLs reduce bandwidth and update size.
- **OCSP**
  - Provides near-real-time status.
  - Depends on responder availability and privacy considerations.
- **OCSP stapling**
  - Improves performance and privacy by letting the server present a signed status response.

OCSP responder considerations:
- The responder itself may need a dedicated signing certificate.
- The responder certificate must be trusted for OCSP signing, not general server authentication.
- High availability matters because revocation checks can affect user access.

**Trust stores and enterprise distribution**

Clients trust certificates through local trust stores, enterprise policy, or application-specific pinning and configuration.

Common enterprise trust distribution methods:
- **Group Policy** in Windows domains.
- **MDM** (<abbr title="Mobile Device Management">MDM</abbr>) for mobile and managed endpoints.
- Configuration management tools such as Ansible, Intune, Jamf, or SCCM.
- Application-specific trust bundles for containers, Java keystores, or embedded systems.

Practical examples:
- Install an internal root CA into the Windows Trusted Root store through Group Policy.
- Push a corporate CA profile to iOS and Android devices through MDM.
- Distribute a CA bundle to Linux systems under `/etc/ssl/certs/` or the platform-specific trust path.
- Import CA certificates into Java trust stores with `keytool`.

Example Java trust store import:
```bash
keytool -importcert -alias corp-root -file corp-root.crt \
  -keystore cacerts -storepass changeit
```

Trust distribution risks:
- Overbroad trust can enable internal man-in-the-middle inspection if misused.
- Stale trust stores can break service connectivity after CA rotation.
- Application-specific stores may not follow OS trust updates automatically.

**Certificate transparency, pinning, and modern trust controls**

Certificate Transparency (<abbr title="Certificate Transparency">CT</abbr>) is a public logging system that helps detect misissued publicly trusted certificates. Browsers and public PKI ecosystems increasingly rely on CT expectations.

Key points:
- CT logs are append-only public records of certificate issuance.
- They improve detection of unauthorized or mistaken issuance.
- They do not prevent issuance by themselves; they improve visibility and accountability.

Certificate pinning:
- Pinning binds an application to a specific certificate or public key.
- It can reduce trust in external CAs, but it is operationally fragile.
- If the pinned key changes unexpectedly, clients can fail hard and cause outages.
- Modern ecosystems often prefer safer alternatives such as CT, short-lived certificates, private PKI, or managed trust policies.

Use pinning cautiously:
- It is more suitable for controlled applications than for general web browsing.
- It requires a robust key rotation strategy and backup pins if supported.

**Certificate chain and path building**

Path building is the process of constructing a valid chain from a leaf certificate to a trusted root.

Important terms:
- **Leaf certificate**: the end-entity certificate presented by the service.
- **Intermediate certificate**: signs leaf certificates and is itself signed by another CA.
- **Root certificate**: self-signed trust anchor installed in the trust store.

Operational rules:
- Servers should usually present the leaf certificate plus required intermediates.
- The root certificate is typically not sent by the server because clients are expected to already trust it.
- Missing intermediates are a common cause of TLS failures.

Example deployment layout:
- `server.crt` = leaf certificate
- `intermediate.crt` = issuing CA certificate
- `fullchain.pem` = leaf plus intermediate(s)
- `privkey.pem` = private key

Example Nginx configuration:
```nginx
server {
    listen 443 ssl;
    server_name app.example.com;

    ssl_certificate     /etc/ssl/certs/fullchain.pem;
    ssl_certificate_key /etc/ssl/private/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
}
```

**Self-signed certificates and wildcard certificates**

Self-signed certificates are signed by the same entity whose public key they contain. They are useful for testing, internal labs, and some tightly controlled environments, but they are not inherently trusted by clients.

Use cases:
- Lab systems
- Temporary bootstrap environments
- Internal appliances with explicit trust distribution

Limitations:
- They do not provide third-party trust.
- They often trigger browser warnings unless manually trusted.
- They are poor substitutes for a managed PKI in production.

Wildcard certificates:
- Useful when many subdomains share one identity boundary.
- Reduce operational overhead for large fleets of subdomains.
- Increase blast radius if the private key is compromised.

Operational caution:
- A wildcard certificate for `*.example.com` does not cover `example.com` unless the apex is also included.
- Wildcards are not a replacement for proper service identity design.

**Key sizes, algorithm choice, and lifecycle impact**

Algorithm choice affects performance, interoperability, and security posture.

General guidance:
- **RSA**
  - Broad compatibility.
  - Larger keys and signatures than ECC for similar security.
  - Common in legacy and mixed environments.
- **ECC**
  - Smaller keys and faster operations in many cases.
  - Good for mobile, IoT, and modern TLS.
  - Requires careful curve selection and library support.
- **DH/ECDH**
  - Used for key agreement rather than long-term identity.
  - Ephemeral variants support forward secrecy.
- **DSA**
  - Legacy and less common in modern deployments.

Key size considerations:
- Larger keys generally increase security margin but also increase CPU cost and certificate size.
- Shorter keys may be acceptable for ephemeral session use but not for long-lived CA keys.
- CA keys often use stronger protection and longer lifetimes than end-entity keys, but they still require periodic rotation.

**Backup, archival, and destruction**

Key backups are necessary for recovery, but they create risk. Treat them as highly sensitive assets.

Backup and restore guidance:
- Back up CA private keys, database state, and configuration together.
- Protect backups with encryption and strict access control.
- Test restore procedures before an incident.
- Store backups separately from production systems and, ideally, in separate security domains.

Archival considerations:
- Some organizations must retain old keys or certificates for legal, forensic, or decryption purposes.
- Archived keys should be clearly labeled, access-controlled, and time-limited.
- Do not keep archived keys longer than policy requires.

Destruction guidance:
- Use secure wipe or zeroization methods appropriate to the medium.
- Destroy HSM objects using vendor-supported secure deletion.
- Ensure backup media and snapshots are also retired.
- Document destruction for audit and compliance purposes.

**Practical enterprise workflows**

A mature PKI supports repeatable workflows for issuance, renewal, and revocation.

Example server certificate lifecycle:
1. Generate the key pair on the server or in an HSM.
2. Create a CSR with SAN entries for all required names.
3. Submit the CSR to the CA through ACME, EST, SCEP, or a portal.
4. Receive the signed certificate and intermediate chain.
5. Install the certificate and restart or reload the service.
6. Monitor expiration and automate renewal.
7. Revoke the old certificate if the key changes or compromise occurs.

Example revocation response:
- If a private key is exposed, you revoke the certificate immediately.
- You generate a new key pair.
- You issue a replacement certificate.
- You update dependent services and clients.
- You review logs and access controls to determine impact.

Example ACME-style automation concept:
```bash
certbot certonly --webroot -w /var/www/html -d app.example.com -d api.example.com
```

This kind of automation is common because it reduces manual renewal failures and supports short-lived certificates.

**Common exam traps and distinctions**

- **PFS** is a property, not a protocol.
- **CSR** requests a certificate; it is not the certificate itself.
- **Root CA** is usually offline; **intermediate CA** handles routine issuance.
- **SAN** is the primary hostname validation field; CN is legacy.
- **Revocation** is different from expiration.
- **Soft-fail** revocation behavior favors availability; **hard-fail** favors security.
- **Wildcard** certificates cover one label only.
- **Pinning** can improve control but increases operational fragility.
- **CT** improves visibility, not direct prevention.
- **Private keys** must be protected, backed up carefully, and destroyed securely when retired.

**Quick reference commands**

Inspect a certificate:
```bash
openssl x509 -in cert.pem -noout -subject -issuer -dates -text
```

Verify a chain:
```bash
openssl verify -CAfile root-bundle.pem -untrusted intermediate.pem leaf.pem
```

Check a remote TLS service:
```bash
openssl s_client -connect example.com:443 -servername example.com -verify_hostname example.com
```

Create a private key and CSR:
```bash
openssl req -new -newkey ec -pkeyopt ec_paramgen_curve:P-256 \
  -keyout server.key -out server.csr
```

Revoke a certificate in a CA-managed OpenSSL workflow:
```bash
openssl ca -revoke server.crt
openssl ca -gencrl -out crl.pem
```

**Study focus for CISSP**

When you see PKI questions, ask:
- Who is trusted, and why?
- Where is the private key stored?
- How is identity validated before issuance?
- How is revocation checked?
- What happens if the key is compromised?
- Which algorithm and key size fit the use case?
- How are trust anchors distributed and maintained?

If you can explain the lifecycle of keys and certificates, the difference between trust and validation, and the operational controls around issuance and revocation, you are covering the core CISSP expectations for this topic.
