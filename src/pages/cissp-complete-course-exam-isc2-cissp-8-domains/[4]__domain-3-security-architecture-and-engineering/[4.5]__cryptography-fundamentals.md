## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Cryptography overview**

Cryptography protects confidentiality, integrity, authenticity, and nonrepudiation by transforming data and controlling who can use keys. In CISSP context, you need to know not only the algorithms, but also how they are approved, deployed, rotated, validated, and attacked.

At a high level, cryptography answers four questions:

- Can only the intended party read the data?
- Can you detect tampering?
- Can you prove who sent it?
- Can you trust the key, certificate, or protocol that enabled the exchange?

**Core security goals and primitives**

Cryptographic systems are built from a few basic primitives that are often combined in protocols and applications.

- **Encryption** protects confidentiality by converting plaintext into ciphertext with a key.
- **Hashing** produces a fixed-length digest for integrity checking and password storage support.
- **MACs** (Message Authentication Codes) provide integrity and authenticity with a shared secret.
- **Digital signatures** provide integrity, authenticity, and nonrepudiation using asymmetric keys.
- **Key exchange** lets two parties establish shared secrets over an untrusted network.

A useful exam distinction is that hashes alone do not authenticate a sender. A hash detects change, but anyone can compute it. A MAC authenticates because only parties with the shared secret can generate a valid tag. A signature authenticates because only the private key holder can sign.

**Symmetric cryptography**

Symmetric encryption uses the same secret key, or a closely related key, for encryption and decryption. It is fast and suitable for bulk data, disk encryption, VPN payloads, and session traffic.

Common symmetric algorithms and modes:

- **AES** (Advanced Encryption Standard) is the modern baseline symmetric cipher.
- **3DES** is legacy and slow; it is generally deprecated.
- **ChaCha20** is a modern stream cipher often paired with Poly1305 for authenticated encryption.
- **CBC** (Cipher Block Chaining) is older and error-prone if used without integrity protection.
- **GCM** (Galois/Counter Mode) is an authenticated encryption mode widely used in TLS.
- **CTR** (Counter mode) turns a block cipher into a stream-like construction but needs separate integrity protection.

Practical guidance:

- Prefer authenticated encryption such as **AES-GCM** or **ChaCha20-Poly1305**.
- Avoid using encryption without integrity protection.
- Never reuse an initialization vector (IV) or nonce where the algorithm requires uniqueness.

Example OpenSSL encryption with AES-GCM:

```bash
openssl enc -aes-256-gcm -salt -pbkdf2 -iter 100000 -in secret.txt -out secret.enc
```

Example OpenSSL symmetric encryption with AES-CBC for legacy interoperability only:

```bash
openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000 -in secret.txt -out secret.enc
```

> CBC mode is vulnerable to padding-oracle style issues if the application leaks decryption errors or timing differences. In modern designs, prefer AEAD (Authenticated Encryption with Associated Data).

**Key length, security strength, and algorithm choice over time**

CISSP expects you to understand that key length is not the same as security strength, but they are related. Security strength is the estimated work factor needed to break the system.

Typical exam-relevant equivalences:

- **AES-128** is generally considered strong for most uses and offers 128-bit security strength.
- **AES-256** offers a higher margin and is often chosen for long-term protection or policy requirements.
- **RSA** needs much larger keys to reach comparable strength.
- **ECC** (Elliptic Curve Cryptography) achieves similar security with much smaller keys than RSA.

Common equivalence examples:

| Approximate security strength | RSA key size | ECC key size |
|---|---:|---:|
| 128-bit | 3072-bit | 256-bit |
| 192-bit | 7680-bit | 384-bit |
| 256-bit | 15360-bit | 512-bit |

Practical implications:

- **RSA-2048** is still widely seen, but many environments prefer **RSA-3072** for stronger long-term assurance.
- **ECC-256** is efficient and common in modern TLS and mobile environments.
- Larger keys increase computational cost, handshake time, and storage overhead.
- Algorithm choice changes over time as hardware improves and cryptanalytic guidance evolves.

**Approved, legacy, and compliance-driven cryptography**

In exam context, you should distinguish between algorithms that are merely available and those that are approved by policy or standards.

- **FIPS** (Federal Information Processing Standards) defines U.S. government requirements for certain cryptographic modules and algorithms.
- **NIST** (National Institute of Standards and Technology) publishes guidance, recommendations, and validation programs that heavily influence enterprise and government cryptography.
- **Approved algorithms** are those accepted by a policy or standard for a given use case.
- **Legacy algorithms** may still exist for compatibility but should not be used for new designs.

Examples of common status distinctions:

- **SHA-1** is legacy and collision-prone for many trust uses.
- **MD5** is broken for collision resistance and should not be used for security.
- **3DES** is legacy and generally phased out.
- **AES**, **SHA-2**, **SHA-3**, **HMAC**, and modern **ECC** curves are commonly accepted in current designs, subject to policy.

A CISSP-style answer often depends on context:

- If the question asks for a modern, approved choice, select current standards.
- If the question asks about compatibility with older systems, recognize that legacy algorithms may still appear but are not preferred.
- If the question mentions compliance, think about **FIPS-validated modules** and organizational policy, not just algorithm names.

**Hashing and password storage**

Hashes are one-way functions used for integrity, fingerprinting, and password storage support. They are not encryption.

Common hash functions:

- **SHA-256** and **SHA-384** are widely used in modern systems.
- **SHA-3** is a newer standard with a different internal design.
- **BLAKE2/BLAKE3** may appear in modern software, though CISSP usually emphasizes SHA-family functions.
- **MD5** and **SHA-1** are not suitable for security-sensitive collision-resistant uses.

Password storage should use slow, salted password hashing functions:

- **bcrypt**
- **scrypt**
- **Argon2**

Why this matters:

- Salts prevent precomputed rainbow table attacks.
- Slow hashing increases attacker cost.
- Password hashing is different from general-purpose hashing because it is intentionally expensive.

Example with `openssl passwd` for legacy-style demonstration only:

```bash
openssl passwd -6
```

Better practice in applications is to use a dedicated password hashing library rather than raw SHA-256.

**Message authentication codes and signatures**

A **MAC** proves that a message came from someone who knows the shared secret and that it was not altered in transit.

Common MAC constructions:

- **HMAC** uses a cryptographic hash plus a secret key.
- **CMAC** uses a block cipher, typically AES, to produce a MAC.
- **Poly1305** is often used with ChaCha20 in modern AEAD schemes.

Use cases:

- API request authentication
- Message integrity between trusted peers
- Protocol-level integrity checks
- Token validation in systems that share a secret

A **digital signature** uses asymmetric cryptography and a private key to sign data.

Use cases:

- Software signing
- Certificate signing
- Document signing
- Nonrepudiation scenarios

Key distinction:

- MACs are faster and simpler but require shared secret distribution.
- Signatures scale better for trust distribution because verifiers only need the public key.

Example HMAC generation:

```bash
openssl dgst -sha256 -hmac "shared-secret" message.txt
```

Example CMAC conceptually in application code:

```text
CMAC = AES-CMAC(key, message)
```

**Asymmetric cryptography**

Asymmetric cryptography uses a public/private key pair. It is slower than symmetric cryptography but solves key distribution and trust problems.

Common asymmetric uses:

- Key exchange
- Digital signatures
- Certificate-based identity
- Hybrid encryption systems

Common algorithms:

- **RSA** for encryption and signatures
- **ECC** for signatures and key exchange
- **Diffie-Hellman** for key agreement
- **ElGamal** in some niche or legacy contexts

Practical exam points:

- Asymmetric cryptography is rarely used to encrypt large data directly.
- It is usually used to protect or establish symmetric session keys.
- Hybrid designs combine asymmetric setup with symmetric bulk encryption.

**Key exchange and forward secrecy**

Key exchange lets two parties derive a shared secret without sending the secret itself over the network.

Important concepts:

- **Diffie-Hellman (DH)** establishes a shared secret over an insecure channel.
- **Ephemeral DH** uses temporary keys for a session.
- **Static DH** reuses long-term keys.
- **Forward secrecy** means compromise of a long-term private key does not reveal past session keys.

Why ephemeral keys matter:

- If you use ephemeral key exchange, each session gets a fresh secret.
- If a server private key is later stolen, recorded traffic is still harder to decrypt.
- Static key exchange increases the blast radius of key compromise.

Common protocol terms:

- **DHE** means ephemeral Diffie-Hellman.
- **ECDHE** means ephemeral Elliptic Curve Diffie-Hellman.
- Modern TLS commonly uses **ECDHE** to provide forward secrecy.

Example OpenSSL TLS connection showing negotiated cipher and key exchange:

```bash
openssl s_client -connect example.com:443 -tls1_2
```

You can inspect the negotiated cipher suite to see whether the session uses ephemeral key exchange and authenticated encryption.

**Certificates and public key infrastructure**

A **digital certificate** binds a public key to an identity using a trusted certificate authority (CA). Certificates are central to TLS, code signing, and enterprise identity systems.

Core PKI (Public Key Infrastructure) components:

- **CA** issues and signs certificates.
- **RA** (Registration Authority) validates identity before issuance in some workflows.
- **Certificate repository** stores published certificates.
- **CRL** (Certificate Revocation List) publishes revoked certificates.
- **OCSP** (Online Certificate Status Protocol) provides near-real-time revocation status.

Certificate fields and concepts:

- Subject and subject alternative name (SAN)
- Issuer
- Serial number
- Validity period
- Public key
- Signature algorithm
- Key usage and extended key usage

Validation process:

1. Build a chain from the leaf certificate to a trusted root.
2. Verify each signature in the chain.
3. Check validity dates.
4. Check hostname or identity binding.
5. Check revocation status where policy requires it.
6. Enforce key usage and extended key usage constraints.

Common TLS trust failures:

- The certificate is expired.
- The hostname does not match the SAN.
- The issuing CA is not trusted.
- The chain is incomplete or misordered.
- An intermediate certificate is missing from the server.
- The certificate uses an unsupported algorithm or weak key.
- The client cannot reach revocation services and policy requires strict checking.

Chain-building pitfalls:

- Servers often forget to send the intermediate certificate.
- Clients may have multiple possible chains and choose the wrong one if trust stores are inconsistent.
- Cross-signed roots can create confusing alternate paths.
- A certificate may validate in one environment but fail in another because trust stores differ.

Certificate pinning:

- Pinning binds an application to a specific certificate, public key, or CA.
- It reduces reliance on the broader PKI ecosystem.
- It can block man-in-the-middle attacks if the pin is correct.
- It can also cause outages when certificates rotate unexpectedly.

> Pinning is operationally risky if you do not plan for rotation, backup pins, and recovery procedures.

Example OpenSSL certificate inspection:

```bash
openssl x509 -in server.crt -text -noout
openssl verify -CAfile ca-chain.pem server.crt
```

Example TLS server configuration snippet for Nginx:

```nginx
server {
    listen 443 ssl http2;
    server_name app.example.com;

    ssl_certificate     /etc/nginx/certs/fullchain.pem;
    ssl_certificate_key /etc/nginx/certs/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
}
```

Example Apache TLS configuration:

```apache
<VirtualHost *:443>
    ServerName app.example.com
    SSLEngine on
    SSLCertificateFile /etc/pki/tls/certs/fullchain.pem
    SSLCertificateKeyFile /etc/pki/tls/private/privkey.pem
    SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1
</VirtualHost>
```

**Revocation, CRL, and OCSP behavior**

Revocation answers the question: “Has this certificate been invalidated before its expiration date?”

Two common mechanisms:

- **CRL** publishes a signed list of revoked certificate serial numbers.
- **OCSP** lets a client query the status of a specific certificate.

Important edge cases:

- **CRL distribution points** are URLs embedded in certificates that tell clients where to fetch revocation lists.
- **OCSP responder trust** matters because the client must trust the responder’s status information.
- **Soft-fail** means the client accepts the certificate if revocation checking cannot be completed.
- **Hard-fail** means the client rejects the certificate if revocation status cannot be confirmed.

Exam-oriented implications:

- Soft-fail improves availability but weakens security.
- Hard-fail improves security but can break access when revocation infrastructure is unreachable.
- Some browsers and applications use stapling or caching to reduce latency and privacy leakage.

OCSP stapling:

- The server fetches a signed OCSP response and “staples” it to the TLS handshake.
- This reduces client-side OCSP lookups.
- It improves privacy because the client does not contact the CA directly for every connection.

Example OpenSSL OCSP query:

```bash
openssl ocsp -issuer issuer.crt -cert server.crt -url http://ocsp.example-ca.com
```

**Key management and lifecycle governance**

Cryptography fails more often because of poor key management than because of weak algorithms. CISSP expects you to understand the full lifecycle.

Key lifecycle stages:

- Generation
- Distribution
- Storage
- Use
- Rotation
- Backup and recovery
- Revocation
- Destruction

Operational controls:

- Use a **key ceremony** for high-value root keys and CA keys.
- Enforce **separation of duties** so no single person can create, approve, and export sensitive keys alone.
- Store private keys in an **HSM** (Hardware Security Module) when possible.
- Limit key export and require dual control for sensitive operations.
- Rotate keys on schedule and after suspected compromise.
- Destroy retired keys securely so they cannot be recovered.

Key ceremony considerations:

- Document who is present.
- Record serial numbers and hashes.
- Use tamper-evident controls.
- Verify the environment before generating or importing keys.
- Keep audit logs and chain-of-custody records.

Key backup and escrow risks:

- Backups improve recoverability if a key is lost.
- Backups also increase exposure if they are stolen.
- Escrow can help with enterprise recovery or legal access, but it creates a high-value target.
- If you escrow encryption keys, you may weaken confidentiality if the escrow process is not tightly controlled.

HSM and CA administration:

- HSMs protect private keys from direct extraction.
- CA administrators should not also be the only approvers for certificate issuance.
- Root CA keys are often kept offline to reduce attack surface.
- Intermediate CAs handle day-to-day issuance and can be rotated more easily.

**Cryptographic attacks and misuse patterns**

CISSP often tests whether you can recognize attack classes and the conditions that enable them.

Common attack concepts:

- **Brute force** tries all possible keys or passwords.
- **Dictionary attacks** try likely passwords or passphrases.
- **Chosen-plaintext attack** lets the attacker encrypt known plaintexts and study outputs.
- **Chosen-ciphertext attack** lets the attacker submit crafted ciphertexts for decryption and observe behavior.
- **Replay attack** reuses captured valid messages.
- **Downgrade attack** forces a protocol to use weaker algorithms or older versions.
- **Man-in-the-middle (MITM)** intercepts and potentially alters traffic between two parties.

Why these matter:

- Weak protocol negotiation can allow downgrade attacks.
- Missing nonces, timestamps, sequence numbers, or session binding can allow replay.
- Error messages and timing differences can create oracle behavior for chosen-ciphertext attacks.
- Legacy protocol support can reintroduce known weaknesses.

Replay protection mechanisms:

- Nonces
- Timestamps
- Sequence numbers
- Session identifiers
- Expiration windows
- Signed tokens with freshness checks

Example of a signed API request pattern with nonce and timestamp:

```http
POST /transfer HTTP/1.1
Host: api.example.com
X-Nonce: 8f3c2d1a
X-Timestamp: 1714147200
X-Signature: base64(hmac_sha256(secret, method|path|nonce|timestamp|body))
```

**TLS and protocol configuration examples**

TLS is the most common real-world cryptographic protocol in enterprise systems. It combines certificates, key exchange, symmetric encryption, and integrity protection.

Modern configuration goals:

- Use TLS 1.2 or TLS 1.3.
- Prefer AEAD cipher suites.
- Prefer ephemeral key exchange.
- Disable weak protocols and ciphers.
- Use strong certificate chains and proper hostname validation.

Example OpenSSL test of a specific TLS version:

```bash
openssl s_client -connect www.example.com:443 -tls1_3
```

Example `curl` request with certificate verification:

```bash
curl --tlsv1.3 --cacert ca.pem https://app.example.com
```

Example `curl` with client certificate authentication:

```bash
curl --cert client.crt --key client.key https://vpn.example.com
```

Example Java trust store import:

```bash
keytool -importcert -alias corp-ca -file corp-ca.crt -keystore truststore.jks
```

Example Linux trust store update pattern:

```bash
cp corp-ca.crt /usr/local/share/ca-certificates/
update-ca-certificates
```

**Common protocol and deployment pitfalls**

Real systems fail when cryptography is configured incorrectly, even if the underlying algorithms are strong.

Typical pitfalls:

- Using self-signed certificates in production without a controlled trust model.
- Trusting any certificate that chains to a broadly trusted root without hostname validation.
- Reusing IVs or nonces in modes that require uniqueness.
- Using RSA for bulk encryption instead of hybrid encryption.
- Accepting weak cipher suites for compatibility.
- Leaving legacy protocols enabled for convenience.
- Failing to rotate keys after personnel changes or suspected compromise.
- Storing private keys on shared file systems without access controls.

**Exam cues and decision rules**

When you see a CISSP cryptography question, use these rules of thumb:

- If the goal is confidentiality at scale, choose symmetric encryption.
- If the goal is identity and nonrepudiation, choose digital signatures.
- If the goal is integrity with a shared secret, choose a MAC.
- If the goal is secure session setup, choose ephemeral key exchange and authenticated encryption.
- If the question mentions compliance, think **FIPS** and **NIST** guidance.
- If the question mentions trust failures, inspect certificate chain, hostname, revocation, and trust store issues.
- If the question mentions long-term protection, consider key length, algorithm strength, and future cryptanalytic margin.

**Quick comparison of common primitives**

| Primitive | Main purpose | Shared secret required | Nonrepudiation | Typical use |
|---|---|---:|---:|---|
| Symmetric encryption | Confidentiality | Yes | No | Data at rest, VPN payloads |
| Hash | Integrity fingerprint | No | No | File checks, password support |
| HMAC/CMAC | Integrity + authenticity | Yes | No | API auth, message protection |
| Digital signature | Integrity + authenticity | No | Yes | Certificates, code signing |
| Key exchange | Shared secret setup | No | No | TLS, secure channels |

**Practical study reminders**

- Memorize the difference between hash, MAC, and signature.
- Know why AES-GCM and ECDHE are preferred in modern TLS.
- Understand why certificate validation is more than “is it signed?”
- Be able to explain soft-fail versus hard-fail revocation behavior.
- Remember that key management is a governance problem as much as a technical one.
- Recognize that legacy algorithms may still exist, but approved modern choices are preferred for new designs.
