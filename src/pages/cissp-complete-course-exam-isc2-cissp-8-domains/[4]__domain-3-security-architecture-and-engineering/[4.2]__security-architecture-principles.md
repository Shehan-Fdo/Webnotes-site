## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Security architecture principles overview**

Security architecture principles guide how you design systems so they remain secure under real-world constraints. In CISSP terms, you are not just memorizing controls; you are learning how to choose designs that reduce risk, preserve trust boundaries, and support business goals without creating unnecessary complexity.

A strong architecture balances confidentiality, integrity, and availability while also considering usability, maintainability, scalability, interoperability, and portability. Exam questions often present tradeoffs, so you should identify the principle that best fits the scenario rather than selecting the most restrictive control by default.

- **Security by design** means security is built into the architecture from the start.
- **Security by obscurity** relies on hiding implementation details and is not a primary defense.
- **Defense in depth** layers controls so one failure does not expose the entire system.
- **Least privilege** gives subjects only the access required to perform their tasks.
- **Least common mechanism** reduces shared components to limit unintended interaction.
- **Fail-safe defaults** deny access unless explicitly allowed.
- **Economy of mechanism** favors simple, small, and understandable designs.
- **Complete mediation** checks every access to every object every time.
- **Open design** assumes the design is public; security depends on keys and controls, not secrecy.

**Core CISSP security design principles**

The standard principle set is often tested as a group. You should be able to recognize each principle and explain why it matters in architecture decisions.

- **Simplicity / economy of mechanism**
  - Keep the design as small and understandable as possible.
  - Simpler systems are easier to test, audit, and secure.
  - Complex control paths increase the chance of logic errors and hidden dependencies.

- **Minimization**
  - Reduce the number of components, services, permissions, and exposed interfaces.
  - Minimize attack surface by removing unused software, ports, accounts, and APIs.
  - Apply this to hosts, containers, network paths, and identity roles.

- **Layering / defense in depth**
  - Place multiple independent controls at different layers.
  - Use network filtering, endpoint hardening, application controls, and monitoring together.
  - Layering helps when one control fails or is bypassed.

- **Least privilege**
  - Grant only the permissions needed for the current task.
  - Use role-based access control (RBAC), just-in-time access, and scoped service accounts.
  - Review privileges regularly to prevent privilege creep.

- **Separation of duties**
  - Split critical tasks across multiple people or roles.
  - Prevent one person from approving, executing, and auditing the same sensitive action.
  - Common in finance, change management, and key administration.

- **Need to know**
  - Restrict access to information only to those who require it for their job.
  - This is narrower than role membership alone and often appears in classified or sensitive environments.

- **Fail-safe defaults**
  - Default to deny when a rule, policy, or condition is missing.
  - This is the safer choice for access control, firewall rules, and authorization logic.

- **Complete mediation**
  - Validate access on every request, not just the first one.
  - Do not rely on cached authorization if the risk model requires continuous enforcement.

- **Open design**
  - Do not depend on secrecy of the architecture for security.
  - Use strong cryptography, access control, and monitoring instead of hidden implementation details.

- **Psychological acceptability**
  - Security controls should be usable and intuitive.
  - If controls are too hard to use, users bypass them or create shadow processes.

- **Work factor**
  - Make the cost of attack exceed the value of the target.
  - Increase attacker effort with stronger controls, segmentation, and monitoring.

- **Compromise recording**
  - Detect and log security violations so you can investigate and respond.
  - This principle supports auditability and incident response.

**Architecture tradeoffs and evaluation criteria**

CISSP architecture questions often ask which design is best under competing requirements. You should evaluate the system against operational and business criteria, not only security strength.

| Criterion | What it means | Exam relevance |
|---|---|---|
| **Scalability** | The design can grow without major redesign | Important for cloud, enterprise identity, and distributed systems |
| **Interoperability** | Different systems can work together | Common in federated identity, APIs, and multi-vendor environments |
| **Portability** | The system can move across platforms or environments | Relevant for containers, virtual machines, and cloud migration |
| **Availability** | The system remains accessible when needed | Often tied to redundancy, failover, and resilience |
| **Maintainability** | The system can be updated and supported efficiently | Simpler designs usually win here |
| **Auditability** | Actions can be traced and reviewed | Important for accountability and compliance |
| **Usability** | Users can operate the system correctly | Poor usability often weakens security in practice |

A secure design is not always the most restrictive design. For example, a highly segmented system may improve confidentiality but reduce interoperability and operational agility. On the exam, choose the answer that best matches the stated business requirement.

**Security boundaries, trust, and controlled communication**

A **security boundary** is the point where trust changes. It separates one protection domain from another, such as a user workstation from a server, a guest VM from a host, or an internal network from an external one. A boundary is meaningful only if it is enforced by policy, access control, or isolation.

- **Perimeter** refers to the outer edge of a protected environment.
- **Boundary** refers to any point where access rules change.
- **Trust boundary** is where data or control crosses from one trust level to another.

Common boundary examples include:
- A firewall between a user network and a server network.
- A hypervisor boundary between virtual machines.
- An API gateway between external clients and internal services.
- A container runtime boundary between containers and the host OS.

**Trusted path** and **trusted channel** are frequently confused, but they are different.

- **Trusted path**
  - A secure, direct path between the user and the trusted computing base (TCB).
  - Used when the user must be sure they are interacting with the real security mechanism.
  - Example: pressing <kbd>Ctrl</kbd> + <kbd>Alt</kbd> + <kbd>Del</kbd> on some systems to reach a secure logon screen.
  - Example: a hardware-backed secure attention sequence that prevents a fake login prompt from intercepting credentials.

- **Trusted channel**
  - A secure communication path between two trusted entities.
  - Used when systems exchange sensitive data and need confidentiality and integrity.
  - Example: TLS between a client and an internal API.
  - Example: an IPsec tunnel between two sites carrying administrative traffic.

Practical examples:
- A user authenticates through a secure desktop that bypasses normal applications, creating a trusted path to the OS login mechanism.
- A payment service sends transaction data over mutual TLS, creating a trusted channel between application endpoints.
- An admin connects to a management interface through an out-of-band network with certificate-based authentication, reducing exposure to the production network.

> A boundary is only as strong as its enforcement. If traffic can bypass the control point, the boundary is not effective.

**Isolation: logical vs physical**

Isolation reduces the chance that one workload, user, or process can affect another. CISSP questions often ask you to distinguish between logical and physical isolation.

- **Physical isolation**
  - Separation by hardware or location.
  - Examples:
    - Separate air-gapped network for highly sensitive systems.
    - Dedicated hardware security module (HSM) for key storage.
    - Separate servers or racks for regulated workloads.
  - Stronger against some classes of attack, but more expensive and less flexible.

- **Logical isolation**
  - Separation enforced by software or configuration on shared hardware.
  - Examples:
    - Virtual private cloud (VPC) subnets and security groups.
    - Virtual machines on the same host.
    - Containers with namespace and cgroup isolation.
    - VLANs on shared switching infrastructure.
  - More scalable and cost-effective, but depends on correct configuration and platform security.

Use physical isolation when the risk, regulation, or threat model justifies the cost. Use logical isolation when you need flexibility, density, and easier operations, but ensure the shared platform is hardened and monitored.

**Redundancy and fault handling**

Redundancy supports availability and resilience, but CISSP expects you to distinguish the stages of fault management.

- **Fault avoidance**
  - Design the system to prevent faults from occurring.
  - Examples:
    - Use mature components and tested configurations.
    - Apply change control and code review.
    - Remove single points of failure before deployment.

- **Fault detection**
  - Identify that a fault has occurred.
  - Examples:
    - Monitoring, health checks, alerts, and integrity checks.
    - Heartbeats between clustered nodes.
    - Log analysis and anomaly detection.

- **Fault recovery**
  - Restore service after a fault.
  - Examples:
    - Failover to a standby node.
    - Restore from backup.
    - Rebuild a failed container or VM from a known-good image.

Related redundancy patterns:
- **Active-active** systems share load and provide resilience if one node fails.
- **Active-passive** systems keep a standby ready to take over.
- **N+1** means one extra component beyond the minimum required.
- **Load balancing** improves availability and can support maintenance without downtime.

Redundancy improves availability, but it can also increase complexity. If the design becomes too complex to operate correctly, the benefit may be reduced.

**Formal security models**

Formal models define how access and information flow should work. On the exam, choose the model that matches the security objective.

- **Bell-LaPadula (BLP)**
  - Focuses on **confidentiality**.
  - Key ideas:
    - No read up.
    - No write down.
  - Use when preventing lower-clearance subjects from reading higher-classified data is the goal.
  - Example: military classification systems.

- **Biba**
  - Focuses on **integrity**.
  - Key ideas:
    - No read down.
    - No write up.
  - Use when you want to prevent low-integrity data from contaminating trusted processes.
  - Example: protecting a financial ledger from untrusted inputs.

- **Clark-Wilson**
  - Focuses on **integrity** through well-formed transactions and separation of duties.
  - Uses:
    - Constrained data items (CDIs)
    - Unconstrained data items (UDIs)
    - Transformation procedures
    - Integrity verification procedures
  - Use when business process integrity matters more than strict classification.
  - Example: accounting systems, ERP workflows, and transaction processing.

- **Brewer-Nash / Chinese Wall**
  - Prevents conflicts of interest by restricting access based on prior access history.
  - Use when one consultant must not access competing client data.
  - Example: legal, consulting, and financial advisory environments.

- **Noninterference**
  - Actions in one security domain should not affect what another domain can observe.
  - Useful for reasoning about information flow and covert leakage.

Exam-oriented selection:
- Choose **Bell-LaPadula** for classified confidentiality scenarios.
- Choose **Biba** when integrity contamination is the main risk.
- Choose **Clark-Wilson** for business transaction integrity and separation of duties.
- Choose **Brewer-Nash** when conflict-of-interest restrictions matter.

**Covert channels and information flow**

A covert channel is an unintended communication path that can leak information outside approved channels. CISSP often tests whether you can recognize that a system may be secure at the access-control layer but still leak data through side effects.

- **Storage channel**
  - Information is transmitted by changing shared storage or state.
  - Example: one process encodes data by altering file attributes, cache state, or shared memory values.
  - Example: a container writes to a shared resource that another container can observe.

- **Timing channel**
  - Information is transmitted by modulating response times or resource usage.
  - Example: a process delays responses to encode bits.
  - Example: cache timing differences reveal whether data is present.

Mitigation strategies:
- Reduce shared resources.
- Enforce strong isolation.
- Randomize or normalize timing where practical.
- Monitor for unusual resource contention.
- Apply **least common mechanism** to minimize shared state.

Covert channels are especially relevant when you need strong isolation between tenants, workloads, or security domains.

**Security mechanisms and implementation controls**

Architecture principles become real through mechanisms that enforce them.

- **Reference monitor**
  - Conceptual control that mediates all access to objects.
  - Must be tamper-resistant, always invoked, and small enough to verify.
  - Often implemented through kernel security mechanisms, hypervisors, or policy engines.

- **Security kernel**
  - The hardware/software core that enforces the security policy.
  - Should be minimal and isolated from less trusted code.

- **Trusted computing base (TCB)**
  - The set of components that must be trusted to enforce security correctly.
  - Smaller TCBs are easier to validate and secure.

- **Isolation mechanisms**
  - Virtualization, containers, sandboxing, and microsegmentation reduce blast radius.
  - Strong isolation supports multi-tenant and zero-trust designs.

- **Access control models**
  - DAC (Discretionary Access Control) lets owners decide access.
  - MAC (Mandatory Access Control) enforces centrally defined labels and rules.
  - RBAC (Role-Based Access Control) maps permissions to roles.
  - ABAC (Attribute-Based Access Control) evaluates attributes of subject, object, action, and context.

**Security design in enterprise scenarios**

Architecture principles show up in everyday enterprise decisions.

- A finance team uses separate approval and payment roles to enforce separation of duties.
- A cloud platform uses security groups, subnets, and IAM roles to create layered controls.
- A healthcare system uses encryption, audit logging, and least privilege to protect patient records.
- A multi-tenant SaaS platform uses logical isolation, per-tenant keys, and strict API authorization to reduce cross-tenant risk.
- A regulated environment may add physical separation for key management or sensitive processing.

When evaluating a scenario, ask:
- What is the primary security goal: confidentiality, integrity, or availability?
- Is the main risk misuse, compromise, or leakage?
- Does the design need to scale, interoperate, or move across platforms?
- Is the control preventing access, detecting misuse, or recovering from failure?

**Exam cues and common traps**

- If the question emphasizes secrecy of design details, remember that **open design** is preferred over security by obscurity.
- If the question emphasizes fewer moving parts, choose **simplicity** or **minimization**.
- If the question emphasizes multiple layers of control, choose **defense in depth**.
- If the question emphasizes secure login or direct interaction with the OS security mechanism, think **trusted path**.
- If the question emphasizes secure communication between systems, think **trusted channel**.
- If the question emphasizes preventing data leakage across domains, think **noninterference**, **covert channels**, and **least common mechanism**.
- If the question emphasizes business process integrity, think **Clark-Wilson**.
- If the question emphasizes classified confidentiality, think **Bell-LaPadula**.
- If the question emphasizes data integrity against untrusted inputs, think **Biba**.

**Quick comparison of common principles**

| Principle | Primary goal | Typical exam clue |
|---|---|---|
| **Least privilege** | Limit permissions | “Only what is needed” |
| **Separation of duties** | Prevent fraud or abuse | “No single person can complete the action” |
| **Defense in depth** | Multiple barriers | “Layered controls” |
| **Economy of mechanism** | Simplicity | “Small, simple, understandable design” |
| **Fail-safe defaults** | Deny by default | “Access denied unless explicitly allowed” |
| **Complete mediation** | Check every access | “Every request is validated” |
| **Open design** | Avoid secrecy dependence | “Design is public, keys are secret” |
| **Least common mechanism** | Reduce shared components | “Minimize shared resources” |

**Practical architecture examples**

A secure web application architecture might include:
- A reverse proxy at the edge.
- TLS termination with strong certificates.
- An application firewall or API gateway.
- Separate application and database subnets.
- Role-based access for administrators.
- Centralized logging and monitoring.
- Per-service identities and least-privilege secrets access.

A secure admin access pattern might include:
- A bastion host or privileged access workstation.
- MFA (Multi-Factor Authentication).
- A trusted channel such as VPN or mutual TLS.
- A trusted path to the authentication mechanism.
- Session recording and audit logs.

A high-assurance environment might include:
- Physical separation for key material.
- MAC enforcement for sensitive data.
- Minimal TCB.
- Formal change control.
- Redundant systems with tested recovery procedures.

**Key takeaways**

Security architecture principles help you choose designs that are secure, usable, and supportable. For CISSP, focus on the relationship between the principle and the scenario: confidentiality versus integrity, simplicity versus complexity, logical versus physical isolation, and prevention versus detection versus recovery.

If you can identify the security boundary, the trust relationship, the formal model, and the tradeoff being tested, you can usually eliminate distractors quickly.
