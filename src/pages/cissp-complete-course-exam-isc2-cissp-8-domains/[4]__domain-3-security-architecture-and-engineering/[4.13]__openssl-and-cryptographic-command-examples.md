## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**OpenSSL in security operations**

OpenSSL is a widely used command-line toolkit for cryptography, certificate handling, and TLS (Transport Layer Security) testing. In exam and admin contexts, you use it to generate keys, create CSRs (certificate signing requests), inspect certificates, validate chains, test servers, and convert between common formats.

It is important to distinguish between what OpenSSL can do and what it proves. A command may confirm syntax, parse a certificate, or verify a signature, but it does not automatically establish application trust, hostname trust, revocation status, or policy compliance unless you explicitly configure those checks.

- Use OpenSSL for **inspection** when you need to read certificate fields, extensions, or key details.
- Use it for **validation** when you need to test signatures, chains, or TLS negotiation.
- Use it for **conversion** when you need PEM, DER, PKCS#12, or bundle extraction workflows.

**Core file formats and terminology**

OpenSSL workflows often depend on the file format you are handling. The same certificate or key can appear in different encodings, and the wrong assumption is a common source of failure.

- **PEM** is Base64 text with header and footer lines such as `-----BEGIN CERTIFICATE-----`.
- **DER** is binary encoding, often used by Java and some Windows tooling.
- **PKCS#12** (`.p12` or `.pfx`) is a container that can hold a private key, certificate, and chain.
- **CSR** is a request sent to a CA (Certificate Authority) to obtain a signed certificate.
- **CA bundle** is a file or directory of trusted root certificates used for verification.

Typical inspection commands:

```bash
openssl x509 -in server.crt -text -noout
openssl pkey -in server.key -text -noout
openssl req -in server.csr -text -noout
```

**Generating keys and protecting private keys**

Private keys should normally be encrypted at rest unless a specific automation workflow requires otherwise. In OpenSSL, `-aes256` or similar options encrypt the private key with a passphrase. The historical `-nodes` flag means “no DES” in older documentation, but in modern OpenSSL usage it effectively means “do not encrypt the private key.”

> Be careful with `-nodes`: it writes the private key unencrypted. That is useful for some automated bootstrapping workflows, but it increases exposure if the file is stolen.

Generate an RSA private key and protect it with a passphrase:

```bash
openssl genpkey -algorithm RSA -aes256 -out server.key -pkeyopt rsa_keygen_bits:3072
```

Generate an EC (elliptic curve) private key:

```bash
openssl genpkey -algorithm EC -out ec.key -pkeyopt ec_paramgen_curve:P-256
```

Create a CSR from a private key:

```bash
openssl req -new -key server.key -out server.csr
```

Create a self-signed certificate for lab use:

```bash
openssl req -x509 -new -key server.key -sha256 -days 365 -out server.crt
```

If you need an unencrypted export for a controlled automation task, you can remove the passphrase:

```bash
openssl pkey -in server.key -out server-unencrypted.key
```

**Certificate inspection and extension review**

Certificate inspection is one of the most common OpenSSL tasks. You often need to confirm the subject, issuer, validity period, serial number, and extensions such as SAN (Subject Alternative Name), Key Usage, and Extended Key Usage.

```bash
openssl x509 -in server.crt -noout -subject -issuer -dates -serial
openssl x509 -in server.crt -noout -text
```

Look for these fields in the output:

- **Subject** identifies the certificate owner.
- **Issuer** identifies the signing CA.
- **Not Before / Not After** define the validity window.
- **SAN** lists DNS names and IP addresses the certificate covers.
- **Key Usage** limits allowed cryptographic operations.
- **Extended Key Usage** narrows the certificate to roles such as server authentication or client authentication.

Extract only the SAN extension:

```bash
openssl x509 -in server.crt -noout -ext subjectAltName
```

Extract only key usage information:

```bash
openssl x509 -in server.crt -noout -ext keyUsage -ext extendedKeyUsage
```

A certificate may be valid but still unusable for a specific purpose if the required extension is missing. For example, a TLS server certificate should usually include `serverAuth` in EKU and the correct DNS names in SAN.

**Chain validation and trust stores**

OpenSSL verifies certificate chains against trusted roots and, optionally, intermediate certificates. This is useful for troubleshooting trust failures, but it does not by itself confirm hostname matching or revocation unless you request those checks.

```bash
openssl verify -CAfile root-ca.pem -untrusted intermediate.pem server.crt
```

This checks whether the chain can be built to a trusted root using the provided intermediate certificate. It does not automatically verify the hostname used by an application.

For hostname-aware TLS verification, combine trust validation with hostname checks in client testing tools:

```bash
openssl s_client -connect example.com:443 -servername example.com -verify_hostname example.com -CAfile /etc/ssl/certs/ca-certificates.crt
```

Trust store locations vary by operating system and distribution:

- **Debian/Ubuntu** commonly use `/etc/ssl/certs/ca-certificates.crt`
- **RHEL/CentOS/Fedora** commonly use `/etc/pki/tls/certs/ca-bundle.crt`
- **macOS** and **Windows** often rely on platform trust stores rather than a single file
- **Custom environments** may use `-CAfile` for a bundle or `-CApath` for a hashed directory

If verification fails, check:

- Whether the correct root CA is trusted
- Whether the intermediate certificate is supplied
- Whether the certificate is expired or not yet valid
- Whether the hostname matches the SAN
- Whether the certificate purpose matches the intended use

**Certificate signing and simple CA workflows**

OpenSSL can act as a simple CA for lab or internal workflows. The `ca` command signs CSRs using a CA private key and certificate, and it can maintain a basic index and serial number database.

A minimal CA signing flow often looks like this:

```bash
openssl req -new -x509 -days 3650 -key ca.key -out ca.crt
openssl req -new -key server.key -out server.csr
openssl ca -config openssl.cnf -in server.csr -out server.crt
```

A more explicit signing command using `x509` for a lab certificate is:

```bash
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out server.crt -days 365 -sha256
```

Use `ca` when you want CA database tracking and policy enforcement. Use `x509 -req` for simpler lab signing tasks. In production, certificate issuance is usually handled by a dedicated PKI (Public Key Infrastructure) system or CA service.

**Converting between PEM, DER, and PKCS#12**

Format conversion is a frequent administrative task. Many interoperability issues are simply encoding mismatches.

Convert PEM certificate to DER:

```bash
openssl x509 -in server.crt -outform der -out server.der
```

Convert DER certificate to PEM:

```bash
openssl x509 -in server.der -inform der -out server.pem
```

Convert a private key from PEM to DER:

```bash
openssl pkey -in server.key -outform der -out server.key.der
```

Convert a certificate and private key into PKCS#12:

```bash
openssl pkcs12 -export -inkey server.key -in server.crt -certfile intermediate.pem -out server.p12
```

Extract a certificate from PKCS#12:

```bash
openssl pkcs12 -in server.p12 -clcerts -nokeys -out server.crt
```

Extract the private key from PKCS#12:

```bash
openssl pkcs12 -in server.p12 -nocerts -nodes -out server.key
```

> When exporting from PKCS#12, `-nodes` again means the extracted private key is written without encryption. Use it only when you need an unencrypted key for a controlled workflow.

**Extracting certificates and keys from bundles**

Administrators often receive bundles containing multiple certificates and keys. OpenSSL can split them for inspection or deployment.

View the contents of a PKCS#12 bundle:

```bash
openssl pkcs12 -in bundle.p12 -info -noout
```

Extract only the CA chain:

```bash
openssl pkcs12 -in bundle.p12 -cacerts -nokeys -out chain.pem
```

Split a PEM bundle manually by locating the `BEGIN` and `END` markers, then saving each object into its own file. This is common when a vendor supplies a single file containing multiple certificates.

**Signature verification and digest commands**

Digital signatures prove integrity and authenticity when you verify them with the correct public key. The public key must match the signer, and the file format must be appropriate for the command you use.

Verify a detached signature with a public key file:

```bash
openssl dgst -sha256 -verify public.pem -signature software.sig software.iso
```

This expects `public.pem` to contain a raw public key, not a certificate. If you only have a certificate, extract the public key first:

```bash
openssl x509 -in signer.crt -pubkey -noout > public.pem
```

Create a detached signature with a private key:

```bash
openssl dgst -sha256 -sign private.key -out software.sig software.iso
```

For file integrity checks, a digest is useful but not a signature. A hash confirms that data has not changed, but it does not prove who created it.

```bash
openssl dgst -sha256 software.iso
```

Other useful digest and utility commands:

```bash
openssl rand -hex 32
openssl rand -base64 32
openssl base64 -in file.bin -out file.b64
openssl base64 -d -in file.b64 -out file.bin
```

Use `openssl rand` to generate random bytes for salts, tokens, or test data. Use `base64` when you need to encode binary data for text transport.

**Public key encryption and hybrid encryption**

Public key encryption is usually used to protect a small secret, not a large file. In practice, you encrypt a random symmetric session key with the recipient’s public key, then use the symmetric key to encrypt the bulk data. This is the hybrid encryption pattern used by many real systems.

A common RSA-based pattern is:

1. Generate a random symmetric key.
2. Encrypt the data with the symmetric key.
3. Encrypt the symmetric key with the recipient’s public key.
4. Send both the encrypted data and the encrypted session key.

Example with AES (Advanced Encryption Standard) for the data and RSA-OAEP (Optimal Asymmetric Encryption Padding) for the key:

```bash
openssl rand -out session.key 32
openssl enc -aes-256-cbc -pbkdf2 -salt -in secret.txt -out secret.txt.enc -pass file:./session.key
openssl pkeyutl -encrypt -pubin -inkey recipient.pub -in session.key -out session.key.enc -pkeyopt rsa_padding_mode:oaep -pkeyopt rsa_oaep_md:sha256
```

Decrypt in reverse order:

```bash
openssl pkeyutl -decrypt -inkey recipient.key -in session.key.enc -out session.key -pkeyopt rsa_padding_mode:oaep -pkeyopt rsa_oaep_md:sha256
openssl enc -d -aes-256-cbc -pbkdf2 -in secret.txt.enc -out secret.txt -pass file:./session.key
```

For RSA encryption, specify padding explicitly. Raw RSA is not appropriate for general data encryption, and OAEP is the safer modern choice for many workflows.

**Symmetric encryption with `enc`**

The `enc` command is commonly used for file encryption and decryption. In modern OpenSSL, you should usually combine it with `-pbkdf2` and a strong iteration count because older default password-based key derivation behavior is legacy and weaker.

Encrypt a file with a passphrase:

```bash
openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000 -in report.txt -out report.txt.enc
```

Decrypt it:

```bash
openssl enc -d -aes-256-cbc -pbkdf2 -iter 100000 -in report.txt.enc -out report.txt
```

If you omit `-pass`, OpenSSL may prompt for a password depending on version and context, but exam prep should emphasize that modern workflows should explicitly use `-pbkdf2` rather than relying on legacy defaults.

**TLS client testing with `s_client`**

`s_client` is one of the most useful troubleshooting tools for TLS. It lets you inspect the certificate chain, negotiated protocol version, cipher suite, and server behavior.

Basic connection test:

```bash
openssl s_client -connect example.com:443 -servername example.com
```

Force a protocol version:

```bash
openssl s_client -connect example.com:443 -tls1_2 -servername example.com
openssl s_client -connect example.com:443 -tls1_3 -servername example.com
```

Check the negotiated cipher suite:

```bash
openssl s_client -connect example.com:443 -servername example.com -brief
```

Verify the hostname and trust chain together:

```bash
openssl s_client -connect example.com:443 -servername example.com -verify_hostname example.com -CAfile /etc/ssl/certs/ca-certificates.crt
```

Use `-verify_hostname` with trust validation enabled. Hostname checking alone is not enough if the certificate chain is not trusted.

Useful troubleshooting flags:

- `-showcerts` displays the full certificate chain sent by the server.
- `-state` shows handshake state transitions.
- `-msg` prints protocol messages for deeper debugging.
- `-cipher` restricts the cipher list for older protocol versions.
- `-alpn h2,http/1.1` tests Application-Layer Protocol Negotiation.

Example:

```bash
openssl s_client -connect example.com:443 -servername example.com -showcerts -alpn h2,http/1.1
```

**Local TLS testing with `s_server`**

`s_server` is useful for lab testing, certificate validation, and reproducing client issues without touching production systems.

Start a simple TLS server:

```bash
openssl s_server -accept 8443 -cert server.crt -key server.key
```

Test it from another terminal:

```bash
openssl s_client -connect 127.0.0.1:8443 -servername localhost
```

You can also test protocol and cipher behavior locally:

```bash
openssl s_server -accept 8443 -cert server.crt -key server.key -tls1_2
openssl s_client -connect 127.0.0.1:8443 -tls1_2
```

This is useful when you need to confirm whether a client and server can negotiate a specific protocol version or cipher suite.

**Key and certificate matching**

A common troubleshooting task is confirming that a certificate matches the private key used by the server. For RSA keys, you can compare the modulus, but this method is limited to RSA and should not be treated as a universal key-matching technique.

RSA-only comparison example:

```bash
openssl x509 -in server.crt -noout -modulus | openssl sha256
openssl rsa -in server.key -noout -modulus | openssl sha256
```

If the hashes match, the RSA certificate and RSA private key correspond. For EC or EdDSA keys, compare the public key material instead:

```bash
openssl x509 -in server.crt -pubkey -noout > cert.pub
openssl pkey -in server.key -pubout > key.pub
diff cert.pub key.pub
```

**Password and secret utilities**

OpenSSL also includes utilities that are useful in scripts and exam scenarios.

Generate a password hash for systems that use OpenSSL-compatible password formats:

```bash
openssl passwd -6
```

The `-6` option uses SHA-512 crypt format on many systems. This is not the same as encrypting a file; it is a password hash generation utility.

Generate random data for salts or test vectors:

```bash
openssl rand -hex 16
openssl rand -base64 24
```

These commands are useful when you need unpredictable values for lab work, tokens, or initialization data.

**Practical troubleshooting checklist**

When an OpenSSL task fails, work through the problem systematically.

1. Confirm the file type with `file` or by inspecting PEM headers.
2. Verify that the certificate, key, or CSR is the expected object.
3. Check whether the private key is encrypted and whether you have the passphrase.
4. Confirm that the certificate chain includes the correct intermediate certificates.
5. Verify SAN, EKU, and validity dates.
6. Check whether the trust store contains the correct root CA.
7. For TLS, confirm hostname, protocol version, and cipher compatibility.
8. For signatures, confirm that the public key matches the signer and that the correct digest algorithm is used.

Example diagnostic commands:

```bash
file server.crt server.key server.p12
openssl x509 -in server.crt -noout -text | sed -n '/Subject Alternative Name/,+2p'
openssl verify -CAfile root-ca.pem -untrusted intermediate.pem server.crt
openssl s_client -connect example.com:443 -servername example.com -brief
```

**Exam-focused distinctions**

OpenSSL questions often test whether you understand the difference between related operations.

- **Hashing** checks integrity, not identity.
- **Signing** provides integrity and authenticity.
- **Encryption** provides confidentiality.
- **Certificates** bind a public key to an identity through a CA signature.
- **Trust validation** depends on trusted roots, chain building, and policy checks.
- **Hostname verification** is separate from chain validation and must be explicitly enabled in client workflows.

**Quick reference examples**

```bash
# Generate an encrypted RSA private key
openssl genpkey -algorithm RSA -aes256 -out key.pem -pkeyopt rsa_keygen_bits:2048

# Create a CSR
openssl req -new -key key.pem -out request.csr

# Inspect a certificate
openssl x509 -in cert.pem -noout -text

# Verify a chain
openssl verify -CAfile root.pem -untrusted intermediate.pem cert.pem

# Convert PEM to DER
openssl x509 -in cert.pem -outform der -out cert.der

# Create a PKCS#12 bundle
openssl pkcs12 -export -inkey key.pem -in cert.pem -certfile chain.pem -out bundle.p12

# Test a TLS server
openssl s_client -connect host:443 -servername host -showcerts
```

**What to remember for CISSP**

OpenSSL is a toolset, not a trust decision engine. You use it to generate, inspect, convert, sign, encrypt, and test cryptographic objects, but you still need to understand the surrounding security controls: trust stores, certificate policy, hostname validation, key protection, and operational handling of private material.

The most common exam traps are assuming that a certificate automatically proves trust, assuming that a chain check includes hostname validation, or assuming that a file encryption example is secure without modern KDF settings such as `-pbkdf2`.
