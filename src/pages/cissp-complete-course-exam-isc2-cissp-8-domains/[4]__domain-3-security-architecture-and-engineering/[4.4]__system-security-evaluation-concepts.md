## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**System security evaluation overview**

System security evaluation is the disciplined process of determining whether a system’s security controls, architecture, and implementation meet a stated security requirement. In CISSP terms, you are usually judging both **effectiveness** and **assurance**: does the control work, and how confident are you that it works consistently?

Evaluation is not a single activity. It spans design review, implementation testing, operational assessment, and ongoing monitoring. In practice, you use different methods depending on the system type, acquisition model, and risk tolerance.

- **Security objectives** define what the system must protect.
- **Assurance** reflects confidence in the correctness and completeness of the protection.
- **Evaluation evidence** comes from tests, reviews, scans, audits, and formal certification artifacts.
- **Acceptance** means a business or authority decision that the residual risk is acceptable.

**Core evaluation concepts**

A useful way to think about evaluation is to separate the security property being checked from the method used to check it.

- **Verification** asks whether the system was built correctly against its specification.
- **Validation** asks whether the system satisfies the actual security need in the real environment.
- **Static testing** examines code, configuration, or design without executing the system.
- **Dynamic testing** evaluates behavior while the system is running.

This distinction matters on the exam. A secure design can still fail validation if it does not address the real threat model, and a system can pass a functional test while still containing hidden implementation flaws.

**Verification vs validation**

Verification is usually narrower and more technical. You compare the implementation to a requirement, standard, or design baseline. Validation is broader and more operational. You ask whether the control is suitable for the intended use and threat environment.

- **Verification examples**
  - Reviewing firewall rules against the approved rule set
  - Checking that encryption is enabled with the required algorithm
  - Confirming that a secure build matches the hardened baseline
- **Validation examples**
  - Testing whether the firewall actually blocks the attack paths used by real adversaries
  - Confirming that the encryption protects data in the deployed workflow
  - Determining whether the hardened build still supports business operations

A common CISSP scenario is that a system “meets the spec” but still fails in production because the spec omitted a threat. That is a validation failure, not just a verification issue.

**Static and dynamic security testing**

Static testing is valuable early because it is cheaper and safer. Dynamic testing is valuable because it reveals runtime behavior that static analysis cannot prove.

- **Static testing**
  - Source code review
  - Binary analysis
  - Configuration review
  - Infrastructure-as-code review
  - Policy-as-code checks
- **Dynamic testing**
  - Penetration testing
  - Runtime vulnerability scanning
  - Fuzzing
  - Authentication and authorization testing
  - Adversarial simulation in a test environment

Static methods are strong for finding logic errors, insecure defaults, and missing controls. Dynamic methods are strong for finding exploitable behavior, integration failures, and environmental weaknesses.

**Trusted computing base and reference monitor**

The **trusted computing base** (TCB) is the set of hardware, firmware, software, and configuration components that must be trusted to enforce the system’s security policy. If the TCB fails, the security model fails.

Within the TCB, the **security kernel** is the core component that implements the most critical security functions. It is typically small, isolated, and designed to be easier to verify than the rest of the system.

The **reference monitor** is the abstract concept of a mechanism that mediates all access to objects and subjects according to the security policy. For a reference monitor to be effective, it should be:

- **Always invoked** for every access attempt
- **Tamperproof** so attackers cannot bypass or alter it
- **Small enough to analyze** and test thoroughly

In exam questions, if you see a description of a mechanism that must mediate all access, think reference monitor. If the question asks about the trusted core that must be protected and minimized, think TCB and security kernel.

**Assurance and evaluation rigor**

Assurance is not the same as security strength. A system can be very secure in practice but have low assurance if it has not been thoroughly evaluated. Conversely, a highly assured system may still be a poor fit for the business if the wrong controls were selected.

Higher assurance generally means:

- More formal methods
- More independent review
- More exhaustive testing
- More documentation
- Higher cost and longer timelines

Lower assurance generally means:

- Faster evaluation
- Less evidence
- More reliance on vendor claims or limited testing
- Lower cost, but less confidence

This tradeoff is central to CISSP. You choose the level of rigor based on risk, mission criticality, and regulatory expectations.

**Common assurance frameworks and ratings**

The most recognized formal evaluation model is the **Common Criteria** framework, which uses **Evaluation Assurance Levels** (EALs). EALs describe increasing rigor of evaluation, not necessarily increasing functional security features.

- **EAL1**: Functionally tested; basic confidence
- **EAL2**: Structurally tested; more design and test evidence
- **EAL3**: Methodically tested and checked; stronger development discipline
- **EAL4**: Methodically designed, tested, and reviewed; often a practical commercial target
- **EAL5**: Semiformally designed and tested; higher rigor and cost
- **EAL6**: Semiformally verified design and tested; strong assurance for high-risk environments
- **EAL7**: Formally verified design and tested; very high rigor, very expensive, and uncommon

Higher EALs usually imply more documentation, more independent analysis, and more formal evidence. They do not automatically mean the product is “more secure” for every use case; they mean the evaluation was more rigorous.

> A common exam trap is assuming a higher EAL always means a better product. In reality, it means a more rigorous evaluation against a defined target, which may or may not match your environment.

**Formal certification and validation processes**

Formal evaluation usually produces documented evidence that a product or system meets a defined security target. In practice, this evidence is reviewed by an independent assessor, lab, or authority, then accepted by a sponsoring organization or customer.

Typical formal process flow:

1. Define the security target, scope, and evaluation criteria.
2. Identify the product, system boundary, and assumptions.
3. Collect design, implementation, and test evidence.
4. Perform independent assessment and testing.
5. Record findings, deviations, and residual risks.
6. Issue an evaluation report, certification, or accreditation decision.
7. Accept or reject the system based on business and risk criteria.

The important distinction is that **certification** is the technical evaluation of the system, while **accreditation** is the management decision to accept the risk and authorize operation. Some organizations use different terminology, but the CISSP exam often expects this conceptual split.

Evaluation results are typically documented in:

- Security target or system security plan
- Test plans and test results
- Vulnerability assessment reports
- Independent evaluation reports
- Risk acceptance memos
- Authorization or accreditation letters
- Remediation tracking records

In practice, acceptance is not just a signature. It usually requires that findings are categorized, compensating controls are identified if needed, and residual risk is explicitly approved by the appropriate authority.

**COTS vs custom-developed systems**

Evaluation differs depending on whether the system is commercial off-the-shelf (**COTS**) or custom-developed.

- **COTS systems**
  - You often rely on vendor documentation, third-party certifications, and independent testing
  - You focus on configuration, integration, patching, and known limitations
  - You may have limited visibility into source code or internal design
- **Custom-developed systems**
  - You can evaluate source code, architecture, and development practices directly
  - You can tailor tests to the exact threat model
  - You usually need stronger internal governance over secure development and change control

For COTS products, the main question is often whether the product can be securely configured and operated in your environment. For custom systems, the question is whether the design and implementation are secure by construction and remain secure after changes.

**Baseline configuration assessment and benchmarks**

A baseline is the approved minimum configuration for a system. Baselines are essential because you cannot evaluate security without a reference point.

Baseline assessment often includes:

- Operating system hardening settings
- Account and privilege settings
- Logging and audit settings
- Network service exposure
- Patch level and version state
- Cryptographic settings
- Endpoint protection and host firewall settings

Benchmarks are standardized hardening guides or configuration references used to compare a system against a known-good state. Common examples include vendor hardening guides and industry benchmark standards.

Practical uses of baselines and benchmarks:

- Detect configuration drift
- Support secure build images
- Standardize server and endpoint hardening
- Provide evidence during audits and assessments
- Reduce ambiguity during incident response

Example baseline checks on Linux:

```bash
# Check listening services
ss -tulpn

# Review installed packages and patch state
apt list --installed 2>/dev/null | head

# Inspect SSH hardening
grep -E '^(PermitRootLogin|PasswordAuthentication|PubkeyAuthentication)' /etc/ssh/sshd_config

# Review firewall rules
nft list ruleset
```

Example baseline checks on Windows:

```powershell
# Review local security policy-related settings
secedit /export /cfg C:\Temp\secpol.cfg

# Check running services
Get-Service | Where-Object {$_.Status -eq 'Running'}

# Review Defender status
Get-MpComputerStatus
```

**Security testing methods in evaluation**

Different evaluation methods answer different questions, and CISSP expects you to choose the right one for the scenario.

- **Code review**
  - Finds insecure logic, injection risks, and weak error handling
  - Best when source is available
- **Configuration review**
  - Finds weak defaults, exposed services, and policy gaps
  - Best for hardening and compliance checks
- **Vulnerability scanning**
  - Identifies known weaknesses at scale
  - Best for broad coverage, but it can miss context and exploitability
- **Penetration testing**
  - Demonstrates exploitability and attack paths
  - Best for validating real-world risk
- **Fuzz testing**
  - Sends malformed inputs to expose crashes and parser flaws
  - Best for protocol, file format, and input-handling components
- **Regression testing**
  - Confirms that fixes did not break existing security behavior
  - Best after patches or configuration changes

A useful rule: use static methods to find what is present in the design or code, and dynamic methods to prove how the system behaves under attack or stress.

**Continuous monitoring and reassessment**

Evaluation is not a one-time event. Security controls degrade as software changes, threats evolve, and operational conditions shift. Continuous monitoring keeps the evaluation current.

Continuous monitoring typically includes:

- Ongoing vulnerability scanning
- Log review and security event monitoring
- Configuration drift detection
- Patch and asset inventory tracking
- Periodic control testing
- Risk reassessment after major changes

Frameworks such as continuous monitoring programs and control assessment cycles help organizations move from point-in-time compliance to ongoing assurance. The key idea is that you reassess after significant events such as:

- Major software upgrades
- Infrastructure migrations
- New integrations
- Security incidents
- Changes in threat intelligence
- Material business process changes

In practice, continuous monitoring produces evidence that controls remain effective over time, not just on the day of the audit.

**Documentation and acceptance in practice**

Evaluation is only useful if the results are documented clearly enough for decision-makers to act on them. Good documentation supports traceability from requirement to test to decision.

A solid evaluation record usually includes:

- Scope and system boundary
- Security requirements and assumptions
- Test methods and tools used
- Findings and severity ratings
- Evidence artifacts such as screenshots, logs, and reports
- Remediation actions and retest results
- Residual risk statement
- Formal acceptance or rejection decision

Acceptance usually depends on whether the remaining risk is within tolerance. If not, the system may be denied authorization, placed in a limited operating mode, or approved only with compensating controls.

**Scenario-based CISSP examples**

Use the scenario details to decide which evaluation concept applies.

- A vendor claims a product is secure, but you need evidence that it blocks unauthorized access in your environment.
  - Choose **validation** and dynamic testing.
- You need to confirm that a hardened server image matches the approved build standard.
  - Choose **verification** and configuration assessment.
- You want to find insecure coding patterns before deployment.
  - Choose **static analysis** or code review.
- You need to prove that a web application resists real attack attempts.
  - Choose **penetration testing** or dynamic testing.
- You are evaluating a safety-critical or high-assurance component where the trusted core must be minimized.
  - Focus on the **TCB**, security kernel, and reference monitor properties.
- You are deciding whether to buy a commercial product or build a custom one.
  - Compare **COTS** evaluation evidence against the visibility and control available in custom development.

**Practical evaluation workflow**

1. Define the security requirement and system boundary.
2. Identify the trust assumptions and the TCB.
3. Select the evaluation method based on risk and evidence availability.
4. Compare the system against baselines, benchmarks, and policy.
5. Test both implementation correctness and real-world behavior.
6. Document findings, residual risk, and compensating controls.
7. Obtain formal acceptance or remediation approval.
8. Reassess after changes and on a continuous monitoring schedule.

**Exam focus points**

- **Verification** checks conformance to specification; **validation** checks fitness for use.
- **Static testing** inspects artifacts without execution; **dynamic testing** observes runtime behavior.
- The **TCB** is the trusted set of components; the **security kernel** is its core enforcement mechanism.
- The **reference monitor** must mediate all access, be tamperproof, and be small enough to analyze.
- **Common Criteria EALs** indicate increasing evaluation rigor and cost, not automatic superiority.
- **COTS** evaluation emphasizes vendor evidence and secure configuration; custom systems allow deeper internal review.
- **Baselines** and **benchmarks** provide the reference state for configuration assessment.
- **Continuous monitoring** extends evaluation beyond a one-time audit and triggers reassessment after major changes.

**Quick comparison table**

| Concept | Primary question | Typical method |
|---|---|---|
| Verification | Was it built to spec? | Review, test, inspection |
| Validation | Does it meet the real need? | Operational testing, scenario testing |
| Static testing | What is wrong in the artifact? | Code review, config review, SAST |
| Dynamic testing | How does it behave at runtime? | Pen test, DAST, fuzzing |
| Certification | What does the technical evaluation show? | Independent assessment |
| Accreditation/authorization | Should the business accept the risk? | Management decision |

**Memory aid**

Think in layers:

- **Design**: what should happen
- **Implementation**: what is built
- **Operation**: what actually happens
- **Governance**: who accepts the risk

If you keep those layers separate, CISSP evaluation questions become much easier to answer.
