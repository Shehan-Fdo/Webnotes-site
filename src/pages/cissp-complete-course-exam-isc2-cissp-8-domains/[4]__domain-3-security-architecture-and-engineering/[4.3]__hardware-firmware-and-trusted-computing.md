**Trusted computing overview**

Trusted computing focuses on establishing a trustworthy execution environment from power-on through runtime. The goal is not to make a system “perfectly secure,” but to reduce the chance that malicious code, altered firmware, or unauthorized hardware changes can subvert the platform before the operating system (OS) even starts.

At a high level, you care about four layers:

- **Hardware trust anchors** such as the motherboard, CPU, chipset, and trusted platform modules.
- **Firmware integrity** across BIOS/UEFI, option ROMs, and device firmware.
- **Boot-chain validation** so the platform only starts approved code.
- **Runtime assurance** through attestation, isolation, and hardware-backed key protection.

**Hardware trust foundations**

A secure platform starts with a trustworthy hardware base. In CISSP scenarios, the important idea is that if the root of trust is compromised, everything above it becomes suspect.

- **Root of trust** is the minimal set of components assumed to be trustworthy at startup.
- **Hardware root of trust** means the trust anchor is implemented in silicon or protected hardware rather than software alone.
- **Chain of trust** means each stage verifies the next stage before handing off execution.
- **Measured trust** means the system records cryptographic measurements of components for later verification, even if it does not block execution.

Common hardware-related controls include:

- **Trusted Platform Module (TPM)** for secure key storage and platform measurements.
- **Hardware Security Module (HSM)** for high-assurance key generation, storage, and cryptographic operations.
- **IOMMU** (Input-Output Memory Management Unit) to restrict DMA (Direct Memory Access) from peripherals.
- **Secure enclaves / TEEs** (Trusted Execution Environments) for isolated code and data execution.

**BIOS vs UEFI**

The firmware interface matters because it controls the earliest boot behavior.

- **BIOS** (Basic Input/Output System) is the legacy firmware model.
- **UEFI** (Unified Extensible Firmware Interface) is the modern replacement with richer security and boot management features.

Key differences:

| Area | BIOS | UEFI |
|---|---|---|
| Boot model | Legacy, MBR-oriented | Modern, GPT-friendly |
| Security features | Limited native security | Secure Boot, measured boot, firmware signing support |
| Extensibility | Minimal | Modular drivers, richer pre-boot environment |
| Disk support | Commonly tied to MBR limits | Supports GPT and large disks more naturally |
| Recovery/management | Basic | Better support for firmware capsules and rollback controls |

Legacy BIOS security limitations are important for exam scenarios:

- It has weaker native support for cryptographic boot validation.
- It is more dependent on trust in the firmware vendor and physical access controls.
- It often lacks modern protections such as signed firmware update workflows and robust measured boot integration.
- It is easier to pair with legacy boot paths that bypass modern security controls.

In practice, you often disable legacy compatibility support module (CSM) when you want a fully UEFI-based secure boot posture.

> If a system must support legacy BIOS boot, treat it as a higher-risk exception and compensate with stronger physical security, tighter firmware control, and stronger endpoint monitoring.

**Secure Boot, measured boot, and trusted boot**

These terms are related but not identical.

- **Secure Boot** prevents unauthorized bootloaders and OS loaders from executing by verifying digital signatures before execution.
- **Measured boot** records hashes of boot components into TPM Platform Configuration Registers (PCRs) so the system state can be attested later.
- **Trusted boot** is a broader concept describing a boot process designed to establish trust in the platform, often combining verification and measurement.
- **Measured launch** is a more specific term used when the launch of a component is measured into hardware registers for later trust decisions.

A useful way to remember the distinction:

- **Secure Boot** answers: “Should this component be allowed to run?”
- **Measured boot** answers: “What exactly ran?”
- **Trusted boot** answers: “Can I establish confidence in the boot path?”
- **Measured launch** answers: “Can I record the launch state in hardware for attestation?”

These are often combined:

- Secure Boot blocks known-bad or unsigned boot components.
- Measured boot records the boot chain for remote verification.
- Trusted boot uses both to reduce boot-time compromise risk.

**TPM 1.2 vs TPM 2.0**

The TPM is central to modern trusted computing. It provides hardware-backed key protection, measurements, and attestation support.

**TPM 1.2** is older and more limited:

- Uses a narrower algorithm set.
- Has less flexibility for modern cryptographic agility.
- Is less aligned with current enterprise and compliance expectations.
- Is often considered legacy in modern CISSP contexts.

**TPM 2.0** is preferred because it offers:

- **Cryptographic agility** with broader algorithm support.
- Better alignment with modern security requirements and platform attestation workflows.
- More flexible authorization and policy structures.
- Stronger fit for current OS features such as BitLocker, measured boot, and device health attestation.

Why TPM 2.0 matters in exam scenarios:

- It is the expected baseline for modern Windows and enterprise endpoint security designs.
- It supports stronger and more flexible trust models than TPM 1.2.
- It is better suited to current compliance and lifecycle expectations.

Typical TPM uses include:

- Storing disk encryption keys indirectly, such as protecting BitLocker volume master keys.
- Sealing secrets to platform state so they only release when the boot measurements match expected values.
- Supporting remote attestation by proving platform integrity to a verifier.

**TPM concepts you should know**

- **Endorsement key (EK)** is a unique, hardware-bound key pair used to identify the TPM.
- **Storage root key (SRK)** or equivalent hierarchy keys protect other keys and objects.
- **Platform configuration registers (PCRs)** store measurements of boot components.
- **Sealing** binds a secret to specific platform state.
- **Unsealing** releases the secret only when the expected measurements are present.

Practical example with Linux TPM tooling:

```bash
tpm2_pcrread sha256:0,2,7
tpm2_getrandom 16
```

Example of checking TPM presence on Windows:

```powershell
Get-Tpm
```

**Remote attestation**

Remote attestation lets a verifier assess whether a remote system booted and is running in a trusted state.

The trust model usually includes:

- **Attester**: the device being evaluated.
- **Verifier**: the party checking the evidence.
- **Evidence**: measurements, quotes, logs, or signed claims.
- **Trust anchor**: the verifier’s trusted reference values and certificate chain.

Important attestation components:

- **Attestation Identity Key (AIK)** is a TPM-backed key used to sign attestation data without exposing the TPM’s unique identity key directly.
- **Endorsement certificate** helps establish that the TPM is genuine and tied to a trusted manufacturer or provisioning chain.
- **Verifier trust assumptions** include trusting the reference measurements, the certificate chain, the attestation policy, and the freshness of the evidence.

Attestation is only as strong as its assumptions:

- If the verifier trusts bad reference hashes, the attestation result is meaningless.
- If the AIK provisioning process is weak, identity binding can be spoofed.
- If the evidence is stale, the device may have changed after measurement.

A common flow is:

1. The device measures boot components into TPM PCRs.
2. The device signs a quote with an attestation key.
3. The verifier checks the signature, certificate chain, and PCR values.
4. The verifier compares the measurements to a known-good baseline.
5. The verifier decides whether to grant access or quarantine the device.

**Intel TXT, AMD hardware-rooted trust, and trusted boot**

Hardware-rooted trust technologies help establish a more reliable launch environment.

- **Intel TXT** (Trusted Execution Technology) is designed to help create a measured and protected launch environment.
- **AMD hardware-rooted trust** is commonly associated with platform features that support secure launch and measured boot concepts on AMD systems.
- These technologies are relevant when you need stronger assurance that early boot code has not been tampered with.

In exam terms, they relate to:

- **Trusted boot** by helping establish a trustworthy launch state.
- **Measured launch** by recording launch measurements in hardware.
- **Remote attestation** by providing evidence that the platform started in an expected state.

You do not need to memorize vendor-specific implementation details as much as the security purpose:

- Establish a hardware-backed launch trust anchor.
- Reduce reliance on mutable software-only boot code.
- Support later verification of platform integrity.

**Firmware security controls**

Firmware is a high-value target because it executes before the OS and can persist across reboots and disk wipes.

Important firmware protections include:

- **Write protection** to prevent unauthorized modification of firmware flash.
- **Signed firmware updates** to ensure only trusted firmware images are installed.
- **Rollback prevention** to block downgrades to vulnerable firmware versions.
- **Recovery or dual-bank firmware** so a failed update can fall back to a known-good image.
- **Secure/verified boot firmware** that validates firmware components before execution.

Exam-oriented implications:

- If firmware can be rewritten without authorization, an attacker can implant persistent malware.
- If rollback protection is absent, an attacker may downgrade to a vulnerable version even after patching.
- If recovery mechanisms are weak, a failed update can brick the device or force insecure recovery paths.

Practical enterprise controls often include:

- BIOS/UEFI admin password.
- Firmware update approval workflow.
- Physical access restrictions to prevent flash tampering.
- Centralized firmware inventory and patch management.

Example UEFI hardening checklist in a deployment script or runbook:

```text
- Disable CSM/legacy boot
- Enable Secure Boot
- Enable TPM 2.0
- Set BIOS/UEFI admin password
- Enable firmware update signing enforcement
- Enable rollback protection if supported
- Disable unused boot devices
```

**Supply-chain assurance and hardware provenance**

Trusted computing is not only about runtime integrity. It also depends on how hardware is sourced, received, and validated.

Key supply-chain controls include:

- **Hardware provenance** to verify where components came from and whether they were sourced through approved channels.
- **Tamper-evident seals** to show whether packaging or chassis access occurred before receipt.
- **Secure receiving** processes to inspect shipments, verify serial numbers, and confirm asset tags.
- **Asset validation** to compare delivered hardware against purchase orders and approved configurations.

Good receiving workflow:

1. Inspect packaging for tamper evidence.
2. Verify serial numbers, model numbers, and asset tags.
3. Confirm firmware baseline and hardware revision.
4. Record chain-of-custody information.
5. Quarantine anomalies before deployment.

These controls matter because a compromised device can arrive with:

- Modified firmware.
- Unauthorized hardware implants.
- Substituted components.
- Preloaded malicious management interfaces.

**HSMs and key protection models**

An HSM is a dedicated device for generating, storing, and using cryptographic keys in a protected environment. It is used when software key storage is not sufficient.

Common deployment models:

- **On-prem appliance**: a physical HSM installed in a datacenter for local control and low-latency access.
- **Network HSM**: a shared HSM accessed over a secure network by multiple applications or servers.
- **Cloud HSM**: a provider-managed HSM service integrated with cloud workloads.
- **Cluster or high-availability HSM**: multiple HSMs configured for redundancy and failover.

Why organizations use HSMs:

- Protect root CA (certificate authority) keys.
- Secure code-signing keys.
- Support payment and regulated workloads.
- Enforce separation of duties and dual control.
- Reduce key exposure in application memory and disk.

Operational considerations:

- **Latency** can affect transaction-heavy workloads.
- **Throughput** matters for signing-intensive services.
- **Redundancy** is critical if the HSM is a single point of failure.
- **Backup and key escrow** must be designed carefully to avoid weakening security.
- **Partitioning** may be used to isolate tenants or applications.

Example use cases:

- A code-signing service uses an HSM so private signing keys never leave hardware.
- A PKI (Public Key Infrastructure) root CA is kept offline in an HSM-backed environment.
- A cloud application uses a cloud HSM for envelope encryption key protection.

**IOMMU, VT-d, and DMA protection**

Peripheral devices can attack memory directly through DMA. An IOMMU helps prevent that by translating and restricting device memory access.

- **Intel VT-d** is Intel’s virtualization technology for directed I/O.
- **AMD-Vi** is AMD’s equivalent I/O virtualization and DMA protection capability.
- These features are especially important for defending against malicious or compromised peripherals.

Why this matters:

- A rogue Thunderbolt device or PCIe device can attempt DMA attacks.
- Without IOMMU protection, a peripheral may read or overwrite sensitive memory.
- With IOMMU enabled, device access can be constrained to approved memory regions.

Example Linux verification:

```bash
dmesg | grep -i -E 'iommu|dmar|amd-vi'
```

Example UEFI/BIOS scenario:

- Enable `VT-d` or `IOMMU`.
- Disable unused external expansion ports if possible.
- Require device authorization for hot-plug peripherals.

**Thunderbolt and external device security**

Thunderbolt and similar high-speed external interfaces can expose DMA risk if not controlled.

Common security levels and controls include:

- **User authorization** for new devices.
- **No PCIe tunneling** unless required.
- **Pre-boot access restrictions** to prevent unauthorized devices before OS policy loads.
- **Kernel DMA protection** where supported.

Practical hardening examples:

- Set Thunderbolt security to a mode that requires user approval.
- Disable Thunderbolt pre-boot support on systems that do not need it.
- Pair Thunderbolt with IOMMU/VT-d so DMA is constrained.

Example policy-style checklist:

```text
Thunderbolt:
- Security level: User Authorization
- Pre-boot support: Disabled unless required
- DMA protection: Enabled
- Approved devices only: Yes
```

**Trusted execution environments and secure enclaves**

A TEE or secure enclave isolates sensitive code and data from the rest of the system, including the OS in some designs.

Examples include:

- CPU-backed enclave technologies.
- Secure world / normal world separation on some platforms.
- Application enclaves for secrets processing.

Benefits:

- Protects keys and sensitive computations from ordinary OS compromise.
- Reduces exposure of secrets in memory.
- Supports confidential computing use cases.

Limitations and exam-relevant risks:

- **Side-channel attacks** may leak information through timing, cache behavior, power use, or speculative execution effects.
- **Speculative-execution vulnerabilities** can undermine isolation guarantees if the platform or software is not patched.
- Enclaves do not eliminate the need for secure input, secure output, and strong attestation.
- A compromised host may still influence enclave behavior through resource starvation or malicious orchestration.

In CISSP terms, treat TEEs as a defense-in-depth control, not a complete trust solution.

**Boot-chain and firmware configuration examples**

A secure enterprise endpoint often combines multiple controls.

Example UEFI hardening goals:

```text
- UEFI mode only
- CSM disabled
- Secure Boot enabled
- TPM 2.0 enabled
- BIOS admin password set
- Boot order locked to internal disk
- External boot disabled or restricted
- Firmware update controls enabled
```

Example Windows verification commands:

```powershell
Confirm-SecureBootUEFI
Get-Tpm
```

Example Linux boot and firmware inspection:

```bash
mokutil --sb-state
efibootmgr -v
```

Example configuration logic for a deployment baseline:

1. Disable legacy boot paths.
2. Enable Secure Boot and TPM 2.0.
3. Enforce firmware passwords and update signing.
4. Restrict external boot and hot-plug DMA-capable devices.
5. Validate the device with attestation before granting access.

**Common exam scenarios**

- A laptop is stolen, but disk encryption keys remain protected by TPM-sealed state, reducing offline compromise.
- A server fails a remote attestation check because PCR values do not match the approved boot baseline.
- A firmware downgrade attack succeeds because rollback protection is disabled.
- A malicious Thunderbolt device attempts DMA access, but IOMMU and Thunderbolt authorization block it.
- A code-signing key is protected in an HSM so the private key never appears in application memory.
- A legacy BIOS system cannot support modern secure boot controls, so it requires compensating physical and monitoring controls.

**How to choose the right control**

Use the control that matches the threat:

- Use **Secure Boot** to block unauthorized bootloaders.
- Use **measured boot** and **attestation** to prove what actually ran.
- Use **TPM 2.0** for modern platform key protection and measurement workflows.
- Use **HSMs** for high-value cryptographic keys.
- Use **IOMMU/VT-d** to defend against DMA-capable peripherals.
- Use **firmware rollback prevention** to stop downgrade attacks.
- Use **supply-chain validation** to reduce tampering before deployment.
- Use **TEEs** when you need isolated execution, but remember their side-channel limits.

**Quick comparison**

| Control | Primary purpose | Key limitation |
|---|---|---|
| Secure Boot | Blocks unauthorized boot code | Does not prove runtime integrity by itself |
| Measured boot | Records boot measurements | Needs a verifier to be useful |
| TPM 2.0 | Hardware-backed keys and measurements | Not a full security solution alone |
| HSM | Protects high-value cryptographic keys | Cost, complexity, and operational overhead |
| IOMMU/VT-d | Restricts DMA access | Must be enabled and correctly configured |
| TEE / enclave | Isolates sensitive code | Still exposed to side-channel risks |

**Exam takeaways**

- **BIOS** is legacy and weaker; **UEFI** supports modern boot security.
- **Secure Boot** prevents unauthorized execution; **measured boot** records evidence for later trust decisions.
- **TPM 2.0** is preferred over TPM 1.2 because it better fits modern cryptographic and attestation needs.
- **Remote attestation** depends on AIKs, endorsement certificates, reference measurements, and verifier trust assumptions.
- **Intel TXT** and similar hardware-rooted trust features support trusted and measured launch concepts.
- **HSMs** protect critical keys and are deployed as appliances, network devices, cloud services, or clustered systems.
- **Firmware security** must include write protection, signed updates, rollback prevention, and recovery paths.
- **Supply-chain assurance** starts before boot with provenance checks, tamper evidence, and secure receiving.
- **IOMMU/VT-d** and Thunderbolt controls reduce DMA risk from external devices.
- **TEEs** improve isolation but do not eliminate side-channel or speculative-execution threats.
