## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Enterprise secure design overview**

Secure design for enterprise systems focuses on building systems that remain trustworthy under attack, failure, and misuse. For CISSP, you need to connect design principles to practical controls: how data is classified, how trust boundaries are enforced, how identities are verified, and how resilience is achieved without creating new weaknesses.

A strong design starts with business requirements and risk tolerance, then maps those needs to architecture, control selection, and operational constraints. You should think in layers: people, process, technology, and governance all influence whether a design is actually secure.

- **Defense in depth** uses multiple independent controls so one failure does not expose the system.
- **Layering** separates functions into tiers, such as presentation, application, and data layers.
- **Abstraction** hides implementation details behind stable interfaces, reducing complexity and attack surface.
- **Data hiding** restricts access to sensitive internals and exposes only what is necessary.
- **Modularity** breaks systems into smaller components that are easier to secure, test, and replace.
- **Simplicity** reduces error-prone complexity and makes security controls more reliable.

**Core secure design principles**

These principles are often tested as design concepts rather than product features. You should be able to explain how each one reduces risk and where it applies.

- **Least privilege** gives users, services, and processes only the permissions required for their task.
- **Need to know** limits access to information even when a user has general authorization.
- **Separation of duties** prevents one person or process from completing a critical action alone.
- **Fail secure** means the system defaults to a safe state when something goes wrong.
- **Complete mediation** checks authorization every time a subject accesses an object, not just once.
- **Open design** assumes the design is known; security depends on secret keys and strong controls, not obscurity.
- **Economy of mechanism** favors small, simple mechanisms that are easier to verify.
- **Psychological acceptability** means security controls should be usable enough that people do not bypass them.
- **Least common mechanism** reduces shared components that could leak data or create coupling.
- **Work factor** makes attacks expensive enough that they are not practical.

A useful exam distinction is that secure design principles are not the same as control categories. For example, least privilege is a design principle, while role-based access control is one way to implement it.

**Trusted computing base, reference monitor, and security kernel**

The **trusted computing base** (TCB) is the set of hardware, firmware, software, and controls that must be trusted to enforce the security policy. If the TCB is compromised, the system can no longer be assumed secure.

The **reference monitor** is the conceptual mechanism that mediates every access to every object. It should be:
- Always invoked
- Tamper resistant
- Small enough to be verifiable

The **security kernel** is the hardware/software component that implements the reference monitor concept. In practice, the kernel or hypervisor may enforce access decisions, but the CISSP idea is broader: the enforcement point must be reliable, isolated, and complete.

Design implications:
- Keep the TCB as small as possible.
- Minimize privileged code paths.
- Isolate security-critical functions from general application logic.
- Use strong logging and monitoring around the enforcement layer.
- Prefer well-reviewed platform security features over custom enforcement code.

> If the enforcement mechanism can be bypassed, it is not a true reference monitor.

**Security architecture patterns**

Enterprise systems often use architectural patterns to reduce exposure and contain compromise.

- **Zero trust** assumes no implicit trust based on network location.
  - Every request is authenticated and authorized.
  - Device posture, user identity, and context influence access.
  - Microsegmentation limits lateral movement.
- **DMZ** (demilitarized zone) separates public-facing services from internal systems.
  - A classic three-tier DMZ places web servers in the DMZ, application servers behind it, and databases deeper inside.
  - A screened subnet design uses filtering devices to control traffic between zones.
  - A dual-firewall design can place the DMZ between two firewalls for stronger separation.
- **Microservices** split applications into small services with independent deployment.
  - Each service needs its own authentication and authorization checks.
  - Service-to-service trust should use short-lived credentials and mutual TLS where appropriate.
  - Shared databases and overly broad service accounts weaken isolation.
- **Tiered architecture** separates presentation, business logic, and data access.
  - This supports containment and clearer trust boundaries.
  - It also makes it easier to apply different controls to each tier.

Practical design guidance:
- Place internet-facing components in the least trusted zone.
- Avoid direct database exposure from public tiers.
- Treat internal APIs as hostile until authenticated and validated.
- Use network segmentation, host hardening, and identity controls together.

**Data classification and design impact**

Data classification drives nearly every security decision. If you do not know the sensitivity of the data, you cannot choose appropriate encryption, access control, retention, or segregation.

Common classification levels include:
- Public
- Internal
- Confidential
- Restricted or highly sensitive

Design decisions influenced by classification:
- **Encryption**
  - Highly sensitive data may require encryption at rest and in transit.
  - Some data may need field-level encryption rather than full-disk encryption.
- **Access control**
  - Confidential data may require role-based access control, attribute-based access control, or approval workflows.
  - Restricted data may require stronger authentication and tighter logging.
- **Retention**
  - Sensitive data should be retained only as long as necessary for legal or business needs.
  - Shorter retention reduces breach impact and compliance burden.
- **Segregation**
  - Different classes of data may need separate databases, storage buckets, or tenants.
  - Segregation reduces blast radius and simplifies policy enforcement.
- **Monitoring**
  - Higher-classified data usually requires stronger audit trails and alerting.
- **Backup and recovery**
  - Backup copies must inherit the same or stronger protections as the source data.

A practical rule is to design for the highest classification that the system will process, not just the average case.

**Privacy by design and data minimization**

**Privacy by design** means privacy requirements are built into the system from the start, not added later. **Data minimization** means you collect, process, and retain only the data you truly need.

Key practices:
- Collect only required fields.
- Separate identity data from behavioral or transactional data when possible.
- Use pseudonymization or tokenization when full identity is not needed.
- Limit secondary use of data to approved purposes.
- Build deletion and retention workflows into the design.
- Provide transparency and consent mechanisms where required by policy or law.

Why it matters:
- Less data means less exposure in a breach.
- Smaller datasets are easier to classify, protect, and audit.
- Minimization supports compliance with privacy regulations and internal governance.

**Threat modeling and misuse cases**

Threat modeling helps you identify likely attacks before implementation. A common approach is STRIDE:
- **Spoofing** identity
- **Tampering** with data
- **Repudiation** of actions
- **Information disclosure**
- **Denial of service**
- **Elevation of privilege**

Example misuse case-to-control mapping:

| Misuse case | Likely risk | Design control |
|---|---|---|
| Attacker reuses stolen API token | Spoofing, unauthorized access | Short-lived tokens, token binding where supported, MFA for issuance, revocation |
| Client sends oversized JSON payload | Denial of service | Rate limiting, request size limits, WAF rules, queue backpressure |
| User modifies hidden price field | Tampering | Server-side validation, schema enforcement, immutable pricing rules |
| Insider denies approving a transaction | Repudiation | Signed audit logs, time-stamped records, dual approval |
| Service account reads another tenant’s records | Information disclosure | Tenant isolation, row-level security, per-tenant keys |
| Low-privilege user calls admin endpoint | Elevation of privilege | Strong authorization checks, separate admin plane, least privilege |

A simple STRIDE example for an API:
- **Spoofing**: stolen bearer token
- **Tampering**: altered request body
- **Repudiation**: missing audit trail
- **Information disclosure**: verbose error messages
- **Denial of service**: request flooding
- **Elevation of privilege**: broken object-level authorization

**Secure interface and API design**

Interfaces are trust boundaries. Every API, message queue, file import, and remote procedure call should be treated as an attack surface.

Core controls:
- **Authentication tokens**
  - Use short-lived access tokens and separate refresh tokens where appropriate.
  - Prefer scoped tokens with audience restrictions.
  - Revoke or rotate credentials when compromise is suspected.
- **Rate limiting**
  - Prevent brute force, scraping, and resource exhaustion.
  - Apply per-user, per-IP, and per-client limits as needed.
- **Input validation**
  - Validate type, length, format, range, and encoding.
  - Reject unexpected fields when possible.
- **Schema enforcement**
  - Use strict schemas for JSON, XML, and message payloads.
  - Fail closed on malformed or unknown structures.
- **Authorization**
  - Check object-level and function-level permissions on every request.
  - Do not trust client-side checks.
- **Error handling**
  - Return generic errors to clients.
  - Log detailed diagnostics internally.
- **Replay protection**
  - Use nonces, timestamps, or one-time tokens for sensitive operations.
- **Transport security**
  - Use TLS for API traffic and mutual TLS for service-to-service trust where appropriate.

Example API gateway policy:
```yaml
rateLimit:
  requestsPerMinute: 120
  burst: 20
auth:
  tokenType: oauth2_bearer
  requiredScopes:
    - orders:read
    - orders:write
validation:
  strictSchema: true
  maxBodyBytes: 1048576
logging:
  auditSensitiveFields: true
```

Example cURL request with a bearer token:
```bash
curl -sS https://api.example.com/v1/orders/123 \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Accept: application/json"
```

**PKI and certificate management**

Public key infrastructure (PKI) supports identity, encryption, and trust at scale. Enterprise design must include operational controls, not just cryptography.

Important PKI concepts:
- **Certificate authority** (CA) issues and signs certificates.
- **Trust anchor** is the root of trust, usually a root CA certificate.
- **Certificate revocation** removes trust in a certificate before expiration.
- **Certificate pinning** binds an application to a specific certificate or public key.

Revocation methods:
- **CRL** (certificate revocation list)
  - A signed list of revoked certificates.
  - Can be large and may be stale between updates.
- **OCSP** (Online Certificate Status Protocol)
  - Real-time status checking for a certificate.
  - Faster to update, but depends on responder availability.
- **OCSP stapling**
  - The server provides a recent OCSP response to clients.
  - Reduces client privacy leakage and improves performance.

Operational considerations:
- Protect root CA keys offline when possible.
- Use intermediate CAs for day-to-day issuance.
- Rotate and monitor certificates before expiration.
- Maintain trust store hygiene across endpoints and services.
- Remove obsolete trust anchors to avoid shadow trust.
- Use pinning carefully because it can break recovery if keys change unexpectedly.

> Certificate pinning improves resistance to rogue CAs, but it can create availability risk if you cannot rotate keys safely.

**Secure storage and data protection**

Storage design should match the data classification and threat model.

Encryption options:
- **Full-disk encryption**
  - Protects data at rest on the device.
  - Does not protect data while the system is running or against authorized users.
- **Database encryption**
  - Transparent data encryption protects files and tablespaces.
  - Useful for reducing exposure from stolen media or backups.
- **Field-level encryption**
  - Encrypts specific sensitive columns such as national IDs or payment data.
  - Limits exposure even if the database is partially compromised.
- **Tokenization**
  - Replaces sensitive values with non-sensitive tokens.
  - Useful for payment systems and systems that do not need the original value.
- **Backup encryption**
  - Backups must be encrypted separately and protected with strong key management.
  - Backup keys should be separated from production data keys when possible.

Key management design:
- Separate encryption keys by environment and data class.
- Use hardware security modules (HSMs) or cloud key management services for high-value keys.
- Rotate keys according to policy and risk.
- Restrict who can decrypt backups.
- Log key access and administrative actions.

Example PostgreSQL-style column encryption pattern:
```sql
-- Conceptual example: store only encrypted sensitive values
CREATE TABLE customer_identity (
  customer_id UUID PRIMARY KEY,
  national_id_ciphertext BYTEA NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```

Design tradeoffs:
- Field-level encryption improves confidentiality but can reduce searchability and indexing.
- Tokenization simplifies compliance for some workflows but requires a secure token vault.
- Backup encryption is essential, but key separation matters as much as the encryption itself.

**Identity, access control, and trust boundaries**

Enterprise systems should enforce identity at every layer where trust changes.

Design choices:
- Use centralized identity providers for workforce access.
- Use service identities for machine-to-machine communication.
- Prefer role-based access control (RBAC) for stable job functions.
- Use attribute-based access control (ABAC) when context matters, such as location, device posture, or data sensitivity.
- Apply just-in-time access for privileged operations.
- Separate administrative planes from user-facing planes.

Trust boundary examples:
- Browser to web application
- Web application to API
- API to database
- Internal service to external SaaS
- On-premises network to cloud tenant

Each boundary should have:
- Authentication
- Authorization
- Input validation
- Logging
- Error handling
- Monitoring

**Resilience, redundancy, and single points of failure**

Secure design must also survive outages and attacks. Availability is part of security.

Key resilience patterns:
- **Redundancy**
  - Duplicate critical components so one failure does not stop service.
  - Can be active-active or active-passive.
- **Load balancing**
  - Distributes traffic across multiple nodes.
  - Helps absorb spikes and isolate failing instances.
- **Clustering**
  - Multiple systems work together as one logical service.
  - Can improve availability and failover.
- **Failover**
  - Traffic moves to a standby system when the primary fails.
- **Geographic diversity**
  - Separate sites reduce the impact of regional outages.
- **Graceful degradation**
  - Nonessential features are disabled before core functions fail.

Single point of failure analysis:
- Identify components with no backup.
- Check power, network, identity, storage, DNS, and certificate dependencies.
- Review management planes, not just application nodes.
- Test failover regularly.

Example architecture checklist:
- Two load balancers instead of one
- Multiple application instances across availability zones
- Replicated databases with tested failover
- Separate backup storage and restore validation
- Redundant DNS and identity services

**Virtualization, containers, and platform isolation**

Virtualized and containerized environments need security controls at the platform layer, not just inside the workload.

Virtualization considerations:
- Harden the hypervisor and management plane.
- Limit administrative access to the host and console.
- Separate tenant workloads where required.
- Monitor for VM escape risks, even if they are rare.

Container considerations:
- Use trusted base images from controlled registries.
- Scan images for vulnerabilities and malware.
- Sign images and verify signatures before deployment.
- Run containers with least privilege and read-only filesystems where possible.
- Avoid privileged containers unless absolutely necessary.
- Use namespace boundaries to isolate processes, network, and mounts.
- Apply cgroup limits to reduce resource abuse.
- Use Kubernetes network policies to restrict pod-to-pod traffic.

Example Kubernetes security controls:
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-except-app
spec:
  podSelector:
    matchLabels:
      app: orders
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: api-gateway
  egress:
    - to:
        - namespaceSelector:
            matchLabels:
              name: payments
```

Additional container design points:
- Use separate service accounts per workload.
- Mount secrets only where needed.
- Avoid sharing host paths across containers.
- Treat the cluster control plane as highly sensitive.

**Hardware-rooted trust and platform integrity**

Modern enterprise design often depends on hardware-backed trust.

Key concepts:
- **Secure boot** verifies each boot stage before execution.
- **TPM** (Trusted Platform Module) stores cryptographic material and platform measurements.
- **Attestation** proves to a remote party that a device booted into a known-good state.
- **Measured boot** records boot measurements for later verification.

Why these matter:
- They reduce the risk of bootkits and firmware tampering.
- They support device trust decisions in zero trust architectures.
- They help validate endpoint posture before granting access.

Practical uses:
- Require attestation before allowing access to sensitive applications.
- Bind disk encryption keys to TPM-backed measurements.
- Use secure boot on servers, laptops, and cloud instances where supported.

**Logical, physical, and administrative controls**

CISSP often expects you to distinguish control types.

| Control type | What it is | Example |
|---|---|---|
| Logical | Technical enforcement in systems | MFA, ACLs, encryption, firewall rules |
| Physical | Protection of facilities and hardware | Locks, badges, cameras, cages |
| Administrative | Policies and procedures | Data classification policy, access reviews, training |

Design implication:
- Use all three control types together.
- Do not rely on administrative policy alone when technical enforcement is possible.
- Physical security still matters for servers, backup media, and network gear.

**Practical enterprise design example**

Consider a customer portal that stores personal and payment data.

A secure design might include:
- Public web tier in a DMZ
- Application tier in a private subnet
- Database tier isolated from direct internet access
- API gateway with OAuth 2.0 tokens, rate limiting, and schema validation
- Field-level encryption for payment identifiers
- Tokenization for card data
- Separate keys for production and backup encryption
- RBAC for support staff and ABAC for sensitive records
- Centralized logging with tamper-evident audit trails
- Active-active web tier with failover across zones
- Secure boot and TPM-backed attestation on managed endpoints
- Signed container images and network policies in the cluster

This design reduces exposure at each boundary and limits the blast radius if one component fails.

**Exam-focused distinctions**

- **Layering** is about organizing functions into tiers; **segmentation** is about separating trust zones.
- **Abstraction** hides complexity; **data hiding** restricts access to internal state.
- **Zero trust** removes implicit network trust; **DMZ** reduces exposure for public services.
- **CRL** is a revocation list; **OCSP** is a live status check.
- **Tokenization** replaces sensitive values; **encryption** protects values but still preserves the original data.
- **Redundancy** improves availability; it does not replace security controls.
- **Secure boot** protects startup integrity; **attestation** proves that integrity to others.
