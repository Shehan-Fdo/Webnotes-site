## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Secure hybrid cloud architecture overview**

A secure hybrid cloud combines on-premises infrastructure with one or more cloud service models while preserving consistent security, governance, and recovery controls across both environments. The core challenge is not just protecting workloads in each location, but ensuring that identity, network segmentation, logging, policy enforcement, and incident response work as a single system.

Hybrid cloud security usually spans:
- **IaaS**: Infrastructure as a Service, where you manage operating systems, applications, and data.
- **PaaS**: Platform as a Service, where the provider manages more of the runtime and platform stack.
- **SaaS**: Software as a Service, where you mainly govern identity, data, and configuration.

In CISSP terms, you should think in terms of shared responsibility, trust boundaries, and control inheritance. The cloud provider secures the underlying platform, but you remain responsible for identity, data classification, access policy, workload configuration, and monitoring.

**Reference architecture and landing zone design**

A secure hybrid cloud usually starts with a **landing zone**, which is a standardized, governed cloud foundation for accounts, subscriptions, networking, logging, and security controls. A landing zone reduces ad hoc deployments and gives you a repeatable baseline for new workloads.

Common landing zone elements include:
- A **management account** or **root subscription** for centralized governance and billing.
- Separate **shared services**, **security**, **logging**, and **workload** accounts or subscriptions.
- Centralized identity integration with the enterprise directory.
- Baseline guardrails for regions, tags, encryption, and network controls.
- Central log collection and security monitoring.

A common reference structure is:
- **Identity account**: hosts directory integration, federation, and privileged access workflows.
- **Security account**: hosts SIEM (Security Information and Event Management), SOAR (Security Orchestration, Automation, and Response), threat detection, and security tooling.
- **Logging account**: receives immutable audit logs and control-plane events.
- **Shared services account**: hosts DNS, directory sync, patch repositories, and inspection services.
- **Workload accounts**: isolate applications by environment, business unit, or sensitivity.

This separation supports least privilege, blast-radius reduction, and clearer shared control documentation.

**Network architecture patterns**

Hybrid cloud network design should reflect trust boundaries and inspection points. Two common patterns are **hub-and-spoke** and **flat network**.

| Pattern | Strengths | Weaknesses | Best use |
|---|---|---|---|
| Hub-and-spoke | Centralized inspection, routing, DNS, and egress control | More routing complexity | Enterprise hybrid environments |
| Flat network | Simple to deploy initially | Weak segmentation, larger blast radius | Small, low-risk environments |

In a **hub-and-spoke** design:
- The **hub** contains shared services such as firewalls, VPN or direct-connect termination, DNS, proxy, and inspection appliances.
- The **spokes** host application workloads and connect to the hub for controlled east-west and north-south traffic.
- Traffic between spokes can be forced through the hub for inspection.

In a **flat network**:
- Workloads share a broader address space or fewer segmentation boundaries.
- It is easier to build, but lateral movement risk is higher.
- It is usually not ideal for regulated or high-assurance environments.

A secure hybrid design often uses:
- On-premises segmentation with VLANs, ACLs, and internal firewalls.
- Cloud virtual networks or VPCs with subnets per tier.
- Route tables that force sensitive traffic through inspection points.
- Private connectivity such as VPN, Direct Connect, ExpressRoute, or Interconnect equivalents.

**Cloud-native security services and network controls**

Cloud security is not limited to CSPM (Cloud Security Posture Management), CWPP (Cloud Workload Protection Platform), or CASB (Cloud Access Security Broker). You also need native controls that enforce traffic policy and application protection.

Key controls include:
- **Security groups**: stateful instance- or workload-level allow rules.
- **Network ACLs (NACLs)**: stateless subnet-level filters.
- **Cloud firewall**: managed or virtual firewall for centralized policy enforcement and egress control.
- **WAF**: Web Application Firewall, which protects HTTP/S applications from common attacks such as SQL injection and cross-site scripting.
- **IDS/IPS**: Intrusion Detection System / Intrusion Prevention System for threat detection and blocking.
- **API gateway security**: authentication, authorization, throttling, schema validation, and request inspection.
- **Private endpoints**: keep service traffic off the public internet.
- **DDoS protection**: absorb or mitigate volumetric and protocol attacks.

Practical distinctions:
- Use **security groups** for workload-specific allow lists.
- Use **NACLs** when you need subnet-level coarse filtering or explicit deny behavior.
- Use **WAF** in front of internet-facing web apps and APIs.
- Use **IDS/IPS** where you need deeper packet or flow inspection.
- Use **cloud firewall** or centralized inspection for egress filtering, segmentation, and policy consistency.

Example AWS-style security group rule:
```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 443 \
  --cidr 10.20.0.0/16
```

Example Azure Network Security Group conceptually:
```bash
az network nsg rule create \
  --resource-group rg-app-prod \
  --nsg-name nsg-app \
  --name Allow-HTTPS-From-FrontEnd \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.10.0.0/24 \
  --destination-port-ranges 443
```

**Zero trust in hybrid cloud**

Zero trust architecture assumes no implicit trust based on network location. Every request must be continuously verified using identity, device posture, context, and policy.

Core zero trust principles:
- **Verify explicitly**: authenticate and authorize every access request.
- **Use least privilege**: grant only the minimum access needed.
- **Assume breach**: design as if attackers already have a foothold.

In hybrid cloud, zero trust usually includes:
- Strong identity federation between on-premises directory and cloud identity provider.
- **Conditional access** based on user risk, device compliance, location, and application sensitivity.
- **Device posture** checks such as encryption, patch level, EDR (Endpoint Detection and Response) status, and jailbreak/root detection.
- **Continuous verification** through session risk evaluation and reauthentication for sensitive actions.
- **Policy Decision Point (PDP)** and **Policy Enforcement Point (PEP)** separation.

A simple zero trust flow:
1. The user authenticates through the identity provider.
2. The device posture is evaluated.
3. The policy engine checks user role, device compliance, location, and risk.
4. The PDP returns an allow, deny, or step-up authentication decision.
5. The PEP enforces the decision at the app, proxy, or gateway.

Common enforcement points:
- Identity-aware proxy
- VPN or ZTNA (Zero Trust Network Access) gateway
- API gateway
- Cloud access proxy
- Application load balancer with integrated auth

Example conditional access logic:
- Allow finance users only from compliant devices.
- Require MFA (Multi-Factor Authentication) for privileged roles.
- Block access from high-risk geographies.
- Require step-up authentication for data export actions.

**Identity federation and privileged access**

Hybrid cloud depends on consistent identity. You typically federate enterprise identity into cloud services rather than creating unmanaged local accounts.

Important identity patterns:
- **SAML** (Security Assertion Markup Language) for browser-based federation.
- **OIDC** (OpenID Connect) for modern application and API authentication.
- **SCIM** (System for Cross-domain Identity Management) for automated provisioning and deprovisioning.
- **MFA** for privileged and remote access.
- **PAM** (Privileged Access Management) for just-in-time elevation and session control.

Practical implementation goals:
- Centralize authentication in the enterprise IdP.
- Disable long-lived shared admin accounts.
- Use role-based access control and attribute-based access control where possible.
- Separate human admin access from workload-to-workload service identities.
- Rotate secrets and prefer short-lived tokens over static credentials.

Example federation concept:
```text
User -> Enterprise IdP -> Cloud IAM role assumption -> Temporary credentials -> Resource access
```

Example AWS CLI role assumption:
```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/SecurityAuditor \
  --role-session-name audit-session
```

**Configuration management and infrastructure as code**

Configuration management and infrastructure as code (IaC) are essential in hybrid cloud because manual changes create drift, inconsistency, and audit gaps. IaC means you define infrastructure in version-controlled templates and deploy it through controlled pipelines.

Benefits:
- Repeatable builds
- Faster recovery
- Better change tracking
- Easier compliance review
- Reduced configuration drift

Common tools and patterns:
- Terraform, CloudFormation, ARM/Bicep, or similar declarative templates.
- Git-based change control with pull requests and approvals.
- Policy-as-code for guardrails.
- Automated drift detection against the desired state.

Secure IaC practices:
- Store templates in version control.
- Review changes through peer approval.
- Scan templates for insecure defaults.
- Parameterize secrets rather than hardcoding them.
- Use separate state and credentials per environment.
- Restrict who can apply changes to production.

Example Terraform security group:
```hcl
resource "aws_security_group" "app" {
  name        = "app-sg"
  description = "App tier security group"
  vpc_id      = aws_vpc.main.id

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["10.10.0.0/24"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["10.0.0.0/8"]
  }
}
```

Example drift detection workflow:
1. Compare deployed resources to the approved template.
2. Flag unauthorized changes.
3. Investigate whether the change was emergency remediation or malicious drift.
4. Reconcile the environment back to the approved baseline.
5. Record the exception and update the change record if needed.

> If you allow direct console changes in production, you should treat them as exceptions and reconcile them quickly. Untracked drift weakens both security and auditability.

**Logging, monitoring, and evidence collection**

Hybrid cloud logging must cover both control plane and data plane activity. Control plane logs show who changed what; data plane logs show what traffic or workload behavior occurred.

You should collect:
- Authentication and authorization logs
- API activity and administrative actions
- Network flow logs
- WAF and firewall logs
- OS and application logs
- Cloud audit logs
- Key management and secret access logs
- Container and orchestration logs if applicable

A secure logging pipeline usually includes:
- Centralized ingestion from cloud accounts and on-prem systems.
- Time synchronization using NTP (Network Time Protocol).
- Immutable storage or write-once retention for critical logs.
- Role separation between log producers, log readers, and log administrators.
- Alerting on suspicious changes to logging configuration.

Example logging pipeline:
```text
Cloud audit logs -> Central log account -> SIEM -> Alerting -> SOAR playbooks
```

Practical example for AWS-style audit logging:
```bash
aws cloudtrail create-trail \
  --name org-trail \
  --s3-bucket-name central-audit-logs \
  --is-multi-region-trail
```

For incident response, cloud evidence acquisition should include:
- API audit logs
- Instance metadata and configuration snapshots
- Disk snapshots
- Object storage access logs
- IAM policy history
- Security group and firewall rule history
- Container image digests and deployment manifests

**Backup and recovery architecture**

Backup design in hybrid cloud must protect against accidental deletion, ransomware, insider threat, and provider-side or regional failure. Backups should be encrypted, immutable where possible, and tested regularly.

Key backup controls:
- **Encryption at rest** and in transit.
- **Immutability** using object lock, WORM (Write Once, Read Many), or equivalent retention controls.
- **Air-gapped backups** or logically isolated backup accounts.
- Separate backup credentials from production credentials.
- Backup copies in a different account, subscription, or region.
- Restore testing on a defined schedule.

Backup architecture patterns:
- **3-2-1 rule**: three copies, two media types, one offsite copy.
- **3-2-1-1-0**: three copies, two media types, one offsite, one immutable or offline, zero verified errors.
- Tiered recovery objectives based on business impact.

Restore testing frequency should match criticality:
- Critical systems: frequent restore validation, often monthly or more often.
- Important systems: quarterly testing may be acceptable.
- Low-risk systems: periodic testing, but still documented.

Practical backup considerations:
- Back up identity configuration, not just data.
- Include infrastructure templates and secrets recovery procedures.
- Test application-consistent restores, not only file-level restores.
- Verify that backups are recoverable into a clean environment.

Example backup policy concept:
```json
{
  "backupVaultLock": true,
  "retentionDays": 365,
  "crossAccountCopy": true,
  "crossRegionCopy": true,
  "encryptionKey": "alias/backup-key"
}
```

**Incident response in cloud environments**

Cloud incident response differs from traditional on-prem response because evidence can disappear quickly and actions are often API-driven. Your playbooks should include cloud-specific containment and preservation steps.

A cloud-focused incident response sequence:
1. Confirm the alert and identify the affected account, subscription, project, or tenant.
2. Preserve evidence before making destructive changes.
3. Quarantine the account or workload.
4. Revoke or rotate credentials and tokens.
5. Snapshot affected systems and volumes.
6. Collect control-plane logs and configuration history.
7. Escalate to the cloud provider if provider services or shared infrastructure are involved.
8. Eradicate the threat and restore from trusted baselines.
9. Conduct lessons learned and update controls.

Cloud-specific containment actions:
- Disable compromised access keys or service principals.
- Move workloads to a quarantine security group or isolated subnet.
- Suspend peering or route propagation if lateral movement is suspected.
- Lock down object storage buckets and key management policies.
- Freeze infrastructure changes if you need to preserve state.

Order of operations matters:
- Capture volatile evidence first when feasible.
- Snapshot disks before powering off instances if memory is not the priority.
- Preserve logs and IAM history before rotating everything.
- Avoid destroying compromised resources until you have enough evidence.

Provider escalation paths may include:
- Support case for suspected platform issues
- Abuse or security reporting channels
- Dedicated enterprise incident contacts
- Legal and compliance notification workflows

**Compliance, governance, and shared controls**

Hybrid cloud security must map technical controls to governance requirements. CISSP often expects you to understand how frameworks and shared responsibility intersect.

Common frameworks and governance references:
- **ISO/IEC 27001** for information security management systems.
- **NIST** guidance such as NIST SP 800-53, 800-37, and 800-207.
- Industry or regional privacy and regulatory requirements as applicable.

Shared control documentation should define:
- Which controls are provided by the cloud provider.
- Which controls are customer-managed.
- Which controls are shared.
- How evidence is collected for audits.

Examples of shared controls:
- Provider manages physical security and core infrastructure.
- Customer manages IAM roles, data classification, encryption key usage, and application configuration.
- Both parties may contribute to logging, incident response, and resilience.

Governance practices:
- Maintain a control matrix mapping requirements to technical implementations.
- Use tagging and asset inventory for ownership and classification.
- Enforce policy exceptions with expiration dates and approvals.
- Review privileged access and network exceptions regularly.

**Key management and secrets integration**

Key management is central to hybrid cloud because encryption only helps if keys are protected and access is controlled. Use centralized KMS (Key Management Service) or HSM (Hardware Security Module) integration where possible.

Good practices:
- Use customer-managed keys for sensitive workloads when policy requires it.
- Separate key administration from data administration.
- Rotate keys according to policy and risk.
- Restrict key usage by service, account, or application identity.
- Log all key access and administrative actions.

Practical patterns:
- Encrypt backups with a dedicated backup key.
- Use envelope encryption for application data.
- Store secrets in a managed secrets vault rather than in code or environment files.
- Integrate cloud KMS with on-prem applications through approved APIs and service identities.

Example secret retrieval pattern:
```bash
aws secretsmanager get-secret-value \
  --secret-id prod/db/password
```

**Practical hybrid cloud topology example**

A common secure design for a regulated application looks like this:
- Users connect through an identity-aware access layer.
- Internet traffic terminates at a WAF and load balancer.
- Application traffic enters a spoke network with restricted security groups.
- Shared services and inspection live in a hub network.
- On-premises systems connect through private connectivity and route through the hub.
- Logs flow to a separate logging account.
- Backups replicate to an isolated backup account with immutability enabled.

Example traffic flow:
```text
Internet -> WAF -> Load Balancer -> App Subnet -> Database Subnet
On-prem -> VPN/Direct Connect -> Hub Firewall -> Spoke Workload
Workload -> Central Logging -> SIEM
Workload -> Backup Vault -> Immutable Copy
```

**Exam focus and decision points**

When answering CISSP-style questions, choose the option that best improves governance, segmentation, and resilience without creating unnecessary complexity.

Look for these priorities:
- Identity-first access over network trust.
- Segmentation and centralized inspection over flat connectivity.
- Immutable, tested backups over simple backup existence.
- Version-controlled IaC over manual configuration.
- Centralized logging and evidence preservation over ad hoc troubleshooting.
- Shared control clarity over assumptions about provider responsibility.

Common traps:
- Assuming the cloud provider handles application security.
- Using public exposure when private endpoints or federation would work.
- Treating MFA as a complete zero trust solution.
- Ignoring restore testing after implementing backups.
- Failing to preserve evidence before containment.

**Quick reference checklist**

- Define a landing zone with separate accounts or subscriptions.
- Use hub-and-spoke networking for controlled inspection and segmentation.
- Apply security groups, NACLs, firewalls, WAF, and IDS/IPS appropriately.
- Enforce zero trust with conditional access, device posture, and continuous verification.
- Federate identity and use short-lived credentials.
- Manage infrastructure through version-controlled IaC and drift detection.
- Centralize logs in an immutable repository and SIEM.
- Design backups for immutability, isolation, encryption, and restore testing.
- Build cloud-specific incident response playbooks.
- Map controls to ISO 27001, NIST, and shared responsibility documentation.
