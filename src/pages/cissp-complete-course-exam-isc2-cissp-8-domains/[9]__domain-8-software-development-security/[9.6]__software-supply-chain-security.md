## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Software supply chain security overview**

Software supply chain security protects the integrity, authenticity, and trustworthiness of software from source code through build, release, distribution, deployment, and runtime. The goal is to reduce the risk that a malicious actor, compromised dependency, tampered build system, or poisoned update channel can introduce untrusted code into production.

A useful exam distinction is that supply chain security is broader than vulnerability management. Vulnerability management asks, “Is this component known to be vulnerable?” Supply chain security also asks, “Where did this artifact come from, who built it, what was changed, and can I prove it has not been tampered with?”

- **Integrity** focuses on detecting unauthorized change.
- **Authenticity** proves the artifact or metadata came from a trusted source.
- **Provenance** records how an artifact was produced.
- **Traceability** links deployed software back to source, build inputs, and approvals.

**Threats across the software lifecycle**

Attackers target the weakest trust boundary in the chain. Common attack paths include malicious open-source dependencies, compromised maintainer accounts, poisoned build pipelines, stolen signing keys, tampered container images, and compromised software update infrastructure.

Typical risk areas include:

- **Source repositories** with weak access control, exposed tokens, or unreviewed merge rights.
- **Build systems** that trust unpinned dependencies, shared runners, or internet access during compilation.
- **Artifact repositories** that allow unsigned or unverified packages.
- **Deployment environments** that run images without signature verification or runtime integrity checks.
- **Update channels** for endpoints, IoT devices, and firmware that lack rollback protection or secure metadata validation.

> A supply chain compromise often succeeds without exploiting the application itself. The attacker instead abuses trust in the people, tools, or processes that produce the application.

**Source code repository security**

Repository security is the first control point because source is the root of trust for most software. If an attacker gains maintainer access, they can insert malicious code, alter release tags, or approve unsafe changes.

Key controls include:

- **Multi-factor authentication (MFA)** for maintainers and administrators.
- **Protected branches** that require reviews, status checks, and signed commits where feasible.
- **Protected release workflows** that separate code review from release approval.
- **Least privilege** for contributors, bots, and automation accounts.
- **Token hygiene** with short-lived credentials, scoped permissions, and secret rotation.
- **Commit and tag signing** to help detect unauthorized changes.
- **Audit logging** for repository events, branch protection changes, and credential use.

Practical controls in GitHub or GitLab often include:

```yaml
# Example policy concepts, not vendor-specific syntax
branch_protection:
  require_pull_request_reviews: true
  require_status_checks: true
  require_signed_commits: true
  restrict_pushes: true
  enforce_admins: true
release:
  require_two_person_review: true
  sign_artifacts: true
  publish_from_tag_only: true
```

Repository compromise is a supply chain risk because the attacker can publish a malicious version that appears legitimate. Protecting the SCM (source code management) platform itself matters as much as protecting the code.

**Build environment hardening**

The build environment transforms source into artifacts, so it must be treated as a high-trust system. A compromised build runner can inject malware, exfiltrate secrets, or alter outputs without changing source code.

Important hardening measures include:

- **Network isolation** for build jobs that do not need external access.
- **Ephemeral runners** that are destroyed after each job.
- **Pinned dependencies** and locked manifests to reduce dependency drift.
- **Hermetic builds** where inputs are controlled and reproducible.
- **Minimal base images** and reduced toolchains to shrink attack surface.
- **Secret segregation** so build jobs only receive the credentials they need.
- **Build log review** to detect accidental secret leakage.

Build caching improves speed but can create trust issues if cache contents are poisoned or shared across untrusted jobs. Treat caches as potentially hostile unless you control their provenance and access.

A secure build pipeline often looks like this:

```bash
# Example: isolate a build container and avoid host secrets
docker run --rm \
  --network=none \
  -v "$PWD:/src:ro" \
  -w /src \
  golang:1.22 \
  sh -c 'go build -trimpath -mod=readonly -o /out/app ./...'
```

Build-time secret leakage is a common failure mode. Avoid passing long-lived credentials into compilation steps unless absolutely necessary, and prefer short-lived tokens or workload identity.

**Provenance, attestations, and trust roots**

Provenance answers who built the artifact, from what source, using which dependencies, and under what conditions. An attestation is a signed statement about that provenance or about another property, such as test results or vulnerability scan status.

Common ecosystem concepts include:

- **Sigstore** for signing and verification using modern keyless or managed-key workflows.
- **Fulcio** as a certificate authority in the Sigstore ecosystem.
- **Rekor** as an append-only transparency log for signed artifacts and attestations.
- **in-toto** for supply chain layout and step-by-step attestations.
- **SLSA** (Supply-chain Levels for Software Artifacts) as a maturity framework for build integrity and provenance.

Verification is more than checking a signature. You also need to validate the trust root and the certificate chain or identity binding behind the signature. In practice, policy engines may require:

- A trusted issuer or certificate authority.
- A valid identity claim for the builder or maintainer.
- A matching artifact digest.
- A required attestation type, such as build provenance or test attestation.
- A transparency log entry or equivalent auditability.

Example verification flow:

```bash
# Example Sigstore-style verification concept
cosign verify \
  --certificate-identity "build@company.example" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  ghcr.io/example/app:1.2.3
```

Policy enforcement is critical. Verification should happen automatically in CI/CD and at deployment time, not only during manual review. If the artifact does not meet policy, the pipeline should fail closed.

**SBOM standards and use cases**

A Software Bill of Materials (SBOM) is a machine-readable inventory of software components, dependencies, and sometimes licenses, versions, hashes, and relationships. SBOMs help you answer what is in a product and where it came from.

Common SBOM formats and standards include:

- **SPDX** (Software Package Data Exchange)
- **CycloneDX**
- **SWID** tags in some enterprise and endpoint contexts

SBOMs support several operational use cases:

- **Vulnerability management** by identifying affected components when a new CVE (Common Vulnerabilities and Exposures) is disclosed.
- **Compliance** by documenting third-party and open-source components, licenses, and distribution obligations.
- **Incident response** by quickly locating where a vulnerable or malicious component is deployed.
- **Asset visibility** by improving software inventory across applications, containers, and firmware.
- **Supplier assurance** by requiring vendors to disclose component composition.

An SBOM does not prove that software is safe or authentic. It is an inventory, not a trust guarantee. You still need provenance, signatures, and verification controls to confirm that the SBOM itself is associated with the correct artifact and has not been altered.

Practical example of an SBOM workflow:

```bash
# Generate an SBOM for a container image
syft ghcr.io/example/app:1.2.3 -o spdx-json > app.spdx.json

# Scan the SBOM for known vulnerabilities
grype sbom:app.spdx.json
```

In incident response, SBOMs help you answer:
- Which products contain the affected library?
- Which versions are exposed?
- Which business services depend on the vulnerable component?
- Which systems need emergency patching or compensating controls?

**Vulnerability scanning, malware scanning, and provenance verification**

These controls solve different problems and are not interchangeable.

| Control | Primary question | Typical output | Limitation |
|---|---|---|---|
| Vulnerability scanning | Is a known CVE present? | Findings, severity, remediation | Does not prove artifact origin |
| Malware scanning | Does the file match malicious patterns or signatures? | Detections, quarantine actions | May miss novel or targeted malware |
| Provenance verification | Was this artifact built by a trusted process? | Pass/fail trust decision | Does not guarantee absence of vulnerabilities |

Use vulnerability scanning to manage exposure, malware scanning to detect known malicious content, and provenance verification to confirm the artifact came from an approved build path.

**Software update frameworks for endpoints, IoT, and firmware**

Update security is critical because attackers often target the update channel to distribute malicious code at scale. Secure update frameworks protect metadata, signatures, and rollback behavior.

Important controls include:

- **Signed update metadata** so devices can verify authenticity before installation.
- **Rollback protection** to prevent installation of older vulnerable versions.
- **Anti-downgrade controls** that reject versions below a minimum trusted level.
- **Staged rollout** to limit blast radius if an update is defective.
- **Recovery mechanisms** such as dual-bank firmware or fail-safe partitions.
- **Time-bound trust** where expired metadata cannot be reused indefinitely.

Firmware and IoT environments often have constrained hardware, so update verification must be efficient and resilient. If possible, use secure boot plus signed updates so the device verifies both the boot chain and the update payload.

A simplified update trust model:

```text
Vendor signing key -> signed metadata -> signed update package -> device verification -> install -> rollback protection check
```

If rollback protection is absent, an attacker may force a device onto an older vulnerable version even when the update itself is signed.

**Supplier acquisition and contract controls**

Supply chain security is not only technical. Procurement and legal controls help enforce security expectations before software is purchased or integrated.

Useful contract clauses include:

- **Security requirements** for secure development, patching, and vulnerability disclosure.
- **Right to audit** or independent assessment rights.
- **Notification requirements** for breaches, compromised signing keys, and material vulnerabilities.
- **SBOM delivery** and update obligations for component changes.
- **Third-party assurance evidence** such as SOC 2 reports, ISO 27001 certification, or penetration test summaries when appropriate.
- **Support and patch timelines** for critical vulnerabilities.
- **Subprocessor or subcontractor disclosure** where relevant.

These clauses matter because a supplier compromise can become your compromise. Contractual obligations create leverage for remediation, transparency, and timely notification.

**Runtime supply chain protections**

Even after deployment, you still need to detect tampering and enforce integrity. Runtime controls reduce the impact of a compromised artifact that bypassed earlier checks.

Common protections include:

- **Container runtime restrictions** such as read-only filesystems, dropped Linux capabilities, seccomp, and AppArmor or SELinux profiles.
- **Image signature verification** before admission to the cluster.
- **Admission control policies** that block untrusted registries or unsigned images.
- **Runtime integrity monitoring** for unexpected file changes, injected processes, or altered binaries.
- **File integrity monitoring** on critical hosts and containers.
- **Behavioral detection** for suspicious network connections or privilege escalation.

Example Kubernetes admission concept:

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicy
metadata:
  name: require-signed-images
spec:
  # Conceptual example: enforce signature verification before pod admission
  validations:
    - expression: "object.spec.containers.all(c, c.image.startsWith('ghcr.io/company/'))"
      message: "Only approved registries are allowed"
```

Runtime controls are especially important when a malicious dependency is discovered after deployment or when a build compromise is detected late.

**Enterprise response to a compromised dependency or maintainer**

A mature response process should cover both technical containment and business communication. The response differs depending on whether the issue is a vulnerable dependency, a malicious package, or a compromised maintainer account.

A practical enterprise flow is:

1. **Identify the scope** using SBOMs, dependency graphs, package manifests, and deployment inventories.
2. **Verify the threat** by checking signatures, provenance, package hashes, and repository audit logs.
3. **Contain exposure** by blocking the package, disabling the maintainer account, revoking tokens, or quarantining affected builds.
4. **Rebuild from trusted inputs** using clean runners and pinned dependencies.
5. **Redeploy verified artifacts** only after provenance and policy checks pass.
6. **Rotate secrets and keys** if any build, release, or repository credentials may be exposed.
7. **Notify stakeholders** including legal, procurement, incident response, and affected customers if required.
8. **Document lessons learned** and update controls, such as branch protections or dependency allowlists.

If a maintainer account is compromised, the immediate concern is not just code tampering. You must also assume release tags, package publishing rights, and signing credentials may be at risk.

**Repository, build, and release controls in practice**

A strong supply chain program combines preventive, detective, and corrective controls across the lifecycle.

- **Preventive**
  - MFA for all privileged identities
  - Protected branches and release approvals
  - Dependency pinning and allowlists
  - Signed commits, tags, and artifacts
  - Ephemeral, isolated build runners
- **Detective**
  - Audit logs for SCM, CI/CD, and package registries
  - Provenance and attestation verification
  - SBOM comparison against deployed assets
  - Runtime integrity monitoring
- **Corrective**
  - Key rotation and token revocation
  - Rebuild and re-sign from trusted sources
  - Rollback to known-good versions
  - Incident response and supplier notification

**Exam-oriented distinctions**

- **SBOM** tells you what is inside software.
- **Provenance** tells you how software was built.
- **Signature verification** tells you whether the artifact or metadata was signed by a trusted identity.
- **Attestation** provides signed evidence about a build, test, or scan result.
- **Secure boot** protects the device startup chain.
- **Secure update frameworks** protect the delivery and installation of new software.
- **Runtime integrity** protects the running system after deployment.

**Practical command examples**

Generate and inspect an SBOM:

```bash
syft dir:. -o spdx-json > sbom.json
jq '.packages[] | {name, versionInfo}' sbom.json | head
```

Verify a container image signature:

```bash
cosign verify ghcr.io/example/app:1.2.3
```

Check a package for known vulnerabilities:

```bash
grype dir:.
```

Inspect a Git repository for suspicious release history:

```bash
git log --show-signature --decorate --oneline --graph --all
git tag -v v1.2.3
```

**Key takeaways**

- Supply chain security protects the path from source to runtime, not just the code itself.
- SBOMs improve inventory, vulnerability response, and compliance, but they do not prove trust.
- Provenance, attestations, and signatures help verify origin and build integrity.
- Repository, build, and release systems are high-value targets and need strong access control.
- Secure update frameworks and runtime controls are essential for endpoints, IoT, containers, and firmware.
- Procurement and contract controls are part of the security model, not an afterthought.
