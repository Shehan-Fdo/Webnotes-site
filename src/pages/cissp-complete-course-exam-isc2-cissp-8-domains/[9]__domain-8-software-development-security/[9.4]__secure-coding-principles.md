## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Secure coding principles: the CISSP view**

Secure coding is the practice of designing, implementing, and validating software so it resists abuse, misuse, and exploitation. For CISSP, you should think beyond language syntax and focus on how code, frameworks, libraries, runtime settings, and operational controls reduce risk across the application lifecycle.

The core idea is simple: you prevent classes of vulnerabilities by making unsafe behavior hard or impossible, then you verify that the controls actually work. That means secure defaults, least privilege, input validation, safe memory handling, strong session controls, and repeatable review and testing.

**Core secure coding goals**

- Reduce the attack surface by removing unnecessary features, endpoints, and dependencies.
- Enforce **least privilege** in code, runtime, database access, and service accounts.
- Validate and normalize all untrusted input before use.
- Prefer vetted libraries and frameworks over custom security logic.
- Fail securely when validation, authentication, or authorization checks fail.
- Log security-relevant events without exposing secrets or sensitive data.

Secure coding is not only about preventing injection. CISSP-style questions often test whether you understand the broader software assurance model: standards, reviews, testing, configuration hardening, and lifecycle governance.

**Secure coding standards and organizational controls**

Organizations should not rely on individual developer judgment alone. They need written standards, coding guidelines, approved libraries, and review gates that make secure behavior repeatable.

- **Secure coding standards** define required practices for languages and platforms.
- **Coding guidelines** translate those standards into team-specific rules and examples.
- **Approved libraries/frameworks** reduce the chance of reintroducing known weaknesses.
- **Threat modeling** helps you identify abuse paths before implementation.
- **Security requirements** should be part of design, not added after release.

A practical control set often includes:
- Mandatory peer review for security-sensitive changes
- Static analysis in the build pipeline
- Dependency scanning for known vulnerable packages
- Secure configuration baselines for frameworks and services
- Release approval for high-risk changes

> If a team cannot explain how it prevents unsafe patterns from entering production, it does not have a mature secure coding program.

**Memory safety and memory corruption risks**

Memory corruption is a major secure coding topic, especially in C and C++ and other environments with manual memory management or unsafe pointer use. CISSP expects you to recognize the vulnerability classes and the mitigations.

Common memory safety issues include:

- **Buffer overflows**: writing past the end of a fixed-size buffer
- **Stack overflows**: corrupting stack memory, often through oversized local writes or recursion abuse
- **Heap overflows**: corrupting dynamically allocated memory
- **Integer overflows/underflows**: arithmetic wraps around and causes incorrect allocation or bounds checks
- **Use-after-free**: accessing memory after it has been released
- **Double free**: freeing the same memory twice, corrupting allocator state
- **Format string attacks**: attacker-controlled format strings leak memory or write to memory

Example of unsafe C code:

```c
#include <stdio.h>
#include <string.h>

void copy_name(const char *input) {
    char buf[16];
    strcpy(buf, input);   // unsafe: no bounds check
    printf(buf);          // unsafe: format string risk
}
```

Safer approach:

```c
#include <stdio.h>
#include <string.h>

void copy_name(const char *input) {
    char buf[16];
    snprintf(buf, sizeof(buf), "%s", input);
    printf("%s\n", buf);
}
```

Mitigations for memory corruption:

- Use safe library functions with explicit bounds checks.
- Prefer managed languages when appropriate, but do not assume they eliminate all security bugs.
- Validate lengths before allocation and copy operations.
- Use compiler and runtime protections such as:
  - Stack canaries
  - Address Space Layout Randomization (ASLR)
  - Data Execution Prevention (DEP) / non-executable memory
  - Control-flow protections where available
- Avoid manual pointer arithmetic unless necessary.
- Free memory exactly once and set pointers to `NULL` after release when practical.
- Use static analysis, fuzz testing, and code review to catch edge cases.

For integer safety, check arithmetic before allocation:

```c
#include <stdlib.h>
#include <stdint.h>

void *alloc_items(uint32_t count, uint32_t size) {
    if (count == 0 || size == 0) return NULL;
    if (count > SIZE_MAX / size) return NULL; // overflow check
    return malloc((size_t)count * size);
}
```

**Language and runtime-specific secure coding concerns**

Different platforms have different risk profiles. CISSP questions may ask you to identify the right control for the environment.

- In **C/C++**, unsafe pointers, manual allocation, and unchecked copies are common sources of memory corruption.
- In **Java, C#, and similar managed runtimes**, memory corruption is less common, but logic flaws, deserialization issues, injection, and insecure reflection still matter.
- In **JavaScript and web runtimes**, cross-site scripting, prototype pollution, and unsafe deserialization patterns are common concerns.
- In **Python, Ruby, and scripting environments**, dynamic typing and extensive library use can hide injection, path traversal, and unsafe command execution risks.

Secure coding practices by environment:

- Use vetted standard libraries instead of custom parsing or crypto.
- Avoid shelling out to system commands when a native API exists.
- Treat deserialization as untrusted input unless the format and parser are explicitly safe.
- Use type-safe APIs and strict validation for file paths, URLs, and command arguments.
- Keep runtime and framework versions patched.

**Input validation, output encoding, and canonicalization**

Input validation is necessary but not sufficient. You should validate data at trust boundaries, normalize it, and encode it for the target context.

- **Allowlist validation** is preferred over denylist validation.
- **Canonicalization** converts equivalent representations into a single form before validation.
- **Output encoding** prevents data from being interpreted as code in the destination context.

Examples:
- HTML context: encode `<`, `>`, `&`, `"`, and `'`
- SQL context: use parameterized queries
- Shell context: avoid shell interpretation or escape carefully
- LDAP, XML, and command-line contexts: use context-specific safe APIs

A common mistake is validating input before canonicalization. For example, a path may appear safe until encoded traversal sequences are decoded.

**Secure database access**

Database security is more than preventing SQL injection. CISSP expects you to understand privilege design, stored procedure risks, and ORM limitations.

Key practices:

- Use **parameterized queries** or prepared statements.
- Grant database accounts only the permissions required for the application.
- Separate read, write, and administrative accounts when possible.
- Restrict access to specific schemas, tables, and stored procedures.
- Audit database access and sensitive query activity.
- Protect secrets used for database authentication.

Example of parameterized SQL:

```python
import sqlite3

conn = sqlite3.connect("app.db")
cur = conn.cursor()
cur.execute("SELECT id, email FROM users WHERE email = ?", (email,))
row = cur.fetchone()
```

Least-privilege database account example:

```sql
CREATE USER app_reader IDENTIFIED BY 'strong-secret';
GRANT SELECT ON app_schema.orders TO app_reader;
GRANT SELECT ON app_schema.customers TO app_reader;
-- No INSERT, UPDATE, DELETE, or DDL privileges
```

Stored procedure caveats:
- Stored procedures are not automatically safe if they build dynamic SQL.
- They can still be vulnerable to injection if parameters are concatenated into SQL strings.
- They should use parameter binding internally and enforce authorization.

ORM misuse risks:
- Object-relational mappers can still generate unsafe queries if developers concatenate raw fragments.
- “Convenience” methods that accept raw SQL strings can bypass protections.
- Lazy loading and broad object exposure can create overfetching or data leakage issues.

**Session management and token handling**

Secure session management protects authenticated users from hijacking and fixation attacks. It is a frequent exam topic because weak session controls undermine otherwise strong authentication.

Important concepts:

- **Session fixation**: an attacker sets or predicts a session identifier before login and reuses it after authentication.
- **Session timeout**: the session expires after a fixed period of inactivity or absolute lifetime.
- **Idle timeout**: the session ends after no user activity for a defined period.
- **Logout invalidation**: logout should destroy the server-side session and invalidate the client token.
- **Token revocation**: compromised or no-longer-valid tokens should be rejected.
- **Refresh tokens**: long-lived tokens should be protected, rotated, and revocable.

Best practices:
- Regenerate the session ID after authentication and privilege changes.
- Use short-lived access tokens and longer-lived refresh tokens only when needed.
- Store refresh tokens securely and rotate them on use if the platform supports it.
- Invalidate server-side sessions on logout, password reset, and account compromise.
- Bind session cookies to secure attributes:
  - `Secure`
  - `HttpOnly`
  - `SameSite`
- Set reasonable idle and absolute timeouts.
- Detect concurrent or anomalous session use where appropriate.

Example secure cookie settings in a web framework:

```http
Set-Cookie: sessionid=abc123; Secure; HttpOnly; SameSite=Lax; Path=/; Max-Age=1800
```

Example session regeneration flow:

```text
1. User submits valid credentials.
2. Server creates a new session identifier.
3. Server invalidates the pre-authentication session.
4. Server stores the authenticated identity in the new session.
```

> Do not rely on client-side logout alone. If the server still accepts the token, the session is not truly invalidated.

**Secure defaults and framework hardening**

Frameworks and services should ship with secure defaults, but you should verify them. Many breaches happen because debug features, permissive headers, or weak cookie settings remain enabled in production.

Common hardening actions:
- Disable debug mode and verbose error pages in production.
- Remove sample accounts, demo endpoints, and test credentials.
- Set secure HTTP headers where appropriate:
  - `Content-Security-Policy`
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options`
  - `Referrer-Policy`
  - `Permissions-Policy`
- Enforce HTTPS and redirect HTTP to HTTPS.
- Configure secure cookie flags in the framework.
- Turn off directory listing, stack traces, and detailed exception output.
- Restrict CORS (Cross-Origin Resource Sharing) to trusted origins only.

Example hardening snippet for a web app:

```python
# Pseudocode for framework configuration
DEBUG = False
SESSION_COOKIE_SECURE = True
SESSION_COOKIE_HTTPONLY = True
SESSION_COOKIE_SAMESITE = "Lax"
CSRF_COOKIE_SECURE = True
```

Example security headers:

```http
Content-Security-Policy: default-src 'self'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: no-referrer
```

**Secure use of cryptography**

Cryptography is often misused in application code. Secure coding means using approved algorithms and libraries correctly, not inventing custom schemes.

- Use vetted cryptographic libraries.
- Prefer modern, approved algorithms and modes.
- Never hardcode secrets, keys, or salts in source code.
- Use strong random number generators for tokens and keys.
- Separate encryption from authentication when the library requires it, or use authenticated encryption modes.
- Rotate keys according to policy and protect them with proper access controls.

Common mistakes:
- Using weak hashes for password storage
- Reusing initialization vectors where uniqueness is required
- Rolling your own encryption
- Using predictable tokens generated from timestamps or usernames

**Secure coding for mobile, desktop, and embedded applications**

Application security is not limited to web apps. CISSP may ask about platform-specific concerns.

Mobile applications:
- Protect sensitive data in local storage and backups.
- Use platform keystores for secrets.
- Avoid logging tokens, PII (personally identifiable information), or credentials.
- Validate deep links, intents, and inter-app communication.
- Assume the device may be rooted or jailbroken.

Desktop applications:
- Protect local configuration files and cached credentials.
- Validate file paths and user-supplied file names.
- Avoid unsafe auto-update mechanisms.
- Restrict plugin and macro execution where possible.

Embedded and IoT (Internet of Things) applications:
- Minimize exposed services and debug interfaces.
- Remove default credentials.
- Use secure boot and signed firmware updates when available.
- Protect physical interfaces such as JTAG or serial consoles.
- Plan for constrained memory and CPU without sacrificing input validation.

**Secure code review and testing**

Validation should combine human review and automated testing. No single technique finds every defect.

Common validation methods:
- **Manual code review**: effective for logic flaws, authorization mistakes, and insecure assumptions
- **Static analysis**: finds patterns such as unsafe calls, tainted data flows, and dead code
- **Dynamic testing**: exercises the running application
- **Fuzz testing**: feeds malformed or unexpected inputs to trigger crashes and edge cases
- **Penetration testing**: validates exploitability from an attacker perspective
- **Abuse-case testing**: checks how the system behaves under malicious or unintended use

A practical review workflow:
1. Review security-sensitive changes manually.
2. Run static analysis and dependency scanning in CI/CD.
3. Execute unit and integration tests for security controls.
4. Fuzz parsers, file handlers, and network-facing inputs.
5. Perform penetration testing before release for high-risk systems.

Example fuzzing target concept:

```c
int parse_message(const unsigned char *data, size_t len);
```

This kind of function is a good fuzz target because it processes untrusted input and may hide boundary bugs.

**Abuse cases and secure design thinking**

Abuse cases describe how an attacker might misuse a feature. They complement use cases by forcing you to think about failure modes and adversarial behavior.

Examples:
- A user repeatedly requests password resets to cause denial of service.
- An attacker uploads a file with a deceptive extension to trigger code execution.
- A low-privilege account attempts to access another tenant’s data.
- A mobile app accepts a forged deep link that bypasses normal navigation.

Use abuse cases to drive:
- Input validation rules
- Authorization checks
- Rate limiting
- Error handling
- Logging and alerting

**Practical examples**

Memory corruption example:
- A C function copies user input into a fixed-size buffer without bounds checking.
- An attacker sends a longer string and overwrites adjacent memory.
- Mitigation: bounds-checked APIs, compiler protections, fuzz testing, and code review.

Session fixation example:
- A web app accepts a session ID from the URL or fails to regenerate the session after login.
- An attacker preloads a known session ID and waits for the victim to authenticate.
- Mitigation: regenerate session IDs after authentication and reject session IDs from unsafe sources.

Secure database privilege example:
- The application only needs to read order data.
- The database account should have `SELECT` on the required tables only.
- It should not have `DROP`, `ALTER`, or broad schema privileges.

Framework hardening example:
- Production deployment disables debug mode.
- Cookies use `Secure` and `HttpOnly`.
- Security headers are enabled.
- Error pages do not reveal stack traces or SQL details.

**Exam focus and common traps**

- Do not confuse input validation with output encoding; both are needed in different places.
- Do not assume stored procedures are inherently safe.
- Do not assume ORM use eliminates injection risk.
- Do not treat logout as complete unless the server invalidates the session or token.
- Do not rely on memory-safe languages alone; logic flaws and injection still matter.
- Do not forget secure defaults, patching, and dependency management.
- Do not ignore mobile, desktop, and embedded platforms when the question asks about application security broadly.

**Quick recall checklist**

- **Memory safety**: buffer overflow, stack/heap overflow, integer overflow, use-after-free, format string
- **Session security**: regenerate IDs, idle timeout, logout invalidation, token revocation, refresh token rotation
- **Database security**: parameterized queries, least privilege, stored procedure caution, ORM scrutiny
- **Framework hardening**: disable debug, secure cookies, security headers, HTTPS, safe error handling
- **Validation**: manual review, static analysis, fuzzing, penetration testing, abuse-case testing
- **Software assurance**: standards, guidelines, vetted libraries, secure SDLC controls
