**Secure coding checklist: purpose and scope**

A secure coding checklist turns security requirements into repeatable engineering controls. You use it to reduce common software flaws early in the lifecycle, when fixes are cheaper and less disruptive than post-release remediation.

The checklist should cover the full path from design to deployment:
- **Requirements**: define security and privacy expectations before code is written.
- **Design**: reduce attack surface with trust boundaries, least privilege, and secure defaults.
- **Implementation**: prevent injection, memory corruption, auth bypass, and insecure deserialization.
- **Testing**: verify controls with static and dynamic analysis, code review, and abuse cases.
- **Release**: ensure dependencies, build artifacts, and signing are controlled.
- **Operations**: monitor for vulnerabilities, patch quickly, and preserve traceability.

A strong checklist is specific enough to be actionable, but not so rigid that teams ignore it. It should map to the application’s risk profile, language, and deployment model.

**Core secure coding checklist items**

Use the checklist to verify that the application handles data, identity, and trust boundaries safely.

- **Input validation**: validate all external input on the server side.
  - Prefer allowlists over denylists.
  - Enforce type, length, range, format, and encoding checks.
  - Normalize input before validation when canonicalization issues matter.
- **Output encoding**: encode data for the target context.
  - HTML-encode for web pages.
  - URL-encode for query strings.
  - SQL parameterization is preferred over manual escaping.
  - Command arguments should be passed as structured parameters, not concatenated strings.
- **Authentication**: verify identity securely.
  - Use MFA where appropriate.
  - Store passwords with adaptive one-way hashing such as `bcrypt`, `scrypt`, or `Argon2`.
  - Protect against brute force with rate limiting and lockout controls that avoid denial-of-service risk.
- **Authorization**: enforce access control on every request.
  - Check object-level and function-level permissions.
  - Use least privilege and deny by default.
  - Avoid relying on client-side checks.
- **Session management**: protect session tokens.
  - Use secure, `HttpOnly`, and `SameSite` cookie attributes where applicable.
  - Regenerate session identifiers after authentication or privilege change.
  - Set reasonable expiration and idle timeout values.
- **Cryptography**: use approved algorithms and correct modes.
  - Use TLS for data in transit.
  - Use modern authenticated encryption where possible.
  - Never roll your own crypto.
  - Store keys in a protected key management system, not in source code.
- **Error handling and logging**: avoid leaking sensitive details.
  - Return generic errors to users.
  - Log enough detail for investigation without exposing secrets.
  - Protect logs from tampering and unauthorized access.
- **Secrets management**: keep credentials out of code and images.
  - Use environment variables, secret stores, or vault services.
  - Rotate secrets regularly.
  - Remove hardcoded keys, tokens, and passwords from repositories.
- **Dependency management**: control third-party risk.
  - Pin versions where appropriate.
  - Review transitive dependencies.
  - Track known vulnerabilities and update promptly.
- **Secure defaults**: ship with the safest reasonable configuration.
  - Disable debug mode in production.
  - Turn off unnecessary services and endpoints.
  - Require explicit opt-in for risky features.
- **Memory safety**: especially important in languages with manual memory management.
  - Prevent buffer overflows, use-after-free, and integer overflow.
  - Prefer safe libraries and bounds-checked operations.
- **Concurrency and state**: avoid race conditions and inconsistent authorization decisions.
  - Protect shared resources.
  - Use atomic operations or locks where needed.
  - Re-check authorization at the point of use.

**Practical secure coding examples**

Parameterization prevents SQL injection by separating code from data.

```python
import sqlite3

user_id = input("User ID: ")
conn = sqlite3.connect("app.db")
cur = conn.cursor()

cur.execute("SELECT email FROM users WHERE id = ?", (user_id,))
row = cur.fetchone()
print(row)
```

A vulnerable pattern concatenates untrusted input into a command.

```bash
# Vulnerable
grep "$USER_INPUT" /var/log/app.log
```

A safer pattern avoids shell interpretation by passing arguments directly in code.

```python
import subprocess

pattern = input("Pattern: ")
subprocess.run(["grep", pattern, "/var/log/app.log"], check=True)
```

For web applications, output encoding helps prevent cross-site scripting (XSS).

```html
<!-- Unsafe if user data is inserted directly -->
<div>{{ user_comment }}</div>

<!-- Safer when the template engine escapes output by default -->
<div>{{ user_comment | escape }}</div>
```

**Secure development lifecycle integration**

Secure coding works best when it is embedded into the software development lifecycle (SDLC), not added at the end.

1. Define security requirements during planning.
2. Perform threat modeling to identify likely abuse paths.
3. Review architecture for trust boundaries and privilege separation.
4. Enforce secure coding standards in code review and pull requests.
5. Run static application security testing (SAST) and dependency scanning in continuous integration.
6. Validate with dynamic application security testing (DAST), fuzzing, and manual testing.
7. Gate releases on unresolved high-risk findings.
8. Monitor production for new vulnerabilities and suspicious behavior.

Threat modeling helps you decide which checklist items matter most. For example, a payment API needs strong authorization, input validation, and audit logging, while an internal batch job may emphasize secrets handling and dependency control.

**SBOM review: what it is and why it matters**

An <abbr title="Software Bill of Materials">SBOM</abbr> is an inventory of software components used to build an application. It improves visibility into direct and transitive dependencies, which helps you assess supply-chain risk, vulnerability exposure, and license obligations.

You review an SBOM to answer questions such as:
- What components are included?
- Which versions are in use?
- Which components are transitive dependencies?
- Are any components known to be vulnerable?
- Are there unsupported or end-of-life packages?
- Are there licensing or provenance concerns?

An SBOM does not eliminate risk by itself. It gives you the data needed to make informed patching, exception, and procurement decisions.

**Common SBOM formats and fields**

SBOMs are commonly represented in machine-readable formats such as:
- **SPDX**: Software Package Data Exchange.
- **CycloneDX**: widely used for application security and dependency tracking.
- **SWID**: Software Identification tags, more common in some enterprise contexts.

Typical SBOM fields include:
- Component name
- Version
- Supplier or publisher
- Package identifier
- Hashes or checksums
- Dependency relationships
- License information
- Build metadata
- Provenance or source references

A useful SBOM should be accurate, current, and tied to a specific build artifact. If it does not match the deployed version, it has limited operational value.

**How to review an SBOM**

Review the SBOM as part of release validation and vulnerability management.

1. Confirm the SBOM matches the exact build or container image you intend to deploy.
2. Identify direct dependencies and transitive dependencies.
3. Compare component versions against vulnerability intelligence sources.
4. Check for unsupported, abandoned, or end-of-life packages.
5. Review licenses for policy conflicts.
6. Verify hashes or signatures when available.
7. Determine whether vulnerable components are actually reachable or used in the deployed path.
8. Decide whether to patch, replace, mitigate, or accept the risk.

A component may appear in an SBOM but not be exploitable in practice. For example, a library used only in test code may not affect production risk, but you should still understand why it is present.

**Example SBOM review workflow**

A practical review process often combines automation and human judgment.

```bash
# Example: generate an SBOM with Syft
syft packages dir:. -o cyclonedx-json > sbom.json

# Example: scan the SBOM for known vulnerabilities with Grype
grype sbom:sbom.json
```

You can also inspect container images directly:

```bash
# Generate an SBOM for a container image
syft registry.example.com/app:1.2.3 -o spdx-json > app.spdx.json

# Inspect the image contents
docker image inspect registry.example.com/app:1.2.3
```

For dependency ecosystems, package managers often provide lockfiles that support reproducible builds:
- `package-lock.json` for npm
- `yarn.lock` for Yarn
- `poetry.lock` for Python
- `Gemfile.lock` for Ruby
- `go.sum` for Go
- `Cargo.lock` for Rust

Lockfiles are not SBOMs, but they help establish the exact dependency set that an SBOM should reflect.

**Supply-chain risk indicators**

When reviewing an SBOM, pay attention to signals that increase risk.

- **Outdated versions**: older packages may contain known vulnerabilities.
- **Transitive sprawl**: many nested dependencies increase attack surface.
- **Unmaintained packages**: abandoned components may never receive fixes.
- **Typosquatting risk**: suspicious package names can indicate malicious lookalikes.
- **Unsigned or unverified artifacts**: weak provenance makes tampering harder to detect.
- **Build drift**: the deployed artifact differs from the reviewed artifact.
- **License conflicts**: incompatible licenses can create legal and operational issues.

> If the SBOM does not correspond to the exact artifact in production, treat it as advisory rather than authoritative.

**Secure coding and SBOM relationship**

Secure coding reduces the chance that your own code introduces vulnerabilities. SBOM review reduces the chance that third-party code introduces vulnerabilities. You need both because modern applications depend heavily on external libraries, frameworks, and container layers.

- Secure coding addresses:
  - injection flaws
  - broken access control
  - insecure session handling
  - unsafe deserialization
  - cryptographic misuse
- SBOM review addresses:
  - vulnerable libraries
  - hidden transitive dependencies
  - unsupported components
  - provenance gaps
  - license and compliance exposure

Together, they support defense in depth across the software supply chain.

**Practical controls for teams**

Teams usually implement secure coding and SBOM review through policy and automation.

- Add secure coding rules to pull request templates and code review checklists.
- Run SAST, dependency scanning, and secret scanning in CI/CD pipelines.
- Fail builds on critical vulnerabilities unless an exception is approved.
- Generate an SBOM for every release artifact.
- Sign artifacts and verify signatures before deployment.
- Track exceptions with expiration dates and compensating controls.
- Re-scan released artifacts when new CVEs are published.

A simple CI pipeline might look like this:

```yaml
name: build-and-scan

on:
  push:
    branches: [main]

jobs:
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: docker build -t app:ci .
      - name: Generate SBOM
        run: syft app:ci -o cyclonedx-json > sbom.json
      - name: Scan vulnerabilities
        run: grype sbom:sbom.json --fail-on high
      - name: Secret scan
        run: gitleaks detect --source . --no-banner
```

**Common exam angles**

For CISSP-style questions, focus on the control objective rather than the tool name.

- Choose **input validation** and **output encoding** to prevent injection and XSS.
- Choose **parameterized queries** instead of string concatenation for database access.
- Choose **least privilege** and **server-side authorization** to prevent unauthorized access.
- Choose **secure secret storage** and **rotation** to reduce credential exposure.
- Choose **SBOM review** to identify vulnerable third-party components and supply-chain risk.
- Choose **artifact signing** and **hash verification** to improve provenance assurance.
- Choose **patch management** when the SBOM reveals a vulnerable component in use.

**Quick reference checklist**

Use this as a compact review aid before implementation or release.

- Validate input on the server side.
- Encode output for the correct context.
- Parameterize database and command interactions.
- Enforce authentication and authorization separately.
- Protect sessions with secure cookie settings.
- Store passwords with strong adaptive hashing.
- Keep secrets out of source code and images.
- Use approved cryptography and TLS.
- Log securely without exposing sensitive data.
- Scan dependencies and generate an SBOM for each release.
- Review SBOM entries for version, vulnerability, license, and provenance issues.
- Sign artifacts and verify integrity before deployment.
- Reassess risk when dependencies or threat intelligence change.

**Key takeaway**

Secure coding prevents many defects before they ship, while SBOM review gives you visibility into the software you consume. In practice, you use both to reduce application risk, improve traceability, and support faster, more informed remediation decisions.
