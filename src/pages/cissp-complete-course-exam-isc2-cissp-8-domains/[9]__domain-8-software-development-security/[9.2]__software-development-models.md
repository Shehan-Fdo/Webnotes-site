**Software Development Models in CISSP context**

Software development models describe how you plan, build, test, release, and maintain software. For CISSP, the key idea is not memorizing a single lifecycle diagram, but understanding how each model affects security governance, assurance, change control, and evidence quality.

The Software Development Life Cycle (SDLC) is the broader framing concept. A model is the method you use to move through SDLC phases. In exam scenarios, you should identify where security activities fit into the lifecycle and how the chosen model changes the timing of reviews, approvals, and testing.

- **SDLC** phases commonly include:
  - Requirements
  - Design
  - Implementation
  - Testing
  - Deployment
  - Operations and maintenance
  - Retirement/disposal
- Security should be integrated throughout, not added only at the end.
- The model determines how often you revisit requirements, design, and test evidence.
- Regulated environments usually favor stronger documentation and traceability.

**SDLC as a CISSP framing concept**

CISSP questions often ask where a control belongs in the lifecycle. You should think in terms of checkpoints:

- Requirements phase:
  - Define security, privacy, legal, and compliance requirements.
  - Classify data and identify trust boundaries.
  - Establish acceptance criteria for security.
- Design phase:
  - Perform architecture and threat modeling reviews.
  - Choose secure patterns, cryptography, and access control mechanisms.
  - Validate segregation of duties and least privilege.
- Implementation phase:
  - Apply secure coding standards.
  - Conduct peer code review and static analysis.
  - Manage secrets, dependencies, and build integrity.
- Testing phase:
  - Run dynamic testing, penetration testing, and abuse-case testing.
  - Verify security requirements and negative test cases.
  - Confirm fixes and regression coverage.
- Deployment phase:
  - Review release approvals, change records, and rollback plans.
  - Validate configuration hardening and environment parity.
- Maintenance phase:
  - Monitor vulnerabilities, patching, logging, and incident response.
  - Reassess risk after changes or new threats.
- Retirement phase:
  - Archive or destroy data according to retention rules.
  - Revoke access, decommission services, and sanitize media.

A secure SDLC is not a separate model; it is a way of embedding security controls into whichever model you use.

**Waterfall model**

Waterfall is a linear, sequential model. Each phase is completed before the next begins, and changes become more expensive as the project moves forward. This model is easy to understand and document, which is why it appears often in exam scenarios involving formal approvals, fixed scope, and regulated delivery.

Waterfall fits best when requirements are stable and well understood. It is weaker when the environment changes quickly or when users need to refine requirements through feedback.

- Strengths:
  - Strong documentation and traceability
  - Clear phase gates and sign-offs
  - Easier governance and audit evidence
  - Predictable milestones and budgets
- Weaknesses:
  - Late discovery of requirement errors
  - Limited flexibility for change
  - Security issues may surface late if reviews are deferred
  - User feedback arrives too late to reshape design

In CISSP terms, Waterfall often supports stronger formal control, but it can create risk if security is not validated early enough. A common mistake is assuming that “more documentation” automatically means “more security.” It only helps if the right security reviews happen at the right time.

**Waterfall security checkpoints**

- Requirements:
  - Security requirements are captured up front.
  - Compliance and privacy obligations are documented early.
- Design:
  - Architecture review and threat modeling occur before build.
  - Cryptographic and access control decisions are approved.
- Implementation:
  - Code review and secure coding checks happen during build.
- Testing:
  - Security testing validates the finished product before release.
- Deployment:
  - Change control board approval and release validation are common.
- Maintenance:
  - Patching and defect fixes follow formal change procedures.

Example governance flow in a Waterfall project:

```text
Business requirements -> Security requirements -> Architecture review -> Build -> QA/security testing -> CAB approval -> Production release
```

A practical control example might look like this in a change record:

```yaml
change_id: CHG-2026-0412
model: waterfall
security_gate:
  requirements_signed_off: true
  threat_model_completed: true
  sast_passed: true
  pen_test_required: true
  rollback_plan_approved: true
release_status: pending_cab
```

**Agile model**

Agile uses short iterations, frequent feedback, and incremental delivery. It is designed to adapt to changing requirements, which makes it useful when users cannot fully define needs at the start. In CISSP questions, Agile is often contrasted with Waterfall on documentation, change control, and assurance timing.

Agile does not mean “no process.” It means the process is iterative and collaborative. Security must be built into each sprint or iteration, otherwise the team can accumulate risk quickly.

- Strengths:
  - Fast feedback and early value delivery
  - Better adaptation to changing requirements
  - Frequent testing and integration
  - Security can be validated incrementally
- Weaknesses:
  - Documentation may be lighter unless intentionally enforced
  - Scope can drift without strong governance
  - Security work may be deprioritized if not made explicit
  - Traceability can suffer in poorly managed teams

For CISSP, the important distinction is that Agile can support security well if you define security stories, acceptance criteria, and release gates. If you do not, it can become difficult to prove compliance or consistent control execution.

**Agile security integration**

- Requirements:
  - Add security user stories and abuse cases to the backlog.
  - Define acceptance criteria for confidentiality, integrity, and availability.
- Design:
  - Review architecture incrementally as features evolve.
  - Use lightweight threat modeling for each epic or major change.
- Implementation:
  - Use peer review, linting, and static application security testing (SAST).
  - Manage dependencies and secrets in the pipeline.
- Testing:
  - Run automated unit, integration, and security tests every sprint.
  - Include dynamic application security testing (DAST) where practical.
- Deployment:
  - Use continuous integration/continuous delivery (CI/CD) controls with approvals.
  - Promote only signed and tested artifacts.
- Maintenance:
  - Feed vulnerabilities and incidents back into the backlog.

Example Agile backlog items:

```text
US-104: As a customer, I can reset my password securely.
AC:
- Token expires in 15 minutes
- MFA required for high-risk accounts
- Reset link invalidated after use

SEC-22: As a security requirement, all API responses must avoid leaking stack traces.
AC:
- Error handling returns generic messages
- Logs retain correlation IDs only
```

Example pipeline gate:

```yaml
stages:
  - build
  - unit_test
  - sast
  - dependency_scan
  - deploy_staging
  - dast
  - approval
  - deploy_prod
```

**Spiral model**

The Spiral model is risk-driven and iterative. It combines elements of prototyping and systematic development, with each loop focused on identifying and reducing the highest risks first. This model is especially relevant when requirements are uncertain, technology is new, or security risk is significant.

A classic Spiral loop has four quadrants:

1. Determine objectives, alternatives, and constraints.
2. Identify and resolve risks.
3. Develop and verify the next-level product.
4. Plan the next iteration and obtain stakeholder evaluation.

The key exam point is that each loop is not just “build more software.” It is a structured risk-management cycle. You evaluate alternatives, prototype where needed, and involve stakeholders before committing to the next pass.

- Why Spiral matters:
  - It explicitly prioritizes risk analysis.
  - It supports early validation of difficult requirements.
  - It is useful for large, complex, or high-risk systems.
- What makes it different:
  - Risk assessment is central, not incidental.
  - Prototypes may be used to reduce uncertainty.
  - Stakeholder review happens every cycle.

**Spiral loop details**

- Quadrant 1: Objectives, alternatives, constraints
  - Define what you are trying to achieve.
  - Compare design or implementation alternatives.
  - Identify constraints such as budget, regulation, latency, or platform limits.
- Quadrant 2: Risk analysis and mitigation
  - Evaluate technical, security, schedule, and operational risks.
  - Build prototypes, proofs of concept, or simulations.
  - Choose mitigations for the highest risks first.
- Quadrant 3: Development and verification
  - Build the selected solution incrementally.
  - Verify that the current iteration meets objectives.
  - Perform testing appropriate to the iteration’s maturity.
- Quadrant 4: Stakeholder evaluation and planning
  - Review results with business, security, and technical stakeholders.
  - Decide whether to continue, change direction, or stop.
  - Plan the next loop with updated risk information.

Example Spiral decision record:

```text
Loop 2 objectives:
- Validate SSO integration with legacy identity provider
- Reduce risk of token replay
- Confirm audit logging meets retention policy

Alternatives:
- SAML federation
- OIDC with broker
- Custom token bridge

Selected mitigation:
- Prototype OIDC broker in staging
- Perform threat model and replay testing
- Review with IAM, security, and operations
```

A practical governance pattern for Spiral is to require a formal risk review at the end of each loop:

```text
[Loop Review]
- Objectives met: yes/no
- Residual risk: low/medium/high
- Security findings: 3 open
- Stakeholder approval: pending
- Next loop authorized: yes/no
```

**V-model**

The V-model extends Waterfall by pairing each development phase with a corresponding test phase. The left side of the “V” defines and designs the system, while the right side validates and verifies it. CISSP often uses the V-model to emphasize traceability between requirements and tests.

This model is useful when you need strong assurance that each requirement has a matching test. It is common in safety-, security-, and compliance-sensitive environments.

- Left side:
  - User requirements
  - System requirements
  - High-level design
  - Detailed design
- Bottom:
  - Implementation
- Right side:
  - Unit testing
  - Integration testing
  - System testing
  - Acceptance testing

The main advantage is traceability. The main limitation is that it still assumes relatively stable requirements, so it is less flexible than Agile.

**V-model security mapping**

- User requirements ↔ Acceptance testing
  - Verify business and security acceptance criteria.
- System requirements ↔ System testing
  - Validate end-to-end security behavior.
- High-level design ↔ Integration testing
  - Confirm secure interfaces and trust boundaries.
- Detailed design ↔ Unit testing
  - Check secure logic, input validation, and error handling.

Example traceability matrix:

| Requirement | Design element | Test type | Evidence |
|---|---|---|---|
| MFA required for admin access | IAM design | Acceptance test | Test case AC-17 |
| Encrypt data at rest | Storage design | System test | Encryption validation report |
| Log privileged actions | Audit design | Integration test | SIEM event capture |

**Joint Application Development and Rapid Prototyping**

Joint Application Development (JAD) is a structured workshop approach that brings users, developers, analysts, and sometimes security staff together to define requirements quickly. It is useful when you need fast consensus and clear business input. In CISSP scenarios, JAD helps reduce misunderstanding, but it does not replace formal security analysis.

Rapid Prototyping builds a quick model of the system to clarify requirements or validate feasibility. It is especially useful when users cannot articulate needs well or when the technology is uncertain. The risk is that a prototype can be mistaken for a production-ready system.

- JAD:
  - Focuses on collaborative requirements gathering
  - Improves stakeholder alignment
  - Can reduce rework caused by misunderstood requirements
- Rapid Prototyping:
  - Focuses on learning and validation
  - Helps expose usability and feasibility issues early
  - Must be controlled so prototype code does not become insecure production code

> A prototype should usually be treated as disposable unless it is intentionally hardened and moved into the delivery process.

Practical distinction:

- Use JAD when you need to define what to build.
- Use prototyping when you need to test whether a design or requirement is realistic.

Example JAD agenda:

```text
1. Business objective review
2. Security and compliance constraints
3. Data classification and access needs
4. Workflow mapping
5. Exception handling and audit requirements
6. Sign-off on requirements baseline
```

Example prototype boundary control:

```text
Prototype environment:
- No production data
- No external internet access
- Separate credentials
- Time-boxed access
- Explicit disposal date
```

**Outsourcing and offshoring**

Outsourcing means a third party performs development or related services. Offshoring means the work is done in another country, often by a third party or a captive team. CISSP questions often focus on governance, accountability, and control when development is not fully internal.

The security principle is simple: you can delegate work, but you cannot delegate accountability. You still own the risk, the requirements, and the acceptance criteria.

- Risks:
  - Reduced direct oversight
  - Intellectual property exposure
  - Data residency and legal concerns
  - Time zone and communication gaps
  - Supply chain and dependency risk
- Controls:
  - Contractual security requirements
  - Right to audit
  - Background checks where lawful and appropriate
  - Secure development standards
  - Access restrictions and monitoring
  - Source code escrow or continuity planning when needed

A third-party development model should define who approves requirements, who reviews code, who tests security, and who can deploy. If those responsibilities are unclear, the organization loses control over assurance.

Example vendor control checklist:

```text
- Security requirements included in contract
- NDA and IP clauses signed
- Secure coding standard accepted
- Vulnerability disclosure process defined
- Code review evidence retained
- Pen test rights negotiated
- Data handling and retention rules documented
```

Example access control for an outsourced team:

```bash
# Create a restricted role for vendor developers
aws iam create-role \
  --role-name VendorDevRole \
  --assume-role-policy-document file://vendor-trust.json

aws iam put-role-policy \
  --role-name VendorDevRole \
  --policy-name LimitedRepoAccess \
  --policy-document file://repo-readonly.json
```

**Secure SDLC integration points**

A secure SDLC adds security activities at multiple checkpoints rather than relying on one final test. The exact implementation depends on the model, but the control objectives remain similar.

- Requirements security:
  - Define confidentiality, integrity, availability, privacy, and compliance needs.
  - Capture misuse cases and abuse cases.
  - Set measurable acceptance criteria.
- Design review:
  - Review architecture, trust boundaries, and data flows.
  - Evaluate authentication, authorization, logging, and cryptography.
  - Perform threat modeling.
- Code review:
  - Use peer review and automated analysis.
  - Check for injection, insecure deserialization, weak crypto, and logic flaws.
- Testing:
  - Validate security requirements with positive and negative tests.
  - Include SAST, DAST, dependency scanning, and penetration testing as appropriate.
- Deployment:
  - Enforce change approval, artifact integrity, and rollback readiness.
  - Harden configurations and verify environment settings.
- Maintenance:
  - Patch vulnerabilities, monitor logs, and reassess risk.
  - Revisit threat models after major changes.

Example secure SDLC control flow:

```text
Requirements -> Threat model -> Secure design review -> Code review -> SAST/DAST -> Release approval -> Monitoring and patching
```

Example command-line checks in a CI environment:

```bash
# Dependency scan
npm audit --production

# Static analysis example
semgrep --config auto src/

# Container image scan example
trivy image myapp:1.2.3
```

**Model comparison for CISSP exam prep**

| Model | Change tolerance | Documentation | Assurance style | Best fit |
|---|---:|---:|---|---|
| Waterfall | Low | High | Phase-gated | Stable, regulated projects |
| Agile | High | Medium unless enforced | Continuous and incremental | Changing requirements |
| Spiral | High | High for risk artifacts | Risk-driven iteration | Complex, high-risk systems |
| V-model | Low to medium | High | Verification and validation traceability | Assurance-heavy environments |

Use this comparison to answer scenario questions:

- Choose Waterfall when the problem is stable and formal sign-off matters.
- Choose Agile when requirements evolve and feedback is frequent.
- Choose Spiral when risk is the dominant concern.
- Choose V-model when traceability between requirements and tests is critical.

**Exam-focused distinctions**

CISSP questions often hide the answer in the process objective rather than the label.

- If the scenario emphasizes formal sign-off and sequential phases, think Waterfall.
- If it emphasizes short iterations and customer feedback, think Agile.
- If it emphasizes risk analysis and prototypes, think Spiral.
- If it emphasizes matched verification and validation steps, think V-model.
- If it emphasizes collaborative requirements workshops, think JAD.
- If it emphasizes a quick learning artifact, think Rapid Prototyping.

Common traps:

- Confusing a prototype with a production release.
- Assuming Agile removes documentation or security controls.
- Treating testing as a single phase instead of a lifecycle activity.
- Ignoring third-party development governance.
- Forgetting that security requirements belong at the start, not only at the end.

**Practical governance examples**

A regulated organization may require the following controls regardless of model:

```text
- Security requirements baseline approved before build
- Threat model updated for major design changes
- Code review required for all merges
- Automated security scans in CI/CD
- Independent security testing before production
- Change approval and rollback plan for release
- Vulnerability remediation tracked to closure
```

Example Git-based merge policy:

```yaml
branch_protection:
  require_pull_request_reviews: true
  required_status_checks:
    - build
    - unit-test
    - sast
    - dependency-scan
  dismiss_stale_reviews: true
  require_signed_commits: true
```

Example release gate for production:

```text
Release checklist:
- Security sign-off complete
- Critical vulnerabilities: 0 open
- Pen test exceptions approved
- Backup and rollback verified
- Monitoring alerts enabled
- Business owner approval recorded
```

**How to think about the lifecycle in scenarios**

When you read a CISSP question, identify three things:

1. The lifecycle phase where the issue appears.
2. The development model that shapes the process.
3. The control that best reduces risk without breaking governance.

For example, if a vendor wants to “test security after deployment because requirements are still changing,” the better answer is usually to move security earlier and use iterative validation, not to postpone assurance. If a project needs formal evidence for auditors, a model with stronger traceability and phase gates is usually more appropriate.

**Key takeaways**

- **SDLC** is the lifecycle framework; the model is the method used to move through it.
- Security belongs in requirements, design, implementation, testing, deployment, and maintenance.
- Waterfall favors control and documentation but resists change.
- Agile favors adaptability and frequent feedback but needs explicit security governance.
- Spiral centers on risk analysis and stakeholder evaluation in each loop.
- V-model strengthens traceability between development and testing.
- JAD and Rapid Prototyping help clarify requirements, but prototypes are not production by default.
- Outsourcing and offshoring increase governance and supply chain risk, so accountability and oversight remain essential.
