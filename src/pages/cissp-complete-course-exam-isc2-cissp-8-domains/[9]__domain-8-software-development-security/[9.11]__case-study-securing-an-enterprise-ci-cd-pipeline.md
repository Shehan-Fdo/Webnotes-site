## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Enterprise CI/CD security overview**

A CI/CD (Continuous Integration/Continuous Delivery or Deployment) pipeline automates code build, test, package, and release activities. In an enterprise, it becomes a high-value target because it can move trusted code into production, sign artifacts, and access secrets, infrastructure, and cloud APIs.

The security goal is to preserve **confidentiality**, **integrity**, and **availability** across the software supply chain. You protect source code, build systems, artifacts, deployment credentials, and the runtime environment as separate trust boundaries.

- **Source control** is often the first trust anchor and must be protected with strong authentication and branch controls.
- **Build systems** need isolation because they execute untrusted code from pull requests and dependencies.
- **Artifact repositories** must preserve integrity with signing, immutability, and provenance.
- **Deployment targets** require least privilege, change control, and rollback capability.

A useful mental model is to treat the pipeline as a chain of custody. If an attacker can alter code, dependencies, build scripts, secrets, or deployment steps, they can inject malicious logic into production without directly compromising the final server.

**Threat model and attack surface**

The pipeline expands the attack surface beyond application code. Common threats include credential theft, dependency poisoning, malicious pull requests, build agent compromise, insecure secrets handling, and tampering with artifacts or deployment manifests.

Typical enterprise risks include:

- **Compromised developer accounts** that bypass code review or push malicious changes.
- **Dependency confusion** or typosquatting that pulls a malicious package from a public registry.
- **Build server compromise** that modifies binaries, injects backdoors, or exfiltrates secrets.
- **Artifact tampering** where unsigned or mutable artifacts are replaced after testing.
- **Misconfigured runners** that expose tokens, Docker sockets, or cloud metadata services.
- **Infrastructure-as-Code (IaC)** drift that causes production to diverge from approved configurations.

> A pipeline is only as trustworthy as its weakest privileged component. If the build agent can reach secrets, registries, and production APIs, it becomes a crown-jewel system.

**Core security principles for CI/CD**

Use layered controls rather than relying on one tool. The most effective controls reduce privilege, isolate execution, verify integrity, and make changes traceable.

- **Least privilege** limits what each job, service account, and human operator can do.
- **Segmentation** separates source, build, test, artifact, and deployment zones.
- **Strong authentication** protects developer, maintainer, and service identities.
- **Integrity controls** verify code, dependencies, artifacts, and configuration.
- **Nonrepudiation** comes from signed commits, signed artifacts, and audit logs.
- **Separation of duties** prevents one person or one system from approving and deploying unreviewed changes.

**Source control protections**

Source repositories are the starting point for most pipelines. If attackers can alter source or pipeline definitions, they can control downstream automation.

Recommended controls:

- Require **multi-factor authentication (MFA)** for all privileged users.
- Enforce **branch protection** on main or release branches.
- Require **pull request reviews** and status checks before merge.
- Restrict who can modify pipeline definitions, build scripts, and IaC templates.
- Use **signed commits** and verify signatures where supported.
- Protect secrets from being stored in repository history.

Example Git configuration for signing commits:

```bash
git config --global user.signingkey ABCD1234
git config --global commit.gpgsign true
git commit -S -m "Update deployment manifest"
```

Example branch protection policy concepts:

```text
- Require pull request approval: enabled
- Require status checks: enabled
- Require linear history: enabled
- Restrict force pushes: enabled
- Restrict who can push: enabled
```

**Pipeline design and trust boundaries**

A secure pipeline separates duties across stages and minimizes the blast radius of compromise. Each stage should have only the access it needs.

A common enterprise flow looks like this:

```text
Developer workstation
   -> Source control
      -> CI build runner
         -> Test environment
            -> Artifact repository
               -> CD deployment controller
                  -> Production environment
```

Security implications by stage:

- **Developer workstation** should use endpoint protection, disk encryption, and phishing-resistant MFA.
- **Source control** should store pipeline-as-code and enforce review on changes.
- **CI build runner** should be ephemeral when possible and isolated from other workloads.
- **Artifact repository** should store immutable, signed outputs.
- **CD controller** should deploy with scoped credentials and approval gates.
- **Production environment** should accept only validated artifacts and approved configuration.

**Secrets management**

Secrets are one of the most common failure points in CI/CD. API keys, tokens, certificates, and signing keys must never be hardcoded in scripts or stored in plain text variables.

Best practices:

- Store secrets in a dedicated **secrets manager** or vault.
- Inject secrets at runtime and keep them short-lived when possible.
- Scope secrets to a single pipeline, environment, or job.
- Mask secrets in logs and prevent echoing them to consoles.
- Rotate secrets regularly and immediately after suspected exposure.
- Use workload identity or federated identity instead of long-lived static credentials when possible.

Example environment variable usage in a job:

```bash
export DEPLOY_TOKEN="$(vault kv get -field=token secret/prod/deploy)"
curl -H "Authorization: Bearer $DEPLOY_TOKEN" https://api.example.com/deploy
```

Example GitHub Actions-style secret reference:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        env:
          API_TOKEN: ${{ secrets.API_TOKEN }}
        run: ./deploy.sh
```

> Never pass secrets through command-line arguments when the process list or shell history could expose them.

**Build agent hardening**

Build agents and runners are high-risk because they execute code from many contributors and often have access to sensitive systems. Treat them as ephemeral, locked-down systems.

Key controls:

- Use **ephemeral runners** that are destroyed after each job.
- Run jobs in containers or virtual machines with strong isolation.
- Disable unnecessary services, shells, and package managers.
- Prevent access to the host Docker socket unless absolutely required.
- Block access to instance metadata services unless the job needs cloud credentials.
- Patch runner images frequently and rebuild them from trusted baselines.
- Log job execution, environment changes, and artifact outputs.

Example hardening ideas for a Linux runner:

```bash
sudo useradd --system --no-create-home --shell /usr/sbin/nologin ci-runner
sudo chmod 700 /opt/ci-runner
sudo systemctl disable --now docker.socket
```

**Dependency and supply chain security**

Modern pipelines consume third-party libraries, base images, and build tools. These dependencies are a major supply chain risk.

Defensive measures:

- Pin dependency versions to reduce unexpected changes.
- Use private mirrors or approved registries for packages and container images.
- Verify checksums and signatures where available.
- Generate and review a software bill of materials (**SBOM**) for critical releases.
- Scan dependencies for known vulnerabilities and license issues.
- Block direct internet access from build jobs when feasible.

Example package pinning:

```text
requests==2.32.3
cryptography==42.0.8
```

Example container image pinning by digest:

```dockerfile
FROM ubuntu:24.04@sha256:9b8f...
```

Example SBOM generation with Syft:

```bash
syft dir:. -o spdx-json > sbom.spdx.json
```

**Artifact integrity and provenance**

Artifacts should be immutable once tested and approved. If an attacker can replace a build output after validation, the pipeline loses trust.

Use these controls:

- Store artifacts in an **immutable repository** or enable write-once policies.
- Sign artifacts and verify signatures before deployment.
- Record build provenance, including source commit, build environment, and dependency set.
- Promote the same artifact through environments instead of rebuilding per environment.
- Separate build and release permissions so the build job cannot directly deploy to production.

Example signing with `cosign`:

```bash
cosign sign --key cosign.key registry.example.com/app:1.4.2
cosign verify --key cosign.pub registry.example.com/app:1.4.2
```

Example provenance concept:

```text
Artifact: app-1.4.2.tar.gz
Source commit: a1b2c3d4
Builder: ci-runner-07
Timestamp: 2026-04-26T10:15:00Z
Checksum: sha256:...
```

**Infrastructure as Code and configuration control**

IaC tools such as Terraform, CloudFormation, and Ansible make infrastructure repeatable, but they also create a path for large-scale misconfiguration if changes are not reviewed.

Good practices:

- Treat IaC files as code and subject them to the same review controls.
- Validate plans before apply.
- Separate plan and apply permissions.
- Store state securely and restrict access to it.
- Detect drift between declared and actual infrastructure.
- Review changes to security groups, IAM roles, firewall rules, and secrets references carefully.

Example Terraform workflow:

```bash
terraform fmt -check
terraform validate
terraform plan -out=tfplan
terraform apply tfplan
```

**Access control and separation of duties**

CI/CD environments often fail when one identity can both change code and release it. Separation of duties reduces fraud and accidental release of unapproved changes.

Practical patterns:

- Developers can commit code but cannot approve their own changes.
- Build service accounts can compile and test but cannot modify production.
- Release managers can approve deployment but cannot alter source code.
- Security teams can review pipeline policy and audit logs without direct deployment rights.
- Production access is time-bound and logged.

A role model might look like this:

| Role | Typical permissions | Restrictions |
|---|---|---|
| Developer | Commit code, open pull requests | No direct production deploy |
| Reviewer | Approve merges | No code push to protected branches |
| CI service account | Build, test, publish artifacts | No interactive login |
| CD service account | Deploy approved artifacts | No source write access |
| Security admin | Review logs, policies, alerts | No routine app changes |

**Logging, monitoring, and auditability**

You need visibility into who changed what, when, and from where. Logs support incident response, compliance, and forensic analysis.

Log and monitor:

- Authentication events for source control, CI, CD, and artifact systems.
- Pipeline job start, stop, success, and failure events.
- Changes to pipeline definitions, secrets, and approval rules.
- Artifact creation, signing, promotion, and deployment events.
- Privileged actions such as role changes or manual overrides.

Useful telemetry sources include:

- Source control audit logs
- CI/CD platform logs
- Cloud control plane logs
- Endpoint detection and response (**EDR**) alerts on runners
- Security information and event management (**SIEM**) correlation rules

Example log query concept:

```text
Find deployments to production that occurred without a matching approved change ticket.
```

**Testing and validation controls**

Security testing should be built into the pipeline rather than added at the end. Automated checks reduce the chance that vulnerable code reaches production.

Common pipeline tests:

- **Static application security testing (SAST)** for source code issues.
- **Software composition analysis (SCA)** for dependency vulnerabilities.
- **Dynamic application security testing (DAST)** for running application behavior.
- **Container image scanning** for OS and package vulnerabilities.
- **IaC scanning** for insecure cloud or network settings.
- **Secret scanning** to detect leaked credentials.

Example security scan commands:

```bash
trivy image registry.example.com/app:1.4.2
gitleaks detect --source . --no-banner
checkov -d terraform/
```

Use gating thresholds carefully. For example, you might block production deployment on critical vulnerabilities but allow lower-severity findings to proceed with a documented exception process.

**Change management and release governance**

A secure pipeline still needs formal change control. Automation does not eliminate governance; it changes how you enforce it.

Recommended controls:

- Require change tickets for production releases.
- Use approval gates for high-risk deployments.
- Maintain rollback plans and tested recovery procedures.
- Record deployment windows and maintenance exceptions.
- Tie releases to versioned artifacts and approved commits.
- Use canary or blue-green deployments to reduce blast radius.

Example release gate sequence:

1. Developer submits pull request.
2. Reviewer approves code and pipeline changes.
3. CI runs tests, scans, and builds signed artifact.
4. Security or release manager approves promotion.
5. CD deploys the exact signed artifact to production.
6. Monitoring validates health and triggers rollback if needed.

**Cloud and container-specific considerations**

Cloud-native pipelines often rely on container registries, managed runners, and IAM (Identity and Access Management). These components introduce additional trust dependencies.

Focus areas:

- Restrict IAM roles to the minimum required actions.
- Use short-lived tokens and federated identity for cloud access.
- Protect container registries with image signing and access control.
- Scan base images and rebuild frequently.
- Avoid privileged containers unless there is a documented need.
- Limit Kubernetes service account permissions and namespace access.

Example Kubernetes deployment hardening snippet:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: ci-job
spec:
  serviceAccountName: ci-job-sa
  containers:
    - name: build
      image: alpine:3.20
      securityContext:
        runAsNonRoot: true
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
```

**Incident response for pipeline compromise**

If you suspect pipeline compromise, treat it as a supply chain incident. The response must include containment, eradication, recovery, and validation.

Immediate actions:

- Disable compromised accounts, tokens, and runners.
- Freeze deployments and block artifact promotion.
- Preserve logs, build outputs, and repository history.
- Revoke signing keys if they may be exposed.
- Rebuild trusted runners from clean images.
- Verify the integrity of deployed artifacts and rollback if needed.

Recovery should include:

- Revalidating source commits and dependency versions.
- Re-signing clean artifacts with protected keys.
- Reviewing all pipeline changes made during the incident window.
- Updating detection rules for the observed attack path.

**Exam-focused takeaways**

For CISSP, focus on the security objectives and governance implications rather than memorizing one vendor’s toolset. The exam often tests whether you choose controls that reduce risk at the right layer.

High-value concepts to remember:

- **Least privilege** and **separation of duties** are foundational.
- **Integrity** of code, dependencies, and artifacts is critical in CI/CD.
- **Secrets management** must prevent leakage in logs, scripts, and environment variables.
- **Immutable, signed artifacts** are safer than rebuilding differently per environment.
- **Audit logs** and approval workflows support accountability and forensics.
- **Ephemeral, isolated runners** reduce persistence and lateral movement.

A secure enterprise pipeline is not just a DevOps concern. It is a governance, risk, and assurance system that directly affects production trust.
