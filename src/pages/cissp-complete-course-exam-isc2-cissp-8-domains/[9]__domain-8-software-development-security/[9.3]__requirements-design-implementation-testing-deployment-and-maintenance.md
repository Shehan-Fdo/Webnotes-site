## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Security lifecycle overview**

The security lifecycle connects business needs to technical controls. In CISSP terms, you translate requirements into design, implement controls, verify them through testing, deploy with minimal risk, and maintain them over time. The goal is not just to build a secure system, but to keep it secure as threats, users, and business needs change.

A strong lifecycle is iterative:
- Requirements define what the system must achieve and what risks it must address.
- Design turns those needs into architecture and control choices.
- Implementation applies the design in code, infrastructure, and operations.
- Testing validates that controls work as intended.
- Deployment introduces the solution into production safely.
- Maintenance preserves security, availability, and compliance after release.

**Requirements gathering and analysis**

Requirements are the foundation of secure system development. If you miss a security requirement early, you often pay more later to retrofit it. Good requirements are specific, testable, and traceable to business objectives, legal obligations, and risk decisions.

Security requirements typically come from:
- Business goals and mission needs
- Risk assessments and threat modeling
- Legal, regulatory, and contractual obligations
- Privacy requirements and data classification
- Availability and resilience targets
- Audit findings and prior incidents

Common security requirement categories include:
- **Confidentiality** requirements for protecting sensitive data
- **Integrity** requirements for preventing unauthorized modification
- **Availability** requirements for ensuring service continuity
- **Accountability** requirements for traceability and nonrepudiation
- **Privacy** requirements for lawful collection, use, retention, and deletion

A useful way to write requirements is to make them measurable:
- “All administrative access must use MFA” is better than “Admins should be secure.”
- “Logs must be retained for 365 days” is better than “Keep logs for a long time.”
- “RTO must be 4 hours and RPO must be 15 minutes” is better than “Recovery should be fast.”

<dl>
<dt><abbr title="Recovery Time Objective">RTO</abbr></dt>
<dd>The maximum acceptable time to restore a service after disruption.</dd>
<dt><abbr title="Recovery Point Objective">RPO</abbr></dt>
<dd>The maximum acceptable data loss measured in time.</dd>
</dl>

Requirements should also distinguish between:
- **Functional requirements**: what the system does
- **Nonfunctional requirements**: how well it does it, including security, performance, and reliability

Examples of security-related nonfunctional requirements:
- Authentication must support centralized identity and MFA
- Sensitive data must be encrypted at rest and in transit
- Audit logs must be tamper-evident and time-synchronized
- Privileged actions must require approval or dual control
- Backups must be tested regularly for restore capability

**Security design principles**

Design is where you choose the architecture and control patterns that satisfy requirements. CISSP expects you to prefer designs that reduce risk systematically rather than relying on a single control.

Core design principles include:
- **Least privilege**: grant only the access needed to perform a task
- **Need to know**: limit access to information required for the job
- **Defense in depth**: layer controls so one failure does not expose the system
- **Separation of duties**: split critical tasks among multiple people or roles
- **Fail secure**: default to a secure state when a failure occurs
- **Economy of mechanism**: keep designs simple to reduce mistakes
- **Complete mediation**: check every access request, not just the first one
- **Open design**: security should not depend on secrecy of the design
- **Psychological acceptability**: controls should be usable enough that people follow them
- **Least common mechanism**: minimize shared components that create coupling or leakage

A secure design also considers:
- Trust boundaries and data flows
- Attack surface reduction
- Secure defaults
- Resilience and redundancy
- Logging, monitoring, and alerting
- Key management and secrets handling
- Identity and access management integration

A practical design process often starts with data classification and trust boundaries:
- Identify where sensitive data enters, moves, and exits the system
- Mark boundaries between user devices, application tiers, databases, and third parties
- Decide where authentication, authorization, encryption, and logging occur

Example trust boundary sketch:
```text
[User Device] -> [WAF / Reverse Proxy] -> [Web Tier] -> [App Tier] -> [Database]
      |                |                    |              |             |
   MFA/SSO         TLS termination      Input validation  RBAC        Encryption at rest
```

**Architecture and control selection**

Different architectures support different security goals. You choose controls based on risk, not habit.

Common control types:
- **Administrative controls**: policies, procedures, training, background checks
- **Technical controls**: firewalls, encryption, IAM, EDR, DLP
- **Physical controls**: locks, guards, badges, cameras, mantraps

Control behavior matters too:
- **Preventive** controls stop incidents before they happen
- **Detective** controls identify events after or during occurrence
- **Corrective** controls restore systems or reduce impact
- **Deterrent** controls discourage attacks
- **Compensating** controls provide an alternative when the primary control is not feasible
- **Recovery** controls help restore operations after an incident

Examples:
- MFA is preventive
- SIEM alerts are detective
- Backups are recovery
- A security guard is deterrent
- Network segmentation can be compensating if application-level controls are limited

Design should also account for common security models:
- **Bell-LaPadula** focuses on confidentiality
- **Biba** focuses on integrity
- **Clark-Wilson** emphasizes well-formed transactions and separation of duties
- **Brewer-Nash** supports conflict-of-interest controls in dynamic access environments

These models are most useful when you need to reason about access patterns, especially in regulated or high-integrity environments.

**Implementation and secure configuration**

Implementation is the phase where design becomes reality. This includes code, infrastructure, cloud services, identity configuration, and operational procedures. A secure design can fail if implementation is inconsistent or misconfigured.

Implementation best practices:
- Use secure baselines and hardened images
- Automate configuration to reduce drift
- Store secrets in a vault, not in code or plain text files
- Apply patching and dependency management continuously
- Enforce code review and change control
- Validate input and encode output in applications
- Log security-relevant events consistently
- Restrict administrative interfaces and management ports

Example Linux hardening commands:
```bash
# Update packages
sudo apt update && sudo apt upgrade -y

# Check listening ports
sudo ss -tulpn

# Enable firewall
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw enable

# Verify service status
systemctl status ssh
```

Example SSH hardening snippet:
```conf
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers alice bob
ClientAliveInterval 300
ClientAliveCountMax 2
```

Example application configuration for secrets handling:
```yaml
database:
  host: db.internal
  username: app_user
  password: ${DB_PASSWORD}
logging:
  level: INFO
  redact:
    - authorization
    - cookie
    - password
```

In cloud environments, implementation often includes:
- Identity and access management roles and policies
- Security groups and network ACLs
- Encryption keys and key rotation
- Object storage access controls
- Logging services and alerting rules
- Infrastructure as Code (IaC) templates

Example AWS-style least privilege policy concept:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": ["arn:aws:s3:::example-bucket/reports/*"]
    }
  ]
}
```

**Secure development and change control**

Secure implementation depends on disciplined development and change management. Without control, even good code can introduce vulnerabilities or outages.

Key practices:
- Use version control for all code and configuration
- Require peer review for changes
- Separate development, test, and production environments
- Track dependencies and third-party components
- Use static and dynamic analysis tools
- Treat infrastructure changes like software changes

Common secure development techniques:
- **Input validation** to reject malformed or dangerous input
- **Output encoding** to prevent injection in browsers and interpreters
- **Parameterization** to reduce SQL injection risk
- **Error handling** that avoids leaking sensitive details
- **Authentication and authorization checks** at every sensitive action
- **Secure session management** with short-lived tokens and proper invalidation

Example of parameterized SQL:
```python
cursor.execute(
    "SELECT id, role FROM users WHERE username = %s",
    (username,)
)
```

Change control should answer:
- What is changing?
- Why is it changing?
- What is the risk?
- Who approved it?
- How will it be tested?
- How will it be rolled back?

A strong change record includes:
- Scope
- Impact analysis
- Test plan
- Backout plan
- Approval evidence
- Implementation window
- Post-change validation

**Testing and verification**

Testing confirms that controls work as intended and that changes did not introduce unacceptable risk. Security testing should happen throughout the lifecycle, not only at the end.

Common testing types:
- **Unit testing**: verifies small code units
- **Integration testing**: verifies components work together
- **System testing**: validates the full system
- **Regression testing**: ensures changes did not break existing behavior
- **User acceptance testing (UAT)**: confirms business requirements are met
- **Security testing**: focuses on vulnerabilities and control effectiveness

Security-focused testing methods:
- **Static Application Security Testing (SAST)**: analyzes source code without running it
- **Dynamic Application Security Testing (DAST)**: tests a running application from the outside
- **Interactive Application Security Testing (IAST)**: combines runtime instrumentation with testing
- **Software Composition Analysis (SCA)**: identifies vulnerable third-party dependencies
- **Penetration testing**: simulates attacks to validate exploitability
- **Fuzzing**: feeds malformed inputs to find crashes or logic flaws
- **Vulnerability scanning**: identifies known weaknesses in systems and software

Testing should verify:
- Authentication and authorization controls
- Encryption in transit and at rest
- Logging and alerting behavior
- Input validation and error handling
- Segmentation and firewall rules
- Backup and restore capability
- Failover and disaster recovery procedures

Example test commands:
```bash
# Check TLS certificate details
openssl s_client -connect example.com:443 -servername example.com

# Scan a host for open ports
nmap -sV -Pn 10.0.0.10

# Verify HTTP security headers
curl -I https://example.com
```

A practical security test plan often includes:
1. Define the control or requirement to validate.
2. Choose the test method and success criteria.
3. Prepare test data and environment.
4. Execute the test and record results.
5. Remediate findings and retest.
6. Obtain sign-off before release.

> Testing proves the control works in the tested conditions; it does not guarantee the system is secure in all conditions.

**Deployment strategies**

Deployment introduces the solution into production. The main objective is to reduce risk while preserving service availability and user trust.

Common deployment approaches:
- **Big bang**: switch all users at once; fast but risky
- **Phased rollout**: release in stages to reduce blast radius
- **Pilot**: deploy to a small group first
- **Parallel**: run old and new systems together for comparison
- **Blue-green**: maintain two production environments and switch traffic between them
- **Canary**: release to a small percentage of users or hosts first

Deployment considerations:
- Validate backups before cutover
- Confirm monitoring and alerting are active
- Ensure rollback procedures are tested
- Freeze nonessential changes during the release window
- Communicate maintenance windows and user impact
- Verify access controls in production, not just test environments

Example blue-green flow:
```text
Users -> Load Balancer -> Blue (current)
                     \-> Green (new, tested)
Cutover: shift traffic from Blue to Green
Rollback: shift traffic back to Blue
```

Example Kubernetes rollout command:
```bash
kubectl rollout status deployment/webapp
kubectl rollout undo deployment/webapp
```

Deployment should include post-release validation:
- Service health checks
- Authentication tests
- Log review
- Performance checks
- Business transaction verification

**Maintenance and continuous improvement**

Maintenance keeps the system secure after deployment. This is where many organizations lose security posture if they do not monitor, patch, and review continuously.

Maintenance activities include:
- Patch operating systems, applications, and firmware
- Rotate keys, certificates, and secrets
- Review access rights and remove stale accounts
- Monitor logs, alerts, and anomalies
- Reassess risk after major changes
- Test backups and disaster recovery plans
- Update documentation and runbooks
- Retire obsolete systems and data safely

Patch management is especially important because known vulnerabilities are often exploited quickly after disclosure. A good patch process includes:
- Asset inventory
- Vulnerability prioritization
- Testing in a staging environment
- Scheduled deployment
- Verification and reporting
- Exception handling for systems that cannot be patched immediately

Example maintenance commands:
```bash
# Check for pending updates on Debian/Ubuntu
apt list --upgradable

# Review recent logs
journalctl -p warning -S "24 hours ago"

# Check disk usage
df -h
```

Maintenance also includes lifecycle controls for data:
- Retention schedules
- Secure archival
- Legal hold procedures
- Secure deletion or sanitization
- Media disposal and destruction

When systems age out, decommissioning must be deliberate:
- Revoke credentials and certificates
- Remove DNS entries and firewall rules
- Archive required logs and records
- Sanitize storage media
- Update inventories and CMDB records
- Confirm third-party integrations are removed

**Verification, validation, and assurance**

Verification asks whether you built the system right. Validation asks whether you built the right system. CISSP often expects you to distinguish these concepts.

- **Verification** compares implementation to specification
- **Validation** compares the system to business need and intended use

Assurance comes from evidence that controls are effective:
- Policies and standards
- Test results
- Audit logs
- Independent assessments
- Configuration baselines
- Change records
- Incident and remediation history

A mature program uses continuous monitoring:
- Configuration drift detection
- Vulnerability management
- Security information and event management (<abbr title="Security Information and Event Management">SIEM</abbr>)
- Endpoint detection and response (<abbr title="Endpoint Detection and Response">EDR</abbr>)
- Metrics and key risk indicators

**Common pitfalls and exam focus**

CISSP questions often test whether you choose the earliest, most cost-effective, and most risk-reducing action.

Watch for these patterns:
- Fixing problems in production that should have been addressed in requirements
- Relying on a single control instead of layered defenses
- Confusing testing with deployment approval
- Treating security as a one-time project instead of a lifecycle
- Ignoring maintenance, patching, and decommissioning
- Overlooking documentation and change control

High-value exam takeaways:
- Requirements drive design; design drives implementation; implementation must be tested; deployment must be controlled; maintenance must be continuous.
- Security is strongest when built in early, not bolted on later.
- Controls should be measurable, traceable, and aligned to risk.
- Operational discipline matters as much as technical strength.

**Practical checklist**

Use this checklist to structure a secure project:
- Define business, legal, and security requirements
- Classify data and identify trust boundaries
- Select controls using defense in depth
- Build with secure coding and hardened configurations
- Review changes and manage secrets properly
- Test functionality, resilience, and security
- Deploy with rollback and monitoring
- Maintain with patching, logging, access reviews, and decommissioning

**Quick reference examples**

Example secure service startup on Linux:
```bash
sudo systemctl enable nginx
sudo systemctl start nginx
sudo systemctl status nginx
```

Example firewall rule concept:
```bash
sudo ufw allow from 10.0.0.0/24 to any port 443 proto tcp
```

Example log review for authentication failures:
```bash
grep -i "failed password" /var/log/auth.log
```

Example simple deployment checklist:
1. Confirm backup completion.
2. Confirm test results and approvals.
3. Deploy to pilot or canary group.
4. Monitor logs, metrics, and alerts.
5. Expand rollout or roll back based on results.

**Summary**

The security lifecycle is a chain of dependent activities. Requirements define the target, design selects the controls, implementation realizes them, testing proves them, deployment introduces them safely, and maintenance keeps them effective. For CISSP, focus on the relationship between these phases and the principle that security must be planned, built, verified, and sustained throughout the system’s life.
