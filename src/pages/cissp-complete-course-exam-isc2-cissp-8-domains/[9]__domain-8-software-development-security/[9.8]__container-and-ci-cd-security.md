## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Container and CI/CD security overview**

Containers package an application and its dependencies into an isolated runtime unit, while CI/CD (Continuous Integration/Continuous Delivery or Deployment) automates build, test, and release workflows. Security must cover the full path: source code, build system, artifact registry, deployment platform, runtime, and monitoring.

The main risk is that automation increases speed and blast radius. A compromised dependency, runner, registry, or cluster admission path can spread malicious code quickly across environments. Good security therefore focuses on trust boundaries, least privilege, provenance, and verification at every stage.

- **Threat model** the pipeline end to end: developer workstation, source control, build runner, artifact store, registry, deployment controller, and runtime.
- Treat every artifact as untrusted until you verify its origin, integrity, and policy compliance.
- Prefer controls that are enforceable automatically, because manual review does not scale in CI/CD.

**Container fundamentals and security goals**

A container shares the host kernel, so it is not a virtual machine. Isolation comes from kernel features such as namespaces and control groups (cgroups), plus additional controls like seccomp, AppArmor, and SELinux. This means container security is mostly about reducing privilege, limiting kernel attack surface, and preventing container escape or lateral movement.

Security goals include:
- Minimize the image attack surface.
- Prevent privilege escalation inside the container and on the host.
- Restrict network, filesystem, and process capabilities.
- Ensure images are signed, scanned, and traceable.
- Protect secrets and credentials used by builds and deployments.

**Container image security**

A secure image starts with a trusted base image and a minimal package set. Smaller images reduce patching burden and lower the chance of including shells, package managers, or debugging tools that attackers can abuse.

Common practices:
- Use official or vendor-maintained base images when possible.
- Pin image references by digest, not only by tag.
- Remove build tools from runtime images.
- Run as a non-root user.
- Set a read-only root filesystem when the application allows it.
- Drop Linux capabilities that are not required.

Example Dockerfile using a multi-stage build:

```dockerfile
# Build stage
FROM golang:1.22-alpine AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /out/app ./cmd/app

# Runtime stage
FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /out/app /app
USER nonroot:nonroot
ENTRYPOINT ["/app"]
```

Why this pattern matters:
- The build stage contains the Go toolchain, but the runtime stage does not.
- Distroless images reduce the available attack surface.
- `CGO_ENABLED=0` helps produce a static binary for simpler runtime dependencies.

**Image trust, signing, and verification**

Image signing helps you verify that an image came from a trusted build process and was not altered after release. Common tools include Cosign and Notary-based workflows. In practice, you should verify signatures in CI/CD and at deployment admission time.

Typical verification flow:
1. Build the image.
2. Scan it for vulnerabilities.
3. Sign the image digest.
4. Verify the signature before deployment.
5. Enforce policy in the cluster or deployment platform.

Example Cosign usage:

```bash
cosign sign --key cosign.key registry.example.com/payments/api@sha256:abcd1234...
cosign verify --key cosign.pub registry.example.com/payments/api@sha256:abcd1234...
```

For attestations, use a structured predicate that matches your policy requirements, such as provenance or SBOM (Software Bill of Materials) data. Attestations are only useful if your policy engine knows how to validate them.

> Do not rely on tags alone for trust. Tags are mutable in many registries, so `:latest` or `:1.4.2` can point to different content unless the registry enforces immutability.

**Registry security controls**

A container registry is a high-value target because it stores deployable artifacts. Security controls should cover authentication, authorization, immutability, scanning, and auditing.

Key registry controls:
- **Authentication and authorization**: Use strong identity, role-based access control, and short-lived tokens where possible.
- **Immutability**: Prevent overwriting released tags in production repositories.
- **Vulnerability scanning**: Scan images on push and on a schedule to catch newly disclosed issues.
- **Access auditing**: Log pulls, pushes, deletes, and permission changes.
- **Retention and cleanup**: Remove stale artifacts without breaking traceability.
- **Network restriction**: Limit registry access to trusted networks or private endpoints.

Example policy ideas:
- Allow developers to push only to `dev/*` repositories.
- Allow CI service accounts to promote only signed images.
- Deny deletion of release tags in production repositories.
- Alert on pulls from unusual geographies or outside business hours.

A practical promotion pattern is to copy the exact digest into the target repository using a registry-aware tool, rather than assuming a local tag rename is enough. For example:

```bash
# Example: copy by digest with a registry-aware tool
skopeo copy \
  docker://registry.example.com/dev/payments/api@sha256:abcd1234... \
  docker://registry.example.com/prod/payments/api:1.4.2
```

This preserves the artifact identity more reliably than a local `docker tag` alone.

**CI/CD pipeline threats and defenses**

CI/CD systems are attractive targets because they often hold source code, signing keys, cloud credentials, and deployment permissions. Attackers may target the pipeline to inject malicious code into trusted artifacts.

Common attack scenarios:
- **Build poisoning**: Malicious code changes the build output or build scripts.
- **Dependency confusion**: A build pulls a public package instead of an internal one.
- **Malicious pull requests**: Untrusted contributors trigger workflows that expose secrets.
- **Runner compromise**: A compromised build agent steals credentials or modifies artifacts.
- **Fork-based secret exfiltration**: A forked repository triggers workflows with access to secrets.
- **Supply-chain tampering**: A dependency, base image, or build plugin is replaced upstream.

Defenses:
- Separate trusted and untrusted workflows.
- Do not expose secrets to untrusted pull requests.
- Pin dependency versions and verify checksums.
- Use ephemeral runners and clean workspaces.
- Restrict outbound network access from build jobs when possible.
- Require code review and branch protection for protected branches.
- Sign build outputs and verify them before release.

Example GitHub Actions hardening pattern:

```yaml
name: build
on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: make test
```

Security notes:
- Keep `permissions` minimal.
- Avoid giving pull request jobs write access unless necessary.
- Use separate jobs for untrusted validation and trusted release signing.

**Secrets management for containers and pipelines**

Secrets are often mishandled because teams place them in environment variables, image layers, or plain text configuration files. Containers make this worse if secrets are baked into images or copied into logs.

Good patterns:
- Store secrets in a dedicated secret manager such as HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, or Google Secret Manager.
- Inject secrets at runtime, not at build time.
- Rotate secrets regularly and revoke them after incidents.
- Use short-lived credentials and workload identity where possible.
- Avoid committing secrets to Git, even in private repositories.

Kubernetes Secrets are useful, but they are not a full secret-management solution:
- They are base64-encoded, not encrypted by default.
- Access depends on RBAC (Role-Based Access Control) and cluster security.
- They can be exposed through logs, misconfigured mounts, or overly broad permissions.

Example Kubernetes Secret mount:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:
  username: appuser
  password: S3cureP@ssw0rd
---
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
    - name: app
      image: registry.example.com/payments/api:1.4.2
      env:
        - name: DB_USERNAME
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: username
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
```

Better operational patterns:
- Use external secret operators or CSI (Container Storage Interface) secret drivers to fetch secrets at runtime.
- Mount secrets as files when the application supports file-based credentials.
- Rotate secrets and restart workloads safely when needed.
- Audit secret access separately from application logs.

**Kubernetes control-plane security**

Kubernetes security is not only about pods. The control plane is a critical trust anchor, and compromise there affects the entire cluster.

Important areas:
- **API server hardening**: Restrict access, enable strong authentication, and use authorization policies such as RBAC.
- **etcd protection**: Encrypt data at rest, restrict network access, and protect backup files.
- **kubelet security**: Disable anonymous access, secure the read-only port, and restrict direct node access.
- **Admission control**: Enforce policy before workloads are admitted.
- **Audit logging**: Record API activity for detection and forensics.

Practical controls:
- Use TLS for all control-plane communication.
- Limit API server exposure to trusted networks or private endpoints.
- Enable encryption at rest for Kubernetes secrets in etcd.
- Rotate certificates and credentials regularly.
- Restrict cluster-admin privileges to a small set of operators.

Example API server hardening concepts:
- Disable anonymous authentication.
- Use `NodeRestriction` and RBAC.
- Enable audit logging with retention.
- Use admission controllers to block risky workloads.

**Pod Security Admission and workload policy**

Pod Security Admission (PSA) enforces Pod Security Standards through namespace labels. It is not a separate cluster-wide policy object; instead, the namespace label determines whether pods in that namespace are evaluated under `privileged`, `baseline`, or `restricted` controls.

Example namespace labels:

```bash
kubectl label namespace payments \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/audit=restricted \
  pod-security.kubernetes.io/warn=restricted
```

Why this matters:
- `enforce` blocks noncompliant pods.
- `audit` records violations.
- `warn` informs users without blocking.

Typical restricted settings include:
- Run as non-root.
- Drop privilege escalation.
- Disallow host namespaces.
- Restrict hostPath volumes.
- Require seccomp profiles.

**Admission policy examples**

Admission policies let you enforce security rules before a workload starts. Use them to block privileged containers, enforce image signing, or require approved registries.

Example Kyverno policy to require images from an approved registry:

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-approved-registry
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-image-registry
      match:
        resources:
          kinds:
            - Pod
      validate:
        message: "Images must come from registry.example.com"
        pattern:
          spec:
            containers:
              - image: "registry.example.com/*"
```

Example Gatekeeper-style constraint idea:
- Deny pods that run as root.
- Deny hostPath mounts unless explicitly approved.
- Deny images without signatures or attestations.

**Runtime isolation and kernel hardening**

Runtime security reduces the impact of a compromised container. Since containers share the host kernel, kernel-level controls are essential.

Key controls:
- **seccomp**: Filter system calls to reduce kernel attack surface.
- **AppArmor**: Apply Linux Mandatory Access Control profiles.
- **SELinux**: Enforce type-based access control on supported systems.
- **Capabilities**: Drop unnecessary Linux capabilities.
- **Read-only root filesystem**: Prevent unexpected writes.
- **User namespaces**: Map container users to non-root host users where supported.

Example Kubernetes pod with seccomp and capability drops:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: hardened-app
spec:
  securityContext:
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: registry.example.com/payments/api:1.4.2
      securityContext:
        runAsNonRoot: true
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop:
            - ALL
```

Example AppArmor annotation:

```yaml
metadata:
  annotations:
    container.apparmor.security.beta.kubernetes.io/app: runtime/default
```

Sandboxed runtime options:
- **gVisor**: Adds a user-space kernel boundary for stronger isolation than standard containers.
- **Kata Containers**: Runs containers inside lightweight virtual machines for stronger workload isolation.

Use these when:
- You run untrusted code.
- You host multi-tenant workloads.
- You need stronger isolation than standard container boundaries provide.

**Network security for containers**

Container networking should follow least privilege just like host networking. Flat east-west access inside a cluster can make lateral movement easy after one compromise.

Controls:
- Use Kubernetes NetworkPolicy to restrict pod-to-pod traffic.
- Segment namespaces by application or trust level.
- Restrict egress to only required services.
- Use service mesh mTLS (mutual TLS) where appropriate.
- Avoid exposing unnecessary NodePorts or hostNetwork pods.

Example NetworkPolicy allowing traffic only from a labeled namespace:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-api
  namespace: data
spec:
  podSelector:
    matchLabels:
      app: database
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              name: api
          podSelector:
            matchLabels:
              app: payments
      ports:
        - protocol: TCP
          port: 5432
```

Important detail:
- The `namespaceSelector` only works if the source namespace is labeled `name=api`.
- If you omit namespace labels, the policy may not match as intended.

**CI/CD artifact integrity and SBOM management**

An SBOM lists the components in a software artifact. It helps you answer what is inside the image, which versions are present, and whether known vulnerabilities affect the release.

Why SBOMs matter:
- They support vulnerability management.
- They help with incident response and impact analysis.
- They improve supplier and dependency transparency.

Common practices:
- Generate an SBOM during the build.
- Store it with the artifact or as an attestation.
- Scan dependencies and images continuously.
- Rebuild regularly to pick up patched base images.

Example with Syft and Grype:

```bash
syft registry.example.com/payments/api:1.4.2 -o spdx-json > sbom.json
grype sbom:sbom.json
```

Supply-chain maturity often maps to SLSA (Supply-chain Levels for Software Artifacts). Higher SLSA levels generally mean stronger provenance, hermetic builds, and better resistance to tampering. For exam purposes, remember the direction of improvement:
- More automation.
- Stronger provenance.
- Reduced manual intervention.
- Better integrity guarantees.

**Trusted base images and dependency hygiene**

Base image selection is a security decision, not just a convenience choice. Prefer images from trusted sources with clear maintenance and patching practices.

Good selection criteria:
- Official vendor or project source.
- Active security patching.
- Minimal package footprint.
- Clear digest pinning and release notes.
- Compatibility with your runtime needs.

Dependency hygiene:
- Pin versions in lockfiles.
- Verify checksums or signatures where available.
- Use private mirrors for approved packages.
- Review transitive dependencies.
- Remove unused dependencies from builds.

Example package pinning concepts:
- `go.sum` for Go modules.
- `package-lock.json` or `npm ci` for Node.js.
- `requirements.txt` with hashes for Python when feasible.

**Container orchestration isolation and multi-tenancy**

In shared clusters, isolation must be enforced at multiple layers:
- Namespace separation.
- RBAC boundaries.
- Network policies.
- Pod security controls.
- Node pool separation for sensitive workloads.
- Dedicated clusters for high-risk or regulated workloads.

Use node selectors, taints, and tolerations to keep sensitive workloads away from general-purpose nodes.

Example:

```yaml
spec:
  nodeSelector:
    workload-tier: sensitive
  tolerations:
    - key: "dedicated"
      operator: "Equal"
      value: "sensitive"
      effect: "NoSchedule"
```

This is useful when you need stronger separation for regulated data, production workloads, or untrusted tenants.

**Incident response and forensics in container environments**

Containers are ephemeral, so evidence can disappear quickly. Incident response plans should preserve logs, metadata, and runtime state before the pod or node is recycled.

Preserve:
- Container logs from the runtime and application.
- Kubernetes audit logs.
- Registry access logs.
- CI/CD job logs and runner metadata.
- Image digests, signatures, and SBOMs.
- Node-level evidence if compromise is suspected.

Practical response actions:
1. Capture the running pod spec and node placement.
2. Save logs to a central system before deletion.
3. Snapshot the container filesystem or node disk if policy allows.
4. Record image digest and deployment revision.
5. Revoke credentials and rotate secrets.
6. Rebuild from trusted sources rather than patching a suspected image in place.

Useful commands:
```bash
kubectl get pod app-7d9f8c7d9f-abcde -o yaml > pod.yaml
kubectl logs app-7d9f8c7d9f-abcde --all-containers=true > app.log
kubectl describe pod app-7d9f8c7d9f-abcde > describe.txt
```

**Practical hardening checklist**

Use this as a quick review for exam prep and real deployments:
- Build minimal images with multi-stage builds.
- Run containers as non-root.
- Drop all unnecessary capabilities.
- Use seccomp, AppArmor, or SELinux.
- Pin image digests and verify signatures.
- Scan images and dependencies continuously.
- Protect registry access with strong auth, immutability, and auditing.
- Keep secrets out of images and source control.
- Restrict CI/CD permissions and isolate untrusted jobs.
- Harden the Kubernetes API server, etcd, and kubelet.
- Enforce admission policies and namespace-based Pod Security controls.
- Segment networks and limit egress.
- Preserve logs and artifacts for incident response.

**Exam focus reminders**

For CISSP-style questions, look for the control that best reduces risk at the earliest point in the pipeline:
- Prevent secret exposure by not injecting secrets into untrusted jobs.
- Prevent malicious deployment by verifying signatures and enforcing admission policy.
- Prevent lateral movement by using network policy and least privilege.
- Prevent host compromise by dropping capabilities and using runtime isolation.
- Prevent supply-chain tampering by pinning digests and validating provenance.
