**Code review and dependency management overview**

Secure code review is a verification activity that helps you find defects, insecure patterns, and policy violations before software reaches production. Dependency management reduces risk from third-party and open-source components by controlling what you use, where it comes from, and how you update it.

In CISSP terms, this topic sits at the intersection of secure software development, supply-chain security, and change control. You are not just looking for bugs; you are validating that the codebase and its dependencies are trustworthy, maintainable, and aligned with security requirements.

- **Code review** focuses on source code, configuration, and build logic.
- **Dependency management** focuses on external libraries, packages, containers, and transitive components.
- **Supply-chain risk** includes malicious packages, compromised maintainers, typosquatting, and outdated vulnerable versions.

**Why these controls matter**

Modern applications rarely consist only of first-party code. They rely on package managers, container images, frameworks, and build tools that can introduce vulnerabilities even when your own code is clean. A strong review and dependency process helps you catch:

- Injection flaws and unsafe input handling
- Hardcoded secrets and weak cryptography
- Insecure defaults and dangerous permissions
- Vulnerable or abandoned libraries
- License and provenance issues
- Build pipeline tampering

**Secure code review methods**

Code review can be manual, automated, or hybrid. The best programs combine human judgment with tooling because each approach finds different classes of issues.

- **Manual review** is best for logic flaws, authorization mistakes, and business-rule bypasses.
- **Static Application Security Testing** (**SAST**) scans source code or bytecode without executing it.
- **Peer review** improves quality by having another developer inspect changes before merge.
- **Threat modeling** helps reviewers focus on high-risk flows, trust boundaries, and abuse cases.

A practical review process usually starts with high-risk areas:

- Authentication and session handling
- Authorization checks and role enforcement
- Input validation and output encoding
- File upload and path handling
- Cryptographic operations
- Error handling and logging
- Deserialization and command execution
- Secrets management and configuration

**What to look for during review**

Reviewers should look for patterns that create security defects even when the code appears functional.

- **Injection risks**
  - SQL injection from string concatenation
  - Command injection from shell execution
  - Cross-site scripting from unescaped output
- **Access control failures**
  - Missing server-side authorization checks
  - Insecure direct object references
  - Privilege escalation through parameter tampering
- **Sensitive data exposure**
  - Secrets in source code, logs, or test fixtures
  - Weak token storage or insecure session handling
  - Unencrypted sensitive data in transit or at rest
- **Cryptographic misuse**
  - Weak algorithms such as MD5 or SHA-1 for security purposes
  - Hardcoded keys or reused initialization vectors
  - Custom crypto instead of vetted libraries
- **Reliability and safety issues**
  - Null dereferences and exception leaks
  - Race conditions and unsafe concurrency
  - Resource exhaustion from unbounded input

> A code review is only effective if reviewers understand the intended security behavior. Without requirements, reviewers may confirm that code “works” while missing that it violates policy.

**Review techniques and tooling**

Different review techniques serve different goals. Use them together rather than treating one as sufficient.

| Technique | Best for | Limitation |
|---|---|---|
| Manual review | Logic flaws, authorization, business rules | Slow and inconsistent |
| SAST | Pattern-based insecure code detection | False positives and limited runtime context |
| Dynamic Application Security Testing (**DAST**) | Runtime behavior and exposed endpoints | Misses unreachable code and internal logic |
| Interactive Application Security Testing (**IAST**) | In-context findings during execution | Requires instrumentation |
| Software Composition Analysis (**SCA**) | Third-party dependency vulnerabilities | Does not find first-party code flaws |

A mature workflow often includes:

1. Developer writes code and runs local checks.
2. Peer review validates logic and security controls.
3. Automated scanners run in the CI/CD pipeline.
4. Security findings are triaged and remediated before release.
5. Exceptions are documented and approved through change control.

**Example secure review checklist**

Use a checklist to make reviews repeatable and less dependent on memory.

- Are all inputs validated on the server side?
- Are authorization checks enforced for every sensitive action?
- Are secrets absent from code, logs, and configuration files?
- Are cryptographic functions using approved libraries and modes?
- Are errors handled without leaking stack traces or internal details?
- Are dependencies pinned and reviewed for known vulnerabilities?
- Are build and deployment scripts protected from unauthorized changes?

**Dependency management fundamentals**

Dependency management is the process of selecting, pinning, updating, verifying, and removing external components. It is a security control because dependencies can introduce vulnerabilities, malicious code, or operational instability.

Common dependency types include:

- Application libraries and frameworks
- Package manager modules
- Container base images
- Operating system packages
- Build tools and plugins
- Infrastructure-as-Code modules

A dependency can be:

- **Direct**: explicitly declared by your project
- **Transitive**: pulled in by another dependency
- **Optional**: installed only in certain environments or features

Transitive dependencies are especially important because they can create hidden risk. You may approve one library and unknowingly inherit dozens of others.

**Core dependency controls**

Good dependency management includes both technical and governance controls.

- **Version pinning**
  - Locks a dependency to a known version or range
  - Reduces surprise changes from upstream releases
- **Allowlisting**
  - Limits packages to approved sources or registries
  - Helps prevent typosquatting and unauthorized imports
- **Integrity verification**
  - Confirms the package has not been altered in transit
  - Uses hashes, signatures, or trusted registries
- **Vulnerability scanning**
  - Checks dependencies against known CVEs
  - Supports prioritization by severity and exploitability
- **Patch management**
  - Updates vulnerable components in a controlled way
  - Balances security with regression risk
- **Removal of unused dependencies**
  - Shrinks attack surface
  - Reduces maintenance burden

**Practical examples by ecosystem**

Different ecosystems use different tools, but the security principles are the same.

```bash
# npm: install from lockfile and audit for known issues
npm ci
npm audit

# Python: inspect installed packages and vulnerabilities
pip freeze
pip-audit

# Java: inspect dependency tree with Maven
mvn dependency:tree

# Container image scanning example
trivy image nginx:1.27
```

For package managers, lockfiles are important because they preserve exact versions and reduce build drift.

```json
{
  "name": "example-app",
  "version": "1.0.0",
  "dependencies": {
    "express": "4.19.2"
  }
}
```

A lockfile or pinned manifest helps ensure that the build you test is the build you deploy. Without it, a minor upstream release can change behavior or introduce a vulnerability unexpectedly.

**Secure package sourcing**

You should trust packages only from controlled sources.

- Use official registries or internal mirrors when possible.
- Prefer signed packages or verified publishers when supported.
- Restrict who can publish to internal repositories.
- Block direct internet access from build systems when policy requires it.
- Review package metadata, maintainer history, and release cadence.

Common supply-chain threats include:

- **Typosquatting**: malicious package names that resemble legitimate ones
- **Dependency confusion**: internal package names resolved from public registries
- **Compromised maintainer accounts**: attackers publish malicious updates
- **Abandoned packages**: unpatched vulnerabilities remain indefinitely
- **Malicious updates**: legitimate packages later receive harmful code

**Dependency update strategy**

Updating dependencies is a security activity, not just a maintenance task. The goal is to reduce exposure while avoiding unnecessary breakage.

A practical update strategy includes:

1. Classify dependencies by criticality.
2. Monitor advisories and vulnerability feeds.
3. Apply security patches quickly for exposed components.
4. Test updates in a staging environment.
5. Promote changes through normal change control.
6. Remove or replace unmaintained libraries.

Use semantic versioning carefully. A version range such as `^1.2.3` may allow compatible updates, but it can also introduce behavior changes. For high-risk systems, pin exact versions and update deliberately.

**Example dependency policy**

```yaml
# Example policy concepts for a CI pipeline
allowed_sources:
  - registry.npmjs.org
  - pypi.org
  - repo.internal.example
require_lockfile: true
fail_on_high_severity: true
block_unapproved_licenses: true
```

This kind of policy helps enforce consistency across teams and prevents ad hoc dependency choices.

**Software Bill of Materials**

A **Software Bill of Materials** (**SBOM**) is an inventory of software components used in an application. It improves visibility into what you ship and helps you respond faster when a vulnerability is disclosed.

An SBOM can help you:

- Identify affected products during incident response
- Track transitive dependencies
- Support compliance and procurement reviews
- Compare deployed components against known advisories

Common SBOM formats include SPDX and CycloneDX. The exact format matters less than the ability to generate, store, and query the inventory consistently.

**Build pipeline and repository protections**

Dependency management is only as strong as the pipeline that enforces it. If attackers can alter source, build scripts, or package sources, they can bypass review.

Protect the pipeline with:

- Branch protection and mandatory pull requests
- Signed commits or signed tags where supported
- Least privilege for build agents and service accounts
- Segregation of duties between developers and approvers
- Immutable build artifacts
- Restricted access to artifact repositories
- Logging and alerting for package publication and build changes

A secure repository workflow often includes code owners, required approvals, and automated checks before merge.

```text
Developer branch -> Pull request -> Peer review -> SAST/SCA checks -> Merge -> Build -> Artifact signing -> Deployment
```

**Common review and dependency anti-patterns**

These patterns often appear in insecure environments and should trigger deeper inspection.

- **Skipping review for “small” changes**
  - Small changes can still introduce backdoors or weaken controls
- **Blindly accepting scanner output**
  - Findings need triage, not just volume
- **Using latest by default**
  - Unpinned dependencies can break builds or pull in risky updates
- **Vendoring without review**
  - Copying code locally does not remove the need for provenance checks
- **Ignoring transitive dependencies**
  - Vulnerabilities often hide several layers deep
- **Storing secrets in package scripts**
  - Build scripts and CI variables are common leakage points

**Real-world examples**

A web application uses a database library that concatenates user input into SQL queries. Manual review catches the flaw because the code path is business-logic specific, while SAST flags the string-building pattern.

A containerized service depends on an old logging library with a known remote code execution vulnerability. SCA identifies the vulnerable version, and the team updates the dependency and rebuilds the image.

A build pipeline pulls a package from a public registry because an internal package name was not protected. Dependency confusion occurs, so the organization enforces internal namespace controls and registry allowlisting.

**How to prioritize findings**

Not every issue has the same urgency. Prioritize based on:

- Exposure to untrusted input or external users
- Privilege level of the affected component
- Exploitability and known public exploits
- Data sensitivity
- Internet-facing deployment
- Ease of remediation and regression risk

A low-severity issue in a public authentication service may deserve faster attention than a higher-severity issue in an isolated internal tool.

**Exam-focused distinctions**

For CISSP, distinguish between related concepts:

- **Code review** is about inspecting code for defects and policy violations.
- **Vulnerability scanning** is about identifying known weaknesses, often with tools.
- **Patch management** is about applying updates to fix vulnerabilities.
- **Configuration management** is about controlling system settings and baselines.
- **Change management** is about approving and tracking modifications.
- **Supply-chain security** is about trust in external code, packages, and build inputs.

You should also remember that secure development is a lifecycle activity. Review and dependency controls work best when they are built into design, development, testing, deployment, and operations.

**Practical checklist for secure teams**

- Define secure coding standards and review criteria.
- Require peer review for all production changes.
- Run SAST and SCA in CI/CD.
- Pin versions and maintain lockfiles.
- Scan container images and base layers.
- Generate and retain SBOMs.
- Restrict package sources and publishing rights.
- Track vulnerabilities to remediation completion.
- Reassess dependencies regularly for abandonment or risk.

**Key takeaway**

Secure code review finds flaws in first-party code, while dependency management reduces risk from third-party components and the software supply chain. Together, they help you prevent vulnerabilities, improve trust in releases, and maintain control over what enters production.
