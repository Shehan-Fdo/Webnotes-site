## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**API security overview**

An <abbr title="Application Programming Interface">API</abbr> exposes application functions and data to other software, mobile apps, partners, and browser clients. API security focuses on protecting the interface itself, not just the backend service. That means you secure transport, identity, authorization, input handling, error handling, logging, rate limits, and the operational lifecycle of endpoints.

APIs are attractive targets because they often bypass user-facing controls and directly expose business logic. A secure API design assumes hostile clients, untrusted networks, and active probing for hidden endpoints, weak authorization, and data overexposure.

- **Attack surface** grows when APIs are versioned poorly, documented incompletely, or left running after decommissioning.
- **Business logic** flaws often matter more than classic injection issues because APIs expose workflows and state changes.
- **Least privilege** applies to both human users and service-to-service callers.

**Common API styles and why they matter**

Different API styles change the security model, payload format, and attack surface. You should recognize the tradeoffs rather than memorize labels only.

| Style | Typical format | Security considerations |
|---|---|---|
| REST | JSON over HTTP | Strong reliance on HTTP verbs, status codes, authz per resource, and content-type validation |
| SOAP | XML over HTTP or other transports | XML parser hardening, WS-Security, schema validation, and XML signature wrapping risks |
| GraphQL | JSON over HTTP | Query depth/complexity limits, field-level authorization, introspection control, and overfetching/underfetching risks |
| gRPC | Protobuf over HTTP/2 | Strong schema enforcement, transport security, method-level authorization, and metadata handling |
| WebSocket APIs | Persistent bidirectional channel | Session binding, origin checks, message authorization, and long-lived connection monitoring |

REST is common because it maps well to HTTP and is easy to consume. SOAP still appears in enterprise and legacy environments, especially where message-level security or formal contracts matter. GraphQL can reduce round trips but can also expose too much data if field-level authorization is weak. gRPC is efficient for internal service-to-service communication, but it still needs authentication, authorization, and transport protection.

**Transport security**

API traffic should use <abbr title="Transport Layer Security">TLS</abbr> to protect confidentiality and integrity in transit. For modern deployments, prefer TLS 1.2 or TLS 1.3 and disable obsolete protocols such as SSL, TLS 1.0, and TLS 1.1. Cipher selection should favor forward secrecy and authenticated encryption, such as AES-GCM or ChaCha20-Poly1305, depending on platform support.

Certificate validation is not optional. Clients should verify:
- The certificate chain to a trusted root
- The hostname or subject alternative name matches the API endpoint
- The certificate is not expired or revoked, where revocation checking is supported
- The trust store is controlled and not silently overridden

> Do not disable certificate verification in test code that later reaches production. That pattern creates a common man-in-the-middle exposure.

Use <abbr title="Mutual TLS">mTLS</abbr> when both sides must prove identity at the transport layer, especially for internal service-to-service calls, partner integrations, and high-trust environments where client certificates can be managed securely. Standard TLS is usually enough for public clients when you authenticate at the application layer with tokens or keys. mTLS adds operational overhead, so use it when the stronger client identity assurance is worth the certificate lifecycle cost.

A practical NGINX example for TLS hardening:

```nginx
server {
    listen 443 ssl http2;
    server_name api.example.com;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers off;
    ssl_ciphers TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256:TLS_AES_128_GCM_SHA256;

    ssl_certificate     /etc/ssl/certs/api.example.com.crt;
    ssl_certificate_key /etc/ssl/private/api.example.com.key;

    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
}
```

For mTLS in an API gateway, you typically require a client certificate and map certificate identity to a service account or partner identity:

```nginx
server {
    listen 443 ssl;
    ssl_client_certificate /etc/nginx/ca/client-ca.pem;
    ssl_verify_client on;

    location /internal/ {
        proxy_pass http://backend;
        proxy_set_header X-Client-Cert-Verified $ssl_client_verify;
    }
}
```

**Authentication mechanisms**

API authentication answers the question: who is calling? Authentication is distinct from authorization, which answers what the caller may do.

Common mechanisms include:
- **API keys** for simple identification and coarse access control
- **Bearer tokens** such as OAuth 2.0 access tokens for delegated access
- **JWTs** (<abbr title="JSON Web Tokens">JWT</abbr>) for self-contained claims
- **HMAC-signed requests** for integrity and caller authenticity
- **mTLS client certificates** for strong machine identity
- **Session cookies** for browser-based clients, usually with CSRF protections

API keys are often used for service identification, metering, and low-risk integrations. They are not inherently secret-proof or user-aware. Treat them like credentials:
- Store them in a vault or secret manager
- Rotate them on a schedule and on compromise
- Issue them through a controlled process with separation of duties
- Scope them to the minimum required environment and permissions

A common lifecycle pattern is:
1. Generate a unique key per application, environment, or partner.
2. Store the key in a secrets manager, not in source code.
3. Deliver it to the runtime through environment injection or secret mounts.
4. Rotate the key before expiry or after suspected exposure.
5. Revoke the old key and verify dependent systems have switched.

JWTs are widely used because they can be validated without a database lookup if the signature is trusted. A JWT has three parts separated by dots:
- Header: algorithm and token type
- Payload: claims such as subject, issuer, audience, and expiration
- Signature: integrity protection over header and payload

Validation should check:
- Signature using the correct algorithm and key
- `iss` (issuer)
- `aud` (audience)
- `exp` (expiration)
- `nbf` (not before), if present
- `iat` (issued at), where relevant
- Token type and intended use

Common JWT mistakes include:
- Accepting `alg=none` or allowing algorithm confusion
- Failing to validate audience
- Using long-lived tokens without revocation strategy
- Putting sensitive data in the payload because JWTs are only encoded, not encrypted
- Logging full tokens or returning them in URLs

> Never place bearer tokens or API keys in query strings. URLs are commonly stored in browser history, proxy logs, analytics, referrers, and server logs.

HMAC-signed requests are common for partner APIs and internal services. The client signs a canonical representation of the request using a shared secret. The server recomputes the signature and compares it in constant time. Include a timestamp and nonce to reduce replay risk.

Example pattern:

```http
POST /v1/payments HTTP/1.1
Host: api.example.com
Date: Tue, 26 Apr 2026 10:00:00 GMT
Content-Type: application/json
X-Api-Key: abc123
X-Nonce: 8f2c1d
X-Signature: base64(hmac-sha256(secret, canonical_request))
```

The server should reject:
- Stale timestamps outside an allowed window
- Reused nonces
- Signatures that do not match the canonical request exactly
- Requests where headers or body were altered in transit

**Authorization and access control**

Authorization is where many API breaches occur. You must enforce access control at the resource, method, and sometimes field level. Do not assume that a valid token means the caller may access every object.

Common models include:
- **RBAC** (<abbr title="Role-Based Access Control">RBAC</abbr>) for role-driven permissions
- **ABAC** (<abbr title="Attribute-Based Access Control">ABAC</abbr>) for policy decisions based on user, device, location, time, and resource attributes
- **Scopes** for delegated OAuth permissions
- **Object-level authorization** for per-record access
- **Field-level authorization** for sensitive attributes such as SSNs or payment details

A secure API checks authorization on every request, not just at login. For example, a user with access to `/accounts/123` should not automatically gain access to `/accounts/124`. This is a classic <abbr title="Insecure Direct Object Reference">IDOR</abbr> risk.

Practical policy examples:
- A customer may read only their own profile
- A support agent may view masked payment data but not full card numbers
- A partner integration may create orders but not refund them
- An internal service may call only the methods it needs

Example gateway policy concept:

```yaml
routes:
  - path: /v1/admin/*
    allow:
      - role: admin
    deny:
      - role: partner
  - path: /v1/customers/*
    allow:
      - scope: customer.read
```

**Input validation and content handling**

APIs must validate structure, type, length, format, and semantic meaning. Validation should happen as early as possible, ideally at the gateway and again in the application. Schema validation reduces parser ambiguity and helps prevent injection, deserialization issues, and unexpected business logic paths.

Key controls:
- Enforce `Content-Type` and reject unexpected media types
- Validate JSON against a schema
- Limit request size and nesting depth
- Reject unknown fields when strict contracts are required
- Normalize encodings before validation
- Sanitize data before passing it to downstream systems

Content negotiation can create risk when the server accepts multiple formats without strict controls. For example, accepting both JSON and XML may expand parser attack surface. If XML is required, disable external entity resolution and harden the parser against XML External Entity (<abbr title="XML External Entity">XXE</abbr>) attacks.

Example JSON schema validation concept:

```json
{
  "type": "object",
  "required": ["customerId", "amount"],
  "properties": {
    "customerId": { "type": "string", "minLength": 1 },
    "amount": { "type": "number", "minimum": 0 }
  },
  "additionalProperties": false
}
```

For XML, disable dangerous features in the parser:

```java
factory.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true);
factory.setFeature("http://xml.org/sax/features/external-general-entities", false);
factory.setFeature("http://xml.org/sax/features/external-parameter-entities", false);
```

**Browser-facing API security and CORS**

When a browser calls an API on another origin, <abbr title="Cross-Origin Resource Sharing">CORS</abbr> controls which origins may access the response. CORS is not an authentication mechanism. It is a browser policy that helps prevent unauthorized cross-origin reads.

Important points:
- Use a strict allowlist of trusted origins
- Avoid `Access-Control-Allow-Origin: *` for credentialed requests
- Do not reflect arbitrary origins
- Set `Access-Control-Allow-Credentials: true` only when needed
- Limit allowed methods and headers
- Keep preflight responses narrow and accurate

Example safe-ish CORS pattern for a known web app:

```http
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true
Access-Control-Allow-Methods: GET, POST
Access-Control-Allow-Headers: Content-Type, Authorization
Vary: Origin
```

For browser clients using cookies, protect against cross-site request forgery (<abbr title="Cross-Site Request Forgery">CSRF</abbr>) with anti-CSRF tokens, same-site cookies, and origin checks. If you use bearer tokens in browser storage, understand the tradeoff: local storage is vulnerable to script access if cross-site scripting occurs.

**Error handling and response hardening**

API errors should be useful to legitimate clients but not reveal internal implementation details. Return standardized status codes and concise messages. Avoid stack traces, SQL errors, file paths, framework names, and internal hostnames.

Good practices:
- Use consistent error envelopes
- Return `4xx` for client errors and `5xx` for server errors
- Log detailed diagnostics internally, not in the response
- Avoid exposing whether a username exists when that would aid enumeration
- Suppress sensitive headers such as backend version banners

Example error response:

```json
{
  "error": {
    "code": "ACCESS_DENIED",
    "message": "You are not authorized to access this resource.",
    "requestId": "7f3a2c1b"
  }
}
```

Response hardening also includes:
- Removing unnecessary headers like `Server` and `X-Powered-By`
- Setting `Cache-Control: no-store` for sensitive responses
- Using `Content-Security-Policy` for browser-delivered API documentation or portals
- Avoiding verbose debug modes in production

**Rate limiting, quotas, and abuse prevention**

APIs need controls that limit brute force, scraping, credential stuffing, and denial-of-service attempts. Rate limiting controls request frequency, while quotas cap usage over a longer period. Both should be tied to identity, IP, token, tenant, or a combination of factors.

Common patterns:
- Per-IP limits for anonymous traffic
- Per-token limits for authenticated traffic
- Per-tenant quotas for partner APIs
- Burst plus sustained limits to absorb short spikes without enabling abuse

Example gateway configuration concept:

```yaml
rateLimit:
  key: "$jwt.sub"
  requestsPerMinute: 120
  burst: 20
quota:
  key: "$apiKeyId"
  requestsPerDay: 100000
```

A WAF (Web Application Firewall) or API gateway can enforce these controls before requests reach the application. This is useful for reducing load and blocking obvious abuse, but it does not replace application-layer authorization.

Example with a reverse proxy using NGINX rate limiting:

```nginx
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

server {
    location /v1/ {
        limit_req zone=api_limit burst=20 nodelay;
        proxy_pass http://backend;
    }
}
```

**Discovery, inventory, and attack surface reduction**

Attackers often find APIs through documentation, mobile apps, JavaScript bundles, logs, and forgotten test environments. Reducing discovery opportunities lowers risk.

Controls include:
- Maintain an API inventory with owners, environments, and data classifications
- Remove or protect debug, admin, and test endpoints
- Require authentication for documentation portals when appropriate
- Disable directory listing and verbose route enumeration
- Use consistent, non-revealing error messages for unknown routes
- Verify decommissioned endpoints are actually removed from gateways, DNS, and backend services

Shadow APIs are endpoints that exist outside the approved inventory. You can detect them by:
- Comparing gateway logs with source control and API catalogs
- Scanning DNS, load balancers, and cloud load balancer listeners
- Reviewing mobile app traffic and reverse-engineered clients
- Monitoring for unexpected paths, methods, and user agents
- Running periodic asset discovery and configuration audits

Decommission verification should confirm:
1. The route is removed from the gateway or ingress.
2. The backend handler is disabled or deleted.
3. DNS and certificates are retired if no longer needed.
4. Monitoring confirms no legitimate traffic remains.
5. Logs show only expected rejection traffic after shutdown.

**API testing and verification**

Security testing should be continuous, not a one-time event. Combine automated checks with manual review for business logic flaws.

Useful testing methods:
- **Schema validation testing** to confirm strict request and response contracts
- **Authorization testing** to verify object-level and field-level access controls
- **Fuzzing** to send malformed, oversized, or unexpected inputs
- **Dependency scanning** to identify vulnerable libraries and transitive packages
- **Dynamic testing** against running endpoints
- **Static analysis** of source code and configuration
- **Negative testing** for invalid tokens, expired tokens, and tampered signatures

Example `curl` checks for basic verification:

```bash
curl -i https://api.example.com/v1/accounts/123
curl -i -H "Authorization: Bearer $TOKEN" https://api.example.com/v1/accounts/123
curl -i -H "Authorization: Bearer $TOKEN" https://api.example.com/v1/accounts/124
```

You should expect different results only when the caller is legitimately authorized. If the same token can access another user’s object, you likely have an IDOR issue.

For fuzzing and contract testing, tools such as `ffuf`, `burp suite`, `owasp zap`, or schema-aware API test runners can help identify hidden routes, parameter tampering, and parser weaknesses. Use them in authorized environments only.

**Secrets management and key governance**

API credentials are high-value secrets. Store them in a dedicated secrets manager or vault, not in code repositories, CI logs, or shared chat tools. Access to secrets should be limited by role and environment.

Good practices:
- Issue separate credentials for dev, test, staging, and production
- Rotate secrets on a schedule and after incidents
- Use short-lived tokens where possible
- Restrict who can create, view, export, or revoke secrets
- Audit secret access and changes
- Avoid embedding secrets in container images or mobile apps

A practical vault-oriented workflow might look like:
1. The application authenticates to the vault using its workload identity.
2. The vault returns a short-lived API token or signing key.
3. The application uses the secret in memory only.
4. The secret expires automatically or is rotated before expiry.

**Logging, monitoring, and detection**

Logs are essential for incident response, but they can also leak sensitive data. Log enough to support forensics without storing secrets.

Log:
- Request ID or correlation ID
- Caller identity
- Timestamp and source
- Endpoint, method, and status code
- Authorization decision outcome
- Rate-limit or quota events

Do not log:
- Full bearer tokens
- API keys
- Passwords
- Full payment data
- Sensitive personal data unless required and approved

Monitoring should alert on:
- Spikes in 401, 403, and 429 responses
- Repeated access to unauthorized object IDs
- Unusual geographies or user agents
- Enumeration patterns across sequential IDs
- Unexpected new endpoints or methods
- Certificate failures or mTLS handshake errors

**Third-party and partner API governance**

Partner APIs introduce contractual and legal risk as well as technical risk. CISSP-style governance expects you to consider data ownership, retention, breach notification, and service expectations.

Review:
- Data-sharing agreements and privacy requirements
- Service-level agreements (<abbr title="Service Level Agreement">SLA</abbr>) and support expectations
- Security requirements for authentication, encryption, logging, and incident response
- Right-to-audit clauses and evidence requirements
- Data minimization and purpose limitation
- Offboarding and credential revocation procedures

When a third party consumes your API, define:
- What data they may access
- How long they may retain it
- Whether they may cache or redistribute it
- How they must protect credentials
- How you will revoke access if the relationship ends

**Practical secure API checklist**

Use this as a quick exam and design review aid:
- Enforce TLS 1.2+ and validate certificates
- Use mTLS for high-trust machine-to-machine scenarios
- Authenticate with the least powerful mechanism that meets the need
- Validate JWTs, signatures, timestamps, and audiences correctly
- Apply object-level and field-level authorization on every request
- Reject unexpected content types and validate schemas strictly
- Limit rate, quota, payload size, and query complexity
- Harden CORS and CSRF controls for browser clients
- Return minimal errors and suppress internal details
- Inventory endpoints and detect shadow APIs
- Store secrets in a vault and rotate them regularly
- Test with schema checks, fuzzing, authz tests, and dependency scans
- Govern partner access with contracts, SLAs, and revocation processes

**Exam focus reminders**

API security questions often test whether you can distinguish similar controls:
- **TLS** protects data in transit; it does not replace authentication or authorization.
- **Authentication** proves identity; **authorization** limits actions.
- **CORS** controls browser access; it does not secure the API by itself.
- **Rate limiting** reduces abuse; it does not validate business logic.
- **Input validation** reduces malformed requests; it does not replace access control.
- **Logging** supports detection; it must not leak secrets.

A strong answer usually combines preventive, detective, and governance controls rather than relying on one mechanism alone.
